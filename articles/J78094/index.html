フォルダーの準備


ホームディレクトリ（または都合の良い場所）に、未来のパッケージのすべてのファイルが置かれるフォルダーを作成します： mkdir ~/supersh



。 次に、 パッケージのルートを呼び出します 。

パッケージのルートに「DEBIAN」フォルダーを作成します（大文字で、これは重要です！）。 このフォルダーには、パッケージの生成を制御する情報が含まれており、パッケージのインストール時にディスクにコピーされません。

また、パッケージルートフォルダーには将来の「ディスクルート」が含まれます。パッケージをインストールすると、すべてのファイル（「debian」フォルダーを除く）が/ルートに解凍されます。 したがって、スクリプトは、パッケージのルートを基準にして次のように配置する必要があります： "usr / bin / super.sh"

黒地に白：

mkdir -p ~/supersh/DEBIAN #

mkdir -p ~/supersh/usr/bin #

cp super.sh ~/supersh/usr/bin/ #





その結果、次のことができます。

supersh/DEBIAN/

supersh/usr/

supersh/usr/bin/

supersh/usr/bin/super.sh

DEBIAN / conffiles：構成ファイルのリスト


通常、パッケージには、/などでホストされている構成ファイルの空白が含まれています。 明らかに、パッケージ内の構成が更新されると、ユーザーは編集した構成を失います。 この問題は、「config.d」タイプのフォルダーを使用して簡単に解決できます。その内容はメイン構成に含まれ、重複するオプションを置き換えます。

「DEBIAN / conffiles」ファイルを使用すると、問題をさまざまに解決できます。構成ファイルのリストが含まれています（1行に1つ）。 パッケージの現在のバージョンでこれらのファイルのいずれかが更新されると、ユーザーは構成のバージョンの競合に関する警告を受け取り、削除、置換、またはマージを選択できます。

構成を掘る各Linuxoidは、必ずこの状況に直面しました:)そして、ここから脚が伸びていきます。

各行には、各構成への完全な絶対パスが必要です。 例：

/etc/supersh/init.conf

/etc/supersh/actions.conf

DEBIAN /（preinst | postinst | prerm | postrm）：インストールスクリプト


合計で、1つのパッケージに最大4つのスクリプトを作成できます。

スクリプト 予定
デビアン/ preinst パッケージをインストールする前に実行されます：インストールを成功させるために何かを準備できます
デビアン/ postinst パッケージのインストール直後に実行されます。インストール済みのパッケージを構成して、使用できるようにします。 パッケージもここでインタラクティブに設定されます：これはdh_inputとDEBIAN / templatesファイルを使用して行われます
デビアン/ prerm パッケージがアンインストールされる直前に実行されます。通常、このスクリプトはパッケージのインストールパスをクリーンアップするため、何も残されません:)
デビアン/ postrm パッケージが削除された直後に実行されます：残りをクリーンアップします


これらのスクリプトで発生したエラーは決してログに記録されないことに注意してください。スクリプトのリターンコードよりも興味深いものはどこにも保存されず、ログは手動で行う必要があります。 私のパッケージのいずれかのユーザーがLinux Mintにインストールできず、理由を把握するためにエラーログ（存在しない）を要求することさえできませんでした:)

各スクリプトの冒頭で次のディスクを使用することをお勧めします。発生したすべてのエラーをsyslogに保存します。

#!/bin/bash

set -e # fail on any error

set -u # treat unset variables as errors



# ======[ Trap Errors ]======#

set -E # let shell functions inherit ERR trap



# Trap non-normal exit signals:

# 1/HUP, 2/INT, 3/QUIT, 15/TERM, ERR

function err_handler {

local exit_status=${1:-$?}

logger -s -p "syslog.err" -t "ootync.deb" "supersh.deb script '$0' error code $exit_status (line $BASH_LINENO: '$BASH_COMMAND')"

exit $exit_status

}



... ...



exit 0

DEBIAN /テンプレート：ダイアログテンプレート


すでに述べたように、DEBIAN / configスクリプトでは、ユーザーに質問をすることができます。行を入力し、オプションの1つを選択し、チェックマークを付けます...これは、debhelperパッケージdebconfのbash関数の「ライブラリ」によって行われます。 ここでは考えません。

DEBIAN / templatesファイルには、ダイアログボックス（GUIまたはncurses）の表示に使用されるデータが含まれています。 ファイルには、空の行で区切られたブロックが含まれています。 各ブロックは、特定のダイアログボックスで使用されるリソースを定義します。

すべてのタイプのダイアログのヘッダーは標準です：

Template: supersh/template-name

Type: string

Default: Default-value

Description: Dialog-title

␣Dialog-text







テンプレート-テンプレートの一意の（1つのパッケージ内の）識別子。 スクリプト内で特定のダイアログを呼び出す必要がある場合、この名前が使用されます。

タイプ-テンプレートのタイプ。 次のタイプが定義されています：文字列、パスワード、ブール値、選択、複数選択、テキスト、メモ、エラー。

デフォルト値-デフォルト値：ユーザーは単純に受け入れることができます。

説明-制御ファイルと同様に、短い説明と長いテキストの2つのフィールドで構成されます。 1つ目は「ウィンドウ」のタイトル、2つ目はユーザーに必要な内容のより詳細な説明です。 「enter」などの単語を使用することはお勧めしませんが、すぐに「Greeting script」、「Mount point」などのエッセンスを使用することをお勧めします。

ひも テキスト文字列プロンプト
パスワード パスワードプロンプト。

明らかな理由により、このタイプのテンプレートにはデフォルト値はありません:)
ブール値 チェックマーク:)「true」または「false」の文字列値があります

#!/bin/bash -e

. /usr/share/debconf/confmodule



case "$1" in

configure|reconfigure)

#

db_input medium "supersh/greeting" || true #

db_go || true #

#

db_get "supersh/greeting" # $RET

greeting="$RET"

echo "$greeting" > /etc/supersh/greeting.txt

;;

*)

echo "config called with unknown argument \`$1'" >&2

exit 1

;;

esac

#

db_input medium "supersh/greeting" || true #

db_go || true #



#

db_get "supersh/greeting" # $RET

greeting="$RET"

echo "$greeting" > /etc/supersh/greeting.txt







ここにはすでに不快な待ち伏せがあります：db_input関数には中程度のダイアログの優先度が与えられていることに注意してください。 debconfでは、最小優先度を設定できます。優先度がそれ以下のダイアログは表示されませんが、デフォルト値（デフォルトテンプレート）が使用されます！ これが正確に行われないようにするために、重要な優先順位を使用します:)さらに、GUIからインストールする場合、質問を出力するためのしきい値が高くなり、それらの多くはまったく表示されません。

可能な優先順位：低-デフォルトが常に使用され、中-デフォルトは通常非常に適切であり、高-デフォルトは望ましくない、重要-ユーザーの注意が不可欠です。

パッケージを収集します！ :)



やった！ すべての必要なファイルが作成され、必要なフォルダーにあります。 これで、パッケージを収集できます:)

最初に行うことは、ユーザーパッケージのルートとルートグループ内のすべてのファイルを再帰的に公開することです：root（または必要に応じて他の）。 パッケージファイルは、ファイルのアクセス許可と所有者の両方を保存するtar.gzアーカイブにパッケージ化されているため、これが必要です。 したがって、次を実行する必要があります。

$ sudo chown -R root:root .





ただし、これは必要ありません。 アーカイブを作成するときに、ファイルの所有者をルートに置き換える優れたfakerootコマンドがあります。

この例では、スクリプトに実行可能性ビットが必要です。

次に、フォルダーに戻り、パッケージのルートフォルダーが表示されるようにします。パッケージは、 軽いキック自体で作成されます。

$ fakeroot dpkg-deb --build supersh





作成されたパッケージは、* .debパッケージの命名順序と一致するように名前を変更する必要があります。<package name> _ <version> _ <architecture> .deb

$ mv supersh.deb supersh_1.0-1_all.deb





すべて、パッケージの準備ができました！

独自のパッケージリポジトリを作成する



これで独自のパッケージができました。 それらのいくつかがあり、さらに依存関係がある場合は、ローカルのマイクロリポジトリをすぐに上げてパッケージマネージャーソースのリストに含める方がはるかに便利であることがわかります:)ここでは、簡単なHowTo「独自のリポジトリを作成する方法」について説明します。 関連するドキュメントを読むことで、アイデアを簡単に開発できます:)

最初にヘルパーをインストールします。

$ sudo apt-get install reprepro







将来のリポジトリの説明


リポジトリセンター-その説明。 主なものは、リポジトリコンポーネントのリストです。 「ソフト」および「ゲーム」コンポーネントを作成します。

将来のリポジトリ用のフォルダを選択します。 すべてのアクションはそのルートから実行されます。

次の内容のconf / distributionsファイルを作成します。

Description: my local repository

Origin: Ubuntu

Suite: testing

AlsoAcceptFor: unstable experimental

Codename: karmic

Version: 5.0

Architectures: i386 amd64 source

Components: soft games

UDebComponents: soft games







単純なリポジトリを作成するビジネスでは、すべてのフィールドが基本的な役割を果たしているわけではなく、「何が何であるか」を視覚的に判断するためにのみ使用されます。



リポジトリを作成


リポジトリについて説明します！ 次に、説明に基づいて空白を生成します。 コマンドは、リポジトリのルートで実行されます。

$ reprepro export

$ reprepro createsymlinks





そして、完成したリポジトリを/etc/apt/sources.listに追加します。

deb file:///path/to/repo/ karmic soft games





Webサーバーを使用してこのリポジトリを共有することもできます。



リポジトリ内のパッケージを管理する


リポジトリのルートに、追加する* .debファイルを配置し、karmicディストリビューションのソフトコンポーネントに追加します。

reprepro -C soft includedeb karmic *.deb





これで、パッケージマネージャーからパッケージを入手できます。

パッケージの削除：

reprepro -C soft remove karmic supersh
