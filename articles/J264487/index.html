PHPサイトを高速化し、最適化します。 PHP用のサーバーをセットアップするときに選択するテクノロジー





この記事は、PHPサイトの所有者、開発者、およびシステム管理者からの質問に答えるのに役立ちます。





  • サイトを最適化し、作業を高速化する方法は？
  • サイトの立ち上げ時のテクノロジーに応じて、サイトはどの程度の速度で機能しますか？
  • サーバーまたはVPSをセットアップするとき、どのテクノロジーを使用する必要がありますか？




典型的な問題：

ある時点で、サイトが開き始め、動作が遅すぎます。 ホスティング会社が過剰な負荷またはリソースのオーバーランのためにサイトをブロックすることが起こります。 この状況で何をすべきか？



サイトが高すぎるトラフィックを使用し始めているか、リソースを集中的に使用するモジュールがインストールされているか、攻撃が行われているか、サイトがウイルスに感染している可能性があります。 いずれにせよ、これらのケースにはすべて共通点があり、これはすべてのホスティングサイトのすべてのサイトにとって問題です。



そして、PHP用のサーバーについて話す場合、そのような問題は、サーバー上の他の重要な環境設定と同様に、PHPコードの実行方法です。

コードに問題があるかどうかに関係なく、トラフィックが多いかどうかに関係なく、多くはサーバーの設定に依存します。 すべての言葉が空の言葉に聞こえるわけではなく、この記事は書かれています。



このレビューでは、最も一般的なDrupal 7.33コンテンツ管理エンジンの1つにインストールしたサイトをテストします。



テストには、PHPホスティングの1つのコンポーネントのみが選択されます。 NginxおよびApache2 Webサーバー、 mod_phpおよびphp- fpm モジュール 、phpバージョンphp53およびphp56をテストし 、 apcおよびopcacheオプティマイザーがサイトの速度にどのように影響するかを確認します。





もちろん、これらのパラメーターは、サイトの速度が依存する設定の一部にすぎません。 しかし、意図的に

レビューを無限にしないように、これに限定します。

与えられた：




  • オペレーティングシステムCentos 6.7
  • データベースサーバー：MariaDB 10.21
  • すべてのサイトセッションはmemcacheに保存され、セッションのセットアップ速度がサイトのパフォーマンスに与える影響を取り除きます。
  • すべてのテストで、フロントエンドはWebサーバーnginx 1.93です。 Apache2の場合、Nginxはバランサーとしてだけでなく、静的な戻り値としても機能します。 Apache2を使用しない構成の場合-直接WebサーバーはNginxです
  • 構成NginxおよびMariaDBには、最大のパフォーマンスを達成することを目的とした多くの最適化が含まれていますが、テストのすべての参加者にとって、これらの設定は同じであるため、それらの影響は無視する必要があります
  • opcacheおよびapcパラメーターは、ほとんどのサイトに最適で汎用的であるため、Bitrixの推奨事項から取得されます








どのようにテストしますか？




ローカルネットワーク上にzabbixサーバーとそのタスクが毎分あります：



  • テストサイトのメインページを開き、ページ上の特定のコンテンツを待ち、サーバーからの応答がコード200であることを確認します。
  • 次のステップは、サイトの管理パネルでの承認です。これは、適切なPOSTリクエストを送信することで行われます。 ページ上のテキストと応答コードと埋め込み標準の調整。 このステップはWebサーバーのほぼすべてのサブシステムに適用され、多くの点でその速度はデータベースとの対話の速度に依存します
  • 最後のステップは、サイト管理パネルを終了し、終了ページで応答コードとテキストを確認することです
  • 各ステップの最後に、zabbixはhtmlコード内のphpコードのレンダリング速度を細心の注意を払って測定して記録し、ブラウザーに表示して、結果のグラフを表示します
  • 各被験者について、値は1時間記録され、その時間の平均値が結果になります
  • テストはローカルネットワーク内で行われるため、インターネット接続の速度の結果への影響は除外されます。
  • 便宜上、すべての結果を昇順で表示します。 つまり 最初の結果が最も遅くなります。 すべての構成は条件付きの番号の下に配置されました。これにより、結果の方向性が向上します
  • トップチャート-コード生成速度;値が高いほど良い。 下のグラフ-サーバーの応答時間と値が低いほど良い
  • テストされたサイトは生活を送り、定期的にデータベース操作を実行し、スケジュールされたタスクを実行します。これが、グラフの曲線が上下する理由です。








テスト：





1. opcacheオプティマイザーを使用しないNginx + php-fpm56





アーキテクチャ別-これは最も高度なオプションの1つです。 パフォーマンスの面で-最大の失望。







パフォーマンスには多くのことが望まれますが、このようなオプションは、Apache2のオプション＃2よりもはるかに優れた負荷に耐えます。 また、このオプションはRAMをより効率的に消費します。







2. Apache2 + apcオプティマイザーなしのmod_php53





最も典型的なホスティングオプション。 人気のあるホスティングプロバイダーの90％がこのオプションを使用しています。 php53は開発者によって長い間サポートされていませんが、インターネット上にはまだこのバージョンで動作している多くのサイトがあります。







このオプションは非常に遅いだけでなく、Apache2ワークフローの不足やサーバーのRAM不足のために、すぐに小さな負荷に陥ります。







3. opcacheオプティマイザーを使用しないApache2 + mod_php56の動的部分であるNginxによるバランスと静的





このオプションは、最新のサイトのソリューションとして作成されました。 PHPの最新バージョンを提供しようとするホスティング会社によって提供されます。 一般的な考えによると、このバージョンのPHPは以前のものよりも高速で安全であるはずです。







残念ながら、すべてのサイトがこのバージョンで完全に動作できるわけではありません。 PHPのほとんどすべての新しいバージョンは、いくつかの古くて「安全でない」機能のサポートを停止し、「古い」コードの操作を中断します。

オプティマイザーなしのPhp56自体はかなり遅く、mod_phpはクラッシュし、負荷がかかっているサーバー上のすべてのメモリを占有する傾向があります。







4. apcオプティマイザーを使用しないNginx + php-fpm53





コードオプティマイザーのエラーが原因で問題が発生しないようにする、かなり高度な構成。 この場合、「互換性のある」バージョンのPHPインタープリターが使用され、リソースを集中的に使用するApache2もバンドルから削除されます。











5. Apache2 + mod_php53 + apcの動的部分であるNginxによるバランスと静的





別の一般的なバリエーション。 多くのホスティング会社がこれを使用し、デフォルトで使用するか、コントロールパネルでオプティマイザーを有効にすることができます。

通常、Apache2は、リンク変換やCNCなどの.htaccessルールをそのまま残します。







オプティマイザーを使用しないオプションと比較して、3.5倍の速度向上が得られます。

Apache自体（独自のモジュールmod_phpを使用）は、php-fpmを使用したバージョンよりも作業に多くのリソースを消費します。 Apache2は、モジュールの1つがクラッシュしたり、サーバーのRAMをすべて埋めたりすると、クラッシュする傾向があります。







6. Nginx + php-fpm53 + apc





複雑な.htaccessを必要としない古いエンジンのサイトに最適なオプション







これは、高負荷下で満足のいく速度と信頼性の高い動作を実現するために、古いサイトを上げる必要がある場合に使用するオプションです。







7. Apache2 + php-fpm53 + apcの動的部分であるNginxによるバランスと静的





複雑な.htaccessを持つレガシーサイトのオプション。 たとえば、古いBitrixインストール。







これは、レガシーサイトに最適です。 この構成は、高負荷に耐え、互換性があり、非常に生産的です。

.htaccessルールとApache2アドオンが必要な場合に最適です。

欠点の1つはphpの古いバージョンで更新されていないバージョンですが、選択の余地がない場合はこれが最良のオプションです。 Bitrix、Joomla、その他の一般的なCMSの最新バージョンではなく、古いバージョンに最適です。







8. Apache2 + mod_php56 + opcacheの動的部分であるNginxによるバランスと静的





mod_phpの欠点をすべて備えた、かなり生産的ですが、リソースを集中的に使用する構成。







十分に高速ですが、負荷がかかるとサーバーのメモリが不足し、速度が大幅に低下します。







9. Nginx + php-fpm56 + opcache





最も生産的なオプション。







これは、最新のすべてのサイトに最適なオプションです。 負荷を十分に保持し、パフォーマンスの面で最高の結果を示します。 これはまさに、タスクがサイトのパフォーマンスを最適化し、速度を上げることであるときに使用するオプションです。

唯一の欠点は、.htaccessを使用できず、すべてのmod_rewriteルールをNginx構文に書き換える必要があることです。

Apache2モジュールも機能しません。 使用されている場合、このオプションは機能しません。







10. Apache2 + php-fpm56 + opcacheの動的部分であるNginxによるバランスと静的





.htaccessが必要なサイトに最適なオプション。 Bitrixの新しいバージョンに最適です。







php-fpmによる負荷を十分に保持します。 ほとんどのサイトでこのオプションを積極的に使用しています。

その結果：





  • 私のテストでは、意図的にNginxの静的な戻り値に合格したため、実際には、Apache2のすべてのバリアントは遅くなる可能性があります。 これは、PHPインタープリターの速度を測定した結果に対する静的収益率の影響を排除するために行われます。 Apache2の弱点の1つであると同時にNginxの強みは、静的出力の速度です。 特に、高負荷時に顕著です。 さらに、Nginxは「遅い接続」を攻撃する傾向が少ない
  • mod_phpは利用可能なすべてのサーバーメモリを非常にすばやく占有し、ロード時にパフォーマンスを失います
  • php-fpmは、設定でメモリをより効率的に、より安全に、より柔軟に消費します。 場合によっては、高速で高負荷がありません。
  • このテストの特異性は狭く、ここではDrupalエンジンの機能を確認しました。他のエンジンの動作は異なる場合がありますが、一般的な傾向は同じです。








そして最も重要なこと-サイトの速度は、サーバーまたはホスティングの構成に依存します。 適切なアーキテクチャを選択すると、サイトの速度が5倍に向上します。
