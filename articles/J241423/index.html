Laravelのパッケージを作成する

こんにちはHabr！



Laravelフレームワークは急速に人気を集めており、すでに多くのファンを獲得しています。 この記事では、Laravel用の単純なパッケージの開発、およびcomposer.jsonの1行でパッケージをプロジェクトに追加するためにpackagist.orgで作成したパッケージの公開について説明します。



それで、これについてのすべては何ですか



公式のドキュメントに書かれているように、パッケージはLaravelに新しい機能を追加する主な方法です。 Laravel専用のパッケージを作成します。これは、エントリのリストと記事を表示するためのページを備えたシンプルなブログです。 私たちは、カスタマイズの可能性がたくさんある優れた普遍的なブログを書くことを目指していません。パッケージを作成することの詳細はすべて、日常的なタスクを解決するときにプロジェクトでコードの重複を避けるのに役立つ独立した拡張機能としてのみ考慮します。



最後に、それを取得します。







私たちが学ぶこと



  • プロジェクトの機能を拡張するパッケージを作成する方法。
  • パッケージ内の基本的な構造を理解しましょう。
  • ルートを追加する方法を理解しましょう。
  • テンプレートを保存する場所を見つけます。
  • スタートアップに追加する必要があるクラスを作曲家に示します。
  • 移行を作成し、テーブルに初期データを入力してみましょう。
  • パッケージの外部リソース（css、js-ファイル）を利用可能にします。




必要なツール



私はPhpStorm IDEを使用してMacOS用に開発しており、サイト自体は公式Vagrantボックス-Homesteadで回転します。

habrには、Vagrantイメージの設定方法とLaravel自体のインストール方法に関する有用な記事が既にありました。



パッケージは最終的に何を表しますか？



彼には2つのルートがあります。


  • 投稿のリストがあるページ。
  • 別の投稿ページ。




いくつかのテンプレート：


  • ブログの一般的なレイアウト。
  • エントリのリストがあるページ。
  • 投稿リストのエントリの短いビュー。
  • 記録の全景。




フィールドを持つ1つの投稿モデル：


  • 役職
  • ユニークな投稿リンク
  • 投稿テキスト
  • 作成時間
  • 更新時間




また、エントリのテキストから最初の120文字が取得される計算フィールドカット。

すべてに加えて、必要なデータをテンプレートにバインドするために、ページ表示ロジックと追加のコンパイラファイルをグループ化するコントローラークラスを作成します。

結論として、投稿を含むテーブル移行ファイルを作成し、一定量の初期データを含むシードクラスも作成します。

アプリケーションのパブリックフォルダーにパッケージスタイルを公開します。



仕事を始める



これで、Laravelがインストールされ、お気に入りのIDEが開かれ、開始できます。







名前とメールのパラメーターをapp / config / workbench.phpファイルに書き込みます。将来、これらはパラメーターとしてcomposer.jsonで使用されます。

これらのパラメーターを登録したら、コンソールで実行します。 



php artisan workbench vendor/package --resources


vendorはサービスプロバイダーの名前で、packageは作成するパッケージの名前です。



たとえば、githubでのログインはcherryoffであり、作成したパッケージnbblogを呼び出したいので、コマンドを実行する必要があります。 



 php artisan workbench --resources cherryoff/nbblog








resoursesフラグは、Laravel固有のフォルダー（移行、ビュー、構成など）も作成する必要があることを示しています。

すべてが順調に進んだ場合、プロジェクトのルートに表示されます：







Laravelがアプリケーションの起動時にパッケージを自動的にロードするには、 providers配列のapp / config / app.phpファイルに行を追加する必要があります。 



 'Cherryoff\Nbblog\NbblogServiceProvider',




サービスプロバイダーのクラス名は[Package] ServiceProviderスキームに従いますが、完全なクラススペース名はこの前に指定されます



パッケージ構造



srcフォルダーには、おなじみのフォルダーの構造があり、その名前はそれ自体を表しています。

src / Cherryoff / Nbblog /フォルダーにはサービスプロバイダーのクラスが含まれており、パッケージのすべての補助クラスがそこに配置されていることに注意してください。

NbblogServiceProviderクラスをご覧ください 。 registerメソッドは、パッケージが登録されるとすぐに呼び出され、 bootメソッドは、リクエストを処理する前に毎回呼び出されます。



ブログの作成に焦点を当てる



パッケージのsrcフォルダーで、2つのルートを含むroutes.phpファイルを作成します 。 

 Route::get('/blog/', array( 'as' => 'posts_list', 'uses' => 'Cherryoff\Nbblog\NbblogController@showList' )); Route::get('/blog/{link}', array( 'as' => 'post', 'uses' => 'Cherryoff\Nbblog\NbblogController@showPost' ))->where('link', '[A-Za-z-_]+');




コントローラーとして、まだ作成されていないコントローラーへのフルパスを指定します。

次に、 コントローラーフォルダーで、次の内容のNbblogControllerコントローラーを作成します。 



 <?php namespace Cherryoff\Nbblog; use Illuminate\Support\Facades\App; use Illuminate\Support\Facades\Config; use Illuminate\Routing\Controller; use Illuminate\Support\Facades\Redirect; use Illuminate\Support\Facades\Response; use Illuminate\Support\Facades\Request; use Illuminate\Support\Facades\View; use Illuminate\Support\Facades\Input; use Illuminate\Support\Facades\File; use Illuminate\Support\Facades\Session; use Illuminate\Support\Facades\URL; class NbblogController extends Controller { public function showList() { return 'Posts list'; } public function showPost($link) { return 'Get post:'.$link; } }




さて、これでパスが設定され、コントローラーは簡単なメッセージで応答するはずです。 しかし、リンクsandbox.local / blogをたどると、ページが見つからなかったという例外が発生します。 問題は、パッケージが独自のパスを持っていることをアプリケーションが認識していないことです。これを修正するために、 NbblogServiceProviderクラスのbootメソッドの最後にroutes.phpファイルを含めます。 



 include __DIR__.'/../../routes.php';




しかし、その後でも何も機能せず、エラーをキャッチします。



クラスCherryoff \ Nbblog \ NbblogControllerは存在しません



これは、パッケージ内の作曲家が新しく追加されたコントローラーについて何も知らないことを示しています。 composerに必要なファイルを探す場所を示すには、 classmapセクションのcomposer.json （私たちのパッケージ!!!）に「src / controllers」という行を追加してから、実行します 

 composer dump-autoload

次に、 modelsフォルダーとその中にPost.phpファイルを作成します。 ファイルリストを以下に示します。 



 <?php namespace Cherryoff\Nbblog; use Illuminate\Database\Eloquent\Model; use Illuminate\Support\Str; use Illuminate\Support\Facades\Config; /** *     */ class Post extends Model { protected $table = 'posts'; //     protected $appends = array('cut'); //     protected $fillable = array('header', 'link', 'article'); //   public static $rules = array( 'header' => 'required|max:256', 'link' => 'required|between:2,32|unique', 'article' => 'required' ); public function getCutAttribute(){ return Str::limit($this->attributes['article'], 120); } }




ここで、 モデルフォルダーをプロジェクトのスタートアップセクションcomposer.jsonに追加し（上記の「 src / controllers 」行を追加する必要があります）、コントローラーの場合と同じようにcomposer dump-autoloadを実行する必要があります。



Laravel WebサイトでEloquentモデルの詳細を読むことができます。



コンソールの行により、パッケージの移行が作成されます。 



  php artisan migrate:make create_nbblog_posts_table --bench="cherryoff/nbblog"




src / migrations /フォルダに、新しく作成された移行のクラスが表示されました。 彼のupメソッドでは、次のように記述します。 



 Schema::create('posts', function (Blueprint $table) { $table->increments('id'); $table->string('link', 32); $table->string('header', 256); $table->text('article'); $table->timestamps(); $table->softDeletes(); });




downメソッドでは： 



 Schema::dropIfExists('posts');




したがって、 upメソッドでは、モデルに必要なフィールドを持つテーブルを作成し、 downメソッドでは、存在する場合は削除します



それでは、移行を完了する時が来ました。 



 php artisan migrate --bench="cherryoff/nbblog"




データベース接続を正しく構成すると、コンソールに次のようなものが表示されます。 



 Migration table created successfully. Migrated: 2014_10_23_115450_create_nbblog_posts_table

 class NbblogController extends Controller { public function __construct(){ $this->layout = View::make('nbblog::layout'); } public function showList() { $this->layout->content = View::make('nbblog::list'); } public function showPost($link) { $post = Post::where('link', '=', $link)->firstOrFail(); $this->layout->content = View::make('nbblog::post', $post); } }
