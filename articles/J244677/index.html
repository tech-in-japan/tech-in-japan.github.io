NetApp FASシステムのバックアップアーキテクチャ

この記事では、SnapProtectアーキテクチャが高度なストレージテクノロジーとFASシリーズのストレージシステムの利点を活用して、 「NetAppバックアップパラダイム」を実現する方法を説明します。 SnapProtect（SP）ソフトウェアは、NetApp FAS ストレージにあるインフラストラクチャ全体のバックアップライフサイクル、バックアップ、およびリカバリを管理するように設計されています 。 SPは、アプリケーションとの接続、バックアップバックアップの一貫性、リポジトリ間のレプリケーション/アーカイブ管理、カタログ化、必要に応じたデータリカバリ、バックアップ検証、およびその他の機能を提供します。 FASは 、「プライマリ」 ストレージ 、「スペア」（ DR ）、およびデータアーカイブの両方のタスクに使用できる非常に用途の広いシステムです。



SnapProtectコンプレックスは、次の主要コンポーネントで構成されています。



  • SnapProtect Management Server（CommServeライセンス）を搭載したサーバー、クラスタリングはフォールトトレランスに使用されます（アプリケーションレベル+ データベースクラスタリング）
  • MediaAgentがインストールされているサーバー。
  • IDataAgentエージェント（iDA）。 OS 、ファイルシステム、アプリケーション、およびホストOSの他のコンポーネントと統合するためにホストにインストールされます。
  • SPは、Oncommand Unified Managerを介してNetAppリポジトリと通信します。








そのようなiDAエージェントがあります。

  • VSA -VMWareおよびHyper-Vの場合
  • Windows / Unix / LinuxでのOracle（ RACを含む）
  • Exchange（ DAGを含む）
  • MS SQLの場合
  • スネアポイント用
  • Win / Unixファイルシステム用Qsnapドライバー
  • NAS NDMP iDA
  • DB2 Unix / Linux
  • Windows上のLotus Domino
  • Active Directory iDA




前述のように、一貫性はホスト上のiDAエージェントを使用して実行されるため、ストレージ側のハードウェアアシスタントSnapShotを削除する前に、エージェントはアプリケーションを「準備」します。 このようなバックアップ方法は、営業日の途中で何度も実行できます。そのようなアプリケーションに接続しているクライアントはこれらのプロセスに気付かないからです。



ストレージ間の（一貫した）スナップショットの複製は、SnapMirrorまたはSnapVaultを使用して実行できます。

プライマリシステムまたはバックアップシステムからテープライブラリへのデータの注入は、 SANおよびNASデータのメディアエージェントを介して実行されます。 また、 NASの場合、 NDMPおよびSMtapeプロトコルを使用して、NetAppストレージからテープライブラリにデータを直接アップロードできます。残念ながら、この場合、データカタログはサポートされていません。



カタログ化は、NetAppリポジトリからのクローンスナップショットの増幅を使用した後処理として実行されます（FlexCloneテクノロジー）。



データリカバリは、リモートストレージ、ローカル、およびテープライブラリの両方から実行できます。

データの種類、回復の種類、およびこのデータの配置に応じて、いくつかの回復方法（方法を言うこともできます）があります。



そのため、スナップショットからローカルストレージにボリュームまたはqtree全体を復元するには、スナップショットからのストレージを使用したインスタントリカバリテクノロジー-SnapRestoreを使用できます。 同様のリカバリをリモートシステムから行う場合、SnapMirrorまたはSnapVaultテクノロジーを使用して「リバースレプリケーション」を実行する必要があります。 最新のデータだけでなく、リモートストレージにあるスナップショットの1つも複製できます。



「グラニュラーリカバリ」が必要な場合、内部ロジックはリモートまたはメインシステムのスナップショットのクローンを作成してメディアエージェントに接続するように見え、必要なオブジェクトがメディアエージェントから必要なホストに転送されます。 テープライブラリからの回復は、メディアエージェントを介してホストに至るまで、同様の方法で行われます。



ストレージ施設に必要な技術：

したがって、まとめると、FlexClone、SnapRestore、レプリケーションライセンスの1つ（SnapVault / SnapMirror）、およびSnapProtect自体がメインサイトに存在する必要があると言えます。



バックアップストレージでは、そこでカタログ化を実行する場合はFlexCloneが必要です。 バックアップ/復元スキームでスナップショットからリモートストレージにボリュームをすばやく復元する機能が必要な場合、そのようなストレージにはSnapRestoreテクノロジーが必要です。 メインシステムからリモートシステムにデータを複製しているため、リモートシステムで複製技術（SnapVault / SnapMirror）をサポートする必要もあります。 他のすべては、SnapProtect自体のサポートが必要です。



その結果、NetAppは、「2サイトモデル」の場合、SnapProtectアーキテクチャを展開するためのプライマリストレージとバックアップストレージの追加ライセンスを厳しく規制します。







個別リカバリ。

リカバリ粒度とは、 LUN /ボリューム全体をデータで復元するのではなく、 LUNにある個々のオブジェクトまたはファイルを復元する機能を意味します。 たとえば、仮想インフラストラクチャの場合、そのようなオブジェクトは、仮想マシン、仮想マシンファイル、vDisk、または仮想マシン内の個々のファイルです。 データベースの場合、このようなオブジェクトは個別のデータベースインスタンスにすることができます。 Exchangeの場合、個別のメールボックスまたは個別のレターなどにすることができます。



SAN +詳細なバックアップと復元

SANネットワークでSnapProtectアーキテクチャを使用する場合、個別の専用LUNに配置するために、詳細なリカバリが必要なアプリケーションデータ（Exchange、Oracle、MS SQLなど）を保存する必要があります。 このようなアプリケーションを使用したサーバー仮想化の場合、同じルールが適用されます。これらのアプリケーションのデータを別のRDMドライブに配置する必要があります。 ストレージシステム側のLUN自体は、それぞれ個別のボリュームに存在する必要があります。 たとえば、Oracle データベースとSANネットワークを使用した仮想化の場合、SnapProtectの代わりにSnapMrotagerを使用した場合とまったく同じルールを使用して、このデータベースのデータタイプごとにRDMに接続された個別のLUNを割り当てる必要があります。 つまり SnapProtectのディスクのパーティション分割と配置のロジックはSnapManagerの場合と同じです。ここでは同じルールが適用されます。 実際、両方のケースでハードウェアアシスタントスナップショットを一貫して削除するためのテクノロジーは、同一ではないにしても類似しています。



この例は、 ギーク誌の記事「SnapManager for Oracle＆SAN Network」でより詳細に検討できます。
