JMeterの概要

この記事は、私が働いている会社のテスト部門によって作成されました。

ハブでJMeterについて書くのは初めてではありませんが、記事の焦点がかなり狭いか、理解しにくいです。 したがって、この記事は初心者にとって興味深いものです。



はじめに



どういうわけか私は新しいツール-JMeter-を習得し始めましたが、アクションのガイドとして本格的なガイドを見つけることができませんでした。 徐々に、かなり有用な記事やブログ、ユーザー文書を整理して、何かを理解し始めました。 そして、何かを始めたいと思っているのは私だけではないかもしれないと決めました。 構築の原則とツール自体のロジックのいくつかを習得します。 そのため、ここに開始方法に関する記事があります。

そのため、この記事では、次の3人のユーザーのためにスクリプトを作成する例を見ていきます。

1.ログイン;

2.アルバムに新しい写真を追加します。

3.プロフィール写真を変更します。

そのため、これはソーシャルネットワークの通常のビジネスプロセスです。



そして、これらの目的のために、そのような組み込みコンポーネントが使用されます：

  • HTTPプロキシサーバー
  • 録音コントローラーx3
  • ユーザー定義変数
  • CSVデータセット構成x3
  • 常時タイマー
  • 均一ランダムタイマー
  • HTTP自動化マネージャー
  • HTTPリクエストx6
  • デバッグサンプラーx3
  • HTTP Cookie Manager
  • 一度だけのコントローラー
  • おそらく他のいくつか。


記事は詳細だから大きい。 結果は次のようになります。





自動スクリプト記録



一番下の行は、ブラックボックスメソッドを使用してテストするサイトのスクリプトを作成するとき、その操作のすべての微妙さと機能がわからないということです。 そのため、たとえば、そのサイトは、私のスクリプトの最終的なベースになり、Symphony2パッケージを使用して作成されました。 そのため、ログイン/パスワードデータの送信は、/ログインページのフォームから直接ではなく、存在しない/ login_checkページから行われます。 そして、これらすべてに対して、ユーザーに対して非表示になっている別の空のフィールドが送信されます。

HTTPプロキシサーバーは、このような落とし穴の検出に役立ちます（これは、他のすべての場合と同様に、インライン要素です）。 プロキシサーバーの動作をエミュレートし、受信したリクエストと送信したリクエストをすべて記録します。



準備作業



HTTP Proxy Serverはまだテストツリーに追加しません。 受け取った「ステップ」をどこに書き留めるかについて話し合います。 後で、ツリーに要素を追加すると、結果を記録する場所がいくつかあることがわかります。 これらは、「コイル」（ スレッド ）、「ワークベンチ」（ワークバンチ ）、または「記録コントローラー」要素に直接書き込むことができます。 後者の方法を使用することをお勧めします。 まず、必要に応じてログ全体を一度に無効（ctrl + t）にできます。 第二に、テスト構造の追跡と形成が改善されます。

追加および編集のためのすべてのアクションは、コンテキストメニューでマウスの右ボタンをクリックすることで発生します。 作成したアイテムは、単にドラッグアンドドロップできます。

だから： テスト計画->追加->スレッド->スレッドグループ ; スレッドグループ->追加->ロジックコントローラー->レコーディングコントローラー

すべての要素に一度に明確な名前を付けることを強くお勧めします。 私の場合、これらはそれぞれ「Pictures」と「Login」になります。





HTTPプロキシサーバーを使用したスクリプトの作成



準備手順が完了したら、必要な要素を「ワークベンチ」に追加します： ワークベンチ->追加->非テスト要素-> HTTPプロキシサーバー

多くの設定がありますが、今のところ興味があるのはポートのみです。 ポート8080がビジーの場合、たとえば8089を設定できます。このマシン（8089）を使用しています。 また、Target Controllerフィールドのデフォルトでは「use recording controller」であることがわかります。 つまり ログはそこに記録されます。

残りのフィールドには触れませんが、サイトに関するもう少しの情報を受け取ったら、いくつかのフィールドが必要になります。



ポートを設定したら、ブラウザの設定に移動します。 私はIEを使用している。 ツール-インターネットオプション-接続-ネットワーク設定ボタン 。 「Use proxy server ...」というdawsを配置し、2番目のdawを削除します。 「詳細」を見て、「例外」に書かれていることを確認することをお勧めします。 私の実務では、テスト中にサイトが新しいドメインに移動されましたが、例外になっています。 仮想ユーザーのアクションがテストされたサイトに記録されず、Yandex、twitter、Facebookへのあらゆる種類のリンクが完全に記録される理由を理解するのに約30分かかりました。 一般に、サイトのドメインが突然そこにある場合は、削除します。

「アドレス」フィールドに「Localhost」と書き込み、JMeterからのポート「8089」を指定します。 もちろん、引用符なしで、すべて。



これで、すべてを録音する準備ができました。 プロキシでJMeterに戻り、下部で[スタート]ボタンをクリックします。 クリックすると、ブラウザのすべてのユーザーアクションが、作成した「ログイン」記録コントローラーに記録され始めます。 アクションをいくつかのステップで記録します。 スクリプトが大きすぎて編集が難しい場合があります。 したがって、最初に実行してからデバッグするのは、サイト上のユーザーの承認です。

そのため、[開始]をクリックして、サーバーでの認証手順（まだないかもしれませんが、私は持っています）とサイトでの認証手順を実行します。 JMeterに戻り、「停止」をクリックします。 録音コントローラーを見てみましょう：たくさんの子供がいます。 これがサーバーに送信されたもので、サーバーが応答しました。 勉強して掃除します。





スクリプトのデバッグ



スクリプトのデバッグとは、さまざまな.jpg、.png、およびサードパーティのリソースへのリンクを削除することです。 スクリプトにはこれらのサードパーティリソースの半分以上が含まれています。これらはさまざまなソーシャルネットワークやYandexマップとの接続です。 これらはすべて削除できます（削除を押します）。 一般に、.jsもクリーンアップできます。 主なことは、ユーザーの認証情報を体内で渡すリクエストを見つけることです。 さて、美しさのために、ユーザーがログインするページに導くリクエストを見つけてください。 したがって、彼らは一緒に「ページに行った-ログイン」したユーザーの行動の束をモデル化します。

幸い、必要なクエリを分離できたとします。 さて、または私たちはそれらを特定したように思われます。 読みやすい名前を付けてください！ 選択したかどうかを確認するには、スクリプトを実行する必要があります。 私の場合、スクリプトを実行すると、エラーが発生して実行されます。 実際には、テストのためにサーバーにログインする必要があります。 これを行うには、HTTP Autorization Manager（ スレッドグループ-追加-構成要素-HTTP Autorization Manager ）をツリーの上部に追加します。 直感的です。 リソースのアドレスは、 必ず http：//、ユーザー名、パスワードで書き留めます。



サーバーでの承認に承認マネージャーが必要な場合、Cookieはサイトでの承認中に必ず使用されます。 承認マネージャーの後にCookieマネージャーを追加します（ TestPlan-追加-構成要素-HTTP Cookieマネージャー ）。 ユーザーがログインする必要があるということになると、おそらく常にこの要素を追加する必要があります。

スクリプトを開始してチェックする前に、結果ツリーの表示要素（ コイル-追加-リスナー-結果ツリーの表示 ）を追加します。ここで、実行を確認できます。 この要素は、常に下部に配置すると便利です。 ああ、ブラウザの設定で「プロキシを使用する」チェックボックスも削除する必要があります。まだ必要ありません。



だから、実行することができます。 （ctrl + r）を開始します！

幸運なようです。 そして、ログインしなくても、少なくともリクエストはサーバーに届きました。 ユーザーが正常にログインしたことを確認するには、ツリーの回答を見てください。 [ 応答データ]をクリックします 。 サーバーの応答では、ユーザー認証が成功したことを示す情報を探しています。 私の場合、それはユーザー名になります。 承認後、ページに表示されます。



スクリプトはまだきれいにする必要があるかもしれません。 たとえば、HTTPヘッダーマネージャーを削除しました。 そして、それらがなければ、すべてが正常に動作します。 各要素が必要であることを確認するには、すぐに削除する必要はありません。この要素を無効にするだけで十分です（ctrl + t） 。

次に、このビジネスに3人のユーザーを引き付けます。 これを行うには、まずクエリのパラメーター化に関する小さなケーススタディを実行します。 特定の1人のユーザーをパラメーター化します。



パラメータ化



クエリをパラメーター化するには、一番上にユーザー定義変数を追加します。 スレッド->構成コントローラー->ユーザー定義変数 下部に追加ボタンがあります。 プッシュし、2つの変数で駆動します。 ユーザー名とパスワードの2つのパラメーターのみが変更されます。 それらに適切な名前を付けます：userNameとpassword。 そして、値として、現在のユーザーの特定の値を入力します。



それらを使用するには、ログインリクエストに移動し、特定の値の代わりに$ {variable name}の形式で変数名を書き込みます。 この例では、ログイン-$ {userName}、パスワード-$ {password}のValueフィールドに書き込みます。



パラメーターの受け渡しをデバッグするには、別の要素-Debug Samplerを追加します。 スレッド->追加->サンプラー->サンプラーのデバッグ 。 リクエストに応じてログインします。 要素の設定では、JMeter変数のみを残し、残りをfalseに設定します。 これは、変数のみをキャッチするためです。 そして、まだ他にはありません。



結果ツリーの表示で結果を確認します。 変数として渡されたのは値であり、クエリ全体が必要なものを返したことを確認する必要があります（上記を参照）。



これで、3人のユーザーにデータを転送できます。 これを行うには、CSV Data Set Configを使用します。



CSVデータセットの構成



メモ帳で.csvファイルを作成することをお勧めします。 ユーザーの場合、ファイルは「UserName; Password」という形式の3行のセットである必要があります。 私のファイルは以下のスクリーンショットに示されています：



次に、CSV Data Set Config要素（ Add-Config Elements-CSV Data Set Config ）をツリーに追加します。 コイルではなく、テスト計画のユーザー変数の下に追加します。 ファイルのファイル名として、ファイルへのフルパスとその名前を指定する必要があります。 以下に（選択した）エンコーディングを示します。 すべてのデータが英語の場合、フィールドは空白のままにできます。 以下に、ファイルから読み取られる変数を書き込みます。 私たちにとっては、ユーザー名とパスワードになります。 ファイルで使用されるため、区切り文字としてセミコロンを指定する必要があります。 以下では、少し英語を知っている人にとって、論理変数の名前を翻訳することは難しくありません。 コイルを停止するだけでなく、最後に到達したときにファイルの繰り返しをfalseに設定する必要があります。



次に、これらの変数をユーザー定義変数から削除します。 空のままになるという事実は怖くない。 その後、コイル自体で、スレッドの数 （ Number of Threrads ）が3に設定されます。

これで、スクリプトを実行し、デバッグサンプラーと結果ツリーの表示で結果を確認できます。 データがサーバーから返されたかどうか、ユーザーの名前が含まれているかどうかを忘れずに確認してください。





アルバムに写真を追加します



ユーザーがログインすることを学んだので、スクリプトの次の部分に進みましょう。 サイトの特定のアルバムに写真をアップロードすることを教えます。 これを行うには、録音コントローラーをコイルに追加し、「Photos」という名前を付けます。

結果のスクリプトを短縮するために、サイトのユーザーを事前に承認し、写真のアップロードを開始するページに移動します。 その後、ブラウザー設定でプロキシを再度オンにし、ワークベンチでProxy Server要素を使用します（ 別の記録コントローラーを指定します ）。 また、 除外するURLパターンでは、不要な拡張子を「*。\。Xxx」の形式で指定します。



開始、フォトアルバムに記入、終了。 スクリプトのクリーニングを開始します。 同様に、美容のために、「アルバムに行った-写真を追加した」という一連のリクエストのみを残すことができます。 次のようになりました。



実際、「アルバムに行った」というリクエストは手動で記録されました。 大したことではありません。これはページアドレスを使用したGETリクエストです。

ファイルをサーバーに送信するリクエストの本文に注意してください。 価値があるのは、パスとファイル名だけです。 ここでは、各ユーザーが自分の写真をアップロードできるように、それらをパラメーター化します。 ファイルへのパスから始めましょう。 テストのために、それは興味がありません。 すべての写真がフォルダにある可能性があります。 しかし、テストでは何度も発生する可能性があります。 したがって、適切な形式の規則に従って、変数に書き込みます。 ユーザー定義変数に戻り、 filePath変数に写真のあるフォルダーへのパスの値を追加します。たとえば、「** D：\ jakarta-jmeter-2.5.1 \ images \ **」です。 最後にスラッシュが付いていますが、後で明確になります。

いつものように、最初に1つのファイルでスクリプトをテストします。 そのため、別の変数fileNameを作成します。その値は、「 img.jpg 」などの特定のファイル名になります。



リクエスト本文に戻ります。 [リクエストでファイルを送信]セクションで、パスを次のように記述します：$ {filePath} $ {fileName}。 指定した値を変数（ユーザー定義変数からの値）に代入すると、正しいパスが取得されていることを簡単に確認できます。 リクエストが送信するパラメーターを見てみましょう。 ファイル名もあります。 代わりに$ {fileName}を入力してください。



この部分のユーザーシミュレーションが終了したので、新しいデバッグサンプラーを追加します。 さらに一歩進んで考えてみましょう。コイルを永久に回転させると（設定するまで）、各ユーザーは何度も何度もログインします。 これを回避するには、Only Onceロジックエレメント（ Thread-> Add-> Logic Controllers-> Only Once Controller ）をコイルに追加します。 サイト上のユーザーの承認を担当するレコーディングコントローラーの上の適切な場所に配置し、子としてコントローラーをドラッグします。



ブラウザでプロキシを無効にします。 スクリプトを実行します。 すべてが正しく完了すると、選択したアルバムに新しい写真が追加されます。 何か問題がある場合は、結果の表示ツリーのサンプラーのデバッグを参照してください。

アバターを変更する



実際、これは前の段落と変わりません。 同様に、サーバーに写真をアップロードする必要があります。 したがって、私たちもすべてを行います：

  1. 別の録音コントローラーを作成する
  2. プロキシを構成する
  3. サイトでアクションを実行する（スクリプトを作成する）
  4. スクリプトをきれいにする
  5. 1つのアバターのパラメーター化
  6. デバッグ
  7. 多くのアバターのパラメーター化


アバター名と同じcsvファイルを使用しましたが、写真を別のフォルダーにアップロードし、そのための新しい変数filePath_Avatarsを作成しました。

すべてがうまくいけば、それは良いことです。 そうでない場合は、Debug Samplerをご覧ください。

スクリプトを生きている人のように見せるために、遅延を追加する必要があります。 たとえば、サーバーに写真をアップロードした後、ユーザーが画像がサーバーにアップロードされるのを待っているかのように、一定のタイマーを3秒間（3000ms）置くことができます。 また、ガウスランダムタイマーのアバターを2秒から6秒に変更するには、ユーザーはアバターを検索して選択します。 また、各HTTPリクエストの信頼性を高めるために、[HTMLファイルからすべての埋め込みリソースを取得]ボックスをオンにすることができます。つまり、通常のブラウザーと同様に、ページから可能なすべてが読み込まれます。

したがって、すべてが正常に機能し、すべてが正常であるとします。 それだけです タスクは完全に完了しました。 これで、一晩中サイトを読み込むことができます。 Threadに戻って、 foreverチェックボックスを設定しましょう。 これは、「スレッドの停止」イベントが発生するまでスクリプトが実行されることを意味します。 CSV Data Set Configでは、このようなイベントの可能性をFalseに設定します。 これは、テスターが手で停止するまでテストが実行されることを意味します（メニュー実行->停止） 。 最終画面は、記事の冒頭に配置されています。

このテストは、いくつかのソーシャルメディアのありふれたビジネスプロセスをシミュレートします。ユーザーは昨晩のパーティーからオンラインに行き、アルバムに写真をアップロードしました。 このようなプロセスを永遠に続けて、写真をアップロードしてアバターを変更する3人のユーザーを獲得しました。 プロセスは、他のさまざまな要素を追加することで変更できます。 たとえば、Only Onceを設定することで1枚の写真のみをアップロードできますが、アバターを無限に変更できます。 最大1000人のユーザーでデータベースを補充する場合は、サーバーとオフィスのローカルマシンからスクリプトを直接実行します。これは、アプリケーションサーバーに非常に大きな負荷がかかります。
