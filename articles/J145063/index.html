Windows Azure Blob StorageとAmazon Simple Storage Service（S3）の比較–パートI

こんにちは、同僚の皆さん。

2つの有名なクラウドプラットフォームのストレージサービスの比較の次の部分の翻訳に精通することをお勧めします。 残りの翻訳は、処理時にレイアウトされます。

シリーズの最初の部分-Windows AzureテーブルストレージとAmazon DynamoDBの比較

この記事シリーズでは、 Windows Azure Blob Storage ServiceとAmazon Simple Storage Service（S3）のコア機能を比較します。 このパートでは、基本的な概念、価格設定、およびBLOBコンテナーとバスケットの機能の比較について説明します。 パートIIでは、ブロブとオブジェクトを比較します。 パートIIIでは、結論、パートIVでは、Windows Azure Blob StorageとGoogle Cloud Storageの比較を行います。



略語： Windows Azure Blob Storage-WABS、Amazon Simple Storage Service-AS 3。

概念的には、WABSとAS3は機能システムにおいて類似しています。 要するに、両方のシステムは、クラウド内のファイルシステムと見なすことができ、ユーザーは大量の非構造化データ（通常はファイルの形式）を保存します。

両方のシステムで、ゼロ個以上のブロブまたはオブジェクトを含むブロブまたはバスケットのコンテナを1つ以上作成できます。

両方のシステムは、キューとメッセージ、および通常はREST APIラッパーである他の高レベル言語のライブラリを操作するためのREST APIを提供します。 両方のシステムには一時的な命名規則があります-この記事の執筆時点では、WABS APIバージョンは2011-08-18で 、AS3は2006-03-01です。



一般に、両方のシステムは同様の機能を提供しますが、その一部を以下にリストします。

  • どちらのシステムも、本質的に2レベルの階層を持つクラウド内のファイルシステムです。
  • 両方のシステムは、大量のデータの信頼できる安価なストレージを提供します。
  • 両方のシステムは、コンテンツを不正アクセスから保護します。
  • どちらのシステムでも、1つのオブジェクトの多くのバージョンを保存できますが、バージョン管理メカニズムは各システムで独自の方法で機能します。
  • どちらのシステムでも、それぞれのデータ配信ネットワーク（CDN）を介してBLOBコンテナーとバスケットのコンテンツを公開し、待ち時間を減らしてコンテンツをキャッシュすることができます。
  • 両方のシステムは、データを保護するためのアクセス制御メカニズムを提供します。 AS3にはいくつかのオプション（ Amazon Identity and Access Management（IAM）、バケットポリシー、ACL、クエリ文字列認証 ）がありますが、WABSは署名ベースの保護（ ACLおよび共有アクセス署名）を提供します。


もちろん、違いがあります：

  • WABSはHTTP（RESTを使用）のみをサポートし、AS3はHTTP（REST、SOAP）、およびポイントツーポイント方式を使用してコンテンツを配信するためのBitTorrentをサポートします。
  • AS3には、非常に大きなボリュームをインポートおよびエクスポートするためのパブリックメカニズム（ Amazon Import / Export ）があります（Amazonディスクに大量のデータを提供し、ベンダー自体がデータをストレージにロードする場合）。 WABSでは、これはまだ利用できません。
  • AS3では、オブジェクトがいつ自己破壊するかを指定できます。 WABSでは、これはまだ利用できません。
  • AS3では、 Amazon DevPayを使用して、リソース消費に基づいて顧客に請求できます。 これは、SaaSアプリケーションを構築するための優れた機能であり、WABSではまだ利用できません。 WABSでは利用できないがAS3で利用できるもう1つの機能は、リクエスターペイバケットです。バスケットに保存されているデータにアクセスするユーザーは、データ消費に対して料金を支払います。
  • AS3では、サーバー側暗号化（SSE）を使用したデータ暗号化が可能です。 WABSでは、これはまだ利用できません（Trust Servicesはすでに利用可能です） 。
  • AS3は、仮想ホストスタイル（例：http : //mybucket.s3.amazon.com/myobject ）およびパススタイル（例： http ： //s3-eu-west-1.amazonaws.com/mybucket/ myobject ）、WABSはパススタイルのみをサポートします（例： http ://myaccount.blob.core.windows.net/myblobcontainer/myblob）。
  • AS3は冗長性削減ストレージ（RRS）を提供します-お客様は、AS3が提供する標準の冗長性（99.999999999％の耐久性と99.99％の可用性）よりも少ない冗長性（99.99％の耐久性と99.99％の可用性）でデータを保存して、コストを削減することを決定できます。 これは、データストレージコストがそれほど重要ではなく、簡単に再現できない場合にデータストレージコストを削減するのに便利だと思います。 WABSは、1つのレベルの冗長性を提供します。


コンセプトの



これらの2つのサービスについて詳しく説明する前に、いくつかの概念を明確にすることが重要だと思います。 WABSおよびAS3の基本概念に精通している場合は、このセクションをスキップできます。

BLOBコンテナーとバスケット ：これらのサービスがクラウド内のファイルシステムである場合、WABS BLOBコンテナーとAS3バケットをフォルダーまたはディレクトリと見なしてください。 WABSストレージアカウントまたはAS3アカウントでは、ブロブまたはオブジェクトをそれぞれ含むことができる0個以上のブロブコンテナーとバスケットを持つことができます。

コメント：

  • ネストされたblobコンテナやバスケットなどはありません。 どちらのサービスも、ネストなしで2レベルの階層を提供します。 ただし、どちらのシステムでも、プレフィックスを使用してフォルダー階層の錯覚を作成できます。
  • コンテナとバスケットの数に制限はありません。
  • どちらのシステムでも、静的なWebサイトをコンテナまたはバスケットに配置できますが、AS3では、ルートおよびドキュメントインデックスを決定し、4xxエラーページ（404エラーページなど）を編集することもできます。
  • 両方のシステムは、リソース要求をログに記録する機能を提供します。この機能は、ASでは「ロギング」、WABSでは「ストレージ分析」と呼ばれます。 違いは、AS3ではロギングがバスケットレベルで機能するのに対し、WABSではストレージアカウントレベルで機能することです。 AS3では、ロギングデータは個別のユーザー定義バスケットに配置されますが、WABSでは、ロギングが有効になったときに自動的に作成される定義済みのテーブルとコンテナーに配置されます。


BLOB とオブジェクト ：WABS BLOBとAS3オブジェクトは、BLOBコンテナーとバスケットにあるクラウドファイルシステム内のファイルです。

コメント：

  • 格納されるBLOBとオブジェクトの数に制限はありませんが、AS3ではこの数は単純に不明ですが、WABSではこの数はストレージアカウントのサイズ（100 Tb）によって制限されます。
  • AS3のオブジェクトの最大サイズは5 TB、WABSでは1 TBです。
  • WABSには2種類のBLOBがあります：ブロック（ストリーミング（写真、ビデオ、ドキュメントなど）に便利で最大サイズが200 GB）、およびページ（ランダムアクセス/記録操作に便利で最大サイズが1 TB）です。 ページBLOBを使用する一般的なケースは、Windows Azureの役割でVHDをディスクとしてマウントすることです。 AS3にはこのような分離はありません。
  • 両方のシステムは、豊富なブロブとオブジェクト管理機能を提供します。 他の操作をコピー、ダウンロード、ダウンロード、および実行できます。
  • 両方のシステムは、コンテンツを不正アクセスから保護する機能を提供し、アクセス制御リストメカニズムはAS3でより詳細に構成され、バスケット内の各ファイルに独自のACLを作成できます。 WABSでは、すべてがBLOBコンテナーレベルで発生します。


価格



両方のシステムを使用する場合、「資本」コストはありません。 価格設定モデルは比較的単純で、消費に基づいています。 どちらのシステムでも、請求書は使用量に基づいて請求され、3つのコンポーネントで構成されます。

  1. トランザクションの数 ：支払は、行われたトランザクションの数に従って行われます-大まかに言って、1つのトランザクションはシステム内の1つの関数呼び出しです。 2つのシステムには大きな違いがあります-WABSでは、トランザクションコストは固定（10,000トランザクションあたり0.01ドル）です。AS3では、トランザクションのタイプによって異なります。 PUT、COPY、POST、LIST操作を実行する場合、トランザクションに対してより高い価格（米国標準地域で1000トランザクションに対して0.01ドル）を支払い、GETなどではより低い価格（米国標準地域で10,000トランザクションに対して0.01ドル）を支払います。 AS3のすべての削除リクエストは無料です。
  2. ストレージ ：各システムに保存されているデータの量に対して支払います。 AS3では、価格は冗長性のレベルに依存します。
  3. トラフィック ：システムとの間で転送されるデータの量に対して支払います。 執筆時点では、両方のシステムが無料の着信トラフィックを提供しています。 同じリージョン内のADDBとAmazon EC2の間で転送されるデータは無料です。 異なる地域のADDBとAmazon EC2の間で転送されるデータは、関税に従って課金されます。 WATSでは、発信トラフィックのみが支払われます。


特別な価格設定モデルも利用でき、両方のシステムが異なる支払いパッケージを提供します。 価格設定の詳細-https : //www.windowsazure.com/ru-ru/pricing/details/はWABS、 http： //aws.amazon.com/s3/pricing/はAS3の場合。

AS3

コンテナの削除/バケットの削除

はい

はい



この関数は、ブロブコンテナーまたはバスケットを削除します。

コメント：

  • この操作は同期のように見えるかもしれませんが、実際にはそうではありません。 BLOBコンテナーを削除する要求を送信すると、削除のマークが付けられ、アクセスできなくなります。その後、ガベージコレクションプロセス中に削除されるため、コンテナーを削除する実際の時間は、このコンテナー内のデータのサイズによって異なります。 私の経験では、非常に大きなコンテナの削除には数時間かかる場合があり、この時点で同じ名前のコンテナを作成しようとするとエラーが発生します（競合エラー-HTTP 409）。 この点で、この時点で何をすべきかを計画する必要があります。
  • AS 3では、バケットを削除する前に空にする必要があります。 まず、バスケットからすべてのオブジェクトを削除してから、それを削除する必要があります。




綿棒

AS3

リストBLOB / GETバケット（リストオブジェクト）

はい

はい



この関数は、コンテナまたはバスケット内のブロブとオブジェクトのリストを取得するために使用されます。 システム内の関数は、次の場合に同じことを行います。

  • 両方の機能により、結果の選択を目的のオブジェクト数に制限できます。
  • 両方の関数には、1回の関数呼び出しで返すことができるオブジェクトの最大数があります。WABSではこれは5000、AS3では1000です。
  • どちらの関数もセパレーターをサポートしています。 セパレーターは、ブロブまたはオブジェクトをグループ化するシンボルです。 最も一般的に使用される区切り文字は/です。 前述のように、両方のシステムは2レベルの階層をサポートしており、セパレーターを使用すると、フォルダータイプ階層の錯覚を作成できます。 たとえば、次のオブジェクトがあります：images / a.png、images / b.png、images / c.png、logs / 1.txt、logs / 2.txt、files.txt。 関数を呼び出して区切り文字/を渡す場合、両方のシステムが次の値を返します：images、logs、files.txt。
  • どちらの機能も、プレフィックスを使用したサーバー側のフィルタリングをサポートしています。 リクエストにプレフィックスが含まれる場合、両方のシステムは、このプレフィックスで始まる名前を持つオブジェクトを返します。 上記の例を使用して、セパレータなしで接頭辞「images」を渡すと、両方のシステムは次の値を返します：images / a.png、images / b.png、images / c.png。
  • 両方の関数は、実際には継続トークンであるマーカーを使用でき、このマーカーで始まるオブジェクトのリストの受信を開始する必要があることを両方のシステムに示すために使用されます。
  • 両方のシステムは、アルファベット順にオブジェクトを返します。


違い：



  • WABSの1つの関数呼び出しは、最大5000のブロブ、AS3-1000オブジェクトを返します。
  • リストを受け取ったら、ブロブのスナップショットを返す必要があることをWABSに示すことができます。 AS3には、このための個別の機能があります（ バケットオブジェクトの取得バージョン ）。
  • リストを受け取ったら、ブロブに対してメタデータを返す必要があることをWABSに示すことができます。 AS3では、ブロブのメタデータはサポートされていません。
  • リストを受け取ったら、まだ確認されていない（コミットされていない）BLOBのリストを返す必要があることをWABSに示すことができます。 部分的にロードされると、AS3はすでに完全にロードされているオブジェクトのみを返すことができます。

AS3

BLOBサービスプロパティの設定/ PUTバケットロギング

はい

はい





どちらのシステムにもクエリログ機能があり、この機能はデフォルトでは有効になっていません。 WABSでは、この関数はStorage Analyticsと呼ばれ、すべてのWindows Azureストレージサービス（テーブル、キュー、BLOB）に使用されます。



ロギングの違い：



  • WABSでは、ログはブロブストレージレベル、AS3ではバスケットレベルで有効になります。 つまり、WABSでログを有効にすると、ストレージアカウントのすべてのBLOBコンテナーで有効になりますが、AS3でログを有効にすると、特定のバスケットでのみ有効になります。 この場合のAS3は、特定のオブジェクトのログを有効にできる「低レベル」機能を提供し、さらにプレフィックスを指定できます。ログは、このプレフィックスで始まる名前のオブジェクトに対してのみ有効になります。 たとえば、データと呼ばれるバスケットがあり、画像とエラーオブジェクトが含まれ、サイトイメージタイプの別のオブジェクトを追加する場合、サイトイメージタイプのオブジェクトに対してのみサイトイメージプレフィックスを使用してロギングを有効にできます。
  • WABSでは、ログはテーブルストレージ（$ MetricsTransactionsBlob）およびblobコンテナー（$ログ）のシステム定義テーブルに保存されますが、AS3では、ログはユーザー定義のバスケットに保存されます。
  • WABSでは、最大20 TBのログを保存できます。ログは最大365日間保存でき、その後削除されます（この時間は設定可能です）。 AS3には制限がありません。
  • WABSでは、アカウント所有者のみがこのデータにアクセスできますが、AS3では、このデータへのアクセスとユーザー権限（フル/読み取り/書き込み）を制御するアクセスメカニズムを実装できます。

この関数は、コンテナまたはバスケットのACLを指定するために使用され、WABSで1つ以上のアクセスポリシーを指定することもできます。

BLOBコンテナーの場合、ACL値は次のようになります。

  • 完全なパブリック読み取りアクセス（コンテナ） ：コンテナとBLOBデータは匿名で読み取ることができます。 クライアントは、匿名リクエストでコンテナー内のBLOBのリストを受け取ることができますが、ボールトアカウント内のコンテナーのリストを受け取ることはできません。
  • BLOBのみのパブリック読み取りアクセス：このアクセス権を持つコンテナー内のBLOBデータは匿名要求で読み取ることができますが、コンテナーデータは使用できなくなりました。 クライアントは、匿名リクエストでコンテナ内のブロブのリストを受け取ることはできません。
  • パブリック読み取りアクセスなし（個人） ：コンテナーとBLOBデータは、アカウント所有者のみが取得できます。


バスケットの場合、ACLは次のものと等しくなります。



  • 読み取り ：バスケット内のオブジェクトのリストを取得できます
  • WRITE ：ごみ箱内のオブジェクトを作成、上書き、削除できます。
  • READ _ ACP ：バスケットのACLの受信が許可されます。
  • WRITE _ ACP ：ACLへの書き込みリサイクルが許可されます。
  • FULL _ CONTROL ：READ、WRITE、READ_ACP、WRITE_ACPパーミッションはこの値で付与されます。




AS3で便利なのは、ユーザーに異なるアクセス許可セットを与えることができることです。たとえば、user1にはREAD ACLを、user2にはWRITE ACLを、WABSにはそのような柔軟性はなく、アクセス許可はblobコンテナーにのみ設定されます。



WABSで便利なのは、ACLに加えて、このコンテナのアクセス許可の一時セットを定義するコンテナに対して最大5つのアクセスポリシーを設定できることです。 たとえば、ブロブコンテナに対する書き込み権限を持つアクセスポリシーを作成できます。これは1日のみ有効です。 ポリシーを使用すると、署名付きの特別なURLを生成してユーザーに提供できます（柔軟な共有アクセス署名機能）。 署名を使用すると、一定の期間、より詳細なレベルでコンテナおよびBLOBにアクセス権を発行できます。

バケットのバージョン管理を配置する

バージョニングにより、オブジェクトの複数のコピーを管理できます。 この関数を使用すると、バスケット内のオブジェクトのバージョン管理を有効にできます。PUT、POST、COPY、DELETEのいずれかの操作が実行されると、オブジェクトの新しいバージョンが作成されます。 デフォルトでは、この機能はオフになっています。



WABSは、同様の「 スナップショットブロブ 」機能をサポートしています。この機能を呼び出すと、現在のバージョンのブロブの読み取り専用コピーが作成されます。



2つのシステムのバージョン管理の違い：



  • AS3では、バージョン管理はバスケットレベルで指定され、バージョン管理を有効にすると、そのバスケット内のすべてのオブジェクトに適用されます。 WABSでは、スナップショットブロブの機能はブロブレベルで動作し、バージョン管理を有効にする必要があるブロブで明示的に呼び出す必要があります。 この意味で、WABSはより柔軟なバージョン管理システムを提供しますが、これにより開発者はBLOBスナップショットを作成する責任を負います。
  • WABSでは、ブロブが削除されると、そのスナップショットがすべて削除されます。 AS3では、そうではありません。
  • WABSでは、元のBLOBのブロックとは異なるスナップショットブロックに対してのみ料金を支払います。 AS3では、オブジェクトの各バージョンはオブジェクトと見なされ、それらのそれぞれに対して支払いが行われます。

まとめ



概念的には、どちらのサービスも同様の機能を提供しますが、Amazon S3はより機能が豊富です。これは、Amazonがシステムを開発するのに2年以上かかったためです。 Windows Azure Blob StorageがS3に追いつき、不足している多くの必要な機能を実装することを本当に願っています。 個人的には、次のようなS3機能が大好きです。

  • 静的なWebサイトをホストする機能。
  • 柔軟なアクセス制御。
  • コンテンツの自動削除。
  • より「リッチな」バージョン管理-ソースが削除されても、そのコピーは残ります。
  • 冗長ストレージの削減のようなもの。




ただし、次のような機能があれば素晴らしいと思います。



  • 保管されているオブジェクトを削除せずにバスケットを削除する機能。 これは大きなオーバーヘッドを課すと思います。
  • いくつかの機能を組み合わせます。 たとえば、バケットの取得、バケットのバージョン管理の取得、およびマルチパートアップロードのリストは、簡単に1つの関数に結合できます。
