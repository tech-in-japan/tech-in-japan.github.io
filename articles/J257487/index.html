Junos Fusionとは何ですか？

今日、CIOにデータセンターの主な問題について尋ねると、おそらく次の回答からいくつかのバリエーションが得られます。

「既存のインフラストラクチャを複雑にすることなく、データセンターを拡張できるようにしたいと考えています。」

-全体的なコスト、運用コストを削減する必要があります。 したがって、大規模なデータセンターの運用を簡素化し、コストを削減するものはすべて歓迎します。

「近い将来、専門のクラウドインフラストラクチャとプライベートクラウドインフラストラクチャの組み合わせが期待されます。」 したがって、同じビルディングブロックを使用して環境を構築し、あるタイプのアーキテクチャから別のタイプに移行する柔軟性を提供できるソリューションが必要です。

-提案されたアーキテクチャはすべて、クラウドアプリケーションだけでなくデータセンター内のユビキタスL2通信を必要とする既存のアプリケーションをサポートする必要があります。

-データセンターのあらゆる部分をダウンロードできる必要があります。 ワークロードモビリティには、追加のネットワーク設定は必要なく、完全に自動化する必要があります。

「ソリューションは、複数のモジュールを簡単に接続でき、これらのモジュール間のワークロード分散も利用できるスケーラビリティのシンプルさを提供する必要があります。」



ジュニパーネットワークスは最近、これらの各問題に対処し、グローバルにそれを行うデータセンターのアーキテクチャを導入しました。 Junos Fusionと呼ばれるこのソリューションは、大規模な運用を簡素化するだけでなく、大規模なデータセンターのプロビジョニングも簡単な運用にします。 猫の詳細の下。



一般的なビュー

新しい開発では、データセンターのアクセスレベルと集約レベルを組み合わせて、クラスター内の1つまたは2つのデバイスから大規模なデータセンターを管理するためのシンプルな集中型インターフェイスを提供します。 これにより、リソースの割り当て、データセンターの構成と運用が簡素化されます。



Junos Fusionは、さまざまな標準ベースのテクノロジーを利用してスケーラビリティを提供します。 たとえば、IEEE 802.1BRプロトコルを使用すると、データセンターのトップオブラックスイッチ（「コンパニオンデバイス」と呼ばれる用語）が、1つまたは2つのアグリゲーションスイッチ（「アグリゲーションデバイス」）のリモートラインカードとして機能できます。 Junos Fusionでは、最大64個の集中管理された関連デバイスを使用できます。 これにより、データセンターの運用が大幅に簡素化され、アクセスレベルで個々のスイッチをローカルで管理する必要がなくなります。 さらに、Junos Fusionは疎結合システムとして設計されており、さまざまなコンポーネントが相互に最小限に依存し、ルーティングでネットワークを構築するためのプロトコルに類似したプロトコルを使用して通信します。 これにより、システムのシームレスなスケーリングが可能になります。



インテリジェントなTop-Of-Rackスイッチを個別のラインカードとして使用すると、ローカルL2およびL3トラフィックスイッチングなどの追加サービスを実装できます。これらはラック内で切り替える必要があります。 関連デバイスは、集約デバイスで実行されているJunosオペレーティングシステムとは別の、Linux Forwarding Operating System（LFOS）と呼ばれるWindriver Linuxディストリビューションで実行されます。 Junosは、ラック間およびデータセンター外のすべてのトラフィックに対して豊富な機能セットを提供します。



IEEE 802.1BRは、中央デバイスからリモートラインカードを管理する方法を定義する単純なプロトコルであることを理解することが重要です。 関連するデバイスのソフトウェアの更新、FRU（電源および冷却ユニット）の監視、イベントの監視など、データセンターの機能に不可欠なすべての機能を提供するわけではありません。 この機能はすべて、集約デバイスと関連デバイス間でJSON-RPC APIを使用して実装できます。 新しいアーキテクチャでは、Junos Space Network Director GUIのおかげで、これらの機能はすべてお客様がすでに利用できます。







Juniper MXシリーズルーターまたはQFX10000スイッチはJunos Fusionの集約デバイスとして機能し、EX4300またはQFX5100スイッチは補助デバイスとして使用でき、ユーザーは1 GbEから40 GbEのインターフェースを使用して柔軟なアクセスオプションを提供します。

これにより、あらゆるサーバー、ストレージ、またはサービスをJunos Fusionに接続できます。



主な機能

Junos Fusionアーキテクチャは非常に回復力があります。 集約デバイスは、アクティブレベルの制御を備えたクラスター構成で動作し、高いデータセンターの復元力を提供します。 異なるバージョンのJunosに基づいて動作し、別のレベルの独立性と管理レベルの分離を作成できます。



上記のように、1つの中央デバイスまたは2つのデバイスのクラスターから最大64個の補助デバイスを管理できます。SNMP、ログ収集、FRU監視、および構成は、それを使用して実行されます。 さらに、集約デバイス間で設定を複製できるため、デバイスに障害が発生した場合に最も関連性の高い構成を使用できます。 設定は、Junos CLIまたはJunos Space Network Directorを使用して実行されます。



Junos Fusionでは、集約デバイスはJunosで実行され、関連デバイスはWindriver LinuxベースのLinux Forwarding Operating System（LFOS）で実行されます。 このシステムにより、関連するデバイス上のソフトウェアを他のデバイスから完全に独立させることができます。 ソフトウェアバージョンを更新するために、オペレーターはいわゆる更新グループを整理し、特定のデバイスを組み合わせて使用​​できます。 すべてのサーバーとストレージシステムは、関連するデバイスのペアにデュアル接続できるため、メンテナンス中であってもダウンタイムを最小限に抑えることができます。 実際、各デバイスでのソフトウェアの更新には約30〜45秒かかります。 したがって、すべてのデータセンターを3,000ポートで更新するのに1時間もかかりません。







従来のTop-Of-Rackスイッチは付随デバイス（リモートラインカード）として使用できるため、単純なマルチプレクサでは不可能な多くの追加サービスを使用することが可能になります。 それらの1つは、関連デバイスのローカルスイッチングです。 インテリジェント関連デバイスは、帯域幅の使用を最適化するために、分散マルチキャスト、ストリームベースのアップリンク選択、ローカル監視などの機能を提供します。



集約デバイスと関連デバイスのソフトウェアは実際には相互接続されていないため、新しいモジュールを追加するときにソフトウェアを更新する必要はありません。 関連するデバイスをシステムに接続する場合、他の関連するデバイスまたはアグリゲーションデバイスのソフトウェアに影響を与えることなく、LFOSをシステムに入れるだけで十分です。



また、Junos FusionがQFX10000スイッチとMXシリーズマルチサービスルーターを集約デバイスおよび拡張ポートとして使用できることも注目に値します。 これにより、EVPNやVxLANなどのデータセンター向けの重要な機能を取得できます。







データセンターの利点

Junos Fusionアーキテクチャにより、既存のインフラストラクチャを複雑にすることなく、データセンターを簡単に拡張できます。 そのため、100個のポートを備えたデータセンターは、簡単に3000個まで拡張できます。新しいラックは、関連付けられたデバイスのローカル構成を必要とせずに、集約デバイスに追加ポートとして追加するだけです。



Junos Fusionは、データセンターリソースの提供だけでなく、ソフトウェアの管理、監視、更新も簡素化します。 これにより、ITインフラストラクチャの資本コストと運用コストを削減できます。 「自動VLANセンス」機能を使用すると、ユーザーグループのすべてのセグメントを事前に構成できます。 このシステムはシームレスなモビリティを提供し、すべての手動操作を高速化します。 Junos Space Network DirectorとVMware vCenterを統合すると、新しいサーバーと仮想マシンをネットワークに追加する一般的な操作タスクが簡素化されます。 ワークロードモビリティは完全に調整されており、ユーザーが手動で構成する必要はありません。 これにより作業が簡素化されるだけでなく、オペレーターエラーの可能性が排除されます。



新しいアーキテクチャは、EX4300、QFX5100、QFX 10000スイッチ、およびMXシリーズルーターの組み合わせを使用します。 顧客がJunos Fusionを使用して、または水平方向にスケーラブルなL3構成でデータセンターを構築する場合、すべてのブロックは非常に柔軟であり、そのセットをさまざまなデータセンターアーキテクチャに使用できます。
