エントリー



レビューの4番目の部分は、 Fabricサービス専用です（ レビューの最初の部分、レビューの2番目の部分、レビューの3番目の部分 ）。



2014年の秋以来、かなりよく知られているCrashlyticsサービスはFabricの一部であり、特殊なSDKのセット（または公式Webサイトが言うように、「モジュール式のクロスプラットフォームモバイル開発スイート」）であり、それぞれ個別にまたは共同で統合できます。 レビューでは、モバイルアプリケーションの配布を担当するサービスとSDKであるCrashlytics Kitのみに焦点を当てます。



評価システム：各セクション（登録と統合、基本機能、追加機能、継続的統合）ごとに10ポイントのスケールでサービスが評価されます。 合計スコアが勝者を決定します。



布地



登録と統合





サービスへの登録は、アプリケーションの原則に基づいています-新しいアカウントを作成したい人は誰でもリクエストを残すことができます。これは、私たちの経験に基づいて、できるだけ早く検討されます（それほど前ではなく、従業員の申請は1時間以内に承認されました）。 SDK Fabricの統合が迅速かつ簡単であると言うのは不要です。 このような便利なステップバイステップの統合ウィザードを提供する配信サービスは他にありません。 自分でこのプロセスを実行することをお勧めします。これは、通常の日常業務が楽しく、興味を持って実行されるまれなケースです。





























評価：9/10。



主な機能



注（2015年6月15日に更新）：iOS向けアプリケーションの配布について話すときは、現時点では、AppleのTestFlightを除くすべてのサービスが、AdHocプロビジョニングプロファイルを使用するときに利用可能なデバイスの数を引き続き考慮しなければならないことに注意してください したがって、使用されるデバイスの最大数は、デバイスファミリごとに100です（この条件は、すべての開発者プログラムが1つに統合されたというWWDCでの発表後、2015年6月8日に変更されました）。 最大の制限は、登録済みデバイスのリストは、iOS開発者プログラムのサブスクリプションを更新した後、年に1回のみ「リセット」できることです（将来、この条件は変更される可能性があります）。



主な機能に関して、Fabricは、考えられる他のすべてのサービスと概念的に異なります。 そのため、たとえば、新しいプロジェクト（Fabric用語では「アプリケーション」）の追加は、サービスWebサイトではなくデスクトップクライアントで実行されます。











クライアントアプリケーションとサービスWebサイトの両方で、新しいユーザー/テスターを追加できます。 最初のケースでは、グループにテスターを含めずにテスターを追加することができます。











2番目の場合、既存のグループを選択するか、新しいグループを作成する必要があります。ここで、すべての新しいチームメンバーが追加されます。







注：このサービスは、TestFlightの古いバージョンからテスターのリストをインポートする機能を提供します： dev.twitter.com/crashlytics/beta-distribution/testflight-migration 。 ただし、2015年3月からTestFlightの古いバージョンは使用できなくなったため、リストを事前にエクスポートしておくと、この機能が役立ちます。



CSVファイルからテスターのリストをインポートすることができます。形式は次のとおりです。 

First name,Last name,E-mail




次の手順は、招待状、デバイス登録、テストアプリケーションのインストールなど、他のサービスの同様の手順と一致します。











アプリケーションの以前のバージョンのインストールは非常に簡単です。最新バージョンをインストールするボタンの下には、特定のユーザーが使用できるすべてのバージョンのリストがあります。











評価：9/10。



追加機能



レビューの前の部分で、エラーの開発と修正のプロセスを高速化するのに役立つ追加機能を実装するためのいくつかのオプションを既に検討しました。



Fabricが提供するものを見てみましょう。



1つは、クラッシュレポートのシンボル表示です。 これは、サービスの元の名前であるCrashlyticsからも明らかです。



SDK開発者は、レポートの収集と処理が時計仕掛けのように機能し、最大限の効率で動作するように取り組んでいます。 わざわざ言い直すことなく、私は単に公式ウェブサイトからリストを引用します。



  • 防御的レポート：クラッシュしたプロセスは、カーネルがそれらを終了するアクションをとる前に、しばしば大きな損害を被ることになります。 これにより、二次クラッシュが発生する可能性があり、クラッシュ処理コード自体が正しく動作できず、失敗します。 キャッシュファイルの解析が予期せずクラッシュしないように、ファイル処理コードを非常に防御的なものにすることに投資しました。
  • スタックの巻き戻し：クラッシュ検出の最も難解な側面の1つは、スタックの巻き戻しです。これは、クラッシュに直接つながる履歴コードの実行を決定する一見無知な機能です。 SDKは、objc_msgSendおよびAppleのiOS ABIに準拠する他の多くのメソッドの場合に機能する呼び出し命令を決定します。
  • Mach Exceptions：iOSおよびOS Xで利用可能な最低レベルのシステムを使用してクラッシュをキャプチャします。MachException APIを使用すると、他のメカニズムの不確定な動作の一部なしに、すべてのクラッシュを一貫してキャプチャできます。 これにより、開発者がXcodeで表示するデータにはるかに近いデータも得られます。
  • キャッチされない例外レポート：アプリがキャッチされない例外をスローすると、Crashlyticsが復活します。 Crashlyticsは、例外がスローされた時点でデバイスのスタックトレースと状態をすばやく記録し、クラッシュ情報をサーバーに送信して処理します。 すべてのクラッシュを簡単に可視化できます。
  • キャッチされた例外レポート：Crashlyticsはキャッチされていない例外のためだけではありません！ catchブロックに1行のコードを追加すると、キャッチされた例外がCrashlyticsに処理のために報告されますが、例外を自由に処理できますが、ユーザーにとっては最適です。 致命的なクラッシュについて収集した完全なスタックトレースとすべてのデバイス情報を取得できるため、アプリがクラッシュしなくてもバグを特定して修正できます。
  • 完全に自動化された難読化解除：antツール、mavenツール、またはIDEプラグインを使用して構築されたアプリのProGuardedスタックトレースの難読化解除を自動化します。 私たちはあなたのために足掛かりをしました。 既存のワークフローに組み込まれたSDKは、アプリにほとんど影響を与えません。


詳細情報は、 こちらおよびSDKの3番目のバージョンのプレスリリースから入手できます 。



デバイスからFabricサーバーにレポートを送信した後（「クラッシュした」アプリケーションを再起動した後に発生します）、サービスは電子メールで通知を送信し、クライアントがインストールされているコンピューターにアラートを表示します。







欠点として、2番目のケースでは、（HockeyAppクライアントで行われるように）目的のプロジェクト、クラス、および選択したコード行でXcodeを起動する可能性がないことに注意できます。



トラッカーや他のサービスとの統合を設定することもできます。その後、そこに通知が送信されます。 大きなプラスは、複数のサービスを一度に接続できることです。 欠点は、この機能に関連する背景情報が不足していることです。











注：BitbucketおよびHipchatとの統合は、テストプロジェクトで構成されました。 その結果、新しいクラッシュに関するメッセージはHipchatでのみ受信され、新しいチケットはBitbucketで作成されませんでした。







Fabricは、次の3つの追加ツールを使用して、クラッシュレポート関連情報の有用性を高める機会を提供します。



カスタムキー：フレームワークは、後続の分析に重要な変数の値を保存するための特別なメソッドを提供します。 



 + (void)setObjectValue:(id)value forKey:(NSString *)key; + (void)setIntValue:(int)value forKey:(NSString *)key; + (void)setBoolValue:(BOOL)value forKey:(NSString *)key; + (void)setFloatValue:(float)value forKey:(NSString *)key;




注：変数の最大数は64です。詳細については、 こちらをご覧ください 。



カスタムロギング： 「CLS_LOG」マクロを使用すると、デバイスのログにサービスメッセージを書き込むことができますが、標準のNSLogとは異なり、このマクロを使用すると、ログがクラッシュレポートに添付されます（ 詳細 ）。



注：アプリケーションが起動された時点から、ログ全体がレポートに保存されます。 正規表現を使用してログを検索することが可能です。



ユーザー情報：アプリケーションのユーザーを識別することもできます（識別子、名前、および電子メールアドレス-more ）。 使用するときは、Appleのプライバシーポリシーを考慮する必要があります。

継続的インテグレーション



注：このレビューの継続的インテグレーションの一部として以下に説明するサービスを使用することのニュアンスは、アップルが提供するソリューションに基づいていることを意味します。 つまり、Mac OS X ServerとXcodeボットの助けを借りて、このような統合の構成について詳しくは説明しません。おそらくこれは別の記事のトピックになるでしょう。 希望する人は、たとえばAppleの公式ガイドを参照して、自分でトピックに慣れることができます。



アプリケーション配布ダウンロードAPIを使用するには、API_KEYおよびBUILD_SECREを取得する必要があります。 これらは、「設定->組織->特定の組織の詳細情報を含むページ->組織名の下の非表示フィールド」ページで見つけることができます。











ダウンロードは、フレームワークに含まれている特別なユーティリティを使用して実行されます。使用例は次のとおりです。 



 /path/to/Crashlytics.framework/submit <API_KEY> <BUILD_SECRET> -ipaPath /path/to/my.ipa -emails TestEmail@crashlytics.com,AmazingTester@twitter.com -notesPath ~/Notes/ReleaseNotes.txt -groupAliases GroupAlias,GroupAlias2




注：emails、notesPath、およびgroupAliasesはオプションのパラメーターです。



オプションの完全なリスト： 



 ﻿ -emails list of individual addresses (comma-separated)﻿-groupAliases list of group aliases from web dashboard (comma-separated) -notesPath absolute path to .txt file -notifications enables/disables email notification to testers (YES|NO)

おわりに



レビューでは、iOS用のモバイルアプリケーションを配布するための4つのサービスの長所と短所を検討し、それらを評価しようとしました。 最初は勝者が明らかだったように思えましたが、最終的には、より慎重に研究した結果、すべてがそれほど明確ではありません。



総合評価の正式な基準に従って、サービスは次のように配布されました。



  1. ファブリック-34ポイント。
  2. HockeyApp-33ポイント。
  3. Ubertesters-32ポイント。
  4. テストフライト-15ポイント。




そもそもファブリックは、このサービスの最大の利点は、クラッシュレポートの収集と処理が無料で高効率であることです。 欠点の中でも、開発者に対する一定の方向性*と、フィードバックを収集するための組み込みシステムの欠如があります。



*ここで私はサービスの狭い専門化を意味します：チームメンバーが開発者でない場合、アプリケーションでは他の追加機能は利用できません（仕様に従ってアプリケーションに必要なものを除く）。 また、アプリケーションがクラッシュせずに機能する場合、フィードバックを提供するには、他のサービスを使用する必要があります。



HockeyAppとUbertestersの2位と3位-評価の違いはわずか1ポイントです。



HockeyAppの利点は、 クラッシュレポートのシンボル化のサポートと、他のサービス（アプリケーションをクラッシュさせる原因となった行でプロジェクト/クラスを開く）と比較したXcodeとの密接な統合です。 欠点の中で、通常の無料料金プランの欠如とレビューを収集するためのシステムの限られた可用性に注意することができます。



現時点でのUbertestersは、将来の展望、サービスの観点から最も興味深いように見えます。 フィードバックを収集するための十分に開発されたシステムとテスターの行動を追跡するためのシステムにより、特定の範囲の問題を最大限の効率で解決できます。 現在、クラッシュレポートのシンボル表示に対する組み込みのサポートの欠如は欠点ですが、今後の開発で解消されると思います。 すべての基本機能にアクセスできる無料の料金プランがあることも、このサービスの大きな利点です。



最後にあるTestFlightは部外者のように見えますが、これは一時的な状況であり、Appleのサービスとエコ開発システム全体のさらなる発展により、現在のリーダーを圧迫する可能性があります。WWDC2015からの最新ニュースは楽観的です。



一連のレビューが有用であり、アプリケーションの配布に適したサービスを選択できる人に役立つことを願っています。 追加の質問にお答えできることを嬉しく思います。
