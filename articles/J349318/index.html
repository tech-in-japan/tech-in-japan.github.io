第4レベルに進みます。 これはトランスポート層です。 そのため、転送する必要があるファイルがあります。 同じsales.pptファイルにします。 これはPowerPointプレゼンテーションファイルです。 シスコでは、PowerPointを使用できることを非常に誇りに思っています。 そして、私たちだけでなく、多くの人がそれを使用したいと思っています。







これが75枚のスライドの30時間のPowerPointプレゼンテーションであると想像してみましょう。そうです、これはシスコです:)そして、ファイルサイズが約27メガバイトであるとします。 このサイズのファイルをネットワーク経由で送信することはよくあります。 また、ネットワークを介して、大きなファイルは一度に送信されることはなく、小さなファイルで送信されます。



それは本のようなものです。 ファイルを本と比較しましょう。 この本をネットワーク経由で送信したい場合、ページが非常に多いため、全体を送信できませんでした。 それなら私は何をしなければなりませんか？ チャプターとページに分割するか、または何らかの方法でさらに多くの数に分割し、その後、それらをはるかに簡単に送信できます。 受信するページ数を決定し、一度に1枚ずつ引き裂いて個別に送信する必要があります。つまり、ファイルをミニドキュメントに分割します。 これは、データセグメンテーションと呼ばれます。



したがって、トランスポートレベルでは、ファイルを送信すると、ネットワークを介したデータセグメンテーションが開始され、サーバーに到達します。 sales.pptファイルをいくつかのセグメントに分割する必要があります。 しかし、このファイルのいくつの章、いくつのセグメントを送信する必要がありますか？







たとえば、10個のチャプターを1つのファイルセグメントとして、または20個のチャプターとして送信します。 これを行うには、プロセスを構成する必要があります。 あるクライアントから別のクライアントにファイルを転送する必要があります。 これを行うには、それらの間にセッションを確立し、一方から他方への論理通信トンネルを作成する必要があります。 そして、次のことが起こります。 ファイル転送プロトコル（FTP）を使用してこのファイルを要求するユーザーは、受信するファイルが75の部分に分割されることを報告します。 転送する必要があるセグメントが75個あることがわかりました。 そして、最初に「ねえ、私はあなたに送ったこのファイル用に75個のセグメントがあります」と言ったら、彼はこのファイルを部分に分割したことがわかります。 したがって、トンネルの反対側には75の異なるセグメントが予想されます。



しかし、あなたと私は、すべてのセグメントがサーバーに到達することを確認する必要があります。 アプリケーションによっては、たとえばファイル転送アプリケーションの場合、FTPプロトコルを使用できます。これにより、サーバーへのデータ転送が完了し、ネットワーク経由でファイルを送信するための信頼できるサービスになります。



そのため、私のアプリケーションは信頼できるFTPを使用することにしました。 しかし、その「信頼性」とは何であり、どのように保証されていますか？ そして、それはそのように提供されます。 ファイルを部分に分割し、最初のセグメントを送信します。 サーバーが反対側でこのセグメントを受信すると、「ほら、最初のセグメントを受信しました。」 そして彼は答えます：「あなたは私に一秒を送ることができます。」 などなど。 彼は、前のセグメントをすでに受け取っていることを知っているので、2番目のセグメントを受け取り、3番目のセグメントを要求します。 データアプリケーションは、特にこれを活用するように設計されています。



私たちは皆、TCP / IPなどの用語を知っており、聞いています。 さて、TCPという用語はどういう意味ですか？ TCPは、データストリームに予備接続を提供し、データ損失の場合にデータを再要求し、1つのパケットの2つのコピーを受信したときに重複を排除するデータ転送制御プロトコルです。これにより、送信データの整合性を保証し、転送結果を送信者に通知します。 たとえば、3番目のパケットが失敗し、サーバーが3番目のパケットを受信しません。 彼はプロトコルに従って待機し、しばらく待ってから「3番目のパケットをもう一度送信してください」と尋ねます。 そして、再送信が発生しない場合、そのようなリクエストがいくつかある場合、「リクエストタイムアウト」を取得します。







そして、これはあなたのアプリケーションに何を与えますか？ 信頼性。ファイルが正常に転送されたことを保証します。 したがって、データアプリケーションは常にTCPを最初に使用する必要があります。 このプロトコルによって、ネットワーク全体が正確に「吸い込まれ」ます。 はい、これは専門用語であり、冗談ではありません。 さまざまなレベルのエンジニアから何度も聞いています。 彼のおかげで、データベースが整然と並んでいます。



まあ、それはいくらですか？ 人生には自由なものは何もありません。 おそらく私の犬の愛、Doc。 彼は私の気分に関係なく私と一緒に踊るのが大好きです。 これはおそらく、人生で唯一無料のものです。 別の犬-私の妻、ベラの犬は私をまったく愛していません。 しかし、ここの価格はいくらですか？ サーバーが世界の別の場所にある場合でも、サーバーが応答するのを待つ必要があるため、時間とトラフィックがかかります。



TCPフローティングウィンドウ機能のおかげで、このメカニズムを少し回避し、このプロセスを少し短縮する方法を見つけました。 複数のセグメントの転送があり、障害がなかった場合、システムはネットワークが信頼できると結論付けることができます。たとえば、時間を節約するために一度に6セグメントと7セグメントを送信できます。 また、ネットワークはまだ信頼性が高いため、8セグメントと9セグメントをすぐに送信することもできます。 また、データ転送を高速化します。 その結果、以下を結論付けることができます。ネットワークの信頼性の程度に応じて、任意の数のセグメントを同時に送信できます。 TCP / IPプロトコルの標準により、一度に最大15個のセグメントを送信できます。 しかし、一度に複数のセグメントの転送中にエラーが発生した場合はどうでしょうか？ 1つのパケットで50から64のセグメントを送信すると、63でエラーが発生したとします。 サーバーは言う-もう一度63を送ってください、私はそれを受け取りませんでした。 コンピューターは63を送信し、その後エラーを回避するために一度に送信するセグメントの数を減らします。 したがって、パケットデータウィンドウの期間は可変です。







次に、ネットワーク経由でファイルを送信することを選択できる別のアプリケーションレベルのプロトコルについて説明します。 これは簡単なファイル転送プロトコル-TFTP（Trivial File Transfer Protocol）です。 そして、その定義を見ると、信頼性なしにネットワークを介してファイルを転送すると言われています。 例に戻り、セグメントの送信にこのTFTPプロトコルを選択したことを想像してみましょう。 転送プロセスはどうなりますか？ 私たちのコンピューターはサーバーに「ねえ、ここには75個のセグメントがあります、キャッチしてください」と言います。 そして、私はすぐにそれらを送り始めます。 このプロトコルの利点は何ですか？ 転送速度は高速ですが、信頼性はどこですか？ データベースアプリケーションには使用しないことをお勧めします。 TFTPアプリケーション層プロトコルは、トランスポート層プロトコル（UDPまたはユーザーデータグラムプロトコル）-ユーザーダイアグラムのプロトコルに基づいています。 したがって、送信する内容に応じて、TCP / IPまたはUDP / IPを選択できます。 ほとんどの場合、データ転送レイヤー自体が、提示された2つから目的のプロトコルを選択します。
