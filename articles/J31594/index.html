シングルサインオン認証（SSO）



SSOを使用すると、ユーザーは認証に1回合格した後にポータルリソースにアクセスできます。 ユーザー資格情報の再入力は不要です。



AccessManager



Java Enterprise Systemでは、アクセスマネージャーコンポーネントがSSOを管理します。 ポータルサーバーは、SSOが実行されているシステムのコンポーネントの1つです。



SSOアダプタサービス

-サーバーがAccessManagerに統合されていないサードパーティのサーバーからデータを受信するために、Portal ServerはSSOアダプタサービスを使用します。 SSOアダプターを介してアクセスされるサービスは、メール、スケジューリング、アドレス帳などを提供できます。



ポータルサーバーデスクトップ



ポータルサーバーは、情報を集約してユーザーに表示します。 ユーザーがブラウザに表示するものは、標準デスクトップ（標準デスクトップ）と呼ばれます。

標準デスクトップに加えて、モバイルデスクトップ（モバイルデスクトップ）と音声デスクトップ（音声デスクトップ）があります。



標準デスクトップは、コンテンツが行と列に配置されたWebページです。 ポータルサーバーは、2つの方法を使用してデスクトップにコンテンツを送信します。

Portal Serverコミュニティの役割



（ポータルサーバーで実行されているコミュニティサービスのユーザーの役割）

役割には次の3つのタイプがあります。

所有者 （コミュニティの所有者）、 メンバー （コミュニティのメンバー）、 訪問者 （訪問者）。

コミュニティの所有者は、コミュニティを作成および削除したり、メンバーシップを付与したり、コミュニティへのアクセス権をユーザーから奪ったりすることができます。 コンテンツに関するあらゆる操作（記事の作成、ファイルのダウンロード、削除など）を実行できます。

コミュニティメンバーは、コミュニティアクティビティ（調査、ディスカッション）に参加し、コンテンツを作成し、他のコミュニティメンバー、コミュニティに関する一般情報を見ることができます。

訪問者はすべての基本的な権利を奪われ、コミュニティで作成されたコンテンツを表示できます



コミュニティユーザー向けのサービス：



Wiki

•ディスカッション—コミュニティでのディスカッションは、コミュニティ内で情報を共有するためのメカニズムです。 サブスクリプション、情報取得メカニズムがあります。

•検索（検索）-コミュニティメンバーはコミュニティポータルを検索できます。

AJAXTableContainerProvider



参照 ：

Java SystemPortal Server 7.1 Update 1 AJAXテーブルContainer.pdf




プロバイダーは、非同期JavaScriptおよびXML（AJAX）機能をポータルフレームワークに統合します。 コンテナは、ポータルページの個々のチャネルとポートレットを非同期にロードする機能を提供します。 ページに3つの高速チャネルと1つの低速チャネルがあるとします。 コンテナは最初の3つのチャネルをダウンロードし、ユーザーはWebページのコンテンツが完全にロードされるのを待たずに作業を開始できます。

コンテナ機能

•ドラッグアンドドロップによるチャネルと個々のポートレットの配置

•イベントの非同期インタラクション：チャンネルを最小化および拡張し、ユーザーのページからチャンネルを削除します。

•ページ全体をリロードすることなく、個々のチャネル、ポートレットを個別に更新します。

•ページ全体をリロードせずにポートレットを編集します。

•チャンネルをすばやく簡単に追加および削除します。

•ポータルページのカスタマイズ可能な外観。

AJAXTableContainerProviderの制限

•ユーザーブラウザーでJavaScriptを有効にする必要があります。

•コンテナは、AJAXコンポーネントの機能を自動的に追加しません。 開発者自身が、AJAXコンテナの機能を使用するために適切なコードを作成する必要があります。

•デフォルトでは、チャネルとポートレットの編集モードはユーザーから隠されています。 実際には、AJAXコンテナでは、編集モードでのウィンドウ幅が同様のモードで小さくなっています。

•別のコンテナ内。

•ポータルページに配置できるAJAXコンテナは1つだけです。



結論：

そして最後に、7.1ポータルは、あらゆる時代の人々の既存の標準であるWebSphereポータルにはるかに近いものです。

IBMが非常に保守的な会社であることを思い出したいと思います。 現在、ポータル5.0からポータル5.1へのアクティブな移行が行われていますが、Java 1.3は5.0ポータルで動作し、Java 1.4はポータル5.1で動作し、Java 1.5のみがポータル6.1で表示されました。

また、Sanaポータルでは、1.6未満で記述できます。

WebSphereのバージョン6.1には、AJAXTableContainerProviderに類似したものがあります。これは、ポートレットがオーバーロードされ、互いに独立してイベントに反応するため、ページの再読み込みは必要ありません。

残念ながら、7.1はJSR-168でのみ機能し、IBMポータルは仕様に加えてJSR-168仕様をサポートします。

現在、168番目と286の両方の仕様を保持するSanovoポータル7.2が利用可能です。 ところで、IBMは286仕様を承認したため、同社のポータルはネイティブのSanovo仕様でも動作することが予想されます。

IBMポータルと一緒に、WCM-Web Content Managerを配置できます。これは、IBMの解釈におけるCMSの一種です。 興味があれば、私はその独創性が何であるかを伝えることができます。 WCMで作業しなければならなかったとき、そのアーキテクチャに非常に驚かされました。 WCM 2.0はポータル5.0上にあり、WCM 5.1はポータル5.1上にありました

彼らはポータルとコンテンツマネージャーのバージョンを同期することにしました。 バージョンに加えて、WCMでさらに重要な変更が行われましたが、それについては改めて説明します。

ポータル用のCMSに加えて、IBMにはまったく同じSSOテクノロジーがあります。 ところで、そのために、5.0ポータル（WebSphere Portal Server）および5.0アプリケーションサーバー（WebSphere Application Server）で興味深い不具合が発生しました。

同じユーザーを使用して、1つのブラウザーからポータルにログインし、次にアプリケーションサーバーの管理コンソールにログインするとします。 アプリケーションをデバッグするとき、このようなニーズがしばしば発生します。

管理コンソールに入った後、ポータル上のセッションは強制的に強制終了されます。 アプリケーションサーバーで変更を行い（たとえば、ポートレットが移動するWebサービスを再起動する）、ブラウザーのポータルタブに切り替えて、セッションが無効になっていることを確認します。

ポータルに再ログインし、ポートレットをテストし、管理コンソールでブラウザータブに戻り、管理コンソールのセッションが停止していることを確認します。 管理コンソールにログインすることにより、ポータルのセッションを強制終了します。

このポータルの一般的なユーザーセッションとこのポータルのアプリケーションサーバーを使用した、奇妙なカントです。



一般に、7.1ポータルはIBMポータルに比べてそれほど劣っていません。 主なことは、JSR-286がサポートを取得し、稼働を開始すると、IBMはポータルテクノロジーの分野で深刻な競争相手になる可能性があるということです。
