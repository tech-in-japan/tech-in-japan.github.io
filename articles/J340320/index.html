無料のYouTubeブロードキャストDevOps Conference DevOops 2017

以前に私の投稿を見たことがあれば、JUG.ruグループの私たちはITエンジニア（開発者、研究者、テスター、システム管理者）のために大規模な技術会議を開催していることがわかります。 それで、数週間前、私はコミュニティのメンバーの一人にブログ投稿でカンファレンスをサポートするよう要請しましたが、それは断固として拒否されました。 チケットが支払われるので、あなたは悪の軸上にあり、コミュニティを支援せず、業界を助けず、豊かな企業を楽しませ、彼らがさらに「企業化」するのを助けます。



それから私は、数か月でYouTubeのレポートのすべてのビデオ録画を共有し、オープンJUGを作成し、一般的に、私たちは業界にとって興味深い製品を作成し、目標を追求していないという事実にのみお金を払うと答えました人事部門または営業部門の拠点を集める企業）：人々が必要とし、誰が購入するかのレポートを選択します。これは私たちの切り札とパンです。



しかし、私は別のことを忘れました。会議の無料放送です。 ブログを詳しく調べると、2017年のすべてが会議のメイントラックの公開放送を行っていることがわかります。



2017年10月20日午前9時30分（モスクワ時間）に、 DevOops 2017カンファレンスのメインホールから無料のオンラインブロードキャストが開始されます！ そして、単に放送するだけでなく、2k放送、最終的にあなたのRetinaと4kモニターが何か役に立つために必要になるでしょう！







ブロードキャストリンク



放送はYouTubeから、明日9:30に始まります。 それまでの間、リマインダーを接続してチャンネルに登録します（数か月以内にすべてのレポートを投稿します）。





放送番組



9:30-10:30 //オープニング、JUG.ruグループチームとのインタビュー、会議の主催者とパートナーからの開会の挨拶。



10 : 30-11 : 20 // スケールアウトしてください：持っていない問題を解決する 、Corey Quinn、編集者、lastweekinaws.com



基調講演では、Coreyが、会議レポートから得られた優れたインフラストラクチャに関する優れたストーリーが、なぜ本当にでたらめなのかについて話します。 そのようなレポートを聞いて、あなたはあなたのインフラストラクチャがたわごとだと思うようになります、あなたはたわごとだ、そして一般的に、あなたはセールスマネージャーとして仕事を得なければなりませんでした。 Coreyは内部からこのようなシステムをたくさん見てきたので、すべてのシステムが実際に不完全である理由を説明します。少しリラックスして、新しいスタックに移動するためにゴミ箱に捨てずにプロジェクトで作業を続ける必要があります。



英語で報告してください。 興味があれば、会議の前にコーリーからミニインタビューを受けました。





11 :40-12: 30 // 遅延の原因を適切に責める方法：分散トレースとZipkinの紹介

エイドリアン・コール、ピボタル



応答時間分析は、有罪の検索です。どのコンポーネントがユーザーに顕著な遅延を引き起こしましたか？ マイクロサービスの台頭により、容疑者の輪が拡大し、「調査員」になることがより困難になりましたが、ジプキンは助けることができます。 エイドリアンとコールは理論的な部分を説明し、小さなデモを見せます。





12: 50-13 :40 // SmartMonitoring-オドノクラスニキ 、セルゲイシャラポフ、オドノクラスニキのビジネスロジックの監視



Odnoklassnikiは困難な状況に直面しています。そのような場合、よく知られたソリューションに満足せず、独自のソリューションを作成します。 その結果、ソーシャルネットワークにはそのような問題について多くの専門知識があり、すぐに共有することができます。 この場合、彼らはポータルで異常を見つけるだけでなく、それらの間の接続を表示する独自の監視システムの作成について話します-管理者と開発者の生活を楽にします。





14: 25-15 :15 // システム管理者から男性まで 、Alexander osminog Titov、Express 42



通常、DevOpsについて話すとき、2つの相反する関係者が思い出します-異なるタスクとアプローチを持つ管理者と開発者。 アレクサンダーは、一般に、ビジネスにも側面があり、一般に、最終的な分析では、すべてがタスクに良い影響を与えるはずであることを思い出します。 したがって、彼のレポートは、「システムとしてのDevOpsについて：ビジネスにどのように役立つか、DevOpsのエンジニア側の能力、DevOpsのソフトウェア生産方法によって解決できるビジネスタスク、DevOps生産に至るまでに発生する可能性のあるエラーそれらを回避または停止する方法。」





15:35-16:25 // クラウドへのインフラストラクチャの継続的な提供 、ポールスタック、ジョイエント（旧Hashicorp）



Paulは現在Joyentで働いていますが、最近までHashiCorpの従業員でした-この会社のPackerおよびTerraformツールをよく知っているのは驚くことではありません。 また、DevOopsでは、「インフラストラクチャをクラウドに継続的に提供する」というテーマが、その使用例によって明らかになります。 予備調査では、多くの人がこのレポートを待っていることがわかりました。





16 : 45-17 : 35 // Kubernetes（別名 Failing Demo Talk） 、Ray Tsang、Google＆Baruch Sadogursky、JFrogでの運用アプリケーションのトラブルシューティングとデバッグ



Baruch jbaruchはJFrogの開発者支持者であり、彼は開発者支持者のGoogle Cloud Platformに加わります。 当然のことながら、Kubernetesアプリケーションの実稼働デバッグでは、Google Cloud PlatformとJFrog Artifactoryの両方が関連する可能性がある状況を考慮します。 ただし、これらのプロジェクトについてだけでなく、Kubernetes自体の組み込み機能についても説明します。





17:50-... // 規模のDevOps：3つの行為におけるギリシャの悲劇 、Baruch Sadogursky、JFrog＆Leonid Igolnik、CA Technologies



最後に、会議はBaruchとLeonid Igolnikの基調講演で終了します。これは、会社がさまざまな段階で成長するにつれてDevOpsの問題が生じることについてです。 このような3つの段階があり、結果は「3つの行為におけるギリシャの悲劇」です。



すべてがどのように感じるかについてのアイデアを与える2つの組織の詳細があります。 第一に、講演者はトガスやその他の環境を要求し、ギリシャの悲劇の雰囲気を適切に再現するのを助けました。 第二に、彼らは聴衆に、この公演の前の休憩中であってもパーティーのためにアルコールを準備するように頼み、ある意味でパーティーを開いた。 一般的に、一日の終わりに、数字やグラフを知覚することが難しくなると、そのような問題が明らかに発生しないパフォーマンスがあります。



メインプログラムによると、どういうわけか判明しました。 しかし、それだけではありません！



インターミッション放送







多くのオンラインブロードキャストの問題は、空の休憩とコーヒーブレークです。 会議の参加者がコーヒーを飲み、スピーカーと通信している間、視聴者はスタブを見て、次のレポートが始まるのを待たされます。



私たちはこの問題を独自の方法で解決しました-休憩中に、会議でのイベント、およびスピーカーとのインタビューが放送されます。 インタビューは、 フィレニアムと共にオレグチール （おそらくJavaまたはDevOpsについての彼の記事を読んでいます）-退屈することはありません。 質問が突然発生した場合、会議の電報チャネルで質問できます： t.me/devoopsconf
