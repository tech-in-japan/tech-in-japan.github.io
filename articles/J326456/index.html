GitLab 9.0：サブグループとデプロイボードがリリースされました

バージョン8.0のリリースから 18か月後に、最近GitLab 9.0をリリースしました 。 この間、 GitLabに多くの重要な変更を加え 、毎月22日に新しいバージョンをリリースしました。 8.0から出てきたものを簡単にまとめ、バージョン9の新しい機能と古い機能を比較してみましょう。 または、9.0に登場した機能に移動できます。









アイデアから生産まで



最新のリリースにより、GitLabは製品開発に対するチームのアプローチを変更しました。 ほんの数分で、コンテナスケジューラにGitLabを展開し、自動的に展開されたレビューアプリでCI / CDを追加し、ChatOpsを使用してサイクルタイムを分析できます。 バージョン9.0では、展開ボードで展開を表示し、Prometheusを使用してアプリケーションのパフォーマンスを追跡できます。 マスタープランに基づいて、GitLab 9.0は開発者向けの膨大なDevOpsツールセットを提供します。 仕組みを見てみましょう。









使いやすさとデザイン



バージョン8.0では、UIのほぼすべての要素を変更し、使いやすさを大幅に改善することにより、GitLabインターフェイスを更新しました（数か月前にロゴを更新しました）。使いやすさの改善、小さなCSSの調整からメインUXスレッドの変更まで、すべての分野で主な問題点を見つけて処理します。 各リリース8.xでは、徐々に設計を開発しました。 また、GitLab 9.0では、 全体、グループ、およびプロジェクトのナビゲーションを簡素化する点で大きな進歩を遂げました。 これは、GitLab機能セットがより強力になっている重要な変更点です。







UXを改善し続けるために、GitLabの将来に影響を与えることができる研究パネルがあります。 このグループに参加することで、あなたは最初に新しい機能を試してみることができ、製品の改善においてそれらについてのあなたの考えを考慮に入れます。

デジタルワークコラボレーション



GitLabは、デジタルコラボレーションのためのチーム作りを支援します。 GitLabでのコラボレーションの基盤である問題に多くの変更を加えました。 たとえば、タスクの重み（ 8.3 ）、それらをマージ要求にリンクする（ 8.3 ）、タスクを別のプロジェクトに移動する（ 8.6 ）、強力なフィルターおよび検索インターフェース（ 8.16 ）などです。 また、さまざまなステージ（GitLab言語の「リスト」）を使用してタスクのフローを制御するためのシンプルなメカニズムを提供する発行ボード（ 8.11 ）もリリースしました。 GitLab 9.0はボードをさらに改善し続け、マイルストーンとの統合を改善します。







GitLab 9.0でサブグループを導入できることを嬉しく思います。これは、GitLabでのコラボレーションを改善するためのもう1つの大きなステップです。 グループ内のグループのこの強力な新しいパラダイムは、多数の異なる部門を持つ大規模な組織であっても、チームベースおよびチームファーストの原則に基づいたチームワークを支援します。 私たちの使命は、 誰もが貢献できるようにすることです 。 バージョン9.0は、どこにいてもプロセスを簡素化するのに役立ちます。そのため、実際に組織内の全員が共通のタスクに貢献できるのです。

データベース負荷分散（EE）



データベースクエリの負荷を分散すると、クエリの量とサーバーの負荷を分散できます。 原則として、これにはpgpoolなどのサードパーティソフトウェアの使用が必要です。 バージョン9.0以降、GitLab Enterprise EditionはPostgreSQLを使用する場合のクエリ配布をサポートしています。







このアプローチには、たとえば、メインデータベースの負荷とメモリ使用量の削減、クエリへの応答時間の短縮など、多くの利点があります。 さらに、負荷分散により、過度に要求の厳しいクエリが他のサーバーのデータベースクエリに影響を与えないため、GitLabの機能に悪影響を及ぼす可能性が低くなります。







GitLabクエリバランサーは、データベースの障害にも対応します。 メインデータベースが応答しない場合、またはセカンダリへの切り替えがある場合、バランサーは短い休止の後に操作を繰り返します。 また、セカンダリデータベースが応答しない場合、アロケータは再び使用可能になるまでそれらを無視します。 このようなシステムを最も透過的に機能させるには、各ホストにクエリバランサー（HAProxyなど）を使用する必要があります。







クエリ配布システムを使用する場合、コピー遅延の問題があるという事実に注意を払う価値があります。 たとえば、セカンダリデータベースからの書き込みおよび後続の読み取りの際に、このデータベースにまだデータがない状況が発生する可能性があります。 この問題を解決する1つの方法は、同期コピーを使用することです。 ただし、この方法では、コピーの遅延によりリクエストの処理時間が大幅に増加する可能性があります。 さらに、何らかの理由でデータのコピーが使用できなくなった場合、システム全体が停止する可能性があります。







この問題を解決するために、「スティッキーセッション」アプローチが使用されます。ユーザーがメインデータベースでレコードを開始すると、このユーザーのセッションはメインデータベースを使用し続けます。 これは、タイムアウトが期限切れになるまで（30秒）、または記録されたデータがすべてのセカンダリベースで利用可能になるまで続きます。







データベースバランサーのセットアップの詳細については、 データベース負荷分散のドキュメントを参照してください。

























更新されたナビゲーション（CE、EE）



GitLabでは、ほとんどのビジネスプロセス（製品開発だけでなく）はGitLab.com自体で行われます。 したがって、ナビゲーションの重要性を正確に把握しています。 特に、毎日数時間GitLabを使用しているユーザーにとって、毎日のタスクでスムーズで直感的で効果的なものにしたいと考えています。







ナビゲーションデザインはこの目標を達成するための鍵であるため、9.0ではインターフェイスを変更し、デザイナーからのベストプラクティスとユーザーフィードバックを結び付けました。 一見、何も変わっていません。 これは特別です。 すでにうまく機能していたものを綿密に分析し、問題のある領域のみを変更しました。







ナビゲーションメニューの1番目と2番目のレベルのタブを再グループ化しました（場合によってはマージして名前を変更しました）。 [アクティビティ]タブが[プロジェクト]タブにネストされました。 リポジトリ、タスク、マージリクエスト、およびパイプラインのメインタブは、「アイデアからプロダクションへ」という意味で、それぞれ左から右に配置されています。 「グラフ」タブの第2レベルのタブは、再グループ化されて他の場所に移動しました。 分析からのデータと希望を使用して、各メニュー項目の場所を決定しました。 変更の詳細については、こちらをご覧ください。







もう1つの重要な変更点は、ポップアップサイドバーです。 左上にある、あまり目立たないドロップダウンメニューに置き換えられました。これは、画面コンテンツの大部分をカバーしていません。 以前は、設定にドロップダウンメニューを使用していました。プロジェクトまたはグループのページの右上にある歯車をクリックすると表示されます。 これで、設定がタブ付きナビゲーションインターフェイスに移動し、調和がとれ、使用が簡素化されました。

ボードリストのタスクの順序を変更する（CE、EE）



Issue Boardsは、さまざまな段階（GitLabの「リスト」）を移動するタスクを管理して、アイデアから生産にすばやく移行するための優れた方法です。 しかし、ユーザーは同じリスト内であってもタスクを並べ替えたり優先順位を付けたいことがよくあります。 バージョン9.0では、簡単なドラッグアンドドロップを使用して、タスクボードのリスト内のタスクの順序を変更できます。







Community Editionの発行ボードの詳細については、ドキュメントをご覧ください 。













マイルストーンボード（EES）



GitLabタスクボードを使用すると、1つのマイルストーンでタスクグループを管理できますが、そのたびに関連するマイルストーンフィルターを選択する必要があります。 GitLab 9.0 EESでは、特定のマイルストーンに関連するタスクボードを作成できます。 これにより、個々のマイルストーンに固有のボードを作成できます。







新しいマイルストーンごとにタスクを計画および実行するため、新しいボードが役立つと考えています。 これにより、マイルストーンを簡単に切り替えることができ、同時に、以前に完了したマイルストーンを保存して確認することができます。







エンタープライズエディションの発行ボードの詳細については、ドキュメントを参照してください 。













API v4（CE、EE）



APIは、タスクの自動化とGitLabの管理に最適なツールです。 GitLabをプロジェクトの最初から最後までより良い開発環境にするために毎月追加する新しい機能をサポートするように、APIを絶えず改善しています。







継続的な反復により、既存のAPIにいくつかの矛盾が生じています。 このリリースでは、APIの4番目のバージョンを紹介します。 私たちは、RESTful API標準により一貫性をもたせ、準拠させようとしました。







APIの3番目のバージョンは2017年8月までしかサポートしません。このバージョンを使用して、アプリケーションに必要なすべての変更を加えることをお勧めします。







バージョン4の変更点を確認して、違いを確認してください。







災害復旧（アルファ）（EEP）



会社の規模に関係なく、自然災害や人的要因に対するインフラストラクチャの回復力を確認する必要があります。 このタスクのベストプラクティスの1つは、異なる場所にある少なくとも2つのサーバー（1つはメイン、1つはセカンダリ）にデータを保存することです。 これは、ダウンタイムとデータ損失のリスクを減らすために実装する必要があります。 GitLabに組み込まれているディザスターリカバリーの多くのリクエストを受け取りました。 今日、私たちはそのサポートに向けた最初のステップを表しています。







GitLab 8.5では、GitLabにGeoが付属しています。 これは、メインインスタンスをミラーリングする1つ以上の追加のGitLabインスタンスを開始できる機能です。 Geoの主な目標はgit fetch



プロジェクトのクローン作成とgit fetch



を劇的に高速化することです。 Geoはこのシナリオに完全に対応しますが、障害を完全にサポートするためにこのテクノロジーを使用しないという1つの欠点があります。ディスクに保存されたファイルは複製されません。







この欠点の修正に積極的に取り組んでおり、GitLab 9.0では、災害復旧スクリプトのサポートの第1段階をリリースしています。 このリリースでは、Geoにいくつかの重要な変更が加えられました。







  • LFSを使用する場合、LFSオブジェクトは追加のノードに自動的に複製されます（ マージ要求 ）。
  • これで、すべてのファイルのダウンロードがデータベースに書き込まれます（ マージ要求 ）。

    これにより、後続の反復でこれらのファイルを複製できます。
  • リポジトリを自動的に埋めるための新しいプロセスが登場しました

    （ マージ要求 ）。
  • これで、ユーザーインターフェイスで追加ノードを無効にできます。
  • それまでの間、上記のすべての機能はアルファ版のみにあります。Geoの主な機能は 、長距離にわたるプロジェクトのクローン作成と受信に使用されますが、商用利用に適しています（本番対応）。
