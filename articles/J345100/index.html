Java以上のものですか？

別のJavaプロジェクトを終えて、私は蓄積されたうるささの理由を理解しようとしました。 はい、私はJavaを愛していますが、...がいらいらする「しかし」がいくつかあります。 私は非常に多くの定型的なコードを書く必要があり、コンパイラ自体がその生成を処理できますが、もちろんIDEは役立ちますが、これは問題の解決策ではなく、松葉杖です：何かが変わった場合は、それを再生成してきれいにする必要があります nullをチェックします！ これは歯痛です。良い方法では、最も適切でない瞬間にNullPointerExceptionの形で「予期しない」状態に陥らないように、常にそれらを行う必要があります。 要するに、自然界に他に何が現れたのか、そしてこれが私をJavaに置き換えることができるかどうかを見たいという願望がありました。 さらに、この比較の参加者を説明することは理にかなっています。 私は完全な分析のふりをしていないことをすぐに言わなければなりません。残念なことに、各言語を適切に知る時間はあまりありませんでした。



私が持っていた応募者の必須要件：



•汎用言語

•クロスプラットフォーム（少なくともWindows / Linux）

•安定性

•静的型付け

•自動メモリクリーニング

•完全なオブジェクト指向のパラダイムのサポート

•優れたIDEサポート（Eclipse、IDEA、または最悪のNetBeans）

•既存のフレームワーク/ライブラリへのフレームレスアクセス

•Javaパフォーマンス



さらに、私は非常にしたい：



•コンパイル時に言語によって提供されるヌルポインターセキュリティ

•Cからの初歩を持たない単純なキャスト

•合理的なオペレーターのオーバーロードの可能性

•最終的に基本タイプをクリーニングする。 さて、イチジクはあごひげを生やした時代からすべての種類のフロート/ダブル、ショート/イント/ロングの新しい言語を引き込みますか？ 多分これは、数学的なコプロセッサを持つi386プロセッサの時代には理にかなっていましたが、そうではないかもしれません。 これらの微妙な点はすべて、ハードウェアにより近いオペレーティングシステムのカーネル内に存在し、上に突き出ることはありません。 基本エンティティ：整数と浮動小数点数、それだけでいいので、コンパイラに詳細を操作させます。



•ラムダの調和した共存

•一般的な思慮深く全体的な言語設計

•表現力、ASCIIアートが少なく、神秘的な構造、聖水でスプレーしたい

•文字がサロゲートペアやバイト単位ではなく、本格的なUnicodeポイントで表される通常の行。



このレビューには、ネイティブバイナリまたはCにコンパイルされた言語のレイヤー全体が含まれていなかったことに注意する必要があります。おそらくそれらは素晴らしいのですが、D、Nim、Eiffel、Rust、Goなどのいくつかのオプションを試しました私はこの研究には入らないという結論。 チェックは非常に簡単でした-HelloWord。 出力バイナリは通常、メガバイト単位で測定されます。 すなわち コンソールにダム行を印刷し、重量で3〜4Kを占有する小さなプログラムで、重量はメガバイトです！ あなたはささいなことを言います。 はい、私は理解しています、言語の作成者は子孫のすべての力と富をバイナリに押し込みますが、どういうわけかこれを最適化することができました！？ 印刷のみが必要な場合、世界にあるすべてのものをバイナーにバブルするのはなぜですか？ さて、静的ライブラリの最小限の必要な部分と、動的ライブラリのその他すべてを取り出してください。 誤解がない限り、DとRustの作成者は、これらの言語をオペレーティングシステムのカーネルの作成に適していると位置付けています。 本気ですか？ しかし、この「美しさ」、メガバイトの補助言語コード、そして多数のWindow / Linuxシステムコールをすべて、カーネルにドラッグすることをお勧めしますか？ 一般に、このクラスの言語は、「クロスプラットフォーム」と「安定性」の要件に適合しませんでした（「自動メモリクリーニング」と「オブジェクト指向」では、すべてが常にスムーズであるとは限らず、インターフェイスが追加を忘れる場合があり、クラスの代わりに一部が疑問視されます）代替案を提供する） 言語とオペレーティングシステムの間に非常に厚い層があり、このコードの品質と安定性には多くの疑問が生じます。 たぶん彼はとてもいいです分かりませんが それでも、JVMとJava SDKは長年にわたってテストされており、さまざまなオペレーティングシステムで良好に動作します。 したがって、以下で説明する言語はすべて、多かれ少なかれJVMに基づいています。 次に、簡単な説明で、またはむしろ主観的な印象で、参加者のリストに目を向けます。

Ceylon（1.3.3） -この素晴らしい言語は、有名なJava Hibernateフレームワークの作者であるGavin Kingによって2011年に作成され、現在RedHatコミュニティによってサポートされています。 言語は非常に美しく、考え抜かれ、表現力豊かです。 おそらく、Javaが下位互換性ヨークなしで開発されていたら、Javaはこのように見えたでしょう。 この言語には、上記の「ウィッシュリスト」のすべてが含まれています。 私は本当にこの言語のファンになり、それを学ぶのにほとんどの時間を費やしましたが、残念なことに、それに関する情報はあまりなく、コミュニティはあまりにも大きく活発ではありません。 私の好みでは、この言語はほぼ完璧ですが、欠点について言及したいと思います。 Eclipse / IDE用のプラグインは非常によくできていますが、最適化は最高レベルではなく、要するに恥ずかしく遅くなります。 テストの実行（特別なテストフレームワークがあります）には7（！）秒かかり、テスト自体は即座に実行されますが、開始は非常にゆっくりです。



ほとんどの場合、これは2番目の欠点によるものです-モジュール性システムが言語に縫い付けられている、いや、モジュール性自体が欠陥であるとは言いたくない、私は言語に組み込まれた特定のものが非代替になることを意味し、これは悪いです。 したがって、Ceylonでは、モジュールには独自の形式（.carファイル）があり、これはすべてJBossモジュールシステムに基づいています。 何らかの理由で言語の作成者でさえも「保護」を放棄しました。これにより、たとえば「テンプレートメソッド」などのデザインパターンを実装することができなくなりました。



Scala（2.12） -この言語の作成者は、基本的に「しかし、人々のようにすべてをやろう」という原則に導かれたようです。 通常の*完了_の代わりに、なぜですか？ どうやら、「だれも推測しないように」（c）。 配列は通常[]であることを誰もが知っているので、（）を実行して、脳がそれらを取り出すようにします。 それでは何が意味するのでしょうか？ ジェネリックにしましょうか？ しましょう...など。 <-bとはどういう意味ですか？ 私は、よく考えて、コレクションbからコレクションaにすべてをクローン、または詰め込んでいると思いましたが、いや、これはコレクションのバイパスです...



なんで？ はい、Scalaであるため、Tau Cetiであるため、血液は紫色であり、脳の働きも異なります。要するに、彼らはXML言語に取り込んで縫い付けています。おそらく、Microsoft Office言語に直接統合する必要があります。どうして？ 厳密にxlsファイル形式を言語標準に導入します...最初は、これはすべて厄介です（標準クラスの愚かな名前-StringOps、どうですか？）、誰かが彼女のこの岩を吐き出し、言うでしょう。 そして、支持者は肩をすくめて肩をすくめ、彼らは私たちの超小型回路に登ることをためらわない、と言います。あなたは気の毒なことを理解していないので、私たちはエリートです。 しかし、真剣に、普通の言語は、残念ながら、ヌルポインターに対する保護はなく、オプションでしかありません。 初心者には滑りやすいスポットや落とし穴がありますが、これが集合的な創造性の良い解決策であるかどうかはわかりません。



Kotlin（1.2） -カササギの巣を思い出した：こちらは銀のスプーン、こちらはキャンディーのラッパー...みんなのモットーは、どうやら「私はあなたがたから目をくらませた」という歌だった。 実績のある他の言語のソリューションを組み合わせて、組み合わせてみませんか？ 言語の作成者はこれを実用的なアプローチと呼びます。 時々、それは私には思えますが、彼らは味や何かの欠如（または「美しい」という気持ちが少しの間）に失望しました。 return @ forEach（何らかの直接的な電子メール）などのゲームを見て、私は知的に呪い、それを使わないという誓いを与えたいです。 しかし、一般的に、私はまだ言語、簡潔な、あなたが食べる必要があるすべてが好きでした。 式の割り当てが禁止されていて、基本的な型がCeylonのようにクリーンアップされなかったことは残念です（おそらくJavaとドッキングするときの労力が少なくなります）。



Fantom（1.0.69）はそのような「それ自体」のファントムであり、2005年からすでに存在しているようですが、存在しないようです。 Tuzikの湯たんぽのようにJavaを独自のJVMに引き裂くかのように、彼はそれを単に物語のために流通させました。一般に、JVMとしてブロードキャストできるいくつかの自作fcodeにコンパイルされる言語を見るのは興味深いことでしたCLRで。 言語自体について何を言いますか？ ジェネリックは「まだサポートされていません」と読みましたが、この「これまで」は数年前から続いており、明らかにそうであり続けると思われますが、この言語に対する私の実用的な関心は薄れています。 正確には、一般的なサポートがありますが、組み込みのコレクションに対してのみであり、これは産業レベルの開発ツールとしては十分ではありません。 一方では、null許容型があるようです-「Str？ str」ですが、コンパイラはnullをチェックせずにstrの参照を妨げません。 それは残念です...



D（dmd 2.078.2）-JVMからではなく、追加の参加者として、スポーツの利益のためだけに後で追加されました。 残念ながら、「人間の顔」をしたIDEがないため、Dで書くのはあまり快適ではありません。 このサイトにはIDEの全リストが掲載されており、D自体で書かれているものもありますが、実際にはすべてが判明しています。 これらはすべて、快適なコード記述には適していません。 本当に、どうにかして「cack」できるのは、Ideaのプラグインですが、ドットの後でもカーテンコードはありません。 作業に最も適しているのはEclipseのDDTプラグインですが、その作者は、Dに失望したようで、プロジェクトの作業をやめると書いています。残念です。

地図作成データを含む前のプロジェクトの巨大なテキストファイルが手元にあったので、テストアプリケーションは次の仕事をします。 ファイルのあるディレクトリへのパスはコマンドラインに与えられ、参加者は拡張子でファイルをフィルタリングしてファイルのリストを取得し、それぞれを処理する必要があります。 処理は、行ごとの読み取りと行の解析で構成され、セグメント（つまり、始点と終点の座標を持つ地形上のセグメント）である場合、その長さを計算し、特定のポイントから特定の半径内にあるすべてのセグメントをマップに配置します。 マップ内のキーは、セグメントのグループの識別子です。 プログラム自体は、操作全体の実行時間をミリ秒単位で検出し、見つかったセグメントグループの数を表示します。
