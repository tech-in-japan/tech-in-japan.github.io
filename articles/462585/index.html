Why NestJS



Recently, quite a lot of backend frameworks have appeared in the JavaScript community. And if in terms of functionality they provide capabilities similar to Nest, then in one it definitely wins - this is architecture. The following NestJS features allow you to create industrial applications and scale development to large teams:







  • using TypeScript as the main development language. Although NestJS supports JavaScript, some of the functionality may not work, especially when it comes to third-party packages;
  • the presence of a DI container, which allows you to create loosely coupled components;
  • the functionality of the framework itself is divided into independent, interchangeable components. For example, under the hood, both express and fastify can be used as a framework; to work with a DB, nest from the box provides binders for typeorm , mongoose , sequelize ;
  • NestJS is platform independent and supports REST, GraphQL, Websockets, gRPC, etc.


The framework itself is inspired by the frontend Angular framework and conceptually has a lot in common with it.

Set up work with the database



I chose PostrgreSQL as the DBMS for this article. They don‚Äôt argue about tastes, in my opinion, this is the most mature DBMS with all the necessary capabilities. As already mentioned, Nest provides integration with various packages for working with databases. Because my choice fell on PostgreSQL, it would be logical to choose TypeORM as the ORM. Install the necessary packages for integration with the database:







yarn add typeorm @nestjs/typeorm pg









In order, what is each package for:







  1. typeorm - a package directly from the ORM itself;
  2. @ nestjs / typeorm - TypeORM package for NestJS. Adds modules for import into project modules, as well as a set of decorators-helpers;
  3. pg - driver for working with PostgreSQL.


Ok, the packages are installed, now you need to start the database itself. To deploy the base, I will use docker-compose.yml with the following content:







docker-compose.yml 
 version: '3.1' services: db: image: postgres:11.2 restart: always environment: POSTGRES_PASSWORD: example volumes: - ../db:/var/lib/postgresql/data - ./postgresql.conf:/etc/postgresql/postgresql.conf ports: - 5432:5432 adminer: image: adminer restart: always ports: - 8080:8080


As you can see, this file configures the launch of 2 containers:







  1. db is a container directly with the database. In our case, postgresql version 11.2 is used;
  2. adminer - database manager. Provides a web-based interface for viewing and managing the database.

This configuration will be used for cli typeorm.







Let us dwell on this configuration in more detail. In lines 3 and 4, we get the username and password from the environment variables. This is useful when you have several environments (dev, stage, prod, etc). The default username is postgres, the password is example. Otherwise, the config is trivial, so we will focus only on the most interesting parameters:







  • synchronize - indicates whether the database schema should be automatically created when the application starts. Be careful with this option and do not use it in production, otherwise you will lose data. This option is useful when developing and debugging an application. As an alternative to this option, you can use the schema:sync
