Tarantoolは、すべてのデータをRAMに保存する高性能ストレージです。 多くの場合、最初はそのような異論を耳にします-「しかし、それは非常に高価です」。 しかし、RAMとフラッシュメモリを比較すると、RAMはフラッシュよりも安価であることがわかります。



Tarantoolは永続的なストレージです。 データはRAMに保存されますが、データの変更はすべてアトミックです。 Tarantoolに記録されているものはすべてディスクに書き込まれます。



ここで、私は気を散らし、過去から例を挙げたいと思います。 MySQLの歴史を思い出すと、最初はすべて、MySQLユーザーコミュニティがトランザクションを必要とせず、データストレージの信頼性と原子性は必要なかったという事実から始まりました。 MyISAMのみがMySQLの安定したストレージサブシステムであり、Montyがトランザクションを必要としないことをユーザーに納得させようとしたMySQL 2000を思い出します。



MySQLの経験に基づいて、トランザクションは現代の産業用ストレージシステムに必要であるという信念を形成しました。 これは、システムアーキテクチャの残りのパラメーターを決定する基本パラメーターであるため、Tarantoolはすでにアトミック操作をサポートしており、現時点では本格的なトランザクションサポートを開発しています。 次のバージョンでは、すでにそれがあります。



データアクセス言語の分野では、TarantoolソリューションはSQL言語ですが、他のプログラミング言語もサポートされています。





ただし、データモデルの観点から見ると、Tarantoolはキーバリューストレージ、つまり 主なデータアクセスシナリオは、キーごとに値を取得することです。 したがって、すべてのSQL機能が実装されているわけではありません。



単純なINSERT / UPDATE / SELECT / DELETEクエリに加えて、Luaストアドプロシージャをサポートしているため、使用しているデータアクセススクリプトを非常に柔軟に決定できます。 ストアドプロシージャを使用すると、Tarantoolをデータサーバーからビジネスロジックサーバーに変えることができます。 任意のデータ保存形式、データの一貫性の任意の不変式、任意のクエリをプログラムすることが可能になります。

インデックスは単純または複合のいずれかであり、検索には任意のインデックスを使用できます。 ただし、各スペースでは、単純な主キーを指定する必要があります。これは、挿入、更新、または削除の際にアクセスするために使用する必要があります。 たとえば、フィールド3および5、またはフィールド7、9、11を介して、複合セカンダリキーを介してアクセスできます。

パフォーマンスから始めます。 一般に、これはRAM内のデータストレージであるため、パフォーマンスは最初、たとえばMySQLなどの従来のDBMSのパフォーマンスよりも大幅に高くなります。 複数の接続には非同期I / Oと非同期サポートを使用します。 10,000の活性化合物を簡単に保持できます。 先読みログにすべての変更を書き込んでいるという事実にもかかわらず、1台のマシンでのパフォーマンス。 すべての変更をディスクに記録すると、1秒あたり100,000のUPDATEとINSERTに達します。 このマシンでの読み取り要求については、1秒あたり260,000 SELECTのパフォーマンスを達成しました。 これらはそのようなスーパーディジットではありません。MemcacheまたはRedisのパフォーマンスに匹敵します。

質問はとても良いです。



スライドが参照するベンチマークは10個のスレッドで作成され、各スレッドは一度に10個のリクエストを送信しました。 ベンチマークは、クライアント側が複数の要求をバッチで受信し、応答を並行して送信するように構築されました。 サーバーがパケットを送信するとすぐに、クライアントはパケットを読み取りました。 なぜこのようなパフォーマンス測定スキームを選択したのですか？ Tarantoolをロードするためには、まだ100台以上のクラスターが必要ですが、自由に使える車は2台しかなかったためです。 パフォーマンスを測定するこのアプローチでは、遅延はもちろん、実際の負荷で観察できるものよりも高くなります。 実際、単一のリクエストクエリの実行時間は、データを変更するかどうかに大きく依存します。 SELECTの遅延は最小限です。サーバーはキーからメモリからデータを読み取り、クライアントに送信するだけで、ロックはありません。



一般的に、Tarantoolはロックフリーシステムであり、データ構造とキャッシュのレベルですべてのデータへのアクセスは順次実行されますが、ネットワークレベルでは、多くの接続の並列処理がサポートされます。 このアプローチを実装するために、いわゆる「コルーチン」を使用します。これは、「ライト」または「グリーン」ストリームとも呼ばれます。





要求がデータを変更する場合、変更はディスクに記録する必要があります。 バッチ書き込みを実装するには、深刻なDBMSで「グループコミット」と呼ばれるアルゴリズムが使用されます。つまり、いくつかの蓄積された変更を同時にディスクに書き込みます。 フロントエンドとバックエンドの2つの駅を結ぶ列車として想像するのが最も簡単です。 ディスクに記録する必要があるすべての要求は、ネットワーク（フロントエンド）の操作を担当するプロセスのプラットフォームに蓄積されます。 次に、これらの要求はすべて「トレインに浸漬」されます。つまり、バッファリングされてディスクに書き込まれます。 1つのクエリグループがディスクに書き込まれるとすぐに、「トレイン」が次のクエリグループに戻ります。 したがって、INSERT / UPDATE / DELETE要求の実行の遅延は、プラットフォームを使用している瞬間に依存します。 最新の機器の最大値は0.01秒で、ハードディスクの任意のシリンダーの任意のブロックの記録時間です。
