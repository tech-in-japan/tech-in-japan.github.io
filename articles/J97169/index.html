Androidフォンのファームウェアの開発と変更。 パート2

最初の部分では、公式のファームウェアをRUU形式から更新パッケージ形式に再パックする方法を学びました。これにより、変更した回復パーティション（ recovery rom ）を上書きすることを恐れずに、作成したファームウェアを使用できるようになりました。 それまでの間、HTCは優れたリソースと戦っていますが、ファームウェアの研究と改善を続けていきます。

前のパートでは、クロックのようにロードして動作するファームウェアを作成しましたが、その基本機能を拡張したいと思います。 最も要求される拡張機能の1つは、スーパーユーザー特権（ root ）での作業のサポートです。 busybox統合も含まれます。 さらに、システムの起動時に任意のスクリプトを実行し、 ramdiskをニーズに合わせる方法を学習します。



ビジーボックス





busyboxは、モバイルシステムにとって非常に重要な、小さいサイズとパフォーマンスに焦点を合わせたコンソールUNIXユーティリティのセットです。 Androidシステムとともに、独自のユーティリティセット（システムに必要な最小限の機能を提供するツールボックス ）が提供されます。その結果、量的および機能的にはよりシンプルになります。 システムにbusyboxが存在することで、一方で開発者はデバイス上でリモートで作業するときに快適に感じることができます。一方で、複雑なスクリプトを記述できるようになり、たとえばrun-partsを使用してブート時に独自のスクリプトを起動するメカニズムを実装できます。 一部のAndroidアプリケーション（特にrootを使用するアプリケーション）では、busyboxの存在が必須であることも考慮する価値があります。



私たちの生活を簡素化するために、すでに収集されているbusyboxをプラットフォームに使用します。 たとえば、悪名高いモダコの1.16.0をここでピックアップ できます 。 勇敢で時間がある人は、 公式ツリー （アンドロイドツールチェーンに適応するためのファイルとlibc のアンドロイド実装のためのバイオニック ）からbusyboxを構築するか、 crosstool-ngとuCLibcでXVilkaアセンブリ命令を使用するか、 バイオニックの問題がすでに解決されているcyanogenmodの枝。

したがって、 busyboxを追加する前に、システム内のいくつかのイデオロギーの問題を解決する必要があります。

  • busybox / toolboxを使用する一般的なスキームでは、実行可能ファイルを配置し、 アプレットの名前でリンクを作成する必要があります。 リンクは、 $ PATH変数からのパスでアクセス可能でなければなりません。 busyboxのリンクはどこに保存しますか？ / system / bin ？ /システム/ xbin ？ 別のディレクトリ？
  • 事前に作成されたリンクを使用するか、ファームウェアをインストールする段階で生成しますか？

    最初の部分の例では、 ツールボックスに事前作成されたリンクを使用しました。 ただし、シンボリックリンクに特定の問題があるWindowsシステムでファームウェアが開発されている場合、このオプションは困難です。
  • busyboxのビルドに応じて、 ツールボックスですでに使用可能なコマンドセットをオーバーライドできます 。 誰がより重要ですか？

    busyboxは幅広い機能を提供します（例： chmodの権利の文字割り当て）が、一方で、セットのサイズを小さくするためにbusyboxから既に実装されているユーティリティを除外したい場合があります。

Ramdiskの変更





ただし、リンクを別のフォルダーに配置するだけでは不十分です。 システムの実行可能ファイルの検索パスを拡張する必要があります。 これを行うには、変数$ PATHを補足する必要があります。これは、HTC Heroのアンドロイドの場合、 / init.rcで定義されています

  1. $ adb shell cat /init.rc | grep "エクスポートパス"
  2. エクスポートPATH / sbin：/ system / sbin：/ system / bin：/ system / xbin


すべてのルートディレクトリファイルは、 ramdiskの一部です（他のサブディレクトリと同様）。 最初の部分の知識を使用します ： bootをアンパックし、その後PATH変数を変更できます

  1. $ cat habrrom.boot / ramdisk / init.rc | grep "エクスポートパス"
  2. エクスポートPATH / sbin：/ system / sbin：/ system / bin：/ system / xbin：/ system / xbin / bb


ramdiskを取得する前に、さらに便利なことを行います。

生活を簡素化するために、次のものが必要です。

  1. 書き込み権限でファイルシステムをマウントする機能
  2. USBデバッグはデフォルトで有効になっています。 そうしないと、何かがうまくいかない場合、 adbdサービスが無効になり、その結果、ブート時にadb logcatが利用できなくなるため、理由に関する少なくともいくつかの有用な情報を取得できません。
  3. システムの起動時に任意のスクリプトを実行する（最も一般的な使用例： app2sdをオンにし 、 dalvik-cacheを/ cacheに移動する ）


これを行うには、視線を/default.propに変更します。これには、とりわけ次の設定が保存されます。

  • ro.secure-システムのあちこちで使用されますが、関心のあるプロパティのうち、書き込み用のadbリマウントが含まれ、また、 adbシェルはスーパーユーザー権限で実行されます。 値を0に設定します
  • persist.service.adb.enable - adbdサービスを開始します。これは、 adbを使用して電話とリモートでやり取りするのに役立ちます。 1で公開




初期化スクリプト





3番目の段落を実装するには、初期化スクリプト/init.rcを変更する必要があります。 このスクリプトは、いつものように、Android Init Language上の独自の自転車アンドロイドスクリプトです。 私はアンドロイドソースのフォーマットに慣れることを提案します。 独自のサービスを追加する機能に関心があります （コマンドセットが厳密に制限されているため、コマンドをトリガーに直接追加することはできません）。
  1. service sysinit /system/bin/logwrapper /system/xbin/busybox run-parts /system/etc/init.d disabled oneshot



  2. service sysinit /system/bin/logwrapper /system/xbin/busybox run-parts /system/etc/init.d disabled oneshot



  3. service sysinit /system/bin/logwrapper /system/xbin/busybox run-parts /system/etc/init.d disabled oneshot



つまり サービスsysinitを呼び出しました。このサービスは、開始時にrun-parts /system/etc/init.dを実行し、対応するフォルダー内のすべてのスクリプトをアルファベット順に実行します。 / dev / null （デフォルト）の代わりに出力をロギングシステム（ logcat経由でアクセス可能）にリダイレクトするには、 logwrapperが必要です。

サービスを開始する必要があります（ 無効なオプションでは、このサービスは自動的に開始されません）。 これを行うには、最後のコマンドを起動時トリガーから独自のトリガーに移動して、スクリプトの実行を終了し、サービスを呼び出します。

  1. ＃今ではない
  2. ＃class_start default
  3. sysinitを開始します
  5. プロパティ：habr.sysinit.done = 1
  6. ＃今
  7. class_startのデフォルト


また、 / system / etc / init.dに2つの単純なスクリプトを作成して、これがすべて機能することを示します。 1つ目は標準のエコー （ Hello World !、それなしの方法）を実行し、2つ目はキャッシュをクリアし、 initスクリプトに既に終了したことを伝え、さらにロードを続行できます。

  1. $ cat habrrom / system / etc / init.d / 00banner
  2. ＃！/ system / bin / sh
  3. echo "Hello habrahabr！";
  4. $ cat habrrom / system / etc / init.d / 99complete
  5. ＃！/ system / bin / sh
  6. 同期;
  7. setprop habr.sysinit.done 1


最後のタッチ-実行する権利を設定します（すべて同じ-ファームウェア更新スクリプトで）
  1. set_perm_recursive 0 2000 0755 0755 SYSTEM:etc/init.d





スーパーユーザーの権利。 根





多くの場合、おそらく最も差し迫った問題です。 それを読んだ患者のために。

ソースコードはsuの独自の実装を持っているという事実にもかかわらず、その主な目的は他のユーザーの権限でアプリケーションをスーパーユーザーとして（またはadb shellから）実行することであるため、私たちには適していません。

ChainsDD-アプリケーションスーパーユーザー2.1の 開発を使用します。 アプリケーションの本質は、おなじみのsudo 、またはWindowsシステムの管理者承認モードに似ています。 アプリケーションがその権限をスーパーユーザー権限にアップグレードする場合（だけでなく）、ユーザーに許可を求めます。 また、受け入れられたルールを記憶し、今後のすべての通話に自動的に適用することもできます。

Android 2.1用に既にアセンブルされたアプリケーションを使用しますが、ソース自体からそれを調査およびアセンブルする機会は常にあります 。

アプリケーションは2つのコンポーネントで構成されています。

  1. ネイティブモジュールsu 権利の変更をリクエストすると、リクエストを独自のアクティビティにリダイレクトし、ユーザーの反応を待ちます
  2. Superuser.apkアプリケーション- ユーザーとの対話を行い、設定を保存するインターフェイスのセット（ Activity ）。






示された機能をファームウェアに追加するには、システムの適切なセクションにモジュールを追加し、

  1. $ cp su habrrom / system / xbin /
  2. $ cp Superuser.apk habrrom / system / app /


実行可能ファイルの権利を設定します（一部のアプリケーションは絶対パスでsuを呼び出すことができるため、 / system / bin /にリンクを追加します）
  1. symlink ../xbin/su SYSTEM:bin/su set_perm 0 0 06755 SYSTEM:xbin/su



  2. symlink ../xbin/su SYSTEM:bin/su set_perm 0 0 06755 SYSTEM:xbin/su
