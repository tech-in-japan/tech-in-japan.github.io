クラウドサービス：ビデオ監視からビデオ分析まで

クラウドインフラストラクチャに基づくサービスとしてのビデオ監視（VSaaS、サービスとしてのビデオ監視）は、最も関連性の高い業界トレンドの1つです。 英国のIMS Researchによると 、グローバルVSaaS市場は2011年に5億ドルに達し、2014年には10億ドルを超える可能性があります。つまり、3年で倍増する可能性があります。



同時に、米国のABI Researchによると、ビジネス向けのビデオ分析ツールの市場は2011年から2016年にかけて2.5倍以上増加し、9億ドルに達する見込みです。



「サービスとしてのビデオ分析」は、これら2つの市場の交差点にあり、オペレーターの介入なしでビデオ分析に焦点を当てていますが、今日のVSaaSという用語は、分析なしでリモートで表示および記録する機能を意味することが多くなっています。



この記事では、市場の現状とVSaaSテクノロジーを分析し、ビデオ分析を使用して、VSaaS市場の発展に対する主な障害-サブスクライバーの発信チャネルを制限すること-を克服する可能性も検討します。



IMS Researchによると、VSaaS市場の成長は、個々のユーザー、中小企業、および州からの需要によって推進されています。 ユーザーの特定のセグメントにとって、VSaaSサービスは、ネットワークビデオレコーダー（NVR）およびビデオ管理システム（VMS）に基づく従来のソリューションよりも魅力的です。 したがって、VSaaSの商業化モデルは、投資収益率を保証しないハードウェアとソフトウェアのソリューションのコストではなく、消費者が特定のサービス（ビデオ録画、セキュリティサービスの自動呼び出し、データ収集、分析レポートの作成など）を支払うことを前提としています。 VSaaSサービスは、保存されたビデオの量、観測ポイントの数、およびシステムのユーザー数の点で非常にスケーラブルです。



VSaaSを積極的に開発している企業は約30社あり、その数は毎月増加しています。 これらの企業のターゲット市場は、小売チェーン、建設業者、小規模オフィス、家庭です。

世界、特にロシアでのVSaaSサービスの拡大に対する最も重要な障壁は、ローカルネットワーク外の通信チャネルの帯域幅が不十分であることです。 カメラによって生成されたデータストリームと平均加入者接続速度を比較します。

一方では、H.264などの最新の圧縮アルゴリズムを使用する場合でも、標準解像度カメラ（0.4メガピクセル）は0.5から4メガビット/秒のデータストリームを生成し、高解像度カメラ（13メガピクセル）は1から良好な表示条件では10 Mbit / s、悪い条件では最大50 Mbit / s。 多数のカメラを備えたシステムでは、データの送信、保存、分析のコストが重要になります。



一方、発信チャネルの平均速度は、2012年6月のNetIndexデータによると、世界で4 Mbps、ロシアで13 Mbpsです。 非対称アクセステクノロジー（LTE、4G、ADSL、ケーブルモデムなど）を使用する場合、サブスクライバーからVSaaSアプリケーションへの発信チャネルは着信の410倍小さくなります。 したがって、VSaaSサービスでは、多数のカメラ、特に高解像度カメラ（1メガピクセル以上）からのビデオのリモート表示および録画ができません。



推定によると、世界では、カメラで記録されるビデオデータの量の年間成長率は50％以上であり、チャネル容量の増加はわずか20％です。 したがって、VSaaSのニーズと通信チャネルの機能のギャップは、毎年30％ずつ増加しています。



ビデオ分析は、加入者の発信チャネルの問題、およびビデオをクラウドに保存する問題を解決できる唯一のテクノロジーのようです。 LAID（LinearArrayofIdleDisks）などのビデオを保存する費用対効果の高い方法の出現にもかかわらず、大容量の保存

クラウド内のビデオボリュームは、VSaaSサービスの最も高価なコンポーネントです。



ビデオ分析は、ユーザーが必要とするデータのみをビデオに残す特殊なエンコーダーと見なすことができます。 H.264などのユニバーサルエンコーダーは、画像内の各要素の重要度を「理解しない」ため、冗長データを効果的にフィルター処理してVSaaSサービスを提供できません。 たとえば、標準的なエンコーダでは、背景の小さな人と前景の多数の雨滴を区別できません。 人と各小滴が同じ詳細でエンコードされている場合、フローは送信と保存のために実質的に冗長になります。

原則として、ユニバーサルエンコーダーとビデオ分析が一緒に使用されるため、それぞれを個別に活用できます。

明らかに、通信チャネルの負荷を減らすために、ビデオ分析はサブスクライバー側で機能する必要があります。 さらに、 顔認識などの一部の種類のビデオ分析では、最大精度を達成するために非圧縮ストリームのビデオ分析が必要です。 これらの理由により、クラウドインフラストラクチャは、加入者側に機器がなければプライマリビデオ処理に経済的に使用できません。



一方、クラウドインフラストラクチャを効果的に使用して、次の次元でビデオ監視システムを拡張できます。

  • ビデオおよびビデオ分析メタデータの保存。
  • 新しい観測対象の接続（たとえば、コンセント）;
  • 新しいメタデータ分析およびアーカイブ検索機能の実装。
  • 多数のユーザーにサービスを提供しています。


ビデオ分析の精度が高いほど、通信チャネルとクラウドストレージの負荷が低くなります。 したがって、ビデオ分析の精度指標がわかっている場合、サービスプロバイダーまたは消費者はビデオ分析を使用することの経済的利益を簡単に計算できます。

例を考えてみましょう。 大規模な太陽光発電所（太陽熱設備） の周囲を保護するために 、300台の高解像度カメラ（1.2メガピクセル）が使用されています。 通常の気象条件では、合計

ビデオストリームは1.8 Gb / sです。 悪条件の下、つまり、夜間などの信号がうるさい場合、流量はほぼ2倍に増加して3.5 Gb / sになります。 従来の動き検出器を使用すると、ビデオデータの量を平均で80％削減できます。つまり、0.40.7ギガビ​​ット/秒に削減できます。

残念ながら、 モーション検知器はすべてのカメラの光と気象条件のグローバルな変化に即座に応答するため、ピーク負荷はこの範囲よりも何倍も大きい場合があります。



ソーラーステーションの領域にいる人々を検出するための専門的なビデオ分析の使用により、平均負荷を0.010.02 Gbit / sに、ピーク負荷を0.05 Gbit / sに減らすことができました。 したがって、 従来の動き検出器と比較して 、ビデオ分析は通信チャネルとクラウドストレージの負荷を40倍以上削減しました。



カメラのフラッシュカードまたは加入者側のゲートウェイサーバー（ゲートウェイ）のハードディスクにビデオをローカルに保存すると、データバッファリングにより発信チャネルをより効率的に使用できます。 ビデオ分析イベントが発生すると、カメラまたはローカルサーバーゲートウェイがローカルメモリにビデオを記録し、利用可能な通信チャネルを介してクラウドストレージに転送します。 このようなイベント記録モードでは、記録速度が最大ビデオデータレートよりも高くなる場合があります。



ローカルストレージは、カメラ（ローカルサーバー）とクラウドアプリケーション間の通信チャネルが一時的に切断された場合に、ビデオ監視システムのフォールトトレランスを確保する手段でもあります。



VSaaSアプリケーションの一部としてビデオ分析を適用するより洗練された方法の例は、Synesisによって開発されたビデオデータランキングテクノロジーです 。 ここでは、単純なDVRのように、ビデオ品質の低下により帯域幅が制限された通信チャネルを介したビデオ送信は発生しませんが、ユーザーにとって最も重要な情報の優先送信により、ビデオフラグメントの優先度はビデオ分析によって自動的に決定されます。 このアプローチにより、クラウドディスクスペースの使用効率も向上します。ローカルストレージまたはクラウドでのビデオデータの保存時間は、その重要性に依存する場合があります。



結論として、VSaaSサービスを提供するための従来のビデオ管理システム（VMS）とクラウドベースのアプリケーションの主な違いを考慮してください。

  1. VSaaSでは、より洗練されたテクノロジーを使用して、観測対象とクラウドの間、およびクラウドとサブスクライバーの間の通信チャネルを制御します。 速度制限だけでなく、アドレス変換（NAT）とファイアウォールを備えた複雑なトポロジも考慮する必要があります。
  2. VMSは、CCTVのモニターの前に常設オペレーターがいることを想定していますが、VSaaSサブスクライバーは、ほとんどの場合、アラームメッセージの後またはビデオアーカイブ（レポート）の分析のためにのみブラウザーを介してシステムに接続します。
  3. ポイント1と2の結果として、ビデオ分析はVMSよりもVSaaSでより重要な役割を果たします。 ビデオ分析を使用して、データの送信、アラームメッセージとレポートの形成を制御します。
  4. VSaaSユーザーインターフェイスは、ブラウザーまたは携帯電話に基づいて実装する必要がありますが、VMSはインストールされたアプリケーションをより頻繁に使用します。
  5. ローカルストレージのサポートは、VMSよりもVSaaSアプリケーションで重要です。
  6. VSaaSは、VMSよりもはるかに多くのユーザーとカメラに対応する必要があります。
  7. 原則として、プロのシステムインテグレーターはVMSアプリケーションを構成し、エンドユーザーはVSaaSアプリケーションを使用します。VSaaSアプリケーションは、VSaaSインターフェイスのシンプルさに高い要求を課します。
  8. VSaaSサービスオペレーターやその加入者は、VMSユーザーよりも最終機器（カメラ、ローカルビデオサーバー）のコストに敏感です。
