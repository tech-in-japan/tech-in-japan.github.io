ゲームからキャンディーを作りましょう。 マップアニメーションがなければ、すべてが何らかの形で鋭くandいように動作します。 メインレイヤーの上に別のグラフィックレイヤー（読み取り-透明なキャンバス、キャンバス）を追加すると、デッキとテーブル、テーブルとプレイヤー、デッキとその上のプレイヤーの間にカードの「レース」を配置します。 回転、歌、ダンス。 アニメーション化したいカードは、外部のキャンバスに転送されるだけで、隣人や背景を誤って消去しないように、中国の店で象になることを恐れずに、すでに自由に移動します。 そして、パスの終わりに、彼女は下のキャンバスに戻り、きちんと振る舞います。 必要ない場合は、ゲームメニューからアニメーションをオフにすることもできます。 ちなみに、クリックハンドラーはすぐに外部キャンバスにハングアップします。



次の入力パラメーターを使用して、キャンバス上のマップの動きを設定します：速度Vおよびポイント、初期（X0; Y0）および最終（X、Y）。 移動するとき、移動の開始から時間tiが経過した後の位置にスプライトを描画します。 そして、以前はもちろん、古い位置からスプライトを消去します。 前のレンダリングが完了した直後に、次のレンダリングを呼び出します。 したがって、コードが実行されるデバイスが許す限りスムーズな動きを保証します。 時間を参照すると、ブラウザの短期的なブレーキに関係なく、全体として動きが均一になります。 計算原理は次のとおりです。



モーションコール関数の開始時に、モーションのいくつかのパラメーターを定義します

そして、移動のサイクルでは、毎回、すでに時間内に座標を計算します 



 //,    ,  var ti = Date.now()/100 - t0; //  (Xi;Yi)      var Xi = constX * ti + X0; var Yi = constY * ti + Y0;


そして最後に、モーションのセグメントに属するこれらの座標を確認します。 つまり、XiはX0とXの間にあり、YiはY0とYの間にある必要があります。この条件が満たされる限り、（Xi; Yi）にスプライトを描画し、モーション関数をループします。 さらに、コードは理解できると思うので、詳細にペイントしても意味がありません。 原則全体を説明しました。
