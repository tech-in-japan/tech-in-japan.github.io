小売店対オンラインストア

それは小売店から始まりました。 シンプルなWebサイトと満足のいくSEOにより、顧客への電話と訪問が提供されました。 最後に、トップのターゲット検索クエリの数がクリティカルマスを超えました。 そして...顧客はアパートやオフィスへの配達を積極的に要求し始めました。



ここでIMが必要かどうかを把握しましょう。

または-ミラーストーリーの場合。 小売を追加した場合、売上はどのように変わりますか？

だから。 プロジェクト「Bambolo」の例に関するフルコーンのコレクション



カーニバル衣装の販売には、顕著な季節性があります。 年間の需要には、ハロウィーンと新年の2つのスパイクがあります。 残りの時間に100％の販売を行った場合、急増中の需要はそれぞれ1000％と1400％になります。 IMを使用して需要を増やし、売上の増加をスムーズにすることは可能ですか？ 需要のピーク時にMIが販売を拡大する方法。 そして、どんな問題があなたを待っています。



最後に、「小売とIMの併用が総売上に与える影響」を理解しようとします。



2011年夏 小売店があります-IMを開始します。 問題を解決する理由：



  1. IM用のテレフォニーの選択（ 2回直接参照... ）

    地域ごとに8 800を取る予定です。
  2. 私たちは宅配便サービスの中で入札を行います（参照してください。 完璧を探して... ）
  3. CRMシステムを選択します。 要件：

    重要です
    • CEO向けに最適化。 最適化がない場合、販売はありません。
    • スケーラブル。
    • IMウェアハウスと小売店を同期できます。 これを行わない場合は、サイトに書かれている理由を常に説明しますが、実際にはすべてが異なります。 不快な仕事。
    • ビジネス目標の変更に利用できます。


    できれば

    • 安い。 すべてがどのように発展するかは知られていないからです。
    • Web管理情報へのアクセス


  4. 支払いシステムを選択します。

    Platron.ru。 なんで？ 全員の手数料率は同じであるため、全員にAPIがあります。 最初は、別の会社に焦点を当てました。 しかし、ハッカーの攻撃を仕掛けたために監督が逮捕されたことに恥ずかしかった。

  5. 小売店で商品を会計する方法を選択します。




解決策として、自作のCMSを選択しました。 なんで？

技術的には、小売販売とIMを介して、1つの倉庫から商品を償却する2つの方法があります。 会社が最初に小売店に登場した場合、通常は1Cを受け取り、それにIMを添付します。 商品のデータベースは既に存在し、スタッフはインターフェースに慣れています。 1Cのもう1つの間違いは、実際には業界標準であることです。 専門家とターンキーソリューションが常に見つかります。 短所は誰もが知っています。 ただし、この状況での主なものは2つのベースです。つまり、ベースの同期は常に弱点になります。

私たちの状況では、1Cにはへその緒がなく、ゼロから作業しました。



そのため、エンタープライズ管理システムがあります。小売販売の可能性があるサイトです。 既製のソリューションは見つかりませんでした。 したがって、彼らは自分でやった。 以下にその仕組みを示します。



同期ではなく単一ベース

製品が小切手に分割されると、そのサイトで購入できなくなります。 同様に、サイトで注文するとき-製品は店で突破しなくなります。 サイトで-常に現在の商品の量。 彼が店で同時に購入され、拠点が同期されるまでサイトで注文されたことが判明しません。



小売販売用-キャッシャーの職場

管理パネルのセクションの1つです。 動作するには、ラップトップ、バーコードスキャナー（HQ）、プリンター、および4kbpsのインターネットが必要です（つまり、モデムでも動作します）。 モバイルでスケーラブル。 たとえば、売上がピークの場合。 新しい職場は、USBを介したラップトップ+インターネット+ HQスキャナーです。



サイトの管理パネルのすべての倉庫会計

製品は、倉庫で資産化され本社に接着されたのと同じ秒からサイトに到着します。 遅延なし。 商品を作るとき、購入と販売のコストが示されます。 検索エンジンでのプロモーション用にすぐに準備された製品カード。

もちろん、管理会計は世界中のどこからでも、またどの携帯電話からでも手元にあります。



迅速かつ簡単なサポート

高価な1Cプログラマを雇う必要はありません。 より多くの人がphpで安く書いています。



事前に考えていなかったこと：

  • 宅配便で働くことの現実。

    そう思いました。 統合を実行しましょう。CMSは、特定の時間までに注文があるかどうかを確認します。 存在する場合、クーリエはAPIによって呼び出されます。 運送状は、住所、受取人、重量、商品リスト、金額などのCMSデータから作成されます。 実際、次のとおりです。 私たちは宅配便サービスの非標準顧客です。 したがって、すべての宅配便の従業員が当社を理解しているわけではありません。 APIはどのように宅配便業者を呼び出すことができますか？ 週に一度、彼らは宅配便の必要性、荷物の発送などについて私たちにしつこく質問します。 時々到着する宅配便-手で領収書に記入する必要があります。

    2012年春の状態では、主要な技術的問題は削除されました。 拒否された商品を返すプロセスが確立されています。
  • 店内の人的要因。

    在庫を実施し、基地を整理しました。 そして、物語の1つが起こります：あなたから何かの未知の数が盗まれているか、売り手がいくつかの合成衣装の一部を売るでしょう。

    インベントリを再度取得する時が来ました。 この質問はより組織的ですが、売り手が一見意図的な決定にどのように影響を与えることができるかを示しています。
  • ジオドメイン

    タイムマシンがあれば、サンクトペテルブルクとモスクワの両方で1つのドメインを宣伝することに投資することはなかったでしょう。 すぐにジオドメインの作成を開始します。 なぜこれがそんなに重要なのですか？ そのため、Yandexの主張なしに、最も重要な地域で数十のドメインを実行できます。 ロシアの総人口のわずか6分の1がモスクワとサンクトペテルブルクに住んでいます。 残りの地域で顧客が見つかる可能性があります。 購買力が低下していますか？ しかし、競争は何倍も低くなっています。

    ちょうど今、ジオドメインの実験を開始しました。 これから何が来るのか、少し後で-地域統計を収集するときに書きます。 あぁ！ すでにこれを経験した人はいますか？ コメントを書いてください。 あなたの経験は今、私たちにとって非常に役立ちます。
  • Yandex.Server-サイトの検索エンジンとして

    サイトには何百もの製品がありますか？ すぐに、売り手でさえ混乱し始めます。 価格、完全性など 関連する製品検索を行うまで、顧客はサイトをさまよい、電話で苦しめます。
  • 商品用フィルター

    製品の名前がわかっている場合、検索は有効です。 そうでない場合は、フィルターと製品カテゴリーが必要です。 利用可能性、性別、色、サイズを確認します。
  • 顧客の質問

    作成およびテスト済み。 サイト上-商品の実際の量。 使いやすさは普通のようです。 証拠のキャプテンになる必要があるという事実に備えてください。

    クライアント：あなたはサイトに5つのピースがあると書いています。 いくつありますか？

    あなた：5。 サイトには実際に利用可能な番号が示されています。

    これに関する時間とリソースの無駄遣いを止める方法は、まだわかっていません。

mysql 5.1.62上のデータベース、バックアップサーバーへのレプリケーションが構成されています。 繰り返しますが、負荷が高い場合は、すべての読み取り操作をバックアップサーバーに切り替えることができます。

php 5.3.2-箱から出して

ホスティングにはユニークなものはありません-ほとんどすべてがすぐに使用できます。 ピーク時には、システムは1日あたり5.2k個の一意のページを保持し、表示深度は5.6ページでした。 ピーク時に、1秒あたり2.1 phpスクリプトリクエストに達しました。 これは統計をカウントしていません。

サイト上のほとんどすべての情報は、最小限のキャッシュでリアルタイムに計算されます。 現在のニーズに対しては、それで十分であり、ユーザーのフローを増加させながら生産性を操作する余地を与えます。



結果

オンライン売上は、総売上を2〜3倍に増やします。

その結果、さまざまな販売元の月ごとの変動への依存度が低下しています。
