Windows 95ユーザーインターフェイスの設計

3年前、Windows 95の新しいユーザーインターフェイスを設計したプロセスと結果に関するマイクロソフトの従業員Kent Sullivanによる興味深い科学記事に出会いました。それ以来、Webページは消えてしまいました。これが私がデジタルPlyushkinである理由の1つです。



この記事では、Windows 3.1のプログラムマネージャーシェルの一般的な問題について説明し、「初心者」用に別のシェルを開発するためのオプションを検討します。 System 7の時代に非常に人気があったAppleのAt Easeプログラムの精神で作成されたと思われます。小学校でAt Easeを始めた方法をよく覚えているので、子供たちはFinderのハードドライブに煩わされる必要がありませんでした。



したがって、これは、ケントが「Windows 95ユーザーインターフェイス：ユーザビリティエンジニアリングのケーススタディ」というタイトルの記事で逐語的に書いたものです。 ドキュメントが失われないように公開します。





まとめ



多くの人々が、Microsoft Windows 95などの大規模な商用ソフトウェア製品のユーザーインターフェイスの開発に関与しています。 このプロジェクトには、広範なタスクと積極的なスケジュールがあります。 ここでのプロジェクトの概要では、UI開発プロセスの管理性を改善するために、ユーザビリティ設計、反復開発、問題追跡の原則をうまく適用した経験について説明しています。 特定の設計上の問題とその解決策についても説明します。



はじめに



Windows 95は、Windows 3.1およびWindows for Workgroups 3.11に対する広範な更新プログラムです。 Windowsのほぼすべての部分で、多くの変更が発生しました。 ユーザーインターフェイスも例外ではありませんでした。 この記事では、設計チーム、そのタスク、および作業プロセスについて説明します。 次に、プロジェクトが反復開発や問題追跡などのユーザビリティ設計の原則をどのように適用したかを説明します。 特定の設計問題とその解決策を例として示します。



設計部門



Windows 95ユーザーインターフェイス開発チームは、プロジェクトの初期段階で1992年10月に結成されました。 1992年12月、ユーザビリティサービスを提供するアシスタントとしてグループに参加しました。 チームは、デザイン、グラフィックデザイン、ユーザビリティテスト、コンピューターサイエンスの分野の専門家で構成された、真に学際的なチームでした。 プロジェクト中に従業員の数は変動しましたが、平均して私たちは12人でした。 ユーザーインターフェースを実装するプログラマーは多数います。



設計目標



設計チームは、2つの非常に広範なタスクに取り組みました。



  • コンピューターとWindowsを使い始めたばかりの人がWindowsを簡単に習得できるようにします。
  • Windows 3.1は、通常のWindows 3.1ユーザーと経験豊富なWindows 3.1ユーザーの両方で、既にコンピューターを使用したことがある人にとって使いやすいです。


5,000万を超えるWindows 3.1および3.11のインストールに加えて、ほとんど手付かずの家庭用PC市場があるため、最初から、より優れた製品をリリースするタスクは簡単な作業ではないことが明らかになりました。 慎重な設計とテストを行わないと、特定のカテゴリのユーザーに対してのみ使いやすさを向上させる製品を作成する可能性がありますが、他の何百万ものユーザー（既存または潜在ユーザー）に対しては使いやすさが低下します。 私たちは平均的なユーザーや上級ユーザーの問題をよく理解していましたが、初心者が経験する問題についてはほとんど知りませんでした。



設計プロセス



非常に広範なタスクと積極的な作業スケジュールを考えると、製品がリリースされる前（ユーザーインターフェイスの設計とプログラミングに約18か月かかりました）、当初から、従来のカスケードモデル（「ウォーターフォール」）を使用した開発では最適なソリューションを達成するための十分な柔軟性が得られないことがわかりました。 実際、従来のアプローチでは使用できないシステムが作成されるのではないかと心配していました。



カスケードモデルでは、システム設計は複数の部分に分割され（通常は仕様の記述段階に限定されます）、通常、品質管理活動中のプロセスの終わり近くでユーザビリティテストが行​​われます。 これは、デザインを作成し、ユーザーでテストし（おそらく他のデザインと比較して）、変更を加え、フィードバックを収集するには不十分であることに気付きました。 幸いにも、反復開発を支持してカスケードモデルを放棄したいという願望は、会社の他の部門での同様の試みと一致したため、その利点と適用性の具体例がありました。

Windows 3.1との比較



最初の実験室の実験から、Windows 95以前に存在していた問題と新しい設計に特有の問題をよりよく理解するために、Windows 3.1ベースが必要であることが明らかになりました。 まず、Windows 3.1ユーザーが実行する20の最も一般的なタスクに関する市場調査データを収集しました。 次に、これらの20のタスクを念頭に置いて、Windows 3.1とWindows 95を比較するいくつかの実験室研究を実施しました。 また、Windows 3.1のプロの教師（および比較のためにMacintosh）にインタビューし、オペレーティングシステムのどの側面をユーザーに教えるのが簡単で難しいと考えるかを理解しました。



主な結果：



  • Windows 3.1では、新規参入者は、画面にすぐに表示されないプログラムを見つけて開くのに平均9.5分かかりました。 プロトタイプのWindows 95では、結果はそれほど良くありませんでした。 こうした結果は、市場調査データ（および常識）がユーザー向けのプログラムの立ち上げが最重要課題であることを示しているという事実を考慮すると、明らかに受け入れられません。
  • 新規および一部の平均的なユーザーは、マウスの使用、特にダブルクリックで大きな問題を抱えています。 その結果、彼らはしばしばコンテナ内の要素を見つけることができず、そのアクセスはダブルクリックによってのみ開かれました。
  • 初心者と多くの平均的なユーザーは、チームを検索するためにほぼ排他的に視覚情報に依存していました。 彼らはメニューバーとツールバーに依存していましたが、トレーニング後もポップアップ（コンテキスト）メニューを使用しませんでした。
  • 最も上級ユーザー以外は、重複する複数のウィンドウを効果的に管理する方法を理解していませんでした。 初心者はほとんどすべての問題を抱えていました。ウィンドウを最小化するとき、別のウィンドウで閉じられた場合は「左」になると考えていました。 ユーザーがコンピューターのRAMを使い果たし、最初の実行中のコピーに切り替える代わりにプログラムの多数のコピーを起動したため、教師から多くの話を聞きました（そしてこれを実験室で観察しました）。 平均的なユーザーはより多くの経験を示しましたが、特にプログラムマネージャーやMicrosoft Wordなどの複数ドキュメントインターフェイス（MDI）アプリケーションでは問題もありました。 市場調査データにより問題が確認されました。平均的なWindowsユーザーの40％が一度に複数のプログラムを実行しなかったことが判明しました。
  • 初心者ユーザーは、階層ファイルシステムに困惑していました。 通常、平均的なユーザーは階層を管理していましたが、多くの場合、困難を伴い、使用するプログラムのデフォルトディレクトリにすべてのドキュメントを保存していました。 この問題（特に初心者向け）は、Macintoshユーザーの間でも見られました。


方向転換



Windows 95の初期のプロトタイプでは、Windows 3.1のいくつかの要素（たとえば、デスクトップ要素が実際のコンテナーになった）を変更しましたが、他の要素は変更されていません（ファイルマネージャーとプログラムマネージャーのアイコンなど）デスクトップで）、彼らはあまりにもデザインを変更することを恐れていたので。 Windows 3.1と根本的に異なる製品は、何百万人もの既存のユーザーを混乱させ、失望させる可能性があることを知っていましたが、これは明らかに受け入れられません。



ただし、Windows 95のプロトタイプとWindows 3.1で収集されたデータは、現在の設計を維持できないことを示しています。 基本的なタスクを実行する初心者ユーザーの結果は容認できないほど貧弱であり、多くの平均的なユーザーは、Windows 95は単に異なるが最高のシステムではないという意見を表明しました。



私たちは後退し、数日間状況を考えることにしました。 設計チームは、オフィスの外でフィールドミーティングを開催し、その時点で収集されたすべてのデータ（基本的なユーザビリティスタディ、インタビュー、市場調査、サポート情報）を確認しました。 データの議論の中で、最も頻繁に実行されるタスクに集中する必要があることに気付きました。 また、Windows 3.1との接続性に注意を払いすぎていることにも気付きました。



実際、実行可能なソリューションはWindows 3.1のように見える必要はありませんが、さまざまな理由で、あらゆるレベルのユーザーにとって間違いなく魅力的なはずです。 真に便利なシステムは、さまざまなユーザーのニーズに応じて拡張できることを理解しました。学習と学習が​​容易であると同時に、経験豊富なユーザーに効率（ショートカットまたは代替方法を使用）を提供します。



高速反復ステージ



新しいデザインに取り組み始めたとき、「習得しやすいが使いにくい」という古典的なパラドックスを避けたいと考えました。 UIの基本機能はスケーラブルでなければならないことを常に念頭に置いています。 この目標を達成するためには、さまざまなアイデアをすばやく試して比較し、最も有望と思われるアイデアを繰り返し実行する必要があることを知っていました。 これには、非常に効果的な設計および評価プロセスが必要でした。



UI仕様プロセスの進化



当初から反復的な開発プロセスを選択しましたが、カスケード方式には1つの要素があります。それは、モノリシック設計仕様（「仕様」）です。 最初の数か月で、仕様は急速に成長し、数百人分の作業を反映しました。 しかし、ユーザーのテスト中に発見された問題により、仕様に記載されている設計は突然古くなっています。 重要な決定は、仕様を変更して新しいアイデアを反映し、反復に必要な貴重な時間を失うか、仕様の更新を停止し、プロトタイプとコードが実際の仕様の役割を果たすようにすることです。



議論の後、グループは2番目のオプションを選択しました。 このような変更により、外部のグループが私たちの仕事を追跡することは困難になりましたが、反復を最大速度で実行できました。 また、変更は予想外の効果をもたらしました。仕様のほとんどが口頭で存在し、会話と従業員のオフィスのホワイトボードで議論されたため、チームをより強く結集しました。 多くの「廊下」の会話が続き、それはプロジェクト全体を通して続いた。



関心のあるすべての関係者が常に最新のデザインを使用できるように、次のイベントを開催しました。



  1. デザイン部門の定期会議 。 毎週（場合によってはより頻繁に）ミーティングで、全員が自分の仕事を一般プロジェクトで確認し、各従業員の仕事が他の人にどのように影響するかについて議論しました。
  2. ユーザビリティチャートとユーザビリティテストの結果をメールで送信します 。 設計チームのメンバーは、今後のユーザビリティテストと完了したテストの結果に関する定期的な通知を受け取り、設計の進捗状況を把握します。
  3. 正式なユーザビリティトラッキング 。 Windows 95規模のプロジェクトでは、特定されたユーザビリティの問題、それらをいつどのように修正する必要があるかを記録する標準的な方法が必要であることがわかりました。 このプロセスについては、「オープンチケットの追跡」の章で詳しく説明します。
  4. 外部グループ向けの定期的なデザインプレゼンテーション 。 プロジェクトが進行するにつれて、ますます多くのグループ（Microsoftの内外）が私たちが何をしているかを見たいと思ったので、そのようなプレゼンテーションを行いました。 プレゼンテーションは最新の状態を維持しやすく、設計についてタイムリーに議論できるため、ドキュメントを送信するよりも効果的です。


初心者向けの個別のUI



最初に検討した重要な設計変更は、初心者ユーザー向けの個別のUI（「シェル」）です。 デザインはVisual Basicですばやくスケッチされ、ユーザビリティラボでテストされました（図4）。 設計ではユーザーアクションの可能な選択肢が非常に小さなアクションセットに制限されているため、テストは良い結果を示しましたが、テストに参加するユーザーが多いほど、制限が明確になりました。



  1. 初心者向けのシェルが必要な機能を1つしかサポートしていない場合、ユーザーは（少なくとも一時的に）シェルの使用を拒否する必要がありました。
  2. 理論的には、ほとんどのユーザーは、経験を積んだ後、シェルを初心者に任せて標準インターフェースに移動する必要があります。 しかし、初心者のシェルで得た経験は、必ずしも標準のシェルに移行するとは限りません。
  3. 一般的に、初心者向けのシェルは、ユーザーが起動する他のすべてのプログラム（テキストエディター、スプレッドシートなど）とは異なります。 その結果、ユーザーはコンピューターと対話する2つの方法を学習する必要があり、混乱を招きました。

報告書



他のプロジェクトと同様に、実践は真実の基準であるため、要約統計をいくつか示します。



ラボテスト



560人のユーザーを対象に、64段階の実験室テストを実施しました。それらの50％はWindows 3.1の平均的な経験がありました。残りは初心者、上級ユーザー、および他のオペレーティングシステムのユーザーです。これらの数値には、他のチーム（Exchangeメールクライアント、FAXプログラムなど）から受け取ったテストコンポーネントは含まれません。これらのコンポーネントのテストは、約25段階で175人が参加しました。



問題の特定



プロジェクト中の主要なシェルコンポーネントについては、699個の「ユーザビリティレポート」がデータベースに入力されました。これらのうち、148の肯定的な結果と551の問題。問題には、次の3つの重大度レベルのいずれかが割り当てられました。



  • レベル1：ユーザーは、タスクまたは一連のタスクを完了できません。
  • レベル2：ユーザーは、1つまたは一連のタスクを完了する際に重大な困難を経験しますが、それでも引き続き完了することができます。
  • レベル3：ユーザーは、タスクまたは一連のタスクを完了する際にわずかな困難を経験します。


特定された551の問題のうち、15％がレベル1、43％-レベル2および42％-レベル3を受け取りました。



問題解決



このプロジェクトでは、問題に関する5つの解決策を使用しました。



  • 解決されました。チームは問題を修正し、ユーザーのソリューションを正常にテストしました。
  • スケジュール済み。チームは問題の修正プログラムを開発しました。その実装を楽しみにしています。
  • 疑わしい。チームは問題を解決するかどうか、またはその解決が可能かどうかを知りません。
  • 部分的に。チームはソリューションを開発し、満足のいく結果が得られたユーザーでテストされましたが、まだいくつかの疑問が残っています。
  • 決まっていない。チームは問題を解決するつもりはありません。


プロジェクトの終わりまでに、「計画済み」または「未確定」の解決策に関する問題はすべて、他のカテゴリのいずれかに移りました。問題の81％が解決に成功し、8％が解決に「部分的に」残り、11％が未解決のままでした。ほとんどの場合、解決策がない理由は技術的な制限であり、作業スケジュールの制限である場合もありました。



結論



多くの部門の従業員にとって、Windows 95プロジェクトは反復開発、ユーザビリティテスト、問題追跡の最初の経験でした。



反復開発



おそらく、反復開発へのコミットメントの最良の証拠は、Windows 95の元のUIデザインの詳細が最終製品で変更されずに保存されていることです。設計プロセスの最初の段階では、必要な変更の規模と範囲を予想していませんでした。プロトタイプと製品を仕様として使用する反復的な開発、およびユーザーに対する継続的なテストにより、問題を解決するためのさまざまな方法をすばやく探索することができました。



このグループは設計の反復に非常に慣れていたため、プロジェクトの終わり近くで最新の設計作業の一部を完了する必要があるときに急いで感じました。繰り返しの時間はありませんでした。デザインを微調整して再テストする時間がないことに失望しました。



指定プロセス



「プロトタイプまたはコードは仕様です」アプローチは一般にうまく機能しましたが、時間が経つにつれて自然に手順が洗練されていました。たとえば、特定のリリースのすべてのプロトタイプは、それらをインストールおよび起動するための指示とともにパブリックドメイン（すべての従業員向け）に投稿され始めました。



設計チームは元の仕様からドキュメントを編集し、早期のフィードバックのために配布しました。ただし、プロトタイピングおよびユーザビリティテストが開始されると、多くの場合、仕様により読者がプロトタイプに送られて関連情報が取得されました。実際、プロトタイプは仕様のより豊かな形式であり、作成に要する時間が短いことがわかりました。同時に、彼には他の有用なアプリケーション（ユーザビリティテスト、デモなど）があります。プロトタイプは、より意味のあるレビューを刺激します。これは、レビュアーがシステムを理解するために想像力をそれほど結びつける必要がないためです。



ユーザビリティテスト



設計と反復テストにより別々の機能を作成することができましたが、個々の部品を慎重に適合させるための鍵となったのは製品全体のユーザビリティテストでした。既に述べたように、収集されたデータに基づいて、UIの表現と参照セクションのトピックに変更を加えました。そのテストを実施していなかった場合、製品はそれほど効率的で楽しいとは思えませんでした。



問題追跡



修正されたユーザビリティの問題の高い割合は、すべてのチームメンバーの熱心な献身なくしては不可能でした。問題追跡フレームワークにより、プロセス制御が改善され、問題が視界から逃れることがなくなります。ただし、チームが可能な限り最高の品質の製品を作成することを信じていなかった場合、修正は行われませんでした。この信念の主なものは、おそらく最初は正しい結果が得られないこと、そして間違った結果が正しいものと同じくらい製品を作成するために有用で重要であるという理解でした。



「部分的に」および「未決定」とマークされたすべてのチケットは、新しいデータベースに転送されました。これらは、Windowsの次のバージョンを設計するための出発点になりました。プランナーとデザイナーは、この情報を毎日使用し、サポートレポートも分析しました。
