プログラミング言語の歴史：100％「純粋な」C、単一の「プラス」なし





Cプログラミング言語の人気を過大評価することはほとんどできません。特に過去の業績を思い出してください。 おそらく、すべての開発者は少なくともその存在を知っており、最大でプログラムを作成しようとしました。 Cは、C ++、Objective-C、C＃、Javaなどの言語の先駆者です。



Microsoftは、.Netプラットフォーム用のネイティブ言語を開発するためにCのような構文を選択しました。 さらに、多くのオペレーティングシステムはCで記述されています。



もちろん、Cは完璧ではありません。この言語の作成者であるケントンプソンとデニスリッチーは、長い間開発してきました。 Cの標準化は今日まで続いています。 45年以上前から存在し、積極的に使用されています。



多くの場合、1つではなく2つのプログラミング言語-C / C ++に関連付けられています。 ただし、以下の説明は純粋なCに関するものです。



C言語はALGOL言語（ALGorithmic Languageの略）にまでさかのぼります。ALGOL言語は、1958年にスイスのチューリッヒ高等技術学校での会議でヨーロッパおよびアメリカのコンピューター科学者の委員会と共同で作成されました。 この言語は、FORTRAN言語のいくつかの欠点に対する答えであり、それらを修正する試みでした。 さらに、Cの開発はUNIXオペレーティングシステムの作成と密接に関連しており、ケントンプソンとデニスリッチーもこれに取り組みました。



Unix



MACプロジェクト（Multiple Access Computer、Machine-Aided Cognition、Man and Computer）は、1963年にMITでの純粋な研究プロジェクトとして始まりました。



MACプロジェクトの一環として、CTSS（互換性のあるタイムシェアリングシステム）オペレーティングシステムが開発されました。 60年代後半には、BBN、DTSS、JOSS、SDC、Multiplexed Information and Computing Service（MULTICS）など、他のいくつかのタイムシェアリングシステムが作成されました。



Multicsは、MIT、Bell Telephone Laboratories（BTL）、およびGeneral Electric（GE）の共同開発で、GE-645コンピュータ用のタイムシェアリングOSを作成します。 Multicsを実行している最後のコンピューターは、2000年10月31日にシャットダウンされました。



ただし、BTLは1969年初頭にこのプロジェクトから撤退しました。



彼の従業員の何人か（ケン・トンプソン、デニス・リッチー、ストゥ・フェルドマン、ダグ・マキロイ、ボブ・モリス、ジョー・オサンヌ）は自分で仕事を続けたいと思っていました。 トンプソンは、GE-635で宇宙旅行ゲームに取り組みました。 最初にMultics用に作成され、次にGE-635上のGECOSの下でFortranに書き換えられました。 ゲームは太陽系の体をシミュレートし、プレイヤーは惑星または衛星のどこかに船を上陸させなければなりませんでした。



このコンピューターのソフトウェアもハードウェアも、このようなゲームには適していませんでした。 トンプソンは代替手段を探していて、所有者のいないPDP-7の下でゲームを書き直しました。 メモリは8K 18ビットワードのボリュームで、美しいグラフィックスを表示するためのベクターディスプレイプロセッサもありました。

トンプソンとリッチーは、GEでクロスアセンブリを完全に開発し、パンチテープにコードを移植しました。 トンプソンは積極的にそれを好まなかったため、ファイルシステムから始めて、PDP-7のOSを書き始めました。 それがUNIXの誕生です。

トンプソンは、利用可能な手段を使用して、彼の設計に従って設計された快適なコンピューティング環境を作成したいと考えました。 明らかに振り返った彼の意図は、管理の基礎としてのプロセスの概念、ツリーベースのファイルシステム、ユーザープログラムとしてのコマンドの解釈、テキストファイルの簡略化された表示、デバイスへの一般的なアクセスなど、Multicsの革新の多くを吸収しました。


PDP-7 UNIXは、BCPL言語の影響下で作成された高レベル言語Bの始まりでもあります。 デニスリッチーは、Bは型のないCだと言いました。 BCPLは8Kのメモリに収まり、トンプソンによって慎重に再設計されました。 Cで徐々に育ちました

1973年までに、C言語は非常に強力になり、もともとアセンブリ言語PDP-11 / 20で作成されたUNIXカーネルのほとんどはCで書き直されました。 これは、アセンブリ言語以外の言語で記述された最初のオペレーティングシステムカーネルの1つでした。


Cは、UNIXオペレーティングシステムの作成中に取得される「コンパニオン製品」であることがわかります。



Cの祖先



ALGOL-60言語に着想を得て、ケンブリッジ大学の数学研究所は、ロンドン大学のコンピューター部門とともに、1963年にCPL（Combined Programming Language）を作成しました。



CPL言語は複雑であると見なされ、これに対応して、Martin Richardsonは1966年にBCPL言語を作成しました。その主な目的はコンパイラを書くことでした。 現在は実際には使用されていませんが、一時的な移植性のために重要な役割を果たしました。

BCPLは、1970年代初期にOS6オペレーティングシステムを含むいくつかの興味深いプロジェクトで使用され、一部は初期のXerox PARC設計で使用されました。


BCPLはB（B）言語の祖先として機能し、1969年にすでによく知られているAT＆Tベル電話研究所で開発されました。ケントンプソンとデニスリッチーも同様です。



当時の他のオペレーティングシステムと同様に、UNIXはアセンブリ言語で記述されていました。 アセンブラープログラムのデバッグは非常に苦痛です。 Thompsonは、OSのさらなる開発のために高レベルの言語が必要であると判断し、小さな言語Bを考案しました。ThompsonはBCPLを基礎としていました。 言語Bは、型のないCと見なすことができます。



多くの詳細で、 BCPL、B、およびCは構文的に異なりますが、ほとんど同じです。 プログラムは、一連のグローバル宣言と関数の宣言（プロシージャ）で構成されます。 BCPLでは、プロシージャはネストできますが、プロシージャを含むプロシージャで定義された非静的オブジェクトを参照できません。 BとCは、より厳しい制限を導入することでこの制限を回避します。ネストされたプロシージャはまったくありません。 各言語（Bの最も古いバージョンを除く）は、個別のコンパイルをサポートし、名前付きファイルのテキストを含めるためのツールを提供します。



Bの作成中に広まった構文の変更とは対照的に、BCPLの基本的なセマンティクス（その型構造と式評価規則）は変更されていません。 どちらの言語もタイプレスであり、むしろ、「ワード」または「セル」という固定長のビットのセットという単一のデータ型を持っています。 これらの言語のメモリはそのようなセルの配列であり、セルの内容の意味は適用される操作に依存します。 たとえば、「+」演算子は単純にマシンの加算命令を使用してオペランドを追加しますが、他の算術演算もオペランドの意味に関係ありません。



BCPLもBもCも、言語の文字データを区別しません。 文字列を整数ベクトルとみなし、一般的な規則をいくつかの規則で補足します。 BCPLとBの両方で、文字列リテラルは、セルにパックされた文字列文字で初期化された静的領域のアドレスを意味します。



Cの作成方法



1970年、ベル研究所はこのプロジェクトのためにPDP-11コンピューターを取得しました。 BはPDP-11で実行する準備ができていたため、トンプソンはUNIX部分をBに書き換えました。



しかし、モデルBおよびBCPLは、ポインターを操作する際にオーバーヘッドを暗示しています。言語の規則は、ポインターを単語の配列のインデックスとして定義し、ポインターを単語のインデックスにしました。 実行中にポインターを呼び出すたびに、プロセッサーが予期していたバイトのアドレスへのポインターのスケーリングが生成されました。



したがって、文字とバイトのアドレス指定に対応し、今後の浮動小数点計算のハードウェアサポートの準備をするには、入力が必要であることが明らかになりました。



1971年、リッチーはBの拡張バージョンを作成し始めました。最初はNB（New B）と命名しましたが、言語がBと大きく異なるようになったとき、名前はCに変更されました。



抽象オブジェクトを特徴付けるだけでなく、カタログから読み取ることができるビットのセットを記述する構造も必要でした。 コンパイラは、セマンティクスに必要な名前へのポインタをどこに隠すことができますか？ 構造がより抽象化されており、ポインタの場所がどこかに隠れている場合でも、複雑なオブジェクト、おそらく構造を含む配列を含む構造にメモリを割り当てるときにこれらのポインタを正しく初期化する技術的問題をどのように解決しますか任意の深さまで？



解決策は、型のないBCPLと型付きのCの間の進化連鎖の決定的な飛躍でした。リポジトリ内のポインタの実体化を除外し、代わりに式で配列名が言及されたときに生成しました。 今日のCで保持されているルールは、値-配列が式で言及されると、この配列を構成する最初のオブジェクトへのポインターに変換されるということです。



Cをその前身と最も明確に区別する2番目の革新は、このより完全な型構造であり、特に宣言の構文における表現力です。 NBは、基本的なint型とchar型をそれらの配列とそれらへのポインタとともに提供しましたが、それらを組み合わせる他の方法はありませんでした。



汎化が必要でした。任意の型のオブジェクトに対して、そのようなオブジェクトを配列に結合したり、関数から受け取ったり、ポインタにしたりする新しいオブジェクトを記述することができるはずです。




C言語からの画像 ：M.ウェイト、S。プラタ、D。マーティン



このような複合型のオブジェクトについては、その一部であるオブジェクトを指す方法が既にありました。配列のインデックス付け、関数の呼び出し、ポインターを使用した間接アクセス演算子の使用。 同様の理由から、名前を宣言する構文が作成されました。これは、これらの名前が使用される式の構文を反映しています。 だから

int i、* pi、** ppi;
整数、整数へのポインター、および整数へのポインターへのポインターを宣言します。 これらの宣言の構文は、i、* pi、および** ppiがすべて式で使用されるとint型になるという事実を反映しています。 同様に

int f（）、* f（）、（* f）（）;
整数を返す関数、整数へのポインターを返す関数、整数を返す関数へのポインターを宣言します。

int * api [10]、（* pai）[10];
整数へのポインターの配列、整数の配列へのポインターを宣言します。



これらのすべての場合において、変数の宣言は、宣言の先頭にある型を持つ式での使用に似ています。



70年代：「トラブルの時」と誤った方言



1973年までに、この言語はUNIXを書き換えることができるほど安定しました。 Cへの切り替えには、移植性という重要な利点がありました。 Bell Labsの各マシンにCコンパイラを作成することにより、開発チームはUNIXをそれらに移植できました。



C言語の出現に関して、Peter Moylanの著書「The Cに対する事例」では、次のように書いています。 必要だったのは、アセンブラーまたはマシンコードのレベルでしか実装できなかった何かをすることができる言語でした。」



Cは70年代に発展し続けました。 1973-1980年代に、言語は少し成長しました。型構造は、符号なしのlong型、共用体、列挙型を受け取り、構造はクラスオブジェクトの近くになりました（リテラルの表記のみが欠落）。

K＆R は次の言語機能を導入しました 。



•構造（構造データ型）;

K＆R Cは、Cコンパイラがサポートする言語の最も重要な部分と見なされることがよくあります。 ANSI Cのリリース後も、すべてのコンパイラがANSI Cをサポートしているわけではなく、優れたK＆R CコードがANSI Cに当てはまるため、長年、プログラマーがプログラムから最大限の移植性を達成することを望む最低レベルと見なされていました。



人気の高まりとともに、問題が発生しました。 新しいコンパイラーを作成したプログラマーは、K＆Rで説明されている言語を基礎として使用しました。 残念ながら、K＆Rでは言語の一部の機能があいまいに記述されていたため、コンパイラーはしばしばそれらを自由裁量で解釈していました。 さらに、この本は、言語の機能とUNIXオペレーティングシステムの機能を明確に区別しませんでした。

K＆Rの発行後、Xiは発展を続けたという事実によって状況は悪化しました。Xiはそれに新しい機会が追加され、古い機会は切り取られました。 まもなく、言語の包括的で正確かつ現代的な記述が明らかに必要になりました。 そのような標準がなければ、言語の方言が現れ始め、それが移植性を妨害しました-言語の最も強い側面です。



基準



1970年代後半に、C言語はBASICに取って代わり始めました。BASICは当時、マイクロコンピュータプログラミングのリーダーでした。 1980年代には、IBM-PCアーキテクチャに適合し、その人気が大幅に高まりました。







C言語標準の開発は、米国規格協会（ANSI）によって行われました。 1983年に彼と一緒に、X3J11委員会が設立され、標準の開発に従事しました。 標準の最初のバージョンは1989年にリリースされ、C89と呼ばれていました。 1990年に規格にわずかな変更を加えて、ISOに採用されました。 その後、彼はコードISO / IEC 9899：1990で知られるようになりましたが、プログラマーの間では、標準が採用された年に関連付けられた名前C90が修正されました。 標準の最新バージョンはISO / IEC 9899：1999であり、C99としても知られ、2000年に採用されました。







C99標準の革新の中で、変数の宣言の場所に関する規則の変更に注意する価値があります。 新しい変数は、複合ブロックの先頭やグローバルスコープだけでなく、コードの途中で宣言できるようになりました。



C99のいくつかの機能：



•インライン関数（インライン）;

•プログラムテキスト演算子でのローカル変数の宣言（C ++など）。

•long long int（32ビットから64ビット数値への移行を容易にするため）、明示的なブールデータ型_Bool、複素数を表す複素数型などの新しいデータ型。

•可変長配列。

•制限付きポインターのサポート（制限）。

•構造体の名前付き初期化：struct {int x、y、z; } point = {.y = 10、.z = 20、.x = 30};

•C ++から借用した//で始まる単一行コメントのサポート（多くのCコンパイラは、追加として以前にそれらをサポートしていました）。

•snprintfなどのいくつかの新しいライブラリ関数。

•stdint.hなどのいくつかの新しいヘッダーファイル。



C99標準は、現在のすべてのCコンパイラで多かれ少なかれサポートされています。 理想的には、標準を遵守し、ハードウェアおよびシステムに依存する呼び出しを使用しないCで記述されたコードは、ハードウェアベースおよびプラットフォームに依存しないコードになりました。



2007年に、次のC言語標準の作業が開始されました。 2011年12月8日は、C言語の新しい標準を公開しました（ISO / IEC 9899：2011）。 新しい標準の一部の機能は、GCCおよびClangコンパイラで既にサポートされています。



C11の主な機能：



•マルチスレッドのサポート。

•Unicodeサポートの改善。

•一般化されたマクロ（型ジェネリック式、静的オーバーロードを許可）;

•匿名の構造と関連付け（ネストされた構造へのアクセスを簡素化する）。

•オブジェクトの配置の管理。

•静的アサーション。

•危険なgets関数の削除（安全なgets_sを支持）;

•quick_exit関数。

•関数指定子_Noreturn;

•新しい排他的なファイルオープンモード。



11年という標準にもかかわらず、多くのコンパイラはまだC99バージョンさえも完全にはサポートしていません。



Cの批判対象



エントリーのしきい値がかなり高いため、最初のプログラミング言語としてトレーニングで使用するのが難しくなります。 Cでプログラミングするときは、多くの詳細を考慮する必要があります。 「ハッカーの間で生まれることで、多くの場合安全ではない適切なプログラミングスタイルを刺激し、混乱を招くコードの記述を促進します」とウィキペディアは書いています 。



ピーター・モイランは、より深く合理的な批判を表明しました 。 彼は12ページをCの批判に捧げました。 以下にいくつかのフラグメントを示します。



モジュール性の問題
Cでのモジュラープログラミングは可能ですが、プログラマがいくつかのかなり厳格な規則を順守している場合のみです。



•各モジュールには、ヘッダーファイルが1つだけ必要です。 エクスポート可能な関数のプロトタイプ、説明のみを含む必要があります（コメントを除く）。



•ヘッダーファイル内のコメントのみが、このモジュールに関する外部呼び出しプロシージャに認識される必要があります。



•整合性を検証するには、各モジュールが独自のヘッダーファイルをインポートする必要があります。



•別のモジュールから情報をインポートするには、各モジュールに#include行と、実際にインポートされている内容を示すコメントが含まれている必要があります。



•関数プロトタイプは、ヘッダーファイルでのみ使用できます。 （Cには、関数がプロトタイプと同じモジュールに実装されていることを検証するメカニズムがないため、このルールが必要です。したがって、プロトタイプを使用すると、「欠落関数」エラーをマスクできます）。



•モジュール内のグローバル変数、およびヘッダーファイルを介してインポートされた関数以外の関数は、静的と宣言する必要があります。



•「プロトタイプなしの関数呼び出し」というコンパイラ警告を提供する必要があります。 このような警告は常にエラーと見なされる必要があります。



•プログラマは、ヘッダーファイルで指定された各プロトタイプが、同じモジュール内の同じ名前で実装されたプライベート（つまり、通常のC用語では非静的）でない関数に対応することを確認する必要があります。 残念ながら、C言語の性質により、これを自動的に検証することはできません。



•grepユーティリティの使用には疑いを持つ必要があります。 プロトタイプがその場所にない場合、これはおそらく間違いです。

ポインターの問題
データ構造の理論と実践のすべての進歩にもかかわらず、ポインターはプログラマーにとって本当の障害です。 ポインターを使用することは、プログラムのデバッグに費やす時間のかなりの部分を占め、開発を複雑にする問題の大部分を引き起こすのはポインターです。



重要なポインターと重要でないポインターを区別できます。 私たちが理解する上で重要なのは、データ構造の作成と維持に必要なポインターです。



データ構造を実装する必要がない場合、ポインターは重要ではないと見なされます。 典型的なCプログラムでは、重要なポインターよりも多くの重要なポインターがあります。 これには2つの理由があります。



1つ目は、C言語を使用するプログラマーの間では、配列要素を表示する場合など、他のアクセス方法が決してそれらに劣らない場合でも、ポインターを作成することが伝統となっていることです。



2番目の理由は、関数のすべてのパラメーターを値で渡す必要があるC言語の規則です。 Pascal VARパラメーターまたはAda言語のinoutパラメーターに相当するものが必要な場合、唯一の解決策はポインターを渡すことです。 これは主に、Cプログラムの可読性の低さを説明しています。



重要なポインターを入力/出力パラメーターとして渡す必要がある場合、状況は悪化します。 この場合、関数にはポインターへのポインターを渡す必要があります。これにより、経験豊富なプログラマーでも困難が生じます。
