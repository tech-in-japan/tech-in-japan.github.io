時間がないことよりも良い時間：JIRA APIをその場で拡張する

問題を解決するためにアプリケーションで使用可能なAPIが十分ではなく、コードをすぐに変更する可能性がない場合、どうすればよいですか？







この状況での最後の希望は、 java.lang.instrumentパッケージの使用です。 既に実行中のVMでコードを使用してJavaで何をどのように行うことができるかに興味がある人は、catにようこそ。



Habrには、すでにバイトコードの処理に関する記事があります。



  • JVMサービスのJavaエージェント
  • AOPの理論と実践。 Yandexでの方法
  • アスペクト指向プログラミング。 基本


ただし、これらのテクノロジーの使用は、原則として、ロギングまたはその他の単純な機能に限定されます。 しかし、計装を使用してアプリケーションの機能を拡張するために詐欺をしようとしたらどうでしょうか？



この記事では、アプリケーションに新しい機能を追加するために、Javaエージェントアプリケーション（ OSGiとByte Buddyライブラリの両方）を計測する方法を示します。 この記事は主にJIRAで作業する人々にとって興味深いものですが、使用されるアプローチは非常に普遍的であり、他のプラットフォームに適用できます。



挑戦する



したがって、JIRA APIを2、3日調査した後、タスクの作成/変更時にフィールド値の相互検証を実装するのが普通であることを理解しています（1つのフィールドの有効な値が別のフィールドの値に依存する場合）。 アプローチは機能していますが、難しくて便利ではないので、空き時間に計画Bを得るために研究を続けることにしました。



最近のジョーカーは 、 バイトバディライブラリに関するラファエルウィンターハルターの 講演を 特集しました。これは、より便利な高レベルシェルで強力な低レベルバイトコード編集APIをラップします。 このライブラリは現在非常に人気があり、特に最近ではMockitoとHibernateで使用されています。 とりわけ、Rafaelは既にロードされたクラスのByte Buddyで変更する可能性について話しました。



「これは思考だ！」と思い、仕事を始めます。



設計



まず、Rafaelのレポートから、すでにロードされているクラスの変更は、javaエージェントの起動時に使用できるjava.lang.instrument.Instrumentationインターフェイスを使用することでのみ可能であることを思い出してください。 VMの起動時にコマンドラインを使用するか、JDKに付属するプラットフォーム依存のAttach APIを使用してインストールできます。



ここには重要な詳細があります-エージェントは削除できません-そのクラスはVMが終了するまでロードされたままです。



attach APIのサポートに関してJIRAに関しては、ここではJDK上で実行されることを保証できません。さらに、JIRAが実行されるOSを保証することはできません。

次に、JIRA機能を拡張するためのメインユニットはアドオン -ステロイドのバンドルであることを思い出します。 つまり、すべてのロジックは、それが何であれ、アドオンとして設計する必要があります。 これは、システムに変更を加える場合、べき等で切断可能でなければならないという要件を意味します。



これらの制限を考えると、グローバルに2つのタスクがあります。



  • エージェントのインストール：アドオンのインストール中に実行し、二重インストールに対する保護を提供し、LinuxおよびWindows、JDKおよびJREでのエージェントのインストールをサポートする必要があります。

    なぜなら エージェントを削除することはできません。その更新にはアプリケーションの再起動が必要です-これはOSGiの概念に適合しません。 したがって、エージェントの責任を最小限に抑えて、その更新の必要性ができる限り発生しないようにする必要があります。



  • インストルメンテーションの実装：アドオンのインストール中に発生する必要があり、クラスの変換のべき等性を確認し、検証ロジックの拡張性を確認する必要があります。


コンポーネント間で責任を分散させると、次のスキームが得られました。

メインロジックを呼び出した後、必要に応じて結果を変更できる、コードによる初期パラメーターのカスタム検証が呼び出される必要があります。



ByteBuddyは、アイデアを実装するための2つのオプションを提供します。割り込みの支援とアドバイスメカニズムの支援です。 アプローチの違いは、Raphaelのプレゼンテーションのスライドにはっきりと現れています。









Interceptor APIは十分に文書化されており、どのパブリッククラスもその1つとして機能できます 。詳細はこちらをご覧ください 。 Interceptor呼び出しは、元のメソッドの元のバイトコードINSTEADに埋め込まれています。



この方法を使用しようとすると、2つの重大な欠点が見つかりました。



  • 一般に、元のメソッド、さらにはメソッド呼び出しのオブジェクトを取得する機会があります 。 ただし、 ロード済みクラスのシグネチャの変更に関する制限のため、すでにロード済みのクラスをインスツルメントすると、元のメソッドが失われます（同じクラスのプライベートメソッドとして保存できないため）。 したがって、元のロジックを再利用する場合は、自分で再作成する必要があります。



  • なぜなら 実際に別のクラスのメソッドを呼び出す場合、クラスチェーン内のクラス間の可視性を確保する必要があります。 クラスがOSGiコンテナー内でインスツルメントされている場合、可視性に問題はありません。 ただし、この場合、JIRA APIのほとんどのクラスは、OSGiの外部にあるWebappClassLoaderによってロードされます。つまり、Interceptorのメソッドを呼び出そうとすると、当然のClassNotFoundExceptionが発生します。


プロジェクトの作業の過程で、この問題を解決するオプションが生まれましたが、 アプリケーション全体のクラスをロードするロジックに干渉するため、徹底的なテストをせずに使用してスポイラーの下に置くことはお勧めしません。



ブートローダーの問題を解決する
主なアイデアは、WebappClassLoaderの親チェーンを切断し、そこにClassLoaderプロキシを挿入することです。これは、WebappClassLoaderの実際の親にダウンロードを委任する前に、BundleClassLoaderを使用してクラスをロードしようとします。

この場合、WebappClassLoaderを介してOSGiクラスをロードできます。 注意すべき唯一のことは、OSGiを使用してクラスをロードしようとしないことです。そのロードは、OSGiの外部に委任されます。 これは明らかにループと例外につながります。

BundleProxyClassLoaderコード： 



 class BundleProxyClassLoader<T extends BundleWiringImpl.BundleClassLoader> extends ClassLoader { private static final Logger log = LoggerFactory.getLogger(BundleProxyClassLoader.class); private final Set<T> proxies; private final Method loadClass; private final Method shouldDelegate; public BundleProxyClassLoader(ClassLoader parent, T proxy) { super(parent); this.loadClass = getLoadClassMethod(); this.shouldDelegate = getShouldDelegateMethod(); this.proxies = new HashSet<>(); proxies.add(proxy); } private Method getLoadClassMethod() throws IllegalStateException { try { Method loadClass = ClassLoader.class.getDeclaredMethod("loadClass", String.class, boolean.class); loadClass.setAccessible(true); return loadClass; } catch (NoSuchMethodException e) { throw new IllegalStateException("Failed to get loadClass method", e); } } private Method getShouldDelegateMethod() throws IllegalStateException { try { Method shouldDelegate = BundleWiringImpl.class.getDeclaredMethod("shouldBootDelegate", String.class); shouldDelegate.setAccessible(true); return shouldDelegate; } catch (NoSuchMethodException e) { throw new IllegalStateException("Failed to get shouldDelegate method", e); } } @Override public Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException { synchronized (getClassLoadingLock(name)) { log.trace("Trying to find already loaded class {}", name); Class<?> c = findLoadedClass(name); if (c == null) { log.trace("This is new class. Trying to load {} with OSGi", name); c = tryToLoadWithProxies(name, resolve); if (c == null) { log.trace("Failed to load with OSGi. Trying to load {} with parent CL", name); c = super.loadClass(name, resolve); } } if (c == null) { throw new ClassNotFoundException(name); } return c; } } private Class<?> tryToLoadWithProxies(String name, boolean resolve) { for (T proxy : proxies) { try { final String pkgName = Util.getClassPackage(name); //avoid cycle if(!isShouldDelegatePackageLoad(proxy, pkgName)) { log.trace("The load of class {} should not be delegated to OSGI parent, so let's try to load with bundles", name); return (Class<?>) this.loadClass.invoke(proxy, name, resolve); } } catch (ReflectiveOperationException e) { log.trace("Class {} is not found with {}", name, proxy); } } return null; } private boolean isShouldDelegatePackageLoad(T proxy, String pkgName) throws IllegalAccessException, InvocationTargetException { return (boolean)this.shouldDelegate.invoke( FieldUtils.readDeclaredField(proxy, "m_wiring", true), pkgName ); } }

インストルメンテーションを実装するための2番目のオプションは、アドバイスを使用することです。 この方法は文書化されたものよりもずっと悪いです。実際、例はGithubのチケットとStackOverflowへの応答でしか見つかりません。



しかし、すべてがそれほど悪いわけではありません
ここで私たちはラファエルに敬意を表する必要があります-私が見たすべての質問とチケットには詳細な説明と例が提供されているので、それを理解することは難しくありません-これらの作品が実を結び、さらに多くのプロジェクトでバイトバディを見ることを願っています。



デフォルトでは、アドバイスメソッドがクラスコードに埋め込まれているという点で、最初のものと異なります。 私たちにとって、これは次のことを意味します。



  • ClassLoadersを使用しないダンス
  • 元のロジックの保存-元のコードの前または後に特定のアクションのみを実行できます


完璧に聞こえます。元の引数、元のコードの結果（例外を含む）、さらに元のコードの前に機能したアドバイスの結果を取得できるシックなAPIが提供されています。ただし、常に「but」があり、埋め込みにはコードにいくつかの制限があり、埋め込みが可能です。



  • すべての埋め込みコードは1つのメソッドで実行する必要があります
  • メソッドには、埋め込み先のクラスにアクセスできないクラスメソッドへの呼び出しを含めることはできません。そして匿名（さよならラムダ！）
  • 例外ポップアップはサポートされていません-例外はメソッド本体で明示的にスローする必要があります


Byte Buddyのドキュメントにはこれらの制限の説明はありませんでしたが

、アドバイスのスタイルでロジックを記述してみましょう。覚えているように、必要な計測を最小限に抑える必要があります。これは、特定の検証チェックを無視することを意味します-新しいチェックが表示されたときに、validateCreate / validateUpdateが呼び出されたときに実行されるチェックのリストに自動的に追加され、DefaultIssueServiceクラスのコードを変更する必要がないことを確認します

OSGiでは、これは簡単ですが、DefaultIssueServiceはフレームワークの範囲外であり、ここでOSGiテクニックを使用することはできません。



突然、JIRA APIが役立ちます。各アドオンは、このプラグインを検索できる特定のキーを持つPluginクラス（多くの特別な機能を備えたバンドルのラッパー）のオブジェクトとしてJIRAで表されます。



キーはアドオン構成で設定され、プラグインAPIはDefaultIssueServiceと同じクラスローダーでロードされます-したがって、アドバイスでプラグインを呼び出し、このプラグインに付属するクラスをロードすることを妨げるものは何もありませんたとえば、チェックアグリゲーターなどです。



その後、再び標準のcom.atlassian.jira.component.ComponentAccessor＃getOSGiComponentInstanceOfTypeを介してこのクラスのインスタンスを取得できます。
