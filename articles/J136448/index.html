VMwareの物理サーバーと仮想サーバーの移行

昔々、遠くの国にサーバーがありました。 技術仕様によると、その時間は悪くありませんでした-Intel Core Quad Q6600 2.4GHz 8GB RAM Intel DQ965GFマザーボード、3ware7xxx / 8xxx raidコントローラ、およびraid 1アレイの300GBのSATAディスク2個。



そして、RAIDのドライブの1つがこのサーバー上で死ぬことを決定した-そして一度決定すると、彼は死にました。 1つのディスクが死んだ場所を想定するのは自然なことで、2番目のディスクは死ぬ可能性があります-変​​更する必要があります。 はい、ディスクスペースを広げても痛くありません。

どういうわけか、罪を犯して、新しい2TBディスクを半分で購入しました-遠いブルジョアの国々の危機はハードドライブにありました。 サーバーは重要でしたが、しばらくオフにして苦しめることができました-過小評価がありました。



ソフトウェアを更新することにしました...



サーバーにはCentOS 4.5 x64があり、古いサーバーは拷問され、vmwareサーバー2ndがインストールされ、サーバーの下には、ms SQLと一部のプログラム、freebsdおよびsuseを備えたWindows 2003サーバーを備えた3つの仮想マシンが既にありました。 過去数年間、重要なことのうち、ウィンドウのみが必要でしたが、将来はさらに複数の仮想マシンをそこに配置する予定でした。つまり、プラットフォームを変更する必要があるということです。

vSphereハイパーバイザー-およびシンプルなESXiで停止しました。 インターネットを調べたところ、そのような特定のハードウェアにESXiをインストールした経験はありませんでした-Intel 82556DMネットワークカードとRAIDコントローラーは動作しない可能性があります-公式の互換性リストで検索しても良い結果は得られませんでした。 追加のネットワークカードをインストールするのは不便で、時間がかかります。 まあ、あなたは襲撃なしで行うことができます。



一般に、試してみることにしました。 テクニカルサポートに手紙を書いて、新しいハードドライブをインストールしてesxi 5.0をインストールできるようにします。 しばらくして、彼らはesxi 4.1u1をインストールしたと答えました。RAIDは見えませんが、組み込みのICHコントローラーで動作し、両方のハードドライブを見ることができます。 彼らは私たちにパスワードを与えたので、私たちはこの幸福を見に行きました。



行って見ました。 うまくいくようです。 2番目のタスクは、Windows Server 2003仮想マシンをvmwareサーバーからvmware vsphereハイパーバイザー環境に移行することです。 いくつかのオプションを思いつきました-

最初にディスクイメージまたは完成した仮想マシンをcentos / linuxでストレージにアップロードし、Rawデバイスマッピングとして古いディスクを接続し、コンバーターを入れてesxiに変換します。

2つ目は、ext3パーティションをesxiにマウントするために試してみることです-しかし、残念ながらこれは不可能なようです。

3つ目は、SCP / FTP / HTTP経由でコピーし、そこで変換してから、SCPまたはvclient経由でコピーすることです。

4番目は、古いcentos vmwareコンバーター内に配置し、それを所定の場所に変換することです。

5番目 -Windowsで別の仮想マシンを起動し、そこにコンバーターを配置します。



技術サポートがディスクをUSB-SATAアダプターに挿入し、サーバーに接続すると考えていましたが、むき出しのESXiはもちろん、このオプションで何ができるのか理解できませんでした。 それから、彼らは彼らがネイティブのオペレーティングシステムを除いてRAIDアレイが見られるべきであると疑うことを書いた。



Centosにコンバーターをインストールしようとしましたが、インストールされましたが、何らかの理由で、リモートに接続できませんでした。おそらく、標準のポートがビジーだったため、コンバーターのインストール時に指定された非標準ポートが原因でした よく考えてみると、仮想マシン全体が最初にクライアントを介してローカルマシンでクライアントにダウンロードされ、その後、30ギガのトラフィックと神がインターネットを禁止したという一般に、信頼性の低いオプションでした。

さらに、コンバーターはファイルをファイルに変換しません。ホストまたはvCenterへの接続が必ず必要です。



彼らはext3パーティションをesxiにマウントしようとしませんでした-マウントコマンドオプションでは、ext3がファイルシステムのように見えませんでした。



一般に、これを行いました-仮想マシンのファイルをパックし、コンピューターにダウンロードしました。 VMwareコンバーターをインストールし、ローカルのesxiホストに変換しました。 信頼性と互換性のために、sshを介してローカルesxiに接続し、tarを使用してオフにした仮想マシンをアーカイブしました。仮想マシンのディレクトリは/ vmfs / volumes / datastore /にあります。 zパラメーターは、アーカイブを圧縮します。

結果のファイルは、17GBのうち7GBです。



新しいハードドライブをインストールしてesxiを実行するためにテクニカルサポートに書きました。 次に、リモートのesxiプログラムWinSCPに接続し、アーカイブのアップロードを開始しました。 スピードはちょうど殺されました-30kB / s、約3日間のコピー。 後で判明したように、再開はサポートされていません。esxiのt​​arは、一見破損したアーカイブを処理する方法を知りません。 念のため、標準のvclientを使用してアーカイブをリポジトリにアップロードすることにしました-速度は約10倍高く、7時間でアーカイブがダウンロードされました。



次に、sshを使用してリモートesxiに移動し、tar om仮想マシンを展開し、インベントリに追加して起動しました。 vmwareツール、ネットワークカードを更新し、ネットワーク設定を復元しました。



信頼性を向上させるために、Windows 2003サーバーを備えた仮想マシンを作成しました（vmwareメモリ圧縮技術が機能し、使用するリソースが少なくなるように）。 灰色のIPアドレスを持つネットワークカードを両方のマシンにもう1つ追加し、別のvSwitchを作成し、VMKernelを追加し、その中の管理トラフィックをチェックして、同じ灰色の範囲からIPアドレスを割り当てました。 veeamバックアップをインストールして構成しました。 インターネット上の有料トラフィックを削減し、セキュリティを確保するために、veeamは「グレー」IPアドレスを介してesxiに接続し、1台のストレージハードドライブから2台目の仮想マシンのコピーを作成します。



インターネット経由でのコピーの速度が遅い場合、単純なものは最大で30分です。



おそらく、いくつかの移行オプションを見逃しました-移行のためのほぼ同じタスクがまだいくつかあるため、誰かが賢い考えを教えてくれるかもしれません。
