MS SQL Server 2005以降でデータベースメールを設定する

この投稿で説明されているものはすべて、Microsoft SQL Server 2005およびそれ以前のバージョン（2008、2008 R2、Denali）に関連します。 また、この記事のフレームワークでは、「SQL Server」という語句は、Microsoft SQL Serverのみを意味し、MySQL、PostgreSQL、FireBird、および理論的にはSQL Serverと呼ばれる他のDBMSを除きます。



データベースメールとはどのようなものですか？



データベースメールはSQL Server 2005に登場し、SQLMailコンポーネントを置き換えました。 データベースメール（以前のSQLMail）は、SQL Server自体（より正確には、データベースエンジンコンポーネント）によって電子メールメッセージを送信するために使用されます。 データベースメールとSQLMailの基本的な違いは、後者はサードパーティクライアント（Microsoft Outlook）を使用してメッセージを送信することです。これは、SQL Serverがインストールされているマシンにインストールする必要があり、データベースメール自体はSMTPを介してメールサーバーと通信します。



これは何のためですか？



実際、必要なのかどうかは誰もが自分で決めます。 使用例には次のものがあります。

  • マネージャーは、毎朝、昨日の結果をメールで受け取りたいと思っています。販売された製品の数と数、最大のリクエストを行ったクライアント、延滞しているクライアントなどです。 つまり、クエリの結果はデー​​タベースメールを使用して送信できます。
  • データベース管理者は、どのタスク（ジョブ）またはサービスプランがエラーで完了した（またはその逆、正常に完了した）かに関するメールの情報を受け取りたい。
  • データベース管理者は、特定の重大度レベルのエラーに関する、またはサーバーで発生した特定の番号のエラーに関する情報をメールで受け取りたい。
  • データベース管理者は、データベースのサイズがハードドライブのサイズに近いという情報をメールで受け取りたいと考えており、新しい機器を購入するためにボスを振るときです。
  • データベース管理者は、メールで何かを受け取りたいと考えています。
したがって、私の意見では、データベースメールはSQL Serverデータベース管理者にとって最も有用であり、モバイルオペレーターが電子メールからSMSへのサービスを提供していることを考えると（正直なところ、PPSOSには強くありません。絶対に無料です）、通知の非常に迅速な手段にもなります。



何が必要ですか？



そのため、まず、SQL Serverバージョン2005以降が必要です。 SQL Server 2008 R2を使用していますが、この場合、設定に違いはありません。 Microsoftによると、データベースメールは、Express Edition（ proof ）を除くすべてのエディションで使用できますが、 ここでは、このコンポーネントを有効にする方法について詳しく説明します。

2番目にデータベースメールが機能しないのは、SQL Serverがインストールされたマシンからアクセス可能なメールサーバーと、レターの送信元のアカウントです。 私の例では、mail.ruがそのようなサーバーの役割を果たします。

3番目は、SQL Serverのアカウントのsysadminロールのメンバーシップです。これは、このロールのメンバーのみが構成できるためです。



行こう



まず、SQL Server Management Studioを使用してSQL Serverに接続します。 当然、すべての構成手順は事前に定義されたストアドプロシージャを使用して実行できますが、特にこのための便利なウィザードがある場合は、クエリに対して可能なことは何もしようとしません。

[管理]ブランチを開き、データベースメールアイテムを選択し、マウスの右ボタンでそれをつついて、[データベースメールの構成]を選択します。 ウィザードのウェルカム画面は、有用な情報がないため、「このゴミを二度と見せないで」とすぐに確認できます。

そして今、私たちはすでに選択に直面しています。 データベースメールを構成し、データベースメールのプロファイルとアカウントを変更し、プロファイルのセキュリティを変更し、最後にシステム構成を変更します。 以前にデータベースメールを使用したことがないので、これまで変更するものはありません。最初の項目「データベースメールのセットアップ」を選択し、「次へ」をクリックします。



そして、最初の驚きがあります。 SQL Serverは、実際には、データベースメールなどの機能が無効になっていることを明確に報告します。それを有効にしますか？ これがまさにここに来た理由なので、「はい」と答えて次の画面に進みます。



My First DBMail Profileという名前の新しいプロファイルを作成しましょう-未実行のタスク（ジョブ）に関するメールを送信し、1つのアカウントを追加するために使用されます-これのために、「追加」ボタン（まだ空のアカウントのリストの右側）をクリックしますレコード）。



投稿する前に、実験のためにTestDBMail@mail.ruを登録しました。 当然、これらのすべてのフィールドに「自分用に」入力する必要があります。 たとえば、社内のメールサーバーでは、レターを送信するための承認は必要ないため、[匿名認証]を選択できます。 この画面の表示名は、受信したメッセージの[差出人]フィールドに表示されるものです。アカウント名は、SQL Serverのアカウントの内部名です。

この段階では、サーバーへの接続を確認する方法はなく、テストレターを送信する方法もありません。 記入中に間違いがあった場合、検索に時間がかかることがあります。 注意してください。

これで、プロファイルが作成され、アカウントが追加されました。 「次へ」をクリックして、次の予定を確認します。



この画面、特に[プライベートプロファイル]タブを見ると、以前は注意していなかった質問があるはずです。なぜ複数のプロファイルを作成する機能が必要なのですか。



DatabaseMailUserRoleロールに含まれる各msdbユーザーには、プロファイルを割り当てることができます。 または数個。 これを行うには、「アクセス」チェックボックスをインストールします。 デフォルトプロファイル-「はい」が設定されている場合、sp_send_dbmailストアドプロシージャを使用する場合、デフォルトでプロファイル名を省略できます。このユーザーのデフォルトでマークされたプロファイル名はそこで置換されます。

ただし、[パブリックプロファイル]タブに戻り、新しく作成したプロファイルの属性をデフォルト= "はい"およびパブリック= "はい"に設定します。 これで、このプロファイルは、DatabaseMailUserRoleロール（およびsysadminサーバーロールのユーザー）に含まれるすべてのmsdbユーザーが使用できるようになります。 [次へ]をクリックすると、セットアップウィザードの最後から2番目の画面が表示されます。



ここで設定できます：

  1. アカウント再試行の試行-特定のアカウントを使用して電子メールを送信しようとする回数（プロファイルに複数のアカウントを追加できることを覚えていますか？ここで、最初に、SQL Serverは指定された回数だけ優先度1のアカウントに代わって電子メールを送信しようとします。それでも手紙が消えない場合は、優先度の低いアカウントをソートします）
  2. アカウント再試行遅延（秒）-電子メールの送信を再試行する前にSQL Serverが待機する正確な秒数
  3. 最大ファイルサイズ（バイト）-SQL Serverはメッセージに添付ファイルを追加できます。 このパラメーターを使用して、このような添付ファイルのサイズを制限できます。
  4. 禁止されている添付ファイルの拡張子-ユーザーが.vbsファイルを生成して送信できないようにするための添付ファイルの許可の禁止
  5. データベースメール実行可能ファイルの最小有効期間（秒）-データベースメールは個別のDatabaseMail（90-110）.exeファイルです-このパラメーターは、アクティビティがない場合にSQL Serverが実行中のプロセスを「強制終了」する期間を決定します
  6. ログレベル-このパラメーターは、データベースメールの実行時にログに書き込まれる情報の量を決定します。
最後の画面では、作成されるアカウントとプロファイル、および作成されたプロファイルを使用する機会が追加されるユーザーが表示されます。

SQL Serverはすぐにセットアップを完了し、どのアイテムがどのような理由で完了できなかったか（何かが失敗した場合）を表示します。 すべてが正常であることを確認した後、ウィンドウを閉じます。

データベースメールの設定が正しいことを確認します。 もう一度、[管理]に移動し、[データベースメール]を右クリックして、[テストメールの送信]を選択します。



ご覧のとおり、作成したプロファイルはすでにプロファイルとして選択されており、「Subject」および「Message text」フィールドは自動的に入力されています。 当然、これはすべて変更できますが、デフォルト値は私に合っています。 苦労せずに手紙の宛先アドレスを指定するだけです-SQL Serverが手紙を送信するのと同じアドレスを書いています（怠を許してください-1つで十分な場合に2つのメールボックスを作成するのはなぜですか？入ってくる手紙について気にしないでください、彼はそれらを読むことができません。 そして、「テストメールを送信」ボタンをクリックします。

ビンゴ！ 手紙が届きました。 実際、タスクが完了したと想定できます。 しかし、ここでは、メールのチェックに失敗するが、必要な手紙が表示されない貧しい人々のために、少し余談します。

まず、データベースメールを確認します。 SSMSで、[管理]に移動し、[データベースメール]を右クリックして、[データベースメールログ]を選択します。 おそらく、サーバーに接続するためのパラメーターを誤って指定しただけです。これは、奇妙なことに、よくある間違いです。

2番目に確認することは、Service Brokerがmsdbデータベースに含まれていることです。 メッセージの送信に使用されるのは彼です。

新しいクエリを作成して実行します。

SELECT is_broker_enabled

FROM sys.databases

WHERE name = 'msdb'





戻り値が異なる場合、Service Brokerはオフになります。 次のように有効にできます：

ALTER DATABASE msdb SET ENABLE_BROKER





Service Brokerが有効になっている場合、すべての設定が正しく行われます-冗談ではありませんが-メールを再度確認します。 手紙がない？ [テスト電子メールの送信]ボタンをクリックした後に表示されるウィンドウにある[トラブルシューティング]ボタンを開き、Microsoftのヘルプで一般的なエラーを探します。

データベースメールを構成し、テストレターが届いたと仮定します。



そして、それから何？



そして、楽しみが始まります。 これで、最終的にデータベースメールを使用して、SQL Serverから何か問題があるという通知を受け取ることができます（または、その逆も同様です）。

SQL Serverは、タスクが正常に完了した（またはその逆-失敗した）という通知を送信する場合があります。 より正確には、これは実際にタスクを実行するSQL Serverエージェントによって実行できます。 彼がこれを行えるようにするには、次のことを行わなければなりません。

まず、演算子を作成します。 つまり、通知を受け取る人。 これを行うには、SQL Serverエージェントを開き、[演算子]を選択して「新しい演算子」を突く



表示されるウィンドウで、名前（このデータベース管理者がいます）と通知の送信先の電子メールアドレスを入力します（mail.ruで同じ問題のあるメールボックスを指定します）。 [OK]をクリックすると、すぐに演算子が作成されます。

これで、データベースメールプロファイルが構成され、オペレーターが通知を受け取りました。SQLServerエージェントを構成するために残ります。 これを行うには、マウスの右ボタンでそれ（SQL Serverエージェント）をクリックし、[プロパティ]項目を選択して、[アラートシステム]タブに移動します。

ここで、「メールプロファイルを有効にする」



この直後に、メールシステムとメールプロファイルアイテムがアクティブになります。 メールシステムでは、データベースメールを選択するか、SQL Mailの邪悪な愛好家として（ただし、個別に構成する必要があります）、メールプロファイルでは、SQL Serverエージェントがメッセージを送信するために使用するデータベースメールプロファイルを選択できます。 これはパブリックプロファイルでもプライベートでもかまいませんが、この場合、SQL Serverエージェントを実行しているWindowsユーザーは、SQL Serverのログインとmsdbのユーザー（DatabaseMailUserRole）を持つ必要があります。 プロファイルに煩わされたくないので、パブリックプロファイルの選択に同意し、[OK]をクリックします。

それだけです 次に、SQL Serverエージェントを再起動する必要があります。 これは、SSMSまたはSQL Server構成マネージャーから実行できます。 自分で選択してください-あなたにとってより便利な場所から。

再起動後、すべてが正しく構成され、メッセージが実際に到着することを確認します。 これを行うために、既知の存在しないパスにあるデータベースの1つをバックアップしようとしています。

作成したタスクのプロパティに移動し、[通知]タブに移動し、[電子メール]チェックボックスをオンにして、データベース管理者という名前の最近作成した演算子を選択します。 3番目の列は変更せずに残します-「ジョブが失敗したとき」、つまり タスクがエラーで終了した場合にのみ通知を受け取ります。 ただし、さらに2つのオプション（タスクが正常に完了する場合、または少なくとも何らかの形でタスクが完了する場合）をエラーありまたはエラーなしで選択できます。



設定が完了したら、タスクを実行します。
