PHP 5.3.18のセットアップ



初めに、いつものように。 私はconfigs php + apcを尋ねます： yadi.sk/d/0OuPvlBS0t01O



-一般的なPHPセットアップ

一般的に、php.iniファイルには興味深いものは見つかりませんでした。 ただし、次の点に注意する必要があります。



dotdebからパッケージを取得する場合は非常に重要です。

  • date.timezone = Europe / Moscow-タイムゾーンを設定する必要があります。設定しない場合、日付関数は大量の警告を生成します。 理由はわかりませんが、この問題はPHPを新しいバージョンに更新した後、またはDebianのリポジトリとは少し異なる方法でコンパイルされたdotdebリポジトリが原因で発生しました。 この場合、モスクワのタイムゾーンが設定されます。
  • 非常に重要： session.save_path = "/ var / lib / php5" -これはこのように設定する必要があります！ Debianの標準によれば、セッションはそこにあるはずです。 理由はわかりませんが、この変数はdebianアセンブリでは設定されていません。 デフォルトですでに構築されていることがわかります。 Dotdebはデフォルトでセッションを/ tmpディレクトリに書き込みますが、これは完全に間違っています。 サーバー上のパーティションの構成に注意してください：25G /、20Gスワップ、100G / var、〜800G / home。 このパーティション構成は、debian用に設計されました。 この不確実性の結果、本番サーバーでまったく予期しない事故が発生しました！ その結果、顧客が苦しむ可能性があります。 事故は、セクションが突然詰まったために発生しました。 問題が見つかり、すぐに修正されました。 何らかの理由で、おそらくアクセス権が原因で、ガベージセッションコレクタが/ tmpディレクトリで機能しませんでした。 （ UPD from： kamazee ：これは、 古いセッションを構築するdebianの方法によるものです）


パフォーマンスとリソース：

  • max_execution_time = 30-スクリプトの実行時間。デフォルトで残されています。 一方で、スクリプトは無限に実行可能ではないため、制限があります。 一方、通常、無限ループは存在しないため、制約を特に厳しくする必要はありません。 潜在的な無限ループを使用する場合は、この設定をより強くツイストすることをお勧めします
  • memory_limit = 128Mは、スクリプトが消費できるメモリ量です。 PHPは、この設定で指定された以上のメモリを割り当てません。 この設定は、サーバーで実行されるタスクに応じて示されます。 リソースをより正確に計算するには、このパラメーターを可能な限りひねり、ソフトウェアを可能な限り高くするようにしてください。
  • post_max_size = 70M-このパラメーターは、memory_limitに割り当てられたメモリーの合計量よりも小さくする必要があります。 すでに図で示したように、フロントエンドレベルでは、nginxは64Mを超える要求本文を見逃しません。 さらに読み進めるために、このパラメーターは重複して取得されます。
  • upload_max_filesize = 64M-アップロードできる最大ファイルサイズはnginx構成で検証されるため、これを大きくする意味はありません。 空のPOSTでこのような問題が発生しないように、オーバーラップを行うことをお勧めします
  • PHPをセットアップするとき、重複の順序に注意してください：memory_limit-post_max_size-upload_max_filesize。 示された順序で一方が他方より大きくなると、すべてが安定して動作します。
  • max_file_uploads = 20-1つのリクエストで転送できるファイルの数。 このパラメーターは、全員が決定する必要があります。 たとえば、平均的な写真の重さは3メートル以下であると推定されており、一度に20枚程度の写真が表示されます。 サーバーでは、写真/ドキュメントの一括アップロードを行うことができます。 実際にどのように実装すべきかはわかりませんが、理論的には機能するはずです。


PHPの存在を隠す

  • session.name = SESSIDは、秘密のためにPHPSESSIDからSESSIDに変更されました。
  • Exposure_php = Off-サーバー上でPHPが実行されていることを示すヘッダーを無効にします。使用しているものに照らしてはなりません。
  • user_agent = "EXAMPLE-SERVER"は興味深いものです。 たとえば、PHPでスパイダーを作成する場合、これはデータを要求するときにリモートサーバーに送信されるユーザーエージェントヘッダーになります。
  • 残りは変更されません


エラー出力に関するいくつかの言葉（+ PHP 2の存在を隠す）：

  • error_reporting = E_ERROR （E_ERROR | truezemezからのE_WARNINGの修正はコメントを参照）-プロダクションがさまざまな問題の記録を特にロードしないように、このパラメーターをオフにしました。 左、最も重要なもののみ。 一方、高品質のソフトウェアを使用している場合は、最大エラー/問題出力を設定する必要がありますE_ALL | E_STRICT、実際の条件で問題を見つけるため
  • display_errors = Offはウェブマスターによる非常によくある間違いです。画面のエラー表示をオフにしてください。ユーザーはそれを見るのが非常に不快で、クラッカーはごちゃごちゃです！
  • log_errors = On-当然、ログを出力しない場合、ログを書き込む必要があります！
  • 残りは変更されません


-APCキャッシングの設定（バージョン3.1.9）：

APCはオペコードをキャッシュし、さらにデータをキャッシュできます。 Yiiフレームワークが推奨しているため、APCを使用しています。 Yiiはこのキャッシュを操作するためのクラスをすでに作成しています。 上記の理由により、他のキャッシュを検討または比較しませんでした。



デフォルトのファイル構成は空で、すべての値がデフォルトに設定されています。 特に、32Mはキャッシングに割り当てられています-コードには十分で、データはありません。

指定したファイル構成では、次の点に注意する必要があります。

  • apc.shm_size = 512M -debian / dotdebでは、APCはMMAPサポートでコンパイルされています。 したがって、apc.shm_segmentsパラメーターは考慮されないため、この設定のみを使用する必要があります
  • apc.file_update_protection = 3-このパラメーターが大きいほど、APCがコードファイルが変更されたかどうかをチェックする時間が短くなります。 秒単位で示されます。 サーバー上のコードをめったに更新しない場合は、このパラメーターを設定できます60
  • デフォルトで残された他の設定があります。


-メール（）関数の設定：

箱から出して、メール（）関数は動作しないはずです。 動作するには、適切に機能するSMTPメールサーバーが必要です。 前の記事では、このようなサーバーをインストールしました。

セットアップで次の項目を設定する価値があります。

  • sendmail_path = / usr / sbin / sendmail -t -i -fno-reply@example.com-このコマンドを使用して、送信済みヘッダーを正しく構成します。 スパムを回避するには、正しいヘッダーが必要です。 正しいヘッダーに関しては、私の記事を読んでください： PHPスクリプト用のメール送信サーバーの適切な設定、mail（）関数の設定（もちろんこの記事は恐ろしい混乱ですが、それでも何が起こるかを説明しています。
  • mail.add_x_header =オフ -いつものように、必要のないもので輝かないでください




MySQL 5.5.28を構成する



ドキュメントを読んでいると、ファンタジーレベルでの設定の癖に出会いました。 特定の設定および特定のハードウェアでデータベースから最大30％の最大パフォーマンスを引き出すテストを実行します。

これはありません。 常識とドキュメントに従ってください。



構成へようこそ： yadi.sk/d/T1ov6qAF0t82y



構成ファイルの構造：

  • /my.cnf-ディレクトリの基本設定が書き込まれ、他の構成ファイルをロードするメイン構成ファイル自体
  • /conf.d/.keepme-削除すべきではないことを示す簡単な名前。 デフォルトのままでした。
  • /conf.d/mysqld_base_perfomance.cnf-データベース全体のパフォーマンスの調整。 これらの設定は、使用されるストレージの種類には関連付けられていません。
  • /conf.d/mysqld_common.cnf-グループ化できなかった一般的な設定。
  • /conf.d/mysqld_innodb.cnf-InnoDBリポジトリ設定
  • /conf.d/mysqld_log.cnf-ログ設定
  • /conf.d/mysqld_myisam.cnf-MyISAMリポジトリ設定
  • /conf.d/mysqld_query_cache.cnf-クエリキャッシングの設定
  • /conf.d/mysqld_thread.cnf-スレッド設定


my.cnfファイルでは、次の設定に注意してください

  • max_connections = 300-各PHPプロセスが2つの独立した接続を作成できるという計算から、150 * 2に等しい接続数を増やしました。
  • max_allowed_pa​​cket = 2M-送信パケットのサイズを増やしました
  • datadir = / home / mysql-設定中に、mysqlを除く他のユーザーのアクセス保護を備えたmysqlの/ homeに個別のディレクトリが作成されました。 これは、/ var / lib / mysql（間違えなければ）から/ homeパーティションにデータを転送するために行われます。これは、すべてのデータが保存されるサーバー上の最大のディスクパーティションです。


-変数を表示

データベースが実行する操作を確認するために、 SHOW VARIABLESが作成されました。 これにより、入力/出力操作の数、キャッシュ内のヒット数、ファイルのソート数などについて学ぶことができます。 よく知っておくことをお勧めします。

パフォーマンスに影響を与える各設定の隣の構成では、 SHOW VARIABLESのパラメーターを調べる必要があるリンクが示されます（参照）。 その他の詳細については、ドキュメントを参照してください。



注意すべき設定：

  • /conf.d/mysqld_thread.cnf thread_cache_size / thread_concurrency-コアの数によって設定されます。
  • mysqld_base_perfomance.cnf tmp_table_size = 8M-一時ファイルのソートが多い場合、このパラメーターを増やす必要があります。 サーバーの機能を覚えておく必要があることを忘れないでください。 私たちの場合、一時テーブルでのソートに費やすピーク時の8M = 2.4Gでの300接続です。
  • mysqld_query_cache.cnf query_cache_size = 256M-他のデータベース設定とは異なり、クエリキャッシュのサイズを乱用しないでください。 256Mで十分です-デフォルト値の8倍。 ドキュメントの詳細
  • mysqld_myisam.cnf concurrent_insert = ALWAYS -MyISAMに競争力のある挿入があることがわかったとき、私はうれしく驚きました。このテーブルに対する私の態度を再考しました。
  • mysqld_myisam.cnf key_buffer_size = 256M-多くのMyISAMテーブルを使用する場合、このパラメーターは大きくする必要があります。そうでない場合は、安心できるように256に増やすことをお勧めします。
  • 重要： mysqld_innodb.cnf innodb_buffer_pool_size = 5Gは最も重要なパラメーターです。デフォルトでは、InnoDBテーブルは小さなサーバーでも動作するように設定されていません。 このパラメーターを構成しない場合、ディスクサブシステムに激しい負荷がかかります。 理想的には、このパラメーターは、すべてのデータベースの合計サイズよりも少し大きくすることをお勧めします。
  • mysqld_innodb.cnf innodb_log_buffer_size = 32Mは、InnoDBテーブルログの場合のみ同じです。 （これらはトランザクションログであり、mysqld_log.cnfファイルで設定された通常のログではありません）
  • 読むのは非常に興味深いです：mysqld_innodb.cnf innodb_buffer_pool_instances = 1-フィクションの危機にthisして、このパラメーターの全面的な戦いがあります。 適切に接続されたdimitrik.free.fr/blog/archives/2012/10/mysql-performance-innodb-buffer-pool-instances-in-56.htmlの情報を見つけることができるリンクを次に示します。 最後まで彼らが何について書いているのか理解できませんでした。 私はそれを読み、チャートを見て、リスクを冒さずに1を離れました。




他のすべてのパラメーターは、構成ファイルで見つけることができます。
