無秩序とデータストレージ：SANの未来はありますか

翻訳者注： ZDnetの専門家であるRobin Harrisが「 SSDが古くなった理由 」に関する資料を公開しました（適応バージョンはまもなくブログでリリースされます）。 この出版物のもう1人の専門家であるJason Perlowは、この記事に基づいて、ネットワークストレージシステム（SANおよびNAS）のマイナス面と展望について議論することを決定しました。



以下に示す資料には、かなりの数の技術用語が含まれており、それらの翻訳によって不正確さが生じる場合があります。 翻訳のタイプミス、間違い、不正確さに気づいたら、私たちに連絡してください。すぐに修正します。







現在使用中のSASおよびSATAインターフェースは、数十年にわたって開発されてきた進化したSCSIおよびATAテクノロジーに過ぎないことを理解することが重要です。 それ以来、サーバーテクノロジーははるかに進歩し、データ処理アプリケーションの必要性が大幅に高まっています。



明らかに、業界の代表者は、既存のストレージテクノロジーをさらにアップグレードすることを検討する必要があります。 ただし、SSDからより多くのIOPSを圧縮するタスクは科学により関連しており、実用的な観点からは、データストレージの総所有コストの問題がはるかに重要です。



大企業では、データストレージの整理の問題がますます深刻になってきています。特に規制業界では、法規制に従ってデータを長期間保存する必要があります。 さらに、この情報には簡単にアクセスできる必要があります。これにより、アーカイブテクノロジーの使用が複雑になります。



率直に言って、SANとNASには多額の費用がかかり、企業のデータセンターのハードウェアコストの3分の1がかかります。 はい、そのような製品のメーカーは非常に高い評価を得ているため、製品は最も重要なデータを保存するために使用されますが、正直なところ-冷蔵庫ほどの大きさのディスクで満たされたこれらのボックスには魔法はありません



すべての同じSATA、SAS、さまざまなインターフェイス、コントローラー、および論理デバイス番号（LUN）の作成を担当する特別なソフトウェアの内部。 これらのコントローラーのほとんどは、UNIXのプロプライエタリバージョンまたはBSDバリアント上で実行されます。ユーザーにはこのいずれも表示されません。 彼にとって、LUNを作成した後のSANまたはNASは真のブラックボックスです。



この高価な狂気は終わらせることができますが、これには大企業の代表者による非標準で創造的なアプローチが必要です-つまり、スケーラブルなサービスプロバイダー（ハイパースケール）と同じようにストレージを扱う必要があります。 スケーリングについて話すとき、Amazon Web Services、Microsoft Azure、またはGoogle Computer Engineを意味します。



これらの企業は、EMCおよびNetAppハードウェアのみを使用してクラウドストレージシステムを作成できたと思いますか？



もちろん、そうではありません。 SANやNASの代わりに、そのような企業はJBOD-「 単なるディスクの束」のアレイを使用します。 彼らは、一般に入手可能で安価な機器、JBOD、および経験豊富なエンジニアの助けを借りて、より安価なインフラストラクチャを作成することができました。



このようなインフラストラクチャを構築するための標準はありません。さらに、このプロセスは、クラスターストレージを作成するために光ファイバーを使用する必要性について歴史的に意見があるという事実によって複雑になります。



ただし、10ギガビットおよび40ギガビットイーサネットネットワーク、 RDMAイーサネットカード、 SMB 3.0ネットワークアクセスプロトコルの出現により、状況は急速に変化しています。



概念は非常に単純です-組織は単純にファイルサーバーの多数の「ヘッド」を既存のスイッチドイーサネットインフラストラクチャに接続し、SAS 15KとSSDで構成される多くのJBOD（たとえば、DataOn、Dell、またはSupermicro）を階層構成で接続して使用しますSASクラスタのこれらの「ヘッド」を使用します。







次に、これらのアンカーファイルサーバーは、SMB 3.0を使用してデータへのアクセスを提供する仮想化システムまたは物理システムに接続されます。 このようなシステムの弾力性は、ストレージを管理するOSに依存し、SANやNASで行われているようなコントローラーに組み込まれた一部の秘密ソフトウェアには依存しません。



上の画像で説明されているスクリプトは、組み込みのWindows Server 2012 R2に付属のMicrosoft スケールアウトファイルサーバー （SoFS）を使用し、ストレージスペースコンポーネントを使用して動作します。 ここで使用されるハードウェアは、Dell R820ラックサーバーおよびMellanox RDMAカードと組み合わせたDataOn DNS-1660Dです。



上記の構成では、毎秒100万IOPSを超える一定の速度を実現できます。



デルは、 MD1220 PowerVaultを使用したJBOD SoFSアレイの構築に関する論文を公開しています 。 一般に、JBOD、SASおよび10 Gb / sイーサネット接続を使用する一般的なx86アーキテクチャハードウェアの任意の組み合わせが機能します。



Microsoftに加えて、JBODに基づくアーキテクチャの構築に関与する他のベンダーがあります-たとえば、 Nexenta （SolarisのZFSに基づく）、Linuxの場合、Red Hat Resilient Storageコンポーネントを含むHA-LVMおよびGFS / GFS2があります 。 Ubuntu Serverに相当するものはClusterStackと呼ばれます。



ここでの結論は、SANおよびNASソリューションの信頼性と検証にもかかわらず、ストレージの最高のパフォーマンスと「復元力」を確保するというヘゲモニーが終わりに近づき、他のツールがすぐに広く配布されるということです。



近い将来、増え続けるデータのストレージを節約したい企業幹部は、クラウドプロバイダーが使用する方法に頼ることができます-JBODとソフトウェア定義ストレージ（ これについては、 Habré に関する記事があります ）を最新のサーバーOSに組み込み、また、クラウドにバックアップを保存できるアプリケーションにクラウド統合ストレージ（ CIS ）を使用します。
