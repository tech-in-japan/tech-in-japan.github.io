はじめに



アジャイルの哲学の普及により、ロシアのITスペシャリストは毎年、開発チームの製品のカスタマイズと管理の分野でますます多くの専門知識と能力を獲得しています。 ただし、最も古い、生産的でアクセスしやすいバージョンのJira Data Centerを使用すると、依然として多くの疑問が生じます。 この投稿では、実際に適用するJira DataCenterの原則とメカニズムについて説明します。 Jiraクラスターの構造についての話から始めます。



Jira DataCenterとは何ですか？



Jira DataCenterは基本的にサーバーバージョンですが、共通のデータベースと共有インデックスを使用する機能を備えています。



Jira DataCenter自体は、製品およびアプリケーションとして、フォールトトレランスと負荷分散を提供しないことを理解することが重要です。 モジュールとシステムはこれに責任があり、アトラシアン製品とは関係ありません。



言い換えれば、アトラシアンはクラスターでの作業をサポートしていますが、クラスター化自体は外部手段によって実装されており、その選択は非常に豊富です。



製品の詳細な説明は、アトラシアンのWebサイトで入手できます。



いくつかの構築オプションがあります。



1.独自のインフラストラクチャ上

2. Amazon Cloud（AWS）で

3. MSのクラウド（Azure）



この記事では、独自のインフラストラクチャのソリューションについて説明します。



Jira DataCenterはどのような問題を解決しますか？



Jira Data Centerは、次の目標の達成を支援します。



  1. フォールトトレランスの実装。
  2. 高負荷下での安定した動作を保証します。 Jira Sizing Guideによると、高負荷とは大規模/エンタープライズ規模のインスタンスを指します。
  3. メンテナンスが必要な場合の継続的な運用の確保。 この時点で、私は別に住みます。 多くの場合、アプリケーションを更新する必要がありますが、すべての企業がこれを迅速かつ目に見えないように行う能力を持っているわけではありません。 この問題は、クラスタリングと、いわゆるゼロダウンタイム更新スキームを使用することで解決されます。


これらの問題は、クラスタリングとスケーラブルなアーキテクチャによって対処されます。



Jira DataCenterのコンポーネントは何ですか？



次の図に示すように、Jira DataCenterクラスターはいくつかの専用マシンのコレクションです。





図1. Jira Data Centerのアーキテクチャ



  1. アプリケーションノード（アプリケーションノードまたはクラスターノード）。 すべてのワークロードとリクエストを受け入れて処理します。 ノードの役割は、同一のコンテンツとインストールされたアプリケーション、およびマウントされた共有ファイルシステムを備えた通常のサーバーによって果たされます。
  2. ファイルのインポート/エクスポート、プラグイン、キャッシュなどの標準機能を備えたファイルシステム（共有ファイルシステム）。 ファイルサーバーは、共有フォルダーまたはリソースが作成される別個のサーバーでもあり、ノードにマウントされて共有ファイルに使用されます。
  3. 共有データベース この場合、データベースサーバーは別のサーバーであり、MS SQL、PostgreSQL、MySQL、Oracle上に構築できます。
  4. ロードバランサー ユーザーリクエストを配信し、ノードに配信します。そのうちの1つが失敗した場合、バランサーはリクエストをほぼ瞬時に他のノードにリダイレクトします。 その作業のおかげで、ユーザーは1つのノードの障害に気付くことすらありません。 バランサーの動作については、別途説明します。


Jira Data Centerクラスタートポロジ



JDCでクラスターを構築するための基本原則を示します。



  • Jiraインスタンスは共通のデータベースを共有します。
  • Luceneインデックスはリアルタイムで複製され、インスタンスごとにローカルに保存されます。
  • 添付ファイルは共通のリポジトリに保存されます。
  • Jiraインスタンスはキャッシュの一貫性を監視します。
  • いつでも複数のインスタンスを同時にアクティブにできます。
  • 利用可能なクラスターロック。
  • バランサーはアクティブなノードのみにリクエストをリダイレクトするように設定されていますが、非アクティブなノードにリクエストを転送することはできません。また、1つのノードにすべてのセッションをアドレスすることもできません。


すべてのノードはアクティブとパッシブに分けられます。 アクティブノードは、次の点で異なります。



  • リクエストを処理する
  • バックグラウンドのプロセスとタスクを実行する
  • スケジュールされたタスクは、1つ以上で構成できます。
  • すべての実用的なシナリオで、状況は標準のJira Serverを使用しているように見えます。 したがって、パッシブノードは要求を処理せず、タスクを実行しませんが、短期的なワークロード（システムの起動時、プラグインのロードやインデックス作成など）を引き受ける役割を果たします。

フォワードおよびリバースプロキシ



ロードバランサーは、直接プロキシとしてもリバースプロキシとしても機能します。 この違いは、stackoverflowに関するこのコメントの著者によって十分に説明されています。



1.「転送プロキシ」（転送プロキシ）。 この場合のプロキシイベントは、「直接プロキシ」が元の要求者に代わって別のWebサイトからデータを取得することです。 例として、インターネットに接続されている3台のコンピューターのリストを示します。



X =インターネット上のコンピューターまたはクライアントコンピューター

Y =プロキシWebサイト、proxy.example.org

Z =アクセスしたいウェブサイトはwww.example.netです

通常、X-> Zから直接接続できます。ただし、一部のシナリオでは、Xに代わってY-> Zを使用した方がよい場合があります。これは、X-> Y-> Zです。



2.「逆プロキシ」（逆プロキシ）。 同じ状況を想像してください。サイトYでのみリバースプロキシが構成されています。 通常、X-> Zから直接接続できますが、シナリオによっては、Zの管理者が直接アクセスを制限または禁止し、最初に訪問者にYを通過させる方が良い場合があります。 X-> Y->Z。

このケースは、ユーザーXがデータをYと交換していることをユーザーXが認識しているため、ユーザーXがZにアクセスしていることを知らないという点で「直接プロキシ」と異なります。サーバーZはクライアントには見えず、外部プロキシサーバーYのみが外部に見える。 リバースプロキシは、クライアント側の構成を必要としません。 クライアントXは、Y（X-> Y）とのみ対話していると考えていますが、実際には、Yは接続全体をリダイレクトします（X-> Y-> Z）。



次に、ソフトウェアバランサーの使用を検討します。



どのソフトウェアバランサーを選択しますか？



私たちの経験では、スティッキセッションモードをサポートし、また最もよく使用されるWebサーバーの1つであるため、ソフトウェアバランサーの最適な選択肢はNginxです。



スティッキーセッションは、クライアント要求が同じグループサーバーに送信される負荷分散方法です。 Nginxには、Cookieを使用してバランスを取るスティッキーメソッドがありますが、商用バージョンのみです。 しかし、外部モジュールの使用という無料の方法があります。



このモジュールはCookieを作成するため、各ブラウザーが一意になります。 次に、Cookieを使用してリクエストを同じサーバーにリダイレクトします。 Cookieが存在しない場合（最初の要求など）、サーバーはランダムに選択されます。

このリンクとこのHabrapostでスティッキーメソッドの詳細を読むことができます。

Jira DataCenterクラスターを作成する手順



クラスタリングの場合、Jiraがインストールされた既存のインスタンスまたは新しいインスタンスを使用できます。 この例では、異なるOSへの新しいインスタンスのインストールについて説明します（システムの汎用性を示すため）。



1.データベースサーバーから始めましょう。 既存のものと新しいものの両方を使用できます。 繰り返しますが、説明のために、Windows Server 2016 + PostgreSQL 9.4が選択されました。 OSをインストールし、PGサーバーをインストールし、PG Adminをインストールし、ユーザーとデータベースを追加します。



2. Ubuntu 16.04 LTS OSで最初のノードを作成します。 必要なパッケージ、リポジトリの更新をインストールします。



3. Jira DataCenterをダウンロードしてインストールし、データベースを起動して構成します（念のため、Atlassianに詳細なガイドがあります ）。



4. Jiraをオフにし、ノードをオフにします。

サービスジラストップ



5.さらに操作するには、Jira自動実行を一時的に無効にすることをお勧めします。

update-rc.d -f jira remove



6.スイッチをオフにしたノードのクローンを作成します。



7.最初のノードを起動し、Jiraをオフにします（デフォルトでは、インストール後のJiraは自動実行に設定されます）。



8. 2番目のノードを実行し、Jiraをオフにします。



9.バランサー用に個別のインスタンスを作成します。 Ubuntu 16.04を選んだのは、 それは非常に高速で単純であり、ライセンスの形で追加費用を必要としません。



10. nginxをインストールします（この例では、バージョン1.13.4が使用されています）。



11. nginx-sticky-module-ngをダウンロードして解凍します。

git clone bitbucket.org/nginx-goodies/nginx-sticky-module-ng.git



12.モジュールを再コンパイルおよび追加するために、nginxを準備します。



13. nginx-sticky-module-ngモジュールでnginxをコンパイルします。 私の場合、コンパイル行は次のようになりました。

./configure --prefix = / etc / nginx --sbin-path = / usr / sbin / nginx --modules-path = / usr / lib / nginx / modules --conf-path = / etc / nginx / nginx。 conf --error-log-path = / var / log / nginx / error.log --http-log-path = / var / log / nginx / access.log --pid-path = / var / run / nginx pid --lock-path = / var / run / nginx.lock --http-client-body-temp-path = / var / cache / nginx / client_temp --http-proxy-temp-path = / var / cache / nginx / proxy_temp --http-fastcgi-temp-path = / var / cache / nginx / fastcgi_temp --http-uwsgi-temp-path = / var / cache / nginx / uwsgi_temp --http-scgi-temp-path = / var / cache / nginx / scgi_temp --user = nginx --group = nginx --with-compat --with-file-aio --with-threads --with-http_addition_module --with-http_auth_request_module --with-http_dav_module- -with-http_flv_module --with-http_gunzip_module --with-http_gzip_static_module --with-http_mp4_module --with-http_random_index_module --with-http_realip_module --with-http_secure_link_module --with-http_sl_module_with_module -with-http_sub_module --with-ht tp_v2_module --with-mail --with-mail_ssl_module --with-stream --with-stream_realip_module --with-stream_ssl_module --with-stream_ssl_preread_module --with-cc-opt = '-g -O2 -fstack-protector- param = ssp-buffer-size = 4 -Wformat -Werror = format-security -Wp、-D_FORTIFY_SOURCE = 2 -fPIC '--with-ld-opt ='-Wl、-Bsymbolic-functions -Wl、-z、relro -Wl、-z、now -Wl、-as-needed -pie '--add-module = / usr / local / src / nginx-sticky-module-ng



14. /etc/nginx/nginx.confファイルを見つけて、.bakにコピーし、nginxをリバースプロキシモードに設定します。



15.次に、ファイルサーバー（できればフォールトトレラント）も必要です。 たとえば、 NFSボールを作成したWindowsサーバーを選択しました 。



16.各ノードで、NFSサポート用のパッケージをインストールします。

apt-get install nfs-common



17. / media / jiraフォルダーを作成して実行します。

chmod -R 0777 / media / Jira



18. NFSボールを共有ボールとしてマウントします（ルートフォルダーではなく、/ media / jiraなどにマウントすることが必須です）-すべてのノード



19.1。 さらに、手動マウント（単一）を実行することもできます。

sudo mount -t nfs -O uid = 1000、iocharset = utf-8 xx.xx.xx.xx：/ jira / media / jira

ここで、xx.xx.xx.xxは、NFSボールを持つサーバーのIPアドレスです。



19.2。 または、すぐに自動マウント（OS起動時）：

mcedit / etc / fstab

最後に次の行を追加する必要があります。

192.168.7.239:/jira / media / jira nfs user、rw 0 0

次に、保存して終了します。



20. IDを割り当てます：最初のnode1ノード、2番目のnode2ノードなど。

＃このIDはクラスター全体で一意である必要があります

jira.node.id = node1

＃すべてのJiraノードの共有ホームディレクトリの場所

jira.shared.home = / media / jira



21.最初のノードでjiraを実行します

サービスジラスタート

チェック：

システム->システム情報->クラスターONとノード番号を探します。



22. nginxバランシングのセットアップ



23.以来 以前は、ノードでJira自動起動を無効にしてから、次のコマンドで有効にできます。

update-rc.d -f jira enable



24.クラスターの動作を確認し、必要に応じてノードを追加します。



クラスターの開始順序



1.共有ファイルシステムサーバーを有効にする

2.ロードバランサーを有効にする

3. node1を有効にします

4. node2を有効にします

5. ...



クラスター停止順序



1. service Jira stopコマンドを使用して、両方のノードでJiraを停止します

2.ノード2をオフにします

3.ノード1をオフにします

4.ロードバランサーをオフにします

5.ファイルシステムサーバーの電源を切る
