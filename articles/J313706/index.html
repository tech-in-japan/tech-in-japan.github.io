Xcodeビルドシステム：高度なチュートリアル



Xcodeのプロジェクト構成は、宇宙船のコントロールパネルのように見えます。 多くの場合、人々はアセンブリシステムの仕組みを理解していますが、用語を混同しています。 この記事では、プロジェクトの構造、ターゲット、構成設定、ワークスペースについて説明します。 行こう！



トピックが新鮮であると言っていない-Appleは2011年にwwdcでそれをカバーしました 。 そしてまた今年 。

したがって、新鮮さの記事を追加するために、ファッショナブルな技術を例えとして使用します。 たとえば、Xcodeが食べ物を印刷できる3Dプリンターであるとします。 そして、私たちはフードサービスアウトレットのキッチンとして使用することにしました。



順番に考えてみましょう。以下で説明します。

  1. プロジェクト
  2. 対象
  3. ビルド設定
  4. ビルド構成

プロジェクト



たとえば、人々にピザを食べさせたいので、3DプリンターPizza.project用の特別なモジュールを作成します。このモジュールには、抽象的なピザの調理方法に関する情報（フォーム、調理時間、使用可能な材料など）が含まれています。







ドキュメント：

プロジェクトは、ソフトウェア製品の構築に必要なファイルとリソースのリポジトリです。 プロジェクトには、製品の構築に使用されるすべての要素が含まれ、これらの要素間の関係をサポートします。 プロジェクトは、プロジェクト内のすべてのターゲットのデフォルト設定を定義します。


プロジェクト機能：


  • ソースファイル、ライブラリとフレームワーク、画像、インターフェイスビルダーファイル（xibki、ストーリーボード）へのリンクが含まれています。
  • Project Navigatorでファイルを構造化するためのグループについて説明します（これらは、ディスク上のプロジェクトの構造とは関係のない紛らわしい疑似フォルダーです）。
  • アセンブリ構成について説明します（詳細は後ほど）
  • ターゲットの説明が含まれています（詳細は後ほど）


ボンネットの下
拡張子が.xcodeprojのファイルは、フェーダーで開いて、Projectが割り当てられた機能を実装する方法を確認できるフォルダーです。

このように：



内部には次のものがあります。

  1. .Pbxprojファイル。 まれなiOS開発者は、それをフリーズする必要はありませんでした。 これは、プロジェクトに関するほとんどすべての情報が含まれているため、これが絶えず変化している巨大なファイルであることを知っています。 ソースとリソースへのリンク、ファイルグループの説明、ターゲットのリスト、ターゲットの依存関係、プロジェクトアセンブリ構成、ターゲットアセンブリ構成など。 一般に、このファイルの機能と内部構造の説明は、別の記事に値します。
  2. Xcuserdataフォルダー。 プロジェクトの状態（開いているファイル、選択されているターゲットなど）、ブレークポイント、現在のXcodeユーザーのスキームに関する情報を保存します。 まだ行っていない場合は、このフォルダーを.gitignorファイルに追加する必要があります。
  3. Xcshareddataフォルダー。 名前から推測できるように、フォルダーは、アプリケーション開発仲間と共有したい設定（スキームとブレークポイント）を保存するという点で、前のものとは異なります。

    • ブレークポイントを共有するには：ブレークポイントナビゲータに移動し、共有するブレークポイントを右クリックして、[ブレークポイントの共有]をクリックします。
    • スキームを共有するには、[製品]-> [スキーム]-> [スキームの管理]に移動し、共有するスキームの横にある[共有]列をチェックします。


  4. .Xcworkspaceファイル（内部ワークスペース）。




対象



誰も私たちから抽象的なピザを買わないので、特定のピザのいくつかのモデルを3dプリンターに追加でアップロードする必要があります。これは、Pizza.projectに基づいて、特定のピザの印刷方法（使用可能な原料の種類、温度どんな大きさであるべきかを調理し続ける）。 そのようなモデルがターゲットになります。







ドキュメント：

ターゲットは、どの製品をビルドするかを正確に決定し、一連のワークスペースファイルまたはプロジェクトからプロジェクトをビルドするための指示を含みます。


名前が示すように、ターゲットは、私たちの仕事で私たちが向かっている目標を表します。 4つの軸のうちの1つ、ライブラリまたはフレームワーク、拡張機能またはウィジェット、一連のテストのいずれかの下にあるアプリケーションです。 Swiftでは、各Xcodeターゲットは個別のモジュールとして提示され、パブリックAPIを介してインポートおよびアクセスできます。 最も重要なのは、同じソースとリソースを使用できる複数のターゲットを同じコードベースに配置できることです。 最終製品のタイプと、それが動作するオペレーティングシステムに応じて、ターゲットの責任は異なります。

ターゲットの機能：


  • 製品のビルドに使用されるソースとリソースのセットを定義します。 プロジェクトに特定のリソースまたはソースコードを含むファイルを含めることができます（ビルドフェーズの[ソースのコンパイル]セクション、またはファイルインスペクターで目的のファイルを選択）。



  • プロジェクトのビルド設定を保存します。 他の製品を組み立てるための設定を変更せずに、製品の一部の設定を変更できます。



  • アセンブリのフェーズについて説明します。 これは、製品を取得するためにxcodeが実行する一連の順序付けられたアクション（バンドルへのファイルのコピー、フレームワークのリンク、リソースおよびソースコードの作成など）です。 この順序を変更したり、シェルスクリプトの形式でフェーズを追加したりできます。 多くの場合、フェーズの順序でスクリプトの適切な場所を選択する必要があります。たとえば、バンドルからファイルを変更する場合は、Xcodeがファイルをバンドルにコピーした後に適切なスクリプトを使用する必要があります。


ターゲットに依存関係を追加して、Xcodeが製品のビルド順序を決定できるようにすることもできます。 ピザの例に戻り、PizzaComboターゲットを追加することにより、2つのピザコンボを顧客に提供できます。







スクリーンショットでわかるように、このために、フレームワークの形式でハワイのピザとマルガリータのターゲットを作成する必要がありました。 次に、それらを依存関係としてPizzaComboターゲットに追加するだけで、これらの製品を別の製品内で使用できます。







ビルド設定



適切なピザを作るための適切な材料に加えて、3Dプリンターはいくつかの規則に従う必要があります。 それらのいくつかは、一般的にピザプロジェクトに固有のものです（たとえば、すべてのピザは同じ形式で印刷する必要があります）、いくつかは特定のピザに固有のものです（たとえば、ハワイのピザはマルガリータとは異なる温度で調理する必要があり、そうでなければパイナップルは乾燥します）。



ドキュメント：

アセンブリのカスタマイズは、アセンブリプロセスの特定の側面の実行方法に関する情報を含む変数です。


ビルド設定セクションを最初に見たとき、最初の関連付けは次のとおりでした。







ただし、幸いなことに、日常の開発で利用可能なすべての設定とそのキーを知る必要はありませんが、常に少数しか使用していません。



各設定は、キー、値、および人間が読める形式のキーのタイトルで構成されます（そのため、[ビルド設定]タブに収まるようになります）。



設定値は、プロジェクトレベルとターゲットレベルの両方で設定できます。 一部のレベルで設定値が設定されていない場合、前のレベルの値を継承します。



継承は次の順序で実行されます（小さいものから大きいものへ）。



  • プラットフォームの基本値
  • プロジェクトxcconfigファイルからの値
  • プロジェクトで設定された値
  • ターゲットxcconfigファイルの値
  • 目標値


「ビルド設定」セクションの「レベル」ボタンを使用して、特定の設定の値がXcodeで設定されているレベルを理解できます。 緑色は、アセンブリ中に設定される値を示します。







ユーザー定義の設定を追加して、たとえばビルドフェーズまたはInfo.plistファイルで使用できることを知っておくことが重要です（ただし、いくつかの制限はあります）。



条件設定を設定する可能性についても言及する価値があります。 つまり、製品を構築するプラットフォームに応じて、同じセットアップに対して異なる値を設定します。 設定をより「スマート」かつ柔軟に分離するには、ビルド構成を使用する必要があります。



ビルド構成



たとえば、各ピザを通常の食事バージョンで提供することにしました。 ピザの食事バージョンはそれほど揚げられず、揚げたハムはゆでたものに置き換わり、生地は塩がなくなります。 その場合、同じピザの2つのバリエーションは、プリンターの設定全体によって異なります。 既存のピザごとに新しいターゲットを作成するのではなく、印刷用に異なる設定セットのみを適用するために、アセンブリ構成を使用します。



ドキュメント：

アセンブリ構成は、特定の方法でターゲット製品を組み立てるために使用されるアセンブリ設定のセットを定義します。



デフォルトでは、Xcodeは2つのデバッグ構成とリリース構成を作成しますが、必要な数の構成を追加できます。



したがって、通常のピザとダイエットピザの構成を作成することにより、各構成に対応する異なる設定ごとに2つの異なる設定を設定できます。



このようにして、たとえば 、テストサーバーと本稼働サーバーを操作するためのアプリケーションビルドを分離できます。



アセンブリ構成をxcconfigファイルに移動して、.pbxprojファイルの役割の1つを削除し、設定を変更しながらマージを簡単にすることができます。 これは、たとえば、このツールを使用して実行できます 。



ワークスペース



そこで、ピザの印刷を開始し、いくつかのピザオプションを追加し、顧客からフィードバックを収集したところ、ほとんどの人がピザが好きでしたが、ロールと一緒に食べたいと思いました。 Pizza.projectに基づいてロールを作成することはできないため、Sushi.projectプロジェクトを3Dプリンターに追加し、そのベースに基づいて特定のロールのモデル（ターゲット）をいくつか作成します。 ピザとロールを印刷できるようになりましたが、お客様は一緒に食べたいので、そのようなオファーをします。 土地からのプロジェクトを含むPizza.workspaceを作成しています。 これで、特定のピザと特定のロールの両方を含む、結合されたビジネスランチのターゲットを作成できます。 また、このプロジェクトでは、ピザを印刷するとき、ロールを印刷するとき、およびビジネスランチを組み合わせて印刷するときに使用する水ギセルを印刷するためのサードパーティライブラリを含めることができます。 たとえば、ビジネスランチの最初のバージョンに、ハワイアンピザ、フィラデルフィアロール、サードパーティライブラリの水ギセルを含めるとします。











ドキュメント：

ワークスペース-Xcodeドキュメント、グループ化プロジェクト、およびその他のドキュメントを全体として使用します。


簡単に言えば、ワークスペースにより、任意の数のプロジェクトと関連ファイルを単一のコンテナに結合できます。 これにより何が得られますか？

ワークスペース関数：


  • 含まれる各プロジェクトの各ファイルへのアクセスを提供します。
  • スコープを拡張します。 これにより、オートコンプリート、検索、定義への移行は、ワークスペースに含まれるすべてのプロジェクトで機能します。
  • Xcodeがターゲットの依存関係を解決し、適切な順序でプロジェクトをビルドするのに役立ちます。 たとえば、あるプロジェクトのターゲットのSwiftモジュールを別のプロジェクトにインポートできます（これらのプロジェクトが同じワークスペースにある場合）。


ボンネットの下
ワークスペース-拡張子が.xcworkspaceのパッケージ。

.xcodeprojで行ったように、.xcworkspaceの内部を見てみましょう。 最初に目にするのは、contents.xcworkspacedata-ワークスペースに含まれるプロジェクトとファイルを記述するXMLコードです。



すでにおなじみの.xcuserdataおよび.xcshareddataフォルダーも確認できます。 たとえば、スキームを保存する場所（プロジェクトレベルまたはワークスペースレベル）を自分で決定できます。 選択に応じて、.xcodeprojのフォルダーのいずれか、または.xcworkspaceのフォルダーのいずれかに表示されます。



結果は何ですか



今日、プロジェクトはプロジェクトファイル、ターゲット、構成、および基本設定のリポジトリであることを学びました。 ターゲットは、このターゲットが属するプロジェクトに基づいて製品を構築するための一連の指示です。 そのアセンブリ設定は、アセンブリのいくつかの側面を記述するキーと値のペアであり、プロジェクトレベルとターゲットレベルの両方で設定できます。 このアセンブリ構成は、各アセンブリ構成の値のセットであり、いくつかの一般的な考えに基づいています。 そして最後に、ワークスペースは、いくつかのプロジェクトを便利に共有するためのコンテナです。 この記事を読んだ後、用語の混同がなく、アセンブリフェーズ（ワークスペースまたはターゲット）の責任者を覚えていることを願っています。
