祖父UXデザイン：レイアウトデザイナー-ペスト

みなさんこんにちは。 テクノロジーを打ち破り、5年の休憩の後にレイアウトに戻ることにしました。 テクノロジーが機能し始めたばかりで、レイアウトデザイナーが安くなったことに驚いた。 しかし、ドキュメントに「UX」という言葉が示されている場合、作業コストは同じ生産コストで少なくとも2倍に増加します。これはかなり興味深い提案です。 私はビデオファイルを見に行きました（ 神、ロシア語で100万のビデオファイルがあります ）が、少しd然としました（言葉が変わりました）。



ここでは、フォームの素材を考慮していません。「私たちは、釘を石、レール、木に打ち込もうとします。 さらに結果を評価し、フォームの材料：「胃の上で横転することを学んだすべての子供たちは確実に補い始めなければならない」メッセージを画面に表示する文字-テキスト、写真、その他のものを取得します。 そして、コード外の専門家になることをやめます。



10年前のように、接頭辞を付けたほとんどのコーダーはサーカスの熊として機能します。 トリックは私たちの目標であり、プロ意識の尺度です。



それからいくつかのポイント、クマになるのをやめ、人間になろうとする方法。



簡単です。いくつかのことを理解して消化するだけです。



テキストを表示するにはブラウザが必要です



すべての基本タグおよびセマンティックタグは、テキストをエンドユーザーに転送する順序でマークアップするために必要であり、「ドキュメントの構造を完全に転送しました」状態に最小化する必要がある余分な負荷です。 ライブラリ/カタログシステムに関して実際の画期的な意味を持つ唯一のタグは、HTMLアンカーです。



この概念の再考は、たとえばYandexでのサービスとしてのブラウザーの開発に基づいていますが、ベースでは、カーネルでは、ブラウザーは引き続きテキストを表示します。 また、ブラウザはタイプセッター（UI / UXスペシャリスト）の作業とはほとんど関係がありません。 食べ物のテーブルのような。



ブラウザはあなたなしで表示されます



文書ツリーを作成した後、タイプセッターとしての役割-技術スペシャリスト-は完全に使い果たされます。 それから、「私はそれを提出する方法をよく知っている」という分野であなたの自我とあなたの上司/顧客の自我に満足します。 これらはすべてブラウザに表示され、ユーザーはそこで読むか、そこで行うことを読みます。



自我満足はかなり有利なビジネスです。



この作業をすべてシステムから差し引いても、ユーザーはリンク「ドキュメント-検索システム」を通じて必要な情報を受け取ります。この場合、セマンティクスはレイアウト設計者の実際の役割です。 これらの言葉を確認して、スマートフォンのブラウザを見てください。 常に表示されるボタンの1つは、「インターフェイス全体を破棄し、既に必要なものを読む」ことです。



さらに、デフォルトでは、ドキュメントは完全に意味論、適応、高速、スケーラブルなデザインで表示されます。 あなたはすべてを壊してスタートします。



あなたは何も制御しません



あなたが持っているのは、テストで使用したブラウザではすべてが順調に進んでいるという大胆な仮定であり、以前のブラウザではすべてが十分にうまくいくという希望があります。 少なくともドキュメントとアンカータグの基本構造の外に少なくとも何かを定義するいくつかのWeb標準があるという合意の錯覚。



あなたが見ているスクリーンに特化したプログレッシブエンハンスメント/グレースフルデグラデーションの問題は2〜3年の問題であり、あなたが再び口beforeのないクリアテキストであることを理解してください。



これは、コンテンツによってテクノロジーがどのように拒否されるかを示す素晴らしい写真です。



あなたは、標準のサポートで起こるすべての責任があります。



ブラウザ自体は非常に正常です。 問題は、フローを壊す/インタラクティブにする/ドキュメントにインターフェースを追加するという問題から始まります。 これらの質問はすべて、ドキュメントを読んだ人ではなく、ドキュメントを分類して内容を「改善」しようとした人によって提起されました。



問題は、ほとんどの人々（コーダーを含む人々）がまだ最も精通した生き物ではないということです。 そして、成功した商業製品が答えるのは、まさに大多数の要求です。



最も影響を受けるブラウザは、リクエストに最も迅速に応答するブラウザです。 インターネットエクスプローラーの死は、彼が大衆市場の要求に完全に応えたという事実から来ました。 手動によるサイトの作成が限界的な職業でなくなり、サービスを開始するとすぐに、標準が有益だったため、突然標準のサポートが可能になりました。 そして、最終的に製品を殺すことができます。



したがって、あなたはブラウザを1つずつ選択するのではなく、抽象的に合意された原則に従って群衆を選択する野bar人の群衆の単なる代表者です。



ほとんどのブラウザの問題は問題ではありません。



たとえば、よく知られているタスクは、1つのテキストを複数の列に表示する方法です。 このタスクにはいくつかのエレガントなソリューションがありますが、その中に必要なものはありません。



Webでは、複数の列の1つのテキストを読む必要がないためです。



そしてどこでも（水平スクロール、物理的なページングの模倣などを参照）



UX / UIはどこにありますか？



レイアウト外。 シンプル-インターフェースは何かの問題です。 「何か」がなければ、それはボタンと状態の集合です。



テクノロジーは何の意味もありません



それはただ一つのアイデアの継続的な改善の流れであり、最後まで行く必要はなく、自信を持って適切なツールを選択し、そのライフサイクルを評価する方法を学ぶ必要があります。 5年後、はい、1年後にはすべてが異なります。 速度に焦点を当てます。 その後、あなたは、終了してサポートする個人を見つけます。



アイデアを理解し、それを伝えれば、すべてがそれ自体に結びつきます。



テキストを学ぶ



テキストは、思考を伝える最も効果的な方法です。 それ以外はすべてワニのゲームで、結果はあまり予測されていません。



UX / UIプロセス全体が新しいコントロールでテキストを編集しています。



  • 完成した作品全体で情報を提供します。
  • フローブロックから離れ、フローテキスト、フローの意味を参照してください。
  • テキストは拡大縮小できます。 フォントではなく、テキスト。




意味を見つけて追加する



インターフェイス内のすべてが共同で消費されます。作業によって何も追加されない場合、最善のことはそれを行わないことです。 コンテンツ（テキスト、オーディオ、ビデオ、あなたと一緒にいるもの）を扱うプロセス全体を覚えておいてください-最初から最後まで、それが何であるかを理解し、それを愛します。



自分でやります



テキストを書き、オーディオ、ビデオを録音します。 状態を感じ、元の素材の品質を評価し、追加できる場所と失う場所を完全に理解する必要があります。



しない方が良い



誰もが最も効率的に独自の製品サイトを作成できるように、チームが必要です。 いくつかのパラメーターを変更しても製品の劣化は明らかに起こらないが、生産の迅速化と安価なサポートにつながるレイアウトがある場合は、デザイナーに行き、何をどのように説明する必要があります。 紙の上で。



プロセスとしてのタスクは結果よりも興味深いことを理解していますが、自分で管理する必要があります。



なぜ
