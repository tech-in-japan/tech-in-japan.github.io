JPoint 2018でのパフォーマンスとランタイム

会議プログラムは、トピックごとに少なくとも1つの良いレポートを選択するように設計されています。 JPointは2日以内に開催され、約40のレポートがあるため、すべての主要な問題が何らかの形で強調されます。







この短い投稿では、主にパフォーマンスとランタイムに関するレポートに行く人として私が気に入ったレポートについてお話します。







スケーリング、クラスター、その他すべてについては考慮しません。LightbendのChristopher BateyがAkkaについて話し 、ConfluentのVictor Gamov が Kafkaについて話します。









ディスクラマー 記事は公式ウェブサイトのプログラムの内容の印象について書かれました。 以下はすべて私自身の考えであり、レポートからの引用ではありません。 テキストには、誤った仮定や不正確な記述が含まれている可能性があります（確かにあります）。


性能



冗談の記事「アセンブリ言語を挿入したJava」を覚えていますか？ コメントの中で、 apanginはVMStructsについて話をすると述べた。 「VMStructs：なぜアプリケーションはJVMの内部について知る必要があるのか​​？」 レポートはVMStructsの使用に専念しています。これは、TLAB、コードキャッシュ、定数プール、メソッド、シンボルなど、JVMの内部構造について学ぶことができるHotSpot仮想マシンの特別なAPIです。通常のプログラム。 Andreyは、VMStructsが実際のツール（Odnoklassnikiで使用）の開発にどのように役立つかの例を示します。







2番目の講演「Javaのハードウェアトランザクションメモリ」は、DevexpertsのdxLab研究グループの研究エンジニアであるNikita Kovalによるものです。 今月初めにJBreakにいた場合、彼はそこでまったく異なることについて話していることに気付くかもしれません（最新のマルチコアアーキテクチャと特別なアルゴリズムのパワーを使用して高速マルチスレッドハッシュテーブルを書く）。 同じレポートで、トランザクションメモリについて説明します。トランザクションメモリは、現代のプロセッサで徐々に表示されますが、普通の人にどのように使用するかはまだ不明です。 Nikitaは、それを使用する方法、OpenJDKに既にある最適化、およびJavaコードから直接トランザクションを実行する方法について話す必要があります。







そして最後に、 Enterprise Without Brakes 。 血なまぐさい企業のない私たちはどこにいるのでしょう！ Sergey Tsypanovは、ドイツ銀行ドメインのLuxoftでパフォーマンスの問題を扱っています。 このレポートでは、アプリケーションのパフォーマンスを低下させるパターンを調べます。コードレビューで見られるほど軽く、IDEで赤く強調されないほど複雑です。 すべての例は、運用環境で実行されているアプリケーションのコードに基づいています。







プロファイリング



眼球プロファイリングについては、3つのレポートが取り上げられました。 Sasha Goldsteinによる最初のレポート、 「JVMアプリケーション用のLinuxコンテナパフォーマンスツール」 。 サーシャは、ハードコアパフォーマンスの連続制作者です。 昨年、JPointで、彼はJVMでのBerkeley Packet Filterの使用について素晴らしい講演を行いました（YouTubeの投稿を見ることを強くお勧めします）。コンテナ化の詳細を見るのは時間の問題でした。 世界は雲と港湾労働者になり、それが私たちに多くの新しい問題をもたらします。 お気づきかもしれませんが、ほとんどの低レベルのデバッグおよびプロファイリングシステムは、コンテナに適用されると、さまざまな機能や妨害物でいっぱいになります。 Sashaは、BCCやperfを含むGNU / Linuxプラットフォームで最新のツールを使用するというプリズムを通して、主なシナリオ（CPU使用率、IO応答性、共有データベースへのアクセスなど）を検討します。







「マイクロ秒およびプロセッサー命令まで正確にプロファイリングする」は、ライファイゼンバンクのセルゲイ・メルニコフが作成した2番目のプロファイリングレポートです。 興味深いことに、低遅延のJavaコードに取り組む前は、IntelでC / C ++ / FORTRAN言語のコンパイラパフォーマンスエンジニアとして働いていました。 このレポートにはパフォーマンスもあります！ :-)また、プロセッサのハードウェア機能とIntelプロセッサトレーステクノロジについても説明します。これにより、プロファイリングの精度を向上させ、プログラムセクションの実行を再構築できます。 そのようなレポートはほとんどありません（たとえば、Tracing Summit 2015でAndi Kleenレポートを見つけることができます）。通常、多くの疑問が残り、Javaに関しては実用性に欠けています。 ここには、両方の世界（IntelとJavaの両方）を訪れた人がいるだけでなく、ディスカッションエリアで見つけて不快な質問をすることもできます。







3番目のレポートは、 「ユニバーサルプロファイラーとその場所」です。 Excelsior JET（AOTコンパイルの最適化に基づく認定Java SE実装）の開発者の1人であるIvan Uglyanskyによって作成され、ランタイム、GC、クラスの読み込み、マルチスレッドのサポート、プロファイリングなどを処理します。 レポートの本質は、最近、Excelsior JETで実行されているアプリケーションのプロファイルを収集する必要があったことです。 サポートされているすべてのシステムとアーキテクチャで、アプリケーションを再コンパイルせずに、許容できるパフォーマンスでこれを行う必要があります。 プロファイリングの通常の方法が同時にこれらすべてのポイントに適合しないことが判明したので、私は自分の何かを考え出さなければなりませんでした。 Ivanは、AOTに適したプロファイリング方法、JVM内からコードをプロファイリングする場合の余裕、およびプロファイラーの汎用性に対して支払う必要があるものを教えてくれます。







カスタムランタイム



ランタイムは、要するに、JVM言語で高レベルのコードを取得し、低レベル（たとえばマシンコード）に変換して実行プロセスを制御するものです。 通常、アセンブラー、コンパイラー、インタープリター、仮想マシンなどがあります。 ランタイムの機能は、適用されるタスクのパフォーマンスの機能を定義します。







プログラムに最初に注目するのは、 JDKに関するAlibabaレポートです。 ブラックジャックとコルーチンを使用して独自のJDKを作成することを夢見ていない人はいませんか？ しかし、これは地獄のような労働、痛み、苦しみであることは誰にとっても明らかです。 しかし、アリババではそれが判明しました。 ここに彼らが持っているものがあります：







  • GCのオーバーヘッドなしで非表示領域のオブジェクトを選択できる特定のメカニズム。
  • JVMに直接組み込まれた軽量スレッド（コルーチン）は、非同期プログラミングに必要です。
  • ライブプロファイリング機能
  • 別の素敵なささいなこと


はい、私たち（OpenJDKを使用している一般大衆）は間もなくProject Loomを使用します。 しかし、微妙な違いがあります-Loomでのコルーチンの開発は、主な目標である繊維に次ぐものです。 ファイバーは区切られた継続を必要としますが、パブリックAPIにすぐに、またはまったく登場する必要はありません。 アリババでは、これらすべてがすでに独自に洗い流されているようです。







私が理解しているように、これは「独自のプロプライエタリJDKを使用する」というカテゴリからのレポートではなく、同様の機能の開発をマスターするか、OpenJDKでの不在と戦う人々のためのガイドです。 たとえば、プロファイリングツールは、プロファイル対象の領域とワークロードに依存します。製品ごとに異なります。 Alibabaのスピーカーは、彼のツールについて話すだけでなく、ワークロードを分類するプロセスについても話します。これは、そのようなツールを正しい方向に開発しています。







ところで、コルーチンについて話しているので。 それらはKotlinにバージョン1.1（ 実験状態 ）から登場し、それらについてはJetBrainsのRoman Elizarovによる報告があるでしょう。 この小説では、非同期プログラミングへのアプローチの進化、それらの違いと類似性について説明します。 さらに、現在のKotlinにあるものが、おなじみのasync/await



よりも優れているのはなぜかという公式の立場を聞きます。







行き過ぎないようにするために、Alibaba JDKは珍しいエコシステムの唯一の代表者ではありません。 もちろん、Azul Zingについてのレポートがあり、OpenJ9については2つ （ 1、2 ）あります。







Azul製品の内部に関するすべてのレポートには、私にとってある種の悲しみがあります。 私の人生の中で、私は彼らのクールではあるが非常に高価な決定を使って選挙の輪に入らなければならなかった。 したがって、私にとっては、この新しいレポートは 、ネイティブOpenJDKと競合するテクノロジーに関する情報源として理論的に重要である可能性が高くなります。 現在、OpenJDKでは、AOTテーマが積極的に開発されています-OpenJDK JDK 9には、既にAOT（64ビットLinuxのみ）が組み込まれており、 SubstrateVMがあり、 Metropolisプロジェクトの実装までは改善されます。 残念ながら、JavaのAOTでは、それほど単純ではなく、現代のインフラストラクチャの一部を壊すことは非常に不愉快です（ Nikita Lipskyによる、曲がって設計されたOSGiについての壮大なレポートを思い出してください）。 Azulは、 ReadyNowと呼ばれる既製のAOTソリューションをZingに組み込み、JITとAOTの最高の品質を組み合わせようとしています。これがこのレポートの目的です。







コメントで正しく指摘されているように、スピーカーを紹介する必要があります。 つまり、Azulの主要な開発者であるDouglas Hawkinsは、15年間Javaに携わり、バイオインフォマティクス、金融、小売などのさまざまな分野に参加しました。 彼がjavamirに住んでいればいるほど、彼はJVMの内部に深く入り込み、Azulを離れてZingの仕事をするようになり、そのReadyNowのメイン開発者になりました。 つまり、これはバリケードの両側を訪れた人です。応用開発者としてもシステム開発者としても、そしてその結果として非常にユニークな経験をしました。







一方、OpenJ9 は今すぐダウンロードできます 。 IBMがEclipse Foundationで仮想マシンをオープンして以来、その周辺には多くの誇大宣伝がありました。 一般の人々には、HotSpotをHotSpotに置き換えることができ、OpenJDKのライブラリを簡単に再利用できること、使用されるメモリの量を減らす必要があること、GPUに何かを転送できることについて、一定のアイデアと事実があります...そして、一般に、それだけです。 （ちなみに、GPUは一般的に一般的に黒魔術のように見えます-良い、過去にJoker Dmitry Alexandrovは「JavaとGPU：私たちはどこにいますか？」という素晴らしい報告をしました。まだビデオはありませんが、 スライドを見ることができます）。







最初のレポート、 「Eclipse OpenJ9 JVM：深いダイブ！」 ValhallaとPanamaに取り組んでいるIBM J9開発者であるTobi Ajila氏は、インタープリター、JVMTI、ラムダの改善などの長い実績があると言います。 どうやら、OpenJ9のいくつかの技術的機能の説明があるでしょう。これにより、クラウドソリューションやその他のパフォーマンスが重要なことをオーバークロックできます。 2番目のレポート「Eclipse OpenJ9 GCテクノロジーの詳細」は、同じくIBMのOpenJ9ガベージコレクタアーキテクトが主導します。4つのガベージコレクションポリシー、それらを使用する場所、およびそのすべてが内部でどのように機能するかについて、非常に実用的なストーリーがあります。 これらのレポートを聞いた後、OpenJ9にまつわる魔法のオーラがわずかに減少することを願っています。
