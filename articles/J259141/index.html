OpenStreetMapカードのロシア語化と多言語化

OpenStreetMapデータを使用してWebマップを開発する場合、正しいロシア名でマップを表示する方法についてしばしば疑問が生じます。 あなたのカードがロシアだけを示している場合、この問題は発生しません。 ただし、たとえば中国の地図を見ると、そのような豊富な象形文字が好きになる可能性は低く、そのような地図で北京を見つけようとしても無駄に失敗する可能性があります。







自由が大好きなOpenStreetMapプロジェクトでは、地理オブジェクトの名前を異なる言語で保存できることが知られています。 このために、 name：ru 、 name：enまたはname：esなどの特別なタグが使用されます。最も重要なのは、 OpenStreetMap参加者によって埋められます。 もちろん、最も詳細な碑文は、ユーザーが話す言語で作成されます：ロシア語-ロシア語、中国-中国語、アフリカ諸国-現地語。 ナイジェリアの一部の道路でロシア語の翻訳が行われる可能性はほとんどありませんが、それでも主要な地理オブジェクト（国、都市、河川など）には翻訳があります。 この小さな地図情報は、ロシア語を話すユーザーが、たとえば中国の地図を開いて、その上の主要な名前を見つけるのに十分です。 したがって、リソースはもう少しユーザーフレンドリーになります。



独自のWebカードでこれらの翻訳タグを使用する方法に関する賢明な指示を見つけることを目指している場合、Runetのスペース全体またはインターネットでさえ、 OpenStreetMap Webマップのローカリゼーションのさまざまな側面に関する情報のみが散在していますが、「箱入り」の指示はほとんど見つかりません。 一方で、この問題は最も難しいアーキテクチャ上の問題ではありません。なぜなら、 このテクノロジーは、 OpenStreetMapプロジェクト自体の一部として開発されました。 しかし一方で、有名なRostelecom SatelliteなどのローカライズされたWebマップの完成したプロジェクトでは、ローカリゼーションの問題は解決され、開発者が直面した問題とそれらの解決方法はITコミュニティによって話されていません。







多言語サポートを備えたWebマップのデザインもありますが、それらは論理的で使いやすいものではありません。







この記事の目的は、この異種の情報を、 OpenStreetMapマップをローカライズするための完全な指示の形でまとめることです。 この記事では、 OpenStreetMapデータに基づいて北にタイルを展開することに関する最小限の情報を、 Habré 、 Gis-Lab'e 、および他の特別なプロジェクトの他の出版物は、このトピックを詳細に明らかにしています。 この記事のフレームワークでは、 よく知られている指示に従って、タイルのリポジトリ（Webマップイメージが「ステッチ」されるイメージタイル）をデプロイしますが、詳細な説明は行いません。 タイルストレージの展開に関する質問がある場合は、最初に記載されている手順をご覧ください。



したがって、パッケージが必要です（必要なすべての依存関係を含む）：

この記事の目的のために、本格的なタイルサーバー（Apache httpd、mod_tile、およびレンダリング済み）をデプロイする必要はありません。 ただし、タイルサーバーの展開の方向ベクトルは以下のとおりです。詳細な手順については、上記のリンクを参照してください。



1. OpenStreetMapからベクターデータを取得する



このデータを（osm2pgsqlユーティリティを使用して）Postgresqlデータベースにインポートすると、Mapnikレンダリングパッケージがラスタータイルを生成します。



そのため、 OpenStreetMapミラーの1つから、XML形式（OSMファイル）または圧縮バイナリPBF形式で指定された領域のマップをダウンロードします。

planetファイルは、次のコマンドでダウンロードできます。 



wget http://download.bbbike.org/osm/planet/planet-latest.osm.pbf.md5 wget http://download.bbbike.org/osm/planet/planet-latest.osm.pbf md5sum planet-latest.osm.pbf


このデータは重要であることに注意してください。 たとえば、惑星Planet.osm全体の非圧縮OSM XML形式のベクターデータを含むファイルは1TB以上、XML形式では圧縮bz2、41.8GB、バイナリPBF形式18.1GBを使用します。 これらの形式はすべて同じデータを保存します。 当然、惑星全体ではなく特定の地域のみが必要な場合、これらの数値ははるかに小さくなりますが、フォーマット、OSM XML、OSM XML、圧縮bz2、およびPBFフォーマットのデータボリュームの割合は変わりません。 Postgresqlにインポートする前に、BZ2およびバイナリ形式PBFで圧縮されたOSM XML形式のベクターデータが前処理（アンパック、解析）されるため、ダウンロードしたファイルをPostgresqlデータベースにインポートする時間が長くなることに注意してください。 そのため、16GBのRAM、2TBのHDD（Ubuntu 14.04 64x）を搭載したクアッドコアCore i5マシンでは、 Planet.osm.pbfのインポートに2週間かかりました。 同様のPlanet.osm.bz2ファイルのインポートには、1桁の時間がかかりませんでした。 私の考えでは、osm2pgsqlユーティリティはRAMの量を要求しています（最適なメモリ管理のため、osm2pgsqlユーティリティの--cacheおよび--cache-strategyキーの機能について読んでください）、プロセッサコアの頻度と数（ファイルの解析時、プロセッサコアのロードを制御するために参照してください） （osm2pgsqlユーティリティの--number-processesキー）、およびハードドライブの速度（Postgresqlでのデータの挿入時およびインデックスの作成時、最適化については、osm2pgsqlユーティリティの--disable-parallel-indexingキーを参照してください）。 SSDがある場合、Postgresqlデータベースへのデータのインポートははるかに高速になります。



この記事では、中国の地域を次のように使用します。 デフォルトではRussifiedではなく、サイズが比較的小さいので、デバッグ技術の段階で便利です。 中国のベクターデータファイルは、 geofabrik.deプロジェクトのWebサイトから次のコマンドを使用してダウンロードできます。 



 wget http://download.geofabrik.de/asia/china-latest.osm.pbf


2.受信したベクターデータOpenStreetMapを更新します



実際、データファイルがパッケージ化されてからマシンに展開されるまでの間に、OpenStreetMapでいくつかの変更が行われる可能性があります。 したがって、インポートする前に、osmcutilsパッケージのosmupdateコマンドでデータファイルを更新します。 



 osmupdate china-latest.osm.pbf new-china-latest.osm.pbf


OSMファイルにタイムスタンプが含まれていない場合、osmupdateユーティリティはエラーを返します。 この場合（ファイルのタイムスタンプがわかっている場合-ファイルのダウンロードページで公開される場合があります）、次の形式でコマンドを実行する必要があります。 



 osmupdate china-latest.osm.pbf 2015-05-13T14:48:07Z new-china-latest.osm.pbf


osmupdateユーティリティは、diffファイルを個別にダウンロードし、callコマンドの最初の引数で指定されたファイルに適用します。 OSM XML形式の大きなファイル（planet-latest.osm.bz2など）でosmupdateユーティリティを実行することはお勧めしません。 同じosmcutilsパッケージのosmconvertユーティリティによるファイルの予備処理が必要です。osmupdateユーティリティは、変換後のファイルで数日間機能します。 大きなファイルの場合は、PBF形式を使用することをお勧めします。 ただし、このようなファイルをPostgresqlにインポートする場合は、OSM XMLファイルをインポートするよりも時間がかかることに注意してください。 いわば、両刃の剣。 個人的に、私は常にPBF形式を選択します。



3. osm2pgsqlユーティリティの設定で必要な言語のサポートを構成します



デフォルトでは、すべての言語のサポートは/usr/share/osm2pgsql/osm2pgsql/default.styleスタイルシートにインストールされていません。 マップ上の碑文には、次の行で/usr/share/osm2pgsql/osm2pgsql/default.styleファイルに設定されている名前タグが使用されます。 



 node,way name text linear


たとえば、ロシア語、英語、スペイン語をサポート後に追加します。 



 node,way name:ru text linear node,way name:en text linear node,way name:es text linear


これらの3行は、osm2pgsqlユーティリティに、PBFファイルからタグ値name：ru 、 name：en 、 name：seもインポートするように指示します。 他のローカライズされた名前は無視されます。



4. OpenStreetMapベクターデータを保存するPostgresqlデータベースを作成します



最初に、信頼認証を構成し（パスワードを入力しないため、この記事のフレームワーク内でこれは必要ありません）、必要なデータベースユーザーを作成する必要があります（ 手順を参照）。 この記事の一部として、認証と標準のpostgresユーザーを信頼するように制限しています。 そのため、たとえばchinaなどのデータベースを作成し、必要な拡張機能を接続します。 



 createdb -U postgres china psql -U postgres -d china -c 'CREATE EXTENSION hstore; CREATE EXTENSION postgis;'


5.ベクターデータをPostgresqlデータベースにインポートする



インポートはosm2pgsqlユーティリティによって実行されます。 ユーティリティキーの説明は、ユーティリティのヘルプに記載されています。 いくつかの説明情報が英語のリソースで提供されています。 



 osm2pgsql -s -m -d china -U postgres --drop new-china-latest.osm.pbf


--dropスイッチを使用すると、データベースが占有するディスク容量を減らすことができるため、新しいPBFまたはOSM XMLファイルからデータベース内のデータを後で更新する機能が犠牲になります。 中国の場合、データベースのサイズは約500MBから92MBに減少しました。 データベースが占めるサイズを確認するには、既存のデータベースに接続されたpsqlコンソールで次のコマンドを入力します。 



 SELECT pg_size_pretty( pg_database_size( 'china' ) );

7. OpenStreetMapからスクリプトをインストールしてタイルを生成します



ディレクトリ/ home / osm / mapnikを選択して、必要なMapnikスクリプトをすべて保存しましょう。 OpenStreetMapからスクリプトをポンプしてタイルを生成し、小さなスケールでのタイルの生成を簡素化するために使用される世界の形状（シェイプファイル）を抽出し、MapnikスタイルのXMLファイルを作成してデータベースにアタッチします。 



 svn co http://svn.openstreetmap.org/applications/rendering/mapnik /home/osm/mapnik /home/osm/mapnik/get-coastlines.sh


次に、地理オブジェクトのロシア語、英語、スペイン語の署名用に、それぞれタイルをレンダリングするための3つのファイルスタイルを作成する必要があります。 これを行うには、特別な環境変数MAPNIK_PREFIXを設定します。これにより、Postgresデータベース内のベクターデータテーブルのプレフィックスを設定でき、スクリプト/home/osm/mapnik/generate_xml.pyが読み取り、データベースへの擬似クエリに置き換えます。 



 export MAPNIK_PREFIX='china_ru' /home/osm/mapnik/generate_xml.py /home/osm/mapnik/osm.xml china_ru.xml --dbname china --user postgres --accept-none export MAPNIK_PREFIX='china_en' /home/osm/mapnik/generate_xml.py /home/osm/mapnik/osm.xml china_en.xml --dbname china --user postgres --accept-none export MAPNIK_PREFIX='china_es' /home/osm/mapnik/generate_xml.py /home/osm/mapnik/osm.xml china_es.xml --dbname china --user postgres --accept-none


これらのコマンドは、タイルをレンダリングするための3つの特別なスタイルを作成し、ロシア語、英語、スペイン語の必要な表現からレンダリング用のデータを読み取ることができます。



8.タイル生成スクリプトでは、タイル生成のための中国の領域を示します



スクリプト/home/osm/mapnik/generate_tiles_multiprocess.pyには次の行があります。 



 bbox = (-180.0,-90.0, 180.0,90.0) #          render_tiles(bbox, mapfile, tile_dir, 0, 5, "World") #     0-  5-


これらの行の後、次のように記述します。 



 bbox = (85.0,19.7,132.5,40.8) #       render_tiles(bbox, mapfile, tile_dir, 6, 15, "China") #     6-  15-  exit()


これにより、必要な言語の署名を含む6〜15スケールの中国のタイルが生成されます。 exit（）命令を使用すると、スクリプト/home/osm/mapnik/generate_tiles_multiprocess.py（Muenchen、Muenchen +、Muenchen ++、Nuernberg、Karlsruhe、Karlsruhe +、Augsburg、Augsburg +、Europe +）でデフォルトで指定された領域のさらなるレンダリングを拒否できます。 また、単一のスレッドでレンダリングプロセスを開始するスクリプト/home/osm/mapnik/generate_tiles.pyもあります。 ただし、使用しません。 複数のプロセッサコアがある場合、スクリプト/home/osm/mapnik/generate_tiles_multiprocess.pyは一般に高速に動作します。



9.選択した場所のタイルを生成します



Mapnikが生成されたタイルを保存するディレクトリを作成する必要があります。 



 mkdir /home/osm/mapnik/tiles mkdir /home/osm/mapnik/tiles/ru mkdir /home/osm/mapnik/tiles/en mkdir /home/osm/mapnik/tiles/es


システム環境MAPNIK_MAP_FILEおよびMAPNIK_TILE_DIRの変数の値を変更して、タイルの生成を3回開始する必要があります。 スクリプトはこれらの変数を読み取り、Mapnikに渡してタイルのレンダリングを構成します。 



 export MAPNIK_MAP_FILE=/home/osm/mapnik/china_ru.xml export MAPNIK_TILE_DIR=/home/osm/mapnik/tiles/ru/ exec python /home/osm/mapnik/generate_tiles_multiprocess.py export MAPNIK_MAP_FILE=/home/osm/mapnik/china_en.xml export MAPNIK_TILE_DIR=/home/osm/mapnik/tiles/en/ exec python /home/osm/mapnik/generate_tiles_multiprocess.py export MAPNIK_MAP_FILE=/home/osm/mapnik/china_es.xml export MAPNIK_TILE_DIR=/home/osm/mapnik/tiles/es/ exec python /home/osm/mapnik/generate_tiles_multiprocess.py


タイルが生成されます。



10. Webマップにタイルを表示する



これを行うには、作成したタイルリポジトリを接続する最も単純なhtmlページを作成します。 ページファイルは、生成されたタイル/ home / osm / mapnik / tilesのディレクトリの隣に配置する必要があります。 Webマップを表示するには、 LeafletJS javascriptフレームワークを使用します。

注釈



  • 記事の元のバージョンでは、著者は中国ではなくキプロスの例を使用して地図のロシア化/ローカライズの例を示すことを計画しました。 キプロスの占有面積は中国よりもさらに小さいため、データのデータベースへのインポートが速くなり、タイルの生成が速くなります。 しかし、判明したように、「ローカル」言語にはラテンアルファベットに類似した文字が含まれています。 したがって、英語への翻訳は実証的ではありませんでした。 ちなみに、キプロスと中国のデータベースのサイズ（--dropキーの有無にかかわらず）は、これらの国が占める面積に大きな違いがあるにもかかわらず、プラスまたはマイナス数メガバイトでした。
  • この記事の著者は、記事の一部としてロシア語、英語、ドイツ語のサポートを実装する予定でした。 しかし、英語とドイツ語の翻訳はほとんどの場合同じ方法で書かれているので、記事の最終版では、ドイツ語をよりわかりやすくするためにスペイン語に置き換えました。
  • Planet.osmの読み込みに関する著者の実験は完了していません（キプロスと中国の地図の場合のように）。 事実、ファイルをPostgresデータベースにインポートするのにかなりの時間（2週間以上）かかっていたため、osm2pgsqlユーティリティ（およびそのスタイルファイル/usr/share/osm2pgsql/osm2pgsql/default.style）のこれらの設定またはその他の設定を「再生」できませんでした著者の時間。 そのため、この記事にはこのような大量のデータの処理に関する特定の統計情報がありません。
  • china_ru_point、china_ru_roadsなどのデータベースビューを作成し、システム環境変数MAPNIK_PREFIXに必要な翻訳をテーブルのプレフィックスに強制することは、必要な翻訳を取得する唯一の方法ではありません。 generate_xml.pyスクリプトによって生成されたMapnikスタイルファイルでは、TextSymbolizerタグでマップ上の署名を表すための形式を構成できます。 特に、 銘刻文字[name]の生成された名前フィールドを[name：ru]に置き換えると、Mapnikは名前から取られた銘刻文字を入れます：ruフィールドのplanet_osm_point 、 planet_osm_line 、 planet_osm_polygon 、 planet_osm_roads tables 。 しかし、実践が示すように、地理的オブジェクトの名前の翻訳は、これらの名前自体よりも大幅に少ないです。 おそらくこれが、 多言語マップのプロジェクトマップが地理オブジェクトの署名なしで「裸」である理由です。 この点で、china_ru_ *のビューで作成した名前\ ruの連結は、より普遍的です。
  • 複数の言語をサポートする予定がなく、定期的にデータを更新する予定がない場合（つまり、osm2pgsqlユーティリティの--dropキーを使用する場合）、データベースビューの面倒な設計を1つのUPDATEクエリに置き換えて、 名前フィールドの既存の値を置き換えます名前を連結するには、表planet_osm_point 、 planet_osm_line 、 planet_osm_polygonおよびplanet_osm_roadsの ru \ nnameを使用します。 この場合、システム環境変数MAPNIK_PREFIXを強制的に設定する必要はありません。 そして、予想どおり、Mapnikは選択した言語のタイルのみを生成します。
