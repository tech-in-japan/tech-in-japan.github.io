Ural .NETユーザーグループミーティング





今日、2009年4月23日に、 Ural .NETユーザーグループのメンバーの会議が、Ural State University（USU）の数学部で開催されました。 会議では、トピックに関する2つの興味深いレポート「Unity Application Block-Unity Container」と「Composite Application Library」が読まれました。 それは非常に面白かった、詳細はカットの下で。



Unityアプリケーションブロック要素概要レポート-Unityコンテナー



最初のレポートは、スタニスラフチェルニッチキン（クローバーIT、プログラマー）によって作成されました。 レポートでは、 Unityについて非常に独特でした：スタニスラフは、例を挙げて分解する代わりに、下から、アーキテクチャから始め、Unity自体の内部構造とObject Builderの基本メカニズムをすべて棚に置きました（興味深いものがあります） acervからの記事 ）。 このアプローチはフレームワークのメカニズムを非常に深くカバーすることを可能にしましたが、実際には何も実際に実証することを許可しませんでした。 その結果、スタニスラフはそのトピックについて非常に深い知識を示しましたが、私は彼が誰にも理解されておらず、完全に理解されていないという印象を受けました。 はい、いくつかの点がはっきりしませんでした。



複合アプリケーションライブラリ



最初のレポートが確実に数時間続いた場合、2番目のレポートは著しく短くなりました。 Alexey Kofman（ソフトウェア開発者、Extensive）は、Composite Application Libraryについての資料を提示しました。これは、たとえば、そのようなインターフェイスのさまざまな部分がさまざまな開発グループによって開発された場合に、複雑なモジュラーインターフェイスでプログラムを実装できるフレームワークです。 その結果、Composite Application Libraryのコンポーネントの定義が示され、操作のメカニズム、長所と短所が説明されました。 Alexeyは、Silverlightを含むいくつかの例を示しました。 このレポートは最初のレポートよりも短くなりましたが、理解しやすく、コードと実行中のアプリケーションの両方のデモが多数ありました。



まとめ



一般的に、会議は成功しました。 たくさんの参加者がいましたが、私は20〜25人ほどは期待していませんでした。 あらゆる種類の質問が行われ、物議を醸す問題の議論があり、誰かが経験を共有したり、調整したりしました。 スタニスラフとアレクセイは素晴らしいです、私の意見では、彼らは非常に興味深く、関連する資料を用意しました。
