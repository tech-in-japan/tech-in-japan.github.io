2. 2つのチェック-4つのセル

素朴なアプローチは、2つの隣接セルの現在のセルとの同等性をチェックして、すべてのサブセルをバイパスすることです。 この場合、検証結果は、振動したテクスチャーの座標の中心へのシフトです（図を参照）。 最初は、すべての四分の一がテクスチャのコーナーピースとして受け入れられ、接続されると青い円を形成します（すべての側面に青い隣人）。 垂直方向の近傍との不等式は、選択したテクスチャー部分を垂直方向に中央にシフトします（2つの位置、反対の垂直角の後に選択されます）。 チェック後に何も移動していない場合（青い円全体が残っている）は例外として扱うことが望ましく、セル全体を四分の一ではなく一度に表示する必要があります。

これで、素朴な方法は終了します。

（さらにコードでは、タイルの初期位置は0xfc30の定数になります。水平/垂直シフトは、配列の必要なセルの対応するHEXビットで機能します）

2つのセルの等価性の検証は、実際には常に4つのサブセルに影響し、2つはそれぞれ8つのサブセルを持っていることに注意することが重要です！ （額のメソッドから）この8倍のアクセラレーションを取得して、一括処理します。 図を参照





これがアルゴリズムのコアです（JSの例）： 
function XFCR(map, w, h){ // "eXpansion Fast Cell - Rounded Squares" for(var xfc = [], n = 0, C = map[n - 1], D; n < w * h; xfc[n++ - w - 1] ^= 0xfc30, C = D){ if(C ^ (D = map[n])) xfc[n - w - 1] |= 0x8800, xfc[n - 1] |= 0x0088; if(D ^ map[n — w]) xfc[n - w] |= 0x202, xfc[n - w - 1] |= 0x2020; } return xfc; }




したがって、各サブセルの代わりに目的のセルをバイパスし、結果の配列に変更を加えるだけです。 また、近傍の有無の組み合わせは、2つの独立した要因として知覚できることに注意してください（上部の近傍の存在はxに沿ってテクスチャをシフトし、左側の近傍はyに沿ってテクスチャをシフトします）。 また、要因は独立しているため、同期に依存する必要はなく、非同期に処理できます。 ちなみに、xfcRSは一般に完全に並列化されています！

要因を見つけるとき、対応するX /ゲームを結果の配列に配置します（これはすべて注意が必要ですが、見ると、インデックスビットに非常に便利に格納されています）。
なぜ値をパッキングするのかについての解説
-「メモリは今それほど重要ではない」場合、すぐに答えます。この場合、メモリ圧縮は加速につながります。 セル固有の命令は、一度に複数のセルを処理して問い合わせます。 また、レンダリング時に、セルを四半期ごとに処理する必要性をすばやく判断するには、0xfc30でインデックスの値を比較するだけで十分です-この定数は、このセルの環境が均一であり、破損せずに（パッキングなしで、4つのサブセルすべてを個別にチェックする必要がある）バルクで表示できることを示します）;

-実際、結果のバイナリ形式には2倍の情報が含まれています（ただし、各HEXビットにはタイルのレディディスプレイスメントインデックスが格納されます。同時に、4セルのシフトの座標が2バイトに格納されます）。 非常に大きなボリュームで突然実行する必要がある場合、サブセルごとに2ビットのみを保存できます。 「マジック」定数0xfc30をレンダリングロジックに入れ、サブセルの位置を考慮する必要がありますが、4つすべてがバイト全体を占有します。 そして極端なトリックに行くと、条件の値のみを保存し、各サブセルに正確に半分のビットを費やすことができます=）、同時に、xfcRSアルゴリズムの本質は既にゼロに巻き戻されますが、その本質は便利なインデックスマップを提供することで隣人をすばやく比較するためです。 （ちなみに、たとえばCでは、Int32でハードウェアを有効にすることにより（サブセルごとに）バイトの配列をパックすることができます。これは、メモリによって失われる可能性が高くなりますが、速度と利便性によって勝ちます）。





-！ 重要！ 同じ理由で、xfcRSが結果の配列を返すゾーンは、セルの半分（1つのサブセル）で置き換えられます。 そして、エッジセルを計算することは不可能であるという事実は主なものではありません。 主なことは、インデックスをシフト（およびそれに応じて再グループ化）することで、完全にあふれた2x2サブセルをチェックして迅速に識別するのに便利なブロックが得られることです。 -例：目的のセルの2x2ブロックが着信マップに存在する場合、その結果、輪郭に沿ってセルの半分だけ平滑化され、中央（セルの床に移動）に1つのセル全体が存在し、インデックスから簡単に決定できます。 空のセルの定義は必要ありません。パススルーアルゴリズムの定義により、空のセルは既に標準のタイルアルゴリズムで描画されているためです。 例のレンダリングコードは、単にそれらをスキップします。
ポストレンダリングの例： 
 function draw(map, w, h){// View: //(multipass - empty cell background pattern needed) var sub = XFCR(map, w, h); // Compute sub-tiles by XFCR algorithm for (var n = 0, y = 4, j = h; --j; y += 8, n++){ for (var x = 4, S, i = w; --i; x += 8, n++){ if((S = sub[n]) ^ 0xfc30){ //go sub-tiles 4x4 x4: if(map[n]) ctx.drawImage(fuse, S << 2 & 12, S & 12, 4, 4, x, y, 4, 4); if(map[n+1]) ctx.drawImage(fuse, S >> 2 & 12, S >> 4 & 12, 4, 4, x + 4, y, 4, 4); if(map[n+w]) ctx.drawImage(fuse, S >> 6 & 12, S >> 8 & 12, 4, 4, x, y + 4, 4, 4); if(map[n+w+1]) ctx.drawImage(fuse, S >> 10 & 12, S >> 12 & 12, 4, 4, x + 4, y + 4, 4, 4); }else if(map[n]) ctx.drawImage(full, x, y); //full sub-tile block 4x4 }; }; }

4.拡張機能/追加機能
-なぜなら 一般に、xfcRSはレンダリングについては何も言わず、インデックス付きの境界テーブルのみを提供します。このテーブルに追加データを追加することができます。これは、大きな均一な領域内に「パス」を配置するのに役立ちます。 外部データソースから取得したこれらのパスは、周囲のセルが変更されてもブロックされません。 また、最初は空のスペースに「配置」されているため、パスが表示されないことも興味深いです。 そしてそれらを見つけるためには、小麦粉を特別に振りかける必要があります（灰、ゲームデザイナーの好みに応じて選択してください）。

-xfcRSは、ローカルエリアを処理するように変更できます（この場合も、トリミングおよびスムージングされたサブセルが目的のサブセルにどのように関連するかを想像する必要があります）。

これにより、ローカルマップの変更を非常に効率的に再描画できます。
