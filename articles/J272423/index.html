グラフィックス



意味


Qtでは、3DグラフィックスにOpenGL ES 2.0を使用できます。次のクラスがこれに役立ちます。

  • QGLWidgetは廃止されたクラスで、QWidgetの子孫です。 Qt 5.0より前に使用
  • QOpenGLWidget-新しいクラス。QGLWidgetと同じ機能を実行しますが、その実装で中間フレームバッファーを使用します
  • QOpenGLWindow -OpenGLグラフィックスを表示するように設計されています。ウィジェットではなく、依存関係ウィジェットモジュールをプルしません




これらのウィジェットを試して、いくつかの結論を出しました。

  • QGLWidgetは、弱いスマートフォンではQOpenGLWidgetよりもはるかに高速です
  • QOpenGLWidgetは、ウィンドウのサイズを変更するときに内部フレームバッファーのサイズを変更しません。おそらくバグまたは機能です。射影行列。 このすべてについて、このウィジェットを使用したパフォーマンスは非常に小さかった
  • QOpenGLWindowは最適な候補です。 グラフィックシーンのビューポートとして使用できず、子ウィジェットを追加できないため、Qtを使用してユーザーインターフェイスを作成できません。 しかし、彼は3つすべての中で最も生産的です




ゲームを実装するには、QOpenGLWindowを選択し、ユーザーインターフェイスを自分で循環させる必要がありました。



フレームの更新は、swapBuffersの後に生成されるframeSwapped（）シグナルを使用して実装されました。 この信号を使用すると、タイマーを使用する場合よりもスムーズなフレーム変更を実現できます。



アニメーションを計算するには、フレーム時間を計算する必要がありました。 これを行うために、私はQTimeを最初に使用しましたが、これは悪い考えです。このクラスは不均一な時間を考慮しているためです。 また、このクラスを変更する権限はミリ秒に制限されていますが、これは不安定なFPSでのスムーズなアニメーションには不十分です。



ドキュメントを考えて参照した後、 QElapsedTimerクラスを使用することにしました。 このクラスは単調な時間を使用しようとし、最大ナノ秒の分解能を持ちます。

最後から2番目のバージョンのスクリーンショット




レベルテクスチャは、すべてのブロックの大きなアトラスであり、16ピクセルの辺を持つ32個のキューブを収容します。 このアプローチでは、レイヤーを描画するときに、現在のテクスチャを常に再インストールする必要はありませんが、複数の呼び出しで頂点バッファーを使用してレベル全体を描画できます。



問題は、他のブロックに隣接するブロックの極端なピクセルで発生しました。 テクスチャ座標を丸めると、アーチファクトがブロックの端にストライプの形で現れ始めました。 この問題を解決するには、ブロックのエッジのピクセルを複製する幅1ピクセルの境界線を各ブロックに追加しました。 そのような独特のCLAMP_TO_EDGE。



残念ながら、このアプローチでは、レベルにmipmapを使用できませんでした。 頂点バッファーのテクスチャ座標は、このオフセットを1ピクセル分考慮する必要があります。詳細が低減されたテクスチャを作成する場合、このオフセットは小さくなり、詳細レベルに応じて異なるレベルのジオメトリを使用する必要があります。 私はこれを気にしないことに決めました、なぜなら テクスチャ自体のレベルは256x256ピクセルしかないため、バイリニアフィルタリングではパフォーマンスはそれほど向上しませんが、実装にはさらに困難が伴います。



また、32個のブロックしかありませんが、レベルプロパティでは、90、180、および270度回転したブロックを配置する機能を設定し、ブロックテクスチャ間の切り替えのアニメーションを設定して、ゲームの視覚的なコンポーネントを多様化することもできます。 ただし、ファンの回転の効果を作成するために、いずれかのレベルでのみアニメーションを適用しました。



シェーダー


Qtにはシェーダーを操作するための便利なクラスがあります。 QOpenGLShaderProgramを使用しました 。 このクラスを使用すると、頂点シェーダーとフラグメントシェーダーを追加し、それらをコンパイルし、リンクし、ユニフォームとattrubuteを設定できます。 クラス自体は多くのOpenGL呼び出しの単なるラッパーであり、したがって、このクラスの呼び出しの間にGL呼び出しを直接使用することでクラスを破壊できるという理解において、本格的なオブジェクトではありません。



便利なことに、ESシェーダーがデスクトップとモバイルデバイスの両方で正常にコンパイルされるように、クラスは自動的にdefineを追加します。 これは主に、デスクトップ上では何にもならない精度指定子に適用されます。



いくつかのブロックのライティングとアニメーション、キャラクターシェーダー、インターフェイスシェーダーなど、ゲームワールド用に個別のシェーダーを作成する必要がありました。

バーHPシールド



このバーをシェーダーで実装することにしました。最初は非常に奇妙なアイデアのように思えましたが、最終的には避けられないものとして受け入れ、次のように実装しました。

シェーダーコード 
#ifdef GL_ES precision highp int; #endif varying highp vec4 v_position; varying highp vec2 v_dim; uniform lowp vec4 u_color; uniform highp float u_tg; uniform highp float u_value_a; uniform highp float u_value_b; uniform highp int u_step; uniform lowp vec3 u_pallete[3]; void main() { int hstep = u_step/2; int w = int(v_dim.x); int h = int(v_dim.y); int wd = u_step*3; int pos_bl = int(v_position.x - u_tg*v_position.y + v_dim.y); int pos_br = int(v_position.x + u_tg*v_position.y); int pos_l = (pos_bl - wd*(pos_bl/wd))/u_step; lowp float b0 = float(pos_l == 0) * float(pos_bl <= int(u_value_a*v_dim.x)); lowp float b1 = float(pos_l == 1) * float(pos_bl >= int((1.0-u_value_b)*v_dim.x)); lowp float b2 = clamp(float(pos_l == 2) + 1.0-(b0+b1), 0.0, 1.0); highp float p = abs(2.0*(v_position.w - 0.5)); highp float out_p = (1.0 - 0.25*p); lowp float i = float(int(v_position.y) > hstep) * float(int(v_position.y) < h - hstep); lowp float o = (1.0-i)*float(pos_br >= h) * float(pos_bl <=w); lowp float a = i*float(pos_br >= h+u_step) * float(pos_bl <=w - u_step); lowp float b = i*float(pos_br < h+u_step) * float(pos_br >= h); lowp float c = i*float(pos_bl > w - u_step) * float(pos_bl <= w); highp float pr = (1.0 - p)*a; gl_FragColor = vec4( u_pallete[0]*pr*b0 + u_pallete[1]*pr*b1 + u_pallete[2]*pr*b2 + u_pallete[0]*out_p * b + u_pallete[1]*out_p * c + mix(u_pallete[0], u_pallete[1], v_position.z)*out_p*o, clamp(a+b+c+o, 0.0, 1.0)*u_color.a); }




また、さまざまなAndroidデバイスとそのグラフィックアクセラレータについても述べたいと思います。 私は問題に遭遇しました：

  • シェーダーコードは1つのデバイスで問題なくコンパイルされ、別のデバイスでエラーが発生します
  • シェーダーは正常にコンパイルされますが、一部のデバイスでは正しく機能しません。 たとえば、PowerVRでは、ローカル変数がユニフォームまたは属性と同じように呼び出された場合、これはエラーにはなりませんが、シェーダー自体は正しく動作しなくなります
  • 一部のデバイスでは、変数または関数が組み込み関数と同じように呼び出されるとエラーが発生します。 たとえば、可変ミックスまたはクランプ
  • 一部のデバイスでフロートの精度に問題がありました




このすべての結果-モバイルゲームでシェーダーを使用する場合は、グラフィックアクセラレーターの最も一般的なモデルでそれらをテストしてください。 シェーダーがコンパイルされてコンピューターで動作する場合、これは隣人の携帯電話で動作するという意味ではありません。 新しいVulkan APIは、さまざまなシェーダーコンパイラの問題を解決し、このクレイジーな世界に秩序をもたらすはずですが、これは未来のビジネスであり、今日、私たちは持っています。



音



検索する


音は一般的に別の歌です。 自分で録音することもできますが、これはかなり面倒で、通常のマイクと聴覚が必要です（私たちの場合ではありません）。 また、インターネット上で目的のサウンドを見つけることができます。

Creative Commons 0ライセンスでサウンドを検索して、数十個ある可能性のある各サウンドの作成者を示さないことが望ましいです。 通常の無料のサウンドを見つけるのはかなり難しいように思えるかもしれませんが、そうです。 問題は、それらが少数であるということではなく、それどころか、それらの多くがあり、そのほとんどが恐ろしいことです。 サウンド検索は、多くのサウンドを聞いて、最適なサウンドを選択する必要があるプロセスです。



手段


Qtには、サウンドを出力するためのQSound 、 QSoundEffect 、 QAudioOutput、およびQMediaPlayerクラスがあります。 最初のバージョンでは、QSoundEffectを使用してエフェクトを出力し、QMediaPlayerを使用してサウンドを出力しました。 しかし、結局のところ、それらはすべて適合していません。



QMediaPlayerのみが圧縮オーディオファイルを処理できますが、このクラス、より正確には、Androidでの実装にはいくつかの不快な瞬間があります。

  • 音の始まりでフリーズします。 音楽がループ状態で再生される場合、繰り返されると、アプリケーション全体に顕著な遅延が生じます。
  • リソースからメディアファイルを読み取ると、アプリケーションデータフォルダーに一時ファイルが作成され、それ自体が削除されるとは限らないため、アプリケーションのサイズが大きくなります。 それは偶然に発見されたのです。


QSoundおよびQSoundEffectは、非圧縮のwavファイルでのみ機能します。 QSoundEffectは遅延なしでサウンドを出力するように設計されており、サウンド自体をループさせることができますが、Androidで使用する場合、「AUDIO_OUTPUT_FLAG_FAST denied by client」というメッセージがしばしばログに表示されます。 これは、デバイスごとに異なる不正なサンプリングレートが原因です。 一部のデバイスは44KHzを飲み込み、一部のデバイスは48KHzを必要とし、オブジェクト自体はwavファイルに記録されているとおりにサウンドを送信します。 サウンドリソースのサイズは、アプリケーションのサイズのかなりの割合でした。



これらすべての欠点は、ゲームにサウンドを追加した後、サウンドと音楽を再生するときに具体的なFPSの沈下が現れるという事実につながりました。



解決策は、これらのクラスを放棄し、サウンドにSFMLライブラリを使用することでした。 SDLに似た非常にシンプルで軽量なライブラリ。 グラフィックス、サウンド、入力デバイスを操作するための便利なクラス。 このライブラリは、mp3（ライセンス、すべてのもの）の操作方法を知りませんが、それ以上のことを行います。 エフェクトと音楽にはogg形式を使用しました。

Google Playについて



開発者コンソール自体は非常に便利なツールですが、統計と広告の機能は他のアカウントに関連付けられています。



自分でアプリケーションを完全に開発するには、少なくともAdWords、AdSense、AdMob、Google Analyticsアカウントが必要です。 この場合、接続はそれらの間で確立されます。 これらのアカウントはすべて個別のGoogleサービスであり、さまざまな技術サポートと設定があります。 また、AdMobアカウントにはAdWordsとAdSenseアカウントが必要です。 ただし、これらすべてのアカウントは、1つのコピーでメインのGmailアカウントにリンクできます。 しかし、実践が示しているように、あるサービスを開くと、別のサービス、3番目のサービスなどに新しいアカウントを作成するように提供されるため、最初からこのすべてに混乱する可能性があります。



どういうわけか、魔法のように、技術サポート担当者は何が起こったのか説明できず、2つのAdWordsアカウントを作成して1つのメールにリンクし、一方1つのアカウントを開発者のコ​​ンソールに、もう1つのアカウントをAdMobにリンクしました（これについては知りませんでした）。



広告キャンペーンをチェックするために1つのアカウントに500rを投げました。 これに対処するために、テクニカルサポートのヒントに従って、アカウントの1つを「左」のメールに転送し、アクセスを閉じました。 これはすべて、最初に私のメールから両方のアカウントを操作できなくなり、その後、自分の切断と接続を繰り返して、パフォーマンスが戻りました。 しかし、結局のところ、AdMobは機能しなくなりました。 AdMobは500rよりも重要だったので、AdMobが機能するようにすべてにアクセスできなくなることを祈りながら、この手順全体を新たに実行する必要がありました。 そして、もちろん、それらの500rは接続されていないアカウントにハングアップしたままになりました。



したがって、これには注意してください。



翻訳



ゲームで


ゲームメニューをロシア語と英語の2つの言語に翻訳しました。 ゲーム言語の選択は、システムロケールによって実行されます。 システムがロシア語である場合、ゲームはロシア語で、その他の場合はすべて英語で行われます。



Qtのテキスト情報を翻訳するために、クラスによって実行される組み込みのメカニズムがあります 
  QTranslator myTranslator; myTranslator.load(":/translations/neverfall_" + QLocale::system().name()); a.installTranslator(&myTranslator);


翻訳が必要なすべての行はQObject :: tr（）関数に渡されます。QObjectの子孫ではないクラスの場合、QApplication :: translate関数を使用できます。配列で宣言された文字列にはマクロQT_TRANSLATE_NOOP、QT_TR_NOOPを使用できます。



しかし、これは戦いの半分に過ぎません。 翻訳自体を作成する必要があります。これは、lupdateおよびlreleaseプログラムによって行われます。 1つ目は、これらの関数とマクロを含むソースコードから情報を収集し、翻訳用の情報を含むXmlファイルを作成します。

2番目は、xmlファイルからqmバイナリファイルを収集します。このファイルはQTranslatorに直接ロードされます 。
