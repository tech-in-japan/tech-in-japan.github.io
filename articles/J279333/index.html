Oracle Databaseの新しいバージョンを開発する過程で、オラクルは、現代のIT業界の2つの主な傾向を考慮することが重要でした。 まず、最近の特徴である価格と利用可能なRAMボリュームの傾向。 結局、RAMのコストは毎年30％低下し、今日の一般的な企業サーバーには128 GBのメモリ容量が搭載されており、多くのサーバーには1 TBのメモリが搭載されています。 これは、データベースをRAMに直接配置する方法を学習すると、データベースへのクエリが数十倍から数百倍速く実行され、リアルタイムのビジネスインテリジェンスの可能性が開かれることを意味します。







第二に、ITコストの低下に直面して、開発者は、より迅速に革新し、アプリケーションサポートを簡素化できる新しいツールが必要であることを覚えておくことが重要です。 したがって、たとえば、新しい開発を既存の企業インフラストラクチャと統合するために、開発者は、複雑で開発が困難なモノリシックアプリケーションから、マイクロサービスのアーキテクチャ、つまり、独立して展開されるサービスのセットであるアプリケーションに移行しています。 また、新しいアーキテクチャを使用するには、データベースが新しいツールと新しいプログラミング方法をサポートしている必要があります。

Data Guard Far Syncと呼ばれる別の新しいテクノロジーであるOracle Database 12cは、長距離にわたってデータ損失をゼロにし、メインデータベースから遠く離れた場所にデータベースバックアップを保持できるようにします。 データファイルを持たない追加の特別なデータベースインスタンスは、同期モードでメインデータベースからの変更を受け入れ、これらの変更をリモートデータベースインスタンスに非同期的に転送します。これにより、同期モードの信頼性と非同期モードのパフォーマンスの両方が保証されます。

Oracle Database 12.1.0.2：インメモリー



インメモリの開発において、オラクルは、リアルタイムのビジネス上の意思決定のためのリアルタイム分析を可能にするテクノロジーを作成しようとしました。 Oracleの競合他社がIn-Memoryオプションを使用するために別のデータベースや他のテクノロジーを必要とする場合、Oracle Database In-Memoryオプションがデータベースに組み込まれ、1つのパラメーターでオンになり、アプリケーションに対して完全に透過的であり、すべてと互換性があることが非常に重要ですデータベース機能。 このオプションを使用するお客様の経験から、トランザクション処理は2倍に加速され、行は通常よりも3〜4倍速く挿入され、分析の要求は実際にほぼ瞬時にリアルタイムで実行されます。



この技術の意味は、テーブルの行とインデックスブロックを格納する通常のバッファキャッシュの隣に、RAM内のデータ用の新しい共有領域が作成され、列形式で格納されることです（図1）。 したがって、このテクノロジーは同じテーブルデータに対して行と列の両方のストレージ形式を使用し、データはアクティブでトランザクション的に一貫性があります。 すべての変更は、最初に従来のバッファキャッシュで行われ、次に列キャッシュに反映されます。



同時に、テーブルのみが列キャッシュに反映され、インデックスはキャッシュされません。 また、データは読み取られても変更されていない場合、バッファキャッシュに保存する必要はありませんが、データが変更された場合、キャッシュとバッファキャッシュの両方に保存されます。 そのため、分析では列ストレージがより効率的であるため、インメモリは分析作業を高速化します。



さらに、In-Memoryオプションを使用すると、パフォーマンスを犠牲にすることなく分析インデックスを削除できます。これにより、ディスク領域が節約され、In-Memoryにある任意の列を使用してクエリを作成でき、クエリをすばやく作成するために追加のインデックスを作成する必要がありません。







Oracle Database In-Memoryの重要な要素は、ハードウェアのサポートです。 特に、このテクノロジーは、グラフィックスの処理を目的とした一連のSIMD命令（単一命令複数データ値）をサポートします-In-Memoryは、これらの命令をプロセッサに組み込む場合、これらの命令を使用して列の複数の値を一度に述語と比較し、列のスキャン速度を大幅に加速します1秒あたり最大10億行。

その他の改善



  • 高度なインデックス圧縮は、インデックスを圧縮してディスク容量を削減し（一部のデータベースでは、インデックスがディスク容量の半分を占有します）、キャッシュをより効率的に使用します。
  • 完全なデータベースキャッシュ。 自動的にオンになり、使用可能なすべてのメモリを最大限に活用し、データベースがメモリに収まる場合にパフォーマンスを潜在的に改善します。 ALTER DATABASE FORCE FULL DATABASE CACHINGコマンドを使用すると、NOCACHE LOBオブジェクトを含むテーブルを含む完全なキャッシュを強制的に有効にすることができます。
  • 大きなテーブルの自動キャッシュ。 データベースがメモリ全体に収まらないが、いくつかの大きなオブジェクトが配置されている場合に使用できます。 DB_BIG_TABLE_CACHE_PERCENT_TARGETパラメータを使用すると、バッファキャッシュ内の大きなテーブル用に個別の領域を選択できます。 通常のバッファーキャッシュデータがブロックレベルでキャッシュされる場合、大きなテーブルはキャッシュされ、完全にそれらのアクセス頻度に基づいてこのキャッシュ領域から削除されます。
  • テーブルに設定されたAttribute Clusteringディレクティブは、同じ列値を持つ行がディスク上にある列値でデータを配置します。 このディレクティブは、レコードの一括挿入やテーブルの移動などの直接データ読み込み操作中に機能します。 属性クラスタリングは、次のようにデータ圧縮に役立ちます。 高度な圧縮オプションを使用すると、順序付けされたデータの圧縮が向上します。 ただし、Oracle Database 12.1.0.2の別の新機能であるゾーンマップと共有する場合、属性クラスタリングは最大のメリットをもたらします。 Oracle ExadataまたはSuperclusterで使用可能なゾーン・マップには、行範囲に指定された列の最小値と最大値が含まれており、問合せの実行時に不要なデータを迅速に除外できます。 これらのテクノロジーはアプリケーションに対して完全に透過的であり、クエリのパフォーマンスを向上させ、物理読み取りの数を減らし、高度に選択的なクエリのI / Oの数を大幅に減らし、ディスクの使用を最適化します。
  • 概算カウントは、異なる列値の概算カウントの関数です（つまり、すべてのクエリが正確な結果を必要とするわけではありません-たとえば、「先週私たちのサイトに何人の異なる訪問者がいましたか？」という質問に対する回答です）正確な計算よりも高速で、95％の信頼係数で97％以上の精度を実現します。
