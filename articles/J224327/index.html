エンタープライズ1Cサーバーを実装するときにライセンスを節約する例

こんにちは、この記事はこの作品の続きとして書かれました。 1C会社はしばしば批判され、多くの場合客観的ですが、1Cは選択の自由を提供することを私の例で示したいと思います。少なくとも現時点では敬意に値します。 また、少しのお金を数えます。





プロローグ。





私たちの小さな会社の主な活動は、ITアウトソーシングです。 むしろ、マーケティングと熱狂的な目的のために、私たちは病棟のITインフラストラクチャを少し標準化できるソリューションテンプレートを作成し、クライアントは節約を実現するための最も重要なものを受け取ります（レポートを賞賛しなければ、誰も気付かないでしょう）。 クライアントは20〜200人の小規模企業です。 そのようなソリューションの1つは、無料のLinux + Postgres SQLプラットフォームでの1Cエンタープライズサーバーの実装です。 長い間噛み砕かれてきたため、この記事には別の技術的な実装はありません。 1Cフランチャイジーの標準オファーと2014年5月の経済的オプションの比較のみが行われます。



タスク番号1。





データベースをファイルモードからSQLバージョンに移行します。最大20クライアントを使用できます。



2つのオプションの計算。





1Cをサポートしていないため、すべての推奨事項：ファイルベースからSQLへの切り替え、ライセンスの購入、およびハードウェアの必要性について、クライアントは付属の1Cフランチャイジーから受け取ります。 次に、私たちと相談し、1C + Linux + Postgres SQLに基づく代替ソリューションを提供し、自分で実装します。

説明とニュアンス：





  1. 1Cの実装者（宇宙ではもちろん、クライアントでお金を節約しようとしている人もいますが、私たちはこれらに遭遇していません）デフォルトでUSBオプションキーを提供し、ファイルライセンスよりもかなり高価です。 当然、キータイプの選択は実装プラットフォームに依存しません。 この表は、Linuxプラットフォームに有利なトリックであることがわかりました。 それでも、これは提案の綿密な評価ではなく、実践の新鮮な例であることを思い出させてください。 客観性のために、私の意見では、実装者は支出を増やすためではなくUSBキーを好む傾向があることを明確にしなければなりませんが、使用の信頼性とさらなるメンテナンスの容易さのために、Linux + Postgres SQLの実装に関しては「なおさら」です。
  2. 多くの場合、小規模企業向けに64ではなく1C：Enterprise x86キーも購入しますが、この場合、Postgres SQLは64ビットバージョンを使用し、1Cエンタープライズサーバーは32ビットです。 60人までの組織の使用は、容認できると考え、論文は主観的です。
  3. 作業コストは考慮されません。 この例では、サービスコントラクトに含まれているため、クライアントではゼロになります。 実装とサポートはほぼ同じであると想定しています。

例1からのニュアンスへの追加





  • Kind 1Cでは、エンタープライズサーバーキーなしで12クライアント用にLinux上で1Cエンタープライズサーバーを使用できますが、Windowsにはありません。 10人のユーザーがファイルで生き残ることができるので、ボーナスは疑わしいですが、それでもいいです。






結果。





多くの場合、国内での貯蓄は、あらゆる種類のシステム事業を台無しにします。 このケースはまだ別のカテゴリに属しているように思えます。 3年前、Linuxに1Cエンタープライズサーバーを企業の標準として導入したとき、私たちは1C実装者が、同様の「コーシャ」ではないLinux + Postgres SQL接続でサポートされる構成の機能について責任を負うことを、本当に心配せずに聞きましたこれにより、クライアントがパニック状態になります。

計算にさらに数十個の重要な矢印を引くことができる可能性があり、客観性を主張することは困難ですが、問題の財務的要素の一般的な考え方を伝えたかったのです。
