現代の世界はAPIにかかっています

今日の世界は、アプリケーションプログラミングインターフェイス-APIにかかっています。 それらにより、ネットワークに接続されたWebアプリケーション、モバイルアプリケーション、およびデバイスを介してデータを受信し、サービスを利用することが可能になりました。 ますます、インターネット上の相互作用はAPIを介して実行されます。 APIのおかげで、新しいビジネスモデルが出現しており、インターネットは普遍的なビジネスプラットフォームになっています。







APIには産業上の拘束力はありません。経済のさまざまな分野の企業は、その用途にビジネスの価値を見出しています。 GartnerとForresterのレポートによると、API管理ソフトウェアの市場は急速に成長しています。

















ほんの数年前、同じビジネスの異なる部門間の相互作用は通常、統合バスを介して提供されていました。 しかし、APIポータル（APIが公開されているポータル）を介した対話のモデルは非常に便利であることが判明したため、現在では企業によって内部的に使用されています。







部門間の相互作用のモデルを選択したとしても、今日の企業はAPIに基づいたソリューションに傾倒しているのはどうしてでしょうか。 現在の技術モデルの本質は何ですか？ゲームの新しいルールは何ですか？







オープンAPI-MODまたは必要性？



オープンAPIを使用することは、時代の流行やトレンドだけでなく、市場の要件への対応でもあります。 銀行、通信会社、保険会社はすでに、外部での使用、パートナーとの統合、財務フローの自動化のためにサービスを公開しています。 エンターテインメント、運用サービス、物資のサプライヤが参加する日はそう遠くないようです。







欧州では、金融フローの革新への関心は、革新、競争、安全を促進する、より均一で透明で開かれた決済市場を創出するために発行された欧州議会の支払い指令PSD2によってサポートされています。 ロシアでは、オープンAPIの開発は、金融市場参加者のシステムを効果的に統合するために必要な重要な要素として公式に認められています。







ロシアの州とその金融部門は、すでにオープンバンキングの必要性を認識しています。 銀行APIを外部組織に提供することは、金融市場参加者のシステムを効果的に統合するための重要な要素として認識されています。オープンAPIをリリースするイニシアチブは、中央銀行、Banki.ruポータル、モスクワ証券取引所、国立清算センター、国立決済預託機関によってサポートされています。 一部の銀行はすでにオープンバンキング戦略を策定し、さらなるアクションのモデルを決定し、オープンAPIを介してシステムおよびサービスへのアクセスを正式に発表し、対応する作業を開始しました。











webMethods APIポータルのAPIのリスト

国内のモバイルオペレーターは、パートナーのビジネスを開発するためのAPIを備えた新しいプラットフォームも提供しています。 これにより、電気通信プロバイダーは、オファーを組み合わせて販売市場を拡大することで、パートナーをサポートできます。







ロシアの銀行と通信プロバイダーは、まさにソフトウェア開発者として最初に認められた企業であり、市場は製品の管理、マーケティングキャンペーンの設定、潜在的な顧客とのやり取りのための大規模なデジタルプラットフォームとして認識されています。 製品チーム、顧客、企業、および顧客は、オープンであるほど、製品がよりオープンになり、事業を展開する市場の共通のエコシステムにより早く統合されることを理解しています。 したがって、オープンAPIを使用します。これは、開発者がやり取りする合理的かつ効果的な方法であり、新製品が市場に参入するまでの時間を大幅に短縮できます。







さらに、オープンAPIは、Yandexなどのソフトウェア開発者によってパートナーに提示されます。 Russian Postは、APIを介した外部アプリケーションとの統合も提供します。これにより、サードパーティのサイト、アプリケーション、経理、ドキュメント管理システムにRussian Postサービスを埋め込むことができます-たとえば、サイトに追跡機能を追加します。







そしてもちろん、オープンAPIを使用した製品の作成は、Software AGなどのソフトウェア開発者自身にとって当然のことです。 製品がより完全に文書化され、より適切に管理されるほど、より多くのユーザーを獲得できます。







しかし、オープンAPI管理は上記の誰にも与えられていません。 適切な技術スタックがなければ不可能です。







APIプラットフォームの開発者とその仕組み



前述のGartner Magic Quadrantによると、APIフルライフサイクル管理システムの市場のリーダーは、Google、CA Technologies、IBM、Software AG、MuleSoft、Red Hat、TIBCO Softwareです。 最近の調査では、ForresterはIBM、Google、Software AG、Rogue Wave Software、およびWSO2のリーダーを呼び出しています。







Forresterのレポートによると、「APIはデジタルトランスフォーメーションの重要な基盤です。 顧客体験の最適化、顧客とパートナーの統合デジタルエコシステムの作成、企業による画期的なデジタルイノベーションの活用、運用効率の向上、プラットフォームビジネスモデルの基盤の構築を支援します... API管理ソリューションは、サプライヤーとAPIのユーザー間の関係を管理する中心的な役割を果たします「開発者とアプリケーションプロバイダーは、それらをデジタルビジネスの成功に不可欠なビジネスアプリケーションと見なすべきです。」













API管理インターフェイス



「完全なAPIライフサイクル管理がなければ、デジタル戦略のプラットフォームを作成し、エコシステムを構築し、効果的な製品を発売することはできません」とガートナーはレポートに追加しています。







APIライフサイクル全体を管理するためにシステムは何を提供しますか？ 通常、APIライフサイクル管理スタックには、読みやすいポータル上のAPI公開ツールが含まれています。その主なユーザーはサードパーティの開発者であり、操作環境、消費、メンテナンス、APIのバージョン管理、および廃止措置の手段です。 一部の開発者（Software AGを含む）は、APIの計画、設計、実装、およびテストツールも提供します。

Software AGでは、APIの管理を担当していましたが、「内部対話」とも呼ばれていました。 ミドルウェア、アプリケーション統合ソリューション、エンタープライズサービスバスを作成するためのシステム、およびサービス指向アーキテクチャに基づいてシステムを作成するためのツールを拡張および改善しました。







2004年には、統合バスに加えて、パートナー間の相互作用とデータ交換のために設計されたB2B Trading Networks製品を作成しました。 取引日の結果に基づく継続的な監視、サービス、データ交換など、パートナー関係の非常に古典的なユーザーシナリオを実装しました。 その後、まだオープンAPIと呼ばれていませんでした。







最後に、5年前、webMethods API管理プラットフォームの一部として完全なAPI管理ライフサイクルを導入しました。 2014年にAPI開発者向けにwebMethods API Portalを開始し、2016年にwebMethods API Gateway API 、ポータル、メディエーション、ライフサイクル管理ツールの機能を1つのプラットフォームに統合しました。 これらのツールは、採用された技術標準でのAPIの開発、アセンブリ、承認、公開をサポートし、Software AG Hybrid Integration＆APIプラットフォームの一部です。

APIプラットフォームの選択方法



Forresterは、APIを管理するためのソリューションを選択する際には、提案されたソリューションが複雑かどうかを最初に考慮する必要があると考えています。つまり、API開発者向けのポータル、API管理用のポータル、APIゲートウェイが含まれていると考えています。 一部のソリューションは、API設計および開発ツール、統合プラットフォーム、リアルタイムサービス管理プラットフォームなどの追加コンポーネントを提供することが強調されています。







Forresterは、API管理ソリューションは、関連するプラットフォーム、統合製品、またはビジネスアプリケーションから分離可能な、真のスタンドアロン製品でなければならないことをさらに強調しています。







最後に、このレポートの著者は、多数の本格的な実装を行っているソリューション開発者を信頼する価値があると考えています。 Software AGのAPI管理ソリューションクライアントには、Michael Kors（ハイエンドのアパレルおよびアクセサリーのメーカーおよびサプライヤー）、American Electric Power（北米最大のエネルギー企業の1つ）、Outerwall（小売販売用の自動キオスクのサプライヤー）、Dick's Sporting Goods（スポーツの小売ネットワーク）が含まれます財）、EDF（フランス最大のエネルギー会社および世界最大の原子力発電所）など。







APIプラットフォームを選択する際に考慮する必要があるこのパラメーターのリストに、さらにいくつかの要因を追加する必要があります。







1.産業ごとに、経済の仕組みは異なり、収益化スキームも異なります。 検討しているAPIプラットフォームの開発計画を評価します。 それはあなたの事業セグメントの現実を反映していますか？ 実装のビジネスタスクを決定し、ソリューションのビジネス要件のリストを作成し、そこから機能要件およびアーキテクチャ要件のリストを導出することが重要です。 おそらく、このリストは、APIソリューションだけでなく、追加のコンポーネントの選択も決定するでしょう。













APIポリシー管理



2. APIプラットフォームが顧客、またはむしろIT部門の期待に応えることが非常に重要です。 プラットフォームは実装と運用に便利である必要があり、顧客（クラウド、物理、またはハイブリッド）にとって快適な技術展開モデルをサポートする必要があり、その機能は現在のニーズに対応する必要があり、その開発計画は1年または2年前の将来のニーズに対応する必要があります。







3. APIポータルには、広範な分析機能、開発者向けのテストインターフェイス、およびAPIメタデータに基づいてドキュメントを生成する機能が必要です。 開発者の社会的協力、クライアントSDKの生成、収益化の手段を提供する必要があります。

4. APIゲートウェイは、セキュリティ（認証、承認、セキュリティポリシーの管理、攻撃に対する保護）、サービスの仲介、ルーティング、および負荷分散機能を提供する必要があります。











ユーザー登録確認



5. APIライフサイクル管理ツールは、内部サービスと外部サービス、マイクロサービスと従来のサービス、技術サービスとビジネスサービスの関係、およびカタログ内のさまざまな種類の「資産」のサポートを提供および評価する必要があります。







6.製品開発の速度と市場投入までの時間に依存する意思決定の総所有コストの問題は非常に重要です。これは、開発者が採用する慣行と彼らが使用するテクノロジーの影響を受けます。







7.多くの場合、APIプラットフォームの開発者が答えを持たない質問-顧客とパートナーの間で契約がどのように作成され、請求がどのように行われるか-ベンダーは、契約を作成する技術的可能性の実装に関する推奨事項を持っている可能性が最も高いです。
