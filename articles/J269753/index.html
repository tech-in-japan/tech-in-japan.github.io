Symantec Backup Hot-Addの機能のテスト。 コピーとデータ回復の速度を上げる

2014年、Trinity Uralは、4つのSuperMicroサーバーノードと、高可用性機能を提供するVMware vSphereソフトウェアを実行する外部EMC VNX 5400ストレージシステムを含む一連の機器にConvenient Moneyを導入しました。 Symantec Backup Execがバックアップ/リカバリツールとして選択されました。



今日、マイクロクレジットサービスの需要が集中的に増加している状況で、「Convenient Money」という会社は地域での地理的存在感を拡大しています。 重要な点は、絶えず成長するデータの安全性を確保することです。この点に関して、バックアップとリカバリのシステムには細心の注意が払われました。



顧客は、Symantec Backup Execを使用した現在のコピー速度で、データベースの成長に関連してバックアップ規制を遵守するという問題に直面していました。 生成されたSymantec Buckup Execパフォーマンスレポートは、LAN上で約1,033MB /分のデータコピー速度を記録しましたが、割り当てられた時間内にメインSQLデータベース（1TB）をコピーするには不十分でした（図1）。





図 1.バックアップ速度の割合。



現在のバックアップデータベースの安全性を最小限の材料コストで保証しながら、1 TBのボリュームを超えるデータのバックアップとリカバリの速度を上げるというタスクが与えられています。

SQLデータベースのバックアップ速度を向上させる可能性と、Symantec Buckup Execを使用した最大可能バックアップ速度についての疑問が生じました。



ソリューションを解決するために、FC 8 G /ビットを介してIBM Storwize V7000ストレージシステム（8 x 300 GB SAS 10KプールRAID）に接続されたサーバーノードでVMware ESXI仮想インフラストラクチャが再作成された（バージョン5.5）トリニティウラルサイトにスタンドが設置されました0は読み取り用、プールは8 x 300 GB SAS 10K（書き込み用RAID 0）、SQLデータベースは仮想サーバーに展開されました。





図2。 LANポートを集約し、Symantec Backup Hot-Add機能を使用する場合のコピー速度。



テストの前に、2つの主要な仮定が行われました。



  1. 問題は帯域幅が不十分であるため、コピーの速度を上げるにはLANポートを集約する必要があります。
  2. おそらくVMwareのソフトウェアの制限により制限が課せられます。


テストの最初の段階では、LANポートの集約によってデータのコピーおよび復元の速度を大幅に上げることはできないことが判明しました。テスト中に各側で2つ/ 4つの集約ポートを使用し、集約をオフにしてもコピー速度は変化しませんでした（図2）。



Symantec Backup Hot-Add（SANによるデータベースのコピー/復元（FC 8 G /ビット））の機能をテストすることが決定されました（図3）。 Backup Exec 2012でSAN（Direct SAN）を介したバックアップを使用する場合、仮想マシンのリカバリ機能は機能しません。 バックアップホットアドを使用すると（図2）、LANを使用した場合よりもコピー/復元の速度が少なくとも2倍速いことが判明しました。 したがって、ホットアド機能はバックアップにプラスの効果をもたらします。SANを介したバックアップの特定のケースでは、速度はチャネルの帯域幅ではなく、アレイ内のスピンドルの数によって制限されました。





図3 Symantec Backup Hot-Addバックアップスキーム。



このコピー/復元速度の1 TBデータベースの場合、LANバックアップには約9時間かかりますが、SANには約4.5時間かかります。



さらに、SQLエージェントツールを使用してSQLデータベースのバックアップ/復元をテストしたところ、いくつかの制限が特定されました。



  1. データベースが別のRDM（未加工デバイスマッピング）ディスクにある場合、データベースは、デフォルトで物理設定のディスクタイプが仮想の場合にのみバックアップされます。 vmdkファイルからのデータベースバックアップと比較して、RDM仮想ディスクからのバックアップ速度は同じです（読み取り速度300 MB / s、書き込み速度50 MB / sから300 MB / sに変動します（テストでは、物理ハードドライブStorwizeの速度に依存します）。
  2. データベースが仮想モードの別のRDMディスクにあり、オペレーティングシステム内でディスクがGPT（GUIDパーティションテーブル）として使用されている場合、バックアップエラーが発生しますwww.symantec.com/business/support/index?page=content&id=TECH69498&actp=search&viewlocale= en_US＆searchid = 1429181110280 GRTを無効にするか、ドライブをMBR（マスターブートレコード）として使用する必要があります。
  3. 増分バックアップの実行の失敗。 データベース全体のバックアップが作成され、データベース内の行が変更され、増分バックアップが開始されると、データベース全体がコピーされます。


Convenient Moneyの場合、1 TBのデータベースでは、データベースをオペレーティングシステムと共に1つのVMDK（仮想マシンディスク）ファイルに配置するか、仮想MBRモードの別のRDMディスクに配置するオプションを使用します。 仮想マシンにあるSQLデータベースの増分バックアップはできません。フルのみです。



ホットアド機能を使用すると、古いSymantec NetBackup製品に切り替えたり、既存のバックアップデータベースを保存したりすることなく、データのコピー/復元の速度を大幅に向上させることができます。



テスト完了時の論文：



  1. LANポートを集約する場合、コピー/復元速度の直線的な増加はありません。 4つのポートを集約する場合、LACPの2つの集約ポートと比較して、速度は実質的に変化しませんでした。
  2. シマンテックのホットアド機能を使用すると、既存のインフラストラクチャへの追加投資がなくても、コピー/復元の速度が大幅に向上します（サーバーとストレージで8 G /ビットの無料FCポートが利用できる場合を除く）。
  3. VMwareソフトウェアのコピー制限は確認されていません。
  4. データベース用のSQLエージェントを使用してコピー/復元を構成する機能があります。特に、完全バックアップのみが可能です。


ITインフラストラクチャの監査は、既存のインフラストラクチャの動作を最適化するための推奨事項とともにお客様に対して実施され、SANバックアップ/復元を構成するアルゴリズムは、Symantec Backup Hot-Add機能を使用して策定され、バックアップ組織の規制を満たしています。
