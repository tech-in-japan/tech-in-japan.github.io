ソフトウェアローカリゼーションのための7つのオンラインサービスの概要

おそらく、インターフェイスが複数の言語を持っているすべての製品は、ローカライズプロセスを整理するという問題に直面しています。



実際、この質問は多言語アプリケーションだけではありません。 機能を設計するとき、そのために必要なすべてのテキストが事前に作成されることは常に遠いため、開発者は頻繁に「ドラフト」テキストを使用します。 通常は次のようになります。「まあ、ここまで何かを書いたら、それを修正します。」



このレビューでは、ソフトウェアのローカリゼーションのために7つのオンラインサービスをテストする印象を説明し、主要な指標に従って比較します（記事の最後の表を参照）。



免責事項 ：このレビューは完全または客観的であることを意図していません。 特定のRuby on Railsプロジェクトで翻訳プロセスを整理する問題を解決するプロセスでテストしたいくつかのサービスで作業するという印象を共有することにしました。 おそらく、短所はプラスのように見え、その逆もありますが、それはすべてプロセスと個人的な印象に依存します。 また、私自身は翻訳者ではないため、重要なポイントを見逃す可能性があります。



各サービスについて、その主要な機能を自由形式で説明し、記事の最後に小さな比較プレートを作成します。 すべてのスクリーンショットはクリック可能です。 さあ、行こう！



localeapp.com



私たちのアプリケーションはRuby on Railsで書かれているため、当然、まず魅力的なスローガン「Railsアプリのローカライズ」でこのサービスに注目しました。 一般に、 Ruby on Rails I18nの記事には、開発者-開発者、クライアント-充填というこのサービスの簡単な概要が既に記載されています。 サービスは残りを処理しますが、それでもレビューをより全体的にするために印象を書きます。



今後、localeappには1つの欠点があります（ただし、唯一の欠点ではありません）。そのため、使用を拒否しました。 実際には、翻訳するだけでなく、デフォルトロケールのレビューも行う必要がありました。 また、localeappはインポートされたファイルのすべてのキーを「Complete」としてマークします。 つまり、デフォルトのロケールをそこにロードしたので、すべてが「完全」であることがわかりました。レビュー用にマークするには、キーごとに手動でこれを行う必要があります。 私はこれについて開発者に手紙を書いたが、彼らはイエスと言った：「...完全なロケールを不完全としてバッチマークする簡単な方法はない」 私たちにとって、これは、アプリケーションで架空のロケールを開始し、未承認の翻訳を追加し、レビューを進めながら、それらを通常のロケールに転送する必要があることを意味します。 まあ、私は何を言うことができます...そのような単純なバッチ操作の可能性の欠如のために、サービスは顧客を失うことができるのは残念です。 そして、すべてが順調に始まり、このサービスが気に入りました。



翻訳者インターフェース：







主な機能：



  • このサービスは主にRailsアプリケーションに焦点を当てています（ただし、これは翻訳しかできないという意味ではありません）。
  • アプリケーションに接続し、1つのコマンドでロケールをインポート/エクスポートできる同じ名前のgemがあります。 また、このgemは、ページが更新されるときに、欠落しているすべての翻訳をlocaleappに自動的にプッシュできます（デフォルトでは開発時のみですが、構成できます）。 さて、最もおいしい： gemはステージングサーバーでデーモンとして実行でき、指定された間隔でlocaleappからの更新を自動的に遅延させてサーバーに適用するため、翻訳者は作業の結果をすぐに見ることができます。
  • 人気のあるライブラリとすべての接続された宝石からの翻訳言語リソースのリストに遅延して含めることができます。 ライブラリのキーは課金されませんが、そのセットには制限があります：devise / kaminari / rails v2.3 / rails v3 + more。
  • さまざまなキーステータスがありますが、完全と不完全の2つだけです。 コンテキストメニューには「レビュー用にマーク」機能もありますが、それが何をするのか正確にはわかりません。 目に見える変化はありません。
  • 主要な変更の履歴が保持されます（テキストをバージョン管理せずに、変更の事実のみ）。
  • ロールを持つプロジェクトにユーザーを追加する機能があります。

    -開発者（すべての機能とロケールへのアクセス）;

    -翻訳者（翻訳にのみアクセスし、キーを作成または削除できません）。
  • 素敵で、とても使いやすいインターフェース。
  • このインターフェイスでは、新しいキーを作成したり、キーを削除したりできます（トランスレーターの役割を除く）。
  • 検索は、検索文字列のエントリを、キーと翻訳テキスト自体のどこでもすぐに検索します。
  • 有料のプロ翻訳を注文することは可能です。
  • オープンソースプロジェクトの無料サポートがあります。
  • 小規模な閉鎖プロジェクト（最大500個のキーと最大2つのロケール）には無料の関税が適用されます。


弱点：



  • 翻訳のためにコメントやスクリーンショットを残す方法はありません。
  • 翻訳を未承認として一括でマークすることはできません。 つまり、デフォルトロケールのレビューは、ダミーロケールを使用して上記のライフハックを介してのみ実行できます。
  • 他のサービスと比較して、翻訳インターフェースの非常に貧弱な機能（ただし、誰かにとっては最小限のインターフェースがおそらくプラスになります）。
  • 私たちにとって最も重要なマイナス：公式の宝石は、非常に奇妙なことに、転送と逆インポートをマージします。 私たちと一緒に、彼は絶えず行を交換しました（変更がなかったとしても）、また（これは本当に不名誉です！）あるymlファイルから別のymlファイルに断片を押し込んだので、そのようなインポート後の変更を理解することは不可能でした。 私はこの問題が解決できることを除外しませんが、私はサポートに書き始めて理解しませんでした、なぜなら さまざまな理由でこのサービスを使用しないことにしました。 念頭に置いてください。




webtranslateit.com



ほとんどのプロジェクトに適合する機能的に非常に豊富なサービス。 ただし、一部の人にとっては、機会が多すぎると感じるかもしれません。



翻訳者インターフェース：







主な機能：



  • インターフェースの非常に豊富な機能：便利なフィルター、グループ操作など。
  • localeapp.comとは異なり、デフォルトのロケールで強制ロケールはありません。 デフォルトでインポートされたすべての新しいキーは、ステータスが「未読」になっています。 既存の翻訳を簡単に確認したり、グループ操作ですべてを「完了」とマークしたりできます。
  • 多くの異なるキーステータス：完了、未校正、未翻訳、検証のため、非表示、廃止。 ラベルにタグを追加することもできます。 おそらく、ビジネスプロセスを実装するのに十分です。
  • 検索はLocaleappと同じくらい便利です：すぐにどこでも検索します（キーとコンテンツで発生します）。 正規表現で検索することが可能です。
  • 各キーには、開発者からの短い付随テキストとディスカッションを添付できます。 開発者からのコメントは、コンテキストを確認できるページへのリンクを翻訳者に提供する必要がある場合に非常に役立ちます。
  • プロジェクト用の独自の用語集を作成（または別のプロジェクトからコピー）できます。 用語集の単語が翻訳されたテキストに表示される場合、サービスはその意味と提案された翻訳を表示します。
  • グローバル翻訳メモリから単語翻訳オプションを自動的に提供します。 確かに、それは奇妙な方法で行われます。フレーズが翻訳されると、翻訳オプションは最初の単語でのみ提供されます。
  • 各キーの変更履歴のサポート（翻訳テキストのバージョン管理を含む）。
  • ロールを持つプロジェクトにユーザーを追加する機能があります。

    -マネージャー（すべての機能と翻訳へのアクセス）;

    -翻訳者（指定されたロケールへの翻訳のみへのアクセス、およびオプションで翻訳を承認する機能）;

    -クライアント（翻訳の表示と読み取り/書き込みのみ可能）。
  • URLまたはクリップボードからの直接貼り付けにより、コンピューターからファイルをインポートできます。
  • 新しい非翻訳キーの出現イベントのサポートがあります。 新しいキーをチェックする期間（毎日/毎時）を設定し、これについて翻訳チームに通知する必要があることを示すことができます。 私の意見では非常に快適です。
  • Webフックのサポートもあります。 各転送後、サービスは指定されたURLにPOSTリクエストを送信し、行われた転送とそのステータスに関する情報を送信します。
  • クロスプラットフォームユーティリティのWebTranslateIt Command Line Interfaceがあり 、 コマンドラインから基本的な操作（プル/プッシュなど）を実行できます。
  • ワークフローオートメーションの記事では、ステージングサーバーで自動翻訳更新を実装するためのシナリオを提供しています。 Localeappとは異なり、既製のソリューションはありませんが、rubyで記述されたweb_translate_itユーティリティをベースとして、必要に応じて更新を提出することを提案します（例が添付されています）。
  • 完全なREST APIが存在する場合 。
  • Google翻訳とBing Translatorを接続して自動翻訳を行うことができます。
  • 小規模な閉鎖プロジェクト（最大500キー）には無料の関税が適用されます。


弱点：



  • ファイルを手動でダウンロードするだけで、何との自動同期もありません。 Github Webフックを理解し、翻訳の更新を遅らせることができますが、これは公開リポジトリにのみ適しており、 特定の設定が必要です。
  • 少しぎこちないインターフェイスは、ずさんな印象を与えます。 しかし、一般的に、すべてが直感的です。
  • オープンソースプロジェクトの無料サポートはありません。
  • サービスインターフェイスから有料翻訳を注文する方法はありません。




lingohub.com





このサービスには、webtranslateitのような翻訳の点で豊富な機能はありませんが、私には非常に注目に値するように思えました。 さらに、私の意見では、彼は言語リソースをプロジェクトと同期する（特にGithubを介して）最も便利で柔軟な実装を持っています。



翻訳者インターフェース：







主な機能：



  • githubと統合し、そこからすべてのymlファイルをインポートできます（指定されたブランチにコミットするときに、リポジトリから更新を手動および自動で更新できます）。 さまざまなインポート戦略をサポートします。
  • 選択したブランチのgithubに変更をプッシュバックできます（これは、ボタンを押すことで手動でのみ行われます）。
  • Githubと統合する場合、RoR、yii、CakePHP、Androis、iOSアプリケーションなどのプロジェクトの言語ファイルの変更を追跡するために必要な、最も一般的なウォッチパターンのセット-マスクがあります。
  • アプリケーションの翻訳を更新するためのgemがあります。
  • コメントを追加して、スクリーンショットを翻訳にアップロードできます。
  • 2種類の翻訳エディターインターフェイス（クラシックと更新済み）があります。 更新されたものの方が好きでした（スクリーンショットでは彼でした）。 古典的な形式では、キー自体は表示されませんが、何らかの理由でこれは緊張します。 更新されたフォームでは、キーボードのホットキーを使用すると便利です（習得する必要があるのはそれらだけで、難しいことはありません）。
  • ドラフト、未翻訳、翻訳済み、レビュー済みのさまざまな主要ステータスがあります。
  • 翻訳のエラーを見つけるのに役立つ自動LingoChecksがあります。 プロジェクト設定では、次の検証ポリシーを設定できます。a）翻訳元テキストのすべてのプレースホルダーが翻訳で使用されていることの検証。 b）翻訳元テキストの改行数が翻訳の数と一致することを確認します。 c）翻訳の文字数がソーステキストとn％を超えて変わらないこと、またはソーステキストからn文字を超えないことを確認します（ボタンの文字数に制限がある場合に非常に役立ちます）。 d）翻訳元テキストの特定の用語が翻訳に含まれていることを確認します（たとえば、未翻訳の名前）-これはカスタマイズ可能です。 エクスポートして戻すと、すぐにLingoChecksのステータスを確認し、すべてのエラーを修正するよう翻訳者に依頼できます。 それはかなり興味深い機能のように思えました。
  • キーのステータスをグループ化して変更することができます。
  • 各キーの変更履歴のサポート（翻訳テキストのバージョン管理を含む）。
  • プロジェクト管理者、開発者、または翻訳者のロールを指定して、ユーザーをプロジェクトに追加できます。 後者の場合、アクセスするロケールを指定できます（オプションで、このロケールで翻訳をレビュー/承認する権利を与えることもできます）。
  • パブリックAPIがあります 。
  • オープンソースプロジェクトは無料です。


弱点：



  • githubへの自動プッシュは非常に不明瞭に実装されました！ インターフェイスの設定を把握しようとしていたときに、知らないうちにリポジトリに大量のコミットをプッシュしました。 プロセスが開始されたことの確認または表示はどこにもありません。 結果はgithubでしか見ませんでした。
  • 彼らは検索で賢すぎます。 どうやら彼らはそれを強力で柔軟性のあるものにしたかったのですが、それは必ずしも使いやすいとは限りません。 たとえば、「site CMS」というテキストは「site cms」と「site cm *」で検索されますが、「site cm」では検索されません。 誰もが星のことを考えているわけではありません。
  • 小規模な商業プロジェクトでも無料の料金はありません。 しかし、最小計画は月額4.99ドルからです。
  • サービスインターフェイスから有料翻訳を注文する方法はありません。




transifex.com





アプリケーションをローカライズするためのかなりよく知られたサービスですが、その機能は、奇妙なことに、類似物よりも劣っています。 多くの場合、オープンソースプロジェクトの翻訳に使用されます。



翻訳者インターフェース：







主な機能：



  • コメントと指示を翻訳に追加できます（「指示」は明らかにコンテキストの説明を意味します）。
  • 提案された翻訳（提案）を追加できます。
  • キーには変更の履歴がありますが、どういうわけか奇妙です。 翻訳をいくら変更しても、履歴に残るエントリは1つだけです。
  • 独自の用語集を作成すると、システムは翻訳されたテキスト内の用語集を認識し、既製の翻訳を提供します。
  • 次の翻訳ステータスが利用可能です：未翻訳、未レビュー、レビュー済み。
  • キーを含むタグのサポートがあります。 一対の予約済みタグ「notranslate」と「locked」。それぞれ、翻訳者による翻訳（オリジナルのコピーのみ）を許可せず、通常はキーを使用して何もしません。
  • キーのグループ処理が可能です。 バッチ処理には、「検索と置換」機能があり、便利です。
  • イベントに関するユーザーの通知がサポートされています（ただし、通知の翻訳されていない新しいキーの出現については通知がないようです）。
  • 以前に翻訳された単語を記憶し、将来それらの翻訳を提供する翻訳メモリメカニズムがあります。 翻訳メモリは複数のプロジェクトで共有できます。
  • さまざまな役割を持つユーザーをプロジェクトに追加し、それらをチームにまとめることができます。

    -組織管理者

    -プロジェクトメンテナー

    -チームマネージャー

    -言語チームコーディネーター

    -レビュアー

    -翻訳者

    役割の割り当ての詳細については、 ドキュメントをご覧ください。
  • プロジェクトとターゲット言語を選択することにより、2つの利用可能なベンダーのいずれかからプロの翻訳を注文できます。
  • 基本的な機能（プル/プッシュ/セット/削除など）を実行できるコンソールユーティリティがあります。
  • 外部機械翻訳サービスのサポート：Microsoft TranslatorおよびGoogle Translate。
  • APIも存在します。
  • オープンソースプロジェクトは無料です。


弱点 ：



  • ファイルを1つずつ手動でアップロードする場合のみ、パブリックURLでのみ自動更新が可能です。
  • メインのソース言語をオンラインで編集することはできません。 「残念ながら、ソース文字列をオンラインで編集することはできません...」と書かれています。
  • 英語、フランス語、ドイツ語、ポルトガル語、スペイン語以外の翻訳言語をインターフェースに追加できませんでした。 たとえば、ロシア語を追加する方法がわかりませんでした。 UPD：コメントのNightOrionは、この関数が隠されている場所を示しました： http : //habrahabr.ru/post/229031/#comment_7761613
  • 不便な翻訳エディターが主な職場です。 客観的というよりも個人的な感情のレベルにあります。 すべてがそこにあるように見えますが、不快です。 「a la Excel」の表形式ビューは私に近いですが、ここでは左側のリストでフレーズを選択する必要があり、追加機能を備えた翻訳ウィンドウが右側に表示されます。 現在のテキストが対応するキーを表示するには、どこかに登る必要があります。
  • 検索では、テキスト、キー、または他の何かを検索するものを指定する必要があります。 デフォルトでは、検索フレーズは翻訳テキストのみで検索されます。 これは便利ではありません。フレーズを入力してすべての結果を確認する方が一般的です。
  • 小規模な商業プロジェクトでも無料の料金はありません。




oneskyapp.com





翻訳者インターフェース：







主な機能：



  • プロジェクトの「翻訳スタイル」を指定できます：正式/非公式または明示的に指定されていません。 翻訳者のために文体的なコメントを書くことができます。 この機能は私にとって興味深いものでした。なぜなら、 同じ単語を多くの言語に翻訳するには、採用されている情報の表示スタイルに応じて、異なる単語やフレーズを使用できます。 さらに、おそらく、異なる翻訳者をより適切に調整するのに役立ちます。
  • 翻訳されたフレーズが元のフレーズの長さを超えることができる回数、および翻訳者がこの値をわずかに超えることができるかどうかを指定できます。
  • 必要に応じて、翻訳の投票を有効にし、自動的に確定するために転送が受け取る投票数を設定できます。 これは通常、オープンソースプロジェクトで実践されています。
  • 異なる役割を持つユーザーをプロジェクトに追加することができます。

    -翻訳者（翻訳して翻訳に投票できます）

    -レビュー担当者（翻訳者と同じですが、翻訳を承認する場合があります）

    -モデレーター（校閲者と同じですが、翻訳を変更および削除できます）。
  • 次のペアの翻訳ステータスが利用可能です：未確定/確定済みおよび未承認/承認済み。 既製の翻訳をエクスポートする場合、「承認済み」および「最終化済み」ステータスのキーのみがファイルに含まれます。 翻訳者は「承認済み」ステータスの翻訳を引き続き編集できますが、キーが「最終」ステータスに変更された場合は何も編集できなくなります。 ヘルプでステータスとその意味の詳細を読むことができます。
  • キーにコメントを残すことができます。
  • 通常、検索はキーとテキストの両方を検索します。
  • 教授を注文できます。 次のオプションを使用した翻訳：

    -翻訳されたフレーズのみ。

    -承認されていないフレーズのみ。

    -最後の注文以降に更新されたフレーズ。

    機械翻訳も利用できます（Google翻訳とBing）。これは、もちろんはるかに安価です（ただし、まだ有料です！）。 必要に応じて、完了した翻訳のレビューアをさらに注文することもできます（1単語あたり0.05ドル）。 翻訳者へのコメントを注文に添付できます。
  • 独自の用語集を作成できますが、翻訳を直接追加することはできません。 ただし、用語集の単語は翻訳中にテキスト内で強調表示され、それらをクリックすると、この単語の翻訳を追加するよう求められます。
  • ロールを持つプロジェクトにユーザーを追加する機能：

    -モデレーター（翻訳を翻訳、投票、承認、または変更できます）

    -レビュー担当者（翻訳の変更を除き、モデレーターと同じ）

    -翻訳者（翻訳と投票のみ）。

    ユーザーごとに、どの言語にアクセスする必要があるかを指定できます。
  • プロジェクトはグループにまとめられます（プロジェクトグループ）。 プロジェクトのグループには、App Storeの説明、モバイルアプリ、Webサイト/ Webアプリ、ゲーム、ヘルプデスク（Freshdesk）、ドキュメントなど、さまざまなエンティティの翻訳プロジェクトが含まれる場合があります。 エンティティを選択したら、そのタイプを指定する必要があります。 たとえば、「Website / Web App」の場合、通常のWebサイト、通常のWebアプリ、Facebookアプリ、その他を選択するよう求められます。
  • 翻訳されたテキストに元のプレースホルダーがない場合、自動的に警告します。 翻訳者がこのプレースホルダーの意味を理解しやすくするために、翻訳モードでプレースホルダーテキストにカーソルを合わせるとポップアップする説明を追加できます。
  • 競争の瞬間：翻訳者のすべての成果が記録されます。 初心者、見習い、エキスパート、マスター、または達人の4つのカテゴリで次の成果のいずれかを取得できます。
  • APIは伝統的に存在します。 PHP、Python、Rubyには既製のラッパーがあります。
  • オープンソースプロジェクトは無料です。
  • 最大5人の共同編集者（座席）を持つ民間の商業プロジェクトには無料です 。 価格設定ページでは、以前は月額69ドルでしたが、現在は「無料」となっています。 必要に応じて、理解してください。


弱点：



  • 手動ファイルアップロードのみ。 統合および自動更新はありません。
  • 時々、ステータスペアの組み合わせで奇妙な動作をします。 たとえば、「未承認」の翻訳を作成できますが、同時に「最終版」になります。 ただし、編集することはできません。
  • キースクリーンショットをキーに追加するのは私にとっては珍しいことです。フレーズを翻訳するときにすぐにどこでもできる場合は（同じスクリーン上で）、スクリーンショットをアップロードできる別のセクションに移動する必要があります。 スクリーンショットを読み込んだ後、エリアをマークし、マークされたエリアに対応するキーを示すことができます。 元々のアイデアでしたが、それは私には不快に思えたので、私はそれを短所にしました。 ところで、このサービスは、このアプローチが実践されている唯一のものではありません。
  • リバーシブルなアクションでさえも無限の確認（「Are you sure ...？」という形式のポップアップ）は非常に迷惑です。 一般的に、インターフェースはあらゆる機会にあらゆる種類のローダー、フェーダー、およびその他のハードワークのインジケーターを表示することをためらわず、サービスの何らかの減速の印象を与えます。 さらに、インターフェースはしばしばバグがありました。




crowdin.net





翻訳者インターフェース：







主な機能：



  • Webアプリケーションの翻訳のインライン編集をインターフェースで直接整理することができます。 これを行うには、ダミーのロケールを作成し、サイトに小さなJSコードを挿入します。 次に、アプリケーションをこのロケールに切り替えてCrowdInにログインすると、ユーザーはサイト上の任意の行をクリックして翻訳を入力できます。
  • 「擬似ローカリゼーション」の機能。 それは、サービスが「翻訳」を生成するという事実にあります。これは、アプリケーションがローカライズのためにどれだけ準備ができているかを明らかにするのに役立つはずです。 新しい行は、使用可能なプリセット（フランス語、キリル文字、中国語、またはアラビア語のアルファベットの使用をエミュレートする際のプリセットの本質）のいずれかに従ってランダムに生成されます。 この場合、新しい行が既存の行よりも長くまたは短くなるパーセントを指定できます（たとえば、ボタンやテーブルセルなどの問題のある領域の行の長さに沿ってマージンがあるかどうかを確認するため）、および既存の翻訳を置き換える特別な文字（確認するため、アプリケーションが中国語やアラビア語のテキストなどをどの程度正確に表示するかなど）。 一般的に、それをする時間と欲求を持っている人のためのクールなこと。
  • プロジェクトでグローバル翻訳メモリを使用する機能。 翻訳メモリメカニズムは、これらのタイプのサービスにとってはささいなものです。プロジェクト間で、または特定のプロジェクト内で機能します。 メモリは、翻訳の進行に応じて自動的に満たされるか、事前に作成された翻訳をインポートすることによって満たされます。
  • 翻訳者インターフェースで自動翻訳者（Microsoft TranslatorおよびMyMemory）からのオファーを表示できます。
  • プロジェクト用に独自の用語集を作成できます。 必要に応じて、翻訳者がそれを補充できるようにすることができます（デフォルトでは、プロジェクトの所有者、マネージャー、修正者がこれを行うことができます）。
  • 基本的なエクスポート設定：翻訳されていない行をスキップします。 承認済みの翻訳のみをエクスポートします。 翻訳されていない地域の方言をソース言語から自動的に入力します。
  • リビジョンの1つにロールバックする機能を備えたベースファイルのリビジョンの履歴が保持されます。
  • TranslatorおよびProofreaderの役割を使用してプロジェクトにユーザーを追加したり、プロの翻訳者を招待したりできます。 各翻訳者には、プロジェクトでの指標とアクティビティの概要があります。
  • プロジェクトから、特定の翻訳者のすべてのアクション（翻訳、検証、または音声）を削除できます。このようなことは他のどこでも見たことがありません！
  • プロジェクト参加者間の内部コミュニケーションのシステムと、プロジェクトの一般的なディスカッションを作成する機能があります。
  • 現在のファイルまたはプロジェクト全体で単語を検索して置き換えることができます。
  • 翻訳者からいくつかのキーを隠すことができます。
  • コマンドラインから作業するためのcrowdin-cliユーティリティがあります。
  • 翻訳の変更について外部サービスに通知するためのWebフックのサポート。
  • パブリックAPIがあります 。
  • 教授を注文する能力。 インターフェースで直接翻訳（5つのソースが利用可能）。
  • オープンソースプロジェクトは無料です。


弱点：



  • 検索では、テキストと翻訳のみで検索しますが、キーでは検索しません。 たとえば、キー検索は私たちにとって非常に便利です。なぜなら、翻訳者は多くの場合、どのキーにページ上の特定のテキストがあるかを尋ねるからです。
  • 何らかの理由で、デフォルトでは、サービスからエクスポートされた翻訳済みファイルは、インポートされたファイルと同じ名前になります。 つまり テスト用にen.ymlをダウンロードし、ロシア語に翻訳することを示し、翻訳してアップロードして戻しましたが、ファイル名は「en.yml」のままでした。 ファイルに正しい名前を付けるには、設定で個別に命名規則を設定する必要があります。
  • 小規模な商業プロジェクトでも無料の料金はありません。
  • oneskyapp.comと同じ（便利ではない、私見ではない）タイプの翻訳のスクリーンショットをダウンロードします。最初に、スクリーンショットが読み込まれ、次にその上に行がマークされます。
