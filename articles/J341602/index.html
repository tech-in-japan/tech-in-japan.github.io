.kt拡張子は、ファイルがKotlinファイルであることを意味します。



MainActivity：AppCompatActivity（）は、 AppCompatActivityを拡張することを意味します。



さらに、すべてのメソッドにはキーワードfunが必要です。Kotlinでは、使用する必要はありません。 必要に応じてできます。 Javaのように、アノテーションではなくoverrideキーワードを使用する必要があります 。



それはどういう意味ですか？ savedInstanceState：Bundle？ ？ これは、savedInstanceStateがBundle型またはnull型であることを意味します 。 Kotlin nullは安全な言語です。 あなたが持っている場合： 



 var a : String


aは初期化する必要があり、 nullにはできないため、コンパイルエラーが発生します 。 つまり、次のように書く必要があります。 



 var a : String = "Init value"


さらに、これを行うとコンパイルエラーが発生します。 



 a = null


NULL可能にするには、次のように記述する必要があります。 



 var a : String?


これがKotlin言語の重要な機能なのはなぜですか？ これにより、 NPEを回避できます。 Android開発者はすでにNPEにうんざりしています。 nullの作成者であるTony Hoar Evenでさえ、発明に対して謝罪しました。 null許容のnameTextViewがあるとします。 変数がnullの場合、次のコードでNPEを取得します。 



 nameTextView.setEnabled(true)


しかし、実際にはコトリンは良いです、彼は私たちがそれをすることさえ許さないでしょう。 演算子を使用しますか？ または演算子!! 。 演算子を使用する必要がありますか？ ： 



 nameTextView?.setEnabled(true)


文字列は、nameTextViewがnullでない場合にのみ実行されます。 それ以外の場合、演算子を使用すると!! ： 



 nameTextView!!.setEnabled(true)




nameTextViewが nullの場合、NPEを取得します。 これは冒険家向けです:)。

これはKotlinの簡単な紹介です。 続行すると、Kotlinの別の特定のコードの説明を停止します。

3. ConstraintLayout



activity_main.xmlを開くと、このレイアウトがConstrainLayoutであることがわかります。 iOS向けのアプリを作成したことがあれば、 AutoLayoutについて知っています 。 ConstraintLayoutは本当に彼のように見えます。 彼らは同じヒクイドリアルゴリズムを使用します。 



 <?xml version="1.0" encoding="utf-8"?> <android.support.constraint.ConstraintLayout xmlns:android="http://schemas.android.com/apk/res/android" xmlns:app="http://schemas.android.com/apk/res-auto" xmlns:tools="http://schemas.android.com/tools" android:layout_width="match_parent" android:layout_height="match_parent" tools:context="me.fleka.modernandroidapp.MainActivity"> <TextView android:layout_width="wrap_content" android:layout_height="wrap_content" android:text="Hello World!" app:layout_constraintBottom_toBottomOf="parent" app:layout_constraintLeft_toLeftOf="parent" app:layout_constraintRight_toRightOf="parent" app:layout_constraintTop_toTopOf="parent" /> </android.support.constraint.ConstraintLayout>


制約は、 ビュー間の関係を記述するのに役立ちます 。 各Viewには 、各側に1つずつ、4つのConstraintが必要です。 この場合、 ビューは各側の親によって制限されます。



[ デザイン ]タブでTextViewの 「Hello World」を少し上に移動すると、[ テキスト ]タブに新しい行が表示されます。 



 app:layout_constraintVertical_bias="0.28"






[ デザイン]タブと[ テキスト]タブが同期されます。 [ デザイン ]タブの変更は、[ テキスト ]タブのxmlに影響を及ぼし、その逆も同様です。 Vertical_biasは、彼のConstraintの垂直ビュートレンドを示します。 垂直方向に中央揃えする場合は、次を使用します。 



 app:layout_constraintVertical_bias="0.28"

4.データバインディングライブラリ



データバインディングライブラリについて聞いたとき、最初に自問した質問は、「 ButterKnifeは非常にうまく機能します。さらに、xmlからViewを取得するのに役立つプラグインを使用しています。これを変更する必要があるのはなぜですか？」 データバインディングについて詳しく知るとすぐに、最初にButterKnifeを使用したときと同じ気持ちになりました。



ButterKnifeはどのように役立ちますか？



ButterKnifeは、退屈なfindViewByIdを取り除くのに役立ちます。 したがって、ビューが5つある場合、Butterknifeを使用しないと、ビューをバインドする5 + 5行があります。 ButterKnifeには5行あります。 以上です。



ButterKnifeの何が悪いのですか？



ButterKnifeはまだコードサポートの問題を解決しません。 ButterKnifeを使用したときに、xmlのViewを削除し、Activity / Fragmentクラスのバインディングコードを削除しなかったため、実行時に例外が頻繁に発生しました。 また、Viewをxmlに追加する場合は、再度バインドする必要があります。 これは非常に退屈です。 あなたは連絡を取り合う時間を無駄にしている。



データバインディングライブラリはどうですか？



多くの利点があります！ データバインディングライブラリを使用すると、わずか1行のコードでビューをバインドできます。 それがどのように機能するかをお見せしましょう。 プロジェクトにデータバインディングライブラリを追加しましょう。 



 // at the top of file apply plugin: 'kotlin-kapt' android { //other things that we already used dataBinding.enabled = true } dependencies { //other dependencies that we used kapt "com.android.databinding:compiler:3.0.0-beta1" }

コトリンのゲッターとセッター



Javaのように.setText（）メソッドがないことにすでに気付いているかもしれません。 ここで立ち止まって、Javaと比較してKotlinでゲッターとセッターがどのように機能するかを説明します。



まず、セッターとゲッターを使用する理由を知っておく必要があります。 これらを使用してクラス変数を非表示にし、メソッドのみを使用してアクセスを許可します。これにより、クラスクライアントからクラス要素を非表示にし、同じクライアントが直接クラスを変更するのを防ぎます。 JavaにSquareクラスがあるとします： 



 public class Square { private int a; Square(){ a = 1; } public void setA(int a){ this.a = Math.abs(a); } public int getA(){ return this.a; } }


setA（）メソッドを使用して、正方形の側に負の値を設定することをクラスクライアントに禁止します;それは負であってはなりません。 このアプローチを使用して、プライベートを作成する必要があるため、直接インストールできません。 また、クラスのクライアントが直接取得できないため、ゲッターを提供する必要があります。 このゲッターはを返します。 同様の要件を持つ10個の変数がある場合、10個のゲッターを提供する必要があります。 そのような行を書くことは退屈なことであり、通常、私たちは心を使いません。



Kotlinは開発者の生活を楽にします。 あなたが電話した場合 



 var side: Int = square.a


これは、に直接アクセスしているという意味ではありません。 これはと同じです 



 int side = square.getA();


Javaで。 その理由は、Kotlinがデフォルトのゲッターとセッターを自動的に生成するためです。 Kotlinでは、特別なセッターまたはゲッターがある場合にのみ指定する必要があります。 それ以外の場合、Kotlinは自動的に生成します： 



 var a = 1 set(value) { field = Math.abs(value) }


フィールド ？ これは何ですか 明確にするために、このコードを見てみましょう。 



 var a = 1 set(value) { a = Math.abs(value) }


これは、Kotlinワールドではプロパティに直接アクセスできないため、 setメソッド内でsetメソッドを呼び出していることを意味します。 これにより、無限の再帰が作成されます。 a = somethingを呼び出すと、自動的にsetメソッドが呼び出されます。

フィールドキーワードを使用する理由と、セッターとゲッターの動作方法が明らかになったことを願っています。



コードに戻りましょう。 Kotlin言語のもう1つの優れた機能を紹介したいと思います 。

[ ビルド]をクリックしてデータバインディングライブラリを生成し、必要なクラスを作成し、 MainActivityに戻って2行追加します。 



 binding.repository = repository binding.executePendingBindings()


アプリケーションを実行すると、 TextViewに 「Habrahabr Android Repository Article」が表示されます。 良い機能ですね :)



しかし、以下を実行するとどうなりますか？ 



 Handler().postDelayed({repository.repositoryName="New Name"}, 2000)


新しいテキストは2秒後に表示されますか？ いいえ、表示されません。 リポジトリ値をリセットする必要があります。 このような何かが動作します： 



 Handler().postDelayed({repository.repositoryName="New Name" binding.repository = repository binding.executePendingBindings()}, 2000)


ただし、プロパティを変更するたびにこれを行う必要がある場合は退屈です。 Property Observerという優れたソリューションがあります。

まず、 Observerパターンとは何かを説明しましょう。rxJavaセクションでこれが必要になります。



すでにandroidweekly.netについて聞いたことがあるかもしれません。 これは、毎週のAndroid開発ニュースレターです。 受け取りたい場合は、メールアドレスを入力して購読する必要があります。 後で、必要に応じて、サイトからの登録解除を停止できます。



これは、 Observer / Observableパターンの一例です。 この場合、Android WeeklyはObservableであり、毎週ニュースレターをリリースします。 読者はオブザーバーであり 、彼らはそれを購読し、新しいリリースを待ち、それを受け取ったらすぐにそれを読み、彼らの何人かがそれを気に入らないと判断したら、彼/彼女は見るのをやめることができる



この場合、 プロパティオブザーバーは、 リポジトリインスタンスの変更をリッスンするXMLレイアウトです。 したがって、 リポジトリは監視可能です。 たとえば、 Repositoryクラスのnameプロパティがクラスインスタンスで変更されるとすぐに、xmlを呼び出して更新する必要があります。 



 binding.repository = repository binding.executePendingBindings()




データバインディングライブラリを使用してこれを行う方法は？ データバインディングライブラリは、 BaseObservableクラスを提供します。これは、 Repositoryクラスに実装する必要があります 。 



 class Repository(repositoryName : String, var repositoryOwner: String?, var numberOfStars: Int? , var hasIssues: Boolean = false) : BaseObservable(){ @get:Bindable var repositoryName : String = "" set(value) { field = value notifyPropertyChanged(BR.repositoryName) } }


BRは、 Bindableアノテーションが使用されるときに1回自動的に生成されるクラスです。 ご覧のとおり、新しい値が設定されるとすぐに、それについて学習します。 これで、アプリケーションを起動でき、2秒後にexecutePendingBindings（）関数を再度呼び出さずにリポジトリ名が変更されることがわかります。
