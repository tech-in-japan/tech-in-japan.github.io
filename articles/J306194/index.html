*そして、あなたのバックアップはどれほど古くなっていますか？ （オデッサ。）







定期的なバックアップ（たとえば、毎晩）を実行する最も高価なシステムでさえ、1つの重要な制限があります。最後のバックアップ後にコンピューター上で行われたすべては、決して保護されず、コンピューターがクラッシュすると回復不能に失われます。



たとえば、Dead Soulsの最初の部分が昨日作成され、夜間にバックアップが作成されたとします。 そして今日、第二部が書かれ、別のバックアップの時が来る前でさえ、感情的な衝動で破壊されました。



継続的データ保護（CDP）テクノロジーは、このような状況に対処するように設計されています。 ディスクに書き込まれるすべてのものが同時にバックアップに送信されます。



WindowsおよびLinux環境で実際の例を使用して、Arcserve RHA（レプリケーションおよび高可用性）製品でこれがどのように行われるかを詳しく見てみましょう。

はじめに



2つのサーバーがあるとします。



バトルサーバー（マスター）には、常に更新されるデータが含まれています。 変更を監視し、変更のステートメント（ジャーナル）を「それが->になった」の形式で保持します。次に例を示します。

時間 ウォッシュファイルの内容 変更ログ
10:30:51 母SOAPフレーム
10:30:52 MOM SOAP PAPU ファイル「wash」、オフセット10、「RAM」->「PAP」


変更ログは常にバックアップサーバー（レプリカ）に送信され、そのファイルに適用されます。 したがって、戦闘サーバーとバックアップサーバーのファイルは同一になります。



いくつかの詳細：

  • 変更はバイトレベルで追跡されます。つまり、1バイトが変更されると、ログにはデータブロック全体ではなく、このバイトに関する情報のみが含まれます。
  • 変更ログは、通常のIPネットワークを介して送信されます。つまり、バトルサーバーとバックアップサーバーを、さまざまな都市を含むかなりの距離に分散できます。
  • 接続が切断されると、データがバッファに蓄積され、接続が復元されるとすぐにデータが転送され、バックアップサーバーに適用されます。
  • 変更履歴（特定のボリューム、たとえば最後の500メガバイト）は、バックアップマシンに保存され、ファイルの内容を特定の時点の状態に戻すことで逆の順序で適用できます。




1.アーキテクチャ







上記のスキームが機能するためには、マスターおよびレプリカマシンにEngineと呼ばれるコンポーネントをインストールする必要があります。これは、ファイルの変更を追跡し、あるマシンから別のマシンにデータをレプリケートするメインジョブを実行します。



これら2つのエンジンの動作を設定するために、コントロールサービスを別のマシン（マネージャー）にインストールします。 このマシンは、設定、レポートの受信、管理対象マシンでの個々のアクションの実行にのみ必要です。 常にオンにしないでください。



最後に、インターネットブラウザーで管理サービスに接続し、そこからWindowsアプリケーションをダウンロードすることにより、ユーザーインターフェイスを取得します。

3.2。 特定の時点でのMS SQLデータベースの回復



MS SQLがマスターマシンにインストールされていると仮定します。 レプリカマシンにMS SQLをインストールすることはできません。これは、DATAディレクトリのリポジトリとしてのみ機能します。







MS SQL複製スクリプトはファイル複製スクリプトに似ていますが、私たちの生活を楽にする3つのことが追加されています。



1.正しい初期同期方法がブロックごとに選択されます。 複製を開始する前に、システムは2つのマシン上のデータが同一で​​あることを確認し、異なるものを同期する必要があります。



「ファイルサーバー」タイプのシナリオでは、単純化された初期同期がデフォルトで使用されます。この場合、ファイルのサイズと変更時間が一致する場合、ファイルは同一と見なされます。 このような仮定により、ファイルサーバーでの初期同期時間を大幅に節約できますが、データベースではまったく受け入れられないため、MS SQLスクリプトはファイルを構成するすべてのデータブロックを次々に比較します。



2.データベースファイルの場所を明示的に指定する必要はありません。この情報は自動的に取得されます。





2.データがメインマシン（マスター）に復元されると、MS SQLサービスは自動的に消滅し、手動開始モードになります。 回復後、すべてが元の場所に戻ります。





誰かが「更新」コマンドの「場所」を書くのを忘れたと仮定します。 



begin transaction; update dbo.Table_1 set name=''; commit;


（これを自分でやった人は誰でも転倒の深さを理解するでしょう）



前のケースと同様に、レプリケーションスクリプトを停止し、「ツール->データリカバリ」を実行しますが、バックアップマシンだけでなくメインマシンに対してもリカバリを選択します。







データ破損の瞬間よりも前の時点を選択し、復元を実行するだけです。









3.3。 特定の時点でのLinux上のMySQLデータベースの復元



2つの共同マスターおよび共同レプリカマシンに、CentOS 6.5とMySQLが（同じディレクトリに）インストールされます。



厳密に言えば、スタンバイマシン（co-replica）にMySQLをインストールすることはできませんが、もう少し先に進んで、スタンバイMySQLサーバー上の複製データをキャッチしようとします。



共同マスターマシンでは、MySQLサービスが実行されており、共同レプリカマシンでは、オフになっています。



「カスタムアプリケーション」などの新しいスクリプトを作成します。





「ファイルサーバー」スクリプト（セクション3.1）とは異なり、「カスタムアプリケーション」スクリプトにはデフォルトでブロックごとの初期同期があります。 もう一度繰り返します。「ファイルサーバー」タイプのシナリオでは、デフォルトで単純化された初期同期が使用されます。この場合、ファイルのサイズと変更時間が同じであれば、ファイルは同一と見なされます。 このような仮定は、ファイルサーバーでの初期同期時間を大幅に節約しますが、データベースではまったく受け入れられないため、「ファイルサーバー」などのスクリプトはMySQLレプリケーションでは受け入れられません。







レプリケーションのディレクトリが設定されている画面で、MySQLデータ（この場合はvar / lib / mysql /）を使用してディレクトリを設定します。 mysql.sockソケットファイルがないことを確認してください。 ある場合は、 my.cnf構成ファイルで別の場所を指定します 。







以前のすべてのシナリオと同様に、回復中に過去のある時点へのロールバックを許可することを忘れないでください。







共同マスターマシンで、テーブルTable_1を確認します。 



 mysql> select * from Table_1;
 + ---- + -------------- +
 |  id | 名前|
 + ---- + -------------- +
 |  1 | イワノフ|
 |  2 | ペトロフ|
 + ---- + -------------- +
セット内の2行（0.01秒）




そして、すべてのエントリの名前を同じにすることでそれを台無しにします： 



 mysql> update Table_1 set name = 'Sidorov';
クエリOK、影響を受ける2行（0.00秒）
一致した行：2変更：2警告：0




ここで、特定の時点の復元機能を使用して、すべてを返します。 前と同様に、レプリケーションスクリプトを停止し、両方のマシンでデータを復元します。 しかし、最初に、mysqldサービスを停止します。 



 [root @ co-master mysql]＃service mysqld stop
 mysqldの停止：[OK]




ベースがまだ破損していない時点を選択します。







その後、マスターマシンでMySQLサービスを開始し、データが復元されることを確認します。



そして今-驚き！ 共同レプリカマシンでMySQLサービスを開始し、バックアップマシンで作業データベースを取得しましょう。 Arcserve RHAの高度な機能に近づき、失われたシステムの代わりに、最新のデータを備えたバックアップシステムが動作するようになります。



今日、バックアップMySQLサービスを手動で上げました。 ただし、Arcserve RHA製品は、障害が発生したシステムのように見えるように、バックアップシステムを個別に起動できます。 たとえば、必要なサービスを開始し、DNSのエントリ、IPアドレス、さらにはマシンのNetBIOS名を変更します。 これは「高可用性」などのシナリオで実行できますが、これは別の記事のトピックです。
