「本とインタラクティブを組み合わせたプロジェクトに取り組む」：Kay Horstmann on Books and More







Javaについては非常に多くの本があり、Javaに関する「本」もいくつかあります。 これらには、Kay HorstmannとGary CornellによるCore Java（Javaのロシア語版。ProfessionalLibrary）が含まれます。 それは言語自体よりわずか1年遅れて登場し、すぐにトピックに関する主要な情報源の1つになりました。 そして、次の20年間で、10のエディションに耐え、Javaの新しいバージョンに関する情報を綿密に補充し、1世代以上のJava開発者が成長するようにしました。



KayはJavaの革新に引き続き注目しており、サンクトペテルブルクのカンファレンスの秋に、Joker は Java 9の優れた点を説明します。 そして彼の到着を見越して、私たちは彼に多くのことについて尋ねました：本に取り組むこと、オンラインコースがそれらに取って代わることができるかどうか、そして学界と業界の違い、そしてJavaの未来について。



-「コアJava」が非常に有名で重要になったという事実は、何らかの形でその作業に影響しますか？ たとえば、他の著者よりも徹底的にファクトチェックを行う必要があると感じていますか？



-「コアJava」についての親切な言葉をありがとう。 各著者が実際に適切にチェックしていることを願っていますが、これが多くの人にとって確かであることは確かです。 しかし、「Core Java」が初めてリリースされたとき、いくつかの書籍が競合し、単にAPIドキュメントを繰り返しましたが、ドキュメントは必ずしも正しいとは限りませんでした。 これを発見したのは、APIを実行する小さな（ただし「おもちゃ」の例ではなく）現実的なプログラムを作成したときです。 これがどれほど複雑か想像するのは困難です。stackoverflow.comはまだではなく、Javaソースコードへのアクセスは制限されていました。



「Java Concurrency in Practice」などの一部の本は、コード例を使用せずに当然のことと思っていることを理解できるほど優れた専門家によって書かれています。 しかし、新しいテクノロジーが開発されると、多くの本を急いで書かなければならないことを認めましょう。 この場合、著者が良い例を提供することが非常に重要です。 クラスに動物名があり、メソッドに「Woof」と「Meow」が表示されている本を見る​​と、気に入らない。 コードを現実的なコンテキストで見たい。 私にとっての標準は、Cに関する古典的なKernigan and Ritchieの本で、ほとんどすべてのスニペットが作業プログラムからのものです。



「コアJava」が最初に公開され、Java APIの多くのものが正しく機能しないか、プログラマーにとって不便であると言われたとき、彼らはSun Microsystemsのエディターに叫びました。 しかし、それは本の評判を本当に助けたと思います。 Java SE 9エディションに取り組んでいる今、私はまだ何がうまくいくのか、何がうまくいかないのかを説明しようとしています。 幸いなことに、私の編集者に誰も怒鳴りつけません。



-何十年もの間、1冊の本に取り組んでいます-それは何ですか？ 一部のフラグメントが古くなっているように見えるときに「レガシー」が発生しますが、「それを取り出して捨てる」ことは単に不可能です。 ブックの更新はコードのリファクタリングのように見えますか？ そして、何年も前に書かれたものは、「私はどんな愚かなことを書いた」という感覚を呼び起こしましたか？



-非常に良い質問です。 この本を書き始めたとき、私は教科書の著者でもあり、開発者でもあり、1つのバージョンだけでなく、すぐに次のバージョンについて考えることは自然でした。 この本は、言語とAPIとともに成長するように設計されています。 しかし、APIが飛躍的に成長し、「コアJava」が2つの分厚いボリュームに変わったとき、私たち自身も皆と同じように驚きました。



はい、新しいエディションの作業はリファクタリングのように感じます。 多くの場合、新しい方法で何かを行うと、他のアプローチが時代遅れになり、何百ものサンプルコードを処理する必要があります。 読者が古いものと新しいものの混同によって混乱しないように、私はそれらが適切なすべての例で新しい機能を使用するのが好きです。 単純なケースはダイアモンド演算子であり、「new ... <...>（...）」という形式の式を見つけるために必要でした。 しかし、ラムダ式が追加されたとき、私はサンプルプログラムの半分以上を書き直さなければなりませんでした。



Core Javaを3ボリュームに増やしたくないので、すべてのエディションでいくつかのものが削除されます。 その中には、もはや誰も気にしない古いバージョンのバグに関する情報、言語の改善のために時代遅れになった「回避策」、およびAPIの大規模な「刈り取り」があります。 むかしむかし、CORBAの非常に詳細な説明がありましたが、それを取り除くことは非常に簡単な解決策でした。 RMIの説明が削除されたとき、一部の読者は怒っていましたが、誰かがこれをおもちゃ以外の例で最後に使用したのはいつですか？ そして今、次の難しい決定が選択されています：Swingで何をすべきか？



技術的なエラーがテキストに組み込まれることもありますが、読者が報告できるように、気付いたエラーのリストを保持しています。 私はこれらの「バグ報告」を非常に真剣に受け止め、エラーが1つの出版物よりも長く続かないようにします。 幸いなことに、「私がいかに愚かだったのか」という状況はめったに起こりませんが、時として私の視点が発展することがあります。 たとえば、ネストされたクラスが登場したとき、javapを介した逆コンパイル結果を引用して、ローカル変数のキャプチャがどのように機能するかを詳細に説明しました。 後で考えました。 メソッド呼び出しが技術的にどのように機能するかなど、同じ詳細で説明しましたか？ いいえ、読者はそのような詳細を必要としませんでした。 実際には、プログラマはメソッド呼び出しとローカル変数キャプチャの両方を処理できるという点でコンパイラを信頼する準備が整っているため、無駄な詳細は削除しました。



別の面倒なことは、特定のバージョンに関連する情報の増加です。 「Java 6（貧弱なもの）があれば、Java 7以降でこれを行うことができます。Java9では、他のことができるようになったため、ほぼ達成できます。」 「せっかちなコアJava」では、たとえばJavaScriptをすでに所有しているプログラマーを想像して、Javaの最新バージョンがどのように機能するかを簡単に説明します。 とても爽やかです。 しかし、もちろん、多くのユーザーは異なるバージョンに対処することを余儀なくされているため、従来の「コアJava」は依然としてこれを考慮します。



-一部の書籍は限られた読者にアピールしますが、Core Javaは非常に異なる知識と経験を持つさまざまな人々によって開かれています。 「みんなのために」書くのに苦労していませんか？ それらに対処するのに何が役立ちますか？



-困難があり、初心者の読者にはすべての高度な資料の理解を期待できないことは避けられません。 オブジェクト指向プログラミングのみを学習している読者は、継承とインターフェイスを注意深く理解し、リフレクションまたはモジュールを一lanceするだけで、専門家は反対のことをすることを期待しています。 APIに関する章があると、読者は興味のない章を簡単にスキップできます。



最近では、プログラミングに関する本を最初から最後まで読む人はほとんどいません。 私の読者の多くは、特定のトピック専用のページを開くだけのSafari Booksを使用しています。 そして、Impatiientシリーズの本を構造化して、同じ本の他の場所を常に参照せずに、この形式で資料が簡単に理解できるようにしました。



-長年にわたってJavaについて書いて詳細に説明すると、「Javaでどれだけのことが完全に間違っている」と感じますか？ 何か改善しようとしましたか？



-もちろん、Javaのような大規模で複雑なプラットフォームになると、何かがおかしいことがたくさんあります。 しかし、めったにないことはひどく間違っていることです。たとえば、仮想マシンの内部問題であるはずだったプリミティブ型の存在です。 （そして、将来、新しいバージョンのJavaでこれが修正されることを願っています。）



しかし、もちろん、多くの小さな迷惑な要素があります。 たとえば、APIがストリームをバイト配列に読み込むメソッドを提供するのに20年かかったのはなぜですか？ Unicode APIがなぜこんなに厄介なのですか？ 他にも数十名の名前を付けることができます。



残念ながら、修正はそれほど簡単ではありません。 他のオープンソースプロジェクトでは、誰かが単にもっと良いものを提案し、議論が起こり、改訂が行われ、そして鋭い異議がなければ変更が受け入れられます。 そして、Javaの場合、まったく逆です。 変更を積極的に促進し、規制構造を通過するにはリソースが必要です。 もちろん、これは便利です。つまり、Java 1.0プログラムが正常にコンパイルされる可能性が高いことを意味します。 しかし、それはまた、誰もがどんな変化からも恐れる。



例として、このクラスにmainメソッドが含まれていない場合、コマンドラインで指定されたクラスのコンストラクター（String []パラメーター付きまたは引数なし）を呼び出してプログラムを実行する必要があることを一度提案しました。 私も実装しました-VMの起動コードはそれほど複雑ではありません。 なんでそんなこと気にしたの？ なぜなら、最初の講義で公開された静的なボイドメインに対処する必要のない生徒や教師にとっては、天国からのマナだからです。 「Hello、World！」は次のようになります。 



public class Greeter { public Greeter() { System.out.println("Hello, World!"); } }

ただし、時間が経つにつれて、APIの多くの迷惑な機能が修正される可能性があります。 おそらくオラクルの人々は、私たちと同じように腹を立てているという事実のためです。

-最近スタンフォード大学では、CSの入門コースを変更し、Javaの代わりにJavaScriptを使用することにしました。 これについてどう思いますか？



-コンピュータサイエンスの入門コースは、実際には特定の言語のコースではありません（いずれにしても、そうすべきではありません）。 ループと配列、アルゴリズム、分解、およびデバッグでは、生徒の問題が発生します。 Java、C ++、Python、JavaScriptのいずれを使用するかに関係なく、学生はpublic static void mainやJavaScriptの直観に反する機能よりもループで立ち往生する可能性が高くなります。



したがって、コース自体が焦点を当てていない限り、どの言語を使用するかは重要ではありません。 ポインターに関する詳細を生徒に迷惑させるC ++のコースを見ましたが、これはプロセスに寄与しませんでした。 そして、スタンフォード大学には常に一流のコースがあり、JavaScriptを使用するとJavaを使用した場合よりも悪化することはありませんが、それ以上のことはないと思います。



-あなたはJavaについて何十年も書いており、時間の経過とともに状況がどのように変化したかを見てきました。 Javaの将来についてどう思いますか？ 彼女はそのピークを過ぎており、徐々に人気が失われますか？ それともまだ先に進んでいますか？ScalaやKotlinのような若いJVM言語はエコシステム全体を助けますか？



-Javaの登場直後、急速な離陸の際、Javaが「物語の終わり」になるという非常に明確な感覚がありました。 小さなスマートカードから強力なサーバーに至るまで、Javaはそれらすべてを指揮するようになりました。



これで完全に間違っていることがわかりました。 Javaはサーバーに根ざしていますが、現在はNode.jsが頭の奥に息づいています。 技術的な理由ではありませんが、フルスタックの開発者はクライアント側をJavaScriptで記述したくなく、サーバー側をJavaで記述したいためです。 今日、Java UIがまれであることを考えると、これは問題です。 もし私がJavaの王だったら、JavaがAndroidの言語のままであることを確認し、ブラウザーのJavaからJavaScriptへのテクノロジーを促進しました。 しかし、私がJavaの王ではないことは明らかです。



また、データサイエンスなどの分野で、Pythonについて多くの熱意を持っています。 対応するライブラリをJavaで記述できなかった理由はありませんが、Javaで記述されていません。 そして彼らはコトリンにも手紙を書きません。 Scalaには独自の注目すべき領域、特にSparkがあります。



私の予測は非常に退屈です。 C / C ++が地盤を失い続け、PythonとJavaScriptが成長し、新しい素敵な言語（JVMだけでなく）が自分自身を宣言するが、同じ人気を達成することはないでしょう。 そして何年もの間、技術プレスはJavaが今でも非常に人気があることに驚かされるでしょうが、Javaは何年もの間人気を失っていると歓迎されてきました。
