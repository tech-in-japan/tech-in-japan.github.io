Internet Explorer 9：最も包括的な開発者ガイド（およびその他）、パート3



これは、IE9ブラウザーイノベーションガイドの第3部です。 最初の部分では、CSS、HTML5、およびDOM標準のサポートをレビューしました; 2番目の部分はIE9でのSVGテクノロジーの実装に専念しました。 第3部では、IE9でのWOFF Webフォント、ICCプロファイル、TIFFおよびJPEG XR形式の新しい標準、JavaScriptエンジンの変更、開発者ツール、およびIE9のユーザーインターフェイスのサポートについて説明します。



WebフォントとWOFF



カスタムフォントをWebページに導入した歴史には長いルーツがあります。 多くの人にとって、CSS2の@ font-faceプロパティを実装した最初のブラウザは、1997年に13年前にリリースされたInternet Explorer 4.0であったことを知るのは啓示になるでしょう。 ただし、この技術が広く使用されていなかったため、@ font-faceはCSS 2.1仕様から削除されましたが、このプロパティの実装は4.0以降のIEのすべてのバージョンで行われています。



IEでの実装のフォント形式は、ごく普通のものではないことに注意してください。 これはEmbedded OpenTypeと呼ばれ、OpenType形式のサブセットであり、フォントの著作権を保護し、未使用の文字を圧縮および削除することでフォントファイルサイズを削減できるように設計されています。 Eotファイルは、TrueTypeおよびOpenType形式の対応ファイルよりも大幅に小さくなりました。 EOT形式は、2008年にオープン形式としてW3Cに標準化するために送信されました。



@ font-faceはCSS3仕様に戻り、Firefox 3.5以降、Safari 3.1以降、Opera 10以降、Chrome 4.0以降のバージョンでのみ他のブラウザに実装されました。 ただし、Internet Explorerとは異なり、これらのブラウザーではTrueType （TTF）およびOpenType （OTF） フォントを埋め込むことができます。 この場合の著作権といくつかのフォントの巨大なサイズの問題は解決されません。



新世代のWebフォント技術は、 WOFF （Web Open Font Format）標準でなければなりません。 2010年4月8日に、Microsoft、Mozilla、Operaが共同でWOFF 1.0仕様をW3Cに送信しました。 このリクエストは、4月19日にW3C Webサイトに投稿されました 。 WOFFはEOTと同じ機能を提供します： sfntに基づいており、特別なツールを使用して、文字セットを削減でき、 zlibを使用して圧縮され、作成者が暗号化またはその他のデータ。



WOFF形式はFirefox 3.6でサポートされており、Internet Explorer 9でもサポートされます。



TIFFおよびJPEG XRをサポート



Internet Explorer 9は、TIFFおよびJPEG XR画像形式をネイティブにサポートしています。 さらに、IE9はWindows Imaging Component（WIC）を介した画像デコードを使用します。これにより、デコード速度を最大30％向上させることができます。



TIFFですべてが明確で広く知られている場合、JPEG XRはまだ広く採用されていない新しい標準です。 JPEG XRはISO / IEC 29199-2：2009標準であり、 ITU-T T.832：2009の一部です。 Microsoftによって開発されたこの標準は、古いJPEG形式を大幅に改善し、 多くの機能を追加します 。 JPEG XRの主な利点は、JPEGと同じファイルサイズで、はるかに優れた画質を提供することです。 Internet Explorer 9のこのテストページでこれを確認できます 。また、TIFF形式のブラウザーサポートを示しています。



ICCプロファイル



有名な国際機関であるInternational Color Consortiumは、 ICCプロファイルを標準化しています 。 ウィキペディアによると：「次の仕様により、作成段階から最終印刷までさまざまなアプリケーションやデバイスを使用するときに、システムが正確な色再現を維持できます。 カラーレンダリングを担当するデバイスには、独自のICCプロファイルがあります 。



したがって、正確な色を再現するには、ICCプロファイルのサポートが必要です。 また、Internet Explorer 9は、ICC v2およびICC v4に対してこのようなサポートを提供します。 ICC組織の簡単な公式テストにより、ブラウザーがどの程度正確に色を表示するか、およびサポートしているICCプロファイルのバージョンを決定します。



Chakra-新しいIE9 JavaScriptエンジン



MIX10カンファレンスで、Microsoftはブラウザのその他すべての技術革新の中でも、 Chakraと呼ばれる新しいJavaScriptエンジンを発表しました。 このエンジンは、初期段階でさえ、IE8を7倍上回る優れたパフォーマンス結果を示しました。 Platform Preview 2の改善により、成功が確実になりました。







チャクラの特徴は何ですか：







-JavaScriptのバックグラウンドコンパイルとマルチコアプロセッサの使用。 アプリケーションが起動すると、IE9は高速インタープリターを使用してページを読み込みますが、バックグラウンドコンパイラーはJSコードをマシンコードにコンパイルし、完了するとそれを返します。 したがって、ページが開始され、最初のページの読み込みが高速になったときに、ブラウザーはコンパイラーの結果を待機しません。



-型を最適化するために、IE9エンジンは動的言語で現在使用されている手法を使用します：型表現、多型インラインキャッシュ、動的型推論など。



-クイックインタープリターは、ページ上のコードの即時実行が必要な場合に使用され、効果的なオペコード、型による最適化、レジスタベースのポジショニングエンジンを使用します。



-JSライブラリのすべてのコンポーネントを大幅に改善しました。文字列、オブジェクト、配列、特に正規表現を操作します。



新しいチャクラエンジンの詳細については、このビデオをご覧ください。IE9の高性能JavaScriptエンジンの内部 。 そこで、Internet Explorer 9に関連する他のビデオを見ることができます。



開発者ツール



IE8に開発者ツールが組み込まれたという事実は大きな前進であり、たとえばJavaScriptプロファイラーなど、当時はいくつかの革新的なものがありました。 ただし、ツールのパフォーマンスは、HTMLページの複雑さに大きく依存していました。 大きなページでは、ツールの速度が低下し、それらを使用することは不可能でした。



IE8開発者ツールに対するもう1つの不満は、HTTPリクエスト/レスポンスの監視と分析の欠如です。 このようなツールがIE9に登場したことを知ってうれしかったです。











さらに、非常に大きなDOMを使用したページでの作業も改善されましたが、あまり望んでいませんでした。 パフォーマンスが最終バージョンに引き下げられることを期待しましょう。 私も注意したいささいなことから：



-エージェント文字列を変更する機能







-JavaScriptコンソールが表示されました（当然のことながら、IE8にありましたが、現在は別のタブになっています）







IE9の2つのテストビルドにはJavaScriptプロファイラータブはありませんでしたが、開発者はこの機能が改善および更新されることを約束します。



外観



このセクションは完全に私の疑いであり、現実とは無関係かもしれません。 実際、現在のIE9テストビルドは見苦しい方法で表示されます。コントロールはなく、閲覧履歴に戻ることはできません。アドレス入力行はありません（アドレスはctrl + oで入力します）。







この種の2つの合理的な説明を見つけます。



-テストバージョンが実際には使用されず、町民の間で分散されないように、単純化が意図的に行われたため、最終バージョンへの翻訳が困難になる可能性があります。 言い換えると、IE9の外観はハードコアになっているため、テスト目的でのみ開発者が使用します。



-簡略化は、最終バージョンでのIE9の外観がIE8とは根本的に異なるという事実により行われます。 彼は準備ができていないか、事前に輝きたくないので、一定の時間サプライズを準備します。



私は2番目の仮説がより好きです。 IE9は、多くの視覚効果を備えたWPFインターフェイス（もちろん、WPFのすべてのものと同様に、GPUアクセラレーション）を取得するものと思われます。 マイクロソフトがWPF上に根本的に新しいインターフェイスといくつかの率直に成功したソリューションを備えたブラウザーを既に構築していることを知っている人はあまりいません。 これはブラウザ- ピボットです。







ピボットデータで動作するように設計されていますが、通常のブラウザのように使用できます。 IE8には多くの革新があります：下部のサムネイルの形式のタブ、ページにタグを付け、ピボットデータの形式でそれらを操作する、機能スクロール（左に引くと、画面に収まるまでページの縮尺を縮小するなど）自分でダウンロードして評価することをお勧めします。



私はPivotの多くのアイデアが好きですが、それは重く、明らかにピボットデータを扱うこと以外には最適化されていません。 ただし、Microsoftが、最高のアイデアを使用して、インターフェイスの点でIE9をPivotに似たものにすることを願っています。



私の意見では、IE9の将来を拡大する可能性があるかもしれません。 インターフェイスが根本的に変更された場合、IEのプラグインシステムも変更されます。 将来の拡張システムが、.NET 3.5および4.0で利用可能な新しい成功したMEFアーキテクチャ上に構築されることを願っています。



おわりに



3つの記事で、新しいInternet Explorer 9で開発者とユーザーを待ち望んでいることについて話そうとしましたが、それでも作業を続けていますが、今でもブラウザーの開発者がIEの多くの新しいものとメカニズムのサポートを導入してきた機能の大きな進歩が顕著です。 IE3では、CSS3 / HTML5 / SVGのサポート、新しい非常に高速なJavaScriptエンジン、GPUアクセラレーション、および新しい外観が待ち望まれています。



すぐに、6月中旬または下旬に、IE9の3番目のテストバージョンが利用可能になります。このバージョンでは、HTML5ビデオサポートおよびその他の革新が登場する予定です。 最新情報をお届けし、この一連の記事を新しい情報で続けます。
