セクション「eコマース」



興味深い質問：



- 「類似製品-実装のロジックは何ですか？」

機能的な類似製品をサイトに実装する方法は？ データベース内のクエリが1つだけの場合、これは機能しないと思います。 ジョブの説明：特性（価格、色、その他の技術仕様）のある製品があります。 選択した製品が存在しない場合（可用性= 0）、このカテゴリの商品から最も類似した製品を提供します。 特性のデジタル値を比較するにはどうすればよいのでしょうか？もっと理解できますが、たとえば白のアルファベットの特性を扱っている場合はどうでしょうか？ もちろん、辞書を使用できます。 PHP + Mysqlでの実装。 サードパーティのスタートアッププロジェクトの開発は提供されません。そのような機能のための余分なお金の無駄だと思います。
3段階で。

1.入手可能な商品の最も近いデジタル特性は比較的ありません。それぞれ個別に上下し、関連性の重みを持つリストを形成します。

2.テキスト変数の置換と、関連する重みのリストの形成を伴うレーベンシュタイン距離アルゴリズムを使用して、リスト内の検索（項目1から）。

3.これら2つのリストと重量インジケータの合計と、その後の重量値による並べ替えの組み合わせ：最大重量が最も類似した代替品になります。

PS：データベースに「黄色」、「黄色」、「砂」、「砂」、「明るい黄色」、「明るい砂」がある場合-ここでは、手順2の前に使用するために手動で作成されたシノニマイザーを使用する必要があります。
-最も簡単なオプション

同じカテゴリの製品を提供します。 シンプル、大胆、機能的。



-より複雑

同じカテゴリからですが、価格の対象となります。



-より複雑

基本的な特性で製品を検索する関数を作成します。たとえば、衣料品店の場合、この特性は衣類のサイズです。 まあ、それに応じて同じカテゴリを見てください。



-倒錯

製品ごとに、同様の製品を手動で割り当てます。 CMSでこの実装のようなものを見ました。



多くの方法がありますが、あなただけがあなたが望むものを選ぶことができます。

セクション「インターネット」



興味深い質問：



「WebサービスがAPIと呼ばれるようになったのはなぜですか、それらの違いは何ですか？」

近年、多くの場合、Webサービス（通常はRESTfulまたはRESTに似た曲線、SOAP / XML-RPCははるかに少ない）がHTTP / REST APIまたは単なるAPIを呼び出すことがよくあります。 これにはロジックがありますか、それとも単なるMODですか？



PS：念のため、「API」が何であるかを明確にします。原則として、Win32 APIの開始以来知っています:-)
1）W3Cからの標準のグループがあり、そこで「Webサービス」の概念が使用されているため、旧式（SOAP / XML）Webサービスがそう呼ばれました（https://www.w3.org/TR/ws-arch/#id2260892 ）;

2）したがって、REST APIはWebサービスとは異なるように呼び出され、同時にこの相互作用では「すべての」標準はHTTPであることを強調します。 それ以外のすべて-要求/応答形式、RESTアーキテクチャへの準拠の程度-は特定のサービスによって決定されるため、HTTP上で機能し、RESTアーキテクチャに対応するこのような対話コントラクトを指す一種の「軽量」用語が必要でした。 したがって、REST API / Web APIは、「Webのアーキテクチャに従ってHTTPの上で動作するコントラクト」と解釈されます。
