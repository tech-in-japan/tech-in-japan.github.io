Oracle Database 12c：パフォーマンスのチューニング

データベース時間



データベースのパフォーマンスの最適化について話す前に、このパフォーマンスの測定方法を説明する必要があります。特に、データベースに接続している多くの人々は依然として「応答時間」インジケータを持っているため、多くの人々はDBMSパフォーマンスの普遍的な尺度と考えることに慣れています。







いわゆる応答時間の問題は、残念ながら相対的なことです。 エンドユーザー、システム管理者、ネットワーク管理者、およびデータベース管理者にとって、それは異なり、純粋なDBMSの応答時間だけでなく、アプリケーションサーバー上のビジネスロジックコードのパフォーマンス、Webインターフェイスのパフォーマンス、ネットワークインフラストラクチャコンポーネントの相互作用、およびファイアウォールにも依存します、ロードバランサなど。したがって、データベースのパフォーマンスを適切に表すには、DBMSが実行に費やした時間を表すデータベース時間インジケータを使用する必要があります。 データベースに到着してから結果の選択の最後のフラグメントまで、特定の呼び出し（要求）を満たします。 データベース時間のより厳密な定義は、アクティブな実行中、またはDBMS呼び出しが完了するまでアクティブな待機中のユーザープロセスが費やした合計時間です。







データベース時間の概念は、Enterprise Managerの[トップアクティビティ]ページ（図1）で完全に示されています。ここでは、アクティブなセッションの数と、多色グラフの形式で、DBMS時間の分布、つまり 実際にはデータベース時間。



カスタマイズツール：昨日と今日



以前は、DBMSチューニングプロセスは暗闇の中をさまようようなものでした-管理者は、オプティマイザーの動作に影響を与える可能性のある特定のチューニングパラメーターを適用しようとして、値を変化させ、常に変化に気付かないエンドユーザーからのフィードバックに焦点を当てました最悪の側面。



Find-Fix-Validateと呼ばれる最新のパフォーマンスチューニング方法（図2）により、DBMSパフォーマンス分析ツールを使用してパフォーマンスの問題を正確に診断し、Tuning Packに含まれる自動チューニングツールを使用してそれらを解決し、パッケージReal Application Testingに含まれるテスト。







問題を探しています



もちろん、Oracle Database 12cを含むOracle DBMSの最新バージョンは、文字通り「パフォーマンスセンサーにこだわっています」。また、主な作業（クエリの実行、最適化、結果の出力、ユーザーアクションの調整）に加えて、彼らが何をしているかを常に報告します。コールの待機イベントと時間特性。 したがって、DBMSが特定のアクティビティに費やした時間を常に正確に把握しています。



Oracleはこの機会に非常に優雅にアプローチし、診断情報にアクセスするための独自の言語を発明しなかったが、SQL言語とEnterprise Manager 12cのグラフィカルインターフェイスを使用してアクセスできる特別なデータベーステーブルの形式で設計したと言わなければなりません。 したがって、Oracle DBMSのユーザーのさまざまな問題の根本原因の監視、診断、および検索が最大限に簡素化されます。



必要な情報を含む多くのテーブルがありますが、すでに数百のテーブルがあり、それらの統計はかなりばらばらで累積的です。 しかし、もちろん、データを比較分析するための特別な診断リポジトリ自動ワークロードリポジトリ（AWR）が開発されているため、インジケータを手動で比較する必要はありません。定期的に（デフォルトでは1時間ごとに）テーブルから診断情報が削除されます-さまざまな待機クラス、メトリック、基本統計、 SQLクエリなどの統計。 AWRデータはデータベースに保存され、診断レポートに使用されます。 AWR Baselinesテクノロジーを使用すると、営業日、レポート期間、給与などの営業活動をAWRイメージの間隔と比較することでベンチマーク時間間隔を作成し、選択した間隔のパフォーマンスの比較分析を定期的に実行できます。 このテクノロジーにより、負荷変動をより迅速に分析でき、データベースのパフォーマンスを診断しやすくなります。



AWRレポートは、デフォルトでHTML形式で保存されます（図3左）。 新しいタイプのレポートがあります-これはパフォーマンスハブと呼ばれます-データベースの統計を便利で視覚的なグラフィック形式で表示します（図3右）。







Oracle Database 12.1.0.2では、AWRレポートのもう1つの非常に便利な形式であるActive-HTMLレポートが導入されています。 Enterprise Managerのナビゲーション機能とドリルダウン機能を組み合わせてオフライン分析を行い、他のアクティブレポートと同様に保存およびメール送信できます。EnterpriseManagerで表示する必要はありません。



AWR Warehouseは、AWRデータの長期保存用の中央リポジトリです。 別の専用データベースに保存されます-このように、AWR画像の保存期間を少なくとも無期限に延長して履歴を分析し、1年前、2年前などを確認できます。AWRWarehouseはすべてのエンタープライズデータベースパフォーマンス画面に統合されていますマネージャであり、任意の期間の比較レポートを取得できます。



Active Session History（ASH）機能は、Oracle Database 10gで導入された監視ツールです。 ASHは、アクティブなセッションのステータスのスナップショットを毎秒取得し、特別なメモリ構造に書き込みます（表1を参照）。 実際には、これはリアルタイムの監視です。 Enterprise Manager 12cは、ASH Analyticsと呼ばれる非常にユーザーフレンドリーなグラフィカルセッションをActive Session Historyに導入します。







データベースに組み込まれている自動データベース診断モニター（ADDM）advisorは、診断を解釈し、パフォーマンス低下の根本原因を見つけるのに役立ちます。 AWRレポート自体は、簡単に混同されるかなり膨大なドキュメントです。 ADDMは、ワークロードリポジトリに格納されている統計を解釈し、問題の根本原因を見つけるのに役立ちます。 ADDMは、データベース時間の消費を分析し、それをセッションアクティビティと相関させ、特定の推奨事項を含むレポートを生成します。 ADDMがデータベースのパフォーマンスの問題を通知するだけでなく、問題の原因を特定し、影響の程度に応じてランク付けすることが重要です。



最後に、Oracle Database 12では、Real-Time ADDMの改良バージョンが導入されました。これは、いくつかのパラメータのしきい値を超えると自動的に起動します。たとえば、セッション数、過剰なCPU時間の消費、競合およびデータベースのパフォーマンスを低下させるその他のイベントです。 新しいReal-Time ADDMは、リアルタイムでの問題の自動検出と分析、深刻なパフォーマンス問題の自動診断を特長としています。



ADDMレポートは、履歴分析のためにAWRリポジトリに格納されます。 Real-Time ADDMは緊急監視ツールでもあり、データベースに正常に接続できない場合（「デッドロック」の場合）、特別な診断接続を実行し、DBMSメモリから直接診断を削除できます。 データベースに組み込まれたアドバイザーは、問題の診断と原因の特定に役立ちます。 最初のパフォーマンス分析に便利な形式でAWRおよびASHからの情報が含まれているため、ADDMレポートを使用して分析を開始することをお勧めします。



Enterprise ManagerでのSQLクエリの実行の詳細な分析のために、リアルタイムSQLモニタリングウィンドウがあり、特定のSQLクエリの実行方法、実行されるプラン、および実行プランのどのステップでほとんどのリソースが消費されるかを監視できます。 リアルタイムSQLモニタリングには、実際の行や推定行など、非常に興味深い指標がいくつかあります。 リアルタイムSQLモニタリングを使用して、PL / SQLプロシージャの実行を制御することもできます。



問題を修正します



実践では、ほとんどのデータベースパフォーマンスの問題は、SQLクエリが原因で発生することが示されています。誤って記述されているか、さまざまな理由で非効率的に実行されています。 不完全な統計、オプティマイザの新しいバージョン、不正なパラメータ、競合-SQLクエリが正しく実行されない場合の理由は1000あります。 Oracle SQL Tuning Advisorツールは、同じコストベースオプティマイザー（CBO）SQLクエリオプティマイザーを使用して、ただし特別なチューニングモードで、問題のあるSQLクエリのパフォーマンスを改善するための推奨事項を提供します。 分析では、実際のAWRデータと履歴AWRデータを使用し、代替クエリ実行プランを特定します。 並列SQLプロファイルを使用しているときに、SQLクエリが2回以上加速される場合、SQL Tuning Advisorはそれを推奨します。 また、SQLチューニング・アドバイザは様々な推奨事項をチェックし、どのSQL問合せが分析され、それを構成するために行われた推奨事項に関するEnterprise Managerのレポートを受け取ります。



Oracle Database 12cで導入されたSQL Access Advisorツールの新しいバージョンは、大量のSQL負荷の分析時間を大幅に削減できます。 SQL Access Advisorは単一のSQLステートメントを分析しませんが、一定期間のSQLロードを分析します。 現在では、生産性が大幅に向上し、データベースオブジェクトの分析が10倍高速になりました。



SQLアクセス・アドバイザとSQLチューニング・アドバイザの両方には、Enterprise Managerのグラフィカルなインタフェースがあります。



結果を確認する



Real Application Testingの一部であるSQLパフォーマンス・アナライザ（SPA）ツールは、Oracle Database 10.2、11gおよび12cでテストを提供し、SQLレスポンス時間に対するデータベース・システムの変更の影響を予測し、各SQLテスト負荷実行のSQLパフォーマンス結果を決定し、差異を分析します特定のSQLクエリのパフォーマンス結果を比較します。 同時に、SQLチューニングセット（STS）でSQLの負荷をキャプチャする場合、作業システムのパフォーマンスへの影響は最小限です。



新しいSQLパフォーマンスアナライザー機能-Enterprise ManagerのSPAクイックチェックを使用すると、クエリプランに影響するオプティマイザーパラメーターなどの変更をすばやくチェックできます。 たとえば、セットに1,000件のリクエストがあり、その結果、そのうち10件のみが計画を変更する場合、SPAクイックチェックはまずこれら10件のリクエストを識別し、それらの比較テストを実施します。



それとは別に、プロアクティブなアプローチ、つまり DBMSの問題を防ぐ方法について-クエリプランの管理についてですが、これはこの記事の範囲外の別のトピックです。
