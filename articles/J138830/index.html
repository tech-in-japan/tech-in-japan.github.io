Djangoの共有ホスティング（FreeBSD + Apache + mod_python / mod_wsgi）

Djangoで新しいプロジェクトを作成したら、もう一度Webサーバーの構成を変更します。 そして、それは大丈夫のようですが、犬を散歩させるよりも頻繁に設定を変更するだけです。 どういうわけか正しくありませんか？ 私は同意します（犬も私は気にしません）。 解決策は共有ホスティングです。 インターネットに関する記事をいくつか読んだ後、httpd-vhosts.confでサイトの構成をたたきます。 しばらくの間、それを喜ぶことさえあります。 時間が経ち、ローカルホストには2つ半のサイトではなく、多数のプロジェクト（小さなプロジェクトも含む）があります。 そして、もう一度httpd-vhosts.confを開いて、オプションを追加するために、「少年はいましたか？」と考えるようになりました。本当にこのサイトを作成しましたか？ 実際、プロジェクトからのそのようなオクローシカでは、何かを見つけることはもはや簡単ではありません。 また、たとえば、プロジェクトの半分がmod_pythonモジュールを使用してWebサーバーとやり取りし、もう一方がmod_wsgiモジュールを使用してやり取りすることを考慮すると、作業が一般的に悲しくなります。 はい、そして人生も悲しいものになります（そしてあなたの犬がドアでどのように悲しいか）。



ここでは、仮想ホスティングをより使いやすくする方法と、上記のモジュール（mod_pythonおよびmod_wsgi）を構成する方法を説明します。



私はこれについての私の考えでアメリカを発見しているとは思わない。 しかし、私の例が、この構成のセットアップに費やす時間を短縮し、自分自身（または犬）により大きな利益をもたらすように誰かが助けてくれると嬉しいです。







それで、私が持っているものと、ソロ演奏の最後に何をすべきかを説明しようと思います。 Python 2.7、Django 1.3、およびApache 2.2 WebサーバーがFreeBSD 9.0オペレーティングシステムにインストールされています。 私のホームディレクトリには、すぐに使用できる2つのdjangoプロジェクトが含まれています。 事は小さいです-ApacheとDjangoの相互作用のためのモジュールを選択してインストールする必要があります。また、いくつかのdjangoプロジェクトの同時管理のために共有ホスティングを設定する必要があります。 treeコマンドで取得したディレクトリツリーを次に示します。





サーバーにDjangoをデプロイするには、Webサーバーと対話するモジュールをインストールして構成する必要があります。 そのようなモジュールがいくつかあります。 どちらを優先するかは、システムの特性と特性、選択を行う人の経験と心理的特性、彼の正気の度合い、デバイス/開発者/手の曲率、路上の天気などによって異なります。 簡単に言えば、好みの問題です。 誰もが自分のお気に入りのモジュールをサポートする多くの議論を持っていると確信しています。 私は彼らのすべてを聞きたくないので、議論しません。 やり取りする必要があったモジュールの構成のみを提供します。



すぐにスーパーユーザーに代わって行動することに注意したいので、リストにはsuやsudoのようなコマンドはありません。 特権の低いユーザーのアカウントでシステムを構成する人々のこれらの魔法の言葉を忘れないでください。

さあ、行こう！



MOD_PYTHON



mod_pythonモジュールを備えたApache Webサーバーは、これまで常にDjangoの主な作業環境と考えられてきました（Djangoプラットフォームの作成者が、負荷の高いニュースサイトの基盤として使用したのはこの組み合わせでした）。 このモジュールは、Webサーバー内にPythonインタープリターを実装し、サーバーの起動時にPythonで記述されたコードをロードします。 Apacheプロセスの実行中、コードはメモリに残ります。 この今日のApache + mod_pythonの束は、十分にテストされ、十分に文書化された展開オプションです。 しかし、このモジュールはすでに時代遅れであり、そのサポートはDjango 1.5で削除されます。 したがって、 Djangoの公式Webサイトでは 、新しいプロジェクトをデプロイする際にmod_wsgiモジュールの使用を検討することを強くお勧めします（このトピックについては、この記事で後ほど説明します）。



mod_pythonです。 モジュール自体はポートにあるため、そこからインストールします。 

# cd /usr/ports/www/mod_python3 # make install clean


インストール後、モジュールはその設定をApache構成ファイル（httpd.conf）に追加します。 バージョン管理システムを使用してhttpd.confファイルを保存する場合（「通常のファミリ」である必要があります）、ポートをインストールする前に、必ずファイルをキャプチャしてください。

次に、前述のhttpd.confを編集用に開きます（個人的にeeエディターを使用しますが、viなどの根本的に異なるものを使用することもできます）。 

 # ee /usr/local/etc/apache22/httpd.conf


このファイルで次の行を確認する必要があります。 

  1.  LoadModule python_module libexec/apache22/mod_python.so


    （mod_pythonモジュールをロードします。これは、前のステップでファイルに自動的に追加されるはずの行です。） 
  2.  User www Group www


    （権限のないユーザーに代わってサーバーを起動します。Apacheをrootとして実行することが推奨される場合もありますが、実行しないでください。攻撃者がWebサーバーに侵入すると、追加報酬としてシステムのroot権限を取得します！） 
  3.  ServerName localhost


    （Webサイトの名前。実際のホスト名であり、適切なDNSエントリが必要です。ただし、テスト中は、DNSエントリの代わりに、/ etc / hostsのエントリを使用できます。実際、これが私たちのすることです。 。ローカルホストでテストする場合、この行を追加する必要はありませんが、再起動するたびにWebサーバーがあなたの方向に疑い深く鼻を鳴らします。


すぐに、忘れないように、/ etc / hostsにエントリを追加します（これらは私のサイトの名前です）： 

 127.0.0.1 gallery.ru museum.ru




次に、共有ホスティングをセットアップします。 共有ホスティングの場合、複数のホスト名が同じマシンを指します。 サーバーは異なるドメインへのリクエストを区別し、適切なページを提供する必要があります。 仮想ホストサービス機能を有効にするには、httpd.confファイルのInclude etc / apache22 / extra / httpd vhosts.confの行をコメント解除し、すべてのサイトの構成を1つのhttpd-vhosts.confファイルに記述します（上記を参照）。 ただし、唯一のApacheサーバーはかなり適切な数の仮想ホストを提供できるため、そのような各ホストの構成をIncludesディレクトリ（/ usr / local / etc / apache22 / Includes）の個別のファイルに保存する方がはるかに便利です。 このディレクトリに含まれるファイルの名前の拡張子が.confである場合、Apacheはその後の再起動時にこのファイルの構成を自動的に含めます。 さらに、多数のファイルで迷子にならないように、それらが提供するドメインの名前を含む名前を割り当てます。

そのため、httpd.confファイルの最後に、次の指示を追加します。 

 Include etc/apache22/ Includes/*.conf NameVirtualHost *


次に、Includesフォルダーで、将来のサイトの名前でファイルを作成します。 

 # cd /usr/local/etc/apache22/Includes # touch gallery.ru.conf # touch museum.ru.conf


あとは、新しく作成したファイルに構成設定を追加するだけです。 gallery.ru.confファイルのみをリストします。 



 <VirtualHost *> ServerName gallery.ru ServerAlias www.gallery.ru ServerAdmin test@test.ru DocumentRoot /home/valera/djcode/gallery <Location "/"> SetHandler python-program PythonHandler django.core.handlers.modpython SetEnv DJANGO_SETTINGS_MODULE gallery.settings PythonDebug On PythonPath "['/home/valera/djcode'] + sys.path" </Location> <Location "/media/"> SetHandler None </Location> </VirtualHost> <Directory /home/valera/djcode/gallery> Order allow,deny Allow from 127.0. </Directory>




museum.ru.confファイルおよびサイトのすべての後続の構成は、同じ方法で入力されます。 museum.ruプロジェクトでは、画像を使用していません。そのため、次の行はその構成に含まれていません。 

  <Location "/media/"> SetHandler None </Location>


それだけです。 私が思い出したい唯一のことは、Webサーバーが画像ファイル、jsスクリプトなどを含むディレクトリへの書き込みアクセス権を持っている必要があるということです（私の場合、これはメディアフォルダです）。 これを行うには、メディアフォルダーを含むディレクトリで、次の手順に従います。 

 # chgrp www media # chmod g+w media


サーバーを再起動します。 

 # /usr/local/etc/rc.d/apache22 restart


ブラウザの行にサイトの名前を入力し（私の場合はgallery.ruまたはmuseum.ru）、無駄になっていない時間を楽しんでください！



MOD_WSGI



mod_pythonの代替として位置付けられています。 モジュールの主な利点は、mod_pythonと比較してメモリ消費量が少なく、パフォーマンスが高いことと、デーモンモードで動作する能力です。

では、もう少し簡単にしましょう。 特定の場合にこれを行う理由は説明しませんが（すべての答えは前のモジュールの説明で見つけることができます）、単にアクションの順序と必要なコマンドを示します。

ポートからモジュールをインストールします。 

 # cd /usr/ports/www/mod_wsgi3 # make install clean


httpd.confを開いて編集します。 

 # ee /usr/local/etc/apache22/httpd.conf


このファイルに次の行があるかどうかを確認します。 

  1.  LoadModule wsgi_module libexec/apache22/mod_wsgi.so


    （mod_wsgiモジュールをロードします。この行は、モジュールをポートからインストールするときに自動的に表示されます。） 
  2.  User www Group www


    （非特権ユーザーに代わってサーバーを起動する） 
  3.  ServerName localhost


    （ウェブサイト名）


そのため、httpd.confファイルの最後に、次の指示を追加します。 

 Include etc/apache22/ Includes/*.conf NameVirtualHost *


/ etc / hostsファイルのエントリは、前のモジュールと同様に1対1です。 つまり、私の場合： 

 127.0.0.1 gallery.ru museum.ru




次に、Includesフォルダーで、将来のサイトの名前でファイルを作成します。 

 # cd /usr/local/etc/apache22/Includes # touch gallery.ru.conf # touch museum.ru.conf


次に、新しく作成したファイルに構成設定を追加します。 gallery.ru.confファイルのみをリストします。 



 <VirtualHost *> ServerName gallery.ru ServerAlias www.gallery.ru ServerAdmin test@test.ru WSGIScriptAlias / /home/valera/djcode/gallery/mod.wsgi Alias /media/ /home/valera/djcode/gallery/media/ </VirtualHost> <Directory /home/valera/djcode/gallery> Order allow,deny Allow from 127.0. </Directory>




museum.ru.confファイルおよびサイトのすべての後続の構成は、同じ方法で入力されます。 museum.ruプロジェクトでは、画像を使用していません。そのため、次の行はその構成に含まれていません。 

  Alias /media/ /home/valera/djcode/gallery/media/


リストに記載されているmod.wsgiファイルを作成するだけです。 

 # cd /home/valera/djcode/gallery # touch mod.wsgi


その内容は次のとおりです。 

 import os, sys sys.path.append('/home/valera/djcode') os.environ['DJANGO_SETTINGS_MODULE'] = 'gallery.settings' import django.core.handlers.wsgi application = django.core.handlers.wsgi.WSGIHandler()




メディアディレクトリを書き込み可能にします。 

 # chgrp www media # chmod g+w media


サーバーを再起動します。 

 # /usr/local/etc/rc.d/apache22 restart


ブラウザの行にサイトの名前を入力し（私の場合はgallery.ruとmuseum.ru）、再び喜びます！





ラストワン





その結果、1つのフォルダーにサイトを管理する構成ファイルの束を取得します。 さらに、すべての構成は名前で分割され、各構成には平均30行以下のコードが含まれます。 モノリシック構成と比較して、すべての設定が1つのファイルに集中している場合、このアプローチにより、サイトでのトラブルシューティングを簡素化できます。

異なるモジュールを介してサーバーと対話するために異なるサイトが必要な場合、必要な上記の方法で各構成を構成できます。 主なことは、両方のモジュールのロードがhttpd.confファイルに存在することを確認することです。 

 LoadModule python_module libexec/apache22/mod_python.so LoadModule wsgi_module libexec/apache22/mod_wsgi.so


私のように、すべてのプロジェクトを1つのディレクトリに保存してから、新しい構成を作成するには、次の手順で十分です。 作成しなければならないものと同様の構成を見つけます。 ファイル名を変更しながらコピーしてください。 そして、その中の古いサイトの名前をすべて新しいサイトの名前に置き換えます。 それは： 

 # cp old_site.ru.conf new_site.ru.conf # sed –i –e 's/old_site/new_site/g' new_site.ru.conf


その結果、新しいサイトをWebサーバー構成に追加するには、キーボードで上記の2つのコマンドを入力してサーバーを再起動するのと同じくらいの時間がかかります（mod_wsgiモジュールを使用する場合、mod.wsgiファイルのコピーは追加されます）。
