スナップショット



負荷の高い仮想マシンでVMwareおよびHyper-Vスナップショットを使用したのは誰ですか？ うまく機能しませんか？ スナップショットが多いほど、仮想マシンの動作は悪くなり、削除される時間が長くなり、この削除によりこのマシンのパフォーマンスがさらに低下します。 NetApp ONTAPの場合、スナップショットは些細なことであり、作成時または削除時にパフォーマンスの問題はありません。 VeeamやCommVaultなどのバックアップソフトウェアとの多くの統合により、一貫したスナップショットを取得できます。 VMware vVolを使用すると、個々の仮想マシンごとにきめ細かなストレージを備えたハードウェアアシスタントスナップショットを削除できるため、ハイパーバイザーのスナップショットに起因する問題を排除できます。 このような非抑制スナップショットは、バックアップとDRに役立ちます。 パフォーマンスに影響を与えないハードウェア支援ストレージスナップショットを使用できる場合は、常に使用する必要があります。

階層化またはFabricPool



FabricPoolについて最初に言うことは、他のピアリングテクノロジーと同様に、主にストレージリソースを節約するために設計されており、これが顧客のお金であることです。 予算が制限されていない場合、愚かにもオールフラッシュシステムを購入し、そのシステムにスナップショットを3年間保存し、さらにバックアップサイトで同じシステムの別のシステムを購入し、バックアップクラスターでメトロクラスターと3番目のシステムを構築できます。 しかし、原則としてそのようなアーキテクチャにはお金はありませんが、生産的な（またはその重要な部分）ボリュームのSSDにはまだ関心がありますが、バックアップ、スナップショット、予約にはすでにお金がありません。 これがFabricPoolが助けになる場所です。



FabricPoolは、高速ストレージと低速ストレージを組み合わせて、SSDとS3クラウドで構成されるハイブリッドプール（集合）を作成できるテクノロジーです。FlashPoolキャッシングテクノロジーでの動作と同様です。 FlashPoolの場合と同様に、すべてのデータは最初にSSDレベルのホットレベルに移動し、コールドデータは低速で安価なレベルに移動します。これにより、SSDの高価なスペースが節約されます。



FabricPoolは、DataFabric戦略の一部として実装されるテクノロジーです。これにより、すべてのNetApp製品の統合が拡大し、データモビリティの可能性が拡大します。



NetAppは、新しい機能を作成するとき、他の機能を確認せず、「チェックのために」機能を追加しようとしません。 代わりに、非常によく考えられ、消費者から需要がある技術がリリースされます。 これは、新しい階層化機能FabricPoolにも適用されます。 SSDの需要が高まるにつれて、このテクノロジーへのインストール数と関心が高まっているため、NetAppは、高価なSSDをより効率的に使用できるようにするFabricPool機能をリリースしました。



FabricPoolは、コールドデータを集約SSD（AFF、FAS、ONTAPクラウドシステム）からコールドオブジェクトレベル（Amazon S3クラウドまたはNetApp StorageGRIDオブジェクトストレージ ）に移行するためのテクノロジーです。



FabricPoolを実装してデータをObject Storageに移動するには、2つの段階があります。 9.2RCでは、スナップショットのみをオフセットする機能が利用可能になります。 そして、次の段階は、ボリュームのアクティブなファイルシステムからコールドデータをバージョン9.2GAから利用可能なコールドレベルにシフトする機能です。 FlexProup、MetroCluster、ONTAP Select、SnapLockなどはFabricPoolではサポートされていません。



FAS / AFFおよび3方向コールドデータオブジェクトストレージ上のFabricPool


Amazon S3サードパーティのオブジェクトストレージを備えたFAS / AFFシステムでFabricPool機能を使用するには、ONTAPに追加されるテラバイトライセンスを購入する必要があります; FabricPoolライセンスを追加するとすぐに10TBが追加されます。 その後、テラバイトを追加できます。



FAS / AFFおよびStorageGRIDのFabricPool


FabricPool機能がFAS / AFFストレージシステムで動作するには、Amazon S3を使用する場合にのみライセンスが必要です。 NetApp StorageGRIDの場合、FabricPoolは動作するためのライセンスを必要としません。



FabricPoolおよびONTAP Cloud


ONTAP Cloudシステムの場合、パブリッククラウドに住んでいる場合、FabricPoolを有効にするためのONTAP Cloudによるテラバイトライセンスもありません。顧客は、S3オブジェクトストレージで実際に消費したコールドレベルスペースのみをクラウドプロバイダーに直接支払います。 FabricPoolは現在、ONTAP Cloud for Amazonでのみサポートされています。







FabricPoolと災害復旧


前と同様に、SnapMirrorレプリケーションを使用してDRプラットフォームを構築し、2つのONTAPストレージシステム間でデータを転送します。 非常に頻繁に、お金を節約するために、よりシンプルで安価なストレージがDRサイトで購入されました。 現在、FabricPoolでは、この問題に対して少し異なるアプローチが可能です。 たとえば、メインサイトでAFFシステムが使用されている場合、同じまたは類似しているがAFFシステムをバックアップサイトに配置できます。 オールフラッシュストレージでは、予算の大部分はコントローラーではなく、SSD自体によって消費されます。 FabricPoolを使用すると、AFFシステムをバックアップサイトにインストールできますが、コールドデータはコールドオブジェクトレベルに移動するため、メインストレージの使用可能なスペースの10％〜20％のみです。 したがって、ディスクに保存することはできますが、パフォーマンスを低下させないコントローラーを配置することはできません。 ホットレベルで新しいホットブロックのスペースがない場合、コールドオブジェクトストレージから直接読み込まれます。



クラウドプロバイダーのFabricPoolにとって興味深いものは何ですか？



ストレージは高速ですが、非常に高価なレベルのデータストレージを提供します。 NetApp FabricPoolストレージ所有者は、節約する方法です。 また、一部のお客様は、国の法律に従って、Amazon S3にデータのすべてまたは一部を海外に持ち込むことはできません。 クラウドプロバイダーのFabricPoolにとって、これらは新しい顧客であり、追加のストレージサービスです。



クラウドプロバイダーに必要なもの



FabricPoolテクノロジーは、新規および既存の顧客がSSD（高価なレベル）を使用して生産性の高いストレージシステムのスナップショット（バックアップ）とコールドデータをデータセンターからパブリッククラウドにコールド（安いレベル）に移行できるようにする階層化テクノロジーです。 パブリッククラウドでは、NetApp StorageGridソフトウェア（VMware、KVM、およびDockerイメージの形式で利用可能）を展開する必要があります。 クラウドプロバイダーの場合、NetApp StorageGridは1か月あたりテラバイト単位でライセンスされます。これは非常に便利です。このサービスの顧客がいる限り、クラウドプロバイダーはライセンス料を支払います。 StorageGridを使用するNetAppストレージシステムの所有者には、クラウドプロバイダーにあるライセンスは不要です。



プロバイダーのFabricPoolコールドストレージサービスコストを計算する



コストには、プロバイダーの次の基本コストが含まれます。



  • StorageGridソフトウェアにはシステム要件（CPU、メモリ）があるため、これらはプロバイダーのコストです
  • 顧客は、IPを介してサービスプロバイダーにそれぞれ接続するか、クラウド内のStorageGridにhttpsポートを転送する外部の白いIPアドレスのみを接続します。 外部の白いIPとおそらくファイアウォールはいくらかの価値があります。 または、VPN接続用の白いIPでもあります。このIPで、顧客はコールドデータを追い越します。 VPNの構築にも費用がかかります。 またはそれは専用の通信チャネルですか
  • StorageGridノードの展開に使用されるディスクスペースは、1回割り当てられ、ノードの寿命全体を通して変化しません。 ただし、ノードの数は、負荷と占有スペースの増加に応じて時間とともに増加する可能性があります。 5つのStorageGridノード（仮想マシンでも可）の最小数：3つのノード（ストレージノード）+ 2つの管理ノード（ゲートウェイノードおよび管理ノード）。
  • 毎月-TeAbyte NetAppライセンス
  • そして、顧客からの実際のデータを保存するために使用される実際のディスク容量。


結論



FabricPoolテクノロジは、DataFabric戦略の一部として、コールドデータ用のONTAPとオブジェクトストレージを統合します。 ストレージ所有者にとって、FabricPoolはオールフラッシュシステムとSSDアグリゲートの高価なリソースをより効率的に使用し、非アクティブなコールドデータを安価なレベルにシフトします。これにより、所有コストを抑えてNetAppオールフラッシュストレージシステムをより手頃な価格にできます。 オブジェクトストレージは、成長に応じて簡単に拡張でき、理論的にはスペースの制限はありません（現在のSoft Limit 70PB）。 クラウドプロバイダーにとって、FabricPoolは以前にハードウェアにのみ生産的なデータを保存していた新しい顧客を引き付ける方法です。
