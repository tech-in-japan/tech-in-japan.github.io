Conan：C / C ++の依存関係マネージャー







こんにちは。 今日は、C / C ++の最新の依存関係マネージャーであるConanについて説明します。 あなたが既に彼と積極的に仕事をしているなら、あなたはあなた自身のために何か新しいものを見つける可能性は低いでしょう。 そうでなければ-猫の下でお願いします。



依存関係マネージャーが必要な理由



LinuxベースのディストリビューションまたはmacOSのユーザーであれば、必要な依存関係を強化することはおそらく問題ではありません。ディストリビューションには、必要な<library_name> -devパッケージが含まれている可能性があります。 しかし、あなたがWindowsユーザーである場合、私はしばしば問題に遭遇したと思います。どのようにして中毒をプロジェクトに追加できますか。 サードパーティライブラリがヘッダーのみの場合、ヘッダーファイルを適切なものにコピーして使用を開始するだけです。 ただし、通常はライブラリをアセンブルする必要があり（C ++には動物園があるため、これはそれほど単純ではないことが多い）、コンパイルしたライブラリを必要な場所に配置します。 そして、その後のみ使用できます。



さらに、できれば1つのコンパイラでライブラリをコンパイルすること、ABIの互換性などを忘れないことを忘れないでください。 はい、コンパイル自体に非常に長い時間がかかります（Hi Boost）。 普通の人間にとっては多すぎますね。 一方、一部の人々はpip 、 npm 、 cargo 、 mavenなどを使用するだけです。 たくさんの神経を節約してください。



以前の試み



npmの類似物が欲しいのですが、C ++用です。 文字通り1つのチームでプロジェクトに必要な依存関係を引き上げ、頭痛の種を知りません。



C ++用のこのようなツールを作成する試みはすでに行われています。 たぶんあなたの何人かはbiicodeのようなものを覚えています。 残念ながら、彼は長命を命じました。 ここでは、患者の死因を見つけることができます。 興味深い事実：Conanとbiicodeの作成者はほぼ同じ人たちです。 合理的な疑問が生じます。コナンはbiicodeの運命を繰り返しますか？







コナンに代わる現代的な選択肢があります。 最も有名なのはvcpkg 、 hunter 、 buckarooです。



インストール方法



まず、Conan自体をインストールする必要があります。 これを行うには多くの方法があります：お気に入りのパッケージマネージャー、pip、brew。 Ubuntu / DebianおよびWindows用のバイナリビルドもあります。 著者自身が、pipを介したインストールの使用を推奨しています。 これらのメソッドが気に入らない場合は、 ソースコードからいつでもコンパイルできます 。



特徴



Conanは、コンパイラ、コンパイラバージョン、オペレーティングシステム、OSのビット深度、アセンブリのタイプ（リリース/デバッグ）、標準ライブラリのバージョンなどを追跡します。 これらのパラメーターを使用して、Conanは構成用にアセンブルされたバージョンを見つけようとします。 運がよければ、リポジトリにはすでにアセンブルされたバージョンがあり、何もコンパイルする必要はありません。



OS /コンパイラ/などの組み合わせでも不運なら。 パッケージが見つからなかった場合、Conanは必要なライブラリのソースコードを自動的にダウンロードし、作業中のマシンでコンパイルします。 その後、あたかも誰かによって既にアセンブルされているかのように、ライブラリを安全に使用できます。



使い方



このようなtimer.cppがあり、コンパイルと動作を強制する必要があります。 



#include "Poco/Timer.h" #include "Poco/Thread.h" #include "Poco/Stopwatch.h" #include <iostream> using Poco::Timer; using Poco::TimerCallback; using Poco::Thread; using Poco::Stopwatch; class TimerExample{ public: TimerExample(){ _sw.start();} void onTimer(Timer& timer){ std::cout << "Callback called after " << _sw.elapsed()/1000 << " milliseconds." << std::endl; } private: Stopwatch _sw; }; int main(int argc, char** argv){ TimerExample example; Timer timer(250, 500); timer.start(TimerCallback<TimerExample>(example, &TimerExample::onTimer)); Thread::sleep(5000); timer.stop(); return 0; }


ドキュメントの標準的な例を考えてみましょう。 プロジェクトにPOCOライブラリを含めます（ CMakeを使用するとします）。 これを行うには、プロジェクトフォルダーで、次のテキストを含むファイルconanfile.txtを作成します。

[requires]

Poco/1.8.0.1@pocoproject/stable



[generators]

cmake







このファイルの内容を見てみましょう。 Requiresセクションでは、必要な依存関係をNameLibraries / VersionLibraries @ MaintainerName / Package Stabilityの形式で説明します。 メンテナーの名前は、パッケージをサポートする個人または組織です。 安定性は、単にパッケージの安定性を示す指標です。 通常、安定したテストを使用します。 安定したパッケージのみを使用することをお勧めします。



ジェネレータセクションでは、統合するものを示します。 この場合、CMakeとの統合を示します。 CMakeと統合するためのConanは、conanbuildinfo.cmakeファイルを自動的に作成します。このファイルには、CMakeLists.txtで使用するヘッダーファイルと既にコンパイルされたライブラリへのパスを持つ変数が定義されます。 



 project(FoundationTimer) cmake_minimum_required(VERSION 2.8.12) add_definitions("-std=c++11") include(${CMAKE_BINARY_DIR}/conanbuildinfo.cmake) conan_basic_setup() add_executable(timer timer.cpp) target_link_libraries(timer ${CONAN_LIBS})


5-6行目に注意してください。 5行目では、以前に生成されたconanbuildinfo.cmakeを接続し、6行目では、単にConanを魔法のように動作させるマクロを定義します:-)



これで組み立ての準備が整いました。 また、CMakeの標準的な起動の前に、Conanに依存関係のインストールを依頼する必要があります。 これは、次のコマンドで実行されます。

conan install /path/to/conanfile.txt





通常、チームは次のようになります

conan install ..









コナンが中毒を引き起こす方法-ここに楽しい部分があります



依存関係の検索順序：



  • ローカルキャッシュを確認します。
  • 接続されたリポジトリで必要なレシピを探しています。 リポジトリが接続されている順序で厳密に検索が実行されることに注意してください。 パッケージが2つのリポジトリにある場合、最初のリポジトリが見つかります。
  • レシピが見つかり、構成用に構築されたライブラリが既に存在する場合は、リポジトリからパッケージをダウンロードして、ローカルキャッシュに保存するだけです。 パッケージは、依存関係がすべて自動的にポンプアウトされることに注意してください。
  • 構成にアセンブルされたライブラリがない場合（または構成の依存ライブラリが収集されていない場合）、コナンはこれを通知し、マシン上でパッケージのビルドを開始するために--build missingフラグを付けて再度実行するように要求します。


誰かがアセンブルしたライブラリを使用したくなく、他の人のアセンブリを信頼しない場合、ライブラリとそのすべての依存関係がマシン上で強制的に再構築されるように、常に--buildフラグを指定できます。



おめでとう、すべての依存関係がConanでプルアップされています（運がよければ）！



コナンとビルドシステムとの統合



目的のバージョンのライブラリをダウンロードするだけでなく、プロジェクトに正しく接続することも重要です。 そして、これは開発者にとってできるだけ簡単に行われることが望ましい。 したがって、ConanはCMake、Visual Studio、Xcodeなどと統合できます。 あなたはいつでもここで完全なリストを見つけることができます 。



独自の依存関係を管理する方法としてのコナン

Conanは、誰かが既に作成したリポジトリから依存関係をインストールするためのクライアントだけではありません。 また、この依存関係マネージャーの作成者は、 conan-serverプログラムを親切に提供してくれました。これにより、いくつかのアクションで既製のライブラリーを使用して文字通り独自のサーバーを作成できます。 独自の依存関係管理インフラストラクチャを構築したい企業にとって非常に便利です。



既存の問題



パッケージマネージャーの価値は、使いやすさとパッケージの数によって決まります。 最初のものに特別な問題がない場合（コナンの使用は非常に便利です）、パッケージの数に特定の問題があります。 メインのリポジトリを見ると、それらの2つが見られます： conan-centerとconan-transitです。 同意する、これは誤解を招く-どの1つを使用するのか？ ここで、少し話をする必要があります。



最初は、Conanには1つのリポジトリがあり、そこに任意のユーザーがパッケージをロードする権利がありました。 このため、Conanリポジトリはすぐに1つの大きなゴミダンプに変わりました。 著者は、これは継続できないと判断し、パッケージを徹底的にチェックした後にのみ取得されるリポジトリを作成することを決定しました。 コナンセンターがありました。 そして、古いジャンクリポジトリを削除せず、単に読み取り専用モードにして、conan-transitという名前に変更することにしました。



オープンソースライブラリのパッケージはどこに保存されていますか？



すべてのパッケージはJFrog Bintrayによって保存されます。 少し前まで、コナンはBintrayインフラストラクチャに移動したため、 CDNだけでなく、より容量の大きいパッケージリポジトリにもアクセスできるようになりました。 移行後の最も興味深い革新の1つは、 OSSパッケージ用に完全に無料の独自のリポジトリを作成できることです。 プライバシーが必要な場合は、30日間試用版を使用できます。その後、料金を支払う必要があります。



Bincrafters



たくさんのライブラリがあり、誰でも使える既製のパッケージが欲しいです。 Conanチームは現在非常に小さく、主にパッケージマネージャー自体の開発に従事しているため、すべてを連続してパッケージ化する時間はありません（ただし、パッケージングを含む開発者を常に探しています）。 ただし、ライブラリをパッケージ化する必要があります。そうしないと、Conanに意味がありません。 そこで、Bincraftersチームが登場しました。これは、オープンソースライブラリを積極的にパックするマニアのチームです。 彼らは現在、追加のCI能力のレンタルのための資金を調達するためのプログラムをPatreonに開設しました。 チームに参加したい場合-どういたしまして。



XライブラリがConanにない場合はどうなりますか？



開発のこの段階では、これは非常に頻繁に起こります。 ここにパッケージを作成するリクエストをいつでも残すことができます （その前に、誰もあなたに尋ねていないことを確認してください）。 アプリケーションが既に存在する場合は、このライブラリが必要なものであるというコメントを単に書いてください。 そして、もちろん、あなた自身がライブラリをパッケージ化することができます、誰もが感謝を言うだけです。
