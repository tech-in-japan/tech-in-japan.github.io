Microsoft Azure RemoteAppに基づいたVDIの手間なし

昨年12月、Microsoft Azure RemoteAppはプレビュー版から商用版に移行しました。 本質的に、RemoteAppを使用すると、どの組織でもターミナルサーバーインフラストラクチャを迅速に展開し、RDPを介したリモートアクセス用に企業アプリケーションを公開できます。 このインフラストラクチャのみが、ローカルサーバーと組織ストレージではなく、仮想マシン（VM）とMicrosoft Azureクラウドストレージに基づいて構築されます。 この記事では、RemoteAppを使用する最も単純なシナリオを実装する方法を説明します。 詳細については、MVAポータルのコース「 Azure RemoteAppを使用していつでもアプリケーションを使用する 」を参照してください。





主なアイデア



組織内のモバイルワーカーがビジネスアプリケーションの特定のセットにアクセスする必要があると想像してみましょう。 理想的には、どこからでも、どこからでも、どのデバイスからでも、どのオペレーティングシステムが使用されていてもです。 おそらく、ターミナルサーバーを展開し、必要なアプリケーションをインストールして、リモートデスクトップサービスを使用して公開することにより、最大限のソリューションを理想に近づけることができます。 RDPクライアントは、Android、iOS、Windows、Windows Phoneのすべての主要プラットフォームで利用できるようになったため、実質的に「任意のデバイス」について話すことができます。 さて、「任意の場所」については、インターネット接続があればどこでも予約できます。 このソリューションはITプロフェッショナルに馴染みやすく理解しやすく、サーバーとアプリケーションは一元的に監視および更新されるため、必要なレベルのセキュリティを提供します。



しかし、すべてに欠点があります。 提案されたアプローチの明らかな欠点は、コストとスケーラビリティです。 公開アプリケーションの複雑さとリソース消費、およびそれらに接続するユーザーの数に応じて、1台のサーバーではなく複数台のサーバーが必要になる場合がありますが、十分なストレージがあり、質問の価格が美しいアイデア全体を終わらせることができます。 しかし、必要な容量を持っている/獲得している場合でも、必要に応じてより多くのユーザー/アプリケーションにサービスを提供できますか？ ソリューションのキャパシティをどれくらい迅速に、また、どのくらいの価格で増やすことができますか？



ここで、説明したソリューションが組織のローカルITインフラストラクチャではなく、クラウドに実装されていることを想像してみましょう。 オンデマンドでスケーラブルで、非常に迅速。 SLAを提供します。 消費によって支払われます。 まさにこれがMicrosoft Azure RemoteAppサービスが提供するものです。 技術的な観点から見ると、Microsoft AzureはWindows Server 2012 R2とRDSの役割を持つ1つ以上のVMを作成します。 VMの作成に基づいたイメージは、ローカルネットワークで準備します。 このイメージでは、必要なアプリケーションを追加し、イメージをMicrosoft Azureにアップロードします。 このような画像は必要なだけ作成できます。 Microsoft Azure管理ポータルでは、どのアプリケーションを利用可能（公開）および誰（組織のユーザーにアクセスを許可）するかを指定する必要があります。 また、Microsoft Officeのコンポーネントが公開されたビジネスアプリケーションとして機能する場合は、イメージを準備する必要もありません。MicrosoftAzureがすべてを行います。 魅力的に聞こえますか？ 仕組みを見てみましょう。





展開オプション



Microsoft Azure RemoteAppには、クラウドとハイブリッドの2つの展開オプションがあります。



クラウド展開の場合、Microsoft Azureで発生したVMは、オンプレミスインフラストラクチャとはまったく関係ありません。 まあ、ほとんど何も。 つまり、VM内のビジネスアプリケーションは、作業にローカルActive Directoryまたはローカルネットワークの他のコンポーネントを必要としないことが理解されます。







問題は、どのユーザーがこの方法で公開されたアプリケーションに接続できるかです。 図からわかるように、認証にMicrosoftアカウントを使用するか、ローカルのActive Directoryエントリを使用できます。 後者の場合、ローカルインフラストラクチャとの接続は依然として存在し、組織のADディレクトリをMicrosoft Azureと同期する必要があります。 この同期の構成方法については、コースの2番目のモジュール「 コーポレートデバイス 」で詳しく説明します。 ハイブリッド資格情報を管理する方法 。」



クラウド展開オプションを使用する場合、インストールされたアプリケーションとともにVMイメージをAzureにアップロードするか、既製のイメージを選択できます。 この記事の執筆時点で、Microsoft Azure RemoteAppには2つの既製のイメージが用意されています。WindowsServer 2012 R2とWindows Server 2012 R2およびプリインストールされたOffice 365アプリケーション（Office 365サブスクリプションが必要です）。 自分でイメージを更新します;完成したイメージについては、自動更新がサポートされています（サービスパック、修正など）。



ハイブリッド展開オプションでは、Microsoft Azureで生成されたVMがVPNトンネルをローカルネットワークに関連付け、さらにローカルADドメインに含まれます。 この場合、公開アプリケーションに接続するときの認証はドメインアカウントに基づいている必要があるため、Microsoft AzureとのADディレクトリ同期のセットアップはすでに要件です。







ハイブリッドモデルでは、準備した画像と適切なアプリケーションセットをクラウドにアップロードする必要があります。 ただし、ローカルネットワークとのトンネルがサポートされているため、ドメインネットワークの他のサービスやアプリケーションに依存するものを含め、実際には組織のあらゆるビジネスアプリケーションになり得ます。 ハイブリッド展開のRemoteApp VMは、インフラストラクチャの別のIPセグメントのように見えます。





Office 365クラウドアプリケーションの展開



ここで、オフィススイートへのリモートアクセスが必要だとし、Microsoft Azure RemoteAppのクラウド展開を段階的に実行します。 これは、新しいMicrosoftクラウドサービスを知るための最も簡単で視覚的な方法です。



クラウド展開は4つのステップで構成されます。

  1. RemoteAppコレクション（RemoteAppコレクション）の作成
  2. Active Directory同期の構成（オプション）
  3. アプリを公開する
  4. ユーザーアクセス設定




RemoteAppコレクション（RemoteAppコレクション）の作成



Microsoft Azure 管理ポータルに移動し、 REMOTEAPPセクションを選択して、左下隅の[ 新規 ]をクリックします。







クラウド展開の場合は、 QUICK CREATE （ハイブリッドモデルの場合はそれぞれCREATE WITH VPN ）を選択します。 作成するコレクションの名前を指定し、リージョン、プラン、およびイメージを選択する必要があります。 この場合、Office 365がプレインストールされた完成したイメージが選択されます。







プランには、基本と標準の2つのタイプがあります。 基本的なものは、比較的軽量なアプリケーション、たとえばWebアプリケーション、データ入力用のフロントエンドアプリケーションなどの起動を想定しています。標準計画は、Microsoft Officeなどのより重いアプリケーションに適しています。 プランの価格と接続ユーザーの最大数は異なります 。基本プランでは400ユーザー、標準プランでは250ユーザーです。



コレクションの作成には約30分かかります。 STATUS列に「アクティブ」と表示されたら、コレクションの使用を開始できます。









Active Directory同期の構成（オプション）



クラウド展開オプションでは、Microsoftアカウントまたはドメインアカウントに基づく認証を使用できることを思い出させてください。 後者の場合、ローカルADとAzure Active Directoryの同期を構成する必要があります。 このプロセスについては詳しく説明しません。詳細については、 ここまたは上記のハイブリッドクレデンシャル管理コースを参照してください。





アプリを公開する



作成されたコレクションをクリックすると、クイックスタートページに次のステップが表示されます。







この図は、パブリケーションとアクセス構成の両方がすでに完了していることを示しています。 これは、コレクションを展開したときに既製のイメージを選択したという事実によるものです。 その結果、Microsoft AzureウィザードはVMを上げ、オフィスアプリケーションを公開し、デプロイされたアカウントに代わってアカウントへのアクセスを許可しました。



ただし、いつでもPUBLISHINGセクションに移動して、公開アプリケーションのリストを表示/変更できます。
