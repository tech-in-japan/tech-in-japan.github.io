Enterprise Jabberサーバー：追いつき、Googleを追い抜く

誰もがGoogle Appsについて知っていると思います。 これは、社内でメールやチームワークを整理するのに最適なサービスです。 ただし、いくつかの小さな欠点があります。最初の状態で提供され、Google Appsを使用する際のすべての企業文書、メール、通信はGoogleサーバーに保存されます。



その結果、多くの場合、深刻な企業は困難な道を選択します-自分のサーバーで必要なすべてのサービスをサポートするために。 もちろん、この方法には多くの利点があります。 会社のシステム管理者は、あらゆる方法で構成できます。 ただし、重大な欠点が1つあります。Googleがすべてを構成して接続している場合、すべてを手動で構成する必要があります。 また、システムに同じ美しく便利なWebインターフェイスを提供することはできません。



ただし、実践が示すように、Googleの支援がなくても、企業に柔軟で強力なインフラストラクチャを簡単に展開できます。 猫の下で、XMPPサーバーをメールシステムに統合する方法を説明します。これにより、最終的にはGoogleよりもはるかに優れた結果が得られます。



メールサーバー



まず、メールサーバーが必要です。 MTAの賢明な選択は、最もモジュール性があり、柔軟性があり、しかも信頼性の高いPostfixです。 オプションなしでエンド受信者にメールを配信するには、IMAPサーバーが必要です 。これにはDovecotを使用します。



Dovecotを使用すると、ほぼすべての標準ストレージをユーザーベースとして簡単に使用できます。 たとえば、ADを使用します。 さらに、Dovecotはサードパーティのアプリケーションに自身を介してログインする機能を提供します。特に、Postfixはこれを実行できます。 したがって、非常に便利なスキームが得られます。Dovecotはユーザーデータベースとの通信を担当し、システムの他のすべてのコンポーネントは、単純に認証を確認するためにデータベースにアクセスします。 利点は明らかです。データベースを変更したい（たとえば、ADを放棄してOpenLDAPを使用したい）-何も再構成する必要はなく、Dovecotを新しいストレージに送信するだけです。



Dovecot + Postfixの束を構成する方法は、これらのプログラムのドキュメントに書かれています。 セットアップの結果、SMTPおよびIMAPサーバーを個人的に取得しました。これらのサーバーは、ドメインログイン/パスワードによってどのユーザーに許可されています。 ところで、DovecotとPostfixには非常に優れたドキュメントがあります。これは、これらの製品の重要な利点でもあります。 したがって、このバンドルの設定に問題はないはずです。



Jabberサーバー



あとは、サーバーを既存のJabberメールシステムに固定するだけです。 これを行うことは途方もなく初歩的です。 最初にejabberdをインストールする必要があります。 なぜ彼なの？ それが最も柔軟だからという理由だけで。 この製品の豊富な機能について多くの記事が書かれているので、Dovecotを介してejabberd認証の整理にまっすぐ進みます。 幸いなことに、ejabberdは外部認証をサポートしています。必要なのは、Dovecotを介してユーザーデータを検証するスクリプトを作成することだけです。



小さな余談：Dovecotを介した外部認証メカニズムは、Dovecotが作成した2つのUNIXソケットを介して機能します。 これらのソケットのパラメーターは、Dovecotの設定で指定されています 。それらの詳細については、 公式wikiをご覧ください。



インターネットで少し騒ぎ立てて、私はそのような果物をここで見つけました： http ://search.cpan.org/~sasha/Authen-SASL-Authd-0.04/lib/Authen/SASL/Authd.pm



それから大胆にコードを突いて、ejabberdに必要な動作を追加して、最終バージョンを入手しました。



#!/usr/bin/perl



use 5.010 ;

use IO :: Socket :: UNIX ;

use IO :: Select ;

use MIME :: Base64 qw ( encode_base64 ) ;



# Dovecot

$dovecot_auth_master = '/var/run/dovecot/auth-master' ;

$dovecot_auth_client = '/var/spool/postfix/private/auth-client' ;



sub read_until {

my ( $sock , $re , $timeout ) = @_ ;

my $sel = new IO :: Select ( $sock ) ;

my $result = '' ;

while ( $result !~ /$re/m ) {

$sel -> can_read ( $timeout ) or die "Timed out while waiting for response" ;

defined recv ( $sock , my $buf , 256 , 0 ) or die 'Error while reading response' ;

$result . = $buf ;

}

return $result ;

}



sub dovecot_auth {



my ( $login , $passwd ) = @_ ;



utf8 :: encode ( $login ) ;

utf8 :: encode ( $passwd ) ;



my $base64 = encode_base64 ( " \0 $login \0 $passwd" , '' ) ;



my $service = "ejabberd" ;

my $timeout = 3 ;



my $sock = new IO :: Socket :: UNIX ( Type => SOCK_STREAM , Peer => $dovecot_auth_client ) or die ;



my $resp = read_until ( $sock , 'DONE' , $timeout ) ;



die unless $resp =~ /^VERSION\t1\t\d+$/m ;

die unless $resp =~ /^MECH\tPLAIN/m ;



$sock -> send ( "VERSION \t 1 \t 0 \n CPID \t $$ \n AUTH \t 1 \t PLAIN \t service=$service \t secured \t resp=$base64 \n " ) or die ;



$resp = read_until ( $sock , '\n' , $timeout ) ;



$sock -> close ;



return $resp =~ /OK/ ;

}



sub dovecot_user {



my $login = shift @_ ;



utf8 :: encode ( $login ) ;



my $service = "ejabberd" ;

my $timeout = 3 ;



my $sock = new IO :: Socket :: UNIX ( Type => SOCK_STREAM , Peer => $dovecot_auth_master ) or die ;



my $resp = read_until ( $sock , 'VERSION' , $timeout ) ;



die unless $resp =~ /^VERSION\t1\t\d+$/m ;



$sock -> send ( "VERSION \t 1 \t 0 \n USER \t 1 \t $login \t service=$service \t secured \n " ) or die ;



$resp = read_until ( $sock , '\n' , $timeout ) ;



$sock -> close ;



return $resp =~ /USER/ ;

}



# Reading information from ejabberd

while ( 1 ) {

my $nread = sysread STDIN , my $buf , 2 ;

unless ( $nread == 2 ) { exit }

my $len = unpack "n" , $buf ;

$nread = sysread STDIN , $buf , $len ;



my ( $op , $user , $domain , $passwd ) = split /:/ , $buf ;



# Filter dangerous characters

$user =~ s/[."\n\r'\$`]//g ;

$passwd =~ s/[."\n\r'\$`]//g ;

$domain =~ s/[."\n\r'\$`]//g ;



my $result = 0 ;

if ( $op =~ /auth/i ) {

$result = dovecot_auth ( $user , $passwd ) ? 1 : 0 ;

} elsif ( $op =~ /isuser/i ) {

$result = dovecot_user ( $user ) ? 1 : 0 ;

}



my $out = pack "nn" , 2 , $result ;

syswrite STDOUT , $out ;

}





短いメモ：CPANからのソースモジュールは現在動作する可能性が低いです。 Dovecotとの正しい対話のために、関数encode_base64を呼び出すとき、2番目の引数として空の文字列を指定する必要があります。 上記のスクリプトでは、これはもちろん考慮されています。



Dovecotを介した認証の場合、ejabberdプロセスにはDovecot認証ソケットへの読み取りおよび書き込み権限が必要であることに注意してください。 Ejabberdは、ユーザーejabberdに代わって動作します。これは、奇妙な偶然の一致により、ejabberdグループのメンバーです。 したがって、このユーザー（またはグループ）に、Dovecotソケットとそれらが配置されているディレクトリの両方にrw権限を与えることを忘れないでください（何らかの理由で、最後の項目を忘れる場合があります）。



スクリプトをejabberd構成に追加することは残っています。

{auth_method, external} .

{extauth_program, "/etc/ejabberd/auth.pl"} .





実際、これは実質的な部分の終わりです。 これで、メールに加えて、ユーザーはJabberでもドメインログインでログインできるようになります。 また、メールシステムの設定時にあまりゆがめなかった場合、各ユーザーのJIDとメールは一致し、username @ domain.comのようになります。



ただし、Googleに追いついて追い越すことは残っています。 まあ、これは非常に簡単です。 これは、ejabberdで必要な機能を有効にする完全に標準的な方法です。 私の観点から最も興味深い機能は次のとおりです。



  • 会議 、彼らはまたMUCです。 標準モジュールmod_mucがそれらのサポートを担当します。必要なデフォルトパラメータを設定するだけです。
  • 会議のロギング 。 mod_muc_logモジュールがこれを担当します。 彼はすべての会議ログをHTMLファイルの形式で1つのフォルダーに入れます。残りは、このフォルダーにApacheを設定し、ローカルネットワークからのみ受信したリソースへのアクセスを許可することです。
  • 個人的な通信の記録 。 この機能を有効にするには、mod_log_chatモジュールを再インストールする必要があります。これにより、すべてのログがHTMLファイルの形式で1つのフォルダーに再び配置されます。 次に、このフォルダを定期的にスキャンして必要なファイルを正しい電子メールアドレスに送信する簡単なスクリプトを作成できます。 したがって、Googleと同様の機能が得られます。
  • 一般的なロースター 。 社内で非常に役立つ機会です。 Mod_shared_rostersがそれを担当します。 ユーザー名簿を柔軟に管理し、グループ全体を追加できます。
  • オフラインメッセージのサポート 。 オフにしないでください（ その場合はmod_offline ）
  • 輸送 このアイテムにはコメントは必要ないと思います。 既存のIMネットワークを介して簡単に通信する方法を従業員に提供できます。


念のため、次のことに注意してください。 メール（および私の場合はAD）を組み合わせた認証システムを設定しているため、Jabberサーバーでの登録に関連するすべてをオフにする必要があります。 たとえば、同じmod_register。 目がカルスにならないように。



まとめ



そのため、単純なスクリプトを使用して、メールとJabberを相互に接続しながら、使用する製品のフレームワーク内ですべてを自由に設定することができます。 そして、最も機能的で、柔軟性があり、信頼性の高いツールのいくつかがそれぞれの分野で使用されていたため、実際、タスクのフレームワーク内で可能な限り最も強力なシステムであることが判明しました（企業通信用のメール+ IM）。 Exchangeのような統合されたソリューションはなく、Google Apps、もちろん、closeもそのような機能を提供できません。 同時に、ユーザーの観点からは、結果として得られるシステムは完全に透過的であり、可能な限りシンプルに使用できます。また、管理者の観点からも同様です。 管理者が構成中にすべてを理解する必要がない限り（およびすべてを正しく構成する必要があります！）、さらにサポートは基本よりもわずかに少なくなります。



PS誰もが興味を持っているなら、Dovecot + Postfixバンドル（IMAP ACL、管理者からのメールボックスへの直接アクセス、クライアント、メーリングリスト、パブリックフォルダー、サブスクリプション、ユーザーによる権利の委任など）から多くの興味深いものを絞り出す方法と方法を説明できますサーバーを介してThunderbirdメールクライアントのパラメーターの柔軟な構成を整理し、ユーザーがクライアントで何かを変更する必要性（および必要に応じて機能）を節約します。
