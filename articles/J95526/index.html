2.アップデーター



ここで、既存のすべてのスレッドを自動的に更新したいと思います。 さらに、個々のストリームのTTL（リフレッシュレート）を考慮することが望ましいです。 また、このTTLを時刻ごとに塗り付けたいと思います。 私の研究によると、プロトコルがまったく存在しないか、現実に対応していないため、プロトコルに依存しませんでした。 いずれにせよ、それだけでは十分ではありません。



私は、ストリームの更新頻度を決定するための独自のシステムについて考え始めました。
  • TTL-1時間のストリームの記録間の平均時間間隔（秒単位）（最小2分、最大1時間）
  • 各ストリームには、過去10日間の24時間ごとの平均TTLのリストがあります。
  • 過去10日間の実際のデータに基づいて、翌日の予測が形成されます。これは、各時間の平均TTL値を表します
  • ストリームが更新されるたびに、システムは現在の時間（0〜23）の平均実際TTLを再カウントします
したがって、システムは2分ごとにランチタイムに特定のストリームを更新し、この期間中に10日間、適切な更新を定期的に行う必要があるとしましょう。 これまで、システムはより頻繁にスムーズに適応および更新します。 また、たとえば、夜間、このストリームは1時間に1回更新されます。



実際、更新手順自体は前述のフィードプルであり、重要ではないが何を更新するかです。

そしてここで、これらすべてをキューに入れたいことをスムーズに理解します。 しかし、私は彼らの組織について少し後で話します。



ちなみに、 PubSubを固定し、ハブを起動することを計画しています。



3.発見



確かに、便利なrssリーダーのスキルのリストには、htmlページでrss / atomフィードの検索を含める必要があります。 ユーザーがWebサイトのアドレス（たとえばwww.pravda.ru ）を入力するだけで、システムは実際にそこにアクセスして、購読可能なフィードを検索する必要があります。



しかし、この手順はユーザーのリクエストで直接行うことはできないという事実によって複雑になります。これはWebサーバーのタスクではないため、非同期に実行する必要があるためです。 ユーザーの要求に応じて、最初にそのようなストリームがデータベースに存在するかどうかを直接確認し、ディスカバリキャッシュ（2時間続く）を調べ、何も見つからない場合はこの問題をキューに入れ、最大5秒待ちます（方法について待って、後で説明します）。 この時間内にタスクが完了しない場合、{wait：true}のスタイルでjsonを返すことでスクリプトを終了します。 その後、一定のタイムアウト後、クライアント側はサーバーに同じリクエストを行います。 タスクがバックグラウンドで完了するとすぐに、その結​​果がディスカバリキャッシュに格納されます。



この手順に関連するいくつかのニュアンス：
  • 異なるエンコーディング-ヘッダーまたはヘッダーのいずれにもエンコーディングが示されていない場合があります...バイトで決定する必要があります（常に機能するとは限りません）
  • 1つのページに2つの同一のフィードがあり、1つのRSSが別のAtomである場合があります-この場合、いずれかを選択する必要があります
  • それが機能することを確認するために各フィードを追加でリクエストし、実際のタイトルと説明を取得する必要があります
  • リダイレクト
  • アイコン（同じ問題）
  • 標準と有効性（同じ）
この部分はかなり粗雑だと思います。なぜなら、RSSやAtomが何であるかを知る必要のない人々のためにサービスを提供しようとしているからです。 たとえば、私が最も普通のユーザーとして、突然私のお気に入りで唯一のvkontakte.ruを購読したい場合、ブルーベリージャムが吹いているのがわかります。 少なくとも、将来的にはla grが生成したfeedを実装したいと考えています 。 最低限として、便利で人間的な検索を行います。

ちなみに、このページには具体的には代替物はありませんが、同じサイトの他のページにはどこかにあることがよくあります。 ユーザーが頻繁に入力するサイト上のrss / atomストリームをサイレントに検索するバックアップクローラーを作成するというアイデアがありました。



結論：

さまざまな種類の外部ソースを処理する場合、ランダムにスローされたドキュメントを探して巨大なゴミを掘っているようです。

特定の改良が必要です。 使いやすさの観点から、このページでの代替の単純な検索では十分ではありません。 もっと普遍的な何かをする必要があります。

5.既読/未読のエントリ



これは、スレッドの停止、さらに多くのサブスクライバーが潜在的に存在する可能性があるシステムにとって、かなり重要なタスクであることが判明しました。 最も重要なことは、水平方向にスケーリングできることです。

各エントリエンティティでは、それを読んだユーザーのリストを保持しています。 このリストには少なくとも100万個の識別子が存在する可能性がありますが、mongodbアーキテクチャのおかげでパフォーマンスに影響はありません。 また、別のコレクションには、読み取り時間などの追加情報が格納されます。 インデックスは付けられておらず、統計のためだけに必要であるため、すべてが迅速に機能します。



ユーザーごとに、ユーザーがサブスクライブしているすべてのスレッドについて、カウンターの最終更新日が保存されます。

ユーザーがページを更新すると、システムはこの日付よりも後に表示された各ストリームの新しいレコードの数を見つけ、それを未読の数に追加します（単純な増分）。 ユーザーがレコードを読み取ると、単純なデクリメントが発生します。



別のストリームによる未読エントリの取得も非常に簡単です。



しかし、フォルダ内の未読のみを選択することはすでに問題です。 ニュアンスを明確にしたくありませんが、これはmongodbには単純に結合がないという事実によるものです。 単純な要求またはいくつかの要求では、CodeWScopeを介してのみ解決できます。 インデックス、スケールすることは不可能です-m / r。 これは現在、潜在的なボトルネックです。

9.1。 キューマネージャー



これが最初のノードデーモンです。 そのタスクは、行を読み取り、タスクをワーカーに配布し、作業のプロセスを監視することです。
  • 1人のマネージャーが多くのキューを処理できます
  • システム内のマネージャーは、サーバーごとに1つ、必要な数だけ実行できます。
  • それぞれ独自の方法で構成できます。たとえば、異なるマネージャーが異なるキューのセットを操作できます。
  • キューの構成は異なる場合があり、次のパラメーターがあります
    • 同時に働くワーカーの最大数（実際の数は負荷に応じて調整されます）
    • タスクバッファーのサイズ（タスクの種類とワーカーの数に応じて構成する必要があります）
    • ワーカーの最大アイドル時間（アイドル時間の場合、自動的にメモリを削除して解放します）
    • ワーカーの最大寿命（php-cliの場合、長く生きてはいけません。時々再起動した方が良いでしょう）
    • ワーカーの最大メモリ使用量（超過するとすぐに、kill）
    • タスク実行のタイムアウト（タスク実行中にワーカーがスタックした場合、それを強制終了し、タスクをキューに戻します）
    • タスクが失敗する可能性のある回数
  • キューからタスクが選択されると、ロックが設定されます（ロックにはmemcacheが使用されます）
  • タスクに結果がある場合、memcacheに保存されます
  • 各キューには独自のワーカーがあり、特定のインターフェイスを持つjsクラスである必要があります
    • 現時点では、そのようなものが1つだけ動作します-インポート（さらに先になります）
  • php-cliを子プロセスとして実行し、jsonで通信するWorkerPhp.jsもあります
    • そのような労働者（プロセス）の寿命は1つのタスクで終わるわけではありません。マネージャーは、「脂肪が増えた」と気付いて解雇されるまで、1つずつ実行できます。
    • 実際には、キューごとに4つを超えるphpプロセスが同時に開始されません
  • POSIXシグナルを理解する
  • 正しく完了した場合（-9を強制終了しないでください）、すべてのタスクをメモリからキューにきちんと返します
  • 各マネージャーは、REPLインターフェースでポートを開きます。ポートを開いて、どのように作業しているかを尋ねることができます。 また、その場で再起動せずに構成を変更することもできます。
ところで、キャッチされなかった例外は、1つのスレッドのみを配置しますが、プロセス全体ではなく、喜んで配置します。

そして、このすべて-500行のコード（コメント付き）。



結論：

イベントI / Oは、ほとんどのサーバーアプリケーションが動作するために必要な方法です。 ロックは、本当に必要な場所にのみ配置する必要があります。

ノードを介してphpをプロキシすると、良い結果が得られ、時間を節約できました。

多数の作業は1つのプロセスのみで処理されます（php-cliはカウントしません）。 JSワーカーはそこで非同期に非常に突然作業します。



9.2。 コントローラー-パブリッシュ/サブスクライブハブ



多くの場合、バルクタスク（たとえば100）を並列で、さらには非同期で実行する必要があります。 しかし、線はブラックホールです。 そこに100個のタスクを送信します...そして、結果を得るために2回目にmemcacheに連絡することさえも、不利益です。

それでもキューをバイパスできます。ソケットを使用してマネージャーに直接連絡し、マネージャーにこれらのタスクを完了するように依頼して、同じ接続で応答を待つことができます。 しかし、このオプションは適切ではありません。何十人ものマネージャーがいる可能性があり、どのマネージャーに連絡できるかわかりません...要するに、これは間違っています。



そして、コントローラー（ノード）を作成しました。 通常、システム全体で唯一のものですが、スツールと同じくらいシンプルです：
  • すべてのマネージャーがコントローラーとの永続的な接続を開きます
  • タスクの結果またはファイルの場合、マネージャーはコントローラーに詳細を通知します
  • 「一方で」それに接続し、特定のタスクまたはタスクリストをサブスクライブできます。
  • タスクに関する情報が到着すると、コントローラーはすべてのサブスクライバーに通知します
  • 加入者が多くのタスクを期待している場合、コントローラーは到着時に通知します
  • PHP用のクライアントがあります（ブロッキング）
  • ガベージコレクション
実際、PHPスクリプトがコントローラーに接続し、タスクの結果を5秒間待ち、その後ユーザーをインターフェイスに戻すのは、前述の「検出」手順です。



結論：

パブリッシュ/サブスクライブスキームは、非ブロッキング環境で非常に効果的です。

100パーセントの結果は必要ありません。 結局、100のうち5つのタスクが何らかの理由で完了しなかった場合、原則としてこれは恐ろしくなく、作業を続けます。



9.3。 Feed-updater（バックグラウンド更新）



ノードプロセス、システム全体に1つ。 定期的にデータベースに接続し、更新が必要なフィードのリストを（TTLデータを使用して）受信し、それらをキューにスローします。



9.4。 キュー



競合状態を回避するために、各タスクに対して一意のmd5識別子が生成されます。 この識別子はキューに配置され、このタスク自体のデータはmemcachedにあります。 ほぼすべてのタスクのサイズは固定されておらず、memcacheqはこれに対応していないため、そうすべきではありません。 マネージャーがタスクを取得すると、ロックを設定します。これは、memcachedエントリでもあります。 これにより、実行中に同じタスクのキューに直接再入することを回避できます。

この場合のmemcachedは他の目的で使用されるため、Redisをこのすべての代替として検討する予定です。 彼が倒れた場合、行全体が失われます。



また、キューをユーザーとシステムの2つのグループに分けました。 最初は優先事項です。

これにより、ユーザータスクを妨げることなく、バックグラウンドアップデータで使用されるfeed-pull-sysが追加されました。



結論：

この実装はまだ非常に粗雑です。

キューは、ドロップするたびに復元する必要があります。

より高度なロックシステム-ミューテックスを使用する必要がありますか？

ユーザープロセスとバックグラウンドプロセスには、異なる優先順位が必要です。
