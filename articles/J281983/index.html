最初からDebianパッケージを作成する

Debianパッケージをゼロから作成することは、一種の魔法のプロセスです。 「Debianパッケージをゼロから構築する」というリクエストでGoogleを起動すると、多くの結果が得られますが、どれも必要なものにはなりません。 間違いなく、Debianで使用されるコマンドの優れた概要を見つけることができ、十分に深く掘り下げると、基本的なDebianパッケージの作成に役立ついくつかのコマンドを見つけることができますが、何が起こっているのかを説明することはできません。 「ハプニング」とは何かに関する詳細な情報を得ることができます。この投稿では、これについて部分的に触れます。







まず、いくつかの依存関係をインストールすることから始める必要があります。 このチュートリアルはすべてUbuntu 14.04に基づいて作成されていますが、ほとんどのDebianベースのオペレーティングシステムに適しています。 次のコマンドを実行して開始します。 



sudo apt-get install build-essential dh-make


次に、すべてのコンポーネントを配置するパッケージを作成する必要があります。 この場合、パッケージは「mylittledeb」と呼ばれます。 次のコマンドを使用して、パッケージをインストールします。 



 mkdir mylittledeb touch mylittledeb/Makefile touch mylittledeb/hello.c


hello.cのコンテンツには、古典的な「Hello、World」が含まれている必要があります。 



 #include <stdio.h> int main() { printf(“Hello, World\n”); return 0; }


メイクファイルには次の内容が含まれている必要があります。 Makeファイルに慣れていない人のための情報-コマンドを実行するときにスペースではなく「_」を使用する必要があり、エラーが発生するリスクがあります。 また、バイナリファイルを削除した後、ターゲット要素のクリーニングが||true



に設定されることに注意してください。 将来、作成前にmake clean



実行することにより、指定された不一致およびその他のDebian要素を使用します。 



 all: gcc hello.c -o hello clean: rm hello || true


この時点で、ユーザーが「Hello、World」を印刷できるDebianパッケージが作成されました。 これは最もおもしろい部分ではありませんが、さらに多くのことがあります。 この時点で、 make



を実行make



とバイナリhello呼び出しが生成され、「Hello、World」のバイナリ出力が実行されるmake



を確認します。 これが起こらなかった場合、それは何かがうまくいかなかったことを意味し、先に進むために修正する必要があります。



最後に、実際にパッケージの作成を開始できます！ Debianパッケージを初期化するには、以前にインストールした便利なdh_make



プログラムを使用します。 以下に示すのと同じ設定で次のコマンドを実行すると、エラーが発生するはずですが、これは計画的なエラーです。 この場合、dh_makeとは何かを理解し、より高度なパッケージを使用する際に後で発生する可能性のある他の問題を解決する方法を理解することが重要です。 



 dh_make -p mylittledeb_0.0.1 Type of package: single binary, indep binary, multiple binary, library, kernel module, kernel patch? [s/i/m/l/k/n] s Maintainer name : root Email-Address : root@unknown Date : Sun, 10 Apr 2016 14:38:32 -0400 Package Name : mylittledeb Version : 0.0.1 License : blank Type of Package : Single Hit <enter> to confirm: Could not find mylittledeb_0.0.1.orig.tar.xz Either specify an alternate file to use with -f, or add --createorig to create one.


エラーメッセージを受け取りました。 ここで、このエラーに関連するいくつかのことを検討してください。 まず、orig.tar.xzファイルとは何ですか。 次に、なぜ-pフラグが使用されたのですか？ 簡単な質問から始めましょう。 dh_makeは、現在いるディレクトリの名前を調べてパッケージ名とバージョンを確認するため、-pフラグが使用されます。 多くの場合、この方法でディレクトリを呼び出すのは愚かであるため、-pフラグを初めて実行すると、dh_makeはデータを<パッケージ名> _ <バージョン>の形式で送信します。



では、どのようなファイルorig.tar.xzを見つけましょう。 公式文書では、これは単なるソースコードのtarballであり、この場合はディレクトリの現在の状態にすぎないと述べています。 ただし、おそらくこのorig.tar.xzファイルには特別なものがあります。 それでは、作成方法を見てみましょう。 次のコマンドを実行すると、dh-makeのソースコードを分解できます。 mylittledebパッケージでこれを行うと、すべてのファイルをクリアする必要があるため、tmpでこれを行うことができます。 



 apt-get source dh_make


ソースコードが用意できたので、次に何が起こるか見てみましょう。 dh_makeスクリプトを開くと、内部のPerlファイルを見つけることができます。 Vimでorig.tar.xzを検索すると、次の行に進みます。 



 system('tar', 'cfJ', “../$package_name\_$version.orig.tar.xz”, “.”);


これは単なる古いtarアーカイブファイルです。 ただし、すべてがうまくいかない場合に備えて、これらの魔法のDebianスクリプトはすべて作成されていることをある程度理解しておく必要があります。 では、mylittledebフォルダーに戻って、次を実行します。 



 dh_make -p mylittledeb_0.0.1 --createorig


これで、mylittledebフォルダーに次の内容のDEBiANフォルダーが表示されます。 



 changelog compat control copyright docs init.d.ex manpage.1.ex manpage.sgml.ex manpage.xml.ex menu.ex mylittledeb.cron.d.ex mylittledeb.default.ex mylittledeb.doc-base.EX postinst.ex postrm.ex preinst.ex prerm.ex README.Debian README.source rules source watch.ex


.exおよび.EXファイルはファイルの例です。 それらのほとんどは必要ありません。



changelog-このファイルはパッケージのバージョンを制御し、最後の更新以降に変更された内容も簡単に説明します。 メインファイルは次のようになります。 



 mylittledeb (0.0.1–1) unstable; urgency=low * Initial release (Closes: #nnnn) <nnnn -    ITP> — root <root@unknown> Sun, 10 Apr 2016 15:00:11 -0400


「mylittledeb」には「0.0.1」という名前が含まれ、これはパッケージのバージョンであり、最後の「1」はDebianのバージョンです。 Debianパッケージが対象となり、さまざまなディストリビューションに移植される不安定なディストリビューションは、このプロセスの外部で行われます。 この場合、これはすべてUbuntu 14.04上に構築されているため、単にtrustyを使用します。 すべてを実行すると、次のようになります。 



 mylittledeb (0.0.1–1) trusty; urgency=low * Intial package release — root <root@unknown> Sun, 10 Apr 2016 15:00:11 -0400


パッケージに署名する場合は、最後の行にGPGキーに関連付けられた名前を含める必要があります。 ただし、パッケージに署名することはできます。 現時点では、この項目は無視できます。



compatは魔法のファイルであり、常に数字の9を使用する必要があります。これは、見つけることができる唯一の情報です。 なぜ正確に9ですか？ さて、 debhelperパッケージのすべてのツールで使用されます。これにより、Debianファイルの互換性が確保されます。



control-制御ファイルには、「apt-cache show mylittledeb」を実行したときに表示されるパッケージに関する独立した情報が含まれています。 これはすべてDebian wikiで非常によく説明されています 。 パッケージは次のようになります。 



 Source: mylittledeb Section: devel Priority: optional Maintainer: root <root@unknown> Build-Depends: debhelper (>= 8.0.0) Standards-Version: 3.9.4 #Vcs-Git: git://git.debian.org/collab-maint/mylittledeb.git #Vcs-Browser: http://git.debian.org/?p=collab-maint/mylittledeb.git;a=summary Package: mylittledeb Architecture: any Depends: ${shlibs:Depends}, ${misc:Depends} Description: An example debian package that says “Hello, World”


パッケージのすべての依存関係が含まれているため、依存関係アセンブリを正しく構築する必要があることを理解する必要があります。 プログラムを実行する依存関係のみを含むDepends



に追加しないでください。 ここで重要なのは、{shlibs：Depends}と{misc：Depends}は、dh_shlibdepsコマンドによって呼び出される2つの魔法の行であるということです。 このコマンドは、バイナリのニーズである依存関係を定義するのに役立ちますが、これはすぐにはわかりません。



著作権は最も明白なファイルであり、パッケージを一般に公開することに関心がある場合は、単にこのファイルを含める必要があります。



docs-このファイルには、キットに含まれるすべてのファイルの名前をリストできます。



ルール-パッケージの作成に関するすべての情報を含むDebianルールファイルで、Makefileの特別なバージョンです。 このファイルは、Debianファイルの作成に使用される追加の目標を示しています。 デフォルトでは、これは基本的なコマンドを起動するだけの非常にシンプルなファイルです。 ただし、たとえば、アセンブリの前にディレクトリをクリーンアップする方法など、何かを再定義する必要がある場合は、次のようにする必要があります。 



 #!/usr/bin/make -f # -*- makefile -*- # Uncomment this to turn on verbose mode. #export DH_VERBOSE=1 override_dh_auto_clean: rm /tmp/random.file %: dh $@


使用可能なすべてのオーバーライドは、このwiki記事と、ルールファイルの詳細な説明にリストされています。 デフォルトでは、すべてのdebhelperツールはこの方法でこのファイルを使用します。たとえば、ディレクトリをクリーンアップする場合は、新しいパッケージを作成する前にこれを行う必要があり、次の「dh clean」コマンドが実行されます。 このコマンドは順番にdh_testdir、dh_auto_clean、dh_cleanを呼び出し、これらはいくつかのperlスクリプトを順番に呼び出します。



現時点ではこのファイルを編集する必要はありませんが、独自のDebianパッケージを作成するときに必要に応じて後で機能を設定できることに注意してください。



ソース/フォーマット-このファイルは非常に単純なようです。 現時点では、そのままにしておくことができます。 これはコンパイル形式であり、以前に作成したアップストリームtarballにパッチを適用する特定の方法であることを知ってください。



* .install-このファイルは自動的に作成されないため、mylittledeb.installというファイルを作成する必要があります。 インストール時に追加されたファイルは、バイナリのインストールに使用されます。 パッケージをインストールするときに、システムにあるファイルまたはディレクトリが使用されます。 「hello world」バイナリがあるため、これをユーザーシステムに配置する必要があります。 次のファイルは、「hello world」バイナリをユーザーディレクトリ/ usr / binに配置します。 



 hello /usr/bin


.exおよび.EXで終わる残りのファイルには、さまざまなファイルの機能の説明が含まれています。 もちろん、それらを保存する方が良いですが、削除してもひどいことは起こりません。
