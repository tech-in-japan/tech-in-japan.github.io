メディアパパラッチに関するストーリーの最初の部分


第1部では、メディアスピーカーにたどり着くまでの経緯と、その前に経験したオプションの数について説明しました。今から話を続けましょう。







写真のさまざまなソース



私たちが直面した次のタスクは、MediaPickerの写真が3つの異なるソースから取得できることです。



  • カメラで撮影した写真は、アプリケーションフォルダーのディスクに保存されました。

  • ユーザーのフォトギャラリーから写真を選択することもできます。

  • 最後に、既に投稿された広告を編集用に開くと、それらはネットワークから取得されます。



もちろん、3つではなく1つのエンティティが必要でした。そのため、写真で動作するコードでは、branchesいブランチを作成する必要がなく、新しいデータソースが突然出現した場合に変更から保護します。



画像を操作するときに実行する必要がある4つのアクションを特定しました。



  • 最も一般的なアクションは、もちろん、インターフェースに表示することです。 そして、写真の小さなサムネイルごとに画面のサイズに合わせて調整する必要がなく、フルサイズの写真を3 x 4000ピクセルのサイズで保存しておくといいでしょう。



  • 次は、元のイメージの受信です。たとえば、サーバーに送信したり、ディスクに保存したりします。 繰り返しますが、ここではできるだけメモリをロードしたくないので、NSData形式の圧縮表現が必要です。たとえば、JPEGからビットマップに画像をデコードするためにシステムリソースを費やす必要はありません。



  • また、画像のサイズを調べることが必要な場合がありますが、これも最適な方法でより適切に行う必要があります。つまり、可能であれば、完全にダウンロードせず、そのためだけにメモリでいっぱいにしないでください。 多くの場合、サイズはファイルのメタデータから取得できます。または、サーバーはURLの横にあるJSON構造の個別のプロパティとしてそれを送信できます。



  • 最後に、画像がネットワークからダウンロードされたが、ある時点で絶対に必要ないことがわかった場合（たとえば、表示されるはずだった画面を閉じた場合）、ダウンロードをキャンセルできると便利です。



もちろん、イメージが必要なときにローカルで利用できない場合があるため、最初の3つのポイントのAPIは非同期である必要があります。 メモリに冗長データをロードせずにUIに画像を表示するには、必要なサイズを見つける必要があります。



  1. これを行うには、表示が行われる領域のサイズとその使用方法を知る必要があります：画像を完全にフィットさせたいのですか、それとも一部を犠牲にして内部に空きスペースがないようにしますか（コンテンツモードのAspectFitと同様）およびUIViewのaspectFill）。



  2. APIは非同期である必要があるため、結果の画像をUIImageViewに渡すハンドラーが必要です。



  3. ネットワークから写真をアップロードする必要がある場合もありますが、同時に同じイメージのキャッシュバージョンがローカルにありますが、それより小さくなります。 そして、この縮小版をブート時にビューに代入すると、ユーザーはダウンロードが高速であるという印象を受けることになります。



  4. そのため、プログレッシブ値を設定するdeliveryModeパラメーターも問題ありません。要求された画像の不良バージョンに反対ではなく、品質が向上するたびにハンドラーを数回呼び出すことができます。 最良とは、最良のバージョンの画像を使用してハンドラーを1回だけ呼び出すことを意味します。



したがって、リストされたパラメータを使用して画像を要求する方法は、次のようになります。 



func requestImage( viewSize: CGSize, contentMode: ContentMode, deliveryMode: DeliveryMode, handler: @escaping (UIImage?) -> ())


最初の3つのパラメーターを1つの構造に結合することで短縮します。 これにより、メソッドシグネチャを変更せずに、必要に応じて他のパラメーターを追加できます。 



 func requestImage( options: ImageRequestOptions, handler: @escaping (UIImage?) -> ()) struct ImageRequestOptions { let viewSize: CGSize let contentMode: ContentMode let deliveryMode: DeliveryMode }


結果のバージョンはまだ確定する必要があります。 まず、UIImage型はハンドラー句パラメーターで明示的に指定されており、UIKitを削除して、iOSだけでなくこのメソッドを使用できるようにしました。



したがって、UIImageは、後でUIImageに変換できるものに置き換える必要があります。 この基準を満たし、iOSとmacOSの両方に存在するタイプがあります-これはCGImageです。



したがって、InitializableWithCGImageプロトコルを作成します。 



 protocol InitializableWithCGImage { init(cgImage: CGImage) }


幸いなことに、UIImageとNSImageには既にそのようなイニシャライザーがありますので、これらのクラスに空の拡張機能を追加し、このプロトコルへの準拠を正式に説明するだけです。 



 extension UIImage: InitializableWithCGImage {} extension NSImage: InitializableWithCGImage {}


UIImageをこのプロトコルに置き換えると、このようなメソッドシグネチャが取得されます。 



 func requestImage<T: InitializableWithCGImage>( options: ImageRequestOptions, handler: @escaping (T?) -> ())


最後に、リクエストがキャンセルされるように注意する必要があります。 これを行うには、戻り値ImageRequestIdをrequestImageメソッドに追加します。これにより、リクエストをさらに識別できます。 



 func requestImage<T: InitializableWithCGImage>( options: ImageRequestOptions, handler: @escaping (T?) -> ()) -> ImageRequestId


もう1つ小さな変更が残っています。



先ほど、deliveryModeをプログレッシブに設定すると、ハンドラーを数回呼び出すことができると述べました。 このハンドラー内で、イメージの最終バージョンと中間バージョンのどちらで呼び出されたかを理解しておくと便利です。 したがって、ImageRequestResult構造体に渡します。この構造体には、画像自体に加えて、リクエストの結果に関するその他の有用な情報が含まれます。 



 func requestImage<T: InitializableWithCGImage>( options: ImageRequestOptions, handler: @escaping (ImageRequestResult<T>) -> ()) -> ImageRequestId struct ImageRequestResult<T> { let image: T? let degraded: Bool let requestId: ImageRequestId }


したがって、インターフェイスに画像を表示するために画像を要求する方法の最終バージョンに到達しました。



他の3つのメソッドは単純で、そのうちの2つは本質的に単なる非同期ゲッターです。 



 protocol ImageSource { func requestImage<T: InitializableWithCGImage>( options: ImageRequestOptions, resultHandler: @escaping (ImageRequestResult<T>) -> ()) -> ImageRequestId func fullResolutionImageData(completion: @escaping (Data?) -> ()) func imageSize(completion: @escaping (CGSize?) -> ()) func cancelRequest(_: ImageRequestId) }


したがって、ピッカーのモデルとしての使用に最適なImageSourceプロトコルを取得しました。ディスク、ネットワーク、およびユーザーのフォトギャラリーからの写真の3つの場合にのみ実装します。



フォトギャラリー



iOS 8以降、フォトギャラリーへのアクセスはPhotos.frameworkを介して行われます。 ギャラリー自体は直接PHPhotoLibraryオブジェクトで表され、写真はPHAssetオブジェクトで表されます。







インターフェイスに表示できる写真のビューを取得するには、UIImage出力を提供するPHImageManagerを使用する必要があります。



この変換を実行するメソッドは次のようになります。 



 func requestImage( for: PHAsset, targetSize: CGSize, contentMode: PHImageContentMode, options: PHImageRequestOptions?, resultHandler: @escaping (UIImage?, [AnyHashable: Any]?) -> ()) -> PHImageRequestID


ご覧のとおり、これは独自のImageSourceプロトコルで画像を取得する方法に非常に似ています：同じターゲットサイズ、コンテンツモード、いくつかのパラメーター、非同期結果ハンドラー。



ImageSourceの最初の実装はPHAssetの単なるラッパーであったため、これは驚くことではありません。したがって、この特定の署名に大きく依存していました。



残念ながら、PHImageManagerの操作を調査する過程で、いくつかの滑りやすい点に遭遇したため、独自のrequestImageメソッドの本体は、この標準メソッドの単一の呼び出しで構成されていなかったようです。



これらの最初のものは、コレクションビューで写真を表示するという古典的な問題を解決することに現れました。



  1. PHImageManagerは、リクエストがキャンセルされた後にresultHandlerがどのように呼び出されるかについては一切保証しません。 呼び出されない場合もあれば、呼び出される場合もありますが、同時にUIImageを取得する場合もあれば、代わりに-nilを取得する場合もあります。 正確に何が起こったのかを把握する必要がないように、クライアントコードを簡素化したかったのです。



  2. そのため、ImageSourceのresultHandlerを呼び出すための厳密なルールセットが登場しました。その1つは、リクエストのキャンセル後にresultHandlerを呼び出すべきではないと述べています。



この問題の解決策は非常に簡単でした。 resultHandler PHImageManagerには、入力する2つのパラメーターがあります。1つ目はUIImage、2つ目は情報ディクショナリーで、すべての有用な情報が含まれています。 



 //  resultHandler PHImageManager' let cancelled = (info?[PHImageCancelledKey] as? NSNumber)?.boolValue ?? false || cancelledRequestIds.contains(requestId) if !cancelled { //  "̆" resultHandler }


この情報の中には、リクエストがキャンセルされたかどうかを判断できるフラグがあります。 ただし、このresultHandler呼び出しがキューに入った後にリクエストがキャンセルされた場合、このフラグは送信されません。 したがって、キャンセルされたrequestIdの配列をImageSource内に保持し、リクエストの存在を確認する必要がありました。



2番目の問題は、iCloudからの写真に遭遇したときに現れ、読み込み時にアクティビティインジケータを表示する必要がありました。



このような負荷を追跡する唯一の方法は、PHImageRequestOptionsオブジェクトにプログレスハンドラーを設定することです。これは、画像が要求されたときにPHImageManagerに渡されます。 



 class PHImageRequestOptions { //  PHImageManager var progressHandler: PHAssetImageProgressHandler? // ... }


ダウンロードの開始と終了の事実を追跡するだけでよいため、要求パラメーターを使用して、このような2つのクロージャーを独自の構造に追加しました。 そして、progressHandlerの最初の呼び出しで単純にonDownloadStartをプルした場合、onDownloadFinishではそれほど簡単ではありませんでした。 



 struct ImageRequestOptions { //  ImageSource var onDownloadStart: ((ImageRequestId) -> ())? var onDownloadFinish: ((ImageRequestId) -> ())? }


幸運なことに、progressHandlerから画像が100％ロードされたと報告された場合、progress == 1の値に対応し、この場所でonDownloadFinishを呼び出しました。 



 phImageRequestOptions.progressHandler = { progress, _, _, _ in if progress == 1 { callOnDownloadFinish() } }


ただし、これは発生しない可能性があり、progressHandlerの最後の呼び出しは100％未満の進行で発生します。 この場合、既にresultHandler内で、ダウンロードが完了したかどうかを推測しようとしています。 



 //  resultHandler: let degraded: Bool = info?[PHImageResultIsDegradedKey] let looksLikeLastCallback = cancelled || (image != nil && !degraded) if looksLikeLastCallback { callOnDownloadFinish() }


このコールバックで提供される情報ディクショナリには、画像の最終バージョンと中間バージョンのどちらを受け取ったかを示すIsDegradedフラグがあります。 そのため、この段階では、リクエストをキャンセルした場合、または画像の最終バージョンが来た場合にダウンロードが完了したと想定するのが論理的です。
