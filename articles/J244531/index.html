次のプロジェクトでSpring Bootを使用する必要がありますか？





過去10年間の私のプロフェッショナルなJavaプロジェクトのほとんどは、 SpringまたはJEEに基づいていました。 どちらのプラットフォームも非常に自信を持って開発されていますが、それでもさまざまな問題に悩まされています。



SpringとJEEは、大規模なJavaプロジェクトや、複雑なエンタープライズソリューションで作業する大規模な開発チームのゴールドスタンダードです。 しかし、Javaコミュニティの成熟の必然的な結果は、以前の熱意と革新の部分的な喪失であり、両方の信頼性に大きく打撃を与えました。





JEEはここ数年で劇的に変化しましたが、EJB 2.xのバージョン以降廃止されたプラットフォームの作成者によって指定されたアプローチとソリューションについて、開発者から非難されています。 多くの人がJEEを「J2EE」と呼んでいますが、名前の変更は8年前に行われました！



Springも著しく改善されていますが、すべてのユーザーがこれを認識しているわけではありません。 コード構成から透過的で非破壊的なSpring 3.x以降の最新アプリケーションを使用して作成する可能性はありますが、ほとんどのプロジェクトはXMLファイルと時代遅れのアーキテクチャソリューションに過度に魅了され続けています。 問題は、多くの開発者が何らかの理由で以前のバージョンで実行されたアプローチを変更しないことです。



これらの理由により、他の言語とフレームワークへの道が開かれました。 Javaはおそらく12年以上前に個人プロジェクトで最も人気のある言語でしたが、今日の若い開発者はPython、Scalaなどにもっと注意を向け始めているようです。 過去10年間で最も影響力のあるWebフレームワークであるRailsとRuby SinatraベースのWebフレームワークは、過去5年間で多くのマイクロフレームワークを生み出しました。



Javaの世界でのGroovyとGrailsは 、RubyとRailsからの最初の真剣な答えでした。 現在では、最も保守的なエンタープライズ開発チームのツールボックスにもあります。 ただし、新しいJVMベースのフレームワークはさらに進歩しています。 JEEとSpring APIを使いやすいPlayシェルインターフェースにラップする代わりに、フレームワークは、一見基本的なJavaサーブレットモデルでさえ、いわばゼロから破棄し始めました。



Springは、もはや最新の革新的なツールとは見なされません。 開発者は、特に古いアプリケーションでそれを使用しますが、主に必要なためであり、彼ら自身がそれを望んでいるためではありません。 Playやまったく異なるプラットフォームがいたるところに存在する一方で、個人プロジェクトでSpringを使用した開発者と最後に話したことを思い出せません。



これは残念です。Springは、正しく構成できれば非常に強力なツールだからです。 Spring Data JPAは、DAOクラスを作成せずにリレーショナルデータベースのシンプルな管理を提供します。また、Spring Data JPAを使用すると、NoSQLデータストアを正しく構成できれば、同じ機能を取得できます。 さらに、Springの助けを借りて、 エンタープライズ統合を実行したり、WebまたはAndroidアプリケーションで最も人気のあるソーシャルサービスのAPIにアクセスしたり、 Spring Securityを介してアプリケーションのセキュリティを管理したりできます。設定する。



Springベースのアプリケーションの構築は、非常に苦痛なプロセスです。 これは、テクノロジーを選択する際の既存の選択肢が多数あるためです。 たとえば、すでに時間をかけてJdbcTemplateとHibernate / JPAを学習している場合、Spring Data JPAを学習して使用する価値はありますか？ Spring Data RESTとSpring HATEOASの違いは何ですか？



もう1つの複雑な要因は、Springが非推奨のものをほとんど認識せず、技術の選択に関する情報に基づいた決定を下し、さまざまな競合を排除し、他の頻繁に発生する問題を解決するのに十分な情報を提供しないことです。 問題の修正例をインターネットで検索すると、検索結果の上部に古いアプローチや解決策へのリンクが表示されます。 ほとんどの場合、これは、注釈ベースの構成の実装と保守が容易であるにもかかわらず、XML構成が広く普及しているためです。 ThymeleafやVelocityのようなクリーンなテンプレートシステムを使用する代わりに、ほとんどのアプリケーションは引き続きJSTLでJSPを使用します。



これらの問題を解消することへの渇望は、Springの作成者の間で消えることはありません。 Railsコマンドラインツールに触発されて、 Spring Rooを紹介しました。これは、コマンドラインを介してWebコントローラーやJPAエンティティなどの要素を作成できる迅速な開発システムです。 ただし、非自明な使用では、Spring Rooをマスターすることは、アプリケーションを手動でビルドするのとほとんど同じくらい困難です。 多くの開発者は、Rooがプロジェクトに普遍的に追加する「マジック」を提供する豊富な注釈とAspectJファイルに追われました。 Rooはプロジェクトから簡単かつ痛みを伴わずに削除できると述べていますが、必要に応じて、現実は理論よりも厳しいです。 そして、たとえ成功したとしても、AspectJをJavaに変換すると、魔法のコマンドラインツールを使用できなくなります。



Spring Bootは、次世代のSpringアプリケーション構成簡素化ツールです。 これは自動コード生成の手段ではありませんが、プロジェクトアセンブリの自動化システムのプラグインです（MavenおよびGradleをサポート）。



プラグインは、Springアプリケーションをテストおよびデプロイするための機能を提供します。 mvn spring-boot：runコマンドは、ポート8080でアプリケーションを起動します。これは、かなり人気のあるMaven Jettyプラグインに似ています。 さらに、Spring Bootを使用すると、アプリケーションを完全なTomcatコンテナが埋め込まれた別個のjarファイルにパックできます。 このアプローチは、Playアプリケーションのデプロイメントモデルから借用しています（同時に、従来のwarファイルを作成することもできます）。



Spring Bootの主な利点は、クラスパスの内容に基づいたリソース構成です。 たとえば、Mavenプロジェクトのpom.xmlファイルにJPA依存関係とPostgreSQLドライバーが含まれている場合、Spring BootはPostgreSQLの永続化モジュールを構成します。 Web依存関係を追加すると、デフォルトでSpring MVCが構成されます。 永続性が必要な場合、Spring BootはHibernateをHSQLDBデータベースを持つJPAプロバイダーとして設定します。 Webアプリケーションを作成しているが、さらに何も指定しない場合、Spring BootはThymeleafテンプレートシステムのビューリゾルバーを構成します。



デフォルトの設定といえば、Spring Bootはこの点に関して非常に直感的です。 あなたは彼の設定の選択にいつも同意するとは限りませんが、少なくとも彼はあなたに作業モジュールを提供します。 これは、特にデフォルト設定で作業を開始し、既存の代替案を検討しながら構成を変更できる初心者の開発者にとって非常に便利なアプローチです。 同意する、これは難しい質問をたくさん得るよりもはるかに優れている。 さらに、プロジェクトの公式ページには 、「Hello world」レベルですべての主要なタイプのプロジェクトをすばやく理解し、実際に実装できる完全なチュートリアルが多数あります。







実際、アプリケーションインフラストラクチャを構築するには、pom.xmlに必要なモジュールを追加します。 



... <parent> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-parent</artifactId> <version>1.0.0.RC3</version> </parent> ... <properties> <start-class>com.mypackage.Application</start-class> <java.version>1.7</java.version> </properties> ... <dependencies> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-web</artifactId> </dependency> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-data-jpa</artifactId> </dependency> <dependency> <groupId>com.h2database</groupId> <artifactId>h2</artifactId> <version>1.3.174</version> </dependency> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-test</artifactId> <scope>test</scope> </dependency> </dependencies> ... <build> <plugins> <plugin> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-maven-plugin</artifactId> </plugin> </plugins> </build> ...




メインJavaクラスを次のように記述します。 



 @Configuration @EnableAutoConfiguration @ComponentScan @PropertySource("classpath:application.properties") public class Application { @Bean MyCustomService myCustomService() { return new MyCustomService(""); } public static void main(String[] args) { ApplicationContext ctx = SpringApplication.run(Application.class, args); System.out.println("Let's inspect the beans provided by Spring Boot:"); String[] beanNames = ctx.getBeanDefinitionNames(); Arrays.sort(beanNames); for (String beanName : beanNames) { System.out.println(beanName); } } }




ほとんどの場合、デフォルト設定を変更するには、POMファイルを変更するだけです。 そのため、上記の例では、H2データベースの依存関係の追加が示されています。 この変更を確認したSpring Bootは、デフォルトのHSQLDBではなくH2データベースのJPA永続性モジュールを構成します。 デフォルトのTomcatの代わりにJettyを組み込みコンテナーとして使用する場合は、適切な依存関係を追加するだけです。



現時点では、Spring Bootはまだ初期段階にあり、もちろん、安定したレベルに移行する前に、多くの変容を経験する必要があります。 本格的なシステムの構築に使用するには時期尚早かもしれませんが、さまざまな種類の個人、トレーニング、およびテストプロジェクトに非常に適しています。その実装は、有用な機能の作成とはまったく関係のない、望ましくない量の非生産的で日常的な作業を取り除くことが非常に重要です。



Spring Bootの深刻なツールとして将来的にSpring Bootを開発するという潜在的な見通しのコンテキストでは、許容できる技術文書（英語のみであり、Spring Frameworkの800ページの百科事典と比較してまだやや弱い）の存在が特に勇気づけられます。
