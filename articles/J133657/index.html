Mobile Professional Daysでハリコフに行った方法

すべての良い一日。

2011年11月19日に、ハリコフ国立無線大学（KNURE）で、Mobile Professional Daysカンファレンスが開催されました。

これは、「モバイルプラットフォームiOS、Android、Bada、およびWP7向けの最新のモバイルアプリケーションの作成」を引用したものです。

主な聴衆は再び、引用：

  • プログラマー
  • プロジェクトマネージャー;
  • デザイナーとユーザビリティのスペシャリスト。
  • QAスペシャリスト。
  • 生徒と教師。


私の心が溶けたのはデザイナーとユーザビリティの専門家でした その後、私はためらうことなく会議のウェブサイトに登録し、蒸気機関車のチケットを購入し、ハリコフに向かいました。

興味のあるすべての人にすぐに言わなければなりません。それほど悪くはありません。それはあなたが望む方法ですが、会議は悪くありませんでした。

30分ごとのレポートに誰が何を期待しているかはわかりませんが、そのレベルは主に初心者向けであり、専門家ではない人向けです。 個人的に、私は自分が計算しているものを見たので、それをレビューしようとします。

10：00-10：30 Pavel Kravchenko

チームリーダー/ PM / iOS開発者（Ciklum、Service2Media、Kharkov）

アジャイルモバイル開発方法論



最初のスピーカーはPavel Kravchenkoで、モバイル開発者の分散チームのチームリーダーと彼の経験を聴衆と共有しました。

報告書は悪くありませんでしたが、チームのリーダーシップで特別な何かについて秘密のベールを開いたとは言いません。

パベルはかなり陽気な人であり、リーダーとして、彼はチームで前向きな気分を維持しようとしていると思います。 しかし、賢明なリーダーで明確に表現できるプラスは、すぐれたマネージャーではなくグローバルなマイナスになります。 そして、このケースで何をすべきか、チーム自体、ポールは言いませんでしたが、彼の意見を聞くのは面白いでしょう。

プレゼンテーションのいくつかの興味深い瞬間：





合計：

  1. かなり議論の余地のあるフレーズが聞かれました。分散したチームは、従来の「シングルオフィス」チームの有効性の約70％を提供します。 私はこの数字は非常に楽観的であると言うでしょう：私の観察によれば、3人以上の分散したチーム、神は禁じていますが、60パーセントを与えるでしょう。 3未満の場合、差はほぼ均等になります。 それはすべてリーダーに依存しますが。
  2. スピーカーは、地域の分割、言語の壁、時間帯、その他の人々の仕事に関連する主なリスクと問題について説明しました。 ここでそれらを解決する方法について、レポートの時間が終了し始めたので、それはくしゃくしゃになったことが判明しました。 しかし、ここで聴衆は私たちを失望させず、カテゴリからの純粋に実用的な質問が行きました：「承認された予算を超えたらどうなりますか？」


デザイナー向け：

  1. ここでは、主にフリーランサーにとって興味深いものになります。タスク、期限、費用の変更の問題に遭遇することも非常に多いからです。 すでに処理中です。 また、どれだけスムーズかつ迅速にすべてを解決できるため、多くの場合、プロジェクトを保存して完了する能力は異なります。
  2. 突然、神が禁じた場合、デザイナーの分散チームを率いて（かつてこれをやったことがない）、その後すぐにZenの練習を開始する必要がありました。 「今、あなたは軍隊にいます、息子」そして顧客自身の「特徴」と一緒に、あなたはあなたの病棟の「特徴」に対処しなければなりません。 そして最も攻撃的：誰もが正しいでしょう、そしてそれらの間であなた。 すべてを自分でやろうとしないでください-すぐに作業を委任し、プロジェクトの制御点を正しく設定することを学んでください。 最も困難な瞬間に、同じような状況にいることを思い出してください。そうすれば、「あなた」と「現在」の行動がどれほど似ているかに驚くでしょう。




10：00-10：30 Pavel Taikalo

iOS開発TechLead（Stanfy、キエフ）

iOS開発：始まり



聴衆が近くにいないという事実と、報告の間にギャップがないため、私は遅れて、大勢の人々と一緒に、物語が続いたときにすでに爆発しました。

しかし、この不幸な事実と、私は開発者ではないという事実と、レポートが特にそれらに焦点を合わせているにもかかわらず、私はそれが好きでした。

明らかに、賢明に、そしてむしろ陽気に、Pavelは開発者に、アプリケーションを正しく作成するために知っておくべきことを伝えました。 デザイナーとして、スピーカーがすべてのフロントエンド開発者にHIG Appleを読むように促したとき、私は泣きそうになりました。 特に、彼は多くの「明白な」ものや「直感的な」ものでさえ、繰り返すこと、あるいは直感的、直観的に悪くはないが、それがビジネスにどのようになるかを示唆した...

また、Pavelは重要な事実を強調しました。それは、私がますます頻繁に気づき始めたものです。 開発者はエンドユーザーからますます離婚するようになっています 。 エミュレータはデバイスよりも高速で、ほとんどの場合、開発者のデバイスは2番目のiPhoneよりも高速です。



合計：

  1. マテリアルを学びます。特に、HIGには、Appleインターフェースの構築の原則と一般的なユーザビリティの両方を理解するのに役立つ多くのことがあります。
  2. 基本的な必要事項の知識があるかどうかを確認します（残念ながら、Pavelのプレゼンテーションは見つかりませんが、ジュニア、シニア、アーキテクトの開発者が知っておくべきことのリストを示すスライドがありました）。
  3. 既製のGUIテンプレート（iPhoneとAndroidの両方で使用可能）を使用します-ファイナルで多くの時間を節約し、エラーを減らします。




デザイナーに：

  1. 私たちにとって、ガイドの知識は必須の原則です。 当然のことながら、特に非標準のコントロールを使用している場合は、これらを回避できる場合があります。 結局のところ、ガイドは必ずしも厳密な要件ではなく、多くの場合推奨事項でもありません。 特にAndroidで。 ただし、これには注意が必要です。 私が何をお勧めしますか（特に始めている人に）：



    • www.idemployee.id.tue.nl/gwmrauterberg/lecturenotes/ui - guide - line - collection.htm （ガイドの優れたコレクション）

    • developer.apple.com /library/ios/#documentation/userexperience/conceptual/mobilehig/Introduction/Introduction.html（iOS用のHIG）

    • developer.android.com /guide/practices/ui_guidelines/index.html（Androidガイドライン

    • mobile.tutsplus.com/category/tutorials/mobile - design - tutorials （モバイルインターフェースの作成に関する優れたレッスンと記事のセット）。

    • www.teehanlax.com/blog/iphone - gui- psd- v4 （iOS GUI psd）

    • mobile.tutsplus.com/tutorials/mobile - design - tutorials /5- great- sites-for- ios - design - inspiration （インスピレーションのサイト）

    • www.onextrapixel.com/2011/11/16/50 - useful - and - free - web - ui - mobile - ui - and - wireframe - kits （さまざまな便利なクジラのセット）

    •Smashing Magazinesの非常に便利な本もあります。これは、特にモバイルデバイスのデザインに関するものです。特に、この問題に関するPhotoshopでのカラープロファイルの設定に関する優れた章です。
  2. 描画を開始する前に開発者と話す機会を決して無視しないでください。 カスタムコントロールとカスタム要素を導入する可能性について議論します。 ご存知のとおり、アプリケーションアーキテクチャを一緒に調整することもできます。 一般的に、それは役に立ちます。
  3. さまざまなデバイス（マシン）で設計をテストしてみてください。 これは、モバイルデザインだけではありません。 誰もが8 GBのRAM、2ギガバイトのビデオカード、3ギガヘルツ以上の周波数を持つプロセッサを持っているわけではありません。 また、モニターはまったく調整されていません（残念ながら事実ですが）。慎重に選択されたシェードやエレガントで控えめな質感を誰も評価することはありません。


11：30-12：10アレクサンダー・エゴシン

モバイルプロダクションの責任者（アラワルエンターテイメント、ノボシビルスク）

モバイルゲーム：開発は戦いの半分に過ぎません



会議の開始前でさえ、アレクサンダーは「招かれた星」として位置づけられていました。

私は彼のプレゼンテーションを読むことに興味がある人とそうでない人に強くお勧めします。

その人は多くのパフォーマンス経験を持っているとすぐに感じます。これは、プレゼンテーションのデザイン方法とスピーカーのプレゼンテーション方法の両方で見ることができます。すべてがしっかりしていて、面白くて便利です。



アレクサンダーが伝えようとしたことの本質は、ゲームの開発を始める前に、次のことをする必要があるということです。

  • アイデアをテストするのは良いことです。それはあなたとあなたの愛する人以外の誰にとっても面白いですか、
  • ターゲットオーディエンスを評価し、
  • 収益化と開発の計画を立て、
  • 追加の「チップ」について考えます。




さまざまなタイプのデバイスの適応機能をすぐに検討してください。 たとえば、iPhoneのタッチスクリーンコントロールの要素はiPadにとって必ずしも便利ではありません。手の動きや指の長さが十分ではありません。

ゲームのローカライズの重要性については別の議論がありました。



また、お金を得る価値があるものとそうでないものを理解することも良いことです。 特に、一部のゲーム「チップ」の直接マネタイズに加えて、オファー（換言すれば広告）によるマネタイゼーションもあります。

一般的に、スピーカーは収益化の問題を「創造的に」扱うことを提案しました。 たとえば、非常に難しいレベルでチップを販売したり、ゲーム通貨の異なるセットを作成したりします。



当然、Alawarが開発者に提供する条件は表明されました-そして突然、これらは観客に座っています。



私の意見：会議の最高の報告の一つ。 アレクサンダーがAndroidタブレット市場をブランド化したという事実にもかかわらず、恥ずかしくて、彼はまったくそこにいないと言いました。

バッグにGalaxy Tabが入っていて、その上にカンファレンスのレポートを追加したのはどうしてですか。 もちろん、私はまだ市場を作っていませんが、クリスマスツリー、スティック、世界中のAndroidタブレットの販売レベルを見てください。 もちろん、これまでのところ、iPadの売り上げに近いわけではないので、古いiOSデバイスはどれくらい経験がありますか？！ 原理的には比較できませんが。 Montezuma's TreasuresをGalaxyにダウンロードすると、Alexanderが会社の立場を明確に表現していることに気付きました。「Androidタブレットはありません！」-ゲームがクロールして見た目が怖い



合計：

  1. アイデア、開発計画、収益化方法、対象読者の関連性を評価するまで、ゲームの開発を開始しないでください。
  2. すぐに一連の「チップ」を考え出します。アプリケーションの起動後、その導入により、ユーザーのデバイスでのゲームのライフパスを延長できます（より多くのアップデートとギフト-長寿命）。
  3. ローカライズを軽視しないでください。
  4. iOSとは異なり、Androidはさまざまなデバイスで構成されているため、さまざまな解像度に移植または適応する場合は、主なデバイスに注目してください。
  5. 自問してみてください。すべてのデバイスがマルチタッチ、スライドなどをサポートしていますか。
  6. 収益化で無理をしないでください。ただし、アプリケーションの開発時に実装することを忘れないでください。
  7. テストを怠らないでください。
  8. Alawarは著者と開発者を探しています。


デザイナーに：

  1. プロトタイプを無視しないでください。 特に、アプリケーション（ゲーム）がスマートフォンとタブレットにあることがわかっている場合。 片方の手（またはiPadの場合は2つ）でデバイスを持ち、親指で画面の周りに半円を描き、ユーザーがすべてのコントロールに届くように腱を切る必要があるかどうかを確認します。
  2. 競争力のあるアプリケーション（もしあれば）を勉強するのを怠らないでください。 準備段階で問題を解決しなければ、それらの欠点はほぼ100％になります。
  3. ジェスチャーを考えてください！ インターフェースの直観性（特にそれが主にあなたとアプリケーション開発者にとって「直感的」である場合）と「これはすべてのiPhoneユーザーに明らかである」という事実に集中することは必ずしも価値がないかもしれません。 実践が示しているように、常にではありません。 多くの場合、特別な制御により管理が簡素化され、アプリケーション全体の使いやすさが向上します。
  4. デバイスでデザインをテストしてください。 それを借りることができる誰かが常にあります。
