エントリー



Expressは、最高のNodeフレームワークの1つです。 優れた開発者サポートと多数の便利な機能を備えています。 Expressには、それを扱うためのすべての基本事項を網羅した素晴らしい記事がたくさんあります。 しかし、今はもう少し掘り下げて、本格的なWebサイトを作成した経験を共有したいと思います。 一般に、この記事はExpress自体に関するだけでなく、開発者が利用できる他の劣らない素晴らしいツールとの組み合わせについても説明しています。

Node.jsに精通し、システムにインストールしたこと、およびその上にいくつかのアプリケーションを既に作成していることを信じています。

ExpressフレームワークはConnectに基づいています 。 これは、多くの便利な機能を備えたミドルウェア機能のセットです。 ミドルウェア関数とは何かに興味があるなら、ここに小さな例があります： 



var connect = require('connect'), http = require('http'); var app = connect() .use(function(req, res, next) { console.log("That's my first middleware"); next(); }) .use(function(req, res, next) { console.log("That's my second middleware"); next(); }) .use(function(req, res, next) { console.log("end"); res.end("hello world"); }); http.createServer(app).listen(3000);


ミドルウェア関数は、基本的に、リクエストオブジェクトと応答オブジェクトをパラメーターとして受け入れる関数であり、次に呼び出されるコールバック関数でもあります。 各ミドルウェア関数は、応答オブジェクトを使用して応答するか、ストリームを次のコールバック関数に渡すかを決定します。 上記の例では、2番目の関数でnext（）メソッド呼び出しを削除すると、文字列「hello world」はクライアントに送信されません。 一般的に、これがExpressの仕組みです。 それは間違いなくあなたの時間を節約するいくつかの定義済みのミドルウェア機能を備えています。 たとえば、リクエストボディを解析し、コンテンツタイプapplication / json 、 application / x-www-form-urlencodedおよびmultipart / form-dataをサポートするボディパーサーです。 または、 Cookieヘッダーを解析し、キーがCookieの名前であるオブジェクトをreq.cookiesフィールドに入力するCookieパーサー。

実際、Expressはそれ自体をConnectフレームワークでラップし、ルーティングロジックなどのいくつかの機能で補完して、ルーティングプロセスをスムーズに進めます。 以下は、GET要求を処理する例です。 



 app.get('/hello.txt', function(req, res){ var body = 'Hello World'; res.setHeader('Content-Type', 'text/plain'); res.setHeader('Content-Length', body.length); res.end(body); });




設置



Expressのインストールには2つのオプションがあります。 1つ目は、その説明をpackage.jsonファイルに配置してnpm installコマンドを実行することです（このパッケージマネージャーの名前は「no problem man」を表しているようなジョークがあります:)）。 



 { "name": "MyWebSite", "description": "My website", "version": "0.0.1", "dependencies": { "express": "3.x" } }


フレームワークコードはnode_modulesディレクトリに配置され、インスタンスを作成できます。 私は代替手段を好みます：コマンドライン。 npm install -g expressを実行してExpressをインストールするだけです。 これにより、新しいCLIツールが作成されます。 たとえば、次の場合： 



 express --sessions --css less --hogan app


Expressは、構成済みのアプリケーションフレームワークを生成します。 以下は、 エクスプレス（1）コマンドオプションのリストです。 



 Usage: express [options] Options: -h, --help output usage information -V, --version output the version number -s, --sessions add session support -e, --ejs add ejs engine support (defaults to jade) -J, --jshtml add jshtml engine support (defaults to jade) -H, --hogan add hogan.js engine support -c, --css add stylesheet support (less|stylus) (defaults to plain css) -f, --force force on non-empty directory


ご覧のとおり、使用可能なオプションのリストは小さいですが、これで十分です。 通常、 lessライブラリをCSSプリプロセッサとして使用し、 hoganテンプレートエンジンも使用します。 この例では、セッションのサポートが必要です。--sessionsオプションはこの問題を解決します。 チームが完了すると、プロジェクトの構造は次のようになります。 



 /public /images /javascripts /stylesheets /routes /index.js /user.js /views /index.hjs /app.js /package.json


package.jsonファイルを見ると、必要なすべての依存関係がここに追加されていることがわかります。 しかし、まだインストールされていません。 これを行うには、npm installコマンドを実行します 。その後、 node_modulesディレクトリが表示されます 。

上記のアプローチは、ユニバーサルと呼ぶに値しないことを理解しています。 たとえば、ルートハンドラを別のディレクトリなどに配置する必要がある場合があります。 しかし、次のいくつかの章で示されるように、生成された構造に変更を加えます。これは非常に簡単です。 これに基づいて、 express（1）コマンドを単にテンプレートジェネレータとして使用することをお勧めします。

始める前にやらなければならないことがいくつかあります。 構成のセットアップはそのようなものの1つです。 小規模なサーバーが、ローカル、中間、戦闘の3つの異なるサーバーに展開されることを想像してみましょう。 当然、3つの環境すべての設定は異なるため、このような状況に十分に柔軟なメカニズムを実装する必要があります。 ご存知のように、各nodejsスクリプトはコンソールアプリケーションとして実行されます。 つまり、現在の環境を決定するコマンドにパラメーターを簡単に割り当てることができます。 後でテストを書くのが便利になるように、この部分を別のモジュールにラップしました。 /config/index.jsファイル： 



 var config = { local: { mode: 'local', port: 3000 }, staging: { mode: 'staging', port: 4000 }, production: { mode: 'production', port: 5000 } } module.exports = function(mode) { return config[mode || process.argv[2] || 'local'] || config.local; }


現時点では、モード（モード）とポート（ポート）の2つのパラメーターしかありません。 ご想像のとおり、アプリケーションは異なるサーバー上の異なるポートを使用します。 そのため、 app.jsファイルでサイトのエントリポイントを更新する必要があります。

データベース



動的なWebサイトを構築する場合、情報を保存するデータベースが必要です。 このため、私はこの指示のためにmongodbを選択しました 。 MongoはNoSQLデータベースです。 ここにインストール手順があります 。私はWindowsユーザーなので、Windows のインストール手順に従いました 。 インストールが完了したら、デフォルトでポート27017をリッスンするMongoDBデーモンを起動します。したがって、理論的には、このポートに接続してmongodbサーバーとメッセージを交換する機会があります。 ノードスクリプトを介してこれを行うには、 mongodb module / driverが必要です 。 この手順のソースファイルをダウンロードした場合、モジュールは既にpackage.jsonファイルに追加されています。 そうでない場合は、 依存関係プロパティに「mongodb」：「1.3.10」を追加し、 npm installコマンドを実行します。 次に、mongodbサーバーが実行されているかどうかを確認するテストを作成します。



/tests/mongodb.spec.jsファイル： 



 describe("MongoDB", function() { it("is there a server running", function(next) { var MongoClient = require('mongodb').MongoClient; MongoClient.connect('mongodb://127.0.0.1:27017/fastdelivery', function(err, db) { expect(err).toBe(null); next(); }); }); });


mongodbクライアントメソッド.connectのコールバック関数は、 dbオブジェクトを受け取ります。 後でデータの管理に使用します。 これは、モデル内でアクセスする必要があることを意味します。 データベースを照会するたびに新しいMongoClientオブジェクトを作成することはお勧めできません。 そのため、Expressサーバーの起動をデータベースに接続するコールバック関数に転送しました。 



 MongoClient.connect('mongodb://127.0.0.1:27017/fastdelivery', function(err, db) { if(err) { console.log('Sorry, there is no mongo db server running.'); } else { var attachDB = function(req, res, next) { req.db = db; next(); }; http.createServer(app).listen(config.port, function(){ console.log('Express server listening on port ' + config.port); }); } });


環境構成設定ファイルがあるため、mongodbサーバーのホスト名とポートをここに配置し、接続URLを次のように変更することをお勧めします。



'mongodb：//' + config.mongo.host + '：' + config.mongo.port + '/ fastdelivery'



http.createServer関数の呼び出しの直前に追加したミドルウェア関数attachDBに特に注意してください 。 この小さな追加のおかげで、リクエストオブジェクトの.dbプロパティを入力します。 良いニュースは、ルート決定中にいくつかの機能を設定できることです。 例： 



 app.get('/', attachDB, function(req, res, next) { ... })


したがって、ExpressはあらかじめattachDB関数を呼び出して、ルートハンドラーに入るようにします。 これが発生すると、リクエストオブジェクトに.dbプロパティが設定され、それを使用してデータベースにアクセスできます。

モデル


モデルは、アプリケーションのデータを処理するものです。 MongoClientオブジェクトによって返されるdbオブジェクトへのアクセス権が必要です。 さまざまなタイプのモデルが必要になる可能性があるため、モデルにはそれを拡張する方法も必要です。 たとえば、 BlogModelまたはContactsModelモデルを作成する場合があります。 これを行うには、最初にテスト/tests/base.model.spec.jsを作成する必要があります。 そして、これらのテストを作成することにより、モデルが必要なことだけを行い、それ以外は何もしないことを保証できます。 



 var Model = require("../models/Base"), dbMockup = {}; describe("Models", function() { it("should create a new model", function(next) { var model = new Model(dbMockup); expect(model.db).toBeDefined(); expect(model.extend).toBeDefined(); next(); }); it("should be extendable", function(next) { var model = new Model(dbMockup); var OtherTypeOfModel = model.extend({ myCustomModelMethod: function() { } }); var model2 = new OtherTypeOfModel(dbMockup); expect(model2.db).toBeDefined(); expect(model2.myCustomModelMethod).toBeDefined(); next(); }) });


実際のdbオブジェクトの代わりに、モックオブジェクトを渡すことにしました。 これは、後で特定の何かをテストしたい場合に行われます。これは、データベースからのデータに依存します。 このデータを手動で定義する方がはるかに簡単です。

プロトタイプmodule.exportsを変更する必要がありますが、元のコンストラクターはそのままにしておく必要があるため、メソッド拡張の実装は少し難しいです。 幸いなことに、コードのパフォーマンスをチェックするテストをすでに作成しています。 上記のコードスニペットのバージョンは次のようになります。 



 module.exports = function(db) { this.db = db; }; module.exports.prototype = { extend: function(properties) { var Child = module.exports; Child.prototype = module.exports.prototype; for(var key in properties) { Child.prototype[key] = properties[key]; } return Child; }, setDB: function(db) { this.db = db; }, collection: function() { if(this._collection) return this._collection; return this._collection = this.db.collection('fastdelivery-content'); } }


ここでは、2つのヘルパーメソッドが定義されています。 データベースからのコレクションのdbオブジェクトとゲッターのセッター。



提出


ビューは画面に情報を表示します。 簡単に言えば、ビューはブラウザーに応答を送信するクラスです。 Expressは、これを行う簡単な方法を提供します。 



 res.render('index', { title: 'Express' });


応答オブジェクトは、生活を楽にする優れたAPIを備えたラッパーです。 ただし、独自のモジュールを作成して、その機能に組み込むことを好みます。 ビューの標準ディレクトリはテンプレートに置き換えられ、ビューの古いディレクトリはベースビュークラスを提供します。

この小さな変更には、別の変更が必要です。 テンプレートファイルが別のディレクトリにあることをExpressに通知する必要があります。 



 app.set('views', __dirname + '/templates');


まず、必要なものをすべて決定し、テストを作成します。その後、実装を開始します。 次の要件を満たすモジュールが必要です。



  • そのコンストラクターは、応答オブジェクトとテンプレート名を受け取る必要があります
  • データオブジェクトを受け入れるテンプレート出力メソッドが必要です。
  • 拡張可能でなければなりません。


驚くかもしれません：なぜプレゼンテーションクラスを拡張する必要があったのですか？ 彼はresponse.renderメソッドを呼び出すだけではありませんか？ 実際、実際には、別のヘッダーを送信したり、応答オブジェクトを操作したりする必要がある場合があります。 たとえば、JSONで送信されたデータを処理します。 



 var data = {"developer": "Krasimir Tsonev"}; response.contentType('application/json'); response.send(JSON.stringify(data));


これを毎回行う代わりに、 HTMLView クラスとJSONViewクラスを用意することは素晴らしいことです。 または、XMLデータをブラウザに送信するXMLViewクラス。 同じコードを繰り返しコピーして貼り付けるよりも、そのような機能が既に実装されている大きなWebアプリケーションを作成する方が簡単です。

以下は、 / views / Base.jsクラスのテストです。 



 var View = require("../views/Base"); describe("Base view", function() { it("create and render new view", function(next) { var responseMockup = { render: function(template, data) { expect(data.myProperty).toBe('value'); expect(template).toBe('template-file'); next(); } } var v = new View(responseMockup, 'template-file'); v.render({myProperty: 'value'}); }); it("should be extendable", function(next) { var v = new View(); var OtherView = v.extend({ render: function(data) { expect(data.prop).toBe('yes'); next(); } }); var otherViewInstance = new OtherView(); expect(otherViewInstance.render).toBeDefined(); otherViewInstance.render({prop: 'yes'}); }); });


レンダリング（視覚化）をテストするには、レイアウトを作成する必要がありました。 この場合、テストの最初の部分で、Expressフレームワークの応答オブジェクトをシミュレートするオブジェクトを作成しました。 テストの2番目の部分では、基本クラスを継承し、独自のレンダリングメソッドを使用する別のViewクラスを作成しました。

クラスコード/views/Base.js ： 



 module.exports = function(response, template) { this.response = response; this.template = template; }; module.exports.prototype = { extend: function(properties) { var Child = module.exports; Child.prototype = module.exports.prototype; for(var key in properties) { Child.prototype[key] = properties[key]; } return Child; }, render: function(data) { if(this.response && this.template) { this.response.render(this.template, data); } } }


これで、テスト用のディレクトリに3つのテストがあり、 jasmine-node ./testsコマンドを実行すると、結果は次のようになります。 



 Finished in 0.009 seconds 7 tests, 18 assertions, 0 failures, 0 skipped


コントローラー


ルートについて話し、どのように決定したのか覚えていますか？ 



 app.get('/', routes.index);


上記の例のルートの後にある記号「/」はコントローラーです。 これは、 リクエスト 、 レスポンス 、コールバック関数next（）を受け取るミドルウェア関数です。 



 exports.index = function(req, res, next) { res.render('index', { title: 'Express' }); };


上記は、Expressを使用する場合のコントローラーの外観を説明しています。 express（1）コマンドは、 routesという名前でディレクトリを作成しますが、私たちの場合は、 controllersと呼ばれる方が良いでしょう。 ファイル構造がこのパターンに一致するように名前を変更しました。

単なる小さなアプリケーション以上のものを構築しているため、拡張可能な基本クラスを作成することをお勧めします。 すべてのコントローラーに機能を追加する必要がある場合、この基本クラスはアドオンを実装するのに最適な場所です。 繰り返しますが、私は最初にテストを書いているので、基本クラスに必要なものを定義しましょう：

  • オブジェクトを取得して新しい子インスタンスを返す拡張可能なメソッドが必要です
  • 子インスタンスにはrunメソッドが必要です。これは古き良きミドルウェア関数です
  • コントローラを識別する名前プロパティが必要です
  • 基本クラスから継承された独立したオブジェクトを作成できるはずです。


ほんの数点ですが、後で機能を追加することができます。

テストは次のようになります。 



 var BaseController = require("../controllers/Base"); describe("Base controller", function() { it("should have a method extend which returns a child instance", function(next) { expect(BaseController.extend).toBeDefined(); var child = BaseController.extend({ name: "my child controller" }); expect(child.run).toBeDefined(); expect(child.name).toBe("my child controller"); next(); }); it("should be able to create different childs", function(next) { var childA = BaseController.extend({ name: "child A", customProperty: 'value' }); var childB = BaseController.extend({ name: "child B" }); expect(childA.name).not.toBe(childB.name); expect(childB.customProperty).not.toBeDefined(); next(); }); });


以下は/controllers/Base.jsの実装です： 



 var _ = require("underscore"); module.exports = { name: "base", extend: function(child) { return _.extend({}, this, child); }, run: function(req, res, next) { } }


もちろん、各子孫クラスには、独自のロジックを実装する独自のrunメソッドが必要です。

展開



Express Webサイトをデプロイすることは、Node.jsに他のアプリケーションをデプロイすることと本質的に違いはありません。

  • ファイルはサーバーにアップロードされます
  • ノードプロセスを停止する必要があります（実行中の場合）
  • 必要な依存関係をすべてインストールするには、npm installコマンドを実行します
  • メインノードスクリプトが再度実行されます


Nodeは非常に新しい製品であり、すべてが期待どおりに機能するわけではありませんが、常に改善が行われていることに注意してください。たとえば、foreverモジュールはNode.jsの継続的な動作を保証します。次のコマンドを実行して実行できます。 



 forever start yourapp.js


これはサーバーで使用するものです。この小さなツールは、1つの大きな問題を解決します。node yourapp.jsコマンドを使用してアプリケーションを起動すると、スクリプトが失敗するとすぐにサーバーがクラッシュしますが、永久にアプリケーションを再起動するだけです。

私はシステム管理者ではありませんが、ApacheとNginxでノードアプリケーションを統合した経験を共有したいと思います。統合は開発プロセスの一部だと考えているからです。ご存じのように、Apacheは通常ポート80で実行されます。つまり、localhostまたはlocalhostのブラウザーにアクセスすると：80、Apacheサーバーが提供するページが表示されます。おそらく、ノードスクリプトは別のポートでリッスンしています。したがって、要求を受信して​​目的のポートに送信する仮想ホストを追加する必要があります。たとえば、アドレスexpresscompletewebsite.devの下で、ローカルのApacheサーバーで作成したばかりのサイトをホストする必要があります。最初に行う必要があるのは、ホストファイルにドメインを追加することです。 



 127.0.0.1 expresscompletewebsite.dev


その後、Apacheサーバーの構成ディレクトリにあるhttpd-vhosts.confファイルを変更する必要があります。 



 # expresscompletewebsite.dev <VirtualHost *:80> ServerName expresscompletewebsite.dev ServerAlias www.expresscompletewebsite.dev ProxyRequests off <Proxy *> Order deny,allow Allow from all </Proxy> <Location /> ProxyPass http://localhost:3000/ ProxyPassReverse http://localhost:3000/ </Location> </VirtualHost>




サーバーはポート80からの要求を受け入れますが、ノードスクリプトをリッスンするポート3000に要求を転送します。

Nginxの構成ははるかに簡単で、率直に言って、Node.jsベースのアプリケーションをホストするための最良の選択です。 Apache構成と同様に、hostsファイルにドメインを追加する必要があります。その後、Nginxがインストールされている場所にある/ sites-enabledディレクトリに新しいファイルを作成します。ファイルの内容は次のようになります。 



 server { listen 80; server_name expresscompletewebsite.dev location / { proxy_pass http://127.0.0.1:3000; proxy_set_header Host $http_host; } }




同じホスト構成でApacheとNginxを同時に起動することはできないことに注意してください。これは、どちらもポート80でリッスンするためです。さらに、戦闘サーバーで上記のコードスニペットを使用する場合は、インターネットでより詳細なサーバー構成情報を検索できます。私が言ったように、私はこの分野の専門家ではありません。
