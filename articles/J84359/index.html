iPhone開発：アプリ内購入の統合

アプリ内購入は、アプリケーションの販売やアプリケーションから追加機能を直接整理するためのシンプルで便利なメカニズムです。 In-App Purchasesは簡単に統合でき、新しい販売チャネルを開きます。 App Storeとのやり取りは、バージョン3.0以降のSDKに付属するStoreKit.frameworkを使用して実行されます。





一般的な情報



アプリ内購入には次の3つのタイプがあります。

  • 消耗品
  • 非消耗品
  • サブスクリプション


消耗品は消耗品タイプです。 このタイプの購入は数回購入できます。 たとえば、 エリミネイトゲームでは、プレーヤーがエネルギーを購入しますが、時間が経つと無駄になり、再度購入するか、エネルギーが回復するまで3時間待機します。



非消費型は非消費型です。 購入は1回のみです。 通常、新しいテーマ、追加レベルなどのロックを解除するために使用されます。



サブスクリプション -何かへのサブスクリプション 。 たとえば、追加の機能を開くプレミアムアカウントを持つWebサービス用のiPhoneアプリケーションを作成できます。 サブスクリプションを使用すると、たとえば1か月間または1年間、アカウントをアクティブにできます。



アプリ内購入は、2つのモデルを使用して実装できます。

  • 組み込みモデル
  • サーバーモデル


組み込みモデルを使用すると、機能のロックを解除できます。 原則として、このモデルを使用すると、開発者は事前にそれらを埋め込むことが推奨されます。 この実装では、StoreKitは機能の支払いのみを担当します。 一般に、StoreKitを使用して、購入が成功したか失敗したかを確認できます。



サーバーモデルはより柔軟です。 このモデルには、iPhoneアプリケーション、Appleサーバー、サーバーという3つのエンティティが参加しています。 新しい機能はすべてサーバーに保存されるため、新しい機能や製品を追加するときにアプリケーションを更新する必要はありません。 モデルは次のように機能します。

  1. iPhoneアプリケーションは、サーバーに製品のリストを要求し、
  2. iPhoneアプリケーションは、ユーザーに新製品を表示し、
  3. ユーザーが購入する（または購入しない:)）
  4. iPhoneアプリケーションは、StoreKitを介してAppleサーバーからの購入をリクエストします。
  5. StoreKitは応答を返します。
  6. iPhoneアプリケーションがサーバーに応答を送信します
  7. 回答が再度チェックされます（回答はAppleから送信されたことが確実です）。
  8. その後、iPhoneアプリケーションはサーバーから新しい製品をダウンロードします。


この投稿では、組み込みモデルについて説明します。



実装



AppPurchasesExampleテストケースを使用して、組み込みのアプリ内購入モデルの操作をデモします。 これは、3つのウィンドウを持つ小さなiPhoneアプリケーションです。 最初の（メインでもある）ウィンドウは、デフォルトでユーザーが利用できます。 このウィンドウには、他の2つのウィンドウに関する情報が含まれます。これらのウィンドウは、お金のためにのみロック解除できます。



ステップ1.アプリIDを作成する



iPhone開発者プログラムポータルにアクセスして、[アプリID]タブを開きます。







画面の右上隅で、[新しいアプリID]をクリックします。 次に、アプリケーションに関する情報を入力します。 私の例では、次のようなフォームに記入しました。







バンドル識別子を作成するために、Appleは識別子の一意性を保証し、アプリケーションを公開する際のさらなる問題からあなたを救う逆DNS記法の使用を推奨します。 バンドル識別子に「*」文字を使用しないことが非常に重要です。 「com.wordpress.indiedevelop。*」と入力すると、アプリ内購入は機能しません。



次に、アプリIDのアプリ内購入を有効にします。 [アクション]列の目的のバンドル識別子の反対側にあるアプリIDのリストで、[構成]をクリックします。 [アプリIDの構成]フォームが表示されます。このフォームで、チェックボックス「アプリ内購入を有効にする」を有効にする必要があります。







ステップ2.開発プロファイルの作成



  1. 左の列で[プロビジョニング]をクリックし、[開発]タブに移動します。
  2. [新しいプロファイル]をクリックして、必要な情報をすべて入力します。 [プロファイル名]フィールドは任意です（「InAppPurchasesExample Dev」と書きました）。
  3. 表示されるリストで、作成されたプロファイルのステータスは保留中です。 ページを更新するか、別のタブに移動するか、その逆を行う必要があります。プロファイルをダウンロードできるようになります。
  4. プロファイルは、xCodeでダウンロードおよびインストールできます。 インストールするには、プロファイルをダブルクリックしてクリックするか、プロファイルをxCodeアイコンにドラッグします。


プロファイルが正しくインストールされると、オーガナイザーがxCodeで開き、次のようなものが表示されます。







ステップ3. iTunes Connectでアプリケーションを作成する



アプリ内購入アプリケーションをテストするには、iTunes Connectで作成する必要があります。 これを行うには：

  1. iTunes Connectにアクセスして、[アプリケーションの管理]-> [新しいアプリケーションの追加]をクリックする必要があります。
  2. 「製品には暗号化が含まれていますか？」という質問に対して 否定的に答えてください。
  3. アプリケーション名、説明、バージョン番号、カテゴリなどを指定する必要があるフォームに記入します。 すべてが非常に簡単です。 「SKU番号」フィールドのみが問題を引き起こす可能性があります。 このフィールドは一意である必要があります。「IAPEX」と入力しました（アプリ内購入の例の略）。
  4. 「アップロード」フォームで、「アプリケーションバイナリを後でアップロード」フラグを設定する必要があります。 テスト例の他のすべてのパラメーターとフォームは重要ではありません。




ステップ4. iTunes Connectでアプリ内購入を作成する



  1. iTunes Connectで、[アプリ内購入の管理]-> [新規作成]をクリックし、目的のアプリケーションを選択します。
  2. バンドルIDを選択し、購入情報（タイプ、名前、価格など）を入力します。また、「製品ID」も入力する必要がありますが、これは任意ですが、リバースDNSの使用をお勧めします。 アプリケーションのバンドルIDと機能名から製品IDを作成するのが最適です。 私の例では、次のようになります。






テストアプリケーションでは、製品ID「com.wordpress.indiedevelop.InAppPurchasesExample.f1」と「com.wordpress.indiedevelop.InAppPurchasesExample.f2」を使用して2つのアプリ内製品を作成しました。 タイプが非消耗品の両方の機能。



ステップ5.テストユーザーを作成する



アプリ内購入をテストするには、少なくとも1人のテストユーザーを作成する必要があります。 これは単純に行われます：

  1. iTunes Connectで、[ユーザーの管理]-> [アプリ購入テストユーザー]に移動します。
  2. [新しいユーザーを追加]をクリックします
  3. ユーザー情報を入力してください


ユーザーのメールは本物である必要はありません。 この例では、テストユーザーを1人作成しました。







重要なポイント。 Appleとの契約を完了していない場合、In-App Purchaseは機能しません。 契約を完了するには、連絡先情報、銀行情報、税務情報を指定する必要があります。



ステップ6.プログラミング

ステップ7.テスト



アプリ内購入をテストするには、アプリケーションをコンパイル、インストール、実行する必要があります。 テストするときは、次のことに注意してください。

  1. デバイスでのみ購入をテストできます。
  2. テストする前に、iPhoneでiTunesを終了する必要があります。 これは、「設定」->「ストア」->「サインアウト」で行います。
  3. テスト中に、機能の購入を提案するメッセージに[環境：サンドボックス]が表示されます-これはテストモードの兆候です。
  4. テストアカウントでのみテストできます（ステップ5を参照）
