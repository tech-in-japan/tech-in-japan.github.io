データの種類とバックアップ方法



ファイルサーバー



バックアップなしでオンラインでファイルを回復するには、シャドウコピーメカニズム（共有フォルダーのシャドウコピー）を使用すると便利です。 通常、その操作のために、ファイルサーバー自体のディスク領域の5〜20％を予約すれば十分です。 スナップショットを作成するためのスケジュールでは、営業日の終わりと正午を指定できます。 5％の予備では、約14枚の画像を保存できます。実際の数は、ディスクサイズとデータ変更の強度によって異なります。







組み込みのWindowsバックアップツールを使用してバックアップできます。 Cobian BackupとHandy Backupの非常に信頼できるツールもあります。 Cobian Backupは、Unicode、FTP、圧縮、暗号化、増分および差分タイプのバックアップをサポートする無料のアプリケーションです。 Handy Backupには、コピーからの同期やデータ復元など、さらに多くの機能があります。 Windowsバックアップの動作を検討します。



バックアップサーバー上のリモートネットワークフォルダーに保存できるデータのコピーは1つだけであることに注意してください。 そして、次のバックアップジョブがそれを上書きします。 ただし、いずれにしても、データの単一のコピーを保存するのは危険です。



この制限を回避する簡単で効果的な方法があります。 バックアップ用のディスクを、 iSCSIプロトコル経由でバックアップサーバーから接続する必要があります。 Windowsバックアップは、そのようなドライブをローカルと見なします。



最初のバックアップは、保存されるデータの量に等しくなります。 Windowsバックアップはファイルベースのバックアップ方式ではなくブロックを使用するため、次の増分コピーは実際に変更された数のディスクブロックを使用します。



増分バックアップは、変更されたデータのみの記録です。 つまり、データベース全体を毎回コピーする必要はなく、データベースの完全なコピーを一度作成してから、実際に変更するだけです。 ただし、データの以前のバージョンは保存されず、新しいバージョンがその上に書き込まれます。



対照的に、差分バックアップは、以前のバージョンを保持することを意味します。 たとえば、データベースのコピーを毎日作成することにより、その週の以前のコピーをすべて保持します。 これにより、特定の状態にすばやくロールバックできます。 差分コピーでは、変更されたデータはフルコピーとは別に記録されます。



Windowsバックアップは追加の構成を必要とせず、 ストレージを完全に管理します 。



完全および増分バックアップの自動管理。 完全バックアップと増分バックアップを管理する必要がなくなりました。 代わりに、Windows Serverバックアップはデフォルトで、完全バックアップのように動作する増分バックアップを作成します。 単一のバックアップから任意のアイテムを回復できますが、バックアップは増分バックアップに必要なスペースのみを占有します。 さらに、Windows Serverバックアップでは、古いバックアップを定期的に削除して新しいバックアップ用にディスク領域を解放するためのユーザーの介入は必要ありません。古いバックアップは自動的に削除されます。






バックアップ用に実際に保存されたデータの2つのボリュームを割り当てることをお勧めします。 これは、1か月半から2か月の深さで毎日コピーを保存するのに十分です。 頻度-毎日。



Microsoft SQLサーバー



Microsoft SQLサーバーは、3種類のバックアップをサポートしています。



  • 完了します。 データベースは完全にコピーされます。
  • 差動 前回のバックアップ以降に変更されたデータベースページがコピーされます。
  • 増分 。 トランザクションログがコピーされます（完全復旧のデータベースの場合）。


完全バックアップを作成する頻度を決定する必要があります。

ガイドラインの1つは、バックアップの期間です。 営業時間後または週末に行う必要があります。 バックアップ操作により、サーバーに顕著な負荷がかかります。 夜または営業日に完全なコピーを実行することが不可能な場合、そのようなタスクは週末に実行されます。



2番目のガイドラインは、差分コピーの量と差分コピーの期間です。 後続の各差分コピーは、前のものが含まれるため、大きくなります。 最後のフルコピーから時間が経過するほど、インクリメンタルコピーが非線形に長く作成されます。 結局、完全なコピーの場合はデータベースファイルを順番に読み取ることができ、増分の場合は変更されたページをランダムな場所で読み取る必要があります。



増分バックアップの頻度は、障害の結果として失われるデータベースの量によって異なります。 1時間の作業を失う準備ができている場合（つまり、1時間前にデータベースを復元する場合）、1時間に1回増分バックアップを実行する必要があります。 より頻繁に可能ですが、サーバーの負荷を覚えておいてください。 データベースのバックアップは、データの整合性を確保するための1つの方法にすぎないことに注意してください。 データ回復中の単純なように、データ損失が許容できない場合は、AlwaysOnやLog Shippingなどのメカニズムを使用します。



サーバーですぐに実行する必要がある重要な設定は、バックアップの圧縮を有効にすることです。 これにより、バックアップデータの量がほぼ半分に削減されます。 ディスク上のバックアップファイルのバックアップを開始する場合、データベースの実際のサイズから空白ページを引いたボリュームが予約されることに注意してください。







ディスク上のストレージに割り当てられるボリュームの推奨事項は、データベースの少なくとも2つのフルサイズです。 ただし、これは最低限の要件です。会計士は、多くの場合、過去1年ごとにデータベースの完全なコピーと、今年の以前のレポート期間の完全なコピーを保持する必要があります。 少なくとも1か月の深さで毎日コピーを作成する必要がある場合もあります。

仮想マシン



ほとんどのバックアップ製品では、オペレーティングシステム内のエージェントを使用せずに、すべてのディスクで仮想マシンをコピーできます。 Veeam Backup＆Replicationを使用すると、フルバックアップと増分バックアップを実行できるだけでなく、新しいフルコピーを合成し、増分を古いフルコピーに「ローリング」できます。



無料版では、完全なコピーのみを作成できます。これは、バックアップウィンドウと転送されるデータの量に悪影響を及ぼします。 Windows重複排除を有効にすると、ディスクに保存されるバックアップデータの量を減らすことができます。 コピーが仮想マシンから取得されると、* .vibファイルがディスクに保存され、各仮想マシンについても同様に保存されます。 それらは非常に効果的に重複排除されます。 夜にバックアップされ、1日で重複排除されます。 これは実証済みのスキームですが、有料版の製品を使用する必要があります。



Windows Deduplicationが後処理モードで機能することを考慮すると、ストレージに割り当てられるディスク容量の推奨値は、少なくとも3つのフルサイズの仮想マシンです。 コピーの頻度はサーバーによって異なります。 静的コンテンツを含むWebサーバーの場合、週に1回以上コピーすることは意味がありません。

ディスクサブシステム



原則として、バックアップ操作は、データストレージサブシステムに高い要求を行いません。 メインジョブの記録テンプレートは線形であり、ランダムI / Oプロファイルによる高負荷は、バックアップの重複排除中にのみ発生します。







2.5 "SFFディスクと3.5" LFFディスクから選択できます。 SFFディスクを選択する理由はありません。 このタイプのディスクは容量が少なく、高価です。 1つのサーバーからより多くのIOPS（2倍のディスク-2倍のIOPS）を削除する場合に不可欠です。 同じ理由で、提案されているSFFディスクのほとんどは、1万回転のスピンドル速度を持つSASです。



バックアップサーバーの最適な選択は、7200 rpmのスピンドル速度を持つ容量のあるSATA / SASディスクです。 同時に、理論上、SASディスクは、SATAの関連製品よりも少し多くIOPSを提供します。したがって、価格差がそれほど重要でない場合は、それらが優先されます。 ただし、一般的に、ディスク間のバックアップ時間はバックアップサーバーにとって非常に重要です。



インスタントVMリカバリ機能の使用を計画している場合、バックアップストレージのパフォーマンスがワークロードと比較的一貫している必要があることは明らかです。 多くの場合、ブレーキをかける機械は、作動しない機械よりもさらにひどいです。







ストレージボリュームは、上記の推奨事項を考慮して計算できます。 ただし、バックアップはデリケートなものであり、実際のデータが手元にある場合にのみ、成長を正確に計算して予測できます。



バックアップ用のソフトウェア製品を購入した場合、バックアップのサイズは、ディスクにデータを保存する方法と、組み込みの重複排除/圧縮メカニズムの効果に依存します。

Veeam独自の重複排除の結果はかなり控えめであり、Windows Server 2012 R2を使用してこれを行うことをお勧めします。 Microsoft重複排除の使用を計画している場合は、重複排除ボリュームごとに1コアと350 MBのメモリのシステム要件に注意する必要があります。 推奨される最大ボリュームサイズは2 TBです。

ネットワーク



最小ネットワークインターフェイス速度は1ギガビット/秒です。 この要件を満たす機器を見つけることは困難ですが、スイッチに障害が発生する可能性があります。ネットワークポートを選択するときは注意してください。 100mbit / sでは、1 Tbのデータのバックアップは28時間続きますが、これは比較的許容範囲内です。 ただし、営業日中に追加のコピーを作成する必要がある場合、10倍の待ち時間は自分にとってより高価です。



EtherChannelまたは複数のIPアドレスを使用して速度を上げることを試みることができますが、そのような構成は維持するのがより難しく、最終的な速度は常に期待を満たしません。



VMware仮想化と専用のSANネットワークを使用する場合、有料製品はVMFSボリュームから直接データを読み取ることにより、コピー速度を大幅に向上させることができます（SAN転送）。

汎用サーバー



あなたがそれをいじる準備ができているなら、 通常のサーバーは良い選択肢でしょう。 どのオペレーティングシステムを選択するかにかかわらず、WindowsでもLinuxでも、ニーズに合わせて構成を作成する十分な機会があります。 キャッシュを備えた適切なRAIDコントローラーへのデータストレージを信頼できます。WindowsストレージスペースまたはZFSでソフトウェアアレイを構築できます-選択はお客様次第です。 バックアップシステム自体を同じサーバーにインストールできます。



サーバーのフォームファクターを選択する場合、2Uの高さのサーバーに留まることが最適です。 このようなサーバーでは、原則として、12個のLFF（3.5インチ）または24個のSFF（2.5インチ）ドライブをインストールできます。 さらに、サーバーの背面にSFFディスク用の2つのスロットを配置することが一般的になりました。 システムパーティションまたはSSDキャッシュの下で使用できます。



1つまたは2つのプロセッサ？ サーバープロセッサには、1つのチップに4個から非常に素晴らしい22個のコアを含めることができるため、バックアップサーバーには2個のプロセッサは不可欠ではありません。



ただし、場合によっては、2つのプロセッサのコストは、同じコア数のプロセッサよりもわずかに多いか、少ない場合があります。 プロセッサを1つだけ取り付けると、すべてのPCI-Eスロットが機能するとは限りません。



このような制限の例は、インテルのWebサイトに記載されています 。 また、Lenovoは、シングルプロセッサ構成のデュアルプロセッサマザーボードを搭載したx3650サーバーでは、スロットが1つしか得られないと警告しています。
