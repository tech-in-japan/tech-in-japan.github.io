Microsoft StorSimple-自動災害復旧

本日のブログでは、IT専門家のHenrik Motzkusが、Microsoft StorSimpleのコア機能の一連の詳細なレビューを紹介します。 いくつかの個別の側面について説明します。 最も重要なものの1つは災害復旧です。 Microsoft StorSimpleの基本インストールの後、信頼性の要件を満たす設定を選択する必要があります。 Henrikは、ファイルサーバーの例を使用して、このシナリオの技術的な詳細について説明します。



StorSimpleは、ローカルユーザー向けにMicrosoft Azureデータセンターからシンプルで高速、費用対効果の高いデータストレージを提供するように設計されたハイブリッドMicrosoftストレージソリューションです。 基本を理解し、このソフトウェアの操作に関する基本的な知識を得るには、 「Microsoft Azure StorSimple：ハイブリッドクラウドへの簡単なエントリ」の記事を読むことをお勧めします。



1.自動災害復旧のアーキテクチャ



このシナリオのアーキテクチャはどのように見えますか？ ファイルサーバーは、 Windows Server 2012 R2を搭載した仮想マシン（VM）上で実行されます 。 フェールオーバークラスター構成があります。 データを保存するために、サーバーはStorSimpleをiSCSIボリュームとして使用される共有クラスターボリューム（OTC）として使用します。 StorSimpleはAzureのボリューム（CloudSnapshots）を反映し、その後ドメインユーザーがボリュームにアクセスします。 このアーキテクチャは優れた可用性を提供します。 すべてのコンポーネントが複製されます。 ワークロードおよび運用コストは最小限に制限されています。









出典：マイクロソフト



2.自動災害復旧には何が必要ですか？



適切なエンタープライズディザスタリカバリ環境を作成するには、何を知る必要がありますか？ このセクションでは、ABの構成に必要な以下の個々のコンポーネントについて説明します。



  • Microsoft AzureのiSCSIボリューム用のクラウドベースのデバイス
  • Azureまたはローカルネットワーク上のActiveDirectory。
  • Azure SiteRecoveryストレージ
  • Azure Automation Service


以前のブログ投稿で、Microsoft StorSimpleの機能について説明しました。 このコンテキストでは、クラウドスナップショットにも注意する必要があります。 これらは、iSCSIボリュームからBLOBストレージへの暗号化されたコピーと同様に、データセット全体を安全にコピーするために使用されます。 これは、クラウドデバイスがMicrosoft StorSimpleに役立つ場所です。 Microsoft StorSimpleクラウドアプライアンスは、Microsoft Azure VM上で実行される仮想アプライアンスです。 このクラウドデバイスは、ハードウェアデバイスで同じ機能を提供し、iSCSIボリュームをサーバーで使用できるようにします。 ただし、次の条件が不可欠です。クラウドデバイスは64 TBを超えるデータを処理できません。 したがって、ハードウェアデバイスのメモリが64 TBを超える場合、iSCIボリュームを複数のクラウドデバイスに分割する必要があります。 クラウドデバイスはAzureの標準VMであるため、この制限が設定されています。 そして、私たちが知っているように、VMはそれぞれ1023 GBのメモリ容量を持つ64台以下のハードドライブをサポートします。



3.自動ディザスタリカバリのステップバイステップガイド



次に、ユーザーを認証する方法を見つける必要があります。 環境の複雑さに応じて、以下のオプションを以下に示します。



  • ドメインコントローラーは、Azure Site Recoveryを使用して、ドメインコントローラーが1つだけでユーザーがほとんどいない、それほど複雑でない環境のVM全体を表示できます。 事故が発生した場合、リフレクションは単純に再起動します。 その後、Azure構造のすべての変更を会社のデータセンターで繰り返す必要があります。 私はこの方法の支持者ではなく、場合によっては機能しません。



  • Azureに別のデータセンターをセットアップして、多数のデータセンター、ユーザー、および高い血圧変化率を持つ大規模な環境を構成することをお勧めします。 その後、このデータセンターは構造全体の一部となり、レプリケーショントポロジの一部として、Microsoft Azureの前に構造内のすべての変更を自動的に表示します。


4. Microsoft Azure Site Recovery



次に、クラッシュ時にユーザーが（DFS、FailoverClusterなどを介して）共有リソースにアクセスするためのファイルサーバーも使用できるようにする必要があります。 この場合、 Microsoft Azure Site Recoveryを使用して、Azureデータセンター内の対応するVMを最後のビットまで正確なコピーとして複製します。 次の図は、特定のシナリオでのMicrosoft Azure Site Recoveryの基本原則を示しています。









Microsoft Azure Site Recoveryの仕組み（ソース：Microsoft）



これを行うには、 Azure Site Recoveryプロバイダーが必要です。AzureSite Recoveryプロバイダーは、Hyper-Vホストでの操作の調整を担当します。 Microsoft Azureデータセンターを使用するには、ポート443を使用します。また、VMごとに回復サービスエージェントが必要です。 必要に応じて、現在のVMでスクリプトを実行して、事故時に対応する操作を実行します。



5. Microsoft Azure Automation Service



最後に、重要なこととして、別のリソースに切り替えるプロセスを自動的に実行する機能を提供します。 Microsoft Azure Automationは、この目的のためだけのものです。 ユーザーは、Azure Automationサービスアカウントをセットアップするか、プールから事前定義されたRunbookを選択できます。 Runbookを個人アカウントに追加した後、ワンクリックでいくつかの操作を開始できます。



別のリソースへの移行をテストしても、サーバーに影響はありません。 VMはAzureを起動するだけで、ボリュームはVMにマッピングされます。 しかし、その前に、StorSimpleボリュームがコピーされて仮想デバイスに接続されるため、完璧な競争の場が形成されます。



別のリソースへの計画的な移行では、「フレンドリーアプローチ」を使用して、ローカルデータセンターのVMを終了し、その後のAzureへのアップロードを行います。 これを行うには、既存のクラウドスナップショットを仮想デバイスにマッピングします。 出来上がり！



次に、別のリソースへの計画外の移行中に、VMがAzureに読み込まれ、StorSimpleボリュームコンテナーが別のリソースに転送されます。 その後、ボリュームはVMにあり、データへのファイルサーバーアクセスが復元されます。









ソース：マイクロソフト



特に、完成したRunbookにより、プロセス全体が非常に単純になります。 テンプレートスクリプトは既に作成されていますが、個々のシナリオに合わせて変更できます。



6.結論と見通し



ここで説明するハイブリッドソリューションは、提供されるファイルサービスの優れた可用性、簡素化された管理、およびコスト削減の優れた可能性を提供することを約束します。 結局のところ、Microsoft Azureデータセンターが事故の場所として使用されています。 さらに、進行中のハードウェアとソフトウェアの移行に割り当てられた場所は古くなっています。



将来、マイクロソフトはハイブリッドデータセンター向けのソリューションの開発に注力します。 Azure Site Recovery、Azure StorSimple、またはAzure Active Directoryは、オンプレミスインフラストラクチャとAzureを完全に接続するための既製のソリューションを既に提供しています。
