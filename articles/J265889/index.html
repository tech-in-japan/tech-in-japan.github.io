.NET Native-これは、ユニバーサルWindowsプラットフォーム（UWP）の開発者にとって何を意味しますか？

Windows 10では、マネージ言語（C＃、VB）のWindowsベースのユニバーサルアプリケーションは、.NET Nativeを使用してストアのコンパイルプロセスを経てマシンコードになります。 この記事では、それがどのように機能し、それがアプリケーション開発プロセスにどのように影響するかについてさらに学ぶことを勧めます。 以下に、.NET Native開発チームの代表者とのビデオインタビューおよび関連記事の翻訳を示します。









.NET Nativeとは何ですか？



.NET Nativeは、Visual Studio 2015でユニバーサルWindowsアプリケーションを作成するために使用されるプリコンパイルテクノロジです。.NETNativeツールは、マネージコードILライブラリをネイティブライブラリにコンパイルします。 各マネージ（C＃またはVB）ユニバーサルWindowsアプリケーションは、このテクノロジーを使用します。 アプリケーションは、エンドデバイスに到達する前に自動的にネイティブコードにコンパイルされます。 これがどのように機能するかをさらに詳しく知りたい場合は、記事「 .NETマシンコードを使用したアプリケーションのコンパイル 」をお勧めします 。



.NET Nativeは私とアプリケーションにどのような影響を与えますか？



特定のインジケーターは異なる場合がありますが、ほとんどの場合、アプリケーションはより速く起動し、より速く実行され、システムリソースの消費は少なくなります。

  • コールドスタート速度が最大60％向上
  • ホットスタート速度が最大40％向上
  • マシンコードへのコンパイル時のメモリ消費量の削減
  • インストール中にデスクトップ.N​​ETランタイムに依存しない




アプリケーションはネイティブコードにコンパイルされるため、ネイティブコードの実行速度に関連したパフォーマンスの向上が得られます（C ++パフォーマンスに近い）。 ただし、産業用プログラミング言語のC＃またはVBおよび関連ツールを引き続き利用できます。



また、ビジネスロジックを記述するためのさまざまなAPI、および組み込みのメモリ管理と例外処理メカニズムを備えた.NETで利用可能なプログラミングモデルの全機能を引き続き使用できます。

つまり、C ++に近いパフォーマンスを備えたマネージド開発という、両方の長所を最大限に活用できます。 それは素晴らしいことではありませんか？



デバッグとリリースのコンパイル設定の違い



.NET Nativeでのコンパイルは複雑なプロセスであり、通常は.NETでの従来のコンパイルに比べて時間がかかります。 上記の利点は、コンパイル時に価格が設定されます。 アプリケーションを実行するたびにネイティブにコンパイルすることを選択できますが、ビルドが完了するまで待機する時間が長くなります。 Visual Studioツールは、開発経験を可能な限りスムーズにすることで、これをより適切に管理するのに役立ちます。



プロジェクトをビルドしてデバッグモードで実行すると、アプリケーションにパッケージ化されたCoreCLRの上にILコードが使用されます。 .NETシステムアセンブリがアプリケーションコードに追加され、アプリケーションはMicrosoft.NET.CoreRuntime（CoreCLR）パッケージへの依存関係を考慮します。

これは、可能な限り最高の開発体験が得られることを意味します。つまり、高速なコンパイルと展開、広範なデバッグと診断、および.NETで開発するときに使用する他のすべてのツールのパフォーマンスです。



リリースモードに切り替えると、デフォルトで、アプリケーションは.NET Nativeビルドチェーンを使用して起動します。 パッケージはマシンコードにコンパイルされるため、パッケージに.NET Frameworkライブラリが含まれている必要はなくなりました。 さらに、CoreCLRパッケージとは異なり、パッケージは最新バージョンの.NET Native環境に依存するようになりました。 デバイス上の.NET Nativeランタイムは、常にアプリケーションパッケージと互換性があります。



リリース構成のローカルネイティブコンパイルを使用すると、エンドユーザーの環境に近い環境でアプリケーションをテストできます。 開発中は、このモードで定期的にテストすることが重要です。



慣れるのに良い経験則は、開発プロセス中にこの方法でアプリケーションをテストし、.NET Nativeでのコンパイルに起因する問題を見つけて修正することです。 ほとんどの場合、問題は発生しませんが、.NET Nativeではうまく機能しないいくつかのことを知っています。 たとえば、4より大きい次元を持つ配列。 最終的に、ユーザーは.NET Nativeを介してコンパイルされたアプリケーションのバージョンを受け取るため、アプリケーションが配信される前にすべてが事前に機能することを確認しておくと便利です。



ネイティブコンパイルモードでテストするのに適しているだけでなく、AnyCPUアセンブリの構成構成が消えていることにも気付くことができます。 .NET Nativeの登場により、ネイティブコンパイルはアーキテクチャに依存するため、AnyCPUの構成は意味を成しなくなりました。 これの追加の結果は、アプリケーションをパッケージ化するときに、3つのアーキテクチャ構成（x86、x64、およびARM）をすべて選択して、アプリケーションができるだけ多くのデバイスで実行されるようにする必要があることです。 それでも、これはユニバーサルWindowsプラットフォームです！ デフォルトでは、Visual Studioは、以下のスクリーンショットに示すようにまったく同じ方法でビルドするように構成されています。





デフォルトでは、3つのアーキテクチャがすべて選択されています。



AnyCPUライブラリを構築し、UWPアプリケーションで適切なDLLを使用できることに注意することが重要です。 これらのコンポーネントは、プロジェクト設定で指定された対応するアーキテクチャのバイナリライブラリにコンパイルされます。



最後に、.NET Nativeへの切り替えから生じた、よく知られたアプローチの最後の重要な変更は、ストア用のパッケージの作成方法です。 .NET Nativeの主要な機能の1つは、コンパイラがクラウドで動作できることです。 Visual Studioでストアのパッケージを作成すると、ストア用の.appxuploadとローカルインストール用の「test」.appxの2つのパッケージが作成されます。 .appxuploadパッケージには、MSILアセンブリと、アプリケーション（AppxManifest.xmlで指定）で使用される.NET Nativeのバージョンへの明示的な参照が含まれています。 その後、このパッケージはストアに送信され、同じバージョンの.NET Nativeコンパイルチェーンを使用してコンパイルされます。 コンパイラはクラウド内にあるため、アプリケーションをローカルで再コンパイルすることなくバグを修正するために再利用できます。





.appxuploadパッケージがストアに送信されます。 テストフォルダーには、ローカルインストール用のappxパッケージが含まれています

これの2つの結果：最初に、開発者として、アプリケーションのリビジョン番号（4番目の番号）にアクセスできなくなります。 何らかの理由でクラウドでの再コンパイルが必要な場合、ストアはこの番号をアプリケーションパッケージのバージョン管理方法として予約します。 心配する必要はありません。 他の3つの番号を引き続き制御できます 。



2番目に留意する必要があるのは、ストアにアップロードするパッケージについて注意する必要があるということです。 ストアはマシンコードにコンパイルされるため、ローカルの.NETネイティブコンパイラによって作成されたネイティブアセンブリを読み込むことはできません。 Visual Studioは、これを把握して適切なパッケージを選択できるようにします。





「はい」を選択してストアにアップロードします。



ヘルパーを使用してアプリケーションパッケージを作成する場合、ストアにアップロードするパッケージを作成するかどうかをVisual Studioから尋ねられたら、[はい]を選択する必要があります。 また、「アプリバンドルの生成」オプションに「常に」を選択することをお勧めします。これにより、ダウンロードの準備ができた単一の.appxuploadファイルが作成されます。 ストア用のパッケージの作成に関する詳細な説明は、記事「 Windows 10用のユニバーサルWindowsアプリケーションのパッケージ化 」を参照してください。



その結果、作業方法の主な変更点は、.NET Nativeの使用によるものです。

  • リリースモードでアプリを定期的にテストします。
  • パッケージのリビジョン番号を0のままにしてください。VisualStudioでは変更できませんが、他のエディターでは変更できません。
  • ストアのパッケージの作成中に収集された.appxuploadのみをストアにアップロードします.UWPアプリケーションの.appxをダウンロードすると、ストアでエラーが表示されます。




.NET Nativeを使用するための追加のヒント



.NET Nativeが疑われる問題に遭遇した場合、そのような問題のデバッグに役立つテクニックがあります。 デフォルトでは、リリース構成によりコードが最適化され、デバッグで使用されるアーティファクトの一部が失われます。 その結果、リリース構成をデバッグしようとすると複雑になります。 代わりに、.NETネイティブコンパイルを使用できるようにすることで、独自の構成を作成できます。 コードを最適化していないことを確認してください。 これについては、「。 NETネイティブWindowsユニバーサルアプリのデバッグ 」の記事で詳しく説明しています 。



問題をデバッグする方法がわかったので、それを回避する方法を学ぶのはさらに良いことではないでしょうか？ これを行うには、NuGetを使用してMicrosoft.NETNative.Analyzerをアプリケーションに配置します（パッケージ管理コンソールから「Install-Package Microsoft.NETNative.Analyzer」コマンドを使用できます）。 開発中に、コードが.NET Nativeコンパイラと互換性がない場合、アナライザは警告を表示します。 互換性のない.NETスペースの小さなサブセットがありますが、ほとんどのアプリケーションはこのような問題に遭遇することはありません。



.NET Nativeへの切り替えからダウンロード時間の改善を独立して評価したい場合は、自分で測定できます。



既知の問題と解決策



Windows Application Certification Kit（WACK）を使用してアプリケーションをテストする際に留意すべき点がいくつかあります。

  1. コンパイル手順を完了していないUWPアプリケーションでWACKを実行すると、重大な問題が発生します。 次のようになります。

    • uwphost.dllのAPI ExecuteAssemblyは、このアプリケーションタイプではサポートされていません。 App.exeはこのAPIを呼び出します。
    • uwphost.dll APIのDllGetActivationFactoryは、このアプリケーションタイプではサポートされていません。 App.exeには、このAPIに転送するエクスポートがあります。
    • ap-ms-win-core-synch-11-1-0.dllのAPI OpenSemaphoreは、このアプリケーションタイプをサポートしていません。 System.Threading.dllはこのAPIを呼び出します。
    • api-ms-win-core-kernel32-legacy-11-1-0.dllのAPI CreateSemaphoreは、このアプリケーションタイプではサポートされていません。 System.Threading.dllはこのAPIを呼び出します。


    解決策は、パッケージを正しく作成し、適切なパッケージでWACKを実行することを確認することです。 この問題を回避するには 、 ビルドの推奨事項に従ってください。

  2. リフレクションを使用する.NET Nativeのアプリケーションは、Windows App Network Kit（WACK）で失敗して、修正のためにWindows.Networking.Vpnへのリンクが表示されることがあります。 問題を解決するには、rd.xmlファイルに次の行を追加し、パッケージを再構築します。 

    < Namespace Name=”Windows.Networking.Vpn” Dynamic=”Excluded” Serialize=”Excluded” Browse=”Excluded” Activate=”Excluded” />
