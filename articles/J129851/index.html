System Centerとは何ですか？ パート1

System Center製品ファミリの新しいバージョンであるバージョン2012のリリースを見越して、原則としてSystem Centerとは何かを思い出すのに場違いではないと思います。 第1に、このような簡単な概要は、特定の製品、その機能、およびユースケースをさらに詳細に検討するための出発点として役立ちます。 第二に、この情報がmicrosoft.comで表示される方法が好きではありませんでした。 それは、おそらく、ITマネージャーまたはマーケティング担当者の観点から、そこにある情報は正確に表示されるはずですが、何らかの理由で私は常に明確さと特異性に欠けていました。

そのため、System Center（SC）は、ITインフラストラクチャを管理するための専門製品のファミリーです。 インフラストラクチャがMicrosoftプラットフォーム上に構築されている場合、SCから最大の効果が得られます。 ただし、異種環境でさまざまなSCツールを使用することも驚くほど便利です。 次に、SCに含まれる特定の製品と、それぞれが解決する最新バージョンおよびタスクをリストします。 現在、SCに含まれるもの：



System Center Operations Manager 2007 R2-主要なITインフラストラクチャオブジェクトのプロアクティブな監視用。

System Center Configuration Manager 2007 R3-サーバーおよびワークステーションの構成用。

System Center Virtual Machine Manager 2008 R2-仮想マシンと仮想化サーバーの集中管理用。

System Center Data Protection Manager 2010-重要なデータのバックアップとリカバリ用。

System Center Service Manager 2010-インシデント管理を自動化します。

Opalis-組織のITプロセスのこの統合に基づくSC製品と自動化製品の統合。

System Center Essentials 2010-中規模組織のITインフラストラクチャの監視、保護、管理用。



それでは、ファミリーの各製品を特徴づけましょう。 この投稿では、最も有名で人気のある最初の3つから始めます。



System Center Operations Manager 2007 R2（OpsMgr）



OpsMgrは、主要なITインフラストラクチャ施設のプロアクティブな監視用に設計されています。 まず第一に、重要なオブジェクトはWindowsサーバーですが、Linuxサーバー、アクティブなネットワーク機器、または特定の.NETアプリケーションなどにすることもできます。



単一の監視コンソール 。 セットアッププロセス中に、管理者はOpsMgrに監視するオブジェクトを指示し、選択したオブジェクトにOpsMgrエージェントをインストールします（監視はエージェントなしでも可能です）。その後、オブジェクトの状態に関する情報がOpsMgrサーバーに定期的に送信され、いわゆるオペレーターコンソールに蓄積および表示されます。 したがって、オペレータは実際には、1つのウィンドウでインフラストラクチャの状態に関する操作情報を受け取ります。



管理パック 。 OpsMgrは、特定のサーバーの監視方法、分析する必要があるパラメーター、および潜在的な問題を示すこれらのパラメーターの値についてどのように「学習」しますか？ OpsMgrを監視できるようにするには、適切な管理パック（MP）をOpsMgrにダウンロードする必要があります。 たとえば、Windows Server用のOpsMgr MPにダウンロードすると、OpsMgrは、システムログで確認する必要があるイベント、監視するパフォーマンスカウンターなどを「認識」して、すべてがサーバーで正常であることを「理解」します。 Exchange用のMPをダウンロードすると、OpsMgrは、「メール」サービスが正常に機能していると結論付けるためにExchange Serverの設定の範囲を「理解」します。 そのように、OpsMgrは、たとえば、仮想マシン内にインストールされた特定のSQL Serverデータベースのステータスに関する情報を提供できます。 したがって、この製品は、物理インフラストラクチャと仮想インフラストラクチャの両方の監視を提供します。



プロアクティブな監視 。 各MPには、MPが設計されている製品またはサービスのいわゆる正常性モデルの説明が含まれています。 監視プロセス中に「正常な」状態からの逸脱が検出された場合、オペレーターのコンソールにアラートが表示されます。 同時に、アラートが発生した場合に何をする必要があるかに関する情報（知識ベース）がMPに含まれています。 場合によっては、OpsMgrは自動的にアラートに応答し、必要なアクションを実行できます。たとえば、ログが特定のレベルに達するとSQL Serverトランザクションログをクリアするスクリプトを実行できます。 しかし、このアーキテクチャの主な結果は積極的です。 OpsMgrは、この問題が実際に発生し、インフラストラクチャとユーザーの操作に影響を及ぼす前に、差し迫った問題について警告します。



そしてもう1つの重要なポイント。 製品またはサービスのMPは、製品またはサービス自体を開発する人によって開発されます。 MP for Exchangeは、Exchangeチームによって作成されます。 そして、それらのほかに、誰がどのパラメータとどのように監視するかをよく知っていますか？ microsoft.comから無料でダウンロードできるMicrosoftサーバー製品ごとにMPがあります。 しかし、LinuxサーバーやCISCOルーターを監視するためのMPもあります。 確かに、サードパーティのMPは通常無料ではありません。



System Center Configuration Manager 2007 R3（SCCM）



管理者はSCCMを使用して、会社のサーバーとワークステーションの構成に関連するいくつかのタスクを解決できます。 そして最も重要なことは、リモートで決定することです。 SCCMは、コンピューターユーザーの希望に関係なく、ほとんどのアクションを実装するという意味で、管理者にとってはかなりタフで魅力的なツールです。 後者は、もちろん、Active Directoryインフラストラクチャの存在と、対応するワークステーションおよびサーバーの管理権限を前提としています。

管理するコンピューターにSCCMクライアントをインストールする必要があります。 Windows 2000以降、MSオペレーティングシステムのみがサポートされることに注意してください。さらに、SCCMはネットワーク上のコンピューターを自動的に検出するためのいくつかのオプションを提供し、検出後、SCCMクライアントの強制的かつ非表示のインストールが可能です。 SCCMコンソールにクライアントをインストールすると、組織のコンピューターが表示され、さまざまなサブセットでさまざまなタスクを実行できます。



ハードウェアとソフトウェアのインベントリ 。 SCCMは、どのメモリスロットとマザーボード上のどのスロットがインストールされているかを通知しません。 しかし、たとえば、ハードウェアの点でどのコンピューターが会社のスーパービジネスアプリケーションの新しいバージョンをインストールする準備ができているかを理解することは、さらに役立ちます。 また、この新しいバージョンはどのマシンにすでにインストールされています。



ソフトウェアとアップデートをインストールします 。 SCCMを使用すると、目的のコンピューターグループに必要なソフトウェアを自動的かつリモートでインストールできます。 さらに、グループポリシーとは異なり、SCCMはmsi形式のインストールファイルを持つ配布に制限しません。 この機能の特殊なケースは、更新プログラム（パッチおよび修正）のインストールです。



必要な構成 。 Desired Configurationメカニズム（目的の構成）は、クライアントマシンにインストールされているソフトウェアが内部ポリシーに準拠しているかどうかをチェックします。内部ポリシーは、従来、組織の成長とともにますます増えています。 たとえば、オペレーティングシステムの正しいバージョンがインストールされているかどうか、必要なすべてのアプリケーションがそれに応じてインストールおよび構成されているかどうか、禁止されているアプリケーションがあるかどうか。



オペレーティングシステムのインストール 。 この機能により、管理者は、SCCMエージェントが既にインストールされている管理されたコンピューター、またはPXE、ブート可能なDVD / CDディスクなどを使用した管理されていない（または空の）コンピューターに後で自動的にインストールできるオペレーティングシステムのイメージを作成するためのツールを提供します



診断とトラブルシューティング 。 ITスペシャリストがマシンを制御して問題を解決できるようにするキーボードインターセプトなど、リモートコントロール用の特定のツールがあります。

したがって、OpsMgrがすべてを見る目である場合、SCCMは目に見えない管理者の目と手です。



System Center Virtual Machine Manager 2008 R2（VMM）



VMMの主な目標は、複数の仮想マシン（VM）の便利な集中管理を提供することです。 数十の他の仮想マシンが実行されている物理サーバーが2つある場合、フルタイムのHyper-Vコンソールとうまくやり取りすることができます。 数十台のサーバーと数百台、さらには数千台の仮想マシンを備えたデータセンターを扱っている場合、そのような仮想環境を特別なツールなしで管理することはほとんど不可能です。 「集中型VM管理」の概念の背後にあるものを詳しく見てみましょう。



複数の物理サーバーを管理します 。 VMMでは、1つのコンソールから複数の物理サーバーを管理し、異なる仮想化テクノロジ（仮想サーバー、Hyper-V、ESX）をサポートできます。 同時に、これらのサーバー上のすべてのVMですべての基本操作を実行できます：作成、削除、構成、開始、一時停止、シャットダウン。 さらに、次のVMを作成するとき、VMMは一連の基準全体に対して特定のVMに最適な物理サーバーを提供します。 VMMはクラスター構造を区別します-クラスター上にVMを作成すると、必要なパラメーターをすべて備えたクラスターノードの1つに正しく配置されます。



ライブラリ 。 VMMを使用すると、指定したプロパティでVHDファイルとVMテンプレートを保存できるライブラリを整理できるため、マウスを数回クリックするだけで、目的のOS、ネットワーク設定、その他のパラメーターを使用してVMを作成できます。



VMの移行 。 VMMは、P2V（物理から仮想）およびV2V（仮想から仮想）移行をサポートします。 最初のケースでは、VMMは移行用に指定されたサーバーを分析し、このサーバーに対応するパラメーターを使用してVMを作成します。 2番目の方法では、VMMはVMファイルをVirtual ServerまたはESX形式からHyper-V形式に変換します。

バージョン2008 R2以降、VMMはLive Migration（LM）テクノロジーをサポートしています。 管理者は、VMMコンソールからVMを1つのクラスターノードから別のクラスターノードに直接移行できます。 さらに、前述のOpsMgrとの統合を構成できます。 この場合、VMM自体がVMの移行を決定できます。たとえば、クラスターノードの1つで現在のCPU負荷が大きく、他のノードが比較的空いている場合です。



セルフサービスポータル 。 このポータルにより、ユーザーは管理者権限なしでVMを作成、実行、使用できます。 ユーザーが（もちろん、管理者によって許可されたものから）Webページで必要な特性を選択するだけで十分です。その後、VMM（ライブラリのテンプレートに基づいて）が指定されたパラメーターでVMを自動的に作成して起動します。 言い換えれば、VMはサービスの形で消費者に提供されます。 拡張バージョンのセルフサービスポータル2.0は、追加の無料パッケージとしてダウンロードできます。



運営 VMMは、レポートの作成、任意のVMへのリモート接続、およびVMの現在の状態の評価のためのツールを管理者に提供します。 また、VMMのもう1つの非常に重要なプロパティは、管理者がコンソールで実行するすべての操作が最初にPowerShellスクリプトに変換され、次にスクリプト自体が実際に起動されることです。 管理者は、将来このようなスクリプトを保存、変更、使用して特定のタスクを自動化できます。
