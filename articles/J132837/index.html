Speakeasyを使用したAtlassian ConfluenceとJIRAの拡張

なんで？









JIRA 、 Confluence、または他のアトラシアン製品の「フィッティング」の問題に興味を持ったことのある人なら誰でも、これらの目的のためのアトラシアンSDKがあることを知っています。









そして、このSDKを少なくとも1回掘り下げた人は、特に真の道を導く古い同志がいない場合、これらすべてに取り組むのにどれだけの時間と小さな灰色のボックスを費やすべきかを想像します。











重要なプロジェクトの作業の最盛期で、壊滅的な時間不足で、顧客の1人がConfluenceの小さなものをねじ込むか、JIRAのリンクを非表示にするように要求するとき、特に失望します。 そして、クライアントは、彼らが気分を害する習慣を持っているものを待つように求められなければなりません。 そして、クライアントはしばしば残念ながら何も助けられない専任のプログラマーを持っているという事実によって問題は悪化します。 アトラシアン製品のプラグインの開発に関する知識がなく、数日で解決しません。









どうやら、アトラシアンの人たちもこれを理解しているため、 Speakeasyテクノロジーをユーザーと管理者に提示しました。 技術自体はもはや新しいものではありません-初期のベータ版のプレゼンテーションは、プラグイン開発者向けの競争であるCodegeist 2011と一致するようにタイミングが取られましたが、実稼働システムにインストールすることを恐れて、多かれ少なかれ安定したバージョンが出てきているようです。









Speakeasyは、Atlassian SDKとは異なり、ユーザーに表示されるもののみをカスタマイズできます。JavaScript、マークアップ、またはスタイルを追加してページを変更します。 ただし、これはSDKを使用するよりもはるかに簡単です。 FirefoxアドオンのGreasemonkeyでも同様のことが行われますが、ここでは1つのブラウザーに限定されることはなく、ユーザーが他のブラウザーに具体的にインストールすることを強制しません。









誰がこれを必要としますか？



SpeakeasyをすべてのユーザーがJIRAとConfluenceをカスタマイズできるようにするマステクノロジーと呼ぶのは魅力的ですが、明らかにこれは完全に真実ではありません。 ほとんどの場合（常にではありませんが）、基本的なJavaScriptプログラミングスキルとHTML + CSSレイアウトが必要です。 ただし、このためにAtlassian SDKのドックを読む必要はありません！ そして、これは一分間、英語で何百もの興味深く興味をそそるページです。









したがって、Speakeasyから利益を得ることができる人々の間で、私はランク付けします。







  • 開発者、さらにはWeb開発者。 これらの人々は、追加のトレーニングなしでほとんどすぐに、Speakeasyを使用して拡張機能を作成できます。
  • 管理者 管理者は、多くの場合、プログラミングの方法も知っており、お気に入りのユーザー向けに何らかの拡張を行うことができます。 また、Speakeasyを使用する場合、ユーザー自身が拡張機能を使用するかどうかを決定すると考えると、相互に排他的な願いの間を操作するための大きなスペースが得られます。
  • 開発者の「はい、そうしますが、SDKを研究するのに2週間かかり、さらに1か月間実験を​​行います」というフレーズの後、マネージャーは悲しむことはなくなり、明確な（またはほぼきれいな）良心をもって「学ぶことは何もありません！ すぐにそれを行ってください、親愛なる！」




そしてすぐに戦いに！



シンプルで便利な拡張機能をゼロから作成しましょう。 すべての例はConfluence 3.5用です。 最新のバージョン4.0はまだ人々にあまり提供されていません。 ただし、次のほとんどすべてが4番目のバージョンに適しています。 したがって、JIRAの拡張機能の開発に基本的な違いはありません。









例0、これはまったく例ではありません-Speakeasyのインストール



まず、サーバーにSpeakeasyをインストールし、必要な権限を配布する管理者が必要です。









Speakeasyの最新バージョンではなく、 プラグイン交換および対応するmavenリポジトリで配布されます 。









プラグインのインストール後、どのユーザーグループが拡張機能を作成する権利を持ち、どのユーザーグループがこれらの拡張機能を自分自身に含めることができるかを決定する必要があります（管理-プラグイン-既存の管理-Speakeasyプラグイン-構成）。









デフォルトでは、管理者がSpeakeasyに拡張機能を含めることは許可されていないことに注意してください。 これは明らかにセキュリティ上の理由で行われます。 ただし、テストシステムでは、「管理者に拡張機能の有効化を許可する」設定をオンにすることができます。









例1-履歴



この例では、ページの履歴へのリンクを目立つ場所に追加して、ドロップダウンメニューで毎回そのページの背後をクロールしないようにします。









Speakeasy拡張機能のあるページに移動します。











[インストール]リンクをクリックし、表示されるダイアログでウィザードの使用を選択し、ウィザード自体で、拡張機能の一意の識別子、名前、および説明を入力します。











その後、そのような「パラメーター」と標準のテストコンテンツを含む拡張機能が作成され、インストールされます。 「詳細-編集」ボタンを使用すると、シンプルで便利なWebエディターにアクセスでき、拡張機能の数行をプログラムできます。













JavaScriptのメインロジックを含める必要があるため、main.jsファイルに興味があります。 









/** * @context page */ var $ = require('speakeasy/jquery').jQuery; $(document).ready(function() { var href = $('a#action-view-history-link').attr('href'); $('a#view-change-link').after(", <a href='"+ href +"'>view history</a>"); });




このような行を書くことは、JSとjQueryを知っている人にとっては難しいことではありませんが、JavaScriptを使っていない人はgoogleを使って同様のコードを書くことができます。 それとは別に、コメントには「@context page」というエントリのみを記載します。 このスクリプトを実行するコンテキスト（読み取り、ページタイプ）を示します。 Wikiページにのみ興味があり、このページでコンテキストのリストを見つけることができます 。

例2-情報提供



標準UIで何かを変更することは素晴らしいことであり、変更されたUIに新しい情報を入力できるとさらに良いです。









詳細については、JavaScript / JQueryのすべての機能（REST、SOAP、XML-RPC）を使用できます。 幸いなことに、ConfluenceとJIRAにはそのようなインターフェイスがあります 。









最もシンプルなRESTインターフェイスを使用してみましょう。このインターフェイスでは、Confluenceからさまざまなエンティティに関する情報のみを取得できます。 ページを作成した日付を要求して、最後の編集の日付の横にある標準のConfluenceインターフェイスにページを有機的に配置します。









この場合のコードは次のようになります。 







 /** * @context page */ var $ = require('speakeasy/jquery').jQuery; $(document).ready(function() { var pageId = $('input#pageId').attr('value'); var serviceUrl = '/rest/prototype/1/content/' + pageId + '.json'; $.getJSON( serviceUrl, function(data) { $('li.page-metadata-modification-info a:first').after(" on " + data.createdDate.friendly); }); });




変更されたUIは次のとおりです。











なぜなら ページの読み込み時にすべてのスクリプトが機能するため、ユーザーは「日付がなく突然表示された」などの「ちらつき」に気付きません。









例3-時間厳守



Speakeasy拡張機能には、アトラシアン製品固有の機能も含まれています。 その中にはWebItemでの作業があります。 簡単に言えば、ページのメニュー項目を標準項目の隣に追加する機会があります。









これを行うには、拡張機能内にweb-items.jsonファイルがあり、必要なメニュー項目を記述しています。 例に直行しましょう： 









 [ { "section" : "system.content.action/primary", "label" : "Revert", "url" : "", "cssName" : "stiltsoft-web-item", "weight" : 25 } ]




これにより、[ツール]メニューに[元に戻す]という新しいアイテムが作成されます。 URLパラメーターは必要なアドレスに設定できますが、少し複雑でURLを設定せず、JavaScriptでこのアイテムにロジックを追加しました（以下を参照）。









アイテム自体は前のページのリビジョンにすばやくロールバックするために必要であり、それに応じてJavaScriptコードはこのリビジョンの数を計算します。 









 /** * @context atl.general */ var $ = require('speakeasy/jquery').jQuery; $(document).ready(function() { var pageId = $('input#pageId').attr('value'); var revision = $("meta[name='page-version']").attr('content'); $('a.revert-web-item').attr('href','/pages/revertpagebacktoversion.action?pageId='+pageId+'&version='+(revision-1)); });

例4-インタラクティブ



SpeakEasy開発者は、恋人たちが美しいインターフェースを作成することを忘れませんでした。 Speakeasy自体には、openOnePanelDialogメソッドを使用して、Confluence / JIRA自体のスタイルで美しいダイアログを作成する機能が含まれています。 これはコードでどのように見えるかです： 









 /** * The main module * * @context atl.general */ var $ = require('speakeasy/jquery').jQuery; var dialog = require('speakeasy/dialog'); $(document).ready(function() { $('span#title-text').after("<span class='stiltsoft-source'><a href='#' id='stiltsoft-view-source'>View source</a></span>"); $('a#stiltsoft-view-source').click(function(){ var url=$('a#action-view-source-link').attr('href'); $.get(url,function(data){ dialog.openOnePanelDialog( { 'header':'Page source', 'content':$(data).find('div#content'), 'submitLabel':'OK', 'cancelClass':'stiltsoft-cancel', 'submit':function(dialog,callback){ callback.success(); } }); } ); return false; }); });




このコードは、ページタイトルに[ソースの表示]リンクを追加します。実際には、ページのソースコードを含む美しいダイアログが表示されます。 CSSを使用した拡張機能には、追加のダイアログボタンを非表示にする魔法がまだありましたが、たとえば、これは最も重要ではありません。









このような何か、それは画面上に見えます：











1つのダイアログだけでは不十分な場合は、 AUIライブラリ（Atlassian User Interface）を使用する必要があります。 これは、Atlassian製品のインターフェースを組み立てる基本的な要素（ダイアログ、ポップアップ、ドロップダウンメニューなど）をすべて含む、完全にクライアント側のJavaScriptライブラリです。 したがって、AUIを使用する拡張機能の外観は、他のインターフェイスと変わりません。









この技術にすぐに「突入」するには、 AUI Demo 、 AUI Sandboxにアクセスして、そこの要素を少し試してみることをお勧めします。
