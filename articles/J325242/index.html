SQL Server仮想化



なぜサーバー仮想化なのか？



サーバー仮想化が検討されている7つの主な理由は次のとおりです。



  1. ハードウェア使用率の増加 。 統計によると、ほとんどのサーバーは日常のタスクを実行するときに15〜20％の負荷がかかります。 1台の物理サーバーで複数の仮想サーバーを使用すると、ハードウェアの購入を大幅に節約しながら80％に増加します。



  2. ハードウェア交換コストの削減。 仮想サーバーは特定の機器から解放されているため、物理サーバー群を更新するためにソフトウェアを再インストールおよび構成する必要はありません。 仮想マシンは別のサーバーに簡単にコピーできます。



  3. 仮想サーバーを使用する柔軟性を高めます。 負荷が変化する状況で複数のサーバーを使用する必要がある場合（製品環境でのテストや作業など）、仮想サーバーは最適なソリューションです。物理サーバーの負荷が増大している場合、他のプラットフォームにシームレスに転送できるためです。



  4. サーバーインフラストラクチャの管理性の強化。 仮想サーバーを集中管理し、負荷分散とライブマイグレーションを提供できる多くの仮想インフラストラクチャ管理製品があります。



  5. 高可用性を提供します。 仮想マシンのバックアップの準備と復元は、はるかに短い時間で済み、より簡単な手順です。 また、機器に障害が発生した場合、仮想サーバーのバックアップを別の物理サーバーですぐに起動できます。



  6. スタッフの節約 。 長期的に見た仮想サーバー管理の簡素化は、会社のインフラストラクチャを提供する専門家の節約を伴います。 仮想サーバーを管理するためのツールを使用する2人が4つの機能を実行できる場合、少なくとも年間15,000ドルを受け取る2人の特別な専門家が必要なのはなぜですか？ それでも、仮想化の分野で有資格者を訓練するには、かなりの訓練も必要であることを念頭に置く必要があります。



  7. 電気を節約します。 中小企業にとっては、この要因はもちろん重要ではありませんが、大規模なサーバー群を維持するためのコストに電気（電力、冷却システム）のコストが含まれる大規模なデータセンターでは、この点は非常に重要です。 1つの物理サーバーに複数の仮想サーバーを集中させると、これらのコストが削減されます。

SQL Server仮想化のデメリット



SQL Server for SharePointが物理サーバーに展開される主な理由となる主な欠点をリストします。



  1. システムの信頼性を操作するアプローチを再構築する必要性。 実際、複数の仮想マシンが同じ物理サーバー上で実行されているため、ホストに障害が発生すると、すべてのVMとそれらで実行されているアプリケーションに同時に障害が発生します。



  2. 負荷分散。 仮想マシンの場合、SQLサーバーは通常プロセッサ（またはメモリ）の多くのコンピューティングリソースを使用します。これは、プロセッサ時間（メモリ）も必要とする他の仮想マシンおよびホストアプリケーションの動作に影響します。 SQLサーバーを備えた仮想マシンのみがホスト上にある場合でも、最適なレベルのパフォーマンスを調整するには、複雑なアクションが必要です。 詳細については、Michael Ootyの記事（こちら）をご覧ください。 管理者は、実行中の仮想マシンが負荷の少ないサーバーに自動的に移動するルールまたは負荷のかかったサーバーを「アンロード」するルールを確立して、負荷を分散する必要があります。

ディスクサブシステムの準備



SharePointは、独自のデータストレージ機能を備えたSQLサーバーを使用してデータを保存します。 この点で、これらの機能を考慮して、物理レベルと論理レベルの両方でディスクサブシステムを準備すると、企業ポータルの最終的なパフォーマンスに大きな影響があります。



Microsoft SQL Serverが保存するデータは8 KBページに分割され、ページはいわゆる64 KBエクステントにグループ化されます。 詳細については、 こちらをご覧ください。 これに応じて、ディスクサブシステムの構成は、ディスクサブシステムのすべての物理レベルおよび論理レベルでのエクステントの一貫した配置を確保することです。



最初に開始することは、RAIDの初期化です。 同時に、データベースが配置されるアレイのストライプサイズは64 KBの倍数にする必要があります（できれば正確に64 KBである場合）。



ハードドライブまたはSSDのパーティションの位置合わせ



SQL Serverのパフォーマンスを改善する問題を考慮する場合、ディスクアレイ上の物理的な配置に関してファイルシステムを調整する手順は非常に重要です。 ここで詳細に説明されます 。 ハードディスクまたはSSDでは、512バイトの最初の63セクターは、オペレーティングシステムのインストール中またはディスク管理操作中に作成されたファイルを配置するため、および特別なプログラム（ハードディスクのパーティションマネージャー）を使用するために予約されています。 整列手順の本質は、整数個のファイルシステムクラスターが1つのストリップアレイディスクアレイに収まるように、開始パーティションオフセットを選択することです。 そうしないと、ファイルシステムからデータの1つのクラスターを読み取る操作を実行する状況が発生する可能性があり、ディスクアレイから物理読み取りの2つの操作を実行する必要があり、ディスクサブシステムのパフォーマンスが大幅に低下します（パフォーマンスの損失は最大30％になる可能性があります）。 ソリッドステートドライブの場合、パフォーマンスの損失はわずかに低くなりますが、読み取り/書き込み操作の頻度が増えるため、寿命がすぐに尽きます。

ディスクパーティションを管理するためのさまざまなソフトウェアを使用して、イメージからオペレーティングシステムをセットアップする場合、パーティションアライメントの違反に関連する状況が発生する可能性があります。 この点で、追加の保険としてSQLサーバーをインストールする前に、次のセクションで説明する相関関係を再確認することをお勧めします。

ストライプユニットサイズ



Windowsには、ディスクに書き込むための最小データブロックのサイズ（ストライプユニットサイズ）を決定するための標準ツールはありません。 このパラメーターの値は、ハードドライブに関するベンダーのドキュメントまたはSAN管理者から確認する必要があります。 ほとんどの場合、ストライプユニットサイズインジケーターは、64 Kb、128 Kb、256 Kb、512 Kb、または1024 Kbです。 前に説明した例では、64 Kbの値が使用されました。

磁気ディスク（単一またはRAIDの一部）の場合、シーケンシャル操作（Seq Q32テスト）は、他のテストの結果を10x-100x回上回ることがよくあります。 これらのメトリックは、多くの場合、リポジトリへの接続方法に依存します。 ベンダーが主張するMb / sの数は理論上の制限であることを覚えておく必要があります。 実際には、それらは通常、主張されているよりも5〜20％少ないです。

頻繁に使用されるコンテンツデータベースの推奨事項を推進する



連携して動作するように、または大量のアップグレード操作を実行するように設計されたサイトでは、SharePointコンテンツデータベースファイルを保存するディスクの選択に特別な注意を払う必要があります。 次の推奨事項を考慮する必要があります。



  • コンテンツデータベースのデータファイルとトランザクションログを異なる物理ディスクに配置する
  • ポータルソリューションの設計では、コンテンツサイズの増加を考慮してください。 最も最適なのは、各コンテンツベースのサイズを最大200 GBまでサポートすることです。
  • 頻繁に使用されるSharePointコンテンツデータベースの場合は、複数のデータファイルを使用して、それらを別々のディスクに配置することをお勧めします。 データベースの突然のシステムクラッシュを避けるために、データベースサイズの制限設定を使用しないことを強くお勧めします。


コンテンツデータベースの予想サイズを計算する



コンテンツデータベースの予想サイズを計算する一般的な式は次のとおりです。



（（D * V）* S）+（10Kb *（L +（V * D）））、ここで：



  • D-個人のサイト、ライブラリ内のドキュメントおよびページを考慮したドキュメントの数、来年の増加の予測。
  • V-カウント—ドキュメントのバージョン。
  • S-文書の平均サイズ（見つけられる場合）;
  • 10 Kb-定数、SharePoint 2013で予想される1つのドキュメントのメタデータの数。
  • Lは、来年の数の増加の予測を考慮した、リスト項目の予想数です。
