ISTQBitかISTQBitでないか？ Test Managerの高度なレベルの認定の準備

ISTQB（International Software Testing Qualifications Board）は、テスト専門家向けの国際的に認められた認定プログラムです。 ISTQB証明書を取得するには、認定プロバイダーが主催する試験に合格する必要があります。



現在、認定には3つのレベルがあります。基本（基礎）、上級（上級）、および専門家（専門家）であり、前のレベルの対応する証明書の可用性は、次のレベルの認定に合格するための条件の1つです。 上級レベルはいくつかのモジュールに分かれています。そのうちの1つについては、この記事-Test Managerで説明します。

トピックに深く入り込まないように、利用可能な資格のスキームの詳細については、公式ウェブサイトwww.istqb.orgをご覧ください 。



ISTQBを準備するための認定およびサードパーティのトレーニングセンターは数多くありますが、自分で試験に合格することは非常に現実的であり、この記事の目的はこの企業での経験を共有することです。 また、最後に私たちの会社の例にロシアの現実のこの認証の関連性についていくつかの言葉。



試験はどうですか



検査手順は基本レベルとほとんど変わりません-これは、1つ以上の回答オプションを選択できるテストです。



国際的なプロジェクトに取り組んでいるとき、英語版のプロの語彙は私にとって最も身近で親しみやすいので、言語の選択はありませんでしたが、ロシア語でも取ることができます。



試験は225分間続きますが、そのうち45分は、母国語以外の言語で試験に合格する場合はオプションです。 Foundation Level Examとは異なり、約50ページのテキストが質問としてあなたを待っており、その読書は割り当てられた時間の大部分を占めるため、ボーナス時間が役立ちます。



正常に完了するには、65％のポイントを獲得するだけで十分です（正解数と混同しないでください）。 質問のさまざまなカテゴリには独自の重みがあります。たとえば、一般に、状況に関する質問は、用語や定義に関する直接的な質問よりも多くのポイントを与えます。



実際、準備について



ISTQB Rex Black- Advanced Software Testing-volume IIの元大統領の本から受け取った包括的なトレーニング資料。 そこでは、各章の終わりに、 公式ウェブサイトwww.istqb.org （試験文書セクション）に記載されている質問に加えて、質問の例が記載されています。



試験の対象は次のとおりです。



各ブロック内で注意を払うことをお勧めするものをさらに検討します。



テストプロセスとテスト管理



これら2つの密接に関連するポイントは、すべてのポイントの約半分を占めているため、それらに注目するのが理にかなっています。

通常、テストの質問は、テストにおけるリスクベースのアプローチに従います。 このアプローチの概念は、欠陥の確率と特定の機能の重要度に基づいてQAリソースを割り当てることです。 このコンテキストでのリスクとは、品質リスクを意味し、実装されるとソフトウェアの品質が低下し、テストプロセスはリスクを軽減する直接的な方法として定義されます。



完全に合理的な側面と関連する利点に加えて、このアプローチを完全に遵守することは、プロセスの形式が比較的高いことを意味します。 たとえば、識別とリスク評価には、技術関係者およびビジネス関係者の代表者との特別セッションが含まれます。 柔軟な開発（優先度と要件の頻繁な変更）および/または限られたリソースの状況では、このイベントの有効性は平凡な場合があります。 ただし、このアプローチを理解することで、テスト問題を解決する際に正しい方向に思考を向けることができます。



レビュー



QAの用語では、静的テストの形式としてレビューを意味するのが慣習ですが、このセクションでは管理レビューと監査について話します。場合によっては、テストマネージャーが関与します。

そのようなレビューの主題は、リリースの準備のステータスと進捗状況、製品のリスクとその評価、製品の要件（管理者が実装のために実装することが期待される場合）です。

このトピックに関する質問は、目標、参加者、結果を評価するためのメトリックなど、さまざまなタイプのレビューの構成に関連しています。



欠陥管理



私に出くわした質問に答えるために、バグトラッカーと常識の十分な経験がありました。

質問は、欠陥のライフサイクル、その属性、およびテストと開発プロセスを評価および改善するための統計の使用に関するものです。 たとえば、導入と検出のフェーズ、根本原因、影響を受けるコンポーネント、または機能（欠陥クラスターを識別するため）などの情報が役立つ場合があります。



テストプロセスの改善



このセクションでは、TMMi（Testing Maturity Model integration）、TPI Next（Test Process Improvement Next）、CTP（Critical Testing Processes）、STEP（Systematic Test and Evaluation Process）、IDEAL model（開始-診断-確立-行為-学習）およびその他の略語、その解読と原則。 テストプロセスを改善するのは、すべて業界標準またはフレームワークです。



テストツールと自動化



主な関心事は、自動テスト用のツールの選択と実装、および抽象的な例を使用した他のタイプの補助ユーティリティの基本原則です（特定の製品への参照なし）。 たとえば、ツール自体のコスト、その使用コスト（従業員のトレーニングを含む）、起こりうるリスク（最も一般的なものとして、高い期待と自動テストの開発とサポートに必要なリソースの過小評価など）が考慮されます。



人のスキル



人的要因は成功する生産的なチームの主要な側面の1つですが、ISTQB Test Managerモジュールの資料でそのようなチームを構築し、やる気を起こさせるトピックは非常に表面的にカバーされています。

この事実は試験の準備を簡素化しますが、個人的な発達のために、他の情報源（あなた自身の好みからの例-ブライアントレーシーのリーダーシップと動機付けのチーム管理）を参照することは理にかなっています。

このトピックに関するテストの質問には、テストチームの長所と短所を必要なスキルの観点からかなり単純なレベルで評価し、従業員のトレーニングまたはチームの拡大によって特定されたギャップを埋める計画を​​立てることが含まれます。



なんでこんなこと？



上級試験の受験を考えている場合、ほとんどの場合、すでにCTFL証明書を取得しているだけでなく、その利点（またはその欠如）についての一般的な意見もあります。 したがって、詳細に立ち入らずに、トピックに関するいくつかの考えを要約します。



この認定の実際的なメリットは、テスト管理に関する優れた本を読むことと比較できます。 試験の資料では、一般にQAシニア/リードレベルの専門家の責任に匹敵する著者のリスクベースのアプローチの概念における、SDLCのさまざまな段階での品質管理プロセスの計画、テスト監視、報告、最適化のトピックを詳細にカバーしています。 さらに、上記で推奨されている本の著者は、ソフトウェアテストと品質保証の分野でコンサルタントとしての経験を頻繁に使用しており、外国の同僚の経験からの例を使用して、論争のある問題の代替ソリューションを知る機会を提供しています。 もちろん、試験自体を受験する必要はありません。



私の観察によると、適切な証明書の入手を希望することは、ロシアよりも欧州企業の欠員によく見られます。 しかし、私たちの市場では、このアーティファクトの非常に具体的なアプリケーションがあります。 たとえば、Luxoftはカスタムソフトウェア開発をリードしており、多くの場合、企業の顧客から資格検証要件が出されます。 この場合、試験は雇用主によって支払われ、合格するかどうかの疑いは自分で決定します。 従業員証明書は、顧客へのビジネス提案を策定するため、およびRFPと入札に参加するために使用されます。 さらに、従業員が専門的な成長のために特定の目標を設定している場合、証明書を取得することは彼の給与に影響を及ぼす可能性があります。

会社の関心の文脈では、従業員またはコミュニティが共同で認証の準備をするための内部トレーニングコースを持つことが適切ですが、まだこれはありません。



ISTQB認証に関するかなり一般的な懐疑論については、私はそれを部分的に共有します。 特に、認証のための認証に関しては。 私の意見では、比較的低い合格スコアは、最小限の準備で試験を実行可能にします。 そして、証明書の入手可能性が専門的スキルに影響を与えるかどうかは、個人的な優先順位のみに依存します。
