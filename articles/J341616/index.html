アフリカのBitrix24実験





「ああ、アフリカでは川はとても広い！」

アフリカでは、山はとても背が高い！」



これらの線は、私たちを遠い子供時代に連れて行き、同じ名前のおとぎ話から赤ずきんの冒険を思い出させます。 しかし、今日はこの作品の芸術的な部分についてではなく、 南アフリカ共和国の 1つの大企業にBitrix24を導入した方法について説明します。



顧客のニーズを特定することから始めました-私たちの前には、CRMシステム統合サービスを提供する3つの会社がありましたが、私たちが理解したように、1つの会社がサーバーに箱入りバージョンをインストールしたばかりで、2つはそこで何かをしましたが、終わりませんでした、そして3番目の会社-顧客が何を望んでいるかを理解していませんでした。



顧客は、人口の売掛金の支援の分野で働いている大規模なコンサルティング会社です。 60人がコールセンターで働いており、チーム全体で約150人です。 比較的小規模ではありますが、同社は数十万人の南アフリカ人が経済的自立を支援しています。



基本的に、同社は「 コールド 」コールに取り組んでおり、そのために多くの異なるソフトウェアを使用していました。 順番に始めましょう。 リードのリストはBitrix24にアップロードされ、コールは別のサービスから送られたため、マネージャーは同じリストをあちこちに何度も送信する必要がありました。 その結果、コールセンターのマネージャーは、すべてのシステムの情報を1つのリードのみに追加および変更するために平均3分間費やしました。つまり、 マネージャーの効率は低かったのです。



Bitrix24がどのように配置されているか、つまりサーバー側で確認した後、最初にバックアップを削除してから再インストールすることにしました 。 タイムゾーンが異なるため、このアクションを夕方遅くに実行しました。 バックアップを削除して、作業を開始しました。



Bitrix24はプロバイダーのサーバーにインストールされていたため、メモリ量やその他のさまざまな特性の変更を増やすために、顧客は常に彼に連絡する必要がありました。 これらの問題を解決するために、Bitrix24を社内のサーバーにインストールし、開発者が推奨するように、 CentOS 7プラットフォームにインストールしました。



彼らは展開し、見、チェックし、テストし、出来上がりました-誰も置換に気づきませんでした！



システムのパフォーマンスは良好なレベルであったため、緊急の問題に進みました。 これは大規模なコールセンターであるため、電話の問題を解決する必要がありました。 従業員の効率の向上は、あらゆる組織のすべてのリーダーに関係する重要な側面です。 Bitrix24の電話については、FreePBXコントロールパネルでアスタリスクを使用することにしました。



必要なアスタリスクのバージョンを間違えないようにするため、ソースコードからインストールすることにしました。インストール時に市場で最新のものだったため、13番目のバージョンを選択しました。 つまり、アスタリスクを構築するために必要な依存関係を最初にインストールしました。



Centos 7にインストールし、ここに



このソフトウェアのインストール手順：



1-システムアップデートをインストールします。



システムが正しく機能するには、rootとしてログインしてシステムを更新する必要があります。 これは、次のコマンドで実行されます。 



[root@centos-7 ~]# yum update


2-次に、必要なパッケージをインストールする必要があります。



システムを更新したら、次のコマンドを使用して必要なパッケージと依存関係をインストールする必要があります。 



 [root@centos-7 ~]# yum install gcc gcc-c++ php-xml php php-mysql php-pear php-mbstring mariadb-devel mariadb-server mariadb sqlite-devel lynx bison gmime-devel psmisc tftp-server httpd make ncurses-devel libtermcap-devel sendmail sendmail-cf caching-nameserver sox newt-devel libxml2-devel libtiff-devel audiofile-devel gtk2-devel uuid-devel libtool libuuid-devel subversion kernel-devel kernel-devel-$(uname -r) git subversion kernel-devel php-process crontabs cronie cronie-anacron wget vim






パッケージを確認する必要があります。これらが必要な場合は、[Y]をクリックしてアクションを続行します。 以前にインストールしたものにもよりますが、ダウンロードには平均して約125〜130 MBかかります。 すべてが正しく行われた場合、インストールされたパッケージ、更新、および依存関係のリストが表示されます。



3-MariaDBをカスタマイズする



MariaDBはここで必ず必要です。これがないと、データベースの操作が何度も複雑になります。



インストールするには、次のコマンドを実行する必要があります。 



 sudo yum install mariadb-server


次に、デーモンを実行します 



 sudo systemctl start mariadb


カスタマイズ： 



 [root@centos-7 ~]# systemctl enable mariadb [root@centos-7 ~]# systemctl start mariadb [root@centos-7 ~]# systemctl status mariadb


その後、次のものが必要です。



4-libjanssonのインストール



ダウンロード： 



 [root@centos-7 ~]# wget http://www.digip.org/jansson/releases/jansson-2.7.tar.gz


開梱： 



 [root@centos-7 ~]# tar -zxvf jansson-2.7.tar.gz


構成： 



 [root@centos-7 ~]# cd jansson-2.7 [root@centos-7 ~]# ./configure --prefix=/usr






そして、カスタマイズされたパッケージをコンパイルします。 



 [root@centos-7 jansson-2.7]# make clean [root@centos-7 jansson-2.7]# make & make install [root@centos-7 jansson-2.7]# ldconfig


次に、アスタリスクをインストールし、FreePBXコントロールパネルを構成しました。



つまり、ソースコードからインストールしました。 



 [root@centos-7 ~]# wget http://downloads.asterisk.org/pub/telephony/asterisk/asterisk-13.5.0.tar.gz






アーカイブを解凍した後、適切なフォルダに移動して、アセンブリする前にアーカイブを構成する必要があります。 



 [root@centos-7 ~]# tar -zxvf asterisk-13.5.0.tar.gz [root@centos-7 ~]# cd asterisk-13.5.0 [root@centos-7 asterisk-13.5.0]# ./configure --libdir=/usr/lib64






この操作が正しく実行されると、ASCIIアートのスタイルでアスタリスクのロゴが表示されます。





さらに、非常に重要なポイント：アスタリスク自体が機能し、非常に良好に機能しますが、管理の面では不便なので、いくつかのモジュールを接続しました。



1-設定メニュー



それに入るには、次のコマンドを実行する必要があります。 



 [root@centos-7 asterisk-13.5.0]# make menuselect


原則として、いくつかのモジュールはすでに接続されていますが、個々のケースに適したモジュールから選択できます。



次に、お客様のIVRを構成する必要がありました。 顧客はこの問題を非常に真剣に受け止めたと言わざるを得ません。彼は各メニュー項目のテキストを自分で登録し、 レコーディングスタジオを借りて、高品質のトラックを作ってほしいと頼みました。



以下に、.mp3サポートモジュールを接続する方法を示します。 これを行うには、「 format_mp3 」を選択します





次に、 Core Sound Packagesセクションに移動し、必要なオーディオパッケージをマークします。



私たちは音声制御をうまくセットアップし、その結果から、私たちはその甘いことに喜びました。



その後、FreePBXをインストールしました。



これを行うには、 Apacheをわずかに変更する必要があります 



 sed -i 's/\(^upload_max_filesize = \).*/\120M/' /etc/php.ini sed -i 's/^\(User\|Group\).*/\1 asterisk/' /etc/httpd/conf/httpd.conf sed -i 's/AllowOverride None/AllowOverride All/' /etc/httpd/conf/httpd.conf systemctl restart httpd.service


ダウンロードして実行する 



 cd /usr/src wget http://mirror.freepbx.org/modules/packages/freepbx/freepbx-13.0-latest.tgz tar xfz freepbx-13.0-latest.tgz rm -f freepbx-13.0-latest.tgz cd freepbx ./start_asterisk start ./install -n


これで完了です！



ただし、経験豊富なシステム管理者にとって、これは難しい仕事ではありません。



理論的には、ほぼすべてを完了しましたが、今では主な問題の1つであるオペレーター接続が残っています。 2つのテレフォニープロバイダーが提供され、両方を接続する必要がありました。 彼らは、隠された広告を作らないように名前を表明しないことに決めました。 SIPを介してテレフォニーオペレーターを接続し、AsteriskとBitrix24の相互作用を構成しました。



2つのSIPトランクを接続することはそれほど難しい作業ではないように見えますが、問題が発生する場合があります。その方法を説明します。 WebインターフェイスのFreePBXの[ トランク]セクションに接続すると、提供されたデータを持つオペレーターが追加されます。 通常、これはログイン、パスワード、サーバーアドレスです。 このデータを追加すると、 PBXで2、3、または10個のトランクを取得できます。



ところで、情報の迅速な交換と良好なコミュニケーションのために、Bitrix24とAsteriskを同じサーバーにインストールすることをお勧めします。いずれにしても、より信頼性が高く、高速です。



そして、私たちと同様に、SIPを接続することに決めた場合、Bitrix24でのテレフォニーの操作のために、SIPコネクタのライセンスを購入する必要があることを忘れないでください。これは、毎月の支払いです。

問題があります-解決策があります。 はい、多くの人がBitrix24が常にHTTPSで動作すると言うことができますが 、そうではありません-ボックス版ではSSL証明書をインストールする必要があります。 再確認して、すべてうまくいきました！

時間が経ち、プロジェクトはうまく機能し、何も悪いことはありませんでした。 アフリカの暑い日差しとサファリパークでの象の叫び声に誰もが（少なくとも顧客は）喜んでいました。 しかし、嵐の前は穏やかだった。

彼らは理由を探し始めました。 彼らはBitrix24で、その後アスタリスクで、そしてその後サーバー自体で罪を​​犯しました-彼らは完全なハードウェア診断を実行しました。 しかし、何も助けませんでした。 ご存知のように、会社自体のシステム管理者のようなものが大好きです。彼らの知識や経験の質に影を落としたくはありませんが、システム内の何かを変更したい場合は、それを設定するすべての人が」そしてそれに取り組みました。 プロバイダーの変更については通知されませんでした。 さらに、事故の1日後に失敗が発生しました。当然、パラメーターの競合が発生しました。 原因とその説明の検索には、従業員にとってかなりの労働時間がかかりました。 さらに、最近まで顧客はZoya Kosmodemyanskayaであり、問題の真の原因について話をしたくありませんでした。

そして、Bitrix24を使用する別のボーナスがあります。 Bitrix24がビジネスを行うための優れたツールセットであることは誰もが知っています。 当初、ユーザーはCRMモジュール、タスク、プロジェクト、およびグループのみを使用していました。 しかし、しばらくしてから、ユーザーが増え、ワークスペースからBitrix24は社内のソーシャルネットワークになりました！



最近、Bitrix24の速度に関するリクエストが頻繁になりました。 標準構成のDell PowerEdge T20 RAIDアレイを使用しました。 パフォーマンスを確認しました-原則として、データ処理速度はこのソフトウェアの最小要件の一部ですが、これらは単なる数値です。 実際、システム全体の速度は低下しています。 私たちは解決策を探し始めました。



彼らがBitrix24をどのように使い始めたのかを見てみると、率直に言って少し驚いていました-Buzzを見ると、 Facebookフィードになっていることがわかります-面白い写真、面白い話、投稿などがたくさんあります。 一般的なチャットと個人的なチャットの両方が、 Telegram 、 WhatsApp、またはFacebook Messengerに似始めました。まったく異なるトピックに関する多くのメッセージです。 Bitrix24内の通話にも同じことが当てはまります。人々はお互いに電話をかけ始めました。私は普通の携帯電話や固定電話のサービスに頼りません。



もちろん、このすべてがボックス版のBitrix24-Enterprise製品が使用されているため、データ処理の速度に反映されています。 サイトの正常性および分析データを表示すると、SQLクエリの数が70％増加し、 SQLデータの書き込み速度は同じままであることがわかりました。 7200 rpm、16 GBのRAM、 Xeon 3.0でそれぞれ1 TBのHDDを2つ使用しました。 なぜなら 良いプロセッサが使用されていますが、私たちはそれに触れませんでした。 Bitrix24には8GBのRAM、およびアスタリスク-2が必要です。したがって、原則として、RAMを増やす必要はありませんでした。 さらに、1つのパラメーターだけがラメでした-データベースの書き込み速度。
