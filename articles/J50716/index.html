$ mv darkk-redsocks-77a490422b701875a9fda2bd7d58e62cb7481f64 darkk-redsocks-2009013101

ディレクトリの名前を変更する最初のアクション：バージョンは、日付に現在の日付のバージョン番号を加えた番号になります。今回はバージョンが01になります。2番目は「コーシャ」ソースアーカイブを作成します。 ちなみに、ルールカタログには「name-version」という名前を付け、アーカイブには「name_version.orig.tar.gz」という名前を付ける必要があります。 前者の場合、バージョンの前にハイフンが必要で、後者の場合はアンダースコアです。

制御ファイルを引き続き使用する必要があるため、すぐに少し説明します。 このファイルには、パッケージのすべての説明が格納されます-アーキテクチャが構築されている対象、パッケージに付随している人など。原則として、すべてが明確であり、説明項目のみが対象です。 それはcな方法で配置されています。最初の行では、「Description：」という語の直後に短い説明が書かれています。 「 Packagename is a description 」行で置換できるように、ルールに従って記録します。 つまり、サブジェクトと "is"（可能性のある記事）のない小さな文字で書かれており、末尾に句読点が含まれていません。 この場合、パッケージの名前は短い説明に記載しないでください。また、1つの文に収まり、60文字を超えないようにする必要があります。 次の行から始まるのは、パッケージの完全な説明です。 説明の各行は、再び80文字（これは、すべての種類のスクリプトを含む一般的な要件です）以内で、スペースで始まる必要があります。 説明に空の行を挿入する場合は、その行にスペースとピリオド文字を入れます。 さて、良いDebianの伝統によれば、ファイルは空行で終わるべきです。

最大かつ最も頻繁な石は認可された清潔さです。 友よ、あなたはこれに何人の人が出くわしたか想像できません。 プログラムを取り出してパッケージに入れると、debian-legalから次のように伝えられます。「男、src / face / ob / table.cppファイルが別のプログラムから取られたのを知っていますが、著作権で指定していませんか？ ？ また、src / super / karambol /ディレクトリのすべてのコンテンツは、一般にフリーではないライセンスの下で配布されます。 ああああ。」 しかし、エラーを修正し、なんらかの方法でプログラムの作者に非フリーコードではなく無料のアナログを書くように説得し、パッケージをリポジトリに含めたとします。 しかし、1週間後、細心の注意を払ったユーザーが次のことを発見しました。 ここで、メッセンジャーでは、クライアントアイコンがここに表示されます。 そして、各ロゴは常にそのプログラムのライセンスの下で配布されます。 したがって、ICQ 6.5独自のクライアントアイコンをアーカイブにアーカイブする権利はありません。 もちろん、これらの言葉で、debian-legalは眉をひそめ、よく狙ったキックでリポジトリから既に受け入れられたパッケージnafigを投げます。 そのため、ところで（私が伝えると約束しました）、qutIMパッケージはubuntuに受け入れられ、すべてのライセンス問題が解決するまでDebianで対処しようとさえしません。 プロプライエタリなアイコンだけでも、コミュニティ全体をスリッパで駆り立ててくれます。

別の非常に重要な問題は、奇妙なことに、プログラムの作者です。 アップストリームで修正、追加、または何か他のものを受け入れるように簡単に説得する著者のために祈ってください。 そのような著者は希少だからです。 たとえば、Pidgin srcパッケージをご覧ください。 debianコミュニティからのパッチの大規模なバンドルが含まれています。 上流では、ピジンはそれらを摂取しません。 そして、あなたがGnomeに何か良いものを持ち込もうとするなら、彼らはTorvaldsでさえそれを困難にしたと言う。

それでも、時々、間違ったディレクトリの配置や、スニーカーのプログラムへのショートカットのアイコンの不在または間違ったサイズ/形式が表示される場合があります-リンティアンはこれを誓うのが大好きで、まずそれを聞く必要があります。 しかし、これらはすべてかなり修正しやすいバグですが、最初の2つの問題は本当に深刻で永続的です。
