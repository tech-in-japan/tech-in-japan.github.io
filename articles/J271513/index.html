N個のキューをレーキするために、N個のプロセスを作成できます。 各プロセスで、目的のキューに接続し、まったく同じサイクルを実行する必要があります。 これは実用的なアプローチですが、キューが多数ある場合は、Tarantool Queueへの接続が多くなります。 サーバーの物理メモリを消費する多くのプロセスも起動されます。 「多くの接続」では、Tarantoolでの作業が可能な限り効果的になりません。 また、プロセスでは、GoogleおよびAppleサーバーへの接続を維持する必要があります。 繰り返しますが、保持しているGoogleまたはAppleサーバーへの接続が少ないほど、ロードする負荷が少なくなるため、サーバーのリソースをより多く使用できます。



「PythonのTarantoolを使用した非同期操作」の記事で、Tarantoolへの単一の接続を使用するとパフォーマンスが顕著に向上する理由を詳しく説明しました（これはワークロードにとって非常に重要です）。 このアプローチはここで適用できます。 ソースの擬似コードを少し変更して、キューをかき集めます。 それをasyncioに適合させます。 



 import asyncio import aiotarantool_queue @asyncio.coroutine def worker(tube): while True: task = yield from tube.take(.5) if not task: break # process(task.data) yield from task.ack() loop = asyncio.get_event_loop() queue = aiotarantool_queue.Queue("127.0.0.1", 3301, loop=loop) workers = [asyncio.async(worker(tube), loop=loop) for tube in (queue.tube('q1'), queue.tube('q2'), queue.tube('q3'))] loop.run_until_complete(asyncio.wait(workers)) loop.run_until_complete(queue.close()) loop.close()

しかし、RedisとRabbitMQはどうでしょうか？



RedisやRabbitMQではなくTarantool Queueを使用する理由は何ですか？ 特定の製品に有利な選択はそれほど単純ではありません-RedisとRabbitMQの両方を検討しました。 Redisにはプロトタイプもありました。 これらのソリューションはすべて、十分なパフォーマンスを備えています。 しかし、ここでの問題は「誰が速いか」だけではありません...



まず、キューをメモリ内ではなく信頼できるものにしたいのです。 WALを備えたTarantoolは、RedisやRabbitMQよりも信頼性が高く見えます。



各キューシステムには独自の特性があります。 Redisにはpub / subメカニズムがあり、問題の解決には適していません-キューが必要です。 Redisにはリストとrpush / blpop操作があり、データが表示されるのをロックして待機しますが、take / ackメカニズムはありません。 私たちの生産では、この特定のメカニズムによって信頼性が提供されます-それが最高であることが繰り返し証明されています。



RabbitMQは、キューのさまざまなパターンが豊富です。 問題を解決するには、RabbitMQ機能の一​​部のみが必要です。 そのパフォーマンスは非常に高いですが、ディスクへのデータの保存をオンにすると、負荷時のパフォーマンスが大幅に低下します。 RabbitMQを実行するには、RabbitMQインスタンスを再起動するだけでなく、運用上の問題を診断できる経験豊富なシステム管理者が必要です。



RabbitMQの特別な注意は、Python APIとasyncioのコネクタに値します。 キューAPIはコールバックに実装されます。 コールバックからのコードは複雑になり、保守が難しくなります。 asyncioでmessage.ackを作成するには、 Futureを作成して待つ必要があります。 このようなコードは非常に複雑に見えます。 また、1つの接続に複数のput / takeを送信することもできませんでした。



asyncioを備えたRedisの方がはるかに優れています。asyncioの作成者自身による素晴らしいコネクタがあります。 非常に高速に動作します。



RedisおよびRabbitMQでは、Tarantoolのようなデータベースとluaにそのようなデータ統合はありません。 原則として、生産タスクには、ボックスからもう少し「ロジック」が必要です。 Tarantoolでは、これはluaのおかげで簡単に実現できます。 たとえば、カウンターまたはデータキャッシュの保存を開始したり、キューに入れられたインスタンスの統計を直接保存したりできます。 これにより、Tarantoolはさまざまな問題を解決するのに便利です。
