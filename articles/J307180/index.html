アンドレイ・パンギン



-JavaとC ++、今最も人気のある言語は何だと思いますか？ 両方とも大人ですが、誰がより成熟して洗練されていますか？



-まず第一に、私はこれらの言語間に競争があるとは思わない。 それぞれに独自のニッチがあり、完全に共存します。 従来、Javaの人気はわずかに高くなっています。 Javaプラットフォームは、アプリケーションをデバッグおよび保守するための強力なツールに惹かれます。 ただし、C ++の重要性を過大評価することは困難です。 素晴らしい歴史を持つ言語であるにもかかわらず、現在も積極的に開発を続けています。C++ 14標準が多くの新しい興味深い機能を備えたC ++ 11に慣れたのは開発者だけでした。



-負荷の高いサーバーの世界でどの言語を使用できますか？ 異なる言語で個別のシステムモジュールを開発し、特定のタスク用にそれらを強化することは理にかなっていますか？ 可能であれば（希望しますか、それとも機会がありますか）、C ++を使用して問題を解決しますか、それとも1つの言語のみですべてを行いますか？



-負荷の高いサーバーでは、誰もが異なることを理解しています。 1秒間に何千ものネットワークリクエストが発生するものもあれば、大量のデータを並列処理するものもあります。 さまざまなツールがさまざまなタスクに適しています。 Odnoklassnikiには、特に画像およびビデオ処理に関連するC ++で記述されたモジュールがあります。SIMD計算とプロセッサの最も効率的な使用が必要です。 ただし、ほとんどのシステムではJavaで十分です。 さらに、以前にCで開発され、JNIを介して呼び出されたコードフラグメントは、Javaで徐々に書き直されました。その結果、不要なコピーとJNIのオーバーヘッドをなくしたため、パフォーマンスも向上しました。



-JavaでUnsafeを使用することは正当化されますか？ なぜC ++を使用しないのですか？



-なぜUnsafeを使用するかについての全体的なレポートがあります。 特にオフヒープでの作業やネイティブコードとのやり取りなど、Unsafeなしではできない多くのシナリオがあります。

アプリケーション全体をC ++で記述したい場合、統計の収集、監視、サーバー間の通信などのために、すべての一般的なフレームワークとプロトコルを再実装する必要があります。 、これは低レベルの操作を担当しますが、残りの開発は通常のJavaで行い、シンプルで理解可能なコードを記述するための最良のパターンを厳守します。 エコシステム全体が単一のプラットフォームで開発されている場合は、はるかに便利です。



-エンタープライズシステムと可能なソリューションを開発する際の最も一般的なパフォーマンスの問題は何ですか？



-まれに、Javaプラットフォーム自体のパフォーマンスに依存する場合。 通常、問題を解決するには、アルゴリズムを置き換えるか、スケーリング、つまり鉄を積み上げます。 最も一般的なボトルネックは、ネットワーク帯域幅またはディスクI / Oです。 ただし、JVMについて話すと、ガーベッジコレクターがパフォーマンスの主要な問題、または唯一のパフォーマンス問題をもたらす場合があります。 したがって、ヒープを過度に大きくしないようにします。最大50ギガバイトですが、さらに多くの場合、アプリケーションごとに4〜8ギガバイトです。 私たちは、ヒップを超えて大容量を移動しようとしています。 大規模で負荷の大きいキャッシュを作成するためのフレームワークを作成しました。 ヒップと比較したこのキャッシュの追加の利点は永続性、つまりデータを失うことなくアプリケーションを再起動できることです。 これは共有メモリを使用することで実現されます。起動直後に、アプリケーションは共有メモリオブジェクトをプロセスのアドレス空間にマップし、すべてのデータを含むキャッシュがすぐに利用可能になります。

-非常に大まかに見ると、C ++とJavaの違いはランタイム層にあります。ランタイム層は、とりわけ、あらゆる種類の最適化を実行します。 どちらが望ましいか：マシンのアーキテクチャー機能を手動で使用する（C ++）、または動的なJVM最適化に依存する方が良いですか？ 特定のことについて話す場合、自動的にゴミを収集するか、手動で制御する方が良いでしょうか？



-アダプティブコンパイルと自動メモリ管理は、Javaの強みです。 この点で、仮想マシンは優れており、静的コンパイラーを上回りました。 しかし、主なことはそれでさえありません。 提供するセキュリティ保証のためにJVMを選択します。 まず第一に-メモリの不正な操作による致命的なエラーからの保護。 ポインターに関連する問題や、アンマネージコードで配列の境界を超える問題を探すのは、桁違いに困難です。 そして、このようなエラーを修正するコストは、メモリへの直接アクセスを可能にする小さな速度の利点をカバーしています。 上記のように、時々Unsafeを使用します。これらの場合、C ++と同じリスクに自動的にさらされます。 はい、JVMクラッシュダンプを理解する必要がある場合がありますが、これは楽しいアクティビティではありません。 それが、私たちがまだ純粋なJavaを好む理由であり、アンマネージコードは緊急の場合にのみ使用します。

成熟度に関しては、ここではすべてが複雑であるため、最初に言語の機能、コンパイラの機能、および標準ライブラリの機能を分解する必要があります。



最初から始めましょう-言語を使って。 そこにも問題もあります。 Javaの問題は、言語が最も近い競合他社ほど速く開発されないことです。その結果、機能が非常に遅くなり、希望どおりに機能しません。 C＃は若いが、ラムダが最初に登場したことは注目に値します有能で成功しました。



C ++に関しては、ここでの主な問題は、C ++とC言語との100％の互換性です。これは、誰も必要としない言語機能の膨大な荷物を自動的に意味します。 一方、2000年代のC ++の停滞は、言語に人気を追加しませんでした。 開発者は常に新しい機能を提供する必要があります。 これで状況は改善されました-C ++では、ラムダ（ちなみに、C＃/ Javaよりも表現力があります）、変数の型推論、および関数から返される値さえもあります。一般に、言語は何らかの形で進化します。



これは言語に適用されます。 コンパイラーについて。 ここでは、まず、比較が完全に正しいわけではありません。 JVMはJITです。つまり、バイトコードを取得して、現在のプロセッサの理想的な表現に変換し、すべての最適化を適用できるという考え方です。 これは理論上は良いように思えますが、Javaでどのように機能するかはわかりませんが、.NETの世界では、このアプローチは、C ++コンパイラの最適化と比較して、もちろん実際には何もしません。 あなたが数学をしている場合、または私はこれを言います：たとえば、あなたがネットワーク上で数学的な.NETライブラリを購入する場合、これはC ++の単なるラッパーになります。



はい。C++コンパイラについては、インテルC ++を計算タスクに使用しています。つまり、プロセッサの製造元から提供されているコンパイラです。 非常に多くの不利な点があります。MSVCに比べて言語機能が少なく、サポートに連絡しなければならない厄介な間違いがたくさんありますが、このサボテンは単純な理由で最適化されています。 インテルのコンパイラは、最も効率的なコードを生成します。 もちろん、それは単一のコードではありません。ここでは、Intel Parallel Studioのすべての能力を備えています。これは、並列化のためのスレッドビルディングブロック（ちなみに、Microsoft Parallel Patterns Libraryの類似物）、および直接使用しなくてもIntel Math Kernel Libraryです。 MATLABなどを介して間接的に使用します。 ここでは、MKLのようなライブラリがIntelのスタッフによって既に最適化されていることを明確にする必要があります：ここでは、ベクトル化、並列化、さらにはMPIによるクラスター並列化（FFTの場合など）がすぐに使用できます-つまり、それを使用して使用します。 もちろん、プロファイリングの手段についても言及する価値があります。これもIPSの一部です。 これは非常に強力なツールキットであり、開発者がパフォーマンスと正確さの点でコードを最適化するのを支援するという目標を本質的に設定します-メモリプロファイルもあるため、リークなどすべてを簡単に見つけることができます。



最後に、「if」について-すべてがシンプルで、Javaが勝ち、C ++ですべてが悪い。 C ++標準ライブラリのインターフェース自体が少し狂っているとは言いませんが、問題はすべてが「レガシー」であるだけでなく、機能がほんのわずかしかないことです。 とにかく、ファイルシステムのサポートや何らかのストリームサポートのようなものを得ました。 そして、ここに文字列があります。スペースで部分文字列に打ちたいです-これは標準ライブラリにはありません、つまり、サードパーティのライブラリを取得する必要があります（まあ、良いことはBoostのようなものがあり、多くの有用なものがあります）。 しかし、開発は本当に遅くなります。 Electronic Artsなどの多くの企業は、標準に満足していないため、STLの実装を作成しています。 副次的に、多くの人は基本的に新しいゼロからのライブラリ、特定のSTL2が必要であることを認めていますが、もちろん、それをStandard Library 2またはそのようなものと呼ぶ方が正しいでしょう。



たとえば、メインパッケージマネージャーが存在しないなど、まだ多くの問題があります。それがあったとしても、ライブラリはどのように機能しますか。 Javaまたは.NETでは、単純にバイナリを配布できますが、C ++では基本的に並べ替える必要があります。 誰もまだこの問題を実際に決定しておらず、これも開発を遅くします。 他の人のライブラリを使用して、それがあなたのために機能するようにするためだけに30分を費やすこともあります。
