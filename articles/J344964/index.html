ESIAとの統合に関するFAQ





2018年の初めから始まり、私たちの生活の最も多様な分野（インスタントメッセンジャー、遠隔医療に関する法律など）を含む法律の変更は、私たちの生活への情報サービスの浸透の増加という1つのことによって結ばれています。 実生活のように、人がサービスを受けるには、身元を明らかにする必要があるのは自然な事実です。 オフライン生活では、承認の手段は市民のパスポートであり、オンラインの分野では、政府はESIAを識別と認証の単一の手段として認識することを決定しました。



彼女についてお話したいと思います。 これは入門記事であり、教育プログラムと言えます。 必要に応じてまだ知らない人に会うために、プロジェクトでESIAを使用し、州の最新情報を入手できます。 それで、それはどのような動物であり、政府はどのようにそれを考慮していますか。



ロシアの電子政府インフラストラクチャの枠組み内での通信省は、統一識別認証システム（FSIS ESIA）を作成し、開発しています。その目的は、ユーザーの登録、識別、認証、および承認のプロセスを合理化および集中化することです。



FSIS ESIA：



1.情報システムに、ユーザー（個人および法人、公的機関）を確実に識別するためのソリューションを提供します。



信頼性は、次の事実により達成されます。



  • ESIAへの個人の登録は、識別に重要な基準の検証に関連付けられています。
  • ESIAは、ロシア連邦の法律に従って、ESIAに含まれる情報の保護を提供します。


2.ユーザー指向であり、機能を提供します。



  • 公的機関のさまざまな情報システムにアクセスするときに、単一のアカウントとサポートされている広範な認証方法を使用した識別と認証。
  • ESIAに投稿された個人データの管理、および公的機関の情報システムへの提供の管理。


ESIAの主な機能：



  • 以下を含むユーザーの識別と認証：

    • ワンタイム認証。ESIAユーザーに次の利点があります。ESIAで識別および認証手順を通過した後、ユーザーは再識別および認証なしで、1セッション内でESIAを使用して情報システムにアクセスできます。
    • さまざまな認証方法のサポート：パスワード、電子署名、および2要素認証（永久パスワードとSMSメッセージの形式で送信されるワンタイムパスワード）;
    • ユーザー識別の認証レベルのサポート（簡易アカウント、標準アカウント、検証済みアカウント）。
  • 識別データの維持、すなわち、個人、法人、団体および組織、団体および組織の役人および情報システムの登録簿の維持。
  • 次のESIA機能にアクセスできる公的機関の許可された人の許可：

    • ESIAの政府職員の登録簿を維持する。
    • 情報システムに関する権限のディレクトリを維持し、ESIAに登録されたシステムリソースにアクセスする権限をESIAユーザー（ESIAのESIA担当者として登録）に付与します。
    • 上記の権限を下級国家権力者に委任する。
  • ESIAに登録されている情報システムに関連するユーザーの力に関する情報の維持と提供。


現時点では、政府機関や特定の種類の営利組織がシステムをESIAに接続できます：保険会社、信用機関（銀行）、証券市場の専門的参加者、非政府年金基金、マイクロファイナンスおよびマイクロクレジット組織、ならびに通信事業者。



法律は時間の経過とともに調整され、ESIAへの参加が許可されている組織のリストが拡大しています。



状況に詳しくない人は、「状態」という言葉で、関連するすべてのコスト、ライセンス、および機器を備えた国内の暗号化アルゴリズムを使用して保護する必要がある通信チャネルをすぐに想像します。 しかし、それがいかに馬鹿げていたとしても（または悲しいことに）、この国の身元確認の主なプラットフォームは、外国の暗号化（およびどこへ行くか）に取り組んでいます。



したがって、このプラットフォームのサービスを使用する場合、 Cloud4Yインフラストラクチャを含む広大な国の適切な場所にリソースを配置できます。



営利団体はESIAから何を得ることができますか？



利用可能な情報のリストは、次の項目に依存します。



  1. ESIAに接続する組織のカテゴリ
  2. ESIAへの接続に使用される方法


通信省は、営利団体が利用できるデータのリストを制限しています。 通常、彼らはフルネーム、パスポートの詳細（シリーズと番号、発行者と発行者）、市民権、アカウントの「確認」のサインとESIAのアカウント識別子に関する情報のみを受け取ることができます。



政府機関は、ESIAからユーザーとその組織に関する完全なデータセットを受け取ることができます。 これは次の情報です。



  1. 個人データ（名前、性別、生年月日、出生地、市民権）
  2. 身分証明書（SNILS、TIN、パスポートと市民権、出生証明書、運転免許証、軍事ID、強制医療保険）
  3. 連絡先情報（メール、携帯電話、自宅の電話番号、登録住所、住所）
  4. 子どもに関する情報（個人データと文書）
  5. 車両に関する情報（登録番号と証明書）
  6. 組織および個々の起業家に関する情報（名前、PSRN、TIN / KPP、法的形式、法的住所、連絡先、支店、従業員のリスト、従業員の資格情報、組織の車両）
  7. アカウントデータ（ESIAのアカウント識別子、アカウントの「確認」のサイン）


情報は、ESIAでユーザーが入力する範囲で提供され、この情報を提供するユーザーの同意が必要です。



接続方法



組織のサイトを接続するには、いくつかの非常に簡単な手順を実行する必要があります。







一般に、ESIAに接続するには以下が必要です。



  1. 組織がシステムをESIAに接続できることを確認してください。
  2. ESIAプロファイル Webアプリケーションを使用して、組織のディレクターがESIAに組織を登録します。









  3. また、責任ある従業員を組織のアカウントに添付し、特別なアプリケーションであるESIAテクノロジーポータルへのアクセス権を割り当てる必要があります。 ディレクターが従業員にそれ以上の操作を委任する予定がない場合でも、ESIAテクノロジーポータルへのアクセスを明示的に許可する必要があります。



    ESIA Technology Portal Webアプリケーションを使用して、組織の担当責任者は次のことを行う必要があります。
  4. システムアカウントをESIAに登録します。 システムのニーモニックを作成するか、ESIAに接続されたシステムがすでにSMEVに接続されている場合は、SMEVへの接続ポイントの既存のニーモニックを使用します。
  5. 証明書をシステムカードにアップロードします。



    組織の責任者には次のものが必要です。
  6. テスト環境および産業環境でのESIAプログラムインターフェイスの使用に関する規制に従って、電子メールアプリケーションで交互に送信します。







    プラグ可能なシステム開発者：
  7. 現在の文書「 ESIAの使用に関する方法論的推奨事項 」に従ってESIAと対話するためのコードを独自に開発したか、既製のソリューションを使用して、ESIAに接続するためのシステムを完成させると、市場にそのような利点があります。
  8. ESIAテストおよび産業環境での相互作用をデバッグします。






ここで、2018年1月1日以降、SAML 2.0プロトコルでの対話は許可されなくなります （既存のシステムのみ）。 ESIAに接続するには、OAuth 2.0 / OpenID Connectプロトコルを使用する必要があります（両方のオプションが利用可能になりました）。



システム認証



基本形式のOpenID Connect 1.0を介した統合の推奨ユーザー認証シナリオは、次のシナリオに従って発生します。



  1. ユーザーは、クライアントシステムのWebページで[ESIA経由でログイン]ボタンをクリックします。
  2. クライアントシステムは、認証リクエストを生成してESIAに送信し、ユーザーのブラウザをアクセスを提供する特別なページにリダイレクトします。
  3. ESIAは、利用可能な方法のいずれかを使用してユーザーを認証します。 ユーザーがまだESIAに登録されていない場合は、登録プロセスに進むことができます。
  4. ユーザーが認証されると、ESIAは、クライアントシステムが識別と認証のためにユーザーに関するデータを要求することをユーザーに通知し、クライアントシステムが要求する情報のリストを提供します。
  5. ユーザーがクライアントシステムによる認証の許可を与えると、ESIAはクライアントシステムに特別な認証コードを発行します。
  6. クライアントシステムは、識別トークンのリクエストをESIAに生成します。これには、リクエストの早い段階で受信した認証コードが含まれます。
  7. ESIAは、要求の正当性（たとえば、クライアントシステムがESIAに登録されていること）および認証コードを確認し、識別トークンをクライアントシステムに渡します。
  8. クライアントシステムは、IDトークンからユーザーIDを取得します。 識別子が受信され、トークンが検証されると、クライアントシステムはユーザーが認証されたと見なします。 識別トークンを受信した後、クライアントシステムは、対応するアクセストークンを受信した後、ESIA RESTサービスを使用してユーザーに関する追加データを取得します。






接続するかどうか



オペレーターにとって、メッセンジャーに関する法律の施行に関連して、この問題は実質的に解決されました。



2017年7月29日付の連邦法第245「連邦法の改正」「通信に関する」に従って、通信事業者は加入者に関する情報の正確性を検証する必要があります。 法律は検証方法のリストを定めており、その1つは、オペレーターがCMEAを介して接続している場合、州および地方自治体のサービスの統一ポータルまたは政府機関の情報システムの使用です。



「通信に関する連邦法」の改正は、2018年6月1日に施行されます。それまで、通信事業者は、SMEVおよびESIAを使用してシステムの動作をテストできます。
