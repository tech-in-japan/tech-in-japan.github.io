Twitter APIの新バージョンの変更はすべての人に影響します

Twitter APIの新しいバージョン （v1.1）のリリースが公開されました。それに応じて、6か月以内にアプリケーションを提供する必要があります。 最も重要な恐怖は確認されていません-10万の障壁は、公式クライアント（Tweetbot、Echofonなど）の主要機能をコピーするアプリケーションのみに関係します。 しかし、私たちの恐ろしい空想の残りは実現しました。



彼らの短いレビュー さらに。



リクエスト制限の引き上げ（レート制限）



以前は、1つのトークン（ユーザー）からのリクエスト数の制限は1時間あたり350リクエストでしたが、今ではすべてがさらに面白くなっています。

まず、時間間隔は60分ではなく15分ごとに分割され、ほとんどの方法では15分ごとに15リクエストのみが許可されます（1分間に1回）。 例外（180リクエスト/ 15分）は以下の方法です：

  • 検索/ツイートを取得する
  • GET statuses / oembed（ところで、このメソッドはアプリケーションなしでは呼び出せなくなりました）
  • ステータスの取得/ show /：id
  • ユーザー/ルックアップを取得
  • ユーザーの取得/ショー


それだけですか？ はい、それだけです。 他のすべてのメソッドは、15分以内に15回まで呼び出すことができます。 したがって、 ストリーミングAPIを使用することを常にお勧めします 。



JSONのみ



Twitter API v 1.1は、需要が低いため、XML、Atom、またはRSSをサポートしなくなりました。



すべてのリクエスト-アプリケーションでのみ



これで、すべてのTwitter APIリクエストに、OAuth 1.0aに準拠した認証情報を添付する必要があります。 現時点では、すべてのリクエストにはユーザートークンが必要ですが、アプリケーションのコンテキスト内での認証は数週間以内に追加されます-これはユーザートークンを含まないタイプのリクエストであり、アプリケーションに関する情報のみです（たとえば、前述のoembedまたはツイートの検索） 。



道路セクションのルールを更新



Twitterのコンテンツを使用して何が可能、何が不可能、そして何を行う必要があるかに関する開発者向けの情報を含むセクション 。 ショートスクイーズ：

  • ツイートを表示する場合、コンテンツを変更したり、アクションの名前やアクション自体の本質（リプレイ、リツイート、お気に入りなど）を変更したりすることはできません。
  • 1,000,000人を超えるユーザーがいる場合は、交渉テーブルへようこそ。 そして他の方法はありません。
  • 100,000人以上のユーザーを持つ従来のtwitterクライアント-私たちはあなたに従います。
  • ユーザーを驚かせないでください。
  • Twitterコンテンツを共有しないでください。 アプリケーションがTwitterからのデータを保存している場合-他の人に転送できない場合は、他のクラウドプラットフォームに投稿してください。
  • スパムしないでください。
  • ユーザーのプライバシーを尊重します。 「プライベート」コンテンツを公に投稿しないでください。
  • クエリの頻度制限を欺こうとしないでください。 ルールはルールです。
  • 広告を表示する場合は、Twitterコンテンツから明確に分離されていることを確認してください。
  • サービスごとに1つのアプリケーションのみ。 アプリケーションごとに複数のサービスはありません。




設計上の推奨事項が要件になりました



多くのポイントがあります。 要するに、つぶやきはウィジェットのように見えるはずで、それ以外はほとんど何もないはずです。 要件の完全なリストを読むことをお勧めします。





デフォルト設定の変更



include_entitiesパラメーターはデフォルトでtrueに設定されています（ツイートには、リンク、写真、ハッシュタグ、ビデオに関するすべての情報が含まれます）。

include_rtsパラメータはデフォルトでtrueに設定されています（フィードにリツイートが表示されます）。
