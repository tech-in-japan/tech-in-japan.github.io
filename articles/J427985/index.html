Onkyo ISCP / eISCP：ネットワークを介したOnkyoデバイスの管理

Habrの読者の多くは、オンキヨーのオーディオ機器について知っているか、少なくとも聞いたことがあると思います。 最新のネットワークプレーヤーとA / Vレシーバーには、Linuxが搭載されており、有線/無線ネットワーク接続が可能です。 オンキヨーは、そのようなデバイスであるオンキヨーコントローラーのリモートコントロール用に独自のモバイルアプリケーションを提供しています 。 このアプリケーションがどのように機能するかに関する情報はほとんどありません-フォーラムにはパンくずがあり、githubにはいくつかのプロジェクトがあります。







しかし、ネットワーク上で、このアプリケーションの基礎であるIntegra Serial Communication Protocol over Ethernet（eISCP）の説明を見つけることができます。 プロトコルは興味深いです。 Habréでは、このプロトコルに関する記事は1つも見つかりませんでした。 一方で、この所有権はオンキョー以外では使用されていないように見えるため、これについて悲劇的なことは何もありません。 一方、自分のプレーヤーやオンキヨーのレシーバーを自分で操縦したい愛好家がいる可能性があります。 また、この記事は、純粋に理論的な好奇心から、さまざまなネットワークプロトコルに関する知識を収集する人にとっても興味深いものです。 興味があれば、猫の下でお願いします。



この記事のトピックに関する公式情報はほとんどありません。 したがって、プロトコルコマンドのみを説明しているが、それらの使用の機能については何も述べていないため、見つかったドキュメントだけに依存しません。 デバイスファームウェアの調査だけでなく、tcpdump / wiresharkを使用したネットワークトラフィックの分析から多くの情報を得ることができました。 ここから始めます。



デバイスの特定のモデルは重要ではありません。 これはネットワークプレーヤーであるとしか言いようがありませんが、これは写真の1つに似ていますが、注目を集めています。 外部USBキャリアからだけでなく、ミュージックサーバー（DLNA）からも音楽を再生でき、Spotify、Deezerなどのインターネットラジオおよびストリーミングサービスもサポートしています。 当然、プロトコルはこの多様性をすべてサポートする必要があります。



ポート分析



検索エンジンで適切な質問をするためには、まずどのようなプロトコルが一般的に使用されているかを理解する必要がありました。 つまり、最初のステップはポート分析です。 したがって、デバイスはネットワーク上にあり、そのアドレスは192.168.1.80です。 ポートの全範囲をスキャンします。 



> nmap -sS -p0-65535 -T5 192.168.1.80 PORT STATE SERVICE 80/tcp open http 4545/tcp open worldscores 5000/tcp open upnp 8008/tcp open http 8009/tcp open ajp13 8080/tcp open http-proxy 8888/tcp open sun-answerbook 10001/tcp open scp-config 60128/tcp open unknown


多くの興味深いことが発見されています。



  • 80 / tcpは明確です-これはデバイス構成ページです。 私のモデルでは、ネットワークのセットアップとファームウェアの更新のみがあります。 再生コントロールはありません。 これにより、「http://192.168.1.80/album_art.cgi」などの動的リンクを通じて、現在再生中のトラックの画像にアクセスできます。
  • 4545 / tcp-最新のファームウェアの更新後に表示されます。 Nmapは彼について何も知りません。 接続しようとすると、すぐにjsonに現在の再生ステータスが送信され、毎秒更新が送信されます

    私が言ったように、このポートは最新のアップデートで登場しました。 単語からのドキュメントはまったくありません。 軽量のコントロールパネルの開発に役立つ場合があります。 しかし、この方向で私はまだ掘っていません。
  • 5000 / tcp-nmapは、Apple AirTunesとして定義します。 このプロトコルのサポートはドキュメントに記載されているので、本当のようです。
  • 8008 / tcp、8009 / tcp-目的は明確ではなく、nmapはそれらについて何も知りません。

  • 60128 / tcp-最後に、この記事の主人公であるeISCPプロトコルポート。 Nmapも彼について何も知りません。 秘密のベールを開けましょう。


トラフィック分析



次に、どのポートと、公式アプリケーションがデバイスと通信するかを確認しましょう。 これを行う最も簡単な方法は、ルート化されたAndroid上です（ただし、公式アプリケーションには同じローカルサブネット上の管理対象デバイスが必要なため、エミュレーターではありません）。 これを行うには：



  • Onkyo Controllerが既にインストールされているAndroidデバイスにAndroid tcpdumpをインストールします
  • rootとしてadb経由でAndroidデバイスにアクセスします。 



     > adb root && adb shell root@fiber-bs1078:/>
  • 内蔵SDカードの任意のディレクトリに移動します。 



     root@fiber-bs1078:/> cd /sdcard/work root@fiber-bs1078:/sdcard/work>
  • tcpdumpを開始します（ファイル内のレコードを使用） 



     root@fiber-bs1078:/sdcard/work> tcpdump -vX -i any -w onkyo.dump host 192.168.1.80 tcpdump: listening on any, link-type LINUX_SLL (Linux cooked), capture size 262144 bytes
  • Onkyoアプリケーションを起動し、そこから音楽の再生を開始します
  • 数百のパケットが入力されたら、Cttl + Cでtcpdumpを停止します
  • ターミナルに戻り、ADBを起動して、ファイルを動作中のコンピューターにコピーします 



     root@fiber-bs1078:/sdcard/work> exit > adb pull /sdcard/work/onkyo.dump . [100%] /sdcard/work/onkyo.dump
  • Wiresharkを起動して、そこで何が起こるかを確認します。 



     > wireshark onkyo.dump &


実際、通信はポート60128で行われます。 たとえば、アプリケーションはプレーヤーにリクエストを送信します。







そして彼は彼に答えます：







それで、私たちは記事のまさに本質に来ます、すなわち：そのような手紙のための上の写真の中のそれは何ですか-ISCP？ この略語は、もともとRS-232ポートを介してOnkyoデバイスを制御するために設計されたIntegra Serial Control Protocolを意味します（このテーマに関する古い興味深い記事があります）。 その後、プレフィックス「e」を追加して拡張され、eISCP-Integra Serial Communication Protocol over Ethernetが判明しました。 プロトコルの両方のバージョンについては、技術文書「AVレシーバー用の統合シリアル通信プロトコル」で説明されています。 ドキュメントの最初のバージョンの日付は2012年10月31日で、最後に見つかったバージョンは2017年9月4日です。見つかったすべてのバージョンは、デモプロジェクトのリポジトリに収集されます 。 さらなるプレゼンテーションは、このドキュメントとプレイヤーの実験の両方に基づいています（ただし、このドキュメントでは明示的に言及していません）。



メッセージ仕様



クライアント（モバイルアプリケーションなど）とデバイスは、短いテキストメッセージを交換します。 番号が含まれている場合、それらは通常、16進形式で表示されます。



クライアントからデバイスへのメッセージ形式は非常に簡単です。







メッセージは「！」で始まり、ターゲットデバイスコード、コマンドの3文字、任意の長さのパラメーターの文字列が続きます。 CR（0x0D）またはLF（0x0A）、またはCR + LFの組み合わせで終了します。



コマンドに応じて、デバイスは同じタイプのメッセージ（何らかのパラメーターの要求の場合）、または異なるタイプまたはメッセージの組み合わせ（トラックの切り替えなどの複雑なアクションのコマンドの場合）で応答します。 デバイスからクライアントに送信されるメッセージの形式は同じです。 違いは最後のバイトのみです。







プロトコルの説明文書には100を超えるさまざまなコマンドが含まれていますが、私のデバイスはこの文書から30を超えるコマンドをサポートしています。 つまり、コマンドのセットと有効なパラメーターの両方が特定のデバイスに依存します。



コマンドは論理グループにグループ化できます。 例として、これらを強調します。



  1. 一般的なデバイス管理：
    • NDN：デバイス名。
    • UPD：ファームウェアアップデートの確認とインストール。
    • PWR：電源オン/オフ。
    • NRI：拡張デバイス情報。
    • NTC：標準のリモートコントロールのコマンド（再生コントロールを含む）。
    • CAP：RIコネクタに接続された外部アンプを制御するコマンド。
  2. 再生中のトラックに関する情報：
    • NAL：アルバム名。
    • NAT：アーティスト名。
    • NTI：トラック名。
    • NFI：トラックファイル情報（形式、ビットレート）。
    • NJA：トラックに添付された画像（たとえば、インターネットラジオが選択されている場合、ラジオ局のエンブレム）。
    • NTM：トラック内の現在の時間位置。
    • NTS：ステータス、巻き戻しが有効かどうか（たとえば、インターネットラジオの場合は許可されません）。
    • NST：リピートおよびランダムプレイコントロール。
  3. 音楽ライブラリのナビゲーションと管理：



    • SLI：ソースの選択（例：USB、ネットワークサービス）。
    • NSV：特定のネットワークサービス（インターネットラジオ、ミュージックサーバーなど）を選択します。 デバイスのプレイリストはネットワークサービスにも関連していますが、これはユーザーインターフェイスの観点からは完全に明らかではありません。 そして、電源を切る（コンセントから抜く）と、このプレイリストは削除されます！
    • NLT、NLA：ライブラリのセクション（フォルダー）を介したナビゲーション。
    • PQA、PQR、PQO：プレイリスト管理：追加、削除、並べ替え。


当然、このリストは完全にはほど遠いので、このプロトコルの範囲と機能を示すためだけにリストを提供しました。



パラメーターに関しては、すべてのメッセージを2つのグループに分けることができます。 最初のグループには、ほとんどのメッセージが含まれます。 このグループの場合、パラメーター文字列には英数字または16進数形式のデータが含まれ、バイトごとに解析されます。 たとえば、TuneIn Radioサービスに切り替えると、プレーヤーはNLTメッセージを送信します。パラメーター「0E01000000090100FF0E00TuneIn Radio」を持つ現在のリストのヘッダーに関する情報は、仕様に従ってデコードされ、次の情報を提供します。 



 SERVICE=TUNEIN_RADIO; UI=LIST; LAYER=SERVICE_TOP; CURSOR=0; ITEMS=9; LAYERS=1; START=NOT_FIRST; LEFT_ICON=NONE; RIGHT_ICON=TUNEIN_RADIO; STATUS=NONE; title=TuneIn Radio


ほとんどすべてのメッセージには、「！1NLTQSTN」などの「QSTN」パラメーターがあります。 このリクエストは、このタイプのメッセージに対応する現在のステータス情報を返すためのプレーヤーへのリクエストを意味します。 それはほとんど常に機能しますが、プレーヤーが内部のムードに応じてそのような要求を無視する場合、まれな例外があります。



2番目のグループはメッセージです。パラメーターはXMLであり、XMLパーサーを使用して解析する必要があります。 上記の例から、TuneIn Radioセクションで、XML形式のアクティブリストに関する情報に応答するNLAリクエストを送信できます。

つまり、プレーヤーはテキスト情報（偶然、プレーヤーのディスプレイに現在表示されている）を提供するだけでなく、適切なアイコン（フォルダー、音楽トラック、現在再生中のトラック）もアドバイスします。



場合によっては、プレーヤーはクライアントアプリケーションにテキストメッセージを表示したり、ユーザー名などの追加パラメーターを要求したりします。 これを行うために、プレーヤーはユニバーサルメッセージNCP（ユニバーサルダイアログ）を送信します。ここでは、ユーザーに表示する必要のある構造がXMLで記述されています。

ISCP over Ethernet（eISCP）



上記のフォームのメッセージは、ケーブル（RS-232）を介した送信を目的としています。 古いレシーバーには、9ピンRS-232コネクターが装備されていました。 このコネクタの代わりにネットワーク接続（有線または無線）を使用し始めたとき、これらのメッセージをTCP / IPで送信するためにラッパーでラップする必要がありました。 そのため、このようなパッケージにISCPメッセージがラップされているeISCPプロトコルが登場しました。







ネタバレの下にあるのは、特定のコード（変数コード）、一連のパラメーター（変数パラメーター）、および特定のプロトコルバージョン（変数バージョン）を持つメッセージのパッケージを完全に生成するプロシージャのコードです。 手順は非常に単純なので、Javaコードは1000語以上を言うと思われます。

情報交換の実施



もともと低速ケーブルでの伝送用に設計されたメッセージ自体は非常に小さいものです。 さらに、プレーヤー自体もかなり控えめです-条件付きAmazonサーバーに送信される膨大な量の統計情報を背景に、プレーヤーがISCPを介してクライアントに自発的に提供する情報量は悲惨です。 プロトコル仕様には、プレイヤーがこの情報またはその情報を送信するタイミングと条件についての言葉はありません。 そのため、モバイルクライアントがデバイスの現在の状態に関するすべての必要な情報を常に取得できるように、十分な時間をかけて実験する必要がありました。



一般に、プレーヤーとの通信は要求/応答スキームに基づいています。 さらに、特定の状況では、単一のリクエストが制限されません。 処理する必要がある私のプレーヤーのいくつかの重要なイベントがあります。



  • 接続を確立します。 接続時に、プレーヤーはスタンバイモードまたはオンになっているか、再生モードまたは一時停止になっている可能性があります。 ネットワークサービスまたはUSB上で、入力チャネルスイッチがどの位置にあるかをすぐに見つけることも重要です。



    したがって、接続が確立された直後に、PWR要求（アクティブまたはスタンバイ状態）、UPD（ファームウェアの更新がある）、NRI（デバイスに関する一般情報）、SLI（入力スイッチの位置）、NJA（トラックの画像の送信モード）を送信することは理にかなっていますリンクまたはストリーム）。 私のプレーヤーは、再生ステータスと現在の位置を独自のイニシアチブで具体的に送信します。
  • 再生を開始します。 この状況では、プレーヤーはトラックに関するすべての情報を送信します。 しかし、接続するとき、プレーヤーがすでに何かをプレイしているとき、プレーヤーは何も送信しません。 また、プレーヤーがトラックを切り替えると、すべての情報が送信されるわけではありません。



    リソース集約型の一般的なソリューションは、NSTメッセージ（再生ステータス）を追跡し、この状態が「再生」に切り替わった場合、すぐに7つのリクエストを送信します：NAT（アーティスト）、NAL（アルバムタイトル）、NTI（トラックタイトル）、NFI （ファイル情報）、NTR（トラック番号）、NTM（現在の再生時間）、NMS（トラックメニュー）。 プレーヤーのファームウェアには機能があります。 たとえば、プレイリストを再生するとき、プレーヤーは再生中のトラックの番号を教えたくありません。 ただし、一般的には、再生の現在の状態を十分に詳しく知ることができます。
