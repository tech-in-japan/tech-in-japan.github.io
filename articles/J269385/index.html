AからZへのDrupalキャッシュ

はじめに



サイトのページはサイズが大きくなっており、コンテンツを非同期に読み込むという問題が生じています。 しかし、メインページの読み込み後に簡単に画像を読み込むためのライブラリがある場合、ページのHTML部分（ブロック）の非同期読み込みでは、すべてがはるかに複雑になります。 たとえば、サイトのパフォーマンスを損なうことなくDrupalで非同期ブロック読み込みを実装するには、キャッシングの仕組み、使用方法、最適化方法を理解する必要があります。 その後、計画された計画「遅延読み込み」に従って、サイトのブロックを読み込みます。



Drupal Cacheの概要



Drupalのキャッシュは、最小のシステムリソースと最大速度でユーザーに返すことができる頻繁に要求されるデータを格納する中間バッファーです。 このバッファは、データベース内のテーブル、ハードディスク上のデータファイル、またはネットワークジャーナルデータウェアハウス内のキーと値のセットです。 DrupalでWebアプリケーションをキャッシュするためにPHPと統合できる情報を保存するためのオプションが適しています。



Drupalでキャッシュを使用する1つの例は、メニューキャッシュです。 ほとんどの場合、メニューはサイトの頻繁に変更される部分ではありません。 ユーザーがサイトの新しいページにアクセスしたときに表示するメニューを、毎回再構築する（すべてのアイテムを取得し、相互に依存し、テンプレートに表示する）メニューは意味がありません。 さらに、これが1000アイテムのカタログのメニューである場合、出力用に非常に長い時間再構築できます。 なぜユーザーを待たせるのですか？ したがって、Drupalは、管理者が指定した時間、一度作成したメニューをキャッシュし、キャッシュからユーザーに表示します。これにより、メニューを再構築する長時間の操作を回避できます。



Drupalには（基本モジュールに）組み込みキャッシュが含まれており、/ admin / settings / performanceで構成できます。 開発のいずれかに新しい機能を追加するとき（テンプレートまたはモジュールのいずれであっても）、定期的にクリーニングすることを忘れないでください。



キャッシュはいくつかの方法でクリアできます。

  • / admin / settings / performanceページの「Clear Cache」ボタンを使用します
  • admin_menuモジュールをインストールし、左端のタブで[すべてのキャッシュをフラッシュ]を選択します
  • リンク「空のキャッシュ」（キャッシュのクリア）を使用して、ブロックDevelブロック（モジュールDevel）
  • Drush：drush cache-clear themeコマンドを実行する（テーマキャッシュのみがクリアされます）
  • プログラムでdrupal_rebuild_theme関数を呼び出すことにより（テーマキャッシュのみがクリアされます）


Drupalキャッシュ-キャッシュセグメント



Drupalキャッシュは、単一の場所に保存されるのではなく、セグメント化されます。 デフォルトでは、各キャッシュセグメントは個別のテーブルとしてデータベースに保存されます。 これにより、頻繁に更新されるキャッシュセグメントを、MemcachedやBoostなどの他のキャッシュストレージシステムに移動できます。 これにより、キャッシュの操作のパフォーマンスも向上します。データ量が少ない場合、レコードの操作が高速になります。 さらに、特定のアクションでは、キャッシュを部分的にクリア（セグメント化）できます。



Drupalにはさまざまなキャッシュセグメントがあります。 キャッシュセグメントは、バージョン7.XXと6.XXで類似しています：



1. {cache} -共有キャッシュストレージのセグメント。 これには分類できないデータが含まれているか、それらの新しいキャッシュセグメントを作成しても意味がありません。



2. {cache_block}-Blockモジュールがオンになったときに追加されました（カーネルに含まれています）。 Drupalテーマ領域をロードすると、この領域のすべてのブロックにデータがロードされ、必要に応じて、hook_block_view（）フック呼び出しをスキップして、キャッシュからブロックが構築または表示されます。 hook_node_access（）を使用してマテリアルアクセスを操作するモジュールが有効になっている場合、ブロックのキャッシュが無効になっていることを考慮する価値があります。 また、hook_block_info（）を使用してプログラムでブロックを作成する場合、ブロックのキャッシュパラメーターを制御できることも知っておく必要があります（詳細については、ドキュメントを参照してください）。



3. {cache_filter}-Filterモジュールは、フィルターによって処理されたテキストのキャッシュを保存するための独自のテーブルを作成します。 フィルタが複雑になるほど、テキスト処理により多くのプロセッサ時間が費やされます。 したがって、可能な限り、check_markup（）へのすべての呼び出しはキャッシュされます。 {cache_filter}テーブルのキャッシュIDは、規則format_name：language：text hashに従って収集されます。



4. {cache_form} -サイトを高速化するために他のキャッシュデータが保存されている場合、このキャッシュはパフォーマンスに一切影響しません。 Forms APIを使用して構築されたフォームは、脆弱性の観点から完全に安全である必要があります。 フォームが作成されるたびに、{cache_form}セグメントに保存されます。 多くのフォームと訪問者がいる場合、cronを実行して1〜2時間ごとにキャッシュをクリアしないと、{cache_form}には印象的なサイズに拡張するプロパティがあります。



5. {cache_menu} -メニューモジュールがオンになるとオンになり、インターフェースを介して作成されたすべてのメニューからのリンクのリポジトリになります。 キャッシュIDは、次のルールリンクに従って作成されます。menu_name：tree-data：language：パラメーターのハッシュ。



6. {cache_page} -匿名ユーザーのキャッシュページデータを保存します。 現在のページのキャッシュが見つかった場合、hook_boot（）とhook_exit（）の2つのフックのみが呼び出されます。 残りのフック（hook_init（）などを含む）はスキップされます。 これは、「匿名ユーザーのページをキャッシュする」チェックボックスにより、サイトのパフォーマンス設定セクション（admin / config / development / performance）で有効になっているキャッシュです。



7. {cache_update} -更新マネージャーモジュールはこのセグメントを追加します。 含まれているモジュールのすべてのリリースのデータを保存します。



Drupalバージョン7.XX（バージョン6.XXではない）で使用可能なキャッシュセグメント ：



1. {cache_path} -システムパス内のエイリアスの検索を高速化するために、システムパスとそのエイリアス間の対応を保存します。



2. {cache_image} -Imageモジュールによって予約されており、画像のさまざまな操作に関する情報のストレージとして使用できます。



3. {cache_bootstrap}-Drupalの起動時に初期化されるデータを保存するキャッシュセグメント。



4. {cache_field} -このセグメントには、すべてのフィールドのデータが格納されます。 キャッシュIDは、ルールフィールドentity_type：entity_idによって形成されます。



サードパーティのモジュールは、独自のキャッシュセグメントを作成することもできます。 たとえば、ハッキングされたモジュールのセグメント、l10n_update、token、views：

  • cache_hacked
  • cache_l10n_update
  • cache_token
  • cache_views
  • cache_views_data


カーネルとは関係なく、CToolsモジュールによって作成されるCtoolsオブジェクトキャッシュに言及する価値があります。 CToolsオブジェクトキャッシュは、現在編集中の大きなオブジェクトを保存するためのスペースを提供するキャッシュセグメントです。 たとえば、変更されたビューのキャッシュは、Viewsモジュールに保存される前にCToolsオブジェクトキャッシュに保存されます。 他のキャッシュセグメントとは異なり、追加のフィールドsid（セッションID）-現在のユーザーセッションの識別子があります。 そのおかげで、変更されたデータはオブジェクトを変更したユーザーにのみ表示されます。 このセグメントには有効期限フィールドがなく、インターフェースを介してキャッシュがクリアされても削除されませんが、このセグメントから1週間前のキャッシュが削除されると、Cronによって1日に1回クリアされます。



ページのライフサイクル



ユーザーがWebサーバーからページを要求すると、次のことが発生します（たとえば、Drupalバージョン7.XX）：



Drupalカーネルは、次のパラメーターのいずれかで起動されます。

  • DRUPAL_BOOTSTRAP_CONFIGURATION ：構成のみを初期化します。
  • DRUPAL_BOOTSTRAP_PAGE_CACHE ：キャッシュレイヤーを初期化します。
  • DRUPAL_BOOTSTRAP_DATABASE ：データベース層を初期化します。
  • DRUPAL_BOOTSTRAP_VARIABLES ：可変層を初期化します。
  • DRUPAL_BOOTSTRAP_SESSION ：セッションを使用した作業の初期化。
  • DRUPAL_BOOTSTRAP_PAGE_HEADER ：ヘッダー層を初期化します。
  • DRUPAL_BOOTSTRAP_LANGUAGE ：ページの言語レイヤーを初期化します。
  • DRUPAL_BOOTSTRAP_FULL ：Drupalを完全にロードします。 また、入力されたデータをチェックおよび修正する機能も追加します。


詳細については、includesフォルダーにあるbootstrap.incファイルのdrupal_bootstrap（）関数を参照してください。



drupal_bootstrap（）がDRUPAL_BOOTSTRAP_FULLパラメーターを指定して実行される場合：

  • システム関数ファイルの接続。
  • すべてのレイヤーとプライマリ設定の初期化。
  • 含まれるすべてのモジュールのファイルの接続。
  • Drupalでパスとそのエイリアスを操作するための変数とシステム関数の初期化。
  • 含まれるスキンの初期化。
  • これは、module_invoke_all（）を実行することにより行われます。 Drupalモジュールをロードして対話するためのAPIを実装し、現在含まれているモジュールのすべてのフックを登録します。


その後、menu_execute_active_handler（）関数が呼び出されます。これは、ナビゲーションメニューを定義し、ページ要求をサイト上のパスにバインドされた関数呼び出しに変換します。 また、この関数内で呼び出しが行われます。

  • drupal_deliver_html_page（）関数。ページデータをHTMLとしてユーザーのブラウザーに返します。 この関数内でdrupal_render（）関数が呼び出されます。この関数は、データを表示するだけでなく、テンプレートエンジンを使用してページを再生成する代わりに、データをキャッシュセグメントの1つに保存してそこから取得します。
  • drupal_page_footer（）関数。必要に応じてページキャッシュ（「cache_path」と「cache_bootstrap」）を設定し、モジュールがhook_exit（）によるページのクローズに応答できるようにします。 Cronは必要に応じてすぐに起動されます。


一般的な場合：

  1. 必要な変数と関数はすべて初期化されます。
  2. このURL（{cache_page}）にキャッシュがあるかどうかを確認します。 存在する場合、彼は戻り、そうでない場合、さらなるアクションが実行されます。
  3. フィールド（{cache_field}）、コンテンツ（{cache_filter}）、メニュー（{cache_menu}）、ブロック（{cache_block}）、画像（{cache_image}）およびエイリアス（{cache_path}）のキャッシュがあるかどうかをチェックします。 キャッシュが使用できない場合、必要なデータを取得して処理し、キャッシュに保存する操作が実行されます。 受信したデータはテーマ機能に転送されます。
  4. テーマ関数はページを作成し、指定されたURL（{cache_page}）でキャッシュします。
  5. データがユーザーに返されます。


Drupal 7.Xを使用したプログラミングキャッシュ



キャッシュを操作する最も一般的な方法は、cache_set関数を使用して、開発したモジュールのデータをキャッシュに保存することです。 cache_get関数を使用してそれらから抽出するだけでなく。 



<?php // ,     : my_module_data. if ($cache = cache_get('my_module_data')) { //   ,   . return $cache->data; } else { //   ,   . $my_data = '   '; //      : my_module_data : {cache}. cache_set('my_module_data', $my_data, 'cache'); //  . return $my_data; } ?>


次のいずれかの関数を呼び出すことにより、my_module_dataという名前のキャッシュデータをクリアできます。 



 <?php //        //   (    ). cache_clear_all(); //    {my_module_data}. cache_clear_all('*', 'my_module_data', TRUE); //    {my_module_data} , //   Cache ID  c 'my_module'. cache_clear_all('my_module', 'my_module_data', TRUE); ?>


これは、ページのサイト管理者のインターフェース（ example.com/admin/config/development/performance）から[キャッシュのクリア]ボタンをクリックして実行できます。



example.com/admin/config/development/performanceページで[キャッシュのクリア]ボタンをクリックすることに依存せずに、データを一定時間キャッシュする場合は、データをキャッシュする秒数だけ追加のパラメーターをcache_set関数に追加する必要があります。 



 <?php cache_set('my_module_data', $my_data, 'cache', time() + 360); ?>


Drupalでは、データストレージ用に独自のキャッシュセグメントを作成できます。 このモジュールを作成しましょう：mymodule。 これを行うには、サイトディレクトリ：./sites/default/modulesにフォルダーmymoduleを作成します。 その中に、内容を含むモジュール記述ファイルmymodule.infoを作成します。 



 name = My module description = "      ." core = 7.x version = 7.x-1.x-dev files[] = mymodule.module


また、hook_schema（）を使用して、キャッシュセグメントのデータを格納する新しいテーブルを作成するmymodule.installファイルを作成します。 



 <?php /** * Implements hook_schema(). */ function mymodule_schema() { //    : {cache}. //         //      . $schema['mymodule'] = drupal_get_schema_unprocessed('system', 'cache'); $schema['mymodule']['description'] = 'Cache table stores some example data.'; return $schema; }


作成したセグメントを示すcache_set関数の使用を開始する前の最後のステップは、mymodule.moduleファイルを作成し、example.com/admin/config/development/performanceの[キャッシュのクリア]ボタンをクリックするとキャッシュを自動的にリセットすることです。 



 <?php /** * Implements hook_flush_caches(). */ function mymodule_flush_caches() { //          . return array('mymodule'); }


example.com/admin/modulesページでモジュールを有効にし、その後、たとえば、モジュールによって作成されたキャッシュの独自のセグメントに次のデータを保存します。 



 <?php cache_set('my_module_data', '     ', 'mymodule'); ?>


あまり知られていない関数cache_is_emptyがあり、指定された名前でデータがキャッシュに保存されているかどうかを調べることができます。 



 <?php cache_is_empty('my_module'); ?>


sub_module cache_example、module：examplesのページ： www.drupal.org/project/examplesで、Drupal 6.Xおよび8.Xのキャッシュの操作方法を常に確認できます。



データベースからキャッシュを削除する



セグメントキャッシュは、たとえば、MemcachedまたはRedisに移行できます。



各店舗には独自のサポーターがいます。 Evgeniy Maslovskiy（Spleshka、www.drupal.org / u / spleshka ）は記事の執筆を支援し、Memcachedの支持者であり、彼のサイトではMemcachedとDrupalの統合について詳しく説明しています。 彼が説明したMemcachedとの統合モジュールには、他のソリューションに勝る追加の利点があります。キャッシュを受信するときにデータベースへの接続をバイパスし、他のモジュールのhook_exit（）およびhook_boostrap（）呼び出しをバイパスします。



この記事では、Redisでのキャッシュレンダリングについて説明します。これは、Memcachedよりも簡単なセットアップで気に入っています。 両方のリポジトリの速度の比較テストは、別の記事で検討されます。



サーバーのRedis設定はインターネットで確認できます。 Drupal統合モジュールをwww.drupal.org/project/redisからダウンロードし、。/ sites / all / modulesディレクトリーに解凍します。 その後、Drupalの./sites/default/settings.php設定ファイルに変更を加えて、次の行を追加します。 



 $conf['redis_client_interface'] = 'PhpRedis'; $conf['redis_client_host'] = $relationships['redis'][0]['host']; $conf['redis_client_port'] = $relationships['redis'][0]['port']; //     PHP    Redis. $conf['redis_client_interface'] = 'PhpRedis'; $conf['cache_backends'][] = 'sites/all/modules/redis/redis.autoload.inc'; $conf['cache_default_class'] = 'Redis_Cache'; //     {cache_form}      . $conf['cache_class_cache_form'] = 'DrupalDatabaseCache'; //           : {mymodule} //     Redis. $conf['cache_class_mymodule'] = 'Redis_Cache';


モジュールフォルダー内のREADME.txtファイルを読むことを忘れないでください。すべてのモジュール設定が記述されているためです。
