C＃LINQのPHPへの移植

C＃のLINQクエリの機能は、PHPで書き直されました。 当初、このようなライブラリは既に存在するため、このライブラリはトレーニングとして考えられていましたが、その後、すべての人に公開することが決定されました。 C＃とPHPの機能は非常に異なるため、LINQをPHPに1対1でコピーすることはできません。 提案されたソリューションの特徴は、可能な限りC＃と同一のイテレータ、遅延ラムダ式、およびLINQメソッドのシグネチャへの方向付けです。 すべての標準LINQメソッドは自然に実装されます。 プロジェクトの機能の説明と、このソリューションが選択された理由の説明をカットの下で。







なぜLINQですか？

新しい技術は常に興味深いものです。 なぜ発生したのか、どのような問題を解決し、どのように解決するのか？ そのような機能の1つが、データシーケンス（配列、データベース応答、コレクション）をクエリするためのSQLに似た言語であるLINQ（Language Integrated Query）です。 例えば 



var q = from c in db.Customers where c.Activity == 1 select new { c.CompanyName, c.ItemID, c.ItemName };


C＃では、この構文のサポートは言語レベルで組み込まれていますが、実際には構文糖であり、次の形式に変換されます 



 var q = db.Customers. Where((c) => c.Activity == 1). Select((c) => { c.CompanyName, c.ItemID, c.ItemName });


ここで、 Where



およびSelect



関数はデータシーケンスに対して定義されていますが、処理ロジックは単一の要素に対して設定されています。 関数型プログラミングの精神で作られています-すべてが関数であり、関数の計算結果は入力パラメーターのみに依存します。 関数の純度の要件により、副作用によるエラーを事前に排除できます。 他にもプラスがあります：

  • 処理の順序は重要ではないため、セットに対する処理は並行して実行できます。
  • ローカル列挙の代わりに、LINQクエリを（部分的に含む）通常のSQLデータベースクエリにまとめることができます。 外部からのリクエストは同じままです。
  • 個々の要素の処理ロジックは分離されています。 他のコレクションに再利用したり、他のコレクションと組み合わせたりできます。
PHPで同じツールを使用できると便利です。 C＃のように、追加の計算オーバーヘッドがあまりないのはいいことです。 そして、そのようなライブラリがありますが、それぞれが異なる方法でこの機能を実装しています。 その理由は、使いやすいLINQ実装は、PHPが必要としない、模倣する必要のあるC＃機能の多くを利用しているためです。 そして、誰もが異なる好みを持っています。



なぜコピーできないのですか？

PHPのライブラリをコピーするには、一見しただけではC＃のどの機能が不十分なのかをリストします。

  • 緩いタイピング 。 それはむしろ利点です。
  • オーバーロードメソッドはありません。 これは前の段落の結果です。 たとえば、文字列、配列、オブジェクトを比較する必要がある場合、名前がcmpString



    、 cmpArray



    、 cmpLaptop



    3つの異なる関数をcmpLaptop



    か、1つの大きな関数内に置く必要があります。 両方のソリューションは悪いです。 最初のケースでは、名前の型情報がこれらの関数が使用されるコードを「詰まらせます」。 2番目の場合、機能を拡張することは困難です。
  • クラス拡張はありません。 メソッドを記述して別のクラスのメソッドであるかのように呼び出すことはできません。つまり、名前空間を接続するだけではIEnumerable <T>を拡張できません。 しかし、PHPには、他の場所で実装された静的メソッドを呼び出すことができる魔法の__callメソッドがあります。 確かに、目的のクラスを変更する必要がありますが、これは常に可能とは限りません。 IDEでのサポートについても忘れておく価値があります。
  • 良好な収益を生成するジェネレータはありません（php <5.5）。 一方で、それは構文糖です。 \Iterator



    を返す関数を作成できます。この関数は、 next()



    が次の要素の値を計算し、その状態をクラス属性に保存する匿名関数を呼び出します。 しかし、コードのサイズは何度も増加し、その有用なシェアは減少します。 ジェネレーターはバージョン5.5で登場しましたが、このバージョンが普及するまでに長い時間がかかります。
  • ラムダ式はありません。 これらは、機能が制限されているような小さな匿名関数ですが、構造に関する情報は保持されます。 関数の関数を記号的に計算したり、式を別の言語、たとえばSQLにエクスポートしたりするために使用できます。 PHPで匿名関数を作成できますが、その構造に関する情報はないため、SQLへのエクスポートはありません。
  • オペレーターのオーバーロードはありません。 また、 $f = (Exp::x()+1) * 2



    ようなものを記述し、 Closure



    の子孫であるExp::x()



    メソッドによって返されるクラスの加算および乗算演算をオーバーロードすることにより、ラムダ式を美しくシミュレートすることはできません。
言語でコレクションを操作するための基本クラスの呼び出し方が異なる（ IEnumerator



がIterator



、 IEnumerable



がIteratorAggregate



）、PHPに配列のメソッドがないなどの小さな違いがたくさんありますが、これはすべて簡単に解決できます。



どうする 何をする

仕事を始める前に、私は他の人の実装からではなく、C＃の印象の下で書く他のソリューションを特に探しませんでした。 最初のオプションは数晩で書かれました。 その後、彼は長い間MSDNからすべての標準メソッドを移し、余分な機能を切り取り、.NETに準拠したロジックをもたらしました。 年の初めに、私は他のプロジェクトの機能を比較し、コードを修正し、githubでプロジェクトを公開しました。 開発中、主に次の点に重点が置かれました。



イテレータはすべてであり、すべてはイテレータです



ライブラリは配列から配列にコピーする複雑なループの代わりに、イテレーターで動作します。 PHP 5.5より前では、イテレーターを処理するために、 \Iterator



インターフェースを実装するクラスを作成し、コンストラクター\Iterator



渡して入力パラメーターとして処理する必要がありました。 



 $data = new \ArrayIterator([1,2,3]); $even = new \CallbackFilterIterator($data, function ($item) { return $item % 2 == 0; } );


現在のイテレータからデータを読み取るとき、データは親イテレータからプルおよび処理され始めます。 イテレータを実行するオーバーヘッドは最小限であるようです。 一般的なシーケンス処理タスクには、15を超える反復子が実装されています 。

  • CallbackFilterIterator-アイテムのフィルタリング
  • CallbackIterator-関数による無限シーケンスの生成
  • DistinctIterator-一意の要素を発行する
  • ExceptIterator、IntersectIterator-減算、2つのシーケンスの交差
  • GroupingIterator-キーによるグループ化
  • IndexIterator-キーによるアイテムのソート
  • JoinIterator、OuterJoinIterator-キーによる2つのシーケンスの厳密な、厳密ではないバインディング
  • LimitIterator-一連のシーケンスを発行する
  • ProductIterator-複数のイテレーターのクロス積
  • ProjectionIterator-要素（キー）の投影
  • ReverseIterator-順序を逆にする
  • SkipIterator-条件がtrueの間要素をスキップする
  • TakeIterator-条件が真のときの要素の問題
  • LazyIteratorは抽象クラスです。 反復子は、要素が最初に読み取られたときに構築されます。
  • VariableIterator-親イテレータは、イテレータを開いた後に変更される場合があります
イテレータは、他のLINQ機能とは独立して使用でき、別の名前空間に移動することもできます。



遅延計算も機能します



匿名関数がLINQメソッドの式として渡されると、最高の実行速度、美しい強調表示、およびIDEでのリファクタリングが可能になりますが、式の構造に関する情報は失われます。 これにより、SQLなどの別のプログラミング言語で式を再構築することはできません。 ラムダ式とは異なります。 この問題を解決するために、多くの著者が有効なPHPコードの文字列を「式」として渡します。 文字列は、シーケンスの各要素に対してevalを使用して評価され、SQLなどの別の言語に再フォーマットできる可能性があります。 独自の文字列形式、たとえば$x ==> $x*$x



考え出す人もいます。 この場合、IDEでのコードの強調表示とリファクタリングは失われ、実行は長くなり、コードはキャッシュされず、安全ではなくなります。



提案されたライブラリには、複雑な式を簡単に作成できるツールが作成されています。 式の構造に関する情報は失われず、後で再利用できます。 基礎となるExpressionBuilder



クラスは、ストリーミングモードで計算ツリーを作成し、逆ポーランド（後置）レコードにエクスポートします。 たとえば 



 $exp = new ExpressionBuilder(); $exp->add(1); $exp->add('+',1); $exp->add(2); $exp->export(); // [1, 2, 2, '+']


操作の優先順位と括弧がサポートされています。 Expression



クラスは、アンロードされた配列を実行し、データに遭遇した場合、それをスタックにスローし、 OperationInterface



型のオブジェクトに遭遇した場合、制御をそれに転送します。 オブジェクトは必要な数の引数をスタックから取得し、結果を計算してスタックにスローします。 スタックの最後に、1つの値（結果）が残ります。 上位レベルでは、 LambdaInstance



クラスとそのLambda



デコレーターを使用して式が構築されます。 機会の例。

  1. 引数、定数へのアクセス 

     /*   */ $f = Lambda::v(); $f = function ($x) { return $x; };
  2. 数学演算、比較演算、論理演算 

     $f = Lambda::v()->add()->v()->mult(12)->gt(36); $f = function ($x) { return $x + $x*12 > 36; };
  3. 括弧 

     $f = Lambda::v()->mult()->begin()->c(1)->add()->v()->end(); $f = function ($x) { return $x * (1 + $x); };
  4. 文字列操作 

     $f = Lambda::v()->like('hallo%');
  5. 配列生成 

     $f = Lambda::complex([ 'a'=>1, 'b'=>Lambda::v() ]); $f = function ($x) { return [ 'a' => 1, 'b' => $x ]; };
  6. オブジェクトのプロパティとメソッド、配列要素へのアクセス 

     $f = Lambda::v()->items('car'); $f = Lambda::v()->getCar(); $f = Lambda::car; $f = function ($x) { return $x->getCar(); };
  7. グローバル関数を呼び出す 

     $f = Lambda::substr(Lambda::v(), 3, 1)->eq('a'); $f = function ($x) { return substr($x,3,1) == 'a'; };
  8. 値のLINQメソッド 

     $f = Lambda::v()->linq()->where(Lambda::v()->gt(1)); $f = function (\Iterator $x) { return new CallbackFilterIterator($x, function ($x) { return $x > 1; }); };
もちろん、ラムダ式を計算するとき、追加のサイド操作が実行されます。 関数(x)=>x+1



場合、 Lambda



計算速度は関数の直接呼び出しより15倍遅く、構造自体は800に対して3600バイトのメモリを必要とします。プロファイラを使用して分析し、速度を上げてメモリを削減する方法を見つけ出す予定です。



インターフェースで会い、実装を見送ります
