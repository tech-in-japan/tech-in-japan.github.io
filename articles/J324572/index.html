JVMはそれほど重くない

基本的には、ClojureはJVMであるという答えです。 同様に、このがらくたはとても重いです。



これは、数週間前にSlackグループのZA Techチャンネルに登場しました。 Clojureでのいくつかのスピーチの間に、スピーカーは何度も何度もそのような発言をした。



この機会に、私はSlackで話をしました 。 今、私はより広い読書と議論のために書きます。



まえがき



また、JVMは重いと思っていました。 これは、PHPに比べて2000年代初期のことです。 .NETやColdFusionのような他のヘビーウェイトがありました。 PerlやPythonのような軽い代替品がありましたが、そのときはWindowsに座っていたので、ActivePerlとActivePythonも少し重いものでした。



Herokuに小さなJRuby実稼働アプリケーションをデプロイしたときに、最初にJVMの「恐怖」を克服しました。 この小さな怪物は1日に1つのタスクを実行するだけでした。 彼は多数のPDFを生成し、それらをiSignにアップロードして（現在は機能しません）、保管と配布を行っています。 ISign自体は、3つのAMIをホストする従来のRailsアプリケーションでした。 ストックJVM上のこの小さな恐竜（ -server -Xmx=512M



除く）は非常に高速にPDFを生成したため、起動するたびに文字通り3モードのクラスターを殺しました。



私はまだ彼が仕事で少し重いと思っていたが、このugいアヒルの子に恋をした。



私は多かれ少なかれJRubyの開発とサクセスストーリーに従い、 Rubyfuza 2015でCharles Nutterと素晴らしい時間を過ごしました 。 私はとても刺激を受けて、JRubyでテストを実行するだけのRubyプロジェクトでプルリクエストを開き始めました 。 私は恥ずかしいことに、それを終えました。



そして2016年までに



2016年11月に、Railsアプリケーションをゼロから作成しようとしました。 数ヶ月ぶりに、私は自分のマシンでRubyでプログラミングしました。 brew upgrade



アップデートはrbenvにクラッシュしたため、インストールされているすべてのバージョンのRubyを破棄しましたが、気付くことすらありませんでした。



Jozi.rbでWebSocketに関するプレゼンテーションを行いました。



私の講演の冒頭で、ReactとRailsを共有するという一般的な感覚のために、 React on Railsリポジトリをいじりたいと思いました。 私は数か月間リフレームを使用しており、ベアReactでリフレームできることを確信していました。



列車はレールから外れ、非常に壮観でした。



1つのテンプレートアプリケーションのクローンを作成して実行するには、Xcodeを更新し、Xcodeのコマンドラインツール（合計6 GB以上）を更新し、RubyとBundlerの新しいバージョンをインストールし、テンプレートアプリケーションにインストールをリンクする必要がありました...簡単ですよね？ ほとんどのRailsアプリケーションと同様に、このテンプレートアプリケーションは依存関係グラフのどこかにlibv8に依存しており、これだけで1 GBを超えるサイズが追加されます。



全体の運動には数時間かかりました。



このような印象的なデモンストレーションで遊んだ後、私はこの作品がコイントスに変わることに気付きました。 したがって、私は彼を知っているので、Emberのフロントエンドをより良くすることにしましたが、十分な時間がありませんでした。



繰り返しますが、nvmを更新し、ノードの許容バージョンをインストールし、ember-cliをインストールし、npm と bowerを介してアプリケーションを生成し、依存関係をインストールする必要があります。



私は少しプレーして敗北を認めましたが、代わりに同僚と私の経験を共有しました。 それは屈辱的で、本当に屈辱的でした。 私は長い間参加していた世界で見知らぬ人のように感じました。



JVMが重いというステートメントに戻ります。



どのように計量しますか？



  • ダウンロード時にJDKサイズが大きすぎますか？
  • 彼は職場で多くのリソースを使用していますか？
  • ライブラリは多くのディスク容量を占有していますか？
  • 展開が重すぎますか？
  • それは一日にあなたを遅くしますか？


JVMに関する感情的な障壁を克服するのに役立ついくつかの質問を次に示します。



それで、それらの世話をして、実際に何が起こるか見てみましょう。



初期ボリュームは本当にそんなに大きいのですか？



JVMのこの「重さ」は 、おそらく大量の初期インストールから始まる純粋なFUDです。 約200 MBのJDKダウンロードと約15 MBのノードまたはRubyダウンロードを比較しています。 しかし、これはほんの始まりに過ぎません。 NodeとRubyの両方で、システム上にCコンパイラが必要です。わずか数百メガバイトです。 さらに悪いことに、おそらく実稼働環境でコンパイラが必要です！



NodeとRubyの両方に対する大量の負荷がこのような小さな連続したステップで隠れています。 費やした時間は言うまでもなく、停止してボリュームを計算する場合は、200 MBがより最適なオプションになります。



独自のウェブキットを作成するための手順のコンパイルを開始したくはありません。新しいツールがリリースされた直後に書き換えるのではありません。







JVMの操作はとても難しいですか？



JVMは高速で、おそらく最速の環境の 1つです。 時間が経つにつれて、より速く簡単になります。 何千人もの優秀なエンジニアが改善に取り組んでおり、過去21年間でさらに多くの技術者が貢献してきました。



実行の実際のスレッド、マルチコアのサポートがあり、地獄のように構成することも、そのままにしておくこともできます。 おそらく知っておくべき唯一の有用なことは、JVMのメモリ制限を設定することです。そうすれば、この環境の制限を操作する際に不思議を示します。



Herokuにデプロイしますか？ java -server -Xmx512m beast.jar



。 これで十分でない場合は、誰かにアドバイスを求めるお金があるでしょう...ああ、まあ、または単にStackOverflowを見てください。



チャールズと他のJRubyの連中が推進し続けている重要なポイントは次のとおりです。 ジェスチャがなければ、アプリケーションは、JVMのリリースごとに（JRubyの進行に関係なく）おそらくより速く実行されます。

展開はとても太りすぎですか？



あなたはおそらく私が話すことを予測することができます...



ビルド時には、単一のJARファイルを作成します。 アプリケーションがあらゆるプラットフォームで動作するためのすべてを備えています。 必要な場所にJARを置き、JVMで処理するだけです。



アプリケーションが何らかの大規模なアプリケーションサーバーで実行されるという要件はありません。JARファイルで直接、生産的なHTTPサーバーに簡単にリンクできます。 Nodeの人たちも、Rubyの人たちもそうしていますが、どういうわけか、JARファイルは独りでいることはできませんか？ 私もそう思いました...



たとえば、実稼働apt-get install build-essentials



を実行する必要がなくなりました。



JVMで毎日



8 GBのメモリを搭載した2012 MacBook Proで少なくとも5つの JVMプロセスを実行しています。 彼らは絶えず毎日働いています。 5つのRailsアプリケーションを同時に実行しようとしたことはありません。



なぜ5つ？ 2つはDatomic （データ収集とコンソール）用、1つはバックエンドAPI用、もう1つは作業中のフロントエンド用です。 バックグラウンドで自動テストが行​​われることもあります。 これらのプロセスJVMのかなりの部分が同じバイトをメモリにロードする必要があるため、macOSのメモリ圧縮がおそらく役立つと確信しています。











10か月前にこれについて話してくれたなら、私は笑うだけでした。 5人以上の JVMプロセスを開始するのは誰ですか？ クリッピングのために手を差し出すことはできますが、これは決して行いません。



ああ、クラスパスと他のすべてのクレイジーなものはどうですか？ Clojureの優れたツールキットのおかげで、彼らはまったく触れません。 これは、npmまたはbundlerを使用するのと同じ理由で、パスを含めることを心配しないでください。 これを行うことはできますが、おそらく気付かない別の問題があります。



Joys REPL



JVMインスタンスを継続的に停止および起動する必要がある場合、間違いなく夢中になります。 昔はJRubyについて悩んでいました。 幸いなことに、Clojureと優れたREPLを使用すれば、JVMインスタンスを再起動する必要があるのは、まれに私が実際にどこかでめちゃくちゃになったときだけです。 愚か者に対するかなり良い防御策があります。 Figwheelは何日も問題なく機能しています。



おわりに



JVMをオブジェクトとして評価するときは注意してください。 Javaをあらゆる意味で言語として評価しますが、Javaを仮想マシンと混在させないでください。



私もあなたのように考えていました。 JVMはそのカバだと思っていました。 幸いなことに、私はこれらの制限を取り除くことができ、それをサポートする何千人もの巨人の仕事の結果を得ました。



いかなる場合でも、この投稿を「ノードの終わり」または「Rubyの終わり」の兆候と見なさないでください。 それを新鮮な視点と考えてください。 JVMにアップグレードできない場合は、少なくとも自分の世界で肥大化を減らす方法を検討してください。
