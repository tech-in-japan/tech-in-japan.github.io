WebAssembly：新しい時代の始まり

ウェブには素晴らしい未来があります。

Brendan Ike は昨日、ウェブ開発コミュニティを爆破しました。ウェブは、JavaScriptよりもはるかに優れた動作をする新しい低レベルバイナリコンパイル形式を受け取ります。
Google、Microsoft、Mozilla、およびいくつかの独立した専門家がW3C WebAssembly Community Groupで新しいプロジェクトに取り組んでおり、彼らが取り組んでいるのは小さなことではありません。



WebAssemblyは次のとおりです。


  • JavaScriptの改善：すべての重要なことをwasmに実装し（WebAssemblyの略-翻訳者のメモ）、標準JavaScriptモジュールとしてインポートします。
  • 新しいプログラミング言語： WebAssemblyは、バイナリ形式で抽象的な構文ツリー（JavaScriptなど）を定義します。 コードを記述して、テキスト形式のエラーを取り除くことができます。 WebAssemblyは読みやすいです。
  • ブラウザーの改善：ブラウザーはバイナリ形式を理解します。つまり、開発者はバイナリをコンパイルでき、現在使用されているJavaScriptテキストファイルよりもはるかに多くを圧縮できます。 ファイルが小さいほど、ダウンロードは高速になります。 コンパイル時間を最適化する可能性に応じて、WebAssemblyのコードはJavaScriptよりも高速に送信および実行できます。
  • コンパイルの目的：他の言語がWebプラットフォームのスタック全体を通じて一流のバイナリサポートを受け取る能力。


これはJavaScriptにとって何を意味しますか？


この質問に答える前に、時間を巻き戻しましょう。 React、Angular、Backbone、jQueryの前の時代に戻ってください。



ここにいる。

ネットワークは、メッセージ配信システムを使用して表示されるハイパーテキストドキュメントのセットですが、まだ相互接続されていません。 最初のWebサーバーは、CERNのNeXTコンピューター（下記参照）で起動されました...






1991年の庭で。 私は灰色に変わる時間がありませんでした。 私は10,000番目のテキストベースのクエストゲームをハックしようとしています（それらをすべて数えていません）。



この目的のために一種のプログラミング言語を選択しました。 当時、私はすでにBASICとPascalにうんざりしていました。 Cを使用したかったのですが、できませんでした。BorlandTurbo C ++の箱入りバージョンに保存していました（文字通り、説明書とインストールディスクが入ったパッケージボックスに入っています）。 それから私は最も単純なターボアセンブラーさえ持っていませんでした。



アセンブリ言語で記述し、DOSのコマンドラインと「デバッグ」を使用して実行可能プログラムに「コンパイル」しました。 それがおかしいと思うなら、信じてください。 DOSで作業するのに十分幸運だった人でさえ、「デバッグ」を使用してアセンブラーをデバッグし、既存のコードを逆アセンブル（リバースエンジニアリング）できることを知りません。



かっこいい？ いや 嫌いだった。 新しいBorland Turbo C ++に座って人間的にプログラムできる瞬間を待つことができませんでした。 一般的に、彼らは私にそれを与えました。 やった！



Borland Turbo Assemblerがプリインストールされているため、私はBorland Turbo C ++が好きでした。 なに？！ すばらしい高レベルのオブジェクト指向C ++ツールがあるのに、なぜアセンブリ言語でプログラムするのですか？



時には、脳を溶かすことなく、裸の鉄またはそれに近い鉄で作業したいことがあります。 ちなみに、C ++を使用する前に多くのマシンコードを記述したことを言及しましたか？

私は夢中です。
アセンブリ言語でコードを書くとき、実際の結果を達成することはそれほど簡単ではありません。 それでは、なぜWebAssemblyが必要なのでしょうか？



WebAssemblyが必要なのは、Javascriptの柔軟性がすべてあるため、その中に見たい多くのことを想像することはまだ困難であり、そこに追加する必要がある言語の機能は言語を複雑にするだけであり、 ユーザーを混乱させるからです 。



WebAssemblyを使用すると、あらゆるプログラムに使用できる単純な低レベルのコードブロックを操作できます。



JavaScriptとどう違うのですか？ キーワードは「低レベル」です。 プリミティブを定義します。これには、これらのタイプに対する多くのタイプと操作、それらのリテラル、制御フロー、呼び出し、ヒープなどが含まれます。



これらは非常に単純なプリミティブです。 複雑なことは何もありません。 複雑なオブジェクトシステム（プロトタイプまたはその他）はありません。 組み込みの自動「ガベージコレクション」は存在せず、ガベージコレクションが必要になったときに定期的に停止します。



実際、WebAssemblyを一般的な言語とは考えない方が便利です。 チューリングマシンに少し似た車のように見えますが......



WebAssemblyとは何ですか？


WebAssemblyは、 バイナリ形式で格納される抽象構文ツリー （AST）を定義します 。 バイナリは、 小さなアプリケーションを作成できるので素晴らしいです 。 バイナリコードをデバッグする方法を疑問に思っていることでしょう。



幸いなことに、ブラウザで動作するデバッガが積極的に開発されており、抽象的な構文ツリーが（比較的）読みやすいテキスト形式で表示されます 。 例を示したいと思いますが、今のところあまり多くはありません。 おそらく、WebAssemblyのコードは、JavaScriptで手書きで記述された同様のコードよりも読みにくいかもしれませんが、少なくともASM.jsと同じくらい読みやすいでしょう。 おそらくもっと簡単です。 見てみましょう。



WebAssemblyは何のために使用されますか？


とりわけ、スレッドとSIMD（単一命令、複数データ）の単純な作業、つまり、1つのコマンドストリームと複数のデータストリームで使用できます。 多くのデータブロックをキューに入れてから、1つのコマンドを作成して同時に処理できます。



これは、ストリーミングビデオの並列処理がプロセッサによって処理されることを意味します。 後れを取らない場合、JSでこの決定について聞いたことがありますが、JavaScriptで低レベルの問題を解決するのはいつも不便でした。



このような場合、おそらくオブジェクトシステム、「ガベージクリーナー」、および動的なクエリ処理を忘れてください。 データストリームをキューに入れて、できるだけ迅速かつ効率的に処理します。



アプリはどうですか？


現在、 Ableton Live （音楽の作成）やAdobe Premiere Pro （ビデオの作成）などのアプリケーションは、Webへの移植にはあまり適していません。 これは可能ですが、まだ難しいことに注意してください。 解決すべき多くの問題がまだあります。 たとえば、リアルタイムアプリケーションのデータを最適に同期する方法を決定する必要があります。



さらに、処理機能のネットワークを介して大量のデータを転送するアプリケーションを開発することも可能です。 たとえば、ギターペダルのアプリケーションなど。 人々はそのようなことを考えるとき、通常JavaScriptについて考えません。 多くの人は、これを試してみるのも純粋な狂気だと思うでしょう。



しかし、いずれにせよ、JavaScriptは、夢見ることができるアプリケーションのほとんどを構築するための本当に素晴らしい言語です。
WebAssemblyは、JavaScriptのギャップをシームレスに埋めます。
JavaScriptの機能にはギャップがあり、これは決して秘密ではありません。 彼の最も忠実なファンでさえ、言語が過度に「飲み込む」ことを試み、柔軟性と効率性を失うという主張に反対することはほとんどありません。 昨日、JavaScript自体に機能を追加するだけで、これらのギャップに対処できると思いました。 Brendan IkeはFluent会議でも同じ道を提案しました。 拍手しました。



ただし、この間ずっと1つのポイントを見逃していました。誰もが高水準言語でプログラミングしたいのですが、同時に、速度を上げる必要があるときにアセンブリ言語に「シンク」できます。

WebAssemblyは、JavaScriptの速度を向上させることがあります！
今日、WebAssemblyについての記事がWeb全体で数百ではないにせよ、何十もあると確信しています。 それらのほとんどが、以前に書いたものではなく、このプログラミング言語のまったく異なる機能に専念することは間違いありません。 ところで、私も彼女に喜んでいます。



WebAssemblyを使用すると、Web開発でより多くの言語を使用できます


もちろん、実際には、WebAssemblyがWeb開発で他のプログラミング言語を使用する必要はありません。 たとえば、JavaScriptのコンパイルで素晴らしいAAA ゲーム エンジンが既にあります。

JavaScriptが遅いと聞いた場合は...そうではありません。
WebAssemblyは、ほとんどのJS開発者がJavaScriptで見たくないものを追加します。 機能自体が必要ですが、JavaScriptには間違いなく場所がありません。 さらに、他のプログラミング言語からのコンパイルを使用して、これらすべての機能を取得できます。



実際、WebAssemblyは、この目的のために特別に作成された代替コンパイラを提供します。



これで、たとえば共有メモリチェーンに大きく依存するコードを移植するのがはるかに簡単になります。 WebAssemblyのコンパイラーを記述する方がJavaScriptのコンパイラーを記述するよりも簡単であると確信しています。これは、最初のものが言語関数を特定の抽象構文ツリーに最適に転送することを保証するためです。



すべての古いプログラミング言語がWeb上で問題なく使用できるようになったという事実は良いことですが、これは主なことではありません。 事実は次のとおりです。
WebAssemblyは、開発者が新しいプログラミング言語で作業を開始する優れた理由です。
ネットワークには素晴らしい未来があります。 そのため、手遅れになる前に、影から抜け出してビジネスに取り掛かることをお勧めします。

Q： JVMを使用しないのはなぜですか？

A：プラグインを使用してJVMをブラウザーに追加しようとする試みが複数回ありました。 残念ながら、これについては良いことは何もありません。 JavaScriptには仮想マシンが組み込まれているため、別の仮想マシンを追加すると、2番目の接続APIセットが表示され、仮想マシンがDOM、ネットワーク、センサー、入力デバイスなどにアクセスできるようになります。 このために何かを犠牲にしなければなりません。 たとえば、仮想マシンのプロセスは、利用可能なリソースをどのように分散しますか？ この質問に答えることは、思ったより難しいです。



最初は、WebAssemblyはASM.jsポリファイルとして機能します。つまり、JavaScript仮想マシンを使用できます。 これに基づいて言語の設計が開発されたため、WebAssemblyは、代替の仮想マシンが提供できるよりもスムーズにブラウザと統合できます。



Q： WebAssemblyの登場は、将来多くの新しいプログラミング言語が登場することを意味しますか？ これは断片化につながりますか？
JavaScriptは完全に安全です。 その生態系は今後数年間繁栄します。 WebAssemblyは、断片化というよりも、パフォーマンス、多様性、および前進に関するものです。
A：JavaScriptは、サーバー側だけでなく、小型コンピューターやロボットなどの組み込みシステムのプログラミングにおいても常に非常に深刻な競争を繰り広げてきました。 開発されたエコシステムと専門の開発チームとのかなり良い選択肢が多数存在しているにもかかわらず、Nodeは新興企業や営利企業のサーバーでの存在感を急速に高め続けています。



さらに、JavaScriptは強力なエコシステムだけでなく、開発者の銀河に直面しても優れたサポートを提供しています。 さまざまなモジュールの人気グラフを記事に追加するのが好きです。 これらの曲線を見て驚く。
