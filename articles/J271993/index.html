PVS-Studioアナライザーの実験版があり、C＃-projectsを分析でき、世界中に表示できます。 これはリリースではなく、ベータ版でもありません。 これは、現在のPVS-Studioのビルドです。 ユーザーまたは潜在的なユーザーからC＃サポートに関するフィードバックをできるだけ早く取得したいと考えています。 そのため、C＃プロジェクトでPVS-Studioの新しいバージョンを試してみて、結果、欠点、希望を表明する愛好家を募集しています。 そうそう、もちろんこの記事では、次のプロジェクトであるSharpDevelopの検証結果について説明します。



PVS-Studio



ここで重要な質問の1つは、「C＃用の別のコード分析ツールを作成する理由」です。



移動する場所と理由を明確に理解するために、潜在的なユーザーと自分自身の両方に回答してみましょう。



C / C ++言語用のPVS-Studioアナライザーを作成し、開発を継続しています。 このアナライザーは、さまざまなタイプのエラーを識別するための多くの興味深いユニークなアイデアを実装しています。 時間が経つにつれて、実装された診断の多くが特定のプログラミング言語に関連付けられていないことが明らかになりました。 使用する言語は関係ありません。 タイプミス、不注意によるエラー、またはコピーペーストの失敗が常にあります。



そして、私たちの経験を別のプログラミング言語であるC＃に適用しようとすることにしました。 これがどれほど成功するか、時間はわかります。 私たちは、多数のC＃開発者から利益を得ることができる非常に興味深いツールを徐々に作成できると考えています。



現在、私たちの仕事は、できるだけ早く潜在的なユーザーからフィードバックを受け取ることです。 PVS-Studioアナライザーのフルバージョンはまだ準備ができていません。 現在、診断はほとんどありません（執筆時点では36個ありました）。 ただし、このバージョンをインストールして試すことはできます。 そして、これを行うすべての人に感謝します。 一般的に正しい方向に進んでおり、分析装置全体が動作していることを確認することが重要です。 そして、新しい診断を非常に迅速に追加します。



したがって、興味のある方は、次のリンクからPVS-Studioの実験バージョンの最新バージョンをダウンロードすることをお勧めします： http : //files.viva64.com/beta/PVS-Studio_setup.exe



ご注意 時間が経つにつれて、上記のリンクは無効になります。 したがって、公開から1か月以上経過してからこの記事を読む場合は、現在の配布バージョンをインストールすることをお勧めします： http : //www.viva64.com/en/pvs-studio-download/



読者がPVS-Studioを試したことがない場合は、記事「 PVS-Studio for Visual C ++ 」を読むことをお勧めします。 ご覧のとおり、C ++に焦点を当てていますが、実際には違いはありません。 インターフェイスの観点から見ると、C ++プロジェクトとC＃プロジェクトのどちらで作業しても違いはほとんどありません。



フィードバックと提案を送信するには、フィードバックページを使用できます 。

フラグメントN15



PVS-Studioアナライザーは、プログラマーがスイッチ（）で何かを行うのを忘れるかもしれない状況を特定しようとしています。 警告するかどうかを決定するロジックは非常に複雑です。 誤検出が得られることもあれば、明らかなエラーが見つかることもあります。 これらのプラスの1つを検討してください。



そのため、コードには次のような列挙があります。 

 public enum TargetArchitecture { I386, AMD64, IA64, ARMv7, }


場所では、このリストのすべてのバリアントが使用されます。 

 TargetArchitecture ReadArchitecture () { var machine = ReadUInt16 (); switch (machine) { case 0x014c: return TargetArchitecture.I386; case 0x8664: return TargetArchitecture.AMD64; case 0x0200: return TargetArchitecture.IA64; case 0x01c4: return TargetArchitecture.ARMv7; } throw new NotSupportedException (); }


しかし、疑わしい場所があります。 たとえば、アナライザーは次のコードに注意を向けました。 

 ushort GetMachine () { switch (module.Architecture) { case TargetArchitecture.I386: return 0x014c; case TargetArchitecture.AMD64: return 0x8664; case TargetArchitecture.IA64: return 0x0200; } throw new NotSupportedException (); }


PVS-Studio警告：V3002 switchステートメントは、「TargetArchitecture」列挙型のすべての値をカバーしていません：ARMv7。 ImageWriter.cs 209



ご覧のとおり、アーキテクチャがARMv7の場合、大文字と小文字は区別されません。 これが間違いかどうかはわかりません。 しかし、これはまさに間違いであるように思えます。 名前ARMv7は列挙の最後にあります。つまり、最後に追加されたことを意味します。 その結果、プログラマはGetMachine（）関数の修正を忘れて、このアーキテクチャを考慮する必要がありました。

おわりに



ご覧のとおり、C＃言語だけでは愚かなエラーを防ぐことはできません。 したがって、私はこの写真をここに明確な良心をもって持ち込むことができます。







ユニコーンは長く生き続けます。ユニコーンは、C＃プログラムのエラーを見つけることを学びました。



しかし、真剣に、それから：
  • プログラミングするとき、私たちは皆、複雑なだけでなく、単純な間違いも犯します。 合計すると、単純なエラーの検索に多くの時間が費やされます。 そして時々 たくさん 。
  • 静的コード分析のツールを使用すると、コードを記述する段階でも、多数の単純なエラーを検出できます。 このようなツールを使用すると、多くの時間を節約でき、多くのエラーの検索とデバッグに費やすことができます。
  • 静的解析で最も重要なことは、定期的な使用です。 静的分析の1回限りのチェックには意味がありません。 その全体的なポイントは、エラーがコードに現れた直後にエラーを見つけることです。 まれなチェックは時間がかかり、ほとんど役に立ちません。 結局のところ、すでにこの瞬間までにすばやく簡単に発見できるこれらの間違いは、汗と血で修正されました。
