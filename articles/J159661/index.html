SemonixのPhilipp Torchinsky：SmartOSをインストールし、Node.jsを使用する

11月29日、サンクトペテルブルクのYandexオフィスで、 科学セミナー 「DTrace-コードの検証作業」を実施します。 YaC 2012の人々は、Illumos Kernel Cloud Computing Infrastructure （および他の多くの人々） に関する私の講演を聞いていました。Semonixでは、SmartOSに密接に関連するクラウドテクノロジーに取り組んでいます。 セミナーでは、DTraceテクノロジーを使用して、詳細なパフォーマンス分析を行い、アプリケーションの動作を詳細に研究する方法を説明します。 Habréで検索するとSmartOSに関する記事が1つしか見つかりません。 そのため、すでにそれをよく知っている人がセミナーに参加するため、SmartOSのインストールとNode.jsの使用について事前に書くことにしました。



まず、SmartOSの目的を説明します。 要するに、その目的は仮想マシンのホストシステムになることです。 JoyentやMITACクラウドサービスなど、パブリッククラウドとプライベートクラウドの基盤としてよく使用されます。 LinkedInはJoyentサービスを使用します。モバイルサーバー部分はすべて、Joyentクラウドで実行されるNode.jsで作成されています。 これについては、SemonixブログのSmartOSに関する記事とillumosに基づいたクラウドで詳細に説明しましたが、すでに述べたYaC 2012の講演で話しました。



SmartOSの典型的なアプリケーションは、データセンターシステム、ロードされたWebアプリケーションを開発および運用するためのシステム、プライベートまたはパブリッククラウドを作成するためのプラットフォームです。 この記事に取り組むとき、私はStu Rudnageの記事を積極的に使用しました。StuRudnageは、私とほぼ同じSmartOSインストールパスに従いましたが、VirtualBoxの代わりにVMwareを使用しました。



テストインストールを計画しています



あなたがSmartOSを試してみたい人の一人だとしましょう。 どうすれば簡単ですか？ 私は、WindowsまたはLinuxが仮想マシンで実行されるのを防ぎ、既製のSmartOS + Node.jsを使用できるようにする、最も簡単な方法で進めました。 VirtualBoxでSmartOSを起動します。 VirtualBoxの仮想マシンでKVMが機能しないことは明らかであり、残りは大金を稼ぐ必要があります。



SmartOSは、Windows、Linux、およびSmartOS自体を実行しているゲストのホストシステムになることができます。 この記事では、SmartOSを実行するゲストシステムのオプションのみを検討します。 軽量仮想化（SolarisのゾーンとFreeBSDのセルに似たものと同じ）を実装しているため、物理コンピューターでのみサポートされているKVMは必要ありません。 構成全体を次の図に示します。 SmartOS（ホスト）は、ラップトップ上のWindowsに関連するゲストであり、このSmartOS内のゾーンで実行されているSmartOSシステムに関連するホストであるシステムです。

2つのネットワークインターフェイスを持つ仮想マシンにSmartOSをインストールします。 なんで？ 非常に簡単：ホストシステム（この場合はWindows）から通常のsshをSmartOSに接続して、SmartOS内のゲストシステムごとに少なくとも1つのウィンドウを開くことができるようにします。



コンソールでグラフィカルモードを使用し、適切な量のxtermまたはgnome-terminalをそこで実行してみませんか？ 実際、グラフィックス（ "X"-X）はSmartOSにインストールされていません。 X11のサポートをSmartOSに移植するプロジェクトがありますが、まだ完了していません。このプロジェクトの主任ボランティアであるJonathan Perkinから、作業の期限が不明であることが通知されました。 したがって、sshからSmartOSに5つのウィンドウを開きたい場合、ホストシステムからVirtualBoxのSmartOSへのTCPアクセスが必要です。



さらに、VirtualBox Guest AdditionsはSmartOSゲストマシンにインストールできないため、ゲストシステムと、記事を書いているホストシステムとの間でカットアンドペーストを行うことができます（端末からテキストをコピーしたい）。 理論的には、Guest Additionsをインストールできると思います（基本的にそこにカーネルモジュールをインストールする必要があります）が、Solarisに似たシステムの場合、Guest Additionsは.pkgパッケージの形式で提供されます。 ホストシステムからsshでアクセスするよりも、パッケージを解凍して手動でモジュールをカーネルにねじ込む方が問題が多いと判断しました。



なぜ2番目のインターフェイスが必要なのですか？ そのため、SmartOSはインターネット上のパケットに対してNATを通過できます。 タスクの条件に関するこの制限は、VirtualBoxによって課せられます。NAT/ DHCPを備えた仮想マシンにホストシステムからアクセスできるようにするには、ブリッジネットワークを介してのみ可能です。これは必ずしも便利ではありません。 あまり便利ではない（または特定のホストシステムやVirtualBoxのバージョンでサポートされていない）ケースの説明は、この記事の範囲を超えていますが、信じてください-これは起こります。 最も単純なオプションは次のとおりです。物理コンピューターが接続されているネットワーク上のDHCPアドレスが不足しているか、sshがフックされる前にシステムにこのアドレスが発行されたたびに確認したくない場合。



VirtualBoxで2つのネットワークインターフェイスを作成する方法



2つのネットワークアダプターがあるように仮想マシンを構成するのは簡単です：（ネットワーク）タブで、最初のアダプターがホストアダプターモードでアクティブであるだけでなく、2番目のアダプターがNATモードでアクティブであることを示す必要があります（下図を参照）。 SmartOSの場合、Solaris 10仮想マシン10/09以降（64ビット）のタイプを選択することに意味があることに注意してください。 Solaris 11オプションの方が優れている場合があります。 問題も遅延もなく機能しました。

SmartOSで2つのネットワークインターフェイスを作成する方法



SmartOSのシステム構成は、/ usbkey / configファイルに保存されます。 / usbkeyファイルシステムは、/ etcとは異なり、ディスクからマウントされます。/etcは、LiveCDまたはLiveUSBから常にロードされ、RAMディスクにマウントされます。 したがって、/などからファイルに加えられた変更は再起動時に失われ、/ usbkeyに保存されます。

ファイル/ usbkey / configは、コンピューターでSmartOSを初めて起動したときに作成され、デフォルトではシステムの最初のインターフェイスのみを記述します。 2番目のものを表示するには、ファイルをわずかに変更する必要があります。 変更するには、インターフェイスのMACアドレスを見つける必要があります。 これを行うには、次のコマンドを使用できます 

 dladm show-phys -m

仮想マシンのイメージをダウンロードします。 

 imgadm import 60a3b1fa-0674-11e2-abf5-cb82934a8e24




次に、仮想マシンの説明を作成します。これは、ロードしたばかりのイメージに基づいて作成し、SmartOSホスト内で実行します。



説明はJSONファイルの形式で作成されます（プロセスの詳細はsmartos.orgのドキュメントで説明されています。ここに実際の例を配置します。ここでは/zones/template.jsonに配置します）。 ファイルはファイルシステムにある必要があります。ファイルシステムはディスク上にあり、再起動の間に保存されます。最も便利なのは/ゾーンです。

dataset_uuidに注意してください-これは、リポジトリから受信したばかりのイメージに属するUUIDと同じです。 ゾーンには、「内部」と「外部」の2つのネットワークインターフェイスもあります。 この設定は、実際のデータセンターで使用されます。内部ネットワークは、データセンター管理者が働く内部インターフェイスに接続され、外部ネットワークは、クライアントの発信元であるインターネットに接続されます。 

 { "brand": "joyent", "dataset_uuid": "60a3b1fa-0674-11e2-abf5-cb82934a8e24", "max_physical_memory": 1024, "nics": [ { "nic_tag": "admin", "ip": "dhcp", "gateway": "dhcp" }, { "nic_tag": "external", "ip": "dhcp", "gateway": "10.0.3.2" } ] }




現在、新しいSmartOS仮想マシンを作成しています（簡潔にするため、引き続き「ゾーン」と呼びます）。 



 vmadm create -f /zones/ template.json




すべてがうまくいった場合、次のコマンドを入力して、起動された新しいゾーンを確認できます。 

 vmadm list UUID TYPE RAM STATE ALIAS 570cccb2-0511-400b-9143-7616b2ca8a3d OS 1024 running -




これで、SmartOSコンソールまたはsshから新しいゾーンに接続できます。 

 zlogin 570cccb2-0511-400b-9143-7616b2ca8a3d


注：同じイメージから多くのゾーンを作成できるため、ゾーンのUUIDはゾーンイメージのUUIDとは異なります。 同時に、1つのSmartOSホストシステムで最大8191台の仮想マシンを起動できます。



sshを介してゾーンに接続するには、どのアドレスが割り当てられているかを明確にする必要があります;このためには、まずzloginを介して参加し、次にアドレスを見つける必要があります。 

 zlogin 570cccb2-0511-400b-9143-7616b2ca8a3d ipadm show-addr

Node.jsでDTraceを使用する方法は？



ノードのDTraceプロバイダーは（サプライズ！）ノードと呼ばれます。 これはUSDTタイプのプロバイダーであるため、センサーはnode * ::: <センサー名>の形式で指定するか、 node3297 ::: <センサー名>などのノードプロセスの特定のPIDで指定する必要があります。



Node.jsでDTraceを使用してサーバーの動作を監視する実際の例は、呼び出しを記録することです。 さまざまなゾーンからのサーバーが受信する要求を見つけるには（ホスティングプロバイダーの一般的なケース）、短いスクリプトを実行します。 

 # dtrace -L /var/lib/dtrace -n 'node*:::http-server-request{printf("%s: %s of %s\n",zonename, args[0]->method, args[0]->url)}' -q 570cccb2-0511-400b-9143-7616b2ca8a3d: GET of / 570cccb2-0511-400b-9143-7616b2ca8a3d: GET of /favicon.ico 570cccb2-0511-400b-9143-7616b2ca8a3d: GET of / 570cccb2-0511-400b-9143-7616b2ca8a3d: GET of /favicon.ico 570cccb2-0511-400b-9143-7616b2ca8a3d: GET of / 570cccb2-0511-400b-9143-7616b2ca8a3d: GET of /favicon.ico


リクエストの送信元アドレスを確認するには、スクリプトをわずかに変更できます。 

 # dtrace -L /var/lib/dtrace -n 'node*:::http-server-request{printf("%s: %s of %s\n",args[1]->remoteAddress, args[0]->method, args[0]->url)}' -q 192.168.56.1: GET of / 192.168.56.1: GET of / 192.168.56.1: GET of /favicon.ico 192.168.56.1: GET of / 192.168.56.1: GET of /favicon.ico 192.168.56.1: GET of / 192.168.56.1: GET of /

おわりに



SmartOS設定の特別な魅力は、一度インストールすると、更新後に繰り返す必要がないことです。 最新のlatest.isoをダウンロードし、以前のものと置き換えて、新しいものから起動できます。 すべての設定、仮想マシン、およびその中のファイルは自動的に取得されます。これは、誰も何も変更しないためです。ダウンロードされたイメージのみが変更され、設定はそれとは別に存在します。
