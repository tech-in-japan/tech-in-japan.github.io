IOMeter Load Generatorを使用したNetApp FASストレージテスト

ストレージシステムのテスト中、専門家は、複雑なシステム全体のチューニングの微妙な違いにより、パフォーマンスの問題に遭遇することがよくあります。 ストレージシステムの負荷テストを行うには、負荷シミュレータを起動するだけでは不十分です。ストレージシステム、ネットワーク、ホストを構成する必要があります。 通常、設定の大部分はホストで実行されます。



この記事の目的は、NetApp ストレージシステムをテストする際に、システム管理者とシステムインテグレーターが設定とニュアンスを理解できるようにすることです。 また、この記事では、NetApp FAS 2240コントローラーの最大限のパフォーマンスを得るのに問題がなかったことにも注意してください。デモ機器のテスト時には、 ストレージに十分なディスクがなく（最大144）、サーバーは1つしかなかったためです。 つまり ここでの最大パフォーマンスは、デモスタンドの特定の構成、つまり、より多くのディスク、サーバー、およびネットワーク接続のコンテキストで考慮されるため、同じ2240コントローラーでより良い結果を得ることができます。 2240コントローラー、一部のテストでは4つのSSD 100GB MLCドライブを使用しました。



以下は、 ストレージシステムの負荷テストの包括的な説明です。IOMeter負荷シミュレータの設定、ホストとNetApp FAS ストレージシステムのセットアップ方法、負荷統計の削除、および結果の解釈。 既存のデモスタンドの例を使用して、すべてのコンポーネントを構成する方法と、NetApp FASを使用する他の可能な構成オプションを検討します。

SHD


すべてのディスクは、コントローラー間で均等に分割されます：12対1および12対1。 各ディスクセットで、1つのaggr0アグリゲートが作成され、各アグリゲートで1つのvol0システムパーティションと1つのvol1パーティションがアグリゲートのすべての空きスペースを占有し、1つのムーンが各パーティションで作成され、利用可能なすべての空きスペースを占有します。 原則として、最長のRAIDグループから可能な限り最大の集約を作成することをお勧めします。 月を作成するときに、 LUNが提示されるホストに対応するジオメトリが選択されました。



SANネットワーク



無料のBrocade SANHealthユーティリティを使用してSANスイッチの監視とトラブルシューティングを行うと便利です。 光ファイバコネクタが詰まっていると、パフォーマンスの問題が発生する可能性があります。 実生活から：リンクが壊れているため、ラウンドロビンポリシーを有効にしたマルチパスドライバーは、 HBAアダプターに巨大なキューを作成しました。これにより、I / Oシステムで大きな遅延が発生し、 CPU使用率が高くなりました。

また、 FCリンク自体、たとえば4Gは物理的に400 MB / sを超えないことを忘れないでください。 より多くのトラフィックを生成すると、ホストのCPU使用率が増加することにも気付くでしょう。 FC 1/2/4/8 Gでは8b / 10bコーディングスキームが使用されていることを考慮して、リンクの帯域幅を計算するのは簡単です。 オーバーヘッドで20％を失います。 FC 16Gは、より経済的な64b / 66bコーディングスキームを使用します。 以下は、FC 4Gリンクの最大スループットを計算する例です。

4Gbit = 500 MByte; 500 Mバイト* 8/10 =そのようなチャネルの理論上の最大帯域幅400 Mバイト。



Oncommand Core + Performance Adviser


負荷テストを実行しないOncommand Coreホストのグラフ形式で表示することにより、統計を収集するためのユーティリティセットをインストールすることをお勧めします。 support.netapp.comからDownloads Softwareでダウンロードできます（NetApp NOW IDが必要です）。 NetApp管理コンソールに入るためのログインとパスワードは、ホストのログインとパスワードに対応しています。

NetApp ストレージユーザーからの無料ライセンスが必要です。 FASシステムを監視します。 インストール中に、「 Express 」ではなく「 Standart Edition 」を選択して、統計の収集とグラフ作成を有効にします。

両方のLUNに与えられたRDM上のゲストOS

NetApp Host Utilities 6.0.2はWindowsにインストールされます。

場合によっては、 VMFSのパフォーマンスはRDMよりも優れています。 そのため、テキストで説明されている負荷パターンを使用したテストでは、IOmeterはVMFSを使用してほぼ300 MByte / secを示し（テキストの上部の図を参照）、 RDMを使用して約200 MByte / secを示しました。

Vorker（負荷を生成するプロセス）は、IOMeter GUIと同じマシンと、ボットネットネットワークに似た同じインターフェースから管理される他のマシン（LinuxやSolarisなどの異種マシンを含む）の両方で実行できます。



ロードの最後に、IOMeterはCSVファイルの形式でレポートを提供します。 しかし、実際には、このレポートはあまり信頼すべきではないことが示されています。 IOMeterは負荷の生成には適していますが、負荷統計の収集には適していません。



IOMeterが起動し、2つのワーカーが作成され、それぞれに1つのLUNが割り当てられます。







  • ブロック8K
  • 読み取り67％/書き込み33％
  • 100％ランダム
  • I / Oを常に4Kに揃える必要があります
  • 優れたI / O 128




I / Oの調整は常に4Kである必要があります。これは、最新のOSおよびストレージの場合に最適です。



未処理のI / Oの最大256、パラメータの数値128は、高負荷のデータベースと同様の高負荷を意味します 。 2つのワーカーを作成し、1つのLUNを1つ、2番目の秒を1つ割り当てます。





テストファイルの最大サイズ「Maximum Disk Size」を設定します（0はすべての空きディスク領域を意味します）。値はセクターで設定されます。たとえば、204800セクター（1セクターは512バイトです。 そして、「Write IO Data Patern」フィールドで負荷パターンを設定します。

DataOntap 8.1.4 7-Modeから8.2にアップグレードし、ホスト上のHBA FirmWareを更新する必要がありました。 また、ESXi環境でハードディスク（ RDM LUN ） のタイムアウトを設定する必要があります。

この項目を無視しないでください。 一連のユーティリティにより、正しい遅延、 HBAのキューサイズ、およびホスト上のその他の設定が設定されます。 接続されたLUNとその詳細情報をストレージ側から表示します 。 ユーティリティのセットは無料で、netapテクニカルサポートサイトからダウンロードできます。 インストール後、ユーティリティを実行します

MBR


Windows 2008では、 LUNを作成するときに、「Windows 2008」ディスクジオメトリを選択する必要があります。

MBRテーブルでディスクを初期化する場合、IOMeterでのテスト中にフォーマットせずにファイルシステムまたはファイルシステムを作成せずに、ブロックデバイスとして使用できます。



GPT


GPTを使用してWindows 2003用の LUN を作成する場合、「Windows GPT 」ディスクジオメトリを選択する必要があります。その上にファイルシステムを作成し、テストファイルを作成してロードします（ファイルの作成には時間がかかる場合があります）。このテストでは、ファイルシステムレイヤーが追加されます

これまでのところ、最新バージョン1.1.0のブロックデバイスとしてIOmetrでサポートされていません。

IOMeterは、 GPTとして初期化されたブロックデバイスではまだ動作しません。 したがって、2つの方法があります。

  • netapでLUNを作成するときに、2TBを超えてはならないMBRでブロックアクセスを使用し、「Windows 2008」を選択して、ホスト上のそのような衛星をMBRとして初期化します。
  • GPTディスクをフォーマットし、ファイルシステムで使用します。 ブロックデバイスの上部のテストはGPTでは機能しません

ESXiのパフォーマンス


ESXiホストの負荷統計情報を含むグラフは、通常、MS perfmonと同じです。 個別のインストールと構成を必要としないため、生成された負荷の初期評価に非常に便利です。 ホストがコマンドを使用して動作するブロックのサイズに関する統計を収集するための便利な機能がコマンドラインから利用可能です： 

 vscsiStats -s -w YYYYY -i XXXX
ここで、YYYYYは仮想マシンのwouldGroupIDで、XXXXはその仮想SCSIディスクのhandleIDです。 詳細については、この記事をご覧ください 。

また、次のコマンドを使用して、I / Oおよびラテンのディスクサブシステムの負荷を収集できます。 

 esxtop -b -a -d %SEC% -n %COUNT% | gzip -9c > /root/esxtop.out.gz
ここで、 ％SEC％は測定間の秒数、 ％COUNT％は測定の数です。

VMware Webサイトで結果のログを分析するためにESXplotユーティリティをダウンロードできます。これは、組み込みのWindows perfmonユーティリティを含むいくつかの形式に出力を変換でき、非常に便利です。
