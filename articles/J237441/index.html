履歴：Linux用QuakeLiveネイティブブートローダー





id SoftwareがLinuxおよびMacのサポートを停止し、Launcher for QuakeLiveをクロスプラットフォームフレームワーク（！）を使用してC＃で記述されたスタンドアロンアプリケーションにしました（ OpenGLで Webページをレンダリングするため）。 安定した最大数のfpsのためにdistrを散らかしました（私はまだ弱いビデオカードを持っています）が、それでも遅くなりました。 今、私は私の自由な時間に私のお気に入りのゲームをプレイするために私が何をしたかをお話します。このゲームは、7歳のときにコンピューターに座って始めました。 私はこれまで人々にそのような吐き気をすることに無関心なままでいません。 ネコの下で、Linuxでのゲームのパフォーマンスを改善する方法と、独自のランチャーを作成した方法について説明します。



1. OpenGLの最適化



私はNVIDIAグラフィックスカードを持っているので、ここにあるものはすべてNVIDIA用です。 これらの設定は具体的なパフォーマンスの向上をもたらすと言わなければなりません。



「画面」セクションでxorg.confに追加します（そうでない場合は、nvidia-settingsユーティリティで作成できます：



オプション「NoFlip」「True」

オプション「NoRenderExtension」「False」

オプション「NoLogo」

オプション「AllowGLXWithComposite」「true」

オプション「RenderAccel」「true」

オプション「UseEvents」「false」

オプション「TripleBuffer」「false」

オプション "RegistryDwords" "OGL_MaxFramesAllowed = 0x0; EnableBrightnessControl = 1 "




具体的な設定を行う主な設定についてのみ説明します-これはTripleBuffer = falseおよびOGL_MaxFramesAllowed = 0x0です。 1つ目は、名前が示すとおり、トリプルバッファリングを無効にし、2つ目は、ビデオカードの画像全体がレンダリングされた後にレンダリングする必要があるフレームの数を設定します（たとえば、60ヘルツモニターに対して60フレームがレンダリングされます）。 これらの設定の主な目的は、InputLagを減らし、生産性を高めることです。 多くのレビューがこの設定についてでした。 ここを読むこと 、および開発についてはここを読むことをお勧めします 。



また、ゲームでは、 後処理を完全にオフにし、 r_vertexlight = 1を有効にする必要があります。



2.プロセスの優先順位



遅れがまだ残っていたので、私はより深く掘り始めました。 誰かが、reniceを介してプロセスの優先順位を上げると役立つと考えたかもしれません。 その試みを見ていないとすぐに-最小（20）ワインサーバーに、最大優先度に-反対に、すべて一緒にゲームを減らします。 私を信じて、これは結果を与えません。



3.自分のLauncher.exe



群れはさらに深くなっています。 私は簡単な結論を下しましたps ax | grep quakeで 、いわゆるawesomium_process.exeを見ました。 このプロセスについては説明しませんが、quakelive.exeが組み込みのWebブラウザー用にそれを生成するというだけです。 問題は（私が信じているように）awesomiumは常にロギングで動作するため、30秒ごとに0.5秒の絶対的な何らかのラグが絶対にあり、それに加えてゲームは常にFPSが少ない。 最初は私は彼を殺そうとしましたが、数秒後にquakelive.exeが再びそれを生成します。 それから私はそれを止めようとしました（ kill -stop pid ）-そしてすべてが時計仕掛けのようになりました。 もちろん、2つのモニターがあり、コンソールは常に開いていますが、これは常に自分の手でこれを行うことに煩わされました。 私は、MacやLinuxをプレイしているかなり多くの人々を知っているという事実について考えました。



そしてここで、VinesなしでネイティブのLauncherを作成することにしました。







言語（Python）が選ばれたのは、多くの解析と文字列を処理する必要があることに気付いたからです。 まあ、一般的に、これはすべてより速く起こります。



最初にquakeliveの仕組みを分析しましたが、サードパーティの支援が必要です（ただし、約5％であり、廃止されました（APIが変更されました））。 私のお気に入りのWiresharkと、驚くべきことに、 FireBugがこの問題を助けてくれました。 後者の助けを借りて、quakelive.comでブラウザがどのように、何をするか（その機能が削除されました）を調べました。 しかし、結局のところ、機能の80％はスタンドアロンクライアントに隠されており、これはexeです！ この場合、Wiresharkが役立ちました。



最初に、 netstatユーティリティを使用してquakelive.exeが何をするか、クライアントがアクセスしているIPを調べました。 その後、これらのIPをWiresharkに送り込み、パケット分析を使用して、ほとんどの場合、これはサイトのようにquakelive.com:80に過ぎないことに気付きました。 さて、ここではすべてが簡単です-連続してすべてを注意深く分析し、それを偽造します。 現時点では、ほとんどすべてを偽造しており、quakeliveクライアント全体がどのように機能するかを知っています。 ただし、興味深い点が1つあります。チャットです。



私はすぐにそれがXMPPであると推測しました。 私はLinuxの友人（そのうちの1人はピックアップゲームの1つのircチャネルの管理者です）に尋ねました。彼はそれが本当にあり、それがどのように機能するかを「読む」ためのリンクさえ与えました（誰かが小さなアルゴリズムを書いた独自の言葉で、quakeliveチャットのクライアントを作成する方法）。 これは、実際にはquakelive.comドメインの通常のxmppアカウントです。 しかし、問題があります-チャットとゲームの両方が接続されています。友人がいる場所（サーバー）やその他のトリック（名簿、友人として招待）などに関する通知-すべてがXMPPを使用し、すべてのデータが暗号化されます。 Wiresharkでは表示されません。 私はより簡単に行動しました-私は自分のpythonクライアントを介してquakelive.com xmppサーバーに接続し、私に来たネットワーク上のすべてのデータを分析しました。 次に、お気に入りのqutIMを介してリソースに接続し、すべてが思い通りに機能することをもう一度確認しました。



少しオフトピックで続き


ランチャーの目標は、ゲームWebブラウザーで使用されていたクライアント（およびawesomium_process.exe遅延プロセス）を分離することでした。 問題は、XMPPサーバーで認証に必ず合格する必要があるため、ゲームなしでゲームを開始すること（およびサーバーにアクセスすること）が不可能だったことです。 このプロセスがなくてもサーバー上でゲーム自体を認証できるのか、頭脳で疑問に思ったのですが、正しいコマンドと数日間の助けを借りてこれを達成しました-Webブラウザなしでquakelive.exeを実行し、サーバーにアクセスしてfps awesomium_process.exeの場合のように、着実に最大に維持され、痙攣はありませんでした。 これらの開発は、私のランチャーにも含まれています。







まとめ

Pythonで記述されたすべてのdebベースのディストリビューション用のアプリケーションがあり、可能な限り最高のパフォーマンスでゲームを実行できます。



さらに、私は世界のさまざまな地域でいくつかの連絡先を作成しました。そのうちの1人は有名なQLPrismの著者で、彼は他の人と同じように私の自由時間に私のアプリケーションをテストしています。



最後に、プログラミングの考え方がなければ（誰もが助け合うときです）、少なくとも何らかの形で助けてくれたすべての人に感謝します。



クライアントにできること：

1.ゲームの自動更新。

2.サーバーへの呼び出しですぐに起動します（リンクを押してEnterキーを押します）

3.サーバーブラウザ

4.独自のサーバーの起動（プロアカウントがある場合）



インストール： 

sudo add-apt-repository ppa:broken/ppa sudo apt-get update sudo apt-get install qllauncher
