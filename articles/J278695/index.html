フォーミュラ計算



ユニットの変換は難しい作業です。 第一に、多くの単位があり、すべての係数と変換と間違えられないことは困難です。 第二に、いくつかの変換は自明ではありません。 フィートからメートルに変換する際に、ある係数で除算してから別の係数で乗算する必要がある場合、たとえば華氏を摂氏に変換するには、もう少し試す必要があります。



数式の計算は簡単な作業ではありません。 なんらかの方法で記述し、何らかの方法で解析し、何らかの方法で変数を置換する必要があります。 幸いなことに、研究の過程で、 NSExpression



の副次的性質について説明する記事にNSExpression



。これにより、いくつかの算術式を計算できます。 次のように機能します。 



 [[NSExpression expressionWithFormat:@"(23-7.5)*40.0/21.0+273.15"] expressionValueWithObject:nil context:nil]


また、（例のように）単純なものを計算したり、 ドキュメントにリストされている関数を使用して、もう少し複雑な計算を行うことができます。



NSNumber



およびNSExpression

の動作を決定することを忘れないでください。 そうしないと、計算が中断します。 これは、次のコードで実行されます。 



 [NSDecimalNumber setDefaultBehavior:,   NSDecimalNumberHandler]

ユニット付きのストレージテーブル



ユニットのリスト自体も簡単に保存できません。 結局のところ、次のものが必要です。



  • それぞれの係数または式を理解します。
  • 特定のケースごとに最も適切なユニットを選択できるように、優先順位を保存します。
  • このために、単位測定システムのタイプが保存されます（たとえば、SIまたはインペリアル）。
  • ユニット名の翻訳は、各言語および可能なすべての単語形式（ロシア語の場合、5つの単語形式など）に対して行われます。 これには、ユニットの可能なシンボル、名前の同義語（arまたはweavingまたはsquare decameter）も含まれます。
  • 一部のユニットは「クラスター」に結合されます。たとえば、1メートルはフィートに、10メートルはヤードに変換されます。 2番目のユニットはクラスターから選択されます。
  • 最後に、メニューに正しく表示するために、ユニット自体をカテゴリに結合し、ユニットをフォーマット/変換するときにこの情報を考慮する必要があります。


最初は、ユニットをテキストファイルの形式（編集しやすい）で、基本情報、各言語ファイルのローカライズごとに個別に保存します。 これが速度のファイルです（無料） 



 knot kn,kt 0.514444 3 impAdd #   11  -      —  295 /  1062 /. Mach M 295.0464 2 other speed of light c 299792458 0.11 siAdd meter per minute m/min 60 0 siAdd2 centimeter per second cm/s 100 0 siAdd2 ^minutes per kilometer min/km FORMULAE(16.666666667/X,16.666666667/X) 0 other ^minutes per mile min/mi FORMULAE(26.805555556/X,26.805555556/X) 0 other


それらから、もともと作業に使用されていたJSONファイルを取得します。 残念ながら、これは柔軟性がなく、十分に高速ではありませんでした。 したがって、すべてのデータがSQLiteデータベースにパックされ、必要に応じてそこから読み取られます。 データベース形式は、次のようなJSONファイルの構造をほぼ繰り返します。 



 [ { "fml": "", "abbrs": [ "m\/s" ], "us": "si", "id": 1, "tag": "meter per second", "pri": 3, "to": 2000002, "cof": 1, "names": [ "meter per second" ] }, ... ]


メインデータを含むベースに加えて、検索ツリーも必要です。 Angstromは2つのモードで動作します：



  • 「数」、「時間の単位」、「2つの単位」の3行がある場合のアプリケーション内の翻訳。 この場合、行で2つの単位を見つけて、結果を計算する必要があります。 認識は通常どおり行われています。 文字が存在するノード内の部分文字列でツリーを実行し、各ノードにユニットのリストを添付します。 目的のノードに移動し、ユニットのリストを取得し、オプションを表示しました（または、必要な場合は最初のノードを取得しました）。
  • 改行。 例えば、私たちは何かを口述で（口頭で、または標準の口述を使って）口述するか、別のアプリケーションからコピーしました。 この場合、手順はおよそ次のとおりです。

    • 文字列を単語に分割します
    • 不必要なものをすべて削除し、準備し、魔法の最初の部分を実行します（たとえば、システムはこれを「秒」としてではなく、通常は正しくない「秒」として認識するため、秒を置き換えます）。
    • パーシム番号。 ここでは、マジックの2番目の部分が適用されます。 実際には、 NSNumberFormatter



      は文字列をNSNumberFormatter



      として解析できるため、たとえば「34」と認識できる「34」です。 これは、数字だけでなく、この数字を選択する必要がある行があるため、正しく使用するのが非常に難しいスーパー機能です。 可能な最大数を解析するために、すべての拡張範囲を使用して、単語に従って実行する必要があります。
    • 残りの単語を1つまたは2つの単位に解析しようとします。 ここでは、ユーザーが好きなように好きな順番で話すことができるので、魔法が一番です。 特定のマーカーワードに正しく応答する一連のヒューリスティックを作成する必要があります。 ここでの基本は、通常のバージョンと同じユニットツリーでの検索です。


これらのモードは両方ともユニットツリーを使用します。 たとえば、SQLiteからの標準のテキスト検索を使用できますが、トークンと設定をいじる必要があるため、自分で作成することにしました。 複雑さはあちこちで似ていますが、自分の場合は最適化の機会が増えます。

最適化



最適化は、常にアプリケーション開発者が直面する問題です。 技術開発の速度により、単にそれを叩くか、「単純なこと」を行うことができますが、それで十分です。 Angstromは、たとえばApple Watchや古いiPhone 4（iOS 7は引き続きサポートされています）など、非常に極端な条件で使用する必要があります。 これらのデバイスでは、メモリがほとんどなく、プロセッサが比較的低速です。 したがって、すべてを最適化する必要があります。同時に、将来的にはさらに10倍の異なるユニットが存在する可能性があることを忘れないでください（現在は約1050個あります）。



最適化には3つの主要なポイントがあります。



  • アプリケーションの開始。 開始をデバッグするために、Instrumentsを使用し、開始から可能なすべてを破棄します。 すべての更新-開始から数秒後に開始します。 画面上のもの以外のユニットのダウンロードはありません。重いリソースはありません。
  • 検索ユニット。 これを行うために、検索ツリーを作成し、情報を文字ごとに別のファイルに分割しました。 私たちは手紙を入力しました-この手紙に関するファイルだけをダウンロードしました。 ファイル自体には、情報が非常にコンパクトな形式（識別子、単純なリスト）で保存されます。
  • メモリ使用量。 ここでのテクニックは、メインメモリコンシューマがユニット単位のベースであるため、前の段落と似ています。 SQLiteは、ファイルの断片化された検索ツリーとともに、問題をうまく解決します。


Angstromの開発の前でさえ、あるリソースで非常に制限されたタスクがある場合、それは非常に便利であることがわかりました。 たとえば、アプリケーションがApple Watchで動作するためには、速度を最適化する必要があります。最初のバージョンの時計は非常に遅く、自然なテキストを解析する必要があります。これにはかなりの時間がかかります。 また、バージョン1.8では、解析が一度に複数の言語で行われるため、インターフェースが英語であっても、ロシア語で口述することができます（口述自体は現在どの言語が使用されているかを示しません）。 このような「不良」デバイスの最適化により、残りのパフォーマンスが向上し、より近代的かつ高速になります。



Today Extensionを作成するには（バグが発生し、うまく機能しないため、現在は無効になっています）、厳しいメモリ最適化が必要でした。 私は彼にクリップボードの行を解析できるようにしたかったのです（数行だけを表示するのではなく）。これには本格的なアプリケーションが必要でした。 面白いのは、Appleがこのタイプの拡張機能で使用可能なメモリ量について話していることです。 「十分ではない」と彼らは言う。 どれだけ足りないのですか？ 「小さいほど良い！」具体的な数字はありません。 したがって、メモリによる最適化-限界まで。



これはすべて、最適化の要件を極限まで高めます。 すべての既知の手法を適用し、新しいデータ構造を作成し（より正確には、忘れられた古いものを使用）、杖を並行して突いて、インストルメントが示すものを注意深く見て、スローダウンするアルゴリズムを時々見て、より複雑になりますが、より効果的。
