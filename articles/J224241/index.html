新しいISPmanager 5の導入方法

この記事では、ISPmanager 5コントロールパネルの存在の最初の日からどのように実装したか、どのような困難に直面し、最終的にどのような結果を達成したかを説明します。



関税計画の大幅な改訂を長い間計画してきました。 私たちは、多くの質的な変更を行い、不要なオプションを放棄し、半実用的で時代遅れのテクノロジーを本当に望んでいました。



そこで、新しいハードウェアプラットフォームに切り替え、仮想サーバーのリソースを大幅に増やし、仮想化を管理するためのソフトウェアプラットフォームと、クライアントが仮想サーバーで受け取るソフトウェアを更新できるようにしました。



Runetで最も人気のあるISPmanagerコントロールパネルの新しいバージョンの今後のリリースについて知っていたため、関税の更新を最大3か月遅らせる必要がありました。 最後に、11月末に、パートナーのISPsystemがISPmanager 5のベータ版をリリースしました。正直なところ、私たちは完全に満足することはできませんでした。



2013年12月5、13日にISPmanagerがプリインストールされたテンプレートの短いテストの後、ついに離陸しました！ やった！







この6か月間「飛ぶ」方法



12月 ゲイン高さ

クライアントは新しい料金でサーバーを注文し始め、新しいパネルで質問をし始めました。



キャプテンはシートベルト信号を点灯しました

ISPmanager 5には、ISPmanager 4にあった通常の機能がなかったという事実に直面しました。たとえば：

  • ドメインミキシング機能
  • デフォルトドメインを設定する機能
  • ユーザーのインポートなど


しかし、銀の裏地はありません。 新しい機能に感謝します。

  • nginx + php-fpmを使用する機能
  • ファイルマネージャーのエディター（図1）
  • 新しいより快適で美しいテーマ
  • パネルなどからソフトウェアを更新する機能

バグ報告がなければ、物事は成し得なかったでしょう。 Bugzillaおよびチケットシステムを通じて報告されたエラー。



メールシステムの動作には多くの欠点がありました。たとえば、EximとCentOSのpostgreyを使用したパネルのカスタムインストールでは、ユーザーを作成できませんでした。 アルゴリズムは次のようなものでした：

ユーザーを作成し、同時にメールドメインを作成すると、パネルはpostgreyファイルを見つけられず、ユーザーはパネルに作成されませんでしたが、メールドメインなしでそのようなユーザーを作成することは不可能でした。



バックアップのロジックも少し驚いた。 すべてのWebドメイン、メールドメイン、および一般的にすべての要素は、個別のアーカイブに作成されます。 一方で、それは後で便利かつ迅速に解凍されますが、このヒープ内で何かを手動で見つけることは非常に困難です。



DNSの動作にエラーがありました。 最初は、ドメインを作成するときに、任意のネームサーバーを指定することはできませんでしたが、後にこの機能が追加されました。 ただし、domain.comなどのドメインを作成するときに、同じドメインのゾーン（ns1.domain.com、ns2.domain.com）のネームサーバーを指定すると、誤ったドメインゾーンが作成されます。 また、ドメインゾーンが正しくないため、パネルからこのようなドメインのゾーンを編集することはできなくなりました。

エラーが修正されるまで、ゾーンファイルを手動で変更する必要がありました。



4月-5月。 通常の飛行

現在までに、上記の問題とその他のエラーの両方が修正されています。 すべての最悪は終わった。



デブリーフィング



6か月後、ISPmanager 5で新しい関税が開始された日から、調査を実施することが決定されました。 ISPmanager 4を放棄しようとする際、調査の最初の目標は、ISPmanager 5が選択する顧客の割合とISPmanager 4を好む顧客の数を調べることでした。

この調査の2番目の目的は、技術革新がサポートサービスの負荷の増加を引き起こしたかどうかを判断することでした。



この調査には、2013年12月13日までに注文された仮想サーバーが含まれていました。 ISPmanager 5のディスクテンプレートを使用して注文された仮想マシンの数と、ISPmanager 4のディスクテンプレートを使用して注文された仮想マシンの数が計算されました。

どこで：

A-ISPmanager 5（ISPmanager 4）を持つ仮想サーバーの数

B-パネルで注文されたサーバーの総数



この研究の結果を図1に示します。



図1-ISPmanager 5およびISPmanager 4を使用した仮想サーバーの注文数の割合



図から、今日のクライアントの約80％がISPmanager 5で仮想サーバーを選択していることがわかります。ISPmanager4のシェアは約20％にすぎません。



調査の第2部では、「一人当たりのチケット」を計算しました。 計算は次の式に従って実行されました： C / D

どこで：

C-ISPmanager 5（ISPmanager 4）を持つVDS所有者からのチケットの数

D-ISPmanager 5（ISPmanager 4）でアクティブなVDSの数



初めてのことですが、調査には仮想サーバーが関係しており、仮想サーバーの注文日は2013年12月13日より前ではありません。 結果を図2に示します。





図2-アクティブなサービスの数に対するチケットの数の比率



チャートが下がっている理由を説明します。 12月13日以降に注文された仮想マシンのみが考慮されました。 したがって、毎月仮想マシンが増えていますが、毎月チケットはありません。 グラフは、ISPmanager 5を持つサーバーの所有者がサポートに連絡する可能性が高いことを示しています。 しかし、時間の経過とともに、ISPmanager 5の曲線はISPmanager 4の曲線に近づきます。これは、サポートの負荷が正規化されることを意味します。



ISPmanager 5での個人的な経験に基づいて、私はそれがよりうまく機能したと言うことができます、すなわち：メッセージは明確で有益であり、「デッドロック」状況が少なく、問題状況に対する自動解決策があり、インターフェースは私たち、管理者、およびクライアントの両方にとってより理解しやすい、更新のリリースはほとんど目に見えません。いずれの場合も、顧客に問題を引き起こさず、彼らの側で質問の急増を引き起こしません。



残りの重大な欠点のうち、バージョン4から5への移行メカニズムはまだ完全に開発されていないか、そうではありませんが、すべての状況に適しているわけではありません。 既存のサーバーでパネルの新しいバージョンにアップグレードできますが、OSの古いバージョンがある場合は、ISPmanager 5アセンブリはLinuxおよびFreeBSDディストリビューションの最新バージョンのみであるため、最初に更新する必要があります。 非標準の設定、プラグイン、その他の「ドピルキ」がたくさんある場合、コンバーターは完全には機能せず、多くの手作業が必要になります。 古いバージョンのパネルを備えたサーバーから新しいサーバーにアカウントを移行することは非常に不足しています。なぜなら、最新のオペレーティングシステムとソフトウェアを備えた新しいサーバーを注文し、それを構成し、サイトを単純に移行する方が簡単だからです。



要約すると、新しいサーバーにはISPmanager 5をお勧めします。古いバージョンのパネルから新しいバージョンのサーバーにアカウントを転送する問題が解決したら、ISPmanager 4の販売を完全に中止する予定です。
