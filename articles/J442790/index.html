サンクトペテルブルクのAllure Server Meetupの登録は受け付けています





3月21日、Wrikeのサンクトペテルブルクオフィスで開催されるAllure Server Meetupに招待します。

各テスト部門は、スピードとスピードでスピードを上げることに取り組んでいます。 Wrikeでは、多数の自動テストを作成し、可能な限り迅速に実行すると同時に、サポートとドキュメントの作成に最小限の時間を費やすよう努めています。 自動化フレームワークとツールに関して、長い間qameta.ioと協力してきました。 そのため、1年以上にわたって、新しいTMSシステムであるAllure.Serverを実行してきました。これは、テスト文書の作成と自動テストの管理に関するアイデアにぴったりです。 Allure.Serverミーティングでは、このシステム、このシステム上での使用パターンおよび技術的な実装について詳しく説明します。



プログラム内：



Anton Bashkirov-アリュールサーバー：Wrikeのテストケース管理の変革



Allure Serverを使用するWrikeで、「セルフテストからのドキュメント化」の原則に基づいてテストドキュメントを作成するプロセスをどのように編成したかについてお話します



通常のドキュメント作成プロセスは次のようになります。QAマニュアルはチェックリストを作成→テストケースを作成→自動化のためにそれらを提供→自動テストの変更に応じてドキュメントをサポート。 QAとQAAを共通のチームの作業フローに統合し、統一されたドキュメントシステムに移行することで、このチェーンのコストを簡素化および削減する方法を見つけました。



すべてのテスト成果物を製品に関するドキュメントとして使用することを学び、自動テストコードの注釈を積極的に使用して、サーバーアリュールデータベース内の対応する属性に関連付けます。 。



Ivan Varivoda-検疫テスト、または10Kセレンテストで狂わない方法



残念ながら、自動テストの一部が一時的に正しく動作しなくなる状況は、テストの自動化では珍しいことではありません。 おそらく、これらはテストフラスコであるか、そのパフォーマンスがインフラストラクチャの問題またはバグの影響を受けていたためです。



プロジェクトに含まれるテストの数が少なく、頻繁に追跡されない場合-特別な問題はありませんが、テストの数と毎日の開始の増加に伴い、破損したテストをできるだけ早くオフにし、「オフ」テストを制御できることが非常に必要になります。



テストをすぐにオフにする方法、テスト検疫とは何か、なぜ必要なのか、Wrikeでどのように機能するのか、Allure Serverがそれとどう関係するのかについて説明します。



ディスカッション：Artyom Eroshenko、Mikhail Levin



聴衆からの質問に答える対話の形式で、テスト管理の開発の問題と方向、Qametaとの相互作用、およびAllureサーバーのさらなる計画について説明します。



3月21日から19:00にWrikeオフィスでお待ちしています。
