GitLab 10.2リリース：カスタマイズ可能なタスクボードとGitLab Geo Shared







このリリースでは、計画、展開、信頼性などを改善する機能を追加しました。









作業計画の改善



私の意見では、GitLabのタスクは水と多くの共通点があります。それらは生活に必要ですが、大量にdrれる可能性があります。







一般的な概要のあるチームワークでは、特定のコンテキストで重要なタスクのみに絞り込むことが有用です。 以前、GitLabではフィルターを適用して、特定のマイルストーンまたはタスクボードラベルに関連付けられたタスクを表示できましたが、これは一時的な解決策にすぎませんでした。 おそらく、あなたはブックマークを付けて、同僚のリンクをタスクボードに送信する必要がありました..







このリリースでは、マイルストーン、ラベル、ウェイト、タスク実行者ごとに選択を保存できるタスクボード設定を導入します 。これにより、各チームメンバーは同じタスクを見ることができます。







高速化されたフェッチ



開発チームはますます地理的に分散されています...これはGitが非常に人気がある理由の1つです。それは本質的に分散されています-ローカルリポジトリにはプロジェクト履歴の各コミット、ファイル、ブランチのコピーが含まれています。 ダウンロード可能な履歴は、開発プロセスを大幅にスピードアップします。







ただし、リポジトリの物理インスタンスが1つしかない場合は、分散コマンドから離れた場所に配置される可能性があります。 これにより、多数の小さなファイルのダウンロードが遅くなります。 このリリースでGitLab Geoが一般公開されたことをお知らせいたします。 GitLab Geoを使用すると、リモートコマンドに物理的に近接して、GitLabの読み取り専用レプリカ（インターフェイスを含む）を実行できます。







フォールトトレランスとスケーラビリティ



GitLabは単一のアプリケーションとして設計されているため、ソースコードリポジトリ、タスクトラッキング、CI / CD、および監視用の単一のデータウェアハウスが含まれています。 このような単一のアプローチは、すべての参加者の可視性、効率の向上、プロジェクトの作業の簡素化に貢献します。







GitLabは多くのチームの開発コアであるため、時間に関係なく問題なく動作するはずです。 このリリースでは、 PostgreSQL High Availabilityが公開され 、GitLabのPostgresクラスターを簡単に構成および実行できるようになりました。 Omnibusと自動フォールトトレランスシステムに基づいた簡単なインストールプロセスにより、開発者が外部の問題に気を取られないようになります。

Canary Deployment Performance Assessment（EEP）



GitLab 9.1では、カナリア展開をGitLab CIに追加しました。これにより、新しいリリースの起動プロセスの制御レベルが向上しました。 このアプローチの本質は、グローバルなリリースの前に、製品の新しいバージョンの展開が少数のノードでのみ実行されることです。そのため、変更はユーザーのごく一部に影響し、予期しない合併症の場合、これらの変更は簡単に修正またはロールバックできます。







GitLab 10.2では、このような「カナリア」バージョンのシステムメトリックを追跡する機能を追加します。これにより、異なるバージョンの実際のパフォーマンスを簡単に比較できます。 開発者は、変更が生産性に与える影響をすばやく評価し、本格的な展開を実行できるかどうかを判断できます。 そして、最良の部分は、GitLabを離れることなくこれらすべてを実行できることです！

改善されたイベント監査（EES、EEP）



GitLab Enterprise Editionの監査イベントは、制御とレポートの強化に役立ちます。 GitLab Enterprise Edition Starter（EES）では、各プロジェクトまたはグループのイベントのリビジョンを表示できます。 GitLab Enterprise Edition Premium（EEP）には、すべてのプロジェクトイベントが1箇所に集中する集中ログが含まれています。







GitLab 10.2 EESおよびEEPには、プロジェクトまたはグループに対するアクションの追加イベントが含まれるようになりました。 例：







  • プロジェクトまたはグループを作成する
  • プロジェクトまたはグループを削除する
  • プロジェクトへの新しい所有者の移動、名前変更、または割り当て
  • グループの可視性を変更する
  • ユーザーをグループに追加する（ユーザーのアクセス権を表示する）
  • ユーザー権限を変更する
  • グループへのプロジェクトの追加またはグループからのプロジェクトの削除


GitLab EEPでは、リモートプロジェクトまたはグループのリビジョンデータがサーバー上の共通ログに保存され、管理者がアクセスできるようになりました。 また、アクションを実行するユーザーのIPアドレスも含まれています。

プロジェクトマイルストーンからグループマイルストーン（CE、EES、EEP）へのプロモーション



プロジェクトを大きくした場合、プロジェクトのマイルストーンをグループのマイルストーンに簡単にアップグレードできます。 目的のプロジェクトのマイルストーンページに移動し、ボタンをクリックして、それをグループのマイルストーンにします。 タグと同様に、グループのすべてのプロジェクトで同じ名前を持つすべてのプロジェクトマイルストーンは、グループの1つのマイルストーンに結合されます。 そして、以前に結合されたマイルストーンの1つに関連付けられていたすべてのタスクとマージ要求は、グループの結果のマイルストーンに関連付けられます。 以前のプロジェクトのマイルストーンに関連付けられたプロジェクトタスクボードは、グループのマイルストーンに関連付けられます。

パフォーマンスの改善（CE、EES、EEP）



パフォーマンスはGitLabの重要な部分であり、数十万人のユーザーに拡張できます。







GitLab 10.2には、Sidekiqを使用してタスクを並行して実行する新しいGitHubインポーターが導入されています。これにより、GitHubからプロジェクトをインポートする時間が大幅に短縮されます。 GitHubへのリクエスト数の制限に達すると、新しいインポーターは作業を中断し、制限がリセットされたときに作業を続行します。 この場合、Sidekiqハンドラーはいずれも動作を停止しません。







https://github.com/yorickpeterse/ogaのような小さなプロジェクトの場合、インポート時間は5分から30-60秒に短縮され、 https：//github.com/kubernetes/kubernetesのような大規模なプロジェクトの場合、数時間ではなくインポート時間週は6.5時間になりました。







新しいGitHubインポーターの詳細については、 CEマージリクエストを参照してください 。







さらに、GitLab 10.2には、マージ要求のダウンロードの高速化に焦点を合わせた12のパフォーマンス改善が含まれています。 さらに、タスクの読み込みを高速化し、サーバーリソースを大幅に消費する極端なケースの数を減らしました。
