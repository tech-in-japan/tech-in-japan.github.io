1.1概要



このドキュメントで説明する考えと例は、変更を連続したコミットのグループに分割することの重要性と、それらに適切なメッセージを書き込むことの重要性を明確に示す必要があります。 これらの手順が広く適用される場合、これによりOpenStackのgit履歴の品質が大幅に向上します。 gitの履歴を効果的に変更するには、ニンジンとスティックが必要です。 この文書はニンジンである必要があり、利点と利点に人々の注意を引き付けますが、他の人にとっては、Gerrit Collective Code Inspection Systemの使用は鞭のように振る舞います。 ;-P







言い換えれば、Gerritで検査するとき、コードの正確さだけを見る必要はありません。 検査するときは、まずコミットメッセージを調べて、メッセージ自体の改善を示す必要があります。 多くの論理的な変更を結合し、送信者がそれらを別々のコミットに分割することを必要とするコミットを詳しく調べてください。 インデントのフォーマットの変更が機能の変更と組み合わされていないことを確認してください。 また、コードのリファクタリングが機能の変更などとは別に修正されるようにしてください。







Gerritの一連のパッチの処理が完全に理想的ではないことにも言及する価値があります。 ただし、一連のパッチの作成を避けるために、これを正当な理由と見なすべきではありません。 使用するツールは開発者のニーズに対応する必要があります。また、それらはオープンソースであるため、修正または改善できます。 ソースコードは「よく読まれ、時々書かれます」。したがって、最も重要な基準は、コミュニティの多数の開発者の助けを借りてコードの長期サポートを改善することです。もう一度コード。







そして今、詳細な原則だけでなく、良い習慣と悪い習慣の例も。







1.2変更の構造的な分離



適切なコミットを作成するための基本的なルールは、コミットごとに「論理変更」を1つだけ提供することです。 この重要なルールには多くの理由があります。







  • コードの変更が少ないほど、潜在的な弱点のチェックと特定がより迅速かつ容易になります。
  • コミットが正しくないか壊れていることが判明した場合、不正なコミットがロールバックされる可能性があります。 また、元のコミットと混同するコードに他の無関係な変更がなければ、それははるかに簡単になります。
  • git bisectコマンドを使用するなどのトラブルシューティングを行う場合、小規模で整形式のコミットは、コード内で問題が発生した場所と時期を特定するのに役立ちます。
  • git annotateまたはgit blameでgitの履歴を表示するとき、小さな整形式のコミットは、コードの問題部分を見つけて切り分け、問題の断片がどこから来たのかを理解するのに役立ちます。


1.2.1コミットを作成する際に避けるべきこと



悪い習慣の一般的な例を知り、理解する必要があります：







  • フォーマットのインデントとスペースをコードの機能的な変更と組み合わせます。


変更をフォーマットすると、重要な機能の変更があいまいになり、レビューアがコミットの正確さを正確に判断することが難しくなります。 解決策： 2つのコミットを作成します。1つはフォーマットの変更、もう1つは機能の変更です。 通常、インデントが最初に行われますが、これは厳密な規則ではありません。







  • 2つの無関係な機能変更を混合します。


繰り返しますが、レビュアーは、2つの無関係な変更を混在させると、欠陥を特定することがより難しくなります。 壊れたコミットを後でロールバックする必要がある場合は、2つの無関係な変更を見つけて解決する必要がありますが、新しいエラーが発生するリスクもあります。







  • 1つの巨大なコミットで優れた新機能を送信します。


新しい関数のコードは、実装全体が存在する場合にのみ有用である可能性があります。 ただし、これは、1回のコミットで機能全体を提供する必要があるという意味ではありません。 多くの場合、新機能は既存のコードのリファクタリングを伴います。 新しい関数が実装されているコミットではなく、個別のコミットでリファクタリングを実行することが非常に望ましいです。 これは、リファクタリングに意図しない機能変更がないことをレビューアーとテストスイートが確認するのに役立ちます。 最近書かれたコードでさえ、多くの場合、独立して考慮できるいくつかの部分に分割できます。 たとえば、新しい内部APIまたはクラスを追加する変更は、個別のコミットで行うことができます。 繰り返しますが、これによりコード検証が容易になります。 また、新しい機能をマージする準備ができていない場合、他の開発者がgit cherry-pickを使用して作業の小さな塊を取得できるようにします。 新しいパブリックHTTP APIまたはRPCインターフェイスの追加は、実際の内部実装とは別のコミットで行う必要があります。 これにより、作成者とレビュー担当者は、現在選択されている内部実装にとって簡単な設計を選択するだけでなく、APIまたはRPCの全体的な設計について考えるようになります。 パッチがパブリックHTTPに影響する場合は、APIImpactフラグを使用します（ 外部リンクの有効化を参照）。

1.3コミットメッセージの情報



コミットメッセージのテキストは、コード自体が変わるのと同じくらい重要です。 メッセージを作成するとき、留意すべき重要なことがいくつかあります。







  • レビューアが最初から問題が何であるかを理解していると思い込まないでください。


バグレポートといくつかのコメントを読んだとしても、問題が何であるかが常に明確ではないことがよくあります。 コミットメッセージには、元の問題の明確な説明を含める必要があります。 特定のバグは、問題の特定方法に関する興味深い歴史的背景にすぎません。 レビューアは、エラー追跡システムでこの欠陥を調査することなく、提案されたパッチが正しい決定を行うために何を行うかを理解する必要があります。







  • レビューアが外部のWebサービスまたはサイトにアクセスできると想定しないでください。


6か月後、電車/飛行機/バス/ビーチ/バーに乗っている人が問題を修正してgitの履歴を表示すると、オンラインエラーレポートまたはドキュメントを使用してサーバーにアクセスできる保証はありません。 分散ソースコード管理（SCM）システムは、リポジトリ内のすべての情報にアクセスするためにオンラインである必要がなくなったという点で、大きな一歩を踏み出しました。 コミットメッセージは、gitのメリットを享受し続けるために、完全に自給自足でなければなりません。







  • コードが自明/自己文書化であると想定しないでください。


ある人にとって明らかなことは、別の人にとって完全に理解できない場合があります。 もちろん、明らかな誤字や書式の変更を修正する場合を除き、変更ごとに、元の問題の内容と修正方法を常に文書化します。







  • 変更が行われた理由を説明してください 。


よくある間違いは、開発者がこのように決定した理由を説明せずに、コードの記述方法を単に説明することです。 最も重要なことは、変更の意図と動機を説明することですが、コードの一般的な構造を説明することも忘れないでください。







  • コミットメッセージを読んで、コード構造の改善を示しているかどうかを確認します。


多くの場合、大きなメッセージを説明するとき、変更を2つ以上の部分に分割する必要があることが明らかになります。 戻って複数のコミットに分割することを恐れないでください。







  • 何をチェックするかを適切に決定するために十分な情報を提供します。


Gerritが新しいパッチパッチに関する電子メールアラートを送信すると、電子メールには最小限の情報、主にコミットメッセージと変更されたファイルのリストが含まれます。 大量のパッチがあるため、すべてのレビュー担当者がパッチを詳細に調査することを期待しないでください。 したがって、コミットメッセージには、これが見なければならないパッチであることを潜在的なレビューアに警告するのに十分な情報を含める必要があります。







  • コミットの最初の行が最も重要です。


Gitでは、コミットメッセージの最初の行には特別な意味があります。 ブランチや、テキスト用のスペースがあまりない他の多くの場所をマージするときに、git annotateチームのメッセージ、gitkビューアプログラムで電子メールの件名として使用されます。 変更自体の簡単な説明に加えて、影響を受けるコードの量を忘れないでください。 たとえば、これがlibvirtドライバーに影響する場合、最初の行のどこかに「libvirt」を指定します。







  • 現在のコードの制限を説明します。


変更中のコードに将来の機能拡張や既知の制限がある可能性がある場合は、コミットメッセージでそれらを必ず記載してください。 これは、著者がより広い視野を検討し、長期的な欲求と比較した短期的な目標に関してどのような妥協が行われたかをレビューアに示しています。







  • このパッチに固有のコメントを含めないでください。


つまり、コミットをやり直す場合は、メッセージに「パッチセット2：リベース」を追加しないでください。 変更をマージしても、違いは生じません。 ただし、変更に関するコメントとしてGerritにメモを書きます。 これは、レビュー担当者がパッチセット間で何が変更されたかを見つけるのに役立ちます。 これは、「追加された単体テスト」、「修正されたローカリゼーションの問題」、またはコミットの全体的な目的に影響を与えないその他のパッチなどのコメントにも適用されます。

1.3.1外部リンクの包含



メッセージは主に人間による解釈を目的としていますが、常にマシン処理を目的としたメタデータが含まれています。 OpenStackでは、「Change-Id」に加えて、バグ識別子へのオプションリンク、ブループリント（スキーマ/ドキュメント）へのリンク、DocImpactフラグ、APIImpactフラグ、およびSecurityImpactフラグが含まれます。







  • 「Change-Id」行は、Gitフックによって生成される変更を記述する一意のハッシュです。 rebaseコマンドでコミットを編集する場合、検査後に変更されるべきではありません。また、パッチバージョンを追跡するためにGerritによって使用されます。
  • 文字列 'bug'は、いくつかの方法でエラーを参照できます。 レビュー担当者がLaunchpadでエラーにすばやくアクセスできるように、Gerritはreview.openstack.orgのパッチを表示するときにエラーリンクを作成します。

    • Closes-Bug：＃1234567-コミットが参照しているエラーを完全に排除して閉じることを意図している場合は、「Closes-Bug」を使用します。
    • 部分バグ：＃1234567-コミットが部分的な修正であり、このバグを修正するためにさらに作業が必要な場合は、「部分バグ」を使用します。
    • Related-Bug：＃1234567-コミットが指定されたエラーに単に関連している場合は、 'Related-Bug'を使用します。
  • コミットが特定の機能を実装することを意図している場合、行「blueprint」は、Launchpad blueprint（Launchpad Webサイトの文書）という名前を示す必要があります。 レビュー担当者がLaunchpadのドキュメントにすばやくアクセスできるように、Gerritはreview.openstack.orgでパッチを表示するときにブループリントへのリンクを作成します。
  • DocImpact行には、 DocImpact行と、この変更がドキュメントに影響する理由に関するコメントが含まれています。 DocImpactを別の行に配置します。 レビュー担当者が変更の理由を理解できるように、このフラグを使用して、ドキュメントがパッチに含まれているか、変更がドキュメントに影響することを示します。 できるだけ多くの情報を含めてください。 このフラグがコミットメッセージに含まれると、Gerritはopenstack-manualsプロジェクトのバグレポート（openstack命令）を作成して、実行を追跡およびシーケンスするか、必要に応じてopenstack-api-siteに移動します。
  • APIImpact行には、 APIImpact行と、この変更がパブリックHTTP APIに影響する理由に関するコメントが含まれています。 APIImpactを別の行に配置します。 このフラグを使用して、パッチがパブリックHTTP APIを作成、更新、削除するか、パブリックAPIの動作を変更することを示します。 できるだけ多くの情報を含めてください。 このフラグがコミットメッセージに含まれている場合、関連するすべてのレビューはこのリンクを使用して見つけることができ、 API_Working_Groupドキュメントのページも関連するレビューを見つけるのに役立ちます。 さらに、Freenode ＃openstack-sdksチャネルの IRCを介してAPIワーキンググループに直接連絡できます 。
  • SecurityImpact文字列には、単にSecurityImpact文字列が含まれます。 変更がセキュリティに影響するため、OpenStack Security Groupによるレビューが必要であることを示すために使用されます。
  • UpgradeImpact行には、 UpgradeImpact行と、この変更が更新に影響する理由に関するコメントが含まれています。 これは、変更が継続的な展開、またはNからN + 1への更新を行っているユーザーのアップグレードの結果であることを示すために使用されます。 また、影響を受けるプロジェクトのリリースノートのアップグレードノートセクションへの更新を示す場合もあります。


すべてのメタデータはマシン用であり、人々にとって二次的な重要性があるため、コミットメッセージの最後にすべてのメタデータをグループ化する必要があります。







注：Gitを使用する多くのオープンソースプロジェクトでは、「Signed-off-by」タグ（「git commit -s」コマンドを使用して生成される）を含めるのが一般的ですが、これはOpenStackには不要です。 コードをGerritに送信する前に、OpenStackはすべてのメンバーがCLA（Contributor License Agreement）に署名することを要求します。これは同等の目的を果たします。







特定のパッチに取り組んだ人々を示すために、コミットメッセージでname-name@example.com （ Collaboration ）のCo-Authored-Byを使用することをお勧めします。 これは複数の著者の承認に関する合意であり、私たちのプロジェクトは、統計収集ツールがそれらを分析する際にこれらのフィールドを監視することを奨励しています。







1.3.2コミットメッセージの構造の概要



  • 最初の行に、変更の簡単な説明を入力します。
  • 最初の行の後に空の行を挿入します。
  • 次の行に、変更の詳細な説明を書き、必要に応じて分類します。
  • 最初の行は50文字に制限する必要があり、ピリオドで終了しないでください。
  • 後続の行は72文字で囲む必要があります。

    • vim（ほとんどのディストリビューションの$ EDITOR環境変数のデフォルト）は、自動的に文字列をラップできます。 ほとんどの場合、vimrcサンプルファイル（ /usr/share/vim/vim74/vimrc_example.vim



      ）を~/.vimrc



      にコピーするだけです。
    • 段落を編集した後、エスケープキーを押して、段落内にカーソルを置き、組み合わせgqip



      を入力することで、段落をラップできます。
  • 必要に応じて、「Change-id」、「Closes-Bug #NNNNN」、「blueprint NNNNNNNNNNNN」の行を最後に配置します。


例： 







   libvirt get_cpu_info    API  get_cpu_info   libvirt      XPath      XML .       LibvirtConfigCaps.        . DocImpact Closes-Bug: #1003373 Implements: blueprint libvirt-xml-cpu-model Change-Id: I4946a16d27f712ae2adf8441ce78e6c0bb0bb657
