ASP.NET Core Today：長所と短所

ASP.NET Coreには、現在の形式でASP.NETを置き換える可能性がすべてあります。 今すぐASP.NET Coreにアップグレードする必要がありますか？ これについて専門家と話しましょう：



  1. Dino Espositoは、ライター、コンサルタント、トレーナー、テクニカルエバンジェリストであり、DDDおよびCQRSの概念の専門家および普及者として認められています。
  2. Maurice de Beijerは、独立したコンサルタント、MVP、およびオンラインコースであるThe React Tutorialの著者です。
  3. Andrey TerekhovはフルスタックEPAM開発者であり、サーバープレレンダリングのスペシャリストです。










3人全員が1週間後にサンクトペテルブルクでDotNext会議でASP.NET Coreに関するプレゼンテーションを行います。



ディノとモーリスは英語で回答しましたが、以下に回答の翻訳を公開します。





-ディノ、 あなたのレポートは主に開発管理に携わる人々、ソフトウェアアーキテクトとCTOに向けられています。 それは正確に何ですか？



-ASP.NET Coreは単なる新しいバージョンではありません。 これは完全に新しいプラットフォームであり、時代の変化です。 この移行は 、企業に多くのメリットをもたらします。コードのコンパクト化、パフォーマンスの向上、スケーラビリティの向上です。 しかし、見返りにどの価格を支払う必要がありますか、どのくらいのコードを書き換える必要がありますか？ ASP.NET Coreへの移行のために実行する手順、最初から書き直さなければならないサブシステム、移行戦略を説明します。



「ASP.NET Coreへの切り替えを既に考えている人と、待つべき人に答えることを約束します。」 RC2出力に切り替えることを誰に勧めますか？



-誰も。 ASP.NET Coreはまだ未熟です。 データアクセスなどの多くの基本的なものは、まだ完全に設計されているわけではありません。 現在使用しているコードがリリースバージョンで動作するという保証はありません。 今、私は最終的にASP.NET Coreが2年で落ち着くと思います。

プログラマー向けのツールの開発に携わる企業には、真に.NET Coreを監視する必要があります。 JetBrainsのような企業。 彼らはツールでお金を稼ぎ、ツールは発売日から新しいプラットフォームをサポートしなければなりません。 企業部門の開発者には、待つことをお勧めします。



- アプリケーション開発者が利用できる移行戦略は何ですか？



-ASP.NETの全期間にわたって変更されなかったメカニズムの多くは、ASP.NET Coreで突然変更されました。 たとえば、承認、セキュリティ、アプリケーションとサーバーインフラストラクチャの相互作用。 これらすべての変更を検討し、馴染みのあるものを新しい方法で実装する方法を見つけます。 適切に設計されたアプリケーションの場合、移行はより簡単になり、専任の開発チームが妥当な時間内に行うことができます。 特にプラットフォームに依存するコンポーネントを使用する銀行ソフトウェアなど、企業部門のその他の多くのアプリケーションでは、移行が完全に不可能になる可能性があります。



-アーキテクチャパターンとしてCQRSを使用すると、移行を緩和できますか？



-CQRSは非常に一般的な概念であり、アプリケーションコード全体を特定の2つのサブシステム（読み取りと書き込み）に分割することを規定しています。 このアプローチは、独立したチームが両方のサブシステムを移植する並行作業の可能性があるため役立ちます。 CQRSが使用されているかどうかに関係なく、アプリケーションレイヤーがより適切に分離され、アプリケーションがASP.NET実装の詳細から抽象化されるほど、コードのローカライズと書き換えが容易になります。 大規模なアプリケーションを新しいプラットフォーム全体に移行する必要はありませんが、部分的に徐々に移行できます。





「 モーリス、 最初のレポートで何について話しますか？」



-15年以上にわたるASP.NETプラットフォーム。 さらに、System.Webの作成時には、従来のASPとの後方互換性をサポートするために大量のコードが含まれていました。 この間、プラットフォームは十分な量のコードを蓄積しましたが、これはもはや必要ではなく、古くなっています。 マイクロソフトは難しい選択に直面しました。後方互換性を放棄するか、新しいプラットフォームを発表します。 彼らは2番目のオプションを選択しました。 同時に、既存のランタイムを放棄しなければなりませんでした。 マイクロソフトは常に、Windows上ですべてを作成および実行することに焦点を当てた会社です。 既存のASP.NETも例外ではありませんでした。 現在、状況は大きく変化しています。会社の戦略の重要な位置がAzureとLinuxを採用し始めました。



ASP.NET Coreは、従来のASP.NET MVCのエラーに関する作業であり、ゼロから開始する機能です。 また、Microsoftは、若い開発者の間でRubyやNodeJSと同じくらい人気を博そうとしています。



-新しいプラットフォームへの移行とはどういう意味ですか？



-.NET Coreには、これまで使用していた多くのコンポーネントが含まれていません。 System.Web、Webフォーム、トランザクションスコープ、WPF、Winフォームを忘れてください。 彼らはもういません。 単純なASP.NET MVCアプリケーションの場合、変更はほとんどなく、移行は簡単です。 多数の.NET FrameworkおよびASP.NETパイプラインクラスを使用するより複雑なクラスの場合、状況はより複雑です。 何かが機能するかもしれませんが、何かは機能しません。 コードの一部を最初から書き直す必要があります。 ASP.NET MVCとWebApiが結合されたため、WebApiは追加の問題を引き起こす可能性があります。 多くのライブラリとnugetパッケージはまだ準備ができていません。 そのため、一部のアプリケーションは、新しいバージョンのライブラリが表示されるまで切り替えられません。



-ASP.NET Framework 5とASP.NET Coreの見通しは何ですか、コミュニティは各実装の支持者に分割されませんか？



WebフォームからASP.NET MVCへの移行に似た状況を待っていると思います。 ASP.NET Frameworkは長い間サポートされます。 最初は、ASP.NET Coreで開発されるアプリケーションはごく一部です。 その数は増えますが、遅かれ早かれ誰もがASP.NET Coreに切り替えることになるでしょう。 まだ多くのWebフォームアプリケーションが動作しています。 そこには、実動で動作するCOBOLコードがまだあります。 ただ動作するため、誰もそれを書き換えません。 ただし、今ではWebフォームで新しいアプリケーションを開発することは誰にも起こりません。誰もがMVCを選択します。 遅かれ早かれ、ASP.NET FrameworkとASP.NET Coreでも同じプロセスが発生します。 後者は、最新の開発標準を満たすより多くの機能を提供します。



-たとえば、Dockerを使用できるかどうか。 これは、 2番目のレポートのトピックです。



-もちろん。 Dockerは、「マシン上で動作する」問題の解決策として位置付けられています。 最近まで、プラットフォームに非常に依存していました。ASP.NETはWindowsの99％を意味していました。 はい、Monoはありましたが、Microsoftの公式サポートはありませんでした。これは、多くの開発者にとって、そのようなオプションを検討しようとさえしなかったことを意味します。 Monoは現在Microsoftが所有しており、同社はLinuxおよびMac OSを公式にサポートしています。 WindowsまたはLinuxでDockerコンテナーを実行できます。 講演では、Dockerの基本概念を説明し、DockerコンテナーでASP.NET Coreアプリケーションを実行する方法を示します。 これは、バイナリファイルではなく、クラウドインフラストラクチャ（Azureなど）で実行できる構成済みのアプリケーションを配信できることを意味します。



-マイクロソフトは、広告キャンペーンにおいて、高いパフォーマンス、拡張性、クロスプラットフォームという珍しい立場に依存しています。 ASP.NETはNodeJSにcっていると思いませんか？



-ある意味では、NodeJSとASP.NETは常に競合他社でした。どちらもバックエンドのプラットフォームです。 しかし、実際、それらの間にはもちろん、闘争はありませんでした。 新世代の開発者、いわゆるヒップスター開発者は、RubyとNodeを好みます。 .NETとJavaの側にいる、古い世代の企業環境の人々。 .NET Coreは明らかに、より若々しく、トレンディで人気のあるものを目指しています。 はい、そうです、将来的には、.NET CoreとNodeJSの間の本当の対立が予想されます。





-アンドレイ、DotNext について教えてください。



-ASP.NET Coreの最新のSPAアプリケーションが開発段階と運用段階の両方で、特に同形JSとWebコンポーネントの使用に関して、新しい機会について。



-ASP.NETはどこにありますか？



-JavaScriptServicesライブラリに関するものです。 その助けを借りて、さまざまなバックエンドへの柔軟なルーティングを構成し、1つの屋根の下ですべてを収集できます。 これは、さまざまな技術で書かれたマイクロサービスの動物園を管理するのにうんざりしている人に役立つかもしれません。 さらに、このテクノロジーにより、サーバーの事前レンダリングにより、最初のページを取得する時間を大幅に短縮し、SPAアプリケーションの検索エンジンのインデックス作成機能を提供できます。 同時に、最新のリッチWebスタックのすべての利点（ビルダー、トランスパイル、ホットリロードなど）が保持されます。 webpack -hrmではなく、 dnx webと書くだけで十分です。



-どのJavaScriptフレームワークがサポートされていますか？



-React-実稼働対応、Angular 2-ご自身の責任とリスクで：まだ開発中です。



-ライブラリは公式にASP.NET Coreに含まれているようです。 彼女はRC2または1.0を入力しますか？



-当初、開発はSteve Sandersonによって開始され、開発全体はコミュニティによって行われました。 その後、ASP.NET Coreに含めることが決定されました。 スティーブは他のタスクに切り替えました-ある時点で、プルリクエストの受け入れに問題がありました。



したがって、ASP.NETを正式に含めるという決定がポイントです。 これは、特定の人ではなく、誰かが対処する誰かが常に存在することを意味します。 理論的には、ASP.NETの主要部分と共にリリースする必要があります。
