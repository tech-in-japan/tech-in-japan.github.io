Oracle JDKは料金を支払う必要があります。 現在のオプションは何ですか？

Javaの関連性についての議論は続いています。 時代遅れになりつつあり、より高度な言語がそれに取って代わりつつあると言う人もいます。 コアスキルとして毎年Javaを選択するスペシャリストの数を挙げて、意見が一致しない人もいます。 いずれにせよ、既存のソフトウェアのほとんどがJavaで記述されているという事実を無視することはできず、顧客は内部と外部の両方でJavaを使用し続けます。 このような状況では、Javaに関する新しい実装やグローバルニュースを避けるべきではありません。 EPAMのスイスオフィスの主任開発者兼ソフトウェアアーキテクトであるPeter Verhasは 、Oracleが導入している変化とそれが私たちに与える影響についての彼の分析を共有しました。









Javaには最近多くの変更が加えられており、言語の進化の仕方が根本的に変わったようです。 新しい機能が言語に表示されると便利です。 しかし、これらの変更だけではありません。 オラクルは、この会社がリリースするJavaバージョンのサポートモデルと更新サイクルに大幅な変更を加えました。 これらすべてが、変化、誤報、誤解によって引き起こされる大きな混乱をもたらしました。 この記事では、Javaエコシステムの終末論的な終わりを信じ始める前に、ユーザーがイノベーションについて知る必要があるすべてを要約します。



ただし、最初にいくつかの点を明確にする必要があります。



OracleはJavaを所有していますが、JavaはOracleではありません。 クラスパス例外を使用してGPLv2.0でリリースされたJavaソースコード。 実際、これは、誰でも無料で言語のソースコードを使用および変更できることを意味します。 Oracleを含むすべての人が禁止されていることは1つだけです。コードを変更し、クローズドソースライセンスで公開することです。 これは、オラクルのマーケティングまたはビジネス上の利益が必要とする場合でも、有料版のみのJavaの商用利用が法的に不可能であることを意味します。 しかし、私はまだ技術の専門家です。 この点に対処するには、GPLv2ライセンスのテキストをダウンロードして弁護士に相談するのが100％最善です。

Javaアセンブリをリリースしてサポートを行っているのはオラクルだけではなく、他にも多くのベンダーがあります。 もちろん、Oracleは、言語とツールの開発に最も投資する主要なプレーヤーとして、それをサポートする最も権威のある商業会社です。



実際、OracleにはJavaソースコードだけでなく独占的な権利もありません。 開発プロセス、Javaの定義の変更、API-これらはすべてJava Community Processの手にあり、誰でも参加できます。 個人の場合、これは完全に無料です。 このグループのメンバーは、Intel、IBM、Credit Suisse、Software AG、RedHatなどの企業です。 Oracleではなく、Javaの将来を決定します。 そして、JPMSの最終バージョンの承認が非常にスムーズに進まなかった最後の時を見ることができるように、彼らは彼ら自身の意見を持っています。



OracleがJavaではなくJavaを「所有」していると判断したこの紹介の後、Oracleが導入した開発サポート手順とJavaロードマップの変更を見てみましょう。



Java 9、10、11の後の変更



Oracleは、JDK11以降、Oracle JDKは無料ではなくなることを発表しました。 そして、最初はこの声明は怖いです。 私たちが無料で使っていたものは、もはやそうではなくなります。 実際には、「支払わなければならない」ことはほとんど論理的です。 しかし、「ほとんど」という小さな言葉は、理由があります。 使用したサービスがあり、それが無料でなくなった場合、2つの選択肢があります。後で使用するために支払うか、それ以上使用しないかです。 Oracle JDKを引き続き使用する必要がありますか、それとも別のものに切り替えることができますか？ それが本当の質問です。



それに答えるには、Oracle JDKとは何か、そこにある代替手段を理解する必要があります。 別の重要な変更は、新しいバージョンのリリースサイクルと長期サポートの概念（英語-長期サポートまたはLTS）に関するものです。 まず、リリースサイクルの変更、次にOpen JDKとの混乱、最後にLTS構造について説明します。



リリースサイクル



Javaは1996年に登場しました。 Javaの最初のいくつかのバージョンは、ほぼ定期的にリリースされました。







ただし、図を見ると、Java 5.0のバージョンが遅れていることがわかります。 Java 6は以前のバージョンでも急いでおらず、最大のブレークはJava 6とJava 7の間でした。その後も、新しいバージョンはあまりリリースされませんでした。 Java 8は多くの新機能を受け取りましたが、2年の待ち時間がかかりました。 状況はJava 9と同様でした。Java9はさらに大きな変化を経験していましたが、3年待ちました。



このような更新リリースサイクルの問題点は、進化し続ける言語市場で競合する言語自体が十分に速く新しい機能を提供できないことです。 モジュール式システムの開発に3年かかる場合、プロセスをスピードアップすることはできません。 ただし、Java 9に登場したものの一部は以前にリリースされた可能性があります。 しかし、以前のリリースがなかったため、彼らは単に開発者に連絡することができませんでした。 すでに利用可能であった関数は、このリリースで最もゆっくりと開発された関数の出現を待つべきでした。 これは、従来のカスケード開発モデルの問題の1つです。



Oracleは、6か月ごとにアジャイルに移行してバージョンを作成することを決定しました。 2017年9月から、3月と9月に新しいJavaリリースが予定されています。 一部の機能がこれらの日付に対して準備ができていない場合、次のリリースに転送されますが、残りの機能は準備ができているため、最新のものに分類されます。



一般的に、これは正常です。 一方で、Javaコミュニティと開発者はこれに満足しているはずです。 一方、サポートの問題は残っています。 多くのJavaバージョンをサポートできるのは誰ですか？ これが、長期サポート（LTS）が導入された理由です。これについては、後ほど説明します。 ただし、最初に、Oracle JDKおよびOpenJDKが何であるかを明確にする必要があります。



Oracle JDKおよびOpenJDKであったこととなること



Java 9まで、Java JavaアセンブリOracle（Sun Microsystems）のバイナリリリースがあり、ほとんどの開発者や企業が本番環境で使用していました。 このバイナリはオープンソースJavaに基づいており、Java Flight Recorder、Java Mission Controlなどの追加の企業ツール、およびApplication Class-Data Sharingなどの他の機能がいくつか改善されています。



OpenJDKアセンブリはオープンソースJavaから作成されましたが、改善はありませんでした。 このJavaアセンブリは、Javaのプロパティを実験したり、コードを変更したり、単にオープンソースコードの信奉者である理由を問わず使用したりする人々によって使用されました。 ただし、ソースコードをダウンロードし、自分でコンパイルする必要があります。



Java 9から、OracleはOracle JDKと並行してOpenJDKの提供を開始しました。 同社はまた、2つのバージョンの機能、パフォーマンス、および安定性の差を縮め、それが実現したらOracle JDKを有料にすることを発表しました。 当時、OpenJDKとOracle JDKの両方は、Oracleからバイナリアセンブリとして無料で入手できました。 これらはJava 9とJava 10でした。



Java 11のリリース後、これは終了しました。 OpenJDKは引き続き無料で利用できますが、本番システム用のOracle JDKは有料になりました。 2つのバージョンの間に大きな違いはありません。 商用バージョンにはインストーラーがありますが、OpenJDKにはZIPファイルしかありません。 他にも違いがありますが、Javaユーザーはそれらに注意を払うべきではありません。 この記事では、Oracleの製品管理担当シニアディレクターであるドナルドスミスが、技術的に詳細な相違点のリストを説明しています。



さらに、Oracle JDKは、他の（商用のユーザーでも）ユーザーには無料です。 Oracle JDKは次の用途に自由に使用できます。



•開発

•テスト

•プロトタイピング

•デモンストレーション



OpenJDKを他の目的に使用するか、Oracleに支払い、Oracle JDKを使用してサポートを受けることができます。 サポートは良いです。



長期サポートリリース



6か月ごとに新しいバージョンのリリースサイクルに移行すると、サポートの問題が発生しました。 誰もが自分の利益のためにこれほど多くのバージョンをサポートすることはできません。 2007年にアプリケーションにJava 6をインストールした場合、11年でOracleからサポートを受けることができます。 6か月ごとの新しいバージョンのリリースは、22の異なるバージョンの同時サポートを意味します。 これは、Oracleまたはすべてのリリースをサポートすることを決定した他のベンダーにとって大きな負担になります。



オラクルの戦略は、3年ごとに1つのリリースを特定し、長期的なサポートを受けることです。 そのような最初のリリースはJava 11で、2026年までサポートされます。 この期間中、Java 17も2021年9月にリリースされ、現在の計画に沿って7年間の長期サポートが提供されます。 他のすべてのいわゆる機能リリースは、次のバージョンのリリースまでのみサポートされます。







サポートはOracle JDKにのみ適用され、JDKおよびサポートの支払いを行った人にのみ適用されることに注意してください。 ただし、OpenJDKを使用する予定がある場合は、運命に翻弄されたとは思わないでください。 私たちにとって、何も変わっていません。 以前にサポートの支払いをしなかった場合、とにかくそれを受け取りませんでした。 支払い済みの場合、引き続きこのサポートを支払い、取得します。



以下は、さまざまなバージョンのサポート終了予測に関するOracleの記事へのリンクです。



OpenJDKの新しいバージョンはありますか？ もちろん。 以前は、バグを修正するJavaオープンソースアップデートがリリースされましたが、リリースされる保証はありませんでした。 これは、すべての参加者がこれに興味を持っていたために起こりました。 将来的には、すべてが以前のように残り、Javaは新しいバージョン、新しいアセンブリを受け取ります。



どのバージョンのJavaをインストールするのが最適かを選択するための3つの主要な戦略があります。



  1. Oracle JDKとサブスクリプションに対してOracleに支払い、長期サポート付きのリリースを使用します。 これは、信頼性の要件を満たすために、Javaインフラストラクチャを使用する大規模な企業が行うべきことです。
  2. OpenJDKに切り替え、長期サポートのあるリリースのみを使用し、OpenJDKで利用可能なバグ修正バージョンをインストールします。 これは、Javaインフラストラクチャのコストを変更できない、または変更したくない、過去に商用サポートなしで実行でき、長期的にも実行できると考えている企業にとって実行可能なソリューションです。
  3. OpenJDKに切り替え、表示されたらすぐに6か月ごとに後続のリリースをインストールします。 これは、最新のテクノロジーソリューションを必要とする企業向けのオプションです。


異なるマシンに異なるバージョンのJavaをインストールすることにより、戦略を組み合わせることができます。 何を選択しても、次のサイトからプレビルドのバイナリバージョンをダウンロードできます。

まとめ



どうする？ OpenJDKを使用しますか、それともOracle JDKに支払いますか？ これは収益性の問題であり、オラクルのサポートがあなたにとってどれほど重要かということです。 あなたがスタートアップであり、素晴らしい製品アイデアを持つ新星でありながらポケットに穴があるために、ビジネスがOracle JDKサポートを購入できない場合、OpenJDKはニーズを満たします。 お金を払う必要はありません。 さらに、あなたがスタートアップの場合、技術専門家が問題を解決するまで数時間サーバーが動作しなくても、それほど重要ではありません。 ダウンタイムの余裕がない場合、おそらく成熟した会社で働いているため、Oracle JDKを使用してサポートにサインアップする必要があります。 また、異なるモデルを組み合わせて、24時間年中無休で使用できる製品サーバー、Oracle JDK、およびビジネスにとってそれほど重要ではない内部アプリケーション上のOpenJDKにインストールすることもできます。
