なぜ関数型プログラミングが主流なのですか？

関数型プログラミングを新しい現象と呼ぶことはほとんどありません。 私が大学にいた90年代前半に、私たちはLISPで何かを書く必要がしばしばありました。 25年が過ぎたように思えますが、すでにこのプログラミング言語は私よりも古いものでした。



関数型プログラミングの利点は、一般の人々によって長い間認識されてきました。 ソフトウェアの開発の成功は、多くの場合、既存のメカニズムを最大限に簡素化して、新しいアプリケーションが現代のユーザーの要件に適応できるようにすることです。 そして同時に、私たちは急いで、無制限の可能性を備えた製品を短時間で消費者に提示する必要があります。 開発したアプリケーションをいくつかの純粋な機能に分割でき、検証が容易な場合、タスクに対処するのがはるかに簡単です。 そのようなアルゴリズムには、世界規模での結果に基づいて計算されるトリッキーな副作用や抽象的な定式化はありません。



しかし、1つの関数型言語が普遍的に認識されていると主張しているわけではないようです。 私は、70年代のCの独特な世界的覇権を意味し、後にC ++、Java、そして最終的にJavaScriptに冠を譲りました。 第2層の候補（Python、PHP、Perl、Ruby、C＃...）のどれも機能的なもののカテゴリーに分類されませんでした。



そして、信じられないようなことが起こりました。 はい、Clojureについて話しています。



冗談です！ 新しい機能言語が世界の舞台に登場してすぐにリーダーになったとは誰も言いませんが、今回は明らかに、突然前面に出たチャンピオンの成長する可能性に対処しています。 現在、React / Reduxでモバイルアプリケーションの作成に取り組んでおり、ここに私が書いたコードを示します。 



aliasesList.reduce((allMatching, matching, index) => { if (matching.count() > 0) { const expanded = allAliases.take(index) .concat(List.of(matching)) .concat(allAliases.skip(index + 1)) .filter(aliases => aliases.count()) .reduce((current, aliases) => current ? current.flatMap( i1 => aliases.map(i2 => i1.concat(i2)) ) : aliases.map(alias => List.of(alias)), null); return allMatching.concat(expanded); } else { return allMatching; } }, new List());




では、なぜスペシャリストは長い間、関数型プログラミングを見失っていたのですか？ そして、なぜ今日ではそんなに一般的になっているのですか？





名詞でダウン



最初の質問に対する答えは、ソフトウェア開発の主な傾向の1つによって大部分が決定されます。 卒業後、私はオブジェクト指向の津波の波に完全に圧倒されたため、LISPで1行も書いていません。 ソフトウェア開発者のコ​​ミュニティは、この方法論を究極の真実として認識しました-数年後、Javaの出現により、実践は定着しただけでした-そして、何らかの理由で機能的プログラミングの利点を永続的に無視しました。



カプセル化、継承、およびポリモーフィズムを備えたOOPは、新しいプログラムの作成プロセスの簡素化に希望を与えました。 しかし、最近、この方法の欠陥がより明らかになりました。 オブジェクトの厳格な階層と明確なクラス特性の欠如は、OOPの進化を著しく妨げました。 徐々に、不確実性は例外なくプログラミングのすべてのコンポーネントに及ぶため、望ましくない相互運用性と予測不可能な副作用なしにはできません。 これはまさに、関数型プログラミングの動詞がOOPの基になっている名詞に取って代わる方法です。



私は少し冗談を言って、少し前にClojureに言及しましたが、最近の関数型プログラミングの復活に貢献したのはまさにこの方言でした。 古き良きLISPでは解決できなかった多くの実用的な問題について話すと、Clojureとともに機能開発者の世代が登場しました。 そのため、OMが作成されました。OMなしでは、現代のJSエコシステムの進化を想像することは不可能です（これについては後で説明します）。





JavaScriptは目の前で成長しています



OOPの失望に加えて、先に述べた2つの質問もいくつかの追加の傾向に関連しています：本格的な汎用プログラミング言語としてのJavaScriptの出現と、広大なJavaScriptエコシステムにおける関数型プログラミング方法論の包含。



10年の変わり目に、サーバーが依然としてほとんどのWeb開発で重要な役割を果たしている一方で、クライアント側のスクリプトが補助機能を実行することが明らかになりました。 はい、Gmailなどの個々のAjaxアプリケーションは、ブラウザ自体で動作する個別のページとしてアプリケーションを使用するのがいかに便利かを示しました。サーバーは主に集中データウェアハウスです。 通常の固定アプリケーションを連想させる高速Webアプリケーションは、CGIのかさばる前任者よりも1桁高いことが判明しました。



2008年のGoogle Chromeのリリース後、特にJavaScriptからのV8エンジンの発表後、傾向は強まりました。 高速JavaScriptの作成は、本格的な単一ページアプリケーションを開発するための前提条件でした。 そして再び、GoogleはAngularJSでリードしています。 V8は、主にサーバー製品の開発に使用される別個のJSプラットフォームであるNode.jsでの地位を強化しました。



突然、JavaScriptは子プログラミング言語のように見えなくなりました。 AngularJSと同様の開発ファミリ全体、およびNodeのおかげで、JSへの関心が非常に高まっており、過去数年間でこのプログラミング言語に対する信じられないほどの需要につながっています。





機能と形態



そして、これらすべてが関数型プログラミングと何の関係があるのでしょうか？ Brendan H.（JSの作成者）は、1995年にブラウザスキームを作成するためにNetscapeに連れて行かれたことを思い出します。 後に製品を「Javaに似た」ものにするために設定されたマニュアルの説得に同意する必要がありましたが、開発では関数型プログラミング言語の重要な特性（ファーストクラス関数など）を保持していました。 彼女は（誇張ではあるが）「C構文を使用したLISP」と呼ばれた。



インターネットには無数の例があります。そのおかげで、JSの助けを借りて、命令型、オブジェクト指向型、または単に嫌なスパゲッティコードを書くことができることが明らかになりました。 ただし、JavaScriptをCに似た他のプログラミング言語と比較すると、真の関数型言語を作成するための優れたプロトタイプになります。



2011年に、機能コマンド「stalwarts map」および「reduce」がJS（技術的にはECMAScript）に追加されました。 簡潔な構文と完全に合理的な構造を備えたECMAScript 2015（ECMAScript 6）では、オブジェクトの作成やユーザーが気に入った太字の矢印など、他の多くの機能的なツールが登場しました。 ES2015のすべての機能がブラウザでサポートされているわけではありませんが、Babelトランスコンパイラのおかげで、ES2015機能（さらにはES2016）を使用し、結果をユニバーサルES5の形式でコンパイルできるため、これは重要ではありません。





通常の反応



AngularJSはJavaScriptブームの始まりを示しましたが、製品はJavaとOOPに根ざしているため、機能開発のランクに記録すべきではありません。 過去数年にわたって、Facebook Reactは優れた高速なクライアント側JSフレームワークに進化しました。 はい、FluxアーキテクチャとImmutable.jsライブラリの印象的なパフォーマンスがなければ、これもFacebookチームのメリットの1つです（機能的なJSの開発は、FacebookがJSフレームワークの市場からGoogleを追い出す方法とどのように一致するのでしょうか）。



Reactは、Webアプリケーションの開発に対して非常に機能的なアプローチを作成しているようです。 実際、Reactの最新バージョンでは、統合によって副作用が発生しない純粋なコンポーネント用の単純化された構文が導入されました。



ただし、Reactはデータの表示のみを専門としているため、FacebookはFluxを使用してアプリケーションアーキテクチャを拡張することを推奨しています。 Fluxは基本的な要素（ストレージ、アクション、集中型サービスマネージャー、単方向データフロー）のセットにすぎないため、この開発と同時に、競合他社は迅速に独自の代替案を作成し、分野のリーダーであると主張しました。 さらに、Reduxが登場する前は、このような平等な戦いで勝者はいませんでした。



Reduxメカニズムは、次の原則に基づいています。単一の集中データウェアハウス、状態の変更は、特定のアクションと純粋な機能（リデューサー）の使用の結果としてのみ可能です。



Immutable.jsと組み合わせて使用​​すると、ReduxはほぼClojure Omの双子になります。 理論的には、関数型プログラミングとJavaScriptをいつでも組み合わせることができましたが、React、Redux、Immutable.jsは、最終的にJS開発者に問題の実用的な解決策を提供しました-副作用のないクリーンなレデューサーの開発により、ES6構文のフレームワーク内で製品の使い慣れた状態を変更できます-大規模な機能アプリケーション。 このアプローチの人気（したがって、関数型プログラミングの普及）は驚くべきものです。数か月前にReduxでないことに初めて気付いたとき、GitHubには約500個の星がありました。 今日-11,000以上。

JavaScript機能への関心が高まり続けると主張するのは時期尚早です。 最近、JSは不安定な期間を繰り返し経験しているため、例外的であると主張する数え切れないほどの新しいフレームワークが突然登場し、かなり面白く見えました。 しかし、関数型プログラミングの豊かな血統と新しいトレンドの根源は非常に説得力があります。 また、個々のプロジェクトは失敗しますが、ソフトウェア開発の未来は機能しているようです。
