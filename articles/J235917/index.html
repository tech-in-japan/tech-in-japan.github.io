iOS用のhabraklaviaturaを開発します



多くの場合、Habrを読むには、iPhoneおよびiPad用のHabrahabrモバイルアプリケーションを使用します。 記事を読むのには十分便利ですが、コメントを書くのにはあまり便利ではありません。特にフォーマットタグを使用してそのようなものを書きたい場合はそうです。 不便です。すべてのタグを手動で入力する必要があるため、間違いを犯しやすく、その結果、commentいコメントを残してしまいます。



そこで、キーを押すと、開始タグと終了タグがテキストフィールドに追加される、独自のキーボードを作成するというアイデアがありました。 この場合、テキストの書き込みをすぐに開始するには、カーソルをそれらの間に直接置く必要があります。 また、スワイプジェスチャーを使用してカーソルを移動できるようにする必要があります。主観的に、フィールドに指を引っ張り、虫眼鏡が表示されるのを待って、指を動かし、カーソルが必要な場所に移動することを期待するよりも便利です。 最後に、HabrパーサーでサポートされていないSarcasmタグとBoreタグを処理します。 キーボードにはこれらの目的のための特別なキーが必要であり、タグのデザインはキーボード設定で構成可能で、誰もが好みの外観を指定できるようにする必要があります。



iOS 8のリリースにより、Appleはアプリケーションの拡張機能を開発できる新しいAPIを開きます。 キーボード（カスタムキーボード）は、このような拡張機能の代表の1つです。 それについて説明します。 この記事では、新しいAPIが提供する機能、制限、バグ、habraclavキーボードの開発方法、キーボードをAppStoreに表示する方法、したがってユーザーのデバイスに表示する方法について学習します。



機能と制限



サードパーティのキーボードを作成するためのAPIへのアクセスを開いたAppleは、アプリケーション間の狭いブリッジを構築しました。 一方では、各アプリケーションはまだ独自のサンドボックス内にありますが、他方では、一方のアプリケーションに入力されたデータをもう一方に移動したり、サーバーに直接送信したりできます。 この機能は、ユーザーデータのセキュリティの観点から非常に深刻であるため、Appleは何ができて何ができないかを厳密に定義しています。 詳細な説明に移る前に、キーボードを含むすべての種類の拡張機能は、メインアプリケーション（コンテナーアプリケーション）の一部としてのみモバイルデバイスにインストールできることを明確にしたいと思います。 たとえば、通知センターのHabrahabrアプリケーションの最新バージョンにウィジェット拡張が追加されました。



そして、私たちにはどんな機会がありますか：

  • サードパーティのキーボードは、デバイスにインストールされているほとんどすべてのアプリケーションで使用できます。 これを行うには、ユーザーはデバイス設定にインストールされたキーボードを手動で追加する必要があります。 幸いなことに、アプリケーション開発者はアプリケーションでサードパーティのキーボードの使用を禁止できますが、それについては後で詳しく説明します。
  • キーボードは、コンテナアプリケーションと開発者サーバーの両方とデータを交換できます。 このような交換の許可はユーザーによって設定され、この機能は最初は無効にされていました。
  • キーボードは、位置情報サービスとアドレス帳にアクセスできる場合があります。 アクセスするには、キーボード設定と対応するサービスの設定の両方でユーザーの許可が必要です。
  • キーボードは、内蔵辞書にアクセスして、入力されたテキストの自動修正および補完のオプションを表示できます。 データは次のソースから取得されます。
    1. デバイスアドレス帳（姓と名はペアで提供されません）;
    2. スマートフォンの設定でユーザーが指定した略語。
    3. 一般的な語彙。


実際、これらはすべて主要な機能です。 制限に移りましょう：

  • 標準キーボードから継承することはできません。 つまり、組み込みのキーボードを基本として使用し、ジェスチャ処理を追加してカーソルを制御すると失敗するため、最初からすべてを行う必要があります。 さらに、標準キーボードは原則として再作成できません。 その理由は次の段落で説明されています。
  • 他のタイプの拡張機能と同様に、サードパーティのキーボードはマイクにアクセスできないため、音声入力をサポートできません。
  • サードパーティのキーボードを使用して非表示のテキストを入力することはできません（ secureTextEntry = YES



    ）。 つまり、フィールドがパスワードの入力を目的としている場合、ユーザーは標準キーボードのみを使用できます。
  • カスタムキーボードは、タイプUIKeyboardTypePhonePad

  • キーボードには、入力フィールドのテキストを強調表示するアクセス権がありません。 つまり、ユーザーの介入なしに選択を変更しても機能しません。
  • 標準のキーボードとは異なり、サードパーティのキーボードはフレームから抜け出せません。 つまり、たとえば標準キーボードの一番上の行のキーを長押しすると、キーボードの上に何も表示できません。

デモンストレーション







転記



キーボードと他の種類の拡張機能の両方の公開プロセスは、通常のアプリケーションの公開と実質的に違いはありませんが、拡張機能自体がいくつかの技術要件を満たしている必要があります。

  • 拡張機能は、コンテナアプリケーションの一部としてのみ公開できます。 単独では存在できません。 アプリケーションがアンインストールされると削除されます。 ところで、この制限はiOS専用です。 Macの場合、拡張機能は個別に配布できます。
  • 拡張機能の展開ターゲットは、> = iOS 8.0でなければなりません。 メインアプリケーションは、以前のバージョンのオペレーティングシステム用に公開できます。 古いバージョンでは、拡張機能は使用できません。 つまり、たとえば、Habrahabrアプリケーションの開発者がこのようなものを実装する場合、Deployment Targetを上げる必要はありません。
  • 拡張のターゲットには、64ビットアーキテクチャ（arm64）のプロセッサのアセンブリの設定が含まれている必要があります。 アプリケーションが拡張機能とのデータ交換を実装する場合、同じ要件を満たす必要があります。 好きなことを言ってください。しかし、64ビットプロセッサのサポートについて考える必要があります。 複雑で低レベルのコードを持つ大規模なプロジェクトの場合、これには非常に時間がかかる場合があります。


Appleはまた、レビューガイドラインにいくつかの新しい段落を追加しました。 それらはキーボード拡張に固有です：

  • キーボードを開いたら、ユーザーは次の画面に進むことができます。
  • インターネットに接続していない場合、キーボードは操作可能な状態のままにしてください。
  • キーボードは、次の種類のキーボードを実装する必要があります。数値と10進数（つまり、入力フィールドに数値の入力が含まれる場合、キーボードはこのための便利な方法を提供する必要があります）。
  • キーボード拡張を含むアプリケーションの主要なカテゴリは、ユーティリティである必要があります。 また、アプリケーションはユーザーに独自のプライバシーポリシーを提供する必要があります。
  • キーボードは、機能を向上させるためにのみユーザーデータを使用する必要があります。


元の推奨事項
  • キーボード拡張は、次のキーボードに進むための方法を提供する必要があります。
  • キーボード拡張機能は、ネットワークアクセスなしで機能し続ける必要があります。機能しない場合は拒否されます。
  • キーボード拡張機能は、App Extension Programming Guideで説明されているように、数字および10進キーボードタイプを提供する必要があります。そうでない場合、拒否されます。
  • キーボード拡張機能を提供するアプリには、ユーティリティのプライマリカテゴリとプライバシーポリシーが必要です。そうでない場合、拒否されます。
  • キーボード拡張機能を提供するアプリは、iOSデバイス上のキーボード拡張機能を強化するためにユーザーアクティビティのみを収集するか、拒否される場合があります。


上記のすべての要件が満たされている場合、基本的に既知のいくつかの手順を実行します。

  1. 開発者ポータルにアクセスして、拡張機能のアプリIDを作成します。 メインアプリケーションの識別子の続きである必要があります。 たとえば、アプリケーションのアプリIDがcom.company.application.keyboard



    場合、拡張機能のアプリIDはcom.company.application.keyboard



    ます。 次に、新しいアプリIDに対して、プロビジョニングプロファイルを作成する必要があります。 このデータは、Xcodeのターゲット設定で指定する必要があります。
  2. 拡張機能がメインアプリケーションとのデータ交換を実装する場合、開発者のポータルでアプリグループを作成し、拡張機能とアプリケーションのアプリID設定でアプリグループを使用できるようにする必要があります。 実際、このステップはアプリケーションのテスト段階で実行する必要があります。そうしないと、データ交換が表示されません。
  3. iTunesConnectで拡張機能のスクリーンショットと説明を追加します。 Appleは、これらの目的のための特別なフィールドを追加していません。 これらのデータは、メインアプリケーションに関する情報とともに入力されます。
  4. メインアプリケーションのターゲットを収集し、レビューに送信します。 拡張機能は既にアプリケーションパッケージに含まれています。
  5. それだけです、待つことしかできません。
