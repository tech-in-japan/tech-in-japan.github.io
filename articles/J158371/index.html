フィルタを定義しましたが、まだアクションを割り当てていません。 なぜなら 特定のスクリプトを実行する必要がある場合は、フィルターにアクションを追加します。

ここで、このテキストを/etc/udev/rules.d/ディレクトリ（ルート権限が必要）に任意の名前で保存します（99-lumix.rulesがあります）。 udevデーモンは、条件に一致するフィルターが見つかるまで、このディレクトリからのファイルをアルファベット順に処理します。 適切なルールを見つけた後、ファイルのそれ以上の処理は実行されません。 したがって、ルールの数は小さく設定でき、より早く処理され、他のルールによってブロックされません。

そして今、私たちは接続されたカードから写真をソートするスクリプトを書いています。 元のバージョンでは、ディスク全体のソートのみがサポートされていました。 運用中に、既存のディレクトリ（蓄積された未ソート）も処理したかったので、少し変更して追加する必要がありました。 ここでは、ディスクとディレクトリの両方からメディアファイル（メディアファイルの拡張子はスクリプトのfindコマンドパラメーターで指定されます）を並べ替えることができる2番目のバージョンを検討します。



まず、このスクリプトのアルゴリズムについて簡単に説明します。 処理するもの（ボリュームまたはディレクトリ）を決定します。 ボリュームの場合は、最初にマウントする必要があり（マウントポイントが既に取得されており、このスクリプトでのみ使用されている場合、スクリプトは作業を完了します）、最後にアンマウントすることを忘れないでください。 マウント後、このカードの設定が保存されている隠し.PhotoSortファイルの存在が確認されます。 この不在は、通常のカードが挿入されたことを示す信号として機能し、何もする必要はありません。 このファイルでは、ディスクの名前（ロギングのみに必要-最後の説明）と、標準が適合しない場合はファイルを保存するディレクトリを指定できます（携帯電話用に別のストレージを作成することを考えました）。 エクスポートディレクトリは、コマンドラインまたはudevルールファイルでスクリプトの2番目のパラメーターとして指定することもできます。



次に、指定されたディレクトリとサブディレクトリのすべてのメディアファイル（カメラが以前は知らなかった異なるディレクトリに写真を保存するため）は、作成された時間でソートする必要があります。 落とし穴があります：Linuxでは、ファイルが作成された時間などはありませんが、ファイルへの最後のアクセスの時間しかありません。また、画像を操作するときに更新されるため、信頼することはできません。 ファイル名も不可能です。02.05で作成されたDSC08655.JPGは、29.04からMOV08554.MPGの後に続き、19.04からP1170007.JPGの後に処理されます。 この問題は、1つのイベント（別名「パーティー」）を撮影したさまざまなメーカーの複数のカメラがある場合に特に深刻です。 EXIFは私たちの助けになります-特に、DateTimeOriginal属性とCreateDate属性があります（これらの間にいくつかの違いがあります：少なくともビデオファイルには最初のファイルがなく、何らかの理由で読み取られなかった場合、2番目のファイルが読み取られます）通常の状況での属性は変更されません。 しかし、特定のディレクトリから開始し、これらの属性でソートされたすべてのネストされたディレクトリからファイルを読み取る方法がわからないため、すべてのファイルの名前がループで変更されます（exiftool自体がこれを行うことができますが、ログ用のループを残しました）、スナップショットまたはビデオを作成する時間を追加します名前で（後でファイル名で時系列に並べ替えることができるように）。



2番目のサイクルでは、ファイルをディレクトリに配布します。現在のファイルと前のファイルとの区切りが5時間（5 * 60 * 60）を超えると、新しいディレクトリが作成されます。これが、ファイルを時系列で並べ替えた理由です。 5時は天井からの数字です。 あるイベントの写真がなかったので、ショットの間隔がこの時間よりも長くなりました。 POI間の長い旅程で小旅行がありましたが、道路は5時間もかからず、もしそれ以上であれば、すでに別の都市でした。 別の日に通過する結婚式や休日があり、日付は変わりましたが、イベントは同じままでした。 したがって、小さなブレークは前のファイルと同じディレクトリに保存されます。 ディレクトリ名はYYYYMMDD_HHとして設定されます-クロック（新しいイベントの最初のファイルから取得）は、同じ日の2つのイベントのためにディレクトリ名に追加されます。そうでない場合、2日目に行くと同じディレクトリ名を取得しますが、作成は失敗しますが、さらに保存されますファイルは同じディレクトリに実装されます。



発生するすべてがログに書き込まれます（これがスクリプトの機能を確認する方法です）。また、ファイルをディスクに戻すコマンド（デバッグに役立ちました-ファイルの一部を実行し、テストメディアファイルを古い名前でマップに戻しました）。 追加の処理（回転、縮小、署名）-私は使用しません。とにかく、それらを見て、削除して、すべてを実行できます（写真に署名を​​付けません）。 これを実行できる場所は、スクリプトテキストのコメントで示されます。

使用する
スクリプトは、指定されたディレクトリとその中に含まれるすべて（メモリカードの場合はそのルートであり、すべてがより深い）を1つとして処理します。 



 ./PhotoSort.sh sdh1
したがって、sdh1ドライブをソートするために、スクリプトがudevから呼び出されます。この場合、ルートに設定があるファイルがチェックされます。ディスクは私たちに与えられた、またはディレクトリは最初の文字によって決定されます：〜/ orの場合。 -ディレクトリを意味し、そうでない場合はディスクを意味します。 



 ./PhotoSort.sh ~/AllFromParty
そして、〜/ AllFromParty（自分を含む）で囲まれたすべてのディレクトリからすべての写真とビデオファイルをソートできます。すべては、スクリプトで指定されたデフォルトディレクトリに保存されます（photo = "/ mnt / temp / Photo"）。ソースディレクトリに複数のカメラからのファイルが含まれている場合、カメラ間の時間が同期されていないか、設定されていないために、ファイル名に矛盾がある可能性があります。これは、異なるカメラから異なるディレクトリへのファイルの配布を引き起こす可能性があります。次に、並べ替える前に、EXIFタグで時間を調整する必要があります。exiftool "-DateTimeOriginal + = 00.00.0000 02:37:30" * .JPG-現在のディレクトリのすべてのJPGファイルについて、DateTimeoriginalタグの時間を将来2:37:30にシフトします。 



 ./PhotoSort.sh ~/AllFromParty /media/backup/Photoes
上記と同じですが、すべてが/ media / backup / Photoesディレクトリに保存されます
