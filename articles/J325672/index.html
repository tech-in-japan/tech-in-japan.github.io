Apache Igniteプロジェクトはますます頻繁に聞かれます。 しかし、その開発者の1人であるVladimir OzerovがHabréで以前に指摘したように、一言で言えばプロジェクトを説明することは困難であり、その結果、多くの人が最も基本的なものから始めてまだ疑問を持っています。 一般的なプロジェクトとは何ですか？ Apache IgniteとGridGainはどのように比較されますか？ インメモリデータグリッドとインメモリデータファブリックの概念はどのように関連していますか？

ウラジミール・オゼロフ（建築家）





-簡単なものから始めましょう。プロジェクトとは何ですか。また、類似の製品とはどう違うのですか？



-Apache Igniteは、分散インメモリコンピューティングのためのユニバーサルプラットフォームです。 約10年前に登場した最初のコンポーネントは、分散キャッシュ（データグリッド）とmap-reduce（計算グリッド）でした。 当時、私たちと競合他社の両方は、ソリューションを「分散キャッシュ」として位置付けていました。 時間が経つにつれて、ビジネス要件が増大し、新しいモジュールが追加されました。 主なもの：ANSI SQL'99のサポート、ストリーミングデータの読み込み（ストリーミング）、クラスターへのユーザーアプリケーションの展開（サービスグリッド）、サーバー上のJavaクラスなしでオブジェクトを保存する機能（バイナリオブジェクト）、多数のサードパーティインターフェイスとの統合、プラットフォーム（Webセッション、Springキャッシュ、Hibernate L2キャッシュ、Hadoop、Spark、Kafka、JDBC、ODBCなど）。



ある時点で、「分散キャッシュ」と「インメモリデータグリッド」の確立された概念は、製品の本質を反映しなくなりました。 その後、「インメモリデータファブリック」という用語が登場しました。これは、メモリ内の分散コンピューティング用のソフトウェアであり、さまざまなデータソースとその消費者向けの多数のインターフェイスを備えています。 これが、「インメモリデータグリッド」という用語を超えることを急ぐことのない類似のソリューションとは一線を画しています。



-プロジェクトはすぐにオープンソースになりませんでした-これはどのように起こりましたか？



-最初は、この製品は閉じた商用GridGainソリューションとして開発されました。 オープンソースの可能性を認識して、ほとんどのソースコードをGitHubで公開しましたが、開発は引き続き会社の手に委ねられました。 これは一般的なビジネスモデルです。外観、使用、干渉はありません。 しかし、さらに進んだ。 2015年、オープンソースはApache Software Foundationに引き渡されました。 決定は容易ではありませんでした。コードの制御を失い、新しい名前を探すことを余儀なくされました。 そのため、 Apache Igniteがありました 。 Apacheブランドと今や正直なオープンソースがプロジェクトに深刻な刺激を与え、その人気を高めているという事実に賭けられました。 そして、私たちは間違っていませんでした。ユーザーの数は指数関数的に増加し始めました。



さらに興味深いことに、人々はプロジェクトに積極的に投資し始めました：アイデアの生成、バグの修正、ドキュメントの改善、新しい機能とモジュールの開発、本の執筆、会議での講演。 これは素晴らしいです！ 私たちは、新しい貢献者を引き付ける努力を続けています。 このインタビューを読んだ人は誰でも、本当に複雑なシステムの開発経験を積むことで貢献できます。



-それは素晴らしいように聞こえますが、このスキームのお金はどこにありますか？ GridGainとApache Igniteはどのように比較されますか？



-商業企業は、特に重要なビジネスアプリケーションにおいて、クリーンなオープンソースで作業する余裕はほとんどありません。 誰が不足している機能を追加し、すぐにバグを修正し、質問に答えますか？ これらはすべて、基本的な商用製品であるGridGain Professional Editionの一部として利用できます。 これは、Apache Igniteコードベースとサポートおよびホットフィックスです。



さらに、ビジネスは多くの場合、セキュリティ、回復力、および高可用性に高い要求を課します。 この機能は、GridGain Enterprise Editionに実装されています。 Community Editionに加えて、認証とユーザー承認（セキュリティ）、データセンター間のデータレプリケーション（データセンターレプリケーション）、クラスターを停止せずにGridGainバージョンを更新する機能（ローリングアップグレード）、システムステータスを監視するユーティリティなどの追加機能が含まれます。 。



-製品を把握しました。技術的なコンポーネントについて話しましょう。 研究開発はどのようにしていますか？



-技術的な議論は、開発シートで公開されています。 タスクを設定し、意見を収集してから、アーキテクチャと特定の行動計画を決定します。 各コンポーネントには、対応する機能を熟知しているメンターがいます。 困難な場合、すべてをヒープに収集するのは彼です。 十分な情報に基づいた意思決定を行うには、多くの場合、文学や科学出版物を扱う必要があります。 単に既製の答えがない場合があり、あなたは先駆者になります。 これは私たちの仕事をエキサイティングにします。



-コードをどのようにテストしますか？



-これは複数レベルのプロセスです。 開発者がコードを記述し、テストでカバーします。 さらに、これは単体テストだけでなく、統合テストでもあります。 コードを分離してチェックし、1つのノードで、複数のノードでマルチスレッドと障害をシミュレートします。 テストはパブリックCIサーバー（TeamCity）で実行されます。 次に、コードは必須のピアレビューを通過します。 プロセスの2番目の部分はQA部門によって管理されます。 ほとんどの場合、自動（Python、Java）のテストケースのセットを通じて機能を実行します。 これにより、必要なレベルで品質を維持できます。



-「誰でも貢献できる」という質問については、プロジェクトは普通ではないので、結局のところ、全員ではないという感じがあります。 GridGain開発者の要件は何ですか？また、どこで人を見つけていますか？



-実際、Apache Igniteはハイテク製品であるため、基本的な知識が重要です。 古典的なアルゴリズム、データ構造、マルチスレッドで始まり、分散システムとDBMSの内部構造の作成の原則で終わります。 そのような手荷物を持つ人を見つけるのは難しい作業です。 私たちはこれを理解しており、したがって、私たちは優れた基盤を持つ人々を探しており、すでに戦いの中で適切なレベルに引き上げています。 プロセスのオープン性は非常に役立ちます。 従業員は誰でもdevリストのアーキテクチャに関する議論に参加できます。 これに加えて、かなりフラットな組織構造があり、誰もがコードを書いており、これも適応に貢献しています。



アレクセイ・ドミトリエフ（ロシア支部長/エンジニアリング担当副社長）

-Sberbankのような大規模な顧客がいますか？また、あなたのソリューションは、予算の大きい巨人や小企業にも適していますか？



-このソリューションは誰にでも適しています。 Sberbankには大量のデータがあるため、巨大なハードウェアコンプレックスを使用して、そのようなデータをすばやく操作します。 原則として、中小企業はデータが少なく、コストははるかに控えめです。 さらに、この製品では、データへのアクセス頻度でデータを区別し、アプリケーションが最もアクセスするデータのみをRAMに保持できるようになりました。 まあ、そして最も重要なこと：私たちの製品は、他のマルチプロセッササーバーだけでなく、シンプルで安価なオフィスコンピューターでも動作します。 それらを同じネットワークに接続し、GridGainを配置するだけです。 高価な特定の「ハードウェア」は必要ありません。

私の意見では、Sberbankは、世界のITクライアントバンクの分野で最も急速に成長している分野の1つです。 彼は、ヨーロッパおよびロシアの多くの企業のロールモデルです。 ロシアの潜在顧客の多くは、Sberbankとの共同作業を注意深く見ているため、販売の観点から、この市場は私たちにとって最も重要なものの1つであると思います。
