Win-Win：SPB .NETコミュニティ会議でのDotNextスピーカーとその逆

6月は、 SPB .NETコミュニティプログラマの.NETコミュニティの年です。 私は何が起こったのかを要約して伝えません。なぜなら、その前ではなく、新しい会議の準備に忙しくて、12番目です！ 6月2日木曜日に、DotNext会議の主演講演者である.NETパフォーマンスの第一人者Sasha Goldshteinと.NETユーザーグループMinsk Sergey Shchegrikovichの積極的な参加者を招待しました 。 私たちが大好きなトピック：パフォーマンス、メモリトラフィック、デバッグ、CoreCLR、内部、切断...











SPB .NETコミュニティ



したがって、6月2日に、パークインバイラディソンプルコフスカヤホテルの会議ホールで、SPB .NETコミュニティの会議が開催され、DotNext会議のスピーカーが私たちの場所で話をします： サーシャ ゴールドシュテインとセルゲイシュチェグリコビッチ 。 一週間前、私は提案されたサーシャからのトピックの選択を伴う調査を発表しました 。 .NET Garbage Collection Performance Tipsに投票した人のほとんどが、民主主義の実際の例になりました。 GCの内部構造に関する質問に関心のあるすべての人を待っています。マネージドメモリの操作におけるボトルネック、メモリトラフィックを最小限に抑える方法、およびSasha Goldsteinによるスピーチのその他の微妙なポイントを見つけます。 それまでの間、私は過去のDotNextで最高と認められた彼のレポートを見ることをお勧めします。







そして、 Sergey Shchegrikovichは、レポート「CoreCLR：Brave new world」で会議を開きます。 このレポートでは、CoreCLRの調査と作業の経験を共有し、CoreCLR、SOS、およびファイナライザーのデバッグプロセスを詳細に検証します。 セルゲイはミンスクから会議に飛び、新しい資料を持って私たちのところに来ることに同意しました。 したがって、私たちの会議はすでに国際的な性格を獲得しています。



会議への参加は無料で、座席数は限られています。 サインアップ！



ドットネクスト



翌朝、同じ部屋で、ロシアで最もハードな.NET会議DotNextが始まります 。 20以上のレポート、3つのスレッド、主催者、プログラム委員会、そして最も重要なスピーカーの数百時間の仕事！ 経験、知識、努力の真髄。 これについて書いているのは、内側のキッチンを見て、このイベントの準備に参加したからです。 たとえば、非常に強力なプログラムで起こったことに満足しています。



DotNextカンファレンスのスピーカーの中には、私たちのミーティングで最初の報告をした人たちがいます。 最初の会議の目標の1つは、舞台に立ち、話したい人全員への報告書の作成を支援することでした。 本当にクールな開発者の多くは、どこから始めればよいかが明確でないという理由だけで、自分の経験やプロジェクトについて語りません。 そして、この状況では、私の意見では、誰もが負けています。 コミュニティでは、本で読むことのできないユニークな体験を交換することはありません。また、開発の新たな地平を見る人もいません。 彼らのアイデアのコミュニケーションと議論は、しばしば最も予期せぬ方向に弾みをつけます。 そして、人前で話すことは別の種類の薬物であり、依然として合法です。 公演後、あなたはもはや止められないほどのエンドルフィンの波にびしょ濡れになります。



ウィンウィンゲーム



DotNextの過去以来、SPB .NETコミュニティのスピーカーが選ばれ、優れたプレゼンテーションを提供してきました。 今回はイゴール・ラブチン、ローマン・ネボリン、ウラジミール・コシェレフです。



Igor Labutinがトピック「WCFの診断」を発表します 。これは、SPB .NETコミュニティの第10回会議からの彼の報告の続きです。 今回、イゴールは、WCFアプリケーションの問題を発見から診断する方法について、明らかなものから非常にハードコアなものまで説明します。 あなたの人生が血なまぐさい企業であり、WCFが「ああ、捨ててREST / WebAPI / HipstoServicesですべてを書き換える」のではなく、あなたにとって実用的なツールであるなら、このレポートはあなたのためです。 「WCF：便利ですか、それとも問題ですか？」というビデオをご覧ください。また、多くの便利なことがあります。







ローマン・ネボリンは、 「 。 NETプラットフォームでの機械学習について」というトピックを紹介 します 。 これは、.NETプラットフォームでの機械学習の可能性と機能についての講演の続きです。 今回は、.NETおよび機械学習の機能を示す活気のあるデモに重点を置いています。 そして、もちろん、パフォーマンスとベンチマークなしで何をすべきか。 Romanのレポートの最初のバージョン：







ウラジミール・コシェレフ「すべての静的アナライザーが同じように役立つわけではありません 。 」 講演の最初の部分は、ガイドラインの違反、コピー＆ペーストのエラー、ソースコードのタイプミスを探す一般的なツールに焦点を当てます。 2番目の部分は、NullReferenceExceptionやリソースリークの発生などのより複雑なエラーと、それらの検出方法についてです。 Coverity Preventなどの純粋な静的分析と、IntelliTest（Pex）などの静的-動的分析の両方が、このようなエラーの検出に役立ちます。 これらの分析の類似点と相違点、それらが相互に交換されないが相互に補完する理由、および生産におけるそれらの使用について説明します。 私たちの会議で、ウラジミールは.NETアプリケーションの静的解析方法に関する概要レポートを読みました。 その後、ISP RAS、JetBrains、Positive Technologiesの専門家と静的アナライザーの使用の実践と開発における問題についてのラウンドテーブルを開催しました。 残念ながら、円卓会議のビデオは存続しませんでした（直接会議に出席するもう1つの理由）。また、ウラジミールのレポートのビデオを利用できます。
