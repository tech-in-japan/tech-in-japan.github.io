3. SetMatchMode（SPH_MATCH_EXTENDED2）およびSPH_MATCH_ANYなどが実際の検索に適さない理由



形態が機能し、クエリ「Castrol 5W40 oils」には、テキスト「Oil」と「15W40」のドキュメントがあります。「*」記号を使用し、単語「oils」で検索する必要があります。このためには、クエリビルダーが必要です。 「SPH_MATCH_EXTENDED2」モードです。 SPH_MATCH_EXTENDEDもありますが、私が理解しているように、それは古いバージョンであり、モードの新しいバージョンを使用することをお勧めします。



SPH_MATCH_ANYモードでは、クエリ内の単語が増えると結果の数が減ることを実現することはできません。 SPH_MATCH_ALLモードでは、部分的な出現と単語形式によるモードの同時操作を実現することはできません。 便利になるまで他のモードを何気なく見ていたので、それらについては何も言えません。



「Castrol 5W40 oil」というフレーズでのスフィンクスへのリクエストは次のようになります。 

 ( | **) & (Castrol | *Castrol*) & (5W40 | *5W40*)




重要：使用されるインデックスの設定には2行が必要です： 

 min_infix_len = 3 enable_star = 1


最初の方法では、右側と左側の単語の部分的な出現、つまり単語の末尾と先頭から検索できます。 2行目では、リクエストで「*」を使用できます。 たとえば、単語の左側（先頭から）の出現のみが必要な場合は、min_prefix_lenを使用できます。



入力文字列を処理し、正しいリクエストを生成する関数： 

 private function GetSphinxKeyword($sQuery) { $aRequestString=preg_split('/[\s,-]+/', $sQuery, 5); if ($aRequestString) { foreach ($aRequestString as $sValue) { if (strlen($sValue)>3) { $aKeyword[] .= "(".$sValue." | *".$sValue."*)"; } } $sSphinxKeyword = implode(" & ", $aKeyword); } return $sSphinxKeyword; }

また、フォーラムと開発者のウェブサイトで非常によくある質問は、「フレーズの正確な出現回数を増やす方法」です。つまり、ドキュメント「Wanted_word」の重みは「Wanted_wordおよび多くのテキスト」よりも高くなります。 回答-SPH_RANK_SPH04を使用する必要があります。SPH_RANK_SPH04は、この典型的なタスクのために特別に作成されたものです。



4. SetSortMode（SPH_SORT_RELEVANCE）、SetFieldWeights-関連性によるソートとインデックスフィールドの重みの設定



このメソッドは、スフィンクスによって返されたデータのソートされた配列の中で、どの結果がより高いかを決定します。 SPH_SORT_RELEVANCEの場合、結果はいわゆるソートされます。 「関連性」。 関連性は、望まない限り、純粋に算術規則に従って機能し、GoogleやYandexの検索とは異なります。 つまり、魔法のようなものではありません。インデックスの重み、フィールドの重み、ドキュメント内での検索ワードの出現回数、および他のドキュメント内でのこのワードの頻度の乗算と加算。



最も単純なケースでは、インデックスフィールドの入力ウェイトを設定します。 

 $oSphinxClient->SetFieldWeights(array ( 'code' => 50, 'brand' => 40, 'part_name' => 10, 'price_group_name' => 5, ));


出力では、「関連性」=「総重量」でソートされた配列を取得します。ここで、重量は整数値です。 これらの数値は、自分自身の関連性を調整することで制御できます。つまり、より重要なフィールドにはより多くの重みを割り当てる必要があります。 この例では、最も重要なフィールドはコードcodeです。



5. SetLimits（0.20）-結果の出力の制限



この方法は最も簡単で、Maysklevskyの制限0.20と同様に機能し、それに応じて必要です。つまり、ステッパーを構築するための分割データを取得します。 このプロジェクトでは、最初の20（一定）の結果が必要です。3つ以上ある場合、それ以上の手順はありません。



6. AddQuery、RunQueries-複数クエリの構築



Multiseriesは、スフィンクスに1つのリクエストではなく、複数のリクエストを送信する必要がある場合に、リクエストをバッチ処理するための非常に便利なソリューションです。 この例では、すべてのパーツグループに同じリクエストを送信して、グループのリストと各グループのレコード数を取得します。 つまり、約100件のリクエストが送信され、1つの結果がsphinxへの1つの接続で返されます。 また、1つのリクエストパッケージで許可される同時リクエストの最大32個の「制限」が「解決」されました。



コード例： 

 $aPriceGroup=Db::GetAll(Base::GetSql("Price/Group",array( 'visible'=>1, "where"=>" and pg.code_name is not null", ))); if ($aPriceGroup) { $aResultAll=array(); $i=0; foreach ($aPriceGroup as $aValue) { $oSphinxClient->SetFilter('id_price_group', array($aValue['id'])); $iQuery = $oSphinxClient->AddQuery($sSphinxKeyword, 'price_group'); $oSphinxClient->ResetFilters(); $bAddedUnrunQuery=true; $aPriceGroupAssoc[$iQuery+(32*$i)]=$aValue; if ($iQuery && !($iQuery % 31) ) { $aResultQuery=$oSphinxClient->RunQueries(); $aResultAll=array_merge($aResultAll,$aResultQuery); $sLastError=$oSphinxClient->GetLastError(); $i++; $bAddedUnrunQuery=false; } } if ($bAddedUnrunQuery) { $aResultQuery=$oSphinxClient->RunQueries(); $aResultAll=array_merge($aResultAll,$aResultQuery); } }


進行中のタスクには期限があったため、開始時にすべての複雑な要求を掘り下げるために、タスクは設定されませんでした。 したがって、mysqlのgroup byに似た「グループ化された」クエリの問題を解決する自転車をおそらく書いたでしょう。 一方、スフィンクスのグループ化を理解した場合、マルチクエリを使用できる例はありません。



そのため、コメントでは、より正確なリクエストの例を歓迎しますが、これはスフィンクスのグループ化によるものです。



7. SetFilter、ResetFilters-マルチリクエストにフィルタリングを追加して、受信データを制限します



フィルターを使用するには、最初にフィルターを使用するフィールドをインデックスインデックスフィールドに登録する必要があります。 この例では、これはid_price_groupフィールドです。 

 sql_attr_uint = id_price_group


したがって、コードは次のように使用されます。 

 foreach ($aPriceGroup as $aValue) { $oSphinxClient->SetFilter('id_price_group', array($aValue['id'])); $iQuery = $oSphinxClient->AddQuery($sSphinxKeyword, 'price_group'); $oSphinxClient->ResetFilters(); //... }


つまり、foreachループでは、マルチリクエスト内の各リクエストに対して、フィルターが最初に設定され、追加後に他のリクエストに対して機能しないようにリセットされます。 私の意見では、すべてが論理的で明白であり、困難は生じないはずです。



8.単語形式-同義語を使用し、「C＃」などの非標準の単語形式の制限を克服する



シノニムと非標準の（独自の）単語形式を機能させるには、次の単語形式のファイルをインデックス構成に含める必要があります。 

 wordforms = D:\Sphinx\data\wordforms.txt


ファイル自体には、たとえば、UTF-8エンコーディングのデータセットが含まれている場合があります。 

 bosh > bosch  > bosch CASTROL > CASTROLL  > CASTROLL  > CASTROLL


つまり、左側にはすべての可能な同義語があります-これらの単語の右側の意味。 さらに、たとえば、「bosch」が既に右側にある場合、左側にあるべきではありません。 少なくともこれが行われた場合、検索は期待どおりに動作しません。



この例では、「Castrol 5W40 oil」というクエリを使用できますが、「Castrol 5W40 oil」と同じことがわかります。 「C＃」の例では、標準のインデックススキームに従って処理されず、設定したとおりに手動で動作するように、このような非標準の単語形式を含める必要があります。 プロジェクト内のフレーズの意味を正確に知っているのはあなただけです。たとえば、「C＃」=「ミュージシャンのCシャープの前」



サーバー上の設定ファイルとサンプルにはそのような機能はありません。サンプルのみが提供されていますが、プロジェクト同義語の既存の構造にはまだ実装されていません。
