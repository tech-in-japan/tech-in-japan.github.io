バグトラッカーではありませんが...

真のIT従業員は、「“のpor」を料理するのが大好きです。 そして、このおridgeが同僚を養うためにおいしいことが判明した場合、それは一般的に素晴らしいことが判明します。



サービスの結果として、私は常にさまざまなバグおよび問題トラッカー（以下、単にバグトラッカーと呼ぶ）のインストールに対処する必要があり、その中には非常に多くの非標準ソリューションがありました。 私は自分で何かを展開する必要があり、クライアントで何かを「覗き見」しましたが、観察結果を共有することは有用でしょう。







SQADaysカンファレンスですでにこのトピックについて話しましたが 、18分間のビデオを見るのが面倒な人のために、記事ですべてを簡単に説明します。







意味？



では、なぜ非標準のバグトラッカーを使用する必要があるのでしょうか？ これには2つの理由があります。

  • 一般的なコストの節約。 通常、普通の従業員はそのような質問をすることに慣れていませんが、それでもある場所があります。
  • このツールまたはそのツールの非標準的な適用（失敗した場合でも）を試みると、このツールをよりよく研究し、日常のタスクにより効果的に使用できます。




免責事項



さらに進む例は、実際の生活から取られています。 ただし、実装が成功しても、すべてが正常に機能するとは限りません。 考えてみてください、古い同志と相談してみてください。



まあ、私はすぐに、 JIRAの使用に重点が置かれることをすぐにおpoびします（私にトマトを投げることができます） 私はこのシステムを日常的に使用していますが、他のバグトラッカーも忘れられていません-次のほとんどすべてをそれらで実行できます。



行こう！





ヘルプデスク



私が観察しなければならなかったバグトラッカーの最も頻繁な、非常に標準的なアプリケーションは、テクニカルサポートサービスでヘルプデスクとして使用することでした。







このアプリケーションの利点には、開始されたHelpDesk呼び出しとバグのリンクの利便性が含まれます（HelpDeskとの統合は毎日改善されていますが、すべてが理想的ではありません）。2つのシステムを操作するための技術サポートを学ぶ必要はありません。 ただし、このようなシステムは個別のHelpDeskよりもやや不便であることを理解する必要があります。多くの呼び出しを処理する必要がある場合は、そのほとんどがサポートサービスによって決定されるため、個別のHelpDeskシステムを検討することをお勧めします。



ポイントに到達します。 このようなシステムはどのように構成できますか？



最初に、メールで電話を受ける機能を提供する必要があります 。 幸いなことに、多くのバグトラッカーはメールからアプリケーションを作成できます。

  • JIRAの場合、これらは既存のメールハンドラーとすばらしいJira Enterprise Mail Handlerプラグイン（JIRA 5以降に支払われています）です。
  • Mantisの場合-EmailReportingプラグイン
  • Bugzilla email_in.plスクリプトの場合




これらのプラグイン/ハンドラーを使用すると、レターを解析し、作成したアプリケーションの件名行に件名を保存し、レターからアプリケーションに添付ファイルを添付し、対応するアプリケーションに応答メッセージを添付できます。



ただし、HelpDeskを自分で展開する予定がない場合でも、日常業務で電子メールを介してアプリケーションの作成とコメントを使用してみてください。 とても便利だと思う人もいます。



次に、 Webを介したアプリケーションの受け入れを確認する必要があります 。 通常、バグトラッカーでは次のことができます。

  • クライアントの権利を大幅に削減して、クライアントが自分のアプリケーションのみを参照し、内部のキッチンを掘り下げることができないようにします。
  • 自由に販売される箱入りの製品またはサービスがあり、新しい顧客がいつでも表示される場合は、自己登録を開いてください。
  • ユーザーを登録せずに匿名でアプリケーションを作成する機能を有効にします。 ライセンス費用がシステム内のユーザー数に依存する商用のバグ追跡システムがある場合に特に役立ちます。




JIRAの場合、 Issue Collectorプラグインに注目する価値があります。このプラグインを使用すると、Webサイトに後で投稿するための便利なフィードバックフォームを構成できます。







残念ながら、プラグインはまだ湿っていて、すべての場合で確実に動作しない場合があります。



通常、サポートサービスは、厳密に定義された時間内にインシデントを解決する必要があるサービスレベル契約 （SLA）を持つことを意味します。 そして、忘れっぽさを根絶するために、管理者はガイドを提出して、あらゆる種類のリマインダーを設定します。



  • JIRAの場合、定期的にアクティブ化されるサービスとすべての大まかな作業を行うJellyスクリプトに基づいて、 比較的簡単なソリューションを適用できます。 さて、十分な知識があれば、サービスはJavaで記述され、重要なロジックを持つことができます。
  • Bugzillaにはウィニング機能があります。
  • Mantisにはプラグインとスクリプトもあります。




官僚



調整、サッカー、アプリケーション、従業員のマルチレベル階層、自動化のために何が良いでしょうか？ 実際、組織の一部の官僚的側面の自動化は、それを単純化するのに役立たない場合は、より透過的に行います。



バグ追跡システムを使用して、機器の購入の調整、プロジェクト予算の追跡、ドキュメントの操作などを行うことができました。



このようなカスタマイズの基本は、通常、ワークフローの作成、つまりアプリケーションのライフサイクルです。 つまり アプリケーションにどのような条件があり、それらの間でどのような移行が可能か。 さらに、アクセス権を設定することにより、個々の移行は特別に訓練された責任のある同志のみに許可されます。







  • JIRAには、便利で美しい2つのワークフローエディターさえあります:)バージョン5.0以降、美しいグラフィックが主なものになりましたが、古いエディターの下部に移動できます 。 また、上記のワークフローはJIRAからのものであり、コピーされています。
  • Bugzillaには元々アプリケーションのライフサイクルが修正されていましたが、バージョン3.2からは「ねじれ」ます
  • Mantis はコードを少し掘り下げる必要があります
  • Redmineではすべてがタブレットの形で作られています




開発者の日常業務で独自のワークフローを作成するのに役立ちます。 たとえば、「テスト中」や「顧客による受け入れ」などのタスクの新しい中間状態が導入される場合があります。



通常、ワークフローの使用は、いわゆるカスタムフィールドと新しいタイプのアプリケーションの作成と密接に関係しています。 カスタムフィールドは、追加情報の保存に役立ちます。

  • 機能が作成されているクライアント
  • テストするテスター
  • チーム内でデモを開催する中間期間




新しいタイプのアプリケーション（「ドキュメント承認」、「調達」、「設計」、「テスト」など）により、ワークプロセスをより視覚的にするだけでなく、各タイプのアプリケーションに個別のワークフローを添付することもできます。 したがって、「開発」タスクの場合、ライフサイクルに「テスト」状態を導入することが理にかなっているのであれば、「設計」タスクの場合、これは必要ないかもしれません。



メーリングリスト







バグトラッカーは貴重なツールであるだけでなく、通常、メールトラッカーのシステムに組み込まれています。 また、これらの通知を活用して、組織内にメーリングリストを作成できます。 レシピは簡単です：



  • 「ニュースレター用」の別のプロジェクトを作成します
  • プロジェクト内では、ニュースレターのトピックごとに1つのアプリケーションを開始します。 たとえば、「ダイニングルームのスケジュール」または「注意室の運転手」。
  • 官僚制度の程度に応じて、特定の人々/グループは、通知の受信者のリストに強制的に入力されるか、または自分自身でウォッチャー（要求に応じて通知を受け取る）になることができます。
  • アプリケーションに新しいコメントを追加すると、そのテキストが適切な人に送信されます。




リソース共有



同じサーバーが異なる負荷テストで同時に使用されることが多い場合、または2人の従業員がデモンストレーションのために出張で同じiPadを同時に取得しようとする場合は、リソース制御を確実に行う必要があります。



バグトラッカーを使用して同僚間で良心を培うことは困難ですが、組織のレベルを上げることができます。 レシピ：



  • 一度に1人の従業員のみが使用できるリソース（サーバー、車、デモ用iPad）ごとに、1つのアプリケーションが開始されます。 たとえば、「LOAD-02 Server Busy」。
  • 単純なワークフローは、「空き」状態と「使用中」状態から構成されます。
  • 人がアプリケーションを「無料」状態から「使用中」状態に移行すると、これはリソースを自分自身に持って行ったことを意味します。 同時に、アプリケーションが自動的に割り当てられるため、誰がリソースを所有しているのかを誰でも見ることができます。 メール通知を設定すると便利です。
  • 通常、リソースを使用した人または管理者のみが戻る権利を持ちます。
  • ここでリマインダーをねじ込むこともできます（上記のSLAを参照）。これにより、このリソースがまだ必要かどうかが数日で明らかになります。








当然、人がリソースを使用して「システムに記録されない」ことを禁止することはできませんが、良い言葉と便利なツールの助けを借りて、1つの良い言葉よりも多くのことができます。



投票







特定のアプリケーションを実装するためのかなり一般的な投票機能も、その利点を生かすことができます。 「オフィスパーティーの場所」または「誕生日にディレクターに何を渡すか」を自問した場合、バグトラッカーを安全に使用できます（ディレクターがそれを使用しないようにしてください）。 レシピ：



  • 投票プロジェクトが作成されています
  • 新しい投票ごとにドラフト版が作成されます
  • オプションアプリケーションが作成され、このダミーバージョンに割り当てられます
  • 人々の投票
  • 時間が経過すると、アプリケーションは閉じられます（そのため、投票できなくなります）
  • レポートは、投票の降順で並べ替えアプリケーションを使用したバージョンで作成されます




JIRAとBugzillaには、投票用の組み込み関数があります。 Redmine用のプラグインがあります 。



テスト管理



製品の次の各バージョンで手動で実行する必要がある一連の回帰テストを保存するシステムとしてバグトラッカーを使用するファンがいます。 私自身はこの方法のファンではありませんが、人々はそれを使用しているため、何かがあります。 レシピ：



  • 新しいアプリケーションタイプ「テスト」が作成されます。
  • このタイプのアプリケーションでは、カスタムフィールド「実行手順」と「期待される結果」が追加されます（1つのフィールドで可能ですが、テストクラシックはinしています）。
  • サブタスク「テストの実行」が作成され、「テストに失敗しました」、「テストに合格しました」、「テストに失敗しました」、「テストを完了できません」という状態になります。
  • テストがXYバージョンで実行される予定である場合、彼はサブタスクを作成し、それがこのバージョンに割り当てられます。
  • テスト後、サブタスクは目的の状態に移行します。
  • その後、サブタスク「テストの実行」に関するあらゆる種類のレポートを作成して、プロジェクトの光を理解できます。




このソリューションの利点：テストからバグトラッカーのテスト/バグを参照するのは非常に便利です。「閉じられていない」テストのあるバージョンを閉じることはできません。 一般に、テスト管理用の専用システムほど便利ではない場合があります。



技術拡張オプション



バグトラッカーを構成する機能（および以前のすべてのレシピは基本的に設定）に加えて、多くの場合、バグトラッカーの機能をプログラムによって拡張できます。



すべての主要なバグ追跡システムには、何らかの自動化に使用できるリモートアクセスインターフェイスがあります。 たとえば、一部のイベントや外部システムからのアプリケーションを自動的に投稿します。



  • JIRA： REST、SOAP、XML / JSON-RPC
  • Mantis： MantisConnect SOAP
  • Redmine： REST
  • Bugzulla： XML / JSON -RPC




プログラミングに煩わ​​されたくない人は、 コマンドラインクライアントを使用して自動化できます。



  • Jira CLI
  • Trac-admin
  • Bugzilla pyugz
  • Redmine-cli
