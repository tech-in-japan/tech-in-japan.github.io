なぜwsgiとnginxですか？



これにはいくつかの理由があります。 まず、解決策は高速で、メモリをほとんど消費せず、持続可能でなければなりません。 Nginxはそのように作成されました。 さらに、 パフォーマンス テストによると、nginx + wsgiバンドルは、メモリ消費量が少なく、高負荷下での高いフォールトトレランスを提供します。 第二に、システムが単純であるほど、信頼性が高いことが知られています。 wsgiとfastcgiを比較すると、後者については、アプリケーション自体が実行される別個のサーバーであると言えます。 誰かがこのサーバーがクラッシュしないことを確認する必要があります。 一方、wsgiはCのアプリケーション（nginx）からのpythonスクリプトの呼び出しであるため、インターフェイスを構成したら、エンティティの数を半分にして、Webサーバーとのみ通信します。



さらに、理論的には単にwsgiがエンドユーザーの方が高速であるという仮定がありますが、理論的には50人のユーザーが同時にサーバーに接続し、同様に重いページを要求したとします。 fastcgiの場合、nginxはすぐにfastcgiサーバーに50リクエストを行い、応答を待ちます。 純粋に理論的には、同時に到着した同じリクエストが実行中のアプリケーションを大量に生成し、プロセッサ時間を均等に競い、すべてが一度に実行されます（実行中のfastcgiアプリケーションの数に制限がない場合）。 Nginxはwsgi自体に届いたリクエストを処理し、同時リクエストの最大数はワーカーの数に等しいため、それらはキューに入れられ、ユーザーはほとんどすぐに最初の回答を受け取ります。 しかし、後者はfastcgiの場合と同じ時期に来ます。行き先はありません。

Nginx設定



nginx.confで最初に修正する必要があるのは、ワーカーの数です。 通常、最適なワーカー数はマシンのコア数に等しいと聞いたことがあるでしょう。 mod_wsgiを使用すると、状況は少し変わります。 実際、アプリケーションはこれらの同じワーカーのコンテキストで動作するため、同時に実行されるプロセスの数はワーカーの数に等しくなります。 理想的な条件下では、コアの数で見たワーカーの数がプロセッサーに正確に100％をロードします。 しかし、条件は完全ではなく、アプリケーションの実行は時々中断されます。 たとえば、データベースサーバーが別のマシン上にある場合、ディスク操作の実行時またはデータベースのクエリ時。 したがって、最適なワーカー数はコア数の2倍に等しいように思えます。

サーバーの再起動



Pythonはphpではありません。 ダウンロードされたアプリケーションは、ソースから分離され、その変更については何も知りません。 したがって、コードを更新するときは、時々Enginixを再起動する必要があります。 幸い、Enginixは非常に優れたサーバーです。すべてのクライアントを切断して、それを強制終了して再度実行するスクリプトを記述する必要はありません。 これは、サービスを提供しているクライアントを即座に切断することなく、ワーカーをソフトに再起動する機能を提供します。 これは、HUPシグナルをホストプロセスに送信することで実行できます。 したがって、私たちがする必要があるのは、単純なスクリプトを書いて、それに名前を付け、例えばrestart-serverを書くことです：



sudo kill -HUP `cat /usr/local/nginx/logs/nginx.pid`







このスクリプトを実行するたびに、アプリケーションが再起動します。 さらに、1時間に1回など、このスクリプトを冠に掛けることを強くお勧めします。 これはパフォーマンスに大きな影響を与えませんが、スクリプトが流れ始めればメモリを節約できます。



軟膏で飛ぶ



残念ながら、結果のサーバーにはアキレス腱があります。 10人が何らかの種類のブレーキページを同時にロードすることを決めた場合（pdf生成またはアバターのピンチなど）、あなたが10人の労働者しかいない場合、他の要求はありません（たとえば、gif画像などの最も軽いものも処理されます）少なくとも1人の労働者は無料ではありません。 残念ながら、他の通信方法をテストしませんでした。fastcgiまたはhttp_passを使用した場合と同じ効果が得られる可能性がありますが、別のTCP接続が使用されているため、このように動作しないようです。 しかし、私が言ったように、Enginixは素晴らしいサーバーだからです。 彼はリクエストを重いものと軽いものに分けます。キューに重いリクエストがたくさんある場合でも、少なくとも1人のワーカーを解放すると、最初にすべての軽いリクエストを吐き出し、そのあとで重いリクエストを受け入れます。 言い換えれば、軽いリクエストは、受け取った順番に関係なく、常に重いリクエストよりも優先度が高くなります。
