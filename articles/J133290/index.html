N9のカレンダーフィード：概要と開発方法

この投稿は、「 Smart Posts for Smart Posts 」というコンテストに参加しています。



Nokia N9は、多くの点で優れたデバイスです。 しかし、箱から出してすぐには、非常に重要な機能はありません。 携帯電話のホーム画面を一目見ただけでは、先のカレンダーでどんなイベントが待っているのかを理解することは不可能です。 この厄介な問題を解決するために、私はCalendar Feedアプリケーション（ OVI Store 、 source ）を開発しました。 カットの下で、アプリケーション自体（少し）とそれがどのように作成されたか（投稿の大部分）について詳しく説明します。



注意、テキストがたくさんあります。 準備ができたら...



カレンダーフィードアプリケーション。 これはOSに完全に統合されており、ユーザーではなくOSを介して再度呼び出されます（ただし、ユーザーは[更新]ボタンを使用してアプリケーションデータの更新を急ぐことができます）。 設定はシステムに完全に統合され、その一部のように見えます。 動作中、アプリケーションは定期的に（20分ごと）、古いカレンダーを削除しながら、カレンダーから最も近いイベントを持つ要素をフィード（ロシア語版ではチャンネル）にアップロードします。 この要素は常に一番上にあります（その時刻は現在のプラス1日として設定されます）。 アプリケーションにはいくつかの設定がありますが（まだ非常に若く、少しだけ実行できます）、基本的な機能のすべての機能には十分です。 要素内のイベントの数を設定して、今日のイベントを将来（今日があまりない場合）に追加するかしないか、将来のイベントの日数に制限（切断の可能性がある場合）を設定できます（翌年または月のイベントを表示しないように）。

Harmattan DBus署名ストレージ機能


私はこのパズルを数晩夕方に解決しなければなりませんでしたが、最終的に（ hexdump 、 stringsなど、およびそのような母親を使用して）メソッドシグネチャを渡して正しく呼び出されるようにする方法を理解することができました。



次の署名があります。

com.nokia.Calendar / com.nokia.maemo.meegotouch.CalendarInterface.showMonthView 2011 11 25

デバイスでこれらのパラメーターを使用してこのメ​​ソッドを呼び出すと、2011年11月25日に選択された日付の月表示でカレンダーが開きます。



このメソッドを呼び出すには、要素のアクションに次の行を追加する必要があります。

com.nokia.Calendar / com.nokia.maemo.meegotouch.CalendarInterface showMonthView AAAAAgAAAAfb AAAAAgAAAAAL AAAAAgAAAAAZ



さて、インターフェイスとメソッド名の間にはスペースがあり、すべてが明確です。 QDBusMessageコンストラクターでは、 2つの異なるパラメーターにも分割されます。 しかし、3つの数字の代わりに3つの奇妙な文字列引数とは何ですか？ あなたが知っているとき、すべてが非常に簡単です、hehe（夕方に2つまたは3つを殺し、バイナリとdbusメソッドで掘り返しました）。 これは、 QDataStreamにシリアル化されたQVariantオブジェクトのBase64です。



そのような行を生成するためのヘルパーメソッド： 

 QString CalendarFeedPlugin::base64SerializedVariant(const QVariant &value) const { QByteArray ba; QDataStream stream(&ba, QIODevice::WriteOnly); stream << value; return ba.toBase64(); }


dbusメソッドのシグネチャが格納されているHarmattanのあらゆる場所で、このような表記法が使用されていることが疑われます。



Syncfw



フィード要素を作成する方法を学びました。 ここで、何らかの方法で定期的に追加する必要があります。 バッテリーを食べる悪魔を囲い込むのではなく、可能な限りシステムに統合して、いつ更新するかを決定することをお勧めします。



この目的のために、Harmattanにはプラグインを作成できるSyncFWフレームワークがあります。 このプラグインは、指定されたパラメーターに従ってポーリングされます。 プラグインは、メタデータと共有ライブラリの2つの部分に論理的に分割されています。



重要な点： PR1.0では、新しいプラグインをインストールした後にデバイスを再起動する必要があります。 PR1.1では、この問題を回避できます（インストールスクリプトのセクションを参照）。 ただし、プラグインの更新には常に再起動が必要です。 これがないと、古いメタデータが新しいメタデータと共に使用されるだけです（再起動せずにアンロードできますが、この方法はわかりません）。



もう1つの重要なポイント：例でcalendarfeed名が使用されている場合は常に、プラグイン名が必要であることが理解されています（この場合、ライブラリはlib <plugin_name> -client.so、つまり、この場合はlibcalendarfeed-client.soと呼ばれます）。 プラグインの名前にハイフンがあってはなりません。 理想的には-ラテン文字のみ、数字の使用も許可されると思います（チェックしませんでした）。



メタデータ


プラグインを説明するには、次のファイルが必要です。

  • /etc/sync/profiles/client/calendarfeed.xml
  • /etc/sync/profiles/service/calendarfeed.xml

/etc/sync/profiles/client/calendarfeed.xml

クライアント側のプロファイルを作成するだけです（共有ライブラリがクライアントとして機能します）



/etc/sync/profiles/service/calendarfeed.xml

サービスプロファイルを作成し、オフラインで動作できることを示します。 つまり、プラグインは正常に機能するためにインターネット接続を必要としません（Harmattanのドキュメントではこれは不可能であると記載されていますが、実際には機能しています）。 顧客プロファイルも示されます。

同期プロファイルを作成します。 プロファイルノードのコンテンツ：

  • displayname-UIに表示される名前
  • 有効-プロファイルがアクティブかどうか
  • use_accounts-アカウントフレームワークを使用するかどうか
  • 非表示-このプロファイルはUIに非表示です
  • プロファイル-サービスプロファイル


ここで最も重要な部分は、もちろんスケジュールです。 このノードからの情報に従って、プラグインがポーリングされます。 この場合、曜日に関係なく20分ごとに質問されます。



しかし、週末と平日の夕方には30分ごとに、平日は15分ごとに1回質問する必要がある場合はどうすればよいでしょうか？ 30分で明らかです。 20から30に変更すると、すべて問題ありません。 追加のスケジュール時間を設定するには、 ラッシュノードをスケジュールサブノードとして有効にする必要があります。 

 <rush begin="00:00:00" enabled="true" interval="15" end="17:00:00" days="1,2,3,4,5"/>

このヘッダーには、最低限必要なクラスに近いクラスが含まれます（ 保護されたスロットとプライベートセクションが削除された場合、最小クラスになります）。 それ以外はすべて仮想メソッドのオーバーライドです（ほとんどの場合、純粋仮想）。



仮想bool init（）;

これは最初に呼び出され、プラグインがそれ自体を初期化できるかどうかを確認するために必要です。 成功した場合、trueを返し、プロセスはさらに進みます。



仮想ブールuninit（）;

プラグインがアンロードされる前に呼び出され、通常はここでクリーニングが行われます。



仮想ブールstartSync（）;

このメソッドは、更新を開始できるかどうかを確認し、更新を開始する必要があります（非同期で）。 更新の開始に成功した結果を返します。



virtual void abortSync（Sync :: SyncStatus status = Sync :: SYNC_ABORTED）;

ユーザーが更新をキャンセルしたときに呼び出されます。 アカウント（このようなボタンがある場合）を操作する必要があります。この場合、ユーザーは更新をキャンセルする機会がありません。



virtual Buteo :: SyncResults getSyncResults（）const;

最後の更新の結果を返します。



仮想ブールのクリーンアップ（）;

アカウントが削除されたときに呼び出されます。 この場合も、これは必要ありません（ただし、ドキュメントでは、後で使用できる可能性があると書かれています）。



virtual void connectivityStateChanged（Sync :: ConnectivityType type、bool state）;

接続ステータスが変更された場合に呼び出されます。 プラグインがネットワークで動作しない場合、このスロットも空になります。

更新ボタンの更新


インターネットなしで更新する場合と同様に、ドキュメントでは、更新ボタンは埋め込みアプリケーションのみを更新できることが示されています（読み取り：Twitter、Facebook、RSSフィード、APモバイル）。 しかし、私たちは独自の方法で進んでいるので、このステートメントを自分で検証しようとします。



Feedデータベース（ /home/user/.config/meegotouchhome-nokia/eventsfeed.dataファイルにあり、通常のSqliteデータベース）には、イベント、画像、refreshactionsの3つのテーブルがあります。 この場合、最初の2つはほとんど関心がありませんが、3番目は非常に興味深いものです。 彼女には属性アクションが 1つしかありません。 Feed要素を追加することで名前がすでによく知られているので、ここでdbusメソッド呼び出しのシグネチャを（上記の形式で）配置しようとします。 そしてそれは動作します！ ドキュメントは再び間違っていました。 さて、大丈夫ですが、今では必要な機能を実装できます。 システムへの最大限の統合の道をたどっていますが、アプリケーションがこのボタンに応答しない場合は非常に奇妙です。



さて、ボタンにバインドする方法を決定しましたが、プラグインをスケジュール外で呼び出す方法は？ これを行うには、1つの引数（クライアントプロファイルの名前）を持つdbusメソッドcom.meego.msyncd / synchronizer com.meego.msyncd.startSyncがあります。



細かい部分が残っています...アクションをこのテーブルに追加する方法は？ 実際、SQLクエリではありません。 幸いなことに、dbusのaddItemの隣にはaddRefreshActionメソッドがあり、署名付きの行を渡すことができます。



少し後に、 からaddRefreshActionを呼び出す場所を見つけます。 それまでの間、アプリケーション自体を続行します。

アプレット作成


MeeGo Harmattanは、設定で有効にできるアプレットを作成する3つの方法を提供します。

  • 宣言的 -設定セットはxmlファイルによって記述されます
  • バイナリ -DcpAppletIfから継承した共有ライブラリを作成します
  • 外部 -通常のアプリケーションを使用する


3番目のオプションは、設定からアプリケーションに切り替えるときに遅延を引き起こす可能性があるため、まったく適切ではありません（また、ドキュメントではお勧めできません）。

2番目のオプションは最も柔軟ですが、最初のオプションよりも複雑です。

最初のオプションは最小限の柔軟性ですが、同時に可能な限りシンプルです。



特に複雑な設定は必要ないため、最初の方法を使用します。



それには2つのファイルが必要です。

GConfItemを使用する


GConfが使用されていますが、それでもGConfを操作するためのQtクラス-GConfItemがあります。 それから2つのメソッドが必要です： value（）とset（） 最初はQVariantのパラメーターの値を返し、2番目はパラメーターの値を変更します。



ただし、すべてのタイプがサポートされているわけではありません。 次のルールが適用されます（gconfおよびqvariant機能による）：

  • QVariant ::無効-gconfパラメーターがありません
  • QVariant :: Int 、 QVariant :: Double 、 QVariant :: Boolは同等の型に変換します
  • QVariant ::文字列はutf8エンコードでのみサポートされます
  • QVariant :: StringList -utf8の文字列のリスト。
  • QVariant ::リストリストQVariant :: Int 、 QVariant :: Double 、 QVariant :: BoolまたはQVariant :: String （ QVariant :: StringListとして返されます ）、すべての要素は同じ型でなければなりません
  • 他のすべてのタイプ（両側）は無視されます。


宣言型アプレットの場合、アプレットは文字列、ブール値、整数を提供するため、これは完全に適しています。

GConfItemの例 

 bool fillWithFuture = false; GConfItem fillConfItem("/apps/ControlPanel/CalendarFeed/FillWithFuture"); QVariant fillVariant = fillConfItem.value(); if (fillVariant.isValid()) fillWithFuture = fillVariant.toBool(); else fillConfItem.set(false);

postinst 

 #!/bin/sh /usr/bin/aegis-exec -s -u user dbus-send --dest=com.meego.msyncd --print-reply /synchronizer com.meego.msyncd.installPlugin string:'calendarfeed' /usr/bin/aegis-exec -s -u user dbus-send --dest=com.nokia.home.EventFeed --print-reply /eventfeed com.nokia.home.EventFeed.addRefreshAction string:'com.meego.msyncd /synchronizer com.meego.msyncd startSync AAAACgAAAAAYAGMAYQBsAGUAbgBkAGEAcgBmAGUAZQBk' gconftool -s /apps/ControlPanel/CalendarFeed/EnableFeed -t bool true /usr/bin/aegis-exec -s -u user dbus-send --dest=com.meego.msyncd --print-reply /synchronizer com.meego.msyncd.startSync string:'calendarfeed' exit 0
