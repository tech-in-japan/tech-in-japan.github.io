IdMを実装します。 パート3.1。 IdMが必要であることは明らかです-次は何ですか？ 目標、タスク、利害関係者

一連の記事の前の部分で 、IdMが必要かどうか（つまり、アクセス制御）と、IdMソリューションを実装する価値があるかどうかを判断する方法について説明しました。 少なくとも、この問題について考えていることを示す兆候は何であるかを判断しました。 次は？



IdMのトピックを始めるには、いくつかの重要な決定事項があります。



  1. 目標と目的。 利害関係者。
  2. 従業員にアクセス制御システムを実装する際に使用するアプローチとプラクティス、導入する手順とプロセス、必要な操作を企業のビジネス活動に統合する方法
  3. 使用する技術的ソリューション（ドメインポリシーから始まり、IdMソリューションで終わる）および必要な機能を決定する方法
  4. 技術的な解決策の対象者
  5. 予算を策定し、正当化する方法は？ （これは最も興味深く、燃えているトピックです）
  6. 経営陣へのプレゼンテーション。
  7. プロジェクトを開始するときに考慮すべきことは何ですか？


それでは、ポイントを見ていきましょう。









1.目標と目的。 利害関係者。



IdMが何かをしなければならないことが明らかになったとします 。 いったい何？ 実装する機能 どのポリシーを導入および制御しますか？ そして、このトピックでの作業から何を期待しますか：プロセスの整理、管理の確立、監査の実施？ それとも自動化が必要ですか？



頭に浮かぶ最初の答え-IdMはあなたの会社で需要があるものでなければなりません。



ITIL、Cobit、ISO / IEC 2700xなど、一般的な標準、知識のコード、および一般に受け入れられている慣行は、すべての人に簡単な真実を伝えるために一生懸命努力しています：ミッション、戦略、文化に合った特定の企業に適したもののみを実装する必要がありますおよび組織構造。 次のような特定のビジネスに対する実装された各サービスおよびシステムの影響を考慮する必要があります。



  • ユーザーの利便性 （この場合は内部）とビジネス継続性。 （多くの場合、これらは密接にリンクされた概念です：銀行のオペレーターがひどいインターフェースと不適切な機能のために支払いをほとんどできない場合、何人の顧客がキューから吐き出して別の銀行に出ますか？）
  • 金融要素 （サービスは負担になりません）。
  • システムの信頼性 （1か月あたり1時間以下のシステムダウンタイムの可能性があるSLAは、一部のシステムで非常に人気があります）。
  • 情報セキュリティのレベルは十分でなければなりません 。 すなわち 「試してみるだけで」ペストリー販売会社でSOCを構築する必要はありません。 同時に、銀行などの重要で高価な情報が見つかった会社では、最も単純な決定とポリシーを無視するべきではありません。 この場合、IdMパラダイムで情報セキュリティを検討します。


思い出させてください。IdMとは、一連の測定、プロセス、技術的手段を意味します。つまり、すべての企業が複雑で高価な技術的IdMソリューションを必要とするわけではありません。 さらに、誰かがプロセスを紙の上ではなく実際に正しく構築するだけで十分です。 プロセスのためのプロセス、セキュリティのための安全性、ITのためのITは必要ありません。それらは割り当てられた機能を果たすだけであるため、ビジネス目標の達成に役立ちます。



それはどの結論ですか？ サイトを見つけて変更を加える必要があります。変更を加えることで、ユーザーの生活を改善し、ビジネスに利益をもたらします。 何をする必要があるかを理解するには、完全な調査を実施する必要があります。 論理的なものがあります：「 どこから始めればいいの？」



3つの簡単な質問から始める価値があります。



  1. 何が欲しい？
  2. なぜこれが必要なのでしょうか？
  3. なぜこれが必要なのですか？


すぐに答えが出たら、いいです。 質問への回答は修正するのに役立ちますが、この段階で目標と目的の定義が完了したとは考えないでください。 回答では、あなたの視点と質問を読んだときに思いついたものだけを述べました。 このプロジェクトに参加する動機をより明確に理解するだけでなく、プロセスの他の参加者の 意見やニーズを知ることは非常に重要です。 そして、この情報を何らかの方法で修正します。



それとは別に、研究結果の固定を強調します。



  • 事前にどのように調査を行い、回答を記録するかを考えてください。
  • 記入するアンケートとテンプレートを準備します。
  • 便利なプロトコルフォームまたはスコアカードを作成します。


多数の人とチャットするだけで、それぞれが異なる、時には予期しない何かを報告して尋ねる場合、誰が何を言ったのか、提案したのか、尋ねたのかを後で覚えることはほとんどありません。 自分の記憶だけに頼らないでください。 正式には組織フォームではなく、単にノートに書き留めます。 それは価値があります。



情報を収集するプロセスに進み、それに基づいてIdMの開発と実装の目標と目的を決定し、プロセス、対策、技術的ソリューションの要件を理解します。



最初に行うことは、ビジネス開発戦略（ある場合）を読むことです。 正式なドキュメントのみである場合もあれば、まったく存在しない場合もあります。 そうでない場合は、会社のリーダーに行って、ビジネスの最も差し迫ったニーズは何か、近い将来の会社の発展は何か、そしてITおよび情報セキュリティサービスの仕事からの個別の期待があるかについて話し合う必要があります。



このようなフィードバックの収集は定期的に行うことが望ましいです。 一方で、これは会社の経営陣と連絡を取り合うため、「 突然発生する 」要望や要件にショックを受けることはありません。 一方、受け取った情報により、ユニットの作業に対する満足度を確認し、ビジネスの利益のために機能する真に合理化された調和の取れたシステムを構築できます。



ユーザーが専門家に馴染みのある言葉で報告し、不平を言うことを期待することはほとんどありません。 「 品質 」と「 作業のスピード 」（測定方法と測定方法が不明な場合がほとんどです）、「 コスト削減 」、「 使いやすさ 」、「 シンプル 」、さらには「 美しく、イライラする 。」 または単に「仕事に行け！」あなたは聞いたことすべてに対処する必要がありますが、一度連絡をとれば、将来のリーダーシップの願いを理解しやすくなります。 まあ、そして：「誰がしようとしません、それ...」（あなた自身は何を知っていますか）。



2番目のステップは、ユーザー、IT /情報セキュリティサービス、およびプロセスに関係するビジネスサービス（人事部など）とのコミュニケーションです。 ユーザーの目を通してプロセスがどのように見えるかを理解し、ユーザーと一緒に「通過」する必要があります。 便利なサービスを構築してセキュリティを確保するには、ユーザーを知る必要があります。

さらに、各ユーザーは重要です。 一般ユーザー、中間管理職、経営陣の悲しみや困難を聞く価値があります。



同様のアプローチは、同僚との作業に役立ちます。ITと情報セキュリティ（どちら側にいるのか）。 マネージャーとパフォーマーの両方のアドバイスを求め、IT /情報セキュリティの専門家の意見に耳を傾けることをお勧めします。 マネージャーはIT /情報セキュリティサービスの一般的な概念を知っています。誰が何に責任を持ち、新しいシステムを運用するための計画は何か、落とし穴は何か、規制当局の要件などは何でもあります。 同時に、アクセス権の付与と取り消し、アカウントの作成、パスワードを忘れたユーザーとのやり取り、さらにテクニカルサポートに電話で問い合わせるプロセス全体を、管理者よりも優れた人が教えたり見せたりすることはありません。 押すと、すべてが消えました... 」。 また、ServiceDeskまたはSEDでフォームがどのように遅くなるかを教えてくれます。

  • 機密情報の取り扱い 、アクセスは厳格な規則に従って発行され、常に監視される必要があります。


各ユニットのヘッドと話し合うと便利です（階層の最上部から始めて、そのステップを下っていきます）。 私を信じて、あなたは多くの新しい予期しないことを学ぶでしょう。 たとえば、一部の従業員は1週間または2週間アクセスできない場合があります。 休暇中（病気休暇中）の直属の上司はフォームに署名できないため、アクセスを失った従業員は、すでに解雇された専門家のブロックされていない「記録」の下で作業します（これが実際のシナリオです）。
