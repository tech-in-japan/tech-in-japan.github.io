Android StudioのIntent Senderプラグインとその個別の開発機能の概要

Androidアプリケーションを開発するとき、多くの場合、作業を最適化して時間を節約するためにadbに頼らなければなりません。 Adbは、たとえば、長いアクティビティチェーンをバイパスして、目的のアクティビティをすぐに実行したり、外部条件が満たされるのを待たずにブロードキャストメッセージを送信したりできる唯一のツールです。 adbを使用すると、端末を使用して長いコマンドを生成するすべての不便さが伴い、デバイスにインテントを送信する際にadbの処理を高速化および簡素化するプラグインを作成することになりました。



IntelliJ IDEAのプラグインを開発する方法に関する情報と詳細な説明を見つけることは問題ではないため、記事の最後で、このトピックにいくつかの段落を当てて、遭遇した機能を示します。 今、私の仕事の結果を説明したいと思います。



そのため、 Android Intent Senderは次のadbシェルコマンドに関連付けられています。



  • ブロードキャスト-コマンドを使用すると、ブロードキャストメッセージをクライアントに送信できます。
  • StartActivity-起動アクティビティ
  • StartService-サービスを開始します


これらのコマンドを送信するときに、必要なすべてのパラメーター（アクション、データ、カテゴリ、コンポーネント、MIMEタイプ、フラグ）および追加データを備えたインテントを添付できます。



プラグインの外観を、必要なUI要素を備えた個別の作業ウィンドウとして設計することが決定されました。 プラグインへのアクセスは、[表示]-> [ツールウィンドウ]メニュー、画面の左下のアイコン、および右パネルから実行します。







プラグインウィンドウは、デバイス/エミュレーター選択ウィンドウ、インテントパラメーターセクション、コマンド送信用のボタンのメインセクションに分かれています。 プラグインは、デバイスが接続および切断されたときにデバイスの自動更新を実装します。 インテントパラメータセクションには、入力フィールドに加えて、履歴ボタン（セクションタイトルの右側）、コンポーネント入力フィールドの隣にある現在のプロジェクトのクラスピッカー、フラグピッカー、追加のデータ追加ボタンがあります。



現在のプロジェクトのクラスピッカーにより、コンポーネント（起動されたアクティビティなど）の選択が大幅に簡素化されます。 ボタンをクリックすると、プロジェクトクラスを含むウィンドウが開き、検索してプロジェクトツリーに切り替えることができます。 クラスを選択すると、パッケージの境界線に必要な「/」セパレータが適切な場所に自動的に置き換えられます（たとえば、com.weezlabs.blabla / .MainActivity）。







フラグピッカーを使用すると、インテントに追加する1つ以上のフラグを選択できます。







デバイスにコマンドを送信するには、 「 ブロードキャストを送信」ボタン、 「アクティビティを開始 」ボタン、 「サービスを 開始 」ボタンを使用します。



一般的な場合にadbを介してアクティビティとサービスを開始するには、このコンポーネントのマニフェストにandroid:exported=”true”

  1. 放送受信機のテスト
    ここではすべてが簡単です。必要なパラメーターを指定し、インテントを送信します。 たとえば、onReceiveのコードを持つレシーバーがあります。 



     @Override public void onReceive(Context context, Intent intent) { String action = intent.getAction(); if (action.equals("simple_intent_action")){ Log.d("SimpleReceiver", "Got intent with action: " + action + " and string extra: " + intent.getStringExtra("string_extra")); } }


    適切なIntentFilterを指定してReceiverを登録します。 



     IntentFilter simpleFilter = new IntentFilter("simple_intent_action");


    次に、プラグインでブロードキャストメッセージを送信するには、アクションフィールドで「simple_intent_action」を指定し、 「Add extra」ボタンをクリックして、 「string_extra」キーとテキストでエクストラを追加します。 「ブロードキャストを送信」をクリックして、ログで次のようなものを確認します。

    Got intent with action: simple_intent_action and string extra: Awesome string

    コンポーネントピッカーのプラグインでこのアクティビティを開始するには、MessageActivityを選択し、ユーザーが自動的に追加されることを確認します。 次に、キー「messageId」と目的の値を持つlong型のオブジェクトを追加します。 すべての準備が整ったら、 「アクティビティを開始」をクリックして、指定されたメッセージが表示された起動済みアクティビティを確認します。



    onCreateメソッドのMessageActivityがログを書き込む場合：

プラグイン開発プロセス自体については、プラグインに埋め込まれたアイデア自体を実装する複雑さによって複雑さが決まると言えます。 環境を準備して直接設定しても問題は発生せず、インターネットには多くの情報があります。 環境をセットアップするために必要な手順を簡単に説明します。



  • AndroidStudioに使用されるIDEAブランチを探しています。 これを行うには、 apiVersion = "135.1286"に類似したAndroidStudioApplicationInfo.xmlの行を探し、そこからブランチ番号を取得します。私の場合は135です。AndroidStudioApplicationInfo.xmlでは 、次のリンクを確認します。 idea133 / adt-branding / src / idea / AndroidStudioApplicationInfo.xml

    当然、リンクは厳密には推奨されていません。android.googlesource.com/ platform / tools / adt / ideaへのリンクをトリミングしてAndroidStudioの目的のバージョンに移動し、フォルダーを歩いて、必要なAndroidStudioApplicationInfo.xmlを簡単に見つけることができます。
  • 説明に従ってIntelliJ Community Editionをダウンロードし、以前に見つけたブランチに切り替えます（135を持っています）
  • プラグイン開発用のSDKを作成する

  • 最後から2番目のステップ：Androidプラグインの依存関係をSDKのクラスパスセクションに追加します。

次に、IntelliJプラットフォームプラグインタイプの新しいプロジェクトを作成します。IntelliJプラットフォームSDKドキュメント 、 Open APIおよびプラグイン開発コミュニティ 、ハブに関する多数の記事、githubの例を使用して、プラグインの作成を安全に開始できます。



UI要素のswingフレームワークを使用して、toolWindowコンポーネントに基づいてプラグインを実装することにしました。 JavaのUI、特にSwingでの作業に出会ったことがない人として、私はその機能に少し適応する必要がありましたが、特別な困難はありませんでした。 おそらく、AndroidのUIでの作業との類似性が役立ったでしょう。



さらに、adbを操作するために、Android SDKの一部であり、パス$ {SDK_HOME} / sdk / tools / libにあるddmlibライブラリが非常に役立ちました。 このディレクトリには他にも多くのjarがありますので、ご覧になることをお勧めします。突然、一部のライブラリがタスクの解決を簡素化します。 また、 org.jetbrains.androidパッケージにも注目します。これは、Androidパーツを使用する際に非常に役立ちます。 そのため、彼の助けを借りて、Android SDKディレクトリへのパスが取得され、パッケージがマニフェストから取得されました。 念のため、プラグインはAndroid変数へのパスをANDROID_HOME環境変数で検索し、前の手順でSDKが見つからなかった場合にパスを手動で設定する機能も提供します。 adbを使用する場合、プラグインはコンソールの応答を解析します。これにより、場合によっては送信が失敗したことを理解し、対応するエラーを返すことができます。



コマンドの履歴を保存するために、プラグイン設定に保存されているjsonが使用されます。



Windows環境でのプラグインの開発に関連して、OS Xでプラグインを起動したときに、 NoSuchMethodExceptionでクラッシュするケースがいくつかありました。 その理由は、Windows用のJavaとOS Xの違いです。OSXで作業している同僚を妨害し、それらを使用してデバッグする必要がありました。



プラグインをテストおよびデバッグした後、jetbrainsプラグインリポジトリに追加するためにアプリケーションが送信されました。最初のファイルバージョンはすぐにモデレーションを経て、アプリケーションの送信後1時間以内にダウンロードできました。
