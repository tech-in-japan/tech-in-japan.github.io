マルチスレッドWSH VBScript

松葉杖



すべての良い一日。 WSH VBScriptのマルチスレッド（または「マルチプロセッシング」）のメカニズムを実装する松葉杖を提供します。 インドコードにアレルギーのある人にはお勧めしません。

マルチスレッドの潜在的な利便性について最初に考えたのは、情報を処理して視覚化するという比較的単純なタスクに関する自発的な決定の際に一度来ました。 また、Visual StudioとヘルプデスクのようなIDEのインストールは互換性のない概念であるため、VBAとWSHのみが「合法的な」ものとして残りました。 その時、後者が選ばれました。 そして、現在の管理上の制限と使いやすいメソッドを取得したいという要望により、このマルチプロセッシングの実装の要件が決まりました。

  • コードは、通常のスクリプトのように単一のファイルにあります。
  • WSH x86 / x64では、最小限のActiveXを使用し、WMIなしで機能するのは組み込みのみです。
  • 速度とパターンよりも、十分な機能と利便性が重要です。


機能：

  • 起動されたスクリプトはサービス機能を実行します。 メインコードは後続のプロセスで実行されます。
  • 「フルバージョン」では、メインコードはクラスにのみ配置され、そのいずれも別のプロセスでインスタンス化できます。 新しいプロセスごとに、スクリプトはユーティリティクラスのインスタンスを作成し、指定されたターゲットクラスのインスタンスを作成し、ターゲットメソッドを呼び出し、その進行を制御し、作成されたクラスインスタンスを参照するグローバルスクリプトコンテキストに変数を作成します。 初期化されたターゲットクラスのオブジェクトは前のスクリプトに返され、実行中のスクリプトのターゲットクラスの初期化と完了のイベントは「処理」されます。
  • 「ライトバージョン」では、ロジックは似ており、メインコードのみがsub'ov内に配置され、オブジェクトは前のスクリプトに返されず、起動されたもののインデックスになり、イベント処理はありません。
  • 新しいプロセスを作成するために、スクリプトはそれ自体を再帰的に起動し、呼び出されたメソッドに関するデータを渡し、名前付き引数の情報を識別します。
  • 単一の実行スペースはなく、各スクリプトはそのデータとオブジェクトをコンテキストに保存します。 データを交換するには、Grayフォーラムのトピック「スクリプト間でのデータとオブジェクトの交換-2」で説明されているGlobalContainerを使用して、最初のスクリプトのMeオブジェクトが後続のものに転送されます 。 ターゲットメソッドの完了後の各スクリプトは、許可が完了することを期待しています。このような遅延により、受信した情報を取得できます。
  • WSH WScriptとCScriptで同じように機能し、ScriptCryptorでコンパイルされたexeファイルの形式で機能します（ただし、class_terminateメソッドはexeでは機能しません）。


短所：

  • onエラーの使用によるデバッグの難しさは次に再開され、多くの方法で実行されます。 すべて同じですが、これは松葉杖です。
  • かさばる。
  • スクリプトの完了順序は完全に開発者の責任です。 当然、完了したスクリプトのオブジェクトにアクセスするとエラーが発生します。


実際：

  • メソッドを複数のプロセスが使用できないようにするためのmutexとメカニズムは実装されていません。
  • 他のコンテキストからスクリプト変数にアクセスしているときに障害が発生する場合があります。
  • CScript.exeコンソールウィンドウを閉じても、エクスプローラウィンドウはメモリに残ります。

ライトバージョンのマルチプロセッシングメカニズム機能：



サブネームを起動

サービスアルゴリズムを起動し、スクリプトの実行は常にその呼び出しから始まります。

subnameは、新しいプロセスで実行されるターゲットsubを示す文字列です。

このメソッドの引数は、メインコードの実行を開始するために最初に起動されたスクリプトでのみ使用されます。

新しい各スクリプトで：

  • 変数は、実行中のスクリプトのグローバルスペースで宣言されます。

    scene-最初のスクリプトのMeオブジェクトへの参照、

    container -GlobalContainerエクスプローラーウィンドウへのリンク、

    signature -GlobalContainerを識別するために生成された文字列、

    見つかった、失われた -シーンのみ-作成および完了したスクリプトプロセスの数、

    subname-文字列、スクリプトの目的を含む、

    jobs-シーンで作成された実行スクリプトのMeオブジェクトを含む辞書へのリンク、

    id-このスクリプトのシリアル番号-辞書のキー、

    release = False-ターゲットsubが完了すると、Trueを割り当てた後にスクリプトが完了します。

    state = 4-このスクリプトが配置されている状態。
  • アイテムがジョブディクショナリに追加されます。


startproc （サブネーム）

サブネーム-launch（）を参照してください。

新しいスクリプトプロセスを作成し、スクリプトのMeオブジェクトにアクセスできるときに、状態4（getstate（）を参照）の準備が整うまで待機します。 実行中のスクリプトのID-IDを返します。



createproc （サブネーム）

サブネーム-launch（）を参照してください。

待機することなく、新しいスクリプトプロセスを作成し、数値-実行中のスクリプトのIDを返します。 各プロセスの準備が整うのを待たずに、ループで複数のプロセスを非同期的に作成するために使用します。 このようなアプリケーションでstartproc（）を使用するよりも著しく高速です。



getjob （ターゲット）

target-作成されたスクリプトのグループの番号、スクリプトIDまたは文字列、サブネーム。

Meスクリプトオブジェクトへのアクセスを提供します。 idが見つからないか、スクリプトが完了した場合、スクリプトのMeオブジェクトへの参照を返します-Nothing。



getstate （ターゲット）

target-スクリプトの番号、ID。

スクリプトの状態を決定します。 数値、実行ステップを返します。

1つのプロセスが作成されました（新しいプロセスexec）、

4スクリプトが実行中（初期化済み）、

24個のターゲットサブが完了しました

28個のスクリプトがリリースされました

64が見つかりません（ホストが見つかりません）、スクリプトは終了しました。



isresponsive （ターゲット）

ターゲット-getstate（）を参照してください。

Meスクリプトオブジェクトの可用性を決定します（状態4〜28）。 ブール値を返します。



executecommandコマンド

commandは指示を含む文字列です。

インタープリターを呼び出して、グローバルスクリプトスペースでステートメントを実行します。



varname、valueを共有します

varnameは変数の名前を含む文字列で、valueは任意の値です。

最初のスクリプトのグローバルスペースでvarnameという名前の変数を宣言します。これは、シーンプロパティの形式ですべてのスクリプトで使用可能になり、valueの内容を変数に割り当てます。



newvar varname

varnameは、変数の名前を含む文字列です。

グローバルスクリプトスペースで新しい変数を宣言します。



無料のターゲット

target-作成されたスクリプトのグループの番号、スクリプトIDまたは文字列、サブネーム。 ターゲットsubの実行後、スクリプトの終了を許可します。 単一のスクリプトまたはグループで動作します。



ジョイント （ターゲット、状態、タイムアウト）

target-free（）を参照、state-getstate（）を参照、timeout-ミリ秒単位の秒単位の数値。

状態スクリプトが到着するまで待機します。スクリプトのグループでは、それぞれが状態に到達するまで待機します。 待機はタイムアウトによって制限されます。タイムアウト= 0は無制限の待機を意味します。 ブール値、Truth-待機は終了、False-タイムアウトを返します。 スクリプトを同期するように設計されています。 たとえば、スクリプトが開始するまで待つ必要がある場合-4、スクリプトは完了します-64。



割り込みターゲット、タイムアウト

ターゲット-free（）、タイムアウト-秒単位のwscript.timeoutの値を参照してください。

class_terminateメソッドの実装により、スクリプトを通常の完了に変換します。 スクリプトでダイアログボックスが開かれた場合、タイムアウトの一時停止後にのみ完了します。 class_terminateメソッドでダイアログボックスを再表示すると、完了が停止します。



プッシュ名、値

name-文字列、プロパティ名、value-任意の値。

nameという名前のGlobalContainerプロパティにvalueの内容を入れます。



ポップ （名前）

name-文字列、プロパティ名。

GlobalContainerからnameという名前のプロパティの内容を返します。

マルチプロセスクラスのメソッド：



クラス名、メソッド名、エイリアス名を起動します

サービスアルゴリズムを起動し、スクリプトの実行は常にその呼び出しから始まります。

クラス名、メソッド名、エイリアス名-文字列。 これらは、ターゲットクラスclassnameと、新しいプロセスで実行されるこのクラスmethodnameのメソッドを示します。クラスclassnameのインスタンスは、aliasname（aliasnameが空の文字列の場合はclassname）という名前の変数に配置されます。

このメソッドの引数は、メインコードの実行を開始するために最初に起動されたスクリプトでのみ使用されます。

新しい各スクリプトで：

  • 変数は、実行中のスクリプトのグローバルスペースで宣言されます。

    scene-最初のスクリプトのMeオブジェクトへの参照、

    host-このスクリプトのマルチプロセスクラスのインスタンス、

    process -classnameという名前のターゲットクラスのインスタンス。この名前は、このスクリプトのプロセスを作成するときに前のスクリプトによって示されたものです。

    祖先は、前述のスクリプトのターゲットクラスのインスタンスです。
  • 変数は、グローバルスペースシーンおよび以前のスクリプトで、 aliasnameという名前で宣言されます-このスクリプトのターゲットクラスのインスタンスにリンクします。
  • ホストプロパティが設定されます。

    primary-最初のスクリプトのマルチプロセスクラスのインスタンスへの参照、

    祖先 -前のスクリプトのマルチプロセスクラスのインスタンスへの参照、

    parent-このスクリプトのMeオブジェクト、
