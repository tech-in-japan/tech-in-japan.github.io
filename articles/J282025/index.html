Magentoのパフォーマンスの向上

...またはコレクションの適切な動作。



パフォーマンスの問題があったほとんどすべてのMagentoプロジェクトで見たエラーについてお話したいと思います。 Magentoを使用する場合、他人のコードを監査する必要がある場合があります。 したがって、あなたのサイトのパフォーマンスを改善し、将来の間違いを避けるのに役立つ経験をあなたと共有したいと思います。



この記事では、Magento 1について説明しています。*ただし、説明されていることはMagento 2にも当てはまります。



パフォーマンスの問題があるほとんどすべてのプロジェクトで、次のようなものを見つけることができます。 



$temp = array(); $collection = Mage::getModel('catalog/product')->getCollection()->addAttributeToSelect('*'); foreach ($collection as $product) { $product = $product->load($product->getId()); $temp[] = $product->getSku(); }
間違った



の代わりに 



 $temp = array(); $collection = Mage::getModel('catalog/product')->getCollection()->addAttributeToSelect('sku'); foreach ($collection as $product) { $temp[] = $product->getSku(); }
そうだね



この理由は非常に簡単です。



  1. ロード後、必要な属性はありません
  2. インターネットの「プログラマー」もそうです
  3. 「悪化しない」という原則に従って、余分な属性をロードする


何が間違っているのか、パフォーマンスで何ができるのかを理解するには、コレクションの操作に集中することをお勧めします。



  1. Eav /フラットテーブル
  2. キャッシュ
  3. コレクションの適切な作業


そしてもちろん結論。





EAV /フラットテーブル



EAVは、属性が属するエンティティ、属性自体、およびその値が異なるテーブルに配置されている場合のデータストレージアプローチです。



MagentoのEAVエンティティには、製品、カテゴリ、カスタムメーター、住所のカスタムアドレスが含まれます。 属性自体はeav_attributeテーブルに保存されます。



Magento 5の合計属性値タイプは、text、varchar、int、decimal、およびdatetimeです。 もう1つのタイプがあります。静的です。エンティティを含むテーブルにあるという点で他の5つのタイプと異なります。



属性テーブルは、このテーブルまたはその属性がどのテーブルまたはどのタイプであるかを示し、Magentoはすでにどこに書き込むか、どこから読み取るかを知っています。



このような値の格納により、セットの属性を非常に簡単に実装できます（各エンティティが独自の属性を持つことができる場合、またはまったく持たない場合）。新しい属性の追加はデータベース内のもう1行です。 別のストアの1つの属性に新しい値を追加しました-この属性の値の表の新しい行。



データベースに保存される方法
エンティティ：

製品-catalog_product_entity、

カテゴリ-catalog_category_entity、

顧客-customer_entity、

顧客の住所-customer_address_entity



属性：

eav_attribute

catalog_eav_attribute

customer_eav_attribute



値：

* _text

* _varchar

* _int

* _decimal

* _datetime



フラットは、すべてが1か所にあり、不必要な作業なしで製品とそのすべての属性を取得するために追加のテーブルを必要としない、私たち全員に馴染みのあるアプローチです-SELECT * FROM plate WHERE id = some kind of id and that's。



EAVエンティティから、フラット表示はカテゴリと商品にのみ使用できます。



データベースに保存される方法
製品：

catalog_product_flat_1 // * _N store_view

カテゴリー：

catalog_category_flat_1 // * _N store_view



フラットテーブルに属性を含め、一般にフラットテーブルの使用を有効にするには、次の手順を実行します。
管理パネルの[カタログ]> [属性]> [属性の管理]



属性に次の値のいずれかがある場合、Magentoは属性をフラットテーブルに追加します。







管理パネルの[システム]> [構成]> [カタログ]



Magentoは、以下にリストされているエンティティに対してフラットテーブルを使用します。







次の事実に注意してください。



  1. フラットテーブルは、カテゴリページ、グループ製品の製品リスト、および実際にコレクションが使用されている場所でのみ使用されます。 モデルでloadメソッドを使用する場合、管理パネルの製品ページでは使用されません。
  2. フラットテーブルを有効にした後、インデックスを再作成する必要があります。そうしないと、Magentoは引き続きEAVテーブルのみを使用します
  3. フラットテーブルを有効にした後、Magentoは引き続きEAVを使用しますが、変更を保存するときにフラットテーブルへの変更のコピーを開始します


なぜこれがすべて必要なのか、なぜフラットアプローチをどこでも使用しないのか？ 長所と短所のピボットテーブルを見てください
EAV：

+ Flatよりも柔軟なシステム

+新しい属性を追加するときにデータのインデックスを再作成する必要はありません

+ほぼ無制限の数の属性

+すべての属性は常に利用可能です。

+明示的に指定されていなくても、静的属性（sku、created_at、updated_at）は常に選択範囲に存在します

-致命的エラー：存在しない属性の取得/フィルタリング時にメンバー関数getBackend（）を呼び出す

-パフォーマンス



フラット：

+パフォーマンス

+フラットテーブルに追加された既存の属性のみを選択/フィルタリングに適用できます

-行サイズ（最大65,535バイト、つまり85 varchar 255）および列数（InnoDBは最大1000、一部は最大4096）の制限

-コレクションを使用する場合にのみ使用されます（ロード時には常にEAVが使用されます）

-結果は、EAVを使用したリクエストの発行とは異なります（静的属性なし）

-インクルードの再インデックスが必要な場合、そうでない場合はEAVテーブルが使用されます

-新しい属性を追加する場合、フラットテーブルのインデックスを再作成する必要があります





キャッシュ



もちろん、データベース内のクエリを高速化する方法と、キャッシュが私たちを保存し、すべてがキャッシュされている場合、一般にコレクションがどのように動作するかを理解する必要があるのはあなたそれぞれです。 簡単に答えます-キャッシュはあなたを救いません。 Magentoに表示されるキャッシュは、コレクションを自動的にキャッシュしたり、データのインポート時や何かのカウント時などに使用するカスタムコントローラーやモデルで動作しません。 さらに、キャッシュに入れる前に、なんらかの方法でキャッシュに入れてすぐにユーザーに表示する必要があるためです。



Magento 1のキャッシュの種類。*：







  • 構成 -構成ファイルをキャッシュします
  • レイアウト -レイアウトファイルをキャッシュします
  • HTML出力をブロック -phtmlテンプレートをキャッシュします 。 デフォルトでは、トップメニューとフッターのフロントエンドでのみ使用されます。
  • 翻訳 -CSV CSVファイルのキャッシュ
  • コレクションデータ --> initCache（...）メソッドを使用するコレクションをキャッシュします。 デフォルトでは、core_store、core_store_group、core_websiteコレクションのみが初期化中にキャッシュされます。
  • EAVタイプと属性 -eav属性をキャッシュする必要がありますが、キャッシュはしません。 Magneto CE 1.4以降は呼び出されない1つのメソッドで使用されます
  • Webサービスキャッシュ-api.xmlファイルをキャッシュします
  • ページキャッシュ （FPC）-すべてのHTMLをキャッシュし、CMS、カテゴリ、製品ページのみをキャッシュします。 httpsプロトコルの場合、パラメーターを取得しても無視されますか？No_cache = 1、Cookie NO_CACHE
  • DDLキャッシュ （非表示）-データベースへのDESCRIBE呼び出しをキャッシュします。書き込み操作で使用されます


...そしてコレクションを自動的にキャッシュするものはありません。

 $temp = array(); $collection = Mage::getModel('catalog/product')->getCollection()->addAttributeToSelect(...); foreach ($collection as $product) { $product = $product->load($product->getId()); $temp[] = $product->getSku(); }


収集サイクル。 内部に負荷モデルがない場合：

コレクションのキャッシュ



キャッシュなし： 



 $collection = Mage::getModel('catalog/product')->getCollection() ->addAttributeToSelect('*');


initCacheメソッドの使用： 



 $collection = Mage::getModel('catalog/product')->getCollection() ->addAttributeToSelect('*') ->initCache(Mage::app()->getCache(),'our_data',array('SOME_TAGS'));


カスタムキャッシュの実装： 



 $cache = Mage::app()->getCache(); $collection = $cache->load('our_data'); if(!collection) { $collection = Mage::getModel('collection/product')->getCollection()->addAttributeToSelect('*')->getItems(); $cache->save(serialize($collection),'our_data',array(Mage_Core_Model_Resource_Db_Collection_Abstract::CACHE_TAG)); } else { $collection = unserialize($collection); }


キャッシュを使用せずに、Magentoが提供する方法を使用し、キャッシュモデルの方法に基づいて、どこにも見たことがない松葉杖を使用したサンプルを考えてみましょう。 すべてのテストで、リクエストをコンパイルした後、データをロードし、コレクションをオブジェクトの配列に変換したことに注意してください。



結果




キャッシュがなければ、実際には驚くべきことは何もありません...すべてが通常通りです。



しかし、マゼンタのキャッシュを使用して、時間が長くなったのを見て驚いた。 EAVコレクションは最初に商品テーブルからエンティティを読み込むため（これはまさにキャッシュされているものです）、それから別のリクエストで属性値を選択し、オブジェクトを埋めるため、一般的にEAVキャッシングは愚かな仕事です。 そこのフラットでは、1つのテーブルのすべてが追跡されます。 しかし、それでも、データベースを使用するよりもキャッシュを使用する方が時間がかかります（ファイルシステムとredisの両方でテストしました-カンマの後の4桁目が異なります...つまり、2kエンティティには存在しません）。 InitCacheメソッドの本質は、最初にコレクション自体のすべてのデータ（ページネーション、フィルター、イベントなど）を収集し、sql要求からハッシュを作成してキャッシュ内で検索し、そこに何かがある場合、 anserlsizes、すべてのイベントと後続のメソッドが起動されます。 これはプロセス全体で最も遅い手順です。データベース内の単純なクエリよりもキャッシュが遅いのはここです。 しかし、その後、データベースにリクエストを送信しません...すでにそれほど怖くはありません。



私がひざの上に書いたキャッシュを使用した別の例があります。そこで、コレクションの最終結果をキャッシュし、すべてのイベントをバイパスして属性をロードします。 これは、EAVおよびフラットコレクションで機能します。



count（）およびgetSize（）の適切な使用



getSize（） 



 $size = Mage::getModel('catalog/product')->getCollection() ->addAttributeToSelect('*') ->getSize();


カウント（） 



 $size = Mage::getModel('catalog/product')->getCollection() ->addAttributeToSelect('*') ->count();


結果




メソッドの違いは、 count（）がコレクション内のすべてのオブジェクトをロードし、通常のphpカウントでオブジェクトの数をカウントし、数を返すことです。 getSizeはコレクションをロードしませんが、制限、注文、選択可能な属性のリストがなく、COUNT（*）のみが存在するデータベースへの要求をさらに1つ生成します。



両方の方法の使用例は次のとおりです。



データベースに値があるかどうか、またはその数を知る必要がある場合は、getSizeを使用します。コレクションをロードする必要がある場合、または既にロードされている場合はcount（）を使用します。コレクションにロードされた要素の数を返します。



getFirstItemおよびsetPage（1,1）の適切な使用



getFirstItem（） 



 $product = Mage::getModel('catalog/product')->getCollection() ->getFirstItem();


setPage（1,1） 



 $product = Mage::getModel('catalog/product')->getCollection() ->setPage(1,1) ->getFirstItem();


負荷（） 



 $product = Mage::getModel('catalog/product')->load(22);


結果




getFirstItemの問題は、コレクション全体をロードしてから、foreachの最初の要素を返すだけで、そうでない場合は空のオブジェクトを返すことです。



setPage（別名$ this-> setCurPage（$ pageNum）-> setPageSize（$ pageSize））は、選択範囲を厳密に1レコードに制限します。これにより、結果のロードが大幅に高速化されます。



読み込みでもgetFirstItemより高速ですが、1つの要素のコレクションから取得するよりも読み込みが遅くなることに注意してください。 これは、ロードが常にEAVテーブルで機能するためです。





結論



上記のすべてを要約するために、Magentoで作業するすべての人にアドバイスしたいと思います。



  • コレクションから取得したオブジェクトに対してloadメソッドを再度呼び出さないでください。
  • 必要な属性のみをダウンロードする
  • プロジェクトに該当する場合は、フラットテーブルを使用します。
  • countを使用してロードされたコレクションの結果をカウントし、getSizeを使用してすべてのレコードの数を取得します
  • setPage（1,1）または同様のメソッドなしでgetFirstItemメソッドを使用しないでください
