1.はじめに。


独自のデータベースを作成する場合は、データベースの設計ルールを遵守することをお勧めします。これにより、データの長期的な整合性とメンテナンスの容易さが保証されます。 このガイドでは、データベースとは何か、およびリレーショナルデータベースを設計するためのルールに従うデータベースを設計する方法について説明します。



データベースは、大量の関連情報を保存および受信できるプログラムです。 データベースは、 情報を含むテーブルで構成されます 。 データベースを作成するとき、作成する必要のあるテーブルと、テーブル内の情報間に存在する関係について考える必要があります。 つまり、データベースの設計について考える必要があります。 前述のように、 適切なデータベース設計により、データの整合性とメンテナンスの容易さが保証されます。



構造化照会言語（SQL）。


データベースを作成して、情報を保存し、必要に応じてこの情報を取得します。 これは、データベースに情報を挿入、挿入（ INSERT ）できなければならず、データベースから情報を選択できる（ SELECT ）ことができることを意味します。

3.リレーショナルデータベースの特性。


リレーショナルデータベースは、大量の情報をすばやく保存および取得できるように設計されています。 以下は、リレーショナルデータベースとリレーショナルデータモデルのいくつかの特性です。



キーの使用。


テーブル内のデータの各行は、主キーと呼ばれる一意の「キー」によって識別されます。 多くの場合、主キーは自動的にインクリメントされる（自動インクリメント）番号（1、2、3、4など）です。 キーを使用して、さまざまなテーブルのデータをリンクできます。 1つのテーブルの主キー値を別のテーブルの行（レコード）に追加して、これらのレコードをリンクできます。



構造化照会言語（SQL）を使用すると、キーでリンクされているさまざまなテーブルのデータを一度に選択できます。 たとえば、usersテーブルの識別子（id）3（マイク）を持つユーザーに属するすべての注文を、ordersテーブルから選択するクエリを作成できます。 キーについては、後の部分で説明します。





このテーブルのid列は主キーです。 各エントリには、多くの場合、番号である一意の主キーがあります。 usergroup列は外部キーです。 名前から判断すると、明らかにユーザーグループを含むテーブルを指します。



データの冗長性の欠如。


リレーショナルデータモデルのルールを考慮して作成されたデータベースプロジェクトでは、ユーザー名などの各情報は1か所にのみ保存されます。 これにより、複数の場所でデータを操作する必要がなくなります。 データの重複はデータの冗長性と呼ばれ、適切なデータベース設計では避ける必要があります。



入力制限。


リレーショナルデータベースを使用すると、列に格納できるデータの種類を決定できます。 整数、10進数、小さなテキストチャンク、大きなテキストチャンク、日付などを含むフィールドを作成できます。





データベーステーブルを作成するとき、各列にデータ型を指定します。 たとえば、varcharは最大文字数が255の小さなテキストフラグメントのデータ型であり、intは数字です。



データ型に加えて、RDBMSでは、入力できるデータをさらに制限できます。 たとえば、長さを制限するか、この列のエントリの値の一意性を強制します。 後者の制限は、ユーザーのログイン名（ログイン）または電子メールアドレスを含むフィールドによく使用されます。



これらの制限により、データの整合性を制御し、次のような状況を防ぐことができます。



-番号が表示されると予想されるフィールドにアドレス（テキスト）を入力します

-このインデックス自体の長さを100文字で指定して、リージョンインデックスを入力します

-同じ名前のユーザーを作成する

-同じメールアドレスを持つユーザーを作成する

-誕生日フィールド（日付）に重量（数値）を入力します



データの整合性を維持します。


フィールドのプロパティを調整し、テーブルを相互にリンクし、制限を設定することにより、データの信頼性を高めることができます。



権利の譲渡。


ほとんどのRDBMSには、特定のユーザーに特定の権限を割り当てることができるアクセス許可設定があります。 ユーザーに許可または拒否できるアクション：SELECT（選択）、INSERT（挿入）、DELETE（削除）、ALTER（変更）、CREATE（作成）など これらは、構造化照会言語（SQL）を使用して実行できる操作です。



構造化照会言語（SQL）。


データの保存、選択、変更など、データベースに対して特定の操作を実行するために、構造化照会言語（SQL）が使用されます。 SQLは比較的理解しやすく、 SQL JOINステートメントを使用して複数のテーブルから関連データをフェッチするなど、スタック選択。 前述のとおり、このガイドではSQLについては説明しません。 データベース設計に焦点を当てます。



データベースの設計方法は、データベースからデータを取得するために完了する必要があるクエリに直接影響します。 これは、ベースがどうあるべきかを考える必要があるもう1つの理由です。 適切に設計されたデータベースを使用すると、クエリをよりクリーンでシンプルにできます。







移植性。


リレーショナルデータモデルは標準です。 リレーショナルデータモデルのルールに従うことにより、データを比較的簡単に別のRDBMSに転送できるようになります。



前述したように、データベースの設計は、データとその関係を特定し、この問題を解決した結果を紙に（またはコンピュータープログラムで）作成することです。 データベースの設計は、データベースの作成に使用するRDBMSとは無関係です。
