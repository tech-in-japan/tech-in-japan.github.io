使用できないかもしれない新しいio.js機能

オリジナル： 使用しないかもしれない新しいio.js機能



io.jsプラットフォームは急速に開発されており、Nodeエコシステム全体に大きな影響を与えています。 短期間で、io.jsで多くの修正が行われ、パフォーマンスが改善され、新しい機能が追加されました。



io.jsの開発を行っていない場合でも構いません-この記事では、プラットフォームの主な機能を簡単に紹介します。 終了するとき、つまり古いものから新しいものへとそれらを考慮します。 必要に応じて、関連するドキュメントへのリンクを提供します（英語）。 各新機能について説明し、最初のバージョンを示します。 コードは、io.jsで利用可能な標準のES6機能を使用して記述されています。 それでは始めましょう...



ES6機能（1.0.0）


当初、io.js開発者のタスクの1つは、V8の機能にアプローチすることでした。つまり、利用可能なすべてのES6機能を最大限に活用することでした。 デフォルトのビルトインジェネレーターや他のES6機能は 、フラグがなくても非常に安定して動作すると信じられていました。 ES6の情報を急いで探してはいけないことをお勧めします。まずは、記事をご覧ください。この記事へのリンクを提供します。



ドキュメントリンク： ES6の機能



注 ：デフォルトでは、Node 0.12にはいくつかのES6機能があります。 --harmonyフラグの下にさらに多くの機能があります。 ただし、ES6関数の操作とアクセシビリティはio.jsとは異なります（たとえば、デフォルトでは、ジェネレーターはNodeに存在しません）。

EPIPE、EMFILE、ENOENTとは何か疑問に思ったことはありませんか？ 私は個人的にこの質問を何度も自問しました。 ありがたいことに、io.jsではバグのドキュメントが大幅に改善されました。

スレッドの簡単な実装（1.2.0）


正式には、ストリームを適用する必要がある場合、ベースストリームを展開し、ストリームのタイプに応じて1つ以上のメソッドを適用します。 たとえば、Transformストリームは、_transformメソッドと_flushメソッドの使用をオプションで想定しています。 以下の例は、Streamストリームがファイルを行に分割する方法を示しています。 

var stream = require('stream') var liner = new stream.Transform( { objectMode: true } ) liner._transform = function (chunk, enc, done) { var data = chunk.toString() if (this._lastLineData) { data = this._lastLineData + data } var lines = data.split('\n') this._lastLineData = lines.splice(lines.length-1,1)[0] lines.forEach(this.push.bind(this)) done() } liner._flush = function (done) { if (this._lastLineData) { this.push(this._lastLineData) } this._lastLineData = null done() } module.exports = liner


メソッドのすべての「内臓」が外側にあるため、コードはかなり乱雑に見えます。 これで、これらのメソッドをオプションとしてコンストラクターに渡すことができます。 アンダースコア（_）を削除するだけです。 ライナーを使用すると、コードを変更して（ES6の拡張機能からオブジェクト表記を使用して）、次のものを取得できます。 

 'use strict' const stream = require('stream') let liner = new stream.Transform({ // Include any existing constructor options objectMode: true, // This is the _transform method transform (chunk, enc, done) { let data = chunk.toString() if (this._lastLineData) { data = this._lastLineData + data } let lines = data.split('\n') this._lastLineData = lines.splice(lines.length - 1, 1)[0] lines.forEach(this.push.bind(this)) done() }, // This is the _flush method flush (done) { if (this._lastLineData) { this.push(this._lastLineData) } this._lastLineData = null done() } }) module.exports = liner


ドキュメントリンク： ストリームコンストラクター



ドメイン（1.2.0）のすべてのIPアドレスを表示する機能


以前のバージョンのio.jsでdns.lookupコマンドを登録した場合、最初のアドレスのみが与えられます。 オプション{all：true}があり、アドレスの配列全体を取得できます。 

 'use strict' const dns = require('dns') // Returns first address dns.lookup('google.com', console.log) // => '173.194.46.40' 4 // Returns all resolved addresses in an array dns.lookup('google.com', { all: true }, console.log) /* => [ { address: '173.194.46.40', family: 4 }, { address: '173.194.46.38', family: 4 }, ... { address: '2607:f8b0:4009:804::1007', family: 6 } ] */


ドキュメントリンク： dns.lookupについて



UnhandleRejectionイベント（1.4.1）


約束を拒否しても、誰もそれを知らない場合、本当に拒否されたと考えることはできますか？ もちろん、これは起こりましたが、再び-誰もそれについて知りません！ ご想像のとおり、ここにコードの多くの複雑なエラーの原因があります。 たとえば、次のコードに注意してください。 

 'use strict' let delay = function (ms) { return new Promise(function (resolve) { setTimeout(resolve, ms) }) } delay(2000) .then(function (data) { data.foo = 'hello' })


間違いありませんか？ 遅延関数は、一定時間後に実行されるPromiseを返します。実行されると、未定義とReferenceErrorが発生します。 これは、未定義の値でfooプロパティにアクセスしようとしたために発生します。 一方、私たちは何が起こったのか決して知ることができず、生の失敗に誰も注意を払わなかったため、頭の後ろを引っ掻いて驚いたでしょう。 一部のライブラリでは、そのような約束の可能性が指定されていますが、ES6にはこれに関する情報がありません。 幸いなことに、プロセスを介してunhandledRejectionイベントを使用してコードをテストできます。 

 process.on('unhandledRejection', function (er) { console.log('got unhandled rejection', er.stack) })


ES6でpromiseを使用する場合、失敗が気付かないようにunhandledRejectionを設定することをお勧めします。



ドキュメントリンク： process.on（「unhandledRejection」）



注 ：また、rejectionHandledイベントもあります。 イベントループの次のステップで発生するプロミスの失敗に対処する必要がある場合に使用されます。 rejectHandledは、unhandledRejectionで誤ってふるいにかけられたプロミスを無効にする必要がある場合に役立ちます。 詳細については、 ドキュメントを参照してください。

バッファー＃indexOfメソッド


次に、文字列、バッファ、または番号を使用してバッファ内を検索する便利な方法を説明します。 このメソッドは、Array＃indexOfと同じように動作します。つまり、バッファで最初に見つかった一致の開始位置のインデックスを返します。 必要に応じて、追加のパラメーターとして開始インデックスを設定できます。 

 'use strict' const assert = require('assert') let buf = new Buffer('abc def ghi') assert.equal(buf.indexOf('abc'), 0) assert.equal(buf.indexOf('bc'), 1) assert.equal(buf.indexOf('def'), 4) assert.equal(buf.indexOf('c'), 2) assert.equal(buf.indexOf('c', 4), 11)


追加のメソッドlastIndexOfはまだ議論中です。

プリロードモジュール（1.6.0）


これで、スクリプトの実行中またはREPLを使用してモジュールをプリロードできます。 これは、-rまたは--requireフラグを使用して実行できます。 例： 

 iojs -r ./foo -r bar my-app.js


次と同じ： 

 require('./foo') require('bar') require('./my-app')


プリロードモジュールは、io.jsを使用する新しい可能性を提供します。 たとえば、便利な作業アプリケーションに新しい機能を追加できます。 または：順序付けられていないアレイの状態のスナップショットを取得して、データ漏洩を排除するアプリケーションを作成します。 これを行うには、この構成をサポートする必要はありません。 

 if (cfg.useHeapdump) require('heapdump')


必要な機能が必要な場合にのみ、プリロードされたモジュールでアプリケーションを実行する必要があります。 

 iojs -r heapdump app.js


別の使用例は、コンパイラー（Babel、CoffeeScriptなど）を使用する場合です。 たとえば、Babelを使用してES6またはES7でコードをコンパイルする場合、次のように記述する必要があります。 

 require('babel/register') require('./my-actual-app')


これで、ネイティブアプリケーションを使用せずにコマンドラインから直接設定できます。 

 iojs -r babel/register my-actual-app.js


注 ：現時点では、BabelはNodeを完全にサポートしています。 特別なツールを使用すると、上記の操作を実行できます。



同期I / Oによるトラブルシューティング（2.1.0）


Synchonous I / Oは、特にシェルスクリプトにとって便利ですが、多くのアプリケーション（サーバーなど）のパフォーマンスが大幅に低下します。 コードでSyncを検索できますが、命名スキームに従わないサードパーティモジュールを使用するとどうなりますか？ これは--trace-sync-ioフラグが助けになる場所です。 同期要求が実行されると、スタックトレースが通知されます。



簡単な例を見てみましょう： 

 'use strict' const http = require('http') const cp = require('child_process') http.createServer(function (req, res) { let stdout = cp.execFileSync('whoami') res.end(`${stdout}\n`) }).listen(3000)


各リクエストの同期コードを処理するHTTPサーバーがあります。 このオプションは推奨できません。 したがって、iojs --trace-sync-io server.jsを実行してからhttpL // localhost：3000に切り替えると、コンソールに次の警告が表示されます。 

 WARNING: Detected use of sync API at spawnSync (child_process.js:1241:27) at execFileSync (child_process.js:1291:13) at /Users/wavded/Projects/whats-new-iojs/server.js:6:19 at emitTwo (events.js:87:13) at emit (events.js:172:7) at parserOnIncoming (_http_server.js:474:12) at parserOnHeadersComplete (_http_common.js:88:23) at socketOnData (_http_server.js:325:22) at emitOne (events.js:77:13)


すべての少し


注目に値するいくつかの興味深い機能：

  1. require（ './'）とrequire（ '。'）（1.6.2）を書くことができます。
  2. console.logまたはutil.inspectを使用すると、ES6、Promise、Map、およびSetオブジェクトが美しく見えます（2.0.0）。
  3. os.tmpdir（）は、すべてのオペレーティングシステム（2.0.0）で同様に機能します。 以前は、他のエラーは言うまでもなく、一部のオペレーティングシステムではスラッシュが生成されていました。 これで、この問題に遭遇することはありません。
  4. その他の多くの改善：クエリ文字列の解析（1.6.1）、メモリtlsの削減（2.0.0）、より高速なprocess.nextTick（2.0.0）および単一引数のutil.format（2.1.0）。
  5. 段階的なセキュリティの改善io.js。 Diffie-Hellman方式を使用してキーを交換する場合、パラメーターは少なくとも1024ビット（2.1.0）でなければなりません。
