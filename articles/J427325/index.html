ただし、単にビューモデルクラスを記述するだけでは十分ではありません。 ビューモデルプロパティまたはビュープロパティが変更されたことをバインドメカニズムに何らかの方法で通知する必要もあります。 これには、 PropertyChangedイベントを含む特別なINotifyPropertyChangedインターフェイスがあります。 基本クラスBaseViewModelのフレームワークでこのインターフェイスを実装します 。 将来的には、インターフェイスの実装を複製しないように、この基本クラスからすべてのビューモデルを継承します。 そのため、 ViewModelsディレクトリをプロジェクトに追加し、 BaseViewModel.csファイルをこのディレクトリに追加します 。 次のプロジェクト構造が得られます。

SynchronizedTextプロパティが変更されたときにPropertyChangedイベントを発生させ、このイベントを監視する必要があることをビューに通知する必要があります。 そのため、イベントをトリガーするには、ビューモデルコードを変更します。 







 public class MainWindowViewModel : BaseViewModel { private string _synchronizedText; public string SynchronizedText { get => _synchronizedText; set { _synchronizedText = value; OnPropertyChanged(nameof(SynchronizedText)); } } }


ここで何をしましたか？ テキストを格納するための非表示フィールドを追加し、既存のプロパティにラップします。このプロパティを変更するときは、非表示フィールドを変更するだけでなく、ベースビューモデルで定義されたOnPropertyChangedメソッドを呼び出し、ベースでも実装されたINotifyPropertyChangedインターフェイスで宣言されたPropertyChangedイベントを発生させますモデルを表示します。 テキストが変更されるたびにPropertyChangedイベントが発生し、変更されたビューモデルのプロパティの名前が渡されます。







まあ、ほぼすべて、フィニッシュライン！ PropertyChangedイベントをリッスンするビューを指定するために残ります。 







 <TextBox Text="{Binding Path=SynchronizedText, UpdateSourceTrigger=PropertyChanged, Mode=OneWayToSource}"/> <TextBlock Text="{Binding Path=SynchronizedText, UpdateSourceTrigger=PropertyChanged, Mode=OneWay}"/>


どのトリガーによって更新が行われるべきかを示したという事実に加えて、この更新が追跡される方向（ビューからモデルへ、またはその逆）も示しました。 テキストボックスにテキストを入力するため、ビューの変更のみに関心があるため、 OneWayToSourceモードを選択します。 テキストブロックの場合、すべてが正反対です。ビューに表示するためにビューモデルの変更に関心があるため、 OneWayモードを選択します。 変更を両方向で追跡する場合、 Modeをまったく指定することも、 TwoWayを明示的に指定することもできません。
