まれな言語-IDEなしで、しかし便利にコーディング

IDE vs Notepad / VIM / Emacsのトピックで青くなるまで議論することができます。 IDEが存在する限り。 ただし、使用する必要のある言語（または方言）に簡単に出くわすことができます。そのためのIDEは存在しないか、有料で高価なだけであるか、既に古くて珍しいOSのエミュレーターでしか実行されません。 しかし、あなたは働く必要があります。 幸いなことに、腕を選択して「シャープ」にするだけでよい汎用ツールがあります。 以下は、今日のまれなバージョンのアセンブラーでのVIMバンドル、トランスレーター、およびいくつかのctagの使用について説明しています。



翻訳者のみがいます


開発チームで仕事を始めてすぐに、おおまかに言ってPDP-11の1806BM2のアセンブラに出会いました。 さらに、ターゲットデバイスは、信号を変換および処理するための少数の部品を備えた特定のボードであり、機器はWindows（オプションなし）を備えたPCです。 開発中のデバイスは異なり、それらのプログラムは300〜3000行のコードです。



このプロセスに参加したとき、デバイス名のフォルダーがプロジェクトに対応していることがわかりました。 フォルダーには、アセンブラーファイル（すべてのasmファイルは同じ名前を持ち、コンパイルはそのように構成されています）、パラメーターなしのbatファイルがあり、パラメーター付きの別のbatファイルを呼び出し、次に、アセンブラーexeファイルを実行します（自然に横たわっています）すぐそこ）。 さらに、プロジェクトフォルダーには、バイナリとリスト、バイナリをROMファームウェアファイルに変換するプログラム、変換用の補助テキストファイル、ファームウェアファイルをプログラマーが理解できる形式に変換する別のプログラム、ファームウェアファイル（プログラムはROMの代わりにフラッシュに書き込まれ始めました）、さらにdirinfo.txtで、誰もが何かを書くことは非常にまれでした。



作業は次のようになりました。コンソールファイルマネージャーでは、組み込みエディターによってプログラムテキストが修正されました。 編集後、最初のバッチファイルが起動されました（混同しないことが重要です！）。 次に、コンソール出力のエラーを確認します。 構文上の妨害がない場合-プログラムはROMエミュレーターに流し込まれ、ハードウェアの操作性をチェックします。 そのため、プログラムは追加され、期待される動作を開始するまでチェックされます。 次に、生成されたファームウェアがROMに書き込まれます（後で、ROMがフラッシュに置き換えられました）。



最初はいつものように働きました。 利便性次第ではありませんでした。コマンドシステム、プログラム作成の機能、ソフトウェアが記述されているデバイスのハードウェアを理解する必要があります。 何かが判明し始めると、改善の考えが生じ始めました。 まず、構文の強調表示、エディターからのコンパイラーの起動、および通常の検索と置換が必要でした。



選択の小麦粉


最初は、あらゆる種類のインターネットでIDEを見つけようとしました。これはPDP-11、つまりDCK、BC、UKSCであり、見つけるべきものがあるはずだからです！ そして発見されましたが、UKSCのエミュレータでのみ機能しました。 コンソールIDEもエミュレータも環境にもエミュレータを実行する必要もありませんでしたが、これは最も重要なことではありませんでした。 最悪なことに、結果のバイナリは「従来の方法」で作成されたバイナリとは異なりました。 私は理由を理解していませんでした-エミュレーターで希少性を終了するか、さらに悪いことに修正ソフトウェアを書くのに十分ではありませんでした。 私は好みと遊女で自分のワークスペースを整理することにしました。



私はもっ​​と真剣にテキストエディタを探し始めました。 検索プロセスで出会ったことから、私はUltraEditが一番好きでした（特にUEStudioバージョン）。 テキストファイルの比較、トランスコーディング、列選択、コンパイラだけでなく、現在のファイルの名前を使用して作成中のその他のマクロ、ツールバー拡張機能の起動など、いくつかの利点がありました。 すべてが良いです-しかし、支払われ、そして私の収入は安くはありません。 私はMiBEditorを試しました-印象的な製品（無料で1つのヘルメットで作られています）ですが、そうではありません。 簡単なLinuxの流行が助けました。 VIMについて学び、それを使用してみて、その機能を感知し、Wndowsのバージョンを発見しました。



プリセット


最初に行ったのは、vimrcで外観と動作を設定することでした：暗いウォンバットカラースキーム（黒に明るいグレー、好みに応じて調整）、行番号、行番号、スマートインデント、タブによりデフォルトで85幅に増加、無効なbak-files、開いているファイルによる作業フォルダーの自動変更。 彼は、ビープ音ではなく点滅する仮想呼び出しをオンにし、ツールバーを無効にし（何らかの理由で、私は使用していませんが、メニューを残しました）、印刷中にマウスを隠し、開いている分割バッファー間でCtrl-Tabスイッチを切りました。



ロシア語の文字を含むフォントを選択しました： 

set guifont=Courier_New:h10:cRUSSIAN "


ロシア語での作業を便利にするために、レイアウトに何かを追加しました。 

 set keymap=russian-jcukenwin "   set iskeyword=@,48-57,_,192-255 "  . .    set iminsert=0 "  - -  set imsearch=0 "    —  lang ctype Russian_Russia.1251


F12にレイアウトスイッチを掛けました： 

 imap <F12> <C-^> cmap <F12> <C-^>


現在、英語レイアウトでは、すべてのキーボードコマンドが正常に機能し、システムのCtrl-Shiftではなく入力言語を切り替える必要がありますが、VIM F12では、現在の言語が英語で「-INSERT-」または「-INSERT（ language）-"ロシア語。 少し不便です-通常、私のキーボードレイアウトはロシア語ですが、ここではVIMの起動時に英語に切り替えることを忘れてはなりません。



上記のすべてを個別のvimrc_rgに記録し、デフォルトの設定ファイルに接続しました。 

 source $VIM/_vimrc_rg


複数のプラグインもインストールしましたが、常に文字通り1つを使用します-QuickBuf（qbuf.vim、script_id = 1910）。これにより、F4で1つのウィンドウのオープンバッファーを切り替えることができます。 現在、タブは必要ありません（UltraEditで慣れていました）、さらに不快に思えます。 UltraEditのように、選択したブックマークを作成するインストール済みのビジュアルマーク（script_id = 1026）を時々覚えています。



そこで、VIMをプレーンテキストファイルのメインエディタとして使用し始めました。 私の要件の1つである通常の検索と置換は、すぐに使用できます。使用方法を学んでください。 さらに、多くの言語の強調表示と自動インデント-バッチファイルの編集などが非常に便利になりました。



バックライトと翻訳者の設定


次に、編集を強調表示して埋め込むための新しいファイルタイプa1806を導入しました。 vimfilesフォルダーのルートにfiletype.vimファイルを作成しました。 

  augroup filetypedetect au BufNewFile,BufRead *.msa setf a1806 au BufNewFile,BufRead *.a51 setf asm8051 augroup END


それからしばらくの間、彼はアセンブラーファイルに適切な拡張子を与え始めました-1806のmsa、MCS-51アセンブラーのa51、デフォルトでIntelアセンブラーに設定されたasmと混同しないように。



次に、vimfiles \構文で、彼はa1806.vimファイルを作成し、他の言語のハイライトファイルからやり直しました。 インターネットで見つかったキーワード、PDP-11のテキスト情報。 UKNTSおよびSMコンピュータに関するペーパーブックがいくつかありますが、Webページからコピーする方が簡単です。

このファイルには、特に次の行が含まれています。 

 syn match a1806Error "#@.*\|[a-zA-Z0-9_$]\{7,}:"


すばやく入力して間接アドレスを入力するとき、@＃と＃@を混同することがよくあり、このパターンは入力中のエラーを強調しています。 他の一般的なタイプミスは、この方法で特定できます。



構文の強調表示があります。 今コンパイル。 ユーティリティmakeとctagsを理解すると、ここではそれらがヘルパーではないことが明らかになりました。 アセンブラはありますが、リンカはありません-プロジェクトをモジュールファイルに分割することはできません。 makeの目標は1つだけで、そのためのファイルもありますが、気にする必要はありません。 ctagsでも同じです-ファイルは1つだけで、検索する特別なものはありません。ラベルはアスタリスクで簡単に見つかります。 長いファイルはブックマークよりも便利です。これはVIMにもあります。 しかし、すべてのプロジェクトフォルダーのアセンブラーはfiです！ C：\ドライブのルートにbinフォルダーを作成し、PATHに書き込みます（別のアセンブラーのmakeとctagsをそこに配置します）。 そこで、アセンブラーとmk1806.batを配置しました。マイクロヘルプと渡されたパラメーターの有効性のチェックに加えて、バイナリとリストが消去され、その後アセンブラーが呼び出されます。 

 :VALIDDIR CD %2 %3 %4 %5 %6 %7 %8 %9 :WORK2 if not exist *.bin goto NOBIN attrib -r *.bin del *.bin :NOBIN if not exist *.lst goto NOLST attrib -r *.lst del *.lst :NOLST if exist %1 C:\bin\cmplr1806\asm.exe %1 -m if not exist %1 echo Error: File %1 not exist!


バッチファイルは、ファイルの名前とそのパスの2つのパラメーターを受け入れます。 makeの出力を損なわないように、ファイル内のコマンドを表示するモードは無効になっています（@ ECHO OFF）。



vimfiles \ ftpluginでa1806.vimファイルを作成しました。 

 " Language: 1806 assembler " Maintainer: " Last Change: "           lcd %:h "    setlocal makeprg=C:\bin\mk1806.bat\ %:t\ %:p:h "      setlocal autowrite nmap <buffer> <F9> :make<CR> imap <buffer> <F9> <Esc>:make<CR> "     set efm=%-G,%-G\ BK-10/11%.%#,%-P\ Opened\ %f,%Eline\ %l\\,%.%#,%Z%m,%-GFile\ compiled%.%#


Errorformatは経験的に決定されました（asm.exeにはドキュメントがまったくありません）。つまり、エラーなしでさまざまな特別なエラーを含むファイルのアセンブルを開始し、コンソール出力を調べ、VIMヘルプのテンプレートをコンパイルしました。

エラー出力の例（明確にするために行番号を手動で追加）： 

 1 2 BK-10/11 Assembler 3 4 Opened 1.ASM 5 Line 61,Address 000320: 6 Bad operator [AKSJDFH]. 7 Opened 1.lst 8 9 File compiled 346 bytes. 10


この場合、エラー形式は次の要素で構成されます。

  • ％-G、-空行が発生しても無視します。 （「G」は一般情報であり、「％-」は出力から除外されます。「G」の後に何も表示されない場合は、空の行を意味します）。 1、3、8、および10行に適用されます。
  • ％-G \ BK-10 / 11％。％＃、ここで '％。％＃'は正規表現パターン '。*'の作成に使用されます。つまり、「BK-10 / 11」で始まる行（コマンド「：set」は「\」でエスケープされ、無視されます。 2行目に適用されます。
  • ％-P \ Opened \％f、-単語「Opened」の後のファイル名（％f）を持つ単一行メッセージ（％-P）を解析します。「％-P」は行が表示されないことを示しますが、ファイル名を覚えています。次のエラーそれに関連します。 4行目と7行目に適用されます。
  • ％Eline \％l \\、％。％＃、-複数行エラーメッセージ（％E）の始まり。「line」の後に行番号（％l）が続き、カンマ（「\\、」-コンマをエスケープする必要があります）バックスラッシュ。ただし、「：set」コマンドで複製する必要があります。次に、出力に関係する行（「％。％＃」=「。*」）（この例では、アドレスですが、数字として表示されます）エラー-「320エラー」。これはバグではなく機能です。
  • ％Z％m、-実際のエラーメッセージ（％m）を含む複数行メッセージ（％Z）の終わり。 タイプ5と6の行ペアでは、％Eパターンと％Zパターンが連動します。
  • ％-GFile \ compile％。％＃繰り返しますが、出力からスローされる一般的な情報、9行目のルール。


efmパターンによれば、ファイル名は4と7に格納されていますが、行1..4と7..10は出力されません。 行5と6がない場合、出力は空で、エラーはありません。 タイプ5と6のラインは、複数のペアになる場合があります。 これらのペアのそれぞれがエラーメッセージリストアイテムを形成します。



ほら！ F9で自動保存を使用してコンパイルします。 構文エラーが検出されると、VIMはすぐに、見つかったエラーのリストの最初のエラーに対応するコード行にカーソルを置きます。



十分なデバッガはありませんが、あまりにも多くのものが割り込み、メモリへの直接アクセス、および無関係なハードウェアに結び付けられているため、ここでは役に立ちません。 VIMに組み込まれている完全なファイル比較をまだマスターしていません。主にWinMergeを使用しています。 それでも、それははるかに便利になりました。



エンパワーメント


そして1806年は私たちから少し離れ始め、NM6403プロセッサをベースにしたデバイスで作業が始まりました。 アセンブラーでも。 そして、ソースファイルの次の拡張子を思い付かないように、asm.vimファイルをvimfiles \ ftpluginに追加しました。 

  if getline(1) =~ '^;.*a1806\>' set filetype=a1806 endif if getline(1) =~ '^//.*a6403\>' set filetype=a6403 endif


拡張子が.asmのアセンブラファイルでは、最初の行の入力が開始されました

「; A1806」または「// a6403」。 vimfiles \ syntax \ a1806.vim（「アセンブラー言語の説明」に従って非常に簡単かつ迅速に記述）およびそのようなvimfiles \ ftplugin \ a6403.vimを強調する構文を追加しました。 

 " Language: 6403 assembler " Maintainer: " Last Change: setlocal makeprg=c:\bin\make.exe "           lcd %:h "      setlocal autowrite nmap <buffer> <F9> :make<CR> imap <buffer> <F9> <Esc>:make<CR> nmap <buffer> <F10> :make TAGS<CR> imap <buffer> <F10> <Esc>:make TAGS<CR> set efm=\"%f\"\\,%l\ :\ %m


これが機能するために、彼は各プロジェクトフォルダーにあり、他の開発者が使用するbatコンパイラー起動ファイルに隣接するMakefileを作成しました。 私のMakefileには、TAGSターゲットと、ALLターゲットからこのターゲットへの呼び出しがあります。 

 TAGS : @c:/bin/ctags.exe --options=c:/bin/a6403.ctags $(SRC_ASM) *.asi


$（SRC_ASM）はASMソースをリストし、* .asiにはインクルードファイルが含まれます。



上記のフォルダーC：\ binには、make.exe、ctags.exe、a6403.ctagsがあり、ctags.exeのパラメーターとして渡されます。 

 --langdef=a6403 --langmap=a6403:.asm --language-force=a6405 --regex-a6403=/^<([a-zA-Z0-9_]+)>/\1/L,labels/ --regex-a6403=/^[ \t]*const[ \t]+([a-zA-Z0-9_]+)/\1/C,consts/ --regex-a6403=/^[ \t]*#define[ \t]+([a-zA-Z0-9_]+)/\1/D,defs/ --regex-a6403=/^[ \t]*global[ \t]+([a-zA-Z0-9_]+)[ \t]*:[ \t]*(word|long)/\1/V,vars/


箱から出したctagsは、NM6403について何も知りません。 最初の行は、新しい言語名を定義します。 2行目はファイル拡張子について説明しています。 拡張子は、たとえば「.asm.asi.inc」のように連続して記述されます。 後ろに文字がないドットは、拡張子のないファイルの使用を意味します。 ただし、これは特に便利なオプションではありません。ここでは.asi拡張子を指定しませんでしたが、ctagsを呼び出すときに* .asiを指定しました。ユーティリティは必要なものをすべてスキャンしました。 4行目では、同じ拡張機能のctagsに組み込まれている言語ではなく、この特定の言語に対して上記の拡張機能を使用しています。 残りの行は、「<Label>」という形式のラベル、「const Name」という形式の定数、「＃define Expression」という形式のマクロ定義、および「global Name：word」という形式のグローバル変数をそれぞれ説明します。 「。 ここで、gf-go fileコマンド、カーソルの下の名前がインクルードファイルを表示するのに非常に便利なファイルへの移行を思い出したいと思います。



別の微妙な点：IDEを含むいくつかの異なる開発ツールが作業コンピューターにインストールされます。 それらのいくつかはボード上に独自のmakeを持ち、それらのバイナリへのパスはインストール中にPATHで指定されます。 さらに、一部のmakeは、GNU makeの規則に従って作成されたMakefileに神経質に関連しています。 一般に、それらはバッチファイルであってもかまいません。 そのため、使用するmake（および同時にctags）へのパスをどこにでも書かなければなりませんでした。次のIDE / CADをインストールするたびにPATHを並べ替えて制御するよりも簡単で信頼性が高いです。



合計


今では、ファイル拡張子を気にせずにさまざまなアセンブラで作業し、さまざまなファイルに散らばっているコードを「ジャンプ」し、エディタを離れずにコンパイルし、コンパイル後に構文エラーのある行を取得する機会があります。 類推により、VIM +翻訳者+ ctagsバンドルの範囲を他の言語と一緒に使用する必要がある場合、それらの範囲を非常に簡単に拡張できます。 適切なプラグインを選択することにより、タグの操作に利便性を追加できます。 Makefileを使用して目的のトランスレーターにチューニングしてmakeを固定できます。直接トランスレーターを使用することもできます。 もちろん、IDEではありませんが、ノートブックでもありません！
