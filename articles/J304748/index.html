コンパイラ、インタープリター、およびJITの概要

PHP 7の誕生によって、抽象的な構文ツリー、ジャストインタイムコンパイラ、静的分析などに関する議論は終わりませんが、これらすべての用語はどういう意味ですか？ これらは、PHPの生産性を高める魔法の特性ですか？ もしそうなら、それはどのようにすべて機能しますか？ この記事では、プログラミング言語がどのように機能するかの基本を調べ、PHPスクリプトなど、コンピューターが起動する前に完了する必要のあるプロセスについて説明します。

トランスコンパイル



プログラムを「高速」にするにはどうすればよいですか？ これはさまざまな方法で実行できます。 Facebookで開発されたそのうちの1つはHipHopと呼ばれます（現在使用されているHHVMではなく、「古い」HipHopシステムを意味します）。 HipHopは、ある言語（PHP）を別の言語（C ++）に変換しました。 変換結果は、C ++コンパイラを使用してバイナリコードに変換できます。 彼のコンピューターは、インタープリターの形で追加の負荷なしに理解して実行することができます。 その結果、膨大な量のコンピューティングリソースが節約され、アプリケーションの実行速度が大幅に向上します。



この方法は、ソースからソースへのコンパイル、トランスコンパイル、またはトランスコンパイルとも呼ばれます。 実際には、バイナリコードへのコンパイルではなく、既存のコンパイラによってマシンコードにコンパイルできるものへの変換です。



トランスコンパイルを使用すると、バイナリコードを直接実行できるため、パフォーマンスが向上します。 ただし、このメソッドには欠点があります。コードを実行する前に、最初にトランスコンパイルを行い、次に実際のコンパイルを行う必要があります。 ただし、これはアプリケーションに変更が加えられた場合、つまり開発中にのみ行う必要があります。



トランスコンパイルは、ハード言語をより単純かつ動的にするためにも使用されます。 たとえば、ブラウザはLESS、SASS、SCSSで記述されたコードを理解しません。 しかし、その後、ブラウザが理解できるCSSに置き換えることができます。 CSSは保守が簡単ですが、追加でトランスコンパイルする必要があります。



編集



すべてがさらに速く動作するためには、トランスコンパイルの段階を取り除く必要があります。 つまり、私たちの言語をすぐにバイナリコードにコンパイルし、解釈やトランスコンパイルの形で追加の負荷をかけずにすぐに実行できます。



残念ながら、コンパイラを書くことは、コンピューターサイエンスで最も難しいタスクの1つです。 たとえば、バイナリコードにコンパイルする場合、32ビットLinux、64ビットWindows、または一般的なOS Xで実行するコンピューターを検討する必要がありますが、解釈されたスクリプトはどこでも簡単に実行できます。 PHPのように、スクリプトがどこで実行されるかを心配する必要はありません。 特定のOS向けのコードが存在する場合がありますが、他のシステムでスクリプトを実行することはできませんが、これはインタープリターの問題ではありません。



しかし、トランスコンパイルの段階をなくしても、コンパイルから逃げることはできません。 たとえば、C（コンパイルされた言語）で書かれた大きなプログラムは、ほぼ1時間コンパイルできます。 PHPでアプリケーションを作成し、変更が機能するかどうかを確認するまでさらに10分待つ必要があると想像してください。



最善を尽くして



解釈の実行に時間がかかり、コンパイルが実装が難しく、開発に時間がかかる場合、PHP、Python、Rubyなどの言語はどのように機能しますか？ 彼らはかなり速いです！



これは、解釈とコンパイルの両方を使用するためです。 それがどうなるか見てみましょう。



架空の言語をバイナリコードに直接変換するのではなく、それに非常によく似たもの（「バイトコード」と呼ばれる）に変換できるとしたらどうでしょうか。 このバイトコードがコンピューターの動作に非常に近い場合、その解釈は非常に高速になります（たとえば、1秒あたり数百万のバイトコード）。 これにより、アプリケーションはコンパイルされたものとほぼ同じ速度になり、インタープリター言語のすべての利点が保持されます。 最も重要なことは、変更ごとにスクリプトをコンパイルする必要がないことです。



とても魅力的です。 実際、多くの言語は同様の方法で動作します-PHP、Ruby、Python、さらにはJava。 ソースコードの行を順番に読み取って解釈する代わりに、これらの言語は異なるアプローチを使用します。



  • 手順1.スクリプト（PHP）を完全にメモリに読み込みます。
  • ステップ2.スクリプトを完全にバイトコードに変換/コンパイルします。
  • ステップ3.インタープリター（PHP）を使用してバイトコードを実行します。


実際、さらに多くのステップがあり、実際にはプロセス全体がはるかに複雑です。 ただし、一般に、説明されている3つの手順は、コマンドラインからスクリプトを実行するか、Webサーバーを介して要求を実行するのに十分です。



プロセスは簡単に最適化できます。Webサーバーを起動し、各リクエストがindex.php



スクリプトを実行するとします。 メモリに毎回ロードするのはなぜですか？ リクエストごとにすばやく変換できるように、ファイルをキャッシュすることをお勧めします。



別の最適化：バイトコードを生成した後、後続のすべてのリクエストに使用できます。 したがって、それもキャッシュできます（最も重要なことは、ソースファイルが変更されたときに、バイトコードが再コンパイルされることを確認してください）。 これは、PHPのOPCache拡張機能のように、オペコードキャッシュが行うこととまったく同じです。コンパイルされたスクリプトは、過剰なダウンロードやバイトコードへのコンパイルなしに後続のリクエストですばやく実行できるようにキャッシュされます。



最後に、高速化の最後のステップは、PHPインタープリターによるバイトコードの実行です。 次のパートでは、これを従来の通訳者と比較します。 混乱を避けるため、このようなバイトコードインタープリターは、「仮想マシン」と呼ばれます。これは、ある程度までマシン（コンピューター）の作業をコピーするためです。 これを、VirtualBoxやVMwareなどのコンピューターで実行されている仮想マシンと混同しないでください。 これは、Javaの世界のJVM（Java仮想マシン）やPHPの世界のHHVM（HipHop仮想マシン）のようなものです。 PythonとRubyには独自の仮想マシンがあります。 ある意味では、これらはすべて非常に専門的で生産的なバイトコードインタープリターです。



各VMは、特定の言語によって生成された独自のバイトコードを実行しますが、互いに互換性がありません。 Python VMでPHPバイトコードを実行することはできません。逆の場合も同様です。 ただし、PHPスクリプトをバイトコードにコンパイルするプログラムを作成することは理論的には可能であり、これはPython VMによって理解されます。 したがって、理論的には、PythonでPHPスクリプトを実行できます（深刻な課題です！）。

ソースからバイトコードへ



生成されたバイトコードを効率的に実行できるようになったので、ソースコードをそのバイトコードにコンパイルするタスクは残ります。



次のPHP式を検討してください。

それらはすべて等しく当てはまり、同じバイトコードに変換する必要があります。 しかし、それらをどのように読みますか？ 実際、独自のインタープリターでは、コマンドを解析し、スペースで区切ります。 つまり、インデントやスペース、1行でブラケット、2行で折り返しなどを使用できるPHPとは異なり、プログラマーはコードを同じスタイルで記述する必要があります。まず、コンパイラーはソースを変換しようとしますトークン内のコード。 このプロセスは、字句解析またはトークン化と呼ばれます。

解析



トークンを解析するとき、言語を構成するいくつかの「ルール」に従う必要があります。 たとえば、ルールがある場合があります。プログラムで最初に検出されたトークンはT_OPEN_TAG（<？Phpに対応）でなければなりません。

抽象構文ツリー



言語をトークン化して解析した後、バイトコードを生成できます。 PHP 5.6までは、解析中に生成されました。 ただし、プロセスに別のステージを追加すると、パーサーにバイトコードではなく、いわゆる抽象構文ツリー（AST）を生成させることができます。 これは、プログラム全体が抽象的に表現されるツリー構造です。 ASTは、バイトコードの生成を単純化するだけでなく、変換する前にツリーを変更することもできます。 ツリーは常に特別な方法で生成されます。 ツリーノードはif式であり、必然的にその下に3つの要素があります。

その結果、プログラムをバイトコードに変換する前に「書き直す」ことができます。 これはコードを最適化するために時々使用されます。 開発者がループ内で変数を繰り返し再計算し、変数の値が常に同じであることがわかっている場合、オプティマイザーはASTを書き換えて、毎回再計算する必要のない一時変数を作成できます。 ツリーはコードの小さな再編成に使用できるため、不要になった変数の削除など、高速に動作します。これは常に可能であるとは限りませんが、プログラム全体のツリーがある場合、このようなチェックと最適化ははるかに簡単に実行できます。 AST内では、変数が使用される前に宣言されているかどうか、または条件ブロックで割り当てが使用されているかどうかを確認できます（ if ($a = 1) {}

これは静的分析と呼ばれます。これにより、開発者が調和のとれた安全で高速なコードを作成するのに役立つ新しい機能、最適化、検証システムを作成できます。
