分散システムをスケーリングおよび構築するためのツールとしてのJMS



今日の世界では、テクノロジーと生活のルールは常に変化しています。 洗練されたソフトウェアは、このような変更にどのように適応できますか？ 新しいコンポーネント、新しいテクノロジーを統合して、その動作を中断せずにシステムのパフォーマンスを向上させる方法は？ 最も簡単な方法は、システムを同種の要素で構成されるモノリシックブロックとして認識するのをやめて、その違いにかかわらず、コンポーネントが通信できるようにするレイヤーを作成することです。 JMSは、このアプローチを実装するツールの1つです。

2. OracleのDmitry Chuykoと次のレポート。



CompletableFutureはすでにここにあります



標準ライブラリは、並列コンピューティングの構築を支援し、すべてのコアを有効に活用する必要があります。 Javaでは、Thread、Future、およびExecutorSeviceがかなり前から存在しています。 ただし、多くの場合、スレッドを「手動で」使用すると、典型的なエラーや困難が生じます。 最大のパフォーマンスを達成することは困難な場合があり、データ処理の一般的なスタイルに適合することは常に可能ではありません。 Java 8はソリューションを提供します。 たとえば、プルモデルの場合はストリーム（juStreams）を使用し、継続する場合はCompletableFutureの方が便利です。 ブロッキング呼び出しとコールバックの地獄を取り除くのがどれほど簡単か見てみましょう。 次のトピックについて説明します。

3. NaumenのEgor Titovとレポート：



JerseyベースのSwaggerを使用したRESTful APIのドキュメント化



現在、アプリケーションスキームは、クライアントとサーバーの2つの部分に分かれているため、非常に人気があります。 アプリケーションの両方の部分は、独立したチームによって開発できます。 この場合、多くの場合、自己文書化コードでは不十分であり、コマンドの「共通基盤」を説明する文書が必要です。 Swaggerはこれに役立ちます-RESTful APIのオンラインドキュメントと簡単なSDK作成オプション。 Jerseyは、サーバー側のRESTful APIの実装です。 これらのツールの使用方法を教えて示し、SDK生成のデモを試みます。
