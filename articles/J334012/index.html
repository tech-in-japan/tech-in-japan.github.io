リリースCLion 2017.2：新しいプロジェクトを作成し、IDEのパフォーマンスを向上させるためのウィザードでのClang-Tidy、C ++ 17との統合

こんにちは、Habr！ 今年の夏、私たちは場所を失望させました（確かにサンクトペテルブルクのCLionチーム）が、 CLion 2017.2の新しいリリースが成功したことを願っています！ この投稿では、新機能、重要なバグ修正について説明し、コメントで質問したり、特定の計画について質問したりする機会を提供します。









要するに、CLion 2017.2のリリースは以下に特化しています。



  • コードアナライザーの機能の強化（これは、組み込みツールとサードパーティツールの両方に適用されます-CLang-Tidy）
  • 新しいプロジェクトの作成ウィザードのC ++ 17
  • MSVCのPCHサポート（以下のすべての略語を解読します！）
  • デバッガーでの強制ステップイン
  • gmockにリンクされたターゲットの Googleテスト構成を自動的に作成します
  • 取り消し不可能な非同期CMakeナビゲーションおよびロードアクション
  • 全体的なパフォーマンスの改善
  • さらに多く！


今すぐ試す準備はできましたか？ 当社のウェブサイトから無料の30日間バージョンをダウンロードしてください ！

詳細が必要ですか？ 以下の詳細。 ちなみに、これらの目的のために特別に準備した小さなデモプロジェクトですべての新機能を試すことができます。



コードアナライザー



新しいクイックフィックスとパーサーの正確性の改善



CLionのクイック修正により、 Alt+Enter



を1回クリックするだけでコードの潜在的な問題を修正できます。 このリリースでは、それらのいくつかを改善しました。



したがって、たとえば、以前は未使用の変数のみを削除できました。 しかし、そのような変数のコンストラクターが自明でない場合、これは完全に正しい修正ではないかもしれません。 現在、CLionには2つのオプションがあります。変数を削除するか、変数を削除して初期化を終了し、コードのセマンティクスを保存します。







別の改善点はキャストです。 CLionは、明示的なキャストが必要な状況を積極的に指摘しています。 しかし、以前のクイックフィックスがC言語のスタイルで型キャストのみを使用していた場合、C ++からの型キャストのバリアント（ static_cast



、 dynamic_cast



、 const_cast



、 const_cast



。







このリリースサイクルを通して、パーサー/リゾルバー/コードアナライザーのバグ修正に積極的に取り組んできました。 多くの変更があり、それらはさまざまな領域にあります。



  • これは、 std::enable_if

  • GCC 5/6のSTLコンテナ内のネストされたテンプレートタイプを使用した正しい作業、

ここで、次のリリースの計画がわずかに異なることを直ちに予約する必要があります。 スポット修正の代わりに、言語サポート内のコードのレイヤー全体を書き直す予定です。



詳細は説明しませんが、以前の経験に基づいて、ほとんどの問題がどこにあるかを分析し、これらの領域をボトムアップでソートし、残りの基になる問題領域を選択したとだけ言います。 それらに対処します。 ちなみに、1年前、私たちはすでにオーバーロード解決のそのような手直しに従事しており、ユーザーレビューは非常に好意的でした。



Clang-Tidy統合



組み込みのコードアナライザーの改善に加えて、Clang- Clang-Tidyに基づく人気のある急速に開発されているツールにより、さまざまなチェックとクイック修正の数を増やすことにしました。 彼の最大の主張は、 C ++コアガイドラインからのチェックの部分的なサポートです。これは、C ++コミュニティのBjörnStraustrupおよびHerb Sutter開発者によって現在積極的に推進されています。 しかし、これは彼の唯一の利点ではありません！



重要な引数は、 独自のチェックを記述してClang-Tidyに追加する機能です。



Clang-TidyのCLionへの統合とは何ですか？ IDEユーザーにとって、Clang-Tidyからのチェックは、組み込みコードアナライザーからのチェックと同じに見えます。







つまり、エラーはエディターでオンザフライで強調表示され、 Alt+Enter



、一部のチェックでは修正（クイック修正）が利用可能です。







デフォルトでは、Clang-Tidyからのすべてのチェックは含まれていません。 オンになっているものとオフになっているものは、 コンフルエンスページで確認できます。 構成を変更するには、[設定]の[設定] | [設定]に移動する必要があります。 エディター| 検査| C / C ++ | 全般| Clang-Tidyを使用すると、コマンドライン形式でオプションを設定できます。





プロジェクトで既にClang-Tidyを使用している人にとっては、この行は非常に馴染みのあるものです。これは、コマンドラインから起動時にClang-Tidyに渡されるオン/オフチェックのリストです。 したがって、たとえば、C ++コアガイドラインのチェックのみを含めるには-*,cppcoreguidelines-*



を記述する必要があります。



自分の小切手はどうですか？ 同様に！ CLionで使用されるClang-Tidyに追加すると、CLionエディターに自動的に表示されます。 確かに、最後に正直に言うと、考慮すべき微妙な点がいくつかあります。



  • Clang-Tidyは、リポジトリから取得する必要があります。リポジトリには、変更を加えたバージョンがあり、CLionで正しく動作します。 LLVMでパッチが受け入れられ（現在レビュー中）、変更がマスターに反映されるとすぐに、メインのLLVMブランチを使用できるようになります。 リポジトリの検索場所とブログで読むことができるパッチの詳細をお読みください。
  • Clang-Tidyを変更するためのUIのリリースはリリースに到達することができなかったため、今のところ、組み込みのClang-Tidyのバイナリを置き換える必要があります。 しかし、これは今後のアップデートで修正される予定です。


新しいプロジェクトの作成ウィザードのC ++ 17



この変更の歴史は、CLion 2017.2にCMake 3.8を含めたことから始まりました。 つまり、このバージョンのCMakeでは、CMakeファイルで言語標準のバージョンを習慣的に示すことが可能になりましたset(CMAKE_CXX_STANDARD 17)



-この行は、新しいプロジェクトを作成するときにCLionによって生成されます。 そして今、C ++標準のドロップダウンリストに新しいプロジェクトを作成するためのウィザードに、追加項目-C ++ 17が登場しました。



MSVCのPCH



約束したとおり、使用した略語を解読します。 CLionのプリコンパイル済みヘッダーのサポート（つまり、そのようなファイルからの文字の正しい解像度とエディターでの正しい強調表示）は、最後のリリースで登場しました。 ただし、これはすべてのコンパイラに適用されるわけではなく、GCCおよびClangにのみ適用されます。 Microsoft Visual C ++コンパイラ（MSVC）は、前回のリリースからも実験モードのCLionでサポートされています。 しかし、PCHの使用が必要であり、普及しているのはMSVCの場合です。 そこで、このケースにこのサポートを追加しました。 トラッカーのリクエストにいくつかの微妙な制限が記述されています。



デバッガー



CLionでデバッグするための真のナポレオン計画がありますが、このリリースでは、バグを修正し、現在の機能を改良することが目標でした。



そのため、CLion 2017.1のリリース後、デバッグ中に逆アセンブラのコードに到達するために通常のステップインアクションを使用することはあまり便利ではないことが明らかになりました。 多くの場合、ユーザーは、デバッガーの通常の手順でソースコードがないすべてのフレームをCLionにスキップさせます。 したがって、Step Into（ F7



）はまさにそのように機能します。 逆アセンブラーのコードにアクセスするには、Force Step Into（ Shift+Alt+F7



）を使用する必要があります。



デバッガのもう1つの重要な変更点は、 デバッグ中に配列を表示するためのレジストリの追加オプションです。 実際、パフォーマンスを向上させるために、CLionは一度に配列全体をロードして表示するのではなく、特定の数の要素のみを表示します。 デフォルトでは、50アイテムです。 次の50を表示するには、[展開]をクリックします。 したがって、50要素ごとに。



これで、 cidr.debugger.value.maxChildren



オプションがレジストリで使用可能になりました。これはデフォルトで50に等しく、Expandのクリックごとにロードする必要がある要素の数を決定します。







値が大きすぎると、デバッグ中に値の計算が遅れる可能性があるため、このオプションには注意が必要です。



GmockおよびGoogle Testの構成



CLionは、Google TestおよびCatchの単体テストフレームワークをサポートしています。 サポートには、テスト結果（テストランナー）の出力を備えた組み込みの特別ウィンドウが含まれます。 テスト結果を取得するには、テストの実行時にGoogle TestまたはCatchの特別な構成を使用する必要があります。 Google Testの場合、gtestにリンクされたCMake内のすべてのターゲットに対して自動的に作成されます。 そして今、gmockにリンクしている人なら誰にでも。



取り消し不可能なCMakeナビゲーションおよびロードアクション



このリリースの焦点は、IDEのパフォーマンスです。 たとえば、コンパイラマクロ（コンパイラ定義済みマクロ）のより最適な処理により、インデックス作成時間を短縮しました。 この方向で他の改善がありました。



IDEのパフォーマンスに関する通常の作業に加えて、より多くの制御機能をユーザーに移行することにしました。 これを行うために、ユーザーは何らかの理由で非常に長い時間がかかるアクションを待つことができないように、非同期でキャンセルされたIDEでいくつかのアクションを実行し始めましたが、すばやく簡単にキャンセルできます。



最初のそのようなアクションは、CMakeの再起動とナビゲーションアクション（ サブクラスに移動、オーバーライドされた関数に移動 ）でした。 CMakeの場合、機能するCMakeコマンドをキャンセルするボタンが対応するツールウィンドウに追加されました。







コマンドが正常に完了したか、このウィンドウのCMakeコマンドの出力によってキャンセルされたかを簡単に理解するために、対応するテキストがログの最後に自動的に追加されます（ [Finished] 、 [Reloading cancelled] ）。



ナビゲーションアクションの[ サブクラスに移動]および[ オーバーライドされた関数に移動 ]については、非同期で動作するようになりました。これらをキャンセルするには、結果が表示されるポップアップウィンドウの外側の領域をクリックします。







パスインターフェイスの検索、バージョン管理システムのサポートの改善など



CLionはIntelliJプラットフォームに基づいた製品です。つまり、次の変更が適用されます。



  • パス検索ユーザーインターフェイスでの検索の改善
  • Git Revert、Reword、コミットメッセージの編集設定など、ネイティブVCSサポートの新機能
  • 強化された高解像度モニターのサポート（HiDPI）


これらの詳細については、HatelliのIntelliJ IDEA 2017.2のリリースに関する投稿をご覧ください。
