Windows Phoneマーケットプレイスのフィードバックシステム

良い一日。

誰もがマーケットプレイスでアプリケーションを公開する方法を知っています。 プロセス（かなり自動化された）は何度も説明されています。 しかし、アプリケーション開発のベクトルをさらに設定する方法はありますか？ このフィードバックをお寄せください。





フィードバック




アプリケーションが大規模な商業プロジェクトである場合は、おそらくQA（テスト）ステージがあったか、少なくともエンドデバイスで起動していました。 次に、GUIデザイナーが最後の仕上げを導入し、エンドユーザーがアプリケーションを使用できるようにします。



レビューを残すにはどうすればよいですか：

  • ブラウザ経由（http://www.windowsphone.com）
  • ズーン経由
  • Windows Phone 7ベースのスマートフォン経由




これらのすべての場合、ユーザーは自分のアカウントでログインし、次のようなレビューを残します。



  • レビュー日
  • 著者
  • 格付け
  • 見出し
  • 説明
  • 地域（マーケットプレイス内）




アプリケーションの各ユーザーは、アプリケーションの現在のバージョンに対して1つのレビューを残すことができますが、変更することはできます。 したがって、アプリケーションに関するフィードバックを取得できます。 また、出版のさまざまな場所でのレビューをさまざまな方法で更新できることにも注意してください。 多くの場合、ダッシュボード（create.msdn.com）にレビューがあるものの、まだサイトに表示されていない状況があります。 これは、ユーザーが電話から彼を離れたという事実によるものです。



評価-1〜10の数字。5つ星のシステムに変換されます。 したがって、この種類のマーケットプレイスで評価を確認できます。





この評価はアプリケーションページで生成され、「ratingLarge fourPtFive」という名前が付いていることがわかります。 平均評価（この場合は8.588）をアスタリスクに変換し、丸めることで判明しました。 評価ステップ：半星。 この瞬間は少し隠されていますが、私はこれがまさに起こっていると仮定します。



これでリコール構造がわかりました。 彼と仕事をする方法は？



フィードバックを処理する




最も重要な要素は、レビューの内容です。 ユーザーが間違いを犯した場合、またはレビューを書く際にロジックに導かれなかった場合を考慮しない（誤ったレビューは常に検出が困難です）場合、人が遭遇した問題の説明またはアプリケーションに関する熱狂的な喜びの応答のいずれかが含まれている可能性が高いです。 後者の数を増やしてみます。



レビューの説明




アプリケーションを改善するためにレビューを処理するとき、誰かに役立つかもしれないいくつかのヒントがありました：

  • すべてを心に近づけないでください。 問題に焦点を当てる-より便利です。
  • あなたとユーザーに合ったソリューションを考え出す
  • レビューで議論しないでください（非常に悪いオプション）
  • アプリケーションの明確なビジョンを持っている（開発計画ではなく、主なタスク）
  • 1つの悪いレビューの後にアプリケーションを変更しないでください




1つの怒っているレビュー（ほとんど不快）がアプリケーションページに表示されたとき、私は非常に心配していたことを覚えています。 多くの感情がありましたが、アプリケーションをより良くしたいという願望が広まりました。



著者への手紙




作成者の名前（ログイン）のみが利用可能になりましたが、作成者がZuneメッセージシステムを介して作成できるアプリケーションがあります。 これはWindows Phone 7 AppReviewsアプリです。 Tom Verhoeffによる投稿。 彼のブログ 。



主な機能：

  • アプリケーションがあるすべての地域のレビューを表示する
  • データをCSVにエクスポート
  • 統計を見る
  • レビューを見る
  • レビュアーへの書き込み機能




著者をクリックして、メッセージを入力します。 メッセージのテキストは制限されています。 簡潔に連絡先情報を残すことを忘れないでください。 次に、メッセージを送信します。 確認はありませんが、地域ru-RUについては「お住まいの地域では利用できません」と表示されますが、メッセージは送信されました。



その他のヒント




アプリケーションのヘルプまたはいずれかのページで、作成者に書き込み機能を追加します。 これにより、プロセスが簡素化され、可能な回答を待つ必要がなくなります。 ユーザーはエラーを報告できます。



技術的なエラーが含まれている場合、Stacktraceを失効日と比較すると便利です。 処理されていない場合は、表示されます。



この情報が開発者に役立つことを願っています。
