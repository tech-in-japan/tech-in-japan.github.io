ビジュアルおよびテキストプログラミング



プログラムの原動力は何ですか？ 何が有用な結果を生み出しますか？ もちろん、アルゴリズム。 アルゴリズムは、プログラムが作成された効果を作成します。 アルゴリズムは単独では機能しません。 データ構造と連動します。 しかし、アルゴリズムはプログラムの大部分を占めています。







歴史的に、プログラムのアルゴリズムはソーステキストの形式で記述されています。 テキストがアルゴリズムを表す最良の手段であることを疑う人はほとんどいません。 アルゴリズムは、CやJavaScriptなどのプログラミング言語の関数内でエンコードされます。 鳥瞰図からアルゴリズムを理解したい人のために、擬似コードが提供されています。 ただし、テキストには重大な問題があります。 実際には、人はソリッドテキストに対して最適化されていません。 人間はグラフィックスの知覚に最適化されています。 テキストは比較的新しい発明ですが、生物は何百万年もの間グラフィック情報を処理してきました。







これに基づいて、アルゴリズムをグラフィカルな形式で作成することは論理的です。 エンジニアを見てください。 彼らはどこでも図面を使用しています。 プログラマーが悪いのはなぜですか？ 彼らも、アルゴリズムの図面を作成できます。 ここで議論する人もいます：視覚プログラミングはおそらく非効率的です。 UMLは不便であり、フローチャートは簡単に混同されます。 従来の方法、つまりテキストでプログラムすることをお勧めします。 構造プログラミングでは、少なくとも構造があり、秩序と均一性を提供します。 それに、チャートを描くのは長くて難しいです。 描画よりも速く入力します。







では、プログラマーは一生テキストだけで作業する運命にあるのでしょうか？

おそらく、すべてがそれほど悪いわけではありません。 DRAGON、BPMN、LMLアクション図など、順序と構造も持つアルゴリズムを表現するための視覚言語があります。 ここでは、視覚アルゴリズム言語DRAGONを見てください。







DRAGONでのプログラミング方法



DRAGONは独立したプログラミング言語ではありません。 彼は たとえば、JavaScript、Python、C ++などのテキスト言語。 DRAGONは、テキスト言語とともに、DRAGON-JavaScript、DRAGON-Python、またはDRAGON-C ++というハイブリッド言語を形成します。







ハイブリッド言語でのプログラミングは次のとおりです。







  1. ドラゴンスキームを描画します。
  2. アイコンの中に、対応するプログラミング言語で小さなコードを配置します。
  3. トランスレータプログラムは、DRAGONスキームをソースコード付きのテキストファイルに変換します。
  4. このテキストファイルは、通常どおりプロジェクトに含まれています。

    ダイアグラムからのコード生成は現在、いくつかのエディターでサポートされています。 この記事の例は、DRAKONエディターで作成されています。


チャートからコードを生成する



図では、DRAGONが実行のフローを制御します。 したがって、アイコンのソースコードには、 if 、 else 、 switch 、 case 、 for 、 whileなどのキーワードを含めないでください。







アイコンの内部には、算術式、値の割り当て、関数呼び出し、比較など、単純な単一値のコードのみがあります。 しかし、分岐とループは、DRAGON言語の構築によって実装されます。







また、論理式を使用することはお勧めしません：and、or、not。 それらはまたドラゴンによって描かれています。







コード生成は次のとおりです。







  • 各ダイアグラムから関数が作成されます。
  • チャートの名前が関数の名前になります。
  • 関数パラメーターは、グラフ名の右側にある「公式パラメーター」アイコンから取得されます。
  • 関数の本体は、ダイアグラムの構造とアイコンの内容に基づいて生成されます。

グラフィック言語の利点



DRAGON言語には興味深い運命があります。 その基本原則は、ダイクストラ自身によって定められました。 DRACONは、ロシアの宇宙産業の腸で現在の形を獲得しました。 ドラゴン語の規則が偶然に生じたものではないことは注目に値します。 それらは最初にフォーカスグループで実行され、次に実宇宙プロジェクトで研ぎ澄まされました。

それでは、ドラゴンの強みは正確には何ですか？







まず、DRAGONはグラフィック言語です。 また、グラフィック言語には、テキストよりも基本的な利点があります。







第一に、思考は線にランダムに散らばるのではなく、正方形またはアイコンの中に囲まれています。 1つの考え-1つの小さな正方形。 さまざまな考えが1つの混乱に結びつくことはありません。







第二に、アルゴリズムを通るパスを指（または注視）で追跡できます。 他を探す必要がない場合。 行に従って、目的のボックスに入ります。 質問への答えを探すためにソースコードに目を通す必要はありません。次に何が起こったのでしょうか。







チャートには、魔法のような特性もあります。 人が図を見ると、突然、さらに理解が深まります。 以前は見えなかった接続が明らかになります。 テキストではめったに起こりません。







特別な人間工学的ルール



しかし、ドラゴンは単なる図ではなく、慎重に考え抜かれた図です。 ドラゴンダイアグラムは、従来のダイアグラムよりも読みやすくなっています。 これは、特別な人間工学的手法により保証されています。 ここにそれらのいくつかがあります。







  • ラインを越えることは禁止されています。 一般的に。 交差点により、視覚アナライザーは線が接触していると疑ってしまいます。 これらの疑いは、追加のメンタルワークを作成します。 不要な作業は破棄する必要があります。
  • 先頭にはアルゴリズムの名前が含まれ、常に図の左上隅にあります。 したがって、始まりを探す必要さえありません。 それは通常の場所です。
  • チャートには一方の端しかありません。 道路で何が起こっても（例外を除く）、私たちは常に終わりを迎えます。
  • 直線のみが許可されます。 曲線や曲がりはなく、不必要なねじれもありません。
  • 厳密に垂直線と厳密に水平線のみが許可されます。 傾斜線は禁止されています。 数学愛好家の説明：ドラゴンダイアグラムは平らな長方形のグラフ（マンハッタングラフ）です。 人間の視覚装置は、まっすぐな直交線で接続されたオブジェクトを即座にキャプチャします。 しかし、「アメリカンドリームカーブ」がどこに導くかを追跡するには、読者のさらなる集中が必要です。
  • ドラゴンスキームは上から下に実行されます。 このルールは、次のアイコンを探すためにチャートの熱狂的なアイスキャンの必要性を避けます。 次のアイコンは常に下部にあります。 アイコンの入り口は上にあり、出口は下にあります。 次のアイコンの場所がわかっているため、矢印は不要です。 シンプルな線で十分です。 各アイコンの横にある矢印は視覚的なノイズです。 矢印からアイコンを接続するタスクを削除したら、特別なミッションをそれらに割り当てることができます。 ドラゴンでは、矢印はサイクルを意味します。
  • 分岐は右側のみで発生します。 これは、予測可能性と均一性の点で非常に役立ちます。
  • 同じ垂直線にあるアイコンは同じ幅にする必要があります。 これにより、アイコンが単一の全体に属しているという感覚が得られます。 全員が同じ幅を持ち、スタートアップアイコンがない場合、目はダイアグラム上を簡単かつ自由に滑る。


上記のルールには非常に実用的な価値があります。 一方では、彼らはアーティストの狂気を抑えるように設計されています。 それらと混同するチャートを作成することはより困難です。 一方、図表に認識をもたらします。 チャートは作成者だけでなく理解可能になります。

シルエット



シルエットはDRAGONの本物のダイヤモンドです。 シルエットを使用すると、ダイアグラムを論理的な部分に分割できます。 プログラミングでは、通常、ルーチンを使用した分解がこれに使用されます。 サブプログラムは効果的な方法です。 ただし、サブルーチンを視覚的にメインプログラムの近くに配置したり、パラメーターを渡したり値を返したりすることに煩わされたくない場合があります。 シルエットはこれらの目的に最適です。 シルエットの別のアプリケーションは、有限状態マシンです。 しかし、これについては別の場所で説明します。

アルゴリズムは平面上で分解できないため、線の交点がないことが起こります。 この場合、状況に応じて、サブルーチンまたはシルエットを使用した分解が使用されます。







シルエットは、1つのブロックに接続されたいくつかの小さな図で構成されています。 これらの小さなチャートは、シルエットブランチと呼ばれます。 各ブランチの上部には「ブランチキャップ」アイコンがあり、下部には「アドレス」アイコンがあります。 このブランチの名前はブランチのヘッダーに配置され、次のブランチの名前はアドレスに示されます。 ブランチの名前は、図の上部にある1本の水平線上にあります。 このため、ブランチのみを実行することでアルゴリズムの本質を把握できます。 シルエットは3つの王室の質問に答えます。







  1. 問題の名前は何ですか？
  2. いくつの部品で構成されていますか？
  3. これらの部品は何と呼ばれていますか？

DRAGON言語でプログラミングするとき、疑問が生じます。ダイアグラムはどのくらいのサイズにすべきですか？ 答えは、少ないほど良いということです。 ビジュアルシーン内のオブジェクトが少ないほど、鮮明になります。 テキストプログラミングでは、このようなガイドラインがあります。機能全体が画面に完全に収まるのが良いでしょう。 DRAGONスキームについても同様のアドバイスを提供できます。 巨大なチャートを避けてください。 アルゴリズム全体が完全に表示されると、理解しやすくなります。

DRAGONでのプログラミングには、大きなモニターを使用することをお勧めします。 少なくとも1080ドットの高さ。 その場合、DRAGONスキームを人為的に短縮する必要はありません。







DRAGON-outline silhouetteは、画面の高さで配置する必要がありますが、必ずしも幅で配置する必要はありません。 シルエット図は非常に広く、2000ピクセルよりもはるかに広い場合があります。 これは正常です。 シルエットのすべての枝を同時に見る必要はありません。 主なことは、現在作業しているブランチが画面に完全に表示されることです。







ドラゴンのプログラミングに対する批判



DRAGONでのプログラミングに対する批判の主な方向性を検討し、それらに答えを出そうとします。







  • 「ドラゴンダイアグラムは、テキストプログラムよりも多くの画面スペースを占有します。」それは本当です。 しかし、ドラゴンの仕事は、複雑さをそのまま示すことであることに留意する必要があります。 プログラムの読者は、頭の中で複雑な構造を解凍しないでください。 それらは彼に明示的に示されなければなりません。
  • 「シンプルなアルゴリズムは、テキスト形式でより良く見えます。」 おそらく。 Hello worldはどの言語でもエレガントに見えます。 しかし、実際には、すべてが単純なわけではありません。 少なくとも1つのifが表示され、別のifにネストされるとすぐに、ドラゴンが勝ちます。
  • 「ドラゴンには例外を表示する手段がありません。」そのような問題があります。 DRAGON言語に最近例外が追加されましたが、すべての実装が例外をサポートしているわけではありません。 実装が到着するまで、適切なプログラミング言語でtry / catchブロックを作成できます。
  • 「DRAGONダイアグラムの描画には時間がかかります。」専門のエディターでは、DRAGONダイアグラムの描画は、たとえばVisioの場合よりもはるかに簡単です。 そして、それらのいくつかでは、描画は書くのとほとんど同じくらい簡単になりました。
  • 「diffとmergeのツールはありません。」これは残念ながら真実です。 バージョン管理システムを使用する場合、生成されたソースファイルを比較する必要があります。
  • 「DRAGON回路用のデバッグツールはありません。」それは本当です。 ただし、生成されたコードをデバッグすることはできます。 このコードが図のどこに属しているかを示すラベルがあります。

ドラゴンツール



DRAGON言語の最初の実装は、GRAPHITE-FLOXシステムでした（図17）。 GRAPHITE-FLOXは1986〜1996年に作成されました。 FSUE SPC APのスペシャリストの名前は V.D.の指示の下でのピリュギナ パロンジャノバ。 この環境は、ロケットや宇宙船の制御システムの設計を目的としていました。







GRAFIT-FLOXは閉じた開発であるため、比較的少ないことが知られています。 GRAPHITE-FLOXを使用して作成された宇宙船のリストは、 ここで表示できます 。







90年代初期に、別のDRAGONエディターが作成されました。 開発はM.V. Institute of Applied Mathematicsで行われました L.K.が率いるケルディシュ アイシモント。 Eisymont Editor（図18）はダウンロードして実行できますが、サポートされなくなりました。 エディターはMS DOSで作成されているため、DOSBoxは最新のコンピューターで実行する必要があります。







2008年、Gennady TyshovによるDragon IPのエディターは光を見ました（図19）。 IP Dragonは積極的にサポートおよび開発されています。 Dragon ISは、ダイアグラムからプログラムコードの生成を実装します。 Dragon IPの興味深い機能の1つは、プログラミング言語のコードと自然言語の説明を1つのアイコンで配置できることです。 Dragon IPの絶対的な利点は、いわゆる「アイコンの計算」です。 アイコンの計算は、ユーザーが図を描くのに役立ち、図がDRAGON言語の規則に違反しないようにする編集方法です。 Dragon ISの欠点のうち、非標準のユーザーインターフェースとコード生成時の不便な点に注意することができます。 IP Dragonは商用製品です。







DRAKON Editorは、もう1つの最新のDRAGONエディターです（図20）。 DRAKON Editorは、Stepan Mitkinが率いる愛好家グループによって開発されました。 DRAKON Editorはアイコン計算をサポートしていません。 これは、ベクターグラフィックエディターのように、DRAGON回路がプリミティブから手動で組み立てられることを意味します。 ただし、DRAKON Editorのユーザーインターフェイスは可能な限りシンプルです。 Dragon ISよりも使い慣れたスキームに従って構築されています。 DRAKON Editor環境の主な利点は、プログラミングとコード生成の利便性です。 DRAKON Editorは、C、C ++、C＃、Java、Processing、JavaScript、Lua、Erlang、Python、Tcl、Verilog、AutoHotkey、D、Goなどのいくつかのプログラミング言語をサポートしています。 一部の言語では、有限状態マシンを生成できます。 noolsエキスパートシステムのルールがサポートされています。 UTOPIST E. Tyugu言語のサブセットを実装しました。 DRAKON Editorはオープンソースです。







DRAGON言語の興味深いアプリケーションは、Oleg GaripovのIntegrator CodeViewプロジェクトで考案されました。 CodeViewを使用すると、既存のコードを相互接続されたDRAGONスキームのセットとして視覚化できます。Integrator CodeViewの特性は、視覚化されるのは個々のメソッドではなく、コールグラフ、スタックなどを含むプロジェクト全体であるということです。IntegratorCodeViewは、アルゴリズムだけでなくデータも明確に表示するという点でもユニークです。Integratorシステムのデータ視覚化エンジンは、DRAGONと連携して機能します。







DRAKON Editor Webは、DRAGON言語に基づいた商用クラウドソリューションです。DRAKON Editor Webは、技術的なタスク、ビジネス手順、チェックリスト向けに設計されています。DRAKON Editor Webは、DRAKON Editorとはまったく関係がなく、ダイアグラムからのコード生成をサポートしていません。DRAKON Editor Webの利点の中でも、便利なエディター、コラボレーション、モバイルデバイスのサポートに注目すべきです。
