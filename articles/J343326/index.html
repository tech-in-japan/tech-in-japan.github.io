LizardFS-クラスターファイルシステムの簡単な概要

興味深いクラスターファイルシステム。 それほど大きくないプロジェクトの1つに実装しましたが、一般的なGlusterFSまたはNFSソリューションよりもうまく機能します。 クライアント側で接続するときにFUSEを使用します。 これは他のクラスターFSよりもうまくいくとは思いませんでしたが、実際にはすべてが非常にポジティブであることが判明しました。

3か月後、マスターサーバーの再起動さえ行われました。FS自体とは無関係の理由で、利用できる最高のクラスターFSであり、多少複雑です。 バグではありません。データストレージとして機能するだけでなく、ほぼ毎分データが常に更新され、多数のファイルが上書きされるなど、問題なく平均データ量を保持します（この場合、1TBおよび大小のファイル）。



一般に、LizardFSまたはMooseFS（商用）は、少なくともペタバイトの情報用に設計されています。



自宅で使用する人には微妙な違いがあります。 このファイルシステムは、ファイルの行ごとの書き込みを目的としていません。つまり、ループから行をアンロードしてPHPからディスクに直接ファイルを書き込むと、各変更を常に同期しようとするため、明らかな理由でFSの速度が大幅に低下し始めます。



したがって、行ごとの記録を作成するには、単にtmpfsフォルダーをどこかにマウントし、このフォルダーでファイルを1行ずつ生成し、そこから完成したファイルのmvをクラスターFSに作成するか、インタープリターで直接RAMにデータの配列を作成し、一度にファイルにマージします。 つまり、アトミック操作、ファイルのコピー、ファイルの移動、ファイルの読み取りに最適ですが、最も重要なのはファイル全体の完成であり、問​​題はありません。 forおよびループの助けを借りてのみ、ファイルを1行ずつファイルに書き込む必要はありません。



フォルダ内のレプリカの数を設定する機能は素晴らしいです。



この記事には、cronにインストールするのに十分な単純な監視スクリプトマスターとシャドウサーバーウィザードを追加します。 



#!/bin/bash #Variables SRV="172.24.1.1" #Every 15 minutes repeat alert if have troubles test -f /tmp/$SRV.lizardfs && find /tmp/$SRV.lizardfs -mmin +15 -delete #Check if no have previous alert flag file if [ ! -e /tmp/$SRV.lizardfs ]; then #Check command CHECK_LIZ=`lizardfs-probe metadataserver-status $SRV 9421 | grep "running\|connected" | awk '{print $3}' | sed '/^$/d' | tr -d ' '` #Send Alert if have troubles if [[ "$CHECK_LIZ" != "running" && "$CHECK_LIZ" != "connected" ]] ; then echo "$CHECK_LIZ" > /tmp/$SRV.lizardfs #Send Alert /home/scripts/telegram/telegram.sh --service "$SRV LizardFS" --event "Status: $CHECK_LIZ" ; /home/scripts/sms/msms.sh --service "$SRV LizardFS" --event "Status: $CHECK_LIZ" ; fi fi

LizardFSの利点：



小さなファイルを読み取るモードでの非常に高速な作業。 GlusterFSを使用した多くの人は、これらのサイトを返すためにクラスターFSを使用すると、操作が非常に遅くなるという問題に直面しました。



非常に良いレベルでの大きなファイルの読み取り速度。



大きなファイルの書き込み速度はネイティブファイルシステムと大差ありません。特に、クラスターの下部にSSDディスクがあり、クラスター内のサーバー間に1〜10ギガビットの接続がある場合はそうです。



Chown / ls -la操作はネイティブFSの場合よりも遅くなりますが、それほど遅くはありません

GlusterFSの場合のように。



再帰的な削除操作-非常に高速。



mfsmountユーティリティの非常に柔軟な接続オプション。



メタデータの複製は簡単かつ簡単に設定でき、複数のシャドウシャドウマスターサーバーとメタロガーサーバーを同時に使用できます。



メタデータはRAMに保存されます。



LizardFSがローカルサーバーからデータを取得することを好むのは素晴らしいことです。 2つのレプリカの例を使用して読み取りを高速化すると、LizardFSクライアントはローカルサーバーからファイルを自動的に取得し、2番目のサーバーからネットワークを介してファイルを取得しません。



サブフォルダーに目標（レプリカの数）を設定する機能、つまりLizardFS / var / lizardfs / importantに3つのレプリカを作成すること、および/ var / lizardfs / not_importantにレプリカなしで1つのFS内で設定することができます。 それはすべて、チャンクサーバーの数と必要なタスクの実行に依存し、チャンクサーバーごとに複数の個別のディスクを使用することもできます。



EC-Erasure Codingなど、さまざまなデータ複製モードをサポートしています。

このモードでの読み取り速度はそれほど大きくありません。



ファイルシステム自体をマウントするクライアントを除き、すべてがLXCコンテナーで機能します。 ホストマシンにマウントし、マウントポイントをLXCコンテナーに転送することで解決します。

練習できます。



新しいノードを追加するときの自動リバランス。

削除されたノードや不要なノードの非常に簡単な削除。

小さなファイルの大きなストリームの書き込み速度は非常に遅いです。 Rsyncはそれをよく示しています。

制限を設定します。



/etc/security/limits.confに以下を配置する必要があります。 



 * hard nofile 20000 * soft nofile 20000
