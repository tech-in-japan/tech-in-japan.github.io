NetApp 7-Mode：文書化されていない機能またはSnapVaultからDRを作成

NetAppバックアップパラダイムに関する記事の続きで、 FASシリーズの「アーカイブコピー」を「バックアップ」に変換する文書化されていない可能性についてお話したいと思います。 NetApp FAS ストレージシステムの特徴は、すべてが統合されていることです。 統合とは、1つのデバイスがブロックプロトコルとファイルプロトコルの両方でホストへのアクセスを提供するだけでなく、その使用方法も提供することです。 FASシステムは、仮想化、データコンプライアンス、アーカイブコピーの保存、災害復旧ソリューションの構築などに使用されます。 同じストレージシステムで一度に多くの機能を実行できます。 したがって、機能ごとに1つの「専用」デバイスを保持する必要はなく、緊急に「スペア」 ストレージシステムが必要な場合は、データアーカイブ用のストレージシステムなどから常に「再プロファイル」できます。 この汎用性のおかげで、オペレーティングシステム、コマンドライン、およびすべての構成原則はすべてのFASシステムで同じであるため、これらの各タスクを再トレーニングする必要はありません。



この記事では、組み込みソリューション「NetApp上のデータのアーカイブ」をソリューション「災害復旧」に変換する方法を説明します。



ビジネスの観点から、災害復旧とアーカイブは次の点で異なります。

  • アーカイブ（SnapVault）-このソリューションは、データを長期間保存し、変更からデータを保護し、その後コピーされた場所（または別の場所）に復元することを目的としています。
  • 災害復旧（SnapMirror）-災害時にバックアップサイトに切り替える（およびそれに応じてデータを変更する）ためのバックアップサイトでのデータのストレージ。




例で説明しましょう：SnapMirrorレプリケーションが構成された少なくとも2つのストレージシステムがある場合、そのようなスキームでは、そのうちの1つがソース（プライマリ）の役割を果たし、レシーバーの2番目の役割（セカンダリ）を果たします。 事故が発生した場合、レプリケーションが（リンクブレークだけでなくbreakコマンドで）壊れると、セカンダリシステムは複製されたミラーを読み取り専用モードから読み取り/書き込みモードに切り替えます。 つまり 「事故が発生した場合の予備サイトへの切り替え」（災害復旧）ソリューションを作成するためのツールです。 論理的には、両方のシステムに同じパフォーマンスをプラスまたはマイナスして、あるサイトから別のサイトへのすべての交換ノードに適切なレベルのパフォーマンスを提供することができます。







SnapVaultはバックアップ（セカンダリ）システムにアーカイブするように設計されていますが、後ですべてのデータをプライマリシステムまたは3番目のシステムに復元します。 アーカイブタスクでは、データを常に変更されていない状態で保存することが非常に重要です。 この場合、すべてのアーカイブが保存されているセカンダリシステムは、どのモデルでもかまいません。 低速で安価な大型ドライブを備えた最も安いNetApp FASモデルを使用するのは理にかなっています。 たとえば、 FAS2554またはFAS2520です。



SnapMirrorとSnapVaultの仕組みの詳細については、関連する記事を読んだり、技術的な観点からこれら2つのソリューションを比較したりできます。



QSM SnapMirrorはSnapVaultとほぼ同じように機能し、同じアルゴリズムを使用し、 QSMを除く内部複製デバイスSnapMirrorにはブロック複製VSMも含まれます。 ブロックレプリケーションは同期または非同期にできますが、SnapVaultとQSM SnapMirrorはスケジュールされたデータのみを非同期にレプリケートします。 ただし、そのために、リモートシステムに到達するとスナップショットの形式で転送されたSnapVaultアーカイブを公開できます（ WAFL ファイルフォールディングは同様の方法で機能します）。 したがって、SnapVaultはさまざまな種類のアーカイブを提供し、変更されたデータのみをスナップショットの形式でリモートシステムに転送し、重複排除された形式で保存します。 つまり スナップショットは増分バックアップに類似しています。用語を標準のバックアップ用語に近づけると、NetAppスナップショットは逆増分バックアップに類似しています。 このようなバックアップ（スナップショット）は「フルバックアップ」で収集する必要がなく、システムのロードに時間を費やす必要はありませんが、アーカイブは通常の増分バックアップなどの場所を占有します。 他のメーカーのCOWテクノロジーを使用したスナップショットの「クラシックな実装」は、NetAppのスナップショットテクノロジーとは異なるという事実に注目したいと思います。 そのため、 COWテクノロジーで動作するスナップショットは、その数が増えるとパフォーマンスが低下します。 詳細



SnapVaultの欠点は、関係が壊れた場合に、セカンダリシステムのデータが読み取り/書き込みモードにならないことです。これらはアーカイブですよね？



これに関連して、SnapVaultをアーカイブするためのライセンスは、SnapMirrorよりもはるかに安価です。 私はここで価格について話さないつもりであるとすぐに言わなければなりません、これらの質問は、許可された人、あなたのインテグレーター、ディストリビューター、および代表者にお願いします。 私はエンジニアであり、価格設定ではなくテクノロジーを使用しています。

SnapVaultの場合とSnapMirrorの場合の両方で、ライセンスは「テラバイト単位」ではなく、「コントローラー単位」であることに注意してください。 ソースと受信者の両方でライセンスが必要です。 例を挙げましょう.2つのHA （フォールトトレラントペアの2つのコントローラー）システムがあり、1つはメインシステムに、もう1つはバックアップサイトに、4つのコントローラーの量で、SnapMirrorまたはSnapVaultであるかどうかのレプリケーションのために、4つの対応するライセンスを購入する必要があります。



SnapVaultを使用しているときに、アーカイブを読み取り/書き込みモードに移行する必要が非常に簡単で、緊急、緊急、または緊急の場合、セカンダリシステムに基づいてDRサイトを上げることがあります。 つまり、メインシステムよりも脆弱であっても、アーカイブシステムをDRシステムに変換します。 「内部」のQSM SnapMirrorは同じSnapVaultであるため、文書化されていないサービスコマンドを使用してこれを行うことができます。



あなたはあなた自身の危険とリスクですべての操作を実行することを思い出させてください、著者は義務と保証を負いません。



SnapMirrorライセンスを追加すると、代表オフィスから試用ライセンスをリクエストできます。通常、これは非常に高速なプロセスです。 



netapp-sec1-7M> license add SNAPMIRRORCODE


SnapMirrorがオフになっていることを確認します。 



 netapp-sec1-7M> snapmirror off


FlexCloneライセンスをお持ちの場合、またはトライアルライセンスを注文した場合は、バックアップアーカイブに触れたり破損したりすることなく、データの完全なシンコピーを作成して使用するために使用することをお勧めします。 ちなみに、たとえばIBM Storwizeブロックのクローン作成とは異なり、NetAppのシンクローン作成を思い出してください。このようなクローンがいくつあってもパフォーマンスは低下しません。 ライセンスがなく、バックアップを変換する場合は、この手順をスキップして次に進みます。 



 netapp-sec1-7M> vol clone create new_infra -b backup


クローンがコピーされるときと「SnapVault関係」



次に、変換のためのコマンドの魔法の組み合わせを実行します。 



 netapp-sec1-7M> options snapvault.enable off netapp-sec1-7M> priv set diag; snapmirror convert /vol/new_infra/nfs; priv set netapp-sec1-7M> options snapvault.enable on


SnapVaultはQtree *でのみ機能するため、Qtreeを変換する必要があります。 Qtree自体がSnapMirrorに変換されるだけでなく、複製の「関係」も変換されます。 同様に、「他の方法」をQSMからSnapVaultに変換できます。



SnapMirrorをオンにする 

 netapp-sec1-7M> snapmirror on


SnapMirrorが有効になっていないと、関係を切り離すことはできません（現在はSnapMirror関係が変換されています）

関係を停止します。 そして最後に、ボリュームをRW状態に変換するために（必要な場合）引き裂かれます。 



 netapp-sec1-7M> snapmirror quiesce /vol/new_infra/nfs netapp-sec1-7M> snapmirror break /vol/new_infra/nfs


必要に応じて、SnapMirrorライセンスを削除します。 



 netapp-sec1-7M> license delete SNAPMIRRORCODE


その結果、現在のNetAppファイルシステムで、それに複製されて記録に使用できる最新のデータを確認できることに注目したいと思います。 ただし、いくつかの方法で古いスナップショットを復元できます。



たとえば、 ストレージコマンドラインからNDMP ndmpcopyコマンドを使用してスナップショットペンでコピーするか、非表示の「snapshoot」フォルダーに移動して、必要なすべてのデータを標準的な方法でホスト経由でコピーします。



別のSnapRestoreライセンスを使用して、現在のファイルシステムを必要なスナップショットに即座にロールバックすることもできます。



3番目のオプションは、アーカイブ（qtreeを含むボリューム）のクローンを作成する段階で、NetApp ファイルシステム上の実際のデータの状態ではなく、保存されたスナップショットの1つをクローンすることができます 。





* Qtree- SnapVaultおよびSnapMirror QSMが ボリュームレベルでデータを複製および復元できるようになったため、DataONTAp Cluster-Mode（Clustered ONTAP）を使用したNetApp FASシステムでのレプリケーションアプリケーションは不要になりました。
