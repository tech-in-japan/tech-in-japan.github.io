Oracleのストーリー-商業的に成功した最初のリレーショナルデータベースエンジン





70年代半ばまで、データベース内の情報は、デスクトップオペレーティングシステムでまだ使用されている古い階層、つまり「ツリーのような」原則に従って配布されていました。



リレーショナルDBMSの最初のプロトタイプは、20世紀の70年代にすでに存在していました。 しかし、そのようなシステムの効果的な実装を達成する可能性を信じる人はほとんどいませんでした。 ただし、1980年代の終わりまでに、リレーショナルシステムがグローバルなDBMS市場を支配していました。



この点で、多くの企業は、広告を目的としたDBMSを「リレーショナル」として位置付け始めました。 しかし、彼らは常にこれには十分な理由がありました。 したがって、リレーショナルデータモデルの作成者であるEdgar Coddは、1985年にすべてのRDBMSが満たす必要のある有名な「12 Coddルール」を公開しました。



リレーショナルデータベースの最初のプロトタイプの1つはSystem Rでした。これは、1976年に登場したIBMプロジェクトです。 彼は、将来のOracleの創設者に独自のリレーショナルデータベース管理システムを作成するよう促しました。 、しかし彼自身は商業的に成功しませんでした。



Oracleのクリエーターの主任はLarry Ellisonであり、Oracleの前にBob MinerとEd OatesとともにCIAのプロジェクトに取り組んでいました。 多くの情報筋は 、彼にコード名「Oracle」が与えられたと言っています。 1977年、若いプログラマーのラリーエリソンがイェールを中退して、自分のビジネスを始めました。 ラリー・エリソンの処分で、それはたった1200ドルでした。 彼は上記の2人の友人に投資するように説得しましたが、これからの初期資本はたった500ドルで成長しました。







1977年6月16日に、ソフトウェア開発研究所は、カリフォルニア（米国）でEd Oates、Bob Miner、およびLarry Ellisonによって設立され、間もなくRelational Software Incに改名されました。 若いプログラマーは、リレーショナル代数の原理に基づいて構築されたデータベース管理システム（DBMS）の開発を始めました。



Oracle 2



Oracle DBMSの最初の商用バージョンはOracle 2と呼ばれていました。この動きは、システムが信頼性があり、時の試練に合格したことを顧客に明らかにすることでした。



70年代後半、Oracle DBMSの主な競争上の利点は、膨大な量の情報を高速に処理できることでした。これは、すべての専門家が指摘しています。 強力なスーパーコンピューターであるメインフレームを必要としたSystem Rとは異なり、Oracle 2はより「ミニチュア」なマシンでの情報の処理に対処しました。 これらの利点やその他の利点により、80年代前半にDBMSが急速に普及し始めたという事実に至りました。

Ellison氏と同僚は、IBM System R DBMSとの互換性を実装することは困難であると考え、IBMがソースコードを開示することをためらうことが重要な問題になりました。 2つのシステム間の互換性の結果、達成することはできませんでした。




ラリー・エリソン-Oracleファウンダー



Oracleは、歴史的に、クライアント/サーバーアーキテクチャの最初で最も先進的な実装の1つになりました。 Oracle開発者にとって、移植性とスケーラビリティは常に最優先事項でした。 これは、DBMS市場で会社の成功を達成する上で重要な役割を果たしました。



Oracle 2は、RSX-11オペレーティング環境でDigital Equipment PDP-11ミニコンピューターを実行しました。 Oracleの大部分はPDP-11アセンブラで記述され、個々のコンポーネントは当時は新しいC言語で記述されていましたが、当時はすでにシステムは移植可能で、他のPDP-11オペレーティング環境（IAS、RSTS、UNIX）で動作していました。 次に、Oracleを新しいVMSに移行することにしました。 その結果、オラクルは急速に成長しているVAX市場のエンタープライズ情報システムの広大なニッチを引き継ぎました。



システムのもう1つの重要な機能は、新しいSQLクエリ言語の機能（サブクエリ、結合操作など）の完全な実装です。 これにより、SQLプログラマーの生産性は何度も向上しました。

標準SQL（IBM）は、CONNECT BY操作で拡張されました。これにより、ツリー構造を処理できるようになり、SQLシステムに固有のものになります。
もちろん、DBMSの作業には長い時間がかかりました。 たとえば、Oracle 2では、トランザクションはサポートされていませんでした。データベースの更新プロセス中に障害が発生した場合、データベースの以前の状態を復元することはほとんど不可能でした。 そのため、情報の損失を避けるために、ユーザーはしばしばデータベースのバックアップを余儀なくされました。



1982年10月29日に、同社はOracle Systemsと改名されました。



Oracle 3および4



Oracle 3は1983年に市場に参入しました。Cで完全に書き直されました。これにより、Oracleの幅広いプラットフォームへの移植性の問題を解決するのに大いに役立ちました（少なくとも20ありました）。さらに、アトミックトランザクション実行が実装されました。それぞれまったく実行されなかったか、トランザクションがすべてのデータベース変更で正常に完了したか、またはそれによって行われたすべての変更をロールバックしました。







Oracle 4のリリースにより、システムはVMおよびMVSを実行する大型コンピューターと、640キロバイトのRAMを搭載したパーソナルコンピューターに移植されました。



データベースアクセス制御モデルも実装されました。これにより、クエリの結果がクエリの開始時のデータベースの状態と矛盾しないことが保証されました。 これにより、読み取りと書き込みのプロセス間のよく知られている矛盾が解消されました。



Oracle 5



1985年、Oracleはバージョン5.0をリリースしました。これにより、クライアント/サーバーアーキテクチャが初めて導入されました。 さらに、同社はクライアントとデータベース間または2つのデータベース間の透過的な接続を提供するネットワーク製品であるSQL * Netをリリースしました。



バージョン5.1では、分散クエリが最初に実装されました。これにより、異なるノードに物理的に配置されたデータにアクセスできるようになりました。 相互作用する複数のサーバーは、物理的に間隔を空けて配置された多くのデータベースのユーザーにとって、単一の論理データベースの錯覚を作り出す可能性があります。



1986年3月12日に、Oracle Corporationの株式の最初の公募が行われました。 高い成長率により、オラクルは1986年に5500万ドルの利益で公開IPOを行い、わずか3年で10倍の5億8400万ドルの利益を得ました。

バージョン6の開発者は、リアルタイムトランザクション処理向けの大規模な情報システムを構築するためのツールを作成しようとしました。



シーケンスジェネレーターと書き込みレベルのロックが導入されました。 同時に、OracleはOS / 2、Xenix、Banyan Vines、およびMacintosh用の最初のマルチユーザーネットワークデータベースサーバーになりました。



バージョン6では、基本的に新しい機能が組み込まれ、後で完全に実装されました。



  • SQLクエリは、PL / SQL手続き言語の構造と組み合わせて使用​​し、匿名プロシージャとしてサーバーに送信して実行できます。
  • PL / SQLは、SQL * Formsでアプリケーションプログラミングツールとして使用されました。
  • 参照整合性制限の宣言的な定義が、構文レベルでデータベーススキーマの説明に導入されました（ANSI / ISO標準に準拠）。

その結果、Oracleは倒産に近づき、InformixやSybaseなどの競合他社が徐々に市場シェアを拡大​​し始めました。

Oracle 7



Oracle 7 DBMSバージョンには、I / Oの効率の一般的な向上、中央処理装置の使用、メモリの操作に加えて、多くの革新的なアーキテクチャソリューションがありました。



  • サーバー上の共有SQLキャッシュ：サーバーは、以前に分析およびコンパイルされ、現在キャッシュ内にあるクライアントから送信されたSQLクエリを認識します。これにより、分析、最適化、翻訳時間、およびストレージに必要なメモリが節約されますSQLクエリ
  • クライアントごとに個別のプロセスではなく共有サーバープロセスプールを使用すると、メモリを大幅に節約できます。


データベース管理の分野でも、いくつかの革新が導入されました：ミラートランザクションログ、データログの動的作成、テーブル、インデックスおよびその他の物理オブジェクトの使用に関する統計情報を収集できるANALYZEコマンド、システムリソース（CPU、入力/出力など）の使用に関するユーザープロファイル。さらに。



バージョン7では、ANSI / ISO標準に従って、参照整合性の宣言的制限が完全に実装されました。 これらの制限（主キーおよび外部キー）のフレームワーク内で、ユーザーは特定の主キーに関連付けられたエントリのカスケード削除を指定できます。 PL / SQLプロシージャは、データベーススキーマレベル（ストアドプロシージャ）で記述され、アプリケーション、他のプロシージャ、およびトリガーによって呼び出されます。



もう1つの重要な革新は、データベーストリガーです。

トリガーはペア（イベント+アクション）であり、イベントはテーブルエントリの削除/エントリ/更新であり、アクション（トリガー本体）はイベント発生時に実行されるPL / SQLプロシージャです。



トリガーは、操作レベル（DELETE、INSERT、UPDATE）または個々の行のレベル（FOR-EACH-ROWトリガー。さらに、古い行と新しい行の値を操作できます）で定義できます。 トリガーを使用すると、整合性、アクセス権、値の出力などを制御するための複雑なルールを実装できます。
役割の導入により、セキュリティと整合性の管理が大幅に簡素化されました。
ロールは、データベースオブジェクト（INSERT、UPDATE、SELECTなど）へのアクセス権とシステム権限（CREATE TABLE、ALTER SYSTEMなど）のセットです。 ロールを定義すると、データベース管理者は単一のコマンドを使用して、ユーザー権限を付与してアプリケーションを操作できます。
