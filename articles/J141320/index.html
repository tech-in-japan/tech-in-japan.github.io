GNU / linuxのリモートルートルートの置換



ルートファイルシステムの交換に直面することがあります。 ブートディスクを用意してサーバーにアクセスすることは難しくありません。 ただし、再起動せずに、sshを介してリモートでルートFSを交換した経験を共有したいと思います。



マウントされたFSを交換する理由はさまざまです。

  • LVMへの転送/
  • 翻訳/ mdraid
  • ファイルシステムタイプの置換（ext3 => btrfs）
  • ファイルサイズの縮小（resize2fs）




例として、ルートFSを/ dev / sda2からLVMに移動します

説明は、gentoo GNU / linuxの例で実行されます。

この方法は、Debian GNU / linuxでもテストされました。



この操作中にデータを失ったことがないという事実にもかかわらず、 誰もバックアップをキャンセルしなかったことをすぐに言わなければなりません！



与えられた



データセンターにリモートサーバーがあります。

私の場合は、KVM仮想マシンが実行されているホストシステムになります。



ルートFSがあるハードドライブ/ dev / sdaは、3つのmsdosパーティションに分割されています。

  • sda1 200Mb /ブート
  • sda2 2GB /
  • sda3 197GB LVM


タスク



  1. ルートFSをLVMの/ dev / sda2パーティションから「sys」グループの「root」論理ボリューム（/ dev / mapper / sys-root）に移動します
  2. ルートFSのサイズを2 GBから3 GBに増やす


始める前に



-lsofユーティリティが必要になります。 作業を開始する前にインストールする必要があります。

-問題を解決する過程で、サーバー上のすべてのプロセスを再起動する必要があることを理解する必要があります。

-重要！ 私はすでにLVMに精通している人々のためにこの記事を書いており、ブート時にカーネルがinitramfsの助けなしではそれ自体をマウントできないことを理解しています！



解決策



ソリューションを段階的に説明します。

計画はこれです：

1. LVMで新しい論理ボリュームを作成します

2.マウント用のフォルダーを作成する

3.古いルートFS /読み取り専用モードでの再マウントの準備

4.ルートFS /読み取り専用を再マウントします

5.バックアップ（オプション）

6. sda2デバイスからLVMルートボリュームにルートFSをコピーします

7. FSのサイズを変更します

8.コピーをマウントする

9.ルートFSを交換します

10.他のすべてのFSのマウントポイントを返す

11.新しいルートFSからアプリケーションを再起動します

12.ルートFSを変更した後の操作



1. LVMで新しい論理ボリュームを作成します

トムのサイズは3GBで、ルートと呼ばれます

lvcreate -L 3g -n root sys 

 作成された論理ボリューム「ルート」


2.マウント用のフォルダーを作成する

mkdir / mnt / oldroot / mnt / newroot

これは、古いルートFSを一貫した状態で新しい場所にコピーするために必要です。

しかし、FS自体は、おそらくプロセスによってデータを書き込むために使用される可能性が最も高いです。

ルートFSを転送すると、これらのソケットはアプリケーションに関連付けられなくなります。

これは解決されました：

4.ルートFS /読み取り専用を再マウントします

mount -n -o remount、ro /

すべてが成功すると、コマンドはサイレントに終了します。

「mount：/ is busy」という行が表示される場合、ルートFSはまだビジーです。 手順3に戻って確認します。 たぶん、あなたは何かを忘れました。



私も何かを予見していなかったかもしれないことを除外しませんが、この段階でこのステップを踏まなければ、先に進むことはできません。 まだ何も変更できていません。 停止したプロセスを戻すだけで機能します。

8.コピーをマウントする

マウント-n / dev / sys / root / mnt / newroot



9.ルートFSを交換します

これが私がこれを始めた理由です。

この瞬間から危険な時期が来ます。

重要：ルートFSを変更した後、SSHセッションが中断すると、システムは新しい接続を確立できなくなります！

代替FS：

cd / mnt / newroot 

 ピボットルート。  mnt / oldroot


現在の作業フォルダー（cwd）によるロックを回避するために、これらの2つのコマンドを記述された形式で正確に実行することが重要です。

このコマンドの後、ボリューム/ dev / sys / rootは/に落ち、sda2パーティションは/ mnt / oldrootのマウントポイントを変更します。 さらに、他のすべてのマウントされたFSもマウントポイントを変更します。 たとえば、/ devファイルシステムは/ mnt / oldroot / devに移動します。



10.他のすべてのFSのマウントポイントを返します（古いルートFSを除く）

標準FSを転送します。ほとんどのファイルには次のものがあります。

mount -n --move / mnt / oldroot / proc / proc

mount -n --move / mnt / oldroot / dev / dev

mount -n --move / mnt / oldroot / sys / sys



これで、/ proc / mountsを見ることができます。他の場所に戻す必要があるもの

猫/ proc /マウント| grep oldroot

私の例では、耐えます

mount -n --move /mnt/oldroot/lib64/rc/init.d /lib64/rc/init.d

mount -n --move / mnt / oldroot / var / distfiles / var / distfiles

mount -n --move / mnt / oldroot / var / db / var / db



これから、あなたは危険から脱します。 新しいSSHセッションが正常に開きます。

12.ルートFSを変更した後の操作



12.1。 fstabを変更することを忘れないでください

LABEL =タグを使用しているため、何も変更しません 

  LABEL = root / ext3 noatime 0 1


12.2。 古いルートFSをアンマウントします

umount / mnt / oldroot

rmdir / mnt / oldroot / mnt / newroot

他の人は使用しません

古いルートFSを保持したい場合は、ブートローダーを混乱させないように、LABELとUUIDを変更することをお勧めします。

tune2fs -L oldroot -U $（uuidgen）/ dev / sda2

私自身にとって、古いFSの必要性はもはや見当たりません。 削除します。

ワイプfs / dev / sda2 -o 0x438



12.3。 LVMに切り替えるときにinitramfsを追加/変更することを忘れないでください



12.4。 ブートローダーの再構成を忘れないでください
