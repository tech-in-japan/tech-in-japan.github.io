ソーシャルネットワークの最高の出版物

私の電報チャンネル： https : //t.me/winc0de

こんにちは。 空き時間には、ソーシャルプロジェクトを行います。 私の友人と私は、さまざまなソーシャルネットワークに十分な数の「パブリック」を持っているため、さまざまな実験を行うことができます。 書き込みの問題は、公開できる関連コンテンツとニュースを見つけることです。 この点で、最も人気のあるページから投稿を収集し、指定されたフィルターに従ってそれらを発行するサービスを作成するというアイデアが生まれました。 最初のテストでは、ソーシャルネットワークVKontakteとTwitterを選択しました。



テクノロジー



まず、データウェアハウスを決定する必要がありました（ところで、保存されたレコードの数は200万を超えています）。この数値は毎日解消されます。 要件は次のとおりです。大量のデータを非常に頻繁に挿入し、それらをすばやく選択します。



その前に、私はすでにnosqlデータベースについて聞いていたので、試してみたいと思いました。 私が行ったデータベースの比較については説明しません（mysql vs sqlite vs mongodb）。

キャッシングとしてmemcachedを選択しました。後で、その理由とケースを説明します。

Pythonデーモンは、データベースからすべてのグループを同時に更新するデータコレクターとして作成されました。



MongoDBとデーモン



まず、グループからの出版物のコレクターのプロトタイプを作成しました。 私はいくつかの問題を見ました：

  • ストレージ容量
  • APIの制限


すべてのメタデータを含む1つのパブリケーションには約5〜6KBのデータが必要です。中間グループでは、約20,000〜30,000レコード、グループごとに約175MBのデータが取得され、これらのグループが多数あります。 そのため、興味のない出版物をフィルタリングするタスクを設定する必要がありました。



あまり発明する必要はありませんでした。「テーブル」は2つだけです。1つ目は解析と更新が必要なグループのレコードを保存し、2つ目はすべてのグループのすべての出版物の範囲です。 今、私には不必要で悪い判断さえしているようです。 グループごとにテーブルを作成するのが最善であるため、レコードの選択とソートが簡単になりますが、200万であっても速度は失われません。 ただし、このアプローチにより、すべてのグループのサンプル全体が簡素化されます。



API



VKontakteソーシャルネットワークからのデータのサーバー処理が必要な場合は、任意のアクションに対してトークンを発行できるスタンドアロンアプリケーションが作成されます。 そのような場合のために、次のアドレスでメモを保存しました。



http://oauth.vk.com/authorize?client_id=APP_ID&redirect_uri=https://oauth.vk.com/blank.html&response_type=token&scope=groups,offline,photos,friends,wall









APP_IDの代わりに、スタンドアロンアプリケーションの識別子を挿入します。 生成されたトークンにより、指定されたアクションにいつでもアクセスできます。



パーサーアルゴリズムは次のとおりです。

グループIDを取得し、ループですべてのパブリケーションを取得し、各反復で「悪い」投稿をフィルター処理してデータベースに保存します。

主な問題は速度です。 vkontakte APIを使用すると、1秒間に3つのリクエストを実行できます。 1回のリクエストで、合計100の出版物（1秒あたり300の出版物）を取得できます。

パーサーの場合、これはそれほど悪くありません。1分でグループをマージできますが、更新にはすでに問題があります。 グループが多いほど、更新が長くなり、それに応じて、問題はそれほど速く更新されません。



解決策は、ヒープへのAPIへのリクエストを収集し、一度に実行できるexecuteメソッドを使用することでした。 したがって、1回のリクエストで、5回の反復を行い、500のパブリケーション（1秒あたり1,500）を取得します。これにより、グループは約13秒で「ドレイン」されます。
