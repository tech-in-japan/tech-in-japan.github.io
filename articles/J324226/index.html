古い情報システムをCUBAプラットフォームに移行する







技術とソフトウェア開発ツールの急速な開発は、情報システムの基礎となる技術が関連性を失い、大きな負担になるという事実につながります。 たとえば、Visual Basic 6.0またはDelphi 7で記述されたプロセス自動化のための会社開発を考えてみましょう。これは、控えめに言っても、「すべてのWeb、すべてのクラウド」という新しいトレンドと結びつかず、開発者の野心に対応していません。







古いIPを新しいテクノロジーに移行し、リーダーシップを発揮するという問題は、伝統的にお金にかかっています。 開発者にとっては、データモデルの転送と標準画面のテンプレートプログラミングがマイナスになります。 ただし、多くの場合、新しいISの開発および実装の段階で古いISの操作性を維持する必要があるため、すべてが複雑になります。 いずれにせよ、私の経験では、製品は完全に死に、プログラマーとユーザーの両方に苦痛を与えるか、IPの更新が緊急の必要であるという理解に至ります。







記載されている問題に加えて、レガシーシステムをCUBAに移行する際のプラットフォームベンダーとしての要望の増加に加えて、プログラマーにとってこのプロセスをできるだけ簡単にし、管理を安価にするメカニズムを追加することにしました。







猫の下で、データモデルと標準のCRUD画面を転送するための最小限の労力で古いシステムをアップグレードする方法に関する段階的な指示。









バージョン2.1以降のCUBA Studioで導入された移行機能を実証するために、データモデルと標準のCRUD画面を転送する労力を最小限に抑えながら、古いシステムをアップグレードする方法を段階的に説明しました。 Microsoftはエンタープライズ情報システムの迅速な開発のための一般的なツールであるLightSwitchのサポートを公式に廃止したため 、 LightSwitch Vision Clinicデモアプリケーションを例として選択し、CUBAプラットフォームに転送して、実行されたアクションの詳細な説明とともにGitHubに投稿しました。 ソフトウェアの近代化の必要性または経験がある場合は、提案された例に従うことをお勧めします。







リバースエンジニアリングには、 CUBA Studioに統合された移行ツールを使用します。 古いデータベースからメタデータを抽出し、それを使用して自動的に：







1.既存のデータベースに基づいてデータモデルを生成し、







2.データベースを更新して、必要なCUBAプラットフォームシステムテーブルを作成します。







3.標準のCRUD UIを生成します。







したがって、メインルーチンはCUBA Studioに引き継がれます。 ただし、標準とは異なるビジネスロジックと画面設計は、新しいアプリケーションに手動で転送する必要があります。







結果として得られるものに興味がある場合は、セクションまでスクロールしますMSSQLなしでアプリケーションを実行できますか？











ソースアプリケーションに関するいくつかの言葉



次に、移行の犠牲者として選択したレガシーアプリケーションの構成要素について説明します。 Microsoftの公式Webサイトから抽象アイクリニックのデモアプリをダウンロードしました。 これはMicrosoft Lightswitch開発環境で書かれており、Microsoftは数か月前に公式にサポートを停止しました。







実際、これはLightSwitchの主な機能を示す標準の3層アプリケーションです。データモデルの作成、足場画面、エンティティの計算された属性です。 例VisionClinicは、相互接続された2つの異なるデータベースにデータを同時に保存します。 簡単にするために、これらの2つのベースを1つに組み合わせました。







CUBAシステムテーブルは同じデータベースに作成されます。 これらの変更は、古いLightSwitchアプリケーションで使用されるテーブルに影響を与えないため、同じデータベースインスタンスで実行できます。













注：レガシーアプリケーションデータベースは、追加のデータストアとしてCUBAアプリケーションに接続できます 。 この場合、古いデータベースは完全に変更されずに残り、キューバのテーブルは個別に保存されます。







Lightswitchのユーザーインターフェイスは次のようになります。 製品リスト画面が最も気に入りました。他の画面よりも少し複雑です。













一般的に、これはエンタープライズアプリケーションの画面の古典的なレイアウトです。メインメニューが一番上にあり、画面が同じウィンドウに新しいタブの形で開きます。 必要に応じて、指示に従って自分で作成するか、単にダウンロードすることができます 。 アプリケーション全体から、データベースのみが必要です。 インストール方法については、対応するセクションで詳しく説明します。











CUBAベースのアプリケーション



将来を見据えて、私たちが最後に何を得たいかを言います。 従来の3つのレイヤーで構成される完全に機能するJavaアプリケーションを作成し、元のLightswitchアプリケーションのUIとビジネスロジックを繰り返す本格的なWebクライアントを作成します。 既に述べたように、アプリケーションは既存のテーブルの構造を変更しないため、両方のアプリケーションが同じデータベースインスタンスで並行して動作できます。







以下のスクリーンショットは同じ製品画面を示していますが、すでにCUBAアプリケーションで比較できます：

















ステップ1.環境とデータベースのインストール



1.プラットフォームのドキュメントのインストール手順に従って、CUBAプラットフォームをダウンロードしてインストールします。







2. 公式サイトからMS SQL Server 2012+をダウンロードしてインストールします。 サーバーが既にインストールされている場合は、この手順をスキップしてください。







3. MS SQL DBインスタンスでcreate-db.sqlスクリプトを実行して、VisionClinicデータベースを作成します。







4.新しく作成したVisionClinicデータベースでinsert-data.sqlスクリプトを実行して、テストデータを入力します。







5. ここに示すように、SQL ServerおよびWindows認証モードを有効にし、ユーザーsaとしてログインします。 saのパスワードを忘れないでください。CUBAアプリケーションからデータベースに接続するために必要です。











ステップ2. CUBAでプロジェクトを作成する



1. CUBA Studioサーバーを起動し（1）、ブラウザーでStudioを開きます（2）。













2.新しいプロジェクトを作成し（1）、プロジェクト名フィールドにvision-clinicと入力し（2）、[OK]をクリックします。













3. [ プロジェクトプロパティ]セクションで、VisionClinicデータベースへの接続を確立し、次のフィールドに入力します。







  • データベースの種類-Microsoft SQL Server 2012 、ここではインストールされているバージョンを示します（1）
  • データベースURL- localhost 、またはSQL Serverがインストールされているコンピューターのドメイン名/ IP（2）
  • データベース名-visionclinic 、この名前はcreate-db.sqlスクリプトがステップ1、パラグラフ3（3）で実行されたときに作成されました
  • データベースユーザーとデータベースパスワード-SQLサーバーのsaのログインパラメーター、または管理者権限を持つ別のユーザーを使用します（4）








4. [ 接続のテスト （1）]をクリックし、Studioが必要なデータベースに接続されているかどうかを確認します。 「 接続に成功しました （2）」というメッセージを受け取ったら、「OK」（3）をクリックして入力したパラメーターを保存します。

















ステップ3.移行-データモデルとCRUD UIの生成



このステップでは、既存のデータベースの構造とデータを操作するためのシンプルなユーザーインターフェイスに基づいて、データモデル（エンティティ）を生成します。







1. [データモデル]タブに移動し、[モデルの生成]リンクをクリックします。







2. CUBA Studioは、プラットフォームの動作に必要な独自のテーブル（セキュリティ設定、状態監査、動的属性など）を作成するために、データベースの更新を要求します。 更新を確認します。













3.「データベースからモデルを生成」ウィンドウが表示されます。 ここでは、データベーステーブルとその列のエンティティとそのフィールドへのマッピングを構成します。







4.上部パネルの[設定]ボタンをクリックして、一般的なマッピングパラメーターを構成します。 デモアプリケーションでは、どちらのフレームワークでもデータベースのエントリを作成または変更した人を追跡できるため、事前定義されたキューバのフィールドcreateTs、createdBy、updateTs、updatedByを、ほとんどすべてのテーブルにある既存のデータベースの対応するシステム列に関連付けます：Created （1）、CreatedBy（2）、Modified（3）、ModifiedBy（4）。







RowVersion（5）は、Lightswitchが楽観的ロックに使用する別のシステム列です。 残念ながら、これらのフィールドのタイプが一致しないため、同じ目的でプラットフォームで使用されるバージョン列にバインドすることはできません。 CUBAアプリケーションでオプティミスティックロックをサポートするには、別のフィールドを作成できますが、テーブル構造を変更しないことに決めたため、この列をマッピングから削除し、[OK]をクリックします。













5.上部パネルの[テーブルの表示]ボタンをクリックし、新しいアプリケーションでエンティティを作成するテーブルを選択します。 [すべて選択]ボタンですべてを選択し、sysdiagrams（1）から選択を削除します。これはデータベーススキーマが格納されているMS SQLサービステーブルであり、移行には必要ありません。 次へ（2）をクリックします。













6.この段階では、個々のエンティティとそのフィールドのレベルで、マッピングをより細かく構成できます。 たとえば、CUBAでは、Studioが画面名やメニュー項目ヘッダーなどの基本的なUI文字列を生成するため、すべてのエンティティの名前は単数形でなければなりません。 したがって、エンティティの名前を変更して、あらゆる場所から複数の遺産を削除します。 [予定]テーブルを選択し、[マッピングの編集]ボタンをクリックして、クラス名を編集し、[OK]をクリックして変更を保存します。













7.名前に複数形を持つ他のエンティティ（InvoiceDetails、Invoices、Patients）に対してこの手順を繰り返します。







8.この段階では、エンティティにインスタンス名-エンティティ（レコード）の文字列表現を指定して、UI（テーブルセルやドロップダウンリストなど）に表示することもできます。 [製品]テーブルを選択し、[マッピングの編集]をクリックして、[インスタンス名]フィールドでProductName属性を選択し、[OK]をクリックします。













9.これらのパラメーターはすべて後で修正できるため、何かを見逃したり、どこかでミスをしたとしても怖くありません。 次へをクリックします。







10.これで、データへのアクセスを提供するエンティティのCRUD画面を自動的に作成できます。 CUBAプラットフォームは、いくつかの標準画面テンプレートを提供します。







  • ブラウザ-データベースからレコードのリストを表示するための画面;デフォルトでは、データフィルタが含まれ、ページネーションがサポートされています。
  • エディター-個々のレコードを表示および編集するための画面。
  • ブラウザとエディタ-両方の画面が作成されることを意味します。
  • 単一画面-ブラウザー画面にエディターを埋め込むため、すべてのレコードのリストが左側に表示され、選択したレコードの詳細が右側に表示されます。


注：バージョン6.4以降、プラットフォームはカスタム画面テンプレートの作成をサポートしています。 また、画面はStudioおよびIDEで後で編集できます。







11.次のテンプレートを選択して、「次へ」をクリックします。













データベース更新スクリプト画面には、スクリプトは表示されません。 計画どおり、ソーステーブルは変更されません。 [保存]をクリックして、プラットフォームがデータモデルとCRUD画面を生成するのを待ちます。











ステップ4.最初の実行



この段階から、アプリケーションをすでに起動して、新しく作成した標準UIを確認できます。







1.メインメニューの[実行]-[アプリケーションの開始]コマンドを使用して、CUBAアプリケーションを起動します。













2. Studioの左下隅にあるWebクライアントリンクを使用して、アプリケーションに移動し、新しいブラウザータブで開きます。













3.デフォルトのユーザー名とパスワードはadmin / adminです。確認してアプリケーションを入力します。













4.作成した画面は、アプリケーションの上部にある[アプリケーション]メニューから利用できます。













5. [製品]画面を開き、単一画面テンプレートの標準レイアウトでどのように見えるかを確認します。













もちろん、ビジネスロジックの転送とインターフェースの設計にはさらに時間がかかりますが、この段階では、オープンソーステクノロジーに完全に基づいたクロスプラットフォームアプリケーションの最初のバージョンがすでにあります。











さらなる開発



さらにアプリケーションをどうするか、ここでは完全な選択の自由があります。 CUBAプラットフォームは、プラットフォーム自体のソースコードだけでなく、生成されたアプリケーションコードの変更も禁止していません。 Javaの世界のサードパーティテクノロジーをアプリケーションに統合することもできます。







たとえば、標準の製品ビュー画面では、十分な割引情報、類似の製品および画像の表がないことがわかります。 プラットフォームとCUBA Studioを使用すると、元のLightSwitchアプリケーションの画面を簡単に表示できます。







拡張された製品エンティティのソースコードと製品画面記述子は、GitHubプロジェクトで表示できます。 エンティティ宣言と画面記述子へのすべての変更は、CUBA Studioビジュアルエディターで作成されたことに注意してください。













コードに対するすべての手動変更は、 スクリーンコントローラーに登録されます 。 Productエンティティのソースコードで計算された属性を作成する例も見ることができます（ Product＃getCurrentPriceメソッドを参照）。











MSSQLなしでアプリケーションを実行することは可能ですか？



デフォルトでは、GitHubアプリケーションは軽量のHyperSQLデータベースで使用するように構成されています。 したがって、リポジトリからクローンを作成するだけで実行でき、起動時にStudio自体が必要なデータベースを作成します。 確かに、このデータベースは空であり、アプリケーションのユーザーインターフェイスを介して自分で入力する必要があります。







使用するデータベースのタイプを変更する場合、CUBA Studioでこれは非常に簡単です。別のDBMSに移行するには、[プロジェクトプロパティ]セクションでプロジェクト設定を開き、ドロップダウンリストからデータベースタイプを選択します。 Studioはデータベースを作成および更新するための新しいスクリプトを作成し、選択したデータベースからアプリケーションを実行できるようになります。













要約すると、ソフトウェアの近代化の必要性または経験がある場合は、ツールを評価し、場合によっては移行メカニズムを改善するためのオプションを提供することをお勧めします。 サポートフォーラムでフィードバックをお送りします。
