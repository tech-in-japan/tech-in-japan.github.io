チェックとバランスのシステム


インフラストラクチャの設計は、ボーイングの制限と経験の組み合わせから生まれました。



ボーイングでの経験からの問題の1つ-ラック内のサーバーがほこりを収集し、それらが過熱し始める。 したがって、最も負荷の高いコンピューター（これはX-Plane）はサーバールームに配置する必要があります。



キャブからUSBを引き出します-火がついたら、オンボードコンピューターが必要です。



モバイルプラットフォームに搭載されている場合、SSDまたはディスクレスのいずれかです。 メモリー内のSSDが2回オーバーヒートしたため、ディスクレスロードについて学習します。 Windowsにあるすべての「オンボード」ソフトウェアは、iSCSIになります。



ネットワーク経由で6台のコンピューターをダウンロードします。帯域幅が必要です。つまり、トランクをサポートするスイッチが必要です。 選択はD-Link DGS-1210-16で決まりました。



X-Planeは複数あり、それらのシナリオは、iSCSI、バックアップなどに加えて一致する必要があるため、ファイルサーバーまたはNASが必要です。 価格と品質の比では、Synology RS-812 +は最大3GBのメモリアップグレードを考え出しました。 同時に、トランクをサポートします。



サーバールームはクリーンである必要があります。つまり、ドアを閉める必要があります。つまり、リモートアクセスを最大化し、IP KVMである必要があります（実際には、VNCを使用するのが便利でした）。



さらに、オンボードコンピューターが配置されているコックピットスペースに登る際に、彼女はまったく笑わないため、少なくとも1つはオンとオフを切り替えることができなければなりません。 そのため、「電源」ボタンを押すことができる鉄筋コンクリート作業装置が必要です。 よく知られている電子技術者によって注文されました。一連の反復の後、電源ボタンを「押す」ことができるEthを備えたデバイスを入手しました。

次のようになります。





トップボードは「マスター」であり、「スレーブ」はRS-485を介して接続されます。 各「スレーブ」には8つのチャンネルがあり、各チャンネルには2つの個別の入力（電源LEDとHDDを接続するため）と電源ボタン用の1つの出力があります。



弊社のプロジェクターは高価で希少です。予備冷却なしでは電源を切ることはできませんが、入力電圧がなくなった場合は、人に頼らずに自動的にプロジェクターとオンボードコンピューターの電源を切る必要があります。 そのため、ネットワークインターフェイスを備えたUPSが必要であり、プロジェクターにも搭載されているため、使用する必要があります。



一般に、数週間の設計の結果、次のようなスキームが得られました。





4 GBのメモリを搭載したCeleron G540 2.5 GHzと、ギガバイトのメモリを搭載したRadeon HD 6450チップに基づくロープロファイルグラフィックスカードを搭載したオンボードコンピューターがあります。



X-Plane用グラフィックサーバー-8GB RAMとミラー内に2台の500GBドライブを搭載したCore i5-3470。 絵を描くためのGeForce GTX 670 c 2048MBです。 オペレーティングシステムとして、Ubuntu 12.04 LTS 86_64があり、X-Planeは64ビットです。



ダイナミクスを計算するサーバー-Core i7-3770のWin7_32では、統合ビデオカードを使用します。 ここではX-Planeは32ビットであり、モバイルプラットフォームとのインターフェース用のプラグインは32ビットおよびWindows専用であるため、サーバーはWindowsの下にあります。

電気


ボーイングでは、間違った電気ネットワークの問題に直面したため、今回は座って単線図を描きました（見つけられません）。

アイデアは、すべてのラインを自動機で保護し、さらに感電から人々を保護することでした。



負荷計算によると、キャブ全体（プロジェクター、コンピューター、I / Oコントローラー、機内換気、スイッチ）には3000VAで十分です。

5台のサーバーでは、NAS、スイッチ、KVMもほぼ同じ必要があります。



外部タクシーファン、インストラクターステーション、ランプドライブは1500VA未満で済むため、古い1500 SmartUPSが役立ちました

合計UPSのコストは4つです。2つのIppon Smart Winner 3000 VAと追加のバッテリーパック、および2つの1500VA SmartUPSがボーイングの改良の1つから残っています。

サーバールームは3ワット、もう1ワット-キャビン全体にかかっています。 キャビンとインストラクターステーションの換気口が1500にぶら下がっていますが、もう1つは在庫があります。



自律的なエネルギー源（UPS）の可用性を考えると、人々を保護するためのRCDはUPS、つまり 私たちの無停電装置はシールドに「取り付けられています」。

X-Plane：同期と起動


X-Planeデータを同期するために、最終的にNASからrsyncが選択されました。

私はNFSボールから直接実行しようとしましたが、WindowsとLinuxの両方、X-Planeのディレクトリ構造があることを考えると、それは酔ったせん妄で発明され、各構成には独自のスクリプトと一般的なデータがあると判断しましたX-Planeが起動または再起動するたびにネットワーク経由で同期します。



ちなみに、ボールから直接X-Planeを起動するというアイデアを考えながら、バイナリを編集して、書き込み用に開いたすべてのファイルが同じディレクトリにあるようにしました。 このステップの写真を次に示します。



すべてが正常に完了し、2日後に更新が行われました。 その後、rsyncを実行することにしました。



起動スクリプトは次のようになりました。 

#!/bin/bash LOGFILE="/var/log/xplane-starter.log" log_msg () { DATE=`/bin/date` /bin/echo "$DATE: $@" >> $LOGFILE } log_msg "Starting xplane-startup.sh" while [ 1 ]; do log_msg "Rsyncing" /root/bin/rsync_64.get >> $LOGFILE /bin/sed -i".bak" '/UNSAFE/d' "/xplane.64/Output/preferences/X-Plane Screen Res.prf" /usr/bin/pactl set-card-profile 'alsa_card.pci-0000_01_00.1' 'output:hdmi-stereo-extra1' /usr/bin/xinit /root/bin/xinitrc.64 >> $LOGFILE 2>&1 & XINIT_PID=$! log_msg "xinit pid = $XINIT_PID" while [ 1 ]; do sleep 1 # Check that the process is still alive if [ -e /proc/${XINIT_PID} -a /proc/${XINIT_PID}/exe ]; then sleep 1 else log_msg "Process dead, restarting" break fi done done




X-Planeは絶対にウィンドウマネージャーを必要としないため、実際の起動はxinitを介して行われます。

オンボードコンピューター監視


ProjectMagentaのフリーズするコンポーネントに対処するために、PowerShellスクリプト全体がオンボードコンピューターで記述され、さらに.NET要素も記述されました。



スクリプトは、失敗したUSBデバイスの検出から始まり、その後、すでに他のすべてのものでカバーされていました。



コンポーネントは、このスクリプトを通じてヒープに起動され、プレイ中に誰かがドロップアウトしたかどうかを確認します。



EHIDについては別の行で説明します。

EHIDは、タクシーハードウェア（I / Oコントローラー）とソフトウェアコンポーネント間のインターフェイスです。 NDAの下で配布され、私はそれに署名し、現在私は仕様の誇りに思う所有者です。 これまでに、X-Planeの分げつを使用したバグのガス化に適用しました。これについては、以下で説明します。

一番下の行は、コンポーネントA320_EHID.exe（したがって、ボーイングの場合はB737_EHID.exe）があり、コックピットのすべてのコンポーネントの「ツリー」と、アナログ軸、LED、トグルスイッチ、および一般的なものが含まれています。

各アイテムには独自の名前と32ビットIDがあります。



アプリケーションはハードウェアに直接アクセスしませんが、EHIDにアクセスします。 通信-TCPに加えて、イベント駆動型。 ポーリングはすべてEHIDで実装されているため、この概念は人生を大幅に簡素化します。



この仕様は、ハードウェアとEHIDの間、およびEHIDとアプリケーションの間の2つのレベルをカバーしています。

USB HIDの上にプロトコルの説明を含む下位レベルが表示され、コントローラーファームウェアにも実装されます。

X-Plane、Tiller、および松葉杖


ティラーは、フロントピラーヘッドコントロールハンドルです。 地上でのタキシングに必要です。

2つの方法で操縦できます-ティラーとペダル。 特定の速度に達すると、ティラーは動作を停止し、ペダルのみが残ります。



問題は、X-Plane（その著者によって認められているように）でデータ参照を「失った」ことです。 ティラーの位置を書き込むことができる内部変数。

さらに、ジョイスティックがX-Planeに接続されていない場合、彼は私たちがキーボード上を飛ぶ悲惨な貧弱な煮物だと信じているので、フロントハンドルを操作してサイドステアリングホイールからのコマンドをオンにする必要があります。



X-Plane Austin Meyerの著者（この間、彼はヒステリックなヤギであることが判明しましたが、ごめんなさいmoi）と、この件に関して非常に嵐のような通信を行いました。彼はリリースの1つでこのジョイントを修正することを約束しました。 待っています。



待っている間に、一方では仮想ジョイスティック（VJoy）にしがみつく松葉杖を書き、もう一方ではネットワークを介してEHIDがティラーハンドルの位置の値を取得します。

X-Planeでは、単一の軸を割り当てました。これはティラーであり、X-Planeの目には不正であることがなくなったため、サイドターン機能はサイドスティックによってオフになりました。

この松葉杖には複雑なものも珍しいものもないので、ソースを持ってくる理由はありません。 私の意見では、.NETで1時間で書かれています。
