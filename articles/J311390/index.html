Clever Geek Handbook

ある強迫観念の物語、またはPhotoshopのカレンダースクリプトの書き方







この記事では、最初から最後まで、Photoshopでバグとイースターエッグを使用してカレンダーを作成するスクリプトをどのように作成したかについて説明します。 たくさんのテキスト、いくつかのコード、少しのイラスト。



はじめに



それはすべて、小さな紙にスケッチしたアイデアから始まりました。 特定の年のカレンダーをPhotoshopで直接作成できるスクリプトを書きたかったのです。 最初は、サイズ6の年間カレンダーを2か月間取得する必要がありました。 ただし、開発プロセス中にウィッシュリストが大きくなり、その結果、目的の設定のリストがユーザーが指定できるようになりました。



  • 年
  • 月と曜日の名前
  • 週の始まり（月曜日、日曜日、またはその他の日から）
  • あなたの週末（任意の日の休日としてマーク）
  • 週番号表示
  • 要素間のいくつかの間隔
  • 要素の相対的な位置と日のシーケンス
  • テキストレイヤーの色、フォントとそのサイズ
  • 休日（特別に作成されたテキストファイルが読み込まれ、そこから休日の日付が取り出され、カレンダーにマークされます）-特別な欲求


また、「プレビュー」と同じような機能が必要でした。設定の結果を明確に反映するテスト月が作成されました。



Photoshopでこれを行うことが重要なのはなぜですか？ 写真やイラストを使ってカラフルなカレンダーを作成し、できる限り手間をかけず、できればこのプログラムで作業したいだけです。 たぶん誰かがそれを望んでいます。



ツールキット



だから、計画があり、必要なツールを選択する時が来ました。



言語。 Photoshopは、AppleScript、VBScript、およびJavaScriptのスクリプト言語をサポートしています。 ここの誰もが最初のものはMacOSに、2番目はWindowsに、3番目は両方に適していることを理解していると思います。 JSが選ばれたのは、しばらく前に私がその基礎を学んだからであり、そのようなスクリプトを開発することは、この方法で優れた実践として役立つ可能性があります。



テキストエディター。 私は通常の「ノートブック」に最初の100行を書きましたが、それほど長く続けることはできませんでした。 素晴らしいマルチカーソル機能を備えたSublime Textを思い出しました。



XBYTORのスクリプトツール。 実際、そのツールの1つであるActionToJavascript.jsxのみを使用しましたが、非常に便利でした。 このスクリプトは次のように機能します：入力では、記録された操作がPhotoshopで指定され（これはMS Officeのマクロに似ていますが、閉じた形式のみです）、出力は* .jsx拡張子を持つファイルであり、低レベルのコマンドを使用して同じことを実行する機能が含まれています元の操作として。



Photoshopのスクリプトリスナー。 Adobeの公式プラグイン。 上記のツールと本質的に同じことを実行しますが、わずかに異なる原理で動作します-プログラムウィンドウでユーザーのアクションを監視し、デスクトップ上の2つのテキストファイルに書き込みます。 最初のファイルはJavaScriptで、2番目はOSに応じてVBSまたはAppleScriptです。 そして、はい、彼は常に書いています。プログラムを開始する前に、プラグイン名の先頭にチルダ（ ~



）を追加して削除またはオフにする必要があります。



Adobe ExtendScriptツールキット。 実際、このツールは、Adobeプログラムのスクリプトを作成するためのネイティブソリューションですが、作業の最後でのみ使用しました。 これにより、見知らぬ人からのソースコードを閉じることができます。 出力は、読み取り不能なコンテンツを含む作業スクリプトです。



ドキュメンテーション これがないと、「こんにちは！」というポップアップよりも複雑なものを書くのは非常に問題になります。 記事の最後にいくつかのリンクがあります。



最適化



スクリプトを記述するプロセスで作業を最適化する必要がありました。 最初のブレーキは、スクリプトがすでに3か月を作成できたときに始まりました。 最初は、すべての作業を1つのPhotoshopドキュメントでやりたいと思っていました。 しかし、最初の月の後、速度が大幅に低下し始めました-そして、1月がフライバイだった場合、新しい月ごとに作成時間が増加しました。 すでに2月は1分半、3月は3分でした。 簡単に説明しました-各日付、各碑文は個別のテキストレイヤーであるため、成長中のレイヤースタックで新しいレイヤーを作成して移動するのに時間がかかりました。 したがって、しぶしぶ初めて、毎月の作成を新しいドキュメントに転送し、結果を単にメインドキュメントにコピーしました。 したがって、月を作成する時間を増やすという問題は解決されました。今では、平均して毎月、同じ時間で作成されています。



次の驚きは、受け取った月をそれぞれ正しく配置するために、受け取った月のそれぞれのサイズを調べる必要があるときに起こりました（ユーザーは異なる間隔を設定できます）。 月の高さと幅の値を返すはずの単純な関数をすばやく作成し、実行しました。 Photoshopが固くハングします。 彼に寛大な10分間を与えた後、彼は決して蘇生しなかったので、プロセスを殺さなければなりませんでした。 私はいつか彼が終わったと思うが、10分はそれ自体では多すぎる。 問題は再びドキュメント内のレイヤーの数であることが判明しました-それらの数が多すぎて適切な数をすぐに取得できませんでした。 そしてしぶしぶ再び、私は毎月作成された段階にサイズの計算を移動しました。 作成された月はサイズによってすぐに決定され、測定値は後で使用するために配列に記録されました。 簡単に聞こえますが、実際にはかなり時間がかかりました。



間違い



ここでまず第一に、私のテスターに​​大きな感謝を申し上げたいと思います。彼らなしでは、現在よりもコードにさらに多くの欠点を残していたでしょう。



レンダリングのエラー。 これは、ユーザーインターフェイスを使用してスクリプトを作成し、レンダリング時に要素サイズ（ボタンなど）の複雑な再計算を使用する場合のヒントです。WindowsおよびMacOSでスクリプトの動作を確認します。 MacOSシステムで1つの場所でスクリプトがクラッシュするケースがありました。 私は長い間理由を探しましたが、それは以下でカバーされていました：ボタンの1つのサイズの計算は、増加の方向で他のボタンに関連して調整されるべきであり、この操作は私にサイクルに駆り立てられました（私は同意する、議論の余地がある決定）。 PhotoshopのすべてのバージョンのWindowsでは、調整可能なボタンが常にメインボタンよりも小さいため、これはうまく機能しました。 しかし、MacOSでは、逆のことが判明しました。メインボタンの調整が1〜2ピクセル少なく、スクリプトは無限ループに入りました。 今、私はそれを修正しました。そして、唯一可能な結論は、インターフェース要素（Cap）を再計算するときに疑わしい決定を使用しないことです。



別の憂鬱な瞬間は、Photoshopの異なるバージョンが異なる方法でウィンドウをレンダリングすることです。特に透明度を持つ要素は、背景でこれらのボタンを再描画しなければならなかった理由です（私には思えますが、私は混乱して他の何かを思い付く可能性があります）









Terry Pratchett irのファンは、Holi WIZZARDボタンに気付いたかもしれません。 私はこれらのものが好きです、これがイースターエッグであるかどうかはわかりません（ちなみに、以下でそれらについて）、1つのアンチウイルスと似たものを見ました-スキャナーはLuke Filewalker（Luke Sky Filealker）と呼ばれます。



エラーは内部的なものです。 まれですが、説明が非常に難しいエラーがあります。 私の場合、ブロックテキストの幅（および場合によっては高さ）の設定がこのような問題であることが判明しました。 説明してみましょう。Photoshopには、2種類のテキスト-ポイントテキスト（短い）と段落テキスト（ブロック）があります。 2番目とは異なり、最初の境界には境界がありません。









そこで、私に次のことが起こりました。このブロックテキストの幅を800ピクセルに設定しました。 出力では、3333ピクセルの幅が得られました。 私は自分の目を信じられず、コードをチェックし始めました。 間違いはありませんでした。 それは夕方で、疲れた頭で罪を犯したので、私は手に葉を置いて特別な注意を払って調べ始めました。 間違いは目に見えませんでした。



30分後、何が起こっているのかを理解するためにすでに必死だったとき、同様の問題とのリンクが目を引きました。 ドキュメントの解像度が72 dpiを超える場合、Photoshopは独自の値を公開します。 この方法で解決されます：ドキュメントの解像度（300に等しい）を取得し、標準の72を解像度で除算し、結果にテキストブロックの目的の幅を掛けます。 その結果、300 dpiのドキュメントで800ピクセルのブロックを取得するには、サイズを800 * 72/300 = 192ピクセルに設定する必要があります。 とても簡単です。



週のエラー。 私が非常に愚かだったもう1つのポイントは、週数の計算です。 無知から、最初の週は1月1日に始まるという絶対的な信念に基づいて、週の番号を付けました。 私はこのことを確信していたので、事前に確認することさえしなかったが、すぐに実装を開始した。 終了したときだけ、1月1日から番号付けを開始することが正しいかどうかを確認しました。 いいえということが判明しました。



この記事を読んだ多くの（またはすべての）人々は、日付と時刻の形式にISO 8601標準があることを知っています。 そして彼によると、週は月曜日に始まり、最初の週は1月の最初の木曜日に当たる週です。 また、最初の週は1月1日からまったく開始されない場合がありますが、たとえば、2日目または4日目から（この2016年のように）開始されます。 そのため、コードのこの部分も完全に書き直す必要がありました。



...配列のエラー？ ここでは何が問題なのかわかりませんが、解決策は興味深いものでした。



フォント名を含む7つのドロップダウンメニューを作成するコードは次のとおりです。 



 for (var i=0; i<7; i++){ fontGroup.nameOfFont = fontGroup.add('dropdownlist', undefined, arrFonts); fontGroup.nameOfFont.selection = fontIndex[i]; } /* fontGroup –  «»   «» arrFonts –     fontIndex –             */


ウィンドウが最初に生成されたとき、すべてが問題なく正常に実行されました。 ただし、ユーザーが保存をロードすると、ウィンドウは閉じられ、新しいfontIndex値で再度開かれます。 設定を読み込んでウィンドウを再作成するときに、フォントの名前が（私の場合は） Aarcover（Plain）：001.001の場合、Photoshopがクラッシュしました 。 残りのフォントは問題なく動作しましたが、これによりプログラム全体がクラッシュしました。



可能性は非常に小さいように思えますが（実際にはそうではありませんが）、非常に悩みました。 問題の最初の解決策はこれでした： 



 alert(arrFonts); for (var i=0; i<7; i++){ fontGroup.nameOfFont = fontGroup.add('dropdownlist', undefined, arrFonts); fontGroup.nameOfFont.selection = fontIndex[i]; }


つまり、このサイクルの前に、arrFonts配列のすべての要素がリストされる警告ウィンドウを呼び出すだけで（このウィンドウが画面全体に表示され、[OK]ボタンが表示されなかった場合）、スクリプトウィンドウは通常どおりに生成されます。 空白行と一貫性のないテキストを含む他の「アラート」は、フォントの配列を使用する場合にのみ役立ちました。



しばらくして、動作する2番目のソリューションが表示されましたが、その理由はわかりません。 



 var tempArrFont = arrFonts.slice(); for (var i=0; i<7; i++){ fontGroup.nameOfFont = fontGroup.add('dropdownlist', undefined, tempArrFont); fontGroup.nameOfFont.selection = fontIndex[i]; }


スライスメソッドを使用して「問題」配列のコピーを作成し、元の代わりにこのコピーを使用することがわかります。 動作し、現在使用されています。 これがなぜ起こっているのか、どういうわけか修正できるかどうかの答えをコメントで見て喜んでいるでしょう。



便利な休憩



2015年12月上旬のどこかで、スクリプトの作成を一時停止しました。 しかし、疲れているか興味を失ったので、私は止めませんでした。 別のスクリプトを書くためにスクリプトから脱線します。 このことについて詳しくは説明しませんが、このスクリプトの開発中に、低レベルのコマンドを使用してテキストレイヤーを作成する方が、マニュアルの標準的な方法を使用して関数を使用するよりもはるかに高速であることがわかりました。 はい、新しい関数では、最初に書いた関数よりもはるかに多くのコード行がありましたが、それでも高速に動作しました。 私たちの低レベルがすべてであると推測することは可能ですが、その瞬間、私は徹底的に愚かで、真ん中でのみそれをチェックしました。
