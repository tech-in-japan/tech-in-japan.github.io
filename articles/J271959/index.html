NetApp ONTAP：SAN環境でのUNMAP

UNMAPコマンドは、T10 SCSIコマンドセットの一部として標準化されており、SAN環境の細い月からデータストアにスペースを解放するために使用されます。 先ほど書いたように、SANとNASのプロトコルは次第にお互いから最高のものを借りてきました。 昔登場した便利な機能の1つは、以前はSANに欠けていた薄いブロックにリモートブロックを「戻す」ために、ストレージシステムとホスト間のフィードバックの可能性です。 UNMAPがSAN環境で使用されることはほとんどありませんが、仮想化環境と非仮想化環境の両方と組み合わせると非常に便利です。



UNMAPチームのサポートがなければ、ストレージ側で作成された薄い衛星は常にサイズが大きくなる だけでした 。 その成長は時間の問題であり、そのような薄い月が最終的にその全量を占めるという事実で常に無条件に終了しました。 彼は最終的に太くなるでしょう。











データストアに3台の仮想マシンがあり、それぞれが100GBを占有していると想像してください。 データストアは、300GBを占有する細い月にあります。 ストレージ側とESXiからの占有スペースは同じで、300GBです。 VMを1つ削除しても、ストレージ側からの月のサイズはまだ300GBであり、ハイパーバイザーの側から見ると、この月に存在するデータストアの占有スペースは200GBです。



これは、傷がホストとストレージシステムの間にフィードバックメカニズムを持っていなかったためです。 つまり、ホストが月に情報ブロックを記録すると、SHDはこのブロックを使用中としてマークしました。 さらに、ホストはこのブロックを削除できますが、ストレージシステムはそれをまだ知りませんでした。 UNMAPチームはこのフィードバックであり、月から不要になったブロックを加熱します。 最後に、私たちの薄い衛星は、獲得するだけでなく、ONTAP 8.2ファームウェアバージョン（クラスターデータ）から重量を減らすことも学びました。



VMware ESXiおよびUNMAP



バージョン5.0では、UNMAP機能が初めて導入され、デフォルトで有効になり、リモートブロックの設定値に達すると自動的に起動されました。以降のバージョンでは、メカニズムはデフォルトで無効になり、手動で開始されました。 VMFS-6（vSphere 6.5）以降、12時間以内にスペースが自動的に解放されます。必要に応じて、スペースの再利用を開始するための手動メカニズムも利用できます。 ここで説明するスペースのリリースは、ハイパーバイザーのレベルで行われることに注意することが重要です。 削除されたブロックは、ゲストOS内ではなく、仮想マシン全体または仮想ディスクを削除した後にのみ解放されます。



UNMAPサポートを備えたESXiおよびONTAPストレージが既にあるが、機能が有効になっていない場合



ストレージ側とハイパーバイザーの側から有効にする必要があります。 まず、月をオフラインにして、ストレージシステムのスペース割り当て機能をオンにします（VMが実行状態のままになっていると、VMが誤ってシャットダウンし、破損する可能性があるため、一時的にオフにするか、一時的に移行する必要があります）。 



lun modify -vserver vsm01 -volume vol2 -lun lun1_myDatastore -state offline lun modify -vserver vsm01 -volume vol2 -lun lun1_myDatastore -space-allocation enabled lun modify -vserver vsm01 -volume vol2 -lun lun1_myDatastore -state online


その後、ESXi側からマップ解除を有効にする必要があります。これを行うには、ESXiがUNMAPサポートを検出するようにデータストアをマップおよびマップする必要があります）： 



 esxcli storage filesystem unmount -l myDatastore esxcli storage filesystem mount -l myDatastore esxcli storage vmfs unmap -l myDatastore


その後、スペースを解放するために、定期的にコマンドを実行する必要があります。 



 esxcli storage vmfs unmap -l myDatastore


UNMAPは、1 MBの倍数のパーティションオフセットでフォーマットされた衛星に対してのみ機能することに注意することが重要です 。 これはどういう意味ですか？ つまり、一度VMFS3をVMFS5に変換した場合、UNMAPは機能しません。 これは簡単に確認でき、変換されたVMFS3にはMBR内訳テーブルがあり、再作成されたVMFS5にはGPTスプリットがあります。 



 # esxcli storage core device partition showguid Example output: Device Partition Layout GUID ------------------------------------------------------------- naa.60a98000486e574f6c4a5778594e7456 0 MBR N/A naa.60a98000486e574f6c4a5778594e7456 1 MBR N/A naa.60a98000486e574f6c4a5052537a7375 0 GPT 00000000000000000000000000000000 naa.60a98000486e574f6c4a5052537a7375 1 GPT aa31e02a400f11db9590000c2911d1b8


[レイアウト]列に注意してください。



オフセットの確認も難しくありません。セクターの長さを調べます。 セクターが512バイトであることを思い出させてください。 

 # esxcli storage core device partition list Example output: Device Partition Start Sector End Sector Type Size ------------------------------------------------------------------------------------- naa.60a98000486e574f6c4a5778594e7456 0 0 3221237760 0 1649273733120 naa.60a98000486e574f6c4a5778594e7456 1 128 3221225280 fb 1649267277824 naa.60a98000486e574f6c4a5052537a7375 0 0 3221237760 0 1649273733120 naa.60a98000486e574f6c4a5052537a7375 1 2048 3221237727 fb 1649272667648


「Start Sector」列と「End Sector」列に注意してください。 したがって、最後のデバイスnaa.60a98000486e574f6c4a5052537a7375には1MBのオフセットがあります（2048セクター* 512 = 1048576バイト= 1024KB）。 ただし、2番目のデバイスnaa.60a98000486e574f6c4a5778594e7456のオフセットは1MBの倍数ではなく、明らかに小さいため、UNMAPは機能しません。



UNMAP（ Delete Status ）がサポートされているかどうかを確認できます： 



 # esxcli storage core device vaai status get -d naa Example output: naa.60a98000486e574f6c4a5052537a7375 VAAI Plugin Name: VMW_VAAIP_NETAPP ATS Status: supported Clone Status: supported Zero Status: supported Delete Status: supported


vSphere 6.5の自動再生スペース



vSphere 6.5以降では、ストレージ上の薄い月へのスペースの自動解放がサポートされています。 VMFS-6データストアごとに、12時間以内にストアに返されるHigh / Mid / Slowスペースのリリースに優先順位を付けることができます。 グラフィカルインターフェイスまたはコマンドラインから、スペースの再利用を開始し、VMFS-6データストアの優先順位を手動で構成することもできます。 



 esxcli storage vmfs reclaim config get -l DataStoreOnNetAppLUN Reclaim Granularity: 248670 Bytes Reclaim Priority: low esxcli storage vmfs reclaim config set -l DataStoreOnNetAppLUN -p high






ゲストOSからUNMAP。



そのため、以前はデータストアから仮想マシンを削除することを検討しました。 仮想マシン全体またはそのディスクではなく、ゲストOS内からデータブロックを削除するときに同じことを行うのは論理的です。 UNMAPはONTAPでストレージ側でサポートされているため、VMDKからデータを削除するときにUNMAPメカニズムが機能します。 ゲストOSの内部から、ストレージ側から、追加の実装は必要ありません。UNMAPを有効にするには十分です。 ハイパーバイザーがこの情報を仮想マシンからストレージにブロードキャストできることが必要です。ストレージは完全にSCSIレベルで実行されます。 そのため、 ESXi 6.0以降では、ゲストOS内のリモートブロックに関する情報を送信できるようになりました。



仮想マシン内からUNMAPを機能させるには、次の条件が満たされている必要があります。



  • 仮想ハードウェアバージョン11
  • VMFS6
  • vSphere 6.0 * / 6.5
  • ムーンストレージは薄くする必要があります
  • 仮想ディスクvirtualkaは薄いはずです
  • ゲストOSファイルシステムはUNMAPをサポートする必要があります
  • vSphere 6.0 CBTではオフにする必要があります
  • 必要に応じて、ハイパーバイザーでUNMAPを有効にします。esxclistorage vmfs unmap -l myDatastore
  • ストレージでUNMAPを有効にする


薄くて薄い



長年にわたり、すべてのストレージベンダーは、シンムーンにシン仮想ディスクを作成しないと言っていました。 しかし、現在はこれが変更されており、仮想マシン内からブロックを解放するには、ストレージシステムにシン仮想ディスクとシンムーンが必要です。 vSphere 6.0では、ゲストOS内部からリモートブロックを返す機能が実装されましたが、 UNMAPを使用する場合、Linuxマシンがサポートされないなど、 いくつかの制限がありました。 VMFSを備えたvSphere 6.0以前のバージョンでは、UNMAP機能が自動的に開始されないため 、コマンドを手動で実行する必要があります。

LinuxゲストOS SPC-4サポート


SPC -4をサポートし、vSphere 6.5で実行されているLinux仮想マシンも、仮想マシンの内部から解放されたスペースをストレージに戻すことができるようになりました。

Linuxマシンがスペースのリリースをサポートしているかどうかを確認する方法は？



Linux仮想マシンでのSPC-4サポートの確認
これを行うには、コマンドを実行します 

 sg_vpd -p lbpv Logical block provisioning VPD page (SBC): Unmap command supported (LBPU): 1 Write same (16) with unmap bit supported (LBWS): 0 Write same (10) with unmap bit supported (LBWS10): 0


Unmap command supported（LBPU）パラメーターを1に設定すると、シンディスクが使用されます。これが必要なことです。 値0は、仮想ディスクのタイプがシック（熱心またはスパース）であることを意味し、UNMAPはシックディスクでは機能しません。 



 sg_inq /dev/sdc -d standard INQUIRY: PQual=0 Device_type=0 RMB=0 version=0x02 [SCSI-2] [AERC=0] [TrmTsk=0] NormACA=0 HiSUP=0 Resp_data_format=2 SCCS=0 ACC=0 TPGS=0 3PC=0 Protect=0 [BQue=0] EncServ=0 MultiP=0 [MChngr=0] [ACKREQQ=0] Addr16=0 [RelAdr=0] WBus16=1 Sync=1 Linked=0 [TranDis=0] CmdQue=1 length=36 (0x24) Peripheral device type: disk Vendor identification: VMware Product identification: Virtual disk Product revision level: 1.0


バージョンversion = 0x02 [SCSI-2]は、UNMAPが機能しないことを意味します。バージョンSPC-4が必要です。 



 sg_inq -d /dev/sdb standard INQUIRY: PQual=0 Device_type=0 RMB=0 version=0x06 [SPC-4] [AERC=0] [TrmTsk=0] NormACA=0 HiSUP=0 Resp_data_format=2 SCCS=0 ACC=0 TPGS=0 3PC=0 Protect=0 [BQue=0] EncServ=0 MultiP=0 [MChngr=0] [ACKREQQ=0] Addr16=0 [RelAdr=0] WBus16=1 Sync=1 Linked=0 [TranDis=0] CmdQue=1 length=36 (0x24) Peripheral device type: disk Vendor identification: VMware Product identification: Virtual disk Product revision level: 2.0




UNMAPが稼働していることを確認しましょう。
これを行うには、コマンドを実行します 

 grep . /sys/block/sdb/queue/discard_max_bytes 1


値1は、ゲストOSがファイルシステムから削除されたブロックのSCSIレベルを通知することを示します。



UNMAPを使用してスペースが解放されているかどうかを確認します。 

 sg_unmap --lba=0 --num=2048 /dev/sdb # blkdiscard --offset 0 --length=2048 /dev/sdb


「blkdiscard：/ dev / sdb：BLKDISCARD ioctl failed：Operation not supported」というエラーが表示される場合、UNMAPは機能していません。 エラーがなければ、 discardキーを使用してファイルシステムをマウントします。 

 mount /dev/sdb /mnt/netapp_unmap -o discard




VMwareハイパーバイザーはUNMAPを非同期的に 、つまり遅延して開始することを覚えておくことが重要です。 これは、実際には、ゲストOS内/ハイパーバイザーデータストア上およびストレージムーン上に、1：1の占有スペースがほとんどないことを意味します。

ゾンビ



ONTAP 9より前のファイル削除操作では、いわゆるゾンビメッセージが生成されていました。
ONTAP 8およびゾンビメッセージ
削除されたブロックを解放すると、ディスク領域を大幅に節約できますが、一方で、ホストがストレージシステムにさらに多くのブロック（たとえばテラバイトのデータ）を解放するように要求した場合、ストレージシステムはこれを行い、完了するまで落ち着きません。これはストレージシステムでの追加アクティビティです。 テラバイトのデータのリリースは、ディスクまたはハイブリッド（オールフラッシュではない）システムで特に顕著になります。 したがって、このようなシステム（テラバイト）で一気にデータを削除する場合は注意が必要です。



この状況にある場合は、テクニカルサポートにご相談ください。wafl_zombie_ack_limitおよびwafl_zombie_msg_cntの値を増やす価値がある場合があります 。 月のすべてのデータを削除する必要がある場合は、ストレージシステムの月全体を削除することをお勧めします。 一方、すべてのFlashシステムは、このような要求の影響を受けにくく、原則として、こうしたタスクに迅速かつ楽に対処します。





ONTAP 9でのファイルの削除と優先順位付け


ONTAP 9は2016年5月にリリースされたことを思い出してください。このバージョンのZombieソフトウェアでは、スペースを解放するためのメッセージは生成されなくなり、代わりにファイルに削除のマークが付けられます。 これに関連して、スペースを解放する作業のロジックが大幅に変更されました。

  • 最初に、データはバックグラウンドで削除されます
  • 第二に、より多くを必要とするボリューム、つまりスペースで終わるボリュームからのスペースの解放を優先することが可能になりました。




結論



ゲストOS、ホスト、およびNetAppストレージ側の両方でUNMAPサポートが実装されているため、シンプロビジョニングを使用してSAN環境のスペースをより合理的に使用でき、その結果、ハードウェアスナップショットでストレージスペースをより合理的に使用できます。 ハイパーバイザーレベルでのUNMAPのサポート、およびゲストOS内でのサポートは、これら2つの一般的なテクノロジーの使用を大幅に促進します。
