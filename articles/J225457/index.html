VSTおよびAU





WDL-OLでは、アプリケーションをVSTまたはAUに簡単に変換できます。 MacでVSTを作成するには、レイアウトを変更します： VST2-> My Mac 64-Bit 実行をクリックします 。 エラーはないはずですが、何も始まって​​いません！ Finderを開き、 〜/ Library / Audio / Plug-Ins / VSTに移動します。 ライブラリフォルダがホームディレクトリに表示されていない場合は、 MyFirstPlugin.vstが表示されます。Cmd + Shift + Gを押し、指定されたパスを貼り付けてEnterを押します。



次のセクションはMac用です。 以下のWindowsバージョン。



Xcodeでのデバッグ





プラグインを実行するには、ホストが必要です。 できれば64ビットバージョンのREAPERをダウンロードし、 アプリケーションフォルダーにドラッグしてください。60日間制限なく使用できます。その後、ライセンスを購入するよう求められます（ただし、引き続き機能します）。 Xcodeで実行をクリックすると、REAPERとデバッガーが起動するように、すべてを構成します。 アプリケーションで行ったのと同じ方法でブレークポイントを使用できます。



製品→スキーム→スキームの編集...に移動し、左側のリストで「 実行 」を選択します 。 実行可能サブ項目で、 その他...を選択します 。







ダイアログボックスを使用して、 REAPER64.appを見つけます 。 次のようになります。







[ OK]をクリックして、もう一度実行します。 REAPERが起動します。 サウンド用の新しいトラックを作成するには、Cmd + T（WindowsではCtrl + T）を押し、その上にFXボタンがあります：







開いたウィンドウで、 myfirstpluginと入力すると、プラグインが見つかります。







選択して、 OKをクリックします。 彼が表示されるはずです。 REAPERを使用する利点は、プラグインで任意のサウンドファイルを処理し、すぐに結果を聞くことができることです。 これは、マイク入力から常に聞くよりもはるかに興味深いです。 さらに、ここでは、プラグインの出力信号の分析など、他のさまざまなプラグインを使用できます。 これは便利になりますが、後になります。

このファイルをダウンロードして、REAPERのトラックにドラッグします 。 トラックの左境界線の近くに配置します。 スペースバーを押して再生します。 低い音が聞こえます：







便宜上、スペースバーをもう一度押すまで無限に再生されるようにサウンドをループするとよいでしょう。 Cmd（WindowsではShift）を押しながら、トラック上のファイルをダブルクリックします。 明るい長方形が時間範囲を示します。 Rを押してループし、実行します。 プラグインノブを回す-音量が変わります！







ノブを回すと、カチッという音がする場合があります。 これは、値を補間せず、信号に急激な差を作成しないためです。 Xcode Stopをクリックして、REAPERを閉じます。



AUバージョンの作成は簡単です。ターゲットをAUに変更します。 REAPER.appを実行する前のようにRunスキームを変更し、 Runをクリックします 。 プラグインがコンパイルされ、そのファイルが〜/ Library / Audio / Plug-Ins / Componentsに表示されます 。 REAPERが起動し、VSTバージョンと同じことができます。



これらすべてが始まり、トラックを作成し、プラグインをスローします。 私はこのようなものを最適化することを好みます。 REAPERですべての準備ができたら（サウンドがループバックされ、プラグインは既にトラックにかかっています）、プロジェクト（MacではCmd + S、WindowsではCtrl + S）をXcodeのプロジェクトフォルダーに保存します（ 〜/ plugin-development / wdl-ol / IPlugExamples / MyFirstPlugin / ）およびreaper-project.RPPという名前。



[ スキームの編集]ウィンドウで、上部の[ VST2]を選択します。 次に、[ 引数 ]タブに移動し、[ 起動時に渡される引数 ] 行の下にあるプラス記号をクリックします。 reaper-project.RPPをプロジェクトフォルダーから新しい行にドラッグします。







AUについても同じことを繰り返す必要があります。 必要に応じて、AUバージョン用に別のプロジェクトを作成することもできます。 これで、REAPERはすべてのジャンクで既に開始されます。 わあ！



必要に応じて、Xcodeで直接実行されていなくても、ブレークポイントをいじってプラグインの変数の値を追跡できます。



Windowsでのデバッグ





エクスプローラーを開き、 C：\ Program Filesに移動します 。 そこでvsthostという名前の新しいフォルダーを作成します。 Savihostプログラムをダウンロードし（ savihostx86.zipが必要です ）、 savihost.exeファイルを解凍してこのフォルダーにコピーします。







Visual C ++で、 MyFirstPlugin-vst2を右クリックし、「スタートアッププロジェクトとして設定 」を選択します。 F5を押すと、プラグインがSavihostで起動します。







何も表示されない場合は、タスクバーのウィンドウをクリックしてください。

もう1つ実行する必要があります。Savihostはすべての例外をキャッチし、プラグインがクラッシュしないことを確認しようとします。 ステージで演奏するのは素晴らしいことですが、今はデバッグする必要があります。 Visual C ++で、 MyFirstPlugin-vst2を右クリックし、 [ プロパティ ] を選択します 。 開いたウィンドウで、「 デバッグ 」を選択します。 図のように、パラメーターの右側に/noexc /noft



追加します（スペースを失わないでください）。







これで、Visual C ++デバッガーに例外がスローされます。



VST3バージョンを実行しようとすると、次のようなメッセージが表示される場合があります（おそらくドイツ語ではありません）。







前にダウンロードしたVST3 SDKでは、アーカイブは\ binサブフォルダーにあります。 32ビットおよび64ビットシステムに必要なフォルダーを作成するインストーラー（ C：\ Program Files（x86）\ Steinberg \ VST3PluginTestHost \およびC：\ Program Files \ Steinberg \ VST3PluginTestHost \ ）が含まれ、 そこにVST3PluginTestHostファイルの必要なバージョンをコピーします.exe インストール中に、パラメーターを変更する必要はありません。



Windowsでは、実際の状態でプラグインのパフォーマンスを確認することもできます。 REAPERを再び使用します。 プラグインの場所を彼に伝える必要があります。 プロジェクトフォルダーには、サブフォルダー\ build-winがあります。これには、特に.dllおよび.vstプラグインファイルが含まれています。 REAPER オプション→設定→プラグイン→VST設定で、これらのファイルを含むディレクトリを追加できます。







次に、REAPERで、目的のプラグインのビルド（x86 / x64）を使用してプロジェクトを作成し、必要に応じて構成（たとえば、プラグインハンドルを特定の位置に配置）して保存します。 Visual Studioのメニュー[ プロジェクト ] →[プロパティ ]で、reaper.exeと構成済みの.rppへのフルパスを記述します。







一般に、 $(VST2_64_HOST_PATH)



およびその他の多くのパラメーターは、WDL-OLディレクトリーのcommon.propsファイルに登録されます。 設定で遊ぶ前にバックアップすることを忘れないでください。



デバッグモードでREAPERを使用しているときにreaper_host32.exeがクラッシュし始めた場合は、設定で[ネイティブのみ]を選択して、このプロセスをバイパスしてプラグインをロードするように設定してみてください。







reaper_host32.exeは引き続きクラッシュする可能性がありますが、それをバイパスしてロードされたプラグインの処理とサウンド出力の中断を停止します。



Macライブラリアラートロック





次にWindowsバージョンが登場します。



ご覧のとおり、次のような警告が表示されます。







警告は、コード内の考えられる問題やバグに注意を払うため、非常に便利です。 この場合、22個の警告は使用しているライブラリのコードに関連しているため、単に修正することはできません。このライブラリの最新バージョンをダウンロードした場合、すべての修正を再度行う必要があるためです。 しかし、そのままにしておくことも良くありません。プラグインのコードをすでに参照している23番目のエラーを簡単にスキップできます。



私の意見では、最善の解決策は、これらの警告をブロックして、新しい警告が表示されたときに、それがコードに関連していることを知ることです。 このために、 -w



コンパイラフラグを使用できます。 ナビゲーターで、青いプロジェクトアイコンをクリックし、 APPターゲットを選択し、 ビルドフェーズに移動して、 ソースのコンパイルセクションを開きます。 MyFirstPlugin.cppを除くすべての行を確認し、 Compiler Flags列をダブルクリックし、 -w



と入力してEnterを押します。







VST2、AU、およびその他の目的のターゲットに対して繰り返します。

アセンブリを再度ビルドします。 3つの警告が表示されます。 プロジェクト設定の検証は簡単に修正できます。 プロジェクト設定をダブルクリックし、すべてのボックスをオフにして[ 完了 ]をクリックします。







現在、2つの警告が残っています。







InfoPlist.stringsをエンコードする情報は、ファイルを開き、右側のユーティリティを使用してUTF-8に変換することで修正できます。







MyFirstPlugin.hに #include



ディレクティブを追加すると、余分なトークンの警告が削除されます 。 



 #pragma clang diagnostic push #pragma clang diagnostic ignored "-Wextra-tokens" #include "IPlug_include_in_plug_hdr.h" #pragma clang diagnostic pop




現在、APPおよびAUターゲットには警告はありません。 また、VST2をビルドすると、 メイン関数の宣言の1つのパラメーターに関する警告がポップアップ表示されます。 これを修正するには、 MyFirstPlugin.cppの#include



ディレクティブを補完します （.hではありません）。 



 #include "MyFirstPlugin.h" #pragma clang diagnostic push #pragma clang diagnostic ignored "-Wmain" #include "IPlug_include_in_plug_src.h" #pragma clang diagnostic pop #include "IControl.h" #include "resource.h"




アセンブリをビルドし、すべてがきれいであることを確認してください！



Windowsでのライブラリコードの警告のロック





一般に、VS C ++はMacのGCC / LLVM以外の警告を生成します。 すべてのVS C ++ 2010警告のリストは、 このページの左側にあります。



たとえば、WDLライブラリ内のファイルに関する警告C4101およびC4129が表示される場合、それらを無視できますMyFirstPlugin.h



#include



前に#pragma



追加し#pragma



。 



  #pragma warning( suppress : 4101 4129 ) #include "IPlug_include_in_plug_hdr.h"




Suppress



は、次の行でのみ警告が無視されることを意味します（つまり#include



）。 そのため、ファイル内の残りのコードに関する警告が表示されますが、これは問題ありません。 「C」なしで警告番号のみを書く必要があることに注意してください。 MyFirstPlugin.cppでも同じことができます。 



 #pragma warning( suppress : 4101 4129 ) #include "IPlug_include_in_plug_src.h"




これは、ライブラリの.hファイルに適用されます。

特定の.cppに関する警告が表示された場合は、プロジェクトエクスプローラーでそれを見つけ、左側のウィンドウで[ プロパティ ]を右クリックし、[ C / C ++]→[詳細]を選択します。 右側には、 特定の警告を無効にするオプションがあります 。







ここで、セミコロンで区切られた警告番号のリストを追加できます。 ご覧のとおり、デバッガーは既にこのファイルの4つの警告を無視しています。 これらの変更は、この特定の.cppファイルのみに影響することに注意してください。 したがって、残りのファイルの警告番号を指定し、適切な設定に追加する必要があります。

新しいプロジェクトを作成するときは、 複製スクリプトを使用して、これらすべての調整を再度行う必要がないようにします。
