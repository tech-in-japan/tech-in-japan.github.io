.NET APIを使用してAutoCAD用のプラグインを作成する（パート1-最初のステップ）

こんにちは、Habr！



AutoCADでの経験について話すことにしました。 多分これは誰かを助けるだろう-まあ、または少なくともそれは面白いように思えます。 



public static string disclaimer = "          AutoCAD.   –        .";




背景



すべてが非常に簡単に始まりました。再びお金の不足を感じたので、私はそれをどこかで探し始める時だと決めました。 そのため、 Freelansimで数週間検索した後 、開発者の空きが発見され、AutoCADとやり取りするプログラムが作成されました。



私はすぐに言わなければなりません：私はその日までAutoCADと通信する機会がありませんでした。 しかし、広告には「仕事の経験は必要ありません」というフレーズが含まれていて、それが私の魂を希望で満たしました。 空席を投稿した人に連絡し、テストタスクを取得しました。



テストのために、図面にいくつかのオブジェクトを作成し、テキストを表示することが提案されました。 数日間、APIに関する情報を検索し、珍しいプログラムで友達を作ろうとしました。 最後に、図が描かれ、テキストが表示され、検証のためにテストタスクが送信されました。 そして数日後、私は突然受け入れられたことがわかりました！ 奇跡、そしてのみ。



次の段落で-私の印象、あざやぶつかり、考えやアドバイス（有害かもしれない）。 開発はAutoCAD 2010の下で行われ、適切なVisual Studio 2013 ExpressがIDEとして使用されました。 開発言語はC＃です。

2.ライブラリプロジェクトの作成



最初の手順は、 ここで非常に明確に説明されています 。 英語を話す人もここに来ることができます 。 確かに、最後のリンクの資料では、Visual Basicに重点が置かれています。さらに、AutoCADのプラグインを作成するためのプロジェクトテンプレートである「AutoCAD .NETウィザード」をインストールする必要があります。 知識のある人々は、このテンプレートは人生を大幅に簡素化すると言います。 私はそれを使用したことがないので、控えめに黙ってください。



メインステージを簡単に複製します。



2.1。 クラスライブラリプロジェクトを作成する


プラグインがAutoCADの古いバージョンを対象としている場合、使用するプロジェクトプロパティで.NETバージョンをすぐに設定することをお勧めします。 たとえば、AutoCAD 2010は.NET Framework 4を使用して作成されたプラグインを読み込むことができないため、使用するバージョンとして.NET Framework 3.5を指定します。



プロジェクトで使用されている.NET Frameworkのバージョンをダウングレードすると、エラーメッセージが表示される場合があります。 私の場合、Visual Studioは「Microsoft.CSharp」アセンブリがないと誓っています。参照から除外するだけです。



2.2。 必要なAutoCAD .NET APIライブラリへのリンクを追加します


この時点で、もう少し詳しく説明する価値があります。 AutoCAD .NET APIには、さまざまなネームスペースで区切られたかなり多数のクラスが含まれています。 同様に、これらの名前空間は、いくつかのコンテナ（つまり、DLLファイル）にまたがって配置されます。



これらのdllファイルは、 inc- <architecture_name>という名前のフォルダーにあります。 したがって、私の場合、 C：\ ObjectARX 2010 \ inc-win32フォルダーからライブラリへのリンクを追加します。



注意：
自宅に32ビットOSを、顧客に64ビットOSをインストールしています。 これまでのところ、深刻な互換性の問題は発生していません。 しかし、それにもかかわらず、関数がInt32



返すという事実と、顧客のInt64



関数にInt64



。 リンカーは非常に動揺していました。 この機能に留意する必要があります。



APIを初めて知ったのは、Web全体にgeneしみなく散らばっているサンプルの少なくとも1つをコンパイルしようとする熱心な試みでした。 そして、 λλの典型的なものは 、頑固にコンパイルすることを望まず、未知の名前空間とクラスを誓った。 悲しみとともに最初のプロジェクトを構築しようとして、ObjectARXに付属するほぼすべてのDLLファイルを含めました。 悪い方法は、あなたがする必要がないことです。

多くのAutoCAD .NET APIクラスの名前は標準の.NETクラスの名前と一致するため、あまり便利ではありません。 たとえば、コードでApplication



クラスを見ると、Visual Studioはこの定義の曖昧さを誓います。その名前のクラスはSystem.Windows

名前空間にあります。 毎回フルネームを書かないために、ファイルの先頭に行を追加できます 



 using acadApp = Autodesk.AutoCAD.ApplicationServices.Application;

2.3。 プラグインコードを書く 


 using System.Windows.Forms; using Autodesk.AutoCAD.Runtime; namespace MyAutoCADDll { public class Commands : IExtensionApplication { //   (   ) public void Initialize() { MessageBox.Show("Hello!"); } // ,     public void Terminate() { MessageBox.Show("Goodbye!"); } //        AutoCAD  «TestCommand» [CommandMethod("TestCommand")] public void MyCommand() { MessageBox.Show("Habr!"); } } }


すべてが非常に簡単です。 まず、必要な名前空間を指定します。 2つ必要です。



最初の名前空間（ System.Windows.Forms



）には、 MessageBox



を表示するMessageBox



クラスの説明が格納されます。 使用可能にするには、同じ名前の.NETアセンブリへのリンクを追加する必要があります。



2番目のネームスペース（ Autodesk.AutoCAD.Runtime



）は、 IExtensionApplication



インターフェイスとCommandMethod



属性を定義します。 さらに、 IExtensionApplication



の説明はAcDBMgd.dllファイルにあり、 CommandMethod



説明はAcMgd.dllファイルにあるため、これらの両方のライブラリへのリンクを追加する必要があります。



したがって、合計で3つのリンクを追加する必要があります。









UPD。 2018年4月9日。 lasteranからの重要な追加：新しいバージョン（おそらくAutoCAD 2013以降）では、 CommandMethod



クラスはAcDBMgd.dllではなくAcCoreMgd.dllに含まれているため、このライブラリへのリンクも追加する必要があります。 新しいバージョンを使用するときは、このことに留意してください！



次に、 Commands



クラスを宣言します。 彼はプラグインの「出発点」になります。 このクラスはIExtensionApplication



インターフェイスから継承されているため、 Initialize



およびTerminate

注意：
AutoCADは、プラグインのロード後にプラグインをアンロードする機能を開発者に提供しません。 したがって、実際には、 Terminate



メソッドは、AutoCAD自体を閉じるときの1つの場合にのみ呼び出されます。

を使用して画面に表示し、ユーザーが入力したデータに基づいて、AutoCADで開いた図面を変更できます。



作成したメソッドをAutoCADコマンドに「変換」するには、 CommandMethod



属性をCommandMethod



ます。 その後の括弧内には、作成されたコマンドの名前が示されており、AutoCAD環境で直接使用できます。



このプロジェクトを組み立てた後、プラグインを使用できるようにします。
