デスクトップアプリケーションのホットキー、または同じボタンを使用してIDEでデバッグする方法から抽象化します。

多くのプログラムを使用する場合、さまざまなホットキーの組み合わせを使用することが、パフォーマンスとユーザーの利便性の鍵となります。 非常に複雑なプログラムの場合、さまざまなアクションの特定のキーの組み合わせを数十個記憶します。 これにより、必要なアイテムを検索するために複数階のメニュー上でマウスを移動するのではなく、実際の作業に集中できます。



したがって、ホットキーの組み合わせの理想的な記憶は筋肉です。 組み合わせが筋肉の記憶にある限り、作業中に対応するアクションの実行に注意を切り替える必要はありません-それは自動的に行われます。



これは良いことですが、このユーザー入力インターフェイスには問題があります。



それらを、ほとんどのhabr住民に馴染みのある環境-統合開発環境（IDE）で作業する例として見てみましょう。



これは、使用の多くの側面（コードの記述、コードのナビゲート、リファクタリング、コンパイルと実行、デバッグ、バージョン管理システムの操作など）を備えた複雑なソフトウェアシステムです。



これらの各側面には、他の側面では使用されない独自のホットキーのセットがあります（コードを記述するときに、デバッガーでローカル変数を表示するためにウィンドウを開くキーの組み合わせを覚える必要はありません）。



最初の問題-不規則に使用される組み合わせは忘れられます



最初の問題は、何らかのアクションが不規則に行われた場合、対応するキーの組み合わせが次回使用するときに忘れられる可能性が高いことです。 さらに、この組み合わせが再び必要になった瞬間に、一度か二度ではなく、むしろ積極的に使用されることが判明するかもしれません。



例：ソフトウェアシステムの大規模なリファクタリングを行うことにしたとしましょう。



実行する必要がある主なアクションは、コードに対してさまざまな半自動化された変更を行うことです（文字の名前変更、コードの個々の機能への抽出、機能の署名の変更など）。 さらに、プロジェクトのある場所から別の場所にジャンプしたり、さまざまなクラス図を調べ​​たり、階層を呼び出したりする必要があります。



この一連のアクションは、通常のコード記述で使用されるものとはまったく異なる可能性があります。したがって、大きなリファクタリングが不規則に行われるため、そのようなアクションのホットキーの組み合わせは時間の経過とともに洗い流されます。



この問題に対処するために、一部のプログラマーは、まれに役立つコマンドのリストを使用してさまざまな虎の巻を作成します。 そのような場合は、チートシート（またはgoogle）を参照する必要があります。



2番目の問題は、システム間の移行の複雑さです。



何らかの理由でIDEを変更する必要がある場合、筋肉のメモリレベルで学習した古い開発環境のホットキーが深刻な障壁になる可能性があります。 再学習する必要がありますが、これは非常に快適で非生産的な職業ではありません。



さらに、新しいシステムの古い組み合わせはまったく異なる意味を持つ可能性があり、特に面倒です。



3番目の問題は、複数のシステムを共有することです。



ここではすべてが非常に悪いことがあります。 異なるシステムでの同じアクションは、異なるホットキーの組み合わせを持つことができます。 この場合、各システムの組み合わせのセットに常に留意する必要があります。 さらに、現在使用されているシステムの種類を常に精神的に制御する必要があります。 ここで重要なのは、「心に留めておく」ことです。筋肉の記憶を忘れることができます。これはパフォーマンスの低下をもたらします。



サーバーがVisual StudioのC ++で記述され、クライアントがIDEAまたはEclipseのJavaアプリケーションであるクライアントサーバーシステムをデバッグしているとします。



サーバーはデータを使用してパケットを形成し、それをクライアントに送信し、クライアントが何らかの方法で処理します。 まず、サーバー側が送信用のパケットを正しく生成するかどうかを把握する必要があります。 次に、サーバーがデータを送信した後、受信したデータがどのように処理されるかを確認するために、クライアント側のデバッガーで必要になります。 さらに、サーバーとクライアント間の通信は、1つのパケットの送信に限定されない場合があります。 この場合、デバッガー間を複数回ジャンプする必要があります。 この場合、ユーザーが実行する主なアクションは、プログラムを段階的に実行するためのコマンドです（ステップオーバー、ステップイン、ステップアウト）。 概念的には、これらのアクションは同じですが、各開発環境では、独自のファンクションキーがそれらを担当します。

VS 日食 アイデア
踏み込む F11 F5 F7
乗り越える F10 F6 F8
出る Shift + F11 F7 Shift + F8


同時に、たとえば、Eclipseの「ステップイン」はF5キーを使用して行われます。これは、VSで「実行の再開」を担当し、不注意によってデバッグプロセス全体を停止させる可能性があります。



そのような状況では、マウスだけでデバッグを開始していることに気づくことがよくありました。はるかに低速でしたが、現在実行しているアクションと環境を正確に把握していました。



目的



キーボード入力をエミュレートするシステムを作成し、ホットキーの組み合わせを使用して開始される多数のアクションセットを持つプログラムの使用を簡素化します。



システム要件：



  • ユーザーがホットキーの組み合わせの実際の値を覚えておく必要はありません。
  • アプリケーションへのユーザーコマンドの送信のしやすさ。 インターフェイスは、コマンドを簡単かつ迅速に見つけて実行できるようにコマンドを編成できるようにする必要があります。
  • システム構成の容易さ-ユーザーは、自分のニーズに合わせてインターフェイスを調整し、特定の時点で関連するコマンドのみを制限できる必要があります。
  • 同様のアプリケーション間でコマンド構成を簡単に再利用する機能。
  • コマンドの実行時にコマンドの宛先ウィンドウがアクティブであることを自動的に制御する機能（上記の問題（EclipseのF5キーおよびVisual Studioデバッガーなど）を回避するため）。
  • クロスプラットフォームアプリケーション-ホットキーの実際の組み合わせだけでなく、オペレーティングシステムからも抽象化したかった。


このシステムの主なアイデアは、コンピューターでサーバーを起動し、キーボードコマンドを生成することです。 彼はバックグラウンドで作業し、その時点でアクティブなウィンドウにコマンドを送信します。 ユーザーの入力/出力に使用されるAndroidデバイスがこのサーバーに接続されています。 すべてのコマンドは、Androidデバイスに表示されるボタンをクリックすることで開始されます。 これが発生すると、サーバーは適切なキーの組み合わせをその側のアクティブウィンドウに送信します。



このアクションに対応するキーの組み合わせのマッチングはサーバー上で自動的に行われるため、このマッピングのいくつかのルールを決定することができます。 これにより、同じ方法で異なるプログラムを操作できます（実行されるアクションのセットが類似している場合）。



サーバー構成



このシステムが機能するには、次の情報が必要です。



  • 作業環境、およびキーボードショートカットをエミュレートする環境に関するデータ。
  • キーボードショートカットデータ。 各アクションでは、各環境でそれを開始するホットキーの組み合わせを指定する必要があります。
  • クライアントデバイスの構成アイテム：コマンドの送信に使用されるボタンの場所、外観、および動作。

その中で、各行は特定のチームに関するすべての情報を表します。 最初の4列はオーバーヘッドです。 ユーザーインターフェイスを構築するためにプログラムで使用されるデータを保存します。

当然、このアプローチはホットキーだけでなく使用できます。 キーボードから頻繁に繰り返されるアクションは、このプログラムを使用して自動化できます。

別の言語の単語またはフレーズを挿入する



多くの場合、ロシア語のテキストを入力するとき、1つの単語、または2、3の文字のために、英語のレイアウトに切り替える必要があります。 ロシア語のレイアウトは、次の文字の存在を認識しません： '<> [] {} $＃@＆^〜、これらのいくつかは定期的に必要です。 さらに、ロシア語のテキストには、英語で繰り返される用語がある場合があります。また、htmlの場合は、タグもあります。



挿入されたテキストの実際の入力の前後にキーボードレイアウトを切り替えるコマンドを追加するだけで、異なるレイアウトでの入力のシミュレーションを実装できます。



たとえば、<html>タグを印刷するには、「+ {COMMA} html + {PERIOD}」の組み合わせを使用する必要があります。 英語のレイアウトが有効になっている場合、正常に動作します。 ロシア語のレイアウトがオンになっている場合、レイアウトを組み合わせの最初と最後に切り替えるためのホットキーを追加する必要があります。 私の場合、これは '％{SHIFT}'（Alt + Shift）です。 したがって、テキストエディターでロシア語のキーボードレイアウトが有効になっている場合、次の文字シーケンス： '％{SHIFT} + {COMMA} html + {PERIOD}％{SHIFT}'が<html>を印刷します。

選択した単語のインプレース変換



コード内の多くの場所でマクロまたは関数呼び出しのいくつかの変数をラップする必要があるとしましょう。 たとえば、関数の1つはパラメータの型を変更する必要があるため、この関数が呼び出されるすべての場所で、そこに渡されたものを新しい型に変換する必要があります。



そのような変更を少なくとも数十個行う必要がある場合、これを自動化するというアイデアは非常に魅力的です。



この場合に実行する手順は次のとおりです。



  1. ダブルクリックして変数を選択します
  2. 「カット」コマンドのシミュレーション（Ctrl + x）
  3. タイピングをシミュレートします-ラッピング関数の名前と括弧
  4. 貼り付けコマンドのシミュレーション（Ctrl + v）
  5. ブラケットを閉じます

おわりに



システムはまだ未加工です-それはむしろプロトタイプであり、最終バージョンではありません。 しかし、私にとっては、すでに現在の状態にあるように思えるので、ホットキーを使用した作業を簡素化し、効率を高めることができます。



この記事では、実装の技術的な詳細については触れませんでした。 興味があれば、これに専念する記事を作成することが可能になります。



実装コードはgithubにあります。 構成ファイルの形式とコマンドライン引数に関するもう少し正式なドキュメントもあります。



ソースコードをダウンロードしてコードをコンパイルしたくない場合は、 Windows用にコンパイルされたバージョンがあります （実行するには、 Microsoft Visual C ++ 2015 Redistributableをインストールする必要があります）。



ソースを自分でコンパイルするには、C ++ 14-互換性のあるコンパイラとブーストが必要です（boost :: asioおよびboost :: program_optionsが使用されます）。 ドキュメントには、gccとVisual Studioを使用してプロジェクトをビルドする方法の説明が含まれています。



実装に対する批判と、システムに追加できるものについての提案を聞いてうれしいです。 また、現時点では、UIの説明の構成ファイルの形式があまり好きではありません。 そのようなインターフェイスを（できればプレーンテキスト形式で）最適に表示する方法についてアイデアをお持ちの方は、コメントしてください。



このシステムに他に何を追加しますか：



  • 1つの要求で複数のコマンドを実行する機能（これにより、入力をシミュレートする機能が大幅に向上します）。
  • アクティブなウィンドウのキーボードレイアウトを自動的に追跡および切り替えます。
  • 構成ファイルを操作する機能を改善し、新しいタイプの構成を追加します。
  • プログラムコードのリファクタリング。
  • 構成ファイルの問題に関するユーザーへのメッセージのシステムを改善します（現在、メッセージは常に情報を提供するとは限りません）。
  • いくつかのキーの組み合わせをシミュレートするバグ修正（以下を参照）。

PS



使用する前に知っておくべき問題と非自明性：



  1. UTF-8エンコードのみがサポートされています。 プログラムが別のエンコーディングで設定をフィードする場合、かなり明白なメッセージによって呪われます。 これを考慮して、正しいエンコーディングでファイルを保存してください。



  2. Windowsでは、Shift + 'smth'形式のいくつかの組み合わせのシミュレーションに問題がありました。 PgUp、PgDn、Home、End、Insert、Delete、および矢印キーに対して表示されます。 Shift + Insertをシミュレートするタイプ '+ {INSERT}'の組み合わせは、実際にはInsertを押すだけをシミュレートします。



  3. 構成例では、多くのホットキーの組み合わせの機能がテストされていません。 いくつかのリストをグーグルで検索し、それらを共通ファイルにマージしました。 したがって、エラーが発生する可能性があります。 プルリクエストだけでなく、設定で見つかった問題の報告にも感謝します。
