Hans Boehm：C ++ 11のスレッドとシェア変数


最後に、C ++では、マルチスレッドが言語標準になりました。 スレッドの設計は、よく知られているboost.threadライブラリの設計にほぼ従っています。 非同期呼び出しの非同期/将来のサポートについて言及しました。 ミューテックス/ロックの使用例があります。

ステファン・T・ラヴァヴェージ：STL11-Magic && Secrets


STLは、標準C ++ライブラリの多くの革新について語っています。

  • 参照カウンター付きのポインター。 テクニック「タイプ消去」。 make_sharedを使用してshared_ptrを作成する方が良い理由 

    auto sp1 = make_shared<T>( args ); // 1 allocation, 24 bytes overhead shared_ptr<T> sp2( new T( args ) ); // 2 allocations, 40 bytes overhead


  • 配置：コンテナ内にオブジェクトを直接構築します 

     v.emplace_back("Carmichael", 3 * 11 * 17); // more efficient, easier to type v.push_back( make_pair("Carmichael", 3 * 11 * 17) );


  • 新しい標準がstd ::ペアをほぼ「破った」方法、またはNULLではなくnullptrを使用する価値がある理由 

     pair<X*, double> p1(NULL, 3.14); //  -  ,    pair<X*, double> p2(nullptr, 3.14);


  • Futures / async、そしてなぜ重いオブジェクトを参照ではなく値で渡すほうがよいのか
  • 範囲ベースのforはVC11に表示され、すでにVC11 Betaに含まれています。 実装方法、およびautoと組み合わせて使用​​する方が良い理由
  • auto：記述しやすいだけでなく、より効率的なコードが得られる場合があります 

     map<string, int> m; //  ? for (const auto& p1 : m) { … } for (const pair<string, int>& p2 : m) { … }




Andrei Alexandrescu：可変長テンプレートはFunadicです


「...結局のところ、彼は誰であるべきかということは素晴らしいことをした-恐ろしい、はい、しかし素晴らしい。」


テンプレートメタプログラミングの第一人者のプレゼンテーションでは、Variadic Templatesについて話しています。可変数のパラメーターでパラメーター化されたテンプレートクラス/関数を作成できるC ++ 11標準の新機能です。 たとえば、Cからprintf / scanf関数のタイプセーフなアナログを記述できるようになりました。ほとんどのC ++プログラマーは、Variadicテンプレートを使用して自分で何かを記述する必要はほとんどありませんが、この機能はライブラリ作成者にとって非常に重要です。 プレゼンテーションは、使用されている構文を見て、Variadicテンプレートの使用方法を理解したい人にとって興味深いものになります。



パフォーマンスには3つの部分が含まれます。

  • Variadicテンプレートが何であるか、なぜ必要なのか、構文が説明され、渡された最初の引数が次のいずれかと等しいかどうかを決定する関数の例が示されます 

     isOneOf(1, 1); // => true isOneOf(1, 2, 6); // => false isOneOf(1, 3.5, 4, 1, 2); // => true


  • Cからのprintf関数の単純なアドオンが実装され、渡されたタイプとフォーマット文字列の対応をチェックします。 

     safe_printf( “%d %f”, 12.1, 1.1 ); // => throws invalid format exception safe_printf( “%d”, 1, 2, 3, 4, 5 ); // => throws too few format specifiers exception safe_printf( “%d %f”, 11, 12.1 ); // => ok, redirects to printf(…)


  • std :: tupleクラスの可能な実装が示されています（C ++ 11標準から） 

     tuple<int, string, double> t; get<0>(t) = 42; assert(get<0>(t) == 42); get<0>(t) = “this will not compile”; // => compilation error get<1>(t) = "forty-two"; get<2>(t) = 0.42;

ハーブサッター：C ++ 11、VC ++ 11以降


Sutterは、VC11の次のバージョンでC ++ 11標準ライブラリを完全にサポートすることを約束します。また、（喜ばしいことに）今月予定のベータバージョンでは、範囲指定および最終/オーバーライドのサポートが表示されます。 VC11のリリース後すぐにいくつかの「帯域外」リリース（Visual Studioリリースに関連付けられていない）をリリースし、新しい標準のサポートを徐々に改善していく予定です：初期化リスト、テンプレートエイリアス、可変テンプレート、constexpr、noexcept、= default / delete、...はい調査に参加する機会。新しい標準のどの機能が最も重要であり、できるだけ早くそれを見てみたい： bit.ly/mscpp11 。



2番目の部分は、C ++でのプログラミングの新しいスタイル、イディオム、および推奨される方法に専念します。 Sutterは、C ++のすべての本は標準のリリースで時代遅れになり、本の各例は新しいスタイルで書き直すことができるし、そうするべきだと主張しています：地球上のすべての本を壊し、地球上のすべてのプログラマを壊しました。 C ++ 11に関する本が少なくとも1つ既に存在しています： bit.ly/meyers11 。 C ++に関する他の有名な本の更新予定日：

  • 実効C ++（マイヤーズ）-2013-2014
  • C ++コーディング標準（Sutter、Alexandrescu）-2015


3番目の部分では、C ++標準の可能な開発計画について説明します。 彼の意見では、スタイルのこのような急進的な変更、およびC ++ 11のようなイディオムは、近い将来望ましくありません。 スピーチの最後に、紋章はC ++の最も弱い側面を効果的に説明し、状況を修正するために何を計画しているかを伝えました。



Chandler Carruth：Clang-マーフィーのミリオンモンキーズからC ++を守る


Clang開発者は、ライブgccで別のC ++コンパイラを記述する必要がある理由について説明します。

  • 悪いエラーメッセージ
  • コンパイル速度が遅い
  • gccの技術的特徴。コード分析ツールの構築には適さない
  • GCCの政治的制限により、コード分析ツールの構築には適さない


最新バージョンは、C ++ 98標準、および部分的にC ++ 11を完全にサポートしています。 Linux、Macで動作します。 開発者の中には十分なWindows開発者がいないため、Windowsのサポートはありません。



問題のあるC ++コードの多くの例と、コンパイラによって生成されるエラーメッセージと警告。 興味深い静的コードアナライザーが判明したように見えますが、実行時の動作を分析する機会もあります。 コンパイラに基づいて、C ++コードをリファクタリングおよび分析するための非常に興味深いツールが開発されています。 たとえば、 不要なincludeを削除するツール 。



アンドレイ・アレクサンドレスク：ハンマーがあったら静的


D言語で証明されているstatic_if構文を言語の標準に追加する提案について話しているので、今では震えなしでは見られないテンプレートコードを大幅に簡素化できます。 その使用方法は次のとおりです。 

 //     It – forward iterator template <class It> It rotate(It b, It m, It e) if (is_forward_iterator<It>::value) {...} //     WithParent       enum class WithParent { no, yes }; template <class T, WithParent wp> class Tree { class Node { ... }; static if (wp == WithParent::yes) { Node * parent_; } Node * left_, * right_; T payload_; }; Tree<string, WithParent::yes> dictionary;




Bjarne StroustrupとAndrew Sutton：C ++のコンセプトデザイン


コンセプトは、C ++ 11言語標準に含まれていない長期にわたる機能であり、採用が大幅に遅れました。 Straustrupは言語の概念をどのように理解しているかを語り、Suttonは技術的な詳細について説明します。 私が理解しているように、言語に概念を追加したいという要望がまだあり、提案は標準化委員会に提出されます。 提案は、オリジナルと比較して簡素化の方向に真剣に作り直されました。 これは概念を使用するコードのように見えるかもしれません：
