Android O：新しいオペレーティングシステムのサポート機能

みなさんこんにちは！ 重要なイベントがまもなく行われます-Android Oのリリース。オペレーティングシステムの新しいバージョンのサポートは、深刻な製品の責任です。 Androidを更新するたびに、多くの開発者は既存の機能の機能を維持し、Androidの新しいバージョンに新しい機能を追加するために一生懸命働く必要があります。







この記事では、Android Oの主な変更点を確認し、それらの影響の可能性を評価します。



















Android Oでの作業中、Google開発者はまずAndroidデバイスでの高速バッテリー消費の問題を解決したため、主な変更はバックグラウンドタスクの最適化とアプリケーションリソースの消費に関連しています。

バックグラウンドロケーションの制限



これは、バックグラウンドでのアプリの位置情報リクエストの数の制限です。 そのため、Android Oから開始し、インストールされているTargetSDKに関係なく、アプリケーションは1時間あたりわずかな場所の更新のみを受信できます。 同時に、これらの制限は、次のパラメーターの少なくとも1つが有効なアプリケーションには適用されません。







  • アプリケーションはバックグラウンドではありません（開始状態または一時停止状態のアクティビティが実行されています）。
  • アプリケーションにはアクティブなフォアグラウンドサービスがあります。
  • 別のForegroundアプリケーションには、現在のアプリケーションへの接続があります（バインドサービスとコンテンツプロバイダーの両方を使用）。


したがって、フォアグラウンドアプリケーションの場合、動作は以前のバージョンのAndroidと同じになります。







フォアグラウンドサービスがない場合、ジオロケーションリクエストが変更されます。







  • 融合ロケーションプロバイダー（FLP）

    • バックグラウンドアプリケーションの場合、アプリケーションが明示的に追加の更新を要求した場合でも、システムは1時間に数回だけ新しい場所を決定します
    • フォアグラウンドアプリケーションの場合、作業のロジックは変更されません。
  • ジオフェンシング

    • バックグラウンドアプリケーションは、Fused Location Providerからの更新よりも頻繁にgeofencing-apiから更新を受信できます。
  • GNSS測定とGNSSナビゲーションメッセージ

    • バックグラウンドのアプリケーションの場合、GnssMeasurementおよびGnssNavigationMessageサービスの更新は提供されません
  • ロケーションマネージャー

    • バックグラウンドアプリケーションの場合、システムは新しい場所を1時間に数回しか決定しません。


バックグラウンド実行の制限



アプリケーションのバックグラウンド操作の制限は、Android Oの重要な変更です。これは、targetSdk "O"に切り替えたときにのみ顕著になります。 targetSdk <= 25の場合、アプリケーションがバックグラウンドにあり、システムがCached状態に移行した場合（システムがいつでもプロセスを自由に強制終了できる場合）、同時にアクティブなコンポーネントがない場合、システムはすべてのWakeLock（インジケーターアプリケーションがこのアプリケーションのスリープ状態になってはいけないということです。 アプリケーションがtargetSDK <Oでない場合、設定で動作ポリシーを設定して、アプリケーションがtargetSDK Oを使用する場合と同じになるように注意することが重要です。







targetSdkを使用したアプリケーションの場合、「O」バックグラウンド実行制限は2つのカテゴリで構成されます。







  1. バックグラウンドサービスを制限する

    アプリケーションがバックグラウンドに切り替わった後、数分で「ウィンドウ」が表示され、その間にサービスを開始して動作させることができます。 この間隔が経過すると、すべてのサービスが停止し、新しいサービスを起動するとアプリケーションがクラッシュします。 Googleは、実行中のバックグラウンドサービスの数を最小限に抑えようとしており、JobSchedulerとGcmNetworkManagerの使用を提案しています。

    バックグラウンドで長いタスクを実行する必要がある場合は、アプリケーションが動作していることをユーザーに明確に伝えるforegroundServiceを使用することをお勧めします。

    Context.startForegroundService()



    呼び出された後、 Context.startForegroundService()



    が5秒以内に呼び出される必要があることに注意してください。そうしないと、システムがANRを表示する場合があります。
  2. マニフェストにブロードキャストを登録する際の制限

    Androidの新しいバージョンは、Android 7.0で開始された作業を継続します（モバイルステーションが変更されるたびにリリフトされたアプリケーションを誰もが忘れないように）。 しかし、第7バージョンでマニフェストに少数のブロードキャストのみを登録することが不可能だった場合、今ではブロードキャストの大部分が存在します。


通知チャンネル



通知チャネル-通知をテーマ別グループにグループ化するツール。ユーザーは直接管理できます。 アプリケーションがtarget> = Oでビルドされている場合、少なくとも1つの通知チャネルをサポートする必要があります。 targetApi <Oの場合、製品内での通知の処理は同じままです。







ウィンドウマネージャー



Android Oでは、他のウィンドウの上に表示できる新しいタイプのウィンドウ（targetSDK O用）が導入されました-TYPE_APPLICATION_OVERLAY。 同時に、いくつかの古いウィンドウタイプは非推奨になり、それらが使用されるとRuntimeExceptionがスローされます。

現在、これらのタイプのウィンドウは、システムアプリケーションでのみ使用できます。







  • TYPE_PHONE
  • TYPE_PRIORITY_PHONE
  • TYPE_SYSTEM_ALERT
  • TYPE_SYSTEM_OVERLAY
  • TYPE_SYSTEM_ERROR


プライバシー（Build.Serial / net.dns / etc）



Android Oでは、ユーザーが識別子へのアクセスを制御できるように設計されたいくつかの機能強化が導入されています。 これらの改善点は次のとおりです。







  • 永続的な（リセット不可能な）デバイス依存識別子の使用に関する制限
  • ネットワークスキャン中にMACアドレスをランダム化するためのPixel、Pixel XL、Nexus 5xなどのデバイス上のWi-Fiチップセットファームウェアの変更に関連するスタックのWi-Fiシステム更新
  • アプリケーションがアカウント情報を要求し、ユーザーデータをより強力に制御できるようにするメカニズムの更新


AccountManager.getAccounts（）



Android O以降、アカウントのリストにアクセスするにはGET_ACCOUNTS権限では不十分です。 現在、targetSDK Oを使用するアプリケーションには2つのオプションがあります。







  • AccountManager.newChooseAccountIntent()



    使用します。
  • オーセンティケーターによって定義されたメソッドを使用して、アカウントにアクセスします。


Android Enterprise



GoogleはAndroid for Workを積極的に開発しています。 Android Oでは、デバイスをより詳細に制御できます。 たとえば、作業プロファイルの場合、設定で個別のロックを開始し、利用可能なシステムアップデートに関する情報を確認できます。







オートフィルフレームワーク



Autofill Framework APIの出現により、Accessabilityを使用する場合よりもユーザーデータをアプリケーションに簡単に入力できるようになりました。 フレームワークを呼び出すときは、オートコンプリートの呼び出しが発生した製品のpackageNameを照合し、入力用のデータを提供する必要があります。 ログインフィールドとパスワードフィールドを正しく識別するには、autoIdのUI要素を正しく識別できるようにするresourceIdコントロールまたはその他のサービス情報を使用してデータベースを作成および維持する必要があります。







ピクチャインピクチャ



Android Oでは、特殊なタイプのマルチウィンドウモードであるPicture-In-Pictureモードでアクティビティを実行できます。 ビデオを表示するアプリケーション専用にこのモードを使用することをお勧めします。







適応アイコン



Android Oからは、アプリケーション用のランチャーアイコンの別のバージョンを提供できます。これは、システムがデバイスメーカーが使用する形式にマスクで簡単に切り取ることができます。 また、そのようなアイコンがある場合、システムはアプリケーションアイコンのアニメーションを作成できます。







ショートカットとウィジェットの固定



Android Oの新機能は、ランチャーでのプログラムによるショートカットとウィジェットの作成です。 ショートカットは、タスクをクリックして特定の意図でアプリケーションのタスクを実行できる個別のアイコンです。 ウィジェットまたはショートカットを正常に接続するには、システム確認ダイアログでユーザーの同意が必要です。







おわりに



Androidで必要なことを何でもできるようになったのは過去のことです。 Androidは、新しい機能を作成するための「合法的な」ツールを拡張し、ハッキングのギャップを埋めて、正直に機能させます。 Android Oは既存のアプリケーションの一部を破壊しますが、エンドユーザーの満足度を高めようとします。 やった、ありがとう！

誰もが良いアプリケーションを作成し、ユーザーについて考え、Androidデバイスの幸せな所有者になることを望みます。
