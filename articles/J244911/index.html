妄想パート1のWordPress

だから、もしあなたがnginxの幸せな所有者であり、高貴な偏執狂者であり、ある意味地獄でワードプレスを置くことに決めたなら...最初に思い浮かんだのは、「この創造物を自由に制限する必要がある！」



誰もが独自のゴキブリを持ち、誰かが一般的にApacheを実行するため、アカウント設定とphp5-fpm設定を省略します。 ただし、ここでは、WordPressの一般的な部分について説明します。 私が何をしたか、何が起こったのか、そしてその理由について書きます。

これはWordpress 4.0の典型的なスイートです。



何が必要ですか？ phpファイルと管理パネルへのアクセスを制限し、静的をレンダリングし、xmlrpcを閉じる必要があります。



管理パネルとphpファイルへのアクセスを制限します



私のバージョンのwordpressでは、ユーザーのコメントを保存せず、xmlrpcを使用しません。 コメントへのアクセスを安全に提供する方法、およびnginxとwordpressのその他の緊急の問題については、この記事の第2部で説明します。もちろん、もしあれば作成します。 Apacheがないため、.htaccessファイルは役に立ちません。

したがって、上記の内部告発者を閉じます。 



場所〜* ^ /（\。htaccess | xmlrpc \ .php）$ {
  404を返します。
 }




その後、xmlrpc.phpおよび.htaccessリクエストを使用すると、404エラーが発生します。 403と200の両方のトロロロを発行することは可能ですが、これはすでに好みの問題です。



次に、残りのものへのアクセスを制限します。 制限により、私は承認リクエスト、つまりauth_basicを意味します。 



場所〜* ^ / wp-admin /（。*（？<！（\。php）））$ {
    auth_basic「パスワードで保護」;
    auth_basic_user_file users / somefile;
    root / path / to / site / root;
    ＃その他のオプション
 }




*このコードは、/ wp-admin /から静的を要求するときにnginxに承認を要求させ、nginxは静的を発行します。



次に、ファイルへのアクセスを制限します。 



場所〜*（/ wp-admin / | / wp-cron \ .php | / wp-config \ .php | / wp-config-sample \ .php | / wp-mail \ .php | / wp-settings \。 php | / wp-signup \ .php | / wp-trackback \ .php | / wp-activate \ .php | / wp-links-opml \ .php | / wp-load \ .php | / wp-comments-post \ .php | / wp-blog-header \ .php | / wp-login \ .php | / wp-includes /.*？\。php | / wp-content /.*？\。php）{
    auth_basic「パスワードで保護」;
    auth_basic_user_file users / somefile;
    root / path / to / site / root;
    ＃その他のオプション
 }




/wp-includes/.*?\.phpのようなエントリには、wp-includes以下のすべてのphpファイルが含まれます。



完了しました。アクセスをクローズしました。 ここで、公開バージョンに必要な要素を選択的に含めます。これは本文でさらに詳しく説明します。



安全なWordpressで安全な投稿を有効にする



wp-login.phpを承認して閉じた後、安全な投稿を書き込み、リンクとパスワードを（投稿から）適切なユーザーにドロップすることにより、ユーザー...は未知のウィンドウを怖がります。 パスワードは、GETパラメーターを使用したpostリクエストとしてwp-login.phpファイルに渡されるので、 Action = postpass 。



nginxは多くの制限を課しています：

  • nginxからの場所では、リクエストパラメータを記述できません。
  • auth_basicはif式では使用できません。
  • 承認に成功した場合に1が渡されるconfig内の変数を持つゲームは、変数が現在のリクエストでのみ有効であるため、何ももたらしません。


どうする？



解決策があります！ 同じフォルダーにwp-login.phpへのシンボリックリンクを作成します。 このwp-postpass.phpがあります。 シンボリックリンクが必要なので、wordpressを更新すると、 wp-login.phpも更新され、リンクによってファイルが更新されます...それがLinuxが大好きな理由です。



次に、nginx configに次のように記述します。 



場所〜*（/wp-postpass\.php）{
    if（$ args〜 "^ action = postpass $"）{
       set $ wppostpass 1;
    }
    if（$ wppostpass〜0）{
       403を返します。
    }
    ＃その他のオプション
 }




この場合、 / wp-postpass.php？action = postpassを要求すると、wppostpass変数の値は1になり、場所は最後まで機能します。 ベアリクエストwp-postpass.phpまたは他のパラメーター（ここでわかるように、$行の先頭から^までチェックされます）の場合、403エラーが発生します。これは、アクセスが閉じられたことを意味します。



そのようなスキームが機能するためには、 ngx_http_substitutions_filter_moduleが必要です 。 構成に登録する必要があります 



 subs_filter 'https://example.com/wp-login.php\?action=postpass' 'https://example.com/wp-postpass.php?action=postpass' gi;




次に、nginxはwp-login.php？Action-postpassリンクをwp-postpass.php？Action-postpassに自動的に変更し、ユーザーはパスワードでログインして保護されたレコードを表示できます。



別のサーバーで静的データを取り出し、CDNを接続します



ロード中、js、css、および小さなgifは役割を果たしません。メモリがある場合、nginxはキャッシュにそれらを保存し、十分なメモリがある場合、すべてのサイト統計をtmpfsセクションに転送できるためです（3.8 GB読み取りおよび書き込み745k iops ' s）。



ただし、1つのサーバーの場合、誰かがファイルを早く取得し、後で誰かが取得します。多くのクライアントがいる場合、1MBのファイル1000個を配信するとき、rateを入力しないとチャネルはたるみます。



これらの場合、キャッシングCDNプロバイダーが発明されました。 たとえば、 cloudflareです。



仕事の原理は、彼らの写真に顕著に示されています：







CDNを使用しない場合、すべての要求は最終サイトに送られ、CDN要求からはプロバイダーのCDNに送られます。これは中間リンクとして機能します。 この場合、1000人のユーザーが1 MBのファイルを要求すると、このファイルはCDNプロバイダーによってキャッシュに対して1回要求され、その後1000人のユーザーが配布されます。 Àla google docsスタイルのDDoSオプションで、 big_photo.jpg ？ Ver = 1 、次にbig_photo.jpg ？ Ver = 2などを要求した場合 中程度のキャッシュモードが選択され（cloudflareにある）、キャッシュのみが静的である場合は機能しません。その後、 big_photo.jpg 、 big_photo.jpg？ver = 1またはbig_photo.jpg？ver = 123を要求すると、サーバーからbig_photo.jpgが要求されます。クライアントが引数付きのファイルを要求した場合でも、彼と彼だけです（それらは単に無視されます）。 これにより、基本的にddosからも保護する必要があるcdnプロバイダーによるddos問題が解決されます。



私はあまり登りませんでしたが、デフォルトの統計は次の場所に保存されていることがわかりました。

  • / wp-content /アップロード/

  • / wp-includes / js /
  • / wp-includes / css /
  • / wp-includes /証明書/

したがって、それらのために場所に新しいルールを作成し、 ngxとngx_http_substitutions_filter_moduleを使用します 。

このモジュールをインストールする必要はありません。書き換えモジュールのみで実行できますが、それだけで便利であり、それを通じてバックエンドからの出力を改善できます。



構成に以下を追加します。 



	 subs_filter_types text / html;
	 subs_filter_types text / xml;




htmlおよびxmlドキュメントの出力をフィルタリングします。



次に：

場所〜* ^ / wp-includes / js /（。*（？<！（\。php）））$ {
	書き換え^ / wp-includes /(.*)$ https://static.example.com/$1 permanent;
 }

場所〜* ^ / wp-includes / css /（。*（？<！（\。php）））$ {
	書き換え^ / wp-includes /(.*)$ https://static.example.com/$1 permanent;

場所〜* ^ / wp-includes / certificates /（。*（？<！（\。php）））$ {

場所〜* ^ / wp-includes / fonts /（。*（？<！（\。php）））$ {

その結果、phpファイルをリクエストしても何も起こりません。 そして、静的（静的なWPの場合はphpではないすべて）を要求すると、ユーザーは静的サーバーにリダイレクトされます。

ルート/パス/へ/サイト/ダミー;

リクエストのフィルタリング



一度に2つのケースがあります。

  1. メディアファイルをダウンロードすると、 example.com /？attachment_id = XXのようなリンクを取得します。XXはこのメディアファイルのページIDです。 それに応じて、1、2、3の順に並べ替えます...ユーザーは、すべてのコンテンツと、そのコンテンツが自分に向けられていない部分を送り出すことができます。
  2. phpにはびらんがいっぱいです。 おそらく、これは言語のアーキテクチャではなく、プログラマーのスキルや、この作成が展開する環境の設定ではありません。 しかし、ワードプレスを配置したら、将来のバグに備えます。


これを行うには、nginxの構成のサーバー{}にコードを記述します 。 



 if（$ args〜* "（attachment_id | eval | duplicate | base64 | substring | preg_replace | create_function）"）{
	 403を返します。
 }




次に、リクエストの引数にattachment_id、eval、duplicate、base64、substring、preg_replace、create_functionが含まれる場合、nginxは403エラーを返し、リクエストは潜在的な脆弱性を実行するためにダイナミクスに渡されません。



nginxのsubs_filterを介したパン



このモジュールの目的はここで見直されました 。



タスク：デフォルトでは、wordpressは現在のウィンドウでメディアファイルへのリンクを開きます。 そして、あなたは新しい必要があります。



解決策： nginx kogfigに小さなコードを追加します。 



 subs_filter '<a href=\'https://static.example.com/uploads/(.*?)\'>' '<a href = \' https：//static.example.com/uploads/$1 \ 'target = \' _ blank \ '>' gi;
 subs_filter '<a href=\"https://static.example.com/uploads/(.*?)\">' '<a href = \ "https：//static.example.com/uploads/$1 \ "target = \" _ blank \ "> 'gi;




その後、ターゲット= "_ blank"がフロントエンドを使用してメディアへのリンクに追加されます。



タスク：どこでもxmlrpc.phpリンク...を削除する必要があります。



解決策： nginx kogfigに小さなコードを追加します。 



 subs_filter 'https://example.com/xmlrpc.php' 'https://example.com/xmlrpc.txt' gi;




xmlrpc.txtでは、イースターエッグを貼り付けることができます。



あとがき



  • example.comとstatic.example.comをサーバーに置き換えます。 例はphp5-fpmだけでなく、apacheでも機能します。
  • この記事では、ダイナミクスをキャッシュせずに、CDNのスタティックのキャッシュのみを考慮しています。 その理由は非常に単純です。無料の関税では、キャッシュの更新時間は30分です。 つまり、ページは30分キャッシュに保存されます。 サイトを更新するたびに開発モードに切り替えることができるようになりました（キャッシュを毎回選択的にリセットする代わりに）。
  • 場所は主要なポイントを示します。 私は読者が私の人生で初めてnginxとwordpressを入れないという事実に焦点を合わせます。 この記事は行動のヒントと考えてください。
  • レギュラーの説明はここにあり、 この記事を読むことができます。 静的変数を配布するときは、 ネガティブルックの構造を使用します 。つまり、レギュラーはlocationの値を取り、末尾から先頭まで見て、 .phpが見つからない場合はファイルが提供されます。 リクエストでphpが見つかった場合、 場所は正しくありません。すべての場所でphpが除外されるため、 ダミーの ルートを選択します。この場合、 phpファイルはなく（空のフォルダのみ）、 404を返します。
  • CloudFlare .comは素晴らしいCDNプロバイダーであり、すでに無料のプランで 、ほぼすべてのプロジェクトに十分な機能を手に入れることができます。
