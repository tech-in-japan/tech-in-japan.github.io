DebianとDevuanがsysvinitで協力

数日前、古典的なUnix / Linux初期化システムsysvinit 2.92 の 次のバージョンが予定より早くリリースされました。 前号の2.91は1か月以上前に発表されました。













ほとんどすべての最新のLinuxディストリビューションが拒否した古い初期化システム（SI）のマイナーバージョンのリリースで注目すべき点は何ですか？また、オープンソースコミュニティとDebian Linuxユーザーにとってこれはどんな喜びですか？







このエピソードは、リリースがDebianとDevuanの2つの敵対的な開発チームの共同作業であり、4年前にsystemdをめぐる状況により分裂したという点で興味深いものです。 しかし、順番に見てみましょう。

デヴアン



この状況に同意しなかったDebian Linuxディストリビューションの開発者は、同じ第8バージョンのリリースの6か月前から長い間悲しみませんでした。 彼らはsystvinitに基づいてフォーク を作成し、 Dev oneというフレーズからDevuanという名前を付けました。







配布キットのハイライトと親OSとの主な違いは、大騒ぎの始まりです。 Devuan LinuxはSIとしてsysvinitを選択しました。 一般に、ディストリビューションキットの開発は、 ロシア語の部分を除外することなく、Linuxユーザーの間で大きな熱意で迎えられました。







6月、配布キットの2番目のバージョンが、その祖先の最新バージョンであるDebian Stretchのパッケージベースでリリースされました。 sysvinitに加えて、SIとしてopenrcを選択することもできます。







Debian開発者の環境、そして実際にLinux OSのさまざまなバリエーションの多数のユーザーの間で、このような分裂を引き起こした原因を理解してみましょう。 実際、GNU / Linuxの歴史には 、 GPLv3かどうか 、 UEFI SecureBootなど、難しい決定と危険な変化がありました。なぜ、これがすべて邪魔にならないのですか？







systemdをめぐる論争



すべての紛争の本質を1つの論文にまとめた場合、 systemdは、システムの初期化とそのサービスに対する最も完全な制御の交換を意味し、Unix哲学を完全に拒否することになると言えます。







そして、最初のものはsystemdの反対者によって激しく争われていますが、2番目のものについては誰も争っていません。 systemdが管理者の生活を楽にすることに誰もが同意するわけではありませんが、 systemdとUnix Wayが両極端であると主張できる人はほとんどいません。







systemdは、アグレッシブな並列化機能を提供し、D-Busソケットとアクティブ化サービスの実行を使用し、必要に応じて起動するデーモンを提供し、Linux制御グループを使用してプロセスを監視し、スナップショットとシステム状態の回復、マウントとマウントポイントをサポートし、依存関係ベースの実装も行います複雑なトランザクションプロセスを制御するためのロジック 。

OSとそのプロセスの初期化を制御するためのこのような包括的なシステムが必要になったのはどこですか？ 私は陰謀論を信じていないので、その理由はsysvinitと他のSIの欠陥にあったと信じています。







1983年に作成されたsysvinitは 、次のような多くの重要なタスクを解決できませんでした。







  • プロセスの並列開始。
  • リムーバブルメディアの検出
  • ソケットベースのサービスのアクティブ化。
  • 異なるプロセスとサービス間の依存関係を確実に制御
  • /dev/log



    を介した早期イベントロギング


これらすべておよび他の多くの不便は最終的にsystemdで解決されました。 構成ファイルの標準化、構文の統一、およびcgroups



基づくサービスの依存関係の管理により、商用Linuxディストリビューションのベンダーは管理者の生活を簡素化し、サーバーパークの管理コストを削減できたと思われます。







しかし、価格はいくらでしたか？ 奇妙な設計によると、レナートポタリングの発案は、通常のSIから「Linuxシステムのビルディングブロックのセット」に移行しました。 プロジェクトのメインページから引用します 。 デバイスの接続、ファイルシステムエントリポイント、ネットワーク接続、システムタイムサービス、ユーザーセッション、システムログなどを制御する状態の一種。







同時に、多くのDE開発者、特にGnomeは、グラフィカル環境の要素をsystemdにバインドし始めました：







  • 電源管理;
  • ユーザーセッション管理;
  • 雑誌を見る。
  • ノートパソコンの画面を閉じた場合、イベントは処理されません。
  • ウェイランド


systemd以外の別のSIを選択した場合、特別なパッチがなければ、これはすべてGnomeでうまくいきません。







このような状況の理由は、多くのプログラムセットで2つのオプションをサポートするのが難しすぎるということです。1つはsystemdを使用し、もう1つは使用しません。 その結果、Linuxディストリビューションで別のSIを選択する方法がない場合にシチュエーションが作成されます。

まあ、SIを選択する方法はありません、それは本当に重要ですか？ OSローダーやDHCPクライアントを選択する可能性がなければ、それが私を大いに混乱させるとは思いません。







多くのユーザーに判明したように、 systemdは多くのことを悩ます。 起動の瞬間からのシステムイベントの早期かつ完全なロギングは確かに優れていますが、バイナリログファイルを備えたLinuxシステムをどのように想像できますか？







Lennart Potteringは優れたプログラマーかもしれませんが、彼のコミュニケーションのスタイルと、検出された欠陥、批判に対する反応は、すべて批判よりも下にあります。 欠陥5644に対する彼の反応を示します。







欠陥自体。 







 # mkdir -p /foo/dir{1,2} # touch /foo/.bar{1,2} # cat /etc/tmpfiles.d/test.conf R! /foo/.* - - - - - Reboot.


Lennart Potteringによるコメント。







これほど多くの問題を考慮するかどうかはわかりません。 ええ、それはUNIXの落とし穴ですが、「rm -rf /foo/.*」はまったく同じように機能します。 。

これは孤立したケースではなく、このスタイルのコミュニケーションは彼にオタクからの悪い評判をもたらしました。 そして、これはsystemdとその作者に対する主張の一部にすぎません。 その範囲、規模、および複雑さのために、システムはSIとして脆弱になる可能性があります-その主な機能。







それでも、 systemdはその機能においてすべての主要なSIより優れていますが、疑問は残っています。 なぜ普通のローカルホスト管理者は、Debian Linuxや他のディストリビューションを選択する機会がなく、SIの使用とデバッグが簡単なのですか？ 実際、すべての人がプロセスとクォータ管理の並行起動を必要とするわけではありません。







systemdの状況についての私の理解は、 より洗練された、しかし同時に、多くの人にとって、それを必要とする人とそうでない人に不当に複雑で不透明なSIが課せられたということです。 多くのLinuxユーザーはこれが気に入らず、今ではLinuxがWindowsのようになったと嘆き始めましたが、それはオープンソースコードのみでした。
