コードエディターに選択するライブラリはどれですか？



Npmは、コードエディターに対して400以上の結果を生成します。 ほとんどの場合、これらは特定のフレームワークまたはプロジェクト用に作成されたいくつかの最も人気のあるライブラリのUIラッパー、同じライブラリのプラグインまたはそれら自身の変更を加えたフォークであり、ブラウザでコードを編集するためではなく、単にキーワードによって出力されます。 そのため、幸いなことに、選択肢ははるかに狭くなります。 いくつかのライブラリ-CodeFlaskは 、軽量ですがあまり機能的ではなく、小さなスニペットやインタラクティブなサンプル用に設計されていますが、デスクトップエディターで使用されている機能を備えた本格的なWeb IDE用ではありません。



最終的に、 Ace 、 CodeMirror 、およびMonaco Editorから選択できる3つのライブラリがあります。 これらの初期のCodeMirrorは、Berlineer Marijn Haverbekeによる個人的なイニシアチブであり、オンラインチュートリアルEloquent JavaScriptでエクササイズコードエディターが必要でした。 エディターの最初のバージョンは2007年にリリースされました。 2010年、Aceの最初のバージョンが同じベルリンのJSConf.euで発表され、Ajax.orgがクラウドIDE Cloud9用に開発しました（実際、AceはAjax.org Cloud9 Editorの略です）。 2016年、Cloud9はAmazonに買収され、現在はAWSの一部です。 最新のMonaco EditorはVS Codeのコンポーネントであり、2015年末にMicrosoftによって公開されました。



各エディターには長所と短所があり、それぞれが複数の大規模プロジェクトで使用されます。 たとえば、CodeMirrorは、ChromeおよびFirefox開発ツール、BitbucketのIDE、npmのRunKitで使用されます。 エース-MODX、カーンアカデミー、コードアカデミーで Monaco-GitLab IDEおよびCodeSandbox内。 以下は、プロジェクトに最適なライブラリを選択するのに役立つ比較チャートです。

ライブラリをサイズで比較することは意味がありません。すべて特定のプロジェクトに接続する方法と方法に依存するためです。完成したファイルをビルドの1つでロードする（これもさまざまです）か、何らかのコレクターを介してnpmパッケージを実行します。 そして最も重要なのは、エディターの使用量です。すべてのスタイルとテーマがロードされるかどうか、使用されるアドオンとプラグインの数と数です。 たとえば、CodeMirrorでは、MonacoとAceですぐに動作するほとんどの機能は、アドオンでのみ使用できます。 表には、CDNの最近のバージョンの文字数と、どの順序が関係するかについての一般的な考えのための圧縮ファイルの重みが示されています。



すべてのライブラリには、ほぼ同じ基本機能セットがあります：コードの自動書式設定、行の折り畳み、切り取り/コピー/貼り付け、ホットキー、強調表示とテーマの新しい構文を追加する機能、構文チェック（CodeMirror-アドオンのみ、Ace-これまではJavaScriptのみ） / CoffeeScript / CSS / XQuery）、ツールチップとオートコンプリート（CodeMirrorで-アドオンを使用）、コードによる高度な検索（CodeMirrorで-アドオンを使用）、タブと分割モード、diffモード、マージツールの実装方法（CodeMirrorで） -1つのウィンドウにプラスとマイナス、またはアドオンを介した2つのパネルのいずれか、 エース - セパレート・リーブ）。 古くなったため、多くのアドオンがCodeMirror用に作成されていますが、その数はエディターの重量と速度の両方に影響します。 モナコは、すぐに使用できる多くの機能を実行できますが、私の意見では、AceやCodeMirrorよりも優れています。



モナコに滞在した理由はいくつかあります。



  1. 私たちのプロジェクトにとって重要と考えられた最も開発されたツール：

    • IntelliSense-ヒントとオートコンプリート。
    • コンテキストメニューおよびミニマップによるスマートコードナビゲーション。
    • すぐに使える2パネルdiffモード。


  2. TypeScriptで書かれています。 コントロールパネルはVue + Typescriptで記述されているため、TSサポートが重要でした。 ところで、Aceは最近TSもサポートしていますが、元々はJSで書かれていました。 CodeMirrorの場合、 DefinitelyTypedには型があります 。
  3. それは最も積極的に開発されており（おそらく、それほど前にリリースされていないため）、バグはより迅速に修正され、プールリクエストが処理されます。 比較のために、CodeMirrorでは、バグが何年も修正されず、松葉杖を松葉杖に乗せて松葉杖を運転したという悲しい経験がありました。
  4. インターフェースとメソッド間の相互参照を含む便利な自動生成（完全性への希望を与える）ドキュメント。
  5. 私たちの好みでは、最も美しいUI（おそらく作成時間にも関連）と簡潔なAPIです。
  6. どの編集者が最も頭痛の種となったのかを開発者の友人に尋ねた後、エースとCodeMirrorがリーダーでした。


また、仕事のスピードについても言う必要があります。 コストのかかる解析は、並列ワーカースレッドで行われます。 さらに、すべての計算はビューポートのサイズによって制限されます（すべてのタイプ、色、レンダリングは、表示されているラインに対してのみ計算されます）。 コードが100,000行未満の場合にのみ、ブレーキをかけ始めます-プロンプトは数秒間計算できます。 また、重いコンピューティングにワーカーを使用するエースは、より高速であることが判明しました：同じ長さのコードでは、プロンプトがほぼ瞬時に表示され、200,000行にすばやく対応します（公式Webサイトでは、400万行であっても問題ないはずです）ネジが加速され、入力が遅くなり始め、プロンプトが100万回後に消えました）。 並列計算がないCodeMirrorでは、そのようなボリュームを引き出すことは非常に困難です。テキストと構文の強調表示の両方がちらつくことがあります。 ファイル内の100,000行は現実の世界ではまれなので、これに目をつぶった。 40〜5万行のモナコでも、すばらしい仕事をしています。

エディターを削除



Monacoウィンドウを破棄する場合、 dispose



メソッドを呼び出す必要があります。そうしないと、すべてのリスナーがクリアされず、これ以降に作成されたウィンドウが正しく機能せず、いくつかのイベントに数回反応します。 



 beforeDestroy() { this.editor && this.editor.dispose(); }
