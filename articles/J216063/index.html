Microsoft Project Server 2010の会社のプロジェクト管理システムの仕様への適応

特定の顧客の要件を満たすために、Microsoft Project Server 2010の最終化における経験を共有したいと考えています。



この記事は、自社に適用されるプラットフォームの新機能について学ぶことができるビジネスユーザーと、Microsoft Project Server 2010の改良方法に関する新しい知識から学ぶことができる技術スペシャリストの両方にとって興味深いものです。



小さな紹介



大企業では、プロジェクト管理は会社の基準によって厳密に規制されています。 原則として、すべてのプロジェクトには明確な分類と明確に定義されたプロジェクトライフサイクルがあります。 さらに、同時に実行される多数のプロジェクトと、さまざまな部門のプロジェクトに関与する多数の専門家。



プロジェクト管理の情報システムは、理想的には次の機能を提供する必要があります（完全ではありませんが、一般的には十分です）。



  • プロジェクトの正当化、要件の仕様、ボリュームの形成、範囲、計画されたリソース、プロジェクト予算の準備を含むプロジェクトの開始。
  • カレンダーとネットワークの計画。
  • プロジェクトのリソース管理。
  • プロジェクトの運用情報を維持し、プロジェクトのステータスを監視します。
  • 設計活動のワークフローおよび設計文書のアーカイブの維持。
  • プロジェクトのリスク、問題、変更要求の管理。
  • プロジェクトに関する運用レポートおよび統計レポートの作成。


さらに、情報システムは、カスタマイズ可能なプロジェクト分類システムから始まり、必要なすべての機能を提供するプロジェクトチームのチームワークのためのカスタマイズ可能な領域で終わる、会社の内部標準に柔軟に適応する必要があります。



企業がプロジェクト管理システムを構築するための基礎として考慮しているオプションの1つは、Microsoft Project Server 2010（2013）です。 しかし、それを会社の要求に実際に「そのまま」または最小限の変更を加えて使用することは可能ですか。また、未解決の問題はどの程度可能で便利かということです。



カレンダーネットワークの計画とプロジェクトのリソース管理の機能に関する質問がない場合、既定では、Microsoft Project Server 2010（2013）が提供するプロジェクトワークスペースの機能はそれほどバラ色ではありません。



問題の声明



実際のプロジェクトの例を示します。



このプロジェクトは、古典的な方法論に従って編成されており、開始、計画、実装、監視と制御、終了の各段階が含まれています。



プロジェクトの開始と計画の段階について詳しく見てみましょう。 システムに（理想的に）求めているのは、企業の基準に基づいたプロジェクトの作成と、プロジェクトの評価と計画に関する共同作業のメカニズム（機能）です。



タスク1.会社の内部標準に従って、必要なすべての属性を含むプロジェクトを作成します。 たとえば、内部予算とポートフォリオの分類に応じたプロジェクトコード。



タスク2.プロジェクトの作成後に、プロジェクトの基本データを生成するための基本機能を提供する作業領域を自動的に取得します。



  • プロジェクトの分類（プロジェクトの目標、決定、結果、利益、および実行可能性の調査）;
  • プロジェクトの組織構造（管理評議会、運営評議会、ワーキンググループ、必要な資格を持つ必要な専門家）;
  • プロジェクトの範囲と範囲（プロジェクトの機能的、組織的、技術的ボリューム、ボリュームの変更手順）;
  • プロジェクトの実装へのアプローチ（実装戦略、仮定と制限、影響分析）;
  • プロジェクト予算（資金調達段階、プロジェクト段階および費用項目ごとの資金調達段階内の作業範囲、総プロジェクト予算）;
  • プロジェクト文書の構造化されたリポジトリ。
  • 最終段階のドキュメントとしてのプロジェクトパスポートの自動生成。


プロジェクトの初期段階では、さまざまな従業員がチームワークに参加しているため（彼らは常にMicrosoft Projectと連携するスキルを持っているわけではありません）、使い慣れたユーザーインターフェイスで作業するための快適なメカニズムを提供する必要があるため、ワークスペース（タスク2）の要件は重要です。
