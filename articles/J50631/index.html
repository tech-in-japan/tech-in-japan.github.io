GPG



そして、GnuPGユーティリティまたは単にGPGから始めます。 これは何ですか GPGは身分証明書であり、あなたの名前の下に隠れているのはあなたであり、危険な敵ではないことをインターネット上で示す能力です。メールやジャバーメッセージを暗号化する能力です。これはファイルに署名し、指紋を別のファイルに保存する能力です これは、単純なmd5ハッシュよりもはるかに信頼性が高いことに同意する必要があります。また、この状態でファイルを保存したのは自分であり、他の誰かではないことも明確にします。 一般に、GPGは家庭で非常に有用で必要なツールです。 私は彼の可能性のすべてを語るのではなく、それらはマナで読むことができますが、最も基本的なものだけを語ります。 だから、チームとしてパッケージを置きます

$ sudo apt-get install gnupg





次に、独自のキーペアを作成する必要があります。 募集します

$ gpg --gen-key





標準タイプのキー（DSAおよびElGamal）も標準サイズ（2048ビット）に適しています。 有効期間（制限なし）も標準に適合します。 次に、自分に関する情報を入力する必要があります。 Debianは例として提供しています：「Baba Yaga（pensioner）<yaga@deepforest.ru>」。括弧内の情報はコメントです。 入力する必要はありません。名前と電子メールだけに制限できます。

: Ivan Ivanov

Email-: ivan@ivanov.iv

:

User ID:

"Ivan Ivanov <ivan@ivanov.iv>"





/ etc / passwdにログイン用に指定された名前と電子メールを入力すると、このキーは、可能なすべてに署名して暗号化するためにデフォルトで使用されます。

次に、パスワードを入力する必要があります。 パッケージに署名する際に何度も入力する必要があるという事実のために、「はい、パスワードなしで別のキーを作成するので、そのような秘密があり、誰もあなたのラップトップからそれを盗むことはありません。」 。 そのような挑発に陥らないでください。 gpgキーはパスポートなどです。 パスワードなしでキーを作成することは、壊れたジッパー付きのバッグを運ぶようなものです-常に私たちの前にそれを保持し、実行中に排他的に実行することでメトロを横断する場合でも、潜在的な泥棒を引き起こすのはなぜですか？ したがって、覚えやすい複雑なパスワードを作成し、最終的に目的のキーペアを取得します。

$ gpg --list-keys

/home/user/.gnupg/pubring.gpg

----------------------------------

pub 1024D/3904FAE7 2009-01-30

uid Ivan Ivanov <ivan@ivanov.iv>

sub 2048g/2DA010F1 2009-01-30





次に、公開キーを何らかのキーサーバーに送信して、誰でも自分自身にインポートできるようにする必要があります。

$ gpg --keyserver subkeys.pgp.net --send-keys 3904FAE7

gpg: 3904FAE7 hkp subkeys.pgp.net





誰でもコマンドを入力できるようになりました

$ gpg --keyserver subkeys.pgp.net --recv-keys 3904FAE7



公開鍵を取得します。 この場合の3904FAE7は、個人キーIDです。

次の2行を〜/ .bashrcに追加します。

export DEBFULLNAME="Ivan Ivanov"

export DEBEMAIL="ivan@ivanov.iv"



、パッケージの作成および署名時に将来使用されます。 これで、今のところGPGを離れて次のパートに進むことができます。



アプト



はい、APTユーティリティの魔法のスイートとパッケージビルドでの使用について説明します。 残念ながら、多くはこのパッケージに非常に表面的に精通していますが、いくつの問題を回避できたでしょう。

まず、 apt-keyユーティリティを覚えておいてください。 サードパーティのリポジトリを接続すると便利です。 このユーティリティは、さまざまなリポジトリに署名できる信頼できるキーのリストを管理します。 コマンドを実行すると
$ sudo apt-key list



次に、ディストリビューションの公式リポジトリに署名したキーの標準リストを取得します。 別のリポジトリのキーをそこに追加する方法は？ たとえば、次のように：

$ wget -O - qutim.org/debian/archive.key | sudo apt-key add -



wgetを使用して、サイトからqutIMリポジトリの公開キーをダウンロードし、それを入力としてapt-keyコマンドに渡します。 これで、このキーで署名されたパッケージは「良好」と見なされ、警告なしにインストールされます。



続けましょう。 このアプローチがなぜひどいのかをお話しすることを約束しました。 もちろん、数十の依存関係を持つインストールされたすべてのパッケージを頭に抱えているSlackwareユーザーであれば、ソースから別のプログラムをコンパイルしてインストールすることは難しくありません（慣れたSlackwareユーザーは、最新バージョンでは本格的なパッケージマネージャーが登場し、おそらくは正直でさえあると言いますcn ... Debianから盗まれましたが、彼らはまだ古い方法ですべての依存関係を手動で追跡することを好みます）。 しかし、私たちは皆、パッケージ配布とは何か、依存関係、バージョン管理を最終的に知っています。 / opt /にpidginを入れることは文明化されていません。単に忘れてしまった場合、メニューのpidginボタンを突いて、壊れたICQで古いバージョンを実行します。

「プレフィックス/ usr /を指定できます」と言います。 もちろん、リポジトリ内の最初のパッケージ更新は、パッチを適用したバージョンを摩擦します。 誰が本当にこれを必要としますか？ いいえ、すべてを文化的に行います。 pidginは大きくてugくて重いパッケージであり、一般的には好きではないので、アセンブリ中にコンパイラオブジェクトファイルをキャッシュするccacheパッケージの例ですべてを説明します。

ご存知のように、このパッケージは、ファイルを「gcc」、「g ++」などのようにします-実行可能ファイル/ usr / bin / ccacheへのシンボリックリンクであり、必要なコンパイラに代わって呼び出されたファイルの名前を調べます。 しかし、ここに問題があります-コンパイル中の多くのソースコードは「c ++」ファイルを使用します。これは適切なコンパイラへのシンボリックリンクであるはずですが、ccacheはコンパイラ「c ++」を認識しません。 どうする？ 私は苦労せず、ソースコードを再訓練せず、余分なファイルを作成せず、汚いハックに対処することにしました。

したがって、ソースパッケージを使用するには、/ etc / apt / sources.listファイルを確認します。 そこに「deb-src」で始まるコメントアウトされた行がある場合は、コメントを外してください。これらは、ソースコードを含むパッケージファイルの入手先に関する指示です。

今、私たちはほとんど子供の頃から慣れ親しんでいたapt-getコマンドを使い、異なるパラメーターで3回実行します。
$ sudo apt-get update

$ sudo apt-get install devscripts lintian

$ apt-get source ccache



sources.listを編集した後に使用可能なパッケージのリストを更新する最初のコマンド、パッケージの構築に必要なスクリプトをインストールする2番目のコマンド、ccacheソースを含むパッケージをダウンロードする3番目のコマンド。 ソースパッケージをダウンロードするために、スーパーユーザー権限さえ必要ないことに注意してください。 一方、APTは次のように伝えます。
:1 http.us.debian.org sid/main ccache 2.4-16 (dsc) [1118B]

:2 http.us.debian.org sid/main ccache 2.4-16 (tar) [86,4kB]

:3 http.us.debian.org sid/main ccache 2.4-16 (diff) [44,1kB]

132k 2s (51,1k/c)

dpkg-source: ccache ccache-2.4

dpkg-source: : ccache_2.4.orig.tar.gz

dpkg-source: : ccache_2.4-16.diff.gz





注：ソースパッケージは3つのファイルで構成されています。.orig.tar.gzファイルは、プログラム開発者によって配布されたものなど、ソースが変更されていないアーカイブです。.diff.gzファイルは、ソースに対するすべての変更を保存するdiffファイルです、debianアセンブリ命令（次回これらの命令について）、およびパッケージに関する簡単な情報を格納する.dscファイル、最初の2つのファイルのハッシュ、およびdscファイル自体は作成者のGPG署名によって署名されます。 必要に応じて、公開鍵を取得してこの署名を検証できますが、パッケージを公式リポジトリから取得したため、このようなナンセンスは処理しません。 したがって、ccache-2.4ディレクトリに移動し、お気に入りのテキストエディターを使用して、execute.cファイル内の行「 x_asprintf(&fname, "%s/%s", tok, name);



」を次の行に変更します。
if (strcmp(name, "c++") == 0) {

x_asprintf(&fname, "%s/%s", tok, "g++");

} else {

x_asprintf(&fname, "%s/%s", tok, name);

}





ほとんどの場合、このプログラムとこのパッチは必要ありませんが、少なくとも次回パッケージを修正する場合は、たとえば、その中のアイコンをお気に入りのものに置き換えるか、「プログラムについて」に「Vasya was here」という行を追加してください。何をすべきか想像できます。

それで、ソースを修正しました。次に何をする必要がありますか？ そうです、パッケージのバージョン番号を増やす必要があります。 すべてを同じディレクトリコマンドに入力します
$ dch -i



これにより、お気に入りのエディター（私はvimを持っています）が起動し、変更ログが既に開かれ、新しいバージョンを記録するためのテンプレートが追加されます。 このコマンドが実行された時点ですでにbashを再起動しているか、単にコンソールを再度開いている場合、変更の作成者は〜.bashrcファイルに追加した名前と電子メールになります。 たとえば、私の提案するテンプレートは次のようになります。

アセンブリに必要なすべてのパッケージをインストールする別の方法は次のとおりです。
$ sudo apt-get build-dep ccache

本当に便利？ エラーの内容を伝えるだけでなく、Debian開発者ドキュメントの適切なセクションに送信します。 ここで、Debianでは、lintianユーティリティはUbuntuに比べてパッケージに関してはるかに厳密であるため、完璧なパッケージを入手してDebianでコンパイルし、ubuntuに移植する場合、蚊は鼻を傷つけません。 この場合、提案されたバージョン番号を意図的に変更することにより、誤って変更したことが唯一の間違いであることが通知されました。 次に、1レベル上のカタログを終了し、アセンブルされたパッケージをインストールします。
$ sudo dpkg -i ccache_2.4-17_amd64.deb





私たちは何を残しましたか？ ここまで読んだ最も注意深い読者はきっと疑問に思うでしょう：もしバージョンが2.4-17より大きいccacheの新しいバージョンが出たらどうなるでしょうか？ この質問には少なくとも2つの解決策があります。 最初のバージョンを組み立てるときは2.4〜17ではなく、明らかに大きく、たとえば5.7〜99です。 しかし、私が予期せずにパッチを当てたproftpdが新しいバージョンに置き換えられたときに、そのようなソリューションでさえ効果がなかったことが判明しました。 2番目のソリューションはもう少しエレガントです。 入力する：
$ sudo aptitude hold ccache



このコマンドはccacheに保留状態を設定します。つまり、現在インストールされているバージョンを修正し、リポジトリから新しいバージョンをインストールしません。
