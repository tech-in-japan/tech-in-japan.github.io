ANSI ASC X12（American National Standards Institute Accredited Standards Committee X12）は、電子文書のサイズが小さいことが重要である（1秒あたり300-1200ビットの速度のモデムの場合）70年代に開発され、各バイトは最大の情報。 したがって、電子文書の「可読性」は放棄され、「情報の密度」が優先されました。


そのため、XMLのように、人間が読み取れる美しさは表示されません。 また、標準では、ブロックといわゆるループが存在する、かなり複雑な階層構造のドキュメントを作成できますが、それでもすべてのブロック（ISA / GS / STを除く）の終了タグも提供されていません。 カタへのリンクを介して、希望する人はフォーマットの構造と説明を詳しく知ることができるので、必要なことだけに触れます。



各タイプのドキュメントには独自のテンプレート構造があり、個々のフィールドとセグメントの意味と目的、それらのタイプと可能な値、および必須およびオプションのセグメントとブロックのリストを示します。 テンプレートのバージョン管理がサポートされており、ドキュメントの対応するフィールドでタイプとバージョン番号に関する情報が送信されます。 特定のタイプのドキュメントのテンプレートを使用して解析および検証する必要があると想定されています。

各ブロックおよびセグメントの目的の詳細な説明は、一般構造12およびこのタイプのドキュメントのテンプレートの構造の分析にあります。 しかし、基本的な概念はすべてのタイプに共通です-パッケージの内容は〜記号で区切られたセグメントで構成されます（例のテキストでは、各セグメントは読みやすいように新しい行に表示されます）。 同様に、各セグメントには*で区切られた任意の数のフィールドを含めることができます。



このような合意により、セグメントのリストとフィールドのリストとして、ドキュメントの線形構造を簡単に取得できます。 ただし、これはドキュメントブロックの階層構造を復元するには不十分です。 このため、すでに述べたように、ほとんどの種類のドキュメントではかなり大きなファイルであるスキームを使用することになっています。 ただし、ドキュメントの検証の問題は考慮せず、解析のみに制限するため、次のアルゴリズムは目的に非常に適しています-ドキュメントセグメントの線形リストの順次分析中に遭遇する各セグメントについて、唯一の質問に対する答えを取得する必要があります：現在のブロックの終わり（同時に次のブロックの始まり）であるか、（他の場合では）現在のブロックの内部セグメントであるかにかかわらず、新しいネストされたブロック。



パーサー



以下のClojureコードは、セグメントの線形構造を階層ブロック構造に解析します。 階層構造を定義するために、宣言的なアプローチが使用されます-最も単純なループデータ構造。ネストされたブロックを形成し、現在のブロックで終わるセグメントのリストがリストされたセグメントに対して個別に設定されます。 もちろん、この構造はドキュメントのタイプに依存し、実際、その階層を定義します。 ただし、以下の解析関数は汎用的であり、解析されたドキュメントのタイプが選択されたテンプレートと一致する場合、この方法で正しく設定された構造テンプレートで動作します。 



 ;; Parse 835 x12 string to hierarchical structure (def loops {"835" {:nested #{"ISA"}} "ISA" {:nested #{"GS"}} "GS" {:nested #{"ST"} :end #{"IEA"}} "ST" {:nested #{"N1" "LX"} :end #{"GE" "ST"}} "N1" {:end #{"SE" "LX" "N1"}} "LX" {:nested #{"CLP"} :end #{"SE" "LX"}} "CLP" {:nested #{"SVC"} :end #{"SE" "LX" "CLP"}} "SVC" {:end #{"SE" "LX" "CLP" "SVC"}}}) (defn parser-core [id ss acc] (let [seg-id (first (first ss)) {:keys [nested end]} (loops id)] (if (or (empty? ss) (and (contains? end seg-id) (not (empty? acc)))) [acc ss] (let [[v ss-] (if (contains? nested seg-id) (parser-core seg-id ss []) [(first ss) (rest ss)])] (recur id ss- (conj acc v)))))) (defn segments [s] (str/split (str/trim s) #"~")) (defn elements [s] (str/split (str/trim s) #"\*")) (defn x12 [s] (first (parser-core "835" (mapv elements (segments (or s ""))) [])))

最初の例は、セグメントの階層構造です。
