アプリケーションでエクスプローラーからドラッグアンドドロップで転送されたファイルのリストを取得する方法

この記事の内容：



多くのプログラムでは、ユーザーがエクスプローラーからアプリケーションウィンドウにファイルをドラッグして開くことができます。 原則として、これは標準のオープニングモデルとは対照的に、メニュー「ファイル->開く」を選択するか、ツールバーの対応するボタンをクリックすることにより、ユーザーにとってより便利です。この場合、ユーザーはダイアログボックスの操作手順をスキップするからです。

この機能をアプリケーションに追加すると、より「プロフェッショナル」になります。



仕組み：



この機能をWindowsアプリケーションに追加するには、主に2つの方法があります。

最初の方法は、標準のWindowsドラッグアンドドロップAPI関数を使用して、付随するウィンドウメッセージを処理することです。

2番目の方法は、アプリケーション間でデータを転送するための高度な方法を提供するOLE（COM）テクノロジーの使用です。

この記事では、最初の最も簡単な方法を検討します。



短いレビュー：



デフォルトでは、Windowsアプリケーションのファイルのドラッグアンドドロップサポートは無効になっています。

この機能を有効にするには、ドラッグアンドドロップ通知を受信することをWindowsに通知し、通知データの処理を担当するウィンドウを指定する必要があります。

このウィンドウは、ユーザーがドラッグアンドドロップ操作を完了したときに到着するWM_DROPFILESメッセージを処理できる必要があります。

このメッセージはDropイベントハンドラーを実装する機会を提供します。Dropイベントハンドラーから、さまざまなAPI関数を呼び出して、転送されたものに関する情報を取得できます。

このメッセージの処理が完了したら、この操作を処理したことを通知し、それに関する通知を受け取りたくないことをWindowsに通知することをお勧めします。







Windows Vista（およびそれ以降）のセキュリティ問題

セキュリティ上の理由から、Windows Vistaは異なるセキュリティ設定を持つウィンドウ間でドラッグアンドドロップできない場合があります。

その結果、アプリケーションによるWM_DROPFILESメッセージの受信がブロックされる場合があります。


ドラッグアンドドロップによるファイル受信を実装するために、Delphiアプリケーションで実行する必要のある手順をステップごとに考えてみましょう。



1.必要なAPI関数にアクセスするには、ShellAPIモジュールを接続する必要があります。 



uses ShellAPI;


2. DragAcceptFiles関数を呼び出し、最初のパラメーターをWM_DROPFILESメッセージの処理を担当するウィンドウのハンドルに渡し、2番目のパラメーターをこのウィンドウが通知を受け取る準備ができていることを示すTrueに渡します。

たとえば、このウィンドウはアプリケーションのメインフォームになります。 



 DragAcceptFiles(Form1.Handle, True);


3. WM_DROPFILESメッセージを処理します。 Delphiでは、アプリケーションのメインフォームのクラスでメッセージハンドラを宣言できます。 



 procedure WMDropFiles(var Msg: TWMDropFiles); message WM_DROPFILES;


4. WM_DROPFILESメッセージを受信したら、DragQueryXXX関数を使用して、転送されたファイルに関する情報を取得します。以下の説明を参照してください。



5.このメッセージの処理が完了したことをWindowsに通知します。

これは、WM_DROPFILESメッセージの処理時に受信したHDROP記述子がパラメーターとして渡されるDragFinish関数を呼び出すことで実行されます。

この関数を呼び出すと、ドラッグアンドドロップ操作に関するデータの保存に使用されるリソースが解放されます。 



 DragFinish(DropH);


6.アプリケーションが終了したら、DragAcceptFiles関数を再度呼び出す必要がありますが、今回はウィンドウがWM_DROPFILESメッセージを処理しなくなったことをWindowsに通知する2番目のパラメーターとしてFalseを渡します。 



 DragAcceptFiles(Form1.Handle, False);


移行されたファイルに関する情報を取得します。



移行されたファイルに関する情報を取得するために、2つの関数DragQueryFileとDragQueryPointを使用します。

これらの関数は両方とも、WM_DROPFILESメッセージの処理時に受信されるHDROP記述子パラメーターを必要とします。



ようこそ、すべての取引-DragQueryFile。


Windows APIのほとんどの機能と同様に、DragQueryFileは、渡されるパラメーターに応じて、いくつかの動作を実装できます。

このような動作は、プログラマがこの関数の正しい使用法を覚えるのが難しいという事実につながることがあります。

そのパラメーターを見てみましょう（Delphiの構文を考慮に入れて）：



  1. DropHandle：HDROP-WM_DROPFILESメッセージによって送信された操作記述子。
  2. インデックス：整数-転送されたファイルのリストを照会するときのファイルインデックス。
  3. FileName：PChar-ファイル名の転送先のバッファーへのポインター。
  4. BufSize：整数-FileNameバッファーのサイズ。


この関数を呼び出すと、次のデータを取得できます。



  • 転送されたファイルの数を取得します。 この情報を取得するには、Indexパラメーターに$ FFFFFFFFまたはDWORD（-1）を渡します。FileNameパラメーターは「nil」でなければならず、FileNameバッファーのサイズを制御するBufSizeパラメーターはゼロでなければなりません。 関数によって返される値は、転送されたファイルの数に等しくなります。 明らかに十分です。
  • Indexパラメーターに基づいてファイル名を保存するために必要なメモリ領域のサイズを取得します（カウントはゼロから始まります。つまり、リスト内の最初のファイルはゼロでインデックス付けされます）。 これを行うには、Indexパラメーターを使用してリストに必要なファイルを指定し、FileNameパラメーターとBufSizeパラメーターを前の段落と同じままにする必要があります。 その結果、関数はファイル名を保存するのに必要なバイト数を返します。
  • 渡されたインデックスに基づいてファイル名を取得します。 これを行うには、Indexパラメーターを使用してファイルインデックスを指定することに加えて、ファイルパスに関するデータと終端のゼロを受信できるバッファー（必要なサイズよりわずかに大きい）を準備します。


これまでのところ、これはすべて不明確に見えますが、記事の最後で、これらのすべての段階を、クラスの形式で実装された独立したコードの形式で検討します。



ドラッグクエリポイント


この関数は、原則として、代替のDragQueryFileであり、呼び出すことで、ドラッグアンドドロップファイルが終了した座標を見つけることができます。



そのパラメーターを考慮してください。

  • DropHandle：HDROP-WM_DROPFILESメッセージによって送信された操作記述子。
  • var Point：TPoint-ドラッグアンドドロップ操作の完了点に関するデータが返されるTPoint構造体。 ウィンドウのクライアント領域で操作が完了した場合、ゼロ以外の値を返します。それ以外の場合、関数はゼロを返します。


すべてを完全に検討しましょう。



新しいプロジェクトを作成します。

「魚」のDelphiアプリケーションがあり、それにドラッグアンドドロップで転送されたファイルに関する情報を受信する機能を追加する必要があります。

このフォームでは、関心のあるイベントのハンドラーを登録します。 



 procedure TForm1.FormCreate(Sender: TObject); begin // ...    DragAcceptFiles(Self.Handle, True); // ...    end;


このコードを使用して、WM_DROPFILESメッセージを受信するようにサインアップしました。

これは、メッセージハンドラーの外観です。 



 procedure TForm1.WMDropFiles(var Msg: TWMDropFiles); var DropH: HDROP; //    DroppedFileCount: Integer; //    FileNameLength: Integer; //    FileName: string; // ,    I: Integer; //      DropPoint: TPoint; //     Drop begin inherited; //   DropH := Msg.Drop; try //     DroppedFileCount := DragQueryFile(DropH, $FFFFFFFF, nil, 0); //        for I := 0 to Pred(DroppedFileCount) do begin //    FileNameLength := DragQueryFile(DropH, I, nil, 0); //  ,          // (Delphi       ) SetLength(FileName, FileNameLength); //    DragQueryFile(DropH, I, PChar(FileName), FileNameLength + 1); // -     (    ) // ...     end; // :  ,     Drop DragQueryPoint(DropH, DropPoint); // ... -      finally //  -   //   DropH    ... DragFinish(DropH); end; //   ,    Msg.Result := 0; end;


最初に、ローカル変数でのDrop操作のハンドルを覚えています。

これは便宜上のみ行われます。将来、この値をDragQueryFile関数の呼び出し（およびDragXXXの残りの部分）に使用し、最初の呼び出しでリスト内のファイル数を取得します。

次のステップは、ループ内の各ファイルの名前をインデックスで取得することです（リスト内のファイルはゼロからインデックス付けされます-これを忘れないでください）。

各ファイルについて、先に説明したDragQueryFileを使用する2番目の方法を使用して取得できるバッファーのサイズを最初に見つけてから、ファイルへのパスを格納する行にメモリを割り当てる必要があります。

結論として、3番目の方法でDragQueryFileを呼び出して、割り当てられたバッファへのファイルパスを読み取る必要があります。



ファイル名に関するすべての情報を読み取った後、ユーザーがファイルをドラッグしたポイントの座標を取得できます。

すべての操作が完了したら、DragFinish関数を呼び出してハンドルを解放します。

例外が発生した場合でも、とにかくこのアクションを実行する必要があることに注意してください。

最後に、Msg.Result構造体に値0を設定して、このメッセージの処理に成功したことを示すことにより、メッセージの処理結果を示します。



アプリケーションの完了時に、メインウィンドウとWM_DROPFILESメッセージの受信を切断します。 



 procedure TForm1.FormDestroy(Sender: TObject); begin // ...    DragAcceptFiles(Self.Handle, False); // ...    end;


これをクラスとして実装します（Delphiの方法）



ベースアプリケーションのコードで直接APIを使用すると、多少の混乱が生じることに同意する場合、この機能を補助クラスの形式で実現する方がはるかに便利であることに同意すると思います。

例として、WM_DROPFILESメッセージを処理できる小さなクラスを次に示します。

それ自体には多くのコードが隠されているので、使用せずに独立して実装する必要があります。

確かに、微妙な違いがあります-WM_DROPFILESメッセージを処理するウィンドウの存在をWindowsに通知する必要があります。



このクラスの宣言は次のようになります。 



 type TFileCatcher = class(TObject) private fDropHandle: HDROP; function GetFile(Idx: Integer): string; function GetFileCount: Integer; function GetPoint: TPoint; public constructor Create(DropHandle: HDROP); destructor Destroy; override; property FileCount: Integer read GetFileCount; property Files[Idx: Integer]: string read GetFile; property DropPoint: TPoint read GetPoint; end;


...まあ、その実装： 



 constructor TFileCatcher.Create(DropHandle: HDROP); begin inherited Create; fDropHandle := DropHandle; end; destructor TFileCatcher.Destroy; begin DragFinish(fDropHandle); inherited; end; function TFileCatcher.GetFile(Idx: Integer): string; var FileNameLength: Integer; begin FileNameLength := DragQueryFile(fDropHandle, Idx, nil, 0); SetLength(Result, FileNameLength); DragQueryFile(fDropHandle, Idx, PChar(Result), FileNameLength + 1); end; function TFileCatcher.GetFileCount: Integer; begin Result := DragQueryFile(fDropHandle, $FFFFFFFF, nil, 0); end; function TFileCatcher.GetPoint: TPoint; begin DragQueryPoint(fDropHandle, Result); end;


原則として、これまでに述べられていないことは何もありません。 すでにすべてのコードを見てきました。

コンストラクタはHDROP記述子を受け入れ、デストラクタはDragFinishの呼び出しでそれを確定します。

ファイルのリストとその番号は、クラスプロパティによって表されます。

すべてのクラスメソッドは、基本的にDragQueryFileおよびDragQueryPoint APIのラッパーです。



TFileCatcherの使用を考慮して、WM_DROPFILESメッセージハンドラを書き換えましょう。 



 procedure TForm1.WMDropFiles(var Msg: TWMDropFiles); var I: Integer; //      DropPoint: TPoint; //     Drop Catcher: TFileCatcher; //   TFileCatcher begin inherited; Catcher := TFileCatcher.Create(Msg.Drop); try for I := 0 to Pred(Catcher.FileCount) do begin // ...     end; DropPoint := Catcher.DropPoint; // ... -      finally Catcher.Free; end; Msg.Result := 0; end;


今後の計画



TFileCatcherクラスを拡張して、ドラッグアンドドロップ操作で使用されるすべてのAPIを非表示にすることができます。 確かに、これにはWM_DROPFILESメッセージを傍受するためにフォームウィンドウへのアクセスが必要です。

この実装の1つのオプションは、フォームウィンドウをサブクラス化することです。 確かに、これはこの記事の範囲外です。 ただし、さらに情報が必要な場合は、Drop Filesコンポーネントのセットをご覧ください。
