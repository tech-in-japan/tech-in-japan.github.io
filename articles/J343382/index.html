IntelliJ IDEA 2017.3。 何が新しくて面白いですか？

他の誰かが最新でない場合、 IntelliJ IDEA 2017.3は今日です 。 伝統を続けながら、リリースで新しく興味深いものが追加または変更されたことについてお話します。スマートなヒント、リファクタリング、ナビゲーション、 Gitを操作するためのツール、データベース、フレームワークのサポートなどです。







行こう！



スマート補完でのコールチェーンの型キャスト



以前のリリース（IntelliJ IDEA 2017.2）では、 メソッド呼び出しチェーンは、 Smart Completionの最初の呼び出しで表示されるようになりました（2回目の呼び出しでのみ使用可能になる前）。 このリリースでは、これらのチェーンにより、コンテキストに基づいた自動型変換も可能になります。







コードフロー分析



数年前、IntelliJ IDEAは、パラメータのNullableおよびNotNullアノテーションを自動的に出力し、ライブラリクラスメソッドの値を返す方法を学びました。 この結論は、メソッド内の実行フローの分析に基づいて行われました。 バージョン2017.3では、パラメーターの注釈の出力は、ライブラリクラスメソッドだけでなく、プロジェクト内のプライベートクラスメソッドまたは最終クラスメソッドでも機能します。







同様の方法での注釈出力の場合、IDEは明示的な注釈宣言のクイックフィックスを提供します。







IDEは、 Stream APIメソッド呼び出し内の潜在的なNullabilityの問題を検出することも学びました。 潜在的な問題を検出した場合、IDEはこれを報告し、可能であれば、コードを自動的に変更して実行中にエラーを受け取らないようにします。







forループをStream API呼び出しチェーンに変換します



StringBuilderに文字列を追加する結果となるforループは、 Stream.collectおよびCollectors.joiningを使用して、自動的にStream API呼び出しのチェーンに変換できます 。







クイックフィックスを使用したブールメソッドのロジックの反転



ブール型のリファクタリングの反転は、メソッドのクイック修正として利用できるようになりました（以前は、 クイック修正はバッチモード 、およびクラスの変数、定数、フィールドでのみ利用可能でした）。







コレクションの個別のメソッド呼び出しとStream API呼び出しチェーン



これが理にかなっている場合、IDEは、コレクションのsortおよびtoArrayの個々の呼び出しを、 Stream API呼び出しチェーンの適切なメソッドに置き換えることを提案しています。







マップとセットの初期化で重複キーを検出する



誤って同じキーの値を2回以上指定した場合、または同じ要素をSetに追加しようとした場合、IDEはエラーの可能性について警告します。







冗長な例外宣言の検出



冗長な例外宣言の検出は、以前はfinalおよびprivateメソッドに対してのみ機能していました。 現在、検出はどのメソッドでも機能します。







この特定の検査は、テストなどのいわゆるエントリポイントでは無効にできます。



非推奨のメソッドの移行



非推奨のメソッドに明示的な代替が指定されている場合、IDEは自動置換のクイックフィックスを提供します。







JUnit 4からJUnit 5への移行



モジュールにJUnit 4とJUnit 5の両方のライブラリが存在する場合、IDEはJUnit 4からJUnit 5への自動移行のクイックフィックスを提供します。







すべてのタイプの実行構成のダッシュボードを実行



Run Dashboardは、以前はSpring Bootアプリケーションでのみ使用可能でしたが、現在ではあらゆるタイプのRun Configurationで使用できます。 これを行うには、[ 実行構成の編集 ]ダイアログを開き、[ デフォルト ]を選択し、適切な構成タイプを[ 実行ダッシュボードタイプ]リストに追加します。







実行ダッシュボードは、いくつかの実行構成を起動および管理するための便利なインターフェイスを提供します。







JVMのデバッガー



このリリースでは、 変数 、 ウォッチ 、 式の評価タブでのブレークポイントの処理と値の表示（ データレンダラーを使用）により、 デバッガーが開始プロセスに導入するオーバーヘッドコストのアカウンティングと最適化にさらに注意が払われました。 検査およびその他の場所。

さらに、任意のデータレンダラーをオンデマンドとしてマークできます。 これは、必要な場合にのみ値が計算され、表示されることを意味します（クリックで）：







Javaタイプレンダラーの設定で、 データレンダラーをオンデマンドとしてマークできます。

Java EE 8のヒントとナビゲーション



新しいリリースは、 Java EE 8標準を完全にサポートし、 非同期CDIイベント 、 CDI Beanインジェクション 、 PushBuilder 、 DisposesおよびProducesの追加のヒント、ナビゲーション、その他の便利な機能を提供します。







SpringおよびSpring Bootのサポート



IDEは、 Spring BootアプリケーションのSpring Facet設定でMVCコンテキストを自動的に検出および構成できるようになりました。







Spring Bootアプリケーション起動およびデバッグウィンドウの[エンドポイント]タブは、実行中のアプリケーションの監視に役立ち、 Spring Boot 2.0 Actuator Endpointsをサポートするようになりました。



Spring Bootアプリケーション構成ファイルの強調表示では、値のタイプが考慮されるようになりました。







Bean間の依存関係図（ Spring Beans Diagram ）に、新しい興味深いモード： Neighborhoodが登場しました。 このモードでは、直接の依存関係のみが表示されます。 依存関係のいずれかを選択すると、選択したビンなどの直接的な依存関係が表示されます。したがって、依存関係間を移動すると便利です。







モジュールを操作する



一部のモジュールを「アンロード」としてマークし、 VCSプロジェクトの更新時に新しいモジュールが作成された場合、それ以前は自動的に「ロード」としてマークされていました。 現在、「ロード済み」とマークされたモジュールがこれらのモジュールに依存しない場合、これらのモジュールは自動的に「アンロード済み」とマークされます。



また、コミットする前に、IDEは「アンロードされた」モジュールがエラーなしでコンパイルされることを確認します。



さらに、表示中のモジュールのグループ化方法が変更されました。 以前は、 プロジェクト構造で作成できるモジュールのグループがこれに使用されていました。 パッケージの構造をグループ化するために使用されるのと同じメカニズムが使用されるようになりました。モジュール名はドット記号で分割され、ツリーを形成します。







プロジェクトの組み立て



以前は、 Gradleを使用しているときに、 Delegate IDEビルド/実行アクションをGradleまたはGradleテストランナーオプションが有効になっている場合、 カバレッジ統計モードでアプリケーションとテストを実行できませんでした。 カバレッジで実行は 、すべての場合に機能するようになりました。



Gradle 、 Android / GradleおよびSBTのプロジェクトのインポート、コンパイル、ビルドのプロセスは、新しいビルドツールウィンドウで確認できます。







RESTリクエストを処理する



これで、新しいエディターベースのツールを使用してRESTリクエストを処理できるようになりました。 これを行うには、プロジェクトに.http権限でファイルを作成し、エディターで開きます。 プロジェクトにファイルを作成したくない場合は、 スクラッチファイルを使用できます 。







コトリン



昨日起こったKotlin 1.2のリリースに続いて、IDEの言語サポートが更新されました。 更新の主な革新は、 JVMとJavaScriptの間で「コードを再利用」する実験的な機能です。







expectキーワードを使用して、クラスまたはメソッドを「ジェネリック」として宣言できるようになりました。 特定のプラットフォームのこのクラスまたはメソッドの実装は、対応するプラットフォームのモジュールの実際のキーワードを使用して宣言できます。 IDEは、構成されたプラットフォームの実装が存在することを確認し、宣言から実装へ、またはその逆に移動できるようにします。







このハブまたは公式ドキュメント（英語）で、「共通モジュール」の作業について詳しく読むことができます。 このようなモジュールの動作の前提条件は、 Gradleプロジェクトのアセンブリです。



スカラ



キャリッジの下の文字の強調表示（ Shift + Ctrl / Cmd + F7を押すと使用できるようにデフォルトでオンになっています）は、文字の暗黙的な使用も強調表示するようになりました。







アプリケーションまたはテストのアセンブリと起動は、インデックス作成に関連付けられなくなり、インデックス作成がまだ終了していなくても動作します。



SBTプロジェクトでは、プラグインが、 SBT構成には指定されていないがローカルIvy Cacheに存在する依存関係インポート式に気付いた場合、IDEはそれをSBT構成に追加することを提案します。







プラグインの新しいバージョンはAmmoniteをサポートし、構文の強調表示、ツールチップ、ナビゲーション、および実行構成の個別のタイプを提供します 。







Scalaプロジェクトの作成が変更されました。 現在、 Lightbend Activator （今年の半ばに存在しなくなった）の代わりに、プラグインはTech Hub Project Starterプロジェクトテンプレートの使用を提案しています。



GitとMercurialを使用する



Log からHereからInteractively Rebaseを呼び出すことで、 Gitの最新の変更をインタラクティブにリベースできます。







ブランチを切り替えると、IDEはワークスペースの状態（エディタータブ、ツールウィンドウなどの場所）を保存します。







便宜上、マージに伴う変更（ merge ）のために、新しいモードが[ログ ： 親への変更を表示 ]に追加されました。 このモードでは、 ログの右側のパネルには、競合のあるファイルの変更だけでなく、マージに参加している各ブランチの個別の非競合変更も表示されます。







このモードは、コンテキストメニューで有効にできます。



Gitの競合する変更を表示するために、 Log IDEは3つのパネルのインターフェースを提供するようになりました。



パッチから変更を送信するとき、IDEは変更の名前、作成者のメール、および変更の説明（パッチに含まれている場合）を自動的に記録します。







異なるコンピューター間でIDE設定を同期する



IDEが職場や自宅、デスクトップコンピューター、ラップトップなどにインストールされることがあります。以前は、各コンピューターでIDEを構成する必要がありました。 しばらく前に、プラグインを使用してこのプロセスを自動化できました。 このプラグインを使用すると、設定は指定されたGitリポジトリに保存されました。 IntelliJ IDEA Ultimate 2017.3には、Gitリポジトリを必要としない新しいメカニズムがあります。すべての設定は、JetBrainsサーバー上の安全な形式で保存されます。 新しいメカニズムのもう1つの利点は、設定だけでなく、直接インストールされたプラグインも同期されることです。 新しいメカニズムを使用するには、 JetBrainsアカウントが必要です。







Dockerプラグインの変更



このリリースでは、Dockerのプラグインの改善を続けています。 最も注目すべきもの：



  • Dockerデプロイメントの実行構成は 、3つの個別の実行構成に分けられます ： Dockerfile （ Dockerfileからイメージを構築し、結果のイメージに基づいてコンテナーを起動）、 Docker Image （ イメージを開始、「 docker run 」 コマンドに相当）およびDocker Compose （サービスを開始）。
  • Docker Image Run Configurationでは 、コマンドラインを直接指定できるようになりました（以前は、IDEで特別なJSONファイルを作成する必要がありました）。
  • Dockerfileの実行構成では、イメージのみを収集でき、コンテナーとしては実行できません。
  • これで、 Dockerfileエディターから（エディターの左側のパネルをクリックすることで）イメージを直接組み立てたり、このイメージに基づいてコンテナーを起動したりできます。
  • IDEは、 Dockerfileおよびdocker -compose.ymlファイルの別名をサポートしています。
  • [設定]のDocker設定が追加されるまで、 Dockerツールウィンドウはデフォルトで非表示になりました。


SVGファイルのプレビュー



これで、エディターでSVGファイルを開くと、IDEは右側にこのファイルの視覚的なプレビューを提供します。







エディターの視覚的境界



IDEの初期には、 右マージン設定がありました。これはエディターに表示され、 コードスタイルルールで考慮されていました。 新しいリリースでは、この設定はでハードラップに名前が変更され 、それに加えて、新しいビジュアルガイドオプションが追加されました。 これを使用すると、複数の視覚的境界のカンマ区切り値を一度に指定できます。 これらの境界線もエディターに表示されます。







主に、これはコードの設計規則がそのようないくつかの境界を参照する場合に必要です（例えば、PythonやPHPで可能です）。



データベースツール（DataGrip 2017.3）



  • 多数のデータソースを使用する場合、これらのデータソースをグループに整理できるようになりました。
  • テーブル内の数値データを操作するために、それらは右揃えになりました。
  • PostgreSQLの場合、 データベースツールバーにはユーザーとロールが表示され、 外部データラッパーもサポートされます。
  • テーブル内のデータをExcelから挿入できるようになりました。
  • テーブルのいくつかの値を相互に比較できるようになりました。
  • DDLへのエクスポートでは、より多くのオプションがサポートされるようになりました（ コンテキストメニュー | SQLスクリプト | SQLジェネレーター ）。
  • JOINのオートコンプリートが少しスマートになりました。
  • Exasolデータベースのサポート。
  • 言語インジェクションを介したSQL Serverの XQueryおよびXPathのバックライト。


JavaScriptおよびTypeScriptのサポート（WebStorm 2017.3）



  • Vue.jsでは、コード補完とプロパティ、プロパティ、メソッドのナビゲーション、およびコードテンプレートの新しいコレクションが登場しました。
  • JavaScriptおよびTypeScriptクラスの新しいプルメンバーおよび抽出スーパークラスリファクタリング 。
  • TypeScriptの新しいリファクタリングExtract type aliasおよびExtractインターフェイス 。
  • ドキュメントを更新し、標準JavaScriptメソッドおよびオブジェクトの自動補完を改善しました。 これらのメソッド用に事前に作成されたインデックスのおかげで、プロジェクトの発見は高速です。
  • JestとMochaの 監視モードのサポート、 Jestのテストによるコードカバレッジのレポート、 Jestスナップショットをワンクリックで簡単に更新する機能。
  • JavaScriptファイルとTypeScriptファイルのインポートは、あるファイルから別のファイルにコードをコピーして貼り付けるときに自動的に追加されるようになりました。
  • WebStormでクライアントアプリケーションをデバッグするには、 Chrome用の追加の拡張機能をインストールする必要がなくなりました。 ただし、 Live Editingには引き続き拡張機能が必要で、現在はデフォルトで無効になっています。
