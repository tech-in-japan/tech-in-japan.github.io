データベース設計


アメリカの哲学者ラルフ・ウォルド・エマーソンはかつて言った：「愚かな一貫性は、小さな心のホブゴブリンです」（何かに続いて愚か、これは多くの愚か者です）。 多くの場合、人々はこのことわざを使用して、攻撃から身を守り、詳細について他者からの質問をします。 Core Dataはデータベースではないと定期的に主張しますが、データベースのように扱いますが、この餌に落ちないことを願っています（著者：「何かがアヒルのように歩く場合、それはアヒル、それはアヒルです。」 このセクションでは、アプリケーションのデータ構造を設計する方法を検討し、このプロセスとリレーショナルデータベースの構造を設計するプロセスとの類似点を描くことで、議論だけでなく、最終的な結果にも役立ちます。 もちろん、いくつかの点で類推はありませんが、議論の中でこれらの点を指摘しようとします。



コアデータのエンティティは、リレーショナルデータベースのテーブルと同じように見えます。 属性はテーブルのフィールドであり、関係は主キーと外部キーによる結合であり、エンティティ自体はテーブル内のまさにレコードです。 データモデルがSQLiteタイプのストレージに「重畳」されている場合、Core Dataは上記および第2 /第3章で説明した方法ですべてを実際に実装します。 ただし、データはメモリ、または何らかのファイル（アトミックストレージ）に保存できることを忘れないでください。ここにはテーブルとフィールド、主キー、または外部キーはありません。 Core Dataは、使用されるストレージのタイプからデータ構造を抽象化することで、設計とデータとの相互作用の利便性と汎用性を実現しています。 Core Dataがあなたの心を捕らえ、Core Dataで単一の全体にマージします。CoreDataで力を感じるか、最適でないデータ構造を使用して壊れたトラフになってしまいます。



コアデータの研究を始めたばかりで、最初のデータモデルを作成している初心者は、データベース構造の従来のモデリングで使用していたように、エンティティのプライマリキーを作成して責任を負うことができないことに常にconstantlyします。 Core Dataには自動インクリメントキーはありません。それらはそこでは必要ないからです。 コアデータは、追加された各オブジェクト（管理対象オブジェクト）に対して一意のキーを作成する責任を負います。 主キーは、外部キーも存在しないことを意味しません。CoreDataは、エンティティ自体間の関係を管理し、必要な結合を実行します。 すぐにやめて！ 主キーと外部キーを宣言することに不快感を感じるのをやめてください！



あなたを泥沼に引きずり込むことができる別の場所は、多対多の関係シミュレーションです。 リーグマネージャーアプリケーションでは、プレーヤーが異なるチームに所属できることを想像してください。 その場合、各チームには多くのプレーヤーがいて、各プレーヤーは異なるチームでプレーできます。 従来のリレーショナルデータベースを作成する場合、3つのテーブルがあります。1つはチーム用、もう1つはプレーヤー用、もう1つはどのプレーヤーとチームが属するかを追跡するためです。 同時に、Core Dataでは、プレーヤーとチームの2つのエンティティが同じであり、通常のチェックマークを設定することで多対多の関係が構成されます。 ただし、コアデータでは、「内部」で3つのテーブルZTEAM、ZPLAYER、Z1TEAMSを使用してこれを実装していますが、これらは実装の詳細です。 3番目または4番目のテーブルがあることを知る必要はありません。CoreDataがこれを処理し、より良い結果をもたらします。



ヒント ：データの保管メカニズムではなく、データに注意してください。



リレーショナルデータベースの正規化


リレーショナルデータベースの設計理論は、正規化（正規化プロセス）と呼ばれるデータモデリングプロセスを組み合わせたものです。その目的は、冗長性を削減または完全に排除し、データベース内のデータへの効果的なアクセスを提供することです。

正規化プロセスは5つのレベル（フォーム）で構成され、レベル6で作業が続行されます。 おそらく、最初の5つの形式の定義を好むのは数学者だけであり、おそらく理解できるのは彼らだけです。 これらは次のようになります。

「関係Rは、RにMVDが存在する場合、たとえばA->-> Bの場合にのみ、Rのすべての属性も機能的にAに依存する場合にのみ、4番目の正規形（4NF）になります。 Rの依存関係（FDまたはMVD）のみがK-> Xの形式です（つまり、候補キーKから他の属性Xへの機能的依存関係）。 同等：RがBCNFにあり、RのすべてのMVDが実際にFDである場合、Rは4NFにあります。



（c） www.databasedesign-resource.com/normal-forms.html



すべてのフォームを説明するのに十分なページも、すべての通常のフォームの定義を調べてその本質を説明したいという欲求もありません。 代わりに、このセクションでは、コアデータに関する各標準フォームについて説明し、設計のヒントを提供します。 特定の標準形式に従うには、これに先行するすべての標準形式に従う必要があることに注意してください。



最初の正規形（1NF）に対応するデータベースは、正規化されていると見なされます。 この形式（レベル）の正規化に従うには、データベースの各レコードに同じ数のフィールドが必要です。 Core Dataのコンテキストでは、これは、管理対象オブジェクトに特定の（事前定義された）数の属性が必要であることを意味します。 Core Dataでは異なる数の属性を持つ1つのエンティティを作成できないという事実に基づいて、Core Dataのモデルは自動的に正規化されると結論付けることができます。



2番目の標準形式（2NF）と3番目の標準形式（3NF）は、非キーフィールドとキーフィールドの関係に関連し、非キーフィールドが属するキーフィールドに関する事実である必要があります。 Core Dataのキーは気にしないので、この質問は削除されます。 ただし、エンティティのすべての属性が、他の何かではなく、その特定のエンティティを説明することを確認する必要があります。 たとえば、フォームの色はプレーヤーではなくチームの状態を表すため、PlayerエンティティにはuniformColor属性を設定しないでください。



次の2つの標準形式、4番目の標準形式（4NF）と5番目の標準形式（5NF）は、コアデータの世界で同じ形式と見なすことができます。 それらは、記述されたデータの冗長性を削減または排除し、 複数の値を持つ属性を個別のエンティティに転送し、エンティティ間に多対多の関係を作成するように促します。 4NFコアデータに関しては、フォームには、エンティティが複数の値を持つことができる属性を持つべきではないことが記載されています。 代わりに、これらの属性を別のエンティティに移動し、エンティティ間に多対多の関係を作成する価値があります。 LeagueManagerアプリケーションのチームとプレーヤーを使用したアプリケーションの例を考えてみましょう。 4つの標準形式と5つの標準形式に違反するデータモデルは、追加の属性を持つ1つのエンティティ（チーム）で構成されます-プレーヤー。 次に、各プレーヤーの新しいチームインスタンスを作成します。その結果、余分な（不要な）チームオブジェクトがいくつかあります。 代わりに、LeagueManagerアプリケーションで使用されるデータモデルでは、多対多の関係をチームエンティティと結合するPlayerエンティティが作成されます。



モデリングプロセスでは、これらの単純なルールを忘れないでください。 たとえば、LeagueManagerアプリケーションのデータモデルでは、TeamエンティティのuniformColor属性は正規化する機能を表します。 フォームの色の名前は有限のセットです。つまり、Colorという名前の追加のエンティティを作成できます。これは、name属性を持ち、Teamエンティティとの多対1の関係で接続されます。

リンクの表示と編集


属性情報を表示するときと同じように、リンクを選択することでリンクのプロパティを表示できます。



リンクにはいくつかのプロパティがあります。 まず、名前と有限のエンティティ（接続が属するエンティティ）があります。 最後のエンティティは、リンクが指すオブジェクトのタイプを記述します。 Playerエンティティの場合、チーム関係はTeamエンティティを指すことが期待されています。 コアデータアーキテクトは、各リレーションシップで逆の反発、つまり反対方向に向けられたリレーションシップを示すことを強くお勧めします。 たとえば、PlayerエンティティはTeamと接続しているため、実装に基づいて、推奨事項に基づいて、TeamもPlayerと接続する必要があります。 実際に、フィードバックを忘れたり、確立したくない場合は、コンパイラーによって警告が生成されます。 追加の2つのプロパティが属性に使用できます：TransientおよびOptional。 一時リンクは、保存操作中にストレージに保存されません。 チームのコミュニケーションが一時的なタイプの場合、アプリケーションを保存してから再起動すると、プレーヤーが所属するチームに関する情報が失われます。 Playerオブジェクトは保存されますが、Teamオブジェクトとの関連付けは保存されません。 一時的な通信は、パスワードやアプリケーションの操作中に取得した情報から取得したものなど、実行時に値を設定する必要がある場合に役立ちます。 接続がオプションの場合、単にnilに設定できることを意味します。 たとえば、プレーヤーにはチームがない場合があります。



通信の多重度により、ソースオブジェクトが関連付けることができる有限オブジェクトの数（1つまたは多数）が決まります。 デフォルトでは、関係はk-oneです。つまり、ソースオブジェクトは1つの最終にのみ関連付けることができます。 これは、Playerエンティティに対してチームを使用する例の場合のみです。プレーヤーは、チームを1つしか持つことができません。 ただし、チームには多くのプレーヤーを含めることができるため、Teamエンティティ内のプレーヤー関係には多対多の関係があります。 これらの値（多くの側から）は、（NSSet）オブジェクトのセットとして表されます。 通信の多重度は、通信中のエンティティの有限セットのパワーを表す最小値と最大値を変更することでも決定できます。



結論として、Core Dataは、元のオブジェクトを削除するときに最終オブジェクトをどう処理するかを知るために、リンク解除のルールを使用します。 Core Dataがサポートするアンインストールオプションは多数あります。 削除には、アクションなし、無効化、カスケード、拒否などの種類があります。 この章の「構成ルール」セクションでは、削除ルールとその意味について説明します。

ルールを削除


削除ルールは、リンクを持つオブジェクトを削除するときにCore Dataが実行する必要があるアクションを決定します。

削除には4つのタイプがあります。

1.カスケード-ソースオブジェクトが削除され、関連する（最終）オブジェクトもすべて削除されます

2.拒否-ソースオブジェクトがいくつかのオブジェクトに関連付けられている場合、削除は行われません

3. Nullify-元のオブジェクトは削除され、関連するすべてのオブジェクトのフィードバックはnilに設定されます。

4.アクションなし-元のオブジェクトは削除されますが、関連するオブジェクトは変更されません



リーグマネージャーアプリケーションでは、プレイヤーコミュニケーションの削除ルールはカスケードです。つまり、チームが削除されると、そのプレイヤーもすべて削除されます。 削除ルールを拒否に変更した場合、チームに少なくとも1人のプレーヤーがいた場合、削除は行われませんでした。 Nullifyで削除ルールを設定すると、すべてのプレーヤーがリポジトリに残りますが、チームへの参照はnullになりますが、チーム自体は削除されます。
