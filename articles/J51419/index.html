重要なデータをバックアップする必要があるのはなぜですか？



ウォーランドがマスターとマルガリータで言ったように、恐ろしいことは人が死ぬことではなく、彼が突然死ぬことです。 同じことがハードドライブにも当てはまります。 私の人生で2台のハードドライブだけが死んだようで、1台は突然死んだ（非常に貴重なデータが失われた）、もう1台はほとんど突然死んだ（データの一部を回復することができた）、私の友人の間の状況は平均してそれほど変わらない。 グーグルによると、ネジの死の統計は主観的ではありません-平均して、ネジは数年間生き続け、60％のケースではSMARTでもネジが突然死にます



ハードウェアに加えて、データ損失のソフトウェア上の理由もあります。WIN95.CIHウイルスがFATを含むディスクの最初の数メガバイトを破壊すると、データに完全にアクセスできなくなります。 FAT構造を理解し、インターネットが存在しない場合に復元するために、1か月の猛烈なプログラミングとDiskEditでの登山が必要でした。 パーティションの転送中にPatrition Magicがクラッシュしました。 仕事中に

Partition Magicは、着陸時にシールドの電気をオフにしました。 最後の2つのケースでは、重要なデータは失われていませんが、これは単なる運です。 ネットワークに接続している場合、攻撃者が重要なデータを破壊および破壊する可能性があります。 この種の損傷から、RAIDアレイは保存されません。



バックアップの実行可能性を評価する方法は？



リスクの硬直性などがあります。これは、リスクの発生確率にお金のリスクのコストを掛けたものです。 この場合のリスクのコストは、失われたすべての情報を復元するためにどれだけお金を払ってもよいかということです。 以下からリスクの確率を推定します。 他の統計情報がないため、ハードウェア障害のみを評価できますが、この理由はすでにわかっています

十分でした。 ハードドライブが3年以内に死亡する確率は、新品であるかどうかに関係なく、少なくとも24％です。 私が書いたように、死が突然になる確率は60％です。 3年間の突然死の総確率は14％です。 ミラーリングのない2つの鉄道を使用する場合、この期間中に少なくとも1つの鉄道が死亡する確率は26％です。

もちろん、これらのディスク上のデータの価値を考慮する必要があります-貴重なデータを備えたハードドライブが2台あり（いずれかが失われると致命的です）、音楽と映画を搭載したハードドライブが1台あります-私は選択した音楽のコレクションを失うことに腹を立てますが、私は生き残ります。



かつて私は学生で、時には月に600ルーブルで生活していましたが、今ではデータセキュリティに投資する余裕があります。 今日、320ギガバイトの外付け（USB）ハードドライブを購入しましたが、この喜びに3,000ルーブルかかりました。 なぜハードドライブなのですか？ 私はたくさんのデータを持っているので-私の心にとって大切な写真だけがそのように40ギガバイトを占有します。はい、私はこれにあまり時間を費やしたくはありません。音楽と映画を除く。 なぜ外部なのか？ 盗難の場合はコンピューターとは別に保存できますが（ラップトップの場合はより適切ですが）、コンピューターに接続されていません（データに致命的なエラーが発生した場合）。



安全性



私は妄想ではありませんが、ハードドライブにはデータが含まれていますが、その安全性は私だけのものではありません。たとえば、見知らぬ人のヌードスタイルの写真を他人の手に渡す必要はまったくありません。 したがって、ディスクは暗号化されています-私は全体を暗号化しました。 debianでは、すべてが数分で箱から出されます。したがって、後でこのデータを簡単に復元できることは間違いありません。主なことは、使用したアルゴリズムのカスケードとパスワードのヒントを紙に書き留めて、ネジで箱に保管することです サードパーティの商業的に助成された資金の使用を強くお勧めします。

ためらうことなく、私は未使用の（しかし、念のため）パーティションを完全にバックアップすることにしました。テストのために、一度インストールしたWIN98でパーティションを取得しました。



# cp /dev/hda1 /mnt/backup/images/





テスト：

# mkdir /mnt/hda1-backup-test

# mount /mnt/backup/images/hda1 /mnt/hda1-backup-test -o loop

# ls /mnt/hda1-backup-test/

boot.ini io.sys ntdetect.com ( )







やった！ うまくいく！



ext2およびext3ファイルシステムに作業データがあるので、ext2 / ext3のファイル属性にバックアップに関する情報が含まれている可能性があるという事実を使用したいと思います。 ただし、マウントされたファイルシステムではダンプを使用しないことを強くお勧めします-www.rhd.ru/docs/manuals/enterprise/RHEL-4-Manual/admin-guide/s1-disaster-rhlspec.html、dump.sourceforge.net/isdumpdeprecated。 html tar、cpio、backup-manager、baculaなどの一般的なバックアップテクノロジーには大きな選択肢がありますが、今のところ、rsync + cpを選択しました。これは、すべての増分アーカイブを解凍して読み取ることなく、結果をどこからでも透過的に読み取ることができるためです。



$ sudo rsync --archive --one-file-system /etc /home /root --delete /mnt/backup/rsync/`date +%F--%H-%M`







最後のいくつかのバックアップを保存する必要がありますか？ cpの--linkパラメーターは、ファイルをコピーするのではなく、ファイルへのハードリンクを作成するという事実を使用してください。



$ sudo rsync --archive --one-file-system /etc /home /root --delete /mnt/backup/rsync/latest/

$ sudo cp --archive --link /mnt/backup/rsync/latest/

/mnt/backup/rsync/`date +%F--%H-%M`







私たちはチェックします：



$ ls /mnt/backup/rsync/

2008-04-03--04-29 2008-04-03--04-34 latest







バックアップとの違いを見たいですか？ 現在の状態をバックアップ状態にコピーする「エミュレート」。



$ sudo rsync --archive --dry-run --verbose --one-file-system /etc /home /root --delete /mnt/backup/rsync/latest/







バックアップの違いを見たいですか？ 同様に。



$ sudo rsync --archive --dry-run --verbose --delete

/mnt/backup/rsync/2008-04-03--04-41/

/mnt/backup/rsync/2008-04-03--04-29/







以上です。
