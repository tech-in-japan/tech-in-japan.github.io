後方互換性が重要な理由





フレームワークを開発する際、そのいくつかのバージョンのリリース後、フレームワークに基づいて独自の拡張機能とモジュールを開発するプログラマーと同様に、フレームワークを購入/ダウンロード/インストールしたユーザー（またはMagentoの場合は顧客）がアップグレードする必要があるという事実に直面しますバージョンは可能な限り軽量でした。



顧客/ユーザーにとって、プロセスは費用対効果が高いはずです。 製品の新しいバージョンへの移行にかかる費用には、アップグレード自体、アップグレード後の自動テスト、回帰テスト、プラットフォームの新しいバージョンによって導入されたバグの修正、カスタマイズおよびサードパーティモジュールの更新が含まれます。 最小限にする必要があります。



プラットフォーム用に拡張機能を開発するプログラマーは、モジュールが可能な限り上位互換であることを望み、システムの変更によってモジュールのコードがどのリリースで破損するかを事前に知って、新しいリリースと互換性のあるモジュールのバージョンを事前に準備できるようにします。



その結果、フレームワーク開発者は予想外のジレンマを思いつきます。より多くの変更（バグ修正を含む）が必要になるほど、後方互換性のない変更が行われ、誰かが壊れてしまう可能性が高くなります。



コードの後方互換性ポリシー









セマンティックバージョニング （SemVer）は、ジレンマに対する答えを提供し、オプション（頻繁に下位互換性のあるリリース）を拒否しないようにします。



フレームワークのコンポーネントのバージョン番号は、MAJOR.MINOR.PATCHの形式で示されます。更新は次のとおりです。

  • MAJOR -APIの互換性のない変更について説明します
  • マイナー -後方互換機能が追加されたことを示します
  • PATCH-後方互換性のあるバグ修正について説明します


後方互換性ポリシーは、コードベースで@apiアノテーションが付けられたエンティティに適用されます。

言語レベルのPHPはこの機能を提供しません。 したがって、この言語で記述されたフレームワークは、長い間「正しい」アプローチを探していました。 たとえば、Magento 1は、当時の他の多くのフレームワーク（Symfony 1）と同様に、コンポーネントをカスタマイズおよび拡張し、子孫のクラスの動作をオーバーライドまたは拡張するためにクラスのいずれかから継承することをフレームワークが提案したときに、継承ベースのAPIを使用しました。 したがって、Magento 1では、プライベートプロパティとメソッドはまったく使用されず、Magentoコア開発者は2つのコントラクト（パブリック-パブリックプロパティ、メソッド、エンティティ定数を形成するコントラクト、保護-エンティティ継承コントラクト）を監視し、両方に後方互換性のない変更が追加されないようにする必要がありました。 コードベース内のこれらのエンティティのすべてのエンティティとすべてのメソッドがAPIである場合、新しい変更を追加してバグを修正すると、いずれの場合でも誰かが壊れることがあります。

API vs SPI（拡張ポイント）





MagentoのPHPコントラクトは、さまざまな方法で使用できます。 そのような用途は3つあります。



  • APIの使用 ：インターフェースメソッドはPHPコードで呼び出されます
  • サービスプロバイダーインターフェース（SPI）の使用 ：インターフェースを実装できるため、新しい実装により現在のプラットフォームの動作を拡張できます。
  • APIとSPIを同時に


モジュールへのすべての呼び出しはAPI（モジュールサービスコントラクト）を経由し、外部開発者やカスタマイザーが代替実装を提供したり、そのまま実装を拡張できるように実装することで、モジュールもインターフェイスを提供することを期待しています。 たとえば、検索機能の場合、リクエストを実行するアダプターにとらわれない契約があります。 また、このコントラクトはAPI（ビジネスロジックコードで呼び出される）として使用されることを前提としています。 インターフェースのセットが提供されますが、サードパーティの開発者は、Sphinxなどの新しい検索エンジンのサポートを追加して、検索アダプターを実装し、その実装を提供できます。 また、これにより、Search APIを使用するビジネスロジックが中断されることはありません 。



APIとSPIは相互に排他的ではないため、コード内で別々に分離しません。 SPIには@api
