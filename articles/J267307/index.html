Hyper-V仮想マシンと通常のコンピューターのバックアップ

仮想マシンと通常のコンピューターのバックアップ-時間の海を奪ったものについての私の経験を共有したいと思います。 安くて美しい作り方。



おそらく、VMWareのバックアップが必要な場合は、支払いの準備をするという事実から始めます。 無料のVMWareは、移行、バックアップなどになるまで無料です。 この場所で、無限のホリバーを開始できますが、私の参加は必要ありません。 私の話は、Windows Server 2012R2上のHyper-Vについてのみです。 記事の一部はVMWareに適用できますが、落とし穴がある可能性があります。



Hyper-Vに無料でバックアップすることも、Windows Serverのライセンスを取得することで既に購入したWindowsツールを使用してバックアップすることもできます。 バックアップの操作の利便性のために（また、私たちはそれを支払いました）、WDSと重複排除（おそらくグループポリシー）を使用します。



1.内部仮想マシンからのバックアップ



1.1。 今日のバックアップ



私たちが知る限り、どのWindowsもバックアップできます。 さらに、インターフェースを介したWindowsバックアップの設定は、最終的にはwbadminユーティリティのバックグラウンド使用に限定されます。 そして、実際には、wbadminは何ができますか？ また、システムパーティションを使用してイメージをバックアップする方法と、個々のフォルダーをバックアップする方法を知っています。 記事のこの部分では、バックアップイメージ（システムパーティション）のみに関心があります。 残りは仮想マシンの特定のデータであり、個別にバックアップする必要があります。 したがって、結論：仮想マシンのシステムパーティション（および通常のコンピューター）に、貴重な情報やデータベース、個々のアプリケーションを格納しないでください。 MS SQL Server / MS Exchange / "Application Server 1C"および非システムパーティションまたは別のディスクにのみある別のセット。



それでは、バックアップが機能するためには何が必要ですか？ そして、必要なコマンドは1つだけです。 



wbadmin.exe start backup -backupTarget:\\\ -allCritical -quiet


実際、このチームには特別な権利が必要ですが、それらについては後で詳しく説明します。 今、一つのことを理解することが重要です。 このコマンドは、単なるバックアップではありません。 彼女は増分バックアップを行います。 さらに、サーバーとデスクトップ（クライアント）では、Windowsバックアップの形式が異なります。 違いは、サーバーオペレーティングシステムの場合、各バックアップのスナップショットを取得するという事実にありますが、デスクトップの場合、スナップショットは常に最後のスナップショットのままになります。 これはどのような増分バックアップですか？ ただし 、イメージ全体をバックアップするのではなく、最後のバックアップ以降に変更された部分のみをバックアップするため、「増分」のままです（つまり、トラフィックが少なくなり、バックアップが高速になります）。



同様の状況に直面した人は、バックアップが常に「増分」（フル）であることに気付くでしょう。 私たちの場合、バックアップはネットワークドライブで行われるためです。 つまり、サーバーWindowsの場合、最後の画像のみが残ります。



その後、彼はサーバーとクライアントOSでのwbadminの動作に違いがないことを明らかにしました。 違いはインターフェースにあるということです。 ハードドライブが-backupTargetキーで指定されている場合（コマンドはデフォルトキー-vssopyを使用します）、wbadminは増分バックアップ（最初のバックアップを除く）を実行します。 または、-vssFullスイッチを追加すると、完全バックアップが作成されます。



1.2。 前の写真の履歴でバックアップ



現時点では、仮想マシンのイメージのバックアップを作成しました。 しかし、これは今日のみの画像のバックアップです。 明日は完全に異なります...しかし、バックアップをバックアップするとどうなりますか？ はい、本当に増分です。 やってみましょう。 



 wbadmin.exe start backup -backupTarget:e: -allCritical -include:d: -quiet #  D -   ,   E -


しかし、これは私にとって十分ではなく、私はこれをしました： 



 ( echo select vdisk file="\\2\2\2.vhdx" echo attach vdisk ) | diskpart


スクリプトは、ネットワークから仮想ディスクを接続します。 バックアップ後、同様のスクリプトがディスクを切断します。 OSは、Eという文字がディスク上に定義されていることを記憶していますが、同じEという文字で他の誰かのディスクをスリップすることは禁じられています。 これを念頭に置いて、アルファベット（X、Y、Z）の終わりに近い文字を使用してください...



すぐに注意してください。今日のバックアップが履歴付きのバックアップと並行して行われる場合、発生できないバックアップが取得されます。



前日のバックアップを取得するには、履歴付きのバックアップが作成されるサーバーのインターフェース（GUI）を使用できます。 さらに、彼はWindowsコンソールでのwbadminコマンドのすべての開始を知っており、覚えています。 回復サービスを使用すると、履歴付きのバックアップで目的のアーカイブを選択できます。



2.仮想マシンのvhdxファイルのバックアップ



簡単かつ自然に作られます： 



 wbadmin start backup -backupTarget:$BackupPath -hyperv:$VMList -Quiet


しかし、いくつかの機能を備えています。 このコマンドはPowerShellで実行し、最初に変数内の仮想マシンのリストを取得する必要があります。 詳細な例については、Googleにお問い合わせください。



Windows Server 2012 R2の仮想マシンのバックアップには、Hyper-Vスナップショットが付属しています。 また、仮想マシンにLinuxカーネルがある場合、またはHyper-Vドライバーがない場合、仮想マシンがサスペンドされることに注意してください。 私はこの方法で仮想マシンをバックアップすることを個人的に拒否しました。 その理由は、Windows Server 2012（R2ではない）では、バックアップの前に仮想マシンを停止する必要があったためです。 はい。そして今、Windows Server 2012 R2では、最初の適切なバックアップ方法がある場合、Linuxの中断は私には不向きです。 （この記事へのコメントにはコメントがあります）。 Windows Server 2012 R2の次の更新後、仮想マシンのバックアップは中断することなく実行されます。 Linux OSはDump（CentOS、Ubuntu）を使用して内部からバックアップすることもできますが、これは私の場合、人形やその他のソフトウェアに関する別のトピックです。



3.バックアップとWDSを復元する



そして今、私の意見では、バックアップに関するこの記事の最も有用な部分です。



WDSはWindows展開サービスであり、Windows Server 2012R2の機能の一部です。 以前は、このサービスはRISと呼ばれていましたが、私はそれを知りませんでした。 一般に、WDSの本質はシンプルです。 DHCP（DHCP Windows Serverの場合は自動的に）に個別のパラメーターの形式で登録し、TFTP WDSブートローダーを介してネットワーク（コンピューターがネットワーク経由で起動するBIOS設定など）を介してコンピューターにダウンロードしました。 次に、WDSブートローダーを使用して、Windowsで使用可能な「ブートローダー」のイメージから選択できます。 ローダーは異なります-これらはインストーラーブートローダーイメージ、PE、およびREイメージです。 インストーラーブートローダーは、WDS内のWindows自体のイメージを引き続き必要としますが、これは、ネットワーク経由でWindowsをインストールする必要がある場合のためです。 バックアップから車をピックアップできるREイメージに興味があります。



WDSでどのように、どのように機能するかについては詳しく説明しません。 ただし、重要な注意事項は次のとおりです。



  1. REブートローダーがネットワーク経由でHyper-V仮想マシン上で起動するが、その中のキーボードが機能しない場合。 おめでとうございます。WinXP以前のREイメージは、Hyper-Vドライバーの存在を認識していません。
  2. システムがバックアップの復元を開始したが、停止した場合。 ハードバックアップ（バックアップが復元されているパーティション）のすべてのパーティションを削除して、再試行してください。 バックアップが破損する可能性があることを忘れないでください。ハードパーティションのすべてのパーティションを削除した後、古い情報が残っていない可能性があります。
  3. バックアップがUEFIをダウンロードしていて、UEFIのないコンピューターにそれを復元したい場合、時間を無駄にしないでください。 ほとんどの場合、バックアップを展開することはできません。
  4. UEFIロードとGPTパーティションを使用したバックアップは、異なるプロセッサ/マザーボードを搭載したマシンに復元できますが、MBR形式のパーティションと通常のBIOSを別のマシンにロードすると、展開される可能性は低くなります。 まあ、私は間違いなく成功しませんでした。
  5. 小さいボリュームのディスクにバックアップを展開しようとすると、これは機能しません。 バックアップのディスクがほとんど空だったとしても。 この場合、ダイナミックディスクを使用した仮想マシンへのリカバリが役立ちます。 このディスクをさらに削減し、新しいバックアップを作成します。 ただし、これはバックアップのUEFIブートローダーでのみ可能です（前の段落を読んでください）。
  6. バックアップを復元する前に、余分なディスクを切断して、それらの情報を上書きしないようにします。


4.重複排除の機能



実行中の仮想マシンを重複排除できます。 今日のバックアップを重複排除でき、履歴付きのバックアップを重複排除できます。 これはすべて、ハードドライブのボリューム（HDDとSSDの両方）に大きなプラスを与えます。 ただし、次のことを忘れないでください。



  1. 重複排除が1 TBを超えるドライブで機能する場合、重複排除オプティマイザーは大量のメモリを使用します。
  2. 重複排除が圧縮データで機能するが、10 TBを超える圧縮ボリュームで機能する場合、重複排除オプティマイザーの期間が長すぎます。 これは、異なるフォルダにある重複排除されたディスクに毎日データをコピーするだけで発生する可能性があります。
  3. HDDにバックアップを保存して必要な場合もありますが、HDDで5〜10を超える量の仮想マシンを使用する価値はありません。 重複排除は、そのような稼働中の仮想マシンの重複排除によってHDDのパフォーマンスがゼロに低下するという事実にのみ関連しています。


5.グループポリシー



ここでは、GPOを使用して、さまざまな方法でバックアップスクリプトのインストールを実装できます。 しかし、重要な点に注意を喚起したいと思います。



  1. 別のユーザーアカウントに代わってのみバックアップします。
  2. グループポリシーにパスワード付きのスクリプトを保存しないでください。
  3. 特別な権限でスクリプトを実行して、すべてのシステム情報を読み取ります。


実際、 結論 ：wbadminツールを使用したバックアップは可能であり、完全に実装可能であり、最も重要なことは実行可能です。 しかし、あなたが時間とあらゆる種類のささいなことに多くの忍耐を持っている場合にのみ。 そのため、たとえば、この記事では、正常に作成されたバックアップを監視する方法については何も言及していません。 Zabbixで監視を行うことができましたが、これについていくつか記事を書くことができます...この記事があなたにとって有益であり、貴重な時間を大幅に節約できることを願っています。
