問題の技術面



翻訳プロセスをサブタスクに少し分解すると、次のリストのようになります。

  • リバースエンジニアリングは、 リバースエンジニアリングという用語でよく知られているプロセスです。 ゲームが調査され、フォーマットが分析され、アルゴリズムが決定され、必要なデータが見つかります。
  • リソースの抽出 -翻訳に必要なリソースを編集用の便利な形式に変換します。 テキストからテキストファイル、グラフィックスから一般的な画像形式など。 これはすべて、手作業（計り知れないアマチュア風の方法）とツールを使用して（最高で翻訳者自身が作成）行うことができます。
  • 翻訳と編集はプロセスの本質です。 これらの作品の結果は、プレーヤーによって評価されます。
  • 翻訳アセンブリ -リソースをゲーム形式に戻し、元のゲームデータをそれらに置き換えます。 理想的には、このプロセスは自動アセンブリ用のツールの起動のみで構成されますが、残念なことに、ほとんどの翻訳者にとっては、必要に応じて変更を加えるたびに手動で編集する必要があります。
  • テストは、多くのエラーを特定し、場合によっては翻訳を改善できる必須の手順です。 自国語でお気に入りのゲームをプレイしたい人の中から、最も責任があり有能な人が選択され、可能な限り多くのゲーム状況を再現するためのリクエストとともに、ベータ版の翻訳が提示されます。 はい、そして外部からの新鮮な外観は常に良いです、チームは単に多くのエラーを見ないかもしれません。




詳細を掘り下げると、翻訳のためにゲームを準備するために完了しなければならないすべてのケースに対して同じアクションのシーケンスはありません。 また、特定の手順を実行できる汎用的な方法はありません。 実際、それは常に即興ですが、それでもほとんど常に見つかるタスクのリストがあります。



最も頻繁に発生する重要なタスクを強調して、それぞれについて個別に説明します。 プラットフォームとして具体的なものは考慮しません-つまり 以下で説明するすべては、PCと他のプラットフォームの両方に当てはまります。PlayStation、XBOX、またはSegaまたはDendy（NES）であっても同様です。



このコンテキストでは、リバースエンジニアリングのほとんどのタスクはデバッガーまたは逆アセンブラーによって解決できるため、個々のケースでのみ言及します。



テキストエンコーディングの定義



テキストがどのエンコーディングに保存されているかを判断することは、完全に簡単な作業に思えます。 そして、ほとんどの場合、これは本当に難しくありませんが、ここでも開発者のアイデアは制限がありません。



表示されたテキストが常にテキストとして正確に保存されるとは限らず、多くの場合、表示する必要があるのはフォントの文字の単なるインデックスのセットです。 多くの場合、主にUnicodeの最初の256文字のエンコードと互換性または部分的に互換性があります。 とにかく、キャラクターとコードの間の正確な対応を確立する必要があります。 ただし、現代のゲームでは、インデックスの代わりに通常のエンコーディングを使用し、テキストをXMLなどの形式にシリアル化します。長い間、誰もパフォーマンスについて考えていません。



エンコードを表すために、「エンコードテーブル」が使用されます。テキストファイルでは、特定のバイトシーケンスの各行に特定の文字シーケンスが関連付けられています。 次のようになります。 

41=A 42=B ... 5A=Z 1E20=Hero 1E21=Item 1E22=Bonus




たとえば、「ヒーローはアイテムを取得します！」というテキストは、「 1E 20 20 6F 62 74 61 69 6E 73 20 1E 21 21



」のようにエンコードされます。 ただし、結果のエンコードが適切なエンコード（Unicodeなど）と十分に互換性があることが判明した場合、通常、テーブルは不要であり、この手順をスキップできます。



エンコーディングを決定してテキストを見つける最も一般的な方法は、いわゆる「相対検索」（相対検索）です。 その本質は、検索されるのは絶対値ではないということです。検索基準は、目的のシーケンスの値の差です。 これを行うには、ゲームで見つかった短すぎない単語を取るだけで十分です。すべてのバイトシーケンスは、要素間の差が元の単語の文字コード間の差に等しいものであることがわかります。



たとえば、「WORLD」という単語には、「57 4F 52 4C 44」と「77 6F 72 6C 64」の両方のシーケンスがあります。 はい、少なくとも「13 0B 0E 08 00」！ このようなシーケンスを見つけて、これがエンコードされた単語であることを確認したら、エンコードテーブルを簡単に作成できます。 この機能を備えた最も有名なプログラムは、 16進エディタTranslhexionです。 Search Relativeのような特殊なユーティリティがたくさんあります。 はい、そして技術的に有能な翻訳者の多くが同様のユーティリティを自分で書いています。







典型的なケース：このスクリーンショットをフォントキャンバスと比較すると、見つかったシーケンスがフォント内の文字のインデックスであることがわかります。





一般に、このような手法は、ほとんどの場合に適用できますが、いくつかのトリックなしでは常に効果的ではありません。 結局、エンコーディングの文字コードがアルファベットの文字と同じ順序になることを保証する人はいません。



たとえば、Final Fantasy for GameBoy AdvanceおよびNintendo DSの多くの部分の再リリースでは、フォント内の文字は出現頻度でソートされ、UTF-8に似た方法がインデックスのエンコードに使用されます。 つまり コードが0x7Fより大きい文字は2バイトでエンコードされますが、最初の128文字は1バイトでエンコードされます。 

 uint16 encode(uint16 code) { return 0x8000 | ((code << 2) & 0x1F00) | (code & 0x3F); }




私の記憶におけるより深刻なケースは、ファイナルファンタジー：PlayStation Portableの20周年記念版です。 各場所には、独自のフォント、独自のテキスト、および結果として独自のエンコーディングが存在していました。 フォントは、テキスト内で見つかった文字のみで構成されており、出現頻度によっても順序付けられています。 ニューラルネットワークを使用して各場所のエンコードを認識するのは良いことですが、その利点は、シンボル透明マスクのピクセルごとの十分な比較でした。



これらの場合、相対検索も問題の解決に適していますが、文字数の違いではなく、フォント内の文字のインデックスの違いで検索する必要があります。 つまり 一連のインデックスを簡単に書き留めることができます。これは、このような検索に非常に適しています。



他のリソースと同様に、テキストはパックまたは暗号化できます。 この場合、ゲームデータの検索は、少なくとも単語の断片がまだパックまたは暗号化されたデータに存在する場合にのみ役立ちます（これはLZ77やRLEのようなアルゴリズムを使用する場合によく起こります）。 したがって、出力はRAMダンプ内の検索である可能性があります。 ダンプをマイニングする機能は、転送が行われるプラットフォームによって異なります。 エミュレートされたコンソールやPCに問題はないはずです。ゲームのメモリにアクセスするためのツールがたくさんあります。 しかし、他の場合には、ゲーム中にコンソールで必要なコードを実行する機能が必要です。そのためには、原則としてコンソールを「ハッキング」する必要があります。 アルゴリズム自体を解析する方法については、次の記事で説明します。



検索ポインター



データがシリアル化された形式で保存されている場合、この項目は安全にスキップできます。 リソースがすぐに使用できる形式で実行可能ファイル（カートリッジを使用するコンソールではほぼ常に当てはまる）に格納されている場合、原則として、各リソースにはポインターがあります。 当然、これはテキストにも適用されます。 さらに、テキストを自由に変更できるようにするには、変更された各行へのすべてのポインターとリンクを見つける必要があります。



一部の初心者にとって、ポインターの概念を理解することがアマチュア翻訳の技術を習得する上で最も難しい障害の1つであることは面白いことです。 原則として、そのような人々は長い間プロセスの技術面に煩わされず、元の行の長さに合うようにテキストを翻訳します。 さらに楽しいのは、スキルを向上させるために、彼らの多くが最終的にプログラミングを学ぶことです。 それは逆の順序で価値があります-そして、すべてがはるかに簡単になります。 本格的なITスペシャリストになった人は、多くの場合、このシーンを離れて、より真剣に仕事を始めます。



非常に多くの場合、すべてのポインターは単一の場所に格納されます。これは通常、「ポインターのテーブル」と呼ばれます。これは、ポインターまたはそれらを含む要素の配列です。 そのような場合、ゲームはインデックスによって行にアクセスし、インデックスはそのようなテーブルからポインタを取得します。 次に、テキストブロック内の任意の行へのポインタを見つけるだけで十分です。テーブルが見つかりました。







しかし、それほど単純ではありません...またはむしろ、必ずしもそれほど単純ではありません。 ポインターの検索を妨げる困難の1つは、「オフセット差」と呼ばれます。 実際には、ポインターは絶対（リソースの論理アドレスまたは物理アドレスを示す）だけでなく、相対（あるアドレスに対するオフセットを示す）にすることもできます。 または、たとえば、PlayStationのような古いディスクコンソールでは、データはメモリに読み込むために準備された形式で保存されることがよくあります。 ファイル内にある間は、ダウンロードが行われるアドレスを知らなくても、ポインターが指すものを把握することはできません。



オフセットの違いがわかるまで、ポインタを明確に計算することは不可能です。 したがって、通常最初にチェックするのはテーブルの存在です。同じ相対検索が役立ちます。 行の先頭間の距離は、目的のシーケンスの要素と見なされます-ポインター値の差はまったく同じになります。 テーブルが見つからない場合は、検索が繰り返され、ポインターの可能なサイズとそれらの間の距離が並べ替えられます（ポインターに加えて、テーブルに他のデータが含まれる場合）。 



 //    char* strings[] = {...}; //     ... struct Message { int characterId; int messageType; char* string; }; Message messages[] = {...};


ただし、すべての猫がパンケーキウィークであるわけではありません。一部のゲームでは、テーブルを使用せずにポインターで直接行にアクセスします。 次に、データの視覚的な分析や逆アセンブラーの使用など、可能な限り差異が計算されます。 もう1つの「昔ながらの」方法があります。バイトを置き換えることでデータセクションが「腐敗」し、ゲームに影響があるかどうかを確認します。 したがって、除外メソッドによって検索範囲を縮小すると、任意の行の出力を担当するコードセクションをローカライズして、ポインターを見つけることができます。 そのような目的のために、 PogankaやVisual Pogankaのような特化したツールもあります。



テーブルを使用しないと、ポインターは実行可能ファイルのコードスペース全体に散らばり、場合によっては単一のコピーに散らばります。 テキストが1か所に保存されている場合、この問題はテキストをスキャンし、各行の先頭へのすべてのポインターを見つけることで解決できます。 ほとんどの場合、これは難しくありません-アドレス空間の性質により、ポインター値と別の値との衝突の可能性は最小限です（たとえば、メモリは0x08000000-0x09FFFFFFの範囲でアドレス指定されるか、データセクションはアドレス0x00472000で始まります）。











しかし、翻訳者にとっては、たとえばゼロアドレスから開始するなど、メモリのアドレス指定があまりうまくいかないこともあります。 そして、衝突を確実に回避することはできません...同じ値を持つポインターまたはデータであるかどうかを確認するために、複数回発生する各値を手動でチェックする必要があります。 また、テキスト自体もファイル全体に散らばっている場合、 IDA Pro用の何らかのスクリプトまたはプラグインを記述することによってのみ、ポインターの検索プロセスを自動化できます。



いずれにせよ、忍耐と労力がすべてを粉砕します。 一度ポインターを見つければ十分であり、このタスクに煩わされずに次のステップに進むことができます。



テキスト抽出



テーブルからテキストを「抽出」する方法は、翻訳者の組織レベルによって異なります。 そのため、最も組織化されていない人（通常は初心者）はまったく気にせず、16進エディタでテキストを直接翻訳します。 もう少し深刻です-PokePerevodや同じTranslhexionのようなすべてのプログラムを使用して抽出します。 より深い知識を持つ人々は、 Kruptarのようなより専門化された自動化ツールを使用します。 最先端の専門家は通常、独自のスクリプトまたはツールを作成して、プロセスを完全に制御できるようにします。



いずれにせよ、ほとんどの場合、プロセスは、エンコード、バイナリタグ、およびゲームで使用されるバイトコードに関する情報を使用して、バイトストリームを読み取りおよび編集可能な形式に変換します。







しかし、開発者が最初にテキストを個別に純粋な形式で保存したという事実はまったくありません。 非常に多くの場合、それは他のデータの一部にすぎません-レベルマップ、シナリオなど。 TES Construction Setのようなゲームエディターに精通している人は、それが何であるかを理解するでしょう。 このような場合、テキストは他のデータと一緒に保存されるため、その構造を「解析」し、テキストとその他の必要な情報を慎重に抽出する必要があります-たとえば、テキストに加えて、出力の座標やダイアログボックスのサイズなどのデータも変更する必要があります 時には、エディター全体がこのために作成され、開発者が使用するツールの機能を部分的に再現します。



一般に、テキスト抽出に対する独自のアプローチがあります。 まず、エンコードテーブルの標準形式は非常に単純であり、すべてのケースをカバーしているわけではありません。 たとえば、テキストの終わり、改行、または画面のクリアを示すためにどのバイトシーケンスが使用されているかを知ることが重要な場合があります。 また、ゲームでは、たとえば、ビットの特定の部分がパラメーターとして機能するマークアップコードを使用できます。 この場合、次のようにこのすべてを書き留める必要があります。 

 1E40=[size=0] 1E41=[size=1] 1E42=[size=2] ... 1E4F=[size=15]

拡張の本質は、命令については、ビットマスクと識別子をテーブルに直接書き込み、それらを決定してパラメータを読み取ることができるということです。 また、単純な文字シーケンスの代わりに、命令のテキスト表現をフォーマットしてバイトコードに戻す方法を説明する特別な文字列を使用できます。 つまり そのようなテーブルは、実行可能ファイルを原始的に逆アセンブルすることさえできます。



エンコーディングテーブル内のこのような命令レコードの形式は次のとおりです。 

 <OpcodeMask>?<OpcodeID>:<ValuesMasks>=<FormatString>


OpcodeMaskはマスク、OpcodeIDは識別子、ValueMasksはパラメーターのビットマスクのコンマ区切りリスト、FormatStringは命令のテキスト表現をフォーマットしてバイトコードに変換するための文字列です（printf関数のフォーマット文字列の修正版）。



たとえば、 1010iiii cccccccc



という形式の命令があり、ポップアップウィンドウに項目番号iの色が表示されています。 「ポップアップ（アイテム、色）」と呼びましょう。 この命令のマスクは11110000 00000000



で、16進数表現ではF000



ます。 命令識別子は10100000 00000000



に等しくなります。 C000



、およびパラメーターマスクは00001111 00000000



および00000000 11111111



、つまりそれぞれ0F00



および00FF



です。



このようなコード化されたテキストを用意しましょう。

0000: 4F 62 74 61 69 6E 65 64 20 61 6E 20 69 74 65 6D Obtained an item





0010: 21 C1 0F 0A 4E 6F 77 20 79 6F 75 20 63 61 6E 20 !##\nNow you can





0020: 6F 70 65 6E 20 74 68 65 20 64 6F 6F 72 2E open the door.







そのようなエントリがあるテーブルを使用してデコードしてみましょう。 

 F000?C000:0F00,00FF=popup(%d, %d)




次の結果が得られます。

アイテムを入手しました！ポップアップ（1、色）

これでドアを開けることができます。


テキストとうまく結合していません。 したがって、タグの助けに頼ろうとします。 XMLタグにします。 

 F000?C000:0F00,00FF=<popup item="%d" color="%d"/>


すでにはるかに良い：

アイテムを入手しました！<ポップアップアイテム= "1"色= "255" />

これでドアを開けることができます。


しかし、「魔法の定数」の代わりに、パラメータのテキスト表現を見た方が良いでしょう。 これを行うために、フォームで転送を宣言する機能を導入しました。 

 %{<member1>[=value],<member2>[=value],...}


この指示が3つのアイテムにのみ適用されるとします：番号0の爆弾、番号1のキー、および番号10のコイン： 

 F000?C000:0F00,00FF=<popup item="%{Bomb,Key,Coin=10}" color="#%2x"/>


どうやら、色についても、より馴染みのある表記法を適用しました。 もちろん、8ビットの色表現でそうするのは愚かなことですが、これはわかりやすくするために頭から取った例にすぎません。



次に、タグの強調表示をオンにするだけで、多少なりとも読みやすいテキストを取得できます。 

 Obtained an item!<popup item="Key" color="#0F"/> Now you can open the door.




命令のセマンティクスがそれほど重要ではない場合（つまり、翻訳がそれらを編集することを意味しない）、短いレコードに頼る方がよいです。 たとえば、次のように：

アイテムを入手しました！{ポップアップ：キー、0F}

これでドアを開けることができます。


パラメーターが1つだけの手順については、短縮記録フォームを提供しました。 以下は、2つの同等の文字列です。 

 FF00?3C00:00FF=<pause time="%d"> 3C??=<pause time="%d">


レコードからわかるように、疑問符でパラメーターのビットを強調表示し、残りのビットはマスクの単位ビットと見なされ、識別子は疑問符をゼロに置き換えることで取得されます。 残念ながら、そのようなレコードは、パラメーターと識別子の次元が4ビットの倍数である命令、つまり 1つの16進文字のサイズ。



テキストプレゼンテーション



これはプロセスの最も重要な部分の1つです。抽出されたテキストは翻訳者に直接送られるため、翻訳者の作業の生産性だけでなく、作成する構造的および意味的エラーの数もそれがいかに適切かによって異なります。 本質的に、このプロセスは、翻訳者の作業を可能な限り簡素化するような方法でテキストを抽出するメカニズムの改良です。



前の段落で説明したテキストを考慮すると、翻訳者にとっては、タグやその他のゴミの目的に関する短い余談で十分です。特に彼を怖がらせるものはありません。 しかし、私の意見では、テキストの表示における主な間違いは、圧倒的な量の技術情報であることに注意する必要があります。 ほとんどのアマチュアにとって、そのようなテキストは非常に異なって見えるでしょう：

ジェニカ：私はこの城でかなり奉仕してきました^

ジェニカ：レナ王女の姉。

サリサは父親と一緒に航海していた^

嵐が襲い、彼女は海で失われました。{END}


翻訳者の代わりに、「 これを翻訳するにはどうすればいいですか」と言います。



残念ながら、断片的な情報から判断すると、これは公式のローカリゼーションのプロセスでも発生します。ここでは、改行として「\ n」を含むXMLまたはINI形式のテキストを提供し、必要に応じて翻訳します。 同時に、そのような場合でも多くの人はWordを変更しません。Wordは、何らかの自動置換、またはさらに悪いことに、ほぼすべての類似した形式の構造を殺すことができます。



もちろん、自動化の範囲に関係なく、通常はすべての品質管理が手動で行われるため、これは簡単に完全なエラーを伴う可能性があります。かつて、「同僚」はダッシュを1つも使わずに非常に大きな翻訳をリリースしました。Wordはすべてを静かに食べ、テキストコンバーターが馴染みのない「正しい」ものに置き換えました。
