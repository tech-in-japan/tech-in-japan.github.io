ピアツーピアbase.network

ジョージ・オーウェル-1984



現代の世界では、さまざまな分散技術が活発に開発されています。 数年にわたり、Bitcoinピアツーピア支払いシステム、分散マイクロブログ（Twister）、分散メッセンジャー（Toxなど）などのプロジェクトが正常に機能しています。 本格的な分散サイトになりました。



部分的に、分散サイトの機能はすでにFreenetやI2Pなどのプロジェクトに実装されています。 ただし、このようなプロジェクトにはいくつかの欠点があります。 たとえば、現時点では、Freenetを使用して、データベースやスクリプトを使用するなどの動的コンテンツを作成または配布することはできません。 このようなネットワークにアクセスするには、特別なソフトウェアをインストールするか、巨大なデータベースをダウンロードする必要がしばしばあります。 多くのピアツーピアネットワークの応答時間は長くなります。 情報の公開から他のユーザーがそれを見るまで、多くの時間がかかります。



提案されたbase.networkシステムは 、ピアツーピアネットワークのこのような不利な点を排除し、集中型システムのすべての技術的利点と分散ネットワークでの情報交換の自由を結合するように設計されています。

これは、短い応答時間と、自律性、フォールトトレランス、およびスケーラビリティの特性を備えた安全な分散ネットワークを作成するための一種の試みです。 このプロジェクトの主な目標は、インターネット上の情報へのアクセスを制限するだけでなく、公開を制御、抑制する組織からの圧力の下でも機能する能力です。 プロジェクトのすべての側面は、オープンソースとして無料で利用できます。 これにより、ソフトウェアが要求されたとおりに動作することを確認でき、すべての開発者が情報の自由な流れを制限する試みからネットワークの保護を改善できます。





base.network-それは何ですか？



これは、検閲の可能性なしにデータの分散分散ストレージ用に設計されたピアツーピアネットワークです。 システムには中央サーバーがなく、個人や組織の管理下にはありません。 ユーザーにオンライン空間での言論の自由を提供するように設計されています。 システムは、ネットワークメンバーが無償で提供する共通のクラスターにサーバーノードを結合することで機能します。 すべてのデータはサーバーノード間で均等に分散され、中央ストレージを必要としません。 参加者は、Web上でさまざまな種類の情報を公開または受信するために、サーバーの帯域幅とディスク容量を提供します。 たとえば、ファイル、データ構造、順序付きリスト。



その結果、このようなデータストレージスキームを使用すると、ブログ、ニュースチャネル、コメント、プライベートグループ通信、メディアリスト、検索インデックス、ディレクトリなどの動的なアプリケーションを実装できます。 これらのアプリケーションの特徴は、システムの分散型アーキテクチャによる中断抵抗です。 もう1つの重要な機能は、アプリケーション開発者を含む第三者による検閲が不可能であることです。



システムにアップロードされるすべてのデータは、公開鍵暗号システムを使用して作成者が署名する必要があります。 作成者の公開キーを知ることにより、特定のデータの作成者をいつでも確認できます。 デジタル署名とデータ暗号化を作成するための暗号化アルゴリズムとして、システムはECDSA（楕円曲線デジタル署名アルゴリズム）に基づいたキー長が256ビットの非対称公開キー暗号化アルゴリズム（特に、secp256k1曲線が使用されます）とSHA-256ハッシュ関数を使用します。



ネットワークの仕組み



このシステムでは、他のピアツーピアネットワークとは異なり、サーバーとクライアントの概念が共有されます。 クライアントソフトウェアは通常のブラウザであり、他の多くのピアツーピアネットワークと比較して、通常のユーザーがネットワークにアクセスしやすくなります。



すべてのネットワークデータは、参加者が無料で提供するサーバーに保存されます。 理想的には、専用サーバーノードが適切なインターネットチャネルにアクセスできるため、ユーザーはネットワークリソースにすばやくアクセスできます。



したがって、ネットワークを維持するシステムの参加者は、サーバーリソースを提供します。 これを行うために、彼らは特別なサーバーソフトウェアをインストールします。これは、本質的に1人の小さなWebサーバーとデータベース、いわゆるサーバーノードです。 ノードを機能させるには、外部IPアドレス（v4）と空きポートを割り当てる必要があります。



データ保存



まず、データストレージの構成を理解するために、ネットワークスペース全体が厳密に指定されたボリューム、たとえば32 GBによって制限されていることを想像してください。 したがって、ネットワークは巨大なハッシュテーブルになり、そのハッシュアドレスにある各ドキュメントは、仮想ネットワーク空間に厳密に配置されます。 1つの物理マシンにすべてのネットワークデータを保存しないように（たとえば、Bitcoinシステムで行われるように）、仮想空間全体をいくつかのセグメントに分割します。 まず、たとえば8つのセグメントに分割します。 各セグメントは4GBです。 現在、専用の物理マシンは、データベースの完全なコピーを保存する必要はなく、その一部のみを保存することなく、個別のセグメントに個別に対応できます。 このような各サーバーノードは、複数のネットワークセグメントをサポートできます。 1つのセグメントに複数の独立したノードが対応します。 マシンの物理アドレスを把握し、どのマシンがどのセグメントにサービスを提供しているかについての情報があれば、そのアドレスでいつでも別のドキュメントにアクセスできます。 ノードへのアクセスは、標準のhttpプロトコルを使用して実行されます。



ユーザーがノードの1つにデータを追加すると、同じhttpプロトコルを使用して、同じセグメントにサービスを提供するすべてのノード間で情報が直ちに同期されます。







指輪



ネットワークスペースは一定の量（32 GB）に制限されているため、ネットワークは巨大なハッシュテーブルのように機能することが上で述べられました。 これはすべて、仮想空間の場合に当てはまり、いわゆるリングのフレームワーク内で制限されます。



リングシステムは、ネットワークの仮想ストレージを自由にスケーリングし、そのスペースを厳密に制限しないために開発されました。 現在のリングがデータでいっぱいになると、ネットワークは次の仮想リングを使用します。 後続の各リングのセグメント数は8倍増加し、セグメントのサイズも数倍増加します。 より正確には、それは2倍の倍数であり、異なるタイプのストレージでは、この数は異なります（以下のストレージタイプを参照）。 したがって、次の各リングの合計ボリュームは前のリングのサイズよりも1桁大きく、より多くのデータを配置できます。 ネットワークの人気の潜在的な成長と、その結果として、マシンの数の増加により、次のリングのサービスが可能になり、ネットワークは本質的に無制限の成長を実行できます。



たとえば、ファイルを保存する場合、最初のリングのセグメント数は8です。2番目のリングでは64などです。 セグメントのサイズは2倍に増加します。 したがって、後続の各リングの音量は16倍になります。 また、たとえば、3番目のリングでは16 TBになります。 そして、5日目はすでに4 PBです！







ゼロリングには常に1つのセグメントがあります。 通常、ネットワークの重要な情報が含まれています。

ドキュメントのアドレス指定



ネットワーク上でドキュメントを受信（または保存）するには、そのアドレスを知る必要があります。 アドレスには任意の文字列識別子を使用できます。 ハッシュは、この文字列アドレス（SHA-256アルゴリズムを使用）から計算されます-いわゆるuid（ドキュメントの一意の識別子）。 ファイルの場合、ファイルのコンテンツからのハッシュは同様の識別子として機能します。 受信したハッシュの最初のビットは、ドキュメントが配置されているセグメントの番号を示します。 例えば



ドキュメントURI： 



  D2/my-domain.base/path/document.txt


パス： 



  my-domain.base/path/document


uid-sha256（パス）（16進数）： 



  49ea72cbd1de7f2d...3cf23d49778


uid（oct）： 



  2236516262750736...44365113570


ドキュメントの座標： 



  storage: D ring: 2 uid: 49ea72cbd1de7f...f23d49778 segment: D22


これで、セグメント番号と、このセグメントにサービスを提供するノードのリスト、およびそれらの外部IPアドレスとポートがわかれば、任意の物理マシンにアクセスして、通常のhttp要求を使用してドキュメントを要求できます。 このようなリクエストを作成するために、ユーザーは特別なソフトウェアをインストールする必要はありません。 クライアントソフトウェアは、通常のブラウザにすることができます。



すべてのネットワークデータが作成されます。 公開時には、各ドキュメントには独自のデジタル署名が必要です。デジタル署名は、作成者の秘密キーによって形成されます。 その後、公開鍵を使用して各ドキュメントの作成者を確認できます。



システム内のすべての文書は、次のパラメーターによって特徴付けられます。

  • storage-ストレージのタイプ。 単一の文字として示されます。 例：F、N、D、P
  • ring-ストレージ内のリングの番号（整数）
  • uid-一意のリソース識別子（hex64）


一般に、単一のドキュメントまたはデータリストへのリンクは次のようになります。 



  <storage:char><ring:int>/<uid:hex64>




ファイルストレージなど、一部のタイプのストレージでは、オプションのリストが若干広くなります。 例：

  • 拡張子-ファイル拡張子。 コンテンツの種類について話します。
  • 援助 -著者の一意の識別子（長さが20文字の公開証明書からのハッシュ）は、指定されたアドレスの特定の著者のみに情報を要求する場合に必要です。


リンクの例： 



 F1/ef43c9adc8aeec6910b4373ba0d9fbf28048ed53ec631ab9fd6fa8ad00a8a3a0.txt D4/930bbc0fa952dcc8eb7e8a711ac953c61f76f86fecd1ba7b0b5c4e74834e12bf?aid=Ipf8SOrpxbgYnNMXszFk


ストレージの種類



現在、システムには4種類のデータストレージが実装されており、それぞれ異なる情報構造を保存できます。

  • F-ファイルストレージ
  • N-ドメイン名ストレージ、証明書
  • D -1人の著者のデータリスト
  • P-公開データリスト


それぞれについてもう少し説明します。



ファイルストレージ（ストレージタイプF）



ファイルは、文字列データとは異なり、アドレス指定にいくつかの詳細があります。 ファイルアドレス（そのuid）は、そのコンテンツからのsha-256ハッシュです。 したがって、異なるユーザーがネットワークにアップロードした同じファイルは同じアドレスになります。 ハッシュをチェックして、正しいコンテンツが指定されたアドレスに返されることを確認してください。 ファイルのコンテンツタイプ（Content-Type）は、リンク内で拡張子（拡張）として直接示されます。



たとえば、次のリンク 
  F1/ef43c9adc8aeec6910b4373ba0d9fbf28048ed53ec631ab9fd6fa8ad00a8a3a0.txt
パラメータを含むドキュメントは対応します：

  • storage = F-ストレージタイプ「ファイル」、
  • リング= 1-最初のリング、
  • uid = ef43 ... a3a0-ファイルの一意の識別子、
  • extention = txt-ファイルタイプ。コンテンツタイプ（content-type）「text / plain」に対応します。


uidが与えられ、8進数のシステムに変換されると、最初の桁でセグメント番号が取得されます。 「7」がそれに対応します。 これで、「F7」という名前のセグメントにサービスを提供するすべてのノードとノードの物理アドレスがわかったので、ドキュメント自体を取得できます。 



 curl -i http://5.9.123.140:8080/-/F7/data/ef43c9adc8aeec6910b4373ba0d9fbf28048ed53ec631ab9fd6fa8ad00a8a3a0.txt HTTP/1.1 200 OK Content-Type: text/plain; charset=utf-8 Cache-Control: max-age=315360000, public Expires: Sun, 17 Aug 2025 15:18:30 GMT Access-Control-Expose-Headers: x-base-author x-base-author: AA3Evvx1AR...i8UONOm92Q== Connection: keep-alive Transfer-Encoding: chunked TEST FILE


次に、受信したドキュメントのコンテンツからハッシュを確認します。 



 curl http://5.9.123.140:8080/-/F7/data/ef43c9adc8aeec6910b4373ba0d9fbf28048ed53ec631ab9fd6fa8ad00a8a3a0.txt|shasum -a 256 ef43c9adc8aeec6910b4373ba0d9fbf28048ed53ec631ab9fd6fa8ad00a8a3a0 -


すべてが正常です：要求されたuidと一致します。

ドメイン名（ストレージタイプN）

このタイプのストレージは、登録ユーザーのドメイン名、ドメイン所有者、公開証明書に関する情報を保存するために必要です。 このタイプのストレージは、ドメインレジストラー専用の記録に使用できます。 現時点では、彼が唯一の人物です。これは.base仮想ドメインゾーンレジストラーです。 レジストラの公開キーは、すべてのネットワーク参加者に知られており、クライアントおよびサーバーソフトウェアに直接ハードコーディングされています。

ストレージリングのゼロリングセグメントサイズは8 GBです。 理論的には、このサイズは、ドメインと証明書に関するすべてのデータをゼロリングの1つのセグメントに排他的に格納するのに十分な時間です。 したがって、ドメイン名による情報の取得は、異なるリングを検索することなく行われます。

したがって、タイプNストレージは一種の巨大なハッシュテーブルです。 ドメイン名によって、ドメイン所有者とその公開鍵に関する情報を要求できます。

データリスト（ストレージタイプD）

ネットワークには、単一のドキュメントに加えて、データのリストを操作して、いわゆるテープを形成する機能があります。 リストは、単一のドキュメントのように、特定の一意の識別子uidによって配置されます。 リストuidは、文字列アドレスからのsha256ハッシュです。 ただし、このタイプのストレージでは、最初の2つとは異なり、特定のソート順とオフセットを使用して、特定の著者のドキュメントのリストを要求することができます。 実際、このストアは、順序付けされたデータを返すことができる単純なインデックスを実装しています。

ユーザーにとってのネットワークの仕組み



平均的なユーザーの場合、特別なソフトウェアをインストールせずにシステムの操作を開始します。 これを行うには、通常のブラウザを使用します。 ALL * .base.network / *アドレスの場合、JavaScriptで記述されたクライアントカーネルコードを含む静的なHTMLページが返されます。







現在、同様のスキームには、既存のドメインネームシステムを使用する必要があるアクセスのためのサイトおよびbase.networkドメインの形式の中央エントリポイントが確実にあります。 ただし、この仮定は、ユーザーが特別なブラウザプラグインをインストールすることで、後で簡単に解消できます。 すべての* .base.networkアドレスに対する同様のプラグインは、現在のドメインネームシステムを使用せず、外部の中央リソースに追加の要求を行うことなく、システムコアのベースコードを含む静的ページを返します。



クライアントコアには、ノード（IPアドレスとポート）の初期リストが含まれています。 エンジンが最初にロードされると、スクリプトはこのリストから任意のノードに接続し、そこから新しいネットワークマップを取得しようとします。これは後でLocalStorageブラウザーに保存されます。 スクリプトは定期的にリストからノードをポーリングし、ネットワークに関する情報を更新します。 ネットワークマップには、すべてのノードのリスト、それらのアドレスとポート、サーバーソフトウェアバージョン、および各ノードがサービスを提供するセグメントのリストが含まれています。 これで、クライアントはネットワークからドキュメントを取得するのに十分な情報を取得できました。 どの文書アドレスについても、そのセグメントがどのセグメントにあり、どのセグメントによってこのセグメントが処理されるかがわかります。 ネットワークからのデータの高速ダウンロードのために、クライアントはそれに最も近いノードを選択します。 クライアントは、ダウンロードしたデータの信頼性をハッシュで検証し、公開キーを使用して作成者の署名を検証できます。



最初にネットワークに入ると、秘密鍵と公開鍵を含むユーザーの匿名証明書が生成されます。 この証明書は、core.base.networkドメイン名前空間のブラウザーのLocalStorageに保存されます。 他のドメインのスクリプトは、このスペースに直接アクセスできず、秘密キーに関する情報を受信できません。







すべての外部http要求は、クライアントコアフレーム（core.base.network）から排他的に行われます。 サイトエンジンは、このための特別なAPIを使用して、このフレームを通じてのみシステムと連携できます。 APIは、サイトエンジンにファイルをネットワークにアップロードし、データを投稿し、データフィードを受信し、フィードにデータを追加するイベントをサブスクライブする機能を提供します。







公開時に、カーネルはサイトエンジンによって送信されたデータに署名します。これには秘密キーを使用し、ネットワークマップを手に入れて、必要なノードに対して直接POST要求を実行します。



登録



ユーザーがネットワーク上のサイトに初めてアクセスすると、カーネルは自動的に匿名証明書を生成します。 匿名証明書には、ネットワークでの作業にいくつかの制限があります。 匿名ユーザーは、ファイルをネットワークにアップロードしたり、個人のドメイン、サイトを登録したり、アクセス権に対応する制限があるトピックにコメントを残したりすることはできません。







証明書の登録は、この証明書によって修正された個人ドメインの登録と同時に行われます。 現在、ドメインと公開証明書の概念は密接にリンクされています。 登録中に、ユーザーは公開鍵とドメイン名を暗号化された形式でレジストラに送信します。登録された証明書をバインドするために、これに登録者は必要になります。 レジストラはユーザーの公開キーに署名を追加し、生成された証明書とドメイン所有者にデータをドメイン名ストレージ（ストレージN）に配置します。ここで、他のユーザーが利用できるようになります。







レジストラが誰であるかについて少し述べる価値があります。 レジストラは、誰もが信頼するルート証明書です。 レジストラの公開キーは、クライアントおよびサーバーソフトウェアに直接埋め込まれます。 すべての登録ユーザーの証明書には、署名が含まれています。 レジストラは、ドメインとその所有者に関する情報に署名します。 ランダムなドメイン登録に対する一種のリミッターです。 これは単なるドメインレジストラであり、情報サイトを変更したり、何らかの形で制御したりする技術的な能力はありません。 彼は、証明書の物理的な所有者、IPアドレスについて何も知りません。



現在、レジストラはネットワークの組織における特定の中心人物であり、分散デバイスシステムの精神にやや反しています。 その後、ドメインの登録と移管を完全に分散化する計画。 既存のドメイン登録スキームは、ネットワークの速度だけのために選択されます。 ユーザーは、すべてのドメイン名の重いデータベースをダウンロードする必要はなく（たとえば、ネームコインで行われたように）、1つのレコードのレジストラの署名を確認するだけです。 さらに、現在の登録方式では登録数が厳密に制限されていないため、参加者の迅速な入力とシステムの人気のさらなる拡大が保証されます。



そのため、証明書がレジストラの署名を持っている場合、証明書は登録されます。 ドメイン所有者は、公開証明書の秘密鍵を持っている人です。 既存の証明書を使用すると、個人ドメインの名前など、所有者に関する情報をいつでも取得できます。 そしてその逆。 ドメイン名により、所有者の公開証明書をいつでも取得できます。 ドメインと証明書に関する情報は、レジストラによって署名されており、Nストレージから常に取得し、信頼性を検証できます。



現在、招待により登録が行われています。 招待コードは、Gmailアカウントを持っている人なら誰でも入手できます。 Gmailアカウントごとに1つの招待を取得できます。 このような招待システムは、システムの運用の初期段階で制御されないドメイン登録を何らかの形で制限するためだけに導入されました。ユーザーが入力したGmailアカウントはどこにも保存されず、招待コードを送信するために直接必要なだけです。匿名性を維持するために、レジストラは招待コードをドメインおよび証明書所有者に関する情報に関連付けませんが、このコードが既に使用されていることにのみ注意します。

サーバーソフトウェアのインストール



プロジェクトとネットワークの開発をサポートするために、誰でもサーバーリソースを提供できます。これを行うには、サーバーにサーバーノードをインストールして実行する必要があります。



  1. ノードをインストールする前に、Node.jsプラットフォームがサーバーに既にインストールされていることを確認するか、指示に従ってインストールする必要があります
  2. githubからリポジトリをダウンロードする 



     git clone https://github.com/basenetwork/base.server-node


  3. Node.js用のsqlite3モジュールをインストールする 



     cd base.server-node && npm install sqlite3 --build-from-source && cd ..


  4. 実行ノード 
     nohup base.server-node/base.node.js --size=32 >/var/log/base.node.log &


サイズパラメータとして、ノードの下に割り当てる準備ができている空きディスク領域（GB）を指定する必要があります。

ノードが機能するには、外部IPアドレスと空きポートを割り当てる必要があります。デフォルトでは、システムはオペレーティングシステムのネットワークインターフェイスリストにある外部IPアドレスのいずれかを自動的に使用しようとします。起動オプション--hostおよび--portを使用して、IPアドレスとポートを手動で指定することができます。 



 nohup base.server-node/base.node.js --size=32 --host=41.34.55.66 --port=2222 >/var/log/base.node.log &


Webサーバーに対してhttpリクエストを行うことにより、ノードの動作を確認できます。 



 curl http://41.34.55.66:2222/-/about
