SATA、SAS、SSD、およびRAIDアレイとそれらの比較

理論上、誰もがディスクサブシステムのパフォーマンスパラメーターを知っています。 しかし、実際には何ですか？ 多くの人がこの質問をし、一部の人は仮説を立てます。 私は一連のテストを実施し、「誰が誰であるか」を決定することにしました。 すべての既知のdd、hdparmユーティリティを使用してテストを開始し、次にfio、sysbenchに移動しました。 UnixBenchおよび他のいくつかのアナログを使用して、一連のテストも実行されました。 多数のグラフがプロットされましたが、テストを続けると、このソフトウェアの大部分は異なるディスクの適切な比較には適さないことがわかりました。

fioを使用して、SAS、SATAの比較表またはグラフを作成することはできましたが、SSDをテストすると、結果が完全に不適切であることがわかりました。 もちろん、私はこのソフトウェア全体の開発者を尊敬していますが、その瞬間、一連の模擬テストではなく、実際の状況により近いものを作成することが決定されました。



テスト結果がプロセッサの種類、周波数、またはその他のパラメータによって歪ま​​ないように、テストパラメータとマシン自体が選択されたことをすぐに言わなければなりません。



テスト1


ファイル作成


8サイクルを超えると、混oticとしたコンテンツと、サイクルあたりのファイル数が徐々に増加する小さなファイルが作成されました。 実行時間は各サイクルで測定されました。



このグラフは、KINGSTON SV300S3 SSDのファイル作成が高速であり、その数にほとんど依存していないことを示しています。 また、これらのディスクのスケールはより単純であることに注意してください

ハードウェアRAIDのSASディスクによると、速度はRAIDのタイプに依存しますが、ディスクの数にはまったく依存しないことがわかります。

しかし、判明したファイルの作成ではなく、ファイルの上書きに時間がかかります。 2番目のテストに進みましょう。

SATA 7,200 rpm MB2000GCVBRディスクの恐ろしい写真はすぐにわかります。 遅い録音と2x 300GB SAS SEAGATE。 したがって、残りの部分を明確にするために、スケジュールから外すことにしました。



最速のサブシステムはシングルSSDキングストンでした。 2位と3位は、8x SEAGATE ST3300657SSと4x SEAGATE ST3300657SSによって獲得されました。 また、アレイ内のSSDの数が増えると、速度がわずかに低下することもわかります。



テスト3


MySQL SQLクエリの組み合わせINSERT、SELECT、UPDATE、DELETE


InnoDBテーブルは、次の構造で作成されました。

CREATE TABLE `table` (

`id` int(10) unsigned NOT NULL AUTO_INCREMENT,

`time` int(11) NOT NULL,

`uid` int(11) NOT NULL,

`status` varchar(32) NOT NULL,

PRIMARY KEY (`id`),

FULLTEXT KEY `status` (`status`)

) ENGINE=InnoDB DEFAULT CHARSET=cp1251;







いくつかのクエリが同時に生成されました。

-INSERT;

-PRIMARY KEYによる選択による更新。

-全文を選択した更新（24から4文字で検索）：WHERE `status` LIKE '％（string）％';

-PRIMARY KEYによる選択でDELETE FROM;

-キーを使用せずに選択したDELETE FROM：WHERE `time`>（int）;

-キーを使用せずに選択したSELECT：WHERE `time`>（int）;

-PRIMARY KEYによる選択でSELECT;

-全文選択での選択（24文字のうち4文字で検索）：WHERE `status` LIKE '％（string）％';

-キーレス選択を伴うSELECT：WHERE `uid`>（int）。

テスト6


線形読み取り




ここでは、わずかに異なる画像が表示されます。 リーダーは、4倍のKINGSTON RAID10 SSDです。ファイルサイズの増加に伴う結果の変化は最小限で、RAID10の8倍のSEAGATEは、700 Mbpsおよび600 Mbpsの速度で速度が徐々に低下します。

1x SSD KINGSTONと2x SSD KINGSTON RAID1のラインは一致していました。 簡単に言えば、線形読み取りには、RAID10または単一のディスクを使用することをお勧めします。 RAID1の使用は保証されません。

2x SAS RAID1と4x SAS RAID10は非常に似ていることがはっきりとわかります。 しかし、ディスクの数が2倍に増えると、速度が大幅に向上します。

2x SSD Intel RAID1の速度は10 GBから30 GBの間でわずかに低下しますが、SATA RAID1と同じ速度になります。

また、SASディスクの大規模アレイの比較を行いました。これは、その数よりもディスク速度に大きく依存していることを示しています。





在庫を確認する時が来ました。

多くの車がありましたが、十分ではありませんでした。 残念ながら、SSD INTEL SSDSC2CT12のインジケーターがそれらの機能であるか、RAIDコントローラーの機能であるかを判断できませんでした。 しかし、私は同じコントローラを信じています。

  1. アレイ内のSASディスクの数が増えると、すべてのインジケータが改善されるだけです。
  2. MySQLの場合、低速サブシステムはSATA RAID1およびSAS RAID1です。 残りには違いがありますが、それほど重要ではありません。
  3. 線形記録の場合、RAID10のSASディスクの大規模アレイとSSDの両方が適しています。 SSDからアレイを使用しても意味がありません。 コストが上昇し、生産性が整っています。
  4. 線形配列は、線形読み取りに適しています。 しかし、実際には、林。 書かずに読んで、ほとんど会うことはありません。
  5. 単一のSSDまたはソフトウェアRAIDでのランダム読み取り。
  6. ランダム録画の場合は、SSDのハードウェアRAIDを使用することをお勧めしますが、単一のSSDではそれほど効果はありません。
  7. 最も重要な指標の1つであるランダム読み取り/書き込みは、SSDからのハードウェアRAIDで最高の結果をもたらします。
  8. 上記のすべてを要約すると、ほとんどのタスクでは、SASの大きなアレイ（> = 8）またはSSDのハードウェアRAIDを使用することをお勧めします。 ただし、一部のタスクでは、単一のSSDを使用する方が適切です。
  9. 主に当社の市場で提供されているSSDの量に基づいて、VDSノードでは、大型SASアレイまたは平均的なプロセッサーと単一のSSDと組み合わせたプロセッサーの最大パフォーマンスを使用する価値があります。 2台のSSDにhw raidを使用すると費用がかかると思います。
  10. 高速なシステムが必要で、ハードウェアRAIDに大きな2x SSDディスクスペースが必要ない場合は、最良の選択です。 パフォーマンスを犠牲にして少し節約したい場合は、ソフトウェアRAIDで1つのSSDまたは2つのSSDを使用できます。
