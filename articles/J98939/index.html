Xaps Minifier。 Silverlightアプリケーションのサイズを削減するためのVisual Studio 2010アドオン

私は、Silverlightアプリケーションを常に使用し、リリースを定期的にアップロードしています。 通常、 MVVMパターンとPrismの実装を使用します。 その結果、アプリケーションアセンブリとマニフェストを含むいくつかのXAPファイルが作成されます。



このアプローチに従って作業するすべての人は、ほとんどのXAPファイルに重複したアセンブリが含まれていることに気付きます。 たとえば、 Prismライブラリを使用する場合、ほぼすべてのXAPファイルにこのライブラリのすべてのアセンブリが含まれます。 Prismは各XAPファイルに約300 Kbを追加します。これにより、アプリケーションサイズが1 Mb以上増加します（4〜5個のXAPファイルがある場合）。 さらに、追加のライブラリ（主にUI要素）により、アプリケーションのサイズがさらに増加する可能性があります。



これらすべての事実により、XAPファイルのサイズを削減する方法を探し始めました。



アイデア



Jeff Prosiseに関するブログ投稿に出くわしたとき、私は1つの問題に取り組んでいました。 彼は、アセンブリをアプリケーションに追加できるが、XAPファイルには追加できないと述べました。 これを行うには、パラメータを設定するだけです



CopyLocal=false







参照リストにある必要なアセンブリごとに。 この場合、プロジェクトは目的のアセンブリを参照しますが、コンパイル中にbinフォルダーに追加しません。



これにより、アプリケーション内のすべての重複するアセンブリを添付できるという考えが得られました。



CopyLocal=false







メインXAPファイルのアセンブリパラメーターのみを変更することなく。 これは、 アプリケーションドメインに自動的にロードされるすべての重複アセンブリを含む必要があるメインXAPファイルであり、XAPファイルの残りのすべてのアセンブリで使用できます。







上記の図は、上記のルールに従ってソリューションが変更された場合にソリューションがどのように更新されるかを示しています。 各プロジェクトが個別のXAPファイルであると想像してください。 この場合、 プロジェクト1とプロジェクト2には同じペアのアセンブリ（ アセンブリ1とアセンブリ2 ）が含まれており、それらにProject1.xapとProject2.xapから除外するようにCopyLocal=false



を設定します。



アセンブリ1は既にメインプロジェクトにあるため、 メインプロジェクトに追加する必要はありません。 対照的に、アセンブリ2（ アセンブリ2 ）をメインプロジェクトに追加して、最初と2番目のプロジェクトの操作性を確保する必要があります。



プロジェクト2（ プロジェクト2 ）にはアセンブリ4（アセンブリ4）が含まれています。これはメインプロジェクトに既に存在します-このアセンブリをプロジェクト2から除外します。



プロジェクト3（ プロジェクト3 ）は変更できないため、 一意のアセンブリのみが含まれます。



最適化されたものを要約しましょう：

プロジェクト 最適化前のXAPのアセンブリの数 最適化後のXAPのアセンブリの数
プロジェクト1 4 2
プロジェクト2 4 1
プロジェクト3 3 3
主なプロジェクト 3 4
合計 14 10


ご覧のとおり、1つのアセンブリによるメインプロジェクトの増加にもかかわらず、アプリケーション内のファイルの総数は減少しています。



当然、これらの変更を手動で行うと処理が遅くなり、エラーが発生する可能性があります。 そのため、これらの操作を自動化するVisual Studioのアドオンを作成することにしました。



アドオンの実装



XAPファイルの最適化が行われる一般的なアルゴリズムのみを説明します。





  1. すべてのSilverlightプロジェクトを取得する 。 検索は、プロジェクトファイルにパラメータが含まれているという事実に基づいています



    ProjectTypeGuids







    プロジェクトの種類（コンソールアプリケーション、asp.netアプリケーションなど）を示すGUIDが含まれています。
  2. メインプロジェクトを検索します。 このプロジェクトから開始して、アプリケーションが起動し、そこに他のプロジェクトからの重複したアセンブリを追加する必要があります。 プロジェクトパラメータ値に基づく検索



    SilverlightAppEntry







    このパラメーターが有効な（プロジェクトに存在する）クラスの名前で初期化されている場合、そのようなプロジェクトはアプリケーションのメインです。

  3. 重複したアセンブリのリストを取得します。 すべてのSilverlightアプリケーションプロジェクトがバイパスされ、2回以上発生するすべてのアセンブリが記憶されます

  4. メインプロジェクトに複製アセンブリを追加します（存在しない場合）。

  5. アセンブリのパラメーターを設定します



    CopyLocal=False







    メインプロジェクトを除く、すべてのプロジェクトで発生します。



このアルゴリズムの結果として、Visual Studioの現在のアプリケーションが更新されます。 すぐにコンパイルして、XAPファイルのサイズの違いを確認できます。



仕事の可視化



Visual Studioでは、さまざまな組み込みメカニズムを使用して、実行中のアドオンの現在の状態を表示できます。



4つのオプションを選択しました。

  1. アニメーション 。 アニメーションアイコンは、アドオンの操作全体を通してステータスバーに表示されます

  2. 進捗状況のインジケータ 。 ステータスバーには、重複する各アセンブリに対する最適化の割合が表示されます

  3. テキストメッセージ ステータスバーには、現在の完了手順に関するテキストメッセージが表示されます

  4. OutputWindowでのアクションのロギング 。 アドオン中に別のウィンドウにすべてのメッセージが表示されます。



アドオンを公開



Visual Studio Galleryでアドオンを公開するには、 Liveアカウントを持ち、ギャラリーのWebサイトにアクセスし、[ アップロード ]をクリックしてアドオン公開ウィザードに従ってください。



アドオンのインストール



アドオンをインストールするための最速かつ最も便利なオプションは、 Visual Studio Galleryを使用することです。 これを行うには、Visual Studio 2010で、 Visual Studio Extension Manager （メインメニュー-[ツール]-[Extension Manager]）を開き、[オンラインギャラリー]を選択して、[オンラインギャラリーの検索]フィールドに「xap」と入力します。 その後、Xaps Minifierを選択して[ダウンロード]をクリックします。アドオンのダウンロードとインストールが開始されます。







アドオンを使用する



Visual Studioをインストールして再起動したら、Silverlightプロジェクトを読み込み、ソリューションのコンテキストメニューを呼び出して、Xapの最小化アイテムを選択する必要があります。







アプリケーションの分析と最適化のプロセスが開始され、進行状況がVisual Studioのステータスバーに表示されます。 最適化プロセスが完了すると、アプリケーションから除外できるアセンブリの数、メインプロジェクトで増加したアセンブリの数、変更されたプロジェクトの数に関する情報を示すダイアログボックスが表示されます。



実際のプロジェクトの最適化



最初のテストプログラムはデモアプリケーションで、Remix10カンファレンスで発表しました。 このアプリケーションには、5つのSilverlightプロジェクトと4つのXAPファイルが含まれていました。 最適化されていないアプリケーションのサイズは1.2 Mbでしたが、最適化後は500 Kbに減少しました。 同時に、メインプロジェクトのサイズは120 Kbから400 Kbに増加しました。



2番目のアプリケーションは、 PDC09で John Papaが発表したPrism Demoアプリケーションです 。 このアプリケーションには、7つのSilverlightプロジェクトと4つのXAPファイルが含まれ、XAPファイルの合計サイズは5.7 MBでした。 最適化後、このアプリケーションは1.6 Mb （!!!）に減少しました。



既知の類似体



Visual Studioアドオンの実装の直接の類似物を知りません。 しかし、Silverlightアプリケーションのサイズを最適化するというトピックは関連性があるため、 Component OneやTelerikなどの企業がツールを提供しています。 私の考えとは異なり、彼らは完全に異なるアプローチを使用します。 彼らのツールは、完成したXAPファイルを分析し、未使用のコード（クラス、XAMLコード）を見つけて、アセンブリから削除します。



私の意見では、これはやや物議を醸す決定です。 私の主張は次のとおりです。

  1. クラスを動的にロードできますが、これを追跡することはできません。 この場合、削除すべきではないすべてのクラス/コントロールを覚えておく必要があります。 これにより、アプリケーションの安定性が大幅に低下したり、テスト部門の負荷が増加したりする可能性があります。私のアドオンはアセンブリを変更しないため、はるかに安全です。

  2. 有料ライブラリのサードパーティプロバイダーは、ライセンス契約にライブラリの変更に関する合意を含めるとは思わない。 これは、まさにコンポーネント1で起こっていることです。 これは単に違法であるように思えます。

  3. 同僚のアレクセイと私は、このプロジェクトに約1週間を費やしました。 上記の企業は、はるかに多くの時間と労力を費やしました。

  4. 私のアドオンは、アプリケーションで使用されているライブラリに依存しません。 同時に、 Telerik Assembly MinifierはTelerikライブラリのみを最適化できます。

  5. アドオンを使用する場合、開発者はライブラリの新しいバージョンが表示されたときにアプリケーションを再最適化する必要はありません



いずれにせよ、私のアドオンは、 Component One XapOptimizerおよびTelerik Assembly Minifierツールと組み合わせて使用​​できます 。



現在の制限



アドオンは現在、次のものでは機能しません。

  • Silverlight for Windows Phone 7;

  • Windows Phone 7のXNA。

  • SymbianのSilverlight。



GUIDのみが異なるこれらのタイプのプロジェクトのサポートを追加するのに技術的な困難はありません。 近い将来に実装されます。



また、アドオンでは、XAPファイルに動的に追加されたアセンブリを削除することはできません。 これは、追加されたアセンブリが他のアセンブリに依存し、後者が明示的にプロジェクトに追加されていない場合に発生します。 この場合、これらのアセンブリをXAPファイルに含まれるすべてのプロジェクトに手動で追加し、アプリケーションを再最適化することをお勧めします。



追加アルゴリズムに伴う最後の制限は、1つのXAPファイルで構成されるアプリケーションを最適化できないことです。
