GentooでのPerlモジュールのインストール

Perlエコシステムの最大の欠点はモジュール管理です（この役割のもう1つの候補は長期的なPerl6ですが、それについては説明しません）。 興味深いことに、このエコシステムの最大の利点は、 CPANモジュールの単一のアーカイブの存在です。 驚くべきことに、彼らはモジュールを組み立てて整理することができましたが、便利なインストール/更新/削除を実装することができました-いいえ。



豊富な選択肢...または別のTIMTOWTDI



この問題には多くの代替アプローチがあります（そして、それらの数はそれらのどれも問題を十分に解決しないことを間接的に示します）： cpan 、 cpanplus 、 cpanminus 、 pip 、 cpansite 、 minicpan / mcpani 、 perlbrew 、 cpan- outdated 、 cpan -listchanges 、 local :: lib 、...



だから、私たちは持つことができます：
  • perl自体のいくつかのバージョン（もちろん、それぞれ独自のグローバルモジュールを備えています） ユーザーのホームディレクトリにインストールされます（perlbrewを参照）。
  • グローバル（perlの開始時に利用可能）およびローカル（通常は個別のプロジェクト内またはユーザーのホームディレクトリにある任意のディレクトリ/ディレクトリからのプラグイン）モジュール。
  • グローバルモジュールには、コア（perlに付属）、サイト（管理者が手動でインストール）、ベンダー（OSのパッケージマネージャーによってインストール）の3種類があります。
  • すべてのグローバルモジュールはサブディレクトリ "version.perl.perl /"にあり、これらのディレクトリは自動的にクリーンアップされません。 そして、perlの新しいバージョンをインストールすると、新しい同様のディレクトリが作成されます。 そして、perlは利用可能なすべての以前のバージョンのディレクトリからモジュールをロードします。 したがって、コア+サイト+ベンダーにperlアップデートの数を掛けます。これは、グローバルモジュールが存在するディレクトリ/オプションの数です。
  • モジュールのソースも異なります。CPAN、プライベートモジュールを備えたローカルCPANオーバーレイミラー、独自のもの、またはCPAN互換システムにないインターネットモジュールからダウンロードしたもの。
そして、このすべての喜びを管理する必要があります：インストール、更新、... Gentooには、グローバルPerlモジュールの管理を簡素化するg-cpanユーティリティがあります。



作業環境



まず、通常どのような条件で作業する必要があるかを少し指定します。
  1. ホスティングにモジュールをインストールすることではなく、サーバー全体で作業します 。
  2. このサーバーは、 いくつかのPerlプロジェクトを実行します 。
  3. 積極的にPerlで書く人-独自のモジュールを書く。 さらに、そのすべてのモジュールをCPANに配置できるわけではありません。 複数のプロジェクトで作業するすべての人は、異なるプロジェクトで同じモジュールを使用します。 したがって、この状況は遅かれ早かれ、モジュールのローカルCPANミラー/オーバーレイを「通常の」手段でインストールおよび更新できるようにする必要があります-これは全体として問題を解決しませんが、人生は楽になります。 CPANローカルミラー/オーバーレイサポートが必要です 。
  4. 問題は常に発生します：プロジェクトに必要なモジュール（私たちとCPANの両方）をグローバルに（私たちは自分のサーバーを持っていることを思い出します）またはローカルにインストールします。 ここには単一の答えはありません。両方のアプローチには長所と短所があります。 私は自分自身でこのように決めました：インストールされたすべてのモジュールを管理（インストール、更新、古い/穴のあるバージョンを追跡、削除）する必要があり、すべてのモジュールを1か所で組み立てることができないため（グローバルモジュールは他のシステム全体で使用されるため、削除することは不可能です）アプリケーション、およびほとんどのプロジェクトでローカルモジュールなしでは実行できません）、少なくともそのような場所の数とインストールされているモジュールの総数を最小限に抑える必要があります 。 これは、 ほとんどのモジュールをグローバルにインストールする必要があるという事実につながります。グローバルモジュールのバージョンが特定のプロジェクトと互換性がない場合にのみ、ローカルにモジュールをインストールする必要があります。 はい、このアプローチの主な欠点は、グローバルモジュールを更新すると一部のプロジェクトが破損する可能性があることです-しかし、私の経験では、これはめったに起こりません。動作中のサーバー-とにかくこれを行う必要がありますか？ :)極端な場合、メイン開発者のワークステーションはテストサーバーの役割に適していますが、更新の最小限のテストが必要です。
CPAN互換のローカルミラー/オーバーレイ（cpansiteを参照）からモジュールをインストールし、ほとんどのモジュールを単一のコピーにグローバルにインストールすることに制限することで、すでに少し楽になりました。 次の問題：モジュールには、 コア、サイト、ベンダーという3つのグローバルインストール場所があります。 そして、このため、しばしば対立が生じます。 Gentooはかつてvendor-site-coreを優先していました。 このため、cpan / cpanplus /などを介して手動でインストールされます。 多くの場合、サイト内のモジュールは利用できませんでした。 ベンダーのバージョンが古い。 また、cpanなどのユーティリティは、モジュールが機能しないことを認識せずにモジュールを際限なく更新できます。 これは、ベンダーからすべてのモジュールを削除し、/ etc / portage / profile / package.providedに手動で書き込むことで解決しました。これにより、ポートがそれらを再度インストールしないようになりました（その後、ポートがdev- perl / *またはperl-core / *。必要に応じて、cpanを介してこれらのパッケージをインストールし、package.providedに登録します。 Site-vendor-coreが優先事項になりましたが、これも問題を完全には解決しません：cpanなどのユーティリティによって更新されたコアモジュールの一部（またはすべて、チェックしなかったかもしれません）は、サイトではなくコアにインストールされ、それらのバージョンの上に、 perl自体とともにインストールされます。 これにより、2つの予期しない効果が生じます。まず、これらのコアモジュールの異なるバージョン（perl-core / *パッケージから）がベンダーにインストールされている場合、cpanユーティリティによるモジュールの「更新」にもかかわらず、ベンダー; 次に、dev-lang / perlパッケージを再インストールすると（たとえば、revdep-rebuildが原因で）、cpanを使用して手動でインストールされたすべてのコアモジュールの更新がロールバックされます。



g-cpan



上記の問題、およびcpan / cpanplusのような「通常の」ツールの不便さを考慮して、すべてのグローバルモジュールが移植ツールやシステム内の他のすべてのパッケージをインストール、更新、削除できるようにスムーズに近づいています。 同時にサイト内のモジュールの手動インストールを完全に放棄すると、優先順位の問題/モジュールの重複/ perlの再インストールもすべて解決されます。 残念ながら、移植にはCPANモジュールのごく一部のパッケージしかなく、私たち自身のプライベートモジュール用のパッケージはありません。 この問題を解決するために、g-cpanユーティリティが開発されました-任意のCPANモジュール（CPAN互換ミラー/オーバーレイからのプライベートモジュールを含む）のパッケージを生成し、ローカルのPortageオーバーレイに保存するか、単にオンザフライでインストールできます。 数年前にそれを使用しようとしたとき、それは非常にうまく機能せず、実質的に役に立たなかった。 しかし、今ではすでに完全に機能しています-少なくとも300のモジュールをインストールすることができ、そのうちの1つだけを手動でebuildを編集する必要がありました。



一方で、g-cpanを使用することの好ましい効果として、あるサーバーでPerlモジュール用の標準​​のGentooベースのバイナリパッケージを作成し、他のサーバーに非常に迅速にインストールする機会を得ます（モジュールの通常のインストール、特にテストには数時間かかり、一部のモジュールでは、インストール中にインタラクティブな対話も必要です。



インストールとセットアップ
  1. 私たちは置きます： 

    emerge g-cpan
  2. g-cpanがebuildを作成するオーバーレイを設定する必要があります。 

     echo "GCPAN_OVERLAY=/usr/local/portage" >> /etc/make.conf
  3. CPANミラーのリストを設定します（最初のアイテムとしてローカルミラー/オーバーレイを含む）。 最も簡単な方法は、cpanから構成済みのリストをコピーすることです。 

     cpanmirrors=$(cpan -J | perl -0777 -ne 'eval;print"@{$CPAN::Config->{urllist}}"') echo "cpan $cpanmirrors" >> /etc/portage/mirrors
  4. インストール中にモジュールをテストする必要があるかどうか-それぞれが自分で決定しますが、特に私のモジュールに対してテストを実行することを好みます。 インストール中のperlモジュールのテストを高速化するには、パッケージのカテゴリ全体にフックを設定する機能を追加する必要があります。 可能なアプローチの1つは、ディレクトリ/etc/portage/bashrc.d/とこのファイル/ etc / portage / bashrcを作成することです： 

     # Useful variables and functions: # # EBUILD_PHASE= # clean # setup # unpack # compile # test # install # preinst # prerm # postrm # cleanrm # postinst # clean # # CATEGORY = app-admin # PF = eselect-opengl-1.0.6-r1 # P = eselect-opengl-1.0.6 # PN = eselect-opengl # PVR = 1.0.6-r1 # PV = 1.0.6 # PR = r1 (will be "r0" for packages without -r) # # einfo () # elog () # ewarn () # eerror () if [ "x${EBUILD_PHASE}" != "x" ]; then if [ -x "/etc/portage/bashrc.d/${CATEGORY}.${EBUILD_PHASE}" ]; then source "/etc/portage/bashrc.d/${CATEGORY}.${EBUILD_PHASE}" fi if [ -x "/etc/portage/bashrc.d/${CATEGORY}/${PF}.${EBUILD_PHASE}" ]; then source "/etc/portage/bashrc.d/${CATEGORY}/${PF}.${EBUILD_PHASE}" elif [ -x "/etc/portage/bashrc.d/${CATEGORY}/${P}.${EBUILD_PHASE}" ]; then source "/etc/portage/bashrc.d/${CATEGORY}/${P}.${EBUILD_PHASE}" elif [ -x "/etc/portage/bashrc.d/${CATEGORY}/${PN}.${EBUILD_PHASE}" ]; then source "/etc/portage/bashrc.d/${CATEGORY}/${PN}.${EBUILD_PHASE}" fi fi
    その後、カテゴリperl-core / *、dev-perl / *およびperl-gcpan / *のすべてのパッケージのテストを有効にできます。 

     cat <<'EOF' >/etc/portage/bashrc.d/perl_module_test FEATURES="$FEATURES test" SRC_TEST="do" EOF cat <<'EOF' >/etc/portage/bashrc.d/perl_module_notest SRC_TEST= EOF chmod +x /etc/portage/bashrc.d/perl_module_* ln -s perl_module_test /etc/portage/bashrc.d/perl-core.test ln -s perl_module_test /etc/portage/bashrc.d/dev-perl.test ln -s perl_module_test /etc/portage/bashrc.d/perl-gcpan.test
    そして、必要に応じて、いくつかのモジュールのテストを無効にするか、何らかの影響を与えます。 

     mkdir /etc/portage/bashrc.d/dev-perl ln -s ../perl_module_notest /etc/portage/bashrc.d/dev-perl/Crypt-SSLeay.test ln -s ../perl_module_notest /etc/portage/bashrc.d/dev-perl/Shell-EnvImporter.test ln -s ../perl_module_notest /etc/portage/bashrc.d/dev-perl/Term-ReadLine-Perl.test cat <<'EOF' >/etc/portage/bashrc.d/dev-perl/Inline.test MAKEOPTS=-j1 EOF chmod +x /etc/portage/bashrc.d/dev-perl/Inline.test


  5. さらに、通常、CPANではモジュールの最新バージョンが最も安定しています。 これは、プライベートモジュールの場合にさらに当てはまります。 したがって、最新バージョンのインストールを強制することは理にかなっています。 

     cat <<'EOF' >>/etc/portage/package.keywords perl-core/* dev-perl/* virtual/perl-* # avoid depending on ~x86 sys-libs/readline dev-perl/Term-ReadLine-Gnu -~x86 # avoid depending on ~x86 sci-mathematics/pari dev-perl/math-pari -~x86 EOF


  6. ここで、g-cpanのCPANインデックスを更新する必要があります。 理論的には、g-cpanにはこのための--cpan_reloadオプションがありますが、現在のバージョン（0.16.2）では機能しません。 また、g-cpanはcpanユーティリティインデックスを使用するため、cpanを実行してcpan> reload indexを実行すると、この問題を回避できます。
  7. サーバー上のすべてのPerlスクリプトで使用されるモジュールのリストを準備します。 多階建てのshパイプラインで管理しましたが、おそらくこのためのPerlスクリプトを作成し、この記事のコメントに投稿する必要があります。 :)
  8. それらのebuildを生成します（ebuildは依存モジュール用にも自動的に作成されます）： 

     g-cpan -g Module::Name1 Module::Name2 …
    一部のモジュールでは、ペンを使用して/usr/local/portage/perl-gcpan/Module-Name/Module-Name-version.ebuildを編集する必要があります。 （これまでのところ、Crypt :: MatrixSSLのようなものに出くわしただけです。）
  9. ディレクトリ/ usr / lib / perl5 / site_perl / *を削除します。 注意 、この時点で、このサーバー上のほとんどのPerlスクリプトが破損します。 残念ながら、ベンダーモジュール（優先度の低い）をインストールする前にサイトモジュールを削除しないと、ベンダーモジュールのインストールおよびテスト中に、ベンダーに既にインストールされているモジュールのサイトバージョンの使用に起因する問題が発生する可能性があります。 ただし、問題は通常解決されるため、しばらくの間「システムを壊す」余裕がない場合は、最初にベンダーモジュールをインストールします。その後、サイトモジュールのあるディレクトリを削除することを忘れないでください。
  10. /etc/portage/profile/package.providedからすべてのperlモジュールを削除します。
  11. まず、モジュールに必要な移植を配置します（package.providedを使用し、現在インストールされていない場合）： 

     emerge -uDNa @world
    そして、必要なもの（最初にTest :: *モジュールを配置し、次にMath :: *を配置するのが理にかなっています-これにより、残りのインストールが高速化されます）。 次のように、インストールの試行を数回繰り返します。 すべてのモジュールが依存関係を正しく綴るわけではありません： 

     until emerge -uDN --keep-going y \ Test-Deep Test-Differences Test-Distribution … \ Math-BigInt-GMP math-pari … \ … do read -p "Press <Enter> to try again" done


更新する


g-cpanユーティリティは、「cpan> upgrade」の完全な類似物を提供しません。 

 g-cpan -u
g-cpanがパッケージを作成したモジュールのみを更新します（つまり、perl-gcpan / *カテゴリから、perl-core / *およびdev-perl / *からのパッケージは更新されません）。 しかし、これはそれほど悪くないかもしれません-最後に、これらのモジュールを更新するには、原則として、ebuildをperl-core / *またはdev-perl / *から/ usr / local / portageにコピーして名前を変更するだけです。 ただし、インストールされているモジュールのバージョンをより細かく制御し、以前のバージョンに簡単にロールバックできる可能性があります。



まとめ



私の意見では、g-cpanはグローバルにインストールされたPerlモジュールの管理を本当に簡素化します。 そして、11項目の巨大なインストールセクションにもかかわらず、本当に複雑なことは何もありません（インストールする必要のあるモジュールを決定する以外は:)）。
