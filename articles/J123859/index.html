オンライン支払いを簡単かつ便利にする。 A1Payシステムの再設計





ロシアで売り手と買い手の両方のためにシンプルで便利な支払いシステムを作成することは、UXスペシャリスト/インターフェイスデザイナーにとって深刻な課題です。 タスクが面白くて難しいほど、プロセスで得られる経験と知識が増えます。 私はこのまさにタスクに取り組んでいます。この記事では、 A1Payインターネット決済システムの再設計と最終化における私の経験を共有したいと思います 。



この記事では、作業の段階、行われた決定、およびそれらが何に基づいているかについて説明します。 1つの記事で、すべての興味深いアイデア、遭遇した困難、その解決策、発見事項などを説明してください。 -。 簡単な作業ではありません。 明らかな理由のために、測定されたKPIとその値など、いくつかの重要な問題は、完全に範囲外のままです。



インターフェイスでの私の作業の開始までに、A1Payはすでにインターネット支払いを受け入れるための有効なシステムでした。その強みは、高い技術力、詐欺防止、パートナーおよび加入者のサポートサービス、合理化された作業プロセスなどです。 しかし、このシステムは使いやすさが非常に不足していました。



会社の経営陣にとって、インターネット決済の分野のシステムとビジネスが成長するためには、ユーザー指向の製品が必要であることが明らかになりました。



支払いシステムのユーザーは、3つの大きなグループに分けることができます。
  • 商人/パートナー-ウェブサイト所有者
  • 購入者-販売者のサイトで購入するユーザー
  • 会社の従業員
この記事では、売り手のインターフェースの再設計について詳しく説明します。 パートナーと呼んでいます 一緒に私たちは共通のことをします。 システムの現在のバージョンをサポートする必要があり、同時に高度な機能と新しいインターフェイスを備えた新しいバージョンの開発が開始されたときに、プロジェクトに参加しました。 私の仕事では、UCDアプローチ（ ユーザー中心設計またはユーザー中心設計、最近はヒューマン中心設計と改名）に焦点を当てています。 このアプローチに従って、ユーザーを開発プロセスに引き付けるために、ユーザーの目標と目的を理解する必要があります。 A1Payの場合、次の手順を定義しました。
  1. 調査：システムとやり取りする際のユーザーの目標、目的、ニーズを調査する。
  2. 監査：既存のインターフェイスの問題を特定して修正します。 UIを標準化するためのユーザビリティガイドラインを作成します。
  3. 設計：最初の2つの段階の結果に基づいて、システムの新しいバージョンのインターフェイスを設計します。
  4. テスト：新しいインターフェイスの導入後にKPIを測定し、ユーザーからフィードバックを取得します。
  5. カイゼン（継続的な改善）：継続的なマイナーシステムの改善。


リサーチ



プロジェクトの作業のアクティブフェーズでは、研究時間は非常に限られています。 この場合、長時間にわたる一連の詳細なインタビューまたは比較的なユーザビリティテストは適切ではありません。



Cooperからキャラクターの方法論について聞いたことがあるかもしれませんが、このプロジェクトでは使用しませんでした。 なぜ-別の議論のための別の問題。 代わりに、ユーザーをグループに分けました。 上記の「販売者/パートナー」グループをサブグループに分けました。
  • 求職者とは、このシステムについて知り、アカウントをまだ持っていない人です。
  • 新しいパートナーは、システムを登録して作業を開始したばかりの売り手です。
  • 経験豊富なパートナーとは、システムを長年使用してきたパートナーです。
  • 元パートナー-何らかの理由でシステムを離れた人。
これにより、結果を追跡できるシナリオに従って、現在の機能とコンテンツがユーザーおよびKPIのさまざまなサブグループのニーズを満たす方法を定義できました。

Web分析ツールとして、Google Analyticsを使用しました。 その中で、訪問の頻度、ナビゲーションチェーン、最も訪問されたページ、コンバージョンなどを追跡しました。 これにより、システムの実際の使用に関する定量的なアイデアが得られました。 記事のある段落では、UXスペシャリスト向けのこのツールの機能について語るのは困難です。 ブライアン・クリフトンの本「Google Analytics。 プロのウェブサイトのトラフィック分析」とAvinash Koshikの本。

フォーラム、ソーシャルネットワーク、ブログ

試作機



現時点では、多数のプロトタイピングツールがあります 。 誰もが自分自身や直面しているタスクに便利なツールを見つけることができます 。 ただし、ポイントはツールにあるのではなく、ツールで作成するものと解決したいものにあります。 このプロジェクトでは、タスクはインタラクティブで非常に詳細なプロトタイプを作成することであったため、 Axureを使用しました 。







詳細なプロトタイプの作成の前に、概念設計が行われました。 この段階で、情報アーキテクチャ、ナビゲーション、対話アーキテクチャなどが作成されました。 ツールを使用する前に、紙の上で「考える」必要があるように思えます。 彼女はすべてに耐えます。







概念設計段階では、ナビゲーションは私にとっては問題でした。 古いバージョンのように、タブとサブメニューで水平にするか、垂直にしますか？ 紙に両方のオプションを描画し（画像の垂直方向）、システムの開発計画と比較して、水平方向のオプションに専念することにしました。 私のソリューションのロジックは次のとおりです。
  • システムに新しいモジュールとソリューションが表示されても、メニューは乱雑ではありません。
  • 左側の空き領域により、画面の作業領域が増加しました。
  • どの時点でも、ユーザーは1つの画面で作業します。 ユーザーがシステム全体の高速ナビゲーションを必要とするシナリオはありません。
  • このようなニーズ（クイックナビゲーション）が発生した場合でも、コンテキストリンク操作によって実現できます。
ナビゲーション要素は、訪問統計と調査データに基づいて並べられました。 最も頻繁な操作とモジュールは、左上にあります。



概念設計のもう1つの重要なポイントは、以前は存在しなかったダッシュボードがメイン画面として表示されることです。 システムを操作する主な頻度のシナリオは、監視です。 ダッシュボードは、システムのステータス、変更、およびその原因を明確に表示する必要があります。 コンテキストが原因である場所では、システムの奥深くにある詳細情報（たとえば、詳細な統計など）へのクイックリンクを提供します。 そこで、収入チャート、残高、口座、ニュース、新しいチケットなど、主要なブロックを含む紙のダッシュボードが描かれました。 モジュールのいくつかの要素は、速度を上げるためにパネルに拡張されました（「クイックリンク」を与えます）。



ダッシュボードの設計については、Stephen Few の優れた書籍、Information Dashboard Design：The Effective Visual Communication of Data 、Aaron Harsmanのプレゼンテーション、またはDashboard Spy Webサイトの例をご覧ください。



概念設計の後、Axureのすべてのシステム画面の詳細な描画が始まりました。 合計で、約30の画面が設計されました。 インタラクティブなプロトタイプにより、計画されたUIをチームでデモンストレーションおよび議論することができました。
