Googleのビーコンプラットフォームにより、物理Webの場合のようにビーコンを物理的に接続してビーコンを再構成する必要がなくなります。通常のURL（ Eddystone-URL ）をブロードキャストするとき、ビーコンに添付される添付ファイルのみを変更することができ、ビーコンをリモートで制御できます。 このソリューションにより、すべてのビーコンをリモートで監視することもできます。 これがどのように実装されているか、さらに説明します。



図からわかるように、ビーコンの管理のために、 Googleビーコンレジストリにデバイスを登録できるGoogle Proximity Beacon APIが提供されています。 登録済みビーコンに関する情報を保存します。 クライアント側では、デバイスと対話するために、 Nearby Messages API 、 Places API、またはその他の任意の裁量を使用することをお勧めします。



要するに、この動作方式では、ビーコンは広告ID （ Eddystone-UID ）をブロードキャストします。これに添付ファイル（ Attachments ）をバインドできます。これについては以下で詳細に説明します。 スマートフォンまたはタブレットでビーコンを見つけた後、 ビーコンレジストリでその広告IDをリクエストします。答えはこの添付ファイルです。 このシナリオは、特別または内部使用のために非公開アプリケーションを作成する必要がある場合に最適です。 または、ナビゲーションやいくつかの追加データの受信など、ユーザーに人気のある既存のアプリケーションにビーコンを操作する機能を追加する場合。



つまり 間違いなくアプリケーションが必要ですが、 Physical Webはどうですか？



Eddystone-URLまたはEddystone-UIDなど 、一部のビーコンは同時に2種類のパケットを送信できるため、追加のメタデータをアプリケーションに提供し、アプリケーションを持たないユーザーにアプリケーション自体を提供できます！



Google Proximity Beacon API



Google Proximity Beacon APIはクラウドサービスであり、 RESTインターフェイスを使用して、ビーコン、またはビーコンに関連付けられたデータを管理できます。



近接ビーコンAPIを使用すると、次のことができます。



  • デバイスを登録する
  • ビーコンへの添付ファイルの添付
  • デバイスを監視する
  • ビーコンから添付ファイルを受信する


Proximity Beacon APIを使用する前に、次のことを理解または理解しておくと役立ちます。



  • OAuth 2.0を使用してGoogle APIにアクセスする
  • RESTfull Webサービス
  • JSONデータ表現


Proximity Beacon APIを使用するには、次の5つの手順を完了する必要があります。



  1. Googleアカウントを作成またはログインします
  2. Google Developer Consoleで新しいプロジェクトを作成するか、既存のプロジェクトを選択します
  3. Proximity Beacon APIをアクティブにします-Google Developer Consoleで、[プロジェクトの選択]をクリックして、使用するプロジェクトを選択します。 左側のメニューで、API Managerをクリックしてから、Proximity Beacon APIを見つけ、そのステータスをenabled



    設定します
  4. OAuth 2.0クライアントIDを取得する
  5. API KEYを取得します 。 APIキーを安全に使用するためのベストプラクティスに精通することを強くお勧めします

ビーコン登録



メッセージをブロードキャストするためのビーコンの有効化と基本設定の後、 Proximity Beacon APIを使用して登録できます。 追加のデータを添付して、ユーザーのデバイスがどこにいるかを理解するのに役立ちます。 Googleビーコンレジストリのエントリには、次のメタデータが含まれる場合があります。



  • 広告ID （必須）
  • 現在のステータス-アクティブ、非アクティブ、廃止。
  • 安定性-配置の予想される安定性を表します。 （安定、めったに動かない、頻繁に動く、絶えず動く）
  • 緯度と経度は、値を度単位で表す二重のペアです。 特に指定がない限り、 WGS84に準拠する必要があります。 値は正規化された範囲内である必要があります。
  • 建物内のUvroenは、人間が読める形式の行で、ビーコンが配置されている階を示します。
  • Google Places APIプレイスID
  • テキストの説明
  • キー/値のペアなどのカスタムプロパティ


メタデータ値の詳細については、 こちらをご覧ください 。



これは、記事の冒頭で説明した非常に追加のメタデータです。これらのメタデータは、ニーズに合わせてアプリケーションで使用したり、全体的なUXを改善したりできます。 たとえば、 Place Pickerに Google Places APIのPlace IDを使用できます。 Place Pickerは、近くの場所、組織、施設などのインタラクティブな地図を表示するダイアログです。 ユーザーは、近くの場所に関する情報を取得したり、ソーシャルネットワークで現在いる場所の情報を共有したりできます。 Place PickerはPlaces APIの一部であり、 Android プラットフォームとiOSプラットフォームの両方で利用できます 。

登録解除



ビーコンが登録されると、 Googleビーコンレジストリから単純に削除することはできません。 ビーコンをオフラインにする方法には2つのオプションがあります。



  • プロジェクトからビーコンを一時的に削除するには、 beacons.deactivate



    を呼び出します。 非アクティブ化後、APIはバインドされたデータまたはビーコン情報を返しません。 ビーコンを動作状態に戻すには、 beacons.activate



    を呼び出す必要があります
  • beacons.decommission



    を呼び出して、プロジェクトのビーコンIDを永久に無効にします。 以前に登録されたIDは使用できなくなります。 ただし、ビーコンに新しいIDを簡単に割り当て、新しいIDでビーコンを再登録できます


ビーコンへの添付ファイルの追加

添付ファイルを作成するとき、次の2つのフィールドに入力する必要があります。



  • namespacedTypeは、 名前空間識別子、スラッシュ、およびデータ型で構成される文字列です。

    Npirmer：surreptitious surreptitious-banjo-145/string



  • data - namespacedTypeフィールドで定義されたデータ型のbase64表現。 例： aGVsbG8gd29ybGQh

モニタリング



ビーコンが適切に機能していることを確認するためにビーコンを展開したら、監視する必要があります。 Eddystone-TLMフレームにより、ビーコンはそれらを検出したデバイスにステータスを報告できますが、ビーコンが多く、それらが展開される領域が十分に大きい場合、特別な監視アプリケーションでビーコンを個別にバイパスすることが問題になる可能性があります。 状況から抜け出す方法があります。 Googleのビーコンプラットフォームを介して送信されるEddystone-TLMフレームを定期的にユーザーの手に渡すことができます。 監視フレームとユーザーデバイス用に特別に設計されたフレーム（ Eddystone-UIDなど ）の交互は、ビーコンの初期化中に設定された同じ時間間隔で発生します。



ビーコンが頻繁に使用されない場所にインストールされている場合、ユーザーがそのようなフレームを受信する可能性を最大化するために、 Eddystone-TLMフレームをターゲットと頻繁に切り替えることが推奨されます。



次のデータがEddystone-TLMフレームで送信されます。



  • バッテリー残量
  • 送信されたフレームの数
  • アクティベーションからの時間
  • あなたの温度


このデータに基づいて、 Googleのビーコンプラットフォームは次の情報を提供できます。



  • 推定バッテリー持続時間
  • ビーコンの移動（加速度計がある場合）
  • 異常に低い検出率


beacons.diagnostics
