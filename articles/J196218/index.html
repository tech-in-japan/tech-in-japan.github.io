すべてを監視する：スクリプトを使用してWindowsおよびLinuxエージェントを拡張する

Zabbixを使用してサーバーおよび他のコンピューター化されたワークステーションのステータスを監視する必要がある場合、これは2つの方法で実行できます。



最初の方法は、SNMPリクエストを使用することです。Zabbixは、SNMPリクエストを使用して送信を行います。 したがって、ネットワークインターフェイスを引き出してロードし、プロセッサ、メモリをロードできます。 さらに、サーバーメーカーは、SNMPを介して鉄の状態に関する多くの情報を提供できます。



2つ目は、 Zabbixエージェントを使用することです。これは、 監視対象システムで実行します。 監視されるパラメーターのリストには 、CPUの読み込み、メモリ使用量などの単純なものだけでなく、ローテーションサポート付きのテキストログファイルの読み取りや、ファイルが変更されたという事実の追跡など、トリッキーなものが含まれます。 システム上の任意のコマンドの出力をパラメーターとして使用することもできます。 Zabbixエージェントの機能はバージョンごとに大きくなります。







Zabbixを介して制御するものがZabbixエージェント機能のリストにない場合はどうすればよいですか？ 開発者が次のリリースで実装するまで待ちますか？ 必ずしもそうではありません。



Zabbixの機能を拡張してサーバーを監視するためのいくつかの標準インターフェースが残されていました。想像力とスクリプトを書くための空き時間がある限りです。 インターフェースはUserParameterとzabbix_senderです。 最初の1つについて説明し、例として、SMARTハードドライブのステータスを収集し、誰かがWindowsマシンで新しいプログラムをアンインストールまたはインストールするタイミングを監視する方法を示します。



少しの材料



サーバー上でZabbixエージェントを少なくとも1回設定している場合、 UserParameterの使用を開始するのは難しくありません。 新しいパラメーターを追加するには、いくつかのことを行う必要があります。



  • 構成ファイルzabbix_agentd.confの最後にフォームの行を追加します 


UserParameter=<>,<>
ここで：



<key>は独自に作成した一意の名前です。 Zabbixでデータアイテムを設定するときに使用します。

<command>は、監視対象ホストで実行されるコマンドです 。



そして、これは非常に単純な例で、Linuxのすべての標準構成にあります。 



 UserParameter=system.test,who|wc -l


そのため、 キーはsystem.testであり、 who | wc -lは、システム内の開いているセッションの数を返します。 追加（または、この行を既に持っている場合はコメント解除）して、先に進みます。



  • Zabbix Webコンソールで、使用したキーで新しいデータ項目を作成します。上記の例を使用する場合、これはsystem.testです。


これを行うには、[データアイテムの作成]をクリックします





そして、設定ファイルで指定されたものと同じキー 、およびZabbixエージェントのタイプを設定します。





  • 設定ファイルの変更を有効にするためにZabbixエージェントを再起動します


最新のデータで結果を観察します ：







UserParameterによるSMARTモニタリング



上記の例は、まったく同じことを行う標準のキーsystem.users.numがすでに存在することを考えると、ほとんど実用的ではありません。



それでは、もう少しリアルに見える例を見てみましょう。



ハードドライブの変更を計画する時期を監視することに関心がある場合は、2つのオプションがあります。



  1. ディスクがハードウェアRAIDコントローラの背後にある場合、原則として、オペレーティングシステムはディスク自体を「認識」しません。 したがって、ユーティリティまたはSNMPサブエージェントを通じてハードドライブのステータスに関する情報を取得する方法を探しています。これらは、RAIDコントローラの製造元に親切に提供した（または提供しなかった）。 コントローラーの個々のシリーズごとに、この情報への独自のパス。
  2. ワークステーション、ソフトウェアRAIDを備えたサーバーなどについてだけ話している場合、オペレーティングシステムからディスクにアクセスでき、さまざまなユーティリティを使用してそれらのステータスを自由に読み取ることができます。 Zabbixの場合、 SMARTMONTOOLSパッケージのsmartctlユーティリティが適しています。


Debianでは、SMARTMONTOOLSをインストールすると次のようになります。 



 sudo apt-get install smartmontools


そしてユーティリティは使用する準備ができています。



システム上の各ドライブについて、最初にSMARTが有効になっていることを確認します。 



 sudo smartctl -i /dev/sda | grep SMART SMART support is: Available - device has SMART capability. SMART support is: Enabled


突然SMARTがドライブでサポートされているが、オフになっている場合は、アクティブにします。 



 sudo smartctl -s on -S on -o on /dev/sda smartctl version 5.37 [i686-pc-linux-gnu] Copyright (C) 2002-6 Bruce Allen Home page is http://smartmontools.sourceforge.net/ === START OF ENABLE/DISABLE COMMANDS SECTION === SMART Enabled. SMART Attribute Autosave Enabled. SMART Automatic Offline Testing Enabled every four hours.


次のコマンドを使用して、SMARTのステータスを確認できます。 



 sudo smartctl -H /dev/sda |grep "test"| cut -f2 -d: |tr -d " "


これは、 zabbix_agentd.confに書き込むコマンド です。 



 UserParameter=uHDD.health,sudo smartctl -H /dev/sda |grep "test"| cut -f2 -d: |tr -d " "


ここで、uHDD.healthがキーです。



Flexible UserParameterによるSMARTモニタリング



これにより、ディスクが2つある場合の対処方法についての合理的な疑問が生じます。 この問題を解決する最も簡単な方法は、 UserParameterがエージェントにパラメーターを渡す機能です 。これについてはまだ説明していません。 しかし、すべては非常に簡単に行われ、すぐに例です： 



 UserParameter=uHDD.health.[*],sudo smartctl -H /dev/$1 |grep "test"| cut -f2 -d: |tr -d " "


Zabbix Webインターフェースのキーでは、パラメーターを*ではなく角括弧で置き換えます。 たとえば、あるデータ項目に対してsdaを記述し、別のデータ項目に対してsdbを記述します。 コマンドでは、このパラメーターは変数$ 1の位置に反映されます。







2番目のディスクのデータ要素を作成します。







しばらくすると、 最新のデータで結果を確認できます。







低レベルのディスカバリーを備えた柔軟なユーザーパラメーターによるSMARTモニタリング



すべてうまくいきました。 しかし、ここで、ディスクが2つではなく22である場合、合理的な疑問が生じます。 そしてここで、 すでに説明した 低レベル検出 （LLD）の素晴らしい機会を手に入れます。



低レベルの検出により、監視システムは、ネットワークノードに存在する類似の要素の数を検出し、テンプレートを使用してこれらの要素に必要なデータ要素、トリガー 、 グラフを動的に作成できます。 システムには、ファイルシステム、ネットワークインターフェイス、SNMP OIDをすぐに見つけることができます。 ただし、ここでは開発者が標準機能を補完する機会を残しました。JSON形式で要素が見つかったという情報をシステムに転送するだけです。 これを使用します。



小さなperlスクリプトsmartctl-disks-discovery.plを作成します。 システム内のすべてのディスクを検出し、この情報をJSONで出力し、ドライブでSMARTが有効になっているかどうかに関係なく情報を送信します。 



 #!/usr/bin/perl #must be run as root $first = 1; print "{\n"; print "\t\"data\":[\n\n"; for (`ls -l /dev/disk/by-id/ | cut -d"/" -f3 | sort -n | uniq -w 3`) { #DISK LOOP $smart_avail=0; $smart_enabled=0; $smart_enable_tried=0; #next when total 0 at output if ($_ eq "total 0\n") { next; } print "\t,\n" if not $first; $first = 0; $disk =$_; chomp($disk); #SMART STATUS LOOP foreach(`smartctl -i /dev/$disk | grep SMART`) { $line=$_; # if SMART available -> continue if ($line = /Available/){ $smart_avail=1; next; } #if SMART is disabled then try to enable it (also offline tests etc) if ($line = /Disabled/ & $smart_enable_tried == 0){ foreach(`smartctl -i /dev/$disk -s on -o on -S on | grep SMART`) { if (/SMART Enabled/){ $smart_enabled=1; next; } } $smart_enable_tried=1; } if ($line = /Enabled/){ $smart_enabled=1; } } print "\t{\n"; print "\t\t\"{#DISKNAME}\":\"$disk\",\n"; print "\t\t\"{#SMART_ENABLED}\":\"$smart_enabled\"\n"; print "\t}\n"; } print "\n\t]\n"; print "}\n";

ここで、スクリプトがZabbixによって自動的に起動されるようにするために、もう1つのUserParameterをzabbix_agentd.confに追加するだけです 。 



 UserParameter=uHDD.discovery,sudo /usr/local/bin/smartctl-disks-discovery.pl


構成が完了したら、Webインターフェイスに移動し、smartctlの新しい検出ルールを作成します。







キーとフィルターに注意してください（{#SMART_ENABLED} = 1）後者のおかげで、SMARTをサポートする検出されたドライブのみが追加されます。 これで、ドライブ名をマクロ{#DISKNAME}に置き換えるだけで、sdaおよびsdbドライブの2つのデータ項目を1つのプロトタイプデータ項目に書き換えることができます。







最後に、Zabbixがzabbix_agentd.confにルートとして登録したコマンドを実行してSMARTを監視する前に、パスワードを入力せずにこのコマンドを実行するユーザーのアクセス許可を追加する必要があります。このため、/ etc / sudoersに次の行を追加します。 



 zabbix ALL= (ALL) NOPASSWD: /usr/sbin/smartctl,/usr/local/bin/smartctl-disks-discovery.pl


SMARTモニタリング用の完成したテンプレートを、トリガーとそのために構成された構成を使用して、残りのデータ要素とともに適用します。

トリガーを追加します。







そして、電子メール通知を送信するアクション ：







監視プロセス全体は次のようになります。Zabbixスクリプトは、現在と前の2つのプログラムリストを比較するエージェントによって毎時間起動されます。 次に、スクリプトはすべての変更を別のファイルに書き込みます。 変更がない場合、ファイルに0x0が書き込まれます



ファイルの内容はZabbixサーバーに送られ、 uDiffProgramms データ項目の値が 0x0以外の場合にトリガーが発生します。 次に、別のアクションが、このコンピューターにインストールまたは削除されたもののリストを含む通知をメールで送信します。







最後に



UserParameterは 、システムの機能を自分で拡張するための優れた簡単な方法です。 言及する価値はあります： zabbix_sender。たとえば、スケジュールに従って（ UserParameterのように）ではなく、何らかのイベントに従って、Zabbixにデータを送信する必要がある場合に適しています。 System.run [] 、これはUserParameterに似ていますが、すべてのエージェント構成を変更する必要がないという点でより便利で、このデータ項目をテンプレートに追加するだけです。 さらに、Zabbix 2.2の次のメジャーリリースでは、エージェントの機能を拡張するもう1つの新しい方法がプラグインです。 楽しみにしています！
