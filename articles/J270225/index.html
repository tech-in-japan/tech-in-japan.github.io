ダブルKCSサイクル。 知識管理について

この記事は前の記事の続きで、KCS方法論に基づいた会社のナレッジマネジメントへのアプローチに関する発表がありました。 要するに、短い定義が与えられ、KCSの導入によって発表された結果が発表されただけでなく、KCSの主なアイデアであるダブルサイクルによる知識管理も発表されました。 彼は今日議論されます。 私は元の練習ガイドを転載したくはないので、できるだけシンプルで短くなるように考えます。184ページもあります。 そして、すべての重要なもの！



用語集


私は練習とテクニックを英語からどのように翻訳するかを長い間考えていました。 その結果、彼は「Practices」と「Receptions」に立ち寄った。 そのため、より調和のとれたように見えました。



それでは、二重ループとは何ですか？



プロセス自体に進む前に、KCSがどのような知識であるか、つまりどの種類の知的資産 （元の知識資産）を区別できるかを示したいと思います。 KCSによると、それらには4つのタイプがあります。



  1. 知識ベース記事。 実際、これは組織のサポートサービスの作業中に得られた経験です。 記事は問題の解決に役立ちます。
  2. 従業員のプロファイル。 この情報は、従業員の経験、スキル、関心、評判に関するものです。
  3. 顧客情報。 たとえば、クライアントのビジネスモデル、組織の目標。 会社のプロフィールが完全であればあるほど良い。
  4. クライアントソフトウェアの構成。


これまで、KCSは最初のタイプでの作業について説明しています。 方法論の以下のバージョンは、残りの3つのタイプに影響しますが、現時点では、上記のすべてが主に記事に適用されることを理解する必要があります。







したがって、ダブルサイクルには、決定サイクルと開発サイクルが含まれます。 あまり明確ではありませんよね？ 各概念が開示されている場合、決定サイクルは、従業員が実行すべきアクションと、ユーザーの問題を解決するためのものです。 また、開発サイクルは、KCSを維持および開発する方法に関するものであるため、KCSはより良く、より速く、より良く機能します。 私の解釈は非常に無料であることを思い出させてください。 この出願方法は、原文を文字通り翻訳するよりも優れていると判断しました。



2つのサイクルにはそれぞれ4つのプラクティスが含まれており、すでに各プラクティスには特定のテクニックが含まれています。 2から8まで。 プラクティスをリストしますが、もちろん、テクニックは分析しません。



決定サイクル



決定サイクルは4つのプラクティスで構成されています。



  1. ワークフロー中に知識を収集します。
  2. 知識の構造化。
  3. 再利用;
  4. 継続的な改善。


KCSでは、問題をすぐに文書化することが非常に重要であり、ユーザーとの会話時にすぐに改善することが重要です。 したがって、問題はクライアントによって説明されます。 問題が解決するかどうかまだ不明な場合でも、この問題を説明する記事はすでにすべての人が利用できるはずです。 KCSによると、このアプローチはいわゆる隠された知識の識別に役立ちます。



記事を構造化することで、それらの適切な構成は記事をナビゲートするのに役立ち、記事の認識が向上します。 これは主にテンプレートを使用して実現されます。 つまり 各記事の構造は類似している必要があり、各タイプの記事は独自のテンプレートを開発できます。 誰もがこの形式に慣れると、情報の検索がはるかに簡単になります。 最も単純な形式では、記事の構造は、問題の説明、環境の説明、解決策、およびメタデータで構成できます。 バグも説明する- 全体の芸術 。 記事が悪いのはなぜですか？



再利用は、知識ベースを構築するための基礎です。 結局のところ、誰も参照することのない記事は誰も必要としません。 ただし、問題は、記事を書いている時点で、それが需要があるかどうかが明確でないことです。 したがって、役立つトリックの1つは、検索を保存することです。 誰もが暗黙のgitルール「早期コミット、頻繁にコミット」を覚えています。 ここでも同じことですが、検索についてのみです。 「早めに検索して、頻繁に検索してください。」 ベストプラクティスは、クライアントとの通信時に直接検索することです。



継続的な改善は、おそらくリストから最も理解しやすい項目です。 記事は絶えず更新され、補足されるべきです。 たとえば、ある記事で不正確に気づいた場合、少なくともそれを書き留め、それを修正するために最大限に注意する必要があります。 これには、重複記事の操作も含まれます。 コードなどの記事は、正しくマージできる必要があります。



1つの重要なルールが決定サイクルのすべてのプラクティスを通過します-
集団所有権。
アイデアは単純です：「記事を使用する場合は、その品質に責任があります。」



開発サイクル



開発サイクルも4つのプラクティスで構成されています。



  1. コンテンツの関連性と品質。
  2. プロセス統合;
  3. 性能評価;
  4. リーダーシップとコミュニケーション。


KCSは、ナレッジベースの記事が対応しなければならないすべての特性を含む、非常にクールな用語「コンテンツヘルス」を考案しました。 たとえば、同じものが異なる名前で呼び出されないように、用語集を導入する必要があります。 ナレッジベースには記事が重複しないようにしてください。 記事の場合、テンプレートと適切に設計された記事の例を作成する必要があります。 これは、KCSで最も重要なポイントの1つです。 8つの異なるトリックが含まれています。 原則として、あなたの会社が既に何らかの形でナレッジベースを持っている場合、オリジナルのこの章を読んだ後、実装のための多くの興味深いものを見つけることができます。



プロセスの統合に関しては、これが最も問題のあるポイントです。 基本的な考え方は、サービスデスクシステムをナレッジマネジメントシステムと統合することです。 例として、ナレッジベースの記事とサービスデスクの特定のアプリケーションを「解決先」リンクを使用してリンクできる場合、記事の品質を評価するための優れたツールがあります。 T、e。この記事が特定の問題の解決に役立った回数。 ナレッジベースがメッセンジャーと統合されている場合はさらに良いです。 同僚のアドバイスや推奨事項を見つけるために、企業のメッセンジャーで検索を使用しなければならなかった頻度を覚えていますか？ そして、もしそれがスカイプではなくたるみなら。



最後の段落では、記事の品質を評価する方法の1つを既に示しました。 実際、KCSは特別なAQI指標である記事の品質指標も導入しています。 また、KCSが機能するためには、従業員の有効性をさらに評価する必要があります。 それらを見つけて、励ましてください。 これは、知識ベース管理システムでのゲーミフィケーションの導入かもしれませんが、何らかの物質的なインセンティブであればより良いです。 また、この時点で、KCS開発者はバランススコアカードの使用を強くお勧めします。



リーダーシップとコミュニケーション-これは、管理とスタッフとの連携に関するものです。 従業員は、何らかの方法で知識ベースを使用してそれに貢献するように動機付けられる必要があります。 社内では、チームワークを促進し、インセンティブおよび報酬プログラムを導入する必要があります。 内部コミュニケーションに大きな重点を置く必要があります。 そして、はい、会社のビジョン、その使命 -それはすべてここにあります。



合計ではなく



KCSに含まれる推奨プラクティスのそれぞれについて段落を1つだけ書いたので、KCSは巨大で広大であるように見え始めます。 そうだと思いますが、私の意見での利点は、あなたがそれのいくつかの部分を取り、あなたの組織でそれらを紹介しようとすることができるということです。 さらに、KCSは「他に何が起こるか」の良い例です。 レーキを踏んで作業の質をすぐに上げることはできません。
