Unityで格闘ゲームを作成します：HitboxおよびHurtboxの実装







解説



hitboxおよびhurtboxとは何ですか？ それは同じことではありませんか？



答えはあなたが誰に質問するかによって異なりますが、記事では、ヒットボックスとハートボックスは、適切な格闘ゲームの場合のように、用途の異なる2つの異なる概念であると考えます。



Hitboxは、攻撃がヒットする場所を決定する非表示の四角形（または球体）です。



Hurtboxも目に見えない長方形（または球体）ですが、プレーヤーまたはオブジェクトがHitboxを使用してヒットできる場所を定義します。









Street Fighter IVのこの画像では、赤いボックスがヒットボックスで、緑のボックスがハートボックスです



ヒットボックスとハートボックスのサイズと位置は、再生されるアニメーションフレームに依存することに注意してください。



Killer InstinctのGif。 ヒットボックスはショットフレームにのみ表示され、剣とともに移動することに注意してください。






Killer Instinctの例には、 Pushbox （黄色の長方形。Hurtboxは空の緑の長方形）という別のタイプの長方形もあります。 プッシュボックスは、キャラクターが物理的に占有するスペースを示す領域であり、キャラクターが重ならないようにします。



戦闘とビートアップのジャンルのほとんどのゲームでは、簡単にするために考慮しない2つのタイプの領域があります。



グラブまたはスローボックスは、キャラクターをつかんだり投げたりできる領域を定義し、 ブロックボックスは、戻るボタンを押した攻撃されたプレーヤーが後方に移動する代わりに攻撃をブロックし始める領域を決定します。



設計の観点から、これらすべての領域は非常に重要です。 攻撃のヒットボックスとハートボックスは、攻撃がダメージを与えるフレームの数だけでなく、この攻撃の死角とプレイヤーの脆弱性も決定します。



ストリートファイターの例でこれの良い説明はビデオで提示されます。



それでは、用語を理解したら、作業に取り掛かりましょう。



何が欲しい



作業したいエリアの各タイプを見て、それらから何が欲しいかを言いましょう：



プッシュボックス：タッチするには2つのプッシュボックスが必要ですが、互いに重なり合うことはありません（そのため、「プッシュボックスエリア」と呼ばれるため、別のキャラクターをプッシュします）。 プッシュボックスは、他のプッシュボックスとのみ対話する必要があります。



ハートボックス：ヒットを記録できますが、物理的な意味での衝突は実行しないでください。 ハートボックスは、ヒットボックスとのみ対話する必要があります。



Hitbox：任意のフレームでHurtboxにオーバーレイするかどうかを確認できるはずです。 Hurtboxとのみ対話する必要があります。



Unity標準コンポーネントの使用



まず、さまざまなタイプの領域を標準のUnityコンポーネントにバインドすることができます。 明らかな選択は、 Colliderです。



Pushboxは、Collider plus Rigidbodyとして直接実装できます。 オブジェクトとの衝突に応答し、他のプッシュボックスと重ならないように、必要なとおりに動作します。



（適切なRigidbody設定を除いて）心配する必要があるのは、「 他のプッシュボックスとのみ衝突することができる 」プロパティの実装です。 Unity物理システムに精通している場合、解決策はレイヤーとレイヤー衝突マトリックスを使用することであることを既に知っています。 わかりやすくするために、Pushboxというレイヤーを作成してオブジェクトに割り当て、 PushboxがPushboxとのみ衝突するようにコリジョンマトリックスを構成できます。















Hurtboxesの場合、isTriggerを使用してColliderを取得できます。 そのため、彼は物理的な意味で衝突を実行せず、彼のエリアに落ちる他のコライダーのみを登録することを保証します。 影響を記録するには、 OnTriggerEnterイベントを実装する同じオブジェクトにスクリプトを追加する必要があります。場合によっては、着信コライダーのタグをチェックして、イベントをトリガーしたコライダーが必要であることを確認し、ゲームに必要なダメージとヘルスの計算を実行する必要があります。 あなたはおそらくこのアプローチに精通しているでしょう。



また、HurtboxレイヤーとHitboxレイヤーを作成する必要があります。HurtboxがHitboxのみと衝突するように、再びLayer Collision Matrixを使用します。



  • Rigidbodyは必要ありませんが、追加するすべてのトリガーは、Rigidbodyを既に持っているPushboxオブジェクトの子になると信じているからです。 これは重要です。なぜなら、リジッドボディまたはその親オブジェクトを持たないコライダーは、UnityエンジンによってStaticとして定義され、 それらを移動してもアクションが発生しないためです。
  • さらに、おそらく、Hitboxプレイヤーと敵のHitboxを区別する必要があります。 Hurtboxについても同じことが言えます。 これにより、プレイヤーのヒットボックスはハートボックスの敵にしかヒットできず、敵のヒットボックスはプレイヤーのハートボックスにしかヒットできません。 味方にダメージを与えたいが、ここではプレイヤーが自分のハートボックスをヒットできないように注意する必要があります。


ヒットボックスは実装が最も困難です。 物理的な衝突を避けるために、isTriggerでColliderを使用できます。実際、Hitboxに含まれるコライダーはありません。 この問題を解決する別の方法があります。Hitboxは、Hurtboxに「入り」ます（ またはオーバーレイをチェックします ）。 とにかく、Colliderが必要です。そうしないと、 UnityはHurtboxでOnTriggerEnterを呼び出しません 。



Hurtboxにダメージを与えるには、同じオブジェクトにスクリプトを追加して、HurtboxがGetComponent <T>を使用して、どの程度のダメージを与える必要があるかを見つけられるようにする必要があります。 これを別の方法で行うこともできます。両方のOnTriggerEnter Colliderに対して呼び出されます。 また、ヒットボックスを必要なときにのみアクティブにし、すべてのフレームではなく、キャラクターが攻撃しないときにアクティブにする方法を見つける必要があります。 これを行うには、 ドキュメントによると、 トリガーイベントが無効なMonoBehavioursに送信され、コリジョンへの応答にビヘイビアーを含めることができるため、スクリプトをオフにすることができます。



コライダーを有効または無効にするか、スクリプトにブール値を追加して、ヒットするかどうかを示します。



問題



  • 階層： OnTriggerEnterに応答できるようにするには、Colliderを使用して各オブジェクトにスクリプトを作成する必要があります。 順序のためにすべてのスクリプトを1か所に保持したい場合は、他のすべてのオブジェクトに呼び出しを委任するためだけにスクリプトを作成する必要があります。
  • 過度：このアプローチでは、Hitboxには必要のない大きな機能があります。
  • イベント：私たちの機能はOnTriggerEnterに基づいています。 Unityイベントの使用は問題ではないかもしれません。 しかし、少なくともそれらの必要性について考える理由があります。 さらに知るには、これを調べる価値があります （ 「Unity APIへの高価な呼び出しの回避」セクションを参照 ）。
  • 視覚的なノイズ：異なる攻撃に異なるヒットボックスを使用する場合、上記の問題が再び発生するだけでなく、エディターウィンドウにも視覚的なノイズがあります。
  • 柔軟性が低い： ColliderをHitboxとして使用する場合、たとえば、BoxからSphereにコライダーの形状を変更する場合、すべてのBoxColliderを手動で削除し、SphereColliderを追加する必要があります（または、この作業を行うエディタースクリプトを記述します）


私たちはすべて自分で作ります



上記からおそらく理解したように、プッシュボックスとハートボックスは、標準のUnityコンポーネントによって非常に便利に実装されます。



ハートボックスにはまだ上記の問題があり、それらのいくつかを解決しますが、抽象化する必要がある主要なエンティティはヒットボックスです。



多数の攻撃とコンボを含む格闘ゲームを作成する場合、オブジェクト内のすべての攻撃を適切に順序付けし、それぞれに対してHitboxの複数の組み合わせを作成できるようにする必要があります。 これを行うには、OnTriggerEnter呼び出しをアクティブな攻撃に厳密に委任するか、同様のことを行うスクリプトが必要です。

















攻撃ごとに個別のHitboxオブジェクトを作成する必要はありません。ここでは、サイズを変更して同じものを使用できます。



ヒットボックス



次の問題を解決するには、新しいコンポーネントが必要です。



  1. Hitboxの動作をさせるには、任意のフレームでHurtboxのオーバーレイをチェックできる必要があります。 彼はHurtbox-sとのみ対話する義務があります。
  2. シーンウィンドウで視覚的に表現します。
  3. カスタマイズ可能かつ柔軟であること。
  4. 理想的には、Unity APIイベントに依存しないでください。
  5. 別のオブジェクトにスクリプトを使用するために十分に独立している。
  6. 特定の攻撃に執着しないでください。 ヒットボックスは、いくつかの異なる攻撃に適用できる必要があります。


振る舞い



最初に、エリアがコライダーに重なっていることをどのように確認しますか？ 答えはUnityEngine.Physicsを使用することです 。



物理学には、このタスクを達成できる多くのメソッドがあります。 必要なフォーム（Box、Sphere、Capsule）を指定し、必要に応じて、ヒットしたコライダー（ある場合）を配列として取得するか、これらのコライダーで埋めるために配列を渡します。 それについては考えませんが、最初の場合は新しい配列が割り当てられ、2番目の場合は既存の配列に入力するだけです。



長方形の領域が何かに当たっているかどうかを確認することから始めましょう。 このために、 OverlapBoxを使用できます。



チェックされた長方形の寸法を設定する必要があります。 これを行うには、長方形の中心、その半分のサイズ、回転、およびヒットする必要があるレイヤーが必要です。 半分のサイズは各方向のサイズの半分です。たとえば、サイズ（2、6、8）の長方形がある場合、その半分のサイズは（1、3、4）になります。



中心として、GameObjectの変換位置を使用できます。回転には、GameObjectの変換回転を使用するか、一般変数を追加して特定の値を設定できます。



半分のサイズはVector3であるため、一般的に使用して使用します。



ヒットするレイヤーの場合、 LayerMaskタイプのパブリックプロパティを作成します。 これにより、インスペクターでレイヤーを選択できます。 



Collider[] colliders = Physics.OverlapBox(position, boxSize, rotation, mask); if (colliders.Length > 0) { Debug.Log("We hit something"); }


正しい設定の場合、および呼び出し中に投影された長方形が対応するマスクのコライダーにスーパーインポーズされると、コンソールにメッセージが表示されるはずです。

オンザフライでコライダーを追加および削除することは、ヒットボックスほど簡単ではありません。 このタスクを達成するための満足のいく方法が見つかりませんでした。 Hitboxで行ったように、いくつかの異なるColliderをスクリプトに追加し、ブール値を使用して目的のColliderを選択できます。 ここでの問題は、必要なすべてのColliderをコンポーネントとして追加する必要があることです。その結果、エディターとオブジェクトに強い視覚的ノイズが発生します。



別のアプローチは、コードを介してコンポーネントを追加および削除することですが、そのようなソリューションは多くの不要なゴミを追加し、おそらくそれほど正確ではありません。
