Java 7からJava 8への移行について



-太陽の下で働き、Javaを内側から見て、Odnoklassnikiに切り替えたときに、クライアント側から見て始めました。 （アプリケーション開発者として）Java 6、7、8を見たことがありますか？ たとえば、JDK 8はJDK 7よりも明らかに優れた製品であると感じていますか？



-すべてのバージョンにバグがあり、すでに使用されています。 JVMが非常に複雑なシステムであることは明らかです。 また、特定のシナリオでJITコンパイラがどのように機能するかを事前に予測することさえできない場合があります。 最も重要なことは、すべての人のためにこれに取り組んでいるポータルの99％があり、2013年以降、大きな低下はありませんでした。 したがって、Javaに満足しています。



-Odnoklassnikiの何パーセントが、秘密ではないとしても、Java 7で動作しますか？また、Java 8で動作しますか？



-Java 6にもシステムがあります。幸いなことに、それらはほとんどありません。 何年も再起動しないサービスがあります。 それらをアップグレードする意味は何ですか？ 仕事と仕事。 そのため、現在起動しているものはすべて新しく、すぐにJDK 8で起動します。これらのサービス（多くの場合デプロイされるサービス）は、すでにほぼすべてがJDK 8に移行しています。 そして毎週新しいものが翻訳されます。



-Java 7からJava 8に移行する際の困難は何ですか？



-驚くべきことに、Java 6からJava 7に移行したときよりも、Java 7からJava 8に移行したときの問題がはるかに少ないことがわかりました。 Unicodeサポートの変更などでした。



Java 8に切り替えたとき、そのような問題はありませんでした。 しかし、JITコンパイラのバグに遭遇しました。



一般に、時々、何らかのバグがある場合、私たちは確かにそれに出くわすように思われます。Oracle自体よりもJavaをよりよくテストできるほど多くのサーバーとそのような負荷があるようです。



たとえば、本番環境でマルチレベルコンパイルを無効にしました。これは、Oracleのスタッフが熱心にやったことです。C1コンパイラに重大なバグがあるため、すぐにC2でコンパイルします。



-まだこのバグを修正しましたか？



-はい、JDK 8u60が問題を解決したようです。 JITコンパイラーでは、オラクルのスタッフが問題を非常に迅速にクリーンアップします。 しかし、重大な脆弱性がありました。仮想マシン全体をクラッシュさせる単純なJavaコードを書くことができました。 ちなみに、バージョン8u40では、コンパイラーで2つのバグが修正され、私が提出（まとめた-著者のメモ）。

-どのGCが使用されていますか？



-ほとんどの場合、適切に調整されたCMS。 一般に、Garbage-Firstよりもうまく機能します。 ただし、ここでもG1が使用されます。たとえば、応答時間を保証するためにSQLサーバーを置き換えるようになったNewSQLの自己記述ソリューションで使用されます。



つまり、平均して、G1はあいまいなCMSよりも少し悪い動作をしますが、同時により多くの保証を提供します。 たとえば、CMSには平均で50ミリ秒の遅延がありますが、最大で300〜400ミリ秒に達することがあります。 そして、G1はおそらくより頻繁に収集し、平均で最大150ミリ秒の休止がありますが、200ミリ秒の制限が与えられ、それに耐えようとしています。
