Visual StudioでDSLを使用する

はじめに



以前は、Visual Studioでの開発を簡素化するツールの開発について考えたことはありませんでした。また、開発に役立つさまざまなサードパーティユーティリティを作成することもよくありました。 しかし、いつものように、転換点が来ています。

かつては、専門的なソリューションを開発する必要があるプラットフォームを開発することがタスクでした。

柔軟な構成の可能性を制限することなく、プラットフォーム上のソリューションの開発を可能な限り簡素化したかったのです。

問題を解決するには、主に2つの方向がありました。

  • カスタムソリューションツールを開発する
  • 開発環境に組み込まれた支援ツールを作成する




独自のツール


このアプローチを使用すると、将来はコーディングせずに実行できます。 これは間違いなく、ソリューションの開発を簡素化および高速化し、エラーの可能性も減らします（もちろん、すべてがきれいに行われない限り）。

しかし、ツール開発者が直面する主な困難は、必要とされる可能性があるすべての必要な機能を可能な限り提供することです。

各ソリューションでは、重要なタスクが発生しますが、ツールを作成する段階では考えられません。 また、ユニバーサルツールを作成する場合、その複雑さが増し、作成のタイミングや使用の複雑さに影響します。



Visual Studioツール


補助ツールを使用する場合、必要な基本コードを可能な限り自動化し、システムの機能を拡張できるようにします。

しかし、このアプローチを使用するには、ソリューション開発者がより高い専門レベルを持っている必要があります。



私の選択


私が思いつくことができる長所と短所を比較検討した後、私は2番目のオプションを選ぶことにしました。 それは私にとって新しいものであり、本当に魅了されました。

次に、DSLを使用してVisual Studio 2010の支援ツールを作成する基本を説明します。 おそらくこれは、この分野のさらなる独立した研究にあなたを押し進めるでしょう。



予備



始める前に、次のディストリビューションをダウンロードしてインストールする必要があります。

  • ビジュアルスタジオ
  • Visual Studio SDK
  • Visual Studio Visualization and Modeling SDK


問題の声明



まず、主な可能性の両方を述べ、このメカニズムが日常の課題を解決するためにどのように効果的であるかを示すことができる基礎に基づいた例を考え出す必要があります。

最初は、このタスクは初歩的なように思えましたが、考えてみると、すべてがそれほど単純ではないことに気付きました。 瞑想に時間を費やした後、私はまだ思いつきました。

WCFサービスを作成するレベルでクライアントサーバーアプリケーションを開発するのに役立つモデルを作成することを提案します。 もちろん、WCFの技術的な側面に重点を置くのではなく、オブジェクトモデルとして取り上げます。



プロジェクトの作成とデバッグ



自分でタスクを設定したら、これ以上考えずに、独自のプロジェクトを作成します。

いつものように、始めに、最愛のVisual Studio 2010を起動します。

プロジェクト作成ウィンドウに入り、プロジェクトタイプDomain-Specific Language Designerを選択します。





次に、ウィザードにより、作成するモジュールのタイプを選択するように求められます。 MinimalLanguageを選択します。 このタイプのプロジェクトでは、事前に作成された要素の最小数。これにより、要素をより早く取り除くことができます。







次に、ファイルの拡張子を設定する必要があります。 この拡張機能を使用して、ファイルをプロジェクトに追加します。

最後の貢献は、作成中のプロジェクトに関する合計情報です。

[完了]をクリックすると、事前に作成された要素を含むプロジェクトが開かれます。これは、例を示すために行われます。







プロジェクトをすぐにコンパイルして、テストしてみることができます。

このタイプのプロジェクトのテストとデバッグには、デバッグスタジオが使用されます。 [実行]をクリックするか、デバッグして実行すると、プロジェクトファイルのテストコピーが作成される追加のスタジオが開きます。

基本的なオブジェクトとそのマッピングの作成


当面の施設の作成を開始する前に、準備要素を作成する必要があります。

  • デザイナーは、オブジェクトを視覚化するデザイナーです。 DSL Explorerで、 Editorオブジェクトを作成します。 その中にプロジェクトの拡張子を設定する必要があります（これはFileExtensionプロパティです）。これを「wcfw」にします。
  • 図は、オブジェクトを表す主要な視覚化要素です。 WCFDiagramと呼びましょう。
  • ルート要素（ Domain Class ）を作成する必要があります。その上で、以前に作成したエディター （これはルートクラスプロパティです）およびダイアグラム （これはクラス表現プロパティです）を構成する必要があります。 ルート要素に「WCFCollection」という名前を付けましょう。
  • Explorerの動作 -DSL Explorerに似たもので、このタイプのファイルを操作するときに表示されます。


これは、ファイルを保存してプロジェクトをコンパイルできるようにするために必要な最小限のリストです。 ただし、コンパイルの前に、ファイルに加えられた変更に従ってコードを再生成する必要があります。 これを行うには、 ソリューションエクスプローラーで [すべてのテンプレートを変換]ボタンをクリックします。







すべてのコードは、 DslおよびDslPackageプロジェクトの GeneratedCodeフォルダーにある* .ttファイル（T4変換を記述する）から生成されます。 新しいオブジェクトが生成されたら、プロジェクトを収集して開始することさえできます。 同じプロジェクトで、同じデバッグスタジオが起動します。 実験プロジェクトに関連するファイルがプロジェクトに追加されたため、すべて削除できます。 ファイルタイプを（他のタイプと同様に）追加し、WCFWizardという名前で検索します。

チャート表示を作成する


オブジェクトがダイアグラムに表示されるようにするには、オブジェクトの視覚的表現を作成する必要があります。 これらの目的のために、 ツールボックス要素があります： ジオメトリ形状 、 コンパートメント形状 、 コネクタ 。 最初の2つはタイプDomain Classのオブジェクトをレンダリングし、最後はDomain Relationshipに使用されます。

コンパートメント形状要素を追加し、WCFServiceShapeという名前を付けます。 この要素には、視覚的表現に関与する多くのプロパティがありますが、生成されたクラスの拡張により、より柔軟な視覚的表現を生成することもできます。

オブジェクトをその表現に関連付けるには、 ダイアグラム要素マップ要素を使用して、WCFServiceとWCFServiceShape間の接続を拡張する必要があります。







コンパートメントシェイプに追加できるものを見てみましょう。

  • コンパートメント-プロパティのコレクションを表示します。
  • ドメインプロパティ-プロパティに表示されるプロパティ。
  • Expand Collapse Decorator-内部コンテンツを折りたたんだり展開したりできる要素。
  • アイコンデコレータ-画像を表示します。
  • テキストデコレータ-テキスト出力。


テキストデコレータを追加し、NameDecoratorという名前を付けます。 次に、それをWCFServiceの特定のプロパティに関連付ける必要があります。そのために、 DSL Detailsウィンドウを使用します。 Diagram Element Mapを選択すると、バンドルに関する情報が表示されます。 [デコレータマップ]タブに移動して、NameDecoratorをアクティブにします。 WCFServiceプロパティを直接使用するため、 プロパティ要素を表示するパスは空のままにできますが、この要素を使用して、関連オブジェクトへのパスを設定し、そのプロパティを表示できます。 Displayプロパティ要素で、名前を選択します。







これで、WCFServiceは自身をチャートに表示することを学習しました。



ツールボックスにオブジェクトを置く


WCFWizard Explorerを使用してオブジェクトを追加できますが、見た目はあまり良くありません。 Toolboxを使用してオブジェクトを作成する機能を追加してみましょう。

DSLエクスプローラーに移動して、 エディターで新しいツールボックスタブを作成します。これをWCFWizard Elementsと名付けます。 これで、 Toolboxの要素のグループを作成しました。ここで、 接続ツール （接続用）と要素ツール （オブジェクト用）の2種類の要素を作成できます。

要素ツールタイプの要素を作成し、WCFServiceToolという名前を付け、リソースに既に存在するアイコンからアイコンを取得します。 最も重要なことは、ドラッグアンドドロップ時に作成されるオブジェクトを特徴付けるClassプロパティを設定することです。 WCFServiceを選択します。

これで、最初のオブジェクトがツールボックスに表示され、チャートに安全にドラッグアンドドロップできます。



「機能」を記述するオブジェクトの作成


上記のWCFServiceの場合と同様のアクションを、WCFFunctionという名前でのみ実行します。 そして、WCFCollectionにではなく、WCFFunctionを（ 埋め込み関係を使用して）WCFServiceにバインドします。 そして、次のようなものが得られます。







これで、サービスに関数を追加できますが、ダイアグラムには表示されません。 束にマッピングを追加します。

タイプConnectorのオブジェクト（プロパティで外部表示を構成できます）をドラッグし、 ダイアグラム要素マップを使用してWCFServiceHasWCFFunctionオブジェクトに関連付けます。

オブジェクトの型を記述するオブジェクトを追加して、関数のパラメーターと戻り値の型を設定します。

WCFTypeという名前のDomain Classオブジェクトを追加し、その中にNamespaceとTypeという2つのテキストプロパティを作成して、型の名前空間と型自体を指定します。 Inheritance Modifierプロパティをabstractに設定します。

また、WCFParamType（関数パラメーターのセットを表す）という名前のDomain Classオブジェクトを追加します。これは、WCFTypeから継承されます（ Base ClassパラメーターをWCFTypeに設定）。

WCFParamTypeの場合は、Nameプロパティを追加して関数内のパラメーター名を識別し、図に示すようにWCFServiceに関連付けます。







次に、WCFFunctionShapeオブジェクトで、コンパートメントを追加します（Paramと呼びましょう）。 WDSLFunctionとWCFFunctionShapeの間のリンクが強調表示されたDSL Detailsパネルに移動しましょう。 [コンパートメントマップ]タブを選択し、図のように構成します。







Displayed elements collection pathフィールドは、（バンドルを使用して）表示する要素のコレクションへのパスを示し、表示するプロパティの名前を指定します。

これで、簡単なコードの生成を開始するのに十分な情報が得られました。
