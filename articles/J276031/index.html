電話からクラウドへ：Azure Mobile Services。 どこから始めますか？



投稿者：Anastasia Belokurova、.NET開発者DataArt。



Azureモバイルサービスクラウドをモバイルアプリケーションに使用する方法と理由を説明します。 これを行うには、小さなアプリケーションを分析し、その例ですべてがどのように機能するかを確認し、実践と理論の両方に触れます。



しかし、その前に、クラウドテクノロジーの利点と、これらのテクノロジーがモバイルデバイスにどのように関係するかを思い出しましょう。



クラウドの利点


「クラウドコンピューティング」という用語はかなり前に登場しましたが、その意味は少し変わりました。 最初は、クラウドコンピューティングは単に分散コンピューティングと呼ばれていました。 しかし今、クラウドプラットフォームは、問題を解決するために使用できる便利なサービスの大規模なセットです。



なぜ雲なのか？ アプリケーションには次の利点があります。

  • スケーラビリティ。
  • 信頼性
  • サービスの大規模な選択。




あなたが小さなオンラインストアを立ち上げていると想像してください-最初に、それがまったく需要があるかどうかを確認したいだけです。 予算が少ない。 そのため、小さなWebアプリケーションを作成してコンピューターに配置します。コンピューターは夜間にはオフになりません。 サイトは外部からアクセスでき、一度に10人のユーザーに耐えることができます。 時間の経過とともに、あなたのサイトは人気を博し、一度に100人のユーザーに耐えるためにすでに必要になっています。 しかし、お使いのコンピューターはそのような負荷に対処できません。 そして、あなたは私のアパートを私のためにサーバールームに変えるべきかどうか考え始めますか？..いいえ、これはひどく不便です！ 次に、ホスティングが提供するサーバーを購入します。 現在、サイトは1分あたり100件、さらには1,000件のリクエストをサポートしています。 最後に、ある時点で、多くのユーザーが24時間体制でいて、購入したサーバーが対応できなくなっているため、グローバル市場に参入していることに気付きます。



ここで、クラウドプラットフォームの最初の利点を紹介します。クラウドプラットフォームは、ほぼすべての時点でほぼすべてのスケーラビリティを提供します。 そして、これは非常に簡単、迅速、便利に行われます。場所と負荷についてではなく、開発について考えるだけで済みます。 これが最初の利点です。



二番目。 多くの訪問者とユーザーがサーバーでホストされているプロジェクトが成功したとします。 そして、met石がサーバーに落ちたり、何か他のことが起こります。 プロジェクトのすべてのデータはそれらに保存されていました...一方、ほとんどのクラウドプラットフォームはレプリケーションをサポートしています。アプリケーションを配置する地域を選択しますが、他の地域のサーバーに自動的に複製されます。 そのため、クラウドプロバイダーは常にデータを安全に保存します：不可抗力の状況では約20分で復元されることを約束します。



第三に、非常に重要な利点もあります。現在、ほとんどのクラウドプロバイダーは、使用できる多くの自転車サービスを提供しています。 これは、すべてのプロジェクトで使用できるコードです。 そのため、毎回同じコードを記述したり、同じアーキテクチャを構築したりすることはできません-この多くは提供されています。 これを活用して、特定のプロジェクトに必要な特定のことだけを考えることができます。



モバイルデバイスとクラウドテクノロジー


モバイルデバイスはどのようにクラウドに接続されていますか？



多くのモバイルアプリケーションは、接続されたインターネットでのみ機能します。 そのため、電話ではなくサーバー上のどこかにあるロジックを使用できます。これにより、デバイスのリソースを節約できます。



現在、日常生活でモバイルデバイスの使用が増加する傾向にあることに注目する価値があります。 そしてもちろん、これらすべてのモバイルデバイス間でデータが同期されていると便利です。 たとえば、すべてのデバイスがApple製である場合、問題はありません。 しかし、デバイスのOSが異なる場合はどうでしょうか？ OSごとに別々にアプリケーションを作成しますか？ これは不便です！ そして、一般データをどこに保存しますか？ したがって、次の問題のリストを強調表示できます。

  • 異なるプラットフォームで実行する必要があります。
  • プラットフォームごとに1つのアプリケーションを作成するのは不便です。
  • 他のアプリケーションからデータにアクセスする必要があります。




クラウドテクノロジーは、これらの問題の解決に役立ちます。 Azure Mobile Servicesの例を使用したソリューションを見てみましょう。



Azure Mobile Servicesを使用してアプリケーションを構築する


見ていくアプリケーションは、タスクのリストです。 これは、Azure Portalにアクセスしてダウンロードできる最も単純なアプリケーションです。 このアプリケーションでは、タスクを作成、削除、名前を変更できます。 アプリケーションには、Azureサービスを操作するための基本機能が含まれています。







このアプリケーションは、OSが異なるさまざまなデバイスで動作し、すべてが同期される必要があります。 PhoneGapを使用してAzureモバイルサービスを使用してモバイルアプリケーションを作成する方法に関するタスクリストの例を見てみましょう。 PhoneGapは、一度コードを記述し、それを複数のプラットフォーム用にコンパイルできるフレームワークです。 つまり、OSごとにネイティブアプリケーションを作成する必要はありません。



PhoneGapに加えて、このようなアプリケーションを開発するには、次のものが必要です。

  • SQL Azureは、すべてのデータが保存されるベースです。 このサービスはモバイルサービスとは独立して機能しますが、モバイルサービスにはSQL Azureとの連携を容易にする機能があります。
  • SDK：Android SDK、Windows Phone 8および/またはXcode用のVisual Studio Express-開発対象のプラットフォームに応じて。 Visual Studioでの開発を紹介します。




私たちが直面している問題を解決するには、Azure Mobileサービスを作成する必要があります。 これを行うには、 Azure Portalに移動して、[モバイルサービスの作成]（計算>モバイルサービス>作成）をクリックする必要があります。

  • タスクスケジューラ。 たとえば、アプリケーションで、たとえば週に1回ユーザーに通知が表示されるようにする必要があります（オンラインストアの場合、これは割引に関する情報である可能性があります）。




データベース


データベースに関しては、これはSQLデータベースです。 任意のクエリを作成でき、必要なインデックスがあります。



非常に良いことは、すべてのテーブルを完全に構築できない動的レイアウトです。 アプリケーションの表に4つの列があり、列が提供されていない他のデータがそこに入力されているとします。 大丈夫-すべてが追加されます！



データベースは、このために設計されたすべての可能なポータルおよびアプリケーションから簡単に管理できます。 Azure Portal（読み取りおよび削除）、SQL Portal（Silverlight）、SQL Management Studio（Windows）、およびREST APIを使用できます。

クラウドサービスで他に何ができますか？


サービスの「スタートページ」をもう一度見てみましょう。







「データ」タブがあります-データベースを操作します。 そこにテーブルのリストが表示され、新しいテーブルを作成できます。 テーブル自体には、その中にあるデータが表示されます。 また、「スクリプト」セクションでは、テーブルを操作するために現在使用可能な操作を確認できます。



次のタブは「API」です。 たとえば、いくつかのテーブル間でより複雑な相互作用がある場合、作成、更新、削除の単純な機能では不十分な場合、APIを作成できます。



「スケジューラ」-タスクスケジューラがあります。



「プッシュ」-通知を作成します。



「アイデンティティ」-アプリケーションで認証メカニズムを構成できます。 識別用のすべてのデータは、スクリーンショットで見た「ユーザー」オブジェクトにあります。



「スケール」-サービスのスケーリング機能：サービスのサービスに割り当てられるリソースの量を変更したり、プッシュ通知の数を設定したりできます。



言語に関しては、.NET、Node.js、Java、PHP、Pythonでアプリケーションを記述し、それらをAzureに配置して、提供されるすべてのサービスとやり取りできるようになりました。



もちろん、Azureは無料ではありませんが、多くの機能を無料で提供する学生向けのDreamSparkプログラムがあります。 毎月の試用版サブスクリプションもあります。



一般に、Azure Mobile Servicesの操作は非常に便利で簡単です。すべての変更はその場で行われ（少なくともNode.jsの場合）、再コンパイルする必要はありません。 コードはどのデバイスからでもすばやく変更できます。 これは、単純なアプリケーションやプロトタイプに最適です。 しかし、複雑なプロジェクトにはあまり適していません-いつか、回避するのが非常に難しい制限に直面するでしょう。
