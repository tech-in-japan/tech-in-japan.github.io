Buildroot-パート1.一般情報、最小限のシステムの構築、メニューからの設定

はじめに



この一連の記事では、buildroot配布システムを検討し、カスタマイズの経験を共有したいと思います。 ここでは、グラフィカルインターフェイスと最小限の機能を備えた小さなOSを作成する実際的な経験を紹介します。







まず、ビルドシステムとディストリビューションを混同しないでください。 Buildrootは、提供されたパッケージのセットからシステムを構築できます。 Buildrootはメイクファイル上に構築されるため、非常に優れたカスタマイズ機能を備えています。 パッケージを別のバージョンに置き換え、パッケージを追加し、パッケージを構築するためのルールを変更し、すべてのパッケージをインストールした後にファイルシステムをカスタマイズしますか？ Buildrootはこれをすべて実行できます。







ロシアではbuildrootが使用されますが、私の意見では初心者向けのロシア語の情報はほとんどありません。







この作業の目的は、ライブローディング、icewmインターフェース、ブラウザを備えたディストリビューションを構築することです。 ターゲットプラットフォームはvirtualboxです。







ディストリビューションを作成する理由 多くの場合、限られたリソースで制限された機能が必要です。 多くの場合、自動化ではファームウェアを作成する必要があります。 汎用ディストリビューションを適合させ、不要なパッケージをクリーンアップしてファームウェアに変換することは、新しいディストリビューションを構築するよりも時間のかかる方法です。 Gentooの使用にも制限があります。







Buildrootシステムは非常に強力ですが、何もしません。 機会を与え、組み立てプロセスを自動化することしかできません。







代替ビルドシステム（yocto、オープンビルドシステムなど）は考慮または比較されません。







入手方法と開始方法



プロジェクトのWebサイトはbuildroot.orgです。 ここで、現在のバージョンをダウンロードして、マニュアルを読むことができます。 コミュニティに連絡することもできます。バグ追跡システム、メールリスト、IRCチャンネルがあります。







Buildrootは、アセンブリターゲットボードのdefconfigで動作します。 defconfigは、デフォルト値を持たないオプションのみを保存する構成ファイルです。 何をどのように収集するかを決定するのは彼です。 この場合、構成busybox、linux-kernel、uClibc、ブートローダーu-boot、およびbareboxを個別に構成できますが、これらはすべてターゲットボードに関連付けられます。

ダウンロードしたアーカイブを解凍するか、gitからクローンを作成した後、buildrootを準備します。 ディレクトリ構造の詳細は、マニュアルに記載されています。最も重要なものについて説明します。







board-各ボードに固有のファイルを含むディレクトリ。 これらは、システムイメージ（iso、sdcart、cpioなど）、オーバーレイディレクトリ、カーネル構成などを形成するためのスクリプトです。

configs-実際にはdefconfigボード。 defconfigは、ボードの不完全な構成です。 デフォルト設定以外のパラメーターのみが保存されます。

dl-アセンブリ用のダウンロードされたソースコード/ファイルのあるディレクトリ

output / target-結果のOSのアセンブルされたファイルシステム。 将来的には、ダウンロード/インストール用のイメージがそこから作成されます

出力/ホスト -構築用のホストユーティリティ

出力/ビルド -コンパイル済みパッケージ







ビルド構成はKConfigを介して行われます。 Linuxカーネルの構築にも同じシステムが使用されます。 最も一般的に使用されるコマンドのリスト（buildrootディレクトリで実行）：







  • make menuconfig-ビルド構成を呼び出します。 グラフィカルインターフェイス（make nconfig、make xconfig、make gconfig）を使用することもできます
  • make linux-menuconfig-カーネル構成を呼び出します。
  • make clean-ビルド結果をクリアします（出力に保存されているすべてのもの）
  • make-システムを構築します。 既に収集されたプロセスは再構築されません
  • make defconfig_name-構成を特定のdefconfigに切り替えます
  • make list-defconfigs-defconfigのリストを表示します
  • ソースを作成する-アセンブリなしでインストールファイルのみをダウンロードします。
  • make help-可能なコマンドをリストします


重要な注意事項とヒント



Buildrootは既にパッケージ化されたパッケージを再構築しません！ したがって、完全な再組み立てが必要な場合に状況が発生する可能性があります。







make packagename-rebuildコマンドを使用して、単一のパッケージを再構築できます。 たとえば、Linuxカーネルを再構築できます。 







make linux-rebuild


Buildrootは、output / build / $ packagenameディレクトリに.stampファイルを作成することにより、パッケージの状態を保存します。













したがって、パッケージを再構築せずにroot-fsとイメージを再構築できます。 







 rm output/build/host-gcc-final-*/.stamp_host_installed;rm -rf output/target;find output/ -name ".stamp_target_installed" |xargs rm -rf ; make


便利な変数



Buildrootには、構成を容易にする一連の変数があります







  • $ TOPDIR-buildrootのルートディレクトリ
  • $ BASEDIR-OUTPUTディレクトリ
  • $ HOST_DIR、$ STAGING_DIR、$ TARGET_DIR-アセンブリディレクトリホストfs、ステージングfs、ターゲットfs。
  • $ BUILD_DIR-アンパックおよびアセンブルされたパッケージのあるディレクトリ


可視化



Buildrootには視覚化する機能があり、最終的なシステムで依存関係図、ビルドタイムライン、パケットサイズグラフを作成できます。 結果は、output / graphディレクトリにあるpdfファイル（svn、pngから選択できます）の形式です。







視覚化コマンドの例：







  • make graph-depends



    依存関係に依存関係ツリーを構築させる
  • make <pkg>-graph-depends



    はパッケージの依存関係ツリーを構築します
  • BR2_GRAPH_OUT=png make graph-build



    は、出力がPNGのアセンブリ時間のグラフを作成します。
  • make graph-size



    プロットパケットサイズを作成する


便利なスクリプト



buildrootディレクトリには、有用なスクリプトを含むutilsサブディレクトリがあります。 たとえば、パッケージの正しい説明を検証するスクリプトがあります。 これは、パッケージを追加するときに役立ちます（これは後で行います）。 utils / readme.txtファイルには、これらのスクリプトの説明が含まれています。







在庫分布を構築しましょう



すべての操作は、rootではなく通常のユーザーに代わって実行されることを思い出してください。

すべてのコマンドはbuildrootのルートで実行されます。 buildrootディストリビューションには、多くの一般的なボードと仮想化のための構成セットが既にあります。







構成のリストを確認します。













qemu_x86_64_defconfig構成に切り替えます 







 make qemu_x86_64_defconfig


そして、組み立てを開始します 







 make


アセンブリは正常に完了しました。結果を確認します。













Buildrootには、Qemuで実行して動作することを確認できるイメージがコンパイルされています。 







 qemu-system-x86_64 -kernel output/images/bzImage -hda \ output/images/rootfs.ext2 -append "root=/dev/sda rw" -s -S


結果は、qemuで実行されているシステムです。













カスタムボード構成の作成



ボードファイルの追加



構成のリストを確認します。













リストにはpc_x86_64_bios_defconfigがあります。 ボードを構成からコピーして作成します。 







 cp configs/pc_x86_64_bios_defconfig configs/my_x86_board_defconfig


すぐにボードディレクトリを作成して、スクリプト、rootfs-overlay、およびその他の必要なファイルを保存します。 







 mkdir board/my_x86_board


このdefconfigに切り替えます。 







 make my_x86_board_defconfig


したがって、ビルド構成（buildrootディレクトリのルートにある.configに格納）は、ロードすることにより、ターゲットのx86-64レガシー（bios）マシンに対応します。







linux-kernel設定をコピーします（後で役立ちます）： 







 cp board/pc/linux.config board/my_x86_board/


KConfigを使用してビルドオプションを構成する



セットアップを実行します。 







 make menuconfig


KConfigウィンドウが開きます。 グラフィカルインターフェイス（make nconfig、make xconfig、make gconfig）で設定することが可能です：













ターゲットオプションの最初のセクションに入ります。 ここで、アセンブリを実行するターゲットアーキテクチャを選択できます。













ビルドオプション-ここにはさまざまなビルド設定があります。 ソースコード、アセンブリスレッドの数、ソースコードをダウンロードするためのミラー、およびその他の設定でディレクトリを指定できます。 デフォルト設定のままにします。







ツールチェーン-ここでは、アセンブリツールキットが構成されています。 彼についての詳細。













ツールチェーンタイプ-使用されるツールチェーンのタイプ。 buildrootまたは外部ツールチェーンにビルトインすることができます（すでにビルドされているディレクトリまたはダウンロードするURLを指定できます）。 異なるアーキテクチャの場合、追加のオプションがあります。 たとえば、armの場合、Linaro外部ツールチェーンのバージョンを選択するだけです。







Cライブラリ-Cライブラリの選択システム全体の動作はこれに依存します。 通常、glibcを使用し、可能なすべての機能をサポートします。 ただし、組み込みシステムには大きすぎる可能性があるため、uClibcまたはmuslがしばしば選択されます。 glibcを選択します（将来、systemdを使用する必要があります）。







カーネルヘッダーおよびカスタムカーネルヘッダーシリーズ-ビルドシステム上にあるカーネルのバージョンと一致する必要があります。 カーネルヘッダーの場合は、tarballまたはgitリポジトリへのパスを指定することもできます。







GCCコンパイラーバージョン-アセンブリーに使用するコンパイラーのバージョンを選択します

C ++サポートを有効にする-システム内のc ++ライブラリをサポートするアセンブリを選択します。 将来的には、これが重宝します。







追加のgccオプション-追加のコンパイラオプションを設定できます。 今のところ必要ありません。







システム構成により、作成されたシステムの将来のパラメーターを設定できます。













ほとんどのポイントは名前から明らかです。 次の点に注意してください。

ユーザーテーブルへのパス-ユーザーが作成されたテーブル（ https://buildroot.org/downloads/manual/manual.html#makeuser-syntax ）。







ファイルの例。 ユーザーuserは、パスワードadmin、自動的にgid / uid、/ bin / shシェル、デフォルトグループユーザー、ルートグループメンバー、コメントFooユーザーで作成されます 







 [alexey@alexey-pc buildroot ]$ cat board/my_x86_board/users.txt user -1 user -1 =admin /home/user /bin/sh root Foo user


ルートファイルシステムオーバーレイディレクトリ-組み立てられたtarget-fsの上に重ねられたディレクトリ。 新しいファイルを追加し、既存のファイルを置き換えます。







ファイルシステムイメージを作成する前に実行するカスタムスクリプト-ファイルシステムをイメージにフォールディングする直前に実行されるスクリプト。 現時点では、スクリプト自体は空白のままです。







カーネルセクションに移りましょう













カーネル設定はここで設定されます。 カーネル自体は、make linux-menuconfigを使用して構成されます。

カーネルのバージョンはさまざまな方法で設定できます：提案されたものから選択し、バージョンを手動で入力し、リポジトリまたは既製のtarballを指定します。







カーネル構成-カーネル構成へのパス。 Linuxから選択したアーキテクチャまたはdefocnfigのデフォルト構成を選択できます。 Linuxソースには、異なるターゲットシステム用のdefconfigのセットがあります。 ここでソースを直接見ると 、正しいものを見つけることができます。 たとえば、ビーグルボーンブラックボードの場合、configを選択できます 。







[ターゲットパッケージ]セクションでは、ビルドするシステムにインストールするパッケージを選択できます。 とりあえずそのままにしておきましょう。 後でパッケージをこのリストに追加します。

ファイルシステムイメージ-コンパイルするファイルシステムイメージのリスト。 ISOイメージを追加













ブートローダー-組み立てられたブートローダーの選択。 イソリニクスを選択













システム構成



Systemdは、カーネルとglibcとともに、Linuxの投稿の1つになります。 そのため、別の段落で構成を行いました。







make menuconfig、ターゲットパッケージ→システムツール→systemdの順に構成します。 ここでは、システム起動時にインストールおよび開始されるsystemdサービスを指定できます。













システム構成の保存



この構成をKConfig経由で保存します。







次に、defconfigを保存します。 







 make savedefconfig


Linuxカーネル構成



Linuxカーネル構成は、次のコマンドで呼び出されます。 







 make linux-menuconfig


Virtualboxグラフィックスカードのサポートを追加する













Virtualboxゲスト統合サポートを追加する













保存して終了します。 重要 ：設定はoutput / build / linux- $ version / configに保存されますが、board / my_x86_board / linux.configには保存されません

 cp output/build/linux-4.19.25/.config board/my_x86_board/linux.config

 make clean;make
