iOS SDKを調べて、文書化されていないAPIを使用する

この章とこの本全体から、iOSプログラミングのほとんどの情報が公開フレームワークに含まれていることは明らかですが、SDKには含まれていません。 これに関するAppleの非公式のポリシーは単純です。これはすべて使用できますが、ご自身の危険とリスクがある場合に限ります。 次のファームウェア更新中にコードが破損する場合があります。 あなた自身がリスクと利益の妥協点を見つけなければなりません。

この章および本書で説明したように、iPhoneプログラミングの最も優れた部分のいくつかは、公開のiPhoneフレームワークに含まれていますが、SDKには含まれていません。 これに関するAppleの非公式のポリシーは明確です：これらのアイテムはプログラムで使用できますが、使用するのは自己責任で行い、ファームウェアのリリースごとにコードが破損する可能性があります。 リスクと報酬のバランスをとるのはあなた次第です。

異なるファームウェアと異なるデバイス：何が壊れるか、どのように修正するか？


文書化されていないAPIがすべてのデバイスで必ずしも同じように機能するわけではないことは明らかです。 私の経験では、ほとんどの場合何も変わらず、APIはすべてのデバイスとすべてのファームウェアで同じように機能します。 したがって、たとえば、 UIDevice-IOKitExtensions拡張機能のすべての機能（IMEIの定義を除く）は、すべてのデバイスおよびすべてのファームウェアで同等に機能します。 iOSの更新時にどのような変更が発生しますか？

以下に実用的なオプションを示します。

  • 公式に文書化されたプログラミングインターフェイスが表示される場合がありますが、文書化されていないインターフェイスは原則として引き続き機能します。 例： 

     void GSEventSetBacklightLevel(float level); //     iOS -[UIDevice setBrightness: (CGFloat)brightness]; //   iOS 5.0


  • プログラムインターフェイスは別のフレームワークに転送されます。 Appleは、複数のフレームワークを1つに結合し、フレームワークの名前を変更または削除できます。 たとえば、WiFiを操作するための関数（Apple80211Open、Apple80211Closeなど）はAeropuerto.dylibからIPConfiguration.dylibに転送されました。
  • 特に脆弱性に関連する場合は、APIを簡単に削除できます。
