Webサービスデザインシリーズ

みなさんこんにちは！



私たち、Denis Beskov （ beskov ）とIlya Polyak （ ilyap1 ）は、Webサービスの設計プロセスに関する一連の記事を公開し始めています。 サイクルは、分析と設計のレベルが明確に区別される3レベルのプロセスを中心に構築されます。

  1. 事業内容
  2. 製品
  3. テック
ここでは、ハブで、技術レベルでの設計のトピックが適切にカバーされています。 スルーデモケースの例を使用して、このレベル、作業の種類、およびデザインソリューションのタイプと、より高いレベルのデザインソリューションの関係を示したいと思います。著者の1人が数年前に開発に参加した映画予約Webサービスのデザインです。



一連の記事は、社内の従業員によって開発された冗長なプロジェクトドキュメント（要件）を中心に構築され、理論的な側面の簡単な説明が含まれており、仕事での次の質問に答えるのに役立ちます。

  1. ソフトウェア要件を説明するにはどうすればよいですか？
  2. ToRにはどのシステム要件を含める必要があり、どの要件を省略できますか？
  3. 要件の説明形式を選択する際にどのようなオプションがありますか
  4. 要件の説明の種類の選択は、プロジェクトのパラメーター（期間、リスクなど）にどのように依存しますか？
  5. どのような決定が、適切なタイプの要件を取るのに役立ちますか？
このサイクルは、カスタムソフトウェア開発の側面には対応していません。 この記事は、非スポーツ環境で働くユーザーを対象としています。 規制に縛られていません。



エンターテインメント情報ポータル用の新しいサービスを設計する例を使用して、次のことを説明します。

  1. 分析と設計の3つのフェーズで行う必要がある決定：「ビジネス」、「製品」、および「テクノロジー」。
  2. 分析と製品設計への古典的なアプローチ。
  3. 要件を提示するためのさまざまな形式（プロセスモデル、用語集、ユースケースなど）。
  4. 要件の表示の形式は、各状況で最も適切です。
このサイクルは、「ひざまずき」ではなく、ソフトウェア開発業界のベストプラクティスに基づいて、Webサービスの設計方法を学びたいプロジェクトマネージャー、初心者システムアナリスト、デザイナー、インターフェイスデザイナー、マーケティング担当者、および開発者向けに設計されています。



ソフトウェア製品の開発要件のいくつかのレベル（ビジネス分析と設計、製品分析と設計、技術分析と設計）で設計決定を行うプロセスは、afisha.ruのチケット予約サービスの例に示されています。
