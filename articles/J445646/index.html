CLion 2019.1：ClangFormat、Clangdによるコードの強調表示、メモリビュー、マイクロコントローラーの初期サポート

こんにちは、Habr！



CLionチームには多くの素晴らしいニュースがあります-チームのサンクトペテルブルクの一部は、他の同僚と一緒に新しいオフィスに移動し、新しいクールな開発者が参加しました 。



新しいバージョンでの作業は、一度にいくつかの面で行われました。



  • C ++言語サポートの改善：Clangdによるコードの強調表示、抽出および名前変更リファクタリングの改善、クラスのメンバー関数が静的に宣言できることの新しいチェック。
  • コードの記述スタイルのその他のオプション ：ClangFormatとの統合、C / C ++の変数命名スタイルのサポート、ヘッダーガードのさまざまなスタイルのサポート。
  • デバッガーの新機能と改善 ：ポインターのメモリ状態の表示-メモリービュー-LLDBの場合は逆アセンブルされたコードの表示、ステップバイステップデバッグの作業の加速。
  • マイクロコントローラ用のCLion 、最初のステップ。
  • CLionで起動/デバッグするためのビルドターゲットと構成を作成する機能。プロジェクトモデルとはまったく関係ありません。
  • C / C ++の文字列リテラルで他のプログラミング言語を使用します。
  • 新しい視覚テーマとその他のプラットフォーム機能。






これらおよびその他の革新については、以下をご覧ください。 また、新しい機能や改善を試すには、 当社のWebサイトから無料の30日間バージョンのCLion をダウンロードしてください 。



C ++言語のサポート



クランド



ご存じのとおり、CLionにはC ++言語をサポートするための2つのツールがあります。1つは完全に独自のもので、2つ目はClangdに基づいています。 彼らは協力して、お互いを補完し、必要な情報を交換します。 同時に、パフォーマンスと機能的完全性の基準が許せば、CLionでC ++コードを操作するためのスマートツールをClangdベースのツールに移行しようとしています。 これまでのところ、コードのリファクタリングについては話していませんが、2019.1でのコードの強調表示はClangdに基づいて既に行われています。 これにより、エディターの応答性が大幅に向上しました。



少し前のリリースでは、エディターでエラーを表示するときにCLionをClangdベースのツールに切り替えました。 エラーテキストがより詳細に表示されるようになりました。 これは、たとえば、関数のオーバーロードに関連するエラーをデバッグする場合に便利です。







これに追加されるのは、Clangdを介して可能な修正（クイック修正）の位置を計算する機能です。 修正自体は、CLion独自のツールによって直接提供されます。



私たちの仕事のもう1つの興味深い分野は、Clangdコード解析ツールで新しいチェックを作成することです。 CLion 2019.1以降では、C ++コードの新しいチェックにより、クラスのメンバー関数を静的と宣言できる時期がわかります。







ちなみに、Clangdでこの代替ツールの設定を管理するには、[設定]> [設定] | [ 言語とフレームワーク| C / C ++ | クランド。

編集者の生産性は私たちの最優先事項の1つです。 このリリースでは、多くの小さな改善に加えて、プロジェクトの初期インデックス作成時間の大幅な改善に注目する価値があります。 常に発生するわけではありませんが、プロジェクトに同じライブラリを使用する場合：CLionはこれを自動的に認識し、それらを使用する新しいオープンプロジェクトでこれらのライブラリのシンボルを再利用できます。 STLまたはBoostに関しては、改善は非常に顕著です！



今年の計画は、C ++のリファクタリングの精度と精度を高めることです。 最も基本的な2つの名前変更と抽出から始めました。 Extractでは、名前空間修飾子（ std::

名前の変更に関連して、クラスまたは構造の名前が、それらが配置されているファイルの名前と一致する場合に注意を払いました。 以前は、常にファイルの名前も変更していましたが、CLionはリファクタリング中に望ましい結果について尋ねます。 名前を変更することも、古いファイル名のままにすることもできます。 その逆も機能します。ファイルの名前を変更しても、クラスの無条件の名前変更は行われません。 （ここのどこかで観客からの叫び声があるはずです：「ついに！」）

LLDBの場合の分解



アセンブラコードは機能別に分割され、最も重要なこととして、GDBの場合だけでなく、LLDBの場合にも表示されます。







ただし、アセンブラコードは、関数のソースコードがない場合にのみ表示されることに注意してください。 いわゆる逆アセンブルオンデマンドモードはまだサポートされていません。



段階的なデバッグパフォーマンス



場合によっては、各ステップで変数が長時間計算されるため、段階的なデバッグが遅れることがあります。 しかし、時には、誰もこれらの計算を必要としないこともあります。コードの特定の領域をできるだけ早く調べ、時には変数のペアの値を調べたいと思います。 CLionでは、段階的なデバッグ中に変数の再カウントを無効にする機会があります-デバッガーコンテキストメニューの[ 変数のミュート ]はまさにそれを行います。 また、値を計算して表示する必要がある場合は、変数の[ ロード ]をクリックできます。







マイクロコントローラー用CLion



Ilya Motor（ elmot ）は、すでにHabrで、CLionをSTM32CubeMXと統合し、OpenOCDデバッガーをサポートするためのプラグインについて書いています。 昨年末、イリヤは私たちのチームに加わり、すでにプラグインを大幅に更新し、CLion内に移動することができました。







更新されたプラグインに関するかなり大規模で詳細なブログ投稿は、 ブログにあります。 ここで、今できる最も重要なことを説明します。



  • [新しいプロジェクト]ダイアログで、STM32CubeMXプロジェクト（ .ioc ）を作成できます。
  • CLionから直接、プロジェクトに対してSTM32CubeMXを実行して、マイクロコントローラーの設定を更新し、プロジェクトのコードを生成します。
  • CLionは、このプロジェクトを操作するための正しいCMakeファイルを生成します。
  • CLionは、ハードウェアの構成ファイル（ボード構成）の選択を提案します。
  • OpenOCDを使用してデバッグするには、特別なタイプ「OpenOCDのダウンロードと実行」の構成を作成する必要があります。 STM32CubeMXプロジェクトの場合、CLionは独自に作成します。 すべての設定を指定したら、CLionから直接マイクロコントローラーでデバッグできます！


イリヤには多くの野心的な計画があるので、あなたのフィードバックは私たちにとって非常に重要です。 CLionの組み込みシステムの開発に興味がある場合は、コメントでお待ちしています！
