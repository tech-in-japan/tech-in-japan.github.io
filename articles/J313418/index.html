vCenter ServerをvSphere 6.0 Update 2mに移行する準備をしています。 パート2

VPSを新しいバージョンに移行する-多くの場合、管理者にとっては「頭痛」ですが、「新しいバージョンにアップグレードしない」オプションには、将来問題が発生する可能性があります。 そのため、仮想マシンを5.5からvSphereのバージョン6に移行したいすべての人に役立つレビューを公開し続けています。







最初の部分では、vSphere Single Sign-OnドメインおよびさまざまなvCenter展開モデルの移行の準備について説明しました。 次に、さらに2つの「ショーのスター」に注目しましょう。WindowsvCenter Serverと、もちろんWindows vCenter Databaseです。



Windows vCenter Serverに関しては、すぐにネットワーク、アカウント、エージェント、インストールされているさまざまなアプリケーションなどに関する多くのポイントがあります。 Windows vCenter Serverと同様に、データは（移行プロセスの前に）事前に評価できます。移行プロセスは、展開モデル（内部/外部）に依存します。もちろん、移行はデータのサイズに依存します。 この投稿は、vCenter Server Windows、データベース、およびネットワークに関するものです。



ネットワーク



ネットワークは、vCenter Serverの移行で最も重要な部分の1つです。 移行を開始する前に、Windows vCenter Serverに正しいDNS構成があり、前方参照ゾーンと逆引き参照ゾーンの代わりになっていることを確認してください。 理想的には、vCenter Serverが静的DNSとして構成されているが、DHCPサポートが必要な場合、vSphere 6.0 Update 2mはこれをわずかな警告でサポートします。



まず、Windows vCenter Serverがホストと同じIPを使用していないことを確認します。 この場合、この構成はアシスタントを介した移行中はサポートされず、移行の準備中であってもブロックされるため、移行できません。



次に、Windows vCenter ServerがDHCPアドレスを持つFQDNを使用する場合、すべてが予約されていることを確認します。そうでない場合、移行アシスタントもそれをブロックします。 また、ネットワークタイムプロトコル（NTP）が有効になっており、Windows vCenter Serverと正しく同期されていることを確認します。



vCenter Serverを使用する際のネットワーク構成について話すときは、ユーザーポートに注意を払う必要があります。 デフォルトでは、インストールプロセス中にvCenter Serverポートで行われた変更はすべて、移行アシスタントで確認されません。









ESXi Dump CollectorおよびAuto Deployでサポートされている構成済みポートのみが、vSphere Web Clientでサポートされます。 移行はポート9123を介して行われます。ポートが既に使用されている場合は、次の空きポートが使用されるか、ポートを指定できます。 最終的に、以下で説明する条件がなければ、移行プロセスは成功しません。 Windows vCenter Server 5.5が引き続き使用可能な状態で、新しいvCenter Server Appliance 6.0に一時IPアドレスをデプロイする必要があります。



Windows vCenter Serverのデータが収集されると、サーバーはシャットダウンします。元のIPアドレスを含むvCenter Serverの特性により、サーバーも移行されます。 一時IPアドレスは、Windows vCenter Serverで利用可能なルーティングネットワークの一部である必要があります



Windowsの設定



Windows vCenter Serverは、移行プロセスで最も重要なアイテムです。 ここで、Windows vCenter Serverに既に含まれているものを簡単に見てみましょう。 他のアプリケーション、アプリケーションの一部、エージェント、スクリプトなどがあります。 VMwareまたはサードパーティのメーカー。 vCenter Serverにインストールされているものだけでなく、仮想サーバーが最終アプリケーションとして使用するものにも注意を払ってください。 移行プロセス中にWindows vCenter Serverおよび関連するすべてのアプリケーションを保存すると、移行後、移行前と同じ仮想マシンが取得されます。 プラグインの更新が必要なアプリケーションや、バックアップの場合は他の変更が必要なアプリケーションがあります（それらについては以下で詳しく説明します）。



Appliance ServerとWindowsの両方のVCenter vSphere 6.0は、イメージレベルのバックアップのみをサポートします。 移行できるかどうかは、vCenter ServerがActive Directoryドメインに接続されているかどうか、またはワークグループであるかどうかによって異なります。 Windows vCenter ServerがActive Directoryドメインに接続されている場合、資格情報を持つユーザーはvCenterサーバーのアカウントをアップグレードする必要があります。 このナレッジベースでは、Active DirectoryのユーザーがvCenter Serverをアップグレードするために必要な権限について説明します。 また、WindowsからLinuxに切り替えたため、ローカルのWindowsユーザーとグループは移行されないことを覚えておく必要があります。 vCenter Serverのアクセス権にアカウントまたはグループを使用する場合、これは非常に重要です。



強調する必要があることの1つ：必要に応じて移行プロセス中にロールバックされるように、Windows vCenter Serverのローカルアカウントにアクセスできる必要があります（移行のロールバックの側面については、次のレビューで詳しく説明します）。



移行プロセス中に新しいバージョンに移行されないという話をまとめると、インストールされているエージェント、ウイルス対策プログラム、監視プログラム、バックアップなども移行されないということです。 vCenter Server Appliance-また、何もサポートしていません。 vCenter Serverがサービスアカウントおよび移行が行われるアカウント（移行アシスタントが起動される）から起動される場合、Windowsセキュリティ権限を持つアカウントから「プロセスレベルトークンを置き換える」機能の権限が必要です。 これらの権限は、移行アシスタントがシミュレートされたアクセストークンを渡すことができるようにするために必要です。



Windows vCenter Serverに既にインストールされているソリューションを見てみましょう。 VMwareまたはサードパーティベンダーに起因するこれらのアプリケーションは、Windows vCenter Serverがシャットダウンされるまで利用できません。 VMware Update Manager（VUM）には、vCenter Serverにインストールできるユーザー向けの汎用アプリケーションが既に含まれています。 移行アシスタントは、インストールされたこれらのアプリケーションとシグナルをマークしますが、もちろん続行したい場合は移行プロセスを続行します。 移行後にこれらのインストール済みアプリケーションが必要な場合は、移行プロセスを開始する前に別のサーバーにインストールすることを強くお勧めします。







これらの移行手順では、外部サーバーへのVUMのインストールについて詳しく説明しています。



特にVUMに関しては、ユーザーごとに個別のインストールごとにこれらのアプリケーションプラグインが必要です。 移行プロセスを開始したユーザーが、アプリケーションに必要なすべてのプラグインを既にインストールしていることを確認してください。そうしないと、移行アシスタントがエラーをスローします。 たとえば、VUMプラグインがインストールされていないユーザーのアカウントに移行するプロセスを開始した場合、エラーが表示されます。



最初の部分ですでに述べたように、移行だけでなく更新も行っています。 考慮すべき点がいくつかあります。



  • リンクモードが構成されている場合、移行が開始される前に構成が削除されます。
  • 移行プロセス中、Windows vCenter ServerとvCenter Server Applianceは、移行エンドポイントで同等の部分を持ちます。
  • すべてのvCenter Server証明書も移行されます。
  • Windows vCenter Serverに保存されているスクリプトは、移行を開始する前に別の場所に移動する必要があります。
  • Windows vCenter Serverの移行が完了したら、ネットワークアダプターから切断し、何らかの理由でvCenterがオンになっていることを確認して、ネットワークの競合がないようにすることをお勧めします。


vCenter Serverデータベース



vCenter Serverデータベースは、Windows vCenter Serverの第2ウィングです。 移行ツールは、Microsoft SQL（ローカルまたはリモート）からOracleへのすべてのタイプのvSphere 5.5データをサポートします。 ローカルにインストールされたデータベースは、Windows vCenter Serverがオフになるまで移行後に利用できません。 vCenter Server 6.0 Update 2mは、イベント履歴と構成を転送する機能を提供します。 これにより、移行時間が長くなることに注意してください。 移行にかかる時間（構成、分析、イベント履歴、またはそれらのないバージョンで）をより正確に見積もるには、 特別なスクリプトを使用します 。 vCenterデータのサイズに応じて、移行時間の短縮に役立つオプションもありますが、移行評価スクリプトを最初に実行して何が起こるかを確認することをお勧めします。 vCenter Serverのデータを高速化する1つのことは、組み込みの統計ツールです。 ただし、エラーが修正されるまで、第2レベルの統計の後は機能しません。 各レベルの統計の使用方法と、いつどのように使用できるかについて説明します。









vCenter ServerのデータがリモートSQLまたはOracleデータベースである場合、ロールバック機能が機能するかどうか、およびデフォルト設定で実行されているかどうかを確認すると便利です。 これについては、vCenter Serverに大規模なデータベースを持っているお客様と常に話し合っていますが、この非常に必要なロールバック機能が動作する（動作している）かどうかは常にわかりません。 vCenter Serverでこの機能を起動すると、データベースが大幅に削減されます。 次のオプションは、vCenter Serverのデータベースの削減とクリーン化にも役立ちますが、経験豊富な管理者が使用することをお勧めします。



vCenter Serverが長期間更新されていないか、データベースが大きい場合は、古いデータを消去する必要があります 。 これを行う前に、すべてのデータのバックアップを作成してください。 これらのヒントは迷惑に思えるかもしれませんが、これらはすべて、移行とアップグレードの前に行うのが最善であると主張します。



データベースの移行準備ができているかどうかを確認する場合は、スクリプトを実行してデータの移行時間を推定し、時間だけでなく、発生する可能性のあるエラーも確認することをお勧めします。 また、移行アシスタントでフォルダーをWindows vCenter Serverにコピーして実行します。 アシスタントは、移行の準備を開始し、Windows vCenter Serverが移行のすべての要件を満たしているかどうか、または他の何かを修正する必要があるかどうかを評価できます。 また、潜在的な変更の表示（移行プロセスのウィンドウ）もあります。



このシリーズの最後の投稿では、このロールバック機能と移行完了後のタスクについて詳しく説明します。
