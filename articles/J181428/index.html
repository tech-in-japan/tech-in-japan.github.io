もちろん、ほとんどのヒントは非常に効率的です。 JRebel、コードのシンプルさ、自動化について。 しかし。

すべてが節度に優れています。 私の意見では、アントンはシンプルさを求めて一瞬をやり過ぎています。 Javaが作成されたのではなく、このアプローチでPHPへの直接の道がありました（holivarovなしでお願いします）。 私はエンタープライズが何であるかを直接知っていますが、今はJ2EEを捨てず、J2SEで複雑で複雑な分散および統合プロジェクトをコーディングしていませんか？ ここで、もちろん、十分な経験があり、これがあなたのプロジェクトにすぎず、あなたが何をしているのか知っているなら、それは本当に簡単で、より美しくなります。 より速く。 また、プロジェクトが大規模な場合、新規参入者の入場基準はどうですか？ Java Bean、Tomkat、JSF、およびHibernateは、事実上の標準として誰もが知っています（そのため、認定されています）。PureJ2SEのあなたのバイクは、たとえ美しく、純粋で、シンプルで、少なくとも1,000回の形容詞であってもそうではありません。 そして、彼らがより良く、より速く働くかもしれないと主張する人はいません。 同様に、アーキテクチャ、管理者、アーキテクト、推定へのアプローチを使用します。 さて、美しいコードのために美しいコードを書くことはできません。 私たちは皆、何らかの形で私たちを養うビジネスのコードを書きます。 また、これはビジネスであり、ユートピアプログラマのサンドボックスではないため、マネージャーやアーキテクトなど、このビジネスの他の参加者のニーズや要望を考慮する必要があります。 彼らはただ絵を描くだけではありません。 私たちがコードを書いている間、誰かが製品のマーケティング、市場への参入、潜在的な顧客/バイヤー/顧客の解決を考えています。そして、プログラマーがそこで何をしているのか知らずに、彼ら全員が仕事をすることができません。

だから私は繰り返します-すべてが適度に良いです。



私はここで批判するつもりはなかった、私はレポートが好きだった。 私自身もKISSの原則を固守しています。 しかし、私はアントンがこの観点にどのように関係しているか、そして彼がそれについて知っているかどうかを本当に理解していません。 または、アントンは、企業で働くことは決してなく、ポジティブなスタートアップからのみドライブを得ることを主張していますか？

そして、私はまた、アンバーの立場をどう見るかについても興味がありますか？

たぶん私自身は正しくないと思いますか？ 一般的に、アントンは「あなたの製品を私に売ってください」と試みます。
