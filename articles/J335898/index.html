Linux History（1993–2003）：ディストリビューションのテスト

オープンソースプロジェクトのユニークな特徴は、彼らの人生が本当に終わらないことです。 ほとんどの場合、ディスクイメージはインターネットからダウンロードでき、ライセンスの有効期限はありません。 したがって、仮想マシンに90年代からLinuxをインストールすることで、過去に飛び込むのは非常に簡単です。







どうだった？ 今日、古代のLinuxディストリビューションはどのように認識されていますか？ 開発の長年にわたって何が変わったのですか？ これを見つけてください。 私たちの旅の最初のポイントは、20年前にcomp.os.linux.announceニュースグループに送られたSlackware 1.01 OSです。



Slackware 1.01（1993）





Slackware 1.01



おそらく、現在検討されているディストリビューションの中で、Slackware 1.01が試してみるのが最も簡単でしょう。 実際、2014年のQemuシリーズの無料画像にはすぐに使用できる画像があるため、システムを手動でインストールする必要はありません（ただし、このような贅沢に慣れない方が良いでしょう）。 



$ qemu-kvm -m 16M -drive if=ide,format=qcow2,file=slackware.qcow2 \ -netdev user,id=slirp -device ne2k_isa,netdev=slirp \ -serial stdio -redir tcp:22122::22


多くのLinux 1993は、予想どおりに見えます。 ls



やcd



などのすべての基本的なコマンドが動作し、すべての基本的なツール（ gawk



、 cut



、 diff



、 perl



、そしてもちろんelvis



- Patrick Volkerdingのお気に入り）が存在し、機能していますが、ちょっとした驚きがあります。 たとえば、オートコンプリート中のBASHの動作方法、および配布キットに圧縮ファイル（ zless



、 zmore



、 zcat



）を調べるためのツールが既に含まれているという事実。 このシステムは、多くの点で驚くほどモダンに見えます。 実際、これは私が期待したことです。



ここで欠けているのは、少なくともある種のパッケージ管理ツールです。 すべてのインストールとアンインストールは完全に手動で実行され、補助的なメカニズムはありません。



一般に、Slackware 1.01は、かなり現代的なUNIXシステムとして、より正確には、現代のUNIXシステムがLinuxユーザーによって認識されると認識されています。 その中のほとんどすべてはおなじみですが、こことそこには違いがあります。 さらに、1993年にリリースされたオペレーティングシステムから期待されるほど多くはありません。



Debian 0.91（1994）



Debian 0.91をテストするために、 Ibiblioデジタルアーカイブから1994年に登場したフロッピーディスクイメージを使用しました。 システムの起動に必要なコマンドは次のとおりです。 



  $ gunzip bootdsk.gz basedsk1.gz basedsk2.gz $ qemu-system-i386 -M pc -m 64 -boot order=ac,menu=on \  -drive file=bootdisk,if=floppy,format=raw \  -drive file=debian.raw,if=ide,format=raw \  -device ne2k_isa,netdev=slirp \  -serial msmouse -vga std \  -redir tcp:22122::22 \  -netdev user,id=slirp


Debian 0.91のブートディスクは、明確な指示を含む単純なシェルを起動します。



インストールプロセスは驚くほどスムーズでした。 ハードディスクのパーティション分割とext2ファイルシステムでのフォーマットから、 basedsk



イメージのコピーまで、7つのステップの実行を目的としたメニューシステムの使用を提供します。 その後、最新のLinuxユーザーがOSに期待できる多くの馴染みのある慣習に対応する、最小限のDebianインストールを自由に使用できます。



今日、Debianはパッケージ管理システムで有名ですが、この初期のリリースではこのシステムの弱いヒントしかありませんでした。 dpkg



コマンドは存在しますが、インタラクティブなメニューベースのシステムです。いくつかのメニューレベルがあり、驚くことではありませんが、ごく一部のパッケージしか利用できません。



このような状況でも、システムの設計方法において、ユーザーの利便性に対する懸念を感じることができます。 動作中のOSの所有者になるには、3つのフロッピーディスクイメージをダウンロードするだけで、簡単なテキストメニューを使用して追加のプログラムをインストールできます。 DebianがLinux業界に強い影響を与えている理由は間違いなくわかります。



ジュリックス/ SuSE（1996）





Jurixをインストールする



SUSEの前身であるJurixには、ディレクトリに編成されたバイナリ.tgz



が付属していました。これは、Slackwareインストールパッケージの構造に似ています。 また、インストーラー自体はSlackwareインストーラーに似ています。 



  $ qemu-system-i386 -M pc -m 1024 \  -boot order=ac,menu=on \  -drive \   file=jurix/install,if=floppy,format=raw \  -drive file=jurix.img,if=ide \  -drive file=pkg.raw,if=ide,format=raw \  -device ne2k_isa,netdev=slirp \  -serial msmouse -vga std \  -redir tcp:22122::22 \  -netdev user,id=slirp


最初のリリースでは特別な検索を行わなかったため、Jurixディストリビューションは、グラフィカル環境で動作するように設計された最初のディストリビューションであることが判明しました。 XFree86はデフォルトでインストールされるため、使用しない場合は削除する必要があります。



設定例のファイル/usr/lib/X11/XF86Config



、後にXorg.conf



として知られるようになりXorg.conf

、 vsync



微調整、 ramdac



カラーカードの再割り当てにより、週末全体がなくなり、その後あきらめました。



Jurixに新しいパッケージをインストールするのは簡単です。 パッケージディスクで.tgz



ファイルを見つけて、次のコマンドを実行する必要があります。 



 $ su -c 'tar xzvf foo.tgz -C /'


パッケージは、ルートパーティションに解凍すると使用できるようになります。 そのため、システムのインストール後にインストールされなかったいくつかのパッケージをインストールしましたが、パッケージを使用するこの方法はシンプル、高速、信頼性が高いことがわかりました。



SUSE 5.1（1998）





SuSE 5.1で実行されているFVWMウィンドウマネージャー



1998年にメリーランド州のコンピューターストアで購入したInfoMagic CD-ROMからSUSE 5.1をインストールしました。 



  $ qemu-system-i386 -M pc-0.10 -m 64 \  -boot order=ad,menu=on \  -drive file=floppy.raw,if=floppy,format=raw \  -cdrom /dev/sr0 \  -drive file=suse5.raw,if=ide,format=raw \  -vga cirrus -serial msmouse


インストールプロセスは、私が既に遭遇したことと比較するとそれほど単純ではありませんでした。 YaSTは、設定ファイルと設定をフロッピーディスクと起動可能なCDに分散させていたため、数回の再起動が必要でした。 システムが私に期待するアクションのシーケンスを理解しようとしていたとき、最初からやり直すのに数回かかりました。 何回かすべてを台無しにした後、YaSTの原理を理解し、3回目はうまくいきました。 これは、Linuxユーザーが後年やらなければならなかったことのヒントでした。



SUSE 5.1では、私の主な目標はグラフィカル環境でした。 セットアッププロセスはおなじみで、マウスをテストし、問題を監視およびトラブルシューティングするために設計されたいくつかの優れたグラフィカルツール（優れたX86Setup



ツールを含む）を使用して実行されました。 グラフィカルインターフェイスが適切に機能するためには1時間もかかりませんでした。ほとんどの時間は、Qemu仮想グラフィックスカードがサポートする解像度と色深度を把握するための私自身の研究に費やされました。



ウィンドウマネージャーにはfvwm

およびctwm



含まれていfvwm

は、ファイルマネージャーの機能を実装する多機能パッケージで、UbuntuのUnity



シェルと非常によく似ています。



一般的に、SUSE 5.1での作業は好印象を残しました;私は特に、グラフィカルな作業環境のセットアップの便利さと速度が気に入りました。



Red Hat 6.0（1999）





Red Hatで実行されているGIMP 1.x



私が見つけた幸運な次のインストールディスクはRed Hat 6.0でした。 注意してください-RHEL 6.0ではなく、Red Hat 6.0のみです。 これは、RHELまたはFedoraが登場する前に店舗で販売されていたデスクトップディストリビューションでした。 使用したディスクは1999年に購入しました。 



  $ qemu-system-i386 -M pc-0.10 -m 512 \  -boot order=ad,menu=on \  -drive file=redhat6.raw,if=ide,format=raw \  -serial msmouse -netdev user,id=slirp \  -vga cirrus -cdrom /dev/sr0


インストールプロセスはきちんと整理されており、高速でした。 インストールするパッケージを選択するとき（ Workstation



、 Server



およびCustom



グループに収集される）、パーティション分割中、またはインストール開始後に中断する必要はありませんでした。



Red Hat 6には、Xウィンドウシステムを設定する段階的なプロセスを整理するように設計されたxf86config



アプリケーションが含まれていましたが、あまり明確ではありませんが、マウスエミュレーションパラメーターをオンにすることが可能になりましたが、Xは後でこのエラーをX86Config



た。 Xのセットアップはまだ簡単なタスクではありませんでした。



Red Hat 6のデスクトップ環境は、実際にはGNOMEでしたが、ウィンドウマネージャーは初期のEnlightenmentであり、サウンドのメインデーモンはそこから取得されました。 ログオンマネージャーとしてxdm



とgdm



xdm



gdm



提案されました。その結果、通常のユーザーはX自体を起動または停止する許可なしでログインできました。これはマルチユーザーシステムで特に重要です。



ディストリビューションにはいくつかの重要なアプリケーションがなかったことに注意してください。 そのため、 gedit



はまだ存在していませんでした。多機能のオフィスアプリケーションはありませんでした。パッケージマネージャーはありませんでした。 RPMパッケージをインストール、表示、およびアンインストールするためのグラフィカルインターフェイスであるGnoRPM



は、 yum



またはPackageKit



が提供するものに最も近いものでした。 グラフィカルインターフェイスで使用できるテキストエディタはgnotepad+



（もちろん、Emacsなしではできませんでした）。



ただし、一般的に、デスクトップは直感的です。 後のGNOME実装とは異なり、この以前のバージョンでは、アプリケーションメニューとランチャーアイコンを含む画面の下部にパネルがあり、仮想デスクトップを管理するためのツールもありました。 このグラフィック環境で迷子になるような時代の平均的なユーザーを想像することはできません。



Red Hat 6は強力なLinuxプレーヤーでした。 明らかに、これにより、彼女は優れたデスクトップOSになる道を真剣に進めました。



マンドラケ8.0（2001）





Mandrake：Linuxの世界におけるターニングポイント



Mandrake 8.0は2001年にリリースされたため、たとえばApple OS 9.2やWindows MEと比較できます。



インストールを正常に進めるために、かなり古いハードウェアをエミュレーターで使用することにしました。 



  $ qemu-system-i386 \  -M pc-0.10 -m 2048 \  -boot order=ad,menu=on \  -drive file=mandrake8.qcow2 \  -usb -net nic,model=rtl8139 \  -netdev user,id=slirp \  -vga cirrus \  -cdrom mandrake-8.0-i386.iso


Red Hatのインストールプロセスは十分だと思いましたが、Mandrakeのおかげですべてがうまくいきました。 インストーラーは使いやすく、シンプルで高速であり、ユーザーは続行する前に構成を確認できます。 Mandrakeインストーラーがすべて正しく行ったため、 XF86Config



構成ファイルをインポートする必要さえありませんでした。





Mandrake 8.0インストーラー



実際、Mandrakeデスクトップを使用することは、他の同様のシステムにあったものと非常によく似ています。 すべてが似ていることに少し驚いた。 私が当時Mandrake Linuxを試していたなら、私の年齢と私が特に精通したユーザーではないという事実を考慮しても、Mandrake Linuxで仕事をすることができたと確信しました。



当時はサイトからプログラムのインストールファイルをダウンロードすることはまだ一般的な習慣ではなかったため、明確なインターフェイス、適切なドキュメント、優れたパッケージマネージャー。



Fedora 1（2003）





ブルーフェドラ、レッドハット



2003年、新しいFedora Coreディストリビューションがリリースされました。 Red Hatをベースにしており、Red Hat Enterprise Linux（RHEL）が同社の主力製品になった後、デスクトップLinuxの役割が割り当てられました。



古いFedora Core 1ドライブを起動するのに特別なことは必要ありませんでした。 



  $ qemu-system-i386 -M pc \  -m 2048 -boot order=ac,menu=on \  -drive file=fedora1.qcow2 -usb \  -net nic,model='rtl8139' -netdev user \  -vga cirrus -cdrom fedora-1-i386-cd1.iso


Fedora 1のインストールは簡単で馴染みがありました。 これは、FedoraとRed Hatが今後9年間使用した同じインストーラーです。 これは、明確で使いやすいグラフィカルインターフェイスです。





Anaconda GUI



Fedora Coreでの作業経験はRed Hat 6または7とは大きく異なります。GNOMEデスクトップは完成されており、主要なサポートアプリケーションがすべてあります。これらはすべて簡潔で専門的に提示されています。



ここからStart Here



アイコンは、ユーザーを3つの場所に導きます。 これは、 Applications



フォルダー、 Preferences



パネル、および設定- System Settings



です。 赤い帽子のアイコンはアプリケーションメニューを指し、GNOMEパネルではOpenOfficeオフィススイートやMozillaブラウザーなどのアプリケーションを起動できます。



概要：過去と未来



2000年代初期には、LinuxベースのOSがすでにかなり優れており、正しい方向に進んでいることは明らかでした。 デスクトップはこれまでになく見栄えが良く、必要なアプリケーションがすべて揃っていて、インストールは他のオペレーティングシステムよりも簡単で効率的でした。 実際、2000年代以降、ユーザーとオペレーティングシステムとの間に相互作用の方法が確立され、今日までほとんど変化していません。 もちろん、何かが変化して改善され、いくつかの革新が現れました。



以下は、いくつかのLinuxディストリビューションの開発について言うべきことです。



  • OS MandrakeはMandrivaに変わり、その後Mageiaになりました。
  • Fedora Coreディストリビューションは単にFedoraと呼ばれていました。
  • UbuntuはDebian OSから登場し、Linuxを大衆に押し上げました。
  • Valveは、SteamOSをゲーミングプラットフォームの公式基盤にしました。
  • Slackwareは今日まで静かに暮らしています。


世界最大のオープンソースプロジェクトがどのように開発されたかを見る機会は、初心者と、この記事のスクリーンショットを「Linuxの歴史」とは関係なく、自分のコンピューターの伝記との両方に持っている人の両方に役立つと思います。 Linuxは進化し続けているので、過去を調べることで、Linux運動の方向性をよりよく見ることができ、驚くべきことに、この運動の一部になることができます。
