注意すべきこと



プロセスで実行されているコンポーネントの観点からプロセスの優先順位について説明しましたが、コンポーネントレベルではなくプロセスレベルで階層が存在することに注意してください 。 プロセス全体をフォアグラウンドプロセスのカテゴリに変換するコンポーネントは1つだけです（たとえば、フォアグラウンドサービス）。 ほとんどのアプリケーションは1つのプロセスで完全に動作しますが、アプリケーションのライフサイクルが起動ごとに大きく異なる場合、またはアプリケーションに非常に困難なタスクがある場合は、時間がかかりますが、軽量であり、最初のプロセスは2番目のプロセスに依存しません、つまり、2つのプロセスを区別することは理にかなっています。 最初のプロセスで重いタスクを実行し、2番目のプロセスで軽いタスクを実行します。その後、メモリ不足の場合は、重いタスクを中断し、軽量なタスクを続行します。



また、プロセスの優先度はコンポーネントレベルで何が起こるかによって決まることを覚えておくことが重要です 。 つまり、長時間実行する必要のある非常に重要なタスクを別のスレッドで、ただしActivity



下から実行すると、プロセスが突然バックグラウンドになったときに不快な驚きを得ることになります。 利用可能なツール（サービス、フロントエンドサービスなど）を使用して、システムが現在実行していることを確認します。

通常、テストにはできるだけ弱いデバイスを購入することをお勧めしますが、フラッグシップでも、アプリケーションが破壊された場合の動作をテストできます 。 アプリケーションを（そのすべてのプロセスとともに）破棄するには、次の手順を実行します。 



 adb shell am force-stop com.example.packagename


複数のプロセスがある場合、最初に次のコマンドの結果の2番目の列（つまり最初の数字）を見て、必要なプロセスのPIDを見つけることができます。 



 adb shell ps | grep com.example.packagename


そして、このプロセスを叩きます： 



 adb shell kill PID


これは、メモリ制限が発生するかどうかに関係なく、アプリケーションがほとんどのデバイスで正しく動作することを保証する最初のステップになります。
