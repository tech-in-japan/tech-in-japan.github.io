スケジューラApp Engine。



App Engineは、スケジューリングアルゴリズムを使用して、アプリケーションがトラフィックを処理するために必要なインスタンスの数を決定します。 アプリケーションが受信した各リクエストに対して、利用可能なインスタンス（アイドル状態または並列haproリクエストを受け入れるインスタンス）にサービスを提供するか、リクエストを待機キューに送信するか、このリクエストの新しいインスタンスを開始するかを決定します。 使用可能なインスタンス、アプリケーションがリクエストに応答する速度（レイテンシ）、リクエストを処理する前に新しいインスタンスを開始して初期化するのにかかる時間に基づいて決定します。 ほとんどの場合、新しいインスタンスを開始することでリクエストをより速く処理できると考えられる場合、新しいインスタンスを開始します。



もちろん、アプリケーショントラフィックには一貫性がないため、スケジューラはアプリケーションのアイドルインスタンスの数を監視し続けます。 これらのインスタンスは、ユーザーへの顕著な遅延なしにトラフィックの急増に対応するのに役立ちます。 スケジューラーは、アプリケーションのアイドルインスタンスが多すぎると判断した場合、リソースを撤回し、インスタンスの1つを停止します。



使用されるインスタンスの数を減らすための戦略。



レイテンシーの削減。


アプリケーションの遅延は、アプリケーションのサービスに必要なインスタンスの数に大きな影響を与えます。 したがって、遅延を減らすと、必要なインスタンスの数に大きく影響する可能性があります。 待ち時間を短縮する手順のリストは次のとおりです。

  • 頻繁に使用される共有データをさらにキャッシュします。 つまり、Memcacheを使用します。 アプリケーションにcache-controlヘッダーも設定すると、サーバーとブラウザーのキャッシュ効率を大幅に向上させることができます。 数秒間のキャッシングでも、アプリケーションの効率を大幅に向上させることができます。 Pythonアプリケーションもランタイムキャッシュを使用する必要があります 。
  • Memcacheをより効率的に使用します。 一連のシングルの代わりにバッチ呼び出しを使用します。
  • クエリ以外のアクティビティにはタスクを使用します。 アプリケーションがリクエスト提供ユーザーの外部で実行可能なアクションを実行する場合、タスクに送信します。 実行がリクエストを完了するのを待つのではなく、これらのアクションをタスクキューに送信することで、ユーザーの顕著な遅延を大幅に減らすことができます。 タスクキューを使用すると、実行速度を大幅に制御でき、負荷をより均等に分散できます。
  • データウェアハウスをより効率的に使用します。 以下の詳細を掘り下げます。
  • URLフェッチリクエストを並行して送信します。
    • 非同期API呼び出しを使用（ Java 、 Python ）
    • URLフェッチの呼び出しをパッケージに結合し（ユーザーリクエスト中に1つずつ処理できます）、非同期URLフェッチと並行してオフラインタスクで処理します。
  • HTTPセッションをJavaで非同期に記録します。 HTTPセッションでは、 <async-session-persistence enabled="true"/>



    をappengine-web.xmlファイルに追加することにより、アプリケーションがデータストアに非同期でセッションデータを書き込むように設定できます。 セッションデータは常にmemcacheに同期的に書き込まれ、リクエストがmemcacheにないデータを取得しようとすると、リクエストはデータストアに送信されますが、データストアにはまだ最新の更新がない可能性があります。 これは、アプリケーションが期限切れのセッションデータを受信するリスクはほとんどないことを意味しますが、ほとんどのアプリケーションでは、速度の向上はこのリスクをはるかに上回ります。


スケジューラを手動で調整します。


コントロールパネルの[アプリケーションの設定]ページには、スケジューラがアプリケーションインスタンスを管理するために使用するいくつかの変数を設定するのに役立つ2つのスライダーがあります。 以下に、パフォーマンスとリソース使用の妥協点を見つけるためにそれらを使用する方法の簡単な説明を示します。

  • アイドルインスタンスの最大数を減らします。 最大アイドルインスタンスを設定すると、アプリケーションのアイドルインスタンスの最大数を制御できます。 この制限を設定すると、App Engineはこの制限を超えてアイドル状態のインスタンスを停止し、追加のクォータを消費したり、追加のコストが発生したりしないようにします。 ただし、アイドルインスタンスが少ないことは、トラフィックの急増時にApp Engineスケジューラが新しいインスタンスを開始する必要があることを意味します。これにより、ユーザーに気付く待ち時間が長くなる可能性があります。
  • 最小待機遅延を増やします。 Min Pending Latencyを増やすと、指定した時間を超えてリクエストがキューに入れられるまで、新しいインスタンスを開始しないようApp Engineスケジューラーに指示します。 すべてのインスタンスがビジーの場合、ユーザーにサービスを提供するリクエストは、このしきい値に達するまでキューで待機する必要があります。 このパラメーターに大きな値を設定すると、実行するインスタンスの数が少なくなりますが、負荷の増加中にユーザーに大幅な遅延が発生する可能性があります。


Javaで同時要求を有効にします。


リリース1.4.3では 、アプリケーションのインスタンスがJavaで複数の同時リクエストを処理する機能を導入しました。 このオプションを有効にすると、必要なインスタンスの数が減りますが、アプリケーションが正しく機能するには、アプリケーションがスレッドセーフである必要があります。 Javaのドキュメントで同時クエリの詳細を読むことができます 。



注 ：Pythonのマルチスレッドは、作業計画で利用可能なPython 2.7のリリースまで利用できません。 Python 2.7では、マルチスレッドインスタンスはより多くのリクエストを処理でき、API応答のブロックを待機しているときにアイドル状態の場合、インスタンス時間のクォータを消費しません。 Pythonは現在、一度に1つのインスタンスに対して複数のリクエストを処理する機能をサポートしておらず、すべての開発者が同時リクエストに適応できるようにするため、2011年11月20日までフロントエンドインスタンスの稼働時間を50％割引します。Python2.7は終了しましたテスト。



アプリケーションストレージ管理。

    、作業中のアプリケーションインデックスを確認できます。
  • データモデルを設計するときは、インデックスの総数を減らすクエリを作成できるかどうかを確認してください。 App Engineがインデックスを作成する方法の詳細については、 クエリとインデックスのドキュメントをご覧ください。
