それで、 TypeScript自体は正確に何ですか -おそらく誰にも秘密ではありません。 しかし、それでも、これはMicrosoftによるJavaScriptへの静的型付けの試みです。 コードの例とそれによって解決できるタスクの例は、 公式Webサイトまたはここのハブにあります。 ハブには同様のTypeScriptの記事が既にあります：一般的な印象 、したがって、繰り返さないために、私は私の個人的な経験に基づいて、言語で作業することの長所と短所を強調することにしました。 言語の長所と短所を思い出してリストするのは簡単ではありませんでした。

  • 多くの一般的なIDEのサポート：

    • Visual StudioのTypeScript
    • ビジュアルスタジオコード
    • ウェブストーム
    • 崇高なテキスト
    • 日食


  • TypeScriptは厳密に型指定され（ オプション！ ）、JavaScript言語でコンパイルされています。 プログラマーはJavaとC＃を学ぶのが簡単です。
  • TypeScriptは、継承、ポリモーフィズム、カプセル化、アクセス修飾子など、多くのOOPの概念を実装しています。 クラス、インターフェース、および（偶数！）抽象クラスがあります。
  • この言語の可能性により、標準のJavaScriptよりも将来の開発とテストが容易な複雑な統合ソリューションをより速く簡単に書くことができます。
  • TypeScriptはJavaScriptのスーパーセットであるため、JavaScriptコードはすべてTypeScriptでも実行されます。 私の意見では、これは競合他社に対する主な利点です。たとえば、GoogleのDartはJavascriptから根本的に再設計された言語です。 JavascriptプロジェクトのTypeScriptへの翻訳に関する記事があります




短所



ほとんどの場合、TypeScriptは賞賛されているように思えます。 したがって、Typescriptの何が問題なのかについての投稿を書きたいと思いました。 結局のところ、短所を見つけるのは簡単ではありませんでした。



  • 開発のプロセスでは、* .ts、* .d.ts、* .map、* .jsファイルを扱っています。 追加ファイルが多すぎるため、プロジェクトが小さい場合は不便です。
  • すべてのブラウザが、不要な設定なしにコンソールでTypeScriptデバッグをサポートしているわけではありません。
  • 多くの自明でないクラス。 クラスに基づいてコードを記述するには、どのプロパティがどこにあるかを覚えておく必要があります。 たとえば、1つのイベントクラスの代わりに、MouseEvent、TouchEvent、KeyboardEventなどがまだあります...
  • 暗黙的な静的型付け。 タイプはいつでもanyとして記述できます。実際、この変数の特定のタイプへのキャストは無効になります。
  • これはトランスポーターであり、常に手元にTSCが必要であることを意味します
  • d.ts宣言はDefinitelyTypedコミュニティによってサポートされており、多くの場合、ライブラリの現在のバージョンと一致しません。 または、複雑なバリアント（汎用関数、複数のタイプの戻り値）を考慮しません




静的型付けについて少し



すべての開発者の生活の中で、彼は喜びのためにコードを書くときがあります。 それがホームプロジェクトであろうと、あるいはゼロから書き始めたばかりのプロジェクトのチームワークであろうと。 これは、同僚とのコードの競合について多くのことを考え、エラーを探す必要がない素晴らしい瞬間の1つです。 しかし、プロジェクトは成長しており、動的に型付けされた言語（JavaScript）でコードを記述すると、型を含む新しい機能とバグが生い茂ります。



この場合の解決策は何ですか？ テストを書く！



しかし、コンパイルの段階でマシンが何をうまく処理できるかを確認する必要があることを認めなければなりません。 静的型付けを使用する言語は、型エラーに関連する追加のテストを不必要に記述することを防ぎます。 したがって、タイプを無視しない場合、TypeScriptにはJavaScriptよりも大きな利点があります。



簡単な例を見てみましょう。



2つの数値を追加できる関数があります。 



function sum(a, b) { return a + b; }




顧客は、ユーザーが追加された番号を入力できるWebフォームを実装することを提案しました。 



 var result = sum( document.getElementById("input1").value, document.getElementById("input2").value, ) document.getElementById("result").innerText = result;




入力フィールドに2と3の2つの数値を書き込み、関数の動作を確認します。 



 23




結果はまったく予想外でした。 問題は、html inputタグのvalueフィールドが「string」タイプの結果を返し、JavaScriptがこれらの数字を追加する代わりに2つのストリング「1」と「2」を接着することです。



もちろん、この例は非常に単純ですが、実際のエラーはより複雑になる可能性があり、開発段階でエラーに気付くことは非常に困難です。



この問題の解決策は、TypeScriptを使用して非常に簡単に解決できます。 



 function sum(a: number, b: number): number { return a + b; } sum(2, 3); // Ok sum(2, "3"); // Error!




すでにコンパイル段階で、エラーを検出することができました。 エラーを検索して追加のテストを作成するよりも、開発にエネルギーを費やす方が良いでしょう。



おわりに



TypeScript開発の長所と短所を考えたとき、短所を強調するのは簡単ではありませんでした。 TypeScriptは、大規模プロジェクトで正当化される以上のものです。 これは、メソッドとクラスに加えて、それらの宣言を記述する必要があるという事実のために、JavaScriptでの開発より時間がかかるという事実によるものです。 それにもかかわらず、JavaScriptに静​​的型付けがない限り、TypeScriptは優れた代替手段です。
