LinuxおよびPostgreSQLを使用して1C Freshをゼロからインストールする

写真付きのこの記事では、ボタンがPostgreSQLおよびLinuxで独自の新しいブラックジャックを展開する方法について説明します。 1C Freshをインストールしようとしており、公式ドキュメントの量に少しthoseれている人に役立つことを願っています。 展開するとき、フリーソフトウェアを最大限に活用します。







1C Freshの構成



1C Freshは、ソリューションを公開するためのテクノロジーです。 つまり、1Cに基づいたさまざまなソリューションへのアクセスを一般ユーザーに公開し、これらのソリューションを共通の管理および保守で単一のシステムに接続できるようにするサービスです。



中心的な要素は、残りの要素をリンクするコントロール構成であるサービスマネージャーです。 サービスのコンポーネント、構成バージョン、ユーザー、データ領域に関するすべての情報を保存し、さまざまな操作を実行するコマンドを提供します。



Service AgentはService Managerの右手であり、アプリケーション構成を処理するサービス構成です。 サービスエージェントの一般的なタスクは次のとおりです。



  • アプリケーションデータベースの構成を更新します。
  • アンロード（ダンプ）を現在のバージョンのアプリケーション構成に更新します。
  • 構成の新しいバージョンの初期データを準備します。
  • アプリケーションデータベースから統計とエラーを収集します。


サービスエージェントは、コンフィギュレータを使用してこれらのタスクを実行します。 最近まで、これはWindowsプラットフォームを必要とする唯一のサービス要素でした。 しかし、1C Fresh 1.0.19テクノロジーの新しいバージョンでは、開発者がこの制限を削除し、Service AgentはLinuxでも動作できるようになりました。



インフォベースは、Service Managerを介して公開されるアプリケーション構成です。 現在サービスモード操作をサポートしている構成：

独自の決定を公開する機会があります。ITSには改善のための推奨事項があります。



これらの要素は基本的なものであり、展開に必要です。 追加機能を提供する他のものもありますが、それらは必要ではありません：



  • アプリケーションゲートウェイ-サービスの内部構造を「隠す」ことができるサービス。 要求をプロキシし、情報セキュリティにリダイレクトします。 すべてのアドレスは同じように見えます。
  • サービスサイト-サービスと対話し、アプリケーションを起動し、アプリケーションへのアクセスを提供するためのシンプルで便利なユーザーインターフェイスを提供します。
  • Availability Manager-サービスリソースが利用できないことに関する情報を保存し、サービスの他のすべてのコンポーネントが機能しない場合でも、サイトおよびフォーラムのユーザーがアクセスできないことに関するメッセージを表示できます。


サービスの他のコンポーネントがあります。 サイトでそれらについて読むことができます

初期機械準備



データベースがあります：fresh-db.knopka.int



データベースサーバーとしてPostgreSQLを使用します。 更新の便宜上、PostgresProリポジトリを接続します。 1Cはディストリビューションも提供するため、安全に使用できます。 



sh -c 'echo "deb http://1c.postgrespro.ru/deb/ $(lsb_release -cs) main" > /etc/apt/sources.list.d/postgrespro-1c.list' wget --no-verbose -O - http://1c.postgrespro.ru/keys/GPG-KEY-POSTGRESPRO-1C | apt-key add - && apt-get update apt-get install postgresql-pro-1c-9.6


データベースごとに、PostgreSQLの個別のインスタンスを提供するクラスターを作成します。 これにより、各データベースを管理し、レプリケーションを実行できます。 



 pg_createcluster 9.6 <clustername> -d <dir> systemctl daemon-reload


< clustername > -クラスターの名前。クラスターで同様にサービスされるデータベースを呼び出します。

< dir > -クラスターが作成されるディレクトリ。



作成中に明示的に指定されない限り、次の空きポートは5432以降に使用されます。 次のようにクラスターのリストを確認できます。 



 pg_lsclusters

リモート管理サーバー（ras）を起動するためのユニットを作成しましょう。 内容を含むファイル/etc/systemd/system/ras.serviceを作成します。 



 [Unit] Description=RAS After=syslog.target After=network.target [Service] Type=forking WorkingDirectory=/opt/1C/v8.3/x86_64 User=usr1cv8 Group=grp1cv8 OOMScoreAdjust=-100 ExecStart=/opt/1C/v8.3/x86_64/ras cluster --daemon -p 1545 ExecReload=/bin/kill -s HUP $MAINPID ExecStop=/bin/kill -s QUIT $MAINPID TimeoutSec=300 Restart=always [Install] WantedBy=multi-user.target




ユニットをアクティブにし、サービスを開始します。 



 systemctl daemon-reload systemctl enable ras systemctl start ras

 @echo off rem %1 -    1: set SrvUserName=< > set SrvUserPwd=< > set CtrlPort=1540 set AgentName=localhost set RASPort=1454 set SrvcName="1C:Enterprise 8.3 Remote Server" set BinPath="\"C:\Program Files\1cv8\%1\bin\ras.exe\" cluster --service --port=%RASPort% %AgentName%:%CtrlPort%" set Description="  1: 8.3" sc stop %SrvcName% sc delete %SrvcName% sc create %SrvcName% binPath= %BinPath% start= auto obj= %SrvUserName% password= %SrvUserPwd% displayname= %Description%


ユーザー名とパスワードを、サービスの起動元のユーザー名とパスワードに変更します。 Descriptionがロシア語で作成される場合、目的のエンコーディングでファイルを作成することを忘れないでください。 プラットフォームバージョンを渡すパラメーターとしてファイルを実行します。

アプリケーション情報ベースでは、データ領域を作成します。 これを行うには、インフォベースウィンドウのメインセクションで[詳細]タブに移動し、[新しい領域を追加]チェックボックスをオンにして、[初期データで領域を塗りつぶす]のチェックを外します。 初期データを準備しなかったため、単に存在しません。

サービスユーザーとサブスクライバーユーザー



サービスモデルでアプリケーションアプリケーションを使用できるようにするには、サービスのユーザーを登録して、自分のデータ領域で作業する権利を与える必要があります。 サービスとサブスクライバーの関係の仕組みは次のとおりです。ユーザーはサービスに登録され、サブスクライバーが作成されます。サービスのユーザーでもあるこのサブスクライバーのユーザーはサブスクライバーに追加されます。



ユーザー登録には、アクティベーションコードを送信するためのメールアカウントの設定が必要です。 サービスマネージャーの[管理]→[メールアカウント]セクションで設定できます。 各設定には独自の設定があり、すべてが直感的に明確であるため、これについては詳しく説明しません。あなたはそれなしで行うことができ、別の方法でアクティベーションコードを取得します。それについては以下で説明します。



パスワードなしでユーザーAnonymousに代わってサービスマネージャーにアクセスし、ログインに成功した後、[登録]をクリックして、サービスの新しいユーザーにデータを入力します。メールアドレスは有効でなければなりません：







「登録」ボタンをクリックすると、作成されたユーザーをアクティブにするかどうかを尋ねるウィンドウが表示されます。



次に、登録時に指定した電子メールに送信されたアクティベーションコードを入力する必要がありますが、「メンテナンス」→「登録のリクエスト」セクションで取得します。リクエストを開き、そこからフォーム0414fb84-cf59-11e7-f094-00155d003608からリクエストコードをコピーします。これが必要なアクティベーションコードです。



サービスユーザーを作成した後、サービスマネージャーはサブスクライバーも自動的に作成し、このサブスクライバーの所有者権限をユーザーに追加します。彼はアプリケーションを管理し、新しいユーザーをサブスクライバーのアプリケーションで一緒に作業するように招待できます。

「アプリケーションの入力」ボタンをクリックすると、ブラウザはアドレスfapache.knopka.int/ext/ea_01/1を開きます。最後の単位はエリアの番号です。ユーザーには権限がなく、他のエリアに入ることはできません。

おわりに



これがすべての機能ではありません。サービスで計画された構成の更新、データの同期を構成できます。アプリケーションゲートウェイを追加すると、サービスインスタンスのスケーラビリティが確保され、新しい情報ベースが追加され、同じ情報ベースの単一の外部アドレスが維持されます。サービスのサイトおよびフォーラム、アクセス不能性の監視などは、1C Freshサービスと一緒に問題なく動作させることができます。これにより、ニーズに応じて機能を柔軟に選択できます。
