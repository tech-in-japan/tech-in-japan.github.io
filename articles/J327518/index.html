チームについて話しましょう

WrikeのスタッフがMicrosoft Teamsをサービスにどのように統合したか、そしてそれが何につながったかについての小さな興味深い話と、統合プロセス自体に関する大きな技術的部分。 ちなみに、彼らはチャットがリリースされた日にこれを行う世界で最初の1つでした。







1 + 1 = 3



Wrikeでの経験では、潜在的な購入者はプロジェクト管理システムを選択するときに、手頃な価格の統合にますます関心を持っています。 実際、快適なボーナスから、製品を既存のエコシステムに適合させることが、クライアントの選択を決定する要因に変わりました。



もちろん、開発者が企業ユーザー向けの最も一般的なツールに集中することは重要であり、Microsoft製品は必然的に優先事項です。 ただし、統合のために統合を行わないことがさらに重要です。 1 + 1は、多数のソリューションがクライアントに付加価値をもたらすという意味で、3を与える必要があります。



したがって、Azure Active Directory、OneDrive、およびOffice 365との統合を一貫してサービスに追加しましたこれらの手順の最終目標は、ユーザーがアプリケーションを数分ごとに切り替えたり、ウィンドウ間で情報をコピーしたり、個人間で情報をコピーしたりしないシームレスなワークスペースを作成することですチームワークのためのツールとサービスは深byではありません。



MS Teamsは論理的な新しいステップでした。 500人以上の従業員を抱える企業の53％が企業のメッセンジャーを使用しています（ SpiceWorksデータ）。 私たちは、作業を議論しているユーザーが議論の結果をすぐに実施することが便利だと考えました。 したがって、チームはマイクロソフトと協力して、そのような状況で最も適切なサービス機能を選択し、チームインターフェイスでの有機的な実装を通して考えました。



その結果、Wrikeユーザーはメッセンジャーからプロジェクトとタスクを直接作成できるようになり、プロジェクトを新しいタブとしてチームに追加できるようになりました。 統合では、ディスカッションと並行して、タスクへの変更（ステータス、実行者、期限、コメント）の導入もサポートしています。







メッセンジャー内のタイムラインでガントチャートを表示できるため、プロジェクトの議論はより実質的で正確になり、議論の参加者の前にいます。







そしてもちろん、明らかな機会が追加されました-タスクに関する通知は、電子メールだけでなく、Teamsのメッセージでも受信できるようになりました。



この機能の開発には2か月かかりました。 この間、チームはあなたに役立つかもしれない興味深い経験を得ました。



技術面について



Microsoft Teamsのタブアプリケーションは、次のもので構成されます。



  • manifest.jsonファイルとリソースファイルを含むマニフェストを含むアーカイブ。
  • ページの要求を処理する外部HTTPサービス：構成、タブのメインページ、削除。


すべてのページは、Microsoft Teamsのiframe内にレンダリングされます。 タブアプリケーションのマニフェストの例は、 MSDNで説明されています 。 マニフェストスキームもあります。 執筆時点では、回路の現在のバージョンは0.4でした。



マニフェストの主なフィールド：



  • Id



    -GUID、異なるアプリケーションまたは異なる環境に対して一意である必要があります。
  • Icons



    -解像度が44x44および88x88で、それぞれ1.5 Kb以下のアイコン。
  • configUrl



    は、ユーザーに最初に表示されるURLです。 アプリケーションの構成が開始されます。
  • canUpdateConfig



    ユーザーが既に追加されたアプリケーションの構成を変更できるかどうかのサイン。
  • needsIdentity



    -Teams JS APIを通じてユーザーに関する情報を取得できます。
  • validDomains



    ドメインが存在する信頼できるドメインの配列。 ワイルドカードが許可されます。


マニフェストとリソースを準備した後、結果のアーカイブをMicrosoft Teamsにアップロードできます。



Microsoft Teamsインフラストラクチャとの相互作用は、 Microsoft Teams JS APIライブラリによって提供されます 。 ライブラリでは次のことができます。



  • Azure Oauth2またはその他のOauth2プロバイダーを介して承認のフローを実行します。
  • Azure Active Directoryに関する最新のユーザー情報、チャネルID、チーム、およびグループを取得する
  • タブ設定の受信と保存
  • イベントのハンドラーを設定します。設定の保存、テーマの変更、タブの削除。
  • タブで選択したオブジェクトへのリンクを生成してクリップボードにコピーします。


ライブラリを使用するには、ライブラリを接続し、アプリケーションの各ページで初期化する必要があります。 



 <script src="https://statics.teams.microsoft.com/sdk/v0.4/js/MicrosoftTeams.min.js"></script> ... <script type="text/javascript"> microsoftTeams.initialize(); </script>


マルチページ構成とAzure ADおよびWrikeアカウントのリンクを備えたサンプルMsTeamsアプリケーションを検討してください。



いくつかのiframe（設定、タブ）およびOauth2リダイレクトを使用したアプリケーションの詳細には、設定と、以前に追加されたタブでのアプリケーションの表示の両方について、複雑ではあるが典型的なフローが必要です。



主なプロセス：



  1. ユーザーは、Microsoft Teamsに新しいタブを追加します。
  2. 権限のないユーザーは、以前に追加されたタブに移動します。
  3. 許可されたユーザーが以前に追加されたタブにアクセスします。


タブを追加するプロセスを次の図に示します。 青色はMicrosoft Teamsページ、緑色-アプリケーションのページ、白色-リクエストを処理するためのコードとアクションを示します。







ようこそページのページには、アプリケーションに関する基本情報と、Azure ADでの承認のフローを開始する[接続]ボタンが表示されます。



コールバックURLハンドラーは以下を行う必要があります。



  • Azure ADでユーザー情報を要求します（たとえば、 Microsoft Azure AD Graph APIを使用 ）。
  • 受信した情報をどう処理するかを決定します。システムに新しいアカウントを作成するか、既存のアカウントを使用します。
  • たとえば、セッションCookieを発行して、システムにログインします。
  • 認証が成功した場合、認証は新しいウィンドウで開始され、コールバックは認証のエンドポイントであるため、コールバックURLページは単純に閉じる必要があります。
  • 承認に失敗した場合、コールバックURLページには、ユーザーに失敗の理由、たとえば、ユーザーによるアクセスの拒否、Azureアプリケーションへのアクセスの欠如などが表示されます。


設定ページは設定ページです。 タブを追加するとき、および以前に追加したタブのメニューの[構成]オプションを使用して要求されます（マニフェストでcanUpdateConfig



フラグが設定されている場合）。



Azure ADのマルチテナントアプリケーションを介してOauth2承認を実行するサンプルコード（つまり、ようこそページのコード例）は次のようになります。 



 <script src="https://statics.teams.microsoft.com/sdk/v0.4/js/MicrosoftTeams.min.js"></script> <input type="button" onclick="startAuth()" value="..."/> <script type="text/javascript"> microsoftTeams.initialize(); var startAuth = function() { microsoftTeams.authentication.authenticate({ url: 'https://login.windows.net/common/oauth2/authorize' + '?response_type=code%20id_token' + '&scope=openid' + '&response_mode=' + form_post + '&client_id=' + <YOUR_AZURE_APP_CLIENT_ID> + '&resource=https%3a%2f%2fgraph.windows.net' + '&nonce=' + <MS_NONCE> + '&state=' + <OAUTH2_STATE> + '&redirect_uri=' + encodeURIComponent('http://localhost/callback'), width: 700, height: 500, successCallback: onAuthSuccess, failureCallback: onAuthFailure }); }; var onAuthSuccess = function() { console.log('Auth success'); document.location.href='http://localhost/setup'; }; var onAuthFailure = function() { console.log('Auth failed'); microsoftTeams.settings.setValidityState(false); }; </script>


パラメータ：



  • redirect_uri



    -GETリクエストidTokenが渡されるコールバックハンドラのURL。 Azure ADのアカウントをシステム内のアカウントに接続するためのアクセストークン、リフレッシュトークン、およびユーザー情報を取得できます。
  • YOUR_AZURE_APP_CLIENT_ID



    アプリケーション識別子。
  • MS_NONCE



    はランダムに生成されたナンスであり、Azure Oauth2の状態に類似しています。
  • OAUTH2_STATE



    -Oauth2状態。コールバックハンドラーで確認する必要があります。

以前に追加されたタブへのユーザーログインのプロセスを検討します。 ここでは次のオプションが可能です。
