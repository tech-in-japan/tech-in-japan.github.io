phpDaemon-非同期アプリケーションフレームワーク

今日は、 phpDaemonについて説明します。これは、I / O（libevent）処理とデーモンに固有の他の低レベルタスクを処理する非同期モジュラーデーモンフレームワークです。 その助けにより、ブラックジャックと売春婦を使用して適切なネットワークアプリケーションを簡単に作成できます。

すぐに使用できるFastCGI、HTTP、CGI、FlashPolicy、Telnet、 WebSocket （！） サーバー -はい、同じ魔法のペンダルが Googleの新しいプロトコルです。 そして、mysql、memcached、mongodbクライアント...そしてさらに、猫の下の完全なリスト。 ネットワーキングは本当に簡単です。 中年のプログラマは、たとえば数時間でIRCボットを作成できます。

実例として、 このチャットをphpDaemon + WebSocket + MongoDB + jQueryに実装しました。 これは、このテクノロジーの利点を示しています。メッセージ配信は瞬時に行われ、データ交換中のオーバーヘッドは最小限に抑えられ、高性能であり、アプリケーションは水平方向に拡張されます。 このチャットのソース （現在は17 kb）。 チャットは、Chrome、FF、IE6 +、Iron、Safariでテストされ、動作していることに注意してください。



どこに申請するのが良いですか？



phpDaemonの範囲は、Web開発とそれ以降の両方で非常に広いです。 その助けを借りて、リアルタイムのマルチプレイヤーゲーム（たとえば、Flash）、インスタントインタラクションを備えたサービス、チャット、IMゲートを作成すると便利です。

本番環境でのプロジェクトの最初の実際のアプリケーションは、個人メッセージの即時配信です。サイトのページには、数キロバイトの目に見えないswfファイル（Flash）があり、サーバーに接続してコマンドを待ちました。 ユーザーがメッセージを受信するとすぐに、Flashはコマンドをページ（Javascript）に送信し、すぐにポップアップアイコンを表示して「クリック」しました。 同僚と私は反応の速さに驚いていました。送信者がマウスの左ボタンから指を離す前に、隣のコンピューターから音が聞こえました。

さらに最近では、サーバーは、アスタリスク（IPテレフォニー用）にAPIを提供する順序で実装され、負荷を完全に保持しています。



アーキテクチャと機能



phpDaemonのアプリケーションには処理ロジックのみが含まれ、すべての低レベルの呼び出しは自動的に発生します。 このプロジェクトは、もっぱらPHPで書かれており、おそらく既にパフォーマンスについて疑問に思っているでしょう。 I / Oは、memcachedやその他の有名なプロジェクトで使用される、確立されたライブラリであるlibeventを通じて提供されます。 同時に、PHPで命令を実行する速度は非常に速く、このグループの他の多くの言語よりも優れています。 通常の同期スクリプトでは、環境の起動とI / Oのロックに時間の大部分が費やされます。たとえば、データベースを照会する場合、通常のスクリプトは応答が受信されるまで実行を継続しません。 phpDaemonにはそのようなロックはありません。リクエストを送信し、必要に応じてコールバック関数をハングアップし、ビジネスに取りかかりました。



phpDaemonは1つのマスタープロセスと多くのワークフローであり、それぞれがアプリケーションランタイムです。 アプリケーションはAppInstanceの後継クラスであり、さまざまなイベントへの反応を記述します。 ワークフローを実行するとき、アプリケーションクラスをインスタンス化し、init（）メソッドを実行します。 init（）メソッドでは、アプリケーションはソケットをバインドしたり、どこかに接続したり、ローカル記述子を開いたりできます（一方は他方を除外しません）。 アプリケーションは、必要なパラメーターを宣言することにより、ワークフロー内の他のアプリケーションと対話することもできます。たとえば、WebSocketServerにルートを登録します。

ソケットをブリッジした後、適切な接続が到着すると、onAcceptedイベント（connId、addr）が発生します。 WebSocketServerのこのメソッドの例：

public function onAccepted($connId,$addr)

{

$ this ->sessions[$connId] = new WebSocketSession($connId,$ this );

$ this ->sessions[$connId]->clientAddr = $addr;

}

ポート843でクロスドメインポリシーを発行するサーバーの重量は1.67 kb- FlashPolicy.phpのみです 。 一方、多くのアナログに勝る明らかな利点は明らかです。1ダースまたは2つのtelnetセッションの後にオープンにすることはできません。 多くの一般的なflashpolicyd DoS実装は、何も送信しないセッションを1つでも開くことです。これらは同期的に処理されており、プロセスは海の天候を待っているためです。



HTTP要求（FastCGIを含む）を処理するために、別個のエンティティ（要求継承クラス）があります。 このエンティティには、入力パラメータ（get、post、cookie、server ...）と状態（実行/スリープ/デッド）があります。 要求はキューに入れられ、ディスパッチャはそれらを順番に呼び出します。 要求を中断する必要がない場合は、すべての作業を実行して完了コードを返すことができるため、ディスパッチャはそれをキューから削除します。 要求が待機を必要とする操作を実行する場合（たとえば、MongoDBへの要求を行う場合）、$ this-> sleep（30）を実行して、最大30秒間スリープ状態にする必要があります。 また、MongoDBリクエストのコールバック関数では、$ request-> wakeup（）を指定するだけで、すぐに夢を追い出します。 その後、ディスパッチャはすぐに彼に連絡します。 要求は、MongoDBから応答を受け取り、実行を継続できます。 何らかの理由で回答が受信されない場合、「申し訳ありませんが、しばらくしてからもう一度お試しください」というメッセージが表示される場合があります。 run（）メソッドでリクエストを完了するには、1を返すか、$ this-> terminate（）を呼び出します。

リクエストヘッダーは、アプリケーションサーバー（ボックス化-FastCGI、HTTP）で受け入れられた後、appResolverに進みます。これは、アプリケーションが誰にリクエストを受信するかを決定し、アプリケーションでbeginRequestメソッドを呼び出します。 次に、彼はRequestクラスのインスタンスを作成して返します。 次に、要求オブジェクトがキューに入れられます。 オプションで、アプリケーションはpushRequestメソッドを使用して独自の「リクエスト」をキューに追加できます。これは、特定の間隔で特定のコードを呼び出すために必要です（たとえば、これはMongoNodeで行われ、カーソルをポーリングします）。

リクエストはPOSTデータ（およびマルチパート）、アップロードなどを完全にサポートしています（ 例ページ ）。さらに、リクエストのUCS-2エンコーディングを手動で処理する必要がなくなりました（％uFFFF）-これは自動的に行われます。

X-Sendfile（ファイルへのWebサーバー要求への応答の記録）およびRequest-Body-File（ファイルからの要求の本文部分の読み取り）をサポートします。 ボディが完全に受け入れられる前に要求の実行を開始でき、アップロード処理をプログラムできます（たとえば、ディスク上の一時ファイルではなく、memcachedにすぐにスローします）。

イスカロプキ



現在、次のモジュールがディストリビューションに含まれています。

  • FastCGI-FastCGIプロトコルを使用してアプリケーションに接続できます。
  • HTTP-HTTP経由でアプリケーションに接続できます。
  • CGI-FastCGI、HTTP、およびその他のトランスポートを使用して、通常のCGIアプリケーションを実行できます。
  • Flashpolicy-ポート843でFlashのクロスドメインポリシーを配布します。
  • WebSocketServer- WebSocketセッションを提供します。
  • MongoClient-MongoDBドライバー。
  • MySQLClient-MySQLドライバー（SphinxQLに接続することもできます）。
  • MySQLProxyはプロキシMySQLサーバーです。
  • MongoNode-MongoDBスレーブノードの実装により、データベース内のオブジェクトの変更時にイベントをハングさせることができます。
  • MemcacheClient-Memcacheドライバー。
  • LockServerは分散ロックサービスです。
  • LockClientは、独自のロックサービスのクライアントです。
  • TelnetHoneypotは、最も単純なtelnetサーバーです。
  • RTEPServer / RTEPClient-クライアントがイベントにサブスクライブしてそれらをアナウンスできるようにするリアルタイムイベントプロトコルの実装。
  • BitTorrentTracker-MongoDBを使用したBitTorrentトラッカー。
  • DebugConsoleは、実行中のプロセスでコードを実行できるa-la telnet対話型デバッグコンソールサーバーです。
また、プロセスと記述子（asyncProcess、asyncStream）を使用した非同期作業用のツールも提供されています。

さらに、サンプルアプリケーションが含まれています。
