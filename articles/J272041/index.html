私たちのプラットフォームを含め、何百万人ものオーディエンスを持つ大規模なオンラインストアはほとんどありません。 ただし、「1C-Bitrix：サイト管理」を使用する店舗の総数は非常に多く、これらを合わせてさまざまな市場セグメントの印象的なオーディエンスをカバーしています。



その結果、プロジェクトのフレームワーク内で内部スタートアップを組織しました。 どちらから取得するかわからなかったため、小さな問題を解決することから始めることにしました。データの収集と保存の方法です。 この小さなプロトタイプは30〜40分で作成されました。







MVPという用語があります-最小限の実行可能な製品、最小限の機能を備えた製品です。 テクニカルメトリックの収集、訪問者からのページ読み込み速度の収集を開始し、プロジェクトの速度に関する分析をユーザーに提供することにしました。 これは、パーソナライゼーションまたはBigDataとは関係ありませんでしたが、すべての訪問者のオーディエンス全体を処理する方法を学ぶことができました。



JavaScriptには、Navigation Timing APIと呼ばれるツールがあります。このツールを使用すると、ページ速度、DNS解決期間、ネットワークデータ転送、バックエンドでの作業、ページレンダリングに関するクライアント側のデータを収集できます。 これらすべてをさまざまなメトリックに分解してから、分析を提供できます。







プラットフォームで作業している店舗の数、収集する必要があるデータの量を把握しました。 潜在的に、これらは1万件のサイト、1日あたり数千万件のヒット、1秒あたり1000-1500件のデータ書き込み要求です。 多くの情報があり、それをどこに保存し、それから作業しますか？ そして、ユーザーに最高速度の分析サービスを提供する方法は？ つまり、JSカウンターは非常に迅速に回答を提供する義務があるだけでなく、ページの読み込み速度を低下させることもありません。



データの記録と保存



当社の製品は主にPHPとMySQLで構築されています。 最初の要望は、すべての統計をMySQLまたは他のリレーショナルデータベースに単純に保存することでした。 しかし、テストや実験がなくても、これは行き止まりであることがわかりました。 ある時点で、記録時またはデータのフェッチ時に十分なパフォーマンスが得られません。 また、このベースの側面に障害が発生すると、サービスの動作が非常に遅くなるか、通常は動作しなくなるという事実につながります。







さまざまなNoSQLソリューションを検討しました。 Amazonには大規模なインフラストラクチャが展開されているため、最初にDynamoDBに注目しました。 この製品には、リレーショナルデータベースと比較して多くの利点と欠点があります。 DynamoDBの記録およびスケーリングがより良く、より速く動作する場合、複雑な選択を行うことははるかに困難になります。 データの一貫性にも問題があります。 はい、提供されていますが、常にいくつかのデータを選択する必要がある場合、常に最も関連性の高いものを選択するという事実ではありません。



その結果、生データのストレージとしてではなく、ユーザーへの情報の集約とその後の配信にDynamoDBの使用を開始しました。



行ではなく列で機能するカラムナーデータベースを検討しました。 しかし、録音中のパフォーマンスが低いため、それらを拒否する必要がありました。







適切なソリューションを選択して、テキストログの書き込みから始まり、ZeroMQ、Rabbit MQキューなどのサービスで終わるさまざまなアプローチについて説明しました。 しかし、最終的に、彼らは完全に異なるオプションを選択しました。



キネシス



そのときまでに、AmazonはKinesisサービスを開発しました。これは初期データ収集に最適なサービスでした。 これは、必要なものをすべて書き込むことができる、一種の大きな高性能バッファーです。 彼は非常に迅速にデータを受信し、成功した録音について報告します。 その後、バックグラウンドで、選択、フィルター、集計などの情報を安全に操作できます。



Amazonが提供するデータから判断すると、Kinesisはワークロードに簡単に対処できたはずです。 しかし、いくつかの質問が残った。 たとえば、エンドユーザー（サイト訪問者）はKinesisに直接データを書き込むことができませんでした。 サービスを使用するには、Amazon Web Services vの比較的高度な認証メカニズムを使用してリクエストに「署名」する必要があります。 4.したがって、フロントエンドにKinesisにデータを送信させる方法を決定する必要がありました。



次のオプションを検討しました。



  • Kinesisにプロキシされる純粋な構成をnginxに書き込みます。 うまくいきませんでした、論理が複雑すぎます。
  • フロントエンド+ PHPバックエンドとしてのNginx。 多くのリクエストがあるとバックエンドが遅かれ早かれ対処を停止するため、リソースの面で困難で高価であることが判明しました。水平方向にスケーリングする必要があります。 そして、プロジェクトが成功するかどうかはまだわかりません。
  • Nginx + C / C ++のモジュール。 開発の面で長くて難しい。
  • Nginx + ngx_http_perl_module。 要求をブロックするモジュールのバリアント。 つまり、このスレッドに入ってきたリクエストは、他のリクエストの処理をブロックします。 バックエンドの使用と同じ欠点があります。 さらに、nginxのドキュメントには、「モジュールは実験的であるため、何でも可能です。」と直接記載されています。
  • Nginx + ngx_lua。 当時、私たちはまだLuaに遭遇していませんでしたが、このモジュールは好奇心が強いようでした。 必要なコードは、JavaScriptにいくらか似た言語でnginx configに直接書き込まれるか、別のファイルで取り出されます。 したがって、必要な最も奇妙で異常なロジックを実装できます。


その結果、Luaに賭けることにしました。

推奨システムの仕組みに関するいくつかの言葉。



訪問者が特定の製品を推奨できる主要なメカニズムは、協調フィルタリングメカニズムです。 いくつかのアルゴリズムがあります。 最も単純なのは、ユーザーとユーザーのマッチングです。 2人のユーザーのプロファイルを比較し、最初のユーザーのアクションに基づいて、次の瞬間に最初のユーザーが注文したのと同じ製品が必要になる類似のアクションを実行する他のユーザーを予測できます。 これは最も単純で最も論理的なモデルです。 しかし、彼女にはいくつかの短所があります。







  • まず、同じプロファイルを持つユーザーは非常に少ないです。
  • 第二に、モデルはあまり安定していません。 ユーザーはさまざまな製品を見て、一致するマトリックスは常に変化しています。
  • オンラインストアの全オーディエンスをカバーする場合、これらは数千万人の訪問者です。 そして、すべてのユーザーに一致するものを見つけるために、このマトリックス自体を乗算する必要があります。 そして、そのような百万番目の行列の乗算は、私たちが使用するアルゴリズムとツールの観点から重要なタスクです。


Amazonのオンラインストアでは、別のアルゴリズムが考案されています-アイテム-アイテム：ユーザー向けの通信は確立されていませんが、「メイン」製品と一緒に購入された製品を含む特定の製品用です。 これは、アクセサリーの販売増加に最も関連しています。電話を購入した人は、ケース、充電、その他を推奨できます。 製品の一致はめったに変化しないため、このモデルははるかに堅牢です。 アルゴリズム自体ははるかに高速です。

プロジェクトのアーキテクチャ







PHPで記述された単純なワーカーでKinesisからデータを読み取ります。 なぜPHPなのか？ それは私たちにとってより馴染みがあり、私たちにとってより便利だからです。 Amazonには、ほとんどすべての一般的な言語のサービスで動作するSDKがあります。 次に、ヒットの初期フィルタリングを行います。検索ボットなどの多数のヒットを削除します。 次に、Dynamo DBですぐにオンラインで提供できる統計を送信します。







Sparkでのモデルの構築など、後続の処理のためのメインデータ配列。 S3に保存します（従来のHDFSの代わりにAmazonストレージを使用します）。 その後の数学、協調フィルタリング、機械学習アルゴリズムは、Apache Mahoutに基づいて構築された推奨事項のクラスターによって処理されます。







  • データ処理には、Apache Hadoop / MapReduce、Apache Spark、Dynamo DB（NoSQL）を使用します。
  • Apache Mahoutは数学計算に使用され、行列乗算の大規模処理はMapReduceパラダイムを使用して行われます。
  • データは、Amazonクラウドの動的コンピューティングクラスターで処理されます。
  • ストレージはAmazon S3 Objectiveで行われ、一部はNoSQL DynamoDBで行われます。


Amazon AWSクラウドインフラストラクチャと既製のサービスを使用すると、多大な労力、リソース、時間を節約できます。 このシステムにサービスを提供する管理者の大規模なスタッフは必要ありません。大規模な開発チームは必要ありません。 上記のすべてのコンポーネントを使用すると、非常に少数の専門家を管理できます。



さらに、システム全体がはるかに安価です。 テラバイト単位のデータはすべて、ディスクストレージを備えた別のサーバーを作成したり、バックアップを処理したりするよりも、S3に入れる方が収益性が高くなります。 インフラストラクチャを構成し、管理し、低レベルのメンテナンスタスクを解決するよりも、Kinesisを既製のサービスとして高め、文字通り数分または数時間で使い始める方がはるかに簡単で安価です。

品質指標



主な疑問が生じます。上記のすべてがどれほど効果的に機能するのでしょうか？ パフォーマンスをどのように測定しますか？



  • ビューの総数に対する推奨のビューの割合を計算します。
  • また、推奨事項に対する注文数と合計注文数も測定します。
  • ユーザー状態マシンで測定を行います。 3週間の行動を示すマトリックスがあり、それらに関する推奨事項を示します。 次に、たとえば1週間で状態を評価し、出された注文を将来提供できる推奨事項と比較します。 一致する場合、正しくお勧めします。 それらが一致しない場合は、アルゴリズムを強化するか完全に変更する必要があります。
  • A / Bテスト。 オンラインストアの訪問者をグループに単純に分けることができます。1つは個人的な推奨事項のない製品を示し、もう1つは推奨事項のある製品を示し、売上の違いを評価します。


A / Bテストを使用して、近い将来にパーソナライズされた推奨事項を処理できるようになります。オンラインストアの所有者は、管理パネルで必要なメトリックを選択および構成し、しばらくの間データを収集し、異なるオーディエンスのコンバージョンを比較して差異を推定することができます。



データによると、コンバージョンの増加率は10〜35％です。 10％でさえ、投資の観点からオンラインストアの大きな指標です。 ユーザーは広告やエンゲージメントにより多くのお金を投入する代わりに、視聴者とより効果的に連携します。
