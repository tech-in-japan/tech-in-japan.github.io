QtDockTile-クロスプラットフォームでのドックの使用！

デスクトップ開発の現在の傾向を考慮すると、ドックのアイデアがますます普及しているという事実に注意を払わないことは困難です。 この原則には、少なくとも3つの一般的な実装があります。Makovskyドック、windiws 7のタスクバー、Unityのランチャーです。 アイコンタスクもkde 4.8でこのリストに追加されます。

図書館の建築





最大の柔軟性と拡張性のために、特定の各ドックの実装は通常のQtプラグインであると決定しました-これにより、ライブラリ全体を再コンパイルせずに新しいAPIのサポートを追加でき、いずれかの実装を使用できない場合、プラグインは起動しません。 プラグインは、特別なシングルトンマネージャーによってロードされます。 各プラグインは、この環境で動作できるかどうかをマネージャーに通知します。その結果、マネージャーは、この環境で動作可能なプラグインでのみ必要なメソッドを呼び出すことができます。

ユーザーは、マネージャーのラッパーである単純なQtDockTileクラスを使用します。 その結果、QtDockTileインスタンスをいくつでも安全に作成できます-それらはドックを中断しません。

ドックメニューには、通常のQtshnoe QMenuが使用されます。 このプラットフォームまたはそのプラットフォームが設​​定する制限のみを覚えておく必要があります。

奇妙なことに、Unityの場合、最短で簡潔な実装が行われました。 すべてのAPIは、非常に単純なdbusメッセージの送信に基づいています。 



 void UnityLauncher::sendMessage(const QVariantMap &map) { QDBusMessage message = QDBusMessage::createSignal(appUri(), "com.canonical.Unity.LauncherEntry", "Update"); QVariantList args; args << appDesktopUri() << map; message.setArguments(args); if (!QDBusConnection::sessionBus().send(message)) qWarning("Unable to send message"); }




appUriはアプリケーションの一意の名前です。この実装では、プロセスの名前と単純に一致し、appDesktopUriはapplication：//$appUri.desktopというフォームのエントリです。

  QVariantMap map; map.insert(QLatin1String("count"), count); map.insert(QLatin1String("count-visible"), count > 0); sendMessage(map);

Unity Launcher APIの制限




  1. バッジはデジタルのみであり、0より大きいのみです。それ以外の場合、0が表示されます。
  2. エクスポートされたメニューにはサブメニューが表示されないため、サブメニューは避けたほうがよいでしょう。
  3. メニューもappmenuにエクスポートされる場合、ドックに表示されません
  4. DBusMenuExporterの実装にはバグがあり、その結果、メニューのチェック状態が反転します

おわりに





ライブラリは非常に使いやすく、拡張も簡単です。 これまでのところ、すべてのプラットフォームにある基本的な機能のみをカバーしています。 将来的には、プラットフォーム固有の拡張機能を追加する方法を検討します。

メニューが問題なくドックにエクスポートされることを保証するには、次の点を満たしている必要があります。

  • サブメニューはありません
  • 切り替え可能なアイテムまたは無効なアイテムはありません
  • 少数のポイントがあるはずです
  • setMenuで設定された後は変更しないでください


そしていくつかのコメント：

  • ドックの正しい作業は、単一のアプリケーションの場合にのみ可能です。ドックをQtシングルアプリケーションまたは他の同様の手段と組み合わせて使用​​します
  • バッジでは、100未満の正の数を使用することをお勧めします


その他の場合、すべてのプラットフォームで何かが利用できるとは限りません。 原則として、これは致命的ではありませんが、これを覚えておく必要があります！
