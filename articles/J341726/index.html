ユーザーロールごとに個別のアプリケーションが実装されました。 スキームの下で、これが行われた理由を説明します。

バックエンド構造





バックエンドレベルで3つのアプリケーションのスキームを実装するために、次の要件を作成しました。



  1. 各アプリケーションには、独自の一意のURL（エントリポイント）が必要です

  2. 各アプリケーションには一意の識別子とアクセスキーが必要です（アプリケーションの作成時にbitrix24によって生成され、作業時に必要です）

  3. 単一のGitリポジトリを使用する

  4. 3つのアプリケーションすべてに共通のデータベースが必要です（3つのアプリケーションはすべて同じエンティティで動作します）



その結果、次のものが得られました。



  1. 各アプリケーションのコードは、個別の仮想ホストディレクトリにあります

  2. すべての特徴的な構成情報は、各アプリケーションの.envファイルに保存されます

  3. gitでは、管理アプリケーションのみをホストしています。 学生および教師のアプリケーションホストには、バックエンドのフレームワークとして使用されるlaravelを「取得」するための最小限のエンティティが含まれます。 同時に、メインホストに格納され、すべてのアプリケーションの作業に必要なエンティティは、シンボリックリンクによって単にリンクされます。 各アプリケーションには独自のバンドルjsとcssがあり、これも管理アプリケーションディレクトリにあり、対応するアプリケーションのエントリポイントを介して接続されています。

  4. 3つのアプリケーションすべてに1つのデータベースがあります。 移行とシードとの競合が発生しないように、データベースディレクトリは、生徒および教師のホストから管理者アプリケーションのgitの下のディレクトリにリンクされます。

エラー制御



どのようにすべてをテストでカバーしようとしても、それは不可能です。常に例外的な状況があります。 いずれにしても、間違いは避けられません。 回避できない場合は、どうにかして制御できますか？



おそらく、サーバーでエラーが突然発生した場合、ログに書き込まれ、その後診断できるという事実に慣れているでしょう。 しかし、フロントエンドで発生する間違いをどうするか？ クライアントで起こることはすべて彼にあります。 エラーが十分に重大である場合、怒っている顧客のフィードバックを通じてそれについて知ることができます。 これは明らかに私たちの方法ではありません。



アプリケーションの生産プロセス中に発生する可能性のあるエラーをキャッチするために、このサービスhttps://sentry.ioを使用し、ひどく楽しんでいます 。 このサービスは、クライアントエラーとサーバーエラーの両方を構築します。 すべてのエラーが追跡され、解決する必要があります。



ボーナスとして、スラックの統合を設定して、先例の瞬間にプロジェクトチャネルに通知が表示されるようにします。 これにより、お客様がサポートシステムでチケットを作成する前にエラーを検出し、状況が重大な場合は迅速に対応できます。

診断



サイトポータルリンクを形成し、スケジュールを作成すると、データの整合性に違反する競合が発生する可能性があります。 たとえば、1人の教師が誤って同じ日に、同じカップルに複数のクラスを割り当てられている場合があります。



人間はロボットではありません。 あなたは間違いなくすべてを追跡することはできません。 このような問題を解決するために、「コントロール」診断画面を収集しました。この画面では、チェックのリストとその結果をチェックリストの形式で配置しました。





診断の仕組み



スケジュールを作成するときなど、問題を引き起こす可能性がある状況で診断を実行すると便利です。 すぐに確認することは可能ですが、より柔軟でスケーラブルなソリューションを適用しました。



チェックリストを拡張可能として計画しました。 一部の操作は、非常にリソースを消費します。 gearmanを使用して、診断でアプリケーションの動作をブロックしないように、「潜在的に危険な」イベントが発生し、管理者アプリケーションが最初に起動されたときに、検証タスクをキューに送信します。



チェックの1つでエラーが発生した場合、制御セクションに移動することを強く推奨して「悪のロボット」を起動します。







開発者がスケジュールの競合などを解決できない場合があるため、「アセンブリ」テストで診断を開始しませんでした。


仕事で使用したもの



技術的な観点から見ると、プロジェクト全体が非常に複雑であるため、テクノロジーの選択に徹底的に取り組みました。 サーバープログラミング言語として、MySQL DBMSと組み合わせてPHPを使用することにしました。 このバンドルは、ウェブ開発の世界で非常に人気があり、さらに私たちのチームによく知られています。



このプロジェクトはCRM Bitrix 24の使用に基づいているため、多くはポータルAPIの使用に関係しています。 多くの場合、さまざまなメソッドの呼び出しが使用されるため、この部分を最大限に簡素化する必要があります。 この目的のために、便利なオブジェクトアプローチを提供するbitrix24-php-sdkを使用することにしました。 同時に、このライブラリの開発にも少し参加しています。



2017年にネイティブPHPを作成することは、特にこのサイズのプロジェクトにとって最良の選択肢とはほど遠いものです。 したがって、フレームワークを選択する必要がありました。 要件のうち、MVC、ORM、移行およびシードの存在は、キューを操作します。 多くのLaravel、YII2、Zend、Phalconなど、ほとんど知られていないフレームワークがこれらの要件に適しています。 私たちのチームは彼と仕事をした経験が豊富だったので、Laravelを選択することにしました。



フロントエンドの観点からは、優れた機器を選択することも重要でした。 React / Redux / React Routerバンドルに注目しました。



Reactの中心にはコンポーネントアプローチがあり、これは私たちに非常に適しています。 1つのコンポーネントを実装してから、アプリケーションの複数の場所で再利用すると便利です。
