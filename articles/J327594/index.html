HTTP APIを使用してメールを送信する方法

通信技術は、1つまたは2つのチャネルを使用するには不十分になっています。 各チャネルには長所と短所があり、これは議論の余地のない事実です。 プッシュ通知を使用して、特定の地域のユーザーをターゲットにして到達できます。 通常、SMSは常に読み取られます。 Viberは、インタラクティブなコミュニケーションと楽しいコンテンツを提供します。 メールは顧客に迷惑をかけません。



いずれにせよ、今日の顧客指向のエコシステムでは、クライアントのニーズに合わせてコミュニケーション戦略を調整する必要があります。 多くの人がすでに選択の基盤となるプラットフォームを持っています。 彼らはすでにGmail、Facebook Messenger、またはViberとOutlook、および他の多くのアプリケーションを使用しています。 重要な要因は、特定の目的でこれらの各チャネルを使用することです。



たとえば、一部の人はメッセンジャーを使用してリアルタイムの会話を行います。 同時に、彼らはGmailを通してブランドからのニュースレターを受け取ることを好みます。 依然として最も信頼性の高い通信チャネルであるため、SMS経由で連絡を取りたい人もいます。 重要な通知の場合に音声通話を受信することを好むクライアントがいます。 間違いなく、顧客ベースはそれを完全にカバーするためにさまざまな方法の使用を必要とします。







なぜメールするのですか？



そのユニークな構造により、電子メールはマルチチャネル通信の重要な部分のままです。 これは最も経済的で目立たないチャネルの1つであり、ブランドとのコミュニケーションに適した方法の1つとなっています。 私たちはニュースレターについて話しますが、メールが最適に機能するケースは他にもたくさんあります。 アカウントの確認からパスワードのリセットとアカウントの復旧まで。



そのため、毎日何十億ものメッセージが電子メールで送信されます。 エリクソンによると、2020年までに、世界人口の70％がスマートフォンを使用し、90％がグローバルモバイルネットワークのカバレッジエリアに参入します。 また、電子メールの50％以上がデスクトップからではなくモバイルデバイスから読み取られることに注意することも重要です。この数は増え続けています。 つまり、ユーザーはいつでもアカウントにアクセスできます。



HTTP APIは、シンプルで多機能な電子メールの送信を提供します。 添付ファイル付きのHTMLメールを送信し、ログと配信レポートを表示できます。 ただし、メールの送信を開始する前に、有効なInfobipアカウントを作成する必要があります 。



アカウントを作成する方法



APIを介して他のサービスを使用する場合、既に有効なInfobipアカウントを持っているため、それを使用してメールを送信できます。



電子メールの送信者の評判はドメイン名とIPアドレスに密接に関連しているため、ドメインに関連するすべての問題を事前に検討してください。 「悪い」送信者になると評判の問題が発生し、これはドメインの状態に大きな影響を与える可能性があります。 「悪い」送信者は、自分への自信を取り戻すために膨大なリソースを費やします。 そのため、メッセージタイプごとにドメインを分離することをお勧めします。



ドメインの計画を考えていない場合は、心配しないでください。 テストドメインを使用してサービスに慣れ、後で独自のドメインの使用を開始できます。 優先ドメインを選択したら、アカウントマネージャーに連絡する必要があります。その後、ドメインをプラットフォームに統合できます。 DNSの構成に関する追加の指示を受け取り、ドメインとプラットフォーム間の目的の通信チャネルを確立したら、次の手順に進み、電子メールの送信を成功させることができます。

support@infobip.comにお気軽にご質問ください。



メールを送ろう



アカウントを作成したら、メールを送信できます。 シンプルなレターまたはリッチコンテンツを含むレターを送信できます。 簡単な手紙の送信を開始することにした場合、これは、リクエストを介して1つのメールメッセージのみを送信することを意味します。







一方、1つまたは複数のアドレスに送信される添付ファイル付きのメッセージである多機能電子メッセージを送信できます。







答え（画像はクリック可能）：







これは、2つのアドレスに1つの添付ファイルが送信された電子メールの例です。 複数の添付ファイルを含むメールを送信する場合は、リクエストに複数の添付ファイルのパラメーターを入力します。



正常に送信された場合、HTTPステータスコードレスポンスのヘッダーは200 OKのようになります。認証なしでメッセージを送信しようとすると、 401 Unauthorizedエラーコードが返されます。



メール配信レポートと送信ログ



メール配信レポート



メールを送信すると、 1回限りの配信レポートが届きま​​す。 配信レポートは、送信したメールが正常に配信されたことを確認します。 レポートは1回しか受信できません。レポートを受信した後、再度リクエストすることはできません。



パラメータを入力して配信レポートを受信できます。また、パラメータなしでリクエストを行うこともできます。







結果（写真はクリック可能）：







パラメータのない配信レポートは、メールのステータス、メッセージID、メールの送受信日時、エラー情報など、送信されたメールに関する詳細情報を提供します。



messageIdを使用して配信レポートを取得することもできます。







もう1つの可能性は、2つの初期配信レポートです。







両方の要求は、パラメーターなしの要求としてメッセージの詳細を表示しますが、違いは、messageID要求が特定のメッセージに関する情報を取得し、2つの配信レポートが2つの初期メッセージに関する情報を提供することです。



メールログ



配信レポートと同様に、送信ログは送信されたメッセージに関する追加情報を提供します。 主な違いは、ログは過去48時間以内に利用可能であり、何度でも要求できることです。現在のステータスに関係なく、メッセージの結果が表示されます。



結果（写真はクリック可能）：







応答には、過去48時間以内にInfobipプラットフォームを介して送信されたすべての電子メールに関する情報が含まれます。 このタイプにはクエリは必要ありませんが、特定のメッセージを見つける必要がある場合は、フィルタリングオプションを使用できます 。



たとえば、ログは1つ、またはこの場合はいくつかのメッセージIDを介して受信できます。







また、パラメーターの組み合わせは、メッセージをより正確にフィルターするのに役立ちます。







このリクエストには2つのフィルター（パラメーター）があります。sentSinceとgeneralStatusは単一のパラメーターとして識別され、応答には、リクエストの日付から送信および配信されたすべての電子メールのメッセージログが含まれます。



ステータスコードは、送信されたメッセージと同じです。 HTTPステータスコードの応答ヘッダーは、正常に送信されると200 OKになり、メッセージログと配信レポートも返されますが、許可なしでメッセージを送信しようとすると、HTTPステータスコード401 Unauthorizedの応答が返されます。
