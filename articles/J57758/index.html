13）Xorgの優先度

このヒントでは、Xorgの優先度を上げることでUbuntuでの作業をスピードアップできます。 このヒントは、Ubuntu（Gnome）とkUbuntu（KDE）の両方のユーザーに適しています。 優先順位はreniceコマンドによって変更されます。 0〜-19の優先順位が上がり、0〜+19の優先順位が下がることを覚えておく必要があります。 この場合、Xorgの優先度を0から-5に変更します



sudo renice -n -5 $（pidof X）



再起動後にXorgの優先度を上げるには、Gnomeにgksu / path / to / scriptを、StartupのKDEにKdesudo / path / to / scriptを入れます。

または、コマンドrenice -n -5 $（pidof X）を/etc/rc.localに追加すると、Xorgが起動した後でもrootから実行されます。



14）Mozilla Firefoxを高速化するためにフィッシング対策を無効にする

注意!!! フィッシングとは何か、オンラインの脅威から身を守る方法がわからない場合は、これ以上読んでこのアドバイスを適用しないでください！ フィッシングは、現在と外見上区別できないサイトへの直接リンク、またはリンクが既知のリソースに類似しているように見えるサイトへの直接リンクを提供するときに、ユーザーを欺く方法です。 このサイトにアクセスすると、ユーザーはアカウントや銀行口座にアクセスできる貴重な情報を詐欺師に通知できます。 フィッシングから保護するために、主要なインターネットブラウザーのメーカーは、同じ方法を使用して、詐欺師に属する疑わしいサイトを開いたことをユーザーに通知することに同意しています。 ブラウザの新しいバージョンにはすでにこの機能があり、これを「フィッシング対策」と呼びます。

フィッシング対策は、決して不要ではない保護です。 フィッシング対策はあなたにとって追加の懸念事項です。 しかし、たとえば、管理している場合は、Firefox用のNoScriptプラグインがインストールされており、未知のサイトであなたを保護します。 または、自分でネットワークセキュリティを監視している場合は、フィッシング対策を無効にしてセキュリティを低下させることを犠牲にして、Firefoxを高速化することができます。

高速化は、各URLがフィッシングアドレスのベースに対してチェックされないという事実により実現されます。 フィッシング対策を無効にしてFirefoxを高速化するには、次の手順を実行します。

1）Firefoxの[編集]-[設定]-[保護]メニュー。 「攻撃の疑いのあるサイトをブロックする」と「詐欺の疑いのあるサイトをブロックする」のチェックを外します。 変更を保存します。

2）Firefoxを閉じます。 Firefoxが削除した後、アドレスの大きなファイルベースを削除する必要があります。 ターミナルとコマンドを起動します



ls -laF〜/ .mozilla / firefox / *。default / urlclassifier *

rm -i〜/ .mozilla / firefox / *。default / urlclassifier *



3）Firefoxを起動して、パフォーマンスと開始時間と応答性を確認します。 次のコマンドでターミナルをチェックインします



ls -laF〜/ .mozilla / firefox / *。default / urlclassifier *



ファイルが以前よりも小さく作成されたこと。 「総SQLiteのクリーニングと再インデックス付け」の上のヒントをお読みください。



短所：ホームディレクトリのFirefoxファイルを削除し、Firefoxプロファイルをバックアップすることをお勧めします。



15）ファイルシステムの高速化

このヒントでは、ファイルシステムに関する設定を変更します。 このアドバイスは、無停電電源装置（UPS）の所有者のみが使用することを強くお勧めします。

Linuxでは、多くのファイルシステムを使用でき、それぞれにパフォーマンスに影響する独自のパラメーターがあります。 必要なパラメーターを指定するには、/ etc / fstabファイルを慎重に編集し、そのコピーを最初に作成する必要があります。

パフォーマンスを向上させるためのファイルシステムオプション：

ext4アクセラレーション

barrier = 0ファイルシステムコードは、コミット[ログ]レコードを作成する前に、すべてのトランザクション情報がログに記録されることを完全に確認する必要があります。 正しい順序で録音するだけでは十分ではありません。 最新のドライブには大きなキャッシュがあり、パフォーマンスを最適化するために記録順序を変更します。 したがって、ファイルシステムは、コミットレコードを作成する前に、すべてのログデータをメディアに書き込む必要があることをディスクに明示的に通知する必要があります。 最初にコミットレコードが作成された場合、ログが破損している可能性があります。 カーネルのブロッキング入出力システムは、「バリア」（バリア）を使用してこの機会を提供します。 簡単に言えば、「バリア」は、「バリア」がメディアに転送される前に送信されたすべてのものまで、それ以降に送信されたブロックの記録を禁止します。 「バリア」を使用する場合、ファイルシステムはディスク上のすべてがいつでも完全であることを保証できます。 バリアbarrier = 0を無効にすることで、ext4パーティションへの書き込み操作を高速化します。



noatime、nodiratime-最終アクセス時刻の記録を無効にします（ファイル/ディレクトリの場合）。 またはrelatime-アクセス時間属性（atime）は更新されますが、ファイルデータが変更された場合（mtime属性）またはそのステータス（ctime属性）のみです。 relatimeの使用をお勧めします。



commit-バッファーをディスクにフラッシュする間隔。 commit = 100をお勧めします



data =ライトバック 。 デフォルトでは、ext4のロギングモードはdata = orderedモードです。 ファイルシステムはメタデータのみをログに記録し、データとメタデータは1つのモジュール、つまりトランザクションにグループ化されます。 このモードは、保証はありませんが、予期しない再起動時にデータを保護します。 データ=ライトバックモードでは、ファイルシステムはデータを記録しません。 システムが予期せず再起動した場合、更新されたファイルのデータが失われる可能性があります。 このモードは、最高のext4パフォーマンスを提供します。 順序付きモードを安全性の低い、ただし高速のライトバックに変更するには、セクションをアンマウントしてsudo tune2fs -O has_journal -o journal_data_writeback / dev / SECTION_NAMEを実行することが必須です。 sudo fsck / dev / SECTION_NAMEのエラーのセクションを確認することをお勧めします。 この記事では、ルート/パーティションのロギングモードの変更については説明も推奨もしません。

syslogのマイナス 。 syslogロギングデーモン（およびそれに代わるrsyslog）は、ログを/ var / log /ディレクトリに書き込み、1行追加して、ディスクキャッシュと長いバッファーバッファーを無効にする同期操作を実行します。 デーモンの動作を変更し、追加するたびに同期しないように指定できます。 デーモン構成ファイル（通常は/etc/syslog.confまたは/etc/rsyslog.d/）を見つけ、パスの前にマイナス記号（ "-"）を追加して/ var / log /の形式のすべてのパスを変更します。 たとえば、次の行がありました
