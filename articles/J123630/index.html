Elephants Love Mango、Part 1/2：Backstage Look in Evernote for Windows Phone 7

WP7のEvernoteクライアントの作成者、Damian Meyersによる投稿







はじめに



1年前、私は大規模な小売企業のIT部門で働き、11人のチームを管理していました。 一般的に、私は10代の頃からソフトウェア開発に従事していましたが、私は常に管理職になりたいと思っていました。 成長企業になる前に、2つの最初のスタートアップを売却したので、管理経験の不足が作業の弱点のままでした。

その後、逆の状況が発生しました。 私はリーダーシップの仕事が好きで、それは私にとって非常に有能でしたが、スキルを失わないようにプログラミングの必要性も感じました。



その結果、暇なときにWindows Mobile 6.5用のEvernoteクライアントを作成しました。 オフラインメモなど、公式アプリケーションでは利用できなかった多くの機能を実装しました。 そして、作業を終えるとすぐに、マイクロソフトはモバイル市場での戦略のリロードとWindows Phone 7（WP7）のリリースを発表しました。



その時、私はすでに経営陣が私のためではないことを理解し始めていたので、EvernoteのCEOであるPhil Libinから電話があり、Evernoteに参加してWP7のサービスクライアントを作成したいかどうか尋ねられたとき、新しいスタートアップについて考え始めました。 それは非常に時間通りでした-もちろん、私は同意しました。 そして最近、このプラットフォーム用のクライアントの最初のバージョンを導入しました。



これは、何が実装され、何が実装されなかったのか、WP7を優れた開発プラットフォームにしているのか、どのような困難があるのか​​についての物語です。 また、コードネームMangoのWP7の次のバージョンのリリースで開かれる可能性についても検討します。





まともな開発プラットフォーム





古いルーツ


.NETとSilverlightをWP7の基盤として選択することで、Microsoftは.NETの既存の開発者の大規模なコミュニティにとってプラットフォームをすぐに魅力的なものにしました。



Silverlightの使用経験がほとんどなかったにもかかわらず、私は長年C＃で開発を続けていましたが、今では新しく獲得したスキルを適用する機会がありました。 Language Integrated Query（ LINQ ）、 ラムダ、ガベージコレクションなどの技術を使用した開発経験がある場合、それらを後で放棄することは困難です。



Silverlightは数年前から存在しており、この技術はインターネット上で普及していませんが、企業環境向けのイントラネット開発では非常に需要があります。

SilverlightとModel View ViewModel（ MVVM ）開発パターンが一緒に使用されます。 ユーザーインターフェイスがXAMLと呼ばれるXML言語として処理されることを宣言します。XAMLは、ViewModelコードに宣言的にバインドされます。 これは、ユーザーインターフェイスを使用せずに、ViewModelコードを個別にテストできることを意味します。 また、Silverlightは非常に便利なアニメーションおよび視覚制御システムを受け取りました。



優れたツールキット



WP7開発者は甘やかされています。 Visual StudioにResharperなどの優れたプラグインを備えた強力で成熟した機能的なIDEがあるだけでなく、Expression Blendの高速なエミュレーターと独立した設計ユーティリティもあるためです。



エミュレーターは、コンピューターのハードウェア仮想化機能を使用して、本格的な仮想マシンのように動作します。 プラスは、高いパフォーマンスを提供しますが、同時に、エミュレータは単に起動しないため、仮想マシン内で開発することはできません。



Blendを使用すると、アニメーションを設計、開発し、アプリケーションのユーザーインターフェイスをカスタマイズできます。 元々はデザイナー向けの有料のハイエンドツールとして開発されたものの、開発者側からの感情の嵐を引き起こしたため、MicrosoftはWP7開発者向けに無料で提供しました。 Blendを使用すると、インターフェイスの外観と相互作用に集中し、コードを気にせずにストーリーボードとアニメーションに取り組むことができるため、操作が簡単で便利です。



コミュニティ



コミュニティ

WP7の開発に関する強力なコミュニティの存在はユニークなものとは言えませんが、おそらくプラットフォームの若さのために、コミュニティで高い活動が感じられます。



Stack OverflowやMicrosoft 独自のフォーラムなどの通常の場所に加えて、 Windows Phone Geekなどのサイトでは、品質に関するヒントやコツ、テーマに関する記事が絶えず流入しています。



コンポーネント

プラットフォームはまだ新しいため、人々はさまざまな場所に新しいフレームワークを投稿していますが、 WP7Contribの周りに人々が集まり始めていることがわかります。



WP7のEvernoteクライアントでは、次を使用しました。

  1. Microsoft WP7 ツールキット
  2. Laurent BunionによるMVVM Light Toolkit
  3. PhoneyToolsのSequen WildermouthのFadingMessageコンポーネント
  4. Colin Eberhardtによる流体リストアニメーション 。




いくつかの困難



データベースの不足


Googleでのインタビューについては、実際には決して使用しないさまざまなハッシュアルゴリズムとソートアルゴリズムの実装について詳しくお尋ねになると、誰もが既に聞いたことがあるでしょう。



この知識が役立つ世界へようこそ。 WP7の最初のリリースでは、開発者が利用できるデータベーステクノロジーはありません。 それは、隔離されたストレージとファイルを検索する機能です。



ノートブックをオフラインで保存する機能は、クライアントで数千のノートを検索する機能と相まって、データベースの欠如を敏感な障害にします。 メモは同じノートブック内にあり、それに関連付けられた0個以上のタグがある場合があります。 ノートはノートブックまたはタグで表示でき、ノートのリストは、見出し、日付、位置情報などのさまざまなパラメーターでソートできます。 一般的に、データベースは非常に役立ちます。



数人がこのギャップに夢中になり、さまざまなレベルの複雑さと完全性のデータベース技術を提案しています。 2010年の終わりと2011年の初めに数週間かけてそれらを試しましたが、私の要件をすべて満たすものは見つかりませんでした。 最終的に、私は自分の自転車を作る方が簡単だと判断し、独自のシンプルなメカニズムを作成しました。



プラットフォームエラー


通常、プラットフォームでバグを発見したと考えるのはかなり予想外です。問題が自分のコードにある可能性がはるかに高くなります。 ただし、Windows Phone 7などの新しいプラットフォームの最初のリリースでは、問題が本当に彼女にあることがあります。



たとえば、ブラウザコンポーネントをフォーカスが受け取った後は、ページ上のコントロールにフォーカスを設定できません。 または、ピボット内の3番目のリストアイテムの選択を復元しようとすると、どのようにクラッシュしますか？ または、ログに予期しないエラーメッセージがありますか？ 私が遭遇したこれらすべてのことはWP7の問題であることが判明し、プラットフォームの若者によって非常に説明できました。 あなたのコードだけが問題を引き起こしているのではないことを、多かれ少なかれ準備してください。



あなたがそこに見つけると期待したいくつかの機能もありますが、それらは単に存在しません-例えば、あなたはサイトからデータストリームをダウンロードできますが、あなたはサイトにストリームを向ける逆の能力を持っていません。 これは、サイトにデータを送信する場合、送信プロセスのすべての情報がメモリにある必要があることを意味します。 非常に大きなファイルをアップロードする場合、これは問題です。



そして最後に、プラットフォーム自体は、IsolatedStorageまたはネットワーク接続へのアクセスなど（UIストリームの参加なしで開始され、もちろんWebClientクラスを使用しない場合でも）、予期しない状況でUIストリーム内のさまざまな機能の起動を必要とします。 。



スムーズなスクロール


プラットフォームが高速でレスポンシブなインターフェースを提供することに重点を置いていることを考えると、画像や可能なダイナミックローディングなど、リストに多くの要素がある場合、本当にスムーズなスクロールを実装するのがどれほど難しいかは不思議でした。 そうそう、リストには何千ものそのようなアイテムがあるかもしれないと言った？



組み込みのListBoxを使用しました。 VirtualizingStackPanelを介してユーザーインターフェイスの仮想化を提供します。つまり、インターフェイスオブジェクトは、表示される要素に対してのみ初期化されます。 また、データの仮想化も可能です。ListBoxが関連付けられているリストにIListが実装されている場合、現在必要な要素のリストを照会するだけです。つまり、必要に応じてデータソースから要素を動的にロードできます。



これらの利点にもかかわらず、私はWP7ツールキットからLongListSelectorも導入しました。 グループ化やアンカーなど、データリストがいつ使用され、いつ使用されなくなったかを示す興味深い機能があります。 ユーザーインターフェイスの仮想化を提供しますが、データの仮想化が欠けています-最初に行うことは、リスト内の各要素をバインドすることです...



スムーズなスクロールを作成するための多くの記事とテクニックがあります。 これは、バックグラウンドスレッドからデータをロードし、スクロール中にダウンロードを一時停止しています。 しかし、多くの要素がある場合、すべてを正しく行うことは依然として非常に難しく、すべての要素がメモリ内にあるわけではありません。



WP7のクライアントの次のバージョンでは、IList要素を操作する特別なアプローチの利点があるため、ListBoxの使用に戻ることにしました。
