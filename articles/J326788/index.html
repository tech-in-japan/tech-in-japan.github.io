測定プロトコル（MP）は、Googleアナリティクス（GA）が情報を収集するための基本的なツールです。 つまり、ページビューやその他のイベントに関する情報をGAに送信するために使用される技術です。



技術面



プロトコルはHTTP経由で機能し、単一のエントリポイントwww.google-analytics.com/collectを持ちます。



データは、POSTおよびGET要求によって送信できます。



必須パラメーター



v-現時点1のプロトコルバージョン（v = 1）。

tidはGAのリソースの識別子です。

cid-ユーザーの一意の識別子（標準のGA JSコードをCookieに入れるか、個別に生成する必要があります-以下で詳細に検討します）。

tは、ページビュー/イベントなどのインタラクションのタイプです（t =イベント）。



つまり、リクエストは次のようになります。 



v=1&tid=UA-123456-1&cid=1482535147.1490885188&t=event


なぜなら 相互作用イベントのタイプを選択した場合、イベントを説明する必須パラメーターを追加する必要があります。



ec （イベントカテゴリ）-イベントカテゴリ（ec =登録）。

ea （イベントアクション）-イベントアクション（ea =フォーム）。 



 v=1&tid=UA-123456-1&cid=1482535147.1490885188&t=event&ec=registration&ea=form


パラメータの全リスト



GAへのイベント転送





次のいずれかの方法を使用してリクエストを送信できます。



投稿 



 User-Agent: user_agent_string // https://en.wikipedia.org/wiki/User_agent#Format POST /collect HTTP/1.1 Host: www.google-analytics.com v=1&tid=UA-123456-1&cid=1482535147.1490885188&t=event&ec=registration&ea=form


ゲット



キャッシュの問題を回避するために、最後のパラメーターとして、各リクエストに一意の値を持つzを追加することをお勧めします。 



 GET /collect?v=1&tid=UA-123456-1&cid=1482535147.1490885188&t=event&ec=registration&ea=form&z=12345 HTTP/1.1 Host: https://www.google-analytics.com User-Agent: user_agent_string


送信されたデータの正確性をチェックするサービス



パラメーターcid、uid、ni



これらのパラメーターに関してよく質問がありますので、詳しく見てみましょう。



CID

cid-クライアントID、ユーザーのクライアントアプリケーション（Webブラウザーなど）の一意の識別子。cidは特定のドメインの_ga Cookieに保存されます。



_ga Cookieには、GA1.2.602320690.1474476467のようなものが保存されます



cidパラメーターでは、2番目のポイントの後にあるすべてのものを渡す必要があります。 つまり、この場合： 



 &cid=602320690.1474476467


_ga Cookieは 、Google Analytics Trackerが初期化されたときにスタンプされます。

cidパラメーターは、Measurement Protocolのすべてのクエリに必要です。 MPを介してGAにイベントを送信するのを遅らせた場合、バックエンド側でユーザーcidを保存する必要があります。



cidがそうでない場合



事例の例：



1）以前にcidユーザーを保存していない

2）ユーザーのCookieが無効になっているか、一部のブラウザー拡張機能がgoogle分析スクリプトの実行をブロックしています。



cidパラメーターはデータをGAに送信するために必要なので、そのような場合、バックエンド側でcidを生成する必要があります。



このようにしてcidを生成します： 



 $cookieId = rand(1000000000, 2147483647) . '.' . time();


cidの自己生成を脅かすもの

送信されたデータは、Googleアナリティクス側のユーザーには連絡しません。



しかし、イベントの数の全体像を把握できます。 なぜなら、バックエンドから測定プロトコルに関する情報を送信していなかった場合、通常は統計に合格していたからです。



uid



システム内のユーザーの一意の識別子（たとえば、サイト上）。 このパラメーターは、ユーザーID機能を有効にするために必要です。



これにより、ログインしているすべてのユーザーの統計情報を含む個別のビューを作成し、各ユーザーのアクション（ビュー、ページ、イベント、支払い）のシーケンスを追跡できます。 クロスデバイスレポートも利用できます。



ユーザーIDの詳細： support.google.com/analytics/answer/3123662



このパラメーターを使用すると、この対話が特定のユーザーによって実行されたことをGAに通知できます。 パラメーターには任意の文字列値を使用できますが、ユーザーの身元を明らかにする情報（電子メールや姓、名など）は送信できません。



ログインしているすべてのユーザーのユーザーIDを接続しました。 しかし、彼らはuidをベースパラメータだけでなく、カスタムディメンション1でも複製しました（これにより、ユーザーIDビューでセグメントを使用できるようになりました）。



メインのGA JSコードでは、これは次のように行われます。 



 ga('create', 'UA-XXXXX-Y', 'auto', { userId: USER_ID, 'dimension1': USER_ID }); ga('send', 'pageview');


これはGoogle Tag Managerを使用してよりエレガントに行うことができますが、これは別の記事のトピックです。



バックエンドからリクエストを送信するには、パラメーターを追加します。 



 &uid=USER_ID&cd1=USER_ID


に



パラメータがあると、新しいセッションを初期化できません。



このパラメーターを使用しない場合、遅延イベントの場合、送信時に新しいセッションが空のパラメーターでGAに作成されます（トラフィックソースはdirect / noneになり、デバイスは（設定されていない）になり、地理はサーバーのIPに対して構築されます）。 つまり、値を持たないセッションであり、統計カウンターを終了し、トラフィックソースに関するレポートを分解します。



たとえば、これが保留中のイベントである場合、CRMでトランザクションステータスが変更されます。 通常、このトランザクションに割り当てられているユーザーがサイトでアクティブなセッションを持っていない場合に発生します。 したがって、＆ni = 1を使用しないと、新しいセッションが作成されます。



一般的に、保留中のイベントには、＆ni = 1を使用します。



実際の測定プロトコル



GA標準トラッキングコード



実行中の最も標準的なJS GAコードでさえ、Measurement Protocolを使用します。 仕組みを見てみましょう。



  1. [開発ツール]パネル（CTRL + Shift + IまたはF12）でGoogle Chromeウィンドウを開きます。
  2. アドレスバーにhabrahabr.ruと入力します。
  3. 開発者向けのパネルでは、[ネットワーク]タブに注目しています。 開発者向けに最初にパネルを開いてからサイトを開いた場合-このタブには情報が含まれます。 シーケンスが異なる場合、タブは空白になります。F5キーを押してページを再度開きます。
  4. 「Filter」行にwww.google-analytics.com/collectを書き込みます
  5. リクエストの詳細を開きます。

CRM統合



目標のコンバージョン率では、トラフィックソースの品質を完全に把握できないという状況に直面しています。



当社のウェブサイトの目標の1つは、無料の入門レッスンを申し込むことです。 ユーザーがそれが何であるかを理解していないが、それでもフォームに記入する場合があります。 当然、すべてのアプリケーションがCRMに分類され、営業チームに負荷がかかります。



GAでは、フォーム=変換が正常に完了したため、不適切なアプリケーションでも変換カウンターが終了します。



各トラフィックチャネルの品質をより詳細に評価するために、追加のパラメーター（ターゲットアプリケーションと入門レッスン）を導入することにしました。



これらのパラメータは両方とも、CRMのトランザクションのステータスによって判断できます。



トランザクションのステータスを変更するときに、CRMがWebhookをサーバーに送信できるため、以下を構成しました。



  • トランザクションがIntroductory Assignedステータスに変わると、イベント「Target Request」を送信します（たとえば、ec = Quality＆ea = Good_lead）
  • トランザクションのステータスがEntry completedになると、イベント「Entry completed」を送信します（たとえば、ec = Quality＆ea = 1Lesson_done）


各イベントに目標追跡を設定しました。 その結果、ターゲットアプリケーションの数を見積もることができました。



トランザクション追跡



私たちは、請求だけでなく、支払い受付システムの口座でのお金の受領の事実を直接検討することにしました。



多くの場合、これは遅れたイベントです。彼らは銀行振込または支払い端末で支払うことにしました。 また、例は「宅配便への現金支払い」です。



支払い受付システムが資金の受領を通知すると、Googleアナリティクスにトランザクションを送信します。 



 v=1&tid=UA-12345678-1&cid=455045511.1409504244&uid=12345&t=event&tr=225&pa=Transaction&ec=Oplata&ea=Individ&ev=225&ni=1&pr1id=12345_individual_id&pr1nm=Individ&pr1pr=225&pr1qt=1


払い戻しまたは一部払い戻し



より正確な統計を得るために、GAには返品（トランザクションのキャンセル）を処理する機能があります。

非常にまれなケースがあるため、自動化しませんでした。



公式ドキュメントの例： 



  // Refund an entire transaction and send with a non-interaction event. v=1 // Version. &tid=UA-XXXXX-Y // Tracking ID / Property ID. &cid=555 // Anonymous Client ID. &t=event // Event hit type. &ec=Ecommerce // Event Category. Required. &ea=Refund // Event Action. Required. &ni=1 // Non-interaction parameter. &ti=T12345 // Transaction ID. Required. &pa=refund // Product action (refund). Required.

オフラインPOS



出席者

Webに情報を送信する機能を備えた店舗訪問者カウンター（オフライン）があります。 測定プロトコルを使用して、この情報をGoogleアナリティクスに転送できます。 GAにページビューを送信すると、マーケティング活動がオフラインポイントのトラフィックにどのように影響するかを追跡できます。



取引

すべての支払いはGAに送信することもできます。ここでは、既にUserIDを使用できます。

将来的には、オフラインで買い物をするユーザーとオンラインで買い物をするユーザーをセグメント化できるようになります。

マーケティング担当者はあなたに感謝します。
