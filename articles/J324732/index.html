タスク「オンラインゲームからの終了」
参加者がオンラインゲームにとどまるか、退場するかを予測することを学ぶ必要がありました。 トレーニング/テストのデータは過去2週間に計算されたものであり、その週にゲームにデータが存在しない場合は注意が必要です。



合計で、過去2週間に計算された12の記号が使用されます。



-maxPlayerLevel-プレーヤーが渡したゲームの最大レベル

-numberOfAttemptedLevels-プレーヤーが渡そうとしたレベルの数

-triesOnTheHighestLevel-最高レベルで行われた試行の数

-totalNumOfAttempts-試行の総数

-averageNumOfTurnsPerCompletedLevel-正常に完了したレベルで完了した移動の平均数

-doReturnOnLowerLevels-プレイヤーは既に完了したレベルでゲームに戻りましたか

-numberOfBoostersUsed-使用されたブースターの数

-fractionOfUsefullBoosters-成功した試行中に使用されたブースターの数（プレーヤーはレベルに合格した）

-totalScore-得点の合計数

-totalBonusScore-獲得したボーナスポイントの総数

-totalStarsCount-得点の合計数

-numberOfDaysActuallyPlayed-ユーザーがゲームをプレイした日数



タスクに提供されるすべてのデータは、トレーニング（x_train.csvおよびy_train.csv）とテスト（x_test.csv）の2つの部分に分かれています。 x_train.csvおよびx_test.csvファイルの各行は、1人のユーザーに対応しています。 行のデータはセミコロンで区切られます。 最初の行には、特性の名前が含まれています。 y_train.csvファイルには、ユーザーがゲームに残っているか、ゲームを離れているかに応じて、それぞれ1または0の値が含まれます。



トレーニング（x_train.csvおよびy_train.csv）サンプルとテスト（x_test.csv）サンプルの両方に、25289ユーザーに関する情報が含まれています。



このタスクの答えとして、テキストファイルが受け入れられます。テキストファイルの各行はx_test.csvファイルの行に対応し、0〜1の値（ユーザーがゲームに残る確率）を含みます。 問題を解決するための品質基準として、対数損失関数が使用されます。



区画の数は1日あたり5個に制限されています。



テストサンプルは、40/60の比率でランダムに2つの部分に分割されます。 最初の40％の結果により、コンテスト全体の評価表における参加者の位置が決まります。 残りの60％の結果は、競技終了後に知らされ、参加者の最終的な配置を決定します。



Loglossがメトリックとして使用されたため、次のことを覚えておく価値がありました。



-多くの分類子は、「箱から出してすぐに」ログ損失を最小限に抑える方法をすでに知っています。

-たとえば0.97や0.03の代わりに1.0または0.0に回答を四捨五入すると、Loglossは「許しません」と誤解します（下の図を参照）。

問題の解決について



最初の提出は、コンテストの開幕直後に行われました。これは、彼の意見では、トップ10のランダムフォレストです。 その後、約1週間を費やしたステップが実行されましたが、生産的ではありませんでした。2次フィーチャが生成され、交差検証の結果に基づいてフィーチャを選択する最も単純な貪欲なアルゴリズムが起動しました。



アルゴリズムは次のとおりです。フォレストをトレーニングし、そこから機能の重要性を取得し、最初の2つから始めて、現在のセットに1つの機能を追加します。 機能を追加することで結果が改善される場合は、機能をセットに残します。 そうでない場合、私たちはそれを投げ返し、再びそれに戻ることはありません。



この単純なアルゴリズムが完了すると、別のヒューリスティックを使用します。一度に1つの機能をスローし、他のすべての機能に取り組んでいるときに結果を確認します。 結果が改善される場合、それは炉内の特徴を意味します。



フィーチャの検索によりフォレストが生成され、アパートメントがウォームアップされる一方で、次のようにロジスティック回帰をトレーニングし、xgboostで遊んでみたい 以前は使用していません（はい、はい）。



最初に、利用可能なサインに対してロジスティック回帰を試みました（もちろん、それらを正規化することを忘れずに）。次に、すべての機能を次のように変換することにしました。



-パーセンタイルで等しいサイズの1 / NのN間隔でフィーチャのすべての値を打ちます

-機能値をoheとしてエンコードします。 例の特徴値が間隔に該当する場合-1を設定し、そうでない場合は0を1つの値から合計して、長さNのベクトル[0 0 ... 0 1 0 ... 0]を取得します。



アイデアは次のとおりです-プレーヤーの異なるレベル（プレイした日数、星の数など）がターゲット変数に非線形に影響し、この非線形をこの変換された形式の線形モデルに持ち込むことができます（ただし、 、切り捨てられた「グループ」バージョン）。



このような変換では、クロス検証で約0.3836が得られました-ロジスティック回帰では悪くありません（後で判明したため、改善される可能性があります）。



それから彼はxgboostに進み、その時点でのランダムフォレストとほぼ同じ（0.3812前後）のクロス検証に十分なソリューションをすばやく得ました。



その後、クロス検証用の3つの優れた分類器の結果を1つに追加し、3つに分けて送信しました。 ちなみに、この状況は私だけでなく、問題を解決したすべての友人や知人にも見られました。 それは面白い点に来ました-分類器が交差検証でより良く振る舞うほど、それは私を公共のリーダーボードに引きずりました。 この状況は意欲を弱めるものではなかったため、タスクへの関心は約1週間失われました。



<過去1週間>



ある日、リーダーボードを開いて、友達がどのようにテーブルに上がったのかを見て、ある日（または2日）にすでに20才でした。 「うわー」と思って、袖をまくり、再び仕事を始めました。



チャートをよく見ると、オンラインゲームに1日参加した人がいて、100以上のレベルを持ち、多くのボーナスとポイントを持っていることがわかりました。 これは、人々が実際に長時間プレイするという考えにつながりました。2週間のウィンドウで、機能を数え、たった1日でそれらをフックしました。 maxPlayerLevel、totalStarsCount、totalNumOfAttemptsの各機能に基づいて、使用方法について数日間頭を痛めたため、プレーヤーが再生する「予測日数」という1つの新しい機能を作成することにしました。 アイデアは次のとおりです-選択した機能（たとえば、maxPlayerLevel）に従って各日で低い値（4パーセンタイル以下をすべて取得）を取得し、多項式でかなりよく近づくかなり良いグラフを取得します（おそらく4番目の程度ですが、おそらく3番目の十分）。 多項式を作成すると、プレーヤーの機能値の根を見つけて、プレーヤーがプレイした日数の予測にコストをかけることができます。 次のようになります。







x軸は日数を表し、y軸は選択されたフィーチャの値を表します。 壊れた曲線は4パーセンタイル以下の同じ特徴値で、美しい滑らかな曲線は近似多項式のグラフです。



多項式コードの近似 
from numpy.polynomial import Polynomial as P class nlSolver(): def __init__(self): self.p=None def fit(self,f_x,f_y): self.p = P.fit(f_x, f_y, 4) return self def get_y_by_x(self,f_x): return self.p(f_x) def get_x_by_y(self,f_y,initial_x): res = max([r.real for r in (self.p - f_y).roots()]) return res




このようにしてさらに3つの属性を取得した後、彼はそれぞれの新しい機能の初期機能（プレーヤーがゲームに費やした日数を除く）を正規化したため、機能の合計数は4倍に増加しました。 たとえば、次のような機能を開発して追加しました。 



 ... X['positiviness'] = X.totalStarsCount*X.totalBonusScore*X.usefulBoosters X['winDesire'] = X.attemptsPerDay*X.numberOfBoostersUsed*X.attemptsOnTheHighestLevel X['positivinessPerWinDesire'] = X.positiviness/(X.winDesire+1) ...


さらに、記号の数の増加を考慮して、彼は2次記号を生成せず、利用可能なものから最良の特徴を選択し、アンサンブルを訓練し、最終的には120位から上昇して60位になりました。



Sklearnのロジスティック回帰は、vowpal wabbitの確率的実装に置き換えられました（私自身の執筆のPythonラッパーを使用しましたが、それについて）、間隔に機能エンコードを使用するというアイデアを放棄し、クロス検証ログの結果をわずかに改善しました（ここで相互予測の結果を得るための松葉杖もほとんどありませんでした）。 なぜなら vwがすでに関与しているので、なぜそこからパーセプトロンを使用しないのか（--nnフラグ）-したがって、別の分類子がアンサンブルに追加されました。



さて、vwのパーセプトロンが関係していたので、MLPClassifierを使用しないのはなぜですか。



その後、ランダムフォレストに基づく2つの分類器がアンサンブルに追加されました。 それらのうちの1つは、特徴の値（すべての例、1パーセンタイル以下および99目以上が分類子のトレーニングに参加しなかった特徴の値）に従って毎日偏差が除去されたサンプルでトレーニングされました。



合計で、競争の終わりまでに、アンサンブルは7つの分類子で構成されていました。 これすべてのために通常の相互検証スキームを作成し、多くの苦労をしない方法は？ 構築スキームは次のとおりです。



  • 各分類子をプロジェクトの個別のフォルダーに配置します。
  • 各分類子は、相互検証の代わりに、列車で相互予測を行い、結果を独自のフォルダー内のファイルに追加し、明確な末尾を付けます（これらのファイルの検索を容易にするため）。
  • 各フォルダーで、トレイン全体に対して1つの分類子をトレーニングし、テストサンプルに述語を作成します。これは、フォルダー内のファイルに特定の末尾を付けて配置します。
  • 分類子フォルダーの上のディレクトリで、一般的なアンサンブルスクリプトを作成します。これは、相互検証のために、必要なすべての分類子フォルダーからすべての相互予測を読み込み、必要なルールに従ってそれらをアンサンブル結果に変換し、必要なメトリック（ログ損失）を計算します。
  • アンサンブル検証結果を受け取った後、各フォルダーから予測を読み込み、同じルールに従ってそれらを変換して、テストサンプルのアンサンブル予測を取得します。
  • 最後の仕上げを追加します。


最後の仕上げ
-サンプルは、同じ機能セットを持つサンプルを見つけましたが、ターゲット変数の値が異なります。 このようなサンプルの場合、確率予測は、正の例の数と負の例の数の比として計算されました（これにより、対数損失が最小限に抑えられます。紙に書き留めて確認できます）。

-「すべてを追加して分類子の数で除算する」代わりに、各分類子の結果に独自の選択された重みを追加し、重みの合計で除算します。 分類子の1つの係数を変更する必要はありません。この方法でスケールを修正し、相互検証の結果が減少するまで残りの係数で「再生」します。 0.0005-0.0007の改善は、手作業ですぐに見つかります（手作業で行うことはできませんが、回帰によって重量を拾います）。
