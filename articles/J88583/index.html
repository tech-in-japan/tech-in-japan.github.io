SCOM 2007およびSensatronics Em1を使用したサーバードレインの温度の監視は、SNMPを実行するデバイスの構成に適しています。



+



Sensatronics機器からSNMP情報を収集することにより、System Center Operations Manager 2007 R2監視システムで温度表示とアラート生成を構成する方法についての短いメモは、ロシアの売り手と米国の公式代表者がこの機器の管理パックがないことを確認しました。 EM1モデルと、温度と湿度の測定値を取得できる2つのセンサー、および現在テストされているSCOM 2007 R2のインストール済みバージョンがありました。





SCOM 07にはSensatronicsの管理パックがないことを繰り返しますが、かなり簡潔なデバイスページに移動すると、ブラウザーでインジケーターを表示できます。



次のようになります。







ページ自体は更新されますが、ページを常に開いたままにして、時々見ているのは楽しみです。 ところで、Nagiosのデバイスからデータを表示する方法の1つは、デバイスのWebページを読み込んで、そこからデータを「切り取る」ことです。 このようなスキームはかつて機能しました。 しかし、現在はOpsMgrについて話しているため、そのような可能性はありません。 結論は、デバイスがSNMPバージョン1をサポートしていることを示唆しています。SCOMはこの方法をサポートしていますが、マッピングへのパスは簡単ではありませんでした。



主にopsmgr.ruの尊敬されているSCOMの第一人者（リンク1、2、3を参照）、netwizard.com.uaの専門家（4）、および外国の専門家（5、6）のおかげです。 管理パック（管理パック）を開発する必要性をほぼ確信しましたが、これを常にxmlで記述し、数晩にわたって喜びを繰り返しました。 これらのSCOMに適合した製品メーカーが、製品とともにMPをリリースするのを本当に助けることを願っています。



私たちのタスクはやや単純で、測定値を表示し、許容可能なインディケーターを超えたときに通知を受け取るだけで、責任者に警告を送信します。



SCOM 2007 R2を構成するいくつかの手順を次に示します。



まず、OIDを読み取るためにMibデバイスが必要です。通常、製造元のWebサイトで見つけることができます。参照番号（7）が役立ちます。 さまざまなプログラムでmibsを読むことができます。私はMibBrowser（8）を使用しました。







必要なMibをロードし、接続に必要なすべてのデータ（IP、SNMPのバージョン、コミュニティストリング）をドライブし、フォレストから必要なパラメーターを選択して、そのOIDを取得します。







オペレーションコンソールを開き、構成を続行します。



まず、コンソールにデバイスを追加して、詳細に説明する必要があります。すべてを助けにしたくはありませんでしたが、完全を期すために、ここで見ます。



管理セクションに移動します。







コンテキストメニューから検出ウィザードを開きます。







アイテムのネットワークデバイスを選択して続行します。







パラメーターを入力し、ディスカバリーを実行します。







肯定的な検索結果では、指定した範囲で見つかったすべてのデバイスが表示され、追加に必要なすべてのデバイスが表示されます。







デバイスが追加されました。次に、モニターとルールの設定に移りましょう。



「オーサリング」セクション（作成とセットアップ）に進みます。







[ルール]セクションで、新しいルールを作成します。







ナビゲートしやすくするために、新しいMPを作成します。







すべてを任意に入力できますが、作業には影響せず、コンソールの表示のみに影響します。







作成、選択します。







パフォーマンス、SNMPパフォーマンスに基づいてルールのタイプを選択します。







次のステップでは、ルールの名前、カテゴリ「その他」、およびターゲット「SNMPネットワークデバイス」を選択します。







次に、データをデバイスに問い合わせる頻度を決定します（2分間残しました）。



また、デバイス上のセンサーの3つのグループの温度インジケーターのOIDを駆動しました。



1.3.6.1.4.1.16174.1.1.3.3.3.3.0-SCOMでは、すべてのオブジェクトは最初のポイントなしで記述されていることに注意してください







完了-[作成]をクリックして、ウィザードの完了を待ちます。 したがって、各センサーに対して行う必要があります。 必要なOIDを以下にリストします。



グループ1



温度-1.3.6.1.4.1.16174.1.1.3.3.1.3.0



湿度-1.3.6.1.4.1.16174.1.1.3.3.1.6.0



グループ2



温度-1.3.6.1.4.1.16174.1.1.3.3.2.3.0



湿度-1.3.6.1.4.1.16174.1.1.3.3.2.6.0



グループ3



温度-1.3.6.1.4.1.16174.1.1.3.3.3.3.0



湿度-1.3.6.1.4.1.16174.1.1.3.3.3.6.0



グループ4

湿度-1.3.6.1.4.1.16174.1.1.3.3.4.6.0



最後に、デバイスからデータが収集されていることを確認できます。



これはすべて何らかの形式で表示されますが、最初に、デバイスのセンサーを簡単に検索するために、将来のパフォーマンス表示で表示するための選択を行うためのグループを作成します。



「オーサリング」セクション（作成とセットアップ）に進みます。







そして、新しいグループを作成します。











メンバーを明示的に指定するには、検索ウィザードを使用して、必要なデバイスをリストから選択します（他のSNMPデバイスが巻き上げられている場合）。











作成する最後のステップで、ウィザードのその他のステップをスキップします。今はそれらについてではありません。 したがって、デバイスを含むグループがあり、グループメンバーをクリックすることでデバイスが機能していることを確認できます。







彼らが見るべきものを見る。







監視モードに移動します







そして、新しいパフォーマンスビューを作成します。







次のオプションを使用します。







これによりビューが作成され、表示する時間範囲、線の色などを設定できます。







最初のタスクが完了しました。



次に、アラートをトリガーするモニターを作成します。 再び、「オーサリング」セクションに進みます。







また、SNMPネットワークデバイスの[モニター]セクションで、可用性セクションに基本的なモニターを作成します。







SNMPチェックに基づいてそれを行います。もちろん、ロシア語版の翻訳はばかげていますが、単純なルールの検知と検出に基づいてモニターを作成します。







次のステップでは、名前、親の選択、ターゲットを除いてほとんど何も行いません。







ここで、テスト対象のセンサーのOIDを示します。これは、OIDを持つ3番目のグループの温度センサーです-1.3.6.1.4.1.16174.1.1.3.3.3.3.0







パラメーターを指定する



/ DataItem / SnmpVarBinds / SnmpVarBind [1] / Value 、協力者が28未満。



つまり、撮影するパラメーターが28度以上である場合、すべてが不良です。







次に、3番目のステップと同じ手順を実行し、チェックされたOIDを示します。







繰り返しますが、2番目の式についても同じことを行い、演算子のみが28以上になりました。







次に、状態を式にマッピングします。







そして、アラートを設定します。 説明では、次の式を記述することにより、モニターがトリガーされたときの温度を示すことができます。



$データ/コンテキスト/ SnmpVarBinds / SnmpVarBind [1] /値$







それだけです これで、センサーごとに複数のモニターを作成できます。







その結果、サーバールームで起こっていることを非常に明確に表示できます。
