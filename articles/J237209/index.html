これらの面白い地域設定

今日は、地域の設定についてお話します。 しかし、最初に、小さなタスク：以下のコードは何を出力しますか？ （コードはC＃ですが、かなり一般的な問題が考慮されているため、代わりに他の言語を想像できます。） 



Console.WriteLine((-42).ToString() == "-42"); Console.WriteLine(double.NaN.ToString() == "NaN"); Console.WriteLine(int.Parse("-42") == -42); Console.WriteLine(1.1.ToString().Contains("?") == false); Console.WriteLine(new DateTime(2014, 1, 1).ToString().Contains("2014")); Console.WriteLine("i".ToUpper() == "I" || "I".ToLower() == "i");


真の値をいくつ取得しましたか？ 0より大きい場合、正解は「依存」であるため、地域の設定について詳しく知る必要はありません。 残念ながら、多くのプログラマーは、異なる環境でのこれらの設定が異なる可能性があるとはまったく考えていません。 また、このプログラマーがすべてのコードにInvariantCultureを設定するのは面倒で、その結果、美しいアプリケーションは非常に奇妙に動作し、他の国のユーザーに届きます。 。 この記事では、地域の設定によって影響を受けるいくつかの重要なポイントの簡単な選択を提供します。







ちょっとした理論：.NETでは、特定の言語と地域の設定に関するすべての情報はCultureInfoクラスを使用して見つけることができます。 以前にカルチャに遭遇したことがない場合、 この投稿は最初の慣れに適しています。 既存のさまざまな地域設定の調査に熱心な経験豊富なプログラマーは、すべてのCultureInfoを手動で表示するのにうんざりするかもしれません。 個人的には、ある時点で疲れました。 したがって、CultureInfoExplorerと呼ばれる小さなWPFアプリケーション（ GitHub 、 バイナリ へのリンク ）が表示され、上のスクリーンショットに示されています。 次のことができます。
  • このCultureInfoによると、その主要なプロパティの意味と、事前に準備された文字列パターンがどのように見えるかを見てください。
  • このプロパティについては、可能な値と、各値に対応するすべてのCultureInfoのリストを参照してください。
  • このパターンについては、変換可能なオプションの可能なオプションを参照してください。各オプションについては、関連するCultureInfoのリストも参照してください。
このプログラムが役立つ読者がいることを願っています。 さまざまな地域の設定について多くを学ぶことができます。 さて、例のために。



数字



数値の表現については、 NumberFormatInfo （ CultureInfo.NumberFormatからアクセス可能）を担当します 。 そして、これは通常の数字だけでなく、パーセンテージと金銭的価値にも言及しています。 正と負の値があることに注意してください。ローカリゼーション/グローバリゼーションを扱う場合、これに注意を払うことが重要です。 少なくともドキュメントに目を通し、利用可能なプロパティを確認することを強くお勧めします。



人に問題を引き起こす最も一般的なプロパティの1つは、 NumberDecimalSeparatorと呼ばれます 。 これは、数値をデコードするときに整数部が何から分離されるかを担当します。 エラーの典型的な例：プログラマは小数の配列を文字列にマージし、それらをコンマで区切ります。 その後、彼は文字列を解析して配列に戻します。 NumberDecimalSeparatorがポイントに等しい場合、すべてが正常です。 たとえば、en-USカルチャーが公開されたため、すべてがプログラマーのために機能し、彼は製品をリリースしました。 この製品はru-RUカルチャーを持つユーザーによってダウンロードされ、悲しみ始めます：結局、彼のNumberDecimalSeparatorはコンマに等しい：要素1.2および3.4​​の配列は、そのようなマージでストリング「1,2,3,4」にマージされ、解析するのに問題があります。 個人的には、同様の問題に遭遇したプログラマーが、フォーマット時に正しいNumberFormatInfoを示すことで通常は解決しようとせず、ドットをコンマに、またはコンマをドットに置き換えることを思いつき始めると、さらに悲しくなります。 原則として、 NumberDecimalSeparatorが任意であることを理解する必要があります。 たとえば、fa-IR（ペルシャ）文化では、スラッシュ（ '/'）と同じです。



利息と通貨についても同様のプロパティがあります： PercentDecimalSeparatorとCurrencyDecimalSeparator 。 これらの3つの値すべてが一致する必要はありません。 たとえば、カザフスタン（kk-KZ）の場合、 NumberDecimalSeparatorとPercentDecimalSeparatorはカンマに等しく、CurrencyDecimalSeparatorはマイナス記号に等しくなります（負の数値が示されるものとまったく同じです）。



整数が文字列に変換されると、数字のみで構成される値が得られると考える人もいます。 ただし、これらの数値はグループに分けることができます。 NumberGroupSizesプロパティはグループのサイズを担当し、 NumberGroupSeparatorはセパレータを担当します（パーセントと通貨は同様のプロパティを持っていますが、再び一致する必要はありません）。 グループのサイズはさまざまです。たとえば、多くのカルチャ （as-IN、bn-BD、gu-IN、hi-INなど） では 、 NumberGroupSizesは{ 3、2 }です。 as-INカルチャの番号1234567は、 「12.34.567」のようになります。 \ u0020スペースはセパレータとして使用できます（たとえば、af-ZAおよびlt-LTで）が、それを見るときは、文字列の解析とフォーマットに別の松葉杖を駆り立てないでください。 ほとんどの場合、通常のスペースの代わりに、不可解なスペースが使用されます\ u00A0（ネイティブru-RU）。



負の数と正の数を示す記号もカルチャに含まれます： NegativeSign 、 PositiveSign 。 ありがたいことに、利用可能なすべての文化において、マイナスとプラスに等しいのですが、これに依存するべきではありません。環境を再定義し、プロパティをプロパティに設定できます。 そして、最も興味深いのは、記号ではなく、正と負の値のフォーマットパターンです。 たとえば、負の数値の書式設定は、次の5つの値を持つNumberNegativePatternを使用して決定されます。 



 0 (n) 1 -n 2 - n 3 n- 4 n -


たとえば、ti-ET文化（Tigrinya（エチオピア））では、-5の値がフォーム（5）に表示されます。 パーセンテージと通貨（ PercentNegativePattern 、 PercentPositivePattern 、 CurrencyNegativePattern 、 CurrencyPositivePattern ）を使用すると、状況はさらに楽しくなります。 たとえば、 CurrencyNegativePatternには、最大16個の値があります。

分析したさまざまな有用な特性の例。 そして、ちょっとしたトリックをやってみましょう。ロシアの文化を少し変えて、新しい意味が私たちの人生を台無しにするように、投稿の最初から例を示します。 



 var myCulture = (CultureInfo)new CultureInfo("ru-RU").Clone(); myCulture.NumberFormat.NegativeSign = "!"; myCulture.NumberFormat.PositiveSign = "-"; myCulture.NumberFormat.PositiveInfinitySymbol = "+Inf"; myCulture.NumberFormat.NaNSymbol = "Not a number"; myCulture.NumberFormat.NumberDecimalSeparator = "?"; Thread.CurrentThread.CurrentCulture = myCulture; Console.WriteLine(-42); // !42 Console.WriteLine(double.NaN); // Not a number Console.WriteLine(int.Parse("-42")); // 42 Console.WriteLine(1.1); // 1?1

日時
