Microsoft SQL Server用Veeam Explorerを使用したデータベースの回復

仮想化されたSQL Serverをバックアップするためのプロセス理論と実用的なヒントを分析した後、Microsoft SQL Server用のVeeam Explorerを使用してデータベースを復元するトピックに移りましょう。 このツールを使用すると、サーバー自体をバックアップから解除することなく、データベースをソースまたは別のサーバーに復元できます。



一般に、Veeam Explorerを含むリカバリプロセスは次のように行われます。







  1. リカバリを担当する管理者は、Veeam Backup＆Replicationコンソールを起動し、お気に入りのリカバリオプション（Veeamのオファーから）を使用して、SQL Serverを（リポジトリにあるバックアップから直接）Veeamバックアップサーバーにマウントします。 SQL Serverインスタンスおよびベース。

  2. Veeam Explorer for Microsoft SQL Serverは、このSQLサーバー上のインスタンスとデータベースの階層に関する情報を収集し、コンソールにユーザーのビューを作成します。 ユーザーはツリー全体を閲覧し、必要なベースを見つけたら、必要な回復スクリプトを選択して実行できます。 次の回復およびエクスポートシナリオがサポートされています。

    • 選択した復旧ポイント（Veeamバックアップサーバーに現在マウントされている復旧ポイント）に対応する状態へ
    • 選択した時点で述べるには
    • 選択したトランザクションの前夜


    これらのシナリオの最初は、ターゲットSQL Serverのみに関係します。 残りについては、次の表に従って、ターゲットサーバーと補助SQLサーバー（ステージングサーバー）の両方が必要になります。

    サーバー ポイントインタイムリカバリ トランザクション回復 ポイントインタイムでエクスポート トランザクションごとにエクスポート
    子会社 未使用 ログを表示するために使用 使用されている ログの表示と「ロール」に使用
    対象 ロギングに使用 ロギングに使用 未使用 未使用


  3. 最後に、データベースはターゲットのMicrosoft SQL Server上で完全に「再作成」され、すぐに使用できます。 エクスポートスクリプトを使用した場合は、目的のサーバーにデータベースを接続できます。


復元する対象と方法を決定する



もちろん、回復ポリシーの要件に焦点を当てています。 目的のシナリオを選択し、実装を成功させるために必要なすべてを準備するには、チェックリストを作成して基本的な質問に答えてください。

  1. SQL回復ポイントの作成時にデータベースを復元しますか、または任意の時点（サーバーの回復ポイント間の間隔を含む）で回復の可能性を提供する必要がありますか？

    後者のバージョンでは、選択したポイントにログを「ロール」するにはログバックアップが必要になるため、サーバーバックアップジョブを構成するときに[ ログのバックアップ ]オプションを選択する必要があります。

  2. データベースを操作（たとえば、テーブルからレコードを削除する）の前の状態に復元する予定ですか？

    このレベルの粒度で回復するには、次のものが必要です。

    • トランザクションログのバックアップ
    • 補助SQLサーバー（要件については後で説明します）。操作のリストをコンソールに表示して、ユーザーが操作を表示して目的の操作を選択し、ログをロールできるようにします。


  3. データベースは元のMicrosoft SQL Serverに復元されますか？

    • Veeamバックアップサーバーとこのサーバーの接続設定を確認してください。
    • リカバリを実行する予定のアカウントに必要なすべての権限があることを確認してください。
    • 復元するSQL Serverのバージョンを確認します（新しいバージョンのサーバーから古いサーバーへのデータベースの復元はサポートされません）。


  4. 誰がデータベースの回復に関与しますか？ あなたは個人ですか、それとも他のユーザー（ユーザーグループ）ですか？

    2番目のケースでは、このためにEnterprise Managerの機能を使用して、回復を委任することが合理的です。



回復プロセス中は、サーバー自体を完全に復元して起動する必要はありません。 代わりに、バックアップコピー（リポジトリに保存されている形式で、つまり圧縮および重複排除後）がVeeamバックアップサーバーにマウントされます。 この仮想マシンのファイルシステムを正しくマウントし、Veeam Backup Browserブラウザーウィンドウに通常の形式で表示するために、特別なVeeamドライバーが使用されます。

次に、データベースを復元するツールを選択できます。



  1. Veeam Explorer for Microsoft SQL Server-データベースを元のサーバーまたは別のサーバー（商用ライセンスが必要）に、サーバー自体の選択されたバックアップ/指定された時間/選択されたトランザクションの前夜に対応する状態に復元できます。 このツールとスクリプトについては、以下で詳しく説明します。
  2. Veeam Backup Enterprise Manager-リカバリオプションを他のユーザーに委任して、特定のサーバーで作業する権利を付与できます（商用のEnterprise Plusライセンスも必要です）。 選択したサーバーのバックアップ/指定した時間に対応する状態へのソースまたは別のサーバーへのリカバリをサポートします。
  3. 特に熱心な人は、 個別リカバリウィザード（U-AIR）を使用して、バックアップ管理者への要求を作成し、選択したSQLサーバーバックアップ時の状態にデータベース、テーブル、またはクエリ結果を復元できます。 この方法では仮想ラボを使用し、さらに微調整が必​​要であり、ツール自体はリクエストに応じて発行されるため、この場合はサポートサービスに連絡することをお勧めします。 まだバージョン7を使用している場合は、 ドキュメントを読むことができます 。




そこで、Veeam Explorer for Microsoft SQL Serverを使用してデータベースを復元することに焦点を当てましょう。



Veeam Explorerを起動する前に何をする必要がありますか？



5つの簡単な準備手順を実行するだけです。 特に必要な場合に時間と神経を節約するのに役立つ小さなチェックリストを作成することをお勧めします。



手順1：ポートが開いているかどうかを確認する



VeeamバックアップサーバーをSQLサーバーのゲストOSに接続するには、ほとんどの場合、バックアップサーバーのTCP 1433ポートとTCP 1434ポートを開く必要があります残り（必要な場合）は、SQL Serverの構成によって異なります。 詳細については、 http：//msdn.microsoft.com/en-us/library/cc646023.aspx#BKMK_ssdeへようこそ。



ステップ2：セカンダリサーバーを準備する（必要な場合）



選択したSQLサーバーのバックアップに対応する状態にデータベースを復元する必要がある場合、追加のマシンは必要ありません。 ただし、別のシナリオを選択する場合は、次の操作を提供するステージングサーバーが必要になります（上記の表も参照）。

  • 選択したトランザクションの前夜の状態にデータベースを復元する
  • 選択された時点または選択されたトランザクションの前夜にあった状態でのデータベース（.MDFおよび.LDFファイル）のエクスポート


デフォルトでは、Veeamバックアップサーバーと共にインストールされたSQL Serverが補助として使用されます。 別のサーバーを指定するには、Veeam Explorer for SQLを起動し（たとえば、 [スタート ]メニューから）、メインメニューから[ オプション ]を選択します。



重要！ ここでの主なことは、SQL Serverのローカルインスタンスを指定し（つまり、Veeam Explorerおよびバックアップサーバーと同じマシン上に存在する）、そのバージョンがベースを復元する予定のSQLのバージョン（つまり、 E. SQL Server 2008を使用して、SQL Server 2014でデータベースを復元/エクスポートすることはできません。



ステップ3：アカウントの権利を割り当てる



プロセスのさまざまな段階で、次の権限が必要になる場合があります（必要に応じて、データベース管理者に連絡して発行してください）。

  1. SQL ServerのゲストOS（バックアップジョブをセットアップするときにGuest Processingステップで指定したもの）へのアクセスに使用する予定のアカウントは、このSQL Serverのsysadmin固定サーバーロールのメンバーである必要があります（そうでない場合、データベースの内容はVeeam Explorerウィンドウに表示されません）。 Veeam Knowledge Baseの記事も参照してください。
  2. Veeam Explorerを実行する予定のアカウント-原則として、これはVeeam Backup Serviceが実行されるアカウントであり、その要件はここで説明されています -データベースエクスポートシナリオでは、移動先のフォルダーに対する権限が必要ですデータベースファイルのエクスポート（少なくともReadおよびWrite ）
  3. データベース（元のサーバーまたは別のサーバー）を復元するためにターゲットSQL Serverへの接続が実行されるアカウントは、ターゲットSQL Serverのsysadmin固定サーバーロールのメンバーである必要があります。
  4. また、WindowsサーバーOSにアクセスするアカウントが必要になる場合があります。このアカウントでは、トランザクションログのバックアップが後続の「ロールアップ」のためにコピーされます（シナリオに応じて、ターゲットサーバーまたはセカンダリサーバーになります）。 このアカウントには、指定されたサーバー上の管理フォルダーへのアクセス権が必要です（少なくとも読み取りおよび書き込み権限が必要です）。


ステップ4：操作の制限と機能を調べる



  1. 現在のバージョンでは、リモートマシン上のSQL Serverインスタンスはセカンダリサーバーとしてサポートされていません。 ローカルインスタンスのみを使用します。
  2. SQL Server Expressを補助として使用する場合（たとえば、Veeamで提供されるSQL Express 2012サーバー）、サポートされるデータベースのサイズに制限があることに注意してください-10 GBです。 詳細はこちらをご覧ください。
  3. デフォルトでは、システムデータベース（マスターなど）は処理から除外されます。 それらを復元する必要がある場合は、たとえば、ゲストファイルを復元する手順を使用できます（ ここで説明します ）。
  4. 現在のバージョンでは、レプリカから選択した特定時点へのリカバリはサポートされていません。
  5. 暗号化されたデータベースを復元する必要がある場合は、Veeam Knowledge Baseのこの記事をご覧ください 。




ステップ5：回復のためのファイルの準備



サポートされているVeeam Backup＆Replicationのリカバリオプションを使用できます。 SQL復元ウィザード（ ここで説明）を使用すると最も便利です。自動モードで必要な操作を実行し、Veeam Explorer自体を起動します。 必要に応じて-たとえば、VeeamZIPで作業している場合-同様の操作を手動で実行できます（ ここで説明します ）-最後のポイントは、Add Databaseコマンドを使用して必要なデータベースをVeeam Explorerコンソールに追加することです（詳細はこちらを参照）。



したがって、データベース回復スクリプト





重要！ Veeam Backup＆Replication EnterpriseおよびEnterprise Plusエディションでは、SQLサーバーでのデータベースリカバリ（[ 復元 ]メニューコマンド）がサポートされています。 すべてのエディション（無料版と標準版を含む）では、データベースファイルをエクスポートしてから、目的のサーバーに添付することで復元できます。



シナリオ番号1。 「選択したバックアップSQLサーバーのように」状態に復元する



アプリケーションを念頭に作成されたリカバリポイント（バックアップ、レプリカ、VeeamZIP）が必要です。 必要な構成設定：

  1. データベースの復旧モデル-任意の可能性があります。
  2. ゲスト処理ステップのバックアップタスクの設定で、[ アプリケーション対応の画像処理を有効にする]チェックボックスを選択する必要があります。







  3. [ 全般 ]タブの仮想マシンを処理するための設定ダイアログで、次のオプションを選択する必要があります。

    • [ アプリケーション]セクション- 正常な処理が必要です（推奨）
    • セクショントランザクションログ - このジョブでトランザクションログを処理する（推奨）







    • [ SQL ]タブの同じダイアログで、任意のログ処理オプションを選択できます（もちろん、この場合に何が起こるかを理解した上で）。




注：これらの設定の説明を忘れたり見逃した人のために-それらは最初のシリーズで説明されました 。



このシナリオ（Veeam Backup＆Replication EnterpriseまたはEnterprise Plusライセンスを持っている場合）を使用すると、ソースサーバーまたは選択した別のサーバーにデータベースを復元できます。



オプション1：移行元サーバーに復元する



このためには、1クリック復元クイックリカバリを使用すると便利です。これは、次のデフォルト設定で行われます。

  1. データベースは「選択したバックアップSQLサーバーのように」状態に復元されます
  2. データベースファイルは、ソースサーバーにコピーされ、ソースインスタンス（バックアップされ、現在運用中のインスタンス）にアタッチされます。
  3. このサーバーに接続するには、Veeam Explorerを実行するアカウントとWindows認証が使用されます。 このアカウントに十分な権限がない場合、Veeamはゲスト処理ステップでバックアップジョブを構成するときに指定したアカウントを使用しようとします。 動作しない場合は、ユーザー名とパスワードの入力要求が続きます（管理者権限を確認してください！）


これらの事実に気付いたので、Veeam Explorerコンソールのツリーから必要なベースを選択し、メニューからコマンドデータベースの復元> current_state_dateをサーバー\インスタンスに復元を選択します 。



注：ここで、 current_state_dateはSQLサーバーのバックアップが作成された日付であり、その内容は現在Veeamサーバーにマウントされ、Veeam Explorerを通じて表示されます。 したがって、「最新」-最後のバックアップが選択された場合、または対応する日付があります。







有用：ツリーで適切なノード（インスタンスまたはサーバー）を選択することにより、複数のデータベース（1つのインスタンスのすべてのデータベースまたは1つのサーバーのすべてのデータベース）を選択できます。



オプション2：選択したサーバーに復元する



（ソースではなく）選択したサーバーに復元するには、ウィザードの手順を実行する必要があります。

  1. [ 復元ポイントの指定 ]ステップで、[ 現在の復元ポイントに復元する ]オプションを選択します。 「SQLサーバーの選択されたバックアップのように」状態への復元：







  2. 次に、 ターゲットSQL Server接続パラメーターの指定手順で、必要なすべてのものを指定します。データベースを復元するサーバーの名前、データベースの名前、ターゲットサーバーに接続するアカウント。 このアカウントに必要な権限があることを確認することを忘れないでください（実際には、上記の準備手順のリストの手順3でした:-)）。







    いくつかの説明的な言葉：

    • 特定のインスタンスに復元する場合は、 サーバー\インスタンス形式で指定します。
    • 指定されたターゲットサーバーのバージョンは元のバージョンよりも低くない必要があります（つまり、SQL 2014データベースをSQL 2008に復元しないでください）。
    • （SQLの以前のバージョンの場合）：ターゲットデータベースがミラーモードではないことを確認します（このような回復はサポートされていません）
    • ターゲットデータベースの名前を指定しない場合、元の名前と同じ名前が（バックアップから）使用されます。




    重要！ この名前のデータベースがターゲットサーバーに存在する場合、デフォルトで削除され、バックアップから復元されたデータベースに置き換えられます。このオプションでは、[ 次へ ]をクリックした後、このアクションを確認する必要があります（キャンセルして新しいデータベース名を入力します）。



  3. 指定したターゲットサーバーがAlwaysOn可用性グループをサポートしている場合は、[ 常にONの復元オプションを指定する]ステップで、復元したデータベースを含めるグループを選択できます。







  4. 次に、ターゲットサーバーのゲストOSをノックする必要があるアカウントを指定します（特に、ベースファイルをそこにコピーするため）。







  5. 次に、これらのファイル（.MDFおよび.LDF）を正確に配置する場所を指定します。







  6. 最後に、[ 復元]ボタンをクリックして、レポートがプロセスを完了するのを待ちます。




シナリオ番号2。 特定の時点への回復



このシナリオを使用すると、データベースを選択した時点（サーバー自体のバックアップコピーを作成する時点とは異なる場合があります）に復元できます。 実際、データベースは「サーバーの最も近い以前のバックアップ」の状態に復元され、ログはバックアップから「ロール」されます。そのため、データベースは目的の状態になります。 したがって、このシナリオを実装するには、次の構成設定が必要です。

  1. SQL Server復旧モデルは、 FullまたはBulk-loggedである必要があります 。
  2. SQL Server復旧ポイントは、アプリケーションの操作を考慮して作成する必要があります（ アプリケーション対応の画像処理を有効にする）。
  3. [ 全般 ]タブの仮想マシン処理設定ダイアログで、次を選択します。

    • [ アプリケーション]セクション- 正常な処理が必要です（推奨）
    • セクショントランザクションログ - このジョブでトランザクションログを処理する（推奨）


  4. トランザクションログを処理するための設定で、バックアップログを選択する必要があります。




注：現在のバージョンでは、このスクリプトは、アーカイブジョブを使用してリポジトリに保存されたレプリカおよびバックアップではサポートされていません。



このシナリオでは、ソースまたは選択したサーバーへのリカバリもサポートされています。 復元ウィザードの対応するステップにアクセスするには、 [データベースの復元]> [特定の時点の状態を 復元]を選択するか、メニューから[ 別のサーバーに復元]を選択します。







ここで、特定の時点への復元オプションを選択し、スライダー（「スライダー」）を使用して、データベースを復元する特定の時点を指定する必要があります。



シナリオ3.選択したトランザクションの前の状態への回復



ここで、ユーザーは選択された期間の操作のリストを表示し、目的の操作を見つけて（たとえば、テーブルからレコードを削除する）、データベースを操作前の状態に復元できます（操作の完全なリストはここにあります ）。

このシナリオを実装するには、次の構成設定が必要です。

  1. SQL Server復旧モデルは、 FullまたはBulk-loggedである必要があります 。
  2. SQL Server復旧ポイントは、アプリケーションの操作を考慮して作成する必要があります（ アプリケーション対応の画像処理を有効にする）。
  3. [ 全般 ]タブの仮想マシンを処理するための設定ダイアログで、次のオプションを選択する必要があります。

    • [ アプリケーション]セクション- 正常な処理が必要です（推奨）
    • セクショントランザクションログ - このジョブでトランザクションログを処理する（推奨）


  4. トランザクションログを処理するための設定で、バックアップログを選択する必要があります。
  5. 補助サーバーが必要です-その助けを借りて、トランザクションログのバックアップからの操作のリストが表示されます。
  6. データベースがAlwaysOn可用性グループの一部である場合、このグループのすべてのノードが同じタイムゾーンにあることを確認します。


注：リカバリーがグループの2次ノードに対して実行された場合、 「指定されたSTOPAT時間は早すぎます」などのエラーが発生する場合があります。 この場合、プライマリノードからリカバリを実行する必要があります。
