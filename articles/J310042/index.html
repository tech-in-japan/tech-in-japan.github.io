フルスタックJS開発に関するシリーズの2番目の記事。







JavaScriptは絶えず変化しています。6か月前のベストプラクティスは時代遅れであるため、最新のテクノロジーに対応するのは非常に困難です。 このようなステートメントは多くの点で真実ですが、これはクライアント側のJavaScriptにより当てはまることに注意してください。 サーバーの場合、すべてがはるかに安定して徹底しています。

JSの癖



JavaScriptはその癖で知られています。 たとえば、有名な0.1 + 0.2は0.3に等しくありません。これは、JavaScriptには10進数型が組み込まれていないためです。 







console.log(0.1 + 0.2 === 0.3); //false


または、変数の値が文字列であることを正しく確認するには、次の構成が必要です。 







  if (typeof myVar === 'string' || myVar instanceof String) { console.log('That is string!'); }


これらの問題の一部は、 ES6バージョンの言語で修正されています。 たとえば、 varを介して宣言された変数のスコープは関数全体です（他のほとんどの言語では、変数にはブロックスコープがあります）。 ES6では、この問題はlet / constを介して変数を宣言することで解決され、 varの使用はまったく推奨されていません。 ただし、下位互換性のためにJSには（永久に）残ります。







サードパーティのライブラリは、JS言語の他の欠陥に対処するために使用されます。 たとえば、正確な10進数の計算には、「big.js」を使用できます。 多くの小さなパッケージがあり、それぞれが同様の問題を解決していますが、名前を覚えてプロジェクトに接続するのは非常に困難です。 したがって、 「lodash」などの汎用ソリューションを使用すると、追加のユーティリティ機能のセット全体を一度に提供する方が便利です。







lodashのWebサイトには優れたドキュメントと例があります。すべてを一度に見る必要はありません。すぐに知り、必要なものだけを将来的に見ることができます。







たとえば、変数に文字列値があることを確認するのは、lodashを使用すると簡単です。 







  if (_.isString(myVar)) { console.log('That is string!'); }


lodashは、関数チェーンの呼び出しをサポートしていることに注意してください。たとえば： 







  let arr = [2, 3, 1, 5, 88, 7, 13]; let oddSquares = _(arr) .filter(x => x % 2 === 1) //filter odd numbers .sort((a,b) => a > b) //sort by default sorts lexicographically: [1, 13, 3, 7] .map(x => x*x); console.log(oddSquares.join(',')); //1,9,25,49,169


サーバー上のTypeScript



JSコードが複雑になるほど、TypeScriptを使用するメリットが増えます。 複雑なサーバー側コードの場合、TSの利点はすぐに明らかになります。







Contosoでは、サーバーのTSコンパイル設定はtsconfig.jsonファイルにあります。







以下の点に注意してください。







  • rootDir / outDir ： rootDirのすべてのコード（すべてのTSファイル）は、outputDirでJSファイルにコンパイルされますが、フォルダー構造は保持されます。 たとえば、ファイル「/server/helpers/emailHelper.ts」は「/build/server/helpers/emailHelper.js」にコンパイルされます







  • sourceMap ：このフラグが設定されている場合、JSに加えて、ソースマップファイルが作成されます（myModule.ts-> myModule.js + myModule.js.map）。 これにより、TSファイルをデバッグでき、ブレークポイントが設定されてtsファイルで機能し、JSコードが実際に実行されます。







  • target ：TSを使用すると、JSのさまざまなバージョンのコードをコンパイルできます。 ただし、ES6の最新バージョンにコンパイルする場合にのみサポートされる機能がいくつかあります。 それらの1つはasync / awaitです。これについては以下で説明します。 サーバーで使用するために、コードはES6でコンパイルされ、6xバージョンのNodeが使用されます。


TypeScriptをグローバルにインストールしている場合は、プロジェクトルートからtsc（TypeScript Compiler）コンソールコマンドを使用して、オプションでwatchパラメーターを使用してプロジェクトをコンパイルできます。 監視モードでは、ソースコードの変更後、再コンパイルが実行されます 







 tsc --watch


アプリケーションを起動するには、「ノードを介して」「build / server / startServer.js」を実行する必要があります。







TSを使用したくない場合は、TSコードをJSに簡単に変換できます。 主なこと：







  • ES6インポートをノードに変換する必要があります
  • 型注釈を削除する
  • asyn / awaitの代わりにpromiseを使用するか、翻訳のためにbabelを接続します


JSの非同期



JSでの非同期（非同期）プログラミングは、他のプログラミング言語から切り替えるときに最も難しい瞬間の1つです。 私は非常に簡単な概要を示します、あなたはリソースのリストでもっと読むことができます。







ノードのパフォーマンスが高いのは、長時間実行されるすべての操作がメインプロセスをブロックしないためです。 このような操作の例としては、データベースでのクエリやファイルへの情報の書き込みがあります。 操作が呼び出された後、別のコードの実行が継続されます。 操作が（将来）正常に完了した場合、または何らかのエラーで完了した場合、次に何をするかを指定する必要があります。







複数のテンプレートを使用して、非同期コードを操作できます。 これらのパターンは、Nodeの開発とともに変化（進化）しました。







  • コールバックは、Nodeで非同期を整理する従来の方法です。 理解は簡単ですが、いくつかの問題があります。複数のネストされた非同期呼び出しを行うことは困難です。 この場合、コードの記述、保守、および理解が困難になります（コールバック地獄）。 データの戻り、エラー処理、生成は、同期JSコードでこれが発生する方法とは根本的に異なります。







  • Promises（約束/約束）-JSでの非同期の最新の標準。 現在、広く使用されています。 Promisは、保留中/拒否/解決済みの3つの状態のいずれかになります。 最初は、期待の状態ですが、将来的には、約束は、エラーまたは実行の成功（結果を伴う）の最終状態のいずれかに移行することを約束（約束）します。 Promiseコンストラクトを使用すると、非同期呼び出しチェーンをより線形に整理できるため、サポートが容易になり、コードの理解が向上します。 この場合、データはreturn経由で返され、エラーはthrow経由で生成され、1つのエラーハンドラーを複数の非同期操作に使用できます。







  • async / awaitは、JSでの非同期の未来です。 現在、stage4の議論の最終段階にあり、その後、JSに新たに承認された追加となります。 Async / awiatは、同期コードと非同期コードの間の線をさらに消去し、それらが調和してブレンドされるようにします。 async / awaitを使用したコードは、記述および保守が可能な限り簡単です。 C＃やPythonなど、他の言語にも同様の構文があります。 async / awaitの短所は、TypeScriptやBabelなど、追加のツールを使用する必要があることです。


さらにいくつかの非同期テンプレート、非同期モジュール、ジェネレーターを使用するcoモジュールがあります。 これらは、promisesおよびasync / awaitの前の中間的な進化段階です。 現在、「co」は追加の変換なしで使用できますが、私の意見では、async / awaitを使用できない場合はpromiseを使用することをお勧めします。







JSで正常に動作するには、基本的な非同期テンプレートを知っている必要があります。 時々、それらのそれぞれに対処する必要があります。 たとえば、async / awaitを使用する場合、Promise.allを介した並列実行などの一部の操作については、Promiseが必要です。Promiseを使用する場合は、Promiseでコールバックの関数をラップする必要がある場合があります。







エラー処理



これは、他のプラットフォームからNodeへの移行を困難にする別の領域です。 まず、エラーを処理する方法は、使用する非同期パターンによって異なります。 コールバックを使用する場合、エラーはコールバック関数の最初のパラメーターとして送信されます。Promiseを使用する場合は、Promiseチェーン内のcatchを介してエラーを処理する必要があります。コードを同期的に呼び出す場合、async / awaitを使用する場合は、try / catchを使用する必要があります







次に、JavaScriptでは、オブジェクトを渡すことで（スローを介して）エラーを生成できることを知っておく必要がありますが、ベストプラクティスは組み込みのErrorオブジェクトを使用することです。 最も単純な形式では、次のようになります。 







  throw new Error('Param value is wrong');


この場合、組み込みのエラーオブジェクトから継承することにより、カスタムエラーオブジェクトを作成できます。 







  function AppError(errorCode, options) { this.code = errorCode; Error.captureStackTrace(this, this.constructor); //merge data props into error object return Object.assign(this, options); } Object.setPrototypeOf(AppError.prototype, Error.prototype); ... throw new AppError('user_not_found', {userId: 5});


Error.captureStackTrace呼び出しに注意してください。これは、スタックトレースをエラーオブジェクトに正しく追加するために必要です。 データをエラーに追加し、エラーコンストラクターの署名を定義できます。







Contosoでは 、AppErrorオブジェクトは既定で文字列パラメーターを受け入れます。エラーの種類とエラーコード、エラーテキストは外部ファイルのコードによって読み取られます。







別のポイント：未処理エラーをキャッチしてログに記録する必要があります。これを行わないと、アプリケーションは理由を明確に示すことなくシャットダウンします。未処理エラーのハンドラーをできるだけ早く接続する必要があります。 







  process.on('uncaughtException', (err) => { console.log(`Uncaught exception. ${err}`); });


バックエンドWebフレームワークの選択



Expressが主な選択肢であり、すべての選択肢は10倍人気があります。 Expressは、ミニマリズムと柔軟性の哲学に従います。 このコンテキストでのミニマリズムとは、追加の構成を行わないと、使用できる機能がほとんどないことを意味します。 たとえば、Cookieのサポートを追加し、HTTP要求の本文を解析し、クライアントのセッションにアクセスするには、適切なモジュールを追加する必要があります。 使用される各機会は直接アクティブ化する必要があります。 柔軟性とは、既存の機能を変更/追加する多くの可能性があることを意味します。







Expressのこれらの機能により、Webアプリケーションと個々のAPIサーバーの両方に最適な選択肢となります。







他のオプションは、完全に独立したフレームワークとExpressベースのフレームワークの2つのタイプに分類できます。







別に：



  • Koaは、Expressの作成者による新しいフレームワークです。 一方で、これはExpressの改良版であり、いくつかの点が優れています。他方では、基本的には同じであり、Expressとの下位互換性はありません。







  • Hapiは完全に独立したフレームワークで、元々は米国最大のスーパーマーケットチェーンであるWallmartによって作成され、財政的にサポートされています。 Expressとは異なり、必要なカスタマイズは少なく、すぐに使用できる機能が増えます。







  • Meteor-サーバー部分とクライアント部分の両方を開発する機会を提供し、開発者に必要なすべてを一度に提供しようとするため、サードパーティのテクノロジーと統合するのはより困難です。


Expressに基づく：



  • ループバック-APIサーバー、特にモバイルデバイス向けに作成されたフレームワーク。iOS/ Android用のDSKがあり、組み込みORMがさまざまなデータベースの操作に使用されます。







  • Sails-特にsocket.ioを介したリアルタイムの相互作用がある場合、サーバーAPIの優れた基盤として位置付けられます。







  • FeathersJs-このフレームワークには他のパッケージ/フレームワーク用の多くのアダプターがあり、目標は標準化を追加し、開発プロセスを容易にすることです。


何を選ぶか



Expressから始めます。時間と希望がある場合は、他のオプションを参照してください。 個人的に、私は他のフレームワークに対処するための十分な動機を持っていませんでした。なぜなら、私は完全に自分に合ったExpressベースの構造を迅速に構築できたからです。 このアプローチの利点は、最初はより多くの作業が必要ですが、フレームワークの詳細への依存が少ないことです。







プロジェクト構造



Contosoプロジェクトの構造は、MVCアーキテクチャに基づいています 。







コントローラー -ルートハンドラー、標準のエクスプレス要求および応答パラメーターを受け入れる関数を含むモジュール。 コントローラーにはデータ検証と基本的なリクエスト処理のロジックがありますが、このリポジトリーとヘルパーが使用されるため、データベースに直接アクセスせず、ルーチン操作を実行しません。







リポジトリ -これはデータアクセスロジック（DB）と追加のビジネスロジックです。ビジネスロジックの一部をコントローラに含めることができますが、これにはリポジトリが望ましいです。







ヘルパー （ヘルパー）-特定の操作を実行-電子メールの送信、エラー処理、ロギングなど







ルーター -ルート（URL）とコントローラーからの対応するハンドラーとの間の通信を記録します。 この場合、APIメソッドのルートとWebページ（ビュー）のルートの間に大きな違いはありません。







タスクは、初期データベースの作成やファイルからのデータのインポートなどのタスクのためのユーティリティスクリプトです。







ビュー —サーバービュー（HTMLテンプレート）は、一般的なレイアウトビューと部分ビューをサポートしています。 Contosoでは、サーバービューは認証ページに使用され、残りのHTMLビューはクライアントで生成されます。







古典的なMVCアーキテクチャでは、モデルは個別に定義されます。ここで、モデルは、コントローラーで作成され、ビューを生成するために使用されるデータを持つオブジェクトにすぎません。







Expressは柔軟なフレームワークであり、必要に応じてコードを整理できます。 この場合、開発者は自分に合った構造を決定します。 単一の正しい構造はありませんが、原則としては間違いなく良いです。







Expressおよびサードパーティパッケージの多くの機能は、ラッパーモジュールを介して使用されます。 これは、将来パッケージを簡単に置き換えたり、標準機能を再定義したりできるため、便利です。







構成



通常、構成はプロジェクトサブディレクトリの物理ファイルに保存されます。 Nodeでは、他のプラットフォームとは異なり、JSONはXMLよりも頻繁に使用されます。







構成値を操作するためのパッケージがいくつかありますが、「config」の方が好まれます。別の一般的なオプションは「nconf」です。







「config」では、特定のルールに従って複数の構成ファイルからデータが読み取られます。 デフォルト値はファイル「default.json」に保存され、ファイル「local.json」の値で上書きできます。 JSON形式に加えて、yaml / xml / hjson / jsなど、他の多くの形式を使用して構成を保存できます。 詳細はこちらをご覧ください 。







デフォルト設定ファイル「default.json」がリポジトリに追加され、それらを再定義できるファイル「local.json」は、.gitignoreを介してリポジトリから除外されます。







構成を保存するには、いくつかの戦略を使用できます。







  • デフォルトの開発者設定 -構成（デフォルトファイル）には、ローカル開発に適した設定が保存されます（ローカルファイルでオーバーライドできます）。 運用環境では、既定のファイルである運用設定（ローカルファイルまたはサーバー名を持つファイル）の既定の設定をオーバーライドする必要があります。
  • デフォルトのプロダクション設定 -逆に、デフォルトの構成ではプロダクション設定が保存され、開発者は開発用の設定をローカルでオーバーライドします。


最初のオプションが望ましい-最初に、本番用の設定をリポジトリに追加することはセキュリティの点であまり正しくありません。そして、第2に、異なる設定値を必要とする本番/ uat /テスト展開の環境がしばしばあります セキュリティを強化するために、プロダクションではデフォルトファイルをまったく使用せずに、ローカルファイルの構成値を完全に設定することができます（詳細については、展開の記事を参照してください）。







Contosoでは、構成はラッパーモジュール 'server / config'を介して使用されます。







ロギング



単純なデバッグのために、私はconsole.logを使用していますが、ログまたは何らかの補助情報にエラーを表示する必要がある場合は、ロギングライブラリの1つを使用する必要があります。







いくつかの一般的なロギングパッケージがあります：winston、bunyan、およびlog4js。 私は「良いものから良いものを求めない」という原則で最も人気のある「ウィンストン」を使用し、問題なくウィンストンを統合することができました。そこには必要なものがすべてあります。







一番好きなライブラリを比較して選択できます。 追加リソースには、ウィンストンとバンヤンを比較する記事があります。







Contosoは、loggerHelperラッパーを介したログ記録を使用します。 ログはエラーの「/ data / logs」ログに保存され、診断メッセージのログは異なるファイルに書き込まれます。







データ検証



外部からアプリケーションに入力するデータの正確性を確認する必要があります。 単純なWebアプリケーションの場合、着信データの主なソースはWebリクエストです：HTMLフォームデータまたはクライアントAJAXリクエスト。







さまざまなレベルでデータを検証できます; Contosoでは、検証はコントローラーレベルで実行され、正しいデータが既にリポジトリに到着していると想定されます。 より複雑なアプリケーションでは、データベースの状態が変更される直前に、リポジトリ内のデータを追加で確認することが理にかなっている場合があります。







着信データをより効果的に検証するには、 「joi」パッケージを使用できます。 これは、独立して動作するhapiプラグイン（表現の代替）です。







データは、リクエスト本文（req.body）またはクエリ文字列パラメーター（req.query）でコントローラーに送られます。 これは、任意のJSオブジェクトであり、その中で何でも使用できます。 しかし、特定のタイプのパラメーター、有効な値のセットをそれぞれ持つ特定のパラメーターセットを取得することを期待しています。







データの正確性を確認するには、予想されるデータを記述するJoiスキームを宣言し、コンプライアンスチェックを実行する必要があります。 







  let schema = { id: Joi.number(), number: Joi.number().required(), title: Joi.string().required() }; let obj = { number: 8, title: 'Gone with the Wind' }; Joi.validate(obj, schema, {/*options*/}, (err, val) => { //... });


Contosoでは 、controllerHelperのloadSchemaメソッドを呼び出すことで Joiが使用されます 。 このメソッドは、Joi.validateをpromiseでラップし、アプリケーションが理解する検証エラーを生成します。







メール送信



電子メールを送信するために、私は人気があり、よくサポートされている「nodemailer」を使用します。







さまざまな転送モード（電子メールの転送方法）を使用できます。 デフォルトでは、ダイレクトモードが使用されます。







このモードは、個別のSMTPサーバーや、Amazon SESやSendGridなどのサードパーティのメールサービスを必要としないため、プロトタイププロジェクトでの作業に適しています。







欠点は、IPアドレスによっては、スパムディレクトリに文字が表示される場合があることです。







ここで輸送の様々なモードの詳細。







HTML形式の電子メールのコンテンツを生成するには、「handlebars」テンプレートを含む「email-templates」パッケージが使用されます。 Contosoでは、電子メールは「/ helpers / emailHelper」を介して送信され、電子メールテンプレートは「/ data / emails」に保存されます。
