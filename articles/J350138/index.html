息を詰まらせるのではなく、Javaアプリケーションを食べるクジラ

こんにちは、愛するhabravchane！ 今日は、Javaアプリケーションをdockerに「フィード」する方法、より良い行動の仕方、してはいけないことについてお話したいと思います。 私はJavaで10年以上開発しており、過去3年間はDockerとの最も近いコミュニケーションに費やしてきたので、Javaでできることとできないことを確信していました。 しかし、仮説は実際にテストする必要がありますよね？



このプロセス全体を、温かいチューブピクセルアートを使用した古き良きコンピューターゲームとして紹介しました。



どのゲームにもふさわしいように、ブリーフィングから始めます。 はじめに、Dockerの広告を見てみましょう。



Dockerの Webサイトでの多くの広告の約束を熟知できます。つまり、開発と展開の速度を13倍に高め、開発の移植性を高めることを約束します（特に、「私のマシンで機能する」秘跡を取り除きます）。 しかし、これは本当ですか？



今、これらの声明を証明/反論しようとします。



レベル1



ゲームに参加しているので、予想どおり、最も単純なレベルから始めます。



最初のレベルでの私たちの使命は何ですか？ おそらく多くの人にとって、これは非常に簡単で理解しやすいものです。最も原始的なJavaアプリケーションであるDockerで「ラップ」する必要があります。







これを行うには、秘跡的なHello JavaMeetup！を出力する単純なJavaクラスが必要です。 また、 Dockerイメージを作成するには、 Dockerfileが必要です 。 構文に関しては、非常に簡単です-基本イメージとしてjava：8を使用し、Javaクラスを追加し（ ADDコマンド）、それをコンパイルし（ RUNコマンドを使用）、コンテナーの起動時に実行されるコマンドを指定します（ CMDコマンド）。



HelloWorld.java 



public class HelloWorld { public static void main (String[] args) { System.out.println("Hello World!"); } }


Dockerfile 



 FROM java:8 ADD HelloWorld.java . RUN javac HelloWorld.java CMD ["java", "HelloWorld"]


Dockerコマンド： 



 $ docker build -t java-app:demo . $ docker images $ docker run java-app:demo








このビジネスをすべて収集するには、実際には1つのコマンドが必要です。これはdocker buildです。 組み立てるときに、画像の名前とそれに割り当てるタグを指定します（このようにして、アプリケーションのさまざまなアセンブリをバージョン管理できます）。 次に、 docker imagesコマンドを実行して、イメージを組み立てたことを確認します。 アプリケーションを実行するには、 docker runコマンドを実行します 。



やれやれ、すべてが完璧に行きました。







はい、ミッションを完了しました。 しかし、私たちと意見を交わす理由があります。 なぜ、あなたは尋ねます、そしてそのような間違いを次回回避する方法？



  • 使用した基本的なdockerイメージは非推奨と宣言されており、dockerコミュニティではサポートされていません。 DockerCon17でさえ、Java EEの世界でよく知られている多くのArun Guptaは、openjdkをベースイメージとして使用することを推奨しています（これについては、イメージの説明と更新日付https://hub.docker.com/_/openjdk/でも示唆されています ）。
  • サイズを小さくするには、Alpineに基づく画像を使用することをお勧めします。この分布に基づく画像は最も軽量です。
  • jdkイメージを使用してコンパイルし、jreの使用を開始します（ディスクスペースを節約しますが、まだ必要です）。


これで、最初のレベルが正常に完了したと想定できます。 2番目に上昇します。



最初のレベルを渡すための便利なリンク



レベル2







Javaを扱う場合、MavenまたはGradleを使用する可能性が最も高くなります。 したがって、プロジェクトとドッカーイメージをビルドするための単一の環境を得るために、ビルドシステムをDockerと何らかの形で統合すると便利です。



幸いなことに、ほとんどのプラグインはすでに作成されています-MavenとGradleの両方。







Docker fabric8ioおよびspotify用の最も人気のあるMavenプラグイン。 Gradleの場合は、プラグインBenjamin Mushkoを使用できます。これは、Gradleの開発者であり、「Gradle in Action」という本の著者です。



Dockerをアプリケーションのビルドシステムに接続するには、gradle構成で、コンテナーを収集して実行するいくつかのタスクを作成するだけでなく、カテゴリからいくつかの一般情報を示します。ベースイメージとして使用するイメージと、組み立てたイメージに付ける名前



冗長にならないようにしましょう。bmuschko/ gradle-docker-pluginとGradle プラグインを使用してください （MavenファンとXML愛好家の皆さん、少々お待ちください！）。



最初のタスクを完了しますが、今はこのプラグインを使用します。 build.gradleの必要な主要部分： 



 docker { javaApplication { baseImage = 'openjdk:latest' tag = 'java-app:gradle' } } task createContainer(type: DockerCreateContainer) { dependsOn dockerBuildImage targetImageId { dockerBuildImage.getImageId() } } task startContainer(type: DockerStartContainer) { dependsOn createContainer targetContainerId { createContainer.getContainerId() } }


gradle startContainerコマンドを実行して、イメージのアセンブリーとコンテナーの起動を確認します。 しかし、「Hello JavaMeetup！」というウェルカムメッセージの代わりに、ビルドが成功したという通知を受け取ります！







どこかで間違えたことがありますか？ 実際には、コンテナの出力をコンソールにリダイレクトするだけです。 



 task logContainer(type: DockerLogsContainer, dependsOn: startContainer) { targetContainerId { startContainer.getContainerId() } follow = true tailAll = true onNext { message -> logger.quiet message.toString() } }


gradle logContainerコマンドを実行し、 ... Hooray、切望されたメッセージと合格したレベル。







実際、それがすべてです。 Dockerfileも必要ありません（ただし、余分なものではありません-Gradleが手元にないことは決してわかりません）。



続けましょう！



レベル3







ほとんどの場合、実際には、アプリケーションは「Hello World」を表示するよりも難しいことを行います。 したがって、次のレベルでは、データベースからいくつかのレコードを取得するSpring Webアプリケーションである複雑なアプリケーションの実行方法を学習します。



ベースとアプリケーション自体を上げるために、 Docker Composeを使用します。 最初に、新しいファイルを作成します（別の新しい構成ファイル、呼吸しますが、停止しませんか？）- docker-compose.yml 。 その中で、データベースのイメージとアプリケーションのイメージを上げるためのサービスを単に規定します。 Docker Compose自体は、現在のディレクトリでymlファイルを見つけ、必要なコンテナーとイメージをピックアップまたは収集します。







この全体を開始するために、最初に画像を収集します。 この例では、Dockerのmavenプラグイン（hooray、XML！）がfabric8ioから使用されたため、最初にmvn installコマンドを実行します 。 



 <plugin> <groupId>io.fabric8</groupId> <artifactId>docker-maven-plugin</artifactId> <version>0.20.1</version> <configuration> <images> <image> <name>app</name> <build> <dockerFileDir>${project.basedir}/src/main/docker</dockerFileDir> <assembly> <mode>dir</mode> <targetDir>/app</targetDir> <descriptor>${project.basedir}/src/main/docker/assembly.xml</descriptor> </assembly> </build> </image> </images> </configuration> <executions> <execution> <id>build</id> <phase>install</phase> <goals> <goal>build</goal> </goals> </execution> </executions> </plugin>








プロジェクトとdockerイメージがアセンブルされるまで待ち、ymlファイルがあるディレクトリに移動してdocker-compose up -dコマンドを実行します。









docker psを実行して、両方のコンテナーが実行されていることを確認します。



Webアプリケーションが動作し、データベースから何かを取り出すことを確認するために、データベース内の何かを直接変更し、アドレスhttp：// localhost：8080 /にアクセスして、目的のデータを確認します。







これはすべて複雑に見えるかもしれませんが、実際には非常に簡単です。 第3レベルが完了しました。 まあ、ほとんど。



まだボーナスレベルがあります。 その上で、少し（非常に小さい）Docker Swarmを再生します 。正確には、 Docker Swarm Modeで再生します 。



ボーナスレベル







Docker Swarm Modeは特に複雑ではありません-Dockerが立つマシンのクラスターです。 ユーザーにとって、このクラスターは1台のマシンのように見え、すべてのチームは、このDocker Swarmがそうでなかった場合とほぼ同じように機能します。



Swarmモードでは、アプリケーションの複数のインスタンスを実行できます-たとえば、負荷分散のため。 また、スタックなどの抽象化がここに表示されます。DockerSwarmを使用すると、アプリケーション全体をデプロイできます。 また、通常のスケーリングと同様に、 スタックの複数のレプリカを拡張できます 。



SwarmモードのDockerコマンド： 



 $ docker service create --name japp --publish 8080:80 --replicas 3 java-app:demo $ docker stack deploy -c docker-compose.yml javahelloworld


実際、コマンド構文は非常に単純で、通常のコンテナーの作成に似ています。

docker composeも使用できます。 



 $ docker-compose scale jm-app=3


さて、過去3つのレベルで、Javaアプリケーションの移植性を達成しました。 最後のレベルに進み、Dockerが「私のマシンで動作する」というフレーズがもはや意味をなさないという声明を確認または反論するときです。



最終レベル







重いアプリケーションがあると想像してください。 または、同じマシン上に存在できる多数のマイクロサービス。 この場合、Javaアプリケーション（正確にはJVM）は、ホストマシンのリソースをめぐる戦いで、私たちの小さなシロナガスクジラと確かに格闘します。 ところで、これはこの記事で詳しく説明されています 。







このレベルでは、コードの例は少なくなりますが、異なるdocker container launch configurationがあります。 Dockerが使用するプロセスとリソースを分離する主な手段は、 cgroupと名前空間です。 しかし、主な問題は、これらすべてのJavaが小さなドラムであるということです。 彼女は貪欲で少し貪欲で、-memoryフラグを使用してJavaアプリケーションでコンテナを作成するときにメモリ制限を設定しても、実際にはより多くのリソースがあることがわかります。 これを確認するには、コンテナ内でfreeコマンドを実行します。 ここから、Java 8の一般的な推奨事項に従って、–Xmxパラメーターを設定し、–memoryパラメーターを–Xmxの少なくとも2倍にします。 Java 9の分野からの朗報-cgroupのサポートが追加されました。







Javaでのメモリリークのシミュレーションは非常に簡単です。 既製の画像valentinomiazzo / jvm-memory-testを取得し、さまざまなヒープサイズオプションとdocker用の--memoryで実行します。









最初のケースでは、コンテナのメモリがJavaアプリケーションよりも少ないため、不明確なエラーが発生します。 そして、OutOfMemoryExceptionを取得したいと思います。 「死んだ」コンテナを調べると、コンテナがOOMKillerによって殺されたことに気付くでしょう。これは、予測できない結果、Javaプロセスのフリーズ、リソースの不適切なシャットダウン、その他のあらゆる不快なものにつながる可能性があります（カーネルパニックにさえ遭遇しました）。 起こりうる最高のことではありません。



レートを上げて、コンテナにより多くのメモリを割り当てます。 今回はOutOfMemoryExceptionをキャッチできます。検査後、OOMKillerがコンテナに触れていないことを確認し、上記の問題をすべて回避します。









最後のレベルが渡されました。要約してみましょう。



再開



それで、私たちはゲームのすべてのレベルに合格し、結果として何を得ましたか？ Dockerは私たちに山を向けると約束していますか？



移植性は私たちが望むほど良くありませんが、Java 9はこれらの問題を解決すると約束しているようです。 柔軟性が増すと、すべてがより快適になります。dockerを使用すると、メインコードからそれほど遠くないコードで環境の再現可能な構成を取得できます。 そのようなものを追跡するのは、ルートの下のどこかで、いつ、何を修正、交換、または破壊した人よりも簡単に追跡できます。 また、一般的に、1台のマシンで多くのコンテナを実行できるため、リソースを大幅に削減できます。これはテスト中に重要になる可能性があります。



つまり、テストや開発には、Dockerが理想的だと思います。 ただし、本番環境で作業する場合は、この場合の負荷が非常に高くなる可能性があるため、より注意する必要があります。 そして、ドッカーの障害を乗り越えることは絶対に不快です。



そして最後に-dockerでJava 9を友達にするために必要な同じフラグ！
