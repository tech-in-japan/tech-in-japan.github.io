C ++例外の処理で難しいのは何ですか？



例外と関連するスタックプロモーションは、C ++で最も優れた手法の1つです。 例外処理は、プログラムのブロック構造と直感的に一貫しています。 外部的には、例外処理は非常に論理的で自然なようです。



スタックオブジェクトを正確に使用すると、非常に効率的で安全なコードを作成できます。ガベージコレクションを使用するシステムとは異なり、リンクの局所性が維持されるため、メモリに対するシステムの呼び出し回数を減らし、断片化を減らし、メモリキャッシュをより効率的に使用できます。



ただし、C ++では、例外は従来、文字通りエラー回復に関連する例外と見なされます。 これが、コンパイラによる例外処理の実装が非常に高価であるという理由または結果であるかどうかを言うのは困難です。 理由を理解してみましょう。

状況はどうですか。



例外処理を実装するには、2つのアプローチがあります。

  • 1つ目は、「敗者に支払いをさせる」という言葉で説明できます。 コンパイラーは、例外が発生しない場合のコストを最小化しようとします。 理想的には、プログラムには追加の負荷がかからず、必要な情報はすべて、プロモーションに便利な形式でコードから離れた場所に配置されます。
  • 2番目の戦略は「誰もが多く常に支払う」ことです。つまり、プロセスにおいて、プログラムはスタックの正しいプロモーションに必要な情報の関連性を維持するために特定のコストを負担します。さらに、例外の場合、スタックのプロモーションは安価です。


いくつかの例：

  • GCC / SJLJ SJLJはsetjmp / longjmpの略です。 むしろ、最初のアプローチを指しますが、コントロール転送スキームのおかげで、 試行ごとに固定料金もかかります。 本質的に、この実装には、両方のアプローチに固有のすべての最悪が組み込まれています。 4番目のバージョンまで、これは主要な例外処理オプションでした。

    名前が示すように、制御の転送はlongjmpの呼び出しを介して実行され、各試行はsetjmpの呼び出しを生成します。 対応するバッファは、各tryブロックのスタックに割り当てられます。

    各関数の最初に、現在のフレームをコンテキストスタックに登録するプロローグが作成されます。 同様に、現在のコンテキストをコンテキストスタックの最上部から削除するエピローグが作成されます。 リソースをクリアするために、各関数の隣にヘルパーコードが作成されます。

    より正確には、コンパイラーは、例外として終了する可能性のある関数からの戻りをキーとしてツリーに記録します。値はクリーニングコードへのポインターであり、関数のコンテキストをクリアするにはこの場所で取得する必要があります。 リンカは、各リンクモジュールのこれらのツリーの断片を単一のプロジェクトツリーに収集します（非常に大雑把です）。 この奇跡はLSDA（言語固有のデータ領域）と呼ばれ、「。gcc_except_table」セクションにあります。

    スローされた例外のtype_infoに基づいて例外が発生した場合、この例外を処理できるブロックが検索されます。 現在のコンテキストからハンドラコンテキストまで（呼び出しフレームのナビゲーションを使用）、コードのアドレスが抽出されて実行されます（ローカル変数のスコープに応じて）この場所で実行する必要があります。 次に、制御が転送されます。



    この方法は非常に高価であるという先入観があります。 すでに、すべてのtryブロックに対してsetjmpが呼び出されているという事実によりますが、これは安くはありません。 実際、プロセッサの状態を完全に維持する必要があります。この場合、数十個のレジスタが存在する可能性があります。 一方、例外の時点では、これらのレジスタのほとんどの内容はすでに役に立たない。 実際には、コンパイラは非常に合理的に機能します。 彼はsetjmpをデプロイし、有用なレジスターのみを保存します（すでにこの情報を持っています）。 著者は、setjmpのコストが非常に高いことを疑っています。



    しかし、本当に目を引くのは、特に重要な場合に、膨大な補助コードです。 コンパイラは、YACCと同様に、スタックマシンのすべての状態をペイントします。 そして、可能であれば、オプティマイザーは冗長性と簡単なコードをクリーンアップしますが、残っているのは十分以上です。

  • GCC / DW2 これは、例外処理の最初のアプローチの例にすぎません。 DW2は、DWARF2（現在3つ既に）-実行可能ファイル内のデバッグ情報を含む、補助情報を格納するための形式を意味します。 結局、デバッグ情報も必要であるため、以前の（上位の）呼び出しのフレーム内を含め、変数の値をいつでも見つけることができます。 したがって、コード生成のプロセスでは、コンパイラーは、変数を格納するときに、変数を配置するスタックに割り当てるものに関する情報を延期します...実際、この形式はDWARFと非常に似ていますが、DWARFと同一ではありません。 GCCの4番目のバージョンの標準。

    GCC / DW2は、GCC / SJLJとほぼ同じLSDAセクションを使用します。



    ご覧のとおり、スタックプロモーションに関連するコスト（例外がない場合）は実質的にありません。 ただし、例外を発生させるコストは高くなります。 さらに、コンパイラーのアーキテクチャー依存部分とそのかなり高レベルのレイヤーの強力な統合に注意することは間違いありません。

制限事項

このソリューションには多くの欠点があります。

  • デストラクタで例外をスローすることはできません。 unw_item_tデストラクタはこのインスタンスへのリンクをまだ削除していません。その結果、デストラクタが再度呼び出されます。
  • new演算子を使用してunw_item_tから継承したクラスのオブジェクトを作成することは非常に危険です。 メモリ自体を管理している場合でも、そのようなポインタは外部のコンテキストまたは外部のストリームに落ちる可能性があり、見ているオブジェクトが突然デストラクタを引き起こし、代謝大惨事を引き起こす可能性があります。
  • unw_item_tから継承されたクラスを別のクラスのメンバーとして集約することはできません。そうしないと、そのデストラクターが2回呼び出されます。
  • 説明されている方法は、ハードウェア例外と統合できません。
  • 例外タイプの制限。 上記では、文字列ポインタのみを使用しました。 プリミティブ型を例外として渡す場合、選択肢は1つだけです。 オブジェクトポインターを例外として使用する場合、RTTIを使用できます。 次のような提案ができます 
     #define CATCH_BLOCK_TYPED(t) \ } else if (NULL != dynamic_cast<t>(__exc)) {
    そしてこれにより、さまざまなタイプの例外を使用する機会が与えられます。 ただし、プリミティブ型の例外をスローすることはできません。
  • ユーザーは、スローされた例外オブジェクトを削除する必要があります。


そしてまだ。

説明されている制限にもかかわらず、説明されている方法には固有の利点があります。

  • シンプル。 数十行のコード-すべてが機能します。
  • 透明性のコンセプト。
  • 簡単な携帯性。 アーキテクチャに依存しません。
利点を維持しながら、この方法の欠点を排除することは可能ですか？ はい、いいえ。 C ++ツールのみを使用すると、これは不可能です。



著者は何で運転していますか？

技術的なせん妄の順序で、上記のスキームを正しく実装するためにコンパイラを変更する方法について考えますか？

上記のソリューションには何が欠けていましたか？ オブジェクトの生成方法に関する知識。

たとえば、オブジェクトが共通ヒープから割り当てられたメモリ上に構築され、スレッド間で移行できる場合、ストリーム依存スタックに決して登録されるべきではありません。 別のオブジェクトに集約されたオブジェクトをどこにも登録しないでください。

そして、同じタイプのオブジェクトを使用しますが、スタックメモリ上で、これを行う必要があります。 もちろん、このスタックオブジェクトへのポインタを別のスレッドに渡すことは可能ですが、どのような状況でこれが役立つか想像することは困難です。

だから：

  • タイプTのスタックオブジェクトの場合、コンパイラは実際にタイプのラッパークラスを作成します 

     template<class T> class __st_wrapper : public unw_item_t { public: virtual ~__st_wrapper() { unreg(); ((T*)data_)->T::~T(); }; private: char data_[sizeof(T)]; };
    目的のコンストラクターTを呼び出す

  • jmp_buf_splice :: root_slice_クラスの静的メンバーは、TLSまたは対応するレジスタ（存在する場合）を介して実装されます。
  • プログラマーは、 data_にあるタイプTのオブジェクトのみを見ています。
  • 仮想デストラクタのないスタックオブジェクトの場合、これはラッパーに表示されます
  • デストラクタで例外をスローできるようになりました デストラクタ自体を呼び出す前に、登録を解除しました。
  • ハードウェア例外（カーネル例外）はサポートされていないため、例外がスローされる時点で、コンパイラーはどのレジスターを「着陸」させる必要があるかを認識し、そうする必要があります。
  • スタックオブジェクトを定期的に破棄するために、コンパイラは__st_wrapperのデストラクターへの呼び出しを作成します
  • 適切なcatchブロックを選択するためのメカニズムはそのままです。 つまり これらのブロックの記述子がコード外にある補助テーブル情報が必要です。
  • setjmpのアナログを使用して制御を移行します。 中間（上記の2つに関して）コントロール転送オプションを実装することを提案します。 Setjmpには重大な欠点があります-バッファサイズは非常に大きく、実際には小さな部分が使用されます。

    一方、DWARFバイトコードの実行は非常に無駄が多いようです。

    したがって、setjmpバッファーの代わりに、実際の値があるスタックポインターを基準にした回復とシフトを必要とするレジスターのリストを格納します。 計算値の場合、値はレジスタに直接格納されます。 これを行うには、追加のメモリがスタックに割り当てられ、それにシフトが与えられます。 実際、一時変数が開始されます。

    コンパイラーは、例外を発生させる前に、関連するすべてのデータをレジスターからアンロードします。その場合、損失なく復元できます。



    それでも、 tryブロックの使用は意識的な行為であることに注意する価値があります。これには一定のコストが伴うという事実には何の問題もありません。 私見これらの（中程度の）コストはさらに便利です 言語ツールに対する責任ある態度を刺激します。

  • 演算子newとnew []を呼び出すときにキャッチする例外はそのまま残されます。 つまり 各tryを内部tryブロックで保護し、例外が発生した場合は以前のイテレーションで作成されたすべてを破棄します。 そして、もちろん、オブジェクトに割り当てられたメモリを返します[s。]
  • スタックオブジェクトの配列を実装するために何もする必要はありません。 ただし、特別なスタックオブジェクト（ new []演算子を呼び出すときに使用されるベクトルと同様のベクトル）を実装することにより、メモリを節約できます。
