「最大の競争相手は、自分ですべてを行う人たちです」-Perconaの作成と開発に関するPeter Zaitsev

パーコナのスペシャリストとの会話を続けています。 素材はとても興味深いです。 私たちの質問には、会社の創設者兼CEOであるPeter Zaitsevが回答しました。 Peterの実績は長く、MySQLとMongoDBに基づくソリューションのサポートと保守に従事する最大のコンサルティング会社の設立と開発です。 O'Reilly、MySQLの共著。 パフォーマンスの最適化 ''; MySQLおよびPostgreSQLに関する最高の技術ブログの1つであるPercona Database Performance Blogの定期刊行物。 自身の会社を設立する少し前に、ピーターはMySQL ABの高性能グループを率いていました。



私たちは、会社設立の歴史、重要な価値、現代の技術市場で働くことの難しさ、会社の開発戦略についてピーターと話し、パーコナの従業員の選択のいくつかの特徴について話しました-ピーターが個人的に監督するプロセスです。



PG Day'17 Russiaで、Petrは初めて 、ロシア語圏の視聴者向けにInnoDBのアーキテクチャとパフォーマンスの最適化に関する著者のコースを提供し、ドキュメント指向データベースとしてMySQL 5.7を使用するための利用可能なツールを確認します 。







PG Day：ピーター、あなた自身、あなたが誰で、何をしているのか教えてください。



Peter： Peter ZaitsevはPerconaの創設者兼CEOです 。 会社は10年以上前に設立されました。 MySQLおよびMongoDBの分野でソリューションを提供しています。 私たちの主なビジネスは、サポート、リモート管理（リモートDBA）、および管理（マネージドサービス）ですが、コンサルティングとトレーニングも行っています。



PG Day：どのようにしてMySQLを中心にビジネスを構築し始めましたか？ なぜこのDBMSを選択したのですか？



ピーター：実際、私は座って何をするべきか考え始めました。 Perconaを設立する前は、MySQL ABで働いていたので、テクノロジーを知っていたので、この業界の人々と出会い、つながりを持ちました。 したがって、機会が現れ、自分で何かをしたいという願望があったとき、選択は非常に明確で明白でした。 ビジネスでは、あなたがする方法を知っていること、あなたが潜在的な成功として見ていることをする必要があります。 Perconaの設立時には、従業員が100人を超える会社になるという壮大な考えはありませんでした。 私は数年間コンサルティングを行い、その後何か他のことをするだろうと思いました。 しかし、スタートは非常に成功し、すべてが行き来し、私たちは関与しました。



PG Day：まだMySQL ABで働いていたとき、技術的なバックグラウンドはありましたか？つまり、技術管理に来ましたか？



ピーター：はい、そうです。 MySQLでは、純粋に技術的な役割で働きました。 昨年私はエントリーレベルのマネージャーでした。 しかし、基本的には技術的な方向性でした。



PG Day：コンサルティングビジネスを開くことに決めた場合、どの基準でテクノロジー、スタックを選択しますか？ 現在の現実に関する決定はどのように変わりますか？



ピーター：いくつかの基準があります。 MySQLでは、2つの理由で非常に幸運だったと思います。 第一に、Perkonaが設立されたとき、MySQL市場は非常に急速に成長し 、十分な専門家がいなかったため、このテクノロジーを実装しようと考えた多くの企業は多額の費用を支払う準備ができていました。 このようなサイクルは、他のテクノロジーでも繰り返されます。 たとえば、Hadoop。 数年前、Hadoopの爆発的な成長が起こったとき、この技術を理解した人々は、 需要が供給を数回超えたため、非常に良い機会を得ました。 今、もちろん、私はこの方向を見ます。 第二に、 成功の重要な瞬間の1つです。好きなこと、自分の魂のあることをすることが重要です 。 お金のためだけに嫌いなことをすると、どこで働いても成功する可能性は低くなります。



PG Day：すべてのソフトウェアは完全に無料です。 売れるプロプライエタリなソフトウェアを作成しません。 閉じたソリューションを販売するようなビジネス開発のモデルをなぜ拒否したのですか？



ピーター：クローズドソフトウェアの市場、または一部の有料機能を備えた基本機能（オープンコア）のオープンソースコードの市場は、Perconaが扱う完全にオープンなソフトウェアの市場よりもはるかに大きいです。 ただし、多くの競合企業もあります。 ベンチャーキャピタルなど、多くのリソースがこの市場に投資され、企業は多くの資金を調達しています。 彼らは私たちがあまり興味を持っていない全く異なるビジネスモデルを持っています。



私たちは、クリーンでオープンなソフトウェアの市場に焦点を当てています 。これにより、他のベンダーとは大きく異なります。 これはMySQL市場で特に重要です。MySQL市場では、まったく新しいテクノロジーを使用せず、ゼロから作成することはありません。



多くの場合、Perkona市場について話している人々は、MongoDB、MySQL、MariaDBなどと比較します。 実際、これは完全に正しいわけではありません。なぜなら、私たちの仕事はプラットフォームやテクノロジーをゼロから作成するのではなく、この分野で最高のオープンプラットフォームにサービスを提供するからです。 閉じた「ローション」でPercona Serverを作成する場合、ソリューションとOracleソリューションなどの違いは非常に小さくなります。 同じOracleですが、よりシンプルで安価です。 これは良い動機付けではありません。 Oracleの運命がユーザーベンダーをロックインさせる独自のソフトウェアである場合、MySQLがオープンコアであっても、これを回避するためにオープンソースを作成しています。 非常に明確でシンプルな違い。



PG Day：会社が設立された瞬間から、MySQLのサポートとコンサルティング市場はどのように発展しましたか？ 確かに、Perconaはより多くの競合他社を持っています。 それらとどう違うのですか？ 競争力を維持し、市場で高い地位を占めることができるのは何ですか？



Peter： MySQLと言えば、Perconaは異なる競合他社のあるいくつかの市場で事業を展開しています。 サポートについて話すと、競合他社である企業はOracleとMongoDBのみで、問題の修正、ソフトウェアのバグなど、 完全なサポートサイクルを提供できる唯一の企業です。 MariaDBもMySQL市場に存在していましたが、現在は市場を去り、MariaDBプラットフォームのサポートのみに注力しています。



ソリューションだけでなく、 MySQLとMongoDBの幅広い技術をサポートしている唯一の企業であるという点で、私たちはユニークです。 たとえば、OracleのMySQLがバックアップ、監視、その他すべての独自のテクノロジースタックをサポートしている場合、OracleのPercona Server、Percona XtraDB Cluster、MySQLテクノロジー、MariaDBのMySQLバージョン、および次のようなクラウドソリューションの両方をサポートします。 Amazon RDSおよびGoogle Cloud SQL。



サポートの分野では、MySQLとMongoDBの両方を同時にカバーする唯一のベンダーです。これは、2つの理由で多くの人にとって非常に興味深いものです。 まず、テクノロジーが進化すると、多くの企業がMySQLの使用量と使用するMongoDBの数のバランスを取ります。 2つのベンダーと協力することは難しく、柔軟性がなく不快です。 Oracleサービスの価格が高騰した場合は、「聞いてくれ、みんな。 100個のMySQLインスタンスのサポートをお客様から購入しましたが、実際には50個のMySQLインスタンスを使用し、残りはMongoDBになります。 ライセンスを翻訳できますか？」 もちろん、彼らは「あなたはできない」と言うでしょう、なぜなら彼らはより少ないお金を受け取るからです。 私たちの側では、両方のソリューションをカバーするサポートを提供し、クライアントが使用することを期待しているものは彼の裁量に残ります。



第二に、1つのテクノロジのみを使用するベンダーは、このテクノロジを可能な限り使用することに関心があります。 MySQLを使用できることを示すことができれば、より多くのサポート契約を獲得できます。 現在の段階では、これら2つのDBMSの間に選択肢がある場合、そのような動機はありません。 クライアントはMySQLまたはMongoDBを使用できます-私たちにとっては、金銭的な違いはありません。 これにより、チーム全体がクライアントの利益に基づいたより客観的なソリューションを提供し、利益相反を回避できます。 おそらく、Perconaの基盤を形成した最も重要なことの1つです。私たちは、ウォレットではなく、本当に最適なクライアントであるソリューションに集中しようとしています。



リモート管理およびリモート管理の分野では、競合他社はすでに異なっています。PythianやDatavailなどの企業です。 「クラウド」企業も、多くの企業の所有者が「サービスとしてデータベース、Amazonのクラウドソリューション、Google Cloud SQLを使用する場合、DBAは必要ないかもしれません」と考え始めているためです。これは完全に真実ではありませんが、この競争の選択肢も私たちにとって興味深い事例です。 しかし、私たちのサービスについて全体的に言えば、最大の競争相手はおそらく自分ですべてを行う人です 。 例として、オフィスクリーニング市場を使用してこれをよく説明します。 ここで最大の競争相手は誰ですか？ おそらく多くの人が言うだろう：「そしてお金を節約しましょう、私たちは自分自身をきれいにして掃除機をかけます！」これが一種の選択であることは明らかです。 そして、そのような選択は、自分のデータベースを管理するか、他の誰かにそれらを与えることを決める人々によって直面されます。



PG Day：リモート管理サービスについて言及しました。 私が理解しているように、これらは企業の外部請負業者として働き、コンサルティングサービスを提供するDBAスペシャリストです。 どうやってそのような方向に来たのか教えてください、今では何が魅力的ですか？



ピーター：まず、これについては外部契約としてではなく、ネガティブな何かに関連していることが多いと思います。 クラウドモデルと「X as a service」などのテクノロジーの継続についてよく話します。 クラウドで何が起こっていますか？ 人々は、サーバーを設置してケーブルを接続できる専門家の完全なスタッフを雇って維持することはほとんど意味がないと気づきました。 多くの企業にとって、これらはあまり重要な決定ではなく、アウトソーシングすることで企業がコアビジネスに集中できるようにします。



テクノロジー企業にとって、これは製品の創造です。 小売業者にとって、これはマーケティングまたはロジスティクスです。 また、データベースの保守作業は、多くの場合、実行する必要がある作業だけになります。 彼らが社内で成功しようとすると、エネルギーが必要になります。 この場合、これらの問題をすべて当社に転送し、製品の開発に専念するよう人々に提供します 。



この点で、これらの企業はマーケティングに多くの投資をしているため、 Database as a Service市場は私たちにとっても役立ちます。 データベースのセットアップと管理自体が不利になる可能性があるという考えを顧客に伝えようとします。 しかし、彼らは最後まで行きません。 たとえば、Amazon RDSを見ると、これはマネージャーが操作できるサービスではありません。 展開するインスタンス、スキームの「設計」方法を誰かが決定する必要があります。 簡単なバックアップ設定、監視には技術専門家も必要です。 そしてこれがまさに私たちが提供できるものです。なぜなら、 私たちはチームの一員になり、これらの問題を完全に解決するために管理部門と開発部門の両方と直接連携できるからです。



PG Day：サポートするテクノロジーの多様化について話しました。 どの時点でMongoDBをサポートすることにしましたか？ それは意味のあるステップでしたか、それとも有機的にうまくいったのですか？



ピーター：一方では、戦略の開発の始めに、テクノロジーを追加したいと考えていました。 クライアントに客観的なソリューションを本当に提示できることが非常に重要だと思います。 sayingがある場合：ハンマーを持っている場合、すべてが釘のように見えます-使用できるツールが1つある場合、それが不要な場合でもすべてがこのツールに適しているようです。 MongoDB市場に機会が現れ、私たちにとって非常にうまくいきました。MongoDBが企業としてMySQLの代替として位置づけるために一生懸命働いたこともあり、この市場には代替サポートベンダーの需要があります。



TokuTekを買収することができました。TokuTekは 、すでにMongoDBとそのフォークMongoDBのサポートサービスを提供しており、市場でよく知られています。 これにより、ゼロからすべてを行うのではなく、新しい市場に参入することができました。 これが起こらなかった場合、MySQL以外の何かを始めるにはもっと時間がかかります。



PG Day：コンサルティングは人を基盤としたビジネスです。 あなたの会社は分散しており、あなたが全員座っている大きな建物はありません。 多数の顧客にどのように対処しますか？ どのような専門家があなたのために働いていますか？



ピーター：現在、約150人の従業員、 30か国、 20か国の米国の州があります。 ほとんどの人は自宅で仕事をしています。 米国のノースカロライナ州には、1日平均5人の小さなオフィスがあります。 会議を開催するのに便利なように作られています。 ゲストがよく来ます。 また、多くの小規模オフィスもあります。



私の意見では、サポート事業とコンサルティング事業を一行にまとめるのはまったく正しくありません。 私たちはコンサルティング会社としてスタートし、数年後には開発が困難になりました。 会社が成長し、構造がより複雑になり、コストが増加したため、 Percona Server for MySQLなどのソフトウェアの開発を開始しました。 多くの場合、コンサルティングはまったく予測不可能です。クライアントが次から次へと来ることもありますが、時には「健康な」季節が過ぎて、何の関係もないコンサルタントがたくさんいることがあります。



そのため、ビジネスモデルに移行しました。 サポートとリモート管理 -1か月または1年間の固定コストのサブスクリプションであり、当社とお客様の両方にメリットをもたらします。 私たちにとって、これは予測可能な収入であり、投資とリソースを計画する能力です。 しかし、顧客にとっては、これは一度スケジュールできる予算でも予測可能です。 過去1年間、サブスクリプションサービスを提供するこのモデルに切り替えましたが、収入の約85％がサポートとリモート管理です。 コンサルティングとトレーニングを行っているのはわずか15％です。



今スケーリングの点で。 サブスクリプションを固定すると、効率を高めるためにテクノロジーに投資できます。 これは、リモートコントロールの観点から最も明白です。 1時間あたりの料金を請求する多くの企業は利益相反を抱えています。 顧客にあまりにも多くの自動化を導入すると、顧客はより少ない時間を支払う必要があります。 しかし、顧客は時間の支払いをするため、自動化を促進し、生産性を向上させるために、イノベーションに投資するための資金を見つけることは困難です。 私たちの観点からは、結果を固定価格で販売するため、主な関心は人件費を削減するために可能な限りすべてを自動化することです。 これにより、自動化に真剣に投資できます 。 自動化されたシステムはより安定しているため、これはクライアントにとって便利です。 コンピューターは人よりもミスを少なくします。 これは、特に大規模なシステムと大規模なチームに当てはまります。



私たちの成功の大部分はこれに関連していると思います。 これにより、スタッフの面で大幅に柔軟になります。 ある種の請求書を顧客に販売し、生産性が5倍高いスタッフのある種のスーパー開発者がいるため、そのために3倍のお金を払いたいとしましょう。 実際、コンサルティングビジネスモデルはそのようには機能しません。 企業顧客は幅広い価格に否定的に反応しますが、ソフトウェア開発の分野における生産性は中級から高級のスペシャリストの間で10倍も異なることが広く知られています。 したがって、サブスクリプションワークモデルに焦点を当てることで、 非常に効率的なスタッフに投資する余裕があります。
