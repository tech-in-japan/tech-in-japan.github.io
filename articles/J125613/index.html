登録



おそらく、IoCコンテナにとって最も重要なアクションは登録です。 それは、それがいかに便利で理解しやすいか、人々がそれを使用するかどうか、そしてツールの不適切な使用の可能性に依存します。



著者は、DSLをできるだけ広く使用し、接続文字列、URL、ファイルパス、およびその他すべてを同じように個別に設定する必要がある場合にのみ、構成ファイルに頼ることをお勧めします。



最も単純なものから始めましょう。デフォルトのコンストラクターでクラスを登録します。



このような単純なクラスセットがあるとします。 

public interface IClass {} public interface IClass1 : IClass {} public interface IClass2 : IClass {} public class Class1 : IClass1 {} public class Class2 : IClass2 {}


これまでのところ、このセットが可能性を実証する主なものになります。 しかし、心配しないでください。これ以上のクラスと接続はより難しくなります。そのようなクラスでもあまり表示されないからです。





ベース



登録は、 Containerを使用して、静的ObjectFactoryクラスを使用して、またはRegistryクラスを使用して可能です。 これらすべてのオブジェクトを使用して、登録を徐々に検討してください。 登録の主なクラスはレジストリであり、残りのクラスは便宜上、その機能をスローします。



静的ObjectFactoryクラスに登録します。 

 public class RegisterByObjectFactory { public RegisterByObjectFactory() { ObjectFactory.Initialize(x => { x.For<IClass1>().Use<Class1>(); x.For<IClass2>().Use<Class2>(); }); } }


すべてがDSLおよびラムダ式を使用して行われます。 DSL自体は非常に簡潔で理解しやすく、結果のコードは意味のある表現に簡単に折りたたまれます。 この場合、 IClass1 インターフェイスに Class1を使用 する 必要があることを簡単に 理解 できます 。



オブジェクトの取得は、次の方法で行うことができますが、直感的でもあります。 

 private static string ObjectFactoryExample() { new RegisterByObjectFactory(); var class1Inst = ObjectFactory.GetInstance<IClass1>(); var class2Inst = ObjectFactory.GetInstance<IClass2>(); return ObjectFactory.WhatDoIHave(); }


この場合、インターフェイスでGetInstanceオブジェクトを取得するメインメソッド。 さらに、完成したオブジェクトを取得するさまざまな方法を検討します。 メソッドは、メソッドから取得した文字列を返すことがわかります。この名前は、発話名WhatDoIHaveを持ちます。 この方法を使用すると、コンテナの内部を診断し、何が、どのように、どこに登録されているかを確認できます。



長い間、フレームワークの作者は彼の子孫に関連してコンテナという用語を受け入れることができなかったため、次のメソッドは長い間隠されていましたが、その後の実装でのみ、静的クラスの背後で内部で実装されたように、自然な登録コースが開かれました。 だから 

 public class RegisterByContainer { public IContainer Container; public RegisterByContainer() { Container = new Container(x => { x.For<IClass1>().Use<Class1>(); x.For<IClass2>().Use<Class2>(); }); } }


一見、すべてが同じで、ラムダが同じですが、今度はクラスを作成し、それを外部に渡し、それによってコンテナにアクセスします。 つまり 繰り返しますが、ObjectFactoryはContainerクラスの静的ラッパークラスにすぎません。



オブジェクトの取得も同じシナリオに従います。 

 private static string ContainerExample() { var container = new RegisterByContainer().Container; var class1Inst = container.GetInstance<IClass1>(); var class2Inst = container.GetInstance<IClass2>(); return container.WhatDoIHave(); }


行の次のオブジェクトはRegistryです。 実際、以前はすべて間接的に呼び出していました。 変更のために、特定のクラスを登録します。

この場合、 ForConcreteTypeメソッドが使用されます。これはの同義語です。 < T>（）の場合。 < T>（）を 使用します。 また、Registryクラスをサブコンテナとして使用し、アセンブルしてから、1つのコンテナ内のアセンブリに転送できることもわかります。 この場合、作成時のレジストリの追加が示されていますが、何も書くことを妨げるものはありません。

プロフィール



StructureMapでは、名前付きプロファイルを使用してクラスマッピングをグループ化できます。 つまり マッピングクラスをすばやく切り替えることができます。 

 public class RegisteringProfiles { public IContainer Container; public RegisteringProfiles() { var registry = new Registry(); registry.Profile("p1", x => x.For<IClass>().Use<Class1>()); registry.Profile("p2", x => x.For<IClass>().Use<Class2>()); Container = new Container(x => x.AddRegistry(registry)); } }


ここでは、クラスClass1とClass2が共通インターフェースによって登録されていますが、異なるプロファイルにあることに注意する価値があります。 必要なクラスを取得するには、プロファイルの名前を取得するSetDefaultProfileメソッドを使用して、コンテナ内のプロファイルを切り替える必要があります。 

 private static string ProfilesExample() { var container = new RegisteringProfiles().Container; container.SetDefaultsToProfile("p1"); var class1Inst = container.GetInstance<IClass>(); container.SetDefaultsToProfile("p2"); var class2Inst = container.GetInstance<IClass>(); return container.WhatDoIHave(); }


プロファイル名には文字列変数のみを使用できますが、これはもはや大きな問題ではありません。 つまり、実際には、例のように線を開いてプロファイル名を書くべきではありません。 カルマに有害です



アクティブなプロファイルを設定した後、通常どおりコンテナを操作できます。 その結果、同じ行を実行すると、 container.GetInstance <IClass>（）; 異なるクラスを取得します。





プラグイン



共通インターフェースを使用して特定のクラスを取得する問題を解決する別の方法があります。これは名前付きプラグインです。



用語について少し。 IntelliSenseとここの少しでは、プラグイン、PluginTypeおよびPluggedTypeという用語を見つけることができます。一般的に、これは取得したいタイプを意味します 。 つまり 前述のすべての例で、IClassはPluginTypeと呼ばれ、Class1またはClass2はPluggedTypeと呼ばれます。 

 public class RegisterAsPlugin { public IContainer Container; public RegisterAsPlugin() { Container = new Container(x => { x.For<IClass>().Use<Class1>().Named("Class1"); x.For<IClass>().Use<Class2>().Named("Class2"); }); } }


例として、共通のインターフェースでクラスを登録することがわかりますが、同時に特定の名前を付けます。 Namedメソッドを使用して、特定のタイプを簡単に照会できるようになりました。 

 private static string PluginExample() { var container = new RegisterAsPlugin().Container; var class1Inst = container.GetInstance<IClass>("Class1"); var class2Inst = container.GetInstance<IClass>("Class2"); var instanceDef = container.GetInstance<IClass>(); return container.WhatDoIHave(); }


この例は、コンテナーにアクセスして、共通インターフェースで特定のタイプを取得する方法を示しています。 ただし、ここで同時に質問を提起します。プラグイン名を指定せずに共通のインターフェイスでGetInstanceメソッドを呼び出そうとするとどうなりますか？



デフォルトの動作は、「誰が最後、つまりパパ」という言い方に従います 。 この場合、 Class2クラスのインスタンスはinstanceDef変数に入ります。 ただし、デフォルトクラスを明示的に定義できます。 これを行うには、プラグインの登録のわずかに異なる形式を使用します。 

 public class RegisterAsPluginWithDefault { public IContainer Container; public RegisterAsPluginWithDefault() { Container = new Container(x => x.For<IClass>() .AddInstances(i => { i.Type(typeof (Class1)).Named("Class1"); i.Type(typeof (Class2)).Named("Class2"); }) .Use(new Class1()) ); } }

この場合、デフォルトのインターフェースに使用されるタイプを伝えるのはUseメソッドです。 ここで次のコードを実行すると

スキャン



論理的な継続は、コンテナ内の型の検索と自動登録です。 共通のインターフェースから2つのクラスを継承するのではなく、50を想像してください！ これらすべての登録と依存関係を自分の手で満たすのは非常に悲しく退屈です。 このような場合、StructureMapには、関心のあるアセンブリまたはフォルダーを実行して適切なオブジェクトを登録するScanメソッドがあります。 したがって、アプリケーションにプラグインの構造を実装し、何らかの方法でMEFと競合したり、MEFと置き換えることも可能です。



Scanメソッドがタイプを見つけて登録するには、いくつかの条件が満たされている必要があります。

  • タイプは明示的である必要があり、ジェネリックタイプは自動的に登録されません。
  • この型にはパブリックコンストラクターが必要です。
  • コンストラクターはプリミティブ型の引数を持つことはできません。
  • 複数の登録は許可されていません。


スキャン方法と動作はオーバーライドされる場合がありますが、現時点ではこれは考慮されません。



スキャンするアセンブリをいくつかの方法で指定できます。

  • アセンブリ名を明示的に綴るか、渡す。
  • 呼び出し元アセンブリにアクセスします。
  • 特定のタイプを含むアセンブリを見つけます。
  • 特定のパスでビルドを検索します。


実験的なアセンブリを指摘した後、さまざまなパラメーターで型を含める/除外する方法を使用して、インポートプロセスをより詳細に構成できます。 詳細については、 ドキュメントを参照することをお勧めします 。 それは時代遅れですが、可能性の一般的なアイデアを提供します。



それでは、より簡単な例を見てみましょう。 

 public class RegisterByScan { public IContainer Container; public RegisterByScan() { Container = new Container(x => x.Scan(s => { s.AddAllTypesOf(typeof (IClass)); s.AssembliesFromPath("."); s.WithDefaultConventions(); s.LookForRegistries(); })); } }


このクラスでは、アプリケーションフォルダーからIClassインターフェイスを実装するすべての型をインポートすることを言います。デフォルトの規則に従ってください。 最後の行は、検索を開始するコマンドです。 以前は、明示的な指示なしにすべてが機能していました。 ただし、LookForRegistriesメソッドを明確に定義する必要があります。



メソッドが機能すると、コンテナで検出および登録された内容を確認できます。 

 private static string RegisterByScanExample() { var container = new RegisterByScan().Container; var instances = container.GetAllInstances<IClass>(); return container.WhatDoIHave(); }


Get All Instancesメソッドが呼び出されていることに注意してください。 登録されたクラスから特定のクラスを取得するメソッドを呼び出すと、StructureMapは「デフォルトで」返すクラスを認識しないため、エラーが発生します。



正直なところ、このような実装では、スキャンコマンドの結果を使用することはできません。 すべてがうまくなり、見つかったクラスを名前で参照できるようにするには、スキャンコードを少し書き換える必要があります。 

 public class RegisterByScanWithNaming { public IContainer Container; public RegisterByScanWithNaming() { Container = new Container(x => x.Scan(s => { s.AddAllTypesOf(typeof (IClass)).NameBy(t => t.Name); s.AssembliesFromPath("."); s.WithDefaultConventions(); s.LookForRegistries(); })); } }


すべてのクラスを名前で登録する必要があるという明確なルールがAddAllTypesOfメソッドに追加されました。 この変更後、特定のタイプを操作できます。 

 var instance = container.GetInstance<IClass>("Class1");




実装



コンテナの操作中に、デフォルトで返されるタイプをオーバーライドできます。 これは主にテストで使用されます。 仕事のデモンストレーション： 

 private static string InjectExample() { var container = new RegisterAsPluginWithDefault().Container; var instance1 = container.GetInstance<IClass>("Class1"); var instance2 = container.GetInstance<IClass>("Class2"); var class1Inst = container.GetInstance<IClass>(); container.Inject(typeof (IClass), new Class2()); var class2Inst = container.GetInstance<IClass>(); return container.WhatDoIHave(); }


以前に、デフォルトのClass1クラスを返すRegisterAsPluginWithDefaultクラスを宣言しました。 Injectメソッドを使用すると、戻り値の型をオーバーライドできます。プラグインの型と新しいクラスを指定するだけです。
