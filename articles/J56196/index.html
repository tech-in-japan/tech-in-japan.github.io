gen_serverの概要：Erlybank



これは、OTPシリーズの概要の2番目の記事です。 gen_serverについて説明し 、これを読む前に銀行システムの基礎を築く最初の記事を読むことをお勧めします。 すばやく把握すれば、 完成したバージョンのサーバーを確認して先に進むことができます。



シナリオ： ErlyBankサーバーを顧客に提供しましたが、顧客は非常に満足していました。 しかし、21世紀は庭にあり、安全で使いやすいATMも求めているため、サーバーを拡張し、ATM用のソフトウェアを作成するよう依頼されました。 ユーザーアカウントは4桁のPINコードで保護する必要があります。 ATMでは、以前に作成したアカウントを使用してログインし、アカウントから入金または引き出しを行うことができます。 美しいインターフェイスを作成する必要はありません;他の人がそれをやっています。



目標：最初に、アカウントのPINコードのサポートとPINコードによる認証を追加して、サーバーを拡張します。 次に、gen_fsmを使用してATMバックエンドを作成します。 データ検証はサーバー側で実行されます。



gen_fsmとは何ですか？



gen_fsmは別のErlang / OTPインターフェイスモジュールです。 有限状態マシンを実装するために使用されます。



この記事では「状態」の概念が2つのことを意味するために使用されるため、事前に謝罪します。

  • state gen_fsm-状態マシンの状態、その操作の現在の「モード」。 gen_serverからの状態（データ）とは関係ありません。
  • ステータス（データ） -サーバーステータスデータは、gen_serverについての以前の記事から学んだことです。


もちろん少しcですが、上記の条件のコンテキストでのみ参照するようにします。



gen_fsmはある状態で動作を開始します。 gen_fsmへの呼び出し/キャスト呼び出しは、特別なコールバックメソッドで処理されます。このメソッドは、gen_fsmの現在の状態（ステートマシン）の名前と呼ばれます。 実行されたアクションに基づいて、モジュールは状態を変更できます（[現在の状態、着信文字]->新しい状態、約）。 ステートマシンの教科書の例は、閉じたドアです。 最初は、ドアは「閉じた」状態です。 開くには4桁のコードを入力する必要があります。 1桁を入力すると、ドアはそれを保存しますが、1桁では不十分なので、「閉じた」状態で待機し続けます。 4桁を入力した後、それらが正しい場合、ドアはしばらくその状態を「開く」に変更します。 数値が正しくない場合、「クローズ」状態のままになり、メモリがクリアされます。 おそらく、gen_fsmを使用してステートマシンを実装する方法について、すでにいくつかの推測をお持ちかもしれません。



gen_serverの場合と同様に、gen_fsmで実装する必要があるコールバックメソッドのリストを示します。 gen_serverに関連することがたくさんあります。

  • init / 1-ステートマシンサーバーを初期化します。 gen_serverとほぼ同じです。
  • StateName / 2 -StateNameは州の名前に置き換えられます。 このメソッドは、ステートマシンがこの状態にあり、メッセージを受信したときに呼び出されます。 その結果、特定のアクションが実行されます。 これは非同期コールバックメソッドです。
  • handle_event / 3 -StateName / 2と同じ。ただし、このメソッドは、マシンの現在の状態に関係なく、クライアントがgen_fsm：send_all_state_eventを呼び出したときに起動します。 繰り返しますが、非同期です。
  • StateName / 3 -StateName / 2の同期バージョン。 クライアントはサーバーの応答を待ちます。
  • handle_sync_event / 4 -handle_eventの同期バージョン/ 3。
  • handle_info / 3 -gen_server handle_infoと同等です。 このメソッドは、gen_fsmの非標準の手段によって送信されたすべてのメッセージを受信します。 これは、タイムアウトメッセージ、プロセス終了メッセージ、または「！」を使用してサーバープロセスに送信されるその他のメッセージです。
  • terminate / 3-サーバーのシャットダウン時に呼び出され、サーバー内の占有リソースを解放できます。
  • code_change / 4-サーバーがリアルタイムで更新されるときに呼び出されます。 現在は使用していませんが、今後の記事で使用する予定です。

gen_fsmの初期化



ATMを起動するには、gen_serverと同じstart_linkメソッドを使用します。 ただし、初期化は少し異なります。 

  init（[]）->
   {OK、無許可、誰も}。


gen_fsmのinit / 1メソッドは{ok, StateName, StateData}



を返す必要があります。 StateNameはサーバーの初期状態、StateDataはサーバーの初期状態（データ）です。 このケースでは、無許可の状態で開始し、データはnobody



設定されます。 状態（データ）は作業しているアカウントの名前になるため、最初は何もありません。 Erlangにはnull / nil / nothingのデータ型がありません。代わりに、例えば誰もいないので、通常は話すアトムが使用されます。



アカウント認証

sync_send_eventメソッドは、gen_serverモジュールの呼び出しメソッドと同等です。 サーバーの現在の状態（最初の引数）にメッセージ（2番目の引数）を送信します。 したがって、このメッセージのハンドラーを作成する必要があります。

       {返信、{エラー、理由}、不正、状態}
  終わり;
不正（_Event、_From、State）->
  返信= {エラー、無効なメッセージ}、
   {返信、返信、未承認、州}。


この関数は、サーバーが無許可の状態にあるときにメッセージを受信する必要があるため、無許可と呼ばれます。 タップ{authorize, Name, Pin}

短い「ありがとう！」条件



ATMを使用した多くの（またはすべての）人々は、短い間表示される小さな「ありがとう！」画面があることを知っています。 実際、実装ではこの画面がなくても安全に実行できます。gen_fsmにタイムアウトのある機能を表示したかっただけです。 5000ミリ秒後、またはメッセージが受信されない場合、状態を「無許可」に戻すので、ATMは次のユーザーで再び開始できます。 コードは次のとおりです。 

 thank_you（タイムアウト、_State）->
   {next_state、無許可、無人};
 thank_you（_Event、_State）->
   {next_state、無許可、無人}。


注：訓練を受けた目では、両方の方法が同等であり、最初のサンプルは不要であることがわかります。 確かに、最初のサンプルを含めて、タウマウトを捕まえるようにしました。
