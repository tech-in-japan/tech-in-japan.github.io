Windows AzureテーブルストレージとWindows Azure SQLデータベースの比較と対照

この記事では、Windows Azureでサポートされる2つの構造化ストレージサービスを比較します。WindowsAzureテーブルストレージとWindows Azure SQLデータベース（以前はSQL Azureと呼ばれていました）。 この記事の目的は、これらのテクノロジーを比較して、それらの一般的およびさまざまな特性を理解できるようにすることです。 この比較は、どのテクノロジーがシナリオに適しているかについて、より情報に基づいた決定を下すのに役立ちます。



はじめに



Windows Azureの永続的なデータストレージに関しては、2つのクラウドオプションがあります。WindowsAzure SQLデータベースとWindows Azureテーブルストレージです。



Windows Azure SQL Databaseは、SQL Serverのコア機能をクラウドに拡張するリレーショナルデータベースサービスです。 SQL Databaseを使用すると、リレーショナルデータベースソリューションをクラウドに展開できます。このアプローチの利点には、管理インフラストラクチャ、高可用性、スケーラビリティ、使い慣れた開発モデル、さまざまなデータアクセスフレームワークとユーティリティが含まれます。 SQL Server環境。 SQL Databaseは、ローカルSQL ServerデータベースをWindows Azure SQLのデータベースと移行、エクスポート、およびリアルタイムで同期する機能も提供します（ SQL Data Syncを使用）。



Windows Azureテーブルストレージは、追加の構造を必要とする大量の非リレーショナルデータを保存するアプリケーションに役立つ、フェールセーフ、ISO 27001認定のNoSQLキー値ストレージサービスです。 このサービスは、スキーマを持たないデータへのキーベースのアクセスを提供します。 ただし、スキーマなしで構造化データを保存すると、データ間の関係を作成できません。



顕著な違いにもかかわらず、両方のサービスは非常にアクセスしやすいマネージドサービスであり、月間99.9％のSLAが提供されます。



テーブルストレージと SQLデータベース



SQL Databaseと同様に、Windows Azureテーブルストレージには構造化データが格納されます。 2つのサービスの主な違いは、SQL DatabaseはSQL Serverエンジンに基づくリレーショナルデータベース管理システムであり、標準の原則とリレーショナルの基礎に基づいて構築されているため、Transact-SQLクエリ、ACIDトランザクション、ストアドプロシージャ、実行可能プロシージャなどの機能を提供することですサーバー側。



Windows Azureテーブルストレージは、アプリケーションデータモデルを特定のスキーマセットにロックすることなく、クラウドベースのアプリケーションをすばやく作成できる柔軟なキーと値のエンティティストアです。 このサービスはリレーショナルデータのリポジトリではなく、SQLデータベースなどのリレーショナルデータを管理する機能（結合やストアドプロシージャなど）を提供しません。 Windows Azureテーブルストレージでは、サーバー側のクエリサポートが制限されていますが、トランザクション機能があります。 さらに、同じテーブル内の異なるレコードは異なる構造を持つことができ、Windows Azureテーブルストレージのこのアプローチにより、単純なリレーショナルデータを効率的に保存および操作できます。



アプリケーションが、リレーショナル関数を使用する必要のない大量のデータを保存および操作する場合、Windows Azureテーブルストレージの方が適している可能性があります。 アプリケーションがスキーマで接続されたデータセットを処理する必要がある場合、SQL DatabaseはWindows Azureテーブルストレージよりも適切なオプションのように見えます。 SQLデータベースを使用するかWindows Azureテーブルストレージを使用するかを決定する前に、考慮すべき要素が他にもいくつかあります。その一部を以下に示します。



技術的な考慮事項



特定のソリューションに適したストレージテクノロジを決定する場合、アーキテクトと開発者は次の推奨事項を考慮する必要があります。



次の場合には、 Windows Azureテーブルストレージの使用を検討してください。



・アプリケーションは、大量のデータ（たとえば、数テラバイト）を安価なストレージに保存する必要があります。



・アプリケーションは、複雑な関係関係のない大きなデータセットを格納および操作します。そのため、サーバー側で結合を使用するか、セカンダリインデックスまたはサーバー側で実行される何らかの複雑なロジックを使用する必要があります。



・アプリケーションには、その構造が異種であるオブジェクトを格納するための柔軟なデータスキームが必要であり、その構造は設計段階で決定するのが困難です。



・ビジネスには、地理的多様性を使用した壊滅的な状況の場合にデータを保存する機能が必要です。 Windows Azureテーブルは、数百マイル離れた同じ大陸にある2つのデータセンターに地理的に複製されます。これにより、災害発生時のデータの安全性がさらに向上します。



・シャーディングやパーティション分割を使用せずに150 GBを超えるデータを保存する必要があります。



・データを手動でシャーディングせずに、高度なスケーリングが必要です。



次の場合には、 Windows Azure SQLデータベースの使用を検討してください。



・アプリケーションは、高度な構造化された関係と関係を持つデータ処理を処理します。



・データは本質的にリレーショナルであり、データの一意性、制限、さまざまな種類のキーのルールによるデータの整合性を確保するために、リレーショナルデータのソフトウェアモデルの主要な原則を実装する必要があります。



・データの量は、割り当てられたストレージユニット（データベース）ごとに150 GBを超えることはできませんが、この制限を超えた場合はデータを複数のデータベースに分割できます。



・既存のデータ指向アプリケーションは既にSQL Serverを使用していますが、既存のフレームワークを使用してクラウドストレージにアクセスし、オンプレミスインフラストラクチャとWindows Azureの間を透過的に移行する機能が必要です。



・アプリケーションでT-SQLストアドプロシージャを使用してデータレイヤー内で計算を実行し、それによってアプリケーションとデータストア間のラウンドトリップの回数を最小限に抑える予定です。



・アプリケーションは、空間データ、さまざまなデータ型、およびユニオン、集計、複雑な述語を含むデータアクセスパターンを使用する必要があります。



・アプリケーションは、ユーティリティを使用して視覚化とBIデータモデルを提供します。

ご注意
Windows Azureの多くのアプリケーションは両方のテクノロジーを利用できますが、これらを組み合わせて使用​​することをお勧めします。

追加情報


・ビルトインファイアウォールを使用して、管理ポータルでルールを構成することにより、ネットワークレベルでSQLデータベースのインスタンスへのアクセスを制限できます。 任意のクライアントがテーブルに接続でき、テーブルはHTTP / HTTPSを介してWindows Azureストレージに接続できます。



・Windows Azureテーブルストレージは、単一レコードおよびエンティティグループトランザクションのすべての挿入、更新、削除トランザクションを保証します。 リクエストごとにスナップショット分離が提供されます。 要求は、要求の実行の開始時からトランザクションの実行中にパーティションの表示を制御します。 複数のテーブル間の一貫性については、アプリケーション開発者が責任を負います。



・Windows Azureテーブルはログをサポートしており、実行中のすべてのクエリを表示できます。 ロギングは、クエリの集約メトリックも提供します。



・現在、Windows Azure SQL Databaseはログとメトリックを提供していませんが、クエリパフォーマンスの問題の診断、データベース接続の監視、アクティブなトランザクションの表示、クエリ実行プランの調査に役立つ動的管理ビュー （DMV）のセットを備えています。



・Windows Azure SQLデータベースはSQL Serverエンジンに基づいているため、TempDBやトランザクションログなど、いくつかの概念を使用できます。 トランザクションログの無制限な増大を防ぐために、SQLデータベースはログのサイズに10 GBの制限を課しています。 SQLデータベースインフラストラクチャはこれらのログを管理しますが、直接アクセスすることはできません。 Windows Azureテーブルストレージには、これらのログに相当するものはありません。代わりに、変更可能なデータではなく、リクエストを追跡するロギングとメトリックがあります。



・マルチテナント環境でのリソースの過剰な使用を防ぐために、両方のサービスは調整メカニズムを使用しますが、これらのサービスの動作原理によって異なります。 たとえば、SQLデータベースでは、 ソフト調整とハード調整の 2つの調整戦略が使用されます 。これらについては、 こちらで確認できます 。
