クラウドでの自動テスト

少なくとも少しテストすれば、検証用の新しいビルドで環境を作成するだけでなく、その方法を理解できます。 ソリューションが複雑でない場合は、すべてを手作業で行うことができます-古いファイルを消去し、レジストリ、データベースを消去し、新しいビルドをコピーします。 ただし、深刻な多層ソリューションを使用している場合は、すべての操作を自動化することをお勧めします。 したがって、人的要因の影響を減らすことができます。 問題が見つかった場合は、スクリプトを修正してエラーを修正し、やり直してください。 Team Foundation Serverには、テスト環境管理環境であるLab Managementを整理するための完全なツールセットが含まれています。 興味深いのは、インフラストラクチャへの投資なしでテストラボを組織できることです。 必要なのはすでにAzureとTeam Foundation Serviceにあります。 これは、インフラストラクチャの一部をクラウドに安全に取り出すことができる、新しいIaaSの場合です。



スクリプト。



開発者はコードをクラウドTFSに保存し、タスクを管理します。クラウドでビルドを作成し、自動テストのためにこれらのビルドを仮想マシンに展開することができます。



自動テストとは、プログラム自体を起動し、ボタンをクリックし、一般に実際のテスターのインタラクティブなアクションを繰り返すテストを意味します。



準備する


http://tfs.visualstudio.comで登録し、プロジェクトを作成します。 その後、テストベンチを設計します。 2つの仮想マシンで構成されます。



  1. デスクトップクライアント ：デスクトップアプリケーションが実行され、インタラクティブなテストが実行されるマシン。
  2. サーバー ：デスクトップクライアントのサービスが実行されているマシン。これはデータベースであり、Webアプリケーションである可能性があります。


さらに、ビルドサーバーが必要です。プロジェクトがソースからコンパイルされる仮想マシンと、単体テストの実行です。



これらのマシンが相互にやり取りするために、Azure管理ポータルの最初のステップは仮想ネットワークを作成することです。







次に、スタンド用の2つの仮想マシンとビルドサーバー用の1つのマシンを作成します。





マシンを作成する場合、1つ前に作成された仮想ネットワークを指定する必要があります。



便宜上、ビルドサーバー用に既製の仮想マシンを使用できます。ライブラリには、Visual Studio 2013 Ultimateがプレインストールされたマシンがあります。 残りのマシンは空です。これはWindows Server 2012または2008 R2です。

クラウドインフラストラクチャ全体が正しく機能するには、クライアント（Visual Studio、Microsoft Test Manager）、TFSサーバー、およびこれらのマシン自体の間の接続に追加のエンドポイントを追加する必要があります。



したがって、残りのマシンについては以下を示す必要があります。



  1. ct -lab ：137,138,139,6901,6910
  2. ag -lab ：137,138,139,6910


さらに、これらの仮想コンピューター間の名前が仮想ネットワークで表示されることを確認する必要があります。 これを行うには、DNSサーバーを作成（または既存のDNSサーバーを使用）し、仮想ネットワークに登録します。 スクリプトのニーズには、ホストの編集で十分です。 DNSのセットアップの詳細については、Azureポータルをご覧ください 。 DNSサーバーがない場合、テストコントローラーとテストエージェントの構成中に、エラーCreateControllerObject：試行0、System.Net.Sockets.SocketException（0x80004005）が発生する可能性があります：そのようなホストは不明です。 これを解決するために、 DNSマシンのアダプターにサフィックスloudapp.netを規定します 。



構成



Test ControllerとTest Agentを使用してct-labとag-labをインストールします。 一般に、Test ControllerマシンにTest Agentをインストールすることは推奨されませんが、そうすることは禁止されていません。 ここに示す3台のコンピューターのインフラストラクチャのシナリオ全体を1台の仮想マシンにインストールできます（これにより、DNSおよびエンドポイントとのダンスが簡単になります）。



ct-labコンピューターは、最初にすべてのテストコントローラーになります。 配布キットからファイルを短時間インストールすると、構成手順が実行されます。





したがって、ag-labマシンにテストエージェントをさらにインストールして構成します。







すべてを正しく行った場合、構成前の予備チェックでは、すべてのポイントで緑色のマークが表示されます。



テストマネージャーの構成



環境インフラストラクチャを作成したら、Microsoft Test Managerでラボ環境を構成する必要があります。





そこに仮想コンピューターを追加し（フルネーム-ag-lab.cloudapp.net）、ドロップダウンメニュー（デスクトップクライアント、サーバー）からロールを指定し、コントローラーを構成します。





構成検証手順の後、しばらくして（3〜5分）、環境内の準備完了ステータスがオンになります。 ag-labマシンはインタラクティブに構成され、テストエージェントはそこで開始する必要があります。





サーバー構成を構築します。



Azureライブラリの完成したVisual Studio 2013 Ultimate仮想マシンにはTeam Foundation Server 2013が含まれていないため、MSDN WebサイトからWebインストーラーをダウンロードし、bc-labマシンにTFSをインストールします。 次に、サーバーの構成手順をスキップして、Build Server構成項目のみを選択します。

必要なパラメーターを指定し、bc-labマシンでビルドコントローラーとビルドエージェントを実行します。 残念ながら、Lab Managementインフラストラクチャは専用のビルドコントローラーでのみ動作します。TeamFoundation Serviceの一部である仮想コントローラーはこの目的には適していません（複数のビルドエージェントを作成することはできません）。



ビルド構成。



スクリプトを機能させるには、bc-labコントローラーに関連付けられた2つのビルド定義を作成する必要があります。



  1. ビルドビルド自体 ：このビルドは、プロジェクトの最新バージョンを取得し、コンパイルし、単体テストを実行します。
  2. テストビルドの展開と起動 ：次に、ビルドはバトンを取得し、バイナリファイルをテストマシンにコピーし、自動テストを開始します。


最初のオプションの設定に問題はないと思いますが、2番目のオプションについて詳しく説明します。 標準パラメーター（ソースの取得元、ビルドコントローラー）を入力したら、[プロセス]タブでビルドテンプレートを変更する必要があります。







[ラボプロセス設定]構成ボタンをクリックし、ウィザードをステップ実行します。 CloudEnvを環境名（Microsoft Test Managerで作業するステップで作成しました）およびコンパイルビルドとして指定します。 次は、展開スクリプトを指定する必要がある重要な手順です。





最初のスクリプトはデスクトップクライアント（ag-labコンピューター）のロールで実行され、2番目のスクリプトはサーバー（これはct-lab）のロールで実行されます。 これらのマシンのc：\ toolsディレクトリにスクリプトを配置する必要があります。 最初のスクリプトは、$（BuildLocation）パラメーターを受け取り、収集されたビルドバイナリをネットワークドライブからコピーします。



SET "inputpath =％1"

xcopy％inputpath％c：\計算機/ E / R / Y



2番目は何もせず、単なるエコーパラメーターであり、ここでは例として示します。 実際、サーバーには何も展開していません。

展開スクリプトがデフォルトで実行されるユーザーは、システムアカウントです。 一方では、これにより、アクセス違反を恐れることなくスクリプト内で何でも実行できますが、他方では、ネットワークとのやり取りが多少制限されます。

パスワードをcmdに登録して、ビルドが配置されているネットワークドライブを手動で接続する必要がないように、最も簡単な方法は、ビルドエージェントがインストールされているマシンでVisual Studio Lab Agent Serviceサービスのパラメーターをやり直すことです：

テスト計画、テスト構成、およびテスト設定は、Microsoftテストマネージャーで作成する必要があります。



自動テストを作成する



最後のステップは、テストを作成および構成することです。 テストプロジェクトに新しいコード化されたUIテストを追加します。 ゼロから作成することも、テスターがアクション記録として記録したものを基礎として使用することもできます。 テストが作成されたら、テストケースにバインドする必要があります。これはVisual Studioで行われます。



その後、このテストケースがテストスイートに追加されます（この場合、Auto-BVTと呼ばれます）。





単体テスト中にCodedUITestが開始されないようにするには、テストカテゴリを使用できます。 テストコードに属性を直接追加するだけです。



次に、ビルドパラメータをフィルタリングします。





どのようにすべてが一緒に機能しますか。



これで、開発者はCalculator-BVTビルドのビルドを開始するだけで十分で、次のことが起こります。



  1. Build Calculator-BVTはBuild Calculatorを呼び出します
  2. bc-labコントローラー上のBuild Calculatorは、クラウドTFSからプロジェクトの最新ソースを受け取ります
  3. 次に、プロジェクトがコンパイルされます
  4. 実行された単体テスト
  5. すべて問題なければ、制御はCalculator-BVTビルドに戻ります
  6. このビルドは展開スクリプトを実行します
  7. そして、テスト計画からの自動テスト

次は何ですか。



プロジェクトを組み立ててテストする方法には、多くのオプションがあります。 たとえば、ビルドでAzure Webロールのプロジェクトをビルドし、自動的に展開できます。 そして、Webプロジェクトの自動テストを実行できます。 テスト環境では、マシンの数が増減する場合があります。 ただし、それらの一部は組織のネットワーク上にある場合もあります。 また、ビルドサーバーをコントローラーと「一緒に」残すこともできます。 一般に、必要なレベルの柔軟性を得ることができる多くの構成オプションがあります。



最も重要なことは、テスト環境を構成するコストを削減できることです。 より多くのステップが自動化されるほど、より多くのテストを実行でき、人的エラーの可能性が少なくなるため、意思決定が改善されます。
