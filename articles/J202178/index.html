Realtek RTL-1185メディアプロセッサでLinuxを実行する





今日、SoC RTD1185 -RTK300 Rev.に基づいたデバッグボードを使用しました。 C1-マルチメディアデバイスの開発用。 この記事のフレームワークでは、このSDKの技術的なパラメーターに精通し、ベースのLinuxカーネルとその上のrootfsを収集して実行し、プロセスのいくつかの問題を解決します。



免責事項：この記事は経験のあるLinuxユーザーを対象にしていますが、少なくとも細部については触れていません。 ご質問があれば、コメントしてください。



Realtekの友人によるチップ上のこのシステム（コード名Jupiter、彼女の兄弟RTD1186など）は、HD MPEG 1/2/4、H.264、VC1、RM / RMVBビデオ形式をデコードするマルチメディアアプリケーション向けに設計されています。 メディアプロセッサは、USB 2.0およびSATAコネクタ、PCI-Expressバス、およびギガビットイーサネットネットワークインターフェイスもサポートしています。



対応するネットワークコントローラーと組み合わせたイーサネットの存在により、最大1ギガビット/秒の速度で有線ネットワークを介してデータを転送できます。 このデバイスは現代の標準ではCPU周波数が小さくなっていますが、このパラメーターは強力なビデオデコーダーによって補正されており、マルチメディアアプリケーションでのこのチップの使用を正当化しています。



RTD1185の技術的パラメーター：

  • CPU：500 MHz
  • HD MPEG 1/2/4およびHD JPEGデコーダー
  • HD H.264、VC1、RM / RMVB、AVSデコーダー
  • CVBS / Sビデオ/ YPbPr / SCART出力付きTVエンコーダー
  • CEC付きHDMI v1.3
  • I2S、SPDIFアウト
  • USB2.0ホストおよびPHY
  • USB2.0デバイスとPHY
  • SATA
  • 10/100イーサネットMACおよびPHY
  • ギガビットイーサネットMAC
  • PCI Express
  • カードリーダー（SD / MMC）
  • DTV録画とタイムシフト
  • デュアルTS-In


Realtek RTL-1185キット



ボードには、IRリモコンが付属しています。 Realtekから「半分機能する」SDKを受け取りました。これは、ファームウェアアセンブリに関するすべての問題を解決しなければならなかったはずです。 「タンバリン」なしでlinux-2.6.12と基本的なrootfsを構築できたため、「ハーフワーキング」と呼びました。 しかし、オーディオおよびビデオプレーヤーのアプリケーションに関しては、「タンバリン」を使用する必要がありました。 まあ、ありがとう。 クジラは下の写真に描かれています。







ボードには次のコンポーネントが含まれています。

  1. SoC RTD11185
  2. NANYAから256MB RAM
  3. 256MB NAND SLC-フラッシュサムスン
  4. 2 x 2.0 USBホスト
  5. FastEthernet RJ45ソケット
  6. HDMIトランスミッター出力
  7. リモートコントロール用のIRレシーバー（リモートコントロールが利用可能、下の写真）
  8. オン/オフを切り替えるロックボタン
  9. 1つのUSBホストに挿入されたRTL8190Uに基づくUSB WiFi
  10. ピンプラグ（UART）
  11. 回復ボタン

ファームウェアの更新


まあ、悪くない。 組み立てられた形式でSDKに提示されたファームウェアの更新を試みます。

  1. まず、USBドライブをフォーマットする必要があります。 

     host# mkfs.vfat /dev/sdx -n firmware
  2. install.imgファームウェアイメージをこのドライブにコピーします。
  3. USBフラッシュドライブを挿入し、ソフトウェアの更新プロセスを開始します。 この「プロセス」は、主に2つの方法で起動できます（他にもあります）。


  • 回復ブートモード：「回復ボタン」を押してボードを有効にする必要があります。 同時に、ブートローダーはフラッシュ内の別のアドレスにカーネルを移動します。つまり、フラッシュ中に間違ったカーネルを縫い合わせてボードからブリックを作成すると、いつでもリカバリモードで起動できます。
  • 標準ダウンロード：GUIで適切なアイテムを選択します。 このオプションについては詳しく説明しません。


すべてが一目でわかるシンプルなものです。 フォーマット。 ファームウェアをUSBフラッシュドライブにコピーします。 デバイスに挿入します。 回復モードで起動します。 フラッシュドライブが自動的にマウントされ、ファームウェアが起動します。 loader_aユーティリティは、これらすべてを担当します。 しかし...

ベースLinuxカーネルとrootfsをビルドします


Realtek SDKを使用してカーネルとrootfsをビルドするときが来ました。 ArchLinux x86_64で作業しているため、通常、Ubuntu 12.10 amd64 chrootの組み込みソフトウェアのすべてのアセンブリ操作をいくつかの理由で実行します。

  1. 多くの場合、ビルドする必要があるすべてのパッケージはすでにUbuntuにあります。
  2. 組み込み開発者は保守的で、多くの場合、ファームウェア（rootfs）を構築するために必要なホストユーティリティが特定の古いバージョンのgccおよびライブラリによってコンパイルされるSDKがあります。また、常に新しいバージョンのgccがarch 。 パッチをいじったり、自分で修正するよりも、chroot rootfsをデプロイする方がはるかに簡単です。
  3. ほとんどのSDKは、Ubuntuでもまったく同じようにデプロイおよびテストされます。 Ubuntu自体でもありません。 同じ成功で、必要に応じて別の配布キットを展開できます。
  4. お客様にchrootと共にすべてを転送することができ、お客様のファームウェアはすぐに高い確率で組み立てられるため、お客様にアセンブリシステムを展開するのは簡単です。 ただし、これは、顧客がWindowsで作業していない場合です（この場合、仮想マシンを使用しますが、これは意味がありません）。


chroot用のUbuntu rootfsは、AURから取得したdebootstrapユーティリティを使用してデプロイされます。

  • ubuntu chrootを展開するwiki.ubuntu.com/DebootstrapChroot;
  • chrootのproc、sysfs、devtmpfs、devptsをマウントします。
  • ホスト＃chroot ubuntu-12.10
  • Ubuntu＃apt-get update
  • ubuntu＃apt-getインストールvimファイルlib32z1-dev libc6-i386 sed dos2unix python g ++-4.4-multilib
  • ubuntu＃useradd -s / bin / sh -m developer


SDKファイルをubuntu-12.10 / home / developer / Jupiterにコピーします。



  • Ubuntu＃su developer / bin / bash
  • ububtu $ cd / home / developer / Jupiter / linux / system
  • ubuntu $ make menuconfig＃NFSおよびNFSルートサポートを有効にする
  • ubuntu $ make PRJ = development.avhdd.jupiter.nand.loongtle


これらの操作の後、カーネルと基本的なrootfsを取得します。



nfsルートを使用するために、rootfsを/ srv / nfs / jupiterにコピーします。 / etc / exportsに対応する行を追加します。

ホスト＃exportfs -a



カーネルを/srv/tftp/vmlinux.develop.avhdd.jupiter.nand.loongtle.binのtftpサーバーにコピーします。



ホストでtftpおよびnfsサーバーを起動します 。 イーサネット経由でデバッグボードに接続されたホストNICを構成します：Ipv4 192.168.0.1/24。
