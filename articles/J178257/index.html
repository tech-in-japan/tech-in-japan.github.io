今日のパスカル、デルファイ、オベロン

みなさんこんにちは。



まず、先日、DelphiとRAD Studio XE4に関するニュースレターがありました。Delphiは、コンピューター、ラップトップ、タブレット、およびMAC、iPhone、iPADを含むモバイルデバイスの迅速な開発のためのマルチプラットフォームツールとして宣言されています。 Androidモバイルプラットフォーム。



DelphiとFreePascalは現在Pascalです。 私のお気に入りのプログラミング言語の1つ。 この言語は、「科学者向けのおもちゃ」から、強力な普遍的な開発ツールに至るまでの長い道のりを歩んできました。たとえば、 Total Commanderなどの優れたプロジェクトが作成されています。



健全なソリューションに基づいて構築されたPascalのコアは、改善と改良に容易に陥り、幅広いクラスの実用的なタスクに適しています。 ロシアでは、Delphiは特に人気があり、現在でも広く使用されています。 ただし、言語の開発はキャパシティビルディングの原則に基づいており、Delphiは次第に扱いにくくなり、PL / 1、C ++、Adaなどのモンスターの複雑さに追いつきます。 高い確率で、蓄積されたソースデータベースなどとの互換性を維持する必要があるため、言語が簡単になることはないと想定できます。 最終的に、このような開発戦略は、複雑さを制御できなくなると、必然的に困難を招きます。



同時に、Pascalの系統的かつ一貫した改良を通じて、数十年にわたって実践的な教育とさまざまな問題を解決してきたプログラミング言語があります。 これはOberonです。これは、ほとんど聞いたことのない最小限の言語の中で最も美しい言語です。 私は安全に言うことができます：今日のPascalは、Delphi（構築方法によって取得）だけでなく、 Oberon-2 / Component Pascal （特定の手段の実行可能性を破棄、修正、および明確化する方法によって取得）です。 言語の作者は互換性の必要性に支配されていなかったため、このトピックについて特に心配していませんでした。 そして、結果として得られる言語は、広義のワンピース、コンパクト、強力、クロスプラットフォームです（最新のプラットフォームだけでなく、 Amiga 、さらにはAtariを含む ）。 Component Pascalでは、Javaマイクロエディション用のMIDletの開発に成功しましたが、 Androidの開発の難しさは予見していません。 1つを除いて：インフラストラクチャの不足について、そして実際にそのようなOberonsの使用について知っている人はほとんどいません。



Oberon-2およびComponent Pascalは、システムおよびアプリケーションプログラミング用のマルチパラダイム言語であり、モジュール式、コンポーネント、およびオブジェクト指向です。 オペレーティングシステム、コンパイラ、およびOberon方言で開発された他のプログラムは、多くの場合、それ自体が芸術作品です。 それらは美しく、並外れた、コンパクトで、すべての賞賛に値します。 Oberonは、アルゴリズムを書く方法（Fortranに置き換わる）および研究用の最小限の言語としてアカデミック環境で人気があります（例えば、 SpringerLinkのWebサイトにOberonに関する出版物がいくつ登場したかを参照） プログラミング教育に最適です。 ロシアには、Informatics-21プロジェクトがあります。これは、Oberonの技術的な利点に加えて、Oberonがプログラミングテクノロジを十分に理解しており、大企業の商業的利益に依存しないためです。 私のプログラミングについての1週間の話の後、私の若いが刺激したものを見てください。 言語がシンプルであるため、愛好家はさまざまなプラットフォーム向けに多くのOberon方言コンパイラを開発しました。



Wirthは、言語用のプロセッサを設計することは常に正しいと考えており、逆も同様です。 しかし、それにもかかわらず、OberonプログラムはC、Javaにうまく変換され、ネイティブおよび.NET / JVMの両方で動作します。 LLVM用のGPCPコンパイラは開発中です。 私は、8ビットZ80プロセッサーおよびZX Spectrumコンピューター用のOberon-2上の開発環境であるZXDevのアイデアを所有しています。 Wirth Oberonの最新版はOberon-07と呼ばれます。 サイズはOberon-2よりもさらに小さく、ARMプロセッサと高解像度アーキテクチャに適合しています。 これがPascalの開発の方向です。 プログラマーの大衆にはほとんど知られていないが、芸術作品としては美しい。 7年以上にわたり、仕事の利益のためにOberonテクノロジーを積極的に使用してきましたが、他には何もプログラムしたくありません。



したがって、私は、PascalやDelphiが好きで、レトロなプラットフォーム向けのコーディングが好きな愛好家にアピールします。 クロスプラットフォーム開発のベースとして設計されたXDev環境は、あなたが思っていたよりもはるかに興味深いツールです。 さまざまなプラットフォームのターゲットコードにさまざまな翻訳スキーム（さまざまな言語から）、開発を容易にするユーティリティ、ビジュアライザー、および特定のプラットフォームとクロスプラットフォーム向けのライブラリセットを実装します。 XDevはモジュール式のコンポーネント環境であり、サブシステムはさまざまなプラットフォームの開発に使用されます。 最も豊富な（そして準備が整った）ZXDevサブシステム（ターゲットプラットフォーム：ZX Spectrum）です。 ZXDevは、CでOberonおよびOberon-2言語からの翻訳スキームを実装し、 SDCCコンパイラーにより、CおよびコンパイラーはすでにZ80プロセッサーのマシンコードに翻訳されています。 このセットには、 smartlibユーティリティ（ライブラリでの便利な作業と、プログラムで実際に使用されるマシンコードのみをターゲットバイナリに含めるため）と、ターゲットTAPおよびTRD形式へのコンバータが含まれています。 コンソール、レーザー、基本、グラフなど、いくつかのライブラリがあります。 ZXDevでの開発は、アセンブラーインサートを使用したOberon-2またはCの両方で実行でき、同時にすべての言語で同時に実行できます。



XDevのすべてのレベルで、ほぼ1つの言語が高レベルツールとして使用されます-Oberon-2。 もちろん、C言語は中間プレゼンテーションに使用され、BATファイルのコマンド言語はOberonコンポーネントとサードパーティツールをリンクするために必要ですが、一般に、特にComponent PascalはOberonとOberon-の正しいスーパーセットであるため、単一言語を目的としています2。 したがって、1つの言語-XDev自体の開発用と、ターゲットプラットフォームのプログラミング用の両方。 私たちのフォーラムでは、Oberon言語の開発を試みており、実際の開発に必要だと思われる手段でそれを増やしています（たとえば、Oberon-07はさまざまなタイプの整数を失い、したがって8ビットプロセッサの開発には適していません）。



しかし、XDev環境では、もちろん、他の翻訳スキームを実装し、他のプログラミング言語を追加できます（計画：Component Pascal、 Modula-2 Revision 2010、Objective Modula-2 、および場合によってはオブジェクトの方言または通常のPascal）およびターゲットプラットフォーム。 XDevコンポーネントフレームワークは、拡張可能でオープンソースであるBlackBox Component Builder Oberon Environment （Oberon Microsystems）を使用します。



ZXDevに加えて、サブシステムWinDev、LinDev、DosDev、およびJmeDevも（Java microeditionバイトコードで）開発されており、原則として、Android / iOS、MSX、ColecoVision、NES / SEGA、Atari、 Amiga、Palm OS、さらにはDV-K / BK-0011のUK-NTの下、Apogee / Krista / RK-86のMikroshの下でも。 そして、あなたは、XDevのために、あなたが興味を持っているあらゆるプラットフォーム（現代とレトロの両方）をサポートする機会を持っています。 これが私の趣味であり、最終的にはプログラミングに対する長年の情熱につながりました。 それはすべて、英国NTのPascal / Fastで始まりました。 :)



だから、私たちの活動について、そしてもちろんOberonテクノロジーについてもっと学ぶために、私たちのフォーラムにみんなを招待します 。 XDevの開発に貢献し、新しいアイデアを支援し、決定に影響を与え、新しいライブラリを開発/適応させたり、新しいプラットフォームを追加したりできます。 言語や何かに制限はありません。XDevの基本は、今日のIT市場を支配しているアメリカのC字型のパラダイムとは質的に異なる、ヨーロッパのプログラミングスクールをモジュール言語で開発することです。 ただし、Cが好きなら、XDevはC開発に非常に適していますが、XDev環境よりも従来のIDEを好むでしょう。
