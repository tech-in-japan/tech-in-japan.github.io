CMSアーキテクチャ

十分な時間があるため、プログラマーとそのユーザーコンフィギュレーター向けに、柔軟で多機能で比較的単純なCMSを作成することに夢中になりました。 いくつかのアイデアは、CMS Drupalの潜在的な機能に触発されました。 そして今、アイデアの実装と成功した中間結果に魅了された私は、作成されたCMSのアーキテクチャについて話すことにしました。 読む過程で、CMS機能の一般的な図は、プログラマーのために、それ自体で作成される必要があります:)



CMSアーキテクチャ



作成されたシステムのアーキテクチャは、モジュール方式に基づいています。 CMSは、キューブの子供の家のようなモジュールで完全に構築されています。 モジュール間の対話の柔軟性と単純さを実現するために、ハードリンクを作成する必要なく、システムの任意のコンポーネントでモジュールを使用するための最小限のアクションを達成する必要があり、他のモジュールを使用してモジュールのアクションを簡単に拡張できるようにする必要がありました。

モジュール



作成されるシステムの観点からモジュールは、名前がモジュールの名前によって決定される静的クラスです。 多くのモジュールが機能するためには、1つの静的クラスでは不十分です-別のファイルに実装された追加のプログラム構造を使用できます。 すべてのモジュールファイルは1つのディレクトリ（フォルダー）に配置され、モジュールディレクトリはモジュールに配置されます。





Webディレクトリのルートには、index.phpファイルとconfig.php基本設定ファイルがあります。 サイトへのリクエストに対して、サーバーはindex.phpファイルを実行します。それからシステムの機能が始まります。



システムモジュール



index.phpでは、Systemモジュールとconfig.php構成ファイルが接続されています。 構成ファイルをロードした後、制御はシステムモジュールに転送されます。

システムモジュールは、他のすべてのモジュールの自動接続を提供し、システムの中核であり、システムを活性化します。 モジュールの自動接続は、php関数__autoload（$ class_name）によって提供されます。 この関数は、システムモジュールファイルで定義されていますが、クラスの外部にあります。 アンロードされたクラスにアクセスするとき、PHPインタープリターはこの関数にアクセスし、関数はシステムモジュールにアクセスします。 システムモジュールは、要求されたクラス（モジュール）の存在、そのphpファイルの場所を決定し、それらを接続します。 



 /** *    (). *      PHP * @param string $class_name   (). */ function __autoload($class_name){ System::IncludeClass($class_name); }




クラスファイルをシステムモジュールにアタッチすると、接続されているクラス（存在する場合）の静的なActivate（）メソッドが自動的に呼び出され、静的クラスを初期化できます。 覚えているように、モジュールは静的クラスであり、多くは作業前に初期化が必要です。



したがって、モジュールを使用するために、事前に宣言または接続する必要はなく、モジュールクラスのメソッドまたはプロパティにアクセスするだけで、モジュールへのリンク（ポインター）を作成する必要はありません。 これは、システムの任意のコンポーネントによるモジュールの単純な使用の原則です。



イベントモジュール



イベントモジュールを使用すると、モジュールによってイベントを生成できます。 イベントは、 Events::Send($module_name, $event_name)



呼び出すことで生成されます。 イベントは他のモジュールのクラスの静的メソッドによって処理されますが、モジュールが自身のイベントを処理することを妨げるものはありません。



システムモジュールは、システムのそれ以上の機能を決定せず、「初期化」（INIT）、「実行」（START）、および「完了」（STOP）の3つのイベントを順番に生成するだけです。 



 static function Process(){ //   : //  Events::Send('System', 'INIT'); //  Events::Send('System', 'START'); //   Events::Send('System', 'STOP'); }


どのモジュールでもこれらのイベントを処理でき、それによりシステムの機能に参加できます。 ただし、システムモジュールだけがイベントを生成するわけではありません。 この図は、システムの主要なイベント（円）と、イベントを処理するモジュールとの接続（ストリップ内の線）を示しています。

エラーモジュール



システムモジュールの初期化イベント（INIT）は、まずエラーエラーモジュールを処理し、自動的に起動します。 エラーモジュールは、標準のPHPエラーおよび例外ハンドラをオーバーライドします。 それについて詳細に話す必要はまだありません。 エラーモジュールは、エラー処理の後にイベントを生成するだけであるため、エラーの原因に関係なく、出力を生成するモジュール（Pageモジュールなど）がユーザーにわかりやすい回答を生成することができます。



セッションモジュール



セッションセッションモジュールの操作には特別なものはありません。モジュールはシステムモジュールの初期化イベント中に起動され、ユーザーセッション識別子を決定してから、リクエストデータとしてセッションデータへのアクセスを提供します。



リクエストリクエストモジュール



システムモジュールの「START」イベントが要求モジュールを処理します。 クエリモジュールは、ユーザーからのすべての着信データを「コンテナ」に収集し、そこからシステムの任意のモジュールで利用できるようにします。 「コンテナ」からデータを取得するには、形式を指定する必要があります-期待値のタイプとサイズの制限。 したがって、モジュールの要件を満たさないデータの侵入はある程度防止されます。 入力データを「コンテナ」に配置するプロセスが完了すると、クエリモジュールは、メイン作業の完了時にAFTERイベントを生成します。



原則として、結論を作成する必要があります-ユーザーのリクエストへの応答、およびそれをクライアントに送信します。 答えは通常htmlページですが、XMLやファイルなどの別の形式のドキュメントでもかまいません。 ユーザー要求（url'a）のパラメーターに応じて、1つのモジュールが出力を生成するために決定されます。より正確には、各出力モジュールは、最終的に1つのモジュールのみが出力を生成するように動作するかどうかを決定します。 出力モジュールは、RequestモジュールのAFTERイベントを処理することで作業を開始します。これにより、新しいモジュールを簡単に追加して、他の形式の出力を生成できます。 原則として、少なくともユーザーに何かを送信するのであれば、どのモジュールでも出力モジュールにできます。



出力を生成する前に、着信データを処理する必要があります。 システムに保存する必要があるフォームデータにすることができます。 これは、Dataモジュールによって行われ、RequestモジュールのAFTERイベントを処理します。

データモジュール



データモジュールデータ-システムの最も重要なモジュールであり、システムの柔軟性と機能力が依存するのはその機能です。 データモジュールはデータ共有を提供します。 強力な機能を使用して、作成、変更、削除、および検索を行うことができます。 データベースモジュールを介してデータベースにアクセスするのは、データモジュールのみです。 データを操作するために、他のモジュールはSQLクエリを使用せず、クエリ条件とデータを操作する方法を記述するために、シンプルだが柔軟な方法を使用します。 条件付きデータモジュールは、データベースへのSQLクエリを作成し、データオブジェクトを返します。 「News」オブジェクトを受信すると、データモジュールに明示的にアクセスしなくても、ニュースに関連するすべての情報、さらにはニュースが属するセクションの作成者を見つけることができます。 データモジュールのおかげで、ユーザーコンフィギュレータ（およびシステム自体）は、独自のデータクラス（独自のデータ構造）を作成できます。 これについては、次の記事で詳しく説明します。 明確にするために、データクラスはプログラミング言語の構造クラスと直接関係がないことを付け加えます。



また、データモジュールはイベントを生成し、データオブジェクトが保存、読み取り、削除されたときに追加のアクションを実行できるように処理します。



ページモジュール



ページテンプレートモジュールは、デザインテンプレートを使用してhtmlページを形成し、ユーザーに送信します。 ページは、着信データ-要求パラメーター（url'a）に従ってブロックから形成されます。 ページはデータの表現であり、多くのフラグメント（ページブロック）もデータであるため、ページモジュールはデータモジュールを集中的に使用します。 ページ生成の原理とモジュールの機能のすべての微妙さは、次の記事のいずれかのトピックです。



画像 ファイルと写真モジュール



ファイルおよび写真モジュールは、データモジュールのアクションを補完するモジュールの1つです。 さらに、ファイルと写真のモジュールは、システムにインストールされると、それぞれデータクラス「ファイル」と「写真」を作成します。 次に、これらのクラスのオブジェクトを使用したデータデータモジュールの作業中に、ファイルまたは写真モジュールが追加のアクションを実行します。 特に、「ファイル」データオブジェクトを作成するとき、ファイルはサーバーにアップロードされ、サーバーファイルシステムに保存されます。 ファイルモジュールは出力モジュールでもあります。 適切な要求があれば、モジュールはダウンロードのためにユーザーにファイルを転送しますが、アクセス制御、統計、およびシステム全体のすべてを実行できます。



写真モジュールの主な機能は、写真を拡大縮小することです。 写真はファイルであるため、サーバーにアップロードできる必要もあります。これには、ファイルモジュールが使用されます。



アクションモジュール



アクションモジュールを使用すると、オブジェクトに対してデータモジュールによって実行されるアクションを分類および制御できます。 理論的には、論理式はアクションの条件になります。現在のユーザー（彼のプロパティ）のチェック、アクションが実行されるオブジェクトのチェックなどを含むことができます。



例：データモジュールデータは、データベースからデータをロードし、読み取りイベントを生成します。読み取りイベントは、アクションモジュールによって処理されます。 アクションのアクションモジュールは、このアクションの条件を確認し、その実装の可能性を計算します。計算の論理結果は、データデータモジュールによって受け入れられます。データデータモジュールは、結果に従うだけです。つまり、データオブジェクトの読み取りを続行または終了します。 条件とアクションもデータオブジェクトです。 アクションのアクションモジュールの機能の複雑さ、およびデータデータモジュールの学習後にアクションを制御する柔軟な可能性を理解することが可能になります。



フォーマットフォーマットモジュール



Formats形式モジュールの目的は、値にさまざまな形式を提供することです。 たとえば、文字列の場合、形式は電子メールアドレス、電話、およびデザイン形式（マークアップ）の表現になります。 形式は、値の形式を決定する値オブジェクトに関連付けられたデータオブジェクトです。 形式モジュールは、形式オブジェクトを提供するだけでなく、形式に対する値の検証も提供します。 検証自体は、データモジュールがデータオブジェクトを格納するときに発生します。 また、データを読み取るときに、形式に応じた値の変換（テキスト形式など）が発生します。 このモジュールの機能には多くの未解決の問題があります。 作業は継続されます。
