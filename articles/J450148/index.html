インテリジェントな診断と自動トラブルシューティング



プロパティの組み込みの問題アラート（アラーム）には、製造元からの情報を含む[ ナレッジベース ]タブがあります。 このアラートをトリガーした問題に関するナレッジベース記事の形式があります。











たとえば、構成に問題がある場合、詳細情報と理由（ 原因 ）および状況の解決方法（ 解決策 ）に関する通知を受け取ります。 また、たとえば、この問題の修正プログラムが既に存在する場合は、サポートに連絡して適切なファイルをリクエストするだけです。そのような例は上のスクリーンショットに示されています。



スマート診断Veeam Intelligent Diagnosticsは、組み込みの知識ベースに従ってアドバイスを提供できるだけではありません。 修復アクションを設定することもできます。



重要！ そのような自動修復アクション（組み込みまたはカスタム）がVeeam Backup＆Replicationサーバーで機能するには、Veeam ONEエージェントコンポーネントをインストールする必要があります。 これはここで詳細に説明されます 。



たとえば、保護されていない仮想マシン（またはそれ以上）が検出されるという組み込みのアラートを検討してください。 ここで何ができますか？



  1. まず、Veeam ONEモニターコンソールでアラーム設定を開き、 [ アクション ]タブに移動します。









  2. ここでは、 アクションリストで、バックアップが検出された場合に、バックアップのないVMの状況を修正する方法を選択します。 可能なアクションの完全なリストは、 ここにあります 。

    [ バックアップジョブにVMを追加]を選択しました 。



    便利：独自の通知のために、実行するスクリプトを指定することにより、独自の修復アクションを構成できます。 これを行うには、[ アクション ]タブの通知設定で[ 追加 ]ボタンをクリックし、[ アクション ]リストで[ スクリプトの実行 ]を選択します 。 次に、[ スクリプトへのパス]フィールドに、目的のスクリプトへのパスを入力します。



    重要！ スクリプトはVeeam ONEサーバーで実行されます。 Veeam ONEサービスアカウントのスクリプトファイルへのアクセスを提供する必要があります。

  3. [ 解決タイプ]フィールドで、選択したアクションを実行するために最初にその人の承認を取得する必要があるかどうかを示します。



    - 自動 -手動操作は不要です。通知がトリガーされた後、プログラム自体が指定されたアクションを開始します。

    - 手動 （組み込みアラートに対してデフォルトで選択）-アラートがトリガーされた後、指定されたアクションに同意する必要があります。 これは次のように行われます。



    1. Veeam ONE Monitorコンソールで、目的のビュー（インフラストラクチャビュー、ビジネスビュー、vCloud Directorビュー、データ保護ビュー）と目的のオブジェクトを選択します。
    2. 右側のパネルで、[ アラーム ]タブに移動します。
    3. 上部の修復可能なアイコンを表示をクリックして、関連するアクションの実装を承認する必要があるアラートを確認します。
    4. 次に、通知のリストで必要な通知を選択し、右クリックして[ アクションの承認 ]アイテムを選択するか、右側の[ アクション ]パネルで選択します。
    5. 必要に応じて、[ 是正措置の承認 ]ダイアログにコメントを入力します（変更のリスト（履歴の詳細）の[ コメント]フィールドと、設定している場合は電子メール通知に表示されます）。
    6. OKをクリックします。


組み込みのアクションは1つしか選択できず、カスタムスクリプトで必要な量を指定できることに注意してください。



Veeam ONEは、イン​​ラインアクションまたはスクリプトの実行を3回試行します。 すべてがうまくいけば、通知ステータスはAcknowledgedに変わり、そうでなければ通知はアクティブのままになります。



ヒートマップ



ヒートマップは長い間Veeamのソリューションに登場しました-Veeam Management Packが最初にそれらを受け取りました（これはブログの記事で、「コンテキストダッシュボード」と呼ばれています）。 現在、それらはVeeam ONE Reporterにも実装されており、同様にインフラストラクチャの状態を把握し、どこで問題が発生したかを迅速に見つけるのに役立ちます。 このすべての視覚化は、[ ダッシュボード ]タブのヒートマップウィジェットをクリックして使用できます。







ここでは、たとえば、バックアップインフラストラクチャの健康な人のヒートマップのように見えます。Veeamバックアッププロキシの負荷は、適度にバランスが取れています。







左側のウィジェットは、リポジトリ内のスペースの大きさを示しています-それほど多くないことがわかります。 中央のウィジェットは緑色で表示され、2つのプロキシサーバーの負荷が均等に分散されます。



注意！ プロキシによっては、ダウンロードが濃い緑色で示されている場合、これはプロキシが完全にフリーである、つまりアイドル状態であることを意味します。



任意のプロキシサーバーをクリックして、日中の負荷の大きさを確認できます-朝、バックアップウィンドウに負荷がかかることがわかります。







次に、バランスのとれていないインフラストラクチャを見てみましょう。そこでは状況が異なります。







ここでは、3つのプロキシがありますが、そのうちの1つは明らかに他のプロキシ（黄緑色のインジケータがあるもの）よりも多くロードされています。 詳細に調べてみると、ハードワークの主な期間は夜間の労働時間にあることがわかります。 残りのプロキシの負荷が低いため、バックアップウィンドウにはかなり長い時間がかかり、負荷を再配分する方が合理的です。



このプロキシのインジケータをクリックしてから、中央のウィジェットの時間間隔をクリックして、不幸なプロキシ輻輳の原因を理解します。ここでは、プロキシの構成や処理するバックアップタスクなどの詳細情報が表示されます。







最も一般的な構成エラーには、プロキシ設定が特定の種類のバックアップタスクに適していない場合が含まれます。たとえば、設定自体がホットアドモードの使用を指定している場合、プロキシ自体は物理サーバーです。 当然、そのようなサーバーはタスク用に選択されることはなく、負荷全体が残りのプロキシに落ちます。 その結果、バックアップウィンドウが長くなりますが、これはもちろん望ましくありません。



プロキシに加えて、ヒートマップでは、リポジトリ（スケーラブルなスケールアウトバックアップリポジトリを含む）のステータスを監視することもできます- リポジトリ使用状況ウィジェットは、1週間のバックアップ中のタスクの並列処理でリポジトリがどの程度ビジーであったかを表示します。



Veeam ONE Reporterのヒートカードの詳細は、 ドキュメント （英語）に記載されています 。



アプリケーション監視



Veeam ONE Monitorに実装されたこの新機能により、仮想マシンのサービスとプロセスの動作を詳細に監視できます。







選択したVMでのSQL Serverサービスの動作を見てみましょう。 まず、この作業を停止または再開できます。VeeamONE Monitorは、サービスコントロールマネージャーと通信して、 開始 、 停止 、および再開の操作を実行できます。



次に、たとえばサービスのステータスを変更するためにトリガーされるアラートを設定できます。このためには、サービスを右クリックして、[ アラームの作成 ] > [状態 ]を選択します 。







たとえば、サービスが5分間機能しない場合にエラーをスローするように、通知設定を設定できます。 この状況では、ほとんどの場合、サービスを再起動する必要があります。 アラートのトリガールールは次のようになります。











これらの設定では、サービスの状態が5分間の実行と異なる場合、通知はエラーになります。 次に、[ アクション ]タブに移動して、修正するアクションを指定します。 この例では、これはサービスを再起動するPowerShellスクリプトの自動起動になります。 スクリプトへのパスを示し、修復アクションのタイプは自動であると言います（確認なし）。











有用：サービスの再開が通知されるようにユーザーを構成することもお勧めします。



もう1つの便利な例は、アラートを設定してサービスの数を増やすことです。 このようなアラートは、たとえば、重要なマシンに1つまたは複数の新しいサービスがインストールされた場合にトリガーされます。 これは、ソフトウェア（またはマルウェア）の不正なインストールの結果である可能性があります。



プロセスのアラートを設定できます。たとえば、リソースの使用方法やパフォーマンスなどです。 これを行うには、[ ルール ]タブの通知プロパティで、 ルール タイプを選択します：ルールタイプ：プロセスパフォーマンス 。



次に、CPU使用率を監視し、指定されたしきい値を下回った場合にアラートを生成することを示します。 しきい値は、エラー（ エラー ）を生成するだけでなく、警告（ 警告 ）に対しても設定できます。











この例では、CPU使用率が10％減少すると警告が発生し、エラーが25％減少します。



Veeam Agentsの新しいレポート



バージョン9.5 Update 4では、Veeam Agentバックアップジョブの作業に関する3つの新しいレポートが表示されました。



  • アーカイブコピーのないコンピューター -その助けを借りて、 どのコンピューターにバックアップがないかをすばやく見つけることができます。
  • コンピューターバックアップステータス -エージェントが動作する物理マシンのバックアップステータスを毎日レポートします。
  • エージェントバックアップジョブとポリシー履歴 -Veeam Agentのすべてのポリシーとタスクに関する履歴データを提供します。


すぐに使用できる十分なレポートがない場合は、複合カスタムレポートバックアップインフラストラクチャカスタムデータを設定し、Veeam Agentの動作に関する必要な情報を含めることができます。



最後の2つのレポートについて詳しく説明します。 まず、 エージェントバックアップジョブとポリシー履歴レポートを見てみましょう。



  1. Veeam ONE Reporterを起動し、ワークスペースビューを開きます。
  2. ビルトインレポートのフォルダーに移動しますVeeam Backup Agent Reports。
  3. 「エージェントバックアップジョブとポリシー履歴」レポートを選択し、レポートのスコープを選択して、レポートに含める期間のデータを指定します。



    有用：同様に、Veeamバックアップサーバーを選択するか、特定のタスク/バックアップポリシーを指定できます。

  4. [ レポートのプレビュー]をクリックし、レポート生成の終了を待ちます。 表示は次のとおりです。









特定の日付のリンクをクリックすると、その日のタスクの動作を確認できます。開始時刻、所要時間、結果のバックアップのサイズ、完全バックアップか増分バックアップかを確認できます。



次に、 バックアップインフラストラクチャのカスタムデータレポートを見てみましょう。 これを使用すると、すぐにレポートを提供しないデータを受信できます。 これを行うには、レポートデザイナーを使用できます。



Veeam ONE Reporterで、[ カスタムレポート]を選択し、 レポートを見つけて指定します。



  • 範囲 ：Veeam Backup＆Replicationサーバーがレポートに含まれるデータ。
  • オブジェクトタイプ ：関心のあるオブジェクトのタイプ（バックアップサーバー、バックアップジョブ、仮想マシン、コンピューター）。
  • 列 ：選択したオブジェクトのタイプに応じて、ここでレポートを使用して分析するプロパティを指定できます。 列に表示されます。 便宜上、[フィルター]をクリックしてフィルターできます。 レポートに含めることができるプロパティは50個までです。
  • カスタムフィルター ：これらの列に独自のフィルターを設定できます。
