stm32f4-discoveryからのSIP電話

みなさんこんにちは。

この記事では、Embox組み込みOSを使用して、stm32f4-discoveryに基づくSIP電話をどのように作成したかを説明します。 stm32f4-discovery（144MHz、192Kb RAM、1Mb ROM）の特性により、そのような実装の可能性について疑問が生じる場合があります。 私たち自身はうまくいくかどうか疑問に思っていました。 答えとして、ビデオを見ることができ、記事自体で-技術的な詳細。









仕組み



このデバイスのソフトウェアは、PJSIPソフトフォンと前述のEmbox OSで構成されています。 PJSIPは、SIPプロトコル、サウンドデコードのサポートを担当します。 Emboxは、PJSIP標準ライブラリ、pthreadライブラリ、ネットワークスタック、オーディオ出力用ドライバー、およびOS機能の残りのセットを提供します。 このために、POSIXインターフェースが使用されます。これにより、Unixのようなオペレーティングシステムから組み込みシステムに、ほとんど変更を加えることなく幅広いクラスのアプリケーションを転送できます。 PJSIPをEmboxに移植することは簡単であると期待されていましたが、特定のハードウェアプラットフォームでPJSIPを起動するのは困難でした。 それらについて-さらに。



私たちの成功を繰り返したい人のために、私たちは小さな指導を用意しました。 stm32f4discovery + stm32dis-bb、com-port接続（拡張カード上）、Linuxホスト、カードとホスト間のイーサネット接続が必要です。



github.com/embox/pjsipを取得して、このディレクトリに移動する必要があります。 以下は、デモの準備に使用したビルドスクリプト、アプリケーション、およびファイルです。



最初の簡単な手順は、検出時にEmboxを実行することです。 ELFを画像形式として使用します。

事前に組み立てられた画像、emboxファイルを取得できます。

または、Emboxを自分で構築することもできます。 これを行うには：

  • github.com/embox/emboxのクローンを作成します

  • 行うことにより、正しい構成をロードします 

    make confload-platform/pjsip/stm32f4discovery


  • やって組み立てる 

     make


  • ターゲットイメージはパスbuild / base / bin / emboxにあります


ファームウェアの後、シリアルポートを介して検出に接続し、ターミナルエミュレータを接続し、115200 8n1の設定を設定する必要があります。 起動後、システムは小さなステータス情報を表示し、起動スクリプトのコマンドを表示して実行し、シェルインターフェイスを提供します。 さらに、すべてのコマンドは検出コンソールで実行する必要があると想定されています。

  • デフォルトのIPアドレスが自分に合わない場合は、 

     route del 192.168.1.0 dev eth0 ifconfig eth0 (ip-) netmask () up" route add () netmask () eth0


  • pjsip_simpleuaを行う


次に、ホストクライアントをビルドして実行する必要があります。 ホストのコンソールに戻ります。

  • www.pjsip.org/release/2.4/pjproject-2.4.tar.bz2をダウンロードして、現在のディレクトリ（pjsip）に配置します。
  • PJSIPホストライブラリをビルドするには、コマンド 

     ./build_pjproject.sh


  • ホストクライアントを直接構築するには、コマンド 

     make


  • 走る 

     simpleua sip:test@(ip- )




これはなぜですか？



通常の組み込みプラットフォームでは、通常とは異なる何かをしたかったのです。 プラットフォームが非常に人気があり安価であることは明らかでした。そのため、8ビットコントローラーではなく、ギガヘルツRaspberryPiやBeagleboneではなくstm32f4-discoveryを選択しました。 彼らは、発見されたものに目を向け始めました：イーサネットに加えて、彼らは音を出力する可能性に注意を引きました。



そして、スターたちが集まりました。「現場で」同僚がLinuxを搭載したOMAP137でsipフォンを開発していました。 そして、特別なソフトウェアを書くために、ある種の苦しみがありました。ビットベイクは、それを何をどのように構築するかを少し考え、何らかの理由でプロジェクトをきれいにするための推奨事項で、最初のアセンブリには数時間かかります。



したがって、私たちは自分たちに課題を用意しました。機能が可能な限り非常に弱いハードウェアの電話にできるだけ似ているSIPデバイスです。



このセクションの追加のポイントはIndemsysの 解説で 、Linuxのみに搭載され、コントローラー（ベアメタルまたはクラシックRTOS）に搭載されていないソフトウェアについて質問しました。



準備する



最初の段階は、SIPプロトコルとそれに伴う音声伝送をサポートするサードパーティソフトウェアの選択です。 Google発行の最初の結果はwww.pjsip.orgでした 。これは、必要なすべてを実行でき、大規模でポータブルなaとファイルでの処理を可能にし、RAMの要件が低い（150kbと主張）。



最初に、アセンブリ記述ファイルを作成する必要があります。 Emboxを初めて聞いた人のために：「モジュール」の概念で動作する独自のビルドシステム（ ハブに関する記事 ）を使用します。 サードパーティソフトウェアのアセンブリを容易にするために、自動ダウンロード、パッチの適用、構成、アセンブリ、およびインストールをサポートする拡張機能を導入しました。



PJSIPライブラリは単なるモジュールです。 このトピックで最初にコミットされたコミット 。 ここでは、アセンブリの説明がホストシステムのアセンブリとどのように異なるか（決して）とPJSIPで必要な変更を評価できます。 SOCK_RDMがないことを報告します。



将来、これらのファイルはもちろん変更されましたが、基本は同じままでした。 さらに、メモリ内の構造のサイズを小さくしたり、コーデックを無効にしたりするために、マクロ宣言が必要でした。 説明の現代的な外観は、ネタバレの下で評価することができます。

音



デモンストレーションのために、最初にメディアストリームのみを送信できるstreamutilと呼ばれる例を使用しました。



ビルドスクリプトを追加し、小さなファイルを振った後、streamutilは、サウンドを出力するためのデバイスが見つからないというメッセージで停止し始めました。 最初から、再生は不要でした。メーカーは、USBフラッシュドライブからwavファイルを再生するためのデモアプリケーションを提供しています。 このコードは、将来の実装の基礎となりました。



一方、問題はインタラクションにどのインターフェイスを使用するかということでした。PJSIP用のサウンドプラグインを開発するか、一般的なサウンドAPIに既存のプラグインのいずれかを使用できます。 その中には、たとえばALSA APIよりも著しく単純なportaudioがありました。 このインターフェイスでドライバーの実装を開始しました。



一言で言えば、インターフェイスは次のように説明されています。コールバックを登録します。コールバックは、必要に応じて、サウンドカードバッファーが空に近いときに呼び出されます。 素晴らしいシンプルなAPIは、コールバックがどのコンテキストで呼び出されるかを規制せず、実際のシステムでは、このために個別のスレッドが起動され、ほとんどの時間スリープし、適切なタイミングでコールバックを実行します。



このようなスキームには「穴あき抽象化」などの問題があります。PJSIPは明示的に別のスレッドを想定しています。 シグナルでコールバックを呼び出そうとすると、デッドロックが発生しました。 スレッドを使用して、「アダルト」システムのようにする必要がありました。 このフェーズの終わりに、ドライバーは次のようになり始めました。



将来的には、可能であれば、 ライトフローを優先して従来のフローの使用を放棄する予定です。 軽量スレッドの主な違いは、独自のスタックがないことです;計算は、最後に実行されたスレッドのスタックで実行されます。 軽量スレッドの場合、利用可能な同期ツールを備えたpthread互換インターフェイスが提供されます。 理論的には、これにより、これらをportaudioストリームの実装として使用できるようになり、メモリ消費が削減されます。



メモリー不足



私たちが苦労した主な問題は、動的メモリの不足でした。



ここで、システムの動的割り当てについて説明する必要があります。 私たちはプールを使用します-固定サイズのオブジェクトの固定数のために静的メモリの予約セクション。 この方法は最も柔軟ではなく、一定量のボイラープレートコードを書くことを余儀なくされますが、代わりに、メモリ消費を制御し、コンパイル段階で潜在的なメモリ不足をキャッチする機会を得ます。 実際、プールのサイズの構成は、見かけよりもはるかに簡単です。



最初に、これらはグローバル構成ファイルで設定されます。ここには、サブシステム、コマンド、管理ポリシーなど、他のすべてのシステムパラメーターがスケジューラーによって記述されます。 このファイルのサイズが100行を超えることはめったにありませんが、この数はシステムの構成に大きく依存します。



第二に、プールサイズは多くの場合自然な要件の結果です。たとえば、stm32f4discoveryでは、オンボードイーサネットとループバックの2つのネットワークアダプターしか使用できません。 驚くべきことに、この確実性は他のサブシステムで見ることができます：開いているファイルの数、スケジューラーの優先順位、irqハンドラーの数など-これはすべて、組み込みシステムでの分析に役立ちます。



PJSIPに戻ると、彼は独自のライブラリを使用して動的メモリ割り当てを行います。これは、標準のmalloc / free呼び出しの上に実装されます。 streamtuil動的メモリの必要性を予測することは非常に困難です。 ただし、束全体がstreamutilによって排他的に使用されるため、システムプールのためにサイズを制御し、拡張できます。



システムで最大のプールの1つは、ネットワークパケットバッファーでした。 その後、小さなハックを行いました。最大パケットサイズを標準の1514バイトから214に変更しましたが、最大パケット数を増やしました。 事実、テストでは次のことが示されました：214バイト以下のパケットは、/からstreamutilに送られるため、このトリックは機能しました。



ヒープを増やすには何度か繰り返す必要がありましたが、その後、streamutilがクリアなサウンドを生成し始めました。



SIPフォン



streamutilは、必要な電話にあまり似ていないデモアプリケーションであることは明らかです。 もっと機能的なものをとる必要がありました。 適切な候補は、PJSIPの参照sip電話であるpjsuaです。 実行を開始した修正済みビルドスクリプト。 メモリ不足。 さて、プールのサイズをさらに引き締めました。 メモリ不足。 その後、漠然とした疑念が私たちを苦しめ始めましたが、原則として、ピジュアの要求を満たすことは可能ですか？



見つける最初の試みは、ホスト上のmalloc / freeへのpjsuaリクエストをトレースすることでした。 しかし、標準ライブラリでは、独自のニーズ（たとえば、日付を文字列にフォーマットするなど）に対して動的な割り当てが多すぎるため、トレース出力が非常に乱雑であることが判明しました。



私たちは別の方法で行きました。標準ライブラリではmalloc / freeを使用しないため、トラックはより有益になります。 メモリ割り当てライブラリにトレースを追加し、beagleboard用にpjsuaでemboxをコンパイルし、qemuで実行しました。 結果は少し混乱しました。断片化とpjsua静的メモリを除いて、ヒープごとに200kbを少し超える程度でした。 stm32f4のこのようなボリュームは適合しません。 2つの解決策がありました：pjsuaを真剣に変更するか、より簡単な代替手段を探します。



幸いなことに、simpleuaデモアプリケーションという代替手段が見つかりました。 Simpleuaダイナミックアロケーショントレースは、約110 kbの最大ライブヒープサイズを示しました。 .data（〜4kb）と.bss（〜25kb）に追加すると、メインスタックとストリーム、ネットワークパケットバッファー、オーディオインターフェイスに十分なスペースがあります。



実装プロセスでは、ヒープ用のいくつかのメモリ領域のサポートを追加し、パケットサイズを増やし（810バイトが必要）、その数を減らす必要がありました。
