コンピュータはプログラムされたものだけを実行できるという一般に受け入れられ広く普及した信念があります。 この誤った判断は、フォームとコンテンツの間に存在する混乱に基づいています。 厳密な文法は、プロセスの厳密な説明を意味するものではありません。 プログラマーはコンピューターの文法の規則に従って非常に正確でなければなりませんが、これらの規則を使用して表現したいコンテンツは自由であり、何にも限定されません。 そして、文法の厳格さはコンピューターによるものではなく、それを使用するプログラマーによるものです。 プログラマーは、自分の考えを非常に正確に定式化する必要さえありません。彼は、ある範囲のコンピューターの解答を頭の中に持っているかもしれません。 プログラマーは、コンピューターを特定のプロセスに制限する必要はありません。 疑いの重みのもとで、彼はコンピューターに新しいプロセスを生成するように依頼するか、特定の選択ルールを提示し、コンピューターに選択を「助言」することができます。 したがって、コンピューターをプログラムする必要がなく、明確に正確に表現する必要があります。


ここで示されている理由は、「デザイン」ではなく、コンピューターが作業にどの程度役立つかということではありません。 多くの理由で、人々は通常これらの機会を過小評価しています。 そもそも、1つの間違いについて警告したいと思います。彼が状況を完全に理解していると信じている多くの人のように、「中程度の」立場を取るべきではありません。 サイエンスフィクションの作家、あらゆる分野の科学者、経済学者、アナリスト、心理学者、さらには論理学者までもが繰り返し、コンピューターが本当に考えられないことを皆に納得させようとします。 「機械が人間のように考えることができると想定すべきではありません。機械はプログラムが提供することしかできないので、独創的で創造的ではありません。」 私たちは皆、似たようなことを聞​​いたことがあり、私たちのほとんどはこの観点に同意します。

（1）従うべき指示のシーケンスとしてのプログラム。


最も一般的で単純な観点：コンピュータープログラムは、特定のデータに基づいて実行される一連の明確な操作です。 簡単なプログラム例を見てみましょう。Xが入力情報として与えられた数値であると仮定します。 



 START: SET R = 1 LOOP: SET S = X/R SET T=R+S SET R = T/2 SET U = R x R SET V = X - U IF |V| > .001, GO TO LOOP ELSE PRINT V


Isaac Newtonに起因するこのプログラムは、Xの平方根のおおよその値を計算します。プログラムの各行では、現在の状況と次に来るものが完全に明確です。 もちろん、そのような状況のシーケンスがRの許容値とすべての計算の終わりにすぐにつながることを証明するために、数学的分析が必要です。 したがって、これらの判断を一連の手順と見なしても害はありません。 「GO TO」コマンドは、「メインループ」のいくつかのコピーの非常に短いバージョンと考えることができます。 もちろん、分析が完了するまで、プログラムは有用であるが理解できないアイデアの表現と見なすことができます。

  1. yは他のオブジェクトの一部として知られていますか？ または、何かの一部であるグループのメンバーですか？

（3）ヒント集としてのプログラム


おびえた人文科学だけでなく、ほとんどのコンピューター「専門家」によって共有されている誤解があります;それは、プログラミングが本質的にフォームとコンテンツの間の基本的な混乱に基づく機能の正確で明確な環境であると言います。 詩人に14行のスタンザを書くタスクが与えられた場合、これはプロセスをより正確にしたものではなかったでしょう。 作曲家がスケールの12ステップすべてを使用する場合、これはフォーム全体を制限しません。 設計者が4次のサーフェスのみを使用する必要がある場合、誰も気付かないでしょう！ したがって、（古い）プログラミング言語の厳密な文法がプロセスの記述の正確さに貢献するような論争でそのような一致を観察することは、単にばかげています。 実際、プログラムがまったく機能するためには、コンピュータの文法（構文）のルールを非常に注意深く守る必要があります。 スペルミスや句読点のエラーは許可されていません！ しかし、これがあなたのプログラムが何をするかを明確に理解するのに役立つと言うのは間違っています。 FORTRAN言語では、プログラムが既に記述されたプロセスを参照するようにしたい場合、「GO TO」固定コマンドのいずれかを使用する必要があります。 「使用」（「使用」）や「GET TO」（「PROCEED ON TO」）などとは言えません。 この場合の構文は非常に厳密です。 しかし、ほとんど何でも「GO」できるので、コンテンツは無料のままです。



そのような厳しさはコンピューターによるものであると仮定するのはさらに間違っています！ 言語を定義したプログラマーによるものです！ BobrovのSTUDENTプログラムでは、必要に応じて「USE（「USE」）を常に入力（「GO TO」）と入力し、単純な状況では「GO」の代わりに「USE」を使用できます。 もちろん、これは柔軟性の単純な例ですが、ほとんどの人が受け入れないのはまさにこの考えです。FORTRAN言語の厳密さと明確さは、もしあれば、厳密な偏見によるものであり、事実の厳密さによるものではありません！



モダンでより柔軟なシステムの例として、Warren Teitelman（ミシガン工科大学の博士論文、1966年）によって開発されたPILOTプログラミング言語を取り上げます。 ） 多くの場合、「プログラム」ではなく「アドバイス」と呼ぶことができます。 これらはケース間で記述され、通常、デフォルトの状況に応じて、または以前のアドバイスの結果として使用されます。 例は、 「宣教師と共食いについて」というよく知られている謎のような問題を解決するためのプログラムを開発するときに導入された次の命題です 。



, m – -1, c – -2 (- m) (- c ), . ( .)







このプログラムはヒューリスティックな検索であり、さまざまなパターンと動きを試し、前進できるものを好み、人々が川を渡るのを助けます。 Teitelmanは最初に基本プログラムを作成します。 しかし、宣教師は食べられており、上記の「アドバイス」は、「前方への動きを測定するプログラムの一部を変更して動きを拒否し、その後、不平等な数の宣教師と人食い人が川のほとりに残る」ことを推奨しています。 タイテルマンによると：

ただし、構文上の制限を減らすことは重要ではありません。 ポイントは、プログラムに加えられた変更の「助言」の性質にあります。 「前進」についての判断は、この「前進」がどのように実行され、正確に「プログラム」のどこに位置するかを知らなくても行うことができます。 状況は別のアドバイスの影響で変化する可能性があり、新しいアドバイスがいつ使用されるのか、彼らが単にそれに注意を払わないのか想像さえできないかもしれません。 アドバイスの影響下で、完全に異なる機能が変化する可能性があり、状況によっては「進行」がまったくなく、いずれにせよ誰かが食べられる可能性があります。 これが発生した場合、その理由を理解する必要があります。
