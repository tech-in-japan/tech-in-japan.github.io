ビデオ検索について一言

今日、この素晴らしい春の日に、ビデオ検索だけでなく、技術的なことについても書きたいと思います。

ロードされたDjangoプロジェクトでのSphinxsearchの動作の実装。













おそらく、ビジネスタスクを設定することから始める価値があります。







  • タイトル、説明、その他のテキストデータで関連動画を検索する必要があります
  • 各動画について、 関連動画を探す必要があります
  • 適切な場所で適切なリクエストを発行する際に、 適切なビデオを表示する必要があります。


非機能要件：







  • dofigビューとビデオの説明の継続的な更新を含むDjangoプロジェクト
  • ライブラリ内の検索エンジンでの作業のカプセル化と、サイト上の他のライブラリとの互換性（まず、Django REST Framework）

Djangoバックエンド検索ベース



Sphinxsearchはmysqlクライアントをサポートしているので、Djangoのバックエンドを使用して必要な検索クエリを作成し、結果をDjangoモデルとして返しませんか？ 誰もがこれを行うことをお勧めしませんが、それは本当にそれほど怖いではありません。 そして、まだ怖い、または単に興味がない人のために、次のセクションにすぐに進むことをお勧めします。







いつものように、githubには便利なものがあります。 たとえば、同じdjango-sphinx-dbは 、djangoモデルから直接エンジンを開始するのに役立ちました。 Django-1.8では、データベースバックエンド実装のプライベート部分が大幅に変更されたため、 django-sphinx-db



移植が問題になりました。 その結果、 django-sphinxsearchプロジェクトが登場しました。これは、開発側からの注意が少し欠けていますが、 すでに本番環境で使用されています。 ちなみに、ここにバックエンドサポートの難しさの例があります。Djangoの新しいバージョンがリリースされ、すべてがバラバラになります。なぜなら、「私の内臓は私が望むものを変える」からです。 ですから、最初から始めなければなりません。

  2. 精神的に最も近いバックエンドが取得され、継承されます。 sphinxsearchの場合、これはMySQL、別名django.db.backends.mysql



    です。
  3. 最も難しいのは、バックエンドが記述されているデータベースと互換性のあるコードSQLCompiler



    生成するようにSQLCompiler

    から文字列を収集するSQLCompiler.as_sql



    メソッドがほぼ100行のモノリスであるという事実について、Django開発者に別の「妖精」を言いたいと思います。 その結果、 LIMIT OFFSET



    をLIMIT start, end



    に変更するには、基本クラスメソッドを呼び出した結果を定期的にLIMIT start, end



    があります。
  4. 検索固有の機能を提供するQuerySetメソッドが追加されました。 たとえば、 SphinxQuerySet.match



    はself.match構造をSphinxQuerySet.match



    追加し、SphinxQL式の構築に必要なデータが含まれます。 一致フィールドは複製され、 SphinxWhereNode.make_atom

検索結果ランキング



通常、検索結果は、検索クエリとの関連性によってソートされます。 それを数えるには？ たとえば、ドキュメントとクエリに同時に存在する単語の数を取得できます。 交差点の単語が多いほど、このクエリにより正確に結果が適合します。 一致する単語の数だけでなく、それらの順序も考慮することができます。 そして、各単語の「希少性」が考慮される場合、それは一般的にクールです。リクエストおよびドキュメント内の前置詞と接続詞の存在は、もはや問題に影響しません。 そのような特性を発明した多くの異なる、有用な、そしてそれほど多くはないので、一般的なケースでは、エンジンが考慮するすべての特性の値の加重和を使用するのが合理的です。







特定のドキュメントを検索クエリに関連付ける特性に加えて、クエリに関係なく、特定の属性を持つドキュメントに重みを追加することができます。 たとえば、高品質でロードされたビデオの出力を増やすため。 または、最近の動画や視聴回数の多い動画に重みを付けます。
