理論



Atheros AR9331には、組み込みコンピューターにとって非常に重要な2つの要素があります：ハードウェアタイマー（システムバスの周波数で4ティック、ゼロに達すると割り込みを生成）とGPIOレッグの割り込み（信号の両側で動作可能）。



残念ながら、ソフトウェアレベルでは、OpenWRTはどちらもサポートしていません。つまり、いくつかの間隔をカウントダウンしたり、いくつかのプロトコルをエミュレートしたい場合は、udelay（）が手元にあり、マルチタスクになることを望みます誰もあなたのプロトコルのエミュレーションの真ん中-首に収まらないでしょう。 GPIO IRQをサポートするために、問題の一部のみを解決する狭い環境で広く知られているパッチがあります。割り込みを取得するだけでは不十分であり、それを処理して、独自のプログラムが既に存在するユーザースペースに配置する必要があります。 もちろん、パッチを適用するには、すべてのファームウェアを自分で組み立てる必要があります。タスクは迅速ではなく、hemoではありません。



タイマーをサポートするものは何もありません。



より正確には、最近までそうではありませんでした。



GPIOからの割り込みをキャッチし、ユーザー空間への信号として送信するカーネルモジュールは次のとおりです。github.com / blackswift / gpio-irq-handler



単純に機能します。擬似ファイル/ sys / kernel / debug / gpio-irqに「+ GPIO PID」という行を書き込む必要があります。GPIOは必要なレッグの番号、PIDはシグナルを送信するプロセスです。 次に、シグナルハンドラー番号42を選択し、着実に待機します。 割り込みを削除-同じファイル内の「-GPIO PID」。



ハードウェアタイマーサポートを実装するカーネルモジュールはこちらです： github.com/blackswift/timer-irq-handler



難しいことではありません：最初に/ sys / kernel / debug / timer-irq write“ + TIMER TICK”、ここでTIMERはタイマーの数です（0から3、システムによって占有されているものはありません。 TICK-マイクロ秒単位の解像度（ティックごとに割り込みが生成されるため、20μs未満に設定しないでください）。 次に、「+ TIMER TIMEOUT PID」があります。ここで、TIMEOUTはシグナルを受信したい期間で、PIDはプロセスです。 シグナル番号は43です。タイマーを1回だけ動作させる場合は、最後に「once」という単語を追加します。 不要な場合は、上記のファイルに「-TIMER」と言うまで機能します。



他のチップには異なるレジスタアドレスがあるため、カーネルモジュールはほとんどの場合AR9331でのみ動作します。 さらに、GPIO IRQを機能させるには、前述のパッチを備えたファームウェアが必要になりますが、現時点では「すぐに」そのようなファームウェアはBlack Swiftでのみ入手できます。



練習する





習得した知識を使用する必要があります。つまり、タイマーと割り込みに役立つ何かを収集するためです。 私にとっては気象観測所になります（KDPVを参照）。 なんで？ 第一に、誰もが理解していること、そして第二に、一般的なDHT22センサーはまだ苦痛です。 1-Wireに類似したプロトコルを備えていますが、ここでは1-Wireに同期がある場合にのみ、ホストがデータレッグをグランドに一度引っ張ると、センサー自体が応答して出力41パルスをパルスします。 「0」と「1」のパルス幅は異なります-28μs対70μs。



デコードするのはかなり不愉快なことです。着信する短いパルスの長さを計算する必要があります。 中断することなく解決されますが、率直に言って、作業の安定性が気に入らないでしょう。 必要に応じて、プロトコルのソフトウェア実装を提供します。 やってみて



割り込みを使用すると、問題は根本的に簡素化されます。多くの作業をせずに、目的のレッグで割り込みをキャッチし、そこから通常のudelay（）で35μsをカウントし、同じレッグのレベルを測定するカーネルモジュールを作成できます。 「1」がある場合、信号は35μsより長く、実際には「1」です。 実際には、それだけです。 カーネルモジュールを私たちより少し難しく書く場合、ソースは次のとおりです： github.com/blackswift/gpio-dht-handler



センサーからデータを取得すると、モジュールはそれをカーネル出力（logreadコマンドを参照）に発行するか、指定されたPIDのアプリケーションに信号44を発行します。 CRCエラーが発生すると、コンソールに「エラー」が、アプリケーションに0が表示されます。







これについては、アプリケーション自体を記述する必要があります。帽子をかぶるかどうかを確認するためにカーネルログにア​​クセスしません。



ここではすべてがささいなものであることがわかります。1分に1回カチカチと音を立てるハードウェアタイマーを開始します（可能な最大カチカチ時間はUINT_MAXマイクロ秒、つまり71分強です）。 シグナルがそこから来て、そのハンドラーがリクエストをDHT22プロトコルモジュールに送信します-単に擬似ファイルに書き込みます。 さらに10ミリ秒後、DHT22からの応答信号が到着します。実際には、ハンドラーがそれを処理します。この例では、ファイルに書き込みます。 



void irq_handler(int n, siginfo_t *info, void *unused) { //every minute counter++; dht_request_data(); } void dht_request_data() { int fd=open("/sys/kernel/debug/irq-dht", O_WRONLY); sprintf(buf, "%d %u", DHT_GPIO, getpid()); // gpio-dht-handler kernel module write(fd, buf, strlen(buf) + 1); close(fd); } void dht_handler(int n, siginfo_t *info, void *unused) { if (info->si_int == 0) // DHT protocol CRC error { if (++failcounter > 10) { printf("Error retrieving data from DHT sensor\n"); failcounter = 0; return; } usleep(3000000); // wait 3 seconds between attempts dht_request_data(); return; } float humidity = (float)((info->si_int) >> 16)/10.0; // 2xMSB float temperature = (float)((info->si_int) & 0xFFFF)/10.0; //2xLSB // ... } int main(int argc, char* argv[]) { int timer = TIMER_TIMER; int tick = 1000*1000; // 1 sec unsigned int timeout = 1000*1000*60; // 60 sec if (!init_handler(timer, tick, timeout)) { printf("Error initializing timer %d\n", timer); return -1; } struct sigaction sig; sig.sa_sigaction = dht_handler; sig.sa_flags = SA_SIGINFO | SA_NODEFER; sigaction(SIG_DHT_IRQ, &sig, NULL); // ... }

github を歩き回ると 、内部で非常に美しく作られたDHT22を操作するための代替モジュールを簡単に見つけることができます。これは、着信パルスの持続時間を文字通り測定します。 マイクロ秒単位。 そして非常に正確に-キャリブレーションされたジェネレーターではテストしませんでしたが、DHT22からのデータは、データシートに示されているデータと3〜4マイクロ秒しか異なりません。



1つの例外を除きます。



OpenWRTの奥深くには、操作中に他のすべての割り込みを偽装するやや邪悪なUSB割り込みハンドラーがあります。長時間動作するため、30〜40マイクロ秒かかります。 その結果、この場合、GPIOからの中断の一部をスキップするだけで、せいぜい長い遅延でそれを受信できます。 残念ながら、1999年のアセンブラーと著作権の世界では、まだ完全には解明されていません。そのため、パルスごとに2つの割り込みを処理するモジュールを脇に置く必要があります。そのような条件下で最初の割り込みをキャッチし、それから35μsを測定する方がはるかに信頼性があります。



ここで、彼は2つの割り込みを食べ、3番目の割り込みをプッシュしました。







さらに、USBがアクティブに動作している場合（たとえば、Webカメラからストリーミングされる場合）、このような遅延が頻繁に発生するため、1つの割り込みのみを必要とするモジュールが危険にさらされます。 ただし、数十マイクロ秒の遅延がそれほど重要でないわずかに重要度の低いものでは、これに気付かないでしょう。しかし、それはすべて同じように不快です。
