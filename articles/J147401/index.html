PVS-Studioについてのいくつかの言葉、それが何かわからない場合



PVS-Studioは、C / C ++ / C ++ 11プログラミング言語を使用するプログラマ向けのツールです。 このツールは、プログラムのソースコードのエラーやその他の欠点を識別する静的コードアナライザーです。 このアナライザーは、Windowsアプリケーション開発者を対象としています。 現在、PVS-StudioはVisual Studio 2005/2008/2010に統合されており、まもなくVS 2012をサポートします。MinGWを使用してビルドされたプロジェクトを検証することができます。 リンクでアナライザーの詳細を確認できます： http : //www.viva64.com/en/pvs-studio/

最後に、PVS-StudioとPC-LintおよびKlocworkの簡単な比較。



PVS-StudioはPC-Lintよりもはるかに使いやすいです。 PC-Lintの主な問題は、潜在的な問題が多すぎることです。デフォルト設定の同じプロジェクトでは、23,000以上のメッセージが表示されます。 （PVS-Studio-42のみ）。 同時に、PC-Lintは繰り返し発生する問題を隠すことはできません（同じ.hファイルが2つの.cファイルから接続されている場合、PC-Lintは.hファイルに問題を2回表示し、PVS-Studioは1回だけ表示します）。 また、PC-Lintでは、1つの問題を誤検知としてマークすることはできません。問題のクラス全体を「オフ」にすることしかできません。



さらに、PC-Lintによって発行された「ガベージ」の山を掘り下げるのが良い場合、PVS-Studioが見つけるよりも実際の潜在的な問題を見つけることができます（たとえば、いくつかのクラスで仮想デストラクタがなく、明確に定義されていないことが見つかりませんでした）コンストラクタとデフォルトのコンストラクタをコピーします）。 もう1つのことは、PC-Lintでは、これらの有用なグレインは通常、残りの中で「 "れ」、それほど重大なメッセージではないということです。



同時に、PC-Lintはより使いやすく、Visual Studioとの統合も非常に簡単ですが、使用するのははるかに困難です。



しかし、一般的に、ここでPVS-Studioは議論の余地のないリーダーです。 ただそれを受け取って、それを使って作業を開始できるからです。PC-Lintについては言えませんが、発行されたすべてのメッセージを表示するには数日かかることがあります。



しかし、Klocwork PVS-Studioと比較すると、残念ながら負けています。 まず第一に、使いやすさ（ビルドごとのステータスモニタリング-修正された問題の数、残っている数、追加された数、および便利なレポートのみ）、汎用性、および重大な潜在的な問題のより良い識別に注目します。 さらに、Klocworkはサーバーベースのマルチユーザーシステムであり、検出された各問題に責任を割り当て、複数のプロジェクトを一度に処理して、それぞれのステータスを簡単に追跡できます。



価格で比較すると、そのようなレイアウトが得られます（開発者のWebサイトからの情報）。

  • PVS-Studioの場合、1年間の使用に対して5ライセンスで3500ユーロ、30ライセンスで9000ユーロを要求します。
  • PC-Lintの場合-1ライセンスで389ドル、3,500ドル-10で無制限。
  • Klocworkの場合、「サーバー+ 20クライアント」パッケージで1年間の使用で30,000ユーロ。


まあ、価格は機能にほぼ対応しており、PVS-Studioは安価であるが重くて不便なPC-Lintと便利で美しいが高価なKlocworkの中間に位置しています。

典型的なPVS-Studioスキャンレポートを見てみましょう。 これはちょうど私の腕に落ちたので、私はそれにリンクを与えます。



タイプミス...うん、なるほど。 配列を超える能力...なるほど。 いくつかのマイナーな欠陥...なるほど。 これらの欠点がプログラムの動作にどのように影響するかはわかりませんが、それらは非常に明確に提示されており、それらに注意を払う必要があると考える場合、その重要性を簡単に評価できます。



コードにこれらのひどいエラーがあるため、プログラムがどのように機能するのか疑問に思う人がいるかもしれません。



すべてが非常に簡単です。 プログラムのテキストで明らかに明らかになっている問題は、古き良きデバッグ、単体テスト、専門家の評価など、他の方法を使用して以前に発見されました。 しかし、他のすべての欠陥を見つけるには、いくつかの努力をする必要があります。 これは、たとえば、異常なデータセットである可能性があります。 または、非定型のユーザーアクションアルゴリズム。 または、異常なエラー表示。 または、コンパイラまたはC ++ライブラリのバージョンの更新かもしれません。



エラーはエラーです。 彼らが言うように、あなたはコンパイラに嘘をつくことはできませんが、すべてのエラーが等しいわけではありません。 これらの非常に小さな欠陥は長年データベースに隠され、その後誰かがそれをPVS-Studioで実行します。そして、「なんてこった、どうして恐ろしいバグ、神MY1111 !!!」-彼は叫ぶ。



そのため、スキャンレポートはそれ自体ではあまり機能しません。とにかく、コードベースに精通している開発者は、示された各問題を評価し、可能であれば解決する必要があります。 PVS-Studioのスキャンレポートは、必要な処理を正確に実行します-検出されたエラーを次々に表示し、簡単な説明を提供します-これ以上は何もしません。

Integrity Reportは、「ミッション」、「シームレス」、「イノベーションに焦点を当てる」などの言葉を含む熱狂的なドキュメントです。 種としては悪くない-少なくともそのようなキーワードの存在は、最初の3ページに十分なマーケティングがあることを明確に示している。



表Aに進みます...ああ、このプレートはプロジェクトのサイズ分布を示しています。 どういうわけか、「分布」という言葉の使用は、収集されたデータが統計的に有意であり、したがって特別な信頼に値することを意味します。 45のプロジェクトがあり、それらに基づいてスケジュールを作成しようとしている場合、「配布」と呼ぶのは非常に愚かです。 ご覧のとおり、700万行を超えるコードを含む2つのプロジェクトがあります。 信じられないほど、言葉はありません。

付録Aに早送りします。表2と表3はエラーをまとめていますが、同時にエラーを分類し、プログラムへの影響の程度を示しています。 見てみましょう...



制御フローエラー。 これは何ですか これは、「switch」というステートメントの「case」の最後に「break;」と書くのを忘れたときです。 あなたは中程度の影響があると言います...わかりました。 すぐに返される「main（）」はどうですか？ これも制御フローエラーであり、平均的な影響があることを教えてくれません。 すべての制御フローエラーが公平であるとは限りません。



NULLポインターの逆参照も平均的な効果がありますよね？ そう、私のコードはあちこちでヌルポインターを逆参照し、ユーザーがキャンディを取得するたびに参照します。 おそらく、レポートの作成者は、コードが最初にNULLでないことを確認せずにポインターを逆参照する場合、NULLポインターの逆参照の可能性を念頭に置いていたのでしょう。 良いニュースは、逆参照するたびにNULL値へのポインターをチェックすると、コードが非常に乱雑になることです。 繰り返しますが、すべてのNULLポインター逆参照エラーが等しいわけではありません。



エラー処理の欠陥。 これは何ですか Win32 API関数のエラーコードをチェックすることを意味しますか？ もちろん、この試みの成功を評価せずにプログラムがファイルを開こうとするたびに、さらに読み続けると、ユーザーは言うでしょう。 フォルダーにアクセスできませんか？ さて、ファイルが保存されたふりをしましょう。 エラー処理のすべての欠陥が同じというわけではありません。



整数型の問題は中程度の影響があります。 正確には、割り当てられたメモリのサイズを計算する際の整数型のオーバーフローは意味がありません。 どれだけ結果が出るかを強調し、これが適切な量であると仮定します。 整数型を扱う際の問題がすべて同じというわけではありません。



安全でないデータを操作すると、平均的な影響があります。 何言ってるの？ 手がかりはありませんが、安全でないデータを操作するすべての状況が同じというわけではないことがわかります。



誤った表現-中程度の露出。 もちろん、好きなように括弧を付けてください-何が悪いのでしょう。



同時アクセスエラー-中程度の影響。 あなたの人生の残りの間、彼らのデバッグを掘り下げて考えてください、それは大丈夫です。
