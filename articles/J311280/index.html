True Image 2017のリリースと開発について-すべてのハードコア機能が用意されています

私たちは最近休日を過ごしました-True Image 2017をリリースし、1年間取り組んでいます。 多くの変更がありますが、最初に目を引くのは、「カジュアルな」ミニマルなデザインです。 最初のリリースが上級ユーザーとシステム管理者向けのツールであった場合、過去数年間、この製品の世界的な人気は非常に多くの人々がバックアップするほどのものです。 モニターとシステムユニットを特に区別しない人を含めます。



したがって、私たちのアプローチは、古き良きハードコア、設定の柔軟性、および多くの重要なツールをすべて保持することですが、インターフェイスの簡素化に焦点を合わせることです。

このリリースでも、iOSとAndroidのデスクトップへのローカルバックアップの方法、Facebookプロファイルのバックアップ（ユーザーMasha Bucketのおかげ）、アーカイブアーキテクチャの操作などを学びました。 主な機能とその開発の難点について説明します。

従来のデスクトップバックアップ



歴史的に始めた最初のモードであるセクターごとのバックアップは、2017リリースでも自然に保存されていました。 仕組みを思い出させてください。ハードドライブの完全な物理コピーを作成し、必要に応じてさらに操作することができます。 これは、データを復元するときに非常に便利です（呼吸困難なディスク上のデータを破損しないようにするため）。 このようなディスク（より正確には、イメージファイル）を直接Acronis True Imageにマウントし、Windowsエクスプローラーで個別のR / Oディスクとして表示することもできます。



反復スキームに従って、従来のバックアップは少し異なります。 最初に、完全なイメージ（ハードディスクのすべてのファイル、例外の対象）が取得され、次に差分がバックアップに追加されます。 スケジュールとすべての詳細は非常に柔軟であり、「カジュアル」ユーザーには最適なデフォルトがあります。



もちろん、緊急ディスクまたは緊急フラッシュドライブを作成して、そこから起動することを引き続き提案します。 USBフラッシュドライブに書き込み、実際にはLinuxのリカバリツールを使用します（Macの場合、「ネイティブな」標準のディザスタリカバリツールとユーティリティはそこに書き込まれます）。 実際には、これをNext-Next-Finishツールとして特別な知識なしで使用することも、画面を切り替えて別のことを行うこともできます。

当然、下位レベルを非常にうまく分解しているため、ローカルコンピューターで緊急ツールを直接作成できます。 EFIで新しいブートブランチを作成しています。ここでは、修正されたコンパクトLinuxおよび回復ツールを記述しています。 何かがカバーされている場合は、別のブランチの同じコンピューターから起動して、バックアップから回復することができます。 これにより、スケジュールされたバックアップのためにハードドライブをコンピューターに持ちたくない（およびクラウドにバックアップしない）怠usersからユーザーを保護します。 それにもかかわらず、もちろん、これは鉄の複雑な故障から救うことはできません。したがって、データを含む別の媒体が依然として重要です。



もう1つの興味深い点は、自動マッピングです。 すでに持っていましたが、このリリースではよりスマートになりました。 事実、ほとんどすべてのメディアから回復が行われていることであり、そのようなメディアがいくつかある場合もあります。 たとえば、ネットワークリソース、ローカルコピーなどです。 最後のバックアップの反復以降にシステムが大幅に変更された場合（たとえば、1年前にバックアップをロールアップした場合、たとえば、一部の企業で新しい職場をセットアップしたときにこれが発生します）、どこでどのように復元するかを正しく理解する必要があります。 典型的な例は、別のハードディスクレイアウトです。 別のケース：バックアップは古いが、このコンピューターからの完全な回復を待ちたくない場合、明らかに破損していないものを除外しようとすることができます。 数時間待たないために。 自動マッピングコードを使用すると、このような場合にすぐに違いを取得して正しく回復できます。

フォルダー同期



モバイルバックアップ



新しいリリースでは、モバイルバックアップのすべてのテクノロジーをほとんど検討しました。 要点-誰もが自分のデータのコピーを他の誰かのクラウドに提供したいとは限らないことを十分に認識しているため、ローカルバックアップ機能を提供しました。 古いクラウドオプションも使用できますが、ローカルバックアップは非常に人気がありました。 私たちは企業の開発にサーバーコード（ネットワーク内の企業デバイスの妄想とバックアップ用に作成されたもの）があり、それを再利用することにしました。 より明確にするために、最初に新しいスキームがどのように機能するかを説明します。



  1. 電話で帰宅します（より正確には、Acronis True Imageの基本バージョンがインストールされているデスクトップと同じWi-Fiネットワーク上にあります）。



  2. 電話のアプリケーションがアクティブになり（iOSのスケジュールに従って、またはAndroidのバックグラウンドから）、現在のネットワーク上のコンピューターに接続する機能を決定します。 これを行うために、電話機はSSL接続を確立するためのメインホストの検索を開始します。



  3. その後、反復バックアップが開始されます。過去1日間に変更されたものはすべて部分に分割され、短いチャンクで満たされます。 ユーザーはホームネットワークで多くの時間を費やすことが想定されているため、両方のオペレーティングシステムのバックグラウンドツールが使用されます。たとえば、iOSの場合、「ウィンドウ」を30秒間開き、短いチャンクを埋めて、次の時間ウィンドウを待つことができます。


困難はほとんどどこにでもありました。 まず、もちろん、iOSおよびAndroidの新しいバージョンでのトラフィック、バックグラウンドでのアクティベーション、および電力消費。 一般に、これはすでに比較的単純であり、既製のレシピがありました。 最初の完全バックアップの後、原則として、数メガバイトの新しい写真、数キロバイトの連絡先、その他のユーザー情報について話します。 充填を担当するコードを書き直しました。チャンクは破損せず、ネットワークで24時間休憩した後でも適切にダウンロードされます。 連絡先が優先的に優先されるようになりました。 同時に、クラウドの塗りつぶしを更新し（バックアップがローカルでない場合）、チャンクの終了を待たずに、受信した直後に連絡先が表示されるようになりました。



電話とデスクトップの間に接続を確立することは簡単ではありませんでした。 QRコードに落ち着くまで多くの方法を試しました-アプリケーションのデスクトップ画面からQRコードを撮影する必要があります。 しかし、このバージョンでも非常に困難でした。最初は、本格的なSSL証明書をすぐにビジュアルコードに縫い付けました。すべてのライブラリリーダーがそれを読むわけではありません。 QRを大幅に削減し、新しいキー交換プロトコルをインストールする必要がありました。



検出自体はさまざまな方法ですぐに実行されるため、仮想マシン内でオペレーティングシステムを起動した場合でも（必要に応じて）、電話はそのペアを検出します。 一般的なホームケースは、電話がWi-Fi経由で接続され、ラップトップがローカルケーブルで接続されている場合です。 解決は、IP4、bonjourサービス、およびホスト名によって行われます。 残念ながら、ロシアのYotaユーザーはまだこの機能がありません。Wi-Fiを配布する「プロバイダー」モデムはNATを作成しますが、これは見つけるのが困難です。



サーバーコードの再利用もまた別の話です。 当初は、Linuxサーバーであり、理論的には、たとえばWinに移植できました。 同時に、コンポーネントのロジック全体が、コマンドを発行する制御サーバーを持つように構築されます。 その結果、企業のバックアップを使用して、このサブネットのコマンドセンターとして機能するマイクロサーバーを作成します。 ホームの場合、そのような「ナノサーバー」はデスクトップでエミュレートされていると言えます。 より正確には、コンポーネントのみが要求を送信し、ローカルバックアップの場合はデスクトップ自体が回答を提供します（クラウドの場合は回答がリモートサーバーによって提供されます）。 このコンポーネント自体は、特にコンソールがすでにバックアップされているものとされていないものを知るために必要でした。



リカバリもモバイルアプリケーションを経由しますが、GUIを介して既にアクティブに起動しています。 もちろん、ほとんどのデータは異なるデバイス間で移行できます。たとえば、iOSからAndroidに移動する場合、写真をドラッグするのが最も簡単です。



Facebookプロファイルのバックアップ



ページをハイジャックしただけである場合-これは一般に、それほど大きな問題ではなく、ほとんどの場合、通常の手段を使用して復元できます。 Facebookが予期せずあなたを禁止した場合、あなたは何かを引き出す可能性は低いでしょう。 ページの技術的な不具合のためにページが死んだ場合も同様です。 ユーザー自身が何か（たとえば、1枚の写真）を誤って削除して、実際に復元したい場合があります。 一般的に、多くの人がFacebookのプロファイルをバックアップすることが可能かどうかを尋ねました。 はい、できます。 今やる。



デスクトップからプロファイルコピーツールを起動すると、Fbページが開きます。ここで、データへの特別なアプリケーションアクセスを許可する必要があります。 これは、Facebook APIのゲートウェイであり、実際、バックアップ作業用のトークンリポジトリです。 次に、Facebook側のAPIを使用してプロファイルの「可視性」を制限し、コピーを開始できます。



すべてがコピーに適しているわけではありません。 たとえば、友人のグラフを取得して保存することはできません。Facebookは（他の多くの個人データとは異なり）その主要な財産を保護します。 しかし、すべての写真、壁全体、すべてのメッセージなどをピックアップすることが判明しました。 写真とビデオ。この場合、すぐに回復できます。APIを使用すると、更新ストリームに表示せずに遡ってアップロードすることができますが、もちろん、いいねやコメントはありません。 これらのデータはいかなる方法でも戻されませんが、一般的には非常に説明可能です。 タイムラインも復元されません。写真とは異なり、すべての記録はバックフィルの時点までに日付が付けられます。 あなたのいいねはバックアップに保存されますが、復元もされません。



Facebook APIは非常にバグが多いことがわかりました。むしろ、非常に非論理的な機能が豊富です。 このツールはもともと大規模なタスクを対象としていませんでした。原則として、アプリケーションは非常に具体的な何かに対する1つまたは2つの要求をプルしますが、一度にすべてではありません。 これにより、同じ増分バックアップを実行するのは非常に困難であるという事実が生じました。 たとえば、既知の日付の現在の状態と以前の状態の違いをすばやく取得することはほとんど不可能です。 通常の方法はなく、今後もそうではありません-これはFacebookのセキュリティポリシーの一部のようです。 既存の制限におけるアルゴリズムの構成について慎重に考える必要がありました。 その結果、私たちは一生懸命掘りましたが、バランスを見つけました。 2番目のポイント-トークンには制限があります。 最長のトークンは永遠ではなく、60日間有効です。 また、Facebookのロジックによると、60日後にダッシュボードに戻ってボタンを押す必要があります。 通常のWebトークンは、ユーザーアクティビティ中に更新されます。プロファイルにアクセスして、そのようなトークンに言及するCookieがある場合、トークンは更新されます。 ライブコールなしで、別のセッションで作業します。 幸いなことに、私たちはSmartTVのメーカーではありません。 それらから、ユーザーは常に電話またはデスクトップでコードを取得し、2か月ごとにリモコンからテレビに打ち込みます。 幸いなことに、ボタンは1つだけです。



とても楽しいテストでした。 Facebookでは、メインネットワークには表示されないテスト仮想ユーザーを作成できます。 残念ながら、これらの「マンネキン」はお互いを好きにならず、コメントすることもできず、他の多くの制限があります。 その結果、「ライブ」ユーザーを作成しました。その主なユーザーは、時間の経過とともにMasha Bucketになりました（まだネットワーク上で見つけることができますが、プロファイルを削除しました）。 最終的には、トップマネージャーのプロファイルを使用しました。長年にわたってアクティブなネットワークライフを過ごし、今日をリードしている人々の非常に「肥大化した」プロファイルが必要でした。 その結果、従業員の1人がバックアップボリュームの測定を開始しました。「50人のIvanovsのデータがFacebookから取得され、48人だけが来ました」などのチケットの説明があります。



変更ログの興味深い点



これはリリースに関する投稿であり、開発機能ではないため、実装に関する多くの興味深い情報を残しておきます。 しかし、同僚と私は、筋金入りの詳細を後で説明しようとします。 それまでの間、私たちは他の興味深いものを変更したことに注意します。



最初の興味深い話は例外です。 たとえば、通常のバックアップスクリプトはシステムのシャドウコピーを取得しません-意味がありません。 バックアップ（ネットワーク経由でアップロードするユーザーにとって特に重要）を促進するために、例外を除いて非常に良い仕事をしました。 一時フォルダー（ブラウザーキャッシュを除く-予想外に重要であることが判明）、システムログ、クラッシュダンプ、その他のシステムに加えて、独自の回復ツールがあるという事実を排除します。 たとえば、明示的に反対を指定しない限り、Dropbox、Yandex.Disk、Google Drive、OneDrive、iTunesなどのライブラリ、およびウイルス対策検疫はバックアップされません。 したがって、設定を見つけ、構成レコードによってシステム内にあるものを判別し、何をすべきか、何をすべきでないかを理解するために、これらのツールをすべて逆にする必要がありました。 Yandex.Diskは最長期間私たちから逃げ出しました。バージョンが異なると、設定がまったく異なる方法で保存されます。 前回のリリースで最初に遭遇したのは、XMLファイルの構成で、彼はSQLiteデータベースに設定を保存し始め、ストレージ形式を変更し始めました...これらの設定を行い、各例外のロジックを理解しようとしました。



逆の方向では、例外も必要でした。 たとえば、新しいシステムで古いアーカイブを作成するために、古いOS、スワップファイルなどからドライバーを復元することは決してできません。 したがって、リリースからリリースまで、このようなルールセットをサポートしています。







また、同じドライブに保存されている場合、同じバックアップのバックアップファイルから除外します。 そうしないと、再帰が発生します。 同時に、不要なファイルのアーカイブ（「Parse1」、「Parse2」フォルダーの圧縮バージョンなど、ユーザー自身がノスタルジックな感覚で表示して保存する）の新しいタイプのストレージがあります-それらをバックアップに含めます。



バックアップデータが保存されるアーカイブは大きく変更されています。 これと昨年、私たちはそれを最適化する方法と、よりアクセスしやすくする方法について多くのアイデアを思いつきました（最終的に、アーカイブファイルからディスクを「透過的に」マウントし、クラウドコピーへのWebアクセスを提供します）。 現在、ローカルバックアップは2番目のバージョンのアーカイブに保存され、アーキテクチャは大幅に仕上げられています。モバイルバックアップは、完全に異なるアーキテクチャを持つ3番目のバージョンに既にあります。 来年はどこでも3番目のバージョンに切り替えると思います。 概して、私たちのアーカイブは「それ自体」の独立したファイルシステムです。 そのため、たとえば、1つのファイルの最大20のバージョン（重複排除の有無にかかわらず）をその中に置くことができます;古いバージョンを削除する独自の手段があります。 ガベージコレクタに類似したものがあります。アーカイブを「クリーン」でき、その中にストレージスペースが作成されます（ファイルサイズ自体は変更されません）。 従来のファイルシステムと同様に、アーカイブ内の削除されたファイルは最適化のために削除されませんが、上書きの対象としてマークされます。 アーカイブの辞書構造により、完全に削除するには辞書セクションの削除が必要です。 また、アーカイブの内部デフラグの独自の手段がありますが、直接のユーザーコマンドなしでは使用しません（削除時にデータを移動しないように）。 事実、情報の移動は損失の脅威を生み出します。 たとえば、一般的な問題は、ユーザーが複数のアダプターを介してデバイスを接続するときの「USBガーランド」です（特に、小さなラップトップのユーザーによく見られます）。 アーカイブサイズを変更してアーカイブ内のファイルを削除しようとすると、すべてのデータがセクターごとに移動します。 長い「ガーランド」では、これにより損失が発生し、そのようなケースが数十件ありました。



また、個々のファイルの回復にも取り組みました。 特に、ローカルアーカイブで非常に迅速に検索されるようになりました。
