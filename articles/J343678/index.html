サーバーレスコンピューティングとは何ですか？



APIの進化に従わなかった場合、Serverlessの表示をスキップした可能性があります。 これは、作成したコードがクラウドで実行されるアプローチです。 直接。 OS、仮想マシン、またはコンテナを持ち上げる必要はありません。 コードを記述して実行します。 魔法。



「Something-there-aaS」を見ると、この画像をグーグルで検索できます。











  • 独自のサーバーを使用する場合は、すべての責任を負います
  • IaaSの場合、オペレーティングシステムの前のすべてが「他の誰か」によって提供されます。 アマゾン、グーグル、マイクロソフト-さて、あなたは理解しています
  • 次にPaaS 。 実行時まですべてがあなたのために行います。 心配するのはデータとアプリケーションだけです。 そして、APIを使用してランタイムに接続します（少なくともほとんどの場合）
  • SaaSとは、既製のアプリケーションまたはサービスを提供することです。 プログラマーとして、心配することはありません


Serverlessはこの状況にどのように適合しますか？









サーバーレスの場合、アプリケーションも開発していますが、アプリケーションとそのデータは管理およびサポートされていません。 そのような決定から何を得ますか？



  • スケーラビリティ -もう面倒を見る必要はありません。 他の誰かがあなたのために今やっています。 コードで何をする必要があるかを説明しましたが、今では適切な数のサーバーでコードを実行することがプラットフォームの関心事です。
  • サポート -記述する必要があるコードが少ないため、サポートする必要があるコードが少なくなります。 実際、「ステージング」コードと「プロダクション」を分離するものはすべて捨てます。 プロトタイプは既に実稼働で実行できます。
  • セキュリティ -PaaSベンダーがアプリケーションのセキュリティに精通していると想定すれば、頭痛の種は1つ少なくなります。
  • 市場投入までの時間 -書くコードが少ないということは、ユーザーにソリューションをより早く見せることができることも意味します。
  • 遅延-APIが提供するのと同じクラウドでコードが実行されるため、チームとプラットフォームの応答の間の遅延は最小限に抑えられます。 一部の人にとってこれは重要ですが、一部の人にとってはそうではありません。


結果として何が得られますか？ スケーリングで保存します。 PaaSソリューションを提供するベンダーは、既に（および他の多くのクライアントに対して）スケーラビリティ、サービスサポート、およびセキュリティを提供しています。 理論的には、彼は自分よりもさらに上手くできる。 これにより、リソースを解放してユーザーに最適なUXソリューションを実装し、アプリケーションをより良く、より迅速に市場に投入できます。 追加のボーナス：コードは、サービスが使用するプラットフォームのAPIに可能な限り近い場所で実行されます（サーバーレスは、通常、何らかのサービスとAPIを提供するプラットフォームの追加機能として使用されます。たとえば、音声、ビデオ、メッセージを操作するAPI- 翻訳者のコメント ） 。



サーバーレス=機能



「サーバーレス」という名前の人気にもかかわらず、 Google Cloud Functions 、 PubNub Functions 、 Twilio Functionなどの製品に反映されているFaaS、「Functions as a Service」を見つけることができます。



最も一般的な例は、おそらくAWS Lambdaです。 また、オープンソースのApache OpenWhiskソリューションもあります。



多くのAPIサービスプロバイダーがサーバーレス機能を提供し始めています。 サービスと通信する独自​​のサーバーはもう必要ありません。「XXX Functions」製品でコードを実行するだけです。



場合によっては、「機能」サービスを無料で利用できますが、ほとんどの場合、ベンダーは「使用量ごとに支払う」モデルに従ってそれらの料金を支払いたいと考えています。

CPaaS（サービスとしての通信プラットフォーム- 約Translator ）に戻って、Serverlessの機能を見てみましょう。



現在、サーバーレスソリューションを提供しているCPaaSベンダーは2つしかないと思います。



  1. VoxEngineを使用したVoximplant
  2. Twilio 機能を備えたTwilio


ロンドンで行われた前回のSignalカンファレンスで、Jeff LawsonはFunctionsがTwilioのサービス開始以来最も急成長している製品であると述べました。 この機能は市場に必要です。



CPaaSは現在非常に複雑であり、CPaaSでサーバーレスがどのように使用されているかを理解することがより重要です。 CPaaSをいくつかのAPIレベルといくつかの製品に分けましょう。



APIレベル



  • TwiMLやNCCOなどのスクリプト言語
  • REST API
  • クライアント側のSDK（ブラウザー、モバイルアプリケーション、冷蔵庫などからの呼び出しの送受信用- 翻訳者コメント ）


製品



  • SMSと音声（電話番号を使用）
  • IPメッセージング、チャット、オムニチャネルメッセージング
  • VoIP（WebRTCを使用した音声とビデオ）


ある程度まで、スクリプト言語の独自のAPIレベルは、サーバーレスの非常に粗雑な形式と見なすことができます。 目的の動作をイベント応答スクリプトとして記述し、WebHooksを使用してプラットフォームに送信します。



REST APIはサーバーレスアーキテクチャで適切に機能します。サーバー間で認証、セキュリティ、またはスケーリングを要求する代わりに、実行されるのと同じサーバーで実行できます。



また、クライアントSDKがあります。 これらはエンドデバイスで実行されるため、サーバーレスの概念がどのように適用されるか想像するのは困難です。 SDKはCPaaSバックエンドと相互作用するように設計されているため、考慮しません。

IaaSベンダーのFaaSサービスを使用できますか？



私は本当にこれら2つの頭字語を1つの文に入れたかったです:)



すべての主要なIaaSベンダー（Azure、AWS、Google Cloud）は現在、サーバーレスを何らかの形で提供しています。 なぜCPaaSでサーバーレスなのですか？ IaaSサーバーレスをCPaaSに接続することはできませんか？



できます。 ただし、2つの異なるベンダーになります。 既に他のAWSサービスを使用している場合、AWS Lambdaのようなものを使用することは理にかなっています。



通信の問題を解決する場合は、サーバーレスCPaaSを使用する方が合理的です。 それと一緒に、外部サーバーレスソリューションと比較して、遅延が削減され、セキュリティが向上します。
