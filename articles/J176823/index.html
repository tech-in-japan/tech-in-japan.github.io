Linux KVMでWindowsを実行する

タスク：典型的なLinuxサーバー上のWindowsで多数の仮想マシンを実行します。



ソリューション：最新のLinuxディストリビューション、「ネイティブ」KVM仮想化テクノロジー、Windows 2003、および以下で説明する設定。



FreeBSD用のKVMコンテナの構成に関する入門記事も参照してください。



ゲストOSの選択



Windows XPはLinux KVMでは不安定です。 主なエラーは、csrs.exeプロセスによるプロセッサの100％の消費（RDPセッションの終了まで）と、HAL.DLLのIRQL_NOT_LESS_OR_EQUALコードによるBSODです。 安定した動作を達成できた場合は、更新プログラムの自動インストールを無効にしてください！ 私たちの経験では、WinXPがKVMの下で動作するために、それらは問題の主要な原因になりました。



Windows 7は正常に動作しますが、Proxmoxカウンターによると、動作には3ギガバイト以上のRAMが必要です。



最良のオプションは、Windows 2003 R2の32ビット版です。

  • 確実に動作します。 virtioを使用すると、
  • XPとの互換性の問題はありません（外観を同じにすることもできます）
  • 800メガバイト未満のRAMを使用します。


4ギガバイトのメモリ（PAEなし）の上限は、発生したすべての問題に対して十分でした。



情報提供の目的で、RuTrackerで公開されているすべてのディストリビューションが適しています。



最初の起動とvirtio 



#!/bin/sh VM_ID="10" MACBASE="00:16:3e:ff:ff" HDA="vm_${VM_ID}.img" HDB="temp.img" HDC="w2k3_r2_ent_rus_x86/ru_win_srv_2003_r2_enterprise_with_sp2_vl_cd1_X13-46484.iso" HDD="virtio-win-0.1-52.iso" sudo kvm \ -enable-kvm \ -boot "menu=on,order=d" \ -m 1024M \ -balloon virtio \ -name "kvm_${VM_ID}" \ -drive "file=$HDA,index=0,media=disk,cache=writeback" \ -drive "file=$HDB,index=1,media=disk,cache=writeback,if=virtio" \ -drive "file=$HDC,index=2,media=cdrom,cache=writeback,readonly" \ -drive "file=$HDD,index=3,media=cdrom,cache=writeback,readonly" \ -net "nic,model=virtio,macaddr=${MACBASE}:${VM_ID}" \ -net "tap,ifname=tap${VM_ID},script=no,downscript=no" \ -vnc "0.0.0.0:${VM_ID}"

推奨Windows設定



最初に、デフォルトでは、WindowsはBSODで完全なメモリダンプを作成します。 最良の場合、これにより再起動が大幅に遅くなります。 最悪の場合、完全にハングアップします。



第二に、自動更新はデフォルトで有効になっており、自動更新の1つがKVMの下で不安定になる可能性があります。



したがって、最初の場所でインストールが完了した後（ドライバーをインストールする前に！）コントロールパネル=>システムに移動することをお勧めします。

  • 自動更新：無効にする
  • 詳細設定=>エラーレポート=>無効化
  • 詳細=>ダウンロードと復元=>設定=>システムクラッシュ=>デバッグ情報の書き込み=>小さなメモリダンプ（64KB）


TCP / IP設定はオプションですが、パフォーマンスがわずかに向上します。 仮想環境では、物理ネットワークを介して送信するときに考慮する必要のある問題はありません。



説明： www.linux-kvm.org/page/WindowsGuestDrivers/kvmnet/registry



準備ができているREGファイル： svn1.sytes.net/linuxkvm/tune-guest-tcpip.reg



その後、ディスク（virt-stor）およびネットワークカード（virt-net）のドライバーのインストールに進むことができます。 それらをインストールすると、「Red Hat VirtIO SCSI Controller」、「Red Hat VirtIO SCSI Disk Device」、および「Red Hat VirtIO Ethernet Adapter」がハードウェアマネージャーに表示されます。



バルーニング



従来のアプローチでは、仮想マシン（VM）の開始時に、特定のサイズ（たとえば、512メガバイト）のRAMブロックを直ちに割り当てます。 その欠点は、VMメモリに未使用のスペースがある瞬間にあり、他のVMやホストシステムでは見逃される可能性があります。



メモリボールリングは、（a）必要に応じてVMにホストRAMを割り当て、（b）使用可能になった未使用のブロックを返すためのメカニズムです。 そのおかげで、最大許容ボリュームを一度に使用しない限り、ホストシステムの物理RAMの量を超える仮想RAMの合計量である多くのVMを同時に起動することが可能になります。 これにより、ホストシステムのメモリは、通常のプロセス間と同様に柔軟にVM間で分散されます。



物理量を超える仮想リソースの作成は、多くのホスティング会社に愛されている「オーバーコミット」および「オーバーセルリング」という用語で示されます。



バロニングには、2つのソフトウェアコンポーネントの協調動作が必要です。

  • ホストシステムのMOM （メモリオーバーコミットマネージャー）。VMからの要求に基づいてVMのRAMの量を変更します。
  • 仮想PCIコントローラーを介してMOMと対話するゲストOSのVMM（仮想メモリマネージャー）。


KVMの最新バージョンのMOMは自動的に有効になり、古いバージョンではコマンドラインで「kvm ... -balloon virtio」を使用して有効にするよう要求されました。



Windowsは、MOMハードウェアマネージャー（devmgmt.msc）と通信するためのゲストデバイスを「未知の種類のPCI標準RAMコントローラー」として認識します。 virt-storやvirt-netとは異なり、そのドライバーは自動的にインストールするようには提供されません。 代わりに、デバイスのプロパティに移動し、「ドライバー」タブで更新を選択し、VirtIO CDのballoon.infへのパスを手動で指定します（ proof ）。 その後、デバイスの名前が「VirtIO Balloon Driver」に変更されます。



ACPI



デフォルトでは、Windows 2003では、ログインパスワードを入力し、[スタート] => [シャットダウン]を選択し、メモを入力して[OK]をクリックするという唯一の方法で自分自身をオフにすることができます。 もちろん、このアプローチはVDSファームでは受け入れられません。 KVM（およびQEMU）はACPIをエミュレートできます。 「system_powerdown」コマンドは物理コンピューターの電源ボタンを押すのと似ていますが、Windowsはそれを無視します。 次の.regファイルで処理されます。 

 Windows Registry Editor Version 5.00 [HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion\policies\system] "ShutdownWithoutLogon"=dword:00000001 "DisableCAD"=dword:00000001 [HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Windows] "ShutdownWarningDialogTimeout"=dword:00000003


実行する準備ができました： svn1.sytes.net/linuxkvm/win-acpi-kvm.reg



詳細な説明を含む元のソース： umgum.com/acpi-windows2003-shutdown



キャッシング



ゲストディスクのイメージがVDSファームにファイルとして保存されている場合、ゲストファイルのキャッシュは2倍になる可能性があります。ゲストOSは、仮想ディスクにアクセスするときに最初にゲストOSをキャッシュし、次に物理ディスクにアクセスするときにファームOSをキャッシュします。



合計で、 3つのメインモードが可能です。

  • none-ホストシステムは、読み取りまたは書き込みのためにイメージファイルをキャッシュしません
  • ライトバック-すぐに書き込み、読み取りはキャッシュされます
  • ライトスルー-読み取りと書き込みはキャッシュされます


異なるバージョンのqemu / kvmおよび異なるオペレーティングシステムでは、デフォルトで異なるモードを使用できます。 たとえば、バージョン1.2より前のQemuはライトスルーを使用し、1.2はライトバックに切り替え、Proxmoxではcache = noneが選択されています。



例外なく、Web上の情報源は、ライトスルーを最も低速なものとして使用しないことを推奨しています。 主観的な評価によると、ライトバックはWindowsを搭載したVMに最適であり、LinuxおよびFreeBSDを搭載したVMには最適ではありません。



ネットワークがハングする



KVMエラーが明らかに引き起こす唯一の深刻な問題は、トラフィックが多い場合のゲストネットワークのフリーズです： bugs.centos.org/view.php?id=5526 （エラーの説明自体の他に、他のバグトラッカーへの重要なリンクがあります）。



議論の参加者によって提案された推奨事項（qemu-kvmとカーネルの更新、vhost-netを使用したコマンドラインパラメーターの変更）は、残念ながらまだ解決できていません。



フリーズするたびに、VNC経由でVMコンソールに移動し、ネットワークインターフェイスをリセットする必要があります。その後、トラフィックは再び正常になり始めます。



PingFailed_ResetNic.au3ファイルを作成し、タスクマネージャーによって数分ごとに呼び出す場合、AutoItを使用してWindowsでこのアクションを自動化できます。 

 #include «EventLog.au3» Local $PingHost = "192.168.0.1" Local $Interface = "LAN" Ping($PingHost, 250) If @error = 0 Then Exit Local $hEventLog = _EventLog__Open("", "RestartNicOnPingFailure") Local $aEmpty[1] = [0] _EventLog__Report($hEventLog, 2, 0, 1, "", "Restart NIC " & Interface & " on failed ping to " & PingHost, $aEmpty) _EventLog__Close($hEventLog) RunWait("netsh interface set interface " & $Interface & " DISABLED", "", @SW_HIDE) RunWait("netsh interface set interface " & $Interface & " ENABLED", "", @SW_HIDE)


CMD.EXEのオプション： rickosborne.org/blog/2007/02/stupid-windows-tricks-restart-network-adapter-when-it-hangs



このような「解決策」は必ずしも満足できるとは限りませんが、場合によっては、マイナスの影響を許容可能な最小値に減らすだけで十分な場合があり、より抜本的な対策の代わりに修正が現れるのを待つことができます。
