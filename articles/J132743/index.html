MODx Revo、ログイン認証セットアップ。 基本設定

この記事の大部分は、 公式rtfmからのレッスンの翻訳、つまり「ログイン」コンポーネントですが、翻訳者が挿入されています。 最高の翻訳には行きませんが、最終的には本質が明確になるはずです。 それにもかかわらず、私はまだロシア語への翻訳を見つけていません。

このレッスンは「基本」と呼ばれていますが、多くの相互接続された部分があります。 このレッスンでは、作成する必要があるページ、完全なユーザー認証のためにページに公開する必要があるチャンク（コードスニペット）およびスニペットについて説明します。



カッコ内の数字はリソースIDです。 これはpage_idであり、異なる場合があります。 ここでは、単に便宜上選択されています。



必要なページを作成します



コードフラグメントを含むチャンクをレイアウトする前に、5ページを作成します。

  • ログインページ（1）：ログインフォームを含むページ
  • パスワードリセット（2）：ユーザーがパスワードリセットを要求できるページ
  • パスワードリセット、ハンドラー（3）：実際にパスワードをリセットする非表示のページ。 ユーザーには表示されません。
  • ユーザー専用ページ（4）：コンテンツがサイトの承認されたユーザーのみに表示されるページ
  • 終了ページ（5）：正常終了後にユーザーがリダイレクトされるページ


これが、現時点でのリソースツリーの外観です。 リソースIDは異なることに注意してください。 この例では、Loginコンポーネント用のページしかありません。







次に、ユーザーとリソースに正しい権利を割り当てる必要があります。



必要なユーザーグループとリソースグループを作成する



MODX Revoには、ユーザーのアクセス許可に関して非常に柔軟な権限の細分化システムがありますが、このトピックでは、トピックに深く入り込まずに必要なことだけを行います。 それでは、始めましょう。

1.セキュリティ→リソースグループ

たとえば、「リソースグループの作成」をクリックし、「ユーザー専用」と呼びます。 [保存]をクリックすると、このページで他の変更は行われません。









2.セキュリティ→アクセス制御

最初のタブ「ユーザーグループ」で「新規ユーザーグループ」をクリックします。 新しいグループに「ユーザー」という名前を付け、「保存」をクリックします。 ユーザーグループは、「ユーザーのみ」のリソースにアクセスできます。 なぜこれが必要なのか、このレッスンの一部として後ほど学習します。









3.同じページ（[ セキュリティ]→[アクセス制御] ）で、作成したユーザーグループを右クリックし、[ユーザーグループの編集]を選択します。









次に、「リソースグループへのアクセス」タブに移動し、「リソースグループの追加」をクリックします。 正常に動作させるには、少なくとも次のパラメーターが必要です。

リソースグループ：ユーザーのみ（作成したばかり）

最小役割：メンバー9999

アクセスポリシー：ロード、リスト、表示

コンテキスト： web

保存します。









4.セキュリティ→ユーザー管理

「新しいユーザー」を作成し、ユーザーのアクセス権の差別化がどのように機能するかを確認します。

この場合、単純なユーザー名とパスワードを使用します。先ほど書いたように、新しいユーザーがUsersグループに属していることを確認することが重要だからです。 これを行うには、[アクセス許可]タブに移動し、[ユーザーをグループに追加]をクリックします。

ユーザーグループ：ユーザー

役割：メンバー

次に、ウィンドウで「保存」をクリックし、コントロールパネルの右隅でもう一度クリックします。









これにより、新しいユーザーが「ユーザー」権限でログインして、「ユーザー専用」権限でページを表示できるようになります。

次に、ページに戻って、対応するページにスニペットとコードスニペットを追加します。



ページにスニペットを追加する





ログインページ（1）



次のスニペット呼び出しコードをユーザーのログインページに配置します。 

[[!Login? &loginTpl=`lgnLoginTpl` &logoutTpl=`lgnLogoutTpl` &errTpl=`lgnErrTpl` &loginResourceId=`4` &logoutResourceId=`5`]]


ログインフォームを表示するチャンク（コードスニペット）には、「パスワード回復」のページへのリンクも含める必要があります。 フォーム呼び出しにあるものを見てみましょう：

＆loginTpl = `lgnLoginTpl`-ユーザーが認証されていない場合、lgnLoginTplチャンクがログインフォームを担当します。

＆logoutTpl = `lgnLogoutTpl`-ユーザーが認証されている場合、lgnLogoutTplチャンクの内容が表示されます

＆errTpl = `lgnErrTpl`-認証用のデータが誤って入力された場合、または単純な単語「error」で入力された場合、チャンクlgnErrTplが表示されます

＆loginResourceId = `4`-認証が成功した後にユーザーをリダイレクトする場所。 リソースIDとして示されます。この場合は4。

＆logoutResourceId = `5`-ユーザーがログインしている場合、ユーザーは終了時にIDが5のページにリダイレクトされます。

また、これらのチャンクは標準であり、好み、レイアウト、実装ごとに独自のチャンクを作成できることに注意してください。 これにより、入力/出力フォームなどを任意の形式で表示することができ、さらにコードに混乱をもたらすこともありません。 ちなみに、標準テンプレートも変更できます。

以下は、まだログインしていないユーザーに提供されるコード、つまり「ログインフォーム」です。 チャンク-lgnLoginTpl 

 <div class="loginForm"> <div class="loginMessage">[[+errors]]</div> <div class="loginLogin"> <form class="loginLoginForm" action="[[~[[*id]]]]" method="post"> <fieldset class="loginLoginFieldset"> <legend class="loginLegend">[[+actionMsg]]</legend> <label class="loginUsernameLabel">[[%login.username]] <input class="loginUsername" type="text" name="username" /> </label> <label class="loginPasswordLabel">[[%login.password]] <input class="loginPassword" type="password" name="password" /> </label> <input class="returnUrl" type="hidden" name="returnUrl" value="[[+request_uri]]" /> [[+login.recaptcha_html]] <input class="loginLoginValue" type="hidden" name="service" value="login" /> <span class="loginLoginButton"><input type="submit" name="Login" value="[[+actionMsg]]" /></span> </fieldset> </form> </div> </div> <a href="[[~2]]"> ?</a>


テンプレート内に「パスワードを忘れた」（パスワードリセット）へのリンクを配置し、IDで参照できます。 私の場合、パスワード回復を使用したリソースIDは2です。



パスワードリセット（2）



まず、このページはほとんどの場合メニューから隠されています。 ほとんどの場合、ユーザーがパスワードをリセットできるフォームです。 ここで、パスワードをリセットするページ（私の場合はID 2）には、パスワードを復元する必要があると言う機会があり、リセット自体は「パスワードのリセット、ハンドラー（3）」ページで行われることを説明します。

次のコードでパスワードリセットフォームを呼び出します。 

 [[!ForgotPassword? &resetResourceId=`3` &tpl=`lgnForgotPassTpl`]]


呼び出しを分析しましょう：

＆resetResourceId = `3`-ページハンドラ（私の場合はID 3）、リセットされるのはそれです。

＆tpl = `lgnForgotPassTpl`-パスワードリセットフォームがあるコードを含むチャンク。

ハンドラーの理由と仕組みを理解するために、写真をご覧になることをお勧めします。











パスワードリセット、ハンドラー（3）



このページはメニューから非表示にする必要があります。 それにもかかわらず、それへのリンクがあっても、それは単に無意味になります。

その本質は次のとおりです。 ユーザーがパスワードのリセットを要求すると、ハッシュと同様に、このリンクとともにメールにメッセージが送信されます。 ユーザーがメールからリンクをクリックすると、MODxはパスワードをリセットし、ユーザーをログインページにリダイレクトします。

ハンドラーページでは、スニペットは次のように呼び出されます。 

 [[!ResetPassword? &loginResourceId=`1`]]
-＆loginResourceId = `1`は、リダイレクトするページを示します。この場合、ログインフォームがあるID 1のページです。



ユーザー専用ページ（4）



これは、ユーザーが正常にログインしたことを確認するページです。 このページのコンテンツは、許可されたユーザーのみが利用できます。

許可されたユーザーのみがページにアクセスできるようにするには、ページに権限を割り当てる必要があります。 これを行うには、リソースツリーで[ユーザー専用ページ]をクリックします。 次に、[アクセス許可]タブに移動すると、[ユーザーのみ]というテキストが表示され、その右側の[アクセス]列にチェックされていないチェックボックスが表示されます。 チェックボックスをマークして、「保存」をクリックします。











終了ページ（5）



これは、ユーザーがアカウントから正常にログアウトした後にアクセスできるページです。 このページには、メインのページ（さようならメッセージなど）、またはログインフォームへのリンクのみを含める必要があります。 例： 

 <p>  !  !</p>

終了ページ（5）



これは必須ではありませんが、テストおよび一般的に開発を容易にするために、終了するための特別なリンクを取得できます。 これを行うには、「新しいリンク」（Webリンク）を作成し、そのページにID 1を入力します。たとえば、次のようなリンクを含むページを呼び出します。 

 [[~1? &service=`logout`]]


これは、この形式で終了するために使用できます。 

 <a href="[[~1? &service=`logout`]]" title=""></a>




考えられるエラー



すべてのページを収集してリンクし、アクセス権を設定したら、すべてを確認する必要があります。 すべてが計画どおりに機能する場合、入り口で「ユーザー専用」ページにリダイレクトされます。 ログインページからアカウントを終了できます。許可されている場合、ログインページに出口へのリンクが表示されるためです。 正常に終了すると、「終了ページ」にリダイレクトされます。



ログインできません！



通常、このような問題は、ページIDが正しく指定されていない場合、チャンク名が誤って記述されている場合、または角括弧が欠落している場合に発生します。 コードを注意深く確認してください。



さらに、上記のスニペットはすべてキャッシュなしと呼ばれることを覚えておく必要があります。 もちろん、キャッシュ形式で動作するものもありますが、ほとんどは動作しません。 感嘆符を使用して、キャッシュされていない形式で呼び出すことができます。たとえば、次のように記述する必要があります。 

 [[!Login]]


そうではありません： 

 [[Login]]




ログアウトできません！



どうしてもアカウントを離れることができない理由の1つは、終了リンクがログインフォームのないページを指していることです。 終了ボタンはログインフォームのあるページを指す必要があることを覚えておく価値があります。



また、パスワードをリセットすると、電子メールメッセージが届かないこともあります。 この場合、サーバーでメールを送信するように設定する価値があります。
