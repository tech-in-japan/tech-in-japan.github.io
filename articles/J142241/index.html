キャッシュ


これはプロジェクトの最も強力な部分です。 キャッシュは、遅いバックエンドの前のシールドになります。 もちろん、多くはすでにそれが何であるかを推測しています。



どのように機能しますか？ それは簡単です：ユーザーがページをリクエストし、nzhinksはこのページがキャッシュにあるかどうかを確認します。そうでない場合、dzhangはそれを生成し、キャッシュに入れてnzhinksに渡します。 つまり 実際、一度サイトを通過するだけで十分であり、その後のすべての時間、サイトはmemcashd + nzhinksの速度で動作します。



本当に素晴らしい？ まあ、ほとんどの場合、このタイプのキャッシュは私たちのケースに適しています。複雑なプロジェクトには、多くの欠点があります。 たとえば、ユーザーが入力したフォームを更新することはできません。なぜなら、このURLを通過するたびに、ユーザーはキャッシュから同じフォームを取得するからです。 ファッションを称賛する必要はありません:)

計画を実行するために、ミドルウェアを作成します。 インターネットには、そのようなバンドルの作業を実装する方法に関する記事があります。 それは以前のバージョンのdzhanga向けに書かれたものであり、rawを見ることなく、それを理解することができませんでした。 私はそれを修正し、バカに更新します。 ミドルウェアでは： 

 import re from django.core.cache import cache from django.conf import settings class NginxMemCacheMiddleWare: def process_response(self, request, response): url = request.get_full_path() cache_it = not settings.DEBUG \ and request.method == 'GET' \ and response.status_code == 200 if cache_it: stoplist = [ x for x in settings.CACHE_IGNORE_REGEXPS if re.match(x, url) ] if not stoplist: cache.set(url, response.content) return response


GETのみをキャッシュし、開発時と404にはキャッシュしません。そしてなぜですか？ しかし、管理パネルまたは攻撃者自身を探しているボットは、あらゆる種類のジャンクでメモリを埋めることができるためです。



構成に戻りましょう： 

 CACHES = { 'default': { 'BACKEND': 'django.core.cache.backends.memcached.MemcachedCache', 'LOCATION': 'unix:/tmp/memcached.sock', 'KEY_PREFIX': 'YOURSITE', } }


ここではすべてが簡単です。バックエンドmemkeshd、ソケット、プレフィックス。 ところで、私はpylibmcをソケットから取得できませんでした。彼はそこに何かが好きではありません。 python-memcachedを使いました。 

 CACHE_IGNORE_REGEXPS = ( re.compile(r'/admin.*'), re.compile(r'/some_url.*'), )


これは、キャッシングが禁止されているURLのリストです。 設定でMIDDLEWARE_CLASSESに新しいミドルウェアを追加することを忘れないでください。



ところで、memcashdのステータスを監視するには、 django-memcache-statusプログラムを使用できます。 彼女は、管理パネルでmemcashdの使用済みメモリのスケールを描画し、あらゆる種類の情報を表示します。 正直なところ、私はそれをあまり使用していませんでした:)フラットページ管理者のためにすでに退屈な管理者を希釈するだけです。 残念ながら、彼女はdjango-admin-toolsと友達ではありません。



Nginx


サーバーdevの設定、後で更新します： 

 upstream dev { server 127.0.0.1:8000; } server { listen 80; server_name dev.local; charset utf-8; location ~^/(media|static) { root /home/user/project; access_log off; break; } location / { if ($request_method = POST) { proxy_pass http://dev; break; } default_type "text/html; charset=utf-8"; set $memcached_key "YOURSITE:1:$request_uri"; memcached_pass unix:/tmp/memcached.sock; error_page 404 502 = @fallback; } location @fallback { proxy_pass http://dev; } }


次の行を参照してください： 'set $ memcached_key "YOURSITE：1：$ request_uri";'？ ここですべての魔法が発生します。 各ページは、キープレフィックス+キャッシュのバージョン+完全なURLによってキャッシュに保存されます。 一度ページを開くだけで十分です。dzhangaはそれをキーのキャッシュに入れ、次にnzhinksがこのキーでそれを取得します。 dzhangaサイトをオフにしても動作します。 memcashdをプルすることなく、すぐにリクエストをバックエンドに送り返します。 別のポイント：ユーザーがjsを無効にしている場合、フルバックモードにget-keysを使用します（noscript：Pがあります）：すべての種類のタブがあるため、uriの代わりにrequest_uriがget-keysのない最後のURLです。



janga 1.3以降、キー全体を自分で生成する必要はありません。 set（）、get（）、delete（）メソッドに一意の部分を渡すだけで十分で、残りはjungが個別に実行します。 この場合のユニークな部分は、絶対ページURLです。



通常のgzipには小さな問題があります：nzhinksはmemkeshdから取得したものを圧縮しません。 私はネットワーク上で4年前にNinxのバージョン0.6のパッチを見つけました。すでに1 *です。 IE6を提供しない場合は、jung自体で圧縮を有効にし、すでに圧縮されたコンテンツを提供できます。 私たちの場合、スターターブラウザーは重要です。 それ以外の場合は、gzipが機能します。

それで、私たちは何をしましたか？



  • mysqlをsqliteで置き換えることでリソースを節約しました（dbをメモリに入れることで少し費やしました:)が、とりあえずは非常に高速になりました。
