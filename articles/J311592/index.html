チャットボット、およびマイクロソフトがこれをどのように支援するか

このレビュー記事は、IT業界で急速に成長しているトレンド（チャットボット、およびその開発プロセスにおけるマイクロソフトの役割）の研究に専念しています。 発生の履歴、ボットの機能、アプリケーションの主な領域と予期しない領域、技術の見通しと限界が考慮されます。



調査のための主要な比較プラットフォームとしてのマイクロソフトの選択は偶然ではありません。 同社は、インテリジェントボットの促進と開発の分野で多くの仕事をしています。 この方向の主なステップの1つは、特定のプラットフォームやオープンソースに結び付けられていない独自のMicrosoft Bot Frameworkボットを作成するためのBuild 2016展示会で発表した無料のフレームワークと考えることができます。



はじめに



一般に、チャットボットは、ユーザーがテキストまたは音声を使用して理解できる言語で、ユーザーとの有意義な対話をシミュレートできるプログラムです。 このようなダイアログの目的は、原則として、コマンドとユーザー要求のボットによる実行です。 基本的に根本的に新しいものではないにもかかわらず、チャットボットは何らかのノウハウとして市場に位置付けられています。 この最後の役割は、ユーザーにとってスマートで、多くの場合、かけがえのないアシスタントなどのボットを中心に作成された特定の感覚によって果たされるわけではありません。



チャットボットは、メッセンジャーの他のユーザーとコミュニケーションをとったり、新鮮なジョークやゲームを楽しんだりするのに飽きさせません。 新鮮な天気予報を送信します。 ピザの注文を手伝います。 あなたの体重と血圧についてのメッセージに応じて、健康的な食事に関するアドバイスを与える リーガルサービスのウェブサイトであなたとの対話を巧みに行い、真の対談者を模倣します; 心地よい声で彼らはあなたの気分について尋ね、答えの分析に基づいて、スマートフォンで楽曲を拾うことができます; 電子メールでサービスのサポートスペシャリストの役割に自信を持って対処する。 これは、インテリジェントボットを実際に使用する例のほんの一部です。 同様に重要な企業タスクを解決する企業ボットの別のグループもあります。









すべての新しいテクノロジーと同様に、チャットボットは、サイト、アプリケーション、プッシュ通知など、より身近な通信方法とエンドユーザーへの情報通信の論理的な開発です。 同時に、このテクノロジーは魅力的です。まず第一に、エンドユーザー、最終プラットフォームへの統合、使いやすさ、インテリジェンス、使いやすさに焦点を当てています。



大規模なITプレーヤー（主にオンライン通信市場）は、ソリューションでチャットボットを使用することの潜在的および可能な利点を長い間評価してきました。 何らかの形のボットは、すべての一般的なインスタントメッセンジャー、インターネットプラットフォーム、モバイルデバイスで積極的に使用されています。



マイクロソフトは、一度に複数の方向で作業を行ったため、傍観しませんでした。モバイルデバイスCortanaのアシスタント。 成功したわけではありませんが、TwitterでボットガールのTayを使った実験は間違いなく興味深いものです。 ボットガールXiaoice（Xiaois）による中国でのより成功した実験。 Skypeのボット。 最後に、さまざまなプラットフォーム用のカスタムボットを作成および統合するためのMicrosoft Bot Framework。



ちょっとした歴史



最初のチャットプログラムは、1964年にマサチューセッツ工科大学のジョセフウィゼンバウム教授によって開発されたELIZAと考えられています。 このプログラムは、ユーザーの心理療法士とのやり取りを模倣しようとし、仕事の驚くべき信頼性と「これについて話したいですか？」などのフレーズによって患者を落胆させます。



この後、このような仮想対談者を作成するためにさらに多くの試みが行われましたが、それらはすべて非常に単純であり、人は自分が機械を扱っていることを簡単に認識できました。



「チャットボット」という用語の使用は、開発者のマイケル・モールディングに起因します。マイケル・モールディングは、1994年にコミュニケーション用プログラムJuliaを開発し、チャットボットと呼びました。 このプログラムは簡単でしたが、それでも多かれ少なかれ有意義な対話を維持することができ、自己学習の機能をサポートしました。



1990年代後半から2000年代初頭に、単純なチャットボットが、IRCやIMプログラムなどで積極的に使用され、ユーザーの会話に多様性を加え、チャットルームをモデレートしました。 そして、もちろん、世界中のAI愛好家や研究者は、ユーザーとの本格的な対話を行うための真に「スマートな」プログラムを作成しようとする試みを放棄しませんでした。



これらすべてと技術の自然な発展は、インテリジェントなチャットボットの作成に大きな進歩をもたらしました。 これまでに、チューリングテストに合格したチャットプログラムが既に作成されています。 2014年6月8日、イギリスのレディング大学が主催するイベントで、サンクトペテルブルクで開発されたプログラムは、実在の人物とのコミュニケーションを模倣することに成功し、一部の試験官を誤解させました。



また、IBM Watsonスーパーコンピューターに基づいて、DeepQAは着信コマンドを自然言語で処理できます。 これに使用されるNuance Communicationsテクノロジーは、Appleがこの会社のデバイスでSiriアシスタントを実行する際にも使用されます。



最新のチャットボットは、エリザなどの先駆的なプログラムでは達成できない高さに達しました。 彼らは自然言語と音声の分析の分野で高度な開発を使用し、高度なAIアルゴリズムを持ち、自己学習が可能です。



適用分野



チャットボットは、メッセンジャー、Webポータル、スマートフォンなど、ほぼすべてのプラットフォームで使用できるようになりました。 さらに、これらはすべて特定のシナリオで、ユーザーの特定のニーズに使用されます。









チャットボットを使用するための主なシナリオに従って、次の分割を実行するのが慣例です。
  • 技術サポート;
  • 指定されたパラメーターによる検索。
  • コンテンツの消費とエンターテイメント。
  • ショッピング。


技術サポート。 多くのサービス会社は、テクニカルサポートを組織するために、ボイスチャットボットだけでなく、電子メールを介したテキストボットを長い間使用してきました。 同時に、これらの企業の一部の声明によると、彼らはスタッフによる顧客のテクニカルサポートに以前に費やした資金の最大40％を節約することに成功しています。 この場合のチャットボットの主な利点は、夜間に休憩をとることなく、同時に多数のクライアントと作業できることです。 短所は通常、そのようなボットの知性の欠如にあります。 困難な場合、オペレーターはクライアントの問題を解決するために接続する必要があります。



指定されたパラメーターで検索します。 私の意見では、インターネットを検索するためにチャットボットを使用することは、従来の作業方法と比較してまだ十分に便利ではありません。 もちろん、スマートフォンやメッセンジャーに組み込まれたボットを使用する場合、別のアプリケーションに切り替えて検索を開始する必要はありません。 しかし、あなたはまだ何らかの形で質問を定式化し、そしておそらく、結果を達成するためにボットからのいくつかの追加の質問に答える必要があります。 さらに、どの検索エンジンでもすぐに意味のある結果が得られます。 Ariel Verberの記事はこの問題に当てられており、ボットとGoogle検索エンジンを使用した検索の使いやすさを比較しています。



コンテンツの消費とエンターテイメント。 この領域はすでにボットによってより良く開発されていますが、ここでは、検索の場合と同様に、前述の問題が発生する可能性があります。 ただし、サブスクリプションコンテンツをダウンロードするボットを使用することにより、状況は大幅に改善されます。 このコンテンツは通常、ニュースと記事で構成されています。 同時に、ボットは必要なコンテンツを配信するだけでなく、目的のトピック、関連性などに応じてフィルタリングします。 また、ボットはユーザーのダウンロード履歴を分析し、コンテンツを推奨できます。 多くの大規模な情報ポータルおよびニュースアグリゲーターは、独自のボットを積極的に使用して、資料へのアクセスを簡素化します。 あらゆる種類のアシスタントボット、ゲーム、エンターテイメントチャットボットもこの領域に起因します。 Telegramメッセンジャーでボットが作成される最も一般的な目的の1つがデートしていることは秘密ではありません。



ショッピング。 おそらく、チャットボットのアプリケーションで最も興味深い分野は、仮想アシスタントと現実世界のオブジェクトをリンクすることを可能にし、特定のユーザーにボットが焦点を合わせ、実際のコミュニケーションを模倣することで、特定の方法でそれらを結び付けることです。 しかし、残念なことに、このユースケースは現在最も開発されていません。 これは、ボットのインターフェースを介して直接商品を直接支払うことは頻繁に不可能であり、このような操作にはこのインターフェースが根本的に不適切であることによるものです。 特定の製品を選択するには、チャットボットが引き続き役立ちます。 しかし、大規模な品揃えと複雑な取引オプションを扱うことは彼の力を超えています。 この場合、仮想ボットトレーダーの仕事は検索エンジン機能に帰着します。



チャットボットの開発は着実に継続しています。 アプリケーションの両方の新しい領域が発生し、古いシナリオが改善されます。 たとえば、北米のティーンエイジャーの間で人気のあるKikメッセンジャーを使用すると、スマートフォンでQRコードをスキャンすることでチャットにボットを追加できます。 たとえば、カフェの仮想ウェイターとチャットして、実際の店員に会う前に注文することができます。 そして中国では、WeChatメッセンジャーで既に支払い機能を実装しています。現在、アプリケーションはチャットボットを含むオンライン取引を積極的に開発しています。



チャットボット市場



今日、チャットボットの使用におけるリーダーシップは、メッセンジャーによって自信を持って保持されています。 また、これは驚くべきことではありません。なぜなら、インスタントメッセージング用のアプリケーションとユーザー間のコンテンツはチャットボットに最適な場所だからです。 インスタントメッセンジャーの積極的な開発が、チャットボットのほぼ普遍的な使用に具体的な推進力を与えました。 現在、世界が25億人のメッセンジャーを少なくとも1つ使用しているという統計を引用しています。 そして、インスタントメッセンジャーの間で人気のあるリーダーは、Facebook MessengerとWhatsAppです。 専門家によると、今後2〜3年で、世界の住民の半数以上がメッセンジャーのユーザーになり、人気のソーシャルネットワークを絞るでしょう。



インスタントメッセンジャーでのチャットボットの使用が普及したもう1つの要因は、大企業のサポートです。 ほとんどの企業は、複雑なインターネットポータルやアプリケーションの開発と比較して、インスタントメッセンジャーでの製品やサービスのパーソナライズされたプロモーションのために、特別なボットを作成および統合できることが、どれほど収益性があるかを徐々に認識しています。 さらに、アナリストによると、世界では新しいアプリケーションに対するユーザーの関心が着実に低下しています。 ほとんどは、テスト済みのインスタントメッセンジャー、ソーシャルネットワーク、およびデバイスにインストールされたプログラムのセットを使用することに慣れています。



チャットボットは、新しい視聴者を惹きつけてサービスを効果的に促進するために、一般ユーザーとエンターテインメントの巨人の両方によって作成されています。 この方向はすでにかなり普及しています。 インターネットでは、あらゆる方向と複雑さのチャットボットを作成および宣伝するための包括的なガイドラインを見つけることができます。 このトピックに関する優れた記事の例 。









インスタントメッセンジャーにチャットボットを導入する動きの先駆者の1人は、Telegramでした。 2015年6月、Pavel Durovの会社はボットAPIプラットフォームを最初に立ち上げた企業の1つであり、ボットをダウンロードするためのストアでもありました。 現在、Telegramメッセンジャーには、約10万の最も多様なボットがあります。 Telegram Bot API 2.0ボットを自己作成するためのAPIの改良版が既に利用可能です。 PHP、Python、Java、C＃、Rubyなど、ほぼすべてのプログラミング言語を使用してボットを作成できます。 この場合、プログラミングスキルなしで、Telegram用の独自のボットを作成できます。 @BotFatherサービスアカウントを使用して新しいボットを登録し、一意のIDを取得するだけで十分です。その後、@ Paquebotアカウントを介して、ボットを@Paquebotにリンクし、新しいボットのすべてのサブスクライバーへのメッセージ送信を開始できます。



他のメッセンジャーもそれほど遅れていません。 Facebookは、2015年3月にメッセンジャーでボットを作成するためのプラットフォームも発表しました。 ボットを作成する機会は、 Messenger Platformサービスの一部になりました。これにより、Facebookチャットを他のアプリケーションに統合できます。 'メッセージを送受信するためのAPIは十分です。 しかし、より複雑なボットを作成する場合は、ボットエンジンを使用してください」と、Facebookの副社長であるデビッドマーカスは述べています。 前述のように、ボットエンジンを使用すると、状況に応じて異なる方法でボットを「教える」ことができます。 競合他社とは異なり、Facebookはすでにユーザーが通常のテキストではなく、画像、リンク、ボタン、その他のやり取りが可能な要素を含むインタラクティブな「カード」を送信することを許可しています。 ロボットとのやり取りにより、ユーザーはレストランのテーブルをすぐに予約したり、商品のカタログをめくりながらいつでも好きな商品を購入したりできます。



チャットボットを作成するための独自のサービスの開発、インスタントメッセンジャーおよび他のプラットフォームへのボットの導入は、長年にわたって単なる良い形ではなく、不可欠なものになっています。 世界中の多くのサービスが、ボットを使用するというアイデアを積極的に推進しています。 検索エンジン：アメリカのGoogleとロシアのYandex。 インスタントメッセンジャー：中国のWeChat、日本線、韓国のカカオトーク、カナダのKikなど



スマートフォンのメーカーも、じっと座っていませんでした。 Appleの音声アシスタントSiriとMicrosoftのCortanaは、音声インターフェイスを積極的に使用してユーザーの要求を満たします。 アメリカのアマゾンには音声アシスタントもあります。



専門家によると、近い将来、チャットボットの既存のプラットフォームへのさらなる統合と、この分野の収益化の増加が期待されています。



マイクロソフトの貢献



2016年3月、Microsoftの専門家は、Tay（@TayandYOu）という名前のチャットボットをTwitterに投稿しました。 Thayは、KikメッセンジャーまたはGroupMeソーシャルネットワークの連絡先リストに追加することもできます。 当初、開発者はプログラムと通信し、ソーシャルネットワークからの匿名の情報の大規模な配列でプログラムを教えました。 少人数のグループもトレーニングに参加しました。 著者によると、甘くて優しい10代の少女のふるまいを模倣したThayは、公開された後、ユーザーから学びながら独学を続けることになっていた。 そのため、ユーザーとのコミュニケーションが始まってからわずか1日足らずで、Thayはナチス、反ユダヤ主義、そして不幸な人間になりました。 プログラムは、それが教えられたすべてを成功裏に学びました。 残念ながら、著者はプログラムに善と悪を区別する能力を入れることができませんでした。 しばらくの間、Microsoftはいくつかの特に不快な発言を編集することにより、Thayの行動を制御しようとしました。 しかしすぐに、実験は完了したと宣言され、Thayは切断されました。 ただし、このイベント中、マイクロソフトは優れた広告を提供し、AIとチャットボットの使用に対する関心を大幅に高めました。



マイクロソフトが関与した別の同様の実験は、より成功しました。 中国のツイッター（Weibo）の類似版では、Xiaoiceと呼ばれる言語ボットが発売されました。 ボットは、17歳の10代の少女の行動を模倣し、意味のある対話を維持し、同情を表明し、通常の人のように非常に予測不可能な動作をすることがよくあります。 どうやら、Xiaoisに取り組んでいるとき、著者はThayに対する以前の否定的な経験を考慮に入れていました。 何百万人もの中国の十代の若者たちはシャオイと「何時間も過ごし、シャオイを「敏感で思いやりのある少女」と呼んでいます。



3月30日、 Build 2016展示会で、Microsoftは、Skype、Slack、Facebook Messenger、Kik、Office 365メール、その他の一般的なサービス用のテキスト、オーディオ、ビデオボットを作成するMicrosoft Bot Frameworkを発表しました。 プラットフォームは、3つの主要部分で構成されています。



  • ボットビルダーSDK：チャットボットを作成およびテストするための一連のオープンソースツール（Node.js、.NET、またはRESTを開発に使用できます）。
  • ボットフレームワーク開発者ポータル：ボットの登録、人気チャンネルへの接続、チューニングとテスト、公開のためのサービス。
  • ボットディレクトリ：ボットフレームワークを使用して作成され、開発者ポータルに登録されたボットのカタログ。ボットのダウンロードと確認、エクスペリエンスの共有のために設計されています。


ボットフレームワークの主な利点には、特定のプラットフォームへのバインドの欠如、オープンソースSDKの可用性、ボットの高度なデバッグおよびテストツール、ビルトイン通信およびWebアプリケーションの管理、30以上の言語への自動翻訳のサポート、およびコミュニティの存在が含まれます。 同時に、Microsoft Bot Frameworkは積極的に開発を続けています。



また、Build 2016展示会では、Windows、Android、iPhone、iPadのボットをサポートするSkypeの新しいバージョンのリリース、およびSkype BotプラットフォームのSDKとAPIのオープンが発表されました。 Skype、Slack、またはTelegramの連絡先リストにボットを追加するには、 ボットディレクトリを使用できます。 iOSおよびAndroid用の新しいバージョンのSkypeでは、ボットの検索はアプリケーションに直接実装されます。



Microsoftは、それほど遠くない将来に、音声通話またはビデオ通話（Skype Video Bots）を介して自然言語でチャットボットとチャットできるようになると述べています。 同時に、チャットボットは高度なAIを備え、一種の3Dモデルになります。 マイクロソフトは、ボットを使用してなじみのあるユーザーアクションを自動化し、音声制御を使用すると、コミュニケーション、ゲーム、ショッピングなどの通常の方法が変わると確信しています。



軟膏で飛ぶ



しかし、チャットボットは本当にすぐに市場から通常のアプリケーション、検索エンジン、人事コンサルタントを追い出すことができるほど大きな潜在能力を持っているのでしょうか？ 彼らは時間の経過とともに迷惑な、あるいは危険な要因になりますか？ この問題については非常に異なる意見があります。









これらの意見の1つ：チャットボットは、単なる新しいユーザーインタラクションメカニズムにすぎません。 これはチェーンの論理的な展開です：サイト-アプリケーション-チャットボット。 さらに、この相互作用では、前述のように、高度なAIは完全にオプションです。 ボットには、よく考え抜かれたコマンドシステムが必要です。 同時に、ボットの有用性は疑いの余地はありません。



もう1つの観点は、実際にはボットが情報を取得するための特別な方法であるということです。 現時点で存在するものはすべてすでに十分です。 そして、App Storeからお気に入りの雑誌の新しい号または新しいアプリケーションをダウンロードするために、インスタントメッセンジャーを起動し、目的のチャットボットを探し、彼と話し、途中で2〜3の主要な質問に答えてから、必要な質問を取得するだけです。 既存のアプリケーションは消えていません。それらはまだ手元にあり、便利です。 そして、通信のプロセスに直接積極的に関与しているチャットボットのみが有用です：翻訳者、エキストラ、アナリストなど。



現在でも、製品を効果的に宣伝できるチャットボットを作成することはできません。 彼がすでに選択したもの、つまり購入を行う必要性を人に納得させるために、人に提供しないでください。 もちろん、スマートチャットプログラムを作成するために実施されている実験は印象的です。 しかし、彼らにはまだ多くの開発が必要です。 最新のチャットボットとATMを比較する場合の例は興味深いかもしれません。 はい、ATMはルーチンアクションをより速く、エラーを減らして実行し、ユーザーと対話します。 しかし、ATMはユーザーにローンを引き付けるよう説得することはできません。



この状態では、主にボットの主な生息地であるインスタントメッセンジャーのメーカーと大規模プラットフォームの所有者が、チャットボットの不可欠性、利便性、特にインテリジェンスの問題を非常に過熱し、維持していると断言できます。 多くの場合、大規模なプラットフォームの所有者のみが、高度なAIと幅広い機能を組み合わせた真にインテリジェントなチャットボットを作成する余裕があります。



これで、さまざまなサービスを使用してボットを作成できます。 メッセンジャープラットフォームで提供される特別なサービスを使用して、ソースコードを1行も書くことなくボットを作成できます。 Chatfuelなどのサードパーティサービスを使用できます。 Microsoft Bot Frameworkなどの無料の開発ツールが利用可能です。 最後に、 Gupshupサービスを通じてボットを作成するためのツールを購入できます。 しかし、ボットは、使い慣れたユーザーアプリケーションと完全に競合するのに十分なほどインテリジェントです。 あなたのチャットボットはあなたの環境に必要不可欠になるのに十分適合しますか？ または、良いが正直なところ十分な単調な仮想アシスタントの大規模なコホートを補充しますか？



おそらく、チャットボット業界は、時間の経過とともに、モバイルアプリケーションやインターネットアプリケーションと同じことを期待するでしょう。 最も求められている「スマート」で快適なチャットボットは、特定の生息地に可能な限り収まります。

アナライザーは、キャスト演算子の使用に関連する疑わしい構造を次のように検出しました。 



protected Type StepType(object step) { var field = step as FieldInfo; var prop = step as PropertyInfo; return (step == null ? null : (field == null ? prop.PropertyType : // <= field.FieldType)); }


指定されたコードフラグメントは、 ステップ変数がFieldInfoまたはPropertyInfoタイプになるまで正しく動作します。 それ以外の場合、変数propに追加のnullチェックがないため、null参照によるアクセスが可能です。



PVS-Studioアナライザーの警告 ： V3105 「parts」変数は、null条件演算子を使用して割り当てられた後に使用されました。 NullReferenceExceptionは可能です。 JwtTokenExtractor.cs 60

 public async Task<ClaimsIdentity> GetIdentityAsync(string authorizationHeader) { .... string[] parts = authorizationHeader?.Split(' '); if (parts.Length == 2) // <= return await GetIdentityAsync(parts[0], parts[1]). ConfigureAwait(false); .... }
