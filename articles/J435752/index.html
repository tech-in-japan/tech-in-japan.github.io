Moscow Python Conf ++ 2019-一部のスピーカーをゼロから準備する最初の会議





「何かをうまくやりたいなら、自分でやりなさい。」 2019年4月5日に、私たちは第4回モスクワPython会議を実施します。私は奇妙な実験を実施することにしました。レポートの準備を逆さまにするとどうなるか見てみましょう。



いつものように、Call for Papersが開きます（開いて、彼には何も起こりませんでした）約1か月半、オーガナイザーはトップスピーカーにレポートを提供するよう説得しますプログラム、そして誰もが息を吐きます。 1つのma-a-a-scarletの欠陥を伴う良い方法：多くの事故。



経験豊富なスピーカーであっても、提案されたトピックについて強力なレポートを作成できるということは事実ではありません。 有名企業の「アクティブ」なスピーカーは忙しいか、別の会議に参加したい場合があります。 プログラム委員会の10人がIVI.ruの腸内にあるPythonの巨大な球体を知らず、誰も書いていないため、多くの興味深いトピックが通り抜けることができます。



私はゴブリンスレイヤーの道をたどり、サイコロを投げるのではなく、自分で会社を訪問し、インタビューを自分で行い、トピックを選択し、最初からスピーカーを準備することにしました。 あなたはそのような申し出から彼らの目を見たはずです...写真はありませんが、カットの下で何が起こったかを教えてくれます。



私たちは会議を秋から春に移したので、秋のモスクワPython Conf ++＃3の完了直後に私は人々に行きました。 大規模なIT企業は人として行動しましたが、DevRelsについては、すでによく知っているか、同じHighLoad ++で話しました：hh.ru、Domklik、Avi​​to、S7、Tinkoff、Iponweb、Yandex、Rambler、Cyan-リストは長いです。 私は皆に同じことを提案しました：



  1. 私は一杯のコーヒーで訪問し、会社のPythonを知っている開発者と話をします。 有望なトピックがある場合-1対1で電話をかけ、コードについて話すことができる人の連絡先を取ります。
  2. 開発者を呼び出し、請求書を収集します。
  3. トピックとテクスチャがある場合は、著者の方法論に従ってスピーカーをゼロから準備することを提案します。 この方法で準備されたスピーカーは、自動的に会議プログラムに入ります。


興味深いのは、私が手を差し伸べてPythonを使用している企業について、誰も拒否しなかったことです。 おそらく誰もがコーヒーの入ったバケツがどのようなものか見たかったのでしょう。 このように：









企業へのインタビュー。 Pythonはありますか？ そしてそれを見つけたら？



Pythonと興味深いトピックを皆で見つけました。 それは通常どのように見えましたか？ 私の周りには、1〜10人の開発者、チームリーダー、技術編が集まって、プロの会議のコンセプトを簡単に提案し、「Pythonの参加により会社で何が行われているのか」と申し出ました。 最初は確信が持てなかったが、この話はすぐに「そしてまだ…」という議論に変わり、約20分後に質問を明確にし、会議に最も有望なトピックを選択しました。



このインタビューの鍵は、なぜ私たちがプロの会議に参加するのかを思い出すことです。 学校や大学の時代から、会議は人々が新しい何かを学び、学ぶ場所であると確信してきました。 これは、「なぜ会議に行くのか」という質問に対して脳が生成する「自動」回答です。 自動で間違っています-できるだけ多くの聴衆に向けて準備するスピーカーが30〜40分のスピーチで「学習」できるものは何ですか？ 新しいことを学び、学ぶことはインターネットに関するものです。 公式ドキュメント、reddit、ハッカーニュース、メディア、Habr、stackoverflow、その他のリソース。



この会議で興味深いトピックは何ですか？



会議-コミュニケーションについて。 インターネットにはすべての情報があります。 しかし、開発者として、私たちはすべての情報を必要としません。 私たちには、最も重要な質問に対する答えが必要です。 テクノロジーを使用し、クールな企業の「製品」でさまざまなタスクを解決した個人的な経験。 主要な専門家会議では、コーヒー1杯（そしてaftapatiのビール）でこれらすべてについて話すことができます。



この角度から会議を見ると、なぜスピーカーがいるのですか？ スピーカーは、会話のトピックを設定し、会社の内部経験について話します。それは、stackoverflowとHabrにないものすべてです。 インタビューでそのようなコンテンツを探しており、「公式文書やスタックオーバーフローで書かれていないあなたと何が起こっているのか」という質問をしています。 そして、興味深いことが常に起こります。Python開発を行うすべての大企業には、独自のタスク、困難、問題、解決策を備えた独自のサーペンタリウムがあります。



Python開発者が行うすべてに関するモスクワPython会議。 バックエンド、ナンバークラッシャー、機械学習、ビッグデータ、Web開発、モバイルアプリケーション、コードのテストと展開、その他多数のトピック。 たとえば、IponwebはPythonのクライアントごとに一意の管理パネルを生成します。 そしてスピーカーは、これでどんな困難が生じるか、そしてそのようなタスクのためにDjangoを適切に裏返す方法を教えてくれます。



ズームを介した目から目への請求書コレクション。 請求書とは何ですか？



対面インタビューの最後に、開発者の連絡先を収集します。開発者は、Zoomを通じて20〜30分間、小規模な電話を手配します。 ビデオ通話の非常に便利なこと：2人のユーザーに対する無制限の無料ビデオ通話、コンピューターにビデオを書き込むことができ、Skypeピアツーピアに典型的な「通話失敗」の問題はありません。 私の意見で唯一のマイナスは、隠された「コンピュータオーディオと結合」ボタンを備えた奇妙なユーザビリティソリューションです。ビデオコールの接続された参加者はビデオを見ますが、他の人の声は聞こえません。 ただし、このボタンについて事前に説明しておけば、他の問題はありません。



私はすでに開発者とコードについて話し合っています-レポートに十分な資料がありますか。 フルタイムのブレインストーミングでは、多くのものが実際とは異なって記憶され、その後の開発者への呼び出しはこのギャップを修正します。



みんなに電話して、私は会社を私の観点から最も適切なトピックに関するオファーにします。 それに応じて、会社はスピーカーを提供します。通常、これは私がズームのためにコードで話した相手です。



スピーカートレーニング



1つのレポートに平均30〜50分のビデオコールが1ダース費やされます。 準備はインタビューの形式で行われ、私は何百万もの質問をして会社のユニークな経験を掘り下げ、スピーカーは知識の源として働きます。 大まかな準備は、次の4つの部分で構成されます。



  1. 請求書の収集 ：最初に、選択したトピックの前の段階で収集されたすべてのものがここにコピーされます。その後、30分間のレポートに十分な資料と価値が表示されるまで、質問でスピーカーを攻撃します。 通常、これは2〜3回の電話で、その間にスピーカーがチームに戻り、詳細とニュアンスを明確にします。
  2. レポートの概要 ：請求書、講演者、および私は事実を整理し、興味深い話が得られるようにします。 通常私は香りを使用しますが、極端な場合には、既製のジャーナリスティックなアプローチを使用できます。 これは最も困難な段階であり、歴史の形成において、いくつかの事実は必要ありませんが、新しい興味深い詳細が表示される場合があります。これについては、さらに学ぶ必要があります。
  3. スライドを作成するのが最も簡単な手順です。 プレゼンテーション計画を立て、何百ものスピーチからの経験を持ち、スライドが必要な理由を理解すると、簡単に作成できます。 予想外の場所にあるリスト、図、面白い写真、矢印は、聴衆が話者の話を追い、キーワードを見て、話の輪郭を失わないようにするのに役立ちます。
  4. 両面スライドの切断 。 スピーチ中に使用する大きなチート：スピーカーは、スピーカーノートセクションに配置され、レポートを多くの多くの小さな孤立したストーリーに分割する、彼の前にこれらの手がかりを見る。 仕組みについては、こちらのビデオをご覧ください。
