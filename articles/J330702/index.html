「OLE DBコマンド」では、たとえば、タイトルまたは価格が変更されたかどうかを確認し、値のいずれかが異なる場合にのみ対応する行を更新するなど、TSQLでより複雑なロジックを記述できます。



明確にするため、DemoSSIS_TargetデータベースのProductsテーブルに新しい列を追加します。

次のコマンドを書きましょう。







チームテキスト： 



 DECLARE @TargetID int=? DECLARE @Title nvarchar(50)=? DECLARE @Price money=? IF(EXISTS( SELECT Title,Price FROM Products WHERE ID=@TargetID EXCEPT SELECT @Title,@Price ) ) BEGIN UPDATE Products SET Title=@Title, Price=@Price, UpdatedOn=GETDATE() WHERE ID=@TargetID END

第二部の結論



このパートでは、小さなディレクトリを同期する方法を検討しました。 ここでは、もちろん、ソース内のデータがまだ削除できる瞬間は考慮されていませんが、必要に応じて、自分で試してみることができます。 削除するときは、追加の要素を考慮する必要がある場合があります。たとえば、他のテーブルから削除されたレコードへのリンクがある場合があります（次の部分ではこれを行う予定です）。

参照整合性に違反しないように、受信テーブルのエントリが論理的に削除される場合があります。たとえば、タイプビット（論理削除フラグ）の削除済みフィールドまたは日時タイプ（論理削除日時）のDeletedOnをこのテーブルに追加できます。


場合によっては、ターゲットデータベースが配置されているサーバー上に、補助的な中間データベースが作成され（通常はステージングと呼ばれます）、ソースからの「生の」データが最初にロードされます。 ターゲットとステージングは​​同じサーバー上にあるため、2番目のステップは、これらの両方のデータベースのセットを操作するSQLクエリ（SQL MERGEコンストラクトまたはJOINコンストラクトを使用するクエリなど）を簡単に記述することです。


SSISは非常に興味深いツールであり、場合によっては統合プロセスを大幅に簡素化できるので、私が思うに、このツールは武器庫に持っていても害はありません。 しかし、当然ながら、すべての重量を量る状況もあります。たとえば、リンクサーバーを使用して純粋なTSQLでプロシージャを記述したり、OOPのすべての機能を使用して他のプログラミング言語でユーティリティを記述したりするなど、他の方法を使用して統合を記述する方が合理的です。



教材を勉強するときは、好奇心を見せましょう。たとえば、表示しなかったタブをクリックし、情報を調べて分析し、矢印をクリックします。独自のプロパティと設定もあります。 あなたにとって興味深いと思われるすべてのものを試してみてください。小さなテストケースを怠ることはありません。 これが例外につながるようにスキームを変更し、この状況から最も適切な方法を見つけようとするコンポーネントにより適切なパラメーターを選択します。
