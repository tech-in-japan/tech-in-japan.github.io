最初のツバメ

「リポジトリ」はOracleデータベースと呼ばれ、「ウィンドウ」を構築することになっていた。 そして「店頭」とは、分析のためのデータを意味していました。 スタッフがゆっくりと拡大しているため、私は最初のレポートを単独で作成しましたが、いつものように「昨日」に作成する必要がありました。 最初は、データベース用とSharePointポータル用のサーバーすらありませんでした。 管理者パスワードを取得した標準ユーザーコンピューターを使用しました。 この同じ管理者パスワードの操作は、規律の重大な違反でした。 ですから、銀行での仕事の最初の日から、ここでの成功は規則を破ることによってのみ達成できるという考えに妥協する必要がありました。 2番目の違反は廃止されたコンピューターであり、何らかの理由でコンピューターから持ち去られることを忘れていました。 銀行の企業ネットワークで使用するように構成されていないため、私はYotaをそれに組み込み、インターネットへのアクセスポイントとして使用しました。 一般的に、手元にある手段ではなく、プログラマーの職場を構築することができました。



私が雇われた部門がプログラミングに関与しておらず、専ら経済的な仕事に従事していたことは注目に値します。 そして、彼らは私と同僚を違法に連れ去りました。 銀行では、人がどのように問題を解決するかを尋ねるのは慣習ではありません。それは上から下に行くだけで、あなたがそれを解決した場合、うまくいけば、うまく行かなければ、トラブルが発生します（実際の不法移民のように）。 したがって、プログラマーだけが管理レポートを自動化できることに気付いた私の直接的なリーダーシップは、リスクを冒して、経済学者を装って私たちを採用し始めました。

Microsoft Reporting Servicesに適さないもの。



しかし、経営陣が好んだのは、レポートで毎日働くことになっている銀行の従業員にとって完全に満足のいくものではなかった。 Microsoft Reporting Servicesを使用する際、ユーザーは次のような不便さに気付きました。



  1. マトリックスレポートは、グループ化されたピボットテーブルのように見えます。 グループは、グループの各メンバーの[+]をクリックして展開できます。 Webフォームではこれは正常に機能しますが、大量のデータを使用してMicrosoft Excelにアップロードすると、すべてが非常に遅くなります。

  2. 一部のレポートでは、特定の値のセットを使用して、レポートのデータベースからデータを受信したクエリが非常に長い時間実行され、その結果、待機間隔を超えたときにエラーが発生しました。 間隔を長くしようとしても問題は解決しませんでした。 サーバーからデータを返すための時間制限があります。 Internet Explorerの場合、これはReceiveTimeoutです。 変更することはできますが、クライアントのマシンでのみ可能であり、そのような機会はありませんでした。
  3. クライアントのレポートのデータ量は、考えられるすべての値を超える可能性があります。 仕事の最初の週に、私はこれに出会いました。 ユーザーは、データベースから100万行をアンロードしたかったため、「突然、エラーがどこにあるのかを比較したいだけです」と言いました。 そのような量の情報をExcelにアップロードするとき、レポートはアップロードを完了できませんでした。 彼はサーバーのメモリを使い果たしていました。 サーバーは最初はかなり弱く、そのようなエラーが状況を停止させました。
  4. MSレポートは、サーバー上のレポートで使用されるクエリを保存します。 しかし、ストレージが配置されているOracleデータベースに保存することを望みました。 同じ問題は、Microsoft SharePointに保存されているレポートへのアクセスのチェックに関するものでした。 このデータをソースシステムからOracleデータベースリポジトリに毎日コピーするユーティリティを作成する必要がありました。



また、銀行では、ユーザーがMicrosoft Excelで最も頻繁にデータを操作することにも注意する必要があります。 それらのほとんどは、Webフォーム上のレポートを必要としません。 レポートやプレゼンテーションを作成するとき、ユーザーは何らかの方法で数字を「再生」したいと考えています。 独自の何かを追加したり、何らかの分析を行ったり、以前のレポートと比較したりします。 会計および統計データのみが標準形式で形成されました。 しかし、そのようなレポートの少数がありました;大半はそれらとさらに働くためにユーザーにデータを提供しました。

Microsoft Analysis Servicesの問題



しかし、スプーン一杯のタールがないわけではありません。 Microsoft Analysis Servicesは管理にはあまり便利ではなく、多くの問題があります。



  1. アクセスの大量配布-たとえば、ユーザーには30種類のキューブへのアクセス権が必要です。 標準的な方法でこの問題を解決する場合、各キューブへのアクセスを個別に許可する必要があります。
  2. 自動パーティション-新しいパーティションを自動的に追加する定期的な方法はありません。
  3. キューブの更新-キューブが多数あり、各パーティションに多数のパーティションがある場合、管理の複雑さが何度も増します。 Microsoft Integration Servicesは、単一のキューブの場合は便利ですが、キューブの数が増えると、利便性が失われます。
  4. ユーザーがActive Directoryから削除されると、リモートユーザーがこのキューブにアクセスできる場合、キューブプロジェクトをサーバーに展開できなくなります。 削除されたアカウントを見つけ、キューブから削除する必要があります。
  5. 変更されたプロジェクトの展開が困難。

    これらの問題を解決するために、Microsoftは、作業を容易にするユーティリティの開発に使用できるAPIを提供しています。 また、作成されたキューブを記述するプロジェクトは、すべての情報をxmlファイルに保存します。 これにより、MS Analysis Servicesの管理を容易にするソフトウェアの開発が大幅に促進されます。
