Guido van Rossumが質問に答えます

先週（ 8月19日-約Per。 ）Pythonのあらゆる側面、およびDropboxへの移行に関して、Pythonの寛大な独裁者であるGuido van Rossumに質問する機会がありました。 時間を失うことなく、グイドはあなたの質問のいくつかに答えました。



GoogleからDropboxへ



ヌルフセインから

こんにちは GoogleからDropboxに移行したきっかけは何ですか？ Googleで何をしたのですか。Dropboxで何をしているのですか？



Guido： Googleで7年間働いた後、環境を変更する準備ができていたので、Dropboxからオファーがありました。 概して、私の仕事はあまり変わっていません。 私はまだ

  • 私は、寛大な生活の独裁者としての役割に応じて、私が通常行うことに時間の50％を費やしています
  • 私はこの組織の普通のエンジニアです（マネージャーではなく、グループ（チームリーダーでもない）を管理することすらありません）
  • 私はよくコード検査を行い、アーキテクチャと設計を開発します
  • たくさんのメールを作成する
  • Pythonでコードを書いています


仕事の詳細はもちろん異なります。 実際、私はGoogleで2つのことを行いました。最初の2年間、最初のオンラインコード検査ツールMondrianに取り組みました。これはオープンソースではありませんが、 Rietveldを生み出しました。 現在、RietveldはPyhon、Go、およびChromiumプロジェクトで使用されています。 その後、Google App Engineに参加し、主にPythonに関連するさまざまなことを行いました。 私の最後の大きなプロジェクトは、データベース用の新しいPython API、NDBでした。

私はDropboxで7か月働いており、最初のプロジェクトはDropbox Datastore APIの設計でした。 皮肉なことに（これは私のせいではありません）、「データストア」という言葉もここにあります。 Dropbox DatastoreとGoogle App Engine Datastoreには類似点があります。

別の楽しい部分。 ほとんどのデザインは私が発明したものであり、Pythonで2つのプロトタイプを作成したという事実にもかかわらず、先月リリースされたSDKはJava、Objective-C、およびJavaScriptのみをサポートしています。 しかし、私はそれに取り組んでいます。 このインタビューのためだけに、プロセスはそれほど速くはなりません:)



Pythonが他の言語に存在するOOPイディオムを避けるのはなぜですか？



from i_ate_god （ このニックネームは信者の感情を怒らせます-約per。 ）

インターフェイス、抽象クラス、プライベートメンバーなど...なぜこれらのものがPythonに欠けているのですか？



Guido： 2つの理由があります：（a）本当に必要ない、（b）コンパイル段階で型チェックがないため、実装が難しい。 Pythonは実験的なプロジェクトとして始まり（当局によって承認または承認されていませんが、禁止されていません）、すぐに実用的なプロトタイプを取得したいと考えました。 したがって、特に必要のない、延期できる機会を削除しました。 また、実行時にすべての型チェックを行う必要がありました。 これにより、Pythonがサポートする機能に自然な制限が生じます。 私は、オブジェクト指向アプローチの宗教的なファンでもありませんでした。 単純な言語を取得したかったのですが、偶然により、よりオブジェクト指向になりました。

Pythonの最近のバージョンでは、あなたが求めているもののおおよその類似物が登場しています。 ただし、常に期待どおりに機能するとは限らず、不要なオーバーヘッドが発生する可能性もあります。 したがって、彼らはしばしばそれらを避けようとしますが、そのようなアプローチのファンもいます。

ebno-10dbから （ 彼のためにニックネームを発明したのは誰ですか？-約per。 ）

Pythonは少なくとも部分的には機能言語であると主張する人もいます。 私が見る限り、あなたはこれに同意しません。 map

質問：プログラミングへの関数型アプローチは原則的にあまり有用ではない、またはPythonに適合しないように思われますか？ いずれにせよ、私はあなたの意見を聞きたいです。



Guido：私は宗教的な好みに基づいて1つまたは別のアイデアを選択するのは好きではありません。デザインを選択する際に実用的であることを試みます（しかし、 あまり実用的ではありません。文の冒頭を参照してください:-)。 実際のコードの読みやすさと有用性に感謝します。 map()

嫌いでした。なぜなら、この関数はほとんど常に次の2つの目的で使用されていたからです。（a）要素の合計を計算し、（b）読み取り不可能なコードを書く したがって、組み込みのsum()

を組み込み関数からfunctools

（これは私にとって不要なあらゆる種類のダンプです:-)。

関数型プログラミング言語について考えるとき、主にHaskellのような信じられないほど強力なコンパイラーを備えた言語について考えます。 このようなコンパイラーの場合、関数型パラダイムは、並列化などの膨大な変換機能を提供するため、非常に便利です。 しかし、Pythonコンパイラには、コードの意味がわかりません。 これも便利です。 したがって、Pythonに「機能」プリミティブを追加することは理にかなっていないと思います。これらのプリミティブが関数型言語でうまく機能する理由がPython言語に合わないからです。 そしてそれらのために、コードは関数型言語に慣れていない人々にとって非常に読みにくくなります（そしてそれらのほとんどはプログラマーの中にあります）。

また、現代の関数型言語のセットは、主流の準備ができていないと思います。 正直なところ、私はHaskell以外の関数型言語をほとんど知りません。 しかし、Haskell ほど人気のない言語は、おそらく実際にはあまり適用されません。 そして、私はHaskellよりも人気のある言語を聞いたことがありません。 Haskellに関しては、これはコンパイラテクノロジであらゆる種類のアイデアをテストするのに最適な言語だと思いますが、その「純度」は常に広く採用されるようになると思います。 モナドに対処する必要性は、多くの人を怖がらせるでしょう。

（同様のコメントはScalaに関連しています。これはおそらく、1つの言語で機能指向とオブジェクト指向のパラダイムを組み合わせた最良の例です。しかし、結果として、あなたはそれを書くのに本当に賢明である必要があります。）

ブラウザーのPython？



Btrot69から

長年にわたって、Webブラウザー内で安全に実行できるサンドボックスでPythonを作成するためのさまざまな試みが行われてきました。 ほとんどの場合、人々はJavaScriptの問題を修正しようとしました。 JavaScriptが動作するようになった（CoffeeScriptのようなクールなものがある）ため、ブラウザにPythonを埋め込むのをやめることができますか？



Guido： 1995年にこれをやめました。つまり、答えはイエスです。 また、PythonをJavaScriptでコンパイルしようとしないでください。 これらの言語のセマンティクスは非常に異なります。 最後に、ほとんどのPython言語をJavaScriptで実装する必要があります。 その結果、パフォーマンスが低下します。 （CoffeeScriptの長所は、もともとJavaScriptにコンパイルされるように設計されていたことです。現在、これらの言語はどちらも同じ方法で開発されており、コンパイルプロセスが容易になっています。）



Python 3



MetalliQaZから

Python 3を使用して移行の現在の状態をどのように評価しますか？ ユーザーの観点からすると、主要な人気のあるライブラリーの移行は非常に遅れており、そのためPython 3への移行が困難になっています。私の職業上のキャリアでは、ほとんどすべてのシステムにPython 3.xがないことがよくあります。 バージョン2.7でさえ、まだまれです。 あなたの意見を聞くのは面白いです。



グイド：どこで働いているのだろうか。 Python 3への移行には時間がかかることに同意しますが、Python 2.7がシステムにない場合は、おそらく石器時代に行き詰まっています。 Googleを辞めたとき、彼らはPython 2.7への内部移行をほぼ完了しました（以前の2.4から2.6への移行は数年前に成功しました）。 このDropboxでは、クライアントとサーバーの両方がPython 2.7を使用しています。 両社はすでにPython 3について考えています。

Python 3への移行に戻ると、私は実際には大きな楽観主義者です。 最も人気のあるライブラリのほとんどは、すでにPy3kで動作するポートを持っているか、移植に関与しています。 （Python Software Foundationは、広く使用されているが、移植を準備するのに十分なコミュニティがないプロジェクトの移植に資金を提供しています）。 時間がかかりますが、トンネルの終わりに光が見えます。 数年後には、新しいコードの大部分はPython 3になります。Python2の使用を完全になくすには、さらに時間がかかります。 繰り返しますが、Windows XPはまだ生きています。
