傘の監視：ビジネス向けのペリスコープ

HPE Operations Bridgeソリューションの紹介は不要です。 多くの企業は、ITインフラストラクチャの個々のコンポーネントからのイベント情報を処理するための効果的なツールとしてそれを使用しています。 ただし、このソリューションは、インフラストラクチャ全体に拡張することができます-その範囲内のすべてのシステムからデータを収集および分析する「傘」のように。







なぜ「傘」の監視が必要なのですか？ 情報技術の開発の歴史に興味がある場合、さまざまなシステムの管理者のニーズへの対応として、さまざまな監視ツールが登場していることがわかります。 新しいタイプのITソリューションの出現により、そのステータスとパフォーマンスを監視することが必要になりました。 管理者はますます効果的な制御を必要としていたため、監視ユーティリティは常に改善されていました。 そして、企業のリーダーは、ビジネスを組織する上で情報技術の役割がどれほど重要であるかを明確に認識し始めました。 したがって、現在のITリソースの現在の状態だけでなく、オフィスの開設、製品の開発、サービスの撤回など、新しいタスクを実装するための使用方法も確認および評価できるツールが必要です。など



まず、ビジネス担当者は予測可能性を必要とします：IT資産が自由に使えるものを理解し、従業員が必要な知識をどの程度持っているか、企業が現在のテクノロジーに投資し、実際的な観点から能力を構築することの利益性を理解する必要があります。 そのような質問が現れたとき、上で説明した個別の監視ユーティリティは包括的な答えを与えるのに役立たないことが明らかになりました。 技術的な管理ユーティリティに対する追加のスクリプトおよびトップレベルのアドオンも、この問題に対処することを許可しませんでした。 したがって、開発者はユーティリティの機能を徐々に増やし、相関の可能性、情報の識別と強化、統計データの分析を追加し始めました-未知の問題を特定し、ビジネス指標に対するIT指標の影響度を確認するまで。 ただし、同時に別の世界的な問題を解決する必要がありました。それは、企業で長年使用されてきたすべての異種監視ツールを統合および統合することです。



この問題を解決する方法を模索した結果、ITインフラストラクチャの「傘型」監視システム、たとえばHPE Operations Bridgeが登場しました。これは、本質的に一連のコンポーネントです。 顧客が状況を明確に理解し、経営上の意思決定の不確実性要因を減らすことに関心がある場合、彼はすでに「傘」の監視を実装し、適切なソリューションを選択するためにかなり成熟しています。



HPE Operation Bridgeには、いくつかの重要なコンポーネントまたはモジュールが含まれています。 それらについて詳しく見ていきましょう。 監視プロセスを開始するために最も重要なことは、情報を収集することです。 したがって、まず、さまざまなソースからの情報を統合することをタスクとするコンポーネント、Operations Manager I（OMI）について説明します。 基本バージョンでは、主な目的を果たします。イベント情報を統合し、優先順位を付け、グループに自動的に配布し、責任者を任命します。 OMIには多くの追加機能があります。 それらの1つは、相関を保証する責任があります。 2つのタイプの相関がサポートされています。ストリーミング（特定のイベントの期待）とトポロジ（インフラストラクチャトポロジ全体でのイベントの分散。CMDBに基づいて動作します）。 トポロジは本質的に動的であるため、たとえば、物理機器または仮想環境がホスト間で移動されると、自動的に変化します。 さらに、企業はITインフラストラクチャの個々のセクションに対して独自の相関スキームを作成できます。同様の領域がターゲットサービスおよびリソースモデルで見つかった場合、このスキームは自動的に適用されます。関係に基づいて、一般的な相関スキームが構築されます。 これは、企業が地理的に分散した広範なブランチネットワークを持ち、それぞれが同じサービスまたは製品を提供している場合に非常に役立ちます。 スキームが1か所で変更されると、相関関係が自動的に再構築されます。



「傘」の下にあるITインフラストラクチャへの新しい要素の追加は、自動的に実行する必要があります。これにより、管理プロセスが簡素化されます。 したがって、次のHPE Operation BridgeモジュールであるMonitoring Automationは、監視および監査設定の自動化を担当します。 これにより、新しいサーバーまたは他の機器、およびITインフラストラクチャに追加された仮想環境は、サービスリソースモデルの監視境界に自動的に接続でき、その後、指定された監視ポリシーが配信されます。 したがって、日常的な監視セットアップ操作の量が削減され、企業はITリソース監視スキーム全体の全体像を把握できます。 サービスリソースモデルのサーバーロールが変更されると、不要なポリシーが破棄され、新しいポリシーが接続されます。 その結果、誤った監視メッセージの数が削減されます。



ITインフラストラクチャの監視に不可欠な要素はレポートです。 Operations Bridge Reporterモジュールがこれを担当します。 ビジネスプロセスの状態から、その運用を保証するITインフラストラクチャのすべてのコンポーネントまで、サービス全体に関するエンドツーエンドのレポートを受け取ることができます。 顧客は、好みに応じてこれらのレポートを送信するためのフォームをカスタマイズできます。 Operations Bridge Reporterの中心には、ビッグデータを処理するために特別に設計されたVertica DBMSがあります。 したがって、会社の長は、機器の作業負荷の程度を独自に判断し、売上の増加やサービスの拡大、または既存のリソースが十分にある場合にデバイスの艦隊を補充するかどうかを決定できます。 このようなレポートは、機器だけでなく、アプリケーション、ネットワーク負荷などによっても取得できます。



データの視覚化は、柔軟なレポートと同じくらい重要です。 また、特別なモジュールであるBusiness Value Dashboardも担当します。 財務、技術、その他の資料を表示できる情報パネルを作成できます。 それらはすべて、CEOであろうと普通の従業員であろうと、特定のユーザーに最適な形式で視覚化され、提供されます。 モジュールはわずか2年前にリリースされましたが、そのアプリケーションにはすでに興味深いシナリオがあります。 たとえば、風力エネルギーを専門とする欧州企業のWindParkは、HP Operation Bridgeを使用して風力発電機の動作を監視しています。 同社は、風の強さ、およびビジネスバリューダッシュボードを介して生成された電力量に関する視覚化されたデータを受け取ります。 実際、HPE Operation Bridgeは、ITリソースだけでなく、生産システムやエンジニアリングシステムの監視にも使用できます。 もちろん、エンジニアリングシステムを監視するための特別なソリューションに代わるものではありませんが、ビジネスアプリケーションのデータとその作業に関する情報を正常に比較します。



企業には、ITインフラストラクチャの運用に関連するさまざまなインシデントが発生することがあり、発生の原因を調査および特定する必要があります。 この場合、HPE Operation Bridgeの別のコンポーネントであるOperations Analyticsは、緊急事態の特定を担当し、救助に出ます。 システムで受信した情報を使用して、相互接続の観点から分析し、隠れた傾向を明らかにするため、条件付きで「タイムマシン」と呼ぶことができます。 たとえば、システムに不具合がある場合、このモジュールはログファイルを分析し、不具合の原因を示すデータを見つけ、不具合の原因となったイベントの関係に関する視覚化された情報を提供できます。 このような関数は、システムの「クラッシュ」の根本原因を特定できるため、インシデントの解析に非常に役立ちます。 これを行うには、情報パネルの「スライダー」をフォールトチェーンの最初に戻します。 Operations AnalyticsモジュールはVerticaを使用しているため、ビッグデータと連携できます。



ご存知のように、システム管理者は、担当するITリソースの状態を定期的に確認する必要があります。 Operations Orchestrationモジュールを使用すると、情報システムをチェックするための日常的な操作を自動化でき、手動検証または自己記述スクリプトを完全に置き換えることができます。



リストされている基本モジュールに加えて、HPE Operation Bridgeには、容量管理、ネットワーク監視、およびエージェントまたはコレクターを使用したデータ収集の編成を担当するいくつかの追加アドオンが含まれています。 後者は、顧客がHewlett Packard Enterpriseソリューションのみを使用して、利用可能なすべてのITリソースを監視する場合に必要です。 下位レベルにサードパーティメーカー（Zabbix、IBM Tivoliなど）からのIT監視システムがあり、HPE Operation Bridgeが「傘」として使用される場合、コレクターのインストールは不要です。 実際、HPE Operation Bridgeは、少なくともトポロジ、イベント、および外部システムからのパフォーマンスメトリックを受け取ります。



HPE Operation Bridgeをゼロから展開する場合、最良のオプションは、データ収集と分析の両方を提供する包括的なソリューションを使用することです。 将来的には、これにより、会社はトレーニングを含めて大幅に節約し、リスクを減らすことができます。 ビジネスが発展するにつれて、このようなソリューションを増やすことができます。 より複雑なのは、ITリソースを監視するための個別のユーティリティがすでに実装されている組織に同様のシステムを展開するタスクです。 従業員は特定の習慣を身につけているため、利害、あらゆる種類の恐怖、または他の解決策への転換を望まないためにロビー活動に対処する必要があります。 そのような場合、「上から」注文するのではなく、各従業員が新しいテクノロジーに切り替えるように動機づけることによって成功を達成できます。 この点で、古いツールと新しいツールの機能と利便性を比較すると便利です。



以前は、ビジネスリーダーには「トラフィックライト」、つまり可能な限り単純なソリューションが必要であるとよく言われていました。 これは完全に真実ではありません。 トップマネージャーは、問題が発生した場所、責任者、既に修正に取り組んでいるかどうか、サービスまたはビジネスプロセスがいつ復元されるかなど、いくつかのことを知る必要があります。 これらすべてのタスクは、「傘」の監視を解決するのに役立ちます。
