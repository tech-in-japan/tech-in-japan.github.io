ASP.NET MVC 2：エリア-説明、使用

ASP.NET MVCの領域の概念は新しい概念です。 エリアの本質は、非常に大きなプロジェクトを、相互にやり取りし続けるいくつかのグループに分割する機会を提供することです。 たとえば、会社には、メインサイト、ストア、製品レビュー、フォーラム、内部管理サイトなどを含む大規模なプロジェクトを作成するいくつかの開発チームがあります。 ASP.NET MVC 2でこのようなプロジェクトを作成するには、プロジェクトの個々の部分を、1つの相互接続されたプロジェクトを構成し続けるエリアに割り当てることを提案します。 個々のプロジェクト領域を大規模プロジェクトから分離することにより、開発者のさまざまなグループがアプリケーションの各部分をより独立して開発できるようになります。



エリアの概念は、マルチプロジェクトソリューションのエリアとしてASP.NET MVC 2 Preview 1に登場しました。 ASP.NET MVC 2 Preview 2のリリース後、エリアのアプリケーションは、1つのプロジェクトでエリアを作成できるように拡張されました。 両方のオプションを検討してください。



いくつかのプロジェクトのエリア



複数のプロジェクトで動作する領域を使用してソリューションを作成する場合、次の手順に従う必要があります。

  • 保護区域に関連して親（またはメイン）になるプロジェクトを作成します。
  • たとえば、フォーラム、ブログ、ストアなどのエリアプロジェクトを作成します。 これらのプロジェクトでは、 Global.asaxファイルを削除する必要があります。さらに、すべてのスクリプトは親プロジェクトから取得されるため、 scriptsフォルダーで作成されたデフォルトファイルを削除できます。
  • 親プロジェクトのプロジェクトエリアへのリンクを追加します。
  • 各プロジェクトに対して、 AreaRegistrationクラスから継承するクラスを作成します。 このクラスでは、 RegisterAreaメソッドとAreaNameプロパティの2つの要素がオーバーロードされます。 RegisterAreaは、アプリケーションコンテキストで領域を登録するためのロジックを実装する必要があり、AreaNameプロパティは領域の名前を返す必要があります。
  • Global.asaxの親プロジェクトでは、 AreaRegistration.RegisterAllAreas（）メソッドを呼び出す必要があります。 起動時にアプリケーションコンテキストに領域を登録するため。
  • エリアを含むプロジェクトを正常に起動するために必要な最後の手順は、各プロジェクトの* .csprojプロジェクトファイル（Visual Basicプロジェクトの場合は* .vbproj ）を変更することです。 そのような各ファイルでは、エリアに関連するコード行のコメントを外す必要があります（以下に表示されます）。
手順をもう少し詳しく検討してください。 図 図 1は、親AreasProjectプロジェクトと、それぞれフォーラムとストアのロジックを実装する2つのプロジェクトエリア- フォーラムとストアの 3つのプロジェクトを含むソリューションの構造を示しています。

図 1 StoreおよびForum プロジェクトでRoutes.csファイルを確認できます。 このファイルは、AreaRegistrationクラスを実装するためにプロジェクトに追加されました。 リスト1のStoreプロジェクトのRoutes.csファイルの内容を検討してください。



リスト1. StoreプロジェクトのRoutes.csファイルの内容

namespace Store

{

using System.Web.Mvc;

using System.Web.Routing;

public class Routes : AreaRegistration

{

public override void RegisterArea(AreaRegistrationContext context)

{

context.MapRoute(

"Store_Default" ,

"Store/{controller}/{action}/{id}" ,

new {controller = "Products" , action = "List" , id = "" },

new string [] { "Store.Controllers" }

);

}

public override string AreaName

{

get { return "Store" ; }

}

}

}







RoutesクラスはAreaRegistrationを継承します。これにより、ASP.NET MVCサブシステムはそれを使用してエリアサブシステムをさらに初期化できます。 Routesクラスには、RegisterAreaメソッドとAreaNameプロパティの2つのオーバーロードされたメンバーが含まれています。



RegisterAreaは、アプリケーションのコンテキストでルートをエリアに登録します。 context.MapRouteを使用して、 Store_Defaultというルートが作成されます。 パラメーターnew string [] {"Store.Controllers"}の最後の値は、このルートに関連付けられたコントローラーが配置される名前空間を定義します。



オーバーロードされたAreaNameプロパティは、エリアの名前を返します。 この名前は、ASP.NET MVCと開発者の両方がさらに使用して、必要な機能を呼び出す必要がある領域を示します。



ストアプロジェクトのルートなどのクラスは、エリアを含むソリューション内のプロジェクトごとに作成する必要があります。



次の手順は、Global.asaxのAreasProject親プロジェクトに次のコード行を追加することです。

protected void Application_Start()

{

AreaRegistration.RegisterAllAreas();

RegisterRoutes(RouteTable.Routes);

}







アプリケーションの起動時にAreaRegistration.RegisterAllAreas（）を呼び出すと、ASP.NET MVCがアプリケーションで使用可能なすべての領域とそのルートを初期化できます。



エリアを含むプロジェクトを開始するために必要な最後の手順は、ソリューション内の各プロジェクトの* .csprojプロジェクトファイル（またはプロジェクトがVisual Basicで記述されている場合は* .vbproj）を変更することです。 これらのファイルにアクセスするには、 UnloadProjectコマンドを使用して、ソリューションからプロジェクトを順番にアンロードします。 ダウンロードしたプロジェクトのコンテキストメニューで、[ 編集] ***。Csproj （図2）を選択します（***はプロジェクトの名前）。

1つのプロジェクト内のエリア



複数のプロジェクトの領域の前の例のソリューションに、親プロジェクトの非表示ファイルを表示するオプションを含めると、次の図が表示されます（図4）。





図 4.非表示の決定要素



親AreasProjectプロジェクトでは、エリアプロジェクトの非表示アイテムがビュー/エリアフォルダーに作成されます。 これは、親プロジェクトがプロジェクト領域の表現を独立して表示するために必要です。



ただし、この隠しファイルの構造は直接使用できます。 ASP.NET MVC 2 Preview 2では、エリアに基づいた単一のプロジェクトでソリューションを作成する機能が導入されています。 このようなソリューションには、すべての領域が実装されているプロジェクトが1つだけ含まれています。 図 図5は、1つのプロジェクトの構造を示しています。ここでは、前のセクションでいくつかのプロジェクト領域を使用して実装されたのと同じ領域が実装されています。





図5 エリアに基づく単一プロジェクトの構造。



このようなプロジェクトを作成するには、前の例と同様に、複数のプロジェクトの代わりに、1つのプロジェクトに地域の階層を作成する必要があります。 これを行うには、プロジェクトルートでAreasフォルダーを作成する必要があります。作成された各フォルダーは個別のエリアになります。 図 5そのようなフォルダーは、フォーラムとストアです。 それらで、コントローラーとビュー用のコントローラーとビューのフォルダーを作成し、前のセクションで行ったのと同じ方法でそれらを実装する必要があります。 リージョンフォルダーごとに、AreaRegistrationクラスの実装を使用してRoutes.csファイルを作成する必要があることに注意してください。 Global.asaxは、AreaRegistration.RegisterAllAreas（）呼び出しを追加して更新する必要があります。 その後、いくつかのプロジェクト領域を持つソリューションに似たプロジェクトを受け取ります。



エリアに基づいて単一のプロジェクトを開発することと、いくつかのプロジェクトエリアを持つソリューションを開発することの素晴らしい違いは、エリアに基づいて単一のASP.NET MVCプロジェクトを作成する場合、プロジェクト間の依存関係を処理する必要がないため、プロジェクトファイルを変更する必要がないことです。 一般に、これにより、エリアを使用したソリューションの作成が多少簡単になります。



機能領域は、さまざまな部分のロジックを複雑なオブジェクト（大規模ソリューションの領域）に分割できる便利なメカニズムです。 ソリューションでの領域の表現方法を選択できます。複数のプロジェクトに基づいて、または1つのプロジェクトのみに基づいて。 いずれにしても、エリアの作成は難しくなく、アプリケーション内の異種機能を簡単に分離できます。
