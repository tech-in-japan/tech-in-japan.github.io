Veeam SureBackupについて









バックアップを確認する必要があります。



導入として、荒れ狂う若者からの簡単な物語。 誰もがリソースがない状況に慣れていますが、バックアップを保存する必要があります。 かつて、システムのバックアップを保存するために2つの500GBディスクが使用されていました。 ご想像のとおり、RAID-1を使用する場合、使用可能なスペースは非常に不足していた500 GBに制限されていました。 Linux LVMを使用することで、信頼性を損なう1000GBのスペースを確保することになりました。



1年以上、あるいはそれ以上の期間にわたって、このリソースにバックアップが追加され、スクリプトはすべてが正常であると報告し、ディレクトリの内容を表示する際にバックアップのリストは目に優しいものでした。



時間が来て、回復する必要がありました。バックアップの1つが破損したとき、すべてがバックアップが行われ、保存され、一般にすべてが正常であると言っていたため、驚きがありました。 後で判明したように、LVMのディスクの1つが崩れ始め、データの一部が復元され、他の部分はなくなっていました。 当時のように絶対にしないでください。



現在、仮想化を使用しない企業はほとんどなく、仮想マシンのバックアップを行わない企業がわずかにあり、バックアップをチェックしない企業がさらに多くあります。



Veeamがバックアップをチェックする方法に興味がある人のために、猫をお願いします。



仮想インフラストラクチャでVeeamバックアップシステムを既に使用しており、毎日すべてのバックアップが完了したという通知をメールで受信するとしますが、残念ながら、これはこれらのバックアップを使用できるという意味ではありません。 実際、これには多くの理由があります。



  • 実例からわかるように、ディスクの問題はディスク上のバックアップに影響を与える可能性があります（私にとっては最悪のケースです）。
  • 一貫性のないバックアップを使用すると、オペレーティングシステムが起動しない場合があります。
  • システムにあるアプリケーションは、システムを復元した後、気分が悪くなる場合があります。


等



Veeam Backup＆Replicationには、SureBackupと呼ばれる興味深いソリューションがあります。これにより、システム内にあるいくつかのアプリケーションと同様に、バックアップを確認できます。 この技術に慣れていない人のために、あなたはここを読むことができます 。 この機会を得るには、EnterpriseまたはEnterprise Plus ライセンスが必要です。



SureBackupアルゴリズムは非常に簡単です。



  1. 分離されたラボは、仮想化クラスター内の任意のホスト上に作成されます。
  2. vPower NFSを使用して、バックアップからの仮想マシンがこのラボで起動されます。
  3. 一連の自動テストが実行されます。
  4. テスト復旧レポートが送信されます。


VPower NFSテクノロジーを使用すると、ハイパーバイザー上の仮想マシンをバックアップファイルから直接実行できます。



SureBackupが実行できるテストのリスト：



  1. 仮想マシンの起動の確認。
  2. Heartbeatオペレーティングシステムの確認（ゲストオペレーティングシステムにVMwareツールが必要）。
  3. 仮想マシンへのpingを確認してください。
  4. スクリプトを実行してシステム内のアプリケーションを確認します（ゲストOSにアクセスするにはアカウントが必要です）。


そして今、それをすべて設定する方法について。

すべてのアドレスは架空のものであり、一致するものはランダムです。



SureBackupはVeeam BRコンソールから直接設定されます。最初に行うことは、バーチャルラボの準備です。 [ バックアップインフラストラクチャ ] → [ SureBackup ] → [ 仮想ラボ ]タブで見つけることができます。

[仮想ラボの追加]をクリックすると、まず何とかしてラボに名前を付け、その説明を示すように求められます。









次に、仮想マシンを起動してテストするホストを指定する必要があります。 本稼働クラスターとは別のホストを使用したい：









次のステップは、ハイパーバイザーに接続されたストレージの1つを選択することです。これは、Veeamが回復に必要な一時ファイルと、仮想ラボでネットワークのテストと構成に必要な仮想マシンをホストします。









ネットワーク経由でテスト仮想マシンと対話するには、プロキシアプライアンスのインストールが必要です。 この機能の使用を拒否できますが、この場合、ネットワークテストなどのテスト機能の多くは、ネットワークを介した仮想マシンへのアクセスが利用できなくなります。 プロキシアプライアンスは、稼働中のネットワーク内の1つのインターフェイスと、構成に応じて残りのインターフェイスを分離して監視する仮想マシンです。 この仮想マシンは、仮想ラボを作成すると自動的に展開されます。システムをインストールしてルーティングを手動で構成する必要はありません。









プロキシアプライアンスを正しく構成するには、この仮想マシンをBackup＆Replicationサーバーで使用できるネットワーク設定を指定する必要があります。



以下を指定する必要があります。



このアプライアンスが使用する仮想ハイパーバイザースイッチからのネットワーク。

このアプライアンスをBRサーバーで使用できるIP設定。















次の手順としてプロキシアプライアンスの設定を指定したら、仮想ラボ用に分離されたネットワークを構成する必要があります。 3つのオプションがあります。



  1. 基本的な単一ホスト -ネットワーク設定の自動構成。 このオプションでは、アプライアンス設定で指定されたネットワークと同様に、1つの隔離されたネットワークのみが作成されます。



  2. 高度な単一ホスト -分離されたネットワークの手動構成。 大多数は仮想マシンに1つのネットワークではなく、VLANに分割された複数のネットワークをそれぞれ使用して、異なるネットワーク設定を持つ仮想マシンのネットワークを正しくテストするには、いくつかのネットワークを作成する必要があります



  3. 高度なマルチホスト -レプリカのテストなどのために、仮想ラボを複数のホストに分割する必要がある場合に使用します。 VMware Distributed Switch（VDS）の機能を使用します。








異なるネットワーク上にある仮想マシンをテストする必要があるため、高度な単一ホストオプションを使用します。



（隔離されたネットワーク）タブで、システムはプロキシアプライアンスが配置されているネットワークに似た1つの隔離されたネットワークを自動的に作成します（バックアップから実験室で実行されている仮想マシンは隔離されたネットワークに接続され、本番環境にブロードキャストされません）。



私の場合、これはV39ネットワークとVLAN ID 39です。









別のVLANから仮想マシン用の別のネットワークを追加する必要があります。 [追加]をクリックすると、vSwitchネットワークのいずれかに対応する実稼働ネットワークを選択し、それを仮想ラボの分離ネットワークにマップする必要があるウィンドウが表示されます。 私の場合、仮想マシンがバックアップから使用するV30ネットワークを追加します。















その結果、次のルールが形成されます：vSwitchでV30ネットワークを使用するネットワークインターフェイスを持つ仮想マシンは、仮想ラボのtest-vLab1 V30ネットワークに接続され、V39ネットワークで仮想マシンはtest-vLab1 V39ネットワークにそれぞれ接続されます。









次のステップはおそらく最も興味深いものです。 プロキシアプライアンスの仮想マシンを使用して、外部から仮想マシンにアクセスできます。これは、プロキシがネットワークと、テストマシンが配置されている分離されたネットワークとの間のルーターとして機能するためです。 バーチャルラボ内にあるマシンの実際のアドレスを偽装することでアクセスが提供されますが、マシン自体は実際のアドレスを使用して隔離されたネットワーク内で動作します。



デフォルトでは、プロキシアプライアンスは最初の分離されたネットワークに対して1つのネットワークインターフェイスのみを作成します。 2つのネットワークV30およびV39に2つのインターフェイスを作成します。

次のステップは、テストグループを作成することです。これは、テスト中に起動される仮想マシンのグループです。 たとえば、DNS、ドメインコントローラー、メールサーバー。 私の例では、すべてがシンプルで、サービスのない2つの独立した仮想マシンです。



Veeam BRで、[アプリケーショングループ]タブに移動し、[アプリケーショングループの追加]を選択します。 グループの名前を示します。









次に、テスト用の仮想マシンを選択する必要があります。 SureBackupがマシンを起動するのはその順序であるため、マシンが追加される順序は重要です。 最初にメールサーバーを起動し、次にDNSを起動することは非論理的です。 マシンを追加するには、[VMの追加]をクリックします。 バックアップファイル、レプリカ、またはストレージスナップショットからマシンを追加できます。









公平には、Move UpおよびMove Downボタンを使用してマシンの順序を変更できることに注意してください。



マシンを選択して[編集]ボタンをクリックすると、このマシンで実行されるテストのリストを指定できます。 これらは、ロール、スクリプト、および起動パラメーターです。 私の場合、マシンが起動してpingに応答することを確認したいので、[スタートアップオプション]タブにのみ興味があります。









関心のある分野：



  1. 最大許容起動時間-SureBackupがシステムの起動を待機する時間。
  2. VMハートビートが存在する-仮想マシンからのハートビートがチェックされます。
  3. VMは任意のネットワークインターフェースでpingに応答します-仮想マシンはネットワークインターフェースを介してpingに応答します。

SureBackupのセットアップと操作はこれですべてです。



vPower NFSについてもう少し：



  1. vPowerNFSは、主にインスタントVMリカバリやSureBackupなどのタスクで使用されます。



  2. vPowerNFSはNFSストレージをハイパーバイザーに接続し、作業の終了時にそれを切断しません。 これは、次回に再度マウントして時間を無駄にする必要がないように行われます。 NFSデータストアを無効にすると、次回に再度マウントされます。



  3. vPowerNFSは、バックアップファイルから仮想マシンを提示します。 マシンのディスクサブシステムで発生したすべての変更は、vPowerNFSサーバーまたは特別に選択されたデータストアに保存されているREDOログに転送され、バックアップファイルは変更されません。 REDOログがvPowerNFSサーバーに保存されている場合、このサーバーの十分なディスク容量に注意してください。
