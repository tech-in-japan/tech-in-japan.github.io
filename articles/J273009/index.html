OttoでDrupal 8をデプロイする

この投稿は、ハットコープの製品に関するレビュー記事がすでに掲載されている最後のHashicorp製品の1つであるオットーと知り合った結果として判明しました。



私の投稿はより実用的です。 設定の複雑さにすぐにつまずかないように、標準プロジェクトをオットーに上げることにしました。そして、それを面白くするために、最近リリースされたDrupal 8を選びました。







設置



でタグ付けされた最新バージョンは、2015年10月20日付けの0.1.2です。 今後は、すぐにMySQLを構成できなかったと言います。CHANGELOGを見て、Layered Dev Environmentsのためにmasterブランチを試すことにしました。



Otto 0.1.3-devにはVagrant 1.7.99+が必要です。 安定バージョンは1.7.4であるため、これはマスターブランチからVagrantも必要であることを意味します。 UPD：Vagrant 1.8がリリースされたため、Vagrantをビルドする必要がなくなりました。



両方のプロジェクトのコンパイルについては、ottoとvagrantのページで詳しく説明されています。GoとRubyの知識がなくても、golangのインストール、gopathの構成、両方のプロジェクトのコンパイルに30分かかりました。

主なことは、追加することを忘れないことです 

export PATH="$GOPATH/bin:$PATH"


ターミナルを再起動し、dev-versions： 'vagrant version'、 'otto version'を実行していることを確認します。



NFSセットアップ
オットーには直接当てはまりませんが、標準的なvirualbox共有フォルダーのひどいスローダウンについて知ったとき、NFSを構成する必要があることに気付きました。



Ubuntuでは、nfs-common nfs-kernel-serverパッケージをインストールし、/ etc / sudoersに数行を追加するだけで十分です。Vagrantは起動するたびにパスワードを要求しません。詳細については、ドキュメントを参照してください。





プロジェクトの準備



Drupal 8をダウンロードします。 

 git clone https://github.com/drupal/drupal.git drupal8 cd drupal8


Appfile



Appfile-プロジェクトのメインのotto設定ファイルは、プロジェクトのルートにある必要があります。



一般に、まったく存在しない場合は、ottoがアプリケーションを実行できる環境の要件を決定します。 アプリケーションのタイプを決定するロジックはどこにも簡単ではありません。たとえば、ルートに* .phpファイルがある場合、プロジェクトはphpとして定義されます。



オットーコンパイル



ottoが必要とする最初のコマンドは ' otto compile 'です。 プロジェクトとそのAppfileを分析し、その後、開発環境と展開環境のすべての構成情報を含むツリーを.ottoフォルダーに生成します。



ドキュメントでは、このフォルダーの内容を自分の手で支配しないことを推奨しています。 しかし、あなたは見ることができ、見るべきです。



.Ottoidも生成されます。これは、アプリケーションの一意の識別子です。これにより、どのottoがその展開を追跡し、履歴を保存し、依存関係として使用します。



この時点で、環境が記述され、デプロイの準備ができました。



otto dev



次に行うことは、ローカルマシンにローカル開発環境を展開することです。

実際、otto devはvagrant upと同じです。



階層化された開発環境


レイヤーの概念は興味深いです。レイヤーはDockerレイヤーに非常に似ています。「otto dev」を初めて実行すると、Vagrant仮想環境が順番に設定され、各ステップの後に写真が撮られます。 レイヤーは、アプリケーションのタイプで説明されています。 これにより、開発環境を再作成する時間を大幅に短縮できます。 したがって、仮想マシンが完全に再作成された場合でも、完成した環境を即座に起動できます。 マシンの再起動と再作成の違い（「otto dev halt && time otto dev」と「otto dev destroy && time otto dev」）は約5秒です。



changelogから判断すると、vagrantは少なくとも次のバージョンではこの機能を受け取りませんが、Vagrantにはスナップショットを作成する機能があります。



「otto dev layers」コマンドでレイヤーを表示できます。 

 $ otto dev layers consul php5.6


その結果、PHPアプリケーションの場合、ottoはさらに指示を発行する必要があります。 

 $ otto dev IP address: 100.66.143.21 A development environment has been created for writing a PHP app. You can access the environment from this machine using the IP address above. For example, if you start your app with 'php -S 0.0.0.0:5000', then you can access it using the above IP at port 5000.




自動チューニングで何が得られましたか？

  • Ubuntu 12.04（現時点では、システムのベースイメージを選択できません）
  • 領事
  • PHP 5.6（オットーはPHPを標準リポジトリからではなく、PPAからインストールします）
  • 作曲家


MySQLはDrupalが機能するために非常に不足しています。



依存関係



MySQLを追加する最も簡単な方法は、依存関係を指定することです。 プロジェクトのルートにAppfileを作成します。 

 application { name = "drupal8" type = "php" dependency { source = "github.com/hashicorp/otto/examples/mysql" } }


この構成は、アプリケーションのタイプ（php）を明示的に示し、MySQLを依存関係として追加します。

ソース内のリンクが無効であるという事実にもかかわらず、オットーはそれを理解するでしょう、 他のソース形式があります 。



mysqlをさらに簡単に追加する方法はありますが、間違っています。それについていくつか説明します。



mysql appfileは、依存アプリケーションをDockerコンテナー、コンテナーイメージへのパス、およびスタートアップパラメーターとして記述します。 結果として、Vagrant内で実行中のコンテナーを取得します。Consulのおかげで、mysql.service.consul（仮想マシン内）でユーザーとroot：rootパスワードで利用できるようになります。 デフォルトのConsulを使用すると、Dockerの使用がはるかに簡単になります。サービス検出の設定の複雑さを理解することはできませんが、使用するだけです。



他のアプリケーションは依存関係になる可能性がありますが、アプリケーションの名前が変更されても、ottoが実行中の内容を理解できる.ottoidファイルを持つことを強くお勧めします。



依存関係は現在、バージョン管理をサポートしていません（将来追加することを約束します）。つまり、すべてのottoコンパイルで、期待したものを取得できない可能性があるため、自分で依存関係をフォークするか、ローカルマシンに配置する必要があります。



「ottoコンパイル」、「otto dev destroy」、「otto dev」を実行します。Drupalを起動するマシンを準備します。 その中でサーバーを実行するために残ります。



otto dev ssh



SSH経由で仮想マシンに入ります。 上記のコマンドを実行して、PHPの組み込みWebサーバーを起動します。devのポート80でrootとして実行しても何も問題はありません。 

 sudo php -S 0.0.0.0:80


その後、「otto dev」を発行したブラウザーでIPアドレスを開いて、「otto dev address」を実行することで明示的に取得できます。



欠点



生コード


一部の場所では、コードはかなりプロトタイプであることがわかります。 たとえば、Vagrantfileは、bashを使用して環境をデプロイします。これは、ansibleやそのようなものではありません。



私はアプリタイプのPHPコードを見て、起動されているアプリケーションがワードプレスであるという定義が見つかりました（wp-config.phpファイルの存在による）、標準のPHPアプリケーションとの唯一の違いはmysqlの自動追加です つまり、Appfileで「type = "wordpress"」を指定するか、「touch wp-config.php」を作成してmysqlを取得できるようになりましたが、もちろん、これを行うべきではありません。



環境は設計されていません


プロジェクトの課題追跡ツールから判断すると、ベストプラクティスはまだ準備ができておらず、どのように準備するかが完全には明確ではありません。 たとえば、PHPの場合、PHPプロジェクトの環境を知らないHashicorpと、Goを知らず、プルリクエストを送信できないPHPプログラマがいます。



Ottoの作者であるMitchell Hashimotoは、Githubに改善点、正しい環境を備えた記事へのリンクなどの提案を書くことに興味のあるすべての人を招待します。Hashicorpは製品への統合を担当します。 後の抽象化が現れると思われます。これにより、さまざまな非囲programmerプログラマが製品の開発を支援できるようになります。



Devはprodとは異なります


「オットーインフラ」および「オットーデプロイ」コマンドには到達しませんでしたが、AWSのみをサポートするようになり、PHPプロジェクトの場合、開発環境はデプロイ環境とは異なることがわかりました。 apache2はmod-phpで上昇します。



開発者がひどくカスタマイズされている


Ottoでは現在、Appfileを使用してVagrantfile開発環境をカスタマイズすることはできません。 .otto / compile / app / dev / Vagrantfileを見ると、NFSと互換性のないマウントオプション/ vagrantが書き込まれていることがわかります。 次のような行を修正します。 

 config.vm.synced_folder '/home/popstas/projects/site/drupal8', "/vagrant", type: "nfs"


次の「オットーコンパイル」はこれらの変更を上書きします。



それを正しく行う唯一の方法は、application type = "custom"を使用してVagrantfileを完全にオーバーライドすることですが、この場合、環境を自動調整する魔法はすべてなくなります。



結論



私は、Ottoをまだ完全に使用する準備ができていないことを自分で結論付けました。これは、製品のバージョン0.1で一般的に示唆されています。 それにもかかわらず、この製品は非常に興味深いものであり、Vagrantに比べてすでに利点があります。すべての欠点はOttoの新規性によって説明され、開発者はそれらについて知っており、削除の計画を持っています。



レイヤーがOttoに完全に表示されるときに、独自のレイヤーを作成する方法が明確ではありません。私にとっては、Vagrantからレイヤーに切り替える十分な理由になります。
