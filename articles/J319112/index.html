VZ7対VZ6：更新する理由はありますか？

過去1年で、当社の主要製品であるVirtuozzo仮想化システムの新しいバージョンがリリースされました。 それ以来、「更新する価値はありますか？」、「7ka 6kaよりも優れているものは？」などの質問が常に寄せられています。 そのため、休暇中にiにドットを付けたいという要望があり、ある投稿でVirtuozzoの最新バージョンと以前のバージョンとの違いについて話しました。











この投稿は、Virtuozzo 6およびそれ以前のバージョンのユーザー、およびオープンプロジェクトOpenVZ（新しいバージョンについてはすでに書いた ）のユーザー向けに準備されたことにすぐに注目したい。 もちろん、これは、Virtuozzoをまだ使用していない人に読むことが不可能または無関心であることを意味するものではありません...ここで、最新バージョンの違いについて客観的に話して、興味のあるhabrochaneが更新が関連性があるか、まだ意味がないかを判断できるようにします。



プラットフォームについて



Virtuozzoプラットフォーム自体がより広範囲になりました。 現在、この製品には、「軽量」仮想マシン、仮想マシンを管理するためのKVMベースのハイパーバイザー、およびソフトウェア定義ストレージを実行できる独自のシステムコンテナーソリューションが含まれています。 利点は、これらすべてが単一の仮想化センターとして機能するため、「軽量」仮想マシンと本格的なVMを作成し、単一のVirtuozzo Storageデータウェアハウスのリソースを割り当て、サーバーにインストールされたハードドライブとソリッドステートドライブに基づいて作成できることです。



ハイパーバイザー



オープンプラットフォームと商用プラットフォームの両方（現在同期的に開発され、同じソースコードを使用）で、KVMハイパーバイザーに切り替えました。 その理由は非常に単純です。近年、KVMは非常に大きな進歩を遂げており、業界の発展のペースで独自のハイパーバイザーに追いついていないことがわかりました。 KVMプロジェクトに参加することがより有望であり、オープンな仮想化システムの開発に貢献するとともに、VZ7で使用するために必要な機能でシステムを補完することがわかりました。 200を超えるパッチと改善が実装されており、KVMを名手により効率的なソリューションにしています。







より安全なコンテナ



ユーザーの要望に応じて、コンテナの多くの革新も実装しました。 Virtuozzoが新しいLinux 3.10+カーネル（RHEL 7）に移行したため、cgroupと名前空間のサポートが登場しました。これにより、カーネルモジュールと比較してより高いレベルの分離が実現します。 通常のコンテナIDの代わりに、Virtuozzo 7は単一のUUIDを使用します。



ライブ移行



Virtuozzoの新しいバージョンは、 CRIUの完全なサポートも実装しました。 これは、コンテナをフリーズし、単純なファイルセットとしてディスクに保存できるLinux用のソフトウェアツールです。 その結果、同じポイントから復元することができ、適切なアクションを実行すると、アプリケーションは作業環境が変更されたことにも気付かないでしょう。 CRIUとP.Haulプロジェクトは、オープンテクノロジーに基づいたコンテナのオープンマイグレーションのための新しいエコシステムの作成を支援します（CRIUは完全にオープンソースのプロジェクトです）。これは、Virtuozzo 6と比較して移行の問題が大きく飛躍します。一部はユーザー空間に実装されており、カーネルの変更はほとんど必要ありません。 これにより、ツール自体の開発では、カーネルとその調整のための永続的なパッチのリリース、およびシステムの更新時の再起動が不要になります。これは、産業環境の重要な問題です。



セキュリティ更新-停止サービスなし



新しいVirtuozzo 7を使用する際の損失を減らすのに役立つもう1つの興味深い機能は、ReadyKernelです。 Virtuozzo Linuxオペレーティングシステムの新しいバージョンは、サーバーを再起動せずにセキュリティアップデートをインストールする機能をサポートしています。 一方で、これにより、パッチをすぐにインストールして、システムのハッキングや侵害のリスクを軽減できます。また、一方で、再起動によるサービスの停止によってお金を失うこともありません。



Virtuozzo 6ユーザーは、再起動せずにカーネル更新をインストールするためのRebootless Kernel Update（RKU）システムにすでに慣れています。 しかし、実際のところ、RKUはまだそれほどうまく機能していませんでした。コンテナを停止し、新しいカーネルをロードしてから、コンテナまたはVMを起動しました。 はい、このメカニズムによりほぼすべてのコアをロードできましたが、それでもある程度のダウンタイムが発生しました。 Virtuozzo 7では、3.10 +カーネルを使用してkpatchテクノロジーに切り替え、すべてがさらにクールになりました。



専門家によると、ReadyKernelを使用すると、10,000台のサーバーごとに1か月あたり最大200時間の管理者作業（http://www.iksmedia.ru/news/5336522-Virtuozzo-Linux-7-ekonomit-zakazchi.html）を節約できます。



バックアップと復元



Virtuozzo 7のバックアップツールも大幅に再設計されました。 このために、新しいAPIが開発され、CBT（変更ブロック追跡）メカニズムが実装されました。 これを行うために、標準のQEMU / KVMメカニズムを使用して、スナップショットを作成するシステムを作成し、コンテナーと仮想マシンのイメージをQCOW形式で保存しました。 Virtuozzo7。これにより、累積バックアップの作成がVirtuozzo 6よりもはるかに高速になり始めました。



さらに、緊急復旧が可能になりました！ ある時点でバックアップシステムが機能しなくなった場合でも、qemu-imgユーティリティを使用してQCOW2ファイルを任意のイメージに変換できます。



メモリ管理



OpenVZ 7とVirtuozzo 7の両方に実装されている新しいメモリ管理技術について、すでにいくつかの言葉を述べました。この新規性は、コンテナ、VM、ストレージシステムの両方を対象としています。 高度なメモリ管理（AMM）は、KSMを含むすべてのメモリ作業と、さまざまなWSSセットのメモリ予測テクノロジを監視します。 これにより、ユーザーエクスペリエンスへの影響を最小限に抑えて、自動的にバランスを取ることができます。 AMMは、WindowsとLinuxの両方のゲストシステムをサポートし、オンラインメモリ管理（OMM）ツールは、コンテナーに適切な量のメモリの可用性を保証します。また、再起動せずにメモリを増減できます。これは、サービスの継続性を保証するもう1つの方法です。



OpenStackサポート



Virtuozzo 7以降では、Libvirtを介したコンテナとVMの操作をサポートしています。 これにより、Virtuozzoはlibvirtを使用して大規模なエコシステム、つまりOpenStackとVirtual Machine Managerで動作できます。 つまり、OpenStackユーザーは、OpenStack APIまたはHorizo​​nパネルを使用して、コンテナとVirtuozzo VMを使用してプライベートクラウドまたはパブリッククラウドを管理するOpenStackの可能性を最大限に引き出すことができます。 新しいVirtuozzoストレージモジュールは、OpenStack用のソフトウェア定義ストレージの作成にも役立ちます



アプリケーションカタログ



今日お話しする最後のことは、既製の構成済みのアプリケーションをすべてのユーザーに即座にインストールできることです。 このようなディレクトリを自分で維持し、更新を追跡することは非常に困難です。 そのため、新しいバージョンでは、Virtuozzo Application Catalogの独自の既製スイートを作成しました。 Bitnamiと共に、WordPress、Redmine、SugarCRM、Alfresco、Drupal、MediaWiki、GitLabなど、仮想マシンまたはコンテナーとして展開する準備ができている常に最新のアプリケーションおよび開発環境のセットをサポートしています。







おわりに



今日は、Virtuozzo 7と以前のバージョンの一般的な違いについて話しました。 あなたが私たちのユーザーであれば、休日が遅れているので、更新の必要性を考慮することができます。 それまでの間、最小限の損失で「7」にアップグレードする方法に関する投稿を準備します。
