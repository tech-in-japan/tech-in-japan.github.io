Meteorのパッケージシステム

バージョン0.9.0（約2か月前）以降、 MeteorはAtmosphereに基づいてパッケージシステムを更新しました。 新しいパッケージシステムのリリースにより、各ユーザーは独自のパッケージを作成および公開できるようになりました。 現在のところ、システムには1800以上のパッケージがあり、約500人の貢献者がいます。



Meteor、Meteorite、Atmosphereは既にハブで何度も書いていますが、このフレームワーク用のパッケージを作成した経験を共有したいと思います。



この記事では、次のパッケージシステムが必要な理由、その使用方法、パッケージの作成、テスト、および公開方法について説明します。



パッケージシステムの主な機能



  • プロジェクトで使用されているパッケージをコマンドラインから追加、更新、削除します。
  • インストール済みパッケージのリスト。
  • パッケージキャッシュ-毎回異なるプロジェクト用に同じパッケージをダウンロードする必要はありません。
  • リモートストレージによるパッケージの検索。キャッシュされたパッケージによる検索も実行できます。
  • セマンティックバージョニングの仕様およびパッケージのバージョン管理に従ったバージョニング。




なぜ別のパッケージ配布システムが必要なのですか？



現在、すべてのWeb開発者がおそらく聞いたことがある少なくとも2つのパッケージシステムがあります。npmとクライアントライブラリのbowerです。 それぞれに数千のパッケージがあり、npmには毎日数百万のダウンロードもあります。 では、これらのシステムを使用して、Meteor上のアプリケーションにライブラリーを提供してみませんか？



[2]新しいシステム自体の作者によると、主な理由は、さまざまなプラットフォーム上でjsコードを実行できることです。ブラウザ、サーバー、または電話で -つまり、jsでコードを1回記述し、どこでも実行できます。これはいくつかの問題を引き起こす可能性があります。



開発者は次の例を示します。たとえば、ソーシャルネットワークFacebookなどのREST APIを使用して、サードパーティサービスからのデータを処理するサービスを作成するとします。 クライアントでは、NodeJS httpのサーバーモジュールまたはさまざまなアドオンでXMLHttpRequest機能を使用する必要があります。 どちらの場合もjsでコードを記述しますが、移植性がないことが判明したため、2つの異なるバージョンをサポートする必要があります。



Meteorのアイデアは、標準のHTTPパッケージを使用して、クライアントとサーバーの両方で同じように機能する同形コードを作成できるということです。 Meteorでのこのような同型のサポートは、データベースを操作するためのパッケージにも拡張されます。



Meteorでは、パッケージアセンブリシステムを使用して、特定のプラットフォームまたは一度に複数のプラットフォームで実行するコードを指定できます;アーキテクチャに依存するコードもパッケージに含めることができます。 システムはこのようなコードを正しく組み合わせることができ、完全にプログラム可能であるため、アセンブリプロセス自体を必要な機能で拡張できます。 これらすべてのために、ビルドシステムは、そのパッケージだけでなく、npmまたは他の同様のサービスからのパッケージでも動作できます。



使用する



Meteorパッケージ名のユーザー名形式は、 packagename @ versionです 。ここで、 usernameは投稿者名、 packagenameはパッケージ名、 versionはバージョンです。 Meteor自体の一部であるメインパッケージには、パッケージの作成者の名前は含まれていません。これらは、 http 、 coffeescript 、 standard-app-packagesなどのパッケージです。



次のコマンドの詳細なヘルプは、次のように入力して取得できます。 



$ meteor help <command>




パッケージの追加、更新、削除



パッケージを追加するには、プロジェクトフォルダーのターミナルで次の指示を実行するだけです。 



 $ meteor add username:packagename


たとえば、 流星は鉄を追加します：router 。 パッケージの特定のバージョンを指定するには、 meteor add username：packagename @ versionを使用できます。たとえば、バージョン0.1.0のパッケージが必要です-meteor add username：packagename@0.1.0を実行します。この場合、指定したバージョン以上がインストールされますが、互換性があります0.1.0 正確なバージョンを示すには、バージョンの前に=記号を記述する必要があります： meteor add username：packagename@=0.1.0 。 特定のバージョンを削除するには、単にmeteor add username：packagenameと入力できます 。



パッケージとMeteor全般を更新するには、次を実行できます。 



 $ meteor update


このコマンドはいくつかのパラメーターもサポートしています： meteor update package1 package2 ...-指定されたパッケージのみを更新し、 meteor update –packages-only-すべてのパッケージを更新し、Meteor自体を更新しません。



パッケージを削除するには、コマンドを使用します 



 $ meteor remove username:packagename




依存関係と制限



フレームワークパッケージシステムはすべての依存関係を自動的に解決しますが、特定のパッケージの1つのバージョンのみを含めることができます。これに関連して、いくつかの制限が発生する場合があります。 たとえば、パッケージAとパッケージBがパッケージCバージョン1.1.0に依存しているが、パッケージAの新しいバージョンがC@2.0.0に依存している場合、 meteor updateを実行してもパッケージAは更新されません。



また、パッケージシステムはインストールされたパッケージのバージョンをロールバックしません。 たとえば、依存関係にパッケージCを含むパッケージAをインストールし、パッケージAの依存関係で指定されたバージョンよりも低いバージョンのパッケージCをインストールした場合、パッケージCは新しいバージョンに更新されますが、パッケージAが削除されるとCバージョンはロールバックしません。



インストール済みパッケージの検索と表示



インストールされているすべてのパッケージとバージョン（指定されている場合）は、 .meteor / packagesファイルに保存されます。 このファイルの内容は次のとおりです。 



 # Meteor packages used by this project, one per line. # # 'meteor add' and 'meteor remove' will edit this file for you, # but you can also edit it by hand. standard-app-packages coffeescript http accounts-base mrt:jquery-ui mrt:moment mrt:nprogress


ファイルは手動で編集できます。保存時にMeteorが起動された場合、パッケージは自動的に更新されます。また、コマンドを使用してインストール済みパッケージとそのバージョンのリストを表示することもできます 



 $ meteor list


.meteor / versionsファイルで、インストールされた依存関係とそのバージョンとともにパッケージの完全なリストを見ることができます



パッケージに関する詳細情報（利用可能なバージョン、説明、サポートされているプラ​​ットフォーム、作成者）を取得するには、次のように入力します。 



 $ meteor show <package>


コマンドを実行して、必要なパッケージを見つけることができます 



 $ meteor search <substr>


検索はサブストリングごとに実行され、フルネームのサブストリングを含むすべてのパケットが返されます。 パッケージ情報はMongoDBにあり、リクエストは次の式packages.find（{name：new RegExp（substr）}）によってほぼ形成されるため、原則としてリクエストに正規表現を渡すことができます。

このガイドがあまりにも膨大であることが判明しないように、パッケージを作成するすべての瞬間を詳細に分析するために、特別なリポジトリのすべての例を参照することをお勧めします 。 すべての例が機能しているため、プロジェクトのクローンを作成して、それがどのように機能するかを手で感じることもできます。



パッケージを管理する



開発者アカウント



独自のパッケージを公開するには、 Meteor Webサイトでアカウントを作成する必要があります。 そして、コマンドを使用してパッケージを管理するコンピューターにログインします 



 $ meteor login




転記



既にパッケージを作成し、テストをデバッグして作成したとします。 コミュニティと共有するには、たった1つのコマンドを実行する必要があります。 



 $ meteor publish --create


このコマンドは、現在のディレクトリにあるパッケージを自動的に収集し、サーバーに公開します。 その後、 meteor search packagenameチームを使用してパッケージを見つけるか、 Atmosphere Webサイトでパッケージを検索してみてください。



パッケージの新しいバージョンを公開するには、 –createフラグなしでコマンドを実行するだけです 



 $ meteor publish


その後、パッケージもアセンブルおよび公開されますが、新しいバージョンを公開するときはpackage.jsの番号を変更する必要があります。変更しないとエラーが発生します。



一部のパッケージには、アーキテクチャ固有のコードが含まれている場合があります。 現在、システムは次をサポートしています：32ビットLinux、64ビットLinuxおよびMac OS。 さまざまなアーキテクチャのパッケージのバージョンを公開するには、必要なオペレーティングシステムを搭載したコンピューターにMeteorをインストールし、ログインしてコマンドを実行する必要があります 



 $ meteor publish-for-arch <username:packagename@version>


パッケージ自体をこのコンピューターにダウンロードする必要はありません。システムが自動的にダウンロードし、アセンブルして公開します。
