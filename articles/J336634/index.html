避けられないサーバーレスの未来はいつですか？

過去10年間、クラウドサービスの利便性に慣れてきました。 クラウドは、いつでも新しいサーバーを取得するエキサイティングな機会です。 次のステップでは、新しいアメニティを提供します-より高いレベルのサービスを提供するプラットフォーム：キュー、API、ゲートウェイ、認証ツール。 次に、普遍的なサーバーレスの幸福はありますか？





レゴアート展の写真



多くの場合、サーバーレスコンピューティングは、サービスとして機能（FaaS）形式でサービスを提供する既存のプラットフォームに関連付けられています。 これらのプラットフォームの多くは期待はずれであり、疑いを持って「サーバーレス」なものすべてを見ることを余儀なくされています。 しかし、これは物事の狭すぎる見方です。



今日は、サーバーレスプラットフォームの開発がすぐに私たちの態度をどのように変えるかについてお話します。 ユニバーサルサーバーレステクノロジーの3つの波を示し、それらの相互作用を示します。これは、FaaS製品よりもはるかに幅広い機会を提供することを目的としています。



「サーバーレステクノロジー」の概念について



サーバーレステクノロジーは常にサービスレベルの抽象化であり、エンドユーザーの利便性のために作成された錯覚です。 当然、ハードウェアはどこにも行きません。 実際、サーバーレステクノロジーには2つの定義的な特性があります。カスタム仮想マシンイメージではなく、目に見えないインフラストラクチャですか。 固定の時間給ではなく、 実際に消費されたリソースに基づいた支払いスキーム 。



これは一見すると思われるかもしれませんが、ほとんど神秘的ではありません。ほとんどのクラウドサービスは、すでにサーバーレスの原則で動作しています。 AWS S3やAzure Storageなどの基本サービスを使用する場合、保存されるデータの量（つまり、使用されるディスク容量のコスト）、I / O操作の数（データへのアクセスを整理するために必要なコンピューティングリソースのコスト）、および転送されるデータの量（コストネットワークチャネルの使用）。 これはすべてハードウェアサーバーに依存しますが、その容量は、このようなシステムの運用のこの側面について考えていない数百または数千の消費者に分かれています。



多くの開発者は、サーバーレス環境でコードを実行するという考えに混乱しています。つまり、ユニバーサルサーバーレスコンピューティングの考えには慣れていません。

コードが機能し始めたかどうかを知るにはどうすればよいですか？ 環境をデバッグおよび監視する方法は？ サーバーソフトウェアを最新の状態に保つ方法は？



これらは理解できる反応と質問です。 機能に関係しないこれらの問題はすべて、優れたサーバーレスアーキテクチャとは何かについて話し合う前に解決する必要があります。 この分野ですでに行われていることを見てみましょう。



第一波：イベント依存コンピューティング



FaaSプラットフォームは、サーバーレスコンピューティングの市場に参入した最初のプラットフォームです。 例には、AWS LambdaおよびAzure Functionsが含まれます。 あちこちで、オンデマンドで呼び出されるコードフラグメントを配置できます。 そのようなアプリケーションはありません。 開発者は、関数、アプリケーションのフラグメントを作成し、適切なタイミングでコードをトリガーするイベントに基づいてルールを設定します。 例：「HTTPリクエストが/ fooに到着したときにXを開始する」、「メッセージがこのキューにあるときにYを呼び出す」。 などなど。





関数は非常にシンプルにすることができます。 たとえば、簡単なタスクを実行するためのいくつかのメソッドの呼び出しで構成されます



開発者はサーバーについて何も知らず、気にしません。 コードは機能します。 この場合、プロバイダーシステムのリソースの機能を使用して計算される、ほんの一瞬だけ支払う必要があります。 スケーリングの問題は開発者の参加なしで解決されるため、関数が呼び出される頻度（1日に1回、または1時間に100万回）は重要ではありません。



したがって、FaaSは間違いなくサーバーレステクノロジですが、天気予報を自分でやることはありません。 これにはいくつかの理由があります。



  • 特定のイベントが特定の操作をトリガーするモデルにすべてのワークロードが削減されるわけではありません。

  • コードをその依存関係から明確に分離することは、常に可能なことではありません。 依存関係によっては、深刻なインストールおよび構成リソースが必要になる場合があります。

  • 開発中の言語やプログラミングパラダイムは、FaaSプラットフォームでサポートされていない場合があります。

  • 通常、既存のアプリケーションをFaaSモデルに変換することは非常に複雑であるため、財政的に実行不可能です。



これらはすべてアーキテクチャ上の問題です。



さらに、いくつかのシナリオでは、FaaSを非常に不便にする純粋に道具的な困難があります。 ツールは絶えず改善されているため、この種の問題については特に心配していません。 私たちが望むほど速くはありませんが、それがあなたがFaaSを使い始めるのを妨げるものであれば、これらの問題はすぐに解決されると確信しています。 主な問題は、これがいつ起こるかです。



第二波：ワークフロースキーム



ワークフローがイベントによって実行される小さなコードのモデルに収まらないためにFaaSシステムとの関係がうまくいかなかった場合、これは2つのことのいずれかを意味します。 プロジェクトがより複雑な組織を必要とするか、プログラムを継続的に実行する必要があるか、またはかなりの期間、本質的に同じものが必要です。



複雑な組織のプロジェクトでは、最初にAzure Logic AppsやAWS Step Functionsなどの製品が取り上げられました。 開発者は、グラフィカルインターフェイスを使用して、長いプロセスを説明するフローチャートを作成できます。 スキームの助けを借りて、関数の呼び出し順序を設定し、プロセス制御を編成するフレームワークに独自の機能を埋め込むことができます。





Azure Logic APPでアプリケーションを作成します。これは、CRMシステムに更新が表示されると、それらをさらに処理するために送信します。 さまざまなシステムを組み合わせた複雑な非同期プロセスの管理は、必要な手順の説明に限定されます。



複雑なシステムの相互作用スキームを整理するためのプラットフォームは、開発を簡素化します。 たとえば、電子ストアに関するレビューに関連する特定のコードを商品の購入から24時間後に立ち上げる必要がある場合、対応するタスクをスキームに追加するだけで十分です。 誰かがあなたの製品にTwitterで言及した場合、何らかのアクションをとる必要がありますか？ これを整理するには、マウスを数回クリックするだけです。 Twitter APIリクエストや回答への応答方法について考える必要はありません。 すぐに使用できるブロックは、プログラムの編成の問題を緩和するだけでなく、ほとんどの一般的なサービスのクライアントコードを記述する必要もなくなります。



スキームを使用するアプローチはビジネスプロセスのモデリングには優れていますが、すべての負荷をサーバーレスプラットフォームに転送することはできません。 これは万能薬ではありません。 たとえば、フローチャートの形式で表示される非常に複雑な決定ツリーは非常に厄介に見えますが、それを使用するのは不便です。 このような状況での通常のプログラムコードは、より表現力豊かです。 さらに、このようなサービスの支払いモデルは、スキームのステップの計算に基づいて、頻繁な要求とタスクを非常に小さな部分に分割することを高価な喜びにします。



要するに、関数を使用するアプローチも回路のアプリケーションも、複雑なアプリケーションを継続的に実行する作業を整理するという問題に対する価値あるソリューションを提供しません。 一見したところ、これはそれほど深刻な問題ではなく、サーバーレスコンピューティングの範囲がかなり狭いだけです。 しかし、1つの簡単な事実がすべてを変えます。過去数十年にわたって作成されたアプリケーションは、少なくとも部分的には継続的なタスクとして設計されてきました。 したがって、サーバーレステクノロジーの第3の波が来ました。



第三波：コンテナ化技術



アプリケーションのコンテナ化技術は、主にDockerの人気により、数年前に大衆化しました。 これらの技術の出現は非常に顕著であることが判明しました。 コンテナは、次世代の仮想マシンと見なすことができます。 最初は、Linuxエコシステムがコンテナ化システム開発者の焦点でしたが、今ではWindowsサポートについて話すことができます。 さらに、MesosphereやCubernetesなどのコンテナ管理プラットフォームがより一般的になりつつあります。



コンテナについて説明する際に考慮すべき最も重要なことは、コンテナをより軽量な抽象化による仮想マシンの通常の代替と見なすことができないということです。 コンテナは、汎用アプリケーションのパッケージ化メカニズムでもあります。 Node.js上のアプリケーションが特別なNginx構成を必要とする場合、コンテナで構成するだけで十分です。 ASP.NET WebサイトがバックグラウンドでWindowsサービスを使用している場合、コンテナイメージに適切なレイヤーを埋め込むことができます。



質問が発生する場合があります。「だから、コンテナの使用は明らかにアプリケーションをパッケージ化するための手段と考えられていますが、サーバーレステクノロジーはそれと何の関係があるのでしょうか？ コンテナは基本的に、コンテナホストで実行されるパッケージ化されたミニサーバーです。 実際、これは一種のサーバーです！ したがって、コンテナはほとんどサーバーレステクノロジーの敵であることがわかりました！」



いい質問ですね。 コンテナアーキテクチャには2つの要素があり、その適切な構成により、コンテナがサーバーレスプラットフォームに貢献できるようになると確信しています。 これは、サーバーレスコンテナーホスティングおよび自動コンテナーメンテナンスです。



サーバーレスコンテナーホスティング



Container Clustersは、効率的な大規模ホスティングプラットフォームです。 コンテナは、仮想マシンと比較して、ワークロードの密度を実際に高めることができます。



ただし、サーバーレステクノロジーに切り替えるには、仮想マシンのタスクを待機している常時動作するモデルを忘れる必要があります。 消費量の詳細な測定である通話ベースの請求が必要です。



これを実装する最初の大規模な提案はAzure Container Instancesで、このサービスの予備バージョンは2017年7月にリリースされました。 このソリューションを使用すると、コンテナを実行するインフラストラクチャを考慮することなく、必要に応じてコンテナを呼び出すことができます。 2ギガバイトのメモリを備えた単一の仮想プロセッサにコンテナインスタンスをデプロイしたいですか？ 



az container create --name JouniDemo --image myregistry/nginx-based-demo:v2 --cpu 1 --memory 2 --registry


ジョブが完了しました。 支払いには、コンテナコールあたり0.0025ドルに加えて、ギガバイトとプロセッサコアあたり1秒あたり0.0000125ドルが含まれます。 したがって、10秒間実行すると、0.002875ドルが請求されます。 これを1時間ごとに1か月間行うと、サービスのコストは2.07ドルになります。



私たちがマイクロビリングを実行する前に、そのような支払いモデルは実際、非常に効果的です。 バッチタスク用に仮想マシンを用意する必要はありません。大量のリソースが緊急に必要な場合、仮想マシンのクラスターを起動するのに時間がかかるのではなく、サーバーレスコンテナーが数秒でリソースを提供します。



しかし、サーバーレスコンテナーの便利なホスティングの可能性を考慮しても、サーバーを含むコンテナー、つまり「ベースイメージのOSを更新する方法」についてのジレンマが残っています。



コンテナサービスと自動化



サーバーレステクノロジーの教訓の1つは、開発者がインフラストラクチャのメンテナンスではなく機能に集中できるようにすることです。 コンテナは優れたツールですが、定義上、コンテナ化された環境をサポートするという形で追加のワークロードが付属しています。



アプリケーションを作成するには、オペレーティングシステムの基本イメージを選択し、このイメージにさまざまな必要なサービスを持つレイヤーを追加し、最後にアプリケーションをイメージに導入します。 新しいバージョンをリリースすることにより、コンテナイメージを再構築するだけです。 ただし、問題は残っています。 OSと依存関係がイメージに組み込まれている場合、どのように更新されますか？ 新しいWindowsパッチまたは新しいLinuxカーネルセキュリティパッチは、実行中のアプリケーションにどのように取り込まれますか？



このような質問への反省は、サーバーレスコンピューティングの性質に反します。 環境の依存関係を維持することは、サーバーレス開発者が行うべきことではないため、このタスクは自動化する必要があります。



ただし、環境へのサービス提供と重要な意思決定の間の境界線は曖昧です。 たとえば、典型的なWebサイトはオペレーティングシステムの更新から恩恵を受けますが、このアクションで危険なものは何も見つからないように思われます。 しかし、自動化と人々が下す決定の間の境界線をどこに引くのでしょうか？ 知らないうちにWebサーバーを更新すべきですか？ たとえば、Node.jsの新しいバージョンを自動的にインストールする必要がありますか？



これはトリッキーな問題であり、無視できない問題の1つです。 この議論を続けるために、MicrosoftのSteve Laskerとのインタビューを数分間聞くことをお勧めします（ .NET Rocks＃1459 、約37分）。



ここでは非常に長い間議論できますが、これはこの資料の範囲を超えていますが、次のような未来を想像してみてください。



  • コンテナには、ワークロードの主要な機能を検証する自動テストシステムが装備されています。

  • コンテナが実行されているプラ​​ットフォームは、基になるオペレーティングシステムの更新に関する詳細を知っています（たとえば、「Alpine Linux 14.1.5には重要なセキュリティ更新プログラムがあります」）。

  • プラットフォームは、新しいパッチをインストールした後、コンテナの動作を個別に検証できます。 ベースイメージが更新されると、事前にアプリケーションコンテナの再構築が試行され、テストスイートが実行され、結果に関するレポートが送信されます。

  • ルールベースの自動更新を取得することもできます。 例：「重要なOSの更新がリリースされたときに、テストが失敗しない場合、実行中のアプリケーションのイメージを自動的に更新します。」



このアプローチにより、コンテナ化されたサーバーがサーバーレスコンピューティングのまったく同じモデルになることが突然判明しました。 このレベルのサービスにはまだ成熟していませんが、これが業界の発展方法です。



コンテナを使用すると、小さなコードで動作する単純なFaaSフレームワークよりもはるかに複雑なワークロードを編成できます。 コンテナは、依存関係を処理する副作用を軽減します。 開発者のタスクを解決することを目的とした他のテクノロジーよりもはるかに強力であることが判明しました。



まとめ



この資料のタイトルには、「いつ...」という質問があります。 明白な答えは「今ではない」だと思います。 既存のサーバーレスコンピューティングプラットフォームには、まだ十分な欠点があります。



たとえば、Azure Container Instancesで24時間年中無休のコンテナロードを実行すると、費用がかかりすぎます。 仮想マシンは、このようなタスクに最適です。 仮想マシンでのOS更新の完全自動化は、もう1つの困難なタスクです。 また、コンテナサービスの自動化もまだ見えていません。 さらに、FaaS製品のログと監視は、AWSまたはAzureを使用する開発者にとって常に頭痛の種です。 Windowsコンテナの適切なサポートはどうですか？ 「私たちはこれに取り組んでいます」と、これまでのところ、他に何も聞いていない。



ただし、ツールは非常に迅速に改善されます。 機能に基づくサーバーレス計算とフローチャートに基づくシステムの両方が、この1年間で大きな進歩を遂げました。 コンテナ技術はまだ活発に開発されていますが、それらの関連性はその役割を果たしています。



さらに、小さな部分への負荷分散の概念、マイクロサービスの概念は、サーバーレステクノロジーの概念と密接に関連しています。 サーバーレスワークロード管理フレームワークの利点を最大限に活用するには、最初にこれらのワークロードを見つけておくとよいでしょう。 これは、集中的な開発が進行中の別の領域です。 Azure Event Gridなどのサービスを使用すると、異なるサーバーレスプラットフォーム間の接続を作成できます。



たとえば、AWS LambdaとAzure Logic Appsをリンクし、いくつかのイベントが発生したら、コンテナーを開始できます。 一般に、これらはすべて同じオペラからのものです。特定の問題を解決することを目的とした小規模なイベントベースのワークロードは、連携して大きな問題を解決します。



ここで説明している未来が来ると、前の世代のアプリケーションを新しいプラットフォームに配置することさえ可能になり、インフラストラクチャへの依存度がはるかに低くなります。 さらに、サーバーレスコンピューティングへの移行には、クラウドでも動作するように設計された、かなり現代的なアプリケーションを設計するための新しいアプローチが必要になることに注意してください。



一般に、サーバーレステクノロジの業界は動的に発展していると言えますが、将来的には多くの新しく興味深いものが期待されます。
