これらのジャンプはそれぞれ、給与、プロジェクトの面白さ、またはその他の利点のいずれかで増加しました 。 それらのそれぞれは非常に苦痛でした-私が1年間管理に従事し、コードを書かなかったとき、インタビューで「ArrayListにはいくつのメソッドがあるか」を思い出すことは非常に困難です。 逆に、ある年が1つの狭い焦点のトピックをコーディングした場合、多数のタスクからマルチタスクに切り替えて年間計画を立てることは非常に困難です（アジャイル？アジャイルとは？）。







これらのジャンプを最も簡単に行えるようにした最も重要なことの1つは、YouTubeの本とチュートリアルでした。 YouTubeについて説明します。給与が15,000ルーブルのJavaジュニアの場合、実際の会議に参加することはできません。食事をするのに十分なお金がありません。 国有企業で「アイデア」と3万ルーブルで働く場合、サンクトペテルブルクからジョーカー（ロシアでの主要なJava会議）に飛ぶことはまずありません。これには十分な給与がありません。 したがって-YouTubeで動画を見る。

JBreakの本質は何ですか



私の主観的な意見では、JBreak 2018：







  • かつて私が個人的に会議に参加したり参加したりすることを妨げていた組織のブロックの問題を解決します。
  • フォームとコンテンツに関する多くの願いを具現化しますが、それらは他では無視されます。


組織の問題



  • すべてはノボシビルスクで行われます。 飛行機のチケットの価格について気にしなくても、すべての雇用者はサンクトペテルブルクで1週間ジョーカーに行くことを許可しますか？ 数えてみましょう：そこに飛んでホテルに落ち着く日。 会議の2日間。 戻る日（4時間のフライト+ 4時間のタイムゾーンの差-これはすぐに1日を差し引いたものです）。 差し迫った締め切りに照らして、彼らがあなたをどこにも行かせないというリスクは大きい。 ただし、JBreakはノボシビルスクで開催されるため、どこへでも飛ぶ必要はありません。
  • 休みになります。 多くの雇用主は会議をナンセンスだと考えており、就業日には行かせないかもしれません。 しかし、JBreakは日曜日に開催されるので、これらすべての禁止事項は地獄にあります。 土曜日は仕事から少しリラックスし、日曜日はJBreakにアクセスして頭を突っ込み、月曜日は新鮮な考えと新しいツールでタスクを開始できます。
  • ノボシビルスクに行くのが難しい場合は、オンライン放送があります 。 オンラインの場合、 特別な種類のチケットがあります。 ちなみに、JUG.ruグループのシベリア会議では、初めてオンライン放送が行われますが、これは即興ではなく、テクノロジーは兄のJBreak-JokerとJPointで開発されました。
  • もちろん、会議の参加者には、すべてのレポートのビデオ録画が非常に迅速に提供されます。 会議を真の知識源と考えるならば、そのような量の有用な情報を一日で覚えることを望むのは狂気である。 幸いなことに、記録があります。
  • チケットは、JPointやJokerよりも大幅に安くなります。 約3回。 もちろん、JBreakは1日だけで、JPointとJokerは2日だけです。

Javaバイトコード検証：いつ、どのように、そして無効にできますか？



Nikita pjBooms Lipsky、Excelsior







Nikitaは、 Excelsiorが開発した認定Java SE実装であるExcelsior JETプロジェクトのイニシエーターおよびリーダーの1人です。 1997年からプロジェクトに携わり、コアから製品特性まで、製品のほぼすべてのコンポーネントの研究開発に参加しました。 特に、Excelsior JETのJVMレベルでのOSGIサポート、Java Runtime Slim Downテクノロジー（2007年からExcelsior JETに実装されたJava SEのモジュール化）、Excelsior JETのJavaバイトコード検証などの作者の1人です。







このレポートでは、Nikitaがバイトコード検証について説明します。 現在、Java開発者はライブラリを使用してランタイムでJavaバイトコードを生成し、Javaで表現することが困難または不可能なさまざまなトリックを効果的に実装しています。 ただし、Java言語を使用する場合、javacコンパイラは、出力が正しいJavaバイトコードになることを保証し、その後、バイトコード自体のレベルまで下がるため、多くの場合、その正確性を自分で監視する必要があります。 そうしないと、生成したクラスをロードするときにjlVerifyErrorを受け取ります。これは、JVMがJavaバイトコード検証を通じてロードするバイトコードの正確さを厳密に監視するためです。 したがって、バイトコードを生成します。多くの場合、バイトコード命令のセマンティクスを知るだけでは十分ではありません。また、Javaバイトコード検証がどのように動作するか、どのバイトコードが正しいと見なすか、またどのバイトコードが正しくないと見なすかを知る必要があります。







このレポートでは、JVMがバイトコード検証機能を実行するミッション、動作のタイミングと方法、アプリケーションのパフォーマンスに影響を与える可能性があるかどうか、無効にすることが危険な理由を明らかにします。

効果的なエンタープライズテスト



セバスチャン・ダシュナー







SebastianはJavaコンサルタントであり、書籍の著者であり、Java（EE）を使用したプログラミングにも携わっているトレーナーです。 たとえば、彼の著書Architecting Modern Java EE Applicationsを知っているかもしれません。 SebastianはJCPに参加し、JAX-RSやJSON-Pなどのグループに参加することにより、Java EE標準の策定を支援しています。 これらすべてに対して、彼はJavaチャンピオン、Oracle開発者チャンピオン、JavaOne Rockstarなどの称号を授与されました。 Javaに加えて、SebastianはDockerなどのLinuxのコンテナー化テクノロジーにも取り組んでいます。







レポートについて少し説明します。 ほとんどの開発者はまだテストの問題を避けようとしています。 テストはソフトウェアの動作にとって非常に重要であるという事実にもかかわらず、テストの開発と保守には、特に既存の機能の変更がテストスクリプトを適応させる場合、ある程度の時間と労力が必要です。 ソフトウェアテストを行うことは解決策ではありません。 これは、企業テストをどのように最も効率的かつ生産的に実行できるのかという疑問を抱かせます。







このレポートでは、Java EEアプリケーションの効果的で実用的な自動テストに必要なものを検討します。 迅速なフィードバック、実質的なカバレッジ、および一定の開発速度を維持する方法について説明します。 ほとんどの場合、さまざまなテクノロジーを使用したさまざまな規模のテストのライブコーディングに専念します。 コンテナとオーケストレーションフレームワークが効果的なエンタープライズテストをサポートする方法を示します。 特に、私たちは、ソフトウェアをマスターするための基本を習得する必要がある、サポートされた高品質のテストコードの開発に焦点を当てます。

過去1年以上に一度も会議を訪れたことがあれば、これは私たちには起こらないことを既に知っています。 各レポートにはディスカッションエリアが付いています 。 参加者とスピーカーの両方-この形式には誰もが喜んでいます-特別に指定された場所で静かに特定の質問をすることができます。スピーカーは、どんなに狭くても、タイミングやについて考えることなく、質問を建設的かつ徹底的に答える機会がありますそれはすべてリスナーにとって興味深いものですか。







「インターネット上のすべてを見つけることができるのに会議に行く理由」という質問に答えるのは、ディスカッションゾーンです。 それは簡単です：あなたの特定の質問を個人的に尋ねること。 さて、JBreak 2018では、このためのすべての条件が作成されます。長い休憩では、トラックオーバーにより、スピーカーが割り当てられたスペースにイラストレーションボード、座席、コーヒーカップを用意します。 今、タイミングでdrれる質問は一つもありません。







オンライン放送の特徴



  • 解像度2560x1440のビデオ。 この権限がある場合、JUG.ruグループのみがJava会議を削除します。
  • 任意の部屋からレポートを見る機会、あなたは個人的に来た参加者と同じくらい多くの機会を持っています！
  • さらに、質問がある場合は、会議のテレグラムチャットを通じてスピーカーに質問し、すぐに回答を得ることができます。
  • 改善された音質、真剣に再設計された録音システム。
  • レポートの合間に、プラットフォームのスピーカーと参加者がディスカッションエリアに向かうと、オンラインブロードキャストレポートの視聴者と、スピーカーとスポンサーへのインタビューを表示します。 それらは@phillenniumと私（つまり、@ olegchir ハブブラウザー ）が主導しています。 これがどのように起こるか-あなたはジョーカー2017の例を見ることができます;
  • 多くの小さな技術的な問題は解決されましたが、大多数は気付かなかったものの、そこにありました。 一般的に、放送を見ている感覚は良くなるはずです。
