コード最適化開発者向けのクイックWebアプリケーションまたは8つのヒントの作成方法

Diafan.CMSを作成するときに私たちが祈る2つの言葉は簡単で迅速です。 原子戦争の場合の大きなライブラリはなく、必要に応じて新しいものがすべて追加されます。 システムの一般的なロジックは長年にわたって改善および洗練されているため、システムでは機能を分岐および変更する余裕があり、コードを理解しやすく、CMSを簡単に開発できます。 これはどのように達成されますか？ いくつかのヒントを作成しました。



私たちのシステムは、高速で最適化されたシステムのタイトルを繰り返し獲得しています。 研究に関するコメントでは、そのような結果がどのように達成されるかについての記事を書くことを約束しました。 そして、そのような記事を他のどこで公開するのでしょうか？



Diafan.CMSに基づいて実際の例がレビューされているという事実にもかかわらず、以下のヒントは、サイト、個々のモジュール、モバイルアプリケーション、およびネットワークプログラムのWeb開発者にとって有用です。



1.ニーズに応じてのみアプリケーションをスケーリングします。



シンプルから複雑に移行します。



プロジェクトにSMS通知が必要だとしましょう。 SMS送信コードは数行で非常にシンプルです。電話の検証とSMS送信サーバーへのGETリクエストだけです。すぐに統合してください。



たとえば、Diafan.CMSが注文のために管理者へのSMS通知を必要とした場合、注文を形成する関数のShopモジュールにこの単純なコードを直接配置しました。



次に、「フィードバック」および「ニュースレター」モジュールでSMS通知が必要でした。 次に、別のファイルinclude / sms.phpのグローバル関数としてこの機能を削除し、次の形式でさまざまなモジュールで静かに使用しました。 



include_once('includes/sms.php'); Sms::send($message, $to);


SMS通知を設定とインターフェイスで補完する必要がある場合、この機能を別のモジュールに移行しました。



また、異なるバックエンドを接続する機能が必要な場合、たとえば、異なるSMSオペレーターと連携する必要がある場合は、モジュールでこれを実行できます。システムの一般的な構造ではこれが可能です。



一番下の行は、最初の段階で新しい機能を操作するための複雑なライブラリを開発する必要がないこと、サイト管理者を混乱させ、ドキュメントの束を書く必要がある100,500の設定をする必要がないことです。 そして、それは必要ないかもしれません。 この段階で必要なことだけを行うことをお勧めします。



システムに「コード-プラグイン関数-プラグイン」という明確な階層がある場合、これは雪だるま式につながりません。 アプリケーションは、開発のあらゆる段階でシンプルかつ理解しやすいままです。



開発の現在の段階で使用されていない不要なコードを拒否すると、1石で2羽の鳥を殺すことができます。



  • 論理エラーとセキュリティ問題を排除する完全なコード制御。
  • アプリケーションの合計サイズは増加しません。


さらに、このアプローチは別の機能に私たちを導いた。 外部コード、プラグイン、ライブラリはほとんど使用しません。 「自転車」エラーにもかかわらず、巨大なサードパーティライブラリを含めるよりも、タスク用に1つの関数を記述する方が良いと考えています。



2. URL処理に注意してください



リンクのサーバー解析は、サイトのすべてのページが通過するボトルネックです。 この場所をサーバーにとってできるだけ簡単にするようにしてください。



ほとんどの場合、サイトページのアドレスナビゲーションでは、サーバーのsite.ru/?PAGE=34の形式のサーバーアドレスの代わりに、サイトの各ページに文字列に適した美しいアドレスがある場合、いわゆる「CNC」（人間が読めるURL）が使用されます。 たとえば、site.ru / o_kompanii /で「会社について」ページが開きます。 ここで、「o_kompanii」は、site.ru /？url = o_kompaniiで解釈される文字列CNCです。これにより、CMSはデータベースでo_kompaniiという名前のページを検索し、そのコンテンツをサイトに表示します。 PAGE = 34で要求するかのように。 ただし、ページの文字列名に加えて、一部のパラメーターはアドレスバーを介して渡すことができ、多くのパラメーターが存在する場合があります。 このようなCNCの分析は非常に困難です。



Diafan.CMSには、リンクを構築するためのテンプレートが1つだけあります：/文字列CNC /数値パラメーター3 /別の数値パラメーター25 / 3、25-これらはパラメーターの値です。 つまり URLに数値パラメーターがある場合、変数に直面していることを意味します。いずれにしても、これは最も一般的な構造です。



すぐにすべての数値パラメーターを破棄し、何かが残っている場合は、データベース内の文字列CNCとそれに対応するページを確認します。 CNCでページが見つからない場合は、1つのパラメーターをCNCに追加し始め、データベースを再度調べます。 はい、そのようなアルゴリズムはいくつかの追加のSQLクエリで構成されていますが、そのような特別なページまたは404ページにのみ使用されます。 それは、残りのすべての（およびそれらのほとんどの）ページのベースをロードしません。



site.ru/?url=page:13&m=3&m2=25など、$ _GETデータを含むいリンクのみがこれよりも簡単です。



3.必要な場合にのみ機能を実行する



サイトのページを開始するときは、まず必要な機能のみをダウンロードします。99％のケースで使用されます。 必要な場合にのみ、他のすべてを実行します。



たとえば、評価をDiafan.CMSの製品に接続する必要がある場合、変数$ this-> diafan-> _ ratingに変更するだけで、「評価」モジュールの接続部分全体が返されます。 疑似変数$ this-> diafan _ ***への最初のアクセス以来、要求されたモジュールの存在がチェックされ、クラスへの正しい接続がクラスのインスタンスとして実行されます。 使用すると便利であり、スクリプト実行の現在の瞬間に余分な機能を遮断します。 ページの評価が不要な場合、まったく開始されません。



4.追加の負荷を作成する機能、オプションで接続



追加機能をすべてのモジュールにグローバルに接続せず、オプションで有効にする機会を与えます。

たとえば、コメントが使用されるモジュールでは、Diafan.CMSには「コメントの接続」オプションがあります。 これは、製品に関するコメントが必要であるがニュースでは不要な状況を除外し、ウェブマスターは単にニューステンプレートのコメント出力を非表示にしますが、CMSはSQLクエリの実行、ニュースに関するコメントの検索、および必要なすべてのデータの生成を続けます。 言い換えれば、無駄なサーバーリソースの無駄遣いです。



サーバーリソースを最小限に使用する他の機能にも同じことが当てはまります。 ヒットカウンター、ニュースの評価、メニュー項目の画像は必要ありませんか？ 設定への接続を取得し、デフォルトで切断します。 これにより、リソースが節約されます。



異なるユーザー（ゲスト、ユーザー、卸売業者など）のサイトのさまざまな特定の素材に対するアクセス制限を柔軟に構成する必要がある場合、「アクセス権」などの隠し設定にも同じことが当てはまります。追加：



...現在のユーザーが要求されたマテリアルにアクセスできるかどうかを確認するために、 a.element_id = 'id_material' AND（e.access = 'yes' AND a.role = 'role_current_user'）として 、 LEFT JOINアクセスします 。



そして、すべてのユーザーがすべてのニュースを見て、これらの状況がはるかに多い場合、なぜこの負荷が必要なのでしょうか？ このLEFT JOINをオプションでSQLクエリに追加することは理にかなっています。 Diafan.CMSでは、この設定のオン/オフは管理者の参加なしで自動的に行われます。 つまり、一部のマテリアルでアクセスを制限する必要がある場合、チューニングとデータベースクエリが複雑であることに注意するだけで十分です。 そうでない場合、要求は単純です。



5.データベースクエリの数を最小限に抑える



準備機能を使用します。



あるモジュールの機能が別のモジュールで使用されることがあります。 ストアモジュールには10個の製品があり、画像モジュールの各画像があると想像してください。 初心者が犯す最も一般的な間違いは、出力サイクルで画像テーブルに追加のリクエストが行われた場合です。 



 $rows = SELECT * FROM shop; //  foreach ($rows as $row) { echo "".$row['id']; //    $img = SELECT * FROM img WHERE shopid=$row['id']; //     foreach ($img) { echo "  ".$img["link"]; //     } }


*ここと以下で、コードは条件付きであり、アルゴリズムの明確さと意味の伝達のためです。



したがって、1 + 10 = 11のSQLサーバーへの個別のSELECTクエリが取得されます。 ただし、商品をリクエストし、受け取った商品のすべてのIDを覚えてから、SQLクエリを1つだけ作成するのが最善です：すべての商品のすべての画像を一度に選択します。 11ではなく、2つのSELECTクエリのみを取得します。 



 $rows = SELECT * FROM shop; //  foreach ($rows as $row) { $goods[] = $row['id']; //     } $img = SELECT * FROM img WHERE shopid IN implode($goods[]); //     foreach ($img) { $images[] = $img['link']; //     } foreach ($goods[]) { echo "".$goods[]; //    echo "  ".$images[$goods[id]]; //    }


しかし、ここでの問題は、ループのロジックに違反し、コードが不快になることです。 使いやすく、サーバーに最適なエレガントなソリューションである準備機能を見つけることができました。 コードは次のとおりです。 



 $rows = SELECT * FROM shop; //  foreach($rows as $row) { $this->diafan->_images->prepare($row['id'], 'shop'); //  } foreach($rows as $row) { $images[$id] = $this->diafan->_images->get($row['id'], 'shop'); //    get echo "".$row['id']; //  echo "  ".$images['id']; //    }


prepare関数は、写真を取得するすべての要素を単純に記憶します。 また、get関数の特性は、最初の呼び出しでSQLクエリが実行され、結果も記憶されることです。 このループからの他のすべてのget呼び出しは、初めて受信したデータで機能します。



したがって、リンク構築のためのCNC要求、タグ要求、評価要求など、あらゆる機能に対して行われています。したがって、Diafan.CMSでは、SQLサーバーへの呼び出しは最小限です。



6.キャッシングを使用する



キャッシュアルゴリズムを記述し、サーバー側のアルゴリズムを使用します。



オンラインストアが落ち着いていて、新しい製品が追加されない場合、毎回データベースに連絡して各顧客の製品のリストを作成するのはなぜですか？ SQLサーバーに一度アクセスし、クエリ結果をキャッシュフォルダー内のテキストファイルに保存して、そこから取得するだけで十分です。 リストに新しい製品が追加された場合、または製品のソートされたリストなどの別のページが要求された場合、キャッシュファイルをリセットして、データベースを再度参照できます。 ただし、キャッシュを更新するために一度だけ。



サイトのすべてのページだけでなく、サイトのセクション、ブロック、リスト、カード、要素の両方をキャッシュできます。 詳細



静的リソースは動的リソースよりもサーバーへの負荷が1桁少ないことを覚えておくことが重要です。そのため、SQLとPHPインタープリターの使用はできるだけ少なくしてください。 デフォルトでは、Diafan.CMSはファイルキャッシュを使用しますが、Memcacheを接続することもできます。 データのみをキャッシュします。 すべてのデザインがキャッシュに入るわけではありません。 これにより、キャッシュの下のスペースを節約でき、その結果、キャッシュをより高速に処理できます。



7.動的なデータ取得を最小限に抑える



各ユーザーを待つよりも、サイト管理者を1回待つ方が適切です。



たとえば、Diafan.CMSでは、1つの製品が複数の価格を持つことができ、それぞれが異なる通貨でも割引が設定されている場合、割引の柔軟なシステムがあります。 各サイト訪問者のサイトでの新しい価格は計算しません。 これは、サイトの管理部分で割引を設定するときに行います。選択した商品のすべての異なる価格オプションを検討し、price_id記号の付いた別のshop_priceテーブルに収集します。 製品の元の価格はid = price_idとして定義されます。 id <> price_idの場合、これは割引またはサイトのメイン通貨での価格の変動です。 また、この表は、この価格が特定のユーザーグループに設定されている場合に、誰に対してすぐに表示されるかを示します。



はい、多くの商品がある場合、割引を追加または変更するとき、管理者は回転するホイールを見ながら待機します。 しかし、サイトへの訪問者は待つ必要はありません。サイトに表示するとき、検索、並べ替え、その他の容量のある操作を行うとき、システムは1つのテーブル、price_idから利用可能なすべての価格から最小のすべてを選択するだけです。



8.テーブルでDB正規化とインデックスを使用する



情報の複雑な構造をさまざまな単純なテーブルに分散し、それらのインデックスを使用します。 1つの複雑なINNER JOINを作成するよりも、2つの単純なクエリを作成する方が有益です。



Diafan.CMSでは、基本的にすべての隣接するテーブルはほぼ同じ構造を持っています：id、element_id、element_type、module_name。 それらの要求は非常に簡単です。

SELECT * FROM WHERE element_id = 3 AND module_name = 'shop' AND element_type = 'element'

また、指定されたすべてのフィールドには、インデックスが必要です。
