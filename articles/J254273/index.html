Bitrix上のプロジェクトのISPmanagerの最適化、またはISPmanagerとVMBitrix（Bitrix環境）の交差方法

2015年4月23日更新



  1. ソリューションはISPmanager 4で正常にテストされました
  2. Apache MPM-ITKモードでは、restore.phpスクリプトがエラーをスローします。その情報は「重要な情報」セクションにあります




なんで？









1C-Bitrix：Web Environment-Linuxなどの製品を既にご存じかもしれません。 非常に多くの場合、これはBitrix CMSに基づくプロジェクトの理想的なソリューションですが、純粋に「コンソール」メニューにプロジェクトで作業している人々に必要な機能がない場合があります。



たとえば、別の人気製品であるISPmanagerを使用すると、管理者がマウスを数回クリックするだけで便利なWebパネルからFTPアカウントを作成でき、Linux管理スキルが必要なコンソールから同様のアクションを実行できます。 ただし、sshコンソールの助けを借りずに、ブラウザのタブからサーバーとプロジェクトを管理したいだけの場合もあります。



ただし、ISPマネージャー（およびこの場合は最新バージョンの5番下）は、企業ポータルであるBitrix24はもちろん、Bitrixサイトでもすぐに使用できる状態にありません。 機能の一部は利用できず、プロジェクトの動作は非常に遅くなります。 残念ながら、関連情報の更新が非常に遅れる場合があるBitrixドキュメントの推奨事項を検討するのに長い時間がかかります。



「 使いやすい 」ISPmanager 4/5インターフェースを介してプロジェクトを操作する必要性と、プロジェクトの速度と機能を維持する必要性のため、これら2つのシステムのある種の「交差」について決定が下されました。







ですから、始める前に、重要な情報を注意深く読んでください





  1. インストールスクリプトは、安定したISPmanager 4/5をインストールし、nginxの「機能」を含めた直後に、クリーンなCentOS 6.6 x64_86でのみテストされました。

    以前にシステムの設定を変更した提案ソリューションを使用する上級ユーザーの場合、 スクリプトを自動モードで完全にインストールすることはお勧めできません-手動でコマンドを実行してください
  2. 著者は 、提案されたソリューションの安定した動作または正しい動作を保証するものではありません 。 すべての指示は自己責任で行ってください。 スクリプトと構成ファイルの現在のバージョンは、アクティブなテストの段階にあります。
  3. システムの「圧縮」モジュールは削除する必要があります。
  4. Apache MPM-ITKモードのBitrix restore.phpスクリプトは、php.iniでshort_open_tagパラメーターをオンにする必要があるというエラーでクラッシュします

    これは、このスクリプトのif条件の誤った使用が原因で発生します。 他のすべてのphpモードでは変調されないため、問題が存在しますが、1C-Bitrixの開発者に既に連絡しており、問題を修正する必要があります（別の質問はいつ行うかです）。 誰かがバグの詳細に興味がある場合-私はコメントで答えます



    これまでのところ、一時的な修正として、記事に添付されたアーカイブの配布キットを使用してサイトを復元します。



    全自動モードでの最適化









    便宜上、ソリューション全体が完全に自動化されたスクリプトと事前に準備された構成ファイルに組み込まれています。



    次のシナリオが考慮されます。



    1. クリーンなCentOS 6.6 x86_64をインストールした
    2. 公式の指示に従って ISPmanager 4/5をインストールしました
    3. ISPmanager 4/5パネルの[機能]セクションで、Nginxが有効化されました








    次に、最適化自体に進むことができます：（出版物の最後にあるすべてのリンク）

    1. 「bitrix-env_5.1.2_patched_2.sh」を実行します
    2. スクリプト「isp_patch_V0.x.sh」と解凍されたアーカイブ「patch_filesV0.1.zip」をサーバー上の任意のディレクトリにコピーします
    3. 「isp_patch_V0.x.sh」を実行します




    おめでとうございます、ISPmanager 4/5を搭載したサーバーはBitrixプロジェクトで動作する準備ができました



    新しいドメインにサイトを展開するためにのみ残ります：（発行の最後にあるすべてのリンク）





    • これを行う方法に関する私のオプションは、別の小さな指示で「 host.pdfを作成する 」（いずれの場合でも、作成されたホストに必要なapacheとnginxの設定を接続することに注意してください）
    • 便宜上、Bitrixをインストールするための配布キットを「 bitrix_install.zip 」アーカイブに投稿します（この場合、 インストールはクイックインストール/ショートインストールモードで行われます）




    どうした





    完全に機能するデフォルトのISPManager 4 /5。Bitrixのプロジェクトを迅速かつ機能を失うことなく操作することを学びました。



    アセンブリはすでに当社で使用されています。EMArtは、サイトとスタンドの両方、および顧客サイトの生産に使用されています。



    あなたのコメント、コメント、提案のいずれかを喜んでいます！







    提案されたソリューションをより詳細に検討してください。





    ステージ1. bitrix-env.sh（または「1C-Bitrix：Web Environment」-Linux）









    Bitrixのクラシックインストールは、スクリプトを介してインストールされるBitrix環境で発生します

    www.1c-bitrix.ru/products/env

    ISPmanagerの最適化を開始するのは、bitrix-env.shの解析です。

    実際、スクリプト自体には特別なものはなく、最初はシステム配布の標準チェックがあり、このチェックの結果に基づいて、必要なソフトウェアがインストールされます。

    スクリプトが「独自の」1C-Bitrixリポジトリを追加し、bitrix-env *およびbx-nginxパッケージをシステムにインストールすると、すべての魔法が発生します。 それらに焦点を当てます。



    したがって、「 bitrix-env_5.1.2_patched_2.sh 」は1C-Bitrix自体からわずかにやり直したスクリプトにすぎませんが、bitrix-envをインストールする際の主要な「トリック」は削除されています。



    ステージ2. bitrix-env.rpm（またはメイン構成ステージ）









    bitrix-envが提供するエコシステムは私たちに適していないため、競合する可能性のあるISPmanagerパッケージ/構成およびBitrix環境を積み上げる必要はありません。



    私の意見では、必要だけを残して、rpmパッケージからisp_patch.shスクリプトにすべてを転送しました（注、centos 6.6 x64でのみテストされました）。



    さらに-もう少し。



    ステージ2.1 bx-nginx（またはプッシュとプルをサポートするNginx）









    Bitrix環境が提供するこのパッケージは、「プッシュアンドプル」モジュールを備えたコンパイル済みのnginxにすぎず、「ビジネスチャット」、「ライブコメント」、「ビデオコール」、「モバイルアプリケーション。」



    ISPmanagerを使用してマシン用の既製のnginxバイナリである、適切にコンパイルされた1C-Bitrixを選択するだけで、コンパイルの手間を省くことができます（スクリプトの22行目と23行目）



    ステージ2.2 bvat.bx（またはソフトウェアパラメーター自動チューナー）







    Bitrix環境の興味深いコンポーネントはbvat.bxスクリプトです。これは、システムのスタートアップに書き込まれ、プロジェクトの操作（主にmysqlサーバーとメモリ消費）を​​担当するソフトウェアパラメーターのチューニングを実行します。 チューナー自体は非常に簡単に機能します。 システムの現在の構成に基づいて、彼は設定の1つまたは別の「プリセット」を有効なパラメーターとして公開します。 構成では、bvat.bxが完全に無効にすることなく設定できるパラメーターの変更を提供していることに注意してください（そうすることはできます）。



    スクリプトでは、25〜31行目でインストールを行います。



    参照Bitrix環境と同様に、/ etc / mysql / conf.d / z_bx_custom.cnfファイル（bvat.bxのバイパスに使用されます）でmysqlサーバーに独自のパラメーターを指定できます。 また、bvat構成ファイルの後に構成ファイルを接続することにより、1つまたは別のパラメーターセットが自動的に適合しない場合、ソフトウェアの残りの部分の自動構成をバイパスできます。



    スクリプトのロジックは自分の好みに合っているため、「自動化」を独自の値で補完することは難しくなく、システムへのインストールは非常に簡単で、構成に含めました。



    ステップ2.3 PHP設定





    php.iniのいくつかのパラメーターを変更します（スクリプトの33〜40行目）

    • memory_limit = 512M
    • pcre.backtrack_limit = 1000000
    • short_open_tag =オン


    PHPのdavモジュールを無効にします

    ステップ2.4 mysqlの設定







    スクリプトの42〜48行で​​、以前に準備および構成したmy.cnfをコピーします。このファイルでは、おそらく既に自分で何かを変更したいでしょう。



    ステージ2.5 bitrixenv.ini（またはBitrixが機能するために必要なphpディレクティブ）









    古典的なBitrix環境の不可欠な部分は/etc/php.d/bitrixenv.iniファイルです。このファイルには、必要なディレクティブ設定がリストされています。



    元のファイルとのわずかな違いで、ファイル/etc/httpd/bx/bx_apache.confとしてISPmanagerを使用してマシンにコピーします（行50〜52）。



    必要に応じて、 このファイルをISPmanager（config-apacheセクション）で作成された個別の仮想ホストにのみ登録するために、 意図的にどこにも含めないことに注意することが重要です。



    ステップ2.6 nginxの構成







    私のスクリプトのnginx設定には「オートチューニング」のシェアがあります（64行目と67行目）

    残りは、Bitrix環境のnginxの構成ファイルから取得されます



    72〜77行目はサーバーレベルの圧縮を担当しています

    79-82-プッシュ＆プル用



    bx / conf / bitrix.confファイルに注意してください 。 ISPmanager（config-nginxセクション）で作成された個々のホストにのみ登録するために、必要に応じて意図的にこのファイルをインクルードしないことに注意することが重要です。



    さらに、ホストセキュリティに関連するオプションが事前に用意されているがコメントアウトされた「いくつかのセキュリティオプション」ブロックがあります。
