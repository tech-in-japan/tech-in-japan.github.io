マップサービスの方法

「ソーシャルコミュニケーションの最も有望な技術の1つは、ロケーションベースのサービス（LBS）で表されます。これにより、ユーザー/モバイル加入者のロケーションを特定できます。

CNews（詳細： www.cnews.ru/reviews/index.shtml?2012/06/06/492022 "）



Mediagatesチームはこの見解を完全に共有しています。 そのため、今日、私たちはHabrasocietyにプロジェクト-Mediagates.ruのジオロケーションサービスをどのように開発したかを伝えたいと考えています。 投稿には、地図作成サービスの選択、タスクの説明、難易度、解決のためのアルゴリズムが含まれています。







プロジェクトMediagates.ruについて



これが何であるかを明確にするために、まずプロジェクトについていくつかの言葉を述べます。 Mediagatesの主な目的は、アーティスト、プロデューサー、プロモーター、会場など、音楽分野の専門家を団結させることです。 プロジェクトの機能とアーキテクチャの詳細については、Habréの次の投稿で説明します。 それまでの間、あなたは私たちのウェブサイトでプロジェクトに関する投稿を読むことができます。



サイトの設計段階でも、情報をより視覚的に表現するには、マップを使用するだけでよいことがわかりました。 近くのアーティストをすばやく見つけたり、ジオターゲティングを使用して自分に関する広告を正しく配置したりできます。 そして今、ベータテストのためにハブロフスク市民に全員を招待します。 この記事の後半でテストできるものについて。





主なタスク



私たちの主で最も重要な要件は、プロジェクトのほとんどのページでインタラクティブおよび静的マップを最大限に浸透させることでした。 サイトの構造を検討した後、プロジェクトのさまざまなセクションに配置できるいくつかの基本モジュールが必要であるという結論に達しました。



  1. 特定の種類のマーカー（イベント、アーティスト、プロモーター、会場、ショップなど）を表示し、日付でフィルターする機能を備えたインタラクティブマップ。
  2. アーティストがマップまたは大きなカレンダーでパフォーマンスと動きを計画し、さまざまな会場に自分自身を提供し、他のユーザーから共同パフォーマンスに関するオファーを受け取ることができるツアープランナー。
  3. サイト上のすべてのコンテンツをマップにリンクできるようにオブジェクトの場所を編集するためのフォーム（たとえば、ユーザーにとっては、何らかのチェックイン機能です）。
  4. リスト内のオブジェクトを示すためにマーカーが付いた静的カード（そうしないと、各ページで数十枚のカードを初期化する必要があり、パフォーマンスとマップサービスの制限の両方の面で不利になります）。 また、ツアープランナーをJPGイメージにエクスポートするには静的マップが必要でした。


Mediagates.ruのカードの機能の詳細については、サイトのブログに投稿された記事「 カードへのすべて 」で読むことができますが、ここではサービスの実装の技術的側面について具体的にお話したいと思います。



もちろん、これらの点はそれぞれ個別に実装が比較的単純で明白であると言うでしょう。 しかし、私たちはそれらのそれぞれを掘り下げ、それらすべてを1つのプロジェクトに結合する方法を理解し、実装中に直面しなければならないすべての困難と落とし穴に対処することを提案します。 まず、非常に重要な決定から始めましょう。マップサービスの選択です。



マップサービスの選択



Habréのこのトピックに関する記事やレビューがたくさんあるので、詳細に説明する必要はないと思います。 ここで遭遇した主な問題は、Google Maps APIサービスを設計する過程で、支払いが発生することでした。 これにより計画が台無しになるのではないかと心配しましたが、詳細な計算を繰り返した後、Google製品を使用するという決定にとどまりました。



確認したマップサービス：



  • Googleマップ
  • Yandex Maps、
  • Cloud Made（またはOpen Street Map）、
  • ビング
  • API 2GIS。


最終分析は2012年2月上旬に行われ、それ以降状況は変わる可能性がありますが、選択パラメーターは次のとおりでした。



  • エイスク市の質（プロジェクトの開発の初期段階では、ロシア連邦が十分に詳細に描かれていることが重要です）
  • 世界中に地図を描く（将来の野心的な計画がある）
  • カードの外観をカスタマイズする意思（結果として、これを放棄しました）、
  • マップの初期化または特定のAPIへのリクエストの数に関する法的制限、
  • ジオコーディングの品質（ロシア連邦を含む）。


比較の一般的な印象は次のとおりです。



  • OpenStreetMapは世界中で最も詳細であることが判明しましたが、最高のGeocoding APIはありません。
  • Yandex Mapsでのマドリード（スペインの首都）の描画レベルは低いです。
  • Bingはロシア連邦に対する準備が不十分です。
  • API 2GISは良いスタートを切りました-将来、この市場の主要プレイヤーと競争することを望みます。
  • Google MapsはファンキーなGeocoding APIを提供しますが、ロシア連邦のいくつかの都市の詳細には苦労しますが、ヨーロッパについては言えません（ナビゲーターの代わりにiPadで都市をナビゲートするためにマップを何度も使用しました）。 Google Pay


その結果、支払いにもかかわらずGoogle Maps APIに最も満足していることに気付きましたが、将来的には別のマップサービスに切り替える必要がある可能性があります。 この点で、クラスをいつでも追加して別のマップサービスを操作できるようにコードを整理し、マップでの作業の大部分を書き換える必要がないようにすることにしました。 言い換えれば、彼らはタスクを2つのサブタスクに分割することを決定しました。それは、サイトでの対話型とマップサービスとの通信です。



Googleは非常に優れたジオコーディングサービス（Google Geocoding API）を提供してくれました。 正直に言うと、私たちの開発者は、プロフェッショナルコミュニティに存在するジョークを確認する機会がありました。「Yandex-すべてがあります。 Google-何も失われませんでした。」：Google Geocoding APIは、ユーザーが元の言語で入力することなく、非常に多くの住所の場所を見つけることができます。



Googleサービスの欠点のうち、ロシアの弱い「知識」と料金（APIへのアクセスとカードの初期化の制限を超えた場合）をもう一度強調する価値があります。 これまでのところ、カスタムカードの初期化の制限が緩和されたため、カードのカスタマイズを放棄する必要がありました。



したがって、選択されるサービスプロバイダーはGoogleです。 だから私たちの処分では：



  • Google Maps API v3、
  • Google Geocoding API v3、
  • Google Static Maps API v2。


次に、フロントエンドとバックエンドの一般的なアーキテクチャを扱うことを提案します。 フロントエンドから始めましょう。



フロントエンドの主要コンポーネント



すでに述べたように、将来、別のカードプロバイダーと連携する機能を迅速に追加できるようにしたいと考えました。 そのため、基本クラスを開発しました。実際には、次のようなクラスの拡張です。



  • google.maps.Map、
  • google.maps.Marker、
  • google.maps.Geocoder。


マーカーについては、名前とポップアップリストを表示するための追加機能が必要でした（たとえば、マーカーのグループ）。 基礎として、Gary LittleからMarkerWithLabel for V3クラスを取得し、必要に応じて変更することにしました。



また、モジュールを操作するためのメインクラスを準備しました。



  1. さまざまな種類のマーカーのフィルターと小さなカレンダー（1つのレーン）を備えたインタラクティブマップ、
  2. 最初のクラスに基づいて構築されたツアープランナー。小さなカレンダーを大きなカレンダーに置き換えます（当然、機能を拡張します）。
  3. オブジェクトの場所を示します。


バックエンドの主要コンポーネント



バックエンドは、次の主なタスクを実行することになっています。



  • フロントエンドからの要求とそのグループ化の条件を満たす、多数のマーカーのデータベースからの選択。
  • リストに要素を表示し、ツアープランナーの現在のマップビューをjpgイメージにエクスポートするための静的マップの生成。
  • ユーザーアクションのマーカー（またはマーカーグループ）を連続曲線と組み合わせます。


ここでは、まずマーカーのグループ化に注意を払う価値があります。次に、カーブマーカーを結合するプロセスと、最後にマップのエクスポートを検討します。



マーカーのグループ化



マップ上の多数のオブジェクトを操作する必要がある場合、マーカーをグループ化する必要が生じます。 一部のブラウザは、数十個の非標準マーカーの処理で速度が低下し始めます。 これを回避するには、マップ上のオブジェクトをグループに結合する必要があります。



グループ化には、フロントエンドとバックエンドの2つのタイプがあります。 1つ目はJavaスクリプトで実装され、2つ目はサーバー言語で実装されます。 Java Scriptのグループ化には、使用できる既製のエンジンが多数ありますが、サーバーからブラウザーに大量のデータを転送することも恩恵のないタスクです。データが多いほど、ブラウザーの処理が遅くなり、最終的には減少しますサイトの速度と情報の認識の悪化。



サーバー側マーカーのグループ化



サーバー側のマーカーのグループ化に落ち着いた後、この領域で既にどのアルゴリズムが発明されているのか疑問に思い始めました。 グリッドベースと距離ベースの2つの最も一般的なクラスタリングのタイプについて話すのが習慣です。 最初のオプションはより単純で、2番目はアルゴリズムと実装の点でよりクールです。 クイックスタートのための最初の道を歩みましたが、距離ベースのクラスタリングにすでに取り組んでおり、近い将来に実装する予定です。



グリッドベースのグループ化の本質は簡単です。各近似（ズーム）ごとに、地球はグループ（比較的言えば長方形）に分割されます。 つまり、複数のオブジェクトが一度にこれらの図のいずれかに該当する場合、それらは1つのマーカーにグループ化され、グループを象徴します。







曲線マーカーの組み合わせ



当初は、地図上の特定のマーカーまたはマーカーのグループを時系列で結合するだけの破線を作成することを考えました。 これは、google.maps.Polylineクラスとgoogle.maps.PolylineOptionsクラスを使用して行うのが最も便利であることは明らかです。 しかし、最初に、GUIデザイナー、そしてプログラマーは、破線ではなく、ユーザーのマーカーを結合する滑らかな（ポイントによる構築が許す限り）曲線を持つ方がはるかに美しいと判断しました。







グループ化と同様に、ユーザーのコンピューターに負荷をかけすぎないように、サーバー側でこの曲線を設計することにしました。 短いテストの後、3次スプラインを使用した補間方法を決定しました。その結果はサーバーからJavaScriptに送信され、google.maps.Polylineを使用してポイントごとに既にレンダリングされています。



アルゴリズムについて詳しく見てみましょう。 数学の観点からは、複雑なことは何もありません。 PHPでは、次のクラスを作成しました。



  1. ポイント-ポイントの座標の視覚的なストレージ
  2. Cubic-（f（x）= d * x ^ 3 + c * x ^ 2 + b * x + aの形式の）関数の値と、特定のxおよび係数a、b、c、dの導関数を取得するためのクラス
  3. ポリ-補間自体

静的マップとJPGへのエクスポート



ユーザーが地図とカレンダーでツアーを計画した後、おそらくソーシャルネットワークで友人に通知するか、ブログでこの情報を公開したいと思うでしょう。 これを行うために、ユーザーのコンピューター上のすべてのマーカーを含む現在のマップビューをJPG形式で保存できるようにしました。

「JPGとして保存」ボタンをクリックすると、Java Scriptはマップの現在のビューとその上のマーカーに関するすべての情報をサーバーに送信します。サーバーは、Google Static Maps APIを使用してマップの目的の部分を読み込み、マーカーを配置して保存します一時ファイルストレージ。ブラウザでユーザーに返されるリンク。



Google Static Maps APIから画像を読み込んだ後、地図上にマーカーを配置しようとするのは、やるべきではない主なことです。良い結果が得られることはまずありません。 ソリューションははるかに簡単です。APIを使用すると、地図上の座標を指定するときに外部画像を配置できます。



ここで、1つの質問が残っています。fb.comまたはvk.comのユーザーの壁に地図を投稿する方が良いでしょうか。 問題を詳細に調査したとは言えませんが、マップをjpg画像としてエクスポートし、それを壁に手動で公開することほど良いものはありませんでした。 バックグラウンドで大きな画像を公開する方法を教えていただければ幸いです。 同時に、この画像はユーザー生成の概念に適合しないため、Facebookに大きなサイズを投稿できることを理解する必要があります。



地図検索



結論として、地図上のオブジェクトの検索に関するいくつかの単語を追加したいと思います。 この機能を考えたとき、グラフィカルインターフェイスの利便性を組み合わせて、最も関連性の高い結果を取得しようとしました。 UIの観点から、私は常にafisha.ruでの検索が好きでした-あなたはあなたが望むすべてを入力し、彼はこのリクエストのためにサイト上にあるものをあなたに伝え、さらにコンテンツタイプでそれをソートします。



その結果、検索エンジン（または、サーバーと呼ばれることもあります）sphinxを使用しました。このスフィンクスは、サイトのメイン検索のベースになっています。 これにより、データベースからユーザーの要求に応じて最も関連性の高いデータを取得します。 また、ユーザーが住所を見つけたい場合は、Google Geocoding APIにリクエストを送信します。



データベースおよび住所による検索結果は、ユーザーが入力した最初の3文字の後に既にタイプ別にソートされて表示され、それらをクリックすると、このオブジェクトが配置されているマップ上の場所にすばやく移動できます。
