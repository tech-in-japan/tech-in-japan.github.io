貧しい人々のためのiSCSIストレージ

良い一日、親愛なるコミュニティ！



この記事では、ディスクストレージを作成した経験を共有したいと思います。その結果、多くの実験、試行錯誤、発見、苦い失望に苦しみました。 そして最後に、いくつかの興味深い、比較的予算と高速なストレージで終わりました。



同様のタスクがある場合、または見出しだけに興味がある場合は、habrakatへようこそ。



プロローグ



そのため、最近私たちの部門は、VMware ESXi 5.1ハイパーバイザーのクラスターに大容量のストレージを提供するという課題に直面しました。 その上に、Dovecotと「クラウド」ファイルストレージ用の暗号化されたmaildirを配置することを計画しました。 予算を割り当てるための前提条件は、会社にとって重要な情報を保存する場所を提供することでした。このセクションは暗号化する必要があります。



鉄



残念なことに、あるいは幸運なことに、このような野心的なタスクのための大きな予算はありませんでした。 したがって、私たちは、マキシマリズムの真の崇拝者として、ブランドの倉庫を買う余裕がなく、割り当てられた材料リソース内で次の鉄を選択しました：



  • サーバーシャ​​ーシSupermicro CSE-836BE16-R920B

    多くの理由がありました。ユニットの数、ハードドライブのサイズ、速度、ケースまたはプラットフォームを一度に選択し、多くのオプションを検討し、インターネットを吸って、最終的にこのオプションに決めました。これがタスクに最適です。
  • マザーボードSupermicro MBD-X9DRI-FO

    主な条件は、4つのPCI-E x8ポートの存在です。
  • Intel XeonプロセッサーE5-2603

    選択は簡単でした-十分なお金でした。 さらに、最初のプロセッサーではなく、2つのプロセッサーを一度にインストールする必要があり、必要であれば、1つの占有スロットで3つのPCI-Eのみが動作するため、購入しましたが、4つ必要でした。
  • Seagate Constellation ES.3はST3000NM0033を駆動します

    SATAの方が安価であり、同じ費用でSASを使用する場合よりもはるかに多くの空き容量を受け取りました。
  • Adaptec RAIDコントローラASR-7805Q

    これはストレージシステムであるため、コントローラーでささいなことを始めませんでした。 このシリーズにはSSDキャッシングがあり、これは非常に便利です。また、キットにはすぐにBBUがあり、これも非常に便利なオプションです。
  • SSD Intel SSDSC2CW240A310

    MaxCacheが動作するためにのみ必要でした（別名SSDキャッシュ）。
  • ネットワークカードIntel X520 DA2

    ネットワークインターフェイスのボトルネックを回避するには、ESXiノードとストレージ間に10Gbリンクを提供する必要がありました。 市場の申し出を研究した後、私たちは最もエレガントなものではなく、10ギガビットのネットワークカードを使用して価格と速度が適切なオプションになった可能性があります。


これにはすべて約20万ルーブルかかりました。



実装



ターゲットを提供する、つまりストレージリソースをコンシューマに割り当てるために、iSCSIとNFSを使用することにしました。 もちろん、最も合理的で迅速な解決策は、FCoEを使用して、対応するオーバーヘッドでTCPに入らないようにすることです。これは、一般にネットワークカードで実行できますが、残念ながら、SFPスイッチはありませんFCoEサポート、500 trの費用がかかるため、購入できませんでした 上から。

インターネットを再び喫煙した後、彼らはvn2vnテクノロジーでこれから抜け出す方法を見つけましたが、ESXiは6.xバージョンでのみvn2vnで動作することを学習します。



Linuxサーバーの企業標準はCentOSですが、現在のカーネル（2.6.32-358）では暗号化が非常に遅いため、OSとしてFedoraを使用する必要がありました。 もちろん、これはRed Hatのトレーニンググラウンドですが、最新のLinuxカーネルでは、データはほとんどその場で暗号化され、残りは必要ないようです。

さらに、現在のバージョン19はRHEL 7のベースとして使用されるため、将来的にシームレスにCentOS 7に切り替えることができます。



ターゲティング



記事を膨張させず、トピックから離れないようにするために、コントローラの突合せ、OSのインストールなど、鉄のアセンブリのすべての面白くないタイプを省略します。 また、ターゲットの説明はできる限り少なくし、ESXiイニシエーターでの作業のみに制限するようにします。



ターゲットから、次のものを取得したいと考えました。

  • 適切に機能するキャッシング-ディスクはかなり低速で、2,000 iopsのみを圧縮できます。
  • ダイレクトディスクサブシステム全体で可能な限り最高の速度を読み取ります（可能な限り多くのiopsを与えます）。


会いましょう、ここにあります。



リオ
linux-iscsi.org

Linuxカーネル3.10.10では、blockioモードで300 MB / sの書き込みと600 MB / sの読み取りを示しました。 彼は、fileioとRAMディスクでも同じ数字を示しました。 グラフは、おそらくESXiイニシエーターが記録の同期を必要とするという事実のために、書き込み速度が非常にジャンプすることを示しました。 同じ理由で、レコードあたりのIOPSの数はfileioとblockioで同じでした。

メーリングリストでは、emulate_fua_writeを無効にすることをお勧めしましたが、これは変更につながりませんでした。 さらに、3.9.5カーネルでは、最良の結果を示しており、これにより、将来についても考えることができます。

説明から判断すると、LIOはさらに多くの機能を実行できますが、ほとんどの機能は商用バージョンでのみ使用できます。 私の意見では、主に情報源であるはずのサイトは広告でいっぱいであり、それが否定的な結果をもたらします。 その結果、彼らは拒否することにしました。



istgt
www.peach.ne.jp/archives/istgt

FreeBSDによって使用されます。

いくつかの例外を除いて、ターゲットは十分に機能します。

第一に、彼はブロックする方法を知らず、第二に、彼は異なるMaxRecとMaxXtranを使用できません。少なくとも私は成功しませんでした。 MaxRecの値が低い場合、順次書き込み速度は250 MB /秒を超えず、読み取りは非常に高いレベル-700 MB /秒でした。 キュー深度32のランダム4k記録から約40Kのiopsを取得しました。MaxRecの増加に伴い、書き込み速度は700 MB / sに増加し、読み取りは600 MB / sに低下します。 Iopは、書き込み用に30Kおよび20Kに低下します。

つまり、何らかの方法で設定を変更することで中間地点を見つけることができますが、どういうわけか真実ではないように見えました。



STGT
stgt.sourceforge.net

このターゲットがハイパーバイザーとのインターフェースのセットアップに問題がありました。 ESXiは絶えずLUNを混同します-相互に受け入れた、またはまったく見えなくなりました。 シリアル番号の誤ったバインドに問題がある疑いがありましたが、それらを構成に登録しても役に立ちませんでした。

速度も喜ばれません。 彼は、読み取りにも書き込みにも500 MB / sを超えることに成功しました。 読み取りのIOPSの数は20Kで、書き込みのIOPSは約15Kです。

結果として-設定の問題と速度の低下。 ごみ。

iscsitarget.sourceforge.net

ほぼ完璧に働いた。 700MB / sの読み取りと書き込み。 約30Kの読み取り、2000の書き込みのIOPS。

ESXiイニシエーターは、システムキャッシュを使用せずに、ターゲットにディスクへのデータの書き込みを強制しました。 また、メールリストで彼についてのレビューをいくらか怖がっていました-多くの人が負荷のかかった不安定な仕事を報告しました。

ESXi 5.1の準備


ハイパーバイザーで次の設定が行われます。

  • iSCSIイニシエーターの設定では、遅延ACKはすべてのターゲットに対して無効になっています。 kb.vmware.com/selfservice/microsites/search.do?language=en_US&cmd=displayKC&externalId=1002598に従って作成
  • イニシエーターパラメーターは、ターゲットパラメーターに従って変更されます。
    InitialR2T =いいえ

    ImmediateData =はい

    MaxConnections = 1

    MaxRecvDataSegmentLength = 1048576

    MaxBurstLength = 1048576

    FirstBurstLength = 65536

    DefaultTime2Wait = 0

    DefaultTime2Retain = 0

    MaxOutstandingR2T = 32

    DataPDUInOrder =いいえ

    DataSequenceInOrder =いいえ

    ErrorRecoveryLevel = 0

    HeaderDigest =なし

    DataDigest =なし

    OFMarker =いいえ

    IFMarker =いいえ

    OFMarkInt =拒否

    IFMarkInt =拒否




ネットワークアダプタの割り込みモデレーションとLROを無効にする必要があります。 次のコマンドでこれを行うことができます 



ethtool -C vmnicX rx-usecs 0 rx-frames 1 rx-usecs-irq 0 rx-framesirq 0 esxcfg-advcfg -s 0 /Net/TcpipDefLROEnabled esxcli system module parameters set -m ixgbe -p "InterruptThrottleRate=0"




これを行う理由：

www.odbms.org/download/vmw-vfabric-gemFire-best-practices-guide.pdf

www.vmware.com/files/pdf/techpaper/VMW-Tuning-Latency-Sensitive-Workloads.pdf



これらの値を再度設定しないようにするには、これらの値をこのスクリプトに追加できます。

/etc/rc.local.d/local.sh




Fedoraの準備


Fedoraの最新バージョンを最小バージョンでダウンロードしてインストールします。



システムを更新して再起動します。 



 [root@nas ~]$ yum -y update && reboot




システムはローカルネットワークでのみ動作するため、ファイアウォールとSELinuxをオフにしました。 



 [root@nas ~]$ systemctl stop firewalld.service [root@nas ~]$ systemctl disable firewalld.service [root@nas ~]$ cat /etc/sysconfig/selinux SELINUX=disabled SELINUXTYPE=targeted




ネットワークインターフェイスをセットアップし、NetworkManager.serviceサービスを無効にします。 これはBRIDGEインターフェースと互換性がなく、これはNFSに必要でした。 



 [root@nas ~]$ systemctl disable NetworkManager.service [root@nas ~]$ chkconfig network on




ネットワークカードでLROを無効にしました。 



 [root@nas ~]$ cat /etc/rc.d/rc.local #!/bin/bash ethtool -K ethX lro off




Intelの推奨に従って、次のシステムパラメータが変更されました。 



 [root@nas ~]$ cat /etc/sysctl.d/ixgbe.conf net.ipv4.tcp_sack = 0 net.ipv4.tcp_timestamps = 0 net.ipv4.tcp_rmem = 10000000 10000000 10000000 net.ipv4.tcp_wmem = 10000000 10000000 10000000 net.ipv4.tcp_mem = 10000000 10000000 10000000 net.core.rmem_max = 524287 net.core.wmem_max = 524287 net.core.rmem_default = 524287 net.core.wmem_default = 524287 net.core.optmem_max = 524287 net.core.netdev_max_backlog = 300000




ターゲット準備


SCSTを使用するには、カーネルにパッチを追加することをお勧めします。 これはオプションですが、パフォーマンスが向上します。

リポジトリの作成中、最新のカーネルバージョンは3.10.10-200でした。 この記事を読む頃には、カーネルは既に更新されているかもしれませんが、これがプロセスに大きな影響を与えるとは思いません。



変更されたカーネルを使用してrpmパッケージを作成する方法の詳細は、次のとおりです。

fedoraproject.org/wiki/Building_a_custom_kernel/en



しかし、問題を引き起こさないために、準備について詳しく説明します。



ユーザーを作成します。 

 [root@nas ~]$ useradd mockbuild




彼の環境に移りましょう： 

 [root@nas ~]$ su mockbuild [mockbuild@nas root]$ cd




アセンブリ用のパッケージをインストールし、カーネルソースを準備します。 

 [mockbuild@nas ~]$ su -c 'yum install yum-utils rpmdevtools' [mockbuild@nas ~]$ rpmdev-setuptree [mockbuild@nas ~]$ yumdownloader --source kernel [mockbuild@nas ~]$ su -c 'yum-builddep kernel-3.10.10-200.fc19.src.rpm' [mockbuild@nas ~]$ rpm -Uvh kernel-3.10.10-200.fc19.src.rpm [mockbuild@nas ~]$ cd ~/rpmbuild/SPECS [mockbuild@nas ~]$ rpmbuild -bp --target=`uname -m` kernel.spec




これで、パッチ自体が必要になります。 SVNリポジトリからSCSTをダウンロードします。 

 [mockbuild@nas ~]$ svn co https://scst.svn.sourceforge.net/svnroot/scst/trunk scst-svn




必要なパッチを〜/ rpmbuild / SOURCES /にコピーします 

 [mockbuild@nas ~]$ cp scst-svn/iscsi-scst/kernel/patches/put_page_callback-3.10.patch ~/rpmbuild/SOURCES/ [mockbuild@nas ~]$ cp scst-svn/scst/kernel/scst_exec_req_fifo-3.10.patch ~/rpmbuild/SOURCES/




カーネル構成に次の行を追加します。 

 [mockbuild@nas ~]$ vim ~/rpmbuild/SOURCES/config-generic ... CONFIG_TCP_ZERO_COPY_TRANSFER_COMPLETION_NOTIFICATION=y ...




kernel.specを編集しましょう。 

 [mockbuild@nas ~]$ cd ~/rpmbuild/SPECS [mockbuild@nas ~]$ vim kernel.spec




変更します： 

 #% define buildid .local


オン： 

 %define buildid .scst




できれば他のすべての後にパッチを追加します。 

 Patch25091: put_page_callback-3.10.patch Patch25092: scst_exec_req_fifo-3.10.patch


パッチ適用コマンドを追加します。残りのエントリの後に追加することをお勧めします。 

 ApplyPatch put_page_callback-3.10.patch ApplyPatch scst_exec_req_fifo-3.10.patch




すべてのアクションの後、含まれているファームウェアファイルでrpmカーネルパッケージのアセンブリを開始します。 

 [mockbuild@nas ~]$ rpmbuild -bb --with baseonly --with firmware --without debuginfo --target=`uname -m` kernel.spec




アセンブリが完了したら、ファームウェアカーネルとカーネルヘッダーファイルをインストールします。 

 [mockbuild@nas ~]$ cd ~/rpmbuild/RPMS/x86_64/ [mockbuild@nas ~]$ su -c 'rpm -ivh kernel-firmware-3.10.10-200.scst.fc19.x86_64.rpm kernel-3.10.10-200.scst.fc19.x86_64.rpm kernel-devel-3.10.10-200.scst.fc19.x86_64.rpm kernel-headers-3.10.10-200.scst.fc19.x86_64.rpm'




再起動します。



成功したら、ダウンロードしてSCSTソースのあるディレクトリに移動し、rootユーザーがターゲット自体を収集することを望みます。 

 [root@nas ~]$ make scst scst_install iscsi iscsi_install scstadm scstadm_install




アセンブリの後、サービスを自動実行に追加します。 

 [root@nas ~]$ systemctl enable "scst.service"




そして、/ etc / scst.confで構成を構成します。 たとえば、私の： 

 [root@nas ~]$ cat /etc/scst.conf HANDLER vdisk_fileio { DEVICE mail { filename /dev/mapper/mail nv_cache 1 } DEVICE cloud { filename /dev/sdb3 nv_cache 1 } DEVICE vmstore { filename /dev/sdb4 nv_cache 1 } } TARGET_DRIVER iscsi { enabled 1 TARGET iqn.2013-09.local.nas:raid10-ssdcache { LUN 0 mail LUN 1 cloud LUN 2 vmstore enabled 1 } }




必要に応じて、特定のアドレスからのターゲット接続を許可または拒否するファイルを作成します。 

 [root@nas ~]$ cat /etc/initiators.allow ALL 10.0.0.0/24 [root@nas ~]$ cat /etc/initiators.deny ALL ALL




構成ファイルを構成した後、SCSTを実行します。 

 [root@nas ~]$ /etc/init.d/scst start




すべてが正しく行われた場合、対応するターゲットがESXiに表示されます。
