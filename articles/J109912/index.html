クラウドの商業運用の開始

消費されたリソースのみが支払われるこの夏を約束したSelectelクラウドは、ついに準備が整いました。



実際、1か月前に開始しましたが、habraeffectを手配するのではなく、顧客を少し受け入れることを決めました。 これにより、発生する困難を慎重に研究し、非常に個別の方法で問題を解決することができました。 「ハーフローンチ」中に、1つの仮想マシンのテンプレートに1つの不快なタイプミスが見つかり、インターフェースにいくつかの小さな欠陥が見つかったため、戦略は報われました。 同時に、静かに、開始前の煩雑さなしに、仮想マシンを管理するための追加機能が実装されました。



ただし、ワークフローについては十分です。後で説明します。



Selectel Cloudは、サーバーが使用したリソースに対してのみ支払う機会です。 また、リソース自体は、高速で快適なサーバー操作に必要な量で提供されます。 自動的に。 したがって、異なる料金プランはありません。リソースの価格しかなく、最終コストはサーバーの負荷によって決まります。



ただし、アイドル状態のサーバーでもリソースを消費することは明らかです（カーネルにはメモリが必要です。オペレーティングシステムがインストールされたディスクはどこかに保存する必要があります）。 予想される質問：Debianの最もコンパクトなバージョンの月間価格。一定の稼働時間で、月あたり約45ルーブルの大きな負荷コストがありません。



そして、はい、あなたは車を止めることができます、そしてあなたが支払うのはディスクストレージだけです、それは同じDebian Miniのために月に3-5 rかかります。 賞味期限は、貸借対照表上の金額によってのみ制限されます。



価格に気を取られることはありません。価格はサイトにあります[1] 。



代わりに、最終的にどのような数字が得られるかについての観察についてお話します。



まず第一に、驚くべきことに、CPU時間は最も要求の少ないリソースです（再び、最近発表したアトムのレンタルは合理的な選択かもしれません）-総コストのわずか5％（平均消費量）仮想マシンごとに1週間あたり約6時間のマシン時間になりました-現代のXeonのパワーを過小評価しないでください。

第二に、最も高価な（および時間消費の観点から予測可能な）RAMは、すべてのコストの約40％を占めています。

リストの3番目はディスクストレージであり、これもかなり予測可能です。



私たちはすべてのリソースを考慮します-最大の質問はディスクアカウンティングです。 私の知る限り、私たちは（webDAVや他のAPIではなく）直接ディスクアクセスのためのディスク操作を行う最初の企業です。



ディスク操作-情報をディスクに読み書きするためのOSからの要求。 ディスク操作は小さくても非常に大きくても（書き込み4kまたは書き込み1MB-明らかに違いがあります）、ディスク操作のアカウンティングを2つのコンポーネントに分割し、それぞれの価格を引き下げます-操作の数は別々に考慮され、記録された操作の量は別々に考慮されます/データを読み取ります。 もちろん、OSのキャッシュにより、ディスク操作の量を大幅に削減できるため、Webページへのアクセスが少なくとも一部のディスク操作を引き起こすことは決してありません。 まあ、価格（私たちは、そのような価格でそれが完済するかどうかはまだ確信していません）-これは、ディスク要求100万あたり3ルーブル、記録ギガバイトあたり10コペックです。



一般的な概念に基づいて、トラフィックの支払いについてもインターネットを同じにしました。 はい、これらの言葉はメガバイトあたりのルーブルやインターネットの夜明けのその他の恐怖の心的外傷的な記憶を呼び起こします。 発展したインターネットと一般的なアリムの時代にトラフィックを支払ったもの。 しかし...着信トラフィックにギガバイトあたり20コペックのコストがかかる場合（発信1p）、これはまったく異なる会話ですか？ 結局のところ、ストリップを購入すると、特定の平均レベルで消費するという事実に暗黙のうちに置かれます。 あなたが天井の下でそれを消費するなら、ストリップを買うと、あなたはすでに完全に詰まったチャネルの本当の価値を見ます。 30％を選択しない場合、なぜあなたはそれを支払うのですか？ 発信トラフィックが多い場合、着信トラフィックに対して暗黙的に料金を支払うのはなぜですか？ 第二のポイント：どのチャンネルが必要ですか？ 10 Mb？ そして、それが十分でない場合、ある瞬間のピークが100を超える場合は？ 私たちはギガビットを提供します-そしてあなたが消費するそのほんの小さな部分に対してのみ支払う能力。



そして再び記憶について。 私が何ヶ月も前に書いた主な機能は、Memory on Demandテクノロジーです-オンデマンドでメモリを割り当てます。 実際、そこには魔法はありません。空きメモリの量（キャッシュは考慮されません）がほぼ一定であることを確認し、空きメモリが少ない場合はメモリをスローし、空きメモリが多い場合はメモリを選択します。 したがって、各プログラムは必要な量のメモリを受け取りますが、それ以上は受け取りません。 もちろん、このテクノロジーは、メモリ消費量が変化するアプリケーション（たとえば、肥沃なApache）にのみ必要です。 Java / Tomcatの場合、メモリはすぐにバックアップする必要があります。SunOracleに連絡する方が「一体何がいいのか」という質問があります。



この投稿は発表のお知らせです。 次の投稿では、その仕組みとその理由について説明します（短いティーザー：pv_opsではなく2.6.34-xen、CPU時間とメモリのカウント方法、debian lennyがない理由、システムのインストール方法、4.01ではなくxen 3.4、なぜxendではなくXCP）。



特に、お客様を当社のネットワークに誘い込むために、クラウドでの作業を開始するための最低額を10ルーブルに設定しました（ただし、支払いシステムでは、残高を補充するための最低額を100ルーブルに制限していますが、これは私たちではありません）。 10ルーブルは、システムで約3日間の騒ぎに十分です。 さらに、当社の支払いモデルは、クライアントが当社に留まっている状況には対応していません。 資金が終了した後、マシンからのディスクを7日間保管します。クライアントが続行することに決めた場合、ディスクの保管料を支払います。 残った場合、何も行われず、試用版はありません。ディスクを取り外して、ストレージを損失して消去します。 言い換えれば、クラウドテストでは何も義務付けられていません。
