WargamingのErlang

Erlangはあまり知られていませんが、かなり興味深い技術であり、そのニッチに非常に強いです。 過去数年にわたって、ErlangはWeb開発でますます使用されてきました。



アーランは物議をかもしている-頑固な支持者と厳しい批評家の両方がいる。 ただし、開発で使用する人はほとんどいません。 そのため、自分の使用経験に基づいてこの言語を称賛したりscったりする専門家はほとんどいません。



Wargaming Erlangは3年以上使用されています。 約12人の開発者と、重い負荷の下で動作するいくつかのプロジェクトがあります。 この記事では、Erlangの経験とビジョンを共有します。







それがすべて始まった方法



Wargamingは、さまざまなサービスがHTTPプロトコルとAMQPの両方で相互作用するマイクロサービスアーキテクチャを使用します。 そのため、 RabbitMQサーバーが積極的に使用されています。 RabbitMQですべてが始まったと言えます。 プログラマーはまだErlangについて知りませんでしたが、アーキテクトとDevOpsはすでに経験を積んでおり、RabbitMQがどこかに落ちたのか、そのログとクラッシュダンプから何を抽出できるのかを突き止めました。 そして、それが落ちないように何をする必要があるか:)

当時のアーキテクトは長い間Erlangに同情していたが、これには十分な理由があるまで開発言語として実装しなかったと言わざるを得ない。



ゲームロジックがPythonで記述されているBig Worldエンジンでゲームサーバーを実行することは秘密ではありません（Pythonは現在も会社の主要な開発言語です）。 当初、ゲーム内チャットはこれらのサーバーを介して機能していました。 しかし、ある時点でサーバーリソース（既に不足していた）をチャットに費やすことは不可能でした。



Chat World of Tanksはゲームと密接に関連しており、多くの特定の機能をサポートしています。 したがって、彼は彼自身の決定を必要としていました。 もちろん、1つのサーバーを既存の負荷および計画されている将来の負荷に合わせてスケーリングできないため、クラスタ化する必要がありました。 Erlangがそのようなプロジェクトに選ばれたのは論理的です。 その後、最初のErlang開発者が会社に来ました。



どうだった



ゲームサーバーに加えて、多くのWebサービス（プレーヤーの登録と承認、支払いなど）があります。 当初は、すべてPython / Djangoで記述されていましたが、Web上のプレイヤーの数が急速に増加した結果、問題が発生しました。

それからTwistedを使い始めました。 しかし、開発者とアーキテクトの両方が複雑すぎると考えており、その開発は高すぎました。 アーキテクトは代わりにErlangを使用することを提案しましたが、管理者は十分なErlang開発者をどこに置くべきかについて合理的な疑念を抱いていました。



一方、Erlangで書かれたプロトタイプはベンチマークで良好に機能しましたが、Twistedは傷つき続けました。 そして、アーキテクトはプロトタイプを超えて、Erlangで1つの小さなWebサービスを作成して開始しました。 ある建築家の努力によって数週間で書かれ、成功したことが判明しました。 次に、2番目のプロジェクト、3番目のプロジェクトがありました。 同社はErlang開発者の検索を開始し、Python開発者は新しい言語を簡単に習得しました（Twistedを整理するよりも簡単な言語もありました）。



今日、同社にはErlang開発者と呼ばれることができる12人の開発者がいます。彼らはErlangでほとんどのコード（またはすべて）を書きます。 それらのいくつかは、数年前からErlangで書いていますが、他のものはPythonから切り替えたばかりです。



私たちはウェブ上で多くの仕事をしています。 Wargamingがリリースするゲームの数は増え続けています。 特殊な（特定のゲーム用の）サービスからユニバーサルサービスに移行しています。 新しいプロジェクトも開始されており、その多くはErlangで書かれています。 欠員では、Python開発者（当社の主要なプログラミング言語）を頻繁に探しますが、Erlangの知識があればプラスになることを述べています:)



その結果、強力なErlang開発者のチームを編成することができました。 その中には、たとえば、セルゲイプロホロフがあり、そのパッチはアーラン/ OTPチームに受け入れられました。 したがって、Erlang仮想マシンにはSergeyコードがいくつかあります。



開発者の意見



社内でさまざまな経験を持つ多くのErlang開発者がいるため、Erlangについて幅広い意見を集めることができました。 経験豊富な開発者と初心者の開発者との間の言語認識の違いに注目することは興味深いです。



経験者は一般的にアーランを前向きに考えています。 そして、これは驚くべきことではありません。さもなければ、彼らはそれについて何年も書かなかったでしょうし、経験することもなかっただろう:) 彼らはより多くの批判と不満を持っています。 Pythonと比較して、Erlangはライブラリの表現力、インフラストラクチャ、および豊富さの点で著しく劣っているため、これも驚くことではありません。



すべての開発者が同意する一般的なポイントから始めましょう。



  • Erlangは、1秒あたりのリクエスト数で測定すると、優れたWebパフォーマンスを示しています。 Pythonで同様のパフォーマンスを達成するには、より長く、より難しく、より高価です。
  • Erlangは高度に専門化された言語です。 問題を解決しようとしないでください。




長所Erlang、一般的な意見：



  • 使いやすい優れたマルチスレッドモデル。 アクターモデルと非同期性により、効率的なマルチスレッドコードを簡単に記述できます。
  • ロードされたプロジェクト、標準およびテンプレートで20年以上使用され、OTPフレームワークに組み込まれています。 それらを使用すると、複雑ではあるが安定した信頼性の高いシステムを簡単に構築できます。
  • Erlangは利用可能なすべてのカーネルを使用できます。 コアごとに個別のサービスインスタンスを実行する必要はありません。
  • 低エントリーしきい値。 この言語は簡単で、すぐに習得できます（ただし、以下に説明する落とし穴がいくつかあります）。




初心者開発者に見られる長所：



  • 閉じたエコシステムであるこのプロジェクトには、外の世界からの何かは必要ありません。
  • 外部のメモリ内ストレージ（Redis、memcached）は必要ありません。サービスのRAMにキャッシュを直接保持できます。
  • 外部スーパーバイザーの下で、独自の構成で各コアの複数のサービスインスタンスを実行する必要はありません。 1つのサービスですべてのコアを使用でき、独自の柔軟なスーパーバイザーシステムがあります。
  • 少ないサービスインスタンス— HTTPまたはAMQPを介したインスタンス間の通信が少なくなります。 同じノード内のプロセス間で通信する方がはるかに便利です。
  • 分散システムをより簡単に開発できます。 Erlangには、クラスターを構築し、ノード間の通信を行うための独自のツールがあります。




経験豊富な開発者に見られる長所



  • 機能的パラダイム、不変のデータ構造。
  • サービスをリアルタイムで監視し、作業ノードで直接問題を検索して修正すると便利です。
  • サービスを再起動せずに構成を変更できます。




短所、一般的な意見



  • ライブラリはほとんどなく、優れたライブラリはさらに少ない。 多くのことを自分で書くか修正する必要があります。
  • 限られた言語構文。 時々、より便利な構文構造が欠けています。 たとえば、let構造はスコープを導入するのに適しています。 数値の接尾辞を持つ変数を作成する必要があることを示します：Var1、Var2、Var3。
  • 言語は保守的で、非常にゆっくりと発展します。 たとえば、マップタイプは数年間実装されています（完全には完成していません）。
  • Webのインフラストラクチャはゆっくりと開発されており、Webフレームワークはありません。
  • この言語は、Python / DjangoやRuby / Rails開発者が使い慣れているものよりも低いレベルのアプリケーションを対象としています。




初心者開発者に見られる短所



  • 文字列は恐ろしく実装されています;文字列を操作するためのツールはほとんどありません。 経験豊富な開発者は一般に文字列を使用しないため、このような問題を忘れています:)
  • 弱い標準ライブラリ、いくつかの機能。 これは、Pythonの背景に対して特に顕著です。 標準モジュールの不整合（関数名、引数の順序、戻り値）。
  • 機能的パラダイム、不変のデータ構造。 経験豊富な開発者にとってはプラス、パラダイムに慣れていない初心者にとってはマイナスです。 これには別の調査が必要です。
  • エラーメッセージを読むのは難しいです。




経験豊富な開発者が見る短所



  • 依存関係を管理し、パッケージを構築するための優れたツールはありません。 Rebar3とhex-正しい方向に移動しますが、まだ未加工であり、広く受け入れられていません。
  • リリースは複雑で、間違ったタスクを解決します。 たとえば、ホットアップデートの問題を解決しますが、これはWeb開発者には必要ありません。 1つのノードを停止することを簡単に心配できるように、サービスを構築する方が適切です。
  • 構成構文は、DevOpsには便利ではありません。 正直なところ、開発者自身にとっては便利ではありません。 設定は、iniなどの単純な形式で記述する方が適切です。




もっと個人的な意見



  • Python開発者は、まったく異なる言語であるにもかかわらず、TwistedよりもErlangを学ぶ方が簡単です。 Erlangプロジェクトは、Twistedプロジェクトよりも単純で、開発が速く、信頼性が高くなります。
  • Erlang開発者コミュニティは小さく、あまり活発ではありません。 各チームは独自のプロジェクトに住んでいます。 他の企業の同僚と経験や知識を共有している人はほとんどいません。 各開発者がプロ​​ジェクトでのみ使用される独自のライブラリセットを作成する一般的な状況。
  • CowboyやRiakのような大規模なオープンソースプロジェクトがありますが、活発なコミュニティが開発するプロジェクトはありません。
  • Erlang仮想マシンは優れていますが、言語自体は保守的すぎて、開発を求めていません。 そのため、Elixirなど、この仮想マシン用の新しい言語のフレームワークで活発な開発が行われています。
  • ErlangとPythonは互いに補完し合い、弱点を補います。 それらを1つのプロジェクトに正しく結合します。 データベースでのビジネスロジック（ORM）の作業はPythonで、マルチスレッドデータ処理、サービス間の通信、バックグラウンドプロセス-Erlangで行うのが最適です。
  • Erlangにはひどい構文があり、苦しんでいます:)




一番下の行は何ですか？



一般的に、開発者はErlangを非難して喜んでおり、プラスよりもマイナスを見つけました。 これは驚くことではありません。なぜなら、テクノロジーの利点にすぐに慣れ、それを当たり前のことと考え、欠点がいらいらして注目を集め続けるからです。 ただし、当社では、開発者自身がプロジェクトの言語を選択します（もちろん、設計者が承認したリストから:)）、このソリューションは上に課されません。 そして、彼らはErlangで書いているので、それほど悪くはありません。 今日、私たちのウェブはPython 2.7、Python 3.5、Erlangを使用しています。 アーキテクトは、Go言語、JVM言語の1つ、他のものなど、他のオプションを試しても構いませんが、これまでのところ、希望する人はElixirでのみ見つかりました。



潜入エントリのしきい値



確かに、多くの人はErlangのエントリしきい値が低いと聞いており、言語を2週間知った後、開発者は本番用のコードを書くことができます。 それは本当ですが、すべてではありません。 言語は本当にシンプルで習得しやすいですが、落とし穴があります。



まず、未経験の開発者の多くがつまずく「子供の問題」、経験豊富な開発者はそれらを解決する方法を学んだが、忘れる。 典型的な例は、gen_serverでデッドロックを取得することです。 これはほとんどすべての人に起こります。 クラブに入るための一種の儀式:)同じサンドボックスから、文字列を使用する必要はなく、バイナリのみを使用することを理解する問題。 プロジェクトでこのルールを遵守するのは簡単ですが、残念ながら、サードパーティのライブラリでは何でもかまいません。このために準備する必要があります。



第二に、幼稚な問題はまったくありません。 そして最初の問題は情報不足です。 Erlangはほんの数冊の良い本を書いています。 それらのほとんどは、言語をそのようなものとして記述し、適用された問題を解決するための使用法ではありません。 多くの重要な問題は、書籍でも文書でもまったく説明されていません。 プロジェクトのビルドに使用するツールは何ですか？ 動作中のノードを監視する方法は？ メールボックスのオーバーフローを検出する方法は？ 他のメモリリークオプションはどうですか？ システムのスケーリングを妨げるボトルネックを見つける方法 使用するライブラリ 一方で、それらは少数です。 一方、人気のあるライブラリには多くの分岐点があります。 そして、それらがどのようなフォークであるか、どのように異なるのか、どのようにサポートされるのかは明確ではありません。 MySQLで動作するドライバーを探している人は、この問題をよく理解しています:)そして、分散システムはまだ取り上げていません。 このすべてについて学ぶことができるのは、別の経験豊富な開発者だけです。 したがって、常にチームに所属していることが重要です。



エリクサーについて



Erlangバイトコードにコンパイルされ、Erlang仮想マシンで実行されるいくつかの言語があります。 最も人気のあるのはエリクサーです。



数年前、Ruby / Railsの開発者はErlangに注目しました。 彼らはRails固有のパフォーマンスの問題に対する解決策を探しました。 パフォーマンスの問題は解決されましたが、Erlangの他のすべてが好きではありませんでした。 そして、当然のことながら、豊富なインフラストラクチャを備えた強力な高レベル言語から、彼らは厳しい低レベルの世界にいることに気付きました。

Ruby / Rails開発者は、そのような生活をしたくないことをすぐに認識し、より快適な環境を自分たちで構築し始めました。 彼らの努力の結果、Elixir言語とPhoenix Webフレームワークができました。それぞれに活発なコミュニティがあります。 3年で、彼らはErlangが10年でしたよりも多くのWeb開発をしました。



Elixirについての意見とErlangについての意見は矛盾しています。 読みやすく、簡潔で強力だと考える人もいます。 他の人は読みにくく、想像力に欠けます。 保守的なErlang開発者はElixirをかっこよくしています。 そして、RubyやPythonからErlangに来た人たちは好奇心でElixirを受け入れます。



言語の長所：



  • 豊富な標準ライブラリ。
  • 高レベルWebフレームワークPhoenix ;

  • 良いドキュメント

短所：



  • 言語は若く、業界での経験はありません。

  • 低バス係数。


コミュニティはインフラストラクチャ、ライブラリ、フレームワークで忙しく、言語の中核は小さなチームによって開発されています。 このチームが資金を失い、誰もそれに代わらない場合、言語の開発と維持が停止するリスクがあります。



Wargaming Webでは、将来のサービスの1つのプロトタイプがElixirで作成されています。 彼がどのように自分を見せているか見てみましょう。
