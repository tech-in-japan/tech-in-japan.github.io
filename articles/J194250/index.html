Impress：Node.js用の多目的アプリケーションサーバー

注目に値する成功にもかかわらず、 Node.jsは依然として特殊な技術であり、主に別の技術スタックで記述されたシステムのボトルネックによってカバーされています。 この状況の理由は、ノード自体に、他の言語で使用され、アプリケーションソフトウェアの高速開発を提供するライブラリが多くないという事実にあります。 たとえば、コード内の異なるURLのハンドラーを分離し、静的ファイルを与え、セッションを整理し、複数のスレッドを実行し、データベースにアクセスし、メモリにデータをキャッシュし、ユーザー権限を制限し、ログを取得してローテーションし、ネットワークAPIを作成し、テンプレートをレンダリングします、URL書き換えの構成、サーバーからクライアントへのイベントの迅速な配信を提供します。これらすべて、およびその他の多くのタスクでは、個別のライブラリー（モジュール）が使用されます。 さまざまなモジュールがさまざまな開発者によって作成されており、参加するのは難しく、競合します。 一般に、ほぼすべてのWebアプリケーションに必要なこの一連の必要な機能をすべて1つのアプリケーションサーバーに結合し、コードの一貫性を高め、アプリケーションサーバーコアを個別のライブラリから収集されたソリューションよりもモノリシックで一貫性のあるものにすることを決定しました。 Impressプロジェクトはすでにプロトタイプとして発表されており、現在、迅速なアプリケーション開発に必要なすべての武器を提供しています。これは、多数のライブプロジェクトでテストされています。 Impressは、印象派が表現主義と異なるのと同じように、広く普及している他のプラットフォームとは大きく異なります。つまり、突然の感情の爆発とは対照的に、全体的でよく考え抜かれた美的印象を生み出します。 しかし、私たちは、他の人のコードを批判することなく、Impressの設計機能を実証するために進みます。

  • 外部ライブラリのサイズ （オプションのロードを含む）、/ node_modulesのすべては56MBです

  • プロジェクト期間：3.5か月（プロトタイプとしての別の1か月前、npmおよびgithubでの公開前）。
  • 開発強度 ：この期間中に47バージョンがリリースされました。 何かが2〜3日ごとに終了します。
  • 依存関係 （現在）： async 、 cluster 、 colors 、 mkdirp 、 mongodb 、 mysql 、 memcached 、 nodemailer 、 geoip-lite 、 uglify-js 、 multiparty 、 iconv-lite


機能と用途



  • 複数ページのWebアプリケーションを作成します 。つまり、サーバーテンプレートエンジン（カーネルに組み込まれています）とページの再読み込みを使用します。
  • クライアント画面を動的に再構築するために、HTMLまたはJSONフラグメントを転送して、AJAXを使用してブラウザーとサーバーAPIの間でデータ交換を行う1ページWebアプリケーションを作成します。
  • 単一ページと複数ページのアプローチが混在するハイブリッドソリューションの作成。
  • ブラウザー、iOS、Androidなどのモバイルアプリケーションを含む、サーバー間およびクライアントとサーバー間の対話のためのネットワークAPIの開発
  • cache.manifest 、IndexedDBまたはWebSQLのローカルストレージ、およびオンラインとオフラインの両方でオフラインで機能するオフラインHTML5アプリケーション。 もちろん、これはImpressのタスクではありませんが、そのような非典型的な場合でも使用する経験があります。
  • 単一のアプリケーションサーバー（マスクを含む）で複数のドメインにサービスを提供します 。 1つのアプリケーションを構成する仮想ホストをセットアップするメカニズム、またはそれらで（1つのポートと異なるポートの両方で）異なるアプリケーションを起動するメカニズム。
  • URL書き換えサポートを使用して他のサーバーおよびポート（リバースプロキシ）に呼び出しを転送し 、config.jsでJSON形式のURLルーティングを設定し、パターンと正規表現を使用します。 転送は、アプリケーションサーバーの一部のコールの処理と組み合わせることができます。 転送を使用すると、複数の言語、サーバー、テクノロジースタックから1つのアプリケーションを構築できます。
  • RAMでのキャッシュ、静的ブラウザーjsファイルの縮小、圧縮可能な形式のgzip圧縮、および「if-modified-since」ヘッダーの受信時のHTTP 304（Not Modified）の正しい戻りによる静的 出力 。
  • 組み込みの認証システムと、MongoDBにユーザーを保存するためのプロバイダーがあります。
  • セッションをRAMに保存し、MongoDBに保存し、リロード中にセッションを復元する、組み込みのセッションシステムがあります 。 認証されていないセッション、つまり 登録せずにユーザーの状態を識別して保存します。
  • プロセス間でタスクを分散するためのいくつかの戦略をサポートし、IPCを介したプロセス間の相互作用の実装により、複数のプロセスで動作する機能（将来、これらの目的でZeroMQを使用し、複数のサーバーに透過的にスケーリングする予定です）。 異なるサーバープロセスに接続されているクライアント間の対話を確立する場合は、メッセージングが使用されます。
  • セッションをプロセスに接着するための2つの方法の実装： 「IPスティッキー」 （IPによる接着）または「クッキースティッキー」 （外部バランサーおよびポートによる多重化と組み合わせて使用​​されるCookieによる接着）。 これにより、同じIPからのすべての接続または（認証後の）要求の繰り返し中に同じCookieを使用して、セッション（状態）を格納する同じプロセスに再度ルーティングできます。
  • 毎日新しいファイルを作成し、古いファイルを削除して、要求のログを記録します（保存された履歴の制限が設定されます）。
  • Impressコアに既に組み込まれているCMSの作業は完了間近であり、すぐにページを編集するための管理インターフェイスを受け取ります。


さて、これらの機能については詳しく説明しません。 既にそれらについて書いているので、例を挙げて見たほうが良いでしょう：

  • サーバー側のJavaScriptおよびRAMテンプレートのキャッシュ。

API（RPC）：STATEfulおよびSTATEless



Impressが開発された主なタスクの1つは、STATEless原則（つまりRESTサーバー）とより興味深いSTATEful原則の両方でアプリケーションサーバーを作成することです。 RESTは、サーバーとクライアントのどちらでもない要求/応答のペアの間で、オブジェクトの状態が保存されることを思い出してください。 クライアント/サーバーアプリケーションのベースとなるRPCとは対照的に、クライアントでモデルを作成し、サーバーでモデルを作成し、それらのインターフェイスをネットワーク経由で接続し、これらのモデル間でイベントと呼び出しをブロードキャストするのが一般的です。 ここでは、ノードを使用して、ワイヤの両端にモデルを展開し、AJAX / JSONを介して呼び出しを同期できます。これは、もちろんアプリケーションアプリケーションにとってより便利です。 RESTは石器時代の重いWebサーバー（ApacheやIISなど）のノードに到達し、そのたびに（それらに関連して）外部アプリケーションを起動し、CGIを介してHTTPプロトコル要求を渡します。 このようなアプリケーションは新しいプロセスを生成します;作業環境を初期化する必要があります。 データベースとの接続の確立、すべてのデータの展開、ファイルシステムからの読み取り（必要な場合）など。 そしてこれらすべては、数ミリミリ秒で作業を完了してメモリを解放し、ベースから切断するためだけです。 ウェブの前に、STATEful APIをRPCとして採用した言語（COM、DCOM、Corbaなど）で書いていましたが、私にとってRESTの概念は常に欠けていました。 そして最後に、ノードの礼拝に切り替えた後、私は幸せでした。 ここでも、メモリ内のデータを拡張でき、リクエストからリクエストへは移動せず、RAMに重いセッションを保存でき、プロセスが完了して再起動したときにそれらをシリアライズ/デシリアライズしません。 そして、RESTはビューステートやステートサーバーのような松葉杖を備えた過去のものであるというビジョンを持っていました。 STATEful APIは、ノードを知っているすべての生き物に全能者によって与えられた最大の祝福であることに気付きました。



新しいAPI URLハンドラーを作成するには、次のものが必要です。

1.フォルダー/api/myAPI/getSomething.json/を作成します

2. post.jsファイルをそこに入れて書き込みます。 

 module.exports = function(req, res, callback) { db.impress.collectionName.find({ fieldName: req.post.fieldValue }).toArray(function(err, nodes) { res.context.data = nodes; callback(); }); }


3. / api / myAPIディレクトリで、access.jsファイルを作成して書き込みます。 

 module.exports = { guests: false, logged: true, http: true, https: true }


httpsの場合は、config.jsで設定を設定し、access.jsでこのフォルダーのhttpを禁止するだけです。 さらに、メレンゲサーバーの再起動用のハンドラーを作成し、別のフォルダーを作成して、そこにファイルにコードを書き込むことができます。 メモリ内のコードに初めてアクセスしたとき、ネジのファイルが変更されると、新しいコードがメモリにロードされ、そこに座って呼び出しを待ちます。
