John Rezigの自己呼び出しコンストラクターと、この決定が根付かなかった理由に関する考察

異常に便利なjQueryライブラリーの著名な作成者であるJohn Rezigのブログからのブログ投稿「 Simple」クラス「Instantiation 」で、4年以上も精神的に戻る時が来ました。 そして戻ってきます。



ただし、Habrahabr の検索結果には「Resig」という単語がまったく表示されないため、過去4年間にわたってこの便利なブログ投稿を翻訳する（または少なくとも再語する）ことを誰も気にせずに考えなければなりません。 、Rezigのブログ投稿を私が自分の主な意図を果たす前に再語すること：Rezigによって提案された問題を解決するためにRezigによって提案された方法が広まっていない理由を考え出すこと。 そして、私は言い直します。 （読み直し自体は、読者に何も追加しなかったとしても、読者にとってはすでに有用です。追加します。）





2007年12月6日に、Rezigはjavascriptで「new」操作を使用してオブジェクトを作成した場合に何が起こるかを調べました（クラスを持つ言語では、「クラスインスタンス」と言います）。 



function User(first, last){ this.name = first + " " + last; } var user = new User("John", "Resig");


Rezigは、javascriptの初心者にとって、関数コード内の「this」の出現がオブジェクトコンストラクターを持っていることを示していることは明らかではないことを正しく指摘しました。 （自分で括弧を付けます：関数がライブラリの腸内にある場合、この状況も文書化する必要があります-そうでない場合、ライブラリユーザーは初心者とそれほど変わりません：誰もが関数の本体でソースコードを読むわけではありません。多くの場合、縮小された読み取り不能な形式で使用されます。）



したがって、Rezigは、遅かれ早かれ誰かが 「 new 」なしで「 User（） 」を呼び出そうとし、それによって頭に2つの不快な問題が一度に生じると推論しました。 まず、変数「 user 」は未定義のままです。関数「 User（） 」はコンストラクターとして意図されていますが、値を返しません。 第二に、さらに悪いことに、そのような（誤って呼び出された）コンストラクター内からこれにアクセスしようとすると、必然的にグローバル名前空間が詰まります。これは、不吉でとらえどころのない結果に満ちています。 John Resigは例を使用して両方の問題を示しました。 



 var name = "Resig"; var user = User("John", name); //   «user»   //  :  «name»  é  «Resig»! if ( name == "John Resig" ) { // !… }


それでも、Resigはさらに指摘し、コンストラクターの呼び出しは便利です。 プロトタイプの継承（プロトタイプからオブジェクトのプロパティを取得する）、つまり実際のコンストラクターを呼び出すことは、単純な関数を呼び出してオブジェクトと同じプロパティを「構築」するよりもはるかに高速に動作するという利点があります。 



 //    : function User(){} User.prototype = { /* … … */ }; //     : function User(){ return { /* … … */ }; }


Rezigは、一方ではコンストラクターとして機能し（クイックプロトタイプの継承を提供する）、他方では" new "なしで呼び出され、その場合はリゾートとして機能するたびに関数を作成するのが良いと、ここから自然な結論を出しました。コンストラクタとして自分自身に。 jQueryネイティブライブラリの独自の関数$（）の例を使用して、Rezigは合理的に示しています。まあ、ライブラリユーザーが「 $（「div」）の代わりに「 new $（「div」） 」と書く必要があると便利でしょうか？ もちろん違います。



幸いなことに、Resigは続けました。これらすべての問題は、関数本体の条件付き表記を使用した簡単な解決策に役立ちます。 



 function User(first, last){ if ( this instanceof User ) { //    : this.name = first + " " + last; } else { //     : return new User(first, last); } }


ここでのinstanceof演算子は、 新しい演算子が関数呼び出しに関与していたかどうかを検出するための最も重要なツールです。これは簡単な例で簡単に表示できます。 



 function test(){ alert( this instanceof test ); } test(); //   alert( false ); new test(); //   alert( true );


このソリューションを見つけて、それが機能していることを確認してから、Rezig氏は次のように述べました。次に、このソリューションを「クラス」のコンストラクターを作成する一般的な方法でラップします。 この目的のために、John Rezigは次の無料のコードをレイアウトしました。 



 // makeClass - By John Resig (MIT Licensed) function makeClass(){ return function(args){ if ( this instanceof arguments.callee ) { if ( typeof this.init == "function" ) this.init.apply( this, args.callee ? args : arguments ); } else return new arguments.callee( arguments ); }; }


このコードを使用する前の例は、前のコンストラクターの本体がプロトタイプの「 init 」メソッドの本体になるように書き直す必要があります。 



 var User = makeClass(); User.prototype.init = function(first, last){ this.name = first + " " + last; }; var user = User("John", "Resig"); user.name //  «John Resig»


「 makeClass 」John Rezigの作業の論理も十分に詳細に説明されています。 「 makeClass（） 」関数はコンストラクターではなく、コンストラクターを作成します-このコンストラクターは、 「 makeClass 」 から返される匿名関数（args）関数です。 「クラス」の名前（結果としてこの関数に与えられる名前）はまだ知られていないため、実行時にサービスjavascriptプロパティ「 arguments.callee 」に頼り、そこから名前を取得します。 別のトリックは、この匿名関数が「 new 」なしで呼び出された場合、コンストラクターの役割で自分自身を呼び出すときにその引数（「 arguments 」）がその内部に再渡されることです（「 return new arguments.callee（arguments） 」） -そして、この引数のセットがargsパラメーターになり、 initメソッドに渡されます 。





John Rezigの思慮深いブログの書き直しは終わりました。 今、私は彼自身があまりにも賢いように思われる場所について最終的に話すことができます。



彼のmakeClassアイデアの不快な要素は、 arguments.calleeプロパティの使用です。 このプロパティは、ブラウザのパフォーマンスの加速に関して問題があると考えられているため（ 何らかの理由で現代のインタプリタの最適化では対応できない）、言語の新しいバージョン（ECMAScript 5）でニュアンスの1つである「厳格モード」も導入されましたこれは、 「 arguments.callee 」の完全な拒否です。 （2009年5月、ジョン・レジグ自身がこれについて言及し 、 Habrahabrに移されました 。）



さまざまなjavascriptやライブラリの作成者のコミュニティにおける「 arguments.callee 」に対するこの敵意は、時間の経過とともに、Rezigovの自己呼び出しコンストラクター自体のアイデアに一部シフトしているように思われます。 " And" new $（ "div"） "は、この考えを支持する強力で説得力のある議論のようです。



自己呼び出しコンストラクターに対する敵意のもう1つの理由は、 明らか に、「 new 」演算子がJavaScript言語の重要な部分であるという考えです 。この演算子の無知は非常に恥ずかしいため、この演算子の不在によって引き起こされるエラーを防ぐ必要はありません。 （コンピューター科学者の特に便利なツールについては、常に自虐的なものがあります。この感情は、助けを必要とする新参者に燃えるような嫌悪感を引き起こす場合があります。「いいえ、それはやりすぎ、簡単すぎます。私はセックスをしました。 ）



これは何度も見ました。



私は、この（2011年）5月にazproductionによってコンパイルされたJavaScript FAQで 、次のように言われたことを覚えています 。



-新しいものを通してオブジェクトを作成することは、より良く、より身近で、イデオロギー的です。 デザイナーは大文字にする必要があります。



-私は規約に基づいていることを好み、コンストラクター内でこれをチェックしません-私は新しいコンストラクターを呼び出さず、したがってグローバルに飛んでいます-これは「愚か者自身」を意味します。 そして、決して私はnewでエラーを奨励しません-一部の人々はこれがグローバルなものであるかどうかを確認し、ユーザーがnewなしでコンストラクタを呼び出し、コンストラクタ内にオブジェクトを作成して返します-これはエラーへの奨励であり、イデオロギー的に間違ったアプローチです。



（引用の終わり。）



また、地図を表示する美しいリーフレットライブラリの作成者であるウラジミールアガフォンキンの事例を覚えています。 この（2011年）8月に、彼はGithubでの合併のリクエストを受け取りました。著者は、各デザイナーの冒頭でこのコードのようなものを置くことを提案しました。 



 if ( !(this instanceof arguments.callee) ){ return new arguments.callee(arguments); }


アガフォンキンはこれに答えた：



-初心者のJS作成者が間違いを犯さないようにすることは非常に便利ですが、ユーザーに間違っていることを伝えるのではなく、誤った構文を許可するような鈍い言語のアイデアは好きではありません。



-「new」がなくてもオブジェクトのインスタンスを作成するのではなく、「 new Error」（「クラスコンストラクターの前に新しいキーワードを置くのを忘れた」）のようなことをした方が良いように思えます。



-そして、もう1つあります。arguments.calleeは現在悪意があると見なされているため、クラス名を明示的に記述する方が安全です。



（引用の終わり。）



リクエストの作成者は、行ってarguments.calleeについて読み、リクエストを取り消しました。 arguments.calleeの欠点とnewの尊重により、再び自己呼び出しコンストラクターの実装が妨げられたことがわかりました。



どのリーフレットユーザーが少なくとも「 クイックスタートガイド 」を読むかは、このライブラリで定義されているグローバルオブジェクトが（明らかに、簡潔にするために） 単に 「 リーフレット 」ではなく 単に「 L 」と呼ばれることに気付くでしょう。 6文字が保存されます。 ただし、コンストラクターの各呼び出しの前に「 new 」とスペースを書き込まないように、さらに4文字を保存することができます。



John Rezigがarguments.calleeを完全に控え、自己呼び出しコンストラクターレコードの単なる良い例（テンプレート、パターン）に制限されていれば、前向きに考えていたと思うことがあります。 



 function User(first, last){ if ( this instanceof User ) { //    : this.name = first + " " + last; } else { //     : return new User(first, last); } }


ただし、もちろん、関数全体に余分なif- wrapperを作成しないために、この例を単純化する必要があります。 



 function User(first, last){ if (!(this instanceof User)) return new User(first, last); //         this.name = first + " " + last; // …      … }


そして、 azproductionによってコンパイルされたJavaScript FAQに敬意を表する必要があります。この簡略化もここにあります。 そこではお勧めできません。



このような良い例に従うのは簡単で楽しいです。 また、コンストラクターコンストラクターよりも理解しやすくなっています-javascriptオプティマイザーをより明確に含みます。
