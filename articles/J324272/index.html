Visual Studio 2015でWinFormsアプリケーションを開発するためのSQLite DBMSでのEntity Framework Coreコードファーストの使用

一見、下の資料はもう1つの「ハローワールド」、「Captain Obviousnessのチュートリアル」のように見えるかもしれませんが、これらはすでにWebに豊富にありますが、この印象は誤解を招きます。 実際、同じ結果をゼロから達成するために、以前にEntity Framework Core（従来のEntity Frameworkと混同しないで）を使用せず、Webおよびドキュメントの広大な領域に既に豊富なEF Coreチュートリアルおよびドキュメントのみで武装したWinFormsプログラマーは、より長い時間がかかる可能性があります彼が仕事に取りかかる前に、彼は素朴に予想していました。 それで私に起こった。 したがって、私の経験を共有したいと思います。



主な問題は、圧倒的多数のEF CoreマテリアルがVisual Studioの使用を暗示しておらず（代わりに、軽量のクロスプラットフォーム.NET Coreチューニングを使用している）、デスクトップでEF Coreを使用したいことを考慮していないことです。 Windows Formsフレームワークに基づくアプリケーション（代わりに、通常はASP.NETコアが代わりに想定されます）。 ただし、デスクトップアプリケーションの形式のソリューションが望ましい（または必要な）場合があり、Entity Framework CoreにはEntity Framework 6と比較していくつかの利点（もちろん、欠点）があるという問題と状況がまだあります。 さらに、Webで利用可能な例では、原則として、データベースへのレコードの追加など、最も基本的なアクションのみが考慮され、その後のデータ抽出の問題さえ考慮されないことが多く、その中にはすでに明らかな点はありません。 また、EF Coreは非常に活発に開発されており、次のような多くの命令の関連性が失われています。 この理由で。



VisualStudo 2015でEF Coreを使用して最小限のWinFormsデモアプリケーションを開発するような一見基本的なタスクを解決しようとすると、例外、予期しない動作、チュートリアルで自明なことを行う方法の理解不足などの一連の問題に遭遇しました。 。 その結果、Googleが質問や実験を行うことで、以下で説明するいくつかの手順をほとんど盲目的に感じる必要がありました。



プラットフォームとして、.NET Framework 4.6.1を選択しました（何らかの理由でバージョン4.6.2はVisual Studioで利用可能なリストに表示されませんが、4.6.1を選択するとすべてが機能しますが、おそらく以前のバージョンで動作しますが、開発環境として-Visual Studio Community Edition 2015 Update 3、DBMSとして-SQLite、OSとして-Windows 7 64ビットとして、私はチェックしませんでした）。



以下は、デモアプリケーションが適切に動作し、エラーメッセージが受信されなかったステップのリストです。
