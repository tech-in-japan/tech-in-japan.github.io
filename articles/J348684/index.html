BLE400ボードとnRF51822の開発

nRF51822マイクロコントローラーでの開発には、Nordic Semiconductorのいくつかのキットがありますが、それらはすべて非常に高価ですが、プログラマーと大騒ぎせずに便利に機能することができます。 同時に、中国のメーカーは、ばかげたお金でコントローラーのデバッグを容易にするボードを見つけることができます。 この記事では、BLE400という名前で販売されているWaveshareボードのクローンを使用する方法について説明します。





このボードは、アンテナと一対の水晶振動子を備えたはんだ付けチップであるNRF51822モジュールでの使用を目的としています。 ボードとモジュールのキットのコストは約20ドルです。







BLE400には、CP2102チップに基づくUSBポートとUSB-UARTコンバーターが含まれ、RX、TX、CTSおよびRTS信号がボード上に出力されます。 回路レベルでは、NordicデバッグボードであるPCA10001、同じポートに接続されたLEDのペア、2つのボタンと部分的に互換性があります。LEDを点滅させたい人のために、これは生活を少し楽にします。 残念ながら、ボードはnRF51822プログラマーの役割を果たす方法を知りませんが、Arduinoタイプのブートローダーで動作するファームウェアがあります。 ボードに正確に組み込まれているものを簡単にリストします。一部は、 ここからのドキュメントの翻訳になります 。





  1. NRF51822モジュールの座席。キーはありません。モジュールを誤って挿入できますが、バッテリーが閉じられ、アンテナが逆になります。
  2. NRF51822モジュールのピンを複製し、プロトタイピングに便利
  3. 接続を容易にするために立ち上げられたI2Cインターフェイスは、SDAコムからの対応する信号を複製します-P0.00 SCL-P0.01 SMBA-P0.02および電源、Waveshareの既製モジュールを使用できます
  4. SPIインターフェース、信号MISO-P0.23 MOSI-P0.24 SCK-P0.25 NSS-P0.30
  5. 電源は、USB経由で接続した場合、5ボルトの電力を使用できます。バッテリーとプログラマーからは3.3のみが利用可能になります。 電源はRT9193で最大電流300mAで作成され、モジュール自体は最大2mAを消費し、最大5mAの3ピンまたは15mAの電流で1ピンを負荷に供給する準備ができています。 在庫の周辺への合計250 mA
  6. 電源を接続したり、PCに接続したりするためのミニUSBジャック。 5ボルトのポートは電源コームに直接接続されています
  7. 簡素化されたSWDプログラミングインターフェイス。 3.3v、SWDIOおよびSWCLKを搭載
  8. 複製されたUARTは、CP2102にも接続されており、仮想COMポートを介してPCで監視できます。 RX-P0.05 TX-P0.06 CTS-P0.07 RTS-P0.12
  9. スタンドアロンテストに役立つバッテリーサイズCR1225
  10. P0.18 P0.19 P0.20 P0.21 P0.22に接続された一連のLED 論理ユニットが点灯しているときに点灯します。 ここでは、RX、TX、およびSPD信号のアクティビティを確認できます
  11. ポートP0.16およびP0.17に接続された2つのボタン。 押されると、論理的ゼロを与える
  12. リセットボタン。 SWDIO信号をゼロにすることによりCP2102およびNRF51822をリセットします
  13. CP2102の仮想COMポートコントローラー、DTRおよびDSRははんだ付けされていません
  14. CP2102からUARTインターフェイスを無効にするジャンパー
  15. LEDおよびボタンを無効にするジャンパー


当初、このモジュールはNordic SemiconductorのnRF Toolbox for BLEを使用するためのコードをフラッシュしました。これは機能をテストするのに十分です。 ボードをPCに接続し、CP2102のドライバーをインストールし、ターミナルを使用してボードに接続します。 私は、PuTTY、ポートパラメータを使用しました。速度38400の日付ビット8、ストップビット1、ソフトウェアフロー制御。



接続されており、英語の文字Yまたはyでドライブしている端末で、応答としてStart ...コマンドを取得します-コントローラーは動作します。



現在、Nordic_UARTという名前の電話機のBLE互換デバイスのリストで使用できます。 Androidアプリケーションをインストールした後、仮想Bluetooth端末を介したキャラクターの送信をいじって、UARTのエミュレーションをテストできます。







プログラミングに進みます。 プログラマーが必要です。Seggerのオリジナルは少し高価ですが、自動車愛好家のおかげで、J-link ARM v8クローンは市場で十分であり、私たちの目的には約15ドル以上かかります。 プログラマー用のドライバー（たとえば、J-Link Software v4.52b +）が既にインストールされていると想定しています。 プログラマからの標準JTAGケーブルまたは4本のワイヤでボードに接続します。VTref、GND、SWDIO、SWCLKを接続するだけです。









接続をテストするには、JLink.exeを実行し、インストールされているチップに関する情報を取得します 



SEGGER J-Link Commander V4.74b ('?' for help) Compiled Aug 19 2013 23:26:57 DLL version V4.74b, compiled Aug 19 2013 23:26:44 Firmware: J-Link ARM V8 compiled Nov 28 2014 13:44:46 Hardware: V8.00 S/N: 20091135 Feature(s): RDI,FlashDL,FlashBP,JFlash,GDBFull VTarget = 3.293V Info: Found SWD-DP with ID 0x0BB11477 Info: FPUnit: 4 code (BP) slots and 0 literal slots Info: Found Cortex-M0 r0p0, Little endian. Found 1 JTAG device, Total IRLen = 4: Cortex-M0 identified. JTAG speed: 100 kHz


次のソフトウェアが必要です



  • Keil MDK-ARM v4
  • nRF51 SDK
  • nRFgo Studio


Keil MDK-ARMのLiteバージョンを取得するには、登録が必要です。残りは次のようにダウンロードできます。 Windows 10ではnRFgo Studioバージョン1.21.2は起動しませんでしたが、新しいバージョン1.15.1は動作します。 pca10001ボードをサポートするSDKの最新バージョンはバージョン6で、 リンクをダウンロードしてください 。 SDKをインストールすると、Keil環境がインストールされているフォルダーに、必要なすべてのファイルを含むディレクトリ\ARM\Device\Nordic\nrf51822



が表示されます。 例はKeilバージョン4を使用するように設計されています。プロジェクトファイルには、uvprojx拡張のあるKeil 5とは異なり、uvproj拡張があります。 実際には、設定ファイルとサンプルの互換性で混乱しないように、Keil 4を使用します。



ボードをプログラムするには、J-Link Commanderコンソールだけでなく、nRFgo Studioを使用することもできます。nRFgoStudioを使用すると、チップをクリアしたり、SoftDeviceを埋め込んでBluetoothで動作したりできます。 これは、nRF51プログラミングメニューから行います。 BLEスタックのコアは、すでにチップにあふれています。 メニュー項目「すべて消去」を使用して消去できます。











次のタスクは、ボードに慎重にはんだ付けされたLEDを点滅させることです。 既に述べたように、LEDとボタンはNordic pca10001ボードと互換性があるため、そこから例を使用し、Keil4 \ARM\Device\Nordic\nrf51822\Board\pca10001\blinky_example\arm\



のフォルダー内のプロジェクトファイルblinky.uvprojを開きます。

F7を押してプロジェクトをアセンブルし、メニューのFlash-> Downloadでフラッシュします。 デフォルトのJTAG充填速度は2000 kHzであるため、デフォルトでは点滅しませんでした。 ターゲットのメニューオプション->デバッグ->設定から修正できます。 100 kHzに置き換えた後、すべてが正常に機能しました。 LED LED0およびLED1が正常に点滅し、マイクロコントローラーが機能しています。 それでもダウンロードできない場合は、nRFgo Studioを使用して_build blinky_arm.hex



フォルダーから、またはJ-LinkプログラマーコンソールからHEXをフラッシュしてみてください。 ただし、最初に、Keilの設定でJ-Linkプログラマーが選択されていることを確認する必要があります。











LEDは優れていますが、Bluetoothが必要です。作業例は、 \ARM\Device\Nordic\nrf51822\Board\pca10001\s110\ble_app_beacon



。 これがiBeaconの実装です。私たちにとって、NordicのソフトウェアBLEスタックを使用するのは興味深いことです。 この例は、機能の点で最も単純なs110スタックバージョン用に設計されていますが、機能を実証するにはこれで十分です。 カーネルは個別にダウンロードする必要があり、多くのバージョンがありますが、ヘッダーのバージョンはカーネルのバージョンと相関する必要があります。 SoftDevicesセクションのリンクからカーネルをダウンロードできます。



当初、バージョン7.0.0のヘッダーファイルにはSDKが付属していましたが、このカーネルバージョンは私にとっては起動しませんでした。 理論的には、スタックのどのバージョンでも動作するはずです。バージョンv6、v7、v8はダウンロード可能です。 SDKの例では、スタックの7番目と8番目のバージョンが機能しなかったため、6.2.1で停止しました。



カーネルをダウンロードして、Cores ARM\Device\Nordic\nrf51822\Cores\6.2.1



などの便利なフォルダーに解凍します。 既存のフォルダー\ARM\Device\Nordic\nrf51822\Include\s110



をクリアし、アンパックされたカーネルのincludeフォルダーからヘッダーファイルをコピーします。



SoftDeviceをフラッシュする必要があります。これにはnRFgo Studioを使用し、Device ManagerメニューからnRF51 Programmingを選択し、プログラマーが落ちないようにします。 [プログラムSoftDevice]タブで、カーネル\ARM\Device\Nordic\nrf51822\Cores\6.2.1\s110_nrf51822_6.2.1_softdevice.hex



のファイルを選択します。 使用されている容量の下に正しい形式のファイルが表示される場合、このバージョンのSoftDeviceでは80kbです。











[プログラム]ボタンをクリックします。 デバイスのメモリカードは次のようになります。







RAMは8kb少なくなり、カーネルは永続メモリの最初の80キロバイトを占有しました。 ターゲットをs110スタックのターゲットに変更し、メニューからnrf51822_xxaa_s110（256K）を選択します











SoftCoreのインストール時にLEDが点滅する例を作成するには、メモリカードに修正を入力する必要があります。これは、[ターゲットのオプション]-[ターゲット]メニューから行います。 ベースアドレスはCODE_R1_BASE = 0x00014000



変更され、空きメモリはアドレス0x20002000から始まります。











スケッチに記入する際にカーネルを上書きしないようにするため、プログラマーの設定を変更します。 これは、[ターゲットのオプション-デバッグ]メニューの[設定]ボタンから行います。











プロジェクトをアセンブルしてコントローラーに入力します。カーネルが正しくインストールされていれば、同じ点滅LEDが表示されます。 iBeaconをシミュレートするファームウェアをアップロードし、フォルダー\ARM\Device\Nordic\nrf51822\Board\pca10001\s110\ble_app_beacon



からプロジェクトを開きます。 点滅するLEDと同様にメモリカードを変更し、収集してデバイスに書き込みます。



すべてが正常に実行された場合、LED0が点灯します。これは、パケットが送信されていることを示しています。 iBeaconを表示できるアプリケーションをAndroidまたはiOSデバイスにインストールし、ダウンロードしました

ビーコンスキャナー 。 デバイスのUUIDはソースコードの定数APP_BEACON_UUIDと一致するため、これがビーコンです。









原則として、この情報は、BLEを迅速かつ非常に安価に使用するには十分です。 欠点としては、時代遅れのKeil 4と100 kHz以上の周波数での不安定なプログラマー操作をサポートします。 開発環境でダウンロードエラーが発生した場合は、アクセス速度を50 kHz以下に下げてください。 プログラマーが落ちて、JLink.exeを実行するだけで元に戻すことができる場合があります。これはおそらく、元のクローンであるためです。
