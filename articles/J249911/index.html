Laravel 5のリリースを待っていた

みなさんこんにちは！ Habrahabrの住民は、Taylor Otwellの雄弁なPHPフレームワークLaravelにまだ懐疑的であるようです。 残念ながら、昨日リリースされた新しい5番目のバージョンを取り上げた人はいませんでした。 この欠点を補おうと思います。 これがどのような通常のフレームワークで、彼のガレージにある自転車については説明しませんが、変更の影響を受けた瞬間のみを取り上げます。 基本的に、この記事は公式サイトからのリリースノートの無料翻訳になります。 意味が失われないように、概念と用語の一部を翻訳しませんでした。



更新された公式ウェブサイトのデザイン



Laravelを初めて知ったとき、リンクをオフにしたことを覚えています。 サイトの同僚は彼の熱意を共有しましたが、数分後、彼はカテゴリ「ピンクが多すぎます」を聞きました。 今後、このような発言は行われるべきではありません。 新しいデザインでは、メインページの上部にある大きなピンクの帽子が取り除かれ、パーソナライズされた特定のソリューションの使用を要求する（フレームワークのコンテキストで）愛国的な見出しがわずかに残っています。



ドキュメントページは以前の外観を保持していますが、バージョンの分離に関してはより論理的になりました。 以前は、開発バージョンはdevとしてマークされていましたが、現在はmasterと呼ばれ、これはgithubのブランチの名前に対応しています。



ファイル構造



app / modelsフォルダーが削除されました。 これで、アプリケーションに関するすべてのコードは、 App名前空間（hello psr-4！）のappフォルダーに保存されます。 標準の名前空間は、 app：nameコマンドを使用してArtisanによって変更できます。



コントローラー、ミドルウェア、およびリクエスト（Laravel 5.0の新しいクラスタイプ）は、次のようにapp / Httpフォルダーにグループ化されます。 これらのクラスは、アプリケーションHTTPレイヤーに属します。 フィルターを持つ単一のファイルの代わりに、各ミドルウェア（フィルターの代替、要求とコントローラーへの移行の間にあるもの）が独自のクラスを取得しました。



app / Providersフォルダー内のクラスは、 app / startフォルダー内のファイルを置き換えるようになりました。 これらのサービスプロバイダーは、エラー処理、ロギング、ルートのロードなど、アプリケーションへのロードクラスを提供します。同じ場所で、独自のサービスプロバイダーを作成できます。



多言語ファイルとプレゼンテーションファイルはリソースフォルダーに移動されました。



契約



すべての主要なLaravelコンポーネントは、 照明/契約リポジトリにあるインターフェースを実装します。 このリポジトリには外部依存関係はありません。 便利で一元化されたインターフェイスのセットを使用して、それらをデカップリングに使用できます。依存関係の実装は、ファサードの優れた代替手段として機能します。

ルートキャッシュ



アプリケーションがルートコントローラーのみで構成されている場合は、新しいArtisan route：cacheコマンドを使用して、 ルートマップの読み込みを大幅に高速化できます。 100以上のルートを持つアプリケーションでこの機能を使用して、アプリケーションのこの部分のダウンロード速度を上げることは理にかなっています。



ルートミドルウェア



Laravel 4フィルターはLaravel 5 HTTPミドルウェアに置き換えられ、おなじみの標準認証フィルターとCSRF保護は新しい方法で書き直されました。 ミドルウェアは、あらゆる種類のフィルターを交換するための単一の一貫したインターフェースを提供し、処理のためにコントローラーに送信される前に、リクエストを簡単に検査および拒否できます。

public function createPost(Request $request, PostRepository $posts) { // }




認証スキャフォールディング



登録、ユーザー認証、およびパスワードの回復が「箱から出して」すぐに利用できるようになり、対応するビューはresources / views / authにあります。 これに加えて、usersテーブルの移行にはフレームワークが付属しています。 これらの単純なリソースを含めることで、認証の整理に時間を費やすことなく、プロトタイプを迅速に開発できます。 許可インターフェースは、 auth / loginおよびauth / registerルートで使用できます。 App \ Services \ Auth \ Registrarサービスは、ユーザーの検証と作成を担当します。



イベントオブジェクト



イベント（orig。：Events）は、文字列ではなくクラスとして定義できるようになりました。例： 

 class PodcastWasPurchased { public $podcast; public function __construct(Podcast $podcast) { $this->podcast = $podcast; } }


前と同じように、イベントを送信できます（元：ディスパッチ済み）。 

 Event::fire(new PodcastWasPurchased($podcast));


イベントハンドラは、データリストの代わりにイベントオブジェクトを受け取ります。 

 class ReportPodcastPurchase { public function handle(PodcastWasPurchased $event) { // } }

コマンド/キューイング



Laravel 4でサポートされているキュー（オリジナル：キュージョブ形式）に加えて、Laravel 5ではキューを単純なコマンドオブジェクトとして提示できます。 これらのコマンドは、 app / Commandsフォルダーにあります。 例： 

 class PurchasePodcast extends Command implements SelfHandling, ShouldBeQueued { use SerializesModels; protected $user, $podcast; /** * Create a new command instance. * * @return void */ public function __construct(User $user, Podcast $podcast) { $this->user = $user; $this->podcast = $podcast; } /** * Execute the command. * * @return void */ public function handle() { // Handle the logic to purchase the podcast... event(new PodcastWasPurchased($this->user, $this->podcast)); } }


Laravelベースコントローラーは、新しいDispatchesCommandsフラグ（orig。：trait）を使用します。これにより、実行するコマンドを非常に簡単に送信できます。 

 $this->dispatch(new PurchasePodcastCommand($user, $podcast));


コマンドは、同期して実行されるタスクにも使用できます（整列しないでください）。 実際、コマンドを使用することは、アプリケーションが解決する複雑なタスクをカプセル化する優れた方法です。

フォームの要件



Laravel 5.0は、「フォームリクエスト」（元のフォームリクエスト）を導入します。これは、 Illuminate \ Foundation \ Http \ FormRequestクラスによって拡張されます。 これらの要求オブジェクトをコントローラーメソッドに埋め込み、ユーザー入力を検証できます。 簡単なFormRequestの例： 

 <?php namespace App\Http\Requests; class RegisterRequest extends FormRequest { public function rules() { return [ 'email' => 'required|email|unique:users', 'password' => 'required|confirmed|min:8', ]; } public function authorize() { return true; } }




クラスが定義された後、コントローラーメソッドでそれについて「ヒント」（orig。：type-hint）を実行できます。 



 public function register(RegisterRequest $request) { var_dump($request->input()); }




Laravel IoCコンテナがFormRequestクラスの実装済みインスタンスを決定すると、リクエストは自動的に検証されます （オリジナル：検証済み）。 これは、コントローラーの（元の：アクション）メソッドが呼び出された場合、 FormRequestで定義されたルールに従ってユーザー入力が正常に検証されたことを確認できることを意味します。 さらに、リクエストが無効な場合、リダイレクト（設定可能）は、クイックメッセージ（オリジナル：フラッシュメッセージ）がセッションに送信される場所、またはJSON文字列で自動的に機能します。 フォームの検証がかつてないほど容易になりました！

シンプルなコントローラーリクエストの検証



Laravel 5のベースコントローラーは、 ValidatesRequests属性（orig。：trait）を使用できるようになりました。 この機能は、着信リクエストを検証する簡単な方法を提供します。 FormRequestsがアプリケーションにとって面倒すぎる場合は、以下をご覧ください。 

 public function createPost(Request $request) { $this->validate($request, [ 'title' => 'required|max:255', 'body' => 'required', ]); }




検証に失敗すると、例外がスローされ、適切なHTTPヘッダーがブラウザーに送り返されます。 また、検証エラーはセッションに「フレアアップ」（元：点滅）しました。 リクエストがAJAXによって送信された場合、Laravelはこれらのエラーを自動的にJSON文字列に変換します。
