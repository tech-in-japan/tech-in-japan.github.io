Oracle Exadata、または設計されたシステムの利点について（パート2）

記事の最初の部分では 、 設計されたシステム Oracle Exadataの基礎となる基本的なアプローチについて説明しました。 さらに深く掘り下げます。







クエリオフロード



Exadataでできること、クラシックデータベースではできないことは何ですか？ おそらく既に推測したように、増分バックアップはストレージシステムレベルでのみ実行され、データベースサーバーのパフォーマンスには影響しません。



大規模なI / Oを必要とし、データベースレベルの負荷が高い操作も、ストレージシステムレベルに転送されます。 また、古典的なアーキテクチャで配列がデータベースサーバーが使用しているブロックのセットをシステムに返す場合、Exadata Storage Serverは、要求で必要な行と列の完全なセット、いわゆるレコードセット、必要な列と列のみを返します。 その結果、データベースサーバーのプロセッサリソースが大幅にアンロードされます。



圧縮



Oracleデータベースは、従来のデータ圧縮であるOLTP圧縮を使用します。 Exadataはハイブリッド列圧縮を実行します-これは、Hybrid Columnar Compressionを使用する場合、データがバッチ形式で保存されることを意味します。これは、高性能で非常に高い圧縮率を実現できるよりコンパクトな形式であるため、大規模なデータウェアハウスに非常に効果的です。 念のため、オラクルはマーケティング資料で、SAPアプリケーションで10倍の圧縮が達成されていると報告しています。 しかし実際には、SAPアプリケーションの場合、実際のデータに対してはるかに高い圧縮率が達成されます。



PCIフラッシュ



PCIフラッシュアーキテクチャは、ドライブよりも高速であるだけでなく、システムのパフォーマンスを向上させます。 デフォルトのソリューションには、「ホット」データを通常のディスクからPCIフラッシュキャッシュに移動する機能が含まれています。 システムは負荷の変化に対して非常に迅速に応答し、データベースが特定のデータブロックを積極的に必要とする場合、それらは高速PCIフラッシュキャッシュに即座に転送されます。 しかし、もちろん、他の追加機能と同様に、これはオフにすることができます。



事前確認



特定のシステムを導入する前に、顧客は常に複雑で面倒なテストに頼らざるを得ません。これは、ソリューションが顧客のITインフラストラクチャでどの程度正確かつ効率的に機能するかを示します。 これを行うには、テストベンチを構築し、テスト方法を開発し、ビジネスにとって最も重要なタスクを特定し、負荷の発生を考慮し、少なくとも数週間は現在のタスクからITスペシャリストの注意をそらす必要があります。 Oracle Exadata for SAPの場合、この問題は、Oracle Exadata SAP Acceleration Pre-Checkと呼ばれる顧客向けの無料の専門サービスによって解決されます。



これは次のように機能します-（オラクル） エンジニアリングシステムへの切り替えを計画しているSAPクライアントの場合、オラクルのスペシャリストはITインフラストラクチャを分析し、SAPソリューションがOracleハードウェアおよびソフトウェアプラットフォームで動作する効率を計算します。 STAD、ST03、およびAWRの分析は、ベストプラクティスに基づいて実行されます。 OracleデータベースおよびSAPアプリケーションの実務分野の既存の経験による。 したがって、サービスはテスト環境を必要とせず、本稼働システムに影響を与えません。



レポートには、システムの合計反応時間、アプリケーションとデータベースの反応時間などの指標に関する情報が含まれています。 同時に、アプリケーションパフォーマンスの最も悲観的な予測が常に提供されます。 実際には、このソリューションは予備分析で示されるよりもはるかに高速に機能します。



護衛



システムの保守とアップグレードに対する従来のアプローチはどのように機能しますか？ システムを1年間運用し、その後、自分で、またはITパートナーを巻き込んで、決定のかなり大規模な監査を行います。ネットワークの混雑、I / Oシステムとサーバーの混雑、最適に組み立てられた構成、パフォーマンスの低下とその原因。 Exadataの場合、すべてがはるかに簡単です-Oracle Exadata Exachkプログラムを実行すると、すべてのシステムコンポーネントの包括的な診断が実行され、Exadata構成に関する完全な情報が提供されます-たとえば、プログラムは、システム内のディスクグループがOracleの推奨どおりにマークアップされていないことを示します 定期的な更新がシステムにインストールされていない場合、これもレポートに反映されます。



技術サポート



ハードウェアおよびソフトウェアシステムの適切な運用のために、テクニカルサポートの組織が非常に重要であることを説明する必要はありません。 Oracleには、Exadata、Exalogic、およびSuperClusterソリューションのテクニカルサポート専用のプログラムがあります。 このプログラムのフレームワーク内で、クライアントケースはハードウェアレベルとソフトウェアレベルの両方で考慮されます。これは、ソフトウェアとハ​​ードウェアシステムのアーキテクチャによって促進されます。



Oracle Classic Serviceは、完全かつ包括的でプロアクティブな24時間のプレミアサポートサポートであり、2時間以内に顧客のリクエストへの応答を保証します。 同じお金で、 工学システムのために、オラクルはプラチナサービスのサポートを提供します。 認定された構成の場合、障害の5分間の通知、問題の解決策、または15分以内の開発ユニットへのエスカレーションが保証されます。後者の場合、開発者は30分以内に問題を解決します。 このようなSLAパラメータは、オラクルのエンジニアがプラットフォームのステータスを積極的に監視し、すべてのレベル（オペレーティングシステムレベル、すべてのコンポーネントのソフトウェアおよびハードウェアレベル、データベースレベル）でシステム情報を読み取るために実現されます。



システムの更新手順は、従来のアーキテクチャよりもはるかに簡単です。Oracleエンジニアが既にテストした四半期ごとの更新を取得するだけです。 したがって、システムの更新に必要な時間、およびそのような更新に特有のリスクが大幅に削減されます。



認証とライセンス



SAPがOracle Exadataコンプレックス全体を認証すること、およびサーバー、ネットワーク、クラスターソフトウェア、ストレージシステム、さらにケーブルを認証することは基本的に重要です。 これにより、ソリューションの品質と正常な動作が保証されます。これは、顧客にシステムを起動するためのすべての段階と期限が非常に明確に決定されるためです。



もちろん、データベースサービスをExadataまたはSuperClusterで実行するには、Oracleデータベースのライセンスが必要です。 SAPを介して、またはOracleから直接、ソリューションのライセンスを取得することができます。

スーパークラスター



3層アーキテクチャのSAPアプリケーション環境は、アプリケーションサーバーとデータベースサービスの2つのコンポーネントで構成されています。 データベースサービスはOracle Exadataにデプロイできますが、SAPアプリケーションサーバーはどこにデプロイできますか？ これは、Oracle SuperClusterで実行できます。



Oracle SuperClusterは、データベースサービスとアプリケーションサーバー（ストレージシステム、データベースサーバー、仮想化システム、クラスターソリューション、アプリケーションサーバー）の機能に必要なすべてを含む、自己完結型のソフトウェアとハ​​ードウェアの複合体です。



Oracle SuperClusterには、ハーフラックスーパークラスターとフルフラッシュの2種類があります。 ハーフラックスーパークラスターには、Oracle T5プロセッサーを備えた2つの計算モジュールが含まれています。 データベースはExadata Storage Server上で実行されます。 構成はハードウェアの観点から完全に複製されており、アプリケーションもソリューションに含まれるSolaris Clusterテクノロジーを使用して保護されます。 災害から地理的構成を保護するために、メインサイトからバックアップへのZFS SAレプリケーションが使用され、データベースサービスの保護にOracle Data Guardテクノロジーが使用され、アプリケーションサーバーの保護にOracle Solaris Clusterが使用されます。 したがって、Oracle SuperClusterは、メインまたはバックアップのデータセンターが完全に失われるなど、障害から保護される完全なソリューションです。







移行



Oracle ACSメンテナンスプログラムの一環として、SAP用の生産的なデータベース移行サービスがOracle ExadataまたはOracle Superclusterで提供されます。 移行プロジェクトはさまざまなオペレーティングシステムで実行されます。UNIX、Linux、Windows、クロスプラットフォームの移行、Oracleデータベースバージョンの更新からの移行が可能です。 移行率は主にハードウェアプラットフォームの速度に依存し、通常は1時間あたり0.5〜1 TBです。
