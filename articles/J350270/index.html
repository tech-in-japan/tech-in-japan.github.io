InfiniBox F2230を確認する

ONLANTAは最近、 Oncloud.ruコーポレートクラウド用にInfinidatのInfiniBox F2230システムを購入しました。 このシステムのレビューはインターネット上にはほとんどないため、このギャップを解消することにしました。 本日は、このストレージシステムについてできる限りお話しするようにします。









Infinidatはイスラエルの若い会社です。 かつてEMC Symmetrixの開発を率いていたMoshe Yanayによって2011年に設立され、後にXIV社を設立し、このシステムのチーフアーキテクトでした。 ところで、2008年のXIVはIBMに買収されました。 そのため、エンタープライズレベルのストレージの製造におけるInfinidatの経験は、この記事の範囲を超えています。 Infinidatシステムの重要な要素の1つは、高い信頼性（99.99999％）です。 ガートナーのアナリストによると、Infinidatはディスクアレイの先見者のリーダーです。



F2230シリーズシステムは18Uしか占有しません。 同時に、ピーク消費電力はわずか3.5 kWであり、250 TBの有効容量で非常に経済的です。 このシステムは圧縮を完全にサポートしており、最大1PBの有効ボリュームを増やすことができます。



会社の主な開発は、高品質の方法で機器を管理できるオペレーティングシステムです。 基本的に、これはSoftware Defined Storage（SDS）です。 これらのシステムは、現在「古典的な」競合他社と積極的に市場で戦っています。 Infinidatは、すべてのハードウェアコンポーネントと独自のラックを備えた完全なシステムとして提供されます。







アンロードとアンパック



システムの中心は、アクティブ-アクティブ-アクティブモードで実行されている3台のサーバーです。









3つのコントローラーを使用すると、コントローラーの1つに障害が発生した場合のパフォーマンスの損失を減らすことができます。 従来の2シングルモード構成では、1つのコントローラーに障害が発生した場合、システムは書き込みキャッシュの安全性を確保できません。 これにより、強制リセットが発生し、遅延が増加します。 3線構成の場合、コントローラーの1つに障害が発生しても、キャッシュは残りの2つのノード間で同期を続けます。







コントローラーとBBU



すべてのコントローラーの合計RAMは384GBです。 システムには24TBフラッシュキャッシュと60個のNL-SASディスクを備えた2つのディスクシェルフもあります。

ディスクシェルフ、背面図



システム内の各ノードには、すべてのノードのグローバルキャッシュとして機能する複数のSSDドライブが装備されています（データブロックは2回キャッシュされません）。 ボリュームのアドレス空間は断片に分割され、断片はコントローラー間で分散され、各コントローラーはその断片を維持し、キャッシュします。 ピースではなくコントローラーにリクエストが届くと、InfiniBandを介して別のコントローラーからリクエストを受け取ります。



ノード間の接続は、InfiniBandを使用して実行されます。 ディスクシェルフは、SASを介してコントローラーに接続されます。 データへのアクセスは、FCとイーサネットの両方で調整できます。



InfiniRAID-Infinidatの論理アーキテクチャは、ディスク全体ではなく、ディスクの小さなブロック（64k）をRAID要素として使用します。 これにより、ディスクの競合が最小限に抑えられ、システム内の各ノードがパフォーマンスを最大化するためにすべてのディスクを常に使用できるようになります。



データはシステムに到着すると、DIFフィールド（データ整合性フィールド-データ整合性フィールド）とともにRAMに配置され（クイックアクセス用）、論理エラーやレコードの損失を防ぐために、すべてのレベルのストレージと処理でデータとともに保持されます。 DIFは、各読み取り操作中にデータとともに読み取られ、すべての検出不可能なエラー（サイレントデータ破損）を検出するためにチェックされ、RAIDを使用して修正されます。



データ配布のアルゴリズムがメモリから永続メディアにデータを書き込むことを決定すると、書き込みのブロックを選択し、その前にそれらを最適化します。 これには、14のセクション（DIFを含む）+ 2つのパリティブロックのグループにブロックを編成することが含まれます。









このアルゴリズムは、これらのグループ（ストライプ）をInfiniBoxのすべてのディスクに分散し、次のことを確認します。



  • 異なるディスク上の各セクション、
  • セクションはシェルフ間で均等に分散されます（シェルフの問題から保護するため）。
  • セクションはすべてのディスクに均等に分散されます（容量とパフォーマンスの最適な分散のため）。








各仮想RAIDグループにさまざまなディスクグループのパーティションが含まれるこのようなデータの配布により、すべてのシステムディスクを一度に使用してデータを回復できます。 したがって、InfiniBoxは、数日ではなく数分で大きなディスクを回復できます。 ディスク障害（旧モデルでは12台のディスク）に対する保護のために割り当てられた十分なスペース、および保護のために未割り当てスペースを使用する機能を使用するため、システムは12台以上の破損ディスクに耐えることができます。









ほとんどの読み取り操作は、SSDにあるキャッシュから実行されます。 これにより、NL-SASディスクの負荷を最小限に抑え、キャッシュに入らなかった少量のデータのみにアクセスできます。



書き込み操作は、DRAMで保護されたキャッシュに分類されます。DRAMで保護されたキャッシュには、ブロックが「クールダウン」する前のすべての変更が含まれます。 データブロックがディスクに送信されると、それらはすべてのディスクに同時に書き込まれるため、システムはそれらを高速で書き込むことができます。 システムが後で読み取られる可能性が最も高いと見なすデータのブロックは、NL-SASディスクだけでなく、SSDキャッシュにも書き込まれ、読み取りが高速になります。









この独自のデータレイアウトにより、InfiniBoxはSASドライブの使用を回避し、代わりにSSDとNL-SASドライブの組み合わせを使用して、高性能と低IOレイテンシを実現できます。







インストールされ、接続され、構成されたシステムは操作の準備ができています



これはユーザーに多数のタスクを実行するシステムであることは注目に値します：ディスクへのアクセスは許可しません-RAIDは事前に実行済み、ディスク間およびコントローラー間の負荷分散は自動的に実行され、コントローラーファームウェアはサポート付きで更新されます









右上隅のこのアイコンは、Infinidatサーバーへの接続が確立されており、サポートが何らかの診断などのためにシステムにアクセスできることを示しています。 サポート担当者がアレイに接続すると、アイコン全体が緑色に変わります。 サポートへのアクセスを無効にし、必要な場合にのみ有効にすることができます。



ストレージ効率



これらのシステムは重複排除をサポートしていないという事実から始めたいと思います。 一般的に。 圧縮のみ。これについて詳しく説明します。



データ圧縮がシステムパフォーマンスに影響を与えないようにするため、Infiniboxは、データがキャッシュからディスクに転送されるときに圧縮します。 現時点では、システムはLZ4圧縮アルゴリズムを使用していますが、ストレージシステムアーキテクチャは、将来、より新しく効率的なものに置き換えることができるように構築されています。 データ圧縮はプロセッサに依存する操作であり、最新のプロセッサを使用している場合でも、システム全体のパフォーマンスに影響する可能性があることを忘れないでください。 圧縮は64KBブロックで行われるため、より高い圧縮率を実現できます。



圧縮はシックボリュームとシンボリュームの両方で機能しますが、これらの場合の動作は異なります。



  • シンボリューム-データ圧縮によるスペース節約は、プールの物理容量で引き続き利用可能です。
  • シックボリューム-この方法で解放されたスペースは、再割り当てに使用できません。


したがって、InfiniBoxがパフォーマンスに影響を与えることなくディスク領域を節約する方法：データは非圧縮形式でRAMキャッシュに格納されます。 データがディスクに書き込まれると圧縮が行われます。 しかし、HDDに記録しているのは、 SSDキャッシュに到達するものは、非圧縮形式でもあります。 Infinidatは、作業中のデータのほとんどがキャッシュされていると想定しているため、圧縮しても応答時間が長くなることはありません。 実際、これは常にそうではないことを理解していますが、小さなブロックの多数のディスクにデータを保存する一般的なアーキテクチャを見ると、応答が大幅に増加することはありません。



このシステムの操作について詳しく知りたい場合は、Infinidat RussiaのテクニカルディレクターであるVasily Kostryukovの記事「 デジタルエンタープライズ向けストレージアーキテクチャ 」を参照することをお勧めします。



複製



InfiniBoxは、同期複製と非同期複製の両方をサポートしています。 非同期複製はスナップショットに基づいています。 同期複製はI / Oを両方のシステム（プライマリおよびバックアップ）に送信し、これにより応答時間が増加します。 ホストは、両方のシステムへの記録後にのみデータ記録の確認を受け取ります。 読み取りはローカルシステムからのみ行われます。







非同期複製



  1. ホストはI / Oを送信してソースInfiniBoxに書き込みます
  2. ソースInfiniBoxがホストへのI / Oレコードを確認します
  3. ソースInfiniBoxはデータをターゲットInfiniBoxに複製します
  4. ターゲットInfiniBoxがソースInfiniBoxへの複製を確認します


システムに従って、システム間の通信の品質が既に構成されている同期レプリケーションの要件を満たさない場合、適切な接続品質を復元するときに、自動的に非同期モードに転送し、同期モードに戻します。



非同期複製は以下でサポートされています。



  • ボリューム
  • ファイルシステム
  • 整合性グループ。


同期レプリケーション：



  • ボリューム。


複製されたエンティティでサポートされていない操作：



  • 除去
  • 回復
  • サイズ変更
  • 書き込み禁止を無効にし、
  • ファイルシステムのエクスポート。








同期レプリケーションモードでは、レプリケーション方向の変更がサポートされています。



非同期複製用のリンクを作成するには、各システムに4つのポート（3つのデータポートと1つの管理ポート）が必要ですが、同期複製には7つのポート（6つのデータポートと1つの管理ポート）が必要です。

インターフェース、RESTful APIおよびSDK



私はインターフェースの多くのスクリーンショットを撮るという伝統から離れ、今回は必要なコメントを伴ったビデオを作りました。





InfiniBoxシステムは、そのAPIにより、クラウドインフラストラクチャ管理システムに完全に統合されます。このAPIを使用すると、システムをすばやく便利に操作できます。 サービスを提供し、リソースの割り当てがコントロールパネルを使用して行われるか、システムが独自のインフラストラクチャのみを提供するかは関係ありません。多数の要素を迅速かつ便利に追加する必要があります。 これはすべて、APIを使用して完全に実装されます。 Python用のInfiniSDKもあります。



ホストを1000個追加しますか？ 問題ありません！ 



#!/usr/bin/python #! -*- coding: utf-8 -*- from infinisdk import InfiniBox from infi.dtypes.iqn import make_iscsi_name datastring = ''' iqn.1990-06.vmware:host1 … iqn.1991-06.vmware:host1000 ''' system = InfiniBox('infinidat_ip', auth=('infinidat_login', 'infinidat_password')) system.login() datastring = datastring.splitlines() for i, iqn in enumerate(datastring): host_name = 'host-' + str(i) host = system.hosts.create(name=host_name) address = make_iscsi_name(iqn) host.add_port(address)
