なぜMitrhilなのか 使用する動機





ミスリル自体について簡単に説明します-内容と内容



MithrilはMVCマイクロフレームワークであり、その表示モデルは仮想DOMに基づいており、大規模なアプリケーションのレンダリングパフォーマンス、またはコンパクトだが複雑なウィジェットレンダリングロジックも追加します。 ミスリルには、完全なフレームワークと呼ぶのに必要なものがすべて揃っています。 コントローラー、モデルデータのバインディングを操作する手段、レンダリングを操作するための完全なシステムなど、ロジックを操作するシステムがあります。 また、低レベルで、コンテンツの再描画の時間とタイプ（複雑な非同期操作など）を正確に制御し、コンパクトに追加できるユーティリティも提供します。

  • ルーター -コンポーネントをページとしてブラウザの特定のアドレスにバインドできます。
  • Webサービス -http（s）経由でオンラインデータリクエストを操作できるようにするメソッド。フレームワーク機能の適応動作が組み込まれています。


これらすべてのコンポーネントとコンポーネントを使用すると、コードの作成中にニーズの90％をカバーするアプリケーションをmithrilで開発できます。



なぜミスリルを選んだのですか？



アーキテクチャの開発中、システム自体にデータの負荷が最小限であり、すべてが典型的なものであり、すべての要求、処理テンプレートの形式での追加のオーバーレイ、出力、および応答を待機しているクライアントを考慮して、クライアントとサーバー組織の間で選択しました-なぜなら、私たちの時代には、コンテンツのダウンロード速度と「瞬時の」読みやすさが非常に重要であり、機器に余分な負荷がかかるためです。



そのため、すぐにサーバー実装オプションを破棄し、少量の血液（および余分な使用済み神経細胞）でさまざまなタスクを解決できるクライアント側ツールのオプションを検討し始めました。

  1. ツールのパフォーマンス。
  2. 便利なデータ編成。
  3. 描画の管理と速度。
  4. ブラウザサポート：IE9以降、モバイルブラウザ。
  5. 冗長な「適用された」相互作用のない便利で実用的なロジック管理
  6. 過剰な機能性を伴わない、ツールの最大のコンパクトさ。
  7. 前の声明から来る-楽器の最大「バニラ」。
  8. ツールに慣れていない新しい開発者の開発への迅速な接続。
  9. SEOを考慮して、将来的に適応する能力。
  10. コンポーネントに依存しない開発-モノリシックソリューションを必要としません。




検討中のツールの中で、 Angular 、 Angular Light 、 Backbone 、 Knockout 、 Reactなどの多くのライブラリを調べました。 また、既製のフレームワークと複雑なツールだけでなく、個別のテンプレートシステム、データライブラリなども検討しました。



他のツールをふるいにかける理由


分析中にこれらのオプションがすべて破棄されたのはなぜですか？ 以下は、これらのツールが選択に関連しなかったリストの簡単な概要です。

AngularとReactは冗長性があるため、すぐには適していません。機能と組織の99％がプロセスで必要ないためです。

バックボーン -過度に冗長なロジックのために排除されましたが、それは遠い時代のアタビズムであり、その開発により、すべてのコンテンツのほぼ直接的な制御、および一般的に暗黙的なロジック構造の選択が可能になりました。

Knockoutは優れたライブラリですが、埋め込まれた一連の操作には大きすぎます。それらの多くがあり、最初はテンプレートでレンダリングするための大量のコード用に設計されています（ロジックおよびモデルのデータに関連）。 Backboneと同様にロジックの制御は非常に具体的ですが、データとバインディングを適切に操作できます。 私たちのアプリケーションでは、この機能は冗長であり、このオプションも放棄することが決定されました。

Angular Lightは、テンプレート、「遅延」ディレクティブおよびコンポーネントを含む、宣言された同様のロジックを持つ、角度のマイクロアナログです。 角度からのディレクティブの元のロジックと動作から、多くは残っていませんが、簡素化されていますが類似したテンプレートシステム、ツリーとディレクティブの初期化をタグ、クラス、属性、および$スコープバインディングの類似性によって比較するシステムです。 優れたツール-しかし、かなり具体的で柔軟性の低いコンポーネント構築ロジックを備えており、その正確な動作を理解することも最初は困難でした。 このオプションは、最後までスペアのままで機能していました。



ミスリル


mithrilがツールの選択を目の当たりにしたとき、それはすでに完全な機能を備えたアプリケーションを書くための一連のツールの非常に成熟した成功したプロジェクトでした。 このツールには、バックボーンやノックアウトなどのアタビズムや過度のテンプレートロジックはありませんでした。実際、ミスリルはDOM要素の小さなラッパーであり、そのほとんどはDOMメソッドに直接結び付けられ、仮想DOMの形で小さな砂糖が追加され、オブジェクトに接続されていましたコントローラーロジック。 同時に、データバインディングと過剰な部分のほとんどがロジックから除外されました。これにより、実装の柔軟性が最大限に高まりました。 彼は、容量のある拡張可能なアプリケーションを開発するときに必要なすべてのミニマリズムと実用性を収集しました。 Angular Lightとは異なり、より透明なコンポーネントロジックがあり、より実用的で統一されています。これにより、コードの保守が容易になり、アーキテクチャの微妙な問題が突然発生することを考えなくなりました。



美をもたらす-砂糖、便利さ、ユーティリティ



既定では、mithrilでのコンポーネントの開発は、開発の利便性を高めるために、絶対に何も規定しないプロセスです。さまざまなツールやライブラリを好みに合わせて使用​​できますが、mithrilでアプリケーションの開発を開始するすべての人に役立つ基盤があると思います。

デフォルトでは、Mithrilは、アーキテクチャ内の独立したコンポーネントである通常のオブジェクト内に編成されます。クラスとして使用する組織の特別な機能は、同じミニマリズムの理由で実装されませんが、これは、オブジェクト開発を拡張または適応できないことを意味するものではありません楽器。



最も単純な場合、コンポーネントオブジェクトをクラスのインスタンスとして使用する必要があります-たとえば、次のように： 

 /*   */ class Component { constructor(opts) { /*     ... */ this.opts = opts; /*      */ this.controller = this.controller.bind(this); this.view = this.view.bind(this); } /*   */ controller(props) { return { paramOne: this.opts.paramOne }; } /*   */ view(ctrl, props, children) { return ( <button onclick={this.method} attr={ctrl.paramOne} /> ); } }; /*   -      */ m.mount(document.body, new Component({ paramOne: 'Hello!' }));

描画


MithrilはVirtual DOMに基づいており、これはその速度とパフォーマンスを提供する特徴的な機能の1つですが、最小限のフレームワークであることを考慮して-レンダリングシステムにはいくつかの機能があります。 。 たとえば、Mithrilでは、たとえばReactでできるように、単一のノードまたはコンポーネントの再描画を他のすべてのコンポーネントとは別個に制御することはできません。 そして、これは理解にいくつかの困難を課します。したがって、Mithrilでの再描画の仕組みを理解することは価値があります。

ユーザーアクセスで使用可能な再描画には2つのタイプがあります。



m.startComputation / m.endComputation-再描画時に状態を整理することを可能にするメソッド-文字通り-それらはVDOMツリーの最初のスナップショットのセグメントを示し、2番目のスナップショットは両方のスナップショットが比較され、変更が見つかった後、パッチを形成し、データをポイントごとに変更します。 これは、たとえば、非同期データをロードする場合に役立ちます。この場合、操作を開始する前に最初のスナップショットを取得し、結果を取得した後に最後のスナップショットを取得します。 機能のうち、メソッドは特定のコンポーネントで呼び出された場合でもツリー全体をグローバルに比較するため、再描画がツリー内の他のコンポーネントに影響を与えないように開発時に注意する必要があります。 簡単な使用例を次に示します。 

 function asyncGetAndRedraw() { /*      */ m.startComputation(); /*   */ SomeAsyncOperation({ ...args }).then(function promised(data) { /*   (   ) */ setTarget = data; /*     ,    ,   */ m.endComputation(); }); };




m.redrawはハードレンダリング用のメソッドです。このメソッドが呼び出された場合、比較は行われません。メソッドはコンポーネントのツリー全体を完全に再描画します。再描画するときはコンポーネントの既存の状態を考慮しないため、このメソッドには注意が必要です。



コンポーネントでプレーンHTMLを使用する


かなり危険で推奨されない操作ですが、テンプレートの純粋なHTMLの行を挿入する必要がある場合があり、 m.trustメソッドがこれに役立ちます。使用方法を次に示します。 



 m.render("body", [ m("div", m.trust("<p>Clean Html</p>")) ]);
