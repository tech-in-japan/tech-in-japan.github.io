Badoo PHP Code Formatter。 オープンソースになりました！

数年前、Badooは従業員数が20人から100人以上に大幅に増加し始めました。 これには、多くの開発プロセスの大幅な見直しが必要でした。 私たちが遭遇した問題の1つは、すべての開発者が単一のコーディング標準に従うようにして、すべてのコードが一貫性​​があり、簡単に保守できるようにする方法です。



この問題を解決するために、次のことができるコードフォーマットツールを実装することにしました。



  1. ファイル自体に触れることなく、リストの形式でフォーマット標準の非準拠に関するメッセージを表示します。
  2. 見つかったすべてのフォーマットの問題を自動的に修正します。
  3. ファイルの一部のみをフォーマットできます（履歴を失わないように、リポジトリ全体をすぐに再フォーマットする必要はありません）。


このようなツールを作成するための基礎となる2つのプロジェクト、PHP BeautifierとPHP Code Snifferを検討しました。 1つ目はコードをフォーマットできましたが、診断を印刷できませんでした。2つ目は、反対に、診断を印刷できましたが、ファイルをフォーマットできませんでした。 残念ながら、これらのプロジェクトは両方とも、私たちの推定では、それらに欠けていた機能を追加するのにあまり適していないため、新しいユーティリティ-phpcf（PHP Code Formatter）が作成されました。 過去2年間、git pre-receiveフックとして機能しており、コーディング標準に従ってフォーマットされていない変更を拒否（！）するように構成されています。



最後に、ユーティリティのソースコードを一般公開します： github.com/badoo/phpcf



機能性



フォーマッタは、基本的に空白を変更するために作成されました：改行、インデント、演算子の周りのスペースなど。 したがって、phpcfは、前述のPHP Code SnifferやFabien PotencierのPHP Coding Standards Fixerなどの他の同様のユーティリティを置き換えません。 それはそれらを補完し、ファイル内のスペースと改行の正しい配置で「汚い作業」を実行します。 ユーティリティがファイルの初期フォーマットを考慮し、選択した標準に対応しないスペースのみを変更することに注意することが重要です（最初にすべての空白トークンを削除してからフォーマットを開始する他のソリューションとは異なります）。



このユーティリティは拡張可能であり、カスタムスタイルセットをサポートしています。 独自のフォーマットスタイルを非常に簡単に定義できます。これにより、当社とは異なる標準が実装されます（当社のコーディング標準はPSRに非常に近い）。

ユーティリティはPHPで記述されていますが、ほとんどのファイルは一瞬でフォーマットされます。 しかし、大きなリポジトリと多数のコードがあるため、接続すると生産性が100倍に向上する拡張機能を作成しました。200万行のリポジトリ全体が「ノートブック」Core i7で8秒でフォーマットされます。 拡張機能を使用するには、「ext /」ディレクトリから収集し、インストールし、php.iniで「enable_dl = On」を有効にするか、拡張機能として登録する必要があります。



phpcfはまず最初に空白を変更し、コード上で最も単純な変換しかできないことをもう一度強調したいと思います。たとえば、短い開始タグを長いタグに置き換えるか、ファイルから最後の終了タグを削除します。 さらに、phpcfは、英語の文字の関数名のキリル文字を自動的に修正できます。 また、スペースで手動で配置された式は影響を受けません。 これはアーキテクチャーによるものです-フォーマッターは、ハードコードされた置換のセットとしてではなく、ユーザーが設定したルールを持つステートマシンとして機能します（フォーマッターには、フォーマットルールに一致する「デフォルト構成」が付属します）。 したがって、「var」を「public」または同様のものに自動的に置き換える場合は、PHP-CS-Fixerに注意することをお勧めします-phpcfとは異なり、空白文字にはほとんど注意を払いませんが、トークンを書き換えることができます。



PHPバージョンのサポート



当初、フォーマッタはPHP 5.3で動作し、それだけをサポートしていました。 現時点では、PHP 5.4および5.5の構文を完全にサポートしており、フォーマッターが機能するには、5.4以上のPHPバージョンが必要です。 PHPの以前のバージョン向けのコードをフォーマットする場合は、それを行うことができますが、phpcf自体はPHP 5.4+を使用して起動する必要があります。



phpcfは、たとえば不均衡な括弧を含む「未完成」ファイルをフォーマットできないことに注意してください。この場合、エラーメッセージが表示され、ファイルは単にフォーマットされません。 さらに、場合によっては、フォーマッタ自体がファイルの構文をチェックしないため、インタープリターの観点から「無効な」コードをフォーマットできます。



PHPの将来のバージョンのサポートに関して、phpcfアーキテクチャは、ファイルに「なじみのない」キーワードとトークンを追加または会議するとき、それらを単に無視してそのままにするようなものです。 したがって、phpcfはすでにPHPの将来のバージョンをサポートしていますが、未知のトークンにはフォーマットルールが適用されないことに注意してください。



Git統合サポート



ユーティリティをダウンロードすると、「チェック」、「プレビュー」、「適用」アクションだけでなく、「-git」サフィックスが付いた同じアクションもあることに気付くでしょう。 Badooでは、バージョン管理システムとしてGitを使用します。デフォルトでは、変更された行のみがチェックおよびフォーマットされます。 開発者に変更された行の番号を思い出させないために、次のように機能する「* -git」コマンドを作成しました。



  1. インデックスに追加された「コミットされていない」変更を表示します。
  2. 現在のブランチで行われたが、origin / masterおよびorigin / <current-branch>に存在しない変更を表示します（対応するブランチはgit push / git pullで更新されます）。つまり、まだリポジトリに送信されていません。
  3. （1）および（2）にある行にのみ書式設定を適用します。


機能ブランチで開発を使用し、同時にマスターブランチに製品コードがあるため、「-git」コマンドはこのフローに合わせて調整され、上記のアルゴリズムに従って変更された行を決定します。



使用例： 

 (master) $ git checkout -b some_feature (some_feature) $ vim test.php #  test.php (some_feature) $ phpcf apply-git #    ,     test.php formatted successfully

実装



作業の過程で、フォーマッタは次の手順を実行します。



  1. フォーマットするファイル名と行番号のリストを準備します。
  2. token_get_all（prepareTokens）を呼び出して、ファイルのトークンのリストを取得します。
  3. フックを実行する機能を使用してトークンを単一の形式に変換し、一部のトークンを他のトークンに置き換えることができます。
  4. プロセス（）メソッドの呼び出し。これは、状態マシンPhpcf \ Impl \ Fsmを使用してすべてのトークンを渡し、フォーマットアクション（exec）の配列を構成します。
  5. exec（）メソッドの呼び出し。生成されたアクションの配列が処理され、最終的な文字列に変換されます。

トークンフック



$ token_hook_namesプロパティは、prepareTokensメソッドがこのトークンを検出したときに呼び出す必要があるメソッドの名前を定義します。 フックは次のように定義されます。 



 <?php namespace Phpcf\Impl; class Pure implements \Phpcf\IFormatter { /* *  $idx_tokens      array(T_SOMETHING => 'T_SOMETHING'), *       * *  $i_value       ,  token_get_all * *           $this->tokens   each() *   : tokenHookStr * *     ,     *        * *   ,      : */ private function tokenHookDoNothing($idx_tokens, $i_value) { if (is_array($i_value)) { $this->current_line = $i_value[2]; return array( array( PHPCF_KEY_CODE => $idx_tokens[$i_value[0]], PHPCF_KEY_TEXT => $i_value[1], PHPCF_KEY_LINE => $this->current_line, ) ); // set correct current line for next token if it does not have line number $this->current_line += substr_count($i_value[1], "\n"); } return array( array( PHPCF_KEY_CODE => $i_value, PHPCF_KEY_TEXT => $i_value, PHPCF_KEY_LINE => $this->current_line, ) ); } }






トークンのフックの説明



トークンのフックの簡単な説明と、フックの出現理由とそのアクションの説明：



  • tokenHookHeredoc、tokenHookStr：デフォルトでは、PHPはHEREDOC、二重引用符、斜め引用符内のテキストを「トークン化」し、そこで変数を選択します。 フォーマッタは行に触れてはならないため、内容は1つのトークンに結合されます。
  • tokenHookOpenBraceは、括弧内の式が長い（デフォルトでは120文字）か、式で改行が使用されている場合、トークン "（"を "（_LONG"に変換します。「long」配列と「short」配列を区別するために使用され、関数の呼び出しと定義;
  • tokenHookCheckUnaryは、演算子が単項演算子（たとえば、「+」、「-」、＆）かどうかを判別します。 ルール内のコンテキスト間の遷移の数を減らすのに役立ちます。
  • tokenHookStaticは、「static :: HELLO」のような呼び出しを「public static function」としての使用から分離します。 また、状態遷移のロジックを簡素化するのにも役立ちます。
  • tokenHookClassdefは、キーワードの後に改行 （たとえば、「const \ n」）があると判断し、「const \ n var1 = 1、\ n var2 = 2;」などの構造を正しくフォーマットするために使用されることを決定します。
  • tokenHookOpenTagは、開始タグが長いことを確認し、開始タグから空白も分離します（「<？php \ n」を2つのトークンに変換します：「<？php」と「\ n」）。
  • tokenHookCloseTagは、ファイルの最後に終了タグがないことを確認します。
  • tokenHookIncrementは、変数のどちら側が演算子「++」または「-」であるかを決定します。 コンテキスト間の遷移のロジックを簡素化するために作られました。
  • tokenHookWhiteSpaceは、式がスペースで正当化されることを決定し、T_WHITESPACEをT_WHITESPACE_ALIGNEDに置き換えます。T_WHITESPACE_ALIGNEDは、フォーマット時に変更されません。
  • tokenHookElseは、elseが単一行かブロックを含むかを決定します。 ロジックを簡素化するために作られました。
  • tokenHookCommentは、単一行のコメントが「//」で始まることを確認し、トークンから改行を分離します（「// something \ n」は「// something」と「\ n」になります）。
  • tokenHookTStringは、このT_STRINGがメソッドまたは関数の名前である場合、T_STRINGの名前をT_FUNCTION_NAMEに変更します。 コンテキスト間の遷移のロジックを簡素化するために作られました。
  • tokenHookBinaryは、ステートメントを次の行に転送する状況を処理するために使用されます。
  • tokenHookCommaは、長い配列の新しい行にラップできるコンマの場合、「、」を「、_LONG」に変換します。
  • tokenHookFunctionは、匿名関数と非匿名関数を互いに分離します。




フックがトークンの内容を変更する場合は、これを行う権利があるかどうかを確認する必要があります($can_change_tokens = !isset($this->lines) || isset($this->lines[$this->current_line]))



。 そうでない場合、フックはトークンの内容を変更するべきではありませんが、その番号を変更し、トークンをそのコンポーネント部分に分割できます。 例はtokenHookOpenTagです。ユーザーがファイルの一部のみをフォーマットするように要求した場合、開始タグの内容はチェックしませんが、開始タグから空白を分離します。 開始タグと単一行コメントの後の行数（およびインデント）を正しく考慮するために、空白の分離が必要です。
