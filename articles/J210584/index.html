メモ帳でAndroid用HelloWorldを作成、収集、起動します





Androidの学習を始めたとき、IDEを使用せずに、Androidアプリケーションを完全に手動で作成およびコンパイルしたかったのです。 しかし、この作業は簡単ではなく、かなり時間がかかりました。 しかし、結局のところ、このアプローチは非常に有用であり、IDEを隠す多くの微妙な点を明確にしました。



基本的に、この記事は、現在の最新のJDK（7）およびAndroid SDK（APIレベル19） のコマンドラインでのAndroidプログラムの構築記事の翻訳と改訂です。 さらに、私は非常に単純化し、自分から何かを追加しました。



メモ帳のみを使用して、非常に小さな教育用Androidアプリケーションを作成します。 そして、それをコンパイルし、アセンブルして、デバイス上で実行します。これはすべてコマンドラインを使用して行います。 に興味がありますか？ お願いします。



エントリー



Android Studioでテンプレートアプリケーションがどれほど複雑で混乱しているかに驚かされました。 単にリソースが山積みされています。 そして、それほどではありませんが、コードとスクリプトを使用します。 やらなければならないことは、HelloWorldを表示することだけです！ さらに、私が調べた本とマニュアルは、ダイアログボックスを使用してIDEAまたはEclipse HelloWorldを作成する方法を説明します-そして、さらにナレーションがそこから出ています。 そして、「フードの下」で何が起こるか-推測しかできません。



長い検索はほとんど失敗しました-このトピックに関する記事はロシア語で見つかりませんでした。英語では時代遅れの情報があり、 stackowerflowとドキュメントを歩きながらファイルで何かを修正する必要がありました。 私がそれを考え出したとき、私はギャップを埋めることに決めました-誰かが便利になるとどうなりますか？ 実際、この記事が登場しました。



トレーニングに使用するのに理想的な独自のテンプレートプロジェクトを作成します。 余分なものはなく、最も必要なものだけがあります。 そして、それをどのように組み立ててAndroidデバイスで実行するかを詳細に分析します。 記事の最後に、最終プロジェクトとともにアーカイブをダウンロードするためのリンクがあります-質問がある場合は、それを確認できます。



したがって、プロジェクトの構成とアセンブリのプロセスを100％認識し、理解することになります。 このテストプロジェクトは、少し改良されたトレーニングを目的としていますが、実際のプロジェクトの強固な基盤として使用できます。



準備する



まず、 JDK 7とandroid SDKがインストールされている必要があります 。 ちなみに、後者はインストール後に少し調整する必要があります。



この記事を読む前の主な要件は、インストールされているソフトウェアに加えて、EclipseまたはAndroid Studioに付属のHelloworldを既にデバイスで実行できることです。 つまり USBドライバーの構成、デバイスでのUSBデバッグの有効化などが必要です。または、エミュレーターを作成して構成します。 これらは非常に基本的なものであり、その検討はこの記事の範囲外です-ネットワークに関する十分な情報があります。 ところで、本からいくつかの章を読むことも不必要ではありません-マニフェスト、リソース、そして実際にはJava言語の基本がどのように配置されているかを理解してください。 そして、この記事では、どの本が沈黙しているかについて説明します。



プロジェクト執筆



開始するには、プロジェクトのあるフォルダーを作成します。 彼女をmytestappと呼びましょう。 その中に、 bin 、 obj 、 res 、およびsrcの 4つのフォルダーをさらに作成します。



空のテキストファイルを作成し、その名前をAndroidManifest.xmlに変更します。 あなたが初心者のWindowsユーザーである場合、拡張機能の表示を有効にする必要があります。そうでない場合、ファイルはテキストのままです。



以下を追加します。 

<manifest xmlns:android="http://schemas.android.com/apk/res/android" package="com.example.testapp"> <uses-sdk android:targetSdkVersion="19" /> <application android:label="TestApp"> <activity android:name=".MainActivity"> <intent-filter> <action android:name="android.intent.action.MAIN" /> <category android:name="android.intent.category.LAUNCHER" /> </intent-filter> </activity> </application> </manifest>


ここではすべてが簡単です。 起動時にMainActivityクラスを開始するTestAppというアプリケーションを作成する予定です。 この小さなクラスを記述するだけで、アプリケーションの準備は完了です。 必要に応じて、uses-sdkタグのandroid：targetSdkVersionプロパティを編集し、使用しているバージョンを入力します。



次に、最も単純なリソース（文字列Hello test app）を作成します。 実際、この行をJavaコードに直接挿入することにより、リソースなしで実行できます。 ただし、いくつかのアセンブリ手順はリソースで機能します。興味深い点を見るために、引き続きそれらを使用します。



resフォルダーにvaluesフォルダーを作成しましょう。 すべてのリソースはフォルダーに分割する必要があります。 次に-空のstrings.xmlファイルを作成します。 それに書いて 

 <resources> <string name="hello">Hello test app!</string> </resources>


それが私たちが必要とするすべてのリソースです。 シンプルでしょ？ 次に、 src内にcomフォルダーを作成し、その中にサンプルフォルダーを作成し、さらに階層の下位にあるtestappフォルダーを作成します。そして、 MainActivity.javaクラスがあります。 そこにコードを追加します 

 package com.example.testapp; import android.app.Activity; import android.os.Bundle; import android.widget.TextView; public class MainActivity extends Activity { @Override public void onCreate(Bundle savedInstanceState) { super.onCreate(savedInstanceState); TextView textView = new TextView(this); textView.setText(getString(R.string.hello)); setContentView(textView); } }


ディレクトリ構造は次のようになります 

 │ AndroidManifest.xml ├───bin ├───obj ├───res │ └───values │ strings.xml │ └───src └───com └───example └───testapp MainActivity.java

組立



今、私たちは最も重要で困難な段階に来ています。 コマンドラインで多くの作業を行うため、ここで指定したすべてのコマンドを1つのファイルに記述し、 comp.batという名前を付けることをお勧めします。 コマンドの後のファイルの最後に、 一時停止を追加して、結果とエラー（存在する場合）を表示できます。



トラックの準備


便宜上、簡潔にするために最初に行うことは、パスを格納する特別な変数を作成することです。 まず、メインディレクトリを定義しましょう。 JDKおよびAndroid SDKへのパスを現在のものに置き換える必要があります。 

 set JAVA_HOME=C:\prog\sdk\jdk\1.7 set ANDROID_HOME=C:\prog\sdk\android-studio\sdk set DEV_HOME=%CD%


次-プログラムへの直接パス。 SDKのカタログを参照し、すべてが適切に配置されていることを確認することをお勧めします。 また、パスに存在するバージョンを調整します。 

 set AAPT_PATH=%ANDROID_HOME%/build-tools/19.0.1/aapt.exe set DX_PATH=%ANDROID_HOME%/build-tools/19.0.1/dx.bat set ANDROID_JAR=%ANDROID_HOME%/platforms/android-19/android.jar set ADB=%ANDROID_HOME%/platform-tools/adb.exe


ちなみに、古いバージョンでは、aaptユーティリティはplatform-toolsにありました-そして、私はそれや他の誰かが他の場所に忍び込むことができることを除外しません。 だから注意してください。 今すぐすべてをチェックすれば、記事の残りの部分はスムーズに進むはずです。



そしてまだ-いくつかの変数で、パッケージとクラスを強制終了します。 それらを変更する場合-コードを実行する必要はありません-すべての設定が最初です。 

 set PACKAGE_PATH=com/example/testapp set PACKAGE=com.example.testapp set MAIN_CLASS=MainActivity


コンパイル準備


そして今、楽しい部分です。 古典的なJavaプログラムが2行でどのようにコンパイルおよび実行されるか覚えていますか？ ここでは同じjavacを使用していますが、それに加えて、さらに多くのユーティリティを使用する必要があります。



そもそも、不思議なクラスRがどのように機能するのか疑問に思ったことはありますか？ 実際、彼は最初、彼の超能力のせいで私を当惑させました。 コンパイル段階で、クラスのフィールドを介して他のディレクトリのXMLファイルにアクセスするにはどうすればよいですか？ プリコンパイラがここで動作していると仮定しました-判明したように。



実際には、特別なユーティリティAAPTがあります-リソースのディレクトリを調べて、同じR.javaを作成します。 すべてが非常にシンプルであることがわかりました-それは単なるクラスであり、整数定数を持つ他の静的なネストされたクラスで構成されています。 それだけです！ こんな感じ

R.java 
 /* AUTO-GENERATED FILE. DO NOT MODIFY. * * This class was automatically generated by the * aapt tool from the resource data it found. It * should not be modified by hand. */ package com.example.testapp; public final class R { public static final class attr { } public static final class string { public static final int hello=0x7f020000; } }




それではあなたと一緒に作成しましょう。 これを行うには、次のコマンドを使用します。 

 call %AAPT_PATH% package -f -m -S %DEV_HOME%/res -J %DEV_HOME%/src -M %DEV_HOME%/AndroidManifest.xml -I %ANDROID_JAR%


何が何であるかを理解しましょう。 AAPT-Android Asset Packaging Tool-文字通り「Androidプロパティのパッカー」。 そのオプション：

  • パッケージ -リソースをパックする必要があると言います （追加も削除もしません）
  • -f-既存のR.javaを上書きします（存在する場合）
  • -m-R.javaを-Jで指定されたパスのルートではなく、適切なパッケージに配置します
  • -S-このオプションの後、リソースを含むディレクトリを指定します
  • -J-このオプションの後に、結果のR.javaを保存する場所を指定します
  • -I-このオプションの後、接続されたライブラリへのパスを示します-android.jarをオンにします


実行後、同じR.javaファイルがsrcディレクトリに表示されます。 ご覧ください。



プロジェクトに魔法はありません。通常のJavaコンパイラにとっては構文的に完全に正しいです。 コンパイルしましょう。 

 call %JAVA_HOME%/bin/javac -d %DEV_HOME%/obj -cp %ANDROID_JAR% -sourcepath %DEV_HOME%/src %DEV_HOME%/src/%PACKAGE_PATH%/*.java


以前にjavaを使用したことがある場合、この行を理解することは難しくありません。 ただし、それでも何が起こっていたかを説明します。

  • -d-結果のパッケージと* .classファイルを配置する場所
  • -cp -classpatch- android.jarライブラリーへのパスを指定します
  • -sourcepath- * .javaファイルが置かれている（または置かれている）パス


実行後、objフォルダーには、バイトコード（* .class）の形式のクラスを含むパッケージが含まれているはずです。 しかし、これは不運です-Android仮想マシンは互換性がありません！ しかし、それは考案されたものであり、これには理由があります。 また、緑色のR2D2で理解できるクラスファイルからdexファイルを作成するコンバーターを使用します。 

 call %DX_PATH% --dex --output=%DEV_HOME%/bin/classes.dex %DEV_HOME%/obj


--dexオプションは、* .classから* .dexに変換する必要があることを示します。 次に、オプション--outputの後に、結果のclasses.dexファイルへのパスを設定します。 このコマンドの最後の引数は、* .classファイルを含むパッケージへの直接パスです



classes.dexがbinフォルダーにあることを確認します。 これで、リソースと一緒にAPKファイルにパックするだけです。 やってみましょう： 

 call %AAPT_PATH% package -f -M %DEV_HOME%/AndroidManifest.xml -S %DEV_HOME%/res -I %ANDROID_JAR% -F %DEV_HOME%/bin/AndroidTest.unsigned.apk %DEV_HOME%/bin


ここで、オプションはR.javaを作成するときに使用したものと似ています。

  • パッケージ -リソースをパックする必要があると言います （追加も削除もしません）
  • -f-既存のAndroidTest.unsigned.apkを上書きします（存在する場合）
  • -M-このオプションの後、マニフェストファイルへのパスを指定します
  • -S-このオプションの後、リソースを含むディレクトリを指定します
  • -I-このオプションの後、接続されたライブラリへのパスを示します-android.jarをオンにします
  • -F-このオプションの後に、結果のAndroidTest.unsigned.apkを保存する場所を示します
  • 最後の引数は、dexファイルがあるフォルダーへのパスです


AndroidTest.unsigned.apkがbinフォルダーに表示されます。 そして、理由のためにそれを呼び出しました！ 彼にはデジタル署名がありません。 Androidは、署名のないアプリケーションのインストールと実行を禁止しています。 しかし、それを作成することは一見すると思えるほど難しくありません 

 call %JAVA_HOME%/bin/keytool -genkey -validity 10000 -dname "CN=AndroidDebug, O=Android, C=US" -keystore %DEV_HOME%/AndroidTest.keystore -storepass android -keypass android -alias androiddebugkey -keyalg RSA -v -keysize 2048 call %JAVA_HOME%/bin/jarsigner -sigalg SHA1withRSA -digestalg SHA1 -keystore %DEV_HOME%/AndroidTest.keystore -storepass android -keypass android -signedjar %DEV_HOME%/bin/AndroidTest.signed.apk %DEV_HOME%/bin/AndroidTest.unsigned.apk androiddebugkey


実際、これらの行はAndroid SDKとは関係のない2つのJavaユーティリティを実行しますが、これらは必須です。 最初はAndroidTest.keystoreファイルを作成し（存在を確認）、2番目はこのファイルをAndroidTest.unsigned.apkに接続します。 AndroidTest.signed.apkファイルが判明します。 これがそのようなワイルドクラフトファイルです。 ただし、batスクリプトを作成したら、それを実行します。自動モードでこれをすべて実行します。



この記事では、これらのユーティリティのオプションの詳細な分析に時間を費やすべきではないと思います。 本質を把握するだけです。AndroidTest.unsigned.apkを取得し、 AndroidTest.keystoreファイルで署名して、 AndroidTest.signed.apkに保存します。 必要に応じて、 ドキュメントを読むことができます 。



「 警告：-tsaまたは-tsacertが提供されておらず、 このjar ... 」という警告が表示される可能性が高くなりますが、注意は必要ありません 。



打ち上げ


apkファイルがようやくコンパイルされたので、実行できます。 USB経由でデバイスを接続するか、エミュレーターを実行します。 それから 

 call %ADB% uninstall %PACKAGE% call %ADB% install %DEV_HOME%/bin/AndroidTest.signed.apk call %ADB% shell am start %PACKAGE%/%PACKAGE%.%MAIN_CLASS%


実際、プログラムが既に存在する場合、最初の行はプログラムを削除します。 繰り返し起動する場合に便利です。 2番目-デバイスまたはエミュレーターにAPKをインストールします。 3番目のものが起動します。 彼女の議論を詳しく見てみましょう。

  • シェル -デバイスでいくつかのコマンドを実行したい
  • am-アクティビティマネージャーコマンドの実行に使用
  • start-アクティビティを開始したい
  • 開始するパッケージの名前とクラスの完全な名前（パッケージを含む）

おわりに



すべてのファイルをアセンブルすると、ディレクトリツリーは次のようになります。 

 │ AndroidManifest.xml │ AndroidTest.keystore │ comp.bat │ ├───bin │ AndroidTest.signed.apk │ AndroidTest.unsigned.apk │ classes.dex │ ├───obj │ └───com │ └───example │ └───testapp │ MainActivity.class │ R$attr.class │ R$string.class │ R.class │ ├───res │ └───values │ strings.xml │ └───src └───com └───example └───testapp MainActivity.java R.java


これで、低レベルでのAndroidアプリケーションのアセンブリがどのように行われるかを明確に理解できます。 IDEを使用する場合-アセンブリが突然間違った場合（そしてこれが頻繁に発生する場合）-必要に応じて状況を解決できます。 また、最終的なapkファイルは約4キロバイトしか消費しないことに注意してください。



プロジェクトアーカイブをアップロードします （[ファイル]-[ダウンロード]をクリックします）。 そこに別の小さなスクリプト-clear.batを追加したことに注意してください。 アセンブリ中に作成されたすべてのファイルを削除します。 そして、comp.batの最初に彼のローンチを置きます。 また、レムの後にコメントを追加しました-ステップバイステップ。



したがって、スクリプトは、署名を含むプロジェクトの完全なクリーニングと再構築を実行し、デバイスでの削除、インストール、実行を行います。
