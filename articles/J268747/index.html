6.1。 リクエストキャッシュ



同じパラメーターで繰り返し実行されるクエリがある場合、クエリキャッシングによりパフォーマンスが向上します。



キャッシュにより、追加のトランザクションオーバーヘッドが発生します。 たとえば、オブジェクトに関するクエリの結果をキャッシュする場合、Hibernateはオブジェクトへの変更がコミットされたかどうかを監視し、それに応じてキャッシュエントリを破棄する必要があります。 さらに、クエリキャッシングの利点は限られており、アプリケーションの使用パターンに大きく依存します。 これらの理由により、Hibernateはデフォルトでクエリのキャッシュを無効にします。



手順6.1。 クエリキャッシュを許可する



  1. hibernate.cache.use_query_cacheプロパティをtrueに設定します 。



    この設定により、2つの新しいキャッシュ領域が作成されます。



    • org.hibernate.cache.internal.StandardQueryCacheはキャッシュされたクエリの結果を保存します。
    • org.hibernate.cache.spi.UpdateTimestampsCacheは、要求されたテーブルに最近の更新のタイムスタンプを保存します。 これらのタイムスタンプは、クエリキャッシュから取得した結果を検証します（つまり、それらを介して、オブジェクトの関連性がチェックされます- およそTransl ）。


  2. キャッシュ領域のタイムアウトを構成する



    キャッシュの実装でタイムアウトまたは有効期限を構成する場合、 UpdateTimestampsCacheの基礎となるキャッシュ領域のタイムアウトを、要求キャッシュのタイムアウトよりも大きい値に設定します。 UpdateTimestampsCacheリージョンに無期限のタイムアウトを設定することは可能であり、推奨されています。 より具体的には、LRUキャッシングポリシー（最長未使用時間）は適切とは見なされません。

  3. 特定のクエリの結果キャッシュを許可する



    ほとんどのリクエストはキャッシングの恩恵を受けないため、キャッシングが普遍的に有効になった後でも、個々のリクエストに対してキャッシングを許可するだけで済みます。 特定のリクエストのキャッシュを有効にするには、 org.hibernate.Query.setCacheable（true）を呼び出します 。 この呼び出しにより、リクエストは実行前にキャッシュを「見る」か、実行後に結果をそこに置くことができます。



クエリキャッシュは、キャッシュ内の関連するエンティティの状態をキャッシュしません。 識別子の値をキャッシュし、結果を入力します。 したがって、要求キャッシュの一部としてキャッシュする必要があるエンティティに対しては、常に2次キャッシュと組み合わせた要求キャッシュを使用します。



6.1.1。 要求キャッシュ領域



クエリキャッシュの有効期限ポリシーをより適切に制御するには、 Query.setCacheRegion（）を呼び出して、特定のクエリの名前付きキャッシュ領域を指定します。



例6.1。 SetCacheRegionメソッド 



List blogs = sess.createQuery("from Blog blog where blog.blogger = :blogger") .setEntity("blogger", blogger) .setMaxResults(15) .setCacheable(true) .setCacheRegion("frontpages") .list();


クエリキャッシュにその領域の1つを強制的に更新させ、キャッシュされたデータを無視するには、 org.hibernate.Query.setCacheMode（CacheMode.REFRESH）を呼び出します。 クエリ用に定義されたリージョンと組み合わせて、Hibernateはそのリージョンにキャッシュされた結果を選択的に更新します。 これは、 org.hibernate.SessionFactory.evictQueries（）を使用してリージョンからレコードを一括削除するよりも効率的です。



6.2。 レイヤー2キャッシュプロバイダー



Hibernateは、一部の二次キャッシュプロバイダーと互換性があります。 Hibernateで定義されているすべての可能なキャッシュ戦略をサポートするプロバイダーはありません。 セクション6.2.3「Hibernateのレイヤー2キャッシュプロバイダー」には、プロバイダーのリストと、そのインターフェースおよびサポートされるキャッシュ戦略が含まれています。 キャッシング戦略の定義については、 セクション6.2.2「キャッシング戦略」を参照してください。

6.2.2。 キャッシング戦略



  • 読み取り専用



    読み取り専用キャッシュは、読み取られたが変更されていないデータに適しています。 クラスター環境でも安全な、シンプルで機能的なアプローチ。
  • 非厳密な読み書き



    一部のアプリケーションでは、データをときどき読み取るだけで済みます。 これは、2つのトランザクションが同じエンティティを同時に更新する可能性が低い場合です。 この場合、トランザクションの分離を制限する必要はなく、非厳密な読み取り/書き込みキャッシュが非常に適しています。 キャッシュがJTA環境で使用される場合、 hibernate.transaction.manager_lookup_classを指定する必要があります。 他の環境では、 Session.close（）またはSession.disconnect（）を呼び出す前に、トランザクションが完了していることを確認してください。

  • 読み書き



    読み書きキャッシュは、データが定期的に更新されるアプリケーションに適しています。 シリアライズ可能なトランザクションの分離が必要な場合は、読み書き戦略を使用しないでください。 JTA環境では、 hibernate.transaction.manager_lookup_classプロパティを設定して、JTA TransactionManagerを取得する戦略を指定します。 非JTA環境では、 Session.close（）またはSession.disconnect（）を呼び出す前にトランザクションが完了していることを確認してください。



    重要です
    クラスター環境で読み取り/書き込み戦略を使用するには、基盤となるキャッシュ実装がロックをサポートする必要があります。 組み込みキャッシュプロバイダーはブロッキングをサポートしていません。



  • トランザクション



    Transactional Caching Strategyは、JBoss TreeCacheなどのトランザクションキャッシュプロバイダーのサポートを提供します。 このようなキャッシュはJTA環境でのみ使用でき、最初はhibernate.transaction.manager_lookup_classを指定する必要があります。

キャッシュされたレコードを同期または削除します。 flush（）メソッドを呼び出すと、オブジェクトの状態はデータベースと同期されます。 同期を回避するために、 evict（）メソッドを使用して、オブジェクトとそのコレクションを一次キャッシュから削除できます。 セッションキャッシュからすべてのエントリを削除するには、 Session.clear（）メソッドを使用します 。



例6.4 一次キャッシュからエントリを削除する 

 ScrollableResult cats = sess.createQuery("from Cat as cat").scroll(); //a huge result set while ( cats.next() ) { Cat cat = (Cat) cats.get(0); doSomethingWithACat(cat); sess.evict(cat); }


エンティティがセッションキャッシュに属しているかどうかを判断する方法 。 Sessionオブジェクトはcontains（）メソッドを提供して 、オブジェクトがセッションキャッシュに属しているかどうかを判断します。



例6.5 L2キャッシュからの削除

SessionFactoryメソッドを使用して、エンティティのキ​​ャッシュ状態、クラス全体、コレクション、またはコレクションのすべてのロールを削除できます。 

 sessionFactory.getCache().containsEntity(Cat.class, catId); // is this particular Cat currently in the cache sessionFactory.getCache().evictEntity(Cat.class, catId); // evict a particular Cat sessionFactory.getCache().evictEntityRegion(Cat.class); // evict all Cats sessionFactory.getCache().evictEntityRegions(); // evict all entity data sessionFactory.getCache().containsCollection("Cat.kittens", catId); // is this particular collection currently in the cache sessionFactory.getCache().evictCollection("Cat.kittens", catId); // evict a particular collection of kittens sessionFactory.getCache().evictCollectionRegion("Cat.kittens"); // evict all kitten collections sessionFactory.getCache().evictCollectionRegions(); // evict all collection data




6.3.1.2。 2次キャッシュとクエリキャッシュの内容を表示する



統計が解決されると、2次キャッシュまたはキャッシュ領域の内容を表示できます。



手順6.2 統計分解能

  1. hibernate.generate_statisticsをtrueに設定します 。
  2. オプション-hibernate.cache.use_structured_entriesをtrueに設定して、Hibernateがユーザーフレンドリーな形式でレコードを保存できるようにします。
