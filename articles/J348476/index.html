2. ZoneMinder



CCTVシステムを使用した最初のCCTVシステムは、1年以上にわたって機能し、ZoneMinderでした。 正直なところ、ZoneMinderのインストールと設定は非常に複雑ですが、これでも使用の主な問題にはなりません。 主な問題は、古いMJPEGカメラをH.264カメラにゆっくり交換し始めたときに現れました。 今はどうなっているかはわかりませんが、数年前に使ったバージョン（1.28）は「箱から出してすぐに」H.264ストリームをサポートしませんでした。望まれるために多くを残しました。 カメラからのビデオをほぼ連続的に記録するか、動き検出が「粗く」なったときに、「重要な」フレームの省略を我慢する必要がありました。 さらに、明らかにプロセッサの負荷が大きいため、H.264をデコードするffmpegプロセスが低下し、記録がまったくなかったことがよくありました。 他の何かを探す必要がありました。







3. AVReg



一般的に、AVRegに移動しました。 AVRegのインストールは簡単ですが、まだ簡単ではありません。 AVRegはフリーモードで動作したため、インストールされた6台のカメラのうち4台からしか録画できませんでしたが、ビデオ録画はすでに安定しています。 AVregはZoneMinderよりも安定しており、ビデオストリームは「落ちません」。 しかし、設定の複雑さと非自明性および「扱いにくい」インターフェースは少し面倒でした。 動き検出の品質にも多くの要望がありました。 AVRegは大量に記録したため、必要なイベントのアーカイブを検索するのは不便です。 最終的には、私には今思えるかもしれませんが、このプロジェクトのカメラあたり1000ルーブルのライセンスの価格は非常に誇張されています。 チャンネルあたりのAVReg 400ルーブルの適切な値札、これ以上。







一般的に、上記のプロジェクトで作業する場合、多くの時間が費やされ、最終的に判明したことは喜びを引き起こしませんでした。 Linux向けNVRのその他の良い点についてアドバイスを求めてGoogleに時折、ビデオ監視システムを編成するための高品質のソフトウェアに少額の料金を払うことはそれほど悪い考えではないという結論に達しました。







4. Xeoma



最初に見えたのはXEOMAでした 。

NVR用に指定されたソフトウェアには、ホームビデオ監視を整理するために必要なすべてのものが含まれていたため、視線が正常に落ちたと言う必要があります。







  • システムはLinux OS（およびWindows、MacOS、Android、iOS、ARM）で動作します。
  • サーバーパーツは、ベアコンソールでクライアントパーツとは別に起動できます（実際には、ネットワーク経由でのサーバーへの接続に制限がある無料バージョンに加えて）。
  • クライアントインターフェイスは、任意のOS（Linux、Windows、MacOS、Android、iOS、ARM）からサーバーに接続できます。
  • 膨大な数のカメラのサポート。
  • 簡単なインストールとセットアップ。
  • PTZコントロール。
  • 高品質のモーション検出器。
  • Webインターフェイスを有効にして使用すると、システムにアクセスできます。
  • 多数の高品質のビデオストリーム処理モジュール。

4.2。 Xeoma CCTVコンソールサーバーの起動



XEOMAサーバーは単に起動されるだけでなく、非常にシンプルです。 正直なところ、特にZoneMinderとAVRegでの試練の後、Linuxにシステムをインストールして起動するプロセスに非常に驚きました。

ここではすべてが簡単です：







  • Linuxを最小バージョンでインストールします。
  • Xeomaの必要なバージョンをポンプで送り出します（リンク： http ://felenasoft.com/xeoma/ru/download/）

    wget http://felenasoft.com/xeoma/downloads/xeoma_trial_standard_linux64.tgz

  • ダウンロードしたアーカイブを解凍します。

    tar xzf xeoma_trial_standard_linux64.tgz

  • 必要に応じて、起動を許可するアプリケーションの属性を設定します

    chmod u + x xeoma_trial_standard.app

  • たとえば、Windowsワークステーションからサーバーに接続するための管理者パスワードを見つけます。

    ./xeoma_trial_standard.app -showpassword

  • サーバーモードでXeomaを起動します（対話モードでの確認のため）。

    ./xeoma_trial_standard.app -core

  • Xeomaの自動起動を設定し、サーバーパーツをLinuxサービス（デーモン）として実行することも驚くほど簡単です;前のコマンドの代わりに、次のことを行う必要があります。

    ./xeoma_trial_standard.app -install -hiddenmode

4.5.1。 動き検出器


一般に、Xeomaが提供するデータの検出とアーカイブのデフォルト設定は、変更なしでデフォルトで正常に機能しますが、Xeomaサーバーを使用する仮想マシンのリソースの制限、および冗長情報を保存する必要がないことを考えると、検出モジュールの構成にすぐに小さな変更を加えることをお勧めしますムーブメント "：













感度のしきい値を変更します。 ビデオフレームのさまざまな変化における現在の感度レベルの読み取り値を少し観察し、最小感度しきい値を決定する必要があります。 検出器がカメラの動きを見逃さないという値10に満足しています。 デフォルトで設定されている5の値は、カメラの「ノイズ」と照明のわずかな変化を実際にキャプチャし、ビデオアーカイブに大量の記録をもたらします。

検出エリアを変更します 。 ブラシと「消去」モードを使用して、動きの修正に関心のない余分な領域を削除します。

4.6。 仮想化の使用に関する制限



Xeomaは仮想環境で動作し、製造業者は誰かにとって重要な制限を確立しています。 特に、完全なシステムマニュアルには次のことが書かれています。







Xeoma（Xeoma Standard）、Xeoma Proライセンス、Xeoma StandardからXeoma Proへの移行および更新の場合、任意の仮想マシンでアクティブ化する機能が追加されます。 これを行うには、仮想マシンでインターネットに常時アクセスする必要があります。 インターネットに障害が発生した場合、ライセンスは無効化される場合がありますが、インターネット接続が再開されると復元されます。

次のように機能します：インターネットが消滅すると、仮想マシンのXeomaサーバーは2日間動作し続け、何も起こらなかったように動作し続け、アーカイブを書き込み、クライアントが接続できるようにしますが、サーバーが再起動するまですべて動作します。







再起動時にサーバーがライセンスアクティベーションのためにインターネットに接続できない場合、サーバーの生産的な作業は停止します。 カメラからデータが受信されない場合、アーカイブは記録を停止します。 クライアントを起動しようとすると、次の警告が表示されます。













これで、インターネットXeomuができるまで、アーカイブの表示にのみ使用できます。 これを行うには、[作業を続行]ボタンをクリックします。







ただし、インターネットへのアクセスを再開した後、1つのトラブルが待っています（少なくとも最新バージョン11.17.24では）。 インターネットへのアクセスが再開されると、Xeomaの機能は完全に動作しますが、アクティブライセンスに関する情報（メニュー>情報>アクティブライセンス）を表示すると、次のように表示されます。













1日ほど待っても状況は解決しなかったため、ライセンスアクティベーションコードを手動で再入力する必要がありました。







仮想マシンで実行されるXeomaの完全に明確なロジックではないため、2つの方法のいずれかで保存することにより、以前に行った設定をリセットしないように保護することをお勧めします。







  • XeomaクライアントGUIのメニューを使用：メニュー>インストール>復元>設定の保存（または設定のエクスポート）。
  • Linux設定ファイル/usr/local/Xeoma/settings.datを使用してコピーを作成します
