MODx Revoワークフロー。 ワークフローの構成、バージョン管理、展開

チャンク、テンプレート、スニペットなど、MODXシステムのすべての基本的な要素はデータベースに保存されます。これにより、これらの要素のバージョン管理の問題が発生するだけでなく、サイトを開発バージョンと本番バージョンに分割することが困難になります。



MODX Revoのワークフローに必要な基本要件は次のとおりです。

  • コード（html、css、js、php）を記述する場所はどこでもバージョン管理、
  • 現在の開発が進行中のサイトの別の開発バージョンを用意し、その後、本番環境にすべての変更を展開し、できればこのプロセスを自動化してください。
  • 開発および展開中の最小限のコピーアンドペースト。

はじめに



この記事を書いている時点で、私はわずか1年以上にわたってWeb開発の経験がありました。 幸運なことに、この分野での旅を始めたのはMODX Revoでしたが、このフレームワークのすべての利点にもかかわらず、時間の経過とともにマイナスに出会うようになりました。 これまでのところ、私は大規模で複雑なプロジェクトのチームで働いたことがなく、人々が通常どのように上記の困難に対処するかを知りません。 インターネット上で特定のソリューションを見つけることができなかったため、この記事で紹介したいワークフローの作成に貢献しました。 すぐに予約しますが、根本的に新しいものは思いつきませんでしたが、インターネットに散らばった情報を集めて、MODX Revolutionを使用する際のワークフローの整理方法に関する1つのガイドにまとめました。



何が必要ですか。

-git

-npm + gulp

仕事には、IDE PHPStormを使用します。





サイトの開発バージョンと本番バージョン



サイトの作業（test、 dev ）およびリリース（combat、 prod ）バージョンを整理する方法を考えるときに最初に思い浮かぶのは、単に2つのコピーを作成することです。 まず、テストサイトで開発し、マネージャーまたはクライアントの承認を得て、すべてを戦場に移します。 シンプルで明確なのは、このプロセスだけが悪夢に変わる可能性があるからです。 すべてが手動で行われ、転送中にエラーが発生する可能性がかなり高いため、間違ったことを確認する必要があります。その後、バトルサイトがテストサイトと完全に一致せず、他の何かを編集する必要があるなどです。 など いいえ、必要ありません。



長い間議論するつもりはありませんが、MODX Revoコンテキストメカニズムを使用するサイトのコピーを1つ使用することを提案するとすぐに言います。 標準のWebコンテキストには、サイトのリリースバージョンが含まれます。 さらに、別の開発コンテキストを作成し、サブドメインで使用します。 必要なのは：

-Webのコンテキストで動作するexample.comドメイン

-devのコンテキストで動作するサブドメインdev.example.com

デプロイするときに、 devのコンテキストで行った変更がWeb上で何らかの形でマージされる必要があります 。



開発コンテキストを作成する



作業のこの部分は、エンジンと必要なコンポーネントをインストールし、ドメイン名を設定した瞬間から始まります。 example.comとdev.example.comの両方のアドレスにアクセスすると、同じMODXスタートページが開きます（両方のドメインのルートディレクトリが共通です）。



次に、管理パネル→システム設定→コンテキストに移動して、新しいコンテキストを作成します。 キーはdevを指定し、好きな名前を付けて、私はDevelopmentと呼びました。 左側のコンテキストメニューに新しいコンテキストが表示されます。 まず、そのためのリソースを作成し、コンテキスト設定に移動して（コンテキストを右クリック→編集→コンテキスト設定）、設定を設定します

-site_start

-error_page

- 無許可ページ 、

作成されたリソースのIDを指定します。 これは、 devコンテキストで作業しているときにページが見つからない場合にシステムがエラーでクラッシュしないようにするために必要です。





約 dev（開発）コンテキストを作成し、標準のWebサイトコンテキストをProductionに変更しました。標準のコンテキストキーは同じままです-web。



フォルダーcore / elements / common / plugins / （必要なフォルダーを作成）で、次の内容のswitchContext.phpファイルを作成します 

<?php /** * @var modX $modx */ /* don't execute if in the Manager */ if ($modx->context->get('key') == 'mgr') { return; } switch ($_SERVER['HTTP_HOST']) { case 'dev.example.com': $modx->switchContext('dev'); break; case 'example.com': break; default: $modx->log(modX::LOG_LEVEL_ERROR, 'Check this plugin! May be your headache coming from here.'); break; } return;


プラグインの機能について説明します。 管理パネルからサイトを操作する場合、つまり mgrのコンテキストでは、何もしません。 ドメインdev.example.comのサイトのフロントで作業する場合、コンテキストをdevに切り替えます。 メインドメインのフロントエンドで作業する場合、何も行いません。 ただし、スクリプトの実行が何らかの理由でdefaultになった場合、ログにエラーが表示されます。 これは、たとえば、サイトを新しいドメイン名に移管するときに発生する可能性があります。このメッセージは、移管後に何も機能しない理由を理解する開発者の作業を楽にするためのものです。 ここで説明する操作スキームをリモートサーバーで確認したとき、このプラグインのドメイン名を修正するのを忘れていたので、ログを間違えて間違っていることがすぐにわかりました。 もう一度、彼は自分に言いました、「ありがとう」。



プラグインファイルの最後にreturnステートメントを配置する必要があることに注意してください。プラグインが何も返さない場合、これはプラグインファイルを接続するときに使用するincludeステートメントの戻り値を上書きするために必要です。



管理パネルから、作成されたファイルをプルする新しいプラグインを作成します 

 <?php $filepath = 'plugins/switchContext.php'; $context = 'common'; $plugin = $modx->getOption('pdotools_elements_path') . "$context/$filepath"; if (file_exists($plugin)) { return include $plugin; }


OnHandleRequestイベントでプラグインの起動を停止します。



これで2つのコンテキストができました。それらを今後の開発で使用する方法を以下に示します。





バージョン管理



GitHubなどにプロジェクトリポジトリを作成しますが、作業ディレクトリには、エンジンに関連しないファイルとフォルダーのみがあります。 このようなもの 

 assets/ |-- dev/ |-- css/ |-- js/ |-- img/ \-- scss/ |-- web/ |-- css/ |-- js/ \-- img/ core/ \-- elements/ |-- common/ |-- plugins/ |-- dev/ |-- chunks/ |-- plugins/ |-- snippets/ \-- templates/ |-- web/ |-- chunks/ |-- plugins/ |-- snippets/ \-- templates/ \-- plugins/ .gitignore gulpfile.js package.json


同時に、展開に失敗した後にロールバックできるように、アセンブリ（ asset / web /およびcore / elements / web / ）から生成されるファイルも制御します。 以下、このフォルダ構造について説明します。



開発者は、通常どおり、ローカルリポジトリで作業し、コミットをリモートリポジトリにプッシュします。 さらに、リポジトリはサイトのルートに存在する必要があります。

サンプル.gitignoreファイル 

  # IntelliJ project files .idea/ *.iml out gen # ignore MODX files /assets/* /connectors/* /core/* /manager/* /setup/* /config.core.php /index.php /.htaccess # ignore node modules /node_modules/ # do not ignore !/assets/dev/ !/assets/web/ !/core/elements/






作業スキーム



上記の開発バージョンと製品バージョンについて話したとき、サイトのコピーを1つ使用することを提案しました。 これは、ローカルコピーがないことも意味します（プロジェクトのローカルgitリポジトリと混同しないでください）。 ローカルマシン上のファイルを編集しますが、この場合、ローカル作業ディレクトリには、エンジンに関連しないファイルとフォルダーのみがあります。 既存のファイルに変更を加えてから、これらのファイルをサイトにアップロードします。 タスクを完了した後、ローカルリポジトリでコミットします。その後、必要に応じて、変更をGitHubにプッシュするか、プロジェクトをそこに保持します。



次に、システムの主要な要素（テンプレート、チャンク、スニペット、プラグイン）のバージョン管理をどのように調整するかを提案し、IDE PHPStormを介してMODX管理パネルと連携してワークフローを説明します。 伝説的なpdoToolsとそれに使用されるFenomテンプレートエンジンは、これに役立ちます。



ご存じのように、pdoToolsパーサーを使用すると、チャンクの本体で外部ファイルを直接接続できます。この機能は、バージョン管理デバイス全体の基礎です。

pdotools_fenom_parser設定（ページでFenomを使用）をYesに設定する必要があります



pdoToolsコンポーネントのシステム設定はpdotools_elements_pathで、デフォルト値は{core_path} elements /です。 それで、それに応じて、 コア/エンジンフォルダーに要素/フォルダー（まだ作成されていない場合）を作成し、内部に次のフォルダー構造を作成します。 

 elements/ |-- common/ |-- plugins/ |-- dev/ |-- chunks/ |-- plugins/ |-- snippets/ \-- templates/ |-- web/ |-- chunks/ |-- plugins/ |-- snippets/ \-- templates/ \-- plugins/


実際には、開発はdev /フォルダーで実行され、チーム全体にとって重要なルールが有効になっています：「誰もWeb /フォルダーを変更しないでください」。 デプロイするとき、 dev /フォルダーのコンテンツ全体がGulpを使用してweb /にコピーされます。 common /フォルダーには、上記の現在のコンテキストを切り替えるプラグインなど、両方のコンテキストに共通のファイルが含まれています。



このフォルダー構造により、Fenomテンプレートエンジンを使用して外部ファイルを接続するときに、プレースホルダー[[* context_key]]の値を使用できます。 

 {set $ctx = $_modx->resource.context_key} {include "file:$ctx/chunks/common/head.tpl"}


1行目では、現在のコンテキストキー（ devまたはweb ）を$ ctx変数に追加し、2行目では、この値をファイルパスで使用しています。 これにより、1つのエンジンで2つのバージョンのサイトを使用できます。





約 テンプレートファイルのチャンク接続



システムの主要な要素を含むファイルに加えて、レイアウトファイルとjsファイルのバージョン管理も制御する必要があります。 通常、これらのファイルは次のようにasset /フォルダーにあります。 

 assets/ |-- css/ |-- js/ \-- img/


古いスキームによると、次の構造を作成することを提案します 

 assets/ |-- dev/ |-- css/ |-- js/ |-- img/ \-- scss/ |-- web/ |-- css/ |-- js/ \-- img/


レイアウトおよびスクリプトファイルを使用する場合、次の点を考慮する必要があります。 たとえば、 dev / js /フォルダーには複数の.jsファイル（たとえば、 calendar.module.jsおよびsearch-form.module.js ）がありますが、 実稼働環境にデプロイする場合、これらのファイルを1つにマージして縮小する必要があります（たとえば、 main .bundle.min.js ）、チャンクに必要なリンク[[* context_key]]を取得して入力することはできません。このような操作を行い、そのような構造を展開する必要があります 

 [[*context_key:is=`dev`:then=` <script src="[[++assets_url]][[*context_key]]/js/calendar.module.js"></script> <script src="[[++assets_url]][[*context_key]]/js/search-form.module.js"></script> <!--   --> `]] [[*context_key:is=`web`:then=` <script src="[[++assets_url]][[*context_key]]/js/main.bundle.min.js"></script> `]]






リソースをテンプレートに関連付ける



フォルダーcore / elements / dev / templates /にテンプレートを作成し、Fenomテンプレート構文を使用してフォルダーcore / elements / dev / chunksからチャンクを接続します。 テンプレートファイルを作成すると同時に、管理パネルでそれに対応するテンプレートも作成する必要があります。 テンプレートが使用されるコンテキストに依存するため、ファイルへの特定のパスを指定できないため、静的テンプレートのみを作成しません。 代わりに、テンプレートの本文に、1行を記述します 

 {include 'file:[[*context_key]]/templates/base.tpl'}


したがって、このテンプレートは、さまざまなコンテキストのリソースの一種のテンプレートファイルスイッチとして機能します。





約 テンプレートファイル接続



プラグインを作成する



この開発アプローチでは、システムへの新しいプラグインの導入を個別に規定する必要があります。

テンプレートの場合と同様に、フォルダーcore / elements / dev / plugins /にプラグインを作成し、そこにプラグインのロジックを記述します。 次に、管理パネルに移動し、対応するプラグインを作成します（静的ではありません！）、次のようにincludeを介してプラグインファイルを接続します 

 <?php $filepath = 'plugins/somePlugin.php'; $context = $modx->context->get('key'); $plugin = $modx->getOption('pdotools_elements_path') . "$context/$filepath"; if (file_exists($plugin)) { return include $plugin; }


もちろん、管理パネルでプラグインがハングするイベントを設定することを忘れないでください。 両方のコンテキストに共通のプラグインが必要な場合は、switchContextプラグインで行われたように、値「common」をコンテキスト変数に追加します。





展開する



だから、私たちは最も重要な瞬間に来ます。 実稼働用のサイトのテストバージョンを展開しています。 まず、サーバー上のリポジトリでいくつかのアクションを実行する必要があります。つまり、SSHキーを作成し、登録して、GitHubのリポジトリ設定に登録します。 ここでは、このプロセスについては説明しません。 以下にキーを生成および登録するためのソースへのリンクを残します。

そのため、都市は眠りに落ち、開発者は作業を停止し、すべてのファイルがコミットされてリモートリポジトリにアップロードされます。 ティムリッドが目を覚ますと、彼の仕事は、リリースに対して行われたすべての作業を可能な限り迅速に展開することです。 この手順を提案します。

  1. ローカルマシンで、ファイルをリモートリポジトリと同期して、ローカルリポジトリ内のすべてのファイルが最新であることを確認する必要があります。
  2. たとえば、「v1.0.666-pre」などのラベルを作成し、v1.0.666のビルドに先行するコミットをマークして、展開が失敗した場合に迅速かつ便利なロールバックを行います。 

     $ git tag v1.0.666-pre


  3. Gulpを使用して、プロジェクトの新しいバージョンを収集しています。 

     $ gulp modx:build


    gulpファイルの例を示します。 ここでは、ファイルをcore / elements / devからcore / elements / web /にコピーするためのタスクを1つだけ説明しました。残りのタスクは、間違いなく自分で書くことができます。 

      'use strict'; var gulp = require('gulp'); var paths = { dist: { // output js: 'assets/web/js/', css: 'assets/web/css/', img: 'assets/web/img/', fonts: 'assets/web/fonts/', modx: 'core/elements/web/' }, src: { // sources js: 'assets/dev/js/**/*.js', style: 'assets/dev/sass/style.scss', img: 'assets/dev/img/**/*.*', fonts: 'assets/dev/fonts/**/*.*', modx: 'core/elements/dev/**/*.*' }, watch: { // files watch to js: 'assets/dev/js/**/*.js', style: 'assets/dev/sass/**/*.scss', img: 'assets/dev/img/**/*.*', fonts: 'assets/dev/fonts/**/*.*', modx: 'core/elements/dev/**/*.*' } }; gulp.task('modx:build', function () { return gulp.src(paths.src.modx) .pipe(gulp.dest(paths.dist.modx)) });


  4. アセンブリの結果、 / asset / web /および/ core / elements / web /フォルダーで新しいファイルが追加されるか、古いファイルが変更されます。 コミットし、ラベルを割り当て、 -tagsフラグでプッシュして、作成されたラベルをリモートリポジトリに送信します。 

     $ git add . $ git commit -m "Build v1.0.666" $ git tag v1.0.666 $ git push --tags


  5. SSHを介してサーバーに接続し、サイトのルートディレクトリに移動して、リモートリポジトリからデータをプルアップし、 ハードリセットを実行して、すべてのサイトファイルをリポジトリに格納されている形式に戻します。 

     $ git fetch --all $ git reset --hard origin/master


  6. 生産での結果を見ます。 開発とプログラミングの世界がどのように機能するかを正しく理解すれば、10件のうち約10件で、何かがおかしくなります。 したがって、コンソールに戻り、コマンドを実行します 

     $ git checkout v1.0.666-pre


    つまり 実動ファイルをビルド前の状態に転送し、何が問題なのかを理解し始めます。







約 コミットツリーの例。



上記では、テンプレート、チャンク、スニペット、およびレイアウトファイルの展開について説明しましたが、リソースについてはどうですか？



また、リソースについてはどうですか？



この問題を考慮してください。 このサイトにはニュースセクションがあり、 やがて出版物の数が非常に多くなり、サイトのニュースアーカイブの新しいセクションを作成することが決定されました。



開発コンテキストでは、適切なコンテナを作成し、アーカイブされたニュースのインスタンスをいくつか追加し、テンプレートが必要な場合はチャンクを作成します。 その結果、 dev.example.comドメインに新しいセクションが表示されます。 顧客による承認後、上記の展開を行いますが、もちろん、すべてのファイルが必要な形式になりますが、本番環境の新しいセクションは表示されません。 もちろん、これは、作成されたセクション（作成されたリソースの全体を意味する）がdevのコンテキスト内にあり、 Webのコンテキストでは使用できないために発生します。

-そして何をすべきか？

-コピーペースト、同志、悲しいかな。

むしろ、転送。 作成したセクションをWebコンテキストに転送してからdevでそのコピーを再作成するだけで、サイトのdevバージョンの構造が実稼働に対応します。

はい、コピーペーストを完全に取り除くことはできませんでした。
