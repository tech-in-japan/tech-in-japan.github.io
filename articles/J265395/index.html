背景



新しい年の後、私は興味深いプロジェクトを探していました。 暇なときにジョイリアクターを読んで、面白いゲームに出くわしました。このゲームは、サイト全体の聴衆がAlanWakeというニックネームの男と一緒に行ったものです。 ゲームは開発オプションを備えたコミックで、訪問者がコメントにコメントを追加して投票しました。 受賞オプションが開発されました。 数回のリリースの後、この宇宙の「ファンメイド」バージョンが登場し、最終的にすべてが制御不能になりました。 7番目の問題までに、著者自身は何が起こっているのか混乱していた。 コントロールができず、ファンは追加のストーリーも考慮に入れたいと考えていました。

「Communication Channel Closed」の第8号。

わずか数週間で、このゲームは4,000人以上の加入者を集めました。 私は、本格的なゲームを開発し、何が起こるかを提案することを著者に書きました。





プラットフォームと対話ツールの選択



ブラウザ、モバイルアプリケーション、Steam（またはその他のプラットフォーム）ゲームの選択肢がありました。

長所と短所を比較検討した後、選択はブラウザーオプションに委ねられました。ほとんどのデバイスでほぼ同じように機能し、その後ゲームをアプリケーション（モバイルとデスクトップの両方）に変換でき、ゲームを開始するために最小限のアクションが必要です。

互いに対話するためのツールとして、彼らはリアルタイムボードを選択しました。 リアルタイムボード-テキストを書いたり、絵を投稿したり、絵を描いたりできるオンラインボードです。 ファイルは、 dropbox内のフォルダーを介して同期されました。

ゲームは構造を取り始めました。







ゲームの要件と形式をすぐに決定しました。

似たような多くのゲームについて話し合い、最高のものをハイライトし、不快な瞬間を排除しようとしました。

要件：

  • これはPvEゲームです。
  • ゲームは周期的です（約1年のサイクル）。 この間、プレイヤーは勝ち負け（すべて）
  • イベントの間に漫画を描く時間があるように、ゲームを測定する必要があります
  • ゲームは、より多くの人にコンテンツを提供しながら、常に存在してアラームを鳴らす必要はありません。
  • 寄付はゲームプレイに重要な影響を与えません。
  • 追加のアカウントは特典を提供しません
  • ゲームは私たち自身に好かれるべきです






開発開始



数日で、 bootstrapを使用してノード上でプロトタイプが作成され、そこで登録、鉱山の構築、およびリソースの流れを楽しむことができました。 私たちが正しい方向に進んでいることが明らかになったとき、私はすべてを新しい流星で開発することを決定しました（後で無駄ではなく判明しました）、ゲームに問題があったとしても、少なくとも良いことがあると主張しました新技術の知識。

彼は職場環境を広げ、流星を設置し、書き始めました。

私はすぐに流星が好きでした：

  • ワンチームスイープ
  • これまでになく簡単に接続できる既製のライブラリがたくさんあります（つまり、ノードはそれほど複雑ではありません）
  • コンテンツ（画像でも）を変更するときのページの自動更新
  • 環境に応じた美しさ/コーナーコード
  • デバッグツール
  • ブラウザコンソールからデータベースに書き込む機能（本番環境では無効）
  • クライアント、サーバー、および一般へのコード侵入。
  • 反応性 すべての顧客に対するデータベースの変更の即時表示。




私は最後のものが好きではなかった、なぜなら もっと細かく制御したいのですが、結局のところ、ドキュメントを読んでいないだけで、すべてが微調整されています。

マイナスのうち、私はリングリンクを使用できないことにのみ注意できます。 もちろん、どんな問題でもそれらなしで解決できると言うことができます...はい、できます。 しかし、これはgotoは常に悪いと言うのとまったく同じです（ほとんどのプログラマーにとっては、使用方法がわからないので悪いです。はい。既に多くの記事に記載されています）。





リソースバランス



しばらくしてから、残高を計算します。 ゲームの「期間」で、私たちはすぐに決定したので、それが基礎として採用されたのは今回でした。 最大1年後、ユーザーは既に「すべて」を持っているはずです。 原則として、上限ははるかに速く達成できますが、一般的に、計算は一般ユーザー向けでした。

建物のコストを計算するために、それらはテーブルに書き出され、「ブルドーザーから」の最大レベル表示値が示されました。 生産を計算するための簡単な指数関数式を書きました。 最初と最後の（最終レベルの）制作の希望がありました-それらは単に「快適な」数字として私の頭から取られました。







すべてが少なくとも何らかの形で表にリストされた後、価格と建設時間の変化に応じて（最初のレベルから最後のレベルまで）、コストと生産の係数を選択し始めましたが、その他の要因。

係数が選択されるとすぐに、予想される開発スケジュールが取得されました。 少し調整した価格、式、そして実装できます！

私はすぐに、テストに応じてバランスを変えることができるようにしようとしました。 初期値と最終値を維持しながら、開発の「滑らかさ」に影響するレベルの数を増減します。

3つの主要なリソース（人、金属、水晶）と2つの追加リソース（名誉、汚い銀河系クレジット）があります。 ゲームのステージによっては、特定のリソース（クレジットを除く）が不足していますが、すべてを正しく計画すれば（一部のテスターがなんとかして）、ダウンタイムなしで実質的に開発できます。

技術部



流星では、コードの分離は非常に便利です。クライアント部分（クライアント）、一般部分（lib）、サーバー部分（サーバー）です。 共通部分は、プロジェクトの両方の部分で等しく利用できます。これにより、ゲームのすべての基本オブジェクトをそこに入れることができます。







CSS


スタイルには、 スタイラスを使用しています。 第一に、余分なものがすべて削除された素晴らしい構文があり、第二に、追加の言語なしでノード上のcssにアセンブルされます。 さて、完成したモジュールは再構築し、その場でスタイルをページにロードします。 スタイルは関連ファイルによって分類されますが、最も「太字」のメインファイルであり、すべてがまだレイアウトされていません。リファクタリングの時期が来ます。



HTML


すべてのテンプレートは、流星に組み込まれたスペースバーテンプレートエンジン用に作成されています。 論理のないテンプレートエンジンはあまり好きではないので、多くの点に我慢しなければなりませんでした。 外観のみに直接関係するすべてのものをテンプレートで直接記述することができると考えています。 単純な比較ごとに、ヘルパーを開始する必要があります。 通常の比較でも： 

 UI.registerHelper('eq', function (a, b) { return a === b; });


まあ、明らかにこれは反応性による制限です。 しかし、それ以外の場合、テンプレートエンジンは優れています。 それは可視領域でうまく機能します;テンプレートを変更することは簡単になります;それらを書くことも難しくありません。



ルーター


鉄のルーターを使用して、アプリケーションパスを決定しました。 かなり便利でよく作られたルーターです。リンクをクリックすると特定のアクションを停止でき、パスを柔軟に記述でき、必要なデータをヘルパーだけでなくテンプレートにすぐに転送できます。 

 Router.route('/logout', function () { Meteor.logout(); this.redirect('index'); });




コントローラー


完全に正しいわけではありません。 前面にコントローラーはありません。 ヘルパーとイベントを各テンプレートに添付できます。 ヘルパーはテンプレートに値を返し、イベントはその中のアクションを担当します。 原則として、ルーターはページコンポーザーとして機能します。

もちろん、論理要素を別のファイルに分類する人はいません。それらをグループ化すると便利です。

エフェクトシステム


私が作ったエフェクトシステムは本当に好きです。 各オブジェクトには、1つまたは複数の効果があります。 効果は、生産に影響を与える、価格（および時間）に影響を与える、戦闘特性に影響を与える、ユニークな特殊効果の4種類に分けられます。

改善点の1つの効果の説明の例を次に示します。 

 new Game.Effect.Military({ pretext: '+', aftertext: '%   ', condition: { type: 'unit', group: 'ground', special: 'infantry' }, priority: 2, affect: 'life', result: function(level) { level = level || this.currentLevel(); return level * 30; } })




値の前後にテキストを記述し、多くの基準に従って効果が適用されるオブジェクトを記述することができます（基準を指定しない場合、効果は適用可能なすべてに影響します）。 優先度-最終値を計算する順序：奇数の優先度には固定値、偶数の割合があります。 つまり 優先度1で+10アーマーを与え、優先度2で30％を与え、その後に優先度3でさらに+5を与える効果がある場合、（基本アーマー+ 10）* 1.3を得る+ 5。

同じ優先度を持つ複数の効果がある場合、それらは最初に合計されてから適用されます。 たとえば、それぞれ10％の2つの効果は乗数（1 +（0.1 + 0.1））を与えます。

特殊効果は、ゲームの固有のパラメーターに影響を与える可能性があります。たとえば、ハイパーモーターの充電時間を短縮できます（1箇所のみにあります）。

オブジェクトに適用可能なエフェクトのリストを取得し、好きなように操作できます。 たとえば、人口の流入を構成するものを推測する。

ミッション

タスクを、タスクの行、トリガータスク、毎日のタスクに分割しました。 実装の面で最もクールな機能は、ゲーム内のすべてのイベントにアタッチできることです。なんとかしてタスクと実行条件を発行します。

これは簡単な方法で達成されました-タスクには、タスクが完了したかどうかにかかわらずtrue / falseを返す関数があります。 例： 

 isDone: function() { return Game.Buildings.has('residential', 'house'); }


したがって、木曜日に雨が降った後でも、チャットメッセージに参加したり、ゲームに参加したりすると、イベントをハングさせることができます。



現時点では、毎日のタスクは単なる調査であり、いくつかの回答オプションと、一部はランダムで、一部は以前のアクションに依存する分離オプションがあります。



ベースで作業する


デフォルトでは、流星にはMongoDBが付属しています。 これは非常に便利なドキュメント指向のデータベースです。 拡張が容易で、クエリの記述が簡単で、変更が簡単です。 すべてのデータをJSONで保存し、厳密な構造なしで、フィールドをオンザフライで追加/削除できます。実際、移行は不要です（古いレコードの新しいフィールドに値を入力する必要がある場合を除く）。

特定のコレクションを操作するために、別のクラスを設定しました。必要なのは、そのコレクションを操作することだけでした。 たとえば、リソースを償却するには、Game.Resources.spend（price）を呼び出すだけです。
