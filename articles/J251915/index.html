トリガーメーリング

最近、電子メールマーケティングでは、特定の消費者グループへの自動郵送がますます使用されています。 典型的なタスク：
  • お誕生日おめでとう
  • 消費者が長い間サイトを訪れていない場合、サイトを呼び出します
  • パーソナライズされたオファーを行う（消費者をセグメントに分割し、各セグメントに手紙を送る）


この記事では、この問題をどのように解決したかを説明します。3年前に開発者が個々のメーリングリストを最初から作成してから、現在、Webインターフェイスを介してマネージャーがメーリングリストを設定します。 このストーリーは、電子メールマーケティングに携わる人だけでなく、特定の消費者サンプルで定期的に複雑な操作を実行する必要があるすべての人にも興味があるかもしれません（非常に抽象的に聞こえるかもしれませんが、最終的にはこのような抽象的な問題を解決しなければなりませんでした）。



最初の実装

3年前、そのようなタスクは非常にまれにしか発生せず、ゼロから実装するたびに発生しました。 この場合、同じ質問が発生しました：
  1. このメールを既に送信した消費者にタグを付ける方法は？
  2. すべての消費者をできるだけ迅速に処理すると同時に、サイト（データベース内の同じレコードにアクセスするサイト）の作業を遅くしない方法


最初の質問への答えは私たちにとって明らかでした。消費者（サイトへの入力、個人データの変更）またはその上（賞品の引き分け、通知の送信）によって実行されるすべての重要なアクションに関する情報はシステムに保存されます。 さらに、消費者によるさまざまなテクニカルノートのアクションを使用します。 そのため、自動メーリングを送信するときに、この自動メーリングがすでに彼に送信されたことを示すマークとして、消費者に特別なアクションマーカーを付けることも決定しました。 ニュースレターを再送信しないために、「消費者にはトークンアクションがありません」という条件が常にニュースレターの条件に追加されます。



2番目の質問では、データベースのロックに関連する多くのコーンを取得し、その結果、次のパターンになりました。
  1. ニュースレターの送信は、条件に一致する新しい消費者がいるかどうかを定期的に確認するwindows-serviceから行われます。
  2. サービスの最初のステップは、 非コミット読み取り分離レベルのデータベースへの1つのクエリです 。 このリクエストにより、レターの送信先であるすべての消費者のIDが取得されます。 分離レベルが低いため、そのような要求はデータベース内のレコードにロックを課さず、その結果、サイトの運用に非常に弱い影響を与えます。 ただし、データの純度を保証するものではないため、より高いレベルの分離で再確認する必要があります。
  3. コンシューマのIDを取得した後、各コンシューマに対して、 Serializableの分離レベルで個別のトランザクションを実行します。 このトランザクションでは、消費者が条件に適しているかどうかを再確認し、適切であれば、彼に手紙を送り、マーカーアクションを発行します。 各コンシューマーは個別のトランザクションで処理されるため、ロックは1つのコンシューマーのデータにのみ課され、他のコンシューマーの作業の影響を受けません。 このようなトランザクションは非常に短いため、この時点でサイトを訪問すれば、手紙の送信先の消費者もあまり問題になりません。 トランザクション分離レベルは正確にSerializableである必要があります。これにより、1通の手紙を誤って2回送信したり、突然条件に適合しなくなった消費者に手紙を送信したりしません。 ただし、同じメーリングを送信できるのは1つのストリームと1つのサーバーからのみであり、相互に排他的な条件を持つ2つのメーリングが同じコンシューマーに送信される可能性が小さいことを忘れる場合は、 Read Committedトランザクションを使用できます。


もちろん、このテンプレートにいくつかのメーリングを実装した後、テンプレートコードを作成することにしました。 このために、 BatchMailingクラスが作成され、新しいメーリングごとに、その相続人を作成して特別なレジストリに登録しました。 相続人は、次のプロパティとメソッドをオーバーロードする必要がありました。
  • マーカーアクションテンプレート （以前はテンプレートをアクションタイプと呼んでいました：これは開発者にとってより理解しやすい用語だと思います）、メール送信時に発行されます
  • 発送方法
  • 追加のアクションを実行するメソッド（たとえば、誕生日の挨拶を送信するとともに、消費者にポイントを発行できます）
  • Expression <Func <Customer、bool >>を生成し、コンシューマーが条件に一致することを検証するメソッド


プロパティと最初の2つのメソッドは問題を引き起こしませんでしたが、 Expressionのコンパイルは非常に困難でした。 この式は2回使用されました-最初にコンシューマのIDを引き出すための非コミット読み取り要求で、次にコンシューマが適格かどうかを再チェックするためのシリアル化可能なトランザクションで。 Linq to SQLで T-SQLに変換できるように作成する必要がありました。 条件は非常に複雑になる可能性があり、常に問題がありました。 多数のテストを書かずにニュースレターを作成することはできません。 さらに、SMSと電子メールを送信するために、BatchMailingから異なる中間相続人を取得しました。 電子メールとSMSの両方を送信する必要がある場合、コピーアンドペーストする必要がありました。 これを修正する方法についてはアイデアがありましたが、クライアントはそれほど頻繁に自動メール送信を要求しなかったため、優先度の低いタスクでした。



継承をコンポジションに置き換える

2年前、次の広告キャンペーンを開発する際、クライアントは一度に8種類の自動メールを送信するよう依頼しました。 さらに、ニュースレターの条件が一部繰り返されました。 もはやそのように生きることができなくなったことに疑いはありませんでした。私はアーキテクチャを書き直すことにしました。 上記のすべての問題に対処するために、私たちの好きなテクニックを適用するだけで十分です：継承をコンポジションに置き換える。 この手法は何度も役に立ちましたので、可能な限り継承の代わりに構成を使用することをお勧めします（まあ、または少なくともこのオプションを検討してください）。 「特定のタスクごとに相続人のオーバーロードメソッドとプロパティがある」という考えで基本的な抽象クラスを作成する場合は、すぐに「代わりに各タスクのクラスのインスタンスを登録して、異なる設定を渡すのはなぜか」と自問してください。 ここで構成が適切でないことが確実な場合にのみ、継承を使用します。 これが適切な場合、常に構成に傾く-これにより、はるかに柔軟で理解しやすいアーキテクチャが得られます。



私たちの状況では：
  • アクションマーカーテンプレートを返すプロパティをオーバーロードする代わりに、このプロパティはクラスインスタンスに付加されます
  • 文字/ SMSを送信して追加のロジックを実行するメソッドをオーバーロードする代わりに、任意の操作がクラスインスタンスに置かれます。これはコンシューマで実行する必要があります。 さらに、操作は他の操作の組み合わせであってもよい
  • Expressionを形成するメソッドをオーバーロードする代わりに、クラスインスタンスに条件が与えられます。 この場合、条件はAND / ORで組み合わせることができます


メーリングの送信に加えて、このエンティティはコンシューマで任意の操作を実行できるため、メーリングリストと呼ぶのは正しくありません。 実際、これは、特定の消費者サンプルでいくつかの抽象的な作業を行うクラスです。 より良いものを発明することなく、私たちはそれをトリガーと呼び始めました（マーケティングでは、そのようなものと呼ばれるので、名前は悪くありません）。 正直に言うと、システムに非常に抽象的なエンティティ（ DoSomeWorkOnSomeCustomersと呼ばれる）を導入したのは少し怖かったです。 しかし、トリガーの専門化には意味がありませんでしたので、私はこれを気にしないことにしました。そして、原則として、クライアントにとってトリガーが何であるかを理解することに大きな問題はありません。

疲れている、自分でトリガーを開始する

上記のアーキテクチャを使用して、既に記述された条件とTriggerActionsを組み合わせて、30分以内にトリガーを設定します。 しかし、これでは十分ではありませんでした。 次のステップでは、トリガーを作成するプロセスから開発者を完全に除外します。 そして、これを一般的にどのように行うかについては、以前のバージョンのアーキテクチャの実装から数か月後に実現しました。 トリガー条件は、管理パネルで使用するフィルターと同様に一対一でした。 当社のフィルターシステムでは、関連するエンティティへのクエリを含む複雑な条件を記述したり、AND / ORでそれらを結合したりすることができます。 フィルターは式を形成します。これにより、データベース内のエンティティを既にフィルター処理できます。 そして、これらすべてのために、UIとシリアライゼーションはすでに書かれています。 トリガーによく必要となるフィルターをいくつか追加するだけでしたが、コンシューマーのリストを使用した通常の作業では意味がありませんでした（たとえば、「アクションからN日が経過しました」）。 TriggerActionsの場合、データベースにそれらを保存するためのUIと構造を記述する必要がありましたが、ここでは、一般的にすべてが明確でした。 ただし、まだ解決しなければならない小さな質問がありました。
  • この時までに、私たちはあらゆる手紙の送信をアクションとして登録し始め、アクションマーカーは不要になりました-すでに誰に手紙を送信したかを判断できました。一般的には、可能な限り不必要なアクションを発行したくない
  • 各コンシューマで特定の操作セットを1回実行する単純なトリガーに加えて、 定期的なトリガーがあります 。 これをすべてデータベースに転送すると同時に、 任意のマーカーの使用を許可する方法を理解する必要がありました。
  • マーケティング担当者は、トリガーを個別に作成するのではなく、サイトで消費者がトリガーと操作の両方を実行するチェーンとしてサイトにアクセスします（サイトにアクセスして何かを実行するように求める手紙→消費者がサイトでいくつかの操作を実行する→ボーナスポイントが付与されます）これについての手紙が送られます）。 すぐに気づかなかった場合、トリガーと操作の間の依存関係を説明するのが難しくならないように、将来のために基礎を残したいと思います
これら3つの問題はすべて、トリガーがコンシューマーで実行されるかどうかを判断する方法に関連しています。 サイトでトリガーと操作ごとに独自のマーカーを設定すると、タスクは大幅に簡素化されますが、システムで不要なアクションを生成したくはありませんでした。 消費者に対するアクションを実行できるかどうかに完全に責任を持つようにマネージャーにフィルターを作成するように強制するという考えさえありました（したがって、トリガーの繰り返し率はフィルターの条件によって記述されます）が、このアプローチはエラーが発生しやすいです。 それでも多くの苦痛な反省の末、私は、追加のエンティティやマネージャーの作業を複雑にすることなく、トリガーの実行を追跡する方法のアイデアを思いつきました。



もっと表現が必要

トリガーは消費者に対して抽象操作ステップ （以前のTriggerAction）を実行し、ほぼ常にこの操作ステップは一意であるため（たとえば、特定の手紙が送信されるか、特定の賞品がこのトリガーからのみ発行される）、ロジックがチェックアウトされて実行されたかどうかを確認できます。 トリガーには操作のいくつかのステップがある場合があるため、マネージャーはどちらをマーカーにするかを選択する必要があります （各ステップの実行をチェックすることは意味がありません）。 ただし、操作ステップで式<Func <Customer、bool >>を返すメソッドを実装するのは簡単です。各操作ステップで1回限りのトリガー用に1つの式を作成し、定期的なトリガー用に別の式を作成する必要があるためです。 ここでは、システム内のユーザーに対するほとんどすべての操作がアクションを提供するという事実によって保存されています。 したがって、操作ステップは、それらに発行されたアクションを除外できます。 操作のほとんどのステップは特定のアクションを生成し、アクションをフィルタリングするために式を形成するメソッドは次のようになります。 
 public sealed override Expression<Func<CustomerAction, bool>> GetIsMarkerExpression(ModelContext modelContext) { return action => action.ActionTemplateId == ActionTemplateId; }

1つのフィールドを持つOperationStepGroupの本質はかなり奇妙に見えますが、異なるエンティティ（トリガー、サイトでの操作など）がリレーショナルデータベース内のレコードのグループを参照できるようにします。 さらに、このエンティティには後で追加のフィールドが出現したため、すべてがそれほど怖いわけではありません。
