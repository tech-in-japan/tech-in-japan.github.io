What EXPLAIN is silent about and how to talk about it

The classic question that a developer comes to his DBA or a business owner, a PostgreSQL consultant, almost always sounds the same: ‚ÄúWhy are queries performed on the database for so long?‚Äù



The traditional set of reasons:



  • inefficient algorithm

    when you decided to make a JOIN of several CTEs for a couple of tens of thousands of records
  • irrelevant statistics

    if the actual distribution of data in the table is already very different from the last time ANALYZE collected
  • "Gag" by resources

    and there is already not enough dedicated computing power of the CPU, gigabytes of memory are constantly pumped or the disk does not keep up with all the database ‚ÄúWishlist‚Äù
  • blocking from competing processes


And if the locks are quite complicated in capturing and analyzing, then for everything else, we need a query plan that can be obtained using the EXPLAIN operator ( better, of course, immediately EXPLAIN (ANALYZE, BUFFERS) ... ) or the auto_explain module .



But, as stated in the same documentation,
‚ÄúUnderstanding a plan is an art, and to master it, you need some experience, ...‚Äù

But reading the plan with the text ‚Äúfrom the sheet‚Äù is very difficult and beloved:



  • the node displays the sum of the subtree resources

    that is, to understand how much time it took to execute a particular node, or exactly how much this reading from the table raised data from the disk - you need to somehow subtract one from the other
  • node time must be multiplied by loops

    yes, subtraction is not the most difficult operation that needs to be done ‚Äúin the mind‚Äù - after all, the runtime is indicated averaged for one node execution, and there may be hundreds
  • well, and all this together makes it difficult to answer the main question - so who is the ‚Äúweakest link‚Äù ?

Substitution of parameters in the request



If you ‚Äúattached‚Äù to the plan not only the request, but also its parameters from the DETAIL line of the log, you can copy it additionally in one of the options:



  • with substitution of values ‚Äã‚Äãin the request

    for direct execution on its base and further profiling 



     SELECT 'const', 'param'::text;
  • with value substitution via PREPARE / EXECUTE

    to emulate the work of the scheduler when the parametric part can be ignored - for example, when working on partitioned tables 



     DEALLOCATE ALL; PREPARE q(text) AS SELECT 'const', $1::text; EXECUTE q('param'::text);
