WebブラウザーとWebRTCトランスポートプロトコル



あるデバイスから別のデバイスにメッセージを送信するには、トランスポートプロトコルが必要です。 この場合、メッセージ内の内容やメッセージの構造は関係ありません。送信できることのみです。 そして受け取った。

5年前、プロトコルについて話すとき、ブラウザは簡単でした。 実際、HTTP / 1.1とその上にあるXHR、SSE、BOSH、Cometとして知られるすべてのハッキングがありました。 メカニックの詳細に興味がある場合は、コメントを残してください。以下の記事で説明を試みます。ただし、少しグーグルで検索すれば簡単に説明を見つけることができます。



HTTP / 1.1松葉杖とともにソリューショングループを呼び出します。 これらのソリューションはHTTP / 1.1を使用します。なぜなら、当時は単に代替手段がなかったからです。しかし、技術的な意味をなさない方法でそれを行います。

Webソケット



HTTP / 1.1ではできないことを行うために、WebSocketが追加されました。 クライアントとWebサーバーの両方が互いにメッセージを送信できる双方向メカニズムを提供します。 これらのメッセージの種類、意味、サポートする形式の種類-Webページ開発者が決定します。



socket.ioまたはあまり人気のないSockJSもあります。 どちらも、使用できない場合にWebSocketをエミュレートするクライアント側のメカニズムを提供します。



WebSocketがうまく機能するとき、socket.ioとSockJSも素晴らしいです。 ただし、うまく機能しない場合があります（これについては、HTTP / 2パートで詳しく説明します）。



WebRTCデータチャネル



ある程度まで、データチャネルはWebRTCでシグナリングに使用されます。



はい 使用するIPアドレス、およびその前にICEを使用することに同意する必要があります。 このためには、追加の信号とトランスポートレベルが必要になります（リストはこの投稿にあります）。 接続を確立した後、データチャネルを信号レベルとして使用できます。



データチャネルを使用して、2つのデバイス間で直接信号を送ることも、（タスクに応じて）仲介者を介して信号を送ることもできます。



データチャネルをトランスポートプロトコルとして使用する理由



  1. シグナリングの遅延を減らします。 理論的には、データチャネルはあなたができる最速のものです。
  2. サーバーの負荷を減らします。 これで、彼はどこかにリダイレクトするためだけにすべてのメッセージを受信することはありません-あなたは彼のために特に意図されているものを彼に送信します。
  3. 個人データの機密性/セキュリティのレベルを上げます-サーバーを介してメッセージを送信しない場合、これは彼が送信されているものを覗かないことを意味します-または何らかのメッセージが起こっていることに気付かないことを意味します。


しかし、実際には、このオプションはほとんど使用されません。 WebRTCの世界では、DataChannelがまだ利用できないときに接続が確立される前に、トランスポートレイヤーが重要です。 また、1つの接続のDataChannelをトランスポートレイヤーとして使用して別の接続に信号を送ることは奇妙です。

WebRTCシグナリングプロトコル



「シグナリング」とは、自分自身を表現する場所です。 またはあなたのサービス。 あるユーザーが別のユーザーと通信できるようにする必要があります。 または、仮想ルームに参加する人々のグループと。 必要なメッセージの種類、意味、外観などを決定します。



これは信号プロトコルです。



転送プロトコルとは異なり、ブラウザで許可されているものではなく、達成しようとしているものによって制限されます。



WebRTCでよく使用される3つの主要なシグナリングプロトコルを検討してください。



一口



SIPは電話の世界から来ました。 彼の主なトランスポートはUDPでした。 次に、トランスポートプロトコルとしてTCPとTLSが追加されました。 その後、SCTPは自身を引き上げました。 ブラウザを介して使用することはできないため、それらを理解することは意味がありません。 そのため、WebSocketはSIPトランスポートとして追加され、単に「SIP via WebSocket」と呼ばれました。 WebSocket経由のSIPはWebRTC（まだ標準化されていない）の前に標準化されており、とりわけ、独自のRFCをすでに持っています 。 上記のすべてが重要なのはなぜですか？ WebSocketを介したSIPの使用は、WebRTCでのみ可能であるためです。



これはSIPについてです。 また、SIPを知っている場合、SIPを愛している場合、または必要な場合は、WebRTCのシグナリングプロトコルとして使用できます。



Xmpp



私はXMPPが嫌いです。



しかし、その理由はよくわかりません。 おそらく私が彼について何か悪いことを言うと、すべての経験豊かなXMPPファン/フォロワー/狂信者がコメントで彼を守るために急ぐからです。 そして、それは私を笑わせます。



XMPPはすべて、ユーザーステータス情報とクイックメッセージに焦点を当てています。 これらが唯一の要件である場合、特に開発者がすでにXMPPを使用して何ができるかを知っている場合、XMPPが本当に勝ちます。



XMPPが気に入ったら、コメントに返信することを忘れないでください-これは下にあります。



所有権



私はニーが嫌いです。 それにもかかわらず、独自の信号プロトコルには多くの利点があります。



たいていの場合、あなたがしたいのは、1人のページに2人のユーザーを置くだけです。 もうありません。 私は大幅に簡素化することを知っていますが、単純化しないと、一般的なプロトコルの冗長性がすべて必要になります。



他の多くの場合、シグナリングを操作するためだけに別のWebサーバーを追加するのは本当に望ましくありません。 1つのサーバーでWebアプリケーション全体を処理する必要があります。 したがって、独自の信号レベルのプロトコルになります。 あなたはそれを呼び出すことはできませんが。 または、信号レベルのプロトコルとは考えないでください。



選択方法は？



常にシグナリングプロトコルで開始します。



インフラストラクチャがある場合、または接続する外部サービスがある場合は、SIPを使用する必要があります。 必要がない場合はスキップします。



XMPPが好きな場合、またはユーザーステータス情報とクイックメッセージの機能が必要な場合は、それを使用します。



WebRTCを追加するサービスが独自のロジックを持っている場合は、既にシグナリングを持っている可能性があります。 したがって、必要なメッセージを独自のシグナリングに追加するだけです。



他のすべての場合、私のアドバイスは、要件を完全に満たす独自の信号ソリューションを使用することです。 このためにSaaSソリューションを使用することもできます。
