StarTeamからSVNへのプロジェクトの移行

良い一日！



StarTeamとは何人の人が聞いていますか？ 数ヶ月前の私のように、他の事柄ではほとんど人はいないと思います。



私の現在の仕事場まで、そのようなボーランド製品のことは聞いたことがありませんでした。 Googleに聞いてみると、この製品はまだ存在し、開発中であることもわかりますが、ご想像のとおり、最新バージョンからは遠く、最後から2番目のバージョンでもありません。 2003年に開発されたバージョン5.3があり、2004年にここにインストールされて流通しました。 そして、ほぼ11年間、彼女は仕事をして解決しました。



イニシアチブの専門家として、この古いモンスターの髪の毛はすぐに動き始め、より現代的なものに移行することが決定されました。これはSVNまたはGitです。 私は彼と同僚との経験があり、経営陣はすべてに同意したため、選択はSVNにかかっていました。 既製のソリューションを探した結果、 PalarionからSVNのImporterが 生まれました。このソリューションでは、ハブに関する記事もあります。 しかし、結局のところ、すべてが単純すぎるわけではなく、バージョン5.3があり、この製品には2005バージョンのSDKが必要であり、非常に問題が多いことが判明しました。 必要なライブラリの名前を変更するなど、あらゆる種類のトリックを使用しても、このインポーターを起動できませんでした。



他に何ができますか？ 次のように、新しいStarTeamサーバーを作成しようとしました。 SVNと統合され、既存のサーバーに接続しますが、どうやらお互いに互換性がなく、サーバーがお互いを認識しなかったため、何もできませんでした。



それで私は何をしましたか？ もちろん、私は本当のプログラマーとして行動しました。自分のツールを書きました！



設計



それでは、ソースデータには何が含まれていますか？ 私が持っている：

  1. StarTeam Server 5.3
  2. StarTeam Client 5.3
  3. StarTeam SDK 5.3
  4. 特定のリポジトリを持つSVNサーバー
  5. コンソールクライアントがインストールされたTortoiseSVN
  6. 譲渡するプロジェクト




StarTeamの紹介



最新のStarTeamがどのように機能するかはわかりませんが、バージョン管理などの機能がないため、現在のバージョンはひどいものです。 各ファイルは個別にバージョン管理されます。このファイルと同時に配置されたものを理解し、しばらくスナップショットを引き出すことは非現実的です。 まあ、他のすべてに-このバージョンは、どのファイルが変更されているかを常に適切に理解することはできません（つまり、変更されたファイルは100％表示されますが、50/50に触れなかったファイルは変更されたと理解されます）。



また最近、ロシア語のすべてのコメントとユーザー名がセット「?????」の形式で表示されるという問題もありました。これは、エンコードがどこかに流れていることを意味します（さらに興味深いことに、エンコードはクライアントのどこにも指定されていません） そして、なぜ誰もその理由を確実に知りません。そして、そのような問題が始まったとき、彼らは確実に言うこともできません。



StarTeamのその他の構成要素は何ですか？ MS SQL 2008に基づくデータベースから。クエリを正しく記述し、必要なものをすべて引き出すことができるように思えますよね。 しかし、S01、S02、Viewセットなどの名前のテーブルを約100個見たため、データベースでの直接クエリ中のエンコードが正しく表示されないため、これらすべてを処理しないことにしました。



行動計画



そのため、興味深いリポジトリにソースプロジェクトがあり、空のターゲットリポジトリがあります。 このプロジェクトでは、 PalarionにSDKに含まれているstarteam80.jarが必要ですが、SDKにはstarteam53.jarしかありません。 すぐに言っておきますが、このプロジェクトの開始前にJavaでプログラムを作成したことはなく、1行のコードでした。 ただし、必要な場合は、IDEをインストールして試してください。 IDEとしてNetBeansを選択し、starteam53.jarの中身を解析し始めました。

このパッケージ内には、StarTeamサーバーを操作できるクラスのセットがすべて含まれています。 BorlandサイトにはSDKのドキュメントがあり、膨大な数のクラスで無意味なリールからあなたを救います。 次に、サーバーに接続してプロジェクトのリストを返すことができる単純なプロジェクトを作成します。 StarTeamのライブラリを2時間苦しめた後、私は望ましい結果を達成しました。 SDKを使用してStarTeamを操作できることが明らかになったため、ストーリー全体を転送する方法を決定するだけで済みます。



しかし！ それでは、StarTeamのすべての問題についてストーリーをどのように再配置しますか？



StarTeamには、各ファイルのタイムスタンプがあります。 コメント付きの個々のファイルの変更履歴と変更の正確な時間を取得できます。これは、これに基づいて構築することを意味します。



StarTeamでコミットが行われると、コミット中の各ファイルにコメントが書き込まれるため、最初のファイルからコメントと作成者を取得するだけで十分で、残りはすでに一致しています。



StarTeamにはファイルを保存する機能があり、プロジェクトに保存します。プロジェクトにはファイルとフォルダーがあり、フォルダーにはさらに多くのファイルとフォルダーがあり、各ファイルにはバージョンがあります。



また、SDKにファイルを単純に抽出するように要求すると、StarTeamはクライアントでデフォルトで構成されている場所にファイルを抽出します。これは私にとってはうまくいかず、最初にファイルストリームに抽出する必要がありますディスクに。



次の移行パスを思い付きました。

  1. StarTeamからローカルドライブ上のフォルダーにすべてをアップロードします。
  2. フォルダを通過してすべてをSVNにアップロードし、コメントフィールドにStarTeamでコミットが行われた時刻を入力します。


次のスキームに従ってアンロードします。

  1. フォルダツリーとプロジェクトファイルの再帰的な走査を開始します。
  2. 最初のファイルを取得します。
  3. 彼の話全体を抽出します。
  4. 履歴を確認し、ファイルをフォルダーに入れてプロジェクトをエクスポートします。
  5. マスク「yyyy.MM.dd.HH.mm」にタイムスタンプを持つフォルダーを作成します（最初は1ミリ秒の精度で実行しましたが、判明したように、1つのコミットからのファイルは異なるリビジョンに入ることがありますが、これは間違っています衝突と問題のアプローチはそうではなかった）;
  6. フォルダーに履歴ファイルを作成します。ここには、作成者の名前、コメント、タイムスタンプが書き込まれています。
  7. 最後のファイルが抽出されるまで行います。




StarTeamからの抽出



したがって、NetBeans、Java、およびこのプログラミング言語での0の経験。 Javaの開発経験がないからといって怖くはありません。Googleがあるからです。プロジェクトは一度限りであり、誰も私から最高の知識を必要としないので、どこでもコードのパフォーマンス/メモリ/美しさを犠牲にすることができます。

SVNに記入



SVNの塗りつぶしパターンは簡単です。

1）フォルダーの完全なリストを取得して並べ替えます（フォルダーに名前を付けると、これに役立ちます）。

2）欠落しているファイルとフォルダーのリストを現在のリビジョンのSVNフォルダーに作成します。

3）Filesサブフォルダーの内容をSVNフォルダーにコピーします。

4）SVNで欠落しているファイルとフォルダーの場合、追加します。

5）現在のSVNの作業コピーをコミットします。



実装中に、いくつかの問題が発生しました。 つまり、コミットへのコメントでは、二重引用符の前にバックスラッシュを配置して、引用符が正しく処理されるようにする必要があります。 また、@記号を含むファイルまたはパスを追加する場合、svnがファイル名を正しく理解できるように、最後に別の@を追加する必要があります。 ここで、コードが悪化したことがわかりました。すぐにやりたかったからです。
