Unity 3Dモバイルゲーム用の動的ダウンロードコンテンツシステム（DLC）の実装

最近、開発中のUnity 3Dの1つのゲームでは、 DLCシステムを追加する必要がありました。 これは当初のように単純ではありませんでしたが、発生した問題にうまく対処し、ゲームはゴールドになりました。 この記事では、DLC実装のバージョンを紹介し、発生した問題とその解決方法について説明します。



問題の声明



ゲームには、プレイヤーがゲームや実際の通貨のために物を買う店があります。 ストアには200を超えるアイテムがあります。 プレイヤーがゲームに参加すると、ストアで20個のアイテムを入手できます。 インターネットがある場合、ユーザーの知識のないゲームはサーバーにDLCをポーリングし、もしあれば、バックグラウンドでダウンロードします。 プレイヤーがストアに再入店すると、DLCのすべての新しいアイテムが表示されます。

まだ場所のセットがあります。 各場所には、テクスチャと.assetファイルのセットがあります。 DLCを通じて新しい場所も追加する必要があります。

DLCからのリソースのロードは同期する必要があります。

プラットフォーム：iOS（iPhone 3GS以降）およびAndroid（Samsung Galaxy S以降）。



DLCコンテンツとゲーム内での作業



ゲームでは、物事はアイテムデータとそのテクスチャに関する情報を含むitemdata.txtファイルによって完全に決定されます。 これは、各DLCに、これらの事柄のDLC +テストに含まれている一連の事柄を含むitemdata.txtファイルがあることを意味します。 ストアが物事のデータベースを要求すると、すべてのDLCからのすべてのテキストファイルを接着し、このファイルを提供します。

同様に、場所については、場所のリストと特性を含むlocationdata.txtファイルと、それらのテクスチャとアセットファイルがあります。

ゲームロジックでリソースをロードするための対応するC＃コードは次のようになります。 



public String GetItemDataBase() { if(DLCManager.isDLCLoaded() == true) { //   itemdata.txt    DLC     string String itemListStr = DLCManager.GetTextFileFromAllDLCs(“itemdata”); return itemListStr; } else { //    TextAsset itemTextFile = Resources.Load(“itemdata”) as TextAsset; return itemTextFile.text; } return String.Empty; }




同様に、テクスチャをリクエストするとき、DLCでその存在をチェックします。 存在する場合は、ゲームリソースからロードします。 存在しない場合は、デフォルトのものをロードします。 



 public Texture GetTexture(string txname) { Texture tx = null; if(DLCManager.isDLCLoaded() == true) { tx = DLCManager.GetTextureFromDLC(txname); } if(tx == null) { tx = Resources.Load(txname) as Texture; } if(tx == null) { Assert(tx, “Texture not find: ” + txname); tx = Resources.Load(kDefaultItemTexturePath) as Texture; } return tx; }




同様に、.assetファイルの場合、GetAsset（string assetName）関数があります。 その実装は似ているので、スキップしてください。



DLCファイル



DLCの内容を決定しました。 格納するものが何であるかを決定することは残っています。



最初のオプションは、DLCをzipアーカイブとして保存することです。 各アーカイブ-テキストファイル+ Nテクスチャ。 ビデオメモリを節約するには、テクスチャをPVRTC形式にする必要があります。 しかし、ここで最初の問題があります-Unityは、PNGまたはJPG形式のファイルシステムからのテクスチャのダウンロードのみをサポートしています[ link ]。 その後、テクスチャをPVRTCテクスチャ[ link ]に書き込むことができます。 これは遅いプロセスです リアルタイムでPVRに変換する必要があります。 また、 .assetタイプのファイルをDLCに、おそらくゲームレベル（.scene）に保存することが計画されているため、このような方法は完全に不適切です。



2番目のオプションは、 AssetBundleを使用することです 。 このソリューションは、ゲームのDLCに最適です。

ドキュメントから判断すると、多くの利点があります。

  • 必要なテクスチャ形式に圧縮されたものを含む、Unityリソースを保存できます。
  • これは圧縮率の高いアーカイブです。
  • シンプルで使いやすい。
  • バージョンパラメーターとハッシュ量（LoadFromCacheOrDownload関数によって読み込まれた場合）をサポートします。これは、DLCのバージョン管理に便利です。




マイナスのうち、そのAssetBundleのみがUnityのProバージョンを必要とし、 暗号化をサポートしていません 。 私たちはこの決定にこだわることにしました。 明らかに魅力的であり、すべての問題を解決できます。



実装（オプション1）



まず、最も基本的な機能を備えたDLCシステムのテストバージョンが作成されました。

最初に、ストアアイテムとロケーションファイルの200奇数のテクスチャがすべて1つのAssetBundleにパックされ、サーバーにアップロードされました。 ファイルは約200 mbであることが判明しました。 AssetBundleでのパッケージ化は、エディターのスクリプトによって実行されました。 AssetBundleでリソースパッケージを作成する方法は、ドキュメントで詳しく説明されています。 また、スクリプトを使用してAssetBundleを作成することもできます 。



次に、ゲームを開始した後、次の手順を実行します。



  1. まず、サーバーからDLCをダウンロードする必要があります。 Unityマニュアルのコードに従ってこれを行います。 次に、ダウンロードしたデータを将来使用するためにディスク上のファイルに書き込みます。 



     // Start a download of the given URL using assetBundle version and CRC-32 Checksum WWW www = WWW.LoadFromCacheOrDownload (urlToAssetBundle, version, crc32Checksum); // Wait for download to complete yield return www; // Get the byte data byte[] byteData = www.bytes; //       ,   byteData = MyDescriptionMethod(byteData); // byteData     .unity3d ... // Frees the memory from the web stream www.Dispose(); //DLC         DLCManager.SetDLCLoaded(true);




    このコードを使用すると、iPhone 3GSのような低いデバイスでメモリクラッシュが発生する可能性が非常に高くなります。 WWWクラスはバッファリングされた読み込みをサポートせず、読み込まれたすべての情報をメモリに保存します。 この問題については後ほど説明します。 この瞬間を思い出して先に進みます。



  2. DLCからリソースをロードしています。

    次に、GetTextureFromDLC（）、GetAssetFromDLC（）、およびGetTextFileFromAllDLCs（）関数を定義する必要があります。 後者の定義は省略します。なぜなら ロードされたリソースのタイプを除いて、最初のものとほとんど変わりません。



    GetTextureFromDLC関数の主なタスクは、DLCから名前でテクスチャを同期的にロードすることです。

    次のように定義してみましょう。 



     public Texture GetTextureFromDLC(String textureName) { // DLC  .     . AssetBundle asset = AssetBundle.CreateFromFile(pathToAssetBundle); //    DLC Texture texture = asset.Load(textureName) as Texture; //       texture asset.Unload(false); return texture; }






上記のコードは、これまでのところ、AssetBundleからリソースを同期的にロードする唯一の方法です。 結局のところ、多くのニュアンスがあります。 順番に並べ替えましょう。



ドキュメントに従ってAssetBundle.CreateFromFile



関数は、ディスクからアセットを同期的にダウンロードします。 ただし、「この関数では非圧縮のアセットバンドルのみがサポートされます。」という警告が1つあります。したがって、非圧縮のAssetBundleのみを同期的にロードできます。 これにより、サーバーからのトラフィックとDLCの起動時間が大幅に増加します。 さらに、UnityはAssetBundleを圧縮から非圧縮に変換することをサポートしていないため、圧縮されたバンドルをダウンロードしてクライアントで解凍することはできません。



読者は、たとえばLoadFromCacheOrDownload関数を使用してAssetBundleを非同期にロードし、必要なリソースを同期的に取得しないのはなぜかと思うかもしれません。 結局のところ、AssetBundleは、ファイルシステムからロードするときに、ファイルヘッダーのみをロードする必要があるため、メモリで少し占有する必要があるのは論理的です。



しかし、そうではありませんでした。 ロードされたAssetBundleは、すべてのコンテンツが展開された状態でメモリに保存されます。 したがって、200から1つのテクスチャをロードするには、Unityは200のテクスチャすべてをメモリにロードし、1つを取り、残りの199のテクスチャのためにメモリを解放します。 これは、デバイスのメモリを測定することで実験的にわかりました。

明らかに、これはモバイルデバイスでは受け入れられません。



まとめ



与えられたオプションは、DLCとそこからのリソースの同期ロードを実装するための唯一の方法です。

非圧縮のAsssetBundleが必要であり、DLCのロード時に時間とトラフィックが大幅に失われます。

このオプションは、比較的小さなAssetBundlesに適しています。 大量のRAMを消費します。



バグの処理（オプション2）



以前の問題をすべて考慮に入れて、それらの解決策を見つけましょう。



大規模なAssetBundleのロードに関する問題は、2つの方法で解決できます。

1つ目は、WebClientクラスを使用することです。 ただし、iOSでは問題がありました。 WebClientは何もダウンロードできませんでしたが、デスクトップ上では完全に機能しました。

2番目のオプションは、ネイティブOS機能を使用することです。 たとえば、iOSのNSURLConnectionとAndroidのURLConnectionは、それぞれ、ディスク上のファイルへの直接的なバッファーロードをサポートしています。

しかし、これはそれほど大きな問題ではありません。 いずれの場合でも、同期ロードのためにAssetBundleのサイズを小さくする必要があります。 したがって、今のところ、サーバーからバンドルをダウンロードする現在の方法は残しました。



さらに深刻な問題は、AssetBundleの同期ロードです。 なぜなら 圧縮されていないだけでなく、メモリ領域をほとんど占有しないため、1つの大きなDLCファイルをいくつかの小さなファイルに分割する必要があります。 ただし、あまりにも小さなファイルに分割すると、それらのファイルが多くなり、ロード時間が大幅に増加します。 ファイルごとに再接続する必要があります。 そのため、ロード時間とトラフィックを節約するために、圧縮したままにする必要があります。



この問題を解決するために、独自のアーカイバを使用することが決定されました。 C＃用のオープンアーカイバライブラリが選択されました。これは、労力をかけずに、UnityのMonoの下でビルドされることが判明しました。



さらに、アクションのアルゴリズムは次のとおりです。



  1. バンドルを作成するときに、非圧縮バンドルを取得するためにBuildOptions.UncompressedAssetBundleオプションが指定されました。
  2. 次に、アーカイバによってバンドルがアーカイブおよび暗号化され、サーバーにアップロードされました。
  3. アプリケーションの実行中に、別のストリームが作成され、バックグラウンドでバンドルがポンプで取り出され、アンパックされて特別なフォルダーに配置されました。




ここで別の問題がありました。 なぜなら アーカイバによって圧縮されたバンドルを使用するようになったため、LoadFromCacheOrDownload関数で圧縮することはできなくなりました。 したがって、DLC用に独自のバージョン管理システムを定義する必要があります。



DLCバージョン管理システムでは、次のソリューションが選択されました。 サーバーで、DLCファイルが配置されているフォルダーで、dlcversionテキストファイルを開始しました。 フォルダー内のDLCのリストとそれらのmd5ハッシュが含まれていました。 これらのハッシュは、サーバーへのDLCアプレットの段階で考慮されました。 クライアントはまったく同じファイルを持っていて、アプリケーションの起動時に、クライアントはそのファイルをサーバー上のファイルと比較しました。 DLCファイルに優れたハッシュが含まれているか、ハッシュがまったくない場合、クライアント上のファイルは古く、クライアントはサーバーから新しいDLCファイルをプルしたと考えられていました。



新しいDLCファイルがダウンロードされて解凍された後、そのハッシュがサーバー1でもう一度チェックされ、その後、古いファイルが新しいファイルで置き換えられ、対応するエントリがクライアントdlcversionファイルに作成されました。



説明したシステムは正常に実装され、正常に機能しています。 唯一の欠点は、バックグラウンドでDLCをダウンロードおよびアンパックする際の小さなfpsドローダウン（遅延）でした。 アプリケーションのメモリ消費のピーク値もわずかに増加しました。
