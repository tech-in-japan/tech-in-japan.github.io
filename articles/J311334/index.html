安心してSwift iOSアプリの依存関係管理

すべての良い一日。 困難な時期には、常にストレスの多い状況に対処する必要があり、プログラムコードの作成も例外ではありません。 誰もがさまざまな方法でストレスに対処します。誰かがバーに行き、誰かが沈黙の中で反対に瞑想しますが、誰もがこのストレスを可能な限り低くし、意図的にストレスの多い状況を避けようとします。



Swiftで書き始めたとき、私は多くの問題に直面しなければなりませんでしたが、その1つは、この言語でのIoCコンテナー間の競合がないことです。 実際、台風とスウィンジェクトの2つしかない。 Swinjectには機能がほとんどなく、TyphoonはObj-C向けに書かれていますが、これは問題であり、それを扱うことは私にとって大きなストレスになることが判明しました。



そして、Ostapが苦しんだので、 Swift用のIoCコンテナーを書くことにしました。



だから、知り合いになる-DITranquillity 、Swift上のiOS用IoCコンテナー、ストーリーボードと統合。



名前についての興味深い話-何百もの異なるアイデアの後、私は「穏やか」に落ち着きました。 名前を思いついたとき、IoCコンテナーを作成する主な理由はTyphoonであるという事実から始めました。 最初は、図書館を台風よりも強い自然災害と呼ぶことについて考えていましたが、台風はストレスであり、私の図書館は反対、つまり平和を提供するべきだと私は違った考え方をしなければならないことに気付きました。



すべてを静的にチェックすることを計画しました（残念ながら、すべてが完全に失敗しました）。また、大きなアプリケーションで台風を使用する場合、頻繁に発生せず、台風のためにxcodeがプロジェクトを収集しないという何らかの理由でアプリケーションの途中に落ちないようにしましたが、アセンブリ中にクラッシュします。



台風愛好家は少し動揺するかもしれませんが、私の意見では、一部のエンティティの命名は台風の命名とは異なります。 Autofacと同じですが、言語の特性を考慮に入れています。



特徴



ライブラリの機能を説明することから始めます。



  • このライブラリは、純粋なSwiftクラスで動作します。 NSObjectから継承してプロトコルをObj-Cとして宣言する必要はありません。このライブラリを使用すると、純粋なSwiftで記述できます。
  • ネイティブ性-依存関係の記述はネイティブ言語で行われます。これにより、簡単なリファクタリングと...
  • コンパイル段階でのほとんどのチェック-Typhoonの後、これは楽園のように見えるかもしれません。多くのタイプミスが実行時ではなくコンパイル段階で検出されるためです。 残念ながら、ライブラリは実行中にエラーを自慢することはできませんが、問題の一部は確実に遮断されます。
  • すべての依存性注入パターンのサポート：イニシャライザー注入、プロパティ注入、メソッド注入。 なぜこれがクールなのかわかりませんが、誰もがそれについて書いています。
  • 循環依存関係のサポート-ライブラリは、プログラマーの介入なしに、循環依存関係のさまざまなオプションをサポートします。
  • ストーリーボードとの統合-ViewControllersに直接依存関係を埋め込むことができます。


そしてまた：



  • オブジェクトの存続期間のサポート。
  • 代替タイプの表示。
  • タイプと名前による依存関係の解決。
  • 複数登録;
  • 初期化パラメーターを使用した依存関係の解決。
  • 依存関係を解決するための短いエントリ。
  • 「モジュール性」の特別なメカニズム。
  • CocoaPodsのサポート。
  • ロシア語のドキュメンテーション（実際には、ドキュメンテーションのドラフトと言う方が正確です。多くのエラーがあります）。


そして、これらはすべて1500行のコードで、そのうち約400行で、異なる数の初期化パラメーターを持つ依存関係の型指定された解決のために自動的に生成されたコードです。

データを使用した作業を追加する



その後、プロジェクトは何かをする必要があります。多くの場合、モバイルデバイスに対して、アプリケーションはネットワークから情報を受け取り、処理して表示します。 簡単にするために、データ処理ステップを省略し、ネットワークからデータを受信する詳細には触れません。 `get`メソッドを備えたServerプロトコルがあり、それに応じてこのプロトコルの実装があると仮定します。 つまり、次のコードがプログラムに表示されます。 



 protocol Server { func get(method: String) -> Data? } class ServerImpl: Server { init(domain: String) { ... } func get(method: String) -> Data? { ... } }


これで、新しいクラスを登録する別のモジュールを作成できます。 もちろん、さらに先に進んで新しいアセンブリを作成し、サーバーでの作業を別のプロジェクトに転送することもできますが、この例は複雑になりますが、ライブラリのより多くの側面と機能が示されます。 または、逆に、既存のモジュールに埋め込みます。 



 import DITranquillity class ServerModule: DIModule { func load(builder: DIContainerBuilder) { builder.register(ServerImpl.self) .asSelf() .asType(Server.self) .instanceSingle() .initializer { ServerImpl(domain: "https://your_site.com/") } } }


ServerImpl型を登録しました。プログラムでは、ServerImplとServerという2つの型で認識されます。 これは登録動作のいくつかの機能です。代替タイプが指定されている場合、明示的に指定されない限り、メインタイプは使用されません。 また、プログラムにはサーバーが1つしかないことも示しました。

publicModulesとinternalModulesの違い
モジュールの可視性には、内部とパブリックの2つのレベルがあります。 Public-このモジュールが表示されることを意味し、このアセンブリを使用する他のアセンブリでは、Internal-モジュールはアセンブリ内でのみ表示されます。 確かに、アセンブリは単なるアナウンスであるため、モジュールの可視性に関するこの規則は、原則に従ってコンテナーに適用されることを明確にする必要があります：ビルダーに直接追加されたアセンブリのすべてのモジュールは、彼が構築したコンテナー、および依存アセンブリのモジュールに含まれますコンテナがパブリックと宣言されている場合にのみコンテナに含まれます。

ロガーを実装します



私たちのプログラムは動作しますが、一部のユーザーにとっては、突然正しく動作しなくなりました。 自宅で問題を再現して判断することはできません-それはいつでも、ロガーが必要です。 しかし、超常現象に対する私たちの信念は既に覚醒しているので、ロガーはファイル、コンソール、サーバー、さらには海の場所にデータを書き込む必要があり、これらすべてを簡単にオン/オフして使用する必要があります。



そのため、関数log（メッセージ：文字列）を使用して基本プロトコル `Logger`を作成し、いくつかの実装を実装します。ConsoleLogger、FileLogger、ServerLogger ...その他すべてをプルする基本ロガーを作成し、MainLoggerと呼びます。 さらに、ログを記録するクラスでは、類似性に関する行を追加します： `var log：Logger？ = nil`、および...そして、実行したすべてのアクションを登録する必要があります。



最初に、新しいLoggerModuleモジュールを作成します。 



 import DITranquillity class LoggerModule: DIModule { func load(builder: DIContainerBuilder) { builder.register(ConsoleLogger.self) .asType(Logger.self) .instanceSingle() .initializer { ConsoleLogger() } builder.register(FileLogger.self) .asType(Logger.self) .instanceSingle() .initializer { FileLogger(file: "file.log") } builder.register(ServerLogger.self) .asType(Logger.self) .instanceSingle() .initializer { ServerLogger(server: "http://server.com/") } builder.register(MainLogger.self) .asType(Logger.self) .asDefault() .instanceSingle() .initializer { scope in MainLogger(loggers: **!scope) } } }


そして、たとえば次のように、ロガーの導入を宣言したすべてのクラスに追加することを忘れないでください。 



 builder.register(YourPresenter.self) .instancePerScope() // ,    scope    Presenter .initializer { scope in try YourPresenter(server: *!scope) } .dependency { (scope, obj) in obj.log = *?scope }


そして、それをアセンブリに追加します。 今、私たちが書いたばかりのものを作る価値があります。



最初に、3つのロガーを登録しました。これは、ロガーという名前で利用できます。つまり、複数の登録を実行しました。 さらに、MainLoggerを削除すると、プログラムは1つのロガーを持たなくなります。1つのロガーを取得したい場合、ライブラリはプログラマがどのロガーを望んでいるかを理解できないためです。 次に、MainLoggerに対して2つのことを行います。



  1. これは標準のロガーだと言います。 つまり、単一のロガーが必要な場合、それはMainLoggerであり、他のロガーではありません。



  2. MainLoggerでは、自分自身を除くすべてのロガーのリストを転送します（これはライブラリの機能の1つであり、複数の依存関係を解決できるため、再帰呼び出しは除外されます。これは、「try！」に相当する高速構文「**！」を使用します。 scope.resolveMany（） `
