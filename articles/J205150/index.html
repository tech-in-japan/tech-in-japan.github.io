Microsoft SharePoint 2013用Veeam Explorer

最近、Microsoft SharePoint 2013をサポートするVeeam Explorer for Microsoft SharePointの新しいバージョンがリリースされましたが、新しいバージョンの機能、詳細なデータリカバリ、フルリカバリとの違い、これらの各方法を使用するのが合理的であることがわかります。













通常、会社が大きくなり、ビジネスプロセスが複雑になるほど、コラボレーション製品の必要性が高まります。 会社が競争力を維持するためには、従業員、特に複雑なビジネスインタラクションに関与する従業員の生産性に注意する必要があります。 Microsoft SharePointが解決に役立つのはこのタスクです。 これは、社内の作業プロセスを比較的簡単かつ迅速に合理化および自動化できる製品の1つであると言えます。 たとえば、その助けを借りて、プロジェクト専用の別の内部サイトを作成し、他のプロジェクト参加者に協力して作業する機会を提供できます。 このようなサイトは、ナレッジベース（たとえば、Wikiの形式）、プロジェクトドキュメントのリポジトリ、プロジェクトニュースを公開する場所などとして機能できます。 SharePointは、作業文書の承認/承認のプロセスも自動化します。









コラボレーション自動化プロセスが実装され、企業がかなりの時間にわたってそれらを実行すると、SharePointサーバーは組織にとって重要なリソースになり、その障害や損傷が大きな損害を引き起こす可能性があります。 さらに、従業員の1人が誤って手紙や文書を削除する可能性がある場合、人的エラーから安全ではありません。 そして、失われた情報と「問題の価格」を迅速に回復する可能性が最初に来ます。









もちろん、いつでもすべてをロールバックできます。 ここで、「すべて」とは、文字通りすべてを意味します。 必要なオブジェクトがまだ存在する場合、システム全体のすべてのデータを過去の特定の制御ポイントにロールバックする必要があります。 これはどういう意味ですか？ つまり、このような完全なロールバックが行われると、システムのすべてのユーザーを「過去に」強制的に「破棄」し、「一定期間のデータ損失」に関連する多くの問題が発生します。 もちろん、予備のハードウェアにサーバーのコピーを復元することもできますが、これには新しい問題が伴います（たとえば、生産的なネットワークのネットワーク上に重複したサーバークローンが出現し、IPアドレスを介した競合が発生するなど）。 これらの問題を解決するために、きめ細かい復元が考案されました。









このような場合にシステムを完全にロールバックしないようにするために、 きめ細かいリカバリ （ ここでこれについて以前に書いた）が必要です。 個別リカバリは2つのタイプに分けられます。









  • ユニバーサル 、つまり 特定のアプリケーションとは無関係に、システム全体をテストラボまたは「サンドボックス」に復元し、その後このサンドボックスで起動して、プログラムとユーザーインターフェイスを介してオブジェクトを復元することにより実行されます。 複雑なアプリケーションを起動するには、元のアプリケーションが依存するサンドボックス内のネットワークサービスで複数の仮想マシンを実行する必要がある場合があります（たとえば、Exchange Serverを復元するには、動作しているDNSサーバーとドメインコントローラーが必要です）。





  • 専門 、つまり 特定のアプリケーション用に設計されており、アプリケーション自体（すべての依存コンポーネントを含む）をサンドボックスで実行することなく、アプリケーションデータファイルから特定のアプリケーションのデータオブジェクトを直接抽出できます。







特殊なリカバリは、紛れもなく便利であり、管理者の労力をあまり必要とせず、ユニバーサルに比べて時間がかかりません。 次に、 Microsoft SharePointユーティリティ用のVeeam Explorerを例として使用して説明します。









Microsoft SharePoint用Veeam Explorerは 、 Veeam Backup＆Replicationの統合コンポーネントとして2013年3月に初めて発表されました。 これは何ですか 実際、これは、最初にVM全体を復元することなく、仮想マシン（VM）の既存のバックアップコピーから必要なMicrosoft SharePointオブジェクトを復元できるユーティリティです。









どのように機能しますか？ Veeam Explorer for Microsoft SharePointは、仮想化されたSharePointサーバーのサポートを提供します。 個別リカバリ方法では、SharePointサーバーが原則として起動されないため、サンドボックスで相互依存するVMのグループは開始されません。 代わりに、バックアップリポジトリから直接、SharePointサーバーデータストアファイルがあるVMディスクがマウントされ、その後、データを直接抽出するために低レベルで操作されます。









最初のバージョンでは、Microsoft SharePoint 2010のサポートのみが実装され、最新リリースでは、Microsoft SharePointバージョン2013のサポートも追加されました。









Veeam Explorer for Microsoft SharePointは、必要なMicrosoft SharePointオブジェクトをVMバックアップから復元でき、完全、増分、圧縮、および/または重複排除を含む、あらゆるタイプのバックアップをリカバリに使用できます。 このユーティリティは、VMディスク上のSharePointデータベースファイルを指定する必要があります。その後、管理者は一般的な一連の操作を実行できます。









  • バックアップリポジトリに保存された（つまり、圧縮および重複排除された形式で）SharePointサーバーを使用したVMバックアップ内のSharePointデータベースを、事前の復元を必要とせずに表示します。





  • SharePointサーバーデータベースのバックアップコピー内のドキュメントを検索します。





  • バックアップからソースSharePointサーバーに直接、元のストレージの場所にドキュメントを復元およびエクスポートするか、これらのユーザーの電子メールアドレスに添付ファイルとして送信します。











本当に、Veeam Explorerのすべての利点は、バックアップからデータを復元するときに正確に明らかになります。 簡単な例を見てみましょう-管理者がSharePointから削除されたファイルを復元する要求を受け取ります。 必要なことは、Veeam Explorer for Microsoft SharePointを開いて、現在のリカバリポイントを選択し、ウィザードでそれを開くことだけです。 その後、管理者はバックアップからデータベースを取得することなく、データベース全体にアクセスできます。 データは通常の「エクスプローラー」のように表示されます。 アプリケーションに正確なファイル名が含まれていない場合、管理者はユーティリティに組み込まれた検索を使用してキーワードで検索できます。 ファイルが見つかったら、その上でいくつかの操作を実行できます。







  • ローカルサーバーに保存。





  • メールで送信。





  • SharePointの元の場所に復元します。











最後に何が得られますか？ きめ細かな削減の場合、RPOは他の方法に比べて数倍小さくなることがあります。 これで、SharePointに障害が発生した場合、サーバーをできるだけ早く復元できるので、安心してスリープできます。
