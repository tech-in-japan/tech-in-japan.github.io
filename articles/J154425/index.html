ローカルマシン上の1バイトではなく、オフィス全体を「クラウド」に転送します

原則は非常に単純です。各ユーザーにブラウザを備えた最も単純なマシンを残し、すべてのアプリケーションとすべてのデータをデータセンターに保持する必要があります。 同じ場所のデータセンターでは、各ユーザー用の仮想デスクトップを作成できます。ここでは、RDPと同様に移動できます。



これらすべてが正常に機能するためには、クラウド内のそのようなデスクトップの可用性、アプリケーション配信、データ管理、I / Oデバイスからエンドユーザーへの情報転送を保証するソリューションが必要です。 これはCitrix XenAppです。



すべてを「箱から出して」ほぼ1.5倍安く動作させるために、クラウドで非常に興味深いSaaSソリューションを作成しました 。 しかし、最初に、IT部門の作業を促進し、費用を節約する主なポイントを見ていきましょう。



使いやすさ



すべての作業マシンが実際にデータセンターにある場合、非常に簡単に制御できます。 割り当てられた権限とポリシーを使用して、特定のユーザー用の個別の仮想サーバーとしてクラウドで実行される稼働中のマシンを作成することについて話します。 たとえば、会社のすべてのエンジニアに1つのアプリケーションを追加する必要がある場合、これはオブジェクトを変更するのではなく、クラスを1回変更することによって行われます。 実際、仮想デスクトップテンプレートの設定で数回クリックします。 更新には20分以上かかりません。必要なすべてのワークステーションにすぐに適用されます。



ユーザビリティサポート



ユーザーは、端末のように動作する物理的にインストールされたマシンを持っています。ブラウザが起動するものであれば何でも構いません。



オペレーティングシステムは重要ではありません。シンクライアントはどこでも実行できます。



ユーザー端末のパフォーマンスは重要ではありません。高負荷のアプリケーションで動作する場合でも、この負荷はデータセンターサーバーにかかります。



このようなコンピューター群は非常に簡単に保守できます。 さらに、全体の魅力は、ユーザーデータもアクセスセンターではなくデータセンターに保存されることです。つまり、故障した場合は、新しいシステムユニットを持ち込むか、ユーザーを次のテーブルに移動して問題を完全に解決するだけで十分です。 繰り返しますが、鉄の単一の故障がデータ損失につながることはなく、鉄の単一の故障が修正するのに15分以上かかることはありません。



アクセシビリティ



リモートデスクトップには、タブレット、電話、システム管理者が必要に応じて自宅からも含めてデータセンターへのログインを許可したコンピューターなど、何からでもリモートデスクトップに接続できます。 これは、いつでもどこでも作業できるという単純な事実を意味します。 自宅でノートパソコンを忘れましたか？ 大丈夫、いつでもどこからでも通常の作業環境にアクセスできます。



建築



XenAppは、Citrix Delivery Center（XenDesktop、XenApp、XenServer、NetScaler、およびCitrix Receiver）の一部であるソリューションであり、Windowsアプリケーションを配信するように設計されています。 このソリューションは、データセンターからのアプリケーションのアプリケーション仮想化、集中化、およびデータ管理を実装し、さらに、場所に関係なく、ユーザーデバイスへの要求に応じて、SaaS（Software as a Service）モデルに従ってアプリケーションへのアクセスを提供します。 独自のICAプロトコルは、以前はアプリケーションの配信に使用されていましたが、現在、XenAppの最新バージョンでは、マルチスレッドの高解像度ユーザーエクスペリエンスプロトコルを使用しています（おかげでトラフィックが最小限に抑えられます）。



アプリケーションをエンドクライアントに配信するには、Citrix ReceiverまたはCitrix SmartAccessのいずれかが必要です。 1つ目は、プロバイダーのサイト、またはこの場合はSaaSモデルを使用してクラウドにインストールされたリモートデスクトップと企業アプリケーションを提供するためのソフトウェアソリューションです。 ただし、その作業にはITサービスからの制御は必要ありません。 Windows、Windows Mobile、iPhone、iPad、Androidデバイス、Mac OS XおよびLinux用のCitrix Receiverの実装があります。 2つ目は、クライアントソフトウェアをダウンロードしないWebベースのインターフェイスです。







仕事の基本原則は次のとおりです。

  1. ユーザーは、ロードバランサーに接続するか、データセンター内のいずれかのCitrixノードに直接接続します。
  2. ユーザーが接続されているターミナルサーバーで、デスクトップが作成されます。
  3. ユーザーはこの仮想ワークステーションにログインします。
  4. ユーザーからCitrixノードへ、そして逆方向に、モニターデータ（下）とキーストロークとマウスの動き（上）のデータのみが移動します。 つまり、実際にはRDPの類似物です。
  5. Citrixノードから、リクエストはデータセンター内でアプリケーションが動作するデータベースに送信されます。


これはどういう意味ですか？



  • ユーザーとプラットフォーム間のトラフィックは最小限です。
  • アプリケーションとそのデータベースはローカルマシンにプルされませんが、データセンターサーバーで実行されます。 前世代のソリューションは、ユーザーがオムスクから「ノック」していて、データセンターがモスクワにある場合、操作が遅くトラフィックが多いという不快な驚きを待っています。
  • 実際、ユーザーは自分の画面、キーボード、およびマウスをデータセンターの中心にある仮想マシンに接続します。
  • 同時に、ユーザーへのデータ転送は十分に保護されています。したがって、セキュリティを侵害する数少ない方法の1つは、ユーザーの端末画面の写真を撮るだけです。
  • すべてのデータはデータセンターにあり、同じ場所のセッション間で保存されます。
  • データセンターにデータが格納されると、オフィスに不可抗力があっても、データはそのまま残り、安全で、第三者がアクセスできなくなります。


実際に、1人のシステム管理者が600台のクライアントワークステーションを管理し、さらにハードウェアの問題を解決するためのenikeyschikの作業をいくつか行うと、状況は非常に現実的であることがわかります。 ワークステーションとして、PCの代わりにシンクライアントのみが使用されます。キーボード、マウス、およびモニターが接続されるボックスのみです。 侵入するものすらありません。原則として、ハードドライブやクーラーはなく、機械的なものはありません。



なぜ「クラウド」が必要なのですか？



これまでのところ、「クラウド」アプローチなしで、すべてのサーバーを購入するだけで、上記のすべてを実装できることは明らかです。 ただし、半分以上の時間アイドル状態で動作することは明らかです。結局のところ、週末、夜、昼食などで使用が停止します。



「クラウド」を使用すると、必要に応じて水平方向に拡張し、使用の事実に応じて支払うことができます（瞬間：サーバー、ライセンスではありません）。 これは直接的なコスト削減です。



クラウドを使用すると、新しいジョブが（少なくとも部門全体ですぐに）立ち上げられたときにスケールアップすることを考えず、既存のジョブの負荷を増やすことができます。



「クラウド」を使用すると、サーバー間で負荷を分散できます。



実際には、3つのオプションがあります。サーバーを購入する従来のスキームでは、マシンが正常に機能しますが、これは非常に不経済です。 別のアプローチでは、マシンを手動でオフにすることができます。また、いくらか節約できますが、それでも不便です。 CROC「クラウド」へのアプローチでは、負荷またはスケジュールに従って自動サーバーのオン/オフを構成できます。 このソリューションは、Citrixファームを管理するための標準クラウドAPIと組み込みメカニズムを使用して機能します。



CROC「クラウド」と連携してどのように機能しますか？



クラウドにターミナルサーバーが必要です。 会社のクライアントがクライアントに接続するとします。 時間に対する負荷をプロットできます。 たとえば、夜間にはリモートデスクトップユーザーはいません。 ピーク時（午後2時）には600人のユーザーがいますが、ピーク負荷（600ユーザー）には20サーバー、300ユーザー-10が必要であり、ある時点で1サーバーで十分であることがわかります。 ターミナルサーバーは、ユーザー数に応じてオンとオフを切り替えます。



その結果、サーバーインフラストラクチャのダウンタイムの費用を支払う必要がないため、節約になります。 同時に、Citrixをクラウドにバインドするためのソフトウェア開発のコストは、この節約により、原則として6か月後に相殺されます。 ソフトウェアは顧客ごとに個別に構成されます。 たとえば、5人のユーザーだけが前のサーバーにスペースを残したときに起動するようにセカンダリサーバーを構成できます。 サーバーをスケジュールどおりに起動することも、ユーザーのデスクトップが2時間を超えて非アクティブである場合にユーザーのデスクトップの自動削除を構成することもできます。 さらに、顧客のその他の要望。



その結果、サーバーを早期に運用するために必要な資金の最大80％が節約されます。 さらに、IT部門は日常業務から解放され、ワー​​クステーションでの混乱が一気に終わりました。



5〜10年後には、ローカルマシンでアプリケーションを実行するオフィスはほとんどなくなると思います。
