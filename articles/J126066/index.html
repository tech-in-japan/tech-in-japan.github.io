迅速なJava Webアプリケーション開発

JavaでWebアプリケーションをどのように開発しますか？

各変更後、どのように実行してチェックしますか？ 再アプリケーションとコンテナの再起動にはどれくらい時間がかかりますか？



さまざまなオプションを見る機会がありました。WARファイルの完全な再構築から、MyEclipse、WTP、サーブレットコンバーターの「コネクター」などのIDEのプラグインの使用まで。 それらのいくつかには明らかな欠陥があり、他のものは非常にうまく機能します-しかし、もっと簡単な方法があります！



ランチャー



この開発方法により、最小限の再デプロイメント時間で 、できるだけ簡単かつ柔軟にアプリケーションを構成できます 。 mainメソッドを使用して1つの単純なJavaクラスを記述するだけで、必要なアプリケーション（いわゆる埋め込みモード ）でJettyサーバーがすぐに起動します。







ミニマリストランチャーは次のようになります。 

import org.mortbay.jetty.*; import org.mortbay.jetty.nio.SelectChannelConnector; import org.mortbay.jetty.webapp.WebAppContext; public class Launcher { public static void main(String[] args) throws Exception { Server server = new Server(); Connector connector = new SelectChannelConnector(); connector.setPort(8080); server.addConnector(connector); WebAppContext root = new WebAppContext("root/src/main/webapp", "/"); WebAppContext reports = new WebAppContext("reports/src/main/webapp", "/reports"); WebAppContext petclinic = new WebAppContext("petclinic/src/main/webapp", "/petclinic"); server.setHandlers(new Handler[]{root, reports, petclinic}); server.start(); } }




このコードは、ポート8080でリッスンするサーバー（サーブレットコンテナ）を起動し、3つのWebアプリケーションを使用します。コードはプロジェクトフォルダ （ root/src/main/webapp



、 reports/src/main/webapp



およびpetclinic/src/main/webapp



）、つまり、ファイルへの変更はすぐに反映され、 何かを再構築して再インストールする必要はありません。



新しいメソッドを追加する場合、再起動する必要があることは明らかですが、この場合、再起動は文字通り1秒以内にできるだけ早く行われます（もちろん、アプリケーションが起動時に複雑なことをしない場合）。 これにJRebelをねじ込んだ場合、通常はチョコレートになります。



それがトリックです。

実稼働環境では、JettyではなくTomcat、JBoss、またはWebLogicなどを使用しています。 問題ではありません。速度、安定性などの開発について話します。 重要ではありませんが、セットアップの容易さ、起動の速度、再展開が重要です。 そして、ここで埋め込みJettyは医師が注文したものです。



お気に入りのIDEからこのメインクラスを実行し、デバッグ、テスト、再起動するだけです。 プラグインは不要で、構成ファイルを探す必要も、XMLを掘り下げる必要もありません。 あなたの指先ですべての設定。 これが人生です！



最小構成で起動するには、 servlet-api.jar 、 jetty.jar 、 jetty-util.jarの 3つのjarファイルだけで十分です。

チューニング



プロジェクトによっては便利な追加機能をいくつか検討してください。



クラスローダー



上記のコードは、すべてのWebアプリケーションのクラスと依存関係を1つのローダー（ClassLoader）にロードします。 何らかの理由で、Webアプリケーションに異なるクラスと依存関係（jar）のセットがあることが重要な場合、これは小さなコードを追加することでも実行できます。 



  WebAppContext petclinic = new WebAppContext("petclinic/src/main/webapp", "/petclinic"); WebAppClassLoader classLoader = new WebAppClassLoader(petclinic); classLoader.addClassPath("petclinic-core/target/classes"); classLoader.addClassPath("petclinic-services/target/classes"); petclinic.setClassLoader(classLoader);




私自身は、それぞれの（競合する）依存関係を持つ多くのWebアプリケーションを一度に実行する必要があるときに、これを積極的に使用しました。 非常にシンプルで非常に便利です。



JSP



昔の人は、忘れられない時代に、JSPのようなものがHTMLの形成にしばしば使用されたことを覚えています。 プロジェクトでJSPも使用する場合は、さらにいくつかの依存関係を追加する必要があります： eclipse-jdtcore.jar 、 jsp-api-2.1- glassfish.jar 、 jsp-2.1-glassfish.jar



JDBCリソースの構成





おそらくより簡単な解決策があるかもしれませんが、このオプションは私のために働いた： 

 import import javax.naming.InitialContext; import com.atomikos.jdbc.nonxa.AtomikosNonXADataSourceBean; static void setupDataSources() { System.setProperty("java.naming.factory.initial", "org.mortbay.naming.InitialContextFactory"); AtomikosNonXADataSourceBean dataSource = new AtomikosNonXADataSourceBean(); dataSource.setUniqueResourceName("jdbc/portal"); dataSource.setDriverClassName("com.mysql.jdbc.Driver"); dataSource.setUrl("jdbc:mysql://localhost:3306/mydb"); dataSource.setUser("myusername"); dataSource.setPassword("mypassword"); dataSource.setMaxPoolSize(10); new InitialContext().createSubcontext("jdbc").bind("portal", dataSource); }




これを行うには、 transactions.jar 、 jetty-naming.jar 、 transactions-api.jar 、 transactions-jta.jar 、 transactions-jdbc.jar 、 atomikos-util.jarを追加する必要があります 。

Webアプリケーションのリストが絶えず変化している場合は、EclipseまたはIDEAのプラグインを作成して、現在EclipseにあるすべてのプロジェクトでJettyを実行することをお勧めします。 このようなプラグインの1つをここで説明します 。web.xmlファイルを持つすべてのプロジェクトを見つけ、残りのプロジェクトをクラスパスでそれらに追加します。

  • 場合によっては、アプリケーションが埋め込みJettyモードでコンピューター上で正常に実行されますが、戦争で組み立てられて別のサーバーにインストールされるとクラッシュします。

    これはまれですが、起こります。 特に、クラスローダーまたはjarバージョンが原因でエラーが発生する場合があります。
