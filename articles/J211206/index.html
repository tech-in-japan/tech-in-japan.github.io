ラピッド開発フレームワークのコーディング（パート2 CQRS）





プレストーリー



私の前の記事は、IML（私たちの主力機能）で始まったIncoding Frameworkの紹介でした。 IMLは、企業プロジェクトで使用される一連のユーティリティ（開発者のチームのようなもの）以上のプロジェクトを開発するように私たちを押しましたが、これは他のコンポーネントが機能しないことを意味するものではありませんが、それどころか「磨かれ」ており、それをあなたに証明しようとします。



特効薬？



以前、私は常に各ソリューションに長所と短所があるというサポーターでしたが、CQRSは私の意見ではNレイヤーよりも優れており、「禁忌」や「副作用」がないため、クリップの最初のカートリッジの候補になります。しかし、まず最初に。



誰かがCQRSについて聞いたことがありますか？



すでにCQRSを使用している人にとっては、最初のセクションは面白くないかもしれません。そのため、「bike」というラベルを付ける前に、殺人機能セクションに慣れておくことをお勧めします。 N層アーキテクチャを使用する人は、CQRSへの切り替えを検討する必要があります。私の提案をサポートするために、CQRSへのパスを説明します。





ああ、いくつの抽象化が便利か（過去の意見）



アプリケーションの開発を始めたばかりのときに、サーバーアーキテクチャとしてN-Layerを選択しました。これにより、アプリケーションが多数のレイヤーに分割され、プロジェクトの異なる部分を互いに独立して設計できるようになりましたが、最終的に次の問題が発生しました：

  • ソースコードの「膨張」、問題は、多くの場合、ファサード層、通信層などのタイ層の追加が原因で発生します。
  • 複数のレイヤーレベルの背後に詳細を非表示にします。


注：N-Layerのアイデアは、特定のレイヤーの置換DLLとも関連していますが、この非常にまれなタスクはIoCを使用してはるかに簡単に解決できます。



Nレイヤーの「悪」の主な原因はほとんどの場合、サービス層です。サービス層は、同様のタスクを1つのクラスに集約することでビジネスプロセスとアプリケーションロジックの作業の詳細を隠します。

  • 密接に関連するメソッドのサポートと拡張
  • 大きなオブジェクトでテストするのは難しい




微妙ではない場合、CQRSの条件付き変更は、大きなオブジェクトを小さなオブジェクトに分割することで構成されます。 

public interface IUserService { void Add(string name); void Delete(string id); List<User> Fetch(Criteries criteries); }


分解後、2つのコマンド（AddUser、DeleteUser）とクエリ（FetchUser）を取得します。これにより、クラスの数は増えますが、クラス内のいくつかの関連メソッドを取り除くことができます。

「今、さらに多くのクラスを書く必要があります！」-これはNレイヤー開発者からの最初の質問ですが、反論として、タスクのアトミック性（オブジェクト間の依存関係なし）を取得することを選び出すことができます。

  1. VCS（バージョン管理システム）の競合が少ない
  2. クラス検索はメソッドよりも簡単です
  3. UserServiceを部分的に分割する必要がなくなりました。1つをナビゲートするのが難しいためです）
  4. コマンドとクエリから形成されたバグトラッカーの問題
  5. アジャイルのスプリントは、コマンドとクエリから形成されます
  6. 小さなオブジェクトのテスト




私たちの実装



すぐにコードを「感じ」たいと思っているせっかちな人のために、 GitHubからソースコード（Incoding CQRS + MVDの例もあります）をダウンロードできます。

注：プロジェクトを開始するには、空のデータベースを作成し、web.configでそのConnectionStringを指定する必要があります（キーメイン）



ディスパッチャー


単一のトランザクション（作業単位）でメッセージ （ コマンドまたはクエリ ）を実行するキー要素。

コントローラー -ディスパッチャーを使用するためのasp.net mvc（コンソール、wpf、owinなど）システムのフレームワークでは、IoCからインスタンスを取得する必要があり、次の2つのメソッドを使用できます。

  • プッシュ-コマンドを実行します
  • クエリ-クエリを実行します 


 public ActionResult Test() { var dispatcher = IoCFactory.Instance.TryResolve<IDispatcher>(); var id = dispatcher.Query(new GetIdQuery()); dispatcher.Push(new DeleteEntityByIdCommand(id)); return something ActionResult; }




作業単位-Commandの実装を見ると、メインコードがオーバーロードされたExecuteメソッドに含まれていることがわかります。これは、Dispatcherの参加なしで呼び出すことができますが、データベースとトランザクションへの接続は開かれません。 

 new DeactivateEntityCommand().Execute(); // without transaction and connection dispatcher.Push(new DeactivateEntityCommand()); // open transaction and connection




メッセージ


CommandBaseとQueryBaseはMessageの子ですが、その動作は作業単位が作成される分離レベルのタイプが異なります

  • コマンド-ReadCommitted
  • クエリ-ReadUncommitted（読み取り専用）


注：制限は厳しいように思えるかもしれませんが、何らかの理由でクエリにデータを保存（削除、挿入）する必要がある場合は、スクリプトを小さなタスクに分割して修正する必要があります。



メッセージには2つの主要なツールがあります。

リポジトリ -データベースインターフェイス、すべてのCRUDスクリプトをサポート

例
作成する 


 Repository.Save(new entity())


読む 


 Repository.GetById<TEntity>(id); Repository.Query(whereSpecification: spec, orderSpecification:spec, paginatedSpecification:spec)


注：クエリ（ページ区切り）は、最も広範なリポジトリメソッドであり、クエリ仕様を使用して、取得するデータを記述します。

更新する 


 var entityFromDb = Repository.GetById<TEntity>(id); entityFromDb.Title = "New title"; // tracking


注：プロバイダーORMが追跡をサポートしていない場合、Repository.SaveOrUpdate（エンティティ）メソッドを呼び出す必要があります

削除する 


 Repository.Delete<TEntity>(id);






イベントブローカー -コマンド間の通信。これにより、 繰り返し発生する「断片」のコードを集約し、イベントとサブスクライバーをカプセル化できます。



タスク：いくつかのアクションの監査

問題：監査を保存するためのコードは同じであり、各コマンドで繰り返す必要があります

サービス層のソリューション：基本クラスServiceWithAuditBaseを割り当てることができますが、監査の複雑さが増すと維持が難しくなり、継承は常に複雑になります。

加入者ソリューション

イベントコード 

 public class OnAuditEvent : IEvent { public string Message { get; set; } }


注：イベントがIEventを実装する条件

加入者コード 

 public class AuditSubscriber : IEventSubscriber<OnAuditEvent> { readonly IRepository repository; public AuditSubscriber(IRepository repository) { this.repository = repository; } public void Subscribe(OnAuditEvent @event) { this.repository.Save(new Audit { Message = @event.Message }); } public void Dispose() { } }


注：サブスクライバーはIoCFactoryを介して作成されるため、ctor（コンストラクター）に注入するか、IoCFactory.Instance.TryResolve（）を使用できます。

コマンドコード 

 EventBroker.Publish(new OnAuditEvent { Message = "New product {0} by {1}".F(Title, Price) });




問い合わせ


カスタムクエリを作成するには、QueryBaseを継承する必要があります。ここで、期待されるデータリターンを指定し、ExecuteResultメソッドをオーバーライドします。 

 public class GetProductsQuery : QueryBase<List<GetProductsQuery.Response>> { public class Response { public string Title { get; set; } public string Price { get; set; } } public string Title { get; set; } public decimal? From { get; set; } public decimal? To { get; set; } protected override List<Response> ExecuteResult() { return Repository.Query(whereSpecification: new ProductByTitleWhere(this.Title) .And(new ProductBetweenPriceWhere(this.From, this.To))) .Select(product => new Response { Title = product.Title, Price = product.Price.ToString("C") }) .ToList(); } }


ネストされたクラスが結果として使用されることを区別できますが、なぜそうではないのか...

データベース（エンティティ）からオブジェクトをすぐに返す -このメソッドには、データベースに接続するセッションのスコープに関連する問題があります。例を考えてみましょう。

例
クエリコード 

 return Repository.Query<Product>();


コントローラーコード 

 dispatcher.Query(new GetProductsQuery()) .Select(r=> new { Amount = r.Orders.Sum(r=>r.Price))


クエリが完了した後、セッションが閉じられ、Ordersフィールドにアクセスするとリクエストがデータベースに送信されるため、マッピングORMのレベルでLazy Load（OLAPオブジェクトのみに関連する）をオフにしないと、エラーが実行時に発生します。



ViewModelはネストされたクラスと同じですが、他のクエリで再利用する機能がありますが、これは非常にまれなシナリオです。



コマンド


Command for CQRSの最初の実装では、説明（AddUserCommand）をエグゼキュータ（UserCommandHandler）から分離しました。これにより、開発プロセスが複雑になり、これらの部分がさらに結合されました。

注：分離の主な理由は、SOAPシステムのDTO（データ転送オブジェクト）モデルのサポートでしたが、asp.net mvcの出現により、関連性がなくなりました。



カスタムコマンドを作成するには、CommandBaseを継承し、Executeメソッドをオーバーライドする必要があります 

 public class AddProductCommand : CommandBase { public string Title { get; set; } public decimal Price { get; set; } public override void Execute() { var product = new Product { Title = Title, Price = Price } Repository.Save(product); Result = product.Id; } }


注：コマンドがデータを返す必要があるシナリオがあります。コマンドメソッドに結果を入れることができます



キリング機能



コンポジット


CQRSは、複雑なタスクを小さなタスクに「分割」するのに役立ちますが、実行トランザクション全体に問題があります。

タスク ：3段階でオブジェクトを保存する

解決策 ：3つのコマンド（Step1Command、Step2Command、Step3Command）に分割します

条件 ：トランザクション 

 public ActionResult Save(Step1Command step1,Step2Command step2,Step3Command step3) { dispatcher.Push(composite => { composite.Quote(step1); composite.Quote(step2); composite.Quote(step3); }); return IncodingResult.Success(); }


コマンドを1つのパッケージにグループ化することに加えて、Compositeでは実行結果を操作できます。 タスクを複雑にし、実行後のStep1が新しい要素のIDをステップ2および3に渡すように条件を設定します。 

 public ActionResult Save(Step1Command step1,Step2Command step2,Step3Command step3) { dispatcher.Push(composite => { composite.Quote(step1,new MessageExecuteSetting { OnAfter = () => { step2.Id = step1.Result; step3.Id = step1.Result; } }); composite.Quote(step2); composite.Quote(step3);}); }




ホットスワップ接続文字列


起動時にアプリケーションが接続文字列を受信しないが、作業中など、システムに入った後、サードパーティのサービスが現在のセッションのアドレスを発行する場合、以前に指定したパスを変更できる必要があります。 

 dispatcher.Query(query, new MessageExecuteSetting { Connection = new SqlConnection(currentConnectionString) });


レガシーシステム


「古い」ベースをラップすることは、Incoding Frameworkの問題ではありません。 さまざまな構成で作業するための追加のインフラストラクチャの作成を避け、この詳細を考慮せずにコマンドとクエリを開発できるツールがあります。 

 dispatcher.Query(query, new MessageExecuteSetting { DataBaseInstance = "Instance 2" });


注：各キーには独自のORM設定があります
