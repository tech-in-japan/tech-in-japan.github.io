Mapzenは、管理ユニットの地理的ディレクトリを作成します。 すべてではありませんが、ほとんどの種、そしてほとんどの種が望んでいます。 地理的ディレクトリは管理単位の大きなリストであり、各管理単位には永続的な識別子と、その場所を説明する多数のプロパティがあります。 ハンドブックを行政単位を巡る議論が行われているが解決されていないスペースと考えることは興味深い。 ディレクトリを「Who's On First」、または要するに「WOF」と呼びます。





Wikipediaによると、Who's on First：

...これは、アボットとコステロによって有名になったコメディシーンです。 このプロットはアボットがコステロ野球選手と呼ぶものに基づいていますが、彼らの名前とニックネームはコステロの質問に対する無意味な答えとして解釈することができます。 たとえば、一baseのプレーヤーは「Who」と呼ばれます。 したがって、「一firstにいるプレイヤー」は、質問（「一firstにいるプレイヤーは誰ですか？」）と答え（「一baseにいるプレイヤーの名前はだれか」）の2つの方法で認識されます。 「Who's On First」は、言葉と名前の戯曲を使用した前世紀初頭のバーレスクスケッチに由来します。 たとえば、「The Baker Scene」（店舗はWatt Street（「What street」と一致）にあります）および「Who Dyed」（所有者の名前はWho（Who））です。 1930年のCracked Nuts映画では、コメディアンのBert WheelerとRobert Woolseyが神話の王国の地図を次のような対話で研究しました。「「どちらが？」（どの都市が？） 1930年代初頭の英国の音楽ホール（英国のボードビル劇場）では、コメディアンのウィルヘイが、ウェアから来たが現在はワイに住んでいる学生ハウに先生がインタビューするシーンで演奏しました。

結局のところ、地理的参照とは何ですか？



前述したように、地理ディレクトリは管理単位の大きなリストであり、各管理単位には永続的な識別子と、その場所を説明するいくつかのプロパティがあります。 ハンドブックを行政単位を巡る議論が行われているが解決されていないスペースと考えることは興味深い。



このアイデアの最も単純で友好的な表現は、人々が同じ地域にどのように名前を付けることができるかを考えることです。 時々、2人がこの場所を同じと呼びますが、異なる方法で書き留めます。 ジオコーダーが依然としてクエリの正確さを維持していること、および誤ったクエリが入力された場合にどれだけ面白いかを思い出してください。 次に、これに世界のすべての言語の数を掛けます。



ガイドの目的を説明する最も簡単な方法は、スペルミスの問題を解決するということです。 すべての地域に単純な（多くの場合、数値であるが最も重要な定数）識別子がある場合、この識別子を使用して、可能なコンテキストで同じ地域を参照できます。 また、言語学的な知恵を考えれば、膨大な時間を無駄にしないでください。



たとえば、都市を「モントリオール」（英語）と呼び、「몬트리올」（韓国語）と「モントリオール」（フランス語）または「MTL」（国際）などの名前を付けることができます。削減）など。 したがって、ガイドが、1つの行政単位（または、楽しみのために、すべての人が同意した1つの幻覚）のこれらすべての表現が共存できるスペースであったら、素晴らしいでしょう。



ハンドブックは議論のために存在しているというより複雑な考えを伝えるために、「行政単位」は社会的、政治的、そしてしばしば非常に感情的なレベルでしばしば論争の対象になるため、大きな問題であることを思い出すことができます。 この問題は新しいものではありません。 人々は、自分自身を覚えている限り、所属のビジョンと領土の境界線のために議論し、時には戦います。











単一の本ではなく、単一の段落でも、単一の文でも、長年にわたって異なる心を持っている、しようとしていない、または特定の領域で共通の意見になりたくない。 おそらくこれは役に立たないかもしれませんが、キーと値のペアのリストは、そのような領域のすべてのニュアンスを修正するために作成され、どれだけ完全でなくても、長期にわたる紛争の解決に役立つと信じたいと思います。

可動性



管理ユニットに関する正規の情報ソースは、一意の64ビット数値IDを持つGeoJSONテキストファイルです。 すべてのコンピューターは、テキストファイルと数字で動作できます。 テキストファイルは、古いテキストエディタで表示または調整できます。 テキストファイルはプリンターで印刷できます。 番号は、データベースによって迅速かつ簡単に索引付けされます。



テキストファイルを使用するのは、データにとって特に重要であるためです：時間の経過に伴う使いやすさ、信頼性、移植性。 古き良きテキスト形式の利点は、他のオプションの利点を上回ります。



たとえば、Googleのプロトコルバッファーは優れていますが、使用するには他の多くのGoogleプログラムが必要です。 ESRIのShapefilesも素晴らしいです。その普及率と長い歴史はフォーマットの利便性を裏付けていますが、少し編集するために特別なプログラムをインストールする必要もあります。



これは、テキストファイルまたは静的ファイルが最良の選択であることを意味するものではありません。 すべては特定のタスクに依存し、必要に応じて、すべてのデータをより軽量で便利な形式に変換しますが、単純なテキストファイルに常にアクセスできます。



ジオソン



GeoJSONは、2つの補足的な理由から主要な交換フォーマットとして使用されます。



  • 現時点で最小限のマークアップを持つこのデータ構造。 誰かがより簡潔なマークアップ言語を思いついたら、それに切り替えます。
  • GeoJSONを操作するためのツールは多数ありますが、重要なことは、使用されている他のすべての形式に変換することです。

管理単位のタイプ



管理ユニットの階層については、3つの「クラス」を特定しましたが、その1つはどのタイプのユニットにも属することができます。 これは、他のクラス（または管理単位のタイプ）が存在できないことを意味するものではありません。 このようなセットから始めることにしました。



共通（C）



これらのユニットは、Who's On Firstのすべての階層とすべての管理ユニットに共通です。



重要なポイント：これは、すべてのオブジェクトがこのクラスの1つ以上の共通の上位オブジェクト（たとえば、国、大陸、または単に惑星地球）を持たなければならないことを意味します。 これは、既存の共通階層に適合するために、特定の場所の階層への特定の追加を妨げません。



共通オプション（CO）



このクラスのユニットは、一般的な階層の一部として暗示されていますが、適切でないため、またはそのようなデータがないために存在しない場合があります。 このタイプの例は郡です。



オプション（O）



階層のこれらの部分は、通常、特定の国または地域に固有です。 たとえば、フランスまたはドイツの多くのネストされた部門。 唯一のルール：オプション（O）タイプは、一般（C）階層内のどこかになければなりません。

階層



Who's On Firstの階層はリストとして表示され、その各要素は完全な階層を含むディレクトリです。 ここのように： 



 "wof:hierarchy": [ { "country_id": "85633147", "region_id": "85683255", "county_id": "102072387", "locality_id": "101750223", "neighbourhood_id": "85794581" } ]




これは、Who's On Firstの地形が複数の異なる階層に属することができるという事実によるものです。 たとえば、都市集積（サンフランシスコ内およびその周辺の「サンフランシスコ湾岸地帯」、5つの地区すべて、さらにはニュージャージー州の一部を含む「ニューヨーク」など）のタイプを考えます。同じタイプ。 再び紛争地域。 この決定に至った理由と方法は、別の記事のトピックですが、要するに：



このソリューションが優れている理由：



  • それは視覚的です
  • 複数の階層を簡単に比較できます
  • ユーザーが過度の頭脳を使って階層全体を復元したり、訪問したばかりの次の「洞察」をサポートしたりする必要はありません。
  • 開発プロセスの変更が容易になります（...「公式」の開始前に宣言します）




なぜこの解決策が悪いか、または悪いようです：



  • 都市の集積をサポートする場合、他の多くのユニット（近隣、地区、サイト）が複数の階層を持つことができ、一部の地区はすべての親ユニットを超えて広がることを意味します
  • ファイルサイズ、ディスク容量、チャネル幅-これらはすべて最初の段落の結果であり、GeoJSONファイルの小数点以下6桁以上のスペースと座標に似ています。






紛争地域



すべての地域と多くの入植地は友好的な冗談のレベルで「論争」されていますが、2つ以上の州が関与しているため、一部の領土をめぐる紛争は非常に深刻な変化をもたらします（そして、いわゆる国際法の非国家主題 ）。 そのような論争は、「友好的な冗談」とはほど遠い、暴力と結果に満ちています。



異議申し立てのステータスの期間中、異議を申し立てられたタイプをそのような地域に割り当てます。 紛争地域は、定義により、その階層に2つ以上の親の状態があります。 このアプローチは、各紛争の状況のすべての事実を反映していません。 一方、上記のように、論争の当事者を強調し、タスクのコンテキストで論争を反映する方法について決定を下すことができます。



親IDと親の権利



テリトリーが異なる階層に属することができたとしても、ほとんどの場合、1人が事実上「制御」されていることを意味します。 たとえば、ゴラン高原はシリアとイスラエルによって争われており、これは階層に反映されていますが、依然としてイスラエルの管理下にあります。 



 "wof:hierarchy": [ { "continent_id": "102191569", "country_id": "85632315", "disputed_id": "85632221" }, { "continent_id": "102191569", "country_id": "85632413", "disputed_id": "85632221" } ], "wof:id": "85632221", "wof:name": "Golan Heights", "wof:parent_id": "85632315",




場合によっては、紛争が最近始まったので、誰が領土を管理しているか、またはそれについて確信がないと言うことができず、まだデータを確認しています。 次に、親レコードを-1に設定します。



割り当てて-2が発生します。 これは「：肩をすくめる：世界は難しいものだ」と解釈されるべきです。 たとえば、カザフスタンのバイコヌール宇宙基地 。



代替/交換可能



地理を扱う際に生じる大きく複雑な哲学的質問の1つ： 単純な調整と根本的な変化を区別する方法は？



この問題は単なる地理的なものではありませんが、地理学では目障りです。 たとえば、ポーランド、フランス、ドイツは、1つの領土を割り当てて、降伏させました（何百年もの間）（場合によってはその後の割り当てがあります）。 それらの境界、および境界の存在期間は、地図作成のためだけでなく、活動の他の多くの領域にとっても重要なコンテキスト情報です。 領土がポーランドに属していたときに準備ができていたが、領土がドイツに属していたときに作成された芸術作品を取ります。 変化する地形をどのように着実に特定しますか？



別の例。 ニューヨーク市には、BoCoCaと呼ばれる非常に厳しい地区があります。 BoCoCaは、ブルックリンのビジネスセンターの南にある3つの近くの地区であるBoerum Hill、Cobble Hill、およびCarroll Gardensの略です。 BoCoCaは、通常の意味での名前でも、ほとんどの人が考える地区でもありません。 一方、多くのマップおよびデータセットでは、これがエリア（および名前）です。 考えがどうであれ、BoCoCaは「存在します」。



Who's On Firstでは、BoCoCaを「マクロ地区」にしました。これには、その名前の由来となる3つの地区が含まれます。

管理単位のタイプは非常に重要なプロパティであり、さまざまなアプリケーションで明らかに使用されています。 アプリケーションが地形に関連するプロパティを処理する方法や理由を知る必要はありません。 そして、このWOF ID 85892915をエリア（Quattroshapesからインポートしたときのエリア）と見なすことにした場合、 おそらく左ヒールの要求に応じて、それほど簡単に変更すべきではありません 。



確かに、BoCoCaを領域とは見なしません。 これについては確固たる意見があります。 BoCoCaは地区と見なされますが、私たちの観点からは、もはやそうではありません。 このような問題を解決する方法は、新しいタイプ（BoCoCa-マクロ）で新しいレコードを作成し、残りのレコードを調整して、あるレコードが別のレコードに置き換えられることを示すことです。



たとえば、地区としてのBoCoCaのエントリは次のようになります。 



 "wof:superseded_by": [102147495], "wof:supersedes": [],




マクロ地区としてのBoCoCaは次のようになります。 



 "wof:superseded_by": [], "wof:supersedes": [85892915]




置き換えられたオブジェクトを個別に説明する方法（およびその有無）を決定するのはアプリケーションです。 たとえば、検索エンジンは、置換されたオブジェクトを個別にランク付けしたり、処理から完全に除外したりできます。

重心



各レコードは複数の重心を持つことができます。 「複数の重心」の組み合わせは、矛盾語のように聞こえます。 「重心」という用語は、任意のジオメトリの焦点領域を示します。 異なる重心は、使用タイプを示すプレフィックスで示されます。 例：



  • geom:latitude



    とgeom:longitude



    は、数学的魔法を使用して取得したコンセンサスポリゴンの中心です。
  • lbl:latitude



    とlbl:longitude



    は、最適な命名lbl:longitude



    座標です。 たとえば、サンフランシスコの埋め立て地にはファラロン諸島が含まれています。これは、この埋め立て地の「中心」が太平洋にあることを意味しており、これは名前を付けるのに最適な場所ではありません。
  • nav:latitude



    とnav:longitude



    は、ナビゲーターが表示するポイントです。 たとえば、同じ場所に到着するトラックの荷台ではなく、救急ステーションの入り口。
