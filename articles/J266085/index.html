RethinkDBを使用したリアルタイムWebアプリケーションの構築

翻訳者から：最近、このかなり興味深いデータベースについて学び、新しい記事に出会いました。 Habréには、RethinkDBに関する言葉がほとんどありませんが、それに関連してこの翻訳を行うことが決定されました。 猫へようこそ！







RethinkDBデータベースは、リアルタイム更新のためのWebアプリケーションの開発を簡素化します。



RethinkDBは、リアルタイムアプリケーション用のオープンソースデータベースです。 アプリケーションの更新を継続的にブロードキャストする変更通知システムが統合されています。 常に新しいデータを要求する代わりに、データベース自体に最新の変更を送信させます。 ストリーミング更新を「サブスクライブ」する機能により、アプリケーションのアーキテクチャが大幅に簡素化され、サーバー側への常時接続を維持するクライアントと連携できます。



RethinkDBはJSONドキュメント用のスキーマフリーリポジトリですが、リレーショナルデータベースの一部の機能もサポートしています。 RethinkDBはクラスタリングもサポートしているため、拡張が非常に簡単です。 組み込みのWebインターフェイスを使用して、シャーディングとコピーを構成できます。 RethinkDBの最新バージョンには、3つ以上のサーバーを持つクラスターの自動フェールオーバーも含まれています。 （ 翻訳者のメモ：サーバーの1つが落下した場合でも、データベースを操作し続ける可能性が暗示されています。）



ReQinkと呼ばれるRethinkDBのクエリ言語は、アプリケーションを記述する言語のコードにネイティブに埋め込まれています。 たとえば、Pythonでコーディングする場合、データベースにクエリを書き込むときにPythonの通常の構文を使用します。 各要求は、必要な操作を正確に記述するために開発者がチェーンに入れる関数で構成されます。



ReQLについて一言

RethinkDBには、従来のJSONドキュメントが保存されるテーブルが含まれています。 JSONオブジェクト自体の構造には、深いネストがあります。 RethinkDBの各ドキュメントには独自のプライマリキーがあります。これは、親テーブルの一意の値を持つ「id」プロパティです。 リクエストの主キーを参照すると、特定のドキュメントを取得できます。



アプリケーションでのReQLクエリの記述は、SQLクエリコンストラクターAPIの使用と非常に似ています。 以下では、JavaScriptで、 ユーザーテーブルの一意の姓の数を決定するためのReQLクエリの簡単な例を示します。 



r.table("users").pluck("last_name").distinct().count()


ReQLクエリでは、各チェーン関数は前の関数から取得したデータを処理します。 正確には、このリクエストの実行順序は次のとおりです。



  • テーブルはデータベース内の特定のテーブルをクエリします
  • pluckは、各エントリから特定のプロパティ（または複数のプロパティ）を取得します
  • disctinctは重複する値を削除し、一意の値を1つだけ残します
  • countはカウントし、受け取ったアイテムの数を返します


従来のCRUD操作も簡単です。 ReQLには、新しいJSONドキュメントをテーブルに追加するために使用できる挿入関数が含まれています。 



 r.table("fellowship").insert([ { name: "Frodo", species: "hobbit" }, { name: "Sam", species: "hobbit" }, { name: "Merry", species: "hobbit" }, { name: "Pippin", species: "hobbit" }, { name: "Gandalf", species: "istar" }, { name: "Legolas", species: "elf" }, { name: "Gimili", species: "dwarf" }, { name: "Aragorn", species: "human" }, { name: "Boromir", species: "human" } ])


フィルター関数は 、特定のパラメーターに一致するドキュメントを取得します。 



 r.table("fellowship").filter({species: "hobbit"})


updateやdeleteなどの関数をチェーンに追加して、 filterから返されたドキュメントに対して特定の操作を実行できます 。 



 r.table("fellowship").filter({species: "hobbit"}).update({species: "halfling"})


ReQLには100以上の機能が含まれており、これらを組み合わせて目的の結果を得ることができます。 フローの管理、ドキュメントの変更、集計、記録などの機能があります。 文字列、数値、タイムスタンプ、地理空間座標を使用して標準操作を実行するための「シャープ化」機能もあります。



サードパーティのWeb APIからデータを取得するために使用できるhttpコマンドもあります。 次の例は、 httpを使用してRedditから投稿を取得する方法を示しています。 



 r.http("http://www.reddit.com/r/aww.json")("data")("children")("data").orderBy(r.desc("score")).limit(5).pluck("score", "title", "url")


投稿が受信されると、それらはポイントでソートされ、上位5つの投稿の特定のプロパティが表示されます。 「フルキャパシティ」でReQLを使用すると、開発者は非常に複雑なデータ操作を実行できます。



ReQLの仕組み



RethinkDB クライアントライブラリ （以下「ドライバー」）は、アプリケーションが開発されているプログラミング言語にReQLを統合する役割を果たします。 ドライバーは、データベースでサポートされるすべての種類のクエリの機能を実装します。 ReQL式は、 抽象構文ツリーのように見える構造化オブジェクトと見なされます 。 ただし、要求を満たすために、ドライバーはこれらの要求オブジェクトを特別な形式「 RethinkDBのJSONワイヤプロトコル形式 」に変換し 、そこでデータベースに転送します。



チェーンを閉じるrun関数は、リクエストを変換し、サーバー上で実行し、結果を返します。 通常、操作を完了するために、この関数にサーバー接続を渡します。 公式ドライバーでは、接続の操作は手動モードで実行されます。 これは、操作後に接続を作成して閉じる必要があることを意味します。



次の例は、JavaScript用のReQLドライバーがインストールされたNode.jsからRethinkDBでクエリを実行する方法を示しています。 このクエリは、フェローシップテーブルからすべてのハーフリングを取得し、コンソールに表示します。 



 var r = require("rethinkdb"); r.connect().then(function(conn) { return r.table("fellowship") .filter({species: "halfling"}).run(conn) .finally(function() { conn.close(); }); }) .then(function(cursor) { return cursor.toArray(); }) .then(function(output) { console.log("Query output:", output); })


rethinkdbモジュールは、RethinkDBドライバーのアクセスと使用を提供します。 このモジュールを使用して、クエリをコンパイルしてデータベースに送信できます。 上記の例では、非同期フロー制御にプロミスを使用していますが、ドライバーは通常のコールバックの操作もサポートしています。



connectメソッドは接続を確立し、 run関数が使用します。 リクエストを完了します。 クエリ自体はカーソルを返します 。これは、データベースのコンテンツに対して開いているウィンドウのようなものです。 カーソルは遅延フェッチをサポートし、大量のデータを反復処理する効果的な方法を提供します。 上記の例では、結果のサイズが比較的小さいため、カーソルの内容を配列に変換することにしました。



ReQLクエリは通常のコードとしてアプリケーションに書き込まれますが、データベースサーバーで実行され、結果を返します。 統合は非常にシームレスであるため、初心者はアプリケーションのコードとデータベースの操作の境界がどこにあるのか混乱することがよくあります。



ReQLチェーンとさまざまな言語への統合により、コードの再利用と頻繁な操作の分離が大幅に向上します。 クエリはアプリケーション言語で記述されているため、クエリの部分式を変数と関数にカプセル化することは非常に簡単で便利になります。 たとえば、次のJavaScript関数は、指定された値を既に含むReQL式を返すことにより、ページネーションを一般化します。 



 function paginate(table, index, limit, last) { return (!last ? table : table .between(last, null, {leftBound: "open", index: index})) .orderBy({index: index}).limit(limit) }


ReQLのもう1つの注目すべき利点は、使い慣れたSQLでの作業も提供し、通常のインジェクション攻撃に対して十分に保証されていることです。 危険な文字列連結の必要性を心配することなく、クエリに外部データを簡単に含めることができます。



セカンダリインデックス、テーブルの結合、匿名関数の使用など、ReQLのより高度な機能の多くは、この記事の範囲外のままです。 ただし、必要に応じて、ReQL API ドキュメントページでそれらに慣れることができます。



変更フィードを使用したリアルタイムWebアプリケーションの作成



RethinkDBには変更通知システムが組み込まれているため、リアルタイムアプリケーションの開発が大幅に簡素化されます。 チェーンの最後に変更機能を挿入すると、発生したすべての変更を反映して、リクエストの結果として連続ストリームが起動されます。 このようなフローはチェンジフィードと呼ばれます（以降、「チェンジフィード」と呼びます）。



通常のデータベースクエリは、従来の要求/応答Webモデルに適しています。 ただし、サーバーへの常時接続またはストリーミングデータを使用するリアルタイムアプリケーションでは、サーバーを常時ポーリングすることは実用的ではありません。 Chengfidsは、定期的なポーリングの代替手段、つまり、更新された結果を常にアプリケーションに送信する機能を提供します。



変更ログをテーブルに直接添付して、その内容に対する変更を追跡できます。 Chengfidsをより複雑なクエリで使用して、必要なデータのみの更新を受信することもできます。 たとえば、 orderBy関数とlimit関数を使用するクエリに変更ログを添付して、マルチプレーヤーゲームの動的なハイスコアテーブルを作成できます。 



 r.table("players").orderBy({index: r.desc("score")}).limit(5).changes()


プレイヤーはポイントでソートされ、最初の5つが表示されます。 この上位5つに変更があるとすぐに、Chengfidは更新されたデータを送信します。 最初にトップ5にいなかったプレーヤーが十分なポイントを獲得し、他のプレーヤーをトップ5から追い出した場合でも、Chengfidはこれを報告し、リストを更新するために必要なすべてのデータを転送します。



Chengfidは、レコードの新しい値だけでなく、前の値も送信するため、結果を比較できます。 エントリのいずれかが削除されると、その新しい値はnullになります 。 表示されたばかりの新しいレコードと同様に、古い値はnullになります 。 ところで、受信データの操作が必要な場合は、 変更後にチェーンに他の操作を追加できます。



changesコマンドでクエリを実行すると、カーソルが返されますが、カーソルは永久に開いたままになります（ウィンドウを覚えていますか？）。 カーソルは、新しい変更が利用可能になるとすぐに表示されます。 以下に、Node.jsのChengfidから更新を取得する方法を示す例を示します。 



 r.connect().then(function(conn) { return r.table("data").changes().run(conn); }) .then(function(cursor) { cursor.each(function(err, item) { console.log(item); }); });


Chengfidカーソルはバックグラウンドで実行されているため、アプリケーションはブロックされません。 Node.jsなどの基本的に非同期な環境では、正しい操作のために追加の措置を講じる必要はありません。 他の言語で作業する場合、おそらく非同期コードのフレームワーク、またはスレッドの手動実装が必要になります。 PythonおよびRuby用の公式RethinkDBドライバーは、TornadoやEventMachineなどの一般的で広く使用されているフレームワークをサポートしています。



現在、 changesコマンドはget、between、filter、map、orderBy、min 、 maxの各関数で機能します 。 他の種類のクエリのサポートは将来的に計画されています。



RethinkDBを使用してリアルタイムWebアプリケーションを作成する場合、WebSocketsを使用して、フロントエンドに更新をブロードキャストできます。 また、 Socket.ioのようなライブラリは使いやすく、このプロセスを簡素化します。



Chengfidsは、水平方向の拡張用に設計されたアプリケーションに特に役立ちます。 アプリケーションの複数のインスタンス間で負荷を分散する場合、通常、メッセージキューやメモリ内dbなどの追加メカニズムを使用して、すべてのサーバーに更新を分散します。 RethinkDBはこの機能をアプリケーションのレベルまで引き上げ、アーキテクチャを平坦化し、追加のインフラストラクチャの必要性を排除します。 各アプリケーションインスタンスはデータベースに直接接続して、新しい変更を受信します。 更新が利用可能になると、各サーバーは対応するWebSocketクライアントに更新をブロードキャストします。



リアルタイムアプリケーションに加えて、Chengfidsはモバイルプッシュ通知およびその他の同様の機能の実装を大幅に簡素化できます。 Chengfidsは、データベースとの対話のイベント指向モデルを表し、このモデルは多くの場合に役立ちます。



RethinkDBスケーリングとクラスター管理



RethinkDBは、クラスタリングと単純な拡張を目的とした分散データベースです。 新しいサーバーをクラスターに追加するには、コマンドラインから--joinオプションを指定して起動し、既存のサーバーのアドレスを指定するだけです。 複数のサーバーを自由に使用できるクラスターがある場合は、各テーブルに個別にシャーディングとコピーを構成できます。 データベースの1つのインスタンスで機能する設定と機能は、クラスターでも正確に機能します。



RethinkDBサーバーには、ブラウザーで直接開くことができるWeb管理インターフェイスも含まれています。 このインターフェイスを使用すると、クラスターの動作を簡単に管理および監視できます。 数回のクリックでシャーディングとコピーを設定することもできます。



RethinkDBでは、ReQLアプローチをクラスター構成に適用できます。これは、微調整と自動化に最適です。 ReQLには、テーブルにバインドしてシャーディング設定を設定できる単純な再構成機能が含まれています。 クラスターは、R​​ethinkDBの一連の特別なテーブルを通じて、その状態と設定に関するほとんどの内部情報も提供します。 システムテーブルをクエリして、設定を変更したり、監視情報を受信したりできます。 Webインターフェースを介して提供されるほとんどすべての機能は、ReQL APIに基づいています。



changelidsをReQLモニタリングAPIと組み合わせて使用​​して、サーバーデータのストリームを受信することもできます。 たとえば、統計情報を含むシステムテーブルに変更ログを添付し、リアルタイムでデータを転送して読み取り/書き込み負荷のグラフを作成する独自の監視ツールを作成できます。



最近リリースされたRethinkDB 2.1には、自動フェールオーバーのサポートが組み込まれています。 新しい機能により、クラスターの可用性が向上し、データベースサーバーがクラッシュするリスクが軽減されます。 プライマリサーバーに障害がある場合、残りのセカンダリワーキングサーバーは新しいプライマリを「選択」し、障害のあるサーバーが起動するか、クラスターから削除されるまでこの役割を実行します。

ほとんどのサーバーがオンラインになるまで、鉄の故障やネットワークの停止はデータの可用性に影響を与えません。
