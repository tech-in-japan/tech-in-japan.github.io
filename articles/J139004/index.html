クライアントコードのデバッグとプロファイリング



JavaScriptコードをデバッグする「クラシック」な方法は、 alert



関数とその派生物を使用することです。 キャリアの初めに、JavaScriptのprint_r



関数を個人的に作成したことを覚えています。配列とオブジェクトのデバッグ情報を出力する機能がなかったからです。 次のようになりました。 

 function print_r(variable) { if (variable instanceof Array || variable instanceof Object) { var key; for (key in variable) alert(key + ' => ' + variable[key]); } else { alert(variable); } }




もちろん、スピーチのプロファイリングはまったく行われていません。



このアプローチにより、 コンソールオブジェクトに関する情報でさえ革命的です。



Webアプリケーションのクライアント側の仕様では、一般的なすべてのブラウザーでコードのデバッグが必要です。 もちろん、Internet Explorerおよび他の通常のブラウザーでのデバッグで十分な場合がよくありますが、すべてのオプションを検討します。



Mozilla Firefox


おそらく、Firefoxはクライアントコードのデバッグの先駆者と言えるでしょう。 長い間、開発に最適なブラウザとして示されていましたが、 Firebug拡張機能のおかげで、HTMLコードの検証を除く必要な機能がすべて含まれている可能性があります。







また、バージョン4から、組み込みのWebコンソールが登場しました。これは、Firebugの[コンソール]タブと[ネットワーク]タブの機能の一部、およびいくつかのCSSデバッグ機能を実装しています。







バージョン6以降、シンプルなJavaScriptエディターが登場しました。これはFirebugの機能の1つも実装しており、ブラウザーでコードを直接記述および実行できます。







バージョン10から、ページのインスペクターが登場しました。これにより、HTMLコードとCSSプロパティを調べることができます。つまり、HTMLタブの機能を実装します。







HTMLコードの検証には、通常、 Html Validator拡張機能が責任を負います。 サイトのメインページhabrahabr.ruのエラー数を示すアイコンだけが、ページインスペクターの画像のブラウザーの右下隅に表示されます。



また、この機会を利用して、このブラウザーには生活を楽にする多くの拡張機能があることを示します。これについては 、Habréで既に対応する記事がありました。

Xdebug


最初に合意したように、サーバーでPHPを使用する場合を検討します。 ここで、「古典的な」デバッグ方法はecho

xdebug拡張モジュールを使用すると、少なくともステップごとにコードを実行し、変数の値を表示できます。これにより、PHPプログラミングが新しいレベルに引き上げられます。 xdebugでの作業の複雑さについては、対応する記事がありました 。 XDebugは通常、GNU / Linuxリポジトリで使用できますが、Windowsでは、dllファイルをコピーしてインストールすることもそれほど難しくありません。



この拡張機能を使用する場合、着信接続（デフォルトではポート9000）がサーバーから開発者のコ​​ンピューターに到着し、それを処理する必要があります。 これを行うには、IDEを適切に構成する必要があります。



ところで、IDEを使用することも、前進するための前提条件です。 一部のプログラマーは、コードハイライト付きのメモ帳とIDEのプログラミングの違いは大きなプロジェクトでしか見られないと考えていますが、個人的には「Hello world！」というプログラムでも違いが見えると思います。 。

プロファイリング


アーカイブを解凍して受け取るディレクトリには、拡張ソースに加えて、プロファイリング結果を調査するためのWebインターフェイスと、アプリケーションをプロファイリングするためのライブラリが含まれています。



プロファイリングの例を示します。 次の要素をアプリケーションコードに含める必要があります。 

 //  ,   //    ,     xhprof_enable(XHPROF_FLAGS_CPU + XHPROF_FLAGS_MEMORY); /* *    */ //  ,  , //     $xhprofData = xhprof_disable(); include_once XHPROF_DIR.'/xhprof_lib/utils/xhprof_lib.php'; include_once XHPROF_DIR.'/xhprof_lib/utils/xhprof_runs.php'; $xhprofRuns = new XHProfRuns_Default(); $namespace = 'some-unique-name'; $runId = $xhprofRuns->save_run($xhprofData, $namespace); echo "<!-- ", $runId, ' ', $namespace, " -->\n";


ここで、定数XHPROF_DIR



は、ダウンロードしたアーカイブを解凍したディレクトリを指します。



結果を分析するには、同じWebインターフェイスが必要です。 $XHPROF_DIR/xhprof_html/



で取得できます-このように任意に指定します。 たとえば、Webサーバーからアクセス可能な場所に配置し、 example.com/system/xhprof/



で利用できexample.com/system/xhprof/



。作業結果を分析するには、次のようにアクセスする必要があります。



example.com/system/xhprof/?run=%runId%&source=%namespace%







同様の結果が得られます。





プロファイリングは、継続的にアプリケーションコードに含めることができます。また、たとえば、特定の確率で、または特定の条件の存在によってランダムに開始するように作成することもできます。 たとえば、次のように： 



 $needProfiler = (mt_rand(0, 100) < 10 or isset($_COOKIE['xhprof'])); if ($needProfiler) xhprof_enable(XHPROF_FLAGS_CPU + XHPROF_FLAGS_MEMORY);


この場合、顧客からの苦情や疑いがある場合、特定の期間のプロファイリング結果を参照できます。 namespace



パラメーターを使用して、アプリケーションのどの部分（どのスクリプト、コントローラー、アクション）がプロファイルされたかを判別できます。
