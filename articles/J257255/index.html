仮想化されたMicrosoft SQL Serverデータベースのバックアップと復元

Veeam Backup＆Replicationのバージョン8.0は、完全な仮想マシンリカバリ、ファイルレベルの個別リカバリ、Microsoft SQL Server用のVeeam Explorerツールを使用したデータベースリカバリの一般的な方法など、仮想化されたSQLサーバーとデータベースのバックアップと復元のための幅広いオプションを提供します。 これらの最後は、今日の私の話です。







もちろん、「ストーブから」開始する必要があります。つまり、回復オプションとバックアップ設定および最も仮想化されたSQL Serverとの関係を理解することから始めます。



Veeamがイメージレベルで仮想マシンのバックアップコピーを作成することは誰もが知っています。つまり、SQL Serverの場合、サーバー上のすべてのインスタンスとデータベースを含むサーバー全体がバックアップされます。 バックアップがトランザクションに関して一貫している（トランザクションに一貫している）ことを確認するには、作成時に、アプリケーションの操作を考慮した画像処理を使用する必要があります（アプリケーション対応の画像処理）。 さらに、サーバーのバックアップコピーを準備して、後で特定のデータベースを任意の時点の状態に復元したり、選択したトランザクションに復元したりすることもできます。



リソースを計画します



後でデータベースを回復するためにSQL Serverバックアップを計画する際に考慮すべきいくつかの要素を次に示します。

  1. インフラストラクチャリソースとバックアップおよびリカバリポリシーの要件、つまり：

    • どのSQLサーバーをバックアップする必要がありますか？

      Veeamは、Microsoft SQL Server 2005 SP4以降で動作します。 すべてのエディションがサポートされています。 Microsoft SQL Server 2012以降では、AlwaysOn可用性グループがサポートされています。

    • データベースに必要なRTOとRPOは何ですか？

      バックアップの頻度もこれらの値に依存します。 原則として、バックアップが頻繁に行われるほど、回復にかかる時間は短くなります。

    • アプリケーションはどの程度集中的にデータベースにアクセスしますか？

      RTOインジケータとともに、バックアップウィンドウ、バックアップジョブスケジュール、およびトランザクションログのバックアップを保存する場所の計算（必要な場合）を設定する際に、この要素を考慮する必要があります。

    • バックアップリポジトリは十分な大きさですか？

      サーバー自体とトランザクションログのバックアップに十分なスペースが必要です。 おおよそ必要なスペースを見積もると役立ちます。 これは、たとえば、計算機を使用するか、 インフラストラクチャ評価レポートパッケージのVM変更率推定レポートを使用して実行できます（Veeam ONEがインストールされている場合）。



  2. 必要なデータベースのログおよび復旧モデル。 トランザクションログを処理するために使用できるオプションと、将来適用できる回復シナリオによって異なります。

    • データベースにシンプルモデルが指定されている場合は、特定の復旧ポイント（バックアップまたはレプリカ）の作成時の状態にのみデータベースを復元できます。 このシナリオでは、SQL Serverはログを自動的に変換します。
    • モデルが異なる場合（ FullまたはBulk-logged ）、自動トランケートは行われず、ログの処理に必要なオプションを構成できます。


  3. バックアップとリカバリに使用されるツールは何ですか？

    • SQL Serverを使用した仮想マシンのバックアップと対応するデータベースのトランザクションログにVeeamを使用する場合、マシンのバックアップタスクの設定で、 このジョブオプションでトランザクションログを処理するオプションを選択する必要があります。







      その後、トランザクション処理の処理設定が利用可能になります（これについては後で説明します）。

      注：別のアプリケーション（SQLではなく、Exchange Serverなど）を使用してマシンのバックアップタスクを構成するときにこのオプションを選択すると、このアプリケーションのトランザクションログの自動トランキングが有効になります。 Exchangeでの作業の詳細については、Habréに関する記事を参照してください 。

    • 仮想マシン全体のバックアップを作成するためだけにVeeamを使用し、サードパーティアプリケーションなどに処理するためにトランザクションログを転送する場合、Veeam Backup＆Replicationに通知する必要があります。つまり、バックアップチェーンが別のツールによって作成されることを示します。 VeeamはフラグCOPY_ONLYでバックアップを作成する必要があります（この場合のみ、チェーンは変更されません）。 これは、[ コピーのみ実行]を選択することで実行されます（別のアプリケーションにログを使用させる） ：







      この後、トランザクションログの処理設定は不要として「非表示」になります。 Veeamは、VSS_BS_COPYメソッドを使用してこのSQL Serverをバックアップし、スナップショットを作成します。 トランザクションログは移動されないため、データベース管理者は将来の運命に注意する必要があります。





ちょっとした理論：プロセスがどのように進むかを理解しています



いつでも、または選択したトランザクションにデータベースを復元する機能を提供するには、以下を作成する必要があります。

  1. アプリケーション（つまり、SQL Server）の操作を考慮した仮想マシンのバックアップコピー
  2. トランザクションログのバックアップコピー。これを既存のバックアップに「ロール」して、データベースを目的の状態にします


Veeam Backup＆Replicationがこれらのタスクを管理する方法を見てみましょう。



そのため、ログをバックアップする必要がある場合（データベースの最も「ターゲットを絞った」回復の可能性が得られます）、SQL Serverのバックアップタスクには、実際には2つの相互に関連するタスクが含まれます。

  1. SQL Serverを使用した仮想マシンのバックアップ-マシンイメージレベルで実行され、以降「親」として表示され、UIで指定した名前（ Test Jobなど ）が付けられます。 ユーザーがコンソールで作成し、スケジュールに従ってまたは手動で起動します。
  2. トランザクションログのバックアップ-以降、「子」として表示され、「親」の名前とSQLバックアップのサフィックスが付きます（この例では、 Test Job SQL Backupを取得します ）。 両方の条件下で、Job Managerコンポーネントによって、ユーザーの介入なしに自動的に作成されます。

    • スケジュールに基づいて実行され、アプリケーションに基づいてバックアップを作成する「親」が少なくとも1つあります（「親」の設定で「 アプリケーション対応の画像処理を有効にする」 が選択されている）
    • ログバックアップオプションが有効になっている（「親」設定でも）




このタスクの「束」は次のように機能します。





ステップ1と2



スケジュールに従って、Veeamバックアップサーバーで実行されているJob Managerは「親」を起動し、SQL Serverを使用して仮想マシンのバックアップを作成し、リポジトリに配置します。



ステップ3



「子」セッションが開始されます。VeeamBackup＆Replicationは、最初にVeeam Log Shipper Serviceと呼ばれるコンポーネントをSQLサーバーのゲストOS内にインストールします。 このサービスは「子供」のセッション全体で機能し、ログの処理を担当します。 特に、セッション中にバックアップするデータベースのログに関する情報を収集し、データをリポジトリに転送する方法を（直接またはログ配布サーバーを介して-ジャーナルプロキシと呼びましょう）。 「子」セッションの終了時に、このコンポーネントは停止し、ゲストOSから破棄されます。 新しいセッションが始まると、すべてが繰り返されます（インストール、データ収集、停止、アンインストール）。



ゲストOS上のトランザクションログのバックアップは、SQL Server自体を使用して作成されます。 また、ゲストハウスでtrankateログを実行します。 結果のコピーは、ゲストファイルシステム上の一時フォルダーに.BAKファイルとして保存されます（デフォルトでは、これは％allusersprofile％\ Veeam \ Backupです ）。



ステップ4



15分ごと（デフォルトの頻度）に、Job ManagerはこのSQL Serverで使用可能なデータベースを識別し、結果のリストを仮想マシンバックアップの内容（Veeam Backup＆Replicationデータベースから取得）のデータと比較します。 したがって、この15分間にリポジトリに「取得」され、対応するSQLサーバーバックアップの隣に置かれるログがあるデータベースが判明します。



注：何らかの理由で以前の間隔でリポジトリに「取得」されなかったログのコピーがまだある場合、Veeamは一時フォルダー内の.BAKファイルを列挙してそれらを識別し、「トランスポートする」としてマークします。



ステップ5



.BAKファイルは、ゲストからリポジトリに（直接または「ジャーナルプロキシ」を介して）転送されます。 ソース側のVeeamトランスポートサービスは、デフォルト設定に従って圧縮を実行します。 リポジトリ側では、「親」タスクの設定に従って圧縮が実行されます（詳細については、ロシア語をご覧ください）。



リポジトリへのデータの「トランスポート」の後、それらは仮想マシンの一時フォルダーから削除されます。 割り当てられた間隔中に何かを「転送」できなかった場合、これらのファイルは削除されず、次の15分間にVeeamが再試行します。



15分間隔（いわゆるログバックアップ間隔）の間に実行される操作を図に示します。







「親」タスクの2つの連続した開始間のこのような間隔の合計（その期間は変更可能）は、「子供」セッションです。

  • 最初のセッションは、「親」設定で「スケジュールのアクティブ化」が示された瞬間に開始されます。
  • 「子」タスクは、「親」の開始間のバックグラウンドで動作し続け、上記の操作を実行します。 「親」の起動により、現在の「子供の」セッションが終了し、新しいセッションがすぐに開始され、「親」とともにステップ1〜5を実行します（上記を参照）。
  • 「子供」のセッションは、「親」の開始直前、または「親」が無効になったときに終了します（メニューコマンドまたはスケジュールの無効化により）。

例



写真を言葉で説明します。 「親」タスクが、5月5日から毎日午後11時に実行されるように設定されているとします。 その後、次のシーケンスが実行されます。

  1. 最初の「子供向け」セッションは、「親」の自動開始スケジュールが有効になった時点で開始されます（仮想マシン自体のバックアップはまだ作成されていません）-この設定は5月5日の7時に行われました。 「子」タスクはアイドル状態で、マシンのバックアップが作成されるのを待っています。
  2. 5月5日の午後11時に「親」が起動され、SQLサーバーのバックアップが作成されます。
  3. 「子」タスクが作業状態になり、15分間隔のシーケンスが開始されます。 新しいセッションの開始時間が近づく前に、ログのすべてのバックアップがリポジトリに正常に「転送」されると、「子」はアイドル状態に戻り、セッションが期限切れになるまで待機します。
  4. 5月6日の午後11時に、「親」が再び開始され、新しい「子供」セッションが開始されます。




注：ログのバックアップには、対応するデータベース（読み取り、サーバー）のバックアップが必要なので、この新しいデータベースを使用したサーバーのバックアップが成功した後にのみ、各新しいデータベースのログがバックアップされることに注意してください。



リポジトリには何が保存されますか？



.VLBファイル形式のトランザクションログのバックアップコピーは、リポジトリ（対応するサーバーのバックアップの隣）に保存されます。 フォルダーへのデフォルトパスはC：\ backup \ <SQL_server_VM_backup_job_name>です。 .VBMファイル形式のバックアップチェーンメタデータもそこに配置されます。



デフォルトでは、.VLBファイルを保存するポリシーは、SQLサーバーのバックアップを保存するポリシーに準拠するように構成されているため、選択したサーバーの復元ポイントにロールされ、必要なデータベース状態を取得できるログが常に存在します。 当然、保存期間の終了後に復元ポイントが削除されると、対応するログバックアップも削除されます。



別のオプションがあります-指定された日数の間.VLBバックアップを保存します。 スペースを節約し、最新の状態に復元するために最新のデータのみを保存する場合に役立ちます。 ただし、サーバーのバックアップおよび.VLBファイルのストレージポリシーが、サーバーの復元ポイントが対応する.VLBよりも前に削除されないように構成されていることを確認する必要があります。そうでない場合、ログを「ロール」しません。

設定するオブジェクトの選択



「親」タスクに含めるオブジェクトを選択するステップでは、次のことを考慮する必要があります。

  • 同じインスタンスにあるデータベースは順番に処理されます。
  • 1つ以上の特定のデータベースを除き、すべてのデータベースのトランザクションログのバックアップを作成する場合は、これらの例外データベースに対して、 単純な復旧モデルを指定する必要があります。
  • SQL Server 2012以降では、VeeamはAlwaysOn可用性グループをサポートします（SQLインスタンスがWSFCノードにあることを確認します）。




実行中のアプリケーション（SQLサーバー）を考慮した仮想マシン処理のセットアップ



ゲストの処理ステップでは、ゲストOSの処理をセットアップするときに、次のオプションを選択します。

  1. 仮想マシンの処理がアプリケーションの動作を考慮に入れるように、[ アプリケーション対応の処理を有効にする]チェックボックスをオンにします。







  2. [ ゲストOSの資格情報]セクションに移動し、 ゲストOSでアクションを実行するアカウントを指定します（ゲストOSからリポジトリへのログのバックアップおよびログトランケートを含む）。 このアカウントには次の権限が必要です。

    • バックアップの場合、処理されたSQL Serverのsysadmin固定サーバーロール。
    • trankeytaの場合-SQL Server 2012またはSQL Server 2014がある場合、少なくとも目的のデータベースのdb_backupoperatorロール、または再びサーバーロールsysadmin 。


    重要な注意： Veeam Backup＆Replication 8.0 Update 2（ビルド8.0.0.2021）以降、指定されたアカウントでのタンキリング試行に失敗した場合、NT AUTHORITY \ SYSTEMアカウントでさらに1回試行されます。



    したがって、SQL Server 2012またはSQL Server 2014の場合、このアカウントに十分な権限があることを確認する必要があります（詳細については、 http：//www.veeam.com/kb1746を参照してください ）。

    SQL Server 2005、2008、および2008 R2の場合、それらのデフォルト設定はローカルSYSTEMに必要な権限を提供します（ただし、誰かがそれらを変更できる可能性は常にあります。この場合、再度再確認することをお勧めします）。

  3. 次に、表示されたリストで[ アプリケーション ]ボタンを押し、必要なSQL Serverを選択して[ 編集 ]ボタンをクリックします。その後、ゲストOSの処理設定を示すダイアログが開きます。
  4. [ 全般 ]タブに移動し、そこにある[ アプリケーション ]セクションを見つけて、[ 正常な処理が必要（推奨） ]オプションを選択します。 したがって、トランザクション的に整合性のあるバックアップバックアップコピーを作成することを目指しています。







  5. 次に、 トランザクションログセクションに移動し、 このジョブオプションでトランザクションログを処理するを選択します-上記のように、これは仮想マシンバックアップとログバックアップの両方がVeeamを使用して実行されることを意味します。
  6. その後、[ SQL ]タブが使用可能になります。ここで、ログをどのように処理するか（つまり、トランザクションログ）を選択する必要があります。 次のオプションが利用可能です。

    • ログを切り捨てます（ログが永久に成長するのを防ぎます） -ログをトランキーにしたいです。 ゲストOSでの作業に使用されるアカウントには適切な権限が必要であることを忘れないでください。 このバージョンでは、Veeamは仮想マシンがバックアップするのを待ってからログトランケートに信号を送ります（VSSを使用して実行されます）。 このオプションを使用すると、選択した復旧ポイントにのみデータベースを復元できます（「ターゲット」復旧は不可能になります）。
    • ログを切り捨てないでください（単純な復旧モデルが必要です） -ログをSQLサーバーにそのまま残したい、つまり Veeamはそれらに対して何もしません（バックアップもトランケートも行いません）。 このオプションでは、データベース管理者が何らかの操作を行う必要があります（そうしないと、ログが大きくなり、多くのスペースを占有する可能性があるため、「このオプションを選択した場合は単純なモデルを使用します」と表示されます）。 また、このオプションを使用すると、選択したポイントまでのみ回復できます。
    • ログを定期的にバックアップする -ログバックアップをSQLサーバー自体のバックアップの隣のリポジトリに保存します。 このオプションを使用すると、あらゆる回復シナリオを使用できます。



      ログをリポジトリに「転送」する頻度を選択します（デフォルトでは、これらは15分ごとに同じです）。 ストレージポリシーを設定します（ 対応するイメージレベルのバックアップに従って推奨）。 結論として、データがリポジトリに「移動」する「ジャーナルプロキシ」を示します。







  7. ゲストOSからリポジトリにファイルを転送するために、2種類のトランスポートがサポートされています。

    • 追加のリソースを必要とせず、少なくともゲストシステムをロードするため、ゲストOSからリポジトリに直接接続するのが最善の方法です。
    • 直接接続を確立できない場合は、ログ配布サーバーまたは「ジャーナルプロキシ」を使用します。これは、Veeam Backup＆Replicationコンソールに追加されたWindowsサーバーです。


    「ジャーナルプロキシ」の選択が自動的に行われるように指定することも、リストから適切なサーバーを自分で選択することもできます。 Veeamは、サーバーの1つをプロキシロールに自動的に割り当てます。2つの基準-データ転送の可能な方法とネットワーク経由の可用性に焦点を当てます。 また、この目的で複数のプロキシを使用することも合理的です（障害が発生した場合）。 ジャーナルプロキシの詳細については、 こちらをご覧ください。
  8. スケジュールに関しては、「親」タスクを最小生産時間で実行することをお勧めします。 「子」タスク（つまり、ログバックアップ）は、デフォルトでは15分ごとにリポジトリにログを「転送」しようとします-この間隔は変更できます。 もちろん、ファイルがリポジトリに転送される間のリアルタイムを考慮する必要があります（つまり、プロセスが実際に15分すべてかかる場合、間隔を5分に設定しないでください）。


それは、一般に、すべてです。 [ OK]をクリックして設定を保存し、ダイアログを閉じて、タスクウィザードに戻り、最後まで進みます。 同時に、「親」タスクの「スケジュール」ステップで、「 ジョブを実行する」チェックボックスを自動的に選択することが必須であることに注意してください。そうしないと、「子」タスクがアクティブになりません。
