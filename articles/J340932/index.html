REST定義



Roy Fieldingは、RESTの定義を提案しました。これは、調整されたアーキテクチャ上の制約のセットです。



クライアントサーバー









まず第一に、彼によると、ソフトウェアシステムはクライアントとサーバーで動作するように構築する必要があります。 これが誰にも驚かないことを願っています。 私たちのほとんどは、ウェブサイト、メールシステム、FTPなどのクライアントサーバーアプリケーションを既に作成していると思います。 今では非常に一般的です。



詳細は説明しません。 サーバーは何らかの方法でクライアント間で分割され、データストレージ、基本機能、およびセキュリティを提供します。これらは一元管理するものです。



顧客の良い点の1つは、それぞれがコンピューターから来ていることです。 したがって、実行する必要がある計算がある場合は、それらを多数の顧客に配布できます。 したがって、顧客はデータ処理、プレゼンテーション、およびユーザーとの対話を提供します。



サーバーは複数のクライアントにサービスを提供します。 そして、これは大きな利点です。 もちろん、システムを正しく設計すれば、1台のサーバーを使用して何千人ものユーザーにサービスを提供できます。







状態を保存しない



2番目の制限は、状態の保存の拒否です。 ASP.NET、PHP、ColdFusion、または従来のASPでは、セッションのアイデアを思いつきます。 セッションCookieがあります。 しかし、この考えはもはや機能しません。 ASP.NETのようなものがあり、1つのWebサーバーを起動すると、少し遅くなり始めました。 その後、別のWebサーバーをインストールしましたが、サーバーにリクエストを送信するたびに、このクライアント専用のスペースがストレージに割り当てられたため、すべてが機能しなくなりました。 このようなもの：







サーバーはその状態を維持しました。 同時に、次の人も入って来ますが、その人にも何らかの病気があります。



サーバーがいっぱいになり、クライアントの1つが離脱するとどうなりますか？ 彼が戻るかどうかわからないので、しばらくの間その状態を維持しなければなりません。 しかし、セッションのタイムアウトはどうあるべきでしょうか？ 5分、10分、24時間？ 判断するのは非常に困難です。 しかし、最大の問題は、さらに多くの機能を提供する別のサーバーを起動し、クライアントの1つが別のサーバーに切り替えようとすると、セッション情報が別の場所に保存されるため、接続が終了することです。



したがって、状態の保存を回避することで、サービスを提供できるクライアントの数とシステムのスケーリング方法に関する柔軟性が高まります。



キャッシング



Royが定義する3番目の制限はキャッシュです。







サーバーに何らかのリソースがあるとします。 クライアントが要求します。 クライアントが同じリソースを再度必要とする場合、再度要求する必要はありません。 彼はすでにそこにあるものを使用できるはずです。



サーバーでタスクを実行すると、リソースが無駄になります。 ビデオまたは写真を処理している可能性があります。 そして、リソースに見合うだけのものを再利用する必要があります。 これが最善です。 戻ってきて同じデータを再度要求した人が同じリソースを取得できるように、処理結果を保存する必要があります。



階層化システム



次の制限は、キャッシュの概念に直接関連しています。 これは、マルチレベルシステムの考え方です。 このアイデアの私のお気に入りの例は、クライアントとサーバー間のキャッシュプロキシです。







この図の顧客は左側にあります。 これらは、スマートフォン（iPhone、Samsung、Android）のさまざまなオプションであると想定します。 プロキシサーバーはチェーンの途中にあります。 大規模な通信会社によって管理されているとしましょう。 サーバーの所有者として、プロキシの右側にあるサーバーの作業に対して料金を支払います。 結局、クライアントからリクエストが来ると、サーバーはいくつかの計算を実行します。 そして、それはお金がかかります-電気、設備の減価償却費など。



通信会社が結果のコピーを保持している場合、誰かが次に情報を要求したときに、プロキシコピーを受け取ります。 リクエストの実行にお金をかける必要はありません。 したがって、プロキシキャッシングのアイデアは信じられないほど優れています。なぜなら、他の人が追加投資なしでソフトウェアを高速化するために支払うことができるからです。



マルチレベルシステムのアイデアの一部として、中間レベルをさらに追加できます。 多層システムでは、サーバーとクライアントはリクエストがどのレベルを通過するかを気にしません。







この例では、クライアントはプロキシサーバーと対話します。 プロキシは、サーバーと対話すると考えていますが、実際には、セキュリティレベル（ファイアウォールなど）に要求を送信します。 次に、彼はサーバーと通信していると考え、実際にロードバランサーにリクエストを送信します。



ロードバランサーはサーバーと対話すると考えていますが、実際には5台のサーバーについて話しています。 新しいものをデプロイし、古いものを閉じることができます。ロードバランサーのおかげで、これは他のレベルに影響しません。



階層型システムの概念により、アプリケーションを計画する際の柔軟性が大幅に向上します。

リソース識別



HTTPおよびRESTが登場する前に、サーバーから情報を取得するには、Gopher、FTP、Telnetなど、数十種類のプロトコルのいずれかを選択する必要がありました。 これらはすべて異なるシステムであり、それぞれがリモート情報およびリモートサービスと対話する独自の方法を暗示しています。



HTTPとRESTはこれを標準化しました。 まず、リソースの識別が標準化されました-URLが表示されました（誰かがURLを思いついたと思うのは面白いです）。 アドレスの一部はプロトコル（HTTP）で定義され、もう一方はリソースへのパス、3番目はポートです。 ネットワーク要求にアクセスできるインターネット上のすべてのものは、これらのURLのいずれかを使用して識別できます。



提出操作



表現操作とは、さまざまな形式のデータを要求できることを意味します。 データが必要な場合は、CSVテーブル、Excelファイル、テキストドキュメント、jpegとして取得できます。 これらはすべて、いくつかの基本的な結果の実際の表現です。



自己記述メッセージ



ネット上で何が起こっているのかを理解できるはずです。 HTTPにはGET、PUT、POST、DELETEリクエストがあります。 それらは説明的なものです：GETは、何かを手に入れたいという意味の英語の単語です。 PUTは何かを置くこと、DELETEは削除することを意味します。 インテリジェントなデバッグ用のツールでネットワークトラフィックを調べて、実際に送信されたリクエストを把握できます。

エンコードのリクエスト



ロイが確認した最後の制限は、オンデマンドで顧客に提供されるコードを作成するというアイデアです。 サーバーがコードを含むデータを送信できる必要があることを意味します。 クライアントはこのコードを抽出し、サーバーから切断して、後で起動できます。



これを行うAPIを見たことがない。 Googleドキュメントオフライン、Gmail、その他多くの1ページアプリケーションなど、多くのWebアプリケーションを見てきました。 後でシステムで起動するために、他の誰かのAPIから実行可能コードを取得するという考えはサポートしていません。 しかし、クライアント上でこのような操作が開始される可能性があることを知ってうれしいです。



電子メールクライアントではない最初のWebブラウザが作成されたときに、私は業界に来ました。 その後、誰かがJavaScriptを実行できると判断し、JavaScriptでメールクライアントを作成してクライアントに送信し、ブラウザで実行できるようにしました。



必須の制限



重要なことは、ロイ・フィールディングが率直に言っていることです：6つの制限があり、そのうちの1つだけがオプションです。







説明した方法でシステムを構築して問題を解決すると、システムはRESTfulになります。 それらを異なる方法で解決すると、システムはRESTfulになりません。 これは、彼女が良いか悪いかを意味するものではありません。 上記の定義は、システムの品質については何も言及していません。 ただし、定義によってRESTful APIを構成するものについて（元のソースに従って）合意することが非常に重要です。



これらの制限が実際の生活、またはコミックヒーローの生活でどのように見えるかを見てみましょう。 RESTに従って、スーパーヒーロー用のソーシャルネットワークコードを記述してみましょう。 楽しいでしょう。

コミック（およびコミック本の映画）の非常に興味深いキャラクターが描かれています。 彼らのためにソーシャルネットワークを作成してみましょう。 ただし、JavaScriptをテストしたくないのと同じように、ユーザーインターフェイスの開発で行き詰まることは望みません。 それでは、APIを作成してみましょう。



エントリーポイントは次のとおりです。 



GET / HTTP/1.1


応答として、次のものを取得します。 



 200 OK Content-Type: application/json { "message": "Welcome to Herobook!" }


here / profilesというエンドポイントを作成してみましょう。これには2つのメソッドがあります。 



 GET /profiles HTTP/1.1


このメソッドは、ソーシャルネットワークに参加したすべてのスーパーヒーローのプロファイルからJavaScript-JSONの配列を返します。 



 200 OK Content -Type: application/json [ { "username": "ironman", "name": "Tony Stark", } ]


現時点ではセキュリティについて心配していませんが、この問題は会話の範囲を超えています。



2番目の方法はPOSTです。 /プロファイルにPOSTを送信することにより、ペイロードをJSONとして提供します。 



 POST /profiles HTTP/1.1 { "username" : "blackwidow" "name": " ", }


答えは、プロファイルとそのア​​ドレスの作成が成功したことに関するメッセージです。 



 201 Created Location: http://api.herobook.local/profiles/blackwidow { "username": "blackwidow", "name": " " }
