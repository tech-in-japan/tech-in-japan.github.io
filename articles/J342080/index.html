VS CodeからLXC / LXDコンテナーでPythonスクリプトを実行/デバッグする

こんにちは、Habr！



この記事では、 VS Codeエディターで数回クリックするだけで、Pythonスクリプトを開発し、隔離された環境で実行するための開発環境のセットアップを共有したいと思います。



隔離された環境として、Ubuntu 17.10 OSを実行するLXCベースのコンテナーの初期構成と作成が検討されます。 なぜDockerではありませんか？ いくつかの議論をすることは、答えるのが難しいです。 部分的に、OS仮想化の概念はプロセス仮想化よりも精神的に主観的に近いですが、ここで説明する原則は、修正を加えて、Dockerコンテナーでコードを実行するのに適しています。







あなたは何を得ますか？



  • コンテナ内でワンクリックでPythonスクリプトを実行する方法と、VS Codeからのリモートデバッグを学ぶ
  • LXDをインストールし、ネットワークを構成し、最初のコンテナーを開始する方法を学ぶ
  • VS Codeソースコードエディターのインストールと構成を確認してください。VSCodeソースコードエディターは、快適にコードを記述してデバッグできる拡張機能のセットです。


ここにないものは何ですか？



  • LXC / LXDの完全なレビューはありません。Linuxコンテナに関する追加情報を自分で見つける必要があります。
  • 開発者のホストで仮想環境を作成するためのpython virtualenvモジュールは、コンテナーの代替とは見なされません。
  • DockerとLXDの精神の比較はありません。経験に基づいて自分で結論を導きます。

バックエンドリポジトリの選択



LXDをインストールする前に、使用するコンテナのバックエンドストレージのタイプを決定する必要があります。 後で変更することはできません。 いくつかのオプションが利用可能です：ZFS、btrfs、LVM、またはホストファイルシステム。 比較のために、公式ドキュメントに比較表があります。 将来、コンテナのCOW （コピーオンライト）スナップショットを簡単に作成できるZFSを選択することをお勧めします。 ZFSユーティリティパッケージをシステムにインストールします。 



$ sudo apt install zfsutils-linux


LXDパッケージをインストールする



問題はありません。パッケージはシステムの標準配信に含まれているため、コンソールでコマンドを実行します。 



 $ sudo apt install lxd


LXDをインストールすると、現在のユーザーがlxdシステムグループに追加されますが、システムで必要な特権を使用して構成を続行するには、ユーザーセッションを再アクティブ化する必要があります 。 ログアウトしてDesktop Menegerのセッションを再入力するか、完全に再起動してください。 idコマンドを使用して、現在のユーザーセッションがlxdグループにあるかどうかを確認できます。 



 $ id uid=1000(user1) gid=1000(user1) groups=1000(user1),4(adm),24(cdrom),27(sudo),30(dip),46(plugdev),118(lpadmin),128(sambashare),129(lxd)


lxdグループが表示されていれば、すべて問題ありません。続行できます。



LXDを初期化する



LXDの初期化は、 lxdグループのメンバーである現在のユーザーに代わってインタラクティブに実行されます。 コマンドを実行し、以下の指示とコメントに従ってください：

LXCネットワーク構成



ここでネットワークを設定せず、コンテナの作成に進むと、コンテナが作成されると、コンテナがどのネットワークにも接続されていないという警告が表示されます。 これはエラーではなく、単にインターフェイスを介したネットワークアクセスがありません。 最初に、必要な数のコンテナを作成し、 lxc network attachコマンドを使用してネットワークを構成し、コンテナをコンテナに接続できます。 次に、構築するネットワークのタイプを決定する必要があります。 LXCネットワークの構築にはいくつかのオプションがありますが、 ブリッジオプション（ホストとコンテナ間の単純なソフトウェアスイッチ）を選択し、それに基づいてネットワークを構築します。



ネットワークブリッジを作成する場合、名前を指定した後すぐにプロパティを設定するか、IPv4およびIPv6サブネットをランダムに選択する自動構成に依存できます。 デフォルトでは、パラメーターはipv4.nat = falseであり、何らかの理由で、コンテナーからの通過トラフィックのパケットは、ホストカーネルでIP転送を明示的に有効にしても、ホストを越えて他のローカルネットワークセグメントにルーティングされません。 どうやらLXDのNATは、ローカルネットワークセグメントからグローバルへのアドレスのトランスレーター以上のものです。このオプションをアクティブにすると、ホストのIP転送もアクティブになるため、明示的に有効にします。 



 $ lxc network create lxdbr0 ipv6.address=none ipv4.address=10.0.3.1/24 ipv4.nat=true Network lxdbr0 created $ lxc network list +--------+----------+---------+-------------+---------+ | NAME | TYPE | MANAGED | DESCRIPTION | USED BY | +--------+----------+---------+-------------+---------+ | ens33 | physical | NO | | 0 | +--------+----------+---------+-------------+---------+ | lxdbr0 | bridge | YES | | 0 | +--------+----------+---------+-------------+---------+


コンテナー構成プロファイルの作成



コンテナの構成は、条件付きでグローバルとローカルの2つのレベルに分けられます。 グローバル構成は、特定のコンテナーまたはコンテナーのグループに接続されている構成プロファイルで指定され、ローカル構成は特定のコンテナーにのみ適用されます。 つまり、プロファイル内のすべての設定がコンテナに継承され、ローカル設定がプロファイルよりも優先されます。 1つのコンテナに複数のプロファイルを添付できます。



LXDプロファイルを作成して、新しく作成されたコンテナがホスト上のlxdbr0スイッチ（ネットワークブリッジ）にリンクされるタイプnicのインターフェースを持つようにします。 



 $ lxc profile copy default network0 $ lxc profile device add network0 eth0 nic parent=lxdbr0 nictype=bridged


コンテナ作成



たとえば、コンテナを作成してpytestと呼ぶと、この名前でホストからアクセスできます。 ubuntuリポジトリからイメージをダウンロードする起動コマンドを実行します：バージョン17.10およびamd64アーキテクチャと、このイメージからコンテナーを作成します。 さらに、オプションは、このコンテナーをnetwork0プロファイルから構成する必要があることを示します。 



 $ lxc launch ubuntu:17.10/amd64 --profile network0 pytest


次に、ネットワークからイメージをダウンロードします。 少し待って、少し後でシステムで使用可能なコンテナとそのステータスを確認できます。 



 $ lxc list +--------+---------+------------------+------+------------+-----------+ | NAME | STATE | IPV4 | IPV6 | TYPE | SNAPSHOTS | +--------+---------+------------------+------+------------+-----------+ | pytest | RUNNING | 10.0.3.13 (eth0) | | PERSISTENT | 0 | +--------+---------+------------------+------+------------+-----------+


コンテナにソフトウェアをインストールする



ここで、必要なソフトウェアをコンテナにインストールします。 システムを更新するコンテナ内でパッケージマネージャを実行します。 



 $ lxc exec pytest -- /bin/bash -c 'apt update && apt upgrade -y'


次に、 pipパッケージマネージャー（Pythonパッケージインストーラー）とptvsd pythonモジュール（Python用Visual Studioリモートデバッグサーバー）をインストールします。 ptvsdモジュールは必ずバージョン3.0.0にインストールする必要があります。最新バージョン（現在の3.2.1.0）では、github.comの問題に示されているエラーを受け取りました。 一時的なソリューションとしてptvsdと対話するVS CodeのPython拡張の作成者は、バージョン3.0.0の使用を推奨しています。 



 $ lxc exec pytest -- /bin/bash -c 'apt install python3-pip -y' $ lxc exec pytest -- /bin/bash -c 'pip3 install ptvsd==3.0.0'


LXDに関しては、セットアップが完了しました。

ディレクトリリンクホスト<->コンテナ



私たちはすでに素晴らしい仕事をしており、最終段階に来ています。 これで、コンテナでスクリプトを実行する準備がほぼすべて整いました。



/ home / user1 / projectsなどに プロジェクトディレクトリを作成します。 将来のPythonプロジェクトをホストします。 このディレクトリをコンテナにリンクします。つまり、コンテナ内でディスクデバイスを作成し、このディレクトリをinternal / mnt / hostfsに関連付けます。 次のコマンドを実行します。 



 $ mkdir ~/projects $ lxc config device add pytest hostfs disk path=/mnt/hostfs source=/home/user1/projects/


/ home / user1 / projectsディレクトリにディレクトリまたはファイルをローカルで作成すると、アドレス/ mnt / hostfsのコンテナにすぐに反映されます。 これを確認するには、コンテナでBASHシェルをインタラクティブに実行し、lsコマンドを発行します。 



 $ lxc exec pytest -- /bin/bash $ ls -l /mnt/hostfs/




コンテナ内のファイルとディレクトリが読み取り専用になり、 nobody：nogroupに等しいUID：GIDを持つことがわかります。 スクリプトを実行するにはこれで十分ですが、コンテナ内のファイルとディレクトリを書き込み可能にする必要がある場合は、 この記事の 「 ユーザー/グループの直接マッピング 」の章をお読みください 。

VSコードでのリモートデバッグ



test.pyファイルの行のコメントを解除して、このファイルを保存します。 



 #### BEGIN. Visual Studio remote debugging server for Python import ptvsd print('Waiting connection on port 3000') ptvsd.enable_attach('my_secret', address=('0.0.0.0', 3000)) ptvsd.wait_for_attach() ptvsd.break_into_debugger() #### END. Visual Studio remote debugging server for Python


リモートデバッグが必要な場合は、これらのコード行を、実行のために起動されるファイルに挿入する必要があります。 コンテナ側のptvsdモジュールはポート3000を開き、外部デバッガーが接続することを想定しています。 必要に応じて、独自の方法でptvsdを構成できます。



[コードの実行]コンテキストメニューを呼び出してコードを実行します。 統合端末では、スクリプトがポート3000での接続を待機しているというメッセージが表示されます。デバッガーに移動して、このプロジェクトの新しい構成を作成します。







launch.jsonファイルを編集用に開き、既にプリセットがあります。 次のコードブロックを貼り付けます。 



  { "name": "Python LXC", "type": "python", "request": "attach", "localRoot": "${workspaceRoot}", "remoteRoot": "/mnt/hostfs/pytest", "host": "pytest", "port": 3000, "secret": "my_secret" }


保存すると、 setting.jsonファイルの横の.vscode / launch.jsonディレクトリに表示されます。 次に、左側の[ Python LXC]ドロップダウンリストを選択します。これは先ほど作成したデバッガー構成です。







[デバッグの開始]をクリックします。 ファイルtest.pyのあるウィンドウに、動作中のデバッガーと、コードがリモートで開始されたことの確認が表示されます。

外部Pythonライブラリの自動補完セットアップ（IntelliSense）



たとえば、プロジェクトでは、Pythonのデフォルトボックスで提供されていないモジュールを使用し、このモジュールをコンテナにのみインストールしました。 ホストにモジュールをインストールせずにエディターでVS Codeを「理解」、オートコンプリート、プロンプト表示するが、コンテナーのモジュールから取得する方法は？

この問題を解決するには、 YOUR_path_to_project / .vscode / settings.jsonでWORKSPACE_SETTINGS設定を開き 、 python.autoComplete.extraPathsパラメーターを追加し、Pythonモジュールを含むコンテナーディレクトリのリストをキーとして指定します。



たとえば、コンテナは/ var / lib / lxd / containers / pytest / rootfs /のホストファイルシステムで利用できます 。 これは、 pytestコンテナのルートファイルシステムです。 Pythonが外部ライブラリをインストールする場所がわかっているので、このパスを簡単に追加して、外部ライブラリへのフルパスを取得できます。 numpyなどのモジュールの場合、フルパスは/var/lib/lxd/containers/pytest/rootfs/usr/local/lib/python3.6/dist-packagesになります。 このように、このパスをpython.autoComplete.extraPaths設定に追加します。







設定を保存して、プロジェクトに戻ります。 numpyモジュールをインポートして、CTRL +スペースキーで自動補完を呼び出してください。 画面のようにできるはずです：







すぐに動作しない場合は、落胆しないでください。VSCodeは結果をキャッシュし、すぐに動作しない場合があります。エディターを閉じて開き、from ... importキーワードの後に​​CTRL + SPACEキーを押してみてください。
