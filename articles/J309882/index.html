AndroidデバイスをC ++プログラムのシンUIとして使用する

私は過去数ヶ月間ゆっくりとやってきたプロジェクトをコミュニティと共有したいと思っています。



まえがき



多くの場合、携帯電話やタブレットからプログラムを管理したい場合がありますが、このために携帯電話で別のアプリケーションを作成することは不適切です（または不可能です）（このプロジェクトでは作業が多すぎます。 ） この状況は私に定期的に発生し、最終的に私はこの問題にきっぱりと対処することにしました。 その結果、システムのアーキテクチャと使用方法がこの記事で説明されます。



目的



Androidデバイスに基づいたプラグインUIをC ++プログラムに実装できるシステムを作成します。 同時に、サードパーティのライブラリに対するカスタムC ++コードの依存関係を最小限に抑え、データ転送プロトコルから抽象化したいと考えました。 システムは、C ++ライブラリとAndroidアプリケーションの2つの部分で構成する必要があります。



システム構成



このシステムには、Androidデバイスがクライアントとして機能するクライアントサーバーアーキテクチャがあり、サーバーはユーザープログラムです。 それらの間の通信は、TCP / IPソケットを使用して実行されます。 通信プロトコルを実装するために、 TAUライブラリが作成されました。



ライブラリが担当する主なタスク：



  • サーバーとクライアント間で送信されるデータの生成と処理
  • クライアントで発生したさまざまなイベント（イベントUIハンドラー）に応答するためのユーザーコードへの制御の転送
  • サーバーとクライアント間の接続の作成と維持
  • クライアントに表示されるUI要素の構成を記述するために必要なデータ構造の生成


ライブラリは、次の名前空間で構成されます。



  • tau :: Communications_Handling-パケットを生成し、クライアントから受信したデータを解析し、ユーザーコードでハンドラーを呼び出します。 クライアントを接続してから切断するまでの間に発生するすべてのことは、この名前空間のクラスコードによって制御されます。



  • tau :: layout_generation-ユーザーインターフェイス要素の場所と動作を記述するJSON構造を作成できる機能が含まれています。 その後、このデータはクライアントに送信され、対応するUIが表示されます。



  • tau :: util-さまざまな補助機能が含まれています。これは、カスタムC ++プロジェクトでライブラリを使用するために必要ではありませんが、多くの場合便利です。 この名前空間のクラスは、サードパーティのライブラリまたは非標準のコンパイラ拡張機能を使用できる唯一のクラスです。 したがって、ここにTCP / IPソケットの操作を担当するクラスがあります-これはプラットフォーム固有のコードです。 現在、boost :: asioおよびC ++ / CLIに基づく2つのネットワーク通信実装があります。 tau :: Communications_Handling以外のすべてのネットワークコールの実装を削除すると、ライブラリユーザーは必要に応じてネットワークパーツ全体を個別に書き込むことができます。



  • tau :: common-ライブラリの他の部分から使用されるクラスを含みます（tau :: layout_generationとtau :: Communications_Handlingの間に依存関係がないように、ここから派生します）


使用します。 例＃1-こんにちは、world



ライブラリを使用するデモを簡単な例から始めます。この例では、画面にウェルカムメッセージが表示されます。 この場合、カスタムコードは次のようになります。



非表示のテキスト 
#include <tau/layout_generation/layout_info.h> #include <tau/util/basic_events_dispatcher.h> #include <tau/util/boost_asio_server.h> class MyEventsDispatcher : public tau::util::BasicEventsDispatcher { public: MyEventsDispatcher( tau::communications_handling::OutgiongPacketsGenerator & outgoingGeneratorToUse): tau::util::BasicEventsDispatcher(outgoingGeneratorToUse) {}; virtual void packetReceived_requestProcessingError( std::string const & layoutID, std::string const & additionalData) { std::cout << "Error received from client:\nLayouID: " << layoutID << "\nError: " << additionalData << "\n"; } virtual void onClientConnected( tau::communications_handling::ClientConnectionInfo const & connectionInfo) { std::cout << "Client connected: remoteAddr: " << connectionInfo.getRemoteAddrDump() << ", localAddr : " << connectionInfo.getLocalAddrDump() << "\n"; } void packetReceived_clientDeviceInfo( tau::communications_handling::ClientDeviceInfo const & info) { using namespace tau::layout_generation; std::cout << "Received client information packet\n"; std::string greetingMessage = "Hello, habrahabr!"; sendPacket_resetLayout(LayoutInfo().pushLayoutPage( LayoutPage(tau::common::LayoutPageID("FIRST_LAYOUT_PAGE"), LabelElement(greetingMessage))).getJson()); } }; int main(int argc, char ** argv) { boost::asio::io_service io_service; short port = 12345; tau::util::SimpleBoostAsioServer<MyEventsDispatcher>::type s(io_service, port); std::cout << "Starting server on port " << port << "...\n"; s.start(); std::cout << "Calling io_service.run()\n"; io_service.run(); return 0; }


クライアントデバイス（ MyEventsDispatcher ）と対話するすべてのユーザーロジックを含むメインクラスは、 tau :: util :: BasicEventsDispatcherから継承する必要があります。 基本クラスの2つのメソッドonClientConnected（）およびpacketReceived_clientDeviceInfo（）をオーバーライドします。 最初はクライアントが接続するときに呼び出されます。 2番目の方法は、サーバーへの接続後にクライアントデバイスに関する情報が来たときに実行されます（接続がクライアントによって送信された後の最初のパケット）。



この場合、最初の方法は簡単です。コンソールに情報メッセージを表示するだけです。 2番目の方法では、サーバーはクライアントにレイアウトを送信します。これは、クライアントに表示するインターフェイスに関するデータです。



ネットワークを介してデータを送信するコードはすべてmain（）にあります。 この場合、 boost :: asioライブラリを使用して通信を実装します。 tau :: util名前空間には対応する抽象化があり、これによりこの例をできるだけコンパクトにします。 ブーストの使用はオプションです-TCP / IPソケットの実装はライブラリと一緒に非常に簡単に使用できます。



編集



例として、コンパイルにはg ++を使用します。 この場合、コマンドは次のようになります。 



 g++ -lboost_system -pthread -lboost_thread -D TAU_HEADERONLY -D TAU_CPP_03_COMPATIBILITY -I $LIBRARY_LOCATION main.cpp -o demo


ご覧のとおり、いくつかの追加パラメーターがコンパイラーに渡されます。



  • ライブラリソースへのパスを含める（オプション-I $ LIBRARY_LOCATION ）
  • ブーストに必要な追加ライブラリ:: asio（オプション-lboost_system -pthread -lboost_thread ）
  • プロジェクトでライブラリをコンパイルする方法を示す追加マクロの宣言（ -D TAU_HEADERONLY -D TAU_CPP_03_COMPATIBILITY ）


このオプションセットは最も一般的なビルドオプションであり、最小限の労力でライブラリをプロジェクトに含めることができます。



必要に応じてそれらをすべて取り除くことができます。 プロジェクト内でライブラリを使用する場合、 -I $ LIBRARY_LOCATIONおよび-D TAU_HEADERONLYを指定する必要はありません。 C ++ 11と互換性のあるコンパイラの場合、 -D TAU_CPP_03_COMPATIBILITYオプションは不要です。 boostへの依存:: asioにはネットワーク部分のみがあり、依存関係なしで非常に簡単に書き換えることができます。



コンパイルおよび起動後、サーバーはポート12345でリッスンを開始します。



電話でクライアントを起動し、接続を作成して接続し、メッセージを表示します。 外観は次のとおりです（PuTTYを介してリモートコンピューターでサーバーを起動し、クライアントをエミュレーターで起動しました）。

前の例からのすべての変更は、 MyEventsDispatcherクラスで行われました。 次のクライアントイベントハンドラメソッドが追加されました。



  • packetReceived_buttonClickボタンイベントハンドラー 。 ボタンIDは、パラメーターとしてメソッドに渡されます。
  • クライアントからサーバーに変数値を転送するパケットのハンドラー： packetReceived_boolValueUpdate、packetReceived_intValueUpdate、packetReceived_floatPointValueUpdate、packetReceived_textValueUpdate
  • packetReceived_layoutPageSwitched要素を持つ表示ページの変更イベントのハンドラー


さらに、接続時にクライアントに送信されるレイアウトもそれに応じて変更されました。



デモがあるので、ハンドラー内のコードは可能な限りシンプルです。イベントに関する情報をコンソールにダンプし、クライアントにさまざまなコマンドを送信します。



すべてのコマンドはpacketReceived_buttonClick（）ボタンハンドラーからクライアントに送信されます（もちろん、これはそこで行う必要はありませんが、より簡単で視覚的です）。



各コマンドは、サーバーからクライアントに送信されるパケットに対応しています。 これらのパッケージの形成と送信は、 BasicEventsDispatcherで定義された特別なメソッドが呼び出されたときに発生します。



  • sendPacket_resetLayout（） -レイアウト全体の置換
  • sendPacket_requestValue（） -入力の1つで変数値を要求する
  • sendPacket_updateBooleanValue（）、sendPacket_updateIntValue（）、sendPacket_updateFloatPointValue、sendPacket_updateTextValue（） -入力の変数の値を変更する
  • sendPacket_changeElementNote（） -読み取り専用テキスト（ボタン、チェックボックス、ラベルのテキスト）を変更します
  • sendPacket_changeShownLayoutPage（） -要素を持つ別のページに切り替えます
  • sendPacket_changeElementEnabledState（） -要素のアクティブな状態を切り替えます（非アクティブな要素は表示されますが、相互作用することはできません）


この例の仕組みは次のとおりです。



UI要素を使用した作業のデモンストレーション






ご覧のとおり、クライアントデバイス上の各アクションに対して、対応するコードがサーバー上で実行されます。 ユーザー入力要素の値が変更されると、サーバーはこの要素の変数の新しい値に関する通知を受け取ります。 ボタンクリックハンドラでは、さまざまなパケットがサーバーからクライアントに送信されます（パケットのタイプは、どのボタンが押されたかによって異なります）。 この例では、ページ切り替えの仕組みも示しています。 ページングレイアウトを使用すると、機能に応じて要素をグループ化できます。 クライアントの画面には常に1つのページのみが表示されるため、インターフェースの負荷が軽減されます。
