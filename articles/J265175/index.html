Objective-Cのプロパティ属性。 初心者向けの手順





プロパティを宣言するときに使用される属性の意味が説明されている場所はたくさんありますが、原則として、そのようなソースはすべて、属性の1つのみの使用を説明するか、他のドキュメントへのリンクの束を含んでおり、その後、引数の追跡を失います。 もちろん、あなたは文学の山を読む必要があるすべての微妙さを学ぶために努力する必要があります。 しかし、初心者にとっては、基本を理解するだけで十分です。 以下では、プロパティの宣言に使用される主な属性、その値、および1つまたは別の属性値を使用する価値がある主なケースについて、できるだけ簡単に説明しようとしました。



簡単な紹介



setterメソッドとgetterメソッドを使用してオブジェクト指向言語のオブジェクトのデータにアクセスすることをお勧めします（これらはミューテーターとアクセサーでもあります）。前者はプロパティ値を設定し、後者はインスタンス変数に直接アクセスする代わりに、現時点でプロパティ値を返します。 メソッドの束を宣言する必要性を取り除くために、必要な書き込みの量を減らすプロパティが発明されましたが、本質は変更しませんでした：プロパティにアクセスするとき、セッターまたはゲッターも自動的に生成できる操作に応じて呼び出されます。 基本的に、プロパティの宣言時に指定された属性によって、データアクセスメソッドの生成方法が決まります。



直接、指導



まず、プロパティが持つすべての属性をグループに分けます。



  1. アクセシビリティ属性（読み取り専用/読み取り/書き込み）、
  2. 所有権属性（保持/強力/コピー/割り当て/ unsafe_unretained /弱）、
  3. 原子性属性（原子/非原子）。
  4. nullability属性（null_unspecified / null_resettable / nullable / nonnull）-xcode 6.3で登場


セッターとゲッターの名前を設定できる属性は考慮しません。使用しているコードスタイルが提供するものを除いて、そのようなルールはありません。 明示的または暗黙的に、すべてのタイプの属性が各プロパティに指定されます。



可用性属性


  • readwrite-プロパティが読み取りと書き込みに使用できることを示します。つまり、setterとgetterの両方が生成されます。 特に指定しない限り、この値はデフォルトですべてのプロパティに設定されます。


  • readonly-プロパティが読み取り専用であることを示します。 この値は、オブジェクトのタスク実行中にプロパティを「外側」に変更することが望ましくない場合、またはプロパティ値が変数に保存されず、他のプロパティの値に基づいて生成される場合に使用する必要があります。 たとえば、UserオブジェクトにはfirstNameプロパティとlastNameプロパティがあり、便宜上、fullNameプロパティの読み取り専用が設定され、その値は最初の2つのプロパティの値に基づいて生成されます。


プロパティを「外部」に変更するのが望ましくない場合、通常、クラスインターフェイスで読み取り専用として宣言され、クラス拡張でreadwriteとして再定義されます。そのため、クラス内では、変数ではなくセッターを介して値を変更することもできます。



所有権の属性


これは、主に手動と自動のメモリ管理が共存しているため、最も広範なタイプの属性です。

ARCが有効な場合、プロパティなどの変数には所有権属性があります。この場合のみ、値のセットがプロパティよりも小さくなります。__strong / __ weak / __ unsafe_unretained。これは、ARCの対象となるデータタイプ（タイプの機能単純なチートシートとなることを意図したものを複雑にしないために、ARCの対象ではないデータは考慮しません。 したがって、プロパティのこの属性の値を説明するとき、ARCが有効になっているときに対応するインスタンス変数が所有権属性の値を持っていることも示します（変数が自動的に作成される場合、この属性の目的の値ですぐに作成されます。変数を自分で定義する場合は、手動で設定する必要があります）正しい所有者属性値）。



  • retain（対応する変数は__strong属性を持つ必要があります）-この値は、生成されたセッターで割り当てられたオブジェクトの参照カウントが増加し、以前に参照されたオブジェクトの参照カウンターが減少することを示します。 この値は、他の値が適切でないすべての場合に、すべてのObjective-CクラスでARCが無効になっている場合に適用されます。 この値は、ARCがまだ使用されていない時代から保持されており、ARCはその使用を禁止していませんが、自動リンクカウントの代わりに強力を使用することをお勧めします。 


//        //       ARC -(void)setFoo(Foo *)foo { if (_foo != foo) { Foo *oldValue = _foo; //            ivar _foo = [foo retain]; //    ,      [oldValue release]; } }


  • strong（対応する変数は__strong属性を持つ必要があります）-この値はretainに似ていますが、自動リンクカウントが有効な場合にのみ適用されます。 ARCを使用する場合、この値はデフォルトで使用されます。 弱くコピーするのに適さないすべての場合に強いを使用し、すべてがうまくいきます。


  • コピー（対応する変数は__strong属性を持つ必要があります）-生成されたセッターの所有権属性のこの値により、対応するインスタンス変数には、割り当てられたオブジェクトに送信されたコピーメッセージによって返された値が割り当てられます。 所有権属性のこの値を使用すると、オブジェクトのクラスにいくつかの制限が課せられます。


  1. クラスはNSCopyingプロトコルをサポートする必要があります。
  2. クラスは可変であってはなりません。 NSString-NSMutableStringなど、一部のクラスには可変サブクラスがあります。 プロパティが「可変」クラスのインスタンスである場合、copyメソッドはその「非可変」相対のインスタンスを返すため、コピーを使用すると望ましくない結果が生じます。 たとえば、NSMutableStringのインスタンスでcopyを呼び出すと、NSStringのインスタンスが返されます。


例： 



 @property (copy, nonatomic) NSMutableString *foo; ... //      - (void)setFoo:(NSMutableString)foo { _foo = [foo copy]; // copy  NSMutableString    NSString,       ,      ,   ,  ,       }


2番目の制限は、コピー値を使用する主な理由を示しています。「可変」サブクラスを持つクラスのインスタンスであるすべてのパブリックプロパティは、この所有権属性値で最適に作成されます。 ミュータブルでないクラスの場合、copyメソッドはretainとして機能します。コピーは行われず、余分なメモリと時間は消費されませんが、プロパティは可変サブクラスのインスタンスの指定から保護されます。 たとえば、NSMutableArrayクラスのオブジェクトをNSArray型のプロパティに設定することは不可能であるため、セッターをバイパスしてプロパティを「外部」に変更することはできません。



例： 



 @interface Foo : NSObject … @property (copy, nonatomic) NSArray *bar; @property (strong, nonatomic) NSArray *barNotProtected; … @end … NSMutableArray *mutableArray = [@[@1, @2, @3] mutableCopy]; Foo *foo = [Foo new]; foo.bar = mutableArray; // bar       foo.barNotProtected = mutableArray; //  barNotProtected       [mutableArray removeObjectAtIndex:0]; // foo.barNotProtected     (@[@2, @3]),  ,    foo.bar    (@[@1, @2, @3]).


  • weak（対応する変数は__weak属性を持つ必要があります）-この値はassignおよびunsafe_unretainedに似ています。 違いは、特別なストリートマジックにより、所有権属性のこの値を持つ変数は、変数が指すオブジェクトが破棄されるときに値をnilに変更できることです。これは、アプリケーションの安定性に影響します（ご存じのように、nilはメッセージに応答するため、これは既に削除されたオブジェクトにアクセスするときにEXC_BAD_ACCESSがないことを意味します）。 所有権属性のこの値は、ARCを有効にしてオブジェクトデリゲートなどへのポインタを格納するプロパティの保持サイクルを除外するときに使用する必要があります。 これは、ARCが無効になっているとき（およびバージョン5.0より前のiOSでARCが有効になっているとき）にサポートされない所有権属性の唯一の値です。


  • unsafe_unretained（対応する変数は__unsafe_unretained属性を使用する必要があります）-このタイプの所有権を持つプロパティは、割り当てられたオブジェクトのアドレスを保存するだけです。 このプロパティへのアクセス方法は、オブジェクトの参照カウントに影響しません。 削除することができ、このプロパティにアクセスするとクラッシュします（したがって安全ではありません）。 ARCが登場したとき、この値はweakの代わりに使用されましたが、iOS 4.3をサポートする必要がありました。 現在、その使用は作業の速度によってのみ正当化されます（もちろん、現在のパフォーマンスでは肉眼ではこれに気付かないでしょうが、弱い特性の魔法にはかなりの時間が必要であるという証拠があります）。したがって、特に最初は使用しないでください。


  • assign（対応する変数は__unsafe_unretained属性を使用する必要がありますが、強い属性または弱い属性を使用する方が適切なARCタイプのみが所有権属性を持っているため、この値は不要です）-単にアドレスを割り当てます。 ARCがない場合、所有権属性のデフォルト値。 ARCの対象とならないタイプのプロパティに適用する必要があります（これらには、プリミティブタイプおよびいわゆる非オブジェクトタイプ（CFTypeRefなど）が含まれます）。 ARCがない場合、オブジェクトデリゲートなどへのポインタを保持するプロパティの保持サイクルを除外するために、weakの代わりに使用されます。


原子属性


  • atomicは、この属性のデフォルト値です。 アクセサとミューテータは、異なるスレッドから同時にアクセスされたときに同時に実行されないように生成されることを意味します（つまり、最初の1つのスレッドがジョブを実行します-値を取得または取得し、その別のスレッドの後にのみ同じことができるようになります）。 実装の特性により、このような原子性属性値を持つプロパティは、アクセスメソッドの1つのみをオーバーライドできません（それらを再定義した場合、両方を再定義し、異なるスレッドでの同時実行に対する保護と混同されます）。 プロパティの原子性とオブジェクトのスレッドセーフを混同しないでください。 たとえば、あるスレッドでオブジェクトのプロパティから名前と姓の値を取得し、別のスレッドで同じ値を変更すると、名前の値が古くなり、姓がすでに変更されている可能性があります。 したがって、多くのスレッドから同時にアクセスできるオブジェクトのプロパティに使用する必要があり、無効な値を受信しないようにする必要がありますが、必要に応じて、完全なスレッドの安全性を確保してください。これだけでは十分ではありません。 とりわけ、そのようなプロパティのアクセス方法は、原子の規模よりも些細な非原子よりも遅く動作することを覚えておく価値がありますが、それでも、ペニーはルーブルを節約するので、必要でない場合は、非原子を使用する方が良いです。


  • 非アトミック-アトミックの反対-このアトミック属性値を持つプロパティの場合、アクセスメソッドは異なるスレッドでの同時実行に対する保護の負担を受けないため、高速です。 ほとんどのオブジェクトは1つのスレッドでのみ使用されるため、この値はほとんどのプロパティに適しています。追加の機能で「重み付け」する意味はありません。 一般に、なぜアトミックである必要があるかを説明できないすべてのプロパティについて、非アトミックを使用すると、高速で簡単でスマートになり、ある種の休日になります。


ヌル属性


この属性は、生成されたアクセス方法には影響しません。 特定のプロパティがnilまたはNULLになる可能性があるかどうかを示すことを目的としています。 Xcodeは、クラスでSwiftコードを操作するときにこの情報を使用します。 さらに、この値は、コードが宣言された動作に反する何かをした場合に警告を表示するために使用されます。 たとえば、nullとして宣言されているプロパティにnilを設定しようとしている場合。 そして最も重要なことは、この情報はあなたのコードを読む人にとって有用であることです。 この属性の使用には、いくつかの制限があります。

  1. プリミティブ型には使用できません（いずれの場合もnil値とNULL値を受け入れないため、必要ありません）
  2. マルチレベルポインターには使用できません（例：id *またはNSError **）


  • null_unspecified-デフォルトで使用され、プロパティをnil / NULLにできるかどうかについては何も言いません。 この属性が現れる前は、これがまさにすべてのプロパティを完全に認識した方法です。 これはヘッダーの内容の点で悪いため、この値を使用することはお勧めしません。
  • null_resettable-この値は、この値を設定すると、実際にはプロパティにデフォルトが割り当てられるため、そのようなプロパティのゲッターがnil / NULLを返さないことを示します。 また、生成されたアクセス方法はこの属性の値に依存しないため、nil / NULLを受け取ったときにivarのデフォルト値を維持するようにセッターを再定義するか、デフォルト値を返すようにgetterを再定義する必要があります対応するivar == nil / NULLの場合。 したがって、プロパティにデフォルト値がある場合は、null_resettableとして宣言します。
  • nonnull-この値は、この属性でマークされたプロパティがnil / NULLではないことを示します。 実際、たとえば、このプロパティの値をnilから取得しようとしても、Xcodeがこれにあまり従わないため、nilを取得できます。 しかし、これはすでにあなたの間違いであり、Xcodeは警告であなたの最善の範囲でそれを指し示します。 このプロパティの値が決してnil / NULLにならないことが確実で、IDEでこれを追跡できるようにする場合は、使用する価値があります。
  • nullable-この値は、プロパティがnil / NULLであることを示します。 null_unspecifiedのように、何もする必要はありませんが、それでも、確実性が高いため、これら2つの間でnullableを使用する方が正しいです。 したがって、null_resettableもnonnullも適切でない場合は、nullableを使用してください。


便宜上、特定のコードブロックのデフォルト値をnull_unspecifiedからnonnullに変更できます。 これを行うには、そのようなブロックの前にNS_ASSUME_NONNULL_BEGINを置き、その後にNS_ASSUME_NONNULL_ENDを置きます。 

 //    @property (copy, nonatomic, nonnull) NSString *foo; @property (copy, nonatomic, nonnull) NSString *str; @property (strong, nonatomic, nullable) NSNumber *bar; @property (copy, nonatomic, null_resettable) NSString *baz; //   NS_ASSUME_NONNULL_BEGIN @property (copy, nonatomic) NSString *foo; @property (copy, nonatomic) NSString *str; @property (strong, nonatomic, nullable) NSNumber *bar; @property (copy, nonatomic, null_resettable) NSString *baz; NS_ASSUME_NONNULL_END
