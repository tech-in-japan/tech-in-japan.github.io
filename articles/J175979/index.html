Fantomを選ぶ理由

Fantomは、フランク兄弟（ Brian Frank、Andy Frank ）によって開発された、オブジェクト指向の静的に型付けされた汎用言語です。 Fantomの主要な機能の1つは、実行される特定の環境から抽象化された強力な標準ライブラリです。 現在、Fantomで記述されたコードは、Javaランタイム環境（JRE）、. NET共通言語ランタイム（CLR）で実行するか、JavaScriptコードにコンパイルできます。 



class HelloWorld { static Void main() { echo("Hello, World!") } }


移植性



Fantomを作成した主な理由は、2つのJava VMおよび.NET CLRプラットフォームで実行できるソフトウェアを書くことでした。 現実には、ほとんどの企業がこれらのプラットフォームのいずれかでソフトウェアを開発しています。 PythonやRubyなどの動的言語でさえ、これらの仮想マシンのいずれかで実行されます。 Fantomは、ある仮想マシンから別の仮想マシンへの移植性の問題を解決するために作成されました。 Fantomのソースコードはfcode（JavaバイトコードまたはILに簡単に変換できるバイトコード）にコンパイルされます。 ブロードキャストは実行時に行われます。これにより、Fantomモジュールを個別のファイルとして展開し、任意のVMで実行できます。



移植性とは、単なるJavaや.NET以上のものを意味します。 前述のように、Fantomはブラウザーで動作するようにJavaScriptでコンパイルできます。 同時に、Fantomはそこで停止するつもりはありません。次の目標は、Objective-C for iPhone、 LLVM 、 Parrotです。



エレガントなAPI



彼らは好みについて議論しませんが（美しさは見る人の目にあります）、Fantomの作成者は美しく便利なAPIに夢中です。 Fantomの基本原則の1つ。 Javaと.NETには、小さな独立した抽象化されたユニット（クラス）への機能分割を最大化するという共通の傾向があります。 Fantomには逆の哲学があります-彼らは、少数ながら強力なユニットでできると信じています。







良い例はjava.ioパッケージで、これには60以上のクラスとインターフェースが含まれています; Fantomでは、必要なものはすべてFile 、 Buf 、 InStreamおよびOutStreamにあります。 そして、これはその使用方法です： 



 Void textIO() { f := File(`text-io.txt`) // write text file (overwrites existing) f.out.printLine("hello").close // append to existing text file f.out(true).printLine("world").close // read text file as big string echo(f.readAllStr) // read text file into list of lines echo(f.readAllLines) // read text file, line by line f.eachLine |line| { echo(line) } }


タイピング



全世界が静的型と動的型のサポーターに分かれています。 Fantomの作成者は、双方が互いに非常に批判的であると信じており、両者の中間を選択します。これは、タイピングシステムに対する中程度のアプローチです。



静的型付けの側面から、Fantomは型のフィールドとメソッドシグネチャの説明を必要とします。 コンポーネント間の通信形式を修正することをお勧めします。 たとえば、文字列と数値で機能するメソッドを作成する場合、コードで直接修正する必要があります。 メソッドやフィールドの署名の静的な型付けとは異なり、コードではしばしば干渉するだけで、不必要なコードを書くことを余儀なくされます。 Fantomの型推論はこの問題を回避します。 たとえば、Javaでの面倒な辞書の作成： 



 Map<Integer, String> map = new HashMap<Integer, String>(); map.put(1, "one"); map.put(2, "two");


Fantomの1行に相当： 
 map := [1: "one", 2: "two"]


ただし、動的型付けが本当に必要な場合があるため、Fantomの主要な機能の1つは、静的型または動的型付けを使用してメソッドを呼び出す機能です。 「。」演算子を使用してメソッドを呼び出すと、コンパイラによって呼び出しがチェックされ、効率的なマシンコードにコンパイルされます。



または、「->」演算子を使用して動的呼び出しを指定できます。 実際、「->」はObj.trapの呼び出しにリダイレクトされます。 デフォルトでは、トラップは「。」のように機能しますが、実行時のみです。 動的設計を定義することにより、この動作を変更できます。 



 if (a is Str) { return a->toInt } obj->foo // obj.trap("foo", [,]) obj->foo(2, 3) // obj.trap("foo", [2, 3]) obj->foo = 7 // obj.trap("foo", [7])


ジェネリック



興味深いことに、Fantomはコードの型付けを少なくしようとしていますが、JavaとC＃は型付けの強化に向かっていますが、ジェネリックはこの傾向を示しています。 完全にパラメーター化されたシステムは、システムの複雑さに直接関係するため、Fantomは現在 、利益と複雑さのバランスを模索しています。



Fantomでは現在、ジェネリックのサポートが制限されています-ユーザーは自分のジェネリックを使用できません。 ただし、3つの組み込みクラスはList、Map、およびFuncです。 たとえば、Fantomの整数のリストは、Int []として宣言されます。 Fantomの作成者は、彼らが中間にいると信じています。ジェネリックはありますが、システムを複雑にすることはありません。



不純物



モデルをサブジェクト領域からコードに比較する問題は、ソフトウェア開発で最も頻繁に解決される問題の1つです。 通常、オブジェクト指向プログラミングでは、このフレーズはモデリングクラスとインターフェイスを意味します。 JavaとC＃は同じアプローチを使用します。クラスは単一の継承をサポートし、インターフェイスは複数の継承をサポートしますが、実装の継承はサポートしません。



JavaまたはC＃を使用したことがある人なら誰でも、クラスまたはインターフェースを作成することを選択することが非常に重要であることを知っています。 クラスを選択する場合、実装を継承する唯一のチャンスを使用しているためです。 大規模で複雑なモデルを使用している場合、インターフェースは追加の負担になります。 たとえば、相続人が異なり、同じインターフェイスを等しく実装する2つのオブジェクトがある場合、機能を複製する必要があります。 複製に加えて、すべての実装に影響するインターフェースのバージョンを変更する問題があります。



JavaとC＃がクラス/インターフェースモデルを使用する理由はたくさんあります。 多重継承は多くの扉を開きますが、これは複雑さの増大とむしろ不快なニュアンスのために起こります。 Fantomは再びmixinsと呼ばれる中央を占めます。 不純物は、実装が保存できるインターフェイスです。 多重継承エラーを回避するために、フィールドを保存する状態などの一部の関数は、不純物が制限されています。 不純物の例： 



 mixin Audio { abstract Int volume Void incrementVolume() { volume += 1 } Void decrementVolume() { volume -= 1 } } class Television : Audio { override Int volume := 0 }


興味があれば、Javaの同等物をここで見ることができます 。



モジュール性



モジュール性は、現代のプログラミング言語が必要とするソフトウェアの重要な側面です。

残念なことに、Javaの過去10年間、私たちはclasspathで本当の地獄を経験してきました。 classpathの問題に加えて、javaは間違った決定を下し、サイズが44MbのJ2SEモノリスに切り替えました。これにより、アプリケーションが大幅に遅くなりました。

.NETはこの問題を非常に深刻に受け止めていたため、バージョンメカニズム、GAC、およびその他のツールのおかげで、Javaの問題の一部が解決されました。 しかし、彼らはzipモジュールのシンプルさを失い、まだ多くの異なるスペアパーツを含むDLLにモジュールをパックし始めたため、モジュールでの作業が困難になりました。



Fantomでは、すべてがポッドと呼ばれるモジュール上に構築されます。 javaの場合と同様に、underは簡単に確認できるzipファイルです。 メタデータは特別なファイル/meta.propsに保存されます。これは、pod.name、pod.version、pod.dependsなどのキーと値のエントリです。 ポッドの依存関係はそのメタデータに保存され、明示的で理解可能な方法で記述されます。



Javaの名前空間でコードを整理するには、パッケージが使用され、jarファイルはモジュールと見なされます。 これらの概念の不一致は大きな問題を引き起こします。 javaクラスの名前はありますが、このクラスがどのjarファイルに存在し、どこからロードするかはわかりません。



Fantomは、「pod :: type.slot」という名前の3つの階層レベル、つまり 最初のレベルでは、常にポッドの名前、次にタイプ名（javaのClassのアナログ）、次にスロットの名前（メソッドまたはフィールド）。 名前空間でのこの一貫した動作により、大規模システムのアセンブリプロセスを簡単に制御できます。

並行性



現在、ほとんどの言語には、スレッド間で1つの共通モデルがあります。 これは、開発者自身がメモリロックを処理する必要があることを意味します。 不正なブロックは、デッドロック、競合状態などの不快なエラーにつながる可能性があります。 これはすべて、並行性管理にとってかなり低いレベルです。



Fantomは、次の手法を使用して並行性をサポートします。

  1. 不変オブジェクト （ストリーミングセキュリティ） 



     const class Point { new make(Int x, Int y) { this.x = x; this.y = y } const Int x const Int y } p := Point(0, 0) // ok px = 10 // throws ConstErr vowels := ['a','e','i','o','u'].toImmutable


  2. 静的フィールドは不変オブジェクトのみを保存できるため、異なるスレッドは共有された可変データにアクセスできません。
  3. スレッド間の通信用のメッセージモデル （アクター）（Erlangスタイル） 



      echo("\n--- echoActor ---") // this actor just echos messages sent to it a := Actor(ActorPool()) |msg| { echo(msg); return msg } // send some messages and have them printed to console f1 := a.send("message 1") f2 := a.send("message 2") f3 := a.send("message 3") // now block for the result of each message echo("Result 1 = " + f1.get) // message 1 echo("Result 2 = " + f2.get) // message 2 echo("Result 3 = " + f3.get) // message 3
