ビルドツールとユーティリティを備えたDjangoプロジェクトテンプレート

Djang上にゼロからサイトを作成する必要がある場合があります。 これらはコンセプトテストとシンプルなページです（プロジェクトでさまざまな強力なツールを使用する場合、それらを使用して名刺サイトを作成し、手作業から自分を救うのはなぜですか？）。



だから、私は「より少ない落書き！」というモットーの下でプロジェクトを公開します。 急いで、デバッグしてください！」Djangoでの高速開発のセット。



これらは2つのパターンです。

  • プロジェクトに必要なすべてのパッケージをプルアップして収集するビルドアウト構成テンプレート
  • プロジェクトテンプレート、つまり、必要なすべてのツールを含む設定、および静的セット（HamlおよびSassの既製ファイル）






設置





Djangoでプロジェクトをアップロードしてプロジェクトを作成するには、次のコマンドを実行するだけです： 



  $ hg clone ssh：//hg@bitbucket.org/siberiano/fastdev-django my_new_project
 $ cd my_new_project
 $ rm .hg / hgrc
 $ python bootstrap.py --distribute
 $ビルドアウトする




（ BitBucketのプロジェクトページ。 ）



プロジェクトの構成：



メイクファイル



コマンドラインスクリプトのセットの代わりに、プロジェクトでMakefileを作成し、コマンドの依存関係をコンパイルしました。たとえば、 実行はbuildoutおよびbootstrapの後にのみ完了できます。 作成する次のルーチン手順を追加しました。



  • run-サーバーを0.0.0.0:8000で起動します
  • shell_plus-拡張djangoコマンドラインを起動します
  • make_messages-htmlおよびhamlからロケールファイルをコンパイルします
  • コンパイル-翻訳をコンパイルします




ビルドアウト



zc.buildoutは、Zopeプロジェクトで開発され、プロジェクトフォルダーまたはユーザーフォルダーで〜/ .buildoutを収集し、プロジェクト環境を追加パッケージ（サイトパッケージ）から分離し、いくつかのレシピを実行する（たとえば、スクリプトをインストールし、別のリンクを追加します）。



Buildoutには、柔軟な設定システムと、さまざまなスクリプトの既製のコレクション、一般に、作業プロジェクトを組み立てるために使用できる開発済みのインフラストラクチャがあります。



追加のパッケージとpyflakes静的コードアナライザーを使用してDjangaをインストールする最も単純な構成の例を次に示します。 



  [ビルドアウト]
パーツ=
    ジャンゴ
    パイフレーク
 
 [ジャンゴ]
レシピ=ジャンゴレシピ
卵= 
     ipython
     django_extensions

 [パイフレーク]
レシピ= zc.recipe.egg
スクリプト= pyflakes
卵=パイフレーク
エントリポイント= pyflakes = pkg_resources：run_script
引数= 'pyflakes'、 'pyflakes'




Buildoutは、特別なサイトからdjangorecipeとzc.recipe.eggレシピをダウンロードして実行します。



私たちにとって最も興味深いのはdjangorecipeです。 このレシピは、manage.pyスクリプトをラップします。このスクリプトは、システムにインストールされている追加パッケージから隔離し、ビルドアウト構成で指定されたエッグのみを接続します。



この構成は、いくつかの頭痛から私たちを救います：



  • どこかで必要な卵を示すことができ、それらは例えばホスティングなどの新しいシステムに自動的にインストールされ、1つの（！）コマンドで
  • すべてのシステムが同じバージョンの卵を産むことを確信することができます（おそらくもっと古いが、動作することが保証されています）




プロジェクトのインストールは、2つのチームで行われます。 



  $ python bootstrap.py --distribute
 $ bin / buildout




プロジェクトの準備ができており、開発者向けのDjangoサーバーをすでに起動できます。 manage.py [コマンド]の代わりに 



  $ bin / django runserver 0.0.0.0:8000




電池



しかし、便利なプロジェクトアセンブリシステムは、開発の加速に必要なものとはほど遠いものです。 急いで書かれたコードを実行しないでください。 静的アナライザーチェックを実行します。 多くのエラーがすぐにキャッチされます。 



  $ bin / pyflakes my_new_script.py
未定義変数：test_string、2行目
定義前に使用される変数：another_var、5行目




（宣言されていない2つの変数へのアクセス。アナライザーがなければ、サイトを2回起動してバグを確認します。）



アナライザーの後、プログラマーにはデバッガーが必要です。 Pythonにはたくさんありますが、チームでチェックしたものの中で最も便利なのはPuDBです 。これは、テキストモードのMidnight Commanderのように機能し、80年代の古き良きQBasicまたはBorlandメディアに似たインタラクティブな環境です。







開始するには、デバッグが必要な場所にコマンドを挿入する必要があります。 



  pudbをインポートします。  pudb.set_trace（）




デバッガー内は非常に構成可能（Ctrl + P）で、そこからIPythonのデバッグコンソールに移動することもできます。 これは強力なPythonコンソールであり、可能なすべてのオートコンプリートと、デバッグモジュールなどのアクションを記録して繰り返すマクロを備えています。



もう1つの非常に便利なツールは、デバッグツールバー、またはDjangoデバッグツールバーです。 発行されたページに直接リンクすることにより、すべてのページ構築パラメーター、すべてのコンテキストのテンプレートと変数のリストを含むパネルがあります。 ページが「落ちない」が、何か間違ったものを表示する場合、コードを編集する必要はありません。コンテキストを見るだけです。



ページがクラッシュした場合はどうなりますか？ 通常、不明瞭なエラーが発生した場合、次のことを行う必要があります。



  • エディターでプログラムファイルを開き、pudb行を挿入します
  • サーバーが再起動するのを待ちます（分厚いプロジェクトでは時間がかかる場合があります）
  • ブラウザを開き、更新をクリックします
  • コンソールに移動し、借方記入を開始します
  • エラーを見つけたら、ファイルに戻ってデバッグ行を削除する必要があります




5つのアクションの代わりにWerkzeugがプロジェクトにインストールされている場合、このアプリケーションは死後のデバッグコンソールをブラウザーウィンドウに直接表示するため、1つのことを行う必要があります。 さらに、呼び出しスタックのどのレベルでも！ 多くの問題では、これで十分です。



コードが正常に機能する場合は、読みやすくするためにコームします。 



  $ bin / pep8 my_new_script.py

閉じられたタグが原因でレイアウトが何回ばらばらになりましたか？ HAML言語には終了タグはありません。 タグは、タブが停止する場所で閉じます。 このようなドキュメントは、ネストレベルを編集および追加しやすく、読みやすくなります-行数が1.5倍、文字数が1/4少なくなります。 文字の損失は属性のJSONのみですが、これは大きな損失ではありません。
