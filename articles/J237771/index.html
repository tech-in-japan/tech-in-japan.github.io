WebKitの内部について少し

WebKitが何らかの形で動作する方法、リソースのロードプロセスがどのように行われるか、そしてこれらすべてで実際に何ができるかについて、私はタスクを自問し、最新のすべてのブラウザーの基礎を知りました。 原則として、問題に関する文書で十分です。

この記事の目的は、上から見たシステムの一般的な見解ではなく、システムで発生するプロセスの1つを正確かつ詳細に分析することです。 私の意見では、これは抽象的な視点よりもシステム全体のより良い考えを与えることがあります。 あるいは、それは単なる小さなレンガであり、開発者は彼のシステムのアイデアの散在するモザイクから情報をコンパイルする必要があります。



構造


  • WebKit1-外部API、各ポート独自の事実上の;
  • WebKit2-その新しいバージョン、私はそれで動作しませんでした。
  • JavascriptCore-JavaScriptエンジン。
  • WTF-Webテンプレートフレームワーク。
  • ThirdParty-一連のサードパーティライブラリ、特にleveldb。
  • WebCoreはWebKitの基本機能です。 彼とテキストのほとんどについて。 すべての機能は名前空間WebCoreにロードされます。 サブプロジェクトで構成されます：

    • ローダー-WebCore :: FrameLoader、WebCore :: DocumentLoader、WebCore :: DocumentWriter-そこから、一般的に、通信に関連するすべてのドキュメント<->外の世界があります。
    • dom-ドキュメントオブジェクトモデル。 WebCore ::ドキュメント。
    • html-WebCore :: HTMLDocument-WebCore :: Document、WebCore :: PluginDocument、WebCore :: HTMLElement、その子孫-WebCore :: HTMLAnchorElementなどの子孫。 HTMLドキュメントモデルの実装。

    • page-WebCore :: Frame、WebCore :: History、WebCore :: ContextMenu-UIまたは高レベルの通信（ブラウザ履歴など）で接続されているすべてのもの。
    • プラットフォーム-さまざまなポートの特定の機能実装。
    • レンダリング、CSS、SVG、ストレージ、プラグイン、インスペクター-名前は彼ら自身を物語っています。 詳細には考慮されていません。

ビルドの問題


見た場合：

This application failed to start because it could not find or load the Qt platform plugin "xcb".





そのため、静的ビルドの場合、次のことを忘れました

Q_IMPORT_PLUGIN(QXcbIntegrationPlugin)







動的なものについては、 libxcb.soが失われたか、または-qt-xcbスイッチを使用してqtbaseを構築した場合、プロジェクトのコンテンツqt / qtbase / plugins / platformで./platformsフォルダーを作成する必要があります（ libqxcb.soファイルは1つで十分です ）。



テストモジュールをコンパイルするときは、 インライン関数を無効にする（ -fno-inline-small-functions ）か 、ライブラリがコンパイルされた定義のリスト全体を取得する必要があります。そうでない場合は、コンパイルされたWebKitと同じincludeヘッダーに分類されます。 WebCore :: Documentでは、m_parser変数のこのエラーが原因で、クライアントコードのオフセットで8バイトが失われ、さまざまな場所でエラーが発生しました。
