Nginxベースのオンラインブロードキャストサーバー

はじめに



みなさんこんにちは！ 数か月前、 「nginxを使用したオンラインビデオのブロードキャスト」という記事がHabrで公開されました。Aecktannは、ビデオ放送用にnginx向けに開発したモジュールnginx-rtmp-moduleの実装経験について語りました。 それ以来、この製品は積極的に開発されてきました。この記事では、この製品について詳しく説明します。



ビデオストリームをクライアントに送信するには、ブロードキャスターが必要です。 ライブストリームまたは録画ビデオのブロードキャスト（VOD、ビデオオンデマンド）です。 ビデオ放送技術は多数あります。 その中でも、RTMPやMPEG-TSなどの従来のプロトコルと、HTTPを介したアダプティブブロードキャストの最近の技術を区別できます。 後者には、HLS（Apple）、HDS（Adobe）、Smooth Streaming（Microsoft）、MPEG-DASHが含まれます。 テクノロジーを選択する際の主な要因は、クライアント側でのサポートです。 RTMP形式でのブロードキャストが現在最も一般的なものの1つである理由です。 HLSプロトコルは、Appleデバイスと一部のAndroidバージョンでサポートされています。



nginx-rtmpをビルドおよび構成します



nginx-rtmpモジュールをnginxに追加するには、他のモジュールと同様に、nginxを構成するときに--add-moduleオプションで指定する必要があります。 



./configure --add-module=/path/to/nginx-rtmp-module


アセンブリおよびインストール後、nginx.conf構成ファイルにrtmp {}セクションを追加します。 構成のルートに追加する必要があります。 例： 



 rtmp { server { listen 1935; application myapp { live on; } } }


多くの場合、この単純な構成で十分です。 myappという名前のRTMPアプリケーションを設定します。 このアプリケーションでは、後でストリームを公開し、そこからストリームを再生します。 各スレッドには固有の名前もあります。 上記の構成に関する1つの重要なニュアンスに注目する価値があります。 これは、nginxワーカーの数が1に等しい場合（通常はnginx.confの先頭に設定されている場合）に当てはまります。 



 worker_processes 1;


多数のワーカーでライブブロードキャストを使用できるようにするには、rtmp_auto_push onディレクティブを指定する必要があります（ 「ワーカーとローカルリレー」セクションを参照）。



ライブストリームの公開と再生



Flashプレーヤー（JWPlayer、FlowPlayer、Strobeなど）を使用して、ビデオを公開および再生できます。 ただし、ffmpeg（およびffplay）は、サーバーストリームのブロードキャストやテストによく使用されます。 次のコマンドでtest.mp4テストファイルのブロードキャストを開始します。 



 ffmpeg -re -i /var/videos/test.mp4 -c copy -f flv rtmp://localhost/myapp/mystream


RTMPは限られたコーデックセットをサポートしますが、H264やAACなどの人気のあるコーデックもその中に含まれていることに注意してください。 テストファイルのコーデックがRTMPと互換性がない場合、トランスコーディングが必要です。 



 ffmpeg -re -i /var/videos/test.mp4 -c:v libx264 -c:a libfaac -ar 44100 -ac 2 -f flv rtmp://localhost/myapp/mystream


ファイルまたは別のソースからストリームをブロードキャストできます。 たとえば、ライブMPEG-TSストリームがvideo.example.com/livechannel.tsにブロードキャストしていると仮定した場合、rtmpでラップすることもできます。 



 ffmpeg -i http://video.example.com/livechannel.ts -c copy -f flv rtmp://localhost/myapp/mystream


ローカルWebカメラからのブロードキャストの例： 



 ffmpeg -f video4linux2 -i /dev/video0 -c:v libx264 -an -f flv rtmp://localhost/myapp/mystream


次のコマンドでffplayを使用してストリームを再生できます。 



 ffplay rtmp://localhost/myapp/mystream


そして最後に、JWPlayerを使用してブラウザからストリームを再生する簡単な例（/ test / wwwモジュールディレクトリで完全に提供）： 



 <script type="text/javascript" src="/jwplayer/jwplayer.js"></script> <div id="container">Loading the player ...</div> <script type="text/javascript"> jwplayer("container").setup({ modes: [ { type: "flash", src: "/jwplayer/player.swf", config: { bufferlength: 1, file: "mystream", streamer: "rtmp://localhost/myapp", provider: "rtmp", } } ] }); </script>




ビデオオンデマンド



このモジュールは、mp4およびflv形式のビデオファイルのブロードキャストをサポートしています。 設定例： 



 application vod { play /var/videos; }


それぞれ再生する場合、ファイル名を指定する必要があります。指定しない場合、すべてがライブブロードキャストに似ています。 



 ffplay rtmp://localhost/vod/movie1.mp4 ffplay rtmp://localhost/vod/movie2.flv




リレー



分散システムを構築する場合、負荷を分散するためにフローを中継できることが重要です

多数のサーバー上。 モジュールは、プッシュとプルの2種類のリレーを実装しています。 最初のタイプのリレー

ローカルで公開されたストリームをリモートサーバーに転送すること、およびリモートサーバーを転送することで構成されます。

ローカルサーバーへのストリーム。 リレープッシュの例： 



 application myapp { live on; push rtmp://cdn.example.com; }


パブリケーションがrtmp：// localhost / myapp / mystreamで開始されると、リモートサーバーへの接続が作成され、mystreamストリームがさらにrtmp：//cdn.example.com/myapp/mystreamにパブリッシュされます。 ローカル公開が終了すると、cdn.example.comへの接続は自動的に終了します。



プルリレーは逆の操作を実行します。 



 application myapp { live on; pull rtmp://cdn.example.com; }


この例では、ストリームrtmp：// localhost / myapp / mystreamをローカルで再生したいクライアントが表示されると、rtmp：//cdn.example.com/myapp/mytstreamを使用して接続が作成され、リモートストリームはローカルサーバーに中継されます。すべてのローカルクライアントにアクセスできます。 クライアントが残っていない時点で、接続が完了します。



モバイル放送（HLS）



iPhone / iPadデバイス、およびAndroidの新しいバージョンへのブロードキャストには、HLSプロトコル（HTTPライブストリーミング）が使用されます。

このプロトコルはAppleによって開発され、HTTP経由で送信されるMPEG-TS / H264 / AACストリームに「切り刻まれ」ます。 m3u8形式のプレイリストがストリームに添付されます。 Nginxは、HTTPを完全にレンダリングする方法を知っています。 そのため、プレイリストとHLSストリームのフラグメントを作成および更新し、古いフラグメントの削除を監視するだけです。 これを行うために、nginx-rtmp-hlsモジュールがあります。 これはhlsディレクトリにありますが、デフォルトでは収集されません。 ffmpegパッケージに含まれるlibavformatライブラリが必要です。 HLSサポートを使用してnginxをビルドするには、構成中にこのモジュールを明示的に追加する必要があります。 



 ./configure --add-module=/path/to/nginx-rtmp-module --add-module=/path/to/nginx-rtmp-module/hls


しばらく前に、ffmpegプロジェクトが分岐したことが判明しました。 そして現在、ffmpegとavconvの2つのプロジェクトがあります。したがって、ライブラリの互換性（というよりは、非互換性）の問題がすぐに発生し始めました。 nginx-rtmpをビルドするには、元のffmpegが必要です。 同時に、一部のLinuxディストリビューションはavconvの使用に切り替えましたが、これはアセンブリには適していません。 この場合、詳細な手順を書きました。



HLSを生成するには、次のディレクティブを指定するだけです。 



 application myapp { live on; hls on; hls_path /tmp/hls; hls_fragment 5s; }


最後に、http {}セクションで、HLSに関連するすべての戻り値を構成します。 



 location /hls { root /tmp; }


次に、mystreamストリームをmyappアプリケーションに公開し、iPhoneブラウザーでアドレスバーにexample.com/hls/mystream.m3u8と入力します 。 さらに、ストリームはhtmlビデオタグに埋め込むことができます。 



 <video width="600" height="300" controls="1" autoplay="1" src="http://example.com/hls/mystream.m3u8"></video>


iPhoneで再生するには、ストリームをH264およびAACでエンコードする必要があります。 ソースストリームがこれらの条件を満たさない場合、トランスコーディングを構成する必要があります。



再コーディング



ビデオをブロードキャストするとき、多くの場合、着信ストリームを異なる品質または他のコーデックにトランスコードする必要があります。 このタスクはRTMPディストリビューションと根本的に異なり、後者とは異なり、CPU負荷が高く、大量のアクティブなメモリ消費を伴い、多くの場合マルチスレッドの使用に依存し、潜在的に不安定です。 このため、メインサーバープロセスに含めるべきではなく、理想的には別のプロセスで実行する必要があります。 この問題を解決するための優れたツールがすでに存在することは注目に値します-これは同じffmpegです。 膨大な数のコーデック、フォーマット、フィルターをサポートし、多くのサードパーティライブラリを使用できます。 同時に、非常にシンプルであり、コミュニティによって積極的にサポートされています。 nginx-rtmpモジュールは、ffmpegを使用するためのシンプルなインターフェイスを提供します。 execディレクティブを使用すると、着信ストリームの公開時に外部アプリケーションを起動できます。 公開が完了すると、アプリケーションも強制的に終了します。 また、実行中のアプリケーションが突然終了した場合、実行中のアプリケーションの再起動がサポートされます。 



 application myapp { live on; exec ffmpeg -i rtmp://localhost/myapp/$name -c:v flv -c:a -s 32x32 -f flv rtmp://localhost/myapp32x32/$name; } application myapp32x32 { live on; }


この例では、ffmpegを使用して、着信ビデオをSorenson-H263にトランスコードし、32x32にサイズ変更して、結果をmyapp32x32に公開します。 同時に複数のexecディレクティブを設定できます。これにより、ストリームで変換を実行し、結果をローカルサーバーとリモートサーバーの両方で他のアプリケーションに公開します。 このディレクティブは、$ app（アプリケーション名）や$ name（ストリーム名）などのいくつかの変数をサポートしています。



労働者とローカルリレー



ご存知のように、nginxはシングルスレッドサーバーです。 最新のプロセッサのすべてのコアを効果的に使用するために、通常は複数のワーカーで実行されます。 HTTPリクエストの処理は通常、互いに独立して行われ、特定の場合（キャッシュの場合など）にのみ、共有データにアクセスする必要があります。 このようなデータは共有メモリに保存されます。



ライブ放送では、状況が異なります。 ストリームを再生するすべてのクライアント接続は、明らかにストリームを公開する接続に依存します。 この場合、共有メモリの使用は非効率的で、不必要に時間がかかり、同期化とパフォーマンスの大幅な低下につながります。 したがって、複数のワーカーを使用するために、UNIXソケットを介した内部リレーメカニズムが実装されました。 実際、このようなリレーは、実際には通常の外部プッシュリレーと違いはありません。 ローカルリレーは、次のディレクティブによって有効になります。 



 rtmp_auto_push on;


構成ファイルのルートセクションで指定する必要があります。 ローカル再送信は、ライブブロードキャストにのみ必要であることに注意してください。



記録



多くの場合、公開されたストリームをディスクに書き込む必要があります。 このモジュールを使用すると、ストリーム（オーディオ、ビデオ、キーフレーム）およびストリーム全体から個別のデータとして記録できます。 ファイルサイズと記録可能なフレーム数に制限を設定できます。 次の例では、各ストリームの最初の128Kを記録します。 



 record all; record_path /tmp/rec; record_max_size 128K;


録音は、/ tmp / recディレクトリでflv形式で行われます。



手動モードで記録を管理し、httpリクエストを使用して記録のオン/オフを切り替えることができます。 これを行うには、制御モジュールを使用します。 彼に関する情報はプロジェクトのウェブサイトで見つけることができます。



承認とビジネスロジック



多くの場合、ビデオの公開と再生には制限または会計が必要です。 これは、それが使用されているプロジェクトのロジックが原因で発生します。 最も一般的なこのようなケースは、ユーザーがビデオを視聴するためのアクセスを許可する前にユーザーを認証する必要があることです。 プロジェクトのビジネスロジックをブロードキャスターに統合するために、モジュールはon_publishやon_playなどのHTTPコールバックを実装します。 サーバーコードは、アドレス、ストリーム名、ページアドレスなど、クライアントに関する利用可能なすべての情報を受け取ります。 HTTPステータス2xxが返された場合、コールバックは正常に完了したと見なされ、クライアントは引き続き動作します。 そうしないと、接続が切断されます。 



 on_publish http://example.com/check_publisher; on_play http://example.com/check_player;




統計



いつでも、何千ものクライアントをサーバーに接続できます。 当然、リストを表示したり、それらによって公開または再生されるストリームのすべての主要な特性を表示したりできるインターフェイスが必要です。 さらに、この情報を視覚的に分析し、プログラムで処理できることが重要です。 nginx-rtmpモジュールにはそのようなインターフェースがあります。 これを使用するには、nginx.confのhttpセクションに次のディレクティブを記述する必要があります。 



 location /stat { rtmp_stat all; rtmp_stat_stylesheet stat.xsl; } location /stat.xsl { root /path/to/stat.xsl/dir/; }


rtmp_statディレクティブには、ストリームを公開または再生しているライブクライアントの完全な説明、アプリケーションおよびサーバーのリストを含むXMLドキュメントの出力が含まれます。 このドキュメントはソフトウェア処理には役立ちますが、視覚分析にはまったく適していません。 ブラウザでクライアントのリストを表示できるようにするために、rtmp_stat_stylesheetディレクティブはXMLスタイルシート（stat.xsl）への相対パスを設定します。 このファイルはプロジェクトのルートにあります。 指定したURLに配布するようにnginxを構成する必要があります。 結果はブラウザで表示できます。



クライアント接続を明示的に切断することは可能です。 これを行うには、記事に記載されていないコントロールモジュールを使用します。



シンプルなインターネットラジオ



記事の最初から、「ビデオ」という言葉を常に使用していました。 もちろん、モジュールはビデオだけでなくオーディオストリームもブロードキャストできます。 以下は、/ var / musicからmp3ファイルをブロードキャストするbashインターネットラジオ局の簡単な例です。 このストリームは、Webページに埋め込まれた単純なJWPlayerを再生できます。 



 while true; do ffmpeg -re -i "`find /var/music -type f -name '*.mp3'|sort -R|head -n 1`" -vn -c:a libfaac -ar 44100 -ac 2 -f flv rtmp://localhost/myapp/mystream; done




適合性



このモジュールは、FMS / FMLE、Wowza、Wirecastなど、RTMPプロトコルで動作するすべてのメインソフトウェアと互換性があり、最も一般的なフラッシュプレーヤーJWPlayer、FlowPlayer、StrobeMediaPlaybackでテストされ、ffmpeg / avconvおよびrtmpdumpでも動作します。



荷重



このモジュールは、非同期シングルスレッドnginxサーバーモデルを使用します。 これにより、高いパフォーマンスを実現できます。 このモジュールは、シングルワーカーモードのIntel Xeon E5320 / E5645マシンで使用します。 このモードでは、既存のネットワークカードの最大帯域幅である2 Gbpsを実現できます。 モジュールのユーザーは、複数のワーカーを使用したローカルリレーモードで同じ比率（コアあたり2Gbps）が維持されることを確認します。 練習では、ブロードキャスターのパフォーマンスは通常、CPUではなくネットワークに依存することが示されています。



他の製品とは直接比較しませんでしたが、「ヘビー」マルチスレッドFMS、WowzaおよびRed5はより機能的であり、実装の特性により、同時に接続されるクライアントの数とCPU負荷に関する決定を大幅に失います。 これは、次のような比較を行った多くのユーザーによって確認されています。 すでに述べた記事で 。
