D-Link DIR-320を3Gルーターにやり直す

多くの人がすでにこのような見出しを複数回目にしていることは間違いありません。 しかし、それらを読むたびに、世界で最も実績のあるモデムのいくつかを購入する必要があることがわかります。さらに、運はあなたにとって有利なはずです。 しかし、最も一般的な問題は、一定時間後にハングしないモデムがないことです。 私が見た最長の間隔は8（8）時間です。 その結果、多くの人はそのような考えを拒否してgiveめ、ルーターがなければ、古いPCをこれらの目的に適合させ、Windows（通常はXP、または手に入れるもの）をインストールします。

PC、さらにはWindowsも選択肢ではなく、手元に「モデム」を持っていなかったため、解決策を模索する必要がありました。

継続に興味がある場合は、「カット」をお願いします。



はじめに



おそらく、導入から多くの人に明らかになったので、小さくて非常に安定した3Gルーターを作成するという課題に直面しました。

これに必要なもの：

  • DIR-320ルーター（利用可能だった）
  • 3Gモデム
  • 外部アンテナ


研究と目標設定



だから、自分で何かをする前に、私は多数のフォーラムやコミュニティを分析することから始めました。 各フォーラムには、モデムのまさに奇跡と関連する問題の無限のシリーズの説明があるブランチがありました。 一部は非常に安定していましたが、容認できないほど低速で、他の人はパフォーマンスに満足していましたが、30分後にしっかりと「ハングアップ」しました。 これはすべて非常に動揺です。



古いPCの「祖国のバンカー」に行く前に、事実を分析し、問題を個別に調査することにしました。



最初の実験では、Linuxを搭載した固定PCを使用することにしました。 利用可能なHuawei E173を使用して3G接続をセットアップした後、彼は一度に10分ごとにping + wgetコマンドの呼び出しを開始しました。 接続は約4時間続き、その後、ATが動作している間、モデムはPPPコマンドへの応答を停止しました。 AnyDATA ADU-500Aで同じ実験を行いました（これは「これら」のリストにあるモデムです）。 奇妙なことに、5時間後にはすべて同じように終了しました。

復活の試みは待望の結果をもたらさず、完全な停電が状況を救っただけです。

もう一度、私はモデムを再起動することに決めました。それは私に気づきました、そして、なぜUSBポートを単に外さないでください。 すぐに言ってやった：

  1. すべてのUSBデバイスのリストからモデムを見つけます。 

    for f in `ls /sys/bus/usb/devices/*/product`; do echo -e "$f\t`cat $f`"; done


  2. ポートを5秒間オフにしてからオンにします。 

     echo “suspend” | sudo tee /sys/bus/usb/devices/1-1/power/level sleep 5 echo "on" | sudo tee /sys/bus/usb/devices/1-1/power/level




実験では、すべてが完全に再起動することが示されました。 また、接続を再度確立できます。 その瞬間から、何をすべきかが非常に明確になりました。



ルーター用に4つのファームウェア（D-Link、Asus、OpenWRT、またはDD-WRT）を取得して、実験を繰り返し始めましたが、残念ながら、USB電源を制御できるファームウェアはありませんでした。 最初は、2.4.xカーネルの問題だと思っていました。 しかし、2.6でも同じ問題が発生します。 さらに、ルーターを再起動した後、モデムは初期化されたままでした。 結論はそれ自体を示唆しているので、ルーターを分解する必要があります。

慎重な検査とボードの「リンギング」により、+ 5V電源レッグが電源に直接接続されていることが示されました。 これは私には合わず、ボードと電源をアップグレードする必要があります。



配布を選択する時が来ました。 このすばらしいルーターにはRAMがほとんどなく、httpd、FTP、SAMBAなど、不要なサービスがそれを消費することを望みませんでした。 最小限のプリインストールパッケージとサービスを使用して、OpenWRTに基づいて独自にディストリビューションを構築することが決定されました。



しかし、配布キットの組み立てとルーターの近代化を進める前に、タスクはターゲットモデム/オペレーターと外部アンテナを選択することでした。



さまざまな事業者のカバレッジマップを見て、MTSが最適であるという結論に達しました。 カバレッジマップに加えて、かなり便利なWEBインターフェイスからすべての統計と管理を実行できるという事実が本当に気に入りました。 約束された最大14.7 Mbit / sおよび最小3 Mbit / sで最高のモデムを購入した彼は、冒険を始めました。

おそらく誰かが興味を持っているでしょう：

  • 3Gモデム（ MTS WeTelecom WM-D300が購入しました）
  • 指向性アンテナCDMA-450（売り手、 一部の不明なメーカーによると悪くない）


購入後、このモデムへの接続を試み始めました。 そして、いつものように多くの質問が起こりました：

  1. CD-ROMとして永続的に表示されます。
  2. このモデムについてのインターネット上の言葉ではなく、 以前のモデルのこの言及が見つかった最大値（WM-D200）


dmesgの分析により、CD-ROMはSCSIとしてマウントされていることが示されました。SCSIと同様に無効にする必要があり、usb_modeswitchはここでは機能せず、実際には必要ありません。

もちろん、これを行うソフトウェアはすぐに作成されました。 しかし、これはすでに誰かによって書かれており、おそらくports / portageまたは他のどこかにすでにあると考えられていたので、私は検索しました。 man'amiを使用したフォルダーの最初のパスは、すばらしいsdparmパッケージがあることを示しました。 インストールして確認すると、明らかになりました-これが必要です。

これですべてが所定の位置に収まり、これ以上質問が残っていないので、仕事に取り掛かることができます。



OpenWRTビルド



すべてのステップはLinuxコンソールで実行されます。 Backfireリリースは2.4カーネルの最新リリースであるため、選択されていることに注意してください。 2.6カーネル自体はプロセッサのパフォーマンスとシステムリソースの両方を非常に要求するため、これは非常に重要です。

  • OpenWRTソースをダウンロードします。 
     svn co svn://svn.openwrt.org/openwrt/branches/backfire openwrt-backfire
  • ソースフォルダーに移動します 
     cd openwrt-backfire
  • すべてのパッケージを更新します 
     ./scripts/feeds update -a
  • sdparmパッケージから追加 
     ./scripts/feeds install sdparm
  • 設定メニューを開く 

     make menuconfig
  • 関心のあるプラットフォームを選択してください 
     Target System (Broadcom BCM947xx/953xx [2.4])
  • 興味のあるパッケージを埋め込みます（デフォルトではモジュールとしても含まれていません） 
     <*> sdparm........................ Read or modify SCSI or USB disk parameters <*> chat................................. Establish conversation with a modem <*> comgt............................... Option/Vodafone 3G/GPRS control tool <*> kmod-scsi-generic........................ Kernel support for SCSI generic <*> kmod-usb-core............................................ Support for USB <*> kmod-usb-ohci............................... Support for OHCI controllers <*> kmod-usb-serial..................... Support for USB-to-Serial converters <*> kmod-usb-storage..................................... USB Storage support <*> kmod-usb2................................... Support for USB2 controllers
  • 他のすべてのアイテムは変更されません。
  • 保存して設定メニューを終了します
  • ツールを収集します 
     make tools/install -j`grep -c processor /proc/cpuinfo` make toolchain/install -j`grep -c processor /proc/cpuinfo`
  • Linuxカーネル構成メニューを開きます 
     make kernel_menuconfig
  • MTDをカスタマイズする 
     [*] Support 2-chip flash interleave
  • 保存して設定メニューを終了します
  • OpenWRTをビルドする 
     make -j`grep -c processor /proc/cpuinfo`
  • ファームウェアを組み立てた後、デバイスにフラッシュする必要があります。 このテーマに関する多くの指示があります（ たとえば、DD-WRT Webサイト ）


ファームウェアの確認



ファームウェアが「フラッディング」され、ルーターが再起動したら、モデムをファームウェアに接続し、ルーターをイーサネット経由でPCに接続します。 次のコマンドを実行して、PCのインターフェイス（私の場合はeth0）にIPアドレスを設定します。 
 ifconfig eth0 inet 192.168.1.2/24


次に、Telnet経由でルーターに接続します。 これを行うには、コンソールから次のコマンドを実行します。 

 telnet -l root 192.168.1.1


パスワードを要求せずにログインが自動的に行われます。 これは根本的に間違っています。 状況を修正するには、コンソールでコマンドを実行してパスワードを設定します。 

 passwd


exitコマンドを実行して、 telnetセッションを終了します。

ssh経由でログインします 。 

 ssh root@192.168.1.1


次のステップは、コンソールでコマンドを実行して、モデムがカーネルによって認識されていることを確認することです。 
 grep "^P\:.*22de.*6803" /proc/bus/usb/devices


このコマンドの結果は、モデムのVIDとPIDを含む文字列になります。 それ以外の場合は、指示に従って実行されていないことを確認してください。



ルーターハードウェアのアップグレード



USBポートの電力を制御するには、無料のプロセッサレッグが必要です。 ただし、これはエンジニアリングボードではなく最終製品であるため、このオプションは適していません。 幸いなことに、このルーターの前面パネルには多数のLEDインジケーターがあります。 それらの1つは、ニーズに非常に適応できます。 私の場合、それはproc-fs（/ proc / diag / led / usb）のUSB LEDに対応していたため、プリンターアイコンの付いたインジケーターで選択しました。



電源管理は、最小限の必要な部品セットを備えた追加のボードによって実行されます。 彼女の図は次のとおりです。



スキームに関するいくつかの言葉。 両方のトランジスタがキーとして機能します。 論理ユニットがCTRL入力に入力されると、トランジスタVT1が開き、トランジスタVT2が開き、電源電圧がUSBポートに供給されます。 そのため、計算は行われなかったため、スキームを最適と見なすことはできません。 たとえば、トランジスタVT1は、実際には、抵抗のようにリーチフィールドでの唯一のNPNトランジスタでした。



プリント回路基板を作成する時間も希望もなかったので、回路基板の小さな部分をその上に費やしました。 最終バージョンは次のとおりです。



緑色のラベルは、回線上の同じ名前のポートを示します。 LEDの脚に直接はんだ付けされた抵抗R4。



段階的な変更手順：

  • ルーターを取り外し、分解します。
  • 不要なR48を取り外します-この抵抗はLEDをプロセッサの脚に接続します



  • ヒューズF1を取り外します-このヒューズは、USB電力線が短絡した場合に保護する必要があります。 その額面はわかりませんが、スキーム（R3）で適用しました。 黄色の文字は、制御ボードの接続ポイントを示していることに注意してください



  • プロセッサからR48抵抗器に信号を導く導体から保護コーティングを除去します。 この導体に、制御基板のCTRLポートをはんだ付けする必要があります



  • コントロールボードを最も便利な場所に貼り付けます。 アンテナソケットのすぐ下の空き領域が気に入ったことがわかりました。 この場合、接続ワイヤの長さは最小になります



  • 写真のように、抵抗R4をLEDのアノードに直接はんだ付けします。 接続線の接続にも注意してください



  • 接続ワイヤのインストールを完了します



  • 最終結果



  • ルーターを構築してオンにする


ルーターソフトウェアのアップグレード



これが「幸福」への最後のステップです。 多数の新しいスクリプトを作成し、既存のスクリプトを変更する必要があります。 だから、すべての行。

sshを介してルーターに接続します。次のすべての手順がルーター上で直接実行されるためです。



USBコントロール用のユーティリティの作成


/ etc / pppフォルダーに移動します。 多くの人は、このフォルダーがターゲットとして選択された理由に興味を持っています。 答えは単純ですので、歴史的に。 / usr / local / sbinの方が適していると思いますが、これは本質を変えません 

 cd /etc/ppp


  • usb-up.shを作成します。 このスクリプトには、USBポートの電源が含まれます。 
     $ cat usb-up.sh #!/bin/sh logger -t "USB PwrCtrl" "Turn-on USB power" echo 1 > /proc/diag/led/usb
  • USBポートを再起動するスクリプトを作成します 
     $ cat resetusb.sh #!/bin/sh logger -t "USB PwrCtrl" "Turn-off USB power" echo 0 > /proc/diag/led/usb # Sleep in 3 sec is an optional, but may be useful for some modem types #sleep 3 logger -t "USB PwrCtrl" "Turn-on USB power" echo 1 > /proc/diag/led/usb
  • pppdプロセスの存在を確認するスクリプトを作成します 
     #!/bin/sh if ps | grep -q /usr/sbin/pppd; then logger -t "PPP watcher" "PPP alive" else logger -t "PPP watcher" "No PPP daemon" /etc/ppp/resetusb.sh fi
  • スクリプトを実行可能にする 
     chmod a+x /etc/ppp/*.sh
  • モデムをUSBに接続するための正しい処理を追加します。 OpenWRTでは、デフォルトでudevがなく、代わりにhotplugデーモンが使用されることに注意してください 
     $ cat /etc/hotplug.d/usb/11-modem #!/bin/sh # Copyright (C) 2013 Sergey Shcherbakov <shchers@gmail.com> case "$ACTION" in add) (grep -q "^P\:.*22de.*6803" /proc/bus/usb/devices) && { logger -t "WM-D300" "Detected ZeroCD of WM-D300 modem" # Checking ZeroCD status while [ "$(sdparm -q --command=ready /dev/scsi/host0/bus0/target0/lun0/generic)" != "Ready" ]; do logger -t "WM-D300" "ZeroCD yet not ready" # Workaround for bug with multiple calls of scripts (/usr/bin/lsusb | grep -q 22de:6801) && { logger -t "WM-D300" "Seems that script called twice and device already initialized" return 0 } sleep 2 done logger -t "WM-D300" "ZeroCD ready and will be ejected" /usr/bin/sdparm --command=eject /dev/scsi/host0/bus0/target0/lun0/generic } (grep -q "^P\:.*22de.*6801" /proc/bus/usb/devices) && [ ! -c /dev/usb/tts/0 ] && { logger -t "WM-D300" "Updating USB Serial driver with D200/D300 vendor and product ID" rmmod usbserial && insmod usbserial vendor=0x22de product=0x6801 } (grep -q "^P\:.*22de.*6801" /proc/bus/usb/devices) && [ -c /dev/usb/tts/2 ] && ! (ifconfig 3g-wan 1>/dev/null 2>/dev/null) && { logger -t "WM-D300" "All modem interfaces installed, turning on WAN" ifdown wan && ifup wan } ;; remove) ! (grep -q "^P\:.*22de.*6801" /proc/bus/usb/devices) && { logger -t "WM-D300" "Modem disconnected, turning off WAN interface" ifdown wan } ;; esac


既存のスクリプトを変更する


  • コマンドを実行してデフォルトのWANインターフェイスを無効にします 
     sed -i 's/interface\ wan$/interface\ wan_dflt/' /etc/config/network
  • 3Gの設定で新しいWANインターフェイス記述子を追加します 
     $ cat /etc/config/network … config interface wan option ifname ppp0 option device /dev/usb/tts/0 option service evdo option proto 3g option username 'mobile' option password 'internet' option pppd_options 'noipdefault maxfail 3' option connect '/etc/ppp/usb-up.sh' option disconnect '/etc/ppp/resetusb.sh' option keepalive 20
  • ルーターの初期化時にUSBポートを追加して自動的にオンにするには、start（）関数の最後にスクリプト/etc/ppp/usb-up.shを追加して、標準スクリプト/etc/init.d/usbを変更する必要があります 
     $ cat /etc/init.d/usb … start() { ... # Turn-on power on USB /etc/ppp/usb-up.sh } …
  • さらに、チャットスクリプト（/etc/chatscripts/evdo.chat）を変更できますが、必ずしもそうする必要はありません。 実際、変更を加えなくても動作するはずですが、私はそれを少しとかすことを好みました。 私のオプションは次のとおりです。 
     ABORT BUSY ABORT 'NO CARRIER' ABORT ERROR ABORT 'NO DIAL TONE' ABORT 'NO DIALTONE' ABORT 'NO ANSWER' REPORT CONNECT TIMEOUT 10 '' AT OK ATDT#777 CONNECT ''
  • 5分ごとにpppdプロセスをチェックするタスクをCronに追加します。 これを行うには、crontab -eコマンドを呼び出して、次の行を追加します。 
     */5 * * * * /etc/ppp/check.sh
  • Cronの実行と実行を許可する 
     /etc/init.d/cron enable && /etc/init.d/cron start
  • WiFi設定は十分に標準であり、OpenWRTの公式Webサイトで詳しく説明されているため、WiFi設定は提供しません。


それだけです。これ以上の変更はありません。 ルーターを再起動することをお勧めしますが、とにかく5分で3Gが「稼働」するはずです。



エピローグの代わりに



ホームトライアルは41日間続きました。 その結果、期間中の単一の障害ではありません。 期待される安定性が達成されました。

「フィールド」に送信する前に、キエフのアパートで帯域幅テストが実施されました。 期待はずれでしたが、最大のファイルは1Mbit / sを超えませんでしたが、大きなファイルに限られていました。 最小pingは220ミリ秒以内でした。 テストを止めないことに決めて、彼は町を出ました。 レベルは最大でしたが、CDMA Rev.Bインジケータはもうオンではありませんでした。 実際、私は14 Mbit / sの希望を抱きませんでしたが、少なくとも1 Mbitがすでに達成されることを期待していました。 しかし、もう少しがっかりすることはありませんでした。最大値はすでに約350 Kbpsでした。

この時点で、MTSのモデムを使用したあらゆる種類の実験は中止されました。 返却できませんでした。 しかし、売り手はアンテナをIntertelecomおよびPeopleNETモデムと連動できるモデルと交換しました。 信じられないほど幸せで、モデムをコレクションに入れておくと、いつか便利になります。
