XenServerでThunderbolt Ethernet Adapterを上げる

こんにちは、Habroコミュニティの皆さん、ここでは、XenServer 6.2でAppleのThunderbolt Ethernet Adapterをマニュアルの下でどのように機能させたかについての記事を書いてくれました。同じことをするなら時間を節約できます。



プレイ中に判明したように、Citrix XenServerに何かをインストールするのはかなり面倒なタスクであり、ソースからドライバーをアセンブルする必要がある場合（これはまさに必要なことです）、とにかくこれは不可能なタスクです。 私の推測では、これは偶然ではなく、セキュリティとホストシステムのパフォーマンスの理由で行われたものです。これは、システム内のすべての追加パッケージがオーバーヘッドを必要とし、システム全体のセキュリティを低下させるためです。





背景（気にする人は、 次のセクションに進んでください）



すでに遠い2010年、私は自宅で仕事をして、24時間体制でそれぞれの難しいタスクを実行する複数の仮想マシンを備えたサーバーを立ち上げました。そして最も重要なことは、この鉄片の隣にいる人々の平和を妨げません。 長い選択の後（ Avitoで広告が正常に見つかったため ）、C2D 2.5Ghz、8GB Ram、2x500Gb HDDの特性を備えたMac Mini Serverが選択されました。先を見据えて、このマシンは実際に戦闘任務中です。ここで、それはすでにテーブルの上に誇張されており、一般的に甘いもので、すべてのそのような白いトップを持つアルミニウムです。 ハイパーバイザーの検索は、最初は自分の愚かさと紙幣の過剰から始まりました。ParallelsServer for Mac 4.0 Mac miniエディションが選択されました。これには約9kの木製費用がかかり、同時に2台（！）の仮想マシンを同時に起動できました。十分で、すべてが大丈夫でした。



しかし、時間が経つにつれて、いつものように、これは見逃され始め、より多くのVMが必要であると言われました-完了しました、すでにより印象的な構成のi7 2.0（4コア8スレッド）、16Gb Ramがそれらに詰め込まれ、すべてがうまくいくといくつかのMac Miniが購入されました、しかし、1コアと1GBのRamを必要とする1台のマシンで2台のVMのみを実行するのはどういうわけか悲しくなりました。インターネットを調べて、さらに興味深いハイパーバイザーを見つけることが決定されました。



研究中にHyperV、ProxMoxが試されましたが、選択はXenまたはむしろXen Cloud Platformで行われましたが、残念ながら長くは続かず、いつかXCPが閉じられたというメッセージが表示され、XenServer今は無料で、何もすることはありません。XenServerにアクセスしてください。 さらに-多分、多くのサーバー、高可用性が必要です、16GB Ramを搭載した別のサーバーXeon e3-1240v2がこれらの問題のために購入されました月には、外出先でのHA移行が可能なXenクラスターとその他の素敵なパンが組み立てられましたが、時間が経つにつれて、n個のVMで1Gb / sの帯域幅がなくなり、VMイメージ自体のように実行されることに気付き始めましたサーバー内のNAS'a、および同じNAS'eに保存されているデータ。 802.3adが可能な合理的なスイッチを配置することを決定し、次にNAS'eでイーサネットチャネルの数を6x1GB / sに増やし、サーバーでは2x1GB / sに増やしましたが、Apple Thunderbolt Ethernet Adapterが手に入るとすぐに開始しました集合農場の朝、ハブに関するこの記事になりました...





さあ始めましょう



そして、私たちは持っています：
  • Apple Thunderbolt Ethernet Adapter（MD463ZM）、Broadcom 10/100 / 1000BASE-T PCIeネットワークカード、NetXtreme BCM57762
  • Thunderboltコネクタを備えたマザーボード（インターフェイスはPCIeにすぎず、ワイヤーの形のみ）
  • XenCenterを使用するWindowsコンピューター


今回のケースでは、XenServer 6.2がインストールされ、SP1にアップデートされた2011年半ばのMac Mini Serverが対象でした。



1.アダプターをコネクターに接続し、ネットワークケーブルを接続してサーバーを再起動します（再起動しないと検出されません）。アダプターが検出されるかどうかを確認します。 

[root@xNode1 ~]# lspci ... 0a:00.0 PCI bridge: Intel Corporation DSL3510 Thunderbolt Controller [Cactus Ridge] 0b:00.0 PCI bridge: Intel Corporation DSL3510 Thunderbolt Controller [Cactus Ridge] 0c:00.0 Ethernet controller: Broadcom Corporation NetXtreme BCM57762 Gigabit Ethernet PCIe


Broadcomが標準ドライバーにこのアダプターのサポートを含めなかった理由は私には謎のままですが、おそらくAppleとの何らかの合意によっておそらくサポートしなかったという事実が残っています。





2.前述のように、ホストシステムに何かをインストールすることは非常に問題があります。すべての不要なパッケージと「make」もそこから切り取られ、デフォルトでレポジトリがCitrixに設定されているためです。もちろん、Citrixレポとbaserepoを通過しますが、これは私たちの場合ではありません。メーカーのポリシーをサポートし、システムを詰まらせたり害を与えたりしないことを前提としています。



Citrixは、システムの「超過」をクリアしたレベルでは馬鹿ではありません。同じカーネルと、ドライバーイメージを構築するために必要なパッケージの全セットを備えたシステムのイメージを作成し、ホストシステムに簡単にインストールできます。 この獣は、Driver Development Kit（DDK）と呼ばれます。ここからダウンロードできます： http : //support.citrix.com/article/CTX139817 （執筆時点では、現在のバージョンは6.2 SP1です）。 このiso-shnikは仮想マシンイメージに過ぎませんが、Xenサーバーにインポートするには、まずオペレーティングシステムにマウントし、次にXen Centerでサーバーを右クリックして選択する必要がありますインポート。 インポートするには、マウントされたイメージのddkフォルダーにあるova.xmlファイルを選択する必要があります。 （インポートする際にイーサネットインターフェースを追加することを忘れないでください）





3.イメージがインポートされた後、ルートにパスワードを割り当てました。ドライバーソース自体を取得する必要があります。http ：//www.broadcom.com/support/ethernet_nic/netxtreme_desktop.php最下部にあり、Linux（tg3）と呼ばれます、habro-residentsの便宜のために、修正済みのソースhttp://my-files.ru/Download/wqet0i/tg3-drv.zipをすでに投稿しています。



ファイルをDDKサーバーにアップロードしてから解凍します。 

 [root@localhost ~]# wget http://my-files.ru/Download/wqet0i/tg3-drv.zip ... 2014-01-14 01:48:13 (776 KB/s) - `tg3-drv.zip` saved [349509/349509] [root@localhost ~]# unzip tg3-drv.zip


Broadcom Webサイトの元のドライバーを使用している場合
tg3-3.133d.tar.gzアーカイブにあるtg3.cファイルを編集する必要があります。 ファイル内の行を見つけます。 

 ... {PCI_DEVICE(PCI_VENDOR_ID_BROADCOM, TG3PCI_DEVICE_TIGON3_57761)}, {PCI_DEVICE(PCI_VENDOR_ID_BROADCOM, TG3PCI_DEVICE_TIGON3_57765)}, ...


デバイス57762の識別子を追加すると、最終的には次のようになります。 

 ... {PCI_DEVICE(PCI_VENDOR_ID_BROADCOM, TG3PCI_DEVICE_TIGON3_57761)}, {PCI_DEVICE(PCI_VENDOR_ID_BROADCOM, TG3PCI_DEVICE_TIGON3_57762)}, {PCI_DEVICE(PCI_VENDOR_ID_BROADCOM, TG3PCI_DEVICE_TIGON3_57765)}, ...


組み立てる前に、ドライバーイメージを組み立てるスクリプトが必要です。 私が修正したスクリプトのバージョンは、 http：//my-files.ru/Download/ntqvvd/tg3-proj.zipで取得できます。元のhttp://discussions.citrix.com/topic/301614-bcm5754-tg3-and-vlan-tagging -do-not-work-in-xs6 / 、ポスト16（登録が必要です）。



ファイルをダウンロードしてから解凍します。 

 [root@localhost ~]# wget http://my-files.ru/Download/ntqvvd/tg3-proj.zip ... 2014-01-14 02:29:38 (72.6 MB/s) - `tg3-proj.zip` saved [1738/1738] [root@localhost ~]# unzip tg3-proj.zip


元のスクリプトを使用している場合
Makefileを編集する必要があります 

 ... RPM_VERSION := 3.110g //     ... build-iso: build-rpms ... cd $(PACKAGE) && /usr/local/bin/build-supplemental-pack.sh --output=$(dir $(ISO)) ... //     ... build-tarball: build-rpms ... cd $(PACKAGE) && /usr/local/bin/build-supplemental-pack.sh --output=$(dir $(ISO)) ... //      ...


それは判明するはずです： 

 ... RPM_VERSION := 3.133d //     ... build-iso: build-rpms ... cd $(PACKAGE) && build-supplemental-pack.sh --output=$(dir $(ISO)) ... //     ... build-tarball: build-rpms ... cd $(PACKAGE) && build-supplemental-pack.sh --output=$(dir $(ISO)) ... //      ...


tg3.specファイルを編集することも必要です 

 ... Version: %{?version}%{!?version:3.110g} //     ...


それは判明するはずです： 

 ... Version: %{?version}%{!?version:3.133d} ...




4.ドライバーイメージのアセンブリに移動します。 

 [root@localhost ~]# cd tg3-drv/ [root@localhost tg3-drv]# make tg3_flags.h [root@localhost tg3-drv]# cd .. [root@localhost ~]# mv tg3-drv tg3-3.133d [root@localhost ~]# tar czvpf tg3-3.133d.tar.gz tg3-3.133d/ [root@localhost ~]# cp tg3-3.133d.tar.gz /usr/src/redhat/SOURCES/


上記のアクションで、スクリプトアセンブリのソースを準備しました。スクリプトを実行するためだけに残りますが、スクリプトを実行する前に、イメージに現在の時刻を設定します。そうしないと、スクリプトは警告で終了します。 

 [root@localhost ~]# cd tg3-proj/ [root@localhost tg3-proj]# date 011311282014.30 //(,  , , , 4  , ) Mon Jan 13 11:28:30 EST 2014 [root@localhost tg3-proj]# make ... Total translation table size: 0 Total rockridge attributes bytes: 780 Total directory bytes: 0 Path table size(bytes): 10 Max brk space used 21000 283 extents written (0 MB)


ファイルtg3.iso、tg3.iso.md5、tg3.metadata.md5などがフォルダーに表示されます。tg3.isoにのみ関心があり、ホストサーバーに送信する必要があります。これは、sshを介してファイルを駆動するscpコマンドを使用して行います。 

 [root@localhost tg3-proj]# scp tg3.iso root@192.168.0.2:/root/tg3.iso




5.最も難しい部分は背後にあり、残りはホストサーバーにドライバーをインストールすることです。このため、サーバーにログインするには、Xen Centerのコンソールから次の手順を実行します。 

 mkdir -p /mnt/tmp mount /root/tg3.iso /mnt/tmp -o loop,ro cd /mnt/tmp/ ./install.sh cd /root/ umount /mnt/tmp shutdown -r now


したがって、再起動後の最後のタッチは、アダプターが使用可能なデバイスに表示されないためです。アダプターを表示するには、サーバーコンソールに移動し、xsconsoleコマンドを使用してマネージャーを呼び出し、[ネットワークと管理インターフェイス]項目で[緊急]を選択する必要がありますネットワークリセット」は、アダプターをリセットし、Thunderbolt Ethernet Adapterを再起動すると使用可能になります。
