Hyper-V VMおよびMSSQLのフェールオーバー

序文の代わりに



フォールトトレランスは、1つのデータセンターのフレームワーク内で理解されます。つまり、1〜2台の物理サーバーの障害からの保護です。

私たちの実装は、ハードウェア、つまりドイツの有名なホストがレンタルするものの点で安価です。

ソフトウェアコストの面では、無料または既に利用可能です。 マイクロソフトのアフィリエイトプログラム、いわば、実行中です。

Windows Server 2012市場の出現に伴い、「サーバーからクラウドへ」、「アプリケーションは常に機能します」という多くの広告がありました。 それが私たちが実装しようとするものです。



もちろん、ホリバー向けのトピックがあります。VMWareやHyper-Vよりも優れているものですが、これはこの投稿のトピックではありません。 私は議論しません。 味と色について-すべてのフェルトペンは異なります。

おそらく、このビジネスをAzureに送信できると言う人もいるでしょう-それはさらに安いことが判明するかもしれませんが、私たちは偏執的であり、fayloveromとvirtualkaでクラスターを「クラウド」にしたいのです。 ところで、もし望むなら、そこに経済を送ることを妨げるものは何もない。







ソリューション要件



以下を使用する特定のプロジェクトがあります。

  1. データベース-MSSQL
  2. バックエンド-IIS
  3. フロントエンド-PHPのある種のアプリケーション




以下を実装する必要があります。

  1. この束は「常に」機能しました。
  2. 「鉄」サーバーの障害はダウンタイムを引き起こしませんでした。
  3. データは失われませんでした。
  4. 何らかの負荷分散がありました。
  5. スケーラビリティ。
  6. 上記を実装するために、ソフトウェア（たとえば、Identity for MSSQL）で庭をフェンスする必要はありませんでした。


この経済はすべて、1つの有名なドイツのホスティングでホストされます。



実装



実装を論理部分に分割します。

  1. ハードウェア要件。
  2. 準備作業
  3. フォールトトレランスMSSQL（バランス要素を使用）。
  4. フォールトトレランスバックエンドとフロントエンド
  5. ネットワーク設定
  6. 耐障害性：別のマイルストーン。




ハードウェア要件


これには、少なくとも4つのサーバーが必要です。 それらが同じデータセンターにあり、できれば1つのスイッチにあることが非常に望ましいです。 この場合、別のラックがあります。 そして、彼らが私たちに説明したように、1つのラックしかないので、彼女のスイッチも専用です。



サーバー


2台のサーバー-仮想化対応プロセッサー、32GB RAM、2HDD X 3TB（RAID 1）

残りの2つはSQLの下に置かれるので、少し変更します（構成では、1つのハードドライブをRAIDコントローラーとそれぞれ300 GBの3つのSASディスクに置き換えます（これらはRAID 5-MSSQLの高速ストレージになります）。

原則として、これは必要ありません。 もちろん、耐障害性は低下しますが、速度がより重要です。

また、フラッシュドライブも必要になります（ただし、以下でさらに詳しく説明します）。

オプション：ローカルネットワークを整理するための個別のスイッチ。ただし、これはプロジェクトの成長に合わせて後から行うことができます。



準備作業


フェールオーバークラスターを上げるため、Active Directoryドメインが必要になります。

その後、SQLサーバーでバックエンドを承認するタスクを簡素化します。

仮想マシンのドメインコントローラーを上げます。

また、ローカルネットワークのアドレス指定を決定する必要があります。

もちろん、DC（ドメインコントローラー）には白いIPアドレスがなく、NATを介して "出力"されます。

すべての仮想マシンの設定ではなく、プライマリDNS：ドメインコントローラー。

2番目のIPアドレスは、発行された白いものに加えて、ローカルネットワークのアドレスを登録する必要があります。

理想的なオプションは、以下のスケーラビリティの観点から説明されています。



フォールトトレランスMSSQL。


MSSQLクラスタリングを使用しますが、古典的な意味では使用しません。つまり、サービス全体ではなく、リスナーのみをクラスター化します。 すべてのMSSQLをクラスター化するには、共通のリポジトリが必要であり、これが障害のポイントになります。 障害点を最小化する道を辿っています。 これを行うには、MSSQL Server 2012の新機能-Always Onを利用します。

この機能の説明は、 SQL Server 2012-Andrew FryerによるAlways Onで詳しく説明されています 。 また、構成方法について詳しく説明します。

要するに、レプリケーションとミラーリングの2つのテクノロジーの組み合わせ。 両方のデータベースインスタンスには同一の情報が含まれていますが、それぞれが独自のリポジトリを使用しています。

読み取り専用レプリカを使用して負荷分散を使用することもできます。 読み取りルートを事前に構成することにより、詳細を読む-SQL Server 2012 Always Onデータベース可用性グループによる読み取り専用ルーティング

一般に、この問題のベストプラクティスは、高可用性と災害復旧のためのMicrosoft SQL Server AlwaysOnソリューションガイドのLeRoy Tuttleによって詳細に説明されています。

MSSQL設定のパス構成が同一でなければならないという事実にのみ焦点を当てます。



フォールトトレランスバックエンドとフロントエンド。


仮想マシンをクラスタリングすることにより、この機能を実装します。

仮想マシンをクラスター化するには、クラスター共有ボリューム（CSV）が必要です。

また、CSVを作成するにはSANが必要であり、クラスターによって検証され、無料である必要があります。 これは簡単な作業ではないことがわかりました。 多数のソリューションが試されました（オープンソースではなく）。 その結果、目的の製品が発見されました。 彼はNexentaStorと呼ばれています

18 TBの未加工スペース、無料のプロトコルとチップの束。

展開時に、 ULPの 運用経験Nexentaまたは2か月後の経験と推奨事項を考慮する必要がある唯一のこと

残念ながら、私たちは自分でこのレーキを歩きました。



Nexentaには定期的に「疾患」もあります。Webインターフェースは応答を停止しますが、他のサービスはすべて正常に機能します。 ソリューションは、 http：//www.nexentastor.org/boards/2/topics/2598#message-2979で入手できます。

だから。 インストールの詳細。

Nexentaをインストールしようとしていますが、インストールは成功です。 私たちはシステムに入り、驚きました。すべての空きスペースがシステムプールの下に費やされ、データを配置する場所がありません。 解決策は明らかであるように思われます。サーバーへのハードドライブの順序を変更し、データを保存するためのプールを作成しますが、別の解決策があります。 これを行うには、フラッシュドライブを使用します。

システムをフラッシュドライブに配置します（このプロセスには約3時間かかります）。

インストール後、システムプールとデータ用のプールを作成します。 フラッシュドライブをシステムプールに接続して同期します。 その後、プールからフラッシュドライブを削除できます。 詳細については、 http：//www.nexentastor.org/boards/1/topics/356#message-391で説明しています 。

そして、そのような写真を取得します。







zvolを作成します。 作成後、ターゲットにバインドし、ISCSI経由で公開します。

クラスターの各ノードに接続します。 そして、それをクラスターの共有ボリュームに追加します。







したがって、クラスターの各ノードのHyper-V設定では、仮想マシン構成とその上のハードディスクファイルの場所を示します。

また、重要ではありません-各ノードの仮想スイッチの名前も同じでなければなりません。



その後、仮想マシンを作成して、それらの仮想マシンを構成できます。

もちろん、OSの選択は、 統合サービスがあるMS WindowsとLinuxに限定されますが、私たちはそれらを使用することになりました。



また、ドメインコントローラーをHyper-Vクラスターに追加することを忘れないでください。



ネットワーク設定


フォールトトレラントSQLが既にあり、フォールトトレラントフロントエンドとバックエンドがあります。

彼らが外の世界からアクセスできるように作られたままです。

ホスティングプロバイダーには、この機能を実装するための2つのサービスがあります。

  1. サーバーに追加のIPアドレスを要求し、それをMACアドレスにバインドすることができます。
  2. サブネット全体/ 29または/ 28を要求し、1ポイントのアドレスにルーティングするように依頼することもできます。


Hyper-vクラスターにもう1つ仮想マシンを作成します。 この目的のために、 ClearOSを使用します 。 CentOSに基づいて構築され、統合サービスをインストールできるため、選択はすぐに彼女にかかった。

インストール後にインストールすることを忘れないでください。そうしないと、ネットワークインターフェースの消失に問題が生じる可能性があります。



彼女には3つのインターフェイスがあります。

  1. ローカルエリアネットワーク
  2. Dmz
  3. 外部ネットワーク




外部ネットワークは、プロバイダーに尋ねた追加のアドレスです

DMZ-プロバイダーから提供されたサブネット。







また、このマシンは、NATを介して仮想マシン（白いIPアドレスを持たない）を解放します。

したがって、ルーティングフェールオーバーも作成しました。 ルーターはクラスター化されており、ノード間を移動することもできます。

ノード自体で、ファイアウォールを設定することを忘れないでください（危険なポート上の信頼できるものとローカルのものを除くすべてのIPからのアクセスをブロックします）。 白いIPアドレスをオフにする方が良いでしょう。

もちろん、複数のホワイトアドレスが必要ない場合は、サブネット全体を強調表示しても意味がなく、外部からのアクセスはポートフォワーディングおよびリバースプロキシを介して実現できます。



フォールトトレランス：別のマイルストーン


前述したように、障害点を減らすための道を進んでいます。 しかし、まだ1つの障害点があります。これがSANです。 ドメインコントローラーを含むすべてのクラスター化された仮想マシンがその上にあるため、このリソースが消失すると、バックエンドとフロントエンドが消失するだけでなく、クラスターが崩壊します。

サーバーがもう1つ残っています。 最後のフロンティアとして使用します。

バックアップサーバーに2番目のドメインコントローラーを使用して仮想マシンを作成し、ADレプリケーションを構成します。

すべてのマシン上のセカンダリDNSサーバーを忘れずに登録してください。 この場合、CSVが消えても、CSVに依存しないサービス、つまりクラスター化されたSQLリスナーは機能し続けます。

CSVの崩壊後、バックエンドとフロントエンドが運用に戻るために、新しいWindows Server 2012機能であるHyper-Vレプリケーションを使用します。 プロジェクトの重要なマシンを4台のサーバーに複製します。 最小レプリケーション期間は5分ですが、それほど重要ではありません。 フロントエンドとバックエンドには、めったに更新されない静的データが含まれています。

これを実現するには、Hyper-Vレプリカブローカークラスターの役割を追加する必要があります。 そして、そのプロパティで複製プロパティを設定します。 レプリケーションの詳細：

Hyper-Vレプリカの概要

そして、もちろん、バックアップを忘れないでください。



スケーラビリティについて少し


このソリューションは、ノードサーバーを追加することで、将来的に拡張できます。

MSSQLサーバーは、読み取り専用ノードを追加し、読み取りルートのバランスをとることにより拡張されます。

仮想マシンは、ハードウェアに縛られることなく、リソースノードのサイズに「膨張」させることができます。

トラフィックを最適化するために、ノードサーバーに追加のインターフェイスを追加し、これらのインターフェイスにHyper-V仮想スイッチを接続できます。 これにより、外部トラフィックと内部トラフィックを分離できます。

Azureで仮想マシンを複製できます。

SCVMMとOrchestratorを追加して、「プライベートクラウド」を取得できます。



このようなものを使用すると、ある程度までフォールトトレラントな独自のクラスターを構築できます。 原則として、すべてのフォールトトレランス。
