  3. 数字ではなくテキストを割り当てることができない

新機能

  7. 値が範囲外の場合、スライダーを非表示にします

4番目のバージョンで、スライダーの操作がupdateLeftValueとupdateRightValueの呼び出しに制限されていた場合、更新されたTrackBarの操作の原則は次のとおりです。

  1. まず、必要な値を直接設定します。例えば

    var scaleObj = $.trackbar.getObject("scale");





    scaleObj.leftValue = 10;





    scaleObj.rightValue = 20;





    scaleObj.rightLimit = 100;



  2. 次に、1つ以上の値アプリケーション関数を呼び出しscaleObj.SetPosition();



    たとえば、 scaleObj.SetPosition();





    scaleObj.ResetLimits();





すべての機能を実証するために、元のデモが行われ、少し装飾されました（追加されたパラメーターは青で強調表示されます）。また、7番で、2つのスライダーに基づいて任意の時間範囲の選択を整理する例を追加しました：上の方が範囲のスケールを決定し、下の方がそれを設定します。 その中で、すべての改善が最大限に、最も明白な使用例に使用されています。



元のスライダーを完成させる上で最も興味深いことは、それらにセリフを追加することでした。 それらの実装は異なる可能性がありますが、私は次のアプローチに決めました。

  1. 最初に、メインdivが編成されます。このdivでは、スケールが配置され、スライダーのある線の下に表示されます
  2. 次に、異なるidのセットが生成され、どのセリフを描画するかが決定されます
  3. シャム双子のようにセリフが結合しないように、セリフはそれらの間に挿入されますが、幅は各スライダーに対して個別に計算されます。
この規模の署名は、ほぼ同じ原則に基づいて構築されています。

このアプローチにより、セリフを簡単かつ迅速に作成してcssで制御できますが、重大な欠点が1つあります。すべてのブラウザーが小数値のすべての符号を使用しない、破棄または丸めるということです。 したがって、多数のセリフ（40を超える）を使用すると、右側のセリフがトラックバーを完全にカバーしていないことが明確にわかり（例1を参照）、セリフの位置と実際の値に矛盾があります。 小さな値では、これは完全に見えません。



ブラウザー間の互換性と既知の欠点

最初、すべての開発はセカンドバージョンのFirefoxで行われました。これにより、セリフ用のかなり良いレイアウトをすばやく簡単に作成し、他のブラウザー用にさらに最適化することができました。 それでは、さまざまなブラウザーのスライダーの動作の分析を始めましょう。



Firefox v2.0.0.13

問題は認められず、すべてが意図したとおりに機能します。 時々発生する可能性のある唯一の問題は、セリフとスライダーの右側の値と多数のセリフの不正確な一致ですが、これは上記のとおりです。





IE v7

奇妙なことに、すべてが正常に動作します。Firefoxの場合とほぼ同じですが、スライダーの右側に既に多くの不一致があります。 さらに、ノッチ自体は数ピクセル高くレンダリングされます。





IE v6およびv8b1

外観では、すべてがIE 7に似ていますが、中央部分に透明性はありません。 それほど重要ではありませんが、特にIE 8では不愉快です。おそらく、cssの透明度の値は単純に誤って定義されており、これは何らかの形で将来的に理論的に修正できます。 IE 6では、7番目の例で、小さなテキストが境界を越えて新しい行に移動しました。





Safari 3.1.1

また、すべてにほとんど問題はありませんが、スライダーの端にはすでにセリフのより強力な不足があります。 これは最初の例で顕著であり、スライダーの約10％がスライダーで覆われていないため、Safariでは50個以上のセリフを作成することはお勧めしません。





この時点まで、テストプロセスは喜びでしたが、すべてが常に希望どおりに行われるわけではなく、レーキがどこかに出てきたに違いないという気持ちがゆっくりと忍び寄りました。 そして、彼らはクロールアウトしました...



Opera 9.27

どこから始めればいいのかわかりません。 原則として、セリフをオフにすると、スライダーの動きを描くときのアーティファクトを除いて、すべてがうまくいきます：スライダーの署名の場所に不快な黄色の痕跡があり、さらに、それらはクイックドラッグアンドドロップで常に消えるとは限らず、ストリップにいくつかの値が表示される場合があります Operaのセリフは、作成の原理が小数幅の使用に基づいているため、非常に災難です。 実験により、オペラは整数のパーセント値のみを認識することが示されています。 そして、彼女は500ピクセルの0.8％が4ピクセルであることを気にしません！ ピクセル（最初の例のように）。 まあ、それが丸められても、そうではありません-小数部分は単に破棄され、0％しか残りません。 私が理解しているように、彼女を納得させることは無意味です。したがって、例1と2では、このアプローチのすべてのパーセンテージではなくすべての喜びを観察します。 幅の値はほぼ全体であり、例のNo. 7のように値が1.02％と6.06％であり、それらが特別に選択されなかったため、残りの部分は幅の値がほぼ全体であり、天気の100分の1を遮断しなかったため、このような運命を回避することができましたUFOの陰謀G）。 彼女はまだ著しく遅くなり、役に立たないという事実について（依存関係は次のとおりです。セリフ付きのスライダーが多いほど、遅くなります）。
