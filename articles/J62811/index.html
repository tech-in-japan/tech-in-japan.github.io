Linux / UNIXとWindowsオペレーティングシステムの比較

はじめに




最近、Linuxユーザーが大量に流入しています。 原則として、これらは既にコンピューターとの通信でかなりの経験を積んでいる人ですが、ほとんどの場合、この経験は1つのシステムに限定されています。 当然、このシステムは、今日のデスクトップで最も一般的なMicrosoft MS Windowsオペレーティングシステムです。 多数のWindowsユーザーもLinuxをインストールするか、「Live CD」から実行して「表示」します。



そして、ここでは、新しいLinuxユーザーが自分の前に「別のWindows」を見ることを期待しているという事実に関連するいくつかの問題が一度にあります。 また、LinuxはWindowsのクローンではありません。まったく異なるシステムであり、異なる基盤、異なる伝統、他の機能、およびユーザーの他の要件を備えています。



私の意見では、このような誤解こそが、いわゆる「聖戦」と呼ばれる非常に多くの原因の一つです。 おそらく、この記事では、そのような戦争の回数を減らさなければ、少なくとも対戦相手の位置をより深く理解し、戦争の激しさを減らせます。



これら2つのオペレーティングシステムのデバイスを深く掘り下げる必要はないと思います。主にユーザーの観点からそれらを検討します。



歴史ツアー（非常に簡単）




比較のために、比較対象のオペレーティングシステムの簡単な履歴をブラッシュアップしても無害だと思います。



Unixの歴史




UNIXオペレーティングシステムは、商用ソフトウェアの時代よりも前に作成されました。 それは「自分たちのための」システムとしてエンジニアによって書かれました。 したがって、当時の高度な概念が組み込まれていました。 さらなる開発では、新しい機能を追加するときに、通常は「正しく」行う必要があると考えられていました。 つまり たとえば、エンジニアリングの観点から「間違った」2つのソリューションから選択する必要がある場合、たとえば、今日の生産性は向上しますが、将来的には困難をもたらす可能性があります。生産性の損失。



UNIXの最初のバージョンはアセンブラーで書かれ、その後システムはSIで書き直されました。 これにより、システムに独自の移植性がもたらされました。 PCでは、PCの開発、またはi386プロセッサでのPCのリリースにより、UNIXが移植されるか、むしろ書き直され（Linux）、これが可能になりました。



1985年、POSIXプロジェクトが開始されました。 これは、UNIXライクなOSインターフェースの標準です。 そのような標準が存在することが主な原因であるため、Linuxは非常に迅速に誕生し、UNIXの自由な実施形態である成熟に達する可能性があります。



最初から現在までのインターネットの開発は、UNIXを実行しているサーバーと密接に関連しています。 最初はコマーシャルで、そして今は無料でますます増えています。



商業化の観点から、UNIXの開発は3つの段階に分けることができます。



  1. 大学の非営利団体。
  2. 商用UNIXシステムの配布。
  3. 無料実装の出現（Linux、FreeBSD）および商用システムの混雑（現在）。




X Window Systemが登場する前は、UNIXはテキストベースのシステムでしたが、グラフィカルでありながら伝統的にテキストベースのインターフェースが追加されていました。



UNIXが最初からマルチタスクおよびマルチユーザーシステムであったことは非常に重要です。 つまり 複数のユーザーが1台のマシンで同時に作業でき、複数のプログラムを同時に実行できます。



すべてのUNIXライクなオペレーティングシステムの特徴は、信頼性を維持しています。

Windowsの履歴




Windowsオペレーティングシステムの起源は、同じ会社の以前のオペレーティングシステムであるDOSで探す必要があります。 すべてのMicrosoftオペレーティングシステムは、主に商用プロジェクトです。 特に商業的および技術的な特定のソリューションの起源を理解しようとする場合、これは常に覚えておく必要があります。



このファミリの最初のOSはDOSでした。 DOS自体は、議論中の主題に間接的に関連しているように見えるかもしれません。 しかし、多くの伝統、ユーザーと開発者ベース、彼らの習慣はそこから来ています。



DOSは、シングルタスク、シングルユーザーのテキストベースのオペレーティングシステムでした。 Windowsの最初のバージョンは仕事に適さないものであり、配布されませんでした。 バージョン3以降、Windowsで作業できるようになりました。WindowsFor Workgroups 3.1バージョンでは、ネットワークで作業する機会があります。 Winodws Series 3はDOSベースのシステムでした。 低信頼性が異なります。



1995年に、新しいバージョン-Windows 95がリリースされました。コードは、部分的に32ビット、部分的に16ビットの組み込みネットワークでした。 Windowsシリーズ3と比較すると、これは大きな前進でした。 信頼性は向上しましたが、UNIXライクなオペレーティングシステムの信頼性はまだ遠かったです。 もちろん、ワークステーションとしての信頼性は十分でしたが、サーバーとしては十分です。 その後、このラインの2つのオペレーティングシステム、Windows 98とWindows Meがリリースされました。 その後、ラインは閉鎖されました。



1993年に、新しいバージョンがリリースされました-Windows NT 3.1。 すでに完全に32ビットシステムでした。 それはゼロから開発され、有名な専門家がその開発のために雇われました。 新しい概念が導入されました。 これにより、信頼性はほぼUNIXライクなシステムの信頼性のレベルまで向上しました。 このOSはすでにサーバーとして機能している可能性があります。 このラインの継続、オペレーティングシステムWindows 2000、Windows XPおよびWindows Vista。



NTラインオペレーティングシステムはマルチタスクで、Windows XPから始まり、複数のユーザーが作業できる機能が登場しましたが、UNIXライクなOSよりも制限が多く、利便性がはるかに劣りました。

ユーザーの観点から見た技術デバイス




Unix




ユーザーの観点から見ると、UNIXは次のように構成されています。



  1. コア。 デバイスで動作し、メモリとプロセスを管理します。
  2. テキストサブシステム。ターミナルを介してシステムを操作します。 さらに、OSのすべての機能を制御するには、テキストサブシステムだけで十分です。 多くのユーザーがこのサブシステムを介してログインできます。 組み込みユーティリティとテキストベースのアプリケーションの両方の豊富なセット。
  3. Xwindowグラフィックス システム内のプロセスとして開始されます。
  4. テキストモードのリモートアクセスシステム。 テキストモードでOSを使用した本格的な作業を可能にします。 リソースをほとんど消費しません。 数十人と数百人のユーザーが比較的弱いコンピューターで同時に作業できるようにします。 セッションの数は、コンピューターのリソースによって制限されます。
  5. グラフィカルモードのリモートアクセスシステム。 複数のユーザーが同時にグラフィカルモードで作業できるようにします。 セッションの数は、コンピューターのリソースによって制限されます。
  6. グラフィカルアプリケーションウィンドウを別のコンピューターに転送するためのシステム。 この別のコンピューターに転送されたアプリケーションウィンドウを使用して、1つのコンピューターでアプリケーションを実行し、別のコンピューターから管理できます。 セッションの数は、コンピューターのリソースによって制限されます。




窓




  1. コア。 デバイスで動作し、メモリとプロセスを管理し、グラフィックスサブシステムを管理します。
  2. グラフィックサブシステム。 ユーザーインターフェイスを提供します。 ユーザーインターフェイスの優先システム。
  3. テキストサブシステム。 テキストユーザーインターフェイスを提供します。 テキストインターフェイスは完全に削除されています。 組み込みメーカーと他のメーカーの両方のテキストモードユーティリティのセットは非常に乏しいです。 テキストモードコマンドの構文と構成は、バージョンによって異なります。 グラフィックモードの上でのみ実行されます。
  4. リモートアクセスシステム。 システムに組み込まれたWindows NT Server 4.0で初めて登場しました。 それ以前は、他社の製品しかありませんでした。 本格的なグラフィカルセッションが開始されるため、多くのリソースを消費します。 リモートアクセスシステムの存在と同時セッションの数は、まったく存在しないか、商業上の理由で異なるバージョンで制限される場合があります。




コンセプト比較




次に、これら2つのシステムでの作業のアプローチがどのように異なるかを見てみましょう。



UNIX：ツールボックスの概念




UNIXはエンジニアおよびエンジニアのために開発されたため、ツールボックス（ツールボックス）の概念に基づいていました。 これはどういう意味ですか？ これは、UNIX用のソフトウェアおよび組み込みユーティリティを作成するときに、ユニバーサルプログラムを作成せず、それぞれがユーザーに必要なすべてのアクションを実行し、小さなタスクごとに、タスクを実行する独自のユーティリティが作成されたことを意味しますが、いいね ユーザーの業務は、これらのユーティリティのセットを使用して、必要な操作を実行することでした。



同時に、このユーティリティセットからアクションのチェーンとシーケンスをコンパイルできるため、頻繁に繰り返される操作を簡単に自動化できます。



ユーティリティが作業結果を相互に交換するために、情報媒体としてテキストファイルが選択されました。 ユーティリティ間で情報を交換するために、「パイプ」が発明されました。 「パイプ」の助けを借りて、1つのコマンドの出力からの情報を2番目のコマンドの入力に転送し、それを処理し、その情報を出力に渡し、3番目の入力に転送できます。



一般に、結果として、UNIXでは、ユーザーのコマンドとオフラインモードの両方で反復アクションを実行する単純なソフトウェアシステムをユーザーが簡単に作成できます。



このアプローチには、長所と短所の両方があります。 一方で、システムをより細かく制御し、構成の柔軟性を高めますが、同時に、システムに入るためのしきい値が増加するか、簡単に言えば、原則として、何かをする前に、基本を学ぶ必要があります。



Windows：トースターのコンセプト




Windowsは異なる概念に支配されています。 この概念は、ユーザーがタスクを簡単に入力できるようにすることです。 通常、Windowsのプログラムは大きく、アクションごとにメニュー項目またはアイコンがあります。 システムでは、プログラムは通常非常に困難に関連付けられています。



Windowsベースのコンプレックスの構築に関する状況は、ほとんどのプログラムが商用であり、独自のバイナリで通常閉じられたデータとファイル形式を使用しているという事実によって悪化しています。 このアプローチにより、コンピューターは、ソフトウェア製造業者による限られた機能セットを実行できるデバイスに変わり、最終的には製造業者が意図したことだけを実行する一種の「トースター」になります。



このアプローチの利点は、準備ができていないユーザーが入力しやすいことです。 マイナス面は、ユーザーは、軽さのように見えることでだまされ、何も学習したり、必要なアクションを実行したりしないことです。 ソフトウェアメーカーも準備中です。 これは、スペースでフォーマットされた大量のドキュメント、セキュリティの無視、およびその結果としてのウイルスの発生の理由の1つです。



おわりに




もちろん、どちらのシステムでも、そのアプローチは100％支配的ではありません。 Windowsでテキストコンソールを使用して.batファイルを作成する機会があるように、UNIXでも「トースター」アプローチにより固有のプロパティを持つプログラムの大規模なセットがあります。 しかし、アプローチには記載されている違いがあり、それは非常に顕著です。
