財務データを転送するためのプロトコル。 使用説明書

トレーダーになり、取引所で取引を開始する方法に関する多くの記事と「ガイド」が書かれています。 アルゴ取引、ロボット、オプション戦略-多くはこれらの用語を聞いています。 残念ながら、これらの「ロードマップ」のほとんどで、多くの人が直面している問題の技術的な側面、つまり取引戦略と対応するデータ転送プロトコルの選択について警告するのを忘れています。





どうする？

まず、貿易の方向の選択を決定し、技術スキルを評価します。

取引には主に4つのタイプがあります。

  1. 手動取引 （長期投資/少数のトランザクション）。 仲介システムの「クライアント」を使用します。 これには、特別な知識や高いコストはほとんど必要ありません。
  2. 高速接続を必要としない戦略によるアルゴリズム取引 。 このタイプの取引には、ブローカーからの自動化ツールの使用が含まれます。 彼らはお金のために安くなり、また特別なスキルを必要としません。
  3. 高速戦略によるアルゴリズム取引 。 このタイプの取引には、取引所のゲートウェイ上にある自作のロボットが含まれます（ブローカーのインフラストラクチャを介して、または取引所のサーバーに直接接続するリモート接続を介して）。 このオプションは最初の2つよりも高価であり、プログラミングのスキルが必要です。
  4. HFTトレーディング 。 このタイプの取引には、コロケーション、または交換ゲートウェイ上での独自の開発が含まれます。 3番目と同様に、かなり高価なオプションです。




1および2タイプ

最初と2番目のタイプの取引では、ブローカーとのみやり取りし、彼が提供するソフトウェアを使用します（したがって、ブローカー会社を選択するときはこの基準に特に注意する必要があります）。 これらのタイプを選択すると、Exchange自体が開発したソリューションは利用できなくなります。

ブローカーが提供するソフトウェア（ブローカーシステム、テクニカル分析システムなどのクライアントの場所）がPCまたはモバイルデバイスにインストールされ、ブローカーのサーバーへの接続（通常はインターネット経由）が行われます。

また、2番目の形式では、一部のブローカーは標準のクライアントソフトウェアに加えて、プログラムインターフェース（API）を介してシステムを接続し、クライアント独自の取引ソフトウェアを開発する機能を提供できます。



3および4タイプ

ここでは、顧客は取引所が提供する直接プロトコルを使用します。 テクノロジーとプロトコルの点では、これらの種類の取引は、サーバー、ソフトウェアコンポーネント、および接続チャネルの場所を除いて、互いに異なっていません。 一般的な接続方法、特に特定の交換プロトコルに対するブローカーのサポートを検討することが重要です。



接続方法について少し。 最も一般的な「ユニバーサル接続スキーム」を使用すると、ソフトウェアコンポーネントのレイアウトトポロジはどのようになりますか？ 証券会社のオフィス/データセンターには、オペレーターのL3を介した専用チャネルを介して交換インフラストラクチャに接続されているルーターがあります。 このルーターでは、3つのVLANが編成されています：DMZ（取引所取引ネットワークのDMZセグメント）、CLT（クライアントソフトウェアのネットワーク）、TST（テスト取引所サービスにアクセスするためのセグメントは、このコンテキストではそれほど重要ではないため、図には示されていません） 署名「VTS」が付いたアイコンは、開発したシステムです。 略語は「外部ソフトウェアおよびハードウェアツール」の略です。これは、交換インフラストラクチャに接続するソフトウェアの正式な用語です。 ユニバーサルコネクションの詳細については、スキーム1を参照してください。

スキーム1。





ユニバーサルスキームに従って接続するのとは対照的に、コロケーション上のロボットを使用したサーバーのレイアウトははるかに簡単です。

スキーム2。





簡単に言えば、（「ペーパーワーク」を除いて）行う必要があるのは、サーバーを取引所データセンターに配置することです（現在はVarshavkaのM1データセンターですが、DubrovkaのDataSpaceデータセンターへの移行は本格的です。 ） 1つのネットワークインターフェイスでは、サーバーは高速ロボット用に特別に割り当てられたアクセスサーバーのプールを確認し、2番目のインターフェイス-インターネットへのインターフェイス-は、ロボットにサービスを提供するためのいわゆる制御チャネルです。

ASTSブリッジ

名前と翻訳の小さな用語の混乱をすぐに明らかにします。交換ゲートウェイはブリッジです。 ゲートウェイは、ゲートウェイとしても翻訳したいと思っていますが、別の話です（ Habréに関する最初の出版物を参照 ）。 アクセスサーバーと呼びます。

ゲートウェイは、1998年以降に存在していたASTS取引および決済システムのネイティブプロトコルです（以前は、ソリューションはTEAP（TCP / IPバージョン）またはTEServer（RS-232バージョン、サポートされていません）と呼ばれていました。） DLLこのプロトコルの「ネイティブ性」により、その主な機能は、ASTS取引および清算システムで動作するすべての市場からのすべてのトランザクションとすべての市場データのサポートです。

このプロトコルの使用は、まず、データと操作のクリア（ポジション、義務、リスクパラメータの表示、さまざまな制限の設定、口座間での証券とお金の転送など）にアクセスする必要がある人にもお勧めです。交渉された取引モードでの入札への参加（つまり、「迅速」な匿名入札を「グラス」で行うのではなく、特定の取引相手との取引を直接締結する）。

APIは、WindowsおよびLinux用の32ビットおよび64ビットバージョンの動的ライブラリとして提供されます。

接続アーキテクチャは次のとおりです。動的ライブラリは、開発したソフトウェアのパッケージに分類されます。このパッケージは、ゲートウェイのいわゆるサーバー部分にネットワークアクセスできるPC /サーバーにインストールされます。 サーバー部分は、ブローカーに配置され、専用ネットワークチャネルを介して交換インフラストラクチャに接続されている一種のプロキシサーバーです。

上記のHFT取引の場合、ソフトウェアがコロケーション条件で取引所データセンターにインストールされると、ゲートウェイのサーバー部分の形式の中間リンクは不要になります-取引所ゲートウェイに直接接続します。

ゲートウェイプロトコルの興味深い機能は、「インターフェイス」のサポートです。 インターフェースは、適切な構造とデータ型を備えた、ユーザーが使用できるテーブルとトランザクションのバージョン付きコレクションです。 取引および清算システムが更新されるたびに、テーブル構造の変更またはトランザクション形式の変更を必要とするユーザーに新しい機会が現れます。 バージョン管理されたインターフェースの存在により、変更の準備ができていないユーザーは、インターフェースの古いバージョンにとどまり、ソフトウェアを変更することはできません。 現時点では、過去数年にわたって作成されたすべてのバージョンのインターフェース（10年！）と接続する機能をサポートしていますが、来年はさらに厳しくする予定です-以下を参照してください。



プラザII

現在、デリバティブ市場に接続するための主要なプロトコルは、Plaza IIプロトコルです。

このプロトコルを使用して接続するために、取引所はCGate APIを提供します。 それは何を与えますか？ 一方で、これにより、入札者は、セクションを制限する機能のクリア、商品の制限の設定、マーケットメーカーの義務の表示など、取引にアクセスするための本格的な機能を実装できます。 一方、これにより、入札者のクライアントは、最小限の機能セットで独自の高速ロボットを実装できます（アプリケーションの送信/アプリケーションの取り消し）。

APIは、WindowsおよびLinux用の32ビットおよび64ビットバージョンの動的ライブラリのセットとして提供されます。



接続アーキテクチャには2つのオプションがあります。

  1. PC /サーバーにインストールされているVTSは、ブローカーのサーバーに接続し、ブローカーのサーバーを介して交換インフラストラクチャに入ります。
  2. VTSPは（インターネット/専用通信チャネルまたはコロケーションゾーンからのアクセスを使用して）直接エクスチェンジのパブリックサーバーにアクセスし、ブローカーのインフラストラクチャをバイパスしてエクスチェンジと直接連携します。




デリバティブ市場のほぼすべてのリリースで、取引所は独自のプログラムコードを変更および改善し、APIとして顧客に送信します。 ユーザーにとっては、これは新しいバージョンのライブラリを含む新しいディストリビューションのように見えます。 コード自体に加えて、ユーザーに提供されるデータの構造はリリースとともに定期的に変更されます。 もちろん、ほとんどすべての場合、データ構造の後方互換性を維持し、ユーザーに新しい構造に適応するのに十分な時間を与えようとしています。



FIXおよびFAST

これらは標準の国際データ交換プロトコルです。 特にRunetのオープンスペースとHabréには、それらに関する多くの情報があります。したがって、簡単なレビューでそれらをオーバーロードすることはありません。 誰でもここで、たとえばここで読むことができます 。



誰が速いですか？

サポートチームで最も人気のある質問の1つ：誰が速いか-FIX / FASTまたはネイティブプロトコル。 ほとんどの場合、「ロボット」の作成者が尋ねます。 データ転送の速度と遅延について多くのことを話すことができます-これは記事全体のトピックです（注意してください！）。 したがって、現時点では、統計に制限されています。 測定の結果として得られた統計は、FIXプロトコルの存在の最初の1年半の外国為替市場で、アプリケーション数のシェアが今後2年間で92％に増加し、99％を超えたことを示しています。 株式市場では、FIXを介して受け取った申請のシェアは60％に達しますが、これは異なる構成と投資家の数によって説明されます。 コロケーションゾーンでは、ほとんどすべてのシステムがFIX / FASTを使用します。

デリバティブ市場の状況は少し異なります。ここではFIXはあまり人気がありません。 システム内のアーキテクチャ上の位置により、速度はネイティブプロトコルの速度よりわずかに劣りますが、遅延特性の改善に取り組んでいます。



ISS

「IIS」と混同しないでください！ :)このプロトコルは、トランザクションではなく、交換データの処理に関連するタスクのセグメントをカバーするため、一般的な列からはやや外れています。 本質的に、これはRestfulコンセプトに従って実装された交換WebサービスAPIです。 http / httpsプロトコルを使用して、相場、取引、インデックス、ボリューム、取引結果などの一般的な市場情報を取得することができます。 このサービスはインターネット経由でのみ利用できるため、データの受信の遅延を最小限に抑えることはできません。

このプロトコルは、ウェブサイトでの株価の表示（moex.comウェブサイト上のすべてのデータがそこから送信されることを含む）、分析のための取引結果のダウンロード、さまざまなデモパネルおよびディスプレイでのグラフの描画、およびインターネットを介したその他の作業に使用されますアプリケーション。 また、ISSを通じてすべてのユーザーが利用できる製品の中で、MOEX Trade Info情報端末とモバイルアプリケーションが動作します（この機会を利用して、iOS / Androidアプリケーションのすべてのユーザーに、これらのアプリケーションの現在のバージョンがすでに絶望的に古くなっていることを伝えたいと思います： ）しかし、すべての要望を考慮して、完全に新しいバージョンに取り組んでいます）。

ソフトウェアを開発およびテストする方法

交換プロトコルに関する情報、および開発目的での関連するすべての交換ソフトウェアの提供は、オープンで無料で、誰もがアクセスできます。 メインプロトコルの説明は私たちのウェブサイトで見つけることができます、そして、FTPに関するより特定の技術文書。

テストの目的で、多数の24時間テストシステムとテクニカルサポートがhelp@moex.comで利用でき、8：00〜24:00に機能します。

開発が完了したら、システムが接続を「戦闘」できるようにするために、認証プロセスを実行する必要があります。 ロギングを有効にしてテストシステムに接続し、いくつかの一般的な操作を実行することで構成されます。 このテストの実行中に、テクニカルサポートの従業員はAPIの使用の実装の正確性を確認します（ソースコードとソフトウェアは調べません。すべてはログのみに基づいています）。
