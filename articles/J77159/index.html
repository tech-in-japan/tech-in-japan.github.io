1.ハードウェアエミュレーター（Xen / HVM、VMWare、Qemu）に組み込まれたVNCサーバー


ハードウェアエミュレーション（VMWare、Qemu、HVMモードのXen）をサポートする多くの仮想化プラットフォームは、実際のデバイスの動作をシミュレートする仮想デバイス（グラフィックアダプター、キーボード、およびマウス）を仮想マシンに提供します。 仮想マシンのオペレーティングシステムは、違いを区別することなく、それらを通常のハードウェアのように扱います。 仮想マシンの動作を制御するプログラムは、これらのエミュレートされたデバイスを内蔵VNCサーバーに接続し、情報を仮想マシンの仮想アダプターにVNCクライアントにリダイレクトし、VNC-から受信したキーストロークに関する情報を仮想マシンのキーボードに転送できます顧客。 仮想マシンで実行されているオペレーティングシステムとプログラムの場合、VNCサーバーとVNCクライアントの存在は見えません。追加のドライバーや設定は必要ありません。仮想化プラットフォームをエミュレートする機器の標準ドライバーのみです。





1. FreeBSD OSが起動する瞬間に、HVMモードでXen仮想マシンのVNCサーバーに接続されたVNCクライアント。



Xenでは、 vfb構成パラメーターでvncタイプを指定することにより、仮想マシンのVNCサーバーを有効にします。 このパラメーターの追加オプション： vnclisten -VNCサーバーが接続を受け入れるIPアドレスを示します。 vncpasswd -VNCクライアントが提供する必要があるパスワード。



たとえば、仮想マシンの構成ファイルに次の行を追加します。



vfb = [ 'type=vnc,vnclisten=0.0.0.0,vncpasswd=qwerty' ]







使用可能なすべてのIPアドレスで接続を受け入れ、qwertyパスワード認証が必要なVNCサーバーを有効にすることを意味します。



仮想化プラットフォームのVNCサーバーを使用する主で包括的な利点：シンプルさ-仮想マシン設定でVNCサーバーサポートを有効にするだけです。



主な短所：機器エミュレーションのサイトでのリソースの非常に大きな消費と、この機能をサポートする限られた数の仮想化プラットフォーム。

2.動作中のXサーバーに接続されたVNCサーバー（GNOME vino）


このオプションは、通常のコンピューター（職場、自宅）のデスクトップへのリモートアクセスに最適です。 GNOMEのVNCを介したデスクトップへのアクセスは単純に含まれています 。 「他のユーザーに デスクトップの表示を許可する」を有効にし、 「 システム/環境設定/リモートデスクトップ」セクションのシステムメニューで他のユーザーにデスクトップオプションを制御できるようにします オンにすると、GNOMEに含まれるVNCサーバー（vino）が起動し、Xサーバーのディスプレイに関連付けられ、VNCクライアントの画面に表示される情報を完全に複製します。



しかし、仮想サーバーでは、事態はさらに複雑になります。 まず、エミュレートされた仮想マシングラフィックアダプターなどでXサーバーを何らかの方法で起動する必要があります。その後、上記のようにこのXサーバーのディスプレイへのVNCアクセスを許可することができます。 ハードウェアエミュレータ上のVNCサーバーと比較すると、1つだけあります。エミュレーションコストの削減です。 しかし、残りの欠点は残っており、新たな欠点が現れます-より時間がかかるセットアップです。 総合すると、このオプションは仮想サーバーの最悪の候補になります。

3. Xサーバーが統合されたVNCサーバー


多くのプログラムがあります-VNCサーバーは、アプリケーションのXサーバーとして独立して動作します。 実際のデバイス（グラフィックアダプター、キーボード、マウス）または仮想マシンでのエミュレーションで動作する標準のXサーバーとは異なり、VNCサーバーに統合されたXサーバーはこの機能部分を必要としません。ハードウェアではなく、VNC顧客によって。 VNCサーバーに組み込まれたXサーバーは、機器を使用するために複雑な構成やコンピューティングリソースを必要としないため、このオプションは仮想サーバーに最適です。



Ubuntu 9.10では、 vnc-server仮想パッケージは、 vnc4server （RealVNC）とtightvncserver （TightVNC）の2つの代替手段をVNCサーバーとして提供します。 それらの間に大きな違いはありません。それらの使用にはわずかなニュアンスしかありません。これについては後で説明します。 次に、vnc4serverを検討します。



原則として、VDSの最小OSインストールにはグラフィカル環境が含まれないため、GNOMEもインストールする必要があります。 RealVNCサーバーとGNOMEをインストールするには、次の2つのパッケージをインストールするだけです。



apt-get install vnc4server gnome-core







3.1手動VNCサーバー


デフォルト設定では、インストール直後に追加設定なしでVNCサーバーを使用できます。 コマンドラインから、ゼロディスプレイにXサーバーを作成するVNCサーバーを起動します（ポート5900でVNC接続を受け入れます）。



vncserver :0







作成されたゼロディスプレイで、GNOMEを起動します。



DISPLAY=:0 gnome-session







最初の起動時に、 vncserverは接続時にVNCクライアントに入力する必要があるパスワードの設定を求めます 。 VNCサーバーのパスワードを変更するには、 vncpasswdコマンドを使用します。



ただし、SSHを介してサーバーにアクセスし、VNCサーバーを手動で起動するたびに、このオプションは日常の作業には適していません。 VNCサーバーの自動起動を設定することをお勧めします。



3.2デーモンモードで実行されているVNCサーバー


再起動のたびにVNCサーバーを手動で起動しないように、起動手順に追加できます。 必要に応じて、新興企業向けの完全なスクリプトセットを作成できますが、実際には/etc/rc.localに次の行を追加してこれを行うだけで十分です。



su -l $USER -c "vncserver :0"







$ USERとして、デスクトップを起動するユーザーのログインを指定する必要があります。 必要に応じて、異なるディスプレイで1人または異なるユーザーに対して複数のVNCサーバーを同時に実行できます。



VNCクライアントが切断されると、デスクトップセッションと実行中のプログラムは終了しませんが、VNCクライアントが接続されたときと同じように動作し続けます。 VNCクライアントをこのディスプレイに再接続すると、ユーザーは切断されていないかのように同じデスクトップセッションになります。 複数のVNCクライアントが同じVNCサーバー（Xディスプレイ）に接続できますが、同じデスクトップで動作します。

3.3 inetdを介して起動されたVNCサーバー


VNCクライアントを接続する場合、オンデマンドでのみVNCサーバーを起動できます。 このため、5900 VNCサーバーのポートはサービスのためにinetdデーモンに転送されます。 VNCクライアントが接続されていない限り、VNCサーバーは起動しません。 クライアント側からの接続で要求を受信すると、inetdはVNCサーバーを起動し、クライアントをそれに接続します。 VNCサーバーはディスプレイを作成し、作業セッションを開始します。 クライアントを切断した後、VNCサーバーはセッションを閉じ、ディスプレイで動作するすべてのプログラムを終了します。



複数のクライアントが同じポートに接続されている場合、独自のディスプレイを持つ個別のVNCサーバーが起動されます。 したがって、接続されているすべてのクライアントは、独立した作業セッションを持ちます。 これにより、GDM経由の認証を使用して、マルチユーザー作業用にシステムを簡単に構成できます。接続後、仮想サーバーユーザーのユーザー名とパスワードが要求され、正しい入力後、このユーザーの作業セッションが開始されます。





2. VNCサーバーに接続するときのGDMを介したユーザー認証。



すでにインストールされているRealVNCおよびGNOMEサーバーに加えて、gdmおよびxinetdをインストールする必要があります。



apt-get install xinetd gdm-2.20







開始するには、GDMディスプレイマネージャーを構成する必要があります。XDMCPを介したユーザー認証要求の受け入れを許可し、gdmからのXサーバーの起動を無効にします。 パッチに従って、構成ファイル/etc/gdm/gdm.confを編集します。



--- /etc/gdm/gdm.conf.orig 2009-09-11 23:46:30.000000000 +0400

+++ /etc/gdm/gdm.conf 2009-11-30 02:36:06.000000000 +0300

@@ -311,3 +311,3 @@

# the security of XDMCP.

-Enable=false

+Enable=true

# Honor indirect queries, we run a chooser for these, and then redirect the

@@ -604,3 +604,3 @@

#

-0=Standard device=/dev/console

+#0=Standard device=/dev/console







または、ファイル/tmp/gdm.conf.patchにパッチを保存して、コマンドで使用します：



patch /etc/gdm/gdm.conf /tmp/gdm.conf.patch







xinetdがVNCサーバーを起動するには、次の内容の構成ファイル/etc/xinetd.d/xvncを作成します。 



 service xvnc { type = UNLISTED disable = no socket_type = stream protocol = tcp wait = no user = gdm server = /usr/bin/Xvnc server_args = -inetd -query 127.0.0.1 -once -SecurityTypes none -extension XFIXES port = 5900 }




Xvncの起動引数には、inetdデーモンを介して起動することをVNCサーバーに通知する-inetdと、VNCサーバーの起動時にローカルディスプレイマネージャー（GDMを介してユーザーを認証する必要があることを意味する-query 127.0.0.1 ） -onceオプションは、クライアントを切断した後、Xサーバーも無効にする必要があることを示します。 -SecurityTypes noneは、ユーザーがgdmを介して認証するため、VNCサーバーのネイティブ認証を無効にします。



xinetdおよびgdmコマンドを再起動します。



service gdm restart

service xinetd restart







VNCクライアントを起動し、サーバーに接続します。 システムユーザーのユーザー名とパスワードを入力し、標準デスクトップを取得します。

VNCサーバーの選択





Ubuntu 9.10では、標準ディストリビューションに2つのVNCサーバーがあります： vnc4server （RealVNC）とtightvncserver （TightVNC）。



サーバーとしてRealVNCを使用している一部のクライアントには問題があります。 たとえば、TightVNCおよびUltraVNCクライアントの多くのバージョンでは、このサーバーで圧縮アルゴリズムを使用できないため、インターネットの速度が低下します。 TightVNCサーバーには、クライアントにこのような問題はありません。 さらに、RealVNCとは異なり、TightVNCサーバーはJPEGを使用してデータを圧縮できるTightエンコードアルゴリズムをサポートしています。写真やその他の複雑な画像を表示する場合、これにより送信データ量が大幅に削減されます。 それ以外の場合、サーバー間に顕著な違いはありません。



TightVNCサーバー構成の違い


inetdを使用したVNCサーバーのインストールの説明では、RealVNCサーバーのすべての設定が提供されました。 TightVNCサーバーとして使用する場合、設定を少し変更する必要があります。



-SecurityTypes noneおよび-extension XFIXES引数はTightVNCサーバーには必要ないため、xinetdの/etc/xinetd.d/xvnc構成ファイルの引数の行は次のようになります。



server_args = -inetd -query 127.0.0.1 -once







GNOMEでキーボードの再マッピングを禁止することも必要です。そうしないと、キーストロークが正しく送信されません。 これを行うには、環境変数XKL_XMODMAP_DISABLEを設定します。 これを行う最も簡単な方法は、次の内容を含む/etc/X11/Xsession.d/10tightvncserverファイルを作成することです。



export XKL_XMODMAP_DISABLE=1







GNOMEセッションが開始されると、ファイルは自動的に実行されます。

デスクトップの調整


デスクトップ設定は、ネットワーク速度に大きな影響を与える可能性があります。 送信されるデータの量を減らすタスクは、画面上の変更を最小限に抑えることと、十分に圧縮可能な画像を使用することの2つのサブタスクに削減されます。



フォントスムージング


ユーザーが扱う情報の主な部分はテキストです。テキスト、メニュー、フォームなどのあるウィンドウです。 含まれるフォントスムージングは​​、テキストの表示時に送信されるデータ量の1.5〜3倍に増加します。これは、文字の境界のグレースケール領域に圧縮率の低い追加情報が含まれるためです。



GNOMEフォント設定ウィンドウでアンチエイリアスをオフにすることはできますが、この無効化はすべてのプログラムで機能するわけではありません-たとえば、Firefoxはフォントをスムーズにします。 したがって、システム構成ファイル/etc/fonts/fonts.confで設定を行うことをお勧めします（14ピクセル未満のすべてのフォントのアンチエイリアスは無効になっています）。 



 <match target="font"> <test name="size" compare="less"> <double>14</double> </test> <edit name="antialias" mode="assign"> <bool>false</bool> </edit> </match> <match target="font"> <test name="pixelsize" compare="less" qual="any"> <double>14</double> </test> <edit mode="assign" name="antialias"> <bool>false</bool> </edit> </match>

デスクトップの背景


デスクトップの背景には、写真や図面が含まれていると、速度が大幅に低下する可能性があります。 ウィンドウまたはメニューの最小化には、ブロックされた背景の部分の再描画が必要です。 また、画像が複雑になるほど、より多くのデータが送信されます。 バックグラウンドの帯域幅コストを最小限に抑えるには、単色のバックグラウンドを使用することをお勧めします。ホワイトが最適です。



デスクトップテーマ


インターフェイス要素の設計も、送信されるデータの量に影響を与える可能性があります。 トラフィックを最小限に抑えるには、グラデーションなしの単色のテーマを選択する必要があります。 ただし、垂直方向のグラデーションを使用するトピックの多くは非常に圧縮されており、速度を大幅に低下させることなく使用できます。



フォントサイズ


システムメニューおよびテキストフィールドのフォントサイズを小さくすると、テキスト領域の幾何学的サイズが小さくなり、その結果、送信されるデータ量が減少します。



VNCサーバーオプション


VNCサーバーには多くのオプションがありますが、次の3つに興味があります。 ジオメトリ -画面サイズ、 深度 -色深度、 deferupdate-変更の転送の遅延。



幾何学


ピクセル単位の画面サイズ。 形式： -geometry WIDTH x HEIGHT 。 たとえば、 -geometry 800x600は幅800ピクセル、高さ600ピクセルです。 画面サイズが大きいほど、送信する必要があるデータの量が多くなりますが、それらの間には直接的な関係はありません。 ディスプレイ上の情報を変更する場合、クライアントはディスプレイの画像全体ではなく、変更された情報が含まれる領域（開いているウィンドウ、開いているメニュー項目、または印刷された文字）のみを転送します。 したがって、1024x768の代わりに800x600を使用する場合、大きなゲインはありません。



深さ


ビット単位の色深度。 形式： -depth NUMBER_BIT 、NUMBER_BIT- 8、15、16 、または32。色深度が大きいほど、ハーフトーンとシェードはより良く表示されますが、より多くの情報を送信する必要があります。 色深度を減らすとデータ量が減り、それに応じて速度が上がります。 インターネット経由で作業するには、16ビットの色深度が非常に適しています。 低速チャネル（<256 Kbps）を使用する場合は、8ビットの色深度を使用することをお勧めします。



さまざまな色深度のエンコードされたチャネル1 Mbpsドロップダウンメニューの送信データ量と表示速度（エンコード：タイト）：

色深度 24ビット 16ビット 8ビット
ボリューム 15.2 Kb 13.8 Kb 9 kb
スピード 0.12秒 0.11秒 0.07秒




VNCクライアントは、サーバーで指定された色深度を無視しながら、8ビット（256色）の低減された色深度の使用を強制できます。



deferupdate


ディスプレイ上の変更の転送の遅延（ミリ秒単位）。 形式： -deferupdate NUMBER_MILLISECONDS 、デフォルトでは-40 。オプションで指定された時間間隔で行われた多数の小さな画面変更を単一の更新に結合できます。 同じ画面領域で発生した複数の変更をマージすると、送信されるデータの合計量が減少します。 このパラメーターを増やすと、対話性が損なわれます。アクションに対する視覚的な応答は、 deferupdateの有効期限が切れた後にのみVNCクライアントに表示されます 。 ネットワーク上でデータを送信する際の遅延を考えると、総反応遅延は作業の利便性を低下させる可能性があります。



仮想サーバーでのほとんどの操作は十分に迅速にレンダリングされ、 deferupdateの値はマイナス効果のために10ミリ秒に減らすことができます。 高帯域幅のネットワークでは、 -deferupdate 0を指定することで遅延を完全にオフにすることができます。



おわりに



inetdを介して起動されたVNCサーバーとチューニングに関する推奨事項を使用すると、エントリーレベルの仮想サーバーであっても、インターネット経由でデスクトップを操作できます。 合計すると、チューニングにより、標準設定と比較して2〜5倍の遅延を削減できます。 VNCサーバーをオンデマンドで起動すると、他のタスクのためにRAMを解放できます。



たとえば、このスキームに従ってアセンブルされたUbuntu 9.10 VNCプリセットは、1 Mbps ADSLチャネルを介して128 Mb RAM / 320 MHz CPUを備えたVDS上のデスクトップで許容可能な作業を提供します。 接続速度を5〜10 Mbit / sに、仮想サーバーの容量を512 Mb RAM / 960 MHz CPUに増やすと、快適さの点での作業はローカルコンピューターでの作業に近くなります。 ほとんどのインターフェイス操作の応答時間は、0.05〜0.2秒の範囲です。
