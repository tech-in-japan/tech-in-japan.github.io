批評：「Firefoxの設定：フリーズとメモリ消費の排除」

また、Firefoxを長年愛用していますが、トピック「 Firefoxの設定：凍結とメモリ消費の排除」の著者とは異なり、頻繁に凍結することはありません。 私の意見では、著者で起こっていることはナンセンスです。

さて、実際、私が言いたかったこと。 著者がキーだけを与えたが、それらの説明を与えなかったという事実に注意を喚起したかった。 明確ではありません：なぜこのキーまたはそのキーを1つまたは別の値に入れる必要があるのですか？ さらに、いくつかのキー値は不確かです。 そこで、ある種の明快さを作ることにしました。

だから...



  1. browser.sessionhistory.max_total_viewer = 0 。 この値は、訪問されたページがキャッシュされないという事実につながり、これは、たとえば、戻るをクリックすることによって再び戻ることにしたページのサーバーからのリロードにつながります。
  2. network.http.pipelining = trueおよびnetwork.http.proxy.pipelining = trueを使用すると、以前のリクエストへの応答を待たずにサーバーにリクエストを送信できますが、 network.http.pipelining.maxrequests = 30-このパラメーターは1から8までの値を取ることができるため、ナンセンスです（ http://kb.mozillazine.org/Network.http.pipelining.maxrequestsを参照）。
  3. nglayout.initialpaint.delay = 0-ブラウザーがデータの最初の部分を受信するとすぐにページの描画を開始することを意味します。 しかし、この最初の部分には、それほど有用な情報はないかもしれません。 このパラメータを変更する疑わしい必要性。
  4. network.dns.disableIPv6 = false-名前をIPv6アドレスに解決しようとします。 Linuxでは、この値はデフォルトで設定されています。 このような試行を無効にするには、おそらくtrueに設定する方が正しいでしょう。
  5. content.notify.backoffcount = 5-率直に言って、このパラメーターの説明を理解できませんでした。 結果のページの不要な再描画を防ぐようなもの。
  6. plugin.expose_full_path = true-ページの読み込み速度には影響しません！ そして、このパラメーターはこれを行います：about：pluginsで表示すると、この値は名前だけでなくフルパスでプラグインを表示します。
  7. ui.submenuDelay = 0-このパラメーターは、ブラウザーのサブメニューにカーソルを合わせてからサブメニューのコンテンツを表示するまでの遅延を制御するため、ページの読み込み速度には影響しません（ http://kb.mozillazine.org/Firefox_:_FAQs_:_About:config_Entries# UI..2A ）
  8. onfig.trim_on_minimize= true-このオプションは、ウィンドウが最小化されている間に、スワップでfirefoxが占有しているメモリからデータを消去できるようにします。 はい、この時間はメモリが解放されますが、これにより最小化状態からの長い回復につながる可能性があります。
