Luaには便利な機能があります。たとえば、論理演算が機能した最後の要素を返します。同様の構成を行うことができます。オブジェクトが存在する場合はアクセスし、存在しない場合はデフォルト値を指定します。 



local txt = a and a.text and a.text:lower() or 'default' -- : local txt = 'default' If a then if a.txt then txt = a.txt:lower() end end


彼らはコールバックを使用するときにしばしば間違いを犯します。 

 local function myFunc() print('Nya') end timer.performWithDelay(1000, myFunc(), 1)


この場合、関数はタイマーに渡されず、実行結果（nil）に渡されます。 関数名の後の括弧を削除するだけです。

一度、指定された時間の後だけでなく、今すぐにタイマーを機能を実行するように伝える方法をIRCに尋ねました。 答え-タイマーを作成したのと同じ場所のコードで呼び出してください。

LuaのOOP


OOPは、メタメソッド（PHPの魔法のメソッドの類似物）を使用して、擬似クラスに基づいて編成できます。 または、単純なテーブルに基づいて構築された非常に良いかもしれません。 さらに、プライベートでもパブリックでも、保護されていても、問題なく編成できます。

OOPコードを表示
空のオブジェクトを使用した最も簡単なオプション： 

 local function newObject() local object = {} return object end local myObject = newObject()


パブリック変数を追加します。 

 local function newObject() local object = {} object.somePublicVar = 100 return object end local myObject = newObject() print(myObject.somePublicVar)


次に、メソッドを追加します。 

 local function newObject() local object = {} object.somePublicVar = 100 function object:setName(name) self.name = name end return object end local myObject = newObject() myObject:setName('Nyashka') print(myObject.name)


コロンの使用に注意してください。 この場合、特別な変数selfが関数メソッド内に表示されます。これは、オブジェクト自体への参照です。 このようなメソッドは、コロンまたは1つのドットで呼び出す必要がありますが、最初の引数としてオブジェクトを指定します。



moneyなどのプライベート変数を追加します。 

 local function newObject() local object = {} object.somePublicVar = 100 local money = 0 function object:setName(name) self.name = name end function object:addMoney(amount) money = money + amount end return object end local myObject = newObject() myObject:setName('Nyashka') myObject:addMoney(50) print(myObject.money) -- nil


ポイントを介してお金にアクセスしようとすると、この変数はテーブルオブジェクト自体ではなく、このオブジェクトを作成した関数のスコープ内にあるため、nilになります。 したがって、この変数はaddMoney（）メソッドで表示されます。



継承： 

 local function newObject() local object = {} object.somePublicVar = 100 local money = 0 function object:setName(name) self.name = name end function object:addMoney(amount) money = money + amount end return object end local function newChild() local object = newObject() return object end local myChild = newChild()


この場合、オブジェクトは空のテーブル{}によってではなく、別のベースオブジェクトによって作成されます。 したがって、newChild関数では、オブジェクトのプロパティとメソッドを追加したり、それらを再定義したりできます。



Protectedはもう少し興味深い実装です。 すべての保護された変数を配置するプライベートテーブルを作成し、returnを介して作成されたオブジェクトと一緒に返します。 相続によってお金を転送したいとします。 

 local function newObject() local object = {} object.somePublicVar = 100 local protected = {money = 0} function object:setName(name) self.name = name end function object:addMoney(amount) protected.money = protected.money + amount end function object:getMoney() return protected.money end return object, protected end local function newChild() local object, protected = newObject() function object:spendMoney(amount) protected.money = protected.money - amount end return object end local myChild = newChild() myChild:addMoney(100) myChild:spendMoney(60) print(myChild:getMoney()) -- 40


メソッドをオーバーライドする必要がある場合は、まったく同じ方法で宣言されます。 新しいものは古いものを消去します。 ラッパーを作成する必要がある場合、古いメソッドが最初に変数に保存され、次に新しいメソッドから呼び出されます。 

 local function newObject() local object = {} object.somePublicVar = 100 local protected = {money = 0} function object:setName(name) self.name = name end function object:addMoney(amount) protected.money = protected.money + amount end function object:getMoney() return protected.money end return object, protected end local function newChild() local object, protected = newObject() function object:spendMoney(amount) protected.money = protected.money - amount end local parent_setName = object.setName function object:setName(name) name = 'Mr. ' .. name parent_setName(self, name) end return object end local myChild = newChild() myChild:setName('Nyashka') print(myChild.name) -- Mr. Nyashka


画面に表示されるオブジェクトの場合、グラフィックオブジェクトを返すCorona SDK APIから関数を呼び出してオブジェクトを作成するときに、空のテーブル{}を置き換えるだけで十分です。 display.newImage（）、display.newRect（）、またはほとんどの場合display.newGroup（）グループを使用できます。 グループを使用する場合、そのようなオブジェクトは他のグラフィック要素で非常に簡単に展開できます。 ただし、グループの数が多すぎるとパフォーマンスが低下するため、コードの速度が低下し始める場合は、できるだけグループを使用しないようにしてください。



モジュールはテーブルでもあり、オブジェクトでもあります。 OOPと同じ原則に従うことができます。 以前は、モジュールはキーワード/汎用モジュール（）を使用して記述されていました。 彼女には作成中のモジュールの名前が与えられ、スクリプト環境を自分のローカルに置き換えたため、「ローカル」の束を書く必要がなく、すべての変数と関数はこのローカル空間に置かれました。 そして、すべてがうまくいきますが、同時にモジュール内では、印刷などのグローバル機能にアクセスできませんでした。 その後、彼らは、モジュール（）関数の2番目の引数として特別な関数を追加すると、すべてのグローバルなものがどこにあるか（_G内）で検索されるというアイデアを思いつきました。 この特別な関数package.seeallがこれを担当しました。 しかし、実際には、すべてのグローバル関数と変数がすべてのモジュール内に表示されるだけでなく、モジュールはモジュール関数によって作成され、グローバル名前空間に配置されます。 おridge。 月のおpor。 グローバルムーンポリッジ。

Luaの速度はCの約20倍遅くなります。LuaJITは数倍遅いだけですが、iOSでのリアルタイムコンパイルは禁止されています。 ピカルカ。 Androidでコンパイルできますが、クラウンではこれは実装されていません。 幸いなことに、Luaがボトルネックになることはほとんどありません。 集中的な計算のみ。



開発速度は非常に重要です。 そして、そのためだけに王冠を許すことができます。



ローカルではないものはすべてグローバルです。 便宜上、変数をテーブルに保存します。

サービスの可視性テーブルで変数を検索するのに多くの時間が費やされます。 まず、現在のブロックまたは関数の可視性、次にチェーンを_G自体に移動します。 これは、グローバル変数の場合に発生します。 したがって、彼らは遅いです。 特定の場所でパフォーマンスを向上させるために、すべての変数と関数をローカルにキャッシュします。 しかし、無理をしないでください。 ほとんどのコードでは、このようなキャッシュを使用しても速度が目立って向上することはありません。 非常に頻繁に呼び出される場所のみを最適化します。 無限タイマー内の関数、またはフレームごとに呼び出されるイベントハンドラーenterFrameを指定できます。 Luaが現在のフレームのすべてのコードの処理を完了するまで、Coronaはレンダラーを起動しません。 パラメータxを変更した場合、関数内で画面上の画像を何度も変更すると、最後の値のみが画面に反映されます。



math.random、table.insert、ipars、pair、その他の組み込み関数などもキャッシュします。



グローバルな可視性の分野でごみを捨てないでください、何かを壊すことができます。 または、単にメモリリークが発生します。



_Gにあるすべてを再定義することは可能ですが、これは確かに推奨されません。 そこにあるものを見るには、その上にある印刷物を見てください。 表示された変数を覚えて、決して使用しないでください。 決して_Gに何も入れないと言う人もいます。 しかし、これは多すぎます。 モジュールのテーブル（名前空間、たとえば「app」など）、構成、または単に組み込みモジュールのように、各シーンに必要なものをたくさん書かなくてもよいように、いくつかの要素を自分で持つことはグローバルな可視性において非常に普通です。



パフォーマンスについて考えるとき、FPS、テクスチャメモリ、およびLuaメモリを監視するためのモジュールを追加します。 そのようなモジュールの追加後初めて、誰もが「そのようなメモリリークはどこから来たのか」パニックに陥りました。 実際、メモリ消費のスムーズな増加は標準であり、1秒あたり最大約10kBです。 ガベージコレクターが次に呼び出されると、メモリは最小値に戻ります。 ガベージコレクターは、アルゴリズムに煩わされる場合に構成できます。 たとえば、計算が遅くならないように、計算の完了後に自分で実行します。



実際のLuaメモリーは、数メガバイトを超えません。 主なボリュームはテクスチャです。 それらを最適化することが望ましい。 ビデオチップ内では、すべてのテクスチャ（任意の画像を意味する）は2の累乗で整列されます。 したがって、最良の結果を得るには、ファイルの2のべき乗を厳守してください。



コロナでは、1秒あたりのフレームレートが30または60に設定されています。常に60を使用するようにしてください。アプリケーションのインターフェイスがはるかに滑らかで見やすくなります。



すべてをアニメーション化します。 アプリケーションの設計で適切なルールを順守している場合、これを実現するのは非常に簡単です。 すべてにオブジェクトを使用します。 少なくともラッパーは、newImageなどの標準関数に対しても機能します。この場合、オブジェクトのジェネレーターでアニメーションを1回追加するだけで、アプリケーション全体にすぐに表示されます。 ただ、彼らと行き過ぎないでください。 ユーザーを待たせるという意味での輻輳または抑制は、アプリケーションの全体的なエクスペリエンスに悪影響を及ぼします。
