この量をサポートしていないマザーボードに16 GBのメモリを詰め込む方法

少し前に、コンピューターの1つに16 GBのメモリをインストールしました。 その上に、Core i5 750を搭載したマザーボードFoxconn P55MXがあります。この古いCPUを交換することは可能ですが、これまでのところは正常に動作し、必要なすべてを実行します。



ここに興味深いものがあります。 マザーボードは16 GBのRAMを公式にサポートしていません。 上記のページの仕様は、最大8 GBがサポートされていることを示しています。 ボードにはスロットが2つしかないので、マザーボードが発売された時点では8GBのストリップは珍しいものであると思われました。 とにかくやってみることにしました。 多くの場合、マザーボードはメーカーが公式に主張しているよりも多くのRAMをサポートしています。



最新のBIOSバージョン（バージョン946F1P06）がインストールされていることを確認し、8ギガバイトのトリムのうち2つを挿入しました。 その後、Ubuntu 16.04をダウンロードし、すべてが完全に機能しました。 ボードがドキュメントに記載されているよりも多くのメモリをサポートしているという私の理論は正しいことが判明し、それを忘れていました。 RAMを追加するのが好きで、ゲームが報われたことを嬉しく思いました。



しかし、数か月後、私はWindows 10をロードしようとしました。基本的に、コンピューターはLinuxで実行されます。 時々、何かをチェックするためにWindowsを起動する必要があります。 その時から楽しみが始まりました。



GRUBが表示されたら、メニューからWindows 10を選択し、Enterキーを押します。 Windowsブート画面が短時間表示された後、すぐに死のブルースクリーンに迎えられました。







停止コード：ACPI_BIOS_ERROR。 私は多くのことをグーグルで調べ、その本質はBIOSのACPIテーブルにある種の問題にあることを発見しました。 とりわけ、ACPIテーブルはオペレーティングシステムにハードウェアの構成方法を伝えます。 WindowsインストールUSBドライブから起動しようとすると、同じエラーが発生しました。 だから、フォックスコンは嘘をついていなかったと思う。 このコンピューターは、実際には16 GBのRAMをサポートしていません。 8 GBに戻すと、ダウンロードが成功しました。 RAMテストも優れていたため、メモリバーとしては悪くありません。



BIOSの修正のためにFoxconnサポートに連絡しようとしましたが、応答がありませんでした。 電子メールアドレスはWebサイトにリストされていますが、機能しません。 おそらくFoxconnはマザーボードを扱っていないでしょう。 彼らもサポートを提供していないようです。



この時点で、普通の人は単にあきらめ、8 GBのメモリを辞任するか、新しいコンピューターを購入します。 しかし、私はそんなに簡単にgiveめたくありませんでした。 理論的には、コンピューターはLinuxで完全に動作するため、16 GBを使用できることを知っていました。 そこで私はACPIを学び、BIOS設定を試し始めました。



いくつかのメモリ設定で遊ぶことができる興味深いBIOSセクションを見つけました。 パラメータの1つは「メモリリマップ機能」でした。 彼女は含まれていました。 BIOSのドキュメントでは、このオプションにより、共有物理メモリの上にマッピングするための「オーバーラップPCIメモリ」が許可されているとされています。 インターネットでの検索では、64ビットOSで起動するときに有効にする必要があることが示されました。 実験のためだけにオフにして、Windowsを実際に起動しました！ しかし、彼女は4 GB未満のRAMしか使用できないと言いました。 しかし、それは良かったです。メモリバーを物理的に取り外すことなくWindowsにアクセスできるようになりました。



Ubuntuも同じです。 メモリの再マッピング機能を無効にすると、RAMが4 GB未満に制限されました。 現時点では、何らかのメモリマッピングの問題があると確信していました。 ACPI_BIOS_ERRORエラーとその原因をより詳細に調査することにしました。ACPI_BIOS_ERRORエラーチェックを説明するこの優れた Microsoftドライバーデバッグドキュメントに出会いました。



この文書によると、以前のバージョンのWindowsではブルースクリーンに表示されていた4つのエラーパラメーターを見つける必要がありました。 Windows 10はデフォルトですべての情報を非表示にしますが、レジストリにエントリを追加することにより、追加のエラー情報の表示を再度有効にできます。 ここに、 superuser.comでの正しい答えがあります 。



それに応じてレジストリを編集した後、BIOSでメモリ再マッピング機能を再度オンにし、Windowsを起動しました。 BSODの左上隅に4つの追加コードが表示されるようになりました。







いいね！ したがって、パラメーター1は0x0000000000000002です。 Microsoftのドキュメントでは、0x02に等しいパラメーター1は、PCIルートバスのリソースのリストの処理に問題があることを示しています。 パラメーター2、3、および4はおかしな値のように見えるため、これらはおそらくポインターです。 そして、ここにポインタしかない場合、Microsoftは、問題はPCIデコード領域がE820 BIOSによって返されるメモリ領域のリストと重複することだと言います。



わかった 多くの情報がありますが、どこかで調査を開始できます。 E820 BIOSを呼び出すとメモリ領域に関する情報がどのように提供されるかという情報を見つけました。 その後、彼はLinuxに戻り、dmesgコマンドを使用して、E820とACPIに特に注意して、すべてのカーネル開始情報を調べました。 見つかったものは次のとおりです。

うん！ 競合の警告が表示されますか？ 私はそれに気づかなかったでしょうが、メモリをインストールした後、Linuxはブートのたびにこのメッセージを表示し始めました。 BIOSでメモリ再マッピング機能を無効にしてLinuxを起動しようとしました。 この場合、最後のe820領域が0x100000000から0x42fffffffffに消えたため、競合メッセージも消え、メインブリッジウィンドウが0x400000000から0xfffffffffになった別の「ルートバスリソース」がリストに表示されました。



それでどうなります。 Linuxは、競合に気付き、ACPIが提供する競合するPCI範囲を無視するため、16 GBで動作しますが、Windowsは嫌悪感を抱いてブルースクリーンを吐き出します。「BIOSに問題があります！」Windowsを責めることはできません。 実際、重複しているため、混乱していることがわかります。



現時点では、続行するかどうかはわかりませんでした。 0x400000000から0x42fffffffffffまでのメモリの最後の768 MBは、マザーボードがPCIに使用するメモリ領域の巨大な領域の先頭にマップされます。 明らかに、マザーボードがPCIを期待している場合、本当に悪いことが起こる可能性があります。 したがって、マザーボードは15.25 GBのRAMのみをサポートしていますか？



しかし... Linuxでは、この追加のPCIマッピング領域のサポートなしで、すべてがうまく機能します！ 大きなPCI範囲が0x400000000の代わりに0x430000000から始まるように、つまり物理RAMの終わりの直後にACPIテーブルを何らかの方法で変更するとどうなりますか。 その後、競合が解消され、PCIマッピングウィンドウのほとんどが引き続き使用可能になります。



呼び出しは受け入れられます。



ACPIテーブルを掘り始めました。 幸いなことに、Linuxはダンプを非常に簡単にします。 これには特別なツールがありますが、sysfsでテーブルを簡単に見つけることができます。 



  / sys /ファームウェア/ acpi /テーブル


ここにあります。 GRUBにACPIテーブルを新しいバージョンに置き換える機会があることも嬉しく思います。 したがって、どのテーブルが関係しているかを見つけた場合、GRUBを使用してこのテーブルの新しいバージョンをインストールできます。 理論的には、Windowsはこれに満足しています。



他のツールの中でも、私はiaslを使用してさまざまなACPIテーブルを解析し、置換する値0x400000000を見つけました。 ほとんどの場合、この値は逆バイト順（リトルエンディアン）でサイズは64ビットなので、 binwalkを実行してすべてのテーブルファイルを検索しました。 



  binwalk -R '\ x00 \ x00 \ x00 \ x00 \ x04 \ x00 \ x00 \ x00' *


OEMBテーブルには1つの結果がありました。 次の64ビットワードは0x1000000000であり、メインブリッジウィンドウの競合メッセージの終了アドレスよりもわずかに大きくなりました。 非常に有望な手がかり。 OEMBテーブルは、ACPI仕様による標準テーブルではないため、特別です。 Linuxは無効なチェックサムについて文句を言いますが、それは問題ではないと思います。 あなたは私が次に何をしたかを推測すると思います...



OEMBテーブルのコピーを作成し、バイト0x04の直前のバイト0x00を0x30に置き換えて、値を0x430000000に変更します（これは逆の順序であることを思い出してください）。 この変更されたコピーをファイル/boot/oemb.datに配置します。 その後、GRUBを使用してOEMBテーブルを私のコピーに置き換え、次のコマンドをブートコマンドのリストに一時的に貼り付けました（Ubuntuを選択した後、GRUBに文字「e」を入力します）。 



  acpi --exclude = OEMB /boot/oemb.dat


アイデアは、OEMBテーブルを除くすべてのACPIテーブルをロードし、次に/boot/oemb.datの内容をロードして追加のテーブルとして追加するようにGRUBに指示するというものです。 これにより、古いOEMBテーブルが新しいOEMBテーブルに効果的に置き換えられます。

GRUB, /etc/grub.d/00_patchbios : 



# This file patches the BIOS in my Foxconn P55MX motherboard to work
# properly when I have 16 GB of RAM installed. It's a nasty hack.
# Basically, the BIOS is hardcoded in the OEMB ACPI table
# to have a PCI address range from 0x400000000 to 0xfffffffff, but
# that overlaps with 16 GB of RAM being installed, because the RAM
# uses up (among other ranges) 0x100000000 to 0x42fffffff.
# This patch changes the table to actually list a PCI range of:
# 0x430000000 to 0xfffffffff
echo "write_byte 0xCF78E0B5 0x30"
