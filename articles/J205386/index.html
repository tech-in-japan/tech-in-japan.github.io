解決した問題



問題の最初のグループは製品の問題です。



なぜ人々は一般にクロスプラットフォーム開発に行くのですか？ 彼女は何のためですか？ プラットフォームごとに個別に記述した場合に発生する問題のリストを収集してみましょう。

  • 各プラットフォームのコードには単一のフローがありますか？ この問題は、さまざまなプラットフォームで作成しているチームに調整や関係がない場合に発生します。 人々は別のコードを見ずに、それぞれを書いているという事実から-自分自身。
  • APIのどこかに行ったことはありますか？ 同じ理由で発生しますが、もう1つ追加されます。変更に関する情報は、2番目ではなく1番目に到達しています。 さらに、APIのこのセクションがめったに使用されない場合、矛盾はまったく気付かない可能性があります。
  • DBスキームに従って同期されましたか？ ドキュメントの共通リポジトリがない場合、この問題の原因は広がります（ただし、前の例のように）。 たとえば、ローカルストレージを使用する各プログラムは、人々が流されて変更について他の人に通知することを忘れているという理由だけで、コミットから他のプラットフォームから分岐し始める可能性があります。
  • 各プラットフォームの単体テスト。 APIが変更されました??? これは大きな問題です。 ビジネスロジックがテスト（適切なプロジェクト内にある必要があります）でカバーされている場合、コードに加えてAPIまたはデータベーススキーマを変更する場合、関連するすべてのユニットテストをN回書き換える必要があります（Nはプラットフォームの数です）。


管理にも問題はありません：

  • Java / Objective-C / ...開発者からのVenegretを調整する必要があります 。 マネージャーが各プラットフォームを完全に理解していると便利です。 そして、彼が前に1つだけを扱ったことがあり、開発者が経験がない場合はどうなりますか？ そうすれば、彼らは耳で麺を放送できるようになります。たとえば、いくつかのことはできないと言います（実際には、解決策を探すのは面倒です）。 そして、顧客が開発者自身である場合、そのたびにあなたは彼らのふざけた態度を赤面しなければなりません。
  • 開発者がクラッシュしましたか？ 同じプラットフォームで見つける必要があります 。 たとえば、次のリリース用でなければ、これは問題になりません。 クロスプラットフォームツールセットを使用する場合、開発者を他のプロジェクトから移動できます。 少なくとも、プログラミング言語は使い慣れています。 これは大きな助けです。
  • 顧客は同じコードに対して何度も支払います。 同意します。3つのプラットフォームで同じことを推進するには、エスキモーに雪を販売するための巨大なスキルが必要です。 「APIとどの程度統合できますか？ あなたはすでにこれを二度やった！」 そして、あなたは静かにそれらを理解しますが、あなたのものを主張し続けます。
  • 外部APIは変更されましたか？ ここでは、経営陣は3つのプラットフォームに料金を支払ったばかりだったため、いらいらしました。今日から、変更に関連するすべてを書き直す必要があります。
  • 各プラットフォームについて非常に正確に推定する必要があるため、 修正価格は苦痛を増します。 また、マネージャーが一部のプラットフォームで料理をしていない場合、問題が発生します。評価できないプログラマーの評価に頼らなければなりません。 「Vasya、あなたは間違いなくPiを掛けましたか？」
  • 顧客は同じことを求めましたが、別のプラットフォームのみを求めました 。 そして、再び、上記の問題の全体の束。


リストされている問題は何を解決しますか？ すべての人々が単一のプラットフォームと最も均一なコードを使用する場合、それらは解決されます。 その後、すべての変更が一度だけ行われ、エラーが一度だけ修正されます。 そして、上記のポイントを通過すると、そのようなアプローチがそれらのほとんどすべてを減らすことを理解するでしょう。



問題の2番目のグループは開発ツールです。







あなたが何と言っても、開発はツールによって決定されます。 実際、便利で、リファクタリング、ルーチンテキストの入力の単純化、入力の自動補完、未知のAPIでの作業の支援が可能であれば、コードはより良く、より迅速に記述され、より安価になります。 これを理解することは、ライセンスの購入に貪欲ではなく、市場にある最高の開発ツールを使用するというアイデアにつながります。 多くの場合、プログラマーが開発ツール市場の新しいプレーヤーを選択できない有能なIDEがありません。 IDEは怖いですか？ キャラクター定義への移行は機能しませんか？ IDEが普及することはありません。 同意：クロスプラットフォーム開発への移行は、おそらくAPIを知らないことを意味します。 そして多くの場合、絶対に独自のAPIがあり、それを教える必要があります。 同時にIDEがコードのドキュメントのサポートを提供せず、メソッドやクラスにカーソルを合わせたときにツールチップにドキュメントを表示すると、インターネットアクセスは時間の浪費であり、意識の流れから抜け出すため、非常に困難になり、憎しみを感じるようになります。

Xamarinプラットフォームについて



それでは始めましょう。 少し歴史を掘り下げると、私が話しているKsamarinプラットフォームは決して新しいものではないことがわかります。 これは非常に古く、Microsoftの.Net Frameworkのクロスプラットフォーム実装に根ざしています。 この実装はモノランタイムと呼ばれ、グラフィック出力ライブラリを除き、.Netの世界のほぼすべてが含まれています（ただし、完全に作成されているわけではありません）。 ただし、このマイナスは、巨大なプラスによって簡単に相殺されます。デスクトップでは、どこでも機能します。 Windows、Linux、およびMac OSでは、完全にデバッグされます。







プラットフォームはソースコードとともに配布されており、無料なので、開発者はそこからお金を得ることができず、モバイルアプリケーションを使用して監視を開始することに決めました。 結局のところ、比較すると、デスクトップの場合、クロスプラットフォームアプリケーションの開発の注文を探す理由はそれほど多くないことがわかります。 しかし、モバイルデバイスの場合、プラットフォームの一部またはすべてのプラットフォーム用のアプリケーションを作成する注文がはるかに多くあります。 さらに、UIに関連するすべてを.Net Frameworkから削除すると、優れたアーキテクチャ、デバッグされたコード、および既存のIDE間の優れたサポートを備えた豊富なインフラストラクチャが得られます。

C＃を学ぶ価値があるのはなぜですか？



統計では、2012年のプログラミング言語の人気指数として、C＃が今年の言語と呼ばれています。 彼らは、C＃の人気が2.3％増加したと推定しました。これは、インデックス内のすべての言語の最大数でした。 もちろん、MicrosoftはWindows 8も起動しました。Windows8では、C＃が主要なプログラミング言語です。 しかし、実際には、すべてが多少異なります。 Xamarinの共同創設者であるNet Friedmanは、C＃がモバイルプラットフォームに最適なプログラミング言語である8つの理由を投稿で強調しました。









  1. 並列コード実行の第一級言語サポートにより、以前は退屈で厄介だったすべてが実装しやすくなり、膨大な数の潜在的なエラーが生じ、保守が容易なコードになりました。 そして、これに自動型推論、ラムダ式、LINQ言語、その他の言語の利点を追加すると、プログラミングは非常に快適なプロセスに変わります。







  2. 豊富な機能 -オブジェクト指向プログラミングとカプセル化により、最大限に再利用できるようにコードを非常に簡単に構築できます。 また、リフレクションや依存関係のインジェクションなどの機能は、アプリケーションの拡張性と柔軟性のための本当に豊富な機会を提供します。







  3. 高度なランタイム 。 ガベージコレクションは、開発者の負担からアプリケーションメモリを管理する負担を取り除くことにより、開発を大幅に簡素化します。 開発者は、ポインターに苦労してメモリリークを制御する代わりに、アルゴリズムやその他の問題の解決に集中できます。







  4. 強く型付けされた言語のように、C＃はほとんどのコンパイル型チェックを行います。これにより、迷惑なエラーの数が減り、開発が高速化されます。







  5. 簡単に適応できます。 C＃は非常に単純な言語であり、それに慣れるのは非常に簡単です。 しかし、あなたが慣れているなら、そこから抜け出すことは非常に難しいでしょう、信じてください。 そして、これに非常に多くのマテリアルを追加すると、初心者デベロッパーが1か所で動けなくなることはありません。







  6. 高速実行 。 iOSのC＃は、最適化LLVMコンパイラを使用します。これは、オペレーティングシステムのベースとなるCおよびC ++言語で使用され、C＃プログラミングの高い生産性と低レベルプログラミング言語の高効率を実現します。 Androidでは、事態はそれほど悪くはありません。C＃はJavaよりも優れています。 これはいくつかの方法で説明できます。C＃は、デフォルトでは仮想タイプではなく、Javaとは異なり、値タイプ、実際のメソッドをサポートします。 また、もちろん、これは、Dalvikが試行済みのMonoランタイムよりも若いという事実によって説明できます。







  7. プラットフォーム関数へのアクセスにより、オペレーティングシステムの機能だけでなく、元々特定のプラットフォーム用に作成されたサードパーティライブラリも使用できるようになり、長年にわたって開発されてきたサポートされているライブラリの備蓄を補います。







  8. そして主な理由： アプリケーションをさまざまなプラットフォームにエクスポートする機能 。 サポートされているすべてのプラットフォームを含むモノラルプラットフォームで作成すると、アプリケーションで世界中の22億台のデバイスに到達できます。 この数字について考えてみてください。 この数値だけでも、誰でも少なくともこのプラットフォームを開発中に試すことができます。
