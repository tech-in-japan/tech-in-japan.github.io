春の裏側

春のHabrは開いているようです。 「Why I Hate Spring」という記事の翻訳者とコメンテーターに感謝します。タイトルの強い否定的なメッセージにもかかわらず、多くの興味深い議論を引き起こしました。また、私の最後の記事2017年のSpringの書き方に答えてくれました 。 前回の記事へのコメントのおかげで、この記事が登場しました。







今回は、Springフレームワークの深intoに飛び込み、その魔法を公開し、基本的なWebアプリケーションが内部からどのように見えるかを確認し、どのようなタスクとSpring Bootがそれを解決するかを理解します。







前の記事へのコメントで、何人かの人々は、春のHello Worldの例はまだあまり示唆的ではないと指摘しました。 Springは、特にSpring Bootを使用すると、シンプルさと全能の感覚を与えますが、フレームワークの基本と内臓を理解していないと、ログのスタックトレースを取得する大きな危険につながります。 さて、何が起きているのかという完全な魔法の感覚を少しでも払拭するために、今日はアプリケーションを前の記事から取り上げ、フレームワーク内でどのように何が起こっているのか、ブートがブロックしている問題を分析します。 対象読者はまだ初心者の開発者ですが、JavaとSpringの経験と基本的な知識があります。 ただし、経験豊富なSpringユーザーにとっては、内部で何が起きているかについての知識を更新することは興味深いかもしれません。







キーコンセプト



ビン



Springの最も基本的な概念でカバーを破り始めましょう。 Beanは、最も一般的なオブジェクトにすぎません。 唯一の違いは、Springによって管理され、DIコンテナ内に存在するオブジェクトをBeanで呼び出すのが一般的であることです。 SpringのほとんどすべてはBeanです-サービス、コントローラー、リポジトリー。実際、アプリケーション全体はBeanのセットで構成されています。 それらは、登録、依存関係として受信、プロキシ、びしょ濡れなどにできます。







DIコンテナ



Springの重要かつ基本的なメカニズム。 外観は非常にシンプルですが、内部には依存関係を微調整するための多くのメカニズムが用意されています。 基本的に、Springアプリケーションは、DIコンテナを介して接続されたBeanのコレクションです。







多くの場合、Springについて議論するとき、軽量のDIコンテナ（Guiceなど）に簡単に置き換えることができ、同じようになりますが、より簡単でシンプルになるという議論があります。 そして、ここで理解することは非常に重要です-Spring DIの価値は、その可用性の事実ではなく、その基本的な性質にあります。 実際、Springエコシステムのすべてのライブラリは、このコンテナ（Spring自体を含む）にBeanを登録するだけです。依存関係の注入を通じて、アプリケーション開発者は必要なコンポーネントを取得できます。 簡単な例： Spring Security OAuthを使用する場合、 application.properties



でOAuthパラメーターを構成application.properties



と、Spring SecurityはOAuth2RestTemplate



を提供します。これをコードに挿入するだけです。 外部APIを呼び出すと、このBeanはOAuthトークンを取得する場所と方法、更新する方法、リクエストに追加する場所などを認識します。 したがって、ここでのDIの価値は、それが私たちのコードとSpring Securityの間の単なる通信メカニズムであるということです。 また、DI実装をGuiceに置き換えるだけでは、Spring Securityがそれを使用し始めることもありません。 また、この新しいDIですべてのSpringライブラリと統合されていない場合、その価値は大幅に低下します。







DIコンテナについて議論するときに多くの人が見逃すもう1つの非常に重要な点は、依存性注入を使用しても、 各コンポーネントのインターフェイスを作成することを意味しないことです。 これは非常に単純なアイデアですが、その単純さのために、必ずしも明白ではないことを何度も見てきました。 さらに、インターフェースが1つしか実装されていない場合、インターフェースを作成することは悪い習慣と見なされます。 すなわち クラス自体はDIオブジェクトである可能性があります。 さらに、インターフェースがなくても、テストで濡れるのを防ぐことはできません。 たとえば、Mockitoは、ウェットクラスを取得する方法を知っています。







コンテキスト



ApplicationContext



インターフェイスによって提示されます。 実際、Springアプリケーションそのものです。 コンテキストは、アプリケーション内で発生するさまざまなイベントに応答して、Beanのライフサイクルを制御する機能も提供します（たとえば、シングルトンとして作成するか、リクエストごとに作成します）。







構成



そのため、アプリケーションがBeanのセットである場合、それを機能させるには、このセットを記述する必要があります。







構成は、使用可能なBeanの単なる説明です。 Springは、アプリケーションを形成するBeanのセットを記述するためのいくつかのオプションを提供します。 履歴オプションは、一連のxmlファイルを使用します 。 最近では、 Javaアノテーションがそれに取って代わりました。 Spring Bootは注釈に基づいて構築されており、原則として、ほとんどの最新のライブラリは注釈を使用して構成することもできます。 第3世代では、 Bean構成は機能的なBean登録アプローチになりました。これは、近々登場するSpring 5の重要な新機能の1つです。







典型的な構成クラスは次のようになります。 







 @Configuration class PaymentsServiceConfiguration { @Bean public PaymentProvider paymentProvider() { return new PayPalPaymentProvider(); } @Bean public PaymentService paymentService(PaymentProvider paymentProvider) { return new PaymentService(paymentProvider); } }


この構成は2つのビンを定義し、2番目のビンは最初のビンに依存します。 そしてここで、SpringがPaymentProvider



ますPaymentProvider



インスタンスを要求すると、Springはコンテキスト内でそれを見つけて提供します。







設定は1つの巨大なファイルで記述する必要はありません。複数のファイルに分割し、 @Import



アノテーションを使用してそれらを結合できます。

簡単な要約



Spring Bootの「魔法」の中心には魔法のようなものはありません; Spring Frameworkの完全に基本的な概念を使用しています。 つまり、プロセスは次のように説明できます。







  1. @SpringBootApplication



    は、 @EnableAutoConfiguration



    アノテーションを介したコンポーネントのスキャンと自動構成が含まれます
  2. @EnableAutoConfiguration



    はEnableAutoConfigurationImportSelector



    クラスをインポートします
  3. EnableAutoConfigurationImportSelector



    は、 META-INF/spring.factories



    から構成のリストをロードしMETA-INF/spring.factories



  4. 各構成は、必要なBeanを登録し、さまざまな条件（ビンの有無/設定、クラスなど）を使用して、アプリケーションのさまざまな側面（Web、JPA、AMQPなど）を構成しようとします。

アクチュエータ



Spring Boot Actuatorは、実行中のアプリケーション用の強力な診断ツールであり、多くの有用な分析を提供できます（さらに、これらのメトリックのセットはアプリケーションから簡単に拡張できます）。

まとめ



Springは依然として大規模で最も簡単なフレームワークではありませんが、Springが提供する高レベルの抽象化の代価です。 また、日々の開発におけるフレームワークのすべての複雑さを知る必要はありませんが、内部からどのように機能するかを知ることは依然として有用です。 この記事が生態系としてのSpringの重要性と価値を理解する助けとなり、特にSpring Bootを使用している場合に起こっていることの「魔法」を少し取り除くことを願っています。 私のアドバイス-私の意見では、フレームワークの腸を掘り下げて、ソースコードとドキュメントをSpring-aでほとんど標準的であるので、恐れることはありません。







また、9月にリリースされようとしているSpring 5には 、単純なアプリケーションを作成し、「魔法」のレベルを下げることを目的としたいくつかの新しいコンセプトがあります（わかっているように、魔法はあまりありません）。 1つの概念はFunctional Bean Registrationです 。これにより、関数を使用してコンテキストでBeanを登録したり、Kotlinで適切なDSLを使用したりできます（Spring 5ではKotlinの多くの優れたサポートが追加されます）。 次に重要なのは、 Functional Web FrameworkとWebFlux（リアクティブWebフレームワーク）の組み合わせです。これにより、Spring MVCに依存せずにWebアプリケーションを作成し、サーブレットコンテナなしで実行できます。 アプリケーションは、アプリケーションコンテキストとDIなしで動作することができ、単純に一連のrequest -> response



関数として記述されます。 詳細については、 こちらをご覧ください （英語）。
