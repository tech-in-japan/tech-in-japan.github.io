サーバーレスCPaaS-Voximplantがこの誇大広告をどのように予測したか

今年、私たちはサーバーレスのコンセプトに関する堅実な記事を翻訳していました。著者は、それが何であり、なぜであるかを指で示しました。 また、ヨーロッパのパートナーが長い間、プラットフォームをサーバーレスCPaaSと名付けていることも知っています。これを明確に確認するため、CEOのAlexei Aylarovは10月16日にアムステルダムで開催されたAPI Daysカンファレンスで講演しました。 Alexeyは、Serverless CPaaSがまもなくユビキタスになる理由と、Voximplantが-突然-最初からこのアプローチを具体化したことを明らかにしました。 カットの下には、スピーチのテキスト適応があり、プレゼンテーションからの抜粋が添付されています。 ようこそ！









以前は優れていた（c）



CPaaSが導入される前は、企業は多くの準備作業を余儀なくされていました。まず、通信事業者を選択する必要がありました。 次に、バックエンドを選択し、これらすべてのインフラストラクチャを作成する必要があります（たとえば、アスタリスク/ FreeSWITCHソフトウェアPBXとファイルを取得します）。 このPBXが条件付きnode.jsのバックエンドを上げるのを支援するために...そして、その後、ビジネスロジックを実装する場所と方法を決定し、それを展開し、監視を構成し、作業中にクラッシュしないことを確認します。 ターンキーソリューションを開始するには、6か月以上かかる場合があります。







CPaaSにより、低レベルの手順をスキップしてすぐにビジネスロジックを開始することができました。このために、クライアントは通信プラットフォームを調査し、インフラストラクチャ、展開、監視を検討する必要がありました。 一方では、バックエンドがどうなるかを考えて、通信事業者を探す必要はありません。 このため、起動時間は品質を損なうことなく短縮されます。 一方で、その側でやるべきことが多く残っています。



ようこそサーバーレス









そして、サーバーレスが起こりました。 そして、ここで重要な点に焦点を当てる必要があります-なぜ少ないのですか？ 計算はどこかで行わなければなりません。

サーバーレスとは​​、サーバーがまったくないことではなく、クライアント側にサーバーがないことを意味します。
この概念には、クライアント側のコンピューティングプロバイダー+ devopsの束が含まれます。 クライアントは独自のインフラストラクチャを維持する必要はありません。計算のためにコンピューティングプロバイダーに支払うだけで十分です。これらのサーバーを24時間365日使用します。 インフラストラクチャの欠如は、このアプローチの2番目の重要な利点につながります。スケーラビリティについて考えることはできません。 プロバイダーは自動スケーリングを提供します。



開発者にとっては、ほとんどの場合、サーバーレスフレームワーク（ Fn Projectやサーバーレス -ああ、皮肉）を使用します。これにより、アプリケーションの開発とアセンブリが簡素化されます。 また、フレームワークはイベントのハンドルを提供できます。 サーバーレスは単なるイベントベースのコンセプトであるため、これは便利です（テレフォニーの例：電話が来た-これはイベントであり、電話に出た-これはイベントなど）。



サーバーレス機能をCPaaSに追加し、条件付きVoximplantを取得します。 その結果、「インフラストラクチャ」のステップがなくなります。ビジネスは特定のサーバーレスCPaaSを調査し、ビジネスロジックを実装し、ラック、サーバーの購入、サーバールームのスペースの確保などを気にせずに動作を静かに監視します。 もちろん、これは理想的なケースであり、各ソリューションは独特です。クライアントが何らかのハードウェアを必要とする可能性がありますが、Serverlessはクライアントにそのようなニーズがないようにあらゆる努力をします。



ユーザーエクスペリエンスの向上



サーバーレスプラットフォームはいわゆる 機能は、クライアントと他のサービスとの間の仲介者です（ FaaSコンセプト）。 たとえば、HTTP要求を受け入れて応答するように、またはHTTPを介してサードパーティのサービスとやり取りするように関数に指示できます。 ここでWebhookと通信固有の対話も処理できます。



ただし、このレイヤーには制限があります。

  • 限られたリードタイム;
  • 不変コンテキスト（ステートレス）;
  • 通信はHTTP経由であるため、コール処理はプラットフォームランタイムから離されます。






当初から、Voximplantアーキテクチャはクライアントがランタイムにアクセスできるようにしていたため、コール管理はドライHTTPではなくJSスクリプトを使用して行われます。 統合ランタイムには、いくつかの利点があります。



  • クラウドJSスクリプトは最新の言語標準ECMA2018をサポートしています。
  • スクリプトはネイティブプラットフォームAPIを使用します 。
  • リアルタイム制御：イベント処理と機能の実行は即座に行われます。
  • ブレークとステートでデバッガーを使用することができます。
  • 人の音声アラートなど、エラー処理に何でも添付できます。 イベントベースのエラー処理の例： 



    function onHttpRequestFailed() { call.say(“Unfortunately, we couldn't process your request, please try again later”, Language.US_ENGLISH_FEMALE) call.addEventListener(CallEvents.PlaybackFinished,(e) => { if (destroy) VoxEngine.terminate() else tryAgain() }) }


a）これは統一されたアプローチであるb）柔軟性を追加するため、イベントベースは特に重要です。 すべてのアクションはイベントとして解釈され、クラウドベースのJSスクリプトで処理されます。音声合成で着信コールに応答する、ボイスメールを認識して電話を切る、SIPにコールを転送する、せん断プッシュ通知など。









その結果、当社のサーバーレスプラットフォームは待ち時間を短縮し、ユーザーエクスペリエンスを向上させ、最終的に完成品の起動時間を可能な限り短縮します。「CPaaSの前の時代」で6か月だった場合、Voximplantで1か月を満たすことができます（すべての承認と会議を含む、即時の開発時間はさらに短くなります）。
