SWD.Starter：C＃+ Selenium WebDriver + PageObjectsのUIテストのクイックスタートオートメーション



この記事では、Selenium WebDriverおよびPageObjectsパターンとともに、C＃自動ユーザーインターフェイステストフレームワークのセットアップ方法を説明します。



オープンソースのスターターキット-SWD.Starter-は、10分で最初のテストを記述して実行するのに役立ちます。 さらに、優れたテスト自動化プラクティスに基づいたフレームワークアーキテクチャを提供します。

すべてのSWD.Starterコードは、ニーズに合わせて完全にカスタマイズできます。





SWD.Starterとは何ですか？



SWD.Starterは、テスト自動化フレームワークのスターターキットです。 すべてのソースコードはGitHubで入手できます： dzhariy / SWD.Starter 。プロジェクトライセンス（ライセンスなし）を使用すると、ソースコードを好きなように使用でき、少なくとも販売できます。



SWD.Starterは、Selenium WebDriverを介してユーザーインターフェイステストの作成と実行を開始するために必要なすべてのインフラストラクチャコードを含む既に構成されたプロジェクトです。



SWD.Starterは、PageObjectsパターンの使用を強くお勧めします。 また、このパターンを使用する場合、美しいアーキテクチャとコードの可読性を維持しながら、新しい自動テストコードを非常に高速に作成できます。



始めるために必要なもの



プロジェクトを開始するには、次のソフトウェアが必要です。



  1. Visual Studio Express 2013 Desktop Edition（理論的には、VS2010とVS2012もサポートされています）
  2. GithubからプロジェクトをポンピングするためのGit
  3. プロジェクトの公式ページからダウンロードできる追加のブラウザードライバーSelenium WebDriver


ソフトウェアをすばやく便利にインストールするために、Windows用のパッケージマネージャー-Chocolateyの使用をお勧めします。

メインページの指示に従って、cmd.exeを開き、コンソールウィンドウで次のコードを実行します。 



@powershell -NoProfile -ExecutionPolicy unrestricted -Command "iex ((new-object net.webclient).DownloadString('https://chocolatey.org/install.ps1'))" && SET PATH=%PATH%;%systemdrive%\chocolatey\bin


次に、同じコンソールウィンドウで、次のコマンドを実行します。

  • cinst VisualStudioExpress2013WindowsDesktop
  • cinst git


次に、コンソールウィンドウ（cmd.exeやFar Managerなど）で、SWD.Starterのクローンを作成するフォルダーを選択し、コマンドを実行します。 

 git clone https://github.com/dzhariy/SWD.Starter.git




これは必須の手順です。それ以外の場合、プロジェクトはコンパイルされません。

chromedriver.exeとIEDriverServer.exeをフォルダーSWD.Starter \ webdriversにコピーします



そして、これはクリーンな仮想マシン （ modern.ieからダウンロードしたもの ）への完全インストールのビデオです

ピンクのポケモン、ネオをフォローしてください！


念のため、ビデオでは、仮想マシン上のWindowsのアクティベーションが必要であることを示しています。

modern.ieのライセンス条件によると、私はそのような画像をテスト目的で合法的に使用する権利があります。 ユーザー同意書では、この場合はWindowsをライセンス認証しないでください。





PageObjectsとは何ですか、なぜ重要なのですか？



簡単に言えば、PageObjectsを使用した自動テストのアプローチでは、ページを操作するためのすべての低レベルコード（たとえば、要素の入力とクリック）を個別のクラスに配置するだけです。



これで、テストはページで直接動作せず、WebDriverの低レベルメソッドを呼び出しますが、各ページに固有の高レベル操作を使用します。



これにより、テストのコード行数が減り、コードが読みやすく、理解しやすく、信頼性が高くなります。

PageObjectsアプローチは、ボットスタイルに代わるものであり、WebDriverメソッドをテストから直接呼び出します。

最初は、ボットスタイルはPageObjectsを使用するよりもシンプルで理解しやすいようです。 しかし、これは自動化プロジェクトが崩壊する可能性のある大きな誤解です。



ボットスタイルを使用する場合、時間の経過とともにテストの数が増えると、テストのサポートにますます多くの時間を費やすことになります。 結果として、自動化フレームワークのサポートは経済的に実行可能ではなく、プロジェクト管理者は手動テストに戻ることを決定します。 また、すでに作成されたコードは、実際のテストアプリケーションに対応しなくなるため、破棄されるだけです。



ボットスタイルのテストは、並べ替えられていない巨大な本の山のようなものです。 「ヒープ」が10冊の本だけで構成されている場合、それほど困難なく把握できます。

しかし、本の数が100に増えたとき、何があなたを待っていますか？ 私を信じて、私はあなたをうらやましくはしません。 彼自身がすでにそれを経験したからといって。





一方、PageObjectsを使用する場合、すべての本を棚に並べ替えることができます。 書店や倉庫には、膨大な数の本が含まれています。 ただし、売り手は必要なものをすぐに見つけることができます。

PageObjectクラスは、Webページを操作するためのコードを便利に整理できる本棚です。 また、一般的なプログラミング言語とIDEは、オブジェクト指向プログラミングを使用する場合に、より多くの機能を提供します。

プロジェクト構造SWD.Starter



SWD.StarterのコアはSwd.Coreです。 次のような興味深いものが含まれています。

  • Swd.Core.Configurationクラス→Config。外部のConfig.configファイルからフレームワーク設定を読み取ります。 このファイル内で、起動したブラウザーを選択したり、設定を追加したりできます。
  • クラスSwd.Core.WebDriver-> SwdBrowser-はすでに言及されています。 ブラウザの寿命を制御します。 また、同じ名前空間内の近くには、ブラウザでの作業を簡素化する便利なメソッドとクラスがあります。
  • Swd.Core.Pages名前空間では、PageObjectsの基本クラスが存続します。




Swd.Coreには共通コードのみが配置されており、将来的には補助的なテストプロジェクトで拡張できます。



このようなテストプロジェクトの例は、DemoProjectです。

テストプロジェクトは、2つのメインサブプロジェクトで構成されています。



  • Demo.TestModel-PageObjectクラス、カスタマイズされた基本クラス、必要なデータ、アプリケーションロジックの断片、およびテスト対象の特定のアプリケーションに固有の他のライブラリ関数の宣言が含まれます。

    通常、テスト対象の単一のアプリケーションには、モデルライブラリが1つだけ存在する必要があります。
  • Demo.TestProjectは、テストスイートを含むプロジェクトです。 このようなテストプロジェクトがいくつかある場合があります。 たとえば、Demo.Tutorialもテスト付きのプロジェクトであり、Demo.TestProjectなどのモデルライブラリ（Demo.TestModel）も使用します。
  • Demo.Tutorial-Swd.Starterを操作するためのガイドを作成する試み。 まだ完了していませんが、今すぐファイル「Ch 00 Introduction.cs」を読んでテストを実行してみてください。
