Windows 8のアプリケーション開発：理由と方法

過去5〜7年にわたって、私たちは皆、アナリストがITの消費化と呼ぶ現象を観察してきました。 IT業界の特徴であるこの現象の歴史と詳細に立ち入ることなく、主なことに注目する価値があります。今日では、悪名高い付加価値の大部分を占めるのは「消費者」です-時には彼らの大きな数で、時には新しいアイテムや一般的に興味深いアイデアを過剰に支払う能力や欲求で。

90年代半ばの消費化をきっかけに普及したWindowsオペレーティングシステムは、コンピューティングのパーソナライゼーションが新たな強力な推進力である真のモビリティとアクセシビリティを獲得したゼロ中頃まで体系的に開発されました。



同時に、Appleの出願により、対応する画面の安価な生産のために、タッチインターフェイスは、ユーザーがタッチとジェスチャを使用してアプリケーションを操作し、指で画面に直接触れると非常に人気を博し始めました。



アプリケーション開発者の心に大きな影響を与えた別の重要な変化は、コンピューティングの全体的な動員の結果でした-アプリケーションの簡素化。 いいえ、人々は「複雑な」アプリケーションをあまり使用せず、パフォーマンスは低下しませんでした。 彼らは仕事やコンピューター（電話、タブレット、またはラップトップの形）で、ほとんど常に頻繁に、外出先で遊び始めました。 多くのアプリケーションが必要です。

しかし、環境とデバイスのサイズの結果として、これらのアプリケーションはすべて非常にシンプルになり、ほとんどの場合、コンテンツまたはゲームを消費することを目的としています。 言い換えれば、アプリケーションを構築するための需要を減らすことなく、 消費のためのアプリケーションに対する非常に大きな必要性が存在します。

もう1つの特徴は、アプリストアの人気です。 このシンプルなアイデアにより、2つの重要なタスクが解決されました。ユーザー-アプリケーションの入手先、開発者-販売場所。 アイデアのすべての単純さのために、長年、いわゆる「販売チャネル」は消費者アプリケーションの開発者にとって頭痛の種でした。 これは、どこで購入し、どこで販売するかについての直接の指示が開発者から購入者への道を最短にする場合に当てはまります。 そしてもちろん、個々の開発者や小さなチームが自分の作品を販売できるシンプルさは、現代のアプリケーションストアの成功の重要な要素になっています。 ちなみに、これはアプリストアと従来の小売店の数少ない違いの1つです。

今日、Visual Studio、AutoCAD、Photoshop、Excel、または1Cを使用して作業している私たちは、15〜20年前と同じソフトウェアおよびコンピューターメーカーの世界で同じタスクを実行します。 スマートフォンやタブレットを手に入れると、「スニーカーと軌道」、つまり売り上げが大幅に増加する商品、またはより多くの場合は付加価値の高い商品の購入者になります。

iPadが登場するずっと前から、Windows開発者は、3つのタスクが解決された新世代のフラッグシップオペレーティングシステムを開発するプロジェクトを開始しました。

1. 3つの方法でバッテリー寿命を延ばす-一般的な最適化、他のアーキテクチャ（ARM）への移植、最後に、アプリケーションが消費するリソースを最小限に抑える場合の開発者向けルールの開発。

2.タッチインターフェイスの最適化。

3.アプリケーションの販売とプロモーションのためのエコシステムを作成します。

その結果、新しいWindowsが結晶化し始め、Microsoftからのオペレーティングシステムのパラダイムにおける最も重要な変化であると同時に、最もリスクの高いものになると約束しました。 Windows 95以降、開発者とユーザーにこのような劇的な変更を提供していません。 また、Windowsが非常に普及していることを考えると、今日の変化は歴史上最大のものです。

同時に、Windows 8は開発者にとって歴史上最大の機会です。 実際、すべてのWindowsユーザーはWindowsストアを持っています。デスクトップ、ラップトップ、タブレットなど、コンピューターのメイン（場合によっては唯一）のアプリケーションソースであるアプリケーションストアです。 Windows 8エコノミーは、別の記事に適したトピックです。 今こそ、アプリケーション開発に移るときです。

メトロとは？



Windows 8のユーザーと開発者にとっての主な変更点は、一言で言えばMETROです。 この用語には、Windowsランタイム（WinRT）とMetro UIの2つの新しい概念が含まれます。

「メトロアプリケーション」とは、新しいインターフェイスだけでなく、新しい開発プラットフォームも意味します。 これは重要です。

メトロスタイルアプリとは何ですか？ 要するに、これは新しいインターフェースパラダイム、新しい効果的なAPI、適切な開発プラットフォームの融合に加えて、システムや他のアプリケーションとの統合です。

従来のアプリケーションとは異なり、Metroスタイルのアプリケーションにはクロムのないウィンドウが1つ含まれており、デフォルトでは画面全体を占めるため、ユーザーが不必要な詳細に気を取られることはありません。



Metroスタイルアプリケーションは、さまざまなレイアウトとビューをサポートして、さまざまな設計パラメーターとデバイス画面サイズで動的で便利なユーザーエクスペリエンスを提供します。





アプリでタッチを使用できますか？ ペン、キーボード、およびマウス入力



Metroスタイルアプリは、ペン、マウス、キーボード、タッチ入力など、さまざまな入力ソースとシームレスに連携します。 これらすべての入力ソースに対して、単一のイベントセットを使用できます。 Metroスタイルアプリには、タッチセンシティブユーザーインターフェイス要素が適切に機能するように、デフォルトのスタイルセットが含まれています。





アプリケーションは互いに対話できます。



アプリケーション契約は、ユーザーが複数のアプリケーションを簡単に検索し、それらの間でコンテンツを共有する機会です。 これらは、アプリケーションの実用的な有用性を拡張し、別のアプリケーションに保存されたデータまたは作成されたデータにアクセスする必要がある場合に、特定のアプリケーションのさまざまなAPI標準と対話する必要を排除します。 同時に、ユーザーは会社の環境に残ります。 ターゲットコントラクトの宣言されたサポート以外、ターゲットアプリケーションについて何も知る必要はありません。動作するだけです。

アプリケーションで利用可能な新しいコントロールとインターフェース領域



Metroスタイルのアプリケーションには、効果的なユーザーインタラクションの整理を容易にするいくつかの新しいコントロールが含まれています。 これらの要素の中で、アプリケーションのコマンドバー（アプリバー）とチャーム（チャーム）を強調表示できます。



アプリケーションコマンドライン（アプリバー）


アプリケーションコマンドラインは、アプリケーションウィンドウの外側にあり、そのメインコマンドインターフェイスです。 コマンドラインを使用して、ユーザー用のナビゲーション項目、コマンド、およびツールを配置します。 デフォルトでは、ユーザーが画面の上端または下端からの方向に指をスワイプすると、アプリケーションのコマンドラインが非表示になり、表示されます。 これは、アプリケーションのコンテンツのカバーであり、ユーザーが画面の端に沿って指をスワイプしたり、アプリケーションを操作したりすると、ユーザーが隠すことができます。





チャーム


チャームは、すべてのアプリケーションで検索、送信、接続、構成、および起動するために使用される定義済みの一貫したボタンのセットです。 これらは、すべてのユーザーが使用するほぼすべてのアプリケーションで実行したい主要な作業シナリオであると考えています。 ユーザーは次のことができます。

  • 自分または別のアプリケーションでホストされているコンテンツを検索します。 さらに、アプリケーション内のコンテンツを別のアプリケーションから検索することもできます。
  • アプリケーションのコンテンツを連絡先またはサービスと共有します。
  • ホーム画面に直接移動します 。
  • デバイスに接続し、コンテンツを送信し、メディアをストリーミングし、ドキュメントを印刷します。
  • 設定を使用して、必要に応じてアプリケーションを構成します。






アプリケーションはアイコンの代わりにタイルを使用します（タイルとアイコン）


ユーザーがアプリケーションをインストールすると、最初の画面にタイルとして表示されます。 タイルをタッチまたはクリックすると、アプリケーションが起動します。



アプリケーションは、タイルが機能していない場合でも、タイルを使用してコンテンツを提供できます。 動的タイル（ Live Tiles）を使用すると、アプリケーションは、最小限のバッテリー消費で簡潔な形式で有用なデータを表示できます。

アプリケーションは、動作するかどうかに関係なく、Webサービスの更新を定期的に要求するようにシステムを構成できます。 さらに、アプリケーションでWindowsプッシュ通知サービス（WNS）設定を使用して、Webサービスから動的タイルにメッセージを直接送信できます。

勝ち



最近まで、Windows開発者は、Win32 APIを介したネイティブと.NET Frameworkを介した管理という2つの主要なAPIグループを使用していました。 同時に、第2グループは徐々に開発され、ユーザーインターフェイスの作成、データとサービスの操作、ソースコードとアプリケーションアーキテクチャの構築などのさまざまなモデルを受け取りました。

一方、Windows自体-つまり、Win32 APIは、基本的な開発モデルの開発に対するそれほど大きな推進力を受けていません。 おそらく最後の重要な現象はCOMでした。 ここ数年、コンピューター自体は静止していませんでした。 あらゆる種類のセンサー、ネットワークデバイス（3G / LTEなどを含む）、カメラ、タッチセンシティブスクリーンがありました。 最後に、電力消費がますます重要になっています。

したがって、新しいWindowsを作成するとき、オペレーティングシステムのネイティブであるいくつかの最小要件と時代のトレンドを満たす新しいAPIを開発する必要があることに気付きました。 その結果、Windowsランタイム（WinRT）が誕生しました。

WinRTがもたらすもの：

1.これは、最適化されたCOM APIと連携するネイティブAPIです。 同時に、管理環境はWinRTの上で動作します（Windows 8の.NET Frameworkは一例です）。

2.これは、ジェネリック型メタデータ形式を含むオブジェクト指向APIです。

3. WinRTには、パーソナルコンピューター（センサー、カメラなど）の最新のプロパティの多くがサポートされています。 ここではエネルギーの節約が非常に重要です。

4.これは、さまざまなモデルをネイティブにサポートする、言語に依存しないAPIです。

a。 C ++

b。 C＃およびVisual Basicを使用した.NET Framework

c。 HTML5およびJavaScript

5. XAMLまたはHTML5でインターフェイスを開発するための最新の宣言モデルは、Windows APIの一部になりましたが、アドオンではありません。

その結果、Windowsアプリケーションは2つのグループに分けられました。

1.クラシックデスクトップアプリケーション。

2.新しいメトロアプリケーション。



WinRT自体は、特定の機能のパフォーマンスを担当する適切な機能に加えて、多くの重要なコンポーネントを受け取りました。



これらのコンポーネントは次のとおりです。

1.メタデータおよび言語予測のサブシステム。

そのおかげで、異なるプログラミング言語で書かれたコンポーネントは互いに呼び出すことができます。

2.ブローカー。 これらのコンポーネントには、2つの重要な機能があります。

a。 アプリケーション間のデータ交換のサポート。 たとえば、電子メールクライアントを介してアプリケーションからデータを送信する場合。

b。 機能と明示的な許可を通じていくつかの重要な呼び出しを保護します。 たとえば、アプリケーションがカメラを使用する場合、システムはまずユーザーに許可するかどうかを尋ねます。 同様に、たとえばGPSの場合、アプリケーションは開発段階でそのような必要性を示す必要があり、ユーザーは位置情報の使用を明示的に許可または拒否できます。

また、メトロアプリケーションで使用および使用できるWindows APIには、他にも2つの重要な部分があります。

1.アプリケーションモデルとWindowsストアAPI。

このグループは、アプリケーションの重要なコンポーネントを管理します。

a。 コンポーネントの構成。

b。 場所へのアクセス、ファイルシステム、ネットワーク、ユーザー識別、システムとのあらゆる種類の統合ポイントなど、システムのセキュリティと機能に影響を与える機能。

c。 Windowsストアとの相互作用およびアプリケーションに関する情報。これは、既に知っているように、ほとんどの場合、そこからインストールされます。

2. DirectXおよびC ++で開発されたコンポーネントとアプリケーションで利用可能なWin32およびCOM APIのサブセット 。

これらは、一方ではゲームおよびマルチメディアアプリケーション（DirectX）の開発者にとって、他方では既存のコード（Win32のサブセット）を移植するために必要です。

1.インストール。

2.実行。

3.取り外し。

1つ目と3つ目では、すべてが多かれ少なかれ明確になっていますが、Windowsストアでは追加の機会が提供されています。 ただし、実行サイクル自体には特別な注意が必要です。

アプリケーションの実行段階は、 ApplicationExecutionState列挙によって正確に記述されます。

WinRTのファイルシステムとアプリケーションデータ



WinRTのもう1つの機能は、ファイルとデータを操作する手順です。 まず、一般的なケースでは、メトロアプリケーションのデータは互いに分離されていると言えます。 これにより、アプリケーション自体とシステム全体の両方の安定性とセキュリティが確保されます。

これらのアプリケーションの詳細についてはこちらをご覧ください 。しかし、肝心な点は、すべてのメトロアプリケーションには3種類のファイルストレージがあることです。

1.ローカル。

2.一時的に。

3.ポータブル（ローミング）。

最初の種類には、名前が示すとおり、アプリケーションがローカルコンピューター上でこのアプリケーションに割り当てられた特別な領域に作成するファイルとフォルダーが含まれます。

2番目のタイプは、アプリケーションがアンロードされるとすぐに、そのような一時データをいつでも削除できることを除いて、最初のタイプと非常に似ています。

3番目のタイプのストレージはローミングです。これは、アプリケーションでデータを最初に作成した同じユーザーが登録されているWindows 8のすべてのデバイス間で同期されるファイルとフォルダーです。 ユーザーはMicrosoft IDで識別され、誰もがLiveIDとして認識しています。



ファイルシステムと権限


これはすべて、アプリケーションがデバイスの残りのディスク領域にあるファイルにアクセスできないという意味ではありません。 これには、 ファイルピッカーとライブラリの 2つのメカニズムがあります。

最初のメカニズムでは、ユーザーがファイルを選択する要求を介してファイルにアクセスできます。 彼はそれらを明示的に選択し、これらのファイルへのアクセスをアプリケーションに許可します。

2番目のメカニズムは、Windows ドキュメントライブラリファイルへの直接アクセスを提供します：ドキュメント、写真、ビデオなど。 このアプローチには2つの条件が必要です。

1.ライブラリにアクセスしたいという事実の宣言 。 宣言は、アプリケーション開発の段階で行われます。 Windowsストアからインストールする場合、ユーザーはアプリケーションがアクセスできるライブラリを確認します。

2.一部の種類のライブラリ（外部デバイス上のドキュメントとファイル） については、アプリケーションとファイルの種類との関連付けを決定する必要があります 。 そして、まさにこれらのタイプのファイルがアクセスを取得します。
