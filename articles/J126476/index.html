それは何を与えますか？ そして、かなり具体的な結果...フルスクリーンで展開するローカルブラウザを起動した後、クライアントは本質的にサーバーを忘れます。 Xとアプリケーションの両方がローカルで同時に実行されているため、サーバーOSは描画されません。つまり、データはネットワークを介して送信されません。 さらに、計算はすべてローカルで同じように行われ、サーバーは完全に不要です。 かっこいい うん！ これにより、通常の顧客を多かれ少なかれ彼らの弱い兄弟から分離することができます。 前者はそれ自体がリソースを提供し、後者はサーバーに寄生します。



通常のクライアントとほぼ同じクライアントがいることを受け入れると、タスクの一部を達成しました。サーバーの負荷が半分になるようにプロセスを最適化することができました。 結局のところ、通常のクライアントは、ブート時と認証中の短い間だけそれを必要とします。 もちろんボトルネックは残っていますが、これは多くの人々が来てクライアントの電源ボタンを押す朝です。 それについて何もする必要はありません。トラフィックの負荷が顕著になります。 しかし、これも比較的相対的です。 理由は次のとおりです。ブートプロセスは2つの段階に分けることができます。

1）PXEをダウンロード-eth1インターフェースから約8 Mbのトラフィックが必要です。 その後、少しの間休止し、初期化とマウント用のメインOSのイメージが要求されます

2）メインOSのダウンロードとは、カーネル、ライブラリ、デーモンの起動、その他のゴミの読み込みです。 この段階では、eth0インターフェイスを介してネットワーク経由で送信される約16 MBのデータが必要です。 これには、自動ログインと最初のアプリケーションの起動も含まれます。



それはどういうことですか？ ネットワークカードの速度が100 Mb / sの場合、これは12.5 MB / sです。 10に切り上げて、衝突などに関連するさまざまな欠陥を排除します。 次。 たとえば、3種類の端末があります。すべての端末が午前9時に同時にオンになった場合でも、わずかに異なる速度で読み込みを開始します。 各クライアントについて、ブート時に〜26 mbのデータを転送する必要があります。つまり、一般的にクライアントごとに〜3秒（もちろん、すぐではありません）、3 * 300を掛けます（懐中電灯から同時に量が増えます）または、1分または2つのクライアントを含む期間）900秒または900/60 = 15分、または900 * 26 = 23,400メガバイトのデータが出力されます。 かなり見た目が悪いですね。 一般的にはもちろんですが、そんなに言っているわけではありません。 結局のところ、2つのインターフェイスがあり、それぞれに独自のデータ量がポンピングされます-900 * 16 = eth0を介して14,400 mbおよびeth1を介して900 * 8 = 7,200 mb。 ネットワークの負荷は高いままですが、各インターフェイスの負荷は大幅に減少します。 残念ながら、とにかくここには非常にボトルネックが残っており、それを回避することはできないようです。 ギガビットインターフェイスをインストールするか、1つの端末のブート時間に依存するかのいずれかです。 これは約45秒に相当し、サーバーからデータをダウンロードするプロセス全体が広がります。

はい、はい、クラスターです！ ポイントです。 その開発中に、LTSPは、ロードバランサー、さらにはWebインターフェイスを備えたサーバークラスターを作成できるようになるまでに成長しました。 「UUUUUUは....」 - 厳しいシベリアロガーを語りました。 さて、LTSPクラスターをデプロイするための 技術的な説明とマニュアルを読みに行きました。



最初に、これらのltsp-clusterコンポーネントが何であるかを理解しましょう。 まず、LTSP-Cluster自体は、LTSP配布キットのデフォルトの配布に含まれるltspサーバー用のプラグインのセットです。 さらに、クラスタリングをサポートするクライアントを構築するには、他に何もインストールする必要はありません; --ltsp-clusterオプションでクライアントの構築を開始するだけです



プロジェクトには次のコンポーネントが含まれます。



  • Load Balancer Serverは、接続されているすべてのアプリケーションサーバーを監視するメインの負荷制御サーバーです。
  • コントロールセンター（Webインターフェイス）-接続されたクライアントとサーバーの負荷に関する情報を取得できるWebインターフェイス。 調整も行います。
  • GNU / Linux / Windows用のLoad Balancer Agent-各アプリケーションサーバーにインストールされ、サーバーの負荷に関する情報をLoad Balancer Serverに送信します
  • Accounts Manager-Load Balancer Agentと同じマシンにインストールされたデーモン。 必要に応じて、ユーザーを動的に作成および削除できます。 まあ、つまり、ユーザーのメインサーバーでltspclientを開始しましたが、アプリサーバーでは持っていないようです...そして自動ログインがあります...ここで、アカウントマネージャーは、サーバーの1つにアクセスするときにこのユーザーの自動作成を監視します。
  • PXE構成エディター-コントロールセンターからパラメーターを受け取り、それらに基づいてPXE構成ファイルを生成します。
  • NX Loadbalancer-SSHサーバーを起動し、nxユーザーの接続を監視し、必要に応じて特定のサーバーにスローします。
