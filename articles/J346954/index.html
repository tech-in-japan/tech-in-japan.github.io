重要！

残念ながら、現代の世界（特にロシア連邦）では、職務記述書は従業員の効率の向上にプラスの影響を与える強力なツールとはみなされていません。 状況は最初から行き詰まっています。法律は職務記述書の要件をどこにも規定しておらず、人事スペシャリストはそれらを書く時間/労力/欲求がありません（3つのうち2つを選択）、ラインマネージャーは「後で」延期します。 ほとんどの場合、裁判所は従業員に有利でない決定を下します（モスクワの司法慣行に依存しています）。

一般的な職務記述書の有無の問題に対する態度は、4つのグループで表現できます。







  1. 手順は次のとおりです。 このアプローチはめったにありませんが、一部の人事マネージャーと従業員は、次のように、各ポジションごとに開発された指示が必要であると確信しています。 それらがなければ、スタッフの効果的な仕事を組織することは非常に困難です。 同時に、多くの人は可用性が何をもたらすかを完全には理解していませんが、可能な限り高品質にするよう努めています。 最大15％の企業。
  2. 指示は必要ありません 。 さらに多くの人々は、人的資源管理において指示は何の役割も果たさないと考えています。 したがって、そのような会社では、特に法律がこれを要求していないため、職務記述はありません。 このアプローチは、企業の約30％で見られます。
  3. 指示が必要ですが、その品質は重要ではありません。 このアプローチでは、職務記述書は会社の従業員の実際の活動とは関係ありません。 合計の50％以上を自由に提供してください。
  4. 手順-これは管理ツールです -最も効果的なアプローチですが、適用される企業はごく少数で、全体の5％までです。 関連する指示が作成されるだけでなく、定期的にレビューおよび更新され、実際に積極的に適用されるという事実にあります。


一般に、職務記述書を作成するための2つのアプローチ、西洋とロシアについて話すことができます。 西洋のアプローチは、機能の説明が文字通り数行かかる非常に短い指示を書くことです。 同時に、機能ごとに個別の規制が作成され、詳細に説明されています。

ロシアのアプローチは、命令が機能を可能な限り完全に記述し、規制への追加の参照なしに記述されているという事実に表されています。







作成のアプローチに関係なく、職務記述書の目的は次のように定式化できます。







  1. 人事選択の段階と労働活動の過程の両方における労働契約の当事者の義務の明確な規制（労働関係の透明性）。

  3. 行動の指針であり、意見の相違が生じた場合の紛争解決の基礎となる（従業員と雇用主間、構造単位内の従業員間、構造単位間、法廷で）。
  4. 人事評価ツール、そしてもちろん企業のビジネスプロセス管理になります。

著者は、プロセスアプローチを唯一の真のアプローチとして課そうとはしていません。 ただし、会社の各機能ユニットはその責任の領域で活動を最適化し、最終的には会社の戦略目標を別のユニットの目標機能に置き換え、その結果、両方の別のユニットの開発を阻害する可能性があることを思い出してくださいと会社。

先に進む前に、プロセスを分類する必要があります。 一般に、分類で強調表示されたグループが将来使用される場合にのみ、何かを分類することは理にかなっており、その存在は人生を「単純化」します。 オブジェクトはさまざまな方法で分類できますが、常に一般的な特性を保持できることを常に覚えておく必要があります。







分類とは、 分類の基礎と呼ばれる共通の特徴に従って研究対象をグループ化することを意味します。













プロセスの分類の理由：







結果の支払い（ビジネスとサービス）
  • ビジネスとは、創造された価値（満たされたニーズ）が会社の外部の消費者によって（直接的または間接的に）支払われるプロセスです。
  • サービス -会社の内部で、消費者の利益のために実行されるプロセス。


階層レベル別（オーバープロセスおよびサブプロセス）
  • オーバープロセスは、より低いレベルのプロセス（現在のプロセスに比べてサブプロセス）の安定した集中システムであり、通常はプロセスのすべての要件を満たします。
  • サブプロセスは、 プロセスのすべての要件を満たすアクティビティの要素であり、より高いレベルのプロセス（現在のプロセスに対するオーバープロセス）の一部です。


抽象化レベル（メタプロセスおよびプロセスのユースケース）
  • メタプロセスは、アクションが実行され、ニーズのグループを満たすことを目的としたオブジェクトから抽象化した結果として得られる特定のタイプのプロセスのクラスです。
  • ユースケースは、プロセスのインスタンスです。


実行の境界（ローカルおよびエンドツーエンド（クロスファンクショナル））
  • ローカルプロセス -それが実行される利益のためにユニット内に完全に存在するプロセス、すなわち プロセスのすべての参加者は1つのユニットの従業員です。
  • 横断的（横断的）プロセス -複数の部門内で実行されるプロセス。 そのタスク（関数、サブプロセス）の一部は1つのユニットのフレームワーク内で解決され、一部は別のユニットのフレームワーク内で解決されます。


記載されている分類に基づいて、情報セキュリティを次のように定義できるようにするプロセスを定義します。
内部消費者の利益のために行われ、すべての部門に影響を与え、特定のオブジェクトに関連して特定の瞬間に特定の個人の利益のために行動する継続的なプロセス。これは高レベルのプロセスの一部です。

プロセスの主な特徴：







  • 目的性。 アクティビティの目標は消費者のニーズを満たすことであり、目標の達成度（有効性）とその費用（効率性）を評価する明確な基準があります。
  • ニーズと製造性の一貫性。 満たされたニーズは一度きりでも一意でもないため、プロセスは特定のステップ（テクノロジー）を実行することによって実行されます。
  • ニーズは、それを満足させようとした後にのみ変化し、時間内に変化することはありません。
  • 活動と責任の完全性。 このプロセスは、消費者のニーズが特定されてから満足するまで実行され、意思決定者が「from and to」のアクティビティを担当します。
  • 「内側から」の管理性。 有効性と効率性には責任があるため、プロセスにはプロセス参加者自身による管理決定が含まれます（必要です）。 さらに、プロセスのすべての参加者は、その有効性と効率性に動機付けられています。

この記事で説明されているアプローチは、次の利益団体のニーズを満たすことを目的としています。







人事部-プロセスのニーズに応じて、候補者の要件を迅速に形成するためのジョブファミリーの形成。

プロジェクト/プロセスオフィス-新しいプロジェクト/プロセスの実装に利用できる一連の能力の形成。

情報セキュリティ-役割モデルの形成。プロセスの役割を中心に、したがって情報の機密性のレベルを中心に形成されます。

情報技術-プロセス自動化の実装における構成セット（サーバー/ユーザー/ AWP）の形成、ライセンスおよび予算支出管理。

  1. 家族は、会社のプロセスの基礎となる役割と能力を結び付けます。 プロセスにおける役割の場所を理解すると、この役割がアクセスできる情報の機密性のレベルを決定できます-役割の能力と情報を効果的に適用する能力に基づきます。
  2. プロセス-役割-コンピテンシーのリストを実装するのは簡単です。 タスク追跡システムとその実装の助けを借りて、各役割の価値を判断し、この役割の価値とその価値を比較するのは簡単です。
  3. タスクの追跡と実行に基づいて、プロセスインスタンスの起動数、複数のプロセス起動のスーパープロセスの必要性、消費者による結果の支払いにつながったプロセス起動の数、そして結果としての利益を把握できます。
  4. 役割/実行者の主要業績評価指標（機能指標）は、役割/実行者が関与するプロセスパフォーマンス指標（ビジネス指標-収益性、収益性、資本化）に直接関連しています。
  5. （ビジネス）プロセスに関与する情報の機密性のレベルを決定します。したがって、情報システム/情報リソースにおいて、IS / IRの構成とコスト、役割ベースのアクセスモデル、ユーザーのAWPおよび職場の構成に直接影響します。


結論の代わりに









会社の構造化および文書化されたプロセス、ジョブの説明の説明とサポート、情報セキュリティの確保はすべて、会社の成熟度を示す要因です。 上の図に従って、0レベルが克服され、はるか後ろに残されます。

プロセス、組織管理、情報セキュリティの確保という3つの側面から、たった1つのことがドライバーになります-顧客満足を達成することは、機能の再現性、結果を時間通りに達成したいという欲求（「誰が速い？」与えられた品質で（「誰が優れているのか」という動機付けで）。







情報セキュリティのためにジョブの説明を更新することによる（ビジネス）プロセスのインベントリは、情報セキュリティを確保するための変更が定期的な作業を必要とする可能性があり、それが会社の一般的なレベルに影響を与える可能性があるオプションの1つを記述する試みです。 したがって、間接的ではあるが達成されたが、当社のプロセスにおける情報セキュリティ対策の統合。
