node.jsのプロトタイプ全体フレームワーク

node.jsでプログラミングしているが、コードでクエリルーティングを書くのにうんざりしている場合、ビジネス目的でグローバル変数を使用することに偏見はなく、過度の自由は大衆に有害であることに同意する場合、全体主義の週末グループはあなたのための代替プラットフォームのプロトタイプを用意しましたWebアプリケーション開発 。 全体主義スタイルは、requireを通じてアプリケーションにフレームワークを埋め込むことを意味するのではなく、フレームワーク構造にアプリケーションを埋め込むことを意味します。アプリケーションコードのフラグメントは、すべてのステップで追加の制限と課せられたコードおよびデータ構造を処理します。 ノードの開発者が主張するように、 専門家よりも少ない人が高性能システムを開発できるという事実について-さて、ここであなたはあなたが彼らがどんなシステム、特に非同期で、どのステップで失われたコールバックとメモリリークをするかを理解しています。 愚か者からの保護という点では、この奇跡のプロトタイプは実現しません。 そしてもちろん、私はあなたから多くの建設的な批判を期待しています。プロトタイプは原油なので、過去10年間にわたって私たちのチームから多くの概念的なアイデアを集めてきました。 Impressという名前でさえ2日前に登場しました、そして、はい-これは最も難しい質問です。



機能と機能



  • ファイルシステムに基づいたURLのルーティング -ディレクトリの作成-URLの一部になり、ディレクトリはすぐに親ディレクトリからすべてのハンドラーを継承します。ディレクトリ自体に配置されるものはすべて、実行ロジックまたはページの外観をオーバーライドします。 なぜなら 主にサーバーとの動的な相互作用を伴う単一ページのアプリケーションに焦点を当てています。ほとんどすべての場合、これらはすべてページの生成に必要ではなく、JSONに基づいたサーバーAPIをすばやく作成するために必要です。 要求処理シーケンスは、最初に要求されたディレクトリまたは親ディレクトリのチェーンでrequest.jsファイルを検索して実行し、アプリケーションルートを見つけるか、またはアプリケーションルートで停止します。次に、HTTP metadの名前に対応する名前のファイルを検索して実行します（get.js、post.jsなど）、テンプレートをレンダリングします。
  • RAMでのサーバー側のJavaScriptとテンプレートのキャッシュ -すべてのハンドラーとテンプレートは、最初にメモリにアクセスされ、繰り返し使用中にすばやく検索するためのインデックスが提供されます。
  • メインアプリケーションを再起動せずにアプリケーションコードを変更する機能 -ファイルが読み取られたすべてのディレクトリが監視され、ディスク上のコードとテンプレートを変更するとキャッシュが更新され、ファイルを削除するとキャッシュがクリアされます。
  • 複数のポート 、ネットワークインターフェイス、ホスト、プロトコルを一度に提供-as ステートフルアプリケーション、つまり 状態をランダムアクセスメモリ（ユーザーセッションとサブジェクト領域の他のオブジェクトの状態）に保存するため、異なるポート（たとえば、HTTPとSSE）で同じユーザーにサービスを提供するすべてのサーバーコンポーネントが共有メモリを持つことが非常に役立ちます。
  • いくつかの打ち上げ戦略：
    • 1つのプロセス （すべてが1つの処理スレッドで処理されます）。
    • 特殊なプロセス （各ポートには、独自のワーカー+メインプロセスマスターがあります）。
    • マルチスレッドの非特化処理 （類似のワーカー、非特化プロセス、マスターはすべてのポートからリクエストを受け入れ、ワーカー間でランダムに分散します）。
    • IPアドレスによる特定のワーカーへのクライアントバインディング （IPによるスティッキー）を使用したマルチスレッド処理。これにより、後続のクライアント要求は、そのワーカーに関連付けられ、セッションとそれに関連付けられた他の状態構造が既に展開されます。
    • アプリケーションによる分離を伴うプロセスの専門化 （リリースに含まれず、テスト済み）-複数のアプリケーションが同じシステムで実行されており、一方が他方に干渉する可能性がある場合は、それらを異なるワーカーに分離することをお勧めします。
    • 特定のURLを処理するための個々のプロセスの発芽（リリースに含まれず、開発中）-これは特に、長いブロッキングロジックを必要とするハンドラー向けです。これらの場合、一般的なリアクターを備えた回路は適切ではなく、ユーザーに「リクエストが処理されました」、ハンドラーをバディ、処理の最後に、IPCを介して親ワーカーに結果を返します。親ワーカーは、たとえばSSEを介して既に結果をクライアントに返します。
  • サブディレクトリ内のテンプレートフラグメントのサポート、継承、再定義を含む組み込みのシンプルなテンプレートエンジン 。 テンプレートエンジンは、ビジネスロジックプロセッサによって生成され、res.contextに配置されたデータにのみアクセスできます。 ここでEJSなどの既製のコードをねじ止めしなかった理由を尋ねると、そのコード、たとえばEJSで、たとえば一連の同期操作、その他の欠陥がありますが、AJAXアプリケーションには複雑なものは必要ありません。 クライアントで、規制されていないテンプレートエンジンを使用します。
  • Cookieおよびセッションのメカニズムのサポート -メモリ内の状態またはMongoDB内の保存状態を含み、ワーカーがクラッシュした場合、次のリクエストはデータベースから状態を発生させ、変更が行われた場合にのみ保存されます。 不明なセッションキーは拒否され、削除されます。
  • 静的コンテンツとストリーミングのリターン -改善すべき点はまだ多くあります。メモリ内の静的キャッシュと、このキャッシュの構成での管理はまもなく表示されます。
  • url書き換えをサポートするビルトインリクエストルーター （リバースプロキシ）-正規表現に従って、個々のURLを別のホストとポートにリダイレクトし、ユーザーからの応答を提供できます。
  • メインアプリケーションを完全に再起動せずに変更できる機能を備えたJSON形式の柔軟な構成 （現在でもクラッシュすることがあります）。
  • 単純なクエリログ-それを開発するか、ターンキーソリューションを固定する必要があります。
  • 組み込みのデータベースアクセス機能 -これまではデータベースとしてMongoのみでしたが、他のデータベースへの独立したアクセスは禁止されていません。

インストールとセットアップ



1. npm （ https://npmjs.org/package/impress ） から配置し ます 

$ npm install impress


2.プロジェクトテンプレートをコピーします -/ node_modules / impress / examples / copyContentToProjectFolderディレクトリの内容がプロジェクトディレクトリ（server.js、setup.js、config.js、sitesディレクトリ）に転送されます。



3. config.jsファイルを設定して、構成セクションを通過します。

データベース -起動時に自動的に開かれ、リストされたコレクションにアクセスできるデータベース：db.dbName.collectionName.find（...）。 設定例： 

 dbName: { url: "mongodb://localhost:27017/dbName", collections: ["collname1", "collname2"] }


セッション -名前は、セッションCookieを生成するための長さと文字セット、セッションを永続的に保存するためのベースの名前です。

クラスター -インスタンス戦略のセットアップ（マルチスレッドタイプ）。

サーバー -名前付きサーバー（インターフェース/ポート）、各ホストフィールド-このサーバーから指定する必要がある名前付きホストの配列、静的-静的をレンダリングするためのファイルマスクの配列、たとえば["/ css / *"、 "/ images / *" 、「/ js / *」、「/ favicon.ico」、「/ index.html」]。

hosts-名前付きホスト（仮想ホスト）。たとえば、「* .myhost.com」のように、命名にマスクを使用できます。

routes-名前付きリクエスト転送ルート。



4. MongoDBでデータ構造を初期化するには、ノードsetup.jsを実行して「y」を押します。



5. server.jsファイルに、追加の初期化コードを記述できます。 

 require('impress'); impress.init(function() { //    });


6.サーバーを起動します 

 $ node server.js
