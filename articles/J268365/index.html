更新する



コレクター自体とそのプラグインの両方を更新する必要があります。 設定ページは127.0.0.1-00-00242 / manage （設定に合わせて調整）にあります。 新しいバージョンがある場合、更新提案のあるボタンがページにハングアップします。 自分で最新バージョンをダウンロードし、コレクターを慎重にオフにし、 jenkins.warファイルを置き換えて再起動することをお勧めします。

jenkins.warを更新するときにエラーが発生する可能性があるため、 念のため古いファイルをバックアップします。 なんで？ 最初に、ファイルは日本のサーバーに表示され（コレクターの作成者は通常の日本人エンジニアです。詳細はこちら ）、次にミラーを介して配布されます。 参照：



  • 最新バージョンへのリンク： http : //mirrors.jenkins-ci.org/war/latest/jenkins.war
  • 完全なアーカイブ： http : //mirrors.jenkins-ci.org/war/
  • Yandex Mirror： http : //mirror.yandex.ru/mirrors/jenkins/
  • 日本語のリポジトリ、これは最新のものです（許容範囲はありますが）： http : //ftp.tsukuba.wide.ad.jp/software/jenkins/


プラグインは127.0.0.1-00-00242 / pluginManager /で構成できます。 プラグインについての詳細は対応するセクションにありますが、既存のプラグインを更新することをお勧めします。



コレクターの初期セットアップ



Unityプロジェクトは並行して収集されません。
しかし
YoungSkipperはそれが可能であると書いています。 ドキュメントを読んだ後、試してみたいという要望がありました。次の記事で変更点を説明し、次に段落を編集します。

つまり、一度に1つのコンピューターで起動できるアセンブリは1つだけです。 デフォルトでは、 Jenkinsは複数のエグゼキューターで単一のコレクターを作成しますが、これは良くありません。 グラフィカルインターフェイスを使用するか、設定で直接127.0.0.1-00-00242 / computer /（master）/に進み、「 実行プロセス数 」を1に変更します。



必要に応じて、他のマシンでアセンブリを上げることができます（たとえば、アップルアセンブリをここに残し、他のプラットフォーム用に他のWindowsプラットフォームに転送します）。 ここではこの可能性を考慮しません。



ここで-127.0.0.1-00-00242 / configureは一般設定セクションです。 以下のパラメーターを変更します。



  1. ホームディレクトリが正しいことを確認します。 たとえば、 / Users / jenkins / WORK / JENKINS_HOME
  2. 詳細設定を展開します。 ワークスペースルートディレクトリは、 Unityプロジェクトをホストする場所です。 「 アセンブリレコードのルートディレクトリ 」は、収集されたビルドのリポジトリです。 JENKINS_HOMEから両方のフォルダーを削除して 、デフォルト値を変更することをお勧めします。 たとえば、 / Users / jenkins / WORK / JENKINS_WORKSPACE / $ {ITEM_FULL_NAME}および/ Users / jenkins / WORK / JENKINS_BUILDS / $ {ITEM_FULL_NAME}にあります。

  3. 前と同様に、「 コレクターの数 」を1に設定します。
  4. 「 組立前の遅延 」は、たとえば60秒に設定されます。 誤ったビルドまたは誤って起動されたビルドを元に戻すのに役立ちます。
  5. Android向けのビルドでは、異なるJDKが必要になる場合があります。 対応するセクションで、名前とパスを規定します（無人インストールのチェックを外します）。 たとえば、 jdk_1.8.0_60の JAVA_HOMEは/Library/Java/JavaVirtualMachines/jdk1.8.0_60.jdk/Contents/Homeにあります 。 このアイテムはオプションであり、アセンブリ中に異なるJDKを使用したいという強い要望があります。
  6. [ Jenkins Location]セクションで、コレクターのアドレス（ローカルネットワーク上のhttp：//192.168.3.69-00-00242/など ）と管理者のメールを指定できます。 一部のプラグインに必要です。

  7. Subversionセクションでは、最新のプラグインの更新により、 SVNの最新バージョンを選択できます。 たとえば、 1.8です。 バージョンは、システムにインストールされているバージョン（最初の2文字まで）と一致する必要があります。一致しない場合、アセンブラーによって自動的に作成されたプロジェクトを手動で編集することはできません。 前の記事の必要なバージョンのインストールへのリンクを参照してください 。




Jenkins CLI



コレクターのリモートコマンドのリストについても言及する必要があります。 ここに必要な情報はすべてhttp ： //127.0.0.1-00-00242/cli/です。 Jenkins-cli.jarがダウンロードされ、たとえば次のコマンドが含まれます。 



java -jar jenkins-cli.jar -s http://192.168.3.69:4242/ safe-shutdown


ローカルネットワーク上の任意のコンピューターからコンソールからコレクターを切り離すことができます（実行中のすべてのアセンブリが最初に自動的に完了します）。 必要に応じて、適切なスクリプトを準備できますが、ハングしたコレクターでは動作しない場合があります。



プラグイン



コレクターの追加機能は、プラグインを通じて提供されます。 対応するセクション-127.0.0.1-00-00242 / pluginManager / installedでそれらを構成（インストール、更新、ロールバック、削除）できます。 完全なリストは近くにあります 。公式wikiはこちらです。 重要なものは以下にあります（残りは更新する方が良いですが、直接作業することはほとんどありません）：



  • 資格情報プラグイン -パスワードを保存できます。 たとえば、 SVNリポジトリの場合。 コレクターの設定-127.0.0.1-00-00242 / credential -store /で、 「グローバル資格情報」を選択し、各リポジトリーの名前とパスワードのリンクを作成します。 パラメーター- ユーザー名とパスワード 、およびGlobal 。

  • Subversionプラグイン -SVNサポート。 すでに設定されているはずです。




アセンブリパラメータを設定するためにダウンロードする必要があるプラグイン（関連プラグインをインストールするプラグインもあります）：



  • 条件付きビルドステップ -アセンブリ中にエラーが発生した場合のコレクターのアクションを構成します。 たとえば、ログ付きのメッセージを管理者に送信したり、スクリプトを実行したりできます。
  • description setterプラグイン -ビルドパラメーターに基づいて、アセンブルされたビルドの説明を作成します。
  • 環境スクリプトプラグイン -アセンブリの内部変数（パラメーター）の宣言。
  • Extensible Choice Parameterプラグイン -アセンブリパラメーターを別のファイルとして、たとえば同じSVNに保存し、リポジトリからの更新後、最新バージョンの設定が適用されます。
  • Groovy Postbuild-ビルドの成功を処理する簡単なスクリプトを作成します。





コレクターの外観を変更し、アクセスを簡素化するプラグイン：



  • Extra Columns Plugin-タスクリストの追加列。 タスクの停止、開始などのためのボタンが含まれています。
  • Safe Restart Plugin-グラフィカルインターフェイスでコレクタを直接再起動するボタン。
  • カスタムジョブアイコンプラグイン -各タスクに特定のアイコンを割り当てます。 たとえば、アセンブリを配置する目的のアプリケーションまたはプラットフォームのアイコンを配置できます。 設定に、 カスタムアイコンアイテムが表示され、そこでアイコンをアップロードできます。 それらは/ Users / jenkins / WORK / JENKINS_HOME / userContent /に保存されます 。 ロード時に、アイコンは64x64のサイズに圧縮されるため、より美しく見えます。

  • Unity3dプラグイン-Unityエディター呼び出しをアセンブリに統合するプラグイン。 強い欲求があれば、それなしで実行できます（コマンドラインですべてのコマンドを記述します）。 インストール後、 Unity3dアイテムがコレクターの設定に表示され、そこにインストール済みエディターのリストを入力できます。 たとえば、値が/Applications/Unity_4_6_8_f1/Unity.appのUnity_4_6_8_f1フィールドを作成します。 たとえば、テストビルドに使用されます。 メインバージョンとして、 Unity_4_6_6_f2と同様に構成されたパスを使用します。 コレクターの各タスクで新しいバージョンに切り替えた後、現在のバージョンへのパスを更新しないように、必要なパラメーターを使用してUnity_Defaultを作成します。

プロジェクト構造



  1. リポジトリをセットアップします。
  2. パラメーターを使用してファイルをカスタマイズします。
  3. ビルドプラットフォームをセットアップします。
  4. Unityを実行するかどうかを構成します。
  5. リポジトリからデータをダウンロードします。
  6. Unityが実行するように構成されている場合、新規ビルドをビルドします。
  7. アセンブリ結果をアーカイブに保存します。
  8. ビルド結果に応じてスクリプトを実行します（たとえば、音声アラーム）。




タスク（ジョブアイテム）の作成



  1. コレクターのメインページには、[ アイテムの作成 ]ボタンがあります。 パラメーターで、無料の構成を持つタスクを選択します 。 名前として、 projectName_platformなどを選択します。 さまざまなタスクの2番目の部分を変更することにより、単純なプロジェクトフィルターを構成できます。 設定に直接移動します。



  2. 複数のタスクに同じリソースを使用することが計画されているため、まずプロジェクトの場所を変更します。 「 プロジェクトの詳細設定 」を検索し、「 別のディレクトリを使用 」を選択して、 ../JENKINS_WORKSPACE/projectName_standaloneのようなものを入力します。 その後、設定の最初に戻ります。



  3. 「 廃止されたアセンブリの削除 」-必要なオプションを選択します。 たとえば、最後の数個のみを保存します。



  4. カスタムアイコンにプラグインがインストールされており、アイコンがロードされている場合は、必要なものを選択します。



  5. 「 これはパラメーター化されたアセンブリです 」-このパラメーターを構成します。 これを行うには、以前にインストールしたExtensible Choiceプラグインが必要です。 テンプレートは次のとおりです。このタイプのパラメーターが追加され、名前が入力され、データの読み取り元が選択されます。 データは、外部ファイルに保存するか（ System Groovy Choiceパラメーターを選択してデータ読み取りスクリプトを作成する）、または「 拡張選択：利用可能な選択肢プロバイダー」セクションのコレクターの設定で事前設定できます 。 ここでは、最初のオプションを使用します。 重要！ すべてのパラメーターで、デフォルトのオプションを事前に選択する必要があります。 ただし、スクリプトを保存して設定ページを再度開いた後にのみ使用可能になります。



  6. リポジトリを構成します。 REPOSITORYパラメーターを作成します。 /Users/jenkins/WORK/repositories.txtなどの外部ファイルに使用可能なすべてを追加します。 次に、スクリプトを実行します。 



     String _file="/Users/jenkins/WORK/repositories.txt"; ArrayList _list=[]; FileReader _reader = new FileReader(_file); String _line = null; while( ( _line = _reader.readLine() ) != null ) { _list << _line; } return _list.sort();


  7. アセンブリプラットフォームの選択。 PLATFORMを作成しますが、スクリプトは外部ファイルの名前のみが異なります。 以下は、4.6バージョンのプラットフォームのリストです。

  8. 設定ファイルの選択-PRESET 



     import hudson.*; import hudson.model.*; ArrayList _list=[]; _list << ""; FilePath _wp = new FilePath(new FilePath(jenkins.root),project.getCustomWorkspace()); hudson.FilePath _dir = _wp.child("Assets/BuildPresets/"); if (_dir.exists() == false) return _list; hudson.FilePath[] _files = _dir.list("*.asset") for (hudson.FilePath _file : _files) { if (_file.isDirectory() == false) { //_list << _file.getName(); _list << _file.getBaseName(); } } return _list.sort();


    このスクリプトでは、「スクリプトで事前定義された変数を使用する」チェックボックスを有効にして 、コレクターにアクセスできるようにします。 コードはJavaに似ています 。ドキュメントはhttp://javadoc.jenkins-ci.org/jenkins/model/Jenkins.htmlにあります。 getCustomWorkspace関数は、以前に指定した正確な場所へのアクセスを提供します。 デフォルトのテンプレートを使用する場合、パスを明示的に駆動する必要があります。



  9. BUILD_UNITYパラメーターを作成します。 そのためには、 Textarea Choice Parameterを個別に選択します。ここで、値TRUEとFALSEを2行に分けます 。



  10. 「 ソース管理 」を設定します。 Subversionを選択し、アドレスとして$ {REPOSITORY} 、無限の深さを入力し、アクセス用のユーザー名とパスワードを選択します（または、まだ行っていない場合は作成します）。 アセンブル時に、入力された式の代わりにファイルからの明示的な式が使用されます。 戦略として、「 最初に元に戻してから更新 」を選択します。



  11. 「 アセンブリトリガー 」を設定します。 たとえば、最新バージョンがリポジトリに表示された後にのみ、タスクを1日に1回定期的に収集できます。 たとえば、「 変更についてSCMに尋ねる 」フィールドの「H * / 8 * * *」は、最新バージョンで1日に3回アセンブリを開始します。 メニューの横に、構文の説明に関するヘルプがあります。



    他のビルドオプションの設定の詳細はこちら 。 アセンブリがリポジトリの最新バージョンに焦点を合わせており、デフォルトのリポジトリ以外のリポジトリが設定で選択されている場合、デフォルトが変更されてもアセンブリは実行されないことに注意してください！ 解決策は、次のアセンブリを手動で切り替えるか、トリガーまたはスケジュールに従ってアセンブリを構成することです。



  12. 最後に「 アセンブリ後の操作」セクションに進み、 Groovy Postbuildステップを追加します。 アセンブリの成功を分析する最も簡単なスクリプトは次のとおりです。 



     String proc = "/bin/sh /Users/jenkins/WORK/build-failure.sh"; if (manager.build.getResult() == hudson.model.Result.SUCCESS) proc = "/bin/sh /Users/jenkins/WORK/build-success.sh"; proc.execute();




    たとえば、エラースクリプトとして、次を使用できます。 



     say "Kill all humans"


  13. この段階で、設定を保存してアセンブリを開始します。 タスクの最後にディレクトリを作成し、リポジトリからプロジェクトをダウンロードして、スクリプトの1つを作成します。 エラーが発生した場合、スクリプトで使用されているすべてのファイルが作成されているかどうか、およびそれらに正しいアクセス権があるかどうかを確認します。



  14. すべてが機能する場合は、設定シートの中央に戻ります。 「 スクリプトから環境変数を生成する」 を選択して、次のように入力します。 



     BUILD_SUBDIR=`awk -F: '/location:/ {sub(/.*location: */,""); gsub(/\\\\/,"/"); print}' Assets/BuildPresets/${PRESET}.asset` BUILD_DIR="${WORKSPACE}/${BUILD_SUBDIR}" BUILD_VERSION=`../svnversion -c . | cut -f2 -d:` APPLICATION_NAME=`awk -F: '/buildName:/ {sub(/.*buildName: */,""); gsub(/\\\\/,"/"); print}' Assets/BuildPresets/${PRESET}.asset` BUILD_ARTIFACT=${APPLICATION_NAME}-${PRESET}-${BUILD_VERSION}.zip echo BUILD_SUBDIR=${BUILD_SUBDIR} echo BUILD_DIR=${BUILD_DIR} echo BUILD_VERSION=${BUILD_VERSION} echo APPLICATION_NAME=${APPLICATION_NAME} echo BUILD_ARTIFACT=${BUILD_ARTIFACT}




    スクリプトの最初の部分では、アセンブリパラメーターを使用して追加の変数を生成します。 たとえば、 PRESETファイルで選択した設定から正確なアセンブリディレクトリとアプリケーション名を検索します。 svnversionファイルはWORKフォルダーに配置されます。これを呼び出すことにより、アセンブリに関する詳細情報を取得できます。 すべてのパスは、現在のディレクトリからの相対パスです。



    2番目の部分は、特にアセンブリプロセス用です。 ヘルプに書かれているように、変数をアセンブリに直接統合するには、この形式で変数を/ stdoutに送信する必要があります。



  15. [ アセンブリ]セクションに移動し、アセンブリステップ条件付きステップ（複数）を作成します 。 開始条件で、文字列の等価条件を選択し、1つのフィールドで等価テスト$ {BUILD_UNITY}を実行し、別のフィールドでTRUEを実行します。 このセクションでは、以降のすべての手順を実行します。 この順序により、データを更新するためだけにタスクを実行できます。



  16. シェルコマンドを使用してステップを追加します。 アセンブリする前に、アセンブリディレクトリをクリーンアップする必要があります。特定の変数を使用します。 



     rm -rf "$BUILD_DIR" || true mkdir -p "$BUILD_DIR" || true


  17. 次のステップは、 Invoke Unity3D Editorです。 パラメーターにUnity_Defaultまたは特定のバージョンを指定します。 アセンブリパラメータは次のとおりです。 



     -quit -batchmode -executeMethod Build.CmdBuild -buildPreset=$PRESET -switchBuildTarget=$PLATFORM


    ほとんどのチームは、 こちらの公式ヘルプに記載されています 。 最後の2つのパラメーターはビルドスクリプト専用です。特定のオプションがない場合は、標準の-buildOSXPlayerを使用できます。 スクリプトと引数の例は、 Unity3dBuilderプラグインプラグイン情報とUnityコミュニティWikiにあります 。



  18. 最後のビルド手順は、結果をアーカイブすることです。 標準コマンド： 



     cd `dirname ${BUILD_SUBDIR}` zip -qr "${BUILD_DIR}/${BUILD_ARTIFACT}" `basename ${BUILD_SUBDIR}`


  19. アセンブリ後の操作セクションで、 アセンブリの説明を追加できます。 たとえば、 $ {BUILD_VERSION} $ {PRESET}です。 コレクターの履歴でビルドの横に表示されます。



  20. アセンブリ結果をアーカイブします。 すべてのファイルをリストできます。この例では、アセンブルされたアーカイブ$ {BUILD_SUBDIR} / $ {BUILD_ARTIFACT}を保存する必要があります。 この手順をスキップすると、アセンブリ結果をダウンロードできなくなります。 設定で結果の自動クリーニングが指定されている場合、後者のいくつかのみが使用可能になります。



  21. 最後のステップは、 Unityのルールでログ解析を設定することです。 その後、1つのプラットフォームのタスクは完全に準備が整います。





新しいタスクを作成する



上記の例では、タスクはMac用にビルドするように構成されています。 WindowsとLinuxでのアセンブリは、リソースを再インポートせずに同じプロジェクトで実行できるため、同じアセンブリディレクトリを使用できます。 これを行うには、新しいタスクを作成し、既存のタスクのコピーをタイプとして選択します。設定では、アイコン、タスク名、デフォルトプラットフォームのみを変更します（ディレクトリは古いままです！）。



表示設定では、タスクごとにフィルターを作成し、メインページにアセンブリに関する必要な情報を表示できます。 最終的に、次のようなものを取得できます。







Androidビルドをビルドする



最初の部分では、 JDKとJREへのリンクが提供されました-here 実際、コレクターが正常に機能するにはJavaが必要です。 Androidビルドを完全にビルドするには、 Android SDKをダウンロードして構成する必要があります 。

正しく動作させるには、ユーザーjenkinsの下でUnityエディターを開き、 Unity / Preferences / External Toolsに移動して、 Android SDKへのパスを構成する必要があります 。 新しいバージョンでは、 JDKへのパスも指定する必要があります。古いバージョンでは、このフィールドがありません。 動作する構成の例：



  • OS X 10.10.5
  • jdk-8u60-macosx-x64
  • Android SDK：ツール24.3.4
  • Android SDK：プラットフォームツール23.0.1
  • Android SDK：ビルドツール23.0.1
  • Android SDK：SDKプラットフォームAPI 23
  • Unity 4.6.6f2
  • Android SDKの場所：「/ユーザー/ジェンキンス/ WORK / AndroidSDK」
  • JDKの場所：「/ライブラリ/Java/JavaVirtualMachines/jdk1.8.0_60.jdk/Contents/Home/」


重要！ エディターの各バージョン（3.xx 4.xxと5.xxには異なる構成ファイルがあります）およびアセンブリが実行される各ユーザー（ jenkinsに十分）を構成する必要があります。



コレクターで新しいタスクを作成するために、 スタンドアロンタスクをコピーできます。 この場合、 カスタムワークスペースディレクトリを変更する必要があります-リソースを別のプラットフォームに再インポートする時間は非常に長くなる可能性があるため、 Androidビルドではプロジェクト用に個別のディレクトリを作成します。 構成ファイル、ビルドプラットフォーム、タスクアイコンを変更します。 保存されたアセンブリファイルにも違いがあります。たとえば、結果をアーカイブすることはできませんが、 $ {BUILD_SUBDIR} / *。Apk、$ {BUILD_SUBDIR} / *。Obbファイルを保存できます。



ビルドツールを変更する方法はありません。 たとえば、 Unityの異なるバージョンで、 JDKとAndroidSDKの設定が異なる1つ、4つ目の世代を構築する必要があります 。 〜jenkins / Library / Preferences / com.unity3d.UnityEditor4.x.plistファイル （ シェルの defaultsコマンドを使用）を編集するアセンブリステップを追加できます。また、 JDKパラメータープラグインをインストールし、 JDKを選択してアセンブリをパラメーター化できます。 誰かがこれを行う場合-結果の購読を停止してください-私はそれを記事に含めます。



重要！ Android SDKを更新する場合、プロジェクト用に作成されたすべてのJavaプラグインを再構築する必要があります。 サイトが言うように：アセンブリ中に、アセンブリラインに-source 1.6 -target 1.6を含めます。
