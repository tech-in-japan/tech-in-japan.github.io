スクラッチマニアの3つの穴

こんにちは



最近、友人と私は、Androidのアプローチで、 Windows Phoneプラットフォーム用のゲームScratch Maniaを公開しました。 ここでは、Kinectに関する最後の出版物から姿を消した場所と、エンタープライズ開発者がゲームを書くのはどのようなものかについてお話します。









ゲームについて
スクラッチマニアでは、プレイヤーは保護層によって隠された絵を推測するように求められます。 保護層は、画面に触れるだけで消去されます。 さらに、プレーヤーは保護層の一部のみを消去できます。これにより、関心が高まり、同時に複雑さが増します。 写真は、スポーツ、車、俳優、都市などのテーマごとにアルバムにまとめられます。 推測された写真ごとに、プレーヤーはゲーム内のコインを受け取ります。これは新しいアルバムと交換できます。



私たちはこのゲームを週末のハッカソンのテーマとして考えました：誰かと一緒になって、フレームワークを作成します-実用的なプロトタイプを作成し、1〜2か月で公開に適した状態にします。 そして実際に素晴らしい年を過ごしました。 1年以上、カール！ 公平を期して、私は仕事は通常夕方、暇なときに行われたことに注意します。 今こそ、重要な技術的解決策を検討するときです。



7回測定する



私がゲームについて話す人々から最も人気のある質問の評価を行う場合、この質問が最初に来ます：「うん、Windows Phone？ 真剣に??。”皮肉なサインでさえ必要ありません。 それでも、C＃を知っているため、このプラットフォームを選択しました。 また、JavaもCから派生した言語であるため、Androidバージョンもバージョン化することにしました。現時点では、準準備状態になっています。 その結果、「スタイリッシュでファッショナブルな若者」Xamarinを使用するか、ネイティブアプリケーションを作成するかを決定することになりました。 悲しいかな、Xamarinは負けました。 主に同僚とオンラインコミュニティからの混合レビューが原因です。 フレームワークとの戦いに多くの時間を費やしたくありませんでした。 それに加えて、価格は痛烈でした-個々の開発者に対して月額約20ドル。



Javaと.NETだけが競争相手ではなく、javascriptなどのエキゾチックなオプションも検討されました。 しかし、彼は遅すぎます。 ところで、後でゲームのサイトにプロトタイプを使用しましたが、良さは失われません。



Windows PhoneとAndroidのバージョンについては、簡単にするために、それぞれWP 8以降とAndroid API 11以降で停止することにしました。 ここでは主にWindows Phoneについて説明します。なぜなら、 Androidについて話すのは時期尚早です。



威勢のいいトラブル開始



最初に計画したように、2日間の休みの代わりに、プロトタイプに約2週間費やしました。 JavaScriptで開始。 アイデアは単純です。キャンバスをページに追加し、その中に絵を描きます。 DOMに追加せずに、別のキャンバスを作成します。 キャンバスのサイズは同じですが、その中に別の絵を描きます。 OnMouseMoveイベントハンドラーで、N×Nピクセルの領域をあるキャンバスから別のキャンバスにコピーします。 次のようになります：





しかし、それで誰も驚かないでしょう。そのため、Windows Formsで次のプロトタイプを作成しました。消去アルゴリズム自体は、画像エディターのエアブラシツールとの類推によって作成されました。 このプロトタイプでは、2つの画像が互いに重ね合わされ、上部のタッチゾーンでアルファチャネル値が変更されました。 javascriptの第一人者はこのようなものを書くことができると確信していますが、これは私たちの間では見つかりませんでした。



プロトタイプのコードはほとんど変更されずにスマートフォンに転送されました。 Windows PhoneとAndroidのコンポーネントはアルファチャネルで異なる動作をすることが判明しました。 Windows Phoneでは、チャネルは事前乗算されますが、Androidではそうではありません。 これは、Androidではアルファチャネル値がピクセルの色成分（RGB）に影響を与えないのに対し、Windows Phoneでは式に従って変化することを意味します。 

A = alpha R = red × alpha / 255 G = green × alpha / 255 B = blue × alpha / 255


この場合、ピクセル値を計算する操作は実際にはピクセルコンポーネントを加算することになるため、この形式で表示される画像は簡単に重なります。たとえば、Rres〜R1 + R2です。 詳細は、Charles Petzold（ amazon ）の著書Programming Windows（第6版）にあります。





しかし、トラブルが一人で来るわけではありません。 Touch.FrameReportedイベント（AndroidのonTouchEvent）は、両方のプラットフォームで非常に遅いです。 たとえば、ハンドラーのタッチポイントでアルファチャネルの新しい値を計算するためにこのイベントをサブスクライブすると、FPSは約20になります。これは非常に小さいためです。 遅延は肉眼で見えるようになります。 Androidでは、彼らはそれを延期し、リリースの近くで修正することにしました。 Windows Phoneでは、約25ミリ秒に1回画面をポーリングし、タッチポイント（通常は複数あります）（ある場合）を受信し、アルファチャネルの値を計算するバックグラウンドスレッドを使用して問題を解決しました。 これにより、FPSレベルを30〜40に保つことができます。



さらに、Windows Phoneでは、画像の一部が無効化されていないことを覚えておく必要があります。変更後は、画像全体を更新する必要があります。 また、タッチ処理に追加の制限が課せられます。一度に処理されるポイントが多いほど良いです。 一方、あまり多くのポイントを処理することはできません。 ジャークが目立ちます。



完璧さに制限はありません



美しい消去は最初の「キラー機能」であり、2番目は写真アルバムです。 このような重要なコードには対応する態度が必要であることを認識して、彼らは設計を真剣に受け止めました。 システムは階層化されました-個々のアルバムの低レベルのパーサーから、インストールされたアルバムやダウンロード可能なアルバムの高レベルのマネージャーまで。

アルバムについて
ゲームでは、写真付きの3つのアルバムが最初にプリインストールされ、残りのアルバムはサーバーに保存されます。 これにより、プレーヤーは興味のあるアルバムを決定してアップロードできます。また、新しいアルバムの配布プロセスを簡素化できます。プレーヤーには、アルバムリストで「新規」とマークされたアルバムが表示されます。

一見、すべてがそれほど怖いわけではありませんが、ダイアグラム上で簡略化されています：一部のタイプが示されていない、すべてのインターフェイスのコードコントラクト、ヘルパークラスなど このファミリは数回リファクタリングされ、対応し、最終的には、ほぼすべてのインターフェイスとファクトリが削除されました。

「透かし」のすべて



ゲームのサーバー部分をMicrosoft Azureにデプロイしました。 元の写真と完成したアルバムはBlob Storageに保存されます。 識別子、物理的な場所（Blob内）、ゲーム内のアクセシビリティなど、アルバムのメタデータを保存するため Azure SQLで使用されます。 これらのコンポーネントの使用は、RESTfulサービスによって隠されています。 これがメインストリームであるため、私たちはついていくことにしました。



実際、私たちのサービスは3つの方法のセットです：アルバムのリストを取得する、アルバムに関する詳細情報を取得する、アルバムをダウンロードする。







アルバムをダウンロードするために、サービスは有効期間が制限されたリンクを生成します。 Azureは、既成のメカニズム-共有アクセス署名を提供します。 この手順の本質は次のとおりです。有効期間はURLに縫い付けられ、その後、URLは秘密鍵で署名されます。 これにより、URLへの変更がなくなります。 日付を指定する必要はありません。コンテナのアクセスポリシーを作成し、URLでこのポリシーを参照するだけです。 ただし、短命リンクの場合、このオプションはあまり便利ではありません（新しい期間ごとに新しいポリシーを作成する必要があります）。

PNGへのデザイン
多数のExpression DesignファイルをPNGまたはJPEGにエクスポートする必要がある場合、自動化せずにプロセスが退屈になる可能性があります。 Expression Design APIを使用して小さなコンバーターを作成しました（これは、1つの一般的な "リフレクター"を使用してアセンブリを調べて取得した、文書化されていないAPIです）。



2つのアセンブリが必要です。

％ProgramFiles％\ Microsoft Expression \ Design 4 \ GraphicsCore.dll

％ProgramFiles％\ Microsoft Expression \ Design 4 \ Microsoft.Expression.Design.dll

そしてそのようなコード： 

 const string from = @"..."; //   *.design  const string to = @"..."; //    *.png  Module.Instance.Initialize(); var doc = Module.Instance.OpenDocument(from, false, null); ExportImageInfoHandler exportHandler = doc.ExportImage(to, (Enum)ImageFormat.Png); exportHandler.Info.Resolution = 96.0; exportHandler.Info.DocumentUnits = UnitType.Pixel; exportHandler.Info.ConstrainProperties = false; exportHandler.Execute();
