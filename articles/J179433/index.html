フレームワーク機能



1. 軽量。 まあ、非常に小さい。 開梱すると、約34キロバイトかかります（標準のウェルカムテキストとロゴ画像を削除した場合）

2. OOPの欠如。 これが利点であるか、その逆であるかはわかりませんが、理論的にはOOPアプリケーションは手続き型スタイルで記述されたアプリケーションよりもはるかに多くのリソースを消費します。 （これが最初に実証された人がまだ基礎を完全に理解していなかった場合、オブジェクトについてどのようなスピーチがあり得るか）

3. MVC。 さて、ここでは何も説明する必要はないと思います。

4. ルーティング。 （私の意見では）ルーティングは非常に使いやすいですが、同時に設定が非常に柔軟ですが、改良が必要です。 デフォルトのルーティングは「Controller / Action / Param-1 / Value-1 / Param-2 / Value-2」です。

5. JSONおよびXML。 これらの形式のいずれかで回答を得るには、それぞれアドレス「.json」または「.xml」を追加する必要があります。 つまり、アドレス "/ goods / view / id / 3"で製品に関する情報を受け取った場合、アドレス "/goods/view/id/3.json"でJSON形式で取得できます。

6. 機能。 コミュニティは1人だけなので、すぐに使用できる機能はあまりありません。 これまでのところ、SOAPディスパッチスクリプトが不十分であり、セッションを使用したユーザー認証が非常に機能しています。

7. バックトレース。 プレゼント。 アプリケーションがデバッグモードでない場合は切断されます。

8. DB。 MySQLのみがサポートされています。 これはマイナスですが、現時点では重要ではありません。

9. ORM。 OOPは原則としてフレームワークに存在しないため、ここにはありません。 ただし、データベースには何らかのバインドがあります。

10. ACL。 現時点では利用できませんが、フレームワークの開発計画（少なくともいくつかの肯定的なレビューを受けた場合にのみ開発されます）はそもそもです。

11. デバッグモード。 存在し、いくつかのエラーの出力、バックトラックの出力、使用された設定に影響します。
