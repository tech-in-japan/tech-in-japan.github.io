アスタリスクにミニコンタクトセンターを作成する際の落とし穴

私たちの組織はサービスの提供に従事しています。 当初、私たちは1つの都市に拠点を置き、コンタクトセンターは1人の女の子と携帯電話で構成されていました。 都市の数が増加するにつれて、電力を迅速に増加できるソリューションを実装する緊急の必要性がありました。 そして、もちろん、選択はIPテレフォニーにかかっていました。



2011年末には、近隣の都市に1つずつ支店が開設されました。 すでに2人のオペレーターがいます！ 今、彼らは定期的に戦いを開催しました。 （私たちはたくさんの数字を作りたくなかったので、どのように間違えたかは後で示します）。 PBXが必要です！ 予想通り、サーバーにシステムを割り当て、Ubuntuをインストールし、ソースからアスタリスクを収集しました（ハードウェアに合わせて最適化する必要があります！）そして、Webインターフェース（FreePBX！誰が疑うでしょう！） 最初はMegafonからのモバイル通信を使用しましたが、ほとんどの通話はネイティブ地域で行われ、すぐに解決策が出ました。無制限のSIMカードを搭載した3Gホイッスルをシステムに挿入しました。 事前ロック解除し、モデムのみのモードを終了しました。 セルラー通信のコストを最適化するためのアスタリスクの構成に関する記事では、MCC / MNCを使用して目的のモデムを決定する方法が示されていますが、より簡単にそのようなトランクを作成しました。



設定=>トランク=>特別なトランクを追加=>発信設定特別なセット=>データカード/ i：123456789012345 / $ OUTNUM $、ここで123456789012345（IMEIモデム）



通常どおり、アウトバウンドルーティングが作成され、トランクが選択されます。



私たちが驚いたのは：

  • マルチチャネルのアウトバウンド通信が曲がっています
  • オペレーターが話している場合、2番目のクライアントは、楽しいウェルカムビデオではなく、退屈なビープ音を聞いています！




解決策は、奇妙なことに、すぐに思いつきました。SIPチャネル（Multifonドライブ！）を介して着信コールをピックアップし、3Gモデムを介して発信コールを送信し続けます。 結果はハイブリッドです。 （ Megaphone Multifon記事のvp7に感謝します-代替SIPクライアントを使用してください）

最も興味深いのは、マルチチャネルの着信回線があることです！ 私たちは手をこすり、顧客からの電話を待ち始めました！ しかし、そこにありました！ 都市間で誰も私たちに電話するつもりはありませんでした！ 次に、各都市で、都市番号の付いたメガホンのSIMカードを購入し、アスタリスクにねじ込みました。 発信の通信は、まだ3Gモデムを介して実行されました。 私はこれがより安価であることに同意しますが、USBポートで電源障害が発生した場合、常にプラグを差し込んでアスタリスクユーザーをデバイス/ dev / ttyUSB *に再割り当てする必要がありました。

1つの落とし穴をまだ待っていました。 時々、SIP経由で受け取ったメガホンからの数字が減少し始めました。 技術サポートオペレーターが1人でも理由を明確に説明できません。 最初は、非ネイティブクライアントを使用していると言われましたが、アプリケーションが機能していないとすでに言い始めたとき、彼らは大量のチケットを作成しました。 それは無駄であることが判明しました！

私自身は、Multifonで作業するときに2つのルールを特定しました。

  • バランスは常にプラスでなければなりません！
  • Simkaは、少なくとも1日1回ネットワークに登録する必要があります。




この時点までに支部都市の数は10個に達しました。 各都市には、番号付きの独自のSIMカードがあります。 コンタクトセンターはすでに約5人です。 3Gモデムは、発信回線上の1つのチャネルに対応できませんでした。 また、発信回線をSIPに転送する必要がありました。 そして、通信コストは急増しました。 MegaphoneのSIPを介して電話をかけましたが、彼は厳しい関税を課しています。ロシアのどの電話でも1分あたり1.5ルーブルが切り上げられました。



新しい通信プロバイダーの検索には約1か月かかり、成功しました！ ロシアの携帯電話では毎分1.4ルーブル、毎秒課金の固定電話では0.8ルーブル未満を提供している（正確には覚えていません）大手プロバイダーが見つかりました。 最も重要なことは、彼らがOutbound Caller IDサービスを提供することです：発信通話の識別子で任意の番号を代用できます（この場合、これらはマルチフォンからSIPを通して通話するときのように、連邦形式ではなく都市形式のメガフォンからの番号です）



現在、すべてがElastixディストリビューションに基づいています。 着信コールはMegafonを経由し、発信コールは2番目のプロバイダーを経由します。 クラウドでは、メインサーバーまたはインターネットチャネルがクラッシュした場合にミラー設定されたElastixのバックアップコピーにより、不可抗力は問題になりません。 コールの流入が増加した場合に、オペレーターの負荷を確実に分散するために、2つのコンタクトセンター（別のセンターを開設）の間でリングを編成することが計画されています。
