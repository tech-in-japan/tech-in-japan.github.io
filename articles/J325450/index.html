Windows 10のARMコアに基づいて、マイクロコントローラー用のプログラムを作成およびデバッグするための無料のアセンブリをセットアップしました

記事を書くというアイデア（これは、原則としてプログラミングの知識がほとんどないマイクロコントローラーで何かを作成したい初心者向けの一連の記事になります）は、友人が仕事をセットアップするのを助けるために私の主要なタスクから少し脱線しなければならなかった後に思い付きましたstm32f103を搭載したボードに基づいた彼の小さなホームプロジェクト用のソフトウェアを書いた水曜日。 約30分、最大1時間かかると予想していましたが、私の友人のラップトップはWindows 10 x64に搭載されていましたが、これは私にとっては珍しいことです（私はUbuntuで働いています）。 その結果、8時間近くかけて簡単なプロジェクトを設定および作成し、多くの明白ではないことに苦労しました。



これと並行して、友人が以前にマイクロコントローラーに出会ったことがないため（「ストアでArduinoを見た」という言葉から）、どのアセンブリ要素が何のために必要であるか、およびこれらの要素が相互作用する方法を詳細に説明する必要がありました。



この資料は、初心者がすばやく問題なくマイクロコントローラーを操作するための完全に無料のインフラストラクチャをセットアップし、最終的なバイナリファイルがどのように組み立てられるかを理解することを目的としています。 このインフラストラクチャをセットアップする段階では、マイクロコントローラーのメーカーとモデルは重要ではありません。 主なことは、ARMコアに基づいていることです。

問題の声明

次のことができるように、束を構成する必要があります。
  1. C（C99）、CPP（C ++ 14）およびASMファイルで構成されるプロジェクトをコンパイルします。
  2. コンパイルされたファイルから単一のファームウェアファイルをアセンブルします（「.elf」、必要に応じて「.hex」、および便利なアセンブリ分析用の「.map」ファイル）。


ソフトウェア実装の選択

タスクを解決するには、次のソフトウェア製品が必要です。
  1. Eclipse Neon 3 for C / C ++（執筆時点の最新バージョン）。 これは、IDE（便利な自動補完機能+デバッグツールとの対話に便利なテキストエディター）として使用され、コードを記述します。
  2. JRE（執筆時点では、最新バージョンは1.8.0です）。 これがないと、Eclipseインストーラー（まあ、Eclipse自体）は起動しません。
  3. GNU ARM Embedded Toolchain（執筆時点では、最新バージョンは5_4-2016q3-20160926でした）。 これは、必要なソフトウェアソリューションの複雑なものです（「gcc」コードのCコンパイラ、「g ++」コードのC ++、リンカー-「ld」、最終ファームウェアをロードおよびデバッグするためのツール-「gdb」など）。ソースコード、解像度が「.elf」のファイル。これはバイナリマイクロコントローラーファームウェアファイルで、その後マイクロコントローラーにダウンロードされます（詳細は以下を参照）。
  4. OpenOCD 0.10.0。 それを使用して、「。elf」プログラムファイルをマイクロコントローラーにロードします（実際、OpenOCDは上記のツールチェーンからのgdbとデバッガーの間のリンクを提供します）。
上記のツールに加えて、いくつかの小さなパッケージを提供する必要があります。これについては、インストールプロセス中に直接説明します。



Eclipse Neon 3を配置しました

前述のように、コードを書くためには、書くのに便利なテキストエディターが必要です（自動補完、プロジェクト検索、ファイルナビゲーションなどのさまざまな方法）。 そして、コードを記述した後、コンパイル、アセンブリ、および実行がキーボードショートカット（またはマウスクリック、誰にとっても便利）の問題であると便利です。

これらの目的のために、私はEclipseを使用します。 エディターに加えて、開発者の生活を大幅に簡素化するさまざまな拡張機能を接続する機能も提供し、すべてのルーチン作業（アセンブリ、リンク、コントローラーへのプログラムのロード）を数回のクリック/クリックに減らします。

JREをインストールする

インストーラーを実行するには、最初にJREをインストールする必要があります。 執筆時点での最新バージョンは1.8.0です。

  1. ダウンロードしたインストーラーを起動します。 JREがないというエラーが表示されます。
  2. 「いいえ」をクリックして、サイトへの移行を待ちます。
  3. サイトで、アイテム「Oracle JRE 1.8.0」を選択します。
  4. 「ライセンス契約に同意する」を選択します。
  5. Windowsのビット深度に応じて、オフラインバージョンをダウンロードします。
  6. ダウンロードしたファイルを実行します。 さらにインストールしても問題は発生しません。

パッチJRE（上記の段落でエラーが発生した場合）

たまたまJREに不快なバグがあり、プラグインを正常にインストールできません。 種まき現象の性質を理解することなく、次のことを行います。

  1. パッチのあるサイトに移動します。
  2. 「ライセンス契約に同意する」項目を選択します。
  3. 「.zip」アーカイブをダウンロードします。
  4. 開梱します。 パッチとテキストファイルのアーカイブには2つのファイル（ "local_policy.jar"および "US_export_policy.jar"）があります。 これらの2つのファイルを許可「.jar」でコピーし、インストールされたJavaアドレスのファイルに置き換えます。 64ビットWindows 10の場合、これが方法です

    C:\Program Files\Java\jre1.8.0_121\lib\security





    jreの新しいバージョンがリリースされるか、OSのビット深度が異なる場合、パスは異なる場合があります。 主なことは、「\ lib \ security」フォルダーに移動し、これらの2つのファイルを置き換えてそこにコピーすることです。 このパッチでは、完全と見なすことができます。 eclipseを再度実行して、プラグインをインストールできます。
エラー写真と交換パス。














GNU ARM Eclipse Windowsビルドツールをインストールする

実際のところ、Windowsではコマンドラインの「make」コマンドはデフォルトでは機能しません（Linuxでは、すべてのディストリビューションでこのアプリケーションがデフォルトでインストールされるわけではありません）。 なぜ「make」アプリケーションが必要なのか、以下で説明しますが、今度はそれを使用する機能を追加するプラグインをインストールするだけです。 これのために。

  1. 公式リポジトリに移動します 。
  2. プラットフォームの「.exe」ファイルをダウンロードします。 執筆時点では、最新バージョンは「gnuarmeclipse-build-tools-win64-2.8-201611221915-setup.exe」です。
  3. ダウンロードしたファイルを実行し、「次へ」、「同意する」、「次へ」、「インストール」、「終了」をクリックするだけです。


GNU ARM Embedded Toolchainをダウンロードしてインストールします

  1. 公式ウェブサイトに行く
  2. Windows用の「.exe」ファイルにコンパイルされた最新の最新バージョンをダウンロードします（右側の列）。
  3. ダウンロードしたファイルを実行します。
  4. OKをクリックします。
  5. 次に「次へ」
  6. 「同意します」
  7. 「インストール」（ここでは、任意のフォルダーを指定できることを説明しますが、インストール中にコンポーネントを登録する際の問題を回避するために、デフォルトのフォルダーを残すことをお勧めします）。
  8. インストールの最後に、「Show ReadMe。」ファイルのチェックを外します。 残りはデフォルトで残されています。
  9. 完了


次の手順は参考情報であり、省略できます。 この場合、開いているコマンドラインを閉じることができます。

  1. 開いたコマンドラインで、引用符なしで「cd bin」と記述します（これは、すべてのツールチェーンコンポーネントの「.exe」ファイルがある場所です）。
  2. 引用符なしで「arm-none-eabi-gcc --version」と入力します。
  3. gccが応答することを確認したら、コンソールを閉じます。

OpenOCDをインストールする

環境とツールチェーンを扱いました。 次に、OpenOCDをインストールする必要があります。これにより、Eclipseはarm-none-eabi-gdbを介してコントローラーを制御します。 ここではすべてが簡単です。

  1. 公式リポジトリに移動します 。
  2. OSの「.exe」ファイルをダウンロードします。 執筆時点では、最新バージョンは「gnuarmeclipse-openocd-win64-0.10.0-201701241841-setup.exe」です。
  3. ダウンロードした「.exe」を実行します。 何も変更せずにインストールします（パスを含む）。


st-link v2にドライバーをインストールする

OpenOCDはコントローラーを制御しますが、これは事実ですが、このためにはデバッガー用のドライバーが必要です。 私たちの場合、これはst-link v2です。 別のデバッガーがある場合、この部分はスキップできます。

偽のst-linkかオリジナルかを問わず、ドライバーは同じです。 始めましょう。
  1. st-link v2の公式ページに行く
  2. ページの下部で、「ST-LINK、ST-LINK / V2、ST-LINK / V2-1 Windows7、Windows8、Windows10用に署名されたUSBドライバー」という説明を含むファイル「STSW-LINK009」をダウンロードします。
  3. 別のページに切り替えるときは、ページの下部にある[ダウンロード]をクリックします。 ここで、VPNを介してインターネットをサーフィンしない場合は、登録する必要があることに注意してください。 これを行う方法については説明しません。 その後、ファイルがダウンロードされます。
  4. アーカイブを解凍した後、OSのビット深度に応じてドライバーインストールファイルを選択します。 私の場合、これは「dpinst_amd64.exe」です。
  5. インストール中に、すべての要件に同意します。 そして、「STMicroelectronicsソフトウェアを常に信頼する」チェックボックスを選択します。
写真をダウンロードしてインストールするプロセス。














すべてがどのように機能するかを理解しています

そこで、必要なすべてのコンポーネントをインストールし、Eclipseでプロジェクトを作成し（プロジェクトの作成と構成の方法については次の記事で説明します）、「Ctrl + B」を押しました。 「debug」アイコンをクリックすることで、コントローラーにダウンロードしてデバッグを直接開始できる、準備が整ったファームウェアファイルが得られました。 しかし、これらすべてのアクションの背後に隠されているものは何ですか？ そして、次のことが起こります：
  1. 「Ctrl + B」を押すと、Eclipseはプロジェクトディレクトリツリー（作成したすべてのフォルダーとその中のファイル）を分析します。 これを行うには、プロジェクトの作成時に指定したファイルパスを使用します。
  2. その後、Eclipseはメイクファイルを作成します。 このファイルには、コンパイルするファイル（プロジェクトに追加し、それらが配置されているディレクトリへのパスを示します）、およびパラメーター（ここではEclipseは「C / C ++ビルド））および最終ファームウェアファイルで以前にコンパイルされたファイルをコンパイルする方法。

    注：makefileには拡張子はありません。
  3. 次に、Eclipseは、作成されたメイクファイルへのパスと、プロジェクトのセットアップ時に指定したビルドパラメーター（たとえば、コンパイルに使用されるプロセッサスレッドの数）を使用してmakeプログラムを呼び出します。
  4. makeプログラムは、makefileの指示に従ってすべてのプロジェクトファイルを調べ、各ファイルの拡張子が「.o」のファイルを受け取ります。 これらのファイルは、メインプロジェクトと同じ方法で（ツリー階層を保持して）Debugフォルダーにあります。 何らかのファイルにエラーがある場合、コンパイルコンソールで赤で強調表示されたこの通知を受け取ります（デフォルトでは、画面の下部にあります）。 コンパイルエラーが発生した場合、それ以降のコンパイルとアセンブリは終了します。 ファイルをコンパイルするには、makeはmakefileツールチェーン（または、そのコンポーネント：gcc / g ++）で指定されたものを使用します。 Eclipseは、プロジェクト設定からツールチェーンへのパスを再び取ります。
  5. 「.c」、「。cpp」、「。S」からすべての「.o」ファイルを作成した後（アセンブラーコードを含むファイルには大文字の「S」拡張子が必要であることに注意してください。そうしないと、メイクファイルを作成するときにこれらのファイルは単に無視されます、エラーを把握するのが難しくなります）プロジェクトからビルドファイルへのパス（プロジェクトに追加する拡張子「.ld」のファイル）でリンカー（インストールしたツールチェーンのarm-none-eabi-ld）を呼び出します。 ここで、リンカは、拡張子が「.ld」のレイアウトファイルで指定された命令（ルール）に依存して、ヒープ「.o」ファイルからパーミッション「.elf」を持つオブジェクトファイルを1つ収集しようとします。 "and" .map "）。 どのコードとデータをプログラムからスローできるかを決定するのはリンカー（リンカー）であることに注意することが重要です（それらがどこでも使用されず、スペースを占有すると見なす場合）。 したがって、1000個のグローバル変数を作成し、そのうち2個のみをコードで使用する場合、残りは（キーワード「volatile」または「何も削除できないセクションに入れる」でプロジェクトから除外できないことが明示的に指定されていない限り） "）最終的なオブジェクトファイルから除外されるため、空きメモリが増えます。
  6. 最終的なオブジェクトファイルを受け取ったら、それをコントローラーに「縫い付け」てデバッグします。 これを行うために、Eclipseはファイル「openocd.cfg」へのパスでOpenOCDプログラムを呼び出します（名前は拡張子「.cfg」であれば何でも構いませんが、IDEからではなくコマンドラインからデバッグする必要がある場合、 OpenOCDの正しい名前のファイルでは、構成ファイルの名前を書き込めないことがわかります（構成ファイルが指定されていない場合、その名前のファイルが自動的に検索されるため）。

    指定されたファイルが存在する場合、OpenOCDは、ファイル内のパラメーターに従って、マイクロコントローラーへの接続を試みます。 接続が正しい場合、接続が確立され、コントローラーデバッガーが動作しているという通知を受け取ります。

    コントローラーに接続するために、OpenOCDは最初にインストールしたst-linkドライバーを使用します。
  7. OpenOCDは、コントローラーとの接続を確立できた後、Telnet接続を開きます。 詳細に説明しない場合、これはarm-none-eabi-gdbがコントローラーを正常に制御できるようにするために必要です。
  8. telnet接続が確立されると、Eclipseは接続先のポート（OpenOCDを開いたポート）でarm-none-eabi-gdbを起動します。
  9. 次に、Eclipseは初期化コマンドをgdbに送信します（フラッシュの消去、ファームウェアファイルの書き込み、メインへのブレークポイントの設定、およびその他のいくつかのオプションパラメーター）。
  10. その後、Eclipseでメインファイルのコードの最初の行へのポインターが表示され、デバッグを開始できます。
純粋に技術的には、ここで行われるすべての操作は毎回手動で行うことができますが、キーを押すだけでデバッグの開始を楽しむ方がはるかに便利です。
