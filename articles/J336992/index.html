モバイルアプリケーションテストのチートシート



同じレーキを踏むのが好きな人はいません。 要件の分析とテストの段階でそれらを回避するために、共通のチートシートを導入しました。 理想的には開発前に特定して修正する必要がある問題が含まれます。 これをチェックリストとして使用し、その助けを借りて、モバイルアプリケーションのロジックをより深く掘り下げ、ユーザーがどのような潜在的な問題を抱えているかを把握しようとします。







この記事は、TKレビューを実施して不整合を見つけることを求められるテスター、およびTKを読んだ後、開発者がフォーマットの質問をよくするアナリストを対象としています。







それではどこから始めますか？







一般的なアプリケーションの質問



ここではすべてが簡単ですが、プロジェクトマネージャーやアナリストと一緒にこれらの問題を明確にすることを忘れないでください。







  1. アプリケーションはどのプラットフォーム用に開発され、どのOSバージョンがサポートされますか？ OSの最小サポートバージョンを常に覚えておく必要があります。 そうしないと、タスクが既に閉じられている場合、ユーザーに対して機能が機能しないことがあります。
  2. どのデバイスでアプリケーションを確認する必要がありますか？ たとえば、アプリケーションはスマートフォンとタブレットの両方で動作するはずです。 または、Apple Watchのサポートがあるはずです。
  3. アプリケーションはどの画面の向きをサポートしていますか？ ポートレートおよび/またはランドスケープ？ 不快な瞬間：スマートフォンで画面の向きの変更がうまく機能する場合、これはタブレットですべてが同じになるという意味ではありません。
  4. 視聴するのに最も重要なデバイスは何ですか？ デバイス上では、アプリケーションは完全に機能しますが、顧客のお気に入りの場所（中国のアンドロイドスマートフォンを挿入）で、すべてが分散しています...
  5. 完璧なピクセルパーフェクトが必要ですか、それとも何らかのエラーが許容されますか？ こんにちは、模擬適合テスト！ レイアウトを引き伸ばすべきか、それとも各画面サイズのレイアウトがあるべきかを明確にするとよいでしょうか？
  6. 他のクライアントアプリケーションはありますか？ たとえば、アイテムの削除または追加を突然開始する管理パネルがあります。 または、実稼働環境にすでに存在するWebバージョン。 主なことは、これについてできるだけ早く知ることです。
  7. モバイルアプリケーションのロジックに影響を与える可能性のある外部デバイスはありますか？ たとえば、信号をアプリケーションに送信するビーコン、またはアプリケーションから情報を印刷するプリンター。
  8. アプリケーションの対象読者は何ですか？ Playマーケット/ AppStoreのすべてのユーザーですか、それとも顧客の会社の50人ですか？


スクリーン解析



  1. 画面の構成とその上で可能なアクション。 画面はどの要素で構成されていますか？ どのようなアクションを実行できますか？ どのような画面条件が可能ですか？ トランジションとは何で、どのスクリーンにつながるのですか？ この画面に戻るときに何を表示する必要がありますか？ これらの質問に対する答えを見つける必要がありますが、ドキュメントに記録する方が適切です。
  2. 画面上のサーバーとの相互作用。 どのようなリクエストが画面に表示されますか？ 画面上で送信されるサーバー要求を理解すると、サーバーが何らかの理由で実装できない要件を特定するのに役立ちます。
  3. タイマーアクティビティ。 たとえば、重要な分析は2分ごとに送信されるか、データが更新されます。
  4. データのキャッシュ。 画面に入るたびに同じデータをダウンロードすると、ユーザーを困らせることがあります。 キャッシュするとき、画面上の情報をいつ更新する必要があるかを考慮する必要がありますか？ キャッシュをクリアする必要があるのはいつですか？
  5. スタブ。 データがない場合は何が表示されますか？ 空白の画面は、ユーザーにとって有益ではありません。 そして、移動したスタブが顧客の不満の原因である可能性があります。
  6. エラー発生時の動作。 エラーが発生した場合、何を表示する必要がありますか？ たとえば、インターネットの欠如、サーバー側のエラー、または文書化されていないエラー。
  7. データの読み込みが遅い。 データの読み込みが遅い場合はどうなりますか？ ローダー、ブロッキングアクション、カスタムアニメーション-すべてを考慮する必要があります。
  8. 他の画面の動作に影響するアクション。 1つの画面でのアクションは、他の画面の動作にどのように影響しますか？ 横断は危険なものです。 特に、開発やテストが画面または個々の機能で行われる場合。 ここで回帰なしに行うことは困難です。 そのため、一部のプロジェクトでは、テストケースを作成する前に、新機能の影響マトリックスを作成します。
  9. 画面上のデータを更新します。 更新はいつ行われますか？ 次のオプションがあり、それらを組み合わせることができます。

    • 画面を開くたびに（ユーザーが画面を開いている間のみデータが存続します）。
    • アプリケーションが起動されるたびに（データはユーザーがアプリケーションを実行している間のみ存続します）。
    • プルトゥリフレッシュ/特別な更新ボタン/タイマー（データはデバイスのローカルストレージに保存され、アプリケーションの再起動時に復元されます）。


次に、アプリケーションでよく使用される機能を検討します。







アプリ内ナビゲーション



  1. サイドメニューを使用します。 どのセクションがルートですか？ ルートの上に開くセクションは何ですか？ ルートパーティション間の移行履歴はリセットされますか？
  2. タブバーを使用します。 セクション内のナビゲーションを深くすると、タブバーは画面に残りますか？ パーティションを再度テーピングすると、ルート画面に戻りますか？
  3. Androidのハードウェアとソフトウェアの「戻る」ボタンの遷移の違い。


ローカリゼーション



サポートされているローカライズの種類：







  1. テキストはアプリケーション内で保護されます。 アプリケーション設定のユーザーは、目的の言語を設定できます。
  2. テキストは、システム設定の言語に依存します。 言語は、システム設定でインストールされている言語に応じて決定されます。
  3. テキストはサーバーから送信されます。 テキストはサーバーから送信され、言語はデバイスの設定またはアプリケーションの設定に依存します。


許可



  1. 通知、地理位置情報、ギャラリー、カメラ、SMSへのアクセスのリクエスト...カスタム画面または単なるシステムアラート？
  2. ユーザーがアクセスの許可を拒否しました。 この場合、アプリケーションはどのように動作しますか？ アクセスのための再要求ロジックはありますか？
  3. ユーザーがシステム設定でアクセスを無効にしました （上記の段落を参照）。


リスト



多くの場合、モバイルアプリにはリストが含まれます。 彼らにとっては、次の点に注意する必要があります。







  1. リストの最初のダウンロード。 一度にいくつのアイテムがロードされますか？ 起動時にどうなりますか？ リストがロードできる最大時間は？
  2. ページングの存在。 スクロール時に要素の読み込みがありますか、それともリスト全体が一度に読み込まれますか？ 負荷がある場合は、境界の要素が消えず、複製されていないことを確実に確認する必要があります。
  3. リストを更新します （上記のオプションを参照）。
  4. セクションの存在。
  5. フィルター/ソートの存在。 フィルターは、ローカルまたはサーバーにすることができます。 完全にロードされるリスト、またはアプリケーション内で配線されるリストの場合、フィルターはほとんどの場合ローカルであり、それらをテストしても特別な問題は発生しません。 アップロードのあるリストの場合、フィルターは多数のチェックを伴う場合があります。 並べ替えについても同様です。
  6. リスト内の各アイテムの構成。 独自の内部ロジックを持つ基本テキストまたは画面全体のいずれかがあります。
  7. 要素との相互作用。 新しいアイテムの追加、削除、非表示、ドラッグアンドドロップ。
  8. すべてのデバイス間の同期をリストします。 例は、すべてのデバイスでファイルを変更した後のファイルの同期です。
  9. スクロール位置を保存します。 セクション間を移動するとき、またはリスト画面に戻るとき、これは非常に重要な機能です。 たとえば、これが投稿のフィードである場合。


リスト検索



  1. オンライン/オフライン検索。 オフライン検索は簡単です。 これは基本的にローカルフィルターです。 オンライン検索およびオンラインフィルターの場合、さらに多くのケースがあります。
  2. 文字検索または検索ボタンをクリックして検索。 文字ごとの検索では、リクエストの数に制限があるはずです。そうしないと、サーバーはアプリケーションからのスパムを無視し始める可能性があります。
  3. 検索文字列をクリアします。
  4. ヒントの存在。
  5. クエリ履歴の可用性。


入力フォーム



  1. フィールドのリストとその説明および機能。
  2. フィールドにデータを保存およびダンプするための条件。 いつ、どのフィールドで値を保持する必要がありますか？ いつクレンジングしますか？
  3. 文字の数と種類の制限。
  4. 選択したフィールドにデータを入力するためのキーボード。 キーボードのタイプ：数値または文字。 キーボードを開いたときにコンテンツをシフトする必要がありますか？ どのような条件下で閉じる必要がありますか？
  5. フィールド間の遷移のロジック。 ボタン「次へ」、キーボードの「次へ」。
  6. 誤って入力されたデータの検証。 サーバーまたはクライアントで確認します。
  7. 特定の条件下でのサーバーへの自動リクエスト。 たとえば、ユーザーが6桁の確認コードを入力した場合。


アカウント



  1. ユーザー登録と承認の要件。 モバイルアプリケーションを介さずに登録することは可能ですか？
  2. アカウントの回復。 たとえば、ユーザーがアプリケーションのパスワードを忘れた場合。
  3. ログアウト。 アカウントのデータクレンジング（特に、プッシュ通知通知のリセット）。
  4. SMSコードによる承認。 電話番号、その形式、可能な国のコードを含むケースを考慮に入れる必要があります。 受信に問題がある場合にSMSを再送信します。
  5. ソーシャルネットワークを介した承認 （下記の「ソーシャルネットワークを介した登録/承認」セクションの詳細を参照）。
  6. 同時に複数のデバイスの認証。 自動ログアウトまたはデータ同期処理。
  7. 無効な古いアカウントトークンのエラー処理。
  8. アカウント情報を変更します。 たとえば、パスワードの変更。


ソーシャルネットワークを介した登録/承認



  1. ソーシャルネットワークを介した最初の承認時にアカウントを作成します。
  2. ソーシャルネットワークからデータをダウンロードします。 ソーシャルネットワークで変化したときの同期。 たとえば、ユーザーとアバターの名と姓。
  3. モバイルアプリケーション、ソーシャルネットワーク、またはブラウザ/ウェブビューを介した承認。
  4. ソーシャルネットワークからのアプリケーションデータへのアクセスを拒否します。


ロールモデル



  1. 役割モデルの説明。 各役割で利用可能なアクションは何ですか？
  2. 異なる役割の代表者間の相互作用。 1つの役割の代表者間の相互作用。
  3. ある役割から別の役割へのユーザーの移行。 これにはどのようなアクションを実行する必要がありますか？
  4. さまざまな役割の代表者の推定割合。 まず、どの役割に注意する必要がありますか？


地図



  1. 最初のロードマップ。 どの領域をロードする必要がありますか？ マップをどこに、どの縮尺で配置する必要がありますか？
  2. 要素の読み込みと描画。 ダウンロードしたアイテムをキャッシュする必要がありますか？ アイテムはいつ更新する必要がありますか？ この瞬間を考慮して、データの高速読み込みとマップ上でのスムーズな移動を確保することが非常に重要です。
  3. マップ上の要素のロジック。 ピン、ピンの上のポップアップ、ピンのカード、ルートの構築。
  4. マップのスケーリング、回転、傾斜のサポート。
  5. アプリケーションが最小化されたときに、位置情報を更新し、現在位置の座標を送信します。


チャット



  1. メッセージの送信がインターネットに接続されていないか、インターネットに問題がある場合の処理​​ケース。 メッセージを再送信します。
  2. グループチャット。 チャット仲間を追加/削除するロジック。 対談者の最大数。
  3. メッセージの送信/受信のステータス。 ここでは、すぐにグループチャットに注意を払うことをお勧めします。 グループチャットの既読メッセージとは何ですか？
  4. チャット履歴をダウンロードします （リストについては質問をご覧ください）。
  5. 追加機能の存在 ：メッセージの削除、「<Insert_name> prints」の表示、送信されたファイルのプレビュー、メッセージの転送。
  6. チャットのメカニズム。 たとえば、テスト中にバグを特定し、問題が発生している面を特定するために、ソケットでのチャットの動作原理を理解する必要があります。


ファイルをサーバーにアップロードし、デバイスにダウンロードする



  1. ファイル形式。 システムはどのファイル形式を処理し、どのファイル形式でエラーを発生させる必要がありますか？
  2. 中断されたアップロード/ダウンロードを再開しています。 自動またはユーザーの確認後？
  3. アップロード/アップロードされたファイルの最大数。
  4. ファイルをダウンロードするためのデバイスのメモリ不足。 実際には、ファイルをダウンロードするだけでなく、データベースに書き込むためにも十分なメモリがない場合がありました。 このような問題を処理する必要がありました。
  5. ファイルの送信/ダウンロードをキャンセルします。
  6. ファイルを相互に置き換える。
  7. 外部SDカードにダウンロードします。
  8. アプリケーションが最小化されている場合、バックグラウンドでダウンロードします。


外部機器



  1. デバイスを接続/切断します。 アプリケーションと通信する通信チャネル（Wi-Fi / Bluetooth）。
  2. アプリケーションロジックに対する外部デバイスの影響。
  3. 外部デバイスの構成。 外部デバイスを管理するシステムはありますか？
  4. 相互作用が発生する最大距離。
  5. 信号強度/電力。 これらのオプションが何に依存するかを調べてください。 たとえば、金属缶にビーコンを隠すと、信号を失う可能性が劇的に増加します。
  6. 複数の外部デバイスを同時に接続します。 たとえば、あるデバイスから別のデバイスに切り替えると、奇妙なケースにつながる可能性があります。
  7. アプリケーションが最小化されているとき/画面がロックされているときの外部デバイスへの接続。


オーディオプレーヤー/ビデオプレーヤー



  1. サポートされているファイル形式。
  2. 再生可能なコンテンツのキャッシュ。 ユーザーの利便性のためにキャッシュする必要があるデータの量を必ず理解してください。
  3. バックグラウンドでプレイします。 アプリケーションが最小化されている場合、データの読み込みは必要ですか？
  4. システムカーテン内のプレーヤーの通知。
  5. Bluetoothヘッドセット、CarPlay、およびその他の外部システムとの統合。


クレジットカード払い



  1. プロファイルへのリンクと銀行カードの削除。 最低額のテストの撤回はありますか？ たとえば、1ルーブルがアカウントに返されます。
  2. タイカードによる支払い。 たとえば、後続の支払いでSMS確認の繰り返し要求がありますか？
  3. カードでバインド/支払いをしようとしたときのエラー処理。
  4. システム内に複数のクライアントが存在する場合のカードのリストの同期。 たとえば、WebバージョンとiOSバージョンがあります。
  5. カメラをスキャンして、カード番号を認識します。


時間、カレンダー、タイマー



  1. カレンダー/時間。 日付と時刻を誤って設定すると、アプリケーションロジックに影響しますか？ 期間を選択できますか？ 許容範囲はどのくらいですか？
  2. タイマー ローカル/サーバー？ サーバータイマーはどのように同期しますか？ たとえば、Androidでは、アプリケーションはデバイスにインストールされた時間ではなく、デバイスの起動時間に焦点を合わせる場合があります。 ユーザーがシステム設定でクロックをどのように変換しても、タイマーは失われません。


通知



  1. 通知のタイプ。 アプリケーションに接続されている特定のイベントの通知はありますか？ または、サーバーが送信する通知をプッシュしますか？
  2. 通知時に利用可能なアクション。 通知を通過するとどうなりますか？ 彼女を閉じますか？ 通知が古くて利用できない場合はどうなりますか？
  3. 特定のアカウントへの通知のバインド。 あるユーザーがログアウトし、別のユーザーがログインしたことをサーバーに示すアクションは何ですか？


もちろん、このチートシートは、モバイルアプリケーションのすべての可能な機能を網羅しているわけではありません。 私たちにとって、それはテスト分析を開始するための出発点として機能します。 要件をテストする際に役立つケースまたはチートシートのコメントを共有してください。
