国内のマルチクレットプロセッサはありますか？

この投稿では、ロシアのプロセッサを元のMulticletアーキテクチャで使用する取り組みについてお話したいと思います。 Embox RTOSをこのプラットフォームに移植することに興味があります。これにより、既にお話ししたSIP電話など、かなり多数のアプリケーションを使用できるようになります。



転送プロセス中に発生した問題と、これらの問題をどのように修正したかについてです。 おそらく、これはこのプロセッサを使用する予定がある人だけでなく、何らかの理由でc99およびgcc標準からc89標準に変更する必要がある人やgccコンパイラと互換性のない人にとっても興味深いでしょう。 また、結論として、このプラットフォームとのやり取りから個人的な感情を追加できるようにします。



ロシアの現実について



私たちが購入したキットは最高級で、かなりの費用がかかります。 もちろん、エルブラスほど高価ではありませんが、同じラズベリーとは比較できません。 しかし、それを忘れた場合、送信されたキットはかなりまともに見えました。 私の意見では、価格はほとんど手作業であるため、限られた当事者に起因するはずです。



モジュール写真




私たちが遭遇した別の特徴は、配達時間でした。 キットは2月に戻ってくる予定で、7月に届きました。 しかし、私たちの側からのいくつかのキック、マルチクレットからの保証と保証書-そして、ついに、待望のパッケージが到着しました。

非常に励みになる状況ではありませんが、少なくともコンパイラーはありました！ c89ですが、彼はソースコードと提供された例をコンパイルしました。 楽観主義は、FreeRTOSをこのアーキテクチャに移植することに関する記事をすでに見ていたという事実にも触発されました。 そのため、松葉杖では動き回ることができなかった困難な課題に突き当たるまで移動することにしました。



コンパイラの技術詳細



MultkletのコンパイラCはlccに基づいており、残念ながら、コマンドラインインターフェイスを含むすべての点で主流のgcc / llvm / iccと互換性がありません。 少なくとも1つのファイルをコンパイルしようとした最初の試みで、サポートされていないフラグがたくさん見つかりました。 最も厄介なのは、-include xhコンパイラオプションがないことで、このオプションには、コンパイル済みファイルの最初の行が#include <xh>であるかのように、前処理用のxhファイルが含まれます。



原則として、これは標準的な手法ではありませんが、この方法では、たとえば、生成された構成ファイル（および多くのファイルがあります）を転送することは非常に便利です。



次の違いは、コンパイラを一般に提供することでした。 実際のところ、lccは単なるコンパイラであり、gccはコンパイラ用のユーティリティ（ld、as、cc、..）のコレクションに過ぎません。 Gcc自体は、これらすべてのユーティリティの一種のラッパーとして機能します。より簡単には、コンパイラドライバーです。 つまり、gccを介してアセンブラーまたはリンカーを呼び出すのは正しいことです。gccは、すべてのパラメーターを、正しくインストールされた環境の対応するユーティリティに転送します。 一般に、LDとASを構築するために別々の変数を導入し、ホストマシンからARユーティリティを取得する必要がありました。 以前に-ld、-as、および-arをクロスツールの名前（mips-elfなど）に割り当てることが可能であり、それらは通常、対応するフラグを使用してSS経由で呼び出された場合、この場合、ユーティリティを明示的に呼び出す必要がありました。



もちろん、リンカーも驚きを示しました。



これらの最初のものは、中間オブジェクトファイルを作成するための-rフラグ（--relocatable）の欠如でした。 メタ情報を生成するときにこれらのオブジェクトを使用します。たとえば、システムの各モジュールの整合性をチェックするチェックサムを計算します。 まあ、一般的に、これは動的モジュールとライブラリを使用することになると非常に便利です。



もっともっと。 静的ライブラリの操作は貧弱です。 ライブラリは-lオプションのみを使用して転送できるだけでなく、フルパスを含めることもできません。 このため、同じ名前の異なるライブラリをリンクすることは基本的に不可能です。 さらに、-start-group /-end-groupのサポートの欠如は不快です。



また、コンパイラにすでに関連している、かなり興味深い不快な事実を追加することもできます。その中で-lオプションがオーバーライドされます。 lccでは、-lccdirの同義語です。

ソース変更



次に、ソース自体に進みましょう。 それらがc99またはgnucであり、コンパイラがc89であることを思い出させてください。 問題が発生したはずであることは明らかです。 最初の問題は、コンパイラーではなく、プリプロセッサーでも発生しました。 このコンパイラは、 mcppプリプロセッサを使用します。 一般に、可変数の引数を持つマクロはgccとは異なる方法で処理されることがすぐに判明しました。「テール」は空にできません。つまり、 1つの必須引数を持つマクロは、2つ未満の引数で呼び出すことはできません。 たとえば、assertなどのマクロを変更する必要がありました。 以前は、確認のための式だけでなく、エラーメッセージ行も受け取ることができました。これは非常に便利な場合があります。 原則として、これはマクロの標準的な動作ではないため、別のマクロassertf（）を作成し、発生したソースをコンパイルしませんでした。



コンパイラに戻ります。 インラインキーワードはc90で登場しましたが、マルチレットコンパイラではサポートされていません。 マクロの代替として静的インライン関数を広範囲に使用します。 出力はcompiler.hファイルの導入で、ifdefがインラインサポートの欠如を決定します。この場合、インラインは空のマクロ（#define inline / * nothing * /）として宣言されます。



コンパイラは、 指定された初期化子をサポートしません。 この問題はプリプロセッサの助けを借りて解決することはできません。また、古い標準をサポートするためにソースコードをダウングレードしたくはありませんでした。 c99-to-c89ツールが助けになり、その場でc99コードがc89に変わります。



また、コンパイラはgnucの拡張であるtypeofキーワードをサポートしていません。 たとえば、マクロminおよびmaxを記述するために使用します。 



/** @return the larger of @aa and @ab */ #define max(a, b) \ ({ \ typeof(a) __max_a = (a); \ typeof(b) __max_b = (b); \ __max_a > __max_b ? __max_a : __max_b; \ }) /** @return the smaller of @aa and @ab */ #define min(a, b) \ ({ \ typeof(a) __min_a = (a); \ typeof(b) __min_b = (b); \ __min_a < __min_b ? __min_a : __min_b; \ })


一般に、整数変数の場合、次のように簡単に記述できます。 



 #define max(a, b) ((a) > (b) ? (a) : (b)) #define min(a, b) ((a) < (b) ? (a) : (b))


これは、ヘッダーファイルのインクルードを単に破棄してそのように定義することにより、いくつかの場所で行いました。



最大の問題は属性でした。 コンパイラーは、アライメントの設定、構造のパッキング、特定のセクションへのデータまたはコードの配置をサポートしていません。 これらのことなしに行うことは非常に困難です。 たとえば、セクションでの作業を非常に積極的に使用しています。システムのロードでさえ、特定のセクションにポインターを配置するという原則に基づいています。 残念ながら、この問題を解決できなかったため、現在の段階では、システムの起動に関する行をシリアルポートに出力するだけに制限することにしました。



実際、マルチクレットアーキテクチャのサポートを開始したとき、すぐにいくつかの質問がありました。 今回、私はカンニングをすることにし、Habréで何かを書いたマルチレットスタッフと連絡を取りました。 本当にありがとうございます。 彼はすぐに私の質問に答えてくれ、彼のおかげで、マルチクレットアーキテクチャ用のstdarg.hを入手しました。 ブートコードはサンプルからも送信されましたが、原則として、マルチレットサイトからダウンロードすることもできます。 誤解もありましたが、それについては広めたくありません。 最終的に、彼らはマルチクレットフォーラムで質問する方が良いと判断しました。
