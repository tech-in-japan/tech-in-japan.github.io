Nutanix Big Update：NutanixOS 4.6





一部の「 マーケットリーダー 」が突然 「ハイパーコンバージェンスを発明」する一方で、Nutanixは5年前から市場で製造販売を行っています。 そして数週間前、Nutanixは大きくて重要なアップデートをリリースし、メジャーリリースを取り入れていますが、さらに大きく重要なアップデートを前もって計画しているので、私たちは恥ずかしがり屋で、これはバージョン4.6です



まず第一に、生産性の急激かつ大幅な増加について話す価値があります。 いいえ、彼らは「コードの忘れられていたデバッグスリープ10を削除しませんでした」、汚れはありませんでした;）開発者は、そのような印象的な増加の次の理由に名前を付けました。



  • 動的メモリ割り当ての減少
  • ロックオーバーヘッドの減少
  • コンテキスト切り替えのオーバーヘッド削減
  • 新しい言語機能とC ++ v11コンパイラーの最適化の使用
  • チェックサムのより詳細な計算
  • さらなる最適化のためのより高速なI / O分類
  • 改善されたメタデータキャッシング
  • 書き込みキャッシュアルゴリズムの改善。




そして、結果は、根拠がないように、そのようなものです。 これは、4つのノードを備えた最上位構成のテストシステムNX-3460-G4です。 ノードあたり2個のE5-2680v3b 512GB DDR4 RAM、ノードあたり2個のSSD 1,2TB、ノードあたり4個のSATA 2TB、合計4ノード4。

結論は、組み込みテストの結果です。 各ノードには、CentOS構成のVMがデプロイされ、1x vCPU 1GB RAM、7 vDisk、およびそれらの6つが大きなブロック（1MB）の順次読み取りおよび書き込みモードでfioを追跡し、小さなブロック（4K）でランダム読み取りおよび書き込みを行います

単純な列挙によるその他の新機能（詳細については、これらの機能のそれぞれについて個別のメモを作成する価値があります）：



Project Minerva内にある組み込みのNutanix分散スケールアウト、単一名前空間ファイルサービスSMB 2.1。 これにより、Nutanixは外部消費者のNASとして機能し、ファイルを「出力」できます。 外部ユーザー用のファイルストレージが必要な領域がまだあるため、多くの人がこれを要求しました。最も簡単な例は、VDIワークステーション用のユーザープロファイルのストレージです。 これで、Nutanixのみでこれを行うことができます。

Project Minervaは、NASサービスを備えた独立したVM（NVM、「NAS VM」）です。つまり、不要な場合はインストールせず、ディスクスペースとメモリを占有しません。必要な場合は、Nutanixを内部から展開してインストールします。 現在作成されているSMB 2.1は、おそらくNFSが将来追加されるでしょう。 SAMBA開発者の1人がそれに取り組んでいますが、MinervaはSAMBAではなく、単一の「名前空間」、単一の名前空間を維持しながら、クラスター内の無制限の数のノードに分散できるように書かれた「ゼロから作成された」ファイルサービスです。

Project Minervaとそのすべての技術的な詳細については、リリースに関して別の記事で説明しますが、Tech Previewのステータスで公開されているので、今すぐご覧ください。







4.5で導入され、CLIで構成されたボリュームグループは、Prism GUIから管理されるようになりました。 ボリュームグループは、Nutanix内で実行されているVMにブロックアクセスセクションを与える必要がある場合にVMで利用できるiSCSIブロックアクセスボリュームです。 そのようなアプリケーションの例は、ESXi上のMicrosoft Exchange、Windows 2008ゲ​​ストクラスタリング、Microsoft SQL 2008クラスタリング、およびOracle RACです。





このスクリーンショットでは、リリースでまだリリースされていないいくつかの機能がぼやけています。



Update 1-Click Upgradeは、BMCおよびBIOSプラットフォームを含めて機能するようになりました。 作業を停止することなく、フルタイムアップデーターを使用してサーバープラットフォームのBIOSを更新できます。 完全に自動的に、ホストプラットフォームはVMからネイバーに移行し、メンテナンスモードに入り、BIOSを更新し、新しいVMで再起動し、VMを復元します。その後、管理者の介入なしにクラスター内の他のホストで順番に実行されます。







Nutanixに組み込まれたハイパーバイザーおよびCVM-Foundationのファームウェアおよびアップロードサービスも更新されます。 これで、ユーザー自身が、Nutanixのエンジニアの関与なしに、ビルトインFoundationを使用して新しいノードをクラスターに追加できます。 工場からは、Acropolis HypervisorとFoundation Foundationが組み込まれたサーバーノードが提供されます。 ユーザーには、管理者のコンピューターから起動された小さなJavaアプレットがあり、クラスターではなく、ネットワークに接続されたサーバーノードを個別に検出します（各Nutanixには、シリアルから生成されたリンクローカルセグメントから一意のIPv6アドレスがあります）必要なものを入力できますESXiなどのハイパーバイザーを追加し、クラスターに追加されたノードを含めます。 便利で高速。







Metro Availabilityは、VMがDRサイトに移行した後にVMを再起動しないという待望の機会を受け取りました。 以前は、NFS記述子を新しいサイトとそのストレージ（データ自体が同期的に複製された）に切り替える必要があったため、これが必要でした。VMはNFSプロキシを介して動作し、動作を続けながら「利益」を移行できます。



もう1つの待望の機能は、セルフサービス復元です。 これで、VMユーザーはVMのスナップショットに保存されたデータを個別に復元できます。 これを行うために、いわゆるNutanix Guest ToolがVMにインストールされ、その助けによりVMはNutanixと通信し、スナップショットを単純なコピーからスナップショットデータを抽出できるVM内の別のディスクとしてマウントできます。 多くの場合、スナップショットに「ロールバック」するユーザー（たとえば、異なるテスト/開発システムを使用することが多い）は、この管理者の手間をかけずに、VM全体を復元せずに、必要な個々のファイルのみを復元することができます。



OpenStack、Nova、Cinder、Glance、Neutronの新しいドライバーとの統合があります。 統合は、特別なサーバーVM（SVM）を介して機能します。そのイメージは、Nutanixでダウンロードおよびインストールできます。 したがって、NutanixはOpen Stackからハイパーバイザーと見なされます。 OpenStackとの統合については、後で詳しく説明します。







cloudinitおよびsysprepスクリプトのサポートが追加されました 。 それは長い間要求され、行われてきました。 これで、作成または複製されたマシンに構成後のスクリプトを設定し、出力で個別の設定を使用してすぐに使用できるVMを取得できます。 数千人。 :)







異なるハイパーバイザー間でクラスター間のレプリケーションも可能になりました。 たとえば、ESXiの実稼働クラスターは、Acropolis Hypervisorの下にあるバックアップデータセンターのクラスターにデータを複製できます。 私はESXiが好きですが、DRのホストソケットにライセンスを払うのは好きではありません。年に数回使用されます-OK、AHVを使用します。
