モバイルプラットフォーム。 ReactNativeを恐れない方法

最初のブログ投稿を「モバイル」トピックに捧げ、アプリケーションの起動と作成のためのグローバルソリューションである「ESFモバイルプラットフォーム」の開発について話すことにしました。



これは何ですか これは、モバイルアプリケーションの大量作成、複製、および運用中の保守を可能にするマルチモジュール製品です。







それで、私たちは順番に、そしてポイントごとに何をしますか



製品開発



React Nativeライブラリを使用して独自のフレームワークを開発しています。 当社のフレームワークは、従業員向けのESFモバイルアプリケーションに含まれており、ネイティブコードを含むモバイルコアです。 モバイルビジネスアプリケーションはJavaScriptで開発されています。



「コンポーネント」または「サービス」と呼ばれるエンティティの典型的なコードフラグメントを取り出します。



現時点では、コンポーネントには3つのタイプがあります。



  • 視覚的なコンポーネント（様式化されたボタン、スイッチ、リスト、スイッチ、ブートローダーなど）;
  • 非ビジュアルサービス（ネットワーク、認証、ロギング、リモート構成など）;
  • モバイルウィジェット（「ダッシュボード」と呼ばれる特別なセクションに集約されたインジケーターを表示できるエンドツーエンドのコンポーネント）。


コンポーネントにはJavaScriptのブリッジラッパーがあり（ReactNativeメカニズムを使用）、JavaScript開発者はJS / TS、React + Reduxスタックを使用してモバイルクライアントとフロントエンドクライアントを作成できます。



各コンポーネントの背後には、ネイティブ実装があります。 Objective-Cをメイン言語として使用し、厳密にアーキテクチャにアプローチし、パフォーマンスと最適化にも注意します。



ReactNativeライブラリの組織には満足していませんでした。なぜなら、 分離され、階層化され、スケーラブルに見えなかったため、コンポーネントライブラリアーキテクチャを発明しました。現在、アーキテクチャは階層化され、VIPER、SOLID、SOAの基本ルールを順守しています。

実際、ReactNativeとVIPERを友達にしました。











Facebookが開発した標準コンポーネントはほぼ完全に放棄し、アプリケーションの構築に必要なコンポーネントを再実装しています。 なんで？



  1. コンポーネントは、ESFのユニークな単一の視覚スタイルを持ち、変更に適していない必要があります（プロパティを介した有効な設定のみ）。



  2. ReactNativeコードをテストでカバーすることはできず、その品質と正しい動作に責任を持つことはできません。



  3. フォークライブラリとそのさらなる開発は受け付けていません。



  4. アーキテクチャの理解はReactNativeの作成者によって共有されておらず、コンポーネントで使用されるコードの特定の部分を注入または再利用することはできません。



  5. ReactNativeはまだディープベータ版であり、動的に変化しています。 バージョンが変更されるまで、下位互換性の原則はサポートされておらず、Facebookからの更新のリリース後にライブラリのコードが正しくコンパイルされることを確認できません。



  6. このコンポーネントを他のプロジェクトやReactNativeコンセプト以外でも使用できるようにする「在庫」のネイティブコードが必要です。



ReactNativeを選ぶ理由



作成時には、次の原則を順守しました。



  1. 同じ技術スタックを持つ開発者がモバイルアプリケーションとフロントエンドアプリケーションの両方を開発できるソリューションを作成します。



  2. 以前に開発された機能をすばやく再利用できます。



  3. アプリケーションの開発とサポートの銀行コストを最小限に抑えます。



  4. 市場投入までの時間を短縮します。


ソリューションを分析して選択するとき、いくつかの出力パラメーターと基準がありました。



  • Web用に開発された銀行内の多数のレガシーシステム。



  • 結果として、JavaScriptコンピテンシーを持つ多数の開発者。



  • JS、React、およびReduxを活用する新しい有望なWeb製品。


最初のタスクの1つは開発スタックの統合とコストの最小化であったため、アプリケーションコードを記述するための言語としてJavaScriptを使用してモバイルアプリケーションを作成できるソリューションを選択し始めました。



CordovaとPhoneGapはすぐに姿を消した それらを使用して、出力で非ネイティブアプリケーションを取得します。



ReactNativeは、ランタイムのObjective-C呼び出しでJSコードを解釈します。これにより、ネイティブコードを実行し、UIViewの子孫からインターフェイスフォームを構築できます。



ReactNativeは、Reactの自然な継続であり、モバイルアプリケーションおよびデスクトップ上のフロントエンドクライアントで実行される統合コードを記述することができます。



テスト中



プラットフォームライブラリの本質的な品質の1つは信頼性であるため、開発中の製品を常にテストします。



フレームワークのすべてのPRおよびビルドで起動される自動ユニットテストがあります。



個々のコンポーネントをテストし、プロジェクトコードを手作業でデモする必要があると考えています。



近い将来、モジュール間の自動統合テスト、UIテスト、およびビジネスプロジェクトでの実装のためのパフォーマンステストが加わる予定です。



また、戦略計画には、クロスコードテスト（JavaScript <-> objc-C）を記述するための独自のフレームワークの開発、およびビジネスプロジェクトのコードを分析するための静的アナライザーの開発も含まれます。



プロセス自動化



私たちは何をするかを自動化します。 これで、個々のプラットフォームモジュール用のCIとCDができました。 企業のジェンキンズに接続された個別の物理ビルドマシンがあります。



テストの実行、コードカバレッジレポート、 dashing.ioでのインジケーターの更新、バージョンの構築、およびnexusでの公開は、現在自動化されているすべてのプロセスのほんの小さなリストです。



DevOpsを開発しており、CI / CDをすべてのアプリケーションプロジェクトと統合し、サポート作業、インシデント分析などを自動化する予定です。



スケーリング



現在、プラットフォームを使用するESF内のすべてのモバイルアプリケーションを構築するためのグローバルアーキテクチャと原則に取り組んでいます。



プロジェクト間で再利用できるように、アプリケーションコードと登録を独立したJSバンドルとして分離するシステムを開発しています。 JSバンドルはnpmパッケージの形式で配布され、コンパイル段階でアプリケーションに接続されます。



このアプローチにより、静的アプリケーションを作成し、アセンブリを通じて更新することができます。 これは問題を解決しません：ビジネスは、ユーザーの役割に応じてアプリケーションのJSバンドルとは異なる「セット」を見たいと考えています（売り手は1つの機能を持ち、ボスは異なります）。この問題は、 1つのバイナリ。



残念ながら、この場合、市場投入までの時間の要件に違反しています。 アプリケーションが大きいほど、特定のバージョンの欠陥を特定することを目的とした回帰テストが長くなります。



JS Bundleをモバイルアプリケーションに動的にロードするという概念が提案されました。 はい、非常にクールで革新的です！ しかし、これは開発、プロトタイピング、およびパイロットを必要とする新しいシステムです。 現在までに、このようなシステムは世界に少数しかありません（特にネイティブアプリケーション用）。



このコンセプトの開発をR＆Dで開始します。 このような野心的なストーリーに興味がある場合は、チームに参加する絶好の機会です:-)



どのように作業しますか？

2週間の配信サイクルと、すべての必須のスクラムアーティファクト（計画、スタンドアップ、グルーミング、デモ、レトロ）があります。



スタンドアップは、毎日問題を更新し、すべてのチームメンバー間で実装されている内容を共有するのに役立ちます。 重要：私たちは、1日15分以上スタンドアップに費やすことはせず、立ったまま過ごすだけです。



定期的にバックログを過大評価し、タスクの詳細を説明します。これにより、計画に費やす時間が短縮されます。 そして、レトロとデモは楽しくて積極的です（スクラムマスターが主催するコンテストやノミネートに参加しています）。



プロセスに加えて、製品の技術面を厳密に監視します。ドキュメント、ガイドを作成し、相談を行い、他のプロジェクトを支援します。



プロセスとアプローチを文書化し、それらを忘れないようにし、次回の活発な議論を最小限にします。
