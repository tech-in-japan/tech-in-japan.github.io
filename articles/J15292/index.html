オペレーターを介さずにユーザーの位置に関連付けられたモバイルサービス





ご存知のように、携帯電話ネットワークでは、ユーザー（加入者）の場所、またはむしろ電話の場所を特定することが可能です。 ロケーションベースのサービスは、LBS（ロケーションベースのサービス）と呼ばれます。 LBSの例として、 モバイル検索があります。これは、MTSユーザー向けのサービスで、子供、友人、車などがどこにあるかを知らせます。 多数の基地局がある場所 （たとえば、市の中心部）での場所の決定の精度は、100メートルに達する可能性があります。



あなたとあなたの友人（同僚、同志）がそのようなサービスを作成することに決めたとしましょう。そのサービスの主な機能は、ユーザーの位置を中心に構築されます。 多くの問題が発生しますが、その一部は技術的に純粋に解決するのは容易ではありません。 私はそれらを解決する別の方法について話します。それはいくつかの点でより複雑になりますが、確かに技術専門家によって解決されます。







これらのタイプのサービスを構築する際の問題は、サービスプロバイダー（つまり、あなた）にユーザー（このオペレーターの加入者）の位置に関する情報を提供するモバイルオペレーターがプロセスに常に参加していることです。 これは、オペレーターとの契約を締結し、そのサービスとの相互作用を規定し、最終的にこのサービスの料金を支払う必要があることを意味します。 さらに、オペレーターは、サービスプロバイダーがこのサービスを誰に提供しているかをユーザーが知らないことをユーザーに保証しようとします。 オペレーターは、特に狂信的で、可能な限りのことをすべて行い、サービスが正確に誰に提供したかわからないようにします。 サービスの提供は次のようになります。オペレーターはサービスプロバイダーに特定のキーと場所を提供し、プロバイダーは何らかのロジックを適用し、サービスの結果をオペレーターに返します。 サービスで提供されたユーザー（キー）に関する情報は、オペレーターによって保存されます。 そして、オペレーターが加入者に何らかの個人情報を提供することに同意したとしても、あなたが理解しているように、組織化されたプログラマーのグループは、集まってLBSに基づいて低予算のスタートアップを開発することはできません。オペレーターとの契約。 特定のアグリゲーターとの契約を締結する方がはるかに簡単です。これにより、オペレーターとの契約が締結され、機能の一部を第三者（つまり、あなた）に委任できますが、これはユーザーとサービスの間のチェーンを延長し、あなたは オペレータだけでなく、アグリゲーターとも共有します。 ところで、私が利益について話していることに気づいていますか？ はい、そのようなチェーンを選択する場合、ユーザーはお金を支払わなければなりません。そうでなければ、ユーザーはあなたに支払わなければなりません。 オペレーターまたはアグリゲーターがあなたと無料で仕事をしたいとは思わないでしょう。 オペレーターが1人ではないことを忘れないでください。ただし、多くのオペレーターがいます。ネットワークの加入者と協力したい場合は、それぞれのオペレーターと合意する必要があります。



ただし、この種のサービスを構築する別の方法があります。 セルラーネットワークを見ると、それは一連のベースステーションです。 各ベースステーションは特定のエリア、場所をカバーします。 各基地局には、各事業者の基地局に固有のいわゆるセル識別子であるセルIDがあります。 加入者の電話は、自分がどのセル（基地局）でカバーされているかを常に把握しています（セルIDを把握しています）。 ベースステーションには脚がなく、通常は家の屋根にボルトで固定されているため、そのアドレスはほぼ一定です。 探究心は、ユーザーが現在位置しているセルの識別子と、それを介してアドレス（〜近似値）を見つけることができるとすでに結論付けています。





タワーの基地局



次の問題は（そして、これは、オペレーターが関与しないサービスの最も基本的な問題です）、マッピング「セルID =アドレス」を見つけることです。 データベース「Cell ID = address」は携帯電話事業者のみが所有しており、この情報は一種の秘密です。

Web 2.0。テクノロジーは、Cell IDデータベース、つまり自分でこのデータベースを構築するユーザーをコンパイルする際に役立ちます。 ユーザーがいる現在のセルの識別子を表示する携帯電話（アプレット）のアプリケーションを作成できます。ユーザーは、最寄りの家の隅に表示される住所を入力できます。 セルIDとアドレスに関する情報はGPRSを介してサーバーに送信され、追加のポイントがユーザーに加算されます（必要に応じてカルマ）。 ギフト、お金、その他何でもユーザーを刺激できます。 都市の街路にいる1,000人のユーザーだけが、オペレーターの基地局に関する情報を収集できることが重要です。 ちなみに、西側には多くの非監視クラブがあり、参加者は自分の都市の基地局でデータを収集し、新しいCell IDに住所だけでなくその地域の写真も提供します。 そのようなクラブの例を次に示します （ http://gsmloc.org ）。 Net Monitoring Clubは、サンクトペテルブルク（ http://www.netmonitor.ru ）でかなり発展した形式です。





基地局カバレッジ



しかし、私の意見では、Cell IDデータベースを作成するための最良のソリューションは、非モニターを引き付けるのではなく、何らかのサービスの既製のアプリケーションでベースステーションに関する情報を直接収集することです。 たとえば、デートサービスを考えてみましょう。 最初の段階では、セルIDでユーザーの位置を特定することはできませんが、ユーザーがいる通りに入るようにユーザーに要求することはできません。 アドレス情報とともに、基地局の識別子とユーザーオペレータの識別子がサーバーに送信されます。 したがって、サービスが提供され、次のセルのアドレスが取得されます。 もちろん、そのようなデータベースはすぐには組み立てられず、ユーザーからの情報を処理するのが困難になります。通りの名前はエラー、音訳などで入力できます。 ただし、後でデータベースを「コーム」することができ、アドレスの入力をユーザーに求めたり、少なくとも頻繁にそれを試行したりする必要はなくなります。



データベースが組み立てられ、このデータベースがほぼ最新の状態に維持されるとすぐに（私が言ったように、基地局は移動しませんが、ネットワークはまだ「呼吸」しています-識別子が変更され、新しい基地局が表示され、古い基地局が消えます）、オペレーターの関与なしに、独自のロケーションベースのサービスを構築できます。 例えば、私が上で書いた出会い系サービス。



想像してみてください：

初秋、土曜日、明るい太陽と黄色の木、正午、モスクワ。 2年生のミハイルは、テアトラナヤ地区のモスクワの通りに行き、携帯電話を取り出します。 クマにはアプリケーションが含まれ（「テベドール」と呼びます-これが実際の名前になります）、「知りたい」という項目を選択し、彼の偏狭なパラメーターを入力します。 この時点でGPRSセッションが開き、ユーザーID、パスワード、その他の認証情報、およびセルIDがサーバーに転送されます。 サーバーはセルIDを受信し、この識別子がデータベース内にある場合、Mishaのアドレスが決定され、そうでない場合、サーバーはMichaelにアドレスを示すように要求します。 ミシャの住所がわかったので、サーバーに残っているのは、ミハイルが指定したパラメーターを持つ少女を見つけることだけです。 説明が似ている少女は、彼とほぼ同じ場所にいて、さっきほどサーバーにチェックインしました。 それだけです サーバーは、Mishaと切望している人を接続するだけです。 ミシャのプロフィールを適切な女の子に渡します。彼女が会議に反対していないことを確認した場合、両者はチャットなどで接続されます。 ミーシャと少女は、お互いに会うための2つのステップを持っています。 これはサービスのおおよそのロジックです。 それは異なるかもしれませんが、本質は明確でなければなりません。




このようなプロジェクトは、4人のプログラマからなるチームの手の届くところにあります。Symbianプラットフォーム用のCrazy c ++プログラマ。 Windows Mobile用のC＃プログラマー。 j2meプラットフォーム用のJavaプログラマー。 PHP（Ruby、Python、Perl、ASP .NETなど）はサーバー側のプログラマーです。 ところで、Symbianのプログラマーは必要ありません。 j2meアプリケーションは、このプラットフォームでうまく機能します。 これらはWindows Mobileで動作しますが、このオペレーティングシステムでのこれらのアプリケーションの外観には、多くの要望が残されています。



ご存知のように、LBSは何にでもねじ込むことができます。 最後に、最寄りのカフェ、映画館、はい、トイレを検索します。 なぜデートサービスについて話しましたか？ VAS（ 付加価値サービス ）市場を見ると、最も人気のあるサービスはメロディーや写真、つまりデートサービスの順序ではないことがわかります。 彼らの聴衆は実に広大であり、欲望の対象に到達する通りで直接会う機会は魅力的な申し出です、私の意見では。 これでお金を稼ぐことに関しては、コンテキスト広告がここで機能します。 たとえば、Coffee Houseが広告の代金を支払った場合、「お互いを見つけた」カップルを簡単に招待して、最寄りのカフェに座り、場所によって識別できます。 多くのオプション。 ユーザーの電話に広告を表示できます。 彼らが今やっているように、あなたは彼らにお金を求めてトップのプロフィールを上げることができます。 多くのオプション。



実装のために。



Windows MobileでCell IDを取得する-既製の例があります。 すべての電話で機能するわけではありません。 たとえば、WM2003ではQtekで動作しませんでしたが、Megafonではしばらくすると、Cell IDを「アンフック」するようになりました。 近所の人からWM5でQtekを受け取りました-準備ができて起動しました。 リンクを提供したフォーラムには、テストされ動作するモデルの説明があります。 C＃（.NETCF、OpenNETCF（必要な場合））およびC ++は、それぞれ無線モジュールとの通信に使用されます。 C ++に関しては、すでにアセンブルされたdllがこのフォーラムにあり、ソースコードもあります。



SymbianでCell IDを取得します。 プロジェクト「CellTrack」があります。 サンプルをダウンロードして実装について議論できる開発者フォーラムがあります。 mobilab.ruには、Cell IDを取得するためのプログラムのソースコードに関する記事があります。 Windowsとは異なり、Symbianではすべてが長い間安定しています。



j2meでセルIDを取得します。 セルID定義は、CLDC 1.1をサポートするデバイスで使用できます。 Location APIライブラリセットは、float型を使用して座標を表します。 Sunには、LBSに関する記事とライブラリ仕様へのリンクがあります。 連中がCell ID定義を別のライブラリに広げて、実際に座標を取得すれば、おそらくすべてがCLDC 1.0でもうまく機能するでしょう。 残念ながら、彼らはすべてを1つのライブラリに結合しました。 別の方法は、 " System.getProperty（" CellID "）を使用することですが、フォーラムで判断すると、この方法はすべてのモデルで機能するわけではありません。



サーバー側については、すべてが明確であり、調査は必要ありません。 私たちの場合、携帯電話は別のhttpクライアントにすぎません。 電話のリクエストを受け取って彼に答えるのはアマチュアです。 linux + apache + nginx + memcached + php5 + mysql5などです。 もちろん、詳細を考えると、html + jsだけでなく、携帯電話用のWebサービス、wmlおよびxhtml（mp）での追加出力も開発する必要があります。 Webサービスに関しては、WMデバイスの参加を考えると、プロトコルの1つはSOAPである可能性があります。 SOAPを使用してVisual Studioでアプリケーションを開発するのは簡単なことです。 他のプラットフォームに関しては、たとえばJSONに基づいた独自のものである可能性があります（解凍用のライブラリはj2me用であり、もちろんSymbian用のc ++です）。



したがって、携帯電話ユーザーの位置を特定することに基づいて、オペレーターを介さずにモバイルサービスを開発することは非常に可能です。 がんばれ！
