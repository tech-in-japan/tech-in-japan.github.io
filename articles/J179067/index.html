どこから始めますか？ 当然、ハードドライブダンプを削除します。 これはLinuxで1つのコマンドで実行されます。

dd if=/dev/sdX of=./dump.bin bs=1M





お茶を飲みに行きます 約10〜15分かかります。 ウィンチェスターは、160ギガバイトと250ギガバイトで使用されます。

実行中のハードドライブで何もしないでください！ 常に画像を使用する必要があります！



画像が撮影されます。 cfdiskを実行してみましょう。





何が見えますか？ ext2ファイルシステムを備えた2つのパーティションと、ディスクの先頭にある疑わしい未割り当て領域。 ファイルシステムはすぐに、UNIXのようなものが内部にあるという考えにつながります。もちろん、Linuxである可能性が高いです。



仮想マシンで実行してみましょう。 私はqemuが大好きなので、使用しています。

これは何ですか 説明します。

0-1BEでは、MBRブートローダーが配置されます。これは、400から始まる最初のディスクからstage2ブートローダーをロードする最も単純なものです。最も興味深い行は300-32Fにあり、ハードドライブ、ファームウェアバージョン、およびモデルのシリアル番号に非常に似ています。 しかし、推測できることは何ですか;）

説明 データ
シリアル番号（20バイト） （12スペース）6VMQWT47
ファームウェアバージョン（8バイト） CC46（4スペース）
モデル名（40バイト） ST3160318AS（29スペース）
私は簡単な方法で進みました。stage2-loaderのアルゴリズムを分解するのではなく、環境変数DRIVE_MODEL、DRIVE_SERIAL、DRIVE_VERSIONからハードドライブデータを取得するために、qemuに小さなコードを追加しただけです。 また、qemuを使用すると、ゲストマシンのメモリをダンプできます。これは便利です。 



 valdikss@valaptop:~/ % DRIVE_SERIAL=" 6VMQWT47" DRIVE_VERSION="CC46 " DRIVE_MODEL="ST3160318AS" qemu-system-i386 disk.img -monitor stdio QEMU 1.4.1 monitor - type 'help' for more information (qemu) dump-guest-memory mem.bin (qemu) quit

実際、すべてのLZMA応答は偽です。 彼は魔法の数を持っていません、ELFの半分も偽物です。 ただし、235個のファイルを含むCramFSは実際のファイルと非常によく似ており、高い確率でinitrdとして使用されます。

メモリから正しいCramFSを取得するのは簡単です。 CramFSがファイル形式でハードドライブに保存されたら、アンパックを試みます。 firmware-mod-kitパッケージのcramfs-2.0を使用しました。 

 valdikss@valaptop:~/ % cramfsck -x root cram1.bin cramfsck: crc error

それだけです。 /etc/init.d/runのようなテキストファイルは正常なようです。 ただし、実行可能ファイルとライブラリは起動されません。 最初は、ゲームは修正されたカーネルまたは修正されたlibcのいずれかを使用すると考えました。 実行可能ファイルにパッチを適用し、hello worldとどの程度異なるかを観察しました。 それらはすべて、 セグメンテーションフォールトで終了するか、さらに悪いことに、 違法なハードウェア命令で終了します。

参照「hello world」をコンパイルしました、なぜなら 最初から__libc_start_mainを呼び出して、どこか間違った方向に飛んでしまいました。 それから彼は再ロックで罪を犯した、なぜなら いくつかはそのままでbeatられ、おそらく修正が含まれていると示唆しましたが、いいえ、再ロックを編集した後、ゲームは開始しようとしましたが、そうではありませんでした。 約3日間考えました。 私は、実行可能ファイルが何らかの形でCramFSを破り、テキストファイルは元の形式のままであるという結論に達しました。 そして彼は正しかった！

なぜなら 元のCramFSを手元に持っていて、Alt + F2を押すだけで仮想マシンのOSコンソールにアクセスできることを思い出して、元の/etc/init.d/runを置き換え、xinit呼び出しを/ bin / shに置き換えてみることにしました。 しかし、変更されたCramFSを何らかの方法でイメージにプッシュする必要があります。 ハードドライブのイメージで「Compressed RamFS」という行を探して、暗号化されていないことに少し驚きました。 もちろん、それだけではなく、「パディング」によって、ハードドライブ上のファイル全体が32KBのデータのブロックと512バイトのvoidに分割されます。 まあ、それは問題ないようです！



たまたま/etc/init.d/runだけでなく/etc/init.d/onceも変更して、mount_tab、mountrd、およびusbdaemonの呼び出しをコメントアウトし、呼び出しのループに注意を払っていませんでしたmount_tab。



そこで、/ etc / init.d / runを編集し、CramFSをアセンブルし、イメージに書き込みました。 さて、元の/etc/init.d/runでFSを収集し、ファイルの違いを探し始めました。 結局のところ、それらの多くがあったので、私は問題が何であるかを理解しようと長い間試みましたが、firmware-mod-kitのcramfs-2.0は少し異なる収集をすることがわかりました。 さて、リポジトリから通常のcramfs-toolsをダウンロードして収集しましたが、突然、チェックサムに加えて、ファイルの最後のバイトが異なることがわかりました。 元のファイルには0x80があり、もちろん収集したファイルには0x00があります。 この状況に驚いて、彼は自分のファイルの最後のバイトを置き換え、ハードドライブとURAのイメージに入れました！ イメージは修正されたFSから始まり、コンソールを取得しました。 ファイルをコピーしても複雑さは発生せず、スタートアップファイルは実際にLinuxシステムで実行されました。このステップで、実際にCramFSが実行可能ファイルを具体的に変更することに気付きました。



IDA PROのmount_tab、mountrd、およびusbdaemonをざっと見てみると、その時点でmount_tabがファイルシステムを適切な場所（ハードドライブ上の2つのパーティション、/ mnt / gameの2つのパーティション、/ SETTINGSの2番目のパーティション）、mountrd AGPおよびPCIバスをスキャンし、復号化してメモリにロードし、最初のセクションにあるファイル「p」、「i」または「u」の1つを/ usr / libにマウントします。これにはビデオカードのドライバーが含まれ、usbdaemonはUSBでの作業を提供しますUNIXソケット経由のドングル。



偽りの喜び



私は座って、ゲームの起動ファイルを調べていますが、それをコピーして取り出したので、疑いが忍び込み始めました。 どういうわけかそうではなく、私が探求しているゲームのバージョンについての言及はなく、一般的に、何らかの形で何かが正しくありません。 このファイルはゲームの異なるバージョンに属していることを認識し、明らかに、USBドングルをチェックした後、彼はどこかから別のファイルを取得して復号化すると仮定しました。 すべてがずっと面白くなりました。

mount_tabを呼び出すときのループを覚えていますか？ 最初の起動では、ファイルシステムを実際にマウントしますが、次の起動では、最初のオフセットのヘッダーに応じて、オフセット0x1F80200または0x2080200から一定量のデータをハードドライブから読み取り、このデータを復号化し、CramFSゲームの起動ファイルを直接置き換えます正しいもの。 私はこの動きが本当に好きでした！ 私は激怒しましたが、同時に開発者の機知に誇りを持っています。
