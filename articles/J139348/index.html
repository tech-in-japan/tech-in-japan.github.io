RealSync-Web開発者向けの一方向のリアルタイムソース同期

パブリックユーティリティRealSync （GPL）の紹介。 彼女の使命は、Webプロジェクトファイルを検索/編集するときに、Sambaネットワークフォルダーの遅延に定期的に苦しむ人々の作業を促進することです。 RealSyncのアイデアは、使い慣れたIDEでローカルマシンのサイトファイルを操作し、RealSyncがリアルタイムで変更をコピーするリモート開発 Webサーバーで、以前と同様に結果を確認することです。 その結果、たとえば、IDE内のすべてのファイルで検索を実行できます。これらはローカルであり、Sambaを介してネットワークフォルダーを介して接続されていないため、検索は非常に迅速に機能します。 一方、Ctrl + Sは、ネットワークフォルダーを介して作業しているときのように、即座にサーバーに到達し続けます。



RealSyncはWindows、MacOS、Linux用のユーティリティであり、 リモートコンピューターにリアルタイムでファイルの正確なコピー （たとえば、PHP、Python、Rubyなどのスクリプト）をローカルコンピューターのフォルダーから格納できます 。自宅で仕事をしています。 ローカルフォルダーで行われたすべての変更は、これらの変更の数と行われた方法（少なくともIDE、少なくともメモ帳またはFar）に関係なく、ほぼ瞬時に（約0.2秒の遅延）サーバーに到達します。



RealSyncとその類似物の主な違いは、インターネット接続、再接続、およびタイムアウトの不安定性に対して非常に強いことです。 この場合、SSH接続が使用され、ユーティリティを最初に起動したときにアクセスが自動的に構成されます（つまり、キーを操作する必要はありません-設定はインタラクティブに行われます）。



実際、偶然にRealSyncを「殺す」ことはほとんど不可能です。 トレイで常に最小化し、その存在を忘れることができます（CPUはほとんど食べません）。 接続が長時間切断されていることをユーティリティが認識すると、使い慣れたRSYNCアルゴリズムが自動的に起動して、多数の相違点をすばやくコピーします。 リアルタイムモードでは、独自のプロトコルがSSH経由で適用されるため、エディターでCtrl + Sを押すと、サーバー上の変更がすぐに表示されます。 ファイル転送には、心地よい「くすくす」（設定で必要に応じて無効にできます）が伴います。また、一時的に接続が失われると、アイコンが赤くなります（接続が復元されると、アイコンは再び灰色になり、RealSyncは蓄積された変更に「追いつきます」）。



そして、Samba、Denver、またはXAMPPがあるのに、なぜこの自転車なのでしょうか？



一般的に、Webスクリプトを開発するにはいくつかの方法があります。 最初の方法は、ローカルWebサーバーを使用することです。 この方法には、多くの利点（制御の強化、移植性の向上、クロスプラットフォームなど）と、次のような多くの欠点があります。ローカルサーバー構成と開発ゾーンおよび運用ゾーンの構成の潜在的な違い、ローカル同期を監視する必要性SQLデータベース、またはインターネット上の単一の処女データベースへのアクセスが遅くなるまで待つなど。 この記事では、この方法については考慮しませんが、確かに生命の権利があります（同じデンバーの少なくとも100万人の登録ユーザーが適用されます）。



2番目の方法は、会社のオフィス（またはデータセンターのテストマシン）にインストールされ、通常の管理者がサービスを提供するWebサーバーを使用することです。 このようなWebサーバーには、開発者ごとに1つずつ、複数のディレクトリが存在する場合があります。 各プログラマは、他の人を邪魔しないように自分のディレクトリで作業し、サブドメイン（またはポート番号）で結果を確認します。 ここでは、この方法を検討します。



Web開発にリモートフォルダーを使用する



結局のところ、リモートサーバー上のディレクトリに何らかの方法で「アクセス」する必要があります。 最も一般的なケースでは、Sambaを介してネットワークフォルダーとしてローカルマシンに単純に接続されます。 この方法には、同時にいくつかの欠点があります。

  1. 許容できる速度で作業するには、サーバーと同じローカルネットワーク上にいる必要があります。
  2. ただし、この場合でも、IDEで大規模プロジェクトのすべてのファイルの検索を開始するのは非常に困難です。遅延が大きすぎます。
  3. 接続が切断された場合（または、禁止されている場合、インターネット経由で自宅で仕事をしています）、奇妙な影響や不快な遅れに備えてください。
不便さ（2）を回避して、SSH経由でサーバーにアクセスし、コンソールでgrepを使用してファイルを検索する多くの人々を知っています。彼らはIDEで動作します。 私には不便さの高さのようです。 さらに、phpStormでは、たとえば、非常に大きなプロジェクトのローカルファイルをすべて検索すると、grepよりもはるかに速く動作し、1〜2秒かかります（それがどのように行われたかは謎です。明らかに、事前にインデックスを作成しています）。 個人的に、私はphpStormを常に快適に使用してローカルファイルを検索しています-1日に平均50回。
もちろん、サーバー上の直接SSHコンソールでVIMを操作したい人もいます。 私は彼らを尊敬しますが、彼らのキャンプに参加することはまずありません。結局のところ、優れたグラフィックIDEによって提供される便利さは私に勝ります。



概して、RealSyncはIDE速度の問題の解決策です。



そのため、検索速度の問題を解決するには、ソースをローカルディスクに保存し、すでに検索する必要があります。 そしてすぐに疑問が生じます。Ctrl+ Sを押すと、ソースコードはどのようにリモートWebサーバーに到達しますか。 いくつかの可能なオプションがあります。

  1. ほとんどすべてのIDEは、Ctrl + Sを押して、変更されたファイルを（FTPまたはSSH経由で）リモートサーバーに転送することをサポートしています。 しかし、問題は2つしかありません。 まず、IDEによってファイルを変更することはできません（たとえば、ローカルマシンのコマンドラインからgit pullを作成します）。 第二に、数日間インターネットの現実から姿を消して、変更されたファイルを返すと、IDEはどのファイルが変更されたかを判断できなくなります（特に、何らかの理由でサーバー上で誤って変更された場合）。せいぜい30分で完全なコピーが開始されます。
  2. RSYNCの永久サイクルで実行します。 これは、プロジェクトがそれほど大きくない場合、インターネットが良好な場合、およびWindowsを使用していない場合（Windowsでは、起動時にRSYNCが非常に遅い）、多少なりとも機能します。 それ以外の場合はすべて、2〜3秒の遅延に備えてください。
  3. RealSync、Unison、WinSCPなどの特別な同期ユーティリティを使用します。
RealSyncは、コミュニケーションが不十分な状況で安定した動作を提供し、プログラマの注意を必要としない（「実行して忘れる」）ツールとして考案されました。 多数の類似物とは対照的に、これが判明した方法です。



また、双方向の同期が必要な場合は？



これは、サーバー上のフォルダーに直接加えられた変更をRealSyncが常に上書きするという事実に関連する非常に一般的な質問です（すぐではない場合は、次回の再接続時に必ず）。 ローカルフォルダー内の変更のみが意味があり、優先されます。



しかし、双方向の同期が必要になる理由について考えてみましょう。

  • たとえば、サーバーのコンソールでGitを使用しますか？ しかし、なぜですか？ 結局のところ、マシンにGitを配置して、ローカル作業とGitのさまざまなGUIのすべての利点を享受できます。
  • スクリプトはドキュメントディレクトリ内のローカルフォルダーに書き込みますか？ このフォルダーをRealSync例外に追加します（これは、構成内の1行または最初の起動時に簡単に実行できます）。 さらに、ドキュメントディレクトリをスクリプトで書き込み可能にすることはアンチパターンです。
  • 自宅のラップトップと別のデスクトップで作業していますか？職場で、Webサーバーフォルダーを介してファイルを同期しますか？ ただし、Webサーバーは同期とバージョン管理の手段ではありません。 より良いDropboxを使用するか、バージョン管理システムにコミットします。
  • サーバー上の作業コピーでシンボリックリンクを積極的に使用していますか？ まあ...しかし、それらの使用をやめようとしませんでした、少なくとも将来的にサポートと展開を簡素化しますか？
一般的に、RealSyncはシンクロナイザーではなく、 レプリケーターであることを理解する必要があります。 Webサーバー上のフォルダーを独立したものとして認識するのをやめ、それを単なる自動「ミラー」と見なし、SSHに手を入れるのをやめた場合（実際、これを行う理由は何ですか？）、すべてが適切に分類されます。 そして、ほとんどの場合、双方向の同期は単に必要ではないことがわかります（しかし、あなたがまだ違う考えを持っているなら、Unisonを使用してください-それが根を張るなら、もちろん）。



RealSyncのインストールと最初の起動



すべてが非常に簡単です。最初にRealSyncをダウンロードしてどこかにコピーする必要があり、次にどこか（スタートアップまたはデスクトップ）で次のコマンドラインを使用してショートカットを配置します。 



 「C：/ Program Files / dklab_realsync / realsync.exe」フォルダーソースへのフルパス
     -またはMacOSおよびLinuxの場合-
 perl / opt / dklab_realsync / realsyncフォルダーソースへのフルパス


最初に起動すると、サーバーアドレス、SSHログインとパスワードを対話的に入力する必要があるウィンドウのようなものが表示されます（パスワードはSSHプロセスによって排他的に要求され、ローカルマシンには保存されません、以下を参照）、リモート上のフォルダーの名前サーバーなど





SSHキーを使用してパスワードなしでサーバーに既にアクセスしている場合、初めてでもパスワードは要求されません。 まだ持っていない場合、RealSyncは公開鍵と秘密鍵自体を生成し、公開鍵はサーバーに依存し、秘密鍵はローカルマシンのホームディレクトリ（RealSyncが認識する特別なフォルダ）に生成されます。
残りの構成設定はファイルFullPathToYourYourFolderSource / .realsyncに書き込まれるため、 次回起動時に質問は表示されませんが、同期が開始されるだけで、プログラムはトレイに最小化されます（Windowsの場合）。



これで、任意のソースを編集し、それらが即座にサーバーに到達することを確認できます。 トレイで起動したRealSyncのアイコンをクリックすると、おおよそ次のウィンドウで作業プロセスを確認できます。





Windows用RealSyncディストリビューションは自己完結型です。すでにRSYNC、SSH、Perlのミニバージョン、およびその他の必要なユーティリティが含まれているため、realsync.exeを実行するだけでよく考えてください。 MacOSおよびLinuxでは、原則としてこれらのユーティリティはすべてシステム上にすでに存在するため、これらのユーティリティが使用されます。
そのため、RealSyncはトレイでハングします。決してオフにしないため、リモートフォルダーを備えたローカルマシンで作業しているような完全な錯覚に陥ります。 同時に、ローカル検索とIDEの操作のすべての利点があります。 それがRealSync開発の目的でした。
