いよいよDebianとUbuntuでのパッケージングのトピックに戻る時が来ました。

言い訳から始めましょう。 4番目とこの部分の間の長い休止の理由は、私の怠laz、私が終了できない卒業証書、星の位置でもなく、以前の記事（1、2、3、4）で言及した迷惑なバグではありませんでしたDebianパッケージ。 最後に理解したように、このバグの迅速な修正を期待するべきではないので、すでにパッケージのアセンブリを習得している人に、パッケージのテストと異なるアーキテクチャ向けのビルドの両方を促進する興味深いツール、pbuilderを研究することをお勧めします。



pbuilderは、インストールしたばかりのDebianまたはUbuntuのイメージを最小限の構成で、つまり最も必要なパッケージのみで保存するアーカイブを作成できる興味深いツールです。



なぜこれが必要ですか？



あなたのsrcパッケージが未知のシステム管理者Vasily Dudochkinによってダウンロードされ、彼のmipselアーキテクチャ用のパッケージをビルドしたいと想像してください。 期待して手をこすりながら、大切なdpkg-buildpackageコマンドを入力すると、ビルドエラーが発生します。 Dudochkinはdし、プログラムの作者を呪い、実質的なバグレポート（「準備中です、助けて!!!!!!」）をアップストリームトラックに書き込みます。 それは、パッケージのBuild-Depends列でgnutls-devを指定するのを忘れたためです。これはDudochkinにはありません。

このような場合、pbuilderは私たちを保存します-Build-Dependsで指定されたすべてのパッケージをダウンロードしてインストールし、ソースパッケージをコンパイルしようとします。これも重要であり、最終的にあなたのイメージを変更せずに保存します。



このアプローチの欠点は何ですか？



2つのマイナスですが、かなり大きいです。

1つ目は、パッケージにsudoまたはrootパスワードが必要なことです。 自宅のマシンで作業している間-すべてが正常で、すべての権利がありますが、何らかの外部サーバーで収集すると、問題が発生する可能性があります。 率直に言って、私はpbuilderをrootなしで動作させようとしましたが、fakerootを完全に実験する忍耐を持っていませんでした。

2番目のマイナスは最高速度ではありません。 実際、システムイメージは.tgzファイルに保存されます。 したがって、一般的なパッケージアセンブリアルゴリズムは次のようになります。イメージのアンパック-アセンブリのキャッシュにないパッケージの収縮-アセンブリ用のパッケージのインストール-パッケージのアセンブル-すべてを正方形に戻します。 このため、アセンブリにはかなりの時間がかかりますが、そのほとんどはtgzイメージの展開です。

次に、何らかのシステムのイメージを作成する必要があります。 たとえば、次のように：

$ sudo ARCH=i386 DIST=hardy pbuilder --create





ここで、私が言及したバグが現れます。 Debianシステムでは、cdebootstrapはUbuntuイメージの作成でひどく動作します。特に、カテゴリカルに私から勇敢なものを収集することを拒否し、すでにすべてのDebianディストリビューションを利用できました。 したがって、この記事は、ubuntuパッケージをビルドするようにubuntuでpbuilderを設定したときにのみ記述されます。



したがって、私たちは自分のためにさまざまなシステムのパックを作成することができ、それをさらに楽しみに使用することができます。 たとえば、このシステムは/home/torkvemada/pbuilder/hardy-i386-base.tgzにあります。



この画像で今できること



彼と一緒に面白いことができます。 すべてが開始された最初のパッケージは、パッケージのアセンブリです。 ソースパッケージがある任意のディレクトリに移動して、収集を試みます。

$ cd mypackage-1.0.19/

$ sudo DIST=hardy ARCH=i386 pdebuild





イメージがアンパックされるまで待機し、アセンブリに必要なすべてのパッケージをダウンロードし（キャッシュが空の状態でしばらく時間がかかり、これらのパッケージがキャッシュから配置されます）、それらをインストールしてからアセンブリを続行します。 すべてが終了したら、パッケージが正常にアセンブルされていれば、見つけることができます。

$ ls ~/pbuilder/hardy/result/*.deb

/home/torkvemada/pbuilder/hardy/result/mypackage_1.0.19-1_i386.deb





他に何をする？ 画像の内部で何が起こっているのかをまだ見ることができますか？ 入力する：

$ sudo pbuilder --login --basetgz ~/pbuilder/hardy-i386-base.tgz





出来上がり！ 私たちはイメージの内側にいます。心から望むものは何でもできます。ただし、多くの人に愛され、テキストエディターからのみ送られているMidnight Commanderもいないことに留意してください。 まあ、それを信じてはいけませんが、私はそれを設定の編集に非常にうまく使用しました。

すでに述べたように、すべての実験はシステムを破壊しません。「exit」と入力して終了すると、コマンド履歴が保存されない場合でも、イメージは元の状態に戻ります。 ただし、画像の内容を変更する必要がある場合に状況が発生することがあります。 たとえば、私のパッケージの1つには、PHP5の完全なソースが必要です。 この問題を回避する方法がわかりません。そのため、パッケージはアセンブリのソースデータを非常に具体的な方法で単に見ています。 pbuilderでソースディレクトリを保存する方法は？ 入力する：

$ sudo pbuilder --login --save-after-login --basetgz ~/pbuilder/hardy-i386-base.tgz





これで、変更後のすべてがリリース後も保持されますので、ご安心ください。 そのため、この機能は非常に慎重に使用する必要があります。



また、特にDebianの不安定なイメージの所有者が幸せな場合は、システムイメージ内のパッケージのバージョンを更新する必要がしばしばあります。 このニーズを満たすチームは1つだけです。

$ sudo pbuilder --update --basetgz ~/pbuilder/hardy-i386-base.tgz







また、外部レポジトリ、たとえば、第4回の記事で説明したホームのrepreproを使用する必要があるかもしれません。 これを行うには、イメージを作成する前に〜/ .pbuilderrcのOTHERMIRRORで行のコメントを解除する必要があります。 複数のミラーが追加されている場合があり、それらは垂直バーで区切る必要があります。 ミラーがネットワーク上ではなく同じコンピューター上にある場合は、BINDMOUNTS行のコメントを解除して、マウントされたリポジトリへのパスを示す必要があります。

サードパーティのリポジトリの必要性を認識した時点で、接続せずにすでにイメージを作成していた場合、これは悲しいことですが、2つの選択肢があります。 1つ目は、構成を修正してイメージを再度作成することです。 cdebootstrapは再び長い間インターネットからパッケージをダウンロードしますが、どうすればいいですか。

2番目の方法は簡単です。 入力する：

$ sudo pbuilder update --override-config --basetgz ~/pbuilder/hardy-i386-base.tgz --othermirror "deb myhost.ru/ubuntu hardy main"
