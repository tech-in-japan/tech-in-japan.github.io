IOmeterを使用したテスト





IOmeterプログラムは、ディスクサブシステムとローカルネットワークのパフォーマンスをテストするための一般的なツールです。 テストは「100％合成」です。



残念ながら、テストプロセスのいくつかの非自明性、時代遅れの外観、本格的なオンラインヘルプの欠如、ドキュメント、およびロシア語の説明は、使用する際に困難を引き起こすことがよくあります。 また、インターネット上では、実際に使用する方法論の詳細な説明、および使用されている用語と機能の説明はありません。



著者は、 aboutnetappブログの所有者に特別な感謝を表明したいと考えています。IOmeter を 使用したスト​​レージシステムの実際のテストに関する投稿で、著者はこの記事を執筆しました。



テストソフトウェアの世界では、現在2つの主要なテストモデルが受け入れられています。 1つ目は、実際の作業の模倣です。 通常、このようなテストには、スクリプトを使用して、テスト中に実際の作業を模倣するいくつかの実際のアプリケーションが含まれます。Word文書を開いて変更し、スプレッドシートを作成して保存します。 このようなテストアプリケーションの典型的な例は、Winbenchテストスイートです。



このような「実際の」テストには否定できない利点があるため、たとえば、事務などの一般的なアクションを超えるテストを作成することは困難またはほとんど不可能です。

第二に、測定技術はかなり暗く、シミュレーションテストで使用されるアプリケーション内で混乱しているため、結果を解釈することがしばしば困難です。



このため、サーバーシステム、特にディスクサブシステムの負荷テストでは、「合成」テストを使用することがよくあります。これは、実際に近い負荷の種類をシミュレートできるかなり明確なアルゴリズムです。 アルゴリズムの明快さが明らかなため、このようなテストは比較テストやパフォーマンスの問題の分析に非常に便利です。



人気のあるIOmeterテストは、2番目のカテゴリである「合成」テストに属します。



このプログラムはもともとIntelで開発され、その後Intelでの開発が停止すると、オープンソースに移行され、コミュニティは徐々に開発を続けました。



このプログラムには独自のWebサイトがありますが、Sourceforgeのプロジェクトページからダウンロードすることをお勧めします。

実際、ダウンロードリンクは、かなり前に凍結した最後の安定版（2006.07.27）を提供しますが、Sourceforgeの最後の安定版はDevel 2008.06.22 rc2であり、かなりの重要な変更と改善を行っているため、使用することをお勧めします。



このバージョンのSourceforge用に次のバイナリがコンパイルされます：X86_64、IA64（ネイティブItanium）、Win32、およびLinux x86_64。 並べ替えから、目的のプラットフォーム用にDynamoを構築（試行）できるため、少し矯正してシャーマニズムを行った後、彼は必要なSolaris 10 / SPARCのために集まりました。



興味深いことは、最初から始まります。 インストール後、プログラムフォルダーにiometer.exeとDynamo.exeの 2つの実行可能ファイルがあります。 実際、IOmeterは「2コンポーネント」スキームに従って構築されているため、テスト対象のシステムを制御システムから分離するだけでなく、多数のテストコンピューターを一度に管理することもできます。 Iometer.exeはGUIを備えた管理プログラムであり、 Dynamo.exeは対応するプラットフォームのコマンドラインユーティリティです。



この分離は非常に便利です。たとえば、テスト対象のシステムが隣の建物のデータセンターにある場合、率直に言って、そこにいるのはあまり快適ではありません（うるさく、寒く、吹いていて、座るのは簡単なことではありません）。 このような2コンポーネントテストシステムを使用して、データセンターのテスト済みサーバーでLoad Generator（Dynamo.exe）を実行し、オフィスに行き、そこからローカルネットワーク経由で接続し、すべてをリモートで管理して、ラップトップで結果を受信できます。



ただし、「分散」テストを実行するタスクがない場合は、これらすべてに煩わされることはなく、iometer.exeを実行するだけです。 同時に、GUIマネージャー画面が表示され、Dynamoがローカルで起動します。



DynamoとIOmeterは両方とも/？で起動できます。 同時に、予想どおり、いくつかの便利な起動キーのリストが表示されます。



IOmeter：

Iometer config_file [result_file [timeout_value]]

Iometer [/ c config_file] [/ r result_file] [/ t timeout_vaue] [/ p port] [/ m 1]



Config_file-構成ファイル。 存在し、有効なicfでなければなりません。

Result_file-結果ファイル（CSV形式）は補足され、存在しない場合は作成されます。

Timeout_value-参加者がログインを待機している秒数。

Port_number-ログインを待機しているポート。

/ m-テスト時に大きなゲージを表示します。



config_fileとresults_fileの両方が設定されている場合、ロードされたconfig_fileの [Test Setup]タブで開始が定義されている場合、Iometerは人間の介入なしでバッチモードで開始を試み、テストを開始および停止し、 result_fileに結果を書き込み、終了したらiometerを閉じます。



テストには合計で何時間もかかるため、バッチ起動機能は非常に便利で便利です。



ダイナモ：

dynamo [/ i iometer_computer_name / m manager_computer_name] [/ n manager_name] [/ c cpu_affinity] [/ p login_port_number]



iometer_computer_nameは、DynamoからログインするIometerを実行しているコンピューターの名前またはIPアドレスです。 このパラメーターがなければ、Dynamoはローカルホストでそれを探します。

manager_nameは、Dynamoがインストールされているコンピューターの名前です。IOmeterの観点からは、「manager」と呼ばれます。 デフォルトでは、これはホスト名です。 IOmeter構成で使用する場合、たとえば、事前に作成された構成で特定のタスクを割り当てる場合は重要です。

manager_computer_nameは、GUIがこのDynamoと通信するDynamoコンピューターの名前またはIPです。 デフォルトでは、最初のネットワークカードのIPアドレスが使用されます。

login_port_numberは、DynamoとGUIの間でログインが発生するポートです。 デフォルトは1066です。ファイアウォールの場合は、忘れずに開いてください。

cpu_affinityは、マルチプロセッサシステムの場合にDynamoが実行される特定のCPUの番号です。 定義されていない場合は、最初のCPUで。



したがって、実行する行は次のとおりです。

track @ unixbox> ./dynamo / i 192.168.1.100 / m 192.168.1.10 / n LG1



この場合、IP 192.168.1.10のマシンでDynamoインスタンスを実行します。これは、192.168.1.100にあるGUIに表示され、その中でLG1と呼ばれます。



/？で指定されていない別のキー これは/ force_rawスイッチです。 このキーは、ディスク上で見つかったファイルシステムを無視し、rawデバイスとしてテストすることを意味します。 ファイルシステムでフォーマットされたディスクはテスト中に破損するため、使用には注意してください。
