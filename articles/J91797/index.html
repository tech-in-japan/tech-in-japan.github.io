URL書き換え拡張機能：一般的なSEOの問題を修正





検索エンジン最適化（SEO）は、サイトにとって非常に重要です。 サイトへのほとんどのトラフィックは検索エンジンから直接送られます。サイトの検索関連性を改善すると、サイトへのユーザーコンバージョンが増加します。 これにより、サイトの収益を明示的または暗黙的に増やすことができます。



この記事では、 URL Rewrite Extensionを使用して、サイトで発生する可能性のある一般的なSEOの問題を修正する方法を説明します。 15分しかかからず、ウェブサイトに4つのシンプルなURL書き換えルールを適用し、検索エンジンからの訪問者とトラフィックを増やすために、コードを変更する必要はありません。 以下にリストする手法は、ASP.NET WebフォームとASP.NET MVCサイトの両方で（ASP.NETサイトでさえも）同じように機能します



Microsoft SEO Toolkitを使用して、サイトのSEOレベルを測定します



数ヶ月前、私は無料のSEOツールキットについて話しました。 この便利なツールを使用すると、SEOルールに準拠しているかどうかサイトを自動的にスキャンし、問題が見つかった場合に通知できます。 使用するサイトにこのツールをダウンロードして使用することを強くお勧めします。



以下は、URLの書き換えルールを適用する前に自分のサイト（ www.scottgu.com ）で実行したレポートの簡単な例です。これについては後で説明します。







検索の関連性とURLの分離



検索エンジンが検索関連性のためにサイトでランク付けする2つの重要なこと：

  1. コンテンツにリンクしている他のサイトの数。 検索エンジンは、インターネット上の多くの人々があなたのサイトにリンクしている場合、それは成功しているとみなされ、したがって、より高い関連性を持つべきです。
  2. サイト上のコンテンツの一意性。 検索エンジンが、コンテンツが異なるサイトまたはサイトの異なる場所で複製されていることを確認した場合、関連性を減らす必要があります。


サイトの構築中には、1つの重要なことに非常に注意する必要があります。サイトの同じコンテンツに異なるURLを使用することは禁止されています。



特に、外部サイトが異なるリンクを使用して同じコンテンツにリンクできるようにすることで、リンク数とページ評価が分離され、URLが1つしかない場合はページ評価が低くなります。欠席します。 理論的には外部サイトがさまざまな方法であなたにリンクすることを禁止するのは良いことですが、実際にはどういう意味ですか？



サイトで発生する可能性のある4つの本当の一般的なSEO問題



以下に、不注意で発生し、同じコンテンツへの異なるリンクにつながる可能性のある最も一般的な問題の4つの例を示します。



問題＃1：デフォルトのドキュメント


IIS（および他のWebサーバー）は、「既定のドキュメント」の概念をサポートしています。 これにより、Webサイトのルートまたはそのサブディレクトリに表示するページを明示的に指定する必要がなくなります。 同意します-便利ですが、これはデフォルトでコンテンツが2つの異なるパブリックアドレスで利用できることを意味しますが、これはすでに悪いことです。 例：

http://scottgu.com/

http://scottgu.com/default.aspx



問題＃2：URLの文字ケース


Web開発者は、検索エンジンのURLで大文字と小文字が区別されるとは思わないことがよくあります。 これは、2つの完全に異なるURLと同様に、指定されたリンクで検索エンジンが機能することを意味します。

http://scottgu.com/Albums.aspx

http://scottgu.com/albums.aspx



問題番号3：そこにスラッシュ、ここにスラッシュ


以下のリンクを見ると、それらは同じように見えますが、そこにはありませんでした-それらは完全に異なっています。 最後のスラッシュは、検索エンジンがこれらのリンクを異なるものと見なす別の状況を作成します。

http://scottgu.com

http://scottgu.com/



問題4：ベースアドレス


サイトでは、接頭辞「www」の有無にかかわらず、2つのオプションがサポートされる場合があります。 この場合、URLは完全に異なると見なされます。

http://scottgu.com/albums.aspx/

http://www.scottgu.com/albums.aspx/



IIS Rewriteを使用してこれらの問題を10分で簡単に修正する方法



サイトの作成中に不注意だった場合、リストされたエラーのいずれかを行うことが保証されます。 これらの問題を解決すると、検索エンジンのランキングが向上し、サイトへのトラフィックが増加します。



幸いなことに、これら4つの問題はURL Rewrite Extensionを使用して修正されます 。 これは、IIS 7.x用のMicrosoftの完全無料の拡張機能です（Windows Server 2008 / R2、Windows Vista / 7）。 そして今、ニュースはさらに良くなっています-アプリケーションのコードを変更する必要はありません。



Microsoft Web Platform Installer （Webサーバーと開発マシンを自動的に構成する無料のツール）を使用して、3分でURL Rewrite Extensionを簡単に設定できます 。 URL Rewrite Spotlightページの緑色の[今すぐインストール]ボタンをクリックして、マシンにインストールします。







アドオンをインストールすると、そのアイコンがIIS 7管理ツールにすぐに表示されます。







アイコンをダブルクリックすると、URL Rewrite管理パネルが開き、アプリケーションとサイトに定義されているURL Rewriteルールのリストが表示されます。







リストにはすぐにルールが含まれないことに注意してください。 パネルの右上隅にある「ルールの追加」をクリックして、サイトの新しい書き換えURLロジックを追加できます。



例1：デフォルトのドキュメントバリアントの処理



この記事の前半で説明したSEOの問題の1つは、「デフォルトドキュメント」です。

scottgu.com

http://scottgu.com/default.aspx



2番目のアドレスに連絡した人を最初のアドレスに自動的にリダイレクトする新しいIIS Rewriteルールを追加することで修正できます。 HTTPリダイレクトを永続的に設定します。これにより、検索エンジンは新しいアドレスを追跡し、受信したコンテンツの識別子としてこのアドレスを使用する必要があります。



このルールを作成する方法を見てみましょう。 以下に示すように、「ルールの追加」をクリックして始めましょう。







「Inbound rules」グループの「Blank Rule」テンプレートを選択して、新しいカスタムURL書き換えルールを作成します。 そして、空のパネルが表示されます。







心配しないで、ルールの設定はとても簡単です。 次の4つのステップでこれを行う方法を説明します。



ステップ＃1：ルールの名前


最初のステップは、ルールに名前を付けることです。 将来的にリストを簡単にナビゲートできるように、適切な名前を設定する必要があります。 このルールを「デフォルトのドキュメントURL書き換え」と呼びましょう。







ステップ＃2：このルールに一致する正規表現を設定する


2番目のステップは、着信URLがパターンに一致したときに実行される正規表現を使用してフィルターをセットアップすることです。 あなたが正規表現に強くなくても心配しないでください-私もそれらと友達ではありません。 あなたは彼らと友達である人を見つけるか、他のサイトからコピーする必要があります。



以下に、目的の正規表現を設定します。

（。*？）/？デフォルト\ .aspx $



このパターンは、Default.aspxで終わるURLで機能します。 「（。*？）」は、直前の文字と0回以上一致します。 「/？」は、スラッシュが存在する可能性がある、またはその逆であると言います。 末尾の「$」文字を使用すると、Default.aspxで終わる必要がある行にルールが適用されることを確認できます。



正規表現のすべての要素を組み合わせることにより、Webサイトのルート（ http://scottgu.com/default.aspx ）だけでなく、サイトの他のサブディレクトリ（ http://scottgu.com/photos/default.aspx ）の作業ルールも取得します） 「大文字と小文字を区別しない」チェックボックスがオンになっているため、アドレスオプション「Default.aspx」と「default.aspx」の両方が機能します。







素晴らしい組み込み関数の1つは、パターンをテストする機能です-「テストパターン」：







上記で、アドレス「products / default.aspx」を追加し、「テスト」をクリックしました。 このURLに関してルールを実行した結果がすぐに得られました。



ステップ＃3：Permanent Forwardingをインストールする


次に、着信URLの正規表現に一致するアクションを設定します。







上記では、「アクションタイプ」ドロップダウンリストを「リダイレクト」アクションに設定します。 「リダイレクトタイプ」はHTTP 301パーマネントリダイレクトであり、検索エンジンがリダイレクトに従うことを保証します。



「リダイレクトURL」プロパティも設定します。

{R：1} /



これは、Webクライアントリクエストを新しいURLにリダイレクトし、元のURLから「Default.aspx」の最後の部分を破棄することを意味します。 たとえば、リクエストhttp://scottgu.com/default.aspxの場合、リダイレクトはhttp://scottgu.com/になり、 http：//scottgu.com/photos/default.aspxの場合はhttp://scottgu.comになります。 /写真/ 。



正規表現の構成要素「{R：N}」。N> = 0はバックリンクと呼ばれ、Nはバックリンクインデックスです。 パターン「（。*？）/？Default \ .aspx $」の場合、着信URLが「products / Default.aspx」の場合、{R：0}には「products / Default.aspx」と{ R：1}-「製品」。 ユーザーをリダイレクトする値{R：1}を使用します。



ステップ＃4：ルールを保存して適用する


最後の手順は、IIS管理パネルの右上隅にある[適用]ボタンをクリックすることです。これにより、アプリケーションのすべてのルートディレクトリ（<system.webServer / rewrite>セクション）のURL書き換えルールがweb.configに保存されます。

< configuration >

< system.webServer >

< rewrite >

< rules >

< rule name ="Default Document" stopProcessing ="true" >

< match url ="(.*?)/?Default\.aspx$" />

< action type ="Redirect" url ="{R:1}/" />

</ rule >

</ rules >

</ rewrite >

</ system.webServer >

</ configuration >





IIS 7.xとASP.NETは同じweb.configファイルで機能するため、Visual Studioを使用して上記のコードを実際にweb.configファイルにコピーして貼り付けることができ、管理パネルですべてを構成する必要がなくなります。 そして、ASP.NETアプリケーションを使用してURL書き換えルールを追加および拡張するのがさらに簡単になりました。



ステップ5：ルールのテスト


ルールを保存したので、テストしてみましょう。 私のサイトで次のリンクを試してください。

http://scottgu.com/

http://scottgu.com/default.aspx



2番目のリンクが最初のリンクに自動的にリダイレクトすることに注意してください。



例2：URLの文字のケース



上記で説明した別の一般的な問題は、検索エンジンの大文字と小文字の区別です。 つまり、検索エンジンは両方のリンクのランキングを個別に計算します。

http://scottgu.com/Albums.aspx

http://scottgu.com/albums.aspx



これを修正するには、最初のリンクに移動した人を自動的に2番目のリンクにリダイレクトする新しいIIS書き換えルールを追加します。 前と同様に、HTTPリダイレクトを永続的なものとして設定します。



ルールを作成するには、URL Rewrite管理パネルの[ルールの追加]ボタンをもう一度クリックします。







「ブランクルール」を作成した以前の状況とは異なり、組み込みルールテンプレート「小文字のURLを強制する」を使用します。 [OK]をクリックすると、次のダイアログが表示され、意図を確認するように求められます。







[はい]をクリックすると、着信URLに大文字が含まれる場合、完全に大文字のURLへの永続的なリダイレクトを自動的に実行する既製のルールが取得されます。







[適用]ボタンをクリックして、すべての受信URLに適用することにより、何も変更せずにルールを保存できます。



私のサイトwww.scottgu.comはASP.NET Web Fromsを使用しているため、ASP.NETに組み込まれた「WebResource.axd」ハンドラーを除外する条件を追加することにより、上記のルールを少し変更する必要があります。 WebResource.axdハンドラーのURLは、ページ上のサーバーコントロールからのみ取得され、外部サイトからは取得されません。



良いニュースは、URLを無効にする条件を追加することです。一部のURLのルールの書き換えは非常に簡単です。 以下の形式でグループ「条件」を展開するだけです。







次に、「追加」ボタンをクリックします。







上記の[条件]フィールドに{URL}を入力し、URLが文字列内の「WebResource.axd」の存在を確認する正規表現条件と一致しない場合にのみルールを実行することを示しました。 これにより、WebResource.axdへのすべてのリクエストが、小文字のURLを表示せずに直接送信されます。



名前に大文字が含まれているリンク.jpg、.css、および.jsファイルなどの静的リソースが既にサイトにある場合は、URLリダイレクトをキャンセルする条件を追加する必要がある可能性が高いことに注意してください小文字で。 これらのルールを追加しなくてもサイトは正常に機能しますが、内部リソースの場合、小文字で追加のリダイレクトを行う必要はないため、このようなルールを追加する方が適切です。



[OK]ボタンをクリックして小文字のルールを適用すると、管理パネルは新しいルールをweb.configファイルに保存します。

< configuration >

< system.webServer >

< rewrite >

< rules >



< rule name ="Default Document" stopProcessing ="true" >

< match url ="(.*?)/?Default\.aspx$" />

< action type ="Redirect" url ="{R:1}/" />

</ rule >



< rule name ="Lower Case URLs" stopProcessing ="true" >

< match url ="[AZ]" ignoreCase ="false" />

< conditions logicalGrouping ="MatchAll" trackAllCaptures ="false" >

< add input ="{URL}" pattern ="WebResource.axd" negate ="true" />

</ conditions >

< action type ="Redirect" url ="{ToLower:{URL}}" />

</ rule >



</ rules >

</ rewrite >

</ system.webServer >

</ configuration >





ルールをテストする


ルールを保存したので、サイトで試してみましょう。 以下のリンクをたどってみてください。

http://scottgu.com/Albums.aspx

http://scottgu.com/albums.aspx



大文字の「A」を持つ最初のURLは自動的に小文字のURLにドロップされることに注意してください。



例3：末尾のスラッシュ



私が言及した別の一般的なSEOの問題は、URLの最後のスラッシュです。 最後のスラッシュは、検索エンジンがURLを2つの異なるものと見なす別の状況を作成します。

http://scottgu.com

http://scottgu.com/



これを修正するには、最初のリンク（最後にスラッシュがない）をクリックした人を自動的に 2番目のリンクにリダイレクトする新しいIIS書き換えルールを追加します。 前と同様に、HTTPリダイレクトを永続的なものとして設定します。



ルールを作成するには、URL Rewrite管理パネルの[ルールの追加]ボタンをもう一度クリックします。







Rewrite URL管理ツールには、定義済みのルールテンプレート「末尾のスラッシュ記号を追加または削除する」が含まれています。



次に、表示されるダイアログで、アドレスにない場合に最後にスラッシュが付いたURLにユーザーを自動的にリダイレクトするルールを作成するかどうかを尋ねられます。







[OK]をクリックすると、準備ができたルールが表示されます。



前の小文字のルールと同様に、WebResource.axdルールからアドレスを除外する追加の条件を追加します。 その結果、web.configファイルで次のようになります。



< configuration >

< system.webServer >

< rewrite >

< rules >



< rule name ="Default Document" stopProcessing ="true" >

< match url ="(.*?)/?Default\.aspx$" />

< action type ="Redirect" url ="{R:1}/" />

</ rule >



< rule name ="Lower Case URLs" stopProcessing ="true" >

< match url ="[AZ]" ignoreCase ="false" />

< conditions logicalGrouping ="MatchAll" trackAllCaptures ="false" >

< add input ="{URL}" pattern ="WebResource.axd" negate ="true" />

</ conditions >

< action type ="Redirect" url ="{ToLower:{URL}}" />

</ rule >



< rule name ="Trailing Slash" stopProcessing ="true" >

< match url ="(.*[^/])$" />

< conditions logicalGrouping ="MatchAll" trackAllCaptures ="false" >

< add input ="{REQUEST_FILENAME}" matchType ="IsDirectory" negate ="true" />

< add input ="{REQUEST_FILENAME}" matchType ="IsFile" negate ="true" />

< add input ="{URL}" pattern ="WebResource.axd" negate ="true" />

</ conditions >

< action type ="Redirect" url ="{R:1}/" />

</ rule >



</ rules >

</ rewrite >

</ system.webServer >

</ configuration >

最初のURL（末尾にスラッシュがない）は、末尾のスラッシュのURLに自動的にジャンプすることに注意してください。 リダイレクトは永続的であるため、検索エンジンは新しいURLに従ってページランクを更新します。



例4：ベースアドレス



そして最後に、SEOの最後の問題である、サイトがURLの2つのバージョンで動作するときのオプション-「www」プレフィックスありとなし：

http://www.scottgu.com/albums.aspx

http://scottgu.com/albums.aspx



これを修正するには、IIS書き換えルールを追加します。このルールは、最初のリンク（「www」プレフィックス付き）をクリックしたすべてのユーザーを2番目のリンクに自動的にリダイレクトします。 繰り返しますが、転送は永続的です。



ルールを作成するには、URL Rewrite管理パネルの[ルールの追加]ボタンをもう一度クリックします。







Rewrite URL管理ツールには、定義済みの「正規ドメイン名」ルールテンプレートが含まれています。



次に、表示されるダイアログで、ユーザーをプライマリURLに自動的にリダイレクトするルールを作成するかどうかを尋ねられます。







上記で、ウェブに使用したい主なアドレスscottgu.comを入力しました。 [OK]をクリックすると、web.configファイルに完成したルールが表示されます。

< configuration >

< system.webServer >

< rewrite >

< rules >



< rule name ="Cannonical Hostname" >

< match url ="(.*)" />

< conditions logicalGrouping ="MatchAll" trackAllCaptures ="false" >

< add input ="{HTTP_HOST}" pattern ="^scottgu\.com$" negate ="true" />

</ conditions >

< action type ="Redirect" url ="http://scottgu.com/{R:1}" />

</ rule >



< rule name ="Default Document" stopProcessing ="true" >

< match url ="(.*?)/?Default\.aspx$" />

< action type ="Redirect" url ="{R:1}/" />

</ rule >



< rule name ="Lower Case URLs" stopProcessing ="true" >

< match url ="[AZ]" ignoreCase ="false" />

< conditions logicalGrouping ="MatchAll" trackAllCaptures ="false" >

< add input ="{URL}" pattern ="WebResource.axd" negate ="true" />

</ conditions >

< action type ="Redirect" url ="{ToLower:{URL}}" />

</ rule >



< rule name ="Trailing Slash" stopProcessing ="true" >

< match url ="(.*[^/])$" />

< conditions logicalGrouping ="MatchAll" trackAllCaptures ="false" >

< add input ="{REQUEST_FILENAME}" matchType ="IsDirectory" negate ="true" />

< add input ="{REQUEST_FILENAME}" matchType ="IsFile" negate ="true" />

< add input ="{URL}" pattern ="WebResource.axd" negate ="true" />

</ conditions >

< action type ="Redirect" url ="{R:1}/" />

</ rule >



</ rules >

</ rewrite >

</ system.webServer >

</ configuration >

SEOを改善する4つの簡単なルール



上記の4つのSEOルールを使用するのは非常に簡単で、既存のサイトのセットアップには最大15分かかります。



URL書き換え拡張機能などのソリューションを使用する利点は、アプリケーションのコードを変更したり、サイトへの既存のリンクを中断したりする必要がないことです。 古いリンクを使用してサイトにアクセスするユーザーは、新しいURLに自動的にリダイレクトされます。 そして、検索エンジンは格付けを上げ始め、あなたのサイトのポジションが上昇します。



書き換えURLルールを将来変更するのは、web.configファイルを編集するか、IIS 7.x管理パネルを使用することで非常に簡単です。







ルールをクリックすると、エディターが開き、ルールを変更して保存できます。
