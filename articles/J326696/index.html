PHPバージョンのパフォーマンス比較







この記事では、PHP 5から実験的なJITブランチ（現在開発中）までのいくつかのベンチマークの結果を検討します。 執筆時点では、PHP 7.2などのメジャーバージョンがPHP 8より前に表示されるかどうかは不明でした。 しかし、実験的なブランチの機能は少なくともPHP 8に含まれると想定するのは理にかなっています。







1994年の導入以来、PHP言語は根本的に変わりました。 最初のリリースは、単に外部CGIプログラムであり、Rasmus Lerdorfの個人プロジェクトとしてさまざまな方法で作成されました。 PHPの3番目のバージョンが大幅に再設計されて以来、言語開発者のグループです。







PHP 3の拡張性のおかげで、言語の機能は急速に成長しました。 ネットワークの操作、解析、キャッシング、データベースサポートなど、さまざまな分野で新しい機能を導入した基本的な拡張機能が追加されました。







言語自体も開発され、多くの改善が行われました。 クラス、インターフェイス、特性、クロージャなどのオブジェクト指向の構築がサポートされています。

しかし、多くの開発者にとって、これは十分ではありませんでした。 この言語の人気に伴い、主にパフォーマンス、スケーラビリティ、より経済的なメモリ消費に関連して、コミュニティの言語に対する要求も高まっています。







ほぼ20年間、言語の作成者はあらゆる種類の要件を満たすために多大な努力をしてきました。 PHP 3の登場により、生産性は大幅に向上しましたが、Zendエンジンが登場したとき、PHP 4でのみこの言語は重大な結果を示すことができました。







2000年に、新しいインメモリコンパイラとエグゼキュータモデルが導入されました。 これにより、PHPのパフォーマンスが大幅に向上し、多くの場合5〜10倍になりました。 その結果、彼らはWebアプリケーションやWebサイトを作成するためのツールとして真剣に考え始めました。 そして今日、PHPはこの言語が登場したときに誰も期待していなかった高さに達しました。







しかし、PHPの人気の爆発的な増加は、パフォーマンス要件の増加につながっています。 幸いなことに、Zendエンジンには近代化の大きな可能性があります。

PHP 5は大きな進歩を遂げることはなく、場合によってはPHP 4よりもさらに遅くなりましたが、Zend開発チームはリリースごとにエンジンを常に最適化しました。PHP5.6の結果、1.5〜3倍高速になりました。







しかし、2015年12月のPHP 7のリリースで大きなブレークスルーが発生しました。 1年後、バージョン7.1が発表され、多くの改良も加えられました。







PHP JITコンパイラーとPHP 8のパフォーマンス向上に対する期待



Zendの非常に有望なバージョンが開発されています。 リリース7.1のバージョンに基づいており、リリースされる時点ではまだ発表されていません。 したがって、これは実験的なJITブランチです。







主な陰謀の1つは、ジャストインタイム（JIT）コンパイルに関連しています。 これは、実行直前にコードを別の形式（ネイティブマシンコード）に変換する手法です。 JITの目標は、プログラムの速度を上げることです。 開発者が約束を守れるかどうか見てみましょう。







PHPスクリプトを処理するためのベンチマーク



この記事では、純粋にプロセッサのタスク、つまりファイルアクセス、ネットワークまたはデータベース接続などのI / O操作を行わないスクリプトの処理スクリプトのパフォーマンスを測定するベンチマークを使用しました。







次のベンチマークが使用されました。







  • bench.php-元のPHPディストリビューションのphp-src / Zendフォルダーにあります。
  • micro_bench.php-同じ場所。
  • mandelbrot.php


ベンチマークは、主要なPHPバージョンの最新のマイナーリリースで実行されました。

同じベンチマークが、5.3.0から5.3.29など、すべての中間リリースで実行されました。 結果は雄弁です。リリースでは顕著なパフォーマンスの改善は見られませんでした。 PHP 5.4からPHP 5.5への移行、またはPHP 5.6からPHP 7への移行など、メジャーバージョン間の移行時にのみ改善が認められました。







これは、PHP 5.4.0とPHP 5.4.45の両方で同じスクリプトがほぼ同じ速度で実行されることを意味します。

プロセッサベンチマークの結果の比較



各ベンチマークに対して、3つの値が与えられます。







  • 時間、s ：実行時間（秒単位）。
  • 相対的な変化、％ ：以前のバージョンと比較した実行時の変化。 ベンチマークがより速く実行された場合、値は正であり、より遅い場合、負です。
  • 絶対的な変更、簡単 ：PHP 5.0と比較して、スクリプトの実行速度はどれだけ速いか。


ベンチマークの実行結果は、次の表で確認できます。













（1）ベンチマークは、まだ実装されていないプロパティを使用するため、5.3より前のバージョンでは実行できません。







（2）ベンチマークを機能させるには少なくともPHP 5.3が必要なため、この列の結果はわずかに偏っています。 PHP 5.0と比較することはできないため、参照用としてのみ使用できます。







（3）これは、mandelbrot.phpスクリプトの修正版であり、実験ブランチのバージョン7.1.0では速すぎたため、速度を正確に測定することはできませんでした。 そのため、スクリプト内で1つではなく100の計算を実行しました。







もちろん、純粋にプロセッサのベンチマークでは、PHPのパフォーマンスのすべての側面を評価することはできません。 データは代表的ではない場合があります。 それでも、これらのデータに基づいて、次のことを結論付けることができます。







  • PHP 5.1は、PHP 5.0の2倍以上の速度です。
  • バージョン5.2および5.3には新しい改良点がありますが、5.1ほど印象的ではありません。
  • 次のパフォーマンスの飛躍はバージョン5.4でした。
  • opcache拡張機能には5.5および5.6が付属していました。 これにより、同じスクリプトが連続して実行されたときにコードのロードが高速化され、生産性が向上しました。 ただし、opcacheはCLIモードで実行されるスクリプトにはあまり役立ちません。
  • PHP 7.0は、パフォーマンスの面で大きなブレークスルーです。 Zendエンジンは完全に再設計され、これらの大規模な変更の結果を確認しています。
  • PHP 7.1では、opco拡張はopcache拡張で最適化されていたため、7.0を超えるパフォーマンスの向上が説明されています。
  • 実験的なJITブランチは、パフォーマンスの次のジャンプを示しています。 しかし、場合によっては改善されません。 コンパイルによってコードが高速化されないため、分岐がさらに遅くなる場合があります。 ただし、この機能はまだ開発中であることを忘れないでください。

ベンチマークの開始



各ベンチマークは、PHP CLI（コマンドラインインターフェイス）を使用して、以下を実行する特別なスクリプトを使用して起動されました。







1） microtime()



関数を使用して、 microtime()



スクリプトを変更し、内部から実行時間を測定しました。 変更後、スクリプトは次のようになりました。 







 <?php $__start__ = microtime( true ); /***     ***/ fprintf( STDERR, microtime( true ) - $__start__); ?>


これは、スクリプトの動作を変更せずに、内部からスクリプトの次元の安定性を確保するために行われました。







2）次に、実行可能なPHPファイルとベンチマークスクリプトの内容がOSキャッシュにあるように、2回のドライランが行われました。







3）スクリプトが5回実行され、最小、最大、および平均の実行時間が保存されました。 この記事では、平均値-「スクリプト実行時間」のみを紹介します。







php.iniでは次の設定が使用されました。

結果の解釈



実行時間は、Unixのtime



コマンドを使用して測定されました。 サンプル出力： 







 $ time php bench.php real: 0m1.96s user: 0m1.912s sys: 0m0.044s


real



の値は、コマンドからそれを中断するまでの時間です（コマンドラインに戻るまで）。







値user



は、ユーザーコード（この場合、実行可能なPHPファイル）の実行に費やされた時間です。







sys



値は、OSコード（カーネル）の実行に費やされた時間です。 この値は最小限に抑える必要がありますが、コードが遅いデバイスなどにアクセスする場合に表示される値よりもはるかに大きくなることがあります。 また、OSの高負荷は値に影響を与える可能性があります。







スタンバイシステムでは、 user + sys



合計値はreal



に非常に近いはずです。 上記の例では、 user + sys = 1,956 , real = 1,960



です。 0.004 sの差は、プロセスではなく、OSのタスク（ディスパッチなど）に関係しています。







同じスクリプトは、3つの異なるPHPバージョンで並行してコンパイルしながら、負荷の高いOSで実行されました。 







 $ time php bench.php real: 0m7.812s user: 0m2.02s sys: 0m0.101s


ご覧のとおり、負荷レベルはランタイム（および場合によってはシステム時間）に強く影響します。 したがって、ベンチマークに別の値を追加しました-オペレーティングシステムのoverhead



。 これは、 elapsed time



とユーザー時間とシステム時間の合計との差です。







ベンチマーク実行中に、スクリプトが数十秒かかった場合でも、99％の時間でこの値が100ミリ秒未満であることを確認しました。
