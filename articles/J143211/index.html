Lightboxウィジェットは、lightbox.jsの最初のバージョンが2005年にリリースされて以来、Web上で標準になりました。 Lightboxはモーダルダイアログボックスを作成し、通常は[次へ]ボタンと[前へ]ボタンを使用してスライド間を移動する大きな画像を表示します。



タッチデバイスの使用のブーム以来、Webサイトはライトボックスを更新して、さまざまな成功度でのジェスチャーの相互作用をサポートしています。 このチュートリアルでは、ジェスチャをサポートするシンプルなライトボックスを作成する方法を示します。 その過程で、実際のパフォーマンスを改善するためのいくつかの簡単なトリックと同様に、タッチインターフェイスの知覚されるパフォーマンスの改善について少し学びます。



タッチデバイス用のコードの記述は、デスクトップ用のコードの記述とは大きく異なります。 できるだけ多くのコードをデスクトップと組み合わせることができます（また、そうすべきです）が、それらの間には常に大きな違いがあります。



比較テストでは、最も一般的なタッチデバイスのパフォーマンスが1998年頃のデスクトップコンピューターに匹敵することが示されています。 通常、元のiMacと同等の約256 MBのRAM、CPUパフォーマンスを備えています。 デスクトップで「ただ動作する」ために使用する方法は、携帯電話やタブレットでは正しく動作しません。 幸いなことに、これらのデバイスは通常、グラフィックス、特に画面上のアイテムの移動用に最適化されています。 IOSおよびAndroid 3.0+デバイスには、ハードウェアアクセラレーショングラフィックスがあります。

実際、これらのデバイスは、まともなグラフィックスカードを備えた安っぽいコンピューターであると想像できます。



私たちは過去20年間、同じ方法でコンピューターと対話してきました。 マウスポインターを移動し、コントロールボタンをクリックします。 ボタン、閉じるブロック、リンク、およびスクロールバーは、ユーザーおよび開発者にとって第2の性質です。 タッチインターフェイスは、まったく異なる一連の規則を表しています。 最も一般的なのは「スワイプ」です。 「スライド」すると、いくつかの要素が一列に並んでいるかのように表示され、ユーザーは「スライド」ジェスチャを使用して要素間を移動できます。



スライドは非常に一般的なパターンであるため、ユーザーにそれを知らせる必要さえありません。ユーザーがリストに似たものを見ると、本能的にスクロールしようとします。



多くの場合、特に低速の接続と低速のデバイスを扱う場合、コードを少し速く実行することはできません。 しかし、知覚の最適化に焦点を合わせると、インターフェイスをより高速に見せることができます。



知覚されるパフォーマンスを最適化する私のお気に入りの例はTiVoです。 13年前、最初のTiVoが到着したとき、それらは非常に低速でした（16 MB RAMと54 MHz CPU！）。 特に何かを再生したり録音したりする場合は、リモコンで何かを押してから何かが起こるまでに、非常に長い時間を要していました。 しかし、誰もTiVoが遅いと文句を言いませんでした。 私の意見では、これは音によるものです。 TiVoインターフェースの最も馴染みのある部分は、ボタンを押した後に鳴るメロディーです。 その音はすぐに再生されました。 TiVoのエンジニアは、次に何が起ころうとも、インターフェースが死んでいないことをユーザーが知っているように、できるだけ早くサウンドをロードしました。 この短い曲は、ユーザーの要求が聞かれたことをユーザーに伝えました。

ネット上で、私たちは同じことを行う契約を作成しました：スピナー。 クリック後、スピナーがすぐに表示されるため、ユーザーは自分が聞いたというメッセージを受け取ります。 携帯電話では、別の方法で行う必要があります。



ジェスチャーは、クリックなどの個別のアクションではありません。 ただし、インターフェイスを高速に見せるために、ユーザーにフィードバックを提供する必要があります。 彼らが身振りで示すように、何らかの方法でインターフェイスを動かし、彼らが私たちに「聞いている」ことを知らせます。



ツール



レスポンシブインターフェイスでは、要素ができるだけ速く移動する必要があります。 移動することにより、インターフェイスがリクエストに応答していることをユーザーに示します。 これにはJavaScriptアニメーションの使用が遅すぎます。 代わりに、CSS変換とトランジションを使用します。パフォーマンスのためのトランスフォームとトランジションを使用して、JavaScriptの実行をブロックせずにアニメーションが機能するようにします。

このチュートリアルでは、すべての動きとアニメーションに対して、CSS変換と遷移を使用します。

可能な限り使用したい別の最適化は、「書き込み専用DOM」と呼ばれるものです。 DOMからのプロパティと値の読み取りは複雑で、通常は不要です。 ライトボックスの場合、初期化フェーズですべての測定値を組み合わせてみます。 その後、JavaScriptで状態を維持し、必要に応じて単純な算術演算を行います。



ライトボックスの作成



このチュートリアルでは、いくつかのサムネイルを含むページを作成します。 サムネイルをクリック（または指でタッチ）すると、ライトボックスが起動します。 その後、ライトボックスで、ユーザーは指で画像をスクロールし、画像にタッチしてライトボックスを終了できます。



ジェスチャのインターフェイスを作成するときは、知覚されるパフォーマンスの重要性に注意してください。 ライトボックスでは、これは、画面上での指のスライドによりスライドが移動することを意味します。 ユーザーがジェスチャーを停止すると、スライドは次の位置に移動するか、スライドの長さが十分でない場合は前の位置に戻る必要があります。



リターンアニメーションは重要です。 彼女のおかげで、ユーザーはインターフェースが死んでいるとは思わないでしょう。

ジェスチャーサポートの追加



ほとんどのタッチデバイスには、写真ビューアが組み込まれています。 これらのさまざまなアプリケーションは、iPhone用のオリジナルのフォトビューアーに続いて、インターフェイスに同意しました。指を左にスライドすると、次のスライドが表示されます。 フィードバックをまったく提供しないこの対話の実装をいくつか見ました。ジェスチャーが完了すると、スライドは単純に置き換えられます。 適切なアプローチは、ライブフィードバックを提供することです。 ユーザーが指をスライドさせると、スライドも一緒に移動し、方向に応じて、次または前のスライドが表示されます。 これにより、ユーザーが写真のストリップを引っ張っているような錯覚が生じます。







タッチイベント処理


Zeptoを含む多くのライブラリには、タッチイベントのサポートが含まれています。 一般に、それらの使用はお勧めしません。 タッチイベントを処理する場合、ユーザージェスチャとともに要素を更新します。 遅延がユーザーに気付くと、インターフェイスが遅く感じられます。 イベントにライブラリを使用した主な理由の1つは、ブラウザーでの正規化を許可することです。 タッチイベントをサポートするすべてのモバイルブラウザーには同じAPIがあります。



この例では、touchstart、touchmove、touchendの3つの感覚イベントがあります。 また、何らかの理由（プッシュメッセージなど）でジェスチャが中断されると、touchcancelイベントが発生します。 開発中に、それらを正しく処理する必要があります。 



 function attachTouchEvents() { var bd = document.querySelector('html'); bd.addEventListener('touchmove', handleTouchEvents); bd.addEventListener('touchstart', handleTouchEvents); bd.addEventListener('touchend', handleTouchEvents); }




イベントハンドラーはTouchEventオブジェクトを受け取ります。 touchstartイベントとtouchmoveイベントには、touch配列のオブジェクトであるtouchesプロパティが含まれています。 指をスライドするには、「clientX」というプロパティが1つだけ必要です。 これには、ページの左上隅に対するタッチ位置の値が含まれています。



iOSデバイスは、最大11の同時タッチをサポートします。 Android（Ice Cream Sandwichの前）は1つしかサポートしません。 ほとんどのインタラクションには、ワンタッチが必要です。 より複雑なジェスチャーは、複数のタッチを心配させます。
