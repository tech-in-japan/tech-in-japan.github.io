私たちの今日の目標は、vSphereクラスターのニーズに対応するフォールトトレラントなオールフラッシュ（つまり、SSDからのみで、これは重要ではありませんが）であり、ブランドの同等品よりも数倍安く、非常に優れたパフォーマンスを発揮します。 Fibre Channelを介して接続しますが、iSCSI、FCoE、さらにはホラー、Infinibandを実行する必要はありません。



シンクロ



名前が示すように、この家全体の基盤は、 LSI （現在のAvago）のSyncro CSと呼ばれるかなりユニークな製品です。



それは何であり、何が注目に値しますか？



実際、これは2つの従来のLSI 9286-8eコントローラー（または内部ポートが必要な場合は9271-8i）と、停電時にコントローラーのフラッシュドライブにキャッシュメモリを保存するための2つのスーパーキャパシタのセットです。 キットのコストは、HA機能のない同様のキットの価格よりも数倍高くなっています。 ただし、DRBDベースのソリューションと比較すると、この違いは、デュアルドライブセットを用意する必要がないことで相殺されます。



しかし、最も興味深いのはファームウェアにあります。 そのおかげで、これらのコントローラーは同じSASネットワーク（エキスパンダー付きのディスクバスケットなど）に接続され、相互に通信を確立し、フェールオーバークラスターモードで動作します。



私たちにとって、これはここで興味深いです：

  • 2台のサーバーで同時に利用可能なRAIDアレイを作成する機能

  • コントローラーレベルでのフォールトトレランス：そのうちの1つ（またはサーバー全体）が死亡した場合、2つ目は引き続き動作し、I / Oを維持します。


SASネットワークを介して、これらのコントローラーは書き込みキャッシュ（いわゆるキャッシュコヒーレンシー）を同期し、ハートビートを交換し、互いに正常であると伝えます。



これらには、経験と血の両方、そしてドキュメントの思慮深い読みによって明らかにされたいくつかの重要な特徴があります。

  • ソリューションの耐障害性により、LSI SyncroはデュアルポートSASドライブのみをサポートします 。 残念ながら、SATA SSDはありません。これは、ストレージのコストを大幅に増加させます。



  • 私が理解しているように、コントローラーは96個までのSASドライブをサポートします。 このドキュメントには、HAストレージドメイン内の最大120のデュアルポートSASデバイスというフレーズも含まれていますが 、これは同じ場所で最大96個の6G SASおよび/またはNearLine-SASデュアルポートHDDおよびSSDと 2行上に矛盾しています。 そのため、最小限にします。



  • コントローラーの1つにアレイを作成すると、このコントローラーは、たとえば、インストールされているサーバーが再起動されるまでそのマスターになります。 その後、所有者の役割は2番目のコントローラーに渡されます。



  • 最初のコントローラーがサービスに戻った後、アレイの所有者の役割は彼に戻りませんが、2番目のコントローラーに残ります。



  • コントローラーはI / Oシッピングメカニズムを使用します。I/ O要求が現在アレイのマスターではないコントローラーに到着すると、SASバスを介してホストコントローラーにこの要求を転送し、実行して送信します。



  • 上記の結果、アレイを備えたスレーブコントローラーの速度は、ホストコントローラーの速度よりも数倍低くなります（線形速度-2倍、IOPS-7（！）時間）。



  • 現時点では、ホストコントローラーに強制的にアレイの所有権を2番目のコントローラーに強制的に移行させる方法はありません。これは非常に奇妙です。 特に、メインコントローラーのソフトリセット（制御されたフェールオーバー）中に、2番目のコントローラーのログにあるメッセージは、メインコントローラーが再起動について警告し、アレイをピックアップするように要求することを示します。 LSIはユーティリティに何かを追加することを約束しましたが、今のところは...リブートのみで、ハードコアのみです。



  • スレーブ（特定の瞬間）コントローラーが挿入されているサーバーOSはホストコントローラーアレイを認識しますが、アレイ（およびこれらのアレイを構成するディスク）のコントローラー管理ユーティリティ（StorCLI、MegaCLI、...）は表示されません。 アレイの所有権がこのコントローラーに渡されるとすぐに、アレイとディスクがユーティリティの出力にすぐに表示され、それらを管理できるようになります


アルア



ホストとスレーブのコントローラーは異なる方法でアレイと動作するため、ALUAパラダイムを使用してストレージを構築する必要がありますが、これはそのような場合にのみ考案されました。 あらゆる種類の機能がたくさんありますが、私たちにとっての意味は、ストレージのポートの一部を最適化 （つまり使用が推奨）としてマークし 、その他を非 最適化 （それぞれ、非推奨）としてマークできることです。 これらのポートはすべて、いつでもI / Oを処理できますが、最適化されたポートへのライブパスがある場合、イニシエーターはそれらを使用して残りを予約します。 すべての最適化されたポートに障害が発生した場合、イニシエーター（この場合はESXi）は少し考えた後、最適化されていないポートでI / Oを開始します。



VMWare ESXiの観点からは、次のようになります。





最初の3つのパスはスレーブストレージに、そして2番目の3つのパスはメインのパスにつながります。メインのパスはI / Oをフルスピードで処理できます。



ALUAを使用していなかった場合、I / O要求（ラウンドロビンモード）は両方のリポジトリに送信され、遅延と速度が不均一になり、基本的にはサブリポジトリのパフォーマンスに依存します。



できた！

SANで動作する多くのプログラム（Microsoft Cluster Servicesなど）はSCSI Persistent Reservationsに依存しています。これにより、LUNをブロックし、LUNであらゆる種類のトリックを排他的に実行できます。 したがって、これらの同じSCSI PR は別のサーバーに複製されません 。



つまり、イニシエーターが最初のサーバーへのパスを介してLUNをロックし、次に2番目のサーバーへのパスを介してロックを設定した場合、このロックは表示されません。 ラウンドロビンモードでの操作の場合、これは特に重要です。パスが絶えず変化しており、ロックが設定されるパスが事前にわからないためです。 VMWare ESXiの場合、SCSI PRはVMFS5バージョンでは使用されず、代わりにAtomic-Test-and-Set命令が使用されるため、これは原則として問題ではありません（ただし、これも複製せず、必要なブロックのみを行います） （LUN全体ではなく、LUNの領域を短時間）、ALUAはESXiに、メインサーバーが稼働している間は2番目のサーバーへのパスを使用しないことを通知する必要があります。



SCSI PR同期を実装するSCSTベースの商用ソリューションがいくつかありますが、私はそれらを試していません。 そして、噂によると、何らかの解決策がおそらくすぐに公開されるでしょう。

ファームウェア（この場合はql2500_fw.bin）をダウンロードし、その場所に配置します。 

 # mkdir -p /lib/firmware # cd /lib/firmware # wget http://ldriver.qlogic.com/firmware/ql2500_fw.bin
