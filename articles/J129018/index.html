Oracle Databaseを使用したXMLの解析

データベースの側でXMLを解析する必要があるのはなぜですか？



しかし、これには多くの理由があり、それぞれに独自の理由があります。 私を含む一部の人は、データベースによる適用ロジックの実装によってまったく軽disされていませんが、一部の人にとっては古風な残骸のようであり、DBMSでXMLを操作するツールの有用性はそのような人々には疑わしいかもしれません。 ただし、アプリケーションの操作中にこのような機会を得ることに役立つことに異議を唱える人はほとんどいません。 たとえば、卸売業者の価格表アプリケーションは、私たちに受け入れられませんでした。彼は、一部のポジションの記録の欠如に混chaとして呪いをかけました。 XMLで2万以上のポジション-犬が急襲した場所を見つけてください。これは特にアプリケーションを混乱させました。 同意します。XMLにリストされた製品のリストを、変換テーブルに接続できるデータセットとして表示して、一度に変換されなかったすべてのポジションを明らかにできるのは素晴らしいことでしょうか。 そして、多くの同様の例を挙げることができます。 XMLメッセージの交換を通じて外部システムと統合するアプリケーションをサポートする機会がありましたが、アプリケーション自体はOracleが提供するツールを使用していなかったにもかかわらず、この製品のサポートで私や同僚にとって非常に役立つことがわかりました。



この記事では、Oracle Databaseツールを使用して、さまざまな複雑さのXMLを解析することがいかに簡単で簡単かを示したいと思います。



ここでDOMパーサーに触れたくありません。 それはDBMS_XMLDOMパッケージによって実装されているとしか言えません。 場合によっては、開発者にとって非常に役立つことがあり、以前に他のメーカーのDOMパーサーに遭遇したことがある人にとっては、それを扱うのは難しくありません。



Oracleの革新的な機能は、XMLType型とそれを操作する手段です。 このタイプは、バージョン9.2以降のOracle Databaseに含まれているXML DBテクノロジーの一部です。



XMLドキュメントのソーステキストは、CLOB、BLOB、VARCHAR2、BFILE型の値としてXMLTypeコンストラクターに渡すことができます。 おそらく、BFILEを使用すると、クライアントではなくサーバーのファイルシステムからファイルをダウンロードできることに注意する価値があります。XMLがクライアント側にあり、リクエストで文字列として送信するのに十分な大きさであれば、おそらくXMLでファイルを配信する可能性を考慮する必要があるためですサーバーファイルシステムへのコンテンツ。



文字列で渡されるコンテンツでXMLTypeインスタンスを作成する例： 

select XMLType( '<hello-world> <word seq="1">Hello</word> <word seq="2">world</word> </hello-world> ') XML from dual


XMLTypeのインスタンスを作成することにより、XMLを解析する最初のti病なステップを試みることができます。 XMLType型はExtractメソッドを実装し、XPatch式を取得すると、この式に一致するXMLフラグメントを返します。 XMLフラグメント（XMLフラグメント）は、適切に構築されたXML（helloformed XML）とは対照的に、ルート要素の不在を許可します（つまり、構成に複数のルート要素を許可します）。



したがって、以下の例では、3つの式が3つのXMLフラグメントを返します。 最初は単語要素のすべての出現を返し、2番目は単語要素の最初の出現のみを返し、3番目は単語要素のテキストコンテンツのフラグメントを返します。seq属性の値は2です。 

 SQL> with demo1 as ( 2 select XMLType( 3 '<hello-world> 4 <word seq="1">Hello</word> 5 <word seq="2">world</word> 6 </hello-world> 7 ') xml 8 from dual 9 ) 10 select t.xml.extract('//word') case1 11 ,t.xml.extract('//word[position()=1]') case2 12 ,t.xml.extract('//word[@seq=2]/text()') case3 13 from demo1 t; CASE1 CASE2 CASE3 --------------------------- -------------------------- ------- <word seq="1">Hello</word> <word seq="1">Hello</word> world <word seq="2">world</word>


ここで、3番目のケースでは、この要素の値ではなく、XMLフラグメントが返されるという事実に大胆に重点を置く価値があると思います。 この値に＆amp;、＆gt;などのエスケープ文字が含まれている場合にのみ、違いが顕著になります。 要素の値を取得するには、extractValue関数を使用します。 ここで、主なXMLTypeメソッドがSQL関数によって複製されること、またはその逆であることに言及します。XMLTypeを操作する主な機能は、そのメソッドの形式で実装されます。 ただし、extractValueは例外です。 extractValueは関数としてのみ表示されます。 残念ながら、XMLTypeはextractValueメソッドを実装していません。 

 SQL> with demo2 as (select xmltype('<a>&lt;&amp;hello&amp;&gt;</a>') xml from dual) 2 select t.xml.extract('a/text()').getStringVal() case1 3 ,extractValue(t.xml,'a') case2 4 from demo2 t; CASE1 CASE2 --------------------------- -------------------------- &lt;&amp;hello&amp;&gt; <&hello&>


おそらく、名前空間を操作するための規則についても言及する必要があります。 すべての直観がこれらの動作メカニズムの正しい理解につながるわけではありません。 関数（およびメソッド）の抽出であるextractValueは、パラメーターの1つとして、名前空間の説明を受け取ります。 このパラメーターで説明されている名前空間は、XPath式で使用できます。 そして、これはまさに私が強調したいことです。 3番目のケースに注意してください。 XMLおよびXPatch式の名前空間には、異なるエイリアスがありますが、解析は成功するため、同じURIがあります。 

 SQL> select extractValue(t.xml,'a') case1 2 ,extractValue(t.xml,'a','xmlns="foo"') case2 3 ,extractValue(t.xml,'y:a/@z:val','xmlns:y="foo" xmlns:z="bar"') case3 4 from (select XMLType('<a xmlns="foo" xmlns:x="bar" x:val="a-val">a-text</a>') XMl from dual) t; CASE1 CASE2 CASE3 --------------------------- -------------------------- -------------------------- a-text a-val


それで、値を抽出することを学んだので、今度はそれらを共有することを学ぶべきです。 最初の例では、最初の例では、XMLからすべての単語要素を選択しようとしましたが、2つの単語要素を取得しましたが、1つのフラグメントで取得しました。 複数のルート要素を含むフラグメントを、それぞれが1つのルート要素を含むフラグメントのシーケンスとして提示するために、パイプライン化されたXMLSeqence関数があります。 この関数は、XMLType値のテーブルであるXMLSequenceTypeを返します。



突然誰かが忘れてしまった場合、パイプライン化された関数はコレクションを返すので、呼び出されたときにテーブル式でラップされることを思い出してください。 これらの関数の結果は、仮想列column_valueまたは式の値（）を使用してアクセスされるため、テーブル式のエイリアスを定義する必要があります（テーブルコレクションの除外）。 突然誰かがこれを知らなかった場合、私はマントラとしてそれを暗記することをお勧めします、理解は時間とともに、そして必要な場合にのみ来るでしょう。



XMLSequenceの最も簡単な使用例： 

 SQL> select extractValue(value(t),'b') result 2 from table(XMLSequence(XMLType('<a><b>b1</b><b>b2</b></a>').extract('a/b'))) t; RESULT ------------------------------------------------------------------------------------- b1 b2


ここで何が起こっているのかを言おうと思いますが、ロシア語では、SQLの場合よりもはるかに乱雑になり、はっきりしないことがわかります。 fromステートメントでは、まずXMLTypeのインスタンスを作成し、XMLテキストを含む文字列を渡します。 次に、extractメソッドを使用して、要素aを含むすべてのb要素を1つのフラグメントに抽出します。 結果のXMLフラグメントは、パラメータとしてXMLSequenceパイプライン関数に渡されます。XMLSequenceパイプライン関数では、文法の規則に従って、table句が使用されます。 この文で記述されたレコードセットには、エイリアスtが割り当てられます。 select-list'eでは、テーブル式tによって返されるオブジェクトのインスタンスを取得します。これはXMLType型を持っています。 テーブル式によって返される各行に対して、このインスタンスにはソースXMLの要素bの1つのフラグメントが含まれます。 このオブジェクトをパラメーターとしてextractValue関数に渡します。 結果は顔にあります。



実際、すべてが私のプレゼンテーションで判明したほど複雑になることにはほど遠い。 少し慣れれば十分です しかし、脳の虐待はまだ完全には完了していません。 この段階で持っているものは、1つのXMLドキュメントに対してのみ機能します。 ラベルに複数のXMLのソーステキストがあり、それらのいくつかを一度に解析する必要がある場合、残っている相関関係を覚えておく必要があります。 ここにも軍隊はありません。 このことは、テーブル式（テーブルコレクション式）専用にOracleによって発明されました。 一番下の行は、テーブル式では、テーブル式自体の前（左）のfromステートメントで定義されたデータセットの値（列）を使用できることです。 実際には、これは耳ほどひどいものではありません。 

 SQL> with demo3 as(select 1 id, XMLType('<a><b>b1</b><b>b2</b></a>') xml from dual 2 union all select 2 id, XMLType('<a><b>b3</b><b>b4</b></a>') xml from dual) 3 select id xml_id 4 ,extractValue(value(t),'b') result 5 from demo3 s,table(XMLSequence(s.xml.extract('a/b'))) t; XML_ID RESULT ---------- -------------------------------------------------- 1 b1 1 b2 2 b3 2 b4


ここで、テーブル式tでは、demo3テーブルのxml値が使用されます。 式は、demo3テーブルの各行に対して計算されます。 これは、このような大げさな言葉-左相関と呼ばれるものです。



説明されている機能は、ほとんどすべての複雑さのXMLを解析するのに十分です。 おそらく、明らかに未知のネストの深さの階層的に提示されたデータのみが、これらの手段で分析することはできません。 このような構造を解析するには、XSLTに頼ってXMLをより読みやすくする必要があります。 XSLT変換は、ソースドキュメントのXMLType、最初のパラメーターとして2番目のXMLType XSLテンプレートを取得し、変換結果のXMLTypeを返すXMLTransform関数によって実行されます。



原則として、これは理論で行うことができます。 結論として、さまざまなレベルのXMLネストから要素を抽出する例を示します。 初心者にとって、これは時々困難を引き起こします。 

 SQL> with demo4 as( 2 select XMLType( 3 '<master> 4 <id>mater id</id> 5 <details> 6 <detail> 7 <id>detail 1 id</id> 8 <sub_details> 9 <sub_detail> 10 <id>sub_detail 1.1 id</id> 11 </sub_detail> 12 <sub_detail> 13 <id>sub_detail 1.2 id</id> 14 </sub_detail> 15 </sub_details> 16 </detail> 17 <detail> 18 <id>detail 2 id</id> 19 <sub_details> 20 <sub_detail> 21 <id>sub_detail 2.1 id</id> 22 </sub_detail> 23 <sub_detail> 24 <id>sub_detail 2.2 id</id> 25 </sub_detail> 26 </sub_details> 27 </detail> 28 <detail> 29 <id>detail 3 id</id> 30 </detail> 31 </details> 32 </master>' 33 ) xml from dual) 34 select extractValue(s.xml,'master/id') master_id 35 ,extractValue(value(dtl),'detail/id') detail_id 36 ,extractValue(value(subdtl),'sub_detail/id') sub_detail_id 37 from demo4 s 38 ,table(XMLSequence(s.xml.extract('master/details/detail'))) dtl 39 ,table(XMLSequence(value(dtl).extract('detail/sub_details/sub_detail')))(+) subdtl; MASTER_ID DETAIL_ID SUB_DETAIL_ID --------------------------- -------------------------- -------------------------- mater id detail 1 id sub_detail 1.1 id mater id detail 1 id sub_detail 1.2 id mater id detail 2 id sub_detail 2.1 id mater id detail 2 id sub_detail 2.2 id mater id detail 3 id


ご覧のとおり、新しいものは何もありません。 すべて同じ左相関。 注意したいのは、テーブル式subdtlの最後にある（+）だけです。 おそらく推測するのが難しくないように、外部接続を使用する必要があることを意味します。 指定しなかった場合、詳細3の文字列は取得されません。



それでは、目の前に何が現れたのでしょうか？ オブジェクトの種類は1つで、比較的限られた機能セットであり、実質的に無制限の機能セットを提供します。 私はこの実装が本当に好きです。 私が最も驚いたのは、Oracle corpがXMLに合わせてSQLのセマンティクスを調整する必要がなかったことです。 説明されているすべての機能（オブジェクト、パイプライン関数、テーブル式）は、このテクノロジで使用されますが、特別に作成されたものではありません。 誰でもそのような実装を実装できることがわかりました。 この大胆な実装は、OracleのSQLエンジンの能力と柔軟性を強調しています。
