TFS 2011の新しいチームブラウザー

おそらくあなたの多くは、ソースコード管理の分野での改善点を伝える投稿で、新しいチームエクスプローラーのスクリーンショットを既に見たことがあるでしょう。 このトピックでは、Team Explorerの一般的なニュースに集中したいと思います。

Visual Studio 2010のチームエクスプローラーはツリー構造を持ち、プロジェクトエクスプローラー（ソリューションエクスプローラー）と非常に似ていることを覚えています。 長い間、チーム内では、このモデルで十分であると考えていましたが、いくつかの制限があります。 しかし、数年前にチームメンバーのPower Toolsを作成したとき、たとえば、写真やユーザーリストを表示する機能をもっと必要としていました。 ツリーが制限されているか、そのようなことができないチームエクスプローラー。 コードレビューの機能を含める計画を​​含め、TFS 11の設計を開始するとすぐに、新しいアプローチが必要であることが明らかになりました。 特に、製品を使用している人々から多くのレビューを受け、新しい追加機能の実装をリクエストしました。



チーム内の私たちは、チームエクスプローラーのインターフェイスを再設計することにしました。 このウィンドウは、他の多くの機能ウィンドウ（または別の方法でページ）を含むフレームであり、ブラウザに似たナビゲーションを提供します。 これらの10個の追加ウィンドウでIDEをオーバーロードしない一方で、これにより必要な機能へのアクセスが容易になると感じました。

設計を開始するとすぐに、解決したいいくつかの新しい問題がありました。 完全なリストからはほど遠いです。

  • モダリティのないインターフェース。 使用するすべてのモーダルダイアログは、不十分なレイヤーインターフェイスにつながります。
  • 非同期エクスペリエンスに移行し、進行状況を表示し、時間のかかる操作をキャンセルする通常の方法を実行します。
  • 作業の開始時にチームエクスプローラーのすべてのデータ（ドキュメント、レポート、作業項目、アセンブリなど）を一度に取得することなく、開始時間とサービスへの接続を改善し、本当に必要な場合にのみこの情報を表示します。
  • 主なユースケースをより明確にします。
  • チームエクスプローラーの外観を更新します。
  • サードパーティの拡張機能にさらに機能を追加します。
  • 実装した一般的なプロセスアプローチを課すことなく、組織の経験に焦点を当てたシナリオを追加します。




メモリを更新するために、TFS 2010の古いチームエクスプローラーのインターフェイスの画像を次に示します。





ホームページに会う



TFS 11の新しいチームエクスプローラーは、ページナビゲーションを使用して、1つのウィンドウでより多くの機能を提供します。 ルート、つまりホームページには、ツールバーに家の形のブックマークがあります。 このページには、基本的なアクションへのリンクもあります。 また、大麻の「前後」の通常のシナリオをサポートし、作業の履歴を記憶します。 さらに、ウィンドウには検索用の入力フィールドがあります。





検索機能は、OutlookまたはTeam Web Access（2010）での実装方法に似ています。 この例では、「作成」という単語を含み、自分に割り当てられているワークアイテムを探しています。 検索バーの下のドロップダウンリストには、さらにフィールドのフィルタリングオプションが表示されます。 ダブルクリックすると、作業項目が開きます。または、クエリを開いて（「クエリとして開く」）、編集して後で保存することもできます。 これまでのところ、検索は作業項目に対してのみ機能しますが、将来新しいデータソースを追加することを望んでいます。





あなたの仕事を管理します



TFS 2010では、ワークフローは次のようになります。

  1. オープンソリューション
  2. いくつかの変更を加える
  3. 作業項目を探しており、必要に応じてそれらをコードの変更に関連付けます。
  4. 変更に対するコメントを入力します。
  5. チェックインする




TFS 11では、このプロセスを少し調整しました。 [マイワーク]ページでは、タスクを確認したり、現在の作業を追跡したりできます。





実行するタスクを決定するとすぐに、メニューのコンテキストで、この作業項目を「進行中の作業」として割り当てる機会があります。





これらの2つの作業項目は現在、現在のマークが付けられています。 コードを変更するまで、「完了」メニュー項目が表示されますが、表示されるとすぐに「チェックイン」項目が表示されます。 暫定版（開発者プレビュー）では、コードの現在の変更がこれらの作業項目に関連付けられます。 最初のベータでは、これらの作業項目を次の状態のステップに進める機能も追加されます。 たとえば、スクラムプロセステンプレートを使用する場合、これらのワークアイテムの状態は「To Do」から「In Progress」に移行されます。 また、この変更はWebダッシュボードでも確認できます。 もちろん、コンテキストメニューを使用して、「進行中の作業」リストから作業項目を削除できます。 ベータ版では、これらのワークアイテムのステータスも前のステップに戻ります。







コードに変更を加えると、すぐに他のリンクが表示されます。







コードに現在の変更がある場合、次のオプションが利用可能になります。

  1. チェックインする
  2. これらの変更のコードレビューをリクエストします。
  3. これらの変更を「シェルフ」（シェルフセット）に延期します。




最後のオプションから始めましょう。 サーバーでコードの現在の変更を個別に保存し、必要に応じてこれらの変更をロールバックできる「シェルフ」（シェルフセット）の概念を既にご存じでしょう。 これは、変更に取り組み、後で延期する非常に便利な機会です。 同時に、チームの別のプログラマーと一緒に、これらの変更を別のコンピューターで表示する機会があります。 この機能に基づいて、コードの変更をサーバーに保存するだけでなく、IDEの現在の設定（ソリューション、ツールバー設定、ブックマーク、ブレークポイントなどを開く）も保存するサスペンドチームを作成しました。 これにより、コードの作成に関与していたが延期する必要があったコンテキストに戻ることができます。 これにより、中断のコストが大幅に削減されます。







Suspendに電話するとすぐに、この作業を実際に延期する前に、自分が何をしたかを簡単に説明する機会があります。 この説明の入力フィールドのサイズが心配な場合は、縦方向および横方向に展開できます。 [一時停止]ボタンをクリックすると、ワークステーションのプロジェクトの現在の状態がロールバックされ、保留中の作業セクションに新しいエントリが表示されます。







拡張された形式では、この遅延作業に関連付けられた作業要素とコードを表示できます。







[変更の表示]リンクを使用すると、変更、作業項目などを確認できる詳細を表示できます。







もう1つの優れた新機能は、未読のワークアイテムのリストです。 作業項目が割り当てられているが、その内容を見たことがない場合、またはまだ表示されていない変更が加えられた場合、Outlookの未読メッセージと同様に太字で強調表示されます。 私にとって、これは何が新しくてまだ何をしていないかをすぐに理解し、コードに関する新しいコメントを見るなどの絶好の機会です。



コードレビュー





チームエクスプローラーの新しいバージョンで何ができるかを見てみましょう。 遅延作業のコードレビューを要求します。 保留中の変更ページのオプションには「その他」の項目があり、表示されるメニューにはコードレビューのリクエスト項目があります。







[新しいコードレビュー]ページで、レビューするチームの人を指定できます。 さらに、レビュー担当者が全体像を把握できるように、名前、プロジェクトエリア、反復を指定し、コメントを付けることができます。 [変更の表示]セクションには、変更のリストから一部のファイルを除外するためのリンクがあります。これにより、校閲者は必要なものだけに集中できます。 Submit Requestコマンドの後、コードレビューのリクエストが登録され、Code Review＆RequestsセクションのMy Workセクションに移動します。 後で現在のステータスを確認し、詳細とレビューを確認できます。







レビューアがメールで通知を設定した場合、コードのレビューのリクエストを受け取ったことを示す手紙をすぐに受け取ります。 いずれの場合でも、彼は自分の[仕事]セクションにこのリクエストを表示します。





リクエストをダブルクリックすると、レビュー担当者が詳細を確認できるコードレビューページに移動します。







この時点で、レビュー担当者はレビューするかどうかを決定し、[承認]ボタンをクリックできます。 または、このタスクを中止して他の人に送信します。 これを行うには、[レビュー担当者]セクションで、新しいユーザーを追加して[拒否]をクリックします。 この場合、コメントを追加することもできます。

レビューアは、リクエストと、このリクエストに関連付けられた特定のコードファイルの両方にコメントを追加できます。 上記のように、コメント入力フィールドは、より大きな説明に対応するために増やすことができます。







レビューアがいずれかのファイルのコードの分析を終了したら、コメントの送信ボタンをクリックしてコメントを付けることができます。 同時に、新しいコメントは太字で強調表示され、最も関連性が高いことが明確になります。 これらのコメントの交換はチャットの形で行うことができ、レビューアとリクエストの作成者が詳細をすばやく話し合うことができます。 最後のステップは、リクエスト全体の作業を完了することです。これには、Finishチームが使用されます。





レビュー後、リクエストの作成者とレビューの有用性についてさらに議論する機会があります。 少なくとも1つのコメントを追加すると、デフォルトで[コメント付き]オプションが有効になります。 あるいは、変更を「見栄えが良い」ものにし、ソースコードのメインブランチに分類されることに同意することをリクエストで簡単に指摘することもできます。 「Needs Work」項目は明らかであり、レビュアーの意見ではコードですべてが大丈夫というわけではなく、メインブランチに「チェックイン」する前にいくつかの変更が必要です。 もちろん、コメント内の各ファイルのすべてのコメントを詳細にメモできます。

レビューが完了し、「完了」ボタンをクリックすると、「マイワーク」ページのコーダーレビューとリクエストのリストからリクエストが消えます。 将来のベータ版では、レビュー担当者は最近のリクエストのリストを表示して、コメントを追加することもできます。



コードレビューリクエストの作成者はこれを見ています。


レビュアーがファイルにコメントを付けるとすぐに、クエリ作成者はこれらのコメントを未読で太字で「My Work」セクションに表示します。



レビュー担当者が作業を終了し、「完了」ボタンをクリックしてフィードバックを提供した瞬間です。







レビューを分析する前に、クエリ作成者は保留中の作業を「シェルフ」から復元します。 再開コマンドはコンテキストメニューから使用できます。将来のベータバージョンでは、「保留中の作業」を「進行中」セクションにドラッグアンドドロップするだけで、シェルフから環境、そのコンテキスト（開いているファイル、設定、ブレークポイント）が自動的に復元されます。







レビューアのコメントを分析した後、作成者は必要な変更を加えてレビューのリクエストを閉じることができます。これは、このタスクの作業が完了したことを示します。



その後、チェックインを行うことができます-これらの変更をバージョン管理システムのソースコードのメインブランチに配置します。 この場合、すべてがうまくいったという通知を受け取ります。





ビルド





アセンブリ管理に関連する機能をさらに追加しました。 「ビルド」ブランチを開いて、最近起動したプロセスのステータスを監視する必要がなくなったなどの簡単なことから始めます。 この情報は、[マイビルド]セクションのアセンブリページですぐに利用できます。 もちろん、これはBuild Explorerを置き換えるものではなく、作業を簡素化します。 また、多数のアセンブリ定義の管理における不便さについて多くのフィードバックを受け取りました。 提供された簡単なソリューションの1つは、このセクションでフォルダーを作成することでした。 これまでのところ、これをどのように改善するかについての最終決定はまだ行っていません。







「マイビルド」セクションには、Countinuous Integrationモードを使用している場合に、開始したか、変更に基づいて開始されたアセンブリが含まれます。 コンテキストメニューには、次のコマンドリストが含まれています。







新しいRequeueコマンドを使用すると、おそらくご想像のとおり、同じパラメーターでアセンブリを再起動できます。 これは、前回環境に関連する障害があったときに便利です。

「すべてのビルド定義」セクションには、すべてのアセンブリの定義が含まれています（すべてではなく、最初の100個）。 ここには、本当に必要なものを見つけるためのフィルターフィールドがあります。 一部のアセンブリ定義を「お気に入り」として指定して、常に手元にあるようにすることもできます。







お気に入りリストについては、最後のビルドの現在のステータスだけでなく、以前のコンパイル試行の履歴も小さなヒストグラムの形で表示されます。 TFS 2010と同様に、レポートリンクをクリックして詳細を表示できます。



ワークアイテム





現在のバージョンのTFS 2011 Developer Previewの作業項目のリストへのクエリを含むページは、TFS 2010と同じように見えます。これまでのところ、コンテキストメニューの一部のコマンドとアクションがこのページに直接移動することを除いて、何かを変更する基本的な計画はありませんリンクの形で。







おそらくベータTFS 2011では、いくつかの新しい変更が表示されます。



性能



すべての「長期にわたる」プロセスを可能な限りバックグラウンドに移行するために、多くの努力をしました。 ただし、Visual Studio UIとの対話的な相互作用により、現在のユーザーストリームでこれらのタスクを実行する必要がある場合がいくつかあります。 開発者プレビューコードが凍結されて以来、パフォーマンスの最適化を継続しており、これらの改善点はベータ版で確認できます。



拡張性



TFS 2010では、チームエクスプローラーの拡張オプションは制限されていました。 ツリーに新しいノードを追加し、コンテキストメニューを変更し、アクティベーションイベントを処理できます（ダブルクリック）。 TFS 2011では、拡張オプションが強化されました。

  • チームエクスプローラーに新しいページを追加し、それを共通のナビゲーション構造にリンクする
  • 既存のページに新しいセクションを追加する
  • ホームページへのリンクの追加（ホーム）
  • ホームページへのサブリンクの追加


チームエクスプローラーで独自のページを作成する場合は、当社のアプローチに従って構成することをお勧めします。 それでも、このページの作成方法は完全に制御できます。 チームエクスプローラーの拡張機能について詳しく説明します。
