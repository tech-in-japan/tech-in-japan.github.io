Due to the fact that Svelte works with the DOM ‚Äúin a different way‚Äù, MaterialUI began to get out all sorts of nasty things related to how the UI components that are added via js to dom are displayed. For example, a simple spinner is displayed every other time:







After a long search, what went wrong, it turned out that the spinner in the Material UI is added via JS after the document is ready (onLoad) but after this event, Svelte starts its addition to the DOM and the JS script Material simply does not see it since the event was before.



This had to be treated by adding to the parent component of Svelte:

Why do you need Svelte?



  • If you work alone and you urgently need to do MVP or a small admin panel. There will be problems with a large admin panel, because when you write new code you will forget the old one or you must write self-documented code, since Svelte does not "force" you to do this.
  • If the bundle size is very important to you. For example, you are making a web app for a country in which there is still no 3G or the Internet is very slow and very expensive. In most parts of the world, expensive programmers, not the Internet, so bundle size within reasonable limits will not be an actual problem in 99% of cases.

  • If you are just starting to learn how to write in JS and doing a front-end, Svelte will give you all the freedom to ‚Äúshoot in the foot‚Äù, at a desk neighbor or arrange a massive shooting in the legs. In fact, you will not need to learn JSX or understand code conventions for a long time, you can immediately do your first front-end project and see that you have already succeeded and something is even working.



Based on the latter, I don‚Äôt understand at all why Svelte maintainers are going to make TypeScript support? We love Svelte just for ‚Äúshooting at the legs‚Äù and TypeScript in my opinion is like all-terrain tracks for the Svelte sports car. And you know, after working with Svelte, I realized how much JS has changed in recent years, and that it is no longer JS.



Svelte gives the opportunity to work in the same old and lamp JS, without PropTypes, Flow and TypeScript.



What did I like?



  • Well readable if / else, which is not in Vue. I would really like to see something similar in Vue.
  • You can create your event in components and get rid of passing callbacks to functions for child, for communication between child> parent.



Why not use Svelte?



  • There is not a single fully working UI Kit. It is difficult to imagine a web app without UIKit; only large projects for which Svelte is just not suitable can write the UI Kit themselves. Screwing something vanilla goes into creating your own UI components (your UI Kit) or in the fight against bugs and ‚Äúlocomotives‚Äù of classes.

  • The implicit nature of some simple errors, and their capture only at the testing stage.

  • If you are going to make a multi-platform application (and now everywhere is mobile first), you will have to rewrite everything to ReactNative or finish the generated NativeScript with your hands (there is an NS generator from Svelte). In fairness, Vue with mobile development is also not doing very well, but it also has some working features.

  • You will have to take care of writing a very clear code convention for your developers in the project, as the tutorial and examples are not made for team development.
