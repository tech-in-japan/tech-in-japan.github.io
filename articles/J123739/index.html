elFinder 2.0ベータ。 パニックにならないでください！

「イースタンカールオブザギャラクシーの多くの文明では、それほど儀式的ではありませんが、elFinderは他のWebファイルマネージャーを置き換えることに成功し、一般的に受け入れられています。

まず、安いです。

第二に、彼は大きな赤いボタンを持っています

手紙で書かれたフレンドリーなアドバイス： パニックにならないでください！ 」



今日は私にとって本当に幸せな日です！

数ヶ月の非常に激しいが、信じられないほど楽しい仕事の背後にあります。

そして、Web elFinder 2.0ベータ版のファイルマネージャーをご紹介します 。



コミュニティ全体の支援、パッチ、翻訳に深く感謝したいと思います。

スタジオの同僚-このプロジェクトのために私を解放するためにすべての手間をかけてくれてありがとう。

そして、私は新しいelFinder開発者667bdrmを紹介したいと思います。 彼はdrupalのモジュールの著者です。

それでも、elFinderが統合されないフレームワークが事実上存在しないことに、私たちは非常に満足しています。 ちなみに、RBC Softはこれを行う最初の会社の1つでした。





イノベーションの説明に進む前に、1.xブランチの継続ではなく、2.0の理由を説明します。

実際、1.xブランチには2つの大きな欠陥があり、

リファクタリング/追加では修正できません-生産性が低く、完全なAPIがありません。

そのため、elFinderはゼロから書き直されました。

避けられない結果として、データ転送プロトコルが更新されました。

しかし、新しいクライアントは問題なく古いコネクタで動作します。



何が起こったのか見てみましょう。



性能



ディレクトリツリーはオンデマンドでロードされます。

ネストの深さは構成で構成され、デフォルトでは1つのレベルがロードされます。



ファイルパスのエンコードをmd5からbase64 +暗号化に置き換える -すべてのファイル操作を高速化しました。

もちろん、base64はファイルパスを隠す信頼できる方法ではありません。

ただし、ルートディレクトリからの相対パスはエンコードされているため、実際の位置に関する十分な情報が得られません。

パスの暗号化は、将来的に追加されます（偏執狂的）。



コネクタによって送信されるJSONサイズが大幅に減少しました。

古いバージョンでは、ディレクトリツリーは個別に、現在のディレクトリの内容とは別に転送されます。

新しいものでは、すべてのデータが重複せずに1つのオブジェクトで転送されるため、約30％節約されます。

ブール型をintに置き換えると、サイズが大幅に増加します。



大量のファイルは、サーバーだけでなくクライアントにとっても問題です。

2000要素のDOMを作成すると、Firefoxが数分間「フリーズ」します（safari / chromeの場合、このしきい値は5000から始まります）。

もちろん、これは非常にまれなケースであると判断することもできますが、私の経験では、ストレージと7000フォルダーを扱う必要のある企業がいるということです。

アイデアをくれたチェコ共和国の同僚に感謝します。Google検索で画像を読み込む方法のように、新しい要素をオンデマンドでDOMに追加してみませんか。

追加の利点-サムネイルは本当に必要な場合にのみ適用されます。



API



elFinderには、クライアント用だけでなくサーバー側用の完全なAPIが追加されました。

どちらもイベント駆動型モデルに基づいています。

サーバーAPIには、クライアントに送信する前にデータを変更する機能があります。たとえば、ファイルの名前を自動的に変更したり、ダウンロードした画像のサイズを変更したりする場合などに便利です。



そして今-最もおいしい



マルチルート+任意のタイプのデータストレージ。

一文の場合-elFinderは任意の数のルートディレクトリをサポートし、それらはファイルシステム上のフォルダだけでなく、

各ルートディレクトリは、独自の「ドライバー」を使用して「マウント」されます。

コネクタのアーキテクチャには3つの層が含まれます。

  1. アプリケーション層。 実際、elFinderクラス自体。 オブジェクトを使用してルートディレクトリを「マウント」します-「ドライバー」。

    クライアント要求を特定のドライバーのメソッドに変換します。

  2. 抽象ドライバー。 elFinderリクエストをストレージアクセスに変換します。

    彼はどのデータを返す必要があるかを知っていますが、データがどのように保存/取得されるかについてはまったく知りません。

  3. 特定のストレージドライバー。

    一般に、彼は誰についても何も知らず、単純なデータ操作しか実行できません。



したがって、新しいタイプのストレージのサポートを追加するには、「ファイル」を使用した基本操作を実装するかなり単純なクラスを記述するだけで十分です。



現在、2つのドライバーがあります

  • ローカルファイルシステム用
  • mysqlにファイルを保存します。

    このドライバーは（いくつかの例外を除いて）非常に機能的ですが、このデータストレージの方法は実稼働に適しているとは思いません。 主に概念を検証するために書かれました。



最後に上限を外すために、クライアントが唯一のデータトランスポートとしてjQuery.ajaxに関連付けられなくなったことを追加します。

つまり、異なるサーバーにあるコネクタとの通信を実装したり、Webソケットを使用したりできるようになりました。

また、マルチルートも同じ方法で実装できます。

データをelFinderのクライアント側が理解できる形式に変換することもできます。

これにより、古いコネクタのプロトコルがサポートされます。

そして今、私たちはスウェーデンのある会社のwebdav / svnのプロキシの実装に取り​​組んでいます。



検索。

elFinderは本当のファインダーになりました:)

検索はすべてのフォルダーで実行されます。 検索を特定のフォルダーに制限する方法はまだありません。



ファイルの種類でフィルターします。

設定でファイルタイプのリストを指定するだけで十分で、ユーザーにはそれらだけが表示されます。



ファイルを並べ替えます。

すべてのソートはクライアントで行われます。 6種類の並べ替えが可能です-名前、種類、サイズ、および先頭のフォルダと同じ。 設定で設定されますが、ソートをその場で変更するためのボタンを作成することもできます。



新しいファイル属性。

  • 「ロック」-削除を禁止する「rm」属性を置き換えました。 同様に、macosの属性-ファイルの削除と名前の変更を禁止します。
  • 「非表示」-ファイルを非表示にします。 書き込み/削除を禁止しません。


ファイルアクセス制御。

前のバージョンでは、1つの方法しかありませんでした-ファイルへのパスからの正規表現に基づくルールの説明。

新しいタイプのストレージの出現により、これだけでは十分ではありません。

たとえば、（id）によるデータベース内のファイルの場合、番号があり、多くの問題が発生します。

以前と同様に、承認/アクセス権などはファイルマネージャ自体の能力の範囲内ではないと考えています。 彼は正しいオプションを取得する必要があり、これはアクセシビリティを実現するのに十分です。 そのため、初期化中に特定のファイルへのアクセス権を「解釈」する関数またはオブジェクト/メソッドを転送する機能を追加しました。

ルートディレクトリごとに、独自のアクセス制御を使用できます。



既存のファイルの上書きを制御します。

これらの名前のファイルが存在するフォルダーにファイルをロード/移動するときに、名前にサフィックスが追加された新しいファイルが保存されるように構成することができます。



ファイル操作でユーザーインターフェイスがブロックされなくなりました

elFinderインターフェースをデスクトップファイルマネージャーインターフェースにできるだけ似たものにする方法を考えて、

Web 2,3,4.0で最も神聖なものに侵入しました-Ajaxスピナーとオーバーレイを殺しました:)

1ダースのファイルをダウンロードにドロップし、ロード中にファイルの名前を変更して、いくつかの新しいフォルダーを作成します。

特定の操作についてサーバーとの通信が指定した期間（デフォルトでは0.5秒）より長く続く場合、現在何を行っているかについての通知ウィンドウが表示されます。

私は告白します、私はcでした、スピナーは全く殺されませんでしたが、創造的に処理されただけで、通知ウィンドウに押し込まれました。

そして、オーバーレイは生き残りましたが、一時的な役割で、さらには透明なものでさえ-誰がそれに気付くでしょうか？



どこにでもドラッグ＆ドロップ。

これで、ファイルをディレクトリツリーから/にコピー/移動できます。 ツリー内のフォルダーにファイルをドラッグしてしばらくおくと、ファイルが開き、サブフォルダーにファイルをドロップできるようになります。

ドラッグアンドドロップからダウンロード-ファイルを現在のディレクトリにドロップするだけです（ブラウザがサポートしている場合）。
