COREmanagerフレームワークに基づいた開発。 パートナーがテクニカルサポートをアウトソーシングするためのソリューションを作成した方法





誰かが自分のビジネスを開始することを決め、仮想ホスティングを提供し、フランチャイズのために花やコーヒーの販売のためのオンラインサービスを開始したい場合、仕事を整理するための多くの既製のツールがあります。 一方、単純にビジネスを開始する場合は、この分野の多数の競合他社に備える必要があります。 「燃え尽き」の可能性が高まります。

競争の少ない開発が不十分な地域に行くと、タスクを自動化するための既製のツールがない場合があります。 はい、最初はすべてを手動で行うことができますが、顧客の数が増えると、プロセスの最適化について真剣に考える必要があります。



自動化ツールを開発するとき、COREmanagerフレームワークを基盤、スケルトン、製品として使用できます。 これにより、コーディングに費やす時間が短縮され、さまざまなプログラミング言語を使用してさまざまな機能を実装できます。



カットの下-ISPlicenseによる外部委託技術サポートシステムの開発の詳細。



ISPlicenseは、競争の激しいビジネス分野で革新的なサービスを提供する会社の代表的な例です。 彼らはホスティングセクターを選択し、市場に参入する既存および新規プレーヤーにサービスを提供するビジネスモデルを構築しました。 ISPsystemソフトウェア製品のライセンスを再販することから始め、当社の主要パートナーの1つになりました。 しばらくして、同社のサービスパッケージは、 アウトソーシングテクニカルサポートの提供によって補完されました。 リソースの不足、ほとんどのクライアントからの地理的な遠隔性、およびその他の理由により、多くのホスティング事業者はサードパーティ組織に技術サポートを提供することを真剣に考えました。



ISPlicenseは、2つの方法でアウトソーシングテクニカルサポートを提供します。



  1. エンドユーザーに無料でテクニカルサポートが提供されるホスティング事業者と連携する場合、契約に従ってホスティング事業者への返信と請求に費やされた時間が記録されます。
  2. クライアントのサポートが無料でない場合、費用は価格表に基づいて形成されます。料金は、費やした時間または提供されるサービスのリストによって異なります。 リクエストが完了すると、ホストに代わってクライアントに請求が行われ、ISPlicenseはその割合を受け取ります。


ここで、自動化ツールを使用せずにサポートプロセスを配置する方法を考えてみましょう。



最初は、さまざまな方法で顧客との協力を整理できます。誰かが従業員のユーザー名とパスワードを入力し、誰かが個々のユーザーを開始し、BILLmanagerを使用して誰かに接続できます。 ただし、ホストされているサーバーの数が非常に多くなり、テクニカルサポートオペレーターが多数のブラウザータブで混乱し、同時にチケット内の新しいメッセージを追跡する必要がある場合を想像してください。



この状況を防ぐために、ISPlicenseは、BILLmanagerおよびその他の課金システムと統合されたCOREmanagerに基づく技術サポートシステムを開発しました。 この製品は、TicketManagerと呼ばれます。



おそらく今、 COREmanagerについてもう少し説明する価値があります。 これは、C ++で記述されたコンストラクタです。



その開発は2010年に始まりました。 COREmanagerは、製品の一般的な機能を別のエンティティに組み込み、コンポーネントの一貫性を確保するために作成されました。 BILLmanager、ISPmanager、VMmanager、DCImanager、およびその他のコントロールパネルは、「カーネル」の拡張機能になりました。これは、最も経験のあるISPsystem開発者とは別のチームによって作成されています。 その結果、開発時間が短縮され、バグの可能性が減少し、最終製品の速度が向上しました。



COREmanagerは無料で、その使用方法を説明した詳細なドキュメントがあり、ほぼすべてのタスクを解決するツールの開発に適用できます。 メニュー構造は、Webインターフェースを介して、またはxmlファイルを作成することにより作成できます。また、そのインタープリターがオペレーティングシステムにインストールされている場合、任意のプログラミング言語を使用してイベント処理メカニズムを実装できます。



したがって、ISPlicenseはCOREmanagerを選択してTicketManagerを作成しました。 はい、テクニカルサポートに既製のソリューションを使用したり、BILLmanagerのプラグインを作成して問題を解決したりできますが、ISPlicenseプログラマーは、COREmanagerでできることを自分で試してみたかったのです。 :)



アウトソーシングサービスを開始した後、時間の経過とともに問題解決の問題が形になり、テクニカルサポートシステムを開発する必要が生じました。 次の前提とタスクが形成されました。



  • ユーザーに相談するために各プロバイダーのパネルにアクセスするたびに時間がかかりすぎます。 さらに、テクニカルサポートの従業員が複数のプロバイダーと連携している場合、従業員は複数のタブまたはブラウザーウィンドウを開く必要があるため、混乱が生じる可能性があります。 同時に、ホスティング事業者はさまざまな開発者からの請求書を使用できます。 したがって、チケットフローを1つの製品に集約し、単一のビューに表示する必要があります。
  • サポートの請求は1時間ごとなので、チケットに費やした時間の便利なアカウンティングを実装する必要があります。
  • 利便性と時間の節約のために、エンドユーザーに来るアカウントの自動生成を実装します（たとえば、有料の管理のため）。
  • ホスティング事業者向けのレポートとアカウントの作成も自動化します。
  • プロバイダーとISPlicenseの両方に対する不要な質問を避けるために、エンドクライアントはテクニカルサポートが外部委託されていることを知らないようにする必要があります。


結果の製品は、チケットシステム自体と、請求クライアントにインストールされるハンドラーの2つの部分で構成されます。 別のテクニカルサポートシステムまたは課金システムと統合できるAPIもあります。



テクニカルサポートパネルの主要な機能を理解してください。

チケットのリストは最小限であり、アプリケーション表示メニューでアプリケーションを操作するためのツールを利用できます。







例に対するアピールを開き、それを使用して実行できるアクションを確認しましょう。







  • 返信します。 オペレーターの後にチケットをブロックし、応答入力フォームを開き、費やした時間のアカウンティングを開始します。
  • ご注意 このリクエストに関連する同僚にメモを残すことができます。 プロバイダーにもエンドクライアントにも表示されません。
  • お客様 エンドクライアントに添付されたメモ用の大きなテキストフィールドを開きます。
  • プロジェクト ホスティング事業者の同様のフィールドを呼び出します。
  • お金を引き出します 。 ホスティング課金システムでのエンドカスタマーからのサービスの提供に対してお金を請求します。
  • 閉じます。 オープンチケットのリストに戻ります。


さらに、ウィンドウにはチケット開始者に関するすべての情報が表示されます。 クライアントのIDをクリックすると、請求が行われます。サーバー名をクリックすると、このサーバーのコントロールパネルに移動します。



呼び出しの受信に関連するサービスに関する情報も表示され、アクセスデータを含む、サービスが展開されているサーバーに関する情報が表示されます。



チケットをロックして、開かれた応答入力フォームのアクティブな要素を確認します。



  • ロック解除 オペレーターからチケットのバインドを解除し、他の人が答えを取ることを許可します。
  • 部門 。 ホスティングプロバイダーの別のユニットにアプリケーションを転送します。 たとえば、技術的な専門家から会計まで、財務上の問題を解決する場合です。
  • 内部解説 。 クライアントおよびプロバイダーから非表示になるメッセージをチケットに書き込むことができます。
  • 過ごした時間 。 チケットへの応答の期間。 数分で測定。 値を手動で変更すると、自動タイミングが停止します。
  • タイムカウンターを停止します。 オペレーターが近い将来に要求を解決するためのアクションを実行しないと想定される状況で使用されます。 たとえば、ISPmanagerを展開する前の従業員は、仮想マシンへのオペレーティングシステムのインストールが完了するまで待機します。
  • ステータス チケットのステータス。



    • 開く リクエストに応じて作業が進行中です。
    • 閉店 リクエストが完了しました。
    • その過程で 。 要求がサードパーティによって実行され、操作が長く、プロセス制御が必要な場合に使用されます。 「書き込み」タスクの邪魔にならないように、要求はリストの一番下にあります。
    • 延期 。 チケットは、「延期」フィールドに示された時間が経過するまで一般リストに表示されません。


  • 内部コメントをプロバイダーに送信します 。 書面はエンドユーザーには表示されませんが、ホスティングプロバイダーの従業員には表示されます。
  • クライアントによるチケットのクローズを拒否します 。 債務問題や違法行為を解決するときに使用されます。


相談中、エンドユーザーはISPのライセンススペシャリストが質問に答えていることを知りません。 彼は、会話が自分のホスティングプロバイダーの従業員とのものであると考えています。



製品の実装には、テクニカルサポートパネルで約5000行、BILLmanagerやその他の課金システムとの統合モジュールで約500行が必要でした。



興味のある方は、TicketManager API 、および以下のネタバレ-BILLmanagerの統合モジュールのソースコードをご覧ください。



必要な機能の大部分がCOREmanagerに既に実装されているため、必要な機能をコーディングよりも単一のリストにまとめることに多くの時間を費やしたことがわかりました。 まあ、インターフェイスもボタンを配置する場所を示すためだけに記述する必要はありませんでした。
