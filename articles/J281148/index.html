「理想的な」戦略の開発



他の多くの場合と同様に、取引所での取引戦略の開発に関しては、理想と現実の2つの方法があります。 開始するには、理想的な戦略開発プロセスを検討してください。



ステップ1.モデルの選択


まず、トレーダーは前述の市場の非効率性の1つを選択するか、新しい非効率性を発見する必要があります。 疑わしいものが特定の市場ロジックを満たす場合、価格曲線を分析するときにいくつかのアイデアが表示される可能性があります。 反対のオプションも適しています-行動パターンの理論から実際のデータでの検証まで。 確かに、すべての法律と非効率性は、おそらく他の市場参加者によって長年にわたってすでに発見され、研究されています。



モデルを決定したら、それに対応する価格異常を選択する必要があります。 次に、定量的な式を使用して、または極端な場合は定性的な基準を使用してそれを記述します。 例として、前の記事の循環モデルを使用できます。







この場合、（Ci）は持続時間、（Di）は位相、（ai）は振幅です。 歴史上最も成功した基盤の1つであるルネサンスメダリオンは、隠れたマルコフモデルとともにサイクルモデルを使用して良好な結果を達成できたことを保証します。



ステップ。 2.検索


また、選択した異常が、トレーダーが取引する資産の価格曲線に実際に現れることを確認する必要があります。 これは、履歴データ-D1、M1、および時間軸に沿った異常の分布を示す目盛り値を使用して確認できます。 全体の質問は、過去を掘り下げる必要があるかどうかです。 回答：可能な限り。 異常の状態が明らかになるまで。 次に、価格曲線の異常を見つけて実証するスクリプトを作成する必要があります。 この例では、これはユーロ/ドルのペアの周波数スペクトルになります。







任意の時間間隔（月と年）でスペクトルがどのように変化するかを調べて、それをランダムデータのスペクトルと比較する必要があります（Zorroなどの一部のプログラミング言語では、曲線をランダム化する特別な関数があります）。 異常の兆候やランダムな分布との違いが見つからない場合、トレーダーは異常の検出方法を改善する必要があります。 そして、他のすべてが失敗した場合、ステップ1に戻ります。



ステップ3.アルゴリズム


次のステップでは、選択した異常に続いて、ポジションをオープンおよびクローズするための取引シグナルを生成するアルゴリズムを作成します。 通常の状況では、市場の非効率性は価格曲線にほとんど影響しません。 したがって、アルゴリズムは、ランダムノイズから異常の兆候を分離するために十分に微調整する必要があります。 同時に、可能な限りシンプルで、最小数の空きパラメーターに基づいている必要があります。 このバージョンでは、スクリプトは正弦波の各窪みとピークの位置を変更します。これは、支配的なサイクルの先頭になります。 



function run() { vars Price = series(price()); var Phase = DominantPhase(Price,10); vars Signal = series(sin(Phase+PI/4)); if(valley(Signal)) reverseLong(1); else if(peak(Signal)) reverseShort(1); }


これがシステムの中核です。 今がバックテストの時です。 この場合のタスクの精度はそれほど重要ではありません。アルゴリズムの境界を決定するだけです。 彼は特定の市場状況で一連の収益性の高い取引を行うことができますか？ そうでない場合は、2つのオプションが残ります。別の方法を使用して書き換えるまたは再作成します。 この段階では、アルゴリズムにトレーリングストップやその他のスタッフィングを詰め込む必要はありません。 それらは結果をゆがめ、一から利益の幻想を作り出します。 アルゴリズムは、時間通りにポジションをクローズする場合にのみ、単純に利益を上げる方法を学習する必要があります。



この段階では、バックテストのデータも決定する必要があります。 動作テストでは、M1とティッククォートで十分です。 毎日のデータは良くありません。 データの量は、異常の期間（第2段階で決定された）とその性質に依存します。 通常、期間が長いほど、テストの精度は高くなります。 10年より深く掘り下げることは意味がありません。 少なくとも実際の市場行動に関しては、市場は10年間で大きく変化しています。 古いデータは誤った結果をもたらします。



ステップ4.フィルター


単一の非効率性が永遠に続くわけではありません。 市場は、相場の任意の行動の期間を通過します。 したがって、成功する取引システムごとに、非効率の有無を判断するフィルターを用意することが重要です。 フィルターは、信号と同じくらい重要ですが、それ以上に重要ではありません。 しかし、さまざまな理由で、しばしば忘れられます。 以下は、取引システムのフィルターを設定するスクリプトの例です。 



 function run() { vars Price = series(price()); var Phase = DominantPhase(Price,10); vars Signal = series(sin(Phase+PI/4)); vars Dominant = series(BandPass(Price,rDominantPeriod,1)); var Threshold = 1*PIP; ExitTime = 10*rDominantPeriod; if(Amplitude(Dominant,100) > Threshold) { if(valley(Signal)) reverseLong(1); else if(peak(Signal)) reverseShort(1); } }


ここでは、価格曲線への支配的なサイクルの期間を中心に、その振幅を測定するバンドパスフィルターが使用されます。 振幅が設定された制限を超える場合、非効率性があることを確認し、それに応じて行動することができます。 また、トランザクションの期間は最大10サイクルに制限されています。これは、2番目の段階で、支配的なサイクルが比較的短い期間で現れたり消えたりすることがわかったためです。



ここで何がうまくいかないのでしょうか？ フィルターを追加するのは、テスト結果を改善するという理由だけで間違いです。 フィルターには、市場の振る舞いやアルゴリズムのシグナルから始まる理論的根拠が必要です。



ステップ5.パラメーターの最適化


何らかの方法ですべてのシステムパラメータが結果に影響します。 ただし、価格曲線に基づいてエントリポイントと出口ポイントを直接決定するのはごくわずかです。 これらの適応可能なパラメーターを特定して最適化する必要があります。 この例では、ポジションのオープンは、曲線の動作を予測するフェーズとフィルターによって設定されたしきい値によって決定されます。 トランザクションの終了時間は、出口点によって決まります。 DominantPhaseフィルター定数やBandPass関数などの他のパラメーターは、値が市場の状況に依存しないため、最適化する必要があります。



多くの場合、最適化は誤って解釈されます。 遺伝的または暴力的な最適化の方法は、多次元パラメーター空間で利益のピークを見つける試みです。 ほとんどの場合、これは生体システムの動作とは関係ありません。 パラメーターの一意の組み合わせを取得できますが、優れたテスト結果が得られます。 わずかに異なる履歴期間でも、完全に異なるピークが得られます。 履歴曲線の最大利益の値を追跡する代わりに、最適化は次のように実行する必要があります。



  • パラメータ値に対するシステムの感度を決定できます。 システムがパラメータの最小限の変更で境界を見落としている場合は、手順3に戻ることをお勧めします。状況が解決しない場合は、最初の段階に戻ります。
  • 最適化により、アクティブなパラメーターポイントを見つけることができます。 これは、パラメータが最も信頼できる領域です。つまり、わずかな変更でも肯定的な結果に大きな影響を与えない領域です。 考えられるように、これらは狭いピークではなく、パラメータ空間の広い丘です。
  • システムをさまざまな資産に適合させ、パラメータの変更を最小限に抑えて株式ポートフォリオを操作できます。
  • 「ライブ」取引のプロセスと並行して、定期的な間隔でパラメータを現在の市場状況に適合させることにより、システムの寿命を延ばすことができます。


パラメーターを最適化するためのサンプルスクリプトを次に示します。 



 function run() { vars Price = series(price()); var Phase = DominantPhase(Price,10); vars Signal = series(sin(Phase+optimize(1,0.7,2)*PI/4)); vars Dominant = series(BandPass(Price,rDominantPeriod,1)); ExitTime = 10*rDominantPeriod; var Threshold = optimize(1,0.7,2)*PIP; if(Amplitude(Dominant,100) > Threshold) { if(valley(Signal)) reverseLong(1); else if(peak(Signal)) reverseShort(1); } }


2つの最適化された呼び出しは、開始値1.0と範囲（0.7..2.0）を使用して、2つの主要パラメーターのアクティブポイントを決定します。 利益率の図では、これらのポイントは赤い列でマークされています。











この実施形態では、オプティマイザは、正弦波位相に対して１．３に近いパラメータ値を選択し、ユーロ／ドルペアの現在の比率に対する閾値振幅に対して１．０の値（ピークに対しては０．９ではない）を選択する。 ここでの終了時間は最適化されていません。



ステップ6.サンプル外の分析


パラメータの最適化により、検証テストの品質が向上します。 ただし、システムはまだ価格曲線に合わせて調整されていません。 したがって、これまでのテスト結果は役に立ちません。 次に、サンプル内およびそれ以降のデータをサイクルに分割する必要があります。 サンプルは、システムのトレーニング、テスト用のサンプル外の分析に使用されます。 これの標準的な方法は、 前方解析です。 動的サンプリングまたは履歴データのスライディングウィンドウを使用して、2つのループを分離します。



残念ながら、フォワード分析では、トレーニング時間とテスト時間という2つの新しいパラメーターがシステムに追加されます。 2番目は重要ではありません。主なことは、比較的小さいことです。 しかし、トレーニング時間は重要です。 トレーニングが短すぎると、効果的な最適化のための価格で十分なデータを取得できません。 長すぎると、結果に悪影響を及ぼします。 トレーニングが行われている間に、市場はすでに変化を「消化」する可能性があります。 したがって、トレーニング時間自体を最適化する必要があります。



5サイクルのステップでのフォワード分析（前のスクリプトでNumWFOCycles = 5に設定）は、バックテストのパフォーマンスを年間収益の100％からより現実的な60％に減らします。 分析の楽観的な結果とエラーを回避するために、わずかに異なるシミュレーション条件で数回実行できます。 システムに制限がある場合、結果はあまり変化しないはずです。 それ以外の場合は、手順3に戻る必要があります。



ステップ7.現実のテスト


サンプルの外部で分析段階を通過した後でも、結果に何らかの歪みが残っています。 システム自体の制限または開発プロセスの欠点（資産、アルゴリズム、チェック時間などの選択）が原因ですか？ これを見つけることは、取引戦略に取り組むことの最も難しい部分です。



最良のオプションは、このためにWhite's Reality Checkと呼ばれる方法を使用することです。 また、パラメータとアルゴリズムの選択に厳密な順序が必要になるため、実用的ではありません。 他の2つの方法はあまりよくありませんが、使用する方が便利です。



  • モンテカルロ 。 データを変更せずにシャッフルすることにより、価格曲線をランダム化します。 次に、システムをトレーニングして再度テストし、再度実行して結果の分布を生成する必要があります。 ランダム化は、すべての価格の異常を取り除きます。 しかし、実際の価格曲線の結果がランダム分布のピークからそれほど遠くない場合、それはおそらく偶然によるものです。 これはどういう意味ですか？ 手順3に戻ります。
  • オプション この方法は、前の方法の反対です。 彼は、ポジティブな結果を得るために、価格曲線のさまざまなバリエーションに学習システムを適合させます。 ほとんどの異常をサポートするオプションは、サンプリングの増加、トレンドの影響の排除、または価格曲線の反転に関連しています。 システムがそのようなオプションで勝つが、ランダムな価格では勝てない場合、あなたは本当に価値のある何かを選びました。
  • ROOSテスト（実際にサンプル外） 。 これは、コンピューターから価格変動に関するデータが削除されるまで、昨年（この例では2015年）のデータを作成するプロセスにおける完全な例外を意味します。 そして、システムが完全に完成した場合にのみ、今年のデータをダウンロードしてテストを実行する必要があります。 ここで、最後のテストで目的の結果が表示されない場合にシステムを改造しないように、すべての意思を拳に集める必要があります。 脇に置き、手順1に戻るだけです。


ステップ8.リスク管理


したがって、システムはすべてのチェックを生き延びました。 これで、リスクの低減と生産性の向上に集中できます。 アルゴリズムとそのパラメーターの入力に戻る必要がなくなりました。 出力で得られるものを最適化する必要があります。 単純な時間制限とキャンセルの代わりに、さまざまなトレーリングストップのメカニズムを使用して終了できます。 例：



  • 一定時間後に終了する代わりに、ストップロスを1時間ごとのロス値に設定する必要があります。 効果は似ていますが、失敗した取引の閉鎖はより速くなり、後で成功します。
  • 特定の利益値がロールバック値を上回ったときにストップロスを設定する必要があります。 利益率を取得しても全体的な生産性が向上しない場合でも、それは有用です。


ストップロスを設定するスクリプトの例を次に示します。 



 function run() { vars Price = series(price()); var Phase = DominantPhase(Price,10); vars Signal = series(sin(Phase+optimize(1,0.7,2)*PI/4)); vars Dominant = series(BandPass(Price,rDominantPeriod,1)); var Threshold = optimize(1,0.7,2)*PIP; Stop = ATR(100); for(open_trades) TradeStopLimit -= TradeStopDiff/(10*rDominantPeriod); if(Amplitude(Dominant,100) > Threshold) { if(valley(Signal)) reverseLong(1); else if(peak(Signal)) reverseShort(1); } }


もちろん、今度は最適化プロセスを経て、出力パラメーターのフォワード分析をもう一度行う必要があります。



ステップ9.現金の分配

ステップ10.実際の取引の準備


取引システムのユーザーインターフェイスを決定する時が来ました。 これを行うには、どのパラメーターをリアルタイムで変更する必要があり、どのパラメーターが一定であるかを理解する必要があります。 次に、取引量管理方法をそれに合わせて調整し、「パニックボタン」を実装して利益を得て、悪いニュースが表示されたときにキャッシュに移動する必要があります。 取引のすべての重要なパラメータをリアルタイムで表示し、システムを再トレーニングする機会を追加し、実際の結果とバックテストの結果を調整する方法を含める必要があります。 トレーダーは、空間と時間のどこからでもシステムを制御できることを確認する必要があります。 これは、5分ごとにチェックする必要があるという意味ではありません。 しかし、昼食時に携帯電話で取引した結果を見る機会があれば、決して害はありません。
