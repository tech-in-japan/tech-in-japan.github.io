アジャイル開発での毎日の会議：15分、これなしではリリースを確認できません

ほとんどのIT企業は、プロセスを最適化し、すべてのチームメンバーの作業を同期するように設計された毎日の内部会議、ステータス会議、または短いスタンドアップに慣れています。 最適なのは、そのような会議が15〜20分を超えない場合です。







今日のアジャイル開発では、どこにも集会はありません。 ただし、スクラム、カンバン、またはXPチームの作業会議は異なり、独自の特性があります。 この記事では、スクラムとかんばんの違いを理解し、会社の成長の重要な刺激因子として、毎日の会議の必要性を支持する議論を提供します。



内部会議は、あらゆる組織の文化の要素です。 会議の目的が何であるかは関係ありません。製品のリリース前に、新しい切手またはタスクステータスのリリースについて繰り返し議論することです。各ミーティングはチームを集め、作業の透明性の原則を保証します。



有益な効果の外観を作成せず、「目盛り」のために会議を開催しないことが重要です。 各ラリー参加者の15分の無駄な労働時間でさえ、会社にとってまともな金額になる可能性があるため、チームのステータスミーティングは有益なはずです。



アメリカのある会社が引用した統計によると、ほとんどの毎日の仕事の会議は午前11時に予定されており、それらの63％は事前に準備されたアジェンダなしで始まります。 開催された会議の33％は、従業員自身によって無効と見なされます。



残念なことに、最後の大きな数は、「古代の」官僚的企業から、そのような会議の役割も同様に重要である若いスタートアップまで、あらゆる企業を追い抜くことができます。



一般的に、アジャイルチームでは、チームとの毎日の会議のグローバルな意味は、タスクを完了するまで迅速、効率的かつ確実に導くことになります。 このようなステータス会議は、しばしばスタンドアップと呼ばれ、「短い会議」に意味があります。



ほとんどの場合、このような会議を開催し、 日常的なタスクを回避する方法に注意を払うプロジェクトマネージャーまたは製品マネージャーは、立ち上げ中に一部のチームメンバーが無駄なエキストラにならないように常に監視する必要があります。 以下に例を示します。



  • 「偽の仕事中毒」とは、会議中に何十もの重要なものを持っている必要があるため、ラップトップから引き離すことができない非常に「忙しい」従業員です。 そのようなチームメンバーの存在は意味を持ちません。 現時点でそのような「働き者」がソーシャルネットワークで単に対応し、注意深く聞いているふりをしていても驚くことではありません。
  • 「目を開けて眠る 。 」 そのようなチームメンバーにとって、次の立ち上げは単純な形式です。「ダニのために」どこかに行き、何かを思いつく必要があります。 彼が聞いているふりをしているかどうかにかかわらず、これはしばしば起こりますが、あなたが今言われたことをもう一度尋ねると、すべてが非常に明確になります。
  • Upstartは非常に活発な従業員であり、誰もが彼の積極的な地位に感謝するように、非建設的に批判やコメントをするのが大好きです。
  • 「すきからのPR 。 」 毎回必要なまたは必要のない従業員は、一般的な議論に利益をもたらすことなく、自分の仕事とプロジェクト/製品にとってのその重要性に集中します。
  • 「気を散らす」 -会議のトピックを回避し、別のトピックに突然切り替えます。たとえば、プロジェクトの予算について話し合います。


スクラムとカンバンでの毎日の会議：違いは何ですか？



スクラムとカンバンのアジャイル開発方法論は、その本質だけでなく、チームが開催する毎日のミーティングでも異なります 。



スクラムチームのデイリーラリーの主な目標は、チームがすべての反復を完了できるかどうかを追跡するか、できるだけ早く完了できない理由を特定することです。 スクラムで立ち上がるのは、人を志向することです。 短い集会では、チームの各メンバーが交代で過去1日の結果とタスクの現在のステータスを発声し、チームが今日特定のタスクを遂行することを約束します。 問題がある場合は、それらも表明されます。



カンバンでの集会の目的は、すべての段階でタスクに費やす時間を最小限にすることです。 カンバン会議はオプションですが、タスクの流れに実際に影響を与える可能性があるため、本質的に助言的なものです。 この集会は、特定のボードとそれらのボトルネック（英語のボトルネック）の特定に焦点を当てています。







  • モデレーターが参加者を収集します。 ほとんどの場合、これはプロジェクトマネージャーまたはプロダクトマネージャーチームです。
  • チーム全体が取締役会に焦点を当てています。 ボードが製品管理サービスのオンラインツールであり、チームが異なる場所に分散している場合、一般的な電話会議が開催されます。
  • チームは、上から下へ右から左へタスクを「通過」し、次のステージへのタスクの迅速な転送のオプションについて説明します。 チームの誰もが発言する権利を持っています。
  • 左端の列はシャットダウンです。 完了に最も近いタスクが優先されます。 タスクが右端の列に速く移動するほど、タスクにかかる時間は短くなります（リードタイム）。
  • モデレーターは、特定のタスクを完了列に転送するのを妨げるものを明確にします。 原因と仮定が作成されます。 タスクがブロックされている場合、そのタスクにはマークが付けられ、ブロックされた理由に関するコメントが添付されます。 そして、各タスクに対して。 以前に延期されたタスクは、チームメンバーによって「整理」されます。 誰もが仕事をするので、次の会議までにこれらの仕事は次の列に移されます。


主な違い：



スクラムラリーは人に焦点を当て、カンバンラリーはタスクに焦点を当てます







かんばん会議中にクラシックスクラム会議の質問をすることもできますが、ここでもチームメンバーではなくタスクに重点を置いています。



  • タスクの移動を妨げるものは何ですか？
  • ストリーム内でタスクはどのように進行していますか？
  • 何をどのように改善できますか？


Daily ScrumからDaily Kanbanに切り替えるのは難しいですか？



スクラムからかんばんに切り替える場合、新しい会議形式への適応はスムーズに進むはずです。 議論の形式は変化していますが、時間の節約の結果はすぐに明らかになります。



タスクの凍結を避ける方法は？



まず、タスクが「フリーズ」する理由を理解し、効果的に加速するためのアクションを取ることが重要です。 「抑制」リリースタスクの理由は次のとおりです。



  • 多くの場合、タスクが完了に近づいているとき、それは実行者にとっての優先事項ではなくなり、彼は自動的に新しいタスクに切り替えます。 そのため、完全に完了していないタスクは、長時間「フリーズ」する可能性があります。
  • 多くの場合、多くのタスクは製品所有者または製品マネージャーによる確認段階で行き詰まります。 これは、後者が他のタスクでビジーであり、現在のタスクがリリースから1ステップ離れているために発生します。 頻繁なカンバン会議は、これらの未完了のタスクを時間内に確認して完了するのに役立ちます。
  • 仕事はそれに割り当てられた時間を埋めます。 反復が完了するまで、ボードのどの部分でもすべてのタスクがハングします。 また、特定のタスクが以前に実行された場合でも、最終段階に到達しない可能性があります。 また、かんばん大会の目標はタスクに費やす時間を減らすことであるため、立ち上げ中にタスクをリリースにプッシュすることができ、プッシュする必要があります。


ラリーを最適化する方法。 または効果的な立ち上がりを妨げるものは何ですか？



アジャイルチームの毎日のスタンドアップの各参加者は、会議をルーチンに変えるのではなく、効果的なイベントに変えるのに役立つ特定のルールに従ってガイドされる必要があります。



  • 遅れないでください 。 毎回、内部会議の記録された時間は、時間の経過とともに必携のイベントになり、チームメンバーによって忘れられることはありません。 モデレーターはこれに注意を払い、全員にとって都合の良い時間に会議を開催する必要があります。 多くのIT企業では、「次の集会に向けて詩を学ぶ」ことから小銭を貯金箱に入れることで小額の罰金を課すことで遅延に対処するのが慣習となっています。
  • 立ち上がって会議にしないでください 。 かんばんでの短い会議形式では、常設事項についての迅速な議論が行われます。 一部のクリエイティブチームは、ゲームボールの助けを借りて次の単語を「伝え」、誰かが叫び声またはマイクを使用します。一部のチームでは、ラリーがバーで開催されます（もちろん、共通の合意により）。
  • 手ぶらで来てください 。 職場にラップトップと携帯電話を置いておくことをお勧めします。 15分のかんばん会議で目をリラックスできます。
  • 大きな問題を解決しないでください 。 短いスタンドアップは、大規模で問題のある議論を目的とするものではなく、個別の時間を割り当ててください。
  • 報告しないで、情報を共有します。 かんばんの立ち上げは、マネージャーへのレポートではなく、チームミーティングであるため、全員と情報を共有できる必要があります。
  • 雄弁のスキルを開発する 。 あいまいな言葉遣いを避け、すべての人に与えられた考えを明確に表現しないでください。 これは、特に会議の時間形式が限られていることを考慮する場合、学ぶ価値があります。 多くの場合、複雑な技術的なストーリーを、すべての人がアクセスできるシンプルなストーリーにまとめることは困難です。 それから、最も重要なものを表明することが重要であり、技術的な詳細は後のために残します。
  • 準備してください 。 ミーティングのステータスの前に、タスクのステータスを知り、明確に理解する必要があり、急いでそこに何がどのようにあるかを覚えてはいけません。
  • 聴衆を尊重してください 。 多くの場合、近くの集会の参加者は、自分のステータスを伝えたり、タスクについて意見を表明したりするだけで、他の人と干渉して人生について話し始めます。






おわりに



生産的な会議は、新鮮なアイデア、明確な決定、短期的な行動計画で終了する必要があります。 製品マネージャーとすべての参加者は、自分のステータスとプロジェクト全体のステータスを完全に理解して職場に戻り、プロジェクト管理のプラットフォームに必要な変更を加えます 。
