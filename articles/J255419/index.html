Windows Server vNextでボリュームレプリケーションを構成する

すべての人に良い一日を！



今日は、Windows Server vNEXTの新しいバージョンで導入され、Technical Previewの予備バージョンでのテストとテストで既に利用可能な非常に興味深い機能、つまりボリュームレベルでのレプリケーションについてお話ししたいと思います。 WS vNEXTでは、この機能はストレージレプリカと呼ばれるようになりました。 それがどんな動物であり、それが何であるか-カットの下の詳細。







複製オプションと機能



ストレージレプリケーション、ストレージレプリカ（SR）は、クラスターまたはWindows Server vNEXTサーバー間でブロックレベルの同期（場合によっては非同期）ボリュームレベルのレプリケーションを可能にするWindows Serverの新機能です。 トランスポートとして、SMB3プロトコルが使用されます。



現在、サーバーサーバーとストレッチクラスターのレプリケーションオプションがサポートされています。 クラスタ間レベルのレプリケーションはまだ実装されていませんが、計画に含まれています。 複製はブロックレベルで行われるため、このメカニズムはファイルシステムが展開されている機器の種類を実際には気にしません。 レプリケーションは、同期または非同期のいずれかです（これまでのところ、サーバー間シナリオでのみ）。 ネットワーク通信メカニズムとして、TCP / IPまたはRDMAを使用できます。 BitLockerベースのデータ重複排除および暗号化メカニズムは、レプリカの上に適用できます。 この素晴らしい災害とフォールトトレラントの技術の奇跡は、これまでPowerShellのみで構成されています。 プロセスを実装するには、TCP 445またはTCP 5445ポートを開いて同じドメインのメンバーにする必要もあります（この観点からの影響の対象はホストであり、ホスト間のボリュームが複製され、同じホスト内でのボリューム複製シナリオが可能です）。 また、複製はデータボリュームに対してのみ可能であり、システムボリュームに対しては可能ではないことを覚えておくことが重要です。つまり、「ディスクC：」を複製することは技術的に不可能であり、この技術はこの目的のために開発されたものではありません。 このテクノロジーは、データ損失ゼロ（同期レプリケーションの場合）またはゼロに近いデータ損失（非同期シナリオ）を提供するように設計されています。



また、チャネルには特定の要件があることにも注意してください。各ファイルサーバーに少なくとも1つの10 Gb / s接続があります。 信頼性のために、サイズが1472バイトの断片化されていないICMPパケットの送信が、5分間の間隔で損失なしに成功することを確認することをお勧めします。







また、USBスティックのようなリムーバブルメディアを複製として、ターゲットとしてもソースとしても使用できないことに注意してください。 また、組み込みの管理者グループのレベルで権限が必要になります。

テストの目的で、VMのすべてのメカニズムを試すことができます。このシナリオの仮想ディスクは固定タイプであり、動的に拡張するものではないことに注意してください。



サーバー間レプリケーションを構成する



さて、ボリュームレプリケーションの構成プロセスに関する入門書をよく理解した後、サーバー間レプリケーションを構成しましょう。 サーバーがプレビュー中であることを思い出してください。このメカニズムは戦闘データには使用しないでください。各参加サーバーには、 「ファイルサーバー」ロールと「ストレージレプリカ」または「Windowsボリュームレプリケーション」機能をインストールする必要があります（ビルドに応じてサーバー。



その後、参加サーバーの1つで管理者権限でPowerShellコマンドレットを実行します。



New-SRPartnership -SourceComputerName sr-srv05 -SourceRGName rg01 -SourceVolumeName d: -SourceLogVolumeName e: -DestinationComputerName sr-srv06 -DestinationRGName rg02 -DestinationVolumeName d: -DestinationLogVolumeName e: -LogSizeInBytes 8gb





私の例では、sr-srv05とsr-srv06の2つのサーバーが関係しています。ログファイル用に別のボリュームが割り当てられています（レプリケーションボリューム、ソースボリューム、ターゲットボリュームよりも1ギガバイト小さい必要があります）。

次のコマンドレットを実行して、参加している両方のサーバーでレプリケーション構成が正常に実行されたことを確認します。



Get-WinEvent -LogName *WVR/admin -max 20 | fl







イベント2200、5005、5015、5001、および5009の存在は、成功の兆候です。
