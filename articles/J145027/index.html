SaaSアプリケーション用のマルチテナントアーキテクチャ

マルチテナンシーとは何ですか？



簡単に言えば、マルチテナンシーとは、単一のサービス（1つのインストールまたは展開）内で異なる組織（つまり、独立したSaaSサブスクライバー）のユーザーに個別にサービスを提供する機能です。 ここでの主なことは、サブスクライバーが互いに分離していることを観察することです。 実際、SaaSアプリケーションに保存したデータを検索するときに他の顧客が利用できる場合、顧客は満足しません。 これは明確な分離違反です。



マルチテナンシーは異なります。たとえば、次のスキームで表すことができます。







最近では、マルチテナンシーはクラウドコンピューティングの用語と一緒に表示されることがよくありますが、実際には、たとえばホスティングサイトなど、何らかの形でのマルチテナンシーが以前に関連していました。 たとえば、SQL ServerとIISを使用すると、データベースとサイトを個別に管理できます。リモートサーバーのIIS管理コンソールにアクセスすると、SQL Serverの場合と同様に、サイトのみが表示されます。管理コンソールでは、データベースのみが表示および管理されます。 また、マルチテナンシーのバリエーションと呼ばれることもあります。すべてのホスティングクライアントは同じWebサーバーまたはデータベースサーバー上に存在しますが、それらのノード/要素/データでのみ動作します。 マルチテナントは、多くのERPおよびCRMシステムの不可欠な部分でもあります（ マルチテナントとシングルテナントのERP-比較 ）。



SaaSアプリケーションの場合、すでにマルチテナンシーを処理する必要がありますが、既存の機能がこれを支援したり、たとえばPaaSプラットフォームを提供するコンポーネントの準備ができたりします。 一般的に、マルチテナンシーに重点を置いてアプリケーションをすぐに設計する方が、後でアプリケーションを「再設計」してサポートするよりも簡単です。



マルチテナントアプローチの利点は何ですか？



マルチテナントアーキテクチャに加えて、各ユーザーに独自のインフラストラクチャ（これは非SaaSアプリケーションで一般的です）が提供される場合、個別のアーキテクチャ（シングルテナント）があります：論理サーバーまたは物理サーバー。 クラウドサービスの場合、このアプローチは正当化されない可能性があります。 必要な数に関係なく、各サブスクライバーに固定リソースが割り当てられます（これにより、ユーザーのSaaSサブスクリプションの最終コストが増加し、その結果、ソリューションのコストが競合他社のコストよりも高くなることがあります）。 このようなクラウドサービス専用のアーキテクチャは、プレミアムサブスクリプションなどにのみ正当化できます。



マルチテナントアーキテクチャは、SaaSソリューションのコンピューティングリソースとストレージのコストを削減する基本的な方法です。必要なリソースの最小数（共有インフラストラクチャ）とその最大負荷によります。 最も具体的なメリットは、マルチテナンシーと「無制限のスケーリング」の組み合わせです（追加のリソースを提供したり、新しいサーバーを追加するために追加の資金を費やす必要がない場合）。 自分で無制限のスケーリングを実装するか、クラウドベースのPaaSプラットフォームを使用することができます。







出典： クラウドコンピューティング ： クラウドにコミットする前にSAASスマートを強化する



それでも、専用モデルまたはマルチテナントを選択するにはどうすればよいですか？



マルチテナンシーは、たとえばデータレベル、リソースレベルなどで異なる場合があります。 そして、これらすべてを組み合わせたり、すべて一緒に使用したり、まったく使用したりすることはできません。



出典： Windows Azureのマルチテナントデータ戦略-パート1



したがって、1つまたは別のアプローチを選択するという問題に対する正解または不正解はありませんが、選択に影響する可能性のある多くの要因があります。 これらの要素については、Widows Azureプラットフォームの例で説明しますが、それらのほとんどは他のクラウドプラットフォームに適用できます。



建築







1.マルチテナントアプリケーションでは、常に専用モード（1人のサブスクライバー）をエミュレートできますが、通常、選択したアプリケーションを複数の独立したサブスクリプションに拡張する方法はありません。



2. 耐障害性 。 マルチテナントアプリケーションは、専用アプリケーションよりもインスタンス障害に対して脆弱です。 専用展開の障害は、このアプリケーションを使用するクライアントにのみ影響しますが、マルチテナントアプリケーションの障害はすべてのクライアントに影響します。



マルチテナンシーアプリケーションのいくつかのインスタンスまたはその一部を使用して、障害のマイナスの結果を減らすことができます。 たとえば、Windows Azureは、ロールの複数のインスタンスを使用して、アプリケーションの複数の同一コピーを展開できるようにすることで、このリスクを軽減できます（このアプローチは、複数のインスタンスを持つマルチテナントモデルを完全に実装します）。 Windows Azureは、これらのロールインスタンス間で要求処理の負荷を自動的に分散するため、アプリケーションは、複数のインスタンスを使用する場合に正しく機能するように設計する必要があります。 たとえば、アプリケーションがセッション状態を使用する場合、Webロールの各インスタンスが状態にアクセスできることを確認する必要があります。 また、Windows Azureはすべてのロールインスタンスを常に監視し、障害が発生するとインスタンスを自動的に再起動します。



一部のアプリケーションでは、すべてのクライアントが同じマルチテナント展開を共有することが適切でない場合があります。 たとえば、使用するアプリケーションの機能に基づいてクライアントをグループ化し、作成されたグループごとに各インスタンスを最適化する必要がある場合があります。 この場合、異なるWindows Azureサブスクリプションであっても、マルチテナントアプリケーションの2つ以上のコピーが必要になる場合があります。



3. サービスレベル契約（SLA） 。 さまざまなレベルのサービスサブスクリプションに対して、さまざまなサービスレベル契約（SLA）を提供できます。 異なるSLAを持つサブスクライバーが同じマルチテナント展開を共有する場合、最高のSLAを達成する必要があります。



4. 法的および規制環境 。 一部のアプリケーションでは、特定の規制または法的問題を考慮する必要があります。 これには、機能の違い、ユーザーインターフェイスでの法的契約の表示、特定のサブスクライバーごとの個別のデータベースまたはストレージエリアの提供、特定の地理的地域でのサービスおよびストレージの場所が必要になる場合があります。 このような状況では、さまざまなユーザーグループに個別のマルチテナント展開を実装するか、サービスのサブスクリプションごとに厳密に専用モードを実装する必要があります。



5. 認証と承認 。 クラウド内のアプリケーションは、独自の認証および承認システムを使用できます。この場合、アプリケーションのサブスクライバーは、アプリケーションのすべてのユーザーの資格情報を作成する必要があります。 一方、顧客は既存の認証システムを使用し、アプリケーションの新しい資格情報を作成する必要を避けることを好む場合があります。 マルチテナントアプリケーションの2番目のオプションは、複数の認証プロバイダーをサポートする必要があることを意味し、クラウド内のアプリケーション認証スキームをローカル認証インフラストラクチャのスキームにマッピングする必要がある場合もあります。 Windows Azureでは、アクセス制御サービスを使用して認証と承認を構成します。これにより、フォーム、Active Directory、LiveId、Google、Facebook、およびその他のOpenIDプロバイダーによる認証を実装できます。 Active Directoryとの統合と、ADサブスクライバーとサービス間のフェデレーションのセットアップについて話している場合は、おそらく専用の展開が必要になる可能性があります。



アプリケーションライフサイクル管理







1. コードベースの維持 。 開発会社の場合、さまざまなクライアントに対して個別のコードベースを維持すると、サポートとメンテナンスのコストが増加し、どのクライアントがどのバージョンを使用しているかを追跡する際の問題が発生します。 これによりエラーが発生し、追加コストが発生する可能性があります。 1つの論理展開を備えたマルチテナントシステムは、アプリケーション全体に対して1つのコードベースを保証します。 複数の展開がある専用モデルを使用して同じコードベースを維持することはできますが、（長期的な結果を伴う）異なるクライアント用にコードをフォークしたい場合があります。 アプリケーションを微調整する必要があるいくつかのシナリオでは、いくつかのコードベースが受け入れられるオプションかもしれませんが、このパスを選択する前に、標準のシステム設定のレベルでカスタム構成またはカスタムビジネスルールを適用する可能性を検討する必要があります（たとえば、インターフェイスを介して） 。 複数のコードベースが必要な場合は、ユーザーコードがユーザーコンポーネントの必要最小限の数だけに制限されるように、アプリケーションを構成する必要があります。



2. アプリケーションの更新 。 マルチテナントアプリケーションを使用すると、更新をすべての顧客に同時に配布できます。 このアプローチは、1つの論理インスタンスのみが更新されることを意味し、メンテナンスの労力を削減します。 さらに、すべての顧客が最新バージョンのアプリケーションを使用しているという自信があり、サポートが簡素化されます。



たとえば、Windows Azure更新ドメインを使用すると、アプリケーションを停止せずに更新を複数のロールインスタンスに配布できるため、このプロセスが簡素化されます。 クライアントに特定のバージョンのアプリケーションに関連付けられた操作手順またはソフトウェアがある場合、更新はクライアントと調整する必要があります。



アプリケーションの更新に関連するリスクを減らすために、逐次更新手順を実装できます。最初のフェーズでは、ユーザーの小さなグループに対してアプリケーションが更新され、2番目のフェーズでは、新しいバージョンに問題がなければ、更新は残りのユーザーに適用されます。



3. アプリケーションの監視 。 単一のアプリケーション展開の監視は、複数の展開の監視よりも簡単です。 新しいロールインスタンスに複数の展開がある専用モデルでは、監視環境設定を設定する必要があり、その結果、プロセス全体の複雑さが増します。 順次更新を使用することにした場合、アプリケーションの2つのバージョンを同時に監視し、監視データを使用してアプリケーションのバージョンの安定性を評価する必要があるため、監視も難しくなります。



4. サードパーティコンポーネントの使用 。 マルチテナントアーキテクチャを使用する場合は、このモードでサードパーティのコンポーネントがどの程度機能するかを評価する必要があります。 サードパーティのコンポーネントがマルチテナントアーキテクチャで動作することを確認するには、いくつかの追加手順を実行する必要があります。



5. テストアクセスと新規顧客の提供 。 このプロセスに構成の変更のみが含まれる場合、新しいクライアントに領域を提供したり、サービスの無料試用の初期化を大幅に簡素化します。 専用モデルの場合は、無料試用版を使用しているクライアントを含め、各クライアントにアプリケーションの新しいインスタンスをデプロイする必要があります。 このプロセスは自動化できますが、マルチテナントアプリケーションで構成データを変更または作成するよりもはるかに複雑になります。



Windows Azureでは、RESTサービス管理APIを使用する必要があるため、プロビジョニングプロセスを自動化（サービス、証明書、ストレージなどのプロビジョニング）することができます。 必要なすべてのアクションのソフトウェア実装は、非常に重要です。 ユーザーがポータルに入ることを許可し、「試用」ボタンをクリックして、数分後に特別に作成された一時的なサブスクリプション（試用版）を取得します。 これは、ユーザーが最初にアプリケーションを試すことができ、次にアプリケーションを購入して使用を継続できるときに、アプリケーションをMarketplaceに配置する場合にも当てはまります。サポートサービスへの電話や手紙はありません。



アプリケーションのセットアップ







1. アプリケーションにアクセスするためのURL 。 既定では、Windows Azureは各アプリケーションに<application name> .cloudapp.netという形式のDNS名を提供します。 専用展開の場合、CNAMEレコードを使用して、サブスクライバーのDNS名（たとえば、 contoso.com ）をアプリケーションにマッピングできます。 マルチテナントアプリケーションでは、http：//.cloudapp.net//の形式のアドレス指定スキームを使用できます。また、CNAMEとホストヘッダーを使用します。 たとえば、 mydomain.ruをhttp：// <multitenat-application-name> .cloudapp.netにマッピングできます。ここで、ホストヘッダーを使用して、要求されたテナントを決定します。 このアプローチの詳細については、「 マルチテナントAzureアプリケーションでのテナントの特定-パート2」を参照してください。 このアプローチにより、これがマルチテナントアプリケーションであるという事実をユーザーから隠すことができます。 申し立ては、テナント用に選択された一意のURLに送信されます。 HTTPS経由でアクセスする必要がある状況は、もう少し複雑になります。詳しくは、記事「 マルチテナントAzureアプリケーションでのテナントの特定-パート3」を参照してください 。



2. クライアントによってアプリケーションを構成します 。 クライアントが独自のコードをダウンロードできるようにすると、アプリケーションが実行する制御レベルが低下するため、アプリケーションがクラッシュするリスクが高まります。 そのため、多くのSaaSプロバイダーは、マルチテナントアプリケーションでのユーザーコードの使用に制限を課しています（障害はすべてのユーザーに影響するため）。 ほとんどの場合、これは単に禁止され、コードが許可されることもありますが、その機能は大幅に低下します。 クライアントがコードまたはスクリプトをダウンロードできるようにすると、アプリケーションのセキュリティに関連するリスクも高まります。



データ層アーキテクチャ







1. 他のテナント（サブスクライバー）からのデータ保護 。 マルチテナントモデルでは、偶然または悪意のある開示のリスクが高くなります。 アプリケーションが他のクライアントと物理的に共有されていることをクライアントが知っている場合、クライアントに自分の個人データが安全であることを納得させることは困難です。 ただし、各クライアントのデータを論理的に分離する高品質のアプリケーションアーキテクチャは、必要なレベルの保護を提供する必要があります。 このようなアーキテクチャでは、次のアプローチを使用できます（詳細なアプローチについては、habrosttie Multi- tenant Data Architectureで説明されています）。

  • 各クライアントのテーブルが個別のスキーマにあるデータベーススキーマ。
  • データベースレベルでアクセス制御メカニズムを使用できるデータベースセキュリティツール。
  • クライアントデータを分離するためのパーティション分割。
  • これらのアプローチの組み合わせ。


上記のアプローチは、Windows AzureストレージとSQL Azureの両方に適用されます。 ただし、それらは異なる価格設定モデルに関連付けられています。 Windows Azureストレージの場合、保存されたデータの量とストレージへのトランザクションの数に応じて請求書が請求されるため、コストは個々のアカウントまたはストレージコンテナーの数に依存しません。 SQL Azureの使用に対する請求は、使用可能なデータベースの数に依存するため、コストの観点から、サブスクライバー（テナント）の最大数に対して1つのデータベースを使用することが最適です。



Windows AzureのテーブルとBLOBを使用すると、個々のアカウント、テーブル、パーティション、コンテナのレベルで異なるレベルのデータ分離を実装できます。 SQL Azureは、データベース、データベースフェデレーション、またはサーバーを個別に分離して実装します。 たとえば、BLOBストレージにデータを保存するモバイルアプリケーションを作成できます。 この場合、ユーザーごとに個別のBLOBコンテナーを作成し、このコンテナーに適切な権限を割り当てることができます（つまり、パブリックアクセスではありません）。



2. データアーキテクチャの拡張性 。 クライアントがデータモデルを拡張して独自のカスタム属性を含めることができるように、データウェアハウスを構築する方法はいくつかあります。 いくつかのアプローチがあります。

  • 各クライアントが個別のスキームを持つアプローチ。
  • 事前定義されたユーザー列のセットが提供されるアプローチ。
  • クライアントが任意の数のユーザーフィールドをテーブルに追加できるようにする、より柔軟なスキームを使用したアプローチ。


SQL Azureを使用する場合の主な問題は、固定データスキーム内で作業する必要があることです。 Windows Azureテーブルストレージを使用する場合、逆に複雑さは固定されていないストレージスキームに起因します。 Windows Azure Tabular Storageを使用すると、同じテーブル内のレコードをまったく異なる構造にすることができます。これにより、コードがより複雑になるため、柔軟性が向上します。



3. データアーキテクチャのスケーラビリティ 。 水平分割により、データウェアハウスを拡張できます。 SQL Azureのスケーラビリティを実現するために、個々の顧客データをAzure SQLの新しいインスタンスに移行できます。
