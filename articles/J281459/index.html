こんにちはボット！ チャットボット-次世代のアプリ？

最近、 // BUILDカンファレンスが開催されました。このカンファレンスでは、マイクロソフトは従来、開発者向けの新しいテクノロジを紹介しています。 初日の基調講演では 、1つの非常に重要なアイデアが表明されました。コンピューターとのコミュニケーションがますます自然になるにつれて、自然言語における対話コミュニケーションの役割が増大します。 これは、Cortana音声アシスタントにアラームの設定を依頼できるという事実だけでなく、他の多くのタスク（ピザの注文、ホテルの予約、チケットの購入など）を対話で解決できることも事実です。 さらに、これはユーザーとコンピューター間の対話だけでなく、より複雑なケースでは、人がCortanaに旅行の計画を依頼し、Cortanaが他のボットと通信してホテルやチケットを注文することができます。







ここで興味深いのは、「通信」自体がさまざまな場所で発生する可能性があることです。Cortanaクライアントを使用するデスクトップ上だけでなく、チャットに従来使用される他のチャネル（Skype、Telegram、Slackなど）でも発生します。 パーソナルアシスタントの性格と記憶は通信チャネルに依存しませんが、クラウドのどこかにあり、いわゆる「会話キャンバス」のいずれかでチャットする準備ができています。



ボットを作成するために、 Microsoft Bot Frameworkの予備バージョンと対応するAPIが提示されました。これについては、以下で説明します。 また、英語でのユーザーとの会話をサポートする独自のシンプルなボットを作成する方法も示します。



次の主要な要素は、ボットを操作するために使用されます。

  • ボットビルダーSDK （C＃およびNode.jsで利用可能）は、主要なボット機能を作成するように設計されています。 これはWebAPIに基づいており、外界とのボットの通信プロトコルを定義します。 SDKの一部として、ボットのデバッグを可能にするエミュレーターと、状態のある長いダイアログなど、いくつかの主要な抽象化の実装を簡素化するクラスのセットがあります。
  • Cognitive Services （以前はProject Oxfordの一部であるLUISとして知られていました ）は、自然言語分析を簡素化し、テキスト文から意味を抽出します。 Webインターフェースを使用して、基本的な構文構成を決定し、対応するユーザーインテント（意図）を自動的に割り当てて、ボットが応答します。
  • ボットコネクタを使用すると、ボットをSkype、Slack、Telegramなどの1つまたは複数の通信チャネルにバインドできます。これを行うには、サイトbotframework.comでボット接続を設定します。
  • 時間が経つにつれて、さまざまなボットへのリンクを公開できるボットのカタログ 。




例：マーフィーボットを知る



ボットの好例は、 http：//www.projectmurphy.net/プロジェクトの一部として作成されたマーフィーです。 サイトを訪れたら、SkypeにMurphyをインストールしてチャットを開始できます（ボットをサポートする最新バージョンのSkypeを使用していることを確認してください）。 このボットには、「Cindy Crawfordがスーパーマンだったらどうする？」（下のダイアログ例を参照）、または「私がプログラマーだったらどうする」（この場合、写真をアップロードするように求められます）などの質問をすることができます。







こんにちはボット！ 私たちは自分に対談者を書きます



例として、Bot Frameworkに基づいた単純なボットの作成方法を見てみましょう。 ボットの心臓と脳は、通常はクラウドに配置されているWebサービスであることを思い出してください。







このようなサービスを作成する最も簡単な方法は、Visual Studio用のボットフレームワークテンプレートをダウンロードすることです（WebツールがインストールされたVisual Studio 2015 Update 1があることを確認してください）。 結果のZIPファイルをVisual Studioテンプレートのあるディレクトリに配置します。通常は「％USERPROFILE％\ Documents \ Visual Studio 2015 \ Templates \ ProjectTemplates \ Visual C＃」です。その後、Visual StudioでBot Applicationタイプの新しいプロジェクトを作成できます。







作成されたプロジェクトでは、 Controllers / MessagesController.csファイルのPostメソッドがボットの主な機能を担当します。 最も単純なHelloボットの場合、このメソッドを次のように実装します。 



public async Task<Message> Post([FromBody]Message message) { if (message.Type == "Message") { var reply = message.Text.ToLower() == "hello" ? "Hello!" : "I do not understand you!"; return message.CreateReplyMessage(reply); } else { return HandleSystemMessage(message); } }




ボットをテストするには、ローカルWebサーバー（F5）でプロジェクトを実行し、開いているブラウザーのアドレスバーからアドレスとポートをコピーして、 Microsoft Bot Framework Emulatorで開きます   （最初にダウンロードしてインストールする必要があります）。 パス/ API /メッセージをサイトのアドレスバーに追加する必要があることに注意してください。







ボットにもう少し便利なことをさせるために、有名なElizaプログラムのC＃ポートを使用しました。 作成されたチャットボットは、GitHubリポジトリからダウンロードできます 。 最初のバージョンでは、ボットの構築スキームは上記のプログラムとそれほど変わりません。答えを生成するためにElizaライブラリメソッドのみが呼び出されます。



このような単純な実装は、Postメソッドがすべてのユーザーからのリクエストを処理するために使用され、それらの間の状態分離がないため、ユーザーとの長いダイアログをサポートするのにはあまり適していません。 これを行うには、セッション識別子（ message.ConversationID ）を明示的に確認するか、BotUserData / BotConversationDataオブジェクトを使用するか、ダイアログをサポートするより高度なAPIを使用します（これらについては今後の記事で説明します）。 それまでの間、ボットはステートレスであると考えて、これらの詳細に注意を払いません。



クラウドでボットを開始するには、まず、結果のWeb APIをAzureで公開する必要があります。 次に、 http：//dev.botframework.comにログインして、新しいボットを登録する必要があります。 ボットに固有のAppIDを考え出し、App Secretを自動的に生成することが重要です。



この後、重要な手順を実行する必要があります-アプリケーションのWeb.Configファイルで、アプリIDとアプリキーを追加する必要があります。 



 <configuration> <appSettings> <add key="AppId" value="YourAppId" /> <add key="AppSecret" value="YourAppSecret" /> </appSettings>




その後、Web Configが更新されるように、ボットをクラウドに再デプロイする必要があります。







これを行ったら、ボットフレームワークサイトのプロパティパネルでボットのパフォーマンスをテストするか、既知のエミュレーターを使用できます。ただし、この場合、正しいAppIDとApp Secretを入力する必要があります。







ボットはエミュレーターで動作しているので、さまざまな通信チャネルにバインドできます。 これを行うには、Bot Frameworkサイトのボットコントロールパネルでこれらのチャネルを追加して構成します。 これを行うには、希望する通信チャネルを選択し、指示に従います。







たとえば、Telegramの場合、特別なBot Fatherボットとの会話を使用してTelegramで新しいボットを作成し、Access Tokemを取得してBot Frameworkサイトに提供する必要があります。 その後、ボットコネクタがTelegram APIとボットの間のすべての中間操作を引き継ぎ、Telegramからボットと通信できるようになります。 ところで、私のボットは電報@ElllizaBotで呼び出されます。あなたは彼と話すことができます！



BUILDのこの英語のレポートでは、ボットフレームワークを使用してボットを最初から作成する方法について詳しく読むことができます。



チャンネルの抽象化



Bot Framework APIの主な利点は、1つの通信チャネルに焦点を当てることなくボットを作成できるようになったことです。 実際、Bot Frameworkは会話および対話メカニズムを実装するための適切なレベルの抽象化を提供し、Bot Connectorはさまざまなコミュニケーションツールとの特定の接続を引き受けます。 この分離により、プログラマはボットを1回作成し、どこからでもボットと通信できます。 Javaの世界で知られているスローガンを言い換えると、この原則を次のように定式化できます： Write Once-Chat Everywhere 。
