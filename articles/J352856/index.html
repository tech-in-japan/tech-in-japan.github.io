AWSサーバーレスアプリケーションフレームワーク

AWSでサーバーレスWebアプリケーション用の小さなフレームワークを作成することにしました。 これをフレームワークではなくワークと呼ぶ方が正しいかもしれません-私は知りません。 ただし、ポイントは、AWSでサーバーレスアプリケーションを迅速に開発するための基盤を作成することです。 コードはGitHubに投稿されており、多くの改良点があります。









この記事では、サーバーレスアプリケーションをローカルで開発およびテストする方法、フロントエンドおよびバックエンドルーティング、Amazonサービスなどに焦点を当てます。 誰も気にしない、猫へようこそ！



序文のようなもの



最近まで、サーバーレスアプリケーションの開発は、ラムダ関数とAPIの完全なローカルテストの手段がなかったため、非常に複雑でした。 アプリケーションを作成するときは、ブラウザでコードを編集して常にオンラインで作業するか、ラムダ関数のソースコードを常にクラウドにアーカイブしてアップロードする必要がありました。



2017年の夏に突破口がありました。 AWSは、 サーバーレスアプリケーションモデル （SAM）と呼ばれる新しい単純化されたCloudFormationテンプレート標準を作成し、同時にsam-localプロジェクトを開始しました。 まず最初に。



Amazon CloudFormationは、JSONまたはYAMLテンプレートファイルを使用して、アプリケーションに必要なすべてのAWSインフラストラクチャを記述することができるサービスです。 これは非常に便利なことです。 それがないと、Webコンソールまたはコマンドインターフェイスを介して手動で必要な多くのリソースを手動で作成する必要があります：ラムダ関数、データベース、API、ロール、およびポリシー...



CloudFormationを使用すると、特別なデザイナーでインフラストラクチャを描画するか、テンプレートに手でインフラストラクチャを書き込むことができます。 いずれにせよ、結果はテンプレートファイルになります。テンプレートファイルを使用すると、数回のクリックで、または1つのコマンドで、アプリケーションに必要なすべてのものを引き続き作成できます。 そして、必要に応じて、このテンプレートを変更し、1つのコマンドで再度適用します。 これにより、アプリケーションインフラストラクチャのサポートがはるかに簡単になります。 インフラストラクチャはコードのようです。



CloudFormationは美しく、そのテンプレートを使用すると 、AWSリソースのほぼ100％ を記述することができます 。 しかし、汎用性があるため、これはかなり「冗長な」形式です。テンプレートはすぐに適切なサイズに成長できます。 これを認識し、サーバーレスアプリケーションの開発を容易にすることを目指して、AWSは新しいSAM形式を作成しました。



条件付きで、通常のCloudFormationテンプレートは低レベル言語で記述されていると想定できます。 また、SAMテンプレートは高水準言語であるため、簡素化された構文を使用してサーバーレスアプリケーションのインフラストラクチャを記述することができます。 SAMテンプレートは、デプロイ時にCloudFrontによって通常のテンプレートに変換されます。



sam-localとは何ですか？ これは、SAMテンプレートで記述されたサーバーレスアプリケーションをローカルで操作できるコマンドラインツールです。 Sam-localを使用すると、ラムダ関数をテストしたり、さまざまなAWSサービスからイベントを生成したり、ゲートウェイAPIを実行したり、SAMテンプレートをチェックしたりできます。



Sam-localは、Dockerコンテナーを使用して、GatewayおよびLambda APIをエミュレートします。 動作原理は次のとおりです。 起動すると、sam-localはプロジェクトフォルダーでSAMテンプレートファイルを探します。 テンプレートファイルを分析し、Dockerコンテナ内のテンプレートに割り当てられたリソースを起動します。APIを開き、ラムダ関数をそれらに接続します。 さらに、サポートは実際のラムダ関数の動作に非常に近いです（制限、使用されるメモリの量、および実行時間を示しています）。

一般に、これらすべてにより、ローカルコンピューターでサーバーレスアプリケーションを作成することができます。これは、従来のWebアプリケーションを作成するよりも複雑ではありません。

フレームワークについて



すでに書いたように、新しいサーバーレスアプリケーションを作成するときにクイックスタートを提供するために必要です。 現時点では、Webトークンの承認のみをサポートしています。 さらに、エラー処理の追加、フォームの操作、表形式データの出力、展開メカニズムの構成を計画しています。 一般に、将来的にAB-ERPリポジトリのクローンを作成し、アプリケーションの作業をすぐに開始できるようにします。



ERPシステムを作成するため、他の製品の名前AB-TASKSおよびAB-DOCとの類推によりAB-ERPと呼びました。 同時に、ERPシステムの作成にはAB-ERPは必要ありません;それに基づいて、サーバーレスWebアプリケーションを作成できます。



アプリケーションには、フロントエンドコードとバックエンドコードがあります。 したがって、プロジェクトルートには2つのフォルダーがあります： lambda （バックエンド）とpublic （フロントエンド）： 



 +---lambda | +---api | +---core \---public +---css | \---core +---img +---js | \---core \---views


AB-ERPは、1ページのWebアプリケーション（SPA）の原則に基づいて動作します。 アプリケーションをデプロイする場合、フロントエンドコードをAWS S3に配置し、その前にCloudFrontを構成する必要があります。 これは、 以前の AB-DOC に関する記事の「開発と展開」セクションで説明されていました。



デプロイ中のバックエンドコードはAWS Lambdaにアップロードされます。



AB-ERPはMariaDBをデータベースとして使用します。 MariaDBはAWS RDSにデプロイされます。 必要に応じて、たとえばAWS DynamoDBと連携するようにAB-ERPを再構成できます。



ユーザーファイルはAWS S3に保存されます。



これは、アプリケーションアーキテクチャの外観です。







バックエンド



現時点では、すべてが非常にシンプルです。 1つのゲートウェイAPIリソースと1つのラムダ関数のみ。



SAMテンプレートは次のようになります。 



 AWSTemplateFormatVersion : '2010-09-09' Transform: AWS::Serverless-2016-10-31 Description: An example RESTful service Resources: ABLambdaRouter: Type: AWS::Serverless::Function Properties: Runtime: nodejs6.10 Handler: index.handler Events: ABAPI: Type: Api Properties: Path: /{proxy+} Method: any


SAMテンプレートには、ラムダ関数であるABLambdaRouterのリソースの1つがあります。 ABLambdaRouterは、APIからの1つのABAPIイベントによってのみ呼び出されます。



API Gatewayリソースは、任意のメソッド（ ANY ）によるURL / {proxy +}のパスへのリクエストを受け入れます。 つまり、言い換えれば、通常の双方向プロキシとして機能します。 Lambda関数はそれぞれ、リクエストに応じて異なるコードを実行するルーターの役割を引き受ける必要があります。



Lambda関数コード（ルーター） 
 'use strict'; const jwt = require('jsonwebtoken'); //process.env.PROD and other env.vars are set in production only if(process.env.PROD === undefined){ process.env.PROD = 0; process.env.SECRET = 'SOME_SECRET_CODE_672967256'; process.env.DB_HOST = '192.168.1.5'; process.env.DB_NAME = 'ab-erp'; process.env.DB_USER = 'ab-erp'; process.env.DB_PASSWORD = 'ab-erp'; } //core modules const HTTP = require('core/http'); const DB = require('core/db'); //main handler exports.handler = (event, context, callback) => { context.callbackWaitsForEmptyEventLoop = false; let api; const [resource, action] = event.pathParameters['proxy'].split('/'); //OPTIONS requests are proccessed by API GateWay using mock //sam-local can't do it, so for local development we need this if(event.httpMethod === 'OPTIONS'){ return callback(null, HTTP.response()); } //require resource module try { api = require('api/' + resource)(HTTP, DB); } catch(e) { if (e.code === 'MODULE_NOT_FOUND') { return callback(null, HTTP.response(404, {error: 'Resource not found.'})); } return callback(null, HTTP.response(500)); } //call resource action if(api.hasOwnProperty(action)) { if(api[action].protected === 0){ api[action](event, context, callback); } else if (event.headers['X-Access-Token'] !== undefined) { let token = event.headers['X-Access-Token']; try { event.userData = jwt.verify(token, process.env.SECRET); api[action](event, context, callback); } catch(error) { return callback(null, HTTP.response(403, {error: 'Failed to verify token.'})); } } else { return callback(null, HTTP.response(403, {error: 'No token provided.'})); } } else { return callback(null, HTTP.response(404, {error: 'Action not found.'})); } }




APIには2つのレベルの階層があります。最初のレベルはモジュールで、2番目のレベルはアクションです。 URLはapi.app.com/module/actionです。 ルーター関数は、着信要求のpathParametersを分析し、目的のモジュールをlambda / apiフォルダーから接続してから、要求をこのモジュール内の目的の関数に転送しようとします。



デフォルトでは、モジュール内の関数は承認を必要とするため、モジュールから関数を呼び出す前に、ルーターはX-Access-Tokenリクエストヘッダー内の有効なトークンをチェックします。 トークンが有効な場合、モジュールから関数が呼び出されます;有効でない場合、エラー403が返されます。



多数の個別のGateway APIリソースと多数のラムダ関数を作成する代わりに、なぜこのアプローチを選択したのですか？ まず、そして最も重要なことは、構成、展開、およびそのようなアーキテクチャの実際の操作の容易さです。 第二に、このアプローチは関数のコールドスタートを最小限に抑えます。 実際、関数に長い呼び出しがない場合、AWSはそのコンテナーを削除し、新しい呼び出しはリクエストの処理に時間がかかります。



このアプローチには欠点もあります。 さまざまなAPIリソースに対して、API Gatewayレベルで特別な設定を行うことはできません。



誰かが質問があるかもしれませんが、なぜゲートウェイAPIが必要なのですか、ブラウザからラムダに直接アクセスしないのはなぜですか？ Gateway APIには多くの利点があります。 Edge Optimizedモードでは、CDNのように機能し、応答のキャッシュがあり、バックエンドへの呼び出しなしでOPTION要求に応答できます（MOCK統合）。これにより、アプリケーションが大幅に高速化されます。 また、DDOS保護と、制限を使用してトラフィックを規制する機能も備えています。 また、サードパーティの開発者向けにアプリケーションAPIを開くこともできます。



フロントエンド



フロントエンドでは、React、Vue.js、Angular.jsのような「大きな」フレームワークを使用しないことにしたため、SPAアプリケーション用の小さなルーターを作成しました。



ルーターは各ページの説明を保存します：どのhtmlテンプレートとどのcss、jsファイルが必要か。 ページが要求されると、ルーターは必要なすべてのファイルをプレーンテキストでロードし、それらを結合して、アプリケーションインターフェイスのdivコンテナーに挿入します。 コンテナに挿入されると、開かれたページのJavaScriptが実行されます。



ルーターコード 
 "use strict"; //ROUTER object const ROUTER = { pages: { "index": ["css/index.css", "views/index.html", "js/index.js"], "login": ["css/login.css", "views/login.html", "js/login.js"] }, open: function(page){ let self = this; $container.html(big_preloader_html); if(self.pages.hasOwnProperty(page)){ const parts = self.pages[page]; let getters = []; let wrappers = []; for (let i = 0; i < parts.length; i++) { if( /^.*\.css$/i.test(parts[i]) ){ wrappers.push('style'); } else if ( /^.*\.js$/i.test(parts[i]) ){ wrappers.push('script'); } else { wrappers.push(''); } getters.push( $.get(parts[i], null, null, 'text').promise() ); } Promise.all(getters).then(function(results) { let html = ''; for (let i = 0; i < results.length; i++) { if(wrappers[i] === ''){ html += results[i]; } else { html += `<${wrappers[i]}>${results[i]}</${wrappers[i]}>`; } } self.updatePath(page); $container.html(html); }); } else { //TODO console.log('404'); } }, updatePath: function(newPath){ if(newPath !== window.location.pathname) { history.pushState({}, null, newPath); } } }




環境設定



コンピューターでプロジェクトを開始するために必要なものはすべて、プロジェクトgithubの READMEで手順を説明しようとしました。 何かがうまくいかない場合は、コメントを書いてください-私たちは助けようとします。 したがって、READMEを補充します。



ローカルテストのために、Node.jsに小さなHTTPサーバーを作成しました。 



 const express = require('express'); const app = express(); app.use(express.static('public')); app.use(function(req, res, next) { req.url = 'app.html'; next(); }); app.use(express.static('public')); app.listen(80, () => console.log('Listening..'))


開始する前に、 node abserver.jsコマンドを使用して実行する必要があります。 要求が到着すると、 パブリックフォルダー内のファイルを検索し、見つかった場合はそれを提供します。 ファイルが見つからない場合は、メインアプリケーションファイルpublic \ app.htmlが提供されます。 SPAアプリケーションが機能するには、これで十分です。 本番環境では、Amazon CloudFrontは同じことを行います。

プロジェクトのさらなる開発のためのいくつかの計画。
