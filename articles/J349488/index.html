LED照明とスマートピクセルRGB（WS2812B）



小売店で500rのRGB LEDチップWS2812Bを搭載したLEDストリップは、1メートルかかりました。

はんだ付けは難しくありません。クロック信号なしで3本のワイヤのみが使用されます。 このテープはコントローラーなしでは機能しないことに注意してください。 電源には、USBコネクタを備えた5V 1A電源が必要です。 ところで、私が持っていたテープの制御信号には3.3Vで十分だったので、レベルコンバーターなしでARMコントローラーをGPIOに接続することが可能です。









Raspberry PI GPIOを使用してWS2812B LEDストリップを制御できますが、オーディオポートと同じPWM SoCジェネレーターが使用されるため、組み込みのオーディオ出力で音楽を再生する際に問題が発生します。 したがって、このオプションはインターネットラジオでは機能しないため、すぐに破棄されました。









STM8 MKを搭載したSP002E LEDコントローラーを使用すると、ロシアの小売店の270rでこのテープを管理する方が簡単です。 ピクセル値を送信するためのUSB接続を使用して、STM32ストックからこのテープ用のコントローラーを組み立てようとするのは興味深いでしょう。 githubでいくつかのライブラリプロジェクトを見つけました。 軽い音楽を作るというアイデアがあります-TODOリストに書き込みます。



インターネットラジオをインストールして構成する



リモート音楽再生のいくつかのオプションを検討できます。



  1. Bluetooth A2DP-これは、Bluetoothスピーカーに実装されているもので、音の圧縮が失われます。 Raspberry PIは単に「リモート」スピーカーに変わり、音楽は別のデバイスから「ストリーミング」する必要があるため、このオプションは適合しませんでした。
  2. ネットワーク経由のPulseAudio-アプローチはA2DPに似ていますが、 module-native-protocol-tcpとクライアントの構成も必要です。
  3. ミュージックプレーヤーデーモン（MPD）-インターネットラジオまたはローカルディスク/ NASからのプレイリストのオフライン再生に最適です。


スタンドアロンのインターネットラジオが必要なので、選択はMPDです！ MPDのトラックを変更し、MPD ControlなどのAndroidクライアントから、デスクトップアプリケーションまたはWebインターフェイスを使用して、プレイリストとボリュームを管理できます。



bcm2835およびGPIO機能およびPWMコントローラーとの競合について述べたように、これはかなりノイズが多く問題の多いサウンドデバイスです。 唯一の利点は、内蔵されていることと、Raspberryに匹敵するUSB​​オーディオを購入する必要がないことです。



Mixer_type、mixer_device、mixer_control、mixer_indexを設定する必要があります。 これが行われる前に、トラックを切り替えるときにデーモンがハングし、プロセスを再起動する必要がありました。



ルートとしてRaspbianにインストールが実行されました。







コマンドを実行します： 



apt-get install mpd


統合されたRaspberry PI 3オーディオを機能させるには、/ etc / mpd.conf構成にパラメーターを追加する必要があります。 



 audio_output { type "alsa" name "My ALSA Device" device "hw:0,0" # optional mixer_type "software" # optional mixer_device "default" # optional mixer_control "PCM" # optional mixer_index "0" # optional } bind_to_address "192.168.1.68" port "6600" auto_update "yes"


「192.168.1.68」は、ローカルネットワーク上のRaspberyアドレスです。

Camelアーキテクチャについて一言



コンポーネントはエンドポイントファクトリであり、コンポーネントにはプロトコル/スキームが関連付けられています。 CamelのほとんどすべてがUniversal Resource Identifierです。 例：http、hdfs、smtps、irc、webcam。 今、 mpd ！









コンポーネントは、データを変換するためのルートに結合されます。 ルートは、使用可能なドメイン固有言語（DSL）の1つで記述されます：java、scala、groovy、spring / blueprint xml



ルートは、たとえば



hawt.ioコンソール。








DSLの消費者は、ルートの先頭にある「from」構成体です。

DSLのプロデューサーは "to"コンストラクトです

Exchangeは、ルートによって作成および処理されるメッセージです。 メッセージの本文（本文）とヘッダー（ヘッダー）が含まれます。









mavenを使用して、Apacheのラクダコンポーネントをモックアップできます。 



 mvn archetype:generate \ -DarchetypeGroupId=org.apache.camel.archetypes \ -DarchetypeArtifactId=camel-archetype-component \ -DarchetypeVersion=2.20.2 \ -DgroupId=myGroupId \ -DartifactId=myArtifactId


Apache Camelコンテキストがプロトコルでコンポーネントを見つけることができるようにするには、src / main / resources / META-INF / services / org / apache / camel / component / mpdファイルでコンポーネントクラスを指定する必要があります

class = com.github.igorsuhorukov.mpd.MpdClientComponent



MpdClientComponentクラスはファクトリーです-com.github.igorsuhorukov.mpd.MpdEndpointクラスのインスタンスを作成および初期化します。 ルートからの要求に応じてMpdEndpointはConsumerのインスタンスを作成し、そのメッセージはコンポーネントによってRouteに送信されます。 消費者はルートの値を返します。



  • volume-現在のボリューム値。範囲は[0,100]です。
  • currentは、現在の曲に関する情報を返します。
  • プレイリストは、リストとしてプレイリストを返します。
  • total-プレイリストからの音楽の合計期間。
  • 経過-時間内にどれだけ聴取されるか。
  • エラーメッセージ、最後のエラーが発生しました。


プロデューサー-CamelのコンテキストでRouteを終了し、MPDコマンドの発行を許可します。



  • play-再生を開始します。
  • stop-再生を停止します。
  • next-次の曲を選択します。
  • previous-前の曲を開始します。
  • ランダム化-次の曲への遷移をランダムな順序でアクティブにします。
  • unrandomize-プレイリストの順次再生を有効にします。
  • volume-再生ボリュームを0〜100の範囲で設定します
  • 歌-プレイリストに曲のリストを追加できます。 そして、これはファイルシステムのパスか、httpプロトコルを使用したインターネットラジオです。 この関数のメッセージ本文は、文字列またはorg.bff.javampd.song.MPDSongオブジェクトのいずれかです。 別のオプションは、文字列のリストまたはMPDSongオブジェクトのリストです。


camel-url-handlerプロジェクトのjava.net.URLとの統合により、このコンポーネントを操作することが可能になります。
