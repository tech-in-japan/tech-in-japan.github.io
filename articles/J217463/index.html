フラッシュからアプリストアまで、フルサイクル

背景



先日、私の最初のアプリはAppleでレビューされ、最初の試行で承認されました。 こうして、Actionscript 3で小さなおもちゃを書いてからappstoreのトップに至るまでの1か月の旅は終わりました。 すべての旅は、ポピーを使用せずにWindowsコンピューターを使用して行われました（ここには軟膏に小さなハエがありますが、それについては後で詳しく説明します）。得られた経験は多くの人にとって大きな助けになると思います。 私自身は、開発プロセス中に多くのGoogleを使用して、インターネット周辺から少しずつ有用な情報を収集する必要があったため、重要なポイントを強調する少しの指示は、興味がある人にとって有用です。







与えられた



ゲーム自体の開発については何もありません。設計、コーディングなどはこの記事の範囲外になります。 フラッシュ上でクロスプラットフォームアプリケーションを作成し、サイドに送信する方法を次に示します。 同じソースから、ブラウザで開き、モバイルプラットフォームでパックするアプリケーションを構築できます。 開発環境-フラッシュ開発。 ただし、Builderでは、すべてがほぼ同じです。 さらに、スターリングフレームワーク（ gamua.com/starling ）を使用しました-その美しさは、a）ハードウェアグラフィックスアクセラレーションをサポートしていることです。 お使いのフラッシュドライブはデバイスで遅くなることはほとんどなく、b）多重解像度をサポートします-異なる画面のサポートに関する心配は最小限に抑えられます



解決策



ステップ1.プロジェクトの作成、開発者アカウント、証明書の登録、プロビジョニング、その他すべて


この場所で、私は自分自身をあきらめてこのリンクをあなたに与えることができます： kpulv.com/115/FlashDevelop_to_iPad_Workflow-ここでは、プロジェクトを作成してからいくつかのiPhoneで最初のビルドを起動するまでの小さな手順と写真を紹介します。 必要なのは、100ドルでApple開発者アカウントとWindowsコンプを取得することです（ポピーはありません！）。アカウントの取得には約1日かかります（手動で調整するため）。 1時間以上かかります。 以前に独自のプロジェクトを既にお持ちの場合は、指示に従って作成されたプロジェクトにソースとリソースを転送し、デバイスをコンピューターに接続し、F5ボタンを押して、デバッグモードでデバイスのゲームがスローダウンする方法をお楽しみください:)これらのブレーキあなたは恐れてはいけません-彼らはデバッグからである可能性が最も高いです。 F5キーを押さずにPackageApp.batを実行し、そこでAd-Hocアセンブリを選択すると、アプリケーションはすばやく動作しますが、ビルドには非常に長い時間がかかります（5〜10分以内のアセンブリは通常です）。 インタープリターモードでは、非常に迅速にアセンブルされますが、動作は遅くなります。



ステップ2.ネイティブ拡張を接続する


Adobe Native Extensions（ane）は、振動、加速度計、アプリ内ゲームセンターなどのデバイスの機能を使用できるようにするライブラリです。 このようなものが必要ない場合は、この手順を安全にスキップしてください。 また、統計コレクション（Flurry）とGame Center自体を固定して、実績やリーダーボードをおもちゃにしたかったのです。 これを行うために、私はflurry aneとAdobe Gaming SDKをダウンロードしました。そこからgamecenter.aneのみが必要でした。 ただし、このgamecenter.aneを使用しないことをお勧めします。これについては後で詳しく説明します。



aneの接続は非常に簡単です（project explorer'eでファイルを見つけて右クリックし、[ライブラリに追加]をクリックします。aneをlibsフォルダーに追加することをお勧めします）。

まず 、デフォルトではライブラリとして接続されており、外部ライブラリが必要です。 これはここで切り替わります：



2番目：プロジェクトにあるSetupApplication.batファイル、batフォルダー、行call adt -package -target %TYPE%%TARGET% %OPTIONS% %SIGNING_OPTIONS% "%OUTPUT%" "%APP_XML%" %FILE_OR_DIR%



最後に、 -extdir lib/



を記述する必要があります。ここで、lib /は.aneがある場所へのパスです。

3番目：プロジェクトフォルダーにあるapplication.xmlで、次のような拡張機能のIDを登録する必要があります。 



 ... <extensions> <extensionID>com.adobe.ane.gameCenter</extensionID> <extensionID>com.sticksports.nativeExtensions.Flurry</extensionID> </extensions> </application>


これらの識別子は大文字と小文字が区別され、常に明らかではありません。 extensionIDを確実に知るには、.aneファイルをzipアーカイブとして開き、ファイル\ META-INF \ ANE \ extension.xml内を調べる必要があります。目的の行はタグ内に書き込まれます。



多田！ これで、ネイティブ拡張を使用できます。 たとえば、com.adobe.ane.gameCenterをインポートするだけで記述できます。*-使用できます。



ところで、Adobe Game Center ANEについてのコメント。 それは無料で存在しますが、ゲームセンターに許可があり、インターネットに接続していない場合、ゲームセンターにデータを送信しようとすると、アプリケーションがクラッシュするという事実につながります。 アドビはこれについて沈黙しており、問題は少なくとも数か月間知られていますが、修正はなく、いつ解決するかは明確ではありません。 さらに、アチーブメントの受信について画面からポップアップするバナーを表示する方法を知りません。 一般に、使用しない方が良いです。 無料の別のANEもありますが、コンパイルするにはiOS SDKが必要です。つまり、このものを使用するアプリケーションはポピーなしではビルドできません。 そして、70ドルのANEがありますが、これへのリンクは提供しません。すでにGoogleに最初の行を提供しています。 他に何も見つかりませんでした。



ステップ3. 1つのコード-多くのプラットフォーム


ネイティブ拡張を使用し始めたその時点で、コードはクロスプラットフォームでなくなります。 つまり、ブラウザで実行されるコードと電話機で実行されるフラッシュドライブからフラッシュドライブを収集することはできません。 そして、それは悲しいです、私たちはそれを望んでいません。 ゲームプレイのテストは、毎回IPAを収集したり、ゲームをデバイスにアップロードしたりせずに、コンピューターのフラッシュドライブの形で行うとはるかに便利です。 このために、松葉杖を思いつきました...曲がっていますが、うまくいきます。



同じフォルダーにプロジェクトファイルのコピーを作成しました。 たとえば、これらはProject.as3projおよびProjectMobile.as3projファイルです。 そして、Document Class'a Main.asファイルのコピー-MainMobile.as。 同時に、Flash Playerプラットフォームは、通常のプロジェクトの設定で指定され、ProjectMobileの設定-Air Mobile、プロジェクトのテスト項目で、「カスタムコマンドの実行」-Run.batが選択されます。 さらに、単純なプロジェクトでは、Mainがドキュメントクラスとして残り、ProjectMobileでは、MainMobileがドキュメントクラスとして機能しました。 次に、シングルトンゲームで、 public static var main:Class



を取得しました

各Mainは、この変数の値をそれ自体に等しく設定します。 つまり、通常のプロジェクトでは、GameクラスはmainがMainであると見なし、モバイルプロジェクトでは、mainがMainMobileであると考え、mainとして単純にアクセスできます。 MainMobileでは、必要なものをすべて.aneからインポートしますが、単純なMainではインポートしません。 さらに、ゲーム自体からモバイルであるチップを呼び出す必要がある場合、mainに対応する変数があればこれを行います。 たとえば、ゲームセンターの開設は次のようになります。 

  if(main.gc){ main.gc.showLeaderboardView() }


現在メインとしてMobile Mainがない場合、gc変数はなく、ゲームセンターは表示されません。 同時に、これが発生するGameクラス自体は、ゲームセンターについてまったく把握しておらず、必要なクラスはインポートされません



すべてのソースが共有されているという事実（もちろんMain.asを除く）にもかかわらず、プロジェクトを簡単に切り替えて、モバイルバージョンまたはブラウザーバージョンのいずれかを収集できます。

再び証明書

最初のステップでは、テスト用の証明書を生成しましたが、配布、配布、または本番用の証明書が必要になります（Apple Webサイトでは、これらの語は両方とも同じことを意味します）。 これは最初のステップとまったく同じ方法で行われますが、少し簡単です。 配布証明書を使用すると、プロビジョニング用のデバイスをリストする必要がありません。テストデバイス用ではなく、アプリのすべてのユーザー用にアプリケーションを構築しているためです。 一般に、.p12および.mobileprovisionファイルを取得して、使い慣れたSetupApplication.batに登録すると、次のようなものが取得されます。 

 set IOS_DIST_CERT_FILE=cert\ios_dist.p12 set IOS_DEV_CERT_FILE=cert\ios_dev.p12 set IOS_DEV_CERT_PASS=******* set IOS_PROVISION=cert\crushTest.mobileprovision set IOS_DIST_PROVISION=cert\crushDist.mobileprovision set IOS_ICONS=icons/ios set IOS_DEV_SIGNING_OPTIONS=-storetype pkcs12 -keystore "%IOS_DEV_CERT_FILE%" -storepass %IOS_DEV_CERT_PASS% -provisioning-profile %IOS_PROVISION% set IOS_DIST_SIGNING_OPTIONS=-storetype pkcs12 -keystore "%IOS_DIST_CERT_FILE%" -provisioning-profile %IOS_DIST_PROVISION%


テストと最終アセンブリを切り替えるたびに変更しないように、IOS_DIST_PROVISION = cert \ crushDist.mobileprovisionという行セットを追加しました（そして、最後の行の％IOS_PROVISION％を最後の行の％IOS_DIST_PROVISION％に置き換えました）。
