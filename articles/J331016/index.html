サーバー監視システムの概要。 muninを...に置き換える

長い間記事を書きたかったのですが、時間が足りませんでした。 この記事で説明されているように、muninのような単純な代替手段を見つけた場所はありません（Habréを含む）。







私はバックエンド開発者であり、プロジェクトに専任の管理者がいないことが非常に多いため（特に製品の初期段階）、私は長い間基本的なサーバー管理を行ってきました（初期セットアップ、構成、バックアップ、レプリケーション、監視など）。 私はこれが本当に好きで、いつもこの方向で何か新しいことを学びます。



ほとんどの場合、プロジェクトには1台のサーバーで十分であり、上級開発者（および単なる責任者）として、いつその制限に直面するかを理解するためにリソースを制御する必要がありました。 これらの目的には、Muninで十分でした。



インターフェース






ムニン



インストールは簡単で、要件はわずかです。 それはperlで書かれており、リングデータベース（ RRDtool ）を使用します。



設置例
コマンドを実行します。

apt-get munin munin-nodeをインストールします

サービスmunin-nodeの開始



munin-nodeはシステムメトリックを収集してデータベースに書き込み、5分ごとにこのデータベースからhtmlレポートを生成し、/ var / cache / munin / wwwフォルダーに配置します



これらのレポートを簡単に表示するために、nginxの簡単な構成を作成できます 

server { listen 80; server_name munin.myserver.ru; root /var/cache/munin/www; index index.html; }




実際にはそれだけです。 1日/週/月/年のCPU、メモリ、ハードドライブ、ネットワークなどのグラフをすでに見ることができます。 ほとんどの場合、データベースが常にボトルネックであるため、ハードディスクの読み取り/書き込みの負荷に興味がありました。



サーバーリソースを監視するには常に十分であり、 uptimerobot.comなどの無料サービスを使用してサーバーの可用性を監視しました。



この組み合わせを使用して、仮想サーバー上のホームプロジェクトを監視します。



プロジェクトが1つのサーバーから成長する場合、2番目のサーバーにmunin-nodeをインストールするだけで十分です。1番目に-構成に1行追加して、2番目のサーバーからメトリックを収集します。 両方のサーバーのグラフは別々になりますが、これは一般的な画像を表示するのに便利ではありません-どのサーバーが空きディスク容量を使い果たし、どのRAMが不足します。 この状況を修正するには、構成に数十行を追加して、両方のサーバーからのメトリックで1つのグラフを集約します。 したがって、これは最も基本的なメトリックに対してのみ行うことをお勧めします。 構成を間違えた場合は、ログに記録された内容を正確に読み取る必要があります。情報を見つけられずに、「pokeメソッド」を使用して状況を修正してください。



言うまでもなく、より多くのサーバーにとって、これは本当の地獄に変わります。 たぶんこれは、muninが2003年に開発されたものであり、元々このために設計されたものではないという事実によるものです。



複数のサーバーを監視するためのmuninの代替



新しい監視システムに必要な品質を自分で決定しました。

  • メトリックの数はmuninの数より少なくありません（キットには約30の基本グラフと約200のプラグインがあります）
  • bashでカスタムプラグインを作成する機能（これらのプラグインのうち2つがありました）
  • サーバー要件が小さい
  • 構成を編集せずに同じチャート上に異なるサーバーからのメトリックを表示する機能
  • メール、スラックおよび電報での通知
  • RRDtoolよりも強力な時系列データベース
  • 簡単インストール
  • これ以上
  • 無料でオープンソース

インターフェース






小さな余談



上記のシステムはすべて尊敬に値します。 ほとんどのLinuxディストリビューションのパッケージから簡単にインストールでき、多くのサーバーで実稼働で長い間使用されており、サポートされていますが、開発は非常に不十分で、インターフェースが古くなっています。

製品の半分はsqlデータベースを使用していますが、これは履歴データ（メトリック）の保存には最適ではありません。 一方では、これらのデータベースは汎用的であり、他方では、ディスクに大きな負荷をかけ、ストレージ中にデータがより多くのスペースを占有します。

このようなタスクには、 ClickHouseなどの最新の時系列データベースがより適しています。

新世代の監視システムは時系列データベースを使用します。それらの一部には分離不可能な部分として含まれ、他は個別のコンポーネントとして使用され、3番目はデータベースなしで機能します。

telegraf（エージェント）-> influxdb（db）-> grafana（インターフェース+通知）

セキュリティを重視



サーバー上のすべてのポートは、信頼するIPからのみ開く必要があります。使用する製品で認証を有効にし、デフォルトのパスワードを変更する必要があります（両方を実行します）。



influxdb



influxdbでは、認証はデフォルトで無効になっており、誰でも何でもできます。 したがって、サーバーにファイアウォールがない場合は、許可を有効にすることを強くお勧めします。 



 #   : influx -execute 'CREATE DATABASE telegraf' influx -execute 'CREATE USER admin WITH PASSWORD "password_for_admin" WITH ALL PRIVILEGES' influx -execute 'CREATE USER telegraf WITH PASSWORD "password_for_telegraf"' influx -execute 'CREATE USER grafana WITH PASSWORD "password_for_grafana"' influx -execute 'GRANT WRITE ON "telegraf" TO "telegraf"' # telegraf      influx -execute 'GRANT READ ON "telegraf" TO "grafana"' # grafana      #  /etc/influxdb/influxdb.conf   [http]   auth-enabled   : sed -i 's| # auth-enabled = false| auth-enabled = true|g' /etc/influxdb/influxdb.conf systemctl restart influxdb
