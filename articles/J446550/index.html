すべての解釈におけるコーディネーターパターンは、この図にほぼ適合しています。













つまり、コーディネーターはプロトコルです 







 protocol Coordinator { func start() }


そして、すべてのcodeいコードはstart

このアプローチは何ですか？ その中で、スタック（ツリー）に依存するために、コントローラーをそのままひねります。 従う必要がある不必要なエンティティを作成しないため。 条件を保存または追跡しないでください。







UIKitエンティティをより詳しく見て、 最終的に何があるのか​​、何を操作できるのかを考えてみましょう。







  1. コントローラスタックはツリーです。 子View Controllerを持つルートView Controllerがあります。 モーダルで提示されるView Controllerは、生成されたView Controllerへのバインディングも持つため、子View Controllerの特殊なケースです。 すぐに使用できます。
  2. コントローラーの本質を作成する必要があります。 それらはすべて異なるコンストラクターを持ち、Xibファイルまたはストーリーボードを使用して作成できます。 これらには異なる入力パラメーターがあります。 しかし、それらは作成される必要があるという点で団結しています。 これで、目的のView Controllerを作成する方法を知っているFactoryパターンができました。 各工場は包括的な単体テストで簡単にカバーでき、他の工場からは独立しています。
  3. View Controllerを2つのクラスに分割します：1. Controllerを表示するだけです。2. Container View Controller（Container View Controller） 。 コンテナView Controllerは、子View Controllerを含めることができるという点で通常のものと異なります-コンテナまたはシンプルなものも含まれます。 このようなView Controllerは、 UITabBarController



    、 UITabBarController



    などのボックスから使用できますが、ユーザーが作成することもできます。 無視すると、すべてのコンテナで次のプロパティを見つけることができます。1.含まれているすべてのコントローラのリストがあります。 2. 1つ以上のコントローラーが現在表示されています。 3.これらのコントローラーのいずれかを表示するように求められる場合があります。 UIKitコントローラーでできることはこれだけです 。 彼らはこのために異なる方法を持っています。 ただし、タスクは3つしかありません。
  4. 工場で作成されたView Controllerを埋め込むために、Controllerの親ViewメソッドはUINavigationController.pushViewController(...)



    、 UITabBarController.selectedViewController = ...



    、 UIViewController.present(...)



    です。 2つのView Controllerが常に必要であり、1つは既にスタック上にあり、もう1つはスタックに埋め込む必要があることに気付くかもしれません。 これをラッパーでラップし、 Action（Action）と呼びます。 各アクションは包括的な単体テストで簡単にカバーでき、それぞれは他のアクションから独立しています。
  5. 上記から、準備されたエンティティを使用して、構成チェーンを構築できることがわかりましたFactory- > Action-> Factory-> Action-> Factoryそして、完了後、任意の複雑なコントローラーのビューツリーを構築できます。 エントリポイントを指定するだけです。 通常、これらの入力ポイントは、 UIWindowが所有するrootViewControllerか、ツリーの最も極端なブランチである現在のView Controllerです。 つまり、このような構成は、 ViewController-> Action-> Factory-> ...-> Factoryのように正しく記述されます。
  6. 構成に加えて、提供された構成を開始および構築する方法を知っているエンティティが必要になります。 これをRouterと呼びます。 状態がなく、リンクを保持していません。 構成が渡される1つのメソッドがあり、構成手順を順番に実行します。
  7. インターセプタークラスを構成チェーンに追加して、ルーターに責任を追加します。 インターセプターには次の3種類があります。1.ナビゲーションを開始する前に起動します。 システム内のユーザー認証タスクと、その他の非同期タスクを削除します。 2. View Controllerの作成時に実行して、値を設定します。 3.ナビゲーションおよびさまざまな分析タスクの実行後に実行されます。 各エンティティは単体テストで簡単にカバーされ、構成でどのように使用されるかわかりません。 彼女には1つの責任しかなく、それを果たします。 つまり、複雑なナビゲーションの構成は、 [事前ナビゲーションタスク...]-> ViewControllerの開始->アクション->（ファクトリー+ [ContextTask ...]）-> ...->（ファクトリー+ [ContextTask ...]）-> [ポストNavigationTask ...] 。 つまり、すべてのタスクはルーターによって順番に実行され、順番に小さくて読みやすいアトミックエンティティを実行します。
  8. 構成で解決できない最後のタスクは残ります-これは現時点でのアプリケーションの状態です。 ユーザーが部分的に渡したため、構成チェーン全体ではなく、その一部のみを構築する必要がある場合はどうなりますか？ この質問は、ビューコントローラーのツリーによって常に明確に回答できます。 チェーンの一部がすでに構築されている場合、それはすでにツリー内にあるためです。 これは、チェーン内の各工場が建設されているかどうかの質問に回答できる場合、ルーターはチェーンのどの部分を完了する必要があるかを理解できることを意味します。 もちろん、これはファクトリーのタスクではないため、別のアトミックエンティティが導入されます- 検索エンジン（Finder）および構成は次のようになります： [Pre-navigation Task ...]-> ViewControllerの開始-> Action->（Finder / Factory + [ContextTask ...]） -> ...->（Finder / Factory + [ContextTask ...]）-> [Post NavigationTask ...] 。 ルーターが最後から読み取りを開始すると、ファインダーの1つが既に構築されていることを通知し、この時点からルーターはチェーンの構築を開始します。 それらのいずれかがツリーで自分自身を見つけられない場合、最初のコントローラーからチェーン全体を構築する必要があります。

  9. 構成は厳密に入力する必要があります。 したがって、各エンティティは1種類のコントローラービューでのみ動作します; 1種類のデータと設定は、swiftがrelatedtypesを使用できるかどうかに完全に依存しています。 ランタイムではなく、コンパイラに依存したいのです。 開発者は意図的にタイピングを弱めることができますが、その逆はできません。
