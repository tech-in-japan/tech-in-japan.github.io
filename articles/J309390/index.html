エンタープライズクラウドストレージ用の機器の選択





データはあらゆるビジネスの基盤です。 保管場所の信頼性が不十分であるか、常時アクセスを提供できない場合、事実上、企業のすべての活動が危険にさらされます。



もちろん、サーバーソフトウェアと適切な構成を正しく選択することにより、情報の安全性とアクセシビリティを確保することが可能であり、必要です。 しかし、鉄も重要です-データを保存および処理する機器。 それが会社のニーズを満たさない場合、ソフトウェアはそれを十分に信頼できフォールトトレラントにすることはできません。



この記事では、エンタープライズクラウドストレージの作成に鉄を選択するアプローチの1つを検討します。



クラウドはなぜですか？



クラウドインフラストラクチャにはいくつかの利点があります。



  • 迅速にスケーリングする機能 。 追加のサーバーとストレージを迅速に接続することにより、ストレージ容量とコンピューティング能力が向上します。 これは、クラウドの負荷が不規則であると想定される企業に特に当てはまります。



  • コスト削減 。 クラウドを使用すると、すべてのコンピューティングプロセスを実行する単一のセンターを作成できます。ディスクスペースを増やすには、新しいサーバーのインストールを編成する必要なく、ドライブを購入するだけで十分です。



  • ビジネスプロセスの簡素化 。 クラウドストレージは、ローカルストレージとは異なり、常にアクセスできる可能性を意味します。 これは、いつでもどこからでもファイルを操作できることを意味します。 従業員は、仕事に必要な情報をより簡単かつ迅速に受け取ることができ、リモートジョブを整理することができます。



  • フォールトトレランスの向上 。 データが複数のサーバーに保存されている場合、技術的な問題が発生した場合の安全性は、1台のマシンのみにある場合よりも高くなることは明らかです。


なぜ私なの？



現在、市場には多くのパブリッククラウドサービスがあります。 多くの中小企業にとって、彼らは本当に良い選択になります。特に、使用されるリソースのみの支払いやサービスのテストに関してはそうです。 ただし、クラウドストレージにはいくつかの利点もあります。 次の場合に役立ちます。



  • 会社の活動は、サーバーの場所に制限を課しています 。 ロシアの州。 機関、および個人データの処理に関与する組織は、ロシア連邦の領土にすべての情報を保存することが法律で義務付けられています。 したがって、外部サーバーをレンタルすることは不可能であり、一般に、機密情報を請負業者に信頼することは非常に望ましくありません。 プライベートウェアハウスを作成すると、法律に違反することなくクラウドを最大限に活用できます。



  • セキュリティポリシーの完全な管理が必要です 。 MicrosoftまたはAmazonサービスでデータ保護がどのように機能するかを正確に知ることは不可能です。 必要と思われる情報を完全に保護するには、自分のクラウドでのみ可能です。



  • 自分用の機器のカスタマイズ 。 レンタルするときは、サプライヤーが提供するものを使用する必要があります。 ただし、独自のサーバーを自由に使用して、ビジネスに特化した特定の問題を解決するように構成したり、ITチームが完全に知っているソフトウェアを使用したりできます。


機器の選択



クラウドストレージ用の機器を購入するとき、しばしば疑問が生じます。車を借りるか、自分で購入するかです。 私たちは、どのような場合に自分のサーバーが不可欠であるかをすでに把握しています 。 ただし、サードパーティのサービスでクラウドを整理できます。 これは特に小規模企業に当てはまります。サーバーの購入予算から必要な金額を常に割り当てることができるわけではなく、独自のサーバールームを作成することも常に可能ではなく、マシンのメンテナンスにお金を費やす必要もないためです。



ただし、クラウドストレージ用に独自のサーバーを使用することに決めた場合は、機器の変更に問題が生じ、費用がかかることに留意してください。 容量が十分でなくても問題ありません。いつでも別のマシンをクラスターに追加できます。 しかし、パフォーマンスが過剰であることが判明した場合、これでは何もできません。 したがって、選択するのは次のことです。



サーバーを使用する目的を明確に定義します 。 私たちの場合、これはファイルストレージです。 したがって、最も興味深いのはドライブです。 それらを選択する際に注意すべきこと：



容量 。 ストレージを使用する従業員の数、サーバーにアップロードするファイルの種類、クラウドへの転送を既に待機している情報の量、および作業データの量が毎年増加している量に依存します。



平均して、テキストファイル、プレゼンテーション、PDF、および少数の画像を扱うには、従業員あたり平均10〜15 GBが必要です。 大量の高品質の写真や写真を扱うには、少なくとも50〜100 GB、またはそれ以上に増やす必要があります。 ビデオおよび音声処理担当者のニーズは、1人あたり数テラバイトに達する可能性があります。 たとえば、バージョン管理されたプロジェクトをサポートする大規模な企業ソフトウェアパッケージを使用している場合、クラウドユーザーあたり約10テラバイトの話をすることができます。 バックアップファイルの容量と会社の予期しないニーズを考慮することを忘れないでください。



RAIDコントローラーに関しては、企業クラウドにオンボードソリューションを使用しない方が良いです。 それらのパフォーマンスは、十分な速度で多数の要求を処理するには不十分な場合があります。 したがって、低価格および中価格帯から個別のモデルを選択することをお勧めします。



また、 処理能力を決定する必要があります。 複数のサーバーに基づいてクラウドストレージを作成する場合、同一または非常に近い構成を選択することをお勧めします。 これにより、負荷分散の管理が多少簡単になります。 一般に、1台の強力なマシンに賭けて高価なプロセッサとRAMを装備するのではなく、2〜3台の車を安く購入する方が適切です。 なんで？



ストレージが起動の可能性なしに静的ファイルのみを受け入れ、提供する場合、プロセッサの電力はそれほど重要ではありません。 したがって、コアの数を追跡せずに、「タクト」が良好なモデルを選択することをお勧めします。 安価なオプションのうち、4コアのIntel XeonプロセッサーE56XXシリーズは悪くありません。IntelCore i5マシンは、より高価なモデルから推奨されます。



サーバーモデルの例



サーバーを自分で構築するのではなく、すぐに使用できる機器をすぐに入手する場合は、ファイルストレージの作成に適したいくつかのモデルに注意してください。



Dell PowerEdge T110 サーバーには2つのコアのみを備えたIntel Core i3 2120プロセッサーが搭載されていますが、それぞれのクロック速度は3.3 GHzであり、クラウドにとってより重要です。 RAMの初期構成はそれほど大きくありません-4 GBですが、32 GBに拡張できます。 サーバーには、ハードドライブが事前にインストールされていないか、1TB HDDドライブとSATAインターフェースの2つのトリムレベルがあります。



Lenovo ThinkServer RS140 それぞれ3.3 GHzの4つのコアを備えた強力なIntel Xeon E3プロセッサを搭載しています。 「すぐに使える」RAM-4 GB、さらに拡張用の4つのスロット。 SATAインターフェースを備えた2台の1 TBハードドライブも含まれます。



HP ProLiant ML10 Gen9 多くの点で、これは上記のモデルに似ています-すべて同じIntel Xeon E3と2テラバイトのHDD。 RAMの主な違い-HPサーバーには、それぞれ4 GBのプレートが2つあります。



ファイルを保存するのに十分なスペースはありますか？



ストレージ容量は、ファイルサーバーの基盤です。 保存されたデータの量と成長のダイナミクスを評価した後、6か月後に予測を間違え、データが計画よりも速く成長するという不快な結論に達することができます。



ストレージ仮想化の場合、仮想マシンのディスクサブシステムを拡張したり、LUNを増加したりできるのは、ほとんどの場合（ストレージ計画に対する合理的なアプローチを使用）です。 物理ファイルサーバーとローカルディスクの場合、機能はより控えめになります。 サーバーに追加のディスク用の空きスロットが存在する場合でも、RAIDアレイに適したドライブを選択するという問題が発生する場合があります。



しかし、広範囲に問題を解決する前に、スペース不足と戦うのに役立つ技術ツールについて覚えておく必要があります。



NTFSディスククォータ



最も古く、最も信頼性の高いユーザー制限メカニズムの1つは、ファイルシステムクォータです。







ボリュームのクォータを有効にすると、各ユーザーが保存するファイルの量を制限できます。 NTFSレベルでユーザーが所有しているファイルは、ユーザーのクォータに分類されます。 このメカニズムの主な欠点は、第1に、特定のユーザーに属するファイルを特定するのがそれほど容易ではないことです。第2に、管理者が作成したファイルはクォータに含まれません。 クォータメカニズムは実際にはめったに使用されず、Windows Server 2003 R2で最初に登場したファイルサーバーリソースマネージャーによってほぼ完全に置き換えられました。



ファイルサーバーリソースマネージャー



Windows Serverのこのコンポーネントは、特定のフォルダーのレベルでディスク領域のクォータを許可します。 従業員用のファイルサーバー上の個人のホームディレクトリ、および共有部門ドキュメント用の専用フォルダーを選択する場合、FSRMが最適です。







もちろん、クォータだけではファイルストレージは増加しません。 ただし、ユーザーは原則として、リソースの公平な（均等な）分割に忠実であり、不足した場合、ディスク容量を拡張するための小さな官僚的な手順を克服する準備ができています。



クォータは、ユーザーが誤って大量の情報を配置した場合のサーバーの過負荷にも役立ちます。 少なくとも、これは他の従業員や部門には影響しません。



さらに、FSRMには、サーバーに保存できるファイルを "スクリーニング"（フィルタリング）するためのメカニズムが含まれています。 mp3-およびavi--ファイルにファイルサーバー上の場所がないことが確実な場合は、FSRMを使用してそれらを保存しないようにすることができます。



NTFS圧縮



通常のファイルはNTFSで圧縮でき、最新のプロセッサのパフォーマンスを考慮して、サーバーにはこの操作に十分なリソースがあります。 十分なスペースがない場合は、ボリュームまたは個々のフォルダーに安全に含めることができます。 たとえば、Windows Server 2012では、より高度なメカニズムが登場し、ほとんどのシナリオでファイルサーバー上のNTFS圧縮を使用することは過去のものになりました。







重複排除



Windows Server 2012には、NTFSボリュームにあるデータを重複排除する機能が含まれています。 これは、重複排除と可変長ブロックの両方と、保存されたブロックの効果的な圧縮の両方を組み合わせた、かなり高度で柔軟なメカニズムです。 さらに、さまざまなタイプのデータに対して、メカニズムは異なる圧縮アルゴリズムを使用できます。圧縮が効果的でない場合は、使用しないでください。 このような微妙な点は、NTFSを使用した従来の圧縮では利用できません。



また、重複排除では、動的に変化するデータの操作速度を低下させないために、過去30日間（この間隔は設定可能）ユーザーが操作したファイルを最適化しません。



ddpevalユーティリティを使用して、空き領域の潜在的な増加を評価できます。 一般的なファイルサーバーでは、節約量は30〜50％です。







ファイルサーバーのパフォーマンス



前述したように、ファイルサーバーは最もリソースを必要とするサービスではありませんが、それでも、ディスクサブシステムとネットワークサブシステムの構成に合理的にアプローチする必要があります。



ディスクサブシステム

線形の読み取りまたは書き込み速度は、ファイルサーバーにとって重要ではありません。 最新のハードドライブはいずれも線形の読み取り/書き込み速度の高い特性を備えていますが、ユーザーが大きなファイルをローカルドライブにコピーする場合、または逆にサーバーにファイルを配置する場合にのみ重要です。



Perfmonの統計を見ると、150〜200人のユーザーの平均読み取り/書き込み速度は非常に低く、1秒あたり数メガバイトです。 ピーク値はより興味深いものです。 ただし、これらのピークはネットワークインターフェイスの速度によっても制限され、通常のサーバーでは1ギガビット/秒（つまり、ディスクサブシステムとの100 MBの交換）であることに注意してください。







通常の操作では、ファイルへのアクセスは非線形であり、任意のブロックはディスクから読み書きされます。したがって、ランダムアクセス操作のディスクパフォ​​ーマンス、つまり最大IOPSはより重要です。



150〜200人の従業員の場合、インジケータはやや控えめです。1〜2以内のディスクキューで、1秒あたり10〜20の入出力操作が行われます。







標準のSATAドライブのアレイは、これらの要件を満たします。



500〜1000人のアクティブユーザーの場合、操作の数は250〜300にジャンプし、ディスクキューは5〜10に達します。 キューがこの値に達すると、ユーザーはファイルサーバーの速度が低下していることに気付くことができます。







実際には、300 IOPSのパフォーマンスを実現するには、少なくとも3〜4個の一般的なSATAドライブのアレイが必要です。



この場合、「生のパフォーマンス」だけでなく、RAIDコントローラーの操作によって生じる遅延、いわゆるRAIDペナルティも考慮する必要があります。 このトピックは、 https：//habrahabr.ru/post/164325/の記事で明確に説明されています 。



必要なディスクの数を決定するには、次の式を使用します。 



Total number of Disks required = ((Total Read IOPS + (Total Write IOPS*RAID Penalty))/Disk Speed IOPS)


4つの操作の書き込みペナルティ、50％の読み取りプロファイル、50％の書き込みプロファイル、75 IOPSのディスク速度、300 IOPSの目標パフォーマンスを備えたRAID-5： 



 (300*0,5 + (300*0,5*4))/75 = 10 .


アクティブなユーザーが多数いる場合は、容量の大きいサーバーまたは10,000 RPMの回転速度のSASなどのより生産的なディスクが必要になります。



ネットワークインターフェイスの速度

ネットワークサーバーの速度が遅いことが、ファイルサーバーを使用する際の遅延の原因の1つです。 2016年、100 Mbpsネットワークカードを搭載したサーバーはナンセンスです。



一般的なサーバーには、1 Gbit / sの速度のネットワークカードが装備されていますが、これにより約100 Mb / sのディスク交換速度も制限されます。 サーバーに複数のネットワークカードがある場合は、それらを1つの論理インターフェイスに結合（集約）して、クラウドのパフォーマンスと可用性の両方を向上させることができます。 幸いなことに、ファイルサーバー（「多くのクライアントが同じサーバーにアクセスする」）の場合、集約はうまく機能します。



HPサーバーの所有者は、独自のHPネットワーク構成ユーティリティを使用できます。







Windows Server 2012を使用する場合、よりシンプルで信頼性の高い方法は、標準のNICチーミングツールを使用することです。
