HTTPに関する簡単な言語

HTTPプロトコルの主な側面について説明します-90年代初頭から今日までのネットワークプロトコルにより、ブラウザでWebページをロードできます。 この記事は、コンピューターネットワークでの作業を開始し、ネットワークアプリケーションを開発し始めたばかりで、公式仕様を自分で読むのがまだ難しい人向けに書かれています。



HTTPは、ハイパーテキストドキュメント（つまり、他のドキュメントへの移行を整理できるリンクが含まれている可能性のあるドキュメント）の転送を目的とした広範なデータ転送プロトコルです。



HTTPの頭字語は、 HyperText Transfer Protocol 、「Hypertext Transfer Protocol 」の略です。 OSI仕様によると、HTTPはアプリケーション（上位、7番目）層プロトコルです。 プロトコルの現在のバージョンであるHTTP 1.1は、 RFC 2616仕様に記載されています 。



HTTPプロトコルには、クライアントサーバーデータ転送構造の使用が含まれます。 クライアントアプリケーションは要求を生成し、サーバーに送信します。その後、サーバーソフトウェアはこの要求を処理し、応答を生成してクライアントに返します。 その後、クライアントアプリケーションは、同様の方法で処理される他のリクエストを送信し続けることができます。



HTTPプロトコルを使用して従来解決されていたタスクは、Webリソース（通常はWebブラウザー）にアクセスするユーザーアプリケーションとWebサーバー間のデータ交換です。 現在、World Wide WebがサポートされているのはHTTPプロトコルのおかげです。



また、HTTPは、SOAP、XML-RPC、WebDAVなどの他のアプリケーション層プロトコルの情報転送プロトコルとしてよく使用されます。 この場合、彼らはHTTPプロトコルが「トランスポート」として使用されると言います。



多くのソフトウェア製品のAPIは、データ転送にHTTPを使用することも意味します。データ自体は、XMLやJSONなどの任意の形式にすることができます。



通常、HTTPを介したデータ転送はTCP / IP接続を介して行われます。 サーバーソフトウェアは通常、TCPポート80を使用します（ポートが明示的に指定されていない場合、クライアントソフトウェアは通常、開いているHTTP接続にデフォルトで80番目のポートを使用します）。



HTTPリクエストを送信する方法は？



HTTPプロトコルに対処する最も簡単な方法は、Webリソースに手動でアクセスしてみることです。 あなたがブラウザであり、アナトリーアリザーの記事を本当に読みたいユーザーがいると想像してください。



彼がアドレスバーに次のように入力したとします。



http://alizar.habrahabr.ru/







したがって、Webブラウザーとして、alizar.habrahabr.ruのWebサーバーに接続する必要があります。



これには、任意の適切なコマンドラインユーティリティを使用できます。 たとえば、telnet：



telnet alizar.habrahabr.ru 80







急に気が変わったら、Ctrl +“]”を押してから入力してください。これにより、HTTP接続を閉じることができます。 telnetに加えて、nc（またはncat）を試してみることができます。



サーバーに接続したら、HTTPリクエストを送信する必要があります。 ちなみに、これは非常に簡単です-HTTPリクエストは2行のみで構成できます。



HTTP要求を生成するには、開始行を作成するとともに、少なくとも1つのヘッダーを指定する必要があります-これはHostヘッダーであり、各要求に存在する必要があります。 実際には、ドメイン名はクライアント側でIPアドレスに変換されるため、TCP接続を開くと、リモートサーバーには接続に使用されたアドレスに関する情報がありません。たとえば、 、アドレスはalizar.habrahabr.ru、habrahabr.ruまたはm.habrahabr.ruです。これらすべての場合、答えは異なる場合があります。 ただし、実際には、すべての場合でネットワーク接続はホスト212.24.43.44で開かれ、接続の開始時に、このIPアドレスではなく、ドメイン名が指定されていても、サーバーにはこれについて通知されません-このため、このアドレスが必要ですホストヘッダーを渡します。



HTTP 1.1の開始（初期）要求行は、次のように構成されています。



HTTP URI メソッド / バージョン



例（このような開始行は、サイトのメインページがリクエストされていることを示す場合があります）：



GET / HTTP/1.1







メソッド （英語の主題文献ではmethodという単語を使用し、 動詞 -"verb"という単語も使用されます）は、コントロールとセパレータを除く任意の文字のシーケンスであり、指定されたリソースで実行する必要がある操作を決定します。 HTTP 1.1仕様は、使用できるさまざまなメソッドの数を制限しませんが、一般的な標準に準拠し、可能な限り幅広いソフトウェアとの互換性を維持するために、原則として、最も標準的なメソッドの一部のみが使用されます。その意味はプロトコル仕様で明確に開示されています。



URI （ Uniform Resource Identifier ）-操作を実行する必要のある特定のリソース（ドキュメントなど）へのパス（たとえば、GETメソッドを使用する場合、リソースの受信を意味します）。 一部のリクエストはどのリソースにも適用されない場合があります。この場合、URIの代わりにアスタリスク（アスタリスク、「*」文字）を開始行に追加できます。 たとえば、特定のリソースではなく、Webサーバー自体に関連するリクエストの場合があります。 この場合、開始行は次のようになります。



OPTIONS * HTTP/1.1







バージョンは、要求が構成されるHTTP標準のバージョンに従って決定します。 これは、ドットで区切られた2つの数値（たとえば1.1 ）として示されます。



特定のアドレスのWebページにアクセスするには（この場合、リソースへのパスは「/」です）、次のリクエストを送信する必要があります。



GET / HTTP/1.1





Host: alizar.habrahabr.ru







行を折り返すには復帰改行文字を使用し、その後に改行文字を使用する必要があることに注意してください。 最後のヘッダーが宣言された後、改行の文字シーケンスが2回追加されます。



ただし、HTTP仕様では、要求を処理するときにLF文字が行区切り文字として使用され、前のCR文字（存在する場合）が無視されるようにHTTPサーバーをプログラムすることをお勧めします。 したがって、実際には、ほとんどのサーバーはそのような要求を正しく処理します。ヘッダーはLFシンボルで区切られ、最後のヘッダーが宣言された後に2回追加されます。



仕様に厳密に従ってリクエストを送信する場合は、エスケープシーケンス\ rおよび\ nを使用できます。



echo -en "GET / HTTP/1.1\r\nHost: alizar.habrahabr.ru\r\n\r\n" | ncat alizar.habrahabr.ru 80







答えを読むには？



応答の開始行の構造は次のとおりです。



HTTP / バージョン ステータスコード 説明



プロトコルバージョンは、リクエストと同様にここで指定されます。



ステータスコード -リクエストの結果を決定する3桁の数字（最初の数字はステータスクラスを示します）。 たとえば、GETメソッドが使用され、サーバーが指定された識別子を持つリソースを提供する場合、この状態はコード200を使用して設定されます。サーバーがそのようなリソースが存在しないことを報告する場合-404。 HTTP 1.1仕様では40種類のHTTPコードが定義されており、プロトコルを拡張して追加のステータスコードを使用することもできます。



ステータスコードの説明 （ Reason Phrase ）-応答コードのテキスト（ CRおよびLF文字を除く）の説明は、回答を読みやすくするためのものです。 説明はクライアントソフトウェアによって考慮されない場合があり、一部のサーバーソフトウェア実装の標準とも異なる場合があります。



開始行の後にヘッダーが続き、応答の本文が続きます。 例： 



 HTTP/1.1 200 OK Server: nginx/1.2.1 Date: Sat, 08 Mar 2014 22:53:46 GMT Content-Type: application/octet-stream Content-Length: 7 Last-Modified: Sat, 08 Mar 2014 22:53:30 GMT Connection: keep-alive Accept-Ranges: bytes Wisdom


応答本文は、最後のヘッダーの後の2つの改行に続きます。 応答本文の終わりを決定するために、 Content-Lengthヘッダーの値が使用されます（この場合、応答には7桁の8バイトが含まれます：単語「Wisdom」と改行文字）。



しかし、以前に行った要求によると、Webサーバーはコード200ではなくコード302で回答を返します。したがって、現時点では別のアドレスでこのリソースにアクセスする必要があることをクライアントに通知します。



自分で見てください： 



 HTTP/1.1 302 Moved Temporarily Server: nginx Date: Sat, 08 Mar 2014 22:29:53 GMT Content-Type: text/html Content-Length: 154 Connection: keep-alive Keep-Alive: timeout=25 Location: http://habrahabr.ru/users/alizar/ <html> <head><title>302 Found</title></head> <body bgcolor="white"> <center><h1>302 Found</h1></center> <hr><center>nginx</center> </body> </html>


Locationヘッダーは新しい住所です。 これで、URI（リソース識別子）が/ users / alizar /に変更され、今回はhabrahabr.ruのサーバー（ただし、この場合は同じサーバー）にアクセスし、Hostヘッダーで指定する必要があります。



それは：



GET /users/alizar/ HTTP/1.1





Host: habrahabr.ru







このリクエストへの応答として、Habrahabr Webサーバーは既にコード200とかなり大きなHTMLドキュメントで応答を発行します。



すでにロールに慣れている場合は、サーバーから受け取ったHTMLコードを読み取り、鉛筆とノートを取り、Alizarのプロファイルを描画できます-原則として、これはブラウザが行う場所です。



セキュリティはどうですか？



HTTPプロトコル自体は、暗号化を使用して情報を送信することを意味するものではありません。 それでも、HTTPには、暗号化プロトコルSSLまたはTLSで送信データのパッケージ化を実装する広範な拡張機能があります 。



この拡張機能の名前はHTTPS （ HyperText Transfer Protocol Secure ）です。 HTTPS接続では、通常TCPポート443が使用されます。HTTPSは、情報を傍受から保護するために広く使用されており、また、原則として、クライアントで証明書が検証された場合に中間者攻撃から保護します。ただし、証明書の秘密キーは侵害されず、ユーザーは署名のない証明書の使用を確認せず、攻撃者の認証局の証明書はユーザーのコンピューターに実装されませんでした。



HTTPSは現在、すべての一般的なWebブラウザーでサポートされています。



追加機能はありますか？



HTTPプロトコルは、かなり多数の拡張オプションを提供します。 特に、HTTP 1.1仕様では、Upgradeヘッダーを使用して別のプロトコルを使用したデータ交換に切り替える可能性が提案されています。 このヘッダーを持つ要求は、クライアントによって送信されます。 サーバーが別のプロトコルを使用してデータ交換に切り替える必要がある場合、サーバーはステータス「426 Upgrade Required」の応答をクライアントに返すことができます。その場合、クライアントはすでにUpgradeヘッダーを持つ新しい要求を送信できます。



この機能は、特にWebSocketプロトコル（ RFC 6455仕様で説明されているプロトコルで、追加のHTTP要求を送信せずに適切なタイミングでデータを転送できるプロトコル）を使用してデータ交換を整理するために使用されます。値が「websocket」のUpgradeヘッダーを持つHTTPリクエスト。サーバーはステータス「101 Switching Protocols」で応答を返し、どちらの側もWebSocketプロトコルを使用してデータの送信を開始できます。



ところで、何か他のものを使用していますか？



現時点では、Webコンテンツを送信するために設計された他のプロトコルがあります。 特に、 SPDYプロトコル （英語の単語speedyと発音されます ）はHTTPプロトコルの変更であり、その目的はWebページの読み込みの遅延を減らし、追加のセキュリティを提供することです。



すべてのデータを単一の接続内で転送できるように、Webページに必要な追加リソースを圧縮、優先順位付け、および多重化することにより、速度が向上します。



2012年11月に公開されたHTTP 2.0プロトコルのドラフト仕様（1999年に公開された最終仕様であるバージョン1.1以降のHTTPプロトコルの次のバージョン）は、SPDYプロトコル仕様に基づいています。



HTTP 2.0仕様のドラフトの準備中にhttpbisワーキンググループが検討した他の提案された実装と同様に、SPDYプロトコルで使用されるアーキテクチャソリューションの多くは、以前はHTTP-NGプロトコルの開発中に取得されましたが、HTTP-NGプロトコルの作業は中止されました1998年。



現時点では、SPDYプロトコルのサポートは、Firefox、Chromium / Chrome、Opera、Internet Exporer、およびAmazon Silkのブラウザーでサポートされています。



それで何？



一般に、はい。 特定のメソッドとヘッダーを記述することは可能ですが、実際には、特定の何か（たとえば、HTTPを介してサーバーと通信するWebサーバーまたはクライアントソフトウェア）を作成する場合は、この知識が必要です。プロトコルの仕組みに関する基本的な理解は必要ありません。 さらに、これらはすべてGoogleを介して非常に簡単に見つけることができます。この情報は仕様書やウィキペディアなどにあります。



ただし、英語を知っていて、HTTPだけでなく、送信に使用されるTCP / IPパケットについても調査したい場合は、 この記事を読むことをお勧めします。



もちろん、実際に使用を開始すると、どのテクノロジーもはるかにシンプルで理解しやすくなることを忘れないでください。
