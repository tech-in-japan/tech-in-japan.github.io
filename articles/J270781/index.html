WPF開発者向けのUWPアプリケーションについて







何か新しいものへの恐怖からWPF開発者をエンコードし、ユニバーサルWindowsプラットフォーム用のアプリケーションを開発する際に彼らを待っている違いについて語りたいと思います。 だから、モニターの前に缶を置いて、私はインストールを始めています。



プログラミング言語とプラットフォームのいくつかの変更が進行中です。 C＃バージョン10がリリースされると想像してみてください。 これは最新バージョンです。 提示？ 紹介しました。 そして、私の意見では、これが発生した場合、言語のこの最新バージョンは定期的に更新されます。



コードの違い（XAMLコード内であっても）から少し目が離せた「不思議な」時がありました。WPF、Silverlight、Windows Phone、WinRT、そして今やUWPです。 混乱しない開発者は何人いますか？ ほとんどの開発者はすべてを念頭に置いているわけではないと思います。 仕事の基本概念を覚えておけば十分です。 コーディングに関しては、IntelliSense、Blendなどの補助ツールが使用されます。 スニペットを使用してどこにも行かないでください。



変更の理由は何ですか：



1.有害な叔父がどこかに座って、誰もが開発プラットフォーム/環境に慣れる瞬間を待っています。 そして、彼は何かを変えます。

2.改善/新機能。

3.ユーザーまたは開発者からのフィードバック（私は気に入らなかったので、それだけです）。 バグ修正。

4.ハードウェアは改善されています。 省エネ、パフォーマンス、または美しい効果を重視。

5.グローバルな何か。 たとえば、UWPでの最後のプラットフォーム統合、または「C＃にXamarinをバイパスしてすべてのプラットフォームに一度に書き込みたいので、他のプラットフォームと同様の新しいコントロールを追加しました」などです。 おそらく数年後にはそうなるでしょう。

6.コメントにバージョンを記入します。



開発者がUWPの知識を必要とするのはなぜですか？ はい、もし.NetアプリケーションでAPIとUWPサービスを近い将来使用することができるからです。 Project Cenntenialを使用すると、このすべてがすぐに可能になる可能性があります。



そのため、新しいプラットフォームに精通することはかなり可能です。 いくつかの違いを説明する短い遠足をしてみましょう。



まず、UWPアプリケーションには、従来のWindowsアプリケーションにはないものがあります-アプリモデルがあります。 アプリモデルとは何ですか？ これは一種の規制です。 アプリケーションのすべての機能の説明-アクセス権、インストール方法、更新方法、情報の保存など。



Windowsストアアプリケーションは、UWPアプリケーションと同様に、アプリケーションのすべての機能と権限を記述するマニフェストファイルを持っています。 これはPackage.appxmanifestファイルです。 グラフィカルエディタとXMLコードの両方の形式で編集できます。 画像エディタのスクリーンショットは以下をご覧ください。

トリガーとバックグラウンドジョブ



.Netアプリケーションは、実行可能ファイルでもサービスでもかまいません。 これらはまったく異なる種類のアプリケーションです。 つまり、exeアプリケーションはそうではないかもしれませんが、同時にバックグラウンドで動作します。 いいえ、もちろん、アプリケーションはトレイで動作します。 しかし実際には、実行されており、最小化されていることがわかります。



8.xおよびUWPアプリケーションに関しては、バックグラウンドジョブを含めることができます。 バックグラウンドジョブはちょっとしたサービスです。 つまり、アプリケーションは動作しない可能性がありますが、システムでいくつかのタスクが実行されます。 さらに、バックグラウンドタスクは、システムの操作中の一部のイベントをトリガーとして「キャッチ」できます。



最も人気のあるトリガーの1つはSystemTriggerです。 これを使用すると、アプリケーションは、インターネットの出現や喪失、ネットワークの状態の変化、ユーザーの接続または切断、SMSの受信、タイムゾーンの変更などのイベントが発生したときに、任意のコードを実行できます。



TimeTriggerとMaintenanceTriggerも非常に人気があります。 両方のトリガーは、一定の時間間隔で任意のコードを実行します。 時間間隔は少なくとも15分でなければなりません。 違いは、TimeTriggerではロック画面にアプリケーションを登録する必要があり、MaintenanceTriggerではデバイスがバッテリーではなくネットワークから動作する必要があることです。



UWPには多くの新しいトリガーがあります。 たとえば、 MediaProcessingTriggerなどの興味深いトリガーを考えてみましょう。これにより、アプリケーションはバックグラウンドタスクの一部としてマルチメディアをトランスコードできます。



ライブラリを使用する



従来のアプリケーションでDLLを使用していた場合、8.xおよびUWPアプリケーションでは、PCLとWinMDランタイムコンポーネントの両方を使用できます。 違いは何ですか？



PCL（ポータブルクラスライブラリ）は、さまざまなプラットフォームのアプリケーションに追加できます。 そして、さまざまなバージョンの.Net Framework、およびWindows 8.xおよびWP、UWP、さらにはiOS / AndroidアプリケーションXamarinの下で。 つまり、一般的なプラットフォームに依存しないコードをこのライブラリにプッシュできます。



WinMDは、8.xまたはUWPでのみ使用できます。 アプリケーションが記述されている言語に関係なく、WinMDと連携できます。 しかし、WinMD自体に複雑な計算が含まれる場合、最高のパフォーマンスを実現するにはC ++で記述する方が適切です。



ただし、UWP用に開発する場合、クラスライブラリ（DLL）を作成できます。



データを操作する



UWPアプリケーションの違いは、データベースと直接連携しないことです。 つまり、組織のサーバーにあるSQL ServerやOracleなどのデータベースは使用できません。 ただし、ユーザーがストアからアプリケーションをダウンロードし、ローカルネットワーク上のサーバーにあるSQL Serverデータベースでアプリケーションが動作し始めると、奇妙なことになります。 ただし、Webサービスを使用してデータを操作できます。 MySQLデータベースにOracle Connector / Netを使用することは可能ですが、現在SSLをサポートしていないため、特に興味深いものではありません。 そのため、データにアクセスするためにサービスを使用するという概念から逸脱しないことをお勧めします。



SQLiteを使用して、アプリケーション内に情報を保存できます。



アプリケーション設定の保存とファイルの操作



アプリケーションパラメータの保存は、デバイスだけでなくクラウドでも可能です。 したがって、アプリケーションをさまざまなデバイスで実行すると、設定はどこでも同じになります。



次の小さなスニペットは、クラウド内のコード呼び出しの数を保存します。 



int timescount = 0; Object roamS = Windows.Storage.ApplicationData.Current.RoamingSettings.Values["times"]; if (roamS != null) timescount = (int)roamS; timescount++; Windows.Storage.ApplicationData.Current.RoamingSettings.Values["times"] = timescount;


Windows.Storage.ApplicationData.Current.RoamingSettingsをWindows.Storage.ApplicationData.Current.LocalSettingsに置き換えると、パラメーターはデバイスにローカルに保存されます。



設定は、複合パラメーターとコンテナーの両方に組み合わせることができます。 設定と同じ方法でファイルをローカルフォルダーとクラウドの両方のデバイスに保存できます。 ただし、これに加えて、一時フォルダーにファイルを保存することもできます。このフォルダーは、必要に応じて、システム（ ApplicationData.TemporaryFolder）でクリアできます 。



さらに、アプリケーションに含まれているフォルダーにアクセスすることができます

Windows.ApplicationModel.Package.Current.InstalledLocation



ディスクに保存されているファイルへのアクセスも、特別なモデルに従って編成されています。 ドキュメント、写真、ビデオなどのフォルダーのコンテンツは、KnownFoldersクラスを使用して取得できますが、この場合、マニフェストでアクセス許可を設定する必要があります。 他のフォルダへのアクセスは、ユーザーがアプリケーションで作業中に自分でフォルダを選択した場合にのみ可能です。 アクセスしたフォルダは保存できるため、アプリケーションを再起動するときに、ユーザーに不要なアクションを強制することはありません。 



  var folderPicker = new Windows.Storage.Pickers.FolderPicker(); folderPicker.FileTypeFilter.Add(".jpg"); folderPicker.FileTypeFilter.Add(".jpeg"); folderPicker.FileTypeFilter.Add(".png"); folderPicker.SuggestedStartLocation = Windows.Storage.Pickers.PickerLocationId.PicturesLibrary; folderPicker.SettingsIdentifier = "picker2"; Windows.Storage.StorageFolder lastFolder = await folderPicker.PickSingleFolderAsync(); if (lastFolder == null) return; String mruToken = Windows.Storage.AccessCache.StorageApplicationPermissions.MostRecentlyUsedList.Add(lastFolder);


その後、次のように最後に保存されたフォルダーを取得できます。 



 String mruFirstToken = StorageApplicationPermissions.MostRecentlyUsedList.Entries.FirstOrDefault().Token; lastFolder = await StorageApplicationPermissions.MostRecentlyUsedList.GetFolderAsync(mruFirstToken);

有効ピクセル



UWPアプリケーションを作成することにより、物理的なピクセルではなく、有効なピクセルで開発することになります。 特別なスケーリングアルゴリズムを使用すると、24ピクセルのフォントサイズが大きなPC画面と電話画面の両方で等しく読み取れることを保証できます。







したがって、開発者は、さまざまなデバイスのピクセル密度や表示距離を気にかけないことがあります。 しかし、彼は高解像度の画像を備えたデバイスで見栄えを良くする必要があります。 作成したばかりの標準的なユニバーサルアプリケーションのAssetsフォルダーを見てみましょう。







一部の画像には、.scale-200の接尾辞が付いていることがあります。 これは、スケーリングサイズが200に設定されていることを意味します。たとえば、ファイルSquare44x44Logo.scale-200.pngを使用すると、その物理サイズは88x88ピクセルになります。 これは200％のスケールです。 利用可能なオプションはすべて、100、125、150、200、250、300、400です。



例：ズームファクター150のデバイス用の画像バリアントを作成する場合、サイズ66x66の画像を作成し、Square44x44Logo.scale-150.pngという名前を付ける必要があります。



これは、アプリケーションの説明で使用される画像だけでなく、UIで使用する画像にも適用されます。
