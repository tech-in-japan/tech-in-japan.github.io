エントリー



この記事は、GCCコンパイラーでCode :: Blocks開発環境を使用している場合に読むべき最初の記事です（この開発環境の標準コンパイラーです）。 プロジェクトのセットアップ方法を説明します。



SFMLをインストールする



最初に、 ダウンロードページからSFML SDKをダウンロードする必要があります 。



互いに互換性のないWindows用のGCCオプションがいくつかあります（例外処理、ストリームモデルなどの違い）。 使用しているGCCのバージョンに一致するパッケージを選択してください。 不明な場合は、MinGW / binディレクトリに存在するファイル（libgcc_s_sjlj-1.dllまたはlibgcc_s_dw2-1.dll）を確認してください。 MinGWがCode :: Blocksでインストールされている場合、おそらくSJLJバージョンがあります。 GCCのバージョンがプリコンパイルされたバージョンのSFMLライブラリで動作しない場合、 SFMLを自分でコンパイルする必要があります 。 難しくありません。



次に、SFMLアーカイブを自分に都合の良いディレクトリに解凍する必要があります。 ヘッダーファイルとライブラリをMinGWインストールにコピーすることはお勧めしません。 特に同じライブラリの複数のバージョンまたは複数のコンパイラを使用する場合は、ライブラリを別の場所に保管することをお勧めします。



SFMLプロジェクトの作成と設定



最初に行う必要があるのは、作成するプロジェクトのタイプを選択することです。 コード::ブロックは、SFMLプロジェクトを含む幅広いプロジェクトタイプを提供します。 使用しないでください！ このタイプのプロジェクトはかなり長い間更新されておらず、SFMLの最新バージョンと互換性がない可能性があります。 代わりに、空のプロジェクトを作成します。 コンソールを削除する場合は、プロジェクトプロパティの[ビルドターゲット]に移動し、コンボボックスで[コンソールアプリケーション]ではなく[GUIアプリケーション]を選択します。



次に、ヘッダーファイル（拡張子が.hppのファイル）を探す場所をコンパイラーに、SFMLライブラリ（拡張子が.libのファイル）を探すリンカーを指示する必要があります。



[ディレクトリの検索]タブの[ビルドオプション]タブに次を追加します。



  • コンパイラ検索ディレクトリ内のSFMLヘッダーファイルへのパス （ <install-SFMLへのパス> / include ）
  • リンカ検索ディレクトリ内のSFMLライブラリへのパス（ <install-SFMLへのパス> / lib ）


これらのパスはデバッグ構成とリリース構成で同じであるため、プロジェクトに対してグローバルに設定できます。







次のステップは、SFMLライブラリ（拡張子が.aのファイル）を使用してアプリケーションを構築することです。 SFMLは、5つのモジュール（システム、ウィンドウ、グラフィックス、ネットワーク、およびオーディオ）とそれぞれのライブラリで構成されています。 ライブラリは、プロジェクトのビルドオプションのリストの[リンカー設定]タブにある[リンクライブラリ]に追加する必要があります。 「sfml-graphics」、「sfml-window」、「sfml-system」など、必要なすべてのSFMLライブラリを追加します（プレフィックス「lib」と拡張子「.a」は省略してください）。







設定に一致するライブラリを指定することが重要です：デバッグ用の「sfml-xxx-d」およびリリース用の「sfml-xxx」。そうしないとエラーが発生する可能性があります。



SFMLライブラリをリンクするときは、ライブラリを正しい順序でリンクしてください;これはGCCにとって非常に重要です。 ルールは、他のライブラリに依存するライブラリをリストの最初に配置することです。 すべてのSFMLライブラリは、sfml-graphics、sfml-window、およびsfml-systemに依存しています。 これは上のスクリーンショットに示されています。



上記の設定を使用すると、DLLファイルを必要とするSFMLの動的バージョンを使用してプロジェクトをビルドできます。 動的ライブラリレイアウトを使用するのではなく、SFMLを実行可能ファイルに直接統合する場合は、ライブラリの静的バージョンをビルドする必要があります。 SFML静的ライブラリの接尾辞は「-s」です。デバッグ構成の場合は「sfml-xxx-sd」、リリースの場合は「sfml-xxx-s」です。



また、プロジェクトのプリプロセッサオプションでSFML_STATICマクロを定義する必要があります。







静的リンクのSFML 2.2以降では、すべてのSFML依存関係も構築する必要があります。 これは、たとえば、sfml-window-sまたはsfml-window-sdをビルドする場合、opengl32、winmm、およびgdi32もビルドする必要があることを意味します。 これらのライブラリの一部は、既に「継承値」セクションにリストされている場合がありますが、追加しても問題は発生しません。

いくつかのSFMLモジュールは他のモジュールに依存している可能性があることに気づいたかもしれません。たとえば、sfml-graphics-sはsfml-window-sおよびsfml-system-sに依存しています。 SFMLライブラリを静的に構築する場合は、依存関係チェーン内のすべての依存関係が満たされていることを確認してください。 依存関係の1つが欠落している場合、リンカーエラーが発生します。



少し混乱している場合でも、心配しないでください。初心者がこの静的リンク情報のすべてで過負荷になるのは完全に普通です。最初にうまくいかない場合は、上記のすべてを念頭に置いてもう一度試してください。 それでも静的リンクに問題がある場合は、 FAQセクションまたはフォーラムで解決策を探してみてください 。



動的（共有とも呼ばれる）ライブラリと静的ライブラリの違い、および使用するライブラリのタイプがわからない場合は、インターネットで詳細を確認できます。 このテーマに関する多くの優れた記事/ブログ/投稿があります。



プロジェクトの準備ができました。すべてが正しく機能していることを確認するコードを作成しましょう。 main.cppファイルに次のコードを追加します。 



 #include <SFML/Graphics.hpp> int main() { sf::RenderWindow window(sf::VideoMode(200, 200), "SFML works!"); sf::CircleShape shape(100.f); shape.setFillColor(sf::Color::Green); while (window.isOpen()) { sf::Event event; while (window.pollEvent(event)) { if (event.type == sf::Event::Closed) window.close(); } window.clear(); window.draw(shape); window.display(); } return 0; }




ここでプロジェクトをコンパイルします。SFMLの動的バージョンでプログラムをコンパイルした場合は、拡張子.DLLのファイル（<path-to-install-SFML> / binにある）をプログラムの実行可能ファイルがあるディレクトリにコピーすることを忘れないでください。 プログラムを実行し、すべてが正しく行われた場合、これが表示されるはずです。







sfml-audioモジュールを使用する場合（静的であるか動的であるかに関係なく）、外部DLLライブラリOpenAL32.dllもコピーする必要があります。 このファイルは、 <SFMLのインストールパス> / binディレクトリにもあります。
