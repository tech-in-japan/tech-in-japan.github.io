Android SMSブラックリスト開発

こんにちは、愛するhabrasociety！



私の質問への回答に基づいて、このトピックを公開します。 「アンチスパム」は、この段階では送信者のブラックリストにすぎないため、作成されたアプリケーションのやや知名度の高い名前です。 ただし、将来的には自動フィルタリングで本当にスパム対策を行う予定です。 以下の資料は、Androidの開発に少なくとも少し精通しており、独自のアプリケーションを開発するために少なくともいくつかの手順を踏んだ人向けに設計されています。重要なポイント。 誰も気にしない、猫へようこそ。





どのように始まったのか覚えていますか？



少し叙情的な余談。 それはすべて、どこかで携帯電話番号を漏らしたという事実から始まりました。 私と同じサイトに登録した友人が同じスパムメッセージを同時に受信するため、私は正確にどこにいるのかさえ疑います。 しかし、今はそれについてではありません。 金曜日に、私はさまざまなクラブから広告を受け取り始め、毎週毎週メッセージの流れが増えました。 私はAndroid OSベースの電話の所有者になったので、この不名誉に対処するアプリケーションを作成することにしました。 プログラマーは簡単な方法を探すことはありません。モバイルオペレーターに連絡したくありませんでした。さらに、Android向けの開発を開始したいとずっと思っていました。



アプリケーションのアーキテクチャとレイアウト



アプリケーションは3つの部分で構成されています。

  1. 実際には、SMSメッセージを受信して​​フィルタリングするフィルター。
  2. 送信者と送信者から受信したメッセージのブラックリストを保存するデータベース。
  3. ユーザーインターフェース


どのように機能しますか？ ユーザーは、受信トレイまたは連絡先から送信者のブラックリストに追加します。 フィルターをオンにして、アプリケーションを閉じます。 メッセージを受信すると、アプリケーションは送信者を分析し、ブラックリストに載っている場合、いわゆるリポジトリにメッセージを配置して、ユーザーがフィルターされたメッセージを後で見ることができるようにします。 送信者が「クリーン」の場合、SMSは「受信ボックス」フォルダーにあります。



最初のプロトタイプ



すぐに言ってやった。 しかし、最初のプロトタイプの実装は1日も続きませんでした。 実際、私はテキストメッセージを受信するイベントをキャッチしようとしました。このイベントのハンドラーでは、送信者がブラックリストに載っているすべてのメッセージをInboxフォルダーから削除しました。 問題は、携帯電話の所有者にSMSの受信を知らせる信号が聞こえるということです。受信トレイを開いても、新しいものは表示されません。 そして、これはどういうわけか良くありません。 そのため、受信トレイに入る前であっても、メッセージを傍受する方法を探し始めました。



メッセージの傍受



Android OSは、SMSの受信、WI-FIのオン、接続の充電などのイベントが、ブロードキャストを介してアプリケーションに通知するように設計されています。 このメカニズムとAndroidの一般的なアーキテクチャの詳細については、 こちら（4つの翻訳）をご覧ください 。 このようなシステムブロードキャスト用のレシーバー（リスナー）を作成するには、次のことを行う必要があります。

  1. BroadcastReceiverから継承した独自のクラスを作成し、onReceiveメソッドをオーバーロードします。
  2. システムに登録し、どのタイプのニュースレターを受け取りたいかを示します。




リスナーを作成する 



public class SMSReceiver extends BroadcastReceiver { @Override public void onReceive(Context context, Intent intent) { Bundle bundle = intent.getExtras(); Object[] pdus = (Object[]) bundle.get("pdus"); if (pdus.length == 0) { return; } Sms sms = Sms.fromPdus(pdus, context); SmsDatabase db = null; try { db = SmsDatabase.open(context); if (Filter.filter(context).isSpam(sms, db)) { abortBroadcast(); db.messages().save(sms); } } finally { if (db != null) db.close(); } } }




onReceiveメソッドは、SMSメッセージが電話に到着するたびにトリガーされます。 intentパラメーターから必要な情報をすべて抽出できます。 注意事項によると、ドキュメントによると、BroadcastReceiverから継承されたクラスは、onReceiveメソッドの実行中にのみ関連します。 これは、指定されたメソッドの実行が終了するとすぐに、システムがクラスのインスタンスを破棄できることを意味します。 これは、クラスの非静的フィールドに情報を保存しないことも意味します。



最初の2行では、 PDU情報を取得します。 大まかに言えば、これは「生の」形式のSMSです。 空かどうかを確認した後、Sms自己記述クラスの静的fromPdus（）メソッドを使用してメッセージ情報を抽出しようとします。これについては後述します。



次に、Filterクラスを使用して、受信したSMSメッセージの送信者がブラックリストに載っているかどうかを確認します。 データベースにメッセージを保存し、abortBroadcast（）メソッドを使用してブロードキャストを中断することがわかった場合。 これは、SMSに関する通知を受信するために登録された優先度の低いすべての受信者は、そのようなイベントが発生したことすら知らないことを意味します。 スパムメッセージを受信した場合にユーザーの邪魔にならないように、受信者に最高の優先順位を設定します（デバイスで音を立てて振動する受信者よりも高い）。 以下の優先順位についてお読みください。



アプリケーションの以前のバージョンでは、onReceiveメソッドで、データベースへの接続が2回開かれました。1回目はメッセージをチェックするときのFilterクラスで、2回目はSMSをデータベースに書き込むときです。 ただし、onReceiveメソッドの実行時間は10秒に制限されており、2つの接続を連続して開くことには意味がないため、このアプローチを放棄し、コードの「美しさ」の観点からコードを「もう少し不正確」にしました。 結局、メソッドが割り当てられた時間を満たしていない場合、Androidは次の受信者のメソッドを呼び出し、ユーザーはSMSの受信について通知されます。



リスナー登録



リスナーを作成しました。 彼のシステムを登録することは残っています。 これを行うには2つの方法があります。

  1. プログラムでregisterReceiver（）メソッドを使用します。 この場合、受信者は、それを登録したコンポーネントが生存している間のみ生存します（通常、これはアクティビティです）。
  2. AndroidManifestを使用します。 この場合、アプリケーションが閉じられていても、さらに電話が再起動されても、受信者は生き続けます！




明らかに、2番目のオプションの方が受け入れられます。 実装方法を見てみましょう。 

 <application android:label="@string/app_name" android:icon="@drawable/app_icon"> …………….    ………………….. <receiver android:name=".SMSReceiver" android:enabled="false"> <intent-filter android:priority="1000"> <action android:name="android.provider.Telephony.SMS_RECEIVED" /> </intent-filter> </receiver> </application>




ここではすべてが非常に簡単です。 受信者クラスの名前（android：name）を指定し、次にintent-filterタグを使用して優先度（android：priority、1000が最大値です。振動して音を出す標準受信者の場合、優先度は999）と購読するイベント（ android.provider.Telephony.SMS_RECEIVED）。



リスナーのオンとオフを切り替える



受信者はデフォルトでオフになっていると言う必要があります。 その結果、ユーザーはSMS受信者をアクティブまたは非アクティブにすることにより、フィルターをオンまたはオフにすることができます。 Filterクラスには、これに対応するメソッドon（）、off（）、enabled（）があります。



ちなみに、Filterクラスはシングルトンクラスです。コンテキストに依存するという事実にもかかわらず、アプリケーション全体に対して1つのフィルターが必要になるためです。 SMSReceiverクラスでは、静的フィルター（）メソッドを介してフィルターインスタンスにアクセスする方法を確認できます。このメソッドは、Contextクラスのインスタンスをパラメーターとして受け取ります。 



 private static Filter _filter; public static Filter filter(Context context) { if (_filter == null || !_filter._context.equals(context)) { _filter = new Filter(context); } return _filter; }




そのような実装が唯一の真の実装であると主張することはありません。 より良いソリューションを提供できる場合は、喜んで検討します。



Filterクラスのコンストラクターで、作業に必要なデータを初期化します。 

 private final ComponentName componentName; private final PackageManager packageManager; private Context _context; private Filter(Context context) { _context = context; String packageName = context.getPackageName(); String receiverComponent = packageName + ".SMSReceiver"; componentName = new ComponentName(packageName, receiverComponent); packageManager = context.getPackageManager(); }




componentNameは、アプリケーションコンポーネント（この場合は受信者）の完全修飾名で、パッケージ名とクラス名（SMSReceiver）が含まれます。

packageManager-名前から、これがアプリケーションコンポーネントを管理するためのクラスであることは明らかです。



フィルターを含むメソッドを考えてみましょう： 

 public void on() { if (!enabled()) { packageManager.setComponentEnabledSetting(componentName, PackageManager.COMPONENT_ENABLED_STATE_ENABLED, PackageManager.DONT_KILL_APP); } else { Util.showMessage(_context.getString(R.string.alreadyStarted), _context); } }




ここではすべてが簡単です。 コンポーネントが既に有効になっている場合（enabled（）は自己記述メソッドです。少し後で検討します）、ユーザーに通知し、オフになっている場合はオンにします。 Util静的クラスは自己記述型であり、さまざまなヘルパー関数が含まれています。 この場合、showMessageメソッドは標準のToastクラスを使用して画面にメッセージを表示します。



フィルターを無効にするoff（）メソッドは、PackageManager.COMPONENT_ENABLED_STATE_DISABLEDフラグが使用され、フィルターが既にオフになっている場合、対応するメッセージが表示されることを除いて、on（）メソッドと完全に似ています。



フィルターの状態をチェックするメソッドはさらに単純に見えます。 

 public boolean enabled() { int enabled = packageManager.getComponentEnabledSetting(componentName); return (enabled == PackageManager.COMPONENT_ENABLED_STATE_ENABLED); }




シラミの確認



Filterクラスには、まだ説明されていないメソッドがもう1つ残っています。 これはisSpam（）メソッドで、実際にはアプリケーションのメインタスクを実行します。 この方法は非常に簡単です。 彼はSMSから送信者メッセージを抽出し、データベースで彼を見つけようとします。 ある場合、メッセージはスパムと見なされます。 



 public boolean isSpam(Sms sms, SmsDatabase db) { Sender sender = sms.sender(); return db.senders().exists(sender); }




SMSクラス



Smsクラスはすでに2回言及されています。 彼についてもう少し話す必要があります。 このクラスは、アプリケーション内のSMSメッセージを表します。 次のフィールドが含まれます。 



 private String _body; private Sender _sender; private long _timestamp;




_bodyはメッセージの本文、_senderはメッセージの送信者、_timestampはメッセージがUNIXタイムスタンプとして受信された時刻です。



送信者クラスはAndroidの標準ではありません。 手書きで作成され、送信者の電話を保存できます。また、必要に応じて、電話帳から送信者の名前を抽出できます。 詳細は検討しません。



Smsクラスには、PDUからメッセージ情報を取得する静的fromPdus（）メソッドがあることがすでにわかっています。 そのコードを見てみましょう： 

 public static Sms fromPdus(Object[] pdus, Context context) { Sms result = new Sms(); for (int i = 0; i < pdus.length; i++) { SmsMessage sms = SmsMessage.createFromPdu((byte[]) pdus[i]); result._body += sms.getMessageBody(); } SmsMessage first = SmsMessage.createFromPdu((byte[]) pdus[0]); result._sender = new Sender(first.getOriginatingAddress(), context); result._timestamp = first.getTimestampMillis(); return result; }




一見すると、この方法は複雑に見えるかもしれません。 しかし、これはそうではありません。 唯一の難点は、1つの論理SMSメッセージを複数の物理的なSMSメッセージに分割できることです。 この点で、私はあなたのためにアメリカを発見したことはないと思います。 この特殊効果は、できるだけ多くの情報を1つのSMSに押し込んでお金を節約するために、常にスペースなしで書き込みを行うすべての人によく知られています。



したがって、サイクルでは、メッセージ本文を断片的に収集し、最初のSMSメッセージから必要な情報を抽出します。 一般に、Smsクラスを使用してこれを終了できます。



データベースについて少し



慢なスパマーのブラックリストと認識されていない作業の結果がデータベースに保存されます。 これは、ユーザーがそこで好きなものを見ることができるようにするために必要です。 理論的には、ユーザーは連絡先から電話番号をブラックリストに追加することができ、所有者と仲直りした後、受信したSMSを読むことができます。



幸いなことに、Androidにはすぐに使えるSQLite DBMSがあります。 標準のJAVAクラスとSQLクエリを使用して、データベースを操作するロジック全体を実装しました。 意地悪の法則によれば、アプリケーションのこの部分を終えた後、HabréのAndroid向けORMに関する記事を読みました 。



私は、接続を開く-データを操作する-接続を閉じるという原則に基づいてデータベースを操作しました。 このDBMSはファイルシステムであり、読み取りまたは書き込み操作中にブロッキングが発生する可能性があるため、この方法はSQLiteに特に適しています。 そのため、長時間の接続を維持しないようにします。



データベースのすべてのラッパーメソッドを検討したくはありません。多くのメソッドがあり、記事はゴムではないからです。 さらに、これらはすべて同じ型であるため、本質を理解するために、SMSReceiverクラスのonReceiveメソッドで使用したメソッドのみを検討します。



エントリ「db.messages（）。Save（sms）」は、db接続の一部として、フィルター処理されたメッセージを格納するメッセージテーブルに移動し、save（）メソッドを使用して新しいsmsメッセージを追加することを示しています。 すべてのクラスとメソッドは自己記述型であることを繰り返します。 



 public void save(Sms sms) { if (!Preferences.get(_context).storeSms()) return; int maxCount = Preferences.get(_context).storedSmsAmount(); if (count() >= maxCount) trim(maxCount - 1); SQLiteStatement insert = _db.compileStatement( "INSERT INTO Messages(Phone, Timestamp, Body)" + "VALUES (?, ?, ?)"); insert.bindString(1, sms.sender().phone()); insert.bindLong(2, sms.timestamp()); insert.bindString(3, sms.body()); insert.execute(); }




コードの最初の行では、アプリケーション設定にアクセスしてフラグを確認します。これは、フィルター処理されたメッセージを保存するユーザーの要望に対応します。 ユーザーがストレージを拒否した場合は、終了します。



次に、再び設定から、ユーザーが設定したフィルター処理されたSMSストレージのサイズを抽出します。 また、現在のメッセージ数がこのサイズを超える場合、受信した順序を考慮して、古いメッセージをリポジトリから削除します。 つまり、最初に受信したメッセージが最初に削除されます。 クラシックラインナップ。



これらのジェスチャの後、Messagesテーブルにデータを挿入するリクエストを作成し、型付きbind（）メソッドを使用して、生成されたリクエストにデータをバインドします。 最後に、リクエストを実行します。



テストとデバッグ



すべて準備ができているようです。 小さなステップが残っています。 問題は、アプリケーションが正しく機能していることをどのように確認するかです。 受信者は少なくとも何かを受け取りますが、データベースは何かを保存しますか？ これは別の大きなトピックのトピックであるため、Androidの単体テストアプリケーションの機能については説明しません。 エミュレータでSMSの受信をシミュレートする方法のみを説明します。



これを行うには、Telnetクライアントを開きます。 気分に応じて、Windows 7でPuttyまたは標準クライアントを使用します（最初に有効にする必要があります）。 localhostに接続し、ポートとしてエミュレータのウィンドウタイトルに表示されるポートを指定します。 原則として、これは5554です。言うまでもなく、同時にエミュレータを実行する必要があります。 挨拶を読んだ後、コマンドの入力を開始できます。



多くのチームがあります。 それらのリストは、helpコマンドを入力して取得できます。 これらのコマンドを使用して、電話機で発生する可能性のある多くのイベントをシミュレートできます。 ただし、SMSメッセージの受信のシミュレーションにのみ関心があります。



これをシミュレートするには、単にコマンドを入力します 

 sms send _ _


ここで、phone_numberは「+」記号で始まる可能性のある数字のシーケンスであり、message_textは、通常SMSで記述するスペースやその他の文字を含むメッセージコンテンツです。



シミュレーションが成功した場合、telnetクライアント端末に「OK」というメッセージが表示されます。
