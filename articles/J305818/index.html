Visual Studio 2015でAngular2を起動します





少し前まで、最初のAngular.jsから2番目に作成されたアプリケーションを移行する問題に興味がありました。 ちなみに、この記事はそれについてのはずです。 しかし、事件は介入した。 その前に、Angular2をNode.jsでのみ調達しました。 そして、ここでは、主にVisual Studioで作業しているため、その下から実行することにしました。 私が経験しなければならなかったすべてのレーキヒットが終わった後、気がついたとき、別の記事でVisual Studio 2015のAngular2展開を分離することにしました。 そして、それは余分なものではないということを教えてくれます。



最初のステップ。 小道具。



最初に、Angular2が依存するほとんどすべてのパッケージがNPMにあるため、Node.jsおよびNPMで動作するようにVisual Studioを準備する必要があります。



NPM Task RunnerとPackage Installerを配置しました 。 これらは、スタジオとnpmの相互作用に役立ちます。



npmとwindowsといえば。
結局のところ、ウィンドウの軸は非常に独創的です。 Node.jsをインストールすると、npmも自動的にインストールされます。 ただし、その後にnpmをグローバルに（-gフラグを使用して）インストールすると、ほぼ確実に他の場所にインストールされます。



そして（ここではウィザードがウサギを取り出します）使用されません。

問題は、Node.jsのインストール中に、Nodeインストーラーが付属のnpmへのパスをPATH環境変数に書き込むことです。 したがって、コンソールの下からnpmを呼び出すと、グローバルnpmではなく、具体的にアクセスできます。



この奇妙な動作を修正するには、以下が必要です



グローバルがインストールされた場所を見つける（npm root -gコマンドを実行できます）

PATHで指定されたパスをnpmノードのnpmグローバルに置き換えます。 システムとユーザーの両方でこれを行うことを忘れないでください。また、マシンを再起動してください。


拡張機能をインストールした後、スタジオを実行して空のWebプロジェクトを作成します。 次に、Visual Studioで使用されるnode.jsとnpmを更新する必要があります。 スタジオはグローバルnpmとnode.jsを使用せず、ローカルを使用します。 彼女は、システムにインストールされているものについて何も知らず、外部Webツールにあるnode.jsに依存しています。 これがわからない場合、スタジオは古いコンポーネントを使用し、アプリケーションの起動に失敗します。 そのため、プロジェクトをクリックして、クイックインストールパッケージを選択します。 表示されるウィンドウで、gulpと入力して設定します。 もちろん、これは絶対に必要というわけではありませんが、率直に言って、私はpackage.jsonを手動で作成するのが面倒です。

gulpをインストールすると、package.jsonが取得されます。これをさらに操作できます。



そして、最初に行う必要があるのは、スタジオで使用するnode.jsとnpmのバージョンを確認することです。



「npm -v」および「node -v」コマンドを使用してスクリプトセクションをpackage.jsonに追加し、タスクrunner-aで実行します。



package.json 
{ "name": "myproject", "version": "1.0.0", "devDependencies": { "gulp": "^3.9.1" }, "scripts": { "getNpmVersion": "npm -v", "getNodeVersion": "node -v" } }




個人的に、私の結果は印象的でした：ノードv0.10.31とnpm 1.4.9。 繰り返しになりますが、これらはVisual Studioが使用するバージョンであり、インストールされたノードもグローバルnpmもこれらとは何の関係もないという事実に注目します。



さて、このジャンクはすべて整理整頓する必要があります。 package.jsonに新しいコマンド「updateNpm」：「npm install npm @ latest」を追加して実行します。 少し待ってから、getNpmVersionを実行します。 Npmは更新され、バージョン3.10.5になりました。



ただし、node.jsでは、このアプローチは機能しません。 正直に言うと、node.jsを更新する方法を見つけられませんでしたが、必要なインスタンスをスタジオに使用させる方法を見つけました。



そのため、まず、node.jsがインストールされている場所を見つけて、そこにパスをコピーします。 次に、[ツール]-> [オプション]-> [プロジェクトとソリューション]-> [外部Webツール]に移動し、ノードの通常バージョンを指す新しいパスを追加します（ない場合は、ダウンロードしてインストールします）。 最上部への新しい道を歩むことを忘れないでください。 スタジオを再起動し（Windowsを再起動する必要はありません）、getNodeVersionコマンドを実行します。 出来上がり、バージョンが更新されます。



タンバリンとのこれらすべてのダンスの後、私はコーヒーに行くことを提案します。 さらに、それも簡単ではありません。



ステップ2 依存関係のインストール。



私たちのproject.jsonを苦しめ続けましょう。 次に、アプリケーションのすべての依存関係を入力します。 依存関係は、ウィンドウに合わせて編集された5 MIN QUICKSTARTと 、小さいながらも非常に誇らしいニュアンスから取得されます。



可能性のあるニュアンス。
Angular2はtypescript上に構築されています。 （先日だけでしたが、クイックスタートは純粋なJavaScriptで始まりました）。 したがって、hello worldアプリケーションの依存関係の1つは、「typescript」のようなtypescriptへの依存関係です：「^ 1.8.10」。 これは、モジュールをロードするときに、typescriptの最新バージョンがロードされることを意味しますが、1.8.10以上です。 突然、クイックスタートが提供するアセンブリは、エクスポート中に表示されるデフォルトのキーワードにより、最新のタイプスクリプトをサポートしません。 少なくともこれは私のバージョンです。 重要なことは、最新バージョンでは、angular2が「離陸」しないことですが、それを修正して1.8.10だけを指定すれば、すべてが問題ないはずです。

バグはすでに送信されており、結果が期待されています。



package.json 
 { "name": "myproject", "version": "1.0.0", "devDependencies": { "gulp": "^3.9.1", "typescript": "1.8.10", "typings": "^1.0.4" }, "dependencies": { "@angular/common": "2.0.0-rc.4", "@angular/compiler": "2.0.0-rc.4", "@angular/core": "2.0.0-rc.4", "@angular/forms": "0.2.0", "@angular/http": "2.0.0-rc.4", "@angular/platform-browser": "2.0.0-rc.4", "@angular/platform-browser-dynamic": "2.0.0-rc.4", "@angular/router": "3.0.0-beta.1", "@angular/router-deprecated": "2.0.0-rc.2", "@angular/upgrade": "2.0.0-rc.4", "systemjs": "0.19.27", "core-js": "^2.4.0", "reflect-metadata": "^0.1.3", "rxjs": "5.0.0-beta.6", "zone.js": "^0.6.12", "angular2-in-memory-web-api": "0.0.14" }, "scripts": { "postinstall": "typings install", "typings": "typings", "cmd": "npm typescript", "getNpmVersion": "npm -v", "getNodeVersion": "node -v" } }




次に、ランナーのタスクの下で、インストールコマンドを実行し、すべてのパッケージがインストールされるまで待ちます。 すべてがうまくいけば、node_modulesフォルダーに他のフォルダーがいくつかあるはずです。その中に「@angular」フォルダーがあります。 表示されている場合、これまでのところすべてが順調に進み、依存関係がインストールされています。 そうでない場合は？ cachem npmの問題が発生する場合があります。 パッケージがインストールされておらず、ランナータスクコンソール全体がエラーで赤くなっている場合は、package.jsonに別のコマンド「npm cache clean」を追加して実行してください。 たぶんこれが役立つでしょう。



ステップ3 Visual Studioのtypescriptを構成します。



タンバリンダンスでさらにいくつかのステップを行う番でした。

最初の.tsファイルを追加しましょう。 ルートにappフォルダーを作成し、空のapp.component.tsファイルをそこに追加します。 想像力を発揮しないで、自分の名前を考えてください。 くしゃみでもすべてが壊れる可能性があります。 スタジオを閉じます。



Visual StudioとTypeScript！ TypeScriptとVisual Studio
すべてのまともな家では、タイプスクリプトの設定は特別なファイルで設定されています-tsconfig.json。 しかし、Visual Studioは気まぐれな女性です。 CommonJSモードでtypescriptを使用する場合（そして、その中で正確に動作する場合）、tsconfig.jsonはスタジオを無視します。 一方、私たちにとって非常に重要な2つのオプションがあります-「emitDecoratorMetadata」：true、

「ExperimentalDecorators」：true。 そのため、これらのフラグは.csprojファイルでペンで設定されます...詳細はこちらをご覧ください



オプションのより完全なセット 
 <PropertyGroup Condition="'$(Configuration)' == 'Debug'"> <TypeScriptRemoveComments>false</TypeScriptRemoveComments> <TypeScriptSourceMap>true</TypeScriptSourceMap> <TypeScriptTarget>ES5</TypeScriptTarget> <TypeScriptJSXEmit>None</TypeScriptJSXEmit> <TypeScriptCompileOnSaveEnabled>True</TypeScriptCompileOnSaveEnabled> <TypeScriptNoImplicitAny>False</TypeScriptNoImplicitAny> <TypeScriptModuleKind>System</TypeScriptModuleKind> <TypeScriptOutFile /> <TypeScriptOutDir /> <TypeScriptGeneratesDeclarations>False</TypeScriptGeneratesDeclarations> <TypeScriptNoEmitOnError>True</TypeScriptNoEmitOnError> <TypeScriptMapRoot /> <TypeScriptSourceRoot /> <TypeScriptExperimentalDecorators>True</TypeScriptExperimentalDecorators> <TypeScriptEmitDecoratorMetadata>True</TypeScriptEmitDecoratorMetadata> </PropertyGroup>






スタジオを閉じると、* .csprojファイルが見つかり、PropertyGroupノードに2つの新しい行が追加されます。 



 <TypeScriptExperimentalDecorators>true</TypeScriptExperimentalDecorators> <TypeScriptEmitDecoratorMetadata>True</TypeScriptEmitDecoratorMetadata>


ここでプロジェクトを開き、プロパティに移動して、新しいタブ-TypeScript Buildを確認します。

EcmaScript 6とModule System-Common Jsを設定します。 保存します。



ステップ4 コード。



これで、app.component.tsにコードの最初の行を追加できます

これをここにコピーして、プロジェクトをビルドします。



app.component.ts 
 import { Component } from '@angular/core'; @Component({ selector: 'my-app', template: '<h1>My First Angular 2 App</h1>' }) export class AppComponent { }




すべてが正常に実行された場合、ビルドはエラーなしで実行されます。 これは私たちの主要なコンポーネントです。つまり、これは私たちのアプリケーションです。



次に、起動するように設定します（ブートストラップ）。 これを行うには、ファイルをアプリフォルダーに追加します



main.ts 
 import { bootstrap } from '@angular/platform-browser-dynamic'; import { AppComponent } from './app.component'; bootstrap(AppComponent);




そして再びプロジェクトをビルドします。 繰り返しますが、エラーはありません。 もちろん、構築することはできませんが、私は何とか落ち着いています。
