質問は少年ではなく、6月です。 「ミドルパイソン開発者」の地位に関するインタビューでの雇用者への22の質問





はじめに



2年間、「ミドルパイソン開発者」の地位の候補として40以上のインタビューに参加できました。 最後の15回のインタビューで、将来の仕事の驚きに遭遇しないように、雇用主に質問する必要があることに気付きました。 候補者が通常雇用主に尋ねる基本的な質問に加えて、私は質問を作成することにしました。 インタビューでこれらの質問をしたとき、インタビュー対象者からさまざまな反応を受け取りました。 誰かが私は細心の注意を払っていると言い、誰かがこれらの問題はあまりにもありふれたものだと考え、誰かが緊張し（赤面）始め、すぐに彼が会議を開いたという不合理な言い訳でインタビューを中断しました。 この記事では、そのようなイベントに参加するための一般的なアイデアについてお話しします。また、雇用者へのインタビューで尋ねる22の質問をお伝えしたいと思います。





一般的なアイデア



中間開発者とのインタビューは、多くの場合、後輩とのインタビューと同じように見えます。



本当にそうです。 これは、多くのチームリーダー/テクニカルディレクターが、真ん中の開発者に何を見たいのか正確に知らないという事実によるものです。 したがって、そのようなインタビューでは、通常「デコレータを書く」または「任意の言語でバブルソートを書く 」ように求められます。

ジュニア開発者が中間開発者とどのように異なるかを理解している人はほとんどいません。 ミドルは1年半の経験を持つ開発者であり、3人の開発者であると誰かが言います。 私の理解では、中開発者とは、小さなプロジェクトまたは大規模なプロジェクトの一部を安全に提供できる開発者であり、その開発者が責任を負っていることです。 また、中間開発者の重要な基準は、誰かのメンターになる能力、または単に新しい従業員がプロジェクトに侵入するのを助ける能力です。



面接は試験ではなく、会社と候補者がどのように一致するかを特定する機会です。



多くの場合、この重要なルールは雇用主自身に理解されていません。 インタビューに出たら、チケットを取り、インタビューの相手に紙で答えることを余儀なくされました。 さらに、このインタビューを通して、10分間話しました。 多くの場合、同じ行動が候補者によって追跡されます。 多くの場合、候補者はすべてに答えることを望み、最初の机から優秀な学生のように振る舞います。 しかし、ここで重要なのは、雇用主が「Python2とPython3の違い」をどれだけよく知っているかについて特に興味がないことです。 面接をどのように保持するか、どのように推論するか、失敗にどのように反応するかなどを雇用主が一般的に理解することははるかに重要です。



ミドル開発者は経験なしではいられません。



もちろん、それは可能ですが、しばらくすると、この開発者のチーフとプロジェクトの両方にとって大きなトラブルが発生します。 経験のない過度に才能のある候補者のために、人事スペシャリストには独自の用語-「強力なジュニア開発者」があります。 ほとんどの場合、このような開発者には適切な金銭的報酬が提供されますが、ジュニア開発者としての責任があります。 ミドル開発者に話を戻しますが、ミドルはしばらくの間開発に携わっており、どのプロセスで構成されているかを理解している人です。 Middleは、経験のない人がトレーニング目的で遭遇する可能性が低いさ​​まざまなツール（監視、展開、プロファイリング、テスト）の使用方法も知っています。



ソフトスキルは、ミドルデベロッパーの地位における重要な要素になります。



ポジションが高いほど、より多くの人が交流する必要があります。 したがって、非常に多くの場合、中堅開発者の職を雇う際に、将来の従業員の心理的なポートレートを作成するために、人事スペシャリストとの追加のインタビューが作成されます。 このインタビューは、技術的なインタビューと同様に真剣に受け止めなければなりません。 これらの人々と働き続けることを理解する必要があります。 そして、あなたの将来の同僚があなたにあまり適していないと感じたら、それ以上の協力をすぐに拒否する方が良いです。



テストタスクは、開発者の中間のポジションに与えられる可能性は低くなります。



この声明は非常に主観的です。 個人的に、私は本当にそのような事実に出会いました。 これは、雇用主があなたの履歴書により興味を持っているという事実と結び付けます。 履歴書が適切にコンパイルされていない場合は、おそらくテストタスクを待つ必要があります。



ご質問



このセクションでは、面接時に雇用主に尋ねる質問の主なリストを提示します。 おそらくしばらくすると、このリストは拡大または縮小します。 これらの質問は、技術面接で正確に尋ねられるべきであり、後で誰とやり取りするのかを正確に確認することをお勧めします。



1.テストはどうですか？ どのテストを書いていますか？ テストに使用するライブラリは何ですか？ （ 工場 、 モキなど）



テストは、開発の非常に重要な部分です。 私の理解では、すべての開発者は、少なくとも何らかの形でテストを作成する必要があります。 テストの欠如を許すことができるのはスタートアップだけです。 スタートアップでは、通常、誰もが古いプロジェクトを必要としないため、移動の流れが変わることがあります。 そのため、そのようなプロジェクトの品質を確保することは時間の無駄でした。 他のすべての企業にとって、この問題に容赦はないはずです。 最初にプロジェクトに新しい従業員を導入すると、コードにさまざまなエラーが発生することを理解する必要があります。 そして、この場合のテストは、彼の個人的な再保険と、彼の決定を生産に注ぎ込む人の再保険です。



雇用主が質問の2番目の部分に答えると、チームが製品の品質を確保し、空室で議論されなかった開発者の可能な責任も理解できるようになります。



この問題に関して、技術専門家がしばしば迷子になり始めることは注目に値します。 チームがテストを書き始めたばかりで、この技術のすべての微妙さにまだ慣れていないという人が時々います。 しかし、時々私はこの答えを聞きました： 「テスターはテストに従事すべきであり、開発者は作成しなければなりません。」 これは絶対に真実ではありません。



開発者は、作成した機能がどのように機能するかを知っているため、必要な最小限のテストを記述する必要があります。 テスターの役に立たないことについて誰も話していない。 ただし、開発者はコードの品質にも責任を持つ必要があることを理解することが重要です。



2.開発者は、コードをリポジトリに送信する前にコードをどうしますか？



この質問は、さまざまなパラメーターに対するコードのローカル検証に関するものです。 以下は、リポジトリに送信する前に通常チェックされるコードの短いリストです。



  • Flake8 - PEP8に準拠するためのコード分析、
  • Pylint-静的コード分析、
  • カバレッジ -テストカバレッジのコード分析、
  • Tox-個々のパッケージの異なるバージョンおよびPythonの異なるバージョンとの互換性についてコードを確認します。


開発にこのケースがないことは重要ではありません。 また、多くの企業ではこのケースはCIで直接使用されており、開発者はローカルで何も起動しません。 これが開発で使用されていなくても、インタビューする人がこれらのツールの基本的な理解を持っているといいでしょう。



3. CI / CDプロジェクトはありますか？ DevOpsエンジニアはいますか？



この質問には落とし穴はなく、会社のデバイスをよりよく理解するようにお願いします。 プロジェクトにCI / CDがなく、DevOpsエンジニアも欠席している場合は、これを行う可能性があります。 したがって、この点はインタビューで議論することをお勧めします。



4.コードレビューはありますか？ どうですか？



誰もがこのイベントの重要性を理解しているため、質問の最初の部分はコメントなしで残すことができます。 しかし、私は個人的にそれがどのように行われるかに興味を持っていたことは注目に値します。 各チームは、マージリクエストを行った開発者を改訂することがよくあります。 しかし、時には、開発者に対してメンター/メンターがいることがあり、開発者をレビューするのは彼です。 コードを修正する人が多いほど、プロジェクトにとってもチームにとっても良いので、最初のアプローチの方が正しいと思います。 チームワーク、集団的責任、 バス要素の増加などの側面に直ちに影響します。



5.どのバージョン管理システムを使用していますか？



現在ロシアでは、 hg 、 svn 、およびその他の古代バージョン管理システムを使用している多くの企業があります。 これは、10年以上市場に出回っている企業に特に当てはまります。 この質問は、会社が新技術の影響を受けやすい年齢をさらにテストします。 また、私は短期間hgを使用して開発に参加しましたが、あまり喜びを感じませんでした。



6. git / hg-flowを使用するか、git / hgを使用する際に特定の方法論を使用しますか？



この質問は、バージョン管理システムに関する前の質問から続きます。 したがって、チームがgit / hgを使用しない場合、それを尋ねても意味がありません。 会社がgit / hgを使用している場合、この質問は開発プロセスのデバッグの程度を示します。



7.開発方法論（スクラム、かんばんなど）を使用していますか？



開発では、特定のアプローチ（方法論）に従うことが重要です。 最も一般的な開発アプローチは反復的です。 このアプローチにより、プロジェクトへの貢献度を判断できます。 私の理解では、チームが何らかの方法論を使用している場合、これは間違いなく良いことです。 これにより、有効性を判断できます。 また、タスクに割り当てられたタイムラインを理解するのにも役立ちます。 学童には1年の4分の4があるのと同じです。そこでは、その年の最終成績を決定するためのマークが与えられます。



8.監視システムはプロジェクト（Sentry、NewRelicなど）で使用されていますか？



プロジェクト内の監視システムの存在は、テストの存在と同じくらい重要です。 エンドユーザーが実行するアクションに基づいてシステム全体の作業を客観的に評価できるのは、監視システムです。 監視システムがない場合は、製造された製品の品質を考慮する必要があります。 それは料理をする料理人のようなものですが、おいしいかどうかは誰にも尋ねません。



9.プロジェクトは、ログを保存し、ログを操作するためのシステム（ELKテクノロジーなど）を使用していますか？



私にとっても、これは重要な指標です。 ELKがない場合、システムの複雑なエラーの原因を特定することは非常に困難です。 この質問は質問番号8ほど重要ではありませんが、複雑なエラーのプロファイリングにおけるチームの経験がどれほど豊富であるかを理解することもお勧めします。



10.プロジェクトで使用されているデータベースは何ですか？ なぜまさにこれらなのでしょうか？



この質問は、インタビュー対象者の能力を評価することを目的としています。 古いデータベースを使用しているとき、「歴史的に起こった」ようなことをよく耳にします。 この答えは不適切だと思います。 技術者は、使用しているデータベースの長所と短所を理解する必要があります。 この質問は、あなた自身がさまざまなデータベースとその違いに精通している場合にのみ尋ねるべきです。



11.プロジェクトで使用されているPythonのバージョンは何ですか？ Python2.xのバージョンが使用されている場合、Python3.xに移行する計画はありますか？ そして、あるバージョンから別のバージョンにどのように移行しますか？



この質問は、前の質問と同様に、インタビュー対象者の能力を評価すること、およびその推論を評価することを目的としています。 雇用主は非常に文盲であり、そのような問題は面接の段階ですでに特定できることを理解する必要があります。 この種の質問をする前に、自分でさらに深く掘り下げることを強くお勧めします。



12.会社はフルスタック開発者またはバックエンド開発者を探していますか？



この質問は、面接前に会社自体が指定しなかった場合にのみ行います。 労働市場でのフルスタック開発者の仕事は非常に頻繁に見つかります。 多くの企業は、これを自分たちにとって有益だと考えています。 私の個人的な経験から、フロントスタックとバックエンドはWebが誕生してからあまりにも異なる方向になっているため、フルスタックの開発者はいないことがわかります。 つまり、「2つの椅子に座ることはできません。」



ほとんどの場合、会社はあなたがフロントエンドを知らないことに満足しており、戦闘で直接学習することを期待しています。 私は、フルスタック開発者の空きが個人的に受け入れられないことを明確にします。 多くの人々は、これをフロントエンドの豊かな世界に突入する絶好の機会であり、1ルーブルを支払うことなく見つけます。



13.コンテナ化技術はプロジェクトで使用されていますか？



この質問は質問3の補足です。



14.このプロジェクトの前に彼が何をしていたか、そして彼がプロジェクトにどのくらい在籍していたかについて、小さなインタビュアーに尋ねます。



この質問は非常に重要です。 インタビューを受けた人の経験が豊かなほど、時間が経つにつれてスキルに影響を与えます。 従業員の離職が遅い小さな会社でこのような質問をするのは特に良いことです。



15.会社は年次/四半期ごとの従業員評価を行っていますか？



従業員が同僚からフィードバックを受け取ると便利です。 会社にこのための特別なイベントがある場合、これは素晴らしいことです。 そうでない場合、心配することはありません。 いずれにせよ、無料の形式で同僚にフィードバックを要求することを禁止する人はいません。



16.会社には処理がありますか？ もしそうなら、それらは相殺され、どのくらいの頻度で発生しますか？



特に学生や看護師の父親の場合、リサイクルを好む人はほとんどいません。 リサイクルを最前線に置いている企業は数多くあります。 会社に精製所がないか、まれであることを理解するには、この種の質問をする必要があります。 会社がたまに精製を行う場合、重要なことは何もありません。 精製がより頻繁になった場合、さらに会社にとどまることの実現可能性を考慮する価値があります。



17.会社の官僚制度はどれくらい強力ですか？ （1から10までのレート）

多くの開発者は、IT分野での官僚制度の存在を認識していませんが、残念ながら存在します。 これは、特に大規模な古い会社または州と協力する会社に適用されます。 注文。 会社の官僚主義の程度は、経営者の想像力にのみ依存します。 通常、官僚機構は、さまざまな正式な申請書、サイト、アクセス、会社の複数の部門間の利益相反、およびWordでの退屈な生文書の作成で構成されます。 このような官僚制度の主な問題は、開発プロセスの非常に強い阻害です。 通常の会社で1営業日で何をするかは、数週間かかります。 簡単に言えば、会社の官僚主義が強ければ強いほど、製品の開発と専門家としての開発が遅くなります。



18.リソースの除去についてはどうですか？



リソースは、従業員、サーバー、ドメイン、ライセンスなどの新しいコンピューターとして理解されます。 この問題は、以前の官僚制の問題に起因することもあります。



19.インタビュアーはプロジェクトの新しい実装とどのように関係していますか？



この質問により、チーム内の民主主義を評価することができます。 また、この質問により、通常の開発者の声がチームとメンターにどれだけの重みを持っているかを理解できます。



20.会社はIT会議の参加者であり、ITトピックに関する出版物はありますか？



この会議は、開発者にとっても会社にとっても、自分自身とその成果を宣言する絶好の機会です。 会社が公開されており、会議に参加している場合は、ある時点でこの機会を利用することもできます。 これがあなたにとって面白くない場合、それについて尋ねることは意味がありません。



21.社内にマイタップはありますか？



ここでは、チーム内またはチーム間の開発者のmitapについて説明します。 ミタパスは非常に重要です。 特定の時間に誰が何をしているかを正確に知ることができます。 人前で話すことに問題がある場合、これはあなたのソフトスキルの開発にも貢献します。



22.会社にはインターンとメンター制度がありますか？



インターンは会社の将来の可能性です。 会社にインターンがいる場合、おそらくあなたはそれらのメンターになり、あなたの個人的な経験を共有することができます。 メンタリングは、開発できる分野の1つでもあります。



おわりに



上記のすべては個人的な経験に基づいた私の考えであり、100％の真の情報ではありません。 この記事の主な教訓は、候補者だけでなく雇用者もチェックする必要があるということです。 また、面接前または面接時に緊張しないでください。 あなたは、インタビュアーがあなたと同じ人であり、また間違っているか、何かを知らないか、または理解していないことを理解する必要があります。
