創造の歴史



そんな感じでした。 私たちが勤務する会社のウェブサイトビルダーuCozは、8年間存在し、ウェブマスターが作成した3500万を超えるプロファイルと、ウェブマスターサイト、フォーラム、ブログが作成した多数の訪問者がデータセンターの腸内に蓄積しています。



これらのすべての人々は、グローバルなuID認証システムによって結ばれています。

about.meプロジェクトは、各uCozユーザーの個々のページの既存の最良のプロトタイプとして選ばれました。これは、21世紀初頭のWeb住民の自己表現という現代のトレンドに合っていると考えています。



about.meの場合と同様に、ユーザーに次の情報を提供します。



1.名刺への印刷に使用できるuid.me/name_nameの形式のURLは、skypeのホームページとして使用でき、メディアで言及されている場合もあります。



2.個人の写真を単一の視覚画像、高解像度の背景画像、自分に関する基本情報（伝記や関心のある領域など）に結合する機能。



3.パーソナルページにユニークな外観と全体的な視覚的一貫性をすばやくエキサイティングに与えることができるデザイナー。









4.最後に、最も興味深いのは、今日、私たちの多くがソーシャルネットワークに積極的に参加していることです。 FacebookやVkontakteのフォーマットに近い人、TwitterやInstagramのマイクロブログに限定されている人、Youtubeで自分の人気チャンネルを持っている人。



そして、ここでルールは真実です-人がより多くの社会活動を示すほど、「ソーシャルネットワークのどれを「メイン」と見なすべきか」という質問がより深刻になります。



uid.meを一種の個人的なオンライン名刺として使用することをお勧めします。 Googleのサービスを使用すると、最も一般的なソーシャルネットワークを自分のプロファイルに添付できます。新しい貴重な知り合いを作成したり、プロファイルでskypeを指定したり、フォーラムで署名したりするときに、リンクを選択する必要はありません。



投稿、ツイート、写真がプロフィールに自動的に表示され、一般的なビューが表示に使用され、「ストリーム」機能がイベントの共通フィードを作成し、発生したすべてを時系列順に結合します。







ところで、 uid.meに個人用ページを作成する場合は、ソーシャルネットワークを介した自動登録を使用することをお勧めします。 「 ログイン 」ボタンのいずれかをクリックすると、登録情報を入力せずに個人ページがすぐに作成されます！







今後の展望として、近い将来ソーシャルネットワークと緊密に統合することで、プロジェクト開発計画が大きく変わると言いたいです。

以下が成功のコンポーネントとして選択されました。



0. Nginx。 彼なしで。



1.すぐに使用できるデータベースは、データを複数のサーバーに分散する問題と、何らかの理由でクラスターからサーバーが物理的に失われる場合のフォールトトレランスの問題を解決します。 この能力では、アクティブなホリバーにもかかわらず、MongoDBが選択されました。



2.機能的プロトタイピングの主要なフェーズと後続のフェーズを損失なく実行できる柔軟なデータスキーム。 MongoDBは再び役に立ちましたが、ここでは便宜上リソースを支払う必要があったため、 「BSONは贅沢品ですか、それとも現代の交通手段ですか？」という質問に対する主な回答を得る必要があります。

機会を最大化するとは、次のことを意味します。



1. JSONデータストレージ形式により、データスキーマ内の通常の親/子/ xリンクを気にせずに、または気にせず、常識に制限されます。 その結果、メインユーザーオブジェクトのネストされた構造は太字であることがわかりましたが、便利です。 一連のフラグ、表示設定、小さなリンクリスト、および以前は一連のニアユーザーSQLテーブルの作成につながった他のすべてを大胆に配置しました。



2.汎用コードがデータモデルに追加されました。これにより、インターフェイスのプロトタイプを作成する段階で、JS機能を向上させることが非常に快適になりました。たとえば、URL / profile / saveを介して、新しいデータでユーザーオブジェクトを拡張したJSONを送信できるようになりました： 



 user.save({ 'style.profile.top': '20px', 'style.caption.tags.color': 'rgba(30, 29, 38, 1)', 'info.first_name': '' });




許可されたユーザーのアクティビティに関連するすべての操作は、500ミリ秒の潜在コレクターを備えた共通の送信機能にパッケージ化され、さまざまなアトミック編集を共通のパッケージに結合しました。



その結果、クライアント側の開発者は、新しいフィールドの使用を開始するだけで、ユーザーオブジェクトの構造を簡単に拡張できました。



もちろん、プロトタイピングフェーズの後、 / profile / saveのサーバー側には、正確さのために不明なフィールドとフィルター処理された値を遮断するコンテキストデータフィルターが装備されていました。



問題は1つだけでした。ユーザーがデータベースに保存できるのは、フィールドが表示される前にプロファイルを最後に編集してから、一部のフィールドがまったく存在しなかったためです。 理想的には、データベースから取得したオブジェクトに魔法のように表示される各フィールドのデフォルト値が必要です。



ORMレベルでは、取得されたすべてのデータの強制拡張が、存在しないすべてのフィールドのデフォルト値とともに追加されました。



円は閉じています。



データベースのエンドツーエンドの更新に頼らずにオブジェクトの構造を動的に拡張し、サーバー側だけでなくクライアント側でも透過的に動作することができましたが、新しいものを追加するプロセスは非常に快適で、プロトタイプからリリースへの移行にはちょうど2つが伴いましたアクション：



1.新しいフィールドデータのルールをextend_rulesフィルターに追加します。

2.このフィールドの予想されるデフォルト値をdefault_userに追加します。
