ベースの簡単な説明



当初から、Eloqueraデータベースは.Net Frameworkに基づいてオブジェクトを保存するために作成されました。これにより、オブジェクトデータベースとリレーショナルデータベースの両方の長所を同時に取り入れることができ、多くの違いを克服できました。 理論的には、Eloqueraは.Net Frameworkファミリーの任意の言語で動作しますが、実際には、動作はC＃でのみテストされています。 開発者の主な焦点はエンタープライズセグメント（100 GB以上）であり、組み込みソリューションではありませんが、組み込みソリューションも注目を集めていません。



Eloqueraの特徴的な機能は非常に印象的であり、常に更新されています。ここに非常に短いリストを示します。

  • 特別なインターフェイスやアダプターを必要とせずに、C＃オブジェクト（.Netプラットフォーム上の任意の言語のオブジェクト）を保存します。
  • 任意のフィールド\プロパティを持つ動的オブジェクトを保存し、任意のタイプのオブジェクトにマップできます。
  • クエリ言語はできるだけSQLに近く、リレーショナルSQLデータベースが存在する必要はありません。 さらに、LINQサポート。
  • オブジェクトが格納された形式（列挙型を含む）でオブジェクトを返します
  • リストおよび配列の形式のパラメーターのサポート
  • クエリの正規表現。
  • テンプレートオブジェクトのサポート。
  • 読み取り専用フィールドとプロパティを復元します。
  • オブジェクトの部分的な返却がサポートされています。 たとえば、ForumTopicクラスが必要な場合、ForumMessagesへのすべてのリンクをプルアップすることはできません。
  • リクエストで返すオブジェクトの深さを指定できます。
  • ストアドプロシージャ。




設置



登録後、公式ウェブサイトwww.eloquera.comから基本ディストリビューションを入手できます。 デスクトップインストール用およびサービスとしてのインストール用（x86またはx64）アーカイブの選択肢が表示されます。 Eloqueraをサービスとしてインストールする場合、落とし穴はありません。



多少なりとも最新のマシンへのインストールには、5分しかかかりません。 その後、あなたの処分は次のようになります：

  • Eloquera Config Studio;
  • Eloquera DB Studio;
  • サービス自体がベースの操作を担当します。


デスクトップバージョンは、x-copyベースでインストールされます。 ファイルを目的のディレクトリにコピーするだけです。 動作させるには、インストール済みの.Net 4 Full Frameworkが必要です。



このアプローチでは、NuGetパッケージは非常に期待されているように見えますが、現時点ではまだ利用できません。 ただし、開発者はこの可能性を認識しており、データベースのスタンドアロンバージョンの場合、カーネルは実行可能ファイルではなくライブラリの形式で発行されるとすぐに、すぐに作成することを脅しています。



基本モード



Eloqueraデータベースはいくつかのモードで動作します：

  • 複数のアプリケーション用の共通データベース、標準モード。
  • メモリ内モード。
  • デスクトップ-アプリケーションに組み込まれたベースのモード。


サーバーに接続してデータベースを作成するためのコード： 

private const string DBName = "ServiceDatabase"; using (var eloquera = new DB("server=localhost:43962;options=none;")) { eloquera.CreateDatabase(DBName); eloquera.OpenDatabase(DBName); eloquera.Close(); }


DBクラスは、データベースへの接続に使用されます。 接続コンストラクターはクラスコンストラクターに渡されます;この例では、接続はサービスに行きます。



Eloqueraは、サービスとして作業するときとオフラインで作業するとき（デスクトップモード）の両方で、インメモリモードをサポートします。 無料版では、メモリベースは最大1 GBを使用できます。 また、データベースのサイズがWindowsがディスク上のスワップファイルに配置しようとするようなサイズである可能性があるため、RAMの物理的な可用性を覚えておく価値があります。 ご存知のように、これは速度をまったく追加しません。



RAMにデータベースを作成することをエンジンに示すために、接続文字列にメモリ内の特別なパラメーターを指定する必要があります。 

 new DB("server=(local); options=inmemory;"))


この場合、データベースはRAMにのみ存在しますが、ファイルシステムには空のファイルが作成されます 。 そのようなデータベースで作業を再開できるようにするには、オプションの接続文字列でpersistパラメーターを使用する必要があります



デスクトップモードで動作するには、接続文字列は次のようになります。 

 new DB("server=(local); options=none;"))


Eloqueraがデスクトップモードで動作を開始するには、EloqueraDBサービスを停止する必要があります。停止しないと、アドレス（ローカル）のサービスに接続できないというエラーが表示されます。



データベースには多くの興味深いものがありますが、最も興味深いものについて説明します。

操作の一般的な構文は次のとおりです。 

 SELECT Alias1,Alias2 FROM Type1 Alias1 [ LEFT | RIGHT| INNER | FULL ] JOIN Type2 Alias2 ONSomeFunction(Alias1.Field1, Alias2.Field2)


SELECTでJOINを使用してクエリを作成するときに2つ以上の異なるタイプが使用された場合、そのような操作の結果は新しいオブジェクトIEnumerable <Dictionary <string、object >>になります。



JOINリクエストは、他のタイプのリクエストと同様に、名前付きパラメーターを使用できます。 式ONを含めます。 ただし、留意すべき制限がいくつかあります。

  • パラメータの値は、クエリを開始する前に決定する必要があります。
  • 名前付きパラメーターを使用して、Selectステートメントの型名を置き換えることはできません。 つまり 選択タイプを呼び出そうとする試みは、ルールの外になります。


それとは別に、JOIN式の構築に関係するクラスの子孫で問題がどのように解決されるかについて言及する価値があります。 標準的な動作は、クエリで継承された型を使用する機能です。 たとえば、TypeCがTypeAから継承される場合、式は次のようになります。 

 SELECT a, b FROM TypeA a INNER JOIN TypeB b ON b.id = a.id


式の条件を満たす場合、「a」フィールドにTypeCオブジェクトを返すことができます。 クエリ結果のクラスのすべての子孫を削除するには、ONLYという単語を使用します。 つまり 次のようにリクエストを書き換えます。 

 SELECT a, b FROM ONLY TypeA a INNER JOIN TypeB b ON b.id = a.id


これで、フィールド「a」には、タイプTypeAのオブジェクトのみが存在します。

パラメータ



パラメータはデータをリクエストに転送するためにのみ使用できるため、パラメータを介してデータを受信することを想像するのは無意味です。



パラメータの使用は非常に簡単です。 

 Parameters param = db.CreateParameters(); param["name"] = "John"


リクエストでパラメータを使用するには、それらを受け入れるオーバーロードメソッドを使用する必要があります。 例： 

 var res = db.ExecuteQuery("SELECT BasicUser WHERE Name = @name", param);


パラメータ値の配列は、データベースによって単純な配列として解釈できます。 

 Parameters param = db.CreateParameters(); param.AddList("emails", new string[]{"john@gmail.com", "david@hotmail.com"}); var res = db.ExecuteQuery("SELECT BasicUser WHERE ALL @emails in Emails", param);


パラメーターは、Top、Skip、およびOrde​​r Byで使用できます。 パラメータにはnullも含まれる場合があります。



ストアドプロシージャ



バージョン4.0以降、Eloqueraはストアドプロシージャをサポートしています。



ストアドプロシージャは、メインデータベースの拡張メソッドと見なすことができます。 これらは、C＃を使用して記述されています。これは、作成者のおかげで、この問題について解釈された方言を発明しなかったためです。 プロシージャを記述するときに、すべての.Net機能を使用できることがわかりました。



十分な言葉で言えば、ストアドプロシージャが実際にどのように作成されるかを示す方が良いでしょう。



最初の手順は、目的のプロシージャを使用してインターフェイスを宣言することです。 このインターフェイスは、クライアント側とサーバー側になければなりません。 

 public interface ICityStatisticSP{ CityCars GetCityCars(string cityName); }


次に、サーバー側でのみ必要なクラスを作成します。 この場合、プロシージャを実装したライブラリをデータベースディレクトリのLibフォルダに配置するか、ストアドプロシージャを含むライブラリへのパスをStoredProceduresPathパラメーターを使用して構成する必要があります。 

 public class CityStatisticSP: StoredProcedure, ICityStatisticSP { public CityCars GetCityCars(string cityName) { Parameters parms = db.CreateParameters(); parms["city"] = cityName; var cars = db.ExecuteQuery("SELECT Car FROM Car JOIN Seller ON Car.SellerID = Seller.ID WHERE Seller.City = @city", parms) .OfType<Car>() .ToList(); … } }


新しいプロシージャを追加する場合、データベースサービスを再起動する必要はありません。



プロシージャを含むライブラリをサーバーにスローした後、次のコードを使用してクライアントでプロシージャを呼び出すことができます。 

 ICityStatisticSP procedures = db.GetStoredProcedureClass<ICityStatisticSP>(); string cityName = "Chicago"; CityCars cars = procedures.GetCityCars(cityName);


データベースのまさにオブジェクトの性質に基づいて、マッピング構成手順やその他の手順を構成する必要はありません。



既存のフォームにストアドプロシージャが存在することで、データベースクエリを単に保存および呼び出しするよりも複雑な問題を解決できます。 すべてのコードがベース側で実行されることを強調する価値があります 。 つまり プロシージャを使用すると、ネットワークの負荷を大幅に削減できます。プロシージャを使用すると、たとえば、新しい要素を作成してデータベースに保存するなど、サーバー側でデータをより効果的にフィルタリングできます。 データベースから多くの要素を何らかの値で追加する必要がある状況を想像することは難しくないと思います。 古いアプローチでは、すべての要素をアンロードし、更新のための条件に従って選択を作成し、更新し、ネットワーク上で新しいデータを上書きする必要がありました。 ストアドプロシージャは、このような問題をより効率的に解決します。



動的オブジェクト（動的）



Eloqueraの開発者によると、彼らはオブジェクトベースにいくつかの魔法を追加し、独自の機能とDynamicと呼ばれるオブジェクトを得ました。 この機能から脚がどこで成長するか、そしてそれがC＃でイデオロギー的に調整されるものを既に推測しています。 動的オブジェクトは、構造化データとしてそれ自体に保存できます。 C＃コードで定義され、任意に構成されたクラスと構造。 さらに、これらのオブジェクトを互いに自由に表現できます。



主な利点は、保存されたオブジェクトにフィールドを追加および削除できることです。データは魔法のようにオブジェクトにアタッチおよび埋め込まれます。

動的オブジェクトが提供する利点：

  • 作成と使用が簡単です。 フィールドやプロパティの欠落を心配する必要はありません。 任意のタイプの任意の数のフィールド をランタイムに追加できます。
  • フィールドのタイプを自動的に判別し、その機能と制限に従ってフィールドを操作します。 動的オブジェクトはインデックス作成を完全にサポートしています。
  • それらはお互いを参照し、お互いに含まれることができます。
  • 動的オブジェクトは非常に高速です。
  • それらは、その場で「ネイティブ」オブジェクトに、またはその逆に変換できます 。
  • 仕事の場合、データスキームは必要ありません。これにより、活動の分野が非常に広くなります。
  • ブログエンジン、CMS、wiki
  • ユーザープロファイルと登録
  • レコード履歴の保存
  • ランタイムでデータをフィルタリングまたは集約する必要がある場合。
  • 異なるタイプの属性の数によって変数を操作するアプリケーション。


最後の2つの点に関して、ユーザーが独立してデータ型を構築し、各ユーザーのインターフェイスを動的に構成できるようにするさまざまなプログラムを意味することを明確にする必要があります。 爆発的な複雑さを伴うことなく、そのようなデータを厳密な形式で保存する合理的な可能性はありません。 Dynamicを使用すると、そのようなタイプを無限の多様性で記述し、単一のスタイルでそれらを操作できます。



別の例はweb 2.0で、ユーザーはすべてにコメントを追加したり、資料を関連付けたりできます。 クラスの厳密な記述である従来のリレーショナルデータベースでは、このようなスキームを実装することは不可能または非常に困難です。



動的オブジェクトの操作の簡単な概要



動的オブジェクトの構造は、含まれるデータとともにいつでも変更できます。 使いやすさとともに、これはデータの進化に十分な機会を提供します。 

 Dynamic @do = new Dynamic(); @do["Title"] = "Some title"; @do["Price"] = 14.99; @do["DateAdded"] = DateTime.Now; db.Store(@do); Dynamic res = (from Dynamic d in db where d["Price"] == 14.99 select d).First<Dynamic>();


.Net 4で導入されたタイプdynamicの変数を使用しても同じことが可能です。 

 dynamic @do = new Dynamic(); @do.Title = "Some title"; @do.Price = 14.99; @do.DateAdded = DateTime.Now; db.Store(@do); dynamic res = (from Dynamic d in db where d["Price"] == 14.99 select d).First<Dynamic>();


LINQを使用する場合、 d [“ Price”]を使用して、式ツリーの構築を明示的に示す必要があることに注意してください。



動的オブジェクトを使用する場合、オブジェクトに含まれるすべてのフィールドを見つけることができます。 



 Dynamic @do = dynamics[0]; foreach (var field in @do) { Console.WriteLine(String.Format("Field Name {0} Value {1} Type {2}", field.Key, field.Value, field.Type)); }


幅広い可能性に加えて、いくつかの制限があります。特に、フィールド名を指定するときに使用できない文字のリストがあります。 これは、SQLクエリをオブジェクトに書き込む際の衝突を回避するためです。 予約文字のリストは次のとおりです。



。、％] [+-$！ *（）
