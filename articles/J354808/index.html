静的解析を使用する理由



ビデオゲームの開発には多数の段階が含まれるという事実にもかかわらず、プログラミングは依然として主要な段階の1つです。 たとえ何千行ものコードを書かなくても、さまざまなツールを使用する必要があります。その品質は使いやすさと最終結果に依存します。 あなたがそのようなツール（たとえば、ゲームエンジン）の開発者であれば、すべてがすでに明確になっているはずです。



静的解析がソフトウェア開発全般に役立つのはなぜですか？



主な理由はいくつかあります。



  • 修正エラーのコストと複雑さは、時間とともに増加します。 静的分析の主な利点の1つは、開発の初期段階でのエラー検出です（コードの作成時にエラーを見つけることができます）。 したがって、静的アナライザーを使用すると、頭痛になる前に多くのエラーを検出して修正することにより、自分と同僚の生活を簡素化できます。
  • 静的分析を使用すると、さまざまなエラーパターン（コピーペースト、タイプミス、機能の誤用など）を特定できます。
  • 原則として、静的分析は、動的分析では検出されにくい問題を特定するのに適しています。 ただし、逆もまた真です。
  • さまざまな警告抑制機能（単一、パターン別など）、無関係な診断の無効化、分析からのファイルとディレクトリの無効化により、負の使用ポイント（たとえば、誤検出）は通常、深刻な静的アナライザーの開発チームによって「スムーズ化」されます。 これにより、アナライザーを適切に構成することで、「ノイズ」の割合を大幅に減らすことができます。 同僚のAndrei Karpovは、 EFLコアライブラリのチェックに関する記事で 、アナライザーが構成されている場合、誤検知の数は10〜15％に過ぎないことを実証しました。


とにかく、これはすべて理論であり、おそらくより実用的な例に興味があるでしょう。 さて、それらを以下に示します。

 public static CrawledMemorySnapshot Unpack(....) { .... var result = new CrawledMemorySnapshot { .... staticFields = packedSnapshot.typeDescriptions .Where(t => t.staticFieldBytes != null & t.staticFieldBytes.Length > 0) .Select(t => UnpackStaticFields(t)) .ToArray() .... }; .... }


PVS-Studio警告 ： V3080 null参照解除の可能性があります。 「t.staticFieldBytes」の検査を検討してください。



Whereメソッドに引数として渡されたラムダ式に注目してください。 コードから判断すると、 typeDescriptionsコレクションには、 staticFieldBytesメンバーがnullになる可能性のある要素が含まれる場合があります 。 そのため、 Lengthプロパティにアクセスする前に、 staticFieldBytes！= Nullを確認します。 演算子は混乱しているだけで、「&&」の代わりに「＆」が使用されています。 これは、左の式の評価結果（ true \ false ）に関係なく、右の式も評価されることを意味します。そのため、 staticFieldBytes == nullの場合、 LengthプロパティにアクセスするときにNullReferenceException型の例外がスローされます。 「&&」を使用する場合、 staticFieldBytes == nullの場合、式の右側は評価されないため、このような状況は発生しません。
