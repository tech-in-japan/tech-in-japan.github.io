デスクトップでのLinux Deployの実際的な使用

Linux DeployはもともとAndroidアプリケーションとして考えられていたという事実にもかかわらず、やがてそのアプリケーションの他のオプションが登場します。 Linux Deploy CLIの出現により、このツールの新しいアプリケーション領域を開く多くの機能が利用可能になりました。







Linux Deploy CLIは、chrootコンテナー内のGNU / Linuxディストリビューションのインストール、構成、および起動を自動化するように設計されたコマンドラインアプリケーションです。 アプリケーションは、通常のデスクトップLinuxディストリビューションと、Linuxカーネルに基づいたモバイルプラットフォームの両方で動作できます。ただし、必要な依存関係が確認されていれば（すべての依存関係を静的に構築できます）。 Linuxディストリビューションのアプリケーションは、chroot環境で実行され、メインシステムと並行して動作し、速度に匹敵します。 Linux Deployの操作はLinuxカーネルのシステムコールに基づいているため、Linuxディストリビューションのみが「ゲスト」システムとして機能できます。



アプリケーションは、スーパーユーザー権限あり（chroot）とスーパーユーザー権限なし（proot）の2つのモードで動作できます。 通常モードでは、ファイル、ディスクパーティション（論理ドライブ）、POSIX互換ディレクトリ、およびRAM（tmpfs）へのインストールがサポートされています。 prootモードでは、インストールはディレクトリでのみ使用でき、いくつかの制限が表示されます。



  • コンテナ内のすべてのユーザーは、コンテナのファイルシステム全体へのフルアクセス権を持ち、現在のユーザーはすべてのファイルとディレクトリの所有者です。
  • システムでの特権操作へのアクセスはありません。たとえば、ping、ulimitなどは機能しません。
  • アプリケーションは、1024を超えるネットワークポート番号でのみ機能します。
  • アプリケーションが作業でchrootシステムコールを使用する場合、特別なfakechrootユーティリティ、たとえばfakechroot / usr / sbin / sshd -p 2222を使用して起動する必要があります。


アプリケーションは、Debian、Ubuntu、Kali Linux、Arch Linux、Fedora、CentOS、Gentoo、openSUSE、Slackwareのディストリビューションの（基本システムの）自動インストールと初期構成をサポートしています。 Linuxディストリビューションのインストールは、公式のインターネットミラーからネットワーク経由で行われます。 また、tar.gz、tar.bz2、またはtar.xzの形式で事前に準備されたrootfs-archiveからの他のシステムのインポートもサポートしています。 このアプリケーションを使用すると、インストールされたシステム（コンテナー）のコンソールに接続したり、コンテナー内のアプリケーションを起動および停止したりできます（さまざまな初期化システムと独自の自動実行スクリプトがサポートされます）。 各インストールオプションは個別の設定ファイルに保存され、各コンテナの設定を行います。 必要に応じて、コンテナを並行して起動できます。 再インストールおよび構成せずに、このコンテナーの後続の展開のために、構成およびコンテナー自体をrootfsアーカイブとしてエクスポートできます。



一般に、Linux Deployのアイデアは、開発、テスト、またはトレーニングに使用できるLinuxディストリビューションを迅速にデプロイし、メイン（ホスト）Linuxシステムに変更を加えずにすぐに削除できる簡単で便利なツールを手に入れたいという要望から生まれました。そしてその完全性を危険にさらすことなく。 PRootプログラムのおかげで、ルート特権なしでLinuxアプリケーションを実行するためのコンテナーを作成できるようになりました。また、 QEMUソフトウェアエミュレーションを使用して、 カーネルレベルのbinfmt_miscモジュールをサポートせずにホストとは異なるアーキテクチャのアプリケーションを実行できるようになりました。



2011年以降の私の主要な仕事以来、Debianを搭載したコンピューターが使用されてきました。 ローカル開発者は、Webアプリケーション（主にJava、PHP、Python）を実行およびテストするためのシステムを定期的に必要とします。 これらの目的のために、仮想システムは通常、VirtualBoxに基づいて、またはローカルのProxmoxの「クラウド」またはDockerで使用されました。 VirtualBoxの主な欠点は、コンピューターリソースの厳密さ、VDIディスクイメージのサイズが大きいこと、比較的低速であること、およびシステムが誤ってシャットダウンしたときにVMイメージがクラッシュする可能性があることです。 「クラウド」を使用することの不利な点は、管理者自身がそのようなシステムの作成のためのユーザーリクエストを処理する必要があることと、セカンダリタスクでの「クラウド」リソースの消費です。 Dockerを使用するには、スーパーユーザー権限が必要です。



今月、実験が行われました; PHP開発者は、仮想サーバーをLDコンテナーに置き換えました。 Debianベースのコンテナが2つ用意されました：Apache + PHP + OCI8およびApache + PHP + MySQL + PhpMyAdmin。 コンテナはローカルネットワーク上の共有ネットワークドライブに配置され、各コンテナのサイズは約150 MBでした。



管理者がこれから受け取ったもの：



  • 準備されたコンテナは、管理者の参加なしで1つのチームで開発者のコ​​ンピューターに展開できます。
  • コンテナでの作業にはスーパーユーザー権限は必要ないため、メインシステムが故障する危険性はありません。


開発者が得たもの：



  • コンテナ内のシステムの展開、起動、および管理は、単一のチームとして管理者が参加することなく実行されます。
  • 事前に準備されたアーカイブからのコンテナの展開は、ネットワーク上で1分以内に行われます。
  • コンテナの起動と停止（Webサーバー+データベース）は即座に発生し、オペレーティングシステムの起動を待つ必要はありません。
  • コンピュータの電源を切ったときに電源を切るのを忘れても、コンテナを破損する危険はありません。 システムイメージは、独自のファイルシステムのない通常のディレクトリです。
  • コンピューターは高速です リソースは、オペレーティングシステム全体ではなく、コンテナ内のソフトウェアの実行にのみ使用されます（この場合、これはVirtualBoxの500 MBではなく、約50 MBです）。
  • サーバーにアップロードする必要なく、IDEディレクトリからソフトウェアの機能を直接チェックします。これは、メインシステムの必要なディレクトリをコンテナに接続するだけで十分です。


そして、これを実現する方法についてさらに詳しく説明します。 次に、LDコンテナの準備と展開に関する指示が提供されます。



スーパーユーザー権限なしでコンテナを実行するには、PRootをインストールする必要があります。 



mkdir ~/bin wget http://portable.proot.me/proot-x86_64 -O ~/bin/proot chmod 755 ~/bin/proot


Linux Deploy CLIをダウンロードしてインストールします。 



 wget -O cli.zip https://github.com/meefik/linuxdeploy-cli/archive/master.zip unzip cli.zip rm cli.zip ln -sf ~/linuxdeploy-cli/cli.sh ~/bin/linuxdeploy


Debian Wheezyベースシステム（64ビット）をデプロイするための「linux」という設定を作成します。 



 linuxdeploy -p linux conf --method='proot' --source-path='http://mirror.yandex.ru/debian/' \ --distrib='debian' --arch='amd64' --suite='wheezy' --target-path='$ENV_DIR/rootfs/linux' \ --chroot-dir='$TARGET_PATH' --target-type='directory' --username='webmaster' --include='bootstrap'


保存された構成を表示： 



 linuxdeploy -p linux conf -x


新しいシステムの展開の開始： 



 linuxdeploy -p linux deploy


rootユーザーでコンテナーコンソールに接続する（exitコマンドを終了する）： 



 linuxdeploy -p linux shell -u root


次に、必要なソフトウェアをコンテナにインストールして構成できますが、前述の機能を考慮する必要があります。 たとえば、Apacheを起動するには、ポート（ファイル/etc/apache2/ports.conf）を8000に変更し、空のパラメーターAPACHE_ULIMIT_MAX_FILES = ""（ファイル/ etc / apache2 / envvars）を設定し、通常のユーザーの下からapachectlを実行する必要があります（ルートではありません）。



SysV初期化システムに基づいた自動実行の構成： 



 linuxdeploy -p linux conf --include='$INCLUDE init' --init='sysv' --init-level='3' --init-user='$USER_NAME' --init-async


パラメータ：INIT_LEVEL-初期化レベルSysV、INIT_USER-サービスを開始するユーザー（デフォルトではルート）、INIT_ASYNC-サービスを並行して開始します。



構成の準備、構成のエクスポート、およびコンテナーのrootfsアーカイブへのエクスポート（tar.gz、tar.bz2、およびtar.xzアーカイブがサポートされています）： 



 linuxdeploy -p linux conf --source-path='linux.tgz' --target-path='\$ENV_DIR/rootfs/linux' --chroot-dir='\$TARGET_PATH' linuxdeploy -p linux conf -x > /path/to/linux.conf linuxdeploy -p linux export /path/to/linux.tgz


「\ $」をエスケープすると、変数の値ではなく、構成内の変数の名前を保存できます。 したがって、構成をインポートすると、これらの変数は対応する値に自動的に置き換えられ、現在の値とは異なる場合があります。 別のコンピューターにコンテナーをインポートするときに使用できる2つのファイル（linux.confとlinux.tgz）があります。 



 cd /path/to linuxdeploy -p linux conf -i ./linux.conf linuxdeploy -p linux deploy


メインシステムのディレクトリをコンテナに接続します（コンテナの〜/ wwwディレクトリを/ var / wwwに接続します）： 



 linuxdeploy -p linux conf --mounts='$HOME/www:/var/www'


コンテナの起動（SysVの場合、スクリプト/etc/rcN.d/SXXname startが実行されます）： 



 linuxdeploy -p linux start


リソースを解放してコンテナを停止する（SysVの場合、スクリプト/etc/rc6.d/KXXname stopが実行されます）： 



 linuxdeploy -p linux stop -u


その結果、開発者と管理者の両方のニーズを満たすソリューションが得られます。 Linux Deployアプリケーションのソースコードは、GPLバージョン3ライセンスで利用できます。
