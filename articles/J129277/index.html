シナリオ ：銀行とATMで私たちが好きな瞬間は、彼らが常に同じ場所にいるということです。 ATMを使用すると、24時間いつでもお金を引き出したり預けたりできます。 または、銀行が営業しているときに銀行の支店に行き、財務への完全なアクセス権があることを確認します。 これを保証するには、ErlyBankオートメーションシステムが常に稼働していることを確認する必要があります。プロセスは常に実行されている必要があります。 ErlyBankは、この目標を実現するように指示しました。 稼働率100％！ （または提供できる限り近い）



結果 ：OTP スーパーバイザーを使用して、実行中のプロセスを監視し、それらがアクティブであることを確認する責任を持つプロセスを作成します。



コントローラーとは？



コントローラーは、子プロセスと呼ばれるものを追跡するプロセスです。 子プロセスが「吹き飛ばされた」場合、コントローラーはこの子の再起動戦略を使用して再起動します。 この方法により、Erlangシステムの永続的な可用性を確保できます。



コントローラは、 監視ツリーと呼ばれるものの一部です 。 適切に作成されたErlang / OTPアプリケーションが起動し、ルートコントローラーから開始します。ルートコントローラーは、子コントローラーを監視し、追加のコントローラーまたはプロセスを追跡します。 考えは、コントローラーがクラッシュした場合、親コントローラーがそれを再起動するということで、ルートコントローラーまで続きます。 Erlangランタイムには、システム全体が監視され、ルートコントローラーが停止した場合に再起動する優れたモードがあります。 したがって、コントロールツリーは常に動作可能です。



コントローラーには、 init / 1のみのフィードバックメソッドがあります。 そのタスクは、子プロセスのリストを返し、各プロセスの戦略を再起動することです。これにより、コントローラーは、監視する対象と何か問題が発生した場合の対処方法を認識します。



eb_serverとイベントマネージャーを分離



gen_eventsに関する以前の記事で実装したことの1つは、eb_serverモジュールの初期化メソッドでイベントマネージャープロセスを明示的に開始することでした。 そのような依存関係を持つサーバーを簡単に起動したい場合、これが唯一の機会でした。 しかし、今では、コントローラーの使用を開始および停止するため、コントロールツリーでイベントマネージャーを開始することができます。 サーバーコードからeb_event_managerの開始を削除しましょう。



これを行うには、イベントマネージャーを起動する行84をeb_serverモジュールから削除します。 また、この場所にadd_handler呼び出しを追加して、 eb_withdrawal_handlerハンドラーをイベントマネージャーに接続します （ 前の記事への追加を含め、この翻訳サイクルで説明されている翻訳を連続して読んで実装する場合、何も追加する必要はありません。 ;イベントマネージャーの開始を除外した後、初期化メソッドのコードは次のようになります-トランスレーターコメント ）。 これで、eb_serverモジュールのinitメソッドは次のようになります。



init([]) ->

eb_event_manager:add_handler(eb_withdrawal_handler),

{ok, dict:new()}.

再起動戦略



コントローラには単一の再起動戦略があり、子孫の仕様と組み合わせて使用​​して、子孫の1つが死ぬかどうかを判断します。 次のリストには、可能な再起動戦略が含まれています。



  • one_for_one-子プロセスの1つが停止すると、コントローラーはそれを再起動します。 他の子孫は触れません。
  • one_for_all-子プロセスの1つが停止すると、他のすべての子が停止し、すべてが再起動します。
  • rest_for_one-子プロセスの1つが停止すると、故人が完了した後に子孫の仕様のリストに定義された子孫の「残り」が、その後すべてが再開します。


再起動戦略は、次の形式で示されます。



{RestartStrategy, MaxRetries, MaxTime}







ロシア語で理解するのは非常に簡単です： MaxTime秒に1回MaxRetriesよりも頻繁に子が再起動されると、コントローラーはすべての子プロセスを終了し、それ自体で動作を停止します。 これは、子の再起動の無限ループを防ぐために行われます。



子孫仕様の構文と基本概念



コントローラーのinitメソッドは、子孫仕様のリストを返す役割を果たします。 これらの仕様は、開始するプロセスとこれを行う方法をコントローラーに伝えます。 コントローラーは、「左から右へ」の順序（リストの最初から最後まで）でプロセスを開始します。 再起動戦略は、次の形式のタプルです。



{Id, StartFunc, Restart, Shutdown, Type, Modules}



:

Id = term()

StartFunc = {M,F,A}

M = F = atom()

A = [term()]

Restart = permanent | transient | temporary

Shutdown = brutal_kill | int()>=0 | infinity

Type = worker | supervisor

Modules = [Module] | dynamic

Module = atom()









Idは 、子孫の仕様を格納するためにコントローラ内でのみ使用されますが、一般的な規則では、モジュールの複数のインスタンスを実行している場合を除き、モジュール名をIDとして使用します。 後者の場合、IDに番号を追加します。



StartFuncは{Module、Function、Args}形式のタプルであり、呼び出しがプロセスを開始する関数を示します。 非常に重要 ：開始関数はプロセスを開始してリンクする必要があり 、 {ok、Pid} 、 {ok、Pid、Other}または{error、Reason}を返す必要があります 。 従来のOTP start_linkメソッドはこのルールに従います。 ただし、独自のプロセスを開始するモジュールを実装する場合は、 spawn_linkを使用してそれらを開始してください 。



再起動は、上記のコードブロックで説明した3つのアトムのうちの1つです。 アトムがrestartとして使用される場合、プロセスは常に再び開始されます。 値が「 temporary 」の場合、プロセスは二度と開始されません。 また、この値が「 transient 」の場合、プロセスは予期せず終了した場合にのみ再起動されます。



シャットダウンは、子プロセスを終了する方法をコントローラーに説明します。 brutal_kullアトムは、終了メソッドを呼び出さずに子を終了します。 ゼロより大きい整数は、正しい完了のタイムアウトを意味します。 アトム「 無限 」はプロセスを丁寧に完了し、永久に停止するのを待ちます。



Typeは、子孫が何であるかをコントローラーに伝えます：別のコントローラーまたは他のプロセス。 それがコントローラーの場合、スーパーバイザーアトムを使用し、そうでない場合はワーカーアトムを使用します。



モジュールは、このプロセスが影響するモジュールのリスト、または「動的」アトムです。 この値のリストにあるケースの95％では、単一のOTPフィードバックモジュールを使用します。 プロセスがgen_eventである場合、「動的」が使用されます。これは、モジュールへの影響が動的であるためです（すぐに定義できないさまざまなハンドラー）。 このリストはリリース管理にのみ使用され、この記事の文脈では重要ではありませんが、リリース管理に関する今後の記事で使用されます。



うわー そんなに短い時間で学ぶべき多くの情報。 子孫仕様の形式とさまざまな再起動戦略を覚えるのにかなり時間がかかったので、すぐにそれを行うことができなくても心配しないでください。 コントローラマニュアルの情報はいつでも更新できます 。



イベントマネージャーの子孫仕様



サーバーがそれに依存しているため、最初に開始するのはイベントマネージャーです。 子孫の仕様は次のようになります。



EventManager = {eb_event_manager,{eb_event_manager, start_link,[]},

permanent,2000,worker,dynamic}.







子孫仕様の構文に関するセクションを読んだ後、このコードは非常に簡単なはずです。 各パラメーターの効果を理解するには、戻って説明を確認する必要がありますが、これはまったく正常です！ 数分ですべてをうなずき、忘れるよりも、コードを長引かせて理解する方が適切です。 仕様の説明で「奇妙な」ことの1つは、モジュールのリストを「動的」（ 動的、この場合、これはアトム-約Translator ）として示すことだと思います。 これは、gen_eventについて話しているために行われ、そこで使用されるモジュールのリストは動的です。 ハンドラーが接続されています（ 作業中に変更できる数-約翻訳者 ）。 それ以外の場合は、プロセスが使用するすべてのモジュールをリストする必要があります。



子孫仕様を含めた後の初期化メソッドは次のとおりです。



init([]) ->

EventManager = {eb_event_manager,{eb_event_manager, start_link,[]},

permanent,2000,worker,dynamic},

{ok,{{one_for_one,5,10}, [EventManager]}}.







各子孫仕様を変数に割り当ててから、戻り値に直接仕様を入れるのではなく、これらの変数の内容を使用して返すことを好みます。 Erlangの最大のトウモロコシの1つは、プログラマーがリストとタプルを深く埋め込んで、どこから終わり、どこから始まるかがわからないときに表示されるので、それぞれを変数に割り当てることをお勧めします。



コントローラーをコンパイルして実行すると（価値があると思います！）、コントローラーに属するstart_linkメソッドを開始した後、 whereis（eb_event_manager）を入力すると、コマンドはイベントマネージャープロセスの識別子（ pid ）を返します。 次に、 exit（whereis（eb_sup）、kill）を実行してコントローラーを強制終了し 、eb_event_manager識別子を再度取得しようとすると、プロセスが強制終了されたため、定義されていないというメッセージが表示されます。



また、楽しみのために、コントローラーの制御下で作業中にeb_event_managerを強制終了します。 数秒待ってからプロセスを確認してください。 彼は回復しなければなりません！
