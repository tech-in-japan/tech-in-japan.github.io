ステップ5.統合テスト



統合テストは、個々のソフトウェアモジュールをグループで組み合わせてテストするソフトウェアテストのフェーズの1つです。

統合テストには3つのアプローチがあります。



ボトムアップ（ボトムアップ統合）

低レベルのモジュール、プロシージャ、または関数はすべてまとめられ、テストされます。 その後、次のレベルのモジュールで統合テストを実施します。 このアプローチは、開発レベルのすべてまたはほとんどすべてのモジュールの準備が整っている場合に有用と考えられます。 また、このアプローチは、テスト結果からアプリケーションの準備状況のレベルを判断するのに役立ちます。



トップダウン統合

最初に、すべての高レベルモジュールがテストされ、徐々に低レベルモジュールが追加されます。 下位レベルのすべてのモジュールは、同様の機能を備えたプラグによってシミュレートされ、準備ができたら、実際のアクティブなコンポーネントに置き換えられます。 したがって、上から下へテストを実施します。



ビッグバン（「ビッグバン」統合）

開発されたすべてまたはほとんどすべてのモジュールは、完全なシステムまたはその主要部分として組み立てられ、統合テストが実行されます。 このアプローチは、時間を節約するのに非常に適しています。



すべてのモジュールが用意されているので、 ボトムアップアプローチを使用します。



反復アプローチ


反復アプローチを使用します。つまり、モジュールを1つずつ、ボトムアップで接続します。 最初に、API +モデルの束、次にAPI +モデル+マッパー+プレゼンター、次に一般的なAPI +モデルマッパー+プレゼンター+ビューを確認します。



否定的および肯定的なシナリオ


統合テストでは、サーバーからの2つの応答シナリオを検討する必要があります。通常の応答とエラーです。 これに応じて、コンポーネントの動作が変わります。 各テストの前に、サーバー（MockWebServer）からの応答を構成し、結果を確認できます。

ステップ6.機能テスト



機能テストは、機能要件の実現可能性、つまり特定の条件下でユーザーが必要とするタスクを解決するソフトウェアの能力を検証するためのソフトウェアテストです。 機能要件により、ソフトウェアの機能、解決するタスクが決まります。



Androidアプリケーションの一部として、ユーザーの視点からアプリケーションをテストします。 まず、カスタムアプリケーションマップを作成します。



アプリケーションマップ




必要なテストケースを作成します。

  • アプリケーションを開き、すべての要素の可視性を確認します
  • テストユーザーを入力し、[検索]をクリックします

    • 受信したデータ-リポジトリのリストを取得し、データ表示を確認します。
    • データを受信して​​いません-エラー表示を確認してください。
  • 2番目の画面に移動し、ユーザー名とリポジトリ名が正しく表示されているかどうかを確認します。

    • ブランチと貢献者のリストを取得し、データ表示を確認
    • 受信されなかったリスト（2つのテスト）、受信されたリストの表示、エラー表示を確認します
    • 両方のリストが受信されない、エラー表示を確認する

ステップ7. TDD









テスト駆動開発とは、テストの作成を通じて開発を定義するソフトウェア開発手法です。



基本的に、次の3つの簡単な繰り返し手順を実行する必要があります。

-追加する必要がある新しい機能のテストを作成します。

-テストに合格するコードを記述します。

-新しいコードと古いコードをリファクタリングします。



略語TDDになじみがない場合は、iOS部門の同僚からの 記事またはTDDハブからの 記事を読むことをお勧めします。



TDDには3つの法則があります。



  • 本番コードは、アイドルテストが行​​われる前に作成されません。
  • エラーに十分な数の単体テストコードは書き込まれません。
  • 現在のアイドルテストに合格するのに十分な量のプロダクションコードは書き込まれません。


たとえば、インターネットからのダウンロードを表示するプログレスバーを作成します。 データがダウンロードされると表示され、データがロードされるかエラーが発生すると表示されなくなります。 すべての開発はTDDで行われます。



この機能の開発は、プレゼンターとフラグメントに影響し、マッパーと日付レイヤーは変更されません。

ステップ8.次は何ですか？



TDDを研究し、ロードマッピングを開発したら、アプリケーションの開発を終了します。 さらなる開発のために、次の記事を読むことをお勧めします。



Android Clean Architecture

Android Clean ArchitectureはFernando Cejasの有名な記事で、Uncle BobのClean Architectureに基づいています。 プレゼンテーション層、ドメイン層、データ層の3つの層の間の相互作用が考慮されます。 habrahabrへの翻訳があります。



VIPER

VIPER（View、Interactor、Presenter、Entity、Routing）の人気が高まっています。VikkoSの記事VIPER Jet-powered VIPERで詳しく知ることができます。 VIPERの主要な原則は、iOSの同僚からの記事とレポートで取り上げられています。



モスビー

Mosbyは、MVPアプリケーションを作成するための一般的なライブラリです。 すべてのメインインターフェイスと基本クラスが含まれています。 ウェブサイト： http : //hannesdorfmann.com/mosby/ Github： https : //github.com/sockeqwe/mosby



Androidアプリケーションアーキテクチャ

Ribotチームのアーキテクチャに関する良い記事-Androidアプリケーションアーキテクチャ 。 AsyncTaskからRxJavaへの移行が考慮されます。 最近、 habrahabrへの翻訳がリリースされました。



Android開発文化文書

Android開発カルチャードキュメントArtem_zinによる #qualitymatters Artem Zinnatullinによる素晴らしい記事とデモプロジェクト。 この記事では、Androidアプリケーションを開発するための8つの原則について説明しています。Githubの例でサポートされています。



おわりに



この一連の記事では、アプリケーション開発のすべての段階について説明しました。 シンプルなMVPベースのアーキテクチャから始めたため、新しい機能を追加するのが難しくなりました。 リアクティブプログラミングとコールバックを取り除くためのRxJavaとRxAndroid、ネットワークとの便利な作業のためのRetrofit、迅速で簡単なビュー検索のためのButterknifeなどの最新のライブラリを使用しました。 Dagger 2はすべての依存関係を管理し、テストの作成時に非常に貴重なサポートを提供しました。 jUnit、Robolectric、Mockito、MockWebServerを使用して、テスト自体を作成しました。 そして、エスプレッソは、テスターを回帰テストの苦労から救いました。



プロジェクトをテストで完全にカバーしました。 単体テストでは各コンポーネントを個別にテストし、統合テストでは全体的な相互作用をテストし、機能テストではすべてをユーザー側から調べます。 プログラムをさらに変更しても、一部のコンポーネントが破損することを恐れることはできません（まあ、またはほとんど恐れません）。何かが落ちて、リリースにバグが入り込みます。 TDDのおかげで、ほとんどのコードはテストでカバーされます（テストなし、コードなし）。 部分的なカバレッジや「コードは作成されましたが、テストの時間はありませんでした」という問題はありません。



すべてのプロジェクトコードはGithub（https://github.com/andrey7mel/android-step-by-step）で入手できます
