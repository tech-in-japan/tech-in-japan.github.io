それは何で、何を食べますか



Couchbaseは、Couchbase、Incによって開発されたNoSQLデータベースの次のラインであり、親であるCouchDBの伝統と問題の直接の子孫です。 これはドキュメント指向のデータベースであり、個々のレコードをドキュメントとして保存することを意味しますが、これは厳密なルールではなく、値（BLOB行まで）はレコードとして機能しますが、この（および他のデータベースも）の魅力はドキュメントベースのデータ保存方法。



ドキュメントベースのデータ保存方法は、すべてのデータがいわゆるドキュメントの形式で保存されることを意味します。 レコードに存在する常識と一般的な論理の原則に関する文書にまとめられたフィールドのセット。 このようなレコードの例としては、たとえば、ログイン、パスワード、電子メールなどのフィールドのリストを持つユーザープロファイルがあります。 この場合、ドキュメントストレージの標準はJSON文字列形式のドキュメント形式です。 この形式は、非常に人気があり、簡単に解釈でき、人間が読める形式であるため、作成者が慎重に選択しました。 しかし、ポイントではありません。 文書とは何か、データベース内でどのように見えるかを把握しておくことが重要です。

コーディングを開始



抽象的なものをコーディングするのは理にかなっていないので、上記の非常に具体的な例、つまりユーザープロファイルを見てみましょう。 ユーザーがいくつかのフィールドを持っているとしましょう：ログイン、パスワード、電子メール、暗黙のうちに-これは整数型の識別子です。 JSON表現では、結果のドキュメントは次のようになります。 
{ "login": "megausername", "password": "my secured password!", "email": "email@example.com" }




まず、このビジネスをデータベースに保存する方法と、そこからデータを取得する方法を見つける必要があります。 これは非常に簡単に行われ、次の例ではっきりとわかります。

ビューとは何ですか？



Couchbaseサーバーのコントロールパネルにアクセスすると、Viewsという素晴らしいものに気付くでしょう。 「開発ビュー」と「生産ビュー」という2つのサブアイテムがあります。 ディスプレイを推測することは難しくありません（Couchbaseのコンテキストでは、選択です）。 空がありますが、そこを台無しにする方法を見つけます。



ビューは基本的にインデックスであり、JavaScriptで記述できる作成のルールです。 はい、はい。 ここでのインデックスは、ユーザーが記述したロジックに基づいて作成され、列挙関数だけでなく、いくつかのセマンティック関数も保持します。 たとえば、電子メールがn文字より長いユーザーをインデックスに含めることも、特定のフィールドのみを表示することもできます。 インデックスを操作するには、JavaScriptしかありませんが、それで十分です。 インデックスは、オンデマンド（データの要求時）で更新されるか、データベースの断片化が特定の割合（バケット設定で指定）に達すると自動的に更新されます。 この点も開発時に考慮する必要があります。



ビューを作成するには2つの方法があります。 1つ目は、開発セクションのコントロールパネルにJavaScriptルールを直接記述してプロダクションに転送するか、PHPから直接プロダクションセクションに移動するスクリプトの説明を使用してsetDesignDocメソッドをプルすることです。



最初に、スクリプトとその構成要素を検討します。 スクリプトは、ドキュメントとそのコンテンツのメタ情報の説明が入力される関数です。 ユーザーのログインに基づいてインデックスを作成してみましょう。 
 function (doc, meta) { //     JSON      login if (meta.type == "json" && doc.login) { //      userId (      ) var userid = meta.id.split("::"); //       / emit(doc.login, parseInt(userid[1])); } }


上記のJavaScriptメソッドのおかげで、インデックス（emitメソッドを使用して生成される）にはログインフィールドを持つエントリのみが含まれることを理解できます。 ご覧のとおり、JavaScript関数はコールバック関数の形式で作成され、このバケットにある各レコードに適用されます。 ビューは、バケットライフサイクルの存在中いつでも作成できることに注意してください。 つまり 新しい機能を追加する必要がある場合は、新しいビューを簡単に追加して、必要に応じてそのまま使用できます。

ただし、この方法でインデックス全体を取得するという事実を考慮する必要がありますが、これは検索にはあまり適していません。 そして、識別子のみを見つける必要がある場合はどうでしょうか？ インデックス全体を手動で反復しないでください。 もちろん違います。 このため、表示する各リクエスト中に、追加のパラメーターを指定できます。 たとえば、ログインで指定されたユーザーの識別子のみを知りたい場合、表示するリクエストで特定のキーを指定する必要があります。 これは次のように行われます。

操作を実行する前に、ユーザー名の一意性を確認する必要があるとします。 たとえば、ユーザーを登録するとき。 当然、このビューの要求を満たし、データベースの応答を分析します。 ただし、同じ名前のユーザーが登録されたばかりで、インデックスを再構築する時間がまだない可能性があります。 当然、インデックスにそのようなレコードが存在しないという情報が届き、競合が発生します。 このような状況を回避するために、ビューインデックスを呼び出すときに、インデックスを再構築する必要性を彼に示すことができます。 つまり このインデックスで実行されなかったすべての操作が最初に完了し、その後クエリが完了して結果が返されます。 これを行うには、値がFALSEの古いオプションを追加します。 これは次のように行われます。 
 $result = $couchbase->view('userFields', "login", array('key'=>'megausername', 'stale'=>FALSE));




このリクエストの結果、インデックスの操作に関連するすべての操作が完了し、データベースにある実際の結果が得られます。 このような特定のデータを扱うときは、この点を考慮することが重要です。
