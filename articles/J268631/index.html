MySQLにはないPostgreSQL機能、およびその逆





多くの人は、それが与えるものを漠然としか理解していないという事実のため、「筋肉」から「迷走」に切り替えることを恐れています。 一部の人々は、Postgresはおそらくあまりにも複雑で、トレーニングが必要であるという考えにとらわれています。 また、移行に伴い何かが失われる可能性があります。 状況を少し明確にしようと思います。



一般的に言えば、誰かが複雑さを恐れている場合、初心者は通常のようにできます。新機能を使用せずに、MySQLからPostgreSQLに「ダム」スイッチします。 SQL-アフリカSQLにもあります。ロケット科学ではありません。 この移行では、（いわゆるプログラミングでは）複雑なことは何もありません。 さて、引用符は異なりますが、構文は少し厳密です。 つまり pgをmysqlとして使用し、他の引用符を付けて開始し、プレイの進行とともに学習します。



さて、一方にはあるが、他方にはない、またはその逆について。 パンの種類を理解する。 もちろん、これは完全なリストではなく、重要度で並べ替えられているわけではありませんが、それでもなお並べられています。



おそらく筋肉にはない罪の欠点から始めましょう。



  1. 進行中の実稼働環境で作業するには、専門的にセットアップする必要があります。 shared_buffers、自動バキューム設定などを正しく設定しないと、深刻な負荷の下ですべてがゆっくりと動作します。 特に厄介なのは、高負荷プロジェクトの場合、ほとんど常にpgbouncer（これはサードパーティの開発）が必要なことです。これにより、キューに接続が蓄積され、進行状況自体への接続数が制限されます。 これがpostgres自体に組み込まれていないのは奇妙です
  2. 自動バキューム。 簡単に言えば、書き込み/削除の高速化を実現するために、プログレッションはガベージの山を残し、その後、特別に訓練されたデーモンによってクリーンアップされます。 特に非常に忙しいベースで自動バキュームを設定したり、愚か者をオフにしたりするのが間違っている場合、テーブルが占める場所が膨らみ、遅かれ早かれ、詰まる、または膨らむことなく、詰まる可能性のあるものすべてが詰まり、ベースが単に立ち上がることができますトランザクションIDが終了しました。 各会議では、誰かがどのように自動バキュームで英雄的に戦って勝ったかについての3-4のレポートがあります。
  3. 最近まで、INSERT IGNOREおよびINSERT ON DUPLICATE KEY UPDATEはありませんでした。 9.5から、アナログが登場しました。 誰もが必要とするような機能の実装に長い間引きずられていたことは非常に奇妙です。
  4. Mysqlでは、クエリ内の変数を直接操作できます。 

    	 SELECT @x：= 0;
         SELECT @x：= @ x + 1 FROMテーブル;


    進行中のそのようなことはありません、少なくとも私はそれを見つけませんでした（あなたが間違えた場合に書いてください）。 つまり、もちろん、何でもできるストレージを作成することはできますが、リクエストに応じて適切に作成することはできません。
  5. 通常のphpmyadminに対応するものはありません。 実際、私が知っているクールな同僚のほとんどは、最初は慣れるのが難しいコマンドラインでSQLを使用しています。 いいえ、あらゆる種類のpgMyAdminなどがありますが、それぞれに何らかの欠陥があります。 たぶん良い有料のものがあるので、私はチェックしませんでした。
  6. Mysqlは誰もが知っている、postgresqlは誰も知らない。 したがって、多くの場合、新しいプロジェクトはpostgresqlで開始することを恐れています。 「データベース」という言葉とmysqlという言葉がまったく同じ同義語である、PHPプログラマーのクラスがあります。 つまり 彼らが貝殻か何かから抜け出すことはどういうわけか困難です。
  7. 彼らは、安価なホスティングは管理がより難しいので、postgresqlを本当に好きではないと言います。 たとえば、postgresにログインできるユーザーを作成するには、sqlクエリを実行してpg_hba.confに書き込むという2つの場所でこれを行う必要があります。


現時点でmysqlと比較した場合の欠点のうち。 他に何か特別なことを知っている場合、mysqlにあるものとpostgresqlにないものをコメントで書きます。 これで、postgresqlの利点：



  1. CTE（共通テーブル式）



    簡単な方法で説明すると、サブクエリを個別に記述して、名前を付け、データベースへの単一クエリのフレームワーク内でこれらすべてを記述することができます。 例えば

  2. IPアドレスを操作します。 たとえば、ip-addressで都市/国をすばやく特定する必要があります。

  11. トランザクション。 CREATE TABLE、ALTER TABLEなどは、単純なクエリと同様に、単一のトランザクションで実行したり、何か問題がある場合は途中でトランザクションをロールバックしたりできます。 時間が経てば、それなしでmysqlを使用する方法を理解できなくなります。

  13. シーケンス mysqlでは、テーブルフィールドごとにAUTO_INCREMENTのみがあり、一度に1つずつチェックされます。 postgresqlでは、このメカニズムはテーブルとは独立して存在し、さまざまなニーズに使用できます。さらに、ループで作成することもできます。
  14. DBAはpostgresqlをメインのトランザクションマシンと見なしているようです。 そこにあるトランザクションは深く、うまく組み込まれているため、挿入と読み取りの両方で、すべてが迅速かつ確実に機能します。 mysqlでは、システムが異なり、ベースがあり、個別のエンジン（innodb、myisamなど）があり、エンジンはすべてトランザクションではありません。 このトランザクションの分離にはいくつかの問題があります。 たとえば、myisamはトランザクションではなく、innodbはトランザクションであり、同じクエリで両方のテーブルを使用できます。 基地が同時にどのように機能するか、予測することは想定していません。おそらく困難で松葉杖です。
  15. 主観的に、postgresqlのバグは少ないです。 彼らがどのようにこれを達成しているかはわかりませんが、私にとってこれは事実です-非常に安定した信頼性の高いシステムであり、重い負荷とデータ量の下でも。
