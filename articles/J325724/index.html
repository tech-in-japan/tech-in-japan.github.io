すべてがオープンワーク：Microsoft AzureでのParallels RAS統合





先ほど、 Parallels Remote Application Server （ RAS ）がIDCによってVCC（仮想クライアントコンピューティング）市場の主要なグローバルプレーヤーの1つとして認識されていることを書きました。 RASは最近、 Microsoft Azureクラウドコンピューティング環境に統合されました。 いくつかの写真、マーケティングゼリー、およびさまざまな仮想マシンでの獣の技術的なテストの結果について説明します。 企業のリモートアクセスの組織について学ぶための記事へのコメントで、Azure +を使用した個人的な経験から質問やストーリーを得るのは素晴らしいことです。



誰（または何）が議論されているかを思い出してください。



Parallels RASは、アプリケーションおよびデスクトップ仮想化システムです。 彼女は主に、iOS、Androidデバイス、ChromeOS、HTML5からリモートサーバーで実行されているWindowsアプリケーションへの従業員のアクセスを整理する必要がある大、中、および小企業の代表者に関心を持っています。 言い換えると、RASでは、「私たちには長い手があります」というフレーズに新しい意味が付けられ、スマートフォンやタブレットを介してリモートデスクトップコンピューターやサーバーでホストされているファイルやドキュメントを操作したい従業員、同僚、パートナーに適用できます。







主要なRASコンポーネント：



  • 発行エージェントモジュール
  • セキュアクライアントゲートウェイモジュール。 エンドユーザーがRASにアクセスできるようにします。 SSLを使用して接続を暗号化できます。 新しいゲートウェイを追加することにより、回復力とスケーラビリティが向上します。 スループット-選択したモードに応じて、ゲートウェイごとに250〜500ユーザー。
  • ターミナルサーバーエージェントモジュール。 RASがMicrosoft RDSサーバーを使用できるようにします
  • 高可用性ロードバランサー（HALB）モジュール 。 Secure Client Gatewayの「カルーセル」バランシングを実行します。
  • VDIホストエージェントモジュール。 RASファームへのハイパーバイザーホスト（VMware、Hyper-V、XenServer）の接続を提供します。






問題を解決する際のシステム管理者の主な「武器」は忘れられていません-レポート。 RASは14種類のレポートを生成できます。ユーザーレポート（ユーザーアクティビティ、セッションユーザーアクティビティ、クライアントデバイス、OS）。 すべて同じですが、グループに関するものです。 サーバーのステータスとアクティビティ 公開アプリケーションの使用など。 これらすべての情報は、7つの異なる形式でエクスポートできます。







なんで？



RAS + Azureケーキのチェリーに戻りましょう。 RAS自体は生きていたのに、なぜAzureでロールアップしたのですか？ 目標は、アプリケーションへのアクセスを展開する顧客の簡素化、高速化、およびコスト削減です。 早く。 シンプル。 なるほど。 これは、RASが世界で公正な市場シェアを占める方法のレシピです。 一部のパトスで申し訳ありませんが、最終的には、中小企業がハードウェアインフラストラクチャの作成を最大30％節約し、従業員にモバイルデバイスからあらゆる規模のデスクトップアプリケーションへのアクセスを迅速に提供できるソリューションです。



統合アーキテクチャ



Azureで仮想マシンを作成すると、Azureリソースグループコンテナーが形成されます。 このコンテナのすべての仮想マシンは、1つの仮想ネットワークに分離されています。







発行エージェントモジュールは、着信接続を配布し、ターミナルサーバー（TS）にインストールされたアプリケーションのエンドユーザー間の接続を構成します。



統合の利点は何ですか？



RASをAzureプラットフォームに統合することは、モバイルユーザーがクラウドでWindowsベースのアプリケーションを使用できることに加えて、Microsoft RDS（リモートデスクトップサーバー）を使用している企業にとって役立ちます。 レドモンドに敬意を払って、彼らの製品には多くの機能があります。



  • サーバーバランシング ：サーバーバランサーの機能は制限されており、サーバーに割り当てられたセッション数と重み評価に基づいてのみ接続を管理できます。
  • ネットワーク負荷分散 ：バランサーはネットワークインターフェイスの障害のみを検出でき、ゲートウェイのステータスは監視しません。 バランサは、ゲートウェイに障害が発生したことを認識せず、引き続きゲートウェイにリクエストを送信します。
  • Windows以外のクライアント ：RDSにはLinuxクライアントとWebブラウザーがありません。 そのため、今日、管理者は、クライアントをインストールすることなく、LinuxマシンまたはブラウザーでWindowsアプリケーションとデスクトップを提供するために、サードパーティソリューションをその上にインストールする必要があります。


Microsoft RDPプロトコルを中心に構築されているParallels RASは、システム管理者がより短時間でより多くのことを行えるように支援します。 さらに、設定と管理が非常に簡単なため、事前のトレーニングなしで操作できます。



仮想マシンごとのユーザー数



LoginVSIを使用して、異なるシリーズ（A、D、およびDv2）のAzure仮想マシンをサポートするRASユーザーの数をテストして調べました。 評価は、タスクワーカーとナレッジワーカーの2種類のワークロードに対して実施されました。 セッション全体でパフォーマンスを測定しました：システムへのログオン、仮想デスクトップのユーザーへの割り当て、テストタスクの処理（下記を参照）、ログオフ。 テスト結果に対するデータキャッシュの影響を排除するために、ワークロードは最初に仮想マシンで48分間スピンし、その後初めてパフォーマンスが測定されました。 仮想マシンの各インスタンスについて、3つのテストが実行され、平均値が取得されました。







測定した：



  • 最小、最大、および平均応答時間
  • CPU負荷
  • メモリ消費
  • ディスクI / O応答時間


タスクワーカーのワークロードプロファイル：スクリプトは、Microsoft Outlook、Excel、Internet Explorer、Adobe Acrobat、およびPDF Writerの操作で実行されました。 このようなワークロードでは、作業環境の生産性に対する要求はそれほど高くありません。 通常、この種のワークロードは、サービスの操作にあまり時間を費やさないユーザーにとって一般的です。

ナレッジワーカーのワークロードプロファイル：Microsoft Outlook、Word、PowerPoint、Excel、Adobe Acrobat、FreeMind、PhotoViewer、Doro PDF Writerで操作するスクリプトが実行され、いくつかのビデオが360p解像度で表示されました。 このようなワークロードでは、環境のパフォーマンスにより高い要求が課されます。



Azure仮想マシンごとのRASユーザーの数：







仮想マシンの数とスケーリングを選択するときに、この経験的データに集中できます。 また、これらのデータは、ニーズに応じて特定のAzure VMシリーズを選択する費用対効果を評価するのに役立ちます。 したがって、数十から数百のユーザーにとって、AzureでのRASへの移行は、VDIを自動化すると同時にコストとサポートの労力を削減することにより、興味深く効果的なものになります。



ただし、これらはおおよそのガイドラインにすぎません。 VMのグラフに41人のユーザーが示されている場合、42を接続するときに作業が不可能になるという意味ではありません。 それはすべて、負荷とパフォーマンスの要件の特定の性質に依存します。
