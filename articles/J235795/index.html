OpenSL ES簡単な説明

このことは、Android API 9（Android 2.3）以降で利用可能です。 一部の機能は、Android API 14（Android 4.0）以降でのみ使用できます。 OpenSL ESは、C ++から呼び出すこともできるC言語インターフェイスを提供し、サウンドを操作するためのAndroid Java APIの一部と同じ機能を提供します。

  • android.media.MediaRecorder

Libaはご想像のとおり、純粋なCで記述されています。したがって、完全なOOPはありません。 特別な構造が使用されます（疑似オブジェクト指向構造（:)と呼びましょう。これは、構造へのポインターを最初の引数として取る関数へのポインターを含む通常のC構造です。C++ではこのようになりますが、明示的に。 ESは2種類の構造です。



  • オブジェクト （ SLObjectItf



    ）は、特定の範囲のタスクを実行し、これらのリソースに関する情報を保存するように設計された一連のリソースの抽象化です。 オブジェクトを作成すると、そのタイプが決定されます。これにより、ヘルプを使用して解決できるタスクの範囲が決定されます。
  • インターフェイス （ SLEngineItf



    、 SLSeekItf



    など）は、特定のオブジェクトによって提供される一連の相互に関連する機能の抽象化です。 インターフェイスには、オブジェクトに対してアクションを実行するために使用される多くのメソッドが含まれています。 インターフェイスは、そのインターフェイスでサポートされるメソッドの正確なリストを定義するタイプです。 インターフェイスは、インターフェイスのタイプを参照するためにコードで使用できる識別子によって定義されます（たとえば、 SL_IID_VOLUME, SL_IID_SEEK



    ）。 インターフェイスのすべての定数と名前は非常に明白なので、特別な問題はありません。




要約すると、オブジェクトはリソースの割り当てとインターフェースの取得に使用されます。 そして、これらのインターフェイスの助けを借りて初めてオブジェクトを操作します。 1つのオブジェクトに複数のインターフェイスを設定できます（ボリュームの変更、位置の変更など）。 デバイス（またはオブジェクトの種類）によっては、一部のインターフェイスが利用できない場合があります。 SLDataLocator_AndroidFD

を使用）、既にそこから再生することができます。 ただし、このオブジェクトはSL_IID_SEEK

音声コンテンツ

オーディオコンテンツをアプリケーションにパックする方法は多数あります。

  • リソース res / raw /ディレクトリにオーディオファイルを配置すると、 リソースのAPIを使用して簡単にアクセスできます。 ただし、これらのリソースへの直接のネイティブアクセスはないため、Javaコードからコピーする必要があります。
  • 資産 アセット/ディレクトリにオーディオファイルを配置することにより、ネイティブマネージャーを使用してC ++からそれらにアクセスできます。 詳細については、 android / asset_manager.hおよびandroid / asset_manager_jni.hヘッダーを参照してください。
  • ネットワーク 。 URIデータロケーターを使用して、ネットワークから直接オーディオを再生できます。 これに必要な許可を忘れないでください（：
  • ローカルファイルシステム 。 URIデータロケーターは、ローカルファイルにアクセスするためのfile：スキームをサポートします。ただし、ファイルがアプリケーションにアクセスできる場合（つまり、別のアプリケーションの内部ストレージからのファイルの読み取りは失敗します）。 Androidでは、LinuxユーザーIDおよびグループIDメカニズムを使用してファイルへのアクセスが制限されていることに注意してください。
  • 記録します。 アプリケーションは、マイクからオーディオを録音し、コンテンツを保存して、後で再生できます。
  • インラインでコンパイルおよびリンクされます 。 オーディオコンテンツをライブラリに直接プッシュしてから、バッファーキューデータロケーターで再生できます。 これは、PCM形式の短い曲に非常に適しています。 PCMデータは、bin2cツールを使用して16進文字列に変換されます。
  • リアルタイム生成 。 アプリケーションは、オンザフライでPCMデータを生成（合成）し、バッファーキューデータロケーターを使用して再生できます。

私は一般的にOOPのファンなので、何らかの方法でCメソッドの特定の機能をグループ化し、それをクラスでラップして、将来的に作業しやすくするようにしています。 OpenALでこれを行った方法との類推により、クラスが登場しました。

    サウンドを操作するための基本クラス。
  3. OSLWav



    WAVを操作するためのクラス。 OSLSoundから継承して、操作用の共通インターフェイスを維持します。 oggを使用するには、OpenALで行ったようにOSLOggクラスを作成できます。 これらの形式ではダウンロードプロセスが根本的に異なるため、このような区別をしました。 WAVはきれいな形式で、そこにバイトを読み込むのに十分です。oggはOgg Vorbisを使用して解凍する必要がありますが、mp3（：
  4. OSLMp3



    Mp3を操作するためのクラス。 OSLSoundから継承して、操作用の共通インターフェイスを維持します。 mp3をストリーミングするので、このクラスは私にはあまり役に立たない。 しかし、ラメなどを使用してmp3をデコードする場合は、load（char * filename）メソッドでデコードを実装し、BufferPlayerを使用できます。
  5. OSLPlayer



    実際には、サウンドを操作するためのメインクラスです。 実際のところ、OpenSL ESの作業メカニズムはOpenALとは異なります。 OpenALには、バッファーと音源（バッファーを掛ける）のための特別な構造があります。 OpenSL ESでは、すべてが異なるプレイヤーを中心に展開します。
  6. OSLBufferPlayer



    。 ファイル全体をメモリにロードする場合、このプレーヤーを使用します。 通常、短い効果音（ショット、爆発など）に使用されます。 既に述べたように、 SL_IID_SEEK



    インターフェースをサポートしていないため、トラック内を移動することはできません。
  7. OSLAssetPlayer



    を使用すると、assetsディレクトリからストリーミングできます（つまり、ファイル全体をメモリにロードしません）。 長いトラック（バックグラウンドミュージックなど）を再生するために使用します。




オブジェクトを扱う原則

オブジェクトを操作するサイクル全体は次のようなものです。

  1. 目的のインターフェイスを指定してオブジェクトを取得します。
  2. (*obj)->Realize(obj, async)



    呼び出して実装します。
  3. (*obj)-> GetInterface (obj, ID, &itf)



    呼び出して、必要なインターフェイスを取得します
  4. インターフェイスを介して作業します。
  5. オブジェクトを削除し、 (*obj)->Destroy(obj)



    呼び出して使用したリソースをクリーンアップします。
