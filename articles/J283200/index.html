クラウドホスト間で仮想マシンイメージを転送するか、デジタルオーシャンにWindows Serverをインストールする

たとえば、Digital Oceanは独自の仮想イメージをダウンロードする機能を提供しません;さらに、インストール用にISOイメージを接続することもできません（KVMがあります-それに感謝します）。

したがって、トリッキーな方法ですべてをsshで実行する必要があります。 主なアイデアは、新しい仮想マシンでルートファイルシステムをアンマウントし、古い仮想マシンからファイル/イメージをアップロードし、grub設定を更新し、ネットワークとfstab設定を更新し、再起動することです。

最初のステップで最も明白な。 アンマウントすることができます/ライブシステムでは、それは退屈ではありますが、本物です。 sshサーバーといくつかのユーティリティをinitramdiskに追加し、そこからすべてを実行する方がはるかに簡単です。 OSのブートのこの段階では、ルートシステムはまだマウントされていません。

実際、記事全体は、ramdiskにsshサーバーを含めるためのユーティリティと2つの解析された例のデモです。







Debianサーバーを準仮想化を使用してXenからDigital Oceanに移行します。



DigitalOcean仮想マシンには、仮想マシン自体からカーネルを変更する機能があることが重要です（DigitalOceanコントロールパネルの[カーネル]タブでは、ホスト内のカーネルを変更するように記述する必要があります）。

Xenのソース：



そのようなデモページを持つnginxがあります





ステップ1.移行のためのソースサーバーの準備



ソースサーバーをオフにしてハイパーバイザーからファイル/イメージを提供するだけでは何も準備できませんが、これは面白くなく、さらにソースは仮想マシンではなくベアメタル上にある可能性があります。 したがって、最初の手順は、ルートファイルシステム（および、接続されているFS（ある場合））を読み取り専用で再マウントすることです。 読み取り専用での再マウントは、完全にアンマウントするよりもはるかに簡単です-何かを書き込むことができるデーモンを釘付けするだけで十分です。 

/etc/init.d/nginx stop apt-get install lsof rsync lsof /


残っているものを見ます。 FD列がwまたはuで終わる行に興味があります



ええ、rsyslogは残った 

 kill 6288 mount -o remount,ro /


それが機能する場合、それが良い場合、そうでない場合は、誰が書き込みのためにファイルを開いたままにしているかを探し続けます。 原則として、追加でインストールされたすべてのデーモンとsyslogをオフにします。



手順2.移行先サーバーでルートFSをアンマウントする



受信者はUbuntu 16.04になります。 理論的には、転送前に受信者が何をするかは絶対に重要ではありません。 これらはすべて上書きされます。initramdisk設定を自動化するユーティリティは、ubuntu / debianで動作します。

したがって、DigitalOceanで作成されたばかりのubuntu 16.04は次のとおりです。 

 apt-get install -y git dropbear && \ git clone https://github.com/roginvs/early-ssh && \ cd early-ssh && \ ./build_deb.sh && \ dpkg -i early-ssh*.deb && \ sed -ie 's/DISABLED=1/DISABLED=0/' /etc/early-ssh/early-ssh.conf && \ update-initramfs -u && \ reboot


受信者のIPアドレスにpingを送信し、TTLで追跡します。 回答のTLLが変更されるとすぐに、これはinitramdiskが起動し、接続を待機していることを意味します。 デフォルトでは、15秒後に誰も接続していない場合、ダウンロードはさらに長くなります。



ssh経由で接続してシェルを取得します。 私の場合、ルート認証はキーごとですが、sshd_configに「パスワードなしのPermitRootLogin」が含まれている場合でも（initbramdiskにはsshdではなくdropbearがあるため）パスワードで機能します。



実際、1つのセクション。



ステップ3.データ転送



dd（受信者のディスクのサイズがソースより小さくない場合は当然）またはrsyncを使用して、すべてのデータを転送できます。 ddは、ファイルシステムに空き容量が十分にない場合、または多くの小さなファイルがある場合に適しています。



ddからsshへのオプション： 

 dd if=/dev/xvda2 bs=65536 | ssh root@198.199.127.149 'dd of=/dev/vda1 bs=65536'




暗号化なしでddを使用するが、圧縮を使用するオプション： 



 #   nc -q 1 -l 8000 | gzip -d | dd bs=65536 of=/dev/vda1 #   dd if=/dev/xvda2 bs=65536 | gzip | nc -q 1 198.199.127.149 8000


隣のコンソールから、信号USR1を常にddに送信して、状況を確認できます。 

 while true; do kill -USR1 `pidof dd`; sleep 10; done




rsyncを使用したオプション：



レシーバーで、ファイルシステムを再作成してマウントします。 

 mkfs.ext3 /dev/vda1 mkdir /mnt mount /dev/vda1 /mnt


ソースがext3であるという理由だけで、ext4ではなくext3ファイルシステム。 次に、ソースサーバーから： 

 rsync -aAXv --one-file-system --progress / root@198.199.127.149:/mnt/


（最後のスラッシュに注意してください。



ステップ4.カーネルの確認、/ etc / fstabの確認、ネットワークの構成、grub設定の更新



レシーバーで、新しくコピーしたファイルシステムをマウントし、そこにchrootを作成します。 

 mount /dev/vda1 /mnt mount -o bind /dev/ /mnt/dev/ mount -o bind /proc/ /mnt/proc/ mount -o bind /sys/ /mnt/sys/ chroot /mnt PATH=$PATH:/usr/sbin bash




カーネルがインストールされていることを確認します（ソースOSが準仮想化を備えたxenにあった場合に必要です） 

 apt-cache search linux-image apt-get install linux-image-3.16.0-4-amd64 dpkg-reconfigure linux-image-3.16.0-4-amd64




/ etc / network / interfacesのネットワーク設定を更新する 

 nano /etc/network/interfaces


ここには小さな落とし穴があります：場合によっては、一部のコアでは、インターフェイスを異なる方法で呼び出すことができます。 たとえば、Ubuntuではens3と呼ばれ、Debianカーネルでは既にeth0です。 念のため、両方のオプションのレコードを作成することをお勧めします。 または、/ etc / default / grubにカー​​ネル設定を追加して、ens名を使用しないようにすることができます（GRUB_CMDLINE_LINUX行を置き換え/追加します）。 

 ... GRUB_CMDLINE_LINUX="net.ifnames=0 biosdevname=0" ...


+ファイル/etc/udev/rules.d/70-persistent-net.rulesを削除して、カーネルがeth0とのインターフェースの読み取りを開始するようにします



/ etc / fstab内のルートfs（デバイスはvda1と呼ばれ、スワップを削除します） 

 nano /etc/fstab




blkidからUUIDを見つけて入力することもできます。 

 UUID="66bd1995-a4e5-418f-a1f7-0ec2f81ac017" / ext4 errors=remount-ro 0 1




さて、ブートローダー： 

 apt-get install grub2 grub-install /dev/vda update-grub2




すべて、すべてをアンマウントして再起動できます。 

 exit #  bash exit #  chroot cd / umount /mnt/dev umount /mnt/proc umount /mnt/sys umount /mnt reboot




再起動後、新しいIPでWebページが開くことを確認します。

ステップ1.ソースの準備



Windowsでは、事態はもう少し複雑なので、最初にVmWareで仮想マシンを作成し、それをハイパーバイザーから転送しました。 インストール中、彼はOSがLinux、Lsi SASディスクドライバーになることを示しました。

）



松葉杖の魔法



松葉杖は、良い方法でunattended.xmlを介して実行する必要があるためです。 ファイルc：\ firstrun.batを作成し、そこに以下を記述します。同時に、mac / ip / mask / gateway / passwordを仮想レシーバーにあるものに置き換えます。 

 @Echo Off :: Mac is in Windows format (dashes)! SET MAC=00-50-56-9E-B8-57 SET IP=192.168.2.85 SET MASK=255.255.255.0 SET GW=192.168.2.1 SET ADMINPW=QWEasd123 echo Started >> c:\firstrun.log date /T >> c:\firstrun.log time /T >> c:\firstrun.log ipconfig /all >> c:\firstrun.log echo "Start to search network interface" >> c:\firstrun.log :start_loop For /f "delims=, tokens=1,3 skip=1" %%a In ('getmac /V /FO CSV') Do ( IF /I %%b == "%MAC%" ( echo "Found card" >> firstrun.log echo %%a >> firstrun.log netsh interface ip set address %%a static %IP% %MASK% %GW% 10 netsh interface ip add dns %%a 8.8.8.8 goto :end_loop ) ) ping 127.0.0.1 goto :start_loop :end_loop net user Administrator %ADMINPW% reg add "HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Terminal Server" /v fDenyTSConnections /t REG_DWORD /d 0 /f NetSh Advfirewall set allprofiles state off reg delete "HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion\Setup\State" /v ImageState /f reg add "HKEY_LOCAL_MACHINE\SYSTEM\Setup" /v OOBEInProgress /t REG_DWORD /d 0 /f reg add "HKEY_LOCAL_MACHINE\SYSTEM\Setup" /v RestartSetup /t REG_DWORD /d 0 /f reg add "HKEY_LOCAL_MACHINE\SYSTEM\Setup" /v SetupPhase /t REG_DWORD /d 0 /f reg add "HKEY_LOCAL_MACHINE\SYSTEM\Setup" /v SetupType /t REG_DWORD /d 0 /f reg delete "HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion\Run" /v DigitalOceanHack /f echo "Ok, reboot" >> c:\firstrun.log shutdown -r -t 20

ステップ2.ソースでルートFSをアンマウントし、パーティションを再パーティションします



前の例のようにramdiskを起動し、ルートFSを移動し、ディスクの先頭に新しいパーティションを作成し、grub設定を更新します。 

 e2fsck -f /dev/vda1 resize2fs /dev/vda1 25G fdisk /dev/vda Command: d Command: n Select (default p): p Partition number: 1 First sector: 2048 Last sector: +716799 ###      Windows = 716800   1 Command: n Select (default p): p Partition number: 2 First sector: 718848 Last sector: +32833535 ###        1 Command: n Select (default p): p Partition number: 3 First sector: 73400320 ###   25 = 52428800      125829120 . Last sector: +52428799 # 25   - 1 Command: t Partition number: 1 Partition type: 7 Command: t Partition number: 2 Partition type: 7 Command: a Partition number: 1 Command: a Partition number: 3 Command: w




最初のセクションはWindows用に起動可能、350 MB、2番目はWindows自体、3番目はそれぞれLinuxです。 これらの数字の由来は次のステップで明らかになりますが、今のところ、最初にランダムなサイズの2つのパーティションを作成して（最後のセクターがWindowsイメージのサイズより小さくならないように）、イメージがUbuntuファイルシステムにコピーされるときにUbuntuからそれらを再作成できます。 最終的に、Windowsのパーティションは、ソースイメージと同じ場所に同じサイズであることが重要です。

一時的にvmdkファイルをそこに置くので、ルートFSに多くを残しました。 理論的には、パイプを介してセクションにすぐに書き込むことができます。

次に、/ dev / vda3の2048セクターから始まる/ dev / vdaで25GBのUbuntu（まだディスクの先頭にある）をコピーします。 

 dd if=/dev/vda bs=4096 skip=256 count=6553600 of=/dev/vda3


（ブロックが512ではなく4096であるため、スキップとカウントは8倍削減されます）

必要に応じて、grub設定を更新し、/ etc / fstabを編集してください（/ dev / vda1を/ dev / vda3に置き換えてください）： 

 mkdir /mnt mount /dev/vda3 /mnt mount -o bind /dev/ /mnt/dev/ mount -o bind /proc/ /mnt/proc/ mount -o bind /sys/ /mnt/sys/ chroot /mnt PATH=$PATH:/usr/sbin grub-install /dev/vda update-grub2 nano /etc/fstab exit cd / umount /mnt/dev umount /mnt/proc umount /mnt/sys umount /mnt reboot




ステップ3.何らかの方法で、vmdkファイルを新しくロードされたubuntuにコピーします



さて、vmdkファイルがすぐにディスクイメージを表す場合（そうでない場合は、「qemu-img convert -p -f vmdk original.vmdk -O raw Converted.raw」を介してrawに変換する必要があります） 

 fdisk Windows2012r2.vmdk




ここで、実際には、/ dev / vda1および/ dev / vda2の値は前のステップからのものです。



Windowsを画像からセクションにコピーします。 

 dd bs=4096 skip=256 count=89600 if=Windows2012r2.vmdk of=/dev/vda1 dd bs=4096 skip=89856 count=4104192 if=Windows2012r2.vmdk of=/dev/vda2


（スキップとカウントも8倍に拡大されます）

/ dev / vda1と/ dev / vda2をマウントして、すべてが正常であることを確認し、必要に応じてcrutchスクリプトの設定を調整してください。



GRUBを更新し、彼がWindowsを見たことを確認します。 

 update-grub2


次に、次の再起動のためにWindowsをオンにします。何か問題が発生した場合に便利です。仮想マシンをオフにしてオンにすることで戻ることができます 

 grub-reboot 2 reboot


なぜなら sysprep + crutchスクリプトを使用してイメージを準備すると、Windowsは数分間隔で2回再起動します。 つまり さらに2回、ubuntuにログインして最後の2つのコマンドを実行する必要があります。



ステップ4.デフォルトでWindowsをブートする



grubのデフォルトにウィンドウを含める代わりに、単純に/etc/rc.localに追加することをお勧めします。 

 sleep 60 && grub-reboot 2 && reboot &


-Windowsの読み込みが停止した場合、Ubuntuにログインして「killall sleep」する時間を確保できます。



再起動後、RDPでアクセスできます。
