Grインターフェース





SGSNとHLR間のインターフェイス。これは、オペレーターのパケットネットワークに非常に必要です。 加入者の認証と承認の手順がGPRS接続中、またはIMSIとGPRSの組み合わせ接続手順中に通過するのは、彼を通してです。 このインターフェースは「純粋な」信号です。 サービス情報の送信のみを目的としており、SS7モデルに基づいており、MAPは上位レベルのメインプロトコルとして使用され（右の図を参照）、正確には、特別なバージョンはMAP-Hです。



Gdインターフェイス



これは、SGSNとSMS-GMSCの間のインターフェースです。 ショートメッセージを送信するためのセンターであり、オプションおよびオプションですが、パケットチャネルを介してショートメッセージを送信するための追加機能をパケットネットワークに導入します。 このインターフェイスのプロトコルスタックは、Grインターフェイスのスタック（SGSNとHLRの間）と同じです。 すべて同じ手順がSS7 MAPプロトコルに使用されます（Grインターフェイスのプロトコルスタック図を参照）。



パケットネットワーク（SMS over GPRS）を介してSMSを送信するためのサービスについては、記事-SMSの代替パスで詳細に検討されています。 オペレータは、このインターフェイスと、パケットネットワーク自体を介してメッセージを送信するためのサービスを実装することを望んでいないことを自分で追加できます。



GFインターフェース



ネットワーク要素データベースにあるオペレーターのIMEIコードデータベース（グレー、ブラック、ホワイトリスト）を照会することにより、加入者のIMEIコードを使用してモバイル端末を使用する正当性を検証できるSGSNとEIR間の別のオプションインターフェイスEIR。 プロトコルスタックはGr、Gdインターフェイスに似ています-MAPプロトコルが使用され（Grインターフェイスの図を参照）、正確にはMAP-Eバージョンです。



中央の国際データベースを使用する場合、または国が独自の内部IMEIコードデータベースを持っている必要がある場合、IMEIコード検証を使用する有効性は大幅に向上します。 そのため、多くのオペレーターは、 IMSI / MSISDNとIMEIの値のペアを照合するためのプラットフォームとして、要素のEIRプラットフォーム（IMEIコードをチェックする要求が行われます）を使用します。 加入者がSIMカードを変更するか、別のデバイスに挿入すると、「思いやりのある」多くの場合煩わしいMMS /インターネット/ WAP /などの設定を自動的に受け取ります。



Gsインターフェース



SGSNとMSCスイッチ間のもう1つのオプションのインターフェース。アクティブなGPRS / EDGEセッション中に受信機能とCS（サーキットサービス）サービスを実行する可能性を導入します。 これは非常に便利なインターフェースです 加入者は、アクティブなGPRS / EDGEセッション中の可用性について心配することなく、より快適に感じることができます。 私自身の経験から言えば、残念ながら、事業者のネットワークが異なるベンダーの機器で構築されている場合、スイッチ（ MSC ）とSGSNの両側のインターフェースを組み合わせることができない場合があるため、同じ事業者のネットワークでもGsインターフェースと、これが不可能な領域を実装できます。



アクティブなGPRS / EDGEセッション中に電話をかけたり受けたりするための追加情報については、電話を妨げるものではなくGPRSの記事に読者を送ります 。



Geインターフェース



加入者との決済取引のために請求データを転送するためのインターフェースは、SGSNと請求プラットフォーム-SCPの間に実装されます。 メインプロトコルとして、CAPモデルSS7アプリケーションが使用されます（Grインターフェイスの図を参照）。



ここで、少し余談をして、パッケージサービスを使用するための、加入者とオペレーターとの相互決済のプロセスについて説明します。 まず、2つのタイプのサブスクライバーがあることを覚えておいてください。

  • 前払い-前払いサービスの加入者
  • 後払い-契約者


プリペイド加入者の場合、請求はリアルタイムで実行されます。 Geインターフェイスを介してSGSNからPDPコンテキストを上げる（およびアクティブにする前に）とき、CAPプラットフォームを介してINプラットフォーム（ SCP ）にリクエストが行われます-請求プラットフォーム、加入者の残高の現在の残高について、そして（必要な残高がある場合）アカウント）特定の時間間隔（オペレーターによって設定されたタイマー）で、プリペイドサービス加入者のアクティブなPDPセッションの継続の可能性に対して繰り返し要求が行われます。



後払い加入者の場合、つまり 契約加入者の場合、請求データの収集は主にSGSN自体で行われます（ただし、GGSNでデータを収集する請求システムもあります）。 SGSNでは、各加入者に対して、いわゆる その後、tftp / ftpプロトコルを介して請求システムに転送され、加入者アカウントが計算されるCDRファイル。ただし、後払いの加入者向けのオンライン請求システムもありますが、オペレーターからはあまり配信されていません。ウクライナのオペレーター。



したがって、これら2つのタイプのサブスクライバーの主な違いは、プリペイドのいわゆる オンライン請求、および後払い-オフライン請求。



GAインターフェース



実際、完全な実装のインターフェースはオプションになりました。 多くの事業者はプリペイド加入者にGeインターフェースを使用し、契約加入者のCDRファイルはSGSN自体で生成されますが、それにもかかわらず、Gaインターフェースを使用して、GGSNに請求計算を実装することが可能です。 真の実装では、このインターフェイスはSGSNまたはGGSNをCG [Charging Gateway]に接続し、ほとんどの実装では、GTP`プロトコルをトップレベルとして使用するパケットインターフェイス（TCP / IP）です（下図を参照）。







ところで、ネットワークが1つのベンダーの機器で構築されている場合、つまり SGSN / GGSNおよびCGは1つのベンダーによって提供され、Gaインターフェイスはベンダーの独自開発に基づいて「完全に」使用できます。 オペレーターに対して閉じられます。



Gn、gpインターフェイス



SGSNとGGSNの接続を実装するために必要な2つの非常に類似したインターフェース。 これらの2つのネットワーク要素が同じPLMN （en）にある場合、Gnは何を使用し、要素が異なるPLMNにある場合はGpを使用します。 加入者は、ローミングでGPRS / EDGEサービスを使用します。



インターフェイスでは2種類のGTPプロトコルが使用されます。

  • GTP-U-ユーザーデータの転送用
  • GTP-C-サービス情報の送信用*


*-たとえば、PDPコンテキストをアクティブ化する場合、SGSNはGTP-Cプロトコルのみを使用してPDPコンテキストアクティブ化要求をGGSNに送信します。






ところで、加入者がローミングしている場合でも、つまりGpインターフェイスは必ずしも必要ではありません。 複数の追加条件が満たされている場合、このインターフェースの使用を拒否することができます（記事「 高価なGPRSローミング 」を参照）。
