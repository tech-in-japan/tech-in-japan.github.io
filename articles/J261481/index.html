建築



Veeam Endpoint Backupは、バックアップタスクを実行するサービスと、それを構成するアプリケーションの2つの主要コンポーネントで構成されています。 これらのコンポーネントは、バックアップが必要なマシンにローカルにインストールされます-Windows 7以降を実行しているマシンでなければなりません（システム要件の詳細については、 こちらを参照してください ）。



製品のインストールが完了すると、マシンに次のコンポーネントが表示されます。

  • Veeam Endpoint ServiceはMicrosoft Windowsサービスであり、実際、バックアップからのマシンのバックアップと復元を担当します。 サービスは自動的に開始され、バックグラウンドで常に実行されています。
  • Veeam Endpoint System Tray Applicationは、バックアップまたは復元プロセスを開始し、そのステータスを監視し、製品のコントロールパネルを起動できるシステムトレイアプリケーションです。
  • Microsoft SQL Server 2012 LocalDBは、構成データが保存されるSQL Serverの「ライトバージョン」です。 SQLiteの類似物と呼ぶことができます。これは、Webブラウザーまたはユーザーソフトウェアの操作データを保存するためによく使用されます。LocalDBには最小限のリソースしか必要ありません。 主な違いは、SQLiteは通常、アプリケーション自体の内部に埋め込まれて実行されるのに対して、LocalDBはVeeam Endpoint Serviceによって起動されるサブプロセスで動作することです。 Veeam Endpoint Backupがアイドル状態の場合、リソースの消費（ベースとプログラム自体の両方）はゼロになる傾向があると言えます。 一般に、バックアップを実行する場合、プロセスは低い優先度で実行されるため、リソース消費はそれほど大きくありません。






Veeam Endpoint Backupは、 Veeam Backup＆Replication v8 Update 2以降と統合されているため、既存の仮想マシンバックアップリポジトリを物理マシンのバックアップに使用できます。 これらのバックアップをリポジトリに保存するには、Veeam Backupサーバーを登録してVeeam Backup＆Replicationインフラストラクチャへの接続を設定する必要があります。この場合、データはサーバーをバイパスしてリポジトリに直接保存されます。 しかし、それについては後で。

バックアップスキーム



Veeam Endpoint Backupは、フルバックアップと、設定された保持期間（N日間）に達するまで保存される後続の増分の「エンドレス」チェーンを作成します。 N日目に、チェーンの最後の増分が記録され、バックアップジョブの次の実行時に、次のことが行われます。

  • 次の増分は、チェーン内の最新の復旧ポイントとして記録されます（直接増分バージョンと同様に、1回の読み取り/書き込みサイクルを使用）。 この場合、バックアッププログラムは、リカバリポイントの数が設定された制限より1大きいと自動的に判断します。
  • 次に、プログラムは、最も古いファイルがチェーン作成の最初に保存されたフルバックアップであり、最も古い増分が最初のフルバックアップの後の行の2番目であることを確立します（特定のバックアップサイクルの1ラウンドのフルバックアップよりも新しい） ;
  • チェーン内の最も古い増分がフルバックアップファイルに埋め込まれ、対応する古いデータブロックが置き換えられます。 これには2つの読み取り/書き込み操作が必要です。1つは増分からデータを読み取り、2つ目はこのデータを完全バックアップファイルに書き込むためです。






  • 増分ファイルはチェーンから削除され、更新された完全バックアップによってその場所が取得されます。 最も古いフルコピーは、保存期間が終了した増分を徐々に「吸収」すると言うことができます。






時間の経過とともに、このような操作は、新しい増分がチェーンに追加されるたびに何度も繰り返されます。



このバックアップスキームの動作を説明するアニメーションは 、「フォワードインクリメンタルバックアップ方式」セクションのナレッジベース記事「 フォワードインクリメンタル-メソッドと保持のアニメーション 」にあります。



この方法は、冗長システムの負荷を最小限に抑えるという点で最も最適であるため、Veeam Endpoint Backupの唯一のオプションとして選択されていることに注意してください。 これは、平均して、バックアップデータの各ユニットに3つのディスクI / O操作が必要であり、そのうち2つはバックアップコンピューターではなくストレージで実行されるため、ソースディスクの負荷が最小限に抑えられるためです。 これにより、バックアップコンピューターを他のバックアップ方法よりもはるかに高速で集中的に使用できるようになり、バックアップジョブ中に、増分変更が読み取られるとすぐにディスク負荷が除去されます。 バックアップがホームNASに行く場合、負荷は中央処理装置から大部分削除されます。



バックアップ保持期間



古いバックアップを削除したことがない場合、しばらくすると、ストレージの大きさに関係なく、空き領域がすべてのストレージで終了します。 ディスクスペースの最後のイベントは突然発生し、誤動作につながる可能性があります。 さらに悪いことに、それは長い間気付かれず、その結果、バックアップは作成されません。 Habrで、このような実際のケースの例については、「 Pixarのケース、またはバックアップのテストの重要性について 」という投稿で既に説明しています。 そのため、リポジトリにバックアップを保存するためのポリシー（保持するバックアップ）をすぐに構成することが重要です。





Veeam Endpoint Backupでは、保持期間は次のようにカウントされます。日は暦日とは見なされませんが、少なくとも1つのバックアップが正常に作成された日です。 たとえば、「コンピューターが使用された最後の5日間のバックアップポイントを保持する」を設定します。 同時に、週の最初の3日間（月曜日、火曜日、水曜日）に3つの成功したバックアップを作成しました。たとえば、木曜日と金曜日にコンピュータの電源を切りました。 次に、土曜日にそれをオンに戻し、土曜日と日曜日にさらに2つのバックアップを取得しました。 7日間で5つのバックアップが作成されたことがわかりました。 その後、月曜日が再び来て、別のバックアップが作成され、最も古いバックアップが古いものとして削除されます。 ポリシーで規定されているとおり、5つのバックアップが再びあります。



スケジュール



ここで、「スケジュール」ステップが提供するものを見てみましょう。 オプションには2つのグループがあります。1つはバックアップの起動の頻度を設定でき、もう1つは起動を特定のイベントにバインドします。





1）毎日のバックアップ （毎日）-たとえば、午前2時に コンピューターが同時に休止状態だった場合、Veeamは「ウェイクアップ」し、バックアップを作成してから、コンピューターをリストから選択した状態にします（これは、Keep Running、Sleep、Shutdown、Hibernateのいずれかです）。



注 ：選択された状態への転送は、[毎日]オプションが選択されている場合にのみ機能します（つまり、スケジュールに基づく定期的な自動バックアップ）。 手動でバックアップを起動した後、そのような転送は実行されません。 さらに、次のことを考慮してください。

  • デスクトップコンピューターのWindowsにインストールされている電源プランが「起動」を許可しない場合、Veeam Backupはそれを修正することを提案します。
  • タブレットでは、多くの場合、デフォルトの接続スタンバイがインストールされます。そのため、自動「起動」も機能しません。


2）イベントバックアップ - バックアップを開始するタイミングを選択できます：ロック（ロック）、ログオフ（ログアウト）、および「バックアップターゲットが接続されたとき」（USBフラッシュドライブ、外部USBドライブ、または以前に保存したネットワークパケットの接続）バックアップ）。 最後のオプションは、バックアップメディアが使用可能になるとすぐにバックアップが開始されることを前提としています。
