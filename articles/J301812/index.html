KitchenCIを使用してSaltStackでプロジェクトをテストする

はじめに





ペットプロジェクトがあります。これは自由時間に行います。 このプロジェクトは、インフラストラクチャの実験専用です。 構成管理には、 SaltStackを使用します 。 SaltStackは、集中型のインフラストラクチャ管理システムです。 これは、スレーブサーバーをセットアップするマスターサーバーがあることを意味します。







プロジェクトの存続期間中、私は小さな一連のレーキを踏みましたが、最終的には非常に便利なアプローチで作業するようになりました。 一般に、これと記事について-それがすべて始まり、どこに来たのか。







木が大きいとき



プロジェクト全体はモノリシックで、すべてが揃っていました。







  • 状態（states）-構成方法と構成に関する指示。
  • データ構造（柱）-州で使用されるデータ。 例：

    • いくつかのタスクのシステムパッケージのリスト。
    • Dockerコンテナーからの状態で使用されるDockerハブからのログイン/パスワード。

      サーバーのリスト、およびそれらに割り当てられた状態とデータ。


プロジェクト全体が1つのgitリポジトリにあり、 gitfsを介してマスターサーバーに接続されていました 。 これは非常に便利です-マスターサーバー上のデータの更新を心配する必要はありません。 SaltStack自体は、リポジトリからすべてを収集します。







gitリポジトリの別のブランチを使用して、インフラストラクチャのテストコピーを作成し、それを介してすべてをテストできます。 ただし、テスト用のインフラストラクチャのコピーを作成するには費用がかかります。







  • 雲の場合はお金のため。
  • いずれにせよ、時間内に-作業状態を取り、実行し、維持する必要があります。


一方、「戦い」は1つの連続した「テスト」であり、それを破っても問題ありません（「恐れない」のは残念です）。 怖くないので、中間の変更を含むすべての変更を、リポジトリへのプッシュを介して展開しました。 控えめに言っても、コミットログは不気味に見え始めました。







  • パッケージのインストールの問題を解決してみてください...;
  • エラーを修正する別の試み...;
  • 魔法...;
  • まあ確かに;
  • さて、確かにすべての番号2。
  • 前回忘れたいくつかの変更。


実際、すべてがそれほど悪くなかったわけではありませんが、一般に、この例では正しい画像が得られます％）







それはさらに悪化しました-時間の経過とともに、私はそれがどのような状態で、どのようにそれをしていたかを忘れ始めました。 それでも、これは個人的なプロジェクトであり、私はそれに常に取り組んでいるわけではなく、時々取り組んでいます。 READMEファイルはすでにこの問題をうまく解決していません。







データを介した接続の状態もありました。 ある状態でデータ構造が変更された場合、異なる状態では同じデータが使用され、他の状態では破損が保証されました。 このため、ある時点で、「戦闘」構成は「ガットアウト」状態になります。 私は数日間バグを修正できます。つまり、その時点でいくつかの条件が機能しなくなる可能性があります。 一般に、これはプロジェクトアーキテクチャの設計が不十分であることを示しています。







しかし、これらすべてのマイナスはあまり気にしませんでした。 私のモードでは、彼らと一緒に暮らす準備ができていました。 データ構造内の何かを変更して、すべての状態を確認するのを忘れていました。 次に、このような「遅延」エラーを長くてつまらないものにします。







解決策はテストを書くことです



テストを書くと、何かを変更すると、自動テストがすべての状態の結果をチェックするという保証があることに気付きました。 やった！ すべてが非常に簡単です。 タスクは明確です。プロジェクトの状態の結果を確認したいです。







それでは、テストのために何がありますか？ SaltStackの成果は、構成ファイル、サービス、Dockerコンテナー、ファイアウォール設定、SElinuxなどです。 これはすべてServerspecテストを使用して完全にテストされます 。







私はこのテーマで出会った記事を思い出すために、私がいた会議を思い出し始めました。 一般に、ロシア語では、実際の善良な人から、1人の著者だけが頭の中で回転していました。IgorKurochkin IgorITLで 、DevConf'e 2015でのライブを聞きました。彼のレポート「コードとしてのインフラストラクチャのテスト」







  • ビデオ （すでにRootconfですが）
  • スライド
  • 記事の形式に改訂された同じレポート


また、 「アジャイルDevOps：テスト駆動型インフラストラクチャ」の問題を理解するための良い記事を見つけました。









すべての資料を読んだ後、 KitchenCIツールが私のタスクに適していることに気付きました。







  • SaltStackで動作します。
  • Vagrant、Docker、lxc、さまざまなクラウドなど、どこでもインフラストラクチャコードを実行します。
  • テストフレームワークをサポートしています：コウモリ、RSpec、Serverspecなど。


今ではすべてを知っているようだ。 理論があり、すべてが私の頭の中でうまくいきました。今では、テストの作成を開始することができますよね？







最初のパンケーキはゴツゴツしている



私はプロジェクトを見て、どういうわけか私の頭の中の理論が私の現実にまったく合わないことを見ました。 私のプロジェクトにアプローチする方法は？ テストを配置する場所は？ それらを実行するには？







答えを探して、私は再びKitchenCIのドキュメントを注意深く読みました。 残念ながら、このツールのChefでの専門化とその機能は非常に注意をそらします。 例はすべて彼のためです。







KitchenCIを詳しく見てみましょう。 このツールでは、次のオブジェクトを操作します。







  • ドライバー -KitchenCIが仮想マシンを起動するプラグイン。 たとえば、浮浪者、港湾労働者、指の移動など。 デフォルトでは、vagrantが使用され、完全に私に合っています-ローカルでテストを実行したいです。
  • エンジン （プロビジョニング担当者）。設定を説明します。 デフォルトでは、これはマスターレスモードのChefです。
  • platform-テスト仮想マシンのベースとして使用されるイメージの名前。
  • 実行するテストスイート （スイート）。 何も変更しない場合、KitchenCIはデフォルトディレクトリでテストを見つけようとします。これはデフォルトセットの名前です。


SaltStackを使用します。 Googleは、SaltStackのプロビジョナーsalt_soloを実装するサードパーティの'kitchen-salt'プロジェクトがあると言っています。 また、詳細なレッスンと使用方法の例もあります。







KitchenCIとkitchen-saltのドキュメントを読んだ後、私は主なものを思いつきました-個々のレシピは設定全体ではなく（Chefの用語で）テストされます。 SaltStackでは、シェフのレシピは式に似ています-独立したプロジェクトに提出される独立した状態。 これらの式は、他のプロジェクトでコードを再利用するために使用されます。 たとえば、 GitHubにはこのような数式が多数用意されています 。







これが私のプロジェクトがKitchenCIに「適合しない」主な理由です-それはモノリシックです。 「リファクタリング」、「コードの一貫性」、「モジュラーアプローチ」などの言葉が頭の中で回転しました。 悲しかった。 プログラマーとして、私はそのような言葉を知らないはずです。







プロジェクトのリファクタリング



チャレンジを受け入れました！ 最初のリファクタリングルールを覚えている限り、明確で達成可能な測定可能な目標が必要です。 通常、これは「なぜ変更を行うのですか？」という質問に対する詳細な回答です。 私の場合、次のように作成されました。







  • メインプロジェクトのすべての状態は、個別の子数式プロジェクトに提出する必要があります。
  • 各式には、説明が記載されたREADMEファイルが必要です。
  • 各式には、この状態が予期するデータストレージ構造の例とともに、 pillar.example



    ファイルを添付する必要があります。
  • 各公式は、公式文書に記載されている要件と推奨事項に従って構成する必要があります 。


タスクのリストを作成した後、私は再び悲しく、コーヒーを飲み、それをやりに行きました。 状態が次々に別の数式に変わった。 メインプロジェクトから状態を削除して、マスターサーバーの構成に新しい数式へのリンクを入力しました。 したがって、プロジェクトのパフォーマンスは、全体の変更中に特に低下しませんでした。







いくつかの州の連結性により、データストレージ構造を再検討する必要がありました。より独立させ、異なる式のために重複しない部分に分割し、同時にデータの重複を避けようとしました。 これはおそらく最も難しいものでした。 このため、一部の状態のロジックの一部が他の状態に移行されました。







作業の結果は、使用されたデータの例と最小限のドキュメントを備えた、簡単で理解可能なほぼ20個の独立した式でした。 結果のデータ構造は著しく単純になりました。 この段階でさえ、私は肯定的な結果を感じました-私は今、私の式が独立していることを知っており、それらを大胆に変更することは可能です。







テスト中



テストの対象として別の数式を検討し始めるとすぐに、KitchenCIの使用方法についてすぐに頭に浮かびました。 例として、最も単純な公式「共通パッケージ」を使用して、テストプロセスを見てみましょう。 この式は、私のサーバーのいずれかで満たす予定のシステムパッケージをインストールします。 これらは私になじみのあるユーティリティです。







NB！ さらに下の行では、すべてのコマンドが数式プロジェクトのルートで実行されます。







これは、式の元のファイル構造は次のようになります。

はい、KitchenCIは次のことを行いました。







  1. CentOS 7に基づいて仮想マシンを作成しました。

ほほほ 中間の変更をマスターにコミットして「バトル」のためにレイアウトすることなく、数式を作成し、それらのバグを修正できるようになりました。 「戦闘」インフラストラクチャは著しく安定しており、私のコミットログが突然来たとしても恥ずかしくないようです。







マシンの内部に入ると、式の結果を手で見ることができます。 







 $ kitchen login centos-7.2


KitchenCIを使用して、数式を実行してそのパフォーマンスをテストする方法を学びました。 手で確認するのは素晴らしいことです。 しかし、自動テストはどこにありますか？ 自動テストで式の結果を確認しましょう。

これを行うには、次の手順を実行します。







  • ディレクトリ./test/integration/default/serverspec



    作成します
  • そして、その中にpackages_spec.rbファイルを配置します


packages_spec.rb

注意！ 接尾辞_specが必要です。 これと他のニュアンスについて読むことができ、一般に、公式Webサイトhttp://serverspec.org/でServerspecを知ることができます。 







 require 'serverspec' # Required by serverspec set :backend, :exec describe package('bind-utils') do it { should be_installed } end describe package('whois') do it { should be_installed } end describe package('git') do it { should be_installed } end describe package('psmisc') do it { should be_installed } end describe package('mlocate') do it { should be_installed } end describe package('openssl') do it { should be_installed } end describe package('bash-completion') do it { should be_installed } end describe package('net-tools') do it { should be_installed } end


時間を節約し、マシンが作成されてゼロから設定されるまで再び待たないように、KitchenCIにテストを実行してもらいましょう。 







 $ kitchen verify centos-7.2








それがすべての魔法です。







KitchenCIを使用すると、上記のすべての手順を1つのコマンドで実行できます：kitchen test。 仮想マシンが作成され、式とテストが実行された後、マシンが破棄されます。







機能テスト



kitchen-saltは、個々の式だけでなく、そのセットもテストできます。 つまり、いくつかの式の最終結果を十分にテストできます。 このテストは、数式が連携して機能するかどうか、および期待される結果が得られるかどうかを示します。 これはすべて、プロビジョナーのオプションのさまざまな組み合わせにより可能です： https : //github.com/simonmcc/kitchen-salt/blob/master/provisioner_options.md これは、KitchenCIとテストをプロジェクトの初期フォームに添付できることを意味しますが、最終的にははるかに良くなったようです。







結論



今では、古い式をゆっくりとテストでカバーし、新しい式を作成し、以前よりはるかに高速に作成します。 そして、いつでも、新しいものと古いものの両方の式の効率に自信があります。 はい、リファクタリングとテストの作成に時間を費やしましたが、ポケットプロジェクトでの作業が明らかに増加しました。 現在、プロジェクトを長期間延期する心配はありません。プロジェクト自体の複雑さや、数式が機能しない理由が不明なため、プロジェクトを継続できません。 はい、リファクタリングは私の個人的な時間の数日をむさぼり食いました。 はい、テストを書くのは退屈ですが、プロジェクトに自信を与えます。 涼しい感じ。
