ExtJSでの複雑なアプリケーションの作成。

まえがき



この記事は、単純なウィンドウまたはフォームを作成する組み込みスクリプトを備えた単一のHTMLページを既に成長しているExt 2.xユーザー、Extが自分の道であると既に決心している人々、および克服する人々大量のコードに関連する問題は、構造化する必要があることを理解しています。

何人、とても多くの意見。 したがって、以下で説明する方法は、唯一可能な方法ではありません。 また、このアプローチを使用して作成されたすべてのアプリケーションが適切であるとは限りません。 種類はありません。

説明したアプローチは実行可能であり、明確に構造化されており、簡単にサポートされ、一言で言えばワーカーです！



複雑なアプリケーションとはどういう意味ですか？



1つのファイルでBorderLayout、グリッドテーブル、およびフォームを使用してビューポートオブジェクトを操作する場合、これは確かに複雑なアプリケーションではありませんか？ グリッドテーブル、フォーム、またはBorderLayoutのあるウィンドウが多数あり、これらすべてが多数のファイルに散在している場合、これはすでに複雑で大規模なアプリケーションですよね？

ドイツ語にはいい言葉が1つあります。Jein= Ja + Nein

これらの両方のステートメントに対する答えはJeinです。 問題は、いつアプリケーションが大きく複雑になるのかということです。 答えは簡単です。まさにそのように感じ始めた瞬間です。 これは、たとえば、コンポーネントの関係などを理解しようとする場合に、特定の場所を検索するときに多数のファイル内をナビゲートすることが困難になるか、問題が始まる瞬間です。

各アプリケーションは簡単に作成するのは簡単ではなく、新しい機能の追加、コード量の増加、CSSルールの追加などを開始するとすぐに大きくなる可能性があると言っても差し支えありません。

新しいアプリケーションを作成するときの最適で安全なインストールは、「複雑なアプリケーションを起動しています！」です。



ファイルとディレクトリ



最初に整理する必要があるもの。 DocumentRootディレクトリはいつでも自由に使用できるため、以下で説明するすべてのサブディレクトリは、それに関連して一覧表示されます。

推奨ディレクトリ構造：

  • ./css（オプションのリンク）
  • ./ext（リンク）
  • ./img（リンク）
  • ./js
  • index.html


「リンク」という言葉は、ディレクトリがファイルが置かれている実際のディレクトリへのソフトリンクであることを意味します。 この手法の利点は、たとえば、Extの新しいバージョンを任意の実際のディレクトリにダウンロードして、リンクがそれを指すように強制できることです。これにより、コード内のパスを編集する必要がなくなります。 新しいバージョンをテストし、すべてが正常である場合は、リンクを残して新しいバージョンを指すようにし、そうでない場合は元に戻します。



  • cssは、すべてのスタイルのリポジトリです。 会社の色やフォントデザインなどのグローバルスタイルがある場合、このディレクトリをリンクとして作成することもできます。
  • ext-前述のように、使用しているExtのバージョンへのリンク。
  • img-画像のあるディレクトリ。 アイコンなどのサブディレクトリも含まれる場合があります。
  • js-アプリケーションのすべてのJavaScriptファイルと、それらのアセンブルされたバージョンが含まれます。
  • index.html -HTMLファイルは、アプリケーションのエントリポイントです。 好きな名前を付けることができます。たとえば、認証操作用に複数のhtmlファイルを作成できます。 ただし、いずれにしても、エントリポイントファイル（onReadyハンドラーを含む）は1つでなければなりません。
  • さらに、アプリケーションのサーバー部分（私の場合は./classes）へのディレクトリまたはリンクを作成できます。 彼女に任意の名前を付けることができますが、開発中のすべてのアプリケーションと一致する名前を選択する必要があります（./server、./phpという名前が適切なオプションです）。


index.html



index.htmlの最小コンテンツは次のとおりです。 

  <html>
 <head>
   <meta http-equiv = "Content-Type" content = "text / html; charset = utf-8">
   <link rel = "stylesheet" type = "text / css" href = "./ ext / resources / css / ext-all.css">
   <link rel = "stylesheet" type = "text / css" href = "./ css / application.css">
   <script type = "text / javascript" src = "./ ext / adapter / ext / ext-base.js"> </ script>
   <script type = "text / javascript" src = "./ ext / ext-all-debug.js"> </ script>
   <script type = "text / javascript" src = "./ application.js"> </ script>
   <title>大きなアプリケーション</ title>
 </ head>
 <body> </ body>
 </ html>


上記のようなファイルを操作できますが、このタイプのファイルと他のすべてのファイルにヘッダー情報を追加することをお勧めします。 ファイルの終わりマーカーも重要です。 この種類の見出しの例を参照してください。

css / application.css



もしあれば、すべてのスタイルをこのファイルに入れてください。 必要な数が少ない場合は、それらに個別のファイルを作成しても意味がありません。 を使用して、ドキュメントに直接書き込むことができます.





それどころか、複雑なアプリケーションを作成していることを忘れないでください。 ドキュメントのタイトルにスタイルを記述すると、遅かれ早かれレンダリングの問題を解決する必要があり、どのスタイルがそれらに影響を与えたかはわかりません。

正しい方法：分割して征服する



全体は、そのサイズに関係なく、いくつかの要素を含むさらに小さなパーツで構成される小さなシステムで構成されます。 開発した複雑なアプリケーションも例外ではありません。 そして今こそ、これらの部品、コンポーネント、およびそれらの間の関係を自分で判断するときです。

ですから、もっと便利に座って、よく考え、スケッチを描き、リストを作成します。何をするかは関係ありません。その結果、アプリケーションを構成するコンポーネントのリストが得られます。 少なくとも主なもの。

定義済みのクラス



さて、アプリケーションのコンポーネントの分析と決定が完了するとすぐに、それらの1つを書き始めることができます。 これを行う最良の方法は何ですか？ 最善の解決策は、標準のExtコンポーネントを拡張するクラスを作成することです。後者は、コンストラクターに渡される値によってすべての設定が既に上書きされているためです。 私はこれらの拡張機能を事前に構成されたクラスと呼んでいます 新しい機能を導入することはめったになく、主に構成に使用されます。 例は、独自の列モデル、ストレージ、並べ替え設定、エディターなどを備えた「Personnel」グリッドテーブルです。

この場合、ウィンドウの構成は次のようになります。 

  var win = new Ext.Window（{
     タイトル： '人事'
    幅：600
     、高さ：400
     、アイテム：{xtype： 'personnelgrid'}
 }）;
 win.show（）;

例を見てみましょう： 

  Application.PersonnelGrid = Ext.extend（Ext.grid.GridPanel、{

     、initComponent：function（）{

     } // eo関数initComponent

     、onRender：function（）{
         this.store.load（）;

         Application.PersonnelGrid.superclass.onRender.apply（this、arguments）;
     } // eo関数onRender
 }）;
 Ext.reg（ 'personnelgrid'、Application.PersonnelGrid）;


ここで何が起こっていますか？ Ext.grid.GridPanelを拡張し、新しい拡張クラスApplication.PersonnelGridを作成して、 personpersongridと呼ばれる新しいxtypeを登録します。

実際、特殊な「人事」グリッドテーブルに変換するのに十分な設定をすべて通常のグリッドテーブルに転送します。 この時点から、アプリケーションのビルディングブロックである新しいコンポーネントが作成され、どこでも（ウィンドウ内、パネル上、独自に）使用して従業員のリストを表示できます。 次のように作成できます。 

  var pg = new Application.PersonnelGrid（）;


またはxtypeを使用して（いわゆる遅延作成）：

     アイテム：{xtype： 'personnelgrid'}
     、....
 }）;


事前構成されたクラスの編成と保管



上記のコードは必要ではなく、onReadyハンドラーで実行されません。DOMで動作するのではなく、単にJavaScriptオブジェクトを作成します。 したがって、別のファイル（ js / Application.PersonnelGrid.js ）に配置する必要があり、そうする必要があり、 index.htmlヘッダーに含まれます。

さて、これまでのところすべてが順調に進んでいます-ほぼすべての準備が整っており、事前に構成されたクラスの説明を続け、それらを./jsに入れ、 index.htmlにインクルードし、インスタンスからアプリケーションを断片としてアセンブルしますパズル。

相互接続メッセージ



左側（西部地域）にリンクのリストがあり、中央（中央部地域）にブックマークバーがある境界線レイアウトが必要だと想像してください。 リンクをクリックすると、中央に新しいブックマークが作成されます。 発生するすべてのロジック、イベントハンドラー、作成コードをどこに配置しますか？ 左または中央？

どこにもありません。 なんで？ 左側にリストを作成して表示する事前構成されたクラスがあり、そこにロジックを配置すると、中央領域がなければその存在は意味を失います。 ブックマークバーなしではリストを使用できません。

ロジックを中央領域に配置すると、結果は同様になります。ブックマークのバーはリンクのリストなしでは存在できません。

左パネルと中央パネルの存在を実現するコンポーネントは1つだけです。これは、フレームプロトタイピングを備えたコンテナであり、コンポーネント間メッセージのロジックを配置する唯一の適切な場所です。

生産システム



アプリケーションの編成に関する合意により、非常に早い段階で多数のJavaScriptファイル（私の場合は約80で毎日数が増えています）に遭遇しますが、これは実際に動作する非テストシステムのパフォーマンスに悪影響を及ぼす可能性があります。



最善の解決策は、すべてのJavaScriptファイルを正しい順序でマージ（連結）し、1つの大きなファイルを作成することです。このファイルは、縮小ツールまたは圧縮ツールのいずれかで圧縮する必要があります。

実動システムが接続します：

  • ext-all.js
  • app-all.js
  • application.js


コードの圧縮およびビルドファイルの作成に関する追加情報は、 別のガイドに記載されています 。
