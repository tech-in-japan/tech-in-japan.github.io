Fremantle SDK（Maemo 5）のクイックインストール。 Qt 4.6をインストールします。 エミュレーターとデバイスで実行およびデバッグする

良い一日habrasociety。 Maemoの一連のプログラミング記事を継続します。 以前に、「ハイブリッド」モードでSDKをインストールおよび構成する方法を書きました。ネイティブOSで記述し、仮想マシンで収集して実行します。



Maemoのプログラミングを始めるのは難しいと考える人もいました。 だから、私は簡単な方法「簡単な方法」©のデモンストレーションから始めて、それがいかに簡単であるかを示すことにしました。 さらに、 maemo 5のt 4.6.2の公式リリースがリリースされた場合、このリリースをFremantle SDK に組み込む方法を示します。 エミュレータでデバッグを実演します。 そして最も重要なのは、通常のネットワーク接続とUSB接続を使用してデバイス上のアプリケーションをデバッグする方法です。



作業の結果は、デバイス上のそのようなウィンドウになります。







注：Scratchboxは、Fremantle SDKであるため、IDEとして使用されます。 QtQreatorは使用できますが、Fremantleは使用しません。 なんで？ はい、彼らが新しいクロスプラットフォームSDK-MADDEをサポートしているので、すでに言及しました（そして結論で言及します）。







まあ、あなたはカットされているので、それはあなたが私の記事を読むことにしたことを意味します。



1. SDKをインストールし、エミュレーターでHelloWorldを実行します。





ここにあるすべてのビデオは非常に良い品質ではありません。YouTubeでは、HDフォーマットで見ることができます。ビデオをダブルクリックするだけでこれを行うことができます。



初めから始めましょう。この時点で、 前の記事を読んでいないか、SDKをインストールするためのやや長い道のりを踏まないことを決めたとします。 しかし、最初の道を進んだ場合、結果の賞品として、ネイティブOSでの開発に便利な環境が得られます（個人的には、もちろん私の意見です）。

しかし、何らかの理由で怠け者の場合、私が説明したオプションがあなたに合わない場合は、別のより簡単でシンプルな方法をお勧めします。



既にFremantle SDKをお持ちの場合は、この手順をスキップして2番目の手順に直接進んでください。



SDKをインストールするには、ダウンロードする必要があります。

  1. Fremantle SDK仮想イメージ -Maemo_Ubuntu_Intrepid_Desktop_SDK_Virtual_Image_Final.7zというファイル。
  2. オプションの7ipアーカイバ（お持ちでない場合）。
  3. VirtualBoxまたはquemuまたはVMWare。 私の例では、VirtualBoxを使用しています。




そして、ビデオに従って進んでください、すべてがそこにはっきりしているようです：





注：ゲストOSのアドオンは、SDK自体に含まれているアドオンではなく、VirtualBox自体から直接インストールします（デスクトップ上のアドオンへのリンク）。 どこにでもダウンロードする必要はありません。VirtualBox自体がすでに付属しており、ディストリビューションは仮想ディスクイメージとして接続されています。



ご覧のとおり、アプリケーションのインストール、構成、および起動は、前述の方法とは異なり、10分未満で完了します。 すべてが非常に簡単です



2. Qt 4.6.2をインストールし、エミュレーターでQtアプリケーションを作成して実行します。



この時点から、前の記事またはポイント1から提案されたインストール方法を使用するかどうかは関係ありません。すべてが両方のインストールケースに関連しています。







ここでのシーケンスは次のとおりです。



まず、SDKにインストールされているQt 4.5を削除します。

fakeroot apt-get remove libqtv4 *



そして、Maemo5の公式Qt 4.6.2をインストールします。

fakeroot apt-get install libqt4-maemo5-dev





そして、ARMELとx86の両方の目的でこれを行う必要があります。 x86ターゲットはエミュレーターで実行およびデバッグするために必要であり、ARMELターゲットはデバイスで実行およびデバッグするために必要です（クロスコンパイル）。 メニューを使用して、scratchboxで目標を切り替えることができます。

sb-menu





今注目!!!! scratchboxのネットワークが機能しない場合があります（パッケージをインストールするコマンドを実行すると、ネットワークにエラーが発生する場合があります）。 これは、scratchboxの小さな問題によるものです。 それらは簡単に除去できます。 これを行うには、scratchboxの外部で、各目的およびDHCP経由で受信した、または静的ネットワークのセットアッププロセス中に作成されたスクラッチボックス自体のresolv.confファイルを（明示的または間接的にユーティリティを介して）コピーする必要があります。

sudo cp /etc/resolv.conf / scratchbox /など

sudo cp /etc/resolv.conf / scratchbox / users / maemo / targets / FREMANTLE_ARMEL / etc

sudo cp /etc/resolv.conf / scratchbox / users / maemo / targets / FREMANTLE_X86 / etc





また、プロキシを使用する場合は、環境変数をエクスポートする必要があることに注意してください。

export http_proxy = ...

export https_proxy = ...



これを別のスクリプトに追加するか、これらの行を〜/ .bashrcファイルに追加するか、apt-getのプロキシを指定します。 

  echo 'Acquire :: http :: Proxy "http：// aptcache：3142";'  > /etc/apt/apt.conf.d/99proxy




この場合、apt-getはこのプロキシを介して機能します。



また、パッケージをインストールする前に、scratchbox内の各ターゲットのパッケージを更新するためにQtがずれることはありません（一般的に、「fakeroot」で始まるすべてのコマンドはscratchbox内でのみ実行され、sudoの外でsudoが実行されます）：

fakeroot apt-getアップデート





その後、Qtプロジェクト「Qt Hello World」を作成し、ビルド構成としてARMELおよびX86のすべてのターゲットを選択します（結局、エミュレーターとデバイスの両方でデバッグする予定です）。



3.ネットワーク経由でローカルエミュレーターおよびデバイス上でアプリケーションを構成および実行します（私の場合はWiFi経由）。





Qtプロジェクトを構築するには、これだけでは不十分です。事実、ESBoxは、システムに統合される非公式のQtとは異なり、/ opt / qt4-maemo5にインストールされる公式のQtの存在を認識しません（異なるディレクトリに散在し、qmakeは/ usrにあります） / local / bin、include is in / usrt / include / qt4など）。 原則として、非公式のQtパッケージの動作は標準ですが、この場合、いくつかのバージョンに対処するのが難しいため、Qtユーザーがパッケージをシステム全体に散らさないことにしたのでしょう。 これを修正するには2つの方法があります。プロジェクトを作成するときに、すべてのパスをIncludeとqmakeに変更するだけですが、簡単にするか、シンボリックリンクを作成できます。

qmakeの場合（ビデオのコメントにタイプミスがあります。パスに「dev」ではなく「bin」と書く必要があります。残念ながら、技術的な理由で修正できません）：

ln -s / opt / qt4-maemo5 / bin / qmake / usr / local / bin



パスを指定せずにqmakeが使用可能であることを確認します。

qmake



qmakeのプロンプトが表示され、最初にQtのバージョンが示され、4.6.2になります。



含めるため：

ln -s / opt / qt4-maemo5 / include / usr / local / qt4



ESBoxに必要です。



この操作は、スクラッチボックスアセンブリ（ARMELおよびX86）のすべての目的で実行する必要があります。これにより、あるバージョンのQtを備えた端末でデバッグし、別のバージョンのデバイスでデバッグしていることがわかりません。 scratchboxでターゲットを切り替えるには、sb-menuユーティリティを使用する必要があることを覚えています。



次に、エミュレーター（X86-develop）用のアプリケーションをビルドし、デバッグ構成（ローカル）を作成します。 まず、エミュレータを起動する招待に同意します。 プログラムが「メイン」機能へのエントリポイントで停止し、HelloWorldを獲得したことがわかります。 すべてがうまく機能します。 やったー！



次に、デバイスでデバッグを実行します。 最初に、ネットワークを介したデバッグのデモを行うことにしました。 必要なのは、デバイス（デバッグ用）およびsshサーバーにgdbサーバーをインストールすることだけです（クライアントはどちらも傷つけません）。

Open SSHクライアントとサーバーは、このメタパッケージmaemo.org/downloads/product/Maemo5/opensshを使用してインストールできます。 リポジトリーrepository.maemo.org/extrasにあります 。 私の記事で新しいリポジトリをインストールする方法をスパイできます。 また、アプリケーションのリンクに移動して、デバイスの[インストール]ボタンをクリックすることもできます。



デバイスに直接gemoをmaemoリポジトリからインストールできます。これはリポジトリにあります

deb repository.maemo.org fremantle / sdk free non-free

deb repository.maemo.org fremantle / tools free non-free



これらの行をデバイスの/etc/apt/sources.listファイルに追加するだけで（または標準パッケージマネージャーを使用）、スーパーユーザーである必要があるため、コマンドを入力する必要のあるデバイスでこのモードを開始します。

sudo gainroot



次に入力します：

apt-get update

apt-get gdbのインストール





これはすべてsshを介して行うことができます。すべてキーボードで手で書く必要はありません:-)。



2番目の方法は、gdbパッケージを手動でダウンロードし、dpkgを使用してインストールすることです（ちょうどそれを行いました）。

このリンクからこのデバイスにパッケージをダウンロードします 。 そして、デバイスに保存します。 したがって、コンソールで（再びssh経由で）そこに行き、次のように入力します

dpkg -i gdb_6.8.50.20090417-0maemo2 + 0m5_armel.deb



scratchboxでは、GDBはすでに立っています。



ESBox設定で、デバイスを作成し、IP N900を「ホスト名」として設定します。 リモートホスト（この場合はN900）で起動構成を作成し、SSHアプリケーションの配信方法を設定します。

SBRSHとSSHの2つのオプションがあります。

SBRSHは、デバイスがファイルシステムにアプリケーションを含むフォルダーをマウントし、その場所から開始する場合です。 さらに、すべてが高速になります。 マイナス：デバイスはコンピューターを認識し（仮想システムの場合、NATまたは転送ポートではなく、ブリッジモードである必要があります）、SSHサーバーはSDKホスト上にある必要があります。

SSH-プロジェクトはSCPによってN900にコピーされ、そこから起動されます。 さらに、仮想マシンをブリッジモードにすることができます。 マイナス：少し遅い。

2番目のオプションを選択しましたが、自分で選択することができます。設定はそれほど変わりません（ただし、ビデオでは最初にSBRSHを作成してからSSHに変更しました）。

リモート接続である「リモート接続」として、構成済みのデバイスを選択し、デバッグを開始します。



起動時に、ユーザーuserのパスワードを入力するよう求められます。 （パスワードは「maemo」です）。 [パスワードを保存]チェックボックスをクリックすることをお勧めします。そうしないと、パスワードの入力を何度も要求され、迷惑になります:-)



そして、voaaaaallaya、それはブレークポイントで開始および停止し、エミュレータ上と同じアプリケーションがデバイス画面に表示されました（私の言葉を聞いてください、むしろ自分でチェックしてください:-)）。



4. USB接続を介してデバイス上でアプリケーションを構成および起動します。



ネットワークを介したデバッグの欠如：遅い。 しかし、USB接続が助けになります。 実際、これはネットワーク経由のデバッグでもあります。これは、アプリケーションのデバッグに使用するUSB​​ネットワークインターフェイスが作成されるためです。 しかし、実際には「ポイントツーポイント」であるため、これが原因で、より高速です。



準備：pc-connectivityパッケージをデバイスにインストールする必要があります。仮想マシンには既にインストールされています。 Extrasリポジトリにあります。 Maemoの以前のバージョンから継承された非常に強力なツール。 つまり、接続プロファイルを作成できます。プロファイルごとに、USB、Wifi、およびBluetoothを介した接続を個別に設定できます。 はい、Bluetoothを使用してデバッグできますが、試したことはありません。 Wifiデバッグは、DHCPサーバーでアクセスポイントを作成します。 完全なドキュメントはこちらにあります 。 しかし、膨大な量のドキュメントに驚かないでください。 私たちの場合、それを読む必要はありません:-)。





ステータスバー（時計のある領域をクリックすると表示されるメニュー:-)から開始する必要があります。PC接続と[USB]チェックボックスを設定し、[適用]ボタンをクリックしてUSB経由でデバイスをコンピューターに接続します（スクリーンショットを作成できませんでした） 、別の電話で写真を撮りました）。



その後、新しい「usb」ネットワークインターフェイスが作成されます。 デフォルトでは、すべてがすでに構成されており、ESBoxのデバイスとして、ネットワークアドレス192.168.2.15のプリインストールされたUSBデバイスcを使用できます。仮想テーブルはアドレス192.168.2.14を受け取ります。 ただし、このアドレススペースがすでに占有されている場合は、別のアドレススペースに変更する必要があります。 PC接続設定は、標準のN900設定メニューにあります。



その後、接続を確認します-SSH経由でデバイスにアクセスします。

あとは、この接続のデバッグ構成を作成し、デバイスで直接実行するだけです。



おわりに





ご覧のとおり、すべてが非常に簡単です。 デバイスの有無にかかわらずMaemo5の作成を開始するのは簡単です。 しかし、個人的には、Qt-creatorを使用できないという事実は好きではありません。 もちろん、仮想マシンとホストの間の共有フォルダーでもコードを編集し、エミュレーターまたはデバイスで手ですべて実行し、Gdbで手動でデバッグするか、ESBoxを使用できます。 しかし、私はすでにQt-Creatorに非常に慣れています。

しかし、解決策があります-MADDE。 これはまったく異なるSDKであり、公式サポートはまだありません（Developer Previewにあります）。Qt-CreatorにはMADDEサポートがありません（公式リリース）が、これらはすべてQt 4.7のリリースで約束されています。 しかし、それでも、これらはすべて（gitリポジトリのメインブランチを使用するか、最新の「スナップショット」を使用して）、仮想マシンを使用せずにどのOSでも使用できます。 masterブランチのGitoriousのQt-Creatorは、すでにMADDEで非常にうまく機能しています。 MADDEには、エミュレータとパッケージの更新という2つの主なものが欠けています。 デバッグするデバイスがあり、必要なパッケージを手でインストールする準備ができている場合は、MADDEが適しています。

開発者向けの次の記事では、MADDEのセットアップと操作のプロセスについて説明します。
