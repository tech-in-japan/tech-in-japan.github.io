ユーザーロケーションメソッド

まえがき



ユーザー認証/登録システムを作成したことがある人なら誰でも、「ユーザーに関する詳細情報を調べるにはどうすればよいですか？」 これは何のためですか？ ほとんどの場合、 この特定のユーザーを識別するために。 時々-さまざまな社会的パラメーター、またはユーザーまたは居住地域の場所に応じて、追加の機能と情報を提供する。 たとえば、ある種のスコアリングを行う場合があります。 この記事では、ユーザーの地理的位置の特定に焦点を当てます。



効果的な決定方法



インターネットユーザーの地理的位置を取得するための多くの方法を考え出すことができます。 そして、これらすべての方法には長所と短所の独自のセットがあり、アプリケーションに応じて多かれ少なかれ効果的です。 次に、私が参加しているプロジェクトで現在使用されているメソッドのみを説明します。 私が直接使用するもの。 プロジェクトが存在する間、十分な統計がすでに収集されており、そこからいくつかの結論を引き出すことができます。





1. ソーシャルからのデータ。 ネットワーク


今日では、さまざまなソーシャルネットワークやブログの認証（または追加情報）アカウントに使用することが非常に一般的になっているため、それらのデータを使用できます。 この方法でユーザーを認証することにより、ユーザーに関する多くの情報を取得できます。 真実は、多くの人がソーシャルを指しているため、その信頼性について話すことはできません。 ネットワークは「本物」ではなく、「望ましい」、または一般的に最初に思いついたものです。 そのようなものを取り除くことは、通常、開発者の主なタスクです。 これを行うには、ユーザーのすべての友人に関する情報を取得し、一般データを確認する必要があります。 たとえば、ユーザーの同僚/クラスメート/クラスメート/友人との最も一般的な居住地を見つけることができます（たとえば、青のソーシャルネットワークではこれは非常に便利です）。これらのデータに基づいて、実際の地域、地域、都市、さらにはユーザーが住んでいる/働いている/勉強している都市のエリア。

また、いくつかの社会で。 ネットワークの場合、ユーザーがオンラインの場合、ユーザーの直接座標を取得できます。 このデータの正確さは、場合によっては、多くのことが望まれますが、少なくともユーザーがいる都市のエリアは、かなり確実に決定できます。



長所：

  • 友人データに基づくスコアリングモデルを使用する場合の比較的高い精度
  • ほとんどのユーザーはソーシャルアカウントを持っています。 ネットワーク
  • 友人のデータを使用して、データの正確性を確認できます




短所：

  • 実装の複雑さ、なぜなら いくつかのソーシャルサービスのAPIを学ぶ必要があります。 ネットワーク、受信データを分析するためのモデルのコンパイルと実装
  • ソーシャルでの有効なアカウントの必要性。 ユーザーのネットワーク（普及しているにもかかわらず、ユーザーにそのようなデータを要求することは不可能だと思います）
  • 友人からのデータを使用した分析を検討する場合の低速




残念ながら、「会社の秘密」の実装を提供することはできません。



2. GeoIPデータ


おそらく誰にとっても最も簡単で手頃な方法ですが、今日のロシア連邦にとっては、それはしばしば不正確です。

なんで？
事実は、現時点では、既存の地域レベルのプロバイダーのほとんどが連邦レベルの事業者に買収され吸収されているということです。 そして、なぜこれが悪いのですか？ しかし、何で。 状況を想像してみてください-「N」の都市には5つの小さなプロバイダーがありました。 それぞれが都市の彼自身のエリアで働き、それに応じてIPv4アドレスの彼自身のプールを持っていました。 また、動的に発行された「ホワイト」IPでさえ、都市の特定のエリアに大まかに結び付けられます。 連邦レベルのプロバイダーが登場し、5つの小規模プロバイダーすべてをアドレスプールとともに購入します。 その後、彼は、この連邦プロバイダーのすべてのネットワークの一般的な形式にネットワークをもたらします。 最後に何がありますか？ この連邦プロバイダーには、特定の地域で、必要に応じて使用される膨大な数のクライアントと膨大な数のIPプールがあります。 つまり、以前は小規模なローカルプロバイダーのプールに属していたアドレスを、まったく別の都市のクライアントに発行できるようになりました。これは、このアドレスプールがこのプロバイダーのすべてのクライアントに使用されるためです。 しかし、もちろん、誰もこのIPが発行された領域を誰にも通知しません。 さらに、明日は他の誰かに発行することができます。



また、ユーザーがプロキシやVPNを使用して別のIPに代わってインターネットにアクセスすることを防ぐことはできません。 この場合、GeoIPはこのプロキシまたはVPNサーバーに関する情報を受信するため、まったく役に立ちません。 プロバイダーが顧客にNAT経由でインターネットへのアクセスを提供する場合も同じことが起こります（そして、無料のIPv4アドレスの数の問題を考えると、これはますます一般的です）が、この場合、通常、少なくともあなたは地区、地域または都市を得ることができます。





この方法は非常に便利ですが、GeoIPデータのみに依存することは常に可能とは限りません-ほぼ瞬時に情報を取得します。 このため、通常、事前にダウンロードされたローカルデータベースが使用されます。



長所：

  • 使いやすく、さまざまな言語で多くの実装があります。
  • 高精度（いくつかの例外を除き、上記を参照）
  • 作業の速度（ほぼ即座に結果を受け取る-結局のところ、これはデータベースへの1つのクエリにすぎません）




短所：

  • IPベースを最新に保つ必要性
  • 受信したデータの正確性を検証できない（複数のデータベースへのクエリによってのみ）
  • 現時点でロシア連邦にとって十分な割合の誤ったデータ（上記を参照）




「これを行う方法」を説明することはあまりありません。なぜなら、それはオンラインであり、 ハブでは、詳細な説明がいっぱいです。 GeoIPデータを取得するための多くの無料のライブラリとツールがあります。 たとえば、PHPの場合、geoip拡張機能を使用できます。



3. JavaScript Geolocation APIを使用する


非常に便利で効果的な方法ですが、モバイルデバイス専用です。 固定コンピューターの場合-GeoIPのみが有用です。 実際、モバイルデバイス（最新のスマートフォン、タブレットなど）の場合、GPS、Wi-Fi、携帯電話の塔からのデータを含む、このデバイスとユーザー承認済みの場所を特定するために利用可能なすべての手段が使用されます。 しかし、ほとんどの場合、モバイルネットワーク（GSM / 3Gモデムの場合はそこからのデータを使用しない）もGPSも持たない家庭用PCの場合、GeoIPデータのみを見つけることができます。 。 そして、それらの精度については、すでに上に書いた。 しかし、私はこの方法を無視しません。結局のところ、インターネットにアクセスするためにタブレットや電話を使用する人が増えています。



その結果、この方法にはかなり狭い範囲のアプリケーション（モバイルデバイス）があります。 または、GeoIPに十分な近似データがある場合。



長所：

  • 正確だ、なぜなら セルタワー、Wi-Fi、GPSでのポジショニングとして使用できます
  • 速いので クライアントソフトウェアは 、位置を決定するために使用されます


短所：

  • 自宅のPCでは、すべてのブラウザーでサポートされているわけではありません
  • ユーザー権限が必要です
  • 実際、モバイルデバイスにのみ適用可能
  • 比較的簡単にデータを偽造する

4. モバイルオペレーターからの「ロケーター」タイプのサービスによる定義


読者の中には、これらのサービスについて聞いたことがある人もいれば、誰かがそれらを使用し、誰かがそれらを企業環境で使用しなければならないと思います。 私は卵会社のロケーターや黄色の縞模様の座標などのサービスについて話しています。 はい、これらのサービスはもともとエンドユーザー向けでしたが、...なぜ私たちがそれらのサービスを使用できないのですか？ この方法を使用する場合、肯定的な側面はほとんどありませんが、データの高精度とほぼ100％の信頼性は何ですか？ しかし、不快な瞬間があります。 まず、これらのサービスは有料です。 第二に-登録中に携帯電話番号を使用する必要性と、短い番号に無料のSMSを送信する要件...この動作は、多くの人を怖がらせることができます。 また、SMS経由で情報を受信するのにかなりの時間がかかります（Webアプリケーションの一部として）。 しかし、場合によっては、この種の、さらには信頼できる情報が必要な場合があります。 さらに、このメソッドは、任意のアクションのSMSからのコードによる確認の代替として使用できます。 そして、このようにして得られた情報を偽造することはほとんど不可能です。



最初のケースと同じ理由で、実際の実装の例を挙げませんが、これがどのように行われるかを少し下に簡単に説明します。



長所：

  • 高いデータ信頼性、ほぼ100％
  • 使用するデバイスやインターネットへのアクセス方法に関係なく、高精度
  • 携帯電話番号を自動的に確認する


短所：

  • 実装とサポートの難しさ
  • 低速のため SMSとユーザーからの応答を送受信するのに時間がかかる
  • 無料ではありません（このサービスのオペレーターの関税は非常に「大食い」です）
  • ユーザーの同意が必要

2） モデムを接続し、シリアルポートデバイスが/ devにあるかどうかを確認します

4） trsms.shファイルを作成する（イベントハンドラー） 




 #!/bin/bash status="$1" file="$2" case "$1" in RECEIVED) header=`head -12 $file | grep -e "^From: " -e "^Sent: " -e "^Received: "` from=`head -12 $file | grep -e "^From: " | awk '{print $2}'` if grep "Alphabet: UCS2" $file > /dev/null > /dev/null; then message=`tail -n +14 $file | iconv -f UCS-2BE -t UTF-8` else message=`tail -n +14 $file` fi #echo -e "$message" | mail -s "Incoming SMS from +$from" admin@yourhost.ru echo -e "$header\n$message\n" >> /var/log/smsd/sms.log if echo $message | grep "    " > /dev/null > /dev/null; then abon=`echo $message | awk 'BEGIN{ FS = " " } $2 { print substr($2, 2, 11) }'` echo -e "\n> \t$abon" >> /var/log/smsd/location.log fi if echo $message | grep "    " > /dev/null > /dev/null; then abon=`echo $message | awk 'BEGIN{ FS = " " } $2 { print substr($2, 2, 11) }'` adres=`echo $message | awk 'BEGIN{ FS = " " } $2 { print substr($2,0,index($2, "  ")) }'` region=`echo $adres | awk 'BEGIN{ FS = ", " } $1 {print $1}'` echo -e "\n> \t$abon\t$adres\t: $region" >> /var/log/smsd/location.log fi ;; esac




これは最小限の機能を備えた例です。 「卵」オペレーターの要求と受信した応答を記録します。 良い方法では、from変数に基づいて、メッセージを受信した番号で条件を追加する必要もあります。 その中で、オペレーターを決定することも可能です。 通常、異なる演算子の数は異なります。

smsdが動作するユーザーに実行する権利を与えることを忘れないでください。
