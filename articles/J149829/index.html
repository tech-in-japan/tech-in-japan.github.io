Sublime Text 2のBuild Systemビルドオプションパネル（または自転車の発明方法）

アイデアと実装：SublimeText2のショートカットを押すと、ビルドシステムによって選択されたファイルによって自動的に生成されたビルドオプションのリストを含むパネルが表示されます。目的のオプションを選択し、Sublime Text 2インターフェイスでプロジェクトアセンブリプロセスが直接表示される方法を確認します。



Sublime Text 2のProjectBuildプラグインの共同開発での貴重な経験をここで共有することをすぐに言わなければなりません。結果として、経験のみが得られました。結局のところ、このプラグインを実装する必要はなく、プロセス全体が自転車の発明であることが判明したからです。 残念ながら、この体験はまだ貴重です。 物語は、「バイク」が最後にのみ言われるようなものになります。



エントリー



以前に書かれた記事「Antを使用したSublime Text 2でのプロジェクトのアセンブリの自動化」の結果、割り当てられたキーボードショートカットを使用して、選択したビルドシステムのさまざまなビルドバリアント（例としてAntを使用）をSublime Text 2で呼び出すことができました。 （私は今、その投稿のトピックで「設定」という言葉を使用します）



私はこのアプローチについて好きではなかったもの：

  1. ビルドシステムが異なれば、バリアントの名前が完全に異なる可能性があり、キーボードショートカットは特定のバリアント名に既に明確に関連付けられていましたが、これは別のビルドシステムにはなかった可能性があります。 私の仮定では、ほとんどの開発者は複数のビルドシステムを切り替える必要はほとんどなく、ほとんどの開発者は1つのビルドシステムで作業します。 また、開発者が必要に応じて、キーボードショートカットを再度変更します。 しかし、あなたは認めなければなりません、これは少し面倒です。 そして、これは覚えておく必要があります。 そして続けて、また
  2. 割り当てたすべてのキーボードショートカットを覚えておく必要があります。 基本的な「F7」、「Ctrl + B」、「Ctrl + Shift + B」があることは明らかです。ただし、オプションごとにもう1つの組み合わせがあります。 これは、すでに存在するキーの組み合わせを再定義するか、指と脳の両方に便利な未使用の組み合わせを見つける必要があることを意味します。 便利な空いているものを見つけようとしましたが、再定義に落ち着きました。 また、これは、再定義時にこれらの組み合わせの元の機能を潜在的に奪う可能性があるため、憂鬱です。 また、将来、新しいプラグイン用の新しいキーボードショートカットが必要になるのを制限するために、さまざまなビルドオプションの呼び出しに便利なショートカットが既に使用されています。




そして、コメントに登録されていないsnegovikufaの Sublime Text 2のProjectBuildプラグインを試しました。 私が本当に気に入ったのは、コマンドのリストを含むドロップダウンパネルです。 リストから項目を選択することにより、適切なコマンドを実行できました。 このプラグインのすべての設定は、Sublime Text 2自体のメニューからアクセスできる別のファイルに保存されていました。このファイルには、名前と対応するコマンドのリストを表示できます。 簡単に言えば、このプラグインはSublime Text 2インターフェースから直接「サードパーティプログラム」を実行するように設定でき、特に必要なターゲットの名前でAntを実行するために使用できます。 AntProjectBuilder.sublime-buildでProjectBuildをねじ込みました。 これで2番目の問題が解決しました。 キーボードショートカットを1つ（「Ctrl + Shift + B」でも）割り当てると、Sublime Text 2インターフェイスのProjectBuildプラグインパネルが形成され、ProjectBuildの構成に従って目的のコマンドを選択して実行できます。



しかし、プラグインは、「サードパーティプログラム」を呼び出している間、Sublime Text 2自体のインターフェイスをブロックしませんでしたが、呼び出しの成功または失敗の痕跡を残さないように機能しました。 たとえば、私のコマンドラインが呼び出され、Antがその中で機能し、消えました。 Sublime Text 2自体のコンソールには痕跡は残っていません。 そして、これはSublime Text 2インターフェースをロックすることでのみ達成されました。つまり、コマンドと呼ばれ、Sublime Text 2インターフェースは一時停止し、このプロセスを実行した後、Sublime Text 2コンソールにすべての出力を吐き出しました。プロセス全体を「途中で」観察することはできませんでした。



そして、この使用法では、ProjectBuildがSublime Text 2ビルドメカニズムの機能を実装する必要がある場合、最初の問題は解決されませんでした-オプションを明示的に指定し、不要な構成もありました。 パネルのオプションのリストを作成するには、ProjectBuildを構成する必要がありましたが、このオプションのリストは、選択したビルドシステムの「* .sublime-build」ファイルに既に直接含まれています。 snegovikufaに連絡して、彼はすぐにGitHubの操作方法を知ってくれ、プラグインの変更を始めました。

公式および非公式のドキュメントがあります。 最初は、Sublime Text 2のAPI機能はそれほど優れていないように見えましたが、十分でした。 欠けていたものはPythonで実装されました。 前述のように、私はPythonを知りませんが、プラグインの意図された機能を開発するために多くのx知識は必要ありませんでした。 また、 snegovikufaは、サードパーティの不可解な瞬間をすばやく修正しました。 興味のために見つけたかったが、できなかった唯一のものは、Sublime Text 2がデフォルトで持っていた可能性のあるすべてのキーと設定値のリストでした。



「* .sublime-workspace」ファイル形式によると、ドキュメントにはこれがJSON形式であると記載されていますが、このファイルをjson解析するときにエラーが発生する可能性があることがわかりました。 実際、データ内のjsonキーは空かもしれませんが、これはProjectBuildプラグインによって次のように無視されました。 



... json.load (f, strict = False)


Sublime Text 2でも、「$ project_path」を持つ変数が「* .sublime-build」ファイルに含まれている可能性があり、Sublime Text 2の開発者がキーボードショートカットのパスで（および他の）使用を提供しなかった理由がわかりません。たとえば、サードパーティの崇高なプログラムが呼び出されます。 おそらくこれはセキュリティ上の理由によるものであるため、キーの組み合わせをうまく機能させるための一部のプラグインは、プロジェクトから「それ自体」のために何かを休ませるわけではありません。

結論



現在、ProjectBuildには設定は必要ありません。 必要なビルドシステム（標準システムまたは「AntProjectBuilder.sublime-build」のような独自のシステム）を選択するだけです。 プロジェクトをSublime Text 2に保存して、ルートフォルダーの1つにプロジェクトマニフェストファイル「* .sublime-workspace」が含まれるようにします。 ProjectBuildプラグインは、現在のビルドシステムに関する情報がこのファイルから取得されるため、このファイルの存在とその一意性を監視します。 Sublime Text 2のビルドシステムを変更するときは、プロジェクトを保存して、上記のファイルが更新されるようにしてください。 ProjectBuildに割り当てられたキーボードショートカットを押すと、ビルドオプションのリストを含むパネルが表示されます。 このリストは、「* .sublime-build」ファイルに存在するSublime Text 2で選択されたビルドシステムのオプションの説明に従って自動的に生成されます。







記事の冒頭で示された問題は正常に解決されました。1つのショートカットが使用され、Sublime Text 2自体のビルドメカニズムおよびそのインターフェイスで動作します。



プラグインには改善の余地があります。たとえば、ビルドシステムファイルで指定できるプラットフォーム固有のオプションは無視されるようになりましたが、個人的には今のところ十分であり、今実装しているものが正確に必要です。



実際、誰の貢献が大きいかを言うのは難しく、結果としてProjectBuildプラグインが本来あるべきものになった場合、それは重要です。 そして、隠すのは罪です。SublimeText 2のインターフェイスと機能にはそのようなものはなかったが、今はそれがあり、それは私によって部分的に書かれたものであるということは何とも言えない誇りです。 （ここで私は「自転車」について知りませんでした。



以前のバージョンのProjectBuildは、コマンドを1つのショートカットにグループ化するためのOneHotkeyプラグインに変換される可能性が高く、必ずしもコマンドをビルドするのではなく、Sublime Text 2プログラムなどのサードパーティコマンドを実行するだけです。 ProjectBuildが使用していた構成ファイルは、ビルドメカニズムの冗長な情報が含まれていたため、ここでは既に有用でした。
