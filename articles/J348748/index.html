教育と研究



今学期、私はPython入門コースのアシスタントティーチャーとして働いています。 可能性がほとんど無限に思えたプログラミングの初期の頃を思い出し、私のプログラムに新しい機能を追加することは楽しくて楽しいプロセスでした。 私はすべての行を誇りに思って、なんと素晴らしいことを友達に自慢し、全能だと感じました。 世界は文字通り私の指の下で変わりました。 興味のあるアイデアや、探求したい人生のあらゆる側面を身につけることができました。 私は自分自身でプログラミングに対する切望のない渇望を開発しましたが、それは今日まで残っています。









これは、プログラミングにもっとスパゲッティがあったら、私が若かった頃のように見えたでしょう。



皮肉なことに、今日書いたプログラムを振り返ると、とてつもなく恥ずかしい思いをします。 私の古いコードは、幼稚園のフィンガーペインティングに相当する芸術的なものでした。 はい、絵は馬に似ていますが、それは子供が自分のしていることを理解していないからです。 私が書いたプログラムには多くのバグがあり、読みにくく、順序が正しくなく、混乱するスパゲッティロジックに満ちていました。



しかし、それがどれだけ面白かったか、そしてどれだけ学んだかは否定できません。 これについては、もう少し詳しく説明しますが、まず、自分の経験と、今学期の学生の行動とを比較したいと思います。 彼らの研究室作業は、データの無意味なシャッフルと、多項式を計算するホーナーのスキームのようなドライで退屈な機能の実装にすぎません。 そして（驚くべきことに）彼らの独立したプロジェクトはさらに面白くありません。 最大の違いは、生徒がプログラムを作成して問題を解決する必要がないことです。 したがって、彼らの経験は、命令を繰り返すことや、自分のプログラムのロジックに従うためにコンピューターのふりをするという事実に限定されます。



もちろん、優れたプログラマーにとっては、指示に従うことと頭の中でコンピューターをシミュレートすることは重要ではないとは言いたくありません。 私は生徒たちが自分の仕事を楽しんでいないことを伝えようとします。 彼らの解決策はタスク自体と同じくらいドライであり、彼らがタスクを終了したとき、彼らはほとんど満足していることに気づきました（彼らのソリューションが完全に間違っていても）。



コースには独自の問題もあります。 たとえば、教授は学生にPythonではなくCパラダイムを教えます（Pythonで正しく物事を行うことさえ学んだとは思わない）、彼はスタックフレーム、レジスタ、およびアーキテクチャのあらゆる無関係な側面について話すときに混乱します。 覚えておいてください、これらの学生は以前にプログラムを書いたことがなく、何人かはコンピュータにほとんど精通せずにコースを開始しました。 スタックフレームが何であるかは、3年間プログラミングを学ぶまで知りませんでした（うち2つは最初の実験の年でした）。



これらすべてのことから、もしあれば、自分のコースで生徒に教え始めることについてよく考えるようになりました。 この投稿では、コンピューターサイエンスに関する私自身の研究がどのように始まったかについてお話します。 最も重要な側面を強調します。プログラミングを好きになった理由と、自然な文脈を例として使用して深い概念を教えてくれた理由です。

ズモブ



コンピューターサイエンスコースの2年目は完全にゲームに専念しました。 さらに、サイドスクロールプラットフォーマーなど、自分たちでプレイするのが好きなリアルタイムゲームについて考え始めました（そう、スーパーマリオブラザーズ、ドンキーコングカントリー）。 私は自分で似たようなことをしようとしましたが、衝突をうまく機能させる方法にすぐに迷いました。 レベルの作成、キャラクターのアニメーション化、画面のスクロールは難しい作業でしたが、私にとっては実現可能です。









Starfoxシリーズに触発された私の最初のサイドスクローラーの1つ。



しかし、キャラクターにブロックにジャンプするように教えることにうんざりしていたとき、私はより良いプロジェクトを見つけました：Zmob（Zombie Mobの略）。 彼は私たちのコラボレーションに触発されました。 友人が特別な方法で2つの円を描く方法を見つけるのを手伝いました：1つの円は大きく動きがなく、2番目の円は小さく、常に最初の円に触れ、2つの中心の間の線はマウスカーソルの位置を通りました。 言い換えれば、小さい円は、円のペアの「方向」を表し、常にマウスの方を向いていました。 これはかなり単純な三角法でしたが、これを行う方法を理解する前に、トップビューのゾンビシューティングゲームで作業することは面白いと判断しました。 それで私はこのプロジェクトを始めました。 ゲームの初期バージョンのスクリーンセーバーは次のとおりです（タイプミスとエラーは完全に保存されています）。









Zmob 1.0スタート画面



プレーヤーは黒い円を制御し、灰色の円は武器を示します。 ゾンビ（青い円）は定期的にランダムな位置に作成され、キャラクターの方向に異なる速度で移動します。 プレーヤーはしばらくの間、フィールド上を歩きながらShiftキーを押しながら走ることができます。 そしてもちろん、プレイヤーは死ぬまでそれらを撃ち、キャラクターが死ぬとゲームは終了します。 作成されるゾンビの数は時間とともに増加し、弾薬は限られています（ただし、一定の数のキル後にさらに弾薬を獲得できます）。そのため、プレイヤーの死は避けられません。 ゲームの目標は、できるだけ多くのポイントを獲得することです。



ゲームプレイはシューターというよりはむしろ羊飼いの仕事に似ていて、難しいことではありましたが、私以外の誰かが10分以上ゲームをプレイすることはないと思います。



重要なことは、私にはたくさんのアイデアがあり、これらのアイデアを実装する方法を理解する必要があるということでした。 私はゾンビが互いに通り抜けないようにしたかった。 私はゾンビを毒する武器を作りたかったので、毒のあるゾンビが健康なゾンビに触れると、ゾンビも毒するようになりました。 私はゲーム内で多くのことをしたかったのですが、当然、私が使用したJava機能の形で決定が下されました。









毒の大砲。 白いゾンビは毒され、青いゾンビは健康です。



たとえば、最初は、ゾンビは単に円で指定されていました。 あるゾンビと他のゾンビを区別する情報はなかったので、単純にx、y座標のリストとして保存できました。 一連の生命と異なる速度を追加し、それらを毒する方法を学びたいとき、私はゾンビクラスを作成して、各ゾンビにその内部状態（毒があるかどうか、速いか遅いかなど）を与えなければなりませんでした。 次に、プレーヤークラス、アイテムクラス、および弾丸クラスを作成しました。



そして、弾丸は私にとって最も興味深い部分であることが判明しました。 「update（）」関数を呼び出したときにのみ、画面上のすべての箇条書きを更新したかったのです。 これは、弾丸を、特化した各弾丸クラスが継承するインターフェースに変換することに似ていることが判明しました。 私はすでにオブジェクト指向プログラミングの必要性と優雅さを見てきました。 これらすべてのことは、これらの愚かなShapeインターフェイスを作成したときに完全に理解できず、入門チュートリアルの背後で繰り返しました。 私は解決する必要のある問題を解決していましたが、継承の理解は私の頭の中に永遠に残っていました。



弾丸の論理はそれ自体が喜びでした。 私が作成した最初の3つの銃は退屈でした：銃、機関銃、散弾銃。 いつものように、彼らは小さな黒い円を散らした。 目立ってクールな銃を作りたかった。 私は最初のアイデアを波ビームと呼びました。









波ビーム：正弦波弾。



波ビームの概念は、弾丸が正弦波に沿ってショットの方向に移動することでした。 しかし、私にとってそれは大きな挑戦になりました。どのようにしてサイン波を任意の角度に向けることができますか？ 弾丸のx座標とy座標はありましたが、サイン、コサイン、タンジェントをランダムに使用しようとした洗練された式はすべて、惨めに失敗しました。 私が達成できた最善の方法は、い正弦波を横に伸ばしたことです。



1週間失敗した後、統計担当の教師（まだ連絡を取り合っている）に行き、必要な魔法の公式を知っているかどうか尋ねました。



しばらく考えた後、彼は教科書を取り出し、回転行列に関するページを見せてくれました。 17歳の私にとって、この式は古代のルーン文字と同じくらい神秘的でした。









その結果、次のコードを作成しました。 



x += frame*Math.cos(angle) + Math.sin(frame)*Math.sin(angle) y += frame*Math.sin(angle) + Math.sin(frame)*Math.cos(angle)


コードを実行したとき、それは非常に完璧に機能したので、喜んで叫びました。 1週間の苦痛と平凡な問題解決の試みの後、解決策はエレガントで美しく、素晴らしいものであることが判明しました。 その後、マテリアル分析を使用して、StarFoxのサイドスクローラーでのジャンプがより自然に見えるようにしました。 せん断や伸縮などの他のマトリックス操作を試しました。 その年の終わりまでに、私は大学で線形代数を学んだほとんどの学生よりも「基礎の置き換え」を理解し始めました（これらの言葉さえ知りませんでしたが）！ それはただ空間の異なる座標系でした。 回転座標、拡大および伸長座標、縮小および逆座標がありました。 面白いゲームプレイを探して、これらすべてを試しました。



そして、数学だけでなくカリキュラムを追い越しました。 年の終わりまでに、私はゲームを「終了」しました。 ゾンビになったときに連鎖反応を引き起こすチェーン武器を作成し、プレーヤーとゾンビに新しいグラフィックを追加して外観を改善しました。 画面とウィンドウのサイズを測定し、背景を正しく表示するスマートタイル配置システムを開発しました。 寸法を手動で測定するのにうんざりしていたので、この作業を行うプログラムを作成しました。 これは些細なことのように思えますが、実際には、コンピューターサイエンスの問題を解決するための核心です。









クラス全体がゲームのベータテストに従事していました。つまり、数日間かけてゲームをプレイし、バグを見つけました。 そして、クラスメートは多くのバグを見つけました。 明らかな（ゼロによる分割エラーは弾丸を狂わせた）目立たない（時間を正しく計算すると、ゾンビはプレイヤーに十分に近づくことができず、お互いにぶつかります）。



別のかなり重要な問題が明らかになりました-速度。 画像を追加した後、Javaライブラリを使用して各フレームの画像を正しい方向に回転させることにしました。 今日、Javaは遅いと言う人もいますが、この部分は非常に遅く、特に数百以上のゾンビに関してはそうです。 私の決定は、たまたま起こったのですが、 キャッシングと呼ばれるプログラミングパラダイムであることが判明しました。 ゲームに必要なすべてのターンを事前に計算し、保存しました。 実際、私が実装したものはlazy-loadingと呼ばれます。 これは、必要なときに計算されたターンのみを保存するために使用される、少し複雑な手法です。



大学の3年生で勉強した動的なWebプログラミングに着手したとき、この手法の名前を知っただけで、データベースのHibernateデータのオブジェクトリレーショナルマッピングについて説明しました。 線形代数の場合のように、私が自分で設定したタスクにより、重要な概念を教えたり教えたりするよりもずっと早く発見または再発明することができました。 私は、概念とそれらが解決できるタスクの種類を深く理解しました。 これは大学で行われた研究とはまったく異なっていました。学生は概念の名前とその意味を覚えていますが、なぜそれが重要で、どのように使用するかを理解できるのは最高の学生だけです。



正直な遡及分析



私は平均的な学生よりも自分の仕事に専念していたことを認めなければなりません。 クラスのごく一部は愚かなことだけに関心がありました。 頭の中に目標を持っていた生徒もいましたが、私と同じ粘り強さでそれを追求しませんでした。 私たち自身のインターネットサーフィンや、Mathers氏が持っていた平凡な本の質を超えて、多くの良い例にアクセスすることはできませんでした。 一部の人にとって選択されたJava言語は、おそらく学習曲線が非常に急でしたが、その結果、害よりも多くの利点をもたらしました。

最悪の部分は、生徒たちが「プログラミングがそのようになるとは思わなかった」と言うとき、 私の心が壊れる ことです。 準備ができていません。 私が彼らに答えることができる最高のこと：「このコースでプログラミングを判断しないでください。 実際、それは楽しいものです。」

プログラミングに学生を興味を持つためには、いくつかのことが必要です。



1.インスタント報酬



私の学生は、自分のコードを理解するのに時間がかかりすぎています。 すぐに効果を確認するには、何かを変更する必要があります。 彼らにはBret Victorが開発したトレーニングツールが必要です（私が話していることを理解するには、10時30分にビデオを巻き戻しますが、すべてを見る価値はあります）。 そして、彼らは視覚的なプログラムに取り組む必要があります。 描画、ゲーム、音楽のプログラム。 多項式導関数が正しいと見なすかどうかの退屈なテストではなく、その効果が非インテリジェントな方法で感じられるプログラム。



2.プロジェクトは有意義であるか、少なくとも興味深いものでなければなりません。



私が自分で勉強していたときと同じように、学生には勉強する機会が必要です。 彼らが自分のプロジェクトに取り組み、教師として、彼らが動けなくなったときに役立つのに十分な知識を持っているようにします（または、さらに良いことに、彼らとブレインストーミングします）。 各生徒に固有のプロジェクトを作成することが不可能な場合は、少なくとも意味のある何かに取り組むようにします。 残念ながら、私が教えているコースの最後の2つのプロジェクトでは、学生はファイルの入出力とマトリックスの追加に従事していました。 ビデオゲームや検索エンジン（複雑に思えるかもしれませんが、ここではudacityのトピックに関する入門コースです ）、または描画/アニメーション、チャットクライアント、数独の解決について学習させませんか？ はい、少なくともGraph APIを介してFacebookからデータを受信する方法を示します。 これらはすべて抽象的であるため、あらゆるレベルの学生がそれらを処理できると同時に、それぞれが特定の概念（チャットクライアントのネットワークを使用する基本、数独のマトリックス、入出力）を操作する能力を必要とします。検索エンジンの一部のファイルなど）。 生徒が取り組むことができる興味深いことはたくさんありますが、教師は、単に彼らに興味深いプロジェクトを与えたくないようです。したがって、彼らは何度も生徒にマトリックスの合計を計算させます。



3.他人のコードを読む能力。



これは学習の不可欠な部分です。 学生は自分のコードを書くことができるだけでなく、他の人のコードも読むことができるべきです。 例を見て、自分の仕事で使用する重要な部分を選択できるようにする必要があります。 彼らはクラスメートとの共同作業を学び、共通のプロジェクトに取り組み、バグについて話し合うときにブレーンストーミングをしなければなりません。 彼らは、コードを批判する方法、映画やカフェを批判する方法を学ぶ必要があります。 学生はソフトウェアについて独自の判断を下し、肥大化したコードや混chaとしたコードを公然と笑し、適切な実装に努める必要があります（まあ、それほど厳しくないかもしれませんが、それでも意識的に準備する必要があります）。



これら3つの原則は、コンピューターサイエンスとソフトウェア開発の中心にあり、残りのすべて（スタックフレーム、遅延読み込み、Linuxシェル）は、学生が興味を持ち、さらに努力するようになるまで待つことができます。 さらに、Linuxシェル、Webフレームワーク、ネットワークセキュリティ、グラフィックス処理の知識を必要とする分野を選択する機会が得られるまで待つことができます 。 Linuxターミナルに触れることなく、その存在を知らなくても、すべての基本的なことや他のことを学びました。 そして、私の現在の学生もこれができることは間違いありません。



そして、学生がコードに完全に没頭しているとき（私のように、スパゲッティコードを1、2年かけて書いている）、彼らはプログラムを整理するエレガントな方法の美しさ、そして有用なプログラムを書くために持つことができる表現力を見ることができます。 ある意味で、プログラミングはアーキテクチャに似ています。優れたプログラムには、形と機能の美しさがあります。 この時点で、新しいパラダイムを自分のパラダイムと比較できるため、システムおよびネットワークプログラミングについて考え始める必要があります。 彼らは批判し、議論し、革新し、あるいは少なくともそれらを評価し、彼らが取り組んでいる「ゾンビプロジェクト」にアイデアを適用することができます。



私は、コンピューターサイエンスの各カリキュラムでは、空白の紙として使用されるいくつかのコースが必要であり、トレーニングの過程で、そのうちの1つをできるだけ早く（最初のコースの一部としても）提供する必要があると考えています。 数学を教えるときと同じ理由で、学生はこの方法で教えられないと思う：適切な教育は大変な仕事だ！ これがどんなに悲しいことに聞こえても、教授（特に研究教育機関）には、慎重に考え抜かれた学生プロジェクトを開発する時間がありません。



そして、トレーニングシステムにいる間、私は違いを生むように働きます。 興味深くなるようにコースを設計し、成功しない将来の従業員を支援します。 高度に構造化されたコースであっても、学生に無料のプロジェクトに取り組みます。



したがって、高校の先生に「Programming as Finger Drawing」というコースを追加します。 （または「白紙の状態でプログラミングする」または「ゾンビを殺す方法」？）私は提案を受け入れています。
