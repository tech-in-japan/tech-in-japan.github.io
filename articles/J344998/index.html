アルファラボでの1日：Java開発





多くの場合、会議に参加する企業と技術面接を行います。 しかし、 Alfa-Bankの IT部門では、1人の開発者に質問を送信するだけでなく、オフィスで1日を過ごして、バックベンダー、フロントエンド、携帯電話を設置します。 結局のところ、全体像があります-使用される技術から会社の一般的なアプローチまで。



彼らは1つの「フルスタック」テキストを作成しようと考えましたが、非常に多くの資料があったため、それを部分に分割する必要がありました。 そして今、あなたが「朝」の最初の部分であり、彼らはJava開発者マキシム・ゴレリコフとキリル・ トルカチョフと話しました。 両方とも最近ジョーカー会議で話をしました。





マキシム・ゴレリコフ



-最初に、会社にどれくらいの時間を費やしているか、最初にどの職に就いたか、そして今何をしているのか教えてください。



-2015年半ばにアルファに来ました。 昔、私は多くのものを開発し、建築に携わっていましたが、やがて私はティーポットであることに気付きました。 Senseチームのメンバーが私に話しかけ、電話をかけてきましたが、断ることはできませんでした。テクノロジースタックは非常に興味深いものでした。



聞いたことがない方は、Senseは銀行内のそのような新興企業であり、モバイルアプリケーションの形の個人的な財務アシスタントです。 彼は毎年恒例のハッカソンで生まれました。 アイデアとプロトタイプはうまくいき、プロジェクトとそれを思いついたチームにお金が割り当てられました。 センスはまだ生きていますが、一般的に、その機能は現在、銀行のメインアプリケーションに徐々に移行されています。



一般的に、私はここで純粋な製品チームのJava開発者として始めました。 数年間、私は多くのことを学び、さまざまなプロジェクトに参加し、現在は主にインフラストラクチャ、開発者向けツール、安定性タスクに取り組んでいます。



-おもしろいですが、ハッカソンのアイデアを銀行の本当の商品に変えたのですか、それとも他の例はありますか？



-孤立したものではありません。 たとえば、昨年、開発者のみで構成されるチーム（製品の所有者、デザイナーなどを持たない）によるハッカソンに参加しました。 チームは非常に緊密に連携しました。実際にラップトップを24時間放置しなかったため、最後にはレモンのように絞られました。 そして、他のアルファ開発者を対象としたプロジェクトがありました...「プラットフォーム」という言葉を発音するのが怖いのです。 「プラットフォーム」を開始した理由：



会社は積極的に成長しており、私たちは常に希望する経験を持つ人々をすぐに雇うわけではありません。 時々、新参者はほとんど経験がなくても目が燃えます。 JUGミーティングの後、人々は来て見回ります。「ああ、シリルが通り過ぎました！」 彼らは学ぶためだけにここに来ます。 そして、そのような人が最初はfireを壊してすぐに参加しなかったように、ある時点で、入場のしきい値を下げるようなプラットフォームを作成することにしました。 テンプレートとライブラリを使用してサービスをすばやく作成できます。



ある時点で、私たちはそれを理解し、新しいチームがJIRAでプロジェクトを設定するのに多くの時間を費やし、ジェンキンを自分自身で育てることに気付きました（すべてのジェンキンに座るのは大きなリスクです。 。 開発者がボタンを押し、JIRAでプロジェクトを設定し、Jenkinsでジョブを作成するなど、コードを生成し、何らかのサービスベースでプルリクエストを作成するまで、これらすべてを統合しました。 このもののアセンブリが表示されたら、開発者はデプロイをクリックすることができ、テストのためにすべてが展開されます。 これにより、開発者はすぐにすべてがどのように起こるかを理解し始め、この製品に採用されているサービスの基本構造を見て、そのスタイルを理解します。



-そして、ハッカソンのプロジェクトは、銀行の本当のニーズに沿ったものにする義務がありますか、それともまったく違うことをすることができますか？



-楽しみのためだけに作られたクレイジーなアイデアがありました。後で真剣に宣伝することは意味がありませんが、その過程で誰もが多くの喜びを得ました。 たとえば、ハッカソンの1つで、シリルと大規模なチームがスマート冷蔵庫を作りました。 これは一般的に銀行の主要製品には適合しませんが、24時間の楽しみでした。 私たちは座って、はんだ付けし、ある種の食料品店を解析しました。 真夜中に、彼らは突然冷蔵庫がなかったことを思い出したので、チームの一人が夜にそれを持って来て、そのようなZILに値しました：











私は一般的に、アルファを実験の精神がある場所だと考えています。 チームに7人がいる場合、これらはすべてのレイヤー（JavaからiOSまで）の開発者であり、開発者の宗教に従ってインフラストラクチャを編成することもできます。これにより、スタートアップの雰囲気が維持されます。 このチーム自体がすべての問題を解決し、決定を下すことができます。



-通常、「銀行」という言葉は「実験の精神」という言葉ではなく、安全、注意、保守主義という言葉に関連付けられています。 あなたの条件下でどれくらい実験できますか？ Kotlinのような「興味深いが、まだ業界標準ではない」何かを使用できますか？ また、JavaまたはSpringの新しいバージョンがリリースされたとき、どれくらい早くそれらに切り替えますか？



-いずれにしても安定性が必要であることは明らかです。 しかし同時に、新しいものを直接本番環境にドラッグする必要はありません。 技術的なバックログにはかなり多くのタスクがあります。 いくつかのツールを作成します。完全に満足できる既製のものがない場合は、試してみてください。



さらに、銀行のSenseのようなプロジェクトには、パイロットステータスがある場合があります。 つまり、小さなクライアントベースがあり、そこにいくつかのことを試すことができます。 そして、現在Alphaにあるスタックの一部はSenseから、一部は他のプロジェクトから来ました。 実験の余地があるとだけ言っておきましょう。 もちろん、GitHubで3つ星を獲得しているリリース候補やテクノロジーを争うことはありません。 しかし、新しい技術を持ち込むために根本的な問題はありません。



決定は、製品に取り組んでいるチームによって行われます。 3つの開発チームが座り、コトリンを戦闘に引きずり込みたい製品がある場合、彼は他のチームからジャビストを説得する必要があります。 そして、会社全体でそれを普及させるために、そしてこれが初心者に推奨されたために、彼は他の製品で人々を説得する必要があります。



新しいバージョンへの移行速度について-ケースごとに、実際にどれだけ必要かによって異なります。 Java 9では、プロジェクトはまだ移動していないようですが、それほど前のリリースではありませんでした。 しかし、Spring 5では、積極的に実験を行っています。最近のジョーカーでは、それについて話していました。 時間が近づいており、興味深いチップがあります。たとえば、Spring Boot 2.0がリリースされるとすぐに、その上のマイクロサービスの一部を追い越すことが可能になります。 ほとんどの場合、「リリースがありました-来週または今週の終わりにいくつかのマイクロサービスが表示されます」というカテゴリから、いくつかのマイクロサービスのかなり迅速な移行が行われます。 マイルストーンバージョンを既に試したので、リリースでは、どのバグが既に閉じられ、どの重要なバグが残っているかがわかるので、自信がつきます。



-そして、法律の側からの厳格な要件（「そのような情報を保存する」）およびセキュリティ（「この情報が部外者に利用可能にならないようにする」）は、空想を飛ぶ代わりに、ポイントで1,000の退屈な要件に対処する必要があるという事実につながりませんか？



-それどころか、これはまさに実験のための素晴らしい空間です。 過去に、Joker Ilya Sergeevはログインフラストラクチャについてレポートを作成しました。このレポートでは、これまで試したことのないあらゆる種類のさまざまなテクノロジを試しました。 また、このインフラストラクチャは、特定の情報を特定の保存期間と特定のユーザーグループのアクセスで保存する必要があるときに生まれました。



その前に、ログをElasticに直接ロードしましたが、この場合、ダッシュボードとモニターを構築するのが便利ですが、そこから大量のデータを一度にアンロードしようとする誰かを許可した場合、それらはすべてそれを置く可能性があります。 そして、Kafkaとキューを介してロギングインフラストラクチャを構築しました。これにより、これを操作する必要があるユーザーがいつでもこのキューに入り、そこから最新のデータをアンロードできるようになりました。 そして、それらがどれだけの形でどのように保存されるのか、これが彼らの仕事だとしましょう。



私たちと一緒に実験したい人は、常に何を見つけるでしょう。



もちろん、セキュリティのため、アクセスの発行に関して制限があります。 「すべての実稼働クラスターがインターネット全体にアクセスできるわけではない」というカテゴリーから。 パイロットにはもう少し行動の自由があります。 多くの顧客がいる深刻な製品は、リスクが大きくなります-より困難です。 時々、警備員と交渉します。 時々、アクセスする必要がないようにインフラストラクチャとそれを操作するツールを構築します。手動で何も設定しないようにします。すべてが自動です。







-側から見ると、銀行はわずかな遅延に対して罰則を課せられた厳格な組織のように見えるかもしれませんが、実際には、労働時間は決まっていますか？



-いいえ。 昨日新しいハードウェアが到着し、新しいロギングクラスタに移行し始め、新しいスキームを導入し、全員が遅く解散したため、Cyrilはまだオフィスに現れませんでした。



しかし、この自由度により、個々のチームはしばしば独自の規律慣行を導入します。 チームの毎日のスクラム会議があり、DSMに遅れたために少額の罰金を課すことが決定されました。 チームのDSMは11:30であり、銀行はすでに具体的な金額を蓄積しています。 もう少し貯めて、このお金のためにバーに行きます。

実際には、決定は製品提供者だけではありません。 通常、チーム全体が製品の作成に関与しており、主な質問が時々聞こえます。







チーム全体が何が行われているのか、そしてその理由を理解していれば、何か質の高いものを簡単に作成できるようになります。 チーム全体が責任を負うため、全員に投票権があります。

ここで長く働くほど、一目ですべてをカバーし、プロセスとアーキテクチャを構築しようとすればするほど、それが最初から最後までどのように機能するかを理解する必要があることは明らかです。 しかし、最初の段階では、頭で渦に飛び込んで支払い、投稿を理解し始めるようなことはありません...チームのアナリストは通常​​、この質問を閉じます。

-はい、経験が積まれても...私はそれを成長と呼ぶつもりはありません。たとえば、開発者の「方向転換」です。 ほとんどの場合、開発者が来ると、彼は製品チームに突入し、仕様、原則、プロセスを徐々に理解します。 最初の数か月後、彼は同化され、「すべてが新しい」という状況がなくなったとき、彼は他の補助機器に従事し始めます。 そして、彼はすべての技術を理解し、新しい技術を取り入れることができ、他の開発者とこれを議論する方法を理解するまで成長すると、ほとんどの場合、彼は目立つようになり、フルタイムのチームで作業を停止します。 彼はむしろ技術を実験し、いくつかのプロトタイプを作成し、初心者がそれに慣れるのを助け、技術的なバックログを処理し、他の人を引き付けます。 彼は、どこが不安定なのか、何を引き上げる必要があるのか​​、どのテクノロジーまたはアプローチを実装するのかを見ています。
