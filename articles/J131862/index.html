かな応答に基づいて大量の受信メールを処理する

多数の受信メールとその応答を処理する問題についてお話します。 明らかに、多数の顧客（電気通信、銀行、ツアーオペレーターなど）を抱える企業は、年間何百万ものメールメッセージを受信できます。 それらのほとんどは非人格です。つまり、会社の一般的なメールボックス（たとえば、support @ myorg.com、info @ myorg.com、billing @ myorg.com）に向けられています。 「ポストセンター」にいるエージェントの数は、数十から数千です。 Ekschendzhのパブリックフォルダーを使用して非常に多くのメッセージを処理することは可能ですが、非常に非効率的です。

次に、アメリカ/イギリス、そして徐々に西ヨーロッパのこの市場のリーダーの1つであるKANAの KANA Response製品が、この問題の解決にどのように役立つかをお話します。



KANA Responseは、他のKANAソリューションよりも優れた製品です。 サーバー側の主なタスクは、pop3アカウントを介して受信メールを取得し、ルールのシステムを使用してメッセージを分類し、異なる優先度を持つキュー、SLA、およびこのキューからのメッセージに応答できるエージェントに配信することです。 また、分類の助けを借りて（ルールの助けと自己学習モジュールの助けを借りて）、メッセージは既に応答テンプレートが添付された状態でエージェントに配信されます。 クライアント部分は、Java WebStartアプリケーションのようにアプレットとしてダウンロードしたり、ローカルアプリケーションとしてワークステーションにインストールしたりできるJavaアプリケーションです。

エージェントの実際のアプリケーションは次のようになります。



左側にはエージェントの現在のメールボックスがあり、キューの優先度とメッセージの経過時間を考慮して、サブスクライブ先のキューからの手紙が送信されます。 その下には、選択された着信メッセージがあります。 このクライアントからのすべてのメッセージ、または現在のメッセージスレッドに関連するすべてのメッセージをすぐに表示できます。 詳細ビューに切り替えて、添付文書、サービス情報、メッセージの完全なソースコード、メッセージ履歴を表示できます。

右側にある応答フォームでは、メッセージカテゴリ（メッセージに応答テンプレートを追加できます）、グリーティングフレーズ、署名を選択できます。 そしてもちろん、答え編集者。 エディターは、HTML形式とプレーンテキスト形式の両方で動作できます。

現在の状態を監視するための統計ウィンドウもあります。



キューごとのメッセージの統計と、このキュー内のメッセージサービスのレベルへの対応（緑、黄色、赤の列）はどこで確認できますか。



管理部分は、システム部分とコンテンツ部分に分けられます。 システム部分では、管理者はチェック対象のPOP3アカウント、送信サーバー、およびシステム（クラスター）全体の機能に関連するその他のパラメーターを決定できます。 コンテンツ部分では、スーパーバイザーは、新しいカテゴリ、キュー、ルール、さまざまな応答の場合のメッセージテンプレート、システムユーザーを定義できます。 次のようになります。



この画面には、特定のメールボックス（support@myorg.local）への準拠について着信メッセージをチェックし、このアドレスを介して受信したメッセージにカテゴリ「デフォルトカテゴリ」を適用する編集ルールが表示されます。 ルールのシステムは非常に分岐しており、メッセージヘッダーのすべての標準フィールドとメッセージのテキストの両方でメッセージをフィルタリングできます（正規表現を使用できます）。 ルールで設定できるアクションには、メッセージの分類、特定のエージェントまたはキューへの受信トレイへの移動、応答の監視を伴う2行目のサポートへのメッセージの送信（KANA応答の外部のメールボックスへ）、パラメータに応じて入力される任意のURLの呼び出しなどの機能が含まれます現在のメッセージ（たとえば、課金システムから自分の住所でクライアントのステータスを受信する）、確認メッセージをクライアントに送信する、分類テンプレートを使用した自動返信、その他の多くの機能。



システムレベルでは、KANA ResponseはOracle / MSSQL / DB2データベース、AIX / RH Linux / Windows / Solaris OSで動作します。 KANA Responseサーバーは、単一サーバーまたはサーバークラスターで構成でき、すべてのKANA Responseコンポーネントを複数のノードで実行できるため、アプリケーションの生産性を向上させることができます。 1年に約1,000万のメッセージと約200の同時エージェントを処理するサーバーをインストールしました。私が知っている最大のインストールは、1日約50万から100万のメッセージと数千のエージェントが同時に動作することです。
