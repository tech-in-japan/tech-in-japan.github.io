QLiveBittorrent-ダウンロード前にファイルを表示するためのコンソールbittorrentクライアント

ちょうど1か月前、 LiveDC-p2pファイルへのクイックアクセスという記事が公開されました。 その意味は、 Erty_Hackwardは 、最終的にダウンロードされる前にファイルを表示できるDCクライアントによって作成されたということです。 これにより、たとえば、ダウンロード開始後数分で映画を視聴したり、巻き戻したり、いつでも視聴したりできます。 また、アーカイブ全体を転送せずに、大きなアーカイブから目的のファイルを抽出できます。



このプログラムのアイデアは本当に気に入りました。 ただし、C＃で記述されています。 そして、Linuxで使用したいと思います。 したがって、著者の許可を得て、Linux用の同様のプログラムを作成することに大きな喜びを感じました。 結果は、コンソールのBitTorrentクライアントQLiveBittorrentです。









仕組み

映画をダウンロードする例を使用して、QLiveBittorrentの作業について説明します。 たとえば、昨晩、私の友人が私たちの学校でプロムのビデオをアップロードしました。それをダウンロードして、この忘れられない光景をもう一度楽しみたいです。



私は映画をダウンロードし始めています。 マウントしたいフォルダを指定します。 そして、私はそれを開こうとします。 フィルムは、速度に応じて1〜2分で開きます。 私は見始めています-十分な速度があれば、通常遅れはありません。 次に、フィルムを中央に巻き戻します。 ブレーキを約30秒間見て、真ん中から見続けます。



次のことが起こります。 最初に、プレーヤーで映画を開くときに、プログラムは映画の最初の数ブロックと最後のブロックをダウンロードして、その長さに関する情報を取得します。 その後、彼女は最初から映画をダウンロードし続けます。 しかし、途中で映画を巻き戻した時点で、プログラムはプレーヤーが要求したブロックを正確にダウンロードし始めます。



ムービーのダウンロードに加えて、このプログラムを使用して、アーカイブ、isoイメージを操作し、膨大な数の写真を含むフォルダーを表示できます。 アーカイブのダウンロードを開始することは非常に便利です。ダウンロードを待つことなく、最初からアーカイブを開き、必要なファイルをすぐに指定してダウンロードします。

最初は、お気に入りのbittorrentクライアントに機能を組み込むことを考えました。 しかし、ソースを表示した後、この欲求は消えました。 何百万ものファイルが、どこでもさまざまなシステムで定義されており、すべてがシートの形であり、何も明確ではありません。 しかし、qbittorrentに感銘を受けて、私は同じツールキットを選択しました。

第二の風

GUIで得点を決めたとき、2番目の風が開きました。 誰がそれを必要としますか？ あまりにも多くの神経が彼のサポートに行きます！ そこで、コンソールのbittorrentクライアントが登場しました。 そして、面白いアイデアが思いついて、座っているトレントをスイングするトレントから分離しました。



その結果、プログラムはダウンロード部分に分割され、ダウンロード部分がダウンロード不足のファイルをマウントし、ダウンロード速度を制限し、通常のトレントクライアントと、ダウンロードされたものを配布し、アップロード速度を制限する座った部分（シードマネージャー）になりました。





興味深い開発ポイント

  1. トレントのダウンロードをテストするときに、ダウンロード速度がルーターの最大速度である802.11G〜2MB / sを超えることに驚いたことがあります。 それは40-50 MB / sに達しましたが、これは原則的に不可能でした！ 私はすべてをキャッシュにダンプしようとしました。 しかし、コンピューター上で2つのクライアントが実行されており、ハードディスクの読み取り速度で互いに情報を交換していることがわかりました。 1つは読み取り、もう1つは書き込みです。 その後、これをテストに使用しました。



  2. クライアントがコンソールであるという事実により、完了信号を処理する必要がありました。 SIGTERMシグナル処理（プロセスの完了を要求するシグナル）を追加しました。 デバッグ中にSIGTERMの「hands」を送信すると、正常にインターセプトおよび処理されました。 ただし、実際には、ターミナルでctrl-cを押すと、信号は処理されませんでした。 端末はSIGTERMを送信せず、SIGINT（端末からユーザーがプロセスを停止する信号）を送信することが判明しました。 びっくりしました。 ctrl-cを押すと、信号は「プロセスを停止する」のではなく、「ユーザーがプロセスを停止したい」ことがわかります。 おそらく、これらのプロセスの違いの深い意味を思い付くことができますが、それは非常に予想外で奇妙でした。



  3. クライアントはコンソールであるという事実にもかかわらず、私はある種のGUIを残したかったのです。 しかし、結局のところ、アプリケーションがコンソールの場合、ベアコンソール（TTY）で実行したいのですが、これはGUIでは不可能です...または可能ですか？ これを行うには、Linuxでのグラフィックスの動作を理解する必要があります。



    そして、それは次のような働きをします：Xサーバーがあり、クライアントがあります。 ある時点で、クライアントが接続します。 問題は、これがいつ起こるかです。 実験的に（gazeメソッドを使用）、 QApplicationの作成時に接続が発生することがわかりました 。 したがって、グラフィックが必要な場合はQApplicationを作成し、そうでない場合はQCoreApplicationを作成します。 したがって、このプログラムは、ベアコンソールとグラフィカルミニインターフェイスの両方で実行できます。



  4. Qtとlibtorrentを混在させることはまったくの地獄です。 結局、私の神経はそれに耐えられず、私はすべてをQtに変換しようと試み始めました。 残念ながら、libtorrentはstdを使用しているため、すべてのコードはQString :: fromStdString（string）およびQString.toStdString（）、さらにベクトルchar、QVector char、QByteArray変換などでハングアップします。



  5. 開発中に、私は不快な問題に遭遇しました。 ある時点で、遅いごちそうがプログラムに接続され、ゆっくりと必要な部分を提供し始めます。 さらに、平均ダウンロード速度は良好ですが、それらによる遅延は許容できません。 私はそれと戦おうとしました。 問題は、プログラムが最初の2分間でファーストフィーストを検索し、最大数を5フィーストに制限するという事実により、ある程度解決されました。 これにより、ほとんどの遅いごちそうを縫うことができます。



  6. 午前0時以降に時刻を減算しても問題がないことを確認するために、プログラムは関数midnight（）を実装します。 深夜0時から5秒以内に、プログラムはソーセージ-画面上の情報を更新すると、すべてのタイムカウンターをリセットし、ユーザーの要求を無視します。 このような珍しい松葉杖。 真夜中の3分前に書きました。



機能リスト

  1. トレントが公開されたばかりの場合、サイドの数とリーチャーの数の比率が小さすぎます。 これには、そのようなトレントを妥当な時間内にダウンロードすることがほぼ完全に不可能です。 そのため、QLiveBittorrentは通常のトレントクライアントに「変わる」機能を追加しました。 これを行うには、「a」キー（英語）を押します。



  2. マウントされたファイルを読み取るとき、必要な部分がダウンロードされるまでプログラムは混乱に陥ります。 しかし、彼らはこれが起こるとき、常にそれから出てきます。 これはプログラムの機能です。



  3. 信頼できる情報源は、600KB / sチャンネルで、外部オーディオトラックと字幕付きの重さ1.45Gの映画が遅れなく再生されると主張しています。



  4. キーボードでctrl-cを押す（またはSIGTERMを送信する）と、プログラムは終了し始めます-データをハードドライブにダンプし、トラッカーに情報を送信します。 データをリセットした後、ctrl-cを押すと、情報はトラッカーに送信されません。 これは、完了できないプログラムに非常に怒っているためです。
