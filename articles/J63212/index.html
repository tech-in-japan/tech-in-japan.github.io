Javaコードの自動品質管理

プログラマーによって書かれたコードは機能している必要があります-成功した作業の最初のルールであり、プログラマー自身と彼の上司全員が同意します。 ただし、コードが正常に機能するという事実に加えて、コメント（内部ドキュメント）、読みやすさ、変更の速度、明示的および暗黙的な標準との互換性の存在など、多くの場合、高い要件が課されます。 コード品質と呼ばれるものすべて。



ただし、検証テストを実行することで評価できるコードパフォーマンスとは異なり、コードの品質はTRUEまたはFALSEの単純な評価ではありません。 さらに、コード品質とは、他の人によるコードの認識の主観的評価のセットを指します。 ただし、品質を評価するタスクを何らかの形で形式化し、可能であれば、このタスクを自動的に完了する方法を提供してみましょう。



評価のポイントの少なくとも最初のリストは何でしょうか？

  • すべてのコードのコンパイル
  • すべてのコードのパフォーマンス
  • プロジェクトで使用されるコードの記述スタイルへの準拠
  • 各コードセクションのドキュメントの可用性
  • 個々の行、関数、ファイルのレベルでのコードの可読性
  • コードに迅速に変更を加え、既存の最小限の機能に影響を与える機能




編集



残念ながら、私たちの生活には、プロジェクトのコードが完全にコンパイルされていない状況がまだあります。 コードツリーには、数年前に書かれたセクションがあり、誰も触れず、誰も変更せず、何らかの形で機能しなくなっただけでなく、コンパイルされました。 同時に、コードのこれらのセクションは、99％、99.9％、または99.98％のケースでは使用されていませんが、Xが来ると（そ​​してマーフィーの法則によれば、彼は間違いなく本番システムに入ります）、顧客は非常に怒っています。



これに対処することは可能ですか？ 必要です！ 手動-プロジェクト内のすべてのコードを完全に定期的にコンパイルし、できれば自動的にコンパイルします。 たとえば、コードの毎週のコンパイルをセットアップし、できればこのコードをいわゆる「統合サーバー」にアップロードします。これには、いくつかの場所に最新の、おそらく動作しないコードがあります。 しかし、このサーバーでは、常にプロジェクトの現在のステータスを把握でき、コンパイルエラーを見逃すことはありません。



別の行で、Java以外のファイルのコンパイルに言及する価値があります。 ほとんどすべてのスクリプトおよびJSPページで、コンパイルエラーがないかファイルをチェックする方法があります。 たとえば、WebLogicでは、Webアプリケーションのプリコンパイルパラメーターを指定できます。 JSPにエラーが含まれている場合、プリコンパイルが中断され、エラーがログに記録されます。 WebLogicには、個々のjspに対して同じ機能を実行するjspcユーティリティも含まれています。



操作性





コードは機能しますか？ どうやって知っていますか？ すべてのコードが現在機能しているのか、または一部がちょうど2週間前に機能していたのかを確認できますか？ しかし、実際に変更を加えると、必要な場所ではなく、必要でない場合でも何かを「壊す」可能性があることはご存じでしょう。



または、あなたの保証は、QA部門（品質保証）がすべてのバグ/問題/パッチに「合格」したという事実に基づいていますか？ まだ何の意味もありません！ まず、一部の企業では、QAはシステムの変更部分のみをテストします。 第二に、QAは怠け者でプロフェッショナルではない可能性があり、最近の無名の企業での経験が示すように、合格するには怠tooすぎるテストに合格することができます。 第三に、ほとんどの人はQAでも働いており、彼らも間違いを犯す可能性があります。



したがって、パフォーマンスを評価するために、どこでも自動テストがすでに使用されています。 これらは、コードが100％操作可能であるかどうかを評価できます。 確かに、これには以下が必要です。

  • 自動テストは
  • これらの自動テストは、機能の100％をカバーする必要があります




最初の項目が組織的なタスクである場合、2番目の項目は技術的なものです。 また、「100％」では、特定の要件の各要件のテストではなく、少なくとも1つのテストの結果としてすべてのコード行が実行されるという事実を理解する必要があります。 この機能は「 コードカバレッジ 」と呼ばれ、文字通りテストでコードがカバーされる度合いを意味します。 次のインジケータが異なります。



  • 関数カバレッジ-メソッド呼び出しの計算
  • 決定カバレッジ-コード実行の可能な方向による計算（制御構造内のthen-elseまたはcase-case-default）。 それぞれの場合に単一のブランチを考慮します。
  • ステートメントカバレッジ-特定のコード行での計算
  • パスカバレッジ-コード実行の可能な方法の計算。 すべてのブランチの結果を考慮に入れるため、意思決定カバレッジよりも広い概念。
  • 条件付きカバレッジ-コード内のブール式および部分式の値の計算の可能な結果の計算。




一部の指標は純粋に理論的であり、通常、パスカバレッジなどの実際のアプリケーションには使用されません。 反復回数が可変のサイクルでこのインジケーターに最大100％到達すると、可能なすべての反復回数をチェックする必要があります...そして、それがユーザーに依存するかどうか？ はい、確認することはほとんど不可能です。 したがって、通常は「機能カバレッジ」または「ステートメントカバレッジ」のいずれかが使用されます（後者から決定カバレッジを取得するのは簡単です）。 オープンソースプロジェクトから、次のことに気付くことができます。

  • emma.sourceforge.net -EMMA
  • cobertura.sourceforge.net -Cobertura

OK、あなたのコードが機能すると仮定します。 彼にも理解してもらいたいです。 これを行うには、多くの手法とルールが使用されます。最初の方法は、コードの設計と特定のスタイルの対応です。 たとえば、当社では次のスタイルを使用しています：「 Sun Java Conventions + {} on new line」。 このような理解可能な概念の著者に感謝したいと思います。 私のIDEでは、「Java Conventions」のデザインルールを採用し、それらを新しいルールにコピーして、1つ（まあ、5つ）の設定のみを変更すれば十分だからです。



なぜすぐにIDEについてですか？ はい、コードのスタイルはプログラマではなくIDEでサポートされるべきであり、プログラマはプログラムが気にしないことだけを確認する必要があるためです。 しかし、プログラムが間違っていたが、プログラマーが従わなかった場合はどうなりますか？ 繰り返しになりますが、検証の自動手段が役立ちます。

  • checkstyle.sourceforge.net-チェックスタイル


しかし、スタイルはデザインとインデント内のスペースの数で終わっていません。 次のステップは、コードで一般に受け入れられている標準または企業標準の使用です。 たとえば、 SerializableクラスにはserialVersionUIDが必要であり、Exceptionは不変である必要があり、catchブロックの例外値は変更されません。 また、これらの契約に違反しても現時点ではエラーは発生しませんが、標準ルールに従うことを期待している人がコードを理解しにくくします。 このような矛盾がないかコードをチェックするプログラムの中で、同じチェックスタイルとFindBugsプログラムについて言及したいと思います。

  • findbugs.sourceforge.net -FindBugs


コード検証のこの領域は、 静的分析と呼ばれます 。 これには、たとえば、Serializableクラス内のserialVersionUIDの存在や、オープンI / Oストリームなどのより複雑なチェックなどの単純なチェックが含まれます。 繰り返しますが、これらのユーティリティは、テストの100％に合格し、100％のカバレッジを持つコードでエラーを見つけることができます。 そのとき、これらのユーティリティは、潜在的なエラーとプログラムの標準プログラミングスタイルとの矛盾を検出します。



EclipseでFindBugsを使用する



部分的な統計分析は、IDEで実行できます（Eclipse-コンパイル時、IDEA-「コード検査」）。 ユーティリティは2つのグループに分けられます-ソースコード（ほとんど）で機能するグループと、コンパイル済みファイルで既に機能するグループです。 FindBugsがコンパイルされたコードで動作することは注目に値します。つまり、上司と顧客の両方がそれをチェックできます。良い結果（絵文字）があることに注意する方が良いと思います。

コードの読みやすさ



もちろん、これまでのところ、Microsoft Word（OpenOffice Writer）でさえ、良い文章と悪い文章を区別することはできません。 最大は、文法的および文体的なエラーを修正することです。 それでも、少なくとも何か。



したがって、コードが「読みやすさ」の概念を形式化することは困難ですが、読み取り可能なコードを読み取り不可能なコードからマークするいくつかの特性があります。

  • 画面に収まる120行以内の単一行サイズ
  • メソッドのサイズは1画面以下であるため、読者はメソッド全体を見て、自分が何をしているかを理解できます。
  • クラスサイズは1000行以下-BLOBアンチパターンなし
  • 「魔法の」定数がない（たとえば、5、7、10 ...読者はこれらの数字が何であり、どこから来たのかを簡単に理解する必要がある）


これらのチェックのほとんどはコードのスタイルに関連しており、すでに呼び出されているユーティリティによってチェックされます。



まとめると





ユーティリティがコードの品質を自動的に監視するために、最終的に何をする必要がありますか？

  • バージョン管理システムがない場合は、入手する必要があります。 とはいえ、私はまだあなたがそれを持っていることを望みます。
  • 会社がコードや制限を記述するための特定のルール（テンプレート）を持っている場合、それらは形式化され、可能であれば、自動ユーティリティの1つが理解できる言語で記述される必要があります。
  • 専用サーバー（統合）をセットアップします。これは、毎晩現在のHEAD / Streamからすべてのコードを取得し、コンパイルしてサーバーに「アップロード」します。 できれば、少なくとも記述したコードについては、サーバーは-ea（アサーションを有効にする）モードで起動する必要があります。
  • コンパイル中に、ソースコードを使用するユーティリティ（Checkstyleなど）でコードをチェックし、その後、バイトコードを使用するユーティリティ（FindBugsなど）でコードをチェックする必要があります。
  • コンパイルの結果、特にエラーがある場合は、電子メールに送信します
  • サーバーを再起動した後、サーバーで一連の自動テストを実行します。
  • テストを実行するとき、カバレッジといくつかの内部パフォーマンス統計を評価するユーティリティを含めると便利です。 結果をデータベースに保存し、しばらくすると、開発プロセス中にコードを改善、スピードアップ、またはスローダウンする方法を示す適切なインジケーターを蓄積できます。
