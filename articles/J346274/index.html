Hi DPIサポートとベクターロゴを備えた純粋なWINAPI Webインストーラー

現在、これはおそらく純粋なWINAPIで記述されたアプリケーションの主要なアプリケーションの1つです。 純粋なWINAPIでいくつかの単純なウィンドウよりも深刻なものを書くことはもはや楽しいことではありませんが、小さなインストーラーが重要です。



2018年は年なので、アプリケーションだけを書くのはどういうわけかあまり良くありません。 時代のトレンドに合わせてみましょう-インストーラーはHi DPIモードをサポートします。 ラップトップでも、デスクトップはもちろんのこと、4Kスクリーンも珍しくありません。 さて、インストーラーはすぐに起動するはずのものなので、私たちにとって本当に難しくないことを節約します。 さて、追加のライブラリなしでベクターグラフィックスを使用するためにチートを試みましょう-美しいロゴも必要です！







まず、高DPIデスクトップアプリケーションの作成方法に関するMicrosoftの最新（今日）のガイドです。 これは、WindowsアプリケーションのさまざまなDPIのサポートモードの難易度を示しています。 そして、すでにいくつの異なる機能が追加されているのか... Windows 10より前は、システムはDPIに従ってフォントを描画することしかできませんでしたが、同時に、インデントやその他のサイズはひどく「フローティング」で、他はすべて引き伸ばされていました。 このモードは、XPスタイルのDPIスケーリングと呼ばれます。 Vistaからは、システムがアプリケーションをバッファーに引き込み、最終的な画像を引き伸ばしましたが、結果もまあまあで、すべてがぼやけていました。 Windows 8.1以降、システムはDPIの変更に関するメッセージをその場で送信し始めましたが、メインアプリケーションウィンドウのみに送信されました。 DPIの変更を確認するためにユーザーセッションを閉じる必要はありません。 そして、Windows 10でのみ、システムはGDI（テキストおよび共通コントロール）を介して行われたすべてを再描画することを学び、すぐにではなく、Creators Updateの後、Windows 10でもすべてを行うことはできません：







したがって、DPI対応アプリケーションを作成する場合は、ドキュメントに従ってUWPを実行する必要があります。このフレームワークのみがDPIをサポートし、すべてを行います。 それ以外はすべて、何らかの形での手動制御が必要になります。



実際、これは実際には必要ありません。すべてを自分で描きます。 最初に、現在のDPIを取得し、システムからのメッセージをリッスンします-DPIをオンザフライで変更します（システムがサポートしている場合）。 次に、必要に応じて単純にウィンドウ/コンポーネントを描画し、DPIを考慮して、プログラムの実行中にユーザーがDPIを変更するか、DPIが異なる別のモニターにウィンドウをドラッグする場合は更新します。 コモンコントロールを使用することはできません。Windows10 1703より前のDPIの変更に対応することはできません。ご自身で行ってください。



ラッパーはすべてのコンポーネント（ウィンドウとコントロール）向けに作成されているため、ウィンドウプロシージャを「C」スタイルで作成するのは好ましくありません。 ここでは、ゼロからでも多くの作業はありません（既にすべてを事前に知っている場合）。そして、すべてのWINAPIプログラマーは、ストレージに同じようなものを長く持っています。 必要なもの：



  1. Windows向けの抽象的なもの（Windows）
  2. コントロールの抽象的なもの
  3. テキスト要素（テキストボックス）
  4. ボタンテキストとエフェクト（ボタン）
  5. 進捗インジケータ
  6. リンク
  7. 設定を選択するためのテキスト付きのフラグ（チェックボックス）
  8. ベクトルのロゴのトリッキーな要素。
  9. さて、プラスそれらの異なる組み合わせ


WINAPIでは、ウィンドウとコントロール（これもウィンドウです）の操作の理論全体を伝えることは適切ではありません。 つまり、次のようなものです。GUI要素が作成されると、システムはそのハンドル（HWND）を返し、作成された要素のデータを指します。 要素とウィンドウにはメインプロシージャがあります-要素への呼び出しを解析し、要素からの要求を実行します（または実行しません）。 要素は、メッセージ（メッセージ自体とその2つのパラメーター：wparamおよびlparam）を介してアクセスされます。 何かをやり直すために、メインプロシージャを独自に再定義し、必要なメッセージを処理し、メインプロシージャを制御して、さらに何かを分解する必要があることを通知するか、すでにすべてを行っています。 他のシステムでは、GUIはほぼ同じです-複雑なものはありません。



さて、あなたはWINAPIプログラマです。 ほぼ。



最終的に行うのはメインウィンドウです。ここでは、いくつかのボタンを押していくつかのオプションを選択できます。 [インストール]ボタンをクリックすると、背景がぼやけたファッショナブルな進行状況インジケーターが表示されます。これは、Windows XPから動作します。 ダイアログボックスを基礎としてみましょう-これが最も簡単な方法です。







では、順番に見てみましょう。HiDPIについてはどうでしょうか。 Windows自体は、アプリケーションマニフェストを使用して、アプリケーションからDPIサポート（dpi対応）を定義します。これは、しばらくの間実質的に必須でした。 たとえば、Windows c 8.1のDPIに加えて、GetVersion関数も変更されました -マニフェストでサポートするシステムが示されていない場合、GetVersionは6.2（Windows 8より古いすべてのシステムでWindows 8でアプリケーションが実行されているかのように）またはリストから最も適切なものを返します追加するシステム。 したがって、マニフェストは多くの理由で必要です。 マニフェストの例はここで見つけることができ 、すべての可能なマニフェストオプションはここにリストされています 。



アプリケーションには、DPIを操作するための4つのオプションがあります。



  1. DPI Unaware - DPIについて何も知りません（すべてはシステムによって行われますが、どのように行われるか）
  2. システムDPI認識 - システムDPIを取得し、最初にアプリケーションを初期化します（Widows Vista）
  3. モニターごと -システムのDPIではなく、アプリケーションが表示されるモニターのDPIを取得します。 ユーザーが異なるDPIの異なるモニター間でアプリケーションをドラッグしたときにメッセージを処理しますが、それらはトップレベルウィンドウにのみ表示されます（Widows 8.1）
  4. モニターごと（V2）はモニターごとと同じですが、システムはすべてのウィンドウにメッセージを送信します。 さらに、システム自体は、ウィンドウの非クライアント部分のサイズ、共通コントロール、およびCreateDialog（Creators Updateビルド1703以降のWindows 10）で作成されたダイアログボックスの初期サイズを調整します。


最後の3つのモードでは、システムは（ほとんど）アプリケーションから「手を離し」、何にも触れず、仮想化せずにすべての座標とサイズをピクセル単位で提供します。 もちろん、上記の点を除きます。 Per-Monitorモードは私たちにとって最適であり、Per-Monitor V2の機能は干渉するだけです。



しかし、これはもちろん問題がないわけではありません。 Visual Studioのデザイナーでダイアログボックスを作成するのが好きなので、深刻な問題が発生します。 マイクロソフト自体は、対話型アプリケーションを手動で作成し、すべての要素を手動でウィンドウに追加することを提供しています -申し訳ありませんが、これを購読していません。 問題は、リソース（ダイアログテンプレート）のダイアログボックスサイズがDLU（DIALOG UNITS）に保存されることです。 また、現在のDLU値は、システムのDPIに応じて変わります。 DPIとDIALOG UNITSが互いに重ねられていることがわかります。 さらに、DPIとは異なり、オンザフライで（DIALOG UNITS）は変更されず、DPIを変更した後は、新しいセッションでのみ機能します。 それだけでなく、 GetDialogBaseUnits（）関数を使用して変更をカウントしようとすると、サイズは変動します。 実際のウィンドウサイズの計算は、 このメッセージを間接的に確認するいくつかの他のパラメーターに依存していることがわかります 。 つまり、アプリケーションの開始時に初期の正しいサイズを示すことはできません。 ダイアログデザイナを拒否したくないので、ウィンドウの初期サイズとウィンドウ内の要素を常にあたかも96 DPIであるかのように示すコードが表示され、DPIに応じて変更します。 ダイアログとその要素のリソースについて説明する必要がありました。 さらに、ウィンドウ/要素のサイズはデフォルトでDLUで乗算され、初期化ウィンドウはこの形式に縮小されます。



しかし、今ではすべてが単純です-ウィンドウのサイズと開始時に要素を調整した後、デフォルトのDPI（96）を使用して、ウィンドウの現在のDPIのウィンドウサイズの初期化を行い、ウィンドウ内のすべての子に対しても行います。 これは1ステップで実行できますが、2つ残しました。 そして、そこで何が起こっているのかはまったく明確ではありません。



ただし、メッセージWM_DPICHANGEDがウィンドウに到着すると、最初のパラメーターは新しいDPIになり、2番目はシステムが送信するウィンドウの推奨サイズと位置になることを忘れてはなりません。 これを考慮に入れずに自分で考えてみると、異なるDPIのモニター間でウィンドウをドラッグすると、ウィンドウが画面の床に隠れます（実装方法によって異なります）。

つまり、ウィンドウ自体のサイズを処理する必要はありません。システムは希望のサイズを送信します。 ただし、子ウィンドウは処理する必要があります（モニターごとのモードバージョン1があります）。



アプリケーションは、単一のウィンドウと、さまざまな共通のプロパティおよびオプションを備えたInstallManagerとして実装されます（シングルトンを介して実装されます）。 したがって、私のプロジェクトでははるかに便利ですが、ここでも干渉しないようです。

ベクトルのロゴなどのトリッキーな要素 



 class LayeredStaticControl : public StaticControl


遠くから始めましょう。 サイズは2つではなく2つ（100％、125％、150％、175％、200％...）になるため、すべての解像度の画像をストックするのは無駄に見えますが、同時にロゴは通常ベクターです。 しかし、WindowsはEMF以外の一般的なものをサポートしていません。これは基本的にEMF / EMF +のGDI / GDI +呼び出しログです。 このオプションは悪くありませんが、いくつかの制限がありますが、非常に複雑なことをする必要があり、必要な画像をEMF +でラップできる場合はそれだけです。 図面を回転させて、エッジを適切にぼかすと、いいかもしれません。 GDI + 1.1を使用するだけで、最小はWindows Vistaです（本質はMetafile.ConvertToEmfPlus関数にあります）。 通常のEMFはアンチエイリアスを行いません。



GDI +を使用しない場合、Windowsのベクターコンポーネントは1つしかありません。これはフォントレンダリングです。 また、1色で実行されるため、ロゴを色に分解する必要があります。



それがlayered_static_controlのすべてです。 いくつかの文字が取得され、目的の色で順番に描画されます。 実際、文字スタイルはすでにフォント内で複合されているため、ここでは特に新しいものはありません。 コンポーネントと色の内部で変更できないため、このメカニズムを使用できない場合を除きます。
