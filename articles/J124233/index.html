「ペニーのエラー」


私のお気に入りの間違いは、わずか1ペニー（セント、時代、または他のコイン）の不一致を引き起こすものです。 追跡が非常に難しく、しばしば丸め誤差が生じるため、私はそれらが好きです。

数年前、私はお金を表示する1つのサイト用の簡単なJavaScript関数を作成する必要がありました。 私はこれを使用しました： 



 <script type="text/javascript"> function GetMoney (amount) {return Math.round (amount * 100) / 100;} </script>


しかし、1.20のような量が1.2の形で画面に表示されることがすぐに明らかになりました。 だから私はこのようにコードを変更しました： 

 <script type="text/javascript"> function GetMoney (amount) { var pounds = Math.floor (amount); var pence = Math.round (amount * 100) % 100; return pounds + '.' + (pence < 10 ? '0' : '') + pence; } </script>


主な違いは、最後の行の余分な0です。 しかし、ペニーは個別に計算されるため、数量を100で割ったときに残りを取得するには％演算子が必要です。このコードでエラーが発生する可能性の低い状況を見つけてください。

これは、ビーズを販売したサイトで発生しました。 その後、ビーズは、分数の値を含むカスタムブレンドを含め、さまざまな形や数量で販売できることがわかりました。 顧客が4.95ポンド相当の1.01品目を購入し、4.00ポンドのみを支払った場合。 金額は4.9995と定義されているため、プログラムはペンスを100に丸め、％100は0ペンスを残しました。 したがって、支払額は4ポンドに減少しました。



丸み付けの単純な欠陥。100個あたり4.95ポンドで販売された101個のビーズに対して、5ポンドではなく4ポンドを支払いました。

私はすぐにコードを修正しました： 



 <script type="text/javascript"> function GetMoney (amount) { var pounds = Math.floor (amount); var pence = Math.floor (amount * 100) % 100; return pounds + '.' + (pence < 10 ? '0' : '') + pence; } </script>


ただし、これは4.9995ポンドから4.99ポンドに切り上げられたため、サーバーからの対応する計算との同期が妨げられたため、適切な修正ではありませんでした。 さらに悪いことに、£1.00相当の0.7を注文した場合、金額は70ペンスから69ペンスになりました！ これは、0.7のような浮動小数点数が0.6999999999999999ではなく2進数で表示され、70に丸める代わりに69ペンスに削減されるためです。

これは本当の「一ペニーの間違い」です。 それを修正するために、最初に別の丸めを追加しました。 



 <script type="text/javascript"> function GetMoney (amount) { var pence = Math.round (100 * amount); var pounds = Math.floor (pence / 100); pence %= 100; return pound + '.' + (pence < 10 ? '0' : '') + pence; } </script>


これで、非常に単純な1つのことを行うために、非常に複雑な4行のコードができました。 今日、私はこの記事を書いているときに、これらすべてを処理できるJavascriptに組み込まれた関数を見つけました。 



 <script type="text/javascript"> function GetMoney (amount) {return amount.toFixed (2);} alert (GetMoney (4.9995) + ' ' + GetMoney (0.1 * 0.7)); </script>




PayPalで割引を提供する


PayPalは「一ペニーの間違い」であり、翼の中で待っています。 多くのサイトでは、注文金額の特定の割合の割引を提供するコードを提供しています。 最後に計算されます。 95ペンスで2つのアイテムを注文した場合、合計金額は£1.90になり、19ペンスの割引を受けるため、£1.71を支払います。

ただし、PayPalはこのタイプの割引をサポートしていません。 PayPalで買い物かごに商品を表示する場合は、それぞれの商品の価格と数量を個別に計算する必要があります。 



 <input name="item_name_1" type="hidden" value="My Difficult Product" /> <input name="amount_1" type="hidden" value="0.99" /> <input name="quantity_1" type="hidden" value="1" />


したがって、アイテムごとに個別に割引を受ける必要があります。 95ペンスから10％割引すると、85.5ペンスになります。 PayPalは小数では動作しないため、86ペンスに丸める必要があります。これにより、PayPalの合計金額は£1.72になります。 85pに切り上げた場合、合計金額は£1.70になります。

この問題を解決するには、各アイテムの割引を個別に計算する必要もありました。 通常の10％×£1.90の計算の代わりに、コードは、ペンスの全額を使用するたびに、アイテムからアイテムへの割引を累積します。 $アイテムが注文アイテムのPHP配列であると仮定します： 



 $discount = 0; $discountpercent = 10; foreach ($items as $item) { $mydiscount = floor ($item->price * $discountpercent) / 100; $item->priceforpaypal = $item->price - $mydiscount; $discount += $mydiscount * $item->quantity; }

時計の変更


このエラーを「エラー」とは呼びません。 特別なまれな状況下で表示されるため、プログラマーの観点からは、これは「欠陥」である可能性が高くなります。 プログラマーが提供できるすべてを超えています。

一週間以上前に完了した注文を強調するこの一見無害なコード行の何が問題なのか推測できますか？

これは、繰り返し注文を実装する低コストの方法でした。 時間があれば、別のテーブルや買い物かごを作成して、繰り返される注文と繰り返されない注文を区別します。 このコードは数か月間は問題なく機能し、3月末には謎の理由で失敗したことが判明しました。

欠陥の解消とその結果には多くの時間がかかり、注文を手動で実行する必要がありました。 原因を特定するのにさらに時間がかかりました。特に、敷地全体を別の日だと考えるようにサイト全体に強制する必要があったためです。

プログラミングのエラーにはさまざまな形とサイズがあります。 この記事では、それらは非常に明白な失敗から信じられないほど微妙な誤算にまで及びます。 そして、彼ら全員がマーフィーの法則を確認しているようです。トラブルが起こる可能性がある場合、それは起こります。



ただし、検出、報告、修正されるエラーごとに、コードにいくつかのエラーが残る場合があります。 それらが見つからない可能性があります（原因となる異常な状態が再現されていないため）、それらは報告されない可能性があります（ほとんどのユーザーがエラーレポートを送信することに煩わされないため）、または不必要なお金や時間のために修正されないコスト。



さらに、エラーは人気のあるサイトでより一般的です-主に多くの人々によって作られているためですが、1つのエラーを修正すると他の場所に新しいエラーが現れる可能性があるためです。 したがって、先を見越して、エラーを注意深く修正する必要があります。
