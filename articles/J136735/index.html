PHPスクリプト用のメール送信サーバーの適切なセットアップ、mail（）関数のセットアップ

このトピックでは、送信メールサーバー、特にPHPのmail（）関数を正しく構成する方法を学習します。 私自身はひどい物言いです。 私は、すべてがその場所にあるすべての場所で、ハッキングに耐えられないことを気に入っています。 私のサーバーからのメッセージヘッダーで完全なナンセンスを見て、私はこれを徹底的かつ無条件に処理しました。 適切な構成とは、大規模なメールシステムのスパムフィルターのニーズを満たし、見た目も美しく意味のあるものを意味します。



ご存じのように、SMTPプロトコルはスパムおよびユーザー認証に対する保護手段を意味するものではありません。したがって、大規模であまり企業ではないプロトコルセキュリティの「修正」を考え出しました。

ドメインがホストされている専用サーバーをインストールする場合は、すべてが正常に機能するようにこれらの設定を実行することを強くお勧めします。



私はこの問題の研究に多くの時間を費やし、各見出しを整理しました。 パッケージの基本的な設定から始め、選択したパッケージから好きなものが好きになりましたが、薄い風水になりました。



タイトルのタイトルはまさにそれを選びました。旅の初めに、私はこのようなものを探していたからです。 postfix、php、dns（ptr、spf）などを設定します。



この記事は、専用サーバーをセットアップする人々 にとっては興味深いものですが、普通のプログラマにとってもそれほど興味深いものではありません 。

誰かがメール機能を置き換えるライブラリ（phpmailerなど）を使用するか、SMTPと直接通信するか、ローカルサーバーに送信できます。 ただし、標準の組み込みおよび構成済みのPHP mail（）関数を使用する方が適切です。この関数は、ローカルの送信メールサーバーにメールを送信し、そのメールサーバーがそれを処理します。 この機能を使用する場合、ローカルメールエージェントはメールをすぐに受け入れたり拒否したりするため、ブレーキは常に最小です。



記事は最適な方法でフォーマットされていません。 申し訳ありませんが、時間がなくなっていますが、覚えている間に伝えたいと思います。さもないと時間がありません。

最初に起動すると、このような美しい見出しがないことがわかります。 受信メールサーバーは、あなたをある種のchummekと見なします。これについては、レターのヘッダーにこれを少しだけ警告して書き、確実にブラックリストに入れます。 ヘッダー内のすべての情報は、スパムフィルターの設定に応じてメッセージを拒否または受け入れるために、すべてのサーバーチェーンと受信エージェントによって使用されます。 ほとんどのドラゴンの方法はブラックリストですが、ソフトの方法は見出しと苦情によって悪い方法を特定します。



また、mail（）関数で4番目のパラメーターを設定し、そこにFromヘッダーを配置すると、From：no-reply@wartur.ruヘッダーも変更されます。設定しない場合、ヘッダーと同じように自動的に検出されますReturn-path：<no-reply@wartur.ru>、これは私の見出しにあります。

このようなスキームは非常に便利です。Fromヘッダーが変更されたサーバーを送信し、Gmailで（ドメインサービスを介して）受信します。 メールは、サーバーが構成されているドメインと同じドメインに属していること、および以下にリストされているいくつかのポイントが満たされていることが望ましいです。

紳士、私はすべてをパッケージに入れるのが好きです。初心者にはアマチュアアクティビティを行うことはお勧めしません。多くのパッケージには既に自動構成用のスクリプトがあり、その結果はわずかに修正するだけで問題ありません。 さらに、リポジトリからは、それ自体が更新され、一般的に少ないため、一般的にzaparaは少なくなります。



-PHPセットアップ

私たちはphpが欲しいので、私がいなくてもあなたです。



PHP設定に移動し、同等に置き換えます： 

 sendmail_path = / usr / sbin / sendmail -t -i -fno-reply@wartur.ru




これは、見出しReturn-Pathを示しました：<no-reply@wartur.ru>



ちなみに、1：サーバーのすべての仮想ドメインに対して、それを修正し、現在の仮想ドメイン用に構成するには、5番目のパラメーターをmail（）関数に渡す必要があります。 

 $result = mail('YOURMAIL@DOMAIN.DOM', 'subject', 'message', 'From ...', '-fno-reply@yourdomine.ru');


方法2：共有ホスティングに5番目のパラメーターを設定する場合は、以下で説明するように、ドメインを構成することもお勧めします。 1 IP =ドメインであるため、PTRレコードで操作を行うことは不可能であることに注意してください。少なくとも2を行うことはお勧めできません。2まだ覚えていません。

私は心からpostfixをお勧めします、そして/etc/postfix/main.cf settingsに行きます

すでにsendmailに遭遇し、システムからそれらを切り取るのがどれほど難しいかに気付いた人のために、私はコマンドをアドバイスします： 

 aptitude purge sendmail




次に、このように同等に行います

myhostname = wartur.ru

#smtp_helo_name = k-wartur.wartur.ru

mydestination = _ALL_、wartur.ru、k-wartur.wartur.ru、localhost.wartur.ru、localhost



（k-wartur.wartur.ru-これはwartur.ruドメインのサーバーです）



「私が間違っていなければ、それは助けになりました。」 配信エージェントが別のサーバーに提示されると、HELOが送信されます。 myhostnameを変更すると、次のヘッダーが変更されます。

メッセージID：<20120121212420.460BC9B7@wartur.ru>



-smtp_helo_nameはデフォルトではmyhostnameで、資料を読んでください



-mydestination-間違っていない場合、これはMTAが処理のためにメールを受信するドメインを設定します。共有ホスティング（自宅のサーバーにインストールしている）がある場合、_ALL_なしでは、示されたドメイン以外からのメール送信に問題が発生します。



さて、あなたは、見出しがすでにはるかに正しいと思います、そうではありません、そうでなければ最初は恐怖がありました。



-メールサービス（メーラー）との対決

そして彼らは私たちを不親切な情報源だと考えています。

私たちのサーバーは自分自身を正しく紹介することができましたが、とにかく私たちを信頼していないのはメーラーだけです。

DNSを使用するドメイン所有者（もちろん、私たちのように読み書きができる場合）は、そのようなサーバーは信頼できるが、他のサーバーは信頼できないことを示すことができるように、アルゴリズムが考案されました。



あなたが今これを持っているべきであるという恐怖の代わりに： 

 Received-SPF：pass（google.com：no-reply@wartur.ruのドメインは、許可された送信者として188.134.79.140を指定します）client-ip = 188.134.79.140;
 Received-SPF：pass（mx64.mail.ru：wartur.ruのドメインは、許可された送信者として188.134.79.140を指定しています）client-ip = 188.134.79.140;  envelope-from=no-reply@wartur.ru;  helo = wartur.ru;




私たちはすでに真実に向かっています。 ちなみに、デフォルトでは、メーラーはすべてをv = spf1〜allとみなします。これにより、ブラックリストに入るためのしきい値が下がります。また、ドメインを使用してスパムを送信することもできます。



DNSにアクセスして、ドメインにTXTレコードを作成します。私の場合はwartur.ru

v = spf1 + mx + a：wartur.ru + a：k-wartur.wartur.ru + include：gmail.com -all



v = spf1

つまり、アルゴリズムのspf1バージョン



+ mx

mx.wartur.ruが受信サーバーである場合など、mxサーバーから送信されたメールを信頼できます。このサーバーが送信する場合は、すべて正常に動作します。



+ a：wartur.ru

wartur.ruドメインを提供するサーバーを信頼できます



たとえば、メールサーバーにpupkin.ruがあったことも追加します。

Pupkinに送信メールサーバーk-wartur.wartur.ruへのドライブを提供しました（共有ホスティングの通常の状況、データベースのように常にローカルにコストがかかります、それだけです）。 その後、 彼は+ aを設定する必要があります：k-wartur.wartur.ru



+含める：gmail.com

gmail.comのすべてのルールをインポートします。また、サーバーのTXTレコードもあります。 （ドメインにGoogleサービスを使用しています）

-メールサービスでの対決、逆PTRの設定

メーラーはまだ追加のティックの逆PTRドメインゾーンを要求します。このティックはサーバーを100％定義しているため、 最も大胆なものの1つであり、正しく動作する場合は非常に信頼され、すべての文字が宛先に届く可能性があります。

X-Mru-PTR：wartur.ru、Googleでも受信したこと：wartur.ru（wartur.ru。[188.134.79.140]）



注：

共有ホスティングでは、これを行うことは許可されません。彼らはこのアドレスにservername.badhosting.ruという独自のサーバーを持ち、通常は独自のIPを持つ独自のサーバーでのみこれを行うことができます。
