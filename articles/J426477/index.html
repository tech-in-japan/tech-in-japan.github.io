パーティションプール情報の取得



このサービス呼び出しは、パーティションプールに関する部分的な情報を提供します。 Nucleus SEの実装は、オブジェクトの命名と一時停止要求がサポートされておらず、タスクを一時停止できないため、返される情報が少ないという点でNucleus RTOSと異なります。



Nucleus RTOSでのパーティションプール情報の呼び出し



プロトタイプを呼び出す：



STATUS NU_Partition_Pool_Information（NU_PARTITION_POOL * pool、CHAR * name、VOID ** start_address、UNSIGNED * pool_size、UNSIGNED * partition_size、UNSIGNED * available、UNSIGNED * allocateed、OPTION * suspend_type、UNSIGNED * tasks_waiting ** first_task;最初



パラメータ：



pool-情報が要求されるパーティションプールへのポインタ。

name-パーティションプールの名前の8文字の宛先領域へのポインター。 ゼロを終了する場所が含まれます。

start_address-パーティションプールのデータ領域の先頭へのポインターを受け取る変数へのポインター。

pool_size-パーティションプールのサイズ（バイト単位）を受け取る変数へのポインター。

partition_size-特定のプール内のパーティションのサイズを取得する変数へのポインター。

available-特定のプールで現在使用可能なパーティションの数を受け取る変数へのポインター。

割り当て -特定のプールで現在使用されているパーティションの数を受け取る変数へのポインタ。

suspend_type-タスク中断のタイプを取得する変数へのポインター。 有効なサスペンションタイプ： NU_FIFOおよびNU_PRIORITY ;

tasks_waiting-このパーティションプール内の中断されたタスクの数を受け取る変数へのポインター。

first_task-最初に中断されたタスクのポインターが配置されているタスクポインターへのポインター。



戻り値：



NU_SUCCESS-呼び出しは正常に完了しました。

NU_INVALID_POOL-パーティションプールへの無効なポインター。



Nucleus SEでのパーティションプール情報の呼び出し



プロトタイプを呼び出す：



STATUS NUSE_Partition_Pool_Information（NUSE_PARTITION_POOLプール、ADDR * start_address、U32 * pool_size、U16 * partition_size、U8 * available、U8 *割り当て済み、U8 * tasks_waiting、NUSE_TASK * first_task）



パラメータ：



pool-情報が要求されるパーティションプールのインデックス。

start_address-パーティションプールのデータ領域の先頭へのポインターを受け取る変数へのポインター。

pool_size-パーティションプールのサイズ（バイト単位）を受け取る変数へのポインター。

partition_size-特定のプール内のパーティションのサイズを取得する変数へのポインター。

available-特定のプールで現在使用可能なパーティションの数を受け取る変数へのポインター。

割り当て -特定のプールで現在使用されているパーティションの数を受け取る変数へのポインタ。

tasks_waiting-このパーティションプールで中断されたタスクの数を受け取る変数へのポインター（タスクの中断が無効になっている場合、0が返されます）。

first_task-最初に中断されたタスクのインデックスを受け取るNUSE_TASK型の変数へのポインター（タスクの中断が無効な場合、0が返されます）。



戻り値：



NUSE_SUCCESS-呼び出しは正常に完了しました。

NUSE_INVALID_POOL-無効なパーティションプールインデックス。

NUSE_INVALID_POINTER-渡された 1つ以上のポインターが無効です。



Nucleus SEでの情報検索とパーティションプーリングの実装



このようなAPI呼び出しの実装は簡単に実行できます。







この関数は、パーティションプールのステータスを返します。 次に、APIコールブロッキングが有効になっている場合、保留中のタスクの数と最初のタスクのインデックスが返されます（それ以外の場合、これらのパラメーターは0に設定されます）。



パーティションプールの数を取得する



この呼び出しは、アプリケーションで構成されているパーティションプールの数に関する情報を返します。 Nucleus RTOSではこの数は時間とともに変化し、戻り値は現在のプール数を表しますが、Nucleus SEでは戻り値はアセンブリ中に設定され、変更されません。

パラメータ：

パラメータ：



欠席

実装



このAPI呼び出しの実装は非常に簡単です。NUSE_PARTITION_POOL_NUMBERシンボルの#define値が返されます。



データ構造



他のすべてのNucleus SEオブジェクトと同様に、パーティションプールはROMとRAMの両方で構造配列を使用します。その数は、設定で指定されたプールの数に依存します。



アプリケーションコードは、直接ではなく、API関数を介してこのようなデータ構造にアクセスすることを強くお勧めします。 これにより、Nucleus SEの将来のバージョンとの非互換性や望ましくない副作用が回避され、Nucleus RTOSへのアプリケーションの移植も簡素化されます。 以下は、サービス呼び出しコードとデバッグの理解を容易にするためのデータ構造の詳細な説明です。



RAMに配置されたカーネルデータの構造



そのようなデータ構造には以下が含まれます。



NUSE_Partition_Pool_Partition_Used [] -現在使用されているプールのカウンターを含む、構成されたパーティションプールごとに1つのエントリを持つ、タイプU8の配列。

NUSE_Partition_Pool_Blocking_Count [] -各パーティションプールでブロックされたタスクのカウンターを含むU8型の配列。 この配列は、API呼び出しをブロックできる場合に存在します。



このようなデータ構造は、Nucleus SEの起動時にNUSE_Init_ Partition_Pool（）を使用してゼロに初期化されます。 これは、各プールの各パーティションを未使用（無料）にするため、論理的です。 次の記事では、Nucleus SEの起動手順について詳しく説明します。



以下は、 nuse_init.cファイル内のデータ構造の説明です。







RAMユーザーデータ



ユーザーは、各パーティションプールのデータを保存するために、RAMに領域を割り当てる必要があります。 RAMのスペースの量は、構成されたパーティションのボリュームに対応する必要があり（以下の「ROMのデータ」を参照）、プール内の各パーティションに追加のバイトが必要です。 データ領域の各セクションの前には、ステータスバイトが1つあります。



ROMデータ



これらには以下が含まれます。



NUSE_Partition_Pool_Data_Address []-ADDRタイプの配列。構成されたパーティションプールごとに1つのエントリがあり、データストレージ領域の先頭のアドレスが含まれます。

NUSE_Partition_Pool_Partition_Number [] -構成されたパーティションプールごとに1つのエントリを持つタイプU8の配列。プール内のパーティションの数に関する情報が含まれます。

NUSE_Partition_Pool_Partition_Size [] -構成されたパーティションプールごとに1つのエントリを持ち、プールのパーティションのサイズを含むU16型の配列。



このようなデータ構造は、 nuse_config.cで （静的に）宣言および初期化されます 。







パーティションプールのデータフットプリント



Nucleus SEのすべてのコアオブジェクトと同様に、パーティションプールに必要なメモリ量は予測可能です。



すべてのアプリケーションパーティションプールのROMサイズ（バイト単位）は、次のように計算できます。



NUSE_PARTITION_POOL_NUMBER *（sizeof（ADDR）+ 2）



API呼び出しのブロックがアクティブになっている場合、すべてのアプリケーションパーティションプールのRAM内のカーネルデータの量は、パーティションプールごとに2バイトのみで、ブロックがアクティブになっていない場合は1バイトです。



RAMにユーザーデータを格納するためのメモリの量はパーティションプールごとに異なりますが、既に述べたように、インデックスnのプールでは次のように計算できます。



NUSE_Partition_Pool_Partition_Number [n] *

（NUSE_Partition_Pool_Partition_Size [n] + 1）



未実現のAPI呼び出し



Nucleus RTOSに実装されているパーティションプールの3つのAPI呼び出しは、Nucleus SEではサポートされていません。



パーティションプールを作成する



このAPI呼び出しは、パーティションプールを作成します。 タスクは静的に作成されるため、Nucleus SEでは必要ありません。



プロトタイプを呼び出す：



STATUS NU_Create_Partition_Pool（NU_PARTITION_POOL * pool、CHAR * name、VOID * start_address、UNSIGNED pool_size、UNSIGNED partition_size、OPTION suspend_type）;



パラメータ：



pool-パーティションプールを管理するためのユーザーユニットへのポインタ。 他のAPI呼び出しでパーティションプールへのハンドルとして使用されます。

name-パーティションプールの名前へのポインタ、終了ゼロを含む7文字の文字列。

start_address-パーティションプールのメモリ領域の開始アドレスを設定します。

pool_size-バイト単位のメモリ領域の合計量。

partition_sizeは、プール内の各パーティションのバイト単位のメモリ量です。 さらに、各セクションに関連付けられた追加の少量のメモリが割り当てられます。これは、使用される2つのデータポインタのおかげで実現されます。

suspend_type-パーティションプールでタスクを一時停止する方法を決定します。 有効なパラメーターオプションはNU_FIFOおよびNU_PRIORITYです。



戻り値：



NU_SUCCESS-呼び出しが正常に完了したことを示します。

NU_INVALID_POOL-パーティションプールコントロールユニットのゼロ値（ NULL ）を示します。

NU_INVALID_MEMORY - start_ address （ NULL ）で定義されたメモリ領域のゼロ値を示します 。

NU_INVALID_SIZE-パーティションサイズが0またはパーティションに割り当てられたメモリより大きいことを示します。

NU_INVALID_SUSPEND-無効なsuspend_type値。



パーティションプールの削除



このAPI呼び出しは、以前に作成されたパーティションプールを削除します。 オブジェクトは静的に作成され、削除できないため、Nucleus SEでは必要ありません。



プロトタイプを呼び出す：



STATUS NU_Delete_Partition_Pool（NU_PARTITION_POOL *プール）;



パラメータ：



pool-パーティションプールコントロールユニットへのポインタ。



戻り値：



NU_SUCCESS-呼び出しが正常に完了したことを示します。

NU_INVALID_POOL-パーティションプールポインターの無効な値を示します。



パーティションプールポインター



このAPI呼び出しは、システム内のすべてのパーティションプールへのポインターの順次リストを作成します。 オブジェクトはポインターではなくインデックスによって識別されるため、Nucleus SEではこれは必要ありません。



プロトタイプを呼び出す：



UNSIGNED NU_Partition_Pool_Pointers（NU_PARTITION_POOL ** pointer_list、UNSIGNED maximum_pointers）;



パラメータ：



pointer_list-ポインターの配列へのポインターNU_PARTITION_POOL ; 配列には、システム内の構成済みプールへのポインターが入ります。

maximum_pointers-配列に配置できるポインターの最大数。



戻り値：



配列に配置されたNU_PARTITION_POOLポインターの数。



Nucleus RTOS互換



Nucleus SEを開発する際の主なタスクの1つは、Nucleus RTOSとの高レベルのコード互換性を確保することでした。 パーティションプールも例外ではなく、開発者の観点からは、Nucleus RTOSとほぼ同じ方法で実装されています。 互換性のない既存の領域も受け入れられますが、最終的なコードの方が理解しやすく、メモリの面でより効率的であることを考慮する価値があります。 ただし、Nucleus RTOS API呼び出しは、Nucleus SE呼び出しとしてほぼ直接使用できます。 将来、Nucleus RTOSのユーザーによるNucleus SEの使用に関する情報を含む記事が計画されます。



オブジェクト識別子



Nucleus RTOSでは、すべてのオブジェクトは特定のタイプのデータ構造（コントロールユニット）によって記述されます。 この制御ブロックへのポインターは、パーティションプールの識別子として使用されます。 Nucleus SEでは、メモリをより効率的に使用するために別のアプローチが必要であると判断しました。 すべてのカーネルオブジェクトは、RAMおよび/またはROMのいくつかのテーブルで記述されます。 これらのテーブルのサイズは、すべてのオブジェクトの構成可能なタイプの数によって決まります。 特定のオブジェクトの識別子は、これらのテーブルのインデックスです。 したがって、 NUSE_PARTITION_POOLはU8 と同等であると判断しました。その後、このタイプの変数（ポインターではない）がタスクの識別子として機能します。 このわずかな非互換性は、コードがNucleus RTOSとの間で移植されているかどうかを簡単に把握できます。 通常、オブジェクト識別子は変更されずに保存および送信されます。



Nucleus RTOSは、パーティションプールの命名もサポートしています。 これらの名前は、デバッグにのみ使用されます。 メモリを節約するために、Nucleus SEからそれらを除外しました。



セクションの数とそのボリューム



Nucleus RTOSでは、プールの合計ボリュームとパーティションのボリューム（さらに2つのポインターを運ぶ）に基づいて、パーティションプールが構成されます。 これらのパラメーターは、UNSIGNED（約32ビット）として定義されます。 Nucleus SEでは、パーティションのサイズ（追加バイトが追加されている）とパーティションの総数に基づいて、パーティションプールが構成されます。 これらのパラメーターは、それぞれU16およびU8として定義されています。



未実現のAPI呼び出し



Nucleus RTOSは、パーティションプールを操作するための7つの呼び出しをサポートしていますが、そのうち3つはNucleus SEには実装されていません。 これらの課題の詳細と除外の理由については、上記で概説しています。
