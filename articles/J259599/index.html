IP PBX 3CX電話システムを監視するための管理ツール（パート1）

IP-PBXは、「ブート」がハングして忘れたように動作するはずです。 このルールは、すべてが可能な限りシンプルで理解しやすく、あまり機能的ではないアナログステーションでうまく機能しました。 IP PBXはもう少し複雑で、少なくとも3CXについて話している場合は、監視する必要があります。



したがって、稼働中のシステムに関するメンテナンススタッフを知るために必要なもの：

  1. ステータス（稼働中かどうか）
  2. ダウンロード（使用できるリソースがあるかどうか）
  3. アラームと通知
  4. ログ形式のアクティビティのトレース


ステータス

外部回線のステータス。これには、VoIPオペレータ、アナログおよびデジタルゲートウェイ、IP-PBX 3CX間のトランクが含まれます。 アクティブな会話と切断された回線を見ることができます。 必要に応じて、管理者は通話を強制的に終了できます。





サブスクライバーのステータス。ここでは、登録されているサブスクライバー、それらのステータス、キューエージェントの状況を確認できます。 ここで、不要なコールを強制的にドロップすることもできます。





システム番号ステータスは、IVR、キュー、グループ、またはファックスから電子メールなどのサービスのステータスを監視するために必要です。 前のセクションと同様に、コールリセットを使用できます。





リモート接続には、最後に成功した登録と失敗した登録の時刻が表示されます。 こちらから追加の登録リクエストを送信することもできます。





そして最後に、3CXサービスは別のセクションに配置されます。 ステータスを確認し、必要に応じて再起動すると便利です。





リソースの読み込み

プロセッサ-メモリ-ディスク領域。

3CXはまだプロセッサの負荷とメモリを監視できません。 これが必要かどうかは大きな問題です。なぜなら、 IP-PBXは、アラームのみを通過させます。 トランスコーディングを行わない場合、Intel Xeon E5プロセッサの1コアで発生する可能性のある負荷は、1000を超える同時呼び出し、つまり 3CXの場合、最大ライセンスは1024の同時コールであるため、上限は事実上達成できません。 たとえば、G.729のトランスコーディングの状況では、70から90の呼び出しがありますが、より狭い帯域幅のコーデックへの変換が必要な場合は、IP-PBXでサーバーを苦しめずに、このビジネスの下に特殊な鉄片を置く方がより便利です。



ディスクについては-オーバーフローに関する通知を受け取る機会があります。詳細は以下をご覧ください。



アラームと通知

通知を受信する必要があるコンマで区切られた電子メールを指定します。

最も重要なこと：

  1. トランクのステータスが変更されました-ユーザーよりも少し早く問題について知ることができます
  2. トランク上のコールの最大数に達しました-コールの最大数のパラメーターは、VoIPオペレーターの設定で指定されています
  3. ライセンスの制限に達しました-アップグレードの時間
  4. IPアドレスはブラックリストに登録されています-これは、ブロッキング時間を増やすための言い訳です
  5. ハードディスクがいっぱいです-ただし、これはサーバーOS専用です
