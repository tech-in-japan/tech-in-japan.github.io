Raspberry Pi Zero Inside Handy Tech Active Star 40点字ディスプレイ





著者は、新しいHandy Tech Active Star 40点字ディスプレイにRaspberry Pi Zero、Bluetoothの「ホイッスル」、ケーブルを配置し、内蔵USBポートが電力を供給します。 その結果、キーボードと点字ディスプレイを備えた、Linuxオペレーティングシステムを搭載した自己完結型の非モニターARMコンピューターが完成しました。 USB経由で充電/給電できます。 電力銀行またはソーラー充電器から。 したがって、数時間ではなく数日間、主電源を省くことができます。







点字ディスプレイの次元の区別



まず、文字列の長さが異なります。 60以上の使い慣れたデバイスはデスクトップコンピューターでの作業に適していますが、40台のデバイスはラップトップでの持ち運びに便利です。 現在、スマートフォンやタブレットに接続されている点字ディスプレイがあり、文字列の長さは14から18です。



過去には、点字ディスプレイはかなり大規模でした。 たとえば、おなじみの40インチは、13インチのラップトップの寸法と重量でした。 現在、同じ程度の親しみで、ディスプレイ上のラップトップではなく、ラップトップの前にディスプレイを置くのに十分なミニチュアです。



もちろん、これは優れていますが、それでも2つの別々のデバイスを膝の上に置くのはあまり便利ではありません。 テーブルで仕事をするとき、苦情はありませんが、ラップトップは別の方法でラップトップと呼ばれ、その名前を正当化しようとすることを覚えておく価値があります。



だから著者は、Handy Tech Starシリーズで長く期待されていた新しいモデルのリリースを待っていました。 2002年に、以前のHandy Tech Braille Star 40モデルがリリースされました。このモデルでは、ボディ領域がラップトップを上に置くのに十分です。 収まらない場合は、格納式スタンドが提供されます。 現在、このモデルはActive Star 40に置き換えられていますが、ほとんど同じですが、電子機器がアップグレードされています。







そして引き込み式のスタンドは残った：







しかし、目新しさの中で最も便利なことは、ほぼスマートフォンのサイズのくぼみです（KDPVを参照）。 プラットフォームが戻ると開きます。 スマートフォンをそこに置いておくのは不便であることがわかりましたが、どういうわけか、空のコンパートメントを使用する必要があります。



著者が最初に思いついたのは、Raspberry Piをそこに置くことでしたが、ディスプレイを購入すると、「raspberry」ではコンパートメント内でスライドしなかったスタンドがコンパートメントを閉じることがわかりました。 さて、もしボードがたった3mmだけ薄ければ...



しかし、同僚がRaspberry Pi Zeroのリリースについて話しました。RaspberryPi Zeroは非常に小さいため、2つがコンパートメントに収まる可能性があり、おそらく3つにもなりました。 64 GBのメモリカード、Bluetooth、ホイッスル、Micro USBケーブルとともにすぐに注文されました。 数日後、これらすべてが到着し、目が見える友人が著者が地図を準備するのを助けました。 すべてがすぐに正常に機能しました。



これのために何が行われたか



Handy Tech Active Star 40の背面には、キーボードなどのデバイス用の2つのUSBポートがあります。 小型の磁気接続キーボードが含まれています。 キーボードが接続されており、ディスプレイ自体がBluetooth経由で機能している場合、コンピューターはBluetoothキーボードとしても認識します。



したがって、スマートフォンのコンパートメントに配置されたRaspberry Pi ZeroにBluetooth「ホイッスル」を接続すると、 BRLTTYを使用してBluetooth経由で点字ディスプレイと通信でき、キーボードをディスプレイに接続すると「Malinka」が機能し、彼女と。



しかし、それだけではありません。 Malinka自体は、それをサポートするデバイスからBluetooth PAN経由でインターネットにアクセスできます。 著者は自宅と職場でそれに応じてスマートフォンとコンピューターをセットアップしましたが、将来的にはもう1つの「ラズベリー」をイーサネットと別のBluetooth「ホイッスル」に接続されたゼロではなくクラシックなものに適応させる予定です。



BlueZ 5およびPAN



BlueZを使用してPANを構成する方法は明らかではありませんでした。 著者はpyhton-script bt-pan（以下を参照）を見つけました。これにより、GUIなしでPANを構成できます。



これにより、サーバーとクライアントの両方を構成できます。 クライアントモードで作業しているときに適切なD-Busコマンドを受信すると、サーバーへの接続を確立した直後に新しいbnep0ネットワークデバイスを作成します。 通常、DHCPは、このインターフェイスにIPアドレスを割り当てるために使用されます。 サーバーモードでは、BlueZには、各クライアントを接続するためにスレーブが追加できるブリッジデバイスの名前が必要です。 通常、ブリッジデバイスのアドレスを構成し、ブリッジでDHCPサーバーとIPマスキングを開始するだけで十分です。



Systemdを使用したBluetooth PANアクセスポイント



ブリッジを構成するために、著者はsystemd-networkdを適用しました：



ファイル/etc/systemd/network/pan.netdev 



[NetDev] Name=pan Kind=bridge ForwardDelaySec=0


ファイル/etc/systemd/network/pan.network 



 [Match] Name=pan [Network] Address=0.0.0.0/24 DHCPServer=yes IPMasquerade=yes


次に、BlueZにNAPプロファイルを構成させる必要があります。 BlueZ 5.36の通常のユーティリティは実行できないことが判明しました。 作成者が間違っている場合は修正してください：mlang（耳の動かし方を知っています）盲目（時々アクセスと量子）の達人



しかし、彼はブログ投稿とD-Busに必要な呼び出しを行うPythonスクリプトを見つけました 。



便宜上、作成者はSystemdサービスを使用してスクリプトを実行し、依存関係の解決を確認しました。



ファイル/etc/systemd/system/pan.service 



 [Unit] Description=Bluetooth Personal Area Network After=bluetooth.service systemd-networkd.service Requires=systemd-networkd.service PartOf=bluetooth.service [Service] Type=notify ExecStart=/usr/local/sbin/pan [Install] WantedBy=bluetooth.target


ファイル/ usr / local / sbin / pan 



 #!/bin/sh # Ugly hack to work around #787480 iptables -F iptables -t nat -F iptables -t mangle -F iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE exec /usr/local/sbin/bt-pan --systemd --debug server pan


DebianにIPMasquerade =サポートがある場合、2番目のファイルは必要ありません（ ＃787480を参照）。



systemctl daemon-reloadおよびsystemctl restart systemd-networkdコマンドを実行した後、 systemctl start panコマンドでBluetooth PANを起動できます



Systemdを使用したBluetooth PANクライアント



クライアント側もSystemdを使用して簡単に構成できます。



ファイル/etc/systemd/network/pan-client.network 



 [Match] Name=bnep* [Network] DHCP=yes


ファイル/etc/systemd/system/pan@.service 



 [Unit] Description=Bluetooth Personal Area Network client [Service] Type=notify ExecStart=/usr/local/sbin/bt-pan --debug --systemd client %I --wait


設定を再起動した後、次のように指定されたBluetoothアクセスポイントに接続できます。 



 systemctl start pan@00:11:22:33:44:55


コマンドラインのペアリング



もちろん、サーバーとクライアントは、Bluetoothを介してペアリングした後に設定する必要があります。 サーバーで、bluetoothctlを実行し、コマンドを与える必要があります。 



 power on agent on default-agent scan on scan off pair XX:XX:XX:XX:XX:XX trust XX:XX:XX:XX:XX:XX


スキャンを開始した後、必要なデバイスがリストに表示されるまで数秒待ちます。 彼のアドレスを書き留めて、pairコマンド、および必要に応じてtrustコマンドを発行して使用します。



クライアントは同じことをする必要がありますが、trustコマンドは確かに必要ではありません。 サーバーは、ユーザーが手動で確認することなく、NAPプロファイルの接続を受け入れる必要があります。



著者は、これがコマンドの最適なシーケンスであるかどうかはわかりません。 おそらく必要なのは、クライアントとサーバーをペアリングし、サーバーでtrustコマンドを実行することだけですが、彼はまだ試していません。



HID Bluetoothプロファイルを使用する権限



「Malinka」は、有線で点字ディスプレイに接続されたキーボードを認識し、Bluetoothを介してディスプレイ自体によってすでに転送されている必要があります。 同じ方法で、 エージェントの代わりにのみkeyboardOnlyコマンドを実行する必要があり、bluetoothctlはHIDプロファイルを持つデバイスを検出します。



ただし、コマンドラインからBluetoothを構成するのは少し複雑です



著者はすべてを構成できましたが、コマンドラインからBlueZを構成するのは不便であることを理解しています。 最初は、エージェントはPINコードの入力にのみ必要であると考えていましたが、たとえば、HIDプロファイルを有効にするには「agent KeyboardOnly」と入力することが判明しました。 驚くべきことに、Bluetooth PANを起動するには、目的のスクリプトを探してリポジトリを登る必要があります。 彼は、以前のバージョンのBlueZにはこのための既製のパンドツールがあったことを覚えています-BlueZ 5ではどこでそれをしましたか？ 突然、新しい解決策が登場しましたが、著者には知られていませんが、表面に横たわっていますか？



性能



データ転送速度は約120 kbpsで、これで十分です。 1 GHz ARMプロセッサは、コマンドラインインターフェイスとして非常に高速です。 著者は、デバイス上で主にsshとemacsを使用する予定です。



コンソールフォントと画面解像度



Raspberry Pi Zeroのフレームバッファーで使用されるデフォルトの画面解像度はかなり奇妙です。fbsetは、656x416ピクセルであることを報告します（もちろん、モニターは接続されていません）。 8x16のコンソールフォントでは、1行あたり82文字と26行が取得されました。



このモードで40文字の点字ディスプレイを使用するのは不便です。 また、著者は、Unicode文字を点字で表示することを望んでいます。 幸いなことに、Linuxは512文字をサポートし、ほとんどのコンソールフォントには256文字があります。コンソールセットアップでは、2つの256文字フォントを一緒に使用できます。 作成者は、次の行を/ etc / default / console-setupファイルに追加しました。 



 SCREEN_WIDTH=80 SCREEN_HEIGHT=25 FONT="Lat15-Terminus16.psf.gz brl-16x8.psf"


注：brl-16x8.psfフォントを使用可能にするには、console-brailleをインストールする必要があります。



次は？



点字ディスプレイには3.5 mmジャックがありますが、著者はMini-HDMIからオーディオを削除するためのアダプターを認識していません。 著者は、「ラズベリー」に組み込まれたサウンドカードを使用できませんでした（奇妙なことに、翻訳者はZeroには何もないと確信していましたが、GPIOにPWMサウンドを出力する方法があります）。 彼は、USB-OTGハブを使用して外部カードを接続し、点字ディスプレイに組み込まれたスピーカーに音声を出力する予定です。 何らかの理由で、2枚の外部カードが機能しなかったため、別のチップセットで同様のデバイスを探しています。



「ラズベリー」を手動でオフにし、数秒待ってから点字ディスプレイをオフにするのは依然として不便です。 そしてすべては、それが切断されると、コンパートメントのコネクタから電力を取り除くためです。 著者はコンパートメントに小さなバッファバッテリーを入れ、GPIOを介して「ラズベリー」にディスプレイをオフにすることを通知し、作業を開始することを計画しています。 そのようなミニチュアUPS。
