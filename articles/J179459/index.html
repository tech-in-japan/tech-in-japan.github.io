2013年のJAVA + Swing。それだけの価値はありますか？

HabréSwingでは好きではありません。 「Swing」を検索すると、ニュートラルまたはネガティブな参照が得られます。 それらのいくつかを次に示します。

  • 「Javaアプレット（Swingと呼ばれる致命的な病気に発展した）」
  • 「スイングは、控えめに言っても、最適なUIフレームワークではありません」
  • 「スイングはひどかった」


Swingが理想的であるふりはしません。 これは真実ではありません。 しかし、私が直面しなければならなかった長所と短所を説明しようとします。



スイングする理由



私は数年間、断続的にSwingを使用しています。 主に夕方に。 写真をグループで保護するためのビジュアルウォーターマークプログラムを書いています。 私は2011年にJavaバージョンを持っていました。Mac用のポートを作成してインターフェイスをなめたかったのですが、プラットフォームごとに別々のプログラムを書く方法がありませんでした。



2011年の初め、クロスプラットフォーム開発UIライブラリは次の状態にありました。

  • QMLはすべてバグでした。コンポーネントの下にメニューが表示され、デモが失敗し、QtCreatorにサポートがありませんでした。 アクセラレートレンダリングは、昨年秋にQt5でのみ登場しました。
  • Qtは適合しませんでした、なぜなら それは完全に「ネイティブ」コンポーネント上にあり、しばしばどこかでレンダリングを変更する必要がありました。
  • Juceは機能的でバグがなく、クラッシュしませんでした。 リーズナブルな費用がかかり、オープンソースでもコストがかかります。 C ++が怖い。 これは、非常に多くの賢い人々が話す素晴らしい言語です。 私の小さな経験とタスクの大きさを考えると、C ++の複雑さは不利です。 さらに、XcodeはC ++をリファクタリングできないことが判明しました。
  • Adobe Airはマルチスレッドをサポートしていません。
  • Mono + GTKこの組み合わせは私の問題を解決できるように思えます。 その瞬間、GTKでアイドル状態のホットキーを使用した明らかな横棒に驚かされました。 MonoDevelopから判断すると、これまでのところ修正されていません。
  • JavaFXは Mac用ではありませんでした。
  • SWTは Swingよりもはるかに簡単で、一般的に優れています。 最後に見たので、SWTには書きませんでした。 すでに多くの時間が費やされており、最初のバグ（ツールバーのボタンの高さが「浮いている」）の実験を終了しました。


当時、JavaはMac OS Xの一部であり、優れたネイティブルックアンドフィールを備えており、Windows JREの重量はわずか12メガバイトでした。 私は単純に成功を確信していました。 その結果、2か月または3か月の作業の後、Java Swingプログラムの最初のバージョンになりました。



現在までに、記載されている問題はQMLおよびJavaFXで修正されています。 したがって、ステージグラフを使用する準備ができている場合は、それらを試運転してください。



QtはDigiaの傘下に入りました。 iPhoneおよびAndroidのベータリリースは、ライブラリのさらなる開発への希望を与えます。



JavaFXは、今年2月にオープンソースライブラリになりました。 OpenJDK との互換性は JDK 9で計画されています。9kaが発表された時点では不明です。 8番目のバージョンのリリースは2014年初頭に予定されています。



いいね



良いところから始めましょう。 突然、私もSwing-hateersにサインアップしたと思いますか？！



すべてのレンダリングはハードウェアアクセラレーションです。 SwingアプリケーションはGPUでレンダリングされるため、開発者は何も必要としません。 これにより、アプリケーションでアニメーションが可能になります。 アプリケーションがフルスクリーンまたは24インチモニターに展開されている場合を含みます。



MVC Swingは巨大であると批判されています。各コンポーネントは、ビュー、コントローラー、モデルで構成されています。 同時に、これにより、既存のコンポーネントに目的の機能をすばやく追加できます。 すべてが非常に柔軟です。



Javaはマネージコードです。 C ++開発者のみが「利用可能な」バグの可能性を取り除きます。 アクセス違反のリスクは最小限に抑えられます。 これは、他のバグがないことを意味するものではありませんが。 たとえば、メモリリーク。



優れた開発環境。 Eclipse、Intellij IDEA、NetBeans-選択は膨大です。 どこにでもリファクタリング、コードのフォーマット、オートコンプリート、単体テストのサポート、



多くのライブラリ。 ネイティブオブジェクト、文字列、Webを操作するLayoutManagers-すべてをリストすることはできません。 これはプラットフォームとしての巨大な切り札のJavaです。



質問に対する多くの答え。 たとえば、各UIライブラリのStackOverflow に関する質問の割合です 。







StackOverflowに関する約100問ごとに、Swingに関する質問があります。 実際には、これはほとんどの問題がすでに解決されていることを意味します。 ほとんどの場合、あなたが滞在しない問題に一対一で対応します。

ネイティブコードを使用する場合、状況はさらに悪化する可能性があります。 モーダルウィンドウ（たとえば、OpenFileDialogを開く）を使用すると、一部のコンピューターでフリーズが発生する状況に遭遇しました。 Java Native Foundationは、ドキュメントの例に従って使用されています。 そして、StackOverflowの恩恵は私を助けませんでした:-)

すべてのBufferedImageオブジェクトがハードウェアアクセラレーションを使用するわけではありません。 BufferedImage.TYPE_INT_ *のみ。 したがって、TYPE_4BYTE *で動作するJDK7からは、TYPE_3BYTEに一貫性がなくなります。



BufferedImageラスタのデータにアクセスすると、GPUを介した画像の描画が停止します。 これが行われる理由は明らかです。ユーザーがデータを変更し、「完成した変更」方法がなく、それらをビデオメモリにいつ転送するかが明確ではありません。 少なくともそれは論理的です。



Visual Watermarkでは、C ++ライブラリを使用して画像をロードし、結果のピクセルをBufferedImageオブジェクトに変換する必要がありました。 一度に1ピクセルずつ変更するのは非常に遅く、画像ラスタバッファに直接書き込む必要がありました。 ラスターでgetData（）を呼び出すとすぐに、すべての写真の加速が停止しました。 DataBufferIntのコードを調べて、リフレクションを使用してこの問題の解決策を見つけ、小さなヘルパークラスを作成しました。 



 import java.awt.*; import java.awt.image.*; import java.lang.reflect.Field; import sun.awt.image.SunWritableRaster; import sun.java2d.StateTrackableDelegate; // Standard library prevents image acceleration once getData() method is called // This class provides a workaround to modify data quickly and still get hw-accel graphics public class AcceleratedImage { // Returns data object not preventing hardware image acceleration public static int[] getDataBuffer(DataBufferInt dataBuffer) { try { Field field = DataBufferInt.class.getDeclaredField("data"); field.setAccessible(true); int[] data = (int[])field.get(dataBuffer); return data; } catch (Exception e) { return null; } } // Marks the buffer dirty. You should call this method after changing the data buffer public static void markDirty(DataBufferInt dataBuffer) { try { Field field = DataBuffer.class.getDeclaredField("theTrackable"); field.setAccessible(true); StateTrackableDelegate theTrackable = (StateTrackableDelegate)field.get(dataBuffer); theTrackable.markDirty(); } catch (Exception e) { } } // Checks whether current image is in acceleratable state public static boolean isAcceleratableImage(BufferedImage img) { try { Field field = DataBuffer.class.getDeclaredField("theTrackable"); field.setAccessible(true); StateTrackableDelegate trackable = (StateTrackableDelegate)field.get(img.getRaster().getDataBuffer()); if (trackable.getState() == sun.java2d.StateTrackable.State.UNTRACKABLE) return false; field = SunWritableRaster.class.getDeclaredField("theTrackable"); field.setAccessible(true); trackable = (StateTrackableDelegate)field.get(img.getRaster()); return trackable.getState() != sun.java2d.StateTrackable.State.UNTRACKABLE; } catch(Exception e) { return false; } } public static BufferedImage convertToAcceleratedImage(Graphics _g, BufferedImage img) { if(!(_g instanceof Graphics2D)) return img; // We cannot obtain required information from Graphics object Graphics2D g = (Graphics2D)_g; GraphicsConfiguration gc = g.getDeviceConfiguration(); if (img.getColorModel().equals(gc.getColorModel()) && isAcceleratableImage(img)) return img; BufferedImage tmp = gc.createCompatibleImage(img.getWidth(), img.getHeight(), img.getTransparency()); Graphics2D tmpGraphics = tmp.createGraphics(); tmpGraphics.drawImage(img, 0, 0, null); tmpGraphics.dispose(); img.flush(); return tmp; } }
