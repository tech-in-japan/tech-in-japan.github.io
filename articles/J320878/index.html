みなさんこんにちは。 この記事では、IoTデバイスとクラウドアプリケーション間のデータ同期に関連する1つの興味深い問題を解決した経験を共有したいと思います。 まず、プロジェクトの主なアイデアと目標について説明し、次にその技術的な側面と実装について詳しく説明します。ContikiOS、データベース、プロトコル、および同様の側面についてです。 結論として、システムの構築に使用されるテクノロジーを簡単にリストします。



プロジェクトについて簡単に



まず、プロジェクトの主なアイデアについて話しましょう。 以下は、完成したシステムの動作原理の概略図です。







クラウドサービスを介して、または（Wi-Fiを介して）IoTデバイスに直接接続するユーザーがいます。 また、インターネット上のどこかにクラウドアプリケーションサーバーがあります。 AWSやAzureインスタンス、専用サーバーなど、何でもクラウドとして機能します。 アプリケーションサーバーとIoTデバイス間でデータを交換するには、何らかのプロトコルを使用して接続を確立します。 IoTデバイスは何らかの形で相互に接続されています（たとえば、イーサネットまたはWi-Fiを介して）。 さらに、テレメトリーデータ（照度や温度のインジケーターなど）を生成するIoTデバイスの別のグループがあります。



合計で、100を超えるデバイス、さらには1000を超えるデバイスを入力できます。 私の主なタスクは、クラウドとこれらのIoTデバイス間のデータ交換を提供することでした。 先に進む前に、システムに課せられた要件に言及する価値があります。



  • IoTデバイス間でデータを同期する必要があります。
  • IoTデバイスからデータを収集する必要があります。
  • IoTデバイスとクラウドの間でデータを同期する必要があります。


技術的な実装







ここではすべてが非常に簡単です。ユーザーはHTTP（S）、WebSocket、または同様のプロトコルを介してアプリケーションサーバーに接続します。 読者向けの小さなタスク：アプリケーションサーバーとIoTデバイス間の接続に使用できるものは何だと思いますか？







MQTTについて考えた場合、間違いなく正しいです！ HTTP（S）を選択した人と同様に。 実際、どのプロトコルも適切です-あなたの好みに合わせて選択してください！ 私の選択は失敗しました-ドラムロール-非同期複製！ データベースのレプリケーションは正常です。







レプリケーションが必要な理由を尋ねられるかもしれません。 答えは簡単です。複製を使用してデータを同期するので、クラウドとIoTデバイスを含むあらゆる場所で同じバージョンのデータベースをサポートできます。 ただし、レプリケーションの実装は非常に困難です。 複製が必要な場合は、それをサポートするデータベースを入手してください。なぜなら、私は繰り返しますが、複製はデータベースに本来備わっているからです。



ここで、プロジェクトで作業する際に検討したデータベースについて、 SQLite 、 Redis 、 MySQL 、 PostgreSQL 、 Tarantoolのいくつかの言葉を言いたいと思います。



それらの特性を比較し、MySQLとPostgreSQLを除くいくつかの部分をIoTデバイス上で実行しようとしました。 以下に、その由来を説明します。



SQLiteは、IoTデバイスにデータを直接保存するための優れたソリューションであることは間違いありませんが、レプリケーションがなく、異なるプロセスからの同時アクセスをサポートしていません。

Redisはマスターマスタレプリケーションをサポートしていないため、双方向レプリケーションが必要なため、問題を解決できません。



MySQLとPostgreSQLはIoTデバイスには重すぎるので、それらをインストールしようとはしませんでした。 ただし、これを行うことに決めた場合は、コメントで自由に経験を共有してください。



私のリストの最後の1つはTarantoolデータベースでした。 私はTarantoolプロジェクトのコミッターであるとすぐに言わなければならないので、プロジェクト自体とそれをうまく開発している人々を知っています。 さらに、Tarantoolにはマスターマスタレプリケーションがあります。 全体として、私にとっては間違いなく最良の選択肢でした。 プロジェクトで別のデータベースを使用できます。 私が伝えようとしている主なアイデアは、IoTデバイスがマスターマスタレプリケーションでデータベースを使用してデータを交換できるということです。



これまで、私は表面的にプロジェクトを知っているだけでした。 それでは、技術的な側面を少し掘り下げてみましょう。



Tarantoolの使用中に発生した問題から始めます。 まず、TarantoolはARMv7アーキテクチャで実行されませんでした。 第二に、Tarantoolは32ビット環境で起動しなかったため、状況が悪化しただけです。 最終的に、これらの問題を解決することができました。 以下は、これを助けてくれた開発ルールです。



  1. CMakeのツールチェーンファイルを使用します。 それ以外の場合、あなたは、私のように、CMakeファイルの修正に多くの時間を費やします。
  2. サイズが指定されていない符号なしの型またはその他の型を使用しないでください。 libcにはuint32_tなどの特別なタイプがあります。 そうしないと、未定義の動作が発生する可能性があります。 このルールは、C / C ++にのみ適用されます。
  3. 自動テストを移植します。


自動テストはIoTデバイスで実行できることが期待されています。 そうでない場合、多くのデバッグ時間を無駄にするリスクがあります。



そのため、マスターとマスターのレプリケーションを備えた作業データベースがあります。 いいね！ 次のステップは、 6LoWPANを使用して、このデータベースがインストールされているデバイスを接続することです。 6LoWPANを介して互いに接続された多くのIoTデバイスのネットワークがあり、そこからすべてのテレメトリデータを収集する必要があることを思い出させてください。





完成したシステムの簡単なスキーム



センサーを備えたデバイスは、電波を介してテレメトリーデータを送信します。 この標準は6LoWPAN（低電力ワイヤレスパーソナルエリアネットワーク上のIPv6）と呼ばれます。 プロジェクトでLoRaWANを使用しなかったことに注意してください。 このテクノロジーの将来の使用を見つけるかもしれませんが、この記事では6LoWPANに焦点を当てます。 したがって、遠隔測定データの収集には、システムの重要な部分であるゲートウェイを使用します。 ゲートウェイは、電波を介して送信されるデータを収集するためのWANアンテナを備えたMIPSデバイス（MIPSはプロセッサのファミリーです）です。 さらに、6LBRアプリケーションがゲートウェイにインストールされ、受信データをIPv6パケットに変換します。

アプリケーションサーバーへの接続は、Tarantool master-masterが提供するレプリケーションを通じて確立されます。 これから2つの有用なプロパティが続きます。



  1. アプリケーションサーバーがデータを変更すると、この更新されたデータはネットワーク上のすべてのIoTデバイスに配信されます。
  2. IoTデバイスがデータを変更すると、この更新されたデータがアプリケーションサーバーに配信されます。


これらのプロパティが私の問題の解決策です。



マスターマスタレプリケーションを使用してIoTデバイスを接続することもできます。 したがって、デバイスとクラウドは、すべてのデータを同期するために使用できるクラスターに結合されます。 すべてのIoTデバイスとクラウドは、それらの間の接続が失われない限り、ほとんどの時間同期されます。 接続が復元されるとすぐに、すべてのデータが再び同期されます。 素晴らしい！



6LBRアプリケーションがインストールされたゲートウェイを使用すると、IoTデバイスと他のIoTデバイス間でデータを交換できます。 各メッセージをMQTTメッセージにラップし、UARTチャネルに送信します。



MQTTブローカーがインストールされたIoTデバイス#Nは、UARTチャネルからこれらのメッセージを読み取ります。 MQTTブローカーは、MQTT接続を介してメッセージをTarantoolにリダイレクトします。 Tarantoolはそれらを読み取り、Tarantoolアプリケーションサーバーは各メッセージに対していくつかのコードを実行します。



#N IoTデバイスは、Tarantoolマスター/マスターレプリケーションを介して他のすべてのデバイスに接続されます。 同じレプリケーションを使用して、すべてのデバイスをクラウドに接続します。
