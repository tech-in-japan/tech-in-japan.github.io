NVIDIA Parallel Nsight 2.0デバッガーのCUDAレビュー

並列コードのデバッグは退屈で費用のかかるプロセスです。 並列アプリケーションの動作が非決定的であるため、並列化エラーを見つけるのは困難です。 さらに、エラーが検出された場合、再び再現することはしばしば困難です。 コードを変更した後、エラーが修正され、マスクされていないことを確認することは困難です。 ほとんどの場合、並列プログラムのエラーはheisenbagsです。 時には、並列プログラム用の最も便利で機能的なデバッグツールが緊急に必要だと感じます。

そのため、1年ほど前に、NVIDIAはNVIDIA Parallel Nsightと呼ばれるCUDAで記述された並列プログラムをデバッグするために、Microsoft Visual Studio 2008 sp1および2010に統合されたツールのパッケージをリリースしました。 これについて、 やがてXaocCPS は habrosociety について書きました。 それ以来、この製品はより高度になり、完全に無料になりました。 これまでの最新バージョン2.0。 可能な構成、インストール、構成、およびNVIDIA Parallel Nsightの主な機能を検討してみましょう。

Parallel Nsightをインストールする



並列コードをデバッグできるようにするには、1台のマシンで2つのCUDA互換GPUを使用した構成で十分です（もちろん、2台のマシンを使用した構成についてお話しする方がはるかに興味深いでしょうが、残念ながら、現在このような構成を組み立てる方法はありません）。

オペレーティングシステムとして、Windows 7エンタープライズエディションx64を使用しました。 次に、2008 sp1以上のMVSが必要です。



NVIDIA Webサイトには、必要なディストリビューションが含まれています。 以下が必要です 。

  • WinVistaおよびWin7の開発者ドライバー
  • CUDAツールキット
  • CUDAコンピューティングSDK
  • Parallel Nsight 2.0。


同じ順序でディストリビューションをインストールします。 スタジオで新しいプロジェクトのウィザードを呼び出すとき、新しいセクション「NVIDIA」（テンプレートはパッケージ「CUDA Toolkit」に含まれています）を追加し、プロジェクトタイプ「NVIDIA CUDA 4.0」を選択してプロジェクトを作成します。 すべてのディストリビューションのインストールが正しい場合、結果のワールドワードをコンパイルして実行できます。

すべて大丈夫ですか？ 次に、Parallel Nsightデバッガーを直接扱います。 マシンはサーバーとコンピューターの両方であるため、最初にホストコンポーネント「Nsight Monitor」を実行する必要があります。 コードを開いて、計算カーネルプロシージャのどこかにブレークポイントを設定し、nsightパネルの特別なボタンでプロジェクトを開始します。 いくつかの点に注意してください。

  1. プロジェクトは事前にビルドする必要があります（nsightアプリケーションの開始ボタンはコンパイルされません）。
  2. プログラムがnsightデバッグモードで実行されている場合、計算コアの外部に設定されたすべてのブレークポイントは無視されます。 これは逆の順序で行われます。プログラムが通常モードでデバッグされる場合、通常のコードの反対側のブレークポイントのみが考慮されます。
  3. Sevenでnsightデバッガーを初めて実行すると、少なくとも2つの問題が発生する可能性があります。WPFアクセラレーターとの非互換性とWindows Aeroです。 それらはオフにする必要があります（最初のレジストリはレジストリに追加することでオフになります：

    Windows Registry Editor Version 5.00

    [HKEY_CURRENT_USER\Software\Microsoft\Avalon.Graphics]"DisableHWAcceleration"=dword:00000001





    2つ目はコントロールパネルから無効にします）またはnsight自体で警告チェックを無効にすることができます：スタジオセット：Nsight->オプション->ローカルデバッグチェックを「True」にオーバーライドしますが、これには問題がたくさんあります。 たとえば、計算のデバイス、デスクトップが描画されるビデオカード、およびnsightデバッグを実行するデバイスとしてコードで指定すると、永久にフリーズします。 Parallel NsightとWPF / Aeroの非互換性が何を意味するのかは明確ではありません。「ローカルデバッグチェックを上書きする」オプションをオンにしてデバッグする際、デバッガー側とメカニズム自体の両方からこれらのメカニズムに問題はなかったためです。

これで、通常のアプリケーションのデバッグと同様に、利用可能な制御値を確認できます。 本格的なウォッチャーを使用すると、アレイを表示できます。 上記のスクリーンショットでは、「A」変数はMatrix3タイプです。

typedef struct {

表示可能な「要素」配列の要素数は、Parallel Nsightデバッガー設定の「最大配列要素」パラメーターによって決まります。

インデックス値blockIdxおよびthreadIdxからわかるように、デバッガーは最初のグリッドブロックの最初のスレッドにあります。 問題が発生します：目的のストリームに移動する方法は？ nsightツールでは、「Nsight Cuda Device Summary」というウィンドウを使用できます。これは、停止したストリームの近くでワープ間を移動できるインターフェイスです。 近傍のサイズは、ビデオチップのハードウェア機能によって決まります。 そのため、停止時にGeForce 200で計算する場合、4ワープの2ブロックが利用可能でした。



同様に、GeForce GTX 460の場合：



31ユニットが利用可能です。 ワープ内の特定のスレッドに移動するには、「Cuda Debug Focus」ウィンドウを使用する必要があります（このインターフェイスでは、ブロック間を移動することもできます）。

繰り返しますが、「最初のストリームの近くに入らないストリームに入る方法は？」です。 このために、条件付きブレークポイントが使用されます。 条件の構文は次のとおりです。

@blockIdx(x,y,z) && @threadIdx(x,y,z)





デバッガは、指定されたストリームで停止します。これに対して、ワープ近傍が使用可能になります。
