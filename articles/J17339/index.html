オレグ、あなた自身について、あなたが何をしているかについて少し教えてください。

私は25歳で、リガ（ラトビア）で生まれ育ちました。 彼は電気通信の学位を取得して卒業しました。

専門分野-放送およびテレフォニーオートメーションシステムのソフトウェアエンジニア。



以前にオープンソースプロジェクトに参加しましたか？

直接いいえ。 wineプロジェクトにはいくつかのパッチが受け入れられています。 暇なときにGnome / Mozilla / wineを選びます。時間が経つとパッチを送信できるようになるためです。メインブランチでコードを受け入れるプロセスであり、最も時間がかかり、常に注意が必要です。

ああ、完全に忘れてしまった、3Dゲームに関連する自分のオープンプロジェクト、たとえば、かつて多くのWin32 wolf3d newwolf.sf.netに慣れ親しんでいた古いゲームのポート



Money for Bug Fixesプログラムについてどのように知りましたか？

サイトlinux.org.ruでは、毎日アクセスしています。



パッチの開発とテストにはどのくらい時間がかかりましたか？

純開発時間を計算するのは困難です。 私は約1年前にこの問題に初めて興味を持ちました。

Linuxで動作するようにクロールした後、システムで自分に合わないもの（大きなプラスオープンソース）を修正する方法を検討し始めました。 それから、プログラムについて知ったとき、私は自分の変更を掘り下げて受け入れ可能な形にし、プロジェクトメンバーのコメントを修正しました-約1か月かかりました。



パッチの作成時に使用されたツールは何ですか？

コンパイラはもちろんgcc（C / C ++言語）です。もちろん、これはプロジェクトによって決定されます。 IDE-Eclipse。geditを簡単かつ迅速に編集できます。 FirefoxはBugzillaとGMailの操作にも使用され、GTK dev.manualを読みました。



Mozilla開発者と連絡を取りましたか？

Bugzillaの助けを借りて、プロジェクトの精神を理解し、現在の作品の主な方向性を知るために、しばらくメーリングリストを読みました。



他のバグの修正に参加しようとしている他のプログラマーに何を言いたい/アドバイスしますか？

コードをもう少し操作して、プログラムのアセンブルされたバージョンを（変更を加えて）毎日使用する-これが最良のテストです。 パッチを送信するとき、プロジェクトメンバーのコメントに耳を傾け、パッチレビューを行っている人々、それをサポートするのは彼らのコードであり、彼らはあなたよりもそれをよく知っています。
