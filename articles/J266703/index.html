サーバーレスアーキテクチャ



「サーバーレス」という用語は、明示的なアーキテクチャが不要であることを意味します。つまり、サーバーなしで、サーバーに展開せずに、プログラムをインストールせずに実行できます。 マネージドクラウドサービスとラップトップでのみ動作します。 以下の図は、主要なコンポーネントとその接続を示しています。マシンインターフェイスとしてのラムダ関数と、それに直接接続するモバイルアプリケーション、およびAmazon S3にある静的サイトにHTTPエンドポイントを提供するAmazon API Gateway。

ステージ1：画像処理用のサービスを作成します



プロセス全体を簡素化するために、Lambdaテクノロジーnodejsに組み込まれているImageMagickライブラリを使用します。 ただし、これは必須ではありません-独自のライブラリが必要な場合は、 JavaScriptまたはネイティブライブラリをロードしたり、 Pythonを実行したり、コマンドラインの実行可能ファイルにコードをラップしたりできます 。 次の例はnodejsに実装されていますが、 Java 、 Clojure 、 Scala、またはAWS Lambdaの別のjvm言語を使用してこのようなサービスを作成することもできます。



以下のコードは、ImageMagickの一種の「ハローワールド」と見なすことができます-コマンドの基本構造（このコマンドはスイッチオペレーター）に精通し、バラの埋め込み画像を抽出して返すことができます。 結果のコーディングとは別に、他のすべてをJSONで書くこともできます。 



var im = require("imagemagick"); var fs = require("fs"); exports.handler = function(event, context) { if (event.operation) console.log("Operation " + event.operation + " requested"); switch (event.operation) { case 'ping': context.succeed('pong'); return; case 'getSample': event.customArgs = ["rose:", "/tmp/rose.png"]; im.convert(event.customArgs, function(err, output) { if (err) context.fail(err); else { var resultImgBase64 = new Buffer(fs.readFileSync("/tmp/rose.png")).toString('base64'); try {fs.unlinkSync("/tmp/rose.png");} catch (e) {} // discard context.succeed(resultImgBase64); } }); break; //     default: var error = new Error('Unrecognized operation "' + event.operation + '"'); context.fail(error); return; } };




まず、サービスが機能することを確認しましょう。 これを行うには、次のJSONをAWS Lambdaコンソールのテストウィンドウに送信します。 



 { "operation": "ping" }




必須の「ポン」応答を受信する必要があります。 次に、実際にImageMagick呼び出しに進み、そのようなJSONを送信します。 



 { "operation": "getSample" }




このクエリは、base64でエンコードされたPNG PNGイメージの文字列表現を取得します：“” iVBORw0KGg ... Jggg ==”。 これらが単なるランダム文字ではないことを確認するには、それらを切り取り、Base64を画像に変換する便利なデコーダー（codebeautify.org/base64-to-image-converterなど）に（二重引用符なしで）貼り付けます。 バラの美しい画像を取得する必要があります。







サンプル画像（赤いバラ）



次に、画像処理サービスを追加して、nodejsラッパーの残りの使用を準備します。 さまざまな操作を提供します。



  • ping：サービスの可用性を確認します。
  • getDimensions：イメージの高さと幅を取得するために識別操作を呼び出す短縮版。
  • 識別：画像のメタデータを取得します。
  • サイズ変更：サイズ変更のための補助的な手順（「内部」変換呼び出し）
  • thumbnail：サイズ変更の同義語。
  • 変換：「ユニバーサル」手順-メディア形式の変換、変換の適用、サイズの再計算などが可能
  • getSample：サンプル画像を取得します。 この操作は「hello world」に対応します




ここのコードのほとんどは非常に簡単です。 このコードは、nodejsに実装されたImageMagickプロシージャをラップし、それらの一部はJSONを受け入れます（この場合、Lambdaによって送信されたイベントはクリアされ、リダイレクトされます）が、文字列の配列として渡されるコマンドライン引数（いわゆる「カスタム」）を受け入れます。 ImageMagickを以前に使用したことがない場合、この機能の1つの側面は明らかではないかもしれません。つまり、コマンドラインのラッパーとして機能し、ファイル名にセマンティクスがあります。 競合する2つのニーズがあります。第1に、クライアントはセマンティクス（たとえば、JPEGに対するPNGなどの出力画像形式）を転送する必要があり、第2に、サービスの作成者はディスク上の一時ストレージを保存する場所を決定する必要があるため、詳細の漏洩を許可しません実装。 これらの両方の問題を同時に解決するために、JSONスキームで2つの引数「inputExtension」と「outputExtension」を定義し、クライアント部分（ファイル拡張子）とサーバー部分（ディレクトリ名およびベース名）を組み合わせて、ファイルの実際の場所を作成します。 完成したコードは、次の画像処理図面で確認できます（そして使用できます！）。



ここで実行できるテストは多数ありますが（以下で実行します）、操作性を簡単にテストするために、ローズの同じ画像を再度抽出し、ネガティブフィルターを使用して返します（色反転を実行します）。 Lambdaコンソールで同様のJSONファイルを使用できます。base64Imageフィールドの内容を画像に一致する文字に置き換えるだけです（このシーケンスは非常に長くなります）。 

 { "operation": "convert", "customArgs": [ "-negate" ], "outputExtension": "png", "base64Image": "...fill this in with the rose sample image, base64-encoded..." }

データスキームは、クライアントとサーバー間で転送する必要があるすべてのオブジェクトを定義します。 「ラムダ主義」はありません。 すべてのオブジェクトは、特別なライブラリやフレームワークのない通常のPOJO（Plain Old Java Object）です。 基になるイベントを定義し、それを展開して操作の構造を反映します。 上記の画像処理サービスを定義およびテストするときに使用したJSONの「javification」があると想定できます。 Javaサーバーも作成する場合は、通常、これらのファイルを共通イベント構造の定義の一部として共有します。 この例では、これらのPOJOはサーバー側のJSONに変換されます。

サーバーレスWebアプリケーションパート1：APIエンドポイント



これで、モバイルアプリケーションと、そのマシンインターフェイスとして機能するグローバルに展開された画像処理サービスができました。 デバイスではなくブラウザで作業することを好む同志のために、サーバーレスWebアプリケーションの作成に移りましょう。 これは2段階で行います。 まず、画像処理サービスのAPIエンドポイントを作成します。 次に、次のセクションで、Amazon S3を使用してサイト自体を追加します。



AWS Lambdaを使用すると、コードをサービスに簡単に変換できます。これは、Webサービスクライアントインターフェイスがここで既に「組み込まれている」ためです。 ただし、これにはクライアント（前のセクションで作成したモバイルクライアントなど）がAWSから提供された認証情報でリクエストに署名する必要があります。 このタスクは、Androidアプリケーションで使用されるAmazon Cognito認証クライアントによって解決されますが、ウェブサイトを通じて画像処理サービスを共有したい場合はどうでしょうか？



これを行うには、別のサーバーであるAmazon API Gatewayを使用します 。 このサービスを使用すると、インフラストラクチャを必要とせずにAPIを定義できます。APIはAWSによって完全に管理されます。 APIゲートウェイを使用して、画像処理サービスで使用されるURLを作成します。これにより、Webユーザーがその機能のサブセットにアクセスできます。 Amazon API Gatewayは、さまざまなAPIへのアクセスを制御するさまざまな方法を提供します：API呼び出しはAWS認証情報で署名するか、OAuthトークンを使用してトークンヘッダーを検証にリダイレクトするだけで、APIキーを使用できます（セキュリティで保護されたアクセスが必要な場合はお勧めしません）またはAPIを完全に作成しますここに示すように、一般公開されています。



ゲートウェイAPIは、多様なアクセスモデルに加えて、この記事の範囲を超えた多くの機能も提供します。 それらの一部は組み込み（DDOS攻撃に対する保護など）で、その他はキャッシュなどにより、人気のあるイメージを繰り返し抽出する際の遅延とコストをさらに削減できます。 API Gatewayでは、クライアントと（マイクロ）サービス間の間接レベルを実装することにより、バージョン管理とプライマリデータの配置（ステージング）に別々の手順を使用して、それらを独立して開発することもできます。 それまでの間、主な問題の解決に焦点を当てます。画像処理サービスをAPIとして提供することです。



それでは、APIを作成しましょう。 AWSコンソールで、[API Gateway]をクリックし、[新しいAPI]を選択して、APIに名前を付けるか、説明を追加します。 「ImageAPI」という名前を選択しました。







次に、新しいAPIのリソースを作成し（「ImageProcessingService」と呼びます）、その中にPOSTメソッドを作成します。 統合のタイプとして「Lambda関数」を選択し、画像処理のサービスとして使用するLambda関数の名前を入力します。 「メソッド要求」構成で、許可タイプを「なし」に設定します（つまり、パブリックエンドポイントになります）。 以上です。







統合をテストするには、「テスト」ボタンをクリックします。







次に、テストペイロード{「操作」：「ping」}を指定します。 APIをLambda関数に正常に関連付けたことを示す、予想される「ポン」の結果を取得する必要があります。



注：以下では、より完全な（より深い）テストを行いますが、通常、pingなどの単純な操作に関連付けられたトップレベルリソースとしてGETメソッドをAPIに追加すると便利です。 これにより、どのブラウザーからでも、APIがLambda関数に正しく接続されていることをすばやく確認できます。 デモアプリケーション（および一般的に）では、これは必須ではありませんが、この手法を好む可能性は十分にあります。



次に来るもの（静的S3コンテンツ）については、CORSをアクティブにする必要もあります。 簡単ですが、いくつかのステップで行われます。 API Gatewayチームはこのプロセスを引き続き簡素化するため、ここでの指示をすぐに繰り返さないようにするために（まもなく期限切れになる可能性があります）、 ドキュメントを参照します。
