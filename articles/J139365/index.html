USE：どうだった

試験については、多くのコピーが壊れています。 しかし、同時に、Habréでは主にコミックまたは不正なタスクの分析が行われ、管理者の目を通して内部から 1回だけ見ていたため、関心が高まりましたが、これは驚くことではありません。 私たちの会社はこのプロジェクトに長い間参加しているので、私たちは何か言いたいことがあります。 ABBYYが参加しました：



•ABBYY FormReader Developer Edition製品（現在はABBYY FlexiCapture Engineと呼ばれる）を使用して試験結果を処理するためのアーキテクチャを設計する際、

•地域の専門家のトレーニングにおいて、

•実装中、

•テクニカルサポート。



そのため、私たちは試験を内側から見続けています。



物語


USEの歴史は1999年に始まり、 戦略研究センターがロシアで統一国家試験を開催する可能性を分析するタスクを設定しました。 主な要件は、試験が大規模で、独立した、客観的で、比較可能になることでした。



ABBYYロシアのリードコーポレートプロジェクトマネージャーであるAlexey Kalininは、計画は非常に有能であると述べています。 しかし、ここでもさまざまな問題がありました（それらだけでなく）以下で読むことができます。



2001年、5つの地域と3万人の卒業生が試験に参加しました。 当時、統一国家試験はまだ統一国家試験ではありませんでしたが、実際には、連邦国家試験センター（FTZ）が実施する「集中型試験」です。統一国家試験では、ハードウェアとソフトウェアがまだ購入されていないためです。 トリッキーな計画だけがあります:)



一般に、スキームは最初は非常に単純に見えました。オペレーターはスキャンステーションでスキャナーを監視し、スキャンは認識ステーションに送信され、そこで何が起こるかを推測します。 この不可解な自動プロセスの終了後、結果は検証ステーションに送信され、そこでオペレータは認識結果をオリジナルと（もちろん電子形式で）比較する機会があります。



2002年には、すでに16の地域があり、30万人が参加しました。 そして、それはすでに実際の試験でした。 自動処理に役立つフォームは正確に自動的に処理されましたが、問題がないわけではありません。スキャン–認識–検証チェーンは、検証者が気を散らしたときにチェーン全体の生産性が低下するように設計されました。 なぜ彼は気を散らされるべきですか？ 事実、フォームの記入はかなり厳密に形式化されています。 したがって、「チャーターに準拠していない」というマークが見つかった場合、たとえば、フォームのいずれかのバーコードが記入された場合（生徒が退屈した場合、何ができるか）、検証者は問題を解決する専門家を引き付ける義務があります。 さらに、そのような行動はしばしば委員会の収集と行為の完了を伴う可能性があります。 同じ2002年には、フォームの2番目の部分で別の問題が発生しますが、これは専門家だけが解決できます。 それは、作業をチェックする専門家の間で作業が自動的に分配されず、その結果、混乱が生じたという事実にありました。



2003年には、ほとんどの卒業生が1つの試験に参加しました。 問題を解決するために、エキスパート間の第2部のタスクの自動配布のメカニズムが実装され（「C」フォームの回答がある領域が自動的に印刷される）、「シニア検証ステーション」スキームにも導入されました。 以前は正規の検証者を「足で歩く」必要があったのはまさにそれらの専門家でした。 現在、問題が発生した場合、通常の検証者はタスクをシニアにリダイレクトするだけで、「コンベヤ」の速度は低下しませんでした。



2004.学生の50〜80％がグループCからの質問を記入し始めなかったため、突然、大量の空白のドキュメントが無駄に印刷されていることが明らかになりました。 ただし、問題があります。ルールに従って、試験の最後に空白のページに必ず「Z」の文字を消し、後で何かを追加できないようにします。 したがって、システムは、このようにして取り消し線で表示されたページを認識するように教えられ、それらは空として定義され始めました。 印刷物の数が大幅に減少しました。



森林が救われ、コンベヤーが確立されたように思えます-これ以上何が必要でしょうか？



2007年までに、検証者の側にはいくつかの過失がありました。彼らはプロジェクトに1年以上参加しており、慣れてリラックスしました。 問題が非常に深刻であるため、人々を通常の状態-極端なマインドフルネスの状態に戻すように刺激する必要があることが明らかになりました-子供の未来は文字通り危機にatしています。 次に、エラーの確率を統計的に推定するメカニズムがシステムに追加されました。 フィールドでまれな値が検出された場合、フォームは追加の検証検証のためにステーションに行きました。 オペレータは、検証者がすべてを正しくチェックし、さらにフォームを世界に公開することを確認するか、検証者に「あたた」をしてドキュメントを検証段階に戻す必要がありました。 このような制御の存在は、検証品質のレベルを返しました。

そのため、ドキュメント処理スキームは最終的な形式を見つけました





2008年、試験は全国を「カバー」します。 到達困難なエリアに到達するために、USEフォームは事前に送信され、タスクは暗号化された形式で送信されます。 パスワードは試験の1時間前に合格します。 試験後、記入済みのフォームはスキャナーでスキャンされ、地域の情報処理センターに送信されます。 このプロセスは、オペレーターの資格が低くなるように設計されています（「体育教師」）。 コンピュータに実質的に不慣れな人が仕事をこなすことができます。



ツール


プロジェクトのこの期間はツールに自然に影響を与えました。VBで始まり、MS SQL 2000で始まり、C＃、MS SQL 2008で続きました。C++は複雑な数学演算にも使用されました。 「OS」：Windows XP、2000。「ジャンク」と言いますか？ 一般に、今でも、彼らが非常に長い歳のとき、最新のWindows 7ですべてをドラッグする前に、非常に深く考える必要があります。



トレーニング


技術専門家のトレーニングは、ABBYYの参加を得て開催されました。 各地域から2人が3日間のコースに参加し、そこでインフラストラクチャのセットアップを学びました。また、各ステーションのオペレーターの役割を試して、問題が発生したかどうかを質問しました。 これらのスペシャリストは、アプリケーション配布キット、自動調整スクリプト、および各職場向けの段階的な指示を残しました。 この形式のトレーニングは非常に効果的であることが実践により示されているため、今日でもその状態は続いています。



サポート


テクニカルサポートでは、標準ツール（電子メール、ヘルプデスク）に加えて、試験の組織に参加している従業員が互いに相談できるクローズドフォーラムが使用されました。 フォーラムは非常に有用であることが判明しました。これは、試験期間中に質問の大半が発生するためです。 5日以内に、質問が間に合い、発生し、解決されるはずです。



どうした


データによると、2006年に満足した上訴のうち、0.3％が「技術的な理由」で満足しました。 それはたくさんですか、それとも少しですか？ ここでは 、卒業生の2％未満が魅力的であると言われていますが、彼らは彼らの約3分の1を満たしています。 合計の約0.6％。 つまり 技術的なエラーは、作品の0.0018％で発見されています。
