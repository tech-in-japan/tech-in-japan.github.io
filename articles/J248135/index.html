Windowsの高DPI

Windowsは、Vista以降、アプリケーションを高ピクセル密度（ドット/インチ、DPI）のモニターに適合させるための2つのメカニズムを提供します。拡大されたシステムフォントと本格的なウィンドウです。 残念ながら、アプリケーションの一部をいずれかのモードで動作させようとする試みは、開発者の怠慢とMicrosoftによる不適切な決定の組み合わせにより失敗する場合があります。



このページは、ユーザーが高いDPI値を設定するときに起こりうる問題を理解して修正するのに役立つことを目的としています。 Windows Store（「Metro」、「Modern UI」）アプリケーションではなく、従来のWindowsアプリケーションのみを考慮することに注意してください。 後者は、独自のスケーリングメカニズムを提供する新しいWinRT APIを使用します。

スケーリング方法



従来、ネイティブのWindowsデスクトップアプリケーションは2つの表示メカニズムを使用します。

  • ディスプレイにアクセスするためのグラフィカルデバイスインターフェイス（GDI）関数。 通常、モニターのサイズとピクセル密度に関係なく、GDI座標は画面のピクセルで直接測定されます。

  • また、Windowsシステムフォントを使用したテキスト出力。 これはオプションですが、ほとんどのアプリケーションはほとんどのグラフィカルユーザーインターフェイス（GUI）でシステムフォントを使用します。



当初、ほとんどのモニターのピクセル密度は約96 dpiでした。 したがって、この機能を使用するGUIは、どのシステムでもほとんど同じように見えました。 しかし、ピクセル密度が増加するとすぐに、そのようなアプリケーションのGUI要素はセンチメートルまたはインチ（実数、モニターに接続されたルーラーを使用して測定されます）の観点から削減されます。 小さなテキストやその他の小さな詳細は、見づらくなってきています。



この状況を改善するために、MicrosoftはWindowsで何らかのスケーリング方法を構築するのが良いと判断しました。 ユーザーが標準の96 dpiよりも高い値でDPIを設定する場合、以下に説明する2つの方法のいずれか（Windows XPまたはVista）が使用されます。 どちらの方法も、画像要素のサイズを拡大しようとします。



Windows XPスタイルのスケーリング



ご想像のとおり、これらの方法の最初のものはWindows XPで登場しました。 実際、この方法は、グラフィカルインターフェイス自体を使用してアプリケーションをスケーリングする方法ではありません。 より高いDPI設定では、システムフォントとシステムのユーザーインターフェイスの一部の要素のみがスケーリングされます（Windows XPのスタイルでは「スケーリング方法ではありません」と呼びます） 。



他のすべてのアプリケーション要素は、1：1スケールで表示されます。 外観の唯一の違いは、システム関数を使用して表示されるテキストと一部のGUI要素が突然大きくなることです。 たとえば、ボタン上のテキスト。 これは明らかな問題を引き起こしますが、これについては後で説明します。



Windows VistaスタイルのスケーリングまたはDPI仮想化



Windows Vistaでは、ユーザーを混乱させるために、明確な名前のない「ディスプレイスケーリング」という奇妙な名前の2番目のオプションが導入されました。 よりわかりやすい名前-DPI仮想化方式を使用します。 この方法を有効にしても、WindowsはWindows XPスタイルのスケーリングを実行します。 前と同じように、すべてのシステムフォントとシステムインターフェイスの一部の要素のサイズは増加しています。



違いは、高いDPI値を正しく使用できるアプリケーションがこれをWindowsに通知する必要があることです。 そのようなアプリケーションは、Win32 API "SetProcessDPIAware"関数を呼び出すか、できればdpiAwareフラグでマニフェストを埋め込むことにより、新しいDPI対応フラグを設定する必要があります。 ただし、アプリケーションにDPI対応フラグがない場合、Windowsは異なる動作をします。まず、96 dpiのスケールで内部ディスプレイを形成し（ 96 のDPIアプリケーションをエミュレートします） 、以前の現在のDPI設定に従って受信した画像をスケーリングします画面出力。



すべてのモニターに最新のiPhoneのピクセル密度（326 dpi）があれば、素晴らしいスケーリング方法になります。 残念ながら、そうではありません。 この方法でスケーリングされたアプリケーションウィンドウは、120 dpiの一般的な解像度で少しぼやけて見えます（@hommはちなみに解像度ではありません）。 したがって、ピクセル密度を120 DPI以下に選択した場合、MicrosoftはデフォルトでDPI仮想化を無効にします。



DPI設定を変更する方法



Windows 7/8では、コントロールパネルを開き、[表示と個人設定]、[表示]の順に選択し、最後に[フォントサイズ（DPI）の設定]（Windows 7）または[カスタムサイズオプション]（ Windows 8）。 次のダイアログボックスが表示されます（Windows 7のWindows 7はほぼ同じです）。





ドロップダウンリストで、希望のDPI設定をパーセンテージで選択できます。100％は96 DPIに、125％はスクリーンショットのように120 dpiに対応します（手動でより正確に値を記録できます） 。 Windows 8より前は、システムフォントのサイズの横に実際のDPI（「インチあたりのピクセル数」）が表示されていました。 Windows 8では、理由は不明ですが、DPI値が表示されないため、自分で計算する必要があります。



また、ルーラー（インチ単位のスケール）を画面に添付し、ドロップダウンリストの値を変更して、画面上のマークと画面上のマークを組み合わせてみることもできます。 下の赤い丸で囲まれたチェックボックスは、Windows XPのスタイルでスケーリングのみを使用するか、DPI仮想化の新しい方法を使用するかを決定します。 スクリーンショットのようにチェックボックスがチェックされていない場合、DPI仮想化が有効になっています。



朗読。 このダイアログボックスは、使いやすいインターフェイスの例です。 一見、これはWindows XPのスタイルでスケーリングを無効にするフラグのようです。 ただし、このスケーリング方法（システムフォントとシステムのユーザーインターフェイスの他の要素を増やすだけ-Windows XPのスケーリング）は、高いDPI値が選択されると常に有効になります。 実際、このフラグは、このメソッドが唯一のものであるか（Windows XPのスタイルでスケールのみを使用する） 、DPI対応フラグを持たないアプリケーションにもDPI仮想化メソッドが適用されるかどうかを制御します。 したがって、このフラグはその名前で指定されたスケーリング方法を制御しませんが、どこにも言及されていない別のスケーリング方法を制御します-そして、ボックスがチェックされていないときに新しい方法を使用できます！



Windows 8のエラー。これに加えて、Windows 8では、このエラーダイアログボックスが表示されます。 原則として、すべてがWindows 7のように機能しますが、フラグの状態は150％以上のDPI値では保存されません。 このボックスをチェックすると、DPI仮想化が正しくオフになります。 ただし、次回このダイアログを開いたときには、チェックボックス自体はオフのままです。



Windows 8.1での変更、またはすべてがぼやけているのはなぜですか？



Windows 8.1では、Windows XPのスタイルでスケーリングするためのフラグがなくなり、「DPI仮想化」は最大120のDPI値で使用されることはありませんが、DPI対応フラグのないプログラムでは常により高い値で使用されます。 一部のアプリケーションがあいまいに見える場合は、それらのアプリケーションのDPI仮想化を手動で無効にする必要があります。



Windows 8.1では、異なるDPI値を持つ複数のモニターを使用できます。 ただし、この機能は、異なるDPI値を持つモニター間を移動する従来のアプリケーションに「DPI仮想化」の使用を強制します。 これを回避するには、新しいオプション「すべてのディスプレイで1つのスケールを選択したい」を使用して、DPI設定でDPIスケーリングを無効にできます。



Windows 8.1では、開発者が「DPI仮想化」を選択的に無効にできるように、200％を構成する特別なスイッチと新しいAPIも追加されています。



ヘルプ、私のシステムフォントは適切なサイズではありません！



DPI設定を変更した後、一部のシステムフォントが大きくなりすぎたり、新しくインストールするには小さすぎたりすることがあります。 考えられる理由は、古いDPI設定に基づいてカスタムデスクトップテーマを使用していることです。 Windowsは、カスタムテーマフォントをスケーリングしません。



実際にカスタムデスクトップテーマを作成して保存する場合は、新しいDPI設定に自分でフォントを適合させる必要があります。 ただし、Windowsには、何らかの理由で、ユーザーの知らないうちにカスタムテーマを「役立つように」作成するという厄介な習慣があります。 したがって、カスタムデスクトップテーマを作成したことがない場合は、それを削除して標準テーマに戻ります。



Windows 7/8では、コントロールパネルを開き、[外観と個人設定]、[個人設定]の順に選択します。 [マイテーマ]行に選択したエントリが表示されている場合、これは、Windowsがシステムフォントをスケーリングしないユーザーテーマを使用していることを意味します。 Aeroテーマ（Windows 7）またはWindowsデフォルトテーマ（Windows 8）セクションの最初のエントリなどの標準テーマを選択し、[マイテーマ]セクションで不要なエントリを削除します。 これで、すべてのシステムフォントが正しく表示されるはずです。

DPIをまったく気にしないアプリケーションは、非常に古いか、作成が不十分ですが、それでも使用されています。 有名な例の1つは、AppleのiTunes for Windowsです。 ここで、開発者はGUIにシステムフォントを使用し、実際のフォントサイズを気にせずに、ウィンドウサイズを96 DPIに厳密にバインドし、DPI値が高くなるとフォントサイズが大きくなるとGUIを自然に歪めます。



そのようなアプリケーションには、「DPI仮想化」をスケーリングする新しい方法が必要です。残念ながら、これにより多くの場合インターフェースがぼやけてしまいます。 そうしないと、テキストのトリミングからコントロールのオーバーラップに至るまでの問題が発生し、GUIが完全に使用できなくなることがあります（残念ながら、これはめったに起こりません）。 長年にわたり、 誤ったアプリケーションのスクリーンショットのサンプルをいくつか収集してきました。

150％の解像度（144 DPI）









GUIを異なるDPI値に調整できるが、DPI対応フラグがないアプリケーション-これらはWindows XP時代の典型的なアプリケーションです。 ここで、開発者は、GUIを作成する前に、システムの実際のフォントサイズを取得するように注意しました。 このようなアプリケーションは、Windows XPのスタイルでスケーリングを使用すると正しく表示されます。 残念ながら、Windowsにこの事実を伝えるDPI対応フラグを設定していないため、デフォルトでは「DPI仮想化」が使用され、GUIがファジーになります。 あなたはこれが気に入らないかもしれないので、あなたはそのようなアプリケーションのためにWindows XPスケーリングスタイルを強制したいかもしれません。

そのようなアプリケーションと150％（144 DPI）の解像度の例






DPI対応フラグを使用して、GUIを異なるDPI値に調整できるアプリケーション -これは、DPI設定に関係なく、完全に手間がかからない最新タイプのアプリケーションです。 DPI対応フラグは、Windows Presentation Foundation（WPF）およびGDI +アプリケーションに対して自動的に設定されます。これらのAPIには、組み込みのスケーリングツールが用意されているためです。 古いGDI APIと（驚くほど）Windows Formsを使用する開発者は、DPI対応アプリケーションに手動でフラグを立てる必要があります。



DPIの変更に適合していないが、DPI対応フラグを持っているアプリケーションは 、DPI値を完全に無視するよりもさらに悪いです。 この例では、120 DPIまで拡張できるがそれ以上ではないGUIアプリケーション、またはJavaFXアプリケーションがあります。 ここでは何もできません そのようなプログラムにWindowsでDPI仮想化を強制的に使用する方法はありません。 DPI対応フラグが設定されると、アプリケーションはそれ自体をスケーリングする必要があります。 開発者が製品を修正するために、または他の何かを使用するためにのみ「見た」ことができます。



アプリケーションのスケーリング方法を選択してください



高DPIを使用することを決定したら、スケーリング方法の選択は、使用するアプリケーションによって異なります。 「DPI仮想化」を無効にするということは、「Windows XPスタイルのスケールを使用する」という間違った名前のボックス（チェックボックス）をオンにすること、またはその逆を意味することに注意してください。



  • DPIに対応し、必要なフラグを設定しているアプリケーションのみを使用することが非常に幸運な場合は、どのスケーリング方法を選択してもかまいません。 すべてのアプリケーションはWindows XPスタイルのスケーリングを使用し、DPI仮想化は使用されません。
  • 十分に記述されたDPI対応アプリケーションのみを使用し、それらの一部が必要なフラグを設定しない場合、「DPI仮想化」をオフにできます。 したがって、すべてのアプリケーションは、スケーリングのためにぼやけることなく正しく表示されます。 モニターのピクセル密度が非常に高く、スケーリングされたビットマップがぼやけて見えなくなった場合は、とにかくDPI仮想化を有効にすることができます。
  • DPIの変更に対応しておらず、DPI対応フラグを持たない1つまたは複数のアプリケーションがある場合、歪んだGUIアプリケーションに対応する準備ができていない場合は、DPI仮想化を有効にする必要があります。 残念ながら、Microsoftはこのオプションを不便に実装しているため、ここで別の問題が発生します。 DPI仮想化は、単一のアプリケーションではなくシステム全体でのみ有効にでき、個々のアプリケーションでは選択的に無効にできます。




Windows 8.1では、この問題に関して選択肢がなくなったことを思い出してください。 120 dpi（125％）の解像度で作業する場合、各プログラムはWindows XPスタイルのスケーリングを使用するように強制されます。また、より高い解像度で作業する場合、DPI対応でない各プログラムはデフォルトの「 DPI仮想化。」



個々のアプリケーションのオプトアウトDPI仮想化



DPI仮想化を有効にすることを決定した後、または120 dpiを超える解像度でWindows 8.1を実行している場合、対応するフラグのないDPI対応アプリケーションのシステムを確認できます。 そして、彼らが彼らが意図されているWindows XPスタイルのスケーリングを使用する能力を返します。 これを行うには2つの方法があります。1つ目は32ビットアプリケーションでのみ機能し、2つ目は汎用で64ビットアプリケーションにも適しています。



32ビットアプリケーション -簡単です。Windowsエクスプローラーで実行可能ファイルを右クリックし、[プロパティ]ダイアログボックスを選択し、[互換性]タブをクリックして、[画面解像度の高い画像スケーリングを無効にする]チェックボックスを選択します。 それだけです。Windows8.1では、64ビットアプリケーションでも動作します。



64ビットアプリケーション -明確な理由はありませんが、Windows 8以前では、64ビットアプリケーションでは上記のフラグが無効になっていますが、レジストリに直接変更を加えると、オプション自体は非常に機能しますが、64ビットアプリケーションのユーザーを困らせることができます！ そのため、レジストリエディターを起動して、次のキーに移動します。



HKEY_CURRENT_USER \ Software \ Microsoft \ Windows NT \ CurrentVersion \ AppCompatFlags \ Layers



ここで、文字列値（REG_SZ）を追加します。名前はアプリケーション実行可能ファイルへのフルパスで、値はHIGHDPIAWAREです。 上記のように、最初にいくつかの32ビットアプリケーションを変更して、このレジストリキーの値の例を確認することをお勧めします。



Windows Vista以降でDPI設定を使用する方法を検討しました。 また、互換性オプションが何のためにあるのか疑問に思ったことがあります-「高画面解像度での画像スケーリングを無効にする」。 システムで何もしないのはなぜですか。システム全体のオプション「DPI仮想化」を有効にしていて、DPI対応フラグを適切に設定せずに、スケーリングを正しく使用するアプリケーションに対してのみ有効です。 Windows XPスタイル。
