JavaScript：コードを明確にするためのパス

作業コードは常に完璧ではありませんが、プログラムテキストを作成するときは、読みやすく、理解しやすく、修正しやすいように努力する必要があります。 コードを明確にするために努力する価値があります。 これを実現するには、エディターを開く前であっても、コードを適切に編成する必要があります。すべてを慎重に計画する必要があります。









何が起こるのかを明確にしたプログラミングは、優れた開発者を通常の開発者から分離するものです。 この記事では、明確なコードに向けて最初の一歩を踏み出すことを可能にするいくつかの基本原則を提供したいと思います。



ここで説明したアイデアはさまざまなプログラミング言語に適用できますが、ほとんどの例はオブジェクト指向のJavaScriptで記述されています。 この言語での開発に対するこのアプローチに慣れていない場合は、 「モジュール」パターンおよびJSのプロトタイプオブジェクト指向プログラミングに関する資料を見ることができます。 これにより、ここで説明する内容をすぐにマスターできます。



単独責任の原則



あなたが家事をしていて、壁にネジをねじ込むためにドリルをしたと想像してください。 仕事をしてツールを下げたとき、ドリルには1つの興味深い機能があることがわかりました。 その主な機能に加えて、ねじ込まれたねじの上に速乾性化合物を噴霧し、ねじを隠し、石膏を模倣します。 これは、ネジがねじ込まれる壁の一部をペイントする場合に非常に便利ですが、これは必ずしも必要ではありません。 さらに、何かに穴を開けるためだけに2回目のドリルを受けたくありません。 ドリルは、1つの機能のみを実行する場合、はるかに便利で信頼性の高いツールになります。 これにより、多くの状況での使用に適した、かなり柔軟なツールになります。



唯一の責任の原則は、コードの一部が1つのことを実行し、適切に実行する必要があることを示しています。 前の例のドリルと同様に、機能を制限すると、実際にコードスニペットの有用性が向上します。 これを念頭に置いてプログラミングすることで、多くの問題が解消されるだけでなく、将来プロジェクトに参加する開発者の作業が楽になります。



責任範囲の観点から関数とメソッドを検討してください。 コードの責任範囲を拡大すると、柔軟性と信頼性が低下し、変更の要求が厳しくなり、エラーが発生しやすくなります。 コードを明確にするために、各関数またはメソッドは単一のタスクを解決する必要があります。



関数が何をすべきかを記述し、同時に「and」を使用すると、この関数はおそらく複雑すぎます。 関数によって解決される問題は、関数の意味のある名前とその引数のリストの助けを借りてのみ説明できるように、十分に単純でなければなりません。その名前も関数によって解決される問題における役割を示します。



最近、電子版のMyers-Briggsテストを作成するタスクが与えられました。 以前はこれをしなければなりませんでした。 数年前、私が最初に同じタスクに出くわしたときに、 processForm



という巨大な関数を作成しました。 彼女は結果を計算し、チャートをプロットし、DOMとデータの視覚化に関連するすべてを管理しました。



問題は、何かを変更しなければならない場合、どこで変更を行うかを正確に理解するためだけに大量のコードを整理する必要があるということでした。 さらに、関数の腸のどこかで問題が発生した場合、エラーを見つけることは非常に困難でした。



そのため、今回同じ問題に直面したとき、すべてのロジックを関数に分割し、それぞれが独自の問題を解決しました。 これらの関数をモジュールオブジェクトにラップしました。 その結果、フォームの送信時に呼び出された関数は次のようになりました。 



 return {   processForm: function() {       getScores();       calculatePercentages();       createCharts();       showResults();   } };


ここでは、完全なアプリケーションコードを見ることができます。



このようなコードは、読み取り、理解、および修正が非常に簡単です。 プログラミングにはほど遠い人でも、ここで何が起こっているかを理解できます。 そして、これらの関数のそれぞれ（確かに、あなたはすでにそれについて推測しました！）1つの問題だけを解決します。 これが行動における唯一の義務の原則です。



たとえば、フォームデータを検証する必要がある場合、巨大な関数を編集する（および場合によってはその作業を中断する）代わりに、新しいメソッドをプロジェクトに追加するだけです。 さらに、このアプローチにより、関連する変数と関数をグループ化できるため、名前の競合の数が減り、コードの信頼性が向上し、必要に応じて他の目的での関数の再利用が大幅に簡素化されます。



覚えておいてください：1つの機能-1つのタスク。 大きな関数はクラスに処理する必要があります。 関数が互いに強く関連し、同じデータを使用する多くの問題を解決する場合、フォームを処理するために大きな関数で行ったように、それを作り直してメソッドでオブジェクトに変えるのが理にかなっています。



コマンドとリクエストの分離



おそらく、私がこれまで読んだ中で最も面白い通信は、David Thorneが彼女の行方不明のMissy猫を検索するためのポスターの作成についてShannon Walkleyに語ったものです。 シャノンがデイビッドにポスターを作るように頼むたびに、デイビッドは彼女がしたことをしたが、彼女自身から何かを追加した結果、シャノンは彼女が期待したものとは異なる何かを受け取った。 この通信は読むのがとても楽しいですが、あなたのコードが同じことをすれば、もはや面白くありません。



コマンドとリクエストの分離は、不要な副作用からコードを保護するための基礎となり、関数を呼び出すときの不快な驚きを防ぎます。 機能は2つのカテゴリに分類されます。 1つはコマンドを実行する関数を含み、2つ目はクエリを実行する関数を含みます。 前者はいくつかのアクションを実行し、後者は質問に答えます。 それらを混合することは推奨されません。 次の機能を検討してください。 



 function getFirstName() {   var firstName = document.querySelector("#firstName").value;   firstName = firstName.toLowerCase();   setCookie("firstName", firstName);   if (firstName === null) {       return "";   }   return firstName; } var activeFirstName = getFirstName();


これは単純化された例であり、関数の操作中に発生する予期しない副作用を簡単に確認できます。 実際には、ほとんどの副作用を見つけるのははるかに困難です。



関数の名前getFirstName



は、関数が名前（名）を返す必要があることを示しています。 ただし、彼女が最初に行うことは、ドキュメントから取得した名前を小文字に変換することです。 関数の名前は、どこかから何かを取得する（要求を実行する）必要があることを示していますが、関数はデータの状態も変更します（つまり、特定のコマンドを実行します）。 これは、関数の名前から推測することが不可能な副作用です。



さらに悪い。 この関数は、リクエストされた名前をcookieに書き込みますが、それについては何も通知しません。 これは、彼女が私たち自身がクッキーで書いたものを書き直し、私たちが期待していることを書き換えるという事実につながる可能性があります。 要求を実行する関数がデータを上書きすることはありません。



覚えておくと便利なルールは、関数が質問に答えた場合、データの状態を変更するのではなく、値を返す必要があるということです。 逆に、関数が何かをする場合、データを変更し、何も返さないようにする必要があります。 コードを最大限に明確にするために、同じ関数が特定の値を返してデータを変更することはありません。



同じコードの改善されたバージョンは次のようになります。 



 function getFirstName() {   var firstName = document.querySelector("#firstName").value;   if (firstName === null) {       return "";   }   return firstName; } setCookie("firstName", getFirstName().toLowerCase());


これは簡単な例ですが、願わくは、アクションを実行するコードと結果を返すコードにコードを分割することでプログラマの意図を明確にし、ミスを避けることができるため、明確に表示されることを願っています。 関数のサイズとコードベースが大きくなると、そのような分離はより重要になります。関数を使用するときはいつでも関数の定義を検索するだけで、この関数が正確に何をするのかを知ることは、誰かの効果的な使用とは言えませんいずれかの時間。

高レベルの接続性



子供がクローゼットにすべてを捨てて部屋を掃除するのを見たことがありますか？ もちろん、キャビネットのドアを閉めた状態では、外見はまともなように見えますが、そのような「クリーニング」の後は何も見つからず、互いに共通点のないものが同じ場所に収まることがよくあります。 コードを書くときに高レベルの接続性を追求しないと、コードでも同じことが起こります。



接続性は、プログラムのさまざまなコンポーネントがどのように適合するかを示す尺度です。 高レベルの接続性は優れており、コードスニペットがより明確になります。 接続性が低いと混乱が生じます。 コードブロック内の関数とメソッドは、それらが実行するアクションの意味で接続されている必要があります。つまり、高レベルの接続性が必要です。



高レベルの接続性とは、共通のアイデアによって結合された構造を1か所に集めることを意味します。 たとえば、データベースを操作するための関数、またはアプリケーションによって解決されるタスクのあらゆる側面に関連する関数を1つのコードブロックまたはモジュールで収集します。 このようなアプローチは、これらのエンティティがどのように編成され、どこにあるかを理解するのに役立つだけでなく、名前の競合を防ぐのにも役立ちます。 30個の関数がある場合、名前が競合する可能性は、論理的に4つのクラスに分割されている30個のメソッドがある場合よりもはるかに高くなります。



2つまたは3つの関数が同じ変数を使用する場合は、何らかの方法でグループ化する必要があります。 実際、これはそれらをオブジェクトに結合する良い機会です。 スライダーなどのページ要素を制御するために使用される関数と変数のセットがある場合、これらすべてからオブジェクトを作成し、アプリケーション内のコード接続のレベルを上げることができます。
