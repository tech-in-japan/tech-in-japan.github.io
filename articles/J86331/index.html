この記事では、ASP.NET MVCに基づいてWebアプリケーションをローカライズするすべての側面を検討します。 このトピックの執筆時点では、最新のMVC 2 RC 2バージョンを使用しています。



始める前に、MVCチームに感謝したいと思います。すばらしい仕事をしてくれた皆さん、このフレームワークを使用するときにWebアプリケーションを作成するプロセスを楽しんでいます。 Ruby on Railsで少し経験した後、このタイプのフレームワークを探していました。



次の問題を検討します。

  1. 検証を表示
  2. シンプルな作物切り替えメカニズム
  3. モデル検証メッセージのローカライズ
  4. DisplayName属性のローカライズ
  5. キャッシュとローカリゼーション
作業するには、Visual Studio 2008 ExpressとASP.NET MVC 2 RC2が必要です。また、新しいMVC 2 Webプロジェクトを作成する必要があります。

もちろん、ローカライズされたテキストの場合、リソースファイルが使用されますが、ストレージには標準のasp.netフォルダーは使用されません。



次のディレクトリ構造をお勧めします。







ビュー -aspxページのリソースファイル。 モデル -ビューモデルをローカライズするためのリソースファイル。



Viewsディレクトリには、リソースファイルが配置されている各コントローラーのサブフォルダーが含まれています（ファイルの数は、サポートされている言語のセットによって異なります）。



モデルには、ビューモデルの各グループのサブディレクトリが含まれます。 生成されたアカウントモデル（LogOn、Register、ChangePassword）には、各言語カルチャのAccountフォルダーとリソースファイルがあります。



リソースファイル



リソースファイルの命名規則に関するいくつかの言葉。 リソースファイルの名前の形式は次のとおりです。

[リソース名]。[文化] .resx



リソース名-ファイル名。 絶対に何でも構いません。 同じリソース名を持つ複数のリソースファイルがある場合にグループ化するために使用されます-それらはCULTUREで指定された異なるカルチャを持つ単一のリソースを構成します。



CULTURE-リソースファイルカルチャのインジケータ。 文化には、 中立と正確の 2つのタイプがあります。 ニュートラルカルチャは、言語コード（ en 、 ru 、 deなど）のみで構成されます。 正確なカルチャは、言語コードと地域コードで構成されます（ en-US 、 en-UK ）



また、カルチャが定義されていないリソースファイルには特別な意味があり、「デフォルト」または「フォールバック」カルチャと呼ばれます。 名前から推測できるように、特定のカルチャリソースファイルでテキストが見つからなかった場合、または特定のカルチャのファイルがない場合、リソースファイルに使用されます。 特にユーザーがサポートされていないカルチャを何らかの方法でインストールできる場合は、基本的なリソースファイルを使用することを強くお勧めします。



リソースファイルのいくつかの例：



MyStrings.en-US.resx-英語アメリカ



MyStrings.en-UK.resx-イギリス英語



MyStrings.en.resx-ニュートラル英語（英語のデフォルト）



MyStrings.ru.resx-中立ロシア語



MyStrings.resxはベースリソースファイルです。

カルチャを変更するには、リクエストごとに CurrentThreadオブジェクトのCurrentCultureおよびCurrentUICultureプロパティを変更する必要があります!!! これを行うには、カルチャを変更するコードをApplication_AcquireRequestState Global.asaxメソッドに配置します（このメソッドはイベントハンドラーであり、リクエストごとに呼び出されます）



Global.asax.csファイルに次のコードを追加します。

protected void Application_AcquireRequestState( object sender, EventArgs e)

{

//Create culture info object

CultureInfo ci = new CultureInfo( "en" );



System.Threading.Thread.CurrentThread.CurrentUICulture = ci;

System.Threading.Thread.CurrentThread.CurrentCulture = CultureInfo.CreateSpecificCulture(ci.Name);

}

DisplayName属性のローカライズ



前の章で述べたように、DisplayName値は、フォーマットにパラメーターを使用する検証メッセージに含まれます。 DisplayName属性について考えるもう1つの理由は、HTMLフォームのテキストフィールドラベルです。これらはDisplayName値の参加によって作成されます。



本当の問題は、DisplayNameがローカライズをサポートしていないことです。これは、値を取得する場所からリソースファイルに関連付ける方法がありません。



これは、DisplayNameAttributeを拡張し、DisplayNameプロパティを書き換えて、ローカライズされた名前を返す必要があることを意味します。 継承クラスを作成し、LocalizedDisplayNameと呼びました。

public class LocalizedDisplayNameAttribute : DisplayNameAttribute

{

private PropertyInfo _nameProperty;

private Type _resourceType;



public LocalizedDisplayNameAttribute( string displayNameKey)

: base (displayNameKey)

{



}



public Type NameResourceType

{

get

{

return _resourceType;

}

set

{

_resourceType = value ;

// nameProperty, set'

_nameProperty = _resourceType.GetProperty( base .DisplayName, BindingFlags.Static | BindingFlags.Public);

}

}



public override string DisplayName

{

get

{

//,nameProperty null

if (_nameProperty == null )

{

return base .DisplayName;

}



return ( string )_nameProperty.GetValue(_nameProperty.DeclaringType, null );

}

}

}





重要な詳細は、要求されるたびにプロパティの値を読み取る必要があるという理解です。したがって、GetValueメソッドは、コンストラクターではなくDisplayNameプロパティのget'erで呼び出されます。

キャッシュとローカリゼーション



奇妙な章ですよね？ キャッシングとローカリゼーションを組み合わせる方法を考えていますか？ OK、次のシナリオを想像してみましょう。HomeControllerを開き、OutputCache属性をIndexアクションメソッドに追加します。

[OutputCache(Duration=3600, VaryByParam= "none" )]

public ActionResult Index()

{

ViewData[ "Message" ] = "Welcome to ASP.NET MVC!" ;



return View();

}





アプリケーションを起動し、[インデックス]ページで言語を変更して、ヘッダーがまだローカライズされていることを確認します。



なんてこった！ キャッシングとローカリゼーションを一緒に使用することはできないとお考えですか？ 心配しないでください、この問題の解決策が存在します:)



OutputCacheについて何を知っていますか、VaryByCustomプロパティについて何を知っていますか？ それを使用する時間です。



インデックスページを初めて要求すると、OutputCacheがページをキャッシュします。 2番目のリクエスト（言語選択リンクをクリックしたとき）で、OutputCacheは何も変更されていないと判断し、キャッシュから結果を返すため、ページは再作成されません。 そのため、言語の選択が機能しませんでした。 この問題を解決するには、ページのバージョンが変更されたことを何らかの方法でOutputCacheに伝える必要があります（アクションが特定のパラメーターを受け取り、VaryByParamプロパティに渡す場合など）。



VaryByCustomはこの問題の解決策の理想的な候補であり、Global.asax.csファイルのSystem.Web.HttpApplicationクラスに特別なメソッドがあります。 このメソッドの標準実装を書き直します。

public override string GetVaryByCustomString( HttpContext context, string value )

{

if ( value .Equals( "lang" ))

{

return Thread.CurrentThread.CurrentUICulture.Name;

}

return base .GetVaryByCustomString(context, value );

}





まず、このメソッドはパラメーター値が「lang」（特別な値なし、VaryByCustomの値として使用される単純な文字列）に一致するかどうかを確認し、一致する場合は現在のカルチャの名前を返します。 それ以外の場合は、標準実装の値を返します。



次に、ローカライズを使用する各OutputCache属性に値「lang」を持つVaryByCustomプロパティを追加します。これがすべてです。 更新されたインデックスアクションメソッドは次のとおりです。



[OutputCache(Duration=3600,VaryByParam= "none" , VaryByCustom= "lang" )]

public ActionResult Index()

{

ViewData[ "Message" ] = "Welcome to ASP.NET MVC!" ;

return View();

}
