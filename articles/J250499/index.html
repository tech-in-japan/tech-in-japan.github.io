VMwareクラウドでハイパーバイザーを使用してSSDキャッシングを行う方法

VMwareは、VMware vSphere 5.1のリリースにより、ESXiサーバーのローカルディスク上のSSDドライブで分散キャッシュを使用する可能性を含む、仮想マシンデータのストレージにおけるいくつかの新しいベンチャーを発表しました。 このテクノロジーの名前はvFlashで、Tech Preview段階にあり、後にフル機能のvSphere Flash Read Cache（vFRC）プラットフォームVMware vSphere 5.5になりました。 そして、これはさまざまなレベルのタスクで使用できる非常に実用的なツールです。



VFRCテクノロジーは、読み取りデータをキャッシュすることにより、ディスクサブシステムとやり取りするクライアントアプリケーションの効率を高め、読み取り操作をアクティブに実行するアプリケーションのパフォーマンスを大幅に向上させるように設計されています。


フラッシュ読み取りキャッシュは、ハイパーバイザーレベルで機能すると同時に、読み取り操作にI / Oシステムを頻繁に使用する仮想マシンのパフォーマンスを大幅に向上させます。 キャッシングには、ホストにローカルにインストールされたPCIeフラッシュカードとSAS / SATA SSDを使用できます。 デバイスはフラッシュプールに結合され、そこから仮想マシンのVMDKディスクにデータキャッシュ用のスペースが割り当てられます。

VMDK（仮想マシンディスク）は、VMwareが仮想マシンのディスクイメージとして使用するために開発したファイル形式であることを思い出してください。


パフォーマンスの観点から、SSDキャッシュはRAMと通常のディスクの間にあります。



VFRCアーキテクチャの概要









vFRCのアーキテクチャ機能は次のとおりです。

vFRCが有効になっているVMDKドライブに読み取り要求が届くと、vFlashに必要なデータがあるかどうかが最初に判断されます。


  • その場合、仮想マシンはキャッシュからデータを受け取ります。 このイベントは「ヒット」（vFRCヒット）と呼ばれます。
  • データがキャッシュにない場合、ESXiはVMDKディスクからデータを読み取り、マシンに提供し、キャッシュにデータを書き込みます。 このイベントは、ミス（vFRCミス）と呼ばれます。 書き込み要求が到着すると、データはVMDKディスクに書き込まれ、キャッシュに非同期的に書き込まれます。


vFRCには何が必要ですか？



vFRC機能をアクティブにするには、次の条件を満たす必要があります。

  • 少なくとも1つのSSDまたはPCIe SSDを備えた構成済みホストが必要です。
  • vSphere 5.5（vCenter 5.5およびESXi 5.5）を使用します。


vFRCはどのようにオンになりますか？



ESXiを使用してデバイスをサーバーに物理的に接続した後、vSphere Flash Infrastructureレイヤーに追加する必要があります。 これは、ホスト設定の[仮想フラッシュリソース管理]タブで実行できます。







仮想マシンでvFRCを有効にするには、ハードディスク設定で[仮想フラッシュ読み取りキャッシュ]項目を使用します。この項目では、キャッシュに割り当てるスペースの量とブロックサイズを指定できます。 vFRCのブロックサイズは、アプリケーションがディスクにデータを書き込むブロックに応じて選択する必要があります。 各ドライブのブロック統計は、ESXiのvscsiStatsユーティリティを使用して収集できます。







構成機能



vSphere Flash Read Cacheを構成するときは、次の点を考慮してください。

  • Enterprise PlusエディションのVMware ESXi 5.5がホストサーバーにインストールされている必要があります。
  • VFRCはvSphere Web Clientを介してのみ構成および管理されるため、VMware vCenter Serverが必要です。
  • 1つの仮想ディスクの最大キャッシュサイズは400 GBです。
  • ホストあたりの最大キャッシュサイズは2 TBです。
  • 仮想ディスクの最大サイズは16 TBです。
  • キャッシュに使用されるSSDの最大数は8です。
  • 仮想マシンのハードウェアバージョンをバージョン10に更新する必要があります。
  • 各仮想ディスクのキャッシュを手動で構成する必要があります。最小値は1 GBです。


IT CITYでの実際のvFRC



そして、個人的な経験から少し。 VMwareクラウドでSSDキャッシュをどのようにテストし、実際にどのような落とし穴に遭遇しましたか？



ソリューションを顧客に提供する前に、それが完全に機能的かつ機能的であることを確認する必要があります。つまり、結果が宣言された機能と完全に一致し、顧客の要件を満たしていることをテストして理解する必要があります。 そして、まだ誰もテストも実装もしていない新規性がある場合は、特に注意してテストする必要があります。 結局のところ、新しいものすべてに隠れた欠陥、バグ、その他の不快な些細なことがあるのは誰にとっても秘密ではありません。



VMwareがESXiサーバーのローカルディスクのSSDドライブで分散キャッシュを使用する新しい可能性を発表するとすぐに、この機能をテストすることにしました。 このテクノロジーは、vSphere 5.5がリリースされる前はTech Preview段階であったため、すでに開発されたソリューションをテストしたかったのです。 私たちのタスクは、構築されたスタンドでvFRCのパフォーマンスを確認することでした。



テストのために、SSDをDell PERC H710P RAIDコントローラーに接続しました。 SSDディスクの数に応じて、各グループに1つのディスクでRAID-0グループを作成しました。







Dell PERC H710P RAIDコントローラーは、接続されているメディアの物理タイプに関する情報を提供できないため、ESXIに接続されているドライブはSSDドライブであることに注意する必要がありました。 これを行うには、esxclコマンドを実行します。







デバイスパラメータでコマンドを実行すると、「Is SSD」フラグの値が「True」に変更されました。







次に、デバイスをvSphere Flash Infrastructureレイヤーに追加しました。 現在の手順は、仮想フラッシュリソース管理オプションを使用してホスト設定で実行されました。







SSDキャッシュをテストするために、Windows Server 2008 R2 x 64 OSベースの仮想マシンと、各仮想マシンにそれぞれ100 GBの容量が割り当てられた2つのVMDKディスクを備えたスタンドを準備しました。

  • OSで識別されるVMDK1、
  • VMDK2-データ用。


次に、仮想マシンのVMDK2ハードディスクのパラメーターで、vFRCがオンになり、キャッシュに100 GBが割り当てられ、4 KBのブロックサイズが決定されました。







基本的な構成手順が完了しました。 次は、含まれている機能の機能を検証するタスクでした。 ただし、仮想マシンを起動すると、起動が拒否され、ウェルカムウィンドウの代わりに、次の内容の「ブルースクリーン」が表示されました。







残りの仮想マシンには明らかな問題はありませんでした。

次に、SSDキャッシュを監視するために研ぎ澄まされたツールを使用し、テスト結果を比較することにしました。 まず、仮想マシンの1つがFIOユーティリティを起動しました。これにより、VMDK2ディスクに必要な量のデータが生成されます。 前述したように、有用なデータに割り当てられたのは彼でした。 FIOユーティリティはさまざまなモードで動作します。「ランダム読み取り」手順に興味がありました。 それが彼らがランドリードモードで起動した理由です。

注：FIOユーティリティの詳細については、 http ：//freecode.com/projects/fioを参照してください 。


FIOユーティリティは、テスト用のパラメーターが書き込まれるジョブファイル（または、より単純には構成ファイル）の使用を意味します。 ユーティリティは、VMDK2ディスクからランダムに生成されたデータに対して読み取り操作を実行します。 読み取り用の構成ファイルでは、読み取りブロックのサイズは固定されています（この場合、4 Kbに等しい）。 次に、ランダム読み取り操作を開始しました。 テスト時間は6時間46分でした。







私は質問に興味がありました：読み取りデータがキャッシュにヒットしましたか？そうであれば、ヒットの割合は何ですか？

答えを見つけるために、vSphere WEBクライアントを介したマシンの仮想ディスクのパフォーマンスのグラフを使用しました。







次のカウンターを見るのは面白かったです：1秒あたりの出力操作の平均数、読み取り遅延、およびキャッシュ使用量の統計を提供するカウンター。 後者は少し期待はずれで、キャッシュにデータが入る割合が非常に少ないことを示しています。 1秒あたりの平均出力操作数（18,689.328）では、キャッシュされたデータの値は4439.389であり、ヒットのわずか23％です。 この統計シナリオによれば、キャッシュは単にアイドル状態であると見なすことができます。



標準ツールでは予期した結果が表示されなかったため、別のツールであるesxcliコマンドに切り替えました。 また、vFRCオプションが有効になっている特定のVMDKドライブの統計でも機能します。 以下のパラメーターを使用してコマンドを実行します。

〜＃esxcli storage vflsh cache stats get







この図では、キャッシュヒット率の値をパーセントで表示できます。 いわゆる「ヒット」vFRCヒット、つまり、仮想マシンが使用するキャッシュのデータの割合を示します。 次の開始時の結果は完全に異なっていたため、問題のチームは数回実行する必要がありました。 1つの値については、キャッシュはまったく機能しませんでした。最初の場合と同様に、キャッシュが機能する他のケースでは、キャッシュに入るデータの割合は96％でした。



彼らは受け取ったものにこだわらず、別のユーティリティを使用しました：esxtop（インタラクティブコマンド「u」（u：ディスクデバイス）を送信）を使用して、キャッシュ使用量の統計を表示しました。 画面に表示された情報によると、次の結果が得られました。読み取り時に、データがキャッシュから直接抽出されました。 1秒あたりの出力操作の平均数が18,689.328であり、SSDキャッシュから読み取られたデータの操作量が18,184.03であるとすると、キャッシュに入るデータの割合は約97％でした。







テスト結果は私たちの期待を完全には満たしていなかったため、VMwareパートナーである主要なサービスプロバイダーとして、ベンダー側の同僚に助けを求めました。



VMwareは、顧客やパートナーとやり取りする豊富な経験を持っています。 製品の機能に関するバグ、ボトルネック、その他の問題を検出した場合、開発者は必要な修正を行うためにあらゆる努力をします。



その結果、2014年の秋にVMware ESXi 5.5 Update 2がリリースされ、Windows Server 2008 R2 x64を実行している仮想マシンのブルースクリーンで説明されている問題が修正されました。



もちろん、リリースされたアップデートには興味がありました。 vFRCを有効にして、以前にレビューしたテストサイトにインストールしてテストすることにしました。 結果は何ですか？ すべての仮想マシンは1つとして起動しました。 このテストに「+」を追加し、カウンターインジケーターに向かって移動します。 テストの最初の時点で、先ほど使用した構成ファイルを使用してFIOユーティリティをrand-readモードで起動し、その後でランダム読み取り操作を開始しました。 ほとんどのカウンターは動作統計を表示し、定期的に誤った値を示していました。 つまり、VMware ESXi 5.5 Update 2は、vFRC統計を表示するという前述の問題を修正しませんでした。 このバグにもかかわらず、vSphere Flash Read Cacheテクノロジーは、この機能を使用した後のプラクティスで示されているように、レイテンシを削減することで仮想マシンのパフォーマンスを大幅に改善します。



次のテストの後、産業環境のホストにSSDキャッシングテクノロジーを導入しました。 今日、vSphere Flash Read Cacheを使用するいくつかのプロジェクトが、特にパフォーマンスを要求するお客様のために、当社のサイトで正常に実装されています。 後者は、システムとアプリケーションの作業を加速した結果に満足しています。
