PHP検証ノート





PHPは、Webアプリケーションの開発に集中的に使用される汎用スクリプトプログラミング言語です。 現在、大半のホスティングプロバイダーによってサポートされており、動的なWebサイトの作成に使用されるプログラミング言語のリーダーの1つです。



コンパイラとインタープリターの場合、原則として、ソースコードとテストには高い品質と信頼性の要件があります。 しかし、PHPインタープリターのソースコードには疑わしい場所がありました。



この記事では、 PVS-Studio 5.18を使用して取得したPHPインタープリターをチェックした結果を調べます。







同一の条件式



V501 '||'の左と右に同一の副次式 '！Memcmp（ "auto"、charset_hint、4）'があります。 演算子。 html.c 396 

static enum entity_charset determine_charset(char *charset_hint TSRMLS_DC) { .... if ((len == 4) /* sizeof (none|auto|pass) */ && //<== (!memcmp("pass", charset_hint, 4) || !memcmp("auto", charset_hint, 4) || //<== !memcmp("auto", charset_hint, 4))) //<== { charset_hint = NULL; len = 0; } .... }


条件式には、同じパラメーターを使用した関数 'memcmp'の呼び出しが含まれています。 コメント/ * sizeof（none | auto | pass）* /は、関数の1つに値「none」を渡す必要があることを示しています。



常に偽の状態



V605式：shell_wrote>-1の検証を検討してください。符号なしの値は、数値-1と比較されます。 php_cli.c 266 

 PHP_CLI_API size_t sapi_cli_single_write(....) { .... size_t shell_wrote; shell_wrote = cli_shell_callbacks.cli_shell_write(....); if (shell_wrote > -1) { //<== return shell_wrote; } .... }


この比較は明らかな間違いです。 '-1'は 'size_t'型の最大値に変換されるため、条件は常にfalseになり、チェック全体が無効になります。 変数 'shell_wrote'には以前に符号付きの型が含まれていた可能性がありますが、リファクタリング後、符号なしの型の操作の機能は考慮されませんでした。



無効な条件



V547式 'tmp_len> = 0'は常にtrueです。 符号なしの型の値は常に> = 0です。ftp_fopen_wrapper.c639 

 static size_t php_ftp_dirstream_read(....) { size_t tmp_len; .... /* Trim off trailing whitespace characters */ tmp_len--; while (tmp_len >= 0 && //<== (ent->d_name[tmp_len] == '\n' || ent->d_name[tmp_len] == '\r' || ent->d_name[tmp_len] == '\t' || ent->d_name[tmp_len] == ' ')) { ent->d_name[tmp_len--] = '\0'; } .... }


符号なしの「size_t」タイプを使用すると、アプリケーションの現在の容量に対して配列要素の最大数にインデックスを付けることができます。 チェック（tmp_len> = 0）は正しくありません。 最悪の場合、デクリメントによりインデックスオーバーフローが発生し、アレイ外のメモリアクセスが発生する可能性があります。 ほとんどの場合、コードは追加の条件と正しい初期データのために正しく実行されますが、「ループ」または配列から出る危険性があります。



符号なしの数の差



V555式「out_buf_size-ocnt> 0」は「out_buf_size！= Ocnt」として機能します。 filters.c 1702 

 static int strfilter_convert_append_bucket( { size_t out_buf_size; .... size_t ocnt, icnt, tcnt; .... if (out_buf_size - ocnt > 0) { //<== .... php_stream_bucket_append(buckets_out, new_bucket TSRMLS_CC); } else { pefree(out_buf, persistent); } .... }


たぶん、elseブランチは実行する必要があるよりも頻繁に実行されません。 符号なしの数値の差はほとんど常にゼロより大きくなります。 例外はオペランドの等価性です。この場合、条件はより有益なバージョンに書き直す方が適切です。

V595 nullptrに対して検証される前に、「function_name」ポインターが使用されました。 チェック行：4859、4860。basic_functions.c 4859

陰湿な最適化



V597コンパイラーは、「最終」バッファーをフラッシュするために使用される「memset」関数呼び出しを削除できました。 RtlSecureZeroMemory（）関数を使用して、プライベートデータを消去する必要があります。 php_crypt_r.c 421

最終的な配列にはプライベートパスワード情報を含めることができます。この情報はゼロにリセットされますが、memset関数の呼び出しはコンパイラによって削除されます。 これが発生する可能性のある理由と危険な理由の詳細については、記事「 メモリの上書き-理由 」およびV597診断の説明を参照してください 。

使用するライブラリを信頼できますか？



サードパーティのライブラリがプロジェクトの開発に大きく貢献しているため、すでに実装されているアルゴリズムを再利用できますが、その品質もメインプロジェクトと同様に監視する必要があります。 この記事のトピックをたどり、サードパーティのライブラリの信頼の問題を単に提起するために、いくつかのライブラリのほんのいくつかの例を挙げます。



PHPインタープリターは多くのライブラリーを使用しますが、その一部は「自分用に」わずかに書き換えられています。



libsqlite



V579 sqlite3_result_blob関数は、ポインターとそのサイズを引数として受け取ります。 間違いかもしれません。 3番目の引数を調べます。 sqlite3.c 82631 

 static void statInit(....) { Stat4Accum *p; .... sqlite3_result_blob(context, p, sizeof(p), stat4Destructor); .... }


ほとんどの場合、彼らはポインタではなくオブジェクトのサイズを取得したいと考えていました。 sizeof（* p）と書く必要があります。



pcrelib



V501 「|」の左側と右側に同一の副次式「（1 << ucp_gbL）」があります 演算子。 pcre_tables.c 161 

 const pcre_uint32 PRIV(ucp_gbtable[]) = { (1<<ucp_gbLF), 0, 0, .... (1<<ucp_gbExtend)|(1<<ucp_gbSpacingMark)|(1<<ucp_gbL)| //<== (1<<ucp_gbL)|(1<<ucp_gbV)|(1<<ucp_gbLV)|(1<<ucp_gbLVT), //<== (1<<ucp_gbExtend)|(1<<ucp_gbSpacingMark)|(1<<ucp_gbV)| (1<<ucp_gbT), .... };


配列の1つの要素を計算する式には、繰り返しの1つ（1 << ucp_gbL）があります。 以下のコードから判断すると、ucp_gbL変数の1つはucp_gbTと呼ばれることもあれば、単に不要なこともあります。



PDO



V595 nullptrに対して検証される前に、「dbh」ポインターが使用されました。 行を確認：103、110。pdo_dbh.c 103 

 PDO_API void pdo_handle_error(pdo_dbh_t *dbh, ....) { pdo_error_type *pdo_err = &dbh->error_code; //<== .... if (dbh == NULL || dbh->error_mode == PDO_ERRMODE_SILENT) { return; } .... }


ここでは、関数の最初に着信ポインタの逆参照が実行され、その後、有効性がチェックされます。



libmagic



V519 「*コード」変数には、値が連続して2回割り当てられます。 おそらくこれは間違いです。 行を確認：100、101。encoding.c 101 

 protected int file_encoding(...., const char **code, ....) { if (looks_ascii(buf, nbytes, *ubuf, ulen)) { .... } else if (looks_utf8_with_BOM(buf, nbytes, *ubuf, ulen) > 0) { DPRINTF(("utf8/bom %" SIZE_T_FORMAT "u\n", *ulen)); *code = "UTF-8 Unicode (with BOM)"; *code_mime = "utf-8"; } else if (file_looks_utf8(buf, nbytes, *ubuf, ulen) > 1) { DPRINTF(("utf8 %" SIZE_T_FORMAT "u\n", *ulen)); *code = "UTF-8 Unicode (with BOM)"; //<== *code = "UTF-8 Unicode"; //<== *code_mime = "utf-8"; } else if (....) { .... } }


エンコードを変数に2回設定すると、1行は不要になり、プログラムの動作がさらに不適切になる可能性があります。



おわりに



PHPは長い間存在しており、人気があるという事実にもかかわらず、この種のプロジェクトはおそらくさまざまなアナライザーによってチェックされますが、そのコードと使用されたライブラリに疑わしい場所が見つかりました。



静的分析を定期的に使用することで、より便利なタスクを解決するために多くの時間を節約できます。
