DIY Micro-ORM（パート1）

このライブラリを作成する動機と、既存のソリューションが悪い理由：

残念ながら、Hibernateのようなモンスターは「重く」、データベースを操作するための独自のAPIを課しています。 通常のJDBCコードと組み合わせて使用​​できるシンプルなライブラリが必要でした（実際、JDBCには何らかのDapper.NETが必要でした）。



ライブラリの作成に使用される基本原則：

  • シンプルさと原子性-ライブラリは1つのjavaファイルであり、プロジェクトに追加するにはファイルをソースに追加するだけです。
  • 非侵入性-ライブラリはそのAPIを課しません。通常のJDBCコードと「ミックス」を使用することができます
  • 独立性-ライブラリはJava SE 5のみを使用します
  • 拡張性-ライブラリは、特定のプロジェクトに必要な拡張機能の追加をサポートします




パラメータのコレクションを使用してPreparedStatementを作成するメソッドの実装。

この種のコードを書きたい： 

PreparedStatement stmt = DB.prepareStatement(conn, "select id, login, email from users where (login={0} or email={0}) and pass={1}", "dimzon", "pass")


ここで、パラメーター値は位置的に渡され、リクエスト本文では{PARAMETER_ INDEX}構造を使用して参照されます。 このようなレコードは簡潔で便利です（そして.NETプログラマーにはおなじみです）。

したがって、次のメソッド宣言があります。 

 public static PreparedStatement prepareStatement(Connection cn, CharSequence query, Object... args) throws SQLException


実装は、2つの問題を解決することで構成されます。

  1. {PARAMETER_ INDEX}構造のエントリのリクエスト本文を解析し、「？」に置き換えます および「argsのインデックス-> preparedStatementのインデックスの配列」のマッピング
  2. 適切なsetXXXXメソッドを呼び出すことによるPreparedStatementのパラメーターの置換
最初の問題は、正規表現を使用して解決されます。 そして、2番目の問題を解決するとき、ニュアンスが生じます。

  1. SQLパラメーターのタイプの判別。 単純なケース（整数、日付）の場合、すべてが単純ですが、非標準クラスが引数として渡された場合の対処方法は明確ではありません。
  2. ヌル値の解釈。 Javaでは、nullは型指定されていません（つまり、null.getClass（）を実行できません）-PreparedStatement.setNullに渡すjava.sql.Typesが明確ではありません。
両方の問題を解決するには、「戦略」パターンが役立ちます。 

 public interface Param { void set(PreparedStatement ps, int index) throws SQLException; }


生活がより簡単になりました-入力にParamインターフェイスの実装がある場合、単にsetメソッドを呼び出します。 標準ケース（整数、日付）、均一性のために、戦略を通じても紹介します。 そのため、パラメータのタイプに応じて適切な戦略を作成する方法が必要です（最初に戦略でない場合）。

ここでは、「工場」パターンが役立ちます。 しかし、私はファクトリー用に別のインターフェースを開始しませんでした。次の抽象クラスを書きました（.NETプログラマーはFunc <T、R>のアナログを見るでしょう）。 

 public static abstract class FN<I,O> { public abstract O apply(I input) throws Exception; }


したがって、ファクトリはクラスFN <Object、Param>の子孫です。

タイプごとにファクトリインスタンスをすばやく取得するには、ConcurrentHashMap <Class、FN <Object、Param >> paramFactoriesを使用します。

このマップにファクトリを追加すると、クラスをそのままprepareStatementに渡すことができます-さらに1つのメソッドが表示されます： 

 public static <E> void registerParamFactory(Class<E> forClass, FN<E, Param> factory)


したがって、非標準の状況（拡張性）の場合、2つのアプローチのいずれかを適用できます。

  • Paramの実装をprepareStatementに明示的に渡す
  • registerParamFactoryでParam作成ファクトリを登録する


最後の質問は、prepareStatementにnullが含まれた場合の対処方法です。 タイプを判別できませんが、「？」の代わりに 「NULL」をリクエスト本文に直接挿入すると、サーバーはそれを把握します。
