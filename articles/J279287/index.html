DHH（David Heinemeyer Hensson）がレールコントローラを整理する方法



Full Stack Radioとの最近のインタビューで、教祖であり救世主であるDHHは、最新バージョンのBasecampでコントローラーのレール構造をどのように整理するかについて説明しました。 ここに彼の聖なる言葉の転写があります：



コントローラーの作成におけるほぼ原理主義者として、私はRESTにコミットし続けています。



コントローラーに不満を感じるたびに、これは私がコントローラーをほとんど持っていないからです。 私はそれらを過負荷にしようとしています。



そのため、Basecamp 3では、ある種のサブリソースが存在するたびにコントローラーを超えていました。 それらはフィルターとして機能できます。



このコードがあり、次のようになっているとします： 



class BananasController < ApplicationController def index render plain: '\_(:-))_/' end end


いくつかのフィルターとドロップダウンメニューを適用する場合、これは別のものになります。 時々、私たちはこれを取り、それのために完全に新しいコントローラーを実行します。



私が特に仕事に使用するヒューリスティックです：コントローラーに新しいアクションを追加したいときはいつでも、このアクションはデフォルトで5つのアクションの1つに含まれていません。つまり、この欲求を満たし、新しいコントローラーを作成するだけです。 ！ だから私はそれを呼び出します。



したがって、InboxController（メールボックスコントローラー）があり、メールボックス内のすべてを表示するインデックスアクションがあるとします。 また、「保留中の新しいアクションを追加したい場合は、保留中のメールなどのようなものになります」という別のアクションを実行できます。



そして、このアクション保留を追加します： 



  class InboxesController < ApplicationController def index end def pendings end end


これは非常に一般的なテンプレートですよね？ そして、私が使用するサンプルでさらに説明できます。 次に、「いいえ、いいえ、いいえ」と言って、Inboxes :: PendingsControllerという新しいコントローラーを作成します。このコントローラーには、1つのインデックスアクションがあります。 



  class InboxesController < ApplicationController def index end end class Inboxes::PendingsController < ApplicationController def index end end


そして、ここで私がその自由を見つけたのは、各コントローラーに独自の適用分野があり、独自のフィルターセットが適用されているという事実です...



そのため、コントローラーの大規模な分布、特に名前空間でのコントローラーの分布があります。 MessagesControllerがあり、Messages :: DraftsControllerコントローラーとMessages :: TrashesControllerがあり、これらすべてのサブコントローラーとサブリソースが同じコントローラー内にあると仮定しましょう。 私の意見では、これはクールです。

コントローラーの分離技術は、CRUDアクションという唯一の欠点にもかかわらず、コントローラー自体がより強力になると改善されます。 この場合の対処方法 このコードは、そのためのコントローラーで作成するだけです。

単純な組織を作るとき、私は驚かないのが好きです。 私は、コードの統一性、統一されたコード、および強力な「構成より規約」が好きです。これが、他のRubyフレームワークよりもRailsを好む理由の1つです。 すべてが同じように編成されているため、日常的な意思決定に費やす時間を短縮し、ビジネスにとって本当に重要な分野で成功することができます。



理論的には、これは、あるコードから別のコードに移動し、非常に短い時間で100％の生産性を実現できることも意味します。 荒野では、人々は「ひどいRailsアプリケーション」を理解するようになります。 ある会社はオブザーバー（私のお茶ではありません）などの建築パターンを使用でき、別の会社はTrailblazer（私のお茶ではありませんが、これについて興味深いアイデアがあります）などの追加のアーキテクチャを使用できます。 4番目は一般的に独自のものなどです。

RESTについて考えるようになります。



RESTは、このアーキテクチャスタイルが統一されておりシンプルであるため、多くの人々に愛されています。 RESTfulを理解する（実際に理解する）と、もう一方を理解しやすくなります。



理論的には、少なくとも：全員の認証？ ;-)ビジネスロジックはアプリケーション間で明らかに異なるため、これを理解する必要がありますが、ロジックはどこでも似ていることをどのように理解しますか。 つまり、Stripeで費用を作成し（誰かのお金を取る）、TwilioでSMSを作成し（つまり送信）、Githubでストレージを取得します。



アクションの代わりに名詞を使用して、RESTを使用するために最初に頭を少し緊張させる必要があります。「支払う」ではなく、「支払いを作成する」ではなく「残高に資金を追加する」ではなく、バランスシートの資本（「バランスで資金を作成する」）など。 少し奇妙かもしれませんが、SOAP、WSDL、その他のものに戻るよりも毎週この価格を支払うことになります（元Java / JEE開発者は私が話していることを知っています）。



さらに、インターフェイスのすべてのビジネスロジック（必ずしも実装である必要はありません）がRESTによって決定されます。RESTは、より簡潔でシンプルなビジネスロジック用に作成されたものであり、 。 それまでは、RESTで何でも表現できることを知っています。それは明らかです。 これにより、制限がなくなります。



CRUDアクションのみを使用して共有コントローラーを表すRESTful Railsルートの例を次に示します。 



  resources :purchases, only: :create resources :costs_calculations, only: :create namespace :company do resource :account_details, only: :update resource :website_details, only: :update resource :contact_details, only: :update end namespace :balance do resources :funds, only: :create end resource :bank_account, only: :update


REST設計を改善するために（特に子リソースが接続されている場合）、実装を気にせずに、タイムスタンプ（POST /残高/資金）の先頭にアクション+リソースを記述するときに、通常RESTを省略します。 それから、名前が割り当てられたとき、私は落ち着いています。 RailsはRESTを適切にサポートしているため、すべてをRailsルートに変換します。これは非常に便利です。
