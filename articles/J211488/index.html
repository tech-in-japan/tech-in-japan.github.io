大胆なプロジェクト向けの柔軟なCMS

この記事では、信じられないほど柔軟なコンテンツ管理システム-強力でシンプルなBooliveについて説明します。



Booliveサイト作成および管理システムは、最高のモジュール性、汎用性、および柔軟性を兼ね備えています。 プログラマーは、日常的な操作を取り除き、有用なプログラミングに集中します。 管理セクションを使用すると、データ構造の作成、作業ルールの構成、要件に厳密に従ってサイトの作成を行うことができます。 何をするにしても、すべてを再利用できます。 マネージャー、管理者、編集者、サイトプログラマーにとって、システムはすべて、直感的で美しいインターフェイスを備えたシンプルなものにしようとします。



柔軟性の基盤-2つの階層



Booliveプロジェクトのアイデアは、実際のアプリケーションからさまざまな人格との魅力的な議論まで、実験で数年にわたって明確なアウトラインを引き継ぎました。



主な要望は、あらゆる目的でデータ構造とロジックを作成する自由です。 システムは、セキュリティや整合性に関連しない人為的な制限を課すべきではありませんが、創造性を制限することなく、簡単かつ柔軟に作成することができます。



階層は基礎として採用されます。 サイト全体は、オブジェクトの1つの大きな階層です。 どのくらい大きいかは、サブジェクトエリアの詳細レベルと詳細の適切性によって異なります。 より詳細に、より多くの柔軟性。 階層では、サイトに関するすべての情報、そのすべてのオブジェクト（コンテンツ、インターフェース、ユーザー）。 すべてに対して単一のアプローチがあります。







オブジェクトは、データとロジックの組み合わせです。 各オブジェクトには、独自の名前、値、およびロジックがあります。 オブジェクト自体はプリミティブです。 ただし、階層を形成する従属オブジェクトを持つことができます。 従属オブジェクトは、親オブジェクトのプロパティとして、または単にネストされた構造を形成するために使用されます。







オブジェクトは階層構造になっているため、アドレス（URI）があります。 実際にはファイルパスとして。 このアドレスは、ブラウザの行を含むオブジェクトへのアクセスに使用されます。 アドレス-オブジェクトのメイン識別子。



新しいオブジェクトを作成する自由と容易さは、以前に作成されたオブジェクトの複製につながります。 複数の部下で構成される複雑なオブジェクトを作成する同じ操作を繰り返すのは退屈です。 手動で、同じタイプのオブジェクトの構造、命名、ロジックの統一を観察する必要があります。 プロトタイピングは、この退屈な複雑さを解決するために使用されます。 これがシステムの主な機能です。 新しいオブジェクトは、既存のオブジェクトに基づいて作成されます。 つまり、あるオブジェクトは別のオブジェクトの意味、ロジック、構造を継承します。 同時に、その意味、ロジックを再定義し、新しい部下を持つか、継承された部下を削除することができます。 プロトタイピングは、複雑なオブジェクトの作成を最適化し、再利用を可能にします。 さらに、更新のフィードバックを維持しながら、他のタスク向けの既存のオブジェクトを損なうことなく、オブジェクトをタスクに合わせて変更できます。 オブジェクトはプロトタイプと連絡を取り合っています。







各オブジェクトは、構造の階層と継承の階層の2つの階層に同時にあります。 わずか2つの階層により、複雑なデータ構造を作成するための驚くべき柔軟性が提供されます。 たとえば、記事をユーザーに関連付けるために、あるオブジェクトを別のオブジェクトに接続する必要がある場合、プロトタイピングが使用されます。 新しいオブジェクトが送信に追加され、そのプロトタイプが通信が必要なオブジェクトになります。







既存のオブジェクトに基づいて新しいオブジェクトを作成する機能は、オブジェクトのライブラリの存在を示唆しています。 これは、オブジェクトが再利用のために保存されるサイト階層のブランチです。 任意のオブジェクトをどこにでも再適用できますが、ライブラリは、そこからサイトを作成するためのさまざまな空白と完全に既製のオブジェクトのコレクションです。 特定のプロジェクトのオブジェクトを使用して、独自のライブラリまたはパッケージを作成できます。 出版、製品、ユーザー、ウィジェット、既製のサイトなどの標準。



ロジックは一意ではありませんが、柔軟なデータ構造の一部です。 各オブジェクトは、独自の独自のロジックを持つことも、プロトタイプロジックを継承することもできます。 誰もが基本的なロジックを持っています-チェック、保存、部下などへのアクセスの機能。 その他のオブジェクトは、そのロジックによって補完されます。 たとえば、ウィジェットには表示機能があり、画像にはズーム機能があります。

異常な管理領域


管理セクション（管理者パネル）の詳細を確認します。 管理パネルの主なタスクは、ユーザー権限を考慮してサイトオブジェクトへのアクセスを提供し、オブジェクトを使用して必要な操作（作成、変更、削除）を実行することです。 ポイントは、サイトでの作業を最適化するための、便利なグラフィカルインターフェイスの管理パネルです。 経験の浅いユーザーには、使い慣れたインターフェイス要素を表示する必要があります。 上級者向けには、システムの完全な柔軟性を明らかにしてください。







管理パネルでの作業は、目的のオブジェクトへの遷移から始まります。たとえば、その後の編集のためにページに遷移します。 中央の領域には、現在のオブジェクトとその下位オブジェクトが表示されます。その中に入ることができます。 上部には、現在のオブジェクトのパス（アドレス）のメニューがあり、すばやく戻ることができます。 左側の部分には、頻繁に使用するオブジェクトにすばやく移行するためのブックマークが表示されます。 ブックマークはいつでも任意のオブジェクトに追加できます。



1つの画面に配置するオブジェクトに関する情報は多数あるため、一番下のメニューでは、オブジェクトが正確に関心のあるものを選択します。 ほとんどの場合、これはオブジェクトまたはそのプロパティの構造です。 上級ユーザーにとっては、オブジェクトのロジック、値、およびオブジェクトのプロトタイプと相続人を表示するのが役立ちます。 オブジェクトにアクセスすると、デフォルトで構造が表示されます。



ナビゲーションの3番目のタイプは、オブジェクトに対して実行される機能です。 機能メニューは、管理パネルの右側にあります。 主な機能は、オブジェクトの表示、新しいオブジェクトの追加、記号の削除と変更です。 使用可能な機能は、選択したオブジェクト、選択した情報、および管理パネルのその他の条件（アクセス権など、リストで同時に選択されているオブジェクトの数など）によって異なります。 機能セットは簡単に拡張可能です。 ブラウザ、オブジェクトエディタを追加して、特定のタイプのオブジェクトまたはその他の作業条件に割り当てることができます。 彼らの助けを借りて、管理パネルには、特定のプロジェクトの利便性の下で強化されたより具体的な機能が備わっています。 たとえば、オペレーティングシステムとしての管理セクション、ファイルとしてのサイトオブジェクト、およびプログラムとしての機能などを類推できます。



既存のページを編集する必要がある場合は、コンテンツに移動し、そこでページを見つけて、そこに移動して編集することがわかります。 簡単なはずです！



コンテンツ-プロジェクトサブジェクトエリア


コンテンツには、サイトで公開されることになっているすべてのものが含まれます。 例：記事、ニュース、セクション、製品、写真、またはプロジェクト固有のコンテンツタイプ。 通常、ページとセクションを扱います。 サイト階層には、出版物を保存するための「コンテンツ」と呼ばれる特別なセクションがあります。 ただし、これはサイトページのみを保存できるという意味ではありません。 このセクションは、階層全体を検索しないように、利便性と最適化のためだけに事前に作成されています。



新しいページを追加するには、最初に「コンテンツ」セクションに移動する必要があります。 新しいページを追加する必要がある場合は、ページまたはサブセクションに移動できます。 右側のメニューに移動した後、「追加」機能を選択します。 クイックピックリストにはすでにページが提供されています。 ただし、他のサイトオブジェクトを選択できます。 ページを追加すると、「ドラフト」という記号が表示されます。 最初にページを編集することをお勧めします。公開するためにページを完全に準備してから、ドラフトサインを削除してください。







柔軟性を感じてください。 別のテキストプロパティをページに追加します。 たとえば、ページエディタで、パッケージ「シンプル」オブジェクト「HTMLテキスト」から「ライブラリから」を追加します。 [追加]をクリックすると、オブジェクトを選択するためのウィンドウが表示されます。 すべてのサイトオブジェクトを選択できますが、ライブラリにあるテキストが必要です。







新しいテキストがエディターに表示され、その順序に従ってサイトページに表示されます。







この例では、追加のプロパティ「text」が1ページのみに追加されます。 サイトのすべてのページに追加のプロパティが必要な場合は、プロトタイプを変更するか、独自のプロトタイプを作成して使用する方が簡単です。



プロジェクトの場合、ライブラリにパッケージを作成し、プロジェクトの詳細を使用してプロトタイプを収集することをお勧めします。 たとえば、カーサロンWebサイトの「車」オブジェクトを作成するには、車のすべての必要なプロパティをすぐに決定し、その前にプロトタイプ「ボディタイプ」、「メーカー」などを作成して車自体で使用します。 このようにして、プロジェクトのサブジェクト領域のプロトタイプが作成されます。 「コンテンツ」セクションでは、客室の車、メーカーのリストなどがすでに直接追加されています。 「車」には、単純なプロパティ（「テキスト」、「数字」）だけでなく、複雑なプロパティ（任意のサイトオブジェクト）を追加して、複雑な構造を形成できることに注意してください。 必要とマークされたプロパティは、すぐに新しいオブジェクトに表示されます。 プロパティがオプションの場合、補足として利用できます。 必要に応じて後で追加できます。



ウィジェットはすべてを表示します


ブラウザーでサイトを表示するには、ウィジェットが使用されます。 ウィジェットは、HTMLコードを返す表示機能を持つオブジェクトです。 ウィジェット自体は、何か、いくつかの静的な情報を表示したり、デザイン要素になったり、ニュースなどの特定のオブジェクトを表示したり、自身の内部に下位のウィジェットを表示したりできます。



階層ブランチは、「インターフェース」セクションのウィジェットから構築されます。 ウィジェットは、ブラウザでのサイトのプレゼンテーションとダイナミクスを完全に決定します。 ルートウィジェットには、HTMLドキュメントの基本構造が表示されます。 下位ウィジェットの表示結果は本文領域に挿入され、ヘッダー、サイドパネル、地下室、中央などのサイトのレイアウトを指定します。 下位のウィジェットが各領域に挿入されるなど。 これは典型的な構造であり、さまざまなオプションが可能です。1つのウィジェットでサイト全体を表示できますが、詳細に柔軟性があることに注意してください。







ウィジェットには動作条件の機能があり、trueを返す場合、ウィジェットは動作しますが、そうでない場合は動作しません。 この機能は、受信データ、設定、またはシステムステータスをチェックします。 ウィジェットは、特定のアドレスで機能するように構成できます。たとえば、アドレスが/ adminで始まる場合、管理パネルは機能します。 下書きにあるか、アクセスできない場合、ウィジェットは機能しません。 夜間のみの作業、表示されたオブジェクトの種類の確認、クエリパラメータの確認など、あらゆる条件をプログラムできます。 ほとんどの条件は、着信データのルールによって設定できます。



管理パネルでウィジェットを操作する場合、「ロジック」モードと「値」モードは、それぞれウィジェットとそのテンプレートのロジックを編集するのに役立ちます。 ロジックは、すべてのオブジェクトと同様に、phpクラスを持つファイルです。 テンプレートはファイル形式のウィジェットの値であり、テンプレートのタイプはファイル拡張子によって異なります。 ロジックとテンプレートもお気に入りのIDEで編集できます。 Booliveシステム自体がファイル作成のルーチン操作を実行します。オブジェクトが独自のロジックまたは独自のテンプレートを必要とすることを管理パネルで示すだけです。 クラス自体のスタブを持つクラスファイルが自動的に作成されます。 関数に新しいロジックを追加したり、他の関数を再定義したりするだけです。







ウェブサイトのインターフェースを作成するには、ライブラリから既製のウィジェットを使用する方が簡単です。テンプレート、ロジック、スタイルをわずかに変更します。 ライブラリにはナビゲーションウィジェット、コンテンツウィジェット、スライダー、フォームがあり、新しいウィジェットが表示されます。 プロトタイピングのおかげで、ウィジェットをサイトのさまざまな領域に追加でき、各インスタンスを独自の方法でカスタマイズできます。 カスタマイズするだけでなく、ロジック、テンプレート、スタイル、スクリプトを変更します。 ライブラリには、既製のレイアウト、実際には既製のサイトもあります。 ほとんどのCMSテーマで「標準」レイアウト用に選択されている場合、Booliveで1つの選択肢を選択すると、最も単純なサイトがポータルになります。 ライブラリから既製のレイアウトを基礎として使用すると、元のレイアウトを損傷することなくレイアウトを調整できます。



汎用性が必要な場合は、ユニバーサルウィジェットを使用する必要があります。 これらの1つは、サイトの中央エリアにあるコンテンツウィジェットです。 提出物のコンテンツウィジェットには、特定のオブジェクト（ページ、セクション、製品、ユーザーのウィジェット）用のさまざまなウィジェットがあります。 各ウィジェットは、そのうちの1つが機能するまで順番に起動します。 全員に作業条件があります-表示されたオブジェクトが特定のオブジェクトを継承しているかどうかを確認します。 その結果、ページはページウィジェットとして、セクションはセクションウィジェットとして表示されます。 特定のページに固有の表示を作成できます。



オートメーションは、オブジェクトのプロパティの表示にも適用できます。 例は、ページ、セクション、メニュー、スライダー、および特に管理パネルのその他の多くのウィジェットです。 これらのウィジェットは、オブジェクトのプロパティを選択し、各プロパティのリストからウィジェットを選択します。



汎用性があるため、任意のオブジェクトをウィジェットに挿入することもできます。 これには実用的なアプリケーションがあります-フィードバックフォーム、承認、計算機の作成。 まず、「Calculator」ウィジェットを作成してから、ページのテキストに挿入します。 汎用性のもう1つの興味深い例は、スライダー（カルーセル）での写真の表示です。 写真の代わりに、ページをスライドリストに追加できます。 最も重要なことは、これはハックではなく、スライドでページを編集する利便性を維持することです。



リクエストの処理


Booliveシステムには、index.phpファイルという単一のエントリポイントがあります。 すべてのリクエストの処理を開始します-サイトが開始します。 サイトのルートオブジェクトが選択され、そのstart（）関数が呼び出され、その関数がインターフェイスオブジェクトを起動し、管理パネルに入ると表示されます。 インターフェースは、すべての下位を順番に開始します。 要求の処理方法は、インターフェイスの構造によって異なります。







インターフェースには、ウィジェットの階層（ウィジェットHTMLマークアップ）があることがすでにわかっています。 ほとんどのリクエストを処理し、HTMLドキュメントをクライアントに返すのは彼女です。 ウィジェットに加えて、インターフェースには他のハンドラーがあり、各ハンドラーは独自の作業条件に従って起動されます。 たとえば、リクエストヘッダーの応答形式がJSONの場合、RESTfulハンドラーは機能します。 フォームハンドラーは、要求を受け入れるフォームのアドレスパラメーターが指定されている場合に機能します。 外部システムとの統合などのために、独自のハンドラーを簡単に追加できます。 インターフェースはすべての部下を順番に起動しますが、全員が独自の作業条件を持っているため、実行されるのは1つだけです。 さらに、プロセスはハンドラー自体に依存します。







ユーザーおよびグループの権利の柔軟なカスタマイズ


データモデルの柔軟性は、ユーザーの構造化にも拡張されています。 ユーザーグループ、グループ内のグループを作成し、それらにユーザーを追加できます。 サイトの独自の機能に突然必要になる場合、システムはユーザーが別のユーザーを追加することを禁止しません。 当然、ユーザーは自分のプロファイルに任意のプロパティを持つことができます。 サイト管理者は、ユーザーを禁止または許可する対象を決定する必要があります。



ユーザーとグループには「許可」プロパティがあります。







ロールは、権利を構成するために使用されます。 読み取り、書き込みなどのアクションのオブジェクトがロールに追加されます。 独自のアクションを追加できます-これは、アクションの名前に対応する名前を持つ通常のオブジェクトです。 アクションはネストされたアクションを持つことができます。たとえば、オブジェクトの値、オブジェクトのロジックの変更、オブジェクトの機能の実行などです。 記録を禁止すると、オブジェクトを作成または変更することはできません。 条件オブジェクトがアクションに追加されます。 それらのタスクは、対応するアクションを実行できる、または実行できない条件を返すことです。 現在、「入場」と「禁止」の2種類の条件が使用されています。入場によって、アクションを実行する権利があるオブジェクトと、許可されないオブジェクトの禁止が決定されます。 何でも条件になる可能性があるため、権利を定義する柔軟性が集中するのは条件です。 許容値オブジェクトに従属するオブジェクトの識別子のリストに含まれているかどうかにかかわらず、オブジェクトの識別子を簡単に確認できます。 継承、親、部下、作成者を確認し、オブジェクトの値などの属性を比較するだけです。 通常、オブジェクトに触れることはできませんが、時間など、システムの状態によって許容値を決定します。 一般に、条件オブジェクトはデータベースクエリ言語で条件を返します（以下について）

ユーザーの権限は、ユーザーがメンバーとなっているすべてのグループの権限によって補完されます。 権限の結合は、論理演算「AND」、「OR」による条件の結合であるため、ネストされたグループまたはユーザーの権限が親グループより優先されます。 ルートグループはすべてを禁止するように設定でき、ネストされたグループまたはユーザーは、何を許可するかを指定できます。 反対に、最初はすべてを許可し、次にポイントワイズを禁止することは許可されます。

プログラムコードレベルでオブジェクトを操作するとき-オブジェクトを保存、削除、または検索するとき、オブジェクトが実際に格納されている場所、DBMSを気にしません。 オブジェクトの操作は、DBMSから完全に抽象化されています。 データモジュールに目を向けると、オブジェクトがどのストレージにあるべきかを構成によってすでに理解しており、対応するストレージのモジュール、たとえばMySQLStrore.phpを参照しています。 さらに、いくつかのストレージを使用して、さまざまなタイプのサイトオブジェクトを格納できます。 ストレージはオブジェクトURIによって決定されます。



外部ストレージHTTPStrore.phpで興味深い実験を行いました。 オブジェクトのURIが「http：」で始まる場合、機能します。 別のサイトへのHTTP RESTリクエストが実行され、サイトがBoolive 2にもある場合、目的の結果が得られます。 はい、システムはサイト間でオブジェクトのプロトタイプを作成できます！



データの操作は特定のDBMSから抽象化されており、クエリクエリでさえネットワーク上を通過できるため、SQLを使用することはできません。 一方で、理解可能で、プロトタイプデータモデルの特性を満たすクエリ言語が必要です。また、一方で、URLまたはPHP配列の文字列として表現でき、フォーマット間ですばやく変換できるように簡単に構造化できます。



配列のクエリ


ネストされた配列では、クエリは少し肥大しているように見えますが、プログラムによる処理では非常に単純です。 要求は、選択する場所、選択の深さ、選択する対象、ソート、制限、およびその他のパラメーターを示します。 サンプリング条件も構造化されています。 



 $cond = array( 'from' => '/Contents/news', //    /Contents/news 'depth' => 3, //    from. 'select' => 'children', //  ? self, children, tree, heirs, parents, protos 'where' => array( //     AND array('attr', 'uri', '=', '?'), //   array('not', array( //    array('attr', 'value', '=', '%?%') )), array('any', array( // ,   OR array('child', 'title', array( //     title array('attr', 'value', '>', 10), array('attr', 'value', '<', 100), )), array('is', '/Library/basic/Number') //   ?   )) ), 'order' => array( //  array('uri', 'DESC'), //   uri array('childname', 'value', 'ASC') //   value    childname ), 'limit' => array(10, 15), //  -   10-   15  'key' => 'name', // ,       );


URL形式のリクエスト


URL形式では、fromパラメーターが短縮されます。その結果、リソースを含む本格的なURLが取得されます。 このオプションはRESTfulを完全に満たし、複雑な選択のためにそのAPIを思いつく必要はありませんが、制限についても考慮する必要があります。 



 $cond = " /Contents/news& depth=max& select=children& where=all( attr(uri,like,%25%25) not(attr(value,eq,m)) any( child(title,all( attr(value,gte,10) attr(value,lt,100) )) is(/Library/basic/Number) ) )& order=(uri,desc),(childname,value,asc)& limit=10,15 ";


次のように機能します。 



 $result = Data::read($cond); //   $result = Data::read("/Contents/news/news1"); //


クエリ言語は、ユーザーアクセス権の条件、ウィジェットの動作条件、およびその他の条件を記述します。 データベースにアクセスせずに、以前に選択したオブジェクトがプログラムで目的の条件に一致するかどうかを確認できます。 また、リポジトリの場合、構造化された条件をSQLなどの言語に変換することは難しくありません。



チーム内で競合はありません。 ビジネス向けのバージョン


システムまたはサイトオブジェクト（読み取りモジュール）をインストールする場合、SQLダンプは使用されません。 すべてのデータを含むサイト全体が* .info JSON形式ファイルにエクスポートされます。 これらのファイルはインストール中に使用されます。 オブジェクトをエクスポートおよび設定する機能は、管理パネルで使用できます。 いつでも、サイト全体をエクスポートし、そのファイルを別のサーバーに転送してインストールできます。 これは、典型的なプロジェクト用にスタジオで配布または使用するためのシステムの特別なバージョンを準備するだけでなく、モジュールを共有するのにも便利です。



infoファイルを使用すると、Booliveシステムのサイトまたはチーム開発で作業する際に、さまざまな開発者のデータベースをマージする問題が解決します。 さらに、情報ファイルの形式のデータはバージョン管理システムに分類され、異なるバージョンのデータを結合する際にそのヘルプの競合が解決されます。







オブジェクトの値がファイルであるか、オブジェクトに独自のロジックがある場合、またはオブジェクトが情報ファイルにエクスポートされた場合、ファイルシステムでオブジェクトの下にフォルダーが見つかります。 フォルダーへのパスはオブジェクトURIと一致します。 ファイルシステムのフォルダー構造は、サイトのデータ構造と同じです。 ただし、サイト全体が情報ファイルにエクスポートされた場合でも、各オブジェクトに独自のフォルダーを用意する必要はありません。 プロパティオブジェクトは情報ファイルを作成せず、親の情報ファイルにコンパイルされるためです。



そのため、システムをインストールするには、すべてのファイルをサーバーに転送し、ブラウザーを介してサイトにアクセスする必要があります。 config.php構成ファイルには、システムを再インストールする場合に備えてIS_INSTALL定数が含まれています。
