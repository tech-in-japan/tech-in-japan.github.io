MSWinでのサービスの実装

操作上のニーズのために、Microsoft Windows用のシステムサービスを作成する必要がある場合があります。



Habréには既に「Windowsサービスを作成する」という記事がありますが、私の意見では、この記事はMSDNに掲載されている簡単な概要にすぎません。 たとえば、エラーが発生した場合のサービスの動作、またはメッセージログへの書き込みについて考えられるオプションは考慮されません。

このようなアプリケーションを作成した経験を使用して、可能な限り最大の情報量を述べようとします。



サービスがデバイスドライバーとして機能する場合（したがって、拡張子が.sysのファイルにある場合）、オプションは考慮されません。あまりにも特殊です。



通常、サービスは単にSCM （サービスコントロールマネージャー）によって起動され、それによって制御されるアプリケーションです。 プログラマーにとって、これは、開始するためにいくつかの追加アクションを提供する必要があることを意味します。

さらに、正しいサービスがユーザーと直接対話しないようにする必要があります。 これは実際にはどういう意味ですか？ アプリケーションを起動すると、このアプリケーションのコンソールウィンドウまたはGUIが表示されます。 Windows 2000 / Windows XP / Windows 2003 Serverオペレーティングシステムを実行するシステムサービスも許可されます。このサービスはインタラクティブと呼ばれます。 しかし、同時に、 Windows Vistaから開始-インタラクティブサービスの禁止。 サービスを開発するとき、対話性を放棄する方が正しいことがわかります。

同時に、MSDN自体には、ユーザーインタラクション用の次のオプションが用意されています。

1） WTSSendMessage（）関数を使用してダイアログを表示する

動作は非常に柔軟性があり、ダイアログを表示してユーザーの反応を待つことができます。ユーザーに通知してアプリケーションを続行することなどができますが、サービスはターミナルセッションで動作するか、現在のターミナルセッションでアクティブなユーザーに依存する必要がありますポップアップウィンドウに応答するための知識と権利（ITに精通していない多くの人々がこの動作を恐れています。これは設計時にも考慮する必要があります）

2） CreateProcessAsUser（）を使用して現在のユーザーセッションで別のアプリケーションを作成する

ユーザーにサービスと対話するためのシンプルなツールが提供されるだけであるが、ソケット（最も一般的なオプション）、またはIPC、COMなどの一般的なコードを多少複雑にする対話を整理する必要があることを理解する必要がある場合、最も正しいオプションしかし、サービス自体を簡素化します。

個別の実行可能ファイルを作成せずに、特定のキーで起動された同じサービスファイルを使用することもできます。これにより、プログラムの配布と更新が多少簡単になります。

3）パラメータMB_SERVICE_NOTIFICATIONを使用してMessageBox（）を呼び出します

最悪のシナリオは、ユーザーの応答の間、サービス（またはそれを引き起こしたスレッド）が「ハング」することです。



システムにサービスをインストールまたはアンインストールするときは、管理者、またはそれと同等の権限を持つユーザーのみがSCMと対話する権限を持っていることも考慮する必要があります。



さらに、サービスの作業フォルダーはシステムのSystem32であり、サービスがインストールされているフォルダーに書き込む権限がない可能性があります。そのため、システム自体のイベントログをログに使用する必要があります（以下を参照）。



コードを引用する前に、NTシステム用のサービスを開発する場合、マクロで展開されることを信頼せずに完全な関数名を個人的に作成することを好むため、 OpenSCManagerの代わりに、通常OpenSCManagerWを記述します 。

サービスエントリポイントは、コンソール関数main（）またはWinMain（）のいずれかです。 イディオロギー的にserisisはコンソールプロジェクトではないため、2番目の関数を使用します。

関数自体では、コマンドラインパーサーを作成し、 install 、 uninstall 、 runおよびstop コマンドの処理を提供する必要があります。必要に応じて実行できます。 パラメーターなしの実行可能ファイルの起動は、SCMを介したサービスの起動として検討します。 これが最も正しい動作です。 誰かがキーなしでファイルを実行しようとすると、サービスは開始されません。必要なコマンドを処理することにより、コンソールのコマンドとパラメーターを覚えることなく、サービスの動作を簡単に制御できます。

ここで何が行われたか見てみましょう：

1）get_path（）/ get_name（）は、サービス実行可能ファイルが置かれているフォルダーの名前とこの実行可能ファイルの名前を取得する機能です。 Samopisnyeので、実装は任意です。

2）SERVICE_WIN32_OWN_PROCESSフラグの値は、ほとんどのサービスに適しています。 例外は、デバイスドライバーおよびファイルシステムとは見なされません。 この記事のフレームワークで2番目に可能なフラグはSERVICE_WIN32_SHARE_PROCESSフラグです。これは、サービスプロセスが他のサービスと共通のメモリ領域を持っていることを意味します。1つの実行可能ファイルに複数のサービスがある場合、これは必須です。 一般的なメモリ領域を使用すると、RAMを節約してサービスのやり取りを簡素化できますが、デバッグが非常に複雑になります。したがって、この機能を使用する理由は、必要な理由とその他の理由を明確に理解することだけです。

3）SERVICE_AUTO_START-システムの起動時にサービスが自動的に開始されます。 この場合、サービスが開始される瞬間は、システムサービスの初期化、つまり ユーザーがログインする前。 サービスがサーバーで実行されている場合、この点を考慮する必要があります。 フレームワーク内のその他の可能なオプション、SERVICE_DEMAND_START-手動で開始、およびSERVICE_DISABLED-まったく開始しません。

興味深い機能は、メッセージを正しい形式で表示するには、Eventlogサービスの再起動が必要なことです。 それ以外の場合は、次のようなものが表示されます。

良い機会は、サービスが独自のメッセージログを保持でき、一般的なアプリケーションに追加できないことです。 これを行うには、メッセージ記述ライブラリを登録するときに、 HKEY_LOCAL_MACHINE \ SYSTEM \ CurrentControlSet \ Services \ EventLog \ Applicationブランチではなく、 HKEY_LOCAL_MACHINE \ SYSTEM \ CurrentControlSet \ Services \ EventLog \ MyServiceを指定する必要があります。

しかし、興味深いニュアンスがあります。

MMCでレジストリブランチを作成すると、対応するフォルダーが「イベント」に表示されますが、場合によっては、イベントログを再起動した後でもこのブランチからのメッセージが正しく処理されません。 このような「重大なケース」では、コンピューターの再起動が必要になりますが、これはサーバーにとって問題となる可能性があります。 しかし、幸いなことに、これは非常にまれです。
