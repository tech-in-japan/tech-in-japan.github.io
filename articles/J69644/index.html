MS VS 2008での.Net Frameworkソースコードのデバッグ

を含むいくつかの記事の翻訳 Shawn Burke（ ありがとう！ ）、ScottGu（ ありがとう！ ）、Paul Krill（ ありがとう！ ）、John Robbins（ 最初のすばらしい感謝！ 2番目のすばらしい感謝！ ）



ここで目指している最終結果は、.Net Frameworkメソッドの呼び出し内でIDEがデバッグできる機能です。 この投稿に必要なものがすべて含まれていることを願っています。 過去数週間にわたって、多くの実験を行い、この問題に関する多くのデータを収集しました。 そして、経験をコミュニティと共有したいと考えています。







方法＃1：.NET FrameworkソースコードをデバッグするためのVisual Studioの構成（Visual Studio 2008のみ。理論的には-2010）



問題がある場合は、開始するためのすべての手順を完了していることを確認してください。 それは私たちと一緒に書かれているとおりです。 他のすべてが失敗した場合は、ドキュメントの最後にあるFAQセクションをご覧ください。 これがうまくいかない場合は、コメントを書いてください。すべて一緒に解決します。



基本設定


まず、MS VS 2008 Express Editionがインストールされている場合、これは機能しません。



1） Visual Studio QFEをインストールします。 この修正プログラムは、ソースファイルを選択するVisual Studioデバッガーの一部であるDLLを更新するだけです。 ダウンロードページで修正に関する詳細をすべてお読みください。



更新プログラムのインストールエラーメッセージが表示された場合は、VS 2008 DVDを挿入して、EXEを再度実行してください。 これにより、修正プログラムを正しくインストールできる場合があります。



2） Visual Studioを起動し、[ツール]> [オプション]> [デバッグ]> [全般]を選択します。 Visual Basicプロファイルで作業している場合は、続行する前に、[オプションダイアログ]ダイアログの下部にある[すべての設定を表示]チェックボックスをオンにする必要があります。



次の設定を設定します。

  • 「Just Meコードを有効にする（管理のみ）」をオフにする
  • 「ソースサーバーのサポートを有効にする」を有効にします


すべてが次のようになります。







3）次に、プロパティツリーから[デバッグ]> [シンボル]を選択します。 ダウンロードするシンボルのソースと、Visual Studioがそれらをキャッシュするハードドライブ上の場所を設定します。

シンボルファイル（.pdb）の場所にアドレスを追加します： http : //referencesource.microsoft.com/symbols

キャッシュの場所を設定します。 この場所に読み取りおよび書き込み権限があることを確認してください。 素晴らしいオプションは、ドキュメントの場所です。 たとえば、C：\ Users \ Stanislav \ VsSymbols。

チェックボックスをオンにします：シンボルが手動で読み込まれた場合のみ、上記の場所を検索します。



これでインストールは完了です！



ソースコードフレームワーク内でのデバッグ


これを行うには、簡単な例を取り上げます。 空のC＃Windowsアプリケーションプロジェクトを作成します。 エントリポイントをForm_Loadに設定します。







ソリューションのデバッグを開始します。 プログラムがブレークポイントで停止したら、Call Stackウィンドウに移動します。 呼び出しスタックで、 System.Windows.Forms.dllで始まる行を右クリックし、[シンボルの読み込み]を選択します。 良いインターネットチャンネルがあればいいのですが。 それ以外の場合は、事前にロールとお茶を用意してください。 ダウンロードするファイルの重量は約10メガバイトです。 Visual Studioは常に応答しないことに注意してください（カルマでマイナスします。ダウンロードインジケーターが良いトーンです。ハングした場合、プロセスを強制終了します。ハングしない場合、強制終了します-残念です）。 ただし、このようなダウンロードは1回行われ、上記で指定したキャッシュのあるディレクトリのコンピューターに保存されます。







これらのアクションにより、北からDLLから文字をロードでき、 System.Windows.Forms.dll内の呼び出しを示すすべての行が黒になります。 手頃な価格。 行番号も利用可能になります。 キャラクターを見る必要があるたびに、右クリックしてシンボルのロードを選択する必要があることに注意してください。







そのため、この瞬間からSystem.Windows.Forms.dllのすべてのシンボルが読み込まれ、そのコードを見ることができます。 コードの通常のデバッグ時とまったく同じ方法でコードを表示できます。 これを行うには、通常どおり、CallStack行をダブルクリックするか、 （F11）を使用してアセンブリメソッド内に入ります。 コードを初めて表示しようとすると、ライセンスが提供され、それに基づいてコードを読むことができます。同意する場合は、[同意する]をクリックします。その後、ソースコードがダウンロードされます。







ここで、デバッグするすべてのアセンブリについて、上記のすべての手順を繰り返します。 コールスタックにないアセンブリをデバッグする必要がある場合は、[モジュール]ウィンドウを開き、そこから文字を読み込みます。











上記の例では、F11でDrawRectangleコードに移動できます。



上級ユーザー向け


通常、Visual Studioはデバッグセッションごとに、デバッグされたプロセスに読み込まれる各DLLの文字を読み込もうとします。 シンボルに関する情報を見つけるには、[オプション]> [デバッグ]> [シンボル]で指定されたすべてのパスをスキャンします。 ただし、文字に関する情報がないDLLを多く使用するプロジェクトがあります。 これらの場合、デバッグ開始プロセスは非常に長くなります。 これが、ユーザーの要求に応じてキャラクターの読み込みを使用することをお勧めする主な理由です。



ただし、文字を自動的にロードする方法があり（これにより「シンボルのロード」ステップが本質的に不要になります）、全体的なパフォーマンスが向上します。 このチェックボックスは、このダイアログに頻繁に戻る必要があるため、システムの上級ユーザーにのみ意味があります。 ちなみに、すばやく入力するには、上の画像のコンテキストメニューで[シンボル設定...]を選択します。



これの基本は、すべてのキャラクターを取得し、ダウンロードしてローカルに保存することです。 これを行うには、「上記の場所を検索...」の隣のボックスをオフにします。



その後、デバッガでプロジェクトを実行します。 はい、ところで、すみません、言うのを完全に忘れました。 お茶とパンに我慢してください。 接続に応じて、スタジオは長時間または非常に長い時間ダウンロードします。 合計で、約50メガバイトをダウンロードする必要があります。 そして以来 これはすべてキャッシュされ、その後は1回限りです。







このプロセスが完了したら、デバッガーを停止し、サーバーアドレスを持つソースを削除して、[OK]をクリックします。







これらの手順の後、すべてのシンボルが自動的にロードされ、早期の制限なしにコールスタックに介入できます。 ロードされたキャラクターに関する情報を含まない別のプロジェクトで作業している場合は、これらすべてのステップを繰り返してください;）



よくある質問


1）[シンボルのロード]を選択すると、[ファイルを開く]ダイアログボックスが表示されます。このダイアログボックスで、ポートは.pdbファイルの場所を示します。

これは、次の4つの状況で発生する可能性があります。

  1. 構成または構成していませんが、サーバーアドレスが正しくありません： referencesource.microsoft.com/symbols
  2. アカウントでVisual Studioがこのパスに沿ってファイルを読み書きできないように、キャッシュのディレクトリを選択しました
  3. 存在しないDLLのシンボルをロードしようとしています
  4. 別のバージョンの.NET Frameworkがインストールされている場合があります。 これは、たとえば、Microsoft Server 2008 Betaがインストールされている場合に発生する可能性があります。 これを確認するには、mscorlib.dllアセンブリのバージョン番号を確認し、2.0.50727.1433である必要があります。そうでない場合は、 .NET Framework 3.5のRTMバージョンをインストールしてください








また、 「C：\ Program Files \ Microsoft Visual Studio 9.0 \ Common7 \ IDE」 （またはVisual Studioをインストールした場所） symsrv.noというファイルを確認します。 このファイルが存在する場合、名前をsymsrv.yesに変更します 。



2）ソースコード内でデバッガーを入力しようとすると、「この場所ではソースを使用できません」というダイアログが表示されます。

まず、FAQのポイント（2）を参照してください。 すべてのキャラクターが正常にロードされたことを確認します。 これを確認するには、[モジュール]ウィンドウのシンボルのステータスの列を確認します。 文字がロードされている場合は、次を確認してください。

過去にMicrosoft Symbol Serverを構成しており、ソースコードを含まないこのDLLのシンボルをダウンロードした場合。 別のキャッシュ位置を指定するか、既存のキャッシュを削除してから、「シンボルの読み込み」コマンドを確認してください。 詳細については、FAQ（3）も参照してください。

「ツール」>「オプション」>「デバッグ」>「一般」ウィンドウで「ソースサーバーを有効にする」チェックボックスを有効にしていることをもう一度確認してください。

文字キャッシュを収集するように構成されたフォルダーに読み取りおよび書き込み権限があることを確認してください

_NT_SYMBOL_PATHがインストールされている場合、これらの設定はすべて上書きされます。 詳細はこちら



3）Microsoft Symbol Serverを使用してキャラクターをダウンロードします。 違いは何ですか？



Microsoft Symbol Serverは、情報を提供せずにシンボルを提供します。 この情報は公開前に削除されます。 参照ソースパスとMicrososft Symbol Serverの両方を使用する場合、これはリストの最初の参照ソースパスの場所です。







ただし、Microsoft Symbol Serverが_NT_SYMBOL_PATHを介して構成されている場合、_NT_SYMBOL_PATHがこれらの設定をオーバーライドすることに注意してください。



4）64ビットOSのサポートはありますか？



はい、64ビットバージョンのPDBが利用可能です。 さまざまなアーキテクチャで動作するDLLがあることに注意してください。 したがって、必要なPDBは1つだけです。



5）Frameworkコードにブレークポイントを設定するにはどうすればよいですか？



最初に、Visual StudioではコードがPDBファイルと完全に一致する必要があります。 多くの場合、PDBファイルは大幅に変更されません。 たとえば、著者は著作権の行を追加しました。 ただし、コードは簡単にデバッグできます。 ブレークポイントを設定し（不完全に塗りつぶされたドットが表示されます）、PDBファイルの場所を指定します（ブレークポイントからのコンテキストメニュー、場所の位置...）







次に、下のボックスをチェックします。







これらの手順の後、安全に作業を続行できます。



6）「Go To Dedefition」などの機能が機能しないのはなぜですか？



この情報は、実行時ではなく、コードを使用した作業中に作成されるためです。



7）一部のメンバーまたはローカル変数が利用できないのはなぜですか？ なぜいくつかのメソッドの中に入ったり、いくつかのコード行を通過できないのですか？



これは、.NET Frameworkがサイズとパフォーマンスのためにコードを最適化することを意味します。 これらの最適化の一部は、不要になった情報を削除します。 デバッグにはこれが反映されます。 ただし、ほとんどのデバッグ情報はまだセッションに存在するため、メソッドの先頭にブレークポイントを設定すると、それを表示できます。 2番目の理由は、一部のメソッドが組み込まれていることである可能性があります。つまり、これらのメソッドの中に入ることはできません。 ただし、ほとんどの場合、自由にデバッグできます。

方法＃3：.NETマスダウンローダー（任意のIDE）



すべてのキャラクターとソースコードをダウンロードするように設計されています。 コマンドラインから動作し、ソースコードで利用可能です。



必要なものをダウンロードするには、リンク（ .NET MASS DOWNLOADER ）に従って、ダウンロードして次のように実行します。



  • 32ビットシステムの場合：NetMassDownloader.exe -d“ C：\ Windows \ Microsoft.Net \ Framework \ v2.0.50727” -d“ C：\ Program Files \ Reference Assemblies \ Microsoft \ Framework \ v3.5”
  • 64ビットシステムの場合：NetMassDownloader.exe -d“ C：\ Windows \ Microsoft.Net \ Framework \ v2.0.50727” -d“ C：\ Windows \ Microsoft.Net \ Framework64 \ v2.0.50727” -d“ C：\プログラムファイル\リファレンスアセンブリ\ Microsoft \ Framework \ v3.5 "-d" C：\ Program Files（x86）\リファレンスアセンブリ\ Microsoft \ Framework \ v3.5 "
