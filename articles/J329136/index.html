Mobyプロジェクトの本質は何ですか？なぜmoby / mobyが突然Dockerのメインリポジトリになったのですか？

1か月前、DockerCon 2017カンファレンスのDockerは、新しいオープンソースプロジェクトであるMobyを公式に紹介しました。 これがDockerで作業する人が必要とするもう1つの追加プロジェクトである場合...気配りのあるユーザーが気づいたように、GitHubの会社のメインリポジトリーdocker / docker-がmoby / mobyへの送信を開始したのはなぜですか？







コンテナ内のアプリケーションを起動する簡単な方法としてDockerを使用している開発者からの先を見越した質問への回答：Moby-プロジェクトはあなたのためではありません。 その外観と内部で行われた変更にもかかわらず、何も「外側」に変更されることはありません（完成品としてのDockerのユーザーにとって）。 そして、これらの順列をもっと理解しようとする人にとって（非常に重要です！）そして、おそらく、それらを使用して問題を解決することさえできます... Docker開発の最新の歴史について簡単に説明します。



Dockerおよび「接続」ソフトウェア：runC、containerd



コンテナを起動するためのプラットフォームとしてのDockerは、開発会社が「 接続 」と呼ぶ多くの補助コンポーネントを使用します（ 配管 、英語から文字通り「配管、配管」と訳されます） 。 数年前から、Dockerエンジニアは、個々のパーツを独立したオープンソースプロジェクトに分離することで、プラットフォームをよりモジュール化しようとしています。



2015年にこのような最初の大きな「ブランチ」 が発表されました -コンテナを起動するためのrunCユーティリティ-同社は、主要な原則を修正したインフラストラクチャ配管マニフェストを公開しました。



  1. 可能な限り、既存の接続ソリューションを再利用して、改善をコードベースに戻します。
  2. 新しい接続ツールを作成する必要がある場合は、簡単に再利用して改善できます。 これにより、誰もが恩恵を受ける公的に利用可能なコンポーネントの数が増加します。
  3. UNIXの原則に従います。複数のコンポーネントが1つよりも優れていますが、複雑です。
  4. 多くの単純なコンポーネントをより複雑なシステムに組み立てるために使用できる標準インターフェースを定義します。 （単一のコマンドチェーンでBashのパイプラインを介して複数のコンソールユーティリティを構築することに似ています。）


その後、 runCユーティリティは別個に、Dockerプラットフォームの1つの機能、つまりコンテナの起動を実行しました。 これにはルートファイルシステムと構成が必要であり、他のツールに関連するすべてのタスク（イメージの取得、展開...）のソリューションを提供しました。



今年の春に、Dockerから分離され、 コンテナーランタイムを実装するコンポーネントであるcontainerdは、同様の運命に陥りました。 この記事で詳しく説明しました。 これらのプロジェクト（runCとcontainerd）は両方とも、別々のGitリポジトリを受け取るだけでなく、CNCF（Cloud Native Computing Foundation）の形式の独立した「ホーム」も受け取りました。



そして、Mobyの最近の発表により、同社はDockerのモジュラーアーキテクチャへのアプローチを一般化および体系化することで、さらに前進しました。



Mobyとは何ですか？



Moby の最初の発表では、多数のコンポーネントのレゴデザイナーのセットと、アセンブリを組み立てるためのフレームワークとを比較しました。 彼らの意図を説明しながら、著者は「ソフトウェアのコンテナ化の動きを発展させ、コンテナを主流として受け入れるようエコシステムを奨励したい」と語った。



より実用的な意味で、Mobyは以下を提供するフレームワークであることが判明しました。



  1. コンテナシステムのすべての重要な側面のためのコンテナ化されたコンポーネントのライブラリ ：オペレーティングシステム、コンテナの実行可能環境（デフォルトは前述のcontainerd）、オーケストレーション、インフラストラクチャ管理、ネットワークサポート、データストレージ、セキュリティ、アセンブリ、イメージの配布など。
  2. サポートされているプラ​​ットフォームおよびアーキテクチャのセット全体に対して、 コンポーネントをトリガーアーティファクトにアセンブルするためのツール 。 x86およびARM用のベアメタル、Linux、Mac OSおよびWindows用の実行可能ファイル、一般的なクラウドサービスおよび仮想化ツール用の仮想マシンイメージ。
  3. Moby Originと呼ばれるリファレンスキット（アセンブリ）のセット 。これは、そのままで、または変更して、または独自のバージョンを作成するための例として使用できます。


Docker自体と同様に、Mobyプロジェクトは、開発時に厳格なガイドラインに従う柔軟なソリューションを提供することを目指しています。 それらは次のとおりです。



  1. 完全なコンテナシステムを作成するために必要なすべてのコンポーネントが利用可能ですが、モジュールアーキテクチャのために（他の実装によって）交換可能です。 ところで、ここに挙げたcontainerdの代替例は、CoreOSの作者によるrktプロジェクトです。
  2. デフォルトでは、ユーザビリティを損なうことなくセキュリティが提供されます。
  3. コンテナーへの方向付け：「Mobyはコンテナーを使用して作成され、コンテナーを起動するために作成されます。」


MobyとDockerの関係



Mobyの出現により、Dockerプラットフォームは最終的に「モノリス」ではなくなり、Mobyのコンポーネントアセンブリ製品になります 。



Mobyプロジェクトは、コンポーネントをビルドするmoby



コンソールユーティリティを提供します。 現時点では、オペレーティングシステムのダウンロード可能なイメージを収集しますが、Dockerですぐに使用され、コンポーネントの多くからコンポーネントを構築します。コンポーネントの多くは独立したプロジェクトになります。


これはまた、ドッカー/ドッカーGitリポジトリが消滅した理由（およびそのmoby / mobyへの移行）を説明します：開発者には、単一の製品（既に構築されたシステム）ではなく、ニーズに応じてそのような製品を構築するためのフレームワーク（つまり、。同じコンストラクター）。 これらの製品の1つはDocker CE（およびDocker EE）で、もう1つは開発用です。



MobyはDockerプラットフォームを構築し、コンテナシステムを構築します



Mobyの出現（およびメインリポジトリの転送）により、多数のDockerエンドユーザーに明確な質問が引き起こされたため、同社の創設者であるSolomon Hykesは、「ユーザー[Mobyの出現]は影響を受けません。 バイナリファイルは変わりません。」



それは誰のためですか？



Moby Webサイトでは、プロジェクトの潜在的なユーザーの次のリストを提供しています。



  • Dockerビルドを準備またはパッチを適用するハッカー。
  • コンテナシステムを作成するシステムエンジニアまたはインテグレーター。
  • 既存のコンテナシステムを環境に適応させることを目的としたインフラストラクチャソリューションプロバイダー。
  • コンテナの世界からの愛好家で、最新のテクノロジーを試してみたい人。
  • 複数のシステムでプロジェクトをテストしようとしているオープンソース開発者。
  • Dockerがどのように内部に組み込まれ、どのように機能するかについて興味があります。


別のリストには、Moby が必要としないリストが含まれます ：コンテナーでアプリケーションを起動する簡単な方法を探しているアプリケーション開発者（個人および企業）（それぞれDocker CEおよびDocker EEからアドバイスされています）、およびコンテナーについて学び、それらを簡単にマスターする方法を見つけたい人（このため、著者によると、 docker.comで十分であり、議論するのは難しいです）。



Docker自体では、Mobyを「実験、新しいコンポーネントの開発、およびコンテナテクノロジーの未来のためのエコシステムでの共同研究のための研究所」として使用しています。



見込み



近い将来、Mobyのおかげで、runCおよびcontainerdで発生したDockerから分離された追加コンポーネント、および補助ツール、愛好家や企業の特定の機能の代替実装、特殊なコンテナのケースに焦点を当てた新しいDockerのようなプラットフォームの出現が期待できます。
