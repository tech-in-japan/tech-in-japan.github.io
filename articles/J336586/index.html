MS SQL Serverの分析、初めて見る人のために（パート2）

パート1



MS SQLサーバーで何が起こっているかを分析し続けます。 このパートでは、ユーザーの作業に関する情報を取得する方法、つまり、誰が何にどのくらいのリソースを費やしているかを調べます。



2番目の部分は、DB管理者だけでなく、実稼働サーバー上のリクエストの何が問題なのかを理解する必要のある開発者（さらに多くの開発者）にとっても興味深いものになると思います。



ユーザーアクションの分析タスクは条件付きでグループに分けられ、それぞれを個別に検討します。



  1. 特定のリクエストを分析する
  2. 特定の条件でアプリケーションからの負荷を分析します（たとえば、ユーザーがデータベースで動作するサードパーティアプリケーションでボタンをクリックしたとき）
  3. 現在の状況の分析


警告
パフォーマンス分析には、デバイスとデータベースサーバーおよびOSの原則を深く理解する必要があります。 したがって、これらの記事のみを読んでも専門家になることはありません。



実際のシステムで考慮される基準とカウンターは、互いに複雑な関係にあります。 たとえば、HDDの高負荷は、多くの場合、HDD自体の問題ではなく、RAMの不足に関連しています。 いくつかの測定を行ったとしても、これは問題のバランスの取れた評価には十分ではありません。



この記事の目的は、簡単な例を使って基本的なことを紹介することです。 推奨事項は「行動へのガイド」と見なされるべきではなく、教育的タスク（現実を単純に反映する）として、また思考の流れを説明するための「思考」オプションと見なされるべきです。

記事の結果に基づいて、数値を使用してサーバーの操作に関する結論を実証する方法を学習してください。 また、「サーバーの速度が低下しています」という言葉の代わりに、特定のインジケーターの特定の値を指定します。



特定のリクエストを分析する



最初のポイントは非常に簡単です。簡単に説明しましょう。 あいまいなことだけを考えてみましょう。



SSMSは 、クエリ結果に加えて、リクエストの実行に関する追加情報を受け取ることができます。



  • ほとんどの人は、「推定実行計画の表示」ボタン（評価計画）および「実際の実行計画を含める」（実際の計画）ボタンでクエリ計画が取得されることを知っています。 それらは、評価計画が要求を満たさずに構築されるという点で異なります。 したがって、その中の処理された行の数に関する情報は推定されます。 実際には、推定データと実際のデータの両方があります。 これらの値の間の強い不一致は、統計が無関係であることを示しています。 ただし、計画の分析-別の大きな記事のトピック-はまだ詳しく説明しません。
  • あまり知られていない事実-プロセッサコストとサーバーディスク操作の測定値を取得できます。 これを行うには、ダイアログの[クエリ] / [クエリオプション...]のいずれかでSETオプションを有効にします。

2番目のセクションでは、プロファイラーを使用します。 サーバーを起動してサーバーに接続したら、ログに記録されたイベントを選択する必要があります。 簡単な方法で、標準のトレーステンプレートを使用してプロファイリングを開始できます。 「一般」タブの「テンプレートを使用」フィールドで、「標準（デフォルト）」を選択し、「実行」をクリックします。



写真






もう少し複雑な方法は、選択したテンプレートにフィルターまたはイベントを追加（または削減）することです。 これらのオプションは、ダイアログの2番目のタブにあります。 選択可能なイベントと列の完全なセットを表示するには、「すべてのイベントを表示」と「すべての列を表示」の項目をチェックします。



写真






必要なイベントのうち、不要なイベントを含めない方がよい-トラフィックを減らすために：



  • ストアドプロシージャ\ RPC：完了
  • TSQL \ SQL：BatchCompleted


これらのイベントは、サーバーへのすべての外部SQL呼び出しをキャプチャします。 名前が示すように（完了）、要求の処理後に発生します。 SQL呼び出しの開始を修正する同様のイベントがあります。



  • ストアドプロシージャ\ RPC：開始
  • TSQL \ SQL：BatchStarting


ただし、リクエストの実行に費やされたサーバーリソースに関する情報が含まれていないため、これらは私たちにはあまり適していません。 明らかに、そのような情報は実行の最後でのみ利用可能です。 したがって、CPUのデータ、読み取り、書き込みの列*開始イベントは空になります。



まだ含まれないその他の有用なイベント：



  • ストアドプロシージャ\ SP：開始（*完了）-ストアドプロシージャの内部呼び出しをキャプチャします（クライアントからではなく、現在の要求または別のプロシージャ内）。
  • ストアドプロシージャ\ SP：StmtStarting（*完了）-ストアドプロシージャ内の各式の開始を修正します。 プロシージャ内にループがある場合、ループ内の反復と同じ数だけ、ループ内のコマンドのイベントが発生します。
  • TSQL \ SQL：StmtStarting（*完了）-SQLバッチ内の各式の開始を修正します。 リクエストに複数のチームが含まれる場合-各チームにイベントがあります。 すなわち 前のものと同様に、プロシージャ内のコマンドではなく、リクエスト内のコマンドに対してのみ機能します。


これらのイベントは、実行ステップの追跡に役立ちます。 たとえば、デバッガを使用することはできません。

サーバー全体のユーザーのアクティビティを分析します



上記のセクションの情報が役に立たない場合にも、人生の状況が発生します。

いくつかの要求は「実行」で非常に長い時間ハングアップし、終了するかどうかは不明です。 問題のあるリクエストを個別に分析するには-したいのですが-最初にどのようなリクエストを決定する必要があります。 プロファイラーをキャッチするのは無意味です-開始イベントをすでに見逃しており、どのくらいの時間待機するかは不明です。



または、ユーザー要求がまったくないか、サーバー自体が積極的に何かをしているかもしれません...



理解しましょう



みなさんはアクティビティモニターを見たに違いありません。 古いスタジオでは、その機能がより豊かになりました。 彼はどのように私たちを助けることができますか？ 「Activity Monitor」には多くの便利で興味深いものがありますが、3番目のセクションはそれについてではありません。 システムの表現と機能から直接取得する必要があるすべて（およびモニター自体は、プロファイラーを設定して実行する要求を確認できるという点で便利です）。



必要なもの：



  • sys.dm_exec_sessions-セッション情報。 接続ユーザーの情報を表示します。 有用なフィールド（この記事のフレームワーク内）-ユーザー（login_name、login_time、host_name、program_name、...）の識別と使用済みリソースに関する情報（cpu_time、reads、writes、memory_usage、...）のフィールド
  • sys.dm_exec_requests-現在実行中のリクエストに関する情報。 また、非常に多くのフィールドがありますが、ごく少数を検討します。

    • session_id-前のビューとの通信用のセッションコード
    • start_time-要求の開始時間
    • コマンド-このフィールドは、名前とは異なり、リクエストではなく実行するコマンドのタイプを含みます。 カスタムクエリの場合-これは通常、select / update / delete /などのようなものです。 （また、以下の重要な注意事項）
    • sql_handle、statement_start_offset、statement_end_offset-クエリテキストを受け取るための情報：ハンドル、クエリテキストの開始位置と終了位置-現在実行されている部分を示します（クエリに複数のコマンドが含まれる場合）。
    • plan_handle-生成されたプランのハンドル。
    • blocking_session_id-リクエストの実行を妨げるロックが発生した場合-ロックを引き起こしたセッション番号を示します
    • wait_type、wait_time、wait_resource-待機の理由と期間に関する情報を含むフィールド。 たとえば、データのブロックなど、一部のタイプの待機では、ブロックされたリソースのコードが追加で示されます。
    • percent_complete-名前では、これが完了の割合であることは明らかです。 残念ながら、進行状況が明確に予測されているチーム（バックアップや復元など）でのみ使用できます。
    • cpu_time、読み取り、書き込み、logical_reads、granted_query_memory-リソースコスト。
  • sys.dm_exec_sql_text（sql_handle | plan_handle） 、 sys.dm_exec_query_plan（plan_handle） -テキストおよびクエリプランを受信するための関数。 以下で使用例を検討します。
  • sys.dm_exec_query_stats-クエリによって分類された実行の要約統計。 リクエストの実行回数と、リクエストに費やされたリソースの数を表示します。


重要な注意事項



上記のリストはほんの一部です。 すべてのシステム表現と機能の完全なリストは、 ドキュメントに記載されています 。 また、 美しい絵の形で主要なオブジェクトの接続図があります-A1に印刷して壁に掛けることができます。



要求テキスト、そのプラン、および実行統計は、手続き型キャッシュに保存されているデータです。 実行時に、それらは利用可能です。 実行後、可用性は保証されず、キャッシュへの圧力に依存します。 はい、キャッシュは手動でクリアできます。 実装計画が「浮いた」ときに行うことが推奨されることもありますが、多くの微妙な違いがあります...一般的に、「禁忌があります。専門家に相談することをお勧めします。」



「コマンド」フィールド-ユーザークエリの場合、実質的には意味がありません-結局のところ、フルテキストを取得できます...しかし、それほど単純ではありません。 このフィールドは、システムプロセスに関する情報を取得するために非常に重要です。 原則として、いくつかの内部タスクを実行し、SQLテキストはありません。 このようなプロセスの場合、チーム情報はアクティビティのタイプの唯一のヒントです。 前の記事へのコメントでは、サーバーが何でビジーであるかについての質問がありました。何もビジーではないように思えますが、おそらくこのフィールドの値に答えがあるでしょう。 私の実践では、アクティブなシステムプロセスの「コマンド」フィールドは、常に完全に理解できるものを示していました。autoshrink/ autogrow / checkpoint / logwriter /など。



使い方



実用的な部分に移りましょう。 使用例をいくつか示しますが、想像力を制限するものではありません。 サーバーの機能はこれに限定されません-あなた自身の何かを発明することができます。



例1：CPU /読み取り/書き込み/メモリを消費するプロセス



まず、どのセッションがCPUを最も消費しているかを見てみましょう。 sys.dm_exec_sessionsの情報。 ただし、CPU上のデータ（および読み取り、書き込み）は累積されます。 つまり、フィールドの数値には、接続時間全体の「合計」が含まれます。 ほとんどすべての人が1か月前に接続し、一度も切断されないことが明らかになります。 これは、彼が現在システムをロードしているという意味ではありません。



小さなコードで問題を解決できます。アルゴリズムは次のようになります。



  1. 最初に選択を行い、一時テーブルに保存します
  2. それからちょっと待って
  3. 2回目のサンプリングを行う
  4. 最初のサンプルと2番目のサンプルの結果を比較します-差は、段落2で発生したコストとまったく同じです
  5. 便宜上、差をクレーム2の期間で割って、平均の「1秒あたりのコスト」を取得できます。


スクリプトの例 
 if object_id('tempdb..#tmp') is NULL BEGIN SELECT * into #tmp from sys.dm_exec_sessions s PRINT '       ' --     , ..      WAITFOR DELAY '00:00:01'; END if object_id('tempdb..#tmp1') is not null drop table #tmp1 declare @d datetime declare @dd float select @d = crdate from tempdb.dbo.sysobjects where id=object_id('tempdb..#tmp') select * into #tmp1 from sys.dm_exec_sessions s select @dd=datediff(ms,@d,getdate()) select @dd AS [ , ] SELECT TOP 30 s.session_id, s.host_name, db_name(s.database_id) as db, s.login_name,s.login_time,s.program_name, s.cpu_time-isnull(t.cpu_time,0) as cpu_Diff, convert(numeric(16,2),(s.cpu_time-isnull(t.cpu_time,0))/@dd*1000) as cpu_sec, s.reads+s.writes-isnull(t.reads,0)-isnull(t.writes,0) as totIO_Diff, convert(numeric(16,2),(s.reads+s.writes-isnull(t.reads,0)-isnull(t.writes,0))/@dd*1000) as totIO_sec, s.reads-isnull(t.reads,0) as reads_Diff, convert(numeric(16,2),(s.reads-isnull(t.reads,0))/@dd*1000) as reads_sec, s.writes-isnull(t.writes,0) as writes_Diff, convert(numeric(16,2),(s.writes-isnull(t.writes,0))/@dd*1000) as writes_sec, s.logical_reads-isnull(t.logical_reads,0) as logical_reads_Diff, convert(numeric(16,2),(s.logical_reads-isnull(t.logical_reads,0))/@dd*1000) as logical_reads_sec, s.memory_usage, s.memory_usage-isnull(t.memory_usage,0) as [mem_D], s.nt_user_name,s.nt_domain from #tmp1 s LEFT join #tmp t on s.session_id=t.session_id order BY cpu_Diff desc --totIO_Diff desc --logical_reads_Diff desc drop table #tmp GO select * into #tmp from #tmp1 drop table #tmp1


コードでは、2つのテーブルを使用しています。＃tmp-最初の選択用、＃tmp1-2番目の選択用。 このスクリプトは、最初の実行時に、1秒間隔で#tmpと＃tmp1を作成および設定し、残りを実行します。 以降の起動では、スクリプトは前の実行の結果を比較のベースとして使用します。 したがって、後続の起動時のクレーム2の期間は、スクリプト開始間の待機期間に等しくなります。 実稼働サーバーですぐに実行できます-スクリプトは「一時テーブル」のみを作成し（現在のセッション内でのみ使用でき、切断時に自己破壊します）、危険を伴いません。



スタジオでリクエストを実行したくない場合は、お気に入りのプログラミング言語で作成されたアプリケーションでラップできます。 コードを1行も使わずにMS Excelでこれを行う方法を示します。

  • 「SET NOCOUNT ON」を追加-なぜなら Excelは行数のトリミングを好みません。
  • 「一時テーブル」を「変数テーブル」に置き換えます。
  • 遅延は常に1秒です-平均値のフィールドは不要です
