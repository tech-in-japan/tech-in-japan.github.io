クロスプラットフォームのオープンソースタイムトラッカー



この記事では、なぜタイムトラッカーが必要なのか、どのように検索したのか、なぜ見つからなかったのか、何が起こったのかについてお話したいと思います。



トラッカーの選択



私はほとんどのキャリアでフリーランサーとして働いています。 アクティビティの詳細は、支払いを伴うすべての契約が時間単位ではなく、時間通りに行われる作業に関するものです。 そのような状況では、気分が良いときに仕事に慣れます-主なことは期限を満たすことです。 つまり、1日中プロジェクトに手を触れないようにし、翌日の16時間はギャレーの奴隷のように休みなく耕すのが一般的です。 顧客にとって、これは通常無関心です。 このモードで作業する場合、タイムトラッカーについては考えません。



2015年の初めに、私は面白いリモートプロジェクトを提供されました。突然、仕事に対する支払いではなく、毎月決まった給与が支払われました。



仕事を始めてから数ヶ月後、私の通常のアプローチがうまく機能していないことが明らかになりました。 締め切りがあなたの前にあり、支払いの事実が仕事の引き渡しに結びついている場合、特に締め切りに違反すると重大な没収につながる可能性があるため、追加の動機は必要ありません。 固定支払いの場合、状況は異なります。 これを実現するために、私は非効率的に働いており、同じペースで仕事を続ければ雇用主が満足する可能性は低いことに気付きました。



3月末に、フリーランスの Habr。Overseerに記事が公開されました。 勤務時間を記録するシステムを選択します 。 この記事は、効率の改善に関する私の考えと完全に一致しました。 それは主にウェブツールに捧げられており、私の仕事には合いませんが、主なアイデアは、時間を考慮して、私が仕事時間を正しく分配し、雇用主を失望させない銀の弾丸のように思えました。



幸運なことに、記事へのコメントで、読者はあらゆる種類のトラッカーを使用した経験を共有し、 Alexey2005 はトラッカーの比較表へのリンクを投稿しました。



さまざまなトラッカーの研究に飛び込むほど、自分に合わないことが明らかになりました。



トラッカーの要件のリストが作成されました。

  1. 手動タイマーはありません。 ユーザーがさまざまなタスクを絶えず切り替えている世界では、彼は煙に向けて出発し、ICQで愛人に答えます。手動タイマーの精度は0になる傾向があります。
  2. オープンソースのみ。 私は妄想的だと思うが、クローズドソースソフトウェアが私の活動に関する情報を他の誰かのサーバーに送信したくない。 数年前、私はそれについても考えず、静かに独自のトラッカーをインストールしました。 しかし、監視の世界における最新の傾向は、2つのことを教えてくれました。

    • 会社にデータがある場合、それを使用します。
    • データはリークされるか、リクエストに応じて提供されます


    禁じられたビットコインを使ったという告発でいつか私に来てほしくありません。

  3. クロスプラットフォーム。 ゲーム開発者として、私は主にWindowsで仕事をしていますが、LinuxとMac OS Xも私の活動の大部分を担っています。 OSの1つだけでアクティビティを追跡するタイムトラッカーでは、完全な画像は得られません。


自動クロスプラットフォームトラッカーには2つ半の部分があります。 それらの中のオープンソースは、私は1つを見つけませんでした。



あなたの書いて



どうする？ 選択は明らかです-トラッカーを作成します。



操作の基本的なアルゴリズムは単純です。プログラムはバックグラウンドでハングし、アクティブなウィンドウを変更すると、ユーザーが切り替えたアプリケーションを記憶します。 ユーザーがボタンを何も押さず、マウスを動かさず、ウィンドウが切り替わらない場合、最後のレコードをそれに応じて調整することでスリープモードに移行します（ユーザーが非アクティブだった時間をとります）。



PCに接続して省電力モードに切り替える（モニターをオフにする）のは論理的ですが、多くの場合、このモードは完全にオフになるか、一時停止が長すぎるため、これに頼ることはできません。

さらに、正確なタイミングのためにプロファイルが必要であることは明らかです。 現在のアクティビティのタイプに応じて、異なるアプリケーションが異なるカテゴリに分類される場合があります。 たとえば、フリーランスのプロジェクトと自分のプロジェクトの両方でメモ帳++を使用しています。



プロファイルを使用すると、アプリケーションをさまざまなカテゴリに割り当てることができます。 したがって、「プロジェクト」プロファイルがアクティブな場合、メモ帳++の使用時間は1つのカテゴリに記録され、「フリーランス」プロファイルがアクティブな場合は別のカテゴリに記録されます。



外部トラッカー



そして今、トラッカーの最初のバージョンが作成され、起動されています。 数日間のアクティブなテストの後、現在の自動化では不十分であることがわかりました。 ヘッダーのスクリーンショットは、Chromeが3時間50分を食べたのと同時に、先延ばしカテゴリーも3時間50分かかったことを示しています。 しかし実際には、Chromeはバグトラッカー、ドキュメント、stackoverflowへのアクセスにも使用されます。 しかし、トラッカーの最初のバージョンは、これらのアクションを区別する方法を知らず、すべてを1つのカテゴリに記録しました。 一時的な解決策の1つは、Chromeを使用してHabr / mail / vkなどを読み取り、Firerefoxですべての作業を行うことです。 しかし、習慣は強すぎたため、結果として、作業の一部はChromeの習慣から行われました。



仮想マシンでの作業中に問題が激化した。 もちろん、トラッカーはそれ自体では見ることができません。VBox内で何が起こっているのか、VirtualBox.exeアクティビティのみを見ることができます。 もちろん、仮想マシンで一度に実行されるアクティビティは1種類のみであり、プロファイルを切り替えることで問題を解決できます。 しかし、当初は手動制御から逃げたいと思っていました。



その結果、外部トラッカーの形でソリューションが生まれました。



外部トラッカーは、タイムトラッカーがアクセスできない環境内で実行され、そこから情報を送信するプログラムです。 たとえば、仮想マシンの場合、このトラッカーはタイムトラッカーと同じように動作しますが、実行中のアプリケーションは記憶せず、udpチャネルを介してポート25855でメイントラッカーにレポートします。



メッセージ形式は非常に単純です：

5文字は固定ヘッダーです。 タイテット

1 int-プロトコルバージョン番号。 今1。

utf8文字列 （1文字列の長さ、次にchar配列）-システムの現在の状態。



たとえば、仮想マシンは現在のアプリケーションをchrome.exe、qtcreator.exe、explorer.exeの形式で単に提供します。 ブラウザの場合、サイトは海であり、ユーザーが操作するメインサイトはステータスとして与えられます：habrahabr.ru、geektimes.ru、vk.com、gmail.com、google.comなど。 残りのサイトは、未定義の一般ステータスになります。 つまり、メインサイトはタイムトラッカーレベルで適切なカテゴリに定義され、未定義は別のカテゴリに移動します。 例えば、私は先延ばしに未定義を書いています。なぜなら、別のゴミを読むときにあなたが入る何百万ものジャンクよりも作業サイトのセットを決定する方がはるかに簡単だからです。



1 int-重複するアプリケーションの数

utf8-重複する各アプリケーションの名前を持つ文字列の配列



重複するアプリケーションとは、トラッカーから新しい（10秒未満）情報がある場合に置き換えられるアプリケーションのリストです。 たとえば、Chromeトラッカーは2つのアプリケーションにオーバーラップします。

chrome.exeおよびGoogle-chrome-stable。



したがって、habrahabr.ruの現在のステータスがそこから来る場合、Windowsでのchrome.exeおよびLinuxでのGoogle-chrome-stableでのアクティビティ中のタイムトラッカーは、Chromeではなくhabrahabr.ruのデータベースに書き込みます。

1符号なしchar-チェックサム。 チェックサムは、パケット内のすべてのバイトの単純な追加と見なされます。



タイムトラッカーと外部トラッカーを組み合わせることで、PCで費やした時間に関する最も正確な情報を取得できます。



原則として、Androidトラッカーなど、外部トラッカーが真に外部であることを妨げるものはありません。 これにより、PC以外のアクティビティを考慮することができます。 ただし、このような機能は必要ないため、実装しませんでした。



トラッカーを使用する



ユーザーの観点から見ると、アプリケーションはトレイ内のアイコンであり、必要なウィンドウを表示するコンテキストメニューがあります。







マイナーな機能：



設定により、アプリケーションの動作の一般的な詳細を決定できます。



プロファイルを使用すると、アプリケーションウィンドウを開かずにアクティブなプロファイルを切り替えることができます。



そして最も重要なこと：



アプリケーション - アプリケーションを分類し（アプリケーションは操作中に自動的にリストに追加されます）、プロファイルを編集できます：







統計 -指定した日付のデータベースから情報を引き裂き、グラフと表を描くことができます。







一方では、統計の表示はタイムトラッカーの重要な部分です。 しかし、トラッカーアプリケーション自体は、データ分析のための単なるシンプルなサーフェスツールである必要があると思います。 主なものはコレクションです。 また、エクスポートされたCSVデータに基づいてLibreCalcで美しい図を描くことができます。



それとは別に、データベースとの作業について議論する価値があります。



最初は、非常に急速に成長し、プログラムで細分化する必要があるのではないかと心配しました。



しかし、最初のバージョンのテストプロセスでは、100個のアプリケーションを備えた3つのOSでの半年間の作業で、データベースは1メガバイトしか消費しないことが示されました。 10年で20メガバイト。 心配するサイズではありません。 ただし、データベースを1つにまとめると、アプリケーションコードは簡素化されます。



統計の観点から各OSに個別のファイルを保持することは無意味なので、私は100 MBのexFATセクションを作成しました。ここにデータベースを配置し、Windows / Linux / Mac OS Xはこの1つのファイルで動作します。 その結果、一般的な統計が1か所で利用可能になります。



オープンソースに行く



アプリケーションの最初のバージョンが作成されてから約10か月が経過しました。 私自身は、アプリケーションを非常に適していると評価しました。 偶然の一致により、1月に記事が公開されました。 なぜ、なぜオープンソースコードを書くのでしょうか。 。 そして、時間をカウントするタスクは非常に重要であり、オープンソースソリューションはそれほど多くないため、タイムトラッカーをgithubに公開しないことを考えました。



そして今、githubを知り、アプリケーションを書き直し、ほぼ1年間の使用で蓄積されたすべてのニュアンスを考慮に入れ、最初のビルドでフォークをダウンロードして作成する準備ができました。



心から話させて

突然、英語の知識が乏しいことが、オープンソースプロジェクトを公開するときに最も困難であることが判明しました。 その瞬間、少なくとも最小限のドキュメンテーションとコメントなしではプロジェクトが意味をなさないことが明らかになった-恥をかかないように、公開されたくないという願望がありました...意味があり、文法的で、スペルに優しい文章は私には難しすぎます...



そして、選択肢がありました：

  1. 何も公開しないでください。
  2. 単音節のコメントに制限します。 おそらく彼らは私が英語を知らないとは思わないでしょう。
  3. 貧弱な英語にふさわしいと思うだけ書いてください。


最初のオプションは受け入れられないようでした。 このプロジェクトは有用であると見なされており、その公開は重要です。 それには数十人のユーザーがいると思うので、未公開のままにしておくのは残念です。

2番目のオプションでは、ソースとアプリケーション自体の操作に関する重要なポイントを完全に描画することはできません。 そして、短い文章ですべてを正確に書くことができる確率は何ですか？



その結果、彼は3番目の選択肢に落ち着きました。 結局、これは英語の試験や本ではなく、ソフトウェアです。 おそらく、より深刻な同志が深刻な学校を修正するか、アプリケーションが私の曲がったコメントで機能する可能性があります。
