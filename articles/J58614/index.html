インターネットインフォメーションサービスチームのプログラムマネージャーRuslan Yakushevへのインタビュー

Ruslan Yakushevは、約2年前にインターネットインフォメーションサービス（IIS）チームのプログラムマネージャーとして働き始めました。 この間、FastCGI、URL Rewriter、Dynamic IP Restrictionsなど、いくつかの主要なIISコンポーネントの開発とリリースを主導しました。

これに加えて、Ruslanの活動の1つは、Windowsプラットフォーム上のPHPアプリケーションの機能とパフォーマンスを改善するためのPHP開発者のコ​​ミュニティとの対話とコラボレーションです。



IISチームに参加する前、RuslanはBizTalk Serverチームに5年間勤務し、Microsoft BizTalk Server 2004および2006のリリースに参加しました。Ruslanの詳細については、彼のブログruslany.netを参照してください。



このトピックではインタビューの質問をしました。



回答を投稿する際の長い遅延をおaび申し上げます。 途中で Zhenya OsovetskyとAlex Mogilevskyへのインタビュー。



ルスランとのインタビューのテキスト



彼らは最近、彼ら自身のためにPHPで何を書いたのですか？



[RY]：過去数か月間、暇なときにWordPressプラグインを作成しています（ http://wordpress.org/extend/plugins/wp-media-player/ ）。 今年の1月に最初のバージョンを公開して以来、約900回のプラグインのダウンロードがありました。 私は最近、いくつかの新機能を含む新しいバージョンをリリースしました。



多くのメタベースノードで、IIS7によって消費されるメモリサイズがIIS6の5倍を超えるのはなぜですか？



[RY]：質問（ dmach ）を尋ねたユーザーが正しく指摘したように、IIS 7はXMLベースの構成システムを使用します。これにより、IIS6と比較してメモリ消費が増加します。 IIS 7では、構成の委任と分離も追加されるため、構成の複数のコピーを保存する必要があります。 同時に、IISチームは多くの最適化を行ってメモリ消費を削減しました。これは、IIS 7が1つのサーバー上に多数のサイトを持つホスティング会社で使用できるサーバーとして位置付けられているためです。 メモリ消費量の増加の問題は、私たちと一緒に仕事をしたホストからは決して言及されていませんでした。 ただし、これは、ホスティング事業者がサイトごとに個別のワークフローを使用しており、すべてのワークフローが同時にアクティブになっているわけではないためです。 dmachは、数千のWebサイトで1つのワークフローを使用しているようです。 これはサポートされている構成ですが、推奨されません。 これにより、メモリ消費が5倍増加する可能性があります。 詳細を知らずに言うのは難しいです。 dmachがメモリ消費の増加を観察する構成の詳細を提供できる場合、それを引き起こした原因をより具体的に把握することが可能になります。 また、最近、IISチームはQFEをリリースしました。これにより、状況によってメモリ消費が増加するバグが修正されました。 dmachはこのQFEをインストールしようとする場合があります。 この問題に関するより具体的な情報については、私のブログから直接私に連絡することができます： ruslany.net/contact



将来のバージョンでは、同じワークフロー内の個々のサイトまたは実行可能な要求によって消費されるメモリの量を確認できるようにしたいです。 そのような可能性は考慮されていますか？



[RY]：いいえ、考慮されていません。 IISで推奨される構成は、サイトを個別のワークフローに分離することです。 IISのすべての監視ツールは、この構成用に構成されています。



IISバージョン6を含む他のすべてのWebサーバーでできるように、IISの次のバージョンはxn-blablablaの形式でIDNドメインのバインディングを理解しますか？ 残念ながら、IISバージョン7ではその方法がわかりません。 詳細はこちら-dmach.livejournal.com/136446.html



[RY]：次のバージョンのIISでは、IIS7と同様にIDNドメインバインディングが発生します。 IIS7でIDNドメインを指定するには、Unicode形式を使用する必要があります。 punycodeへの変換は、IISで自動的に行われます。 IIS7でIDNドメインを操作する方法の詳細については、 blogs.iis.net / thomad / archive / 2008/03/05 / hosting-220-bersites-iis7-support-for-international-domain-namesをご覧ください。 -idn.aspx

この変更により、IIS6からIIS7に移行する際にサイトが破損する可能性があるため、IIS Web展開チームは、IDNドメインを持つサイトをIIS6からIIS7に移行する際にpunycode変換のサポートを追加することを検討しています。



IISバージョン6で、モジュールのIISワークフローの権限とは異なる権限でPHP実行可能プロセスをFastCGIモードで実行することは可能ですか？ そうでない場合、指定された（または他の問題ではない）機能を備えたIISの以前のバージョン用にこのモジュールを開発する予定ですか、それとも現在の形式で修正されて開発されませんか？



[RY]：これは不可能です。 なりすましがありますが、私が理解しているように、これはdmachが正確に求めているものではありません。 技術的には、これは難しくなく、理論的には興味深い機会のように聞こえます。 これについては今まで誰も質問していません。 特定のケースでのなりすましでは不十分な場合について詳しく知りたいと思います。 FastCGIプロセスを実行する権利を設定することが有用となる実際の状況がある場合、FastCGIモジュールの将来のバージョンでこのような機能を追加することで特別な問題は発生しません。 つい最近、FastCGI extension 1.5 Betaをリリースしましたが、レビューと提案はまだ受け付けています。 機能に関する提案は、具体的に私またはフォーラムに送信できます： forums.iis.net/1103.aspx （ またはgaidar-およそGaidarを通じて ）



管理コンソールは、IISの将来のバージョンでキーボード操作と800x600などの最小画面拡張用に最適化されますか？ これは、たとえば、ラップトップ（または、神が私を許してくれるネットブック）でRDPを使用できない未知のモニターのDCで、...とは異なる条件でIISを操作するのに意味があります。



[RY]：IISコンソールはキーボード操作をサポートしています。 800x600の解像度で作業することもできます。 もちろん、これは高解像度ほど便利ではありません。もっとスクロールする必要があります。 私が知っているように、これに関して変更は計画されていません。



将来のバージョンでは、Microsoftは、IIS / FTPに対して、Windowsユーザーに関連付けられていない（またはオプションで）ユーザーベースを作成する予定ですか（たとえば、SQL Serverで現在行われているように）。 IISのこのアプローチの欠点は何ですか？



[RY]：IISには独自のユーザーベースがあります。 FTPで使用できるIIS管理ユーザーのサポートがあります： learn.iis.net/page.aspx/321/configure-ftp-with-iis-70-manager-authentication 。 同じユーザーデータベースを、ウェブサイトlearn.iis.net/page.aspx/159/configuring-remote-administration-and-feature-delegation-in-iis-7のリモート管理に使用できます。



Microsoft IISは、サーバー上の1つまたは2つのサイトをサポートするソリューションとして、または幅広いタスクのソリューションとして位置付けられていますか？ IISバージョン7は、IISユニットごとに何万ものサイトでテストされていますか？ その場合、テストチームはそのようなテストの結果に満足しましたか？



[RY]：Microsoftの内部では、IISは幅広いタスクのソリューションとして位置付けられています。 たとえば、指示の1つはホスティングサポートです。 ホスティングプラットフォームとしてのIIS7の内部テストが多数実施されました。 テスト中、1つのサーバーでさまざまなタイプ（ASP、APS.NET、PHP、静的）の最大4000サイトが使用されました。 サーバー構成は、ホスティング会社に提供するすべての推奨事項を満たしました（ learn.iis.net/page.aspx/31/hosting-guidance-for-the-microsoft-web-platform ）。 この構成では、テスト結果は非常に受け入れられ、ホスティングの要件を満たしました。



開発チームはIISでどのくらい作業していますか？



[RY]：マイクロソフトの他のチームと比較すると、IISチームは非常に小さく、40人強です。



Ruslanはプログラムマネージャーの仕事でどのような困難に直面しましたか？ このような問題はどのように解決されましたか？



[RY]：最大の困難は、私がマイクロソフトで働き始めたときでした。 プログラムマネージャーに何が期待され、何をすべきかはまったく明確ではありませんでした。 開発者はコードを書き、テスターはテストし、PMは具体的なことを何もしなかったようです。 しばらくしてから、何をする必要があるのか​​、どのように役立つのかを理解し始めました。

PMの作業だけでなく、開発者とテスターの間にも存在する別の問題は、一定のランダム化です。 時間が経つにつれて、あなたは非常に厳密にタスクに優先順位を付ける必要があることを理解し始め、気を散らさないことを学ぶ必要があり、必要に応じて誰かがあなたに何かを頼むと拒否します。 そうでなければ、質の高い方法で作業を行うことは不可能です。



今後どのような機能が予定されていますか？



[RY]：これは機密情報であるため、特定の機能については説明しません。 現在、チームにはいくつかの優先的な作業領域があるため、基本的に新しい機能がこれらの領域に関連付けられます。 たとえば、一般的なWebアプリケーションがIISでサポートされ、簡単にインストールおよび構成できることが非常に重要です。 Microsoft Web Platform Installerはこの方向への最初のステップであり、IISチームはこのプロジェクトの作業を継続します。 また、IISでのPHPサポートは引き続き優先されます。これは、ほとんどの一般的なWebアプリケーションが現在PHPを使用しているためです。 Windows上のPHPは、Linuxのパフォーマンスにまだ遅れています。 IISチームはこの方向に積極的に取り組んでいます。
