Scalatestを使用した情報システムの監視と統合テストの実装。 パート1





この記事では、テスト自動化のためにScalatestフレームワークを使用してテストを作成した経験を共有したいと思います。 この記事は2つのパートで構成されます。 1つ目は、基本的なテストを作成して実行するための段階的な指示、2つ目は、テストスタックのより複雑なケースとニュアンス、テストレポートの作成、問題の解決に関する情報の検討です。



テストの自動化には多くのソリューションがあります。 それぞれに独自の特性、利点、欠点があり、エントリのしきい値、使いやすさ、効率、汎用性、適切なタスクの範囲が異なります。 いずれかのプロジェクトの統合テストおよび監視システムを自動化するタスクでは、「Scala」+「ScalaTest」+「SBT」の組み合わせを使用することが成功したソリューションであることが判明しました。



チャレンジ：



統合テストを自動化します。

監視を整理するには-情報システム全体およびその個々の部分の操作性/可用性の自動チェック。

システム障害が発生した場合に問題のあるコンポーネントの迅速なローカライズを提供します。

テストに関するレポートをわかりやすい形式で提供します。



一般的な概念は次のとおりです。情報システムでの処理の開始を開始する特定のアクションが実行されます（サービス、キューマネージャーへのメッセージの送信、Webインターフェイスで特定のアクションシーケンスの実行）。 システムはメッセージを処理し、コンポーネント、サブシステムを介してメッセージを駆動します。 サブシステムがいくつかの基準に従ってタスクを完了したことを確認します：ログ内のレコードの存在、データベース内の新しいレコードの存在、リクエストに対する正解、システムからの応答メッセージの受信、サイトでの表示。 このような各チェックは、成功または失敗する可能性のあるテストシナリオのステップに変わります。



テストに合格すると、いくつかのファイルが添付されたレポート（システム応答、スクリーンショット、クエリ結果）が生成されます。このファイルから、システム内のすべてが問題ないかどうか、問題の場所を確認できます。 サンプルレポートを以下に示します。



















使用される技術の簡単な説明：



Scalaは、機能的およびオブジェクト指向プログラミングの機能を組み合わせて、コンポーネントソフトウェアを簡単かつ迅速に作成するために簡潔でタイプセーフになるように設計されたプログラミング言語です。 jvmの上で実行されます。

Sbt-（scalaビルドツール）-ScalaとJavaで記述されたプロジェクトの自動ビルドシステム。

Scalatestは、アプリケーションをテストするためのフレームワークです。 メインページで「単純に生産的」と読み、「scalatestは、コードと通信の両方を改善するシンプルで明確なテストと実行可能な仕様により、チームの生産性を高めるように設計されています」。

Selenium WebDriver-Webアプリケーションをテストするためのツール、ブラウザーを管理し、ユーザーアクションをエミュレートするためのツールのセット。



このスタックは、その汎用性（win / linux / macosでの開発と起動に特に違いはありません）、その機能の豊富さ（scalaを使用して何かを実装できない場合、わずかに変更されたJavaコードとJavaライブラリを使用できます）、効率、IDEからのサポートに満足しています（IntelliJ IDEA、Eclipse）、優れたドキュメント。 私は特に、この「Just works」（Windowsでのエンコードの問題を省略した場合）、つまりタンバリンによって解決される問題の数はそれほど多くないこと、そしてスタックでの作業を開始するのに情報システムのプログラミングとチューニングの経験は必要ないことを特に喜んでいます。 特に優れたIDEを使用している場合、Scalaでの書き込みは非常に便利です。



Oracle JDKはマシンにインストールする必要があります。できればバージョン8（1.8）を使用してください。以前のバージョンでは、一部のコンポーネントが起動しない場合があります。openJDKではIdeaは機能しません。 www.oracle.com/technetwork/java/javase/downloads/jdk8-downloads-2133151.html 、



テストを開発するには、jetbrainsのIntelliJ IDEA Community Editionを使用します。



これを行うには、アイデアを入れて実行し、scalaプラグインをインストールします。















アプリケーションを再起動します。







新しいプロジェクト、Scala-> SBTを作成します。







名前、ディレクトリを入力し、JDKへのパスを選択し、「自動インポートを使用」チェックボックスをオンにして、build.sbtファイルの変更がすぐに更新されるようにします。







その後、しばらくすると、プロジェクト構造が作成されます。 Ideaがプロジェクトを作成し、インデックスを作成し、リポジトリからコンポーネントの正しいバージョンを取得する必要があるため、これには初めて数分かかる場合があります。 システムはいくつかの警告メッセージをダンプできますが、無視してください。







プロジェクトディレクトリに作成された「build.sbt」ファイルには、プロジェクトに関する情報と使用されているコンポーネントのリストが含まれています。

scalatestフレームワークを接続するには、build.sbtに対応する行を追加します。 



libraryDependencies += "org.scalatest" % "scalatest_2.11" % "3.0.0-M7"




これは、執筆時点での最新バージョンです。

（SBTはMAVENリポジトリーを使用するため、 mvnrepository.com / artifact / org.scalatest / scalatest_2.11 / 3.0.0-M7などのサイトでライブラリーを見つけることができます）。



（SBTでは、バージョンフィールドに「latest.integration」を指定することにより、最新バージョンを自動的にロードすることができます。ただし、これにより問題が発生する可能性があります。アップグレード中の動作の変更。バージョンを修正することをお勧めします）。



このファイルの行は空行で区切られていることに注意してください（これは慣例ですが、行区切りなしでも機能します）。

追加後、アイデアはリポジトリから適切なライブラリを選択してロードします。 この操作にもかなりの時間がかかる場合があります。 更新に問題がある場合があります。プロジェクトを閉じて、再度開くことができます。







フォームの下部にあるインジケータは、プロジェクトの準備状況を示します。 簡単なテストを作成して、scalatestがどのように機能するかを見てみましょう。 テストクラスは、コンテキストメニューから/ src / test / scalaディレクトリに作成されます。







それをDummyTestと呼びましょう。 クラステキスト： 



 /** * Created by user on 26.08.2015. */ import org.scalatest.{Matchers, FreeSpec} class DummyTest extends FreeSpec with Matchers{ "    " in { val num = 2+3 num should be (5) } "    " in { val num = 2+3 num should be (4) } }


テストについて説明します。 ここで紹介するスタイルの中で、私はFreeSpecが好きです。 これを使用します（クラス名の後に「extends FreeSpec」を追加します）。 比較を使用するには「With Matchers」が必要です。たとえば、xはxの値がyの値と一致することを（y）チェックする必要があります。 こことここの値のより詳細な比較。



テストには2つのステップが含まれます。 前者は明らかに正しいのですが、後者は落ちるはずです。 このステップは「2 + 3 = 5」と呼ばれ、レポートに表示されます。 これは引用符で囲まれた文字列であり、キリル文字を使用できます。 の単語は、括弧と「{」、「}」で囲まれたステップの名前と本文を分離します。 ボディのすべてのステップが成功した場合、テストに合格したとみなされます。 何かがうまくいかず、テストがステップに落ちた場合、テストは失敗したと見なされます。



「Val num = 2 + 3」-変数を宣言し、値を割り当てます。



一般に、scalaの広告は次のようになります。



valまたはval VariableName [：DataType] =初期値



val-変更されない値（定数のように見える）、var-変数-コードの実行中に変更されます。 valを使用することをお勧めします。 「Num should（5）」-値を比較し、変数「num」の値が5であることを確認します。



インポートの行にエラーが表示される場合、おそらく必要なライブラリがまだロードされておらず、アイデアは必要なコンポーネントをインポートできません。



Ideaでスクリプトを実行します。 クラス内のコンテキストメニューを呼び出します-「実行」項目が表示されます（クラスを実行することはできません。このオプションは「FreeSpecの拡張」によって提供されます-特性FreeSpecを使用） その後、クラスは画面右上のリストに追加され、ボタンまたはキーの組み合わせ「Shift + f10」で起動できます







Windowsで作業している場合-エラーが表示されます-エンコードの問題。 1251-デフォルトのWindowsエンコードはキリル文字では機能しません。 したがって、ウィンドウの右下部分でUTF-8に変更します。 ダイアログボックスで、「変換」をクリックします。 Linux / Macでは、問題はないはずです。Idfの新しいファイルのデフォルトのエンコードはutf-8です。







もう一度、テストを実行し、フォームの下部にある結果を確認します。







左側にチェックのある特定のツリーがあり、中央にエラーの説明があり、右側に結果があります。 予想通り、2番目のステップが落ちました。



IDEからテストを実行すると、便利な方法でテストをすばやくデバッグできます。 ただし、テストが継続的統合サーバーで実行されると、コンソールから起動されます。 マシンでこれを行う方法を見てみましょう。



開始するには、 0.13.8以上のバージョンの SBTを使用します。 sbtが存在しないか、バージョンが古い可能性があるため、linuxリポジトリからsbtをインストールすることは推奨されません。



sbtをインストールした後、コマンド"sbt --version"



実行してバージョンを確認します。



パラメーター "MaxPermSize"はsbt起動パラメーターで指定されており、8番目のjavaでは必要ないため、警告が表示されます。 無視するか、構成ファイルのSBT設定を変更できます。 「0.13.8」バージョンは私たちにぴったりです。 プロジェクトのSBTディレクトリに移動します。 その中で、コマンドを実行してテストを実行します-"sbt test"。 テストに合格し、最後のステップは緑、薄片状、赤になります。







標準のWindowsコンソールの場合も、エンコードの問題があります。 美しいロシア語のテキストを表示するには、ページのエンコードとフォントを変更する必要があります。 コマンドプロンプトウィンドウのプロパティに「Lucida Console」と入力します。







ウィンドウで、コマンド「chcp 1251」を作成します。 その後、「sbt test」を再度実行します。すべてが美しく表示されるはずです。







また、SBT構成ファイルを変更することにより、ロシア語の文字を正しく表示できます。 基本的な生地の部分は準備ができています。



サイトの状態をテストする例を考えてみましょう。 テストでは、ページが返され、「title」タグの内容が正しいことを確認します。

Get-requestとSeleniumスクリプトの2つの方法で確認しましょう。 Webドライバーを使用してテストを実装するには、「Firefox」が必要です（原則として、グラフィカルインターフェイスを必要としない「HtmlUnit」を含む他のブラウザーを使用できますが、通常、Firefoxでの問題は少なく、多くのLinuxディストリビューションの配信ではデフォルトで存在します）。





Seleniumスクリプトを実行するには、行「build.sbt」を追加します 
 libraryDependencies += "org.seleniumhq.selenium" % "selenium-java" % "2.46.0"


このアイデアは、Seleniumの作業のためにリポジトリからコンポーネントをプルし始めます。



Firefoxの更新後、「selenium-java」ライブラリに含まれるseleniumDriverが機能しない可能性があることに注意することが重要です。 たとえば、「2.46.0」はFirefoxの42バージョンでは機能しません。 バージョンを現在のバージョンに更新するか、バージョンフィールドで「latest.integration」を指定して、バージョンが毎回チェックされ、必要に応じて更新されるようにする必要があります。 



  libraryDependencies += "org.seleniumhq.selenium" % "selenium-java" % "latest.integration"


クラスを作成して呼び出します（たとえば、「GetTest」）。 必要なライブラリをインポートし、クラスの最初に「get」メソッドを定義してページのテキストを取得し、「FirefoxDriver」のインスタンスを作成します（ScalalaTestにあるSelenium DSLを使用するためのドライバーの暗黙的な値）。 



 import org.openqa.selenium.WebDriver import org.openqa.selenium.firefox.FirefoxDriver import org.scalatest.selenium.WebBrowser import org.scalatest.{Matchers, FreeSpec} import scala.io.Source class GetTest extends FreeSpec with Matchers with WebBrowser{ val pageURL = "http://scalatest.org/about" def get(url: String) = Source.fromURL(url, "UTF-8").mkString implicit val webDriver: WebDriver = new FirefoxDriver() "Get   %s   ".format(pageURL) in { get(pageURL) should include("<title>ScalaTest</title>") } "  " + pageURL+ "   " in { go to pageURL pageTitle should be ("ScalaTest") quit() } }


getメソッドは、Webサーバーからページコードを取得し、それを文字列に変換してから、「ScalaTest」サブストリングの存在を確認します。



2番目のテストでは、Firefoxでページを開き、テンプレートでタイトルを確認します。 ステップの名前は文字列なので、文字列を接着するか、formatコマンドを使用して変数を追加できます。 クラスを右クリックして実行します。 結果はフォームの下部で観察されます。







そのため、ページがロードされていること、および正しいタイトルを持っていることを確認するテストをすばやく作成できます。 すでにそのようなテストを使用してサイトを監視し、スケジュールに従って継続的な統合サーバーを使用してサイトを起動し、転倒した場合に関係者に通知を送信できます。



テストで実行できるアクションの例：



•GET / POST要求。 応答の内容を確認します。

•データベースクエリ。 応答の内容を確認します。

•メッセージの生成とキューへの送信。

•キューからメッセージを受信します。 応答の内容を確認します。

•Webインターフェイスの確認（作業の正確性、フォーム上の値の表示、スクリーンショットの取得）;

•コマンドラインスクリプトの実行（たとえば、残りのメモリ、ハードディスク領域の監視）。



システムをテストでカバーするには、各コンポーネントの検証（場合によっては、システムによるメッセージ処理の各段階の検証）を個別のステップまたは個別のテストクラスでラップして、問題を正確に特定し、有益なレポートを作成する必要があります。



次の記事で対処する問題：



•レポート生成。

•構成ファイルの接続。

•テストの並行起動。

•データベースへのリクエスト。

•キューマネージャーと連携します。

•Selenium WebDriverを使用したより複雑なシナリオ。

•システムコマンドの実行。

•時間の処理（一時停止、待機、実行時間の制限）。

•スタックのその他の機能。
