ホスティング事業者、モジュールを設置してください。

ホスティングには通常、このプログラムまたはそのプログラム用のプリインストールライブラリはありません。 また、ホスティング事業者は、必要なバージョンのライブラリをインストールする要求を常に満たすとは限りません。 ライブラリは存在しますが、古くなっています。 または、サーバーがホスティング事業者で故障し、サイトを別のサイトに投げました。サイトソフトウェアが実行されているライブラリと互換性のない新しいバージョンのライブラリがあります（Djangoユーザーは私を完全に理解します）。 どうする 新しいホスティングをもう一度お探しですか？ 常にプログラムを別のバージョンのライブラリに書き換える準備ができていますか？ 私たちの方法は、自分でそれを置くことです。 最悪の場合は、再インストールするだけです。 バイナリPHPモジュールに加えて、一般的なWebテクノロジーのほとんどすべてのライブラリは、ホストに触れたり、ホストに依存したりすることなく、ホームディレクトリに配置できます。 ちなみに、これは共有ホスティングだけでなく、仮想サーバーでも実行できます。プロジェクトは、仮想サーバーを誰がどのように構成したかにわずかに依存します。 難しいですか？



平均的なホスティングUNIXサーバーのホームディレクトリにライブラリを簡単かつ正常にインストールするために必要なものは何ですか？ SSHアクセスが望ましい。 それなしでも実行できますが、何らかの種類のWebシェルを探す必要があります。 ほとんどの場合、sshコンソールがchroot環境で機能しないか、ssh chroot環境がapacheの環境と一致するため、望ましくありません。 ホスティング業者にはSSHがありませんか？ これは考える理由です。 UNIXコンソールのスキルも必要です。 これは、一連の番号を携帯電話の短い番号に送信することほど難しくありません。 すでにサイトの開発を始めているのであれば、これを恐れないでください。



ナレーションのように、$ PREFIXマクロについてよく言及します。 最も一般的なユーザーエラーの1つは、通常、プレフィックスがライブラリファイルがインストールされるディレクトリを参照することです。 実際、ライブラリとプログラムの配布には、手動ファイル、ドキュメント、補助ファイル、バイナリ実行可能ファイル、およびライブラリファイル自体があります。 すべてのインストールシステムは、指定されたファイルに対して、$ PREFIXから始まる条件付きディレクトリ階層を作成します。 たとえば、Pythonモジュールはバイナリファイルを$ PREFIX / binに、手動ファイルを$ PREFIX / shared / manに、ライブラリファイル自体を$ PREFIX / lib / pythonX.X / site-packagesに分解します（XXはPythonインタープリターのバージョンです。ちなみに、バージョンごとに異なるライブラリを配置できます）。 デフォルトのディレクトリ階層に反対しないことをお勧めします。また、一般的なケースでは、$ PREFIX（システムに存在しない私が定義したマクロ）と$ HOME（ユーザーのホームディレクトリ、定義済みシェル変数）は同一ですが、誰もこれらの規則に従う義務はありません。



ライブラリを置くだけでは十分ではありません。サイトソフトウェアにライブラリの場所を伝える必要もあります。 このため、ここで説明するほとんどすべてのテクノロジーは環境変数です。 それぞれのテクノロジーについて、それらについて詳しく説明します。 しかし、それらのアプリケーションはすべてのユーザーに共通です。



sshコマンドラインの環境変数を設定するには（実際にはログイン中のコンソールに対して）、shおよびbashシェル（ほとんどのホスティングサイトではユーザーのメインシェル）の.profileファイルに書き込む必要がありますユーザーのホームディレクトリのルート、次の行：

VARIABLE=VALUE

export VARIABLE





たとえば、Python 2.5モジュールの場合、これは次のようになります。

PYTHONPATH=$HOME/lib/python2.5/site-packages

export PYTHONPATH





この場合、通常は事前定義された変数$ HOMEが使用され、シェルによって自動的に開かれることに注意してください。 shとbash以外のシェルでは、文字列が異なります。これらのシェルのドキュメントを参照してください。 シェル環境変数は、ライブラリをインストールする前に設定するのが最適です。 設定を適用するには、追加後に「ログイン」することを忘れないでください。



cronジョブで環境変数を確認するには、crontabファイルの先頭に書き込みます。

VARIABLE=VALUE





たとえば、vasyaユーザーのホームディレクトリにインストールされたPython 2.5モジュールの場合、これは次のようになります。

PYTHONPATH=/home/vasya/lib/python2.5/site-packages





さまざまなホスティング事業者のサーバー上のホームディレクトリへのパスは、上記とは異なる場合があります。 cronは変数を「開く」ことができず、パスはそのまま完全に記述する必要があるという事実に特に注意を払います。



Webサーバーの使用はもう少し複雑です。 特定のケースに該当するドキュメントをお読みください。 たとえば、Apacheを実行してvasyaユーザーのホームディレクトリにインストールされているライブラリを確認するPerl5 CGIスクリプトの場合、サイトのルートまたはCGIスクリプトがあるディレクトリの.htaccessファイルに次の行を書き込む必要があります。

SetEnv PERL5LIB /home/vasya/lib/perl5





この説明から、Apache Webサーバーで環境変数がどのように設定されているかが明らかです。

ただし、WSGIサーバー（任意）を実行するPython 2.5アプリケーションの場合、WSGI初期化ファイルに次の行を記述する方が適切です。

import sys

sys.path.insert(0,"/home/vasya/lib/python2.5/site-packages")





この例から、このような場合に他のテクノロジーをどのように扱うかが明確になります。



経験豊富なUNIXユーザーの場合、便宜上、$ HOME / binディレクトリを指定してPATH変数を展開し、コンソールからフルパスを導入せずにライブラリに付属するユーティリティが機能し、MANPATH変数（通常は$ HOME / share / man 、インストールされたモジュールのマニュアルページと通常のマニュアルページを使用できるようにするために-manコマンドを使用します。



CPANライブラリ（モジュール）をホームディレクトリ（Perl）にインストールする





CPAN（包括的なPerlアーカイブネットワーク-Perlアーカイブ）は、Perlに関連するすべての中央リポジトリであることを思い出させてください。 CPANモジュールを調べる1つの方法は、 search.cpan.orgにアクセスすることです。



インストールされたモジュールでスクリプトが機能するためには、PERL5LIB環境変数を設定する必要があります。 モジュールを検索するためのいくつかのパスを指定するには、コロンでリストします。次に例を示します。

PERL5LIB=/home/vasya/lib/perl5:/home/vasya/lib1/perl5:/home/vasya/lib2/perl5







CPANにあるモジュールは、手動でダウンロードしてコンパイルできます。 通常、コンパイル命令のシーケンスと必要なモジュールは、通常配布キットの一部であるREADMEファイルにリストされています。 すべてのCPANモジュールは、次のいずれかの方法でインストールできます。ExtUtils:: MakeMakerおよび/またはModule :: Build（詳細については、これらのモジュールのドキュメントを参照してください）。 ExtUtils :: MakeMakerはMakefile.PLファイルを使用します。 ホームディレクトリにインストールするには、環境変数INSTALL_BASEを指定する必要があります。

perl Makefile.PL INSTALL_BASE=~





モジュール::ビルドはBuild.PLファイルを使用します。 ホームディレクトリにインストールするには、-install_baseスイッチを指定する必要があります。

perl Build.PL --install_base ~





Tildaは通常、ユーザーのホームディレクトリの意味でシェルによって「明らかに」されます。これは、記事の冒頭で説明した$ PREFIXです。 チルダの代わりに、環境変数を適切に修正することにより、他のフルパスを指定できます。 残りのキーと優先インストール方法は、通常、モジュール配布内のREADMEおよびINSTALLファイルに記載されています。



Perlディストリビューションには、CPANと呼ばれるモジュールも含まれています。 これにより、依存関係のインストールなど、必要なモジュールのインストールを自動化できます。 モジュールは、手動モードとバッチモードで動作できます。 簡単にするために手動操作を検討してください。 モジュールと対話的に対話するには、次のコマンドを入力します。

perl -MCPAN -e shell





最初の起動時に、プログラムは構成ファイルを作成しようとし、質問をします。 問題は、答えがあると、通常はユーザーのホームディレクトリ内の.cpan / CPAN / MyConfig.pmディレクトリに作成される最終構成ファイルが、モジュールをローカルにインストールする準備ができていないことです。 この方法で入力する必要がある次の行に興味があります。

'build_dir' => qq[$ENV{'HOME'}/.cpan/build],

'cpan_home' => qq[$ENV{'HOME'}/.cpan],

'histfile' => qq[$ENV{'HOME'}/.cpan/histfile],

'keep_source_where' => qq[$ENV{'HOME'}/.cpan/sources],

'makepl_arg' => qq[INSTALL_BASE=$ENV{'HOME'}],

'mbuildpl_arg' => qq[--install_base $ENV{'HOME'}],

'prefs_dir' => qq[$ENV{'HOME'}/.cpan/prefs],





$ ENV {'HOME'}の代わりに、ユーザーのホームディレクトリへのフルパス、または他のパスがあります。例：/ home / vasya



構成が正しいことを確認した後、モジュールをインストールできます。 チーム

perl -MCPAN -e shell





特別なインタラクティブシェルを起動します。 モジュールのインストールは、たとえば次のコマンドを使用して実行されます。

force install _





forceキーワードは、さまざまな理由で実際にマイナスになる可能性があるネガティブなテスト結果の場合に、インストールの失敗を回避するのに役立ちます。



CPANモジュールは、CPANモジュールの管理に関するバッチ（ソフトウェア）作業の可能性も提供します。 モジュールの設定を細かくしたり、バッチ作業をプログラミングしたりするには、CPAN.pmのドキュメントに精通する価値があります。



PEARライブラリー（モジュール）をホームディレクトリにインストール（PHP）





PEAR（PHP拡張機能およびアプリケーションリポジトリ）は、PHPのフレームワークおよびコンポーネント配布システムであることを思い出させてください。 詳細情報はpear.php.netで見つけることができます



pearパッケージマネージャー自体はPHPディストリビューションの一部ではないため、ホストに事前にインストールする必要があります。



主な問題は、phpスクリプトが1つまたは別のコンポーネントをどこから取得するかを「見る」ことです（PHPは、いつものように独自の方法で処理されます）。 このためには、次のように、たとえばユーザーvasyaに対してinclude_path設定を定義する必要があります。

include_path=".:/home/vasya/pear"





最も簡単な方法は、PHPインタープリターにこの変数を伝えることです。

php -d include_path=".:/home/vasya/pear"





または、コンストラクトをスクリプトコードに挿入します。

ini_set("include_path",".:/home/vasya/pear");





これは、cronを介したスクリプトの実行にも適用されます。

スクリプトをApache Webサーバーの下から実行してPEARコンポーネントを含めるには、スクリプトを記述するか、記述する必要があります。

ini_set("include_path",".:/home/vasya/pear");





または、フォームの行を.htaccessファイルに追加します。

php_value include_path ".:/home/vasya/pear"







PEARモジュールをインストールする前に、コマンドを使用してホームディレクトリにコンポーネントをインストールすることをpearに伝える必要があります。

pear create-config $HOME .pearrc





構成ファイルが作成され、将来梨によって使用されます。 もちろん、これをinclude_pathの値に反映することを忘れずに、$ HOMEの代わりに他の場所を選択できます。



これで、必要なパッケージを簡単にインストールできます。 例：

pear install -o PEAR





依存関係のあるPEARシステムの基本コンポーネントをインストールします。



Pythonライブラリ（モジュール）をホームディレクトリにインストールします（Python）





インストールされたモジュールでスクリプトを機能させるには、環境変数PYTHONPATHを設定する必要があります。 モジュールを検索するためのいくつかのパスを指定するには、コロンでリストします。次に例を示します。

PYTHONPATH=/home/vasya/lib/python2.5/site-packages:/home/vasya/lib2/python2.5/site-packages







アーカイブからの手動インストール





通常、キーと優先インストール方法は、モジュール配布内のREADMEおよびINSTALLファイルに記載されています。 通常、これはコマンドです：

python setup.py install --prefix=$HOME





モジュール開発者が--prefixスイッチを無視した場合、問題が発生する可能性があります。



easy_installを使用したインストール





easy_installユーティリティー自体は、 pypi.python.org / pypiリポジトリーでパッケージを検索し、ダウンロードしてインストールできます。 または、既にダウンロード済みのインストール。 Pythonバージョン3.xには存在しません。 このユーティリティはPythonディストリビューションの一部ではありません。最初にsetuptools pypi.python.org/pypi/setuptoolsモジュールをインストールする必要があります。 easy_installを使用してモジュールをインストールするには、次の形式のディレクトリが必要です。

$PREFIX/lib/pythonX.Y/site-packages





ここで、XYはPythonのバージョンです。 たとえば、easy_installを使用してPython 2.5のモジュールをインストールするには、次のコマンドで適切なディレクトリ階層を作成する必要があります。

mkdir $HOME/lib $HOME/lib/python2.5 $HOME/lib/python2.5/site-packages





そのようなディレクトリが存在しない場合、easy_installはエラーをスローします。



その後、パッケージをインストールできます。

easy_install --prefix $HOME _







他のすべての場合と同様に、$ HOMEの例に示されているものの代わりに、環境変数に適切な修正を加えることにより、他のプレフィックスを使用できます。



RubyGemsをホームディレクトリにインストールする（Ruby）





RubyGemsはRubyプログラミング言語のパッケージマネージャーであり、Rubyプログラムとライブラリ、簡単なインストール管理用に設計されたツール、およびそれらを配布するためのサーバーを提供します。 パッケージは、gemsサーバーからインストールできます。 デフォルトでは、gemsはrubygems.orgサーバーでパッケージを探します。 RubyGemsはRubyディストリビューションに含まれていないため、ホスティング業者が事前にインストールする必要があります。



RubyGemsは、説明されている最も「スマートな」インストールシステムです。 コマンドを実行します：

gem environment





GEM PATHSキーワードの後に​​表示されるパスを確認します。 変数GEM_HOMEおよびGEM_PATHに興味があります。 gem自体はすべての変数を正しく定義していますが、念のため、明示的に記述してください。 これは、非標準の場所にパッケージをインストールする場合にも当てはまります。



gemを使用したインストールは非常に簡単です。 次のコマンドを入力するだけです：

gem install _

バイナリライブラリをインストールする





バイナリライブラリのインストールを形式化することは困難です。 ただし、ほとんどの場合、インストールに直面する必要はありません。

ディストリビューションには、コマンドなどの--prefixスイッチを受け入れる構成ファイルが含まれています

./configure --prefix=$HOME
