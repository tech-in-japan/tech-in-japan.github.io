RailsClub 2017：Nick Suttererのインタビュー。 Railsは死ぬ（Rubyは死ぬ）

こんにちは 今日はニックサテラーのインタビューを公開します。 NickはTrailblazerの作成者であり、Railsの貢献者であるRailsのいくつかの宝石と新しい抽象化レイヤーの作成者です。 ニックが講演するRailsClub 2017の前夜、 Dmitry Matveevは彼とRailsの開発に関する仕事について話しました。







今どこにいるの？



今、私はドイツにいます。ハノーバーで仕事をしていて、その前にオーストラリアに4年間住んでいました。 私はオーストラリアに滞在する予定はありませんでしたが、そこに滞在することは私の人生に大きな変化をもたらしました。サーフィンに夢中です！ サーフィンのために休憩して、プログラムのために再び座るのはクールです。 しかし、ここで多くの重要なことが起こっているので、私はヨーロッパに戻りました。この瞬間、私がここに住むことはより興味深いです。



どのようにしてプログラマーになることを決めたのですか、なぜ、どこで、いつですか？



8歳のとき、PCでゲームをするのが大好きで、自分でゲームを作りたかったのです。それが私の夢でした。 父が私に本をいくつか買ってくれて、私はプログラミングを「始め」ました。 私の父もプログラマーです。今ではそれが私たちの家族の伝統だと言えます。



私が学び始めた最初の言語はC ++です。 だから私は8年以来、中断することなくコードを書いてきました:)



コンピュータサイエンスの教育についてはどうですか？



大学なしで、高校でプログラミングを学び始めました。 それから、私はインターネットカフェでウェイターとして働いていたので、上司は会計ヨット用のデータベースを作成したいと考え、助けを求めました。 だから私はPHPに来ました。



Ruby on Railsに切り替えたのはなぜですか？



すべてのルールに従って、PHPとPerlでプログラミングしました。 私はOOPが好きで、シンプルで直感的なインターフェイスを備えた小さなオブジェクトを作成しようとしました。 同僚はかつてRuby on Railsを見せてくれました。 私は興味をそそられましたが、すぐにいくつかの建築上の欠陥を見ました。 私の友人は、コードを書くべき場所とその理由を明確に説明できませんでした。 しかし、Rails 1.1とActive Recordをいじり始めたので、言語自体が気に入りました。 個人的なプロジェクトにRubyを使い始めましたが、PHPはまだ開発中です。



あなたはまだ雇用で働いていますか？



いいえ、私は自分の会社を数年間持っています。 Trailblazerを開発し、Trailblazerが商業プロジェクトで使用し、建築上の問題の解決に役立つことを企業にアドバイスします。



お金のために仕事とオープンソースを組み合わせたい人にどんなアドバイスをすることができますか？



1日数時間、オープンソースで作業し、マネージャーに伝えないでください:)



冗談です。 お金のためだけでなく、コミュニティ全体の問題を解決し、それを支援することも重要です。 まあ、他の会社があなたのオープンソース製品を使用するなら、あなたはコンサルタントとして需要があるでしょう。 したがって、OSSでの作業は経済的に正当化できます。



RailsClub 2017の講演で 「Ruby is dead」と呼ばれるのはなぜですか？



私はMatzからRubyを死んだ言語と呼ぶ公式の許可を得ています:)もちろん、これは挑発です。 RubyとRailsを常に批判しているのはニュースではありません。 私はRubyが好きですが、他の言語と常に比較しています。 レポートでは、JavaとPHPの例を示し、過去10年間でどのように開発され、どの程度改善されたかを説明します。 しかし、Rubyは20年前と同じままであり、これは私たちへの警告であるべきです。 私はRubyにとどまりたいが、進化する必要がある！ 多くの人は、問題を解決する言語で作業したいので、別のスタックに切り替えます。 私の問題は、安定した信頼できる商用アプリケーションを作成する方法です。 はい、Rubyでは、たとえばタイピングサポートなどの新機能を追加するプロセスは非常に困難です。これは、主に下位互換性のサポートが原因です。 しかし、他の言語でなんとかできました！



レポートでmrubyについても説明します。 これは非常に興味深いトピックで、Matzは現在取り組んでいます。 私の意見では、Rubyの主な問題は、アプリケーションから安定した動作を実現することが難しいということです。 それはほとんど不可能です!!! 常に「メソッドが見つかりません」と表示されますが、これが主な生産エラーです。 Matzの「ダック推論」タイピングをRubyに追加する計画は非常に有望です。 入力は本当に便利です。



レールのもう1つの問題は、構造化されていないコードです。



Railsを使用すると、2時間で簡単なアプリケーションを作成できますが、大規模なアプリケーションの場合、これは機能しません。 最初の2週間は幸せですが、それから怒り始めます。 Railsはデモンストレーションに最適です。ある日、初心者が実用的なアプリケーションを作成し、とても喜んでいます。 しかし、問題は、Railsで安定したものを書きたい場合、遠くまで行かないということです。 シンプルで信頼性の高い両方を同時に書きたいのですが、Railsの助けを借りてそれを行うのは難しいでしょう。



OOPの主な問題は、1つのオブジェクトの機能が多すぎることです。 オブジェクトには300個のメソッドが必要だとは思いません。 関数型言語では、すべてに対して個別の関数を作成する必要があり、これは、私の意見ではより優れています。 そうすれば、内部状態を心配する必要がなくなり、OOPで問題がしばしば発生するのはまさにそのためです。



なぜロシアを訪問することにしたのですか、何を期待していますか？



ここが私の最初の旅行です。 Rails Clubにはたくさんのクールな人々とたくさんの素晴らしいスピーカーがいます。ロシアのアフターパーティについての多くの話も聞きました... :)



また、サンクトペテルブルクに会いたいので、ロシアに約1週間半滞在し、大きなインスピレーションを得ることを望んでいます！



Trailblazerについて詳しく教えてください。



楽しい事実-ロシアはTrailblazerの最大のユーザーベースを持っています！ これは、RailsやHanamiなどのフレームワークで使用できる抽象化のセットであり、オブジェクトやポリシーなどから新しい概念を追加し、その結果、ビジネスロジックとワークフローをすべて一緒にリンクします。



今、私は警察のためのプロジェクトに取り組んでいます。コードの60％は、アクセス権を制御するif-else条件のセットです。 Trailblazer 2.1では、BPMNおよびワークフロー図のサポートが追加されます。 これにより、アプリケーションコードから条件コードを抽出できます。 会議で例を示します！



Ruby / Ruby on Railsの将来の見通しは？



難しい質問です！ 私の意見では、Railsは死にかけています。私はこのフレームワークについてあまり楽観的ではありません。 過去10年間に新しい抽象概念も革新もありません！ 何か新しいことをしようとする愛好家もいますが、コアフレームワーク開発チームはこれに特に興味はありません。 レールが詰まっています。



しかし、Rubyには希望があります。たとえば、JRubyなど、多くの新しい実装があります。 したがって、少なくとも今後20〜30年以内にRubyが生き残ると思います。



しかし、私にとってRailsについての良いニュースがあります-より多くの人々がそれらを使用するほど、より多くの人々が彼らの欠点に遭遇し、したがって彼らはTrailblazerを使い始めます:)



新しいプロジェクトの宝石のセットはありますか？



もちろん！ Trailblazerでは、Tamaramaと呼ばれるスタックを使用します。 次のもので構成されます。



-ルーティング用のシナトラ

-ビジネスロジックの先駆者

-データベースの続編またはアクティブレコード

-プレゼンテーションレイヤーのセル



Rubyが遅いという事実について、あなたは何を言うことができますか？



より高速な言語がたくさんあることは知っています。 しかし、問題はないと思います。 たとえば、TamaramaまたはTrailblazerを使用して、Rubyアプリケーションを高速化できます。 そして、はい、Rubyは遅いですが、Railsはそれをさらに遅くします！ 個人的には、Railsを使用していないため、Rubyのアプリケーションのパフォーマンスに問題はありません。

さらに、Rubyはこの方向で絶えず改善されており、言語の最新バージョンははるかに高速になっています。



好きなブログやウェブサイトをお勧めしますか？



私は定期的にブログやニュースサイトを読んでいません。私は前世紀のように住んでいます:)何か貴重なものを見逃すかもしれませんが、不必要な情報の束を取り除きます。 ここに私のヒントがあります：



-他の人の言うことを信頼せず、自分を信頼してください。

-コードをシンプルにします。

-車輪を発明しようとしないでください。



プログラミング本はどうですか？



プログラミングに関する本は2冊しか読んでいません。これらはMartin Fowler's RefactoringとTCP / IP Illustrated、Vol。です。 1.それは10年以上前でした。 TCP / IP Illustrated、Vol。 1.プログラミング、階層化、カプセル化に関する多くの基本的な知識を学びました。 お勧めです。 そして、マーティン・ファウラーは一般的にソフトウェア開発の世界で私のイエスです。彼が壁にぶら下がっている写真さえ持っています。



プログラミングに興味を失わないようにするにはどうすればよいですか？



私は人生と仕事のバランスがとても良いです:)プログラミング以外の多くの興味-食べ物、料理、スポーツ。 そして最も重要なのは、私が本当に愛するものに取り組んでいるということです。 オープンソースでの仕事とコンサルティングは、私が本当に好きなことです。 ですから今、私はこれまで以上に燃え尽きません。
