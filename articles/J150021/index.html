実用的なメタメタモデリング

宣言型アプローチとMDAアーキテクチャには、情報システム（IP：CRM、WMS、プロジェクト管理など）の開発と保守のコストを大幅に削減できる多くの利点があります。 このアプローチは、多くの製品（1Cなど）ですでに使用されています。 それにもかかわらず、それらの宣言的アプローチは、あまりにも狭いタスクのサークルを解決するために使用されます。 この記事では、宣言的アプローチの利点を検討し、IPの構築におけるアプリケーションの適用範囲を大幅に拡大し、実際の問題に基づいて構築されたモデルを確認し、プロトタイプの動作を実証する方法を示します。



私の学士号と修士号はMDAに関連しており、私たちはこれらのアイデアを情報システムの構築に適用するために、1年前からこれらの元クラスメートと協力してきました。 私たちは商用製品を代表していません。私たちがやった/思いついたものはすべて、自由時間に「ひざまずいて」開発されました。



私たちのアイデアは、情報システムの複雑な設計者（1Cなど）とWebフレームワーク（Django、RoR）の両方に適用できます。 あなたの意見やコメントを知ることは興味深いです。 さらに、製品でベストプラクティスを活用するために、協力に関心のある企業を探しています。



それがすべて始まった方法



それはすべて、偶然にも、ワークフローオフィス用の小さなCRM基盤を作成するというタスクが発生したという事実から始まりました。 MS AccessからWebフレームワーク（Django、RoR）まで、さまざまなソリューションを検討し始めました。 しかし、それらはすべて低レベルであることが判明しました。 データベースオブジェクトではなく、「ユーザー」、「イベント」、「タスク」などの観点から作業し、対応する視覚的表現（カレンダー、フィルター付きリストなど）が必要でした。



私たちが提供するもの



情報システムの開発のために、最も一般的に使用されるオブジェクトリレーショナル表記法（OORDBMS、ORM）。PLの1つでビジネスルールが強制的に記述されます。 オブジェクトリレーショナル表記法を、ビジネスルールを宣言的に記述することを可能にする新しい概念で拡張することを提案します。 「実装」の言語を、要件の形式化の言語に可能な限り近づけるよう努めています。



ビジネスルールの宣言的記述が重要なのはなぜですか？



宣言的アプローチと相まって、高レベルの抽象化を使用すると、IPの開発と適応を単純化および加速するだけでなく、 コードレベルからデータレイヤーへのビジネスルールの転送は、開発プロセスだけでなく製品のプロパティにも定性的に影響します。

  • 新しいバージョンのデータベース更新の部分自動化
  • オンザフライシステムアップデート
  • ビジネスアナリストまたはエンドユーザーによるビジュアルデザイナーでのシステムのカスタマイズ
  • 1つのSaaSサービス内で、動的に拡張可能な構成セットをサポートできます


最後の点は特に注目に値します-現時点では、クラウドソリューションの「構成可能性」は非常に低いレベルのままです-最良の場合、既存のエンティティにいくつかの属性を追加できます。 データ領域にビジネスルールを転送すると、サービスのさまざまなユーザーが、サーバーに干渉することなく、このデータをまったく異なるものにすることができます。 クラウド内の各ユーザーは、自分専用のシステムを持つことができます。



タスクの例



IPの設計で発生し、言語を構築するときに反発する特徴的なタスクのセットを考えます。

  1. 表示コスト（価格*数量）、価格に基づくVATを含む価格、VATレートなど ユーザーインターフェイスでは、これらの値は自動的に再計算され、サーバーは受信したデータの正確性を確認する（または自分で計算する）必要があります
  2. より複雑な例：すべての商品の合計購入価格。所定の価格表で推定されます。 サポートは、クライアント側とサーバー側の両方で行う必要があります。
  3. カレンダー上のさまざまなソースによって生成されたイベントを表示します：ユーザーの誕生日、集会の始まり、マイルストーンの終わりなど。
  4. システムの「ユーザー」の構造を特化する
  5. 顧客や会社の従業員など、ユーザーのカテゴリごとに異なるパラメータのセット
  6. 異なる構造を持つオブジェクトでのコードの再利用。

    たとえば、商品の償却、移動、および転記の構造は異なりますが、アクションは同一です
  7. ビジネスルール：

    • 期限切れの各タスク日のエグゼキューターに対するペナルティ
    • 使用期間と1日の費用に基づいた家賃の計算
  8. 通知：

    • その完了のタスクのディレクター
    • 高価な機器を償却する場合の取締役
  9. 承認後の価格表の固定
  10. マネージャーは、自分に割り当てられている顧客のみを表示できます。




新しいコンセプト



オブジェクトリレーショナルモデルに追加する基本概念を考えてみましょう。



クラス



オブジェクトリレーショナル言語にはクラスが既に存在しますが、この概念を大幅に変更し（同時に拡張および切り捨て）、 ER表記の「本質」の概念に近づけました。

  • クラスにはフィールドのみがあります（メソッドはありません）
  • クラスは継承を許可します（また、オブジェクトリレーショナル拡張SQLのように、基本クラスを照会すると、基本クラスだけでなく派生クラスのオブジェクトも返されます）
  • クラスは抽象クラスです（つまり、インスタンスを持つことはできません）。 抽象クラスには、非抽象子クラスに「実装」する必要がある抽象フィールドを含めることができます。 計算可能な式を指定するか、単純な保存フィールドを宣言します

    継承中にフィールドタイプを指定できます（クラスフィールドがクラスAを参照する場合、継承中にBがAの後継である場合、クラスBへの参照を指定できます）。
  • 参照フィールドは次のように分類されます

    • 参照資料
    • 構成（UMLの構成の類似物）-「構成可能な」オブジェクトの有効期間は、「構成」オブジェクトの有効期間によって制限されます
    • 包含（よく知られているプログラミング言語またはモデリング言語で類似物に出会ったことはありません）は構成の拡張です-埋め込みオブジェクトは親の範囲外では考えられません


「包含」の例は、リレーションシップです：クラスとそのフィールド、テーブルとその属性、ユーザーとそのプロファイル。 オブジェクトの「クラスフィールド」は独立していないため、親クラスの外部に存在することはできません。



計算可能なフィールド



XPathに似た言語の式をオブジェクトフィールドの値として設定する機能。 計算可能な式と計算可能なフィールドの概念は、言語の他のすべての拡張機能で使用されます。 ORMフレームワークまたはPL / SQLのユーザーはget / setメソッドを実装するときに言語の機能を使用しますが、これはまったく同じではないことに注意してください 計算情報はチューリング完全言語で記述されます。 式の言語を特にチューリング完全ではないように単純化し、次のようにします。

  • データへの異なるアクセス方法を使用する異なるプラットフォーム（クライアント、サーバー）で効果的に解釈する
  • 依存オブジェクトの依存フィールドを変更するときに式を再計算するための依存関係を自動的に検出


したがって、フィールドの計算方法に関する情報は、オブジェクトの構造の説明に含まれています。これにより、この構造が自動的に許可されます。

  • サーバー上のオブジェクトを検証する
  • ユーザーインターフェイスの計算フィールドを自動的に更新する




インターフェース



インターフェイスはOOPでよく知られていますが、データベースを記述する際に不当に忘れられています。 次のようにインターフェイスを紹介します。

  • インターフェイスは、すべてのフィールドが抽象である抽象クラスです
  • インターフェイスの実装は継承と同じです
  • インターフェイスはクラスと完全に互換性があります。クラスを使用できるところならどこでも、インターフェイスを使用できます（リレーショナル理論の表現との類推によって）。

    つまり、次のことができます

    • インターフェイスで選択を行います（選択の結果は、指定されたインターフェイスを実装するすべてのオブジェクトです）
    • インターフェイスを介してオブジェクトを変更します（インターフェイスフィールドがクラスフィールドへのバインドを通じて実装されている場合）
    • インターフェイスのトリガーを作成します（この場合、トリガーはすべての実装クラスで機能します）
  • クラスはインターフェイスを「提供」できます。 このために、インターフェイスを「提供」するクラスのフィールドを使用して、指定されたインターフェイスを実装する「接続された」クラスが記述されます。

    • クラスは同じインターフェースを複数回提供できます（「グラント」と呼ばれます）
    • 継承とは異なり、「グラント」はクラスの外で記述することができます（「アダプター」パターンで類推できます）
    • 「リンクされた」クラスの構造については、継承と同様に、フィールドタイプを指定することができます


「インターフェースを提供する」という概念は、重要な再利用ツールです。 インターフェイスの実装と同様に、リレーショナル表現と比較できます。 「プレゼンテーション」とプレゼンテーションの大きな違いは計算方法です。異なるエンティティが同じインターフェイスを複数回提供できるため、1つのSELECTですべての「配信」のリストを取得することは非常に困難です（たとえば、カレンダーにすべてのイベントを表示する） 。



言語表現テスト



導入した設計を使用して、上記のタスクを実装する方法を検討してください。

  1. コストを数量ごとの価格、VATを含む価格の積として表示する


    これは完全に「計算可能なフィールド」の概念に当てはまり、式を記述する言語はもちろん、合計と積を記述することを可能にします:)
  2. より複雑な例：すべての商品の合計購入価格。所定の価格表で推定されます。


    購入は、フィールド「総コスト」とオブジェクトのリスト「goods-quantity-value」を含むフィールド「goods」を持つオブジェクトとして記述されます。

    式によって各オブジェクト「product-quantity-value」のフィールド「cost」を設定します。

    • 購入から価格表を取ります
    • 私のものと一致する製品の価格表をフィルタリングする
    • 結果リストから最初で唯一の要素を取得します
    • 彼から価格フィールドを取得します。


    「購入」クラスの「総コスト」フィールドで、「すべての製品レコードの合計-計算フィールドによる数量-コスト」の式を設定します。

    システムのタスクは、依存関係の推移的な閉包を構築することです。そのため、エントリ「product-quantity-value」のいずれかで「quantity」を変更すると、総コストが再計算されます。

  3. カレンダー上のさまざまなソースによって生成されたイベントを表示する


    これを行うには、フィールド「名前」、「日付」、「期間」、「期間」で「イベント」インターフェースに入ります。 ユーザーインターフェイスコードは、常にイベントインターフェイスを使用してイベントを検索および表示します。

    カレンダーがユーザーの誕生日を「見つける」ためには、ユーザーによる「イベント」インターフェースの「プロビジョニング」を次のように記述する必要があります。

    • 「名前」 -「誕生日$名$姓」という表現、
    • 「日付」 -user.birthday
    • 「期間」 -定数式「1年」
    • 「期間」は「終日」という定数表現です
  4. システムの「ユーザー」の構造を特化する


    これを行うために、インターフェイス（たとえば、「ログイン」フィールドと「パスワード」フィールド）で「ユーザー」の概念を説明します。 インターフェイスは、操作の点でクラスと区別がつかず、インターフェイスは任意の構造を持つクラスによって実装できます
  5. 顧客や会社の従業員など、ユーザーのカテゴリごとに異なるパラメータのセット


    さまざまなクラスが同じインターフェースを実装できるため、ユーザークラスはいくつあってもかまいません。 唯一のことは、認証コードがインターフェースを介してユーザーとのすべての作業を行うことです。
  6. 異なる構造を持つオブジェクトでのコードの再利用。


    インターフェイスまたは抽象クラスによって明らかに実装されます。

    実装中のフィールドのタイプを明確にすることができることに注意することが重要です。インターフェース「移動」があり、フィールド「在庫」、「数量」を持つ「移動レコード」などのリストフィールドがある場合、インターフェースの実装中のエンティティ「購入」はリストフィールドは、タイプが「購入記録」でフィールドが「在庫」、「数量」、「購入額」のフィールドであり、これは転送の処理コードには影響しません。

SaaSモードをサポートするシステムを作成したため、1台のサーバーが多数の「プロジェクト」を持つことができ、各プロジェクトには独自の操作対象オブジェクトクラスがあります。 つまり、1つのグループのユーザーがCRMを持ち、2番目のグループが在庫を持ち、3番目のグループも在庫を持ちますが、レストランビジネスに特化しており、これらすべてを1つのサーバーで実行できます。



作業の結果、3つの「アーティファクト」を形成しました。

    • 構造説明
    • プレゼンテーションの説明

  • サーバー： Java + Jetty + MongoDB。 MongoDBは、スキーマを使用しない設計により使用されており、異なる構造を持つデータを1つのコレクションに格納することが容易になりました。 ただし、重大な欠点（トランザクションとJOINの欠如、集約の遅延）のため、「大きく大人になった」ときにRDBMSに切り替える予定です。
  • クライアント： Javascript + MooTools。 プレゼンテーション全体がクライアント上で形成され、REST APIサーバーを介してデータにアクセスします。 クライアントは、言語モデルの操作の実装を繰り返しますが、クライアントが「薄い」という事実を考えると、特にビューの形成のために、単純化された形式で実行されます。 従来の「テキスト」テンプレート化とは異なり、ツリーのDOMレベルでテンプレート化を使用します（このアプローチは非常にまれであり、興味深い場合はそれについて説明します）
