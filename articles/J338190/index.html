ウィキペディアのデータに基づいて家系図を生成する

この記事では、 Selenium Webdriverフレームワークを使用して、 Wikipediaのデータに基づいて、特定の人物（たとえば、 Rurikのロシアの支配者の最初の王朝の伝説的な創設者）の家系図を作成する方法を示します。



この記事では、人の名前を判別する方法、人の子供のページへのリンクを計算する方法、および家系図を生成するためのアルゴリズムを構築する方法を説明します。

Java 、 Selenium Webdriver 、 Chromeを使用します 。 Chromeは、他のブラウザーよりも高速であり、URLをナビゲートすることがプログラムで最も時間のかかる操作であるため、ブラウザーの選択が時間に最も大きく影響するためです。 PhantomJsを使用すると、ブラウザーを完全に放棄して使用できますが、デバッグが難しくなります。 だから私はChromeに落ち着きました。



まず、ブラウザが正常に起動したこと、およびURL https://ru.wikipedia.org/wiki/Rurikにアクセスすると、「Rurik-Wikipedia」という見出しのページが開くことを確認するテストを作成します。

子孫検索アルゴリズム



それでは、最も興味深い部分、つまり特定の人物の子孫を見つけるためのアルゴリズムの開発に移りましょう。 mainメソッドでGenerateGenealogicalTreeクラスを作成します。



既に述べたように、最も時間がかかるのはURLの移行であるため、これらの移行の数を最小限に抑える必要があります。 これを行うには、家系図全体が保存される個人のリストを作成します。 このリストで、現在の人のインデックスを覚えておいてください-現在ページにいる人。 インデックスが低い人はすべて「訪問済み」とみなされ、インデックスが高い人（+現在）はすべて「訪問されていない」とみなされます。 現在の人のページに移行し、彼女の基本データが計算された後、インデックスが1つ増加します。 したがって、現在の人は「訪問済み」のカテゴリに分類されます。 そして、残りの「訪問されていない」人をバイパスするだけです。 いつでも、ページがすでに閲覧されている人は知られています。



子孫のリストの最後に現在の人を追加することにより、家系図に新しい「訪問されていない」人がいっぱいになります。 この場合、リストにまだない子供だけを追加して重複がないようにします（このような状況は、夫と妻が両方とも異なる支部の王朝の祖先の子孫である場合に可能です。例： 夫と妻はルーリックの子孫であり、 夫と妻は子孫ですポールI ）。



家系図は、「訪問されていない」人がいないときに構築されると考えられます。 現在の人物のインデックスが家系図のサイズと等しくなったとき。



アルゴリズムは次のとおりです。



  1. 指定されたURLに基​​づいて王朝の創設者を作成します
  2. 家系図は、王朝の創設者に基づいて作成されます
  3. 「訪問されていない」人がいる限り、サイクル内
  4. 個人は、家系図の現在のURLに基​​づいて計算されます。 この人物は現在の人物として設定されています。
  5. 現在の人物が重複していない場合、彼女の子供のリストが計算され、確立されます。 すべての子がリストに追加されます。
  6. 現在の人がすでに「訪問した」人の間で会っている場合、彼女は削除されます。
  7. 次の「訪問されていない」人への移行があり、それが「現在」と見なされます。


アルゴリズムコード： 



 public final class GenerateGenealogicalTree {    public static void main(String[] args) throws Exception {        String url = getUrl(args);        GenealogicalTree tree = getGenealogicalTreeByUrl(url);        saveResultAndQuit(tree);    }    public static GenealogicalTree getGenealogicalTreeByUrl(String url) throws MalformedURLException {        WebDriver driver = DriverHelper.getDriver();        Person person = new Person(url);        GenealogicalTree tree = new GenealogicalTree(person);        PersonPage page = new PersonPage(driver);        while (tree.hasUnvisitingPerson()) {            String currentUrl = tree.getCurrentUrl();            Person currentPerson = page.getPerson(currentUrl);            tree.setCurrentPerson(currentPerson);            if (!tree.isCurrentPersonDeleted()) {                List<Person> children = page.getChildrenUrl();                tree.setChildren(children);            }            tree.updatingCurrentPerson();        }        driver.quit();        return tree;    } }


GenealogicalTreeクラスには、3つのフィールドがあります。List <Person> allPersons-家系図のすべての代表のリスト、int indexCurrentUnvisitedPerson-allPersonsリスト内の現在の人物のインデックス、およびブールisCurrentPersonDeleted-「現在の」人物が削除されたかどうかのサイン複製）。 



 public final class GenealogicalTree {    private List<Person> allPersons;    private int indexCurrentUnvisitedPerson;    private boolean isCurrentPersonDeleted; }


初期化は王朝の「祖先」に基づいて行われます-私たちが探している子孫を持つ最初の人： 



 public GenealogicalTree(Person person) {    if (person == null) {        throw new IllegalArgumentException("   ");    }    allPersons = new ArrayList<Person>();    allPersons.add(person);    indexCurrentUnvisitedPerson = 0;    isCurrentPersonDeleted = false; }


この時点で、家系図は現在の「未訪問」の一人で構成されています。 「訪問された」人はいない。



すでに述べたように、「訪問されていない」人の存在のリストのチェックは次のように実行されます。現在の人のインデックスが「終わりに達した」場合、「訪問されていない」人はいないと考えられます。 



 public boolean hasUnvisitingPerson() {    return indexCurrentUnvisitedPerson < allPersons.size(); }


血統ツリーのURLは、現在の人のURLです。 



 public String getCurrentUrl() {    return allPersons.get(indexCurrentUnvisitedPerson).getUrl(); }


setCurrentPersonメソッドは、現在の人を特定の人に置き換えます。



最初は、親のページから取得したURLのみを知っています。 したがって、この情報だけを持つ人が家系図に追加されます。 実際、すべての「未訪問」の人は単なるURLです。 setCurrentPersonメソッドは、インデックスが「彼女に到達」し、その人物が最新になった後、その人物を「明確にします」。



インストール済みの「洗練された」人が既に会ったことがある場合（これは、現在の人のURLから以前に遭遇したページの1つにリダイレクトが発生した場合に可能です）、現在の人は削除されます。 その後、現在の人物は削除済みとしてマークされます。 指定された人物が以前に発生していない場合、彼女は現在の人物を「置き換え」ます。 この場合、個人は削除されたとは見なされません。



「以前に会う」という概念は、「訪問した」人だけをチェックすることを意味します。 「未訪問」はチェックされません。 理論的には、現在の人のURLがURLにリダイレクトされると、「未訪問」の「後で」発生する可能性があります。 しかし、これは非常にまれな状況であり、アレイ全体を毎回「実行」する価値はありません。 このまれなケースでは、キューが「到達」したときに重複が削除され、現在の人のインデックスはリダイレクトが発生したURLを持つ人を示します。 



 public void setCurrentPerson(Person currentPerson) {    int indexDuplicate = allPersons.indexOf(currentPerson);    if ((0 <= indexDuplicate) && (indexDuplicate < indexCurrentUnvisitedPerson)) {        removePerson(indexDuplicate);    } else {        allPersons.get(indexCurrentUnvisitedPerson).copyMainData(currentPerson);        isCurrentPersonDeleted = false;    } }

リストに人の存在を常に確認する必要があるのはなぜですか？

重複の発生は、多くの理由で可能です：



  1. 父性は確実に不明です。 たとえば、 Svyatopolk the Accursedの場合、父親はYaropolk SvyatoslavichまたはVladimir Svyatoslavichのいずれかです。
  2. 両方の親は、異なるブランチのRurikの子孫です。 例： ルリックの8世代目の子孫であるグレブヴセスラビッチは、 同じくルリックの子孫であるアナスタシア ヤロポルコヴナと結婚しました（4人目のいとこであり姉妹です）。
  3. ページの誤り： Vsevolod Mstislavichが息子のVolodar Glebovichを持っているのは疑わしい。彼の両親は他の人も記録しており、Rurikovich王朝に属していた。 ほとんどの場合、Wikipediaの単なる誤植です


これらの重複が排除されない場合、新しい繰り返しが生成されます。 重複のすべての子孫は、2回または3回さえ回避されます（ Volodar Glebovichの場合）。



次に、重複している人をリストから削除することを検討してください。 削除される人は、家系図のメンバーの子のリストに含まれている場合があります。 たとえば、両方の親が同じ王朝の代表者である場合、一方の親は「子」の1つのページへのリンクを持ち、もう一方は最初のページにリダイレクトされます。 重複が「単に」削除された場合、2番目の親は存在しない人物へのリンクを持つことになります。



したがって、現在の人物を削除する前に、すべての「訪問済み」人の子供の識別子のリストで見つかった一致の識別子を置き換える必要があります（「未訪問」の子供はそうではありません）。



削除後、現在の人は削除済みとしてマークされます。 



 private void removePerson(int indexDuplicate) {    int idRemovedPerson = allPersons.get(indexCurrentUnvisitedPerson).getId();    int idDuplicate = allPersons.get(indexDuplicate).getId();    for (int i = 0; i < indexCurrentUnvisitedPerson; i++) {        Person person = allPersons.get(i);        person.replaceChild(idRemovedPerson, idDuplicate);    }    allPersons.remove(indexCurrentUnvisitedPerson);    isCurrentPersonDeleted = true; }


Personクラスで、「子」置換メソッドを追加します。 



 public void replaceChild(int oldId, int newId) {    if (oldId == newId) {        return;    }    if (!children.contains(oldId)) {        return;    }    children.remove((Object) oldId);    setChild(newId); }


現在の人に子供を追加することを検討してください。



入り口では、子供として最新に設定する必要がある人々のリストを取得します。

重複の検索の主な違いは、「訪問済み」の人だけでなく、「未訪問の」人の間でも検索することです。 家系図全体の中。

現在の人物が削除されると、例外がスローされます。 実際、子供をインストールする人はいません。



削除されない場合は、パラメーターとして渡されたリストを調べます。 子がすでに家系図に含まれている場合、見つかった重複の識別子が子のリストに追加されます。 子が家系図にない場合、その識別子が子のリストに追加され、さらに子自身が家系図の最後の「訪問されていない」人のリストに追加されます。



したがって、setChildren（）メソッドを使用すると、リストに「データが入力されます」。 



 public void setChildren(List<Person> children) {    if (isCurrentPersonDeleted) {        throw new IllegalArgumentException(            "    .    ");    }    for (Person person : children) {        int index = allPersons.indexOf(person);        int id;        if (index >= 0) {            id = allPersons.get(index).getId();        } else {            allPersons.add(person);            id = person.getId();        }        allPersons.get(indexCurrentUnvisitedPerson).setChild(id);    } }


現在の人物のカウンターを更新する必要があります。そうしないと、家系図が作成されません。 これは次のように発生します。現在の人物が削除された場合、次の「訪問されていない」人物は既に彼女の場所にいるので、削除された人物のサインを現在の人物から単に「削除」するだけで十分です。 現在の人物が削除されていない場合、すべてのデータが「満たされている」とみなし、次の「訪問されていない」人物に進みます。 



 public void updatingCurrentPerson() {    if (isCurrentPersonDeleted) {        isCurrentPersonDeleted = false;    } else {        indexCurrentUnvisitedPerson++;    } }


クロールは世代ごとに実行されます：最初に王朝の創始者（0世代）、次に彼のすべての子供（1世代）から最年長から最年少（ウィキペディアのURLがその順序にある​​ことを意味します）、次に孫（2世代） ）（年長の長男の子、次に次男のように最年少まで）、great孫（第3世代）など、最後の王朝の代表者まで。



当然、すべてが意図したとおりに機能することを確認するために、テストでコードカバレッジを最大100％にすることを忘れないでください。 テストの説明はjavadocで入手できます。



ここで言及する価値のあることの1つは、GenealogicalTreeクラスは非常に安全ではないため、系図ツリー生成アルゴリズムの外部（GenerateGenealogicalTree以外）で使用すると、簡単に正しく動作しない可能性があることです。 この状況での唯一の正しい解決策は、GenerateGenealogicalTreeの内部プライベートクラスとしてこのクラスを転送することです。 しかし、アルゴリズムのテストの利便性のために、これはまだ行われていません。
