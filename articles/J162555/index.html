Nodejsアプリケーションのテスト

前回、フレームワークとしてexpressjsを使用し、テンプレートエンジンとしてjadeを使用して、nodejsでアプリケーションを作成することについて書きました。 今回は、サーバー側のテストに焦点を当てたいと思います。



テストには、次のものを使用します。

-Mocha-テストを記述し、簡単かつ簡単に実行できるフレームワーク。 さまざまなバージョンのレポートを生成し、テストからドキュメントを作成する方法も知っています。

-Should- 「承認」のスタイルのテスト用ライブラリ（正しい名前が見つかりませんでした）

-SuperTest - nodejsで HTTPサーバーをテストするためのライブラリ

-jscoverage-テストでコードカバレッジを評価する







すべてをゼロから作成したわけではありませんが、前の記事のアプリケーションをテストでラップし、app2フォルダーに完全にコピーすることにしました。



まず、必要なモジュールをpackage.jsonファイルに追加します。 必要なのは、mocha、should、supertestです。

package.json 
{ "name": "app2", "version": "0.0.0", "author": "Evgeny Reznichenko <kusakyky@gmaill.com>", "dependencies": { "express": "3", "jade": "*", "should": "*", "mocha": "*", "supertest": "*" } }


npm iコマンドを実行して、必要なすべてのモジュールをインストールします。

jscoverageをインストールします（Ubuntu sudo apt-get install jscoverageの下 ）。

MochaのBDDテストは、 describe（）ブロックで始まります。 、テスト自体はit（）ブロックに書き込まれます。 describe（）ブロック内に配置する必要があります。 describe（）ブロックを相互にネストすることができます。 フックも使用できます：before（）、after（）、beforeEach（）、afterEach（）。 フックは、describe（）ブロック内にも記述する必要があります。 偽のデータベースを操作するモデルをテストするときのフックについて詳しく説明します。

アプリケーションの残りをさらにテストする前に、カバレッジテストを追加します。

パラメーター--encoding = utf8および--no-highlightを 使用してjscoverageの実行を追加し、着信ディレクトリーとしてlibを、発信ディレクトリーとしてlib- covを指定します。 ここで、カバレッジテストのMocha起動を追加し、環境変数COVERAGE = 1を設定し、レポーターとしてhtml-covを指定して、カバレッジテストの結果を含む美しいhtmlページを取得します。

メイクファイル

アプリケーションをテストする


テストでカバーされない最後の部分がありました。 これは直接HTTPサーバーです。 ここでは、4つの状況のみをフリーズしてテストすることにしました。

1）これが通常のリクエストである場合、答えはhtmlに来る必要があります

2）それがajaxの場合、答えはjsonに来る必要があります

3）サイトのルートへのGETリクエストは、タイトルに値「My site」が含まれるページ/オブジェクトを返す必要があります

スーパーテストライブラリのおかげで、このようなテストの作成は簡単で簡単です。

test / app.js 
 var request = require('supertest'), app = process.env.COVERAGE ? require('../lib-cov/app.js') : require('../lib/app.js'); describe('Response html or json', function () { //   ,   //   html it('should be responded as html', function (done) { request(app) .get('/') .expect('Content-Type', /text\/html/) .expect(200, done); }); //  ,   json it('should be responded as json', function (done) { request(app) .get('/') .set('X-Requested-With', 'XMLHttpRequest') .expect('Content-Type', /application\/json/) .expect(200, done); }); }); describe('GET /', function () { //  title ===   it('should be included title', function (done) { request(app) .get('/') .end(function (err, res) { if (err) return done(err); res.text.should.include('<title> </title>'); done(); }); }); //  title ===   it('should be included title', function (done) { request(app) .get('/') .set('X-Requested-With', 'XMLHttpRequest') .end(function (err, res) { if (err) return done(err); res.body.should.have.property('data'); res.body.data.should.have.property('title', ' '); done(); }); }); });




request（）で、http.Serverのインスタンスまたは要求を実行する関数を渡す必要があります。 SuperTestはSuperAgentを使用してサーバーと対話するため、すべての機能を使用してサーバーへのリクエストを作成できます。 応答の検証は、 expect（）関数で、またはハンドラー関数をend（）に渡すことで要求の結果として直接実行できます。



おわりに


アプリケーションのテストを書くのはとても簡単です（そして必要ですら）。 私のテスト用の小さなサンプルコードでも、テストコード自体よりも多くのことが判明しましたが、これらのテストでさえ不完全であり、たとえば、2人のユーザーを同時に作成して保存した場合、エラーはテストでカバーされません。 カバレッジテストは、ユーザーモデルが新しいユーザーが保存された場所のテストでカバーされることを示していますが。

したがって、テスト自体は万能薬ではありません。テストは正しく記述されている必要があり、ささいなことを理解し、問題を引き起こす可能性のある場所を正確にテストする必要があります。
