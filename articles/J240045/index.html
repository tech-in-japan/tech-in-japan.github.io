プロジェクトの準備



HelloWorld、そしてAndroidアプリケーション用のテンプレートプロジェクトを作成する方法は既に知っています。 次に、さらに必要なものを見てみましょう。 これらのツールの存在を知っていれば、インターネット上でそれらの使用方法を簡単に見つけることができます。 結局のところ、主な問題は、ツールの使用方法を理解することではなく、ツールが存在することを見つけることです。



1）プラットフォームに依存しないActionBar（画面上部のメニュー）を実装するには、 ActionBarSherlockが必要です。 公式サイトからダウンロードし、ソースコードの形式でWorkspaceにインポートします。 ライブラリによるいくつかのリソースの過負荷には既知の問題があるため、ライブラリ（jarファイル）だけでは十分ではありません。



2）SDK sdk \ extras \ google \ google_play_services \ libproject \ google-play-services_lib \からソースのGoogle Playサービスの形式でワークスペースにインポートします。 これは、請求と承認に必要になります。



3）プロジェクトライブラリのlibフォルダーに入れます（その前に、インターネットでそれらを見つけます）。

* acra.jar-アプリケーションクラッシュレポートメカニズムACRAを実装します。

* android-support-v4.jar-古いバージョンのAndroidとの互換性を実装します。

* roboguice-2.0.jar、roboguice-sherlock-1.5.jar-依存性注入の実装用。roboguiceでの実装が必要な場合。

* ormlite-core.jar、ormlite-android.jar- sqlite Androidベース用の人気のある「軽量」 ORM 。

* joda-time.jar-日付を操作するためのライブラリ。

* jdom.jar、gson.jar-JSONを操作するため。

* checkout.jar-課金用（このライブラリを選択しました。APIを直接操作するよりもCheckoutの方が便利です）。



CMCの取得と解析



以下では、Androidで4.4（KitKat）よりも低いバージョンで実行されるメソッドを示します。これは、GoogleがこのバージョンでCMC処理のアプローチを根本的に変更したためです。 KitKatを使用した作業の説明は、アプリケーションで実装されるときに追加します。



CMCを使用するには、マニフェストで許可が必要です。 



<uses-permission android:name="android.permission.RECEIVE_SMS" /> <uses-permission android:name="android.permission.WRITE_SMS" /> <uses-permission android:name="android.permission.READ_SMS" />


どこで

RECEIVE_SMS-CMCを受信するアプリケーションへの許可。

READ_SMS-携帯電話のメモリからSMSを読み取る許可。 必要ないと思われますが、この許可がないと記録は機能しません。

WRITE_SMS-電話のメモリにCMCを書き込む許可。



SmsBroadcastReceiverが受け入れたCMCイベントリスナーを作成しましょう。 電話機がSMSを受信し、SMSを処理するためのメインプロセスの実行を開始すると呼び出されます。



SmsBroadcastReceiver 
 //BroadcastReceiver      public class SmsBroadcastReceiver extends BroadcastReceiver { @Override public void onReceive(Context context, Intent intent) { // -    ,    Bundle bundle = intent.getExtras(); //    pdus -      Object[] pdus = (Object[]) bundle.get("pdus"); if (pdus.length == 0) { return; //      } //  CMC Sms sms = SmsFromPdus(pdus, context); // ,   Boolean isClearFromSpam = SuperMegaMethodForResolving Spam(sms, context); if (!isClearFromSpam) { //    -   CMC  abortBroadcast(); return; } } private Sms SmsFromPdus(Object[] pdus, Context context) { Sms sms = new Sms(); for (int i = 0; i < pdus.length; i++) { SmsMessage smsMessage = SmsMessage.createFromPdu((byte[]) pdus[i]); sms.Text += smsMessage.getMessageBody(); //   (CMC   "") } SmsMessage first = SmsMessage.createFromPdu((byte[]) pdus[0]);//    sms.SenderId = first.getOriginatingAddress(); // Date receiveDate = new Date(first.getTimestampMillis()); // sms.RecieveDate = receiveDate; sms.Status = first.getStatus(); // (, , ) return sms; } } public class Sms{ public String SenderId; public String Text; public Date RecieveDate; public int Status; }




onReceiveが10秒未満で完了することが非常に重要です。 メソッドがより長い期間制御を行う場合、実行は中断され、イベントは他のハンドラーに優先順位でディスパッチされます。

私の場合、SuperMegaMethodForResolvingは、連絡先リストとローカル送信者リストにSMSが存在するかどうかを確認します。これには1秒もかかりません。 次に、専用ストリームに制御が与えられ、onReceiveがabortBroadcastを呼び出します。これにより、他のハンドラーがSMS（基本SMSアプリケーションを含む）を受信できなくなります。



次に、CMC受信イベントのSmsBroadcastReceiverに署名する必要があります。 これを行うには、マニフェストのアプリケーションブロックで、SmsBroadcastReceiverという名前のandroid.provider.Telephony.SMS_RECEIVEDイベントリスナーを宣言します。これは、SMS_RECEIVEDシステムイベントをリッスンし、優先度は2147483631です。優先度は最大2 ^ 31です。 同時に、Googleは999より大きい値を使用することを推奨していません。しかし、多くのアプリケーションはそれらを使用します。 このアプリケーションは、私が知っている最高の優先順位を求めます。 



 <receiver android:name="su.Jalapeno.AntiSpam.SystemService.SmsBroadcastReceiver" android:enabled="true" android:exported="true" > <intent-filter android:priority="2147483631" > <action android:name="android.provider.Telephony.SMS_RECEIVED" /> </intent-filter> </receiver>


起動電話での自動起動アプリケーション



デバイスの電源をオンにした後、アプリケーションを起動する必要がある場合があります。たとえば、通知を通じて未確認の不審なSMSの存在をユーザーに通知する場合です。



起動するには、マニフェストの許可が必要です。 



 <uses-permission android:name="android.permission.RECEIVE_BOOT_COMPLETED" />


どこで

RECEIVE_BOOT_COMPLETED-ダウンロードイベントをリッスンする権限



「ダウンロード」イベントServiceBroadcastReceiverのリスナーを作成します。 



 public class ServiceBroadcastReceiver extends BroadcastReceiver { @Override public void onReceive(Context context, Intent intent) { // - } }


その中で、電話をオンにした後、必要なアクションを実行します。



次に、ServiceBroadcastReceiverを電話ブートイベントに署名する必要があります。 これを行うには、マニフェストのアプリケーションブロックで、SmsBroadcastReceiverという名前のandroid.intent.action.BOOT_COMPLETEDイベントリスナーを宣言します。 



 <receiver android:name="su.Jalapeno.AntiSpam.SystemService.ServiceBroadcastReceiver" android:exported="true" > <intent-filter> <action android:name="android.intent.action.BOOT_COMPLETED" /> </intent-filter> </receiver>

Googleトークン認証



Googleのドキュメントなど、体系的で不完全な情報が不足しているため、これに最も時間がかかりました。 したがって、私たちのタスクは、アプリケーションに関するユーザー情報と秘密情報を含むアプリケーションを介してGoogleから署名付きトークンを取得することです。 これにより、トークンが攻撃者によって生成されたものではないと考える理由が得られます。 この問題を解決するには、 CrossClientAuthメカニズムを使用します。



以下を実行する必要があります。

1）アプリケーション証明書を取得し、それを使用してアプリケーションに署名します。 これは、ウィザードを使用してEclipseに簡単に実装できます。 パッケージエクスプローラー-> Androidツール->署名済みアプリケーションパッケージのエクスポートでプロジェクトを右クリックします。 ウィザードは、新しい証明書ストアの作成、証明書の生成、指定したパスワードで保護されたストアへの配置を提案します。 証明書のハッシュは将来必要になるため、忘れずに保存してください。



2） Googleコンソールでプロジェクトを作成します。 次に、作成したプロジェクトを開き、左側のパネルの[Apiと認証]-> [資格情報]タブに移動します。 ここでは、サーバー側とAndroidクライアント用のクライアントIDのペアを作成する必要があります。 [新しいクライアントIDを作成]をクリックします。AndroidアプリケーションのクライアントIDが必要です。







スクリーンショットに示されているように、正しいパッケージ名と証明書の指紋を記入してください。 作成が完了すると、生成された情報とクライアントIDが記載されたプレートが取得されます。 サーバー上で必要です。



次に、Webアプリケーションタイプの新しいクライアントIDを作成します。 私の場合、httpを介したGoogle Webリソースとのやり取りがないため、アドレスは任意に指定できます。 その結果、新しいクライアントIDを取得します。これはクライアントで既に必要です。



3） GoogleAuthUtilなど、クライアントコード全体をインターネット上で見つけることができます。 重要なポイントのみに注目します。Scopeを正しく構成する方法、およびIDを取得する場所
