Data ONTAP 8.3 ADP：ルートデータパーティショニング

Data ONTAP 8.3 cDOTは、NetAppの最大のリリースの1つです。 このリリースの重要な機能の1つは、Advanced Drive Partitioning（ADP）テクノロジーです。 cDOT 8.3の新機能の詳細については、 こちらをご覧ください。



ADPテクノロジーには2つの主な用途があります。

  • ルートデータの分割
  • FlashPoolパーティショニング（ストレージプール）


この記事では、ルートデータのパーティション分割について説明します。





ルートデータの分割



ルートデータパーティション分割は、各ディスクを2つのパーティションに分割するために使用されます。1つは大規模（通常はデータパーティションと呼びます）、もう1つははるかに小さい（通常2つ目はルートパーティションと呼びます）。 さらに、各パーティションは個別のディスクと見なされ、2番目の所有者（コントローラー）から最初のパーティションを独立して持つことができます。 小さなパーティションでは、ルートアグリゲートが両方のコントローラー用に作成され、大きなパーティションではデータアグリゲートが作成されます。 これにより、専用ドライブの代わりにルートアグリゲートの下の小さなパーティションのみを使用できます。 このテクノロジーは、FAS22XX / FAS25XXおよびAFF8XXXXシステムで利用可能であることに注意することが重要です。 ルートデータパーティショニングは、システムに接続された最初のシェルフにあるディスクでのみ有効になります（FAS2XXXの場合、最初のシェルフのディスクはコントローラーと同じシャーシにあると見なされます）。 各パーティションは個別のディスクとして「機能」し、独自の名前を持ち、古典的にディスクで機能するコマンドを使用して、これらの「仮想」ディスクの名前を置き換えます。



FAS8XXXでは、ルートアグリゲートの下に専用ドライブが必要です。 なぜですか？ FAS8000は非常に多数のディスク向けに設計されており、この背景に対して4〜6台のディスクは役割を果たさないためです。 小規模なシステムの場合、これらの4〜6個のディスクはかなりのスペースを節約できます。高価なAFFシステムにも同じことが当てはまり、システムのニーズに合わせて高価で高性能なディスクを使うことは合理的ではありません。



ルートデータパーティショニングは、設定もオフもしないテクノロジーです。 8.3に付属するすべての新しいシステム上にあります。 古いシステム（FAS22XX / FAS25XXおよびAFF8XXXX）が8.3に更新され、再フォーマットされると、ルートデータパーティションが自動的に有効になり、ディスクが2つのパーティションに分割されます。 どこにも尋ねられたり警告されたりすることはありません。



ルートデータパーティショニングを機能させるには、次のものが必要です。





コントローラーごとに少なくとも4つのディスクが存在する（2つのコントローラーで最小8）。



8.3以降の新しいシステムをお持ちの場合は、ルートデータパーティションが既にあります。リラックスしてください。何もする必要はありません。



既存のシステムでルートデータパーティショニングを使用する場合は、それをcDOTに転送する必要があり（7-Modeは8.3およびADPをサポートしません）、すべてのディスクをフォーマットする必要があります（ すべてのデータが破壊されます！ルートデータパーティショニングを有効にする他の方法はありません） 。



  1. 両方のコントローラーを起動し、両方の計測モードに移動します（ブートメニューから）
  2. 両方で、すべての古いアグリゲートを強制終了します（ すべてのデータが破棄されます！それ以外の場合は、ルートデータパーティショニングを有効にしないでください）
  3. disk removeownershipコマンドを使用して、すべてのディスクからオーバーシップを削除します。問題が発生しています。 これは、ルートデータのパーティション分割にとって重要です。
  4. ブートメニューに移動し、 wipeconfigを （数字の代わりに）書き込み、確認、期待、システムが2回再起動します
  5. （7-ModeからcDOTに変換するには、再起動してローダーに入り、ローダーのブートパラメーターを変更して、システムがcDOTをロードするようにします）
  6. ブートメニューをもう一度入力し、各コントローラーで項目4（ すべてのディスクを初期化 ）を選択し、 ディスクエラーが完了するのを待ちます。




各パーティションのサイズは？



システム内のディスクの数、タイプ、およびコントローラーの数（1または2）に依存します。 原則として、コントローラーごとに最低4pドライブが必要です。 パーティションのサイズは、結果として、ルートパーティションで構成されるルートアグリゲートのすべてのスペースを占めるルートボリュームに十分な使用可能スペースが得られるように選択されます。 可能なすべての組み合わせは事前に計算され、システムに縫い付けられていますが、これは構成できません。 可能なすべての組み合わせに関するより正確な情報は、ウェブサイトhwu.netapp.comのAdvanced Drive Partitioningセクションにあります。





ここにいくつか例を挙げます。

FAS2240-2 / FAS2552
  • システムに2つのコントローラーと8つのディスクがある場合、ルートパーティションのサイズは110.82 GiBになります。 したがって、この構成では、コントローラーごとに4つのルートパーティション 、コントローラーごとに1つのルートユニットを取得します。 各ユニットは、データ用の2つのパーティションとパリティ用の2つのパーティションで構成され、パーティションのペアはありません。 各ルートユニットのサイズは2 * 110.82です。
  • 同じシステムに12個のディスクがある場合、ルートパーティションは73.89 GiBを占有します。 各ルートユニットは、次のパーティションの組み合わせで構成されます。3つのデータ+ 2つのパーティ+ 1つのスペア。 各ルートユニットのサイズは3 * 73.89です。
  • 24個のディスクに、27.72 GiBのルートパーティションと、ルートマウント用のパーティションの組み合わせ（8データ+ 2パーティ+ 2スペア）があります。 各ルートユニットのサイズは8 * 27.72です。




FAS2240-4 / FAS2554
  • HAシステムFAS2240-4およびFAS2554および8ディスクの場合、ルートパーティションは 215.47GiBであり、各アグリゲートは次のもので構成されます：2つのデータ+ 2つのパーティション、パーティションスパーなし。 各ルートユニットのサイズは2 * 215.47になります
  • 同じシステム上に12個のディスクがある場合、ルートパーティションは143.65 GiBを占有し、各ユニットは3つのデータ+ 2つのパリティ、1つのスパートパーティションで構成されます。 各ルートユニットのサイズは3 * 143.65です。
  • 24個のディスクでは、 パーティションは53.88GiB、各ユニットは 8日付、2パリティ、2スペアになります。 各ルートユニットのサイズは8 * 53.88になります






使用可能スペース



ルートデータパーティショニングで何が得られ、このテクノロジーの利点は何ですか？

  • 第一に、それぞれが少し「短く」なる場合でも、データ集合に追加できる4〜6枚のディスクを獲得します。
  • 次に、これらの追加の4〜6ディスクからパフォーマンスを「削除」します。 ルートユニットの負荷が軽いためです。
  • 第三に、アクティブ-パッシブ構成を使用し、両方のコントローラーの操作にルートパーティションを使用できますが、同時に1つのコントローラーに1つの大きなユニットを配置できます。






新しいディスクをアップグレードして追加します。



新しいディスクを追加する場合、いくつかのオプションがあります。

  1. 最も簡単な方法は、新しいアグリゲートを作成し、データパーティションに存在する既存のデータアグリゲートに新しいディスクを追加しないことです。
  2. 2番目のオプションは、データパーティションに存在する既存のデータアグリゲートの既存のRAIDグループに新しいディスクを追加することです。 この場合、新しいディスクはデータパーティションのサイズに合わせて切り捨てられ、このRAIDグループに追加されます。 RAIDグループのサイズは無限ではないため、最大28/20ディスク（SAS / SSD 26 + 2、SATA / NL-SAS 18 + 2）まで拡張できます。 そして、最大に達した場合、3番目のオプションに移動する必要があります。
  3. 3番目のオプションは、既存のアグリゲートの新しいRAIDグループにディスクを追加する場合です。これは、最初はデータパーティションを使用するRAIDグループで構成されています。 この場合、最初のグループは2番目のグループよりも少し短くなりますが、これは問題ではありません.ONTAPの最新バージョンでは、この目的のためにRAIDグループの操作メカニズムが特に最適化されているため、1つのユニットでRAIDグループの十分に大きな分岐が許可されています。
  4. 別のシェルフが届くと、このシェルフに新しいユニットを作成し、短縮ディスク上に構築された古いユニットの1つからオンラインで移行できます。 その後、リリースされた短縮ディスクのオーバーシップをネイバーのオーバーシップに変更します。 次に、リリースされた短縮ディスクを同じ短縮ディスクに追加します。 したがって、3番目のオプションをバイパスします。
  5. FAS2240 / FAS2552 / FAS2554をシェルフに変換し、それらを古いシステム（FAS8020など）に接続すると、ルートデータパーティショニングが機能します。 これは、FAS8XXXをルートデータパーティショニングで動作させる唯一の方法です。






欠点



  • 1つまたは複数のディスクに障害が発生した場合、ルートとデータの集約が同時に劣化する可能性があります。 ルートユニットの出力は、HAペアによって保護されているためそれほど怖くありません。ルートユニットが破損しているコントローラーがディスクを処理できない場合、HAパートナーはこれを行います。 ルートデータパーティショニングのないシステムの場合、集約をネイバーに切り替えることは回避できます。
  • ディスクの切り捨ては、ディスクの一部が使用されないという事実につながります。




アクティブ-パッシブとアクティブ-アクティブ



Active-Passiveを使用するのが理にかなっているケースと、Active-Activeを使用する価値があるケースを比較してみましょう。 24台以下のディスクのシステムがある場合、コントローラー間でディスクを共有することは、データが一度に2つのコントローラーによって提供されるという事実により理論的に高くなる可能性がある、いくつかの追加のパフォーマンスの理由でのみ意味がありません。 実際、各FAS2XXXコントローラーは144のディスクをサービスできるように設計されています（パートナーが死んだ場合でも）。 原則として、システム帯域幅はコントローラーではなく、バックエンドディスクサブシステムにかかっていることを理解する必要があります。 したがって、24台以下のディスクの構成では、原則として、1台ではなく2台のコントローラーを使用するだけでは、追加のパフォーマンスは得られません。



Active-Active構成では、3つの追加ディスク（2パリティ+ 1スペア）しか失わないため、バックエンドでのパフォーマンスと容量が増えます。





まとめ 24台未満のディスクを搭載したFAS2XXXシステムの場合、フロントエンドのパフォーマンスとフォールトトレランスが低下せず、ディスクサブシステムのバックエンドのパフォーマンスが向上し、使用可能なスペースが増加するため、アクティブ/パッシブ構成を行うのが理にかなっています。 それ以外の場合はすべて、Active-Active構成を使用します。



アクティブ/パッシブを設定する方法


システムの初期化と初期セットアップが完了すると、デフォルトでアクティブ-アクティブ構成になります。1つのコントローラーからデータ用のパーティションを選択し（ディスク名の最後にP1またはP2の終わりがあります）、2番目に与える必要があります。 選択したパーティションに既にデータアグリゲートがある場合は、削除する必要があります。削除しないと、アグリゲートからディスクを取得できません。 これは、7-Mode shell'aから実行されます （ システムノードの実行-nodeローカル ）。 変更する前に、どのパーティションがルートパーティションであり、どのデータパーティションであるかを確認してください（ aggr status <root_aggr> -r ）。 ディスクの上位ドライブは、それを所有するコントローラーから変更できます（ disk assign <disk.name.P1> -s <new-serial-number> ）。 各パーティションは個別のディスクとして「機能」します。物理ディスクのオーバーシップと同じ物理ディスク上の他のパーティションとは異なる独自のオーバーシップがあり、パーティションのオーバーシップの変更は、物理ディスクのオーバーシップの変更と同じコマンドを使用して実行されます。

例： 

sys::> system node run -node local sys-01> aggr status rootA -r sys-01> disk show sys-01> disk assign 0a.10.14P1 -f -s 536880408


ここで、 P1はディスク0a.10.14のデータパーティションであり、 sys-01コントローラーはまだパーティションを所有しています（コマンドはそこから実行されます）、パーティションは-sフラグの代わりにsys-02ネイバーコントローラー（シリアル番号は536880408）に再割り当てされます-o <neighborname> 。



結論：



システムの内部に隠されているルートデータパーティショニングテクノロジーは、管理者がその作業のスキームを推測することすらできません。 「現状のまま」のシステムを使用すれば、メンテナンスに問題はありません。単に存在し、動作するだけで動作中に問題を引き起こさない「それ自体」であると言えます。 一般に、このテクノロジーは使用可能なスペースとrawの比率を改善し、バックエンドのパフォーマンスも間接的に改善します（特にAFFに関しては、保存された各SSDは何千ものIOPSを追加でき、ルートデータパーティション化により最大6つ節約できます）。 Root-Data Partitioningの複雑さを知っている場合、インストール段階で、フォールトトレランスとパフォーマンスに屈することなく、使用可能なスペースでさらに勝つことができます。
