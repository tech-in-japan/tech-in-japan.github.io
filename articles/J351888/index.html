エスプレッソ：「かわいい動物や危険な捕食者？」

こんにちは、Habrの読者の皆さん！ 今日は、あなたと一緒にAndroidでRecyclerviewをテストします。私の意見では、このトピックは非常に興味深いものです。







Recyclerviewとは何ですか？ これは、リストが作成されるコンポーネントです。 各リストをスクロールしたり、アイテムを追加したり、削除したり、変更したりできます。 要素は、任意の機能単位です。 たとえば、コメントとボタンを入力するフィールドを持つユーザーのリストを作成します。 コメントが入力され、ボタンが押されるとすぐに、サーバーに送信されます。 これで、システムはアイテムをアップグレードまたは削除できます。

要素には多くのコントロール（ボタンなど）を含めることができるため、要素のすべての機能をテストでカバーする必要があります。 次に、Recyclerviewの便利なユーティリティを紹介します。



例



例として、動物のリストを表示する簡単なアプリケーションを取り上げます。 リストデータは、同じタイプのオブジェクトの配列です。 それで、私たちは何を見ますか？ 



[ ... { "type": "PREDATOR", "name": "", "image_url": "https://www.myplanet-ua.com/wp-content/uploads/2017/05/%D0%B2%D0%B8%D0%B4%D1%8B-%D1%82%D0%B8%D0%B3%D1%80%D0%BE%D0%B2.jpg" }, { "type": "HERBIVORE", "name": "", "image_url": "https://cs2.livemaster.ru/storage/b2/40/b9d72365ffc02131ea60420cdc0s--kukly-i-igrushki-suslik-bejli-igrushka-iz-shersti.jpg" }, .... ]


要素は以下で構成されます：



  1. 動物の種-捕食者（捕食者）および草食動物（草食動物）。 それに応じて、動物には1つまたは別のアイコンが割り当てられます。
  2. 動物の名前（トラ、ホリネズミなど）。
  3. 動物の画像へのリンク。

DrawableMatcher



最初の重要なヘルパークラスは、画像がローカルリソースと一致するかどうかを確認することです。 タイプフィールド（サーバーからの応答）とローカルリソースの名前の対応は次のとおりです。



プレデター-ic_sentiment_very_dissatisfied_red_24dp

HERBIVORE-ic_sentiment_very_satisfied_green_24dp



例：このタスクでは、capybaraが草食動物であることが確認されているため、ic_sentiment_very_satisfied_green_24dpの値は、ivAnimalType識別子を持つimageView要素で設定する必要があります。



実際、動物の種類に応じて、確立されたリソースの適合性を確認する必要があります。 このための標準ツールが見つからなかったため、Matcherを作成する必要がありました。これは、送信された識別子に準拠しているかどうかimageView要素の背景プロパティをチェックします。



小さな予約



マッチャーは、ユーザービューの特定のプロパティをチェックして、渡されたパラメーターに準拠しているかどうかを確認するために最もよく使用されます。



例：ドアを描いた独自のカスタムビューを作成しました。 開いた状態と閉じた状態の2つの状態があります。 ドアの操作を確認するために、独自のMatcherを作成できます。 ハンドルを回した後、ドアが閉じた状態から開いた状態になることに同意し、テストではハンドルを引き、独自のマッチャーを使用して、本当に開いていることを確認します。



この種のマッチャーを作成するには、標準クラスパッケージorg.hamcrest-TypeSafeMatcherを使用します。 詳細はこちらをご覧ください。



このクラスは、ユーザービュータイプ（imageView、viewなど）を持つ汎用クラスです。 2つの主な方法を提供します。



  1. matchesSafelyは、コンプライアンスチェックに合格するルートメソッドです。 ジェネリッククラス型のオブジェクトがこのパラメーターに渡されます。
  2. describeTo-オブジェクトが自身を記述する機能のための関数。 未知の一致がある場合に呼び出され、できれば、検索しようとしているものの名​​前と、実際に出会ったものが含まれています。


この場合、リソースの識別子を渡して、クラスのコンストラクターにさらに確認することをお勧めします。 



 public DrawableMatcher(final int resourceId) { super(View.class); mResDrawableId = resourceId; }


記述方法は、テストをデバッグするときに非常に便利です。 たとえば、画像が一致しない場合、インストールされているリソースがチェック対象のリソースと一致しないことを示すメッセージを表示し、両方のリソースのデータを提供します。 



 @Override public void describeTo(Description description) { description.appendText("with drawable from resource id: "); description.appendValue(mResDrawableId); if (resourceName != null) { description.appendText("["); description.appendText(resourceName); description.appendText("]"); } }


ルート適合チェックメソッドでは、ビットマップクラスの標準のsameAsメソッドがチェックされます。 つまり、送信された識別子によってビットマップを作成し、それを背景フィールドに設定された識別子と比較します。 



 protected boolean matchesSafely(final View target) { if (!(target instanceof ImageView)) { return false; } final ImageView imageView = (ImageView) target; if (mResDrawableId < 0) { return imageView.getBackground() == null; } final Resources resources = target.getContext().getResources(); final Drawable expectedDrawable = resources.getDrawable(mResDrawableId); resourceName = resources.getResourceEntryName(mResDrawableId); if (expectedDrawable == null) { return false; } final Bitmap bitmap = getBitmap(imageView.getBackground()) final Bitmap otherBitmap = getBitmap(expectedDrawable); return bitmap.sameAs(otherBitmap); }


これは、特定のリソースのイメージをチェックすることに関するすべてです。



内部要素のイベント



割り当てに従って、「削除」ボタンをクリックする必要があるため、ViewActionの追加の実装を記述する必要があります。 CustomRecyclerViewActionsと呼ばれるクラスのユーティリティを作成しておくと便利でした。 ViewAction実装を返す多くの静的メソッドが含まれています。 この例では、clickChildViewWithIdを使用します。

始めるために、ViewActionインターフェースをさらに詳しく見てみましょう。 次のメソッドで構成されています。



  1. Matcher getConstraints（）-実行のためのいくつかの前提条件。 たとえば、アクションが実行される要素が表示される必要がある場合があります。
  2. String getDescription（）-ビューのアクションの説明。 ログ内のメッセージの読み取り可能な表示を作成するために必要です。
  3. void perform（UiController uiController、ビュービュー）-実行するアクション自体。


それでは、clickChildViewWithIdメソッドの作成に戻りましょう。 ユーザービューの識別子をパラメーターに渡し、そのパラメーターでクリックイベントを実行します。 このメソッドの完全な実装を見ることができます
こちら。 
 public static ViewAction clickChildViewWithId(final int id) { return new ViewAction() { @Override public Matcher<View> getConstraints() { return null; } @Override public String getDescription() { return "     id"; } @Override public void perform(final UiController uiController, final View view) { final View v = view.findViewById(id); v.performClick(); } }; }




アダプター内の要素の数を確認する



また、Recyclerview内の要素の数を確認する必要があります。 これを行うには、独自の実装ViewAssertionインターフェイスが必要です。 それをRecyclerViewItemCountAssertionと呼びましょう。



ViewAssertionインターフェイスは1つの方法です。

void check(View view, NoMatchingViewException noViewFoundException);







より詳細にチェック方法を検討してください。 ユーザー要求を確認するこの方法は、2つのパラメーターを取ります。



  1. assertThat(0,is(view.getId())



    ステートメントが作成されるユーザービューの要素（たとえば、 assertThat(0,is(view.getId())



    ）;識別子を確認するため）。
  2. NoMatchingViewException-特定のマッパーがビューの階層の要素ではない場合、つまりビューではない場合に発生する例外。 そのような例外には、表示とコンプライアンスに関する情報が含まれており、デバッグに非常に便利です。


このクラスの実装は簡単です。 あなたが見ることができる完全なクラスコード
こちら。 
 public class RecyclerViewItemCountAssertion implements ViewAssertion { private final int mExpectedCount; public RecyclerViewItemCountAssertion(final int expectedCount) { mExpectedCount = expectedCount; } @Override public void check(final View view, final NoMatchingViewException noViewFoundException) { if (noViewFoundException != null) { throw noViewFoundException; } final RecyclerView recyclerView = (RecyclerView) view; final RecyclerView.Adapter adapter = recyclerView.getAdapter(); assertThat(adapter.getItemCount(), is(expectedCount)); } }




要素の内部表現を確認する



リスト内のチェック要素のクラスの調査に進みます。 このクラスは、RecyclerViewItemSpecificityViewと呼ばれます。 クラスコンストラクターは、要素識別子とマッチャーの2つのパラメーターを取ります。 識別子を使用すると、すべてが多かれ少なかれ明確になります。この要素を後で確認します。 そして、2番目のパラメーターは、正確に何をチェックするのかという質問に答えます。 動物の例に移りましょう。



リストの6番目の要素が草食動物である「カピバラ」であることを確認する必要があります。 これを確認するには、tvNameフィールドでテキスト「capybara」に準拠しているかどうかを確認する必要があります。 6番目の要素に正確に到達するにはどうすればよいですか？ espresso.contribパッケージには、RecyclerViewActionsクラスがあります。 RecyclerViewのテストに役立ち、エスプレッソライブラリの標準メソッドと完全に結合するさまざまな便利なメソッドが含まれています。 これにより、カバレッジの割合を高めることができます。 残念ながら、RecyclerViewUtilsはすべての機能をサポートしていません。 たとえば、リストカードの項目を明示的に直接確認することはできません。



RecyclerViewItemSpecificityViewクラスを検討してください。 これは、ViewAssertionインターフェースの実装です。 したがって、RecyclerViewItemCountAssertionと同じように機能しますが、目的は異なります。



前述のように、RecyclerViewItemSpecificityViewクラスのコンストラクターは2つのパラメーターを取り、次の構成を表します。 



 public RecyclerViewItemSpecificityView(final int specificallyId, final Matcher<View> matcher) { mSpecificallyId = specificallyId; mMatcher = matcher; }


この場合のチェック方法は次のようになります。



  1. 対応する要素を識別子mSpecificallyIdで検索します。 ここでは、Recyclerview要素の識別子を渡します。
  2. MMatcherコンプライアンスチェック。 このチェックは、クラスのメインタスクです。
  3. コンプライアンスがない場合の判読可能な結論の形成。 


 @Override public void check(final View view, final NoMatchingViewException noViewFoundException) { final StringDescription description = new StringDescription(); description.appendText("'"); mMatcher.describeTo(description); final ViewGroup itemRoot = (ViewGroup) view; final View typeIUsage = itemRoot.findViewById(mSpecificallyId); if (noViewFoundException != null) { description.appendText( String.format( "' check could not be performed because view with id '%s' was not found.\n", mSpecificallyId)); Log.e("RecyclerViewItemSpecificityView", description.toString()); throw noViewFoundException; } else { description.appendText("' doesn't match the selected view."); assertThat(description.toString(), typeIUsage, mMatcher); } }


「まあ、結局のところ-それはカピバラですか？」



すべての要素の準備ができました。カピバラを確認してみましょう。 まず、要素が草食カピバラかどうかを確認するテストを作成します。 



 onView(withId(R.id.rvAnimals)) .perform(scrollToPosition(capybaraPosition)) .check(new RecyclerViewItemSpecificityView(R.id.tvName, withText(""))) .check(new RecyclerViewItemSpecificityView(R.id.ivAnimalType, new DrawableMatcher(R.drawable.ic_sentiment_very_satisfied_black_24dp)));


ScrollToPositionは、RecyclerViewActionsクラスのメソッドです。 リストを選択した位置までスクロールする必要があります。 リスト項目が表示されていない場合、テストは失敗します。 次に、tvNameフィールドのリストをスクロールした要素に文字列「Capybara」が含まれていることを確認します。 また、テスト対象のアイテムは「草食動物」なので、アイコン（ivAnimalType）がic_sentiment_very_satisfied_black_24dpと一致することを確認する必要があります。



次に、アイテムを削除するテストを作成しましょう。 CustomRecyclerViewActionsクラスの静的なclickChildViewWithIdメソッドをどのように使用し、RecyclerViewItemCountAssertionを使用して要素の数が削減されたことを確認する方法を既に推測していると思います。 



 onView(withId(R.id.rvAnimals)).check(new RecyclerViewItemCountAssertion(8)); onView(withId(R.id.rvAnimals)).perform( RecyclerViewActions.actionOnItemAtPosition(0, MyRecyclerViewActions.clickChildViewWithId(R.id.btnRemove))); onView(withId(R.id.rvAnimals)).check(new RecyclerViewItemCountAssertion(7));


具体的には、RecyclerViewActionsクラスのactionOnItemAtPositionメソッドを使用しました。 彼はリストを現在の位置までスクロールし、それによってリスト項目を操作する機会を与えてくれます。
