両面のREQUEST_URIまたは正しいHTTP / 1.1サーバーを探している

Apache mod_rewriteの％{REQUEST_URI}とPHPの$ _SERVER ["REQUEST_URI"]の違いを知っていますか？



wwwプレフィックスを持つドメインから、またはApacheレベルの.htaccessでドメインに301を正しく転送できますか？



最後の質問については、まだ解決策を提供できません。 その理由は、HTTP / 1.1プロトコルです。これは、「自転車を発明した」（サイトのコアを作成した）ときに、より詳細に調査する必要がありました。



「Host：」リクエストのHTTPヘッダーがすべてです。 特定の条件下では、何でもそこにある可能性があり、サーバーはHTTP / 1.1に従って完全に無視する必要があります。 ほとんどの開発者は、SEO最適化などにこのフィールドの値を使用します。 今後、追加のプロキシ（たとえば、nginx）がこの問題を解決すると言います。



サーバーの不正な動作を説明するために、Habr社のサイトを反復処理することにしました。 ダースのサイトでは、彼は手動でそれを行い、その後、一部のサイトが誤った要求に「正しく」応答することを発見しました。 その後、小さなテストユーティリティが作成され、テストするテストパターンとサイトの数を増やすことができました。



HTTP / 1.1でREQUEST_URIを隠すものは何ですか？







理論





HTTP / 1.0



HTTP / 1.0プロトコルから始めます。これは、RfC1945 www.w3.org/Protocols/rfc1945/rfc1945で説明されており、1996年5月の日付です。 目的のページを取得するには、サーバーに接続して1行送信するだけで十分です。 

  GET /path/to/resource.html HTTP / 1.0


プロキシサーバーにアクセスするときは、絶対パスではなく、完全なアドレスを使用する必要がありました。 

  GET http://domain.name/path/to/resource.html HTTP / 1.0


これはすべてセクション5.1.2「Request-URI」で説明されています。



ホストの外観



1つのサーバーが一度に複数のドメイン名を提供できるようにするために、プロトコル作成者は、アクセスされるドメインを含む「Host：」リクエストヘッダーを追加しました。 このヘッダーはHTTP / 1.0標準の一部ではありませんが、一部のサーバーおよびクライアントでサポートされるようになりました。 たとえば、wgetはHTTP / 1.0を介してリクエストを送信しますが、「Host：」を追加します。



HTTP / 1.1



1999年6月（14年前）にHTTP / 1.1プロトコルが登場しました。これは、RfC2616 www.w3.org/Protocols/rfc2616/rfc2616.htmlで説明されています 。 セクション14.23で、新しいプロトコルでは、各リクエストヘッダーに「ホスト」フィールドを含める必要がありました。

クライアントは、すべてのHTTP / 1.1要求メッセージにHostヘッダーフィールドを含める必要があります。 要求されたURIに、要求されているサービスのインターネットホスト名が含まれていない場合、Hostヘッダーフィールドには空の値を指定する必要があります。




さらに、クエリ文字列からRequest-URIに大幅な変更が加えられました（ セクション5.1.2 ）。 前のプロトコルと同様に、プロキシサーバーへの要求には完全なアドレスが必要です（「要求がプロキシに対して行われる場合、absoluteURIフォームが必要です。」）。 ただし、すべてのサーバーはこのようなリクエストに応答する必要がありますが、クライアントはプロキシサーバーに対してのみそのようなリクエストを発行します。

HTTPの将来のバージョンのすべてのリクエストで絶対URIへの移行を可能にするには、HTTP / 1.1クライアントがプロキシへのリクエストでのみ生成する場合でも、すべてのHTTP / 1.1サーバーはリクエストで絶対URIフォームを受け入れなければなりません。




完全なアドレス（たとえば、 http ： //www.w3.org/pub/WWW/TheProject.html ）に移動することになっていたことに注意してください。したがって 、クライアントは絶対パス（abs_pathなど、 / pubなど）のみを使用する必要はありません。 /WWW/TheProject.html ）。 さらに、サーバーはクライアントからの要求に絶対URIで応答する機能を明示的に必要とします。したがって、この場合、クライアントの要求が正しくないという異議は、「クライアントは常に正しい」ため、すぐに除外します。



HTTP / 1.1のホスト



Request-URIの変更は無害に見えるかもしれませんが、 セクション5.2には1つの重要な要件が含まれています。「Request-URIがabsoluteURIである場合、ホストはRequest-URIの一部です。 要求内のHostヘッダーフィールドの値は無視する必要があります。”つまり、要求の解釈 

  GET http://domain.name/path/to/resource.html HTTP / 1.1
ホスト：anytext


要求に一致する必要があります 

  GET /path/to/resource.html HTTP / 1.1
ホスト：domain.name




absoluteURIでクエリを実行するときに「Host：」を無視しますか？



％{REQUEST_URI}および$ _SERVER ["REQUEST_URI"]



mod_rewriteのドキュメントには、次のことが記載されています。

THE_REQUEST

ブラウザからサーバーに送信される完全なHTTPリクエスト行（例：「GET /index.html HTTP / 1.1」）。 これには、ブラウザから送信された追加のヘッダーは含まれません。 この値は、以下の他のほとんどの変数とは異なり、エスケープされていません（デコードされていません）。



REQUEST_URI

「/index.html」など、要求されたURIのパスコンポーネント。 これは、QUERY_STRINGという名前の独自の変数として使用可能なクエリ文字列を特に除外します。


つまり、％{REQUEST_URI}には常に絶対パスがあり、完全なアドレスはありません。



ユーザーが次のリクエストを送信した場合、mod_rewriteを使用せずにドメインに「www」を追加する標準SEOタスクを試してください。 

  GET http://domain.name/path/to/resource.html HTTP / 1.1
ホスト：www.domain.name




記事の冒頭で、Apache mod_rewriteの％{REQUEST_URI}とPHPの$ _SERVER ["REQUEST_URI"]の違いについて尋ねたので、PHPのドキュメントからの抜粋を示します 。

REQUEST_URI

このページにアクセスするために指定されたURI。 たとえば、「/ index.html」。


どこかで設定できますが、私にとっては、PHP / 5.3.13は完全なアドレスで要求されたときにabsoluteURIを返します。
