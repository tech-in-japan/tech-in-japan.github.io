不完全な技術

こんにちは。 Opensource Game Studioチームのプログラマー、Mikhail Kapelkoです。 麻雀ソリティアの最新リリースであるOGS Mahjong 1.0のリリースに近づいたときに学んだことをお伝えしたいと思います（ただし、麻雀シリーズの最後ではありません）。 この記事は前の記事の続きです： 麻雀0.7への長い道のり



1年前の2011年5月に、OGS Mahjong 0.7をリリースしました。これは非常に困難なものでした。 今年2012年9月にOGS Mahjong 1.0をリリースしました。 開発1.0では、すべてのLinuxディストリビューションとMac OS Xでゲームを配布したかったのですが、残念ながら、成功しませんでした。なぜなら、 OGS MahjongはOGREとOISを使用します。





OGRE



OGS Mahjongの2010年の夏に、リソースをリロードせずにゲームウィンドウを再作成することに気付いたとき、OGRE 1.8の不安定な（当時の）ブランチを使用し始めました。 これを実装するための私の変更は、同じ夏にブランチ1.8に反映されました。

そのとき、OGRE 1.8の安定版リリースは1年以内であり、この時点でOGS Mahjong 1.0はすでにOGREの安定版に依存していると思いました。 OGRE 1.8は2012年5月28日にリリースされました。

しかし、この頃には：

1）OIS 1.3（当時は後者）を試しましたが、私たちには合いませんでした（詳細は以下）。

2）別のOGREファイル（ストーリー：Overlay :: setMaterialName）に変更を加えましたが、OGRE開発者には渡しませんでした。 この小さな変更が1年でせいぜいリリースされることをすでに知っていました。



の



OGREはメイン画像コーデックとしてFreeImageを使用します。 FreeImageの主な問題は、開発者がzlib、pngなどのシステムライブラリの使用に抵抗するため、ライブラリがGentooディストリビューションでサポートされていないことです。

編集 ：FreetoはGentoo、メディアライブラリに含まれています。 他のすべての問題は引き続き有効です。



彼らは特別なフラグを使用することでキャラクターの可視性の問題を修正しました 。

したがって、著者が3番目のオプションで提案したとおりに実行しています。 付録には、さらにいくつかのライブラリがあります。



オイス



OISは、まれにしか更新されないライブラリです。 最近リリースされたリリース1.3は、キーストローク処理を完全に台無しにしました。 キーストロークロジックが変更されたため、OGS Mahjongは起動後30秒でキーストロークの受け入れを停止しました。

この問題について開発者に通知しましたが、彼はそれを再現できませんでした。 Linuxのバージョン1.2を維持する必要がありました。

OIS 1.3はMinGWの下では構築されておらず、SVNの最新（当時）バージョンのみです。

Mac OS Xに関しては、OISはここでも私たちを失望させました。なぜなら、Mac OS X用のMahjongの構築を手伝おうとしていた人は、コンパイルエラーのためにOISを構築できなかったからです。

なぜなら バージョン1.3がリリースされ、すべてのLinuxディストリビューションに付属しています。これは、銅製の流域を備えたすべてのLinuxディストリビューションにOGS Mahjongを含めるという要望をカバーしています。 受け入れられない。



CEGUI



CEGUIライブラリーの状況は完全に異なっていました。特にOGS Mahjongの開発に大いに貢献したチーム、特にCrazyEddieからの優れたサポート。 しかし、CEGUIには非常に大きなマイナス面があります。設計者にとって便利なツールはありません。 私はすべてのGUI開発を単独で行いました。GUIは開発中に頻繁に変更されます。 その結果、GUIの開発に必要以上に時間がかかりました。



ご覧のとおり、上記のライブラリは私たちをかなり制限していました。 唯一のプログラマーである私以上に我慢できます。 他の開発者が犯した間違いを常に修正するのではなく、ゲームを作りたい場合、これらのライブラリを放棄する必要があります。 もちろん、他の人の間違いを修正するために一生を費やすことはできますが、幸いなことに、より良い代替手段があります（依存関係と機会に関して）。



Openscenegraph



最近出版された2冊の本のおかげで、OpenSceneGraphを発見しました。

1）OpenSceneGraph初心者ガイド、2010年。

2）OpenSceneGraph Cookboox、2012。

2010年にLightfeatherをOGREに変更したときに、OpenSceneGraphに初めて会った。 複雑なライブラリであるOSGは、そのチュートリアルを理解できなかったため、当時私には魅力的ではありませんでした。 幸いなことに、今回は上記の本が救助に来ました。これは、優れたOSG構造、すぐに使用できる一般的な技術のサポート、およびFreeImageに依存しないことを示しています。 OSGはLinuxディストリビューションにも存在します。

すぐに使用できる一般的な技術のサポートは、OGREにあったもう1つの大きな問題です。 OGREはテクスチャマッピング（シャドウマッピング）をサポートしていると主張されていますが、Cgでのみ可能です。 別の中毒！ OGSLのGLSLのテクスチャシャドウに関するチュートリアルはありませんでした。 私はシェーダーをほとんど理解せず、1か月間テクスチャシャドウを実装できなかったため、OGS Mahjong 1.0でステンシルを使用する必要がありました。

1つのOSGブックには、ソフトシャドウマッピングの例が含まれています。 OSGは、すぐに使用できるいくつかのシャドウレンダリング手法をサポートしています。 シェーダーを作成する必要さえありません。 すでに書かれています！ ステンシルシャドウをソフトシャドウに変更するには、コードを1行だけ変更する必要があります。

OGREフォーラムでGLSLのシャドウの実装について尋ねましたが、結果はありませんでした。 したがって、次回OGREフォーラムに投稿数が表示されたときに、そこにあるすべての質問に答えられているとは思わないでください。 複雑なものについては、まったく逆です。

OSGのもう1つの大きな利点は、主な開発者であるRobertがまだOSGの開発に積極的に関与しており、メーリングリストで質問に回答することです（OSGはすでに12歳です！）。 シンドバッドを去った後、OGREは非常に死んでいます。

LinuxディストリビューションにOSGが存在するのも、FreeImageのようながらくたの依存関係がないためです。 たとえば、OSGはlibpngを使用してPNGを処理します。 最終的には、強度の1つまたは2つの100,500の異なる形式を使用しません。



libRocket



CEGUIの唯一の代替。 現時点でいくつかのチュートリアルを行ったところ、libRocketを初めて見た後には見た目ほど良くも美しくもありませんでしたが、このライブラリはHTML / CSSのサブセットを使用してインターフェイスを記述しています（そう、HTML / CSSサポートは嘘です）。 この事実により、GUIを実行する必要がなくなります。 これが成功した場合、libRocketを引き続き使用します。 それ以外の場合は、CEGUIに戻ります。その時点までに、デザイナー向けのCEEDが実装されることを望みます（Kulik、あなたが始めたものを続けてください！）。



Python



MJIN2（新しいゲームエンジン）の最初の要件の1つは、GUIとロジックのPythonサポートでした。 その後、突然、MJIN2をC ++のライブラリーにして、ロジックとGUIだけでなく、完全にPythonのゲームにしてみませんか？ これがSWIGの出番であり、C ++ライブラリのPythonラッパーを生成できます。



Pythonを使用して、開発のペースを上げたいと考えています。 ゲームとエンジンのコミットの履歴を分析しました。 ゲームのコミットはエンジンの2倍であることが判明しました。 したがって、Pythonでゲームを作成すると、コードが少なくなり、開発が速くなるため、開発速度が向上します。



なぜなら AndroidとiOSをサポートする予定です。パフォーマンスの制限により、これらのプラットフォームでPythonを使用しませんが、ゲームの最終バージョンをC ++またはJavaで記述できます。 MJIN2 + SWIGを使用すると、ほぼすべての言語（C ++ネイティブおよびSWIGを使用する20以上の言語）でゲームを作成できます。



ゲーム開発へのアプローチは大きく変化しています。 それが何を注ぐか見てみましょう。
