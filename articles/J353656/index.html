ほとんどのヒントは、単一の開発者には適していません。



インターネットプログラマー向けのヒントの大部分と、私が実際に単一の開発者としてしなければならないことの間には、不一致があります。 その理由は次のとおりです。第一に、ほとんどのプログラマーは他の人とチームで仕事をしているので、通常はこの前提でアドバイスが与えられます。 第二に、人々によって作成されたソフトウェアのほとんどは非常に長い間存在している必要がありますが、これはインディーズゲームには適用されません。 つまり、プログラマーへのアドバイスのほとんどは、インディーズゲームの単独開発にはほとんど役に立たないことを意味し、このため、他の人には不可能な多くのことを行うことができます。



たとえば、グローバル値を使用できるのは、非常に頻繁に役立つためであり、それらを頭の中に保持できる限り、問題を引き起こすことはありません（詳細については、BYTEPATHチュートリアルのパート10を参照してください ）。 また、コードベースがあまり大きくないため、コードの大部分を頭に置いているため、コードにあまりコメントすることはできません。 誰もゲームを構築する必要がないため、自分のマシン上でのみ動作するスクリプトを作成できます。つまり、この手順の複雑さを大幅に軽減でき、作業を完了するための特別なツールは必要ありません。 私は巨大な関数とクラスを持つことができ、それらをゼロから作成し、それらがどのように機能するかを正確に知っているので、それらの膨大な量は問題ではありません。 結局のところ、それらに関連する問題のほとんどは、長寿命のソフトウェアに取り組んでいるチームでのみ明らかになるため、私はこれをすべて行うことができます。



このプロジェクトに取り組んで、私はこれらすべての「悪い」ことをしたとき、特に悪いことは何も起きなかったことを学びました。 多くの開発者が非常に悪いコード記述方法を使用して素晴らしいゲームを作成したという事実を考えると、意識の端のどこかで、インディーゲームを作成するのに超高品質のコードは必要ないことを常に思い出しました。

ECS



Entity Component Systemパターンは、前の2つのセクションで述べたすべての矛盾の良い実例です。 ほとんどの記事を読んだ後、インディー開発者は継承を悪い習慣と見なし、コンポーネントを使用してレゴコンストラクターからエンティティを作成できること、そしてそれらのおかげで再利用可能な動作をはるかに簡単に使用できることが明らかになり、文字通りゲーム作成のすべてがより簡単に。



定義により、ECSに対するプログラマの欲求は時期尚早の一般化について語っています。なぜなら、物事をレゴブロックと見なし、それらから新しいものを収集する方法を考えると、有用なものと組み合わせることができる再利用可能なフラグメントの観点から考えるからです方法。 そして、プリコンパイルに関するセクションにリストした理由から、これは完全に間違っていると思います！ この正確な科学的グラフは私の立場をよく説明しています。









ご覧のとおり、私が保護する「ヨーロコーディング」の原則は、最初ははるかに単純で、徐々に難しくなります。プロジェクトの複雑さが増し、ヨーロテクニシャンが問題を実証し始めます。 一方、ECSは最初ははるかに複雑です。コンポーネントを作成する必要があり、定義上、単純な作業要素を作成するよりも困難です。 しかし、時間が経つにつれて、ECSの有用性はますます明らかになり、ある時点で彼はヨーロコーディングを打ち負かしています。 私のポイントは、MOSTインディーズゲームのコンテキストでは、ECSが最高の投資になる瞬間は決して来ないということです。

動作を複数のオブジェクトに分割しない



回避できないと思われるパターンの1つは、1つの動作を複数のオブジェクトに分割することでした。 BYTEPATHでは、これは主に「コンソール」ルームの作成方法に現れましたが、Frogfaller（以前に行ったゲーム）ではこれはより明白です。





このオブジェクトは、クラゲの別々の脚からのクラゲの本体と、すべてを一緒に結合し、身体と脚の動作を調整する論理オブジェクトで構成されています。 動作が3つの異なるタイプのオブジェクトに分割され、それらの調整が非常に困難になるため、これはそのようなエンティティをコーディングする非常に不器用な方法ですが、この方法でエンティティをコーディングする必要がある場合（およびゲームに多くのマルチコンポーネントエンティティがある場合）、自然にこの解決方法を選択します。



デフォルトでこの分離を選択する理由の1つは、各物理オブジェクトを1つのオブジェクトのコードに含める必要があることです。つまり、新しい物理オブジェクトを作成する場合は、オブジェクトの新しいインスタンスも作成する必要があります。 実際、これは厳密なルールや必須の制限ではなく、物理APIのアーキテクチャを設計した方法のために非常に便利です。



実際、私はこの問題を解決する方法について長い間考えていましたが、それでも良い解決策を見つけることができませんでした。 異なる物理オブジェクト間で調整する必要があるため、1つのオブジェクトだけの単純なコーディングはひどいように見えますが、物理オブジェクトを正しいオブジェクトに分割し、その後の調整も受け入れられないように思われます。 他の人がこの問題をどのように解決するかわかりませんので、あなたのアドバイスを待っています！

なし



プレイヤーから受け取ったバグの90％は、 nil



変数へのアクセスに関連しています。 私はどのタイプのアクセスがより頻繁である/少ない頻度の統計を追跡しませんでしたが、ほとんどの場合、別のオブジェクトがこの死んだオブジェクトへのリンクを保存し、それを使ってしようとすると、オブジェクトの死に関連付けられます。 これは「平均寿命」問題のカテゴリーに属すると思います。



いずれの場合も、この問題の解決策は通常非常に簡単に実装されます。オブジェクトが存在するかどうかを確認し、その後でアクションを実行するだけで十分です。 



 if self.other_object then doThing(self.other_object) end


しかし、この方法でのコーディングの問題は、再保険をかけすぎた別のオブジェクトを参照することです。また、Luaはインタープリター言語であるため、コード分岐に関するこのようなまれなバグが発生します。 しかし、この問題を解決する他の方法は考えられません。バグの深刻な原因であるため、それらを正しく処理するための戦略を立てることは正しいようです。



将来、あるオブジェクトから別のオブジェクトを直接参照するのではなく、それらのIDを使用してそれらを参照することを検討します。 そのような状況で、別のオブジェクトで何かをしたいときは、まずそのIDでそれを取得し、それから何かをしなければなりません： 



 local other_object = getObjectByID(self.other_id) if other_object then doThing(other_object) end


このアプローチの利点は、何かをするたびにオブジェクトを受け取るように強制されることです。 また、私はこのようなことは決してしません： 



 self.other_object = getObjectByID(self.other_id)


これは、現在のオブジェクトに別のオブジェクトへの永続的なリンクを保持しないことを意味します。つまり、別のオブジェクトの死によりエラーが発生することはありません。 これは、私が何かをしたいたびに多くの冗長性を追加するため、私には非常に望ましい解決策ではないようです。 MoonScriptのような言語は、これに似たようなことができるので、これで少し役立ちます。 



 if object = getObjectByID(self.other_id) doThing(object)


しかし、MoonScriptを使用しないため、これに同意する必要があります。



メモリ割り当てのより優れた制御



特に次のゲームでLuaを使用するので、ガベージコレクションが悪いことを主張するつもりはありませんが、その側面のいくつかはまだ好きではありません。 Cライクな言語では、リークの発生はうっとうしいですが、通常はどこで発生するかを大まかに理解できます。 ただし、Luaのような言語では、ガベージコレクターはブラックボックスのように見えます。 これを調べて、何が起こっているかについてのヒントを得ることができますが、これは理想的な作業方法ではありません。 Luaでリークが発生すると、Cよりもはるかに大きな問題になります。これは、私が所有していないC ++コードベース、つまりLÖVEコードベースを使用することで補完されます。 開発者がどのように各自のメモリ割り当てを設定するのかわかりません。そのため、Luaが予測可能なメモリ動作を実現するのははるかに困難です。



速度の点で、Luaガベージコレクターに問題がないことは注目に値します。 特定の制限で動作するように（たとえば、nミリ秒間起動しないように）制御できるため、問題はありません。 唯一の問題は、n msを超えて実行しないように彼に伝えることができ、フレームごとに生成されたすべてのガーベッジを収集できないことです。 したがって、割り当てられたメモリの量を最大限に制御することが望ましいです。 このトピックに関する非常に優れた記事があります： http : //bitsquid.blogspot.com.br/2011/08/fixing-memory-issues-in-lua.html 、およびこの記事でエンジンに到達したときにそれについて詳しく説明します。

部屋とエリア



私にとって、部屋（部屋）とエリア（エリア）の概念は、オブジェクトを扱うのに非常に適した方法であることがわかりました。 部屋は「レベル」または「シーン」に類似しています。 すべてのアクションはそれらで行われ、それらの多くが存在する可能性があり、それらを切り替えることができます。 エリアは、部屋の中に配置できるオブジェクトマネージャの一種です。 一部の人々は、そのようなAreaオブジェクトを「スペース」と呼びます。 AreaとRoomはほぼこのように連携します（これらのクラスの実際のバージョンでは、AreaにaddGameObject



、 queryGameObjectsInCircle



などがあるなど、さらに多くの機能があります）。 



 Area = Class() function Area:new() self.game_objects = {} end function Area:update(dt) -- update all game objects end
      
      

        
        
        
      

    
        
        
        
      
      

        
        
        
      

    
     


 Room = Class() function Room:new() self.area = Area() end function Room:update(dt) self.area:update(dt) end


これらの概念を分離することの利点は、部屋にエリアが必要ないことです。つまり、部屋内のオブジェクトを制御する方法は固定されていません。 ある部屋で、オブジェクトを別の方法で管理することを決定できます。それから、エリアコードを新しい機能に適応させる代わりに、コードを書くだけです。
