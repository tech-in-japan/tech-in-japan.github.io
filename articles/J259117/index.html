「Jetpack」はそれほど反応的ではない、またはそれなしで行う方法

まず第一に、この記事は、初心者やもう少し上級のユーザーや、5分以内にWordPressのインストールに実際に適合する人にとっては興味深いものになります。



名前が示すように、同様に人気のある自動開発チーム、つまりJetpackからの人気のあるプラグインについて説明します。



Jetpackの無気力について考えましたが、VPSを安価な料金に変更するとすぐに気づきました。 主催者の名前は付けません-これは別の話です。 新しいサーバーの構成はこれでした。



鉄



  • AMD Opteron（tm）プロセッサー6386 SE
  • 2800 MHz
  • RAM 1024 mb


システム



  • Ubuntu Server 14.04
  • PHP 5.5.9
  • MySQL 5.5
  • Nginx 1.8（ポート8080）
  • Memcached
  • ニス（ポート80）


追加のサードパーティプラグインのうち、W3 Total CacheとYoastのWordPress SEOもインストールされました。これは、驚くほどサーバーをあまりロードしません。 しかし、まず最初に。



ですから、Jetpackをシャベルすることにした最初の、そしておそらく最も重要な理由は、 TTFB （Time To First Byte）インジケーターです。 190〜220ミリ秒の範囲で、Jetpackをインストールした後、この数値は320ミリ秒に増加しました。 私は強調します-Jetpackをインストールした後です。 つまり、すべてのプラグインコンポーネントを無効にしたとしても、とにかくWordPressコアに既に接続されているため、抑制を取り除くことはできません。 私はすぐにすべての個々のコンポーネントの代替品を探し始めました。それが起こったのです。



  1. カスタムCSS 、 エクストラサイドバーウィジェット 、 無限スクロール 、 サイトアイコン、モバイルテーマ、サイト検証ツール 、 共有などのコンポーネントは、手が脳に従い、適切な場所から成長すれば自分でねじ込むことができると信じています。 GenerateWPのようなすばらしいサービスについて聞いたことがない場合は、こちらをご覧になることをお勧めします。
  2. ショートコードの埋め込みは良いことですが、Jetpackプラグインを使用している間は、特にWordPressではデフォルトで多くのショートコードを挿入できるので、特に便利ではありません。
  3. Jetpackコメントは多くの人を引き付ける可能性が高いですが、少なくとも一度は、あなたが興味を持っている.jsスクリプトの数を数えることに関与したことがあります。 つまり、Google Pagespeedの良い指標に頼ることはできません。 私が自分で選んだ最も成功した決定の1つは、Disqusからのコメントであり、WordPressに統合するためのプラグインはDisqus Conditional Loadでした。
  4. 正直言って、 Beautiful Math 、 Enhanced Distribution 、 Markdown 、 Post By Email 、 Spelling、Grammarなどのコンポーネントを少なくとも1回使用しました。 個人的にはそれを使用しましたが、本当の利点は見られませんでした。 メールでの投稿に関しては、このオプションはWordPressに長く存在していましたが、Cronを起動する必要があるため、誰もが最初にそれを処理できるわけではありません。




詳細については、 Photon 、 Publicize 、およびProtectの各コンポーネントに触れたいと思います。これらのコンポーネントは、原則として非常に有用なツールであると思われました。 しかし、3つのビューがあります。



1つ目はPhotonで、WordPress.com CDN（コンテンツ配信ネットワーク）から画像をアップロードします。 これは、画像へのリンクがから変換されることを意味します site.ru/wp-content/uploads/2015/05/image.jpg



site.ru/wp-content/uploads/2015/05/image.jpg



を似たようなものに i2.wp.com/site.ru/wp-content/uploads/2015/05/image.jpg



i2.wp.com/site.ru/wp-content/uploads/2015/05/image.jpg



、つまり、検索エンジンはドメインのインデックスを作成しません。



2番目の「しかし」はPublicizeに影響しました。 今日のソーシャルネットワークでの投稿の自動共有は、リソースを促進する上で非常に重要で重要なツールですが、VKontakteのVKontakteでの共有の欠如が少し私を夢中にさせました。 特定のホリバーを挙げて、外国のサービスがしばしばAPIでロシアのソーシャルプロジェクトのサポートを単に無視する理由をたくさん述べたくはありませんが、一方で、VKontakteの大勢の聴衆にとっては無礼です。 Social Networks Auto-Posterプラグインで代替が見つかりました。



WordPressサイトの管理パネルを保護するための最新の、そしておそらく私の意見では、おそらく最も成功したソリューションの1つは、新しいProtectコンポーネントです。 そして、Jetpackプラグインとは別に、このようなすばらしいツールをどのように使いたいか。 結局のところ、 Protectコンポーネントで障害を見つけることはほとんど不可能であり、WordPress.comネットワークレベルで機能するため、パスワードの推測から保護されます。 この点で、私はkovsheninに完全に同意します。彼はProtectも良いツールだと考えています。 考え直さずに、私はまだ2要素認証を選択し、3番目の「しかし」が発生しました。



一般的に、WordPressの世界でJetpackのようなプラグインはリーダーの1つであり、多くの人にとって不可欠なツールです。 しかし、私にとっては、Jetpackは見た目ほど「反応的」ではありません。
