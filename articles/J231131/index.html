APS 2.0を使用してBackup-as-a-ServiceをParallels Automationと統合した方法

HabréはすでにSaaSの開発およびビジネスモデルについて多くのことを書いています。 たとえば、 この記事では、パートナーを通じてサービスを販売する問題を取り上げています。 「ピースグッズ」とサービスが非常に高価な場合でも、パートナーは顧客を登録し、サービスを手動で構成できます。 しかし、このアプローチが大規模なサービスでは機能しないことは明らかです。 そして、SaaSをクラウドプロバイダーのOSS / BSSプラットフォームと統合して、サービスの提供、課金などのプロセスを自動化することが必要になります。







この記事では、OSS / BSSプラットフォームとの統合に必要なものと、オープンスタンダードAPS 2.0を使用してBackup-as-a-ServiceをParallels Automationプラットフォームに統合する方法について説明します。 一般に、これらのヒントは、クラウドサービスの開発者がサイトから販売するだけでなく、サービスプロバイダーに許可する場合に役立ちます。



誰が誰ですか？



まず、サービスとしてのバックアップ、Parallels Automation Platform、およびApplication Packaging Standard（APS）のすべてについて説明します。



Backup-as-a-Service -BaaSについては既に以前に habrに書き込みました 要するに、これはSaaSサービスのタイプの1つです。 クラウドとバックアップの両方を組み合わせます。すべてのデータはWeb経由でコピーされ、プロバイダーのデータセンターに保存され、バックアップクライアントプログラムはユーザーのコンピューターにインストールされます。 これにより、ストレージシステムからシステムソフトウェア（必要性がなくなる）とそのすべてのメンテナンスまで、インフラストラクチャ全体をサービスプロバイダーの側に転送できます。



アプリケーションパッケージングスタンダードは、2007年にParallelsが開発したオープンテクノロジー標準です。 独立開発者（ISV）は、任意のサービスプロバイダーを介してクラウドアプリケーションを販売でき、サービスプロバイダー自身が販売用の新しいクラウドサービスを迅速に展開および開始できます。 現在、世界のAPS標準には500を超えるアプリケーションがあります。



APS 2は、2013年にリリースされた標準の新しいバージョンです。 前のものと比較して、柔軟性、パフォーマンス、およびサービスバスがあり、異なるサービスが相互にやり取りできるようにします（データ交換と同時使用のための相互統合）。 新しいクラウドアプリケーションでも、長年使用されているクラウドアプリケーションでもかまいません。 また、サービスの利用者は、企業サブスクリプションにそれらを追加し、単一のコントロールパネルからすべてのサブスクリプションを管理し、すべてのサービスにアクセスするための単一の認証を受け取ることができます。



Parallels Automationは、サービスプロバイダーまたは通信会社によってインストールされるOSS / BSSプラットフォームであり、APSにパッケージ化されたクラウドサービスを起動および販売し、サービスプロビジョニングから請求、単一アカウントへの統合まで、必要なすべての操作を実行できます顧客および再販機能。



同時に、クラウドサービスは、SaaS、IaaSなど、どのようなタイプでもかまいません：Parallels Automationエコシステム内では、プラグアンドソケットの原理は可能な限り完全に機能します（パッケージ化されたサービスの接続は数分で完了します）。 そのため、開発者がParallelsパートナーネットワーク（可能な販売者の9000を超える）でサービスを販売したい場合、APSを介した統合という論理的な選択肢があります。



たとえば、Parallels Automationで最も売れたクラウドアプリケーションのトップ10は、Microsoft Office 365（現在80万件以上のサブスクリプションを販売）、Microsoft Hosted Exchange（70万件以上のサブスクリプション）、Open-Xchange、SpamExperts Integration、Hostopia、 Mail2World、Microsoft Lync 2010、MozyPro、IDSync、BackupAgent（ご覧のとおり、BaaSはトップ10に入っています）。

OSS / BSSプラットフォームとSaaSサービスの統合



そこで、サブスクリプションのライフサイクル、つまり作成から削除までを検討しました。 次に、プラットフォームがそれ自体で行うタスクの部分と、新しいサービスを接続するときに行う必要がある変更を把握します。



  1. 新しいサービスの場合、販売可能なリソースのセットを定義する必要があります。 プラットフォームは、計画の設定と公開を行います。

  2. クライアントによるサービスの注文と支払いは、通常完全にプラットフォーム上にあります。

  3. サービスのプロビジョニングと初期構成を自動化する必要があります。

    私たちの場合、このためには、次のメソッドAPI呼び出しを実装する必要があります。

    • BaaS側の組織の作成
    • 組織の制限を設定する


  4. クライアントにサービスを独立して管理する機会を与える必要があります。

    クライアントパネルのWebインターフェイスを統合するための3つの主なオプションがあります 。

    • プロバイダーのクライアントパネルにサービスのインターフェースを埋め込むことができます。

      私の観点からは、これは最も望ましいオプションです。ユーザーが1つのパネルからすべてのサービスを管理する方が便利だからです。

    • クライアントパネルから別のサービスパネルにクライアントをリダイレクトできます。 この場合、シングルサインオン（シングルサインオン）のテクノロジを実装することをお勧めします。これにより、クライアントがさまざまなユーザー名とパスワードを何度も記憶、保存、入力する必要がなくなります。
    • 3番目のオプションは、最初の2つの組み合わせです。 最も頻繁に使用されるスクリプトがクライアントパネルから利用可能であり、クライアントが他のタスクを実行するためにサービスパネルにリダイレクトされる場合。


    私たちのサービスのために、私たちは第三の道を行きました。

  5. 通常、サービスの拡張はプラットフォームにありますが、クライアントが使用したリソースの料金を支払う場合、その使用に関する統計を収集する必要があります。

    このケースでは、APIメソッドを定期的に呼び出してリソースに関する情報を収集します。

  6. サービスの停止、削除を自動化する必要があります

    私たちの場合、これは次のメソッドAPI呼び出しです。

    • 組織の有効化/無効化
    • 組織の削除




ご覧のとおり、新しいサービスをプラットフォームに接続する簡単なオプションの場合、それほど多くの変更は必要ありません。 もちろん、特に複数のサービスが相互に作用する場合、複雑なオプションがあります。



上記は、クラウドサービスの提供と販売をサポートするすべてのプラットフォームで一般的です（ご存じのように、多くの通信会社やプロバイダーは、さまざまなベンダーのOSS / BSSプラットフォームを所有している場合があります。 しかし、Parallels Automationとの統合が特に必要です。これは多くのパートナーが使用しているだけでなく、ヨーロッパの最大の電気通信の半分のコストがすでにかかっています。 このために、サービス用のAPSパッケージを作成する必要があります。



Backup-as-a-Service用のAPSパッケージの作成



APSパッケージの開発を開始するには、以下を行う必要があります。

  • サイトに登録する
  • テスト環境をリクエストする
  • 研究文書
  • 開発環境を準備します。


特別なAPS ECLIPSE IDEプラグインが用意されているため、Eclipseで開発を続けています。



APSプラグインがインストールされたEclipse。







Eclipseをセットアップしたら、新しいAPSプロジェクトを作成できます。

APSパッケージの構造を考慮してください。 完成したパッケージは、以下を含む.zipアーカイブです。

  • APP-META.xml-パッケージ記述ファイル。 サービスおよびAPSパッケージに関する一般情報が含まれています。
  • /スキーマ-APSリソースのJSONスキームが含まれます。 通常、/スクリプトからのリソースの説明に基づいて「aps build」コマンドによって自動的に作成されます
  • / scripts-サービス、リソース、およびサービスを提供するためのロジックを記述するスクリプトが含まれています。

    スクリプトはさまざまな言語で作成できます。 パッケージには、次の理由でphpを選択しました。

    • phpには、パッケージ開発を簡素化するAPS PHPランタイムライブラリがあります
    • アクセシブルな例もPHPで書かれています
  • / ui-組み込みWebインターフェース用のHTMLおよびJavaScriptが含まれています。

    組み込みのWebインターフェースを作成するために、 APS JS SDK JSライブラリーが使用されました。これは、使用されると、コントロールパネルのスタイルが継承されるためです。

  • / i18n-ローカライズファイルが含まれています




だから：

  • プロジェクトを作成しました
  • APP-META.xmlにパッケージの説明を追加
  • リソースモデルの定義と説明
  • ユーザーシナリオの定義：ユーザーがサービスを管理する方法。 Webインターフェースの描画レイアウト。
  • Webインターフェースを作成しました
  • サービスとしてのBackup-as-a-Service in /スクリプトへのメソッドAPIコールの実装
  • / i18nの必要な言語にローカライズを追加




ヒント： APSパッケージを作成する開発者には、リソースモデルが記述され、レイアウトの準備が整った段階で、フィードバックの開発を開始する前にAPSチームに連絡することをお勧めします。 私の練習では、これにより多くの時間を節約できることが示されています。

数週間後、パッケージの準備が整い、認定のために合格しました。



APSパッケージ認証



ヒント：パッケージを作成した後、その認定に合格することをお勧めします。 この手順では、パッケージ内のエラーを特定して排除します。



認証プロセス中、ParallelsのAPSチームは、アプリケーションが正常に動作し、プラットフォーム全体または特定のパートナーで動作することを確認します。 必要に応じて、推奨事項が提示されます。 認証は完全に無料です。



認証プロセス：

  1. 認証のためにパッケージを送信する前に、仕様と要件を満たしていることを確認する必要があります 。
  2. その後、パッケージをAPS開発ポータルにアップロードして、パッケージを自動的にテストする必要があります。 その後、システム内にチケットが自動的に作成され、両側の開発者が通信します。
  3. テストが成功した場合は、手動チェックをリクエストする必要があります。 検証結果に基づいて、APSチームはパッケージを改善するための推奨事項を送信します。 これは、認証プロセスが本質的に終了する場所です。
  4. パッケージが認証に合格すると、 APSディレクトリに公開されます。


パッケージの認証には数日しかかからず、パッケージで何を調整する必要があるかに応じて、平均で3日から2週間かかります。



まとめ



パッケージの認証後、Parallels Automationを使用する最初のパートナーにパッケージを提供しました。 BaaSサービスに接続し、計画を立て、最初のテストサブスクリプションを作成するのに1時間もかかりませんでした。 現在、当社のAPSパッケージは世界中のプロバイダーによって積極的にテストされています。



SaaS、PaaS、またはIaaSである他のXaaSサービスは、APS 2標準を使用してParallels Automationに簡単に統合できます。これにより、ロシアを含む世界中の数百のプロバイダーを通じて販売し、他のアプリケーションと統合できます。 また、これは優れた販売チャネルであるだけでなく、アプリケーションが他の誰かとパッケージで販売されている場合の余分な金額でもあります。
