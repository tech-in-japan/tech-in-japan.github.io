Qtのシグナルスロットシステム用Kostylik





みなさんこんにちは。 このことについてお話したいと思います...約2週間前、私は最初にC ++でGUIを操作する必要があり、少しGoogleを試した後、Qtを使用することにしました。 誰もが彼をひどく賞賛し、実際、一見すると彼は非常にふさわしいように見えました。



一見したところ、Qtも良いことでしたが、そのメソッドオブジェクトコンパイラの実装にはいくつかの制限があるため、小さな松葉杖を設計する必要がありました。 このエッセイでは（「エッセイ」という言葉は、ボリュームのある記事を引っ張らないので、より適していると思います）、生じた問題をどのように解決したかについてお話したいと思います。



追加：親切な人々のコメントに基づいて、MOCなしで、一般的に松葉杖なしで行う方法を示す編集が行われました。







それがすべて始まった方法





それはすべて、ライブラリ内のMVCアーキテクチャの実装の一部としてテンプレートコントローラークラスを作成する必要があるという事実から始まりました。 コントローラーは、異種の整数データと対話し、 QSpinBoxをGUIとして関連付けなければなりませんでした 。 すべての殻を破棄すると、次のようなものが出てきました。



テンプレートコントローラー 
template< typename T_IntegralType > class IntegralController { private: T_IntegralType *_modelField; QSpinBox *_view; ... public: QSpinBox *getView() { if (!_view) { _view = new QSpinBox(); /*     */ } return _view; } . . . private: // ,        void setValue(T_IntegralType inValue) { * modelField = inValue; } };






コードをスケッチした後、Qtでのイベント処理について読み、一般的なイベント、特にGUI要素からのイベントを操作するには、 信号とスロットのシステムを使用する必要があることに気付きました（このシステムはこの記事でよく分解されています -これは公式ドックの翻訳のようです） 。



モデルビューアプローチに関する注意
また、Qtのモデルビューアプローチのフレームワークにはデリゲートシステムなどがあり、シグナルスロットシステムを使用せずに、インターフェイスの実装を通じてビューとモデル間のデータ転送を処理できます。 特定の理由により、ライブラリでQtモデルビューを適切に使用できませんでした。





任意のクラスがシグナルスロットシステムにスロットを提供できるようにするには、このクラスがQObjectクラスを継承し、 Q_OBJECTマクロを含める必要がありました。 なんで？ それから私はそれを理解するためにスチームバスを取らなかった。 それは必要です-それはそれが必要であるということです。 さらに苦労せずに、彼は必要なものをテンプレートクラスに追加し、GUIイベント処理の実装について説明しました。



イベント駆動型テンプレートコントローラー 
 template< typename T_IntegralType > class IntegralController : public QObject { Q_OBJECT private: T_IntegralType *_modelField; QSpinBox *_view; ... public: QSpinBox *getView() { if (!_view) { _view = new QSpinBox(); QObject::connect(_view, SIGNAL(valueChanged(int)), this, SLOT(valueChanged(int)); } } . . . private slots: void valueChanged(int inValue) { *_modelField = inValue; } };






すべてが集まった。 まあ、私はなんて素晴らしい仲間だと思ったんだ！ -そして、しばらくの間、このテンプレートコントローラを忘れて、ライブラリを見続けました。 もちろん、彼はそれが得意ではなかったので、無駄に急いで、一般的に早く喜びました。 テンプレートコントローラーの特殊化を使用するコードをコンパイルしようとするとすぐに、リンカーエラーメッセージが次のように降ってきました。

メタオブジェクトシステムで何か間違ったことをしていたのは明らかでした。 MOCについて読み直さなければなりませんでした。 ソースコードを通過するときに、MOCは追加のcppファイルを生成し、作成されるシステムのメタオブジェクトに必要なメソッドの実装が作成されることが判明しました。 このシステムはテンプレートを非常に曲がりくねって動作し、この非常にメタオブジェクトのコードを貧弱に生成します-MOCはコードを生成するときにテンプレートクラスを単に無視するか、通常のクラスとして認識し、テンプレート引数を破棄します。



テンプレートとQtの詳細
シグナルスロットシステムでテンプレートを使用しない理由については、ドックに別の記事もあります。 残念ながら、私の場合、それは最適性に関するものではなく、コードの量を適切に削減し、大量のコピーアンドペーストを回避することに関するものでした。したがって、この記事のポイントは適切ではありませんでした。





MOCにさらに慣れると、MOCが制御するクラスの使用にはさらにいくつかの制限がありました。 最も不愉快なのは、ネストされたクラスでQObjectの完全に管理されたMOCの子孫を記述できないことです。 そして、私はネストされたクラスを作り、その中に住んでいる小さな古典の間の大きなクラスの責任の領域を壊します。 はい、私は知っています、労働プログラマーは名前空間を使用しますが、同時に、私の意見では、プログラムのコンテキストは乱雑であり、ネストされたクラスのセマンティクスは名前空間のセマンティクスとは異なります（最初のケースでは、クラス間の関係の階層を構築します。何か）。



一般的に、MOCシステムからは静的QObject :: connect（...）メソッドを介してイベントをサブスクライブする機能のみが必要であるため、小さな松葉杖を作成することにしました...非常に小さな、小さな松葉杖。



注：判明したように、Kostylikは、曲がったMOCが支配した5番目よりも新しいQtのバージョン用であることが判明しました。 Qt 5バージョンには、このMOCなしで信号スロットを許可するクールなAPIがあります。 この情報を記事の最後に追加しました 。 繰り返しになりますが、道徳は私にとっては、記事の怠authorな著者であり、まったくうまくいっていません。Googleを7回、記事を一度書いてください 。



松葉杖について





アイデアはシンプルでした。QObjectを継承し、信号スロットシステム内での正しい動作のためにクラスを登録するという意味で、MOCに完全に適したミニクラスを作成することです。 このミニクラスは、Qtに依存しないコールバックをこの補助クラスのスロット呼び出しに関連付けるためのメソッドも提供します。



複雑に聞こえますが、例がより明確になると思います。 QSpinBoxからのイベントを処理するコードでは、次のようになりました。



QSpinBoxからのイベントのKostylik 
 class ValueChangedWorkaround : public QObject { Q_OBJECT public: // -,  .     //  FastDelegate (     , ) typedef fastdelegate::FastDelegate1 < int > Callback; private: Callback _callback; public: ValueChangedWorkaround() : _callback() { } void bind(QSpinBox *inSpinBox, const Callback &inCallback) { _callback = inCallback; QObject::connect(inSpinBox, SIGNAL(valueChanged(int)), this, SLOT(valueChanged(int)); } private slots: void valueChanged(int inValue) { _callback(inValue); } };




FastDelegate全般について

GitHubのFastDelegate





私はこのコードをコントローラーで使用しました-そしてそれは機能しました：



松葉杖コントローラー 
 template< typename T_IntegralType > class IntegralController { private: typedef IntegralController< IntegralType > OwnType; T_IntegralType *_modelField; QSpinBox *_view; ValueChangedWorkaround _valueChangedWorkaround; ... public: QSpinBox *getView() { if (!_view) { _view = new QSpinBox(); _valueChangedWorkaround.bind(_view, ValueChangedWorkaround::Callback( &OwnType::valueChanged)); } } . . . private: void valueChanged(int inValue) { *_modelField = inValue; } };






これは落ち着くように思えます...しかし、私は普遍的なソリューションの病理学的なファンなので、産業規模でQtオブジェクトからのさまざまなイベントを処理する松葉杖を作成できるマクロを作成することにしました。







松葉杖ジェネレーター





古いマクロに基づいて新しいマクロを作成し、単にいくつかの識別子をマクロ引数に置き換え、マクロ自体をわずかに一般化することができるように思われます。



松葉杖を生成するマクロ。 バージョン1.0
ここではスラッシュを省略します（「\」など）-激怒します！ 



 define QT_EVENT_WORKAROUND_1_ARG(M_WorkaroundName, M_EventName, M_Arg0Type) class M_WorkaroundName : public QObject { Q_OBJECT public: typedef fastdelegate::FastDelegate1 < M_Arg0Type > Callback; private: Callback _callback; public: M_WorkaroundName() : _callback() { } void bind(QObject *inQSignalSource, const Callback &inCallback) { _callback = inCallback; QObject::connect(inQSignalSource, SIGNAL(M_EventName(M_Arg0Type)), this, SLOT(M_EventName(M_Arg0Type)); } private slots: void M_EventName(M_Arg0Type inValue) { _callback(inValue); } };






このマクロを書いたので、私は自信を持って、今は確かによくできていて、一般的に松葉杖の領主（上記の男のような）であると思いました。 非常に喜んで、私はコードを実行しました...はい、もちろん、何も機能しませんでした。 コンパイルエラーはなく、すべてが収集されましたが、コールバックは呼び出されず、メッセージがログに送信されました。たとえば、 TestWorkaroundクラスに必要なスロットがありません。



さらに掘り下げなければなりませんでした。 QtのMOCはマクロを展開する方法を知らないことが判明しました。 プリプロセッサが実行される前にコードを通過します（つまり、MinGWで-Eフラグを使用してビルドした場合に表示されるコードではなく、処理されていないコードによって実行されます）。

一方、MOCは、「スロット」という単語の後にクラスの宣言ブロックにあるメソッドシグネチャを知っている必要があります-文字列として読み取り、 QObject :: connectを呼び出すときにこれらの文字列名を使用します（SLOTおよびSIGNALマクロがこれらの名前+使用場所）。 したがって、松葉杖を生成するマクロユーザーは、スロットの独自の実装を記述する必要があることが明らかになりました。



私はこのコードの量と複雑さを最小限に抑えようとしましたが、最終的な解決策は次のようになります（すでに最終的なコードであり、不敬なスラッシュが含まれています）。 



 #define SIGNAL_WORKAROUND_1_ARG(M_WorkaroundName, M_CallName, M_Arg0Type)\ class M_WorkaroundName : public QObject {\ public:\ typedef fastdelegate::FastDelegate1< M_Arg0Type > Delegate;\ \ private:\ Delegate _delegate;\ \ public:\ void bind(QObject *inQSignalSource, const Delegate &inDelegate) {\ _delegate = inDelegate;\ QObject::connect(inQSignalSource, SIGNAL(M_CallName(M_Arg0Type)),\ this, SLOT(M_CallName(M_Arg0Type)));\ }\ \ void CALL(M_Arg0Type inArgument) { _delegate(inArgument); }\
-ご覧のとおり、マクロはクラスを完全には記述していません。ユーザーは、CALL（...）メソッドを呼び出すためのスロットの記述を完了する必要があります。 以下の松葉杖ジェネレーターの使用に関する完全な説明...



完全な説明：

1.マクロを使用してどこかに松葉杖クラスを生成します。 この例では、 YourWorkaroundNameと呼ばれる松葉杖を生成します。これは、タイプEventArg1Typeの 1つの引数を取るqtEventNameイベントをラップします。 クラスを生成するコード： 



 SIGNAL_WORKAROUND_1_ARG(YourWorkaroundName, qtEventName, EventArg1Type) Q_OBJECT private slots: void qtEventName(EventArg1Type a0) { CALL(a0); } };




2.ラップされたタイプのイベントを送信できるQtオブジェクトからのイベントを処理する必要があるコードの任意の場所で新しいタイプを使用します（この例では、タイプEventArg1Typeの 1つの引数を送信するqtEventNameイベント）。 松葉杖を使用したコード例： 



 class UserClass { private: QSomeObject *_qTestObject; YourWorkaroundName _workaround; public: UsingClass() : _qTestObject(new QSomeObject()) { _workaround.bind(_qTestObject, YourWorkaroundName::Callback(&UsingClass:: onEvent)); } void onEvent(EventArg1Type inArg) { /* Some actions on callback */ } }




すべて準備完了です。 これで、Qt MOCによる制限なしに、任意のクラスのQtオブジェクトからのメッセージを処理できます。



結論として、いくつかのコメント：

1.提案されたマクロは、単一の入力引数を取るイベントに適しています。 異なる数の引数を処理するには、このマクロ（古いスタイルのスタイル）をコピーして貼り付けるか、C ++ 11の可変長マクロを使用します 。

2.提案されたソリューションは、 FastDelegateライブラリを使用してコールバックを処理します。 ファンクターを使用する場合は、マクロ内のFastDelefateをご使用のタイプに簡単に置き換えることができます。

3.このソリューションでは、エラー処理、アサーションなどはありません。私の場合、このような処理は必要ありません。 味に追加できます。

4.提案されたソリューションが地獄のようなものであることに同意する準備ができており、Qt MOCの制限に対処する他の方法についての提案を喜んで聞きます。 私はあなたの提案を記事に追加し、決定の原作者を示し、私自身から感謝の意を表します。 事前に感謝します！

AlexPublicとVioletGiraffeに感謝します 。新しいAPI信号スロットに鼻を突っ込みました。



松葉杖ジェネレーターについて話すと、次のようになります。



松葉杖ジェネレーター 
 #define SIGNAL_WORKAROUND_1ARG(M_WorkaroundName, M_CallName, M_Arg0Type)\ class M_WorkaroundName : public QObject {\ public:\ typedef fastdelegate::FastDelegate1< M_Arg0Type > Delegate;\ \ private:\ Delegate _delegate;\ \ public:\ template< typename T_ListenerType >\ void bind(T_ListenerType *inQSignalSource, const Delegate &inDelegate) {\ _delegate = inDelegate;\ connect(inQSignalSource,\ static_cast< void (T_ListenerType::*)(M_Arg0Type) > &T_ListenerType::M_CallName),\ this, &M_WorkaroundName::M_CallName);\ }\ \ void M_CallName(M_Arg0Type a0) { _delegate(a0); }\ };




使いやすくなりました。 これで、テンプレートコンテキストを含め、どこでもラッパータイプを作成できます。 このようなもの： 



 template<typename T_SomeQObject> class UserClass { private: SIGNAL_WORKAROUND_1ARG(YourWorkaroundName, qtEventName, EventArg1Type); T_SomeQObject *_qTestObject; YourWorkaroundName _workaround; public: UsingClass() : _qTestObject(new T_SomeQObject()) { _workaround.bind(_qTestObject, YourWorkaroundName::Callback(&UsingClass::onEvent)); } void onEvent(EventArg1Type inArg) { /* Some actions on callback */ } }






ただし、この新しいQt5 APIシグナルスロットを使用すると、MOCとその制限を完全に取り除くことができます。 したがって、松葉杖を使用する代わりに、QObjectからリスナークラスを継承するのが簡単です（MOCへのバインドがなくなるため、リスナーの相続人は定型句などになる可能性があります）。 AlexPublicで提案されているように、C ++ 11以前を使用し、ラムダを使用する 場合 。
