かんばんスタイルのスタンドアップ

スタンドアップミーティング、デイリースクラムミーティング、または単なるプランニングミーティングがITの一般的なプラクティスになっています。 5年前のスタンドアップミーティング：ベストプラクティスとワーストプラクティスで、 スタンドアップのさまざまなニュアンスについて説明しました。 スタンドアップを実施する手法は、すでにあらゆる側面から検討されているように思われます。 計画セッションで何が複雑になりますか？ しかしごく最近、当社はわずかに異なるアプローチを実践し始め、それによりリリースへのタスクのリリースを加速しました。



2014年の夏にモスクワで、 Askhatと私がトレーニングに行き、スクラムとカンバンのスタンドアップの違いに注意を向けたときにすべてが始まりました。 それ以前は、そのようなニュアンスをあまり重視していませんでした。 当社では、プロジェクトの一部にかんばんが使用されていますが、スタンドアップスタイルはスクラムから残っています。



今、私たちはスタンドアップへのアプローチを変更しました。これについては後で説明し、以前と現在との違いを示します。 記事の最後に、さまざまな意見やニュアンスの説明を含むトピックへのより深い没入のためのリンクがあります。



スクラムとかんばんのスタンドアップの違い



スクラムを初めて使用する場合は、初心者向けにDaily Scrumについて読むことをお勧めします。


スクラムでは、スタンドアップは人間中心です-各チームメンバーが交代で最終日の結果について話し合い、当日の約束をし、問題を共有します。 事実、スクラムでは、計画中に反復するためにチームが与える約束を果たすことに重点が置かれています。 約束を果たすためにはすべてを行わなければなりません。 スタンドアップの目標は、すべての反復タスクを完了できるかどうかを追跡し、完了できない場合は、問題をできるだけ早く理解し、アクションを起こすことです。

次に、かんばんについて説明します。 使用していない場合は、無料の本Priming Kanbanを最初に読むことをお勧めします。


かんばんは他の目標を追求しています-リードタイムを最小限に抑えることが重要です。 すべての段階でタスクに取り組む時間。 この点で、スタンバイへのアプローチも変化しています。 これは、ボードに焦点を合わせて行われ、タスクのフローに焦点を当て、 ボトルネックを検出します。 右から左へのタスクの列を調べて、タスクを次のステージに迅速に転送する方法を説明していることがわかります。 各タスクについて議論するとき、チームメンバーは誰でも話すことができます。

一般的に、スタンドアップはかんばんの一部ではありません。 かんばんでは、特に慣行はなく、基本的な原則はほんのわずかです。 スタンドアップは、 継続的な改善の原則を実装する方法の1つと考えることができます。


合計すると、スクラム会議はタスクについて話す人を中心に行われ、かんばん会議はタスクを中心に行われます。 言い換えると、スクラムには、 Programmer 1 <-> * User Story接続があり、かんばんには、 User Story 1 <-> * Programmer接続があります 。



プロセスの説明



かんばんでのスタンドアップの構成を順を追って見てみましょう。 たとえば、ウィキペディアの記事の掲示板を検討します 。

  1. チーム全体がボードの周りに集まります。 チームが分散され、ボードが電子的な場合、全員がこのボードを開いて呼び出します。
  2. リーダーになる人を任命することをお勧めします。 チームのメンバーや、オープンディスカッションの実施方法を知っている他の人が参加できます。
  3. 列を右から左に、タスクの場合は上から下に移動します。 一番下の行は、右端の列が作業の完了を示しているため、完了に最も近いタスクの価値が高いことです。 タスクを右端の列に速く変換するほど、リードタイムは短くなります。
  4. この例では、右端のユーザーストーリー754です。ホストは、「タスク754を[展開済み]列に移動できないのはなぜですか？」と尋ねます。 何人かの人々が理由を伝え、会社の長からの確認を待っていることを説明できます。 この場合、タスクは、そのためにブロックされたというステッカーまたはコメントで明確にマークされます。
  5. 次のユーザーストーリー75.進行役は同じ質問をします。 たとえば、プリプロダクション環境でのテストを担当するチームメンバーの1人が、このタスクを自分の仕事に取り入れていると言います。 彼はタスクを含むカードを受け取り、「プリプロダクションシステムでのテスト」列に「プル」します。 このタスクでは、誰が彼女をステッカーとして働かせたかに注意します。
  6. 次に、チームのリソースがなくなるまで、ボード上のすべてのタスクを調べます。 次のスタンドアップで次の列にタスクを転送するため、または作業をブロックするものを伝えるために、全員が自分でタスクを取ります。




ご覧のとおり、カンバンでは、スクラムとは異なり、プロジェクトの状況を視覚化したボードが必要です。 なぜなら あなたは人々の特定の行動について話しているのではなく、タスクの現在の状況とその流れについて、これらのタスクがどこにあるかを見ることが非常に重要です。 これが、かんばんスタイルのスタンドアップがストーリー重視のスタンドアップまたはワークアイテム参加とも呼ばれる理由です。



さらに、スクラムで行われるように、3つの質問をすることができます（または、カンバンです）。ただし、これらの質問は、チームメンバーではなく、タスクに関するものです。

  1. タスクの移動を妨げるものは何ですか？
  2. タスクはどのように流れますか？
  3. 何を改善できますか？




移行結果



スタンドアップのスタイルをスクラムからカンバンに変更すると、すぐに結果が表示されます。 例として、新しいスタンドアップを適用したプロジェクトの累積フロー図を示します。 3月24日にこれを行いました。状況がどのように変化したかを確認できます。リリースではタスクの出力を増やしました。





累積フロー図の説明のプレゼンテーションで、累積フロー図の詳細を確認することをお勧めします


タスクを凍結する理由



これより前に、タスクが最終段階でハングし、リリースされなかった理由を理解することが重要です。 チームの全員がITプロジェクトの管理プロセスを理解しており、顧客は現在の状況を認識していますが、タスクが長時間待機していることがあります。 いくつかの理由を特定しました。

  • 多数のブロックされたタスク。 たとえば、タスクが最後から2番目の段階に到達し、製品所有者（PO）からの確認を待っています。 この場合、POがビジーになり、フォーカスが他のタスクに変更される可能性があります。 タスクはリリースから1ステップでハングし、多くの作業時間が既に費やされており、少し努力する必要があります。 スクラムスタイルでスタンドアップを作成する場合、このタスクに長い間関与していないため、そのようなタスクをリリースに取り込む人はいません。 かんばんスタイルでスタンドアップを作成すると、そのようなタスクがすぐに表示されます。さらに、各スタンドアップに戻ります。 右から左に行きます。
  • タスクの優先度は変化しています。 タスクは最終段階に到達し、突然あまり重要ではなくなり、チームは新しいタスクに切り替えます。 POは優先順位を変更し、作業を完了することができなかったことが判明しました。 スクラムスタイルのスタンドアップ中に、チームが現在取り組んでいるタスクの議論があるため、最初は、誰もがこの状況に耐えました。 以前のすべてはそれほど面白くありません。
  • 仕事は彼女に割り当てられた時間を埋めます。 。 反復が3週間続く場合、すべてのタスクは反復が完了するまでボード上のどこかでハングします。 いくつかのタスクをより早く実行できる場合でも、最終段階に到達しない場合があります。 スクラムにはリードタイムを短縮するという目標はありませんが、なぜ反復の終了前にタスクをリリースするのですか？ Kanbanはリリースにタスクをプッシュしており、Kanbanスタイルのスタンドアップが支援しています。
  • 多くの作業！=多くの結果。 スクラム会議では、チームは昨日どの程度行われたかを話し合うことができます。 しかし、インパクトワークは、チームが多くのビジネス結果を生み出したことを意味しますか？ おそらくボトルネックの後、多くのボトルネックが形成されたのでしょうか？ タスクフローは重要であり、監視する必要があります。




おわりに



かんばんスタイルのスタンドアップへの切り替えは簡単です。 プロジェクトの作業のどの瞬間からでも使用できます。 現在のプロセスは関係ありません：スクラム、かんばん、スクラムバンなど、プロジェクトの現在の状況を把握して視覚化できる場合は、説明されているスタンドアップの方法が適しています。
