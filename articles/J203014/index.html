Android NDKの概要

Android OS用のアプリケーションを開発するために、GoogleはSDKとNDKの2つの開発パッケージを提供しています。 SDKについては、多くの記事、書籍、およびGoogleの優れたガイドラインがあります。 しかし、Google自体でさえ、NDKについてあまり書いていません。 そして、スタンディングブックから、 Cinar O.-NDK-2012を搭載したPro Android C ++を 1つだけ選択します。



この記事は、Android NDKにまだ精通していない（またはあまり精通していない）人々を対象としており、知識を強化したいと考えています。 このインターフェースから始める必要があるように思えるので、私はJNIに注意を払います。 また、最後に、ファイルの書き込みと読み取りのための2つの関数を使用した小さな例を検討します。 多くのテキストが好きではない人は、彼はビデオ版を見ることができます。



Android NDKとは何ですか？



Android NDK （ネイティブ開発キット）は、C / C ++などの言語を使用してアプリケーションの一部を実装できるツールのセットです。



NDKは何に使用されますか？



Googleでは、まれにしかNDKを使用しないことを推奨しています。 多くの場合、これらはケースです：

  • 生産性を高める必要があります（たとえば、大量のデータを並べ替える）。
  • サードパーティのライブラリを使用します。 たとえば、すでに多くの言語がC / C ++言語で記述されており、既存の素材を使用するだけで済みます。 Ffmpeg、OpenCVなどのライブラリの例。
  • 低レベルのプログラミング（たとえば、Dalvikを超えるすべて）;




JNIとは何ですか？



Java Native Interface -Java仮想マシンを実行するコードを実行するための標準メカニズム。C/ C ++またはアセンブラーで記述され、動的ライブラリの形式で配置されているため、静的リンクが不要です。 これにより、JavaプログラムからC / C ++関数を呼び出すことができ、その逆も可能です。



JNIの利点



アナログ（Netscape Java Runtime InterfaceまたはMicrosoftのRaw Native InterfaceおよびCOM / Java Interface）に対する主な利点は、JNIが元々、JNIで記述されたアプリケーションの互換性のために、特定のプラットフォーム上のJava仮想マシンに対してバイナリ互換性を提供するように設計されたことですJNIについて話しているのですが、Dalvikマシンには接続しません。JNIは、すべてのJava仮想マシンに適したJVM用にOracleによって作成されたためです。 したがって、コンパイルされたC / C ++コードは、プラットフォームに関係なく実行されます。 以前のバージョンでは、バイナリ互換性が許可されていませんでした。



バイナリ互換性またはバイナリ互換性は、プログラムが実行可能ファイルを変更せずに異なる環境で動作できるようにするプログラム互換性の一種です。



JNIの仕組み





C ++の仮想関数のテーブルとして編成されたJNIテーブル。 VMは、このようなテーブルをいくつか使用できます。 たとえば、1つはデバッグ用、2つ目は使用用です。 JNIインターフェースへのポインターは、現在のスレッドでのみ有効です。 これは、ポインターがストリーム間を移動できないことを意味します。 ただし、ネイティブメソッドは異なるスレッドから呼び出すことができます。 例： 



jdouble Java_pkg_Cls_f__ILjava_lang_String_2 (JNIEnv *env, jobject obj, jint i, jstring s) { const char *str = (*env)->GetStringUTFChars(env, s, 0); (*env)->ReleaseStringUTFChars(env, s, str); return 10; }


  • * env-インターフェイスへのポインター。
  • obj-ネイティブメソッドが記述されているオブジェクトへの参照。
  • iとsには引数が渡​​されます。


プリミティブ型はVMとネイティブコードの間でコピーされ、オブジェクトは参照によって渡されます。 VMは、ネイティブコードに転送されるすべてのリンクを追跡する必要があります。 ネイティブコードへのすべての転送されたリンクは、GCによって解放できません。 ただし、ネイティブコードは、渡されたオブジェクトへの参照が不要になったことをVMに通知する必要があります。



ローカルおよびグローバルリンク



JNIは、リンクを3つのタイプ（ローカル、グローバル、および弱いグローバルリンク）に分割します。 ローカルのものは、メソッドが完了するまで有効です。 JNI関数を返すすべてのJavaオブジェクトはローカルです。 プログラマは、VM自体がすべてのローカルリンクをクリーンアップすることを期待する必要があります。 ローカルリンクは、作成されたストリームでのみ使用できます。 ただし、必要がある場合は、DeleteLocalRefインターフェイスのJNIメソッドを使用してすぐに解放できます。 



 jclass clazz; clazz = (*env)->FindClass(env, "java/lang/String"); //  (*env)->DeleteLocalRef(env, clazz);


グローバルリンクは、明示的に解放されるまで残ります。 グローバルリンクを登録するには、NewGlobalRefメソッドを呼び出します。 グローバルリンクが不要になった場合は、DeleteGlobalRefメソッドを使用して削除できます。 



 jclass localClazz; jclass globalClazz; localClazz = (*env)->FindClass(env, "java/lang/String"); globalClazz = (*env)->NewGlobalRef(env, localClazz); //  (*env)->DeleteLocalRef(env, localClazz);


エラー処理



JNIは、NullPointerException、IllegalArgumentExceptionなどのエラーをチェックしません。 理由：



  • 生産性の低下;
  • Cライブラリのほとんどの機能では、エラーから守ることは非常に困難です。


JNIでは、Java例外を使用できます。 ほとんどのJNI関数は、例外自体ではなくエラーコードを返すため、Javaでコード自体を処理して例外をスローする必要があります。 JNIでは、呼び出された関数のエラーコードを確認し、それらの後にExceptionOccurred（）を呼び出す必要があります。これにより、エラーオブジェクトが返されます。 



 jthrowable ExceptionOccurred(JNIEnv *env);


たとえば、一部のJNI配列アクセス関数はエラーを返しませんが、ArrayIndexOutOfBoundsExceptionまたはArrayStoreExceptionを発生させる場合があります。

変更されたUTF-8



JNIは、変更されたUTF-8エンコーディングを使用して文字列を表します。 次に、JavaはUTF-16を使用します。 UTF-8は、通常の0x00ではなく、0xc0で\ u0000をエンコードするため、Cで主に使用されます。 ゼロ以外のASCII文字のみを含む一連の文字が1バイトのみを使用して表現できるように、変更された文字列はエンコードされます。



JNI関数



JNIインターフェースには、独自のデータセットだけでなく、独自の機能も含まれています。 それらは十数個あるので、それらを検討するには多くの時間がかかります。 公式ドキュメントでそれらについて知ることができます 。



JNI関数の例



学習した資料を学習するための小さな例： 

 #include <jni.h> //... JavaVM *jvm; JNIEnv *env; JavaVMInitArgs vm_args; JavaVMOption* options = new JavaVMOption[1]; options[0].optionString = "-Djava.class.path=/usr/lib/java"; vm_args.version = JNI_VERSION_1_6; vm_args.nOptions = 1; vm_args.options = options; vm_args.ignoreUnrecognized = false; JNI_CreateJavaVM(&jvm, &env, &vm_args); delete options; jclass cls = env->FindClass("Main"); jmethodID mid = env->GetStaticMethodID(cls, "test", "(I)V"); env->CallStaticVoidMethod(cls, mid, 100); jvm->DestroyJavaVM();


行ごとに分析しましょう：

  • JavaVM - JavaVMを作成および破棄できる関数を呼び出すためのインターフェイスを提供します。
  • JNIEnv -JNIのほとんどの機能を提供します。
  • JavaVMInitArgs -JavaVMの引数。
  • JavaVMOption -JavaVMのオプション。


JNI_CreateJavaVM（）メソッドは、JavaVMを初期化し、それへのポインターを返します。 JNI_DestroyJavaVM（）メソッドは、作成されたJavaVMをアンロードします。



ストリーム



Linuxのすべてのスレッドはカーネルによって制御されますが、JavaVM関数AttachCurrentThreadおよびAttachCurrentThreadAsDaemonに接続できます。 ストリームは接続されていませんが、JNIEnvにアクセスできません。 重要なのは、GCが起動した場合でも、AndroidはJNIによって作成されたスレッドを中断しないことです。 ただし、スレッドが完了する前に、DetachCurrentThreadメソッドを呼び出してJavaVMから切断する必要があります。



最初のステップ



プロジェクト構造は次のようになります。



図3からわかるように、すべてのネイティブコードはjniフォルダーにあります。 プロジェクトをビルドすると、ネイティブライブラリが置かれる各プロセッサアーキテクチャのlibsフォルダーに4つのフォルダーが作成されます（フォルダーの数は、選択したアーキテクチャの数によって異なります）。



ネイティブプロジェクトを作成するには、通常のAndroidプロジェクトを作成し、次の手順に従う必要があります。

  • プロジェクトのルートで、ネイティブコードのソースコードを配置するjniフォルダーを作成する必要があります。
  • プロジェクトをビルドするAndroid.mkファイルを作成します。
  • アセンブリの詳細を記述するApplication.mkファイルを作成します。 前提条件ではありませんが、アセンブリを柔軟に構成できます。
  • ビルドプロセスを開始するndk-buildファイルを作成します（これも必須ではありません）。


Android.mk



前述のように、これはネイティブプロジェクトをビルドするためのメイクファイルです。 Android.mkを使用すると、コードをモジュールにグループ化できます。 モジュールは、静的ライブラリ（静的ライブラリ、プロジェクトにのみコピーされるlibsフォルダー）、共有ライブラリ（共有ライブラリ）、スタンドアロンの実行可能ファイルのようにすることができます。



最小構成例： 

 LOCAL_PATH := $(call my-dir) include $(CLEAR_VARS) LOCAL_MODULE := NDKBegining LOCAL_SRC_FILES := ndkBegining.c include $(BUILD_SHARED_LIBRARY)


詳細に検討してください：

  • L OCAL_PATH：= $（call my-dir）-call my-dir関数は、ファイルが呼び出されるフォルダーのパスを返します。
  • include $（CLEAR_VARS） -LOCAL_PATHを除く以前に使用された変数をクリアします。 アセンブリは単一のGNU Makeのコンテキストで行われるため、すべての変数がグローバルであるため、これが必要です。
  • LOCAL_MODULE-出力モジュールの名前。 この例では、出力ライブラリの名前はNDKBeginingに設定されていますが、アセンブリ後、libNDKBeginingという名前のライブラリがlibsフォルダーに作成されます。 Androidは名前にlib接頭辞を追加しますが、Javaコードでは、接続時に接頭辞なしでライブラリの名前を指定する必要があります（つまり、名前はmakeファイルにインストールされた名前と一致する必要があります）。
  • LOCAL_SRC_FILES-アセンブリを作成するソースファイルのリスト。
  • include $（BUILD_SHARED_LIBRARY） -出力モジュールのタイプを示します。


Android.mkで独自の変数を定義できますが、次の構文を使用しないでください：LOCAL_、PRIVATE_、NDK_、APP_、my-dir。 MY_などの変数を呼び出すことをお勧めします。 例： 

 MY_SOURCE := NDKBegining.c     : $(MY_SOURCE) ,    , : LOCAL_SRC_FILES += $(MY_SOURCE)


Application.mk



このメイクファイルには、アセンブリの柔軟性を高めるのに役立ついくつかの変数が記載されています。

  • APP_OPTIM-リリースまたはデバッグに設定される追加変数。 モジュールのアセンブリ中の最適化に使用されます。 リリースとデバッグの両方をデバッグできますが、デバッグはデバッグに関する詳細情報を提供します。
  • APP_BUILD_SCRIPT-Android.mkへの代替パスを示します。
  • APP_ABIはおそらく最も重要な変数の1つです。 モジュールを組み立てるプロセッサアーキテクチャを示します。 デフォルトは、ARMv5TEアーキテクチャに対応するarmeabiです。 たとえば、ARMv7をサポートするには、armeabi-v7a、IA-32-x86、MIPS-mipsを使用するか、すべてのアーキテクチャをサポートする必要がある場合、値は次のようになります：APP_ABI：= armeabi armeabi-v7a x86 mips ndkバージョン7以降を使用する場合、すべてのアーキテクチャをリストすることはできませんが、APP_ABI：= allを次のように設定します。
  • APP_PLATFORM-プラットフォームターゲット。
  • APP_STL -Androidはランタイムライブラリlibstdc ++を使用するため、切り捨てられ、開発者がすべてのC ++機能を利用できるわけではありません。 ただし、変数APP_STLを使用すると、アセンブリに拡張機能のサポートを含めることができます。
  • NDK_TOOLCHAIN_VERSION -gccコンパイラバージョン（デフォルト4.6）を選択できます。


NDK-BUILDS



Ndk-buildはGNU Makeラッパーです。 4番目のバージョンの後、ndk-buildのフラグが導入されました。

  • clean-生成されたすべてのバイナリファイルをクリーニングします。
  • NDK_DEBUG = 1-デバッグコードを生成します。
  • NDK_LOG = 1-メッセージログを表示します（デバッグに使用）。
  • NDK_HOST_32BIT = 1 -Androidには、ユーティリティの64ビットバージョンをサポートするツールがあります（たとえば、NDK_PATH \ toolchains \ mipsel-linux-android-4.8 \ prebuilt \ windows-x86_64など）。
  • NDK_APPLICATION_MK-Application.mkへのパスが指定されています。


NDKの5番目のバージョンでは、NDK_DEBUGのようなフラグが導入されました。 1に設定されている場合、デバッグバージョンが作成されます。 フラグが設定されていない場合、ndk-buildはデフォルトでandroid：debuggable = "true"属性がAndroidManifest.xmlにあるかどうかを確認します。 バージョン8より上のndkを使用する場合、AndroidManifest.xmlでandroid：debuggable属性を使用することはお勧めしません（Antプラグインを使用するか、ADTプラグインを使用してデバッグバージョンをビルドすると、フラグNDK_DEBUG = 1が自動的に追加されます） 。



デフォルトでは、64ビットバージョンのユーティリティのサポートがインストールされていますが、フラグNDK_HOST_32BIT = 1を設定することで、32のみを強制できます。 ただし、Googleは64ビットユーティリティを使用して大規模なプログラムのパフォーマンスを向上させることを推奨しています。

Javaコードからネイティブメソッドを呼び出す



Javaのネイティブコードを使用するには、最初にJavaクラスでネイティブメソッドを定義する必要があります。 例： 

 native String nativeGetStringFromFile(String path) throws IOException; native void nativeWriteByteArrayToFile(String path, byte[] b) throws IOException;


予約語「native」はメソッドの前に配置する必要があります。 このようにして、コンパイラはこれがJNIへのエントリポイントであることを認識します。 これらのメソッドをC / C ++ファイルに実装する必要があります。 また、nativeXという語でメソッドに名前を付けることをお勧めします。Xはメソッドの実際の名前です。 ただし、これらのメソッドを手動で実装する前に、ヘッダーファイルを生成する必要があります。 これは手動で行うことができますが、jdkにあるjavahユーティリティを使用できます。 しかし、先に進んで、コンソールからは使用しませんが、標準のEclipseツールを使用してこれを行います。



これで実行できます。 bin / classesディレクトリにはヘッダーファイルが含まれます。



次に、これらのファイルをネイティブプロジェクトのjniディレクトリにコピーします。 プロジェクトのコンテキストメニューを呼び出して、Android Tools-Add Native Libraryの項目を選択します。 これにより、jni.h関数を使用できるようになります。 その後、すでにcppファイルを作成し（デフォルトでEclipseが作成する場合もあります）、ヘッダーファイルに既に記述されているメソッドの本体を作成できます。
