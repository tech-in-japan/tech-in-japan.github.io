無料のESXIを備えた無料のバックアップユーティリティ

どういうわけか、私は比較的大量のディスク容量を必要とするいくつかの個人プロジェクトを手に入れました-約2TB。 適切なVPSがなかった（多くのHDDスペースを提供する人はほとんどいなかった）ので、OVHから専用サーバーを取得し、そこに無料のライセンスでESXI 5.5をインストールしました。



しばらくして、プロジェクトの開発に伴い、管理機能（監視、バックアップ）の構成を開始し、ソフトRAID が約束され、ホスティング業者（OVH）がESXIイメージをロールアップするサーバーがRAIDなしであることがわかりました！ それはちょうど2つのディスクです。 ええ、はい、ESXIはソフトRAIDをサポートせず、ハードのみをサポートしていることがわかりました。 不快になりました。 はい、2TBでは不十分でした。 一般的に、ハードウェアRAIDを備えたより大きなサーバーを使用して、そこにESXI 6.0を配置しました。



そして、2つの問題が発生しました。その解決策をここで説明します。



  1. 最小限のオフラインで仮想マシン（一部は約1TB）をあるサーバーから別のサーバーに転送する
  2. 定期的なバックアップを作成する


少なくとも最低支払額のESXIライセンスがあれば、これらのタスクは両方とも簡単に解決できるとすぐに言います。 実際のところ、無料版のESXIの「ネイティブ」バックアップAPIはオフになっています。 したがって、他の方法を見つける必要があります。



有料ライセンスでは、 vCenterを介した移行オプションがあります 。 Veeam Backupの無料バージョンもあります。これにより、仮想マシンを停止することなく、あるシステムから別のシステムにバックアップおよび転送できます。 しかし、無料のESXIライセンスでは、現在のバージョンであるVeeam 9 はまったく機能しません 。



HPのソリューション-VM Explorerには、無料の無料版があります。



VM Explorer 6.2は無料のESXIで動作しますが、次のとおりです。



  • バックアップを作成するとき、ディスクが薄い場合でも、サーバーから完全なイメージサイズがコピーされます。 つまり、仮想マシンのディスクが500GBで、そこに50GBのみが記録されている場合、500GBがコピーされます。 そして-毎回。 増分バックアップモード（ローカルコンピューターのみ）は有料版であり、テストしていません-それがどれほど効果的かはわかっています。
  • 無料のライセンスでは、ローカルドライブにのみバックアップできます。 つまり、別のESXIホストにコピーするには、既に有料ライセンスが必要です。
  • 無料版にはスケジューラはありません。つまり、手動でバックアップを開始する必要があります。


もう1つの人気のあるソリューションは、 ghettoVCBオープンソースプロジェクトですが、使用するのはやや難しいようで、 ドキュメントは少し時代遅れに見えます。 このプロジェクトについてはすでにHabréでここに書いています 。



私は他の多くのオプションがあると確信しています。 経験豊富な管理者のコメントを読むのは面白いでしょう。 私は彼らが必要なライセンスを購入し、心配しないで経験豊富な仕事だと思うが...



ここで言及することができます：



  • Nakivoには、無料版で2 VMの制限があります。
  • Unitrends 、無料版では1TBの制限があります。
  • シンウェア


これらの製品の使用経験がある場合は、コメントを共有してください。



最終的に、2つのツールを使用することにしました。



  • Xsibackupには無料版があります
  • Ovftool


Xsibackup



バージョン4.4より前は、XsibackupはGithubにありましたが、現在（バージョン6.0.7）XsibackupはGithubから削除されています。作成者のWebサイトからインストールする必要があります。



無料版では：



  • 仮想マシンを停止せずにホットバックアップ。 これは、スナップショット（スナップショット）を使用して行われます
  • ESXIでのCronの構成
  • メールレポート
  • バックアップローテーション
  • 別のESXIホストにバックアップします。 無料版では-rsyncを使用し、ESXIでシャープにします。 有料版では、中間スナップショットの作成による増分バックアップ（ OneDiff ）（私にとってはこれはあまり良い解決策ではありません）とNASを使用した重複排除（ XSINAS ）もあります


Xsibackupのインストール手順
英語で同じ指示-33hops.com/blog_xsibackup-quickstart.html



Xsibackupは、バックアップが必要なESXIホストにインストールされます。

ESXIでは、SSHを有効にする必要があります。

著者のウェブサイトに登録-xsibackupをダウンロード-33hops.com/xsibackup-vmware-esxi-backup.html

次の電子メールで、ESXIにインストールするための無料のキーとスクリプトを受け取ります。 

cd /vmfs/volumes/datastore1/xsi-dir 2>/dev/null || mkdir /vmfs/volumes/datastore1/xsi-dir && \ cd /vmfs/volumes/datastore1/xsi-dir && \ esxcli network firewall unload && \ wget http://a.33hops.com/downloads/?key=64cG...secretKey -O xsibackup.zip && \ unzip -o xsibackup.zip || cat xsibackup.zip && echo "" && \ chmod 0700 xsibackup* && \ rm -rf xsibackup.zip && \ esxcli network firewall load


secretKeyには独自のものがあります。

データストアの呼び出し方法が異なる場合は、パスを登録する必要があります。



wgetを見ると、誰かが頭を振って、誰かのソフトウェアをESXIホストに置くことは安全ではないと言うことができます。 ただし、バックアップの場合は、rootにバックアッププログラムのパスワードを渡します。つまり、どのような場合でも誰かを信頼します。 ローカルコピーの場合、Xsibackupは表示およびチェックできるシェルスクリプトのみを使用します...



次に、バックアップを保存するフォルダーをローカルまたは別のサーバーに作成します。 

 mkdir /vmfs/volume1/datastore1/backup


ホスト間でバックアップをコピーする場合、SSHキーを共有します。 

 ./xsibackup --link-srv=[second.esxi.system.ip]


クラウンを介してバックアップを起動したい場合： 

 ./xsibackup --install-cron


すべてがローカルで機能することをテストします。 

 ./xsibackup --backup-point=/vmfs/volumes/datastore1/backup --backup-type=running --mail-from=email.sender@yourdomain.com --mail-to=email.recipient@anotherdomain.com --smtp-srv=smtp.yourserver.com --smtp-port=25 --smtp-usr=username --smtp-pwd=password --test-mode=true


ホスト間の操作をテストするために、以下を変更します。 

 --backup-point="IP-OF-ESXI:22:/vmfs/volumes/datastore1",  22 -  SSH .


SMTPにTLSが必要な場合、-smtp-sec = TLSがサポートされます。

ローカル、つまり、1つのホストですべてが正常に機能します。バックアップは、ESXIネイティブユーティリティvmkfstoolsを使用して行われます。 すべてが高速で、薄型ディスクは薄型のままです。 ハードドライブでは、約60MB / sの速度が得られました



ただし、リモートホストにコピーすると、HP VM Explorerと同じ問題が見つかりました。ディスクが薄く、小さい部分のみが使用されていても、VMのフルサイズがコピーされます。



著者にその理由を尋ねると、rsyncはホスト間のコピーに使用され、シンディスクの未割り当てブロックをスキップするのに十分ではないと書いています。



テストの際、バックアップを繰り返しても、rsyncはコピー時間を実質的に短縮しないことがわかりました-VM全体のサイズが再びネットワークを通過します。



著者は、rsyncではなく独自のユーティリティをファイルすることを計画していると書いています。 彼らは年末までにリリースする予定です。



私の場合、ホスティング事業者は250Mb / s（〜31MB / s）のネットワーク速度を保証していますが、実際には同じデータセンター内の2つのホスト間でバックアップが10-20MB / sで機能しました。 ここで何が問題なのかわかりません-ネットワーク、rsync、その他の速度を低下させますが、プロセスが長すぎます。



更新 ：私は記事を見つけました -彼らのベンチマークによると、問題はSSH（rsyncが動作する上で）を遅くしていることが判明し、NFSによるとそれは速くなるでしょう。



その結果、ディスクが薄いままであることを嬉しく思います。



VMの転送およびバックアップのプロセス



ホスト間でVMを転送するプロセスは、次のようになります。



  • まず、Xsibackupを使用してローカルバックアップを作成します。
  • 新しいVMを登録しています-前の手順のコピーです。
  • オプション：次のコマンドを使用して、削除されたファイルからVMを「クリーン」できます。 



     vmkfstools --punchzero VMname.vmdk


    -flatのファイルではなく、メインファイルを指定します。
  • 次に、Ovftool（下記参照）を使用して、別のホストにコピーします。 Ovftoolは、使用済みの部分のみが送信されるようにシンディスクを送信できます。
  • その後、最初のホストのVMがシャットダウンし、新しいホストが起動します。


同様に、ホスティング事業者から自宅へのVMバックアップもまだ見ています。 これを行うために、ESXIは自宅で回転しているため、ovftoolはネットワークを介してのみペイロードを送信できます。



フォーラム は 、既存のデータのみが転送されるように、スパースオプションを使用してファイルをNFSにコピーする方法があるように思われると書いていますが、まだわかりません。



増分バックアップを行う方法が見つかりませんでした。



これはすべてコンソールから手動で行いますが、別のホストに転送して最初のバックアップを作成しますが、時間の経過とともにすべてをクラウンで構成すると思います。 たぶん、クラウンの設定方法に関するいくつかの段落をここに追加します。 元の手順はこちら： 33hops.com/xsibackup-cron-how-to.html



したがって、同じサーバー上に最初のコピーが近くにあり、かなり迅速に回復できます。



2番目のコピーは自宅にあります。つまり、 推奨されているように、物理的に異なる場所にあります。 回復するには、ネットワーク経由でアップロードする必要がありますが、これははるかに遅いです。 しかし、これが必要になる可能性も非常に低いです。



Ovftool



完全な英語のマニュアルはこちらにあり、そこからダウンロードできます。 Ovftoolは任意のコンピューターにインストールでき、そこからハイパーバイザーを制御できます。 また、ESXIホストに直接配置することもできますが、これはサポートされている機能ではありません。



ESXiにOvftoolをインストールする
一般に、プロセスは次のとおりです。最初にOvftoolがLinux x64にインストールされ（Ubuntu 16にインストールされました）、次にファイルがESXIホストに転送されます。



手順は次のとおりです。

  • VMwareに登録し、「Linux 64ビット用VMware OVFツール」をダウンロードします
  • Linuxでダウンロードしたファイルを実行し、結果のファイルをESXIにコピーします。 

     sudo /bin/sh VMware-ovftool-4.1.0-2459827-lin.x86_64.bundle scp -r /usr/lib/vmware-ovftool/ root@esx.com:/vmfs/volumes/datastore1
  • 次に、すでにホスト自体で、1つのファイルを編集する必要があります-/ vmfs / volumes / datastore1 / vmware-ovftool / ovftoolそして＃！/ Bin / bashを＃！/ Bin / shに置き換えます





Ovftoolは、ホットモードでVMをコピーする方法を知りません。つまり、仮想マシンをオフにする必要があります。 したがって-上記のXsibackupの必要性。



Ovftoolのいくつかの機能：



  • 「タスクはユーザーによってキャンセルされました」または「エラー：vim.fault.FileNotFound」というエラーがポップアップ表示されていました。 両方のエラーの理由は、元のマシンのCDROMがターゲットホストにないISOを指していたためです。 エラーのテキストからこれを推測してみてください... CDROMデバイスを取り外すことで決定されました（OSのインストールにのみ使用されました）。
  • 私自身はテストしていませんが、ovftoolはスナップショットを保存しないようです。


ESXiで実行する場合にのみ、Ovftoolのいくつかの機能を追加します。



  • ovftoolはローカルで実行されますが、ホストへのパスは、ユーザーとパスワードとともに完全に指定する必要があります。
  • コンソールからパスワードを対話形式で読み取ることはできませんが、コマンドラインでパラメーターとして渡す必要があります。
  • パスワードには、文字、数字、および「-」のみを使用できます。 `＃`のような他の文字を使用しようとしたとき-パスワードは受け入れられませんでした。


ovftoolの使用例：



  • データストアに登録されているすべての仮想マシンをリストします。 



     ovftool -ds=datastore1 "vi://root:password@esx1.com/"
  • 仮想マシンのコピー（登録する必要があります）： 



     ovftool -ds=datastore1 -dm=thin "vi://root:password@esx1.com/vmName" "vi://root:password@esx2.com/"
