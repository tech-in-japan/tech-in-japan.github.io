アクションプラン



さて、サードパーティのコンパイラをねじ込んでVisual Studioの機能を拡張することにしたので、可能な限り拡張しましょう。 これを行うには、やりたいことのリストを定義します。



  1. Visual Studioでプロジェクトのプロパティ（プロジェクトプロパティ）を表示する場合、独自のプロパティがあります。
  2. プロジェクトに含めることができるファイル拡張子のセットを定義します。
  3. コンパイラ、リンカなどが呼び出されるようにビルドシステムを決定するのは自然です。
  4. 独自のプロジェクトを作成して、スタジオが独自の拡張子を持つファイルを開くことができるようにします。
  5. このタイプのプロジェクトを生成するウィザードを作成します。




さて、このリストの要件の少なくとも一部を実装してみましょう。

そのため、 Ruleタグにはいくつかの重要な属性があります。

  • Name = "ConfigurationGeneral" -スタジオは、このName属性値を持つルールを探しています。 ConfigurationGeneralという名前のルールがない場合、ほとんどの場合失敗します。
  • DisplayName = "General" -これはプロジェクト構成ウィンドウに表示される名前です。好きな名前を付けることができます。 私は去りました。「一般」。
  • 説明=「全般」 -説明。 説明する必要がないことを願っています。
  • SwitchPrefix = "-" -この属性は、コマンドキープレフィックスをアセンブリシステムに転送するときに定義します。 コンパイラキーが「/」で始まると仮定します（例：/ Ipathは接続するファイルのパスを定義します）-この属性の値は「/」になります。XC8コンパイラでは、すべてのキーは「-」で始まります。 異なる形式のキーがある場合、この属性は空白のままにするか、まったく指定しないことができます。
  • xmlns-ドキュメント全体の動作の属性、名前空間など。それに応じて変更することは意味がありません。
  • PageTemplate-プロジェクト設定でプロパティページの表示を定義します。genericicとツールのテンプレートを使用します -スクリーンショットでそれらの違いをすべて表示します（次の部分）。


Rule.Categoriesタグは、プロパティページ内のカテゴリを定義するために使用されます。 次の2つのカテゴリがあります。
