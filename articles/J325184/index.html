人の役目を果たすDevOps

テクノロジーのステップバイステップは私たちの仕事を引き受け、最も価値のあるスタックオーバーフローからのCtr-CとネイティブIDEのCtrl-Vに到達すると脅します。 しかし、幸いなことに、あなたよりも優れたプログラミングを学んだ単一のニューラルネットワークはまだありません。 今日は、DevOpsを使用して、モバイル開発者のチーム全体、さらには個人のルーチンを取り除く方法について説明します。











注： Hacker誌の記事の完全版の出版物シリーズを継続します。 著者のスペルと句読点が保存されました。







グローバルに、DevOpsは合理的に自動化されたすべての自動化の最大化の問題を解決します。 目標は単純です-製品のリリースまたは更新の時間とコストを削減することです。 これは、「クライアントへの価値の継続的な供給」と時々言われるように、労働生産性と組織の増加の一種の別のラウンドです。









現時点では、アプリケーションの新しいバージョンのアセンブリ、配布、テストは十分に自動化されています。 この記事では、 Bitrise.ioビルドサービス 、 Xamarin Test Cloudデバイスファーム、およびHockeyAppフィードバックシステムで構成される、ユニバーサルモバイルDevOpsパイプラインを構築するプロセスを示します。 テストの対象として、Javaの一連の標準サンプルからAndroidアプリケーションを使用します。









オンラインサービスの現代世界の注目すべき特徴は、コードを高品質の製品に処理するための非常に便利なパイプラインを構築できるオープンAPIの存在です。







DevOpsチェーンが動作を開始するには、次に構成する必要があります。 まず、一般的な用語でパイプライン操作アルゴリズムを検討します。







  1. 外部イベント（タイマー、リポジトリ内のcommit / pull-request /タグ、Slackからのコマンド、サイト上のボタンなど）がチェーン全体を活性化します。
  2. ビルドシステムは、設定を考慮して、リポジトリから最後または指定されたコミット/ブランチをダウンロードし、ビルド環境を起動して、コンパイル/パッケージ化されたアプリケーションを発行します。 これらの目的のために、Bitriseは既に構成された環境で個別の仮想マシンを展開し、各ビルドの前に、必要な依存関係をWebまたはキャッシュ（Gradle、CocoaPods、NuGet）からダウンロードします。 コンパイル中の基本的なコード検証に加えて、ユニットテストも実行できます。これは、長くて重いプロジェクトのモバイルに関連しています。
  3. 次に、アプリケーションがファイルストレージまたは外部サービスにアップロードされます。 Bitriseを使用すると、サーバー上でテストビルドをホストでき、携帯電話にダウンロードしてインストールするためのダイレクトページが提供されます。
  4. また、自動UIテストのために、アセンブリをクラウドデバイスファームにアップロードできます。


コンベアを作成する



念のため、ソースコードを保存するためのリポジトリから始めます-あるべきであり、それがポイントです。 Bitbucket 、 GitHub、またはその他を使用できますが、より既製の統合のため、人気のあるサービスが望ましいです。 また、ほとんどすべてのIDEでサポートされているため、Gitを使用することをお勧めします。 プロジェクト（この例では、これはAndroid SDKのクラシックナビゲーションドロワーです）が既にBitbucketにロードされており、BitbucketはBitriseに接続されていると考えています。











集める



DevOpsチェーンの開始要素はビルドシステムです。 ここでは、管理者のタンバリンの味と所有権のレベルは友達ではありません。 しかし、Bitriseは最も単純で最も機能的なオプションの1つです。 安価で、時計のように機能し、すべての一般的なモバイルアプリケーション開発スタックでプロジェクトを完全にビルドできます：iOS + Obj-C / Swift、Android + Java、Xamarin（iOS + Android）、Web App（PhoneGap、React Nativeなど） 。 そして最も重要なことは、すぐに使用できるBitriseは、多数の外部システムとの統合を構成できることです。







開始するには、Bitriseで無料試用版を登録し、その中に新しいAndroidプロジェクトを作成し、リポジトリを添付します。 キーストア（およびiPhone用に開発している場合はiOS証明書）を追加して、アプリケーションに署名することをお勧めします。 知り合いや研究のために、作成したワークフローを確認することをお勧めします。











次に、開始ボタンをクリックし、しばらくして完成したアセンブリを取得します。 ログに飛び込んで、プロセス全体の詳細を調べることができます。











Bitriseには、一般的な外部サービス（お気に入りのSlackを含む）を使用できる多数の追加手順があります。 必要に応じて、独自のスクリプトを作成することもできます（たとえば、パッケージの名前を変更するため）。 同時に、インフラストラクチャとその構成とサポートに手間がかかりません！ もちろん、ビルド時間は優れたプログラマーマシンよりも少し長くなりますが、一方で、別のコーヒーブレークの代わりに並行して作業を続けることができます。 必要に応じて、Bitriseはプロジェクトのユニットテストを実行することもでき、これもすべての一般的なプラットフォームとスタックをサポートしています。







数分後（アプリケーションの複雑さに応じて数十分）、アセンブリの準備が整い、インストールパッケージをスマートフォンにダウンロードするためのページが表示されます。 リンクはすべての関係者と共有できます。











テスト中



モバイル開発で最も問題の多い場所の1つは、OS、画面サイズ、価格、ケースの色だけでなく、OS自体のバージョン、Androidでも異なるモデルのファームウェアバージョンによる強力な断片化です。 まともなモバイルスタジオでは、最近まで、数十（または数百）のさまざまなデバイスを保持し、フリートとファームウェアの更新を常に監視する必要がありました。 同じアプリケーションは、顕著な更新のたびに、または公開前に、一見同じように見える多数のデバイスでクロスチェックされました。 画面上に鞭と珍しい指のガラガラとドローバーを追加するために残っています...







幸いなことに、AppleとGoogleはこの側面を通過せず、iOSとAndroidの自動化のためのAPIを提供しました。 つまり、実際のデバイスでOSとやり取りし、テスターが指定したスクリプトを実行することでユーザーの作業をシミュレートできます。







さまざまな企業が、モバイルソリューションのテスト用に実際のデバイスのファームを提供しています。 普遍的で便利なサービスXamarin Test Cloudに焦点を当てます。 しかし、始める前に、コマンドラインジャグリングとカラバッシュのスピリットを実行する必要があります。







Ruby、Ruby DevKit、追加のgemをインストールし、 Cucumber用の最初のスクリプトを作成する必要があります。 幸いなことに、すべてのGoogleが既製の段階的な手順を表示するため、次のGemのみが必要であることに注意してください。







  • ひょうたんきゅうり;
  • ひょうたんアンドロイド;
  • xamarin-test-cloud。

はい、スクリプトはコンピューターへの指示ではなく、人への指示に似ています。 これは、プログラミングの方法を実際に知らないテスターに​​とって便利です。 ただし、コントロールの種類（画像、ボタン）や画面座標の表示など、より複雑な構文を使用することもできます。 スクリプトの数に制限はなく、スクリプトは連続して実行され、その間でアプリケーションが完全に再起動されます。







インストールパッケージとスクリプトを外部クラウドに送信する前に、ローカルのスマートフォンまたはエミュレーター/シミュレーターで書かれたテストを実行することをお勧めします（たとえば、calabash-android runコマンドを使用）。







Xamarin Test Cloudを使用するには、トライアルを登録し、最初の実行（テスト実行）の作成に進みます。











それから、デバイスグループの識別子（--devicesキーの後に示される）とAPIキー（APKファイルの名前の後に示される）が必要になります。これはBitriseで引き続き使用され、パイプラインに新しいステップが追加されます。









統合のセットアップが正常に完了したら、スクリプトを実行し、数分でその作業の結果を確認できます。















Xamarin Test Cloudは、テスト自体の実行に加えて、すべてのシステムログ、アプリケーションによるリソース消費、さらにはスクリーンショットを収集します。 テストが完了すると、詳細レポートが利用可能になり、メールで通知を受け取ります。







Xamarin Test Cloudは、実際のテスターに​​代わるものではありません。一部のエラーは、アプリケーションを手動で長期間使用した場合にのみ検出できるためです。 ただし、クラウドファームはテスト時間を大幅に短縮し、すべてのユーザーシナリオの基本的なヘルスチェックを高速化できます。 また、スクリプトの実行中に作成されたスクリーンショットは、さまざまな解像度とOSバージョン（ボタンが浮いている、テキストが切れているなど）でユーザーインターフェイスエラーを見つけるのに役立ちます。







大規模プロジェクトの典型的なシナリオ-パイプラインは夜間または手動で開始され、必要なスマートフォン/タブレットでの現在のアセンブリの自動UIテスト（ファンの場合はユニットテストを追加できます）、数時間後（または午前中）に品質管理チームが含まれます特定の選択されたスマートフォンでの実行結果に関する詳細レポートが表示されます。 ただし、比較的少数のデバイス（約2500）があり、多くのデバイスが必要であることを理解する必要があります。 したがって、1日あたりのサブスクリプションごとに利用可能なデバイス時間数には一定の制限があります。 特に試用版の場合は、共有できる必要があります。







分析する



コンベアの最後の要素は、フィードバックサービス（およびパートタイム配信システム）です。 現在、クラッシュ、統計、およびユーザーレビューを収集するための多くのライブラリがあります。 HockeyAppサービスに焦点を当てます。







開始するには、ライブラリをプロジェクトに追加し、モバイルアプリケーションでHockeyAppを初期化します。 これで、1回のクラッシュで問題が解決しないことを確認できます。











HockeyAppでのアンロードは、UIテストの結果を分析した後にユーザーが起動する別のスクリプトとして行うのが最適です。 外部のフリーランサーを簡単に接続して、HockeyAppから直接アプリケーションをテストすることもできます。







使用する



DevOpsコンベアを使用すると、非常に多くのデバイスをカバーし、さまざまなメーカーの実際のスマートフォンやタブレットでのユーザーシナリオの組み立てとテストを自動化できます。 最近、ユニバーサル統合ソリューション（Microsoft Mobile Centerなど）が登場し始めており、選択したモバイルスタックとプラットフォームに関係なく、自動化の利点を活用できます。







おわりに



この記事では、Bitrise、Xamarin Test Cloud、およびHockeyAppに基づいたパイプラインの作成について検討しましたが、他のツールを選択したり、独自のインフラストラクチャを展開することもできます。 しかし、ここでは、管理者のタンバリン、時にはひげのあるセーターなしではできません。
