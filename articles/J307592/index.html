前のパート（ パート1およびパート2 ）では、Big Switch Networksのデータセンターネットワーク向けBig Cloud Fabric SDNソリューションのアーキテクチャと機能について説明しました。 最後の記事では、関連するシステムと統合する工場の能力を検討し、レビューの前の部分の簡単な要約を提供します。



Big Cloud Fabric統合



現代のデータセンターネットワークは、他のシステムとは別個に存在することはできません-物理世界と仮想世界が密接に絡み合い、それらの境界があいまいになりすぎています。 仮想化およびオーケストレーションシステムとの統合は、BCFの最も重要な利点の1つです。 同様の機能が多くのメーカーの決定で発表されているという事実にもかかわらず、集中化されたBCFアーキテクチャにより、それらを迅速、簡単、便利に実装できます。



現在、BCFはVMware vSphere、Docker / Kubernetes、およびOpenStackとの緊密な統合と連携しています。 後者の場合、統合の機会は業界で最高のものと見なされます。 Open vSwitchのエージェントはKVMコンピューティングノードにインストールされます。これは工場と完全に同期され、スイッチングだけでなく、コンピューティングノード上の仮想マシン間のルーティングも提供します。 Horizo​​nインターフェースでのプロジェクトの作成と管理は、BCFコントローラーと完全に同期されます。実際、OpenStack Project（BCF Tenant）内では、工場のネットワークインフラストラクチャを管理するためのすべてのアクションをHorizo​​nから実行できます。



vSphereとの統合はvCenterを介して行われますが、ファクトリはvCenterの複数のコピーに関連付けることができ、それぞれがBCFで独自のテナントを管理します。 さらに、ファクトリリソースは、vCenterとOpenStackの複数のコピー間で共有できます。 vCenterとの統合は、次の機能を提供します。



  • 接続されたESXiホストとLAGグループの組織の自動調査。


  • 接続されたESXiに展開された仮想マシンアドレス（MACアドレス、オプションのIPv4）の自動調査。


  • ファクトリ内のセグメントの自動作成と、ポートグループを作成するときのそれらのメンバーシップルール。


  • ESXiホストを使用して、必要なVLANをLAGに自動的に追加します。


  • vMotionの実行中に、特定のセグメントの現在のESXiに仮想マシンが残っていない場合、 インターフェイスグループ （ESXiが接続されている）はセグメントから削除されます。


したがって、ほとんどの日常業務は自動化されており、ネットワーク専門家の介入を必要としません。 ただし、BCFとvSphereの統合オプションはこれに限定されません。 Big Switch Networksは、vCenter用のプラグインを開発し、仮想化チームメンバーの機能をさらに強化し、追加の構成および障害分析ツールを提供しています。 これにより、vCenter管理者は、仮想マシンが配置されているサブネットとセグメントのL3インターフェイスを作成し、VLAN、エンドホスト、サーバー、およびポリシー（ACL、PBR）に関する情報を表示できます。 しかし、最も重要なことは、プラグインにはBCFコントローラーインターフェイスと同じテストパスツールが用意されていることです。



仮想環境管理者は、トラブルシューティングプロセスのほとんどの段階でネットワークエンジニアを煩わせる必要がなくなりました。ソース仮想マシンとターゲットマシン（または外部ホスト）を選択し、工場を通過するトラフィックの完全なテストを行うだけです。



vSphereおよびvCenterプラグインとの統合は、仮想インフラストラクチャ管理者にとってあまりにも多くの機会を提供しているように見えるかもしれませんが、彼らは準備ができていないかもしれません。 しかし、これはそうではありません。 上記のツールは、セットアップとトラブルシューティングのプロセスを容易にし、スピードアップするためにのみ設計されています-チームにとって、不必要にお互いの注意をそらし、簡単なクエリと日常的な操作の実行に対する反応を待ってください。 仮想環境の管理者は、工場で何かを「壊す」ことはできません。その機能は、vCenterが同期されるテナントで終わりますが、同時に日常のタスクを解決するのに十分なツールを備えています。 工場の全般的な管理は依然としてネットワークエンジニアが行いますが、彼の参加は、重大なインシデントの解決またはシステムのグローバルな変更の場合にのみ必要です。



おわりに



ソフトウェア定義ネットワークの概念は、おそらくネットワーキングの世界で最も話題になっているトピックです。 単一のマーケティング資料がそれを迂回するのではなく、各メーカーは何らかの形で彼らの決定における支持を宣言することを求めます。 しかし、継続的な開発と多くの重要な変更にもかかわらず、この概念は長い間商業的に完成した製品の形で具体化されていません。 おそらくその理由は、通信ネットワークのフォールトトレランスの基盤の1つが分散化であり、一方、SDNコンセプトの実装は、単一の制御と制御（制御と管理プレーン）なしではほとんど不可能だからです。



しかし、SDNがもたらす利点、顧客の関心、およびソフトウェアメーカーが絶えず努力を続けているため、ネットワーク市場の巨人は視点を考え直さざるを得ません。 その結果、高度な制御システムは、異なる名前ではありますが、古典的なネットワークでSDN機能をエミュレートするという1つの目標によって統一され、市場にあふれています。 残念ながら、このアプローチはSDNが提供するすべての利点をもたらすことはできず、最も重要なことは、ITのネットワーク世界の長年の問題-アプリケーション、オペレーティングシステム、ハードウェアの分離を解決しないことです。 商用ASICの開発とネットワークオペレーティングシステムの開発者の市場への登場により、分解の問題は部分的に解決されましたが、それらから組み立てられた製品は、サードパーティシステムとの統合のヒントのない独立したデバイスのネットワークのままであるか、独立して組み立てられるように招待されたデザイナーです。



データセンターネットワークは、広い地域に分散しておらず、1つのマシンルームのフレームワークを超えることはめったにないため、その中の通信回線の信頼性は、分散およびキャンパスネットワークの信頼性よりもはるかに高くなります。 したがって、データセンターの集中化とフォールトトレランスの問題は深刻な問題ではありません。 さらに、誰もがモジュラースイッチが好きです！ 使用可能なスペースをすべて満たす一定の能力を備えたケーブルがなければ、データセンターネットワークはおそらく2つの非常に大きなスイッチで構成されます。 ファクトリーアウトレットまたは1RUスイッチの大きなスタックを使用する大手メーカーが推奨する設計が良い例です。 明らかに、そのような考慮事項に正しく基づいて、Big Switch Networksはコンセプトを実際に機能する商用製品に変えることができました。



結論として、レビューのドライスクイーズを行い、Big Cloud Fabricの主なポイントをもう一度リストしたいと思います。



  • ネットワーク機器のメーカーへのハードウェアバインディングの欠如（ハードウェアベンダーロックイン）。 工場出荷時のスイッチとして、ASIC Broadcom Trident 2/2 +およびTomahawkに基づいたさまざまなメーカーのスイッチを使用できます。 これは、スイッチのメーカーとサプライヤーのポリシーに適応する必要がなくなったことを意味します。現在、選択肢があります。 これにより、コストが削減されるだけでなく、機器の購入プロセスも高速化されます。特定のモデルの納入を数か月待つ代わりに、現在利用可能なスイッチを選択できます。


  • 耐障害性。 一元化された制御および管理プレーンにもかかわらず、BCFは非常に信頼性があります。 工場コントローラーはフェールセーフクラスターとして実装され、リーフスイッチはMC-LAGペアに結合できます。すべてのコントローラーに障害が発生した場合、工場は調査対象のすべてのサーバーで引き続き動作します。 Big Switch Networksは、工場の耐障害性の証拠の1つとして、いわゆる カオスモンキーテスト-42,000のエンドホスト/ VMの負荷の下にある16のスパイン/ 32のリーフスイッチの工場では、30分以内に640を超える障害が整理されましたが、これはアプリケーションのパフォーマンスにはまったく影響しませんでした。


  • 制御、管理、統合の単一ポイント。 実際、非常に大規模な分散型モジュラースイッチがあります。 各スイッチを個別に構成する必要はなく、ネットワーク内のプロトコル、ループ、フォールトトレランスを心配する必要はありません。BCFは単一のインターフェイスを持つ単一の工場です。 仮想化およびオーケストレーションシステムとの統合により、ほとんどのプロセスを自動化でき、テストパスやVMware vCenterのプラグインなどのツールは、チーム間のトラブルシューティングおよび通信プロセスを大幅に簡素化します。


  • 広範な分析機能。 工場の物理的および論理的構造に関するすべてのデータを収集、相関、および表示するための強力なシステムが工場コントローラーに組み込まれています。 これにより、トラブルシューティングプロセスが大幅に簡素化され、追加のシステムを展開する必要がなくなります。


  • 大幅なスケーラビリティ 。 1つのBCF PODは、最大32個のリーフ/ 6個のスパインスイッチと46'000個の接続されたエンドホストをサポートします。 したがって、予測可能な一定の遅延時間と2：1のオーバーサブスクリプションを備えたネットワークファクトリを取得します。 工場にスイッチを追加する場合、スイッチ以外は必要ありません。ゼロタッチプロビジョニング機能により、接続された機器を設定する必要がなくなります。 したがって、工場のポート容量を96ポートから1,500ポート以上に増やすには、インストールと切り替えの時間だけでなく、構成に10分かかります。


これに加えて、Big Switch Networksは非常に柔軟な製品紹介の機会を提供します。 まず、これは仮想マシンであるCommunity Editionの形式のBCFコントローラーの特別な無料バージョンです。 スイッチの数とタイプの制限（2つの物理スイッチのみ）、およびサポート（ベストエフォート、電子メールのみ）を除き、商用バージョンのすべての機能を備えています。 リモートラボを使用して、機器がなくてもBCF機能を調べることができます。 最も興味深いトピック別に分類されたいくつかのトレーニングモジュールがあり、その中で頻繁に使用する機能やスクリプトを実行したり、コントローラーインターフェイスに慣れたりすることができます。



一般的に、Big Cloud Fabricファクトリーでのテストと作業から、非常に満足のいく印象を受けました。また、利便性、シンプルさ、幅広い機能性により、他の大規模システムとの激しい競争の状況でもデータセンターネットワークソリューションの市場で需要があると確信しています。
