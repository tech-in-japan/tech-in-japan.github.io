Asp.netでバックグラウンドプロセスを開始する方法

ASP.NETでバックグラウンドプロセスを開始する必要がありました。 疑問が生じました：これをどのように行うのが最善ですか？ SCOTT HANSELMANブログを少し調べてみると 、「 ASP.NETでバックグラウンドタスクを実行する方法 」というエントリが見つかりました。 この記事はそれほど新しいものではありません-2014年ですが、かなり関連性があるので、ロシア語に翻訳することにしました。



数年前、Phil HaackはASP.NETでバックグラウンドタスクを行うことの危険性に関する優れた記事を書きました 。 彼は、バックグラウンドプロセスの開始に関連する3つの主なリスクを特定しました。



  1. 要求に関連付けられていないスレッドの未処理の例外により、プロセスが削除される場合があります。
  2. Webファームでサイトを実行すると、同じタスクの複数のインスタンスを同時に実行する可能性のあるアプリケーションの複数のインスタンスを誤って終了する可能性があります。
  3. サイトが実行されているアプリケーションドメインは、さまざまな理由で、サイトで実行されているバックグラウンドタスクをアンロードおよび削除できます。


もちろん、バックグラウンドタスクを管理する独自のマネージャーを作成できます。 しかし、ほとんどの場合、あなたは間違っています。 開発者のスキルに挑戦する人はいません。 そのようなマネージャーを作成することは、かなり微妙なことです。 そして、なぜそれが必要なのですか？



バックグラウンドでタスクを実行するための多くの素晴らしいオプションがあります。 そして、抽象的な技術だけでなく、既製のライブラリもあります。



ASP.NETアプリケーションには、IISの下の独自のサーバーで実行できるものと、Azureでホストできるものがあります。



バックグラウンドタスクを開始するには、いくつかのオプションを選択できます。



  • 基本：独自のASP.NETサイトでバックグラウンドタスクを作成するために使用できるHangfireまたは同様のライブラリ。
  • クラウド： Azure WebJobs Azure Web Jobs。 Webサイトの外部でバックグラウンドタスクを実行し、負荷を調整できるAzureのネイティブサービス。
  • 高度： クラウドサービスのAzure Webロール 。 サイトに関係なくバックグラウンドタスクを実行できるサービス。負荷を調整し、実行を制御できます。


Azureタスクを使用する方法と、AzureクラウドサービスでWebロールがどのように機能するかについて、Web上に多数の記事とビデオがあります。 ただし、自分のサーバーでバックグラウンドタスクを実行する方法に関する記事はあまりありません。



WEBBACKGROUNDER



サイトが言うように：「WebBackgrounderは、ASP.NET Webアプリケーションでのみ動作するWebファーム互換のバックグラウンドタスクマネージャーの概念のテストです。」 コードは長年変更されていませんが、 NuGetパッケージは 50万回以上ダウンロードされています。



このプロジェクトを使用すると、1つのタスクのみで作業し、Webアプリケーションの実行中にバックグラウンドで繰り返しタスクを管理できます。



このライブラリは明らかにすべての場合に適しているわけではありません。 ただし、ASP.NETアプリケーションの実行中に1つのタスクのみを実行する必要がある場合は、WebBackgrounderで十分です。 



using System; using System.Threading; using System.Threading.Tasks; namespace WebBackgrounder.DemoWeb { public class SampleJob : Job { public SampleJob(TimeSpan interval, TimeSpan timeout) : base("Sample Job", interval, timeout) { } public override Task Execute() { return new Task(() => Thread.Sleep(3000)); } } }


.NET 4.5.2 QUEUEBACKGROUNDWORKITEMで追加



QueueBackgroundWorkItemは、WebBackgrounderの出現に応じて.NET 4.5.2に追加されたと言えます。 これはTask.Runの単なる実装ではありません。 これはもっと何かです：



QBWIは、リクエストに関係なく、バックグラウンドで実行されるスケジュールされたタスクを管理します。 通常のThreadPool要素との違いは、ASP.NETが現在APIを使用して登録されている作業項目の数を自動的に追跡し、アプリケーションドメインがオフになっている場合、ASP.NETランタイムがそれを延期しようとし、実行中のバックグラウンドをシャットダウンできることですタスク。



ASP.NETランタイムは、アプリケーションドメインのシャットダウンを90秒だけ遅らせることができ、タスクを完了することができることに注意してください。 したがって、この時間内にタスクを完了できない場合は、他のより信頼性の高い手段を使用する必要があります。



APIは非常に単純です-「Func <CancellationToken、Task>」を使用します。 以下は、MVCからバックグラウンドプロセスを開始する小さな例です。 



 public ActionResult SendEmail([Bind(Include = "Name,Email")] User user) { if (ModelState.IsValid) { HostingEnvironment.QueueBackgroundWorkItem(ct => SendMailAsync(user.Email)); return RedirectToAction("Index", "Home"); } return View(user); }


FLUENTSCHEDULER



FluentSchedulerは、流advancedなインターフェイスを備えた、より高度で洗練されたタスクマネージャーです。 これにより、タスクを起動するプロセスを実際に制御できます。 



 using FluentScheduler; public class MyRegistry : Registry { public MyRegistry() { //   ITask     Schedule<MyTask>().ToRunNow().AndEvery(2).Seconds(); //       Schedule(() => Console.WriteLine("Timed Task - Will run every day at 9:15pm: " + DateTime.Now)).ToRunEvery(1).Days().At(21, 15); //     –        Schedule(() => { Console.WriteLine("Complex Action Task Starts: " + DateTime.Now); Thread.Sleep(1000); Console.WriteLine("Complex Action Task Ends: " + DateTime.Now); }).ToRunNow().AndEvery(1).Months().OnTheFirst(DayOfWeek.Monday).At(3, 0); } }


FluentSchedulerはIoCもサポートしており、ITaskFactoryインターフェイスを実装するだけで、お気に入りのDependency Injectionライブラリを使用して簡単に接続できます。



FluentSchedulerは.NET Coreで動作することに注意してください。



バックグラウンドタスクで発生する例外を処理するには、JobManagerオブジェクトのJobExceptionイベントを使用できます。 このイベントにより、タスクで発生した例外に関する情報を取得できます。 



 JobManager.JobException += (info) => Log.Fatal("An error just happened with a scheduled job: " + info.Exception);


QUARTZ.NET



Quartz.NETは、.NETに実装されている一般的なJavaフレームワークのバージョンです。 フレームワークは積極的に開発中です。 バックグラウンドタスクを作成するために、QuartzはIJobインターフェイスと唯一のExecuteメソッドを使用します。これは実装する必要があります。 



 using Quartz; using Quartz.Impl; using System; namespace ScheduledTaskExample.ScheduledTasks { public class JobScheduler { public static void Start() { IScheduler scheduler = StdSchedulerFactory.GetDefaultScheduler(); scheduler.Start(); IJobDetail job = JobBuilder.Create<MyJob>().Build(); ITrigger trigger = TriggerBuilder.Create() .WithDailyTimeIntervalSchedule (s => s.WithIntervalInHours(24) .OnEveryDay() .StartingDailyAt(TimeOfDay.HourAndMinuteOfDay(0, 0)) ) .Build(); scheduler.ScheduleJob(job, trigger); } } }


Application_Start内でマネージャーを起動するには、JobScheduler.Start（）を呼び出す必要があります。 開始するには、 Scheduled ActionsおよびQuartz.NET （ Quartz.Netを使用したASP.NETのScheduled Tasksの翻訳）を読むことができます。



このプロジェクトにはかなりまともなドキュメントがあり、使用を開始する前に読む価値があります（少なくとも、 Quartzを使用した開発のチュートリアルをご覧ください ）。



ハンファイヤー



そして、最後のレビューですが、機能性の面では間違いなく、すべてのHangfireの中で最も先進的です。 これは、ASP.NETの非常に高度なバックグラウンドタスクフレームワークです。 オプションで、Redis、SQL Server、SQL Azure、MSMQ、またはRabbitMQを使用して、タスク実行の信頼性を高めることができます。



Hangfireのドキュメントは本当に優れています。 すべてのオープンソースプロジェクトにそのようなドキュメントを持たせたい。



Hangfireの最も素晴らしい機能の1つは組み込みのダッシュボードで、スケジュール、完了したタスク、成功したタスクと失敗したタスクを表示できます。



Hangfireを使用すると、「実行して忘れる」などのタスクを簡単に識別できます。タスクに関する情報はデータベースに保存されます。 



 BackgroundJob.Enqueue(() => Console.WriteLine("Fire-and-forget"));


タスクを遅らせることができます： 



 BackgroundJob.Schedule(() => Console.WriteLine("Delayed"), TimeSpan.FromDays(1));


または、CRONスタイルでタスクを実行します 



 RecurringJob.AddOrUpdate(() => Console.Write("Recurring"), Cron.Daily);


Hangfireでの作業は非常に便利です。 Hangfireには、実際の例に基づいた優れたドキュメントとチュートリアルがあります。



Hangfireは、ASP.NETアプリケーションのバックグラウンドタスクを操作するためのエコシステム全体です。



ライブラリは、オープンソースまたはヌゲットパッケージとして入手できます。



結果（個人的に私から）



自分用のライブラリを選択する一方で、適切な機能が必要になり、一方で、たとえば実行中のタスクに関する情報を保存するためにデータベースを使用したくありません。 そのため、WebBackgrounderやQueueBackgroundWorkItemなどの単純なソリューションも検討しませんでした。



複数のプロセスを開始する必要があることは既に知っています。プロセスは長時間（QueueBackgroundWorkItemでの完了が90秒に制限される）動作する可能性があります。 FluentSchedulerはよさそうですが、もっと欲しかったです。 Hangfireは優れたソリューションですが、タスクキューを格納するにはデータベースをすぐに使用する必要があるようです。 はい、完全に無料ではありません。有料版もあります。



最終的には、これまでのところQuartz.NETを選択しました。かなりまともなドキュメント、すぐに使い始めるのに十分な例、およびニーズの増加に応じて追加できる拡張可能な機能です。



バックグラウンドタスクを起動するための他のライブラリを知っているか、そのような問題を解決した経験がある場合は、コメントを共有してください。
