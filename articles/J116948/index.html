アーキテクチャを網羅的に説明するつもりはありませんが、この記事は、一般的な開発のための、テクノロジーへの遠足です。



したがって、私が言ったように、System PはPowerプロセッサー（RISCアーキテクチャー）に基づいて構築された一連のIBMサーバーです。 主な目的は、ビジネスシステム用のアプリケーションサーバーです。データベースサーバー、企業通信（Lotusなど）、およびWebSphereなどのさまざまなミドルウェア（ミドルウェア）です。

Powerサーバーでは、ネイティブAIX、IBM i、そしてもちろんLinuxの3つのオペレーティングシステムが実行されています。

AIXは、その信頼性、安定性、柔軟性、および管理の容易さで知られる最も古いUNIX'ovの1つです（これは主にSMIT-コンソールシステムコンフィギュレーターによって実現されます。これにより、コマンドラインオプションを気にせずに多くのことができます）。 LinuxのRPMパッケージがインストールされているという事実に、このシステムの研究に衝撃を受けました！ もちろん、そのアーキテクチャの下で、しかしまだ。 ネイティブmcをAIXにインストールすることは難しくありませんでした。ppcアーキテクチャ用のRPMパッケージをダウンロードし、依存関係をダウンロードし、有名なRPM Linuxチームと共にこれをすべてインストールしました。

IBM iは、すべての既知のファミリーにほとんど似ていない別個のオペレーティングシステムです。 私はまだ彼女についてあまり知りませんが、IBM iでAS / 400にサービスを提供している銀行の友人は、彼女をDB2データベース管理システムの最も強力なOSの1つと言っています。

System Pの主要な技術的「チップ」であり、このラインをあらゆる種類から区別します。開発されたハイパーバイザー、「オンデマンドのリソース」、インフラストラクチャを構築および変更する際の柔軟性とモビリティ、エネルギー効率、高可用性を確保するためのクラスタリングのサポート。 もちろん、価格は一見噛み付きますが、最終的には、上記の特性と技術のすべてが初期費用を完全にカバーします。 順番に：



仮想化ツール

これはおそらく最も興味深い部分です。Powerシステムでは、すべてのOSはハイパーバイザーに基づいた仮想環境で動作します。 つまり サーバーには、相互に分離された複数の「ゲスト」を含めることができ、同時に動作してリソースを共有できます。 各論理（仮想サーバー）はLPAR（Logical Partition Access Resources）と呼ばれ、オペレーティングシステムがインストールされます。 サーバー上のLPARの最大数は、プロセッサーとメモリの数に依存します。プロセッサーごとに最大10個のLPARを作成し、各LPARに最小共有0.1プロセッサーリソースと最小256 MBのRAMを割り当てることができます。 理論上の制限もあります。795モデル（行の最後）では、最大254のLPARを作成できます。 したがって、プロセッサプールを仮想サーバーに分割することにより、サーバーの負荷を最適化できます。 何らかの種類のLPAR 0.5プロセッサを搭載し、より多くの負荷をかけます-1.2。

各LPARには、ディスクスペースとネットワークカードも割り当てられます。 これは、VIOS（仮想I / Oサーバー）と呼ばれる特別なサーバーによって行われます。 ネットワークカード、SAN、ローカルディスクスペースなどのリソースを統合し、ユーザーの設定に応じてすべてを仮想マシンに提供します。

しかし、最も快適なのはDLPAR（動的論理パーティション）テクノロジーです。これにより、システムを停止することなく、LPARに動的に割り当てられるリソース（プロセッサー共有、メモリー、I / Oインターフェース）の量を動的に変更できます。 つまり パーティションの負荷を調べて、1つが平均してリソースの90％を使用し、もう1つが30％を使用していることがわかった場合、システムを停止せずに2番目のパーティションからリソースを安全に食い止め、これらのリソースを最初に割り当てることができます

すべてのハイパーバイザー管理は、IVM（Integrated Virtualization Manager）またはHMC（ハードウェア管理コンソール）のいずれかのシステムに属します。 IVMはVIOS用のパッケージで提供され、サーバー全体を管理するための便利なWebインターフェースを提供します。 HMCは、すでに複数のサーバーを管理するために必要なLinuxの簡易バージョンを備えた別個のマシンです。 また、便利なWebインターフェイスがあり、すべてのサーバーをここで制御できます。 もちろん、Webインターフェイスはコンソールと交換でき、システムは両方のオプションを提供します。



オンデマンドのリソース

これはおそらくハードウェアそのものではなく、ライセンスの一部です。 これにより、多数のリソースを備えたサーバーを購入し、一部を非アクティブのままにして、それらのコストの一部のみを支払うことができます。その後、サーバーがすでに不足している場合、アクティベーションのためにIBMに支払い、必要なリソースを追加できます。 すべてのリソースを継続的にアクティブ化する必要はない場合があります。しばらくアクティブ化し、使用したリソースに対して毎分支払うことができます。 これにより、ピーク時にリソースを追加することができ、サーバーが比較的アイドル状態のときにリソースを支払う必要はありません。

別のかなり快適なポイント：たとえば、稼働中のプロセッサに障害が発生し、アクティブ化されていないプロセッサがある場合、稼働中のプロセッサはオフになり、非アクティブなプロセッサは追加料金なしでオンになります。



機動性

インフラストラクチャーを構築するときに、PowerVM Enterprise Editionのバージョンを使用できます。 その主な機能：ライブパーティションモビリティ。 このテクノロジーにより、サーバーは、負荷がかかっていてもLPARを切断せずにLPARを相互に転送できます。 もちろん、高速チャネルやSANアレイなどを使用することをお勧めします。そうすれば、すべて高速になり、お客様も気づかないでしょう。 IBMでは、これがどのように起こるかを示しました。ロードされたLPARは、数秒である建物から別の建物に「移動」し、そこで働き始めました。

学生と一緒に仮想ボックスで実験を行いました。仮想マシンを使用して、仮想マシンをLAN経由で別のコンピューターに転送しようとしました。 この機能は新しく、仮想ボックスにのみ表示され、最も興味深いのは機能したことです。 ただし、後で判明したように、仮想マシンがロードされていない場合のみです。 ロードするとすぐに、ハイパーバイザーは10回の反復を実行し、エラーでつまずきました。 もちろん、クラス内の100メガバイトLANおよびSMBリソース形式の共有ストレージは、SANアレイおよびIBMの10ギガビットオプティクスと比較することはできませんが、試してみる価値はありました。



エネルギー効率

Powerサーバーには、電力と冷却の管理システムが組み込まれており、エネルギーの節約とコストの削減に役立ち、ワットあたりのパフォーマンスが向上します。AdvancedPOWER Virtualizationを使用すると、完全に負荷がかかっていないサーバーを統合してエネルギーを浪費できます。 さらに、EnergyScaleテクノロジーは、独自の電源管理機能を提供できます。 これらの機能により、ユーザーはシステムコンポーネントが消費する電力を測定し、電力消費に関してサーバー環境が効率的になるようにポリシーを構成できます。



高可用性

電力ベースのサーバーをクラスターに組み立てて、高可用性とクラスターのマルチプロセッシングを実現できます。 このテクノロジーはPowerHAと呼ばれていました。 PowerHAは、自動障害検出、診断、アプリケーション回復、およびノー​​ド再統合を提供します。 PowerHAは、適切なソフトウェアサポートと共に、並列処理アプリケーションに同時データアクセスを提供できるため、優れたスケーラビリティを提供します。
