CleverStyle Framework 4リリース

どういうわけか、新しいブランチ（ パート1 、 パート2 、 パート3 ）の最初から複数の記事を書くことはできませんでしたが、4.xシリーズの最初のリリースがリリースされたため、ここでも興味深いことがあります。



すべてについて簡単に



最初に言いたいのは、プロジェクトの名前がCleverStyle CMSからCleverStyle Frameworkに変更されたことです。 ついに！ タイトルとコンテンツの間に混乱はなくなります。



2.xシリーズがサーバー側で大幅な変更を開始し、3.xがクライアントで同等の変更を開始した場合、4.xはどこでも改善をもたらします。



これはフルスタックフレームワークであるにもかかわらず、サーバーの速度が大幅に向上しました。ReactPHPに基づくHTTPサーバーを使用すると、1ms未満のページ生成速度を得ることができ、一般的なスクリプトでのHTML生成が高速になりました。

また、翻訳ファイルのネスト構造の便利なサポート、SQLite、PostgreSQLのサポート、PSR7ミドルウェアとしてのサポート（PSR7互換リクエストオブジェクトからの初期化とPSR7互換レスポンスオブジェクトへの結果出力）のサポート、およびこれに関連するカーネルの便利な抽象化を追加しました。



RequireJSはフロントエンドで広く使用されており、フロントエンドで無条件にロードされる多くのライブラリが必要な場合にのみロードされるようになり、最初のフレームの迅速なレンダリングのための最適化が追加されました。



静的キャッシュ（CSS / JS / HTML）、HTTP / 2サーバープッシュ、 Link: <preload>



などを構築するためのバックエンドとフロントエンドの最適化の交差点で。



一般に、システムはフルスタックのPHPフレームワーク（純粋なマイクロカーネルではありませんが、それほど大胆ではありません）のように見えます。



いつものように、Scrutinizerの評価など、コードメトリックにいくつかの改善がありました:)



ポリマーおよびウェブコンポーネント



Webコンポーネントを使用して、前回の記事と同じことから始めることができます。



ポリマー自体にはまだパッチが適用されていますが、目的は異なります。 重要なPRはすべて受け入れられ、受け入れられない2つのPRはランタイムパッチとして配信され、ビルド自体の影響を受けません。 しかし現在、システムで使用されているPolymerアセンブリから、Shady DOMのサポートは完全にカットされています。これは、システムがShadow DOMのフルバージョン（およびサポートのないフルポリフィル）を使用しているためです。 その結果、ビルドは元のビルドよりも大幅に少なくなります。



Polyfill WebComponents.jsも最適化されています。 まず、IE / Edge（言及されたブラウザーにのみ接続されている）のすべての親友が切り取られます。WebComponents.jsはChromium / Chromeにも接続されていません。



Webコンポーネント自体はますます使用されるようになりました。たとえば、モジュールはその使用に合わせてリファクタリングされました。ブログ、コメント、Disqus、そしてシステムは長い間それらをすべてに使用しています。



システムは、HTMLファイルをより適切に縮小する方法も学習しました。現在、すべてのJSが結合され、初期実装のいくつかのエラーが修正されています。



RequireJS（アラメダ）



これは実際には非常に重要な機能です。 最新バージョンでは、RequireJSの代わりに、同じ開発者Alamedaからのよりモダンでコンパクトな（ただし下位互換性のある）アナログが使用されます。



RequireJSはすべてのレベルで統合されています。

  • カーネルで提供されるautosize、html5sortable、jsshaなどのサードパーティライブラリは、AMDモジュールとして実際に排他的に使用され、それ以外の場合は使用されません（jQueryは5.xでも同様にレンダリングされる予定です）
  • components/modules/Blogs/includes/js/xyz.js



    をBlogs/xyz



    モジュールとして透過的にダウンロードできます（サブフォルダーもサポートされています）
  • Bower / bower_components



    および/またはBower / NPMのnode_modules



    がサイトのルートで検出されると、AMDモジュールが自動的に検索されるため、すぐにrequire(['lodash']).then(function (_) {})



    呼び出すことができますrequire(['lodash']).then(function (_) {})



  • ルートBower / NPMフォルダーと同様に、 Composerアセットプラグインを介してインストールされた依存関係は同じ方法で処理され、パスなしで名前で利用可能になります
  • さらに、ルートにインストールされているかのように、絶対パスのComposerアセットを介してインストールされたモジュール、たとえば/node_modules/d3/d3



    、依存関係が実際にインストールされているようにすべてが機能します！




したがって、必然的にダウンロードされるコードの量は大幅に削減され、5.xシリーズではさらに削減され、開発者はさらに多くの可能性を提供します。



IE 10



時間がなくなり、カーネルでは__proto__のみがIE 10との互換性のために使用されず、残りの互換性の負担は5.xシリーズのIE 11をサポートする新しいOld IEプラグインに移動しました。



リクエスト/レスポンス、PSR7



サーバー側について少し説明します。 システムオブジェクトcs\Request



およびcs\Response



追加されました。どちらもPSR7で動作し、必要に応じて再初期化を提供します。

これらすべてにより、フレームワークのミドルウェアとしての使用が大幅に簡素化されました。 これは、ReactPHPとの統合が現在どのように行われているかの例で見ることができます。



再初期化はカーネルレベルで自動的に提供されるため、システムオブジェクトをまったく再作成せず、代わりに必要な部分のみを再初期化することが可能になりました。 これにより、1msよりも高速にページを生成できます。



抽象化のサポートに加えて、マルチパートリクエストのスタンドアロンの完全ストリーミングパーサーが作成されたため、ファイルのアップロードはPOSTだけでなく、あらゆる方法で機能しますenable_post_data_reading = Off



場合、サイトは中断しません（まだReactPHPファイルのアップロードのサポートについては、近い将来、新しいリリースがhttps://github.com/reactphp/http/pull/41の構成でリリースされることを期待しています ）。



SQLite / PostgreSQL



これらの2つのデータベースドライバはカーネルに完全に追加されましたが、どのモジュールでもまだサポートされていません（これは一時的なものです）。 SQLiteは、データベース自体はまったく必要ない単純なサイトに使用できますが、システムでは、単一の管理者の構成と設定を保存する必要があります。 このシナリオでは、SQLiteの速度が低下することはありません。ほとんどの場合、キャッシングによって使用がゼロになるためです。



セッション



セッションは、良い結末を迎えた興味深いストーリーであることが判明しました。 歴史的に、システムはカスタムセッションシステムを使用します。これは、組み込みのものよりも柔軟性があり、拡張性が高く、非常に高速に動作します。



すぐに、ただしセッションがまったくないほどではありません。 その結果、システムの最新バージョンは、ユーザーが実際にセッションが必要なものを使用するまでユーザーのセッションを開始しません。 これにより、最初にサイトにアクセスしたユーザーの標準ページのデータベースでのクエリを回避できます。



翻訳ファイルの名前空間



この機能は長い間懇願します。 最初に、翻訳のあるオブジェクトで接頭辞がサポートされていました。次に、翻訳自体に接頭辞が煩わしくなり始めたため、簡単に行うことができます。







HTTP / 2サーバープッシュ、プリロード



Chromium / Chromeの最新バージョンは、ページにこのリソースが100％必要であることをブラウザに伝えるプリロードヘッダー/メタタグをサポートします。理由がまだ分からなくてもロードする必要があることは確かです。

HTTP / 2サーバープッシュのために、nghttp2とCloudFlareで同じ属性が使用されます。



フレームワークには静的リソースとその依存関係に関する情報が最初にあり、縮小および圧縮中に、リソース内で使用されている画像、フォント、その他のものも知っているため、この情報を使用して対応するヘッダーを生成しますが、これは最も肯定的です画像はページの読み込み時間に影響します。



CLIインターフェース



また、最新のリリースでは、有用なカーネルレベルのCLI機能が導入されました。 これまでのところ、それを介して利用できるものはほとんどありませんが、すぐにさらに多くのものがあります。 その計画はカーネルとコンポーネントの両方で大きいですが、今では何か便利なことをすることができます、またはコンポーネントでそれを使用することができます:)



更新情報



コンポーネントブログ、コメント、Composer、Disqus、Uploader、ComposerアセットおよびTinyMCEは、最新の便利なカーネル機能を使用しており、フレームワークの最新バージョンでコンポーネントを作成する方法の例です。 ほとんどの部分の残りのコンポーネントは、新しいバージョンとの互換性のために単純に移植されたものであり、システムの潜在能力を100％使用することはありませんが、今後のアップデートでは使用されます。



サードパーティの依存関係は、リリース時に最新バージョンに更新されています。
