ストレステスト中にJava VisualVMを使用する

大規模なWEBアプリケーションを開発する場合、遅かれ早かれ、負荷テストを実施するという疑問が生じます。 私たちのプロジェクトでも同様の質問がありました。 この記事から学習できると判断した方法について。



実際、私たちは何をしますか



タスクは次のように設定されました。



アプリケーションは、少なくとも1,500人の同時ユーザーの負荷に耐えることが必要です。これらのユーザーは、次のシナリオに従って動作します。

1.アプリケーションページを開きます

2.システムにログインします

3.データベースからレコードの大きなリストをアンロードする

4.ログアウト



俳優



IBM WebSphere Application Server 7にデプロイされたアプリケーション

ロードツールとしてApache JMeter 2.7が選択されました

Java VisualVMを使用してテスト結果を確認します



すべてを機能させる方法は？



アプリケーションサーバーの構成


最初に、アプリケーションをApplication Server（以降WAS）にデプロイします。

1.ブラウザのアドレスバーに「https：//％hostname％：9043 / ibm / console / logon.jsp」と入力します

2. WAS管理者のログインとパスワードを使用して、AdminConsole自体に入ります。







3. [アプリケーション]-> [アプリケーションタイプ]-> [WebSphereエンタープライズアプリケーション]に移動して、ページに移動します。

エンタープライズアプリケーション

このページを使用して、インストール済みのアプリケーションを管理します。 単一のアプリケーションを複数のサーバーに展開できます。



（大事なインストールボタンを使用して）アプリケーションをインストールすると、コンソールは次のようになります。







赤い長方形は、プロジェクトのメインアプリケーションを示しています。 残りは補助モジュールです。



いいね！ これに対処しました。 次に、Java VisualVMで動作するようにWASを構成する必要があります。 これを行うには、同じAdminConsoleで次の手順を実行する必要があります。

1. サーバー->サーバータイプ-> WebSphereアプリケーションサーバー-> server1-> Javaおよびプロセス管理->プロセス定義-> Java仮想マシン



選択してください：

旗

デバッグモード[false]



デバッグ引数

-Xdebug -Xrunjdwp：transport = dt_socket、address = 52772、suspend = n、server = y、-Dwas.debug.mode = false



汎用JVM引数

-Dclient.encoding.override = UTF-8 -Dfile.encoding = UTF-8 -Djavax.management.builder.initial = -Djava.rmi.server.hostname =％hostname％-Dcom.sun.management.jmxremote -Dcom。 sun.management.jmxremote.port = 8686 -Dcom.sun.management.jmxremote.ssl = false -Dcom.sun.management.jmxremote.authenticate = false

*引数-Djava.rmi.server.hostname =では、％hostname％の代わりに、WASがインストールされているホスト名を指定する必要があります。 ％記号は設定されていません。







Java VisualVMの構成


引数を設定し、WASサービスを強制的に再起動した後、 Java VisualVM自体を構成する必要があります

1. Sun JDK 6u7以降を使用している場合は、すでにVisualVMを使用しています。 Sun JDK 7の最新バージョンを使用することをお勧めします。JavaVisualVMは$ JAVA_HOME \ jdk1.7.0 \ bin \ jvisualvmディレクトリにあります。

また、Java VisualVMは個別にダウンロードできます。

2. Java VisualVMを起動します。 VisualVMは最初の起動時にユーザー名とパスワードを要求する場合がありますが、拒否しないでください。

3. Java VisualVMの一般的なビュー







ご覧のとおり、VisualVM自体を含むいくつかのアプリケーションがローカルで定義されています。



4.「使用可能なプラグイン」タブの「ツール->プラグイン」メニューで、使用可能なプラグインが表示されます。必要に応じて、必要なプラグインを選択してインストールします。 私たちの目的のためには、それらは必要ありませんが、それらについて知る価値はあります。

5.前の設定が実行されたリモートホストを追加します。







6. JMX接続を追加します。 ホスト名の後に必ずポート8686を指定して、[OK]をクリックしてください。







7.リモートホストに接続した後、[JMX接続を開く]を選択すると、目的のプロセスの概要が表示されます。







ここでは、[サンプラー]タブ（別名プロファイラー）に注目します。 プロファイラーには、CPUとメモリの2つのモードがあります。 それらに費やされた時間の観点から最初のプロファイルメソッド、2番目-作成および破壊されたオブジェクトの観点から。 残念ながら、メモリサンプリングはリモートアプリケーションではサポートされていません。

より詳細なビューを表示するには、目的のメソッドを選択し、スナップショットボタンをクリックします。これにより、さらなる分析に使用されるメソッド（クラス/パッケージ）のツリーが呼び出されます。



重要！


このパフォーマンスプロファイラが手段であるという事実に注意を払う価値があります 。 つまり、アプリケーションコードを書き換え、プロファイラーへの呼び出しを追加します。

ツーリングプロファイラーにはかなり大きなオーバーヘッドがありますが、最も正確な結果が得られます。 このようなプロファイラーの動作は、アプリケーションの動作を大きく歪める可能性がありますが、理論上のオーバーヘッドはすべてのメソッドに均等に分散されます。



JMeterでプロジェクトを作成する


JMeter onHabréでプロジェクトを作成する方法はすでに複数回書かれています。さらに、マニュアルをキャンセルした人はいないため、この記事のこの部分では、遭遇したそれらのニュアンスについてのみ説明します。



ローカルホスト


すばらしいマニュアルにあるように、JMeterで作業するには、次のようにJMeter HTTPプロキシを使用するようにブラウザを設定する必要があります。







しかし、ロシアで働いていて、アプリケーションサーバーがドネプロペトロフスクのCRにデプロイされている場合はどうすればよいでしょうか。また、企業のITセキュリティポリシーは、必要なプロキシを使用しないすべての接続をブロックします。 また、企業のプロキシを介して作業している場合、録音コントローラーでの録音が行われないこともわかりました。

R log4j：警告209および列14の連続解析エラー

R log4j：警告「logger」タイプの要素のコンテンツは、「（param *、level ?, Appender-ref *）」と一致する必要があります。

R log4j：警告ロガー（org.springframework.web.context.support.XmlWebApplicationContext）のアペンダーが見つかりませんでした。

R log4j：警告log4jシステムを適切に初期化してください。

R log4j：警告詳細については、 logging.apache.org / log4j / 1.2 / faq.html＃noconfigを参照してください。

仮定を確認します。
