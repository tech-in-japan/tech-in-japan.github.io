会議プログラム

JVM、並行性、分散および高負荷システム、Java 8/9/10、および選択されたフレームワークとツール（Spring、Spark、BPFなど）の基本とパフォーマンスについて説明します。 それまでの間、プログラムに既に含まれているレポートを見てみましょう。



Alibaba、Sanhong Li-Alibaba JDK（AJDK）による極端なスケーリング



過去2年間、私たちのJava会議の参加者は私、参加者、友人、パートナーに近づき、同じ質問をしました。「Alibabaがないのはなぜですか？」 はい、世界最大のプロジェクトの開発者だけでなく、JVMディレクションのリーダーでもあります。 JPointに必要なもの。 IBMの経験があり、OpenJDKオープンソースの貢献者であるSanhong Liに会い、現在、OpenJDKのカスタムバージョンであるAlibaba JDKの開発者をリードしています。



11月11日、アリババは再び独自の帯域幅記録を破り、毎秒325,000トランザクション（昨年より60％増加！）に達しました。そのほとんどは、数十万のサーバー上の数万のJavaサービスで実行されました。 このレポートでは、SanhongがOpenJDKをタスクに合わせてどのようにカスタマイズしたかについて説明します。



  • オブジェクトを割り当てるための独自の非GCメカニズムの開発では、いわゆる「非表示領域」にオブジェクトを作成し、GCオーバーヘッドなしで通常のJavaオブジェクトとしてそれらを操作できます。
  • JVMにcorutinを統合して非同期アルゴリズムを実装することについて。
  • JVMのカスタム効果的なリアルタイム診断の作成について。


さらに、Alibabaのスタッフが、サービスの負荷の分析に頼って、そのような機能が必要であるという結論に至った方法を正確に学習します。 一般に、このレポートは、ワークロードがどれほど高くても有用です。





PivotalのJuergen Hoellerによる2つのレポート



おそらく、Spring Frameworkオープンソースプロジェクトの共同設立者であるJürgenをご存知でしょう。ご存じない場合は、2003年からリリース管理に携わっており、Spring Frameworkコアプロジェクトを管理してきました（14年間、私の頭に収まりません）。 JPoint 2018で、彼は2つのレポートを作成します。


  • Spring Framework 5：機能のハイライトと隠された宝石



    Spring Framework 5.0の主要な機能は、機能的および事後的な開発アプローチのサポートでしたが、新しいバージョンにはまだ目立たない「ねじれのない」部分が多くあります。



    レポートには、主な機能とユルゲンがお気に入りで最も有用であると個人的に特定する機能の両方の詳細なレビューが含まれます。 さらに、今後の5.1リリースの新機能を確認できます。

  • JDK 9上のSpring Framework 5.0



    JPoint 2018で、ユルゲンは多くのJava開発者にとって緊急の問題になるものについて話す予定です。 もちろん、待望のJDK 9への移行についてすぐに考えました。そのため、Jürgenは、Spring Frameworkの5番目のバージョンがEightだけでなくNineでも機能する技術的側面を明らかにします。 この講演では、Jürgenがフレームワーク開発者がJDK 9との互換性を確保する上で直面する課題を紹介します。 さらに、ここでは、JDK 9でSpringプロジェクトを更新するための具体的かつ実用的な推奨事項を紹介し、アップグレードに関連する利点と「犠牲者」の概要を説明します。





Stephane Nicoll、Pivo​​tal-Spring Boot 2.0を使用したリアクティブWebアプリケーション



Spring Bootアプリケーションの新世代はすでにここにあります。バージョン2.0では、Spring WebFluxリアクティブWebフレームワークを使用できます。 すべてがライブコーディングでフレーバー化されるセッションでは、WebFluxアプリケーションがどのように生まれるかを確認し、Boot：Actuator、Developer Toolsなどの機能を最も効果的に使用する方法を理解します。



WebFluxを使用すると、リアクティブWebコンポーネントを記述し、Nettyなどの新しいHTTPランタイムをサポートできるため、Spring MVCのスタイルで昔ながらの方法を記述するか、「WebFlux.fn」と呼ばれる新しい機能的アプローチを試すことができます。





サンダーマック、ルミニステクノロジーズ-Java 9モジュールへの移行



Java 9では、新しいモジュールシステムが導入されています。 使用する必要はありませんが、使用することで多くの可能性が開かれます。 新しいモジュールシステムを使用するために既存のコードを移行するのはどれくらい難しいですか？ 既存のコードベースをクラスパスから任意のモジュラーシステムに転送するのは、大変な作業です。 モジュールをまだ使用しない場合でも、Java 9クラスパスでアプリケーションを実行すると、新たな問題が発生する可能性があります。 このレポートでは、このような問題の解決策を提案します。



移行を容易にするために、Java 9は自動モジュールおよび匿名モジュールを含むいくつかの機能を提供します。 レポートでは、Spring / Hibernateアプリケーションに基づいた実際のコードの移行の例を分析します。 転送中に発生する可能性のある一般的な問題を考慮してください。 モジュールのシステムと転送の機能についての実用的なアドバイスと理解が得られます。 このレポートは、コードの移行を開始するための優れた準備として役立ちます。 一般に、このモジュール化されたコードが穴の中でどのように泳ぐことができるかを確認しましょう。





Rabea Gransberger、MEKOS-コードをJava 9モジュールにリファクタリング



Java 9では、コードをモジュールに分割し、特定のパッケージにのみアクセスできるようにするため、パブリッククラスをpr索好きな目（および手...およびコード！）から隠すことができます。 さらに、コードをモジュールに分割することで、その再利用性を高め、ソリューションのサポートを簡素化し、技術的負債の増加率を減らします。



Rabeaは、既存のモノリシックソリューションを複数のモジュールに分割するいくつかの方法を紹介します。 どのパーツを分離できるかを理解できるツールを選択し、すべてがスムーズに進むように、リファクタリングでいくつかのトリックを使用します。 最後に、いくつかの複雑な例を見て、それらに対処できるパターンを適用しましょう。

ダグラス・ホーキンス、Azul Systems-ReadyNow-Javaのプロファイリングを備えた「AOT」



JITとAOTのコンパイルの違いを説明するのは無意味だと思います。みんなよく知っています。 JITは最適化されていますが、ウォームアップに時間がかかり、AOTコンパイルでは最適化されていない多くのコードにアクセスできません。



講演は、主任開発者がダグラスであるReadyNowテクノロジーについてです。 ReadyNowは、両方のタイプのコンパイラーの利点を組み合わせて、それらの欠点を回避しようとします。 ここでは、ダグラスが伝道者ではなく開発者であることを強調する価値があります。そのため、プレゼンテーション中にテクノロジーの裏側を見て、これがどのように機能するかを理解してみてください。 ReadyNowが（そして何のために）JITのウォームアップを加速し、最適化を回避できるかどうかを確認し、Java AOTコンパイラーが「先天性」疾患を克服する方法を理解してみてください。





Sasha goldshtn Goldshtein、Sela Group-JVMアプリケーション用のLinuxコンテナパフォーマンスツール



「コンテナ革命」はすでに行われています。周りの人は皆、Javaアプリケーションのパッキング、コンテナ化、デプロイを行っています。 ただし、抽象化と同様に、コンテナは喜びと幸福だけでなく、アプリケーションの監視、プロファイリング、トレースに関連する問題ももたらします。 問題は、コンテナはいくつかの分離チップとリソース制限を備えた単なるプロセスであるという事実にもかかわらず、Linuxで正常に動作する多くのパフォーマンスツールをコンテナで非標準で使用する必要があることです。 いくつかは一般的に嘘をつき始め、単に実行しているコンテナではなく、ホストデータに基づいてレポートします。



Sashaのレポートでは、コンテナーに関する最も一般的なパフォーマンスの問題を解決する方法を説明しています。



  • コンテナ内のリソース消費レベルを理解する方法。
  • CPU消費またはCPUリソース制限の問題。
  • ディスクまたはファイルシステムの遅延の問題。
  • データベースとのコンテナ間アクセス。


このため、Sashaには豊富なツールがあります。perf、非同期プロファイラー、フレームグラフ、BCC、その他信頼できるオープンツールと商用ツールです。
