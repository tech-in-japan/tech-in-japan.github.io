CallKitのすべての「喜び」、またはiOS 10での発信者IDの処理方法





2GISは、ディレクトリの会社の電話番号に関する知識をiPhoneのユーザーと共有したいとずっと思っていました。 Androidプラットフォームはそのような機会を提供しましたが、iOSには長い間適切なツールがありませんでした。



6月にWWDC 2016に行きました。セッションの1つで、 Appleの人々は、「ゴージャスな驚き」、つまりiOS 10の発信者IDをようやくできると言いました。 、彼女は多くの制限のある機能を提供しました。

それだけです。 まあ、それはもっと悪かったかもしれません。



このことから、iOS用ダイヤラーは別のプロセスをスピンするアプリケーション拡張機能であり、オーバーロードしてそのステータスを取得できることがわかります。 必要なもののように見えます。

拡張機能自体では、「電話/名前」の形式で番号を追加し、ブロックする番号を追加できます。



最初のプロトタイプは30分で完成しました。 延長に配線された1台の個人用電話、1台のテスト用電話がロックに追加され、すべてが初めて起動しました。喜びに制限はありませんでした。 未来は非常に明るいように見えました-私たちは、これらすべてが翌日の次のリリースにどのように分類されるかをすでに想像していました。



2番目の「喜び」に出会うまで、メインアプリケーションからダイヤラを有効にすることはできません。 ユーザーを設定の奥深くに送る必要がありますが、これは明らかにこの機能の変換を増加させません。



その後、彼らは一連の番号を追加し始め、3番目の「喜び」が現れました：すべての番号は、決定される前にデータベースに記録する必要があります（これは単なる有名なAppleセキュリティです-着信callerIDにアクセスできないようにするため）。 そして、私たちのベースは、署名付きの約400万の数字です。 それは140 MBのテキスト情報、またはブリキ自体を押すと40 MBであり、これらすべてを何らかの方法で拡張機能に配信する必要があります。



この知識を武器に、「電話/名前」の形式でデータを準備し、すでに実際のプロトタイプを見つけ始めました。



データベース



最初に、彼らはすべての数字を愚かに追加することにしました、そして再び驚き-数字はとにかく追加されるべきではありませんが、昇順で：01、02、911など それ以外の場合、拡張機能はドロップします。 最初のベータ8では、xcode拡張機能はエラーなしでクラッシュしました。



さらに、1,999,999個の数字に制限されていることがわかりました。 はい、正確に1,999,999であり、2,000,000ではありません。これは、4,000,000の数値ともまったく同じではありません。 最初に、彼らは3つの拡張を作りたいと思っていました。それぞれが最大1,999,999個の数字で満たされ、口ひげに吹き飛ばされませんでした。 その後、彼らは地域ごとに分割することを決めました：モスクワ+ピーター、ロシアの残り、外国人。 しかし、彼らはこの決定を拒否しました。より複雑な配信を考え出し、機能の安定性をさらに低下させる必要があり、同時に機能するいくつかの拡張機能の作業も安定していなかったからです。 はい。ユーザーに3つの拡張機能をすべて強制的に含めたくありませんでした。 その結果、ユーザーが設定した都市の数のみを残すことにしました。



最初は、SQLiteを介してデータを配信したいと考えていました。 ノボシビルスクから100,000個の単純なデータベースを構築し、データベースを操作するためのロジックを記述し、デモプロジェクトを開始しましたが、何もありませんでした。 エラーはありません。すべて問題ありませんが、数値は決定されていません。



このケースを掘り下げた後、SQLiteからデータを昇順でプルしようとすると、データベースが30 MBのキャッシュを作成し、拡張機能がメモリからクラッシュすることがわかりました。 Appleのフォーラムを掘り下げた後、5 MBのRAMを使用しない方が良いことに気付きました。 その結果、モスクワ、サンクトペテルブルク、およびいくつかの都市のデータベースを組み合わせた場合、データベースへのクエリを大幅に複雑化し、メモリを最適化してフェッチを高速化し、テストプロセスを複雑にする必要があります。 これらすべてを行う時間はまったくありませんでしたが、不本意です。さらに、ニアベーステクノロジーに関する私の能力は明らかに十分ではありませんでした。

データ配信



次に、このデータを拡張機能に、つまり実際には別のアプリケーションで配信する方法を理解する必要がありました。 ユーザーが新しい地域をダウンロードし、古い地域を削除し、すべてを更新したいので、そこに縫い付けることは機能しません。データは古く、新しい地域が追加されますが、私たちは正確さと関連性についての会社です。



すでにすべてが発明されていて、同じ開発者の2つのアプリケーション間でデータを手探りできるApp Groupsという素晴らしいものがありました。



パスに沿ってメインアプリケーションにファイルを配置できます。 



  + (NSString *)extensionDataPath { return [[[NSFileManager defaultManager] containerURLForSecurityApplicationGroupIdentifier:[self extensionGroupName]].path stringByAppendingPathComponent:@"Dialer"]; }


そして、拡張でそれを手に入れます： 



  NSString *databasePath = [[DGSCallKitExtensionModel extensionDataPath] stringByAppendingPathComponent:manifest.databaseName];


配送に問題はありませんでしたが、ありがとうございます。



次に、目的の形式でデータを準備しました。 深くしすぎない場合は、500 MBの.tsvファイルを108の地域に分散させ、バイナリ形式でそれを追い越し、Jenkinsでジョブを作成して作成する必要があります。 要するに、これにもかなりの時間を費やしました-開発全体の約90％。



このデータを電話に配信するタスクが発生しました（開発の2番目の90％）。



最初は「オンデマンドリソース」テクノロジーを使用すると同時に、xcodeの3番目の永遠に空のタブ（リソースタグ）が必要な理由を見つけることにしました。

要するに、私たちにとってのリソースタグは、天国からの単なるマナです（つまり、オンデマンドでのみダウンロード）。 これにより、一部のアプリケーションリソースにタグをタグ付けし、そのタイプを示し、アプリケーションに入力するときにそれらをbinarに含めなくなります。 次に、NSBundleResourceRequestを使用してダウンロードし、[NSBundle mainBundle]から取得できます。 つまり、他のチームをキックする必要はまったくありません。他のチームを保管する方法や、ユーザーに提供する方法を考える必要はありません。 そして、Apple自身がすべてのデータを保存し、それを取得するための非常に適切なAPIを提供します。 ここでも迅速な統合を約束したもの。



しかし、すべてがそれほどバラ色になったわけではありません。最初のリリースでは、このテクノロジーは非常にひどいものであり、ユーザーの約20％が何もダウンロードできませんでした。 Appleのフォーラムを掘り下げてみると、このような問題だけではないことがわかりましたが、長い間修正されず、まったく反応しませんでした。



リソースタグは別の方法でカットして配信する必要がありました。 その結果、データは都市更新データベースに入力されました。 これで、都市の更新とともに、ユーザーは新しいベース番号を受け取ります。

インストールが正常に完了した後、データベースを削除しました。これは、携帯電話のメモリに既にあるものを保存する理由です。 すべてがそれほど単純ではないことが判明しました。ユーザーが設定に入り、拡張機能をオフにしてからオンにすると、単にオンにするのではなく、拡張機能は完全な更新シナリオに従って動作します。 残念なことに、私たちはこれを考慮に入れておらず、これを行ったすべての人は、それらを更新する可能性なしに基盤を失いました。 次のバージョンでは、この問題をすばやく修正し、関連性のあるデータを電話に残します。

iOS 10.3では、Appleはさらに多くの問題を投げかけました。このバージョンにアップグレードすると、ユーザーがアプリケーションを再インストールするか更新をロールするまで、設定で識別子が消えます。 拡張機能は全体として不安定に動作します。 定期的に（不明な理由や法律のため）、更新時に設定からオフになるか、完全に消えます。 場合によっては、番号を更新する過程で、システムはエラーコードを使用して拡張機能をサイレントに釘付けにします。

そのため、近い将来、ユーザーにとって製品を改善できる可能性は低くなります。



最終的にどのように機能するか：



  1. ユーザーは都市をスイングします。
  2. 都市から、私たちの形式で数値のデータベースを取得します。
  3. ユーザーにインストールされているすべてのデータベースを調べます（拡張機能とメインアプリケーション間の共通のUserDefaultsに保存します）。
  4. 各データベースにはハッシュがあります。 少なくとも1つのハッシュが一致しない場合、または新しいハッシュが表示される場合、すべての新しいデータベースを共有ストレージに書き込み、インストールの準備ができているとマークします。 これは、ユーザーが拡張機能をアクティブにしないが、アプリケーションを最小化し、後で有効にする場合に必要です。
  5. 拡張機能がアクティブな場合、次の方法で再起動します。 



      [[CXCallDirectoryManager sharedInstance] reloadExtensionWithIdentifier:bundleID completionHandler:^(NSError * _Nullable error) {}];


  6. 拡張機能自体で、彼が受け取ったとき： 



      - (void)beginRequestWithExtensionContext:(CXCallDirectoryExtensionContext *)context


    インストールの準備ができているベースがあるかどうかを確認します。 ある場合は、すべてを調べて、次の番号を追加します。 



      [context addIdentificationEntryWithNextSequentialPhoneNumber:phone label:name];


  7. ベースをインストール済みとしてマークします。
  8. 更新ごとにプロセスを繰り返します。
