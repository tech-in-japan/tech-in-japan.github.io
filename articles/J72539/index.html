nginxでキャッシュを使用する場合の落とし穴

Webサーバーとリバースプロキシnginxには、HTTP応答をキャッシュするための非常に強力な機能があります。 ただし、場合によっては、ドキュメントと例では不十分であり、その結果、すべてが私たちが望むほど簡単で単純ではないことがあります。 たとえば、nginxの設定は、血のある場所に書き込まれます。 この記事では、状況を少し改善しようとします。



この記事の内容：a）全ページキャッシュの落とし穴。 b）ローテーションを伴うキャッシュ。 c）キャッシュされたページに動的な「ウィンドウ」を作成します。



nginx + fastcgi_phpバンドルを使用していると仮定します。 nginx + apache + mod_phpを使用している場合、ディレクティブ名をfastcgi_cache *とproxy_cache *に置き換えるだけです



ページをPHP側でキャッシュするかnginx側でキャッシュするかを選択する場合、nginxを選択します。 まず、難なく、「高負荷」についてスマートに話すことなく、1秒あたり5〜1万件のリクエストを送信できます。 第二に、nginxはキャッシュのサイズを個別に監視し、陳腐化した場合と、使用頻度の低いデータを混雑させた場合の両方でキャッシュを消去します。



ページ全体のキャッシュ



サイトのメインページが動的に生成されてもめったに変更されない場合、nginxにキャッシュすることでサーバーの負荷を大幅に減らすことができます。 トラフィックが多い場合、キャッシュが非常に高速に動作するため、短期的なキャッシング（5分以下）でもパフォーマンスが大幅に向上します。 ページを30秒だけキャッシュした後でも、データを更新するダイナミズムを維持しながら、サーバーの大幅なオフロードを実現できます（多くの場合、30秒に1回更新するだけで十分です）。

fastcgi_cache_path：デバッグの容易さも重要です


fastcgi_cache_path / var / cache / nginx levels = keys_zone = wholepage：50m;



fastcgi_cache_pathディレクティブで、レベルの「空の」値を設定します。 これによりパフォーマンスがわずかに低下しますが（ファイルはディレクトリに分割せずに/ var / cache / nginxに直接作成されます）、キャッシュの問題をデバッグおよび診断するのがはるかに簡単です。 私を信じて、あなたは/ var / cache / nginxに登り、そこに何回保存されているかを見なければならないでしょう。



fastcgi_cache_valid：応答コード304もキャッシュする


fastcgi_cache_valid 200 301 302 304 5m;



fastcgi_cache_validディレクティブでは、標準コード200 OK、301 Moved Permanently、302 Foundだけでなく、304 Not Modifiedも強制的にキャッシュします。 なんで？ 304の意味を思い出しましょう。これは、2つの場合に空の応答本文で発行されます。
  • ブラウザが「If-Modified-Since：date」ヘッダーを送信した場合、日付は「Last-Modified：date」応答ヘッダーの値以上です。 つまり クライアントは次のように尋ねます。「これから新しいバージョンはありますか？ そうでない場合は、304を返してトラフィックを節約します。 あれば、ページの本文を教えてください。
  • ブラウザが「If-None-Match：hash」ヘッダーを送信した場合、ハッシュは応答ヘッダー値「ETag：hash」と一致します。 つまり クライアントは次のように尋ねます。「ページの現在のバージョンは、前回リクエストしたものと異なりますか？ そうでない場合は、304を返してトラフィックを節約します。 もしそうなら、ページの本文を教えてください。
どちらの場合も、Last-ModifiedまたはETagはnginxキャッシュから取得される可能性が高く、検証は非常に高速です。 特に200の応答を受け取るクライアントにキャッシュから返されるという事実に照らして、スクリプトにこれらのヘッダーを表示させるためだけにPHPを「プル」する必要はありません。



fastcgi_cache_key：依存関係を慎重に扱う


fastcgi_cache_key "$ request_method | $ http_if_modified_since | $ http_if_none_match | $ host | $ request_uri";



特に注目すべきは、fastcgi_cache_keyディレクティブの値です。 このディレクティブの最小作業値を指定しました。 右側のステップ、左側のステップ。場合によっては、キャッシュから「誤った」データを受信し始めます。 だから：
  • 必要な$ request_methodへの依存 インターネットでのHEADリクエストは非常に一般的です。 HEADリクエストへの応答にボディが含まれることはありません。 $ request_methodの依存関係を削除すると、HEADメソッドを使用してメインページを要求する前に誰かがGETで空のコンテンツを取得する可能性があります。
  • $ http_if_modified_sinceへの依存性は、304 Not Modifiedレスポンスを持つキャッシュが、通常のGETリクエストを行うクライアントに誤って与えられないようにするために必要です。 そうしないと、クライアントはキャッシュから空の応答を受け取る場合があります。
  • $ http_if_none_matchでも同じです。 空白のページを顧客に提供しないように保証する必要があります！
  • 最後に、$ hostおよび$ request_uriへの依存関係にはコメントは不要です。
fastcgi_hide_header：セキュリティ問題の解決


fastcgi_hide_header "Set-Cookie";



fastcgi_hide_headerディレクティブは非常に重要です。 これがないと、セキュリティ上の重大なリスクが発生します。ユーザーはキャッシュ内のセッションCookieを通じて他のユーザーのセッションを取得できます。 （確かに、nginxの最新バージョンでは、この要因を自動的に考慮するために何かが行われました。）これがどのように起こるか理解していますか？ Vasya Pupkinはサイトにアクセスし、セッションとセッションCookieを受け取りました。 その時点でキャッシュを空にすると、Vasina Cookieがキャッシュに書き込まれます。 その後、別のユーザーが来て、キャッシュから応答を受け取り、その中にVasyaのCookieが入りました。 それは彼のセッションも意味します。



もちろん、次のように言うことができます。メインページでsession_start（）を呼び出さないようにすれば、Cookieに問題はありません。 理論的にはこれは真実ですが、実際にはこの方法は非常に不安定です。 多くの場合、セッションは「延期」されて開始され、セキュリティホールが発生するため、コードのどの部分でもセッションへのアクセスを必要とする関数を「誤って」呼び出すだけで十分です。 また、セキュリティは、特定の手法の過失が原因でホールが発生する可能性がある場合、その手法は定義上「漏れやすい」と見なされます。 さらに、セッションCookieを除く他のCookieがあります。 また、キャッシュに書き込む必要もありません。



fastcgi_ignore_headers：タイプミス中にサイトを負荷から「横に」させない


fastcgi_ignore_headers "Cache-Control" "Expires";



nginxサーバーは、PHPが生成するCache-Control、Expires、およびPragmaヘッダーに注意を払います。 ページをキャッシュする必要がない（または既に古くなっている）と言われた場合、nginxはキャッシュファイルに書き込みません。 この動作は、論理的に思えますが、実際には多くの困難を引き起こします。 したがって、ブロックします。fastcgi_ignore_headersのおかげで、見出しに関係なく、ページのコンテンツはキャッシュファイルに保存されます。



これらの困難は何ですか？ これらは再びセッションとsession_start（）関数に関連付けられます。これは、PHPでデフォルトでヘッダー「Cache-Control：no-cache」と「Pragma：no-cache」を設定します。 この問題には3つの解決策があります。
  • キャッシングが想定されているページでsession_start（）を使用しないでください。 既に上で検討したこの方法の欠点の1つは、不注意な動きが1回あれば十分であり、キャッシュされたメインページへの1秒あたり数千のリクエストを受け取るサイトが、キャッシュがオフになると即座に「横たわる」ことです。 2番目のマイナス-2つの場所でキャッシュロジックを管理する必要があります：nginx構成とPHPコード。 つまり このロジックは、システムのまったく異なる部分に「広がり」ます。
  • ini_set（ 'session.cache_limiter'、 ''）を設定します。 これにより、セッションでの作業時にキャッシュを制限するヘッダーの出力をPHPで強制的に防止できます。 ここでの問題は同じです。「ぼやけた」キャッシングロジック。理想的には、すべてのキャッシングを1か所から管理したいからです。
  • fastcgi_ignore_headersを使用してキャッシュファイルに書き込む場合、キャッシュ禁止ヘッダーを無視します。 これは、Win-Winのソリューションであると思われるため、これをお勧めします。


ロータリーキャッシング



静的なホームページはそれほど興味深いものではありません。 サイトに多くの資料があり、メインがそれらの一種の「ショーケース」として機能する場合はどうすればよいですか？ このような「ウィンドウ」では、「ランダムな」素材を表示して、さまざまなユーザーがさまざまなものを見られるようにすると便利です（1人のユーザーでも、ブラウザーでページをリロードすることで新しいコンテンツを受け取ります）。



ソリューションは、ローテーションを使用したキャッシュです。
  1. スクリプトを強制的にランダムにメインページに要素を返し、データベースに必要なクエリを（ゆっくりではありますが）実行します。
  2. 次に、1つではなく、たとえば10ページのバリエーションをキャッシュに保存します。
  3. ユーザーがサイトにアクセスすると、これらのオプションの1つが表示されます。 この場合、キャッシュが空の場合はスクリプトが起動され、空でない場合はキャッシュから結果が返されます。
  4. 別のユーザーが1日でサイトのすべての資料を「見る」ことができるように、キャッシュエージングタイムを小さく設定します（たとえば、1分）。
その結果、スクリプトジェネレーターへの最初の10個のリクエストが「正直に」実行され、サーバーが「ロード」されます。 しかし、それらはキャッシュに「落ち着き」、1分以内にすでにすぐに発行されます。 生産性の向上は、サイトへの訪問者が多いほど大きくなります。

Pragma：no-cacheとCache-Control：no-cacheの違いは何ですか？プラグマがHTTP / 1.0のレガシーであり、古いブラウザーとの互換性のために現在サポートされているからです。 HTTP / 1.1はCache-Controlを使用します。



ただし、ブラウザにはまだキャッシュがあります。また、場合によっては、ブラウザがページを表示するためにサーバーにリクエストを行おうとしないこともあります。代わりに、彼は自分のキャッシュからそれを取得します。なぜならローテーションがあるため、この動作は不便です。ユーザーがページにアクセスするたびに、新しいデータを表示する必要があるためです。 （実際、まだ1つのオプションをキャッシュしたい場合は、Cache-Controlヘッダーを試すことができます。）

キャッシュページの動的な「ウィンドウ」



そして最後に、キャッシュの観点から有用なテクニックの1つを紹介します。サイトのメイン（または他の）ページをキャッシュしたいが、動的でなければならない1つの小さなブロックが干渉する場合は、SSIを操作するためにモジュールを使用します。
