ヴァラとは

Valaプロジェクトは2006年に作成されましたが、一般ユーザーや多くの開発者の間ではまだほとんど知られていません。 それが何であるか、そして最も重要なことには、なぜそれが必要なのか理解している人はほとんどいません。 そして、ヴァラのロシア語を話すITコミュニティの中で、それはまったく神秘的であり、この主題に関する記事はほとんどありません。 私は現在の状況をわずかに修正し、この技術について少し検討することにしました。









Linuxでのデスクトップ開発の現状



遠くから行こう。 デスクトップLinuxでは、2つのテクノロジー（QtとGtk）が長い間支配されてきました（2つの異なる世界と言えます）。 混乱を避けるために、QtはGUIを構築するためのライブラリであるだけでなく、GtkはGLibに基づくフルスタックの多くのライブラリの1つにすぎないことに注意してください。 したがって、デスクトップは、これらのテクノロジーに基づいた2つのデスクトップ環境-KDEとGnomeに分割されます。

Qtを使用する一部の開発者にとっては、すべてが非常にスムーズでクラウドレスです。 細長く美しいAPI、便利な開発ツール、優れたドキュメント、大規模なコミュニティがあります。 Gtkの世界ではすべてがそれほどバラ色ではありません。 最初は、開発はC言語で行われます。この言語では、多くの抽象化と規則を使用してGLibが実装されます。これは、Cにはなかった本格的なオブジェクトモデルです。 GLibを使用するアプリケーションのコードは非常に奇妙に見え、ほとんどの一般的なGUIライブラリの開発者が慣れているものとはまったく似ていません。 さらに、言語自体は、そのレベルが低いため、グラフィカルインターフェイスを備えたプログラムの開発には役立ちません。

その結果、Gnomeの開発者にはいくつかの回避策があります。 1つ目は、Qtで記述し、アプリケーションが他のアプリケーションと比較して異質にならないようにすることです。 2つ目は、Gtkにバインドされているスクリプト言語を使用することです。 そして、これらの十分なものがあります：Python、Ruby、Lua、Lisp、Perl、さらにはPHP。 3番目は、Javaで記述することです。 これらすべてのソリューションには、利便性と開発時間の短縮という明らかな利点があります。 しかし、欠点もあります。 まず第一に、それは実行速度です。これは、C言語と比較して、スクリプト言語では著しく低いです。 プログラムの起動速度も低下します。 Qtにはこれらの欠点はありませんが、上記の異質性は、近年大幅に平準化されているとはいえ、感じられます。

しかし、最近集中的に提供している主なオプションはMonoです。 インタビューで、開発者の1人は次のように言っています。 そして、彼に反対するのは難しいです。 Monoは、最新で高レベルのC＃言語、便利なMonoDevelop IDE、広範なコミュニティを提供します（これは、Gtkアプリケーション開発ではなく、主にC＃の問題を直接支援できることに注意してください）。 いいですね。 ただし、Monoはすべて同じ問題をもたらします。 起動と実行の速度は、十分ではありますが、Cの速度よりも著しく低いです。 さらに、MonoはGnomeの万能薬にはなり得ません。これは、Monoで個別のソフトウェアのみを開発することが可能であり、環境自体またはライブラリの形でのその部分を開発できないためです。

この状況でGnomeプロジェクトがありました。 カスタムソフトウェアは心地よいもので書かれていますが、元のCでは書かれていません。 当然、例外もありましたが、これはCでデスクトップソフトウェアを作成する開発者が比較的少ないという事実を変更するものではありません。 そして、毎年その数は明らかに増加していません。 シンプルでエレガントなソリューションが必要でした。 ご想像のとおり、彼の名前はValaです。



ヴァラとは



まず第一に、Valaは、JürgBillerterとRaffaele Sandriniの2人だけが2006年に立ち上げたプロジェクトです。 Gtk開発者向けの便利で最新のツールキットを作成し、エントリのしきい値を大幅に下げるように設計されました。 このプロジェクトは、ValaとGenieの2つの言語で構成されています。 実際、両方ともまったく同じですが、構文は異なります。 ValaはC＃とJavaに非常に似ていますが、GenieはPythonとBooと多くの共通点を持っています。 Valaの方が人気があるため、それについて説明します。

Valaの概念は非常に単純です。コードは便利な高級言語で記述され、最も一般的なCコードに変換され、実行可能なファイルまたはライブラリにコンパイルされます。 ブロードキャストの透過性により、オーバーヘッドが完全になくなります。 つまり、Valaで開発する場合、Valaに関連する追加のライブラリをドラッグしないでください。 この機能には別の利点があります-完全なバイナリ互換性。 これは、Valaでライブラリを記述でき、Cコード（またはバインディングがある場合は他の言語）から使用できることを意味します。 したがって、Valaは、カスタムソフトウェアだけでなく、Gnome環境自体またはこのプロジェクトの個々のライブラリの開発にも適しています。

プログラミング言語には、最も明白な問題を解決するための一連の関数またはクラスを備えた標準ライブラリがあります。 Valaでは、この役割は一連の標準バインディングによって果たされます。 開発者は、Gnomeが構築されているライブラリのスタック全体を自由に使用できます。 これは、Valaで非常に大規模で複雑なプロジェクトをすぐに作成できることを意味します。 バインディングの完全なリストはここにあります 。 バインディングは、ライブラリ関数と構造がクラス、名前空間、メソッドなどにどのように関連するかを翻訳者に説明するvapiファイルです。 既存のライブラリまたはCコードへの独自のバインディングを作成できます。



言語の詳細



前述のように、ValaはC＃とJavaに非常に似ています。 実際のネイティブプロパティ、シグナル、ジェネリックとデリゲート、抽象クラスとインターフェイスがあります。 多重継承はありませんが、クラスとインターフェースから同時に継承できます。 また、Valaのインターフェースには本格的なメソッドを含めることができるため、問題が部分的に解決されます（存在する場合）。 メソッドのオーバーロードもありません。 ただし、デフォルトのパラメーターと名前付きコンストラクターがあります。 プロパティは変更時に信号を送信するため、非常に便利です。 Valaは、クロージャーとモジュールの透過的なインポートをサポートしています。 一般的に、それは十分な表現力のあるツールを備え、同時にC＃開発者になじみの重大度を保持する現代的で美しいプログラミング言語です。 Valaのメモリ管理は自動参照カウントによって行われるため、開発者は常にそれについて考える必要はありません。 しかし、たとえばC ++のように、手動制御の可能性もあります。

ボタンをクリックするとメッセージを表示する単純なウィンドウプログラムの例を次に示します。

using Gtk;



public static int main ( string [] args) {

Gtk.init ( ref args);



Window window = new Window(WindowType.TOPLEVEL);

window.set_size_request(100, 100);

window.destroy.connect(main_quit);



Button button = new Button.with_label( "Push me" );

button.clicked.connect(() => {

MessageDialog msg = new MessageDialog(window, DialogFlags.MODAL,

MessageType.INFO, ButtonsType.OK, "Some message" );

msg.run();

msg.close();

});



window.add(button);

window.show_all();



Gtk.main();

return 0;

}

私はあなたがすべての行を解析するべきではないと思います、とにかくすべてが明確でなければなりません。 言語の基本についての詳細は、 公式チュートリアルに記載されています。 この例をコンパイルするには、コンソールに次の行を入力する必要があります（コードがtest.valaファイルに保存されている場合）。



valac test.vala --pkg gtk+-2.0 -o test







ツールキット



Valaプロジェクトは、C言語のValaおよびGenie言語の翻訳者を提供します。 一般的なディストリビューションでは、公式リポジトリにあります。 ただし、原則として、これらは廃止されたバージョンです。 したがって、最新のパッケージを含む追加のリポジトリを探すことをお勧めします。 たとえば、Ubuntuの場合、 vala-teamがあります。

ValaにはいくつかのIDEがあります（それらを呼び出すことができる場合）。 まず、 Val（a）IDEです。 強調表示できるのは、ディレクトリツリー、プロジェクトファイルのリスト、自動補完（これまでのところ非常に弱い）、独自のビルドシステム、およびサードパーティシステムのサポートです。 何も設定する必要はありません。すべてが自動的に組み立てられ、起動されます。コードを書く時間があるだけです。

ValaのサポートはMonoDevelopとAnjutaで発表されましたが、個人的には、これらの環境は支援よりもコードの作成を妨げる可能性が高いという印象を受けました。

そして最後に、私が自分で使用するもの： ValenciaはGeditのプラグインです。 彼は自動補完とValaのシンボルのリストをGnomeの標準エディターにもたらします。 もちろん、これでは十分ではありませんが、少なくともコードの作成を妨げることはありません。 Valaの開発ツールの完全なリストは、 http：//live.gnome.org/ValaのIDEサポートセクションにあります。

しかし、プログラムを作成するには、1人のエディターでは不十分です。 ビルドシステムが必要です。 オープンソースの世界では、 CMakeの人気が高まっています。 これは非常に柔軟で強力なソリューションですが、... Valaのサポートはありません。 ただし、これは一時的な現象であり、次のリリースまでに、この迷惑な省略を修正することを約束しています。 それまでの間、プロジェクトの開発者は、Valaでプログラムをビルドするために必要なスクリプトを含む特別なマクロを使用できます。

また、Pythonで記述されたビルドシステムであり、Valaの初期サポートがあるWafは無視できません。 スクリプトはPythonで記述されているため、アセンブリに必要なものすべてと無限の柔軟性を備えた非常に便利なシステムです。 ただし、1つの重大な欠点があります。すべてのディストリビューションで適切に処理されるわけではありません。 特に、Waf はリポジトリから削除される予定であるため 、Debianのメンテナーは開発者に別のシステムへの切り替えを求めています。

最近では、Valaの開発者が非常に興味深い機能を発表しています-Valaはインタプリタモードです。 任意のスクリプト言語のようにスクリプトを書くことができ、それらは手動でアセンブルする必要なしにオンザフライで実行されます。 これにより、言語への親しみが大幅に簡素化され、より楽しくなります。個人的な実験では、プロジェクトの組み立てや設計に煩わされる必要がないからです。



ヴァラの現状と展望



現時点では、Valaの最新バージョンは0.9.3です。 そして、1.0に近づくほど、開発は速くなります。 しかし、すべてがそんなに良いわけではありません。 プロジェクトのロードマップは何度か書き直されましたが、その版の1つによると、Vala 1.0は1年半前に登場する予定でした。 これで、 ロードマップのあるページは完全に消えました。 それにもかかわらず、プロジェクトは活発で活発に開発されています。 開発の最終段階前にもかかわらず、言語構文を含む新しい機能が追加されています。 これは、バージョンの変更時に互換性がなく、個々のコードを書き換える必要があるため、わずかな不便をもたらします。

Valaコミュニティは、小規模ではありますが非常に活発です。 メインのたまり場はメーリングリストで 、Valaに関連するトピックに関する多くの議論が含まれています。 そこで、自分で質問をすることができます。ほとんどの場合、迅速かつ完全な回答が得られます。 しかし、公平のために、すべての質問が応答を見つけるわけではないことに注意する価値があります。 数か月前、私はValaでライブラリの1つに対する完全に機能しないバインディングを見つけました。 パッチを書きましたが、応答がありません。 すぐに状況が再び繰り返され、修正されたバインディングをプロジェクトに直接追加する必要がありました。



私の意見では、Valaの開発はGnomeプロジェクトの最も重要な分野の1つです。 「救い」とは言いたくありませんが、カスタムソフトウェアの開発者に向けたステップがなければ、プラットフォームを普及させることは非常に困難です。 GnomeのValaに関する完全な沈黙は、すべて奇妙に見えます。 Ubuntuの開発者であるCanonicalというこの環境の主なパトロンの側からのこの方向への公式なシフトはありません。 これまでのところ、技術としてのValaは自律的に存在し、その将来は純粋に独自のコミュニティに依存しています。 したがって、幸せな未来への唯一の方法は、このコミュニティを強化し、拡大することです。 どういたしまして
