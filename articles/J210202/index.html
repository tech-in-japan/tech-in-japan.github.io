すべては怠inessから来ています。 他の誰かの非常に大きなプロジェクトがあり、そこで小さな編集を行う必要があるか、スクリプトを作成したが、最適化が必要であることはすぐにはわかりません。 になる方法 現在、非常に多くのデバッグツールを使用できるため、コードを読み取って分析し、画面またはファイル（var_dump（）など）に各ステップを表示することは必ずしも効果的ではありません。 頻繁に発生する...



Xdebug Debugger and Profiler Tool -PHP拡張。 サーバーへのインストールと設定が必要です。 表示可能：関数呼び出しスタック、メモリ割り当て。 機能：プロファイリング、コードカバレッジ分析、無限再帰に対する保護、インタラクティブなスクリプトデバッグ。 Xdebugログ視覚化ソフトウェア： Webgrind -XdebugをプロファイリングするためのWebベースのインターフェイス、PHPで記述されたMacGDBp -Xdebugを使用してPHPアプリケーションをデバッグできるMac OS Xクライアント。 Linux GUI kcachegrind 。 無料です。 多くのIDEと統合します。 xdebugを使用したPHPアプリケーションのプロファイリングを参照してください。 php.iniでオプションを有効にすると： 



html_errors = On


var_dump（）の出力とエラーメッセージをフォーマットします。



Xhprofは、facebookのPHP拡張機能です。 サーバーへのインストールと設定が必要です。 各関数の実行時間、メモリ使用量、レイテンシ、呼び出し数などを収集できます。 この拡張機能は、PECLリポジトリから入手できます 。 こちらのドキュメントをご覧ください[tyts] 。 また、xhprofとFirePHPを使用して、PHPアプリケーションのプロファイリングとデバッグを行います 。 利点のうち、システムにあまり負荷をかけないので、戦闘にかけることができます。 無料です。



DBG （PHPデバッガーとプロファイラー）-PHP拡張。 サーバーへのインストールと設定が必要です。 IDEまたはコンソールから、テストサーバーや実稼働サーバーで作業し、ローカルまたはリモートでスクリプトをデバッグできます。 有料/無料版。



ZendDebug -PHP拡張、Zend Studioの一部（有料IDE）。 サーバーへのインストールと設定が必要です。 これにより、xdebug、Zend StudioのGUI、またはZend Server IDEとほぼ同じことが可能になります。 有料。 以下で詳細に検討します。



MemtrackはPHPの拡張機能です。 メモリリークを検索できます。 Crownによって、またはデーモンとして実行されるスクリプトを確認すると便利です。 無料です。 [tyts]を参照



APD Advanced PHPデバッガーはPHP拡張です。 xdebugとの競合は弱いですが、memtrack機能があります。 IDEとの統合は不十分ですが、コンソールインターフェイスがあります（[ tyts ]を参照）。 無料です。



DTrace + PHPはPHP拡張機能です。 低レベルのデバッグ。 [tyts]を参照してください。 Straceやその他のシステムデバッガーの存在を忘れないでください。これらは、いわば「犬が暴走した」場所を示すことができます。 例えば

strace -p 1111





システムコールスクリプトの分析、PID = 1111。 ネットワークアナライザーwireshark （Windows）、 ngrep 、 tcpdump （Linux）-ネットワークトラフィック、プロトコルなどの分析用



FirePHP -phpで記述されたクラス+ FireFoxの拡張。 phpメソッドを呼び出すことにより、デバッグメッセージをFirebugコンソールに送信できます。 すべての情報はX-FirePHP-Dataのヘッダーを介して送信されるため、ページのメインコンテンツと重複しません。 無料です。 Firebugを使用したPHPのデバッグを参照してください



php-console -Google Chromeのphp +拡張で書かれています。 Google Chrome専用のFirePHPに類似していますが、機能は多少異なります。 無料です。 php-consoleを参照してください



phpで記述されたPHP_Debugクラス。 PHPコードのデバッグに役立ち、スクリプトの実行パスを表示し、すべての変数、ランタイム、インクルードファイル、実行されたクエリ、変数を監視します...この情報はスクリプトの実行中に収集され、その最後に表示され、いつでも使用できます。 無料です。

ZendDebugに戻ります。 私は主にZend Studioを使用しているため、これを使用するのが最も便利です。 これにより、スクリプトの進行状況をすぐに理解でき、IDEからのコードのナビゲーションがサポートされます。 IDE以外のサードパーティツールは必要ありません。 一度セットアップして使用するので、とても便利です。



Zend Studioでのスクリプトのデバッグとプロファイリングは、xdebugまたはZendDebugを使用して少なくとも2つの方法で可能です。 xdebugを使用してサイトをプロファイリングしたのは、今ではうまくいきませんでした。これは不可能であり、デバッグのみです。



ローカルコードのデバッグについては、Zend Studio 5.5 [tyts]の時代に書き戻されました。 それ以来、ほとんど変わっていません。 しかし、Webサーバーとデバッグコードがリモートサーバーにある場合、問題が発生しました。 多くの場合、このようなサンドボックスは外部から閉じられ、作業に必要なポートのみが開かれます。 ただし、SSHを介してこのようなサンドボックスにアクセスできる場合、ファイアウォールの機能を妨げることなくZendDebugを構成できます。



今後、SSHトンネルを作成する必要があることに注意してください。 この場合にSSHトンネルが必要な理由について少し説明します。



デフォルトでは、Zend StudioはデバッグサーバーにHTTPリクエストを送信してリモートデバッグセッションを開始します。 この要求には、ZendDebug（サーバーにインストールされている）がZend Studioへの新しい接続を開始してデバッグ情報を中継するときに使用するリターンアドレスパラメーター（IPアドレスとポート番号）が含まれています。 ところで、Zend Studioに付属のコンポーネントをインストールすることにより、IDEとブラウザの両方からデバッグセッションを初期化できます。これは非常に便利です。







たとえば、コード、WEBサーバー、およびIDEがローカルコンピューターに配置されている場合は、定期的なデバッグセッションが行われます。



ただし、多くの場合、WebサーバーはIDEファイアウォール、ルーター、プロキシなどと共有されます。 これは、SSHトンネルが役立つ場所です。



トンネルの場合、デバッグセッションを確立するプロセスは、2つの主要なステップで構成されます。

-SSHトンネルの作成。

-SSHトンネルを介してトラフィックを送信するためのZend Debugger設定。
