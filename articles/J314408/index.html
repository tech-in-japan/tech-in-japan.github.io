なじみのないシステムにリクエストを送信するための一般的なオプション



なじみのないシステムにリクエストを送信するために、次のオプションを使用しました。



  1. 彼は、テスト対象のシステムに要求を送信できる中間Webアプリケーションを作成し、 HTTP要求コンポーネントを使用してWebアプリケーションにコマンドを送信しました。
  2. テストされたシステムに要求を送信できる中間Webサービスを作成し、 SOAP / XML-RPC要求コンポーネントを使用してWebサービスにコマンドを送信しました。
  3. OS Process Samplerコンポーネントを使用してJMeterからクライアントと呼ばれる、テスト対象のシステム用のコンソールクライアントを作成しました。
  4. テスト対象システムを操作するためのJava（C＃）でライブラリ（API）を作成し、 JSR223サンプラーコンポーネントを使用してライブラリメソッドを呼び出しました 。
  5. JMeter用のプラグインを作成しました。


WebアプリケーションとHTTPリクエストまたはWebサービスとSOAP / XML-RPCリクエスト



JMeterはHTTPおよびSOAPリクエストを送信できることが知られています。 また、統合の問題を解決するために、テスト対象のシステムにリクエストを送信し、 HTTP経由でJMeterからコマンドを受信するWebアプリケーションを作成するオプションがよく選択されます。



例。 HTTPを介してSOAP / MSBin1 プロトコルで実行されるwcfサービスをテストしました。 JMeterは SOAP / MSBin1プロトコルを使用できません。1つのパケットで小さなリクエストを非常に読みにくい形式で送信できますが、大きなリクエストを送信したり、大きなレスポンスを受信したりすることはできません。

最初のソリューションはこれで、 Visual Studioを使用してテスト対象システム（wcfサービス）のクライアントを生成し、 HTTP経由でJMeterからコマンドを受信できるテストWebサービスを作成しました。



または、テストシステムにPython APIがある場合があります。 また、Pythonコードを呼び出すために、 JMeterからコマンドを受け取り、システムに負荷を適用するPython Webアプリケーションを作成できます。



軽量Webアプリケーション





テスト対象のシステムに小さな負荷をかけるWebアプリケーション。 Webアプリケーションにログインすることが重要です。



Webアプリケーションを使用する場合、常にタイムラグが発生します。 JMeterと中間Webアプリケーションとの相互作用時間は、低負荷でもゼロではありません。 したがって、テスト対象システムのリクエストに対する応答時間は、中間Webアプリケーションのログによって測定する必要があります。 JMeterで修正されたリクエスト処理時間は、意図的に長くなります。 したがって、 JMeterのログと中間Webアプリケーションのログの2つのログを維持することが重要です。 Webアプリケーションログを解析し、その統計を作成できるようにします。



負荷の高いWebアプリケーション





負荷が高いWebアプリケーションは、マスターされていない場合、障害点になる可能性があります。



大きな負荷を作成するとき、多くの場合、いくつかのJMeterを実行する必要があります。 中間Webアプリケーションが処理できない可能性がある多くの要求が作成されます。 タイムラグが大きくなります。 負荷が大きいと、Webアプリケーションへのログインが遅れる場合があり、テストされたシステムの応答時間を修正するときに、さらにタイムラグが発生します。 上の図では、中間Webアプリケーションを赤で強調表示し、ログファイルへの矢印が点線で示されています。



頻繁にテストされるシステムには、ロードバランサーと複数のアプリケーションサーバーも含まれます。 クライアントのIPアドレスに従って負荷分散が実行される場合、1つの中間Webアプリケーションを使用する場合、負荷はテスト中のシステムのノードの1つのみに適用されます。 他のノードはアイドル状態になります。 上の図でそれを把握し、テストしたシステムのノードの1つを赤で強調表示しました。



大きな負荷をかける前に、次のことが重要です。



  • 要件を設定し、中間Webアプリケーションの最大パフォーマンスを決定します。
  • テストするときは、テスト対象のシステムのバランス方法を考慮してください。




大きな負荷を作成するには、複数のJMeterと複数の中間Webアプリケーションを使用する必要があります。負荷は均等に分散されます



複数のロードステーションがあり、中間Webアプリケーションを介してロードが処理される場合。 負荷を処理するWebアプリケーションを使用して、各ロードステーションで個別のWebサーバーを実行すると便利です。



その後、テスト対象システムのバランサーがIPアドレスによるアプリケーションサーバーへのクライアントバインディングを使用している場合でも、負荷は均等に分散されます。



Pandasライブラリ（ Pythonから）を使用して多くのログを処理し、それらの統計を生成すると便利です。 また、CSVファイルをすぐに取得できるように、Webアプリケーションログを作成します。 このようなCSVファイルは、 MatPlotLibを使用して、サイクルですぐにダウンロード、分析、プロットできます。 Jupyther 、 Pandas 、およびMatPlotLibはログ処理の忠実なヘルパーです。



コンソールアプリケーションとOSプロセスサンプラー



アプリケーションの起動を通じて負荷をロードするオプションは、次の場合に正当化されます。

  • テスト対象のシステムを操作するためのアプリケーションの形で、既製の便利なクライアントがあります。
  • テスト対象のシステムには既製の単体テストがあり、 JMeterを使用してこれらのテストを実行する必要な数のスレッドを開始し、実行時間を測定するだけです。
  • 暗号化を頻繁に使用する必要があります。C＃/ Javaを知っており、C＃/ Javaで暗号化を使用する方法を知っています。
  • テスト対象のシステムには.NET / Win32 / Java / Python用のSDKがあり、C＃/ C ++ / Java / Pythonを知っていて、SDKの例を使用してコンソールクライアントを簡単に記述できます。
  • 巨大なファイルをダウンロードまたは送信する必要があり、標準のJMeterコンポーネントはそのような要求を実行するときにリソースを大量に消費します。
  • テスト対象のシステムを操作する場合、 JMeterのフォームよりもIDE（ Intellij IDEA 、 Eclipse 、 KDevelop 、 Visual Studioなど）で実装およびデバッグが簡単な複雑なスクリプト。




クライアントアプリケーションの起動によるテスト対象システムの負荷



最初にコンソールアプリケーションを介して負荷を使用したとき、テストでHTTPを介して100 MBを超え、最大数ギガバイトのファイルをダウンロードする必要がありました。 そのような回答を受け取ると、 JMeterは使用可能なすべてのRAMを消費しました。 Wgetが助けになりました。このコンソールツールは、巨大なファイルを簡単にダウンロードし、最小限のRAMを消費し、 gzipとキープアライブをオンまたはオフにできる設定を備えています。 OSプロセスサンプラーとwgetは、チャネルをポンピングしてテストを実行することを許可しましたが、これはHTTPリクエストでは実現できませんでした。



アプリケーションを使用する機能：



  • アプリケーションは別のプロセスであり、大量のRAMを消費します。大きな負荷を作成するには、大量のRAM、高速プロセッサ、または複数のロードステーションが必要です。
  • 各アプリケーションは独自のログを保持するため、後で多くのログを処理して統計を取得するのに便利であることに注意する必要があります。
  • テスト中のシステムへの短いリクエストを1つだけ実行するためにアプリケーションを実行するのは無駄が多いため、アプリケーションにシナリオ全体を実装すると便利です。認証、有用なアクションのいくつかの反復、終了です。


アプリケーションが障害点になることはありません。 それらが機能する場合、タイムラグは中間WebサービスまたはWebアプリケーションを使用する場合よりもはるかに短くなります。 開発とデバッグが簡単です。



コンソールアプリケーションがスクリプトを巧みに実行するためには、擬似コードを使用して標準入力からのスクリプトの受信をコンソールアプリケーションに実装すると便利です。 たとえば、これ： 



#AuthByCertificate "  " #GetInboxDocuments #GetContagentList #SendDocument ".\data\Document1.txt" #SendDocument ".\data\Document2.jpg" #Exit


このようなシナリオのパーサーを作成します。行が＃で始まる場合、これがコマンドである場合、制御はそのようなコマンドのハンドラーに転送されます。 必要に応じて、ハンドラーは実行に必要なパラメーター、たとえばユーザー証明書の名前やファイルへのパスを読み取ります。



そのようなインターフェイスを開発する時間がなければ、スクリプトを関数にラップして、コマンドラインパラメーターを介して関数の名前とそのパラメーターを簡単に転送できます。 



 ConsoleApp.exe /test1 "  " ".\data\Document1.txt" ".\data\Document2.jpg" ConsoleApp.exe /test2 "UserLogon1" "UserPassword1" ".\data\Document1.txt"


通常、シナリオの数は制限されており、テストの前に説明されています。 したがって、このオプションも便利です。



OS Process Samplerを使用して、任意の数のパラメーターを渡すことができます。 JMeter変数を使用して、CSVファイルからパラメーターを読み取ることができます。 しかし、実際には、バッチファイルを記述してコンソールアプリケーションを起動する方が便利であることが示されています。 また、アプリケーション自体ではなく、 OS Process Samplerからバッチファイルを呼び出します。



スクリプトがテストに結果を返す必要がある場合。 この結果のコンソールへの出力を文字列として実装すると便利です。 JMeter側では、応答パラメーターを処理するポストプロセッサを実装します。



既製のアプリケーションが使用され、成功およびエラーに対して特定の応答コードが返されない場合。 次に、 OS Process Samplerの操作ステータスを設定するために最低限必要なポストプロセッサが必要です-アプリケーションが正常に完了したかどうか。



wgetを起動し、ポストプロセッサによってその作業結果を処理する例
wgetを使用して速度制限が設定された大きなファイルをダウンロードするためのバッチファイルwget-download.gzip.bat 。 バッチファイルは、1つのパラメーターを入力として受け取ります-ダウンロードするファイルのアドレス、パラメーター値はOS Process Samplerから転送されます。 テキストwget-download.gzip.bat ： 



 @title %~nx0 @setlocal @cd /d %~dp0 @rem 100 MBit/sec @rem --progress=dot:default ( 1 KByte per line) @rem --progress=dot:binary (384 KByte per line) @rem --progress=dot:mega ( 3 MByte per line) %~dp0wget/bin/wget.exe -S --limit-rate=20m --progress=dot:mega ^ --header "User-Agent: Mozilla/5.0 (Windows NT 6.1; WOW64; rv:44.0) Gecko/20100101 Firefox/44.0" ^ --header "Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8" ^ --header "Accept-Language: ru-RU,ru;q=0.8,en-US;q=0.5,en;q=0.3" ^ --header "Accept-Encoding: gzip, deflate" ^ --output-document=- %1 1>NUL




JSR223 PostProcessorは、 Groovyスクリプトを使用してwgetによってコンソールに表示される応答を解析し、 OS Process Samplerの適切なフィールドに入力して、応答を分析するためにJMeterログですべての必要なデータを分析します。 Groovyを使用することは重要です。Groovyは他のものよりもコンパイルと実行が速いため、 JMeterの記事： greatvovanの BeanShell Samplerを忘れてください 。 ハンドラーテキスト： 



 // prev - http://jmeter.apache.org/api/org/apache/jmeter/samplers/SampleResult.html import java.util.regex.Matcher; import java.util.regex.Pattern; //Length: 1718181 (1.6M) [text/html] Pattern pLength = Pattern.compile("(?im)^Length[:] ([0-9]+) "); Matcher mLength = pLength.matcher(prev.getResponseDataAsString()); if(mLength.find()) { int Length = Integer.parseInt(mLength.group(1)); prev.setBodySize(Length); } Pattern pContentType = Pattern.compile("(?im)^[ ]{2}Content[-]Type[:] (.*)"); Matcher mContentType = pContentType.matcher(prev.getResponseDataAsString()); if(mContentType.find()) { String ContentType = mContentType.group(1); prev.setContentType(ContentType); } Pattern pHeaders = Pattern.compile("(?im)^[ ]{2}([a-zA-Z].*[:].*)"); Matcher mHeaders = pHeaders.matcher(prev.getResponseDataAsString()); String headers = ""; while(mHeaders.find()) { headers = headers + mHeaders.group(1) + "\n"; } prev.setResponseHeaders(headers); // HTTP/1.1 200 OK Pattern pCodeMessage = Pattern.compile("(?im)^[ ]{2}HTTP[/]1[.][0-9] ([0-9]+) (.*)"); Matcher mCodeMessage = pCodeMessage.matcher(prev.getResponseDataAsString()); if(mCodeMessage.find()) { String Code = mCodeMessage.group(1); String Message = mCodeMessage.group(2); if(Code=="200") { prev.setSuccessful(true); } else { prev.setSuccessful(false); } prev.setResponseCode(Code); prev.setResponseMessage(Message); }




ライブラリとJSR223サンプラー







アプリケーションは正常に動作しますが。 それでも、より少ないリソースを費やしたい-プロセスではなくスレッドを作成するために、いくつかのロードステーションとギガバイトのRAMがない場合があります。 JMeter変数を入力として受け入れたいです。 また、場合によっては、スクリプトを作成するときに、stdinアプリケーションに擬似コードを送信するのではなく、柔軟性を高めたいと思っています。 



 TestClient client = new TestClient(); String certificateCommonName = vars.get("certificateCommonName"); //"  " String documentPath1 = vars.get("documentPath1"); //".\data\Document1.txt" String documentPath2= vars.get("documentPath2"); //".\data\Document2.jpg" client.authByCertificate(certificateCommonName); client.getInboxDocuments(); client.getContagentList(); client.sendDocument(documentPath1); client.sendDocument(documentPath2); client.exit();


この場合、サイクルを使用し、一部のメソッドの結果を他のメソッドの入力に転送し、JMeterとアプリケーション開発環境を切り替えることなく、興味深いロジックを使用します。



または、仮想ユーザーごとに1回だけ認証およびログアウトします。 また、 JMeterからの1つのリクエストのフレームワーク内で、追加のタイムラグなしで、テスト中のシステムに対して1つのリクエストのみを実行します。



そのような方法があります-JSR223 Samplerを使用します。 そこからライブラリメソッドを呼び出すのが便利です。



JSR223サンプラーには、多くの便利な機能があります。



ライブラリの3つのJSR223サンプラー初期化、クライアントの作成、システムへの接続に分割できます。 そして、単一スレッド（仮想ユーザーJMeter）のフレームワーク内で、操作が正常に完了し、各ステップの時間を測定できるようになります。



テスト対象システムへのクライアントの初期化（.niのライブラリ、jni4netを使用してロード）。 テストのためにライブラリを一度ロードするだけで十分です： 



 import net.sf.jni4net.Bridge; import java.net.URL; Bridge.init(); Bridge.setDebug(false); Bridge.setVerbose(false); java.io.File clientDllFile = new java.io.File("TestClient.j4n.dll"); Bridge.LoadAndRegisterAssemblyFrom(clientFile);


個別に、スレッドごとに、テスト対象システムのクライアントインスタンスが作成されます。 



 import testClient.Client; //   Client client = new Client(); //     "testClient" JMeter vars.putObject("testClient", client); //          //       client String responseData = client;


そして、このクライアントのメソッド、たとえばauthByCertificateを呼び出すことができます。 



 import testClient.Client; //  ,   Client client = vars.getObject("testClient"); //      String certificateCommonName = vars.get("certificateCommonName"); //"  " //     client.authByCertificate(certificateCommonName);


jni4netを使用すると、.NETライブラリメソッドを呼び出すことができます。 jniだけを使用して、Win32ライブラリメソッドを呼び出すことができます。 ライブラリがすでにJavaで記述されている場合、追加のレイヤーは必要ありません。 もちろん、この方法でPython、PHP、bashのコードを呼び出すことはできませんが、これは必ずしも必要ではありません。



JSR223サンプラーで使用されるGroovy言語を使用すると、コンパイル済みの高速スクリプトを作成できます。



また、ウェブアプリケーションやシンプルなアプリケーションを使用する場合のように、大量のログを処理する必要はありません。 中間コンポーネントの使用によるタイムラグはありません。



リソース消費が最小限に抑えられます。 Groovyでコードを書くことができればよく、Javaに非常に似ています。



JMeterプラグイン



コードを書く気がないなら また、負荷を適用する場合、何十ものメソッドを実行する必要はありませんが、同じタイプの操作を何百ものパラメーター値で実行する必要があります。 JMeter用のプラグインを開発すると便利です。 ライブラリに似ていますが、グラフィカルインターフェイスを備えています。





JMeterのプラグイン



単一のログもあります。 タイムラグはありません。 アプリケーションを切り替えることなくパラメーターを設定することが可能です。



プラグインの開発はそれほど難しくありません。 シンプルなプラグインを開発し、既存のプラグインの機能を拡張し、それらを組み合わせます。 また、既存のものはプロジェクトリポジトリにあります。

  • http://jmeter.apache.org/ ;
  • https://jmeter-plugins.org/ 。


デバッガを使用して数日間は、プラグインの開発方法を理解します。



たとえば、TailSamplerプラグイン（記事TailSampler-Apache.JMeterでのGETリクエストの並列送信を参照）は、コンポーネントの組み合わせです：

  • jmeter.apache.orgからのHTTPリクエストと
  • jmeter-plugins.orgのダミーサンプル 。


また、元のコードは数百行しか含まれていません。
