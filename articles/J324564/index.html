ニューラルネットワーク



ニューラルネットワークは、現在バージョン11.1で紹介しているものを含め、当社が大きな一歩を踏み出す重要かつ「ホット」な分野です。 この知識の分野が世界中でどれだけ早く発展しているかを知ることは私にとって非常に興味深いことであり、Wolfram言語がこの開発の最前線にいたことを認識するのは素晴らしいことです。



私たちの目標は、Wolfram言語に完全に統合されたニューラルネットワーク用の高レベルインターフェースを作成することです。 バージョン11.1では、最近開発されたいくつかの新しいブロックが追加されました。特に、30の新しいタイプのニューラルネットワークのレイヤー（11.0の2倍）と、リカレントネットワークの自動サポートが追加されました。 その考えは、低レベルのライブラリを使用して言語に詳細を自動的に入力させるために、ニューラルネットワークをWolfram言語でシンボリックに定義できるようにすることです。 これは、直接接続のある通常のネットワークにとって非常に便利です。 ただし、リターンネットワーク（可変長シーケンスなど）の場合、低レベルのプログラミングを避けたい場合、これは基本的な必要性です。



Wolframのニューラルネットワークのもう1つの重要な機能は、画像、テキスト、その他のものを自動的にエンコードするように構成されていることです。 バージョン11.1では、 NetEncoderおよびNetDecoder機能は多くの新しいケースに対応しており、すでにWolfram言語に統合されているリストを拡張しています。



統合されたシンボリックインターフェイス全体で、Wolfram Languageは効率的な低レベルライブラリ（現在はMXNet ）を使用しているため、最新のプロセッサとGPU構成が最大限のパフォーマンスで使用されることに注意してください。 ところで、バージョン11.1のリリースにより、ニューラルネットワークの完全な仕様を.wlnetファイルに保存することが可能になりました。



シンボリックオブジェクトとしてのニューラルネットワークの処理には、多くの労力が必要です。 11.1には、新しいニューラルネットワークを象徴的に構築する関数（ NetMapOperatorおよびNetFoldOperator ）があります。 まさに、ニューラルネットワークはシンボリックであるため、操作が簡単です。たとえば、内部から実行していることを制御するために「分解」したり、さまざまなネットワーク要素のパフォーマンスを体系的に比較したりします。



バージョン11.1には、事前学習済みまたは未準備のニューラルネットワークモデルで動作するように設計された便利な関数NetModelがあります。 これまで、少数の既知のニューラルネットワークのみが含まれていましたが、毎週追加し、科学コミュニティ内で開発されているものを見つけ、独自のアイデアをいくつか追加する予定です。

ストレージ言語



定義のリストx = 7または$ TimeZone = 11を作成すると、削除するかセッションが終了するまでリストが残ります。 しかし、すべてのセッション中に定義を長持ちさせたい場合はどうでしょうか？ バージョン11.1では、これはPersistentValueによって可能になりました。



PersistentValueを使用すると、名前（たとえば"Foo" ）および "場所の保存"を指定できます（これは、 PersistenceTimeおよびExpirationDate関数でも可能です）。 「KernelSession」は、値が1つのセッションに対してのみ保存されることを意味します。 ただし、 「FrontEndSession」または「Local」 （コンピューターとの通信）、または「Cloud」 （世界中のクラウドを介した同期を意味する）を適用することもできます。



PersistentValue関数は非常に一般的です。 これにより、さまざまな場所（たとえば、さまざまなプライベートクラウド ）に値を配置できます。 また、 $ PersistencePathが表示順序を決定する場合、 MergingFunctionはこれらの値の結合方法（必要な場合）を決定します。

インフラ強化



Wolfram言語を最近開発する主な方法の1つは展開です。 私たちはこの方向で開発するために多大な努力をしました。



Wolfram Cloudのアップデートは絶えず行われており、非常に頻繁に（そして静かに）サーバーのパフォーマンスとユーザーインターフェースの機能を拡張しています。 バージョン11.1では、いくつかの重要な更新を行いました。



AutoCopy機能が登場しました 。これは任意のクラウドオブジェクトに適用でき、オブジェクトが使用可能になるたびに、新しいコピーが自動的に保存されます。 これは、たとえば、多くの人が個別に変更できるドキュメントを作成する場合に非常に役立ちます（「これらのアイデアを研究してください。ここから始められるドキュメントは...」など）。



CloudDeploy [ APIFunction [...]]は、APIのデプロイプロセスを大幅に簡素化します。 バージョン11.1には、APIの動作の側面を自動化するためのいくつかのオプションがあります。 AllowedCloudExtraParameters関数を使用すると、 「_ timeout」や「_geolocation」などのAPIパラメーターを自動化できるかどうかを確認できます。 AllowedCloudParameterExtensions関数もあります（いいえ、これはシステム上で最も長い名前ではありません。この栄誉は現在MultivariateHypergeometricDistribution関数によって所有されています）。 AllowedCloudParameterExtensions関数を使用すると、 x = valueだけでなく、 x __url = ... 、またはx __json = ...と言うことができます。



また、バージョン11.1では、Wolfram Cloudのプライベートインスタンスと新しいWolfram Enterprise Private Cloudをサポートするさまざまな機能が導入されました。 たとえば、Wolfram Cloudの$ WolframIDに加えて、プライベートクラウドでの認証を許可する$ CloudUserIDもあります。 システム内では、「マルチクラウド識別」に関連するあらゆる種類の新機能がサポートされています（これは難しいですが、Wolfram言語の象徴的な特徴により、あらゆる困難に美しく対処できます）。
