五つください。 またはJIRA 5.0を満たす





数日前、 Atlassianの有名な課題トラッカーJIRAの5番目のバージョンがリリースされました 。 過去数ヶ月にわたって、ベータ版とリリース候補版を十分にプレイしてきましたが、リリースを見て満足しました。 私が喜ぶことについての短い物語は、カットの下であなたを待っています。









以前のメジャーバージョン4.0は、10月にリリースされ、2009年にひげを生やしました。 最新のマイナーバージョン4.4の日付は昨年8月です。 今こそがファイブの時間です。













何をお願いしますか？



JIRA 5.0には、新しい光沢のあるインターフェイスや人工知能は搭載されていません。 このバージョンは、以前は忘れられがちだった管理者や開発者を喜ばせるように見えました。 しかし、エンドユーザーはおいしいパンなしで放置されることはありません。









新機能の完全な概要はここで読むことができますが、私は個人的に私を夢中にさせたものについて説明します。 それでは、始めましょう...









アプリケーションを作成する方が便利です



JIRAで最初にアプリケーションを作成したユーザーの混乱した顔を見たり、フィールドの「足場」で自分自身が時々失われたりする場合、この機能は最適です。













いつでも、生産的な作業に必要なフィールドを選択して、残りを非表示にすることができます。 当然、JIRAでは、デフォルト値を持たない必須フィールドを非表示にすることはできません。









また、便利なチェックマーク「別の作成」もあり、狂暴な怒りで新しいアプリケーションを作成するのに役立ちます。













@メンション



コメントテキストにipetrovなどを入力すると、このアプリケーションの通知を購読していない場合でも、このコメントの通知はその名前のユーザーに送信されます。 また、ユーザー名の自動一致により、作業がさらに簡単になります。













アプリケーションを共有する



Confluenceには便利なメカニズムが長い間登場しており、興味深いページを「共有」し、新しい人々を議論に引き付けることができます。 JIRAには同様の機能があります。













[共有]ボタンをクリックすると、便利なダイアログが表示され、目的のJIRAユーザー、またはJIRAにアカウントを持っていないユーザーの電子メールを指定し、現在のアプリケーションに小さなメモを付けて送信できます。 そのようなことは、アプリケーションを議論するのに役立つとは考えられません（結局、コメントでこれを行う方がよいでしょう）が、間違いなく適切な人々を議論に引き付けるのに役立つでしょう。









このオプションは、JIRA設定で送信メールサーバーを指定するまで利用できないことに注意してください。









リモートリクエストバインディング



JIRAの以前のバージョンは、複数のアプリケーションを相互に接続する機会を提供しました。これにより、重複したアプリケーション、相互に待機しているアプリケーションなどの作業が容易になりました。 Confluenceの関連記事へのリンク、別のバグトラッカーやヘルプデスクのアプリケーションへのリンクを配置したいユーザーは、プラグインを探すか、自転車を発明する必要がありました。 次のことができます。







  • 異なるJIRAインスタンスから2つのアプリケーションをリンクする
  • Confluenceのページにクレームを関連付ける
  • 申し立てを任意のURLにリンク
  • APIを介してこれらの関係をプログラムで作成する






開発者、もっと自由に呼吸してください！



JIRAを他のシステムと統合し、そのためのプラグインを作成する際に、開発者は人生をはるかに楽にする革新の数々を手に入れました。









REST API-さらにクールに



現在、 REST APIはJIRAの悲惨な付属物ではなく、非常に機能的なものです。 これにより、次のことができます。







  • アプリケーションを作成および編集する
  • 申し立てを削除する
  • リンク入札
  • アプリケーション、フィルター、ダッシュボード、アクセス権に関する情報を受け取る
  • そして、他の多くのグッズ


SDKで利用可能なREST APIブラウザは、新しいAPIを探索するプロセスを簡素化します。 本質的に、これはJIRA REST APIを学び、遊ぶ便利な機会を提供するプラグインです。









JAVA API-より安定しました



最後に、JIRA開発者は Java APIを煩わさないことを約束しました 。 APIはクリーンアップされ、所定の場所で書き直され、現在は安定しており、将来のバージョンと互換性があるはずです。 プラグイン開発者は、JIRA 5.0の準備方法についてすぐに読みました 。









結果は何ですか？



トリッキーなカスタマイズやプラグインなしでJIRAを使用する場合、5.0へのアップグレードを妨げるものは何もありません（テストサーバー上のすべてを最初に確認することを忘れないでください）。 この場合の更新は問題なく行われます。









アップグレードを妨げる可能性のある主な問題は、プラグインの互換性です。 現在、「5」のプラグイン交換の〜420プラグインのうち、 〜130が互換性があります。 一方で、5.0のプラグインは簡単に作成できるようになりましたが、同時に4.xから5.0への移植には時間がかかる場合があります。









JIRAで何かを変更したり、テンプレートを編集したり、電子メールハンドラーを追加したり、その他の便利だがハックしたりした場合は、 Answersサービスを忘れないでください。 JIRAの新しいバージョンで変更/削除されたAPIを使用しているかどうかを確認できるCheckupサービスと同様に、ヘルプ。
