Kotlin + Rx2：モバイルアプリケーション開発へのリアクティブで機能的なアプローチ

テクノロジーやライブラリの数は日々増加しており、モバイルプロジェクトを開始または開発するためのテクノロジーとアーキテクチャのスタックを選択する際に迷子になることがよくあります。 リスクを最小限に抑えることは間違いなく可能です。オプションの1つは、Android開発の専門家の意見を聞くことです。







そこで、 90Seconds.comのAndroid開発者であるDenis nekdenis Neklyudov（Android GDE）とStepan stepango Goncharovに注目しました。 インタビューでは、モバイルアプリケーションのアーキテクチャからプロジェクトでのRxライブラリの使用まで、いくつかの重要なトピックに触れます。 それでは始めましょう...





デニス・ネクリドフ



- こんにちは、デニス！ 現時点で開発のコンテキストであなたが注力していることを教えてください。



-こんにちは！ 現在、MobiusのパートナーであるStepan Goncharovと同様に、シンガポールに拠点を置く90Seconds.comの新しいモバイルクライアントを開発しています。 現在のプロジェクトで一番気に入っているのは、Stepanがアプリケーションをゼロから作成し始め、非常に興味深いアーキテクチャを構築できたことです。 さらに、Kotlinをプログラミング言語として使用したり、データを操作するための事後対応的なアプローチなど、ファッショナブルなアプローチであると言えるかもしれません。



- 数年前のインタビューで、「Google Developer Expert」のステータスとそれがあなたにとって何を意味するのかについて話しました。 私の意見では、重要なフレーズの1つは次のとおりでした。「また、私の好きな趣味に参加すること：会議、ポッドキャスト、記事で人々と知識を共有すること。」情報を広め、アイデアを促進する必要性について考えを変えてください。



-私は変更していませんが、 Android開発に関するポッドキャストの 30の問題、これは素晴らしい確認です。 この2年間で出会った、GDEのステータスにある別の問題を共有できます。 「新しいAndroidの新機能」というトピックを思い付くと、ドキュメントを読んでいる多くの人々がいます。20ページすべてを1つの便利なプレゼンテーションにきちんと入れて、「ああ、まあ、それは私です」自分で読むことができました。」 そして、あなたが大多数に知られていないトピックを思いついたとき、あなたはそれを伝えます、そして、彼らは完全に準備されておらず、30分のプレゼンテーションで入門資料の時間がないので、聴衆のほんの一部だけが彼らが話していることを理解しています。 そして、あなたの最初のプレゼンテーションを去った人々が2番目から同じを去らないという事実さえありません。



それが、ポッドキャストを録音することにした理由です。 あらゆるレベルの問題について話し合うので、誰もが耳を傾けることは興味深いでしょう。初心者と業界の専門家が自分自身に役立つものを見つけるでしょう。



- あなたの信条は、最新の開発と変更に遅れずについていくことですか？ 言語の今後の変更に関する知識は、コードを記述する特定のスタイルの順守に影響を与えますか、それともあなたは最新のテクノロジーの支持者ですか？



-私はバランスを維持する支援者です。 「私たちには、まだリリースされていない最新のライブラリがあり、時が経ってもテストされていないアプローチがあります」という極端に耐えることはできません。 また、長年の経験で12個以上のコーンを蓄積してきた人々が新しいことを試みているという完全な保守主義も好きではありません。 今日のビジネスに必要な開発速度と、明日ビジネスに必要なアプリケーションを他のチームとともに拡張、拡張、維持する能力とのバランスをとる必要があります。



- すべての注意がAndroid Nougatに集中していますか？ 開発者がどのようなイノベーションを積極的に受け入れたのか、リリースした瞬間からすでに決心したと思います。



-実際、先日発表されたNougat、Marshmallow、Android O Developer Previewは、Android Lollipopの有機的な開発であり、SDKの根本的な変更を試みることはありません。 一方で、これは良いことです。アプリケーションでOSの新しいバージョンをサポートするために、多くの作業を必要としません。 しかし一方で、Androidの古い欠陥とそのルートアーキテクチャは、毎日多くの苦痛をもたらします。



- あなたの意見では、適用されたMVVMアーキテクチャ「Model-View-ViewModel」が主導的地位を占めており、最新の技術と見なされていますか？ この方向の変化を期待していますか？ このアーキテクチャのアプローチには何をもたらすことができますか？



-アーキテクチャアプローチの主要な位置について話すことは困難です。 どこにも長所と短所があります。 市場で人気のあるほとんどのアプリケーションの外観、動作、および「内部」を見ると、主要なアプローチである「プロダクションでの大失敗」に疑いの余地はないので、MVVMは特に人気があるとは思いません。 また、Androidの世界では、iOSとは異なり、GoogleのDataBindingが少し前に安定したため、これもかなり若い方向です。



- 私の知る限り、レポートの一部はMVVMテンプレートとリアクティブ環境での「ライフ」に当てられます。 これがアプリケーションのアーキテクチャの新しい外観なのか、新しいソースで設計するための古いアプローチのレビューなのかを教えてください。



-これは私たちの元の見解であり、最良かつ最も正確であると主張するのではなく、よく知られたソリューションを組み合わせていますが、独自の成果も持っています。 私たちはこの方法でアプリケーションを実行しますが、Android SDKとアプリケーションライフサイクルの多くの制限との闘いを排除するアプローチを見つけるための絶望感をまだ残していないロシアの同僚とこれを共有したいと思います。



- 現在、開発者には大きなチャンスがあり、同時に多くの落とし穴があり、特に大規模なプロジェクトではすべてを追跡することはできません。 この状況では、依存性注入（DI）を使用するのが優れたソリューションであると思いませんか？



-はい、DIはリファクタリングを高速化する素晴らしい方法です。 しかし、これをいわば脳のDIに変えて、すべてを無差別に注入する必要はありません。 私たちは90秒で、自宅のプロジェクトでもDIが大好きで、Dagger2で実装することがよくあります。 また、プロジェクトでこのアプローチを使用することを全員に推奨します。 ステパンとの私のものを含め、テストされているアーキテクチャと適切なコードサポートを可能にすることについて、Mobiusについてのいくつかのレポートがあります。

ステパン・ゴンチャロフ



- ステファン、こんにちは！ あなた自身と今の仕事について少し教えてください。



-いいね。 私自身についてごく簡単に言えば、2008年、公式リリース1.0より前のAndroid開発に興味を持ち始めました。2009年前半には、Androidマーケットに既にアプリケーションがありました（Google Playがそれと呼ばれたことを覚えている人はほとんどいません）。他のいくつかのサードパーティ市場。 広告から、私は約1,000ドルを稼ぎ、このアプリケーションが実際のデバイスでどのように機能するかを見るために最初のAndroidフォンを購入しました。 同時に、すでにお気づきのように、ゲームの作成に夢中になりました。ゲームの1つは、その後AndroidマーケットとApp Storeに登場しました。 その後、カスタムアプリケーションの開発に切り替えて、これに4年間を費やした後、製品開発に挑戦したいと思いました。 そのため、シンガポールのスタートアップの1つでチームリーダーになり、その後90秒で、最先端の技術を使用して独自のアプリケーションをゼロから作成することができました。



- 2011年に、読者を確実に惹きつけるゲームの作成に関する一連の記事をリリースしました。 今日、これらの材料の需要がありますか？



-残念ながら、私はゲーム開発に関する記事を長い間書いていません。 約1年前、彼はMediumに 、主にKotlinとAndroidに関する小さな記事を追加し始めました。



- 私たちの会話のトピックに直接関連する質問をしたかった。 少し前に、Kotlin に関する通信を見ましたが、「私にとって幸運なことに、KotlinはJavaの世界ではスイスアーミーナイフです」（私にとっては、KotlinはJavaの世界ではスイスのナイフです）という答えが本当に気に入りました。 Kotlinはワンストップ開発ソリューションだと本当に思いますか？



-はい、Kotlinはその欠点にもかかわらず、生産性を大幅に向上させ、開発者の機能を拡張するユニークなツールであり、ほとんどすべてのJavaライブラリで作業できます。

データクラスとデフォルトパラメータの優れた使用例、開発者の生産性を向上させる方法の例。 そして、リストでの作業を示すもう1つのオプション。



-Kotlinの使用によりコードの品質、可読性が向上し、場合によってはnullの不要なチェックを軽減できる状況について教えてください。



-実際、KotlinとJavaの最も顕著な違いの1つはNullable型です。 nullチェックを明示的に行う必要があるため、コードの品質は大幅に向上しますが、NPEを完全に削除することを保証するものではありません。 私のお気に入りのKotlin機能の1つは、標準Javaコレクションの拡張機能セットです。これにより、データの操作が大幅に簡素化され、コード量が削減されます。これにより、その後の変更が容易になり、潜在的なエラーの数が削減されます。 そして、もちろん、インターフェイスのメソッドのデフォルト実装と組み合わせてデリゲートを使用すると、コンポジションを100％使用できます。これは、大量のコードを記述し、維持する必要があるため、Javaで達成するのは非常に困難です。



-Kotlinコードの簡潔さとコンパクトさについて、ネットワーク上で多くの資料が見つかりました。これですべてが明らかになりました。 しかし、そのようなコードのサポートを忘れないでください。 たとえばJavaの開発者は、Kotlinのプロジェクトを常にすばやく理解できますか？ Java / Scalaなどでは珍しい言語の問題/微妙な点はありますか？



-私は2012年にKotlinに興味を持ち始めましたが、経験豊富なJava開発者がKotlinに対処することは難しくないと思います。デニスは最近これを経験しました。 初心者のJava開発者にとっては、すべてがはるかに複雑です。私の観点からすると、Kotlinを学び始めるにはJavaの深い知識が必要です。

多くの場合、Kotlinの機能に関する知識は、特定のライブラリが期待どおりに機能しない理由を理解するのに役立ちます。 微妙な点のうち、多くの一般的なライブラリ（Dagger2、RxBindingsなど）で使用されている注釈処理には注意する価値がありますが、それでも多少の不都合が生じます。たとえば、Dagger2は、@ JvmSuppressWildcardsアノテーションのないジェネリック型では機能しません。 また、初心者はインライン関数とコルーチンのスタックトレースの異常な形式に対処する必要があります。



-Androidの別の開発トピックに触れたいと思います。 かなり長い間 、アプリケーションでRxJavaを使用する利点について聞いたり読んだりしています。 このライブラリについてのあなたの意見を聞くのは興味深いです。 私は、ReactiveXについて常にユニークに肯定的ではない資料をよく見ます。



-はい、あなたは正しいです、このアプローチまたはそのアプローチから常に嫌悪者とファンの両方が存在するでしょう、rxJavaも例外ではありません。 私は他のグループと同様に、アプローチ自体に長所と短所の両方があることを理解していますが、私は2番目のグループにいます。 しかし、すべての欠点にもかかわらず、少なくとも各Android開発者のトピックを完全に理解することをお勧めします。



-RxAndroidおよびRxBindingの形式でのRxJavaの「継続」についてどう思いますか？ プロジェクトでこの機能を使用していますか？



-RxKotlinへの貢献者およびRxDataBindingsの作成者として、同様のRxライブラリに大きなメリットがあると思います。 そして、各プロジェクトでRxAndroidを使用しています。

RxDataBindingsは、DataBindingsライブラリとrxJava2のクラスの相互作用に必要なコードの量を大幅に削減できます。



- 会話をまとめると、今後2年間のAndroid開発の傾向についての意見を聞きたいと思います。 反応性、コンパクトさ、およびコード削減に向けた動きは続きますか？



-確かに、AndroidコミュニティはすでにJavaの冗長性に飽き飽きしており、特にAppleがSwiftを宣伝するのを見てきた。 今後数年間で、Android開発用の代替JVM言語（Kotlin、Scala、Groovy、Ceylon、Clojure）の使用が急速に増加すると思われます。 Rxは、モバイル開発の主要なアプローチとしての地位を引き続き強化する可能性があります。今後数年で、コルーチンの概念に基づいたアプローチとしか競争できなくなりますが、残念ながら、Kotlinだけがそれらをサポートし、その後実験モードになります。
