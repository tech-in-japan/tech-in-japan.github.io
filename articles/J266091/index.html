Paypal経由の支払いをウェブアプリケーションに統合します

この記事では、1回限りの支払いと、WebアプリケーションでPaypalを使用したサブスクリプション支払いの統合について説明します。 これらの例はPHPで実装されていますが、原則として、他のテクノロジーを使用しても問題なく同じことができます。 この方法は、単純さと柔軟性の間の妥協案として選択されます。 これは、トピックをすばやく理解し、Paypalを介して支払いをプロジェクトに統合するのに役立つガイドを作成する試みです。



この記事は、主にこのシステムを使用したことがない人に焦点を当てています。 Paypalの専門家がここで自分自身で新しいものを見つけることはまずありません。 しかし、おそらく、彼らはこの方法の短所を指摘したり、どのように異なる方法で実装できるのかをアドバイスするでしょう。



アカウント作成



このスキームを実装するには、ビジネスアカウントが必要です。 PayPal Payments Standardで十分です。

リンクに従ってアカウントを作成します。



サンドボックスアカウントの作成



Paypal Sandboxを使用して、アプリケーションをテストします。 2つのサンドボックスアカウントが必要です。 バイヤーアカウント（バイヤー）とセラーアカウント（ファシリテーター）。 まず、両方のサンドボックスアカウントにパスワードを設定する必要があります。 これを行うには、開発者セクションの paypalウェブサイトにアクセスしてください 。 ログインして、 ダッシュボードに移動します。 左側のメニューには、[サンドボックス]セクションの[アカウント]タブがあります。 ここでは、2つのサンドボックスアカウント（BuyerとFacilitator）を見ることができます。







プロファイルをクリックし、表示されたモーダルウィンドウで[パスワードの変更]をクリックし、パスワードを保存します。

両方のアカウントにパスワードを設定します。 その後、 Paypal Sandbox Webサイトにアクセスして、ログインしてみてください。



Paypalのセットアップ



次に、資金を受け取るPaypal Facilitatorアカウントを設定する必要があります。 Sandbox Webサイトにアクセスし、ファシリテーターアカウントを使用してログインし、プロファイル設定にアクセスします。 プロファイルメニューを開き、販売ツールのアイテムを選択します。







[オンライン販売]セクションで、[ウェブサイトの設定]を選択し、[更新]をクリックします。 ここで、ユーザーのリダイレクトを有効にできます。 支払いが完了すると、ユーザーはデフォルトで指定されたURLにリダイレクトされます。 ただし、ユーザーを別のURLにリダイレクトすることもできます（以下を参照）。







また、Paypalインスタント支払い通知をアクティブにする必要があります。 これを行うには、「支払いの取得とリスクの管理」セクションで、「即時支払い通知」項目を選択し、「更新」をクリックします。







IPN設定で、IPNリスナーが機能するURLを指定します。 このURLはグローバルにアクセス可能である必要があります 操作の通知を受け取ります。







メッセージ配信をオンにして保存します。 これでアカウントのセットアップが完了しました。 支払いの設定を直接開始できます。



一回払い



まず、1回限りの支払いを行います。 これはおそらく最も一般的な使用例です。 ユーザーは、製品または1回限りのサービスを購入したいだけです。 まあ、私はペイパルの設定で何も変更する必要がもうないことを望みます。 商品と価格のリストはアプリケーションのデータベースに保存されますが、必要に応じて変更できます。 1回限りの支払いには、 支払いボタン（PayPal Payments Standard）を使用します。



データ構造



商品のリストは、アプリケーションのデータベースに保存されます。 製品はいつでも追加および削除および編集できます。 ここでは最も単純な構造が示されており、すべての情報が1つのテーブルに保存されています。



ただし、タスクを複雑にすることができます。 たとえば、注文した商品の数量に応じて価格を変更したり、曜日と時間に応じて価格を変更したりします。



または、注文に多くの異なる製品を含めます。

お支払い方法



まず、注文フォームを作成します。 アプリケーションでフォームを生成し、注文の主なパラメーター（製品名、価格、数量）を指定します。



ここで、価格、名前、数量などを指定できます。 カスタムフィールドは、任意のデータを転送できるという点で便利です。 ここでは、製品ID、ユーザーID、およびその他の情報を渡します。 今後の支払い処理のためにこれらのデータが必要になります。

複数のパラメーターを渡す必要がある場合は、jsonまたはシリアル化を使用できます。 または、on0、on1、os0、os1の形式の追加フィールドを使用できます。 個人的に、私はこれをチェックしませんでした、私はここで情報を見つけました。



以下はフォームの例です： 



<?php $payNowButtonUrl = 'https://www.sandbox.paypal.com/cgi-bin/websc'; $userId = 315 // id   $receiverEmail = 'xxx-facilitator@yandex.ru'; //email  (   paypal ) $productId = 1; $itemName = 'Product 1'; //   $amount = '1.0'; //  ( 1 .) $quantity = 3; //  $returnUrl = 'http://your-site.com/single_payment?status=paymentSuccess'; $customData = ['user_id' => $userId, 'product_id' => $productId]; ?> <form action="<?php echo $payNowButtonUrl; ?>" method="post"> <input type="hidden" name="cmd" value="_xclick"> <input type="hidden" name="business" value="<?php echo $receiverEmail; ?>"> <input id="paypalItemName" type="hidden" name="item_name" value="<?php echo $itemName; ?>"> <input id="paypalQuantity" type="hidden" name="quantity" value="<?php echo $quantity; ?>"> <input id="paypalAmmount" type="hidden" name="amount" value="<?php echo $amount; ?>"> <input type="hidden" name="no_shipping" value="1"> <input type="hidden" name="return" value="<?php echo $returnUrl; ?>"> <input type="hidden" name="custom" value="<?php echo json_encode($customData);?>"> <input type="hidden" name="currency_code" value="USD"> <input type="hidden" name="lc" value="US"> <input type="hidden" name="bn" value="PP-BuyNowBF"> <button type="submit"> Pay Now </button> </form>

ここで、ユーザーは、PayPalアカウントまたは銀行カードを使用して注文の代金を支払うことができます。 その後、ユーザーは当社のWebサイト（リダイレクトパラメーター）にリダイレクトされ、そこで支払いが処理されていることを通知できます。

支払い処理



まず、カスタムフィールドの値を取得する必要があります。カスタムフィールドには、注文ID、ユーザーID、またはその他のもの（アプリケーションのロジックによって異なります）を渡します。 したがって、データベースからユーザー/注文情報を取得できます。 トランザクションIDも取得する必要があります。



Paypalは、同じ取引の確認を数回送信できます。 したがって、確認する必要があり、トランザクションが処理されなかった場合は処理します。 トランザクションが既に処理されている場合、何もしません。



支払いを検証します。 すべてが正常であれば、支払い情報をデータベースに保存し、さらにアクションを実行できます（ユーザーに「プレミアム」ステータスを割り当て、注文ステータスを「支払い済み」など）。 支払いが検証されていない場合は、理由を確認してユーザーに連絡する必要があります。 さらなる操作、特に支払いのキャンセルは、手動で実行されます。

支払い検証



支払いの検証は、アプリケーションのビジネスロジックに大きく依存しています。 特定の条件が追加される場合があります。 たとえば、ユーザーは15ユニットの商品を支払いましたが、利用できるのは10個だけです。そのような注文を見逃すことはできません。



ただし、フォーム生成の段階でそのようなことを確認することは理にかなっています。 不正行為を防ぐためではなく、支払いの検証が必要です（たとえば、ユーザーが支払いの形で商品の量を手動で増やしたが、価格は変更しなかった場合）。



とにかくチェックアウトする価値のあるものがいくつかあります。

  • 支払いとデータベースで価格の対応を確認します
  • 合計コストが0に等しくないことを確認します（前の段落でこのケースを扱っているため、妄想）
  • 正しい受取人が指定されていることを確認してください
  • 支払い状況を確認する
  • 支払い通貨を確認する

サブスクリプション



次に、サブスクリプションの実装を検討します。 原則は、1回限りの支払いと同じです。 定期的な支払いのみ。 したがって、それらの実装はやや複雑です。



いくつかの料金プランが利用できます。たとえば、無料-無料、プロ-ユーザーあたり月額5ドル、プレミアム-ユーザーあたり月額10ドルです。

ユーザーは、未使用期間の払い戻しで退会できます。 また、ユーザーは、別の料金プランへの切り替え、ユーザー数の変更など、サブスクリプション条件を変更できます。



無料購読の場合、PayPalはまったく必要ないことは明らかです。 おそらく、この料金プランは、アプリケーションにユーザーが登録されるとすぐに自動的にアクティブ化されるはずです。 このスキームは、一部のSaaSシステムの一般的な使用方法を示しているという点で優れています。 外出先では、Paypalを使用してこれを実装する方法はあまり明確ではありません。



サブスクリプションを使用するには、追加のテーブルが必要です。

IPN



IPNの場合、すべては基本的に1回限りの支払いと同じです。 確かに、我々はより多くのリクエストを受け取るでしょう より多くのイベントを処理する必要があります：サブスクリプションの作成、サブスクリプションの支払い、サブスクリプションの解除など。 前と同様に、各リクエストの信頼性を確認してから処理する必要があります。



サブスクリプション検証



ここでのすべては、1回限りの支払いよりも少し複雑です。 支払いだけでなく、サブスクリプションの作成、サブスクリプションのキャンセル、サブスクリプションの変更も検証する必要があります。 アプリケーションのロジックに応じて、おそらく他の何か。 たとえば、Proの料金プランには最大100人のユーザーが必要です。 またはそのような何か。 繰り返しますが、フォームを作成する段階で、これらすべてを考慮に入れることができます。



この場合に正確に確認する必要があるもの：



  • キャンセルの場合は、サブスクリプションが存在することを確認してください
  • サブスクリプションで支払うには、それを確認してください

    • 価格は0ではありません
    • サブスクリプションのサイズに等しい支払い額
    • 受信者が正しく指定されている
