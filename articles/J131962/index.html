はじめに



私は、多くの人と同じように、私たちの生活の中には、過度に複雑で混乱しているものがたくさんあり、単純化が必要だと思います。 携帯電話会社から残高がゼロに近づいているというテキストをもう一度受け取ったときに、同じ考えが思い浮かびました。もう一度、毒に行き、承認を通過し、クリックして支払いを繰り返し、確認し、指示しました実行を待っている支払いパスワードは、毒を残しました。 どういうわけか多くのアクション、あなたは思いませんか？ そのように思えたし、実際、それは非常に退屈なプロセスであり、自分に合ったより簡単な方法を見つける必要がある。



プロセスが私を捕まえたので、私は今確信が持てないことに注意したい-これらすべてに時間を節約できたのか、それともそのままにしておくほうが簡単だったのか？ しかし、私はそのような人物です。あらゆる種類の興味深い活動にすぐに夢中になります。そのため、実装プロセス自体が最終結果よりも大きな喜びをもたらしました。



1.サービスとシステムの説明



LISA -Motiv Companyの加入者が個人アカウントの現在の状態を管理し、インターネット経由でサービスを使用できるようにするサービス。



毒 -電子決済システム。 多くの店舗や企業が毒のサポートを実装しています。これにより、商品やサービスの支払いを簡単かつ簡単に行うことができます。 「動機」は、毒を通してバランスを補充する機会が実現された企業の1つにすぎません。



Twitterは、Webインターフェース、SMS、インスタントメッセージングツール、またはサードパーティのクライアントプログラムを使用して、ユーザーが短いテキストノート（最大140文字）を送信できるようにするシステムです。 Twitterの特徴は、投稿されたメッセージが一般に公開されていることです。これにより、ブログと同様になります。

2.プロジェクトの目的



毒ウォレットから携帯電話の残高を補充するシステムの実装は、プロジェクトの主なタスクです。残高の補充は、特定の周期性なしで行われる必要があります。 モバイルアカウントは、必要な場合にのみ補充する必要があります。

  1. 個人アカウントのステータスを取得するための資金;

3.2。 毒を扱うためのツール


毒を扱うには以下が必要です：
  1. ドキュメントを調べます。
  2. アプリケーションを登録（client_idを取得）;
  3. 一時トークンを取得します（1分未満）。
  4. 認証トークンを取得します（1年間）。


図3.4-登録済みアプリケーションに関する情報。



受け取った認証トークンは1店舗（Motiv Company）に制限されており、現在のアカウント情報を見ることができる週に1,000ルーブルの制限があります。 

 scope=payment.to-pattern("1122").limit(7,1000) account-info
これで、自由にメソッドを使用できます。
  • account-info-アカウントの状態に関する情報。この方法は使用していません。
  • リクエスト支払い-支払いのリクエスト;
  • プロセス支払い-支払い確認;
支払いを行うには、支払いテンプレートを作成します。

pattern_id = 1122＆PROPERTY1 = 908＆PROPERTY2 = 9089XXX＆sum =％s

すべてが正しい場合、ポイズンのリクエスト-支払いリクエストを送信すると、ステータスパラメータの値「success」と目的のrequest_idを持つjson文字列を取得します。「success」以外のすべては、失敗と同等であり、さまざまな要因に依存します。たとえば、Yandex側の技術的な問題、店舗側の技術的な問題などです。

すべてが正常な場合は、request_idを含むプロセス支払いリクエストを送信し、支払いを確認します。 

 $yad = json_decode($result, 1); if ( $yad['status'] == 'success' ) { $pay_fields = sprintf($process_fields, $yad['request_id']); curl_setopt($ci, CURLOPT_URL, YM_PROC_URL); curl_setopt($ci, CURLOPT_POSTFIELDS, $pay_fields); $result_pay = curl_exec($ci); curl_close($ci); return json_decode($result_pay, 1); }
支払いが成功または失敗した場合、statusパラメーターを含むjson文字列も取得します。 ステータスパラメータが「成功」の場合、支払い番号とアカウントステータス情報が返されます。

4.仕事と論理



このアプリケーションの主なアイデアは、ボードをタイムリーにすることです。毎日50ルーブルまたは100ルーブルでバランスを補充する必要はありません。この日に電話を使用することはできず、Poisonに実装できるからです。 すべてのコンポーネントがすでに存在し、動作しているため、作業のロジック次第でした。



不要なチェックを省略すると、すべて理解できる形式になります。

1.いつ補充するかを知るために現在の残高を確認します。現在、口座にあるお金の量に常に注意する必要があります。

2.バランスの基準値（100ルーブル）と電流の差を決定します。

2.1。 差は設定値（20ルーブル）よりも大きくなります。

2.1.1。 支払いをします。

2.1.2。 データベースに新しい値を書き込んでいます。

2.1.3。 私はTwitterで書いています。

2.2。 差は設定値（20ルーブル）未満です：

2.2.1。 現在の値をデータベースに記録された最後の値と比較します。差が正の場合、残高は補充されています。

2.2.1.1。 データベースに新しい値を書き込んでいます。

2.2.1.2。 私はTwitterで、外部の誰かが残高を補充したと書いています。

2.2.2。 差が負で、指定された値（-5ルーブル）より小さい場合、現在の値をデータベースに最後に記録された値と比較します。

2.2.2.1。 データベースに新しい値を書き込んでいます。

2.2.2.2。 Twitterでバランスが変わったと書いています。



考慮すべきいくつかの注意事項があります。

-Motiv会社は、10ルーブル未満の支払いを受け付けません-これは、口座の現在の状態と参照設定値との差額が少なくとも10ルーブルでなければならないことを意味します。

-多くの場合、アカウントの補充についてTwitterに投稿しないために、20ルーブルが使用されました。

-上記の5ルーブルの差が理由で取られたため、以前はまったく制限がなく、アプリケーションは10分ごとに残高が7コペックまたはそれ以下変化したことをTwitterで通知しました-それは私を動かしましたあるリミッターを入れて、最初は1ルーブルでしたが、これでは十分ではありませんでした。リミッターは5ルーブルで、もう何も気にしません。

-現在の値とリファレンスの差が100ルーブルを超える場合、何かが間違っていると疑い始め、同じ原理に従って、最初に100ルーブル、残りのお金の部分のバランスを補充します。

結論



この記事を書いている時点で、答えが見つからないいくつかの質問に出会いました。

request-payment-requestへの応答で値「success」が来るものの、長い間、なぜポイズンを介してモバイルアカウントに転送できなかったのか理解できませんでした。 ドキュメントの例は次のとおりです。

pattern_id = 2904＆電話プレフィックス= 921＆電話番号= 9538XXX＆sum = 300.00

電話プレフィックスフィールドと電話番号フィールドの内容は、各ストアには必要ありません。Motivの場合、これらのパラメーターはPROPERTY1およびPROPERTY2という名前です。MTSまたはMegafonの場合、これらのフィールドには他の名前を付けることもできます。 動機の分野は、API Poisonクラブのモデレーターによって促されました。 これらのフィールドの値を知っていると、支払い時に単純に名前からパラメーターをストアパラメーターに再割り当てできないのはなぜですか？



そしてもう1つの質問は、最初の質問に直接関係しています。 何らかの理由で無効なフィールド名を含むリクエストを送信すると、「成功」という値が返され、request_idパラメーターも返されました。これにより、支払いを確認する必要があります。 もちろん、そのような支払いは行われませんでしたが、なぜステータスが成功したのですか？ 開発者のマニュアルには、「成功」は成功した実行であると明確に記載されています。



誰かがこれらの質問に答えてくれたら嬉しいです。
