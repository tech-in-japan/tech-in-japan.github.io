はじめに



このプロジェクトは、サーカスマキシマスのスタンドの下にあるローマの重要な考古学的な場所であるミトレウムの研究に関連しています。 この作業は、エンジニア、建築家、工業デザイナーを卒業した科学技術大学であるミラノ工科大学（PoliMi）と共同で実施されました。 大学に加えて、文化遺産市庁（Sovrintendenza Capitolina）およびトリエステ大学もこの作業に参加しました。



Mitreum内に設置されたセンサーの測定値を収集して、二酸化炭素の濃度、振動レベル、温度、湿度をリモートで監視しました。



Mitreumは今日まで良好な状態で保存されています。 1930年代にローマのオペラハウスが修復されたときに初めて発見されました。 建物は紀元前2世紀のものです。 e。 彼女はおそらく何度も再建したでしょう。 3世紀の広告 e。 下層階は聖域であるミトレウムのために確保されていました。





サーカスマクシムスのローマ（ローマ）



このプロジェクトでは、次のIoTデバイスとテクノロジーが使用されました。



  • Mitreumにインストールされている特別なセンサー：加速度計（デバイス上）、温度計（MCP9700A）、湿度計（808H5V5）、二酸化炭素センサー（TGS4161）、土壌温度計（PT1000）、およびフォトレジスター（VT43N2）。
  • Libelium Waspmotesは、センサーのコンピューティングモジュールとして機能します。
  • MonoソフトウェアとC＃コンソールアプリケーションをフィールドゲートウェイとして使用してRaspbian OSを実行しているRaspberry Pi 2。
  • センサーとゲートウェイを接続するための低速ワイヤレスネットワーク用の802.15.4標準。
  • データを受信するためのAzure IoT Hub：30分ごとに1つのメッセージ。
  • データベースとPower BIにデータを転送するAzure Stream Analytics。
  • データストレージ用のAzure SQL Database。
  • Power BI Embedded。
  • Azure App Serviceのパブリックデータ視覚化機能。


問題の声明



このプロジェクトの主な目的は、この建築記念碑内に設置された特別なセンサーを使用して、ミトレウムの現在の状態を遠隔監視することです。 その中には、加速度計、二酸化炭素、温度および湿度センサーがあります。 データはフィールドゲートウェイに送信され、クラウドインフラストラクチャに送信されて、通常のWebページに視覚的な形式で公開されます。





ウェブサイトのデータの視覚化



PoliMiにとって、Mitreumモニタリングは、遺跡からデータを収集するための参照システムを作成する最初のプロジェクトでした。 将来的には、このプロジェクト用に構築されたさまざまなタイプのセンサー、ゲートウェイ、および単一のクラウドアーキテクチャを使用するため、このプロジェクトではIoTテクノロジが使用され、将来の開発に役立つ可能性があります。



このプロジェクトの主な目標は次のとおりです。



  • さまざまな考古学的な場所に適用できるスケーラブルなソリューションを見つけてください。
  • クラウドインフラストラクチャを、ローカルにインストールされたゲートウェイおよびセンサーと統合します。 このプロジェクトのフレームワークで生じた互換性の問題を解決します。
  • すべてのユーザーがどこからでも監視データを表示できるようにソリューションを見つけます。


センサーとゲートウェイ



まず、タスクをより正確に決定し、プロジェクトの現在の状態を理解するために、Skypeでいくつかの会議を開催しました。 クラウドアーキテクチャの作成を開始したとき、PoliMiの従業員は既にセンサーを選択してMiteriumにインストールしていました。





Mitreumのセンサーの場所



センサーは、定期的な間隔（ミリ秒単位で選択した周波数に応じて）で、小さなデータセットをフィールドゲートウェイに送信し、そこでローカルで処理されます。 次の表からわかるように、各メッセージのサイズは非常に小さいため、30分ごとに、データはすべてのセンサー読み取り値を含む.txt形式の単一ファイルにグループ化されます。 C＃コンソールアプリケーションは結果のファイルを処理し、その後、構造化データがAzure IoT Hubに送信されます。





Mitreumのセンサーの特性



このデータに基づいて、Azure価格計算ツールを使用して、展開前にクラウドインフラストラクチャ全体の月額コストを計算することができました。 このイニシアチブは、クライアントから高く評価されており、プロジェクトの作業を続けることができました。 もちろん、他の利点も役割を果たしました。ソリューションの簡単な展開、スケーラビリティ、移植、高度な資格を持つマイクロソフトの従業員によるプロジェクトのサポートです。

プロジェクト開発の課題



このプロジェクトのフレームワークで最も深刻なタスクの1つは、データ分析でした。 IoT Hubに送信するための正しいメッセージを作成するには、テキストファイルに収集されたすべてのセンサーの読み取り値を分離する必要がありました。 PoliMi大学の専門家は、この問題を解決する上で私たちを大いに助けてくれました。 メッセージの構造を明確に理解して定義したので、データを正しく視覚化することができました。



もう1つの大きな課題は、クラウドインフラストラクチャをセンサーとゲートウェイと統合することでした。

いくつかのSkype会議、対面会議、リモート開発、ディスカッションなど、さまざまな形式の通信を使用してクライアントと協力しました。 合計で、設計、開発、テストを含むプロジェクトのすべての段階で約10日間を費やしました。 この間、システムの実装を整理し、包括的なソリューションを作成しました。

ソリューションとその段階



前述のように、4種類のセンサーが、以下のソリューション図に示されている頻度でフィールドゲートウェイにデータを送信します。



ネットワークゲートウェイはAzure IoT Hubに接続し、30分ごとに約1 KBのデータを送信します。 プロトコルとしてMQTTを選択しました。これは、MQTTが広く普及しており、このプロジェクトのノードモデルで使用するのに便利だからです。



Azure Stream Analyticsはデータストリームを処理し、測定結果を適切なSQLデータベーステーブルに送信します。



WebアプリケーションはPower BIレポートを発行して、エンドユーザー（PoliMiの専門家）が視覚化されたデータを受信できるようにします。

分析用データ



データを収集するために、10個のLibelium Waspmoteセンサーが使用されました 。 彼らのタイプは、考古学者と施設の修復を担当するスタッフによって選ばれました。 Waspmoteコンピューティングデバイスには、以下を含むLibeliumセンサーモデルを使用しました。



  • 加速度計
  • 温度計
  • 湿度計
  • 二酸化炭素センサー
  • フォト抵抗器（VT43N2）
  • 土壌温度計（PT1000）


測定値は、Raspbianオペレーティングシステムを実行し、フィールドゲートウェイとして機能するRaspbery Pi 2デバイスに転送されます。



OSの選択は、Raspbian OSでの広範な経験があり、Windows 10 IoT Baseや他のオペレーティングシステムをほとんど使用しなかったクライアントの好みによって決定されました。 デバイスは、センサーから受信したデータを処理します。 たとえば、ノード3および4（加速度計）から収集された値の場合、高速フーリエ変換（FFT）が実行され、クラウドに送信されるメッセージに追加されます。



ゲートウェイで起動されたPythonアプリケーションは、すべてのデータが収集された30分ごとにテキストファイルを生成します。 Pythonプログラミング言語を選択することは、努力を最小限に抑え、将来さまざまなプラットフォームでソリューションの複製を簡素化するための最良の方法であることが判明しました。 このファイルは、IoT Hubにデータを送信するために使用されます。 以下の画像は、フィールドゲートウェイ上のC＃コンソールアプリケーションによって解析される生データを含むサンプル.txtファイルを示しています。





生テキストファイル



前処理



必要なデータを送信するには、フィールドゲートウェイで実行されているコンソールアプリケーションでテキストファイルを変換する必要があります。 この段階で、コンソールアプリケーションを開発するための言語を選択するという疑問が生じました。 IoT Hub SDKは、PythonとC＃の両方で使用できます。 クライアントと両方のオプションについて話し合った後、この言語をよく知っていたため、C＃でアプリケーションを開発することにしました。 次に、Raspberry PiデバイスにMonoソフトウェアをインストールして、アプリケーションを起動しました。



このアプリケーションは、テキストファイルから抽出されたすべてのセンサーの測定値を含む、IoT Hubに送信する準備ができたJSON配列の測定値を作成します。 最終テスト中に、.txt形式の非常に大きなファイルを使用し、IoT Hubに送信されるメッセージのサイズに関連するいくつかの問題を発見しました。



テキストファイルが256 KBより大きい場合は、小さなファイルに分割する必要があります。 この要件は、デバイスからクラウドに送信されるデータの最大パケットサイズに対するIoT Hubの制限のために発生しました。




テキストデータ解析スキーム



ノード3および4の読み取り値を解析するツールの作成は、送信されるデータの性質により複雑でした。 タイムスタンプ、ノードID、バッテリーレベル、座標、パケット番号、FFTの一部である20個の番号のセットをフィールドゲートウェイに送信する必要がありました。 特定の座標に対する1つのセンサーからのデータ送信は、70ミリ秒以内のタイムスタンプを持つ7つのパケットで構成されます。 これらのパケットを識別し、7つのパケットのすべてのFFT値（JSON配列として表され、文字列として一緒に格納される）を含むテーブルに1行を作成する必要がありました。



テレメトリー送信



前述したように、接続の性質により、まずセンサーの読み取り値を含むテキストファイルを処理する必要がありました。 リアルタイム監視はこのプロジェクトの目標ではなかったため、最終モデルはわずかに遅れて機能します。 最終的に、フィールドゲートウェイによって作成されたテキストファイルを拒否しないことにしました。これは、ログファイルとしてクライアントに役立つ可能性があるためです。 また、開発したパーサーを引き続き使用しました。



Raspberry Piは不透明なフィールドゲートウェイとして機能するため、C＃コンソールアプリケーションをIoT Hubの一意のデバイスとして登録しました。 つまり、接続するセンサーの数に関係なく、ゲートウェイは必要なすべての操作を実行し、データを集約して、IoT Hubに送信するメッセージを作成します。 さらに、このアプローチにより、開発されたアーキテクチャを他の考古学的サイトに再利用でき、スケーラブルで簡単に移植できるソリューションが作成されます。



安全性



Azure IoT Hubでは、各デバイスを名前と対称キーで登録できます。 これは、各デバイスが独自の接続文字列を受け取ることを意味します。



ハッキングされた場合、Azureポータルのコントロールパネルを使用してデバイスとのデータ交換を停止することにより、対応するデバイスをオフにすることができます。



IoT Hubは、送信データを保護するためのターンキー効果的なソリューションを提供します。 プロジェクトの作業を開始する前に、安全要件は提出されませんでしたが、この機能はクライアントによって広く認識されていました。

データの可視化



プロジェクトの最終段階は視覚化です。 データ分析を簡素化するために、ASP.NET WebアプリケーションにPower BIレポートを埋め込むことにしました。 このため、このプロジェクトの2つの重要な目標を実現することができました。クライアントが簡単にパーソナライズできるダイアグラムにデータを表示すること、そして最も重要なこととして、センサー読み取り値への公開アクセスを開くことです。



Webアプリケーションで使用可能なチャート、グラフ、およびデータフィルターは、クライアントとの緊密なコラボレーションで作成されたため、さまざまなセンサー、フィルターなどのさまざまなタブという結果が期待に応えました。
