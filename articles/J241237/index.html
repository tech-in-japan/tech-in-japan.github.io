systemdを使用したCentOS 7サービス管理のチートシート

Systemdは、Linuxオペレーティングシステム用のシステムおよびサービスマネージャーです。 開発中、SysV init初期化スクリプトを使用して後方に設計し、起動時のシステムサービスの並列起動、オンデマンドでのデーモンのアクティブ化、システム状態スナップショットのサポート、依存関係に基づくサービス管理ロジックなどの便利な機能を提供しようとしました。 CentOS 7では、systemdがUpstartをデフォルトの初期化システムとして置き換えます。



この記事では、CentOS 7ユーザー向けにsystemdでサービスを管理するプロセスを検討します。この知識は、Fedoraで長い間使用され、Ubuntu 14.10およびDebian 8で計画されているため、他のディストリビューションで役立ちます。







この記事を読みながら、InfoboxからクラシックVPSとクラウドVPSで systemdを試すことができます。 最新のテクノロジーを使用して効率的な作業を行えるように、最新のOSのサポートをタイムリーに追加するよう努めています。 記事を書くというまさにそのアイデアは、CentOS 7でのサービスの使用に関する別のユーザーの質問から生まれました。



はじめに



Systemdは、systemdユニットの概念をもたらします。 ユニットは、次のいずれかのディレクトリにある構成ファイルで表されます。

  • / usr / lib / systemd / system / -インストールされたRPMパッケージのユニット。
  • /実行/ systemd / system / -ランタイムで作成されたユニット。 このディレクトリは、パッケージからユニットがインストールされているディレクトリよりも上位です。
  • / etc / systemd / system / -システム管理者が作成および管理するユニット。 このディレクトリは、ランタイムで作成されたユニットのディレクトリよりも上位です。


ユニットには、システムサービス、リスニングソケット、システム状態の保存されたスナップショット、および初期化システムに関連する他のオブジェクトに関する情報が含まれます。



systemdユニットのタイプ：

  • .service-システムサービス
  • .target -systemdユニットグループ
  • .automoun t-ファイルシステムのマウントポイント
  • .device-カーネルによって認識されるデバイスファイル
  • .mount-ファイルシステムのマウントポイント
  • .path-ファイルシステム内のファイルまたはディレクトリ
  • .scope-外部で作成されたプロセス
  • .slice-システムプロセスを管理する階層的に編成されたユニットのグループ
  • .snapshot -systemdマネージャーの保存状態
  • .socket-プロセス間通信ソケット
  • .swap-デバイスまたはスワップファイルのスワップ（スワップファイル）
  • .timer -systemdタイマー




CentOS 7のコアシステム機能



  • ソケットベースのアクティベーション 。 起動時に、systemdはすべてのシステムサービスのソケットをリッスンし、このタイプのアクティベーションをサポートし、サービス開始直後にこれらのサービスにソケットを転送します。 これにより、systemdはサービスを並行して開始できるだけでなく、サービスが利用できないときに送信されたメッセージを失うことなくサービスを再開することもできます。 対応するソケットは使用可能なままで、すべてのメッセージがキューに入れられます。
  • D-Busベースのアクティベーション 。 D –バスを使用したプロセス間通信用のシステムサービスは、クライアントアプリケーションがそれらに接続しようとしたときにオンデマンドで開始できます。
  • デバイスベースのアクティベーション 。 デバイスベースのアクティベーションをサポートするシステムサービスは、特定のタイプの機器が接続されたとき、または利用可能になったときに開始できます。
  • パスベースのアクティベーション 。 フォルダーまたはディレクトリの状態が変更された場合、システムサービスはこのタイプのアクティベーションをサポートできます。
  • 全身状態のスナップショット 。 システムは、すべてのユニットの状態を保存し、システムの以前の状態を復元できます。
  • マウントとマウントポイントを管理します。 Systemdは、マウントとマウントポイントを監視および管理します。
  • Aggressive Concurrency Systemdは、ソケットベースのアクティベーションを使用しているため、システムサービスを並行して実行します。 オンデマンドアクティベーションをサポートするサービスと組み合わせて、パラレルアクティベーションはシステムの起動時間を大幅に短縮します。
  • ユニットアクティベーションのトランザクションロジック 。 ユニットをアクティブ化および非アクティブ化する前に、systemdは依存関係を計算し、一時的なトランザクションを作成して、このトランザクションの整合性をチェックします。 トランザクションが完了していない場合、systemdはエラーメッセージを生成する前に、トランザクションを自動的に修正し、不要なジョブを削除しようとします。
  • SysV初期化との下位互換性 。 SystemDは、Linux Standard Base（LSB）仕様で説明されているように、systemdへの切り替えを簡素化するSysV初期化スクリプトを完全にサポートします。




サービス管理



CentOSの以前のバージョンでは、SysVまたはUpstartが使用されていました。 初期化スクリプトは/etc/rc.d/init.d/ディレクトリにありました。 このようなスクリプトは通常Bashで記述されており、管理者がサービスとデーモンのステータスを制御できます。 CentOS 7では、初期化スクリプトはサービスユニットに置き換えられました。



使用方法は、サービス.serviceユニットは初期化スクリプトに似ています。 システムサービスを表示、開始、停止、再起動、有効化または無効化するには、 systemctlコマンドを使用します。 serviceおよびchkconfigコマンドはシステムにまだ含まれていますが、互換性の理由からのみです。





systemctlを使用する場合、ファイル拡張子の指定はオプションです。



以下は、主要なsystemctlコマンドです。

  • systemctl start name.service-サービスを開始します。
  • systemctl stop name.service-サービスを停止します
  • systemctl restart name.service-サービスの再起動
  • systemctl try-restart name.service-サービスが実行されている場合のみサービスを再起動します
  • systemctl reload name.service-サービス構成の再読み込み
  • systemctl status name.service-サービスのステータスの詳細表示でサービスが開始されているかどうかを確認します
  • systemctl is-active name.service-サービスが開始されたかどうかを単純な答えで確認します：アクティブまたは非アクティブ
  • systemctl list-units --type service --all-すべてのサービスのステータスを表示します
  • systemctl enable name.service-サービスを有効にします（システムの起動時に開始できます）
  • systemctl disable name.service-サービスを非アクティブ化します
  • systemctl reenable name.service-サービスを無効にし、すぐに有効にします
  • systemctlは–有効なname.service-サービスがアクティブ化されているかどうかを確認します
  • systemctl list-unit-files --type service-すべてのサービスを表示し、どのサービスがアクティブになっているかを確認します
  • systemctl mask name.service-サービスファイルを/ dev / nullによるシンボリックリンクに置き換え、ユニットにsystemdからアクセスできないようにします
  • systemctl unmask name.service-サービスファイルを返し、systemdでユニットを使用できるようにします




Systemdのターゲットでの作業



SysV initまたはUpstartを備えた以前のバージョンのCentOSには、0〜6の番号が付けられた操作固有のモードを表すランレベルの定義済みセットが含まれていました。CentOS7では、ランレベルの概念はsystemdの目標に置き換えられました。



systemd .targetターゲットファイルは、依存関係チェーンを通じて他のsystemdユニットをグループ化するためのものです。 たとえば、グラフィカルセッションの開始に使用されるgraphical.targetユニットは、GNOME Display Manager（ gdm.service ）およびAccounts Service（ accounts – daemon.service ）システムサービスを開始し、 multi-user.targetをアクティブにします 。 次に、 multi-user.targetは、Network Manager（ NetworkManager.service ）やD-Bus（ dbus.service ）などの他のシステムサービスを起動し、他のbasic.targetターゲットユニットをアクティブにします。



CentOS 7には、ランレベルの標準セットに似た事前定義された目標があります。 互換性の理由から、これらにはこれらの目的のエイリアスもあり、SysVランレベルに直接表示されます。



  • poweroff.target（runlevel0.target） -システムのシャットダウンとシャットダウン
  • rescue.target（runlevel1.target） -回復シェルのセットアップ
  • multi–user.target（runlevel2.target、runlevel3.target、runlevel4.target） -非グラフィカルマルチユーザーシステムのセットアップ
  • graphic.target（runlevel5.target） -グラフィカルなマルチユーザーシステムのセットアップ
  • reboot.target（runlevel6.target） -システムをシャットダウンして再起動します


runlevelおよびtelinitコマンドは引き続き使用できますが、互換性の理由でシステムに残されています。 systemctlを使用して、システムの目標を変更または構成することをお勧めします。



デフォルトで使用されるターゲットユニットを確認するには、次のコマンドが役立ちます。systemctl get – default 。



ロードされたすべてのターゲットユニットを表示するには、 systemctl list-units --type targetコマンドを 使用し、すべてのターゲットユニットを表示するには、systemctl list-units --type target --allコマンドを使用します。



systemctl set-default name.targetコマンドは、デフォルトのターゲットを変更するのに役立ちます 。



現在のターゲットを変更するには： systemctl isolate name.target 。 チームはターゲットユニットとそのすべての依存関係を起動し、他のすべてのユニットを直ちに停止します。



システムのシャットダウンと再起動



CentOS 7では、systemctlはかなりの数の電源管理コマンドを置き換えます。 以前のコマンドは互換性のために保存されていますが、systemctlを使用することをお勧めします。

systemctl halt-システムを停止します

systemctl poweroff-システムをシャットダウンします

systemctl reboot-システムを再起動します



リモートマシンでsystemdを管理する



Systemdを使用すると、SSH経由でリモートマシンを制御できます。 制御するには、次のコマンドを使用します。

systemctl --host user_name @ host_name command 、ここでuser_nameはユーザー名、host_nameはリモート制御されているホストの名前、commandは実行されているsystemdコマンドです。



典型的なsystemd .service



このセクションは、systemdから迅速にサービス管理サポートを行う必要がある場合に役立ちます。 .serviceファイルのすべてのパラメーターの詳細情報は、systemdドキュメントの対応するセクションにあります。 



[Unit] Description=Daemon to detect crashing apps After=syslog.target [Service] ExecStart=/usr/sbin/abrtd Type=forking [Install] WantedBy=multi-user.target


[Unit]セクションを見てみましょう。 サービスに関する一般情報が含まれています。 このようなセクションは、サービスユニットだけでなく、他のユニットにもあります（たとえば、デバイス、マウントポイントなどを管理する場合）。 この例では、サービスの説明を示し、Syslogの後にデーモンを開始する必要があることを示しています。



次のセクション[サービス]には、サービスに関する情報が直接含まれています。 使用されるパラメータExecStartは、サービスの実行可能ファイルを示します。 Typeには、サービスが起動の完了をsystemdに通知する方法を示します。



最後の[Install]セクションには、サービスを開始する目的に関する情報が含まれています。 この場合、 multi-user.targetターゲットがアクティブになったときにサービスを開始する必要があると言います。



これは、最小の作業systemdサービスファイルです。 独自のテストを作成した後、/ etc / systemd / system / service_name.serviceにコピーします。 systemctl daemon-reloadコマンドを実行します。 Systemdはサービスについて学習し、開始できます。
