Visual StudioのHANAデータベースプロジェクトのプラグイン

HANAをデータベースとして使用し、TFSをソース管理として使用するASP .NET MVCプロジェクトでSCRUMに取り組んでいます。 データベースレベルでは、主にビュー（計算、属性、DBビュー）とストアドプロシージャを使用して、サーバーへのトランザクション要求を実行します。



各リリースの終了後、「このリリースで何が変更されましたか？」または「誰がどのような変更を加えましたか？」という質問がありました。この点について、「各変更後にTFSのオブジェクトの状態を追跡しないのはなぜですか？」



その結果、Visual Studio（VS）でデータベースプロジェクトを使用し、データベースにある変更をインポートできるプラグインを作成することにしました。 そこで、このアプリケーションを作成するというアイデアが生まれました。



簡単なものから始めて、Microsoftがどのようなプロトタイプを持っているかを見てみましょう。 例として、Visual StudioのMS-SQLサーバーとデータベースプロジェクトを取り上げ、それらが持つすべての機能を検討します。



  1. MS-SQLで独自のスキーマ（既存のスキーマを使用）を作成し、Visual Studio（DB VS）のデータベースプロジェクトにインポートできます。
  2. データベースでオブジェクトを作成（変更）し、変更をデータベースに変換できます。
  3. ソース管理（私の場合はTFS）で変更を追跡できます。
  4. したがって、開発中に発生したすべての変更とその作成者を追跡できます。


これらすべての機能を維持することは非常に難しいことをすぐに言わなければなりません。特にHANAにはグラフィックビューなどの種類のオブジェクトがあるためです。 VSにインポートします）。



私は簡単な方法で行った。 各機会を個別に検討してください。



HANAでスキーマを作成/使用し、DB VSにインポートします



2つのタスクがあります。



  1. DB VSでプロジェクトを最初から作成し、DBにインポートします。
  2. DBの既存のプロジェクト（スキーマ）を使用して、DB VSにインポートします。


私は2番目だけを選択しました。まず、既存のプロジェクトが既にあります。次に、VSでプロジェクトを作成するためにフィードバックが必要です（これについては少し後で説明します）。これは非常に難しいタスクです。



したがって、すべてのオブジェクトをDBからDB VSにインポートするブリッジを作成する必要があります。



データベースでオブジェクトを作成（変更）し、変更をデータベースに変換する



これは、前のタスクとほぼ同じです。 つまり、既存のスキーマがある場合、オブジェクトをDB VSに持ってくるだけです。 ただし、ここで検討する価値があります。変更された（削除された）オブジェクトがプロジェクトに既に存在する場合、DB VSで編集（削除）する必要があります。 これはTFSの変更を追跡するために必要なので、このタスクも実装しました。



ソース管理の変更を追跡する機能（私の場合はTFS）



変更がDB VSに表示されたら、TFSでそれらを追跡する必要があります。 これを行うには、VSで何らかの方法で、変更されたファイルをチェックインの項目のリストに含まれるようにマークする必要があります。 ここでは、TFSに「.Newフレームワークライブラリ」を使用する必要がありました。 このプラグインの作成中にVisual Studio 2012を使用しましたが、VSの他の（より高い）バージョンでは、プラグインがTFSを変更できるように、Microsoftの必要な（追加の）フレームワークを使用する必要があります。



DB VSプラグイン



プラグインは4つの主要部分で構成されています。



  1. 認証ウィンドウ。
  2. HANAオブジェクトトランスフォーマー。
  3. オブジェクト変更マネージャー。
  4. TFSマネージャー


認証ウィンドウ



このウィンドウでは、プラグインに必要なすべての情報を入力します。 検証が成功すると、システムはデータを記憶し、次のログイン後にデータを使用します。 何らかの理由でプラグインが1つのサーバーに「到達」できなかった場合（たとえば、DBまたはTFSで権限を変更した後）、承認ウィンドウが再び開き、権限を編集できます。







次の2種類のデータを入力する必要があります。



  1. HANAに必要なすべての情報、つまり：サーバーパス、ユーザー名、パスワード、およびスキーマ名。
  2. プロジェクトとTFSに関する情報：[プロジェクトパス]フィールドで、プラグインで使用するDB VSファイルの場所を表示できます。 また、ユーザー名、パスワード、TFSサーバーURLなど、TFS資格情報に関する情報。


[保存]の横にあるチェックボックスをオンにすると、サーバーへの接続に成功するとデータが保存されます。



HANAオブジェクトトランスフォーマー



これはプラグインの最も難しい部分です。 ここでは、システム辞書を使用して、システム内にあるすべてのオブジェクトに関する情報を取得します。 現在の実装でサポートされているオブジェクト：



  1. テーブル。
  2. 制約（外部キー、主キー、一意）。
  3. テーブルの種類。
  4. シーケンス。
  5. 機能
  6. ストアドプロシージャ。
  7. 同義語
  8. ビュー（属性ビュー、計算ビュー、データベースビュー）。


HANAでは、多くのプロジェクトが他のオブジェクト（RESTサービス（XSJS）など）を使用することも知っていますが、この場合は無視しました。 ビューを除くすべてのオブジェクトについて、システム辞書を使用してデータを取得します。 例として、テーブルに関するデータの取得を検討します。



これを行うには、次のクエリを使用してスキーマ内のテーブルの名前を取得します。 



SELECT SCHEMA_NAME,TABLE_NAME,TABLE_OID,COMMENTS,FIXED_PART_SIZE,IS_LOGGED,IS_SYSTEM_TABLE,IS_COLUMN_TABLE,TABLE_TYPE,IS_INSERT_ONLY,IS_TENANT_SHARED_DATA,IS_TENANT_SHARED_METADATA,SESSION_TYPE,IS_TEMPORARY,TEMPORARY_TABLE_TYPE,COMMIT_ACTION,IS_USER_DEFINED_TYPE,HAS_PRIMARY_KEY,PARTITION_SPEC,USES_EXTKEY,AUTO_MERGE_ON,USES_DIMFN_CACHE,IS_PUBLIC,AUTO_OPTIMIZE_COMPRESSION_ON,COMPRESSED_EXTKEY,HAS_TEXT_FIELDS,HAS_TEXT_FIELDS,USES_QUEUE_TABLE,IS_PRELOAD,IS_PARTIAL_PRELOAD,UNLOAD_PRIORITY,HAS_SCHEMA_FLEXIBILITY,IS_REPLICA FROM TABLES WHERE SCHEMA_NAME = CURRENT_SCHEMA AND IS_USER_DEFINED_TYPE = 'FALSE';

最も困難なことは、グラフィック属性ビューに関する情報を取得することでした。 ビューの構造に関するデータを取得する直接の機会はありません。 しかし、考えてみると、HANA Studioはデータベース内の特定のメタデータの助けを借りてこのグラフィックビューを表示します。 また、このメタデータを正確に取得しようとしましたが、各ビューのDBに保存されているXMLファイルを見つけました。 したがって、この質問を解決しました。つまり、ビューに関する情報を取得し、その変更を追跡するために、そのXMLをDBVSに保存する必要があります。 XML構造には、受信パラメーター、送信パラメーター、現在のビューで使用されるすべてのテーブルの名前、データ型、フィルター、結合などのデータがあります。



計算/データベースビューも主にSQLスクリプトとして作成および保存されるため、特別な問題はありませんでした。



オブジェクト変更マネージャー（OCM）



データベースから情報を受け取ったら、DB VSの現在および以前の状態にどのような変更があるかを確認する必要があります。 OCMは変更同期リンクであり、3つの段階でDB VSを同期します。



  1. スキーマに追加された新しいオブジェクトを確認します。
  2. 既存の各オブジェクトの変更を確認します。
  3. スキーマ内のオブジェクトの削除を確認します。


最初の場合：OCMはスクリプトを生成し、ファイルを追加し、DB VSでそれを見つけ、追加のファイルを更新します。



2番目の場合：OCMは、生成されたスクリプトをDB VSの既存のファイルと比較し、オブジェクトが変更された場合、既存のファイルを変更します。



3番目の場合：OCMは、スキーマの現在のバージョンにないオブジェクトファイルを削除します。
