仮想ディスクVirtualBoxを操作する機能

この記事では、VirtualBoxで仮想ディスクを使用する機能、さまざまな読み取り/書き込みモードの使用、スナップショットの原理と構成、入力/出力データのキャッシュ、および情報セキュリティの観点から仮想ディスクを使用するいくつかの側面について説明します。 セキュリティの例に興味がある人のために、 特別な記録モードのセクションにすぐにアンカーをジャンプできます 。



一般的な概念から始めましょう。 VirtualBoxには、データへのゲストオペレーティングシステム（OS）アクセスを提供するための3つの主要な方法があります。 このテキストは、仮想ディスクの使用に焦点を当てています。



仮想ディスクは、適切なコントローラー、IDE、SATA（AHCI）、SCSI、SASを介して接続をエミュレートすることにより、仮想ゲストOSに接続されます。



コントローラーの動作は、物理的なプロトタイプをシミュレートするようにプログラムされているため、IDEコントローラーはSATAよりも低速で実行され、より多くのプロセッサーリソースを消費します 。OSは、適切なドライバーとハードウェアサポートがないと仮想ディスクと対話しません。 たとえば、Windows Vistaより前のWindowsファミリでは、SATAが属するAdvanced Host Controller Interface（AHCI）がサポートされていないため、特に、SATAを搭載したWindows XP仮想マシンは機能しません。



仮想ディスクファイル



VirtualBoxを使用すると、さまざまな仮想ディスクファイル形式を操作できます。 独自のVDIに加えて、VMDK（VMware）、VHD（Microsoft）、Parallelsバージョン2 HDD形式（Parallels）をサポートしています。



各仮想ディスクには一意のUUIDが割り当てられます。これにより、VirtualBoxは各ディスクが1回だけ使用され、ディスクの通常のコピーをゲストOSにインポートできないようにします（これには別のクローン作成手順があります）。



仮想ディスクは固定サイズまたは動的に割り当てることができます。VirtualBoxを使用すると、ディスクのサイズやフォーマットに関係なく、ディスクにデータが含まれている場合でも、ディスクスペースのサイズを増やすことができます。 以下は、 vboxmanageユーティリティを使用してこれを行う方法の例です。 



vboxmanage modifyhd <uuid|filename> --resize <megabytes>


動的に割り当てられたイメージを選択すると、コンテナーのサイズが仮想マシンの作成時に指定された制限に達するまで、セクターがデータでいっぱいになるにつれて、コンテナーファイルは徐々に「成長」します。



コンテナのサイズを定期的に増やすプロセスでは、追加のコンピューティングリソースが関係することを考慮すると、通常、固定サイズオプションを使用する場合の記録速度は、ダイナミックディスクと比較して高速です。 それでも、ダイナミックディスクのサイズが長期的に大幅に増加しない場合、記録速度の差はほぼ横ばいになります。



仮想メディアマネージャー



VirtualBoxは、すべてのゲストオペレーティングシステムで使用されるすべての仮想メディアのレジスタを保持します。 これらはいわゆる「既知のメディア」であり、そのリスト（レジストリ）へのアクセスは、Virtual Media Managerユーティリティ（[ファイル]メニューから利用可能）を使用して取得できます。 このユーティリティは、ファイルへのフルパスやファイルが接続されている仮想マシンなど、各仮想ディスクに関する詳細情報を表示します。 レジストリからの情報は、組み込みの「削除」削除機能を使用して削除できます。











個々のイメージは、「リリース」機能を使用して、アタッチ先の仮想マシンから「デタッチ」できます。



画像の固定を解除すると、1つのボタンを押して元に戻すことはできなくなります;このため、画像をハードディスクとして追加する必要があります。 同様に、スナップショット（ディスクスナップショット）は「アタッチ」されます。











繰り返しますが、上記のUUIDのため、ディスクイメージをコピーして添付することはできません。



スナップショット



ご存知のように、英語から翻訳されたスナップショットはスナップショットを意味します。 メカニズムの原理は単純です。 スナップショットを作成するとき、VirtualBoxは現在のイメージ（複数ある場合はイメージ）を転送し、読み取り専用モードでVMに接続し、個別の仮想ディスクを作成します。その後のすべての記録手順は新しい仮想ストレージで既に実行されます。 さらに、特定のセクターの変更のみが記録されます。つまり、10GBディスクのスナップショットを作成する場合、新しいスナップショットははるかに小さくなり、セクターがいっぱいになると徐々にサイズが大きくなります。











1つの仮想マシンのスナップショットがより多く使用されるほど、ディスク読み取り操作を実行するためにより多くのコンピューティングリソースが使用されると仮定することは論理的です。 実際、スナップショットが2つある場合、最初にVirtualBoxはsnapshot2イメージにセクターがあるかどうかを確認し、ない場合、システムはsnapshot1に変わり、そこで何も見つからない場合、メインディスクにアクセスします。 それでも、セクターテーブル全体がメモリに常に存在するため、負荷は重要ではなく、エンドユーザーにはほとんど認識されません。



スナップショットが削除されると、作成後に行われたすべての変更が以前に「凍結された」イメージと「マージ」されることを覚えておく価値があります。



入出力キャッシュ



パフォーマンスのトピックに触れると、キャッシングに言及するのが適切です。 VirtualBoxは、ホストOSによってキャッシュされている通常のファイルと同様に、最初はイメージファイルで機能します。 これは、速度を上げることを目標に、ばかげて行われます。 ゲストOSが書き込み操作を実行すると、操作はホストOSによってキャッシュされ、操作が正常に完了したというメッセージがすぐにゲストOSに送信されますが、操作自体はゲストOSによって非同期に処理されます。 このアプローチは常に正当とは限りません。 ディスクイメージファイルはボリュームが増加する傾向があり、手順全体が逆の効果を生み出し始めます-ゲストおよびホストオペレーティングシステムの側に二重キャッシュがあり、操作の速度が低下します。



リソースの費用に加えて、キャッシュのもう1つの欠如は信頼性の欠如です。 たとえば、ホストOSが既にゲストに録音が成功したことを通知し、非同期録音プロセスがまだ完了していないときに突然の停電があった場合。 そのようなシナリオは確かにデータの損失につながります。



キャッシュの無効化は次のとおりです。 



 vboxmanage storagectl "VM name" --name <controllername> --hostiocache off


帯域幅



VirtualBoxを使用すると、1つ以上の仮想ディスクの帯域幅を制限できます。



グループ「Limit」を作成し、20 Mb / sの制限を設定します 



 vboxmanage bandwidthctl "VM name" add Limit --type disk --limit 20M


目的のドライブをグループに追加します。 



 vboxmanage storageattach "VM name" --storagectl "SATA" --port 0 --device 0 --type hdd --medium disk1.vdi --bandwidthgroup Limit vboxmanage storageattach "VM name" --storagectl "SATA" --port 1 --device 0 --type hdd --medium disk2.vdi --bandwidthgroup Limit


両方のドライブの合計帯域幅は20 MB /秒を超えません。 この制限は、仮想マシンをオフにすることなくいつでも変更できます。 



 vboxmanage bandwidthctl "VM name" set Limit --limit 100M




画像を記録するための特別なモード



VirtualBoxがサポートする仮想ディスクイメージごとに、形式に関係なく、データを記録するときの動作を、仮想マシン内での操作の結果であるかスナップショットであるかを判断できます。 このようなモードは「非標準」と呼ばれますが、デフォルトではすべてのディスクイメージが「通常」モードで機能します。 モードを「通常」から「非標準」に切り替えるには、前述のVirtual Media Managerまたはコンソールユーティリティvboxmanageを使用できます。 



 vboxmanage modifyhd <disk image>.vdi settype immutable


「通常の」記録モードでは、ゲストOSは物理ディスクから制限なしで読み書きできますAディスクのスナップショットを作成するとき、VirtualBoxはすべての変更が記録される別のファイルを作成します。



「ライトスルー」モードでは、スナップショット機能は機能しません。



「共有可能」モードの操作は、以前のモードの一種です。 ここでも、「スナップショット」を操作する方法はありませんが、同じディスクイメージの複数の同時に動作する仮想マシンを使用する可能性があります（クラスタリングシナリオ）。



名前は似ているが動作原理が異なる「マルチアタッチ」モードでは、複数の仮想マシンに1つのディスクイメージを使用することもできますが、このモードでは、各仮想マシンは独自の独立した「スナップショット」を使用し、1つのVMで行われた変更は他の仮想マシンでは使用できません



読み取り専用モードは、主にCD / DVDイメージの操作に使用されます。 読み取り専用です。



注意すべきモードは「 不変 」と呼ばれます。 名前が示すように、不変の画像は時間とともに変化しません。 不変ディスクへの変更は、仮想マシンが実行されている限り正確に関係します。 仮想マシンをシャットダウンすると、すべての変更が消えます。 ディスクを不変モードにする前に、最初に「通常の」ディスクを作成し、できればインターネットに接続せずに、最適な状態にシステムをインストールして構成し、ゲストシステムがディスクを「取り外し」て不変モードにする準備ができてからでなければなりません。



作業シナリオの1つは、2つのディスクが使用されるスキームです。1つはシステム自体が配置されている不変モードで、もう1つは通常モードまたはライトスルーモードです。 一見すると、これは完全に安全で理解可能な作業シナリオです。「新しい」システムがロードされるたびに。 しかし、すべてがそれほど透明ではなく、いくつかのニュアンスがあります。



まず、不変ディスクには1つの重要な例外があります。 実行中にディスクイメージが取得された仮想マシン （いわゆるオンラインスナップショット） に接続さ れている場合、それらは "ゼロアウト"しません 。 これは、たとえば、ユーザーが不変ディスクを作成し、作業中に仮想マシンをシャットダウンせずに「スナップショット」を作成し、その後、「スナップショット」から開始すると、システム内の後続のすべての操作とアクションに元に戻せない効果があることを意味します。に。 すべてのアクションは、事実上スナップショットで発生します。



主な目標がすべての開始時の「新しいシステム」である場合、スナップショットの使用は控えることをお勧めします。



第二に、個々のイメージの上記の「ゼロ化」は、VirtualBox環境自体によって仮想マシンの有効化/無効化コマンドが送信された場合にのみ発生し、ゲストOS内では発生しません。 たとえば、標準的な方法（[スタート]メニュー、システムの再起動）を使用してWindowsゲストOSを再起動した場合、不変ディスクはリセットされません。



最後に、最後の最も重要なこと-仮想マシン内で発生するすべての変更は物理ディスクに保存され、仮想マシンが再起動されるまでそこに残ります。



現在のコンテナが不変モードに設定された後、VirtualBoxはこのコンテナの使用を中止し、実際、ディスクは読み取り専用モードになります。 すべての書き込み操作は個別のイメージにリダイレクトされ、仮想マシンが動作を開始するたびに、この新しい「個別の」イメージは「リセット」されます。 実際には、ハードディスク上に一時的な「スナップショット」が作成されます。これは、対応する仮想マシンのスナップショットフォルダーにあり、その中ですべての作業が行われます。 仮想マシンがシャットダウンした後、上記の一時的なスクリーンショットは変更されません。



簡単な例を見てみましょう。



ボブは仮想マシンを作成し、OSを構成し、ディスクを不変モードにしました。 ボブは定期的に仮想マシンを使用して、アリスとプライベートに通信します。 起動ごとに、以前の通信、テキスト、ビデオ、または写真のログを含まない「新鮮な」システムがロードされます。 通信を終了すると、ボブは落ち着いて仮想マシンをオフにし、スリープ状態に入ります。

また、各OSの起動前に、Bobがディスクモードが「不変」に設定されていることを確認するとします。



Eveは、仮想マシンがインストールされているコンピューターにアクセスできます。 彼女は、対応する仮想マシンのディレクトリ内のスナップショットフォルダーに移動するだけで十分で、必要な「スナップショット」があります。



すべての通信を確認するためにEvaがしなければならないこと、およびボブがOS内で実行するすべてのアクションの結果は、仮想マシンを起動する前にディスクを「通常」モードにしてスナップショットをアタッチすることです。 さらに、Evaはそのような「スナップショット」のバックアップを毎日作成できます。主なことは、これはボブが仮想マシンを再び起動する前に行われることです。



この状況でのボブの解決策は、作業の完了後、スナップショットフォルダーの内容全体を手動で削除することです。 ドライブがどのモードで動作し、それが望ましいかを常に確認する必要があるという事実は言うまでもなく、GUIのいくつかの要素をブロックすることもできます。 



 vboxmanage setextradata global GUI/Customizations OPTION[,OPTION...]


公平に言えば、ディスクをあるモードから別のモードに転送するには、同じParallelsを最初のバージョンからスーパーユーザーのパスワードが必要であり、作業の完了後すぐに一時的な「スナップショット」が削除されます。
