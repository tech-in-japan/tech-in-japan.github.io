Clever Geek Handbook

QtCreator：Linux 64からLinux 32、win32、win64、Mac OS XへのQtクロスコンパイル。 upx、usb、dmgなど





Qtライブラリを使用すると、真のクロスプラットフォームアプリケーションを作成できます。 作成したコードは、多くのオペレーティングシステム用にコンパイルできます。 しかし、問題はまさに「コンパイル」という言葉にあります。 ターゲットシステムの下で再起動し、カスタマイズされた開発環境をインストールし、その中にライブラリー動物園を設定する必要があることが理解されています。 クロスコンパイルの保存-実行されるプラットフォーム以外のプラットフォーム用の実行可能コードを生成するコンパイル。



Windows 64のクロスコンパイル


通常、最も要求される問題の1つは、最初にLinux用に開発されたアプリケーションのWindowsバージョンの構築です。 この問題の解決例は、 ここまたはロシア語で見ることができます。 mkspecs構成を作成し、適切なディレクトリにQtファイルを配置する必要があります。 この場合、Qtをコンパイルする必要はありません;公式サイトからバイナリをダウンロードできます。

このアプローチにはいくつかの欠点があります。1）QtCreatorはこの方法でインストールされたライブラリについて何も知りません。 2）Windows x64用の公式Qtビルドはありません。 そして、まだ最初の問題とどうにかして戦うことができれば、コンパイルだけが2番目の問題に役立ちます...



クロスコンパイルする前に、クロスコンパイラ自体を直接配置することを忘れないでください（「mingw」という名前のパッケージマネージャーで検索されます）。 そして、公式サイトからqt-everywhereソースをダウンロードします。 解凍されたアーカイブのmkspecsディレクトリで、 win32-g ++フォルダーをwin64-x-g ++にコピーし、 qmake.confファイルの内容を調整します。 私は次のものを手に入れました：

次のパラメーターを使用してconfigureを実行しました。

./configure -xplatform win64-x-g++ CROSS_COMPILE=x86_64-w64-mingw32- -prefix /usr/local/qt4win64 -no-webkit -no-phonon -no-phonon-backend -no-script -no-scripttools -no-multimedia -no-qt3support -fast -nomake demos -nomake examples -nomake tools -device-option -little-endian -qt-zlib -qt-libpng -qt-libjpeg -openssl-linked -no-fontconfig -no-3dnow -no-ssse3 -continue





ここでは、Webkit、フォノン、マルチメディアなどのないQtの最小バージョンを収集します。 オプションの完全なリストは、コマンド./configure --helpで表示できます。



したがって、このようなアセンブリの場合、 x86_64-w64-mingw32- g ++を含むg ++パッケージ-mingw-w64-x86-64をインストールする必要があります（パッケージはubuntuに個別にインストールする必要があります）。



次にmake && sudo make installを実行します。 コンパイルの最初の段階では、ネイティブシステムコンパイラが使用され、Linuxに必要なユーティリティが収集されます。これは、既にWindowsバイナリをビルドするために使用されます。

インストール後、 / usr / local / qt4win64 / bin PE32 + DLLと、いくつかのELF 64ビットLSB実行可能ファイル（ qmake、uic、moc、rccなど）があります。 そのため、QtCreatorに便利です！



インストール後、解凍したディレクトリを削除しないでください-使用されます。

通常、QtCreatorはABIを検出しますが、再確認することをお勧めします。 また、Linuxのシステムx64 GCCは32ビットアプリケーションを生成できることにも気付くことができます。 ただし、これは32ビットバージョンのライブラリも必要であるという事実を否定するものではありません。



コンパイラーの後、Qtプロファイルを追加できます。



ただし、プロファイルを追加するときは、 qmakeでディレクトリを選択する必要があるため、以前に収集したqmake、uic、moc、rccが便利です。 プロファイルの左側にある黄色の感嘆符は警告を意味しますが、QtCreatorはそのようなQtプロファイルを使用できます。 ただし、アイコンが赤の場合、プロファイルは非アクティブです。 これは、ディレクトリ構造が正しくない場合に発生する可能性があります。 または、Qtがコンパイルされたディレクトリを削除した場合。



作成された各プロジェクトで次の設定を行う必要があります。

アクティブなプロジェクトで特定のQtプロファイルを追加するには、「プロジェクト」タブに移動します（Ctrl + 5）：





デフォルトでは、Qtシステムプロファイルのみが[ビルド構成の編集]リストにリストされます。 ただし、[追加]ボタンのリストには、すべてのQtプロファイルがアセンブリパラメーターに追加されています。



メインのビルド設定では、いくつかのコンパイラライブラリをチェックアウトする必要があります。 両方が同じオペレーティングシステムからのものであるように。



ビルドステップ「qmake」および「Build」QtCreatorセットはデフォルトで設定されています。 しかし、特別なステージ「upx」と「dmgbuild」は、プロジェクトに手動で追加しました。 「プロジェクトをビルド」ボタンをクリックするたびに、「upx」ステージが実行されます。 ただし、実行可能ファイルが変更されていない場合、 upxはファイルが既に処理されているというエラーを返します。 エラーの場合、次のステップは呼び出されません。 dmgファイルは、upxが正常に機能した場合にのみ更新されます。



upxステージが機能するには、システムにインストールする必要があります。 ただし、Linux環境で動作し、パッケージマネージャーからインストールされた場合でも、 upxはアプリケーションを圧縮できます：linux32 / 64、win32、macos32 / 64。 すべてのプロジェクトから遠く離れて、実際にupx-compressionが必要であり、このステージは例として示されています。



dmgbuildステップでは、 make_dmgスクリプトを使用しました。 彼はルート権限が必要なので、スクリプトを/ etc / sudoersファイルに追加しました



プロジェクトファイルの変更とサードパーティライブラリの使用


私のプロジェクトではlibusbを使用していますが、これはQtの一部とはほど遠いものです。 プラットフォーム固有のHIDの実装を有効にすることも必要でした。 次の行がプロジェクトファイルに追加されました。 



 macx { INCLUDEPATH += $$PWD/libusbx/ SOURCES += BootLoader/HIDAPI/mac/hid.c LIBS += -framework IOKit -framework CoreFoundation -lusb-1.0 ICON = AqPicFlash.icns } win32: { INCLUDEPATH += $$PWD/libusbx/ LIBS += -lsetupapi -lole32 SOURCES += BootLoader/HIDAPI/windows/hid.cpp RC_FILE = WinIcon.rc } win32: !win64-x-g++ { LIBS += -L$$PWD/libusbx/ -lusb-1.0-32.dll } win64-x-g++ { LIBS += -L$$PWD/libusbx/ -lusb-1.0-64.dll } unix: !macx { CONFIG += link_pkgconfig PKGCONFIG += libusb-1.0 SOURCES += BootLoader/HIDAPI/linux/hid-libusb.c }


Mac OS XおよびLinuxでは、システムlibusbにリンクし、Windowsでは、ビット深度に応じてlibusb-1.0-32.dll.aまたはlibusb-1.0-64.dll.aにリンクします。 .aファイルの名前は変更できますが、アプリケーションは引き続きlibusb-1.0.dllに依存することに注意してください 。 Linuxでは、システムユーティリティpkgconfigを介してlibusbのパラメーターを取得します。 libusbに加えて、各オペレーティングシステムに必要なシステムライブラリとアイコンを接続します。



異なるオペレーティングシステム用の結果ファイルをディレクトリに配布すると便利です。 次の方法で実行できます。 

 macx { DESTDIR = mac OBJECTS_DIR = mac MOC_DIR = mac UI_DIR = mac RCC_DIR = mac } unix: !macx { DESTDIR = linux OBJECTS_DIR = linux MOC_DIR = linux UI_DIR = linux RCC_DIR = linux } win32 { DESTDIR = windows/release OBJECTS_DIR = windows MOC_DIR = windows UI_DIR = windows RCC_DIR = windows } win64-x-g++ { DESTDIR = win64/release OBJECTS_DIR = win64 MOC_DIR = win64 UI_DIR = win64 RCC_DIR = win64 }


win64-x-g ++の目標はwin32に関連していますが、プロジェクトファイルの最後に来て、設定を上書きします。
