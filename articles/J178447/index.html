更新されたWindows Azure IaaS機能の概要

こんにちは、同僚の皆さん！



本日、4月16日に正式に発効した更新（GA-General Availability）後にWindows Azureに登場した新しい機能とメカニズムに簡単に注目したいと思います。

Windows AzureはもともとPaaSプラットフォームでした（トートロジーについてはごめんなさい）が、時間が経つにつれて進化してきました-そして今では、インフラストラクチャマネージャーであるIaaSにとって興味深いものがあります。











どこでどうして



仮想マシンのオプションとWindows Azureのオプションについて説明する前に、まずWindows AzureにVMを展開する方法について少しお話しした方がいいでしょう。



1）管理ポータルを使用します。 ここではすべてが簡単でシンプルです-Windows Azureの古きよきGUIは新しいメニューとフリルで生い茂りました-しかし、これは簡潔さとアクセシビリティにまったく影響しませんでした。







2）コマンドラインツールとPowerShellを使用する-特に複雑なスクリプトの自動化のファンのために！



3）REST API呼び出しを使用します。



VMの展開方法を選択したら、展開に必要な仮想マシンのイメージとサイズを決定する必要があります。 その後、VMの作成プロセスが開始され、その仮想ハードドライブはWindows Azure BLOBストレージに配置されます。



さて、VMのサイズについてお話ししましたので、このテーマに関する提案に精通していただければ幸いです。







ここでは、すべてが非常に明確で理解しやすいように思えます。

これに加えて、要求の厳しいプロジェクト向けに2つの追加のVMタイプもあります。



1）4コア-28 GBのRAM

2）8コア-56 GBのRAM



ただし、注意が必要な点が1つあります。永続ディスクです。

Windows AzureのVMロールと仮想マシンの違いは何ですか？ 全体的な違いは、最初のケースでは、OSドライブが永続的ではなかったことです。つまり、そのようなVMの再起動後にシステムに加えられたすべての変更がリセットされました。 仮想マシンの主な違いは、永続ディスクを使用することです。つまり、再起動を恐れません。再起動後のデータ損失を恐れることなく、そのようなVMを使い慣れた信頼できる方法で操作できます。



これは確かに素晴らしいことですが、VMを展開するためにWindows Azureで利用可能な画像のかなり豊富なリストがあるという事実にも注意を払う必要があります。







そして、 SharePointとBizTalkの画像の存在に注意してください-ここでは言語はすでにIaaSと呼ばれることはほとんどありません-既にPaaSのように見え始めているので..）。

また、カスタムの独自のOSイメージと展開用のイメージについてはどうですか？



ここではすべてが簡単です-インストールおよび構成されたOSからVHDファイルをダウンロードできます。

VMディスクとAzureのテンプレートから後続のVMをデプロイするためのイメージの違いは非常に簡単です：



1）OSは、展開コマンドのsysprep = imageを使用して構成および一般化されました。

2）OSはVHDファイルにあり、Azureにロードされます-Windows Azureドライブになります。



しかし、アクセシビリティについてはどうですか？





まあ、アクセシビリティはさらに美しくシンプルです（もっともっと！？）。



1番目のVM（非クラスター化サービス）で実行されているサービスおよびサービスのインスタンスの場合、SLAは99.9％の可用性を保証します。これは、年間8.75時間のダウンタイムにほぼ相当します。



特定のロールのインスタンスが多数あるクラスター化されたサービスの場合、SLAのこのパラメーターは99.95％です。これは、相当するダウンタイムが年間4.38時間の許容ダウンタイムであるということです。



数は魅力的であるだけでなく、SLAにまつわるものでさえあります-考えるべきことがあります-オファーは本当においしいです！
