UEFIブートローダーの構成。 世界で最も短いガイド

最新のOSの読み込みはどのように機能しますか？ システムをインストールするときに、マニュアルにownれたり何かを壊したりせずにUEFI経由でブートを設定する方法は？







私は「最も速いガイド」を約束しました。 ここにあります：







  1. ディスクにGPTパーティションテーブルを作成する
  2. 数百メガバイトのFAT32パーティションを作成します
  3. UEFIブートローダーをインターネットからダウンロードします

    （ブートローダー自体が必要です。これは1つのバイナリファイルです！）
  4. このファイルの名前を変更し、/ EFI / Boot / bootx64.efiで作成されたパーティションに配置します
  5. テキスト設定を作成し、ローダーが見たい場所に配置します

    （構成の構成と場所は、ブートローダーの特定の実装に依存します。この情報はインターネットで入手できます）
  6. 再起動後、ブートローダーメニューが表示されます

    （Windows 8または10がディスクにインストールされている場合、このマニュアルはポイント3から5に削減される可能性が非常に高いです。）

システムにブートローダーをインストールするための最も「人気のある」手順は次のとおりです。OSインストーラーは、その上に特別なセクションを作成します-ディレクトリ構造とブートローダーファイルを配置します。 その後、特別なユーティリティ（Linuxではefibootmgr、Windowsではbcdedit）を使用して、UEFIチップのファームウェアと対話し、ブートレコードを追加します。 このエントリは、ブートローダーファイル（ファイルシステムのルートから開始）へのパスと、必要に応じてパラメーターを示します。 その後、OSを起動するオプションがコンピューターの起動メニューに表示されます。 Linuxの場合、ブートローダーを完全に省くことができます。 ブートレコードは、すべてのパラメーターと共にカーネルへのパスをすぐに示します。 カーネルはEFISTUBオプション（ほとんどのディストリビューションの標準である）を使用してコンパイルする必要があります。この場合、「EFI実行可能ファイル」のヘッダーが含まれ、外部ブートローダーなしでファームウェアを実行できます。







システムの起動時、ユーザーが必要なブートレコードを選択すると、UEFIファームウェアはまずこのレコードで指定されたディスク上の特別なEFIパーティションを探し、このパーティション上のファイルシステム（必要に応じてFATまたはFAT32）にアクセスし、ブートローダーを起動します。 ブートローダーは、設定ファイルから設定を読み取り、OSをロードするか、ブートメニューを提供します。 何か気づきましたか？ はい。2つのブートメニューがあります。1つはUEFIチップファームウェアレベル、もう1つはブートローダーレベルです。 実際には、ユーザーは2番目の項目の存在を推測することすらできません。メニューに項目が1つしかない場合、Windowsブートローダーは不要な質問なしにその項目の読み込みを開始します。 Windowsの2つ目のコピーをインストールするか、単に不正に再インストールすると、このメニューで画面が表示されます。







通常、ブートレコードを管理するには、インターネット上のマニュアルでUEFIファームウェアとの対話をお勧めします。 これを行うための5つの基本的なオプションがすでにあります：Linuxでのefibootmgr、Windowsでのbcdedit、Macでのある種のソフトウェア、uefiシェルユーティリティのbcfgコマンド（UEFI、ベアメタル、OSなしで起動されます。その非常に特別な形式でコンパイルされた）と特に高品質のファームウェア-グラフィカルUEFIツールを使用して（一般的な言語で、「BIOS設定」で）。







上記のすべての「マルチブック」について、次の考えを簡単に見逃してしまう可能性があります：ユーザーは、ソフトウェアパーツの設定を変更する（たとえば、OSスタートアップパラメーターを追加する）ために、ボード上のマイクロサーキットのフラッシュメモリを上書きすることを余儀なくされます。 ここに落とし穴はありますか？ そうそう！ Windowsはラップトップからレンガを作ることができることもありますし、Linux も 多くの方法でできます 。 ファームウェアの品質は、多くの場合、望まれるものを残します-UEFI標準は、曲がって実装されるか、まったく実装されません。 論理的には、ファームウェアは結果なしですべてのefivars変数の完全な削除に耐え、それらに重要なデータを保存せず、独自にデフォルト値を復元する必要があります-ユーザーがそれらにアクセスするだけで、完全な削除の可能性はゼロにはほど遠いです。 実験の過程で、私は個人的に（幸いにも、可逆的に）Lenovoを繰り返し「ブリック」しました。「設定に移動」オプションを含むブートメニューのすべての項目が消えました。







UEFIブートレコードの操作も砂糖ではありません。 たとえば、efibootmgrユーティリティには「既存のエントリを編集する」オプションがありません。 カーネルパラメーターをわずかに変更する場合は、レコード全体を削除して、既に変更済みのレコードを再度追加します。 文字列には、二重引用符と一重引用符、およびあまり明確ではない順のスラッシュとバックスラッシュが含まれています。 最終的にこの魔法の仕事をしたとき-私はまだルートFSにあるbashスクリプトの形で保存しました： 







efibootmgr -c -L "Archlinux (debug)" -l '\EFI\archlinux\vmlinuz-linux' -u "root=/dev/mapper/vg1-lvroot rw initrd=\EFI\archlinux\initramfs-linux.img systemd.log_level=debug systemd.log_target=kmsg log_buf_len=1M enforcing=0"
