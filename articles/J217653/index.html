Moai SDK 1.5-クロスプラットフォームの2Dゲームエンジン



今日は、モバイルゲームのクロスプラットフォーム開発に1年間使用してきた、あまり知られていないゲームエンジンについてお話したいと思います。 2dの場合、それは完全に私たちに合っており、Unity3dだけがそのエディターのために唯一の競争相手になれます。 MOAI SDKへの適切な注意の欠如は明らかに高い参入しきい値に関連しています -開発者自身（Zipline Games）は、製品を「プロゲーム開発者向けモバイルプラットフォーム」と位置付けていますが、ルア。



なぜ私はMOAIがとても好きだったのですか：

  • エンジンはC ++で記述され、ゲームロジックはLuaで記述されています。 lujitのサポートがあります（iOSではインタープリターモードのみ）
  • オープンソース。 CPALライセンス-ロゴ、タイトル、著者、MOAIへのリンクを読み込み画面またはクレジットで指定する必要があります
  • クロスプラットフォーム：Windows、Mac OS X、Linux、iOS、Android、OUYA、およびemscriptenを介したhtml5の実験もあります。 開発は、Windows、Mac OS X、およびLinuxでサポートされています。 iOS用にビルドするには、Macが必要です。 理論的には、新しいプラットフォームを追加することは難しくありません。OpenGLコンテキストを作成し、入力処理メソッドを呼び出す「ホスト」を記述する必要があります
  • Lua APIは非常に低レベルです。 これはプラスとマイナスの両方です。 効果的な開発には、単に高レベルのLuaフレームワークが必要です
  • 内部では、クールなアルゴリズムとテクノロジーが使用されています。 自動バッチ処理のレンダラー（cocos2dもこれを学んだようです）。 アクションツリー-定期的に更新されるすべてのもの（アニメーション、物理）はツリーの形で提示され、親ノードは経過時間（デルタ）を子に渡します。 これにより、アニメーションの階層を構築し、それらを一度に停止および継続し、再生速度を変更できます。
  • 他の2Dエンジンのようなおなじみのシーングラフはありません。 依存関係グラフは、依存関係を処理するために使用されます。 オブジェクトには多くの属性（座標、色、シェーダーなど）があります。たとえば、あるスプライトの回転を別のスプライトのX座標にバインドするなど、オブジェクト間の依存関係を設定できます。 エンジンは変更された属性のみを計算するため、理論的には更新コストが削減されます。 実際、Maya、Nukeノードアーキテクチャ、UEのマテリアルに似ています。タイプのみが一致する場合、必要なものを必要なものにバインドします。 独自の属性を追加し、それらを処理するコールバックを設定できる特別なScriptNodeがあります
  • タイルマップを操作するためのいくつかのクラス。 六角形、ダイヤモンド、長方形メッシュ、経路探索
  • 3Dのサポートがあります-つまり、すべての変換とオブジェクトは本質的に3Dです




短所と欠陥：

  • ドキュメンテーションのギャップ、例はしばしば動作不能。 小さなコミュニティ
  • アセンブリには、Cmakeが使用されます。 UnityやCoronのように「...の下にアセンブル」するボタンはありません。多くの場合、コンパイラとリンカーのエラーに対処する必要があります
  • Lua apiは完全ではありません。 たとえば、セッターとゲッターのバランスが取れていないことがよくあります（セッターがさらにあります）。 したがって、バインダーの記述を理解し、必要に応じてそれらを展開すると便利です。
  • 編集者なし
  • 統合されたUNTZサウンドエンジンは非常に原始的です。 fmodにはバインダーがありますが、おそらく非常に時代遅れです。 誰もそれらを使用していないようです
  • どうやら、開発者はフレームワークの収益化に成功しなかった-私は断固としてそれを支払い、閉鎖させたくない、そしてMOAIクラウドはそれ自体を正当化しなかった。 ドキュメントやサンプルの整理など、あらゆる種類のゴミについて、Zipline Gamesには時間もモチベーションもありません。すべては主にコミュニティによって行われます。 これは、MOAI SDKが死んだ、という意味ではありません、Zipline Gamesは内部プロジェクトにMOAIを使用し、開発は継続します（最近、box2dまたはシマリス全体をドラッグしないように、単純な衝突を処理する独自のエンジンを開発する過程で、libtessを通じてベクタープリミティブのサポートが登場しました）




インストール（Mac OS X）



OS Xでのインストールについてのみ説明します。 Windowsシステムは手元にありません。



公式リポジトリを複製します。 

git clone https://github.com/moai/moai-dev.git


システムのホストを収集するスクリプトを実行します。 

 cd moai-dev bin/build-osx-sdl.sh


すべてが順調に進み、実行可能ファイルがrelease / osx / host-sdl / bin / moaiで利用可能になり、それへのリンクが作成されることを願っています（〜/ binがPATHに追加されます）： 

 ln -s /Users/vavius/moai-dev/release/osx/host-sdl/bin/moai ~/bin/moai


例を実行し、すべてが機能することを確認します。 

 cd samples/hello-moai moai


回転するfigovinaと歓迎のテキストが表示されます。









Luaでの書き込み



最小限の例から始めましょう-画面の中央に正方形のスプライトを描画します：







コード： 

 -- 1 MOAISim.openWindow ( "sample", 600, 240 ) local viewport = MOAIViewport.new () viewport:setSize ( 600, 240 ) viewport:setScale ( 600, 240 ) -- 2 local layer = MOAILayer.new () layer:setViewport ( viewport ) -- 3 local renderTable = { layer } MOAIGfxDevice.getFrameBuffer ():setRenderTable ( renderTable ) -- 4 local deck = MOAIGfxQuad2D.new () deck:setTexture ( "moai.png" ) deck:setRect ( -64, -64, 64, 64 ) -- 5 local prop = MOAIProp.new () prop:setDeck ( deck ) layer:insertProp ( prop )


アイテムで起こっていることの説明：

  1. ウィンドウとビューポートを作成します。 ビューポートを使用すると、ピクセルにスナップせずに論理座標で作業できます
  2. レイヤーはプロップのコンテナです（プロップは描画可能なすべてのものと呼ばれます）。 レイヤーはカメラに設定でき、現在のビューポートに入るプロップのみがレンダリングされます。 レイヤーは並べ替え（レンダリングの順序）を担当します-X、Y、Z座標を個別に、またはベクトル、優先度（整数）、および等尺性タイルゲーム用の特別な種類のISO_SORTによる多くのオプションがあります。 各レイヤーには空間クエリを最適化するためのMOAIPartitionコンテナーがあります-ヒットテストとレイキャスト、マルチレベルグリッドが内部で使用されます
  3. レンダーテーブル-フレームバッファに順番に描画されるレンダー可能なオブジェクトのリストを設定します。 レンダリング時のレイヤーは、追加されたすべての小道具を描画します。 テーブルにはネストされたテーブルが含まれる場合があり、シーンマネージャーを作成するのに非常に便利です。
  4. デッキ-視覚部分を定義するオブジェクト。 ジオメトリ（この場合、四角形-2つの三角形）、UV座標、テクスチャとシェーダーへのリンクを保存します
  5. プロップ-画面に描かれるオブジェクトの集合画像。 これは、原則として、必ずしも1つの独立したスプライトである必要はありません。また、インストールされているデッキに応じて、3Dメッシュとタイルカードが存在することもあります。


一見単純なタスクのためにスプライトを描画するために多くの線を傷つけていることにすぐに気づきます。 ほとんどの場合、これは1行で行われますが、ここでは少なくとも6つ必要です。 したがって、エンジンによって提供されるAPIは「そのまま」使用されることはなく、多くはLuaでラッパーを作成します。 MOAI SDKのC ++部分には、テクスチャキャッシュ、アトラスからのスプライトのロード、それらの間のシーンおよびトランジションマネージャー、あらゆる種類のボタンおよびその他のネバネバした要素など、多くの身近なものが欠けています。 これらはすべて、Luaでの実装が提案されています。 ほとんどの操作はシーンが初期化されたときにのみ実行されるため、これが全体的なパフォーマンスに大きな影響を与えることはほとんどありません。 もちろん、余分なテーブルを作成したり、古いオブジェクトを使用したり、できることをキャッシュしたりするのではなく、賢明にコードを記述してアプローチする必要があります。



完全な自由



四角形のクワッドは必要ないが、台形が必要で、setRectの代わりにsetQuadを使用するとします。 

 local deck = MOAIGfxQuad2D.new () deck:setTexture ( "moai.png" ) deck:setQuad ( -64, 64, 64, 64, 100, -64, -100, -64 ) --


出てくるものは次のとおりです。









背景全体を写真でカバーします。 テクスチャにGL_REPEATを使用できますが、2の累乗の倍数のサイズでのみ機能し、アトラスの画像を使用することはできません。 したがって、MOAIGridクラスを使用します。 

 local deck = MOAIGfxQuad2D.new () deck:setTexture ( "moai.png" ) deck:setRect ( -0.5, -0.5, 0.5, 0.5 ) local grid = MOAIGrid.new () grid:initRectGrid ( 1, 1, 128, 128 ) grid:fill ( 1 ) grid:setRepeat ( true, true ) local prop = MOAIProp.new () prop:setDeck ( deck ) prop:setGrid ( grid ) layer:insertProp ( prop )


MOAIGridはタイルを操作するために必要です。ここでは、128x128サイズの単一のタイルからマップを初期化します。 次に、fillメソッドでインデックス1を指定します。 たとえば、MOAIGfxQuadDeck2Dでは、単一のテクスチャに対して頂点とUV座標の多くのペアを指定でき、スプライトアトラスを表すために使用されます。 この場合、このデッキには単一のインデックス1のみがあります。繰り返しをオンにして、レンダリングにグリッドを使用する必要があることを示します。









この時点で、アニメーション化されたスクロール背景を簡単に作成できます。 最後に1行追加します。 

 prop:moveLoc ( -128, 0, 0, 4, MOAIEaseType.LINEAR ):setMode ( MOAITimer.LOOP )


ループ内でプロップを1つのタイルの幅に移動すると、エンジン自体が欠落しているピースを置き換えて画面全体を埋めます。









アニメーションとアクションツリー



オブジェクトの移動と回転、およびアニメーション再生モードの設定： 

 local move = prop:moveLoc ( 200, 0, 0, 2 ) --  local rot = prop:moveRot ( 0, 0, 360, 2, MOAIEaseType.LINEAR ) --    Z move:setMode ( MOAITimer.PING_PONG ) -- - rot:setMode ( MOAITimer.LOOP ) --




デフォルトでは、アニメーションはアクションツリーのルートに追加されます。 ただし、グループ化することはできます。 

 local action = MOAIAction.new () action:addChild ( move ) action:addChild ( rot ) action:start () action:throttle ( 0.5 )


これで、両方のアニメーションを一度に停止および開始し、スロットルを使用して再生速度を設定できます。



アクションのシーケンスは、Luaコルーチンを通じて実装されます。 MOAIは、MOAIActionから継承したMOAICoroutineクラスを提供します。これにより、アクションツリーにコルーチンを追加できます。 blockOnAction関数は、アクションが終了するまでyieldを呼び出します。



画像を左右に移動し、極値に達すると、1つの完全な革命を起こします。 

 local function func () local distance = 200 while true do local action1 = prop:moveLoc ( distance, 0, 0, 2 ) MOAICoroutine.blockOnAction ( action1 ) local action2 = prop:moveRot ( 0, 0, 360, 2 ) MOAICoroutine.blockOnAction ( action2 ) distance = -distance end end local thread = MOAICoroutine.new () thread:run ( func )












おわりに



この記事は非常に原始的な例を扱っています-私の目標は、Lua APIのいくつかの側面、つまりその低レベルとモジュール性を示すことでした。 MOAI SDKは、私たちのために決定を下そうとせず、一般に認められている1つの方法ですべてを行うことを強制しませんが、完全な自由を残します。 もちろん、コミュニティは、テクスチャキャッシュ、GUI要素、シーンマネージャーなどを使用して、純粋なLuaにいくつかの高レベルラッパーを既に実装しています。



C ++の知識、選択したプラットフォームのアセンブリのニュアンス、および内部の何かを変更する意欲がない限り、実稼働環境でMOAI SDKを使用することはお勧めしません。 MOAI SDKを使用するほとんどすべての人が独自のフォークを持っていますが、これはメインブランチとは少し異なります。 歴史的に、これはZipline Gamesがプルリクエストをプルする時間がないという事実によるものでした。 しかし、今では一部のコミュニティメンバーが公式リポジトリにアクセスできるようになり、開発がさらに活発になりました。



開放性のおかげで、コードとリソースのライブリロードを直接デバイスに実装できました。 Unity3dのイメージで、エディターをゆっくり見ています。 ライブリロードが開始された後は速度が低下しましたが、頭で十分に開発を加速できます。 ベクターエディターでインターフェイスを収集し、コードに直接エクスポートします（宣言型、例はgist.github.com/Vavius/9868572 ）。 もちろん、これらはすべて、どんなエンジンにも、まったく問題なくcocos2d-xに、クラウンにもっと複雑に、しかし実際に取り付けることができます。 一般に、2Dゲームの場合、ココナッツからMOAIに移行し、後悔することはありません。何とかより大人で、柔軟性があり、よりクールです+コードはきれいで美しいです。
