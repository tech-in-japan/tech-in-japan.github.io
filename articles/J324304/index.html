Symfony Windows開発環境の展開

ボイラーが沸騰し、小川に汗が注がれ、開発者のチームがsymfonyプロジェクトをたゆみなく見ました。







必要なすべての環境を備えたリモートVDS、Ubuntu 16での運用。 GitLabの主なリポジトリ-GitHubの無料のアナログ。 各開発者はローカルマシン上のプロジェクトでリモートで作業し、リポジトリクローンを操作し、symfonyの組み込みWebサーバーのすべての改善をチェックします。







チームの最前線に立つと、Windowsで働いていて、作業環境の展開についてまったく知らなかった人が惹かれました。 この時点で初めて、これまでは、windowsコマンドで作業している人はいないことがわかりました。 その瞬間にのみ、緊急のタスク「Windows上でsymfonyプロジェクトを開発するための環境をデプロイするための最小限の知識と最小限の知識で」が登場しました。







Windowsに座っている開発者のために短いマニュアルを書く必要がありました。 その結果、彼らはマニュアルをHabréの記事の形式にするべきであると決定しました。











先を見て、すぐに解決策を見つけるために、バズーカからスズメを撃ち始め、そのとき初めて、どのモンスターを呼んでいるのかがわかり、もっと簡単なものを探し始めたと言います。







環境要件は何ですか



  • シンプルさとスピード-環境の開発、ソースパッケージなど 既存のチームがこれに多くの時間を費やさないように、できるだけ早く実行する必要があります
  • 私たちの理解-ソリューションは根本的に新しいものであってはならないので、その理解のために大量のマニュアルを軽くする必要がありました
  • Windows開発者にとってのシンプルさ、速度、わかりやすさ-コンソールでのLinuxシステムの使用、コンポーネントのインストールなどに精通している場合 -通常、Windowsで作業しているユーザーは、cmd.exeのさまざまな構成、ログ、およびエントリから、コンピューターがハッキングしようとしているという印象を受けることがあります。
  • 通常のIDEでプロジェクトを操作する-リモートプロジェクトに完全に固執するIDE、すべての変更をオンラインで同期するIDEなどがありますが、そのようなIDEは開発者にとって常に便利で無料で理解できるとは限りません。 、顧客のIDEを変更することは少なくとも奇妙です。 このため、追加の要件が発生します。
  • ローカルマシンでのsymfonyプロジェクトのデバッグ—ローカルマシンでのsymfonyのデバッグは、組み込みWebサーバーをphp bin / console server：runコマンドで実行することにより行われます。 これはとてもクールです localhost：8000を含む作業結果を表示できます。 デバッグモード（/app_dev.php）。 プロジェクトをリモートで開始すると、/ web / app_dev.phpで対応する変更が行われるまでデバッグモードは機能しません。
  • gitリポジトリのローカルクローンを操作します-コミット、プルなどを行います。 （コンソールからではなく）通常のIDEでよりシンプルで便利に


どこから始めましたか



Docker



はい、Dockerは非常にクールなもので、途方もないペースで人気を集めています。 彼らはbitnami-docker-symfonyのようないくつかの便利な食べ物さえ見つけました。







必要なすべての環境をdockerにデプロイできるという予感がありますが、2日間の喫煙マニュアルでは結果が得られませんでした。 誰かがドッカーを使用してsymfony環境をデプロイするためのマニュアルを作成し、それが上記で概説したすべての要件を満たしている場合、私は確かにそれをここに示します。







そもそも、このソリューションはわかりにくいものでした。既成のソリューションは要件を満たしていなかったため、適合しませんでした。







放浪者



私たちは、放浪者の家に環境を展開することができました。 しかし、人件費のプロセス全体は、一部のコンポーネントを手動でインストールする必要がないという違いがあるだけで、Webサーバーをゼロから持ち上げることに似ています。つまり、Windows開発者の理解の原則を満たしていません。







興味深い場合は、vagrantを使用してsymfonyプロジェクトをデプロイする方法に関するマニュアルを別途作成します。 しかし、すぐに言いますが、これは仮想マシンでも動作します。何らかの理由でsymfonyプロジェクトがキャッシュされなかったため、ロードに信じられないほど長い時間がかかりました。 繰り返しますが、app_dev.phpを変更しないと、デバッグモードは使用できませんでした。 実際、ローカルマシンにプロジェクトをロードするときに、仮想マシンで実行されていたWebサーバーに対して呼び出しが行われました。 そして、はい、データベースも仮想マシン上にありました...







皮肉なことに、最後に簡単な決定が下されました







Xammp



最終的にはより簡単で正確になる可能性が非常に高いため、symfonyプロジェクトの環境を通常どおりにデプロイする方法を考えていました。 環境に含まれるもの：







  • php
  • mysql（または他のデータベース、ただしmysqlで作業しています）
  • 作曲家
  • Git
  • 井手
  • symfony


1. XAMMP-php、mysqlの場合



Windows用のPHPインストーラーを探していたところ、PHP開発環境であるXAMPPに出会いました。 必要なすべてのコンポーネント（php、mysql、apache、phpmyqdminなど）が含まれ、必要なサービスを含むマウスをクリックするだけでグラフィカルインターフェイスから制御され、シンプルなインストーラーからインストールされます（ "next"、 "next"、 "next" ...）











PHPとmysqlは、既製のsymfonyプロジェクトをデバッグするために必要なものすべてです。 xamppでmysqlをオンにし、run.exeからプロジェクトディレクトリで「php bin / console server：run」を実行すると、サイトのブラウザーでlocalhost：8000のブラウザーで、app_dev.phpを変更せずにデバッグモードで開きます。







おそらく、xamppの存在は多くの住人にとってニュースではないでしょうが、私たちがハエに落ちた港湾労働者、渡り鳥などに直面したすべての原子爆弾の後、それは奇跡のように思えました。







ものは小さく、リモートリポジトリとやり取りするにはgitが必要であり、symfonyプロジェクトのコンポーネントをローカルにロードするにはcomposerが必要です。







2. Git-リポジトリと通信します。



gitを使用する場合、 Git SCMは素晴らしい選択肢です -便利なnishtyakovのセット：



git bash-* nixユーザーに馴染みのあるコマンドを実行するコンソール（オープンrsaキーの生成を含む）。

git GUI-gitを操作するためのグラフィカルインターフェイス。

目的のフォルダーを右クリックして、目的のディレクトリでコンソールを起動できるシェルインテグレーター







興味深い場合は、Windowsでのgitの完全な構成、RSAキーの作成、リモートリポジトリの複製に関するマニュアルを個別に公開できます。







3. Composer-symfonyプロジェクトのコンポーネントとライブラリをインストールします。



作曲家を置き、プロジェクトを展開します


symfonyプロジェクトには、実際のプロジェクトファイルに加えて、数千のライブラリファイルとキャッシュファイルが含まれています。 デプロイされたプロジェクトのディレクトリ全体をリモートサーバーからコピーする-1〜3時間の履歴。 そのため、デフォルトでは、すべてのコンポーネントとキャッシュは.gitignoreにあり、リポジトリのクローン作成時に移行されません。







必要なすべてのコンポーネントをデプロイするには、 composerが必要です。 ダウンロード、設定。 これで、Git Bashターミナル（および通常の退屈なWindows OSからでも）からcomposerを実行できます。







symfonyプロジェクトディレクトリでターミナルを開き、コマンドを実行します：

作曲家のインストール。







インストール中に、データベースへのアクセス、メールサーバーのセットアップなどに関連するデータを入力する必要があります。 データはすぐに指定できますが、後で指定できます。 いずれにせよ、コンポーザインストールデータベースがなければ、コンソールログの不正使用で終わりますが、これは恐ろしいことではありません。







データベースをセットアップする


コマンドラインまたはxamppに含まれているphpmyadminからデータベースを作成できます。 xamppを開き、mysql、apacheを有効にし、mysqlの横にあるadminを選択します。 Phpmyadminが開きます。 データベースとその中にユーザーを作成する方法、私は説明する必要はないと思う







新しいデータベースとユーザーのパラメーターはプロジェクトのsymfonyの設定に入力する必要があります（app / config / parameters.yml）







次に、データベースをsymfonyプロジェクトで指定された構造に合わせる必要があります。 これを行うには、Git Bashを使用するプロジェクトディレクトリで、いくつかのコマンドを順番に実行します。 







php bin/console doctrine:database:drop -- force php bin/console doctrine:database:create php bin/console doctrine:schema:update -- force php bin/console doctrine:schema:validate


これらのプロセス中にエラーが発生しなかった場合、プロジェクトの作業とデバッグを開始できます。







内部Webサーバーを起動します


Git Bashを使用してプロジェクトディレクトリで、次のコマンドを実行します。 







 php bin/console server:run


任意のお気に入りのIDEで、リモートサーバーとファイルを同期する必要なく、ローカルマシンのsymfonyプロジェクトに接続します。







お気に入りのブラウザーでlocalhost：8000を開きます-symfonyプロジェクトが開き、ローカルマシンですぐにデバッグモードでデバッグします。







すべての変更と開発をコミットし、お気に入りのGit GUIまたは上記で提案したものを使用して、それらをサーバーにプッシュします。







まとめ



symfonyの作業環境は最小限の血で設定され、ローカルマシンで実行され、Windows開発者は仕事を始めることができます。







おそらく他のソリューションがあり、同じドッカーであり、xamppを使用したソリューションには欠点はありませんが、作業に必要なすべてを、すべての初期要件に従って迅速かつ簡単に展開できました。
