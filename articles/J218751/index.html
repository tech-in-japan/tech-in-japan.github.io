PHPまたは別のチャット履歴で非同期に動作する

過去数年間でPHPが急速に発展したことに非常に満足しています。 おそらくあなたも。 愛好家がこのプラットフォームにとどまることを妨げる、常に新しい機会があります。 Hackのリリースに関する最近のニュースは何ですか。



きっとこの記事のタイトルを読んでいる人もにやにや笑って、「ムッシューは倒錯について多くを知っている！」と思うでしょう。 この言語またはその言語のクールさについての議論は決して収まることはありませんが、スタック全体を根本的に変更する前にすべての可能性を絞り出すのが好きなので、私にとっては、言語を変更するためのそれほど多くの条件を個人的には見ていないかもしれません。 最近、Tornadoでチャットを作成することに関する出版物があり、PHPを使用して同様の問題をどのように解決したかについて話したいと思いました。



背景


ある晴れた日、私はWebSocketについて知ることにしました。 この技術に興味をそそられましたが、昨日だけ登場したとは言えませんでしたが、これは多くの欠陥に悩まされていた社会学関連のチャットサービスの開始と一致しました。 これは私に競争のレースに参加する興奮を与えました。 Webソケットの使用は、まったく新しい有望なソリューションのように思えました。



接続は一定で双方向に確立され、クライアント側では、作業は4つのイベント（ onopen 、 onclose 、 onerror、そしてもちろんonmessage）の処理に削減されます。 setInterval、過剰なトラフィック、サーバー負荷によるリクエストはもうありません。



ここで、それが何であるかを理解していない人のために、少し余談します。

2000年代初頭のrunetに精通している人は、すべてが遅くなり、ぎこちなく動作するさまざまなチャットサービスを覚えているかもしれません。

AJAXは少し遅れて登場し、はるかに良くなりましたが、本質的には原則は変わっていません。 送信されるデータ量が少ないためにiframeの使用を拒否し、サーバーの負荷を少し減らすことができない場合を除き、クライアント部分はタイマーに従って一定の頻度でサーバーをポーリングしました。

実際、言及されたチャットサービスは古典的なAjaxチャットでした。



選択したアプローチには、真実とコインの裏側があります。

  • 古いブラウザでのサポートの欠如
  • 手動接続制御
  • サーバーデーモンの使用




最初の問題について本当に心配していなかった場合、対象読者は、WebSocketのサポートが長い間実装されてきた現代のコンピューターとモバイルガジェットを備えた若者でしたが、2番目の問題についてはさらに困難がありました。これについては後で説明します。

デーモンの使用には多くの機能があります。

  1. デーモンを再起動することによってのみコードを更新できます-したがって、chatllanの場合、これはある程度発生します
  2. 致命的なエラーと未処理の例外により、デーモンがクラッシュします-コードは「防弾」で記述する必要があります
  3. デーモンは専用の空きポートを使用する必要があります-これは、厳しいファイアウォールの後ろに座っている人にとって問題です
  4. ノンブロッキング機能を使用する




「常駐プログラム」が何であるか聞いたことがない人で、「run-run-run-die」の原理で動作するWebページのコードを書いただけの人は、デーモンを初めて書くときにパターンブレークを経験します。 たとえば、インスタンス化されたオブジェクトは、データベースタイプのストレージを使用せずに長時間「生存」し、情報を保存でき、デーモンへのさまざまな接続からアクセスできることがわかります。 おそらく、彼の執筆中に、関数をブロックする問題と、非同期のためのPHPシャープニングの欠如に最も深刻に遭遇する可能性があります。



一般に非同期とは何ですか？ 簡単な方法であれば、これは、 コードが互いに独立していくつかのコードを実行する 「並列化」する能力です。

UPD： alekciyは正しく述べました：

混乱する必要はありません。 非同期！=並列。 クラシックJSは非同期に実行できますが、並列には実行できません（シングルスレッドVMを使用）。 有用な読み物： JavaScriptでのタイマーの動作 。



非同期-一貫性のないコード実行の可能性。 並行性-同じコードを同時に実行する機能。




読者が少なくともJavaScriptの基本に精通していることを願っています。 ほとんどの場合、少なくとも次のように書きました： 

var myDomElement.onclick = function() { alert("I'm hit!"); }




小学校でしょ？ ページの一部の要素をクリックするためのイベントハンドラーが決定されます。 しかし、PHPで似たようなことをしようとしたらどうでしょうか。



最初の質問は、「オブジェクトのイベントを特定する場所」で発生します。 2つ目は、「特定のイベントに対してオブジェクトが常にポーリングされるようにする方法」です。 さて、このイベントがポーリングされる特定の無限のサイクルを作るとしましょう。 そして、多くの深刻な制限に直面します。 第一に、ポーリングの頻度は、システムの応答が満足できるように低くなりすぎないようにする必要があります。 また、システムの負荷に問題が生じないように、高すぎてはいけません。 第二に、いくつかのイベントがある場合、最初のハンドラーが動作するまで、もう一方のハンドラーは動作を開始しないという問題があります。 また、数千の接続を同時に処理する必要がある場合はどうでしょうか？



しかし、 ReactPHPは現場に登場し、魔法をかけます。



成分


  • サーバー側は、 Ratchetパッケージに基づいていました。これは、基本的にWebSocketを操作するためのReactPHPのアドオンです。
  • AngularJSのようなjavascriptフレームワークを使用するというアイデアがありましたが、その時点で私はできるだけ早くプロジェクトを開始したかったので、新しいフレームワークの学習は厳しいスケジュールに収まりませんでした。 そのため、最初は裸のjavascriptがあり、それでもjQueryが接続されていました。
  • レイアウトとデザインが面倒くさくなかったので、Twitter Bootstrap 3を使いました
  • ページのタイトルを点滅させたり、通知音を鳴らしたりする代わりに、HTML5通知を使用することは十分に重要だと考えました。
  • 生成されたデーモンには独自のポートが必要だったため、ファイアウォールの問題を解決するために、nginxを使用してWebSocketプロキシを設定しました。 興味のために、SSL証明書もねじ込みました




簡単な構造




サーバーパーツは、2つの非対称コードパーツで構成されています。キャッシックWebページ（インデックス、パスワード回復）とデモチャットサービスです。

メインページは、クライアントWebアプリケーションのロードとセッションの初期化のタスクを解決します。



デーモンは基本的に、MyApp \ Chatクラスの形式のRatchetパッケージのMessageComponentInterfaceインターフェースの実装です。 実装されたメソッドは、 onOpen 、 onClose 、 onError、およびonMessageイベントを処理します 。

onErrorを除く各ハンドラーは、責任の連鎖パターンです。 最も膨大なコードはonMessageで、コントローラーに分解されました。



問題と解決策




  1. 最初に直面しなければならなかったのは、致命的エラー、カスタムハンドラーのないエラー、未処理の例外がデーモンを殺すことでした。 致命的および例外がある場合、問題はテストの助けを借りてのみ解決されます。 私の残念なことに、手は時間の深刻な不足のためにテストに到達しませんでしたが、それでもそうです。 おそらくご存知のような単純なエラーは、カスタムErrorHandler +ロギングを使用するだけで解決されます。

  3. 接続の保持はおそらく重要な問題です。 有線ネットワークまたは適切なWi-Fiを介した通常の接続では、すべてが正常でした。 しかし、モバイルインターネットから電話をかけると、モバイルオペレータは永続的な接続を好まず、いくつかの条件に応じて切断することが明らかになりました。 たとえば、BSの負荷が低く、チャットで全員が沈黙している場合、30秒後に破棄される可能性があります。 そして、それは捨てられないかもしれません。 そこで、予防のために、pingコマンドをサーバーに周期的に送信してアクティビティを作成しました。 しかし、より忙しいBSであることが判明したため、これは機能しませんでした。

    一般に、アルゴリズムの実装は長い時間を要求していました。タイムアウト後に存在するユーザーの配列からのユーザーの遅延切断。 明らかに、これには非同期コードの使用が必要です。 当然、ここでは睡眠はありませんでした。 キューサーバーも含め、あらゆる種類の実装オプションを考えました。 ソリューションが見つかり、シンプルかつエレガントであることが判明しました。ReactPHPを使用すると、EventLoopにかかっているタイマーを使用できます。 次のようになります。

  4. デーモンモードでデータベースに接続すると、パフォーマンス上の理由からデータベースを開いたままにし、接続エラーによるログの目詰まりを最小限に抑えることができます。 いずれにしても、データベースへの接続を保証するために、各リクエストの前に呼び出される松葉杖メソッドをPDOラッパーに追加する必要がありました。 

     protected function checkConnection() { try { $this->dbh->query('select 1'); } catch (\Exception $e) { $this->init(); } }


    悲しいかな、私はよりエレガントな解決策を見つけることができませんでした。 特に既製のpredis-asyncパッケージがあるため、Redisを試す必要があります。
  5. 各ブラウザタブは新しい接続を生成します。 そして、どういうわけか、私はユーザーがクローンによって伝播することを許可したくありませんでした。 同じセッションでの接続を禁止する必要がありました。 この動作は、1つのセッションで任意の数のウィンドウまたはタブで同時に簡単に作業できるクラシックチャットとは異なります。




チャットでできること、他に学ぶこと


主な機能：

  • チャットデーモンは約20 MBのメモリを消費し、この数値は安定しています。 これは悪くありません。
  • 必須の登録がない場合、ユーザーはすぐにチャットに入ります。
  • 登録、承認、およびパスワードの回復。
  • プライベートセッションとプライベートメッセージを行うことができます（個別のチャネルを作成せずに）。
  • 個人のブラックリスト。
  • 社会的タイプのチャットルーレット。
  • ユーザーには見えないように、接続が切断されると再接続が行われます。
  • 接続の重複の防止;
  • 洪水制御。


悪い点：

  • 自作のまともなORMはありません。
  • セッションハンドラも自作です。
  • テストなし。
  • マルチスレッドなし。


ファイナライズの予定：

  • RedisなどのNoSQLデータベースを試してください。
  • 個別のチャンネルルーム。
  • ダウンロード可能なアバター。
  • さまざまな種類の通知を設定します。
  • ユーザーに個人的なメモを設定します。
  • プライベートチャンネルでの「今すぐ印刷」の表示。




プロジェクト開発の2か月後にどのような結論を導き出すことができますか？ PHPにはまだ可能性があります。 少なくとも、イベント指向のパラダイムを使用した作業の開始は準備されました。 しかし、悲しいかな、これまでのところ、言語は追いついて、運動の頭になろうとしていません。 ラチェットとトルネードを比較すると、能力の面ではまだ同等ではありません。 この方向での開発が積極的に加速され続けることを期待しましょう。
