120 dpiおよびemのフォント





Em /％のフォントはすべての人に適しています-IEユーザーはサイズを大きくすることができ、テキストを見やすくするためにブラウザーのデフォルトのフォントサイズを大きくすることを好み、プロのタイプセッターの標準です。



短所：...なくなったようですが、！

しかし、ユーザーの画面解像度が標準の96 dpiに設定されておらず、120に設定されている場合はどうでしょうか？

フォントが大きいサイトは、デザイナーが描いたほど美しくありません。結局のところ、サイト全体ではなくフォントのみが拡大縮小されます。 また、特にラップトップでは、120dpiの解像度がより一般的になっています！ どうする？ pxに戻りますか？



いや！



Operaにはこのバグはなくなり、解決を無視します。 しかしIE9では、記事のコードを使用して、サイト全体でフォントが縮小されたバグがあります。 すべてのブラウザで120dpiの修正を使用しないでください。IE6/ 7のコードのみを残してください。



注意！ 2013年現在、この記事は古く、その情報は理論的な知識にのみ役立ちます。 本番環境でコードを使用しないでください！ 彼はもう必要ありません。

* Operaにはこのバグはなくなりました。一般的には、もうすぐ別のエンジン（Webkit）に搭載される予定です。

* IE6および7は既に履歴項目であり、ほとんどの人がそれらをサポートしていません。

*実際、EMを補う人はほとんどいません。ほとんどの場合、レイアウトでスタイルの独立性（BEMメソッド）のためにPXを再び使用します。



そこで、問題を特定します。



Windowsでは、画面解像度が96 dpiを超えると、IE≤7とOperaはデフォルトのフォントサイズを比例的に増加させます。



これにより、通常、固定幅のサイトで不快なデザインの歪みが発生する可能性があります。

FirefoxおよびSafari / Chromeは画面dpiを無視します。

ラバーモックアップでは、サイズを拡大しても一般に問題は発生しませんが、固定幅のサイトでは、割り当てられた狭い列に拡大文字が収まらない場合があります。 大きなサイズは、残りの未検証のグラフィックの横にいように見えます。



フォントサイズが増加するとどうなりますか：

原則として、フォントのみが拡大されます。 サイトを分割するブロックも増加します（サイズがem /％で指定されている場合） が、グラフィックは増加しません ！



これは、サイズ変更可能なフォントを放棄する必要があるという意味ではありません！



実際、非常に大きなモニターでは、120 dpiは流行ではなく必需品です。小さな活字は見にくいです。 また、ユーザーは、この特定のページと一般的なブラウザー設定の両方で、すべてのサイトでフォントサイズが大きくなるように、フォントサイズを大きくしたいだけです。

サイトの使いやすさを低下させるために、 人々を高く壊さないでください。 しかし、デザイナーが意図したとおり、サイトは見た目が美しいはずです。なぜなら、彼らは洋服で会うからです。




CSSを使用すると、ユーザーの解像度を確認し、それに比例してフォントサイズを縮小できます。 



 html {font-size: 68.75%} /*   11px */ @media all and (min-resolution: 120dpi) { html {font-size: 55%} /*   : 68.75/(120/96) */ }


IE6 / 7の場合、次のコードが必要です。 

 #header { scrollbar-track-color:expression( this.runtimeStyle.scrollbarTrackColor = "#fff", ((screen.deviceXDPI/screen.logicalXDPI) == 1) ? (document.body.style.fontSize = 1/(screen.logicalYDPI/96) +'em') : false ); }
これは1回限りの1つの式で、画面のdpiに比例して基本サイズを変更します。

IE8はどうですか？



IE8では、すべてが非常に美しくエレガントに行われました。ブラウザーは、画面のdpi 5に比例してページのコンテンツ全体を増やします。

screen.deviceXDPI、screen.logicalXDPI、およびページズームに関する叙情的な余談



IE8では、フルページズームを有効にすると、screen.deviceXDPIの値が変更されます。 125％の場合、ズーム= 120、150％-144。100％の場合（スケーリングなし）、screen.deviceXDPIは96に戻ります。Windowsのdpiは問題ではありません。



それが（screen.deviceXDPI == screen.logicalXDPI）に必要なチェックです！

これは、IE8の右下隅にあるユーザーが[ページのズーム]をクリックして変更した場合です。

サイズを120dpi減らすには、CSS3プロパティmin-resolution



を使用します。

修正します：

html {font-size: 68.75%} /* 11px==1em, = 11px x Xem */

@media all and (min-resolution: 120dpi) {

html {font-size: 55%}

html, x:-moz-any-link {font-size: 68.75%}

x:-moz-any-link



は、-moz-any-link疑似クラスを持つ存在しないx要素の選択です。 Geckoブラウザーのみがこの擬似クラスを理解し、残りは行全体を無視します7 。

120dpiのサイトの外観を96にするかどうかはあなた次第です

繰り返しますが、サイトの開始形式について話しているところです。

画面解像度が120dpiのユーザーは、ほとんどのサイトを96とまったく同じように表示します（ほとんどのサイトは固定サイズで構成されているため）。 emに折りたたまれると、すぐに拡大表示されます。 さらに、フォントサイズのみが増加し、グラフィックスは元のままになります。 同意する-サイトはデザイナーが当初意図したほど美しくはありません。



それでは、ユーザーが意図したとおりに表示しないのはなぜですか？

それは任意の秒でその上のフォントサイズを増やすことができます。



ところで、ユーザー自身が120dpiを設定したのは事実ではありません。

このような設定がデフォルトであるラップトップを購入するだけでした。


また、ユーザーがFFまたはSafari（dpiでは無視されます）の基本フォントサイズを大きくした場合、見やすくなりますか？

修正がなければ、emを含むサイトには巨大なフォントが使用されます。

そして、IE8でのみ、サイトは美しく正確にスケーリングされます。



将来の証拠はどうですか？ ブラウザの将来のバージョンに問題はありますか？



近い将来、問題の発生はほとんどありません。

修正は、次の場合にのみ必要です。
  • Safariは、最小解像度のCSS3プロパティを理解し始めますが、画面のdpiに応じてサイズは拡大しません。

    この場合、FF3.1について記述したのと同様のルールでコードを補完します。



  • Internet Explorer 9は最小解像度の理解を開始し、IE8と同じ適応フルページズームを維持します。

    これは、条件付きコメント8を介してIE≥9を修正することで修正されます。



  • Operaは、高dpiでフォントサイズを大きくする代わりに、フルページズームを使用し始めます。

    これは、FF3.1で記述したものと同様のルールで処理されます。


だから、ここに既製のソリューションがあります： 



 html {font-size: 68.75%} /*      - */ @media all and (min-resolution: 120dpi) { html {font-size: 55%} /*   : 68.75/(120/96) */ html, x:-moz-any-link {font-size: 68.75%} /*     - */ } #for-opera927 {/* dont' remove! */}


IE6 / 7の場合：
