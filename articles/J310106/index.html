Magento 2. Le Roi est mort！ Vive le Roi？

Magento（当時はまだ「統一」）で初めて、約6年前に出会いました。 それ以来、私は彼女と一緒に密度を上げたり下げたりしてきました。 当初、この投稿は「Quo vadis、Magento？」と呼ばれたかったのですが、 判明したように、コミュニティはこの人気のある質問を何度も尋ねました。Magentoがebayで購入され、販売され、いつでも「2」今日まで、この質問は関連性が残っています（私はそのような名前を使いたいとさえ思っていたので）。 したがって、ポストは呼び出されたとおりに呼び出されます。













カットの下で、私はこのプラットフォームの見通しについて、自分自身の（つまり主観的な）ビジョンを作成しようとしました-不満と落胆に満ちています。 詳細な計算はありません。 詳細な考えはありません。 証拠なし。 そして最も重要なこと-







はい、主なものは希望なしです。







ルロア



私の練習では、Magentoはおそらく私が遭遇した中で最も複雑なWebアプリケーションです（もう一度、計算の主観性に焦点を当てます！）。 ジョナサン・エドワーズの同僚は、かつて「プログラミングは、コードの克服できない複雑さと要件の裏切りに対する必死の負けの戦いです」（ プログラミングは、克服できないコードの複雑さと要件の裏切りに対する絶望的に失われた戦いです ）。 そのため、この絶望的な闘争におけるMagentoは、私が見た他のすべてのWebアプリケーションよりもさらに進んでいます。







Magentoの基礎はもともと、適応性と拡張性の原則に基づいていました：EAV、モジュール、書き換え、イベント-これらは、多くのユーザー、そして最も重要なのは開発者をMagentoに引き付け、eコマースの主要な位置を占めることを可能にするメカニズムです。 Magentoの主な「機能」は、「すぐに使える」豊富な機能だけでなく、「自分用に」ストアをカスタマイズできる印象的な数のサードパーティモジュール（有料および無料）でもあります。







Magentoは多くの開発者の魅力の中心になっています-誰もが必要とする小さいが無料のモジュールを作成することで良い広告を得るか、基本機能ではカバーされない（またはEnterpriseバージョンでのみカバーされる）特定の顧客のニーズに合わせて商用モジュールを作ることで良いお金を得る） Magento Connectには、Magento 1用に少なくとも966ページのモジュールがあり、1ページあたり10ユニット-ほぼ10,000の拡張機能があります。







ワードプレスには50,000

はい、同じWordPressには現在約50,000個のモジュールがありますが、WordPressは「サイト」のエンジン（一般的にはCMSとユーザー）であり、Magentoは「ストア」（および製品カタログ、倉庫、ショッピング）のエンジンです。カート、注文、返品など）。 さらに、「サイト」に拡張機能を追加するのははるかに簡単です-十分ではありません。 しかし、「ストア」の詳細は、拡張の可能な方向をいくらか制限します。







一方、Magentoはユーザーにとっても魅力の中心となっています。ユーザーはすぐに実行可能なストアを受け取るだけでなく、既製のモジュールを使用して魅力的なテーマを選択し、機能を拡張することもできます。 または、「自分の」モジュールの作成を注文します。 または「あなたの」トピック。 拡張性はMagentoのキャッチーなマーケティングギミックの1つです。 もちろん、実際には、機能の追加/変更は完全に簡単な作業ではありませんでしたが、そのような機会は最初にプラットフォームの作成中にプラットフォームに追加され、競合他社に対して評価する際にうまく利用されました。 その結果、2015年の推定によると、Magentoの店舗数は25万台近くに達しました。













この双方向の魅力は、Magentoプラットフォームに基づくeコマースの分野でのさまざまなソリューションが作成、テスト、順応、または廃棄されるようなエコシステムを作成しました。 最初のバージョンでは完全なテストは行われませんでしたが、インストールの数によってその不在が補われました-一般的に使用される機能を使用し、エキゾチックな構成を構築しない場合、ストアの作業でエラーが発生する可能性はかなり低くなりました。 一定数の既存の大規模店舗も、少なくとも中小企業の観点からは、優れたプラットフォーム機能を示しています。







一般的に、最近では、Magentoはeコマース市場で議論の余地のないリーダーでした。







... est mort



それでは、Magentoの将来を悲観的に個人的に見ているのはなぜですか？ 恐らく彼らがタイタニックのエンジンルームの将来について悲観的に見たのと同じ理由で。 おそらくすべての情報が流れたエドワード・ジョン・スミス船長でさえ、当時のマーケティングサービスの類似物がタイタニック号の沈めないことを確信していた乗客は言うまでもなく、状況の悲劇を最初は理解していなかったでしょう。 しかし、機械室では、破裂したリベット、鋼板間の隙間が見られ、そこに氷の海水が破裂し、泡立ち、泡立ち、ポンプの動作が失敗しました。 機械室では、タイタニック号が水面にどれだけ長く耐えられるかを正確には知りませんでしたが、アメリカに届かないことは確かに知っていました。







Magento 2は間違いなく以前のバージョンの開発です。モジュール性が向上し、作曲家との互換性が実現し、「フロント」とサーバー側の両方で最新のソリューションが使用され、コードが公開さ れテストされ 、パフォーマンスが向上しました。 一見、非常に素晴らしい見通しです。

データ構造



Magento 2の開発者は、データ構造を大幅に作り直す素晴らしい機会を得ました-2番目のバージョンは最初のバージョンと互換性がないことが公式に発表されました。 また、データ構造が大幅に進歩しました。最初のバージョンのデータベースを実際に2番目のバージョンで使用できないように、テーブルとフィールドの名前が変更/追加/削除されました。







しかし、処理の主な動機は依然として「できるだけ少ない変更」であったことは明らかです。 それ以外の場合、DBのstore_idと「deuce」に移行した管理パネルのStoreの不一致を説明するにはどうすればよいですか（DBのstore_idは管理パネルのStore View属性ですが、DBのgroup_idは管理パネルのStoreの属性であり、これは理解できません、これは必要です覚えていますか？ または、同じストアテーブルまたはストックテーブルのデータ間の依存関係があいまいですか？







またはそのような基本的な（または修辞的な？）質問-なぜ製品属性（catalog_product）のみが拡張用に利用できるエンティティの8種類（顧客、顧客アドレス、カタログカテゴリ、カタログ製品、注文、請求書、クレジットメモ、出荷）だけですか？ admin、authorization、stock、sales_rule、cms、評価、レポート、レビューなどのエンティティが無視されるのはなぜですか？

慣性



Magentoには良い歴史があり、印象的なコミュニティがあり、eコマースのパイにはまともな「ピース」がありますが、同時に...プロジェクトのコントロールが失われています。 Magento 2は、変更に対するタイムリーな応答を停止しました。 彼女は大きくなりすぎた。 マイクロソフトのように一度。







おそらく、いくつかの法律が施行されており、それによると、プログラミングだけでなく、一般に複雑なシステムの作成は「必死の敗戦」です。 構造は、未知の複雑な境界線に近づきます。次の各ステップは、前のすべてのステップをまとめて行う価値があり、機能を構築するために、新しいものを作成するのが簡単です-他の原則に従って、より低いレベルの複雑さで同様の機能を実装し、許可する別の構造それ自体が同じ境界に近づくまで、この機能を構築します。 物理学にも同様の境界があります-光の速さ、なぜプログラミングにもありませんか？







そのため、Magento 2の慣性は非常に複雑であるため、開発の管理がますます困難になっています。 上記のデータ構造、コード、バグ修正、リリースで説明されている問題は、「複雑さの上限」に近づいた結果です。
