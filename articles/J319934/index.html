Pinbaを使用したアプリケーションの監視

こんにちは、Habr！ Badooの私たちは、ITコミュニティの生活に積極的に参加しようとしています。多くのオープンソースの技術とツールを使用し、開発を共有しています。







これらのツールの1つはPinbaです。これは、収集中のオーバーヘッドなしで実行中のアプリケーションからリアルタイム統計を受信するためのサービスです。 詳細については、この記事をご覧ください 。







私たちは、Pinbaをプロジェクトで使用し、Pinbaに関連するサクセスストーリーを常に喜んで聞いているすべての人を支援するよう努めています。 この翻訳は、Dailymotionの開発者による同様のストーリーの1つです。







はじめに



Dailymotionは、StatsD、Graphite、collectd、Datadogなどのさまざまなツールを使用して、アプリケーションを監視、分析、および最適化します。 しかし、あまり知られていないものもあります-ピンバ（PHPはもうボトルネックではありません）の略です。







現在、旧式のモノリシックPHPアプリケーションと多くのレガシーコードの移行に取り組んでいます。 代わりに、いくつかの最新のマイクロサービスを展開する予定です。 ピンバを使用して収集するもの：







  • 現在のPHPアプリケーションに関するデータ。 これにより、メンテナンスと展開をより効率的に実行し（パフォーマンスに悪影響を与えるリリースを特定）、考えられる問題を見つけて分析することができます。 さらに、取得したデータは、パフォーマンスを最大限に向上させるために、最初にプラットフォームのどのコンポーネントを改善する必要があるかを判断するのに役立ちます。







  • 新しいマイクロサービスに関するデータ。 さらに、開発言語（現在はPythonまたはGo）に応じて、さまざまなクライアントを使用します（ただし、すぐに使用できるクライアントがあります-約Per。）。


ピンバとは何ですか？



Pinbaの作成者tony2001は 、次のように説明しています。







「これはMySQLをインターフェースとして使用する統計サーバーです。 Pinbaは、UDPを介して多くのPHPプロセスによって送信されたデータを収集して処理し、その後、簡単で理解しやすいレポートの形式で統計を表示します。 さらに、特別なインターフェースを使用して、読み取り専用モードで生データにアクセスし、より詳細なレポートを生成できます。

どのように機能しますか？







PHPのPinba拡張機能は、Webファームのすべてのサーバーにインストールされます。 各PHPストリームの技術データを収集し、応答がクライアントに送信された後、受信したメトリックをUDP経由で中央のPinbaサーバーに送信します（Google Protobufデータ交換形式が使用されます）。 中央サーバーでは、Pinbaエンジンがこれらのメッセージの内容を集約し、MySQLエンジンで読み取り専用のふりをして、既知のMySQLクライアントを使用して表示および選択できるテーブル形式のデータを提供します。 便利で効果的。







一般的な誤解にもかかわらず、Pinbaはデバッガーでもグラフィカルツールでもありません。 本番環境のアプリケーションで何が起こっているかの写真を作成するのは単なるデータビューアーです（リアルタイムでは、最後の1、2分でスライスが作成されます）。 動的メトリックを使用して、パフォーマンスを犠牲にすることなく時系列グラフを作成し、それに応じて、発生する可能性のある問題を時間通りに特定できます。







もちろん、ピンバは万能薬ではなく、すべての問題を解決するわけではありません。 よくある間違いの1つは、このサービスを使用して正確なデータを収集することです（たとえば、直前に発生したイベントのカウンターとして）。 Pinbaサービスはこの種のタスクを対象としていません（既にご存知のように、パフォーマンスを向上させるために、protobufメッセージはUDP経由で送信されるため、途中で一部のパケットが失われる可能性があります）。 このツールを使用して傾向を追跡しますが、アクションを正確に定量化するには適していません。

サービスを使用してどのようなメトリックを追跡できますか？

すべてのPHPプロセスでデフォルトで追跡される技術的指標の範囲は非常に広いです。







  • doc_size-応答サイズ。







  • mem_peak_usage-割り当てられたメモリ（最大サイズ）;







  • request_time-プロセスの作成にかかる時間（マイクロセンド処理）。







  • ru_utime / ru_stime-サーバーリソース（ユーザーおよびシステム）の使用に関する統計。







  • status-HTTP応答コード。







  • memory_footprint-プロセスが占有しているメモリの量。

独自のメトリックの収集



ご覧のとおり、Pinbaは追加設定なしで多くの有用な情報を提供しますが、それだけではありません！ レポートテーブルに加えて、タグレポート *テーブルを使用できます。 メトリックは、コードで定義したタグ値に従ってこれらのテーブルに統合されます。







たとえば、古いDailymotionコードベースには、2種類のタグがあります。







  • I / O：外部からのアクセス、プロセスのブロック（へのアクセス

    MySQL、Elastic、Redis、memcached、または外部APIなどのサーバー、

    例：FacebookまたはTwitter）;







  • algo：私たちが見たいアルゴリズム。


たとえば、memcacheの場合、次のようにタグを使用します。







顧客コード 







 $pinbaTimer = $this->pinbaService->startTimer([ 'group' => 'memcached', 'memcached' => $this->pinba_pool, 'method' => 'get', 'namespace' => 'dm_cache' ]); $value = $this->memcache->get($this->prefix . $key); $this->pinbaService->stopTimer($pinbaTimer);


ピンバサービス 







 public function startTimer(array $tags) { if (!$this->isPinbaInstalled) { return null; } $timerId = 't' . ++$this->incr; $this->timers[$timerId] = pinba_timer_start($tags); return $timerId; } public function stopTimer($timerId) { if ($this->isPinbaInstalled && isset($this->timers[$timerId])) { pinba_timer_stop($this->timers[$timerId]); unset($this->timers[$timerId]); } }


次に、Pinbaデータベースにユーザーテーブルを作成し、コメントを使用してエンジンに格納する内容を説明します。

追加機能



また、Pinbaの別の興味深い機能-タグを使用して、異なるタイプのリクエストを区別できます。 要求には、定義したタグがタグ付けされます。 たとえば、次のクエリにタグを使用します。







  • 形式：応答形式（html、rss、atom、...）;







  • site_content：サイトのコンテンツの言語（en、de、fr、jp、ru、...）;







  • ボット：実際のクライアントまたはボット？ （はい/いいえ）;







  • auth：ユーザー認証（ログイン）？ （はい/いいえ）;







  • プロバイダー：PHPのバージョン間の違いを判断します：標準のPHPまたはHHVMなど。


次に、テーブルを作成し、エンジンのコメントで何をどこに保存するかを示します。 たとえば、report_by_script_nameに基づいてテーブルを作成できます。このテーブルでは、リクエストタグボットの値がnoで、リクエストタグ認証の値がnoのPHPプロセスのデータのみがエンジンによって保存されます。 言い換えれば、これらはログインしていない実際のユーザーです-これは一般的な状況です。 







 CREATE TABLE `auth_no_bot_no_report_by_script_name` ( `req_count` int(11) DEFAULT NULL, `req_per_sec` float DEFAULT NULL, ... `req_time_median` float DEFAULT NULL, `index_value` varchar(256) DEFAULT NULL ) ENGINE=PINBA DEFAULT CHARSET=latin1 COMMENT='report1::tag.auth=no,tag.bot=no';


プロット



ピンバは、受信したメトリックをわずか1〜2分で保存します。 したがって、グラフを作成するには、データを外部リソースに統合する必要があります。 ここでは、MySQLプラグインとペアになったcollectdを使用します。 この場合、クエリはPinbaデータベースに対して行われ、結果はWhisperに保存されます。 次に、WhisperをGraphiteとTesseraまたはGrafanaでクエリします。







おわりに



古いPHPアプリケーションでPinbaを使用すると、次のことが可能になりました。







  • アプリケーションの問題の考えられる原因を検出します。







  • エラーを検出します（一部のルートで不当に多数のサーバー要求）。







  • チャートを作成し、アラートを設定して、リリース後のパフォーマンス低下のインシデントを検出します。


PythonプロジェクトでもPinbaを使用しています。 Pynbaの特殊バージョンは、旧友（現在は同僚）のjohnnooneによって作成されました。







Pinbaの使用中に気づいた唯一のこと：拡張機能は、古いリクエストに関する情報を常に「忘れる」わけではありません。 時々、レポートテーブルには幻の結果が含まれます-1分以上前に発生したアクティビティに関するデータ。 Pinbaを使用して何かが発生していないことを確認する必要がある場合（通常、新しいコードを実稼働環境に展開した後）、これは問題になる可能性があります。 これは、エラーの原因がコードにない数少ないケースの1つです。







これを修正する最も簡単な方法は、Pinbaエンジンを再起動することです。 その結果、グラフに小さなギャップが表示されますが、ファントムデータは間違いなく消えます。 （注：TRUNCATE TABLEはMySQLエンジンではないため、適切ではありません。）
