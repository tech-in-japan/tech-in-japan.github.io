ボタン、ダイアログ、および適用ボタンについて

「アグレッシブ」とがった「丸みを帯びたボタン」のトピックでは、ダイアログボックスの標準ボタンの位置（「はい」、「いいえ」、「OK」、「キャンセル」）について少し議論しました。 この記事では、インターフェイスデザイナーがダイアログボックスに関して犯す主な間違いについて説明します。



ダイアログボックス



習慣から、ボタンをクリックして、1つの結果が得られることを期待している代わりに、まったく逆の状況に陥る頻度はどれくらいですか？ または、出口の確認の過度に難解な対話に「ハング」したとしましょうか？ または、「保存しない？」シリーズの質問や、「はい」、「いいえ」、「キャンセル」のボタンに出会ったかもしれません。 この理由は、開発者が「書かれていないデザインコード」を誤解しているためです。 この問題はフリーソフトウェアの世界では一般的であり、プログラムインターフェイスが厳密に標準化され、慎重に検討されている大企業の製品ではほとんど見られません。 いくつかの例を見てみましょう。



Microsoft Word



これは良い対話です。 シンプルで、必要なものはすべて揃っており、余分なものはありません。 順番に行きましょう：
  • 「変更を保存しますか？」という直接的な短い質問がすぐに出されました。
  • 短く、すぐに理解でき、簡単に理解できる答え：「はい」、「いいえ」、「キャンセル」。
  • 答えのオプションは、唯一の正しい便利な順序です。
  • 正解（例の90％）はすでに強調表示されています。ギャップを安全に押して、さらに作業を進めることができます。
しかし、欠点もあります（そのうちの1つは企業、母親、標準です）。
  • ウィンドウタイトル 質問の本質である「保存しますか？」を複製する方がはるかに正確です。 しかし一方で、この質問は、ユーザーが別のアプリケーションで作業しているときにこの質問がポップアップ表示される可能性があるため、良いことです。 繰り返しになりますが、特に他のプログラムで動作する場合、Wordはユーザーの介入なしに突然終了しますか？ 長所は短所以上です。 したがって、タイトルに質問を複製するか、妥協案として、質問とともにプログラムの名前を指定することをお勧めします（「保存？-Microsoft Word」）。
  • アイコン。 マイクロソフト製品では、対話のトーンを設定するように設計されていますが、多くの場合、開発者はアイコンとして「トーン」記号だけでなく、アクションアイコンがあればそれを使用します。 たとえば、これは次の例で「保存」アイコンを使用して行われます。 「ディスケットまたはHDDと下向き矢印」という質問については触れません。これはこのトピックの範囲外であるためです。
次に、同じ原理で別の対話を分析します。



GNUイメージ操作プログラム（GIMP）



これは悪い対話です。 それは複雑で、情報でいっぱいです。 リストを見てみましょう。
  • 質問が長すぎる形式で表示され、何らかの理由で「閉じる前に」追加されます。 この質問はドキュメントを閉じるときにのみ尋ねられます。なぜ「閉じる前に」と書くのでしょうか？ 繰り返しますが、「画像内」を明確にする必要があるのはなぜですか？ 私はグラフィカルエディタを使用していますが、キャプテンは、画像を扱うことを教えてくれます。 とにかく、3D Maxは「textures％file_name％で覆われた一連の3次元モデルへの変更を保存しますか？」
  • 質問に対応していない長すぎる回答オプション（ hello、bashorg！ ）。 これらのボタンを使用すると、脳に電源を入れて、彼らが私に求めていることを理解させることができます。
  • 回答オプションは、理解可能な1つの順序で配置されています。 なぜ「いいえ」、「キャンセル」、「はい」なのか？ 彼らは「おい、あなたの工芸品はひどい、ねじを詰まらせないで」をどのように示唆するだろうか？
  • 間違った答えが強調表示されます。 デフォルトの文は、ユーザーの1回の動きで何らかのアクションを実行する必要があります。 Windowsにソフトウェアをインストールするためのダイアログを想像してください。各ウィンドウの「次へ」ではなく「キャンセル」にフォーカスが移動します。 バカじゃない？
  • 「画像を保存しないと、過去3分間のすべての変更が失われる」というフレーズが追加される理由は不明です。 喜ぶ、紳士：私はキャプテンエビデンスの職場を計算しました。
ただし、このウィンドウには肯定的な側面があります。
  • 私たちが保全について話していることをすぐに明らかにする有益なアイコン。
  • 有益な（疑わしい）ウィンドウタイトル。
これらのダイアログボックスの違いを感じますか？ 前者は便利ですが、後者はそうではありません。 成功した製品を分析する場合、いくつかのルールを強調できます。 このコンテキストでは、最も重要なルールはボタンの順序です。 便利なオプションは「ポジティブ、ネガティブ、ニュートラル」であり、他には何もありません。 このような状況では、ユーザーは最初のボタン-確認、最後-でこの質問をキャンセルすることに慣れています。 設定のようなモーダルウィンドウについては、以下で説明します。



ここからどのような結論を導き出すことができますか：
  • ユーザーに質問を30分考えさせないように、できる限り簡潔に質問します。
  • 否定的な質問を使用しないでください（「保存しない？」、「はい」、「いいえ」、「キャンセル」）。
  • 一般に受け入れられているボタンの順序のみを使用します（正、負、中立）
  • 回答オプションはシンプルで簡潔でなければなりません。 これは、何でも何時間もチャットできる言語学者のセミナーではなく、すぐに動作するはずのプログラムです。
  • 余分な情報を書き込まないでください。 もちろん、最後の保存からどれだけ時間が経過したかを考えるのはクールですが、ユーザーにとってはすみれ色であり、ダイアログでは気を散らし、場合によってはst迷や怒りさえもします。
  • ウィンドウのタイトルも参考にしてください。
  • アイコンを忘れないでください。 問題の本質を反映し、簡単に認識できるものでなければなりません。


モーダルウィンドウと[適用]ボタン



同じ問題が設定ウィンドウで発生します。 これが正しいウィンドウです（ああ、やはりMS）：



Microsoft Word



2つのボタン：「OK」と「キャンセル」。 そして、右側の「キャンセル」。 そうだったし、そうなるでしょう。 誰もがそれに慣れているので、これは正しいです。 ユーザーがそこに何か名前を付け、突然変更を保存するために心を変えた場合、彼はそこでキャンセルを見ることに慣れているので、右下のキーを盲目的にクリックします（MS製品にあります）。 Linuxはまったく別の問題です。 厳密なルールや基準はありません。ここでは、誰もが望むようにしています。



リズムボックス



面白いですね。 前のウィンドウには通常のボタンレイアウトがありますが。 開発者を導いたもの-あるハリネズミは知っています。 これらの失敗を著者の良心に任せましょう、そして私たち自身が結論付けます：右端のボタンはキャンセルです（これがあなたの特別な場合でない場合）。



Microsoft Windows



もう1つの興味深いオプションは、[適用]ボタンです。 ああ、彼女との思い出は...「OK」をクリックする前に、「適用」が完全に自動的に押されます。 理由は明らかではありませんが、それでも:-)

Microsoftが何らかの理由で拒否したくない過去の遺物。 ダイアログボックスを閉じずに変更を保存および適用することを目的としています。 設定を変更しました-[適用]をクリックしてください。 これは、Windows 95のコンピューターが弱く、「オンザフライ」で設定を適用すると、PCで一時停止が発生したためです。 現在、ドーレン核機械と4つの脳のギグを搭載した「オフィス」馬の時点では、そのようなシステムは単に不要です。



Ubuntu Linux



たとえば、Gnomeは「オンザフライ」で設定を適用するシステムを使用します。 画像にマウスを突っ込むと、すぐに壁紙になりました。 「適用」/「保存」をクリックする必要はありません。 ボタンは1つだけです-「閉じる」。 また、重要ではないダイアログではこれ以上必要ありません。



したがって、結論：「適用」ボタンの使用はほとんどないため、その使用が求められています。



それは、原則として、すべてこの主題に関係しています。 紳士、良い、便利なインターフェース。 ユーザーが喜ぶように！
