CoreOSで1年間働いた方法

CoreOSについて初めて聞いたのは 、2013年9月のYandex Road to Cloud ConferenceでPeter Lemenkovからでした。 それから、私はこのOSの開発に参加することさえ考えられませんでした。







2014年10月にCoreOSを2回目に思い出したとき、Rubyで書かれたマイクロサービス（奇妙なことに、異なるバージョンのRubyを使用していました）を継続的統合に適した環境に移行するタスクを受け取りました。 その後、初めてCoreOSを起動しましたが、使用するのは非常に不便に思えました。 そのドキュメントは表面的なものでした。 CoreOSをクラスター化されたOSに変えたサービスには多くの欠陥があり、絶え間ないエラーのために迷惑な感じを引き起こしました。 インフラストラクチャの一部でさえCoreOSに移行することに疑問の余地はありませんでした。







2015年3月に、CoreOSのコミュニティサポートの一部としてサポートサービスを提供するというタスクが3回目に受けられました。 私がどのように対処したかについて、議論があります。







知人



最初のタスクは、私が働いていた顧客の1人の実稼働システムに近い機能を実行するクラスターを構築することでした。 私はたくさんのKafka-Storm-Cassandraで実験しなければなりませんでした。 概念実証の実行中に、ドキュメントとコードetcdおよびfleetのすべての同じ欠陥に遭遇しました 。 それでも、etcdがすでにシステム上にあるときにZookeeperクラスターを上げることは非論理的でした。 残念ながら、これまでのところ、REST API etcdでZookeeper Juteプロトコルのトランスレーターを作成したいという希望はありません。 最大の困難は、Apache Stormのトポロジーの作成を引き起こしました。 https://github.com/Yelp/pyleusプロジェクトのおかげで、Java / Clojureトポロジーの説明を避けることができました。 実用的な概念実証は、実装の潜在的な顧客の1人にさえ成功裏に実証されましたが、残念ながら、プロジェクトは資金の問題のために実装できませんでした。







練習する



得られた経験とCoreOSを学習する過程でのバンプを使用して、改善を開始しました。 公式IRCチャンネル#coreosから、質問に回答し、ドキュメントを作成しました。 最も重要なことは、入ってくるすべての質問を真剣に受け止め、最も愚かな質問にも答えようとすることです。 私がサポートを提供した新しいテクノロジーは、それらについて質問したユーザーにとっては未知であったことは注目に値します。 ユーザーが質問をしていたとき、ユーザー環境を再現し、詳細を自分で理解し、ソースコードにアクセスし、場合によってはエラーを修正する必要がありました。 このような火の洗礼により、systemd、Linuxカーネル、Cの練習、Goでの記述の学習のニュアンスの多くを研究することができました。







ドキュメント



systemdに関連するユーザーの質問は非常に頻繁にあります。 systemdの公式ドキュメントにはすべての詳細が記載されていますが、ユーザーは例を必要としますが、読む時間はありません（言うまでもありませんが、ユーザーの心をつかむには、既製のサンプルなどが必要です）。 CoreOSをサポートするために、systemdに関する多くの例と追加のドキュメントを書きました。 そのうちのいくつかについては、GoogleはCoreOSドキュメントのページへの最初のリンクを発行しました。 その後、チャンピオンシップがWiki Archlinuxを引き継ぎました。 例と説明は、ユーザーが間違った方法で情報を解釈できるあらゆる場所にあるべきでした。 私の考えで生じたドキュメントまたは質問「what if？」に関するユーザーの誤解のほとんどは、githubでのプルリクエストに変わりました。 ユーザーの質問に対する回答がドキュメントにない場合は、修正してください。







遊ぶ



回答を投稿する前に、可能な限りCoreOSの安定版、ベータ版、およびアルファ版で確認する必要があります。 これを行うには、libvirtの助けを借りて、Ubuntuの既製のクラウドVMイメージを使用して、内部インフラストラクチャの仮想マシンを持ち上げるbashスクリプトを適応させる必要がありました（これについては別の投稿を書くかもしれません）。 必要に応じて、スクリプトは特定のリリースの公式CoreOSイメージ（ https://stable.release.core-os.net/amd64-usr/current/ ）をダウンロードし、それに基づいてスナップショットを作成し、 cloud-configがマウントされた仮想マシンを起動します。 ショットショットを使用する場合、「クリーン」な仮想マシンを作成するプロセスは20秒しかかかりません（SSDをさらに高速に使用する場合）。 仮想マシンのクラスターを作成するとき、ディスクの時間とスペースを節約する1つの基本イメージが使用されます。 たとえば、3つの仮想マシンのクラスターはわずか30秒で立ち上がります。 Vagrantと比較したlibvirt-qemuソリューションの利点。 Vagrantのlibvirtプロバイダーでさえ、そのような速度を提供しませんでした。 当日は、ユーザー環境を繰り返すクラスターを数十回作成および削除する必要がありました。 クラスター全体がたった1つのcloud-configで構成されました。 現在、cloudinitの代わりに、 Ignitionがアクティブに実装されており、システムの初期ブート段階で実行されます。







ヘルムズマン



CoreOS製品と密接に使用されているKubernetesプロジェクトを忘れないでください。 Kubernetesを学習するために、Cloud-configを作成しました。これは、CoreOSのKubernetesチームが作成した公式ドキュメントに記載されている構成を正確に再現します。 また、ユーザーが抱えていた問題を数分で再現し、解決策を提案することも可能になりました。







解決策



ユーザーからの質問に対して、解決策、または少なくとも回避策を発行しようとしました。 CoreOSのサイドラインへのアクセスでさえ、あまり助けにはなりませんでした。なぜなら、 世界のヨーロッパ地域でサポートを提供し、CoreOSチーム全体が太平洋時間帯で働いていました。 私の同僚が地球の反対側で目覚めるまで待たないために、私はしばしばソフトウェアのプロセスを理解するためにソースコードを研究し、時にはコードのエラーをすぐに修正しました。







ご意見



これまで、多くの人がコンテナ化に懐疑的かつ断定的に目を向けています。 DockerまたはCoreOSに言及する価値があるだけなので、** ** ** vnaを注いでください。 各タスクには独自のソリューションツールがあるという事実に常に注意を払っています。 そして、各ツールには長所と短所があります。 システムがすでに仮想マシン上で実行されており、何かを変更したり台無しにしたりする必要がない場合は、さらに機能させます。 コンテナにすべてを強制的に転送することはありません。 ただし、タスクが既に設定されている場合、またはコンテナで遊ぶ自由時間があれば、誤解が生じます。 仮想マシン、キックスタート、および構成管理システムでの作業に慣れている人々は、コンテナに対する古いアプローチを採用し、その後、コンテナが機能しないことを訴えます。 私は炎を繁殖させません。もう一度http://12factor.net/へのリンクを提供します。ほとんどの場合、コンテナは構成内に格納され、ホストに依存しないことを思い出してください。







完璧さ



完璧に制限はありません。 仕事は常にされています。 穏やかな日には、TODOリストで時間を無駄にしました。 リスト内の小さなアイデアや欠点をすべて書き留めましたが、このリストは決して終わりませんでした。







コンテキスト



最も困難なことは、念頭に置いていくつかのタスクに取り組むことでした。 IRCチャンネルで同時に5人のユーザーとやり取りすることがありました。 それは脳を非常によく訓練しますが、長時間のコンテキスト切り替えは非常に疲れます。







艦隊



サポートリクエストの数が減り始めるとすぐに、 フリートプロジェクトに取り組んでいる小さなチームに参加しました。 当時、プロジェクトは放棄されました。なぜなら すべての注意はKubernetesに向けられました。 しかし、systemdをクラスターとして使用する必要があるユーザーがいました。 3か月間、約15の新しい機能テストを作成し、 https：//semaphoreci.comで自動検証を実装しました。 バージョンv0.12.0のリリースまでに、36の問題を解決し、コードに20の変更を加えました。 バージョンv0.13.0までに、約20の問題をクローズし、コードに16の変更を加える予定です。
