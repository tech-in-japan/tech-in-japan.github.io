Gentooを高速化する必要がありますか？

「Gentooを自分用にインストールする予定です。私のプロセッサの機能を使用する方が良いでしょうし、それを最大限に活用するでしょう。」 さて、それを把握しましょう...







プロセッサの一般的な最適化は何ですか？


基本的に、これは、アプリケーションのコンパイル時にMMX、SSE、AES、AVXなどの追加の命令セットを使用することを意味します。 ただし、深く掘り下げると、アプリケーションだけでなく他の最適化も行われます。

次の最適化グループを特定しました。

  • コードの最適化
    • x86命令の追加セット（ MMX、SSE、AES、ATA、AVXなど）のコンパイル時のコード最適化
    • コンパイル中の静的分析中のコードの最適化：末尾再帰の展開、コードの未使用セクションの削除、無意味な条件の無視など。
    • プロセッサキャッシュのヒットを改善するための最適化。
  • カーネルレベルのコードの最適化： Cryptographic APIの暗号化メソッド。




追加の命令セットの最適化については、 Intel 386およびAMD x86-64 GCCオプションページで詳しく 説明されています。 Pentium MMXから始めて、MMXが利用可能になり、AMDが3DNow！を作成し、SSEがPentium IIIに登場しました。 今年私たちを喜ばせたIntel Haswellは、MOVBE、MMX、SSE、SSE2、SSE3、SSSE3、SSE4.1、SSE4.2、AVX、AVX2、AES、PCLMUL、FSGSBASE、RDRND、FMA、BMI、BMI2、およびF16C。

コンパイラはこれにSSEを使用できるため、実数（ FPU ）の操作は、追加の命令セットにも間接的に関連しています。 x87命令よりも高速で、MMXをブロックしません。 これについてもう少し詳しく説明します。



別の重要な機能に注意する必要があります。 SSE、MMX、およびAESの略語を声に出して言及すると、これらの概念に精通している人々は、アセンブラーがソフトウェア内のこれらの命令のファイルをサポートするCプログラマーの困難な生活を伝える写真を頭に浮かびます。 実際、これらの命令セットを使用する方法はすでに3つあります：静的コード分析用のコンパイラーによる自動、 特別なコンパイラー関数の手動による使用、およびアセンブラー挿入の手動による使用（例： MMXプロセッサー向けにコードを最適化する方法 ）。 どの特定のケースでGCCが自動的に一連の指示を使用するか（許可されている場合）は明確ではありませんが、マニュアルにはそのようなケースがあることが明確に記載されています（願わくばコメントに書かれます）。



静的解析コードの最適化については 、 最適化を制御するオプションページで詳しく説明されています。 多くの可能性がありますが、混乱しないように、それらはメタフラグにグループ化されます：O0、O1、O2、O3、Ofast。 これらのフラグの詳細については、リンク先の記事をご覧ください。



プロセッサキャッシュのヒットを改善するための最適化 。 すぐに説明することはできないので、読者に別の記事「 バブル、キャッシュ、分岐予測子」を送信します。 プログラマーは、最適化できる場所を分析するために、 Intel VTune Performance AnalyzerとAMD CodeAnalystを使用できるとしか言えません。 また、 ICC Intel C ++コンパイラーはそのような最適化を自動的に実行できる場合もあるようですが、知識のある人がコメントに今日のGCCの対処方法を追加してくれることを願っています。



カーネルレベルのコード最適化 。 SSE、AVX、AESなどの追加の命令セットを使用して、AES、Twofishな​​どの暗号化APIフレームワークの機能を高速化できます。 これらの関数は、他のカーネルモジュールで使用したり、アプリケーションから外部から呼び出したりできます。



理論を理解したので、その使用方法に移りましょう。



Ubuntuをお持ちの場合


Ubuntuに座っているとします。 オペレーティングシステムのサイズに応じて、接尾辞i386またはamd64のパッケージから選択できます（ 例 ）。 i386は、386以降のパッケージがどのプロセッサでも動作することを意味するものではなく、単にパッケージがx86 32ビットプラットフォーム向けであることを意味します。 同様に、amd64は、x86-64 64ビットプラットフォームのサポートを意味します。 コンソールに入力すると、これを簡単に確認できます。 

gcc -dumpmachine


32ビットUbuntu 12.04 LTSサーバーではi686-linux-gnuが表示され、64ビットではx86_64-linux-gnuが表示されます。

32ビットのPentium 4があり、MMX、SSE、およびSSE2を使用できると仮定しますが、これらはMMXのみのIntel Celeronで動作するはずであり、おそらくPentium Proでも動作しないため、パッケージの生成には使用されませんでしたMMXでさえ。

追加の命令セットは、それ自体でプロセッサを決定し、プロセッサ用のより高速なアルゴリズムを含むパッケージでのみ使用されます。 幸いなことに、これはほとんどすべてのマルチメディアパッケージで発生します。

また、32ビットUbuntuがどのコード最適化に使用されたのかも明確ではありません。 GCCの出力を見ると、-O1から、-O2と-O3から少しだけです。 特定のバージョンのUbuntuのパッケージがデフォルトのコンパイルオプションを使用してシステム自体に習慣的にビルドされている場合、明らかに最適な（合理的な）方法で収集されていません。

最後に、カーネル暗号化APIの関数は最適化されていません。 追加の命令セット用に最適化された関数は、モジュール形式でのみ、i586およびAES（VIA Nano）にのみ存在しますが、デフォルトではロードされません。 また、586のどれが最適化に使用できるかは明確ではありません。



Ubuntu 12.04 64ビットでは、状況はずっと改善されています。 まず、gccはデフォルトで64ビットシステムの拡張機能であるMMX、SSE、SSE2を使用するため、コードをわずかに最適化できます。 次に、x86-64の場合、デフォルトは-mfpmath = sseであり、これは実数の演算を高速化します。

追加の命令セット用に最適化されたカーネル暗号化API関数は、システム内のモジュールに存在しますが、デフォルトではロードされません。 少なくともそれらはオンにすることができます。

そして最後に、gccはUbuntu 32ビットの場合と同じ奇妙な最適化セットでパッケージを収集します。



gentooを持っている場合


ほとんどの場合、マニュアルからこのページを読みます 。 したがって、 -O2および-march = native （または正しいプロセッサ）を設定します。 しかし、おそらくあなたはそもそもそうです：カーネルをセットアップするときに暗号化APIにアクセスせず、自分自身のためにいくつかの命令をスピードアップしませんでしたが、少なくともAESをスピードアップする価値があります。 第二に：おそらく、利用可能な追加のプロセッサ命令（3dnow、mmx、sse、sse2、sse3）にUSEフラグを設定していません。 または、すべてが公開されたわけではありません。 これは、USEフラグで最適化を意図的にアクティブにするアプリケーションの場合、追加の加速なしで放置されることを意味します。



グローバルフラグに加えて、一部のアプリケーションの追加命令をトリガーするローカルフラグもあります。 3dnowext、ssse3、sse4、sse4_1、avx、avx128fma、avx256、aes-niなど。 あなたによってサポートされているすべてのものも公開する方が良いです。




amd64の最新のstage3のデフォルトは、bindist、mmx、sse、sse2です。 残念ながら、bindistは、移植性のために一部のパッケージで追加の指示を無効にします。 バインディストが必要な場合は、一部のアプリケーションでcpudetectionを使用して、バインディストフラグの欠陥を軽減してください。
