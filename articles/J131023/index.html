Windows ExplorerおよびTotal Commanderでの簡単なナビゲーション

Windowsでのナビゲーションの方がずっと便利だと思ったことはありますか？ ショートカットによって目的の名前のフォルダーがドロップダウンリストに表示される場合、次のように、多くの最新のIDEでクラスナビゲーションと同様の機会を追加してみませんか。







考えてから、上の写真の「ナビゲーションアシスタント」という小さなアプリケーションを作成しました。 ここから入手できます 。ソースコードは公開されており、無料で、MITライセンスの下で配布されています。 それが誰かに役立つことを願っています。



なぜこれが必要なのですか？



WindowsエクスプローラーとTotal Commanderのナビゲーションの欠陥は次のとおりです。これは、多くの場合フラストレーションを引き起こします。

  1. 目的のフォルダのネストの深さが大きい場合、マウスを何度もクリックする必要があります

  2. Windowsエクスプローラのデフォルトの検索は非常に遅くなります。 さらに、それを使用するには、エクスプローラー自体を起動し、検索フィールドをクリックして、フォルダーまたはファイル全体の名前を入力し、長時間待つ必要があります

  3. Total Commanderには、アドレスバーにパスを入力する際のヒントがありますが、不便なフルパスまたは相対パス（現在のフォルダーに対する）のいずれかを入力する必要があります



だからこそ、私は「ナビゲーションアシスタント」を書きました。その操作は、JetBrains製品（ReSharper、IDEA、PhpStorm、WebStorm）のCtrl-NとCtrl-Shift-Nの組み合わせのCtrl-NとCtrl-Shift-T日食



このプログラムを使用して得られるボーナスは、最適なディレクトリ構造です。 多くの開発者、特にCode Completeを読んでいる開発者は、脳が作業メモリ内の7±2個のオブジェクトにうまく対処できることを知っています。 したがって、クラス内のフィールドとメソッドの数を制限すると便利であるように、特定のディレクトリ内のサブディレクトリの数も制限すると便利です。



現状では、これは不便なナビゲーションを犠牲にしてのみ達成できます。 ナビゲーションアシスタントを使用すると、ディレクトリの大きな入れ子を心配する必要がなくなります。



使い方は？



これまでのところ、フォルダーでは高速ナビゲーションのみがサポートされており、ファイルではサポートされていません。



必要なフォルダーにすばやくアクセスするには、WindowsエクスプローラーまたはTotal Commanderウィンドウ（または他の任意の、しかし、新しいWindowsエクスプローラーがナビゲーション用にインスタンス化されます）で、特別なキーの組み合わせを押します（デフォルトではCtrl-Shift-Mですが、設定で変更できます） ） メインアプリケーションウィンドウが表示されます。







到達したいフォルダの名前の入力を開始します。



キラー機能：フォルダー名全体を入力するオプション。 たとえば、ドキュメントと設定にアクセスするには、「ドキュメントと」のみを入力できます。 さらに、「ドキュメント」という単語全体を入力する必要はなく、「doc and」（こんにちは、JetBrains！）で十分です。 また、最初の単語からフォルダの名前を入力する必要はありません：「および設定」も適切です。 特に開発者向けには、パスカル/キャメルケースがサポートされています。「Nav assistAssistant」フォルダも検索文字列「nav assist」に割り当てられます。



これらの操作の結果、適切なディレクトリのリストが画面に表示されます。







目的の項目をマウスで選択するか、上/下キーを使用してEnterをクリックするだけです（入力フォーカスはこの時点でテキストボックスに残ることができます）。



ウィンドウが気に入らない場合は、Escキーを使用してトレイにすばやく最小化できます。



Total Commanderではなく、Windowsエクスプローラーでプログラムを呼び出すと、新しいWindowsエクスプローラーウィンドウが開きます（必要なフォルダーと共に）。 設定では、デフォルトのナビゲーターをTotal Commanderに変更できます。



他に何がありますか？



変更できる設定ウィンドウもあります

  1. 改善されたナビゲーションをサポートするプログラムのリスト（たとえば、Total Commanderを無効にする）

  2. デフォルトのナビゲーションプログラム

  3. ナビゲーションアシスタントウィンドウを開くキーボードショートカット

  4. 検索するディレクトリを指定します（たとえば、C：\ Users \のみ、またはD：\のみ）

  5. さらに、結果からいくつかのディレクトリ名を除外できます（たとえば、bin、obj、.svn）

    （より正確には、binフォルダーを含むパスはすべて除外されます。これらは実際には正規表現であるため、bin \ S *のようなものを指定できます）

  6. デフォルトのブートを設定します（起動するたびに内部ディレクトリキャッシュを更新しないようにするため、推奨されます）



ツールチップはすべてのオプションで利用可能です。 設定を取得するには、トレイアイコンを右クリックします。



これはどのように行われますか



このユーティリティは、WPFテクノロジを使用してMicrosoft .Net 3.5で記述されているため、.Net 3.5が存在する場所であればどこでも動作するはずです。



インストーラーはInno Setupを使用して作成されています 。これはインストーラーを作成するすべての人にお勧めします。WiX、Install Shield、Visual Studio Installerよりもはるかに優れています。

  1. 無料で無料

  2. モジュラー（Microsoftテクノロジーに縛られない）

  3. 透明（たとえば、Visual Studioインストーラーのように、テキストスクリプトのみが編集され、視覚的なプログラミングは編集されません）

  4. その結果、インストーラファイルは変更ごとに再作成されません—バージョン管理システムに問題はありません

  5. 柔軟（Pascal / Delphiを使用したスクリプト作成をサポート）。 その前に、私は研究所の最初の年に約7年前にDelphiを最後に見ましたが、InnoSetupの利点はこれを補う以上のものです

  6. そうそう、魔法のようにx32およびx64バージョンのインストールファイルを1つ作成します

コードにはいくつかの重要な点があります。



グローバルキーストローク応答


ナビゲーションアシスタントウィンドウを適時に表示するには、キーストロークキーをグローバルに（つまり、すべてのアプリケーションで）聞く必要があります。 .Netでは、これをそのまま使用することはできません。 そのため、 ここからコードを使用しました （他にもいくつかのソリューションがありますが、グローバルフック/グローバルホットキーフレーズをGoogleで検索する必要があります）。



非アクティブなアプリケーションにフォーカスを設定する


大切なキーの組み合わせがいくつかのウィンドウで押されていることがわかった後、ナビゲーションアシスタントを表示し、フォーカスを設定する（アクティブにする）必要があります。



はい、WPFにはWinApiのSetForegroundWindow関数を呼び出すWindow.Activateメソッドがあり、そのためすべての制限があります。 つまり、メソッドが呼び出されたときにアプリケーションが非アクティブである場合、ウィンドウは動作可能になりません。つまり、フォーカスは古いウィンドウに残ります。 実際、悪意のあるプログラムが現在のフォーカスを突然変更して、ユーザーの作業をあらゆる方法で妨げることを許可しないために、これは正しいです。 しかし、この場合、この制限を回避したいと思います。



解決策はここから役立ちましたが 、他にもいくつかあります（ 1、2 ）が、うまく機能しませんでした。

最初はModel-View-View Modelテンプレートを使用しようとしましたが、次に進みました

Model-View-Presenterは、次の理由によります。

  1. MVPは発表者に追加の抽象化レベルを追加し、それなしでは、すべてのロジックはビューモデルにあり、後者は不当に成長します

  2. このロジックは、意味論的にビューモデルに属していてはなりません（単一の責任、現在）

  3. インターフェイス駆動型の開発は使用できません（ビュー用）

  4. その結果、UIレイヤーは混乱に変わります（ある意味では、職務の分離、モジュール性、明確なインターフェースはありません）



したがって、Model-View-Presenterが使用されるようになり、UIとデータはビューモデル（WPFバインディング）、さらにビューモデルを介して接続されます

  1. プレゼンターからプレゼンテーションインターフェイスメソッドに渡される

  2. インスタンス自体の表現



主な機能はテストでカバーされます（nUnitとMoqが使用されます）。



問題



確かに、アプリケーションには見つからないバグがまだあります。



さらに、私はアイコンの描画の専門家ではないため、白い背景のシステムトレイにあるアイコンは、アンチエイリアシングがないかのように角張っています。 他のすべての場合、アイコンは少なくとも許容できるように見えます（ただし、ヘルプやアドバイスは拒否しません）。



最初の起動時には、一部のフォルダーがユーザーコンピューター上でしばらく検索されるため、数分待つ必要があります。



ナビゲーションアシスタントがコンピュータの最後のシャットダウン時に閉じられなかった場合、またはシステムの起動時に自動的に開始されなかった場合、現在のディレクトリキャッシュが関連しているかどうかを判断することはできません。 したがって、このキャッシュは追加のストリームで更新され、このプロセスが完了するまで、最後に保存されたディレクトリキャッシュ（実際のファイルシステムに完全に対応していない場合があります）が使用されます。



はい、.Net全般とWPFは低速でリソース集約型ですが、我慢する必要があります。
