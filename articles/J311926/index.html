「開発チームは自己完結型の生物です」：Javaと選択の自由に関するSEMrush





SEMrushはいくつかの理由でジョーカーのスポンサーの中で際立っています。 第一に、事業部門：同社はマーケティング担当者向けのプラットフォームを開発しています。 第二に、彼女のCTOのJava会議への会社の参加により、オレグ・クラスノフはCの愛で知られています。第三に、彼女にとって、このジョーカーが最初になります。



これらすべてのため、会社がJavaをどのように使用しているか、開発アプローチ全体、および会議からの期待を尋ねることはより興味深いだけでした。 私たちが話したのは以下のとおりです。



  • エレナ・ダニレンコ、人事ジェネラリスト
  • Oleg Krasnov 、CTO
  • アナトリーチチコフ 、シニアJava開発者


エレナ・ダニレンコ（人事ジェネラリスト）



-SEMrushはJokerの最初のスポンサーになります-会社は最初の経験から何を期待しますか？



-当社では、作業上の問題のほとんどは個人的なコミュニケーションによって決定されます。 したがって、私たちの目標の1つは、お互いに知り合い、自分自身について話し、興味のあるトピックについてJava開発者とチャットすることです。 さらに、Joker Conferenceのような重要なイベントのサポートに参加できることを大変うれしく思います。 私たちはより多くのネットワーキングのために、良いと異なる:)



-時にはIT専門家は「マーケティング」の概念を嫌います-マーケティングプラットフォームを行っているという事実を恐れていますか？



-「マーケティング」の概念そのものが抑止力である可能性が高いですが、この言葉の背後にあるものが明確ではありません。 しかし、私たちはマーケティング会社ではなく、IT会社です。 私たち自身はマーケティングに従事していませんが、この分野の専門家または市場での地位を理解したい人のためのハイテクオンラインプラットフォームの開発に取り組んでいます。



そして私たちにとって重要なことは、潜在的な同僚が私たちに来る製品領域ではなく、彼が私たちの仕事のアプローチに同意するかどうかです。 私たちは開発チームに意思決定の最大限の自由を与えようとしています-そして、製品と開発プロセスに対して完全な責任を負う準備ができている人々とうまく協力し、両方の品質の改善に取り組んでいます。 候補者とコミュニケーションをとる際には、この立場をできる限り明確にしようとします。



-SEMrushには4か国にオフィスがあり、開発者はどこにいますか（主にJavaに興味があります）？



-私たちは、サンクトペテルブルク、プラハ（チェコ共和国）、リマソール（キプロス）で開発を進めています。 サンクトペテルブルクで最も多くのJavaコマンドがあります。現在7つのコマンドがあります。 来年はこの金額を倍にする予定です。 また、Java開発者はリマソールのオフィスで働いています。



オレグクラスノフ（CTO）





-企業のCTOがCを専門とする場合、Java開発者が仕事を選ぶと、そのような企業での役割をサポートすることに不安を感じるかもしれません。 これで何がありますか？



-SEMrushの特異性はこの言語またはその言語ではなく、次のとおりです。開発チームは自給自足の開発生物であると考えています。 また、開発言語を含め、チーム自体が何を使用するかを選択できます。



これは、自由を追加し、責任を追加しますが、状況を管理するプロセスをより困難にします。 レポートの収集と「スティック」の使用からの管理は、説得と動機付けに変わります。 平等のコミュニケーションは私の好みです。



私たちは、Javaを大規模なコミュニティと開発中のエコシステムを備えた優れたプログラミング言語と見なしていることを付け加えます。 Javaで書かれたプロジェクトは、私たちは素晴らしい存在感と発展を遂げており、その数を増やしたいという自信を与えています。



-2014年に、「データベースを作成するタイミング 」というプレゼンテーションを行いました。 これに関連して興味深いのは、SEMrushが「準備するか自分で行う」という質問に対するアプローチです。



言語の選択の場合と同じです。男は自分たちでどの方法に従うか、自分で何かをするか、標準的なソリューションを使用するかを決定します。 歴史的に言えば、多くの製品で標準ソリューションから自己記述型への移行がありました。ご存じのように、適切に記述されたプライベートソリューションは標準ソリューションよりも優れているためです。



レポートで提示された私の個人的な経験について話すと、開発時には安定したHadoopもMongoもなかったので、状況に応じて行動しました:)



-Java vs C ++に関する最近の記事で 、150を超えるサーバーを搭載した製品について言及しましたが、理解したいのは、SEMrushの規模とは何ですか？ 他にどのような大きな製品がありますか、合計でいくつのサーバーがありますか？



-ハードウェアから始めましょう。 現在、約440台の実サーバーと約250台の仮想マシンがあります。 言語に関する技術開発スタック-ANSI CからNode.js、そしてもちろんJavaまで。



私たちの最大のプロジェクトは、バックリンク、ドメインとフレーズ分析、追跡です。 「バックリンク」-最大で最も重い150台を超える「マシン」。その意味は、Googleのようなすべてのインターネットページのバックリンクのデータベースを構築することです。 「Domain＆Phrase Analytics」は、30を超えるローカルデータベースでの検索結果の分析、60を超えるサーバー、会社の最も古い製品です。 「トラッキング」は、最も動的に開発されている製品の1つであり、28サーバー、検索結果におけるサイトの位置のダイナミクスです。



バックエンドからのこれら3つの製品はC / C ++で記述されていますが、「SEO Ideas」や「Site Audit」など、十分な成功を収めたJava製品もあります。これらについては、当社のWebサイトで詳細をご覧ください 。



アナトリー・チチコフ（シニア開発者）





-Javaの場合（個人的にも、一般的にSEMrushでも）、どのテクノロジー/ツールが使用されていますか？



-最小限の時間でユーザーへの価値の提供を促進するすべてのものを歓迎します。 また、常に最新の状態に保ち、言語とライブラリの最新の安定バージョンを使用するよう努めています。



私が関わったプロジェクトについて話をすると、もちろんJavaフレームワークの場合、これはSpring（Boot、Web、Data、AMQP）です。 テキスト分析にはLuceneを使用します。 日常生活では、Lombok、Guava、およびJava 8ストリームを広く使用していますが、外部ストレージメディアに関しては、MongoDB、Redis、およびRabbitMQです。



他のJavaコマンドでは、タスクによってテクノロジが異なる場合がありますが、多くの場合、コアはSpringとJava 8です。誰かが大量の情報をHDFSに保存し、処理にHadoopを使用します。 誰かが複雑なクエリとトランザクションを気にしている-彼らの選択はPostgreSQLです。 挿入速度とクエリの複雑さの間のトレードオフのために、私たちのようにMongoDBを使用するチームがあります。



ビルドとデプロイ-Maven、Gradle、Fabric、Ansible。 多くのチームが本番環境でDockerを使用しています。



ツール-Intellij IDEA、TeamCity、YouTrack、GitLab。 興味深いことに、私たちのチームはXRebelを非常にうまく使用しています。 また、企業レベルでは、コマンド間レビューコードを導入しています。Upsourceを試しています。



-最近話したい大規模なタスクはありますか？



-私は「SEOのアイデア」と呼べるような大規模なタスクだと思います。私たちのチームが始めて、1年ちょっとの間書いているプロジェクトです。 その目標は、通常のマーケティング担当者には理解できない一連のメトリックと競合他社のデータを、アクションに関する特定の推奨事項に変えることです。 人々はシンプルさが大好きなので、今のところ着実に成長しています。



これは、ストリーミングデータ処理用の分散システムです。 並行処理、ワードプロセッシングタスク、外部APIの操作、それらの負荷分散があります。 サーバービジネスロジックは完全にJavaであり、クライアント部分はJavaScriptで記述されています。 私の意見では、Javaは言語としてもプラットフォームとしても、パフォーマンスと使いやすさの両面で、新しい誇大広告テクノロジー（NodeJS、Goなど）の自信を持っています。 それを変更する誘惑や必要は生じませんでした。



-SEMrushでは、C / C ++で多くのことが行われました。そして、Java開発者として、「Javaの方が良い」という気持ちがありませんか？ :)



-C / C ++の使用は、これらのチームが非常にリソースを消費するタスクに直面するため、一部のプロジェクトでは必要です。 最も顕著な例の1つは、インターネット全体のリンクグラフとドメインランキングです。 Javaを使用すると、ここで多大なコストがかかり、Cを使用するという決定により、他の、それほど必要ではない製品のためにより多くのリソースが解放されました。



-Javaは「自己組織化チーム」アプローチでうまく機能しますか？



-はい、結構です。 私たちの場合、チームは実験と製品の安定性について全責任を負います。ここでは、Javaとその成熟したライブラリが最良の側面を示しています。



また、可能な限り、シンプルで明確なフラットモジュールを作成しようとします。水平接続と変更に対する一定の準備が優先されます。 そして、ここでも、Javaは通常、私たちを失望させません。優れたリファクタリングツールと依存関係解決ツールを備えています。



-言語とテクノロジーを独自に選択するチームの「異質性」により、相互運用性に問題はありますか？



-いいえ、問題はありません。 すべてのSEMrushは基本的にマイクロサービスのセットであり、JSONとRESTをそれらの間の「共通語」として使用します。 場合によっては、大量のデータまたはクエリに関しては、バイナリソリューションが可能ですが、これらはルールよりも例外である可能性が高くなります。



-あなたはGroovyとScalaである程度の経験がありますが、この経験はSEMrushでの作業に関連しており、本番環境でこれらの言語を使用していますか？



-私自身は、これらの言語でSEMrushでほとんど書きませんでした。 しかし、一般的に、Groovyは、あまり積極的ではありませんが、多くを使用しています。 私の知る限り、主にQAは彼を愛し、Spockに美しいテストを書いています。 私自身はスクリプト言語としてGroovyに興味がありました。 一般的に、それは快適で簡潔であり、人々が彼を愛している理由を理解しています。 それでも、私が書いたスクリプトは主にPythonで書かれており、雑用の匂いがするときは最高のツールだと思います。



そして、私が知る限り、Scalaはまだ私たちと定着していません。 しばらく前、私自身がこの言語を綿密に研究しましたが、キャラクターに同意したとは言えません。 この言語は単純ではなく、学ぶのに多くの時間がかかり、各プログラマーは非常に異なる方法で記述します。 ScalaテーマでGitHubを閲覧すると、地球外の知性の存在を感じることがあります。 私の意見では、急速に変化する要件とワンボタンインターフェースの時代では、これらはマイナスの可能性が高いです。



私自身もKotlinに興味を持ち、Javaで製品を積極的に書き続けています。 言語が単純で簡潔で安全であり、プラグマティストがプラグマティストのために作成したことを魅了します。 当社はそのようなツールを愛しています。 試してみたいと思う人々の非常に多くが形成される場合、私たちはそれにプロジェクトがあることを除外しません。
