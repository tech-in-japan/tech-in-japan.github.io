アイデアから1つのボトルでのリリースまで。 クラウドベースの開発プロセス管理-Visual Studio Team Services







開発プロセスを制御するためにどのシステムを選択するかを考えると、いくつかの可能性が思い浮かびます。 多くの場合、大企業はアトラシアンスタック（Jira、Bitbucket、Bamboo）を選択します。 中小企業は通常、JetBrainsソリューション（YouTrack、Upsource、TeamCity）を好みます。 オープンソース愛好家は、GitHub / Lab、Jenkins、および多数のタスク管理ソリューションを実験しています。



いずれにせよ、これは何らかの統合を必要とするソリューションのセットです。



ただし、比較的最近から、開発のすべての段階を管理するための完全に統合されたクラウドソリューションをMicrosoftが提供していることを知っている人はほとんどいません。 このシステムは「 Visual Studio Team Services 」（以降VSTSと呼ばれます ） と呼ばれます。



以下が含まれます。



  • プライベートGitリポジトリ
  • スクラム/アジャイルタスク管理システム
  • 継続的な統合と展開（CI / CD）
  • テストモジュール
  • Power BIと統合して分析を構築する機能


これらはすべて5人のユーザーに無料で提供されます。 BizSparkスタートアッププログラムに参加すると、さらに5つ無料で接続できます。



オープンソース企業になるというマイクロソフトの試みは、このシステムに反映されています。 たとえば、LinuxサーバーでJavaScriptプロジェクトをビルドできます。



システムについて知りたい場合は、先に進んでください。



免責事項。 この記事の著者はマイクロソフトと提携していません。 引用されたすべての事実と考えは、このシステムでの私の個人的な経験にのみ基づいており、より肯定的です。



少なくともロシア市場では、システムの需要が高いとは言えません。 私の意見では、VSTSの人気の低さは、特にマイクロソフト以外のソリューションをサポートするという点で、同社の製品に対する歴史的な不信に関連しています。 また、公平を期すために、システムは現在アクティブに更新されていることに注意する必要がありますが、現在、システムを使用して非常に深刻な問題を解決することを妨げません。



システムには膨大な数の設定がありますが、完全に知り合うために、アイデアを生産的な環境で公開されるアプリケーションに変えるために完了する必要のある最小限のアクションを検討します。



システムをさらに詳しく知るために、すべての名前とスクリーンショットは英語のインターフェースに付けられます。 ロシア語の製品ドキュメントを検索しようとすると、いくつかの問題が発生する場合があります。



そのため、Node.jsにWebサーバーを作成するというアイデアがありました。これは、「Hello World」というフレーズですべての人を歓迎します。 アジャイル手法を使用してこのプロジェクトを管理し、コードを中央のプライベートリポジトリに保存し、コードが変更されたときにアプリケーションが自動的に公開されるようにします。



このプロセスを実装するための最小限の手順を以下に説明します。



青、斜体で、小さいが重要な叙情的な余談。



1. Microsoftアカウントを作成します



個人のMicrosoftアカウント（AppleまたはGoogle Idの類似物）をまだお持ちでない場合は、ここでaccount.microsoft.comを取得する必要があります



2. VSTSアカウントを作成する



次に、前の手順で作成した個人アカウントで、Webサイトwww.visualstudio.com/vsoにアクセスし 、[無料で開始]ボタンをクリックして、VSTSアカウントの名前（「vasya-pupkin」など）を見つけます。



すべてのVSTSアカウントは、visualstudio.comドメインの第3レベルドメインとして登録されているため、自分の名前を美しいものにすることは理にかなっています。 この記事の執筆時点では、「vasya-pupkin」という名前は無料でした:)



VSTSアカウントを作成するときは、最初のプロジェクトも作成する必要があります。 フォームに入力するときは、「詳細の変更」リンクをクリックしてプロジェクト名と管理方法（作業の整理）を選択することを忘れないでください。 プロジェクトの名前は、作成後に変更できますが、管理方法はもうありません。



プロジェクトにNodeProjectという名前を付けて、デフォルトの管理手法（アジャイル）のままにします。









3. Gitリポジトリを作成する



プロジェクトを作成したら、Gitリポジトリを操作するページに移動します。 Codeメインメニューからもアクセスできます。









VSTSには、上のスクリーンショットのように1つのメインメニューと、歯車をクリックしたときに使用できるいくつかの設定メニューがあります。



設定メニューには3つのレベルがあります。



  1. アカウントレベル
  2. チームレベル
  3. プロジェクトレベル


ほとんどのことはプロジェクトレベルで行われるため、デフォルトではプロジェクトレベルの設定に留意します。



Git VSTSリポジトリを操作するプロセスは、GitHubを操作するプロセスと変わらないため、詳細については説明しません。



デフォルトでは、VSTSはhttpsプロトコルを介して動作するように提供していることに注意してください。 SSHでの作業をお勧めします。 初期セットアップには少し時間がかかる場合がありますが、さらに作業を行うとさらに便利になります。









4.バックログでタスクを作成する



コードを保存するためのリポジトリを作成したら、タスク管理サブシステムの構成を開始できます。 プロジェクトを作成するときに、アジャイルプロセスを選択したことを思い出してください。



システムのアジャイルプロセスには、タスクの3レベルの階層があります。 最高レベルはEpic（デフォルトでは無効）です。 次は機能、次にユーザーストーリーです。 最初の近似として、ユーザーストーリーはTaskという単語の同義語と見なすことができます。



Epicを有効にするには、コマンドレベルの設定で[作業]に移動します。









小規模なプロジェクトの場合、Epicレベルは省略できます。



方法論によれば、システム内のすべてのタスクは、アジャイルバックログと呼ばれるタスクの一般的なリストに分類されます。 その後、これらのタスクは特定の時点で実装されます。 これらの時点は反復と呼ばれます。



メインメニューの「作業」機能に「Webサーバー」という見出しを付けて作成し、その中に「Hello world」という見出しのあるユーザーストーリーを作成します。









プロジェクトを実装するには、1回の反復で十分です。 スタブはデフォルトでVSTSで3回の反復で構成されているため、新しいものは作成しませんが、現在のものを構成します。 設定は、名前を考え出し、反復の開始日と終了日を示すことです。 これは、[作業]メニューのプロジェクトレベルの設定で行います。



新しい反復を追加する場合は、コマンドレベルの設定に移動してそこに追加する必要もあります。









反復を作成した後、この反復中に解決する問題を示す必要があります。 これは通常、反復の最初の日に行われます。 今からやろう。 タスクは1つ（ユーザーストーリー）しかないため、作成したばかりの反復にドラッグする必要があります。 タスクを開始したら、アクティブ状態に設定する必要があります。 異なる「トリック」を使用するために、アジャイルは現在の反復の容量セクションに記入することにも意味があります。









ここで、コードとタスクを操作するためのVSTSのセットアッププロセスの概要を終了できます。



次に、継続的インテグレーションと継続的展開（CI / CD）プロセスを構成する、より時間のかかるタスクに進みます。



CI / CDが必要なのはなぜですか？ コードを記述した後、どこかに配置する必要があります。 変更のたびに手動でコードを移植しないように、マスターリポジトリブランチ（または別のブランチ）の各更新後に通常開始するプロセスを構成できます。 CIプロセスは、プロジェクトが正常にアセンブルされ、自動テストで（オプションで）テストされていることを確認します。 CDプロセスは、CIプロセスの結果を本番サーバーに配置します。



一般に、CI / CDプロセスは簡単に構成されず、構成は開発技術とプロジェクトの場所によって大きく異なります。 VSTSシステムでCI / CDを操作する原理を理解するために、ローカルサーバーでホストされるNode.JSアプリケーションの最も単純なケースを検討します。



この例では、（クラウドではなく）ローカルマシンでプロジェクトを収集し、そこで公開します。



CI / CDの構成手順に移りましょう。



6. Build / Deploy Agentのインストール



Build / Deploy Agentは、コードを最終リリースに変換するために必要なアクションをコードで実行するプログラムです。 クラウドベースのエージェントを使用する場合は、この手順をスキップできます。



エージェントは、Windows、Linux、またはOSXマシンにインストールできます。 ローカルLinuxマシンへのエージェントのインストールについては、 www.visualstudio.com / en-us / docs / build / actions / agents / v2-linuxで詳しく説明されています 。



パッケージをダウンロードしてインストールし、エージェントをVSTSに接続するためのトークンを取得して、エージェントをエージェントプールに追加するだけです。



7. CIを構成する



CIプロセスは、メインメニュー項目「ビルドとリリース->ビルド」で構成されます。 2段階で構成されています。



  1. 新しいビルド定義を作成する
  2. ビルド定義へのステップの追加








[新しい定義の追加]ボタンをクリックした後、定義済みプロセスの1つを選択するか、ゼロから構成できます。 最初から設定します。









次のステップでは、Buildをビルドするリポジトリのブランチと、このブランチの更新時に自動的にビルドするかどうかを決定するように求められます。 実際にCIを使用します。 ここでは、6節で構成されたエージェントプールも指定する必要があります。









Node.jsプロジェクトを正常にアセンブルするには、作成したビルド定義に2つのステップを追加する必要があります。



  1. リポジトリからエージェントフォルダーにファイルをコピー（ファイルのコピー）
  2. コピー結果をアーティファクトとして公開します（ビルドアーティファクトの公開）。 この非自明な手順は、CDプロセスとの統合に必要です。 CDプロセスを開始すると、エージェントはこのアーティファクトを取得し、プロジェクトを公開するために必要なアクションを実行します。


これらの手順はどちらもユーティリティセクションで利用できます。









手順を設定するプロセスでは、システム変数を積極的に使用する必要があります。 多くの場合、フィールド名の横にツールチップがあります。









8. CDをセットアップする



同様に、CDプロセスはメインメニュー項目「ビルドとリリース->リリース」で構成されます。



新しい定義を作成した後、CIプロセスの後に実行する必要があるステップを選択します。 CIプロセスの終了直後にCDを実行する場合は、対応するボックスをオンにする必要があります。



この場合、CDプロセスは次の手順で構成されます。



  1. 古いファイルをすべて削除する
  2. 成果物をCIから実動サーバー上の目的のフォルダーにコピーします。
  3. npm install（yarn）を実行して、package.jsonファイルに記述されているnpmパッケージをダウンロードします
  4. 新しいコードでアプリケーションを再起動します。 Node.jsプロジェクトの場合、「pm2 reload my_application」のようになります。








明らかに、それぞれの場合、ステップのセットは異なりますが、VSTSはCI / CDの正常な操作のためにできることのほとんどすべてのオプションを提供します。 エージェントがサーバー上でアクションを実行できるように、エージェントが機能する権限を注意深く監視する必要があるだけです。



すべての設定が完了したので、コードの作業を開始できます。 VSTSの統合機能を最大限に活用するには、次の手順に従います。



1.メインメニューの「作業」のタスク （ユーザーストーリー）に移動して、そこからリポジトリに新しいブランチを作成します。









2.ブランチをローカルコンピューターにコピーし （git pull）、アプリケーションを作成し、同じブランチのリポジトリに変更を保存します（git push）。



3.タスクに戻って、そこからプルリクエストを作成しましょう 。









このプルリクエストを閉じると、マスターブランチへの変更がマージされ、CIプロセスが起動します。CIプロセスが正常に完了すると、CDプロセスが開始され、その結果、本稼働サーバーでアプリケーションが公開されます。



CIの後にCDを自動的に実行する必要は必ずしもありません。 CIの結果を手で制御すると便利な場合があります。 これを行うために、VSTSにはCDを実行するための承認を構成する機能があります。



おわりに



したがって、1つのシステムでアイデアからリリースに至るまで、すべてを進めました。 単一のシステムを使用すると、タスク、リポジトリ内のブランチ、およびビルド間の関係を簡単に追跡できます。









さらに、システムを使用すると、 Power BIと統合するときに、さまざまなセクションのアプリケーションでの作業に関する興味深い統計を取得できます。 1つの例は、人と反復による休業時間の数です
