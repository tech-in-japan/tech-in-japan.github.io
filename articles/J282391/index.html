人々は住宅に住んでいます。 それらがなければ、都市の人口は増加しません。 町民は住宅を取得すると、仕事を探し始めます。 たとえば、私はオフィスで仕事に行くことができます。 事務所は、市の予算に対する税額控除の主な源泉です。 誰かが植物のために働きに行くかもしれません。 工場は資源を生産します-金属とコンクリート。 すべての建物には電気が必要です。 これを行うには、発電所を建設する必要があります。 さらに、一部の建物では自分で作業できますが、プレーヤーのエネルギーは消費され、時間が経つと元に戻ります。 たとえば、プレーヤーには金属がありません。 工場で生産されるまで数時間待つ代わりに、プレーヤーは工場の建物に行ってそこで自分で作業することができます。 そしてもちろん、仕事の本質は「仕事」というラベルの付いたボタンを100回クリックすることです）

パブリッシャーエクスペリエンス



最初のステップは、ゲームのリリース、法的問題、財務上の問題についてプラットフォームを規定する契約を作成することです。 契約に署名する前に、質問のリストを作成し、それらに対する回答を得る必要があります。 例：

-どのような改善が必要ですか？

-改良プロセスにはどれくらい時間がかかりますか？

-契約終了の条件は何ですか？

その後、ゲームをファイナライズするプロセスが開始されます。 パブリッシャーは、完了するタスクのリストを作成します。 便利なプロジェクト管理システムをすぐに決定することをお勧めします。 私の場合、それはGitHubでした。 そこで、ゲームのソース、ビルド、その他のファイルを保存したり、タスクやバグを開始したりできます。





Tap Tap Builderに関する主なタスクは次のとおりです。

-トレーニングシステムの完了

-改善されたインターフェース

-分析ツールの統合

-ゲームのパフォーマンスを向上させる

-ゲームストアのコンテンツの作成（アイコン、バナー、説明、スクリーンショット、ビデオ）

-テスト開発



多くのサブタスクは、実装が複雑であるため、リリースまで延期する必要があったことに注意してください。 テスト開発段階では、提出されたすべてのテストが少なくとも正確でなければならないため、テストは開発者側で繰り返されます。

ゲームが終了すると、パブリッシャー側でのテストと検出されたバグの修正の反復が始まります。

そして、標準に持ち込まれ、ビルドがソフトローンチ（ソフトローンチ）に送られます。 Softlanchは、たとえば、1つの国および1つのプラットフォームでのゲームの限定出版物です。 ソフトランチのタスクは、実際のプレーヤーからフィードバックを得て、ゲームの主要な指標（収入、保持率など）を決定することです。 これらのデータに基づいて、世界リリースの適切性に関する決定が行われます。 ソフトランチが失敗した場合、契約は終了し、ゲームは開発者に返され、開発者はそれを独立して公開できます。

スクリプトの最適化



ゲーム内の問題を熱心に探し始めて、最も可能性の高いボトルネックを最適化しないでください。 最初に行う必要があるのは、Profiler（Unity 5の無料バージョンで利用可能）を実行することです。

スクリプトですぐに問題を発見しました。 修正にはそれほど時間がかからず、すべての最適化は情報のキャッシュと呼び出し回数の削減に限定されました。 ちなみに、Memoizerデザインパターンをお勧めします。 その本質は、同一のパラメーターを使用した後続の呼び出しのために、関数の結果を記憶することです。 パラメータは単純な型だけでなく、構造体とクラスでもかまいません。 以下は、このテンプレートの使用例です。 

public static readonly Func<int, int, int> SumMemoized = Memoizer.Memoize<int, int, int>(Sum); private static int Sum(int a, int b) { return a + b; }


Memoizerテンプレートの実装 
 public class Memoizer { public static Func<TReturn> Memoize<TReturn>(Func<TReturn> func) { object cache = null; return () => { if (cache == null) cache = func(); return (TReturn) cache; }; } public static Func<TSource, TReturn> Memoize<TSource, TReturn>(Func<TSource, TReturn> func) { var cache = new Dictionary<TSource, TReturn>(); return s => { if (!cache.ContainsKey(s)) { cache[s] = func(s); } return cache[s]; }; } public static Func<TSource1, TSource2, TReturn> Memoize<TSource1, TSource2, TReturn>(Func<TSource1, TSource2, TReturn> func) { var cache = new Dictionary<string, TReturn>(); return (s1, s2) => { var key = s1.GetHashCode() + "." + s2.GetHashCode(); if (!cache.ContainsKey(key)) { cache[key] = func(s1, s2); } return cache[key]; }; } }


3Dモデルの最適化



スクリプトの問題が解決したら、レンダリングを実行します。 建物モデルが異なる素材（および色）のプリミティブから作成されたという事実は、パフォーマンスに大きな影響を与えました。 統計では、約600の描画呼び出し（自動バッチ処理後）と100,000の三角形が示されました。 比較のために、現時点では、真空中の球形ゲームでは、最大100のドローコールと最大30,000の三角形を使用することをお勧めします。







最初の解決策は、サーフェスの数を減らすことです。 メッシュを編集するためにアセットをインポートする必要がありました。 多くの建物には円柱が含まれており、標準的な円柱には20の面と80の三角形があります。 円柱のジオメトリを12面に単純化すると、建物の小さな角の損失に明確な効果がもたらされました。



次のステップは、プリミティブから不可視の面を削除することです。 その結果、三角形の総数の大幅な削減（約20％）により、FPSは増加しませんでした。 エンジンは、たとえカメラのスコープ内にある場合でも、おそらく目に見えないジオメトリのレンダリングにリソースを費やしません。 結果は無駄なステップです。



バッチ処理



次のステップは、静的バッチ処理の使用を試みることです。 知らない人のためのバッチ処理について簡単に説明します。 バッチ処理とは、描画呼び出しを呼び出す前に、メッシュ（メッシュ）を1つの共通メッシュにグループ化することです。 ビデオカードが一度に1つのオブジェクトを描画する方が、複数の呼び出しの部分に分けて行うよりも簡単です。 いくつかのニュアンスがあります。 マージは、同じマテリアルを使用するメッシュでのみ可能です。 第二に、頂点の総数は通常900を超えてはなりません。第三に、子オブジェクトの変換（位置、回転、スケール）を変更することはできません。 たとえば、「スキャット」モデルの車輪は回転しません。 バッチ処理は静的および動的です。 動的バッチ処理は実行時に実行され、開発者によるアクションは不要です。 動的バッチ処理の結果は、統計ウィンドウに表示されます。







静的バッチ処理は、2つの方法で実装できます。シーンまたはプレハブ上のオブジェクトの[静的]ボックスをオンにします。 これらは、景観要素、植生、建物などの静的なゲームオブジェクトです。 2番目の方法は、実行時にStaticBatchingUtilityを使用することです。



静的ビルドのバッチ処理を行うのは論理的でしたが、それは私がやったことです。 その結果、ゲームはさらに遅くなり始めました！ 判明したように、強制静的バッチ処理は動的バッチ処理の動作を中断します。動的バッチ処理はより効率的に機能し、たとえば、異なる建物に属する異なる論理オブジェクトからのメッシュをグループ化します。 さらに、自動的にグループ化されたオブジェクトは、互いに相対的に移動できます（グループ化はキャンセルされます）。 結論-静的バッチの使用は、その作業のロジックを深く理解している場合にのみ有用です。



シャドウの最適化



次の観察-ダイナミックシャドウを無効にすると、三角形の数が正確に2倍減少し、FPSが約20％増加しました。 影の品質を変更しても、目に見える効果は得られませんでした。 この状況では、単純な標準ソリューションがあります-影をスプライトに置き換えます。 このような影はしばしば静的と呼ばれます。 問題はすぐに発生します-オブジェクトの影を手で描くのではなく、どのようにオブジェクトの影を取得するのですか？！ 最も簡単な解決策は、ぼやけた円を使用することです。 これは、単純なランナーまたはシューターのキャラクターの影に適しています。 この解決策は私には向いていませんでした-影は建物のジオメトリを繰り返す必要があります。 さて、影をダンプするスクリプトを作成する必要がありました。 アルゴリズムは単純です-スクリプトは（プレハブから）建物を順番に作成し、影付きの建物のスクリーンショットを撮り、ディスクに保存します。 次に、2番目のスクリプトは、取得した画像の基本処理を実行し、影を黒で塗りつぶして、背景（色）を削除します。 残念ながら、既成のソリューションは見つかりませんでした。コメントであなたの経験を共有してください。







その結果、小さなアーティファクトが表示されるとき（シャドウを適用およびレイヤー化するとき）に速度が20％向上します。 ところで、影のあるスプライトには、シャドウなどのパッキングタグをインストールする必要があります。 次にUnityがアトラスを作成し、1回の描画呼び出しですべての影が描画されます。 さらにもう1つ注意してください。一部のデバイスでは、動的な影の表示に問題がある場合があるため、静的な影に置き換えることで問題を回避できます。



3Dモデルのさらなる最適化



残念ながら、最適化の結果は満足のいくものではありませんでした-ゲームは強力なデバイスで20-25 FPSを示しました。 モデルをベイク処理する別のスクリプトを作成することにしました。 一番下の行は、新しいメッシュを生成し、テクスチャを適用して建物をペイントすることです。 テクスチャは、新しいマテリアル（色）が表示されたときに自動的に塗りつぶすことができるシンプルな8x8カラーパレットです。 この場合、正しいUVスキャン（UVマッピング）を作成する必要があります。 Asset Storeにはオブジェクトをベイクするためのプラグインがありますが、マテリアルの色に基づいてテクスチャを生成する方法を知りませんでした。 実装の詳細については説明しません。なぜなら、 私の状況は非常に具体的であることを理解しています。 一方、この方法は現在人気のあるボクセルグラフィックスで用途を見つけることができます。 結局のところ、Unityでモデルのプロトタイプを作成する方が、モデリングを勉強するよりもはるかに便利です。 もちろん、あなたのチームには、私のようなモデラーはいません。 誰もいませんでした）







最終結果はすべての期待を上回りました-ゲームの速度は大幅に向上しました。 ゲームは、大都市で50-60 FPSを示しています。 これ以上の最適化は必要ありません。
