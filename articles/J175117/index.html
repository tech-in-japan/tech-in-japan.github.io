短所とCSS



ソーシャルネットワークのモジュールが外部サイトに接続されている場合の標準ロジックは、iframeの使用です。 外部サイトはセッションをドラッグアウトしないため、これは安全です。外部サイトのcssはiframe内のスタイルに影響しないため、cssは正しいです。 このロジックはここでは壊れています。 ネットワークコードはサイトページに直接埋め込まれているため、事前定義された正方形のブロックを挿入するだけでなく、より多くの機能を利用できます。 セキュリティは、ネットワーク上のサイトとトークンを登録することによって大部分が決定されます。 しかし、CSSは解決されていません。 任意のcssルールを使用してページに要素を挿入する方法がわからず、すべてを壊すことなく、外観を100％知っている（各タグのすべてのcssプロパティをリストする場合を除く）。 したがって、プロトタイプでは、サイトがブートストラップで実行されていると想定しています（ネットワークは、ブートストラップでcssをドラッグしません。これにより、元のサイトの設計が崩れる可能性があります）。

ウィンドウマネージャー



ブートストラップポップアップダイアログがどのように見えるかが大好きです。 私もそのウィンドウjqueryプラグインを使用しようとしましたが、複数のウィンドウと間違ったスクロールを作成するときにフェードのグリッチが防止されました。 嘘をついています。 主な障害は、ノックアウトのすべての力が手元にあるという認識であり、彼はjqueryとDOMで忙しかった。 ノックアウトがある場合、DOM操作が嫌いです。 uncrdでは、DOMを使用した操作は、特別に割り当てられたdomInitメソッドの2つの場合にのみ見られます。 したがって、ウィンドウマネージャーwindowManager.jsは、複数の開いているウィンドウとドラッグアンドドロップをサポートするブートストラップスタイルで記述され、すべてのウィンドウパラメーターは監視可能な変数です。



そこで、1つのエレガントなトリックについて説明します。 windowManagerには、ウィジェットを含むウィンドウを開くためのopenメソッドがあります。 開かれたウィンドウ自体もウィジェットであるため、親のwindowWidgetであるchildrenWidgets配列に自身を登録します。 windowManagerを初期化するときに、this.childrenWidgets = ko.observableArray（[]）を明示的に設定するため、ウィジェットバインディングの通常の配列によってオーバーライドされません。 ここでのnakoutの便利さは、observableArrayが通常の配列と同じpush、pop、spliceメソッドを持っていることです。 したがって、ウィジェット内では、バインディングは重要ではなく、通常は配列または監視可能です。 開かれた各ウィンドウは、observableArrayに登録されます。 これは、後者の変更をサブスクライブできることを意味します。 そして今-childrenWidgetsが空でない場合は、サイトのページを暗くして、コンテンツを位置で「釘付け」する必要があります：修正して完成したウィンドウを表示し、そうでない場合はすべてを返します。



modalWindow3.jsモーダルウィンドウ自体は、ブートストラップからのhtml +コンテンツのサイズ、位置、およびドラッグに応じた配置計算であり、超自然的なものではありません。 唯一の機能は、内部ウィジェットの名前、タイトル、およびフッターを、監視可能な変数によって定義されるサブウィジェットにすることができ、パラメーターが変更されると、ウィジェットが自動的に再作成されることです。 このため、1行を記述する必要はありませんでした。すべては、ウィジェットバインディングのupdateメソッドによって提供されます。

ページにリンクがあり、クリックすると、ユーザープロファイルでモーダルウィンドウを開く必要があるとします。 



 <a href="#" data-uncrd="click:open.bind($data,{name:'profile',id:123})">  #123</a>.




クリックすると、モーダルウィンドウが開き（モーダルウィンドウウィジェットの読み込みは必要ありません）、内部にプロファイルデータが表示され、ユーザーデータを要求するときにダウンロードアイコンが表示されます。 これは不器用に機能します。ネットワーク内をサーフィンすると、クリックするたびに、ダウンロードアイコンのある同じ空のモーダルウィンドウが毎回表示されることがわかります。 これを回避するために、ウィジェットのreadyイベントが使用されます。 次のように書くことができます： 



 <a href="#" data-uncrd="click:open.bind($data,{name:'profile',id:123,loading:'after'})">  #123</a> <!--   --> <a class="uncrd-loading-after" href="#" data-uncrd="click:open.bind($data,{name:'profile',id:123})">  #123</a>.




そして、クリックされると、非同期内部ウィジェットがreadyイベントを発行するまで、モーダルウィンドウは開きません。 代わりに、イベントクリックイベント、event.currentTarget要素があり、loadingプロパティまたはuncss-loading-cssクラスの存在を確認し、currentTarget要素の横に絶対配置の読み込みアイコンを描画します。 後とは、要素の後、前-前、上-リンクの中央（写真とアバター内）を意味します。 ウィンドウを開く原因となった要素に読み込みアイコンを添付できなかった場合は、ウィンドウを不自然に開きます。 後者の場合は、たとえば、ページがリロードされると発生します。初期化時に、location.hashに応じてルーターは対応するウィンドウを開きますが、データのロード中にページ上のどの要素にアイコンを描画するかを知りません。



ライブラリを接続してr.jsでコンパイルする



ネットワークの重要な要件は最大自治であることをもう一度繰り返します。 jqueryやknockoutを含むすべてのライブラリはローカルに存在し、システムのコアからロードされます。 そのため、ネイティブのタグと属性の代わりにタグと属性data-uncrd = "..."を使用し、data-bind = "..."を使用するのは無駄です。 すべてのコアライブラリとスクリプトは、ベアr.jsを使用して単一のmain.jsファイルにアセンブルされます。 ここでの設定はbuild.jsです。 しかし、将来的には、 土を土台として使用することを皆に勧めます。 何らかの理由で、r.jsでは、単一のファイルまたはディレクトリ全体を処理できますが、1つのアクションでカーネルファイルからmain.jsをアセンブルし、その隣にウィジェットのサブディレクトリを配置することはできません。 また、main.jsファイルのrequire.config（{...}）を使用してrequire.jsにプロジェクトがある場合、r.jsを使用してビルドするときにこの構成が考慮されないという事実に備える必要があります-すべてのパスはアセンブリ中に指定されるbuild.jsファイルで再指定されます（ノードr.js -o build.js）。



r.js（ ＃341 ）で名前空間を操作するときにエラーが発生する可能性があります。これは、現時点では、名前空間が正規表現を介してdefineおよびrequire変数に代入されるためです。 knockout.jsライブラリのコードでは、amd環境をチェックする構文がわずかに異なり、名前空間の規則性はそこで機能しません。 解決策はまだありません。出力を確認し、常連を追加するか、ライブラリコードを編集する必要があります。
