Unity3Dの等尺性プラグイン





Unity Asset Storeのプラグインを作成し、ゲームでよく知られている等尺性問題の解決に頭を打ち、コーヒーにお金を持たせる方法、Unityに拡張可能なエディターがあることを理解する方法の物語。 カットの下の写真、実装、グラフィックス、思考。



ネクタイ



それは夕方でした、何もすることがありませんでした。 新年は、プログラマーの生活において特別なことを前兆とはしませんでした（個人とは異なりますが、これはまったく別の話です）。 私の白髪を揺さぶり、少なくとも個人的かつ自分のものを書くというアイデアは熟成し、少なくともいくつかの商業的利益さえも目指しました（雇用者を除いて少なくとも誰かがあなたの開発を必要とする私の魂を温めるため）。 これはすべて、Unityエディターを拡張し、エンジン拡張機能を販売するためのプラットフォームを評価する可能性を長く感じたかったという事実と一致しました。



モデル、スクリプト、さまざまなサービスとの統合、資産保管者の研究のために1日を割り当てました。一見すると、すべてがすでに異なるバージョンで、異なる価格とサポートで、異なる品質の実行と開発で、すでに書かれ、統合されているように見えました 検索をすぐに絞り込みました：



  • コードのみ（プログラマーとして）
  • 2Dのみ（2Dが大好きだから+ Unityの「箱から出して」その正気なサポートが登場したばかりだから）


そして、私は等尺性ゲームを作ったときに何匹のサボテンを食べ、何匹のネズミが死んだか、良い解決策を見つけるのにどれだけの時間を無駄にしたか、この等尺性自体を分類して描く試みでどれだけのコピーが壊れたかを思い出しました。 震えを和らげながら、彼はさまざまなキーワードを検索しましたが、あまり言葉ではなく、アイソメで芸術の山以外は見ませんでした。 さて、アイソメ図用のプラグインがあると判断されました。



目標設定



最初に行う必要があるのは、このプラグインが解決する問題と、それが等尺性ゲーム開発者のハードワークでどのように役立つかを簡単に説明することでした。 等尺性の問題：



  • オブジェクトを正しいレンダリングのために距離でソートする
  • エディターで等尺性オブジェクトを作成、配置、移動するためのエディター拡張機能


最初のバージョンの主なタスクが設定されました。最初のドラフトバージョンの締め切りは2〜3日でした。 遅らせることは不可能でした。熱意は壊れやすいものであり、最初の日に準備ができていなければ、簡単に台無しになります。 そして、新年の祝日は見かけほど長くはありませんが、最初のバージョンを公開する時間を設けたいと思います。



仕分け



簡単に言えば、アイソメトリは2Dスプライトによる3Dモデルのように見せるための試みであり、さまざまな種類の問題に変換されます。 主な問題は、スプライトが誤った相互オーバーラップを回避するために、描画される順序で並べ替えられる必要があることです。





スクリーンショットでは、最初に緑のスプライト（2,1）が描画され、次に青のスプライト（1,1）が描画されます





スクリーンショットは、最初に青が描かれたときに間違った並べ替えを示しています



この単純な場合のソートは難しくありません。たとえば、次のようなさまざまなオプションがあります。



  • 画面位置Yでソートします。これは、 （isoX + isoY）* 0.5 + isoZです。
  • 左から右、上から下へ最も遠い等尺性セルから描画します[（3,3）、（2,3）、（3,2）、（1,3）、（2,2）、（3,1）、 ...]
  • そして、多くの興味深い方法で、あまりそうではありません


それらはすべて良好で高速で動作しますが、そのような単細胞オブジェクトまたは細長い（isoZ）カラムの場合のみ:)私は同じ座標に沿って細長いオブジェクト、または幅のない「フェンス」のより一般的なソリューションに興味がありましたただし、希望する高さで同じ方向に伸びます。





スクリーンショットでは、サイズが3x0および0x3の 「フェンス」を持つ3x1および1x3の細長いオブジェクトが正しくソートされています



そして、ここから問題が始まり、どこへ行くかを決める必要がある場所が始まります。



  • 「マルチセルラー」オブジェクトを「ユニセルラー」オブジェクトに分割します。 垂直ストリップにカットし、これらのストリップを並べ替えます
  • より複雑で興味深い別のソートオプションを考えてください


私は2番目のオプションを選択しました。私は論理の特別なアプローチである切断（自動）で、芸術の特別な準備に関与したくありませんでした。 参考までに、最初の方法はよく知られているFallout 1およびFallout 2ゲームで使用されており、ゲームデータを消化するとこれらのストリップを観察できます。



2番目のアプローチは、オブジェクトのソート基準を意味しません。つまり、オブジェクトをソートするための特別な計算値はありません。 誰も信じない（そしてアイソメを行わなかった多くの人-信じない）、1枚の紙を取り、 2x8や、たとえば2x2のオブジェクトを描きます。例として8x2 レンズを追加し、互いに対して異なる位置に並べ替えてみてください。



まあ、数はありませんが、それらの間の関係を計算できます（大まかに言うと、誰かが重複します）。これは、 トポロジカルソートで使用できます。 オブジェクトの依存関係は、アイソメトリック軸上のアイソメトリック座標の投影を使用して計算できます。





スクリーンショットでは、青い立方体は赤に依存しています





スクリーンショットでは、緑の立方体は青に依存しています



2つの軸への依存を決定するための擬似コード（Z軸も同様）： 



bool IsIsoObjectsDepends(IsoObject obj_a, IsoObject obj_b) { var obj_a_max_size = obj_a.position + obj_a.size; return obj_b.position.x < obj_a_max_size.x && obj_b.position.y < obj_a_max_size.y; }


このアプローチを使用して、オブジェクト間の依存関係を構築し、表示Z座標を設定することにより、すべてのオブジェクトとその依存関係を再帰的に走査します。 このアプローチは非常に普遍的であり、最も重要なことはそれが機能することです。 このアルゴリズムのより詳細な説明は、たとえばhereおよびhereにあります 。 また、このアプローチは、 アイソメトリック用の一般的なフラッシュライブラリ（ as3isolib ）でも使用されています。



すべては問題ありませんが、このアプローチのアルゴリズムの複雑さはO（N ^ 2）です。依存関係を構築するには、各オブジェクトをそれぞれと比較する必要があるためです。 しかし、最適化は後のバージョンに延期されました。ただし、何も動かないときに何もソートされないように、遅延再ソートを実行しました。 最適化については後で。



エディター拡張



目標は次のように設定されました。



  • オブジェクトのソートは、ゲーム内だけでなくエディターでも機能するはずです
  • 他のギズモ矢印（オブジェクトを移動する矢印）が必要です
  • 移動時のタイルのオプションの配置
  • 等尺性ワールドインスペクターでのタイルサイズの自動適用とそのタスク
  • 等尺性の寸法でAABBオブジェクトを描画
  • オブジェクトのインスペクターでの等尺性座標の表示。ゲーム世界でオブジェクトのどの位置が変化するかを変更するとき


すべての目標が達成されたため、Unityを使用すると、エディターを大幅に拡張できます。 オブジェクトインスペクターに新しいタブ、ウィンドウ、ボタン、新しいフィールドを追加します。必要に応じて、希望するタイプのコンポーネント用に独自のカスタムインスペクターを作成することもできます。 また、エディターウィンドウに追加情報（私の場合はAABBオブジェクト）を表示して、オブジェクトを移動するための標準の矢印を置き換えることもできます。 エディター内でのソートは、マジックフラグExecuteInEditModeによって決定されました。これにより、オブジェクトのコンポーネントをエディターモードで実行できるようになります。つまり、ゲームと同じことができます。



もちろん、これはすべて、問題やあらゆる種類のトリックがないわけではありませんが、数時間以上苦労することはありませんでした（Google、フォーラム、コミュニティは、ドキュメントに記載されていない問題の解決を支援しました）。

アルゴリズム



100個のオブジェクトとO（N ^ 2）を使用して、依存関係を見つけるためだけに1万回のチェックがありましたが、それらすべてを調べて、表示Zをソートするように設定する必要もあります。 何かを解決する必要がありました。 最後に、私は膨大な選択肢を試しましたが、決断に苦しんで眠れませんでした。 テストしたメカニズムのすべてを説明するのではなく、現時点で停止したことをより詳しく説明します。



最初に、もちろん、表示されているものだけをソートします。 これは、フレーム内に誰がいるのか、フレームに表示されるのか、並べ替えるオブジェクトを追加するのか、退出するときにそれを忘れるのかを常に把握する必要があることを意味します。 Unityは、オブジェクトのバウンディングボックスをシーンツリー内の子と一緒に認識し、すべての子を実行する（そして、追加と削除が可能なため毎回）オプションを選択することはできません。 OnBecameVisibleイベントとそのilkも、親オブジェクトに対してのみ機能するため、不可能です。 ただし、目的のオブジェクトとその子からすべてのレンダラーコンポーネントを取得する機会があります。 このオプションは美しさでは輝きませんが、許容できるパフォーマンスを備えた同じ普遍的な方法は見つかりませんでした。 



 List<Renderer> _tmpRenderers = new List<Renderer>(); bool IsIsoObjectVisible(IsoObject iso_object) { iso_object.GetComponentsInChildren<Renderer>(_tmpRenderers); for ( var i = 0; i < _tmpRenderers.Count; ++i ) { if ( _tmpRenderers[i].isVisible ) { return true; } } return false; }
GetComponentsInChildren関数が使用されるという微妙な点があります。これにより、コンポーネントを含む新しい配列を返すものとは異なり、目的のバッファに割り当てずにコンポーネントを取得できます。



第二に、 O（N ^ 2）で何かをする必要があります。 空間を分割するための多くのオプションを試しましたが、表示スペース内の単純な2次元グリッドに落ち着きました。そこでは、アイソメトリックオブジェクトを投影します。各セクターには、交差するアイソメトリックオブジェクトのリストが含まれます。 考え方は単純です。オブジェクトの投影が交差しない場合、オブジェクト間の関係を構築しても意味がありません。 次に、表示されているすべてのオブジェクトを実行し、必要なセクターでのみ依存関係を構築します。これにより、アルゴリズムの複雑さが軽減され、貴重な速度が向上します。 すべてのオブジェクトのサイズの平均としてセクターサイズを選択しました。結果は私に適しています。

2Dと3Dを混在させる



等尺性ゲームで2dと3dを混合することは、動きと回転のさまざまなオプション（たとえば、アニメーション付きのキャラクターの3Dモデル）のレンダリングを最小限に抑える興味深い機能です。 難しくありませんが、すべてを分類システムに組み込む必要があります。 すべての子を含むバウンディングボックスモデルが必要で、モデルをディスプレイZに沿ってその幅まで移動します。 



 Bounds IsoObject3DBounds(IsoObject iso_object) { var bounds = new Bounds(); iso_object.GetComponentsInChildren<Renderer>(_tmpRenderers); if ( _tmpRenderers.Count > 0 ) { bounds = _tmpRenderers[0].bounds; for ( var i = 1; i < _tmpRenderers.Count; ++i ) { bounds.Encapsulate(_tmpRenderers[i].bounds); } } return bounds; }
そのため、すべての子でバウンディングボックスモデルを計算できます





そして、それはすべて最終的に判明します
