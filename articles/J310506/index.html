ユニバーサルモバイル電子キー

「GoogleのDevice Lab」プロジェクトに参加するよう申し出られたとき、私は非常に興奮したとは言えませんが、もちろん興味がありました。 Bluetoothを介してさまざまなやり取りを行うプロジェクトを実行すると、非常に興味深い実装が得られました。 ただし、Bluetoothの操作中に、所有者がBluetoothの動作範囲内で意図せずに電源を入れたままにしたすべての電話機は、通常より3倍速く充電を失いました。 もちろん、同僚はあまり幸せではありませんでした。 プロジェクトは閉じられなければなりませんでした。 この点で、私は「最後の世代」のビーコンを「粘り強い」手に保持したいとずっと思っていました。 そして、Device Labプロジェクトは私にそのような機会を与えてくれました。









Google Lab Labコンテストの一環として、著者Dmitry Senashenko（ @DmitrySen ）による記事 。



現在、市場には多くのビーコンとそのメーカーがいます。 そしてそれらのほとんどは、建物内の地理的位置を決定するように設計されています。 ビーコンをスーツケースに埋め込んで、空港で信号でビーコンを検索するというカテゴリの興味深いアイデアがあります。 しかし、力があれば、おもちゃのように見えます。 独創的で便利なものを思いついて同時に実装したかったのです。 そのため、ウェブサイトとHabrのオフィスでのいくつかの手続きと、私の手にあるiBKSビーコン：









灯台については多くのことが書かれています。 例：「 Googleのビーコンプラットフォーム。 パート1 「または」 Googleのビーコンプラットフォーム。 パート2 」。 繰り返しはしませんが、Googleのビーコンプラットフォームでビーコンを使用し、APIを介して操作することになっていたと言います。 現実には「i」が点在しています。これについては後で説明します。



私の人生では、電子メール、html、smsなどをサポートするテレフォニーとインテリジェントIVRを使用しています。 したがって、既製のスタンドがあり、必要な機器などを簡単に入手できます。 そして、このトピックに対して、ビーコンを適応させたかったのです。



アイデア



そこで、ビーコンをオフィスのドアへの認証キーとして使用するというアイデアが思いつきました。 誰もが知っているように、すべてのオフィスには、電子ロック付きのドアと入り口にリーダーがあります。 各従業員はカードの形のキーを持ち、ドアに入るためにこのカードをリーダーに取り付ける必要があります。その後、ドアが開きます。 世界では、そのようなドアの数は数十億に達し、カードの数はおそらく1兆になります。 実際にそれらを使用することはあまり便利ではありませんが。 重要な会議に急いでいる場合、カードを抽出する（または、このカードが置かれているオブジェクト（財布、ドキュメントのあるフォルダー）を抽出する）のに時間を浪費するのは非常に退屈です。



ただし、各人のポケットには電話があります。 また、一部にはハンドヘルドエレクトロニクスもあります。 電話を忘れるよりも、カードを忘れた人に会う方が簡単です。 したがって、カードを電話に、リーダーをビーコンに置き換える方が便利で経済的です。 もちろん、この方向には良いアイデアがあります。



たとえば、指紋リーダー付きのドアハンドル、電話のNFCなど。 しかし、NFCテクノロジーはすべてのスマートフォンに搭載されているわけではありませんが、Bluetoothはすべてのスマートフォンに搭載されています。 したがって、技術的にはこのアイデアは実際に実装する準備ができています。



さらに、このソリューションにより、従業員の快適性が向上します。 ドアに近づいて、アプリケーションアイコンをクリックするだけで、承認後、ドアが開きます。 つまり エレベータからでも、ドアに近づいてでも開くことができます。 また、時計からドアを開けたり、マイクを使ってヘッドフォンのパーソナルアシスタントに話しかけたりすることもできます。 重要な要素は認可です。 これをもっと詳しく見てみましょう。 図1を参照してください。



作業の概略図



電話機はビーコンの可視範囲に入り、Beacon Discoverを使用してビーコンを検出し、Nearby messages APIを介してGoogleビーコンレジストリから添付ファイルを受信します。 次に、暗号化されたセッションでWifiまたはモバイルインターネットを介して、電話はこの添付ファイルとIMEIまたは他の一意のコードセットを承認サーバーにインストールするときに送信します。 許可サーバーはアクセス権を識別し、そうであれば、ドアを一意に特徴付ける添付ファイルからのデータに基づいて、対応するロックを開くためのコマンドを電子ロック管理サーバーに与えます。 電子ロック制御サーバーは、アクチュエーターにドアを開くように指示します。









図1.相互作用の編成のスキーム



ここで私に起こった解決策があります、そして今それを実装してみましょう。



私たちは鉄のアイデアを具現化します



G250ゲートウェイを備えたAvaya PBXを自由に使用できました。 これは古いゲートウェイであり、長く廃止されました。 しかし、その新しい代替品であるG450 / G430ゲートウェイには、必要なコンポーネント、つまり電子ドアロック制御ドライブも含まれています。 小規模オフィスのニーズに対応するため、このゲートウェイにはドライリレーが組み込まれています。いわゆる接点閉鎖補助装置です（図2を参照）。









図2. Avaya Gateway G430



このコネクタは、事前定義された電話番号のダイヤルを制御するために使用できます。 この番号をダイヤルすると、ゲートウェイはリレーを閉じ、電子ロックを開くための信号を送ります。



さらに、AvayaがプロデュースしたIVR（Interactive Voice Response）を備えたワークスタンドも利用できました。 これは完全にソフトウェアのサービスであり、音声リクエストだけでなく、メール、SMSも処理できます。このソリューションでは、htmlリクエストが必要になります。 また、このIVRは、電子メールとSMSだけでなく、音声通話も開始できます。 つまり このプロジェクトに関連して、このIVRは変数の転送を伴うhtmlリクエストを受け取り、これらの変数の処理結果に基づいて、特定の番号への音声通話の発信を開始できます。 IVRでの要求処理は、組み込みツールとJavaプログラミング言語の両方を使用して実行できます。 この点で、このIVRは許可サーバーとして使用されました。 G250ゲートウェイを備えたAvaya Aura Communication Manager PBXが電子ロック制御サーバーとして使用されました。 電子ロックも利用できました。



したがって、携帯電話のアプリケーションは、ビーコンの識別子を取得し、Googleビーコンレジストリに連絡し、その添付ファイルを取得する必要があります。その後、htmlリクエストでこの情報とIMEIをIVRに送信します。 G250の接点閉鎖ドライリレーを短絡させて、ドアを開けます。



さて、実装の準備の最後のステップは、Androidオペレーティングシステムを実行する携帯電話を入手することでした。 これは、Android Lollipop 5.1.1を搭載したHTC Desireでした。 もちろん、ラップトップにAndroid Studioをインストールする必要もありました。



すべて準備が整ったので、私は準備を整えました。 いつものように、現実はiに点在しています。



落とし穴



まず、ビーコンをGoogleビーコンレジストリに登録する必要があります。 これを行う最も簡単な方法は、Google Play MarketとApple App Storeの両方で利用可能なGoogle Beacon Toolsモバイルアプリを使用することです。



ビーコンをオンにして、マイクロ回路とバッテリーの間のプラスチックの断路器を取り外し、Googleビーコンツールをインストールして、ビーコンの検索を開始しました。 彼は現れませんでした。 次に、メーカーのアプリケーションiBKS Config Toolをダウンロードしました。 このアプリケーションは、AndroidデバイスとiOSデバイスの両方でも使用できます。 そして-見よ！ ビーコンが現れました。









図3. iBKS構成ツール

図6. Googleビーコンツール



登録用のGoogleビーコンツールでは、ビーコンをiBeaconまたはEddystone UUIDモードにする必要がありましたが、このモードでは頑固に表示されませんでした。 Eddystone URLモードでは、彼はそれを見ましたが、登録を拒否しました。 AndroidとIOSでのこのアプリケーションの動作は同じでしたが、Androidではアプリケーションが最初にプロジェクトを選択し、その後ビーコンの登録を拒否しました。



ビーコンのこの動作の理由は何でしたか、プロジェクト中に見つけることができませんでした。 ビーコン自体に問題があったようです。 もちろん、交換する価値はありましたが、プロジェクトの期間は限られており、Habrのオフィスへの新しい旅行の時間はありませんでした。 iBKS Config ToolおよびGoogle Beacon Toolsを使用した実験から、ビーコンが発信し、一意の識別子を送信していることが明らかになりました。 したがって、デモスタンドのビーコンは、Googleビーコンレジストリから受信した添付ファイルではなく、その識別子によって識別することに決定されました。 一般に、プロトタイプソリューションの場合、このような置換は非常に適切です。



そのため、Android Studioがビーコン識別子の検出を整理できるようにするJavaライブラリを見つけるだけです。 もちろん、GitHubとRadiusnetwork.com.eddystonedemoライブラリが助けになりました。 このライブラリを使用すると、ビーコンによってブロードキャストされる識別子を確認できます。これは、iBKS Config Toolでも表示できます。 ここで説明します 。



大きな灰色のボタンがあるアクティビティが編成されました（図7を参照）。 このボタンは、ビーコンを見つけた後、Clickableの色、名前、およびプロパティを変更しました。 つまり、ビーコンが見つかるまで、ボタンを押しても何も起こりません。 ビーコンが見つかるとすぐに、ボタンが赤に変わり、クリックできます。 ボタンが押されると、URLが呼び出され、IMEI送信でIVRアプリケーションにつながります。IMEIはそれをチェックし、成功すると、SIPを介して電子ロック制御サーバーに呼び出しを形成します。 ビーコン信号が失われた場合（それまでの距離が長くなるか、オフになる）、ボタンの色が変わり、Clickableプロパティが元に戻ります。
