Veeam Backup＆Replication 9.0の新機能：バックアップ作成とストレージ効率の改善

この投稿では、Veeam Backup＆Replicationのバージョンを9.0にアップグレードすることで有効にできる新機能について説明します。 さまざまな新製品の中で、私はいくつかのお気に入りを持っていることを認めています-それらは外観が目立たないように見えるかもしれませんが、大きな利益をもたらします。 もちろん、1つの投稿で全体を説明するわけではないため、すぐに公式ウェブサイトの新製品の全レビューへのリンクを提供します： Veeam Backup＆Replication v9の新機能 。

  • 新しいリポジトリ機能
  • 新しいデータストレージ機能

リモート管理コンソール



最後の投稿には、この新しいコンポーネントの概要がすでにありました。 コンソールを使用すると、リモートのVeeamバックアップサーバーに接続できます。 個別のセットアップとして設定できますが、自動インストールもあります（ ユーザーガイドの説明を参照 ）。 したがって、Veeamは完全に分散されたアーキテクチャをサポートするようになりました。 以前のバージョンでは、Veeam.Backup.Service.exeとVeeamShell.exeのコンポーネントは排他的に共存していました。v9では、エンジニアの努力が最終的に相互依存関係を打ち負かすことに成功しました。





新しいリポジトリ機能



バックアップリポジトリの操作には新しい機能があります。



まず、マウントサーバーを任意のリポジトリに割り当てることができます。 これは、バックアップインフラストラクチャで役割を果たすWindowsサーバーです。ファイルレベル（FLR、ファイルレベルの復元）で復元する場合、このようなサーバーはVMファイルをバックアップからファイルシステムに直接マウントします。 接続サーバーは、リポジトリー構成ウィザードの適切なステップ（ サーバーのマウント ）で設定できます。





マウント操作を有効にするためのさまざまなオプションがあります。

  • したがって、アプリケーションのファイルまたはオブジェクトを復元するには、コンソールが実行されているマシンで最初にインストールが実行されます-表示ツールはコンソールから起動されるため、ユーザーはバックアップの内容を表示し、リカバリに必要なファイル/オブジェクトを選択できます（オフラインであるかサーバーと組み合わされているかは関係ありません）、 Veeam Backup BrowserまたはVeeam Explorerツールのいずれかです。
  • 選択したファイル/オブジェクトを復元するコマンドが与えられた後（通常は運用中のマシンに）、接続サーバーはリポジトリでの作業を開始し、トラフィックはターゲットマシンとの間で行き来します。 これにより、分散インフラストラクチャの復旧が加速されます。
  • Enterprise Managerを使用してリカバリを実行し、バックアップ中にゲストOSファイルのインデックスが作成されたマシンに対して実行すると、1つの接続のみが表示されるため（表示とリカバリの両方）、この場合、トラフィックは常に接続サーバーとターゲットVM。


EnterpriseおよびEnterprise Plusのライセンシーは、スケーラブルなリポジトリを作成できます。 これについては、Habrの記事「無限に「スケーラブルなバックアップリポジトリ 」 についての記事で詳細に説明されています 。 このリポジトリは管理が容易です（プールからターゲットストレージを選択するルーチンタスクが自動化されているため）。さらに、デフォルトで「VMごとのバックアップチェーン」モードがオンになっているため、プール内のストレージのストレージスペースの燃料消費が最適化され、パフォーマンスが向上します





通常のリポジトリの場合、「マシンごと」（VMごとのバックアップファイル）にバックアップコピーを保存するオプションも提供されます。 このオプションを使用すると、VMごとに（タスクごとではなく）記録ストリームが生成されます。 その結果、単一のスレッドのサービス（キューの深さ= 1）がほとんどのストレージシステムのパフォーマンスを制限する要因になるため、キューの深さが増加し、パフォーマンスが向上します。 複数のスレッド（キューの深さ〜= N、つまり、リポジトリ上のタスクの数）を作成すると非常に役立ちます。





さらに、大規模なバックアップの制御の難しさについて、予約なしでより大きなタスクを作成することが可能になりました。 v9のリリースに伴い、システムアーキテクトは、1つのタスクに含まれるVMの数に関する推奨事項を修正する準備をしています-たとえば、250〜300にします。 いずれにせよ、当社独自のテストにより、5,000台の仮想マシンを含むタスクを処理できる可能性が確認されています。



もちろん、バックアップタスクを計画および設定する場合、いくつかの操作（たとえば、合成バックアップの作成、バックアップコピージョブタスク（バックアップサイトへのバックアップの転送）など）がすべてのVMがバックアップを完了するのを待つことに注意する必要があります。 このため、巨大な仕事はお勧めできません。



重要！ バックアップストレージをVM-backupファイルごとに使用するオプションを選択した後、手動モードで、またはスケジュールされた操作を待ってから、アクティブフルバックアップ（アクティブフル）を作成する操作を実行して、アクティブにする必要があります 。



さらに2つのポイント：ソース側での重複排除はどこにも行かないことに注意してください。 2番目-EMC DataDomainストレージシステム（DDBoostを使用）、HP StoreOnce（Catalystを使用）、およびExaGrid（Accelerated Data Moverを使用）で新しいリポジトリを作成する場合、このオプションはデフォルトで有効になっていますが、万が一に備えて、これを確認することをお勧めしますこれらのストレージ設定。



新しいデータストレージ機能



アップグレード後、新しいVeeam Backup＆Replication機能はデフォルトで無効になります。 それらの使用を開始するには、既存のバックアップジョブを再構成する必要があります。 以下のリストの最初の2つのオプションは、以前はバックアップコピージョブ（バックアップコピージョブ）でのみ使用可能でした。 これで、他のタイプのタスクを作成するために使用できます。 これらの新機能は次のとおりです。



デフラグと圧縮。 このオプションの使用は、バックアップジョブに定期的なフルバックアップ（アクティブフルバックアップ）の作成が含まれておらず、データストレージポリシーに定期的なフルコピーファイルと最も近い増分バックアップの組み合わせが含まれる場合に役立ちます。 このような変換により、バックアップファイルが断片化され、その中に「ボイド」が形成されるという事実につながります。 VMの一部がバックアップジョブから削除されたときにも、ボイドが形成されます（バックアップから削除した後、ファイル自体は小さくなりません）。 ファイルサイズを小さくし、ファイルの操作速度をわずかに上げるには、[ 最適化と完全バックアップファイルの最適化]オプションを有効にします。 次のように機能します。指定された時間に、Veeam Backup＆Replicationは空のVBKファイルを作成し、そこからの変換の結果として必要なブロックのみをバックアップファイルからコピーします。 このオプションには制限があり、 ユーザーガイドで説明されています 。



ヘルスチェックと自己修復。 SureBackupに加えて、バックアップが破損していないことを確認するために、[ バックアップファイルの正常性チェックを実行する ]オプションをアクティブ化できます。 この場合、各バックアップを作成するときに、Veeam Backup＆Replicationは作成されたファイルのハッシュとチェックサムを計算します。 次に、次のバックアップを開始する前に、Veeam Backup＆Replicationはファイルのハッシュとチェックサムを再計算し、以前に保存した値と比較します。 ファイルデータが破損していることが判明した場合、Veeam Backup＆Replicationは、元のVMまたは以前のバックアップから必要なデータをコピーすることにより、破損したブロックを置き換えます。





削除済みとしてマークされたブロックを除外します。 NTFSでファイルを削除する場合、実際にはファイルを含むデータブロックをクリアするのではなく、削除済みとしてマークします。 したがって、VMイメージのバックアップイメージを作成する場合（イメージベースのバックアップ）、バックアップファイルのサイズは、ファイルシステムに表示されるサイズを大きく超える可能性があります。 最近まで、「削除された」データのコピーを回避する唯一の方法は、バックアップを開始する前にsdeleteユーティリティ（SysInternals）を使用することでした 。 Veeam Backup＆Replication v9では、 BitLookerオプションを有効にして、バックアップを開始する前にVeeamがMFTパーティションを分析し、削除済みとしてマークされたブロックをコピーしないようにすることができます。





BitLookerの仕組みに関する詳細な記事「 Veeam Backup FREE Editionの新しいバージョン：簡単なレビュー 」がリリースされました。 この技術は現在特許出願中です。



重要！ Veeam Backup＆Replicationの商用バージョンでは、アップグレード後、このオプションはデフォルトで無効になっています-PowerShellコマンドを使用してアクティブ化すると便利です：

asnp VeeamPSSnapin; Foreach ($job in Get-VBRJob) {

$job.Options.ViSourceOptions.DirtyBlocksNullingEnabled = $true;

$job.SetOptions($job.Options) }









ゲストOSを操作するための新機能



ゲストOSとの対話の最適化。 Veeam Backup＆Replication v9に基づいて構築されたバックアップインフラストラクチャには、新しいコンポーネント-Guest Interaction Proxyが登場しました。 これは、バックアップサーバーとゲストOSとの相互作用を目的としています（以前は、この相互作用はバックアップサーバー自体によって実行されていたため、負荷が大幅に増加しました）。 ゲストインタラクションプロキシロールの自動マシン選択を構成すると、Veeamは、バックアップが予定されているVMと同じサイトにある負荷の少ないマシンを選択します。 このアプローチにより、リモートオフィスおよびブランチでのバックアップが簡素化され（コンポーネントが同じサイトにある場合、コンポーネントの相互作用が速くなります）、VMware VIXを使用する必要がありません。





バックアップ中の不要なファイルの除外。 この機能はBitLookerテクノロジーの一部です。バックアップ時に、VeeamはMFTパーティションを分析し、プロキシサーバーのメモリにキャッシュを作成します。このファイルでは、除外するファイルとフォルダーをマークします。 バックアップ中、データはソースVMとキャッシュから同時に読み取られます。 ファイルの除外は、アプリケーションの状態を考慮してバックアップオプションをアクティブにした場合にのみ可能であることに注意することが重要です（アプリケーション対応の画像処理）。 さらに、例外処理によりプロキシサーバーの負荷が増大するため、多くの小さなファイルの除外は、同じサイズのいくつかの大きなファイルの除外よりもゆっくりと処理されます。 たとえば、100,000ファイルをバックアップから除外するには、プロキシサーバーに400 MBのメモリが必要です。





転送ジョブでの完全バックアップコピーの構成



さて、別の操作が自動化されました-以前、バックアップコピージョブの完全バックアップを作成するために、ユーザーはPowerShellスクリプトを使用し、 増分ステップから合成するのではなく、ソースバックアップから復元ポイント全体を読み取るをチェックできますジョブを設定するときにターゲットを設定します。





このオプションは、パフォーマンスの低いストレージシステムに役立ちます。このシステムでは、ファイルの変換、増分バックアップから完全バックアップの合成、非効率的で、完全バックアップのコピーがはるかに便利です。
