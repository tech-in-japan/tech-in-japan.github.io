ProxmoxからVMmanagerへの移行

1〜2個の仮想サーバーを備えたホームサーバーからDCの複数の高性能サーバーに既に成長しているホスティング事業者は、日常的なアクションの自動化と仮想サーバーの管理について考えています。

最も人気のある無料の仮想化マネージャーはProxmoxです。

その利点（無料、オープンソース、コミュニティ）には、これらの利点をカバーする以上の欠点もあります。







  • 有料アップデート
  • 有料テクニカルサポート
  • ロシア語の公式ドキュメントの欠如
  • 専門外のローカリゼーション
  • 相対的なインストールの複雑さ
  • すべての追加機能と統合は有料です。
  • サードパーティの開発者によって作成された多くのコンポーネント






使い慣れたソフトウェアから新しいものに切り替えることは非常に難しく、新しいソフトウェア製品を研究することに加えて、すべてのデータを移行する必要があります。 これは非常に難しいステップであり、可能な限り徹底的にアプローチする必要があります。 多くの場合、無料または安価な製品の選択は、必要な機能の追加、使用する他のソフトウェアとの統合、ローカリゼーションなどの追加コストにつながります。



ISPsystemソフトウェア製品を使用すると、ホスティング事業者は顧客にサービスを提供するプロセス全体を完全に自動化できます。



この記事では、VMmanagerの制御下でProxmoxから仮想マシンを転送する方法を説明します。 VMmanagerは、他の仮想化マネージャーからのコンテナーおよび仮想サーバーのインポートまたは移行をサポートしていません。 ただし、VMmanager APIを使用してこれを行うことは難しくありません。

コンテナをProxmoxからVMmanager-OVZに移行するオプションを検討してください。



ProxmoxからVMmanager-OVZへのコンテナの移行。



移行は、1つのサーバーで実行できるという事実によって簡素化されます。

VMmanager-OVZは、Proxmoxが動作するサーバーと同じサーバーにインストールできます。問題なくコンテナーの動作に干渉することはありません。



これを行うには、インストーラーファイルをダウンロードして実行します-

http://download.ispsystem.com/install.5.sh

次に、いくつかの質問に答えて、必要なソフトウェア製品とそのバージョンを選択します。

公式リポジトリも自動的に接続され、コントロールパネルと関連ソフトウェアのインストールが開始されます。

VMmanager-OVZでは、proxmoxのopenvz設定が問題なく使用されることに注意してください。

唯一の注意：VMmanager-OVZをインストールした後、openvzのサンプル構成ファイルを手動でダウンロードする必要があります。 コンテナを作成する必要があり、proxmox配布キットには存在しないため、ログにコンテナを作成しようとすると、次のエラーが表示されます。



2014-12-03T10：39：28 + 0800 vzctl：CT 100：サンプル構成/etc/pve/openvz/ve-basic.conf-sampleが見つかりません：そのようなファイルまたはディレクトリはありません

2014-12-03T10：39：28 + 0800 vzctl：CT 100：コンテナプライベートエリアの作成に失敗しました



公式リポジトリからopenvzのサンプル設定ファイルをダウンロードし、あるべきディレクトリに配置します：



＃wget --no-check-certificate github.com/blueboxgroup/vzctl/blob/master/etc/conf/ve-basic.conf-sample -O /etc/pve/openvz/ve-basic.conf-sample



乗り換えるときは一点に注目したい。 コンテナの識別番号が一致しない可能性があります。

どちらの場合も、番号付けは100から始まります。Proxmoxで作成されたコンテナーの任意のIDを選択できますが、VMmanagerではこのカウンターは100から始まり、リセットされません。すべてのコンテナーを削除した後、削除されたものの次の番号から番号付けが続行されます。 これには1つの解決策があります。以前に作成したすべての仮想サーバーを削除し、vmmgrプロセスをクラッシュさせる必要があります。その後、番号付けが先に行われます。



VMmanager-OVZのパラメーター設定



  • コンテナーを管理する管理ユーザーを作成する
  • コンテナ所有者ユーザーを作成する
  • 作成されたコンテナのIPアドレスが発行されるアドレススペースを作成します。 設定->ベースIPアドレス
  • この例では、Proxmox上のコンテナが同じアドレス範囲を使用するため、192.168.0.0 / 24サブネットを使用します。
  • 「クラスタ設定」=>「コンテナテンプレート」でテンプレートを作成します。テンプレートは、作成されたすべてのコンテナに対してデフォルトで使用されるパラメータです。






デフォルト設定でコンテナテンプレートを作成します。



Proxmoxには、コンテナを作成するためのさまざまなIPアドレスやテンプレートなどの概念がありません。 新しいコンテナを作成するたびに、新しいコンテナで使用可能なIPアドレスとリソースの両方を手動で指定する必要があります。

VMmanagerでは、これらのアクションは標準化されており、可能な限り便利になっています。 いくつかの関税パターンとIPアドレスの範囲を一度作成する必要があります。 コンテナを作成すると、IPアドレスが自動的に割り当てられ、適切なテンプレートを選択してリソースが指定されます。





Proxmoxでコンテナーを作成する





VMmanagerでコンテナーを作成する



コンテナーの移行は、両方の仮想サーバーマネージャーが管理されたコンテナーリソースに関する情報を格納するさまざまな方法を使用するため困難です。

Proxmoxは、openvz構成ファイルVMmanagerを使用します-mysqlデータベースにすべてを保存し、openvzのファイルに設定を複製します。 したがって、VMmanager-OVZからコンテナを作成し、このコンテナをProxmoxのコンテナに置き換える必要があるという事実により、スキームは複雑になります。



設定が完了すると、サーバー上にProxmoxとVMmanager-OVZの作業コピーが作成されます。

移行プロセスの残りの部分には、いくつかの簡単な手順があります。

  • Proxmoxで作成されたコンテナーの停止
  • コンテナをダンプします。
  • コンテナ構成ファイルの名前を変更する
  • コンテナファイルの削除または名前変更
  • ポータブルと同じパラメーターを使用してVMmanager-OVZにコンテナーを作成する
  • ダンプを新しいコンテナに復元する
  • 新しいコンテナを起動します




このプロセスを簡素化するには、以下に添付するサンプルスクリプトを使用します。 



#!/bin/bash # for i in `vzlist -Ha|awk '{print $1}'` do PASS=`pwgen 10 1` vzctl stop $i vzdump $i -dumpdir /vz-dump/ mv /var/lib/vz/private/$i /var/lib/vz/private/$i.proxmox mv /var/lib/vz/root/$i /var/lib/vz/root/$i.proxmox mv /etc/vz/conf/$i.conf /etc/vz/conf/$i.conf.proxmox /usr/local/mgr/sbin/mgrctl -m vemgr vm.edit id=$i name=`grep HOSTNAME /etc/vz/conf/$i.conf.proxmox|awk -F'"' '{print $2}'|awk -F. '{print $1}'` domain=`grep HOSTNAME /etc/vz/conf/$i.conf.proxmox|awk -F'"' '{print $2}'`. iptype=public ip=`grep IP_ADDRESS /etc/vz/conf/$i.conf.proxmox|awk -F'"' '{print $2}'` user=user1 mem=`grep PHYSPAGES /etc/vz/conf/$i.conf.proxmox|awk -F'"' '{print $2}' |awk -F: '{print $2}'|sed -e 's/.$//'` hdd=`grep PHYSPAGES /etc/vz/conf/$i.conf.proxmox|awk -F'"' '{print $2}' |awk -F: '{print $2}'|sed -e 's/.$//'` cpu=`grep CPUS /etc/vz/conf/$i.conf.proxmox|awk -F'"' '{print $2}'` cpufreq=`grep CPUUNITS /etc/vz/conf/$i.conf.proxmox|awk -F'"' '{print $2}'` ostemplate=debian-6.0-i386-minimal password=$PASS confirm=$PASS preset=1 sok=ok; vzctl stop $i vzrestore /vz-dump/vzdump-openvz-$i-*.tar $i -force vzctl start $i done #




スクリプトについていくつかコメントします。

  1. スクリプトは、コンテナのステータスとアクティベーション時間を考慮しません。 結局のところ、これは球形の真空で考慮された例です。
  2. テンプレートは明示的に指定されています-debian-6.0-i386-minimal。 ProxmoxおよびVMmanager-OVZコンテナテンプレートは一致しない場合があります。 ただし、コンテナを作成するには、テンプレートをスタブとして使用する必要があります。その後、古いコンテナから作成されたダンプのデータですべてが上書きされます。
  3. VMmanager-OVZでコンテナーを作成するために必要なパラメーターの最小数は、Proxmoxで作成されたコンテナーの構成ファイルから取得されます。 必要に応じて、デフォルトで作成されたパラメーター以外のパラメーターがそこで使用された場合、構成ファイルを戻すことができます。
  4. 指定されていないパラメーターはすべて、VMmanager-OVZ => Cluster Settings => Container Templatesで作成された最初のコンテナーテンプレートから取得されます。
  5. 転送されたコンテナの正しい動作を確認した後、ファイルのバックアップとダンプを削除できます。




次に、仮想サーバーを備えたProxmoxからVMmanager-KVMへの移行オプションに移りましょう。



ProxmoxからVMmanager-KVMへの仮想サーバーの移行。



残念ながら、すべてが前のケースほどスムーズに進むとは限りません。

パケット依存関係の問題のためにProxmoxが動作していない同じサーバーに、VMmanager-KVMをメインノードとしてインストールします。 したがって、2番目のサーバーを使用した移行を検討してください。



VMmanager-OVについて上記で説明したものと大差ない、VMmanager-KVMを構成します。



そして、次のアルゴリズムに従って行動します。



  • Proxmoxサーバーから仮想マシンの構成ファイルをコピーします
  • vmmgrに同一の仮想マシンを作成します
  • 止まる
  • Proxmoxサーバーで停止します
  • vmイメージファイルのコピーを作成します
  • ProxmoxサーバーからVMmanager-KVMを使用してサーバーにイメージを転送し、古いイメージを置き換えます
  • VMmanager-KVMでvmを開始します




Proxmox仮想マシンの設定は/etc/pve/qemu-server/.confに保存されます

Proxmox仮想マシンのイメージファイルは、ディレクトリ/ var / lib / vz / images /に保存されるため、ssh-keygenを使用してサーバーに接続するための認証データを入力するたびに、キーペアを作成し、公開キーをProxmoxサーバーに送信します。



cat .ssh / id_rsa.pub | ssh root @ proxmox "cat >> /root/.ssh/authorized_keys"



そして、このスクリプトを使用してすべてを実行できます。 



 #!/bin/bash # scp root@proxmox:/etc/pve/qemu-server/*.conf /root for i in `ls *.conf|awk -F. '{print $1}'` do PASS=`pwgen 10 1` /usr/local/mgr5/sbin/mgrctl -m vmmgr vm.edit id=$i mem=`grep memory /root/$i.conf|awk '{print $2}'` vcpu=`grep cores /root/$i.conf|awk '{print $2}'` cputune=1000 blkiotune=500 vsize=`grep size /root/$i.conf|awk -F= '{print $2}'| sed -e 's/.$//g'` name=`grep name /root/$i.conf|awk '{print $2}'` domain=`grep name /root/$i.conf|awk '{print $2}'`.`hostname` ip= iptype=public preset=1 password=$PASS confirm=$PASS installtype=installtemplate vmi=CentOS-6-amd64 sok=ok virsh shutdown `grep name /root/$i.conf|awk '{print $2}'` ssh root@proxmox “qm stop $i” ssh root@proxmox “cp /var/lib/vz/images/$i//vm-$i-disk-1.qcow2 /var/lib/vz/images/$i//vm-$i-disk-1.qcow2.bak” mv /vm/`grep name /root/$i.conf|awk '{print $2}'` /vm/`grep name /root/$i.conf|awk '{print $2}'`.old echo rsync -avz --partial root@proxmox:/var/lib/vz/images/100/vm-100-disk-1.qcow2 /vm/`grep name /root/$i.conf|awk '{print $2}'` virsh start `grep name /root/$i.conf|awk '{print $2}'` done #




スクリプトにいくつかコメントを残します。

  • Proxmoxを搭載したサーバーは「proxmox」としてリストされます。
  • VMmanagerを搭載したサーバーでProxmoxから仮想マシンの構成ファイルのコピーを使用して操作することにしました。
  • APIを使用して各仮想マシンを作成すると、任意のパスワードが生成されます。
  • パラメーター「vmi =」は、テンプレートの識別子を示し、設定して作業に使用できます。 インストールされているすべてのテンプレートの識別子は、コマンドラインからAPIをリクエストすることで表示できます。

    / usr / local / mgr5 / sbin / mgrctl -m vmmgr osmgr | grep 'installed = ok' | awk '{print $ 1}' | sed -e's / id = // '
  • 前のスクリプトのように、VMmanagerでのサーバーの展開時間は考慮されません。
  • 画像ファイルは前処理なしで転送されます。 ProxmoxとVMmanagerはどちらもqcow2形式を理解します。
  • VMmanagerで作成されたVMの元のイメージファイルの名前が変更され、その隣に配置されます）




コンテナと仮想マシンを転送するためのオプションを説明するとき、デフォルト値でproxmoxテストインストールを使用しました。戦闘サーバーの状況は異なる場合があります。 Proxmoxの実際の使用経験があり、それを共有する準備ができている場合は、コメントを見るのがとても良いでしょう。 ホスティングサービスの分野で成功した移行と最大限の自動化！
