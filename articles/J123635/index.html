1か月以上前、 JavaScriptに関するFAQ記事：質問するで、 「かなり高いレベルでサイトにJSコードを整理するための良いアプローチの例を教えてください。 gmailなどの実装方法について詳しく知るにはどうすればよいですか？」



この質問に答える時が来ました。 それ以来少し締めました Y. Subbotnikの同じトピックに関するレポートを伝えたかったのです。 レポートは非​​常に短く、多くの重要なポイントを捨てなければなりませんでした。 この記事は多かれ少なかれ完全版です。



この記事では、アーキテクチャ、アプローチ、ルールなど、大規模なWebアプリケーションを拡張可能かつサポートする方法について説明します。



Webアプリケーションで作業している場合、ほとんどの時間をコードの作成に費やし、バグを修正します。 新しい機能を追加する部分はごくわずかです。 高度なWebアプリケーションは常に変化しています。 今日-1行、明日-20、明後日-300。

多くの場合、jQeuryであり、イベントをハングさせ、プラグインを接続し、ajaxリクエストを実行します。 プラグインを削除すると、すべてが壊れます。 フレームワーク、プラグイン、およびコードが混在する一種のコードのもつれを取得します。 このもつれはそれほど大きくなく（100行目）、原則として1人が作成および管理します。 ほとんどのサイトは1回作成されますが、まったくサポートされていないため、サイトにとってより有害なものがあり、サイト全体のコストが増加する可能性があります。







このアーキテクチャをGMail、Yandex.Mail、Yahoo！Portal！、Twitterに適用すると、コードの巨大なボール（10000行以上）が得られ、数人が作成されます。 巨大なもつれを解くのは非常に難しく、何も壊さないようにさらに混乱させることは困難です。 Webアプリケーションは常に進化しているため、このような混乱は絶えず解明され、混乱しなければなりません。

サイトコードは構造化されておらず、そのアーキテクチャは高度に接続されています。 Webアプリケーションの場合、このようなアーキテクチャは使用できません。



建築



スケーラブルなアプリケーションの作成を容易にするいくつかのアーキテクチャの1つを検討してください。 アーキテクチャはNC Zakasから取ったもので、本当に気に入っています（彼のプレゼンテーションを見て、それが何であったかを覚えておくと素晴らしいでしょう）、コースで少し変更し、最終結果を例で取り上げます。



枠組み


どのアプリケーションにも、原則としてフレームワークが含まれています。 jQuery、Mootools、YUI、dojoなど、すべてのクライアントフレームワークは単なるツールボックスです。 ツールは、釘を打ち、ボードを切るのに役立ちます。 いくつかのツールは非常に必要ですが、ほこりを集めているものもあります。 基本的なツールがほとんどない場合は、Backbone.js + Underscore.jsなど、より重いツールが接続されています

人生では、jQueryボックスをMootoolsに置き換えるのは簡単です。 jQueryまたはお気に入りのライブラリを今すぐ放棄するコストについて考えてみてください。 簡単に交換するには、ライブラリ関数にラッパーを追加する必要があります-それはアプリケーションのコアかもしれません。

ライブラリラッパーに加えて、カーネルは他の機能を実行します。システムの一部の寿命を制御し、通信インターフェイスを提供し、アプリケーション全体のエラーを監視します。



モジュール


アプリケーションはモジュールで構成されます-これらはカーネルによって管理されるアプリケーションの独立した部分ですが、カーネル自体には直接接続されていません。 重いJavaScriptアプリケーションは、宇宙ステーションと同じくらい複雑です。 ISSには、数十個のモジュールを含む拡張可能なアーキテクチャがあり、それぞれが独自の役割を果たします。 ステーションモジュールはさまざまな国で作成され、さまざまな時期に配信され、多くのモジュールがありますが、それらはすべて全体として機能します。



HTML + CSS + JavaScript +リソースで構成されるWebアプリケーションモジュール



モジュールリソース-ローカリゼーション、記述子、およびその他のプライベートモジュールデータ。



各モジュールは、アプリケーション全体とは別個に存在する必要があります。 各モジュールのタスクは、焦点を絞った機能を実行することです。 モジュールは分離する必要があり、モジュールが他のモジュールについて知らないほど、アプリケーション全体を簡単にサポートできるようになります。 新しいコードを作成して依存関係を追加するときは、明日それを捨てる必要があると考えてください。



弱い接続性を確保し、モジュールの自由度を制限するには、特別な中程度のオブジェクトであるサンドボックスで囲む必要があります。 各モジュールはそのサンドボックス内にあり、それとのみ通信する必要があります。 これは、モジュールが知っている唯一のオブジェクトです。 サンドボックスの役割は簡単です。 彼女は警備員として行動します。モジュールが何ができるか、モジュールが誰と通信できるかを知っています。 サンドボックスは、モジュールとカーネル間の接続を提供します。



モジュールはそのメソッドとサンドボックスを呼び出し、独自のHTML要素を使用できます。 モジュールは、アクションを実行する前に許可を求める必要があります。 モジュールは、グローバルの作成、非標準グローバルの使用、他のモジュールとの直接通信を禁止されています。



サブモジュール


各モジュールは、独自のルールを持つことができるマイクロアプリケーションとして機能します;ルールはグローバルなルールと矛盾するべきではありません。 各モジュールは、そのサンドボックスをそのパーツに委任し、他のモジュールを接続できます（プロジェクトのアセンブリ中）。



カーネルは、モジュールがそのパーツをどのように管理するかを気にしません。各モジュールのグローバルルールが尊重されることが重要です。

モジュール


最初にモジュールから始めましょう。 モジュールの各部分は、動的および静的に接続する必要があります（1つのファイルにアセンブルする場合）、またはその両方。 開発者にとってすべてが可能な限り透過的である必要があります。モジュールがあります-それを使用します、いいえ-ダウンロードして使用します。



私たちのモジュールはいくつかの部分で構成されます：
  1. サンプルレイアウト：スタイルが接続されたHTMLファイル。 このファイルは単体テストで使用され、サンプルを取得してテンプレートを作成できるようにモジュールがどのように見えるかを示します。
  2. テンプレート：モジュールが使用するHTMLブロック。
  3. スタイルと画像：単純なCSSファイルまたは画像ファイル
  4. JavaScript：モジュールコード
  5. 記述子：モジュールの名前、設定、およびリッスンして生成できるイベントのリストを含むJSONファイル。 単体テストを自動生成するために使用され、権利を区別するためにサンドボックスによって使用されます。
