さまざまなプラットフォーム上の何百万ものデバイスにプッシュ通知を配信する問題とタスクの解決について

プッシュ通知は 、インターネットに接続されたモバイルアプリケーションにとって重要です。 サービスバックエンドは、頻繁かつタイムリーな更新を特定のユーザーとデバイスに送信し、ライブタイル更新、ポップアップトースト通知、バッジ更新などの形式で表示できます（詳細については、 この一連の記事を参照してください）。 このようにポーズをとると、プッシュ通知はおそらく、電話、タブレット、およびPC上のアプリケーション開発者が利用できる最も強力なクライアントエンゲージメントメカニズムです。







プッシュ通知を使用して作業を簡素化するのに役立つ多くのツールがあります（たとえば、 Windows Azureモバイルサービスには、このような機能が既に含まれています）。 これらのツールを使用して、1人のユーザーに個別の通知を送信することは比較的簡単なタスクです。 それにもかかわらず、ローカリゼーションの実際の要件、さまざまなモバイルプラットフォームのサポート、ユーザーのパーソナライズを考慮して、低遅延の通知を何百万人ものモバイルユーザーに絶えず送信することははるかに困難になります。



すべての問題の主な理由は、プラットフォーム固有のサービスを使用してデバイスにプッシュ通知が送信されることです。 たとえば、Windowsストアからアプリケーションにプッシュ通知を送信するには、Windows通知サービス（WNS）、Windows PhoneにはMicrosoftプッシュ通知サービス（MPNS）、iOSおよびGoogleクラウドメッセージング（GCM）にはAppleプッシュ通知サービス（APN）を使用する必要がありますAndroid向け。



これらのプラットフォームサービスはすべて、事前に要求され、バックエンドのどこかに保存されている各ユーザー、デバイス、アプリケーションに固有のChannelURIチャネルを使用して動作します。 次に、通知の送信時に、アプリケーションバックエンドは必要なデバイスに到達するためにChannelURIチャネルを介してデータを送信します。 これに複雑な点が1つ追加されます-ChannelURIチャネルは時間の経過とともに期限切れになる可能性があります（したがって、バックエンドで常に更新する必要があります）。 さらに、1つのデバイスに複数のアクティブなChannelURIチャネルが同時に存在する場合があります。 いくつかのプラットフォーム用のアプリケーションの開発中、各プラットフォームは独自の方法でプッシュ通知を処理するため、これらの問題はすぐに増加します。





プッシュ通知のライフサイクル



ホットニュースの通知を送信し、各ユーザーがさまざまなカテゴリにサブスクライブできるシナリオを想像してください。 このシナリオでは、おおよそデータベースに保存される次のデータテーブルを取得します。

デバイスID プラットフォーム チャネリ カテゴリー 言語 ClientVersion
1 窓 channel.uri / 1 {「世界」、「ビジネス」} 英語 V1.1
2 WindowsPhone channel.uri / 2 {「テクノロジー」、「スポーツ」} フランス語 V2.0
明らかに、ユーザーが複数のデバイスを使用できるようにする必要があります。通知とユーザー設定のローカライズを処理する必要がある場合、このデータへのアプローチはすぐに問題になります。



さらに、上記のテーブルを管理するために必要なすべてのコードと、個別のサービスを介して通知を送信するために必要なコードをすべて作成したと考えても、アプリケーションが（幸いなことに！）成功した​​場合、問題が発生する可能性があります。 通知する必要のあるデバイスが数千ある場合、通知を送信するためにループ内のデバイスを単純にソートするプロセスでは、ユーザーが希望する肯定的なエクスペリエンスを得ることができません。 これは、たとえば、通知の配信の遅延が非常に速く許容できないほど大きくなり、通知サイクルの途中で何かが発生した場合、プロセスを復元できないことを意味します。 明らかに、デバイス情報を複数のデータベースに保存（シャーディング）してこれらの問題を解決し、同時に通知を送信する多数の仮想マシンを作成し、ディスクストレージへの送信の進捗を定期的に保存しても、実装するのは依然として困難ですバックエンドの一部として。



要約すると、アプリケーションでプッシュ通知を使用する実用的なソリューションの実装が、いかにすぐに大きな困難なタスクになるかを見てきました。 主な困難は、デバイス情報の処理（特にマルチプラットフォームおよびローカライズされたアプリケーションの開発時）と、数百万のデバイスの通知送信インフラストラクチャのスケールアップにあります。 ここで、 Windows Azure Notifications Hubがプロセスを簡素化する方法を見てみましょう。



通知ハブを使用する理由



Windows Azure Notification Hubsサービスは、非常にスケーラブルなプッシュ通知インフラストラクチャを提供し、クロスプラットフォームのパーソナライズされたプッシュ通知を何百万人ものユーザーに効率的に送信するのに役立ちます。



デバイス識別子を管理します 。 通知ハブを使用すると、各プラットフォームの通知サービス（WNS、MPNS、Apple PNS、またはGoogleクラウドメッセージングサービス）で使用されるデバイス識別子またはURIチャネルを保存および管理する必要がなくなります。 私たちは、特にあなたのために、各PNS（プラットフォーム通知サービス）のサービス応答、デバイス識別子の有効期限などを安全に処理します。



タグとサブスクリプションに基づくマルチキャストとルーティングの効果的なシステム 。 デバイスは、通知ハブへの登録中に1つ以上のタグを示すことができます。 これらのタグは、一連のトピック（お気に入りのスポーツ/チーム、地理的位置、株価指数、論理ユーザーIDなど）に関する通知に対するユーザーの関心を示します。 これらのタグを再処理したり、関連性を気にする必要はありません。 各デバイスに独自の通知ルーティングインフラストラクチャを作成する必要なく、1回のAPI呼び出しで数百万のデバイスにターゲット通知を送信する非常に簡単な方法をアプリケーションに提供します。



高度なパーソナライズ 。 Notification Hubsは、顧客向けの通知のフォーム、フォーマット、およびローカライズを設定できる組み込みのテンプレートエンジンを提供し、独自のバックエンドコードをモバイルプラットフォームに依存せずにクリーンな状態に保ちます。



極端なスケーリング 。 通知ハブは、非常に低い遅延で数百万台のデバイスにプッシュ通知を送信するように最適化されています。 Notification Hubの各インスタンスは、デフォルトで最大500万台のデバイスを処理できます。 また、独自のバックエンドは通知ハブにメッセージを1つだけ送信する必要があり、アプリケーションを作り直すことなく、通知が何百万人ものユーザーに自動的に配信されます。



クロスプラットフォーム 。 Notification Hubs APIを1回呼び出すだけで、モバイルバックエンドは、Windowsストア、Windows Phone 8、iOS、またはAndroidデバイスのユーザーに同時にプッシュ通知を送信できます。



任意のバックエンドから使用します 。 Notification Hubは、既存のサーバーバックエンドに簡単に統合できます。 既存のSDKまたはオープンREST APIを使用した.NETまたはNode.js、JavaまたはPHP、PythonまたはRubyです。 このサービスは、Windows Azureモバイルサービスと統合するだけです。 また、WindowsとLinuxの両方の仮想マシン、クラウドサービス、またはWebサイトでホストされているサーバーアプリケーションからも使用できます。



Bing News：Windows Azure Notification Hubsを使用して何百万ものデバイスにホットニュースを配信する



パブリックプレビューバージョン（2013年1月に開始）の最初からWindows Azure Notification Hubsの使用を開始した最初のアプリケーションの1つは、すべてのWindows 8.x PCに含まれ、Windows Phone 8を搭載したデバイスで利用可能なBing Newsアプリケーションです。 Bing Newsには、ホットニュースを即座にユーザーに通知する機能が必要でした。 これは、いくつかの理由で難しい作業になる可能性があります。



極端なスケーリング 。 各Windows 8.xデバイスには、インストールされたBing Newsアプリケーションが含まれています。つまり、誕生の瞬間から、アプリケーションは毎月何億ものホットニュース通知をユーザーに送信する必要がありました。



優先ニュースレター 。 各ユーザーの関心を考慮して、プッシュ通知をさまざまな市場にブロードキャストするには、効率的なルーティングメカニズム、サブスクリプション、および設定ベースのメールロジックが必要です。



クロスプラットフォーム配信 。 通知の形式とセマンティクスはモバイルプラットフォームごとに異なるため、すべてのプラットフォームのチャネル/デバイス識別子の追跡は困難なプロセスになる可能性があります。



Windows Azure Notification Hubsは、Bing Newsに最適なソリューションになりました。 最近のアプリケーションの更新とともに、通知ハブを使用して、WindowsおよびWindows Phoneの何百万人ものユーザーに毎日プッシュ通知を配信し始めました。







クライアントのBing Newsアプリケーションは、WindowsおよびWindows PhoneのWindows Notification Service（WNS）およびMicrosoft Push Notification Service（MPNS）からそれぞれ対応するChannelURIを受け取ります。 次に、アプリケーションはこれらのチャネルをWindows Azure Notification Hubに登録します。 いずれかの市場でホットニュースの通知を配信する場合、アプリケーションは通知ハブを使用して関連するメッセージをすべてのデバイスに配信します。 Notification Hub APIを1回呼び出すだけで、サービスは興味（たとえば、スポーツメッセージ）によってクライアントを自動的にフィルタリングし、何百万もの顧客に遅延なくメッセージを配信できます。
