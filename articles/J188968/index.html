EazyPhoto：サーバー用の居心地の良い写真ホスティング

最近、flickrが写真を保存するためのテラバイトをまだ提供しておらず、BitTorrent Syncが安全でないアルファバージョンでインターネットにアクセスしたとき、写真を友人などと共有する必要がありました。 しかし、通常プログラマーで起こるように、「共有」という言葉はVKontakteに写真を入れるよりもはるかに費用がかかります。 すなわち：

  • 美しくシンプルなWebインターフェイスを使用して、サーバー上のどこかに写真を配置します。
  • 元の画像を簡単かつ大量にアップロードおよびダウンロードします。
  • 特定のアルバムへのアクセスを制限する機能があります。
  • 写真を複製しないように、これをローカルネットワークストレージに関連付けてみてください。
  • 可能であれば、しばらくしてからサーバーから元の写真を削除し、サイズを変更した写真のみを残します。
  • 無料で最大限に ！ :)


このような考えの後、頭の中でアイデアが浮かび始めます：「私たちは創造する必要があります...」-そしてあなたはもうやめられません。



写真ホスティング会社は現在何を提供していますか？



Googleは、写真をアップロードするためのクラウドサービスを多数見つけました。これは主に写真家を対象としています（それだけではありません）。 すべてのサービスの本質は、一定の金額から始めて月ごとに一定額を支払わなければならないという事実に帰着します。 6か月間、金額が大きく見えない場合は、2年後に不快なリマインダーになります。 nic.ruでru-domainsを年間600ルーブルで支払い、これらのドメインを他のレジストラに移管する必要があることを常に思い出して、類推できます:)。



一方、すでにサーバーがある場合は、最大限に使用する必要があります。 原則として、自分のタスクに合う2つのプロジェクトを見つけました。 これらはPiwigo （以前のphpWebGallery）とKokenです。 最後のプロジェクトは一般的に非常に満足しています。 しかし、それらをよりよく知るようになったので、これも私には合わないことに気付きました。



Dropbox、Skydrive、Google +（Picasa）、Yandex.Fotki-これらの人は、十分なスペースがある限り機能を完全に実行しました。



友達と共有する



「共有」という言葉が私にとって何を意味するのかについてもっと話したいです。 何人かの人々が参加するイベント（自然への旅行か、街を歩くか）の後、多くの写真が表示されます。 処理後、写真（300〜700 MB）がローカルNAS（Synology DS210j）に送信されます。 そして、NASのあるこのフォルダーは、写真に写っている非常に友だちがダウンロードできるように、インターネットに配置する必要があります。

このようなボリュームのブラウザを介したアップロード/ダウンロードは私には向いていません。



しかし、最終的には、アルバムへのリンクが必要です。このリンクは、他の人に捨てることができます。 まあ、アルバムのリストも手に入れるといいでしょう。 単純な通常のプロセス。



最も重要なのは、元の写真とフィルターなしの通常のサイズ変更があることです（VKなどが現在提供しているものではありません）。 それから、みんなが4Kテレビを持っているとき（FullHDは今）、写真のアーカイブを見て、また嬉しくなります。



Bittorrent同期



すべてのプラットフォーム向けのBTSyncの最初のバージョンのリリースに関するニュースを読んだ後、私はすべてに気付きました。 仕事のアルゴリズムは私の頭の中にはっきりと並んでいます：

  1. 写真の最初のアップロードは、BitTorrent Syncを介して行われます。
  2. ユーザーは、フォルダを共有し、読み取り専用の秘密キーを受け取る場所は関係ありませんが、NASまたは自分の場所でこれを行うことができます。
  3. 次に、ユーザーはWebベースの管理インターフェイスに移動し、読み取り専用の秘密キーを介してアルバムを追加します。
  4. WebインターフェイスはBTSyncコマンドを送信して、サーバーにこのフォルダーをダウンロードします。 徐々に、サーバーへのダウンロードが開始されます。
  5. 現時点では、クローンで、またはすべてがダウンロードされたかどうかのチェックがある場所（ファイルインデックス作成など）です。 たとえば、すべてのファイルがデータベースに追加され、小さなコピーが作成されます（つまり、すべてのメタ情報は元のファイルからデータベースに取得されます）。
  6. しばらくしてから、サーバーからオリジナルを削除します（ただし、ダウンロードできたユーザーによって保存されます）。




Linux上のBTSyncにはまだコンソールAPIがありません。 ただし、APIサーバーとして機能できるWebインターフェースがあります。 Goを使用して、BTSyncを操作するためのコンソールユーティリティを作成しました。 それでは、PHPのクラスを作成しました。 そして、それは小さいです：他のすべてを書きます;）

イージーフォト



新しいファイルのディレクトリをハングさせてチェックする小さなデーモン。 写真のプレビューを作成し、データベースに保存します。 BTSyncと通信します。 主なタスクは次のとおりです。

これらのタスクを実装するために、次のパッケージを使用しました。
  • code.google.com/p/gcfg-gitconfigスタイルで構成iniファイルを操作します。 出力は、塗りつぶされた構造です。
  • github.com/disintegration/imaging-画像を操作します（サイズ変更、トリミング、フィットなど）。 gdを使用せず、純粋に移動します。
  • github.com/go-sql-driver/mysql-mysqlのドライバー。
  • github.com/howeyc/fsnotifyは、ファイルシステムの変更を追跡するためのクロスプラットフォームパッケージです。
  • github.com/rwcarlsen/goexif/exif-EXIFデータを読み取るためのパッケージ。
  • github.com/sergeyfast/btsync-cliは、私のBTSync APIパッケージです。


デモプロジェクトで必要なすべての機能をチェックした後、私はこれらすべてを作業バージョンに集め始めました。 最終結果を得るのに約2日かかりました。

どのように機能しますか？
eazyphotodは、新興企業を通じて開始されます（将来、supervisordでやり直す予定です）。 開始後、すべてのアルバムをメモリにロードし、新しい写真のディレクトリのスキャンを開始します。 これと並行して、ゴルチンが起動され、ファイルシステムの変更を追跡し、HTTP要求を処理します（アルバムメタ情報を更新し、ダウンロードアルバムに新しいアルバムを追加します）。

すべての要求（サイズ変更、メタ情報の更新、新しいファイルのイベント）は1つのキューに入れられます 。 

var SyncQueue = make(chan *SyncItem) type SyncItem struct { Album *model.Album FsPhotos model.PhotoList FullSync bool Filename string }


したがって、マップおよびその他の重要な変数への同時アクセスを追跡する必要はありません。 すべてが1つの共通キューに入ります。

データベースの操作も非常に簡単です。

2. goをインストールする

godeb blog.labix.org/2013/06/15/in-flight-deb-packages-of-goをお勧めします

最小goバージョン：1.1



3. eazyphotodをインストールします 

 git clone git://github.com/sergeyfast/eazyphotod.git cd eazyphotod export GOPATH=`pwd` go get code.google.com/p/gcfg github.com/disintegration/imaging github.com/go-sql-driver/mysql github.com/howeyc/fsnotify github.com/rwcarlsen/goexif/exif github.com/sergeyfast/btsync-cli/src/btsync go build -o eazyphotod src/*.go


その後、バイナリを取得しました。

（deb-way：構成を/etc/eazyphotod.confに、バイナリーを/ usr / bin / eazyphotodに入れ、-configパラメーターで構成へのリンクを指定します）。
