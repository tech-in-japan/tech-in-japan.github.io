もう少し批判1C

少し前に、1Cを批判する別の投稿がありました。 投稿は良いのですが…なんとかして。 同僚をサポートする価値があり、さらにいくつかの重要なポイントを指摘します。









始める前に



1C開発者は素晴らしい仕事をし、長い道のりを歩んできました。 現在、さまざまなオペレーティングシステム向けのさまざまなソリューションを迅速かつ十分に実装するための幅広いツールが用意されています。 多くの既成の開発は、多くの中小企業にとって最良の選択です。 この製品はテストされ、保守され、広く文書化されています。 あなたは開発者のマニュアルを読んで驚いています-すべてがどれほど健全に考えられているか。



どんな巨大プロジェクトでもそうですが、あなたは大きなinertia性からどこにも行けず、失敗した決定や無視されたことがあります。



「1Cブレーキ」に関する私の5つのコペック



私はボード（アジャイル）の前に立ち、私の心は異なる処方の5つのタスクに追いつきますが、同じ理由です。 これらのタスクの1つは、最も注目に値するものです。

価格、残高、新製品のダウンロードの処理中-エラーが発行されます-ロックの競合...および新しいアイテムをダウンロードできません...



たまたま「クリーンな」構成に出くわすことはほとんどありませんでした。 これらは主に再設計された構成または大幅に再設計された構成です。 一方では、そのような構成を所有している組織に確実に終止符を打つことができ、単にそれらを使用しないでください。 一方、そのような企業では、原則として、より多くの「動き」があり、働くことはより興味深い場合があります。



一般に、あなたはすでにこれらの2つの単語を見てきましたね？ あなたが最初にこれに出会ったとき、誰にどんな感覚があったのでしょうか。 このタスクに必要な時間を尋ねられたとき、何に答えましたか？



悲しみはそのような現象があるということではありませんが、例えば、これに最初に出会ったとき、どこから始めればいいのかわかりませんでした-1C-keでは、この病気に対処するための定期的な手段がありません。 現象は非常に一般的ですが。 多くの人が、この問題専用のコンフィギュレータメニューにアイテムがあるべきだということに同意すると思います。







今日、ツールはありますが、「 企業ツールキット 」の一部である「パフォーマンス管理センター」の腸内のどこかにあります。これは一般に安価ではありません。 さらに、そのドキュメントはITSのサブスクリプションによってのみ利用可能です（テクノで十分です）。 もちろん、サードパーティ製のツールもあります。たとえば、 これです。



大規模な成長企業では、1Cの開発者が何百人ものユーザーが快適に作業できるように多大な努力を払っていることを知っています。 1C部門の認定された専門家が2人に1人いる場合でも、通常のロックの合計時間の指標は恐ろしい値に達します。 私自身も同様の組織で働き、他の大企業から来た同僚と話をしました。 私はこれがどこでもそうであるふりをしませんが、次の図は私には思えます：ユーザーの質量は特定のしきい値を超えます。 次に、2つのオプションがあります。



オプション1。1C開発者の一部がSQL開発者に置き換えられます。 データベースに接続されているものはすべて、構成から削除されます。 すべてのロジックを含む新しいデータベースが作成されます。 また、フォームは、ストアドプロシージャを呼び出すことによってのみ新しいデータベースと通信します。 他のデータベースとの交換は、同じService Brokerを介して確立されます。 その結果、すべてが非常に高速に機能します。 不快な些細な問題が1つ残っています-値のテーブルを1Cからストアドプロシージャに転送することはできません。 代わりに、XMLまたはJSONが渡されます。



続きがあります。 300ジョブのライセンスの別のパッケージを購入するときが来たら、1C開発者は安くはないという理解があります。 その結果、1Cの「顔」をPHPまたはC＃に置き換えるという考え方が生まれました。



オプション2。会社はSAPを実装しようとしています。



そして、なぜですか？ 私はボードの前に立ってカモメを飲んでいるので、時々いくつかのロックタスクを解決するのがいいと思います。 しかし、どれくらい時間がかかるかはわかりませんが、他にも多くのタスクがあります。



通信と構成



モジュール性はモノリスよりも優れています。 だから？ 開発中に全体を部分に分割することが判明した場合、それは良いことです。 20人以上のユーザーが1Cで作業する場合、全員をさまざまな角度で分割するとよいでしょう。 各方向に独自の構成があることを確認してください。 たとえば、独自の倉庫、独自のコンテンツ、独自の宅配便業者、会計ももちろんあります。 したがって、ロックが少なくなります。 多くの小さな構成は、1つよりも優れています。 開発などが簡単です。



将来的にはそうなると思います。店に行って、さまざまな会社の発展を見て、ビジネスに必要な小さな構成を自分で入力し、システムに接続して使用します。

だから私は何が必要ですか？ 顧客の注文、オンラインのキャッシュデスク、Tinkoffクライアントバンク、Sber銀行クライアント、要返却梱包材...いいえ、要返却梱包材は必要ありません...



プラットフォームの1つのバージョンでスピンしている構成であっても、構成をドッキングするには本当に時間がかかります。 私は、交換がすでに書かれている典型的な構成のドッキングについて話しているのではありません。 時々驚くべきことがありますが。 たとえば、会計士が四半期ごとにドキュメントを自分のデータベースにアップロードする場合（これは人にとって便利です）、アンロードおよびロードの時間は時間単位で測定されます。 あるタイプから別のタイプへ。 私はかつて10時間以上続いた交換を見ました。 何らかのエラーが発生したかどうか想像できますか？ あなたはそれを排除し、再びダウンロードを設定します... bam！ また。

それは起こります。 たとえば、私は純粋に標準のUT11.3からデフォルトの設定（別のプログラム...）でユニバーサルフォーマットを介して記事をアンロードするために置きました。 3時間後：

データの送信中にエラーが発生しました（ログブックを参照）。



交換マネージャーへのパスが指定されていません（空の場合、デフォルトのマネージャーが使用されます）。 雑誌を見る：

イベント：ディスパッチ。

オブジェクト：キャッシュフロー項目、為替差損（e1cib /データ/ハンドブック。記事キャッシュフロー？Ref = ae8500259075533411e7205499471b7a）。



{汎用モジュール。データ交換XDTOサーバー。モジュール（4605）}：

メタデータオブジェクトのAMLが見つかりません。

メタデータオブジェクト：。

例外テキストエラーを呼び出します。



これは何ですか なんで？ 何のために？ AMLがデータ交換のルールであるとしましょう...



オンラインストアとの交換については何も言いたくありません。 いくつかのヒントを与えてください、命名法やプロパティのdofigを持っている人のために、突然誰かが知りません：彼が何時間も「ハンマー」しないように、最初にすることは画像転送をオフにすることです。 写真は、サイトリソース（たとえば、FTP画像サーバー）に個別にアップロードする必要があります。 これを行うために、規制が行われ、その中で、写真の変化を監視し（たとえば、データのバージョンに従って）、それを送信します。 さまざまな企業でさまざまな実装を見てきましたが、本質はこれです。



したがって、アプリケーションソリューションのマッチング。 特別な構成「 データ変換3.0 」があります。 ただし、すべてが機能するためには、「Exchange Manager Through Universal Format」というモジュールのテキストを含むファイルなどの些細なことを指定する必要があります（UT11.3の場合、モジュールには46+千行のコードが含まれます）。 このモジュールのジェネレーターはどこに隠されていますか？ また、新しい空の構成がある場合はどうなりますか？



あなたは言う-誰が邪魔になるのですか？ そして、素晴らしいと思います。両方の構成で同じオブジェクトを作成し、構成ツリーでそれぞれを右マウスボタンで直接突いて、データを転送する構成を選択しました。 オブジェクトモジュールでは、このトピックに関するいくつかのイベント（データを送信する前、データを受信する前）。 そして、すぐに送信されます。 確かに、それはすべてサイエンスフィクションの瀬戸際にあります。 それは火星への男の飛行のようなものです。



それから。 関数「Exchange Plans。Select Changes」が選択されたパックの最大サイズを受け入れた場合、それは素晴らしいことです。 すぐにそこに蓄積されたすべてではなく、たとえば100個のパック。 次の反復で-さらに100個。 などなど。 多くの変更がある場合、より速く動作します。 そして、それは長い間ブロックしません。



1Cとの互換性







自転車を発明するのは悪いことです。 特に私がそれをするつもりがなかった場合。 独自の製品の開発に投資する代わりに、ターンキーソリューションを購入する方が収益性が高くなります。 これは、販売の分析、ロジスティクスのサブシステム、ルート、住所、宅配便アプリケーションなどがあります。 「1Cと互換性がある」バッジを取得するためのルールは次のとおりです。 多くの場合、ターンキーソリューションは1C以外のプラットフォームに基づいて提供されます。 ルールはそのような決定についてはほとんど述べていません：

ソフトウェア製品の以下のグループは、認定のために受け入れられます。

...

8. 1Cとのデータ交換をサポートするプログラム：ファイルレベルのエンタープライズ。

9. 1Cと対話する他の方法を持つプログラム：エンタープライズ（個別に検討）。



一般的な要件



1.ソフトウェア製品は、循環配布用に設計する必要があり、特定の実装向けではありません。

2.製品には印刷されたマニュアルが必要です。

3.マニュアルには、製品と1C：Enterpriseとの相互作用を明示的に記述する必要があります。

...

5.すべてのエンドユーザー向け製品にはインストールツールが必要です。

6.認証製品を修正または変更する場合、開発者は、変更された製品が認証要件を満たしていることを確認します。 認証要件に違反する変更が発生した場合、会社「1C」は証明書を一時停止する権利を有します。



統合ファイル共有製品の要件。



ファイル交換（アップロード/ダウンロード）に基づいて「1C：Enterprise」システムと統合されたソフトウェア製品は、TXTおよびDBFファイルで動作するように設計された標準1C：Enterprise側のEnterpriseツールを使用する必要があります。



それだけです。 ソリューション開発者はアイコンを取得します。 顧客はサードパーティ製品の資料を読み、1Cとの統合があることを確認して購入し、「実装」が開始されます。 これに追加料金を支払うことは珍しくありません。 一部の製品には、外部処理や構成拡張など、非常に簡単な統合方法が含まれています。



EnterpriseDataはありますか？ これは、非1Cソリューションとのインターフェースに使用する必要があるツールにすぎません。 必要なものはすべて、すでに考慮されています。 考慮されていない場合、新しいバージョンの形式に追加されるだけです。 APIとのインターフェイスに処理は必要ありません。さらに、設定が改善されます。



サードパーティのソリューションと構成で単一のエンティティを使用する場合、オブジェクトのロックに関連する問題が常に発生します。 私はシステムに出会いました。そのうちの1つでも、どの変更を保存し、どの変更を保存しないかを決定する特別な権限を持つユーザーに提案されました。 1C-differentとの相互作用については言及していません。



また、別のソリューションでは、編集されたデータを転送しようとしたときに、気にせずにエラーをスローすることに決めました。



オブジェクトロックは、ルールで言及するために、個別に詳細に直接必要です。 すべてのメガソリューション開発者がそれが何であるかを認識しているわけではないようです 。



しかし、最も不快なのは、「1Cと互換性がある」というラベルがない場合です。 そして、それは次のようなものを言います：

1Cには典型的な統合があります：

これが会話全体です。 これらの人に必要な厳しい。 ナットを締めます。 弁護士から学ぶのは興味深いです。彼らのウェブサイトの誰でも、彼が「1Cとの統合」を持っていると書くことができますか？



管理フォーム



管理されたフォームを作成するためのデザイナーはなんてクールでしょう！ 欠点もあります（たとえば、標準の検索文字列をすべてのプレートに使用できない、または何も表示できない）が、一般的にはフォームを作成するのが非常に便利です。 創造性のramp延は可能な限り制限されています。 開発者は、インターフェイスの設計エラーから保護されています（フォームのオーバーロードのみを行うことができます）。 異なる開発者が同じものを取得します。 金型は美しいです。 このために私はこれを思いついたすべての人に感謝したいと思います。 みんな-かっこいいね。



ブラウザですべてを公開して開く機能は、大きなチャンスをもたらし、一般的には素晴らしいです。 いくつかの欠陥がありますが、やがてそれらはなくなります。







コードについては、マネージドフォームのアーキテクチャが開発の速度を落とすようになりました。 開発者やフォームの目的に応じていくつかの統計情報がありますが、単純なものを考慮しない場合、フォームに詳細を入れるだけで、スローダウンは約2〜4倍になります。 これは、フォーム自体と非同期にのみ適用されます。 しかし、どうやらそれについては何もできない。



顧客は、モバイルデバイスなどで、見た目が不満な場合があります。 次に、HTTPサービスを介して個別に作成されたページを提供する必要があります。
