Appleプッシュ通知サービスガイド

この記事は、raywenderlich.comのApple Push Notification Serviceガイドと私の追加の無料翻訳です。



iOSアプリを長時間バックグラウンドに置くことはできません。 バッテリ電力を節約するために、バックグラウンドで実行されているアプリケーションは、限られたアクションセットを実行できます。



しかし、アプリケーションが実行されていない場合でも、何か面白いことが起こり、ユーザーに通知したい場合はどうでしょうか？



たとえば、ユーザーがTwitterで応答を受け取った、お気に入りのチームがゲームに勝った、またはランチの準備ができているなどです。 アプリケーションが実行されていないため、このデータを確認および受信できません。



幸いなことに、Appleはこの問題の解決策を提供しました。 イベントを継続的にチェックしたり、バックグラウンドでアクションを実行したりする代わりに、これらのアクションを実行するアプリケーションのサーバー側を作成できます。



関心のあるイベントが発生すると、サーバー側はアプリケーションにプッシュ通知を送信できます！ プッシュ通知では、次の3つのことができます。



  1. 短いテキストメッセージを表示します。
  2. 短いビープ音を鳴らします。
  3. アプリケーションアイコンのバッジに番号を設定します。


必要に応じてこれらのアクションを組み合わせることができます。 たとえば、サウンドを再生してバッジに番号を設定しますが、メッセージは表示しません。



短いレビュー



プッシュ通知メカニズムの操作スキーム：







アプリケーションをインストールすると、プッシュ通知の受け入れを確認するポップアップメッセージが表示されます。



  1. iOSは、Apple Push Notification Service（APNS）サーバーにデバイストークンを要求します。
  2. アプリケーションはデバイストークンを受け取ります。 トークンはプッシュ通知を送信するためのアドレスであると想定できます。
  3. アプリケーションはデバイストークンをサーバーに送信します。
  4. アプリケーションでイベントが発生すると、サーバーはプッシュ通知をAPNSに送信します。
  5. APNSはプッシュ通知をユーザーのデバイスに送信します。


ユーザーがプッシュ通知を受信すると、メッセージが表示され、音声信号が再生され、アプリケーションアイコンのバッジが更新されます。 ユーザーは通知からアプリケーションを開くことができます。 アプリがプッシュ通知コンテンツを受信して​​処理できるようにします。



iOS 4.0でローカル通知とマルチタスクが表示された場合、プッシュ通知を引き続き使用する必要がありますか？ できたらいいのに！



ローカル通知は時間制限のあるイベントです。 VOIPアプリケーション、ナビゲーション、およびバックグラウンドサウンドの再生のみが、無制限のバックグラウンド実行が可能です。 外部イベントを（閉じている間に）アプリケーションのユーザーに通知する必要がある場合、プッシュ通知を使用する必要があります。



このガイドでは、プッシュ通知システムの操作と、それをアプリケーションに統合する方法について詳しく説明します。



プッシュ通知に必要なもの



プッシュ通知をアプリケーションに統合するには、以下を行う必要があります。



iPhone、iPad、またはiPod touch。 プッシュ通知はシミュレーターでは機能しないため、テストにはデバイスが必要です。



iOS開発者プログラムに登録します。 プッシュ通知メカニズムが統合される各アプリケーションについて、新しいApp IDとプロビジョニングプロファイル、およびサーバーのSSL証明書を作成する必要があります。 これらのアクションは、iOSプロビジョニングポータルで実行されます。



このガイドの例を完全に実行する場合は、プロビジョニングプロファイルとSSL証明書を作成する必要があります。 これを行う方法を詳細に説明します。



インターネットに接続されたサーバー。 プッシュ通知は常にサーバーによって送信されます。 開発プロセス中に、サーバーとして独自のMacを使用できますが、リリースにはVPS（Virtual Private Server）などが必要です。



プッシュ通知を使用するには、安価な共有ホスティングでは不十分です。 サーバーでバックグラウンド実行を実行し、SSL証明書をインストールし、特定のポートで発信TLS接続を構成する必要があります。 ほとんどの共有ホスティングプロバイダーはこれを許可しません。 テクニカルサポートに連絡すると、すべての問題を解決できる可能性が高くなります。 それでも、VPSを使用することを強くお勧めします。



プッシュ通知の構造



サーバーは、プッシュ通知用のメッセージを作成する責任があります。 したがって、これらのメッセージがどのように見えるかを知ることは有用です。



プッシュ通知は、デバイストークン、ペイロード、およびその他の情報で構成される短いメッセージです。 ペイロードは、デバイスに送信される実際のデータです。



サーバーはペイロードをJSON辞書に変換する必要があります。 単純なプッシュメッセージのペイロードは次のとおりです。 



{ "aps": { "alert": "Hello, world!", "sound": "default" } }


中括弧で囲まれたブロックには、キーと値のペアで構成される辞書が含まれます（NSDictionaryと同じ）。



ペイロードは、少なくとも1つのキーと値のペア「aps」で構成される辞書であり、その意味自体が辞書です。 上記の例では、「aps」には「alert」と「sound」の2つのフィールドが含まれています。 デバイスにプッシュ通知が届くと、「Hello、world！」というテキストを含むポップアップメッセージが表示され、標準の音声信号が再生されます。



さらに、「aps」では、他のフィールドを追加して通知を構成できます。 例： 



 { "aps": { "alert": { "action-loc-key": "Open", "body": "Hello, world!" }, "badge": 2 } }


これで、アラートフィールドの値は辞書になりました。 「Action-loc-key」には、「Run」ボタンの代替テキストが含まれています。 バッジフィールドには、アプリケーションアイコンのバッジに表示される番号が含まれています。 プッシュ通知には音声信号は含まれません。



JSONペイロードを生成する方法はかなりあります。 通知音を変更し、独自のフィールドを追加できます。 詳細については、Apple Developersサイトの「ローカルおよびプッシュ通知プログラミングガイド」ページを参照してください。



プッシュ通知はごくわずかです。 ペイロードサイズは256バイトを超えることはできません。 これは、SMSまたはツイートに対応できるのとほぼ同じです。 プッシュサーバーは、改行やスペースに無駄になりませんが、次のようなものを生成します。 



 {"aps":{"alert":"Hello, world!","sound":"default"}}


このようなプレゼンテーションは読みにくくなりますが、より価値のある情報のために十分なスペースを提供します。 APNSは、256バイトを超えるプッシュ通知を拒否します。



プッシュ通知について



彼らは信頼できません！ APNSがそれらを受け入れたとしても、プッシュ通知が配信される保証はありません。



サーバーがプッシュ通知を生成すると、応答なしでそれをAPNSに送信します。 送信後にエンドユーザーへの通知の配信ステータスを確認する方法はありません。 配達時間は数秒から30分までさまざまです。



さらに、iデバイスのユーザーは、常にプッシュ通知を受信できない場合があります。 たとえば、近くにインターネットにアクセスできるWi-Fiネットワークがないか、デバイスの電源が完全にオフになっている可能性があります。



APNSは、デバイスが受信可能になったときに送信された最後の通知を配信しようとします。 しかし、これらの試みには時間が限られています。 タイムアウト後、プッシュ通知は永久に失われます！



彼らは高価になる可能性があります！ データを所有している場合、アプリケーションにプッシュ機能を追加するのは非常に簡単で安価です。 ただし、多数のユーザーがいる場合、またはデータを要求する必要がある場合、コストは急激に増加します。



たとえば、RSSフィードの変更をユーザーに簡単に通知できます。これは、フィードを制御し、変更がいつ行われるか（サイトのコンテンツが更新されるとき）を知っているため、サーバーが即座に通知を送信するためです。



しかし、アプリケーションがユーザーがフィードのURLを入力できるRSSリーダーである場合はどうでしょうか？ この場合、追加されたテープを更新するための追跡メカニズムを考え出す必要があります。



実際には、これは、サーバーがフィードの変更を常にチェックする必要があることを意味します。 多くのユーザーがいる場合、すべてのプロセスを処理し、安定した帯域幅を維持するために追加のサーバーをインストールする必要がある場合があります。 RSSリーダーなどのアプリケーションの場合、プッシュ機能は非常にコストがかかるため、価値がありません。



わかりました、十分な理論。 これらすべてのプッシュを実装するプロセスを学ぶ時が来ました。 しかし、最も「おいしい」ものに着手する前に-プログラミング！ -iOS Provisioning Portalで退屈な設定を行う必要があります。 さて、可能な限り高速にしましょう。



プロビジョニングプロファイルと証明書



プッシュ通知をアプリケーションに接続するには、適切に構成されたプロビジョニングプロファイルで署名する必要があります。 さらに、サーバーはSSL証明書を使用してAPNSに接続する必要があります。



プロビジョニングプロファイルとSSL証明書は互いに密接に関連しており、1つのアプリIDに対してのみ有効です。 これは、サーバーのみがアプリケーションのユーザーにプッシュ通知を送信できるというセキュリティ保証です。



ご存知のように、アプリケーションは開発とリリースに異なるプロビジョニングプロファイルを使用します。 サーバーのプッシュ証明書には2つのタイプがあります。



  1. 開発。 アプリケーションがデバッグモードで、開発用のプロビジョニングプロファイルによって署名されている場合（コード署名ID-「iPhone開発者」）。
  2. リリース（生産）。 アドホックとして作成されたアプリケーション、またはApp Storeでのダウンロードの準備（コード署名識別-「iPhone配布」）は、リリース用の証明書（製品証明書）を使用していることをサーバーに通知する必要があります。 それらの間に矛盾がある場合、プッシュ通知はアプリケーションに届きません。


このガイドでは、開発証明書のみを使用します。



証明書署名要求（CSR）の生成



iOS開発者プログラムに参加した後、どのようにiOSプロビジョニングポータルにアクセスし、開発証明書を作成したか覚えていますか？ 次の手順も同様です。 しかし、それでも、以下で説明するとおりに正確に実行することをお勧めします。 開発者は、証明書に関連するプッシュ通知に関する問題のほとんどを抱えています。



デジタル証明書は、公開鍵と秘密鍵を使用した暗号化に基づいています。 証明書を使用する場合、暗号化について何も知る必要はありませんが、証明書は常に秘密鍵とペアになっていることに注意する必要があります。



証明書は、このキーペアの一般的な部分です。 秘密鍵は秘密にしておく必要があります。 あなただけがそれを所有し、他の誰もそれにアクセスするべきではありません。 秘密鍵なしで証明書を使用することは不可能です。



デジタル証明書を要求するたびに、その署名を要求する必要があります（証明書署名要求[CSR]）。 CSRを作成すると、新しいキーがキーチェーンに入力されます。 次に、CSRを証明機関（この場合はiOS開発者ポータル）に送信する必要があります。この機関は、CSRからの情報に基づいてSSL証明書を生成します。



「キーチェーン」ユーティリティ（「アプリケーション→ユーティリティ（その他）」）を開き、「認証局から証明書を要求する...」オプションを選択します。







このメニューオプションが表示されない場合、または「キーを使用して認証局に証明書を要求する」というテキストを含むメッセージが表示される場合は、 WWDR中間証明書をダウンロードしてインストールする必要があります。 また、秘密鍵が割り当てられていないことを確認する必要があります。



これで、認証アシスタントウィンドウが表示されます。







ここにメールを入力してください。 開発者は、iOS開発者プログラムに電子メールとして登録するために使用したのと同じ電子メールアドレスを使用することをお勧めしますが、これは必須ではありません。



共通名には、「PushChat」と入力します。 何でも入力できますが、わかりやすいものを選択してください。 後でこれにより、秘密鍵を簡単に見つけることができます。



[ディスクに保存]ラジオボタンを選択し、[続行]をクリックします。 ファイルを「PushChat.certSigningRequest」として保存します。



アプリIDとSSL証明書を作成する



iOSプロビジョニングポータルにログインします。



まず、新しいアプリIDを作成します。 プッシュ通知メカニズムを使用する各アプリケーションには、独自の一意のIDが必要です。



サイドバーの「アプリID」項目をクリックして、「新しいアプリID」ボタンをクリックします。







次のようにフィールドに入力しました。



説明：PushChat

バンドルシードID：デフォルト

バンドル識別子：me.evgeniy.PushChat



私の代わりに独自のバンドル識別子-com.yoursite.PushChat-を指定した方が良いでしょう。 Xcodeプロジェクトでは、同じバンドルIDを設定する必要があります。



さらにいくつかのポイント：プッシュサーバーがAPNSに安全に接続するために使用するSSL証明書を生成します。 この証明書はアプリIDに関連付けられています。 サーバーは、アプリケーションにのみプッシュ通知を送信でき、他のアプリケーションには送信できません。



アプリIDが作成されると、リストに表示されます：







「プッシュ通知」の反対側の「開発」列と「生産」列には、「構成可能」という言葉が付いた2つのオレンジ色の円があります。 つまり、アプリIDはプッシュ通知を使用できますが、設定する必要があります。 したがって、「構成」リンクをクリックしてください。







表示されるページで、「Appleプッシュ通知サービスを有効にする」の横のボックスをオンにします。 次に、開発プッシュSSL証明書の行にある[構成]ボタンをクリックします。 「Appleプッシュ通知サービスSSL証明書アシスタント」ウィンドウが開きます。







最初に行うことは、証明書署名要求を生成することです。 これはすでに完了しているため、[続行]をクリックします。







次のステップは、CSRをAppleサーバーにアップロードすることです。 前に生成したCSRファイルを選択し、「生成」をクリックします。







SSL証明書の生成には数秒かかります。 準備ができたら、[続行]をクリックします。







証明書をダウンロードするには、「ダウンロード」をクリックします。「aps_development.cer」という名前で保存されます。 「完了」をクリックした後。







これで有効な証明書が作成され、プッシュ通知メカニズムが開発に利用可能になりました。 必要に応じて、証明書を再度ダウンロードできます。



アプリケーションのリリース準備ができたら、プロセス全体を繰り返して実稼働証明書を生成する必要があります。 すべての手順は似ています。



発言。 実動証明書は1年間有効ですが、締め切り前に再作成できます。



キーチェーンに証明書を追加する必要はありません。 これを行う場合は、以前にダウンロードしたaps_development.cerファイルをダブルクリックします（その後、証明書は秘密鍵に関連付けられます）。



プロビジョニングプロファイルの作成



プロビジョニングポータルをご覧ください。 [プロビジョニング]リンクをクリックし、[新しいプロファイル]ボタンをクリックします。







次のようにフィールドに入力しました。



  • プロファイル名：PushChat Development
  • 証明書：証明書を選択します
  • アプリID：PushChat
  • デバイス：デバイスを選択します


このプロセスは、他のプロビジョニングプロファイルの生成と同じです。 プッシュ通知メカニズムをサポートする各アプリケーションは、特定のアプリIDに関連付けられた独自のプロファイルを必要とするため、新しいプロファイルを作成します。



[送信]をクリックすると、プロファイルが生成されます。 新しいプロファイルは「保留中」に設定されます。 ページをリロードして、ステータスが「アクティブ」に変更されたことを確認します。 これで、プロビジョニングプロファイル（「PushChat.mobileprovision」というファイル）をダウンロードできます。



ファイルをIDEアイコンにドラッグするか、ファイルをダブルクリックして、プロビジョニングプロファイルをXcodeに追加します。



アプリケーションをリリースする準備ができたら、上記のプロセスを繰り返して、アドホックまたはApp Storeの配布プロファイルを作成する必要があります。



シンプルなアプリ



前のアクションは本当にエキサイティングではありませんでしたが、実行する必要があります。 開発者はそのようなことを毎日行わず、証明書がないとプッシュ通知が機能しないため、証明書を生成する方法を詳細に示したいと思いました。



次に、プッシュ通知を受信する単純なアプリケーションを作成します。



Xcodeを開き、新しいプロジェクトを作成します。 アシスタントで「シングルビューアプリケーション」を選択し、次の手順に進みます。







次のようにフィールドに入力しました。



  • 製品名：PushChat
  • 会社識別子：me.evgeniy
  • クラスのプレフィックス：PC
  • デバイスファミリ：iPhone
  • チェックボックス：ストーリーボードを使用し、自動参照カウントを使用する


製品名と会社識別子が一緒になってバンドルIDを形成します。 私の場合、「me.evgeniy.PushChat」です。 プロビジョニングポータル（com.yourname.PushChat）で以前に作成されたアプリIDに対応する製品名と会社識別子を入力する必要があります。



プロジェクトを作成したら、PCAppDelegate.mを開きます。 didFinishLaunchingWithOptionsメソッドを次のように変更します。 



 - (BOOL)application:(UIApplication *)application didFinishLaunchingWithOptions:(NSDictionary *)launchOptions { //  iOS 7   [[UIApplication sharedApplication] registerForRemoteNotificationTypes: (UIRemoteNotificationTypeBadge | UIRemoteNotificationTypeSound | UIRemoteNotificationTypeAlert)]; //  iOS 8   // [[UIApplication sharedApplication] registerUserNotificationSettings:[UIUserNotificationSettings settingsForTypes:(UIUserNotificationTypeSound | UIUserNotificationTypeAlert | UIUserNotificationTypeBadge) categories:nil]]; // [[UIApplication sharedApplication] registerForRemoteNotifications]; return YES; }


registerForRemoteNotificationTypes呼び出しは、このアプリケーションがプッシュ通知を受信することをiOSに通知します。



アプリケーションをビルドして実行します。 これを行うには、シミュレータがプッシュ通知をサポートしていないため、デバイスを使用する必要があります。 Xcodeは新しいプロビジョニングプロファイルを自動的に選択します。 アプリケーションの起動中にエラーが発生した場合は、コード署名IDで正しいプロファイルが選択されていることを確認してください。



アプリケーションが起動すると、プッシュ通知の受け入れを確認するメッセージが表示されます。







アプリケーションは許可を一度だけ要求します。 ユーザーが「OK」をクリックするとプッシュ通知が届き、「拒否」するとプッシュ通知が届きません。 設定で決定を変更できます。







アプリケーションの名前がプッシュ通知設定に追加されます。 ここで、ユーザーはアプリケーションのプッシュ通知を有効または無効にしたり、バッジ、サウンド、メッセージを個別に設定したりできます。







アプリケーションは、有効になっているプッシュ通知のタイプを判別できます。 



 UIRemoteNotificationType enabledTypes = [[UIApplication sharedApplication] enabledRemoteNotificationTypes];


プッシュ通知を受信できるようにするには、アプリケーションにもう1つ追加する必要があるものがあります。 次のコードをPCAppDelegate.mに追加します。 



 - (void)application:(UIApplication *)application didRegisterForRemoteNotificationsWithDeviceToken:(NSData *)deviceToken { NSLog(@"My token is: %@", deviceToken); } - (void)application:(UIApplication *)application didFailToRegisterForRemoteNotificationsWithError:(NSError *)error { NSLog(@"Failed to get token, error: %@", error); }


アプリケーションがプッシュ通知を受信するように登録すると、デバイストークンの取得が試行されます。 これは、デバイスを一意に識別する32バイトの一意の番号です。 デバイストークンは、プッシュ通知の送信先アドレスと比較できます。



アプリケーションを起動すると、デバイスのトークンがXcodeコンソールに表示されます。



My token is: <740f4707 bebcf74f 9b7c25d4 8e335894 5f6aa01d a5ddb387 462c7eaf 61bb78ad>







トークンは、NSData型のオブジェクトである不透明なバイナリデータ構造です。 この目的のためには、32バイトのデバイストークンを知っているだけで十分です。 トークンは、64桁の16進文字として表すことができます。 このような形式を使用します。



シミュレーターでアプリケーションを実行すると、didFailToRegisterForRemoteNotificationsWithError：メソッドが呼び出されます。このメソッドは、シミュレーターがプッシュ通知をサポートしていないという情報を含むエラーを表示します。

プッシュ通知を送信する



前に説明したように、プッシュ通知を送信するようにサーバーを構成する必要があります。 ただし、テストには、Mac OS PushMeBaby用のアプリケーションを使用します。これは、 githubサービスからダウンロードすることもできます。



その後、すべてが簡単です-XcodeでPushMeBabyを開き、以前に作成したSSL証明書（aps_development.cer）をプロジェクトに追加し、ApplicationDelegate.mファイルの編集に進みます。 initメソッドで、次の変更を行います。



デバイストークンを追加します。これは、以前に作成したアプリケーションを起動した後にXcodeコンソールに表示されます。 



 self.deviceToken = @"564f77d3 c1d06866 77a907d5 31d0450a 9cb552ec 5e2373e5 f2cc2c20 ff013232";


前述のペイロードを追加します（引用符は必ずエスケープしてください）。 



 self.payload = "{\"aps\":{\"alert\":\"Hello, world!\",\"sound\":\"default\"}}";


そして、プロジェクトに追加されたSSL証明書の名前を設定します。 



 self.certificate = [[NSBundle mainBundle] pathForResource:@"aps_development" ofType:@"cer"];








ここでアプリケーションを起動し、プッシュボタンをクリックします。 数秒以内にプッシュ通知を受け取るはずです。
