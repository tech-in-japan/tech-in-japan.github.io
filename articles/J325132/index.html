LinuxでRDPプロトコルを使用する1Cのターミナルサーバー：実際の操作経験に基づいたチューニングの推奨事項

この記事では、Xubuntu 14.04ディストリビューションキットに基づいて8.3プラットフォームで1cデータベースを操作するために、RDPプロトコルを使用してターミナルサーバーをインストールおよび構成する際のニュアンスについて、最新のxrdpサーバーの機能と実際の操作の経験を考慮して説明します。



少し前（2016年12月末）に、xrdpサーバーバージョン0.9.1の次のリリースがリリースされました。 同時に、xrdpバックエンドの1つであるxorgxrdp v 0.2.0の安定バージョンがリリースされました。 これらのバージョンは、インストール時にさらに使用します。



小さいが重要な余談



Linuxディストリビューションには、主に古いバージョンのxrdpサーバーバージョン0.6.xが含まれています。これには、1で正常に動作しないバグがあります（たとえば、ロシア語文字のクリップボードは正常に動作しません）。



基本的に、Linuxでのrdpプロトコルに基づくターミナルサーバーのインストールを検討する場合、 X11RDd-o-Matic （ubuntu用）-ハブに関する記事 、infostart に関する記事などの特別なスクリプトを使用して、X11Rdpバックエンドのアセンブリでxrdpサーバーをインストールするオプションを検討します。



X11Rdpバックエンドを使用すると、いくつかの欠点があります。



  • 長いビルド時間
  • 言語を切り替えることで問題を解決することは困難です。 セッションに再接続した後、言語の切り替えが機能しなくなる場合があります。 著者は、開発者のウェブサイトで問題を登録しました-保証されたソリューションは、ソースコードを編集することにより、xorgxrdpバックエンドに対してのみ、著者によって発見されました（この方法は記事の後半で使用されます）
  • 快適なユーザーエクスペリエンスを提供するには、レンダリング速度が十分に速くない場合があります。


xorgxrdpバックエンドを使用すると、これらの問題を回避できます。



この記事では、リモート画面を描画する際のパフォーマンスの低下により、最新バージョンの16.04 LTSではなく、Xubuntuバージョン14.04 LTSへのインストールについて説明します。 この状況には登録済みの問題があります。 設定でコンポジターを無効にすることは助けになりましたが、記事の著者はXubuntuバージョン14.04と同じレンダリング速度を達成できませんでした。



オペレーティングシステムのインストールに関する推奨事項



インストールには、Xubuntu OS 14.04の32ビットバージョンを使用します。 画像へのリンク 。



システムは、ホスト、仮想環境、および仮想環境の両方にインストールできます。 私は通常、KVM仮想化を好みます;実動サーバーがそれにインストールされました。



ディスクをパーティション分割する際、ext4ファイルシステムである従来のスキーム（LVMではない）が使用されました。 仮想マシンに接続されている3つの固定サイズドライブがあります。



  • 22 GB：18 GBのマウントポイント/、残りはスワップ下
  • 5 GB：マウントポイント/ var
  • 5 GB：マウントポイント/ tmp


このパーティションは、使いやすさを考慮して、著者の好みです。 いずれにせよ、ディスク容量を計画するときは、ファイルシステムの負荷を考慮する必要があります。 ファイルデータベースを使用する場合は、SSDドライブを使用することをお勧めします。



サーバーの静的IPアドレスを設定する必要があります。



後続のインストール用のコマンドを入力するために、 OpenSSHサーバーをインストールする必要があります。 



sudo apt-get install openssh-server sudo restart ssh


ファイアウォールをアクティブにして、ポート22を開きます。 



 firewall-cmd --add-port 22/tcp --permanent firewall-cmd --reload


xorgxrdpバックエンドを使用してxrdpサーバーをインストールおよび構成する



開発者のWebサイトには、debian 8にインストールするための指示があり、これを基礎として使用しています。



必要な依存関係をインストールします。 



 sudo apt-get install git autoconf libtool pkg-config gcc g++ make libssl-dev libpam0g-dev libjpeg-dev libx11-dev libxfixes-dev libxrandr-dev flex bison libxml2-dev intltool xsltproc xutils-dev python-libxml2 g++ xutils libfuse-dev libmp3lame-dev nasm libpixman-1-dev xserver-xorg-dev git


ディストリビューションをダウンロードするためのディレクトリを作成します。 



 mkdir -p ~/xrdp-xorg cd ~/xrdp-xorg BD=`pwd` cd "${BD}"


現在の最新のxrdpアセンブリをリポジトリから複製し、rfxcodecコーデックでアセンブルします。 



 git clone git://github.com/neutrinolabs/xrdp cd xrdp git submodule init git submodule update cd librfxcodec ./bootstrap ./configure make cd .. ./bootstrap ./configure --enable-fuse --enable-rfxcodec --disable-ipv6 make


システムにxrdpをインストールします。 



 sudo make install sudo ln -s /usr/local/sbin/xrdp{,-sesman} /usr/sbin sudo update-rc.d xrdp defaults


xorg-xrdpバックエンドディストリビューションをダウンロードします。 



 // Build and install xorgxrdp wget https://github.com/neutrinolabs/xorgxrdp/releases/download/v0.2.0/xorgxrdp-0.2.0.tar.gz cd "${BD}" tar xvfz xorgxrdp-0.2.0.tar.gz cd "${BD}"/xorgxrdp-0.2.0


キーボードショートカットCtrl + Shiftを使用してレイアウトをロシア語から英語に切り替える厳格なルールを設定して、レイアウトの切り替えの問題を解決します 。 セッションを作成することで、正しいことを確認します-1回目のチェック、切断、同じセッションでの再接続-もう一度確認します。



ターミナルで、一時変数の値を割り当てます。 



 kb_model="pc105" kb_layout="us,ru" kb_options="grp:ctrl_shift_toggle"


ソースファイルでコード置換を行います。 



 sed -ri.bak "s/(set.model)[^;]+/\1 = \"$kb_model\"/;s/(set.layout)[^;]+/\1 = \"$kb_layout\"/;s/(set.options)[^;]+/\1 = \"$kb_options\"/" xrdpkeyb/rdpKeyboard.c


xorg-xrdpシステムにバックエンドをコンパイルしてインストールします。 



 ./bootstrap ./configure make sudo make install


/etc/xrdp/xrdp.iniファイルの[X11rdp]ブロックと[Xorg]ブロックを交換して、ユーザーがデフォルトで正しいバックエンドを持つようにすることをお勧めします。 サーバーに接続するときは、セッションのリストでXorgを選択する必要があります。



デフォルトのポートを3389から別のポートに変更する必要がある場合は、ファイル/etc/xrdp/xrdp.iniを編集します。[Globals]セクションで、変数port = 3389の値を必要なものに変更します。



ファイアウォールをアクティブにして、ポート3389（または標準ポートが変更されたポート）を開きます。 



 firewall-cmd --add-port 3389/tcp --permanent firewall-cmd --reload


設定後、サービスを再起動します 



 service xrdp restart


これで、Windowsを備えた標準クライアントまたはrdpプロトコルをサポートする任意のクライアントを備えたlinuxを使用して、サーバーに接続できます。



注意 ：接続する前に、クライアントマシンのキーボードレイアウトが英語に設定されていることを確認する必要があります。そうしないと、xrdpサーバーのセッション選択ウィンドウにユーザー名とパスワードを入力できません。



インストール後、クリップボードが両方向のキリル文字で正常に動作していることを確認することをお勧めします。 また、ロシア語の文字を使用してクリップボードを介してファイルをコピーすることを確認できます（これらは、最新のxrdpサーバービルドで修正された主な問題の1つであり、以前はxrdpベースのターミナルサーバーが運用環境で正常に使用できなかった）



1C：Enterpriseバージョン8.3.9クライアントのインストール



プラットフォームの最新バージョンでは、1cがLinuxベースのOSのネイティブクライアントを多少なりとも快適に作業できる状態にしたため、これを使用し、ワインをインストールしようとせず、Windowsのクライアントの上に配置します。



正しい操作に必要な追加コンポーネントをインストールします。 



 sudo apt-get install libgsf-1-114 sudo apt-get install ttf-mscorefonts-installer sudo apt-get install imagemagick


クライアントおよびサーバー配布キットをサイト1cのフォルダーにダウンロードします。 



 sudo dpkg -i *.deb sudo apt-get -f install


サーバー1cのコンポーネントは依存関係を満たすためにのみ必要だったため、サービスを無効にします。 



 sudo update-rc.d srv1cv83 disable


ネットワークハスプキーが使用される場合、検索パラメーターを書き留めます。 



 sudo mkdir /opt/1C/v8.3/i386/conf sudo su echo [NH_COMMON] >/opt/1C/v8.3/i386/conf/nethasp.ini echo NH_TCPIP = Enabled>>/opt/1C/v8.3/i386/conf/nethasp.ini echo [NH_TCPIP]>>/opt/1C/v8.3/i386/conf/nethasp.ini echo NH_SERVER_ADDR = 192.168.0.100>>/opt/1C/v8.3/i386/conf/nethasp.ini echo NH_TCPIP_METHOD = UDP>>/opt/1C/v8.3/i386/conf/nethasp.ini


ソフトウェアライセンスを使用する場合は、コンフィギュレーターを使用してライセンスをアクティブ化する必要があります（空のファイルデータベースを使用できます）。



1Cプラットフォームによって作成された一時ファイルを使用した正しいセットアップ



このプロセスで、あるユーザーが表の部分を埋めるための外部処理を行い、別のユーザーがそれを行わない場合に問題が発生しました-エラーメッセージが表示されます。 問題は、1Cプラットフォームが一時ファイル/tmp/PrnForm.tmpを作成することです。 ファイルへのパスは、異なるユーザーに対して同じです。 したがって、処理を使用した最初のユーザーは、一時フォルダに自分だけが変更するアクセス権を持つファイルを暗黙的に作成しました。 別のユーザーはファイルを上書きできなくなります。



解決策は、システム全体のumaskを正しく構成し、ユーザーを同じグループに割り当て、さらに一時/ tmpフォルダーへの権限を正しく構成することです。



システム全体のマスクを編集します。 ファイルを編集します。 



 /etc/login.defs


その中で、値を次のように変更します。 



 umask 002


すべての端末ユーザーをUsersグループに割り当てます（以前に割り当てられていない場合は、別のグループに割り当てることができます）。



tmpフォルダーの権限を変更します 



 chown -R root:Users /tmp chmod g+s /tmp


Bitrix Webサイトで交換を使用する場合、/ var / log / bitrixフォルダーに対して同様の設定を行う必要があります。



1Cに関する同様の問題を解決するオプションは、 記事に記載されています 。



ZUP 2.5およびBukh 2.0データベースから宣言をアンロードできないという問題を解決する



Linuxの1Cクライアントの経理担当者が給与および経理部門から宣言をアンロードできないようにする不快な機能が1つあります。ファイルを選択してアップロードしようとすると、「ディレクトリ[ディレクトリ名]へのアクセスがありません」のようなメッセージが表示されます。 アップロードディレクトリの名前が正しいことを確認してください！



解決策が見つかったため、「処理：規制レポートの一般オブジェクト。規制レポートの保存のための設定の設定。フォーム。残念なことに。 、そのような変更については、サポートから構成を削除する必要があります。



リモートオフィスからのファイル共有と印刷を整理するためのガイドライン



サーバーがローカルネットワーク内でのみ機能する場合、問題はありません-sambaはうまく機能します。



企業に分散オフィスのネットワークがある場合、便利なファイル共有と印刷設定を整理するという問題がしばしば発生します。



ファイル共有には、専用のドロップボックスアカウントを使用すると便利です。サーバー上の専用フォルダーはクラウドと同期され、クライアントでは、ドロップボックスフォルダーを共有するか、同じアカウントでクライアントが構成されます。



リモートプリンターへの印刷には、仮想プライベートネットワーク構築スキームを使用すると便利です。この場合、固定IPアドレスを持たないリモートポイントでも問題は解決します。 プリンターへのアクセスは、VPNネットワーク内のIPアドレスによって可能になります。 オプションとして、 NeoRouter Freeを使用することをお勧めします。 この製品のカスタマイズは、この記事の範囲外です。



可能な代替案



残念ながら、現時点では安定して動作する無料の代替手段はありません。 OpenSuse 11.2 + FreeNX（etersoftから）+ WineEtersoft（購入済み）バンドルを使用して8.1プラットフォームに基づく構成を操作することでかなり成功した経験がありましたが、現時点では、新しいインストール用のそのようなバンドルは関連性を失いました-インストールのためにすべてのパッケージが見つかるわけではありませんLinux OSおよび1C構成の新しいバージョンのレンダリング速度はひどいです。



x2Goなどの他の製品のテストに多くの時間を費やしましたが、多くのBUTもあります。Windows8、Windows 10などの最新バージョンのOSから接続する場合、イメージのレンダリングなどにアーティファクトがあります。 一般的に、代替ソリューションが管理機能に何らかの形でまだ適している場合、1Cソリューションを使用した実際の作業ではあまり適切ではありません。



おわりに



移行の可能性を考慮する場合、考慮されていないソリューションでは、商用機器の可用性などの要因を考慮する必要があります。 Linuxですべての取引機器がサポートされているわけではなく、RDPを介して実行できるようにさらに少ないものもあります。



一般に、Windowsライセンスを節約したい場合は、特に最近のXRDPの開発が加速されているため、LINUXベースのソリューションで十分です。 今年の3月には、バージョン0.9.2のリリースが計画されています 。これは、公開されているリリースノートに続きます 。
