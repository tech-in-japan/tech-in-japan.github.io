マイクロソフトとモノのインターネット？ この概念の見方に関する入門記事

こんにちは



今日と次の2週間には、ハブブログにテーマセクションがあります。過去数年にわたって、大企業（Microsoft、Intel、HPなど）と中小企業の両方から多くの注意が払われてきましたが、それほど興味深いものではありません。プロジェクト（例： Aura ）-つまり、モノのインターネット。 私たちが行うすべてをお教えします。









Habrahabrはすでにインターネット上でより多くの人々がいること、インターネットがどのように発展しているのか、なぜそうなっているのか、そしてモノの インターネットとDYIの ために開発しているハブには 、トレーニングを受けた人に数か月間、大規模な紹介なしに、IoTの概念内での当社の位置、我々が行うこと、友人である方法、パートナーとの行動（ ArduinoやCanonicalなど ）蒸気を示す 建築ソリューションで。 私たちは、IoTにおけるビジョンを完全に理解するためにこれを行っています。



しかし、文脈なしでは難しいので、背景についての考えはほとんどありません。 多くの人が長い間予測していたデバイスの数の急速な増加が起こりました（さまざまな分析機関や企業は、Webで簡単に見つけられる異なる番号を定期的に提供します-通常、これらは7ゼロの数字です）およびこれらのデバイスからのネットワークトラフィック。



インターネット上のこれらのデバイスの相互作用は、単にモノのインターネットと呼ばれていました。 ただし、この用語はまったく新しいものではありません。 99年に作られました 。 以前の用語はM2M（Machine-to-Machine、定期的にこれら2つの用語は同じコンテキストで使用されます）であり、通信業界で発生し、 接続されたマシン間のネットワークを介したデータ交換に関連付けられています 。 IoTの本質は同じように縮小できますが、IoTはより広範な概念と見なされており、その内部には、データ交換と可能な相互制御を備えたM2Mよりも多くのツールとテクノロジーがあります。 M2Mの標準化はETSIによって実行されます。 M2M = IoTコンセプトを「接着」します。



15年後にようやくそのような積極的な関心が生じたのはなぜですか？ 90年代のモノのインターネットは、その周りに多数の謝罪者がいたという用語だけでなく、何が不十分でしたか？



さまざまな意見がありますが、スピーチでは常にこの質問をしています。 答えは常に異なりますが、基本的に分布は次の点に減らすことができます。



  • 安価なセンサー（10年で1.30ドル=> 60c） -センサーのコストを削減することで、ほぼすべての環境で誰でもプロジェクトを作成できるようになりました。
  • 安価な通信（10年間で40倍）。 ネットワークインフラストラクチャ（M2M）は、あらゆるIoTプロジェクトの重要なコンポーネントです。 プロセスの遅延を取得する価値があります-データはその意味を失う可能性があります。 人でさえ、特定の数を超える遅延を不満足と主観的に知覚します。 RTまたはnRTシステムはさらに厳しいものです。 一般に、通信インフラストラクチャの信頼性と幅が広いほど、大量のデータが適切な宛先に時間通りに送信されるのが快適になります。 これらの問題の詳細については、 ASUSの同僚に相談してください。
  • スマートフォンと開発プラットフォームの出現。 基礎となる「鉄」インフラストラクチャは優れていますが、何かが本当に主流になるには、このインフラストラクチャを管理するための使い慣れた便利なツールが必要です。 例は、マシンと家のリモートコントロールです。 複数の異なるリモートを作成することも、1つのモバイルアプリケーションを使用することもできます。
  • 広いネットワークカバレッジにより、工場やタワーなどの遠隔地に到達できます。
  • IPv6 -IPv4のアドレス数の期限切れに伴い、ネットワーク上のオブジェクトをアドレス指定するための新しいソリューションの検索が開始されました。 もちろん、技術的な理由だけでなくセキュリティ上の理由から、すべてのデバイスをIPv6でアドレス指定することはできませんが、作業は進行中です。
  • クラウドの出現は、産業や趣味のコミュニティへの関心を呼び起こしました-大量のデータを処理するのに役立つ安価なリソースを得ることは、過去にIoTプロジェクトを完全に展開できなかったブロッカーの1つであることが判明しました。




それにもかかわらず、理由がどうであれ、私たちはすでに新しいソリューションと新しいアプローチを必要とする状況にあります。 IoTが適用されない領域の名前を付けるのは困難です。ここには、スマートホーム、ロジスティクスタスク、医療、およびConnectedプレフィックス（Cars＆Cowsを含む-クラウドでの牛と機械学習について、 プレナリーの作成を参照）機械学習のコーポレートバイスプレジデント、ジョセフシロシュから）。 また、特定のシナリオ（たとえば、同じコネクテッドカー）が既に実装されている場合でも、現在、モノのインターネットの概念に適合するための既存のソリューションの改良が見られます。 たとえば、Microsoft Azureで最近開かれたプログラムでは、Microsoft Azure Machine Learning内でリアルタイムデータ転送の機能を試すことができます（ Habréの詳細を参照してください ）。 機械学習は成熟したトピックですが、サービスとして実装し、リアルタイムデータストリームとしてデバイスからデバイスに何が起こっているかを常に送信するメッセージを接続することで、最初にデータセットを収集してから静的な形で実験に渡すのではなく、すべてを自動化できます。







このすべてをマイクロソフトでどのように見ていますか？







私たちは、基礎となる「鉄」から抽象化して、モノのインターネットの概念を、すでにあるものから始めて見ていきます。 既存のインフラストラクチャの効率を改善することから。 すでにそこにあるものからより多くを得ることから。 これをどうやってやるの？



モノのインターネットを次の3つの観点で検討します。



  • クライアント（これはWindows 10についてです）
  • クラウド（これはMicrosoft Azureについてです）
  • 適用済み（これはMicrosoft Band / HoloLensに関するものです）




交差する場合としない場合があります。 以下では、最初の2つを詳しく見ていきます。







クライアントから始めましょう。 モノのインターネットのエコシステムには、スマートフォン、タブレットなどのさまざまなクライアントデバイスが含まれます。これらは、一方ではユーザーとのインターフェイスを提供し、他方では所有者の行動に関する興味深いデータを蓄積できます。 Microsoft側の状況は次のとおりです。新しいWindows 10オペレーティングシステムには、小型デバイスで使用するために設計された特別なエディションがあります。 IoT編集委員会は既にリリースされており、コミュニティで使用方法と使用方法に関する記事が掲載され始めています。 同時に、Wi-Fiや何らかの方法でUSBやオンボードカメラを含むさまざまな機器のサポートが不足しているという事実があるため、このエディションに今後何が起こるかについて疑問が生じ始めました。 ここでは、IoTバージョンが積極的に開発されており、すべての一般的なシナリオのサポートが実装されることを明確にする必要があります。 もちろん、すべてをすぐに実行できるわけではありません-一部はこのシナリオが本当に需要があるかどうかを確認する必要があるという事実、一部はマイクロソフト内部で行われているアクティブなテストのためです（多くの従業員が会社に積極的に参加しています）テストプログラムでは-各アセンブリのテストには通常かなりの時間がかかります）-したがって、必要なコンポーネントの一部が不足しています。 Windows 10 IoTで何を見たいかをお知らせするのに最適な時期です。







現在、Windows 10 IoTは3つのボード（Raspberry Pi 2、Intel Minnowboard MAX、およびQualcomm Dragonboard 410c）にインストールできます。 これらのボードはそれぞれ独自のタスク用に設計されており、たとえば、自宅で簡単なタスクを自動化したい人はRaspberry Pi 2を好むでしょう。WindowsEmbeddedに近い業界では、MinnowboardとDragonboardが興味を持っています。



この2週間で、これを行う方法についてもう少し具体的になります。



私の意見では、Windows 10 IoTに加えて、Windows 10内のAllJoynサポートに関する非常に重要なニュースを非常に気付かずに通過します。AllJoynは、業界ですでに事実上の標準になった人気のある対話プロトコルです。 来週、Akvelon（Ivanovo）のAllJoynの直接開発者から記事がリリースされます。 要するに、AllJoynの本質は、デバイスの種類に関係なく、WiFiおよびBluetoothを介したアプリケーション、デバイス、およびユーザーの相互作用を実装するために必要なフレームワークおよび対応するオープンソースツールであるという事実に還元できます。 同時に、AllJoynは「それ自体」ではなく、 真面目なプレイヤーが参加するプロジェクトです。







これらすべての主なニュースは、AllJoynがWindows 10に統合されることです。







原則として、Microsoft IoTのクライアント側に適用されるのはこれだけです。 Embeddedに携わっている場合-Windows 10をご覧ください（多くのクライアントとインテグレーターが既にこれを行っています）、Visual Studioでボード上で動作するアプリケーションを作成したい場合- それを行うことを忘れないでください：プレビュー。 コメントや見たいものがある場合は、 必ず私たちに書いてください 。



また、来週、私たちが締結した協力についての資料を書きます-例えば、ArduinoやCanonicalと:)



2つ目のカテゴリであるクラウドに移りましょう。 私たちはずっと長い間クラウドにいます-私たちのクラウドプラットフォームは最近5周年を迎え、スコットガスリーのリーダーシップの下での開発は飛躍的に進んでいます。 ただし、すべてがそれほど単純ではありません。MicrosoftAzureでIoTプロジェクトを構築するには、適切なツールを選択する必要があります。 現在利用可能なすべてのツールがIoTに適しているわけではありません-同じサービスバスキューが数万のデバイスからのメッセージの圧力に耐えられない場合、完全に正常です-これは、メッセージをキューに収集するタスク向けではありません：イベントハブを作成しました。 などなど。



写真を見てみましょう。







写真では、実際のビジネスで使用できるようになったすべてのIoT関連サービスを収集しようとしました。 この場合、方向は左から右（メッセージが送信され、最終的に何らかの出力ツールに送られる）、および右から左（ユーザーがWebサイト上のボタンを使用して、センサーやスプリンクラーなどのエンドデバイスでコマンドの実行を開始する場合） ）



IoTプロジェクトには1つのレシピがないため、各プロジェクトのアーキテクチャは個別に考慮され、複数のサービスまたは1つを使用して、たとえばメッセージを保存できます。



要するに、実際には、最も関連性の高いサービス：



  • イベントハブ（イベントハブ）-さまざまなセンサーから大量のデータを収集します。
  • ストリーム分析（ストリーム分析）。これにより、イベントハブからの「ライブ」データストリームに対する要求を判別できます。これにより、予備処理を「オンザフライ」で実装し、イベントへの反応を整理できます。
  • Azure ML-利用可能なデータで特定のモデルをトレーニングし、予測または結論を形成するためのWebサービスとしてそのモデルへのアクセスを提供できる機械学習テクノロジー。
  • Azure HD Insightは、大量のデータを処理し、MapReduceなどのアルゴリズムを使用してさまざまなデータスライスと集計値を取得するためのクラスターテクノロジーです。
  • Microsoft Azureクラウドプラットフォーム-さまざまな形式（Azure SQLリレーショナルDBMS、Azure Storage準構造化テーブル、NoSQL DBMS）の大量のデータを保存するためのテクノロジー。
