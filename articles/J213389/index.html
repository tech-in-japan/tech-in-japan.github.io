ServiceStack？ いいえ、聞いていません...





まず、ServiceStack自体とは何かという質問に答えましょう。 これは、WebサービスとWebアプリケーションを作成するための本格的なフレームワークであり、C＃とASP.Netの世界最高の機能を組み込んでいます。 -すべての最悪の、不要な、不要なものを取り除きました。 しかし、親愛なるハラジテル、私たちはすぐに警告します、ホリバーはありません、今週の木曜日ではなく、私たちにもありません！ 今後、コンポーネントを単一のシステムに接続するアーキテクチャ上のバリエーションがあり、ServiceStack（サーバーとして）と、ASP.Net MVC（クライアントとして）の両方に記述されていることに注意してください。



このフレームワークはBBC社の腸内で生まれ、 デミスベロッテはそこで働いていました。 フレームワークの開発の始まりは、2007年と見なすことができます。BBCは、マイクロソフトの標準エンタープライズソリューションを使用して、急速に変化する要件の条件での開発へのウォーターフォールアプローチと相まって、穴に直接落ち、「ドローダウン」は映画の主人公ニキータミガルコフによって与えられた定義に近くなります」 Zhmurki」、説明辞書の定義ではなく*。







この問題は、あらゆる小さな変更を行うために、UMLから数日かかるコードまでのサイクル、古いレコードと新しいレコード間のフォーマットの互換性のためのあらゆる種類の松葉杖、およびメジャーリリース間の毎週のダウンタイムを経る必要があることでした。



チームは簡単な質問を投げかけました。サービスはすべて同じですか？ 質問は原則として複雑ではなく、私たちの多くはそれに対して同じ答えをします：



  • これは、特定のプラットフォームまたはテクノロジーを強制する方法ではありません。
  • これは、事前定義された制限を課す方法ではありません。
  • これは、事前に定義されたフォーマットを使用したり、事前に定義されたアーキテクチャに従う理由にはなりません。
  • これは、サービスの提供であり、技術的な言語、再利用可能な機能のカプセル化、および可用性の組織です。
  • 最小限のコストと最大限の再利用で、最も簡単かつ効率的な方法で、できるだけ多くの顧客に可用性をアレンジします。




これがサービスの定義です。 これに加えて、優れたサービスは、今後の変更を助長し、変更が問題にならないように設計する必要があります。



この問題を反映してServiceStackが誕生しました。これにより、チームの半分だけが以前のチーム全体よりも良い結果を達成することができました。



それでも、彼は何が良いですか？





  • より直感的で使いやすい。
  • 標準の.Netソリューションと比較して高速
  • 少ない直接的な手でも、高レベルのコードの再利用を実現できます




最後の点に関しては、もちろん、高レベルのコードの再利用に直接手を携えて、標準のMicrosoftソリューションを使用して達成できる可能性が十分にあることを予約する必要があります。 ただし、この場合も、まっすぐな腕と適切なアプローチと経験が必要です。 ServiceStackアーキテクチャの単純さと比較天才により、スキルの低い開発者はより良い結果を達成でき、コード構造に対するより効率的なアプローチへのコミットメントを開発することもできます。

ServiceStackプロジェクトは、すぐにLinuxにデプロイできます！



したがって、高価なWindows Azureの費用を支払う必要はなく、さらに高価な独自のWindows Serverを維持する必要もありません。 中規模のプロジェクトの場合、通常のサーバーでも、たとえばDebianまたはUbuntuサーバー、または月額30ドルでHetzner VPSを使用できます。 小さなプロジェクトの場合、 Hetznerで10ドル、またはDigital Oceanで5 ドルの最も安いVPSが使用されます。



私たちのパートナーは、nginx + FastCGI Monoを使用してLinuxでServiceStackプロジェクトを展開およびホストした経験があり、これに興味があるなら、一緒に尋ねても、彼らが私たちにこれは詳細に行われます。



技術的な詳細を教えてもらえますか？



不可能です。 ServiceStackの開発者は、新しいASP.Netを構築するためにASP.Net全体を破壊し始めませんでした。 ルート-つまりIHttpHandler-が去りました。 しかし、さらに進むものはすべて破壊され、より軽量な代替手段が使用されました。代わりに、Webサービスの開発の問題を解決するために、それら自体が少し具体的に適合しました。 フレームワークは、キャッシュマネージャー、認証プロバイダー、およびセッションを提供し、独自のRedqベースのMQブローカーも提供します。 ちなみに、フレームワークのフレームワーク内で、 独自のRedis-clientも作成しました。個人的な意見では、これは単純に豪華で、さまざまなコンテキストでフレームワークとは別に使用できます。 標準のIOCコンテナとして、かなり人気のあるFunc軽量コンテナの わずかに変更されたバージョン が使用されます。 確かに、それの代わりに、利用可能な多くの他のIOCコンテナを使用できます。



ServiceStackの基盤はDTO（データ転送オブジェクト）であり、フラットなPOCOオブジェクトに基づいています。 認証プロバイダーとセッション、およびユーザーサービスはPOCOに基づいており、さまざまな実装にさまざまなバックエンドを使用できます。 たとえば、すぐに使用できるように、入力されたセッションは、メモリとRedisに直接保存できます。 型指定されていないセッションは、ダイナミズムの愛好家にも利用できますが、残念ながら、それは多少遅くなります。



残念ながら、1つの記事でフレームワークのアーキテクチャを詳細に検討するのは現実的ではありませんが、好奇心Ha盛なハブロフスク市民からの関心があれば、将来、これに関する一連の記事を書くことができます。 それまでの間、読者の食欲をいじめるためにちょっとしたHelloWorldを書いてください。

  • DTOサービスリクエストのクラスを作成する
  • DTOサービスレスポンスのクラスを作成する
  • サービス自体のクラスを作成し、基本クラスからそれを継承し、実装する必要があるHTTPメソッドで同じ名前のメソッドをオーバーライドします
  • 人生を楽しんで利益を得る（冗談...しかし、 いや、冗談ではない ）




簡単なサービスを書きましょう。



Visual StudioまたはXamarin Studioで新しいプロジェクトを作成します。 手元に最初のものが存在しない場合、幸いなことに、モノの下ですべてがうまく機能します（そして、私たちにとってこの要素は最初は決定的でした）。 そして、最高のIDEを使用しないことに対する補償として、最高のモバイルOSのためにモバイルアプリケーションからこのサービスを使用する方法を示します。



したがって、空のASP.Netプロジェクト（空のWebアプリケーション）を作成します。 ソリューションをSSHelloWorld、プロジェクトをSSHelloAPIと呼びます。



ソリューションエクスプローラーのプロジェクトの場合、右クリックして[Nugetパッケージの管理]を選択します。







「追加」ボタンをクリックすると、ライセンス契約に同意するかどうか尋ねられます。 まあ、なぜなら ロシア語を話すユーザーは、他の誰よりも早くライセンス契約を読み、同意し、ダウンロードしたServiceStack dllがプロジェクトに接続されます。



プロジェクトにGlobal.asaxがない場合（理論上は存在しないはずです）、それを追加し、次のコードをそれが何であるかではなくGlobal.asax.csにコピーします。 



using System; using System.Collections; using System.ComponentModel; using System.Web; using System.Web.SessionState; using ServiceStack; namespace SSHelloAPI { public class Global : System.Web.HttpApplication { public class AppHost : AppHostBase { public AppHost() : base("Hello API", typeof(HelloService).Assembly) {} public override void Configure (Funq.Container container) { } } protected void Application_Start (Object sender, EventArgs e) { new AppHost ().Init (); } } }




コード内のエラーを強調表示することに注意を払わないでください。 サービスを配置するクラスを作成する必要があります。



また、おそらくweb.configファイルを手動で作成する必要があります 。 また、そのコードはすべて次のように置き換える必要があります。 



 <?xml version="1.0"?> <configuration> <system.web> <httpHandlers> <add path="*" type="ServiceStack.HttpHandlerFactory, ServiceStack" verb="*"/> </httpHandlers> </system.web> <!-- Required for IIS 7.0 (and above?) --> <system.webServer> <validation validateIntegratedModeConfiguration="false" /> <handlers> <add path="*" name="ServiceStack.Factory" type="ServiceStack.HttpHandlerFactory, ServiceStack" verb="*" preCondition="integratedMode" resourceType="Unspecified" allowPathInfo="true" /> </handlers> </system.webServer> </configuration>




最後に、3つのクラスを含む新しいHelloService.csファイルを作成します。 



 using System; using ServiceStack; namespace SSHelloAPI { [Route("/hello")] [Route("/hello/{Name}")] public class HelloRequest { public string Name { get; set; } } public class HelloResponse { public string Result { get; set; } } public class HelloService : Service { public object Any(HelloRequest request) { return new HelloResponse { Result = "Hello, " + request.Name }; } } }




すべてがシンプルで明確です。そのようなコードについてコメントすることさえ恥ずかしく思います。 プロジェクトをコンパイルして実行し、ブラウザのアドレスバーで、ホスト名とポートの後に「/ hello / Habrauser」と入力します。 リクエストヘッダー自体によるServiceStackは、クライアントのタイプを決定します。この場合、htmlラッパーでサービスの応答を表示します。 [結果]フィールドには、サービスが私たちをどのように歓迎しているかが表示されます。



Anyメソッドの名前は、すべてのHTTPメソッドに同じロジックを定義したことを意味します。



プロジェクトを開始すると最初に開くメタデータページに注意してください。 自動的に生成されたサービスドキュメントが含まれています。 ご覧のとおり、ServiceStackは外出先でさまざまなエンドポイントを作成し、RESTfulサービスとSOAPサービス（はい、SOAP、親愛なる読者）の両方でサービスを操作できます。







もちろん、このサービスは退屈ですが、記事を過負荷にしないために、すぐにもっと複雑な例を含めませんでした。 しかし、それはあなたにSericeStackの味を感じさせます。興味があれば、今-私たちの有無にかかわらず、もっと味わうことができます。



ServiceStackがモバイル開発者の生活を楽にする方法





ServiceStackのもう1つの良い点は、Windows Phone、Xamarin.iOSまたはXamarin.Android-applicationで作成したサービスを使用すると非常に便利なことです（ただし、最後の2つはXamarinのビジネスライセンスが必要です）。 さまざまな方法がありますが、最も単純な方法（ただし、最も美しい方法ではありません）は、RequestクラスとResponseクラスを別々のファイルに選択し、モバイルアプリケーションプロジェクトへのリンクを介して接続することです。 また、NugetからServiceStack.Clientアセンブリをインストールします。



次に、ボタンクリックイベントでこのようなハンドラーを切断するとします。 



  button.TouchUpInside += (object sender, EventArgs e) => { try { var client = new JsonServiceClient("http://127.0.0.1:8080"); HelloResponse response = client.Get<HelloResponse>(new HelloRequest() { Name = "iPhone" }); resultLabel.Text = response.Result; } catch (Exception ex) { resultLabel.Text = ex.Message; } };

ライセンスはどうですか？





バージョン4.0から、このプロジェクトは商用カラーになりました。 まあ、なぜそれが良くなるのですか？ フレームワークのソースコードはまだ開いていますが、このライセンスでは、次の制限がある小さなプロジェクトにのみ無料でフレームワークを使用できます。



  • 10個の操作（つまり、10個のDTOリクエスト）
  • OrmLiteの10テーブル
  • JSON、JSV、およびCSVシリアライザーを備えた20のタイプ
  • Redis Clientの20のタイプ
  • Redisの1時間あたり6,000リクエスト




まあ、大丈夫、単一のRedisではなく、OrmLiteを除いて他のORMがありますが、もちろん10の操作はすでに重大な制限です。



ライセンスと価格の詳細については、 このページをご覧ください 。



現時点では、BSDライセンスの下で利用可能なプロジェクトで3番目のバージョンを使用しており、4番目への移行のみを検討しています。 「営利目的」に接続したくない場合は、3番目のブランチを安全に使用してプロジェクトで使用できます。これは非常に成熟していて安定しています。
