プロローグ



この話は3年以上前に始まりました。 私たちの小さな会社DAVAはWargamingの一部になり、次にどのプロジェクトを行うべきかを考え始めました。 3年前のモバイルがどのようなものだったかを思い出すために、クラッシュオブクラン、パズル＆ドラゴン、今日の非常に有名なプロジェクトの多くはなかったと言います。 ミッドコアはまだ始まったばかりです。 市場は今日よりも何倍も小さかった。



最初は、新しいユーザーを大きな「タンク」に引き付けるいくつかの小さなゲームを作成することは非常に良い考えであると皆に思われました。 一連の実験の後、これは機能しなかったことが判明しました。 モバイルアプリケーションでの優れた変換にもかかわらず、モバイルからPCへの移行はユーザーにとって深byであることが判明しました。



その後、開発中にいくつかのゲームがありました。 それらの1つはスナイパーと呼ばれました。 主なゲームプレイのアイデアは、スナイパーモードで、防御側に立っているタンクから、AIによって制御され、応答して攻撃できる他のタンクで射撃することでした。



ある時点で、スタンディングタンクは非常に退屈であるように思われ、1週間でプロトタイプのマルチプレイヤーを作成しました。



これがすべての始まりです！



「スナイパー」の開発を開始したとき、モバイルプラットフォームで利用できるテクノロジーを検討しました。 当時、Unityはまだ開発のかなり初期の段階にありました。実際、必要なテクノロジーはまだ存在していませんでした。



欠けていた主なものは、ダイナミックなディテールを備えたランドスケープのレンダリングでした。これは、オープンスペースでゲームを作成するために不可欠です。 Unityにはサードパーティのライブラリがいくつかありましたが、それらの品質は低かったです。



また、C＃では、開発中のデバイスを最大限に活用することはできず、常に制限されることも理解していました。

Unreal Engine 3は、多くの同様の理由からも適切ではありませんでした。



その結果、エンジンを改良することにしました！



これは、以前のカジュアルプロジェクトで既に使用されていました。 このエンジンは、プラットフォームでサポートされるiOS、PC、Macの低レベルの作業がかなりよく書かれており、Androidでの作業が開始されました。 2Dゲームを作成するための多くの機能が記述されています。 つまり、2Dで作業するための優れたUIと多くのものがありました。 ゲームの1つは完全に3次元であったため、3Dパートの最初のステップでした。

約6か月が経過し、小さなミニプロジェクトがBlitzになりました。 完全に新しい要件があります。MMO、AAA品質、およびその当時の元の形式のエンジンではもはや提供できなかったその他の要件です。 しかし、仕事は本格的でした。 ゲームはうまくいき、うまくいった。 ただし、パフォーマンスは平均的で、カードにはオブジェクトがほとんどなく、実際には他の多くの制限がありました。



この段階で、私たちはエンジンに置いた基盤が実際のプロジェクトの報道に耐えられないことを理解し始めました。



当時の仕組み


すべてのシーンレンダリングは、シンプルなシーングラフのコンセプトに基づいていました。



主な概念は2つのクラスでした。



  • シーン-シーンのコンテナ。内部ですべてのアクションが行われました
  • ステージ上。
  • SceneNode-シーン内のすべてのクラスが継承するシーンノードの基本クラス：
  • MeshInstanceNode-メッシュをレンダリングするためのクラス。
  • LodNode-lodを切り替えるクラス。
  • SwitchNode-スイッチオブジェクトを切り替えるためのクラス。
  • SceneNodeの継承者のクラスはさらに15個あります。


SceneNodeクラスにより、仮想メソッドのセットをオーバーライドして、ある種のカスタム機能を実装できました。

再定義できる主な機能は次のとおりです。



  • 更新-更新シーンを作成するために各ノードに対して呼び出された関数。
  • Drawは、このノードを描画するために各ノードに対して呼び出された関数です。


私たちが遭遇した主な問題。



まず、パフォーマンス：



  • レベル内のノードの数が5000に達すると、空の更新関数をすべて通過するのに約3ミリ秒かかることが判明しました。
  • Drawを必要としない空のノードでも同様の時間が費やされました。
  • 作業は常に異種データで行われているため、膨大な数のキャッシュミス。
  • 複数のコアで作業を並列化できない。


第二に、予測不可能性：



  • 基本クラスのコードを変更すると、システム全体に影響します。つまり、SceneNode :: Updateのすべての変更は、何でもどこでも壊れる可能性があります。 依存関係はますます複雑になり、エンジン内の各変更により、関連するすべての機能のテストが必要になることがほぼ保証されました。
  • シーンの残りの部分を傷つけないように、変換などでローカルに変更することは不可能でした。 非常に多くの場合、LodNode（lodを切り替えるためのノード）の最小の変更がゲーム内の何かを壊しました。


状況を改善するための最初のステップ



まず、パフォーマンスの問題を処理し、すぐに実行することにしました。



実際には、各ノードに追加のフラグNEED_UPDATEを導入することでこれを行いました。 彼は、そのようなノードをUpdateと呼ぶべきかどうかを決定しました。 これにより生産性は大幅に向上しましたが、多くの問題が発生しました。 実際、Update関数のコードは次のようになりました。 



void SceneNode::Update(float timeElapsed) { if (!(flags & NEED_UPDATE))return; // the rest of the update function // process children }


これによりパフォーマンスの一部が返されましたが、多くの論理的な問題が予想外の場所から始まりました。



LodNodeとSwitchNode-それぞれ（距離による）lodの切り替えとオブジェクト（たとえば、破壊されたものと破壊されていないもの）の切り替えを担当するノード-は定期的に壊れ始めました。



時々、故障を修正しようとした人は次のことをしました：基本クラスでNEED_UPDATEを無効にし（結局、単純な解決策でした）、完全に気づかれずに、FPSは再び落ちました。



NEED_UPDATEフラグをチェックするコードが3回コメントアウトされたとき、根本的な変更を決定しました。 一度にすべてを行うことはできないことを理解したため、段階的に行動することにしました。



最初のステップは、アーキテクチャを敷設することでした。これにより、将来発生するすべての問題を解決できます。



目標


  • 独立したサブシステム間の依存性の最小化。
  • 変換の変更はlodのシステムを破壊するべきではありません。
  • マルチコアにコードを配置する機能。
  • 異種の独立したコードが実行される更新機能などはありませんでした。 古い機能を完全に再テストすることなく、新しい機能を使用してシステムを簡単に拡張できます。 一部のサブシステムを変更しても、他のサブシステムには影響しません。 サブシステムの最大の独立性。
  • 最大のパフォーマンスを得るために、メモリ内でデータを直線的に配置する機能。


最初の段階での主な目標は、これらすべての目標を達成できるようにアーキテクチャを再設計することでした。



コンポーネントとデータ駆動型アプローチの組み合わせ



この問題の解決策は、データ駆動型アプローチと組み合わせたコンポーネントアプローチでした。 さらに本文では、翻訳が成功しなかったため、データ駆動型のアプローチを使用します。



一般に、多くの人はコンポーネントアプローチについて異なる理解を持っています。 データ駆動型でも同じです。



私の理解では、 コンポーネントアプローチは、いくつかの必要な機能が独立したコンポーネントに基づいて構築される場合です。 最も単純な例は電子機器です。 チップがあり、各チップには入力と出力があります。 チップが適合する場合は、接続できます。 エレクトロニクス業界全体は、このアプローチに基づいて構築されています。 数千の異なるコンポーネントがあります。それらを互いに組み合わせると、まったく異なるものを取得できます。



このアプローチの主な利点は、各コンポーネントが分離されており、独立性が高いことです。 コンポーネントが誤ったデータを送信する可能性があり、ボードが焼損するという事実を考慮していません。 このアプローチの利点は明らかです。 今日、膨大な数の既製のチップを取り、新しいデバイスを組み立てることができます。



データ駆動型とは何ですか。 私の理解では、これはソフトウェア設計へのアプローチであり、ロジックではなくデータがプログラムフローの基礎として使用されます。



この例では、次のクラス階層を想像してください。 



 class SceneNode { //      Matrix4 localTransform; Matrix4 worldTransform; virtual void Update(); virtual void Draw(); Vector<SceneNode*> children; } class LodNode { //  c    LodDistance lods[4]; virtual void Update(); //   Update,       ,    -    virtual void Draw(); //      }; class MeshNode { RenderMesh * mesh; virtual void Draw(); //   };


この階層のバイパスコードは、次のように階層的に見えます。 



 Main Loop: rootNode->Update(); rootNode->Draw();


このC ++継承の階層には、3つの異なる独立したデータストリームがあります。



  • 変換
  • ロダ
  • メッシュ


ノードはノードを階層内で結合するだけですが、各データストリームを順番に処理する方がよいことを理解することが重要です。 階層処理の実際的な必要性は、変換にのみ必要です。



データ駆動型アプローチでどのように見えるべきか想像してみましょう。 アイデアが明確になるように、擬似コードで記述します。 



 // Transform Data Loop: for (each localTransform in localTransformArray) { worldTransform = parent->worldTransform * localTransform; } // Lod Data Loop: for (each lod in lodArray) { // calculate lod distance and find nearest lod nearestRenderObject = GetNearestRenderObject(lod); renderObjectIndex = GetLodObjectRenderObjectIndex(lod); renderObjectArray[renderObjectIndex] = renderObject; } // Mesh Render Data Loop: for (each renderObject in renderObjectArray) { RenderMesh(renderObject); }


実際、プログラムの作業サイクルを開始し、すべてがデータに基づいているようにしました。



データ駆動型アプローチのデータは、プログラムの重要な要素です。 ロジックは単なるデータ処理メカニズムです。



新しいアーキテクチャ



ある時点で、エンティティをベースにしたアプローチでシーンを整理する必要があることが明らかになりました。エンティティは、多くの独立したコンポーネントで構成されるエンティティでした。 コンポーネントが完全に任意であり、相互に簡単に結合できるようにしたかったのです。



このトピックに関する情報を読んでいると、 T-Machineブログに出会いました。



彼は私の質問に多くの答えをくれましたが、主な答えは次のとおりです。



•エンティティにはロジックは含まれず、単なるID（またはポインター）です。

•エンティティは、それに属するコンポーネントID（またはポインタ）のみを知っています。

•コンポーネントは単なるデータ、つまりです。 コンポーネントにはロジックが含まれていません。

•システムは、特定のデータセットを処理し、出力で別のデータセットを出力できるコードです。



これに気付いたとき、さまざまな情報をさらに研究する過程で、 Artemis Frameworkに出会い、このアプローチの適切な実装を見ました。

今日のArtemisはECS（Entity Component System）と呼ばれています。 エンティティ、コンポーネント、およびデータ駆動型に基づいてシーンを整理するための多くのオプションがありますが、その結果、ECSアーキテクチャに到達しました。 この用語がどれほど一般的かを言うのは困難ですが、ECSは、 エンティティ、コンポーネント、システムというエンティティがあることを意味します。



他のアプローチとの最も重要な違いは、コンポーネントの動作ロジックの必須の欠如、およびシステムのコード分離です。



この点は、「正統派」コンポーネントアプローチでは非常に重要です。 最初の原則に違反すると、多くの誘惑があります。 最初の1つは、コンポーネントの継承を行うことです。



その柔軟性にもかかわらず、通常はパスタで終わります。







最初は、このアプローチを使用すると、同様の動作をするがわずかに異なる動作をする多くのコンポーネントを作成できるように思われます。 共通コンポーネントインターフェイス。 一般に、再び継承のtrapに陥ることがあります。 はい、それは古典的な継承よりも少し良いでしょうが、このtrapに陥らないようにしてください。

3番目のクラスはSceneSystemです。 



  /** \brief This function is called when any entity registered to scene. RegisterEntity、UnregisterEntity関数は、シーンにエンティティを追加または削除するときに、シーン内のすべてのシステムに対して呼び出されます。 



  /** \brief This function is called when any component is registered to scene. It sorts out is entity has all necessary components and we need to call AddEntity. \param[in] entity entity we added component to. \param[in] component component we've just added to entity. */ virtual void RegisterComponent(Entity * entity, Component * component); /** \brief This function is called when any component is unregistered from scene. It sorts out is entity has all necessary components and we need to call RemoveEntity. \param[in] entity entity we removed component from. \param[in] component component we've just removed from entity. */ virtual void UnregisterComponent(Entity * entity, Component * component);


RegisterComponent、UnregisterComponent関数は、シーン内のエンティティにコンポーネントを追加または削除するときに、シーン内のすべてのシステムに対して呼び出されます。

また、便宜上、さらに2つの機能があります。 



  /** \brief This function is called only when entity has all required components. \param[in] entity entity we want to add. */ virtual void AddEntity(Entity * entity); /** \brief This function is called only when entity had all required components, and don't have them anymore. \param[in] entity entity we want to remove. */ virtual void RemoveEntity(Entity * entity);


これらの関数は、コンポーネントの順序セットがSetRequiredComponents関数を使用して既に作成されているときに呼び出されます。



たとえば、ACTION_COMPONENTおよびSOUND_COMPONENTを持つエンティティのみの受信を注文できます。 これをSetRequiredComponentsに渡します。



これがどのように機能するかを理解するために、どのシステムを使用しているかを例で説明します。



  • TransformSystem-変換の階層を担当するシステム。
  • SwitchSystem-破壊された状態と破壊されていない状態など、いくつかの状態になる可能性のあるオブジェクトの切り替えを担当するシステム。
  • LodSystem-距離によってlodを切り替える役割を担うシステム。
  • ParticleEffectSystem-パーティクル効果を更新するシステム。
  • RenderUpdateSystem-シーングラフからレンダーオブジェクトを更新するシステム。
  • LightUpdateSystem-シーングラフから光源を更新するシステム。
  • ActionUpdateSystem-アクションを更新するシステム。
  • SoundUpdateSystem-サウンド、その位置と方向を更新するシステム。
  • UpdateSystem-カスタムユーザー更新を呼び出すシステム。
  • StaticOcclusionSystem-静的オクルージョンアプリケーションシステム。
  • StaticOcclusionBuildSystem-静的なオクルージョン構築システム。
  • SpeedTreeUpdateSystem-スピードツリーツリーのシステムを更新します。
  • WindSystem-風計算システム。
  • WaveSystem-ジャバリングから振動を計算するためのシステム。
  • FolliageSystem-景観上の植生を計算するためのシステム。


私たちが達成した最も重要な結果は、異質なものを担当するコードの高度な分解です。 これで、TransformSystem :: Process関数で、変換に関連するすべてのコードが明確にローカライズされました。 彼はとてもシンプルです。 いくつかのコアに簡単に分解できます。 そして最も重要なことは、変換のシステムに論理的な変更を加えることにより、別のシステムで何かを壊すことは困難です。



ほとんどすべてのシステムで、コードは次のとおりです。 



 for (  ) { //    //      //     }


システムは、オブジェクトの処理方法によって分類できます。



  • システム内にあるすべてのオブジェクトの処理が必要です。
    • 物理学
    • 衝突


  • マークされたオブジェクトのみを処理する必要があります。
    • 変換システム
    • アクションシステム
    • 音声処理システム
    • 粒子処理システム


  • 特別に最適化されたデータ構造を使用します。
    • 静的閉塞システム


このアプローチを使用すると、オブジェクトをいくつかのコアに処理するのが非常に簡単であるという事実に加えて、通常のポリモーフィズムパラダイムではやや難しいことを非常に簡単に実行できます。 たとえば、フレームごとにすべてのlodスイッチングを簡単に実行することはできません。 大規模なオープンワールドに非常に多くの宿泊オブジェクトが存在する場合、各フレームを、たとえばオブジェクトの3分の1に処理することができます。 ただし、これは他のシステムには影響しません。



まとめ



  • コンポーネントアプローチにより物事がより独立し、それらを個別に解き、最適化することができたため、FPSが大幅に増加しました。
  • アーキテクチャがよりシンプルになり、理解しやすくなりました。
  • 近隣のシステムをほとんど損なうことなく、エンジンの拡張が容易になりました。
  • 「lodで何かをして、スイッチを壊した」シリーズからのバグは少なく、その逆もあります。
  • これをすべていくつかのコアに並列化する機会がありました。
  • 現時点では、すべてのシステムが利用可能なすべてのコアで実行されるようにするために、すでに取り組んでいます。
