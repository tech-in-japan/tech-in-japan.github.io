郵便配達員



アプリケーションの主な目的は、APIへのリクエストを含むコレクションを作成することです。 コレクションを開くと、開発者またはテスターはサービスの作業を簡単に理解できるようになります。 さらに、Postmanを使用すると、APIデザインを設計し、それに基づいてMockサーバーを作成できます。 開発者はスタブの作成に時間を費やす必要がなくなりました。 サーバーとクライアントの実装は同時に開始できます。 テスターは、Postmanから直接テストを作成し、自動テストを実行できます。 また、コレクションの説明を自動的に文書化するツールは、さらに別の「便利な機能」の時間を節約します。 管理者向けの機能もあります。作成者は、監視サービス用のコレクションを作成する機能を提供しています。







はじめに





1-コレクション、2-フォルダー、3-要求







Postmanが動作する主な概念は、最上位のコレクションと最下位のリクエストです。 すべての作業はコレクションから始まり、クエリを使用してAPIを記述することになります。 順番にすべてを詳しく見ていきましょう。







コレクション



コレクションは、新しいAPIの出発点です。 コレクションをプロジェクトファイルとして表示できます。 コレクションは、関連するすべてのクエリを結合します。 通常、APIは1つのコレクションで記述されますが、必要に応じて、別の制限はありません。 コレクションには独自のスクリプトと変数を含めることができますが、これについては後で説明します。







フォルダー



フォルダー-要求をコレクション内の1つのグループに結合するために使用されます。 たとえば、APIの最初のバージョンである「v1」用のフォルダーを作成し、実行されたアクションの意味に応じてリクエストをグループ化します。「Order＆Checkout」、「User profile」など。すべてはあなたの想像力とニーズによってのみ制限されます。 コレクションのようなフォルダーには、変数ではなく独自のスクリプトを含めることができます。







リクエスト



リクエストはコレクションの主要コンポーネントであり、すべてが開始された理由です。 要求はコンストラクターで作成されます。 クエリデザイナーは、作業する必要があるメインスペースです。 Postmanは、すべての標準HTTPメソッドを使用して要求を実行できます。すべての要求パラメーターは管理下にあります。 必要なヘッダー、Cookie、およびリクエスト本文を簡単に変更または追加できます。 リクエストには独自のスクリプトがあります。 リクエストパラメータの中の「Pre-request Script」および「Tests」タブに注意してください。 リクエストの前後にスクリプトを追加できます。 これらの2つの機能により、Postmanは開発とテストに役立つ強力なツールになります。









1-要求のあるタブ、2-URLとメソッド、3-要求パラメーター、4-応答パラメーター







スクリプト



「Postman Sandbox」は、「Pre-request Script」および「Tests」スクリプトを記述するときに使用できるJavaScriptランタイムです。 「事前要求スクリプト」は、要求の前に必要な操作を実行するために使用されます。たとえば、別のシステムに要求を行い、その実行結果をメイン要求で使用できます。 「テスト」は、テストの作成、結果の確認、および必要に応じて変数に保存するために使用されます。









リクエストのシーケンス（公式ドキュメントから）







リクエストレベルのスクリプトに加えて、フォルダーレベル、さらにはコレクションレベルでもスクリプトを作成できます。 事前要求スクリプトおよびテストとも呼ばれますが、違いは、フォルダー内の各要求の前または後に、またはご想像のとおりコレクション全体で実行されることです。









フォルダーおよびコレクションのスクリプトを使用したクエリ実行のシーケンス（公式ドキュメントから）







変数



Postmanには、変数用のスペースとスコープがいくつかあります。







  • グローバル変数
  • コレクション変数
  • 環境変数
  • ローカル変数
  • データ層変数


グローバル変数および環境変数は、プログラムの右上隅にある歯車をクリックして作成できます。 コレクションとは別に存在します。 コレクションレベルの変数は、コレクションパラメーターの編集時に直接作成され、ローカル変数は実行可能スクリプトから作成されます。 データレベルの変数もありますが、それらはランナーからのみ利用可能です。これについては後で説明します。









可変スペースの優先度（公式ドキュメントから）







Postmanの変数の特性は、置換のプレースホルダーとして中括弧を使用して、クエリコンストラクター、URL、POSTパラメーター、Cookie、どこにでも挿入できることです。









{{domain}}および{{slack_incoming_webhook}}-DNS Checker環境変数は、クエリの実行中に値に置き換えられます







変数はスクリプトからも使用できますが、pm組み込みライブラリの標準メソッドを呼び出すと、変数を取得できます。 







//    pm.globals.get("variable_key"); //     pm.environment.get("variable_key"); //        pm.variables.get("variable_key");


コレクションランナー



選択したコレクションまたはフォルダーからのすべての要求をテストして完了するように設計されています。 起動時に、反復回数、フォルダーまたはコレクションが起動される回数、環境、および変数付きの追加ファイルを指定できます。 コレクションおよびフォルダ内の場所に応じて、クエリが順番に実行されることに注意してください。 実行順序は、組み込みコマンドを使用して変更できます。 







 //      "Create order", postman.setNextRequest('Create order');


すべてのリクエストが完了すると、「テスト」スクリプトからの成功したチェックと失敗したチェックの数を示すレポートが生成されます。









コレクションランナー







コンソール



コンソールを使用してスクリプトをデバッグし、リクエストに関する追加情報を表示します。 コンソールは、単一のリクエストの起動中と、Runnerを介したリクエストパッケージの起動中の両方で機能します。 それを開くには、メインアプリケーション画面の左下隅にあるコンソールアイコンを見つけます。

新しいリクエストを作成し、URLとしてhttps://postman-echo.com/get?userId=777を指定し、GETメソッドをそのままにします。 以下、簡単にするために、Postman開発者が親切に提供するエコーサーバーを使用します。 リクエストを「Habra」コレクションに保存し、「Get user id」という名前を付けると、サーバーからのユーザーIDの受信をシミュレートします。 [テスト]タブに移動して、次のコードを記述します。

ステップ3



2つのテストスクリプトの形式とステータスチェックが同じであることに気付くかもしれません。

行き過ぎるまで、これを修正し、これらのテストをコレクションレベルに転送しましょう。 コレクションの編集を開き、形式とステータスチェックを[テスト]タブに転送すると、リクエストからそれらを削除できます。 これで、コレクション内の各リクエストの「テスト」スクリプトの前にこれらのチェックが自動的に呼び出されます。 したがって、このチェックを各リクエストにコピーする必要がなくなりました。

ステップ5



3つのクエリといくつかのテストから抜け出しました。 完全にテストするときが来ました。 しかしその前に、最初のリクエストに少し変更を加えましょう。URLの「777」値をプレースホルダー「{{newUserId}}」に置き換え、インターフェースを介してHabra.Env環境に値「777」の「newUserId」変数を追加します。 。 コンピューターにusers.jsonファイルを作成し、次のjson配列をそこに配置します。 







 [ {"newUserId": 100}, {"newUserId": 200}, {"newUserId": 300}, {"newUserId": 50000} ]


ランナーを実行します。 「コレクションフォルダ」には、「Habra」を選択します。 「環境」には「Habra.Env」を配置します。 反復回数は0です。[データ]フィールドで、users.jsonファイルを選択します。 このファイルから、「コレクションランナー」は4回の反復を実行する必要があることを理解し、各反復で変数「newUserId」の値をファイル内の配列の値で置き換えます。 配列内の要素の数よりも大きい反復回数を指定すると、データ配列の最後の値で以降のすべての反復が発生します。 この場合、4回の反復後、「newUserId」の値は常に50000になります。「データ」ファイルは、さまざまなnewUserIdを使用して、ユーザーに関係なくすべてが安定するようにします。







「Run Habra」をクリックしてテストを実行します。 おめでとうございます。コレクションの作成に成功し、自動テストの助けを借りて即興の方法をテストしました！

おわりに



プロジェクトでPostmanを使用するようになった主な利点：







  • 可視性-すべてのリクエストは常に手元にあり、開発中の手動テストが容易になります
  • クイックスタート-プログラマーであろうとテスターであろうと、チーム内の新しいプレーヤーをすばやく簡単に参加させる
  • テスト-クエリのテストを作成し、パズル、さまざまなオプション、アプリケーションライフの方法など、それらからすばやく作成する機能
  • CIサポート-newmanを使用してCIにテストを統合する機能（これは別の記事になります）


マイナスには：







  • コレクションの編集はPostmanに関連付けられています。つまり、プロジェクトリポジトリにインポート、編集、およびエクスポートする必要がある変更について
  • コレクションは1つの大きなjsonファイルにあるため、変更のレビューはほとんど不可能です


各プロジェクトには、テストと開発に関する独自のアプローチと見解があります。 私たちは短所よりも長所を見たので、この強力なツールをテストとプロジェクトのAPIリクエストの記述の両方に使用することにしました。 そして、私たち自身の経験から、Postmanが提供するテストツールを使用しなくても、要求の説明をコレクションの形で持っていると非常に役立つと言えます。 Postman Collectionsは、APIのライブの対話型ドキュメントであり、多くの時間を節約し、開発、デバッグ、およびテストを時々高速化します。







PS：

Postmanの主な機能は無料で、毎日のニーズを満たしていますが、無料版にはいくつかの制限があるツールがあります。







  • Postman API-アカウントデータにアクセスし、APIを介してプログラム、コレクション、リクエスト、その他の情報を直接追加、削除、更新できます。 彼に月に1000回連絡することができます。
  • モックサーバー-APIのスタブをPostman開発者サーバーに展開できます。 1か月あたり1000件のリクエストが可能です。
  • API監視-Postman開発者サーバーからスケジュールに従って監視および実行するコレクションを作成できます。 監視システムは、1か月あたり1000件以下のリクエストを行います
  • Apiドキュメント-Postmanアプリケーションのサーバー上で自動生成されたドキュメントを公開できます。 1か月あたり1,000回の視聴を制限する
