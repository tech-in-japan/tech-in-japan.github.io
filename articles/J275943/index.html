はじめに



コードの理解を深め、順次複雑にするために、設計を2つの段階に分割します。プリミティブ（最小限実行可能）と従来のアーキテクチャです。 プリミティブでは、最小限のコードとファイルを管理してから、このコードを改善します。

すべてのソースコードはgithubにあります。 リポジトリのブランチは、記事のステップに対応しています。 ステップ1単純なアーキテクチャ -最初のステップ、 ステップ2複雑なアーキテクチャ-2番目のステップ。

たとえば、Github APIを使用して、特定のユーザーのリポジトリのリストを取得してみましょう。



このアプリケーションでは、Rxを使用しているため、この記事を理解するには、このテクノロジーの一般的な考え方が必要です。GrokaiRxJavaの一連の出版物を読むことをお勧めします。



ステップ1.シンプルなアーキテクチャ



階層化、MVP

アーキテクチャを設計するときは、MVPパターンを順守します。 詳細はこちらをご覧ください：

https://ru.wikipedia.org/wiki/Model-View-Presenter

http://habrahabr.ru/post/131446/



プログラム全体を3つの主要な層に分割します。

モデル -ここでデータを取得して保存します。 出力は観測可能です。

プレゼンター -このレイヤーには、すべてのアプリケーションロジックが格納されます。 Observableを取得し、サブスクライブし、結果をビューに渡します。

ビュー -表示レイヤー。すべてのビュー要素、アクティベーション、フラグメントなどが含まれます。



条件付きスキーム：




モデル



データレイヤーは、Observable <List <Repo >>を提供する必要があります。インターフェイスを作成しましょう。 



public interface Model { Observable<List<Repo>> getRepoList(String name); }


後付け


ネットワークでの作業を簡素化するために、Retrofitを使用します。 RetrofitはREST APIを操作するためのライブラリです。ネットワークでのすべての作業を処理します。インターフェイスと注釈を使用してリクエストを記述することしかできません。



レトロフィット2
Runetのレトロフィットに関する資料は多数あります（ http://www.pvsm.ru/android/58484、http://tttzof351.blogspot.ru/2014/01/java-retrofit.html ）。

2番目のバージョンと最初のバージョンの主な違いは、同期メソッドと非同期メソッドの違いがなくなったことです。 ここでCall <Data>を取得します。これは、同期要求の場合はexecute（）を、非同期要求の場合はexecute（コールバック）を呼び出すことができます。 また、リクエストをキャンセルする待望の機会がありました：call.cancel（）。 以前と同様に、特別なプラグインの助けを借りて、Observable <Data>を取得できます





リポジトリデータを取得するためのインターフェイス： 



 public interface ApiInterface { @GET("users/{user}/repos") Observable<List<Repo>> getRepositories(@Path("user") String user); }


モデルの実装 
 public class ModelImpl implements Model { ApiInterface apiInterface = ApiModule.getApiInterface(); @Override public Observable<List<Repo>> getRepoList(String name) { return apiInterface.getRepositories(name) .subscribeOn(Schedulers.io()) .observeOn(AndroidSchedulers.mainThread()); } }

レトロフィット（およびその中のGSON）は、POJO（Plain Old Java Object）で動作します。 これは、次の形式のJSONからオブジェクトを取得することを意味します。 



 { "id":3, "name":"Andrey", "phone":"511 55 55" }


GSONが値を書き込むUserクラスが必要になります。

そのようなクラスを自分の手で生成する必要は当然ありません;これには、例えばwww.jsonschema2pojo.orgのような特別なジェネレーターがあります。

レトロラムダ


ラムダなしでRxを使用するのは苦痛で、すぐに疲れるたびに匿名クラスを書く必要があります。 AndroidはJava 8とラムダをサポートしていませんが、Retrolambda（ https://github.com/evant/gradle-retrolambda ）が助けになります。 ラムダ式の詳細： http : //habrahabr.ru/post/224593/



レイヤーごとに異なるデータモデル。


ご覧のとおり、3つのレイヤーすべてで同じRepoデータオブジェクトを使用しています。 このアプローチは単純なアプリケーションには適していますが、実際には、APIの変更、オブジェクトを変更する必要性、または何か他のものに出くわすことがあります。 複数の人がプロジェクトに取り組んでいる場合、別のレイヤーの利益のためにクラスを変更するリスクがあります。



そのため、多くの場合、1つのレイヤー= 1つのデータ形式というアプローチが適用されます。 モデルの一部のフィールドが変更されても、これはビューレイヤーに影響しません。 Presenterレイヤーに変更を加えることはできますが、Viewでは厳密に定義されたオブジェクト（クラス）を提供します。 このため、データモデルからのレイヤーの独立性が達成され、各レイヤーには独自のモデルがあります。 モデルを変更するときは、マッパーを書き換えて、レイヤー自体には触れないようにする必要があります。 これはコントラクトプログラミングに似ています。どのオブジェクトがレイヤーに到達し、どのオブジェクトをさらに提供する必要があるかを正確に把握しているため、予測不可能な結果から自分自身と同僚を保護します。



この例では、DTO-データ転送オブジェクト（JSONオブジェクトを完全にコピー）とビューオブジェクト（表示用に調整されたオブジェクト）の2種類のデータで十分です。 より複雑なアプリケーションがある場合、ビジネスオブジェクト（ビジネスプロセス用）またはデータベースオブジェクト（データベースに複雑なオブジェクトを保存するため）が必要になる場合があります

 public class RepoBranchesMapper implements Func1<List<BranchDTO>, List<Branch>> { @Override public List<Branch> call(List<BranchDTO> branchDTOs) { List<Branch> branches = Observable.from(branchDTOs) .map(branchDTO -> new Branch(branchDTO.getName())) .toList() .toBlocking() .first(); return branches; } }

発表者



Presenterレイヤーでは、共通のクラスが必要です。 プレゼンターは、さまざまな機能を実行できます。ダウンロードショーの簡単なプレゼンター、アイテムをロードする必要があるリスト、サイト上のオブジェクトをリクエストするマップ、および他の多くのエンティティがあります。 しかし、それらはすべて、メモリリークを回避するためにObservableからサブスクライブを解除する必要性によって統一されています。 残りは発表者のタイプによって異なります。



複数のObservableを使用する場合、onStopで一度にすべてのサブスクリプションを解除する必要があります。 これを行うには、CompositeSubscriptionを使用できます。すべてのサブスクリプションをそこに追加し、コマンドでサブスクライブを解除します。



また、プレゼンターに状態の保存を追加します。 これを行うには、onCreate（BundledInstanceState）およびonSaveInstanceState（Bundle outState）メソッドを作成して実装します。 DTOをVOに変換するには、マッパーを使用します。



コード例 
 public void onSearchButtonClick() { String name = view.getUserName(); if (TextUtils.isEmpty(name)) return; Subscription subscription = dataRepository.getRepoList(name) .map(repoListMapper) .subscribe(new Observer<List<Repository>>() { @Override public void onCompleted() { } @Override public void onError(Throwable e) { view.showError(e.getMessage()); } @Override public void onNext(List<Repository> list) { if (list != null && !list.isEmpty()) { repoList = list; view.showRepoList(list); } else { view.showEmptyList(); } } }); addSubscription(subscription); } public void onCreate(Bundle savedInstanceState) { if (savedInstanceState != null) { repoList = (List<Repository>) savedInstanceState.getSerializable(BUNDLE_REPO_LIST_KEY); if (!isRepoListEmpty()) { view.showRepoList(repoList); } } } private boolean isRepoListEmpty() { return repoList == null || repoList.isEmpty(); } public void onSaveInstanceState(Bundle outState) { if (!isRepoListEmpty()) { outState.putSerializable(BUNDLE_REPO_LIST_KEY, new ArrayList<>(repoList)); } }
