Docker for WindowsでPHPでプロジェクトを作成する方法

この記事は、Docker for Windowsのユーザー向けのシンプルで明確な指示とヒントのセットです。 この記事は、PHP開発者がプロ​​ジェクトを迅速に選択するのに役立ちます。 問題とその解決策について説明します。 この記事は、WindowsでのDockerの問題を深く掘り下げるための無限の時間的リソースがない人に役立ちます。 著者が以前に同様の記事に出くわした場合、著者は無限に感謝し、著者は多くの時間と労力を節約できただろう。 テキストにはエラーや不正確さが含まれている場合があります。

最初のステップは、システムに新しい管理者を追加することです



これは重要なステップです。スキップするか従わない場合、おそらく次の指示はまったく意味をなさないでしょう。 Windows + Rを実行すると、lusrmgr.msc、「ローカルユーザーとグループ」が開きます。 さらなるユーザー、コンテキストメニュー、その中の「新規ユーザー...」。 必要なパスワードで新しいユーザー（dockerhostなど）を追加します。 パスワードが必要です！ 管理者グループにメンバーシップを追加します。 他のグループを追加しないでください。







次に、Docker設定の[共有ドライバー]で、必要なドライブを選択し、新しいユーザーのデータ（ユーザー名とパスワード）を入力します。 設定を保存します。 以前に作業中のユーザーのユーザー名とパスワードを入力した場合は、新しいユーザーのデータを入力します。 予期しない場所で問題を修正する場合は、仕事用アカウントを使用し、記事をこれ以上読みません。









どの構成から始めますか？



インターネット上には、PHP Webサーバー用のあらゆる種類の構成docker-compose.ymlがあります。 それらのいくつかは、Windowsで起動しません。 docker-compose.ymlジェネレーターを見ることをお勧めします。 ジェネレーターから構成をすぐに取得したので、後で構成を編集しました。 最終結果はgithubにアップロードされました。 ジェネレーターオプションは、いくつかの点で不適切です。 問題の説明とその解決策を以下に示します。









永続的なデータベースストレージのセットアップ



問題：データベースファイルをローカルディスクに保存することはできません。 これは、Windowsでの公理としてとらえ、データがコンテナの外部に保存されるように、許容可能な解決策を見つけようとします。 Windowsの場合、これはボリュームという名前です。 ほんの数行でこの問題を解決できます。 







postgres: volumes: - db:/var/lib/postgresql/data volumes: db:


名前付きボリュームは、MobiLinuxVMドッカー仮想マシンの/ var / lib / docker / volumes /フォルダーにあります。 これらのファイルへの直接アクセスはなく、中間コンテナーを介してのみアクセスできます。 このフォルダーをWindowsで使用可能にする方法を決定するのかわかりません。 名前付きボリュームを管理するには、docker volumeコマンドを使用します。 たとえば、未使用のdockerボリュームを削除してボリュームを整理します。







名前付きボリューム内のファイルの権利とユーザーを増やす必要はありません。Dockerがすべてを処理してくれます。 一部のマニュアルでは、Dockerに永続的なデータベースストレージを設定すると、権利の割り当てに合わせて踊ります。 Windowsでは、名前付きボリュームがないと成功しません。 まあ、最初に起動するかもしれませんが、再起動すると曲がります。 名前付きボリュームを備えた永続ストレージが機能し始めたときのことを想像することさえできません。









SSHキーの追加



また、名前付きボリュームを介してSShキーを追加します。 その理由は、秘密鍵には特別な権利が必要であり、通常のWindowsボリュームではこの多重度が与えられないからです。必要な部分をdocker-compose.ymlから引用します 







 services: php: volumes: - ssh:/root/.ssh volumes: ssh:


これが機能するには、指定されたボリュームのキーをコピーし、秘密キーへの権限を変更し、接続をテストする必要があります。 これらすべてのアクションは、単一のbatファイルとして発行できます。 







 docker run -v first_ssh:/root/.ssh --name helper busybox true docker cp ./.ssh/. helper:/root/.ssh/ docker rm helper docker-compose exec php ls /root/.ssh docker-compose exec php chmod 600 /root/.ssh/id_rsa docker-compose exec php ssh git@github.com


もちろん、スクリプトでは、first_sshの代わりにボリューム名を置き換えます。 ボリュームという名前は、ドッカーによってCOMPOSE_PROJECT NAME + + sshとして形成されることを思い出させてください。 ./.sshの代わりに、キーへのパスを入力します（または、このスクリプトを実行する場所にキーを含むフォルダーを一時的にコピーします）。 キーをgithubに追加した場合、最後にgithubが挨拶します。 スクリプトは1回限りであり、コンテナの最初の起動が成功した直後に実行する必要があります（docker-compose up -d）。 名前付きボリュームを削除した場合を除き、繰り返し起動しても意味がありません。









コンテナ間のネットワーク



1つのdocker-compose.ymlファイルで説明されているコンテナ間を接続するには、何も必要ありません。 ホスト名としてサービス名またはコンテナ名を指定するだけで十分です。 ポート番号を変更する必要はありません-デフォルトのポートが機能します。 これは、別のdocker-compose.ymlからコンテナーにリクエストを送受信する必要がない場合に完了できます。 これにも問題はありません。ネットワークの名前とネットワーク上のコンテナを指定するだけです。 目的のセクションdocker-compose.ymlを引用します 







 services: php: networks: # this network - default # external network - second_default external_links: - ${EXTERNAL_NGINX} networks: default: driver: bridge second_default: external: true


グローバルネットワークセクションでは原則として変数の使用が許可されていないため、ネットワーク名second_defaultは環境変数に配置されないことに注意してください。 可能な場合、phpセクションで変数を使用することは意味がありません。 たとえば、そこに外部ネットワークからコンテナ名を入れます。 .envファイルでEXTERNAL_NGINX = second_nginxを設定した場合、PHPコードでは、ホスト名second_nginxを使用してhttpリクエストを作成するだけで十分です。 ポートはまだ80であり、特別に登録する必要はありません。 githubリポジトリに、異なるdocker-compose.ymlのコンテナー間の接続を確認するためのスクリプトを追加しました。 docker-compose exec php php get.phpを実行して、機能していることを確認するだけで十分です。







コンテナを初めて起動すると、Dockerが誓う場合があります。 エラー：ネットワークsecond_networkが外部として宣言されましたが、見つかりませんでした。 docker network create second_default



を使用してネットワークを手動で作成し、再試行してください。 実際、そうです、dockerプロンプトを使用して、手動でネットワークを作成します。









ログ設定



すべてのサービスからのログを、ローカルでアクセス可能な1つのフォルダーに入れたいです。 これは、通常のボリュームの助けを借りて行われます。 主なことは、サービス自体のロギングを有効にすることです。 phpの場合、これらはphp.iniファイル、nginx、その設定、postgresのオプションです。 phpとnginxを使用すると、すべてが簡単になります-設定に対応するファイルがあります。 postgresの場合、コマンドラインオプションを使用する必要があります（postgresql.confファイルにはまだパスがありますが、もう少し複雑になります） 







 services: postgres: command: postgres -c logging_collector=on -c log_destination=stderr -c log_directory=/logs -c client_min_messages=notice -c log_min_messages=warning -c log_min_error_statement=warning -c log_min_duration_statement=0 -c log_statement=all -c log_error_verbosity=default volumes: - ${LOGS_DIR}:/logs


カブと記事の最後にあるdocker-compose.ymlの全文。 私はまだログの設定に満足していないと言わなければなりません。 そのまま満足しながら。 ロギングを微調整する場合は、対応するサービスのドキュメントを使用できます。









.envの変数



docker-compose.ymlをほぼ普遍的な構成にするための優れたオプションは、.envファイルです。 環境変数を指定することが不可能なグローバルネットワークセクションのため、完全な汎用性は機能しません。 私のファイルの機能のうち、postgressの変数は、docker-compose.yml自体に記述する必要はありません。 PHPでは、データベース接続を使用できます 







 'dsn' => sprintf('pgsql:host=%s;dbname=%s', getenv('POSTGRES_HOST'), getenv('POSTGRES_DB')),

気づいたら、phpとnginxに自分の画像を使用しました。 これにより、テスト中の時間を節約できました。 他のコンテナイメージの使用を妨げるものは何もありません。私の構成は単なるデモです。 独自のイメージを簡単に作成できます。カブの使用済みイメージが作成されるビルドフォルダーを確認します。
