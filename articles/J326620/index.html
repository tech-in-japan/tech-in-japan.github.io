プロジェクトの基本要件：



  • 簡単な実装
  • 簡単な使用
  • 便利なカスタマイズと拡張（突然新しいアクセス許可またはビジュアルを追加する場合）


残りはうまくやろうとしました（たとえば、ビジネスロジックを変更する機能は優先事項ではありませんでしたが、考慮されました）



それでは話しましょう。 これは一般的に便利なことです。 プロジェクトをシンプルにするためには、シンプルなインターフェースを持ち、プログラマーに内部の実装をロードしないようにする必要があります。 この点で、私はAppodealからのアプローチに触発されました。 一般に、エントリポイントは1つ必要です。 つまり オブジェクトに2つの権限を設定し、要求しました。 簡単に聞こえるはずです！



すぐに左の悪魔？ 肩にささやく：「シングルトン...」。 そして、最初の日は素晴らしいソリューションに思えたので、AppDelegateで構成し、コントローラーを必要な場所に表示しました。



しかし、さらに問題がありました：



  • ダイアログボックスを念頭に置くことは非常にまれです
  • すべての許可を取得した後、あなたは間違いなくそれをあなたの記憶に保持する必要はありません
  • 異なる画面で同じ権限を要求する必要はほとんどありません


一般に、パターンにはプラスよりもマイナスが多くありました。 ここでは、プロジェクトを使用するためのオプションに基づいているという事実に注意を喚起したいと思います。 Singletonの使用は正当であると確信していますが、他の状況でもそうです。



ターニングポイントで、私の見解は、既知のAndroid開発者Aleksey Skutarenkoによって確認されました。彼はこのパターンを「疑わしい」と呼びました。 このパターンが嫌いなのか、それとも私のニーズに最適に適用できないのかは不明です。 しかし、20枚の古紙を捨てて新しいものを手に入れるという決断が下されました。 実際、マーカーも終了しました。



それから反対から行くことにした。 プロジェクトの使用方法を教えてください。 私はこれを明確に表しています： 



class ViewController: UIViewController { var permissionAssistant = SPRequestPermissionAssistant.modules.dialog.interactive.create(with: [.Camera, .PhotoLibrary, .Notification]) override func viewDidAppear(_ animated: Bool) { super.viewDidAppear(true) self.permissionAssistant.present(on: self) } }


ソリューションは自然に生まれました。メインクラスが必要であり、PermissionAssistantと呼びます。 そして、ロジックをキーブロックに分割し、便宜上、それらをワードマネージャーと組み合わせます。 そして、それは論理的に、異なるタスク-したがって、異なるクラスがそれらを担当します。





それでは、機能部分を決定しましょう。 明らかに、それらの1つが許可の要求とそれらに関する情報の取得を担当します。 PermissionsManagerと呼びます。 視覚的な部分も含まれているため、PresenterManagerを追加します（Viperの命名規則を所有します。発見者で十分な場合があります）。
