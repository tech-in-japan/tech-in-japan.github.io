Titaniumのモデルとコレクションについて知りたいことすべて

しかし、何らかの理由で彼らは尋ねることを恐れていました。



モデルはアプリケーションの非常に重要な部分です。データがなければどこにも存在しないからです。



この記事では、モバイルアプリケーションAppcelerator Titaniumの開発のためのMVCフレームワークでのモデルの使用のあらゆる側面を可能な限り詳細にカバーするようにします。



モデルへの連絡を試みていない場合は、この記事が数キロの神経を節約することを願っています。



Backbone.js



したがって、Titaniumでモデルを操作するときに最初に知っておくべきことは、それらがBackboneのモデルとコレクションに基づいているということです。 これは、まず、Backboneのドキュメントで説明されているすべてのプロパティとメソッドを使用できることを意味します。 第二に、あなたがまだそうしていなければ、最初に彼女を知ることは不必要ではありません。



念のため、用語に焦点を当てます。モデルは単一のエンティティであり、コレクションは類似したモデルのセットです。



重要：TitaniumはBackboneバージョン0.9.2を使用するため、Backboneドキュメントの一部のメソッドが実装されない場合があります。


Underscore.js



コレクションはUnderscore.jsメソッドも継承し、開発者の作業を楽にします。 たとえば、 .mapメソッド



作成



2つの同等の方法でモデルを作成できます。AppceleratorStudioを使用する方法と手動で行う方法です。

<project> / app / models /フォルダーに特別な形式のファイルを手動で作成できます（実際には、スタジオによって自動的に作成されます）。

モデル記述ファイル形式 


exports.definition = { config : { //   }, extendModel: function(Model) { _.extend(Model.prototype, { //  Backbone.Model }); return Model; }, extendCollection: function(Collection) { _.extend(Collection.prototype, { //  Backbone.Collection }); return Collection; } }




スタジオのヘルプを使用する場合は、プロジェクトのコンテキストメニューを呼び出して、「新規->合金モデル」の項目を選択するだけです。 ダイアログボックスを作成するとき、アダプタのタイプを選択するように求められます：localStorage、properties、またはsql。 アダプターのタイプについて以下に説明します。



基本的な機能のためにこのファイルに記述する必要があるのは、構成ブロックだけです。 データスキーマと同期アダプターのタイプが含まれています。 このブロックには、次のフィールドが含まれている必要があります。

  • columns ：モデルのフィールドを説明する辞書。 キーはフィールドの名前、値はタイプです。 SQLiteのような型： 文字列、varchar、int、tinyint、smallint、bigint、double、float、decimal、number、date、datetime、boolean
  • デフォルト ：列と同じディクショナリ。ただし、同期中に定義されていない場合、値はデフォルトのフィールド値
  • adapter ：このオブジェクトには2つのフィールドが含まれます。

    • type ：同期アダプターのタイプ
    • collection_name ： コレクションの名前。 モデル名と異なる場合があります。 このフィールドの値を使用して、コレクションを作成します。
    • idAttribute ：主キー


たとえば、書籍モデルを作成します。 これを行うには、<project> / app / models /フォルダーにbooks.jsファイルを作成します。

作成されたモデルの例 


 exports.definition = { config: { "columns": { "title": "String", "author": "String" //    sql,     SQLite keywords  }, "defaults": { "title": "-", "author": "-" }, "adapter": { "type": "sql", "collection_name": "books", "idAttribute" : "title", "db_file" : "db_books", //   ,    .   _alloy_ "db_name" : "books.sqlite", //      /app/assets.   ,    [_].sqlite } } }




ここには、本の名前と著者の2つのテキストフィールドを持つモデルがあります。 両方のフィールドのデフォルトは「-」です。 これまでのところ、すべてがシンプルです。



sqlアダプタータイプは、データを取得しようとするときに、組み込みアダプターを使用するモデルがデバイス上のSQLiteデータベースに接続し、booksテーブルからデータを取得しようとすることを示します。

モデル内の独自のプロパティとメソッドは、次のように追加できます。

モデル拡張の例 


 exports.definition = { config : { //   }, extendModel: function(Model) { _.extend(Model.prototype, { //  Backbone.Model customProperty: 'book', customFunction: function() { Ti.API.info(',  '); }, }); return Model; } }




コレクションにも同じことが言えます。 extendModelに独自のメソッドを追加した場合、モデルで使用できますが、コレクションは使用できません。逆の場合も同様です。



使用する



モデル/コレクションにアクセスするには、2つの方法があります。

  • グローバルシングルトンの作成
  • ローカルリンクを作成する
  • Alloyを使用している場合、XMLマークアップで指定することにより


グローバルシングルトン



この場合、モデルはどこでも、すべてのコントローラーですぐに使用できます。 すでに宣言されている場合は、別の宣言でAlloy.Models.instanceメソッドを呼び出すと、この宣言されたモデルへのリンクが返されます。 例はユーザーモデルです;それはどこでも単独である場合もあります。



コレクションについても同様です。



ローカルリンク



Alloy.createModelメソッドを使用して、ローカルリンクを作成できます。 このモデルは、作成されたコントローラー内でのみ使用可能になります。 メソッドはファクトリーであり、モデル名（ファイル名から.jsを引いたもの）とパラメーターを渡します。 そしてそれだけです。



コレクションについても同様です。



XML



<Model>マークアップ要素は、 <Alloy>の直接の子孫でなければなりません。 また、値が「model-file-name-minus-.js」のsrc属性が必要です。 次に、Alloy.Models名前空間のこのモデルのシングルトンがコントローラーで使用可能になります。

XML 


 <Alloy> <Model src="book" /> <Window> <TableView id="table" /> </Window> </Alloy>




コントローラー 


 var drama = Alloy.Models.book; drama.set('title', '  '); drama.set('author', ' ');




この要素にはインスタンス属性もあります。 trueの場合、モデルへのリンクが作成されます（ローカル、1つのコントローラー内でのみ使用可能）。 ところで、この場合、モデルはAlloy.Models名前空間ではアクセスできません。IDでアクセスする必要があります。

XML 


 <Alloy> <Model id="myBook" src="book" instance="true"/> <Window> <TableView id="table" /> </Window> </Alloy>




コントローラー 


 var drama = $.myBook; drama.set('title', '  '); drama.set('author', ' ');




コレクションについても同様です。



同期アダプター



最後に、「作成」セクションで約束されたセクションに行きました。 モデルを操作するには、データスキーマを記述するだけでは不十分です。 また、それらを何かに接続する必要があります。 この接続は同期アダプターです。

アダプタは、Backbone.syncの実装です。 最も一般的なタスクはリモートサーバーでのCRUD操作であるため（個人的には私の経験から）、デフォルトでは、Backbone.syncメソッドは、Model.urlRootおよびCollection.urlプロパティで指定したアドレスに対してRESTful JSONリクエストを行います。 これは公式ドキュメントです。 実際には、モデル/コレクションの感覚をつかむために、restapiアダプターを使用すると非常に便利です。これについては後で説明します。 それまでの間、公式ドキュメントに戻ります。



アダプタには次の4つのタイプがあります。

  • sql
  • 物性
  • localStorage（合金1.5.0以降では使用されないため、考慮しません）
  • あなたのタイプ




SQL



idAttributeキーがadapterセクションのconfigセクションで定義されていない場合、AlloyはテーブルにAlloy_idフィールドを生成し、それを主キーとして使用します。



db_fileおよびdb_name構成フィールド（「作成」セクション）に注意してください。 ここで重要です。

他のタイプとは異なり、このメソッドでは、 Backbone.Collection.fetchは1行のSQLクエリを取得できます。

.fetch（）のクエリ文字列 
 var library = Alloy.createCollection('book'); //   -  collection_name   var table = library.config.adapter.collection_name; //   library.fetch({query:'SELECT * from ' + table + ' where author="' + searchAuthor + '"'}); // prepared statement library.fetch({query: { statement: 'SELECT * from ' + table + ' where author = ?', params: [searchAuthor] }});




つまり、単純なリクエストが必要な場合は、パラメータにクエリキーを指定してオブジェクトを渡し、複雑なリクエストが必要な場合は、リクエストとパラメータの両方を渡します。



ところで、まだidキーを渡すことができます。 この場合、アダプター自体はWHERE id =？..制約を使用することを理解します。

id 
 myModel.fetch({id: 123}); //   ,  myModel.fetch({query: 'select * from ... where id = ' + 123 });






物性



データベースがSQLiteでない場合、独自のアダプターを作成し、データを使用して何でもできます。 これを行うには、app / assets / Alloy / syncまたはapp / lib / Alloy / syncディレクトリにアダプターファイル（myAdapter.jsなど）を作成し、adapter.typeのモデル構成でこのファイルの名前、つまり「myAdapter」を指定します。

実際、このファイルは3つの関数をエクスポートする必要があります。

  • module.exports.beforeModelCreate（config） （オプション）-名前が示すとおり、メソッドはモデルを作成する前に実行されます。 パラメーターは、このモデルの構成を渡しました。 ここでは、たとえば、リクエストアドレスにbaseUrlを追加できます。 変更された構成オブジェクトを返します
  • module.exports.afterModelCreate （オプション） -2つのパラメーターを受け入れます：Backbone.Modelを作成したばかりの新しいクラスとモデルファイルの名前
  • module.exports.sync -Backbone.syncメソッドの実装。 データを返す必要があります




レスタピ



リモートサーバーからデータを受信する必要があり、独自のアダプターを作成したくない場合は、組み込みのアダプターの数には含まれていないが、その仕事（RESTful JSONリクエスト）を実行する素晴らしいrestapiアダプターがあります。

最低限必要な設定-構成のURLフィールド：

Restapiアダプターを使用したモデルの例 
 exports.definition = { config: { "URL": "http://example.com/api/modelname", //"debug": 1, "adapter": { "type": "restapi", "collection_name": "MyCollection", "idAttribute": "id" } }, extendModel: function(Model) { _.extend(Model.prototype, {}); return Model; }, extendCollection: function(Collection) { _.extend(Collection.prototype, {}); return Collection; } }




もちろん、これらはすべて利用可能な設定ではありません;完全なリストについては、ドキュメントを参照してください。 このアダプターで他に重要なことは、そのフェッチメソッドが3つの可能なフィールドを持つ辞書を受け入れることです。

  • データ要求パラメーター
  • 成功したリクエストの成功コールバック
  • エラーのエラーコールバック
