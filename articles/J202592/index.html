クロスプラットフォームモバイルアプリケーションの作成の歴史

モバイルプラットフォーム市場での激しい競争により、1つのオペレーティングシステムに焦点を絞ったアプリケーションのリリースが許可されなくなり、開発者は難しい選択を余儀なくされています。 各OSのネイティブアプリケーションの開発（それぞれのネイティブプログラミング言語）、HTML5を使用したモバイルデバイス用のWebサイトとしての開発、またはクロスプラットフォーム開発用のフレームワークを使用したアプリケーションの作成から選択する必要があります。



この記事では、興味深い統計を提供するだけでなく、空港向けのクロスプラットフォームモバイルソリューションを作成するストーリーを伝えたいと思います。



最初のクライアントはキエフボリスピリ国際空港でした。 開発された製品の特徴は、顧客の企業スタイルに適合し、店舗での公開に成功したことです。 キエフボルィースピリ空港がウクライナの近代的なモバイルアプリケーションの最初の所有者になったことは興味深いことです。 それでも、最初から始める価値はあります。



旅客航空輸送の業界は常に進化しており、そのおかげで乗客の快適性を高めるために最新のテクノロジーが導入されています。 世界中で、空港でのモバイルアプリケーションの可用性はすでに当然の問題になっていますが、奇妙なことに、このプラクティスはウクライナでは一般的ではありません。 そのため、空港情報システムと簡単にやり取りし、顧客の要件に適応できるボックス化されたソリューションを作成するというアイデアがありました。



システムの主な機能には、オンラインフライトディスプレイ、目的のフライトに変更が加えられた場合のユーザーへのプッシュ通知、および他の空港サービス（会議室、VIPサービス、タクシー、ホテル、ショップなど）の普及が含まれます。



ソリューションのクライアント部分の開発



C＃でアプリケーションを作成する従業員の素晴らしい経験を考慮して、 Xamarinフレームワークを使用してアプリケーションのクライアント部分を開発するアプローチを選択しました。 すべての作業はVisual Studioで行われました。 Windows PhoneおよびWindows 8の場合、Visual Studioでの開発がデフォルトのツールであり、AndroidおよびiOSの場合、この機能はXamarinのおかげで利用可能になりました（これにはビジネスライセンスが必要です）



システムに設定された主なタスクの1つは、オンラインで作業するユーザーエクスペリエンスを作成することでした。 ユーザーは、表示されたデータが定期的に更新されているとは思わずに完全に信頼する必要があります。 さらに、システムを設計するときに、すべてのビジネスロジックが実装されるすべてのオペレーティングシステムに対して単一のクライアント側カーネルを作成することが決定されました。



クライアントビジネスロジックの単一の実装とは異なり、プラットフォームごとに、アプリケーションの外観を担当する独自のグラフィカルインターフェイス（コード）を作成する必要がありました。 各OS固有のグラフィック効果を使用するために、プラットフォームごとに個別にこのようなインターフェイスを作成しました。



Xamarinでは、iOSおよびAndroidのユーザーインターフェイスとその要素に標準の開発ツールを使用できます。



iOS用のソリューションを開発するには、とにかくMacが必要であることに注意してください。 私たちの場合、Macには2つの理由が必要でした。まず、アプリケーションをデバッグするには、Macでのみ利用可能なiPhone / iPadエミュレーターが必要です。 次に、このオペレーティングシステムはプロジェクトの自動アセンブリに必要でした。



ソリューションのサーバー側の開発



アプリケーションをユニバーサル化する必要があることを考えると、設計時に最適なソリューションはサーバーパーツをクラウドに配置することになるという結論に達しました。 クラウドプラットフォームとして、MicrosoftのソリューションであるWindows Azureが選ばれました。



彼と一緒に仕事をするのはとても便利で簡単でした。 Azureを使用することのプラス面のうち、次の点を強調します。

1）クラウドでのシステムの便利な展開

Windows Azureにアプリケーションをデプロイするには、Windows AzureポータルまたはService Management API（SMAPI）を使用する2つの方法があります。



2）テスト展開のための便利なメカニズム

テスト（ステージング）と戦闘（プロダクション）の2つの展開セルがあります。 テスト展開では、運用環境に展開する前にWindows Azure環境でアプリケーションをテストできます。 同様に、ステージングとプロダクションの切り替えはワンクリックで行われます。



Windows Azureでは、支払い額は使用したプロセッサ時間に依存するため、ステージング展開を削除してコストを最適化する必要があります。 ポータルで「停止」ボタンをクリックするだけで、サービスは利用できなくなりますが、同時にプロセッサ時間が消費されます。



私たちの場合、節約額が1か月あたり100ドルを超えたことは注目に値します。



軟膏のハエとして、MicrosoftはWindows Azureではログを操作するのは不便だと言います。 もちろん、テーブルストレージを操作するのは非常に不便なので、ログには有料のソリューションを使用できます。



サーバーシステム機能の機能



空港システムからアプリケーションにデータを転送するために、おそらく最も便利で安全な方法ではなく、顧客にとって最速で低コストのftpサーバーを介したxmlファイルの転送を選択しました。 たとえば、空港は、指定された頻度でxmlファイルを生成し、旅客航空輸送に関連するすべてのフライトに関する情報を含めて、ftpサーバーにアップロードします。



長所：

  • シンプルさと実装の速度
  • 空港からのロジックの量を減らし、データエラーの可能性を減らします。
  • 承認の数と期間の削減。


短所：

  • 空港からのすべての混乱を防ぐことができない
  • FTPセキュリティ
  • Windows AzureにはFTPなどのサービスがないため、ftpサーバーはお客様の
  • FTPからデータを受信するシステムには動的IPアドレスがあるため、情報へのアクセスはAzureクラウドのアドレスプールまたはサービスのドメイン名によって制限する必要があります。


また、プッシュ通知をユーザーに送信することにも注意する価値があります。 機能の中で、次の名前を付ける必要があります。

  • プッシュ通知サービスはクライアントへの配信を保証しないため、インターネットチャネルが不良な場合、メッセージが失われる可能性があります
  • 通知の配信順序は送信順序に依存しません
  • Wi-Fiネットワークでは、プッシュ通知を配信するために特別なポートを開く必要があります（セキュリティポリシーが「すべてを禁止し、正しいもののみを許可する」という原則に基づいている場合）。


出版物の特徴



キエフボルィースピリ空港は、すべての一般的なスマートフォン（iOS、Android、Windows Phone）、およびWindows 8を実行するタブレットでアプリケーションを利用できるようにしたいと考えていました。



Windows Phone


以前は、Microsoftストアでアプリケーションを公開した経験はありませんでしたが、公開は成功しました。 わずか数日後、アプリケーションは既にストアで利用可能になりました。



iOS


AppStoreへの公開もスムーズに行われましたが、すべての手順を完了するには1週間かかりました。



Android


GooglePlayでの公開は最も簡単で最速でした。 すべての記述属性を入力し、パッケージ（apkファイル）を添付した後、アプリケーションには「発行済み」のステータスが割り当てられました。 「新規」セクションおよびストアの検索で、数時間以内に利用可能になりました。 ほとんどの場合、遅延はストア自体のデータの更新によるものです。



Windows 8


Windows Phoneストアでの公開のシンプルさに触発され、Windows 8での作業でも同じことが期待されていました。ただし、公開する際にはいくつかの困難がありました。



文言で受け取った最初のシステム障害：

  • アプリケーションが要件4.1を満たしていません。 （アプリケーションにネットワーク機能がある場合は、プライバシーに関する声明を含める必要があります）。 詳細


結局のところ、インターネットにアクセスできるアプリケーションの前提条件は、ストアだけでなく、アプリケーション設定にもプライバシーポリシーが存在することです。



  • アプリケーションが6.8の要件を満たしていません。 （サポートされている各言語のローカライズされたアプリケーションのスクリーンショットを提供する必要があります）。 詳細


スクリーンショットはすべての必要な言語で撮影されたため、このようなメッセージは予想外でした。 それでも、計算中にスクリーンショットの1つが必要なサイズより1ピクセル小さくなり、その結果グラフィックエディターで変更されたと想定しています。 再公開時に、エラーが修正されました。



2番目の拒否は、次の文言で受信されました。

  • アプリケーションは要件2.3を満たしていません（アプリケーションは、独自の説明、タイル、通知、アプリケーションパネルを使用して、または端から指をすばやくスワイプした後に広告を表示しないでください）。 詳細


アプリケーションには広告がまったくなかったため、この結果は非常に驚きました。 変更なしで、再認証のために送信しました。 その結果、そのアプリケーションは合格しました。



選択したアプローチの長所と短所



この世界の他のすべてと同様に、選択されたアプローチには長所と短所の両方があります。 利点の中で、特に次の点を強調したいと思います。

  1. Xamarinにクロスプラットフォームカーネルを実装する決定により、アプリケーションのビジネスロジックを開発およびテストする時間が大幅に短縮されました。
  2. サーバーパーツをWindows Azureクラウドに展開する決定により、次のことが可能になりました。


  • 管理タスクを簡素化する
  • ソリューションを簡単にスケーリング（プロセッサコア、RAM、仮想マシンの数を管理）
  • 利用可能なサービスを使用して、迅速に開発を開始します。


選択したアプローチの欠点は次のとおりです。

  1. Xamarinがサイズを大きくしたため（iOSおよびAndroid）、アプリケーションは非常に重いことが判明しました（約19 MB）-これはAndroidユーザーからの否定的なフィードバックを引き起こしました
  2. Xamarinには、報告する必要のあるバグがあります（すぐには修正されませんが、すぐには修正されません）。


比較のために、アプリケーションのサイズは次のとおりです。

  • Android-19 Mb
  • iOS-19 Mb
  • Windows Phone-2 Mb
  • Windows 8-4.5 MB。
