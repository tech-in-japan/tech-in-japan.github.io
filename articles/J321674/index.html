FIFOの仕組み

FIFOはデジタル技術の重要な要素の1つです。 このメモリは、「最初に入力された最初の左」タイプです（最初の入力-最初の出力）。 FPGA FIFOの開発者として、私はあらゆる場所に囲まれています。 実際、1つのFIFOからデータを取得して別のFIFOに転送するだけです。 しかし、それはどのように機能しますか？ もちろん、最新のCADシステムには、すでに既製の要素があり、アルテラには素晴らしいメガファンクションがあります。 ザイリンクスにはコアジェネレーターがあります。 しかし、標準ソリューションに適さないものがある場合はどうでしょうか？ 唯一の答えは、それを把握して自分で書くことです。



FIFOに関する多くの記事がインターネット上にあり、かつて私は非常に良い賢明な記事に出会いました。 残念ながら、私は今彼女を見つけていません。 次は、FIFOコンポーネントの作成と使用に関する私の個人的な経験です。 完成したアイテムは、 fpga_componentsプロジェクトのGithubにあります。 コンポーネントはいくつかの理由で必要でした：



  1. FIFO XIlinxは再送信モードで動作する方法を知りません-これが主な理由です
  2. FIFOザイリンクスでは、指定されたパラメーターを使用してコンポーネントを作成する必要があります。離婚したコンポーネントが多すぎます。
  3. FIFOザイリンクスにはエラーが含まれています。データ書き込み信号と同時にリセット信号が到着すると、FIFOで1ワードがスタックします。 これは確かに回りましたが、それでも不快です。


FIFOとは何ですか。 一般に、これはデュアルポートメモリ、2つのアドレスカウンター、およびデータの読み取りと書き込み用の2つのマシンです。







FIFOの主な用途の1つは、1つのクロック周波数から別のクロック周波数へのデータの転送です。 これにより、このようなスキームが決まります。 書き込みと読み取りに単一のクロック速度を使用すると、マシンを簡素化できます。



FIFOコンポーネントの外部ポートを見てみましょう。



cl_fifo_m12 
component cl_fifo_m12 is generic( FIFO_WIDTH : in integer:=64; --  FIFO FIFO_SIZE : in integer:=4096; --  FIFO FIFO_PAF : in integer:=16; --    PAF FIFO_PAE : in integer:=544 --    PAE ); port( --  reset_p : in std_logic; -- 1 -  --  clk_wr : in std_logic; --   data_in : in std_logic_vector( FIFO_WIDTH-1 downto 0 ); --  data_en : in std_logic; -- 1 -   fifo flag_wr : out bl_fifo_flag; --  fifo,   clk_wr cnt_wr : out std_logic_vector( 15 downto 0 ); --   --  clk_rd : in std_logic; --   data_out : out std_logic_vector( FIFO_WIDTH-1 downto 0 ); --  data_rd : in std_logic:='0'; -- 1 -   fifo,     flag_rd : out bl_fifo_flag; --  fifo,   clk_rd cnt_rd : out std_logic_vector( 15 downto 0 ); --   rt : in std_logic:='0'; -- 1 -       rt_mode : in std_logic:='0' -- 1 -        FIFO ); end component;




コンポーネントのセットアップ：



  • FIFO_WIDTH-FIFOの幅、任意の幅。
  • FIFO_SIZE-FIFOのワード数。これは64から65536までの2のべき乗です。より大きなサイズが必要な場合は、複合FIFOを実行する必要があります。
  • FIFO_PAF-ほぼ満杯のFIFOフラグのトリガーレベル。
  • FIFO_PAE-フラグのトリガーレベルはほぼ空のFIFOで、フラグは次になります。


ポート名は非常に明白で、フラグに関するコメントがいくつかあります。



FIFOフラグは、bl_fifo_flag型で渡されます。 タイプ定義： 



 type bl_fifo_flag is record ef : std_logic; -- 0 - FIFO  pae : std_logic; -- 0 - FIFO   hf : std_logic; -- 0 - FIFO   paf : std_logic; -- 0 - FIFO   ff : std_logic; -- 0 - FIFO  ovr : std_logic; -- 1 -    FIFO und : std_logic; -- 1 -    FIFO end record;


負論理が使用されることに注意してください。 見つけましたか？ はい、私はまだ155、533、1533シリーズおよび個々のFIFOチップでTTLを使用した恐竜の1人です。 したがって、これらのフラグは私にとって馴染みのあるものであり、何年も前に作られ、現在も使用されています。



フラグef-FIFOが空であることを通知します。 ef = 1の場合、FIFOから1ワードを読み取ることができます。

Paeフラグ-FIFOがほとんど空であることを示します。 FIFO_PAEパラメーターがほぼ決定する程度。 pae = 1の場合、FIFOから読み取れるのはFIFO_PAEワード以下です。

Hfフラグ-FIFOが半分いっぱいであることを示します。

Pafフラグ-FIFOがほぼ完成したことを示します。 FIFO_PAFパラメーターがほぼ決定する程度。 paf = 1の場合、FIFOに書き込むことができるのはFIFO_PAFワード以下です

フラグff-FIFOが完了しました。 ff = 0の場合、FIFOに書き込むことはできません。

ovrフラグがオーバーフローしています。 ovr = 1の場合、これはフルFIFOにレコードがあったことを意味します

フラグとアンダーフロー。 und = 1の場合、これは空のFIFOからの読み取りが発生したことを意味します。



FIFOに書き込む場合、デュアルポートメモリにワードを書き込み、書き込みカウンターを増やす必要があることは明らかです。 または、最初に増やしてから燃やします。 また、読み取り操作中は、出力でデータを修正し、読み取りカウンターを増やす必要があります。 ただし、次の問題を解決する必要があります。



  1. FIFOが完全か不完全かを判断する方法、すなわち 書いてもいいですか？
  2. FIFOが空であるか空でないかを判断する方法は？ つまり それから読むことは可能ですか？
  3. PAE、PAF、HFフラグを正しく形成する方法は？
  4. FIFOのワードカウントは何ですか？


これらすべての質問に対する答えが、書き込みおよび読み取り用のアドレスカウンターの分析にあることは明らかです。 ただし、これらのカウンターは異なる周波数で動作します。 ここから実装の違いが始まります。 カウンター値を別のクロックドメインに転送するために対称スキームを適用しました。 その結果、読み取りマシンと書き込みマシンのそれぞれに、そのカウンターの値と、他のカウンターの遅延値があることがわかりました。 これらの値から、マシンはフラグとFIFO内のワード数の値を形成します。 これは、構造図で表すことができます。







リプレイノード（プロジェクトでは、これはコンポーネントctrl_retack_counter_m12）が数クロックサイクルの遅延でデータを送信することを明確に理解する必要があります。 したがって、FIFOの状態も遅れて変化します。 たとえば、FIFOが空で1ワードが書き込まれた場合、フラグef = 1が少し遅れて表示されます。 FIFOのワードカウント出力にも同じことが当てはまります。 たとえば、16個のワードが空のFIFOに書き込まれる場合、書き込みプロセス中にcnt_wrの出力は値0、1、2、3、... 16を取ります（FIFOからの読み取りが実行されない場合）が、cnt_rdの出力は次のような値を取ります： 0、5、8、12、16。正確な順序は周波数の比率に依存し、予測することはできません。 これは、異なる周波数で動作する基本的なFIFOプロパティです。 ただし、同期スキームによっては、さまざまなニュアンスが存在する場合があります。



空および完全なFIFOは、アドレスカウンターを分析することによって決定されます。 また、書き込み用の2つのアドレス（現在および次）と読み取り用の2つのアドレス（現在および次）があります。 cl_fifo_control_m12コンポーネントでは、これらは信号w_adr、w_next_adrおよびr_adr、r_next_adrです。 さまざまな状態のアドレスの比率を以下の図に示します。



初期状態では、w_adr = 0、r_adr = 0、w_next_adr = 1、r_next_adr = 1です。 w_adr = r_adrの場合、FIFOは空です。







記録するとき、データワードはw_adrに記録され、記録アドレスがインクリメントされます。







これらのいくつかの後、w_adrの値はw_adr_to_rdに転送され（clk_rdのクロックドメインに移動します）、r_adrとw_adr_to_rdが一致しない場合、フラグef = 1が設定されます。 FIFOからデータワードを読み取ることができます。 ただし、1語では十分ではありません。高い伝送速度を得るには、データブロックを操作する必要があります。 そして、ここではPAEフラグを使用する必要があります。 FIFO_PAEワードがFIFOに書き込まれると、pae = 1フラグが設定され、データブロックをすぐに読み取ることができます。 これは、DMAチャネルでの主要な動作モードです。



書き込み速度が読み取り速度よりも速い場合、書き込みアドレスは読み取りアドレスに追いつきます。







この場合、w_next_adrはr_adr、またはr_adr_to_wrに等しくなります（クロックドメインの値のみを比較できます）。 これは、すでに記録されたデータを損なわないように、FIFOが完全であり、それ以上記録できないことを意味します。 ADCの接続は一般的な状況であることに注意してください。 FIFOを介した単一コレクションと呼ばれるこのモードがあります。 このモードでは、ADCはデータを高速でFIFOに書き込み、低速のプロセッサがこのデータを読み取ります。 同時に、FIFOのサイズに対応するデータブロックのみが有効であることがわかります。 通常、DMAチャネルはこのサイズ用にプログラムされています。 データを読み取った後、FIFOはリセットされ、すべてが再び繰り返されます。 ただし、このモードでは、完全なFIFOのレコードが以前のデータを損なわないことが基本的に重要です。



データをブロック単位で書き込む場合は、PAFフラグを使用する必要があります。 paf = 1の場合、FIFO_PAFワードをFIFOに書き込むことができます。



PAEおよびPAFフラグの値は、FIFOが接続されているコントローラーのDMAの要件から選択する必要があります。 たとえば、PCI Expressの場合、4 kBのデータブロックを使用します。 これは、128ビットの256ワードです。 PAEフラグのサイズを272に設定します。 256を少し超えます。FIFOの破壊を防ぐために、これを意図的に行います。 まあ、私は旗の形成スキームを信頼していません。



また、FIFOでワードカウントはどのように決定されますか すべてが非常に簡単です-書き込みアドレスから読み取りアドレスを引く必要があります。 アドレスは次数2の倍数であるため、減算は2 ^ Nを法として行われます。 アドレスのペアが2つあるため、1つのFIFOのワード数に2つの値を取得します（これは量子力学に何らかの関係があるのでしょうか？）。



PAEおよびHFフラグ（読み取り）の値は、r_cntから生成されます。 PAFおよびHF値（レコードごと）はw_cntから生成されます。



FIFOコンポーネントを開発しなければならなかった主な理由は、DACで作業するために周期モードを実装する必要があるためです。 このモードでは、データブロックが記録されます;もちろん、FIFOサイズを超えない任意のサイズにすることができます。 そして、読み取りが開始され、最後に記録された単語を発行した後、最初の単語への遷移がすぐに発生します。 これにより、低速プロセッサを高速DACに接続できます。 FIFOコンポーネントには、巡回モード用の2つの入力があります。 rt_mode = 1は、最後に記録されたワードを発行した後、ゼロアドレスに移動することを意味します。



ただし、入力rtはもう少し必要です。 rt = 1が存在すると、FIFOをいつでもゼロアドレスに変換できます。 時にはそれも私たちと一緒に使用されます。



fpga_componentsプロジェクトには 2つのFIFO が含まれています 。



  • cl_fifo_x64_v7
  • cl_fifo_m12


cl_fifo_x64_v7は長い間開発および公開されました。 また、長い間使用されており、その性能が実証されています。 Core Generatorによって生成されたコンポーネントをデュアルポートメモリとして使用します。 異なるFIFOサイズには独自のコンポーネントが必要です。たとえば、fpga_components \ src \ fifo \ fifo_v7 \ coregenディレクトリには4つのコンポーネントがあります



  • ctrl_dpram512x64_v7
  • ctrl_dpram1024x64_v7
  • ctrl_dpram8192x64_v7
  • ctrl_dpram32768x64_v7


これはすべて、64ビット幅のバス専用です。 他のタイヤおよび他のサイズには、独自のコンポーネントが必要です。 私たちはそれらを軽率に行いましたが、今では大きな山がありますが、それはすでに作業するのに不便です。 Alexander Capitanov（ capitanov ）はこれに注目し、完全に合成されたFIFOを作成するエレガントなソリューションを提案しました。 彼はこれを自分のプロジェクトに実装しました： github.com/capitanov/adc_configuratorコンポーネント： ctrl_fifo_config 主なアイデアは、このVHDLコンストラクトを使用することです。 



 type RAM is array (integer range <>) of std_logic_vector(DATA_WIDTH-1 downto 0); signal Mem : RAM (0 to DATA_DEPTH-1);


このデザインは、デュアルポートメモリに合成されます。 このアイデアは美しく、cl_fifo_x64_v7の改良の結果、FIFO cl_fifo_m12になりました 。
