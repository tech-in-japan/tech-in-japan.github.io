はじめに



それで、私たちは何をするつもりですか？

私は1石で2羽の鳥を殺すことにしました。自分の視点を証明するだけでなく（N900の記述は簡単で有望です）、必要なアプリケーションを作成することもできます。 そして、このアプリケーションは、電車の動きの時刻表です。 それに基づいて入門コースを作成するのに十分簡単であり、私の人生を改善するのに十分有用です（そして、私は他の誰かを願っています）。

私はよく電車を使用します（渋滞に立つ必要はなく、運転に集中する必要がないので映画を読んだり見たりする機会があります）が、このタイプの交通機関に乗る人は電車の間に十分に長い休止がある状況になることがあることを知っていますプラットフォームで待機する必要があります（この場合のプラットフォームはコンクリート構造なので、 コンピュータープラットフォームではなく、高鉄道車両に乗る方が便利です ）。 最寄りの列車のスケジュールをポケットに入れておけば、状況を改善できます。



当然、アプリケーションが最も目立つ場所であるデスクトップ上のN900にあると、より便利になります。 これは、アプリケーションをウィジェットとして作成することで実装できます。 さらに、これはMaemo5の公式開発ツールであるため、Qtで記述します（Maemo5はGTK / Hildonプラットフォームであるという事実にもかかわらず）。

さらに、アプリケーションは、プラズモイドまたはハングアップしてデスクトップコンピューターにスケジュールを表示する小さなユーティリティに簡単に変換できます。 とにかく、このアプリケーションは電車に限定されません。 誰かが予定通りに出発するバスや他の種類の交通機関を利用しているのかもしれません（フェリーの場合もあります:-)）。



スケジュールはどこで入手できますか？



アプリケーションの設計に進む前に、情報自体の入手先を決定する必要があります。



アプリケーションは特定のスケジュールプロバイダーに関連付けられるべきではない（少なくとも現時点では）と考えたため、これは組織側から時間がかかる可能性があるため、この問題に最小限の時間を費やし、アプリケーション自体の開発に専念することにしました。 すぐに目を引く最も簡単なオプションは、アプリケーションを作成することです。これは、スケジュールでhtmlページを解析し、アプリケーションを理解する単一のビューに変換します。

アプローチの利点：

  • アプリケーションは特定の情報源に関連付けられていません（htmlページではない場合もあります）。
  • ソースコードを変更せずにアプリケーションを透過的にスケーリングできます（たとえば、静的コンテンツ、Webサービス、またはプロキシサービスからアプリケーションを提供できます）。


短所：

  • コンテンツの関連性を個別に監視し、スケジュールのソースごとにコンバーターを作成する必要があります。
  • スケジュールは静的で、「ステーションからステーション」タイプのテーブルの形であることがわかります。つまり、ステーション1からステーション2に移動する必要がある場合、1つのスケジュールが必要です。また、ステーション1からステーション3 2）次に、別の別のスケジュールが必要です。
  • プログラム自体からスケジュールを選択する方法はありません。


アプローチの欠点を考えると、私はすぐにこれがすべて一時的なものであると判断しました。 後で、宛先を動的に選択する機会があるように、データ形式を作り直す必要があります。

実装。



ここに私が終わったものがあります：



スキームに従って、サイトからの情報がPythonスクリプトによって解析され、上記のxmlファイルの形式でhttp経由でアプリケーションに提供されることを追跡できます。

RaspParserは、このxmlを解析するQXmlStreamReaderの子孫です。 分析は、再帰降下テクニックによって実行されます。 これは、 Pull Parserの Qt実装です。

実装の主要部分全体がいくつかの行に収まります。

void RaspParser::readRaspItem()

{

Q_ASSERT(xml.isStartElement() && xml.name() == "train" );



RaspItem theItem;

while (xml.readNextStartElement()) {

if (xml.name() == "time" )

theItem.time = QTime::fromString(xml.readElementText().trimmed(), "hh:mm" );

else if (xml.name() == "url" )

theItem.url = QUrl(xml.readElementText().trimmed());

else if (xml.name() == "note" )

theItem.note = xml.readElementText().trimmed();

else

xml.skipCurrentElement();

}

items->append(theItem);



}

if (index.row() >= d->items.size())

ただし、少し説明が必要です。 このクラスは、 前に説明したpimplテンプレートを使用して記述されています。 専用スロットもあります。

データメソッドでは、Qt :: UserRole + 1およびQt :: UserRole + 2の2つの役割が関与しました。 3つのフィールド（時間、メモ、リンク）があり、モデルは1次元リストの形式であるため、デリゲートは何らかの方法でメモとリンクに関する情報を見つける必要があります（「出発時間」はQt :: DisplayRoleのロールを通じてデータとして直接返されます） ）必要に応じて。 これらのロールを介して、デリゲートはこのデータにアクセスできます。

もちろん、リストではなくテーブルの形式で作成することも可能ですが、この場合、データを表すのはリストであり、追加情報を表示する方法はデリゲートのタスクであると判断しました（テンプレートによる）。
