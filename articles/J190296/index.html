顧客のデバイスにインストールするためのXcodeからのAdHocアプリケーションビルドの発行の自動化

iOS向けに開発する場合、ほとんどの場合、開発モードでアプリケーションを構築すること、つまり、デバイスにインストールすることを初心者向けに説明します。 自分のためだけに。

しかし、ある時点で、顧客に「見る」ための作業の結果を提供し始める必要があります。

このために、特別なタイプのアセンブリAdHoc Distributionが使用されます。 Appleにアクセスして、配布プロビジョニングプロファイルを作成する必要があります。 次に、アプリケーションを収集し、このプロファイルで署名します。 この段階でこのアプリケーションをインストールする予定のデバイスのすべての識別子がプロファイルに登録されます。 その結果、AdHocでビルドすると、Xodeは拡張子.ipaのファイルを作成します。このファイルは、プロファイルに登録されているすべてのデバイスに既にインストールできます。 たとえば、iTunesを通じて。



アセンブリをクライアントに伝えるのに最適な方法に関して疑問が生じます。 はい。たとえば、単にファイルをメールで送信したり、ファイルホスティングサービスに置いて、貧しい人に自分のデバイスのiTunesからインストールさせたりすることができます。 しかし、あなたがクライアントの時間を大切にしている、またはあなたに彼にそれを行う方法を説明するのが面倒である、あるいはあなたがただ甘くて楽しい人であるなら、あなたは別のより便利な方法があるかどうかを考える必要があります。



Xcodeからの自動化で、これらのメソッドの1つについて説明します。







長い間、この道ですでに十分なサポーターが働いており、確かに、このタスクへの対処はそれほど悪くありません。 当然のサービスの1つは、たとえばTestFlightです。 肝心なのは、クライアント（またはベータテスター）がサービスを通じてアプリケーションをインストールできるようにすることです。 これを行うには、すべてのベータテスターがデバイスにアプリケーションをインストールする必要があります。 そして将来、彼らがあなたの次のビルドのリリースについてメールで通知を受け取ったとき、彼らはこのアプリケーションを起動し、そこであなたのアップデートを見つけ、彼らのインターフェースを通してそれをインストールします。



しかし、このスキームは理想的ではありません。

まず、これは、TestFlightに登録する方法、自分でアプリケーションをインストールする方法を顧客に説明する必要があることを意味します（AppStoreにはなく、インストールはWebサイトから行われます）。

また、この場合、私たちの側の生活を簡素化しません-Xcodeで最初に別のAdHocビルドを収集する必要があるたびに、そのオーガナイザーウィンドウでAdHoc配布オプションを選択し、正しいプロファイルを指定し、.ipaファイルをディスクに保存します 次に、TestFlight Webサイトにアクセスして、この.ipaを入力し、コメントを書きます。このビルドの対象者にアクセス許可を設定することを忘れないでください。 非常に多くのジェスチャー。

また、多くの場合、サービスは利用できないか、単にダウンロード速度がアプリケーションのインストールプロセスがタイムアウトするほどです。

何らかの理由で、アプリケーションがインストールされていないことが確実であり、TestFlightからのダウンロードを自分で確認したときに、アプリケーションがインストールされていないことを顧客から聞くのは特に喜ばしいことではありません。

その結果、私にとって最後のストローは、最新のプロジェクトを開発しているiOS7へのインストールをまだサポートしていないという事実でした。

そして、我々は両方の当事者にとってより簡単であることが判明したソリューションに到達しました。



これで、上記の手順全体が2つのステップに削減されます。 私は、以前と同様に、Xcodeアーカイブビルドを選択します（これにより、通常AdHocのアセンブリがアセンブルされます）。 その後、アプリケーションが更新されたことをお客様に通知します。 それだけです

これで、顧客は自分のデバイスからインターネット上の特定のアドレスに移動するだけです（アドレスは変更されず、ブックマークに保存できます）。 ポップアップウィンドウが表示されます-アプリケーションをインストールしますか、もちろん彼はそれを望みます。アプリケーションがインストールされます。 誰もが幸せです。

モバイルブラウザを介してアプリケーションをインストールする魔法。



まず、URLを使用してアプリケーションをインストールできる魔法について少し説明します。

これはかなり前の機会であり、これらすべてのサービスは基本的にTestFlightに基づいています。



このようなプロトコルitms-servicesがあります。特に、その助けを借りて、ブラウザーからiOSアプリケーションがインストールされます。 このプロトコルは、特定の構造を持つ.plistファイルがロードされることを意味します。特に、.ipaファイルへのリンクが示されます。 結果として、アプリケーションをインストールする要求としてiOSによって処理されます。

つまり 最初に行う必要があるのは、いくつかの単純な（または実際には望んでいない）HTMLページを作成することです。 次に、魔法の.plistファイルと.ipaビルドを配置します。



ここでは、Amazon S3ストレージがそのようなストレージに使用されることを意味します。 アマゾンウェブサービスへのサインアップは簡単で、ここでは説明しません。 そのようなわずかなボリュームのストレージと配布は、私がここで計算するのが面倒ですが、今はそれについてではありません。



したがって、ファイルを作成します。これがHTMLの外観です。 それを呼び出して、index.htmlと言います。 主なものは 、 itms-services： protocolのリンクが含まれていることです 



<html> <head> <title>My Ad-Hoc Distribution Site</title> <style> li{font-size:60pt margin:20px 0} </style> <meta name="viewport" content="width=device-width" /> <meta name="apple-mobile-web-app-capable" content="yes" /> </head> <body> <ul> <li><a href="itms-services://?action=download-manifest&url=http://mybucketname.s3.amazonaws.com/MyApp1.plist">    MyApp1</a></li> <li><a href="itms-services://?action=download-manifest&url=http://mybucketname.s3.amazonaws.com/MyApp2.plist">     MyApp2</a></li> </ul> </body> </html>




ご覧のとおり、リストはここに実装されています。1つのアプリケーションだけに限定されるのではなく、いくつかのアプリケーションがあります。 そして、それらはそれぞれ個別のリンクにインストールされます。



アドレスitms-services：// ...で変更する必要があるのは、 mybucketnameと.plistファイルの名前だけです。 つまり それに応じて、これらすべてを配置するAmazon S3のバケットの名前と.plistの名前に置き換えます（ファイル名は重要ではなく、アプリケーションの名前と一致しない場合があります）。



.plist自体は次のようになります。 

 <?xml version="1.0" encoding="UTF-8"?> <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd"> <plist version="1.0"> <dict> <key>items</key> <array> <dict> <key>assets</key> <array> <dict> <key>kind</key> <string>software-package</string> <key>url</key> <string>http://mybucketname.s3.amazonaws.com/MyApp1.ipa</string> </dict> </array> <key>metadata</key> <dict> <key>bundle-identifier</key> <string>com.mycoolcompanyname.myapp1</string> <key>bundle-version</key> <string>1.0</string> <key>kind</key> <string>software</string> <key>title</key> <string>MyApp1</string> </dict> </dict> </array> </dict> </plist>




もちろん 、ここでは、 ソフトウェアパッケージキー文字列の名前mybucketnameを置き換える必要もあります。 そして、MyApp1.ipaの代わりに.ipaファイルの名前を示します。 そして、キー文字列bundle-identifierを変更することを忘れないでください。 com.mycoolcompanyname.myapp1の代わりに、このアプリケーションをXcodeでビルドするときに使用したバンドルIDを入力します。

何もする必要がないように、小さなスクリプトを作成します。

Xcodeには、アプリケーションを構築するときに、ほぼすべての段階でスクリプトを詰め込む機能があります。

この場合、アーカイブへのアセンブルにのみ関心があります（これが、AdHoc Distributionのアプリケーションを常に構築する方法です）。

したがって、このスキームの最終段階で事後アクションを追加します。

Xcodeに移動し、回路を編集するためのダイアログを開きます。 次は、いくつかの説明的なスクリーンショットです。 これらはXcode 5のベータ版で作成されていますが、Xcode 4.6にもこの機能があり、この部分のインターフェイスは実質的に同じです。



メインのXcodeウィンドウの左上に示されている場所で、ターゲットの名前をクリックします。 表示されるダイアログで、[スキームの編集...]アイテムを選択します。

スキームを編集するためのダイアログで、Arhiveスキームを見つけ、その名前の横にある小さな三角形をクリックすると、いくつかのサブアイテムが展開されます。 最後の1つは、必要な後処理です。







このウィンドウのドロップダウンボックスで、ターゲットの名前を選択します。 これは、たとえば$ {PRODUCT_NAME}など、スクリプト内のすべての変数がそれに対応して取得されることを意味します。



ここでは、このようなスキップを使用します。 



 SIGNING_IDENTITY="iPhone Distribution: MyCoolCompanyName (G4DHGXDY2)" PROVISIONING_PROFILE="${HOME}/Library/MobileDevice/Provisioning Profiles/20DB9849-4CFE-4005-81F6-1A594119839B.mobileprovision" LOG="/tmp/post-build-upload-to-s3.log" DATE=$( /bin/date +"%Y-%m-%d" ) ARCHIVE=$( /bin/ls -t "${HOME}/Library/Developer/Xcode/Archives/${DATE}" | /usr/bin/grep xcarchive | /usr/bin/sed -n 1p ) DSYM="${HOME}/Library/Developer/Xcode/Archives/${DATE}/${ARCHIVE}/dSYMs/${PRODUCT_NAME}.app.dSYM" APP="${HOME}/Library/Developer/Xcode/Archives/${DATE}/${ARCHIVE}/Products/Applications/${PRODUCT_NAME}.app" /usr/bin/open -a /Applications/Utilities/Console.app $LOG echo -n "Creating .ipa for ${PRODUCT_NAME}... " > $LOG /bin/rm "/tmp/${PRODUCT_NAME}.ipa" /usr/bin/xcrun -sdk iphoneos PackageApplication "${APP}" -o "/tmp/${PRODUCT_NAME}.ipa" --sign "${SIGNING_IDENTITY}" --embed "${PROVISIONING_PROFILE}" >> $LOG echo "Created .ipa for ${PRODUCT_NAME}" >> $LOG echo -n "Uploading to S3... " >> $LOG /opt/local/bin/s3cmd put --acl-public --force --guess-mime-type /tmp/${PRODUCT_NAME}.ipa "s3://mybucketname/MyApp.ipa" >> $LOG echo "Done." >> $LOG /usr/bin/open "https://basecamp.com/path-to-your-project-to-let-people-know-of-new-build"




したがって、開いたウィンドウで、入力の最初の行の/ bin / shを参照してください。 そこにこの値を残します。

次に、構成に合うようにいくつかの値を編集する必要があります。 つまり、SIGNING_IDENTITY変数に正しい値を書き込み、PROVISIONING_PROFILE変数に現在のプロビジョニングプロファイルへの正しいリンクを書き込む必要があります。

新しいXcode 5では、これら2つの値は「コード署名ID」セクションの個別のエントリに分割されています。 したがって、同じアクションを2回実行する必要があります。 最初に名前IDをコピーしてから、プロビジョニングプロファイルのコードをコピーします。



ご覧のとおり、スクリプトは2つの単純なアクションを実行します。 最初に、結果の.appファイルが.ipaファイルに変わります。 次に、この.ipaがS3にアップロードされます。 スクリプトの実行プロセスでは、発生するすべてが書き込まれる.logファイルも作成されます。 この.logファイルは一時フォルダー（.ipaと同じ）に配置され、コンソールシステムユーティリティによって開かれるため、スクリプトの進行状況を確認できます。

次に、s3cmdユーティリティ自体をインストールします。 MacPortsでは、これは非常に簡単です。

 sudo port install s3cmd




パスワードの入力、管理者パスワードの入力を求められ、インストールが開始されます。 約1分かかります。
