過去の記事では、ページの受信と解析、再帰的なクリックスルー、リクエストキューの整理と微調整、Ajaxサイトの分析、サーバーエラーの処理、セッションの初期化、ボットに対する保護を克服する方法が検討されてきました。







この記事では、定期的に更新されるデータのWebスクレイピング、変更の追跡、クラウドプラットフォームを使用したスクリプトの実行やデータの保存などのトピックに焦点を当てています。 Webスクレイピングのタスクと完成したデータの処理の分離、および更新されたサイトで作業する場合に避けるべきことにさらに注意が払われます。







この記事の目的は、タスクの設定から最終結果の取得まで、スクリプトを作成、展開、使用するプロセス全体を示すことです。 いつものように、フリーランスのやり取りでよく見られる実際の問題が例として使用されます。







問題の声明



顧客がBuzzbuzzhome Webサイトでプロパティデータを追跡したいとします。 彼はボルチモアの街にしか興味がありません。 各物件のオファーについて、顧客はオファーページへのリンク、オブジェクトの名前、および価格情報（または価格の代わりにサイトに書かれているもの）を見たいと考えています。







顧客は定期的に情報を更新する機能を必要としています。 実際、彼はいつでもExcel互換ファイルを取得する機会を必要とします。Excel互換ファイルでは、現在のオファーのみが存在し、すべてのオファーの中で新しいオファー（最後に要求されたファイルにはなかった）がマークされ、前回から変更されます。 顧客は、価格情報のみが変更されることを確信しています。







顧客はスクリプトをネットワーク上に配置し、Webインターフェースを介して作業したいと考えています。 彼はどこかに無料のアカウントを持ち、クラウドでスクリプトを自分で実行し（ブラウザーのボタンをクリック）、CSVをダウンロードすることに同意します。 データのプライバシーは彼を悩ませませんので、パブリックアカウントを使用できます。







サイト分析



Buzzbuzzhome Webサイトでは、2つの方法で関心のある情報を取得できます。メインページの検索フィールドに「 Baltimore 」を入力するか（ポップアッププロンプト「 Baltimore、Maryland、United States 」が表示されます）、カタログでBaltimoreを見つける（メインメニューの「Cities」項目、選択州のリストから「 メリーランド 」を選択し、都市のリストから「 ボルチモア 」を選択します。 最初のケースでは、マーカー付きのマップとAjaxを介してロードされたリストを取得し、2番目のケースでは、退屈なリンクのリストを取得します。







サイトにシンプルな旧式のカタログがあるという事実は大成功です。 そうでない場合、多くの困難を克服する必要があります。Ajaxトラフィックの分析ではなく、データ自体について話しています。 事実、地図の横のリストには、ボルチモアのオファーに加えて、地図に載ったばかりのゴミがたくさんあります。 これは、検索が住所ではなく地理座標によって実行されるためです。 さらに、座標による検索結果には、アドレスが正確または不完全に指定されていないオブジェクトが含まれていないことが多いことを考慮する価値があります（このサイトがこれを行わない場合でも、今後そうしないことを保証することはできません）。 ガベージをアドレスで簡単にフィルタリングできる場合、欠落しているオブジェクトについて明確な解決策はなく、顧客に何をすべきかを確認する必要があります。







幸いなことに、Baltimoreがリストされている住所のすべてのオブジェクトを取得する別のページがWebサイトのカタログにあります。 個々のオブジェクトのページは簡単に解析されます（少なくとも名前と価格情報）。







データ検索



データの更新について考えなければ、タスクは非常に簡単です。 リンクを再帰的に通過できないため、最初の記事よりもさらに簡単です。 不動産のオファーへのリンクのリストを取得し、キューに入れ、オファーの各ページを解析し、結果の配列を取得します。 たとえば、次のように：

データ保存



おそらく、VPN、PaaS（Herokuなど）、またはNode.jsサポートを使用した共有ホスティングなどのホスティングオプションの詳細な分析を省略することができます。 これについてはすでに多くのことが書かれています。 人件費を大幅に削減できる、スクレーパー専用のPaaSソリューションがあるという事実からすぐに始めることができます。 screen-scraperのような、微調整の解析のみを必要とする「ユニバーサルスクレーパー」ではなく、独自のスクリプトを起動するための本格的なプラットフォームについてです。 最近まで、このニッチのリーダーはScraperWikiでしたが、現在このサービスは読み取り専用モードに切り替えられており、徐々に別のものに変わりつつあります。 このニッチの新しいリーダーであるMorph.ioサービスを呼び出します。 便利で、無料で、非常に簡単に展開できます。 Morph.ioユーザーは、512メガバイトのメモリと24時間のスクリプトに収まるように求められますが、これが誰かに十分でない場合は、個別に調整してみてください。 このサービスには多くの利点があります。スクレイピング用に研ぎ澄まされた既製のWebインターフェイス、便利なAPI、毎日の自動起動、Webフック、アラートなどです。 Morph.ioの主な欠点は、プライベートアカウントがないことです。 たとえば、秘密の環境変数にあらゆる種類のパスワードを保存できますが、スクリプト自体と受信したデータはすべてのユーザーに表示されます。

変更データの追跡



スクレーパーの観点から、変化するデータを追跡するタスクは単純と複雑に分けられます。







単純なタスクとは、ソース内のデータの変更が破棄されるよりも著しく遅い場合です。 クエリを最適化および並列化する必要がある場合でも、これはまだ簡単なタスクです。 また、単純なタスクでは、変更データの次のバージョンを1回のスクリプト実行で破棄できます。 これは毎回発生するわけではありませんが、主なことは「再開」を実装する必要がないことです。 単純なタスクは、衝突がないこと、または実際に顧客を心配しないことを前提としています。 簡単なタスクでは、Webスクレイピングを使用して次のバージョンのデータを取得し、必要なことを静かに実行できます。







困難なタスクは、スクレイピングを放棄して別のエンジニアリングソリューションを選択する必要がある場合です。







私たちのタスクは簡単です。 1つのストリームでも、1〜2分でデータを取得できます。 データの更新は1日に数回しか必要ありません（顧客に確認する価値がありますが、通常は必要です）。 たとえば、2つの異なるテーブルに古いバージョンと新しいバージョンを保存して、それらをマージできます。







別のテーブルを取得しましょう（ new



と呼びましょう）。 作成し、スクレイピング、メリー、および削除の結果で埋めます。 data



テーブルに、たとえば、レコードが新しい場合は ' new



'、レコードが古いが以前のバージョンから変更された場合は ' upd



'など、書き込むstate



列を追加します。 特定の指定は、最終的なCSVで十分に読まれている限り、重要ではありません。 マージする前に、列全体がNULL



リセットされNULL



。







テーブルのマージは、3つのSQLクエリで実行されます。







新しいエントリを追加する 
 INSERT INTO data SELECT url, name, price, "new" AS state FROM new WHERE url IN ( SELECT url FROM new EXCEPT SELECT url FROM data );


廃止されたエントリの削除： 
 DELETE FROM data WHERE url IN ( SELECT url FROM data EXCEPT SELECT url FROM new )


変更されたレコードの更新： 
 UPDATE data SET state = "upd", price = ( SELECT price FROM new WHERE new.url = data.url ) WHERE url IN ( SELECT old.url FROM data AS old, new WHERE old.url = new.url AND old.price <> new.price )


「第2言語を学ばないように」Node.jsを選択した人は、最小限のSQLで対処し、Javascriptでメインロジックを実装することができます。 基本的に、これは全体的な問題ですが、考慮すべき点が1つあります。低電力ホスティングでのスクリプトの安定性は、データを操作するために特別に設計された言語を使用する場合、はるかに高くなります。 リクエストは最も難しいものではなく、スクリプトがクラッシュする頻度が低い場合、顧客は満足します。







対応するスクリプトフラグメントは次のようになります。 







 var sqlite3 = require('sqlite3').verbose(); // ...  ,      function done(){ var db = new sqlite3.Database('data.sqlite'); db.serialize(function(){ db.run('DROP TABLE IF EXISTS new'); db.run('CREATE TABLE new (url TEXT, name TEXT, price TEXT)'); var stmt = db.prepare('INSERT INTO new VALUES (?, ?, ?)'); for (var i = 0; i < results.length; i++) { stmt.run(results[i]); }; stmt.finalize(); db.run('CREATE TABLE IF NOT EXISTS data (url TEXT, name TEXT, price TEXT, state TEXT)'); db.run('UPDATE data set state = NULL'); db.run('INSERT INTO data SELECT url, name, price, "new" AS state FROM new ' + 'WHERE url IN (SELECT url FROM new EXCEPT SELECT url FROM data)'); db.run('DELETE FROM data WHERE url IN (SELECT url FROM data EXCEPT SELECT url FROM new)'); db.run('UPDATE data SET state = "upd", price = (SELECT price FROM new WHERE new.url = data.url) ' + 'WHERE url IN (SELECT old.url FROM data AS old, new WHERE old.url = new.url AND old.price <> new.price)'); db.run('DROP TABLE new'); db.close(); }); }


このようなスクリプトはGithubにアップロードし、Morph.ioにアップロードして実行できます。 読みやすくするために、クエリのテキストを別々の文字列変数に入れることは価値がありますが、読みやすさは顧客にとって重要ではありません。







重要な注意事項

この記事で説明するスクリプトは、少量のデータにのみ適しています。 現在、ボルチモアには約25件の不動産オファー（Buzzbuzzhome内）しかなく、数分で解体できます。 つまり、サーバーのエラーが原因でスクリプトがクラッシュした場合、スクレイピングが完了するまでデータベース内で何も変更されないため、単純に再起動できます。







同時に、同じサイトにはニューヨークの約1,000件のオファーがあるため、廃棄には40〜50分かかり、Buzzbuzzhomeのサーバーは非常に脆弱です。 この問題を解決するには、サーバーエラー処理を追加する必要があります（コーニー、失敗したタスクをキューに戻し、スクレイピングを短い一時停止にします、2番目の記事を参照）。スクリプトがクラッシュせず、顧客が1時間ごとに再起動する必要はありません。 してはいけないことは、データベースの再起動の間に部分的なスクレイピング結果を保存することです。 実際には、これは「更新された」データの一部が非常に古くなるという事実につながる可能性があります。







さらに、タスクがBuzzbuzzhomeカタログ全体を追跡することである場合、各都市のデータを個別にスクレイピングおよび更新することは理にかなっています。 長い間更新されていた都市のデータを別のテーブル（または何らかの方法で）に保存する必要があります。 これはMorph.ioにとって非常に大きなタスクです（少なくとも制限を満たしていません）。したがって、より強力なクラウドにスクリプトを展開する必要があります（そして、Webインターフェイスを自分で記述します）。 カタログ全体のスクレイピングに費やされる時間は数日で測定され、データはスクレイピングされたデータよりも早く古くなってしまいます。 これは単一の実際の顧客には適さないため、作業を非常に並列化する必要があります。 どのように正確に-それはすでに特定の要件に依存しますが、それは間違いなく必要です。







おわりに



結論として、定期的に更新されるデータを破棄するタスクは、起動間でデータを保持することを伴いますが、クラウドでの必須の展開を意味するものではないことを強調する価値があります。 ターミナルからオペレーティングシステムをインストールして起動するための指示とともに、同じスクリプトを顧客に提供できます。すべてが正常に機能します。 SQLiteからCSVを取得する方法を説明する必要があるため、現在のバージョンのデータの自動エクスポートを追加することをお勧めします。 ただし、この場合、たとえばJSONファイルの代わりにデータベースを使用する理由はありません。 データベースの実際の必要性は、単にメモリに収まらない膨大な量のデータを廃棄するときにのみ現れ、通常のファイルに保存するのは非常に不便です。 しかし、これは別の記事のトピックです。
