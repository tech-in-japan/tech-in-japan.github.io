ゲッターとセッター



Yiiの主な機能の1つは、ゲッターとセッターを介してオブジェクトのプロパティとメソッドに簡単にアクセスできることです。たとえば、 $ model-> userを使用してアクセスすると、モデルプロパティを取得したり、 getUser（）メソッドを呼び出したり、ユーザーリレーションを取得したりできます。 DBは完全な魔法ですが、誰もが気に入っています。

JavaScriptでは、すべてのブラウザーがゲッターとセッターをサポートしているわけではなく、多くのプロジェクトはまだIE8をサポートする必要があるため、Jiiではget（ 'user'）およびset（ 'user'、 'Ivan'）メソッドとして実装されます。 このアプローチは、 Backboneなどの多くのライブラリで見つけることができます。

将来、古いブラウザの必要性がなくなると、実際のゲッターとセッターがそれらを介して機能するget / setメソッドと並行して追加されます。



クラスと名前空間



彼らが言うように、すべての自尊心のあるプログラマーはJavaScriptでクラスの独自の実装を書くべきです。 そのため、ここでは、クラスを実装するために、他の内部プロジェクトで既によく示されているNeatnessライブラリを使用します。



ES6クラス（CoffeeScript / TypeScriptではない）を選択してください。









コメントとgithubには、これに関する小さなホリバーが既にあります。

また、JiiでES6が不足している理由も述べます。

  1. node.jsにはノードの前処理が必要です（io.jsがサポートします）
  2. es6には名前空間はありませんが、Yiiの機能を繰り返すために必要です。 モジュールを介して実装できますが、松葉杖になります。 松葉杖を別の松葉杖に変更します-意味がありません。
  3. ES6はゲッターとセッターの使用を望んでいますが、上記のように古いブラウザーではサポートされていません
  4. 多くの開発者はes6 / coffeescript / typescript形式を知らず、これは追加のエントリのしきい値になります
  5. 異なる形式（ES5およびES6）のコードでの作業が困難。 既存のプロジェクトですべてのコードをes6に変更できるとは限りませんが、es6クラスを継承する必要があり、Jii.defineClass（）を使用してクラスを作成する必要があります-松葉杖はなくなりません。

Yiiフレームワークでは、コンポーネントはYiiを介してグローバルにアクセスできます:: $ app-> ...ただし、 Jiiでは、すべてのコンポーネントをグローバルに配置できるわけではありません。 それらの一部（要求、応答、Webユーザー、セッションなど）はコンテキスト（要求）に関連付けられています。

このようなコンポーネントはコンテキスト（ Jii.base.Context ）で作成され、パラメーターとしてactionに渡されます。これはexpressにリクエストとレスポンスを渡すのと同様です。 



/** * @class app.controllers.SiteController * @extends Jii.base.Controller */ Jii.defineClass('app.controllers.SiteController', /** @lends app.controllers.SiteController.prototype */{ __extends: Jii.base.Controller, /** * * @param {Jii.base.Context} context * @param {Jii.httpServer.Request} context.request * @param {Jii.httpServer.Response} context.response */ actionIndex: function(context) { context.response.data = this.render('index'); context.response.send(); } });


エンティティ









Jiiアプリケーションは、model-view-behavior（MVC）デザインパターンに従って編成され、次のエンティティを持ちます。

  • アプリケーション ：これらはグローバルにアクセス可能なオブジェクトであり、さまざまなアプリケーションコンポーネントの正しい操作と、リクエストを処理するための調整を実行します。
  • アプリケーションコンポーネント ：これらは、アプリケーションに登録され、さまざまな機能を提供するオブジェクトです。
  • 要求コンポーネント ：これらは、要求のコンテキストで登録され、現在の要求を処理するためのさまざまな可能性を提供するオブジェクトです。
  • モジュール ：これらは、MVCのすべてのツールを含む自己完結型のパッケージです。 アプリケーションは、いくつかのモジュールを使用して編成できます。


それらの使用例はデモで見ることができます。



用途



アプリケーションは、Jiiアプリケーションシステムの構造全体とライフサイクルを制御するオブジェクトです。 通常、Jiiアプリケーションの1つのインスタンスは、1つのワーカー（プロセス）Node.jsに到達します。これは、Jii.appを介して利用できます。

入力スクリプトがアプリケーションを作成すると、構成がロードされ、アプリケーションに適用されます。次に例を示します。 



 var Jii = require('jii'); //    var config = { application: { basePath: __dirname, components: { http: { className: 'Jii.httpServer.HttpServer' } } }, context: { components: { request: { baseUrl: '/myapp' } } } }; //       Jii.createWebApplication(config);


Yiiとの重要な違いは、構成が2つの部分-アプリケーション構成（アプリケーションセクション）とコンテキスト（要求）構成（コンテキストセクション）に分かれていることです。 アプリケーション構成は、コンポーネント、モジュールを作成および構成し、アプリケーション自体（Jii.app）を構成します。これらはすべて、ワーカーの起動時に発生します。 次に、コンテキスト構成は、各アクション呼び出し（http要求、コンソールコマンド呼び出しなど）のコンポーネントを作成および構成します。 作成されたコンポーネントは、アクションメソッドの最初の引数として渡されます。



多くの場合、アプリケーションの構成は非常に複雑であるため、ファイルに転送され、いくつかの構成ファイルに分割されます。

アプリケーションには、アプリケーションが機能するためのさまざまな手段を提供する多くのアプリケーションコンポーネントが格納されます。 たとえば、urlManagerコンポーネントは、Web要求を適切なコントローラーにルーティングします。 dbコンポーネントは、データベースを操作するためのツールを提供します。 など



アプリケーションの各コンポーネントには独自の一意のIDがあり、同じアプリケーション内の他のさまざまなコンポーネント間で識別することができます。 次のようにしてコンポーネントにアクセスできます。 



 Jii.app.ComponentID

コントローラー



コントローラは、 MVCアーキテクチャの一部です。 これらはJii.base.Controllerから継承されたクラスのオブジェクトであり、リクエストの処理とレスポンスの生成を担当します。 基本的に、サーバー（ Jii.httpServer.HttpServer ）によってHTTPリクエストを処理するとき、コントローラーは入力データを分析し、それを渡します

モデルで、モデルの結果を[ビュー]（構造ビュー）に挿入し、最終的に発信応答を生成します。



アクション



コントローラーはアクションで構成されます。これは、エンドユーザーがアクセスして、いずれかの実行を要求できるメインブロックです。

機能的。 コントローラーには1つ以上のアクションがあります。



次の例は、2つのアクションviewおよびcreateを持つポストコントローラーを示しています 。 



 /** * @class app.controllers.PostController * @extends Jii.base.Controller */ Jii.defineClass('app.controllers.PostController', /** @lends app.controllers.PostController.prototype */{ __extends: Jii.base.Controller, actionView: function(context) { var id = context.request.get('id'); return app.models.Post.findOne(id).then(function(model) { if (model === null) { context.response.setStatusCode(404); context.response.send(); return; } context.response.data = this.render('view', { model: model }); context.response.send(); }); }, actionCreate: function(context) { var model = new app.models.Post(); Jii.when.resolve().then(function() { // Save user if (context.request.isPost()) { model.setAttributes(context.request.post()); return model.save(); } return false; }).then(function(isSuccess) { if (isSuccess) { context.request.redirect(['view', {id: model.get('id')}]) } else { context.response.data = this.render('create', { model: model }); context.response.send(); } }); } });


ビューアクション（ actionView（）メソッドで定義）では、コードは最初に要求されたモデルIDに従ってモデルをロードします。 モデルが正常にロードされると、コードはviewというビューを使用してモデルを表示します 。



作成アクション（ actionCreate（）メソッドで定義）では、コードは似ています。 彼は最初に、クエリからのデータを使用してモデルをロードし、モデルを保存しようとします。 すべてがうまくいった場合、コードはブラウザを新しく作成されたモデルのIDを持つビューアクションにリダイレクトします。 それ以外の場合は、ユーザーが必要なデータを入力できる作成ビューが表示されます。



ルート



エンドユーザーは、いわゆる*ルート*を介してアクションにアクセスします。 ルートは、次の部分で構成される線です。

  • モジュールID：コントローラーがアプリケーションに属さず、[モジュール]（構造モジュール）に属する場合にのみ存在します。
  • コントローラーID：同じアプリケーション（またはコントローラーがモジュールに属している場合は同じモジュール）の他のすべてのコントローラーの中でコントローラーを一意に識別する文字列。
  • アクションID：同じコントローラーの他のすべてのアクションの中でアクションを一意に識別する文字列。


ルートの形式は次のとおりです。 



 ControllerID/ActionID


または、コントローラーがモジュールに属している場合は次の形式： 



 ModuleID/ControllerID/ActionID


アクション作成



アクションを作成することは、コントローラクラスでいわゆる*アクションメソッド*を宣言することと同じくらい難しくありません。 アクションメソッドは、名前がactionという単語で始まるメソッドです。 アクションメソッドの戻り値は、エンドユーザーに送信される応答データです。 以下のコードは、2つのアクションインデックスとhello-worldを定義しています 。 



 /** * @class app.controllers.SiteController * @extends Jii.base.Controller */ Jii.defineClass('app.controllers.SiteController', /** @lends app.controllers.SiteController.prototype */{ __extends: Jii.base.Controller, actionIndex: function(context) { context.response.data = this.render('index'); context.response.send(); }, actionHelloWorld: function(context) { context.response.data = 'Hello World'; context.response.send(); } });


集団訴訟



アクションは、 Jii.base.Actionまたはその子孫から継承されたクラスとして定義できます。



このようなアクションを使用するには、次のように、コントローラークラスのJii.base.Controller.actions（）メソッドをオーバーライドしてアクションを指定する必要があります。 



 actions: function() { return { //  "error"      error: 'app.actions.ErrorAction', //  "view"      view: { className: 'app.actions.ViewAction', viewPrefix: '' } }; }


ご覧のとおり、 actions（）メソッドは、キーがアクションIDで、値がアクションクラスまたは[構成]（概念構成）の対応する名前であるオブジェクトを返す必要があります。 組み込みのアクションとは異なり、個々のアクションIDには、 アクション（）メソッドで定義されている限り、任意の文字を含めることができます。



別のアクションを作成するには、 Jii.base.Actionクラスまたはその子孫から継承し、public run（）メソッドを実装する必要があります。 run（）メソッドの役割は、他のアクションメソッドと似ています。 例えば 



 /** * @class app.components.HelloWorldAction * @extends Jii.base.Action */ Jii.defineClass('app.components.HelloWorldAction', /** @lends app.components.HelloWorldAction.prototype */{ __extends: Jii.base.Action, run: function(context) { context.response.data = 'Hello World'; context.response.send(); } });


モジュール



モジュールは、モデル、ビュー、コントローラー、およびその他の補助コンポーネントで構成される自己完結型のソフトウェアブロックです。 アプリケーションにモジュールをインストールすると、エンドユーザーはそれらにアクセスできます

コントローラーに。 このため、モジュールはしばしばミニチュアアプリケーションと見なされます。 アプリケーションとは異なり、モジュールを個別に作成することはできません。アプリケーション内にある必要があります。

モジュールの作成



モジュールは、 ベースモジュールパス （ Jii.base.Module.basePath ）と呼ばれるディレクトリに配置されます。 アプリケーションディレクトリと同様に、このディレクトリには、 controllers 、 models 、 views、およびその他のサブディレクトリがあり、これらはcontrollers、models、views、およびその他の要素をホストします。 次の例は、モジュールのサンプルコンテンツを示しています。 



 modules/ forum/ Module.js    controllers/     DefaultController.js      models/     views/       layouts/     default/     DefaultController index.ejs


モジュールクラス



各モジュールは、 Jii.base.Moduleを継承する一意のクラスを使用して宣言されます 。 このクラスは、 ベースモジュールパスのルートに配置する必要があります 。 アプリケーション（ワーカー）の起動中は

対応するモジュールクラスのインスタンスが1つ作成されます。 モジュールコードがデータとコンポーネントを共有できるように、アプリケーションインスタンスと同様にモジュールインスタンスが必要です。



モジュールクラスがどのように見えるかの例を次に示します。 



 /** * @class app.modules.forum * @extends Jii.base.Module */ Jii.defineClass('app.modules.forum.Module', /** @lends app.modules.forum.Module.prototype */{ __extends: Jii.base.Module, init: function(context) { this.params.foo = 'bar'; return this.__super(); } });


モジュール内のコントローラー



モジュールコントローラーを作成するときは、モジュールクラスの名前空間のコントローラーサブスペースにコントローラークラスを配置するのが一般的です。 また、これは、コントローラクラスファイルがベースモジュールパスの コントローラディレクトリに配置されることを意味します 。 たとえば、前の例のフォーラムモジュールでポストコントローラーを説明するには、コントローラークラスを次のように宣言します。 



 var Jii = require('jii'); /** * @class app.modules.forum.controllers.PostController * @extends Jii.base.Controller */ Jii.defineClass('app.modules.forum.controllers.PostController', /** @lends app.modules.forum.controllers.PostController.prototype */{ __extends: Jii.base.Controller, // ... });


Jii.base.Module.controllerNamespaceプロパティを設定することで、コントローラークラスの名前空間を変更できます。 コントローラーがこの名前空間から外れた場合、アプリケーションで行われるのと同様に、 Jii.base.Module.controllerMapプロパティーを設定することでコントローラーにアクセスできます。



モジュールを使用する



アプリケーションでモジュールを使用するには、アプリケーション構成のJii.base.Application.modulesプロパティにモジュールを含めるだけです。 次のコードは、アプリケーション構成でフォーラムモジュールを使用しています。 



 var config = { application: { // ... modules: { forum: { className: 'app.modules.forum.Module', // ...    ... } } }, context: { // ... } };


Jii.base.Application.modules プロパティには 、モジュール構成を含むオブジェクトが割り当てられます。 各オブジェクトキーは、モジュールを他のアプリケーションモジュールの中で一意に識別する*モジュール識別子*であり、対応するオブジェクトはモジュールを作成するための構成です。

モジュールへのアクセス



モジュールインスタンスへのアクセスは、次の方法で取得できます。 



 var module = Jii.app.getModule('forum');


モジュールのインスタンスがあれば、モジュールに登録されているパラメーターとコンポーネントにアクセスできます。 例えば 



 var maxPostCount = module.params.maxPostCount;
