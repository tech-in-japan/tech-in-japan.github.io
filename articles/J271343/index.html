nanoCADは西部のCADソリューションを置き換えることができますか？ 答えを探しましょう...





CADプラットフォームを導入するとき、私たちの国では、CADなどの複雑で多面的なツールの機能を適切に評価し、競合するソリューションを完全に比較し、公平な評価を下せる人はほとんどいません。 ほとんどの場合、そのような作業は地元のCADの第一人者に委ねられています。 現在使用しているソリューションが気に入っています。 その結果、「おいしい」が十分に活用されていない関数のリストが得られ、「それ以外の場合は実装しないでください！」、およびインポート置換のトピックはスムーズに消えていきます。 それに応えて、独立した評価を可能にし、それを実装し、あなたと共有する準備ができたツールを考えました。



はじめに

このツールのアイデアは、平均的なCADユーザーが毎日使用する機能のリストをまとめることです。 ユーザーのグループを選択し、コンピューターでコマンドラインロギングを有効にし、組織のネットワーク上の特定のフォルダーに収集し、分析します。 スキーム自体は非常に便利です。

  • すべてが完全に自動化されています。ログを有効にするだけです。
  • すべてが完全に機密です。プロジェクト自体ではなく、呼び出されたコマンドに関する情報のみがログブックに含まれます。
  • 統計の収集は、ユーザーの毎日の作業を絶対に妨げません。ログへの書き込みは、かなり目立たない操作です。


最も重要なことは、この統計の収集により、最も完全な分析を実行できることです。少なくとも1年間ログを収集でき、プログラムは1つのフォルダーにチーム呼び出しをきちんと収集します。 ユーザーをグループに分けて分析することもできます-統計を収集するために異なるフォルダーを指定するだけです。 そして、ログには、使用されたプラットフォームコマンド、アプリケーションまたは追加の開発が使用されたかどうか、プラットフォームとアプリケーションの使用レベル（コマンドコールの頻度）が表示されます。 偏りのない純粋な統計と主観性。



「このような雑誌をどのように含めるか」という疑問が生じます。 ここではすべてが簡単です。コマンドプロンプトで、コマンド「Parameters」（または英語バージョンの場合は「OPTIONS」）を入力します。 次に、「開く\保存」タブ（または「開くと保存」）に移動します。 そして、「ログファイルを保持する」オプション（または「ログファイルを保持する」-図1を参照）を設定します。 このコマンドの結果、コマンドラインからの情報（コマンドログ）は、LOGFILEPATH変数で指定されたシステムフォルダーに保存されます。





図 1. [ログファイル]オプションを有効にすると、CADソリューションで使用されるコマンドに関する統計を、LOGFILEPATH変数で設定されたフォルダーに蓄積できます。



原則として、これらはすべて3つのシステム変数で構成されます：LOGFILEMODE（ロギングモードの有効化/無効化）、LOGFILEPATH（ロギングフォルダーへのパス）、およびLOGFILENAME（現在のコマンドログへのパス）。 そのようなコマンドと変数はありませんか？ ほとんどの場合、CADを使用しているため、呼び出されたコマンドのログを保持できず、インポート置換はありませんが、競合ソリューションからの移行-これはわずかに異なる話です...



UPG：多くの場合、システム管理者は、ドメインポリシーを使用してこれらの変数を構成する方法を尋ねます（各職場に行かず、ユーザーインターフェイスを介してログのコレクションを構成しないように）。 私たちは答えます-レジストリを通じて。 特に、変数はHKCU \ Software \ Autodesk \ AutoCAD \ <version> \ ACAD-1001：セクションにあります：419 \ Profiles \ << Profile with name >> \（またはユーザーが使用するプロファイル）：

  • ログの有効化：GeneralサブフォルダのREG_DWORD型のLogFileOpen変数の値は「0x00000001」（つまり、ログ）でなければなりません。 変数がない場合は、作成する必要があります。




  • ログファイルの保存パス：Editor ConfigurationサブフォルダーのREG_EXPEND_SZタイプのLogFilePath変数には、パスの文字列値が必要です。 たとえば、「D：\ temp \ log」






そこで、ログを有効にし、ログをフォルダに収集しました-次は何をしますか？



データの前処理

結果のファイルはファイルの山で、内部には使用されているコマンドが隠されています（図2）。





図 2.コマンドログのある典型的なフォルダー-各ファイルの個別のログ。



内部を見ると、コマンドコール（興味のあるもの）、システムメッセージ、ユーザー応答、変数値、エラーメッセージ、警告など、CADで記述されたすべてがコマンドラインに表示されます。 これはすべて処理する必要があり、コマンドのリストを呼び出す頻度に関する情報とともに呼び出す必要があります。 これを行うために、指定されたフォルダー内のログファイルにログオンし、それらを順次処理し、呼び出しの頻度でコマンドの配列を形成するAutoItで簡単なユーティリティを作成しました（図3）。





図 3.ユーティリティStatCAD。コマンドログを処理し、使用するコマンドのリストを作成できます。



StatCADユーティリティの操作は非常に簡単です。ログでフォルダーを指定し、[分析]ボタンをクリックして、作業が終了するのを待ち、[統計を表示]ボタンをクリックして別のウィンドウにリストを取得します。 このウィンドウから、Excelまたはテキスト形式にデータをコピーできます-必要に応じて。 次は？ そして、創造的な仕事が始まります。



受信した統計の処理

まず、見つかったリストでCADプラットフォームコマンドを選択する必要があります。 比較するフルタイムのCADプラットフォームチームのリストが必要です。 現在、約2400のコマ​​ンド、システム変数、および略語のエイリアス（ロシア語と英語の両方）が蓄積されており、これと比較しています-Accessのテーブルの単純な共通部分により、ワンクリックで同様のコマンドと異なるコマンドを取得できます。 最も難しいのは、さまざまなコマンドを使用することです。コマンドを手動で分析し、構文エラーをアプリケーションコマンド、ユーザーが作成したコマンドなどから分離する必要があります。 多くの場合、結果は驚くべきものです（図4を参照）-統計は、プラットフォームのシェアがアプリケーションのシェアを大幅に超えていることを示しています。



図 4.コマンドの予備リストには、アプリケーション呼び出しと構文エラーの両方が含まれる場合があります。



次に、認識されているCADプラットフォームコマンドのリストを、実装されているnanoCAD Plusプラットフォームコマンドのリストと比較できます。これは、Accessの2つのテーブルの共通部分でもあります。 その結果、1対1モード（完全一致）で実装されたコマンドのリストを取得します。



分析結果

最後に、残りのチームのリストは3つの部分に分かれています。

  1. 実装された機能 、つまり nanoCAD Plusのコマンドラインを経由しないコマンドですが、それでもプログラムインターフェースを介して実装され、ユーザーは実際にはそれらの不在に気付かないでしょう。 たとえば、「PDFATTACH」コマンド（PDF基板の挿入）は現在nanoCADコマンドリストに登録されていませんが、挿入\基板...メニューからアクセスできます。
  2. 代替機能 、つまり 設計機能により、他のCADシステムの類似チームとは異なるが、ユーザーは類似またはさらに高いパフォーマンスでタスクを解決できるチーム。 たとえば、nanoCAD Plus 7.0の環境でのテーブルの操作は、ロシアの設計標準と、現在のDWG図面からのデータ収集操作のために強化されています。
  3. 機能が実装されていない 、つまり 現在nanoCAD Plus 7.0で動作していないチームであり、残念ながら、それらを置き換える代替手段はありません。 これらの機能の一部は開発中（バインダー、ツールパレットなど）、実装には分析と追加の承認が必要、一部は明らかにオプションです（「360ウェブサイトの呼び出し」など）。


分析の観点から3番目の部分は最も困難です-各チームを分析し、それが何をするのかを明確にする必要があります。nanoCADPlusプラットフォームに代替手段があります（または既に実装されている場合があります）。 。 これは手動の分析作業ですが、その結果、優れたスピーキングチャートが授与されます（図5）。



図 5. nanoCAD Plusコマンドのリストを、顧客の1人が使用する西部CADプラットフォームのコマンドのリストと明確に比較する図。



しかし、これだけではありません。呼び出したコマンドのリストを作成しただけでなく、呼び出しの頻度に関する統計も収集したことを思い出してください。 そして真実は、1つのことは「コマンドが実装されているかどうか」という質問に対する答えであり、まったく異なるものです-「呼び出されたコマンドは使用されていますか？」 そして、ここで結果を伝えることもできます（図6を参照）-nanoCADに実装されていない機能のシェアは1〜3％に低下する可能性があります！ つまり、ほとんどの場合、ユーザーは機能面でnanoCADとWesternソリューションの違いに気付かないでしょう。



図 6.そして、呼び出しの頻度でチームを比較すると、状況はさらに説得力があります。



結論の代わりに

私が最初にそのような結果を得たとき、私はショックを受けました。 現時点では、nanoCADプラットフォームは機能的な観点で人気のある西欧のソリューションを非常に自由に置き換えることができます。97-99％の場合、ユーザーは仕事用の代替ツールを受け取ります。 これまでに、5つの組織で同様の分析を実施しました。チームの比率はほぼ同じであり、疑問に思うことはありません。



ただし、分析結果は組織ごとに大きく異なる可能性があると思います。 すべての資料をあなたと共有したいのはそのためです。企業で分析を行い、「nanoCAD Plusの代替品をインポートすることはできますか？」という質問に答えましょう。 私が持っているもの：

  • ログファイルを処理し、CADプラットフォームのコマンドに関する統計を収集するStatCAD 3.01ユーティリティ 。 実行可能モジュール（x32およびx64）とAutoItスクリプト（ソースコード）の両方を共有します。
  • 現在知られているコマンド 、システム変数、nanoCADプラットフォームとよく知られている西部CADプラットフォームの両方の略語のリストを含むExcelテーブル 。 さらに、このテーブルには、見つかったさまざまなチームのリストと、それが属するチームのグループ（代替、実装済み、未実装）のトピックに関する分析が含まれています。
  • ユーティリティStatCAD 3.01を実行できるログファイルのあるフォルダーの例 。 もちろん、ここに収集したログファイルを置く方が良いです。
  • ExcelおよびAccessドキュメント分析テーブル -StatCAD 3.01ユーティリティの結果をCLIENT_CMD.xlsxファイルに配置し、Accessでテーブル02ACAD_CROSS、02ACAD_NOTFOUND、03ACAD-NCAD_CROSS、04ACAD-NCAD_NOCROSSを開くだけです-テーブルの名前がそれを物語っていると思います。


収集された統計を自分で分析できない場合は、ログファイルを受信させていただきます。 または、使用するコマンドのリストを含むExcelスプレッドシート（​​添付のアーカイブからCLIENT_CMD.xlsxで埋められます）。 この場合、自分で分析を実行し、その結果を使用してnanoCADプラットフォームを開発できます。 私を信じて、これは私たちにとって非常に有用な情報になるでしょう-これらのデータに基づいて、私たちは製品をさらに開発し、あなたは便利なツールを手に入れます。
