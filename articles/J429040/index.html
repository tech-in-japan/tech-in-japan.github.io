麻酔なしで本番環境にJavaコードをパッチする





ここでは、Odnoklassnikiプロジェクトのさまざまなサービスの開発に役立つ多くのツールの1つのデバイスについて説明します。 社内では「ホットコード置換」（HCR）と呼ばれ、このツールは、運用サービスを停止することなく実行する際の重大で単純なバグを修正するように設計されています。 これは非常に重要な機能です。これは、修正された新しいバージョンのジャンクサービスをレイアウトするというかなり退屈で時間のかかるプロセスを回避し、各ホストの可用性のアテンダントが十分に長い間停止するのを回避し、キャッシュのフラッシュを回避できるためです。



一般に、多くの時間を節約し、エラーが検出された瞬間から修正までの間隔を数時間から数分に短縮します。 ほとんどの場合、計画どおり、コードの軽微なエラーが修正されます。たとえば、プログラマーがnullをチェックするのを忘れ、一部のユーザーはサイトでの特定のアクションによってエラーが発生します。 つまり、メソッド内の複数の行を変更して修正を実行する場合です。 また、このような小さな変更のために、同僚の注意をそらし、生産のために何時間も手間をかける必要がなくなりました。

もちろん、次の更新を待たずに、はるかに多くの変更を加え、同時に新しいクラスを追加し、マネージャーが同時に要求する変更をすばやく行うことができます。 しかし、これはすでに、ムッシューが倒錯について多くのことを知っているならです。



さらに、互いに「パッチ」を無限に配置することができます。



ただし、このツールは全能ではなく、Javaクラスが提供する標準機能java.lang.instrument.Instrumentationとそのメソッドvoid redefineClasses（ClassDefinition ... definitions）に基づいています。



Instrumentation.redefineClassesは、以前にロードされたクラスを新しいバイトコードに置き換えます。 異なる依存関係を持つ複数のクラスを同時にオーバーロードできます。 オーバーロードは、既存のクラスインスタンスを変更せず、継承を変更せず、インスタンスまたはクラスフィールドに影響しません。 メソッドの本体、定数と属性のプールのみを変更できます。 新しいクラスまたはサブクラスを追加できます。 メソッドシグネチャ、インスタンスフィールド、およびクラスフィールドは変更できません。 互換性のない変更を行おうとすると、redefineClassesは原則として機能せず、エラーをスローします。 クラスがオーバーロードされると、コードのオーバーロードされたセクションの実行が中断されず、次回同じメソッドが呼び出されたときに新しいバイトコードが使用されることに注意してください。 したがって、内部に無限に長いサイクルがあるメソッドのコードを修正しようとすると、実際の置換はこのサイクルが終了した後にのみ発生します。



非常に単純な場合：メソッドとポイント内でのみコードを変更できます。

主な困難は、確立されたすべての作業プロセスに適合するツールであるOdnoklassnikiエコシステムで機能するツールを作成することでした。 これは、何百ものホスト上のすべてのサービスと一貫して透過的に対話するだけでなく、柔軟で使いやすいものです。 このツールは、本番環境で継続的に発生する多数の実験、作業、更新に対処する必要があります。



開発者/管理者の観点から、本番環境でバグを修正しようとするパッチをインストールするプロセスはどのように見えますか？ コード内のエラーを見つけて修正するプロセスは省略します。

TeamCityがパッチビルドプロセスを起動します。 最初に、指定されたブランチからプロジェクトアセンブリが作成され、次に、新しいアセンブリが実稼働環境にインストールされたアセンブリと比較されます。 これを行うために、アーカイブからすべてのファイルを抽出し、矛盾を比較し、変更または追加されたファイルのみを選択するビルドツールのプラグインを作成しました。 この場合、両方のアセンブリのJavaコンパイラのバージョンは同じである必要があります。 コンパイラの別のバージョンでは異なるファイルが作成され、ほぼすべてのプロジェクトファイルがパッチに含まれます。 必要なファイルのみが取得される小さなアーカイブを作成することは非常に重要です。 これにより、パッチを多数のサーバーに配信するプロセスが大幅に高速化されます。 ビルドプロセスは、プロジェクトコードのパッチに適しているだけでなく、プロジェクト内のパッチを適用したライブラリを置き換えることもできます。 2つのアセンブリの内容を比較すると、ライブラリ（jarファイル）に違いが見つかります。



3.ビルドが成功した場合、パッチは特別なリポジトリに送信され、結果ウィンドウでキー（またはハッシュ）が発行されます。これは、パッチを一意に識別するために必要です。







何度も何度もパッチを適用でき、同じバージョン番号のビルドはハッシュが異なります。



4.次に、すべてのアクティビティが設定サービスに転送されます。通常のUIでは、どのサービス、どのホスト、どのバージョンのアプリケーションにパッチを適用するかを指定できます。







このように豊富なパラメーターは、必要なレベルの設定の柔軟性を提供します。これは、多くのサーバーの大きな動物園で非常に重要です。 サーバーの一部では、アプリケーションのバージョン番号が異なっており、このコードにパッチを当てる必要はまったくないとしましょう。 または、検証のために、Hot Code Rreplaceが最初に単一のサーバーまたはサーバーのグループで起動され、次にアプリケーションのすべてのインスタンスに分散されます。



5.構成の変更により、選択したサービスは、インストールする必要のあるパッチの種類、バージョン、および検証ハッシュに関する情報を受け取ります。 考えは、すべてのサービスが「パッチをインストール」コマンドを受信し、それから独立して動作するということです。 独自にバージョンを比較し、バージョンが一致し、パッチのハッシュが欠落しているか異なる場合にのみ、リポジトリからパッチアセンブリを個別にダウンロードします。 ダウンロードプロセス自体はHTTPを介して行われ、リポジトリアドレス、ダウンロード試行回数、および再試行間の待機時間をすばやく変更できます。



6.各アプリケーションは、アセンブリのハッシュをローカルで検証し、展開します。 同時に、各ファイルは、Instrumentation.getAllLoadedClasses（）によって返された配列の中で配列内に存在するかどうかをチェックし、すべての新しいクラスとファイルは新しいクラスと一時クラスパスに書き込まれます。 redefineClassesメソッドをパススルーします。



7.プロセス全体：アプリケーションのパッチ適用の必要性に関するシグナルの到着、そのダウンロード、検証、解凍、およびアプリケーションは、アプリケーションの一般ログと独自のログの両方で詳細に記録されるため、不必要なジェスチャーなしでプロセスを監視できます。



8.パッチの正常な適用後、プロセスは、パッチのハッシュを含む特別に構成された行を追加することにより、アプリケーションのバージョンをパッチが適用されたバージョンに変更することで終了します。 ホストのバージョンが予期したバージョンに変更されていない場合は、そのホストのホットコード置換ログに移動し、そこで起こったことを確認します。 これらが通信の問題である場合は、パッチコマンドを安全に繰り返すことができ、目的のホストが再試行します。
