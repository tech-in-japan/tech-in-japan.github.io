VMware Virtual Volumes-展開できませんか？

VMwareは、ソフトウェアストレージと仮想ボリューム（VVol）のサポートを備えたvSphere 6.0のリリースにより、新しいストレージコンセプトに関する長く骨の折れる作業を完了しました。これは今日の投稿です。 メーカーの公式定義から始めます。「仮想ボリュームは、仮想化環境向けに最適化され、インフラストラクチャではなくアプリケーション機能に焦点を当てた効率的な運用モデルを作成することにより、SANおよびNASアレイの仮想化を可能にする新しい統合および管理構造です」



VMwareは、主要なインフラストラクチャコンポーネントの2つの異なるオプションをサポートするために、つまりダブルワークするためにスプレーされることはないため、将来的にはVMFSに代わる仮想ボリュームになると思われます。 まず、ESXiのリリースから数年後にESXを廃止し、次にVSAがVSANを置き換えてから、vSphere ClientからWeb Clientにユーザーを移行しようとしています。 ある時点で、Windows上のvCenter ServerからVCSAに切り替えることを決定するかもしれないと思います。 したがって、同様の運命がVMFSを待っていると想定できます。VMFSは、VVolsを支持して放棄されます-もちろん、「成熟」の特定の段階に達した後、大多数のユーザーが使用を受け入れます。 このアイデアは、VVolsのサポートがvSphereのすべてのエディションに含まれているという事実によっても確認されています。



ただし、まず、専門家がVirtual Volumesの実装に急がないことを勧める理由を確認しましょう。







理由＃1。 ストレージ複製アレイはサポートされていません



これは、特にストレージアレイのレプリケーションがデータ保護およびリカバリ戦略の基本である大規模な組織にとって、非常に深刻な理由です。 VASA 2.0仕様では、Vaultレプリケーションのサポートはまだ発表されていないため、SRMとvMSCはVVolで動作しません-誰かにとっては、これ以上読む必要はありません。 ただし、VMwareはストレージアレイのレプリケーションをサポートしていませんが、一部のメーカーはそのようなサポートを計画またはすでに提供しているため、回避策がいくつか存在します（ただし、VMwareが仕様に含めるまでSRMまたはvMSCで使用することはできません） VASA）。



理由番号2。 これは、新しいソフトウェアストレージアーキテクチャの最初のバージョンです。



はい、開発に何年も費やしましたが、これはバージョン1.0のみであり、他の製品の最初のバージョンと同様に、「成長する痛み」の影響を受けます。 さらに、相互運用性（つまり、レプリケーション）のサポートがある程度不足しています。 実際のストレージアレイからのサポートについては（製造元からもかなりの人件費が必要でした）-初期バージョンでも出てきます。 アーキテクチャの開発に参加したメーカーの中には、HP、NetApp、およびDellを尊敬している人たちがいます。彼らは数年にわたってこれを行ってきたので、使用準備の面で最も成熟している可能性が最も高いのは彼らの決定です。



理由番号3。 すべてのストレージベンダーがVVolを完全にサポートしているわけではありません



現在、VMwareハードウェア互換性ガイドでVVolのDay 1サポートを宣言しているメーカーは4社あります。 もちろん、このリストは、他の製造会社が開発を完了し、VVolsのVMware認定を受けるにつれて拡大します。 現在、メーカーは、VVolのサポートされる機能の範囲を拡大するのに忙しい。



理由番号4。 データセンターへの影響範囲の闘争で誰が勝者になるでしょうか？



ネットワーク管理者に最初にvSwitchについて話し、vSphereのネットワーク要件を表明したときのことを覚えていますか？ 疑いもなく、その男は手から力が抜け落ちるのを感じた。 おそらくあなたは世界を壊すことさえできなかったでしょう:)。 ストレージ管理者との会話でVVolについて話し始めると、今日も同様のシナリオが発生する可能性が高いです。 しかし、歴史が示すように、「抵抗は役に立たない！」-そして、vSphere管理者はネットワーク管理者との戦いに勝ったので、今ではストレージ管理者も倒す可能性があります。 戦いをせずに協力するために、仮想インフラストラクチャとストレージの関係が大幅に改善されるという事実から、VVolが何であるか、どのように機能し、どのように人生を簡素化するかを説明し、協力することをお勧めします。



理由番号5。 このアーキテクチャはどれくらいスケーラブルですか？



VMFSと比較してVVolがどれだけスケーラブルであるかを確認するのに十分なパフォーマンス統計がまだありません。 この点で、それらがより良くなるか、より悪くなるか、同じになるか。 このソリューションは、スケーラビリティの点でVMFSとほぼ同様であるか、それよりもわずかに優れていると思われます。 厳密に言えば、VVolは、VMFSのパフォーマンスを向上させる手段としてではなく、ソフトウェアストレージのまったく新しいアーキテクチャであり、ストレージアレイのストレージユニットとしての仮想マシン向けの新しいアプローチとして開発されました。 RDMとVMFSの比較についても同じことが言えます。パフォーマンスを改善することではなく、VMwareはパフォーマンスが同等であることを証明したので、これはVVolでも起こることを完全に認めています。 VMwareが現在、パフォーマンスを比較するためのベンチマークを実行していることを示す証拠があり、結果がまもなく利用可能になることを願っています。



理由No. 6.ストレージアレイを操作する機能には、個別のライセンスが必要です



長い間、VMwareはvSphere内でこの機能を模倣しようとして、「シンディスク」、仮想マシンのスナップショット、Storage vMotion、Storage I / O Controlなどを提供していました。 VVolの導入によりストレージアレイが再び使用されるようになるため、これらの機能の多くは将来廃止されます。 つまり、仮想マシンのスナップショットを削除するプロセスは、ストレージアレイのスナップショットの作成に自動的に変わります。 同様に、ストレージリソースを動的に割り当てるために「シンディスク」が使用されるようになった場合、将来的にストレージアレイがこれを実行します。 これにより、これらの操作をより迅速かつ効率的に実行できるため、さらに優れています。 ただし、これらの機能を使用するには、ストレージアレイに対応するライセンスが必要です。 製造元によっては、サポートされているメインセットにこの機能が含まれている場合とそうでない場合があるため、特定のケースでこれらの機能をサポートするコストを明確にする必要があります。



理由番号7.バックアップソリューションプロバイダーはいつ強化されますか？



VVolの実装は、プロトコルエンドポイントとVASAストレージプロバイダーに対応するため、「Direct to SAN」バックアッププロセスのアーキテクチャに影響を与えます。 現時点では、バックアップソリューションプロバイダーからのこのアプローチの大規模なサポートに関する情報はまだありませんが、近い将来、もちろんそれらが表示されます。 バックアップソリューションがVVolをサポートする時期をベンダーに問い合わせてください。 たとえば、Veeamは、最近リリースされたVeeam Availability Suite 8.0 Update 2でvVolをサポートしました（詳細については、 こちらをご覧ください ）。



今、ポジティブについて。



VVolの実装を急いで行うべきではない理由のリストは非常に深刻に見えますが、他のすべての点でVVolはvSphereのストレージに最適なアーキテクチャソリューションであり、多くの利点を提供します。 そのため、VSANが仮想マシンレベルでストレージポリシーベースの管理（SPBM）を実装できるように、VVolは共有ストレージに同じ機会を提供します-もちろん、メリットはそれに限定されません。

そこで、 「仮想ボリュームの迅速な実装のために」引数のリストを検討することを提案します。



理由＃1。 常に選択肢と柔軟なオプションがあります



VVolへの「完全かつ取消不能な」切り替えは完全にオプションです。VMFSまたはNFSとともに仮想ボリュームを使用できます。 VVolsは、アーキテクチャのもう1つのオプションです。 Storage vMotionを使用して、仮想マシンをVMFS / NFSからVVolに、またはその逆に簡単に移動できるため、仮想ボリューム上でストレージコンテナーを構成し、必要に応じてマシンを作成または移行できます。



理由番号2。 愛好家と一緒に新しいテクノロジーをマスターしましょう



ESXから他のESXiに遅く移行し、遅かれ早かれ作業の違いとニュアンスを研究しましたか？ vSphere Web Clientに長い間慣れていませんでしたか？ この「レーキ」は過去のままにしておき、VVolが「任意強制」モードでマスターされるのを待ちません。 新製品に慣れるのが早ければ早いほど、他の人がこの方向への最初のti病なステップを始めたとき、あなたはより自信を持って感じるでしょう。 そして、専門家として行動することは名誉と喜びです。なぜなら、あなたが彼らに与えた知識が役に立つとき、それは素晴らしいからです。



理由番号3。 再び空きディスク容量！



vSphere 5.0では、VMwareがUNMAPコマンドで未使用のディスクスペースの自動復帰を導入したため、エラーが発生したため、この操作の自動化を放棄して手動で実行する必要がありましたが、VVolのおかげで自動化が再び可能になりました。 仮想マシンの削除または移動を完全に理解しているストレージアレイは、その直後に未使用のスペースをすぐに戻すことができます。 esxcliを手動で実行する必要はなくなりました。すべてがVVolで自動的に行われます。 さらに高いレベルの粒度で場所を返す必要がある場合でも、ゲストOSでUNMAPコマンドを実行できます。 このアプローチにより、ストレージアレイの使用効率が向上します。



理由番号4。 vSphereのすべてのエディションでの可用性



ストレージアレイがVVolをサポートしている場合、vSphereの別のバージョンに切り替える必要はありません。すべてのエディションがVVolで動作します。 VVolsのセットアップは簡単なので、それは抑止力ではなく、逆になります。



理由番号5。 ストレージポリシーベースの管理への移行を促進



ストレージポリシーベースの管理を既に展開しているVSANユーザーに対応します。 インフラストラクチャ管理に対するこのアプローチにより、ストレージアレイの機能に応じてストレージ管理ポリシーを構成できます。したがって、ストレージシステムのリソースと機能の使用は、仮想マシンに直接指向されます。 ポリシーベースの管理により、仮想マシンのニーズに応じてリソースを割り当てたり、これらの割り当てのコンプライアンスを監視したりできます。 VMwareは、この管理メカニズムの作成に多大な労力とお金を費やしました。これらはすべて、仮想マシンにストレージリソースを割り当てる際の最大の効率と柔軟性を確保するためです。



理由＃6。 ストレージアレイ機能の最適な使用



ストレージアレイは、実際には、読み取り/書き込み操作を実行するためのメカニズムであり、この特定のメカニズムで動作するように設計されたソフトウェアです。 一緒に、それらはデータを扱うための強力なツールです。 VMFSおよびvSphereストレージの使用は、ストレージアレイ自体に何をいつ実行するかを伝える何らかの「中間」がまだ存在することを意味します。 もちろん、そのようなチェーンは最も効果的な作業方法ではありません。 このような「中間者」を排除し、ストレージシステムアレイを自由に制御できる可能性を考えると、最大の効率と最適な結果を得ることができました。 また、ストレージアレイの機能は、同等のvSphereストレージシステムよりもはるかに広く、さらに、アレイはvSphereよりもストレージリソースに関してはるかに透過的です。 当然、ストレージアレイを使用した実装では、動的プロビジョニング（シンプロビジョニング）、スナップショット、実際のQoSなどがはるかに効率的です。



理由番号7。 VVols管理には特別なスキルは必要ありません



VVolを使用するために、ストレージ管理者として再トレーニングする必要はありません。 平均的なIT管理者はVVolの管理タスクを処理できます。VMware管理者の場合、SPBMとVVolのvCenterとの統合により単一の管理ツールが提供されるため、ストレージ管理コンソールを再度開く必要はありません（ポリシーと仮想ボリュームを制御できます） VMwareインターフェイス）。



理由番号8。 みんなのための1つのアーキテクチャ



ファイルまたはブロック？ NFS、iSCSI、またはファイバーチャネル？ VVolはすべてのストレージプロトコルに統一されたソフトウェアストレージアーキテクチャを実装し、いわば、「vSphereとの通信レベル」を1つ実装しているため、まったく違いはありません。 VVolを使用する場合、VMFSとブロックまたはNFSとファイルが存在するかどうかは関係ありません。現在、VVolとストレージプロトコルで使用される多くのユニバーサルコンポーネントがあります。 もちろん、各プロトコルには独自の特性がありますが、VVolsでは、すべてのプロトコルの一意ではなく同一のプロパティを使用するため、「1つの櫛の下ですべてを櫛形にする」ことができます。



理由番号9。 ストレージユニット-仮想マシン



そして、デザートのために、おそらく「おいしい」理由：ストレージアレイが個々の仮想マシンで動作できるようになりました。 VMFSを使用すると、「透明度」はレベルが低くなるのではなく、LUNレベルに広がります。このため、特定のVMFSボリュームにどのマシンがあるのか​​わかりませんでした。 VVolの導入により、パラダイムは変化しています。仮想マシンがストレージユニットになりました。 たとえば、1台のマシンのアレイスナップショットを作成する場合、これは非常に実現可能です（一方、VMFSでは、LUN全体のスナップショットを作成する必要がありました）。 特定のTier-1仮想マシンにQoSポリシーを割り当てる必要がある場合、この機能は仮想マシンレベルで機能するようになったため、これも実行できます。 したがって、ストレージアレイの使用という点では十分ではなかったVMFSの概念から離れています。 これで、VMFS用の巨大なスペースを予約することなく、必要に応じてリソースをストレージに割り当てることができます。 これは、VMwareがVirtual Volumesで具体化した主なアイデアでした。



もちろん、誰もが自分の状況に対する理解に基づいて、仮想ボリュームの使用を決定します。 （まだ、VVolsに切り替える場合ではなく、「VVolsに切り替える場合」ではなく、「VVolsに切り替える場合」と表示されることを忘れないでください。）新しいアプローチには大きな利点があることは間違いありません。仮想マシンの場合-ただし、決定を下す前に、VVolの実装がインフラストラクチャの現在の作業とその開発の見通しとどのように整合しているかを慎重に分析する必要があります。 既に述べたように、NFSまたはVMFSと一緒に仮想ボリュームを使用して（つまり、「部分に象がいる」）新しいアーキテクチャへの移行を徐々に行うことができるため、歓迎します。
