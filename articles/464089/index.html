Automating HTTP requests in the context of Spring

Background



A few months ago, the task was to write an HTTP API for working with a company product, namely to wrap all requests using RestTemplate and then intercepting information from the application and modifying the response. An approximate implementation of the service for working with the application was as follows: 







if (headers == null) { headers = new HttpHeaders(); } if (headers.getFirst("Content-Type") == null) { headers.add("Content-Type", MediaType.APPLICATION_JSON_VALUE); } HttpEntity<Object> entity; if (body == null) { entity = new HttpEntity<>(headers); } else { entity = new HttpEntity<>(body, headers); } final String uri = String.format("%s%s/%s", workingUrl, apiPath, request.info()); final Class<O> type = (Class<O>) request.type(); final O response = (O)restTemplate.exchange(uri, request.method(), entity, type);


... a simple method that accepts the type, body, and request headers. And everything would be fine, but it looked like a crutch and not very usable in the context of Spring.







And while fellow colleagues wrote functional in their branches on the old mechanism, I had the most ingenious idea - why not write these requests ‚Äúin one line‚Äù (like Feign).







Idea



We have a powerful Spring DI container in our hands, so why not use its functionality to the full? In particular, initializing Data repositories using the Jpa example. I was faced with the task of initializing a class of an interface type in the context of Spring and three solutions for intercepting a method call as a typical implementation - Aspect, PostProcess and BeanDefinitionRegistrar.







Code base



First of all, annotations, where without them, otherwise how to configure queries.







1) Mapping - annotation identifying the interface as a component of HTTP calls. 







 @Target(ElementType.TYPE) @Retention(RetentionPolicy.RUNTIME) @Documented public @interface Mapping { /** * Registered service application name, need for config */ String alias(); }


The alias parameter is responsible for assigning the root routing of the service, be it https://habr.com , https://github.com , etc.







2) ServiceMapping - an annotation that identifies an interface method that should be called as a standard HTTP request to the application, from where we want to get a response or perform some action. 







 @Retention(RetentionPolicy.RUNTIME) @Target({ElementType.METHOD}) @Documented public @interface ServiceMapping { /** * Registered service application route */ String path(); /** * Registered service application route http-method */ HttpMethod method(); Header[] defaultHeaders() default {}; Class<?> fallbackClass() default Object.class; String fallbackMethod() default ""; }


Options:







  • path - request path, example alias + / ru / hub / $ {hub_name};
  • method - HTTP request method (GET, POST, PUT, etc.);
  • defaultHeaders - static request headers that are immutable for a remote resource (Content-Type, Accept, etc.);
  • fallbackClass - a class of rejection of a request that was processed with an error (Exception);
  • fallbackMethod - the name of the class method that should return the correct result if an error occurred (Exception).


3) Header - annotation identifying static headers in request settings 







 @Retention(RetentionPolicy.RUNTIME) @Target({ElementType.ANNOTATION_TYPE}) @Documented public @interface Header { String name(); String value(); }


Options:







  • name - title of the header;
  • value - the value of the header.

I‚Äôll briefly explain what this implementation of the bin object does:







  • provides storage of meta-information of interface methods with resource request settings, such as the methods identified by annotations themselves, reject classes, a collection of routing settings models;
  • provides interception of a method call in the application context using CGlib (MappingFactoryBean # getObject ()), i.e. formally, there is no implementation of the called method, but the method is intercepted physically and, depending on the parameters, the annotation and method arguments, the HTTP request is processed.


The third step is to implement the low-level DI component of the Spring container, and specifically the ImportBeanDefinitionRegistrar interface.
