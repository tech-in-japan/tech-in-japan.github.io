なぜチャットボット、またはBitrix24のストーリーが必要なのか

ファン用のボットを作成してチャットする場合を除き、Microsoft Bot Frameworkを使用できる理由と、自然界のプロジェクトでチャットボットを使用する実際のケースがあるかどうかがよく聞かれます。 だから、彼らは存在します。 パートナーの1つであるBitrix24に、ボットをプロジェクトに統合した経験と技術的な詳細を共有するように依頼しました。 システムの開発者であるSergey Pokoevに話を聞きます。SergeyPokoevは、そのアーキテクチャと、Skypeに接続するためのBot Frameworkの使用について説明します。







背景



会社と顧客とのコミュニケーションは長い間変化してきました。電子メールや電話に置き換わり、最初にソーシャルネットワーク、次にメッセンジャーが登場しました。 同意して、今でははるかに便利になりました。そして最も重要なことは、Facebook、VKontakte、What's App、またはTelegramでサポートの質問をより速く尋ねることです。



豊富なチャンネルは素晴らしい機会を提供するだけでなく、いくつかの問題も生み出します。



  • 売上データの説明と分析は困難です。
  • 人事ミスは顧客の不満につながります。


したがって、多くの企業は、データの集約を1か所で考えています。これは、制御と管理が簡単です。 彼らのために、Bitrix24 *サービスにはOpen Linesツールがあります。これについては以下で説明します。



ところで、 「チャットボットが必要なのはなぜですか」という多くの心配の質問にすぐに答えます。私たちの場合、使用は解決する必要のあるタスクに依存します。複雑-カスタマーサポートを自動化できます。



*ヘルプ： Bitrix24サービスは、会社の業務を整理するためのインターネットツールのセットです。 これには、タスクおよびプロジェクト管理、オムニチャネルCRM、会社の内部メッセンジャー、ワーキングソーシャルネットワーク、勤務時間管理、ドキュメント管理などの機能が含まれます。



仕組み：ユーザービュー



クライアントは、ソーシャルネットワークまたはインスタントメッセンジャーで会社の公式アカウントに書き込みます。 このメッセージは、すべてのチャネルに共通の処理キューに到着します。 それから、メッセージはBitrix24メッセンジャー内で応答する従業員間で配信されます。 理解できるように、これは自動化されたプロセスであり、レスポンダーにとって、クライアントが作成したチャネルに違いはありません。







仕組み：開発者からの外観



Microsoft Bot Frameworkとの統合



Microsoft Bot Frameworkは、クラウドでインテリジェントなチャットボットを作成できる環境であり、SkypeやTelegramからSlackやSMSまで、さまざまな通信チャネルを介して通信できます。 プロキシとしても使用できます。 これらのチャネルごとに個別に統合を実装する必要がないため、これは便利です。



Microsoft Bot Frameworkとの統合のために、既存のOpen Lines作業スキームがいくつかの改善とともに使用されました。



特別なコネクタサーバーを作成しました。 これは、ソーシャルネットワーク、インスタントメッセンジャー、オンラインチャット、サイト上のフォームなどの外部通信チャネルとBitrix24クライアント間のリンクです。 サーバーの主な利点：



  • ソーシャルネットワークおよびインスタントメッセンジャーからBitrix24へのメッセージ配信を保証します。
  • Bitrix24クラウドとボックス版の両方で動作します。
  • ソーシャルネットワークまたはメッセンジャーの会社の公式アカウントをBitrix24にすばやく接続できます。








Bitrix24クライアントとコネクタサーバー間のデータ交換は、独自のプロトコルを使用して実行されます。 この場合、サーバーはルーターとして機能し、着信データと発信データの予備処理を行います。



サーバー環境の要件はBitrix24と若干異なります：PHP 5.4以降、MySql 5.5.3、InnoDB。



Bitrix24クライアントとコネクタサーバー間のデータ交換中のオーバーヘッドを最小限に抑えるために、次のことが決定されました。



  • 外部チャネルおよびサーバーからメッセージに添付されたファイルを転送しないでください。 サーバーは、外部ソースからのファイルへのリンクのみをBitrix24に送信します。 そして、Bitrix24クライアントはそれらを自分でダウンロードします。
  • Bitrix24からファイルを送信する場合は、リンクを送信するだけでダウンロードできます。 ファイル自体はBitrix24.Diskに保存されます。
  • メッセージの送信後に頻繁に使用されるアクション（メッセージの受信、送信、配信ステータスの送信、メッセージの読み取りなど）の場合、応答を期待せずに接続を閉じます。


コネクタサーバーのメッセージキュー



サーバー上のメッセージ送信キューは、MySQLに基づいて実装されています。 現在、Redisのようなより洗練された柔軟性のあるツールは必要ありません。 データベースベースのキューは、追加のオーバーヘッドを回避し、維持しやすく、現在の負荷に簡単に耐えることができます。



ポータルがしばらく利用できない場合でも、キューはメッセージ配信を保証します。 着信メッセージが処理され、データベースに配置されます。 別のプロセスがデータベース内のメッセージを継続的に選択して送信します。 同時に、複数のメッセージからパケットを形成し、それらをポータルに送信します。



配信確認がサーバーに到着しない場合、この「Open Line」へのこのポータルへの次の送信は、ブロック期間の終了後にのみ行われます。 各試行の後に、ロックの期間は増加します。 最後の試行が失敗した場合、メッセージの送信は12時間ブロックされます。 合計で、ポータルへのメッセージ送信の最初の試行から24時間になります。 最後の試行の後、このOpen Lineのこのポータルのすべてのメッセージが削除され、ロックが解除されます。



サーバーが確認を受信すると、配信されたメッセージはキューから削除され、このオープン回線のブロックが解除されます。 つまり、新しいメッセージはすぐにポータルに送信されます。

オープンラインの仕組み



チャットは、ソーシャルネットワークまたはメッセンジャーのユーザーからのメッセージで始まります。 単一のエントリポイントが使用されます。 これは、Open Lineハッシュを含むGETパラメーターを持つ単一のURLです。 ハッシュを使用して、システムは着信メッセージの送信先を決定します。



接続の一意性



外部システム（グループ、ボット）の同じエッセンスを複数のポータルまたは同じポータル内の異なるオープンラインに接続しようとする場合があります。 そのようなケースが処理されない場合、異常な状況と障害が発生する可能性があります。 そのため、接続の一意性を確認するメカニズムを実装しました。



異なるチャネルの再接続の試行は、異なる方法で定義されます。 Microsoft Bot Frameworkは、システム内の特定のボット識別子の存在を確認します。 現在の接続が唯一のものではない場合（繰り返し）、通常どおり機能し続け、以前の接続のデータはサーバーから完全に削除されます。 同時に、リクエストがポータルに送信され、このOpen Lineコネクタに「緊急」のマークが付けられるため、管理者の注意が必要です。
