ズボンの雲



現在、ネットワークには多数の異なるいわゆる 「クラウド」ストレージシステム。 ほとんどの場合、世界中のどこからでも、モバイルを含むあらゆるデバイスから、一般ユーザーがデータにアクセスできるようにするために作成されました。



クラウドストレージ市場は非常に飽和しています。外国のソリューションと国内のソリューションの両方があります。



たとえば、最も有名な外国のいくつかは次のとおりです。Dropbox.com 、 GoogleDrive 、 Microsoft OneDrive

国内から： Yandex.Disk 、 Cloud Mail.ru



上記にはそれぞれ長所と短所があります。

たとえば、Dropboxは無料で2GBの空きディスク容量しか提供しません。これは、Yandexの10GBの空き容量と比較すると明らかです。ディスクは明らかに小さいですが、Dropboxにはファイルのバージョン管理システムがあり、これは非常に重要です。



Mail.Ruは100GBを提供します（ソフトウェアをインストールした場合は1TBを提供しますが、この「無料」をカバーします）。しかし、ソフトウェアはLinuxコンソールでは動作せず、WebDAVのサポートさえありません。

（UPD。：https://webdav.cloud.mail.ru/に書いたコメントでは、WebDAVのベータ版がありますが、現在どのように機能し、どのようにさらに機能するかは不明です。自己責任で使用してくださいおよびリスク） 。

一部の人にとっては、これらの微妙さは重要ではありませんが、誰かにとっては、たとえばホームルーターのコマンドラインを使用して写真をクラウドにバックアップすることが基本的に必要です。

また、Web-faceを介してMAIL.RUにファイルをアップロードする場合、最大サイズの制限は2GBになります。ネイティブクライアントを使用してアップロードする場合、最大ファイルサイズは32GBになります。 重要かどうか、これは個々の質問です。



個人的に、私はできるだけ多くの空きストレージスペースを必要とし、グラフィカルシェルなしでLinuxのストレージを操作できるようにするユーザーのカテゴリーに属しています。 大きな利点は、モバイルデバイス（スマートフォン/タブレット）からこのストレージを取得できることです。これは、「ズボンの雲」が必要だからです。



そのような基準に適合するものは何ですか？ 無料のもののうち、多くはありません。



Yandex.Diskを使用すると、 10GBのディスクスペースを無料で永久に使用できます。

リポジトリを操作するために、Linuxコンソールで実行されている特別なソフトウェアyandex-diskがあります。

Debian Linuxは1つのコマンドでインストールします。 ここにインストールと使用の説明があります 。

私自身は現時点では使用していますが、アーカイブはすでに10 GBの制限まで拡大しており、まもなく空き容量がなくなります。 もっと欲しい



したがって、私たちの要件は次のとおりです。



100万ドルとモルディブへの飛行機

Mega.co.nz

URL： https : //mega.co.nz/

彼女はただのメガです。 彼らは無料で50GBを提供し、Windows用のクライアントがあります。 Linuxおよびその他すべてについては、クライアントを少し後にリリースする予定ですが、コンソールバージョンが利用可能になるかどうかは不明です。



Megaを操作するための代替アプリケーションがネットワーク上にありますが、それらはすべて不安定に動作し、バグがあり、落ちます。

多かれ少なかれ安定して動作するように見える唯一のものは、このソリューションです：

http://megatools.megous.com/

私のDebian 7で立ち上がって働いた。



ジャンプ/ダウンロードの速度は地理的な場所によって異なる場合があるため、このサービスは外国のものです。Megaでは、アクティビティを制御するようなものがあります。 短時間で数十ギガバイトをドライブする場合、一時的に速度を下げることができます（30分）。

なぜなら 私のストレージは小さなファイルでいっぱいです-私は怖がっていません。

次のように配置されます。

アーカイブをダウンロードします（Debian x64があります）：

wget megatools.megous.com/builds/megatools-1.9.91-debian-wheezy-amd64.tar.gz





開梱します。 システムのルートに解凍すると、必要な場所にすべてのバイナリとライブラリが散在しています。

次に、ストレージ情報を見てみましょう。

megadf -h -u mister-blister@pochta.com -p moyparol1123





次のようなものを出力します：

Total: 50,0



Used: 0



Free: 50,0





ある種の警告がコンソールに注がれている場合：



«dbus-launch --autolaunch=174196e0fb6ad20eef21a078520f2e6b --binary-syntax --close-stderr» 1: Autolaunch error: X11 initialization failed.\n







このコマンドは次のように実行できます。



dbus-launch megadf -h -u mister-blister@pochta.com -p moyparol1123







ファイルのリストを表示するには、次のコマンドがあります。



megals -h -u mister-blister@pochta.com -p moyparol1123







「/ root / mytest」フォルダーからクラウドにファイルをアップロードする場合は、次のようにします。



megaput /root/mytest/* -u mister-blister@pochta.com -p moyparol1123







たとえば、リポジトリにフォルダーを作成することもできます。



megamkdir /Root/mytest -u mister-blister@pochta.com -p moyparol1123







（リモートフォルダーへのパスは完全に書き込まれることに注意してください）



次に、ローカルフォルダーの内容をその上で同期します。



megasync --local /root/mytest/ --remote /Root/mytest -u mister-blister@pochta.com -p moyparol1123







コマンドの完全なリストとその使用例は、公式の人またはhttp://megatools.megous.com/man/megatools.htmlにあります。

そこで、パスワードとログインを自動化する方法を読むことができます（特別なrcファイルが使用されます）



ちなみに、 メガプットチームまたはメガシンクチームがクラウドへのファイルのアップロードを行ったとき、サーバーのCPU（ Intel®Atom （TM）CPU D525）の負荷は1.3-2.0％でした。 比較のため、 Yandex.Diskの同期条件は1.7％ですが、初期化中（明らかに、非同期データのリストを作成する場合）に最大200％の沈下が発生します。 また、定期的に、すべてのディレクトリが既に「最新」である場合、Yandex.Diskは定期的なチェックを行い、 CPUの1〜2％を使用しますが、真実は非常に簡単です。 Yandexの製品に敬意を表して、作業/ダウンロードの速度は依然として非常に印象的です（データセンターはロシアにあるため、トラフィックの移動にオーバーヘッドはありません）

アカウントでの承認後、紹介リンクを作成して友人に配布できます。 彼らが持ってくる友人ごとに、彼らはあなたに+ 5Gbを、各友人に+ 5Gbを +広告ツイート用に2GBを与えます。 私にはとても人道的なようです。 理論的には、この方法でストレージボリュームを非常に大幅に増やすことができます。実際には、代替メールボックスへの参照リンクで登録することで15 GBのストレージを増やすことができました3つの新しいアカウント（さらに多くの「左」アカウントを作成するのにうんざりしています）

Linuxでは、次のように実行します。

アプリケーションでアーカイブをプルします。

wget copy.com/install/linux/Copy.tgz





開梱

解凍されたアーカイブのディレクトリに移動します。内部にはディレクトリがあります。

armv6h/

x86/

x86_64/





32ビット、64ビットのオペレーティングシステムおよびARMのディストリビューションが含まれています。

目的のディレクトリに移動して、コマンドを実行します。

./CopyConsole -u=mister-blister@pochta.com -r=/root/copycom/ -p=moyparol1123





ここで、 -uスイッチの後にログイン（別名電子メール）が書き込まれ、 -pスイッチの後にパスワードが書き込まれ、 -rの後に「クラウド」と同期する必要があるディレクトリが書き込まれます。



コンソールを起動すると、次のようなものが上昇します。



User from cmdline mister-blister@pochta.com

Root from cmdline /root/copycom

Password from cmdline

Starting copy...success

Logged in as user: 'mister-blister@pochta.com' Copy folder is: '/root/copycom'

All files up to date







「すべてのファイルが最新」状態では、サーバー上のプロセス（Intel®Atom（TM）CPU D525）がCPUの0.3〜1％を占めていました。

ファイルがクラウドにアップロードされたとき、負荷は1.7％から5％に「歩き」 ましたが、基本的に1.7-3％の範囲で維持されました
