ロングエントリー



妻と私はしばしば、ポイントAからポイントBにファイルを転送する必要があります。ポイントAは自宅のコンピューターの1つであり、ポイントBはあなたが知っている誰かのコンピューターか、たとえば職場のコンピューター（フラッシュドライブが禁止されている場合があります） ） さらに、フラッシュドライブ（アパートの任意の場所に配置可能）に移動してファイルをコピーするのも非常に面倒です:)一般に、私はこれらの目的で通常webfile.ruやbox.netのようなサービスを使用し、妻は愚かにファイルを送信します自分（または特派員）にメールで。





これらのオプションはすべて、非常に不便です。 ファイルをホームサーバーに保存し、何らかの便利な方法でダウンロードすることを決定しました。 次のオプションを検討しました。

  • FTPを上げます。 しかし、私はFTPが好きではなく、どこでもそれを使用する機会があるわけではありません
  • WebDAVサーバーを上げます。 しかし、これは、第一に、あまり安全ではありません。第二に、FTPよりもWebDAVの方がさらに問題があります。
  • ブラウザからファイルをダウンロードします。 このオプションでは、停止しました




もちろん、ApacheまたはLighttpdを構成してディレクトリのコンテンツを表示し、承認を固定することもできますが、ファイルが多くダウンロードされている場合、ファイルを検索するときに見苦しくなり不便になります。 さらに、特定のファイルへのアクセスを制限することは困難です（覚えているなら、ファイルを自分にダウンロードするだけでなく、友人に転送したかったのです）。



一般に、私はまだプログラマーであり、山から来た西洋ワサビではないと判断し、自分のニーズに合った簡単なファイルホスティングサービスを書くことができました。



プラットフォームの選択



私の人生では、ASP.NETで多くのことを書いてきましたが、うんざりしています。 私はPHPを十分に知らないので、何らかの理由でPHPが好きではありません。 今、私はRuby on Railsで書いていますが、そのようなタスクにはやや怪しいです。 しかし、私はRuby言語が本当に好きなので、Rubyで他のフレームワークを採用することにしました-同時に、私の視野を少し広げる機会があります。 最初は私の選択はRackでしたが、それらが私の要求に対して禁欲的すぎることがすぐに明らかになりました。 SinatraとCampingという2つの小さなフレームワークがノッチを立てています。 理由がわからないので、シナトラを選びました。

そして、ブラウザでlocalhostアドレス：4567を開きます。 出来上がり！ シナトラからこんにちはを見て、喜んでいます。



何を書きましたか？ 最初の2行では、シナトラを接続します。 そして、まさにDSLが始まります。シナトラのアプリケーションは、次の形式のブロックのセットで構成されています。



METHOD PATTERN do

OUTPUT

end









METHODはHTTPメソッド（get、post、put、delete）、PATTERNはURLパターン、OUTPUTはブラウザーの応答を生成するコードです。 ブラウザからリクエストを受信すると、SinatraはHTTPメソッドとURLパターンを使用して、リクエストに対応する最初の適切なブロックを決定し、このブロックからコードを実行し、受信した文字列をクライアントに返します。



PATTERNについて詳しく説明します。 すでに書いたように、これはURLパターンであり、いくつかの方法で設定できます。

  • 定数（例： '/'、 '/ about'、 '/ hello / world'）
  • 名前付きパラメーター（たとえば、「/：id」、「/ post /：id / show」、「/ user /：name /：action」）を持つ定数-この場合、OUTPUTでは、paramsハッシュを介して対応するパラメーターの値にアクセスできます（params [：id]、params [：name]、params [：action]）
  • ワイルドカードを使用する定数（たとえば、「/ select / * / from / *」、「/ load / *」）-この状況では、paramsハッシュには、アスタリスクに一致するすべてを含むparams [：splat]配列が含まれます（最初の場合、たとえば、「/ select / id / from / users」パラメータへの呼び出し[：splat]には['id'、 'users']が含まれ、2番目の場合は、 '/ load / all / my / money'にアクセスするとparams [：splat]は嘘をつきます['all / my / money']）
  • 正規表現（たとえば、 '％r {/ hello /（[\ w] +）}'）-ここでは、params [：captures]の配列を扱っています。




さまざまなパターンの使用例を次に示します。



require 'rubygems'

require 'sinatra'



get '/' do

'Index page has you. Follow the white rabit.'

end



get '/give/me/the/:key' do

"I don't have any #{params[:key]}. Maybe you should try another url."

end



get '/buy/*/in/*' do

"#{params[:splat][1]} doesn't sell #{params[:splat][0].pluralize}"

end



get %r{^/ where / is /(habr)(a\1)?.ru$} do

"You're looking for <a href=\"http://habrahabr.ru\">habrahabr.ru</a>. It's not here."

end

ちょっとしたデザイン



ここで、ファイルホスティングの計画を開始します。 私が最大限にしたいこと：

  1. ファイルをアップロードする
  2. ファイルをダウンロードする
  3. ダウンロードカウンター（統計用）
  4. 不要なファイルを削除する
  5. パスワード保護ファイル
  6. アクセス制限
  7. 取得が困難なアドレス（リンクが/ダウンロード/ 4のように見える場合は、/ download / 3 / download / 5などもあることが明らかです。リンクが/ download / 6a1941fb0cd47のように見える場合は、他のURLを選択しますずっと難しい）

  8. Webインターフェースだけでなく、Sambaを介してファイルをアップロードする機能




この記事の一部として、次の簡易バージョンを検討します。

  • Webインターフェース経由でのみアップロード/ダウンロード
  • Basic-HTTP認証を使用したセキュリティ
  • 簡単なダウンロードリンク

次に、問題があります-ファイルをどのように保存しますか？



最初に、次のソリューションを使用したいと考えました：アプリケーションのディレクトリの/パブリックフォルダにファイルを保存します（Sinatraでは、パターンを記述せずに、/パブリックフォルダのファイルに直接アクセスできます。つまり、ディレクトリのphoto.jpgファイルです。 myapp.rbと同じディレクトリにあるpublicは、/ photo.jpgで常に利用可能です。 この場合、単に「ls ./public」を呼び出して直接リンク経由でファイルを提供することにより、ファイルのリストを取得します。 問題：直接リンク、それらを隠す可能性なし、ロシアのファイル名と特殊文字の問題、統計の欠如など。 一般に、この場合、実際にはApacheディレクトリリストの類似物がありますが、これは望んでいるものではありません。



したがって、結論：データベースが必要です



このような単純なシステムにsqliteよりも複雑なものを使用するのは愚かです。 彼女を連れて行きます。 rubyのデータベースを操作する問題が残っています。 もちろん、ActiveRecordとそのORMを使用することもできましたが、それもかなり難しいものであると判断しました。 そこで、シナトラだけでなくDataMapperも勉強することにしました 。



DataMapperは、Rubyオブジェクトと同様にデータベースレコードを操作できる小さな軽量フレームワークです。



DataMapperのインストールと構成



インストールは簡単です：



sudo gem install data_mapper







アプリケーションに接続し、「シナトラ」を必要に応じて追加します



require 'dm-core' #

require 'dm-validations' # (not_null )

require 'dm-timestamps' #









作業するデータベースの名前を指定します。



DataMapper.setup(:default, "sqlite3://#{Dir.pwd}/mydatabase.sqlite3")







次に、データを格納するテーブルの構造を記述する必要があります。



class StoredFile

include DataMapper::Resource



property :id, Integer, :serial => true

property :filename, String, :nullable => false

property :created_at, DateTime



default_scope(:default).update(:order => [:created_at.desc])

end









ここで行ったことを検討してください。

StoredFileクラス-インスタンスがファイルを表すクラス。

DataMapper ::リソースを含める-StoredFileクラスがDataMapperリソースであることを宣言します（つまり、データベース内のレコードを記述します）。 これは、stored_filesテーブルがデータベースに作成され、それを操作するメソッドを取得することを意味します。 ActiveRecordとは異なり、クラスからは継承せず、モジュールを接続することに注意してください。これは、OOPの観点からはるかに便利です。

次に、ファイルの3つのプロパティ（=データベース内の列）について説明します-id、ファイル名、作成日。 ：serial => trueは列がauto_increment（値を自動的に取得する）であることを意味し、nullable => falseは明らかにファイル名がNULLであってはならず、created_at列はレコードが作成された時間で自動的に満たされます（このためタイムスタンプ）。

最後に、default_scope（：default）.update（：order => [：created_at.desc]）は、デフォルトエントリは作成時間の降順に並べる必要があることを示しています。



StoredFileの説明の後、次のように記述します



DataMapper.auto_upgrade!







この行は、StoredFileクラスが変更されると、DataMapperにデータベース構造を強制的に更新させます。



データを扱う





したがって、次のようにファイルを操作します。アップロード中に、ID.uploadという名前でファイルをファイルフォルダー（IDはデータベース内のファイルID）に保存し、データベースに元のファイル名を書き込みます。 ダウンロードするとき、ブラウザにファイルを与え、データベースからファイル名の値を名前として転送します。 したがって、ファイルシステムには242.uploadタイプのファイルがあり、ブラウザは「Rollback Report 2008.docx」という名前でそれをダウンロードします。
