作業プロセス



React Nativeの問題は1つだけです。AndroidのAndroidの問題です。 これがシステムにとって問題なのか、Androidにボルトを取り付けた開発者にとって問題なのかはわかりませんが、それには常に奇妙な妨害があります。

最も奇妙なケース：フォーマット文字列に変換された日付：「YYYY-MM-DD HH：mm：ss」は、iOSとAndroidで文字の長さが異なります。どのプラットフォームで余分なスペースがありますか？ 激しいペアにつながるだけで、コードは純粋なjs上にあるように見えます。iOSではすべてが正常に動作しますが、Androidでは何かがうまくいかない可能性があります。 したがって、コードを変更した後は常に両方のプラットフォームでアプリケーションをテストする必要があり、これがどのような問題につながるかはわかりません。



実際、私にとっては、何らかの理由でDateがAndroid上で曲がって動作するのと同様の問題はまれでしたが、momentJSは問題ありません。 すぐに後者を使用してください。 しかし、異なる性質の組版の問題があります。

第一に、アンドロイドは、スタイルに示されているシェーディング方法をまったく知りません。 それには別のパラメータがあります-'elevation'ですが、それだけです。 影の色、半径、透明度-これらはすべてアンドロイドを通過します。

第二に、画面解像度。 アップルフォンは、特にプラスで巨大な解像度を持っているため、必要なフォントサイズのボタンを公開すると問題が発生します.iOSではすべてがうまく見えますが、Androidではすべてが固着します-画面は小さくなります。 幸いなことに、react nativeを使用すると、アプリケーションが起動されたプラットフォームを特定できます。これにより、コード内の何か（スタイルなど）が変更されます。



開発プロセス自体に関しては、Androidのネイティブ開発後に得られる自由感を説明するのは困難です。 Android SDKは、特定の何かのために設計されたツールを提供しますが、それは避けられません。 ここでは、クラスがレイアウトにフックされるアクティビティがあり、少なくともあなたがしていること、そしてアクティビティがあるべきだと考えられました。Google自身でさえ、それについて何もできません。 そのため、彼らはデータバインディングを提供し、このライブラリを使用して、ケースの99％のアクティビティは松葉杖として使用され、レイアウトとViewModelを取得し、途中でModelを渡しますが、レイアウトで既にモデルとViewModelを明示的に指定しました。 この場合、ロジックはまったくありませんが、アクティビティがあります。 情報を送信するには、Intentが必要であり、単純なタイプのみを問題なく送信できます。オブジェクトを渡したい場合は、すべてParcelableを送信します。

コンポーネント



特記事項は、コンポーネントに値します。 Webのユーザーにとっては、これは使い慣れたツールですが、携帯電話のユーザーにとっては、これは単なる特効薬です。



ここには3つの異なるボタンがあります。 実際には違いはありません。これは1つのコンポーネントです。 コンポーネントの急峻性の規模を理解していますか？ ここで「完全に」という言葉から、レイアウトのコピーアンドペーストを忘れることができます。 はい、アンドロイドには理論的にコピーアンドペーストから私たちを救うべきスタイルがあります。 しかし、それらは純粋なコンポーネントに適用されます。 はい、ボタンのスタイルを設定できますが、上の写真のようにボタンを作成するには、Androidの1つのボタンでは不十分です。 これは、TextViewとImageViewの両方が存在する完全に独立したレイアウトであり、これらすべてとレイアウトには独自のスタイルパラメーターがあります。 また、3つのボタンはすべて、これらのコンポーネントの数が異なります。写真がない場所、2つのテキストなどです。つまり、Androidでこれらの3つのボタンすべてを3回完了せずに作成することはできません。 さて、反応でこれをどのように行うことができますか？ ここには3つの魔法があります。



  1. 小道具
  2. 存在する要素のみを表示する
  3. スタイルオーバーレイ


すべての詳細をご覧ください。

Androidでは、最初にレイアウトでコンポーネントを作成し、次にアクティビティでID'shnikによってコンポーネントを見つけ、いくつかのパラメーターを渡します。たとえば、テキストを変更します。 これは、表示するものが事前にわかっている場合に機能します。 試薬では、必要なパラメーターを使用して作成する必要があるコンポーネントを示します。 塩とは何ですか？ 小道具では、簡単なものからオブジェクトまで、問題なくすべてを転送できます。 中央のボタンと同じですが、矢印が異なるボタンがもう1つ必要だとしましょう。 アンドロイドでは、4番目のボタンを描画するために実行し、その後、小道具に別のアイコンを渡します。



しかし、ここでさらに魔法が起こっています： 



{true && <Text> </Text>} {false && <Text>  </Text>}


ポイントは、この方法では存在しないオブジェクトを表示できないことです。 動的型付けでJSを使用します。たとえば、次のように自分で便利に使用できます。 

 const text = this.props.text; {text && <Text> {text} </Text>


textパラメーターをpropsに渡した場合、コンポーネントはそれをレンダリングしますが、それを渡さなかった場合、それはしないので、なぜですか？ 動的型付けに再び焦点を当てる価値があります。 すべてはテキストで問題ありません。テキストはtrue、テキストはfalseではなく、オブジェクトと同じです。 しかし、数字の場合、問題は0を送信することです。JSはこれが間違っていると考えます...

ロジックは到達し始めますか？ 私たちは、その中にある可能性のあるすべての可能なオプションを考慮に入れたコンポーネントを作成し、上記で提示したデザインでそれらをラップします。

そして、ここで別の問題が発生する可能性があります-大きすぎるテキスト、大きすぎる画像などは、コンポーネントの外観を損なう可能性があります。 ここでは、アイテム番号3が役立ちます。 



 <Text style={[styles.defaultStyle, this.props.customStyle]}> {this.props.text} </Text>


スタイルをスタイルの配列として定義します。 これはどういう意味ですか？ これは、スタイルを小道具に渡さない場合、コンポーネントはデフォルトになり、事前に決定したことを意味します。 しかし、カスタムスタイル、たとえば上からのインデントに1つのパラメーターを渡すとどうなりますか？ そして、このコンポーネントにはデフォルトのスタイルに加えて上部にパディングがあります。 ポイントは、デフォルトのスタイルが常に完全に適用され、カスタムスタイルで渡すパラメーターのみが置き換えられることです。 すなわち テキストの色のみを変更したい場合-問題ではありません。サイズ、インデント、その他の喜びは消えません。 また、スタイルにパラメーターが必要ない場合は、カスタムスタイルでこのパラメーターに値nullを渡します。
