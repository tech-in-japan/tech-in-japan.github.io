組み込みシステム：Windowsの特別な目的

こんにちは



強大なハブラマンと最も美しいハブラマンたちよ！ わずかな人が、すばらしいWindows EmbeddedのスーラがHabrの知恵の本にまだ記録されていないことを述べました。 価値のない人に、ロシアの鉄道会社の施設で特別に防弾のWindows®™がどのように構築され、実装されたかについての1つの有益な話を聞かせてください。



どうしてそんな人生にたどり着いたのか



そのため、ある朝、プログラマからのメッセンジャーが、ひどく悲しいニュースとともに、私たちの控え目なシステムソフトウェアの住まいに到着しました。 実際、出張中のプログラマーは職場でソフトウェアを更新できませんでした。 それはシャイタンを取り囲み、ウイルス、トロイの木馬、ポルノバナー、および反撃で埋めました（無実のユーザーはそれとは関係ありませんでした）。



このソフトウェアは、VC ++ 6 + MFC + WINAPIで記述された、信号システムの自動ワークステーションです。





私たちの賢いカリフ、そのようなわいせつを学ぶことは、ソファと呼ばれます。 そして、裁判所の賢者とスターゲイザーは言いました：



ステーションにインターネットがないため、ウイルス対策ソフトウェアはインストールされませんでした（ステーションはロシア全土に散らばっており、更新のためにオフラインベースで旅行することはどういうわけかオプションではありません）。 一部のステーションはイントラネットにアクセスできます。イントラネットには企業のウイルス対策がありますが、第一に：まったくない、第二に：アンチウイルスがシトニックで古くて良いです。



一定の数のユーティリティを使用した後、デスクトップWindowsをより耐久性と信頼性の高い、しかしwin32互換の何かで置き換えることにしました。 適切なプラットフォーム全体から、Windows XP Embeddedが選択されました。 その上にすでに成果があり、実際、そのようなクールなシステムがありました。



Windows XP Embeddedは、Windows XP Professional Editionに基づく組み込みコンポーネントオペレーティングシステムであり、産業用オートメーションシステム、ATM、医療機器、キャッシュレジスタ、スロットマシン、VoIPコンポーネントなど、さまざまな組み込みシステムで使用するために設計されています。書き込み保護フィルター（EWFおよびFBWF）、フラッシュメモリ、CD-ROM、ネットワークからの起動、独自のシステムシェルの使用など、追加の埋め込み機能が含まれます。



現在Windows Embedded Standardとして知られています。





ステロイドの窓



Windows Embedded IDEの統合ツールで何ができるかを考えました。

  1. ディスクへの書き込みを無効にする
  2. ユーザーアカウントの権利を制限する
  3. AWP以外を実行する機能をユーザーから奪うため
  4. 未使用のポートを閉じる
  5. 外部USBドライブのマウントを無効にする
  6. ワークステーションの直接作業に必要ではないアプリケーションおよびシステムソフトウェアをシステムから除外する（メディアプレーヤー、Outlook Express、ペイント、エクスプローラーなど）




しかし、それはすべて退屈でシンプルです。 ボスはモジュラーソフトウェア™のアイデアに触発されました。これは、アプリケーション、システムソフトウェア、および構成を「モジュール」に分割することで構成されています。 このアプローチの利点：



  1. ソフトウェア構造がより合理化されます
  2. より便利なソフトウェアバージョン管理システム（この奇跡の言葉をおaびします。これ以上良いアイデアはありませんでした）
  3. 展開/ソフトウェアの更新速度の向上-大きなファイルは、多くの小さなファイルよりも速く書き込まれます
  4. たとえば、システムの柔軟性が向上-システムソフトウェア（OS）は、アプリケーションプログラムに影響を与えずに読み取り専用でマウントできます。




明らかなマイナスはありません、それは思われる-紹介し、喜ぶ。 しかし、QN​​X（元々は産業用コンピューター用にモジュラーシステムが開発されていた）がWindowsでシンプルかつ自然であったという事実は片頭痛の地獄になり、ウニをクレーンで渡そうとするように見えました。 その結果、実装されたことが判明しましたが、最初に最初のものが必要です。



Windowsトリミング


タスクがポイントでどのように解決されたか：

  • ディスクへの書き込みをブロックするには、RAMのデータマッピングでEWFを使用します


高度な書き込みフィルター-ハードディスクへの書き込みアクセスをインターセプトし、設定に応じてバッファリングするドライバー：

  • 特別に選択された非表示セクション
  • RAM内


この場合、保護されたパーティションへの実際の書き込みは発生せず、ユーザーの要求（いわゆるコミット）でのみ実行されます。MicrosoftがAPIと既製のユーティリティewfmgrを提供します。 バッファリングに特別なセクションを使用する場合、RAMにバッファリングする場合ではなく、データは再起動の間に保存されます。



  • 定義済みのWindowsテンプレート-「通常のユーザー」からアカウントを作成し、自動ログオンに入れます
  • ワークステーションをシステムシェルとして実行するシェルタイプのコンポーネント（システムシェル）を作成します。 explorer.exeをシステムから除外します。 特に巧妙なユーザーは、ワークステーションウィンドウの1つのファイル保存ダイアログからディレクトリ構造を表示できますが、

    • これはプログラマーにとっての問題です（hehe）
    • レコードと制限付きアカウントのフィルタリングを忘れないでください


  • なぜなら 私はVBのWHS方言の大ファンではないので、 netsh.exeファイアウォールを引っ張るバッチファイルを使用しました
  • 追加のコンポーネントでそれを行いました-重要なコンポーネント（つまり、ハードウェアの依存関係に応じて自動的に追加されるコンポーネント+ワークステーション作業に必要なインフラストラクチャ）のみを含むXP Embeddedプロジェクトを作成しました
  • 外部USBドライブの切断は野barな方法で実行されました。つまり、USBドライブのドライバーと新しい機器のインストールウィザードはシステムイメージから除外され、未知のPnPデバイスが接続されたときにレジストリキーを介して新しい機器ウィザードの表示が無効になりました



    PnPデバイスを無効にする問題はいくつかの 方法で解決できます （またdevconもあります ）が、これはそのグローバルな性質、つまり 同時に、他の不正な周辺機器はUSBドライブで無効になりますが、USBマウス/キーボードを交換することは可能です（ドライバーはシステムにあり、デジタル署名が含まれているため、そのようなドライバーは自動的にインストールされます）




画像をマウントする


Windowsにイメージをマウントするためのソフトウェアを見つけるのは長く苦痛でした。 RAMドライブの実装のみが登場しました。これは確かにクールですが、そうではありません。 vhdファイルをマウントするユーティリティがWindows Virtual Serverバンドルに含まれていたことを思い出しましたが、WinXPに追加する方法と、それが合法かどうかを開くことができませんでした。 世紀のプロジェクトはほぼ銅の流域で覆われていましたが、Googleで優れたファイルディスクへのリンクが突然現れたとき、どうすれば血を減らして自分のドライバーを開発しないようにするのかと思っていました。 すべての点で機能します-イメージを論理ディスクとしてマウントできます（メディアをエミュレートする方法はわかりませんが、必須ではありませんでした）、ドライバーとして作成され、XP Embeddedをプロジェクトに簡単に追加できます。すぐにディスクを管理するためのコンソールユーティリティが含まれています起動スクリプトで使用するために必要です。 一般的に-百パーセントヒット。



ドライバーはインストールされていますが、イメージ自体はどこで入手できますか？ Windowsではddを使用し、virtual / dev / nullから必要なサイズの画像を作成します。 ファイルディスクを使用してそれらを接続し、FSのない論理ディスクを参照し、それらにNTFSを作成し、コンテンツをそれらに書き込みます。



イメージの1つに、ログオン後に実行するスタートアップスクリプトを配置し、いくつかの設定（構成ファイルからのIPアドレス、画面解像度などの設定）を行います。 ログオン時までに、イメージが既にマウントされていることは明らかです。 そして、ここで私はたくさんのオプションを試してみました-Schtasks 、これはログオンでタスクを実行することができます（しかし、それはどういうわけか常に動作しませんでした）アプリケーションソフトウェア（オプションとして、タスクをスクリプトからサービスに転送することは可能ですが、この魂のないC ++よりもスクリプトの方が何となくきれいです）。 その結果、集合ファーム-AutoExNtはログオンの前にマウントを実行し、システム構成タスクは管理者に代わって暗号化されたcpauタスクを使用して実行されます（自動ログオンは通常のユーザーアカウントで発生するため）。 これは安定して動作する最初のオプションでしたが、その時までにデバッグを行うと非常にひどくなり、メソッドのさをあきらめてそのままにしました。



さらに、彼は小さなシンプルなスタートアップユーティリティを開発しました。 Explorer Shellはプロジェクトから除外され、レジストリの作成者のキーは誰も処理しなかったため、開始時にプログラムを実行する必要がありました。



これで、Windowsシステムソフトウェアに対する私の苦痛は基本的に終了しました。このすべてのがらくたがターゲットマシンにどのようにインストールされるかを考える時が来ました。





展開



Windows XP Embeddedには、通常のWindows XPとは異なり、ドライブをパーティション分割およびフォーマットするためのツールが含まれていません。 つまり XPeのインストールでは、プロジェクトディレクトリ構造をFAT32 / NTFSでフォーマットされ、アクティブとしてマークされたパーティションのルートにコピーします（ブートフラグが設定されます）。 XPeパッケージには、パーティションに必要なツールを含むWinPEディスクが含まれていますが、このディスクの使用シナリオはかなりあいまいです（たとえば、リカバリディスクとして使用することはできません）。 ユーザーに対するこのような懸念を賞賛して、Microsoft製品をいじってXPeを展開しないことにしました（無料のWindows AIKがあるようですが、Microsoft製品のライセンス制限を把握しようとする以前の経験から、側で何かを探すことにしました）。



GNU / Linuxで運を試してみることにしました。 ntfs-3gドライバーはかなり長い間安定しており、それ自体で動作していました（ところで、かなり奇妙な場所で-Realtekによると、マルチメディアプラットフォーム用のSDKでは、外部ドライブのデフォルトファイルシステムとして使用されています）。 Ubuntu LTS Live-CDを取り、ロゴとUbuntuの言及を削除し（Canonicalはシステムの大幅な変更のためにこのような要件を提唱しているため）、カスタマイズを開始しました。 配布の選択は、個人の好みによって決定されました。



デプロイメント環境の最初のバージョンは、zenityを使用してユーザーと通信するbashスクリプトでした。 シンプルで上品。 UIの面で十分に神性がないとして拒否されました。



2番目のバージョンはMono（C＃-企業標準）で記述されており、GTKのグラフィック要素のほとんどすべてが含まれていました。 高等弁務団は、UIがまだ十分に神聖ではないが、まだ実行可能であると指摘した。



インストール手順自体は、外部ユーティリティの呼び出しで構成されていました。マークアップ、ntfsprogsパッケージのmkfs.ntfs（ntfstools）からFSを作成し、rsyncでインストールメディアからデータをコピーします。



次に、再起動し、Windows Embeddedの展開に直接進みます。
