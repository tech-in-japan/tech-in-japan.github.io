Google Glassのアプリケーションを作成します

数日前、Google Glassのアプリケーション開発を徹底的に練習する機会がありました。 私はこれまで「ポイント」のために他のものを開発するつもりはないので、得られた経験は時間の経過とともに混乱します。 まだ新鮮な印象を共有するために、私はこのトピックを書くことにしました。

Google Glassに興味がある人は誰でも、このガジェットのソフトウェア「詰め物」が何であるかを知っていると思います。 はい、これは適応されたランチャーを備えたAndroid 4です。 はい、「眼鏡」では、adb経由でAndroidアプリケーションをインストールすることにより、通常のAndroidアプリケーションを実行できます。 おそらくミラーAPIについてご存知でしょう。これは最近まで、Google Glassのユーザーに公式にサービスを提供する唯一の方法と考えられていました。 以下に、このツールの使用について少し説明します。 しかし、私が話したい主なことは、まだ公式ではないGlass開発キットを使用して、Google Glassの本格的なAndroidアプリケーションを作成する方法です。



それでは、まずはGoogle Glassを作りましょう


革命的なガジェットの選択された所有者の中にいない場合は、絶望しないでください。 ここからランチャーといくつかの関連するapkをインストールすることで、Androidスマートフォンまたはタブレットからほぼ実際のGoogle Glassを作成できます 。 タイムラインカード、​​通常は音声コマンドの認識、Bluetooth、なんとか機能するカメラ（通常はNexus 7でのみ起動可能）サウンド、およびハングアウトを備えた本格的なインターフェースが得られます。 どういうわけかナビゲーションではうまくいきませんでしたが、おそらくもっとうまくいくでしょう。 初めて起動すると、ランチャーは通常のアプリケーションとしてアカウントへのアクセスを要求します。 彼に権利を与え、ほぼ真のGlass Explorerになります。 少なくとも、ミラーAPIを介して自分にタイムラインカードを送信できます。



GoggleがMirror APIをGoogle Glass所有者のみに提供するのはなぜですか？


通常のプログラマは、新しいツールにアクセスして何をしますか？ もちろん、彼はコードを書き始めます。 次にテストします。 そして、バグがないように見えるとき、彼はとにかく彼の発案を公開します。 これはどこでも正常ですが、Google Glassではそうではありません。 このプラットフォームでは、ユーザーは実世界と仮想世界の間で注意を切り替えません。 この意味でのGoogle Glassはユニークなツールです。 「メガネ」を使用しないプログラマは、特に最初は特に、アプリケーションを目立たなく機能的にすることはできません。 Glass Explorerのユーザーエクスペリエンスは、ガイドラインに完全に置き換わるものではありません。 おそらく、Google Glassのまだ小さなコミュニティを「キャリア」の不快で邪魔なGoogleアプリケーションから保護し、Mirror APIを「隠す」ためです。



しかし、アクセス権があるとしましょう。 それで何ができますか？



配信時間を保証することなく発行および購読します


Google Glassインターフェースの主なパラダイムはタイムラインです。 時計と音声入力を備えたホーム画面の右側には、過去のものである無限のカードのリボンがあります。 Mirror APIを使用するすべてのアプリケーションは、時系列でカードを公開し、これらのカードで発生するイベントをサブスクライブできます。



ユーザーは、カードに添付されたメニュー項目を使用してイベントを生成します。 カードには、定義済みのメニュー項目として、たとえば「削除」または「共有」のほか、アプリケーションで定義されたものが含まれる場合があります。 カードには、付属のカードが含まれる場合があります。 このような「パッケージ」の編成スキームは非常に原始的であり、マルチレベルの構築を許可していません。 一連のカードに同じbundleIdを割り当て、「カバー」となるカードにisBundleCover = trueを設定します。 この場合、「カバー」メニューは使用できなくなります。 ユーザーは、接続されているすべてのカードを取り外した場合にのみ、再び使用できます。



カードは「ホーム」画面の左側に配置することもできます。 これらは「ピン留めされた」カードです。 isPinned = trueプロパティを設定することにより、Mirror APIを介してこのようなカードを追加しようとすることができますが、ほとんどの場合、何も機能しません。 Mirror APIは引き続き、カードを共有フィードにダンプします。 ただし、解決策があります。アクションを使用してメニューオプションにTOGGLE_PINNEDを追加し、ユーザー自身が必要と考える場合はカードを修正します。 カードを更新しても、その状態には影響しません。カードは、管理者またはユーザーが削除するか、ユーザーがUnPinを同じメニューオプションにするまで固定されたままになります。



もちろん、これはMirror APIでできることのすべてではありません。 ユーザーにアプリケーションの「連絡先」を追加して、写真やビデオを手探りする機会を与えることができます。 カードには添付ファイルが含まれる場合があります。 これらの同じカードの外観を形作るには多くの機能があります。 ここでは、これらすべてを試すことができる便利なリソースへのリンクのみを残します。 APIs Explorerを使用すると、Mirror APIとの通信のトレーニングを行うことができ、 プレイグラウンドではカードを「設計」できます。



この場合に重要なことは異なります。ミラーAPIを使用して対話型アプリケーションを作成することはできません。 ユーザーは「インターフェース」で何かを行うことができますが、このイベントがいつGoogleから配信されるかはわかりません。 ユーザーに何かを見せることができます。 ただし、ユーザーが「メッセージ」を受信する時期を予測することはできません。 Mirror APIを使用すると、優れたアプリケーションのアイデアのほとんどは根本的に実現不可能です。 これを理解する必要があります。 そして、あなたはそれに耐えなければなりません。



インタラクティブな何かをする方法は？


そして、ここでGlass開発キットが私たちの助けになります。 公式には、まだ承認されていませんが、まだ公開されていません。 Googleでは、通常のAndroid SDKの使用を推奨しています。 そうかもしれませんが、「ユーザー入力」という点でのGoogle Glassの非常に珍しい特性を忘れないでください。 ボタンはありません。 通常の意味でのタッチパネルはありません。 Glass Explorerが「タップ」および「スワイプ」するのはジェスチャのみです。 OnTouchをキャッチすることはできません。 長押しを傍受する機能はなく、トップダウンジェスチャは予約されており、onBackPressedとしてアプリケーションにキャッチされます。 奇妙なことに、センサーが役立ちます。 このデバイスのうなずきと頭の回転は、ボタンに代わる価値のあるものです。 音声入力では、すべてを置き換える必要がありますが、私たちが望むほど良くはありません。 少なくとも、コマンドを追加して、認識されたときにイベントを受け取ることができませんでした。 しかし、おそらく私は十分に努力しなかったので、あなたはもっとうまくやるでしょう。



一般的に、これは何らかの形で行われます


Google Glassのネイティブアプリケーション、たとえば次のようなものがあります。 そこからglasslib.jarを取得します。これは、おそらくGDKとして後で公開されるものの類似物であると思われます。 それをプロジェクトに追加して、ミラーAPIを使用するのと同じ方法でティムラインカードを操作する機会を得ます。 2つの重要な利点のみがあります。 遅延や制限はありません。 ここでカードをisPinned（true）にすると、ユーザーの関与なしに「ホーム」画面の左側に素直になります。 私たちは、TimlineHelperを通じて、常にサービスからTimlineと連携しています。 通常のスキームは次のとおりです。アプリケーションには、起動時にサービスを開始して終了するアクティビティが1つしかありません。 また、デバイスブートイベントにサインアップし、BroadcastReceiverからサービスを再度上げることは問題ありません。 サービスでは、ユーザーがアプリケーションのカードを持っているかどうかを確認し（このため、そのIDをSharedPreferencesに保存するとよいでしょう）、古いものを削除して新しいものを追加し、再びそのIDを保存します。 



import android.app.Service; import android.content.ContentResolver; import android.content.Intent; import android.content.SharedPreferences; import android.os.IBinder; import android.preference.PreferenceManager; import com.google.glass.location.GlassLocationManager; import com.google.glass.timeline.TimelineHelper; import com.google.glass.timeline.TimelineProvider; import com.google.glass.util.SettingsSecure; import com.google.googlex.glass.common.proto.MenuItem; import com.google.googlex.glass.common.proto.MenuValue; import com.google.googlex.glass.common.proto.TimelineItem; import java.util.UUID; public class GlassService extends Service { private static final String HOME_CARD = "home_card"; @Override public int onStartCommand(Intent intent, int flags, int startid){ super.onStartCommand(intent, flags, startid); GlassLocationManager.init(this); SharedPreferences preferences = PreferenceManager.getDefaultSharedPreferences(this); String homeCardId = preferences.getString(HOME_CARD, null); TimelineHelper tlHelper = new TimelineHelper(); ContentResolver cr = getContentResolver(); if(homeCardId != null){ // find and delete previous home card TimelineItem timelineItem = tlHelper.queryTimelineItem(cr, homeCardId); if (timelineItem!=null && !timelineItem.getIsDeleted()) tlHelper.deleteTimelineItem(this, timelineItem); } // create new home card String id = UUID.randomUUID().toString(); MenuItem delOption = MenuItem.newBuilder().setAction(MenuItem.Action.DELETE).build(); MenuItem customOption = MenuItem.newBuilder().addValue(MenuValue.newBuilder().setDisplayName("Custom").build()).setAction(MenuItem.Action.BROADCAST).setBroadcastAction("net.multipi.TEST_ACTION").build(); TimelineItem.Builder builder = tlHelper.createTimelineItemBuilder(this, new SettingsSecure(cr)); TimelineItem item = builder.setId(id).setText("Hello, world!").setIsPinned(true).addMenuItem(customOption).addMenuItem(delOption).build(); cr.insert(TimelineProvider.TIMELINE_URI, TimelineHelper.toContentValues(item)); preferences.edit().putString(HOME_CARD, id).commit(); return START_NOT_STICKY; } @Override public IBinder onBind(Intent intent){ return null; } }




上記のように、カードには2つの項目のメニューがあります：削除とカスタム。 そして、最初のものがシステムによって処理され、カードを素直に削除すると、2番目のものがブロードキャストをスローし、それをキャッチして処理することができます。

ささいな「Hello、world」にこだわらないように、私は小さなプロジェクトを作りました。 これをより高度な資料として使用して、Google Glassの「ネイティブ」作業の機能を学習できます。 それで、もちろん、私はいつも質問に答える準備ができています。



もちろん、TimeLineをアプリケーションのインターフェイスとして使用することを強制する人はいません。 単純なコントロールでアクティビティを簡単に上げ、ユーザーにそれらの対処方法を教えることができます...ゲームなどのグラフィカルなアプリケーションの場合、これが一般的に唯一の方法です。 しかし、通常のアプリケーションに関しては、私の意見では、この珍しいプラットフォームでは「ネイティブ」スタイルで実行する必要があります。 その後、ユーザーからの温かい歓迎を期待できます。
