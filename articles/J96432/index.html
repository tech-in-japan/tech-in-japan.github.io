Sikuliを使用してドキュメントスナップショットのクリーニングを自動化する

少し前に、長年の解説を本格的なトピックに拡張するように頼まれました。 それだけでは十分に面白いとは思いませんが、便利な機能と快適な機能を組み合わせて、1つの好奇心tool盛なツールと知り合いになるのはいかがですか。そのニュースは最近ITリソース全体に広がっています。



問題



このトピックのフレームワークで解決する主なタスクは、印刷、コンパクトストレージ、djvuでのパッケージ化などのために、書面（書籍、講義など）のスキャンと写真の準備です。

PhotoshopおよびFineReaderは考慮されません。 それらは多くの便利なツールを提供しますが、一般的に言えばお金がかかります。

スキャナーを使用すると、通常はすべてシンプルです。十分な品質の画像が得られるため、最小限の処理を実行できます。

写真でもっと面白い：照明の問題と幾何学的な歪みが追加されます。 残念ながら、少なくとも自動化するための幾何学的歪みの補正は困難です。 しかし、照明と背景があれば、戦うことはかなり可能です。 何をしますか。



ツール



Paint.NETは、レイヤーとフィルターをサポートするWindows用のラスターグラフィックエディターです。

Sikuliは基本的に、グラフィカルインターフェイスとの対話を自動化するためのツールです。 さらに、アプリケーションをテストするための追加機能がありますが、この記事ではそれらに触れません。 Paint.NETでマクロが完全にサポートされていないことを補うために、Sikuliを使用します。

Sikuliの主なキラー機能は、作成されたスクリプトの可視性と単純さである必要があります 。 これは、「見えるものはその仕組み」です。 確かに、プロジェクトの全体的な湿気は印象をいくらか台無しにします。 バージョン0.09で作業しました。 最近リリースされたバージョン0.10では、メインのレーキは削除されましたが、エディターの[元に戻す]機能など、多くの通常の機能はまだありません。

ところで、私は最近QAliberプロジェクトに出会いました。 どうやら、テストされたインターフェイスとの相互作用および全体的な洗練の点で多くの利点があります。 しかし、可視性...一般的には、違いを見て感じることができます:)おそらく、私は時々QAliberを使用しようとします。


Sikuliアーキテクチャには、さまざまな言語で記述されたいくつかのレイヤーが含まれます。
  • 最上位はJython APIです。 本質的に、SikuliスクリプトはPythonプログラムであり、Jython APIによって提供される機能にアクセスします。 （各プロジェクトは％scriptname％.sikuliフォルダーに格納されています。フォルダー内には％scriptname％.pyファイルとPNG画像があります。）著者は、JVMの上で実行される他の言語でトップレベルを実装する可能性に言及します。 Sikuli Java APIをプログラムから直接操作できます。
  • 中間層はJava APIです。 キーボードとマウスで動作し、OpenCVライブラリと対話して、画面上の特定のグラフィックパターンを検索します。
  • したがって、プラットフォーム依存の下位レベルは、C / C ++で実装されたOpenCVライブラリです。
筆者とはあまり似ていないアーキテクチャを説明しましたが、主なことは、システムのアイデアを得ることができることです。



理論



実際、タスクはノイズから有用な信号を分離することなので、適切なアナロジーを使用してアイデアを説明できます。バンドパスフィルターとアクティブノイズリダクションシステムです。



単純なしきい値フィルターはバンドパスフィルターのように機能し、特定の境界線より下の明るさのピクセルを「カット」します（明るさを0に設定し、他のすべての明るさを255に設定します）。 より高度なレベルでは、値がスムーズに変化する2つの境界が設定されます。

画像内の明るさが広範囲にわたって変化する場合、バンドパスフィルターのみを使用しても、有用な信号を失うことなくノイズが「カットオフ」されることはありません。 より巧妙な方法が必要です。



簡単に言えば、アクティブノイズリダクションシステムの動作原理は、「（信号+ノイズ）-（ノイズ）=（信号）」と表現できます。

（信号+ノイズ）は私たちの写真です。 （ノイズ）は背景で、テキストを除くすべてのものです。 （信号）は、したがって、テキストです。

最初は（信号+ノイズ）しかありませんが、この場合、有用な信号（テキスト）の特定のプロパティを使用すると、ノイズ（ノイズ）を簡単に取得できます。細い線で構成されます。

画像が空白のシートのように見えるように、テキストを穏やかに「ぼかす」フィルターを選択する必要があります。 このようなフィルターとしては、 Median Blur （何らかの理由でPaint.Net でノイズを処理する手段として[ ノイズ ]メニューにあります）が適しています。これを逆の目的で使用し、有用な信号を削除します。

確かに、イラストでは物事はそれほどスムーズではない場合があり、それらは個別に処理する必要があります...



アクションのアルゴリズムは次のとおりです。

  1. テキストなしできれいな背景を得るために、元の画像に中央値ぼかしフィルターを適用します。
  2. 元の画像と段落1で取得した画像の差を計算します。
  3. 段落2で取得した画像を反転します（白い背景に暗いテキストが必要です）。
  4. レベルフィルターを適用してコントラストを均一にし、アイテム1-2の後に残ったわずかなノイズを取り除きます。
美しいスキームやイラストがあったかもしれませんが、それでも私は自分の完璧主義をデザイン能力と調和させることはできませんでした（むしろ、彼らの不在）。 意味が非常に透明で、写真がないことを願っています。




自動化



したがって、自動化のタスクは、Sikuliを使用して、記述されたアルゴリズムを使用してPaint.NETで一連の画像を順番に開いて処理することです。

事前に画像フォルダを開いて、Sikuliにアイコンを表示させ、コンテキストメニューからPaint.NETを起動するよりも良い方法は思いつきませんでした...



Sikuli IDEを開き、必要な変数を宣言して新しいスクリプトを開始します。

patterns = [ , , ]

openwith_img =

paintnet_img =

waitfor_img =

edited_text = "_edited"

base_timeout = 30000

negation_mode =

difference_mode =





  • パターン -処理するファイル形式の画像を含む配列。
  • openwith_img 、 paintnet_img-クリックするコンテキストメニュー項目。
  • waitfor_img-Paint.NETを開く操作には時間がかかり、このフラグメントが画面に表示されると完了したと見なされます。
  • edited_text-処理されたファイルの名前に追加される接尾辞。
  • base_timeout-必要に応じてスクリプト全体でタイムアウトを変更しないように、リソースを集中的に使用するすべての操作のタイムアウトの基本値（ミリ秒単位）。
  • negation_mode 、 difference_mode-スクリプトを書いている間、これらの2つのレイヤーブレンドモードを試していました。 したがって、変数として宣言するのが便利でした。


ここでは、Sikuliアプローチの基本的な問題、つまりスクリプトの移植性の制限に注意する必要があります。

ほぼ確実に異なるグラフィック形式のアイコンがあります。 スクリプトに自分で追加する必要があります。 残りの画像は、OSと使用されるレイアウト（VisualStyle）の影響を受ける場合があります。 私の場合、これらはb0seのWindows XPおよびOpus OSです。

結論



  1. 人が受信データストリームから有用な情報を抽出できる場合（テキストの読み取り、キャプチャの解析など）、コンピューターのアルゴリズムを作成してこの情報を抽出できます。 このアルゴリズムの複雑さと汎用性は別の問題です。 アルゴリズムが必要なほど、詳細を考慮する必要があります。 説明したアルゴリズムを使用すると、単純なしきい値フィルターよりも深刻な場合にテキストスナップショットをクリアできますが、制限もあります。
  2. 今日、Sikuli IDEを深刻なツールと考えることは困難です。 「写真を使ったプログラミング」は愚かな仕事だからではありません。 インターフェイスを操作するときにコンピュータービジョンを使用するだけでは信頼性が低く、使用可能なツールは同時にあまり便利ではなく、単純なタスクを解決する場合でも問題が発生することがあります。 同様の問題が発生する別の機会に、QAliberを試します 。
  3. 多くのタスクについて、Sikuli Java APIは、独自のテストツールなどで使用するためのOpenCVの便利なラッパーとして便利になると思います。
