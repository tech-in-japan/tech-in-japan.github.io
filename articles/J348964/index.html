はじめに

あなたがそうでないことを除いて。 近くさえありません。 あなたが実際にあなた自身のアプリを構築しようとし始めたとき、それはマーフィーの法則があなたを襲ったときだからです。



-エリックサイモンズ



実際、 RealWorldはMediumまたはHabrなどのブログ社会プラットフォームのクローンであり、 「Conduit」と呼ばれます。これは、同じ仕様とレイアウトに従ってさまざまなフロントエンドおよびバックエンド（はい、フルスタック）テクノロジーを使用する愛好家によって開発されています。 プロジェクトに参加したい人はだれでも、GitHubのリポジトリに新しい問題を作成する必要があります。そこで、目的の技術スタックを記述し、プロジェクトのスターターキットをフォークして、開発を開始します。 さらに、開発者は、AngularJSでEricによって作成されたデモアプリケーションを使用して、実際の最終結果を確認する機会があります。



実際、それぞれの新しい実装の結果はまったく同じアプリケーションですが、他のテクノロジーまたはアプローチを使用して記述されている必要があります。 React / Redux 、 Elm 、 Angular2 、 React / MobX 、 Svelte / Sapperおよびその他のフレームワークのフォークは 、既に実装および公開されています。 途中で、 Vue 、 Emberなども。







バックエンドテクノロジーの中でも、プロジェクトはNode / Express、Laravel、Django、ASP.NET Core、Railsなどに実装されています。







2017年の終わりに、プロジェクトは、パフォーマンス（最初の意味のあるペイント）、バンドルのサイズ（gzip）、プロジェクトの実装に必要なコードの行数（loc）の3つの基準に従って、最も人気のある 9つのフロントエンド実装の比較を準備しました 。

プロジェクト宣言：



  1. RealWorldプロジェクトの仕様に準拠します。
  2. サーバー作業を完全にサポート（SSRおよびその他すべて）。
  3. 本格的なSPAとしてクライアントに取り組みます。
  4. 検索エンジンによって索引付けされます。
  5. クライアントでJSをオフにして作業します。
  6. 100％同形（一般）コード; *
  7. 実装のために、「ハーフメジャー」と「クランチ」を使用しないでください。 **
  8. 最大のシンプルでよく知られた技術スタックを使用します。 ***
  9. 最終的なバンドルのサイズは、100Kb gzipを超えないようにしてください。
  10. アプリケーションコードの行数は1000 locを超えてはなりません。




*-一般的なコードの読み方について。
これは少し重要なポイントです。 一般的な（同形の）コードを正確にどのように考慮すべきかについて多くのことを考え、そのようなコードは、実装のために私が書いたアプリケーション自体のコードであるという結論に達しました。 つまり、現在のプロジェクトの機能を実装するコード、つまり プロジェクト依存コード。



アプリケーションの基礎（たとえば、Webサーバーのコードなど）のみであり、別のプロジェクトで変更なしで使用できるコードの部分は、アプリケーションコードとして考慮されません。 このコードを「スターターキット」または「定型」などと呼ぶことができます。 いずれにせよ、このコードは特定のビジネスタスクの直接的な部分ではなく、アプリケーションの記述に基づいて一般化されたベースを実装します。



この場合も、アプリケーションコードは、特定のプロジェクトの実装のために私が個人的に作成したコードと見なされます。



**-松葉杖と半分の対策についての私の理解。
同型アプリケーションの例を調べると、次のようなまったく正当化されていないことがよく使用されるという事実に注意を払うことができます。（1）サーバーアプリケーションとクライアントアプリケーションの別々のエントリポイント。 （2）現在の環境（「isServer」など）および共有ファイルのフレームワーク内の個々のコードブランチをチェックするすべての種類の条件ステートメント。 （3）サーバールーティングとデータフェッチは、クライアントルーティングなどとは異なる方法で実行されます。 など



この不十分な実証済みの複雑さとコードの難読化が、少数の開発者が試行することを恐れさせているのではないかと思います。 私の実装では、小さな町の松葉杖の代わりに、建築計画の情報に基づいた決定を使用するよう努めます。



***-スタックについて。
多くの場合、同型アプリケーションに関しては、 クジラ 、 トップ レベルの フレームワーク、または通常の技術スタックを変更する他の特別な ものを使用する必要があるという考えがあります。



したがって、最も単純で、最もアクセスしやすく、よく知られているテクノロジースタックを使用しようとします。 つまり、特別なツールの使用を最小限に抑えます。



ご覧のとおり、リストはかなり野心的です。 ここには、議論の余地のある「同型」と、誰もが切望しているが達成できない「進歩的な強化」があります。 そして、SPAのスタイルで動作するブログプラットフォームですが、同時にSEOをサポートします。 これらはすべて、既存のバックエンド上にある単一のコードベースで、TORに従って変更できません。



そういえば。 同型アプローチを嫌う人の多くは、本格的な同型アプリケーションを作成するためには、特別な方法でバックエンドを書き換える必要があるとしばしば言うので、このプロジェクトは私にとって興味深いものです。 つまり、既存のバックエンドは機能しません。 または、間違いなくNodeJSをバックエンドとして使用する必要があります。それ自体はfu fu fuです。 または、何らかの特別な方法で要件とTKおよびその他のナンセンスを策定する必要があること。



また、同型コードははるかに複雑であり、一般に同型アプリケーションを記述することは難しく、時間がかかり、経済的であると考えられます。 私の「マニフェスト」の最後の3つのパラグラフは、これらの誤解から生じています。



要するに、私にとって、このプロジェクトは、これを完全にまたは部分的に反論する機会であり、同時に楽しむことでもあります。 私は本当に成功することを願っています。 トピックに興味があり、このパスをステップでフォローしたい場合は、切り替えないでください！

建築について







これは、アプリケーションアーキテクチャ全体の外観です。 いくつかの重要なポイント：



  1. Fontendの部分は、RactiveのクライアントサーバーアプリケーションとExpress / Nodeの純粋にサーバーベースのアプリケーションに分かれています。
  2. RealWorldバックエンドは、フロントエンドサーバーの背後にあります。
  3. REST APIへのHTTPリクエストは、フロントエンドサーバーを介してバックエンドにプロキシされます。


ワークフロー



  1. ユーザーはURLをアドレスバーに入力します。
  2. クライアントはサーバーに同期要求を行います。
  3. サーバーは着信要求を処理し、メインアプリケーションコードを実行します。
  4. アプリケーションの準備ができた時点で、サーバーはアプリケーションの現在の状態をHTMLでレンダリングし、応答を返します。
  5. クライアントは受信したHTMLを解釈し、関連リソース（CSS、JS、画像、フォントなど）の読み込みを開始します。
  6. ユーザーはコンテンツを受け取ります。
  7. アプリケーションコードはバックグラウンドでダウンロードされ、初期化されます。
  8. ユーザーはインターフェースを操作します。
  9. アプリケーションのクライアントコードは、アクションを処理し、サーバーへの非同期要求を作成し、状態を更新します。


このプロセスは、同形のWebアプリケーションでは完全に正常であり、この場合でも同様に機能します。

そしてもちろん、すべての種類のpackage.json 、 webpack.config.jsおよびその他の構成ファイルの完全な束。その説明はチュートリアルの範囲外です。



コードを書く



Webサーバーのメインコードは./server.jsファイルにあります。 Express自体に加えて、実用的なタスクを解決するためにプラグイン（ 「エクスプレスミドルウェア」 ）のいくつかを使用します。 以前は、もちろん、必要なモジュールはすべて、 npm iコマンドを使用してpackage.jsonに登録する必要があります。



最初に、サードパーティのモジュールとExpress自体を接続します。 



const express = require('express'), helmet = require('helmet'), compress = require('compression'), cons = require('consolidate');


使用されているすべてのExpress拡張機能は完全に普通のものであり、同型とは関係ありません。 それらのほとんどは非常に標準的です。 以下に簡単な説明と目的を示します。



  • ヘルメット -特定のHTTPヘッダーを設定することにより、いくつかの一般的な攻撃からフロントエンドを保護します。
  • 圧縮 -サーバー応答のgzip圧縮。
  • 統合 -Expressのテンプレートモジュール。


次に、これまで何もしないミドルウェアをいくつか作成し、それらを接続します。 



 const app = require('./middleware/app'), api = require('./middleware/api'), req = require('./middleware/req'), err = require('./middleware/err');




  • app-サーバーでアプリケーションを起動するコードはここにあります。
  • apiは、APIバックエンドのhttp要求のプロキシです。
  • req-すべてのHTTP要求の前処理。
  • err-サーバーエラー処理。


メインサーバーの設定common.jsonを接続し、Expressと拡張機能を初期化します。 



 const config = require('./config/common'); const server = express(); server.engine('html', cons.mustache); server.set('view engine', 'html'); server.use(helmet()); server.use(compress({ threshold: 0 })); server.use(express.static('dist'));




Expressのテンプレートエンジンとして口ひげをインストールします。 なぜ彼は正確に？ Ractiveはテンプレートに口ひげ構文を使用しているだけなので、均一性を実現します。



次に、静的全体も圧縮したいこと、および静的が「./dist」フォルダーにあることを通知します（Webpackもそこに生成されます）。



次に、最後の仕上げ： 



 server.use(req()); server.all('/api/*', api()); server.use(app()); server.use(err()); server.listen(config.port);


コードには特別な説明は必要ありません。 ここでのキーポイントは次のとおりです。



  • / api / *へのすべてのリクエストはREST API（apiミドルウェア）にプロキシされます。
  • 他のすべてのリクエストは、アプリケーション（アプリミドルウェア）を起動し、HTMLを返します。
  • 特定のケース（oauthなど）を処理するために、追加の「ミドルウェア」を追加することが可能です。 中間の「ミドルウェア」が機能しなかった場合、またはnext（）に進んだ場合、制御は常に「アプリミドルウェア」に転送されます。
  • サーバー側のエラーが発生すると、 「errミドルウェア」でストールが発生します。


Webサーバーの概要コード 
 const express = require('express'), helmet = require('helmet'), compress = require('compression'), cons = require('consolidate'); const app = require('./middleware/app'), api = require('./middleware/api'), req = require('./middleware/req'), err = require('./middleware/err'); const config = require('./config/common'); const server = express(); server.engine('html', cons.mustache); server.set('view engine', 'html'); server.use(helmet()); server.use(compress({ threshold: 0 })); server.use(express.static('dist')); server.use(req()); server.all('/api/*', api()); server.use(app()); server.use(err()); server.listen(config.port);




Expressの経験があまりない純粋なフロントエンドの開発者でさえ、このようなプリミティブコードを簡単に理解できるように思えます。 もちろん、彼が「ミドルウェア」とは何か、そして一般にこの概念を理解している場合。



これまでのところ、このコードは実際には何もしません。 いずれの場合でも、プロジェクト内で行う必要があることは行いません。 次の記事では、純粋にサーバー側のコードを追加し、アプリケーション自体のコードに進みます。
