Peat TVのやり方

この記事では、インターネットTVチャンネルの技術的な実装がどのように作成されたか、チームが直面したタスク、開発プロセスでどのツールとサービスが役立ったかについて学ぶことができます。





Peat TVは、Kirill Kislyakovによって設立された文化人類学のインターネットプロジェクトです。 これは、それぞれ、トピックの選択、キャラクターの選択、素材のプレゼンテーションの両方に対する顕著な主観的アプローチを用いた著者のプロジェクトです。 このチャンネルでは、芸術と教育、文学と詩、科学と技術、音楽と歴史、非政治ニュース、興味深い事実などの幅広い問題を短いビデオ形式で取り上げています。

フッテージピートテレビは、素材の蓄積とともに登場します。 各号は、ビデオアートのスタイルで作成された3〜15分の短いビデオです。



▌コンセプト



いくつかの問題は人々の独白または対話であり、時には視聴者に向けられたり、時には彼らは撃たれているという事実に注意を払わずに互いにコミュニケーションをとったりします。 ドキュメンタリー番組の形式で行われた問題があり、インタビューの形式で問題があります。 インタビュアーは対談者のようにフレーム内にいる場合があり、時には見えない場合があり、視聴者には質問をする音声のみが聞こえます。 すべての問題は、特定のトピックを持つプログラムのカテゴリに分類されます。



今日、チャンネルには次のようなセクションがあります。

  • ポートレート -著者インタビュー
  • ブラザーズインザトラッシュ -アルコール消費の文化に関する問題
  • サウンドを見る -問題は音楽の世界のイベントや人に捧げられています
  • 地理の終わり -旅行およびワールドエディション
  • 好きな詩 -ゲストが好きな詩を読む
  • 最初のリットル -文学に捧げられた問題
  • 3番目のシフト -専門家は何をなぜ行うのかを話します




人類学の観点から、最も興味深い問題はポートレートとゴミ箱の兄弟です。 だから、最初のカテゴリーの問題では、人は自分が言いたいことを言い、2番目の問題では、自分が言いたいことを言う。 どちらの形式でも、多くの場合、5分以内に彼自身が提案する以上のことを言うことができます。



チャンネルの視聴者は、文化の問題だけでなく、多くの時代や世界のあらゆる場所に関連する文脈や時間から引き裂かれた思考の流れに従う機会に興味がある人々です。したがって、Peatのビデオは、文化生活で何が起こるか、著者のチームまたは招待されたキャラクターのビューに突然何が入るかのレビュー。 ストーリーは過去のものでも未来のものでもある場合がありますが、関連性には影響しません。

▌プレーヤー



このプロジェクトでは、主要なビデオ形式をサポートする独自のHTML5 / JSプレーヤーが開発されました。

  • oggは、メディアコンテナー形式のオープンスタンダードであり、Xiph.Org基盤のコーデックのメインファイルおよびストリーム形式です。 この形式のビデオは、バージョン4までのFirefoxブラウザーと10.6までのOperaユーザー向けにエンコードされました。
  • mp4は、MPEG-4標準の一部であるメディアコンテナ形式です。 この形式はほとんどのブラウザでサポートされており、統計から判断すると、この形式の動画はほとんどの場合ダウンロードして視聴できます
  • webmは、2010年5月19日に開催されたGoogle I / OカンファレンスでGoogleによって導入されたオープンマルチメディア形式です。 この形式では、オープンVP8ビデオコーデックに基づくロイヤリティは不要です。 標準のより広範な配布が想定されていたため、代替コーデックとして使用されました。


各フォーマットに対して、異なる解像度の4つのビデオオプションを作成します。

  • SD-インターネット速度が遅いユーザーおよび携帯電話向け
  • HD-このモードは、プレーヤーモードでビデオを視聴するためにデフォルトで有効になっています
  • HD720-フルスクリーンでビデオを見るのに適しています
  • HD1080-スマートTVや高解像度スクリーンなどのデバイスからのリリースを視聴する人向けに、この解像度のサポートが少し後に登場しました


統計によると、ほとんどのビューはHDおよびSD解像度のmp4形式のビデオファイルです。



videoビデオの準備と公開



複数の編集者で構成されるチームがリリースに取り組んでいます。 編集者はさまざまな都市にいて、遠隔地で働いています。 それらのそれぞれは、エンコードサーバー（メディアエンコードサーバー）へのFTPアクセスを持ちます。 エディターは、元の（生の）ビデオファイルと共に、リリースの説明とメインページに表示されるビデオのポスターを含むテキストファイルもサーバーにアップロードします。 ビデオがFTP経由でサーバーにアップロードされた後、自動ビデオエンコードのプロセスが開始されます。



  1. コーディングサーバーでは、特別に設計されたプログラムを使用して、ビデオがさまざまな形式と解像度に変換されます。

  2. ビデオのエンコードが完了すると、プログラムはビデオをRackspace CDNエンドポイントにダウンロードし始め、そこからファイルはローカルCDNサーバーに送られます。

  3. ダウンロードが完了すると、新しいリリースを追加するためのリクエストがWebアプリケーションに送信されます。

  4. Webアプリケーションがコールを受信すると、データベースにビデオ情報を追加する手順が実行され、リリースがサイトに送られます。





ビデオを変換するプログラムを作成する過程で、変換を担当するコードを別のライブラリに割り当てることが決定されました。 ライブラリはX.Media.Encodingと呼ばれ、nugetリポジトリから無料でダウンロードできます 。



cloudsクラウドへの移行



開発を開始したとき、サイトは通常の専用サーバーに展開されていましたが、最初はそれで十分でした。 ただし、時間が経つにつれて、通常のサーバーにはいくつかの欠点があることが明らかになりました。

  1. まず第一に、それはビデオファイルに関するものでした-リリースのダウンロード速度は低かったです
  2. 2番目の問題は、リソースが基本的にピーク負荷モードで機能するという事実に関連していました。訪問者のアクティビティは、新しいリリースがリリースされたときに最大になり、リリース間で最小になります。 したがって、リソースがほとんどの時間アイドルになる、より強力なサーバーを購入するか、自動スケーリングをサポートするプラットフォームに切り替える必要があります。




最初に、問題のダウンロード速度に関する問題を解決するために、ウクライナでコンテンツ配信サービスを提供する会社CDN.UAのサービスを使用することにしました。 ちなみに、彼らは非常にプロフェッショナルであり、私たちのプロジェクトでは、サーバー上でいくつかの特定の設定を行いました。 この段階で、ウクライナの視聴者にとって、視聴の問題はほぼ完全になくなりました。 しかし、ジョージア、イスラエル、およびその他の国からのリリースを見た人にとって、視聴の遅れは消えませんでした。 そのため、ビデオを国際コンテンツ配信ネットワークに転送する次のステップが決定されました。 Rackspace CDN は、世界最大のAkamaiネットワークを使用します。





新しいプラットフォームへの移行に関連して、ビデオダウンロードプログラムをわずかに更新する必要がありました。 したがって、ビデオをFTP経由でCDN.UAにアップロードした場合、それを使用してRackspaceにアップロードします

OpenStack SDK： 
using net.openstack.Core.Domain; using net.openstack.Providers.Rackspace; using NLog; using System; using System.Collections.Generic; using System.IO; using System.Linq; using System.Net; using System.Text; using X.Media.Encoding; .... public bool Upload(string path) { const string container = "Video"; const string region = "DFW"; try { var cloudIdentity = new CloudIdentity { Username = _settings.StorageUserName, Password = _settings.StoragePassword }; var contentType = ContentTypeManager.GetContentType(Path.GetExtension(path)); var name = Path.GetFileName(path); _logger.Info(String.Format("Upload: {0} \t with content type: [{1}]", Path.GetFileName(path), contentType)); using (var stream = File.OpenRead(path)) { var cloudFilesProvider = new CloudFilesProvider(cloudIdentity, region, null, null); cloudFilesProvider.CreateObject(container, stream, name, contentType); _logger.Info(String.Format("Uploaded: {0}", path)); } } catch (Exception ex) { _logger.WarnException(String.Format("Error while uploading: {0}", path), ex); return false; } return true; }






ビデオコンテンツがCDNに転送された後、サイトが変わりました。 Webサーバーの負荷は、ビデオを配信する当初の負荷に比べて非常に小さくなりましたが、以前のように訪問者の流入が不均一だったため、サイトをクラウドプラットフォームに移行することにしました。 これで、WebSitesサービスのMicrosoft Azureにサイトが展開されました。 このサイトは、訪問者の数が突然増加した場合の負荷分散とリソースの自動スケーリングをサポートしています。



興味深い点のうち、このサイトには2つの表示モードがあります-夜と昼、これは時刻に応じて自動的に変化します。



▌プロジェクトの開発とサポートのプロセス



プロジェクト開発チームは小規模ですが（当初は4人でしたが、現在はコードの作成に積極的に関与しているのは2人のみです）、プロジェクト開発と公開プロセスを可能な限り最適化することが決定されました。 プロジェクトのソースコードをプライベートBitbucketリポジトリに保存します。 既に述べたように、プロジェクト自体はAzure Webサイトサービスでホストされています。 そして、このサービスにより、GitHubとBitbucketからプロジェクトを自動展開するプロセスを構成できます（実際、ソースの数ははるかに多く、 Dropboxからの公開もサポートされています）。 Azureでは、2つのWebサイトがあります。

  • テスト - マスターブランチからの展開
  • 本番 - クラウドブランチからの展開








新しいコミットが対応するブランチに入ると、プロジェクトの自動ビルドが発生し、正常に完了した場合-サイトの新しいバージョンの公開。 何らかの理由でプロジェクトがクラウド環境で正常にビルドされなかった場合、展開ログを確認することでこの理由をいつでも確認できます。





ただし、この機能は常に、プロジェクトをクラウドに最初に展開する際に一度だけ役に立ちました。



▌モバイルクライアント



プロジェクトの開始から約1年後、多くの視聴者がコンピューターからだけでなくスマートフォンを使用してリリースを視聴する方が便利であることがわかりました。 モバイルクライアントを作成することが決定されました。 3つの主要なモバイルプラットフォーム用に3つのバージョンのモバイルクライアントを作成しました。

  • Windows Phone-バージョン7.8以降でサポート
  • iOS-バージョン5以降からサポート
  • Android-バージョン4.1以降からサポート




モバイルクライアントのロジックは可能な限り単純です。 xml形式では、利用可能なプログラムとビデオリリースのリリースに関する情報をサイトから受け取ります。 また、CDNにあるビデオファイルのURLはモバイルクライアントに送信され、そこからビデオがダウンロードされます。



また、プロジェクトには「内なる声」があります。 彼は、スマートフォンにPeat TVモバイルアプリケーションをインストールしたユーザーと通信します。 ユーザーはリリースのお知らせを受け取ることもあれば、興味深い考えや表現を受け取ることもあります。 Azure Mobile Servicesに基づく「内部音声」が機能します。 リリースされたばかりのモバイルサービスの「内部音声」の作成時に、ロジックはJavaScriptで記述されていたため。 後に、C＃でモバイルサービスのロジックを記述する機能が追加されましたが、すべてがうまく機能するため、このコードをスチールで書き直す必要があります。



興味深いことに、現時点では、ビデオビューの約半分がモバイルアプリケーション経由です。
