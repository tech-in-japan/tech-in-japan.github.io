ユーザーインターフェイスを作成するためのいくつかのパラダイムがあります。



  • すべてのコントロールは1つのXAMLファイルに含まれており、フォーム設計中に結合されます。
  • フォームは論理的な部分、通常はユーザーコントロールに分割されます。 設計中にフォームが作成され、これらのパーツが参照されます。
  • 前のバージョンと同様に、フォームは論理部分に分割されますが、それらを直接参照することはありません。 実行時にパーツがフォームに追加されます。 このようなアプリケーションはコンポジットと呼ばれます。




通常、複合アプリケーションインターフェイスは、ビューと呼ばれる異種のコンポーネントで構成されます。 ビューは通常、アプリケーションモジュールにありますが、これは必須ではありません。 アプリケーションをモジュールに分割する場合、疎結合ビューをユーザーインターフェイスに結合する方法が必要です。 ただし、ビューが異なるモジュールにない場合でも、このアプローチを使用できます。



ユーザーインターフェイスを構築するには、実行時に生成される疎結合の視覚要素で構成されるマークアップを作成できるアーキテクチャが必要です。 さらに、このアーキテクチャでは、ビュー同士が疎結合のままでメッセージを相互に交換できる必要があります。



下の図に示すように、 株式トレーダーリファレンス実装（株式トレーダーRI）は、さまざまなモジュールからShell



で宣言された領域にビューをロードすることにより、実行時にユーザーインターフェイスをコンパイルします。









ユーザーインターフェイスのレイアウト



複合アプリケーションのルート要素はShell



と呼ばれます。 シェルはアプリケーションのマスターページとして機能し、複数のリージョン（ Region



）で構成されます。 リージョンは、実行時にコンテンツがロードされるプレースホルダー要素です。 リージョンは、 ContentControl



、 ItemsControl



、 TabControl



、またはユーザーコントロールなどのさまざまなUI要素に関連付けることができ、コンテンツを制御できます。 リージョンは、アプリケーションの要件に応じて、コンテンツを自動またはオンデマンドでダウンロードできます。



通常、領域のコンテンツはビューです。 アプリケーションのユーザーインターフェイスの一部をカプセル化し、アプリケーションの他の部分から分離します。 ビューは、ユーザーコントロールまたはデータテンプレートのいずれかです。



領域は、ビューのレイアウトと表示を制御します。 領域には名前でアクセスできます。 また、ビューの動的な追加と削除もサポートしています。 リージョンは、ビューが動的にロードされるコンテナと考えることができます。

シェル



シェルはアプリケーションのルートオブジェクトであり、ユーザーインターフェイスの主要部分が含まれています。 WPFアプリケーションでは、 Windows



オブジェクトはシェルとして機能します。 SilverlightアプリケーションRootVisualUserControl

ラッパーはマスターページとして機能し、インターフェイスの基本レイアウトを提供します。 これには、モジュールが必要な表現を代用できる1つ以上の名前付き領域が含まれます。 また、背景、メインメニュー、ツールバーなど、特定の高レベルのユーザーインターフェイスコンポーネントが含まれている場合もあります。



シェルは、アプリケーションの一般的な外観を定義します。 シェル自体に表示されるスタイルと境界線、およびダウンロード可能なビューに適用できるスタイル、テンプレート、テーマを定義できます。



通常、シェルはプライマリWPFまたはSilverlightプロジェクトの一部です。 シェルを含むアセンブリは、ロード可能なビューでアセンブリを参照する場合があります。または、実行時にアセンブリが検索およびロードされる場合、参照しない場合があります。



視聴回数



表現は、アプリケーションのユーザーインターフェイスを構築する主要な要素です。 カスタムコントロール、ページ（ Page



）、またはデータテンプレートをビューとして使用できます。 このビューは、アプリケーションの残りの部分から遠ざけたいユーザーインターフェイスの部分をカプセル化します。 機能の分離とアプリケーション内のインターフェースの一部の再利用の両方に基づいて、プレゼンテーションに含める内容を選択できます。



WPFおよびSilverlightで使用されるコンテンツモデルのおかげで、Prismに関連するビューを作成するのに何も必要ありません。 ビューを作成する最も簡単な方法は、 User Control



を作成することです。 ビューをUIに追加するには、そのビューを作成してコンテナーに追加する方法のみが必要です。 WPFとSilverlightは、これを行うためのメカニズムを提供します。 Prismは、実行時にビューを動的に追加できる領域を指定する機能を追加します。



複合ビュー


いくつかの機能を追加すると、表現が非常に複雑になる可能性があります。 この場合、それを複数の子ビューに分割し、親ビューを使用して子ビューを作成および配置できます。 これは設計時に静的に発生する場合と動的に発生する場合があり、モジュールは実行時にビューを領域に置き換えることができます。 ビューが1つのクラスで完全に定義されていない場合、複合ビューとして参照できます。 多くの場合、親ビューは子ビューの作成とそれらの相互作用の調整の両方を担当します。 イベントアグリゲーターやコマンドオブジェクトなどのPrism機能を使用して、相互に関連性の低い子ビューを設計できます。



ビューとデザインパターン


Prismはこれを主張していませんが、ビューを作成する際にいくつかのデザインパターンの使用を検討する必要があります。 Stock Trader RIおよびQuickStartsは、インターフェイスとロジック間で責任を共有するためのMVVMパターンの使用を示します。



MVVMデザインパターンは、WPF、Silverlight、Windows Phone、Windows RTなどのMicrosoft XAMLプラットフォームと自然に調和するため、ほとんどの場合に推奨されます。 依存プロパティのメカニズムとデータバインディングの豊富なサポートにより、プレゼンテーションモデルとプレゼンテーションが疎結合で相互作用することができます。



プレゼンテーションロジックをプレゼンテーション自体から分離することは、テストとメンテナンスの容易さを促進し、デザイナーとプログラマー間のやり取りのプロセスを改善する重要なステップです。



ビューをユーザーコントロールとして作成し、ロジックを別のコードファイルに配置する場合、ビューのインスタンスを作成し、それを介してロジックをテストする必要があるため、テストするのは困難です。 これは、膨大な量のインフラストラクチャのスタブを作成する必要があるという理由で、いくつかの困難をもたらします。 したがって、インフラストラクチャへの依存関係を削除するには、プレゼンテーションロジックをプレゼンテーション自体から分離する必要があります。



ビューをデータテンプレートとして作成する場合。 このビューに関連付けられたコードがありません。 このため、コンパニオンロジックを別の場所に配置する必要があります。 テスト容易性を提供したのと同じアプローチが、ビューの展開に役立ちます。



ご注意

ユニットテストとUI自動化テストは、カバレッジがまったく異なる2つのまったく異なるテストアプローチです。 単体テストのベストプラクティスでは、オブジェクトを完全に分離してテストすることをお勧めします。 これを実現するには、すべての依存関係の偽物とスタブを作成する必要があります。 その後、単体テストは1つのオブジェクトのみで機能します。 オートメーションUIテストはアプリケーションを実行し、UIにジェスチャーを適用して結果を検証します。 このタイプのテストは、コントロールがアプリケーションロジックに正しく接続されていることを確認するために必要です。


論理を表現から分離すると、責任の分離が促進されます。 これにより、テスト能力が向上するだけでなく、UIデザイナーは開発者とは独立して作業できます。 MVVMパターンの詳細については、パート5：「MVVMパターンの実装」を参照してください。

データバインディングは、依存プロパティシステムによって提供される組み込みの変更通知システムを使用します。 INotifyPropertyChanged

リージョンを作成する


リージョンは、 IRegion
