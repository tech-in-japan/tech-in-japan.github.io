Wolfram言語（Mathematica）は単なるおもちゃです

私は自分の視点を疑っていました。UPDの詳細を参照してください。



Wolfram言語（以降WL）について話をしたいです。 混乱をおかけして申し訳ありませんが、 この投稿はNilisからのこのコメントへの返信の試みから生まれました。



WLは依然として高度に専門化された言語です。 Wolframは単にお金を稼ぐことに決めたため、WLを汎用言語として宣伝するようになりました。 WLが好きなら、それはあなたがまだ他の言語をまだ学んでいないことを意味します。 同じPythonを含む他の言語の束とは異なり、WLはフリーソフトウェアではないことにも注意してください。



約6〜10年前、私が学校にいたとき、私はBASIC、Pascal / Delphi、C / C ++およびWLしか知りませんでした。 OSについては、Windowsのみを扱いました。 スクリプト言語（perl、python、bash）が何であるか知りませんでした。「スクリプト」の概念自体も知りませんでした。 コマンドインターフェイスは時代遅れのインターフェイスであり、唯一の正しいプログラムインターフェイスはグラフィカルなインターフェイスだと思いました。 DelphiとWLでプログラミングしました。 Delphiプログラムは常にグラフィカルです。 Olympiadプログラミングを除き、そこにあるプログラムの要件はテキストである必要があるということでした。DelphiとC ++で作成しました。 WLについて知ったとき、あなたと同じように、私はそれに魅了されました。 WLのコードはDelphi / C ++よりもはるかに短いためです。 そして一般に、WLはDelphiやC ++とは完全に異なっていたためです。 機能的にプログラムする機会があったからです。 WLの例を使用した関数型プログラミングに精通しましたが、「標準的な」関数型言語（Haskell、Lisp、ML）があることを知りませんでした。



ファイルを操作するためのプログラムを作成する必要があるときは、Delphiで作成しました。 そして、GUIで。 彼は窓、ボタンを書いた。 「作業するフォルダを入力する」などのフィールド。これは、bashの1行で実行できることを理解していませんでした。



たとえば、フォルダー内のファイル名をラテン語からキリル文字に変換するプログラムを作成するように求められました。 Delphiでguiを使用してプログラムを作成しました。 「変換したいフォルダを入力してください。」 コードは、おそらく1000行ではなくても、少なくとも100行でした。 これは、これが次のようなGNU / Linuxの1つのコマンドによって行われることを理解しています： prename 'y/abc.../.../' *.avi







私は、Windows用のファイルマネージャー（もちろん、Windows、他のOSの場合、私は知りませんでした）を書くアイデアを思いつきました。WLでスクリプトを作成する可能性があります（同時に、「スクリプト」という言葉を知りませんでした）。 そのため、たとえば、フォルダのFlattenを作成するWLのFlatten関数がありました。つまり、指定されたフォルダ内のフォルダの階層全体を破棄し、すべてのファイルをヒープにダンプします。



それから6年前、大学に行き、GNU / Linuxに精通し、C / C ++でプログラミングを始め、GNU / Linuxの下でbashを始めました。 私は、DelphiとWLでのすべてのプログラミング（数学を除く）がナンセンスであることに気付きました。 すべてのDelphiプログラムをbashの単一行として実装できること。 WLは、数学的な問題にのみ使用する必要があります。 コマンドインターフェイスは時代遅れではないことに気付きました。 そして、ファイルマネージャのアイデアはナンセンスであることに気付きました。 GNU / Linux全体は、すべてのスクリプト言語でスクリプトを作成できる機能を備えたファイルマネージャー（およびIDE）であるためです。



そこでプログラミングを開始しました。先生の名前はボガチョフです。 彼（および他の人々）から、WLは単なるおもちゃであり、数学的問題でもあることを知りました。 WLは、関数をすばやくプロットしたり、導関数を見つける必要がある場合に使用する必要があります。 より困難なタスクについては、使用できません。 なぜなら：

  • Mathematicaとその言語は単なる寄せ集めです。 WLが好きなら、Lispなど、それが由来する言語を使用してください。 Mathematicaパッケージ自体には多数の数学関数が含まれていますが、高度に専門化された数学パッケージで開発されているため、それらはいずれも開発されていません。 たとえば、大きなマトリックス（たとえば、1000 x 1000）をすばやく正確に反転する必要がある場合、WLは使用できません（エラーがある可能性があるため、最適化されておらず、精度が低く、実際のアプリケーションを対象としていません）。 実装を記述するか、C、C ++、Fortranなどの高度に特殊化されたパッケージを使用します
  • WLにエラーがある可能性があります。 たとえば、Cコンパイラでのエラーの確率ははるかに低くなります。第1に、無料のCコンパイラがたくさんあり、第2に、これらのコンパイラは実際のタスクに実際に広く使用されており、
  • WLは、たとえばC ++よりもはるかに遅い
  • WLは数学的に正しくありません。 例： 
     In[9]:= Simplify[(x^2 + x) / x /. x -> 0] 1 Power::infy: Infinite expression - encountered. 0 Infinity::indet: Indeterminate expression 0 ComplexInfinity encountered. Out[9]= Indeterminate In[10]:= Simplify[(x^2 + x) / x] /. x -> 0 Out[10]= 1


    数学的な観点から、ある場合には不定、別の場合には-1を説明することは不可能です。Simplifyは式の数学的な意味をまったく変えてはいけません。 正しい動作は、 (x^2 + x) / x



    単純化せず、xがゼロに等しくないという条件がある場合にのみ単純化することです。 Solveは、特別なケースを考慮しません。 たとえば、 Solve[xa == b, x]



    は、 b/a



    がゼロになることを考慮せずにb/a



    生成するだけです。
  • WLはフリーソフトウェアではありません




Bogachevは、彼の教育活動に加えて、ある会社でテクニカルディレクターとして働いており、私はそこで働いていました。 そこで油田のシミュレーターを作成しました。この地域のお金が莫大であることを説明する必要はないと思います。 そのため、シミュレーターはC ++で記述されており、計算を実行するコード（ディファーや線形方程式系を解く）を含みます。 そしてもちろん、WLではなく、必要な速度がないため、このWLには適切な信頼がないため、一般的にここでWLを使用することは深刻ではありません。



それから、私はイザベル系を含む正式な数学に出会った。 そのため、コンピューターで数学的な操作を実行し、数学的な正確性を確認するための唯一の方法は、形式的な数学であることをお伝えします。 そして、正式な数学のSimplify、Solveなどのこれらの浅瀬はそうではありません。



しかし、そのようなシステム（つまり、正式な数学を操作するためのシステム）はWLに似ていません。 それらの作業は次のとおりです。システムに正式な証明を導入し、それが正しいかどうかを示します。 問題が発生します：WLの動作方法と形式的な数学の正確さを混ぜることは可能ですか？ はい、できます。



Mathematica、Mapleなどは、いわゆるコンピューター代数システムに属します。 したがって、そのようなシステムはすべて、数学的な厳密さの欠如に罪を犯します。 この点に関して、形式数学に基づいたコンピューター代数の真に正しいシステムを書くという考えがあります。 ここで、形式数学の専門家は、この方向の最初の実験の結果を示しています： www.cs.ru.nl/~freek/pubs/holcas.pdf 、少なくとも最初の2ページを読んでください。



したがって、WLの数学については、グラフをすばやく作成したり、導関数を計算したりする必要がある場合は、WLを使用します。 本格的な数学的計算を行う場合は、Bogachevのように汎用言語（C ++など）で自分でコードを記述するか、汎用言語用にプロフェッショナルライブラリを使用します。 計算ではない、つまり数値ではない数学的な操作を実行する必要がある場合は、公式の数学（たとえば、Isabelle）を使用するか、たとえば上記の記事で紹介した公式のコンピューター代数システムを使用して、手動で操作を行います（確かに、そのようなシステムは現在開発の初期段階にあります）。



非数学的な問題の場合、WLはまったく使用できません。 スクリプトを含む他の言語を使用します。 WLが好きなら、関数型言語を見てください。 特に、Lispを見てください。これは、WLが何よりも見た目が正確です。



結果。 WLは単なるおもちゃです。 これは学生のツールですが、科学者ではありません。 たとえば、Large Hadron ColliderのデータがWLで計算されることはほとんどありません。 1CとしてのWL：貧弱で、高度に専門化された、商用の、独自の言語。



この投稿に興味のあるもの（形式数学、関数型プログラミングなど）があれば、尋ねて、答えます。

 In[11]:= N[Integrate[E^(-(x-1)^2)/Sqrt[x], {x, 0, Infinity}]] Out[11]= 0. + 0.411862 I In[12]:= NIntegrate[E^(-(x-1)^2)/Sqrt[x], {x, 0, Infinity}] Out[12]= 1.97373


NIntegrate [...]が受け取った答えは正しいが、N [Integrate [...]]は正しくないという強い疑いがあります。 NIntegrateは、内部で数値微分の単純で愚かなアルゴリズムを実行したためです。 インテグレートとは異なり、インテグラートは、正式な厳密さのない代数的操作の未知の怪物であり、エラーを起こしやすい。 最後に、数値的に積分するC / C ++プログラムを自分で作成した場合にのみ、これらの結果の1つが正しいと考えています。 または、私が紙の上でこの積分を見つけた（または少なくともチェックした）場合。 （それでも、この積分は明らかに本物でなければなりません。）
