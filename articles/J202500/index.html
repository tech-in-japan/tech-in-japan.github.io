Acronis True Image：バックアップ戦略

ハブロ世界の住人であるあなたに挨拶します！ True Imageテクノロジーを引き続き紹介します。 今回は、バックアッププロセスを構成する方法について説明します。

  • データが保護されました。
  • バックアップフォルダ（バックアップであり、バックアップでもあります）がディスクのすべての空き領域を「消費」していませんでした。
  • 異なるバックアップで同じ情報を複製しないでください。


一部には、 Acronis True Imageの記事「 バックアップを作成する方法」でこのトピックについて既に触れました。 もっと詳しく考えてみましょう。





バックアップ戦略は、2つの段階で構成されます。

  • バックアップスキームの作成。
  • バックアップのスケジュール


実際、平均的なユーザーは実際にこれらの問題に対処する必要はありません。 デフォルト設定は、ほとんどの場合に非常に適しています。 ただし、特定のユーザーのデータの詳細、ユーザーがこれらのデータを変更する頻度などを考慮できないことは明らかです。 したがって、自分で微調整することをお勧めします。



バックアップを作成する方法



スキームの作成は、バックアップ方法を理解することから始まります。 そのような方法には、フルバックアップ、増分バックアップ、差分バックアップ（フルバックアップ、増分バックアップ、差分バックアップ）の3つがあります。 なぜ必要なのか、違いは何ですか？ 見ます。



完全バックアップ


ここではすべてが非常に簡単です。 バックアップファイルには、バックアップ用に選択されたすべてのデータが含まれています。







写真：すべてのバックアップがいっぱいです。

このようなバックアップは最も信頼性が高く、しかも最大です。 この場合、回復に必要なファイルは1つだけです。



増分バックアップ


最後のバックアップ以降に発生した変更のみがバックアップファイルに記録されます。







画像：1.tib-完全バックアップ（最初のバックアップは常に完全）、2.tib、3.tib、4.tib-増分バックアップ。

増分バックアップははるかに完全ではありません。 ただし、復元には、以前の完全バックアップ（図の1.tib）と、データの復元元のバックアップで終わる増分バックアップのチェーン全体が必要です。



差分バックアップ


最後の完全バックアップ以降に発生した変更のみがバックアップファイルに記録されます。







図：1.tib-完全バックアップ（最初のバックアップは常に完全）、2.tib、3.tib、4.tib-差分バックアップ。

差分バックアップは完全バックアップではありませんが、増分バックアップです。 復元するには、差分バックアップ自体と以前の完全バックアップが必要です（図-1.tib）。



チェーンとスキーム



さて、ここで最も興味深いことになります。 もちろん、あなたはすでに推測しました。 3つのバックアップ方法は、いわゆるバックアップチェーンのあらゆる種類のオプションを提供します。 チェーンは、1つの完全なバックアップと、それに依存するすべての増分および/または差分バックアップです。 このスキームは1つまたは複数のチェーンで構成され、古いバックアップを削除するためのルールも含まれています。

実際、チェーンには非常に多くのオプションがあります。 しかし、これは理論上です。 実際には、完全、増分、または差分のいずれかの方法のみがチェーンの基礎として使用されます。







「ここではすべてが白い日として明確です！ 常に完全なバックアップを作成してください！」-あなたは言って、あなたは正しいでしょう。 しかし、いつものように、もう1つ「しかし」があります。 完全バックアップは最も重要です。 2 TBのディスクバックアップを実行して申し訳ありませんか？ 次に、これが最適なソリューションです。 しかし、ほとんどの場合、ディスク領域の損失を最小限に抑えながら、最大限の信頼性と可変性を求めています。 したがって、彼らが言うように、理解しましょう。 ここでは、完全バックアップに基づいたスキームで開始します。



完全バックアップに基づくスキーム


完全バックアップのみを作成することが、データを保護する最も信頼できる方法です。 また、バックアップの無制限の膨張を防ぐことも非常に可能です。 クリーニングルールを設定する必要がありますが、それについては以下で詳しく説明します。

このスキームの欠点：

  • 各バックアップの作成には多くの時間がかかります。
  • ディスク容量の大幅な浪費。
  • 少数のバックアップ、つまり 「ロールバック」できる時点。
  • 異なるバックアップでの同じ情報の複製。


このスキームは、システムパーティションを保護するために使用することをお勧めします。 3つまたは4つのフルバックアップで十分です。



増分バックアップスキーム


このスキームでは、1つの完全バックアップとそれに依存する増分バックアップのチェーンが作成されます。 利点は明らかです-バックアップは迅速に作成され、少し重くなります。 完全バックアップを使用したスキームを使用する場合よりもはるかに多くのファイルを再作成できます。 その結果、リカバリポイントを選択する際に最大の変動性が得られます。 しかし、1つの重大な欠点があります-低い信頼性。 バックアップのいずれかが破損している場合、後続のバックアップはすべてゴミに変わります-バックアップから回復することはできません。 信頼性を向上させる方法はありますか？ はい、できます。 最も簡単な方法は、4〜5回など、いくつかの増分バックアップの後に新しい完全バックアップを作成することです。 したがって、複数のチェーンを持つスキームを取得し、チェーンの1つが損傷しても他のチェーンには影響しません。

このスキームは普遍的であり、ディスクとファイルの両方を保護するために使用できます。



差分バックアップに基づくスキーム


このスキームでは、1つの完全バックアップとその差分バックアップが作成されます。 このアプローチは、以前の2つの利点を組み合わせたものです。 差分バックアップは完全バックアップではなく増分バックアップであるため、リカバリポイントを選択する際の平均的なばらつきとかなり高い信頼性が得られます。 しかし、とにかく欠陥なしではできません。 差分バックアップが完全バックアップから時間的に遠く離れるほど「重く」なり、完全バックアップのサイズを超えることさえあります。 ここでのソリューションは、増分アプローチと同じです-差分バックアップを完全なバックアップで希釈します。 保護されたデータの変更率に応じて、2〜5回の差分バックアップ後に新しい完全バックアップを作成することをお勧めします。

このようなスキームは、ディスクスペースが複数の完全バックアップの保存を許可しない場合にシステムパーティションを保護できます。



計画中



ここではすべてが簡単です。 スケジュールを作成すると、設定されたスキームに従って、設定した時刻にTrue Imageがバックアップを更新します。 データが頻繁に変更されるほど、頻繁にバックアップすることをお勧めします。 たとえば、システムパーティションは1か月に1回バックアップできますが、毎日作業するファイルは毎日またはさらに頻繁にバックアップすることをお勧めします。







もちろん、バックアップを緊急に作成する必要がある場合、予定された時間を待つ必要はありません。 バックアップはいつでも手動で開始できます。



クリーニングルール



実践によると、ユーザーはバックアップのセットアップ時にクリーンアップについてほとんど考えることはありません。 しかし、無駄に。 結局、彼らはバックアップがディスクのすべての空きギガバイトを「食べた」ことを発見します。

バックアップスキームを作成するときに、クリーニングルールを設定できます。 次の3つの基準を構成できます。

  1. バックアップチェーンの最大「年齢」。
  2. バックアップチェーンの最大数。
  3. 最大合計バックアップサイズ。


なぜ設定で、個々のバックアップではなくチェーンについて話しているのですか？ ルールをクリーニングする目的は、古くなったバックアップを削除することです。 チェーンは完全バックアップから始まるため、最初に陳腐化するのは彼です。 あなたが知っているように、あなたが完全に削除すると、中毒者は役に立たなくなるので、チェーン全体が削除されます。



クラウドストレージへのバックアップはどうですか？



これまで説明してきたことはすべて、内部または外部のハードドライブ、NAS、FTPサーバーなどに保存するバックアップを指します。 クラウドへのバックアップはどうですか？ True Imageは、ファイルとディスクの両方のバックアップを単純な増分スキーム（1つの完全バックアップと一連の増分バックアップ）でAcronis Cloudに保存し、変更を許可しません。 合理的な質問「なぜ」は簡単です-このスキームはディスク容量に対して最も質素であり、Acronisはクラウド内のバックアップを保証します。

クラウドバックアップをクリアするためのルールは、通常よりも若干単純です。







「年齢」およびクラウドに保存されている各ファイルのバージョンの数によってバックアップを制限できます。 ストレージボリュームによるバックアップの制限は、あまり論理的ではありません。 結局のところ、まず、Acronis Cloudはバックアップの保存専用に使用されます。



それで、乾燥した残留物には何が入りますか。 自分で決める：

  • 保護するデータの量。
  • これらのデータが変更される頻度。
  • バックアップのためにどれだけの空き容量を与えますか。


これに基づいて、バックアップスキーム、計画、およびクリーニングルールを構成します。 それだけです 他のすべては、Acronis True Imageによって処理されます。
