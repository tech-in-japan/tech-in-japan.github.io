VMware：「静止するかしないか？」、仮想マシンを正しくバックアップする





仮想マシンのスナップショットについて非常に多くの記事が執筆されており、このアクションの理論的な部分が過度に説明されています。 私の記事では、問題の実際的な側面と、VMware vSphereプラットフォームのみに焦点を当てます。



では、なぜ「静止」 *スナップショットが必要なのか、スナップショットは何と一緒に食べるのか、そしてそれらの典型的な問題は何なのか？ スナップショットの見方は、主にバックアップの観点から提示されますが、使用の他の側面をある程度明らかにするようにします。

バックアップにスナップショットを使用する



VMware vSphere環境では、スナップショット作成プロセスは2つのオプションによって制御されます。

  • 仮想マシンのメモリの状態を含むスナップショット

  • 静止しているゲストファイルシステムと呼ばれるスナップショット



VMware vStorage API for Data Protectionを使用して仮想マシンをバックアップする場合、最初のオプションは単純に使用されません。この動作の主な理由は次のとおりです。このオプションの場合、増分バックアップの作成時間とサイズが大きくなります（各増分バックアップにはRAMのサイズが追加されます）。 さらに、多くの技術的な問題がありますが、今日では、私たちにはほとんど興味がありません。 別のシナリオを検討しています。



実際、代替オプションは2番目のオプション-静止です。 これは非常に重要であり、その本質は、バックアップを削除するためにゲストオペレーティングシステム（最初はファイルシステム）を準備することです。



静止とは何ですか？



公式記事を翻訳すると、次のようなものが得られます。

「これは、仮想ディスク上のデータをバックアップに適した状態にするプロセスです。 このプロセスには、オペレーティングシステムメモリからディスクまたは特定のアプリケーションに固有の他の高レベルの操作へのダーティバッファのフラッシュが含まれる場合があります。



仮想マシンで何が起きているのかというこの説明からは、実際には明確ではありません。 自分で考えてみましょう。



まず、VMware Toolsは、VMware Snapshot Providerサービスを通じて、ゲストOS内でVSSスナップショットの作成を開始します。 さらに、ゲストOSに登録されているすべてのVSSライター（「 vssadmin list writers 」コマンドで表示できます）は、要求を受信し、対応するアプリケーションのバックアップを準備します（すべてのトランザクションはメモリからディスクに書き込まれます）。 すべてのVSSライターが終了すると、それをVMware Toolsサービスに報告し（再びVMware Snapshot Providerサービス経由で）、スナップショットを削除できることをVMwareに伝えます。

したがって、VMware vSphereのすべてのバックアップアプリケーションは、コマンドを発行してVMwareスナップショットを作成するときに次の組み合わせを使用します（スナップショットを直接作成するプロセスは、VMware自体によって完全に完全に制御されます）。



静止=オン、メモリ=オフ

静止= OFF、メモリ= OFF



この記事では2番目の組み合わせを検討せず、静止プロセスに焦点を当てます。



なぜ静止が必要なのですか？



最も明らかな例は、コントローラードメインをバックアップから復元する際のUSNロールバックの問題です。 VSSを使用せずに（つまり、静止オプションまたはディスクへの書き込みトランザクションを提供する他の手段なしで）仮想化ドメインコントローラーがバックアップされた場合に発生します。



静止オプションで作成したバックアップを復元する場合、タンバリンを使用した追加のアクションやダンスは必要ありません。 InvocationIDは正しくリセットされ、回復後に読み込まれたコントローラーのイベントログに次のエントリが表示されます。

イベントID 1109：バックアップメディアからActive Directoryが復元されたか、アプリケーションパーティションをホストするように構成されています。 このドメインコントローラーのinvocationID属性が変更されました。



Acronis vmProtect 9を使用する場合も、同様の正しい動作を確認できます。 実際、ドメインコントローラーが内部にある仮想マシンのバックアップと復元の一部として、具体的にテストしました。



未処理のスナップショットを使用する場合、USNロールバックだけが明らかに問題になるわけではなく、他のアプリケーション（たとえば、VSSアプリケーションを明示的にサポートするExchange / SQL）は、そのようなスナップショットから回復するときに失敗する可能性があります。



VSSを使用してスナップショットが正しく作成されたことを確認する方法は？



一貫性のある（アプリケーションレベルまでの）スナップショットの作成の正確性を判断するには、いくつかの方法があります。



最も簡単な方法：ゲストオペレーティングシステムを入力し、「イベントビューアー」を確認します（このような貧弱なイベントビューアーを翻訳する必要がありました）。 オプションquiesced = ON、スナップショットメモリ= OFF（記事の冒頭のスクリーンショットを参照）でスナップショットを作成した後、アプリケーションログに対応するVSSライターからのイベントが表示されます。









注：スクリーンショットに表示されているイベントID 12289のVSSからのエラーは、実際には問題ではありません 。 これは3.5 ''ディスクに属しているため、仮想マシン構成からフロッピーを削除するだけです。







この方法はより複雑です。vSphereクライアントからDatastore Browserコンポーネントを使用します。データストアの仮想マシンフォルダーに、静止スナップショットを作成した後、ファイル*** vss_manifests * .zipが表示されます。



ファイル内には、ゲストシステムで見つかったすべてのVSSライターの説明と、writerX.xmlの各ライターのメタデータを含むbackup.xmlがあります。







重要：vss_manifests.zipにbackup.xmlのみが含まれている場合、これは通常、実際にVSSを使用せずにスナップショットが作成されたことを意味します。 したがって、最も興味深いもの、つまりスナップショットに関する問題の研究にスムーズにアプローチします。 以下に、スナップショットの破損の主な原因を示します。 主な危険性はスナップショットの破損ではなく（検出しやすい）、つまりVMwareが成功と報告するスナップショットではなく、スナップショットはそうではないことです。

次に 、静止が実際に機能するためには、VMware ToolsのVSSコンポーネントが実際にインストールされていることを確認する必要があります（当然、VMware Toolsは最新バージョンでなければなりません）。







静止用のvSphereの古いバージョン（3.5以前）では、Legato Sync Driverも使用され、ファイルシステムレベルでの整合性が保証されましたが、アプリケーションレベル（VSSコンポーネントが必要）では保証されませんでした。 現在、このドライバーは実際には使用されておらず、VMware Snapshot Providerに広く置き換えられています。 インストールは、対応するCOM +コンポーネントのVMware Snapshot Provider +サービスの存在によって、ゲストオペレーティングシステム（仮想マシン上）で確認できます。

VMware Snapshot Providerサービスが無効になっているかまったくインストールされていない場合、VMwareは、静止= ON、スナップショットメモリ= OFFオプションでスナップショットを削除すると、成功したことを報告しますが、実際には、システム内でVSSを使用せずに、つまりLegato Syncを介してスナップショットが生成されますドライバー。







Windows 2008以降の場合、動作が異なることに注意してください。ログにはそのようなイベントはありませんが、ボリュームシャドウコピーサービスのみが開始されてから停止状態になります。



3番目に、一般的な静止構成の問題の1つは、仮想マシンの.vmx構成のdisk.EnableUUID = trueパラメーターです。



この設定は、Windows 2008以降を実行しているゲストシステムでのみ意味があります（Windows 2003では、設定は無視されます）。 追加の機能は、vSphere 4.1以降でのみ新しい仮想マシンを作成するときに、このパラメーターが自動的に登録されるという事実です。 つまり、仮想マシンが古いバージョンのvSphereから移行された場合、設定がない可能性があります。







パラメーターがない場合、またはfalseに設定されている場合、スナップショットを作成するときの動作は前のケースに似ています：スナップショットは正常に作成されますが、実際にはVSSは使用されず、結果として一貫性のないバックアップを取得できます。 無効なパラメーターの2番目の症状は、vss_manifests.zipにある空のbackup.xml（VSSライターの説明なし）です。



4番目に 、ゲストマシン内のダイナミックディスクを確認します。 ゲストシステム内に少なくとも1つのダイナミックディスクが存在する場合-システミックであるかどうかに関係なく、VSSは関与しません。 スナップショットは正常に作成されますが、vss_manifests.zipは空になり、ゲストOS内のイベントログも空になります。 このルールは、Windows 2008以降のゲストOSに適用されます。



同じことがIDEディスクにも当てはまります-それらは仮想マシン構成にあるべきではありません（ただし、IDE CD-ROMデバイスの存在は有効であり、スナップショットには影響しません）。 1つのSCSIコントローラの空きSCSIスロットの数は、ディスクの数と等しくする必要があることに注意してください。 たとえば、SCSI1に既に8つのSCSIディスクがある場合、十分なスロットがありません。



5番目 ：ゲストマシン内でVSSをアイドルにします。 これは、多くのresみとVMwareのテクニカルサポートへの呼び出しを引き起こす主要なポイントです。 多くの場合、失敗したスナップショットをVMwareで見る人がいますが、まったく異なる思考の巨人であるMicrosoftを非難する価値はあります。 新しいSQLデータベースのインストールが失敗した後、マシンの静止スナップショットを作成しようとしたときに同じ画像が得られました（インストール中に仮想.isoドライブがアンマウントされましたが、インストーラーは本当に気に入らなかった：-\







この問題は、仮想マシンの通常の再起動によって解決されました。この方法は非常に頻繁に役立ちますが、VSSの内部が完全に壊れない場合があります。 これらの場合、Microsoftが本当に責任があるかどうかを確認する最も簡単な方法は、Windowsバックアップを起動し、システム状態のバックアップを作成することです（誰かが英語の用語に慣れている場合は、システム状態のバックアップ）。 Windowsバックアップ（またはNTBackup）は動作します-問題はVMware側にあり、動作しません-マイクロソフトはできません。
