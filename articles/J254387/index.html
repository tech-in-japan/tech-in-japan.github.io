AggreGate Network Manager：プラットフォーム+ ITインフラストラクチャの包括的監視用ボックス









複雑なITインフラストラクチャの信頼性とパフォーマンスの問題を診断するには、すべてのプロセスとイベントの全体像を把握し、重要な傾向を追跡する能力が必要です。 統一された（「傘」）ITインフラストラクチャ管理システムを実装する必要性は、従来のビジネス上の理由により正当化されます。







  1. ビジネスアプリケーションおよびサービスへの投資収益率を最大化する 。 管理システムにより、企業のITサービスの運用に必要なリソースを評価および最適化できます。 これにより、最小限の運用コストと新しいITサービスの迅速な回収が保証されます。
  2. ITサービスの品質の向上 。 ITサービスの品質を維持および改善するには、その現在の状態を詳細に理解する必要があります。 多くの場合、パフォーマンスの低下は、自社のサービスによって長い間気付かれず、顧客の評判と信頼を損ないます。 自動監視は、定義されたサービスレベル契約（SLA）のサービスのコンプライアンスに関する必要なレポートを提供し、検出されない劣化のリスクを最小限に抑えます。
  3. サービスおよびITインフラストラクチャコンポーネントのダウンタイムを削減します 。 プロアクティブな監視により、多くの機器やアプリケーションの障害を防ぎ、サービスの継続時間と低下の度合いを最小限に抑えることができます。 障害が発生した場合、継続的な監視システムにより、問題のあるコンポーネントを迅速に特定して修正できます。 ビジネスサービスが利用できない状態が数時間続くことによる経済的損失は、常に適切に見積もられるとは限らず、多くの場合、IT管理システムの実装コストと同程度です。
  4. ITインフラストラクチャ運用部門の従業員の効率を改善する 。 現在の状況の分析や構成の管理などの日常業務からITシステム管理者を解放し、インシデントをなくすための緊急タスクを管理システムにより、従業員は1回限りのタスクに解放された時間を使用してプロセスをさらに自動化でき、最終的には増加につながります一般的な業績。




ほとんどすべてのITディレクター、ITインフラストラクチャ運用部門の責任者、およびシステム管理者は、上記の内容を（直感的または意識的に）理解しています。 それでも、ITインフラストラクチャの「成熟度」の程度は、国ごと、産業ごと、企業ごとに大きく異なります。 また、IT管理および監視の自動化へのアプローチは、ITインフラストラクチャ全体の開発の程度に大きく依存します。



ITインフラストラクチャを管理するアプローチの開発のいくつかのレベルを条件付きで区別します。



  1. 監視のない生活 。 すべてのインフラストラクチャコンポーネントとサービスは、運用管理なしで独自に機能します。 インシデントの除去は、インフラストラクチャとサービスの個々のコンポーネントに接続し、何が起こったのかを管理者の「頭の中で」把握し、報告する-通常はログによって行われます。

  2. 事後監視 。 もちろん、「反応」という言葉ではなく、「反応」という言葉から。 この段階で、古典的な初級クラスの監視システムが導入されます。 これは、単純なpingovalkaでも、標準プロトコル（SNMP、WMIなど）またはエージェントを使用してインフラストラクチャコンポーネントを定期的にポーリングするシステムでもかまいません。 収集されたデータは保存され、ダッシュボード（ダッシュボード）を使用して視覚化されます。 イベント（SNMPトラップ、Syslog）の最も単純な統合、および重大なイベントとしきい値違反の通知も実行されます。
  3. プロアクティブな監視 。 インシデントの発生がすぐに発生せず、監視システムがオペレータに否定的な傾向を警告する機能を備えている場合、インシデントを予測し、それらを回避するという点で事後対応型とは異なります。 プロアクティブな監視システムは、通常、動的アラームのしきい値、トラフィック分析、SIPコールによるVoIP監視、IP SLAデータ収集などの追加機能も提供します。
  4. 統合IT管理 。 これまでのところ、ロシアのこの段階では、少数の大企業が存在します。 包括的な、つまり「傘」のIT管理の一部として、1つのシステムは、インフラストラクチャとサービスの監視のすべての機能と自動構成管理の機能を組み合わせ、同時にITSM / Service Deskクラスの企業システムとの統合の単一ポイントを提供します。 このクラスのシステムは、イベントチェーンの複雑な相関のシナリオと、資産インベントリシステムからのデータとの強化のシナリオを実装します。 その結果、根本原因分析技術でシステムをトレーニングすることが可能になります。 事前定義された根本原因検索アルゴリズムのセットは、通常、箱入り製品の一部です。




世界市場のIT監視システムとネットワークの数は非常に多いです。 ただし、ほとんどは、トラフィック監視および分解ソリューション、Javaアプリケーション監視ソリューションなどの高度に専門化された製品です。



普遍的な複雑なシステムについて話すと、円は急激に狭くなります。 Wikipedia（ ロシア語版 、 英語版 ）の記事「ネットワーク監視システムの比較」では、多かれ少なかれよく知られた無料および商用のソリューションが言及されています。



すべての汎用IT監視ソリューションは、条件付きでいくつかのカテゴリに分割することもでき、それぞれに長所と短所があります。



  1. 無料のオープンソースシステム 。 このクラスには、たとえば、Nagios、Cacti、Zenoss、Zabbix、Pandora FMS、OpenNMSが含まれます。 システム自体のコストは、総所有コストよりも常に低いとは限りません。構成ファイルを介したチューニングや、多くの監視シナリオ用のスクリプトを作成する必要性は、最終的に高価です。 管理者の労働時間を支払う費用は間接的であり、長期間にわたって分配されるため、多くの場合、企業には見えないままです。 同時に、このクラスのシステムには、低コストの商用システムよりも柔軟性に優れた利点があります。オープンソースコードにより、経験豊富なネットワークエンジニアは、システムを追加し、有料のテクニカルサポート契約を締結することなく、従来のモニタリングのほとんどのタスクを解決できるためです

  2. 機能が固定された商用製品 。 このセグメントには、WhatsUp Gold、OpManager、Solarwinds、PRTG、AccelOps、AdRem NetCrunch、op5 Monitor、SevOneなど、ほとんどの商用ネットワーク監視ソリューションが含まれます。 このような商用システムは通常、非常に迅速に展開され、標準的な問題をすぐに解決できます。 ほとんどの製品は、標準の監視シナリオの一部として、使いやすさが向上しています。 このようなすべてのシステムの主な欠点は、柔軟性が低く、非標準タスクに適応する能力が低いことです。 多くの場合、製品の最も完全なバージョンが購入されたとしても、「XをYで監視する方法とZを確認する方法」という質問に対する答えは、製品の将来のバージョンで目的の機能を実装することを期待して、ベンダーとの長い会話の見込みがあるため、堅実な「方法」です。
  3. プラットフォームタイプの商用製品ライン 。 このセグメントのプレーヤーは、IBM、CA、HP、EMCといった最大の西洋企業です。 この場合、決定は1つの特定の製品としてではなく、監視および制御タスクのさまざまなグループをカバーする一連の製品として理解されます。 残念ながら、これらの製品はサードパーティ企業によって開発されたことが多く、その後吸収されました。したがって、単一の管理コンソールのレベルで統合された製品は、まったく異なる方法で配置、構成、動作します。 このような製品セットを導入するコストは非常に高くなります。 それにもかかわらず、このタイプのシステムは、「すぐに使える」監視タスクの最大数をカバーし、幅広い非標準タスクを解決できます。




一般に、IT管理システムの市場は、すべてのソフトウェア市場の中で最も競争の激しい市場の1つです。 では、なぜこのクラスの別のシステムであるAggreGate Network Managerの開発に投資することにしたのですか？



この質問に対する答えは完全に些細なものではありません。 監視および制御システムを作成するためのコンストラクターであるAggreGateプラットフォームがあります。 AggreGate SCADA / HMIやAggreGate Fleet Managerなどの垂直市場向けのソリューションの多くは、いくつかの新しいドライバーを変更してデータを収集し、新しいソリューションを美しいマーケティングキャンディラッパーにパックすることで構築されています。 SCADAとは異なり、ネットワーク監視システムには、多くの新しいデータ収集メカニズム（WMI、IMPI、SSH、JMX、SIP、NetFlow、Syslogなど）と、多数の事前定義されたポーリングテンプレート、アラーム、ダッシュボードの両方が必要です。 したがって、投資は当初かなりの額になると約束されていました。

カスタムIT管理ソリューションのプラットフォーム





カスタマイズオプションの点では、Network Managerは他のすべての製品とは大きく異なります。 製品は、標準のAggreGateデータ処理および視覚化ツールを使用して完全に開発されています。 簡単に言えば、データ収集モジュールを除くIT管理システム全体が、システムおよびビジネス分析、および実装部門の従業員によって開発されました。 それらの作業の結果は、何百ものレポートテンプレート、ウィジェット、アラームであり、それぞれをシステムエディターで開き、「自分用」に変更できます。



この機能は、アラームしきい値の変更などの最も単純な設定だけでなく、製品の動作のほとんどすべての変更にも適用されます。 1つの例を見てみましょう：プロセッサ負荷統計の監視。



トップ10ダッシュボードには、CPU使用率が最大の10個のデバイスを示すテーブルがあります。 ダッシュボードアイテムの設定は、管理者がオブジェクトツリー内の個別のオブジェクトとしてアクセスできるリクエストを参照します。 要求テキストは次のとおりです。 



SELECT info.info$description AS device, info.cpuLoad$cpuLoad AS processor_utilization_percentage FROM users.*.devices.*:cpuLoad:info:genericProperties AS info WHERE (info.cpuLoad$cpuLoad IS NOT NULL) AND (info.genericProperties$suspend = 0) ORDER BY processor_utilization_percentage DESC LIMIT 10




このクエリは、すべてのデバイス（ ユーザーに対応。*。デバイス。*マスク）のAggreGateプラットフォームの統合データモデルからcpuLoad変数の値をすぐに取得します。したがって、降順でソートされ、最初の10行に制限されるCPU負荷サマリーテーブルを作成します。 たとえば、特定のタイプのデバイスまたは特定のグループのデバイスのみを表示するように、クエリテキストを簡単に変更できます。



cpuLoad変数はどこから来たのですか？ 監視システムのカーネルは、SNMPデータに基づいて計算するのでしょうか？

しかし、違います。 ビジュアルエディターで完全に作成された別のリソースは、CPU負荷モデルです。 モデルの個々のインスタンスが各ネットワークデバイスに接続されるように構成されます。 このインスタンスは、レポートがデバイスを参照するcpuLoad変数を追加します。 さらに、モデルは、デバイスイメージで使用可能なデータに応じてCPU負荷を計算できるビジネスルールのセットを定義します。



  • デバイスがHOST-RESOURCES-MIB（Windows、Linuxなど）をサポートしている場合、すべてのCPUコアの現在の平均負荷値が取得されます
  • デバイスでWMIが有効になっている場合、Win32_PerfFormattedData_PerfOS_ProcessorクラスのWMIデータが使用されます
  • シスコは、デバイスの計算された過去5分間の平均を使用します。
  • SolarisはCPUのユーザーとシステムの負荷を合計します
  • もう少し複雑な、HP / UXシステムは、計算されたCPU負荷値の代わりに、スイッチがオンになってからプロセッサーがビジーであったミリ秒のカウンターを提供します。 中間分析では、統計チャネルと呼ばれるAggreGateプラットフォームの別のオブジェクトが使用されます。 ミリ秒カウンターデータを変化率に変換し、統計をリングデータベース（RRD）に保存します。 次に、HP / UXのCPU負荷を計算するルールは、最後の1分間にチャネルで計算された平均CPU負荷を参照します。
  • など、製品の出荷には、多くの一般的なデバイスの負荷を計算するためのルールが含まれています。












たとえば、Huaweiなどのルールエディターを使用して新しいルールを追加すると、標準製品に含まれるプロセッサーをロードするためのすべてのアラーム、レポート、ダッシュボードは、CPU負荷モデルも参照するため、Huawei機器と自動的に互換性があります。



監視システムの他のすべてのモジュールも、製品コアへの「ハードコーディング」ロジックの代わりに標準プラットフォームコンポーネントを使用するという原則に基づいて作成されています。 たとえば、スケジュールされたネットワークスキャンを開始するには、カスタム（および削除された）システムスケジューラタスクが使用されます。



Network Manager製品のセットアップ画面とメインオペレーター画面を実装するダッシュボードは、システムのセットアップと日常のタスクを効率的に解決できるようにすることに関連するすべての困難を一般ユーザーから隠すように設計されています。



この記事の目的は、システムのイデオロギーを説明することです。「オーバーボード」には、非常に表面的な製品機能のリストさえあります。 ハブはこれに適したサイトではないと思います。 したがって、以降の記事では、さまざまな非標準の問題を解決する方法の詳細な説明と同様に、参加できた興味深いプロジェクトに注目します。
