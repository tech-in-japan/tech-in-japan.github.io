Snapshot Hunter Snapshot Detector-Veeam Backup＆Replication v8統合機能

最近の記事で取り上げたスナップショットのテーマを続けて、今日は、Veeam Backup＆Replicationがスナップショットの「環境」への影響を最小限に抑える方法について説明します。



さまざまな理由で、vCenterからスナップショットが「見えない」ことがあります。スナップショットに関する情報はレポートに表示されず、UIのどこにも表示されませんが、スナップショット自体はライブおよびストレージでライブになります。 仮想マシンはこのようなスナップショットを冷静に使用します。これにより、ストレージシステムの「使い果たされた」スペースとパフォーマンスの低下により、問題が発生する可能性があります。 「見えない」スナップショットのせいで生じた問題は、サポートに連絡する一般的な理由です。



スナップショット（表示および非表示）の処理方法については、Catを参照してください。









赤ん坊の息子はスナップショットを作成し、赤ん坊に尋ねました：「スナップショットは良いですか、悪いですか？」



ご存じのとおり、VMware環境では、スナップショットはストレージ用の追加ディスクです。 前のポイントが「読み取り専用」状態にある間、すべての書き込み操作を実行します。 利点に加えて、スナップショットの存在は非常に心地よい結果をもたらします：スナップショットが占有するディスク容量は、ストレージシステムの貴重なストレージアレイから奪われ、異なる仮想ディスクで読み取り/書き込みが実行されるため、仮想マシンのパフォーマンスが低下する可能性があります。



VMware管理者は、仮想マシンのスナップショットを長時間読み取りおよび書き込み用に開いたままにしておくと、仮想インフラストラクチャが効果的に機能するために問題が発生する可能性が高いことを知っています。 したがって、そのようなスナップショットの可用性を監視することは合理的です。たとえば、Veeam ONEソリューションに含まれるVMwareレポートパッケージからアクティブスナップショットレポートを定期的に起動します。







これは、スナップショットで状況を制御するための良い方法です。 ただし、状況が十分でない場合があります。



「見えない」スナップショットとは誰ですか？



よく見てみましょう。VeeamBackup＆Replicationでは、バックアップまたはレプリケーションのタスクはすべて、仮想マシンのスナップショットの作成から始まります。 この方法により、「フリーズ」（静止）を正しく実行できます。つまり、 仮想ディスクに保存されたデータの処理を完了するために-これにより、バックアップデータの一貫性が保証されます。 そのため、まず、Veeam Backup＆Replicationはスナップショットの作成要求をvCenterに送信します。 これが発生した後、「凍結」仮想ディスクのデータは、全体または一部がコピーされます（増分バックアップが実行される場合）。 次に、VeeamはvCenterにコミットリクエストを送信します-つまり データのコピー中にスナップショットデルタファイルに書き込まれたすべての変更をVMDKに加え、スナップショットを削除する必要があります-この手順は統合と呼ばれます。



ここで、このようなイベントの切り替えが可能です：vCenterがスナップショットの削除を報告したとしても、実際にはこの削除は行われず、閉じられていないスナップショットは生きたままでした（したがって、vCenterはそれについて知りませんでした）。 たとえば、仮想マシンディスクがHotAddプロキシにアタッチされているときにスナップショットを削除しようとすると、「非表示」スナップショットが表示される場合があります（HotAddプロキシの詳細については、 こちらを参照してください）。



Veeam Backup＆Replication v8では、「見えない」スナップショットの問題を克服することができました。これは、バックアップまたはレプリケーションパスを残す可能性があります。 これは、 スナップショットハンター （「スナップショットハンター」）と呼ばれる機能を使用して行われました。 「ハンター」はそのようなスナップショットを追跡し、それらを自動的に削除します。



画面には何が見えますか？



vSphereが仮想マシンのスナップショットでの作業を完了する（または完了すると考える）とすぐに、対応する通知がクライアントUIに表示されます。







ここで、スナップショットハンターは仮想インフラストラクチャに接続し、この仮想マシンのファイルが保存されているストレージシステムからデータを読み取ります。 バックアップ中に作成されたスナップショットがまだハングしている場合、これに関する情報がVeeam Backup＆Replicationコンソールの現在のタスクのセッションに表示され、自動統合プロセスが開始されます。







履歴ビューで「ハンター」の作業を確認できます。左側のツリーで、 システムノード（システムタスク）を見つけ、検索バーを使用してシステムタスクのリストを除外します。 スナップショットの統合に取り組んでいるタスクのリストが右ペインに表示されます。 このような各タスクはスナップショットハンターです。







これはどうなりますか？



タスクの処理が終了し、スナップショットのコミットに関するレポートが完了した直後（Veeam Backupに関して、「タスク」は1つの仮想マシンまたはマシンに複数の仮想マシンがある場合は1つの仮想ディスクであることを思い出します）、次の条件がチェックされます：

  1. vSphereでは、仮想マシンの[ 統合の必要性]属性の値を確認します。その値が[ はい]の場合、データ統合が発生していないことを意味します。
  2. vCenter Serverデータベースに対して、登録されたスナップショットの数とデルタディスクの数をチェックします-これらが異なる値である場合、これは統合が機能しなかったことも意味します。


検証により、強制的な統合が必要であることが明らかになった場合、Veeam Backupサービスは、システムジョブ（つまり、スナップショットハンター「ハンター」）を別のスレッドで起動することを計画しています。

「ハンター」は、次のアルゴリズムに従って動作するスナップショットの強制統合および削除の手順を開始します（各ステップの後、上記の条件に従ってチェックも実行されます）。

  1. 最初に標準ツールを使用してみます- 統合 （仮想マシンのvSphereクライアントでスナップショット>統合コマンドを選択したときに機能する同じネイティブVMwareメカニズム）を使用します
  2. これが役に立たない場合は、緊密な統合、つまり「スナップショットを作成してから削除」という一連の操作を実行します-VMwareによると、これにより、元に関係なくすべての「表示されていない」スナップショットが強制的に削除されます（表示されているスナップショットのみが変更されずに残ります-例：ユーザーが作成）
  3. 最後に、「フリーズ」で緊密な統合を実行します。つまり、「スナップショットを作成してから、「フリーズ」で削除します（目に見えないスナップショットに同じ効果があります）。


このアルゴリズムで最初に目的の結果が得られない場合（検証では、たとえば、 Needs Consolidation = Yesと表示されます）、4時間の間隔でさらに2回試行されます。



12時間後でもスナップショットを正しく削除できない場合（手順1〜3を完了した後）、Veeam Backupは「ハングした」スナップショットの存在を書面でユーザーに通知します。 つまり、一般設定でメール通知を送信するように設定している場合（ ここで説明します ）、次の内容のメールを受信します。

" VM virtual_machine_nameにはスナップショットの統合が必要ですが、すべての自動スナップショット統合の試みは失敗しました。

最も可能性の高い理由は、何らかの外部プロセスによって仮想ディスクがロックされていることです。 ロックの問題をトラブルシューティングし、vSphere Clientでスナップショットの統合を手動で開始してください。 」



または、ロシア語で：

" VM VM_name仮想マシンの場合、スナップショット統合を実行する必要があります。自動統合の試みは成功しませんでした。ほとんどの場合、外部プロセスにより仮想ディスクがロックされます。仮想ディスクのこの状態の理由を特定し、次を使用して手動でスナップショット統合を実行してくださいvSphere Client。



そして、手動モードで自分で統合を実行する必要がある場合はどうなりますか？



スナップショットの統合と削除を正確に妨げるものを見つけ、根本原因を排除した後、 VMwareが規定する手順に従うことをお勧めします。



これはどのスナップショットで機能しますか？



スナップショットハンターは、すべてのバックアップおよびレプリケーションタスクに対して起動されます（スナップショットは、ソース側で追跡されます）-通常のものと、ストレージスナップショットを使用して実行されるものの両方。 また、vCloud Directorを使用したバックアップおよびVeeamZIPでも機能します。 同時に、Veeam Backup＆Replication中にこれらのタスクによって作成されたスナップショットのみが「キャッチ」され、たとえばユーザー自身が作成したスナップショットは影響を受けません。



このようなメカニズムは、古いバージョンのVeeam Backup＆Replicationタスクの作業中に発生した「見えない」スナップショットを識別するためにも使用されます。



スナップショットハンターのパフォーマンスはパフォーマンスに影響しますか？



インフラストラクチャリソースに関連して、スナップショットハンターは非常に人道的に振る舞います：たとえば、複数のマシンがバックアップタスクに含まれている場合（または1台のマシンが複数の仮想ディスクを使用している場合）、スナップショットは並行して追跡および削除されます ただし、ストレージシステムがレイテンシの観点から過負荷であることが判明した場合（つまり、読み書きの強度がしきい値に達した場合）、操作の強度が低下するまで「ハンター」は統合を開始しません。



また、バックアップタスクを担当するスケジュールが構成されてアクティブ化されている場合、「ハンター」はバックアップウィンドウ（バックアップウィンドウ）を考慮します（ ここで説明します ）。 この場合、スナップショットハンターは、統合を実行する前に、バックアップ用に予約された「ウィンドウ」が閉じているかどうかを確認します。 3回の試行のいずれか（最初の試行を含む）で統合が「ウィンドウ」に収まらないことが判明した場合、統合は起動されず、代わりにユーザーは手動介入の必要性に関する通知を受け取ります。



彼のボタンはどこですか？



スナップショットハンターはデフォルトで常にオンになっています。つまり、手動で何かを構成する必要はありません。 これを無効にする必要がある場合は、レジストリキーHKLM \ SOFTWARE \ Veeam \ Veeam Backup and Replicationで値DisableAutoSnapshotConsolidation（DWORD）= 1を設定することでこれを行うことができます。 ただし、統合が必要であることが判明した場合、 VMware KB記事で説明されているように、ユーザーは統合を個別に実行する必要があるという通知を受け取ります 。
