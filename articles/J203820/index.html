さまざまな場所に分散された大量のデータを使用してクライアントを操作し、何らかの方法で収集する必要がある、真面目なビジネスアプリケーションを作成する必要があるとします。 このタスクを実装するには、多くの新しいテクノロジーとアプローチを学び、多くのコードを書き、デバッグする必要があります。 結果は何ですか？ 失礼な場合は、アプリケーションのサーバー側からクライアントにデータを送信するだけです。 さて、データベースを操作します。 そして、まず何に焦点を当てたいですか？ データ転送の実装について？ または、このデータを処理、操作、提示するロジック上で？ 2番目を選択すると思います。



WCF RIA Servicesは、Silverlightに基づいたデータ転送の自動化とマルチレベルアプリケーションの作成を提供し、開発者の作業の論理に注意を集中させます。 サーバーコードのコピーを1回だけ記述すると、自動的に移植され、クライアント側で使用可能になります。開発者が手動で複製したり、他のトリックを適用したりする必要はありません。 また、クライアントでは、検証、アクセス権など、すべてのサーバー機能が引き続き利用可能です。



このシリーズの記事では、WCF RIAサービスの主な機能に精通し、実際にこのアプローチが提供する利点を確認します。 以下の図は、WCF RIAサービスが担当するものを示しています。





今日は、標準で最もシンプルなSilverlightテンプレートを使用します。 後続の部分では、アプリケーションを完成させ、新しい手法、戦術、技術を使用して知識を増やします。その1つはModel-View-ViewModel（MVVM）パターンと単体テストです。

イベントを入力し、名前、アーティスト、プロジェクト、開始日と終了日を入力できる電子ノートブック。 一般に、複雑なことや異常なことは何もありませんが、何をする必要があるかをよく理解することは、研究対象の技術の枠組み内でこれを行う方法の理解を促進するだけです。



WCF RIAサービスの学習面：



  • プロジェクトリンク-Silverlightクライアントプロジェクトとサーバー側のWebアプリケーションまたはライブラリクラス間のリンク。 サーバープロジェクトは、ドメインサービス、エンティティ、メタデータへのリンクを定義または作成し、サーバーで定義されたコードにクライアントがアクセスできるメカニズムも提供します。 リンクのおかげで、クライアントに必要なすべてのコードはコンパイル時に動的に生成されます。



  • ドメインサービス-WCF RIAサービスのコア。 サーバーで実行できる操作を決定します。 基本的には、CRUD（書き込み、読み取り、更新、削除）になりますが、必要だと思われる他のものでもかまいません。 これらの操作はすべて、WCFのおかげでクライアント側で自動的に利用可能になり、クライアントはサーバー側についてほとんど何も知る必要がありません。



  • エンティティ-サーバー側でエンティティのタイプを決定する必要があり、クライアント側は自動コード生成のおかげでそれらを操作します。 検証およびその他のメタデータを使用することもできます。 サーバーとクライアント側のエンティティタイプは、それらの間でデータを転送するときに、シリアル化/逆シリアル化操作に使用されます。 エンティティは、Entity Framework、LINQ to SQL、またはPlain Old CLR Objects（POCO）を使用して作成できます。



  • ドメインコンテキスト-ドメインサービスのクライアントアナログ。 コードを自動生成します。これにより、サーバー側の機能にアクセスできます。 これには、サービスを呼び出す機能を提供するWCFプロキシが含まれています。また、サーバー側で実行する必要があるリクエストを管理し、エンティティの変更などを追跡します。



  • DomainDataSourceは、クエリを実行し、変更をサーバーに通知するために使用されるデータソースオブジェクトです。 XAMLなどでの直接バインドが促進されます。


手順1：Silverlightアプリケーションとサーバー側へのリンクを作成します。



タスクとすべての関連情報を保存する単純なデータベースを使用します。 これを行うには、プロジェクトの馬にあるTaskManager.sqlスクリプトを実行し、データベースとスキーマを作成し、データベースに初期値を入力する必要があります。



「TasksManager」という名前の新しいプロジェクトを作成します。 WCF RIAサービスを使用する機能は、あらゆるタイプのSilverlightアプリケーションに固有のものです。 ただし、最も単純なものから始めて、「Silverlight Application」テンプレートを選択しましょう。





[OK]ボタンをクリックすると、次のダイアログボックスが開きます。ここで、[WCF RIAサービスを有効にする]にバーディーを追加する必要があります。





そのおかげで、サーバープロジェクトへの必要なリンクが作成され、コード生成や前述のその他の機能が表示されます。

[OK]をクリックして、プロジェクトの作成を待ちます。

ステップ3：ドメインサービスを作成する



次に、RIAサービスカーネル-ドメインサービスを作成しましょう。 サーバー側を右クリックします：TaskManager.Web-「追加」-「要素の作成」。 左側のウィンドウで、カテゴリ「Web」を選択し、右側で「DomainService Class」を選択します。 「 TasksDomainService 」という名前を付けます。





[追加]ボタンをクリックした後、サービス設定ウィンドウに移動します。 ここからは、タスクテーブルでのみ作業するため、それを選択します。 「編集を有効にする」列では、チェックマークは不要です。 またね あなたがそれを置くなら、CRUDのセットを取得します。 そうでない場合は、Get（読み取り）のみ。 OKをクリックします。



コメントなしで生成されたコードは次のようになります。 

  [EnableClientAccess()] public class TasksDomainService : LinqToEntitiesDomainService<TasksModelEntities> { public IQueryable<Task> GetTasks() { return this.ObjectContext.Tasks; } }




LinqToEntitiesDomainServiceクラスは、EDMモデルとRIA WCFサービス間のリンクを提供します。 属性 "EnableClientAccess"は、自動コード生成中にこのメソッドをクライアントパーツに追加する必要があることを示します。 この場合、テーブルからデータを読み取る方法しかありませんが、必要なデータに対して完全なCRUDまたはその他のアクションを実装でき、最終的にはクライアントプロジェクトでも使用できます。 これを行う方法については、次の記事で詳しく検討します。



また、戻り値のタイプ（Queryable）および命名規則（GetName）に関連する規則が使用されることを忘れないでください。 この場合、ObjectContextデータモデルの委任されたエンティティがあります。これは、基本クラスの一部として作成され、Tasksテーブル全体を取得するために使用されます。 並べ替え、フィルターなどを追加することでこのクエリを改善することもできますが、戻り値はIQueryable型ですが、WCF RIAサービスはこれを理解し、この機能をクライアント側で使用できる適切なメソッドを形成します。



プロジェクトを再構築して、すべてが正しく行われるようにします。 クライアントプロジェクトでは、2つのシートのアイコン「すべてのファイルを表示」をクリックすると、クライアント用に生成されたコードを含むTaskManager.Web.g.csという名前のファイルが表示されます。
