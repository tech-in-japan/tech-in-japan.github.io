未決定のMicrosoft App-Vによるアプリケーション仮想化





サーバー仮想化は私たちの生活に長くしっかりと入っています。 その利点は否定できず、さまざまなビジネス分野で広く使用されています。 ただし、アプリケーションの仮想化は、ロシアの企業市場では依然として「新しい」ものです。 多くの人がこの技術に注目しているだけで、すでに世界で定着しています。 最大の外国メーカーや金融機関の多くが、オフィス全体の作業を仮想化アプリケーションに移行するのはなぜですか？ 答えは簡単です-有益だからです。



少し後に、仮想化テクノロジーが提供するいくつかの重要な利点をリストし、アプリケーションの所有コストを大幅に削減します。まず、次のシナリオを検討します。



このような移行は、多くの場合、複雑で痛みを伴うプロセスです。 新しいファイル形式、残りのソフトウェアとの相互作用の新しいスキーム。 機器の一部が古いバージョンのみをサポートしており、迅速に交​​換できないことが判明する場合があります。 作業環境に新しいアプリケーションを入力する場合、オーバーレイは避けられません。ビジネスのダウンタイムを避けるために、古いバージョンにロールバックする必要さえあります。 顧客アプリケーションのサポートを提供する当社のスペシャリストは、これらの問題に直接精通しています。 仮想化の助けを借りて、Microsoft Officeのような大きくて複雑なパッケージであっても、プロセスを大幅に簡素化することができます-同じユーザーの下で異なるバージョンを同時に実行し、必要に応じて、コンポーネントを追加または削除することで構成を簡単に変更します-すべてこれを長時間のダウンタイムなしで実行します。







テクノロジーの仕組みを詳しく見てみましょう。同時に、アプリケーション仮想化の主な利点をリストします。



-インストーラーを介してコンピューターにアプリケーションがインストールされない。

「仮想化」のプロセスでは、プログラムはクリーンなOSイメージにインストールされ、インストールプロセス中に、レジストリおよびファイルシステムのすべての変更が特別なパッケージに記録されます。 実際、展開されたアプリケーションであるこのパッケージは、ユーザーのコンピューターに配信され、オペレーティングシステム自体に痕跡を残すことなく、隔離された環境で実行されます。 この環境は、「サンドボックス」（サンドボックス）または「バブル」（仮想バブル）と呼ばれます。 同時に、このようなバブルで起動されたアプリケーションは、通常のプログラムを「認識」し、それらと対話できますが、それ自体は「見えない」ままです。



-アプリケーションの相互分離。

各仮想アプリケーションは独自の個別の環境で実行されるため、レジストリブランチの上書きや複数の独立したプログラムによるファイルの置換に関連する競合を完全に排除します。 大幅に削減されるか、むしろ、いくつかの非常に大きなアプリケーションの共同テストに不要なコストになります。 必要に応じて、仮想アプリケーションをグループにまとめてから、相互にやり取りできます。 ただし、この場合でも、プログラムが安定して機能することを確認するために、互換性チェックを一度だけ行う必要があります。



-1つのコンピューターで1つのアプリケーションの異なるバージョンを同時に実行する機能。

仮想アプリケーションは互いに分離されているため、同じプログラムの異なるバージョンを仮想化し、同じアカウントの同じコンピューターで実行することを妨げるものはありません。 両方とも機能します。 前述のように、このようなシナリオは、ソフトウェアの古いバージョンから新しいバージョンに切り替えるとき、または教室やテストラボで非常に一般的です。



-通常のインストール中にリソースの共有でエラーが発生した場合でも、 複数のユーザーが1つのアプリケーションを同時に起動する機能 。



-ユーザーにローカル管理者の権限を提供したり、非標準アプリケーションのレジストリへのアクセス権を強化したりする必要はありません。

分離された仮想バブルでは、アプリケーションはすべてのファイルとレジストリキーに完全にアクセスできます（以前のバージョンのApp-Vでは制限がありましたが、最新版では削除されました）。 したがって、以前に必要だった場合でも、昇格した特権で仮想アプリケーションを実行する必要はありません。 これにより、作業環境の安全性が向上します。



-Active Directoryを介して管理される、目的のユーザーグループに対する新しいアプリケーションの即時可用性。

展開されたApp-Vインフラストラクチャでは、ユーザーのグループを仮想アプリケーションにマッピングするだけで十分であり、指定された時間が経過すると、ユーザーのスタートメニューにショートカットが表示されます。 さらに、制御は、任意のブラウザーからWebインターフェースを介して実行されます。 同様に、必要に応じてアプリケーションが起動しないようにすることができます。 また、インストールの待機中にコンピューターを再起動したりダウンタイムしたりする必要はありません。



-障害が発生した場合に作業環境を復元するための時間の大幅な短縮 。



-稼働中のシステムのイメージの管理の簡素化。

アプリケーションを仮想化すると、それぞれが独自のソフトウェアセットを備えた異なるユーザーグループ用のOSイメージを用意する必要がなくなります。 1つの「バニラ」方法に制限し、Active Directoryグループを使用してアプリケーションを割り当てるだけで十分な場合があります。



ここで、現時点で非常に関連性の高い別のシナリオ-あるオペレーティングシステムから別のオペレーティングシステムへの移行 -に言及することを怠ることはできません。 多くの人がこの複雑なプロセスを最後まで延期しており、懸念は明らかです。 互換性の問題、IT部門の大きな負荷、ビジネスのダウンタイムの場合の深刻なリスク。 当社はすでにヨーロッパの8大顧客のインフラストラクチャで移行プロセスを完了しており、アプリケーションの仮想化によりプロジェクト時間が大幅に短縮されると確信できます。 64ビット環境と32ビット環境の両方で動作するようにアプリケーションを構成し、それらが互換性があり、異なるプラットフォームで動作することを確認したら、OSを変更するときに複雑な最終ユーザー作業環境をすばやく構成できます。



次に、Microsoft Application Virtualization（App-V）5.0の例を使用して、仮想アプリケーションのサポート構造を拡張するために必要なものをリストします。



ライセンス：既にリモートデスクトップサービス（RDS）クライアントライセンスを持っている場合は、既にApp-Vを使用できます。 また、App-VライセンスはMicrosoft Desktop Optimization Pack（MDOP）に含まれています。 これは、サブスクリプションとしてソフトウェアアシュアランスプログラムのメンバーが利用できるデスクトップテクノロジーのセットです。



ライセンスを決定したら、それがどのように機能するかを見てみましょう。



  1. アプリケーションは特別にパッケージ化する必要があります。 このために、App-V Sequencerプログラムが使用され、パッケージングプロセスはシーケンス処理と呼ばれます。
  2. 準備されたアプリケーションはネットワークフォルダーに配置し、ユーザーに読み取りアクセス権を付与する必要があります。
  3. ユーザーデバイス（またはターミナルサーバー）にApp-V Clientアプリケーションをインストールします。 ネットワークドライブから仮想パッケージをダウンロードして実行する必要があります。








各項目について詳しく見ていきましょう。



仮想パッケージを準備するには、インフラストラクチャで使用される一般的なクリーンシステムイメージが必要です。 App-V Sequencerアプリケーションをインストールします。 理想的には、仮想マシンとして実行されているコンピューターにSequencerをインストールし、インストール後すぐに「スナップショット」を作成する必要があります。 これにより、次のアプリケーションを仮想化する前に、Sequencerが「クリーン」な状態に戻るコンピューターを簡単に戻すことができます。



起動されたSequencerはアシスタントモードで動作し、最初にパッケージの名前を設定し、インストールを実行するディレクトリを選択してから、アプリケーションのインストール中に行われたレジストリおよびファイルシステムの変更を追跡して保存します。







App-V 5.0では、仮想アプリケーション用に個別のディスクを作成する必要はなく、インストールはデフォルトのディレクトリで実行されます。 ただし、このウィンドウの仮想アプリケーション（PVAD）のプライマリディレクトリは、さまざまな方法で選択できます。 これは、プログラムのインストールディレクトリ、そのサブディレクトリのいずれか、または存在しないフォルダのいずれかです。 この選択に応じて、仮想パッケージ内のファイルの構成が変わります。 この記事では、この点について詳しくは説明しませんが、私たちの経験から、最初の段階でPVADを適切に選択することで、特に古いアプリケーションの将来の多くの機能の問題を解消できる場合があることに注意してください。



インストール中に再起動が必要な場合は、それを実行できます。変更が記録され、中断した瞬間からプロセスが正しく続行されます。 並列（SxS）ライブラリアセンブリ（Microsoft Visual C ++など）は正しく処理されるため、以前にシステムイメージにインストールする必要はなくなりました。







完成したパッケージはファイルのコレクションです：







report.xmlファイル -シーケンス処理中に発生したすべての警告とエラーが保存されるレポートファイル。 問題の診断と解決に使用できます。



.msiファイルは、グループポリシーまたは展開システムを介して仮想パッケージをインストールするためにシーケンサーによって作成されるWindowsインストーラーファイルです。



.appvファイル -仮想アプリケーション自体のファイル。 バージョンApp-V 5.0では、このファイルはオープンzip形式で作成され、必要に応じて任意のアーカイバーで開くことができます。 以前のバージョンとは異なり、ファイルサイズに制限はありません。 残念ながら、シーケンサーを使用せずに直接変更することはできません。



展開構成ファイル、ユーザー構成ファイル -ターゲットコンピューターの展開パラメーターを決定するXML形式の構成ファイル。 テキストエディタでシーケンサーを起動せずに、それらに変更を加えることができます。 これにより、アプリケーションの微調整が大幅に簡素化されます。 ここでは、特に、ショートカットパラメータ、ファイルの関連付け、環境変数が設定され、レジストリキーが変更されます。



一部のプログラムでは、ローカルにインストールされたバージョンの仮想パッケージの機能がシーケンサーの助けと正確に一致することを保証することができません。 この場合、欠落している操作を実行するスクリプトを追加する必要があります。 カスタムスクリプトファイルがパッケージに追加され、その呼び出しがXML構成ファイルで構成されます。 たとえば、次のように： 



<PublishPackage> <Path>powershell.exe </Path> <Arguments>.\Scripts\InstallDriver.ps1 </Arguments> <Wait RollbackOnError="true" Timeout="120"/> </PublishPackage>




構文からわかるように、スクリプトは、使用したい任意の言語で作成できます。 当社のエンジニアは、主にPowershell、VBS、およびバッチを使用してスクリプトを記述します。



パッケージの準備が完了したら、ネットワークフォルダーにコピーして、ユーザーに読み取りアクセス権を付与します。

次に、ユーザーのコンピューターにApp-V Clientをインストールする必要があります...それだけです！

最も単純なケースでは、これ以上は必要ありません。 仮想パッケージは、いくつかの方法で配布できます。



  • シーケンス段階で作成された.msiインストーラーを介して。 たとえば、グループポリシーを使用します。
  • 組織で配布システムを使用する（Microsoft SCCM、Altiris）
  • Powershell 3.0を使用します。 一般に、Microsoft App-V 5.0製品はPowershell環境と非常に緊密に統合されており、非常に便利です。 パッケージの配布とメンテナンスに関連するタスクは、スクリプトを使用して自動化できます。








したがって、最も単純な場合、Microsoft App-Vを使用するには、ITエンジニアのマシンにSequencerがあればパッケージを準備し、ユーザーのマシンにはApp-V Clientでパッケージを実行するだけで済みます。 ただし、App-Vインフラストラクチャの完全な展開を最大限に活用することしかできません。



これを行うには、次のコンポーネントをインストールします。



  • 管理サーバー App-V 5.0インフラストラクチャの基本的な管理機能を提供します。
  • 公開サーバー。 仮想アプリケーションのホスティングおよびストリーミング機能を提供します。
  • SQLデータベース管理。


また、レポートサーバーとそのためのSQLデータベースをインストールすることもできます。



すべての役割は1つのファイル（APPV_SERVER_SETUP.EXE）にインストールされます。







Microsoftには、組織に個別の管理者チームがある場合に備えて、データベースを作成するための個別のSQLスクリプトが用意されています。 インストールグループは、App-V 5.0環境を管理する権限を付与されるADグループも指定します。



インストール後、カスタムApp-Vクライアントを構成する必要があります。 発行サーバーのアドレスを指定し、パッケージ変更情報の更新オプションを指定します。 Powershellを使用してこれを実現できます。



さらに、コンソールおよび管理へのアクセスは、ブラウザーを使用して実行されます。







ネットワークパスを指定してパッケージを追加します。







ADユーザーグループを割り当て、パッケージを公開します。







その後、しばらくすると、ユーザーにはアプリケーションのショートカットと対応するファイルの関連付けが表示されます。 ソフトウェアを使用する準備が整いました。



既定では、App-Vアプリケーションがユーザーに公開されると、パッケージファイルは％PROGRAMDATA％フォルダーにコピーされます。 ただし、クライアントは共有コンテンツストアモードに切り替えることができます。 Powershellでは、コマンドは次のようになります。 



 Set-AppvClientConfiguration -SharedContentStoreMode 1




この場合、ファイルのネットワーク上の場所へのNTFSリンクのみがユーザーのディスク上に作成され、ディスクスペースが大幅に節約されます。 もちろん、このようなソリューションには予備テストが必要であり、純粋な形で使用されることはめったにありませんが、実装されたプロジェクトには、顧客環境でのこのテクノロジーの適用例があります。



複数のプログラムが互いに対話する必要がある場合-「接続グループ」セクションでグループにまとめることができます。 この場合、仮想環境は結合され、グループのすべてのメンバーがファイルとレジストリを利用できるようになります。 そのようなグループを適切に形成し、プログラムの依存関係を識別し、優先順位を設定するには、ある程度の経験が必要です。 個々のコンポーネントを仮想化せずに、システムイメージに直接インストールすることをお勧めします。 これらは、ODBCドライバーまたはデータベース接続設定です。



これは、仮想化があらゆる場面で人命を救うソリューションであるという印象を与えます。 悲しいかな、他の技術と同様に、独自の制限があります。 特に、Microsoft App-Vを使用すると、システムの起動時に起動するドライバーまたはサービスを仮想化できません（ Microsoft Application Virtualization 5.0の仮想化ガイドを参照）。 他のテクノロジーと同様に、App-Vは進化しており、多くの制限が既に克服されています。 たとえば、ExplorerのアプリケーションコンテキストメニューやさまざまなActiveX拡張機能の仮想化は問題ではなくなりました。 ところで、App-Vの場合のこの開発は、時には挑戦でもあります。 サービスパックと修正は、問題の解決に加えて、機能に大きな変化をもたらします。実際には、この技術の新しいバージョンです。 ただし、1000を超える仮想化アプリケーションが背後にあるため 、お客様がビジネスプロセスを中断することなく、できるだけ早く作業環境に新しい機能を導入できるよう支援しています。



そこで、アプリケーション仮想化の利点を調べました。 簡単に言うと、Microsoft App-Vの例を使用して、仮想パッケージの作成と公開の主要な段階を理解しました。 もちろん、これは唯一の仮想化ツールではありません。 Citrix XenApp、VMware ThinApp、Novell Zenworksに言及するしかありません。 選択は、ビジネスのニーズによって異なります。 ただし、仮想環境への移行を実装したお客様をサポ​​ートした経験に基づいて、移行戦略を次のように定式化できます。アプリケーションを仮想化できる場合は、仮想化する必要があります。 長所は否定できません。
