Lync SDK2013。Lync（Skype for Business）に基づいた独自のインスタントメッセンジャーの開発経験

どう？



この記事では、Lyncクライアント用の「美しい」インターフェイスと新機能を作成するためにMicrosoftがLync SDKで提供するものについて説明します。 また、UI抑制モードでのLyncクライアントとの対話についても説明します。UI抑制モードでは、Skype for Businessに基づいて独自の企業メッセンジャーを開発するプロセスで非常に密接に対処する必要がありました。 そして、最も重要なことは、私たちが直面しなければならなかった制限を詳細に説明することです。



どのようにすべてが始まりましたか？



2014年の春に、Lync（現在のSkype for Business）に基づいた企業メッセンジャーを作成するというアイデアが当社で生まれました。 むしろ、最初はメッセンジャーについては話していませんでしたが、Lyncをより効率的に動作するように変更しました。 実際、Lyncクライアントを「通常モード」で使用し、アプリケーションは連絡先を管理するための単なるアドオンでした。 お気に入りの連絡先を画面の幅全体に簡単に配置できるようになりました。これは、デスクトップでの作業に便利で、連絡先への迅速なアクセスを暗示し、グループに整理できます。 このアプリケーションは、EasyLy（「Easy Lync」から）と呼ばれ、「彼ら」に配布されました。





図1.プログラムの最初のバージョン



徐々に、新しい機能がアプリケーションに追加されました。 たとえば、LyncとEasyLyの両方に対応でき、通信の履歴は後者に保存されました。 また、あるマイルストーンの日に、Lyncトレースがタスクマネージャーにのみ残るように、Lyncインターフェイスを放棄し、そのすべての機能を自分で実装することが決定されました。 つまり、Lyncをベースにしたデスクトップ用に独自のより便利なメッセンジャーを作成することにしました。 そして、すべてが順調に始まったようです。



結果として何を得たかったのですか？



当初、Lyncの主な欠点の1つは、会話の履歴を保存および表示するための多かれ少なかれ適切なメカニズムの欠如でした。 もちろん、LyncはOutlookの履歴を（一定の時間後に）保存しますが、これは使用するのが非常に不便です。 さらに、たとえば、保存が行われる前に会話ウィンドウを閉じると、ストーリーがまったく保存されない場合があります。



ユーザビリティに関しては、「ワンクリック」の概念を説いてユーザーの時間を節約し、作業の効率を高め、できるだけシンプルでわかりやすく、便利なアプリケーションの作成を試みました。 私たちが取っておきたいターゲットグループは、スマートフォンで「フィールド」に沿って走るファッショナブルな「モバイル従業員」ではなく、人間工学に基づいた大型モニターの後ろで8時間働く実際のオフィス労働者です。



以前に表示された[お気に入り]タブに加えて、グループ、ダイアログタブ、コメント付きの通話ログ、オンライン会議ウィンドウ、オフラインサブスクライバーへのメッセージ送信機能、カレンダーとの統合、OneDriveおよびその他の多くの興味深い機能を作成するアイデアがありました。 。 既存のLync機能（およびSkype for Business）を破棄することなく、デスクトップUI、既存の機能への迅速なアクセス、および新しい企業機能の作成により、ユーザーに付加価値をもたらすことを計画しました。





図2.これがEasyLyの現在の外観です



開発



最初に行われたのは「お気に入り」タブで、ユーザーはすべての連絡先を整理し、グループに分類することもできました。



Lync SDK 2013を使用して、問題なく作業を開始し、このアイデアを実装しました。すべての連絡先をカードの形式で作成し、このユーザーの会議と会議のリストを表示したり、電話をかけたり（音声通話とビデオ通話）したり、メッセージを書いたりできます。 これらのボタンをクリックすると、標準のLync会話ウィンドウが表示されます。





図3.ホバー連絡先アイコン



とても簡単でした。 成功に勇気付けられて、私たちは先に進み始め、Lync UIを完全に放棄しました（LyncクライアントをUI抑制モードに移行しました）。 これにより、会話、呼び出し（および実際にはすべてのウィンドウ）の標準ウィンドウを放棄し、アプリケーションのすべての基本機能を自分で実装する機会が与えられました。



ユーザー認証については説明しません。説明はここにあります 。 ほぼ1年間実装してきた2番目の最も重要なタブ「ダイアログ」について説明します。





図4.マルチチャットが有効になっている[ダイアログ]タブ



予想通り、テキストによる会話から始めました。 すべてがうまくいきましたが、それほど速くはありませんでした。メッセージを送受信するだけでなく、入力、顔文字、rtfフォーマットのサポートなどを実装する必要があったためです。 ここから Lync IM会話の実装を取得できます 。



ユーザーが柔軟に管理できるSQLiteに基づいて実装したメッセージとコールレコードを保存します。 また、従業員とのコミュニケーションをさらに高速化するために、1つのウィンドウで複数のオープンチャット（最大3つ）を同時にサポートしました。



次に、音声、ビデオ通話、画面共有、アプリケーションのサポートを追加しました。 会話へのアプリケーション共有サポートの追加には長い時間がかかりました。 通常のLyncモードでは、画面は会話自体に表示されます。UI抑制モードでは、リンクコントロールApplicationSharingViewを使用してすべての共有オプションを自分で実現する必要があります。これは、親ウィンドウのハンドルが機能するために必要な共有リソースを表示するために使用されます。



共有をサポートするために行ったことの短いリストを以下に示します。

•共有を表示するためのウィンドウを作成し、親ウィンドウのサイズを変更するときに画像の再描画を実装しました。

•手探りできるリソース（画面、アプリケーション）のリストを作成しました。

•共有リソースのフレームワークを概説しました。

•追加機能を実装しました。たとえば、会議参加者にコントロールを転送する（マウスを動かして「詐欺」ユーザーの画面をクリックできるようにする）。



Lync（UI抑制モード）での共有は気まぐれなことです。彼は未知の理由で脱落するのが好きで、Skype for Business 2016に切り替えたとき、彼はこれをほぼ絶えず始めました。 sharingaを表示すると、黒い画面が表示されることがよくあります。上記のApplicationSharingViewコントロールは再生中なので、何もする必要はありません。 また、共有の正常な起動は、接続速度の影響を大きく受けます。



次に、最後のタブである「グループ」の作成に進みました。これにより、Lyncからグループのリストが読み込まれ、ユーザーはそれらを完全に編集して目的の順序で並べ替えることができます。 このタブは、Active Directory（Lyncでは配布グループと呼ばれます）で既に作成された従業員のグループをホストする企業タブに成長するはずです。たとえば、「パスの注文」、「明細書の作成」などです。 これらのグループのメンバーは、会社の特定の問題に対処する従業員です。 このグループをクリックすると、グループの最初のメンバーのステータスが「オンライン」であるチャットまたは音声通話が開きます。 これらのグループは、Lyncサーバー管理者が編集する必要があります。



もちろん、すべての計画を実施する過程で、定期的に解決する問題がありました。 ただし、Lync抑制モードでのLync SDKライブラリの不適切な操作に関連する特別なグループの問題がありました。 Microsoftに説明を送信し、サポートを受けて解決策を見つけることができると考えました。



問題







結局のところ、Outlookとの統合はUI抑制モードで消えます：

•連絡先のステータスが消えます。

•連絡先カードから会話を作成する可能性はありません。

•「Skypeミーティングの作成」ボタン「終了」。



この記事（ https://msdn.microsoft.com/en-us/library/office/jj900715.aspx?f=255&MSPPError=-2147217396#off15_IMIntegration_HowConnect ）に基づいて、Outlookで完全に機能する連絡先カードを返すことができました。



また、連絡先のステータスを獲得しました。 現在、OutlookはLyncからではなく、アプリケーションからそれらを受信しますが、何らかの魔法の方法でそれらを更新し、絶えず「落ち」て同期を停止します。 つまり、実際には、人はすでにオンラインで「緑」に、そしてOutlookでは-黄色のステータス「退席中」になっています。 Microsoftを支援して、同じ記事へのリンクを送信しました。 彼らはまた、Yammer Microsoft、MSDNフォーラム、Reddit、およびその他のプラットフォームでも書いています...残念ながら、この問題は解決されていません。



落ちたOutlookの「Skype会議の作成」ボタンの問題は、UI抑制モードのレジストリキーがHCU /ソフトウェア/ Microsoft / Office / 15.0 / Lync / ConfAddin / sip /パスに沿って消去されることです。 ここには、Capabilities、InbandInfo、PublicMeetingの3つのキーが表示されます。 Lyncを通常モードで起動し、[Skype会議の作成]ボタンをクリックすると、OutlookはLyncに会議情報を要求し、Lyncは既にレジストリキーを作成します。 UI抑制モードでは、Lyncはこれを行いたくありません。 Lync SDK 2013を使用すると、Lyncサーバーから必要な情報を取得できません。 ソリューションが見つかりました：別のSDK（ https://msdn.microsoft.com/en-us/library/office/dn465943.aspx ）を使用して、サーバーを直接操作します。 UCMAを介して、許可されたユーザーの会議に関する情報をサーバーから取得できました。 レジストリキーを埋めました-そして出来事：Outlookでコレクションを作成する機能を返しました。 ただし、この方法にはいくつかの制限があります。

•UCMAライブラリは、64ビットオペレーティングシステム、32ビットシステムでのみ動作します。解決策はありません。

•ユーザーのコンピューターとサーバーは同じドメインに存在する必要があります。そうでない場合は、信頼されたアプリケーション用にサーバー上に証明書を作成する必要があります。 これを行う方法はここで説明されていますが、クライアント側でのみ機能を実装する必要があるため、このソリューションは適していません。



Lync SDK自体にバグがありました。 最も重要なこと：p2pビデオ会話では、3番目の参加者を追加すると、Lyncクライアント自体が安定してドロップします。 マイクロソフト自体がこのバグを確認していますが、期限に名前を付けていません。 p2p会話を強制終了し、一度に3人でビデオ会議を作成しようとしましたが、SFBクライアントによって3人目の参加者が追加された場合、これを防ぐことはできません。 ビデオチャネルを停止して再開しても、何も起こりませんでした。



また、一部のユーザー（Lyncクライアントのバージョンが異なるユーザー）と画面を共有できないという問題も発生しました。 この問題は次のように現れます。



最初のクライアントはアプリケーション（Lyncバージョン2013）で動作し、2番目のクライアントはSFB 2016で動作し、最初のクライアントと会話を開くと、画面共有ボタンは使用できません。 最初のクライアントが通常のLync 2013（抑制モードではない）に入ると、2番目のクライアントは共有ボタンがアクティブになります。 2番目にもLync 2013またはアプリケーションがインストールされている場合、共有は両側で利用できます。



Lync SDKには、ユーザーにメッセージを配信するためのコールバックがありません。 メッセージがなくなったコールバックのみがあります。 しかし、場合によっては、オンラインで人にメッセージを送信し、例外なくメッセージを残すという状況が発生する場合がありますが、到着時にはすでに対談者はオフラインになっています。 したがって、送信者はメッセージが到着したと考えますが、そうではありません。



多くの場合、連絡先の写真を受信すると「ItemNotFoundException」が発生するため、標準の連絡先画像を表示する必要があります。 経験的に、Lyncが引き続き写真を返すために、写真を受信するためにコードを呼び出す必要がある回数（約3回）を見つけました。 ただし、一部の連絡先では、この方法はまだ機能しません。 さらに、写真のキャッシュ機能を導入しました。これにより、ユーザーが多くの連絡先を追加している場合でも、最初から多数の実行が流れ込むことはありません。



ユーザーグループ、特に「お気に入りグループ」に問題があります。 問題は、一部のユーザーがCanInvoke（DeleteGroup）を呼び出すと、Lyncがtrueを返しますが、これは標準のLyncグループであり、削除できないことです。 このグループのタイプは、ユーザーが作成したグループ（CustomGroup）のタイプと同じであるため、メニューから「グループの削除」項目を非表示にするには、グループ名の追加チェックを行う必要があります。



発見されたLync SDK 2013の制限



•重要な機能の1つは使用できません。ユーザーがオフラインのときに自動転送を構成する機能です。 ユーザーがオフラインでない場合にのみ、一定時間後にリダイレクトを実装しました。

•グループビデオコール（3人以上の参加者）では、参加者のビデオをブロードキャストするストリームは1つのままです（モバイル会議のアナログ）。つまり、Lync自体は、現在マイクに話している人に応じて、異なる参加者からのビデオストリームを変更します。

•ファイル転送は、UI抑制モードでは使用できません。 この問題は2つの方法で解決しました。

-Outlook経由のファイル転送をレターの添付ファイルとして追加。

-ファイルをクラウドにアップロードし、ファイルURLからメッセージを自動的に送信するOneDriveサービスを導入しました。

•会話が会議に変換され、次のようになったときに、会話のコンテキストデータ（使用方法、ここで説明されているhttps://msdn.microsoft.com/en-us/library/office/jj933248.aspx?f=255&MSPPError=-2147217396 ）を使用できません。ローカルではなく、サーバー上に存在します。 この機能を使用して、顧客間でさまざまな情報を同期したいと考えました。

•会議の「件名」フィールドに記録する機能が無効になっています（通常のLyncでも）。何らかの理由で、Lync 2013ではこの機能を終了しました。 MSDNの男性は、Lync 2010に約30の会議があり、それぞれに独自のタイトルがあると不平を言っていました。 Lync 2013への移行に伴い、彼は各会議の参加者のリストのみを持っていました。 現在、ある会話を別の会話と区別することはほとんど不可能です。

•Lync SDK 2013のもう1つの機能は、Lync 2013以降のカスタマーサポートです。 Lync 2010はサポートされていません。 ただし、Lync SDK 2010は、Lync 2010とLync 2013以降の両方で機能します。 マイクロソフトがどのロジックをガイドするかは不明です。



おわりに



開発プロセス中に、Lync SDK + UCMAに基づいて、Lyncの本格的なアナログを完全に実現することは不可能であることに気付きました。 これらのテクノロジーの制限を考慮に入れていなくても、Lync SDKエラーを修正するためにマイクロソフトからサポートを得られなかったのは残念です。



今日、UI抑制モードを放棄せざるを得ません。 必要な機能をすべて実装することは不可能であり、発生した問題を回避するオプションを探しています。
