Spring Bootでトランクまたはテスト駆動開発アプリケーションにピラミッドを構築する方法

が統合テストの必要かつ十分なレベルであると思います-それは興味深いでしょう！







TDDの機能は非常に曖昧であり、測定が困難ですが、Springには、最小限の労力で統合テストと単体テストを作成するのに役立つ多くの機能があります。 例：

テスト駆動開発



TDDは非常に単純なアイデアに基づいています-コードを記述する前にテストを記述します。 理論的には恐ろしいように聞こえますが、しばらくすると実践と技術の理解が得られ、 その後テストを書くという選択肢は明白な不快感を引き起こします。 重要なプラクティスの1つは、 繰り返しです。 すべてを小さく、焦点を絞った反復を行います。各反復はRed-Green-Refactorとして記述されます。







赤のフェーズでは、落下テストを作成します。明確で理解可能な理由と説明で落下し、テスト自体が完了してコードの作成時に合格することが非常に重要です。 テストでは、 実装ではなく動作をチェックする必要があります 。 「ブラックボックス」のアプローチに従ってください。その後、その理由を説明します。







グリーンフェーズでは、テストに合格するために最低限必要なコードを記述します 。 練習して不条理なところに持っていくのは面白いこともありますが（夢中にならないほうがよいです）、システムの状態に応じて関数がブール値を返す場合、最初の「パス」は単にreturn true



。







すべてのテストが緑色の場合にのみ開始できるリファクタリングフェーズでは、コードをリファクタリングして適切な状態にします。 私たちが書いたコードには必要すらありません。そのため、安定したシステムでリファクタリングを開始することが重要です。 「ブラックボックス」アプローチは、リファクタリング、実装の変更に役立つだけで、動作には影響しません。







将来、TDDのさまざまな側面についてお話します。結局のところ、これは一連の記事のアイデアであるため、ここでは特に詳細に言及しません。 しかし、標準のTDD批判に対応する前に、よく耳にする神話をいくつか取り上げます。







  • 「TDDはコードの約100％のカバレッジですが、保証はしません」 -テストによる開発は100％のカバレッジとは関係ありません。 私が働いていた多くのチームでは、このメトリックは測定さえされておらず、バニティメトリックとして分類されていました。 そして、はい、100％のテストカバレッジは何も意味しません。
  • 「TDDは単純な機能でのみ機能します。データベースを使用して実際のアプリケーションを作成し、難しい状態を作成することはできません」は非常に人気のある言い訳であり、通常は「テストをまったく書かないほど複雑なアプリケーションがあるため、不可能です」と補足されます。 Web（SPAの有無にかかわらず）、モバイル、API、マイクロサービス、モノリス、複雑な銀行システム、クラウドプラットフォーム、フレームワーク、さまざまな言語とテクノロジーで書かれた小売プラットフォームなど、まったく異なるアプリケーションでTDDアプローチが機能しているのを見ました。 そのため、「私たちはユニークで、すべてが異なる」という一般的な神話は、ほとんどの場合、テストに投資しない理由であり、本当の理由ではありません（本当の理由もあります）。
  • 「TDDにはまだバグがあります」 -もちろん、他のソフトウェアと同様です。 TDDはバグやバグがないということではなく、開発ツールです。 デバッグが好きです。 IDEのように。 ドキュメントのように。 これらのツールはいずれもバグがないことを保証するものではなく、システムの複雑化に対応するのに役立ちます。


TDDおよび一般的なテストの主な目標は、システムが安定して動作しているという自信をチームに与えることです。 したがって、どのテスト方法でも、テストの数と書き込みを決定しません。 どれだけ必要だと思うか、今すぐにコードを本番環境に入れて機能することを確認するためにどれだけ必要かを書きましょう 。 クイックインテグレーションテストを必要かつ十分な最後通告ブラックボックスと見なし、オプションのユニットテストを検討する人々がいます。 e2eは、以前のバージョンへの迅速なロールバックの可能性とカナリアリリースの存在をテストすることはそれほど重要ではないと言います。 チーム数-アプローチが多いため、自分のチームを見つけることが重要です。

私の目標の1つは、TDDストーリーの「2つの数値を追加する関数のテストを通じて開発する」という形式から離れ、実際のアプリケーション、つまり実際のプロジェクトで収集された最小限のアプリケーションにまで蒸発した一種のテスト方法を検討することです。 そのような半現実的な例として、私自身が抽象化のために発明した小さなWebアプリケーションを使用します 工場 ベーカリーベーカリーケーキ工場 。 毎回個別のアプリケーション機能に焦点を当てた小さな記事を書いて、TDDを通じてAPI、アプリケーションの内部構造を設計し、一定のリファクタリングを維持できることを示すつもりです。
