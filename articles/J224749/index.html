RAM-必要ですか？





実際、なぜRAMが必要なのですか？ すべてのデータ（実行中のプロセスを含む）を不揮発性メモリに保存しないのはなぜですか？ 便利でしょう。電源が切れた-プロセッサが停止し、電源が入った-プロセッサは、停止した命令からさらにタスクを実行し続けました。 インスタント休止状態/ウェイクアップ！ すぐに消える問題の数。 たとえば、多くのタスクでファイルシステムをジャーナリングする必要はまったくありません。



もちろん、利用可能なすべての不揮発性データストレージ方法の速度にとどまりません。 ただし、朗報があります。10年以上にわたって、磁気抵抗メモリが開発されてきました。 書き換えサイクルの速度と回数は動作と同じであり、外部電源の独立性はNVMの独立性に似ています。 少し前まで、このメモリのサンプルが販売されており、8〜16 MBのアレイを組み立てることができました。 これにより、現在このメモリにLinuxを展開し、新しい条件への適応作業を開始できるため、磁気抵抗メモリの価格が通常のRAMと同等になるまでに、馬に乗って武装します。 これで、グラフィカルインターフェイスが不要なタスクにこのテクノロジを適用できるようになりました（たとえば、常にオンボードのLinuxが搭載されたルーターは非常に優れています）。

組み込みプログラマは、古くて不便で危険なCでコードを開発するのにどれくらいの時間を費やしますか しかし...再び良いニュースです！ Matzはmrubyをリリースしました。mrubyは外部依存関係がほとんどなく、Cで書かれており、どこにでも簡単に埋め込むことができます。 1人の日本のファンがRuby Conf 2012でレポートを発表し、1 MiBのコードメモリと128 KiBのデータメモリを備えた環境にこの素晴らしいプログラミング言語のインタープリターを組み込んだ経験を説明しました。 このような特性を備えたプロセッサは、悲惨な3ドルから4ドルのコストがかかります（大量生産）。 同時に、開発の速度とセキュリティが大幅に向上します。 なんらかのオペレーティングシステムを構築することもできます。 Rubyにはポインターがなく、アクセス制御のタスクはそこで（プログラミング言語のレベルで！）それ自体で解決されます。



多くの人は、組み込みおよび解釈された高水準プログラミング言語は相互に排他的な概念であると考えています。 しかし、これはそうではありません！ 例として、自分の人生の例を挙げることができます。 スマートな「フラッシュドライブ」を開発しました。 データストレージ機能に加えて、デバイスには顧客が実装した特別なロジックがありました。 USBスタックとそのプロファイル（大容量記憶装置）は、独自に実装されました。 入れる前に、読み取りと書き込みの速度を何度もチェックし、プロセスを最適化しました。 その結果、生データの読み取りで14 MiB / s、書き込みで10 MiB / sを達成しました。 あまり指向していない人にとっては、これはASICを使用せずにUSB 2.0 Highspeedで実装できるもののほぼ限界であると言えます。 プロジェクトが完了し、顧客がデバイス用の独自のロジックの開発を開始すると、暗号化速度（これはSZI）が非常に遅いことが判明しました。 プロセッサの周波数を最大限に緩めるのを忘れることに気付くまで、私たちは問題が何であるかを長い間調べようとしました。 なぜこれが起こっているのですか？ はい。適切に設計されたエグゼクティブデバイスの標準ファームウェアコードの95％は通常、管理作業です。レジスタの設定、手順を開始するコマンドの発行、エラー処理などです。 すべてのインライン操作は、専用のモジュールによって（理想的に）実行される必要があります。 プロセッサは非知的作業に従事するべきではありません-プロセスを整理し、「非インテリジェントモジュール」がハンマーを実行する必要があります。 したがって、これらの組織プロセスのパフォーマンスは重要なパラメーターではありません。 突然10倍遅くなり始めたとしても、これがデバイスの全体的な速度に影響することはほとんどありません。 しかし、最も複雑に設計されたユニットであるのはまさにこの組織活動であり、コードの大部分を占めるのはまさにそれです。 それでは、開発をスピードアップし、開発者の神経を大事にするために、このコードによって解釈されないのはなぜですか？



私たちはこの分野で研究を行っているだけです。コメントから、これが私たちだけでは面白くないということがわかったら、結果を公開します。 NXP LPC18xxにmrubyを埋め込みます。



スイスのフラッシュドライブシステム管理者





フラッシュのようなデバイスを開発する必要があった最初のプロジェクトを完了して以来、多くの非標準機能を備えたフラッシュドライブが多くの人にとって役立つ可能性があるという考えにがっかりしませんでした。 これらの機能の一部を次に示します。



  • 構成可能なLUNの数 。 LUNは、大容量記憶装置ドライバーレベルでのデバイスの論理パーティションです。 エンドユーザーにとって、LUNは個別の物理ディスクのように見えます 。 たとえば、通常、複数のLUNにカードリーダーがあります（スロットごとに1つのLUN）。 各LUNに対して、メディアタイプタスクを提供できます。通常のハードドライブ、CD、リムーバブル/非リムーバブルなど。 ディスクごとに、アクセスモード設定RW / Rを指定できます。 これは、たとえば、古いプログラムが希望のシリアル番号の挿入されたCDでのみ動作する場合に便利です。 スイスのフラッシュドライブを2つのLUNに分割できます。1つはこのCD用、もう1つは通常のリムーバブルメディア用です。
  • RAIDモード 。 デバイスには、1つではなく複数のメモリチップを配置し、RAID0 / 1モードを実装できます。 そして、1つのLUNがRAID0、2つ目がRAID1、3つ目が通常の「フラッシュドライブ」、4つ目が一般的にCDになるようにできます...
  • 非標準インターフェース 。 時々、システム管理者は…USBフロッピーを手に入れる必要があります。 そして、フロッピーディスクに変装するUSB​​フラッシュドライブを手に入れてみませんか。
  • プログラム可能なボタンとディスプレイ 。 複数のジョブプロファイルがあり、それらをすばやく切り替えることができれば便利です。 これを行うには、現在のプロファイルを表示するために、ケースとディスプレイにいくつかのボタンを提供できます。 さらに、ボタンを完全にプログラム可能にすることができます。たとえば、ボタンの組み合わせの1つを割り当てて、デバイス上のデータを完全に破壊します。 さて、あなたは私が話している分野のアプリケーションについて理解しています..
  • USB 3.0および> 100 MiB / s読み取り。 すでにそれほど多くの機能ではなく、当時の需要です。
  • ビール栓抜きとして働く 。 しかし、システム管理者はそのような機能を必要としませんか？




残念ながら、このデバイスの開発に約6か月を費やす必要のあるチームです。 そして、このクラスのデバイスの開発ですでに豊富な経験を持っているという事実にもかかわらず。 投資なしでこれを行うことは不可能であり、投資家はこのアイデアに興味深いものを見つけません。 しかし、誰かがこのアイデアを実装した場合-私はここでそれについて書いているため、私はただうれしいです。



高解像度DVR





あなたの車のビデオレコーダーの画質が好きですか？ しません 多くのDVR（内部を含む）を見ましたが、すべてに1つの重大な欠点があります。特定の条件下（夕暮れ時または揺れたとき）での損失のある圧縮アルゴリズムの適用により、画面上で10-15の車の番号を見ることも不可能メートル。 しかし、これはDVRの最も重要な目的の1つです。車の番号、または事故やその他の事故に参加した人の人/サインを書き留めてください。 なぜこれが起こっているのですか？



すべてのDVRはビデオストリームを圧縮します。 その場で圧縮を実行できるように、ハードウェアコーデックを使用します。 既存の数学的アルゴリズムを使用してもこのようなコーデックを開発することは非常に困難な作業であり、エレクトロニクス業界の巨人にしか実現できません。 Colossiは、ご存じのとおり、DVRを生産していません（ところで、なぜだろうか？多くの国で、車のDVRが私たちや中国のように広く使用されていないことを知っています）。 幸いなことに、これらの巨人の娘によって（または彼らのサポートで）生産されたいくつかのハードウェアプラットフォームがあります。 DVRの開発者は、このようなソフトウェアおよびハードウェアプラットフォームを基盤として、ケースに（多くの場合、ボードを再マウントせずに）梱包し、構成し、ネームプレートを接着して販売します。 たとえば、よく知られているDOD F900LHDは、Ambarellaプロセッサを備えた、広く知られているヤマハプラットフォーム上に構築されています。



したがって、DVRの開発者自身はデバイスのメイン機能の開発に関与せず、コーデック設定を微調整することさえできない場合があります（アルゴリズムの実装を大幅に変更することは言うまでもありません）。 彼らは彼らが持っているものを使うことを余儀なくされ、他の選択肢はありません（あなたが受け入れられたDVRデザインから逸脱しない場合）。 状況は、ほとんどのメーカーが低コストのセグメントに焦点を当てており、したがって意図的に品質を最大にひねっていないという事実によって悪化しています-DVRはSDカードに関して雑食ではありません。 さらに、ビデオコーデックの構成では、特定のパラメーターに焦点を合わせます。 たとえば、夜間にはっきりと見えるが、写真が不自然になったり、その逆になります。 また、これらのパラメーターを変更することは常に可能とは限りません-コーデックのメーカーが設定しているように、これは通常機能します。 私はいわゆるの市場参加者の一人と話しました 「ロシア語」DVR。 私は、中国の仕入先にコーデック設定で修正することを望んでいるものを叩き込むのが面倒ではない、かなり元気な人に出会いました。 それらは、デバイスの開発者にリクエストを出し、基本的なハードウェアプラットフォームの開発者に（明らかに）頼りにすると、 単純な編集構成にどれだけの費用がかかるかがわかりました。 最終合計は、6個のゼロ（人民元ではない）でした。 とにかく、この構成はブレークスルーにつながることはなく、一部の機能が少し良くなるだけです。



そのため、DVRの開発者はビデオストリームを非常に強く圧縮する必要があるため、カメラの「メガピクセル数」や視野角などは重要ではなくなります。 イベントの一般的なコースのみが表示されますが、詳細は完全に表示されません。 しかし、自宅から職場へ、そして職場から自宅への8時間の旅行を地図上に保存することができます...品質を優先して録画時間をあきらめませんか？ RAMに書き込みませんか？



8 GBのRAMを取ります。 このサイズのRAMに記録できる720×480のビデオ解像度の持続時間はどれくらいですか？ 1ピクセルあたり3バイトx 720×480 x 25フレーム/秒= 26 MiB / s 5分以上の鮮明な写真。 開発履歴が3分以内に始まり、その開発に1分以上かかるような事故によく遭遇しましたか？ そして、事故が発生した後、たとえば、もう1分かかり、すべてをeMMCまたはSDカードに転送できます。



興味深いのは、そのようなDVRをロシアで迅速に開発できることです。 実際、すべてのDVRで最も難しい部分はビデオハードウェアコーデックです。 これらのコーデックは独自のものであり、ライセンスの購入が必要です。 販売された各デバイスから控除を支払う必要がありますが、これでさえ最大の困難ではありません。 最大の難点は、何百万もの製品コピーの保証を提供していない開発者と連絡が取れないことです...
