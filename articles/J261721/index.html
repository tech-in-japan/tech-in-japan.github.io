Androidプロセス：私はあなたを生んだ、私と...





システムがプロセスを不必要に終了させた後、アプリケーションに何が起こるかを考えたことはありますか？ 悲しいことに、多くの人は、それがパラレルユニバースで発生し、彼らに関係がないかのように、それについても心配していません。 初心者は特にこの傾向があります。 リンクスタティックの安定性に対する彼らの盲目的な信仰は、単に驚くべきものです。



この記事では、アプリケーションプロセスに関連する第6戒律違反（殺さない）の結果として発生する可能性のあるいくつかのエラーと、それが他の世界からどれだけうまく戻るかを確認する方法について説明します。



プロセスがシステムによって強制終了される可能性があることは秘密ではありません。 そして、あなたは疑問に思っていました、これを真似することは本当に可能ですか？ システムをアプリケーションに「誘発」して、高貴なメモリを食い尽くす他のアプリケーションをたくさん起動し、適切なアプリケーションを殺すことでシステムがダウンすることを願うことができます。 ある種のシャーマニズムであることが判明しました。これは管理者のパスですが、プログラマーのパスではありません。 アプリケーションを簡単かつ迅速に強制終了できる方法、およびシステムがリソースを解放するように実行する方法に興味がありました。 結局、「実験条件」でこの動作を繰り返すことができれば、開発段階で多くのエラーをキャッチしたり、簡単に再現して原因を特定したりできます。



判明したように、必要なメカニズムはすでにSDKにあり、これは...ドラムロール...「アプリケーションの終了」ボタンです。







クリックすると、次のコードを実行するのに似ています。 

//       . // ,   "Terminate Application"   “1”. System.exit(1);
実際、これはプロセスを強制終了します。 システムが不要なプロセスを削除しようとすると、同様のアクションが発生します。



アプリケーションが完全に停止した後にアプリケーションを再開する状況を再現するには、次の一連のアクションを実行する必要があります。

  1. Android Stidioを使用して、アプリケーションを起動します。
  2. [ホーム]ボタンを押して、アプリケーションを最小化します。
  3. Android Studioで、[Android]タブに移動し、アプリケーションを選択して[アプリケーションの終了]ボタンをクリックします。
  4. 最小化されたアプリケーションを展開します。

状況2：すべての問題の解決策としてのsetRetainInstance



プログラマーが名と姓を使用して、フラグメントダイアログを使用して例を繰り返すことを決めたとします。

そのため、姓と名を入力するためのテキストフィールドを持つDialogFragmentがあります。 このフラグメントは、リスナーデータエントリに設定できます。 onCreate（）でリスナーへの参照を失わないために、setRetainInstance（true）が呼び出されます。

対話コードの一部 
 public class Situation2DialogFragment extends DialogFragment { private OnPersonChooseListener mListener; @Override public void onCreate(Bundle savedInstanceState) { super.onCreate(savedInstanceState); setRetainInstance(true); } public void setListener(OnPersonChooseListener listener) { mListener = listener; } public void invokePersonChoose(Person person) { mListener.onPersonChoose(person); } public interface OnPersonChooseListener { public void onPersonChoose(Person person); } // ... }


画面にボタンがあり、クリックするとこのダイアログが表示されます。

 public void onChoosePersonClick(View view) { Situation2DialogFragment dialog = new Situation2DialogFragment(); dialog.setListener(new Situation2DialogFragment.OnPersonChooseListener() { @Override public void onPersonChoose(Person person) { Toast.makeText( Situation2Activity.this, String.format( getString(R.string.sit02_summary_message), person.firstName, person.lastName), Toast.LENGTH_SHORT ).show(); } }); dialog.show(getSupportFragmentManager(), Situation2DialogFragment.class.getName()); }

どうした

setRetainInstance（true）は、ダイアログの破棄を許可しませんでした。 プロセスの破壊後、それにもかかわらず対話は破壊されました。 アクティビティは、フラグメントを可能な限り復元します。 残念ながら、リスナーは完全に異なるオブジェクトのために完全に異なる場所にインストールされているため、復元されません。 invokePersonChooseメソッドのダイアログでリスナーの呼び出しが発生すると、例外がスローされます。 そして、nullチェックがない場合の問題はありません。 空のリンクに適切に応答せずにヌルチェックを行うと、さらに悪い解決策になります。



どうする

フラグメントからアクティビティにメッセージを送信する方法の多くは、インターネットで説明されています。 残念ながら、すべてが正しいわけではありません。 次の方法は正しく、私のお気に入りの1つです。

  • アクティビティは、目的のインターフェイスを実装します。

    アクティビティコードピース 
     public class Situation2Activity extends Activity implements OnPersonChooseListener { //… @Override public void onPersonChoose(Person person) { Toast.makeText( Situation2Activity.this, String.format( getString(R.string.sit02_summary_message), person.firstName, person.lastName), Toast.LENGTH_SHORT ).show(); } }
  • フラグメントは、getActivityを介してアクティビティへの参照を取得し、対応するインターフェイスに導きます。 さらに、すべての作業はインターフェイスを介して実行されます。

アクティビティに加えて、親フラグメントまたはターゲットフラグメントを使用できます。

ケース3：壊れたスレッド



アプリケーションの本質を次のようにします。

画面上のボタン。 ボタンをクリックすると、進行状況ダイアログが開き、UIとの対話がブロックされます。

進捗ダイアログコード 
 public class ProgressDialogFragment extends DialogFragment { public static final String DISMISS_ACTION = "ru.kamisempai.ProgressDialogFragment.DISMISS_ACTION"; private boolean isWaitingForDismiss = false; private BroadcastReceiver mDismissActionReceiver; @Override public void onAttach(Activity activity) { super.onAttach(activity); mDismissActionReceiver = new BroadcastReceiver() { @Override public void onReceive(Context context, Intent intent) { dismiss(); } }; LocalBroadcastManager.getInstance(activity) .registerReceiver(mDismissActionReceiver, new IntentFilter(DISMISS_ACTION)); } @Override public void onDetach() { if (mDismissActionReceiver != null) LocalBroadcastManager.getInstance(getActivity()) .unregisterReceiver(mDismissActionReceiver); super.onDetach(); } @Override public Dialog onCreateDialog(Bundle savedInstanceState) { final android.app.ProgressDialog dialog = new android.app.ProgressDialog(getActivity()) { @Override public void onBackPressed() { // do nothing } }; dialog.setMessage("Please wait..."); dialog.setCancelable(false); dialog.setCanceledOnTouchOutside(false); return dialog; } @Override public void onResume() { super.onResume(); if(isWaitingForDismiss) dismiss(); } @Override public void dismiss() { if(isResumed()) super.dismiss(); else isWaitingForDismiss = true; } }


コードから理解できるように、このダイアログはブロードキャストメッセージを送信することで閉じることができます。 これは、リンクを保存する必要がなく、アプリケーションコンテキストがあれば十分であることを意味します。

ダイアログを開始した直後に、しばらくの間何かをするスレッドが開始されます。 作業の最後に、スレッドはメッセージを送信して進行ダイアログを閉じます。

ストリームの作成と開始 
 public void onStartActionClick(View view) { new ProgressDialogFragment() .show(getSupportFragmentManager(), ProgressDialogFragment.class.getName()); final Context context = getApplicationContext(); new Thread() { @Override public void run() { try { Thread.sleep(5000); } catch (InterruptedException e) { e.printStackTrace(); } LocalBroadcastManager.getInstance(context) .sendBroadcast(new Intent(ProgressDialogFragment.DISMISS_ACTION)); } }.start(); }


このコードを実行して、アプリケーションの最小化のためにその完了を待たずにプロセスをオフにすると、デプロイ時に問題が発生することはなく、アプリケーションはクラッシュせず、ダイアログが表示されます。 私たちは少し待っています...私たちはまだ待っています...それから私たちはまだ待っています...それは長い間行われていたはずですが、ダイアログは閉じません。



どうした

プロセスが破棄されると、そのすべてのスレッドが停止し、復元されると、メインスレッドのみが開始されます。 したがって、スレッドの実行時間が長すぎる場合は、停止する準備をしてください。 また、長い間何かをする必要はありません。待機通知を使用している場合でも、このレーキを踏むことができます。 パブリック静的最終オブジェクトをブロック用のオブジェクトとして使用すると、プロセスが強制終了されるときに静的オブジェクトが例外でないことを既に知っているため、特に面白いでしょう。

どうする

タスクが別のサービスに移動して新しいプロセスで実行するのに時間がかかる場合、さらにフォアグラウンド通知を出力する場合、プロセスは強制終了されます。 待機通知の場合、すべてが少し複雑になり、特定の状況に依存します。 一般に、スレッドの操作のトピックは非常に広範囲であり、特定のアドバイスを与えることはここでは不適切です。 複雑にならない限り、野生に登ることはできません。そこから抜け出すことはできません。
