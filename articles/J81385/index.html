私たちは、GPRS / EDGEテクノロジーについての最初の部分から始めた、モバイルオペレーターのネットワークでのパケット送信の知識を継続します。 この記事では、認証と承認のプロセス、いわゆる GPRS接続手順、およびサブスクライバーによって要求されたサービスのアクティブ化-PDPコンテキストを上げます。 SGSN側に保存されているデータと、サブスクライバー側にあるデータを見てみましょう。

さあ、行こう...





GPRS接続


GPRS Attachプロシージャの直前および付随のプロセスの一部、さらに具体的には以下を欠場します。

  • 無線リソースの割り当て
  • 基地局と加入者端末間の論理チャネル上のサービス情報交換
  • データ伝送チャネルの確立
  • 端末状態の変更


誰かが詳細に興味があるなら、質問をしてください。 すぐにGPRS接続手順を開始します。これにより、加入者を識別し、加入者が利用できるサービスを決定し、オペレーターのネットワークで加入者のモバイル端末を使用する合法性を確認し（ IMEIチェック手順はオプションです）、実際に、加入者に要求しているサービスをアクティブ化する機会を提供します



認証と承認のプロセス、いわゆる 下の図に示されているGPRS接続手順（写真はクリック可能）。









  1. リクエストを添付

    GPRS接続手順は、モバイル端末、GPRS / EDGEサービスからの加入者の要求から開始します（または加入者がデバイス上のパケットネットワークへの永続的な接続を選択した場合：メニュー->設定->デバイス接続->パケットデータ->パケット接続->オンデマンド/パーマネントアクセス）、つまり モバイルブラウザを開く、メールをチェックする、MMSを送信するなど。これにより、加入者がまだオペレータのパケットネットワークに接続していない場合に、接続要求の送信が開始されます。 サブスクライバーが初めてネットワークに接続する場合、次の基本データがリクエストに含まれます。

    • 接続タイプ：GPRS接続（パケットデータのみのサブスクリプション）、IMSI接続（加入者が音声サービスを作成/受信するために登録されている場合、音声サービスのみのサブスクリプション）、複合接続（複合サブスクリプション=音声+パケットデータ）
    • P-TMSI（加入者がすでにSGSNに「既知」である場合、 IMSIの置き換え）
    • RAI = MCC + MNC + LAC + RAC、つまり 基地局サブネット内の加入者座標

      RAI-ルーティングはアイデンティティ

      MCC-モバイル国コード（国際国コード）

      MNC-モバイルネットワークコード（国内の国際オペレーターコード）

      LAC-ロケーションエリアコード（1つのコードで結合されたベースステーションのセット）

      RAC-ルーティングエリアコード（LAC以下のゾーン）

    • MSネットワーク機能-データ転送に関するサブスクライバー端末機能
    • MS無線アクセス機能-無線伝送に関する加入者端末の機能


  2. 識別要求

    加入者が以前に別のSGSNのサービスエリアにいた場合、新しいSGSNでサービスに切り替えるときに、加入者は識別のためにすべての情報を再提供する必要はありません。 前のアイテムから要求されます。 その時点で加入者がGPRS / EDGEのサービスを使用した場合、つまり 彼はPDPコンテキストを開いていたため、新しいSGSNは、サービスの提供を中断することなく、セッションとともにサブスクライバーを「取得」します。

  3. アイデンティティーリクエスト（Req）/レスポンス（Res）

    この手順は、新しいサブスクライバー、または古いSGSNからデータが送信されなかった（または誤って送信された）サブスクライバーに対して実行され、その後、SGSNは再度、接続要求手順で調べたサブスクライバーからすべてのデータを要求します（P- TMSI [Packet-TMSI]、TMSI [Temporary Mobile Station Identity]は、IMSI [International Mobile Subscriber Identity]によって要求される必要があります。

  4. 認証情報の要求/解像度を送信

    その過程で、SGSNは、加入者のIMSIに基づいて、HLR / AuCにリクエストを送信します。HLR/ AuCは、オペレーターのネットワークの加入者のデータベースです。 HLR / AuC側では、加入者のIMSIは特定のチェックサム/秘密番号-K iに対応します。また、HLR / AuC側では、リクエスト用の乱数を生成するランダムジェネレーターがあります。 その後、いわゆる、いわゆる データのトリプレット[TRIPLETS = RAND + SRES（符号付き応答）+ K s ]。

    • RAND-乱数
    • SRES-結果、アルゴリズムA3によるRANDの乱数の「スイープ」
    • K with-結果、アルゴリズムA5を通じて数値K iを「スイープ」
    次に、データトリプレットがSGSNに送信されます。

  5. 認証および暗号化の要件/解像度

    HLR / AuCから受信した値はSGSN'a側に保存され、RAND番号の値は加入者のモバイル端末に送信されます。これに基づいて、SRESとK の値が加入者側で「計算」されます。 加入者のSIMカードでは、A3 / A5暗号化アルゴリズムと秘密番号K iが 「保護」されています。

  6. アイデンティティチェックリクエスト（Req）/レスポンス（Res）

    この手順はオプションであり、オペレータのネットワークで加入者の端末を使用する合法性を確認できます。 端末のIMEIコードと、ホワイト、グレー、およびブラックリストのベースとの比較を要求します。 加入者がブラックリストに載っている場合、この段階でサービスは拒否されますが、これは理論上すべてです。 実際には、逆のことが当てはまります。 ブラックリストを本当にブロックしてもまだ機能しません（ウクライナの領土を代弁します）。

  7. IMEI Req / Resを確認してください

    EIRでIMEIを確認します。これに基づいて、オペレーターのネットワークで加入者の端末を使用する正当性について決定が行われます。

  8. 場所の更新手順の要件/解像度

    GPRS接続手順中に、加入者の位置情報が更新されます。 SGSNはHLRの情報を更新し、HLRはMSC / VLRのデータを更新します。

    加入者が結合アタッチを行う場合、SGSNはMSC / VLRの加入者情報を更新します
  9. 承諾を添付

    上記のすべての操作が正常に完了した後、SGSNはGPRS Attachが受け入れられ、加入者がオペレーターのネットワークでパケットデータサービスを使用できるようになったことを加入者に通知します。

  10. TMSI再配置完了

    最後の手順は、加入者に割り当てられた新しいTMSI値をMSC / VLRに更新/通知することです。



これは、オペレーターのネットワークでパケットデータ送信を提供するために、まさにサブスクライバーの認証と承認が行われる方法です。 この手順の後、文字「G」（または「E」:)がサブスクライバーの端末に表示され、パケットネットワークへの接続が正常に完了したことを示しますが、これによりサブスクライバーはパケットネットワークでサービス*を使用できなくなります。 また、要求されたサービスのPDPコンテキストをアクティブにする必要があります。

*-成功したGPRS接続手順の後、加入者はGPRS / EDGEネットワーク経由で短いメッセージのみを送信できます。 GPRSを介したSMS。



PDPコンテキストのアクティブ化


GPRS接続手順を正常に完了した後、ユーザーはPDPコンテキストをアクティブ化できます。これにより、パケットデータサービスを使用できるようになります。



コンテキストアクティブ化手順自体は、ダイヤルアップ接続の通信アクティブ化手順に似ています。 これら2つの手順を比較してみましょう。

簡略化すると、ダイヤルアップリンクのアクティブ化手順はスキームとして表すことができます。







それでは、PDP Context'aアクティベーションスケマティックを見てみましょう。







ご覧のとおり、これら2つの手順の類似点は、同じプロトコルを使用することにあり、接続を確立する段階で使用される手順と手順は非常に似ており、通信を確立するプロセスに関係する主要なノードは似ています。



PDPコンテキストをアクティブ化するときに重要なポイントを決定したら、完全な手順を検討し、この手順中に転送されるデータを決定します。



PDPコンテキストのアクティベーションスキームを次の図に示します。







  1. PDPコンテキストリクエストのアクティブ化

    このリクエストでは、非常に多くの異なるデータが送信されますが、次のことに興味があります。

    • QoS要求-要求された加入者サービスプロファイル、接続の質的特性、このフィールドが空の場合、SGSNは割り当てられたプロファイルを決定します
    • PDPタイプ-端末が特定のIP / X.25 / etcサービスに使用するプロトコルのタイプを決定します。
    • アドレス-ネットワークでの通信のためにサブスクライバーに与えられるアドレスのタイプ[IPv4、IPv6、自動]
    • APN * [アクセスポイント名]-ユーザーのセッションを提供するGGSNのアドレスを定義するアクセスポイントの名前。

      *-コンテキストアクティベーションのプロセスにおけるAPNの選択と使用の詳細については、記事「あなたが誰であるかは関係ありません...あなたが持っているAPNが重要です」を読むことができます



  2. DNSクエリ/レスポンス

    要求で加入者から特定のAPNを受信すると、SGSNは完全なアドレスを生成し、APNに追加します。 GOI [GGSN Operator Identifier]、完全なアドレスは次のようになります。



    internet.mnc009.mcc255.gprs 、ここで



    インターネット -加入者の端末に登録されたAPN、

    mnc009.mcc255.gprs-いくつかの仮想オペレーター（ウクライナ）のGOI。



    次に、オペレーターのローカルDNSサーバーへの要求が形成され、その結果は、要求されたサービスをユーザーに提供するGGSNのIPアドレスになります。

    ローカルDNSサーバーが完全なアドレスを「認識できない」場合（たとえば、ローミングサブスクライバーの場合）、要求はトップレベルDNSサーバーにリダイレクトされます（ここでは、すべてがIPネットワークの構造に非常に似ています）。

  3. PDP Context Req / Resの作成

    IPアドレス、IMSI、 MSISDN 、APN（たとえば、内部ネットワークへのアクセスの場合）の発行要求を含む、承認されたユーザーから収集されたすべてのデータは、特別な要求[PDPコンテキスト要求の作成]によってGGSNに送信されます。 このイベントは、サブスクライバーのセッションの請求レコードを開きます。

  4. PDPコンテキストの受け入れをアクティブにする

    加入者端末に送信される応答メッセージには、PDPコンテキストのアクティブ化を完了するために必要なすべての情報が含まれています。 このメッセージを使用して、ユーザーにはオペレーターのネットワーク内の特定のIPアドレスが割り当てられ、サービスプロファイルが合意され、要求されたサービスが開始されます。



PDPコンテキストのアクティブ化手順が成功すると、ユーザー端末では「G」（または「E」:)という文字が四角で囲まれ、パケットデータ送信の使用を象徴します。
