別の方法論：スタハノフ





一般原則



  1. タスクは、実行者の便宜のために作成および作成されます。
  2. gamedisanerの作業の結果は、初心者にとっても完全で理解可能なタスクです。
  3. 実行者は割り当てを読み取り、ステータスを変更します。 （タスクではなくコードを記述します）
  4. 長時間掛かるタスク-削除されます（必要になります-再度開始します）








一般的な説明スタイル



  1. チーム：何を、誰に、いつ行うか。 あいまいさや推測はないはずです。
  2. 完全なフレーズのみ。 アクションの詳細な（ステップバイステップ）説明付き。
  3. 文脈から外れたフレーズは意味のあるままにする必要があります。
  4. すべてのパラメーターにはデジタル値（要素数、画像サイズ、行長）が必要です
  5. すべてのリストには番号を付ける必要があります（チェックリストの作成が簡単です）
  6. 長いリストは、3つの要素のセマンティックグループに分割する必要があります
  7. 図と本文の碑文は逐語的に一致する必要があります。 それらが一致しない場合、これらは異なる碑文です。
  8. エラーや極端な状況（不正な文字、長い行、コミュニケーションの損失、間違った時間、高速で不安定なタパス）に細心の注意が払われます




機能説明



  1. 起動/呼び出し方法は？
  2. すべての回答オプションの完全なリスト
  3. 各アクションオプションに対する反応の完全なリスト
  4. 発生する可能性のあるすべてのエラーの詳細なリスト
  5. 極端な状況の説明（長い回線、通信の切断など）
  6. それを閉じる方法、それの後に行く場所は？




職種



新機能の紹介


  1. ゲームデザイナに割り当てられる大きな機能が作成されます（機能の導入を担当）
  2. ゲームデザイナーは、機能をサブタスクに分割し、それぞれをペイントします。
  3. サブタスクは1人で1日以内に完了する必要があります。
  4. 期限の見積もり、リストの作成、技術の調査などのタスクを含みます。


短いタスク：


  1. 一般へのリンク（大きなタスク）
  2. タスクで他のタスクへの要素またはリンクが言及されている場合、それらを示す必要があります
  3. リストには番号を付ける必要があります
  4. 長いリストはトリプル（意味内の要素のグループ化）に分割する必要があります
  5. 短いタスクには、外部ドキュメントまたは画像へのリンクを含めないでください。必要なものはすべてタスク自体に示す必要があります（外部ソースが消える場合があります）


修正タスク：


  1. 再現方法（簡単な方法）-テストレベルなどが必要な場合があります。
  2. 正確に機能していないもの
  3. 正しく動作する方法（バグが修正されたことを確認する方法）




会員





プロデューサー


  1. ゲームデザイナーにタスクを割り当てます（デザインノートステータス）
  2. ゲームデザイナーによってスケジュールされたタスクのビューとコメント
  3. ゲームデザイナーによってスケジュールされたタスクをサブスクライブして追跡する
  4. 他の待機タスクリストを確認し、ヘルプが必要な場合は介入します


ゲームデザイナー


  1. プロデューサーによって設定されたタスク（サブタスクの形式で1日以内の短いタスク）をスケジュールします。
  2. ペイントされたデザインをプロデューサーとメインプログラマーと調整します。
  3. 最終的にタスクを完了して同意すると、デザインノート->技術タスクのステータスが変更されます
  4. 解決済みタスクを選択し、その実行を確認します
  5. タスクの条件が満たされない場合、タスクは再発見されます
  6. 条件が満たされると、タスクは閉じられます。
  7. 新しい条件の場合（そして今すぐ右の矢印が表示されます）-前のタスクへのリンクを持つ新しいタスクが作成されます。
  8. すべてのサブタスク（ゲームデザイナーによってペイントされた）が閉じている場合-プロデューサーによって設定されたグローバルタスクを閉じることができます。


プログラマー


  1. 現在のタスクが完了すると、タスクを閉じて新しいタスクの選択を開始します
  2. 最初に保留中のタスクを確認する必要があります。他の待機中-続行できますか？
  3. 遅延されたものの中から見つけることができなかった場合、プログラマに割り当てられた最高の優先順位を持つTech Taskの中からのみ新しいものを選択します。
  4. そのようなタスクが3つ以上ある場合-彼はゲームデザイナーにタスクを優先順位でランク付けするように依頼します（高-3以下）
  5. 選択したタスクは研究中です
  6. タスクが不完全であるか、わかりにくい場合-ゲームデザイナーに返送されます
  7. すべてが明確な場合-進行中のステータスを変更し、タスクの完了を開始します
  8. タスクを完了できなくなった場合-待機中その他のステータスが設定されます
  9. タスクは延期されます-これは、ゲームデザイナーとプロデューサーに支援の必要性に関する合図です。
  10. タスクが完了した場合-ステータスを「解決済み」に設定します（リポジトリコミットのコメントを介して変更できます）
