時間の変更、またはロシア連邦の5つの地域でのタイムゾーンの変更

あなたがまだ知らないなら、ロシアの時間帯の数の減少が始まっていることをお知らせします。 5つの地域では、近隣のタイムゾーンへの参加が3月28日の次の日曜日に既にスケジュールされています。 このイベントの1週間以内に残ります！



私はテレビを見ていません。おそらくこの問題については長い間詳細に議論されてきましたが、個人的には、この情報は今週の初めに新しくなりました。

最近まで、ロシアのタイムゾーンの数を5個に減らすという考えは、単に誰かの失敗した提案または別の妄想法案であり、これは確実に拒否され、さらなる議論は行われません。 しかし、この主題に関する「そこまで」のすべての決定はすでに行われており、関連する政府の決定はすでに署名され（V.プーチンによる）、公開されていることがわかります。

ケメロヴォ地域に関する決定は2009年秋に署名され、サマラ地域、ウドムルチア、カムチャッカ、チュコトカに関する決定は先日署名されました。







次に、ITインフラストラクチャ管理の観点から、この移行がいつ、どのように発生するのか、誰が影響するのか、何を期待するのか、どのように準備するのかを検討します。



*決定のテキストへの参照：

  1. 2009年9月14日N 740モスクワのロシア連邦政府令「ケメロヴォ地域の領土における5番目のタイムゾーンの適用について」
  2. 2010年3月17日のロシア連邦政府の法令N 166モスクワ「ウドムルト共和国の領土における2番目のタイムゾーンの適用について」
  3. 2010年3月19日のロシア連邦政府の命令N 170モスクワ「サマラ地域の領土における2番目のタイムゾーンの使用について」
  4. 2010年3月19日のロシア連邦政府の法令N 171モスクワ「カムチャツカ準州およびチュコトカ自治管区の領土における第10時間帯の適用について」


*どこで、いつ？



タイムゾーンの変更は、ロシア連邦の5つの科目に同時に影響します。

1.サマラ地方。

2.ウドムルト共和国。

3.ケメロヴォ地域。

4.カムチャッカ半島。

5.チュコトカ自治管区。



このタイムゾーンの変更をできるだけスムーズかつシームレスにするために、彼らはそれを夏時間に切り替える瞬間と組み合わせることにしました。 つまり 2010年3月28日（日曜日）の現地時間02:00にロシアの他のすべての地域で時計が1時間（現地時間03:00）進む場合、指定された5つの地域で2010年に時計が夏時間に切り替わりますしたがって、現地時間の午前02:00には、これらの地域は時計なしで隣接するタイムゾーンの夏時間に自動的に切り替わります。 さて、これらの地域では、切り替え先のタイムゾーンに従って、さらにクロック転送が実行されます。



この移行後、2010年3月28日に、ロシアにはサマラ時間とカムチャッカ時間の 2つのタイムゾーンが存在しなくなります。



これで、子供の頃から多くの人にとって通常のことを聞くことはなくなります。「モスクワは15時間、ペトロパブロフスク-カムチャツキーの深夜」、カムチャッカはモスクワ時間の16:00に深夜になります。



しかし、これはすべての歌詞です。 実際、私はここでこのニュース自体ではなく、全体としてそのような変化の便宜性について議論したかっただけです。 まず、このタイムゾーンの変更がITインフラストラクチャにどのように影響するか、つまり 将来のイベントを考慮して、ITスペシャリストおよび企業のすべてのシステム管理者のために何を準備するか。



たとえば、組織にはこれら5つの地域のうち4つ（チュコトカを除く）に支社があり、ITインフラストラクチャはすべての支社にあります。 会社が管理するこれらの地域にITインフラストラクチャもある場合は、この問題にも事前に対応する必要があります。



*時間が経つとどうなりますか？



最初に、このタイムゾーンの変更、および夏時間/冬時間への移行では、現地時間の変更、つまり世界標準時に対する現地時間のシフトのみを明確に理解する必要があります。 これらの地域とそのITインフラストラクチャの絶対世界時（UTC）自体は変化せず、ジャンプもしません。中断もジャンプもせずに、均等かつ継続的に継続します。

したがって、ここでパニックに値することは絶対にありません。この移行により、グローバルな問題や一時的な飛躍が生じることはありません。



UTCで動作し、現地時間をまったく使用しない多くのUNIXサーバーでは、このタイムゾーンの変更は一般に見過ごされます。



ただし、一部のアプリケーション、サービス、およびインターフェイスでタイムゾーンへのバインドがあり、現地時間が使用される場合、調整のためにここでシステムの変更が必要になります。 たとえば、すべてのユーザーのデスクトップのシステムクロックに正しい現地時間を表示させる必要があります。 一部のアプリケーションが現地時間でログを書き込む場合、ここでも現地時間の正しい表示が必要です。 さて、現地時間を使用する他のすべてのインターフェースは、それを正しく表示するはずです。



このタイムゾーンの変更により、開発者は問題を抱えることがあります。 たとえば、あるフォーラム/ブログのプロセッサのルールでは、「ユーザーのプロファイルに場所[サマラ地方]または[ウドムルト共和国]が含まれている場合、現地時間UTC + 4（夏UTC + 5）」など各地域。 この場合、3月28日からのローカルタイムハンドラーのルールテンプレートを調整する必要があります。

アプリケーションや他のプロジェクトで現地時間への拘束力があるかもしれない他のケースを考えることができます。



繰り返しますが、ITインフラストラクチャとプロジェクトを注意深く見て、上記の5つの地域のタイムゾーンまたは現地時間に関係があるかどうか、すべての人にアドバイスします。 そして、もしあれば、いくつかの決定を下し、必要に応じて変更をテストする価値があります。



*システムにどのような変更が必要ですか？



ローカルOS内のローカル時間で動作するすべてのアプリケーション/サービスは、通常、システムのローカル時間に特別に結び付けられています。 したがって、デスクトップ/サーバー群（特に、Windowsマシン）の場合、現地時間を調整する必要があります。



もちろん、時間そのものの価値は、機械に触れる必要はありません（時間はグローバルな意味で変化しません）。 対応する地域のすべてのマシンでタイムゾーン（TimeZone）のみを変更する必要があり、2つのオプションがあります。



オプション1

すべてのマシンで以前に設定したタイムゾーンのままにしますが、選択したタイムゾーンのパラメーターで、UTCに対する相対的なシフトを変更します。 つまり、たとえば、カムチャツカ時間の場合、UTC + 12（夏の場合：UTC + 13）の代わりに、マガダン時間のようにシフトUTC + 11（夏の場合：UTC + 12）を示します。

TZedit.exeユーティリティを使用して、またはHKEY_LOCAL_MACHINE \ SOFTWARE \ Microsoft \ Windows NT \ CurrentVersion \ Time Zones \のシステムレジストリを直接変更することにより、Windowsのシステムタイムゾーン設定を編集できます。



オプション2

タイムゾーンのシステムパラメータを変更せずに、単に現在のタイムゾーンとして、近隣地域（タイムゾーンが結合された地域）のタイムゾーンを選択します。



両方のオプションは機能していますが、イデオロギー的には、隣接するものと同じパラメーターで重複したタイムゾーンを作成するのではなく、既存の隣接するタイムゾーンを選択するだけです（つまり、オプション2）。 さらに、政府の決定の文言は、その地域が近隣のタイムゾーンに入ることを示しており、そのタイムゾーンのパラメーターを変更しません。
