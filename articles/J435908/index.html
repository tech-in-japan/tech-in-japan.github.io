禁欲的なWeb：GoとJSのプロトタイプフリーマーケット







みなさん、こんにちは。最新のWebアプリケーションの可能性についての私の考えの結果を共有したいと思います。 例として、コミック用の掲示板の設計を検討してください。 ある意味では、問題の製品は、オタクとシンパサイザーの聴衆向けに設計されており、インターフェイスで自由を示すことができます。 それどころか、技術コンポーネントでは、細部への注意が必要です。







実際、漫画では何もわかりませんが、フリーマーケット、特にゼロで人気のあったフォーラム形式は大好きです。 したがって、次の結論が流れる仮定（おそらく誤り）は1つにすぎません。アプリケーションとの対話の主なタイプは表示であり、2番目のタイプはアナウンスとディスカッションの投稿です。







私たちの目標は、シンプルなアプリケーションを作成することです。 技術的ノウハウ ただし、現代の現実と一致する余分なホイッスル。 私が達成したい主な要件は次のとおりです。







  1. サーバー側：







    a）クライアントへのユーザーデータの保存、検証、送信の機能を実行する

    b）上記の操作は、許容量のリソース（時間を含む）を消費します

    c）アプリケーションとデータが一般的な攻撃ベクトルから保護されている

    d）サードパーティのクライアントとサーバー間対話のためのシンプルなAPI

    e）クロスプラットフォームで簡単な展開







  2. クライアント側：







    a）コンテンツの作成と使用に必要な機能を提供します

    b）インターフェイスは、通常の使用、アクションへの最小パス、画面ごとの最大データ量に便利です。

    c）サーバーとの通信外で、この状況ですべての可能な機能が利用可能

    d）インターフェイスは、再起動して待機することなく、状態とコンテンツの現在のバージョンを表示します

    d）アプリケーションを再起動しても、その状態には影響しません

    f）可能であれば、DOM要素とJSコードを再利用します

    g）実行時にサードパーティのライブラリとフレームワークを使用しません

    h）レイアウトは、アクセシビリティ、パーサーなどのセマンティックです。

    i）メインコンテンツのナビゲーションには、URLとキーボードを使用してアクセスできます

プロジェクト構造



comico / modelパッケージは3つのファイルに分かれています。

model.go-データ型と一般的な機能の説明が含まれます。作成/更新（buntdbはこれらの操作を区別せず、レコードの存在を手動で確認します）、検証、削除、1つのレコードの取得、リストの取得。

rules.go-特定のタイプおよびロギング機能の検証ルールが含まれています。

files.go-画像を操作します。

Mtimesタイプは、データベース内の残りのタイプの最後の変更に関するデータを格納し、どのデータが変更されたかをクライアントに伝えます。







comico / bdパッケージには、データベースとやり取りするための一般化された関数が含まれています：作成、削除、選択など。Buntdbは、ファイルへのすべての変更をテキスト形式で保存します（この場合、1秒に1回）。 dbファイルは編集されず、トランザクションが成功した場合の変更が最後に追加されます。 データの整合性に違反する私の試みはすべて失敗しました。最悪の場合、最後の1秒の変更は失われます。

実装では、各タイプは個別のファイル内の個別のデータベースに対応します（メモリに排他的に保存され、再起動時にゼロにリセットされるログを除く）。 これは主に、バックアップと管理の便利さ、小さなプラス-1つのタイプのデータのみへのブロックアクセスを編集するためのオープントランザクションによるものです。

このパッケージは、たとえばSQLなどの別のデータベースを使用して、同様のパッケージに簡単に置き換えることができます。 これを行うには、次の機能を実装するだけで十分です。 







 func Delete(db byte, key string) error func Exist(db byte, key string) bool func Insert(db byte, key, val string) error func ReadAll(db byte, pattern string) (str string, err error) func ReadOne(db byte, key string) (str string, err error) func Renew(db byte, key string) (err error, newId string)


comico / cnstパッケージには、すべてのパッケージに必要な定数（データ型、アクション型、ユーザー型）が含まれています。 さらに、このパッケージには、サーバーが外部の世界に応答するための人間が読めるすべてのメッセージが含まれています。







comico / serverパッケージにはルーティング情報が含まれています。 また、数行（Echo開発者のおかげ）、JWTを使用した承認、CORS、CSPヘッダー、ロガー、静的配布、gzip、ACME自動証明書などが構成されています。

API



サーバーとデータを交換するには、フェッチAPIを使用します。 更新されたレコードを短い間隔でダウンロードするには、 / pub / mtimesにリクエストを行います。いずれかのタイプの最終変更時刻がローカルのものと異なる場合、このタイプのリストをロードします。 はい、SSEまたはWebSocketsを介した更新の通知と増分ロードを実装することができましたが、この場合は通知なしで実行できます。 私たちは何を得ました：

フィルタリングとページネーションには、ナビゲーションデータに基づいて計算されたSvelteプロパティを使用します。 計算された値の方向は次のとおりです： items （サーバーからのレコードの配列）=> ignoreItems （現在のユーザーの無視リストに基づいてフィルターされたレコード）=> scribedItems （このモードが有効になっている場合、サブスクリプションのリストによってレコードをフィルターします）=> curItemおよびcurItems （現在のレコードを計算します）セクションに応じて）=> filtersItems （検索クエリに応じてレコードをフィルタリングします。レコードが1つしかない場合-コメントをフィルタリングします）=> maxPage （ページあたり12レコード/コメントに基づいてページ数を計算します）=> pagedItem （最終配列を返します 現在のページ番号に基づいた投稿/コメント）。







コメントと画像（ commentsと_images ）は個別に計算され、タイプと所有者レコードごとにグループ化されます。







計算は自動的に行われ、関連するデータが変更された場合にのみ、中間データは常にメモリ内にあります。 これに関して、不愉快な結論を出します-大量の情報および/または頻繁な更新のために、大量のリソースを費やすことができます。

  • トランジションと他のUIイベントの見かけの速度については、可能な限り、バックグラウンドで発生するデータと、現在の計算結果を使用するインターフェイスに関連付けられたアクションでの操作を分離し、必要に応じて再構築し、残りはフレームワークによって親切に実行されます。
  • 入力するフォームのデータは、入力を失うことを防ぐために、入力ごとにlocalStorageに保存されます。
  • すべてのコンポーネントで、不変モードを使用します。このモードでは、フィールドの変更に関係なく、新しいリンクが受信された場合にのみプロパティオブジェクトが変更されたと見なされます。したがって、コード量がわずかに増加するにもかかわらず、アプリケーションを少し高速化します。
