nginxのチューニング

この記事は、CodeCamp 2009での私のプレゼンテーションに基づいています。



私たちの多くにとって、サイトの視聴者が急速に成長し始める待望の日が来ています。 毎朝、私たちは息を切らして呼吸し、グーグルのアナリティックスチャートを見て、一日に次の1000人の訪問者に線が引かれると笑顔でぼやけます。 原則として、トラフィックの増加は技術ベースの成長と一致せず、サイトの速度は低下し始めます。 ここで、システム管理者がゲームに入ります...



どのプロジェクトにも常に最適化するものがあります。webo.inで最適化のヒントを読んで、データベースにeaccelerator 、memcache、インデックス検索フィールドをインストールできます。 これはすべて完了していると思いますが、サイトはまだ遅いです。



nginxを最適化するときです...





Nginxのコンパイル



組み立てるとき、私は通常、「自分が使用していないものをすべて切り離す」というルールに従います。 そのため、役に立たない可能性のあるめったに使用されないモジュール： mail、mail_ssl_module、http_perl_module、http_flv_module、http_dav_module 。 将来、いくつかのMSモジュールが必要な場合、再コンパイルに数分かかります。



コンパイル時に含めることが望ましいモジュール： http_gzip_static_module、http_stub_status_module 。



nginx （％nginx_datadir、... specファイル変数） をコンパイルするためのspecファイルの一部は次のとおりです。 



./configure \ --prefix=%nginx_datadir \ --conf-path=%nginx_etc/nginx.conf \ --sbin-path=%{_sbindir}/%{name} \ --error-log-path=%nginx_log/nginx.error.log \ --http-log-path=%nginx_log/nginx.log \ --http-client-body-temp-path=%nginx_spool/tmp/client \ --http-proxy-temp-path=%nginx_spool/tmp/proxy \ --http-fastcgi-temp-path=%nginx_spool/tmp/fastcgi \ --pid-path=%_var/run/nginx.pid \ --user=%nginx_user \ --group=%nginx_group \ --with-cc-opt="-I %_includedir/pcre/" \ --with-http_ssl_module \ --with-http_realip_module \ --with-http_addition_module \ --with-http_sub_module \ --with-http_dav_module \ --with-http_flv_module \ --with-http_gzip_static_module \ --with-http_stub_status_module \ --with-http_perl_module

単純な設定の概要を説明し、そのディレクティブを分析します。

Nginxサーバーの監視



nginx configでは、セクションを規定します 

  location = /stat { stub_status on; access_log off; allow xx.xx.xx.xx; deny all; }


現在、nginxの統計はhttp://simple.com/statで表示できます。



便宜上、nginxによってホストされているapacheの統計を構成することもお勧めします。 これを行うには、まずmod_rpafモジュールをインストールします。これにより、ApacheはIP nginxではなくクライアントのIPアドレスを「見る」 ことができます （ ここでは、私のバージョンのrpaf srpmをダウンロードしてコンパイルできます ）。 
 LoadModule rpaf_module modules/mod_rpaf-2.0.so # # Mod rpaf # RPAFenable On RPAFsethostname On RPAFproxy_ips 127.0.0.1 xx.xx.xx.xx RPAFheader X-Forwarded-For


同様の機能を提供する代替モジュールmod_extract_forwardedがあります（Fedoraリポジトリで利用可能）。 （ ティモシャに感謝）



統計表示モジュールを接続します。 
 ExtendedStatus On SetHandler server-status Deny from all Allow from xx.xx.xx.xx
現在、Apacheの統計はhttp://simple.com/apache-statで表示できます。





静的コンテンツの種類



条件付きで静的を2つのカテゴリに分割します。
軽いコンテンツ ：html、css、js、xml、rss、txt。 圧縮率が高く、保存スペースもほとんど必要ありません。 いずれの場合でも、nginxを使用すると、パフォーマンスが大幅に向上します。
「重い」コンテンツ ：写真、ビデオ、オーディオファイル。 ボトルネックは、主にディスクシステム、RAMのサイズ、およびチャネルの帯域幅です。 このタイプのコンテンツを配信するタスクは、コンテンツの保存と、実際にはコンテンツの配信という2つに分けられます。 nginxを使用すると、最小限のメモリ消費とCPU時間を実現できますが、負荷を増やしても、RAIDアレイを収集する必要があります。


サイトの速度が低下し始めたらどうするか



まあ、まず、パニックにならないでください。 おそらく、あなたのサイトで人気が出るのはこの瞬間です:)

原則として、スローダウンするのはnginxではなく、バックエンド（動的コンテンツを生成するサーバー）、またはよくあることですが、データベースサーバー、ディスク領域が不足し、スーパーデュパーアナライザーが起動してすべてのリソースを取得します。

この記事では、nginxが疑われる場合について検討します。 サーバーの問題を一時的に緩和するために、何を強化できますか？

  1. worker_processesの数を増やしてみてください。nginxの作成者は、コアの数に応じてそれらをインストールすることをお勧めします。 この数は、「コアの数」〜「コアの数x 2」の範囲でほぼ変化しましたが、接続の確立は高速になりましたが、http-requestの処理は常に効率的ではありません。 worker_connections（1人のワーカーのコネクシンの最大数）がまだあることを忘れないでください
  2. worker_priorityを-5以下に設定すると役立ちます（最大-20）。 他のサービスが著しく遅くなり始める可能性があるため、ここで注意してください。 サーバーでは、このオプションは-5に設定されています。 値が低いほど、nginxの優先度が高くなります
  3. 小さなファイルの大量の出力と低速のハードドライブを使用すると、access_log offログの一時的なシャットダウンが役立ちます。
  4. UPD：noatimeオプションを使用して、配布元のディスクをマウントします。これにより、ディスクへの書き込み操作の回数が減ります。 （ coolspot 、 pwlnw 、 alfaのアイデアをありがとう）
  5. ボトルネックを探して、修正できます。 コマンドが役立ちます：top、iostat、df -h、iptraf
  6. RAMを追加するか、ディスクシステムをアップグレードします（たとえば、RAIDアレイをインストールし、SSDハードドライブを試すことができます）






最適化テクニック



軽量コンテンツ

  1. 仮想ディスク。

    仮想ディスク（tmpfsまたはramfs）、フォルダーjs、css、イメージ（コンテンツではなくデザインに関連する画像が少量ある場合）を作成し、そこに転送し、nginx configに個別に書き込みます
    場所/ js / {

    ルート/var/www/img_virtual/auto.ria.ua/js

    }

    ...

    再起動時に仮想ディスクが自動的に作成されるようにするには、 / etc / fstabに追加します
    なし/ var / www / img_virtual tmpfsサイズ= 1g、モード= 1777 0 0
    （起動時に、1Gディスクが自動的に作成されます）

    ここでは、次の状況に注意する必要があります。静的ファイルがシステムキャッシュに落ちた場合、システムキャッシュからの戻り速度は仮想ディスクからの戻り速度に等しくなります。 言い換えると、配布されるすべての「軽い」静的の総量が少ない場合、仮想ディスクの必要はもうありません（テストを実行してこれを確認するように促してくれたmaxpに感謝します）

  2. Gzipコンテンツの圧縮

    仮想フォルダーで実行する
    for i in `find ./* -type f -name '* .js'`; エコー$ i; gzip -c -9 $ i> $ i.gz; 完了;

    for i in `find ./* -type f -name '* .css'`; エコー$ i; gzip -c -9 $ i> $ i.gz; 完了;

    nginx構成で、次の行にgzip_staticを追加します。
    場所/ js / {

    gzip_static on;

    ルート/var/www/img_virtual/auto.ria.ua/js

    }

    動的ファイルのオンラインパッケージを有効にすることもできます。
    場所/ {

    gzip on;

    gzip_min_length 1100;

    gzip_buffers 16 8k;

    gzip_comp_level 3;

    gzip_typesテキスト/プレーンアプリケーション/ xmlアプリケーション/ x-javascriptテキスト/ css;



    ルート/var/www/auto.ria.ua/

    }

  3. コンテンツプロキシヘッダー

    ヘッダーで静的コンテンツの有効期間を指定すると、負荷も軽減されます。 このために、 expiresディレクティブを使用します 。 コンテンツが変更されない場合、expires maxを使用できます。 1dを期限切れにしても、良い結果が得られます。
  4. ファイル記述子キャッシュ

    これにより、ディレクトリ階層が拡張された小さなファイルが多数ある場合にパフォーマンスが向上します。 存在しないファイルへのアクセスをキャッシュすることもできます。 次のようになります。
    場所/ {

    root / var / www /;





    open_file_cache max = 1024非アクティブ= 600s;

    open_file_cache_valid 2000s;

    open_file_cache_min_uses 1;

    open_file_cache_errors on;

    }



重いコンテンツ

  1. まず、コンテンツの配信時にスワップが使用されているかどうかに注意する必要があります。 スワップが使用されている場合、 スワップが通常のSATAハードドライブにある場合は問題になります。 Swingはsendfileカーネルの方法が好きで、確かにプログレッシブテクノロジーですが、 swapを使用するとリターンに大きく影響し、 sendfileディレクティブでsendfileを無効にし、 output_buffers 2 64kなどの最適な出力を選択しようとします。

    sendfileをオフにした構成例：
    sendfile off;

    tcp_nodelay on;

    output_buffers 2 64k;

    多くはシステムで使用されるカーネルに依存するという事実に注意を喚起します。これはkernels = 2.6.27で助けになりました。
  2. RAMが許せば、最も要求の多いファイルを配置する仮想ディスクを作成します。たとえば、10分ごとに、これらのファイルのリストを調整できます。 これで、 try_filesディレクティブを適用できます。
    場所/ {

    root / var / www /;

    try_files / img_virtual / hot / $ uri ストレージ ;

    }



    ロケーションストレージ {

    proxy_pass http：//バックエンド;

    proxy_set_header Host $ host;

    }

    ファイルが見つからない場合、バックエンドは仮想ディスク上でアクセスされます。



    通常、「ホットコンテンツ」は全体の10％未満です。



    キャッシュ形成プログラムを作成できない場合は、 proxy_storeディレクティブを使用します
    ロケーションストレージ {

    proxy_pass http：//バックエンド;

    proxy_set_header Host $ host;



    proxy_store on;

    proxy_store_accessユーザー：rwグループ：rw all：r;

    proxy_temp_path / var / www / img_virtual / hot /;

    root / var / www / img_virtual / hot /;

    }

    この構成では、要求された各ファイルがキャッシュされます。

    時間が経つにつれて、クラウンを実行する最も簡単な方法であるキャッシュを何らかの方法でクリーンアップする必要があります。
    cd / var / www / img_virtual / hot /

    find ./ -type f -amin +60 -delete

    （ファイルがキャッシュから60分以上要求されない場合、削除します）
  3. ストレージが大きい場合、テラバイトがかかります-問題は運用上解決できません。 フロントエンドでRAIDを構築できます。 フェイクレイドを使用することはお勧めしません。これはアップグレード時や光が消えるときに頭痛の種です！ SASネジを増やし、本格的なRAIDコントローラーを使用します（hostraidはありません！）。 スワップ、スプール、およびキャッシュをそこにマウントし、コンテンツを頻繁に変更せず、キャッシュを頻繁に書き換えないようにするには、SSDハードドライブを使用できます。 これはすぐに動作します。このようなハードドライブには、シークツーシーク、低消費電力（Intel X25-M 0.15Wなど）、良好な転送速度（最大250 MB / s）などの特性がありません。
  4. プロキシされたリクエストをキャッシュします。

    2009年3月23日に、次のベータ版nginx 0.7.44がリリースされました。このリリースでは、プロキシされたリクエストをキャッシュするための実験的なサポートが登場しました。 nginxユーザーにとってこのイベントの重要性を過大評価することは困難です。 著者nginxは、大量の静電気の分布を管理する強力なツールを提供してくれます。

    なぜそのようなキャッシュが必要なのですか？ この質問に対する答えは、主にディスク操作とネットワーク操作の異なる「コスト」に関連しています。 多くの場合、ディスクに「行く」ことは、ネットワークにアクセスするよりもはるかに「安く」なります。 このようなキャッシングの主な目的は、必要な最小限のネットワーク操作とディスクキャッシュの「インテリジェント管理」を削減することです。

    プロキシ要求のキャッシュ設定の詳細をご覧ください。
