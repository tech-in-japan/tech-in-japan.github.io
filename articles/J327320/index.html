ソフトウェア開発の管理をより困難にするものの1つは、計画の複雑さです。 知っておくべきことが2回あります。 時間Aは、コードの記述にかかる時間です。 私たちはそれを本当に不十分に評価します-時間Aを推定する方法の質問に答える科学はありません。しかし、時間Bを推定すると状況はさらに悪化します-コードを正しく動作させるのにかかる時間です。

今年6月にECMA総会に送信されるJavaScriptには多くの追加があります（ これらの革新は実際に認証手続きを経て、新しい仕様の一部になりました-ECMAScript 2015 Language Specification-Ed。 ）。

末尾再帰



最初の最良の方法は、末尾再帰と呼ばれる構造です。 関数が最後に実行する場合、関数の呼び出しの結果（別の関数の呼び出しである可能性があります）を返します。呼び出しの逆シーケンスを作成する代わりに、コンパイラーはジャンプを生成します。 したがって、コードは少し速く動作しますが、これは素晴らしいことです。 しかし、さらに良いこと-一部のモデルでは、これはメモリ消費を大幅に削減します。 したがって、まったく新しいクラスのアルゴリズムを使用できます。



たとえば、継続転送スタイルや、以前の言語では実装できなかった他の種類のプログラミングを使用できます。 したがって、この関数を使用すると、JavaScriptは最終的に実際の関数型プログラミング言語になります。これは素晴らしいことです。



省略記号



省略記号演算子も取得しました。 パラメータまたは引数のリストに入れると、可変数の引数を処理できます。これは本当に素晴らしいことです。



ここには、カレー関数の2つのバージョンがあります。 1つ目はES6に準拠しています。 



function curry(func, ...first) { return function (...second) { return func(...first, ...second); }; }


2つ目は、以前の実装方法です。 



 function curry(func) { var slice = Array.prototype.slice, args = slice.call(arguments, 1); return function () { return func.apply(null, args.concat(slice.call(arguments, 0))); }; }


これはすべてひどいので、2番目のバージョンで正確に何が起こるかについては説明しません。 反対に、最初はかなり合理的なように見えます：私たちがポイントを見るところならどこでも（ここではあなたが望むだけ多くの引数を持つことができます）。 そしてこれは素晴らしい。



この革新により、以前はできなかったことを実行できなくなりました。 しかし、可変数の引数を処理する必要がある場合、このアプローチははるかに優れています。

これで正しく実行できます。 素晴らしいと思わない場合は、以前の基準に従って同じことを試してください。 しかし、ES6を使用すると、ロックなしで完全に非同期の言語で最終的にそれを行うことができます。



定数



変数を定義する2つの新しい方法があります。letとconstで、ブロックスコープの問題を解決します。 良いプログラムでは、ブロックのスコープは必要ないことがわかります。 しかし、以前はJavaScript構文がブロックのスコープのように見えました（その構文構文varを使用）が、そうではなかったので、人々を混乱させました。 混乱が原因でエラーが発生するたびに。 LetとConstは、これを回避できるようにします。



繰り返しますが、新しいデザインでは、以前は作成できなかったプログラムを作成できません。 しかし今では、コードはJavaプログラマを悩ませません。それは良いことです。



再編



別の構文上のキャンディーであるリストラがあります。 また、以前はできなかったことを実行することもできません。 しかし、いくつかの点で、新しい構文ははるかに表現力があります。



たとえば、ここにオブジェクトがあります。 そして、オブジェクトのプロパティからこれらの変数を初期化して、いくつかの変数を作成したいと思います。 



 let {that, other} = some_object; let that = some_object.that, other = some_object.other;


これは非常に単純化された例です。 後で、これを使用する方法の別の例を示します。



弱マップ



WeakMapができました。 WeakMapは、オブジェクトが機能するはずの方法で機能します。 JavaScriptオブジェクトでは、キーは文字列ですが、これは間違いです。 代わりにいくつかの値を使用した方が良いでしょう。 しかし、それにもかかわらず、これらは線です。 WeakMapはこの問題を解決します。 ここでは、任意の値を取り、それをキーとして使用できます。



残念ながら、このことを追加する必要があり、言語が非常に複雑になりました。 さらに、プログラミング言語で使用された最悪の名前（ weak-weak ）と呼びました。 誰も彼らのプログラムに何か弱いものを入れたくない。 しかし、この設計は非常にうまく機能します。



WeakMapを使用すると、以前の言語では作成できなかったプログラムを作成できます。



パターン文字列



最後に、メガストロークリテラルと呼ばれるものです（言語ではテンプレート文字列と呼ばれますが、この名前は好きではありませんが、以前は準リテラルと呼ばれ、さらに混乱していました）。



これは、ES6の複数の行に一致する正規表現です。 この正規表現を宣言するのはひどいです。 いつかもっと良いことをしたいと思います。 しかし、ES6では、これ以上の代替手段はありません。



したがって、すべてのスペースを削除した後、文字列を受け取り、それを正規表現に変換する関数があります。 このような関数を使用すると、任意の式を取ることができますが、すべてをヒープに書き込むことはできませんが、必要な空のスペースをそこに入れて、要素とそれらの相互関係を確認します。 結果は同じになります。



この設計には1つの欠点があります。コンパイラは正規表現を文字列として扱うため、検証を実行できません。 検証は、コンストラクタを呼び出すまで行われません。 しかし、レコードの前のバージョンでは、すべてがヒープに混在しているという事実のために、重大なエラーがコンパイラーを簡単にすり抜けることもありました。 したがって、私たちはこの変換で多くを失うとは思いません。



ところで、正規表現を頻繁に使用する場合は、RegulExと呼ばれるツールを強くお勧めします。 その中に正規表現を入れると、ダイアグラムが構築され、内部で何が起こっているかを正確に見ることができます。 正規表現を書くたびに使用します。

しかし、私の仮説では、これを言語から削除しても、優れたプログラムを作成するのに十分な関数型プログラミング言語が残ることになります。 それをテストするために、私は（これなしで）制限なしでJavaScriptでも書き始めました。 そして、私はこれを持たずに、より少ない労力でより良いプログラムを書き始めたことに非常に驚きました。 とてもいい。 言うまでもなく、これが言語自体に存在することが困難な場合、つまり英語でコミュニケーションをとる場合、言語の構築を意味するのか、代名詞を意味するのかを事前に知ることはできません。 これはかなりトリッキーです。



ヌル/未定義



nullの使用を停止しました。 JavaScriptには、nullとundefinedの2つの最小値があります。 一部の言語では、これを使用すべきではないと考えられています。 JavaScriptは、そのような値を複数持つ唯一の言語です。 しかし、一度に2つ持つのは愚かであることは間違いありません。



一部のフレームワークは、それらを交換可能として扱いますが、そうではありません。 状況に応じて、そのうちの1つだけを使用する必要があります。 それを使用するのは言語自体であるため、undefinedを使用しました。たとえば、欠落しているプロパティにアクセスすると、undefinedになります。



不定の値は、完全に間違ったタイプの欠落オブジェクトの問題の全範囲を引き起こします。 これがES6で修正されることを期待していましたが、これは起こりませんでした。 おそらく常にそうでしょう。 ただし、nullを使用していない場合、この問題は発生しません。 また、誤った値の使用を停止し、JavaScriptに存在すること自体が悪い考えだと判断しました。



このアイデアはもともとCに由来し、Cは0、no、false、およびその他の類似値を表すために使用します。 JavaScriptは同じことを試みました。 しかし、これは混乱を招きます。



のために



forの使用を停止しました。 ES5では、配列を操作するための新しい一連のメソッドを導入しました-それぞれ、マップなどのために、これらのツールを使用します。 配列があり、それを処理する必要がある場合、これらの方法のいずれかを使用します。 そして、それらを便利な方法で組み合わせることができます。 それは本当に表現力豊かです。



それぞれについて



私はそれぞれに使用しません。 ES5では、オブジェクト.keys（オブジェクト）を取得しました。 この構造は、文字列の素晴らしい配列を提供します。 また、プロトタイプチェーンからのものが含まれていないため、結果をフィルタリングする必要はありません。 この方法で配列を取得し、それぞれに対して実行できることは非常に素晴らしいことです。これは本当に表現力豊かな方法です。



前に、私は通常のテール再帰がES6に登場したことを言及しました。 そして、今度はループ（特にwhileステートメント）の使用を完全に停止します。再帰関数のみを使用します。



たとえば、再帰関数です。 未定義を返すまで、この関数を呼び出します。 コードの最初のバージョンは、whileループを使用して記述されています。 



 function repeat(func) { while (func() !== undefined) { } }


2番目のバージョンは、末尾再帰を使用して記述されています。 



 function repeat(func) { if (func() !== undefined) { return repeat(func); } }


ES6では、これら2つの方法は同じ速度で動作し、まったく同じ量のメモリを消費します。 したがって、ループには再帰よりも利点はありません。 これは素晴らしい。



次世代の言語



次世代のプログラミング言語について多くのことを考えました。 彼はどのようになりますか？ JavaScriptが最後のプログラミング言語であることが判明すると非常に悲しいので、これはおそらくJavaScriptを置き換える言語になるはずです。 耐えられなかった。 少なくとも子供たちにとって、世界をより良い場所にしなければなりませんか？



この言語がどのようになるかを考えていました。 どのようなプロパティが必要ですか？ 彼はどのような問題を解決しますか？ 彼が現れたときに彼をどのように認識するのでしょうか？



私が確信していることの1つは、彼が最終的に現れたとき、我々は彼を拒否するでしょう。 そしてこの理由は、プログラマーが感情的で非合理的な異常な人だからです。



これはそうではないと考えています。 私たちはコンピューターの世界で人々の大使の役割を果たしており、コンピューターは完全に合理的だからです。 したがって、私たち自身も合理的であると仮定します。 私たちの多くは社会的スキルを失いましたが、不合理性と感情性は消えていません。 この論文を裏付ける証拠を見てみましょう。



  • 高レベルの言語を良いアイデアとして認識するには、1世代かかりました。 Fortranが多くのアセンブラー言語に登場したとき、開発者（当時はすべてアセンブラープログラマーでした）がFortranへの切り替えを拒否しました。 Fortranはマシンとの相互作用の制御を奪い取ったため、許可できませんでした。 移行は彼らの人生をより良くするだろうが、彼らはそれを放棄した。



  • 行くのは悪い考えだと同意するのに全世代がかかった。 何十年もの間、go toを使用するかどうかを感情的に議論してきました。 そして、非常に賢い人々からの深く考え抜かれた議論と、「これは私の表現方法です」、「効率が必要です」、「私は死にますが、私はそれをwillめません」というスタイルの文の両方が与えられました。 これらの議論はすべて完全に間違っていたことが判明しました。



  • オブジェクトが良いアイデアであることに同意するには世代がかかりました。 1967年、Simulaと呼ばれる言語がノルウェーに登場しました。 私が知る限り、この言語には重要なアイデアが含まれていると認めたのは、世界でたった1人、ユタ大学（アラバマ州）のAlana Kayだけでした。 他のみんなはそれを逃しました。 彼は、Simulaに登場するオブジェクトが非常に表現力があり、子供向けのプログラミング言語を開発できると考えました。 そして、子供たちはこのパラダイムを使用して素晴らしいプログラムを書くことができます。 そのため、彼はこの言語の勉強を始め、ほぼ10年かけて自分の言語を開発し、改善しました。 1980年にようやく公開されました。 そして、それは史上最高の開発されたプログラミング言語-C ++でした。 Alan KayはSimulaのアイデアを取り入れ、あまり理解していませんでしたが、Cに翻訳しました。



    そのため、業界には選択肢がありました。SimulaまたはC ++に従うことができました。 最終的な決定は、プログラミングのためにオブジェクトが何であるかを基本的に理解していない人々によって行われました。 しかし、彼らはC ++を選択しました。なぜなら、言語を使い始めるためにオブジェクト指向プログラミングを理解する必要がないからです。 それ以来、ほぼすべての言語で、SimulaよりもC ++から多くを借りてきました。 そして、私たちはまだ間違っています。



  • そして最後に、 ラムダが良いアイデアであることに同意するのに2世代かかりました。 ラムダは、70年代前半にMITでSqueakと呼ばれる言語で登場しました。 そして、業界はそれに全く注意を払いませんでした。 最終的にラムダを主流にするのに、2世代ではなく4世代かかりました。 時間がかかりすぎて、アイデアの失敗の証拠だと考える人もいました。 しかし、ラムダを使用した関数型プログラミングは、今日扱っている非同期システムや分散システムを操作するときに非常に効果的であることがわかりました。 このアイデアを受け入れる最初の言語はJavaScriptです。 その後、ラムダがPythonとRuby、そして最終的にはC＃に登場しました。 少し前、ついにJavaで。 しかし、JavaScriptが最初でした。



何もかもが長い間続いているのは、心を変えることができないからです。 私たちは、新しい良いアイデアの非常に多くのユーザーを獲得し、それを使い続けることができるようになる前に、世代が引退するか死ぬのを待たなければなりません。



に行ったときに覚えています。 彼の周りの議論は何年も衰えませんでした、そして突然それは静かになりました。 私たちは今彼を取り除くことができますか？ はい、私たちは彼を残しました。 誰もが行くのに苦しんでいますか？ すべての紛争は無意味であることが判明し、パラダイムシフトが起こりました。



人々がこのパラダイムを動かすのは本当に難しいです。 go toのストーリーは、1つの設計を取り除き、プログラムの構造を変えるだけです。 go toの欠如によりプログラミングが容易になることが判明しました。



だからこそ、私は次のプログラミング言語がそもそも拒否されると信じています。



言語分類について



プログラミング言語をどのように分類するかを考えてください。 言語をシステムとアプリケーションの2つの主要セットに分けます。



システム言語は、メモリアロケータ、システムコア、デバイスドライバを記述するために使用されます-これらは非常に低レベルの言語です。 それ以外はすべてアプリケーション言語で作成する必要があります。



おそらくJavaの最大の問題は、Javaの作成者がこの部門のどちら側になりたいかを決定できなかったことです。 そのため、Javaは両方のタスクをサドルしようとしています。



両方のカテゴリに新しい言語が必要です。 たとえば、60年代に登場したCは、今日でも主要なシステム言語です。 しかし、私の仕事は応用言語に焦点を当てています-私たちのほとんどが対処するのはそれらです。



応用言語は、2つのグループに分けることもできます。古典派（ほとんどすべての言語を含む）とプロトタイププログラミング用の言語グループ（JavaScriptを含む）です。



そして、JavaScriptは正しい方向を選んだと思います。 多くの批判を引き起こしているJavaScriptには革新が含まれています。



システムを設計する古典的な学校の言語でプログラムを作成する場合、オブジェクトの分類を行う必要があります。 システムのすべてのオブジェクトを分析して、それらが何で構成されているかを理解し、分類法を実行し、クラスが相互にどのように関連するか、どのように、何を実装するかを知る必要があります。 これはすべて非常に複雑です。なぜなら、プロジェクトの開始時に最も頻繁に行われるからです。 システムがどのように機能するかについての最低限の理解がある時点で。 したがって、常に間違った分類法を取得します。 そのため、誤って選択されたオブジェクトになってしまいます。 そして今からあなたは彼らと戦っています。 これだけでは、問題を正しく解決することはできません。 多重継承が必要ですが、これはすべて機能しません。 , , , , , , . . , , , . , ( ).
