映画ファンのためのデータの視覚化：映画の推奨事項を破棄し、インタラクティブなグラフを作成します



インタラクティブなlastfmマップに出会い 、映画用の同様のプロジェクトを間違いなく作成することにしました。 カットの下で、ストーリーは、データを収集し、グラフを作成し、例として映画検索とimdbからのデータを使用して独自のインタラクティブなデモを作成する方法についてです。 Scrapyスクラップブッキングフレームワークを見て、大きなグラフを視覚化する方法を検討し、ブラウザで大きなグラフをインタラクティブに表示するためのツールを扱います。







1.データ収集：スクレイピー



データソースとして、映画の検索を選択しました。 しかし、その後、これは非常に小さく、IMDbを削ったことが判明しました。 グラフを作成するには、映画ごとに、推奨される映画のリストを知る必要があります。 検索すると、十分なムービー検索パーサーとあらゆる種類の非公式のAPIを見つけることができますが、推奨事項を取得する方法はどこにもありません。 IMDbはデータセットをオープンに共有していますが、推奨事項はありません。 したがって、選択肢は1つだけです。スパイダーを作成します。







ハブのスクレイピングに関するいくつかの記事が既にあるので、可能なアプローチの概要はスキップします。 簡単に言うと、Pythonで記述していて、フレームワークを記述したくない場合は、 Scrapyを使用します 。 あなたが必要とするかもしれないほとんどすべてのものはすでに提供されています。







Scrapyは本当に非常に強力であると同時に非常にシンプルなツールです。 エントリーのしきい値は非常に低いですが、同時に、Scrapyはあらゆるサイズと複雑さのプロジェクトに簡単に拡張できます。 本当に必要なものがすべて含まれています。 ブロックのバイパス、スクレイピングの一時停止および再開などの方法を含む、受信した要素の直接解析およびHTTPリクエスト、処理および保存のためのツールからプロジェクトの管理まで。

1.1リクエスト数の制限をバイパスします。


Kinopoiskは、クモをデバッグする段階でさえ、5秒ごとに5リクエストのパックを1秒のタイムアウトで送信したときに禁止しました。 制限を回避するための多くのオプションがあります。 スクレイピーの場合、トーラスの使用、リストからのプロキシのランダムなソート、または有料の回転プロキシサービスへの接続の既製の例は簡単にグーグルで検索できます。 「週末プロジェクト」があるため、有料のサービスに接続する必要がありますが、実装するのに最も速いオプションである回転プロキシを選択しました。 仕組み：プロキシプロバイダーの特定のIP：ポートに接続し、出力で各リクエストの新しいIPを取得します。 スクレイピー側では、プロジェクトのsettings.pyファイルに1行を追加し、各リクエストでip：portペアのパラメーターを渡す必要があります。







コードでは、次のようになります。

settings.pyで適切なセクションを探し、そこに行を追加します： 







 DOWNLOADER_MIDDLEWARES = { 'scrapy.downloadermiddlewares.httpproxy.HttpProxyMiddleware':543 }


次に、クモのリクエストごとに： 







 scrapy.Request(url=url, callback=self.parse, meta={'proxy':'http://YOU_RPROXY_IP:PORT'})


私にインスピレーションを与えたプロジェクトの作者は、Nightwishから始めてから、lastfmの推奨事項をツリーと同じくらい広く回ってみたので、接続グラフを得ました。 私のアプローチも同様でした。 映画の検索からIDで映画を取得できます。これは、サイト上の映画のシリアル番号に単純に対応しているようです。 すべてのIDを取得するだけでは、ほとんどの映画には推奨事項がなく、マップ上のノイズに変わる単一ポイントになるため、うまく機能しません。 新鮮な映画のIDは500,000のオーダーです-これは視覚的なレンダリングには非常に多いので、上位250の映画のリストから始めて、各映画の推奨事項のリストを繰り返し見ていきましょう。







私は約100,000枚の映画を受け取ると予想していましたが、こすり落とす夜の終わりまでに、クモは約12,600で止まったことがわかりました。 映画検索の推奨事項はここで終わりです。 最初に述べたように、新しいデータを得るためにIMDbにアクセスしました。 IMDbのスクレイピングはさらに簡単です。 完成したプロジェクトを書き換えるのに数時間かかり、新しいスパイダーを開始する準備ができました。 2〜3日間のクロール（厚かましくならないように1秒間に8回のリクエスト）の後、クモは止まり、173 +千の映画を収集しました。 クモの完全なコードはgithubで見ることができます：フィルム検索とIMDb 。

2.1スタイリングの比較



大きなグラフの場合、gephiには2つの妥当なオプションがあります。OpenOrdとForceAtlas 2です。OpenOrdは非常に高速な近似アルゴリズムですが、構成可能なパラメーターはほとんどありません。 ForceAtlasは、他の古典的な力指向アルゴリズムに似ており、より正確な結果を提供し、チューニングに非常に柔軟性がありますが、時間をかけて支払う必要があります。 以下は、グリッドを表すグラフ上の両方のアルゴリズムの操作の例です。









モデルグラフグリッド。 左OpenOrd、右ForceAtlas。





より正確な結果を待つ時間がある場合は、OpenOrdをまったく使用すべきではないと考えるかもしれません。 実際、ForceAtlasがすべてのノードを1つのタイトな塊に収集し、OpenOrdが少なくとも何らかの構造を示す場合、グラフは珍しくありません。







プロセスを高速化するために、OpenOrdを初期近似として使用し、ForceAtlasでグラフを塗りつけました。 画像が少なくとも何かはっきりしているためには、お互いのノードのオーバーラップを排除する必要があります。 このためには、Yifan Huスタッキングを使用すると便利です-クラスターを少し汚すため、およびnoverlapを使用してオーバーラップを完全に除去します。 映画の検索グラフの重複を解消するのに一晩かかったため、週末全体でimdbに対処できませんでした。

5.1 DBSCAN



グラフをクラスタリングするには多くの特別な方法があり、それらはすべて別の記事に値します。 実験として、グラフ用ではない方法を使用しました。 理由は次のとおりです。グラフが視覚的に類似のフィルムの雲に分解されると、 DBSCANを使用してフィルムが特に近接している領域を見つけることができます。 深く掘り下げることなく、このメソッドが何をするのか見てみましょう。 DBSCANという名前は密度ベースのスキャンを表します。つまり、この方法を使用して、かなり近くにあるポイントをマージします。 これは、2つの主要なハイパーパラメーターを介して形式化されます。これは、各ポイントの近傍を探す半径と、近傍の最小数です。
