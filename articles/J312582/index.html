セルゲイウラジミロフ（事業開発を支援するプラットフォームの中央委員会の開発責任者）



-Sbertechで何に取り組んでいますか？



-約1年間、小売店のカード受付端末の新しい会計システムを開発していました。 このプロジェクトは、Cで書かれた古いシステムを置き換え、データベースへの直接接続を操作し、他のシステムとの手動および半手動のやり取りを多数行います。 このアプリケーションは、Sberbank 2018+の新しいプラットフォーム（2018年以降のターゲットプラットフォーム）の一部になります。 クライアント指向のデータストレージアーキテクチャ（Apache Ignite / GridGainテクノロジに基づく）と単一の企業データモデルを使用して、「正直な」サーバー側Javaアプリケーション（クライアントとしてブラウザを使用）として記述されています。 ここには多くの流行語がありますが、それぞれの背後には根本的な変化があり、考え方やビジネスアプリケーションの開発方法に変化はありません。



このプロジェクトの第1フェーズは完了し、同僚は引き続き作業を進め、ユーザーインターフェイスを整理するための新しいプロジェクトに携わり始めています。 繰り返しになりますが、これはHTMLからJavaScriptへのコードの単なる書き換えではなく、フロントエンドとバックエンドの相互作用への新しいアプローチの導入です。

-あなたはオープンソースが好きですか-Sbertekhで作業するとき、どのくらい積極的にそれを使用しますか？



-オープンソースは好きではないかもしれませんが、使用しないのは非常に困難です:-) Javaプログラマーであり、ビジネス向けのソリューションを開発している場合は、ロガーからXMLライブラリまで、クラスパスに12個以上のオープンソースライブラリがあることを意味します。 JSON Sberbank Technologiesでは、オープンソースを使用するだけでなく、コミュニティの積極的な参加を目指しています。 最も顕著な例は、もちろん、Apache Igniteの作業に参加していることです。



オープンソースはただ安いだけではありません。 「市場投入までの時間」の点でより速いことが判明しました。 クローズドソースコード付きの有料製品を使用すると、開発者から必要な機能の修正または改善を2〜3年待つことができます。 または、オープンソースソリューションを使用して、その中のいくつかの欠陥を修正し（それらが存在し、誰も気付いていない場合）、何かを追加または変更する必要がある場合は、それを取得して変更します。 従業員にとって少し珍しい場合でも、資格が必要になりますが、大まかに言えば、クローズド製品の新しいバージョンがリリースされたとき、2〜3年後ではなく、生産で1週間後に機能します。 Sberbank Technologiesでは、これは理論ではありませんが、既存の技術を修正するには費用がかかりすぎたり、通常はより重要であるために技術やソリューションの変更を余儀なくされる実際のケースです。

マキシム・ゼリンスキー（前頭システム開発中央委員会の部長）





-正確に何をしていますか？



-私は、Unified Front-End System（EFS）の作成に参加しています。つまり、EFS内にビジネス機能を直接実装する開発者向けのプラットフォームの作成と開発を担当しています。



将来的には、ユニファイドフロントシステムは、利用可能なすべてのチャネル（インターネットバンク、モバイルアプリケーション、ATM、コールセンター、支店など）を通じて銀行のすべての顧客に絶対にサービスを提供します。 そのタスクは、ソリューションを統合して再利用することです。 もちろん、異なるチャネルには独自の特性がありますが、一般的にプログラムの本質は何かの実装に一度投資し、この実装を異なるチャネルで同様にすることです。



ESFの巨大な規模を考えると、高品質で信頼性の高いソリューションを作成するには、新しいソリューションの市場投入までの時間を短縮し、信頼性とパフォーマンスの面で均一なレベルのサービスを保証するための単一セットのテクノロジーとサービスが必要です。 これらの問題（単一のスタック、サービス、アーキテクチャ）は、ESFプラットフォームのフレームワーク内で私のチームによって対処されています。



-分散システムなどに特化しています-バックエンドではなく、フロントエンドシステムを扱うのはなぜですか？



-私たちの場合、パフォーマンス、フォールトトレランス、および信頼性の面でのフロントエンドは、バッキングよりもさらに興味深いものです。「稼働時間99.99％」などの要件があります。 また、バックエンドが何らかの時点で機能しない場合、クライアントはそれに気付かないでしょう。 彼の支払いはすぐには実行されませんが、システムに入力することは可能です。キャッシュから抽出物を取得することは可能です。



しかし、バックエンドによると、Sberbank Technologiesには別の大きなプログラムがあります。それは、ビジネス開発サポートプログラムです。 これは、In-Memory Data Gridを使用して技術的ソリューションを変換する試みです。 最初は、OracleまたはPostgreSQLに基づいてデータの自動シャーディングを行う試みがありました。 パイロットがいました、彼は肯定的な結果と否定的な結果の両方を与えました。 パイロットの結果によると、「現在の技術は銀行の規模に完全に関連しているわけではありません。将来を見据えましょう」と判断されました。 彼らはIMDGを選択し、同僚はバックエンドのデータ配列をメモリに移動しています。



-あなたのような巨大な規模で、1つの技術的解決策がリソースの非常に大きな増加につながる状況はありますか？



-顕著な例は、JS SPAおよびReactのWebプレゼンテーションレベルのテクノロジーの選択です。 アプリケーションサーバーからクライアントにプレゼンテーションロジックを転送するだけで、従来のサーバー側のWebプレゼンテーションレベルで使用する予定だったアプリケーションサーバーを200個まで保存できました。 同じことがクライアントとサーバー間のトラフィックの削減にも当てはまります。これは、交換がデータではなくHTMLであるという事実により明らかに削減されます。



-リバースエンジニアリングの経験がありますが、Sbertekhで作業する際に実際に役立ちますか？



-とてもそうです。 特に、IBM WebSphere Application Serverを使用する場合、ESFの前に作業していたチームが、負荷のあるアノテーションを使用してJVM内でデッドロックを引き起こし、Javaで終わる重大な欠陥を発見したIBM WebSphere Application Serverで作業する場合、特にリバースエンジニアリングに対処する必要がありますIBM WebSphere MQのドライバー、長い間問題を研究しなければならなかったとき、なぜIBM WAS + IBM MQ + Spring JMSテンプレートの組み合わせが負荷のかかった非永続キューで機能しないのか。
