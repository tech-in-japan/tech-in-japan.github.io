情報システムのカスタマイズについて





当社の主な活動は、企業情報システムの開発です。 カスタムメイドのシステムに加えて、2つの複製製品を作成しています。 もちろん、可能な限り便利で機能的な製品を作ろうとしています。 ただし、実際には、各ビジネスには独自の特性があり、常にシステムの標準機能に対応する準備ができているわけではありません。 特定のクライアント向けのソリューションを完成させるタスクと、そのさらなるサポートがあります。 拡張可能な製品のアーキテクチャを整理するためのアプローチは何ですか？ 開発中にどのような問題が発生する可能性がありますか？ 私たちは何をしましたか？ 以下のすべてについて。



可能なアプローチ



まず、「拡張プロジェクト」または単に「拡張」によって、特定の顧客向けに変更を加えた製品を呼び出すことを明確にします。 次に、製品を拡張するための可能なアプローチのいくつかを見てみましょう。

拡張プロジェクトごとにリポジトリでブランチを分けます
おそらく、これが最初に思い浮かぶのは、主な製品から分岐して新しいブランチに変更を加えることが望ましい結果を得るための最速の方法だからです。 唯一の質問は、この速度に対して支払う必要がある価格です。



情報システムの開発と実装の後、ライフサイクルの最も長く、多くの場合最も痛い段階が始まります-サポート。 拡張プロジェクトの場合、このフェーズは二重に不愉快になる可能性があります。これは、顧客に特別に実装された新しい「機能」だけでなく、拡張のベースとなる製品の新しいバージョンも顧客に提供する必要があるためです。 製品の新しいバージョンからの変更をプロジェクトに取り込むには、1つの方法があります。メインブランチから拡張ブランチに変更をマージします。 しかし、それがどれほど時間がかかり、両方のブランチでコードの同じセクションが大幅に変更された場合に発生する可能性のあるエラーの数を想像してください。



もちろん、製品の新しいバージョンへの将来の翻訳についてすぐに考え、すべての特定の変更がメイン製品のコードから可能な限り離れるようにコードを編成することができます。 理想的な世界ではこれで問題ありませんが、あなたと私は厳しい現実に住んでいます。タスクの締め切りは「昨日」として発表されることが多く、プロジェクトはクールな専門家のコンパクトなチームではなく、昨日の学生の大隊です。 そのような状況では、人々はめったにアーキテクチャについて考えず、最も抵抗の少ない道をたどります-修正する場所を見つけ、古いものを削除し、新しいものを書きました。 ちなみに、これは別の大きな問題につながります-拡張ロジックは製品ロジックと混在しています。



結論：このアプローチは、製品のあらゆる部分を絶対に変更できるため、最も柔軟性がありますが、最初の2、3のアップデートのみが柔軟性を喜ぶ必要があります。 その後、拡張機能のサポートと製品の新しいバージョンへの移行により、大きな困難が生じます。 さらに、情報システムの寿命が長くなればなるほど、サポートにかかる時間はますます長くなります。


動的属性の使用（Entity-Attribute-Valueモデル）
Entity-Attribute-Value（またはOpen Schema）モデルを標準のリレーショナルモデルと組み合わせて使用​​して、新しいエンティティ属性の値を動的に識別して保存できます。 EAVモデルを使用する場合、属性値は通常3列の単一のテーブルに書き込まれます。 ご想像のとおり、その名前はエンティティ、属性、値です。

  • エンティティ-フィールドを記述しているオブジェクトへの参照を保存します。 通常、これはエンティティ識別子です。
  • 属性-属性定義へのリンク（詳細は以下）。
  • 値-属性値自体。


スキーマの必須コンポーネントは、属性のメタデータの説明を格納するテーブルでもあります。

  • 属性タイプ
  • 制限（フィールド長、値が一致する必要がある正規表現など）;
  • UIに表示するコンポーネント。
  • コンポーネントがUIに表示される順序。


製品でこのモデルを使用するには、2つのことを行う必要があります。

  1. メタデータを指定するためのメカニズムを実装します。たとえば、用語属性「終了日」が「契約」タイプのエンティティに追加され、フィールドタイプが「日付」になり、表示コンポーネントがDateFieldになるように指定できます。
  2. 必要な製品画面に動的属性の値を表示および入力するメカニズムを実装します。 メカニズムは、メタデータの説明を含むテーブルでこのエンティティの可能な属性のセットを見つけ、編集用のコンポーネントを表示し、データを含むテーブルで値を検索して表示し、画面を閉じたときにそれらを保存する必要があります。


このアプローチの最も重要な利点は、拡張プロジェクトを作成する必要がないことです。 基本製品は顧客に提供され、セットアップまたは運用の段階で、エンティティの任意の数の動的属性がセットアップされます。



さらに欠点について。 まず、限られたアプリケーションです。 EAVモデルでは、エンティティに属性を追加し、それらを画面上の事前定義された場所に表示することしかできません。 それ以上。 機能の変更やトリッキーなUIコンポーネントの話ではありません。



第二に、EAVモデルはデータベースサーバーに大きな追加の負荷を作成します。 リンクなしでエンティティの1つのインスタンスを読み込むには、テーブルの1つまたは複数の行の代わりに読み取る必要があります。 たとえば、UIのテーブルにインスタンスのリストを読み込むには、通常、N + 1個のクエリ、またはテーブル列の数に応じた結合が必要です。 多くの場合、企業システムのデータベースは最も低速で拡張性の低い要素であるため、このような追加の負荷はシステムを単純に停止させる可能性があります。



第三に、データベースの構造により、レポート用のデータサンプルを作成することは非常に困難です。通常のSQLを記述する代わりに、リレーショナルデータはより複雑なクエリを必要とします。


プラグインのアーキテクチャ
このアーキテクチャにより、追加の機能を個別のアーティファクト（プラグイン）に保存できます。 顧客がいくつかの新しい詳細を望んでいるなら、あなたは彼に基本製品を置き、プラグインを書き、それを接続し、それで完了です。 製品でプラグインを使用するには、拡張ポイントを宣言する必要があります。 これは何ですか 単純な場合、これらはコード内の特定の場所です。 これらの場所では、ロードされたプラグインがソートされ、特定の拡張ポイント用に設計されたプラグインにロジックがあるかどうかが分析され、そのようなロジックが見つかった場合は実行されます。 拡張ポイントの例：メニュー項目、コマンドハンドラ、ツールバーのボタン、新しい画面。



外部スクリプトでのロジックやイベントハンドラの記述など、機能を拡張するために頻繁に使用されるこのようなオプションは、次のようなプラグインのバリエーションに起因する場合もあります。 スクリプトは、プログラムの特定のポイントによって呼び出され、実行されます。



プラグインアーキテクチャを使用すると、すべての新しい機能を製品とは別に保存できます。これは重要な利点です。 製品とプラグインを完全に物理的に分離することにより、製品またはプラグインのバージョンを更新するプロセスが非常に簡単になります-更新されたコンポーネントを置き換えるだけです。



しかし、残念ながら、ここには欠点があります。 一部の製品では、重要ではないことが判明します。一部の製品では、プラグインアーキテクチャを使用できなくなることがあります。 実際、プラグインは、拡張ポイントが定義されている場所でのみシステムを拡張できます。 間違いなく、原則としてそのような場所はほとんどなく、それらはすべて事前に決定されている製品のクラスがありますが、明日拡張する必要があるものを予測することはほとんど不可能な巨大なグループもあります。 潜在的な拡張可能な場所の分析は非常にリソース集約型のタスクになる可能性があり、その結果、完全に正確ではないことが判明する場合があります。 さらに、拡張ポイントはメインコードの複雑さであり、常にエラーとメンテナンスの複雑さが伴います。 メイン製品の拡張ポイントの定義にアプローチすることは、非常に思慮深いものでなければなりません。

新しい拡張機能の作業を開始するとき、私たちはこのプロジェクトがどの「モンスター」（言葉の意味では）に成長できるかを推測することさえしませんでした。 よくあるのは、小さなカスタマイズとして始まったものが、ほぼ完全に書き直されたビジネスプロセスと、画面の半分の追加ロジックで終わることです。 さらに、この製品は新しい機能で拡張することができます。これは独立したシステムには十分です。 例として、大規模な分散企業向けのTESISの拡張プロジェクトでは、財務の自動化、従業員の有効性の評価、およびいくつかのより複雑なモジュールが登場しました。



さまざまな要件、その量、および予測不能性により、上記のいずれの方法も使用できませんでした。 その上、製品バージョンはかなり定期的に公開されます。 これにより、拡張プロジェクトを製品の新しいバージョンに簡単に転送できることが要求されます。



拡張機能の作成とサポートの問題をどのように解決しますか？



拡張プロジェクト



当社のプロジェクトのほとんどは、CUBAプラットフォームを使用して作成されています。 以前の記事の 1つですでに書いています。 拡張プロジェクトの編成方法を理解するには、まずプラットフォーム自体のデバイスを扱う必要があります。



つまり、CUBAは一連のモジュールであり、それぞれが特定の機能を提供します。

  • キューバ-アプリケーションのコア。インフラストラクチャ全体、ビジネスロジックを整理するためのツール、視覚コンポーネントのライブラリ、セキュリティサブシステムなどが含まれます。
  • レポート-レポート生成サブシステム
  • fts-全文検索サブシステム
  • チャート-チャート出力サブシステム
  • ワークフロー-ビジネスプロセス管理サブシステム
  • ccpayments-クレジットカードサブシステム


各サブシステムには、ビジネスロジックを備えた永続的なエンティティ、画面、およびサービスを含めることができます。



新しいプロジェクトを作成するとき、機能が必要なプラットフォームのベースモジュールへの依存関係は、ビルドスクリプトに記述されます。 その後、接続されたモジュールのエンティティ、画面、およびサービスを使用して、プロジェクトの実装を開始します。 物理的には、プラットフォームモジュールはjarファイルであり、サーバーにインストールされるプロジェクトは通常のJava Webアプリケーションです。



プラットフォーム上で既存の製品を拡張する必要がある場合、これを行います。新しいプロジェクトを作成しますが、ビルドスクリプトでは、プラットフォームモジュールではなく拡張可能な製品への依存を示します。 製品自体は、実際には開発プラットフォームとして機能します。





これで、拡張プロジェクト内で、新しいドメインモデルオブジェクトを作成し、CUBAプラットフォーム上の最も一般的なプロジェクトのように新しいユーザーインターフェイスを記述できます。 開発者は、下位モジュールのすべての機能を引き続き利用できます。



最も明らかな利点は、すべての新しいロジックが個別の拡張プロジェクトに保存されることです。 現在の拡張機能の一部として、いつ何を書いたかを常に確認できます。 このアプローチは、プラグインアーキテクチャとEAVのように、開発者を新しい機能の種類に制限するものではありません。



拡張機能を製品の新しいバージョンに変換するプロセスは次のとおりです。
  • ビルドスクリプトで製品の新しいバージョンを指定し、拡張機能を再構築します。
  • 拡張機能が安定した製品APIのみを使用している場合、それだけです-高度な製品を起動してください。
  • 展開したポイントで製品が大幅に変更された場合、拡張コードを変更する必要がある場合があります。 原則として、これはまれにしか発生せず、問題を特定するのは簡単です。 バグ修正製品リリースは常に拡張機能と互換性があります。


拡張機能では、新しいエンティティ、ビジネスロジック、およびそれらの画面を簡単に作成できることは明らかです。 しかし、すでに製品にあるものを変更するにはどうすればよいですか？ たとえば、製品エンティティにフィールドを追加して、既存の画面に表示しますか？



ベースプロダクトエンティティに新しい属性を追加する



タスクを自分用に定義します。ベース製品のUser



エンティティで、住所を保存するフィールドを追加する必要があります。 このような要件は、おそらくお客様の間で最も一般的です。 プラットフォームが上記の動的属性モデルをサポートしていることをすぐに言う必要がありますが、実際にはこのオプションはほとんど使用されません-データサンプリング速度とレポートの容易さは、ほとんどの場合重要な要件です。

実際には、属性を追加する代替方法について。 ORMプラットフォームとして、OpenJPAが使用されます。 製品のエンティティ宣言は次のとおりです。 



 @Entity(name = "product$User") @Table(name = "PRODUCT_USER") public class User extends StandardEntity { @Column(name = "LOGIN") protected String login; @Column(name = "PASSWORD") protected String password; //getters and setters }


ご覧のとおり、これはエンティティの標準的な説明であり、JPAのテーブルおよびデータベース列へのマッピングです。



拡張プロジェクトにエンティティー継承を作成します。 



 @Entity(name = "ext$User") @Extends(User.class) public class ExtUser extends User { @Column(name = "ADDRESS", length = 100) private String address; public String getAddress() { return address; } public void setAddress(String address) { this.address = address; } }


継承メカニズムは、最も重要な@Extends



注釈を除き、OpenJPAの標準です。 ExtUser



クラスがUserクラスの代わりに普遍的に使用されることを宣言するのは彼女です。



これで、 User



エンティティを作成するすべての操作で、拡張エンティティのインスタンスが作成されます。 



 User user = metadata.create(User.class); //  ExtUser


データベースからのデータ抽出操作も、新しいエンティティのインスタンスを返します。 たとえば、ベース製品では、名前によるユーザー検索サービスが返され、次のJPQLクエリの結果が返されます。 



 select u from product$User u where u.name = :name

ビジネスロジックのオーバーライド



機能の再定義に関しては、ここではすべてが非常に簡単です。 プラットフォームインフラストラクチャはSpringに実装されています。 したがって、ビジネスロジックを持つサービスレイヤーは、スプリング駆動型Beanです。 フレームワークによって提供される拡張機能は、必要なビジネスロジックを再定義するのに十分以上です。 これを明確に示すために、再び小さな例を示します。 基本製品の中間層に、価格計算を実行するコンポーネントがあると仮定します。 



 @ManagedBean("product_PriceCalculator") public class PriceCalculator { public void BigDecimal calculatePrice() { //price calculation } }


拡張プロジェクトの価格設定アルゴリズムを置き換えるために、2つの簡単な手順を実行します。



再定義されたコンポーネントの継承を作成します。 



 public class ExtPriceCalculator extends PriceCalcuator { @Override public void BigDecimal calculatePrice() { //modified logic goes here } }


基本製品のbin識別子を使用して、Spring構成ファイルにクラスを登録します。 



 <bean id="product_PriceCalculator" class="com.sample.extension.core.ExtPriceCalculator"/>


これで、Springコンテナは常にExtPriceCalculatorのインスタンスを返します。
