性能



そもそも、今日（安定ブランチが8.2、現在のバージョンは8.2.5）PostgreSQLは、オープンソースの選択肢だけでなく、主要な商用DBMSとのパフォーマンスの面でも競合しています。 Oracleなど。 これはもはや空のフレーズではありません。サンが実施したテストの結果を見てください。 もう遅い象はいません！ 豊富な種類のインデックス、システムのチューニング、非常に大きなボリュームと負荷の操作、複製およびスケーリングシステムの適切な選択の幅広い可能性-これらはすべて、現代の象にとって「厳しい」ものです。 開発速度でさえ、他のDBMSと比較してPostgresを有利に区別します。毎年、私たちは常に重要な一歩を踏み出します。



PostgreSQLバージョン8.3のパフォーマンスの新機能は何ですか？ 多くの変更は重要です。 PostgreSQL開発コーディネーターのBruce Momjianによると、Postgresmanの招待で少し前にモスクワを訪れた人によると、Postgresの開発者が近年忙しくしているシステムパフォーマンスの最適化作業は非常に困難です。 各ステップには、ますます多くの人件費が必要であり、開発者の時間と労力がますます必要になります。



これらの本当に重要な変更の1つは、 HOT（ヒープのみのタプル）と呼ばれる「チップ」と安全に考えることができます。 これはおそらくパフォーマンスの面で最も重要な変更の1つです。 この変更の内容を理解するには、PostgreSQLがいわゆるマルチバージョンアクセス制御モデル（ MVCC 、マルチバージョン同時実行制御）を実装していることを覚えておく必要があります。







HOTの本質は次のとおりです。 以前、このアプローチを実装する前に、テーブルの行を更新するとき、インデックス付き列がこれらの変更に影響を与えたかどうかに関係なく、テーブルの行を更新するたびにすべてのインデックスの新しいバージョンが表示されました（図「HOTなしの更新」を参照）。 ここで、新しいバージョンの行が前のバージョンと同じメモリページに分類され、インデックスが作成された列が変更されなかった場合、インデックスは同じままです。 しかし、それだけではありません。 そのような可能性がある場合、ヒープページ内のスペースの「瞬間的な」再利用が発生します。 当然、VACUUM操作中に実行される作業量が減少します。 図 「HOT update」は、行がどのように更新されるかを概略的に示します。



次の新規性は、まず第一に、多数のWeb開発者にアピールします。 バージョン8.3以降、PostgreSQLのトランザクションは非同期にすることができます 。



つまり、トランザクションコミット操作（COMMIT）が実行されると、PostgreSQLサーバーは、高価なトランザクションログ同期（WAL fsync）操作が完了するまで待機しません。 つまり、すべての論理条件が満たされるとすぐに、トランザクションは正常に完了したと見なされます（必要な整合性制約がすべてチェックされます）。 物理的に、トランザクションログは非常に短い時間で書き込まれます（通常、正常に機能するシステムでは、これは最大200〜1000ミリ秒です）。環境変数synchronous_commitは、トランザクションの状態（同期/非同期）を担当します。 非同期モードへの切り替えは簡単です。

SET synchronous_commit TO OFF;





非同期トランザクションは、fsync操作が無効になっているサーバー操作モードの代替ではないことに注意してください。 実際には、fsync = offモードは、ベースの一貫性のない状態につながる可能性があり（たとえば、予期しない機器の故障や電力損失の場合）、信頼性の高い機器が使用されている場合にのみ推奨されます（たとえば、バッテリー付きのディスクドライブコントローラー）。 新しい機能を使用しても、データの不整合は発生しません。 可能な最大値は、データのごく一部の損失です（これも、深刻なサーバー障害（OS、ハードウェア、電源障害）の場合）。 非同期トランザクションの典型的な例は、いくつかの行の損失が重大でない場合に、テーブルログ（たとえば、ユーザーアクションのログ）に大量の情報を格納するタスクです。 ただし、すべての重要なトランザクションは引き続き同期できます。



別のパフォーマンスの向上は、PostgreSQLがクエリ（SeqScan操作）を実行するときにテーブルを順次スキャンする状況に関連しています。 バージョン8.3より前の場合、このような場合、異なるPostgresプロセスが同じジョブを同時に実行したという状況がしばしば発生しました- 同期スキャン（「同期ビュー」）の実装のおかげで、同じテーブルを同時に見るようになりました1つのテーブルの時間は1回のスキャンでしか実行できません。 これは次のように達成されます。 セッション内で、SeqScanがすでに実行されているテーブル（別のセッション用）にSeqScanが必要な場合、既に実行されているSeqScanの結果への「移動中のジャンプ」があります。 このプロセスが完了すると、必要に応じて、別の不完全なSeqScanを使用して結果が「補足」されます（図を参照）。



チェックポイントプロセスシステム（「コントロールポイント」）によって生じるストレスの影響を軽減する作業は継続しています。チェックポイントはすぐに実行されるのではなく、徐々に実行されます。プロセスは時間内に「広がり」ます。 したがって、この変更の名前はチェックポイント平滑化です。 サーバーを定期的にシャットダウンし、「明示的な」チェックポイント操作（ CHECKPOINTコマンド）を実行している間、データは可能な限り高速にディスクに書き込まれることに注意してください。



パフォーマンスの説明を終了するために、PostgreSQLを使用するシステムのパフォーマンスを改善するために設計された他の変更点の短いリストを以下に示します。

  • autovacuumはデフォルトで有効になりました 。
  • 状況によっては、 複数の自動バキュームプロセスを一度に開始できるようになりました （たとえば、大きなテーブルをクリーニングする長いプロセス中に、小さなテーブルがこのプロセスの完了を待つ必要がなくなりました）。
  • データベースが占有するディスク容量の顕著な減少 ：1）以前は厳密に4バイトを占めていたvarlena型ヘッダー（可変長のデータ型：テキスト、配列など）により、状況によっては1バイトのみ。 2）テーブル行ヘッダーの保存（以前は27バイト、現在は24バイト）。
  • データを変更しないトランザクションの実行は、トランザクションカウンター（xid）の値を増加させません 。これにより、トランザクションカウンターの重複（xidラップアラウンド）の可能性が大幅に減少します。 さらに、この変更は、トランザクションログの転送に基づく組み込みのマスタースレーブレプリケーションの実装に向けた重要なステップです。現在、マスターノードとスレーブノードの間でxidカウンターの非同期化はありません。
  • bgwriterプロセスのパラメータを自動調整するメカニズムが実装されました （バックグラウンドライタは、ダーティなダーティバッファをディスクに書き込む特別なプロセスです）。
  • " …ORDER BY … LIMIT…



    "（いわゆるTop-Nソート ）を使用してクエリの結果を取得するメカニズムを最適化しました：場合によっては、システムは必要なデータページをRAMに保持し、結果の出力を非常に高速にします。
  • これで、（これまで2つの定数の形式でのみ）関数を実行する推定コストと返される行数を指定できるようになりました。これにより、PostgreSQLプランナーは最適なクエリプランを選択できます（例： ALTER FUNCTION log_data(text) COST 100000 ROWS 1
