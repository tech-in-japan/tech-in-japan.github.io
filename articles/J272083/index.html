NetAppはインライン重複排除によりさらに効率的です

数日前、Data ONTAP 8.3.2RC1のバージョンが利用可能になりました。 「RC」はリリース候補版を意味するため、NetAppバージョン命名規則に従って、このリリースはすべての内部テストに既に合格しており、顧客は新機能を評価するだけでなく、ビジネスを実行しているシステムを含む製品に対しても使用できます-重要なタスク。 ベンダーサポートは、Data ONTAPの「RC」バージョンで実行されているシステムに完全に拡張されます。



NetAppの売り手（はい、これも私たちです）は、最適化ソフトウェア（重複排除と圧縮）を使用することで、NetAppシステムでのディスクリソースの高い使用率が達成されることを思い出します（そして絶対に正しく行います）。







以下の情報は、NetApp Data ONTAP 8.3.2RC1の機能の簡単な概要であり、「まだバイソンではない」場合に役立ちます。



重複排除は非常に長い間お客様にご利用いただいており、当初から「オフライン」モードでのみ機能しています-スケジュールに従って、重複データブロックを検索し、それらの1つだけをディスクに残すプロセスが開始されます。 システムは同時に、ブロックからのハッシュ合計を比較するだけでなく、データ自体も比較するため、ハッシュの衝突がないことが保証されます。



これにより、パフォーマンスの問題が回避されます-重複排除を有効にしても、プロセッサ/ストレージメモリの負荷が変化することはありません。 ただし、多くのタスクでは、このような重複排除メカニズムは最適ではありません。 多くの仮想ワークステーション（VDI）のファームがあり、すべてのデスクトップでソフトウェアを更新するか、OSのパッチをインストールする必要があると想像してください。 この時点で、多くの同一のデータブロックがディスクに書き込まれます。 はい、もちろん、重複排除プロセスが実行され、「余分な」ディスク領域が解放されますが、更新自体によって膨大な数の書き込み操作が発生します。 これは、過負荷のバックエンドを取得することを意味し、その結果、このストレージで動作する他のサービスのパフォーマンスに影響を与える可能性があります。



cDOT 8.3.2の登場により、NetAppストレージの所有者には、インライン重複排除（オンザフライ重複排除）というソリューションがあります。 オールフラッシュストレージでは、新しく作成されたすべてのボリュームでインラインデデュープサポートが「すぐに使用可能」に含まれており、データを含む既存のボリュームでも有効にできます（ボリュームを再作成せずに）。 Flash Poolを使用するストレージシステムの場合、オンライン重複排除はSSDへの書き込み操作に対してのみ機能し、新規および作成済みのボリュームの両方で有効にすることもできます。 通常のディスク上に構築されたシステムの場合、新しいモードを強制的に有効にする必要があります。 コマンドでインライン重複排除の動作を制御できます



-inline-deduplication（true / false）オプションによるボリューム効率：



ボリューム効率の変更-vserver SVM_test -volume / vol / volume-001 -inline-deduplication true



以下は、AFFおよびその他のNetApp FASシステムでサポートされる最適化モードの表です。





Dtata ONTAPは、書き込み可能なブロックのハッシュを保存するために、合計RAMの約1％を割り当てます。 高いパフォーマンスを維持するために、開発者はグローバルな重複排除を放棄する必要がありました-実装するには大量のメモリとプロセッサリソースが必要になります。 最近記録されたブロックのハッシュのみがメモリに保存されます-再起動すると、ストレージがクリアされ、統計が再び蓄積されます。



別の重要な機能-ボリュームの移行（データモーション）は、後処理が機能するまで重複排除のすべての利点を消滅させます。 これは、システム内で重複排除されたデータの転送を計画するときに考慮する必要があります。



インライン重複排除の効率を向上させるために、「クラシック」後処理スキームでインライン重複排除を使用することができます。 オールフラッシュストレージの場合、後処理は最初に無効になり（書き込み操作の合計数を最小化するため）、強制的に有効にする必要があります。



インライン重複排除を使用しても、NetAppストレージシステムのデータストレージの効率を向上させる他の機会が失われることはありません。 オンライン圧縮と重複排除を同時に含めると、手順は次のようになります。

  • インラインのゼロブロック再複製-空の（ゼロで埋められた）ブロックは除外されます
  • インライン圧縮-データは圧縮されます
  • インライン重複排除-圧縮されたブロックのハッシュIDがチェックされます。 ハッシュが一致する場合、ハッシュが一致するときに全体が比較され、重複ブロックはディスクへの書き込みから除外されます




グローバル重複排除が拒否されたため、Data ONTAP開発者は、書き込み操作を最適化するためのかなり興味深いメカニズムを実装できました。これは、多くのタイプのワークロードでうまく機能し、システムパフォーマンスへの影響は最小限です。 また、バックエンドの負荷を軽減することにより、場合によってはシステムの統合パフォーマンスの向上について話すことができます。 NetAppシステム（cDOT 8.3をサポート）を使用しているユーザーは、バージョン8.3.2RC1にアップグレードすることにより、インライン重複排除の有効性を確認できます。 もちろん、テストシステム上で、またはパイロットの一部として実験を行うことをお勧めします。新年の前に動作中の製品を再構成することはお勧めしません。



私たちの仕事では、後処理の重複排除中にシステムのパフォーマンスがメインの負荷に対して十分でなくなるまで、さまざまな状況に対処する必要がありました。 これらはすべて、プロジェクト開発段階で評価でき、データストレージシステムを選択するときに必要な電力を確保する必要があります。 トリニティのスペシャリストは、さまざまな顧客の要件に合わせて設計前の調査とストレージシステムのサイジングを行うための広範な知識と豊富な経験を持っています。
