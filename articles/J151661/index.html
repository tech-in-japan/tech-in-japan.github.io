XenハイパーバイザーからビデオカードをゲストOSに転送する

背景



この記事を読んだ後、私のガールフレンドは、2番目のビデオカードをシステムユニットに挿入し、同じハードウェアで2台のマシンを取得することを提案しました。 私たちは試行錯誤の長い共同の道を歩み、興味深い問題に直面しました。その解決策について説明したいと思います。



ハードウェア



  • Gigabyte GA-Q67M-D2H-B3マザーボード
  • BIOSファームウェア-F5
  • セットアップ：CPU-> Intel Virtualization Technology = ON
  • セットアップ：チップセット->ノースブリッジ-> VT-d = ON
  • セットアップ：Sataモード= ACHI
  • Intel Core i5 2500 3.3 GHzプロセッサー
  • メモリ16 GB
  • SSD：OCZ Vertex 3 120GB
  • Intel GMAビルトインマット。 ハイパーバイザーが使用するボード
  • ゲストOS 1に提供される最初のPCIeスロットのATI Radeon HD7770
  • 2番目のPCIeスロットのATI Radeon HD7770、ゲストOS 2に提供
  • マザーボードに統合されたIntelネットワークアダプター
  • XenLANをインストールしてサーバーを管理するための、Windowsと同じLAN上の2台目のPC


開始する



当初、XPCハイパーバイザーに基づくXPC 1.5ベータ仮想化プラットフォーム（Xen Cloud Platform）が実験用に選択されました。 このプラットフォームは完全に無料です。さらに、インストールと試運転には数分かかります。 作業の過程で、XCPの仮想マシンでSkypeを起動することは不可能であることが判明し、すぐにエラーで終了しました。 問題はこことここで説明されています 。 修正するには、ソースに1行追加し、XPCのベースとなっているXenを再構築する必要があります。 そのため、この問題がすでに修正されているCitrix Xen Server 6.0.2の無料版に切り替えました。



ビデオカードドライバーをインストールする場合、別の危険があります。 実際、Catalyst Control CenterはゲストシステムでBSODを呼び出します（XCPとXen Serverの両方でテスト済み）。 そのため、タスクマネージャーを使用してドライバーをインストールする必要があります。ネイティブインストーラーは、ビデオカードをチェックするときにBSODを発生させることもあります。



その結果、ゲストシステムはフルHD7770を自由に受け取り、すべてのテストに合格しました。 ゲストOSを完全にセットアップした後、リンクコピーを作成し（これによりSSDに多くのスペースを節約できました）、そこに2つ目のビデオカードを投入しました。

残念ながら、XCPとXen Serverの別々のUSBポートは転送方法がわからないため、1台のマシンに1つのUSBコントローラー（すべて背面ポート）、2台目を別のUSBコントローラー（ケースにすべてのポートが表示）を投げました。 組み込みのUSBコントローラはPCIでハングアップするため、ビデオカードのように転送されます。



公平に言えば、個々のUSBの転送はXenで行われることに注意する価値がありますが、私の意見では、それを管理することはより困難です（Linuxでの経験が必要です）



これにより、1人のシステムユニットを2人の仕事とゲームに完全に使用し、必要に応じて追加の仮想マシンを同時に実行することが可能になりました。 ビデオカードまたはプロセッサのパフォーマンスの低下は認められませんでした。 作業の中断、フリーズ、上記以外の問題はありませんでした！

ESXi-VMwareハイパーバイザー


おそらく最も人気があり強力です。 テスト中、彼は次のようにお気に入りでした ほとんどすべての点で、私の意見では、XenServerよりも優れています。 まず、仮想マシンを作成するときに、鉄の細かい設定を指定できます。これは、実際の鉄を注ぐためのイメージを準備するのに間違いなく便利です。 それどころか、XenServerは事前に準備されたテンプレートのみを持ち、その裁量ですべてを実行します。 次に、3種類のメモリバランシングが同時に実行されます。また、厳密に定義された量のプロセッサ時間をゲストマシンに「与える」こともできます。 32 GBのメモリ制限がある無料版があります。 PCIデバイスのパススルーは、これを行う最も簡単な方法です。必要なデバイスをチェックオフして、ゲストマシンに追加します。 ただし、一部のビデオカードのみが正常に転送されます。 成功はメーカーに依存せず、AMDとnVidiaの両方が転送されますが、ほんのわずかです。 Radeon 7770を仮想マシンに転送できませんでした。 ドライバーのインストール中に、BSODが取得されます。

転送中に、混乱を招く可能性のある非常に興味深いバグに遭遇しました。 実際、ゲストマシンに2 GBを超えるメモリが割り当てられている場合、ビデオカードを転送した後、マシンは起動しません。 ここで問題の解決策を見つけました。

このハイパーバイザーには別の問題があります。それを受け取って通常のハードウェアに置くことはできません。 認定のみ！ インストールの初期段階で、彼は適切なネットワークカードを見つけられず、続行を拒否しました。 かなり奇妙だ、なぜなら 私のマットに組み込まれています。 ネットワークアダプタカードはIntel 82579であり、最悪のオプションではありません、同意します。 さらに、ビンにDリンクDGE 528Tが見つかりましたが、このボードもサポートしていません。 少しグーグルで、私は次の解決策を見つけました。 特許イメージ、まだESXi 5.1をインストールすることができました。



Xen-GPLライセンスのハイパーバイザー


現在、非常に人気のあるCitrix XenServerや、あまり知られていないOracle VMなど、多くの商用製品がベースになっています。 このハイパーバイザーには3つの用途があります。

  1. Xen自体を使用します（最長の方法）。 この場合、Linuxディストリビューションをインストールしてから、Xenをインストールして構成する必要があります。 同時に、構成ファイルの作成と編集により、仮想マシンとハイパーバイザー自体の構成と起動が実行されます。 コマンドラインを介したすべての制御。
  2. オープンXCP（Xen Cloud Platform）を使用します。 この場合、サードパーティのアプリケーション（便利で高速）を使用してゲストOSと同様にハイパーバイザーを管理する機会が得られます。 最適な組み合わせ：XCP + Citrix XenCenter（無料のCitrix XenServer管理ソフトウェア）。 XCP 1.5ベータ版は、市販のCitrix XenServer 6.0のほぼ正確なコピーであり、無料のXCPの機能はXenServerのエンタープライズバージョンと同等です。
  3. 市販のCitrix XenServerを使用します。 機能が制限された無料版があります。メモリバランシングは無効になり、ビデオカードはXenCenterによって転送されます（コマンドラインを使用して、とにかくカードが転送されます）。 XenServerはXCPに基づいており、XCPと大差はありませんが、CitrixはXenServer用の頻繁なパッチをリリースします。これは選択に影響する重要な要因です。

ハイパーバイザーは、私にはよく理解されていませんが、それについてはほとんど何も言えません。 彼は、ビデオカードを含むPCIデバイスを転送する方法を知っているようです。 私が理解しているように、それは独立したプラットフォームではなく、Linux上に置かれ、その後設定されます。 私はコマンドラインを選ぶのが好きではなく、Linuxで十分な経験がないため、最後に検討することにしました。



XenServer（またはXCP）をインストールして構成し、ビデオカードを転送します



1つは実際には他の製品のクローンであるため、2つの異なる製品を組み合わせました。構成の違いはほとんどありません（それらをマークします）。



  1. XCPおよびXenServerの公式サイトからダウンロードします（上記の[無料で試用]をクリックします）。 XPCを使用する予定がある場合でも、XenServerもダウンロードすることをお勧めします。 client_installフォルダーのインストールディスクにはXenCenterがありますが、これは不要です。
  2. ハードウェアにハイパーバイザーをインストールします。 インストールにはウィザードが装備されており、問題は発生しません。 ただし、iSCSIではなくローカルディスクを使用する場合は、インストール時に仮想マシンを保存するHDDを選択する必要があります。 インストール後、これは次のような困難を引き起こす可能性があります。 コマンドラインからのみ行われます。
  3. 同じローカルネットワーク内の任意のPCのCitrix XenServerイメージからCitrix XenCenterをインストールします（最初のステップでダウンロードしました）。 XenCenterを起動し、サーバーに接続します。
  4. （XPCのみ!!!）XCP 1.5ベータ版でXenCenterを完全に動作させるには、ハイパーバイザーコンソールで次のコマンドを実行する必要があります。 

    cd /opt/xensource/bin /etc/init.d/xapi stop cp -vp xapi xapi.original sed -i 's/1\.4\.90/6.0.99/g' xapi /etc/init.d/xapi start


    その後、XenCenterにタブが表示され、メモリバランシングを有効にしたり、ビデオカードをゲストマシンに転送したりできます。
  5. 必要なテンプレートを選択して、新しい仮想マシンを作成します-Windows 7（64ビット）
  6. XenCenterに表示される仮想マシンの「コンソール」タブに移動して、Windows 7 x64をインストールします。
  7. 「コンソール」タブでxs-tools.isoイメージをマウントして、準仮想ドライバーをインストールします。
  8. ユーザーにパスワードを設定し（ターミナル経由で接続する必要があります）、ターミナルアクセスを有効にします（コントロールパネル->システム->リモートアクセス設定）。 これは、ビデオカードを転送した後にゲストOSにアクセスするために必要です。 画像はコンソールに表示されなくなります。
  9. ハイパーバイザーコンソールに移動します。 募集します 

      lspci


    識別子付きのPCIデバイスのリストを参照してください。 必要なデバイスのコードを覚えています。 たとえば、最初のpci-eスロットのビデオカードの識別子は01：00.0で、サウンドカードはそれに組み込まれている01：00.1です。
  10. 募集します 

      xe vm-list


    仮想マシンのリストを参照してください。 必要なマシンのuuid識別子を覚えています。 たとえば、d103a91d-5c38-844f-14d5-64b3c495eb08。
  11. ビデオカードを転送するには、次のように入力します 

      xe vm-param-set other-config:pci=0/0000:01:00.0,0/0000:01:00.1 uuid=d103a91d-5c38-844f-14d5-64b3c495eb08


    例の値を独自の値に置き換えます。 転送するすべてのpciデバイスをコンマでリストします。
  12. 警告バグ！ ゲストマシンでメモリバランシングが有効になっている場合、転送デバイスは正常に動作しません。

    バランシングにより、メモリの量をオンザフライで変更できます。 たとえば、最大メモリ制限が8 GiBのマシンが2台あります。 ハイパーバイザーに8 GBの空きメモリしかないとします。 次に、1つのマシンの電源がオンになると、ハイパーバイザーは8 GiBの空きメモリをすべて割り当てますが、2番目のマシンを起動すると、ハイパーバイザーは最初のマシンのメモリ量を（停止せずにオンザフライで）変更します。 したがって、それぞれが4 GBを受け取り、すべての空きメモリを使い果たします。

    バグの本質は、ハイパーバイザーの空きメモリ量が仮想マシンの最大メモリ量よりも少ない場合、このマシンが起動すると、転送されたPCIデバイスの一部がそこから落ちるということです。
  13. ゲストOSを起動し、新しいデバイスが表示されたことを確認します。 最後のCatalystをダウンロードし、インストールせずにフォルダーに展開し、デバイスマネージャーに移動して、そこからビデオカードドライバーを更新し、目的のフォルダーを指定します。

    事実、Catalyst Control CenterはBSODを呼び出します。BSODをインストールできたとしても、ビデオカードは安定して動作せず、イメージアーティファクトが定期的に表示されます。
  14. 再起動後、XenCenterの仮想マシンコンソールは、Windowsのブートフェーズ中にフリーズします。 ビデオカードが転送されるので、モニターをコネクタに接続できます！
