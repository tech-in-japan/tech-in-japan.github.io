2017年の春の書き方

最近Habréで公開された初心者向けの古典的な記事の1つで、Javaでの基本的なWebアプリケーションの作成について説明されました。 すべてはサーブレットで始まり、次にJSPページを作成し、最後にそれをコンテナにデプロイしました。 新鮮な外観でこれを見て、私は初心者のためだけにおそらく恐ろしいことに気づきました-シンプルで明確なPHPまたはNode.jsの背景に対して、すべてがシンプルです-コントローラを作成し、オブジェクトを返し、JSONまたはHTMLになりました。 この気持ちを少し払拭するために、「春の新人ガイド」を書くことにしました。 この記事の目的は、JavaでWebアプリケーションを作成すること、さらにSpring Frameworkで Webアプリケーションを作成することは、web.xml、persistence.xml、beans.xmlを苦労して引き裂くことではなく、断片的なカードの家としてアプリケーションを収集することではなく、迅速かつ快適なプロセス。 聴衆は、初心者の開発者、他の言語の開発者、Springを最高のタイミングで見なかった人々です。







はじめに



この記事では、現代のSpringに含まれるもの、Webアプリケーションを開発するためのローカル環境のセットアップ方法、およびデータベースからデータを取得してHTMLページとJSONを提供する単純なWebアプリケーションを作成する方法について説明します。 奇妙なことに、私が検索トップで見つけた初心者向けの記事（ロシア語）のほとんどは、コンテキストの手動作成、アプリケーションの起動、およびXMLを介した構成の両方を説明しています-もちろん、これは現代のSpringでは必要ありません。







春とは？



まず、いくつかの言葉、春とは何ですか。 今日、「春」という用語は、多くの場合、プロジェクトのファミリー全体を意味します。 ほとんどの場合、それらはPivotalとコミュニティによって開発および監督されています。 主な（すべてではない）Springファミリープロジェクトは次のとおりです。







  • Spring Framework （またはSpring Core）

    プラットフォームのコアは、アプリケーションを作成するための基本的なツールを提供します-コンポーネント管理（beans、 beans ）、依存性注入、MVCフレームワーク、トランザクション、基本的なデータベースアクセス。 これらは主に低レベルのコンポーネントと抽象化です。 基本的に他のすべてのコンポーネントで暗黙的に使用されます。







  • Spring MVC （Spring Frameworkの一部）

    個別に言及する価値があります。 主にWebアプリケーションについて説明します。 コントローラ、リクエストマッピング、さまざまなHTTP抽象化などの概念で動作します。 Spring MVCは、Thymeleaf、Freemaker、Mustacheなどの通常のテンプレートエンジンを統合し、さらに多くのサードパーティと統合されています。 したがって、JSPやJSFのような恐怖は必要ありません。







  • 春のデータ

    データアクセス：リレーショナルデータベースおよび非リレーショナルデータベース、KVストレージなど







  • 春の雲

    マイクロサービスアーキテクチャに非常に役立つ-サービスの検出、トレースと診断、クエリバランサー、サーキットブレーカー、ルーターなど







  • 春のセキュリティ

    承認と認証、データへのアクセス、メソッドなど。 OAuth、LDAP、およびさまざまなプロバイダー。







  • 春の統合

    さまざまなソースからのデータを処理します。 FTPから1時間に1回ファイルを取得する必要がある場合は、ファイルを行に分割し、それをフィルター処理して、キューに送信します。これはSpring Integrationに送信されます。


一般的なWebアプリケーションには、Spring MVC、Data、Securityなどのスイートが含まれる可能性が高いでしょう。 以下に、これらすべてがどのように連携するかを示します。







特に注目すべきは、 Spring Bootです。これはケーキのさくらんぼであり（ケーキ自体もそうだと考える人もいます）、XML構成のすべての恐ろしさを回避します。 ブートを使用すると、アプリケーションをすばやく作成および構成（つまり、コンポーネント間の依存関係を構成）し、実行可能な自己完結型の成果物にパックすることができます。 これは、コンポーネントのセットを完全なアプリケーションにまとめるリンクです。 Spring Bootについて知っておくべきことがいくつかあります。







  • コード生成は使用しません。 生成されたコードから、mainメソッドのみが存在します。 アプリケーションを生成するためのユーティリティを探している場合-JHipsterの可能性が高い
  • 構成にXMLを使用しません。 すべてがアノテーションを介して構成されます
  • 自動構成は最大限に使用されます。 Mongoに依存関係を追加し、接続先が示されていない場合-ブートはlocalhost:27017



    を試行しlocalhost:27017



  • 設定を使用したコンベンション 。 ほとんどの構成では、何も構成する必要はありません。
  • 簡単に移動して、デフォルトの構成を「オーバーライド」することができます。 たとえば、設定でホストを指定してMongoに接続すると、 localhost



    が自動的にブロックされます

ベースへのアクセス



まず、対象分野について説明します。 訪問の統計を収集します-誰かがメインページにアクセスするたびに、データベースにこれを書き込みます。 モデルは非常に原始的に見えます： 







 @Entity public class Visit { @Id @GeneratedValue public Long id; public String description; }


一連のコメント「ゲッターとセッターなしではありえない」と「イコール/ハッシュコードはどこにあるのか」を見越して-これらの要素はコードを単純化するために意図的に見逃されています。 このナンセンス（ゲッターと比較メソッド）を書くことを強制するJavaの設計の絶対に巨大なエラー、これは、もちろん、別個の会話です。 ところで、Kotlinはこの問題を解決します。

再びアノテーションを非常に積極的に使用します-今回はSpring Dataからです（より正確には、JPAはデータにアクセスするための密な仕様です）。 このクラスは、1つが自動的に生成される2つのフィールドを持つモデルを記述します。 このクラスの場合、データベース内のデータモデル（テーブル）が自動的に作成されます。







次に、このモデルのリポジトリを作成します。 これはコントローラーよりも簡単です。 







 @Repository public interface VisitsRepository extends CrudRepository<Visit, Long> { }

コントローラーでリポジトリーを使用するには、Spring Frameworkが提供する依存性注入メカニズムを使用します。 これを行うには、奇妙なことに、コントローラーで依存関係を宣言するだけです。 







 @Controller public class IndexController { final VisitsRepository visitsRepository; public IndexController(VisitsRepository visitsRepository) { this.visitsRepository = visitsRepository; } ... }


コンストラクターでVisitRepository



型のパラメーターを見た後、SpringはSpring Dataによって作成されたリポジトリーを見つけて、コンストラクターに渡します。







これで、コントローラーメソッドでデータベースに書き込むことができます。 







 @GetMapping("/") public ModelAndView index() { Map<String, String> model = new HashMap<>(); model.put("name", "Alexey"); Visit visit = new Visit(); visit.description = String.format("Visited at %s", LocalDateTime.now()); visitsRepository.save(visit); return new ModelAndView("index", model); }


RESTコントローラー



次のステップでは、データベースからすべてのレコードをJSON形式で返し、後でクライアントで読み取れるようにします。







RESTの場合、Springには@RestController



と呼ばれる別のコントローラータイプが@RestController



、そのコードは通常のコントローラーとそれほど変わりません。 







 @RestController @RequestMapping("/api") public class ApiController { final VisitsRepository visitsRepository; public ApiController(VisitsRepository visitsRepository) { this.visitsRepository = visitsRepository; } @GetMapping("/visits") public Iterable<Visit> getVisits() { return visitsRepository.findAll(); } }


探すべきもの：







  • 今回は、基本的な@RequestMapping



    を使用して、すべてのコントローラーメソッドの「プレフィックス」を定義します
  • 依存性注入は、通常のコントローラー（Springのすべてのコントローラー）とまったく同じように機能します。
  • メソッドは、テンプレートの名前ではなく、モデルを返すようになりました。 SpringはこれをJSONオブジェクトの配列に自動的に変換します
  • JSONでのシリアル化には永続モデルを使用しますが、これは一般的にベストプラクティスではありません。


ここで、 http://localhost:8080/api/visits



をリクエストするとき（プロジェクトをコンパイルし、 DevToolsにアプリケーションを更新させた後）、必要なデータを含むJSONを取得します。

展開



本番環境でアプリケーションをビルドして実行するには、いくつかのオプションがあります。







  1. Tomcatなどのサーブレットコンテナで、結果のJAR（またはWAR）を押し下げます。 これは最も簡単な方法ではありません;動作するサーブレットコンテナまたはアプリケーションサーバーが既にある場合にのみ選択する必要があります。
  2. Spring Bootの魔法を使用してください。 Spring Bootプラグイン（Spring Initializrで作成されたプロジェクトに自動的に追加される）を使用してコンパイルされたJARファイルは完全に自己完結型です。


したがって、アプリケーションのアセンブリと起動は次のようになります。







  1. ./mvnw package



  2. java -jar ./target/demo-0.0.1-SNAPSHOT.jar





このJARファイルをデプロイするには、インストール済みのJava（JRE）のみが必要です。 これは、いわゆるファットJARです。組み込みのサーブレットコンテナ（デフォルトではTomcat）とフレームワークの両方、およびすべての依存ライブラリが含まれています。 実際、これは唯一のデプロイメントアーティファクトです。ターゲットサーバーに単純にコピーして実行できます。







さらに、ファイルを「実行可能」にして、コマンドラインから簡単に実行することができます（もちろん、Javaが必要です）。







このファイルに基づいて、Dockerイメージを簡単に作成したり、デーモンとしてインストールしたりできます。 詳細は公式ドキュメントに記載されています 。







おわりに



それにもかかわらず、非常に簡潔に判明しました-しかし、最も簡単なSpringの入門コースでさえ、1つの記事のフレームワークに入れることは非常に簡単ではありません。 これが誰かが春の最初の一歩を踏み出すのに役立つことを願っていますが、彼の基本概念は理解しています。







ご覧のとおり、この記事の本文で「Spring Magic」という言葉が何度も聞かれています。 基本的に、これは非常に「魔法の」フレームワークです-氷山の最上部を見てさえ、Springが多くのことをバックグラウンドで実行していることを既に見ました。 これは、フレームワークのプラスとマイナスの両方です。 さらに、間違いなく、多くの複雑な（非常に多くの）ものを1つの注釈または依存関係で作成できます。 欠点は隠れた複雑さです。複雑な問題を解決したり、極端な場合にフレームワークを動作させたり、すべての微妙な点や側面を理解したりするには、それをよく知る必要があります。







「知る」段階をできるだけ簡単にするために、Springには優れたドキュメント、巨大なコミュニティ、クリーンで読みやすいソースがあります。 SpringをRich Hickeyスケールに配置すると、Spring（Spring）は確かに簡単になりますが、単純ではありません。 しかし、（企業だけでなく）現代の企業にとっては、本番環境に対応したアプリケーションを非常に迅速に入手し、周囲のインフラストラクチャではなくアプリケーションロジックに集中するという素晴らしい機会を提供します。
