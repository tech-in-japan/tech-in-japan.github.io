モバイルアプリのデザインパターン。 コマンドプロセッサ

「あたかもあなたが住んでいる場所を知っている暴力的なサイコパスが同伴するかのようにコードを書いてください。」

マーティン・ゴールディング



GooglePlacesAPIを使用してプロジェクトの1つを開発するとき、AndroidアプリケーションとサーバーAPI間のネットワーク相互作用を整理する問題がありました。 AsyncTaskの直観と「麺」は、この種の相互作用を整理する他の方法があるべきだと示唆しました。 そこで、私はCommandProcessorデザインパターンに出会いました。 Androidアプリケーションでのこのデザインパターンの使用についてお話したいと思います。



まず、解決する必要がある問題について説明します。 Google Places APIを使用してアプリケーションを作成し、ユーザーが選択したマップ上の任意の場所のプレビューを表示し、ユーザーが詳細な情報（たとえば、より多くの画像を表示）を取得する場合、選択した場所の指定されたIDに画像を読み込み、すべての画像を既に表示する必要がありました選択した場所に関連しています。 当時の私にとって最も明白な方法は、AsyncTaskを使用することでした。 しかし、いくつかの試みの後、より便利な他の方法があるはずであることが明らかになりました。 AsyncTaskを使用すると、次の理由で不便でした。



1）場所のプレビューを取得するには、まず、ユーザーが選択した場所の隣にあるすべての場所に関する情報を取得するよう要求する必要がありました。



2）受信したIDフォームに基づいて、写真プレビューのリクエストを送信します。



3）プレビューをクリックすると、この場所に関連するすべての写真が表示されます。



したがって、AsyncTaskを使用すると、特定の「ウォーターフォール」が取得され、あるAsyncTaskを別のAsyncTask内で使用する必要があります。 そして、グーグルで調べると、上記のタスクに対処するコマンドプロセッサパターンに関する情報が見つかりました。



CommandProcessorデザインパターンは、サービスリクエストとその実行を分離します。 パターンの主要コンポーネントであるCommandProcessorは、要求を管理し、実行をスケジュールし、追加のサービスも提供します。たとえば、実行の遅延や要求のキャンセルなどの要求の保存です。 [1]から引用した図は、パターンのコンポーネント間の関係を示しています。







パターンの範囲



  • ユーザーがグラフィカルインターフェイスを介してシステムと対話するプログラム。 例：テキストエディター、オフィスアプリケーション。 コマンド（以降、コマンドはクエリと同じ）は、ユーザーがマウスボタン、ホットキー、メニューボタンなどを押した後に何らかのタスクを実行するために使用されます。 コマンドは、実行のためにプロセッサに渡されるオブジェクトです。
  • 分散システムまたは並列システム。 このパターンを使用すると、コマンドを送信して別のスレッドで実行できます。 コマンドなどのキューを作成します。
  • ネットワークと対話するアプリケーション。


実装

ここで、状態に応じて、プロセッサに異なるコマンドを送信するコントローラーが必要です。 要求の結果は、ハンドラーを使用してストリームから送信されます。 



 public class MessageController extends Handler { private static MessageController instance = null; private RequestProcessor processor_; public void init (UpdateCallbackListener listener) { processor_ = new RequestProcessor(listener); } public static MessageController getInstance(){ if (instance == null){ instance = new MessageController(); } return instance; } public void handleMessage(Message msg) { switch (msg.what) { case States.INIT_REQUEST: CommonRequest request = (CommonRequest)msg.obj; processor_.execute(request); break; case States.REQUEST_COMPLETED: List<Result> results = (List<Result>)msg.obj; processor_.updateActivity(results); break; default: break; } } }

IntentServiceを使用した実装



IntentServiceを使用すると、このパターンを完全に実装することもできます。 図を考えてください：







Intentをコマンドオブジェクトとして使用し、プロセッサに渡すことができます。 Creatorはアクティビティであり、コマンドオブジェクトを作成し、このオブジェクトをエグゼキュータ、つまりこの場合はIntentServiceに渡します。 したがって、CommandProcessorの役割は、CustomIntentServiceクラス、つまりonHandleIntent（）メソッドによって実行されます。onHandleIntent（）メソッドは、Intentに含まれるデータに応じて、さまざまな操作を実行できます。 この場合、結果をアクティビティに返すには、BroadcastReceiverを使用できます。



順を追った説明



したがって、要約すると、このパターンを実装するには、以下を実行する必要があります。



  • 抽象コマンドのインターフェースを定義します。 インターフェイスは、個々のコマンドの実装をカプセル化します。 私たちの場合、それはsendRequest（int i）メソッドであり、各コマンド（リクエスト）は別々に実装されていました。
  • チームが結果を呼び出し元に返す方法を決定します。 このように、各アクティビティにonUpdate（）メソッドを使用して、UpdateCallbackListenerインターフェイスを定義しました。
  • 各チームを実装します。 リクエストには2種類あります。1つはルートに関する情報を受信するため（TracksRequest）、もう1つは場所に関する情報を受信するため（PlacesRequest）です。
  • コマンドを送信するコントローラーを実装します。 コマンドの作成と送信は、Abstract FactoryまたはPrototypeを使用して実装できます。 ただし、コントローラーがコマンドの作成者である必要はありません。 例からわかるように、リクエストオブジェクトはアクティビティで作成されました。
  • コマンドとプロセスを受け取るプロセッサクラスを実装します。 リクエストごとに、RequestProcessorクラスはexecute（）メソッドを実行します。




パターンの長所と短所。



利点：



  • テキストエディタなどのアプリケーションでは、異なるインターフェイス要素が同じコマンドを使用できます。 たとえば、「作成」メニューのボタンと、コンテキストメニューの同じ名前のボタンは、同じコマンドにアクセスできます。
  • 要求の結果に応じて、非同期要求を送信し、呼び出しの順序を制御する機能。
  • 既存の機能を損なうことなく、新しいコマンド（リクエスト）の形で新しい機能を追加する際の柔軟性。 新しいコマンドを変更または追加しても、コマンドハンドラーには影響しません。
  • 異なるクライアント（この場合、アクティベーション）は同じコマンドを使用できます。 たとえば、IDによるイメージのダウンロード要求は、さまざまなアクティビティに使用できます。


短所：



  • さらに、各状態を追跡し、それに応じて処理する必要があります。
  • チームの数は非常に多くなる可能性があります
  • 双方向通信。 アクティベーションが要求の作成を開始し、応答が受信された後、結果をアクティベーションに返す必要があります。




おわりに



おそらく、何らかの形で、すでにパターンの実装を見てきました。 しかし、モバイルアプリケーションアーキテクチャのテーマは非常に重要であるため、この記事が役に立てば幸いです。 将来的には、モバイルアプリケーションの開発で使用されるいくつかのパターンを検討する予定です。 コメントに質問を書いて、後でまた会いましょう。
