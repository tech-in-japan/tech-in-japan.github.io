パート1、組織



ご存じのとおり、次の2つの方向に進むことで、雇用主の有能な専門家を獲得できます。

  • 市場で既製の専門家を見つける。
  • 必要な能力を持つ従業員を独立して「成長」させます。


どちらの方法にも長所と短所があります。 ほとんどの企業（当社を含む）は、これらのアプローチを仕事に組み合わせています。 そして、訓練された専門家の検索ですべてが多かれ少なかれ明確である場合、若くて早くなり、許容可能な時間で適切なレベルに成長できる場所はどこですか？



この点で、大学との関係が確立されている企業には一定の利点があります。 特別なコースを読んだり、大学院のデザインを指導したりする専門家は、確実に連絡を取り、新進生とのコミュニケーションを確立します。 しかし、このオプションは決して大規模なものではなく、数か月間で導入されます。



将来の従業員と潜在的な雇用主の間の潜在的な「接触ゾーン」を拡大するために、「キャリア日数」が考案されました-これらは、プロセスの両側が誰が価値があるのか​​を迅速に理解し、十分な注意と責任を持って長く幸せになることができる関係を確立する特別なイベントです。

APIを使用するには、 VK-ruby Rubyライブラリを使用できます。 以下の簡単なスクリプトは、モスクワ州立大学のVMKをトレーニングの教員として指定しているユーザーとその友人（参加者にこのようなことを期待している）に関する情報を取得するように設計されています。 2012-2021年のリリースを示すユーザーのデータが収集されます。



データを取得するスクリプト 
 require 'open-uri' require 'vk-ruby' require 'json' token = "%TOKEN%" (2012..2021).each do |year| app = VK::Application.new access_token: "#{token}" users = [] offset = 0 begin us = app.users.search(university: 2, university_faculty: 23, university_year: year, count: 1000, fields: 'sex', offset: offset) users += us[1..-1] offset += 1000 end while us.length > 1 cmc_uids = Set.new users.map{|u| u['uid']} users.map! do |u| uid = u['uid'] page = open("https://api.vk.com/method/friends.get?user_id=#{uid}").read friends_ids = Set.new JSON.parse(page)['response'] u['friends'] = friends_ids.to_a cmc_friends = friends_ids.intersection cmc_uids u['cmc_friends'] = cmc_friends.to_a u end File.open('cmc_' + year.to_s + '.json', 'w') { |file| file.write(users.to_json.gsub(/,\{/, ",\n{")) } end

第三の競争。 「フラッシュドライブの非表示とシーク」



このコンテストはおそらく最も人気がありました。 おそらく、タスクのリーフレットに加えて、競技者にフラッシュドライブが提示されたためです。 はい、タスクは1つの文に収まります：フラッシュドライブの腸内では、ファイルは「慎重に」隠されています：画像、テキスト、アーカイブ-それらを見つけてください！ しかし、見かけの単純さの背後に、本当の脳の爆発がありました。 なんで？



コンテストを準備するとき、情報を非表示にする方法を見つけなければなりませんでした。 さまざまなアイデアが表明されました。メディアに「失われた」セクションを作成し、そこに情報を保存し、写真からステガノコンテナを作成します。必要な条件は、「オープン」情報（広告、はい）と「非表示」の組み合わせでした。 情報がセクションの外に書かれているとき、私たちはアイデアに落ち着きました。 情報の検索を簡素化するには、最初にメディアをゼロで初期化する必要があります。 情報の一部をセクションの前に、後の部分を書き留めることにしました。 リカバリアプローチに関して、自動リカバリ（データリカバリユーティリティを使用）、手動リカバリ（ディスク用の16進エディタ）、デザート用のシンプルなステガノグラフィの3つの難易度レベルを特定しました。



jpegファイルは、第1レベルの複雑さのオブジェクトになりました。 これは、パーティションの最後から連続したブロックとして1024セクターに「置かれ」ました。 最後に、ゼロの背景に対して、彼は目立っていたはずです。



手動リカバリの対象はテキストでした（bash.imにいくつかの真珠を借りました）。 さらに、テキストは断片化されています。 各フラグメントの最後に、次のフラグメントが配置されているセクターの番号を記録しました。 タスクが単純すぎないように、メインテキストにはUnicodeが使用され、次のセクターへのポインターはASCIIの10進数として表示されました。 セクター番号1024は、チェーンの開始セクターとして採用されました。 ここで、セクター番号2048で始まったセクションと言っておく必要があります。 フラグメントを次のチェーンに配置しました：1024 => 964 => 475 => 235 => 1800 => 1720 => 823 => 615 => 1100 => 1920 => 345 => 1418 => 1634 => 1980 => 1320テキスト「Vulkan-task」。最初のフラグメントに表示され、最後に「だからタスクの終わりに到達しました...」という情報



第3レベルの複雑さのオブジェクトは、jpegファイルの最後に追加されるrarアーカイブです。 ご存じのように、rarアーカイバはファイル全体で最初の署名を検索します。これにより、たとえばjpegなどの任意のファイルの最後にrarアーカイブを保存できます。 jpegが自動的に復元されると、ファイルサイズはヘッダーによって決定され、残りは切り捨てられます。また、探究心のある人だけがアーカイブの存在を判断できます。 すべてのITスペシャリストから尊敬されているA. S.からの引用とともに、ドキュメント「Vulkan.docx」をアーカイブに入れました。 プーシキン。 最後に、「Vulkan」という言葉でアーカイブをパスワードで保護することにしました（「Volcanoはすべての鍵」というフレーズは、アーカイブへのパスワードのヒントとして使用され、最初のフラグメントはタスクリーフレットに置かれました）。



私たちは、競技者が最初に写真を復元し、次にテキストを復元し、次にアーカイブから目と鼻の先になると想定しました。 しかし、限られた時間の条件では、何かが「うまくいかない」可能性があることに気付きました-彼らはいくつかのヒントを用意しました。
