LoadRunnerから.NETを使用する方法

LoadRunnerにはさまざまなテキスト処理に適したAPIがありますが、それでも十分ではない場合があり、汎用関数を使用して拡張する必要があります。 多くの場合、そのような実装は自転車の発明になります。ご存知のように、ほとんどすべてのタスクは既に誰かによって解決されています。 さらに、私はC＃のバックグラウンドが十分にあるため、問題を解決するときに、手元に.NET Frameworkクラスライブラリがあれば、このタスクは簡単に解決されると考えることがよくあります。 原則として、私がJavaプログラマであれば、Java（ほぼすべてを備えています）についても同様の考えがありますが、.NETは私に近いので、それについてのみ説明します。 副作用として、この記事はネイティブコードからCLRコードを呼び出す方法を学びたい人に役立ちます。 このオプションの小規模なパフォーマンス調査も提供されており、動作するVisual StudioプロジェクトテンプレートとLoadRunnerスクリプトが含まれています。



LoadRunnerの.NETおよびJava



まず、魅力的だが悪い選択肢を考えてください。 原則として、.NETとJavaの両方がLoadRunnerで直接動作します。 これらの各プラットフォームには、標準のLoadRunner APIのラッパーであるクラスがあります。 それぞれ.NETおよびJava Vuserモードを選択することにより、それらをすぐに使用できます 。 すぐに言ってみましょう。これらのスクリプト開発モードは、わずかに異なるモード用に作成されています。 .NETモードでは、.NETアプリケーションのアクティビティを記録し、アプリケーションのクラスメソッドを直接呼び出すスクリプトを作成できます。 Java VuserにはJava用の本格的で文書化されたAPIがありますが、記録モードはまったくありません（ .NETと同じ方法でJava Record Replayがあります ）。 このため、それらをWebに使用することは非常に問題が多く、一般的に、この場合の「使用」とは「何らかの形で機能するコードを書く」ことを意味し、それ以上は何もしません。 ツールからロードスクリプトを開発する場合、トラフィックを記録し、それをスクリプト（ドラフトも含む）に変換できることが重要です。これはさらに開発されます。 しかし、ここに問題があります：Webトラフィックの記録（ Webモード-HTTP / HTML ）は、Cコードへの変換でのみ可能です。 コミュニティはこれを長い間待っていましたが、少なくとも新しいバージョン12.00ではC＃またはJavaを選択できるようになることを正直に願っていましたが、これは起こりませんでした。 さらに、.NET用のAPIは実際には文書化されておらず、暗闇の中で熊手を踏んで行動する必要があります（ところで、.NETとJavaラッパーのメソッドシグネチャも異なります）。 .NETクラスで必要なラッパーメソッドを比較的迅速に見つけることができる場合、パラメーターを渡す方法がわかりません。

C＃でカスタムライブラリを呼び出す



C＃でより複雑なロジックを記述し、別のアセンブリにパッケージ化したとします。 LRから彼女にアクセスする方法は？

原則として、すべては前の場合と同じで、リンクをアセンブリに追加するだけで（ソリューションまたは参照を使用）、クラスをアセンブリから呼び出す必要があります。



C＃DLL


何らかの理由で、LoadRunnerはサーバー応答でのみ正規表現を検索できますが、C行またはパラメーター値で正規表現を見つける方法は不明です（誰かが知っているなら、鼻を突く）。 この問題を解決するには、次の関数を記述できます。 



 // C#-,    input   pattern   //   nGroup    nMatch. namespace HplrCs { public static class HplrHelper { public static string GetRegexMatch(string input, string pattern, int nMatch, int nGroup) { try { var re = new Regex(pattern); var matches = re.Matches(input); if (matches.Count < nMatch + 1) return String.Empty; var match = matches[nMatch]; if (match.Success) { if (match.Groups.Count < nGroup + 1) return String.Empty; return match.Groups[nGroup].Value; } else return String.Empty; } catch (Exception ex) { return ex.ToString(); } } } }


このコードを含むアセンブリをグローバルアセンブリキャッシュ（GAC）に配置する必要があります。 これを行うには、Windows SDKの一部であり、Visual Studioと共にインストールされるgacutil.exeユーティリティを使用します。 .NET Framework 4.0 / 4.5の場合、SDKの8番目のバージョンのgacutil.exeの対応するバージョンを使用する必要があります。以前のバージョンは4.0 / 4.5ビルドをインストールできません。



gacutil.exe -i HplrCs.dll







GACでのインストールは、管理者権限で実行する必要があります。 次のようにして、アセンブリがキャッシュに存在することを確認できます。



gacutil.exe -l HplrCs







GACのビルドバージョンを置き換えるには、以前のバージョンの上にインストールを再度実行する必要があります。 古いバージョンを削除する必要はありませんが、インストールが成功したことを確認することが重要です。

Visual Studioでプロジェクトを初めて開くと、使用されている.NET Frameworkのバージョンを更新するように求められます。 何が起こっているのか理解できない場合は、これを行うべきではありません。 バージョンを更新すると、このバージョンとターゲットフレームワーク（プロジェクトプロパティ）を使用して.NETアセンブリをビルドする必要があります。 ところで、インターフェイスのどこでC ++プロジェクトの.NET Frameworkのバージョンを変更できるかはわかりませんでした。 C ++プロジェクトのプロパティに表示されますが、表示するためだけに変更することはできません。 ただし、これはテキストエディターで.vcxprojファイルを開き、XML要素TargetFrameworkVersionを見つけることで実行できます。 したがって、プロジェクトをまだ更新している場合は、TargetFrameworkVersionを必要なものに編集するか、.NET Frameworkの別のバージョンを使用するように切り替えます。



性能



提案されたアプローチのパフォーマンスは、組み込み関数を使用する意図が少なく、CのLRでコードを使用したいほど魅力的です。 何らかの目的で組み込みのLoadRunner関数がある場合は、その背後にネイティブコードがあるため、それを使用する必要があります。これは、より高速に機能することを意味します。



たとえば、約32KBの長さの文字列から部分文字列を見つけてみましょう（これより長くするのは困難です。これは、LoadRunnerがローカルパラメータのスタックに割り当てることができる最大値であるためです）。 私は特に部分文字列検索を古典的なタスクの1つとしています。なぜなら、おそらくそのアルゴリズムは可能な限り最適化されるべきだからです。 



  #define BUFF_SIZE 32700 char buff[BUFF_SIZE]; int i; lr_load_dll("hplr.dll"); // Native DLL. memset(buff, '-', BUFF_SIZE); buff[BUFF_SIZE - 1] = 0; strcpy(buff + BUFF_SIZE - 4, "+++"); lr_start_transaction("Find substring, internal function"); for (i = 0; i < 100000; i++) strstr(buff, "+++"); lr_end_transaction("Find substring, internal function", LR_AUTO); lr_start_transaction("Find substrings, C# function"); for (i = 0; i < 100000; i++) find_substr_net(buff, "+++"); lr_end_transaction("Find substrings, C# function", LR_AUTO);
