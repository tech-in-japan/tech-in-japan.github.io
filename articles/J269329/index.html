Hyper-V vNext-VMバージョンと新機能

アセンブリ10565についてです。



おそらくご存知のように、仮想マシン（以降VM）を2012 R2からvNextに移行する場合、VMの構成は新しいバージョンに自動的に変換されないため、VMを元に戻すことはできません。 VM機能のバージョンへの依存：



1.「ホット」RAMの追加/削除-VMバージョン6.0

2.ネットワークアダプタの「ホット」追加/削除-VMバージョン5.0

3. Linux VMのセキュアブート-VMバージョン6.0

4.実稼働チェックポイント-VMバージョン6.0

5. PowerShell Direct-VMバージョン6.2

6.仮想TPM（vTPM）-VMバージョン6.2

7.仮想マシンのグループ化-VMバージョン6.2



彼らはここでPowerShell Directについてすでに書いているように思えますが、要するにVMBusを介してVMでPowerShellコマンドを実行する機会です。



Enter-PSSession -VMName VMName"





Invoke-Command -VMName VMName -ScriptBlock { Commands }







vNextゲストを備えたvNextでのみ機能します。すべての統合コンポーネントを有効にし、ゲストをオーバーロードする必要があります。



VMバージョンのGet-VM | ft Name,Version



： Get-VM | ft Name,Version



Get-VM | ft Name,Version





vNextで2012 R2と互換性のあるVMを作成します（ビルド10565以降）。 New-VM -Name " " -Version 5.0





VMバージョンのアップグレード： Update-VMVersion -Name " "







更新すると、構成ファイルの形式が変更され、VMの新しいバージョンでは次のファイル形式が使用されます。



1.構成ファイル-バイナリ.vmcx

2.ランタイム状態ファイル-バイナリ.vmrs

3.ディスク-.vhd \ vhdx、変更なし

4.差分ディスクファイル-.avhdx、チェックポイントに使用

5.チェックポイントファイル-チェックポイントを作成すると、.vmrs / .vmcxファイルと差分ディスクファイルが作成されます。



ところで、チェックポイントについては、vNext Production Checkpointが私たちを待っています。 これが意味することは、チェックポイントを作成するとき、VSSはVM内で呼び出されるため、VMの一貫した状態を取得することが可能になります（VSS対応アプリケーションと共に）。 私が理解しているように、そのようなチェックポイントを作成するとき、RAMと状態は保存されません。 このチェックポイントに戻ると、VMはゼロから起動します。 さらに、VMのチェックポイントのいくつかのモード（標準、本番、本番のみ、無効）を選択できます。 ここではすべてが明確であり、本番と本番のみの唯一の違いは、本番はVSSエラーの場合に通常のチェックポイントを作成し、本番のみはそうしないことです。
