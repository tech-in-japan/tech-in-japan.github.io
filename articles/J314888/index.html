デスクトップ仮想化：新機能、どのような傾向、どこへ向かうのか？





仮想化は誰も驚かないでしょう。 サーバー、アプリケーション、ワークステーションなど、すべてが今日仮想化されています。 しかし、ほとんどの一般ユーザーにとって、このテクノロジーはDMB映画のゴーファーのようなものです。「見えない、見えないが、彼はそうだ」 ただし、来るとすべてが変わります-仮想ワークステーション、そしてかさばるシステムユニットやラップトップの代わりに、オフィスワーカーの机に小さな箱が表示されます。 シンクライアント。 もちろん、すべてが少し違うかもしれませんが、それは本質を変えません。 デスクトップ仮想化とは何か、何が起こるのか、そして将来的に私たちを待っているものについて話します。



それでは、VDI（Virtual Desktop Infrastructure）とは何ですか？ 簡単です：物理サーバーでは、仮想マシンはユーザーまたはサーバーOSと共に展開され、インターネットを介して世界中のどこからでも（快適に関係なく作業が必要な場合は64 Kbpsから）任意の「スマート」デバイスからアクセスできます。 。 同時に、ユーザーにとっては何も変わりませんが、IT部門にとっては、データアクセスのセキュリティ、ジョブの可用性、シンプルさ、およびメンテナンスのコストに関して、管理の状況が大きく変わります。 ターミナルアクセスと比較してVDIのコストが比較的高い（場合によっては数倍）にもかかわらず、別の仮想マシンのレベルでクライアントOSからユーザーの作業環境を分離することが前提条件である場合、VDIは新しいポイントやオフィスを展開するための最も経済的なオプションになります会社の存在の地理から。 同時に、現地従業員のすべての機能が保持され、業務データの一元管理により、企業の機密情報が漏洩するリスクが大幅に削減されます。



VDIまたはターミナルアクセス：選択基準



VDIソリューションの主要ベンダーは、中小企業にとって必須のVDIを積極的に宣伝していますが、正直なところ、中小企業は提供されている価格で仮想デスクトップを必要としません。 セキュリティポリシーへの厳密な準拠が突然必要になった場合でも、端末へのアクセスで十分です。 ターミナルアクセスとは、最も一般的な意味で、サーバーのオペレーティングシステムの一部として、すべてのユーザーに共通で、共通の低レベルシステムプロセスなどで実行されているアプリケーションへの多くのユーザーのリモートアクセスを意味します。 これは、ハードウェアサーバー上の各ユーザーに対して、特定のユーザーに必要なオペレーティングシステムとアプリケーションと共に個別の仮想マシンが展開されるVDIとの主な違いです。 ユーザーがサーバーOSに接続している場合でも、1ユーザー= 1仮想マシンの場合、有益なハイブリッドスキームがあります。 これは、いわゆる「サーバーVDI」であり、ターミナルソリューションとデスクトップ仮想化の利点を組み合わせています。



経済の面では、ターミナルサービスが引き続き好ましいオプションです。 制限がない場合（ターミナルモードの主な問題は、サーバーOSとマルチユーザーシステムであるということです）、アプリケーションの正常性がない場合、会社は引き続き最も経済的なオプションとして仮想サービスを使用します。 どのような制限がありますか：



  1. アプリケーションメーカーは、ターミナル環境ではアプリケーションをサポートしないと述べています。 同時に、完全に正常に機能しますが、何かが発生した場合、ベンダーはサポートを拒否します。 たとえば、Autodesk Autocad。 特定のバージョンまでは、ターミナル環境では公式にサポートされていませんでしたが、適切に機能していました。 ただし、 Autodesk Autocad Map 3Dはサポートされていました。機能面では、通常のAutodesk Autocadとほとんど違いはありませんでした。



  2. ベストプラクティスから逸脱した独自開発のアプリケーション。 たとえば、結果として文書化されていない関数または呼び出しが使用されます-アプリケーションは特定のバージョンのOSに「拘束」され、クライアントまたはサーバーに転送できません。 多くの場合、そのようなアプリケーションは、それらの機能の一部を失うか、まったく機能しなければ、どこにも移動できません。



  3. 起動時に管理者権限を必要とする非推奨または「ファイル変更」アプリケーション。 ローカルマシンでは、これは通常の状況ですが、ターミナルモードでは、そのようなソフトウェアを実行するすべてのユーザーは管理者権限で動作します。たとえば、彼が管理者であるため、サーバーをオフにする場合彼らの仕事へのアクセスなしで残されます。



  4. アプリケーションは動作し、誰もが満足していますが、定期的にブルースクリーンがクラッシュします。 たとえば、パーソナルPCで月に1回クラッシュした場合-大丈夫です。 しかし、100ユーザーの端末環境で機能する場合、1日に3回統計的に低下し始めます。


これら4つのシナリオでは、ターミナルサービスの代わりに仮想デスクトップを使用することをお勧めします。



VDIハイパーバイザー



予約をしましょう。VDIでは、主に大企業でジョブの仮想化を実装するプロジェクトを検討しています。 それらは何に基づいており、オプションは何ですか？







実際、今日、仮想職場インフラストラクチャには2つのオプションしかありません（上記のターミナルアクセスと「サーバーVDI」もVDIタスクの解決によく使用されますが、完全なVDIにはカウントされません）：ハイパーバイザーとコンテナー。 ハイパーバイザーが最も一般的であり、これは、Microsoft、VMware、Citrixなどのメーカーの最も一般的な仮想化ソリューションが適用されるタイプです。 さらに、そのようなモデルは最も一般的であるという事実により、「コンテナ」の場合には利用できない多くのデバッグされた「チップ」を持っています。 たとえば、追加のGPUの直接接続の機能は、エンジニアリングおよび設計局でしばしば必要になります。 このテクノロジー（GRID Virtual GPU）は、XenServerハイパーバイザーによって完全にサポートされています。 Citrix XenAppおよびXenDesktopデスクトップへのリモートアクセスを提供するソフトウェアについて考えてみましょう。



今日、XenAppとXenDesktopは互いに違いはありません。 それらを技術的に区別する唯一のものは、クライアントOS-VDI（XenDesktopと呼ばれることが多い）またはサーバーOS-ターミナルサーバー（XenAppの意味）にインストールされるエージェントです。 ただし、EnterpriseエディションのXenDesktopにはVDIとターミナルサービスの両方が含まれることを忘れないでください。 バージョン7以前は、技術的には、これらは2つの異なる製品-異なるアーキテクチャ、管理コンソールでした。 また、ユーザーが単一のシステムを持っているように見える場合、管理者はより複雑でした。実際、ソリューションにはターミナルサービスと仮想デスクトップが含まれており、それぞれ異なる場所で個別に構成する必要がありました。 現在、ソリューションには、ターミナルバージョンと仮想デスクトップ用の単一のコンソール、システムの状態（Desktop Director）、特定のマシンの状態などを監視できる単一のWebポータルがあります。



Citrixデスクトップ仮想化インフラストラクチャは、XenServer、Microsoft Hyper-V、VMware ESXi、Nutanix Acropolisなどのハイパーバイザーで実行されます。 これらは4つのハイパーバイザーであり、システム自体が仮想マシンを自動的に作成、削除、再起動、シャットダウンできます。 したがって、ハイパーバイザーを変更するときに、シームレスに切り替えることができます。 他のハイパーバイザーの場合、XenDesktopは仮想マシンを自動的に作成、削除、管理することなく、物理マシンとして認識します。



XenDesktopを使用すると、従業員の職場を実際に置き換えることができます。 このソリューションの一部として、独自のデータ転送プロトコルが使用されます。 クライアントワークステーションにインストールされているエージェントを使用できない場合は、独自の（Citrix）HTML5実装が使用されます。 これは、1つのプロトコルがライセンスされており、2番目のプロトコルが独自の設計である他のベンダーの製品との違いの1つです。

一部のお客様がまだ産業用UNIX（AIX、HP Unix、Solaris）を使用している場合でも、XenAppおよびXenDesktopのフレームワーク内で、Enterpriseエディションから、XenApp for UNIXをインストールできます。 したがって、Citrix製品は、WindowsサーバーオペレーティングシステムおよびさまざまなLinuxバリアントによってサポートされます。 オフラインモード、つまりMacとWindowsのラップトップで動作する独立したDesktop Player製品を使用する必要がある場合。 これは、エンドユーザーが自分で仮想マシンを展開できることを意味しない企業ソリューションです。これは、管理者の助けを借りてのみ実行できます。



「何を選択するか」を尋ねる場合、特定の目的の製品からではなく、アプリケーション、それらのライセンスオプション、ユーザー要件、およびソリューション全体のコストによって課される制限から始める価値があります。 純粋な形式のXenAppは、競争力のあるユーザーとしてのみライセンスされます。 XenDesktopは、競争力のあるユーザーとユーザー/デバイスの両方としてライセンスを取得できます。つまり、ユーザーとデバイスに個別のライセンスはありません。 この場合、ライセンスサーバーのロジックが機能し、現在の使用中にライセンスの最小数が消費されるように切り替えるオプションを選択します。



ユーザー/デバイスライセンスと競合ユーザー（XenAppおよびXenDesktop）のコストの差は約2倍です。 顧客がXenAppを必要としていることが判明することもありますが、ユーザーデバイスライセンススキームの下で販売されているXenDesktopを購入する方がより有益です。 彼は必要なものを2倍安く受け取りますが、必要に応じて後で仮想デスクトップに接続できます。



VDIコンテナーとクラウド外







それでは、コンテナ仮想化について話しましょう。 今日ではエキゾチックです。そのようなテクノロジーの最も有名な例は、Windows用のParallels Containersです。 このオプションの主な利点は、従来のVDI（ただしターミナルサービスよりも低い）と比較して、サーバー上のユーザーの密度が非常に高いこと（最大250シート）であり、場合によってはMicrosoftソフトウェアライセンスを少し節約できることです。 コンテナ仮想化には、多くの利点がありますが、VDIやターミナルサービスから受け継がれてきた多くの欠点もあります。 Windows Server 2016のリリース以前は、Microsoftはコンテナーテクノロジーをまったくサポートしていなかったため、インフラストラクチャにこの製品を実装した企業は、サーバーOS（およびこれはサーバーOSのデスクトップ）の更新がリリースされると、Parallelsが対応する更新のインストールを確認して承認するまで待たなければなりませんでした。 現在、Windows Server 2016のリリースおよびOS内のWindowsコンテナーテクノロジーの公式サポートの後、状況は変わる可能性があります。



コンテナ仮想化テクノロジは、OS（Windows Server 2008/2012）のカーネルで動作します。これにより、コンテナ間でカーネルオブジェクトを分離し、それらを相互に分離できます。 各コンテナには、独自のプロセス、セッション、ドライバーのセット、およびレジストリとカーネルオブジェクトのツリーがあります。 同時に、完全なデスクトップ仮想化が達成され、ユーザーはサーバーOSのデスクトップ、通常のクライアントオペレーティングシステムで特別なテーマが「再描画」されるので、クラシックバージョンと比較して違いに気付きません。 このオプションは、平均して、ハイパーバイザーの場合よりも1.5倍少ないコンピューティングリソースとストレージリソースを必要とします。 これは、このオプションが従来のVDIよりも明確に経済的であることを意味するものではありませんが、特定の条件下では、サーバー側の処理能力の節約が主なものになり、ビジネスの観点から実装を正当化できます。 現時点では、これはまだ完全に認識されているデスクトップ仮想化テクノロジーではなく（MicrosoftはまだこのVDIオプションをサポートしていません）、ハイパーバイザー仮想化に利用できる機能の一部はコンテナーで利用できません。



VDIへの現在のアプローチは次のとおりです。 次は？ そして、その道は「雲の中へ」行きます。 たとえば、今日、エンタープライズ版のVMware Horison 7では、クラウドオートメーションを使用したデスクトップとアプリケーションの起動が実装されています。 ただし、Microsoftのライセンスルールによると、現時点では、パブリッククラウドからのクライアントデスクトップの配信は禁止されています。 唯一の「救済」は、Citrix製品用です。 2016年初頭、CitrixとMicrosoftは、Microsoft Azureクラウドに展開されたXenDesktopシステムのみが合法的な選択肢であると発表しました。 この場合、ユーザーにWindows 10をデスクトップとして提供できます。 XenDesktopは、Amazon Web Services、Microsoft Azure、またはCloud Platformですぐに展開できるようになりました。



これまでのところ、これらは製品へのクラウド仮想化の「組み込み」の始まりにすぎませんが、「.. aaS」の数が増え続けるITの現実は、「VDI-as-a-a-Service」という略語も現れることを示唆しています。 競合ソフトウェア製品のサポートを実装し、オープンおよびプロプライエタリの最大数のオペレーティングシステムとソフトウェアで動作します。 VDIが待ち望んでいること-時間はわかりますが、変換が停止しないことは明らかであり、よりアクセスしやすくなります。 おそらく、「ファット」クライアントが完全にシンクライアントに置き換わるときをキャッチするでしょう。
