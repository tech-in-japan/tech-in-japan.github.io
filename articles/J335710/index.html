Ethereumスマートコントラクト開発の紹介







最近、ブロックチェーン、暗号通貨、スマートコントラクト、および関連技術を中心に前例のない誇大宣伝が増えています。 最も怠zyで受動的な町民でさえ、あらゆる沼地からこれをトランペットしているという感じがします。 IT、サービス、小売、経済の実業界の分野で歴史と経験を持つ膨大な数の新興企業や企業が、活動にブロックチェーンを導入しようと考えており、100,000以上の企業がすでにこれを行っています。



Initial Coin Offeringメカニズムは、クラウドファンディングへの新たな関心の波に貢献しただけでなく、その助けによって引き付けられる投資の量を増やしただけでなく、暗号通貨とブロックチェーンテクノロジーへの関心を刺激しました。 今日、この分野の新しい欠員に出くわすことが多くなり、一般的にICOを実施し、特にスマートコントラクトを開発するための支援を求めています。



残念ながら、今日ではこの分野の専門家が非常に不足していますが、実際にはスマートコントラクトの開発プロセスは一見すると思われるよりもはるかに単純です。 スペシャリストの不足の問題は、ブロックチェーンテクノロジーが暗号通貨ビットコインの形で大衆ユーザーに届き、スマートコントラクトがずっと後にブロックチェーンにやってきたという事実に直接起因しています。 個人的には、スマートコントラクトの開発に関する資料がまだ非常に少ないという事実にはあまり驚きませんでした。



一方、ブロックチェーンベースのデジタル管轄を構築し、プログラミングスキルを持たない一般ユーザー向けのスマートコントラクトを作成するプロセスを簡素化する会社のテクニカルディレクターとして、テクノロジーを理解し、さらに良くなる技術専門家の必要性を感じています-彼女との仕事の経験。



この記事では、スマートコントラクトの開発に入るためのしきい値が実際には非常に低いことを示し、新しい開発者をこの分野に引き付けようとします。 もちろん、人生のすべてがトレーニング資料のように単純であるわけではありません。また、スマートコントラクトの開発には多くの微妙な点があります。 それにもかかわらず、これは多くの人に馴染みのあるJavaScriptを含む他のテクノロジーにも当てはまります。これにより、原則として、実用的な経験が対処に役立ちます。これは、開発者がプロ​​グラミング、設計、実験、テスト、通常は従事しています。



このレッスンでは、開発を開始するために必要なすべてのツールとフレームワークをインストールし、Solidity言語で最初のERC20トークンを作成し、コンパイルし、ローカルブロックチェーンで公開し、いくつかのテストを作成して実行します。 フレームワークtruffleを使用します。これは、移行管理、コンパイル、依存関係管理、およびテストに役立ちます。 また、 OpenZeppelinフレームワークを使用します。 このフレームワークには、 Solidity言語で書かれた一連の契約とライブラリが含まれており、それらの有用性と安全性がすでに証明されています。



はじめに、フレームワークトリュフをインストールしましょう。 これを行うには、フレームワークのドキュメントに移動し、そこでオペレーティングシステムのインストール手順を見つけます。 インストール後、プロジェクト用のフォルダーを作成します。 私の場合、このフォルダーは「jcr-token」という名前になります。その後、ターミナルで作成したプロジェクトフォルダーを開き、 truffle init



コマンドを使用してプロジェクトを初期化します。



次に、お気に入りのテキストエディターでプロジェクトを開きます（私はAtomを使用しています）。 プロジェクトの構造を見てみましょう。 最初は、ルートディレクトリに3つのフォルダーがあります。



  • スマートコントラクトのソースコードを格納するように設計されたコントラクト。
  • 移行 、ブロックチェーンに契約を公開するために使用される移行ファイルを含みます;
  • 契約のテストを含むテスト。 テストはSolidityとJavaScriptの両方で作成できることに注意することが重要です。何かが理解できない場合でも心配する必要はありません。各部分について詳しく説明し、契約、移行、テストの作成方法を学びます。








お気づきかもしれませんが、トリュフはすでにいくつかの簡単なスマートコントラクトを作成しています。 これらを使用してインストールをテストします。



コントラクトと対話してテストを作成するには、まずブロックチェーンで公開する必要があります。 私たちの場合、これらの目的のためにtestrpcと呼ばれる軽いイーサリアムクライアントを使用することをお勧めします。 通常のイーサリアムネットワーククライアントの動作をエミュレートするだけですが、はるかに高速に動作し、消費するリソースが少ないため、優れた開発ツールになります。 インストール後、ターミナルでtestrpc



と入力するだけで、独自の開発ブロックチェーンを開始できます。



正常に起動すると、ブロックチェーンのホストとポートが表示されます。 プロジェクトのルートディレクトリにある `truffle.js`ファイルを開きます。 このファイルには、プロジェクトの構成が含まれています。 構成内のホストとポートが、実行中のtestrpcのホストとポートと一致していることを確認してください。



では、 truffle compile



入力して、スマートコントラクトをコンパイルしてみましょう。 この手順により、ビルドディレクトリにファイルが生成されます。 コンパイルを初めて実行すると、ディレクトリが自動的に作成されます。 ビルドアーティファクトはこのフォルダーに配置されます。 それらについては少し後で説明します。









契約をテストするには、 truffle test



ます。 JavaScriptまたは他の言語の単体テストに精通している場合は、スマートコントラクトをテストする方法を既に知っています。 ブロックチェーンのエラーの価格は非常に高いため、テストには特に注意してください！ テストを実行する前にコントラクトをプリコンパイルする必要さえありません。`test`コマンドがこれを自動的に行うためです。









最後に、testrpcブロックチェーンで契約を公開する移行を実行するには、 truffle migrate



ます。









その後すぐに、ターミナルにコマンドの出力が表示され、testrpcクライアントにイベントの出力が表示されます。 おめでとうございます！ 最初のスマートブロックチェーンネットワークエミュレーター契約を公開しました。









より高度なコントラクトを作成するには、OpenZeppelinフレームワークを使用します。これには、いくつかの非常に便利なライブラリ、インターフェイス、既製のコントラクトが含まれています。 Solidity言語は継承をサポートしており、既存のコントラクトの機能を簡単に拡張できることに注意してください。 これにより、アーキテクチャを熟考する大きな機会が開かれます。



お気に入りの依存関係マネージャー（npmやyarnなど）を使用して、他のJavaScriptライブラリと同じようにOpenZeppelinをインストールできます。 詳細については、githubのOpenZeppelinプロジェクトページまたはそのWebサイトを参照してください。



そのため、独自のスマートコントラクトの開発を開始するために必要なものをすべてインストールしました。 そして、それほど難しくなかったでしょう？



それでは、テキストエディタに進み、プロジェクトファイルを見てみましょう。 トリュフが初期化中にいくつかの基本的なコントラクトをすでに作成していることに気付くかもしれません。 移行ファイルを除き、これらすべてのファイルを削除しましょう。 余分なコントラクト、ライブラリ、テストを削除し、移行からいくつかのコードも削除します。



カスタムスマートコントラクトのコードを使用して、新しいソリッドファイルを作成します。 Jincor ICOを実行するためにJCRというトークンを作成します。つまり、ファイルにJCR.solという名前を付けます。



Solidityは、かなり新しくダイナミックなプログラミング言語です。 これにより特定の不便が生じますが、その1つは頻繁な変更であり、その一部は後方互換性を破壊し、既に公開されている契約で問題を引き起こす可能性があります。 これを防ぐには、Solidity言語コンパイラのバージョンを指定する必要があります。 個人的には、 `^ 0.4.11`を使用します。



ここで `^`は、厳密に指定されたバージョン0.4.11ではなく、新しいマイナーバージョンとバグ修正にも満足していることを意味します。 npmで行うように、はるかに柔軟なバージョン管理パターンを定義できます。



すでに述べたように、OpenZeppelinライブラリの一部のコントラクトを使用して、車輪を再発明するのではなく、コミュニティのベストプラクティスを含み、定期的にセキュリティ監査を実施する、作業用の安全なプラクティスを使用します。



コントラクトをインポートするには、キーワード `import`を使用し、引用符でインポートしたファイルへのパスを示す必要があります。たとえば、次のようにします。`import" zeppelin-solidity / contracts / token / MintableToken.sol ";`



OpenZeppelinのドキュメントを最初から最後まで読み、スマートコントラクトの実装を確認して、どのように機能するかを明確に把握することを強くお勧めします。 さらに、調査中にコード設計の例を見ることができ、新しい契約を書くためのインスピレーションを見つけることができます。 ソースはnode_modulesディレクトリにあります。



今日使用するMintableトークンを見てみましょう。 私の助けなしにソースコードとドキュメントに慣れることができるので、ここでは詳細に深く掘り下げません。 最も重要なものだけを説明します。



「MintableトークンはOwnable、Standardトークン」であることがわかります。 「is」キーワードは、JavaまたはPHPの「extends」とほぼ同じことを意味します。 Mintableトークンコントラクトは、2つのイベント、1つのパブリックプロパティ、2つの修飾子、および2つの関数を追加します。 これにより、トークンを発行する機能が構成されます。 標準トークンは、以前に取得した許可を持つ別のユーザーに代わってトークンを転送する機能を追加します。 基本トークンは、トークン転送とバランスチェックを定義するERC20Basicインターフェイスの単なる実装です。 OwnableコントラクトはonlyOwner修飾子を追加します。これは通常、サードパーティの関数呼び出しを制限するために使用されます。



次に、契約のソースに戻り、少し変更します。 必要なすべての契約をインポートした後、独自の契約JCRを呼び出します。 契約ファイルと名前に同じ名前を付けてください。



契約では、公開されている名前、記号、小数点以下の桁数（Ethereumの場合は18に等しい）を定義します。 



 contract JCR is MintableToken { string public name = "Jincor Token"; string public symbol = "JCR"; uint public decimals = 18; }


次に、コンストラクタコードを記述します。 コンストラクターは、コントラクトのように呼び出される単なる関数であり、コントラクトオブジェクトがインスタンス化されるとき、つまり新しいインスタンスが作成されるときに呼び出されます。 これは初期化コードのための非常に良い場所です。 個人的には、指定された数のトークンを発行し、すべてのトークンを契約作成者の残高に転送できる契約を作成します。 これを行うには、コンストラクタにamount引数を追加し、契約の作成者を所有者として指定し、指定された数のトークンを所有者のウォレットに発行します。 



 function JCR(uint256 _amount) { owner = msg.sender; mint(owner, _amount); }


結果のJCR.solファイル（githubでも入手可能） 



 pragma solidity ^0.4.11; import "zeppelin-solidity/contracts/token/MintableToken.sol"; contract JCR is MintableToken { string public name = "Jincor Token"; string public symbol = "JCR"; uint public decimals = 18; function JCR(uint256 _amount) { owner = msg.sender; mint(owner, _amount); } }


次に、取得したものをコンパイルしてみましょう。 これを行うには、「truffle compile」と入力します。 ビルドフォルダーを調べて、出力で得られたものを確認しましょう。 最初に目を引くのは、ERC20からMintableおよびJCRに使用されたすべての契約のアーティファクトを受け取ったことです。 アーティファクトはJSONファイルに保存されます。 これらのJSONファイルには、コントラクトの名前、Application Binary Interface（abi）、イーサリアム仮想マシンで後で起動されるバイナリコード、および追加情報が含まれています。















アーティファクトを使用して、スマートコントラクトをブロックチェーンに公開できます。 ただし、最初から良いものに慣れて、移行を追加することでこのルーチンプロセスをすぐに自動化することをお勧めします。 発行されたトークンの数をコンストラクタに渡す必要があることを覚えていますか？ 移行は、このような操作に適した場所です。 2_deploy_contracts.jsファイルを開いて、コードを追加しましょう。 140万トークンを発行します。



deployer.deployメソッドを使用して移行スクリプトでコントラクトを公開できます。このメソッドには、最初の引数としてビルドアーティファクトを渡し、その後に同じ順序でコントラクトコンストラクターに渡される引数を渡します。 この場合、引数は1つだけ-1,400,000で、残りはトリュフが処理します。 展開プロセスをより簡単で楽しいものにするために、アカウントのロックを解除できます（アカウントに代わってアクションを実行できます）。 これを行うには、testrpcを起動するときに、引数 `-u 0`を追加します 



 var JCR = artifacts.require("./JCR.sol"); module.exports = function(deployer) { const tokenAmount = 1400000; deployer.deploy(JCR, tokenAmount); };


新しい移行を開始し、機能するかどうかを確認します。 いいね！ 次に、トークンが期待どおりに機能することを確認する必要があります。 これを行うために、JavaScriptでいくつかのテストを作成します。 「truffle create test」とテスト対象のコントラクトの名前を入力して、テストでJavaScriptファイルを生成します。 テストはSolidityでも作成できますが、これについては後で説明します。



PHPなどの他のプログラミング言語でテストを作成した経験があることを願っています。 モカとチャイのフレームワークに既に出会ったことがあるなら、トリュフがそれらを使用しているので、すべてはあなたに馴染みがあるように思えます。 詳細については、公式ドキュメントをご覧ください。



最後に、契約の作成時に所有者が正しい数のトークンを受け取り、トークン転送機能が期待どおりに動作することを確認してみましょう。 



 var JCR = artifacts.require("./JCR.sol"); contract('JCR', function(accounts) { it("Create 1 400 000 tokens at the owner account", function(done) { JCR.deployed().then(function(instance) { return instance.balanceOf.call(accounts[0]); }).then(function(balance) { assert.equal(web3.toWei(balance.valueOf(), 'ether'), web3.toWei(1400000, 'ether'), "1400000 wasn't in the first account"); }); done(); }); it('Should transfer tokens correctly', function(done){ var token; var amount = 10; var account_one = accounts[0]; var account_two = accounts[1]; var acc_one_before; var acc_one_after; var acc_two_before; var acc_two_after; JCR.deployed().then(function(instance){ token = instance; return token.balanceOf.call(account_one); }).then(function(balanceOne) { acc_one_before = balanceOne.toNumber(); return token.balanceOf.call(account_two); }).then(function(balanceTwo) { acc_two_before = balanceTwo.toNumber(); return token.transfer(account_two, amount, {from: account_one}); }).then(function() { return token.balanceOf.call(account_one); }).then(function(balanceOne){ acc_one_after = balanceOne.toNumber(); return token.balanceOf.call(account_two); }).then(function(balanceTwo){ acc_two_after = balanceTwo.toNumber(); assert.equal(acc_one_after, acc_one_before - amount, "Token transfer works wrong!"); assert.equal(acc_two_after, acc_two_before + amount, "Token transfer works wrong!"); }); done(); }); });
