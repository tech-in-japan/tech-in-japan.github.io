プロキシ



グローバルに、彼らがプロキシサーバーと言うとき、それらはクライアントと受信者の間の仲介者として機能する何かを意味します。

匿名性の確保に関して、プロキシサーバーは次のとおりです。

  • HTTP（Web）プロキシサーバー。 このようなサーバーはHTTPトラフィックのみを許可し、デフォルトでは送信トラフィックにプロキシ使用状況データを追加します。
  • SOCKSプロキシ。 HTTPプロキシとは異なり、SOCKSは自身から何も追加せずにすべての情報を転送します。 SOCKSプロトコルはOSIモデルのセッションレベルにあります。これにより、HTTP、FTP、POPZなどの高レベルプロトコルからの独立性が確保されます。これにより、SOCKSはHTTPだけでなくすべてのトラフィックを通過できます。
  • それとは別に、CGIプロキシまたは「アノニマイザー」に言及する価値があります。これらは基本的に、クライアントが目的のサイトのアドレスを入力するフォームを持つWebサーバーです。 その後、要求されたリソースのページが開きますが、CGIプロキシのアドレスはブラウザーのアドレスバーに表示されます。 CGIプロキシは、他のWebサーバーと同様に、httpsを使用して、自身とクライアント間の通信チャネルを保護できます。


写真にプロキシサーバーのスキームが表示されていますが、ここではすべてが簡単です。





プロキシサーバーの利点：

  • プロキシは安価で、ネット上で多くの無料プロキシを見つけることができます。


プロキシの短所：

  • 信頼プロキシサーバー
  • httpプロキシの場合、HTTPヘッダーをフィルタリングする必要があります。「HTTP_X_FORWARDED_FOR：client、ip1 ...」、HTTP_VIA、HTTP_FORWARDEDなど。
  • プロキシプロトコル（http、SOCKSx）は、HTTP / SOCKS / Elite / Anonymousプロキシとクライアント間の暗号化をサポートしていません。 SSLプロキシとは、クライアントがhttpsリソースを操作できることのみを意味します。
  • プロキシチェーンは無効です。「 こんにちはProxy1、メッセージを送信：」Proxy3に転送、Proxy4に転送、encrypted.google.com / c8e8df895c2cae-anything-else-here-here- encrypted -166baf 'For Proxy2？ "ありがとう！
  • アプリケーションごとにプロキシサーバーを構成する必要があるか、Proxifierなどの個別のsocksifierプログラムを使用する必要があります。


VPN / SSH



VPN、つまりSSHトンネルについても説明します。 なぜなら、いくつかの違いはありますが、基本的な原則は彼らにとって同じです。

VPN操作スキームを図に示します。



現在、次のVPNが商用プロバイダーによって提供されています。

  • PPTP-最も広く使用されており、高速で、構成が簡単ですが、他と比較して「最も安全性が低い」と考えられています。
  • L2TP + IPSec。 L2TPはトランスポートを提供し、IPSecは暗号化を担当します。 このバンドルには、PPTPよりも強力な暗号化機能があり、PPTPの脆弱性に対する耐性があり、メッセージの整合性と関係者の認証も提供します。
  • OpenVPN-安全で開かれているため、多くのロックをバイパスできますが、別個のソフトウェアクライアントが必要です。
  • SSTPはOpenVPNと同じくらい安全で、別のクライアントを必要としませんが、Vista SP1、Win7、Win8のプラットフォームでは非常に制限されています。


ほとんどすべての商用VPNプロバイダーは、OpenVPNとPPTPの2つのプロトコルから選択できます。 あまり一般的ではないが、L2TP + IPSecプロトコルが提供されています。 また、かなりの数がSSTPを提供しています。

別に、DoubleVPNを提供するサービスは、インターネットに行く前にトラフィックが異なる国の2つの異なるVPNサーバー、または4つのサーバーが使用されている場合にQuadVPNを通過することに注意する価値があります。いいよ

商用VPNサーバーの匿名性と信頼性に関する興味深い研究がtorrentfreak.comによって実施されました： torrentfreak.com/vpn-services-that-take-your-anonymity-seriously-2013-edition-130302

VPNプロバイダーに質問がありました：

  • あなたや第三者がクライアントとIPアドレスまたはタイムスタンプをマッピングできるようにログを保持していますか？ その場合、どのデータを保存しますか？
  • あなたの会社はどの司法管轄下で、どのような状況下で第三者にデータを開示しますか？
  • DMCA通知またはその欧州版に相当するものを受け取った場合、それをどうしますか？
  • どの支払いシステムを使用しますか？また、ユーザーアカウントとどのように関連していますか？


要約すると、回答のほとんどのVPNプロバイダーは満場一致です。「ログは保存されません。保存されている場合、非常に簡単に言えば、それらから加入者を計算することは不可能です。 私たちにプレッシャーをかけ、少なくとも何かを出すようにすることは非常に困難です。」 もちろん、ユーザーの匿名性を確保することを主な目的とするサービスから他の回答を期待すべきではありません。



VPN / SSHの利点：

  • 高速かつ便利で、アプリケーションを個別に構成する必要はありません。


VPN / SSHの短所：

  • VPN / SSHサーバー/プロバイダーを信頼する必要があります。


ブラウザと「匿名用プログラム」向けのテーマアドオンのほとんどは、プロキシサーバーとVPNサーバーを使用してクライアントのIPアドレスを隠していることに注意してください。



Tor 素晴らしくてひどい



Torについては多くのことが言われていますが、私はあなたに簡単に伝えようと思います:)

Torは、クライアントがノードのチェーンを介してインターネットに接続するルーターシステムです。 原則として、チェーンは3つのノードで構成され、各ノードはクライアントとリソースのアドレスを同時に認識しません。 さらに、Torはノードごとにメッセージを個別に暗号化し、オープントラフィックは出力ルーターにのみ表示されます。

現在、Torは10の信頼できる（制御）ノード、約900の出力ノードを含む約4200の中間ノードです。

写真では、Tor操作の簡略化されたスキーム



トラフィックはオープンフォームに戻り、出力ノードでは一時的な対称キーで暗号化され、チェーンに沿って送信されます（はい、トラフィック自体は対称キーで暗号化され、これらのキーは非対称キーで暗号化されます）。

彼らは彼からあまりにも多くを要求しているので批判しています：アプリケーションのトラフィックをネットワークに転送すること、グローバルなオブザーバーから保護すること、送信データの機密性などは安全です。しかし、彼は脅威モデルの主な問題を解決します： http-のみを送信する場合のクライアントの匿名性の高さすべての必須ルールに従うトラフィック ： www.torproject.org/download/download-easy.html.en



Torの長所 ：

  • すべてのルールに従い、高度な顧客匿名性。
  • 使いやすさ（Tor Browser Bundleをダウンロードし、起動して使用）。


短所Tor：

  • 出力トラフィックがタップされます。
  • 低速;
  • 管理サーバーの可用性。


フォーラムの1つで、Torに関するアンケートを見つけました。 回答者の数は結果の信頼性を示すものではありませんが、勝利の答えは非常に正しいです:)

I2P



I2Pについて多くの言葉が言われていますが、私は簡潔にすべてを明確に説明しようとします。

I2Pは、インターネット上で実行される匿名ネットワークです。 独自のサイト、フォーラム、その他のサービスがあります。 そのアーキテクチャにより、完全に分散化され、I2PのどこでもIPアドレスは使用されません。

I2Pには2つの主要な概念があります。

  • 「トンネル」は、ノードのリストを通る一時的な単方向パスです。 トンネルは着信と発信です。
  • 「NetDbネットワークベース」。これは、すべてのI2Pクライアントに多かれ少なかれ分散しています。 その目的は、クライアントが特定の受信者に接続する方法に関する情報を保存することです。


NetDbデータベースには以下が含まれます。

  • RouterInfos-ルーター（クライアント）の連絡先の詳細は、トンネルを構築するために使用されます（単純化、各ノードの暗号化識別子です）。
  • LeaseSets-発信および着信トンネルの通信に使用される受信者の連絡先の詳細。


2013年の初めには、I2Pには25,000のルーターと3,000のLeaseSetが含まれていました。

ノードインタラクションアルゴリズムについて説明します。

最初のステップ。 ケイトノードは発信トンネルを構築します。 彼女はルーター上のデータをNetDbに頼り、参加してトンネルを構築します。



ステップ2 ボリスは、発信トンネルと同じ方法で入力トンネルを構築します。 その後、彼は自分の座標またはいわゆる「LeaseSet」をNetDbで公開します（ここでは、LeaseSetが発信トンネルを介して送信されることに注意してください）。



ステップ3 ケイトがメッセージをボリスに送信する場合、ボリスにNetDb LeaseSetを要求します。 そして、発信トンネルを介して、宛先ゲートウェイにメッセージを転送します。



I2Pには特別なOutproxyを介してインターネットにアクセスする機能がありますが、それらは非公式であり、要因の組み合わせで、Tor出力ノードよりもさらに悪いです。 I2P開発者は、「 インターネットが欲しい -Torを使用してください」と言います。



I2Pの長所：

  • 高度な顧客匿名性;
  • ネットワークの安定性につながる完全な分散化。
  • データの機密性：クライアントと受信者間のエンドツーエンドの暗号化。


I2Pの短所：

  • 低速;
  • 「あなたのインターネット。」
