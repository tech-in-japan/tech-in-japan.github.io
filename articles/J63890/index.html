Ubuntuでのext4の問題

主なことについて簡単に説明すると、ubuntu jauntyのext4は不安定です。 比較的小さな空き領域（数GB）を含む特定の状況では、大きなファイルまたは多数のファイルを削除しようとすると、システムがフリーズします。 「リセット」と呼ばれる聴覚障害凍結。 対応するバグレポートがあります。 しかし、それは治療可能です。



私は、おそらくここに集まった多くの人たちと同じように、ずっと前に私の唇をなめ、ext-ext4ファイルシステムの新しいバージョンを歩き回りました。 ext4に切り替えるように私を押した主な理由の1つは、ext3のパーティションから大きなファイルを削除するときの遅れでした。 長い間、私はkubuntu 8.04に座っていましたが、それでもアップグレードすることにしました。 アップデート中に、私は多くの不思議と冒険を見ました。 いくつかの問題を解決するために、いくつかのアップデート、kdeのいくつかのバージョン（4.2、4.2.9、4.3、その後kde3へのロールバック）を再配置しました。その結果、/ var / cache / apt / archivesに適切なサイズのアーカイブがありました。 寸法-2ギガバイト未満、数百のファイル。



そのため、このディレクトリが大きくなるほど、不愉快な不具合に遭遇し始めました-パッケージを一括でインストール/更新しようとすると、鈍いフリーズが発生します。 kdeのあるバージョンから別のバージョンにアップグレードするときにシステムがどのように変化するかを想像できますが、その間にシステムがフリーズします。



もっともっと。 sudo apt-get cleanを実行して100％の保証でキャッシュをクリアしようとすると、即座にフリーズしました。 さらに、これらのファイルの手動削除もハングアップして終了しました。



グーグル検索の後、このバグレポートが発見され、現在のところ、デフォルトでカーネルがインストールされたubuntu jauntyに基づくすべてのマシンが影響を受けます（現在は2.6.28-13-genericです）。



治療法があります。 今日から、そのうちの2つがあります。
  1. 前述のバグレポートからわかるように、修正はすでに提案済みに送信されており、すでにコンパイル中です。 必要なことは、jauntyで提案された更新を有効にして、更新を待つことだけです。 これが役立つかどうかはわかりません、なぜなら これが機能することを確認する必要があります。 更新：判明したように、このメソッドは機能しません。 問題を修正しなかったアップデート。
  2. 2番目のオプションは、ここからなど、2.6.29ブランチから新しいカーネルを手動でインストールすることです 。 この方法の利点は、多くのグラフィックチップがはるかに優れていることです。
ここでの結論は次のようなものです。Jauntyで使用されているカーネルはやや問題があることが判明しました。 これらは、Intelカードの問題とext4の問題です。 適切な更新をインストールせずにext4を使用すると危険です。 したがって、 提案された更新を含めるか、 ext4を使用して待つか、カーネルを更新することを強くお勧めします。
