最初に、 EHcacheについてもう一度説明します。 これは、Java用の高性能でスケーラブルなキャッシュシステムであり、成熟した深刻なプロジェクトです。 キャッシュオプションは、RAMとディスクキャッシュの両方で使用でき、また組み合わせた戦略でも使用できます（仮想マシンまたはサーバーの再起動時にデータの安全性を確保するオプションもあります）。 スケーラビリティは、 JGroup 、JMS、RMIを使用した非同期レプリケーションとキャッシュクラスタリングを使用して実装されます。サードパーティ製品（Terracota）に基づいて分散システムを構築することもできます。 私は何よりも分配性が好きです-各キャッシュの個別の設定（同期/非同期レプリケーション）と、同じJVM（または異なる）内でいくつかの異なるインスタンスを実行する機能との組み合わせ。 EHCacheはデータをJVMプロセスのメモリに格納することに注意する必要がありますが、それに応じて、そのボリュームにいくつかの制限が課せられます（32ビットシステム上）が、誰もディスクキャッシュをキャンセルせず、深刻なサーバーはすでに64ビットです。 20 GB以上のデータがある既知のインストール。 さらに、マルチスレッド機能、競争力のあるアクセス、マルチコアの利用は十分にサポートされています（ここで少し議論することができますが、 JBoss Cacheはトランザクションやその他の利点がサポートされているため、これでさらに優れているようですが、学習とそのAPIは理解するのが非常に複雑であり、EHcacheがすぐに開始された一方で、2つの方法でそれを理解することはできませんでした。



キャッシュは非常に高速で、ネットワークで利用可能なテストで他のキャッシングシステムよりも優れています（ただし、4台のマシンのクラスターではJBoss Cacheが若干良い結果を示しますが、これは少し人気のあるMemcachedを含む）。 EHcacheはインプロセスキャッシュであり、memcachedは独立したデーモンであり、外部ネットワークサービスとして機能するため、比較が少し間違っていることを知っています（したがって、このようなキャッシュサイズの制限はありません）。 同時に、ehcacheのほうが柔軟性とスケーラビリティがはるかに優れているため、メモリ/ディスクと微調整されたキャッシュ設定の組み合わせにより、ehcacheが望ましいと比較するとします。 ここで私は思考に訪れました... memcachedの代わりに（または一緒に）EHcacheを使用することは可能ですが、使い慣れたPHPプラットフォームのままですか？ はい、できます！



問題は、キャッシュ自体に加えて、開発者のコ​​ミュニティもキャッシュRESTサーバーを実装し、RESTインターフェイスまたはSOAPを通じてアクセスできることです。 このようなソリューションは、組み込みバージョンのSun GlassFish v3 Preludeサーバーに基づいて作成され、必要なすべてのコンポーネントと依存関係を含めて自給自足です。 サーバーでの作業は、GET / POST / PUT / DELETE / OPTIONS / HEADメソッドを使用したHTTP経由、またはXML経由のSOAP（HTTPの上部）経由で行われます。 キープアライブ、Last-Modified、ETagなど、すべてのHTTP / 1.1機能がサポートされています。つまり、サーバーはすべての正しいヘッダーを提供します。したがって、送信中またはクライアント自体で中間ノードの組み込みキャッシュを使用できることがよくあります。 興味深い点は、いくつかのデータ形式を処理する機能、正確には、要求に正しいMIMEタイプヘッダーを設定するのに十分なXMLまたはJSON形式で回答を取得する機能です。

メインサーバーファイルを指定した後、それが使用可能になるポート番号と、Webアプリケーション（warファイル）のあるディレクトリへのパスがあります。 起動後、コンソールで接続の進行状況と、実行中のサービスとポートに関する情報が表示されます。たとえば、この方法でJMXサービスが管理に使用できるポートを確認できます。 GlassFishサーバーの組み込みバージョン（これはWebサーバーとアプリケーションサーバーの両方）が使用されるため、その設定と機能は非常に小さくなりますが、GlassFishでなく、本格的なサーバーをいつでもデプロイでき、EHcacheサーバーのみを使用できます。個別に、Webサーバーなし。



デフォルトでは、キャッシュは/ ehcache / restで利用できます-ブラウザにアクセスするか、GETリクエストを実行することにより、現在のすべてのキャッシュ設定の説明を含むXMLドキュメントを取得します。 最初に、構成ファイルには、いくつかのキャッシュの説明が含まれています（たとえば、いくつかのキャッシュが含まれています）。 開始するには、すべての基本設定を削除し、独自のキャッシュを作成することをお勧めします。 レプリケーションのない単純なキャッシュは、ここで行います。



すべてのキャッシュ設定は、編集する1つのxml-file /war/WEB-INF/classes/ehcache.xmlに集中しています。 内部にはすべてのオプションのコメントと説明がたくさんあるので、実験を続けるために基本的なキャッシュを作成する方法について簡単に説明します。



これらのオプションの意味：

  • name-アクセスに使用されるキャッシュの名前（URL内にあるため、コンテンツに同じ制限が課せられます。何よりも-保存されたデータの種類を短く明確に示す）。 単一のサーバーは、異なる名前の多くのキャッシュを持つことができます。
  • maxElementsInMemory-メモリに配置される要素の最大数
  • maxElementsOnDisk-ディスク上の要素の最大数（0-制限なし）
  • eternal-キャッシュライフ設定を無視できることを示します。手動で削除するまで、要素は常にキャッシュ内にあります
  • overflowToDisk-メモリ内のオブジェクトの最大数に達した場合にアイテムをディスクにプッシュできるかどうかを示します
  • timeToIdleSeconds-オブジェクトへの最後のアクセスから無効と宣言されるまでの時間（オブジェクトが永遠としてマークされていない場合）。 オプションのパラメーター
  • timeToLiveSeconds-要素の有効期間（バージョン1.6以降、間違えなければ、このオプションは個々のキャッシュオブジェクトごとに設定できます）
  • diskPersistent-再起動間のディスク上のキャッシュステータスの保存を示します
  • diskExpiryThreadIntervalSeconds -TTL（有効期間）の有効期限についてディスク上のオブジェクトをチェックするプロセスを開始する頻度。
  • diskSpoolBufferSizeMB-ディスク書き込みをバッファリングするためにキャッシュに割り当てられるプールのサイズ。 プールがいっぱいになると、キャッシュステータス書き込みコマンドが非同期的に呼び出されます
  • memoryStoreEvictionPolicy-どのキャッシュオブジェクトをディスクにプッシュするかを決定するための戦略を示します。 LRU（Least Recently Used）、最終使用日、FIFO（先入れ先出し、最初の追加、最初の混雑）、およびLFU（使用頻度の低い、使用頻度）


レプリケーションオプションについてはまだ説明していません。これは詳細な特異性であり、有能なスペシャリストが詳しく説明する必要があります。







  1. < キャッシュ 名 = "testRestCache"
  2. maxElementsInMemory = "10000"
  3. eternal = "true"
  4. timeToIdleSeconds = "0"
  5. timeToLiveSeconds = "0"
  6. overflowToDisk = "true"
  7. diskSpoolBufferSizeMB = "4"
  8. maxElementsOnDisk = "1000000000"
  9. diskPersistent = "true"
  10. diskExpiryThreadIntervalSeconds = "3600"
  11. memoryStoreEvictionPolicy = "LFU"
  12. />
*このソースコードは、 ソースコードハイライターで強調表示されました。




したがって、キャッシュは、メモリ内に1万個の要素、ディスク上に10億個の要素を保持するように構成され、再起動間のデータの整合性を確保するために、ライフタイム設定を使用しません。 私が選択したディスク記録用のプールのボリュームは非常に小さく、ディスク要素の寿命をチェックする時間は非常に長いです（しかし、もっと理想的には必要だと思います-無効にできるかどうか見てみましょう）。 この構成は何のためですか？ 私は、このキャッシュに基づいて単純なキー値データベースを作成することに興味があります（今日、これは非常に人気のあるトピックです）が、外部サービスおよびPHP Webアプリケーション内からキャッシュに直接アクセスできるようにします。 注意点-ライフタイムチェックが不要で、一定のキャッシュが必要な場合でも、timeToIdle / timeToLiveパラメーターを0に設定しないでください。そうしないと、サーバーが起動しない場合があります（キャッシュサービス、サーバーが起動して404エラーが発生します）。



確認するには、編集したehcache.xmlファイルを保存し、サーバーを再起動します。 ブラウザのURLを開きます： localhost ：8080 / ehcache / rest / testRestCache-すべてのキャッシュ設定、および現在のキャッシュ使用統計（ボリューム、データ量、ヒットとミスの割合）を含むXMLドキュメントを取得する必要があります-将来、これは逆アセンブルできますプログラムで目的の形式で表示する（たとえば、管理パネル）。



将来的には、RESTパートのみを検討します。SOAPを使用するには、残りのURLをsoapに変更したり、WSDL形式でサービスの説明を取得したりする必要があります。 パフォーマンスのために、不要なキャッシュやSOAPプロトコル経由のアクセスなど、未使用のものをすべてオフにしました。 サーブレット設定は、 / war / WEB-INFディレクトリのweb.xmlファイルで利用できます。



キャッシュの操作は、HTTPプロトコルを介して要求を送信し、応答を解析することで構成されます。 エラーの場合、応答はテキスト/プレーン形式で、要求本文にはエラーテキストが含まれ、HTTPコードは404になります。たとえば、存在しないキャッシュまたは要素に変更した場合、応答は「Element not found：333」行になります（キー333のアイテムをリクエストしました）。 しかし、これはEHcacheサーブレットによって提供されるURLに当てはまります。エラーが別の部分にある場合、GlassFishから標準の404エラーページを取得します。これは自動解析にはあまり適していません。



サーバー全体で（キャッシュマネージャーを使用して）作業することも、キャッシュと要素ごとに個別に作業することもできます。そのためには、URL行を追加して、パラメーターで目的のメソッドを使用します。



キャッシュ全体（CacheManager）の場合：

  • OPTIONS-メソッドは、使用可能な操作の説明をWADL形式（ Webアプリケーション記述言語 ）で返します


  • GET-サーバーで使用可能なキャッシュとそのパラメーターのリストをXML形式で返します。 これは、上で行ったように、ブラウザを介した通常のリクエストです。たとえば、 localhost ：8080 / ehcache / rest /












階層のさらに下で、URLに特定のキャッシュ名を指定すると、次の操作を実行できます。

  • OPTIONS-上記と同様、利用可能な操作のWADL記述を返します
  • HEAD-キャッシュパラメータの説明とともに同じメタデータを返しますが、HTTPヘッダーの形式であり、応答本文ではありません（GETなど）
  • GET-キャッシュパラメーターとその統計を含むXMLドキュメント。
  • PUT-デフォルトのキャッシュ設定（構成ファイルで指定）に基づいて、新しいキャッシュ（URL文字列で名前が指定されている）を作成できます。
  • DELETE -URLで指定されたキャッシュを削除します。 削除はされますが、クリアされません（キャッシュ要素に対する操作の中で、これには別のコマンドがありますが、奇妙なことに）、次のサーバーのリブートまではそうです（ただし、この点はまだ確認していません）。


キャッシュ要素レベルでは、次の操作がサポートされています。

  • OPTIONS-上記と同様、利用可能な操作のWADL記述を返します
  • HEAD-要素の内容をHTTPヘッダーの文字列として返します（ここではヘルプにあいまいさがあります。他の場合はHEADがGETを複製するため、キャッシュ要素については値ではなくメタデータを返すことが示されています）。
  • GET-応答本文のキャッシュ要素のコンテンツを直接返します。
  • PUT-データをキャッシュに入れます。 データ自体はリクエスト本文で送信され、URLのオブジェクト名、およびオプションのパラメーター、lifetimeは、「ehcacheTimeToLiveSeconds」という名前のHTTPヘッダーで渡すことができます。パラメーターがない場合、キャッシュの説明のパラメーターが使用され、有効な値の間隔-0（永久）... 2147483647（約69歳）。
  • DELETE-指定されたアイテムを削除します。 キャッシュのすべての要素を削除する必要がある場合は、マスク*を使用します。他のメソッドがこれをサポートしていないのは残念です（つまり、キャッシュ自体はJavaAPIで完全にサポートしていますが、RESTレベルではマルチgetはありません）。


アイテムをキャッシュに保存するとき、MIMEタイプを（サポートされているもののリストから）指定できます。抽出すると、必要なデータがすぐに取得されます。 以下がサポートされています。

  • テキスト/プレーン-プレーンテキストまたは任意のデータ
  • text / xml- RFC 3023に準拠したXMLドキュメント
  • application / json-最も興味深いJSON形式（ RFC 4627に準拠）
  • application / x-java-serialized-object-シリアル化されたJavaオブジェクト


実際、これはサーバー自体の全体的な説明です。今では、実用的な部分は、PHPのWebアプリケーションからサーバーを操作する方法です。 最初のアイデアは、Mendcachedのクラスと同様に、 Zend Frameworkの特別なキャッシュバックエンドを作成することでしたが、最初はすべての動作を実験することにしました。 おそらく、私以外の誰にとっても興味深く有用なものであれば、そのようなクラスを書くでしょう。

残りのコマンドは同じ方法で指定できます。 最初に少し制限されますが、Zend_HttpはHEADリクエストをサポートしていませんが、通常は他のリクエストを複製するため、それらのリクエストはあまり必要ありません。 2番目の不便-キャッシュまたは特定の要素に関するメタデータはXML形式で提供されますが、要素はJSONでも使用できます。 統計はすべてのデータとともに提供されますが、別のページに配置することをお勧めします。 3番目の不便-データを抽出および追加する機能が開発されていません。 すぐにいくつかの要素を配置または要求することはできません（ただし、Java API自体はそうです）。 しかし、一度にすべてを削除することはかなり可能です。 さて、セキュリティは決して保証されないので、外部にHTTPアクセスで機密データを保存しないでください。
