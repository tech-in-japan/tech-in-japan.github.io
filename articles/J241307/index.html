Microsoft DocumentDB：記事2、リソース、および概念

最初の記事で既に述べたように、DocumentDBはその機能へのアクセスをRESTfulプログラミングモデルの形式で公開し、データベース内に格納されたエンティティはリソースと呼ばれ、URIによってアドレス指定されます。 これらのリソースにアクセスするには、標準のHTTP動詞、ヘッダー、ステータスコードを使用できます。

DocumentDBの良い例（簡単で思慮深い問題）を準備し、最初の記事への質問への回答をしている間、DocumentDBが実行されるリソースと概念についてもう少し読むことをお勧めします。









DocumentDBリソースモデルは、アカウント内の特定の構造に格納されたリソースのセットで構成され、各リソースには定数URIでアクセスできます。 そのため、すべてはDocumentDBアカウントで始まります。 アカウントは、データベースが格納される論理コンテナです。各データベースにはコレクションが含まれ、各コレクションには、ストアドプロシージャ、トリガー、UDFなどが含まれます。各データベースには、ドキュメントを操作する一連の権限を持つユーザーがいます。 許可はトークンのように見え、コレクションはJSONドキュメントとJSのロジックのコンテナです。



システムリソース（アカウント、データベース、コレクション、ユーザー、ストアドプロシージャ、トリガー、UDF）には固定スキームがあり、ドキュメントと添付ファイルにはスキームに対する制限がないため、ユーザーリソースと呼ばれます。 両方のタイプのリソースはJSONによって記述されます。







1つのAzureサブスクリプション内に多数存在する可能性のある各アカウントは、SSDストレージと固定帯域幅インジケーターを組み合わせたユニットで構成されるプレハブコンテナーです。 ユニットはいつでも追加または削除できます。 アカウント設定は、Microsoft Azure管理ポータル（ portal.azure.com ）またはREST APIを使用して作成および変更できます（プラットフォームの機能の大部分はREST API管理用に公開されています）。

アカウントが最上位の論理コンテナである場合、データベースはコレクションコンテナおよびユーザーです。 アカウントには、好きなだけデータベースを含めることができます。







データベースには、数ギガバイトからペタバイトまで、必要なだけのデータを保存できます。このストレージはすべて、固定帯域幅のSSDで動作します。 ただし、データベースは単一のマシン内で固定されていません。数千のコレクションとテラバイトのドキュメントを格納する大規模なデータベースである場合があります。



コレクションは、次のレベルのネストのコンテナであり、すでにJSONドキュメント用です。 コンテナとしてのコレクションは、統合だけでなく、トランザクションとクエリのスケーリングの単位としても機能します。 最も簡単な拡張方法は、コレクションを追加し、それらにSSDストレージを分散することです。 自動スケーリングはすでに機能しています-ドキュメントが追加または削除されると、コレクションのサイズが自動的に変更されます。 DocumentDBはプレビューであり、操作モードは1つ（標準プレビュー）のみですが、コレクションをスケーリングできる最大サイズは10 GBです。



自動インデックス





DocumentDBでは、システムのスキーマをまったく計画する必要はありません。 ドキュメントはその存在を意味するものではなく、ドキュメントをコレクションに追加するとすぐに、DocumentDBは自動的にインデックスを付けます（=>クエリを実行できます）。 DocumentDBの主な機能の1つは、スキーマとセカンダリインデックスを考慮することなくドキュメントに自動的にインデックスを付けることです。 同時に、安定した安定した数の非常に高速な書き込み操作が、連続する要求中に保証されます。

自動インデックス作成は、インデックス作成ポリシーを選択し、パフォーマンスとストレージを増やすことでわずかに修正できます。 自動インデックス作成を完全にオフにするか、インデックス付けされるドキュメントのみを選択して（インデックスを作成しないドキュメントを選択）、同期（一貫性）モードと非同期（遅延）モードを選択できます（デフォルトでは、各挿入操作でインデックスが同期的に更新されます。置換または削除。この動作は「遅延」モード用に修正でき、たとえば、多数の読み取り操作を伴うコレクションなどでパフォーマンス上の利点を得ることができます）。



マルチドキュメントトランザクション





RDBMSは通常、ストアドプロシージャとトリガーを使用してビジネスロジックを記述し、トランザクションとして実行します。これにより、開発者は、アプリケーションプロジェクト開発言語（JS、Pythonなど）とT-SQLの2つの異なる開発言語を知る必要があります。 ただし、DocumentDBでは、JSプログラム実行モデルをストアドプロシージャとトリガーの形式でコレクションに使用できるため、効果的な同時実行制御、インデックス作成を実行でき、豊富なアプリケーションツールに気を取られることはありません。

DocumentDBは、スナップショット分離を備えたAmbient ACIDトランザクションでこのロジックを独立してラップし、JSプロセスで例外をスローすると、トランザクション全体がロールバックされます。 JSの実行は、バッファプールと同じアドレススペースでエンジン内で実行され、パフォーマンスに良い影響を与えます。

ストアドプロシージャ、トリガー、およびUDF





既に述べたように、ビジネスロジックは、ストアドプロシージャ、トリガー、またはUDFとして完全にJSで記述できます。 JSのアプリケーションは、トリガー、ストアドプロシージャとUDFの実行用に登録でき、トリガーとストアドプロシージャはCRUDできますが、UDFには書き込みアクセス権がありません。また、アクセス許可は、たとえば、前の操作。 各プロシージャ、トリガー、UDFは同時に一定量のリソースを使用しますが、外部JSライブラリにアクセスすることはできません。 割り当てられたリソースを超えると、操作がブロックされます。



プロシージャ、トリガー、UDFを登録してREST APIを使用して実行できます。登録後、ストアドプロシージャ、トリガー、またはUDFはプリコンパイルされ、実行されるバイトコードとして保存されます。



ストアドプロシージャを登録する





ストアドプロシージャの登録=新しいプロシージャ用のリソースを作成し、そのコレクションをHTTP POSTで割り当てます。 



 var storedProc = { id: "validateAndCreate", body: function (documentToCreate) { documentToCreate.id = documentToCreate.id.toUpperCase(); var collectionManager = getContext().getCollection(); collectionManager.createDocument(collectionManager.getSelfLink(), documentToCreate, function(err, documentCreated) { if(err) throw new Error('Error while creating document: ' + err.message; getContext().getResponse().setBody('success - created ' + documentCreated.name); }); } }; client.createStoredProcedureAsync(collection._self, storedProc) .then(function (createdStoredProcedure) { console.log("Successfully created stored procedure"); }, function(error) { console.log("Error"); });




ストアドプロシージャの実行





ストアドプロシージャの実行は、HTTP POSTを使用して、要求本文に必要なパラメーターを転送して再度実行されます。 



 var inputDocument = {id : "document1", author: "GG Marquez"}; client.executeStoredProcedureAsync(createdStoredProcedure.resource._self, inputDocument) .then(function(executionResult) { assert.equal(executionResult, "success - created DOCUMENT1"); }, function(error) { console.log("Error"); });




トリガー登録





トリガーの登録= HTTP POSTを使用してコレクションの新しいリソースを作成します。プロセスでは、トリガーを呼び出す前に実行するか後に実行するか、および実行する操作のタイプ（CRUD）を指定できます。 



 var preTrigger = { id: "upperCaseId", body: function() { var item = getContext().getRequest().getBody(); item.id = item.id.toUpperCase(); getContext().getRequest().setBody(item); }, triggerType: TriggerType.Pre, triggerOperation: TriggerOperation.All } client.createTriggerAsync(collection._self, preTrigger) .then(function (createdPreTrigger) { console.log("Successfully created trigger"); }, function(error) { console.log("Error"); });




トリガーリソースでHTTP DELETEを実行すると、トリガーを登録解除できます。 



 client.deleteTriggerAsync(createdPreTrigger._self); .then(function(response) { return; }, function(error) { console.log("Error"); });
