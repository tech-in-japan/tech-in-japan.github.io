WF121 Wi-FiモジュールとHTTPサーバー



[HOW-TO] WF121 Wi-Fiモジュールの組み込みHTTPサーバーの使用に関するSilicon Labs WebサイトのアプリケーションにHTMLボタンプレス機能を追加する記事を読んだ後、どのように機能するか試してみました。 さらに、モジュールのデバッグボードが手元にありました。



*便利なリンク-記事の最後。



私の計画はシンプルでした-WF121モジュールの組み込みHTTPサーバーの機能とBGScript言語を使用して、Webインターフェースを介してWi-Fi経由で電話/タブレット/ PCから電球（またはデバッグボード上のLED）を管理します

もちろん、TCPコマンドを使用して電球を簡単に制御できますが、このためには、特別なアプリケーションを作成して携帯電話/タブレット/ PCにインストールする必要があります。これは必ずしも便利ではありません。 組み込みのHTTPサーバーの利点は、アプリケーションのインストールを忘れて、どのデバイスでもブラウザーを使用できることです。 実際、これがHTTPサーバーがWF121に実装されている理由です。



WF121とは何ですか？



WF121-組み込みシステム用の2.4 GHz Wi-Fi b / g / nモジュール。クライアントとしてもアクセスポイントとしても機能します。 モジュールの機能は次のとおりです。

  • アクセスポイントモードで最大5つのクライアントをサポート。
  • アクセスポイントモードで1つのクライアントから別のクライアントにトラフィックをリダイレクトする機能。
  • UART、SPI、I2C、USB、GPIOに加えて、イーサネットモジュールへの接続を可能にするRMIIインターフェイスの存在。
  • 構成可能なHTTPサーバーの可用性。
  • BGScriptで記述されたモジュールアプリケーションにロードすることにより、これらすべてを柔軟に管理する機能。


WF121モジュールを制御するための3つのオプションがあります。

  1. 外部コントローラーとBGAPIプロトコルを使用します。
  2. モジュール自体にロードされるBGScriptアプリケーションを使用します。
  3. BGAPIとBGScriptの両方が同時に使用される場合に結合されます。


BGAPIの可能性は、BGScriptを使用して計画を実装するため、今ではほとんど興味がありません。



BGScriptとは何ですか？



BGScriptは、 BluegigaがBluetoothおよびWi-Fiモジュール用に発明したシンプルなスクリプト言語です。 開発者はBGScriptを使用して、独自のアプリケーションを作成し、モジュールにロードすることができます。これにより、モジュールは独立したデバイスになり、外部制御を必要としません。 BluegigaとSiLabsの合併後、Silicon Labsの旗の下で、この言語は進化と改善を続けています。

BGScriptを使用する場合、開発者は、システムで発生するイベントに対するシステムの反応をおよそ次のように説明します。 

event system_boot(major,minor,patch,build,bootloader,tcpip,hw) #  –     Wi-Fi- call sme_wifi_on() #  Wi-Fi end event sme_wifi_is_on(result) #  - Wi-Fi  call sme_connect_ssid(6,"myWiFi") #     myWiFi end


つまり モジュールがオンになっている-> Wi-Fiをオンにする、

Wi-Fiがオンになっている->アクセスポイントに接続するなど。

Wi-Fiコントロールに加えて、BGScriptからインターフェイス、タイマーの操作、TCP / UDPソケット（クライアント/サーバー）の作成、不揮発性メモリへの重要なデータの保存などができます。詳細が必要な場合は、 WF121開発者ガイド （リンク-のBGScript記事の最後に）。

BGScriptの準備ができると、拡張子が.bgsのファイルとしてWF121プロジェクトに含まれます。



WF121プロジェクトとは何ですか？



WF121プロジェクトは、WF121 Wi-Fiモジュールの設定とロジックの説明を含むファイルのセットです。 プロジェクトファイルのリストは、project.xmlファイルに含まれています。 http_servプロジェクトの場合、project.xmlファイルは次のようになります。 

 <?xml version="1.0" encoding="UTF-8" ?> <project> <scripting> <script in="main.bgs" /> </scripting> <software> <binary in="fw/wifi_http_serv.juo" /> </software> <hardware> <uart channel="1" baud="5000000" api="true" handshake="True" /> <sleep interrupts="0x1"/> </hardware> <files> <file path="404.html"/> <file path="main.html"/> <file path="ap.html"/> <file path="appl.html"/> <file path="client.html"/> <file path="index.html"/> <file path="logo.gif"/> <file path="style.css"/> </files> </project>


ご想像のとおり、BGScriptを含むファイルの名前は<scripting>タグに配置されます。

wifi_http_serv.juoファイルは構成に使用できないため、ここでは考慮しません。

project.xmlファイルの<hardware>タグは、一般的にモジュールの外部インターフェイスの構成を記述します-UART、SPI、I2C、イーサネット、ADC、GPIO。 この場合、UART構成の例を示します。この構成は、 Bluegiga WiFi GUIプログラムを使用してモジュールをフラッシュする可能性を維持するのに役立ちます。 あるいは、インターフェースの記述は別のファイルで作成することもできます。 WF121インターフェイスの構成の詳細については、「 WF121構成ガイド」を参照してください。

<image>タグには、コンパイル後に受け取ってモジュールにロードするファームウェアファイルの名前が含まれています。 dfuとhexの2つの形式でファイルを生成できます。 Bluegiga WiFi GUIプログラムで使用されているdfu形式に関心が高まっています。

最後に、<files>タグには、HTTPサーバーに関連するファイルの説明が含まれています。



詳細に興味がある場合は、 WF121構成ガイドにようこそ（記事の最後にあるリンクを参照）。



ソースプロジェクトを変更して、ブラウザから電球をオンまたはオフにします



お気に入りのスマートフォンやタブレットのブラウザから電球（LED）を制御するには、次のものが必要です。

  • プロジェクトにボタンのあるWebページを含める。
  • BGScriptを使用して、個々のボタンを押したときのWi-Fiモジュールの反応を説明します。
  • BGScriptファイルで、LEDが接続されているモジュールの出力を構成することを忘れないでください。


元のプロジェクトにボタンを作成して出力を構成する例がありますが、WEBボタンを押すためのイベント処理を追加するだけです。 合計で、DKWF121デバッグボードで制御できる4つのLED、RD4、RD5、RD6、およびRD7があります。 この例ではLED RD6およびRD7を使用してモジュールのステータスを示すため、LED RD4およびRD5を使用して電球をシミュレートします。 プロジェクトを私たちのニーズに適合させるために、最小限の変更を行います。



1.元の例のHTTPサーバーには、不要な複数のページと多くのボタンが含まれています。これらはすべて慎重に削除でき、LEDを制御するためにボタンのペアを2つだけ残します（プロジェクトのappl.htmlファイルを参照）。 

 <body> </form> <form action="/button.cgi" method="get"> <br /> <Input type="submit" name="1" value="On 1" class="btn_body"> <Input type="submit" name="2" value="On 2" class="btn_body"> <br /> <Input type="submit" name="3" value="Off 1" class="btn_body"> <Input type="submit" name="4" value="Off 2" class="btn_body"> </form> </body>


2.モジュールのWEBページでボタンをクリックすると、イベントhttps_button（数値）が生成されます。 main.bgsファイルでこのイベントへの応答を次のように記述します。 

 event https_button(number) if number = 1 then call hardware_io_port_write(3, $0010, $0010) #   1 –  RD4 end if if number = 2 then call hardware_io_port_write(3, $0020, $0020) #   2 –  RD5 end if if number = 3 then call hardware_io_port_write(3, $0010, $0000) #   3 –  RD4 end if if number = 4 then call hardware_io_port_write(3, $0020, $0000) #   4 –  RD5 end if end


3. RD4およびRD5 LEDが接続されている出力を構成します。 

 event system_boot(major, minor, patch, build, bootloader_version, tcpip_version, hw) ... ... call hardware_io_port_config_direction(3, $0020, $0000) #  RD4   call hardware_io_port_write(3, $0020, $0000) #  RD4  0    call hardware_io_port_config_direction(3, $0010, $0000) #  RD5   call hardware_io_port_write(3, $0010, $0000) #  RD5  0    ... end ...


もちろん、行われた変更は最小限で済み、モジュールの通常の動作には、BGScriptのさらなる「コーミング」と他のプロジェクトファイルの美しさが必要です。

テスト中にMozilla FirefoxまたはIEを使用してWEBボタンを押すと、モジュールで1つのhttps_button（数値）イベントが生成され、Chromeを使用すると最大2つになるため、LEDのオンとオフに異なるボタンを割り当てたことに注意してください！ したがって、Chromeと組み合わせたオン/オフボタンを使用すると、ランプがすぐにオン/オフになり、テスターに​​不快感を与えます。 その結果、LEDをそれぞれオンおよびオフにするために、「オン」と「オフ」の別々のボタンを作成することを好みました。 これはすべて、SDK 1.2.4-73に適用されます。

プロジェクトの準備ができたら、ファームウェアファイルを生成してモジュールにロードするだけです。



プロジェクトをコンパイルし、結果のコードをWF121にロードします



Bluegiga WiFi GUIプログラムは、ファームウェアをすばやくコンパイルしてモジュールにロードするように設計されています。 また、BGAPIの調査だけでなく、モジュールの迅速な構成/テストにも使用すると便利です。 このプログラムは64ビットオペレーティングシステムでのみ動作し、32ビットで起動すると、64ビットが必要であると表示されます。 この制限は回避できるとの意見がありますが、私は試したことはありません。詳細はわかりませんし、嘘もつきません。 Bluegiga WiFi GUIは、サンプルアプリケーションとともにSDKの一部です。



一般に、プロジェクトをコンパイルしてモジュールにロードするときのアクションのアルゴリズムは次のとおりです。

  • PC SDK WF121にインストールします。
  • Bluegiga WiFi GUIを起動します。
  • ポート、速度を選択し、「接続」をクリックします。
  • [ファームウェアのアップグレード]タブに移動します。
  • 「プロジェクトを開く」ボタンでプロジェクトを選択します。
  • [ビルド]ボタンでコンパイルします。
  • 「Boot to DFU mode」をクリックして、モジュールをプログラミングモードにします。
  • 「アップロード」ボタンでファームウェアをロードします。


モジュールにプロジェクトをロードした後、携帯電話、タブレットなどをBluegiga Wi-Fiアクセスポイントに接続し、ブラウザーを起動して、192.168.1.1のページに移動し、ボタンを押してLEDが応答するのを確認できます。 この場合のスマートフォンの画面は次のようになりました。
