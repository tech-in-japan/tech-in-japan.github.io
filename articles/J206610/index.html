1. Appleは最善を尽くし、このテクノロジーをすべてのデバイス（iOSおよびMacオペレーティングシステム）で比較的長い間使用しており、最新バージョンのAndroidおよびほとんどのテレビセットトップボックスでもサポートされています。

AppleのHLSは最も信頼性の高いHTTPビデオ転送プロトコルの1つであり、すでにその信頼性を証明し、時の試練に合格しています。

もちろん、私たちの世界では完璧ではありませんが、アップルはいつものようにトップです。 あなたは私がAppleのファンではないとは思わない、ただ客観的に判断しようとする。

そのため、ビデオ信号とオーディオ信号の伝送に関するいくつかの言葉：ビデオ信号はMPEG-2 TSコンテナにパッケージされ、非常に一般的なMPEG H.264（ビデオ）およびAAC（オーディオ）コーデックが使用されます。 ビデオは出力で異なるビットレートでエンコードされ、その結果、m3u8形式のプレイリストが取得されます。 不正アクセスからコンテンツを保護するために、AES-128アルゴリズムが使用され、HLSを介して送信されるコンテンツを暗号化できます。

これが、Apple HTTP Adaptive Streaming（HLS）テクノロジーを説明する方法です。

2.次に、Microsoftの同様に人気のあるアダプティブブロードキャストテクノロジーであるSmooth Streamingを検討します。

Smooth Streamingは、Silverlightビデオプレーヤー、および自然にオペレーティングシステムであるWindowsおよびWindows Phoneでサポートされているプロトコルです。

Microsoftが持っているもの、Appleが持っていないもの、またはSmooth StreamingがHLSより優れているもの。 これは議論の余地がありますが、これらはアダプティブブロードキャストに対する比較的異なるアプローチであるため、すべて一方向の投影を見てみましょう。 最大の利点は、より汎用的で拡張可能なマニフェストファイル形式（xml）です。 それほど利点はありませんが、現時点では、ビデオのすべてのセグメントが最も一般的なMP4コンテナーにパッケージ化されています。

ビデオ信号とオーディオ信号は最初にセグメント化され、次に上記のようにMP4コンテナーにパッケージ化され、WebサーバーとCDNを介してHTTP経由で配信されます。 H.264コーデック（ビデオ）とAAC（オーディオ）が使用されます。

一般的に、Windowsはコンセプトを非常にうまく宣伝しており、Appleとの建設的な競争を示しています。

3.それで、アダプティブブロードキャストの別のオプションであるAdobe HTTP Dynamic Streaming（HDS）に到達しました。

名前が示すように、このテクノロジーはアドビによって作成され、世界中のほぼすべてのパーソナルコンピューターでサポートされています。

動作原理は、HTTPテクノロジーの他の適応システムと似ています。 この点で、特別な利点を特定することは難しく、HDSがサポートしているDRMシステムはFlash Accessのみだと言えます。

4.また、アダプティブブロードキャストの一般規格のトピックを検討したいと思います。

放送局が直面している現時点での大きな問題は、クライアントのさまざまなデバイスで可能な限り多くのソリューションをサポートし、さまざまなテクノロジーを使用することです。 それは高価であり、必ずしも便利ではありません。

そして、ここでビデオ放送のための単一の標準の広範な導入の必要性があります。 MPEG-LAとISOが昨年発売したMPEG-DASH（HTTPを介した動的適応ストリーミング）は、このような標準になることを目的としています。

これは、最も普遍的で完璧なブロードキャストプロトコルと言えます。 Smooth Streamingのマニフェストファイル（xml）を使用し、ほとんどのコンテナ形式をサポートし、「一度購入すればどこでも再生できる」という原則に基づいてUltraViolet（DRMシステム）と統合します。 UltraVioletは、すべてのデバイスをエンコードする単一の環境を提供します。

この技術がどれほど普遍的であっても、世界標準化団体によってどのように推進されようとも、その実装は遠い将来に期待されています。

それらの1つは、ストリームで送信される同じビデオセグメントの長さ、およびそのようなストリームオプションの数です。

ビデオセグメントを不必要に短くすると、無駄なトラフィック消費が増加します。 なぜこれが起こっているのですか？しかし、各セグメントでは、ビデオに加えて、いわゆるオーバーヘッドが抑制されるためです。 その結果、セグメントが増えるほどオーバーヘッドが大きくなり、それがまさにトラフィックの無駄になります。

しかし、コインの裏返しもあります。ビデオセグメントを長くしすぎると、ユーザーが品質の変化を待つ必要があるため、サービスの品質が数倍低下します。

この場合のバランスは試行錯誤によってのみ見つかるため、問題に対する普遍的な解決策はありません。
