Apache Maven-基本

コメントでMavenに関するトピックを公開した後、 Mavenの操作を開始する方法、開始する場所、pom.xmlファイルをコンパイルする方法、プラグインを取得する場所などについて質問がありました。 このトピックは、開始またはよくある質問の一種になります





用語



他のシステムと同様に、Mavenには独自の用語と概念があります。



プロジェクト構造全体はpom.xmlファイル（POM-プロジェクトオブジェクトモデル）に記述されており、プロジェクトのルートフォルダーに配置する必要があります。 Mavenの重要な概念はアーティファクトです。これは実際、リポジトリに保存されているライブラリです。 何らかの中毒やプラグインの可能性があります。



依存関係とは、プロジェクトでコードをコンパイルまたはテストするために直接使用されるライブラリです。



プラグインは、プロジェクトのビルド時またはその他の目的（デプロイメント、Eclipse用のプロジェクトファイルの作成など）にMaven自身が使用します。



Mavenを使用する最初の段階で、ユーザーは確かにアーキタイプなどの概念に遭遇します。 アーキタイプは、さまざまな種類のプロジェクト（web、swing-projectsなど）のファイルおよびディレクトリの特定の標準レイアウトです。 言い換えれば、Mavenはプロジェクトの通常のビルド方法を認識しており、アーキタイプに従ってディレクトリ構造を作成します。



通常、成果物の名前は、グループの名前、独自の名前、およびバージョンで構成されます。 たとえば、Maven環境では、Springの名前はorg.springframework.spring：2.5.5です。 最後のドメインは常にartifactIdを意味し、その前にあるすべてのもの-groupId-これを覚えておいてください！



上記の記事で詳しく説明されているので、ライフサイクルについては説明しません。 さあ、練習しましょう。



Mavenをインストールする



最新バージョンは、公式Webサイトのダウンロードページからいつでもダウンロードできます 。 アーカイブを任意のディレクトリに解凍するだけです。 次に、Pathに変数を作成する必要があります。この変数では、Mavenへのパスを指定する必要があります。 Win +一時停止-詳細設定-環境変数-上部のウィンドウで[作成]をクリックし、M2_HOMEという名前を入力して、値「C：\ apache-maven-2.2.1」を入力します。 次に、値％M2_HOME％\ binで別のM2変数を作成します。 また、JDKへのパスを持つJAVA_HOME変数があることを確認します。 その値は、「c：\ Program Files \ Java \ jdk1.6.0_10 \」のようなものでなければなりません。 最後に、同じウィンドウでPath変数を作成/変更します。Maven実行可能ファイルを持つ父親がコマンドラインから見えるように、％M2％を書き込むだけです。 次に、インストールのパフォーマンスを確認する必要があります。 これを行うには、コマンドラインに移動してコマンドを入力します 



  mvn –バージョン


次のようなMaven、jre、およびオペレーティングシステムのバージョンに関する情報が表示されます。 



    Mavenバージョン：2.2.1
    Javaバージョン：1.6.0_10
    OS名： "windows 2003"バージョン： "5.2" arch： "x86"ファミリ： "windows"


Mavenは、たとえばC：\ Documents and Settings \ username \ .m2 \ repositoryディレクトリなどの個人フォルダーにローカルリポジトリを作成します



すべて、Mavenが動作する準備ができました。アプリケーションの作成を開始できます。



アーキタイプからアプリケーションを作成する



Maven Webサイトには、アプリケーションで最も人気のあるアーキタイプがリストされていますが、独自のアーキタイプを簡単に作成したり、より具体的なアーキタイプを見つけたりすることができます。



それでは、Webアプリケーションに興味があるとしましょう。適切なアーキタイプを見つけます。これはmaven-archetype-webappと呼ばれます。 コマンドプロンプトの必要なディレクトリで、Mavenコマンドを実行します。 



  mvnアーキタイプ：create -DgroupId = com.mycompany.app -DartifactId = my-webapp 
       -DarchetypeArtifactId = maven-archetype-webapp


これで、名前がjavaの非常にわかりやすいディレクトリ構造を見ることができます-これはクラス、webappです-Webアプリケーションのページ、リソース-クラスパス内のさまざまなリソース（構成ファイルなど）、テスト-ユニットテストなど.p。

依存関係とリポジトリ



原則として、最も人気のあるライブラリは中央リポジトリにあるため、pomファイルの依存関係セクションに直接書き込むことができます。 たとえば、Spring Frameworkを接続するには、次の依存関係を定義する必要があります。 



    <依存関係>
       ...
       <依存性>
          <groupId> org.springframework </ groupId>
          <artifactId> spring </ artifactId>
          <バージョン> 2.5.5 </バージョン>
       </ dependency>
    </依存関係>


バージョンを指定する必要はありませんが、Mavenは最後のオプションを選択しますが、プロジェクトが単にある時点で収集を停止したり、完全に予期せず十分にテストされた場所に落ち始めたりするケースが何度もあったため、個人的にはこれを行うことをお勧めします誰も何も変えません。



通常、特定のものは中央リポジトリにないため、製造元のリポジトリを手動で指定する必要があります。 たとえば、JBoss RichFacesのajaxコンポーネントにJSFフレームワークの依存関係を追加します。



依存関係があるため、すべてが簡単です。 



    <依存関係>
       <依存性>
          <groupId> org.richfaces.ui </ groupId>
          <artifactId> richfaces-ui </ artifactId>
          <バージョン> 3.3.1.GA </バージョン>
       </ dependency>        
    </依存関係>


ただし、JBossリポジトリは、ローカルリポジトリのルートにあるsettings.xmlファイルまたはpom.xmlファイル自体のいずれかに、このリポジトリがすべてのプロジェクトで必要か、またはいずれかでペンで登録する必要があります特定の、それぞれ：

       <リポジトリ>
          <リリース>
             <enabled> true </ enabled>
          </ releases>
          <スナップショット>
             <enabled> false </ enabled>
             <updatePolicy>決して</ updatePolicy>
          </ snapshots>
          <id> repository.jboss.com </ id>
          <name> MavenのJbossリポジトリ</ name>
          <url>
             http://repository.jboss.com/maven2/
          </ url>
          <layout>デフォルト</ layout>
       </ repository>
    </ repositories>


原則として、大規模プロジェクトのサイトでは、ライブラリをMavenベースのプロジェクトに埋め込むために必要なすべての情報が書き込まれますが、非常に長い間アーティファクトを探す必要がある場合があります。 ここでは、 MVNrepository.comが非常に役立ちます。常に、目的のライブラリがどこにあるかがわかります。 しかし、たとえそれがなかったとしても、私自身の経験から、「 <library name> pom.xml」をグーグル検索することをお勧めします。 開発者はもはや配布を気にしないので、プロジェクトは長い間既に閉じられており、リポジトリに入れられないことが起こります。 その後、1つの方法が残っています-コマンドでファイルをリポジトリに手動で追加するには： 



    mvn install：インストールファイル

プラグイン



プラグインは依存関係と同じ成果物であるため、ほぼ同じ方法で説明されます。 依存関係セクションの代わりに-プラグイン、依存関係-プラグイン、リポジトリ-pluginRepositories、リポジトリ-pluginRepository。



すべてがMavenプラグインによって行われます。直接開始されたものでも-プロジェクトアセンブリ。機能を追加したくない場合、このプラグインのみをプロジェクトプロパティで指定する必要はありません。



WTPバージョンを使用してEclipseのプロジェクトを作成するためのプラグインの構成方法を見てみましょう。 2.0。 pom.xmlのプラグインセクションで、次のプラグインを記述します。 

        
    <プラグイン>
       <groupId> org.apache.maven.plugins </ groupId>
       <artifactId> maven-eclipse-plugin </ artifactId>
       <構成>
          <wtpversion> 2.0 </ wtpversion>
       </構成>
    </ plugin>


ここで再びコマンドラインに行き、コマンドを実行します 



  mvn eclipse：日食


Mavenがリポジトリー内のすべてのライブラリーを検出するか、それらをダウンロードして出来上がるまで待ちます-これで、MavenプロジェクトをEclipseプロジェクトとして開くことができます。 同時に、ライブラリは従来のアプローチのようにどこにもコピーされませんが、リポジトリに残り、Eclipseはその変数を介してライブラリへの参照を作成します。



すべてのプラグインの単一のリストは当然存在しません。 公式サイトには、Maven開発者が直接サポートするプラグインのみがあります。 ただし、プラグインの名前は非常に簡単であり、キーワード「maven tomcat plugin」を検索すると、最初のリンクとしてTomcatにプロジェクトをデプロイするプラグインが見つかる可能性が高いことに注意してください。
