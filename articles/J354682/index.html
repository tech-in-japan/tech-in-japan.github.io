.NET Core WebApp + GitHubの継続的インテグレーション（ビルド、テスト、デプロイ）の段階的なセットアップ







みなさんこんにちは。 継続的インテグレーション （CI）は、ソフトウェア開発の世界に長い間浸透しており、多くの場合、その不可欠な部分であり、開発の容易さを維持しながら、より良いコードを作成できます。 そして、最初は、CIのセットアップにはかなりの労力とお金が必要でしたが、今でははるかに手頃な価格で、簡単に、さらには無料になりました。 .NET Coreオープンソースプロジェクト用にCIを構成する方法に興味がある場合は、catを使用してください。

F＃はデモプロジェクトの主要言語ですが、マニュアルはC＃プロジェクトと完全に互換性があります。 F＃が選択されるのは、 この記事とそれに関連するリポジトリは、6月に予定されている2番目の出版物「F＃+ Giraffeで本格的なWebアプリケーションをゼロから構築しています」の基礎となっています。

導入と必要なツール



この記事ではデモリポジトリを使用します 。 リポジトリの各章にはタグが作成されており、その完全なリストは記事の最後にあります。







記事に書かれていることを繰り返したい場合は、以下が必要です。







  • Githubアカウント
  • ローカルgit
  • .NET Core SDK> 2.1。
  • F＃サポート付きVisual Studio 2017
  • Azureのアカウント 。 これは必要ではありません。さらに、「試用」するための無料のアカウントを提供します。


独自のリポジトリを作成する



タグ：InitialMaster







始めましょう。 まず、何らかのバージョン管理システムが必要です 。 gitとgithubを使用します







GitHubに新しいリポジトリを作成し、.gitignoreファイルのテンプレートとプロジェクトライセンスを選択します。 .gitignoreファイルは、余分なファイルがリポジトリに入らないようにするために必要です（ビルドプロセスの結果など）。 コードで許可されたアクションの範囲を理解できるように、ライセンスが必要です。 このプロジェクトでは、VisualStudioテンプレートとMITライセンスを選択しました。







リポジトリを作成したら、クローンを作成する必要があります。完全なコピーを削除して、ローカルマシンに配置します。 これを行うには、 Gitをローカルにインストールしてコマンドを実行します： 







git clone PATH_TO_YOUR_REPOSITORY


これで、GitHubのリモートリポジトリに関連付けられたローカルリポジトリが作成されました。 Gitには非常に優れたドキュメントがあるので、何かにつまずいたらここに来てください 。







バージョン管理システムは最新の発明ではありません。 最初のVCSは1982年に登場しました（1972年にも登場しましたが）。 近代的な開発が事実上不可能である楽器の開発の歴史に関する詳細は、 ここで見つけることができます 。 しかし、Gitは比較的最近登場しました-2005年、Linuxの主要な作成者であるLinus Torvaldに感謝します。


新しいプロジェクト



タグ：NewFSharpProject







リポジトリができたので、プロジェクトを作成する準備ができました。 リポジトリがあるフォルダーに移動し、コマンドを実行して新しいプロジェクトを作成します。 







 dotnet new web -lang F#


このコマンドが機能するためには、 .NET Core SDKをインストールする必要があることが重要です。 Windowsで作業している場合、ほとんどの場合、システムがdotnet.exeの新しいパスを選択するようにOSを再起動する必要があります。 F＃の準備がまだ整っていない場合（これは非常に楽しい言語であるため、お勧めします）、langパラメーターを渡すことはできず、dotnetはC＃Webアプリケーションを作成します。







このコマンドを実行すると、3つの新しいファイルが取得されます。







  • Startup.fs-これがサーバー構成です
  • Program.fs-これはサーバーの開始点です
  • FSharpWebAppWithCIDemo.fsproj-これはプロジェクト構造の説明です


このミニマリズムにもかかわらず、こんにちは世界を起動するには十分です。







ここで、プロジェクトの構造に同意しましょう。 各プロジェクトは個別のフォルダーに配置する必要があり、ルートフォルダーにはプロジェクトに関係なく構成ファイルのみが含まれます。







その結果、この構造のようなものを取得する必要があります（その後、Visual Studioを介してソリューションですべての操作を行います）。







-FSharpWebAppWithCIDemo //プロジェクトのメインフォルダー







--BLL //基本的なロジックを持つプロジェクト







--WebServer // Webサーバーとその設定。







重要！ F＃では、ファイルとプロジェクトの順序が重要です。 プロジェクトツリーの上位にあるファイルは、ツリーの下位にあるファイルにアクセスできません。 これは、Webサーバーが任意のプロジェクトにアクセスできることを意味します（最下部にあります）が、ビジネスロジックを含むプロジェクトは、独自のコードのみでコンテンツである必要があります。 一方では、これはやや珍しいことであり、他方では、すぐにコードを構築します。 たとえば、データベースが何らかのWebサーバーについて知っている必要があるのはなぜですか？ 一般的に、1つのモジュールが他のモジュールについて認識しているほど、他のモジュールで作業中にそのモジュールを壊す可能性は低くなります。 言い換えれば、 低結合高コヒジョン







Linuxシステムに関するもう1つの重要なポイントです。 ご存じのとおり、.NET Coreで作成されたプロジェクトはクロスプラットフォームであり、Linuxシステムで実行できます。 ただし、プロジェクト全体を記述するファイル（.slnファイル）は、完全にクロスプラットフォームではありません。 したがって、Windowsを実行している場合、このファイル内のプロジェクトへのパスは正しく書き込まれません-バックスラッシュ文字「\」を使用します。 残念ながら、これは今のところ問題です。 間もなく修正されます。 ただし、ビルド中に、サーバーまたはローカルマシンで、次の形式のエラーが表示されます。 







  error MSB4025: The project file could not be loaded. Could not find file ... MySuperProject.fsproj.metaproj.


ほとんどの場合、これは.slnファイルに移動し、「\」を「/」に手動で変更する必要があることを意味します。







これでプロジェクトの準備ができました。 それを収集し、正常に組み立てられていることを確認して、先に進みましょう。







Hello Worldを記述する伝統のルーツは、1974年にさかのぼります。 このとき、 Cのプログラミング：チュートリアルガイドが登場しました。 マニュアルに記載されているコード例が、最新のコンパイラによってコンパイルされなくなっていることが重要です。


CI-ビルド



タグ：TravisSupport







F＃には既に動作するWebアプリケーションがあります。 直接開発を開始することは可能ですが、その記事はそれについてではありません。 したがって、 CIプロセスの構成を開始し、最も簡単なビルド構成で開始します。







実際、CIはサービスでもツールでもありません。 これはプログラミングへのアプローチであり、その主な簡略化された考え方は、頻繁なコミット、自動検証、およびエンドユーザーへの結果の配信です。 したがって、CIを構成すると言うと、これはすべてのプル要求で自動ビルドプロセスをセットアップし、テストを実行し、その結果を分析することを意味します。 そして、これらすべての結果はプルリクエストのすぐ下に表示されるので、プロジェクトを誰がどのように破ったかを常に認識します。







新しいブランチを作成して、TravisSupportという名前を付けましょう。 ちなみに、新しい機能を実行している場合は、マスターではなく別のブランチで実行すると便利です。 最初に、あなたはあなたの仲間や彼らのコミットから独立します。 第二に、ローカルブランチでは、メインブランチに落ちて何かを壊すことを恐れずにコミットを行うことができます。 3番目に、私のお気に入りの利点は、-squashフラグでマージすると、すべてのコミットが1つにマージされ、100500回の試行がなかったように見えますが、すべてが初めて判明しました。 重要なのは、ソースブランチを削除することを忘れないでください。そうしないと、作業を続けて再度コミットしようとすると、競合が発生する可能性があります。







しかし、ポイントに近い。 CIの場合、プロジェクトを組み立て、テストし、求めていることを実行するツールが必要です。 そのようなツールとしてTravis-CIを選びました。 これはオープンソースでは無料で、比較的シンプルであり、私はある程度の経験があります。







Travisと統合するには、次のものが必要です。







  • Travisにリポジトリへのアクセス権を付与します。
  • 彼に何が必要かを彼に説明してください。


最初のポイントは非常に簡単です。 開く https://travis-ci.org/にアクセスし、GitHubアカウントを使用してそこにログインします。 次に、使用可能なリポジトリのリストにリポジトリを追加します。 リポジトリが表示されない場合は、SyncAccountボタンを使用します。







2番目のポイントは、はるかに興味深いものです。 CIプロセスは、travis-aが.travis.ymlファイルを使用して制御します。 これがドキュメントです。 このファイルをプロジェクトのルートにコピーし、そこに書かれている内容を分析しましょう。 







 language: csharp dotnet: 2.1.4 sudo: false env: global: - DOTNET_CLI_TELEMETRY_OPTOUT: 1 script: - dotnet build FSharpWebAppWithCIDemo.sln -c Release


  • 言語-アプリケーションが記述されている言語。 TravisはC＃とF＃を区別せず、fsharpディレクティブもありません。 ただし、このファイルはC＃プロジェクトにも適しています。
  • dotnet-dotnetのバージョンをすぐに指定して、収集する情報を確認できます。
  • env-このセクションは環境変数を担当します。 ご覧のとおり、DOTNET_CLI_TELEMETRY_OPTOUTを追加して、ビルドプロセス中にテレメトリ送信をオフにしました。 ここに任意の変数を書き込むことができ、ビルドへの引数として渡されます。
  • スクリプト-このセクションは、travis-aに必要なものを担当します。 この場合、アプリケーションのリリースモードでビルドを実行するだけです。


これは、最も単純で最も基本的なCIシナリオです。 簡単に言えば、プルリクエストごとに、ビルドを収集しようとします。 行く-すべて順調です。 行きません-GitHubとメールで警告が表示されます。 CIプロセスを開始する条件は、.travis.yml自体とサイトのプロジェクト設定の両方で簡単に構成できます。 また、Travisの追加機能（OSの選択など）に興味がある場合は、次のようになります。







ビルド中に「MSB4025、ファイル... .fsproj.metaprojが見つかりませんでした」というエラーが発生した場合、.slnファイル内のバックスラッシュをスラッシュに置き換えてみてください。







ところで、ビルドプロセスを見ると、Linuxで実行されていることがわかります。これは朗報です。







そして、当然、readmeにふさわしいバッジを追加することを忘れないでください。 Travis CIプロジェクトにあります。







バッジの可視性が気に入ったら、 こちらをご覧になることをお勧めします。 十分ではない場合は、そこで独自のバッジを作成できます。


Ciテスト



タグ：CodeCov







それでは、テストの時間です。 一般に、コードを記述する前にテストを記述することなく、 TDDの原則に既に違反しています。 それで、それを修正する時が来ました。







Bllプロジェクトの下で、新しいBllTestプロジェクトを作成し、そこにBllTest.fsを追加します。 同じプロジェクトにxUnitをインストールする

プロジェクトの構造は次のようになります。







-FSharpWebAppWithCIDemo







--ブル







--BllTest







--WebServer







Bllプロジェクトで、Sayモジュールとhello関数を使用して新しいbll.fsファイルを作成します（テストの前にコードを記述してTDDに違反しました） 







 namespace Bll module Say = open System let hello() = "Hello user #" + Guid.NewGuid().ToString()


そして、それをテストします。つまり、この関数を呼び出した結果が常に前のものと異なることを確認します。 ところで、これは保証されていません。 2つのGUIDを生成すると、同じ2つのGUIDを取得できる可能性がゼロではありません。 確かに、このイベントを待つまで太陽は消えますが、まだチャンスはあります。







実際、コード自体もテスト自体も重要ではありません（結局、あなたはすでにこれを聞いたことがありますよね？）。 TravisおよびCodeCovと正しく統合するには、それらの存在のみが重要です。 テストを手動で実行し、合格したことを確認して続行します。 最初のテストを行った後、このテストを実行するにはTravisが必要です。 これを行うには、.travis.ymlでスクリプトセクションに別の行を追加する必要があります 







 - dotnet test -c Release --no-build BllTest/BllTest.fsproj


ここではすべてが明らかです-ビルドなしでテストを実行します（ビルドアーティファクトは前の手順で既に取得されているため）。これにより時間を節約できます。







しかし、これでは十分ではありません。 コードカバレッジ統計が必要です。 このため、 coverlet.msbuildはmsbuildに組み込まれ 、 必要なopencoverを含むさまざまな形式でレポートを作成する素晴らしいツールです。 Nugetタブを開き（またはコンソールからコマンドを実行し）、BllTestプロジェクト用にこのパッケージをインストールします。 他のプロジェクトでは必要ありません。 その後、.travis.ymlの前の行を次のように変更します。 







  - dotnet test -c Release --no-build BllTest/BllTest.fsproj /p:CollectCoverage=true /p:CoverletOutputFormat=opencover


ここで、.travis.ymlのフルバージョンをここで選択できることを思い出させてください







コンソールから同じコマンドを実行して、すべてが機能することを確認できます。 同時に、プロジェクトの現在のコードカバレッジが表示されます。







そのため、テストがあり、実行され、何らかのコードカバレッジレポートを見ることができます。 しかし、これでも十分ではありません。 ここで、このレポートを処理して結果を報告できる誰かが必要です。 ここでCodecovが便利になります。これはOpenSourceプロジェクトでも無料です。 GitHubアカウントを使用してCodecovにログインし、目的のプロジェクトを追加します。 最後に、最後に.travis.ymlを更新するときに、新しいセクションafter_scriptを追加します。 







  after_script: - bash <(curl -s https://codecov.io/bash)


この例を見て、すすり泣きました。 それまでは、カバーオールへのレポートの送信を開始するために約3時間を費やしましたが、ここではすべてが非常にシンプルであることが判明しました...私を幸せにしてくれてありがとう。







すべての準備が完了したので、コミットを行い、100％のカバー率で十分なバッジを取得できます。







ケントベックをTDD の作者と呼ぶ人もいますが、彼は昔のプログラミング本でずっと前に読んだテクニックを再発見しただけだと言います。


CIの展開



タグ：DeployToAzure







息を吸って、コーヒーを飲んで、インフラストラクチャの仕事の終わりが近づいています。 展開プロセスの準備を開始します。







要するに、デプロイメントとは、実行のために作業結果（ビルドアーティファクト）を提供するプロセスです。 これは、手で、または自動モードで行うことができます。これは、怠け者として、私はずっと好きです。 Azureでデプロイを設定する方法を紹介します（ただし、これは既にアカウントがあり、まだ数ドル残っているためです）。 これに合わない場合は、Travisが準備するビルドアーティファクトを使用できます。 dotnet buildをdotnet publishに変更し、結果をサーバーに送信するだけで十分です。次に例を示します。







  • AWS S3
  • Githubのページ
  • NPM
  • エロク
  • Googleアプリエンジン


その他







しかし、今ではそれについてではなく、Azureについてです。 展開のセットアップは比較的簡単で、（何よりも）Azure自体の外部から管理されます。 今からお話しします。







まず、独自のWebアプリケーションを作成する必要があります。 これは非常に簡単です。 Azureに移動し、[リソースの作成]-> [Webアプリ]またはこのリンクをクリックします。 そこで、将来のサイトの名前、サブスクリプション（サブスクリプションは請求先になるため、学生の場合は無料のサイトを選択することをお勧めします）およびサーバーがデプロイされるOSを選択します。 コアを使用しているため、LinuxとWindowsの両方を選択できます。 Windowsを選択する理由は、より快適に感じるためですが、一般的に違いはありません。 Webサイトを構築し、実際に機能することを確認し、展開のセットアップを始めましょう。 これを行うには、プロジェクトのルートフォルダーに2つのファイルを追加します。







  • .deployment //デプロイメントのために呼び出す必要があるものを記述します）
  • build.cmd //展開中に正確に行うことを記述します


一見、build.cmdはかなり怖いように見えますが、内部では非常に些細なことが起こります。







  • Node.jsを確認します//信じられませんが、Microsoftは展開プロセスでNode.jsを使用します
  • パスで変数を設定します//ファイルとサーバーファイルがある場所
  • kudusync //サーバーフォルダーにアーティファクトをコピーするためのユーティリティをインストールします
  • dotnet publishコマンドが実行されます//コードをコンパイルします
  • 公開結果はwwwrootフォルダーにコピーされます。 //アプリケーションを実行するために結果をサーバーに渡します


これらのファイルを追加したら、WebアプリケーションをGitHubのリポジトリにリンクするだけです。 これを行うには、アプリケーションを選択し、DeploymentOptions-> Github-> Project-> Branchを選択します。 ところで、負荷テストもそこで構成できます。 以上で、すべてが同期されてシャットダウンされるまで待機することになりました。







後にAzureになったプロジェクトのコード名は「Project Red Dog」でした。


Gitフック



CIについて話し、 gitフックに言及しないのはおそらく正しくないでしょう。 さらに、記事で説明されている機能が可能になるのはツールであるフックです。







Gitフック、またはフックは、リポジトリ内の重要なイベントの前または直後にカスタムスクリプトを実行できる特別なメカニズムです。 それらを使用して、コミットを変更したり、変更ログを維持したり、自動フォーマットを実行したり、一般にシステムで十分な権限を持つすべてのものを使用したりできます。 デフォルトでは、それらは.git / hooksにあります。







少し風変わりですが、dotnetにはビルド前やビルド後などのフックもあります。 また、カスタムスクリプトを実行することもできます。
