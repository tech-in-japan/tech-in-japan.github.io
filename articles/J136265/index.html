テーブルビューの使用

良い一日！



長い間、Mac OS X / iOSでプログラミングするための別の言語/プラットフォームを強制的に習得させようとしてきました。 ネイティブGUIを使用したアプリケーションの開発は興味深いものです。コンソールアプリケーションは、CおよびC ++から現在流行しているNodejsまで何でも開発できるためです。 Qtのようなクロスプラットフォームフレームワークは、このOSのユーザーが慣れているネイティブのルックアンドフィールを提供しないという理由だけで、ここではあまり適していません。



Objective-C言語とHabrのCocoaフレームワークには十分な量の資料があります。 GUIの観点から見ると、Cocoaは興味深いものであり、ここではほとんどの記事がボタンとテキストフィールドに限定されています。 この誤解を修正し、プロセスのリストを表示するアプリケーションの例を使用して、Table Viewでの作業を説明します。



この記事は完全であると主張しておらず、資料は完全に正しいものであり、主に開発者を対象としています。 エラーや建設的な批判は、コメントで聞いて喜んでいるでしょう。 誰も気にしない、猫へようこそ。





理論



開発には、Table Viewの知識が必要です。 それらから始めましょう。 テーブルビューでの作業の詳細な説明は[1]にあります。ここでは、主なアイデアの簡単な要約に限定します。



テーブルビュータイプ


テーブルビューには2つのタイプがあります。

  • セルベース -各テーブルセルはNSCellのサブクラスで表されます。 ほとんどの場合、これで十分ですが、複雑なセルを持つテーブルを開発する際に困難が生じます。 NSCellのサブクラス化が必要です 。 Mac OS Xバージョン10.6以前では、このタイプのテーブルのみが存在していました。
  • ビューベース -各セルは個別のビューです。 これにより、アニメーションをサポートしていても、それほど困難なく複雑なテーブルを構築できます。 このタイプのテーブルは、Mac OS X 10.7でのみ登場しました。


この記事では、 セルベースのTable Viewを使用することを検討します 。その機能は十分であるため、ソリューションはよりシンプルになり、OS Xの新しいバージョンだけでなく動作します。



テーブルビューへのデータ転送


データは、2つの方法でテーブルビューに転送できます。

  • プロトコル（インターフェイス） NSTableViewDelegate 、 NSTableViewDataSourceの実装を使用し、Interface Builderを介してこれらのプロトコルを実装するクラスにテーブルビューを関連付けます。 Outlets dataSourceとdelegateをバインドする必要があります。 そして、このクラスでは、最も単純な場合、2つのメソッドを実装する必要があります。

    • numberOfRowsInTableView： -テーブル内の行の総数を返します
    • tableView：objectValueForTableColumn：row：-row引数で渡された行の要素を返します。 この場合の列番号は、 objectValueForTableColumnの 2番目の引数を使用してさまざまな方法で決定されます。


  • コントローラーとバインディングの使用（バインディング）。 この手法は、Model-View-Controller（MVC）パターン[2]およびCocoa Bindingsテクノロジーに基づいています。 MVCについて話すことはおそらく意味がありません。誰もがMVCについて聞いたことがあるので、聞いたことがない人は誰でもWikiでリンクを見つけることができます。 Cocoa Bindingsはさらに詳しく説明します。


ココアバインディング


Cocoa Bindingsは、バージョン10.3以降、Mac OS Xで導入されました。 その主な目標は、モデルオブジェクトとビューをリンクするためのコントローラーコードの量を減らすことです。

Cocoa Bindingsは2つのメカニズムに基づいています：

  • キー値コーディング（KVC） -これらのフィールドの名前によってオブジェクトのフィールドにアクセスするためのメカニズム。
  • Key-Value Observing（KVO）-1つのオブジェクトが他のオブジェクトの変更を監視できるようにするメカニズム。


Cocoaバインディングの詳細については、[3]を参照してください。



これで理論が終了したら、実践に進むことができます。



練習する



Xcodeを開き、新しいCocoaアプリケーションプロジェクトを作成します。 プロセスモニターと呼びましょう。 私は公開時に最新バージョンのXcode（4.2.1）を使用しているため、一部の手順が異なる場合があります。



GUI作成


MainMenu.xibファイルを開き、その中のウィンドウを選択します。 480 x 500に設定します。これは、サイズインスペクター（右側のサイドバーのルーラーアイコン）で行います。 そこで、最小サイズ-チェックボックスと必要なサイズを設定します。



オブジェクトライブラリ（右下のサイドバーのキューブアイコン）からウィンドウにテーブルビュー要素を追加します（「テーブル」という単語で検索します）。 デフォルトでは、セルベースのテーブルが作成されます。 Attributes Inspector（右側のサイドバーの行の左側）を開くと、これを確認できます。 ウィンドウ全体を占めるようにテーブルビューのサイズを変更します。 次に、サイズインスペクターを開き、自動サイズ調整画面ですべての要素を選択します。テーブルビューをウィンドウに合わせて拡大する必要があります。



次に、テーブルに列を追加する必要があります。 プロセスのPID、アイコン、プロセスの名前の3つの列を表示します。 デフォルトでは、2つの列がテーブルに追加されます。 この番号を3に変更する必要があります。これを行うには、ウィンドウ内をクリックしてテーブル（テーブルビュー）を選択します。 ここで、少し余談をする必要があります。 実際、ウィンドウオブジェクトは互いに入れ子になっており、目的のオブジェクトを選択するのは簡単ではありません。 2つのオプションがあります：オブジェクトウィンドウのオブジェクトの階層で目的の要素を選択する（Xcode 4で表示されたと思いますが、これまで見たことがない）、または目的のオブジェクトを選択してCommand + Control + Shiftをクリックします テーブルを選択した後、属性インスペクターに移動し、列の値を3に設定します。



ここで、列に名前を割り当てます。 これは、列ヘッダーをダブルクリックして行います。 最初の列の名前はPID、2番目は空白のまま、3番目はプロセス名です。



テーブルの列のサイズを変更します。 これを行うには、列を選択し、サイズインスペクターでサイズを設定します。 最初の（PID）の場合：60、2番目の（アイコン）の場合：40、3番目の（プロセス名）の場合-残りのすべてのスペース。



テーブルの2番目の列のタイプをテキストから画像に置き換えます。 これを行うには、オブジェクトライブラリで画像セルを検索し（「画像」で検索）、2列目にドラッグします。



これらすべてのステップの後、「実行」ボタンをクリックして、次のウィンドウを表示できます。





コントローラーとバインディングの追加


次に、テーブルと対話するコントローラーを追加します。 テーブルに最適なコントローラーのタイプはNSArrayControllerです。 オブジェクトライブラリで検索し（「配列」を検索）、オブジェクトリストにドラッグします。



テーブルのバインダーを設定します。 これを行うには、テーブルビューを選択し、バインドインスペクター（右側のサイドバーの最後から2番目のタブ）に移動します。 そこで、Table Contentセクションで、Contentを選択し、Bind to：Array Controller with Controller Key：ArrangedObjectsをインストールします。



次に、KVC手法を使用して、各列に表示するデータを指定します。 これを行うには、各列（テーブル列）を交互に選択し、バインドインスペクターで、値を[バインド]に設定します。 各列のモデルキーパスの値は、それぞれprocessIdentifier、icon、localizedNameです。 このような値が設定される理由を以下に説明します。



そこで、テーブルをコントローラーに接続しました。 コントローラーをデータに関連付けることは残ります。



まず、これらのデータそのものを決定します。 AppDelegate.hファイルを開き、NSArray型のコンテンツプロパティを定義します。 

@property (retain) NSArray* contents;


ファイルAppDelegate.mに次を記述します。 

 @synthesize contents;


次にMainMenu.xibに戻り、使用するデータをコントローラーに伝えます。 Bindings Inspectorに移動し、Controller Contentセクションで、Content ArrayをBind to：App Delegate、およびModel Key Path：contentsに設定します。

これで、バインダーのインストールが完了しました。AppDelegateのコンテンツ配列に必要なデータを入力するだけです。



実行中のプロセスのリストを取得する


実行中のすべてのプロセスのリストを取得するには、いくつかの方法があります。 これについては別の記事で書きます。 それでは、最も単純でわかりやすい方法、つまりCocoaを使用してみましょう。 彼はプロセスの完全なリストを受け取りませんが、これまでのところ私たちに関係ないようにします。

AppDelegateでupdateProcessListメソッドを定義します。 

 - (void)updateProcessList { NSWorkspace* workspace = [NSWorkspace sharedWorkspace]; self.contents = [NSArray arrayWithArray:[workspace runningApplications]]; }


このメソッドをクラスの最初にして、ヘッダーファイルに定義を指定せずに他のメソッドから呼び出せるようにします。

コードに関する小さなコメント。 最初の行では、クラスNSWorkspaceのオブジェクトを取得します。 このクラスのオブジェクトは、プログラム内にシングルトンの一種であるシングルトンとして存在し、 sharedWorkspaceメソッドによって取得されます。

このオブジェクトには、配列内で実行中のアプリケーションのリストを返すrunningApplicationsメソッドがあります。 すべてがシンプルです。 そして、この配列をコントローラーのデータとして使用します。



この配列の各要素は、 NSRunningApplicationクラスのオブジェクトを表します。 このクラスには、 processIdentifier 、 icon 、 localizedNameなどのいくつかのプロパティがあります 。 それらをテーブルの列に設定したことを覚えていますか？ したがって、KVCのおかげで、 NSRunningApplicationからのこれらの値はテーブル表示中に使用されます。



次に、アプリケーションの起動中に、作成されたメソッドへの呼び出しを追加します。 これを行うには、次の行を追加します。 

 [self updateProcessList];


applicationDidFinishLaunchingメソッドに。



アプリケーション終了


Mac OS Xでは、最後のウィンドウを閉じた後、アプリケーションを終了する必要はないことが認められています。 この例では、これはあまり意味がないため、自動補完を追加します。 これは非常に簡単に行われます-AppDelegate.mに1つのメソッドを追加することにより： 

 - (BOOL)applicationShouldTerminateAfterLastWindowClosed:(NSApplication *)sender { return YES; }


おわりに



これで、アプリケーションの準備ができました。 すべてが正しく行われている場合は、開始後に次のように表示されます。



演習として、プロセスのリストの定期的な更新を実装することを読者に提案します。 これを行うには、 NSTimerクラスとこのクラスのscheduleTimerWithTimeIntervalメソッドを使用できます[5]。



そのため、この記事では、テーブルビューを使用して簡単なアプリケーションを作成するプロセスについて説明しました。 ほとんどのアクションはInterface Builderで実行され、プログラミングはまったく必要ありませんでした。 単純化できるものはすべて単純化されました-ウィンドウコントローラ（ NSWindowController ）、ビューコントローラ（ NSViewController ）、メモリ管理などは考慮されませんでした。



次の記事では、プロセスツリーを表示するために階層リスト（ NSOutlineView ）を検討する予定です。 そして、そこに十分なコードがあれば、githubにリポジトリを作成します。
