銀行カードのPINコードの確認

「Banking Transaction Travels」という記事の登場に関連して、PIN検証のメカニズムに疑問が現れ始めたフレームワーク内で、この問題をいくらか明確にしたいと思います。 この記事では、いわゆる オンラインPIN、すなわち PIN。端末で入力され、トランザクションを許可するシステムにさらに検証するために送信されます。



PINカードのスクロールの問題に直接進む前に、いくつかの理論的な質問について説明します。



応用暗号



DESアルゴリズム


現在、銀行カードを使用したすべての操作の基本的な暗号化アルゴリズムはDESアルゴリズムです（3DESではなく、主にトランスポート層の暗号化操作にのみ使用されます）。 暗号化アルゴリズムの用語と分類に入ることなく、ブロックサイズが8バイトでキーが56ビット（7バイト）のブロック暗号化アルゴリズムであると言います。 実際には、DESキーを8バイトのブロックとして表すのが一般的です。各バイトでは、上位7ビットが重要で、最後のビットは重要ではありません。 キーの整合性を制御するために使用できます（これについては後で詳しく説明します）。



現時点では、56ビットキーはセキュリティの観点からは不十分であるため、暗号化にDESアルゴリズムを使用する代わりに、EDEモードで3DESアルゴリズムを使用することを受け入れます（以降、3DESという名称では、EDEスキームを使用することを意味します）。 通常、バンキング環境では、3DESアルゴリズムは2倍長のキー（112ビット、16バイト）とともに使用され、最初と3番目のステップで同じキーが使用されます（アルゴリズムの説明を参照）。



PIN検証アルゴリズム


現在、主に次の2つのPIN検証アルゴリズムが使用されています：Visa PVVおよびIBM 3624 PINオフセット。



Visa PVV


このアルゴリズムはもともとVisa支払いシステムによって開発されましたが、現在はVisaカードとMasterCardカードの両方で推奨されるPIN検証アルゴリズムです。 このアルゴリズムは、PVV（PIN検証値）に基づいています。PVVは、次の値に基づいて取得された暗号文です。



  • カード番号（以下PAN）
  • PIN検証キーインデックス（以降、PVKI）
  • PIN検証キー（PIN検証キー、以下、PVK）
  • カードのPINコード自体


PVVを受信するために、ブロックはPAN（カードの管理番号を除く最後の11桁）、PVKI、PIN（厳密には最初の4桁）から形成され、PVKを使用して暗号化され、その後、特別な関数を使用して4桁の値が抽出されますPVV [1]の値である数値。 このPVV値は、PINコードを確認するための参照です。 つまり 入力されたPINを使用して操作を受け取り、PAN、PVKI、PVKに基づいて検証すると、新しいPVV値が生成され、カードの参照PVVと比較されます。 値が一致する場合、PINは正しいと見なされ、一致しない場合は正しくないと見なされます。



次の制限は、このアルゴリズムの機能に起因する可能性があります。

  • PVV値からPINを復元することが基本的に不可能
  • サイズが厳密に4桁のPINコードを使用する




IBM 3624 PINオフセット


このアルゴリズムは、元々IBMがIBM 3624 ATMで使用するために開発したもので、どのように使用される予定で、歴史は静かであり、記事の著者にはわかりませんが、この場合、これは重要ではありません。 現在、このアルゴリズムは時代遅れとみなされていますが、いくつかの理由で非常にうまく使用されています。



  • 「古い」地域（西ヨーロッパ、北米）のカードシステムは非常に保守的であり、多くの点で、「十分な」古いシステムで動作します。
  • このアルゴリズムにより、検証値（以下を参照）からPINコード値を復元できます。 特定の条件下で非常に有用




このアルゴリズムの基礎はPINオフセット値（PIN検証値）です。これは、次の値に基づいて取得された暗号です。



  • コントロール値（検証データ、さらにVD）-何らかの値（通常はカード番号の一部ですが、これは必須ではありません）
  • 10進数化テーブル（10進数化テーブル、さらにDT）
  • PIN検証キー（PIN検証キー、以下、PVK）
  • カードのPINコード自体


さらに簡単に説明するために、IBM 3624 PINオフセット方式の場合のPIN検証キーの下に、PVKキーと10進数化テーブル値DTの組み合わせがあります。

PINオフセットを取得するには、PVKキーを使用して制御値VDをポリッシュします。その後、10進数テーブルDTを使用して、取得した値から16桁の10進数のブロックを取得します。

最初のN桁は受信ブロックから取得されます。NはPINの長さです（IBM 3624方式では、最大16桁のPINをチェックできます）。次に、受信ブロックの対応する桁が10を法とする各PIN桁から減算されます。 結果の値はPINオフセット値になります。



HSM


HSM-ハードウェアセキュリティモジュール。安全な環境で暗号化操作を実行するように設計されたソフトウェアとハ​​ードウェアの複合体。 HSM自体には、格納されているデータへの不正アクセスを防ぐための保護が必要です。 その機能には、重要な情報（キー、PINコード値）へのアクセスを防ぐような方法で、カード検証、PIN、暗号メッセージ署名（ MAC ）およびさまざまな暗号化操作を含むさまざまな暗号チェックが含まれます。 ここで説明する価値があるのは、HSMを使用する場合、一般的な場合、オープン形式（つまり、DES / 3DESアルゴリズムに従って操作を実行するのに適した形式）のキー値自体は、この操作が実行された時点でHSM内にのみ存在するということです。 これがどのように達成されるかは、特定のHSMに依存します。 最も一般的なHSMカードシステム（SafeNetとThalesによって製造されたHSM）では、2つのメカニズムが使用されます。



  • いわゆるキー暗号化 さまざまなオプションのHSMマスターキー。 この場合、キー値自体は暗号化された形式で外部メディア（通常は処理システム内）に保存され、操作中にHSMに送信されます。 この場合、操作を実行するとき、HSMは最初にマスターキーを使用してキーを解読し、その後、要求された操作を直接実行します
  • HSMセキュアキーストレージのキーストレージ。 この場合、通常、キー識別子は入力に提供され、HSMがキー識別子を内部ストレージから取得します。


同時に、HSM自体で操作を実行する原則は、どのキーストレージ方式が使用されるかに依存しません。

以下のすべての場所で、これが明示的に述べられていない場合、キーは保護されたHSM形式で送信されます（つまり、HSMを使用して、または識別子としてマスターキーで暗号化されます）。



PINチェック



用語


説明を簡単にするために、いくつかの用語を紹介します。

  • PINブロック-カードのPINコードの値。特別な方法で8バイトのブロックにパックされます。 このプロセスでは暗号化が使用されないことを説明する価値があります。 この場合、パッケージ方法は異なりますが、これは重要ではありません。
  • 暗号化PINブロック-PINブロックの暗号化のために特別に割り当てられたキー（端末キー、銀行キー、支払いネットワークなど）を使用して、DES / 3DESアルゴリズムを使用して暗号化されたPINブロックの値。
  • PIN検証値は、使用されているPIN検証方法に応じて、PVVまたはPINオフセットです。
  • 追加のPIN検証データ-PINおよびPIN検証値を除く、Visa PVV / IBM 3624 PINオフセットアルゴリズムに準拠したPIN検証に必要なデータ。アルゴリズムの説明に記載されているリスト（上記を参照）。




支払システム要件


PIN検証部分では、次を指定できます。

  • ユニットのPINおよびPINのオープン値は、特別に指定されたハードウェアおよびソフトウェアシステム（処理システムまたはEPP側のHSMおよび端末デバイス（ATM、POS端末など）側のその他のひどい略語）の外部で送信、保存、または処理しないでください。
  • 暗号化されたPINブロックは、オンライン/オフライントランザクション処理を担当するシステムで操作が完了した後は保存しないでください（カード自体の発行を担当するシステムもあります。この要件は適用されません）。




PINチェック


既に決定したとおり、PINを確認するには、次のデータが必要です。

  • PIN自体、確認します
  • PIN検証値
  • 追加のPIN検証データ


PINを使用すると、すべてが非常に簡単になります。 前述のとおり、どのような状況でもオープンPIN値を取得することはできません。 したがって、暗号化されたPINブロックのみが必要です。 それに加えて、復号化するためのキーが必要です。 このキーをPPK（PIN保護キーと呼びます。この用語はSafeNet HSMドキュメントから取られています）。



次に、PIN検証値と追加データを決定する必要があります。



最初のオプションは、サービスコードフィールドの後に、テスト値をカードの磁気ストリップに保存することです。 ここでPVVがどこに格納されるかを示すISO 7813の変更されたバージョンを見ることができます 。 トラック形式の説明によると、5文字のPVV値は次のシーケンスを意味することを追加する価値があります：1 PVKI文字と4 PVV文字、およびPINオフセットの場合、PINの5桁のPINオフセット値。 PINの長さが5桁以外の場合、それに応じてPINオフセットサイズが変更されます。 この方法の利点は何ですか。 もちろん-検証に必要なキーを持っている人のためにPINを検証する機能。 ここで、新しいカード製品が発売されると、カードが発行されたキーは通常、決済ネットワークに転送されることに注意してください。 したがって、この方法を使用する場合、PINを検証する機能はカード発行者と支払いネットワークの両方に表示されます。 この方法の欠点には、このオプションにより、カードが再発行されるまでPINカードが静的になるという事実が含まれます。



2番目のオプションは、検証値を何らかのストレージ（通常はカードの認証中にチェックを実行するシステムのデータベース）に格納することです。 この場合、PINを確認するときは、このストアから検証値を抽出し、この値を使用して検証を実行する必要があります。 その結果、この方法を使用する場合、外部システム（同じ支払いシステム）でPIN検証を実行することは不可能であり、 テスト値のストアにアクセスできるシステムでのみ実行されます。 ただし、このようなシステムでは、プラスチックを変更する費用なしでカードのPINコードを変更できます（必要なこと、何をすべきか、この後のどのような問題については、これはこの範囲を超えているため、説明しません）記事）。



必要なすべての情報の受信方法（カード発行者または支払いネットワーク）に関係なく、PIN検証自体はHSMで実行され、検証のためにPPKキーを安全な形式で、PIN検証キーを安全な形式で、暗号化されたPINブロック、 PIN検証値と追加の検証データ。これに応じて、検証結果のみが返されます：正しいPIN、誤ったPIN、その他のエラー。 つまり 検証プロセス中、認証を担当するシステムは、オープンPINコード値とは接触しません。
