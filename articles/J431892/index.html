Veeam Backup＆Replicationを使用して、アップグレードする前に新しいシステムとアプリケーションをテストします

1か月前、MicrosoftはWindows Server 2019の最新バージョンのリリースを発表しました。しかし、GA（一般提供開始）の後、Windows 10 October 2018 Update（別名1809）のように重大な欠陥が発見されました-アップデートをインストールするとデータが失われました（ファイルマイドキュメントは残酷に削除されたため、Windows.oldから復元できませんでした。 メーカーは、問題が修正されるまでリリースをキャンセルせざるを得ませんでした。 そして最後に、11月13日、修復されたバージョンは日の目を見ました 。



また、MicrosoftはSQL Server 2008 R2およびWindows Server 2008 R2のサポートを間もなく完了することを忘れないでください。



当然、ユーザーには新しいシステムへの移行に関して多くの質問があります。

Microsoft Azureクラウドに移行する必要がありますか？ ドメインの機能レベルを安全に上げる方法は？ Azure SQLに移行しますか？ Windows Server 2008 R2を仮想化するか、Azureに移行する必要があるかもしれません。 最新のHyper-Vに移行する必要がありますか？



データセンターで実行されている重要なアプリケーションに対して、ベンダーがサポートするシステムを利用できるようにするには、新しいプラットフォームへの移行が必要です。 したがって、移行が驚くことなく行われることが重要です。 Veeamユーザーは幸運です-そのような操作中のリスクを最小限に抑える良い方法があるので、彼らが言うように、「7回測定し、1回切断します」。



詳細については、猫へようこそ。







「ネットワークサンドボックス」でのテスト



特に、 Veeam DataLabの 「サンドボックス」を使用して、バックアップのリカバリをチェックすることをお勧めします。 彼女は2010年のVeeam Backup＆Replicationリリース（当時はVirtual Labと呼ばれていました）で最初に光を発し、それ以来絶えず更新および開発されています。 今日、それは展開前に新しいソフトウェアをテストするための優れたアシスタントになっています-スタンドアロンのサンドボックスを使用すると、Windows Server 2019へのアップグレード、SQLの新しいバージョンへのアップグレード、またはその他の操作にかかわらず、生産を危険にさらすことなく、システムおよびアプリケーションの計画的な更新と変更をテストできます。



このような「サンドボックス」のデバイスは、次の図に示されています。







「サンドボックス」の作業には次が含まれます。



  • アプリケーショングループ （「 アプリケーションのグループ 」）は、目的のアプリケーションの操作を提供する1つ以上の仮想マシンです。 たとえば、SharePointのWebサーバーとデータベースサーバー、またはドメインコントローラーとExchangeサーバーなどです。
  • プロキシアプライアンスは、プロダクションインフラストラクチャからDataLabサンドボックスを分離する補助プロキシマシンです。 これにより、マスカレードIPアドレスを使用して、本番環境と交差することなく、隔離されたネットワークにIPアドレス空間を作成できます。


このようなサンドボックスの設定については、ユーザーマニュアルで詳しく説明されています。 また、近い将来、Windows Server 2019、新しいSQL Server、およびAzureへの移行をテストするためだけにDataLabを使用する例を含む別のドキュメントが計画されています。







バックアップからクラウドへの再作成



Veeamソリューションに実装されているもう1つの有用なテクノロジーは、Microsoft Azureのバックアップから復元する機能です。 現在、Veeam Backup＆Replicationに組み込まれており、非常に便利です。 さらに、この機能を使用して、新しいシステムとアプリケーション、移行プロセス、ネットワーク接続などをテストできます。 -Azureクラウドのテストインフラストラクチャを実際に上げることができます。 すべてがうまくいけば、Azureへの計画された運用移行中に同じ手順を繰り返します。 この機能について詳しく見ていきましょう。



なぜAzureなのか



Microsoftは、サポート終了後さらに3年間、Windows Server 2008 R2のAzureで拡張セキュリティ更新プログラムを無料で利用できるようにすることを発表しました。 ユーザーは、アプリケーションコードを変更せずにマシンをAzureに移動できます。また、実行するのが早いほど、将来の更新を計画する時間が長くなります。 詳細はこちら 。

バックアップからAzureクラウドへのリカバリを使用すると、Veeamがバックアップできるほとんどすべてのものを転送できることに注意してください：Windowsサーバー、Linuxベースのマシン、vSphereおよびHyper-Vプラットフォーム上の仮想マシンなど。



どのように機能しますか？



Windowsマシンの場合、プロセスは次のようになります。





  1. Azureプロキシを使用している場合、Veeam Backup＆Replicationはそれを有効にします。 このプロキシの詳細については、 こちらをご覧ください 。
  2. Veeam Backup＆Replicationは、バックアップマシンのディスクをVHD形式に変換し、Microsoft AzureクラウドのBLOBストレージにアップロードします。
  3. 次に、これらのディスクがVeeamバックアップサーバーにマウントされます。
  4. ディスクはVM回復のために準備されています。リモートデスクトップのルールがアクティブ化され、ファイアウォールを通過するためのルールが設定され、Microsoft Azureエージェントをインストールするための基盤が準備されています。
  5. Veeam Backup＆Replicationはディスクをアンマウントします。
  6. Azureプロキシが使用されている場合、自動的にシャットダウンします。
  7. Veeam Backup＆Replicationは、Microsoft Azure VMをプロビジョニングされたディスクに登録します。 その後、マシンがオンになり、Microsoft Azureエージェントがインストールされます。


Linuxマシンのリカバリも同様です-ディスクのみが補助VM（ヘルパーアプライアンス）にマウントされます。 詳細はこちらをご覧ください。



リカバリには多くの制限があります。



  • 次のゲストOSがサポートされています。

    • Microsoft Windows Server 2008 / Windows Vista以降
    • Linux- https://docs.microsoft.com/en-us/azure/virtual-machines/virtual-machines-linux-endorsed-distrosによる）。
  • 復元されたVMの1つのディスクのサイズは4095 GBを超えてはなりません。
  • ソースマシンのシステムディスクにGPTパーティション構造がある場合、パーティションの数は4以下にする必要があります。リカバリ中に、そのようなディスクはMBRパーティション構造のディスクに変換されます。
  • Azure Hybrid Use Benefitはサポートされていません。


重要！ Veeamバックアップサーバーで時間が正しく設定されていることを確認してください。正しく設定されていない場合、Veeam Backup＆ReplicationインフラストラクチャにMicrosoft Azureアカウンティングを追加しようとしたとき、またはリカバリを実行したときにエラーが発生する可能性があります。



Microsoft AzureアカウントをVeeam Backup＆Replicationに追加します



リカバリを実行するには、特に、Microsoft AzureアカウントをVeeam Backup＆Replicationインフラストラクチャに追加する必要があります。 同時に、Veeam Backup＆Replicationは、アカウントに関連付けられたサブスクリプションとリソースのデータベースデータを保存し、クラウドへのリカバリ中に、それらを使用してMicrosoft Azureに新しいVMを登録します。 このデータをインポートするための2つのオプションがあります。



  • リソースマネージャーを使用した作業モデル
  • クラシックワークモデル


プロバイダー自体（Microsoft Azure）は、Resource Managerを使用してクラウドに新しいマシンをデプロイすることを推奨しているため、使用します。



Veeam BackupコンソールでMicrosoft Azureアカウントを追加する前に、いくつかの準備手順を完了する必要があります。



  1. 既にMicrosoft Azureアカウントを持っていることを確認してください。 セットアップウィザードはアカウントのみを追加できますが、作成はできません。
  2. [サーバー側のWindows OSを使用している場合] Internet Explorerの設定では、保護モードをオフにする必要があります。オフにしないと、セットアップウィザードの操作中にクラウドにログインできません。
  3. 保護モードを無効にできない場合は、許可サイトのリストに次を追加します。

    • login.live.com
    • login.microsoftonline.com
    • secure.aadcdn.microsoftonline-p.com
    • auth.gfx.ms
    • about：security_veeam.backup.shell.exe


    サーバーマネージャーでInternet Explorerセキュリティ強化の構成を無効にする必要がある場合もあります。

  4. サーバーが配置されているタイムゾーンのVeeamバックアップサーバーに正しい時間が設定されていることを確認します。
  5. Veeam Backupを実行しているマシンでは、Microsoft Azure PowerShell 4.0.2をインストールすることを強くお勧めします。 別のバージョンを使用している場合、問題が生じる可能性があります。 Microsoft Azure PowerShellのバージョンがまったくない場合、Veeam Backupはそれをインストールすることを提案します（詳細は以下を参照）。
  6. ローカルシステムアカウントまたはVeeam Backup Serviceを実行するアカウントのHTTP / HTTPSプロキシを構成する必要があります。 詳細はこちらをご覧ください。


それでは、Azureアカウントの追加を始めましょう。 合意したとおり、Resource Managerでモデルを使用します。



  1. Veeam Backup＆Replicationメインメニューで、 Manage Azure Accountsを選択します 。
  2. [Microsoft Azureアカウントの管理]ウィンドウで、[ 追加 ] をクリックしてウィザードを起動します。





  3. Deployment Modelステップで、 Azure Resource Managerオプションを選択します。
  4. [ 地域]リストから、目的のMicrosoft Azure地域（ Global 、 GermanyまたはChina ）を選択します。
  5. [ 次へ ]をクリックすると、 Veeam Backup＆Replicationは、このマシンでMicrosoft Azure PowerShellが利用可能かどうかを確認します。 そうでない場合は、インストール手順へのリンクとともに警告が発行されます。 インストール後、セットアップウィザードを再起動する必要があります。





  6. サブスクリプションのステップで、「 アカウントの構成 」をクリックします 。 既存のアカウントを入力して、Microsoft Azureポータルにログインする必要があります。 Veeam Backup＆Replicationは、このアカウントの所有者に提供されるサブスクリプションとリソースに関する情報を受け取ります。



    Linuxマシンを復元する場合は、[Linuxベースのコンピューターの有効化を有効にする]をオンにします 。 この場合、Veeam Backup＆Replicationは、復旧に必要なヘルパーアプライアンスをクラウドに展開します。





  7. セットアップウィザードの手順を最後まで実行し、[ 完了 ]をクリックします。


バックアップを準備します



次の種類のバックアップがサポートされています。



  • Veeam Backup＆Replicationを使用して作成された仮想マシンバックアップ（Microsoft WindowsおよびLinux）
  • Veeam Agent for Windowsを使用して作成された物理Windowsマシンのバックアップ。
  • Veeam Agent for Linuxを使用して作成された物理Linuxマシンのバックアップ。


注：物理マシンをAzureに復元するには、そのマシン全体またはバックアップボリュームをバックアップする必要があります。



マシンは、最後の復旧ポイントまたはバックアップチェーンの以前のポイントのいずれかの状態に復元できることに注意してください。 チェーンは、Veeam Backupインフラストラクチャの一部であるリポジトリに保存する必要があります。 既存のバックアップをインポートすることもできます。



リカバリーを実行する



これを行うには、Restore to Azureリカバリウィザードを実行します。



  1. [ ホーム]ビューで、左側のツリーの[ バックアップ]ノードを展開します 。 次に、右側のパネルで、必要なバックアップのノードを展開し、そこで必要なマシンを選択します。
  2. それを右クリックして、 Restore to Microsoft Azureコマンドを選択し、 Deployment Modelウィザードのステップに進みます。





  3. クラウドに復元するときに使用するMicrosoft Azureの展開モデルを示します。 この例では、Azure Resource Managerになります。





  4. サブスクリプションのステップで、次の設定を指定します。

    • サブスクリプションリストには、最初のステップでVeeam Backupに追加したアカウントで利用可能なすべてのサブスクリプションが表示されます。 リソースを使用するサブスクリプションを選択します。
    • [ 場所]リストから、復元したマシンを配置する地域を選択します。 この地域のアカウンティング（サブスクリプション）には、少なくとも1つのストレージシステムがあることを確認してください。
    • リモートリージョンへのリカバリプロセスを高速化する場合は、リストからMicrosoft Azureプロキシを選択して、 AzureプロキシVMを使用することをお勧めします 。 プロキシは、マシンを復元するのと同じリージョンにあることが合理的です。





  5. VMサイズのステップで、 マシンのサイズと、復元されたマシンのディスクが配置されるストレージのアカウントを指定します。

    1. Azure VM構成のリストからマシンを選択し、[ 編集 ]をクリックします。
    2. サイズサイズのリストから、復元されるVMのサイズを選択します。 （デフォルトでは、VMで使用可能なディスクの数に十分な最小値が選択されます）。

      注：ここでは、このVMに割り当てられるCPUコア、メモリリソース、およびディスクスペースの数は、VMのサイズに依存することに注意する必要があります。 Microsoftのドキュメントを詳細に読んでください。
    3. ストレージアカウントのリストから、いわゆる 復元されたVMのディスクを保存するストレージの「ストレージアカウント」。 （選択したVMサイズを覚えておいてください。）VMをクラウドにデプロイするときにAzureプロキシを使用することを指定した場合、このリストには汎用アカウントのみが表示されます（Blobのアカウントは表示されません）。 さまざまな種類のアカウントについては、 ここに記述されています 。




  6. [ リソースグループ]ステップで、復元されたVMの新しい名前を指定できます（デフォルトでは、ソースマシンの名前と一致します）。 [ 名前 ]をクリックして新しい名前を明示的に指定するか、元の名前に接頭辞と\または接尾辞を追加して、それが形成される規則を設定します。



    デフォルトでは、VMの新しいリソースグループが作成されます。 既存のグループにVMを追加する場合は、このステップでこれを行うこともできます。 リストからVMを選択し、[ グループ ]をクリックして、目的のオプションを指定します。



    • VMを既存のリソースグループに配置 （既存のグループに配置）
    • 新しいリソースグループを作成する（新しいグループを作成する）




  7. ネットワークステップで 、復元されたVMを接続するためのネットワークとサブネットを指定します。





最後の手順では、VMを復元する目的を示し、もう一度設定を確認し、[ 完了 ]をクリックして、クラウドへの回復セッションの進行状況を監視します。
