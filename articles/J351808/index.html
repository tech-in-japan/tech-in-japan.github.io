Kubernetesを使用してHyperledger Fabricでブロックチェーンプロジェクトを開発する

こんにちは、ハブロビテス。

私はモスクワにあるIBM Science and Technology Centerの開発者です。 私たちは世界中の他のラボでIBM製品を開発しています。 時間に余裕があり、希望がある場合は、主要な雇用ではないプロジェクトに労働時間の一部を使用することができます。 このアプローチは視野を広げ、創造的なムードをサポートします。 私にとって、この分野はHyperledger Fabricでのブロックチェーンソリューションの開発です。 さらに、このようなプロジェクトは、当社の市場で需要が高まっています。

この記事がブロックチェーンについてではなく、そのようなソリューションを開発するために使用するものについてであることを願っていますが、それでも対象分野から始める価値があります。

この記事は網羅的なものではなく、Hyperledger Composerでブロックチェーンプロジェクトを開発するための実用的なクラウド環境を作成することを目的としています。 実践的な調査の結果、次のインフラストラクチャを取得する必要があります。













ハイパーレジャーファブリックについて少し



無期限に比較できるブロックチェーンプラットフォームは多数あります。 おそらく、ユニバーサルプラットフォームは存在せず、現在のタスク用に選択されたプラットフォームがあります。 私自身、ブロックチェーンプラットフォームから、Hyperledger Fabricを選択しました。 プラットフォームは技術的な観点から私にとって興味深いので、一方で、IBMはHyperledgerコンソーシアムに積極的に関与しています。

Hyperledgerは、Linux Foundationプロジェクトのコンソーシアムです。 Hyperledger Fabric（HLF）は、さまざまなソリューションを構築するためのブロックチェーン技術の実装です。 これは、アクセス許可が制御されたブロックチェーンプラットフォームです。 これは、新しいメンバーの追加が制御されることを意味します。 参加者はネットワーク設計段階で定義され、異なる役割を持つ場合があります。 このようなネットワークの最も理解しやすいシナリオは、多くの独立企業が参加するサプライチェーンです。 例として、ウォルマート、JD.com、IBM、清華大学の食品サプライチェーンの組織化の経験を学ぶことができます。

技術的な観点から見ると、各参加者は組織（またはクラウドインフラストラクチャ）に拡張されたネットワークセクションを持っていますが、他の企業に展開されているネットワークセクションにはアクセスできません。









図からわかるように、各参加者には次のものがあります。







  • 独自のノード（ピア）。ブロックチェーンネットワークの動作を保証し、HLFの一部です。
  • 通常、少なくともWebサーバーとユーザーインターフェイスで構成されるアプリケーション。
  • ノード間の相互作用を提供する接続リンク（注文サービス）。


アプリケーションでは、ユーザーは自分のキーまたはユーザー名とパスワードを使用してログインし、スマートコントラクトにアクセスします。







スマートコントラクトは、分散型台帳にあるデータとやり取りするためのプログラムされたビジネスロジックとルールです。 レジストリには、ネットワーク参加者が作成した一連のトランザクションブロックが含まれます。 後続の各ブロックには、前のブロックのハッシュが格納されます。これにより、情報の一部を変更することが不可能になります。 ブロックの正確性に同意して分散レジストリに追加するプロセスは、コンセンサスと呼ばれます。 さまざまなブロックチェーンプラットフォームには、さまざまなコンセンサスメカニズムがあります。 仕事の証明（PoW） 、 ステークの証明（PoS） 、 ビザンチンのフォールトトレランス（BFT）などです。

バージョン1.0以降、HLFはOrdering Serviceの存在に固有であり、クラスター内の参加者間で分散され、出現ブロックのトランザクションのシーケンスを担当します。 設定されたパラメータに従って、ネットワーク内のトランザクションを収集し、ブロックを形成します。 注文サービスに障害が発生した場合、ネットワーク上のトランザクションはログに記録されなくなりますが、データ自体は変更されません。

これで、基本的な概念を理解できたので、練習に移ります。







独自のHLFネットワークを展開する方法



すべてのHLFコンポーネントはDockerコンテナーで動作し、 Docker ComposeまたはKubernetesを使用してデプロイできます。 さらに、スマートコントラクト自体もDockerコンテナーで実行されます。

HLFを展開するには、既存のkubernetesネットワーク、minikubeを使用するか、クラウドKubernetesをサービスとして使用できます。

IBMクラウドインフラストラクチャで kubernetesネットワークを無料で展開できます。







IBM Cloudでkubernetesネットワークを展開する方法。



IBM Cloudの使用を開始するには、以下を行う必要があります。







  • 登録する
  • IBM Cloud CLIをインストールします
  • IBMクラウドの追加機能をアクティブにするためのクーポンを受け取り、アクティブにします（クーポンはセミマニュアルモードで発行されますが、残念ながらそれらの数は限られています）。
  • kubectlをインストールします 。
  • オペレーティングシステムにGitクライアントをインストールする
  • DockerとKubernetesが何であるかについて読んでください。


IBM Cloudに登録すると、インフラストラクチャを管理するためのグラフィカルインターフェイスにアクセスできるようになります。 アプリケーション（Cloud Foundry、Kubernetes、OpenWhisk、仮想マシン、専用サーバー）をホストするための数百のサービスとアプローチにアクセスできます。 仮想マシンと専用サーバーを使用するには、クレジットカードを追加する必要がありますが、kubernetesの一部を無料で入手できます（上記のクーポンを使用）。

これを行うには、左上隅でメニューボタンをクリックし、[ コンテナ ]を選択します。

kubernetesクラスターコントロールパネル、 Dockerレジストリ 、およびヘルムチャートが表示されます。







左側のメニューから[ クラスター ]を選択すると、kubernetesクラスター作成インターフェイスと、クラスターをデプロイするリージョンの選択が表示されます。 各リージョンに無料のクラスターを作成できます。

[ クラスターの作成 ]ボタンをクリックした後、 クラスターの種類を選択して[ 空き ]を選択し、名前を見つけます。







クラスターの作成には時間がかかるため、コンソールに切り替えるのに時間がかかります（コーヒーを（れる時間はありません）。 IBM Cloud CLI（bxツール）をインストールしたら、アカウントで動作するように設定する必要があります。







  • kubernetesで動作するようにbxのプラグインをインストールします。 

    bx plugin install container-service -r Bluemix
  • IBM Cloudにログインします 

     bx login -a https://api.eu-gb.bluemix.net
  • クラスターを作成した地域を示します（英国と南国は異なる場合があります）。 

     bx cs region-set uk-south
  • クラスターの構成を要求します。myclusterはIBM Cloud GUIで入力した名前です。 

     bx cs cluster-config mycluster
  • 前のステップの結果として表示されるコマンドを実行します。 クラスターにアクセスするためのキーの場所をkubectlユーティリティに通知します（構成ファイルへのパスは変更される場合があります）。 

     export KUBECONFIG=/path/mycluster/kube-config-mil01-mycluster.yml
  • kubernetesワーカーのパブリックIPを確認します。 

     bx cs workers mycluster


これで、あらゆる目的でkubernetesクラスターの使用を開始する準備が整いました。 クラスターは無料で、ブロックチェーンネットワークの展開を妨げることのない機能の制限がいくつかあります。







無料のクラスター制限
  • 1人のワーカー（仮想マシン）のみ。
  • ボリュームの不足;
  • アプリケーションロードバランサーが利用できない。
  • Ingresを使用できない、代わりに[NodePort]を使用できる（ https://kubernetes.io/docs/concepts/services-networking/service/ ）


kubernetesを初めて使用する場合は、最初のDeploymentとサービスのデプロイを試してみる価値があります 。 ブロックチェーンプロジェクトの開発の世界にすでに突入する準備ができている人は、次のステップに進むことができます。







kubernetesにHLFコンテナーを展開する



最近DockerとKubernetesで作業している人たちのために、kubernetesにアプリケーションをデプロイするには2つのものが必要であることを思い出させてください。







  • コンテナーが開始するDockerイメージ。
  • kubernetesでのコンテナの動作のyaml設定。


オープンソースのHypeledger Fabric開発者コミュニティが私たちのために仕事をしてくれました。 HLFを使用したDockerイメージは既にDockerHub上にあり、クラスターに自動的にロードされます。 そして、yaml設定全体がgithubリポジトリにあります 。 執筆時点では、リポジトリにはHyperledger Fabric 1.0.3の構成が含まれています。 コンソールに戻り、次のコマンドを実行します。 







 git clone https://github.com/IBM-Blockchain/ibm-container-service.git cd ibm-container-service/cs-offerings/scripts/ ./create_all.sh --with-couchdb


--with-couchdb



、デフォルトのデータベースの代わりにCouchDBをデプロイするために必要です。 これにより、クエリを使用してHLF Composerからデータを取得できます。







Hyperledger Composerとは何ですか？



Composerはブロックチェーン開発者向けのツールです。 これにより、ブロックチェーンソリューションの開発速度を数か月から数週間に高めることができます。 オンラインのサンドボックスComposer Playgroundがありますが、その中で設計されたすべてのビジネスロジックはブラウザで起動されます（ブロックチェーンエミュレーション）。 この場合、Composerは、kubernetesにデプロイされた稼働中のHyperledger Fabricネットワークに接続されます。







Hyperledger Composerでモデルを作成するには、参加者（参加者）、資産（資産）、およびトランザクション（トランザクション）を記述する必要があります。 このアプローチにより、選択したプラットフォームや使用したツールではなく、ブロックチェーンプロジェクトの開発をタスクの条件に変換できます。 トランザクションコード（ビジネスロジック）はJavaScriptで記述され、新しいスマートコントラクトが起動されると、新しいDockerコンテナーは作成されず、トランザクションコードは解釈のために既存のコンテナーに転送されます。



Hyperledger Composerのもう1つの利点は、記述された資産とトランザクションに基づいてRESTインターフェースを自動的に作成できることです。これにより、ユーザーインターフェースの作成をすぐに開始できます（したがって、MVPを取得できます）。







最初のブロックチェーンプロジェクトの開始



IBMクラウドでHyperledger Composerを開始した後、<public_ip_of_k8s_cluste>：31080でWebインターフェースにアクセスできます。 デフォルトでは、必要なすべてのDockerコンテナーがNodePortテクノロジーを使用してパブリックアクセスに転送されます（開発者向けの環境を開発していることを忘れないでください）。







Composerはgithub-projectに接続されており、既存のプロジェクトの1つをデプロイするか、独自のプロジェクトの開発を開始できます。







その結果、Hyperledger Fabricに基づいたプロジェクトを開発するための無料のクラウド環境を手に入れました。
