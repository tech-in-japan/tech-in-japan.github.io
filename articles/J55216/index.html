前の記事で、単純なサイトのオブジェクトモデルを作成する例では、データベースからのエンティティのシングルダウンロードが、Object :: Create（$ id）構造を持つ識別子を使用して実行され、どのエンティティ（ほとんどの場合はクラス）、どの識別子、これらのエンティティを作成し、極端な場合はデータベースを調べるだけでした。 実際には、存在を推測することしかできない、つまり識別子に関する情報がないエンティティに関心がある場合、識別子によるエンティティの読み込みは問題になります。 さらに、特定の条件を満たす複数のエンティティを一度にロードする必要があります。



たとえば、店舗では、シリアル番号やバーコードで製品を選択することはありませんが、その意味を理解することはありません。関心のある製品のプロパティを調べます。 たとえば、サイトのメインページで、最新のニュースを表示する必要があります。最新のニュースは、データベース（オブジェクトデータモデル）から、Newsクラスの10個のオブジェクトを作成日でソートして選択することになります。 このようなクエリを実装するには、オブジェクトモデルの機能を考慮して、エンティティを選択するための条件（検索条件）を記述するための柔軟な方法が必要です。 条件に基づいて、条件を満たすエンティティの識別子のデータベースから直接選択するためのSQLコードを作成する必要があります。識別子があります-エンティティがあります。



オブジェクトアプローチを使用して、検索語を作成します。 階層は、論理関数やその他の関数を表すオブジェクトから作成されます;理論的には、数学関数として解釈できます。 以下は、「ニュース」クラスのエンティティを取得するための条件の例です。 次のように復号化されます。属性「sys_name」（Q :: Attrib）が値「news」（Q :: Comp）に等しいクラス（Q :: IsClass）に属するエンティティ（Q :: Entity）を選択します。 



$cond = Q::Entity( Q::IsClass( Q::Comp(Q::Attrib('sys_name'), '=', 'news') ) );


条件は、クエリクエリオブジェクトの作成に使用されます。 QueryオブジェクトはSQLコードの条件を解釈し、Execute（）メソッドが呼び出されたときに実行します。 クエリの結果は、見つかったエンティティの配列です。または、Count関数で条件が始まった場合、結果は整数（条件を満たしたエンティティの数）になります。 エンティティの検索は、SQLのLIMITパラメーターで行われるように、番号で制限し、オフセットを実行できます。制限は、クエリを作成するときの2番目と3番目の引数、またはクエリオブジェクトのSetCount（）およびSetStart（）メソッドを使用して実行されます。 



 //     $cond       10 //    (0)  $q = new Query($cond, 10, 0); //  .  –    Object $list = $q->Execute(); if (sizeof($list)>0){ echo $list[0]->getP('head')->getA('value'); //    }


条件はクラスCondAttrib、CondClass、CondComp、CondCount、CondEntity、CondIsExist、CondLink、CondLog、CondNotExist、CondParamのオブジェクトから作成されますが、新しい演算子とCond *クラスを直接使用する代わりに構文を単純化するために、静的Qクラスが使用されますファクトリメソッド）条件オブジェクトが作成されます。



条件には、引数（エンティティ、属性、関係）、エンティティの属性またはプロパティの有無をチェックする関数、エンティティまたはエンティティ自体のプロパティの数をカウントする関数、括弧の役割も果たす論理関数OrおよびAnd、および属性値を比較する関数が含まれますまたは、操作 '='、 '>'、 '<'、 '<='、 '> ='、 '<>'、 'like'を使用した数量計算関数の結果。



SQLとは異なり、フェッチする場合、選択する場所（テーブルから）を指定する必要も、結果の結合やグループ化も必要ありません。 クエリでは、条件の助けを借りて、単に選択するものが示されます。 オブジェクトモデルでは絶対にすべてがオブジェクトであるため、オブジェクトを除いて選択するものはなく、オブジェクトを検索する代わりに、条件を満たすオブジェクトの数を見つけることが唯一の方法です。 したがって、条件は常にエンティティ引数またはエンティティカウント関数で始まります。 



 // -   $cond1 = Q::Entity( ); // -   $cond2 = Q::Count(Q::Entity( ));


最初のケースでは、もちろん、クエリクエリで必要なエンティティの数に制限がない場合、オブジェクトモデルのすべてのエンティティ（すべてのオブジェクト、クラス、および関係）を含む配列が返されます。 2番目の場合、数値が返されます-エンティティの合計数。



絶対にすべてのエンティティに関心があることはめったにないため、エンティティの引数引数条件Q ::エンティティ（$ cond）は条件$ condによって補完され、フィルターとも呼ばれます。 条件を継続するかのように、目的のエンティティに含める必要がある属性とプロパティを決定します。「...を持つエンティティ」



属性



エンティティ属性の値に条件を設定できます。 これを行うには、属性Q :: Attribを示す引数引数と比較関数Q :: Compを使用します。 次の条件は、「sys_name属性がリンク値と等しいエンティティ」を意味します 



 $cond = Q::Entity( Q::Comp(Q::Attrib('sys_name'),'=','link') );


エンティティの値または一般条件のチェックがあいまいな場合、論理関数「AND」および「OR」（Q :: LogAndおよびQ :: LogOr）で条件を補完できます。 論理関数「AND」および「OR」は、ネストの複雑な条件を作成できるブラケットの役割も果たします。



次の条件のクエリは、sys_name属性がリンク値と等しいか、sys_name属性がラベル値と等しく、is_define属性が0であるエンティティを返します。 



 $cond = Q::Entity( Q::LogOr( Q::Comp(Q::Attrib('sys_name'),'=','link'), Q::LogAnd( Q::Comp(Q::Attrib('sys_name'),'=','label'), Q::Comp(Q::Attrib('is_define'),'=',0) ) ) );


前の記事で作成について説明したオブジェクトモデルでは、クラスとリンクはsys_name属性を持っているため、上記の条件の結果は、リンククラスと、プロパティを定義しないシステム名ラベルを持つリンクになります。 結果は、異種エンティティになります。 与えられた条件が実際に適用される可能性は低いが、検索におけるエンティティのタイプに無関心であることをよく示していますが、条件は値が比較される属性の存在を明確にしないことに注意してください。 検索は、人とほぼ同じ方法で実行されます。 たとえば、ランプ、日没、トラクター、ボール、血液、牛乳、CCCPフラグなど、さまざまな種類のオブジェクトの中で赤であるすべてのものを見つけるには、問題なく検索を実行します。主なものは赤です。



値ではなくパラメーター



属性を比較するスカラー値の代わりに、Q :: ParamパラメーターをQ :: Comp比較関数で使用できます。これにより、生成された要求の後でもパラメーター値を設定できます。 したがって、1つのクエリを複数回使用して、パラメータの値を再割り当てする可能性があります。 



 $cond = Q::Entity( Q::Comp(Q::Attrib('sys_name'),'=', Q::Param('par1')) ); //   $q = new Query($cond); //    $q->SetValue('par1', 'news'); //   $list1 = $q->Execute(); //     $q->SetValue('par1', 'name'); //   $list2 = $q->Execute();


属性値を比較することに加えて、関数function Q :: IsExist（）によって、それらの存在に条件を単純に置くことができます。 次の条件は、値属性を持つエンティティを定義します。結果は、文字列と数字のすべてのオブジェクトになります。 



 $cond = Q::Entity( Q::IsExist(Q::Attrib('value')) );


プロパティ



ソートに関しては属性に戻り、エンティティのプロパティの条件について説明します。 プロパティは、接続と、接続を確立するエンティティの組み合わせです。 クエリの条件は、プロパティの不在/存在、またはエンティティのプロパティの数に単純に置くことができます。 しかし、最も興味深いのは、プロパティがオブジェクト（繰り返し）であるため、接続自体または接続が確立されるエンティティの属性とプロパティに条件を設定できることです。 考えてみてください。接続と接続が行われるオブジェクトは、属性とプロパティで最も重要な条件が目的のエンティティと同じになる可能性があるため、すべてを含む条件を作成できます。 たとえば、特定のプロパティを持つエンティティを検索できます。プロパティであるオブジェクトにはオブジェクトとの接続（プロパティ）が必要であり、オブジェクトには特定のテキストを含むvalue属性が必要です。 これらすべてを条件にすると、特にシステム名がどのような種類のプロパティであるかを指定することさえできません。 検索は、条件からわかっているものに対してのみ動作します。 一般的に、抽象性、汎用性、柔軟性。



それでは、必要なエンティティのプロパティの存在をチェックする条件から始めましょう。 実際、条件でプロパティを指定するという事実によってプロパティの存在が決まるため、条件関数Q :: IsExistを使用する必要はありません。 



 $cond = Q::Entity( Q::Property( ) );


同じこと： 



 $cond = Q::Entity( Q::IsExist(Q::Property()) );


上記の条件はプロパティの存在を決定し、指定されていないものがあるため、この条件でのクエリは、少なくとも1つのプロパティを持つすべてのエンティティになります。つまり、「プロパティを持つエンティティ」です。



逆は、プロパティがないことを確認しています。 プロパティがないことを判断するには、Q :: NotExist条件関数を使用します。 以下の条件の結果は、プロパティを持たないすべてのエンティティになります。 



 $cond = Q::Entity( Q::NotExist(Q::Property()) );


特定のプロパティを調整する方法は？ Q ::プロパティ（$ link_cond、$ entity_cond）には、接続の条件と接続が確立されるエンティティの条件をそれぞれ決定する2つの引数があります。 繰り返しますが、通信もオブジェクトなので、その条件はエンティティの場合とまったく同じです。 たとえば、属性条件。



次の条件は、プロパティの関係に属性条件を使用します。 この条件でのクエリは、複数のプロパティ（サイズ= 0は複数を意味します）を持つエンティティを返し、ニュースのみがそれを持ちます。 より正確には、サイズ属性がゼロに等しい接続は、コメントを定義するニュースクラスにあり、最初のニュースにはコメントが2つあり、2番目のニュースにはコメントがありません（前の記事の図を参照）。 



 $cond = Q::Entity( Q::Property( Q::Comp(Q::Attrib('size'),'=',0)) );


次に、リレーションシップが作成されるエンティティに条件を追加します。リレーションシップが作成されるエンティティのプロパティに条件を設定します。 次の条件は、複数のプロパティを持つエンティティを返します。プロパティであるエンティティは、プロパティを指定せずに独自のプロパティを持ちますが、値属性に「First」というテキストのフラグメントが含まれるエンティティに関連付けられます。 条件は抽象的であることが判明しました。 ニュースオブジェクトにはコメントがあり、そのタイトルは「最初」という単語で始まるため、実際には最初のニュースになります。 ニュース項目と解説の関係は、条件にちょうど合う 



 $cond = Q::Entity( Q::Property( Q::Comp(Q::Attrib('size'),'=',0), //  Q::Entity( //   ,     Q::Property( null, //     Q::Entity(Q::Comp(Q::Attrib('value'),'like','%%')) ) ) ) );


プロパティの数をカウントする条件の例を示しましょう。 次の条件は、4つ以上のプロパティを持つエンティティを返します。 彼らは最初のニュースとクラスのカテゴリになります。 このニュースには、カテゴリ、見出し、テキスト、および2つのコメントへのリンクがあり、合計5つのプロパティがあります。 カテゴリクラスには、名前、説明、およびオブジェクトのプロパティを定義する4つの関係（合計6）があります。この条件では、考慮する関係を指定しませんでした。 



 $cond = Q::Entity( Q::Comp(Q::Count(Q::Property()), '>', 4) );


プロパティに条件を追加することを制限するものはありません;目的のエンティティでプロパティの役割を果たすエンティティでプロパティカウントを実行することもできます。



本質である本質



実際には、他のエンティティのプロパティであるエンティティを検索することがしばしば必要です。 たとえば、特定のニュースアイテムに属するコメントを検索します。 これらの目的のために、プロパティ引数引数と同様に条件引数Q :: IsPropertyが使用されますが、唯一の違いは、関係と関係を所有するエンティティを定義することです。 そうでなければ、すべてが同じです-コミュニケーションの条件、本質の条件。 必要なエンティティを参照するエンティティの数を計算し、少なくとも2人の所有者が必要であるという条件を設定することもできます。 



 $cond = Q::Entity( Q::Comp(Q::Count(Q::IsProperty()), '>=', 2) );


指定された条件は、クラス「string」、「long_string」、「category」、および「Events」という名前のルートカテゴリのオブジェクトを返します。これらすべてのエンティティには複数の所有者がいます。



クラスメンバーシップ



別の重要でよく使用される条件は、Q :: IsClassクラスに属していることです。 クラスに属することは実際にはオブジェクトのプロパティですが、チェックするための特別な方法が必要です。 継承の階層を考慮する必要があります。つまり、クラスメンバシップの検証は、接続が確立されるエンティティの通常の検証に要約されません。 たとえば、ニュースオブジェクトはクラス「news」に属しますが、クラス「content」および基本クラス「id」にも属します。 以下の条件はすべてのコメントを返します。 



 $cond = Q::Entity( Q::IsClass( Q::Comp(Q::Attrib('sys_name'), '=', 'comment') ) );


クラスはオブジェクトでもあるため、エンティティだけでなくクラスの条件も調整できます。属性とプロパティを確認してください。



Q :: LogAndおよびQ :: LogOr論理関数の使用は、属性値を比較することによってのみ実証され、プロパティQ :: Property、Q :: IsProperty、クラスQ :: IsClassおよびQ関数に属する条件引数を含めることができます。 :: IsExistおよびQ ::必要な柔軟性を提供するNotExist。



仕分け



ソートなしの検索は良くありません。 並べ替えを使用すると、関連するトピック、人気のある製品などを見つけて適用し、サイトのメインページで人気のあるコンテンツを作成できます。 しかし、条件がエンティティの構造について何も指定しない場合に、条件の考えられるすべてのバリアントを考慮して、必要なエンティティをソートする方法は？ 実際、ソートの定義は非常に単純であり、検索SQLクエリの実行時にDBMSで直接実行されます。



当然、スカラー値-属性値でソートできます。 並べ替えオプションは、Q :: Attrib属性の条件引数引数で指定されます。 したがって、並べ替えられた属性の定義と並べ替えパラメーター（降順または昇順、並べ替え順序-並べ替え可能な属性が複数ある場合は、並べ替え順序を指定できます）、および必要なエンティティに存在する属性の条件が結合されます。



目的のエンティティに必ずしも属さない属性でソートできます。少なくとも、目的のエンティティとの一連のリレーションシップに間接的に関連付けられているエンティティの属性でソートできます。 属性の関係は、プロパティであるリレーションとオブジェクトに設定できることを覚えていますか？ そのため、属性で並べ替えることもできます。 サイト上のニュースは、作成日とニュースの著者の評価によって同時にソートできることがわかりました。 または、購入した製品を、購入した店舗の名前で並べ替えるようなものです。製品に直接関係のないプロパティで並べ替えます。



理論的には、エンティティのプロパティの数、たとえばニュースのコメントの数でソートすることは可能ですが、一時的にこの機能は実装されていません。



また、属性（その名前）は並べ替えパラメーターで指定されますが、同時に、属性値の型についての説明は必要ありません。 特に、value属性は異なるクラスのオブジェクトに存在し、最も重要なことには、異なるクラスの値のタイプに違いがあります。 異なる長さ、整数、実数の文字列。 ただし、このような状況でも、ソートは機能します。 次の条件は、value属性を持つオブジェクトのソートされたリストを返します。 ソートは、値属性に従って昇順で行われます。 2番目の引数（true）はこの属性によるソートを決定し、3番目（1）-ソート順序（いくつかのソート可能な属性に関連）および最後の属性ソートタイプ（false-昇順、true-降順） 



 $cond = Q::Entity( Q::IsExist(Q::Attrib('value', null, true, 1, false)) ); //       $q = new Query($cond,0,0); $list = $q->Execute();




おわりに



検索用語を作成するための提案されたアプローチは複雑に思えるかもしれませんが、それはSQLおよびプログラミング言語で行われているように、条件の文字列記述にすべて慣れているためです。 オブジェクトアプローチは、第一に、プログラム分析とSQLクエリの自動生成に便利です。第二に、意味のあるGUIインターフェイスを簡単に作成できるようになります。



さらに大きな懸念事項は、パフォーマンスの問題と、これらの非常に柔軟な条件から生じるSQLクエリです。 要求はそれほど怖いものではありませんが、複数のLEFT JOIN-nov sinや他のデザインを使用することにより、柔軟性を実現することは困難です。
