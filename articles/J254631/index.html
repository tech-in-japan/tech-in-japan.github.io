最初のステップで新しいプロジェクトを作成する場合、最も簡単な方法は、アプリケーション配布キットをロードするときに自動入力フォームを使用することです。



私の意見では、少し不便なのは、次のステップはアプリケーションの最初のリビジョンのディストリビューションを個別にダウンロードすることであるため、ダウンロードされたディストリビューションが意図された目的のためにサービスによって使用されないことです（リビジョンはバージョン/アセンブリを示すためにサービスによって使用される用語です）



概要ページには、プロジェクトに関する基本情報と統計の概要が含まれています。



注：システム内の通知は、デフォルトで、 管理者またはマネージャーの役割を持つユーザーの監視モードと、 QAまたは開発者の役割を持つユーザーの非 監視モードに設定されます。 無視モードは、システムが通知をまったく送信しない場合にも使用できます。







上記のモードの説明については、 よくある質問のページ「通知ステータスとは何ですか？ どのような場合に通知されますか？」。



新しいリビジョンを追加する場合、最初のステップはアプリケーション配布パッケージ（* .ipa）をダウンロードすることです。



次のステップでは、改訂に関する基本情報が設定されます。



ダウンロード後、リビジョンは非アクティブになります。つまり、テストデバイスにインストールすることはできません。 テストを開始するには、[ スタート ]ボタンをクリックする必要があります。



そして、次の画面で、この改訂版が利用できるチームメンバーを指定します（同時に、すべての参加者に通知を送信することができます）。



その後、リビジョンはステータスが進行中になり、デバイスにインストールできます。



注：プロジェクトにデフォルトのユーザーグループがある場合、それに含まれるチームメンバーがすぐに選択されます（最初の列の位置をオンに切り替えます）。そのようなグループがない場合は、ユーザーを手動で選択する必要があります。 各リビジョンへのアクセスは、全員または一部のユーザーに対して停止できます。これにより、プロジェクトの作業中に発生する組織上の問題を効果的に解決できます。 また、Ubertestersを初期化するときに、アプリケーションへのアクセスを許可することはできますが、サービスの機能であるUbertestersLockingModeDisableUbertestersを使用する機能を無効にすることができます。



2月のサービス更新（ 詳細 ）が追加されました。これには、改訂の説明を編集する機能が含まれます。



先ほど言ったように、チームへの招待状は電子メール（管理-チーム-メンバーの追加）で送信され、CSVファイルからインポート機能を使用することでこのタスクを促進できます。



ユーザーは、グループに追加するだけでなく、3つのロールのいずれかを定義できます（これらは、限られた数のユーザーにアプリケーションを配布するために使用されます）。



ご注意 一般に、サービス内では、ロールは次のように階層的に分散されています。組織レベルではAdminとMemberの 2つのロールがあり、プロジェクトレベルではMemberはQA 、 DeveloperおよびManagerです。 重要なニュアンス：各プロジェクトの最初のリビジョンは、 管理者またはマネージャーの役割を持つユーザーがロードする必要があります。 チームメンバーはさらに改訂版をダウンロードできます（ QAを含む）。 このサービスは、特定の企業でビジネスプロセスを構築するために考えられるすべてのオプションを考慮するように設計されています。 役割の詳細については、 よくある質問ページをご覧ください。

追加機能



ソフトウェアをテストするとき、最も完全で信頼できる情報を取得することが非常に重要であることをもう一度強調します。 残念ながら、現時点では、Ubertestersにはクラッシュレポートをシンボル化する機能がありません（この点で他のサービスと比較して失われます）が、荷物には別の、多くの場合より便利な機能があります：テスト中のアプリケーションから直接チケットを作成する機能、およびまた、スクリーンショットも提供します。



この機能は、デフォルトで画面の右上隅に配置される丸いボタンを押した後にユーザーが使用できますが、他の便利な場所に自由に移動できます。











サービスのエラーを追跡するシステムは、特殊なサービスに完全に代わるものではないかもしれませんが、各チケットにスクリーンショットを添付する機能と組み合わせることで、そのタスクに完全に対処できます。











各スクリーンショットには、メモ（鉛筆、マーカー、テキストラベル、矢印、スクリーンショットの一部を切り取る機能などの編集ツール）を添付できます。 撮影されたスクリーンショットはデバイスに残り、再利用できます。 デバイスの標準写真ライブラリからスクリーンショットを追加することもできます。



注：新しいチケットに関する通知を含むレターには、作成中にスクリーンショットがアップロードされたかどうかに関する情報は含まれません。 私の意見では、このようなデータ、少なくとも添付されたスクリーンショットの数があると便利です。



サービスWebサイトでは、すべてのチケットに関する情報が非常に明確に表示されます。1つのリビジョンおよびすべてのリビジョンの一般的な統計を表示できます。



ステータスに関する統計に加えて、情報はタイプ（バグ、機能、改善、タスク、クラッシュ）、優先度、およびエラーの再生頻度別に表示されます。



重要な利点は、各チケットが特定のリビジョンに関連付けられていることです。これにより、テスターからのフィードバックを処理しやすくなり、収集される情報の価値が高まります。

ダウンロードを自動化するためのAPIへのアクセスは、プロファイル設定で個別に有効化されます （APIの説明はhttp://ubertesters.com/upload-api/にあります ）



curlを使用してAPIを使用できます。 



curl -X POST http://beta.ubertesters.com/api/client/upload_build.json -H "X-UbertestersApiKey:PERSONAL_API_KEY" -F "file=@upload.ipa" -F "title=build title" -F "notes=build notes" -F "status=in_progress" -F


パラメータ： 

  X-UbertestersApiKey- required, your personal 'API access KEY'. Please see item 1 file - required, path to your build file (.IPA or .APK) title - optional, your build title notes - optional, build description, plain text status - optional - pending - default, create a new revision only - in_progress - create and start revision stop_previous - optional - true - stop all previous revisions ( if any) - false - keep current statuses of previous revisions


注：完全自動モードで新しいバージョンをダウンロードして公開するには、プロジェクト設定（ 管理-配布グループ ）でデフォルトでテスターのグループを作成する必要があります。この場合、「status = in_progress」キーを追加すると、新しいバージョンがすぐに使用可能になりますこのグループのすべてのメンバーに。 一度に複数のグループをデフォルトグループとして設定できます。
