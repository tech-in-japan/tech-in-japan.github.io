gRPC-リモートプロシージャコール用のGoogleのフレームワーク



リモートプロシージャコールの場合、物事はよく知られているコミック「14 Standards」とまったく同じでした。古代DCOMとCorba、奇妙なSOAPと.NET Remoting、現代のRESTとAMQP（はい、知っています） -これは正式にはRPCではありません。用語を説明するために、最近特別なトピックを作成しましたが、それでもすべてRPCとして使用され、何かがアヒルのように見え、アヒルのように浮かぶ場合は- わかります ） 。



そしてもちろん、コミックブックのシナリオに完全に準拠して、Googleは市場に参入し、最終的に最も正確な最後のRPC標準を作成したことを発表しました。 Googleは理解できます-21世紀にペタバイトのデータを古くて非効率的なHTTP + RESTで駆動し続けると、すべてのバイトでお金を失うのはばかげています。 同時に、他の誰かの標準を採用して、「これ以上良いものを思い付かなかった」と言うのは、彼らのスタイルではまったくありません。



したがって、「gRPCリモートプロシージャコール」の略であるgRPCを満たします。これは、Googleからのリモートプロシージャコールの新しいフレームワークです。 この記事では、以前の「14標準」とは異なり、世界（または少なくともその一部）をキャプチャし、Windows + Visual StudioでgRPCビルドを構築しようとする理由について説明します指示は必要ないこと-公式文書には5つの重要なステップがありませんが、それなしでは何も起こりません）、また、リクエストと回答を交換する簡単なサービスとクライアントを作成してみてください。



なぜ別の規格が必要なのですか？



まず、周りを見てみましょう。 何が見えますか？ REST + HTTP / 1.1が表示されます。 いいえ、すべてがありますが、クライアントとサーバー間の通信の大空の4分の3を閉じるのはこのクラウドです。 よく見ると、RESTは95％のケースでCRUDに縮退していることがわかります。



その結果、次のことができます。

  • HTTP / 1.1プロトコルの非効率-非圧縮ヘッダー、完全な双方向通信の欠如、OSリソースの使用に対する非効率的なアプローチ、余分なトラフィック、余分な遅延。
  • データとイベントモデルをREST + CRUDで取得する必要があります。これは、しばしば地球上の風船のようになり、Yandexに、間違いなく非常に良い記事を書くことを強制します。エレメンタル-PUTまたはPOSTを召喚する呪文を唱えるのはなぜですか？ そして、どの種類のHTTPコードを返すのかというと、「3セル進んで新しいカードを引く」という意味です。




これがgRPCの始まりです。 箱から出してすぐに：

  • データ型とシリアル化を記述するためのツールとしてのProtobuf 。 実際には非常にクールで実証済みのもの。 実際問題として、パフォーマンスが必要な人は、Protobufを服用する前に、輸送を別に気にします。 これですべてが完了しました。
  • トランスポートとしてのHTTP / 2。 そして、これは非常に強力な動きです！ 完全なデータ圧縮、トラフィック制御、サーバーからのイベントのトリガー、複数の並列リクエストに1つのソケットを再利用することの魅力はすばらしいものです。
  • 静的パス-「サービス/コレクション/リソース/リクエスト」はもうありませんか？ パラメーター=値。」 これで、「サービス」とその中身だけが、モデルとそのイベントの観点から説明されます。
  • メソッドをHTTPメソッドにバインドせず、戻り値をHTTPステータスにバインドしません。 欲しいものを書いてください。
  • SSL / TLS、OAuth 2.0、Googleサービスを介した認証、および独自の（たとえば、2要素）
  • 9言語のサポート：C、C ++、Java、Go、Node.js、Python、Ruby、Objective-C、PHP、C＃プラス、もちろん、少なくともBrainfacでバージョンを取得して実装することを禁止する人はいません。
  • Googleの公開APIでのgRPCのサポート。 すでに一部のサービスで機能しています 。 いいえ、RESTバージョンはもちろん残ります。 しかし、選択肢がある場合は自分で判断してください-たとえば、モバイルアプリケーションのRESTバージョンを使用して、1秒でデータを提供するか、同じ開発コストで0.5秒動作するgRPCバージョンを使用します-何を選択しますか？ そして、あなたの競争相手は何を選びますか？




GRPCアセンブリ



必要なもの：

  • Git
  • Visual Studio 2013 + Nuget
  • CMake




コードを受け取ります



  1. GithubからgRPCリポジトリを取得します
  2. コマンドを実行する 

    git submodule update --init


    -これは、依存関係（protobuf、opensslなど）をダウンロードするために必要です。




Protobufをまとめる



  • grpc \ third_party \ protobuf \ cmakeフォルダーに移動し、そこにビルドフォルダーを作成します。
  • コマンドを実行する

    cmake -G "Visual Studio 12 2013" -DBUILD_TESTING = OFF ...
  • Visual Studioで前の手順で作成したprotobuf.slnファイルを開き、アセンブルします（F7）。

    この段階で、貴重なアーティファクト-protoc.exeユーティリティを取得します。これは、gRPCをリンクするときに必要なデータとlibファイルのシリアル化/逆シリアル化コードを生成するために必要です。
  • grpc \ third_party \ protobuf \ cmake \ build \ Debugフォルダーをgrpc \ third_party \ protobuf \ cmakeフォルダーにコピーします。

    もう一度-Debugフォルダーを1レベル上にコピーする必要があります。 これは、gRPCとProtobufのドキュメントにある種の矛盾です。 Protobufは、ビルドフォルダーですべてをビルドする必要があると言っていますが、gRPCプロジェクトのソースはこのフォルダーについて何も知らず、grpc \ third_party \ protobuf \ cmake \ Debugで直接Protobufライブラリを探します




gRPCを構築する



  1. ファイルgrpc \ vsprojects \ grpc_protoc_plugins.slnを開いて収集します。

    前の手順でProtobufビルドを正しく完了した場合、すべてがスムーズに進みます。 これでprotoc.exeのプラグインがあり、シリアル化/逆シリアル化コードを生成できるだけでなく、gRPC機能を追加できます（実際には、リモートプロシージャコール）。 プラグインとprotoc.exeは、grpc \ vsprojects \ Debugなどの1つのフォルダーに配置する必要があります。
  2. ファイルgrpc \ vsprojects \ grpc.slnを開いて収集します。

    アセンブリの過程で、Nugetは必要な依存関係（openssl、zlib）を開始してダウンロードする必要があります。 Nugetがない場合、または何らかの理由で依存関係をダウンロードしなかった場合、問題が発生します。

    ビルドの最後に、gRPCを介した通信のためにプロジェクトで使用できるすべての必要なライブラリが用意されます。




私たちのプロジェクト



gRPCを使用してHabrahabr用のこのようなAPIを作成しましょう

次のメソッドがあります。

  • GetKarmaは、ユーザー名を含む文字列を受け取り、そのカルマの値を含む小数を返します
  • PostArticleは、すべてのメタデータを含む新しい記事を作成するリクエストを受信し、公開結果（記事へのリンク、公開時刻、公開が失敗した場合のエラーテキストを含む構造）を返します。




これはすべて、gRPCの観点から説明する必要があります。 次のようになります（タイプの説明については、 protobufのドキュメントを参照してください）。 



 syntax = "proto3"; package HabrahabrApi; message KarmaRequest { string username = 1; } message KarmaResponse { string username = 1; float karma = 2; } message PostArticleRequest { string title = 1; string body = 2; repeated string tag = 3; repeated string hub = 4; } message PostArticleResponse { bool posted = 1; string url = 2; string time = 3; string error_code = 4; } service HabrApi { rpc GetKarma(KarmaRequest) returns (KarmaResponse) {} rpc PostArticle(PostArticleRequest) returns (PostArticleResponse) {} }




grpc \ vsprojects \ Debugフォルダーに移動し、そこで2つのコマンドを実行します（ところで、公式ドキュメントではこの場所にエラーがあり、引数が正しくありません）。 

 protoc --grpc_out=. --plugin=protoc-gen-grpc=grpc_cpp_plugin.exe habr.proto protoc --cpp_out=. habr.proto


出力では、4つのファイルを取得します。

  • habr.pb.h
  • habr.pb.cc
  • habr.grpc.pb.h
  • habr.grpc.pb.cc




これは推測するのが難しくないので、上記のプロトコルを使用してメッセージを交換できる将来のクライアントおよびサービスのワークピースです。



プロジェクトを作成しましょう！



  1. Visual Studioで新しいソリューションを作成し、HabrAPIと呼びます。
  2. 2つのコンソールアプリケーション、HabrServerとHabrClientを追加します。
  3. 前の手順で生成されたhおよびssファイルをそれらに追加します。 4つすべてをサーバーに含める必要があり、クライアントにはhabr.pb.hおよびhabr.pb.ccのみが含まれます。
  4. grpc \ third_party \ protobuf \ srcおよびgrpc \ includeフォルダーへのパスを追加フォルダーの追加のプロジェクト設定に追加します。
  5. 追加のライブラリディレクトリのプロジェクト設定でgrpc \ third_party \ protobuf \ cmake \ Debugにパスを追加します。
  6. 追加の依存関係のプロジェクト設定にlibprotobuf.libライブラリを追加します
  7. リンクのタイプを、Protobufが作成されたのと同じタイプに設定します（[コード生成]タブの[ランタイムライブラリ]プロパティ）。 この時点で、必要な構成でProtobufをビルドしなかったことが判明する場合があり、戻って再構築する必要があります。 私はあちこちで/ MTdを選択しました。
  8. Nugetを介してzlibとopensslに依存関係を追加します。




今、すべてが順調に進んでいます。 確かに、まだ何も機能していません。



お客様


ここではすべてが簡単です。 まず、habr.pb.hで生成されたスタブから継承されたクラスを作成する必要があります。 次に、GetKarmaメソッドとPostArticleメソッドを実装します。 第三に、それらを呼び出して、たとえば、コンソールに結果を表示します。 次のようになります。 



 #include <iostream> #include <memory> #include <string> #include <grpc/grpc.h> #include <grpc++/channel.h> #include <grpc++/client_context.h> #include <grpc++/create_channel.h> #include <grpc++/credentials.h> #include "habr.grpc.pb.h" using grpc::Channel; using grpc::ChannelArguments; using grpc::ClientContext; using grpc::Status; using HabrahabrApi::KarmaRequest; using HabrahabrApi::KarmaResponse; using HabrahabrApi::PostArticleRequest; using HabrahabrApi::PostArticleResponse; using HabrahabrApi::HabrApi; class HabrahabrClient { public: HabrahabrClient(std::shared_ptr<Channel> channel) : stub_(HabrApi::NewStub(channel)) {} float GetKarma(const std::string& username) { KarmaRequest request; request.set_username(username); KarmaResponse reply; ClientContext context; Status status = stub_->GetKarma(&context, request, &reply); if (status.ok()) { return reply.karma(); } else { return 0; } } bool PostArticle(const std::string& username) { PostArticleRequest request; request.set_title("Article about gRPC"); request.set_body("bla-bla-bla"); request.set_tag("UFO"); request.set_hab("Infopulse"); PostArticleResponse reply; ClientContext context; Status status = stub_->PostArticle(&context, request, &reply); return status.ok() && reply.posted(); } private: std::unique_ptr<HabrApi::Stub> stub_; }; int main(int argc, char** argv) { HabrahabrClient client( grpc::CreateChannel("localhost:50051", grpc::InsecureCredentials(), ChannelArguments())); std::string user("tangro"); std::string reply = client.GetKarma(user); std::cout << "Karma received: " << reply << std::endl; return 0; }


サーバー


サーバーにも同様の話があります-habr.grpc.pb.hで生成されたサービスクラスから継承し、そのメソッドを実装します。 次に、特定のポートでリスナーを起動し、顧客を待ちます。 このようなもの： 



 #include <iostream> #include <memory> #include <string> #include <grpc/grpc.h> #include <grpc++/server.h> #include <grpc++/server_builder.h> #include <grpc++/server_context.h> #include <grpc++/server_credentials.h> #include "habr.grpc.pb.h" using grpc::Server; using grpc::ServerBuilder; using grpc::ServerContext; using grpc::Status; using HabrahabrApi::KarmaRequest; using HabrahabrApi::KarmaResponse; using HabrahabrApi::PostArticleRequest; using HabrahabrApi::PostArticleResponse; using HabrahabrApi::HabrApi; class HabrahabrServiceImpl final : public HabrApi::Service { Status GetKarma(ServerContext* context, const KarmaRequest* request, KarmaResponse* reply) override { reply->set_karma(42); return Status::OK; } Status PostArticle(ServerContext* context, const PostArticleRequest* request, PostArticleResponse* reply) override { reply->set_posted(true); reply->set_url("some_url"); return Status::OK; } }; void RunServer() { std::string server_address("0.0.0.0:50051"); HabrahabrServiceImpl service; ServerBuilder builder; builder.AddListeningPort(server_address, grpc::InsecureServerCredentials()); builder.RegisterService(&service); std::unique_ptr<Server> server(builder.BuildAndStart()); std::cout << "Server listening on " << server_address << std::endl; server->Wait(); } int main(int argc, char** argv) { RunServer(); return 0; }
