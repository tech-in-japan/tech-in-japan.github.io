Broadway-ブラウザでのGTK3インターフェイスのレンダリング（HTML5）

ローカルへのインストールが常に可能であるとは限らないアプリケーションへのアクセスを提供することが必要な場合があり、必ずしも必要ではありません。 おそらくここでの最良の解決策は、JS / PHPなどのWebインターフェイスです。 しかし、場合によっては他のより簡単な方法がありますか？ 特に、アプリケーションの移植性を維持する必要がある場合、およびそのような機能を実装するためにコードで余分なことをほとんど行わないほうがよい場合です。

この機会はブロードウェイによって提供されます-長い間新しいものではありませんが、GTK3のバックエンドの影にとどまります。これにより、すべてが長い間踏まれているように思われる場所に新しい機能をもたらすことができます。













GTK Broadwayとは







ブロードウェイについては、すでに2011年にhabrで語られました。 ただし、それ以降、このオプションの照明領域ではほとんど変更されていません。

UPD、ありがとうawoland ：

最初は、これは6.3 X Window System 11（X11R6.3）リリースのテクノロジーと内部コード名です。

「ブロードウェイは、X Window System 11リリース6.3（X11R6.3）開発努力の内部コード名でした。 X11R6.3プロジェクトは、Xを使用してWorld Wide Web上の対話型アプリケーションを作成およびアクセスできるようにするための画期的なイニシアチブでした。 X11R6.3を使用してWebにリンクされたアプリケーションは、現在静的HTMLドキュメントにアクセスするのに使用されているのと同じツールであるWebブラウザーで検索、アクセス、実行できます。



主なアイデアは、通常の既におなじみのGTK3に基づいて単一のバージョンのコードを作成することです。これは、変更を加えることなく、同時にかつ事実上、古典的なグラフィカルアプリケーションとして機能し、ブラウザーでHTML5とWebソケットを使用してインターフェイスをレンダリングできます。 バージョン3.8以降では、接続用のパスワードと、1つのサーバーで多くのアプリケーションを実行する機能を設定できるようになりました。



GTK3のバージョン



公式には、BroadwayはGTK3でリリースされましたが、バージョン3.8以降でのみ、このサブシステムは迷惑なエラーを取り除きました。 3.10.7を使用します。これは、使用の原則を変更し、多くのエラーを修正し、HTTPサーバーを別のアプリケーションとして作成したためです。 したがって、3.10についてお話します。すべて同じで、すべてが彼に届くからです。



動作原理







GTK3と一緒に、GTK3（broadway d ）と統合されたHTTPサーバーがインストールされます。 起動時に、たとえば/run/user/1000/broadway1.socketなどのソケットを作成し、GTK3アプリケーションがこのソケットに接続するのを待ちます。

別のポート（画面番号）を指定し、接続のパスワードを設定できます （> = GTK 3.8）。



なぜそれが必要ですか



このような操作モードは、現在標準となっているPHP / JS / Javaとそれらのilkに基づいた標準インターフェースを置き換えるふりをしません。 ただし、この方法では、たとえば仮想マシンの形でサービスを作成できます。これにより、サーバー側で高いパフォーマンスを確保しながら、特別なインターフェイスを開発する時間を無駄にすることなく、ユーザーにコンピューティングサービスまたはユーティリティへのアクセスを提供します。 たとえば、XenServer仮想マシンにアクセスするためのコンソールを作成するためにそれを使用し、現在は望遠鏡のカメラへのリモートアクセスを実装しています。

セグメンテーション違反につながります。 次の注意点は、ブラウザで行われるすべてがホストマシンに関連して行われるということです。 たとえば、ローカルブラウザマシンでファイルを選択し、アプリケーション側でそれを処理することはできません。 または、libnotify呼び出しにより、ブラウザを備えたマシンではなく、ホストにポップアップが表示されます。

一方、サーバーリソースへのアクセスの簡素化など、他の利点もありますが、たとえばgtk_file_chooser_set_local_only、gtk_file_chooser_set_filterを介してセキュリティに注意を払い、別のユーザーのjailを構成し、このユーザーのWebバージョンを起動する必要があります。最小のディレクトリ構造。

練習する





アプリケーションでBroadwayを使用する可能性を検討する方法を検討してください。

Gladeをインターフェイスコンストラクターとして使用します。 1つのアプリケーションで2つのバージョンのインターフェイスを使用します。1つは通常の使用用で、もう1つはブラウザ用です。ブラウザおよび/またはモバイルデバイスでの作業のニュアンスを考慮しようとします。

GTKでは、メモリバッファからインターフェイスをロードできます。これを使用します。 Makefileで、Gladeインターフェースが保存されているXMLファイルからの変換をuint8_t配列に追加します。これにより、アプリケーションがロードされます。 これにより、すべてを1つの実行可能ファイルに保存できます。 



 all: xxd -i desktop.glade ../src/desktop.h; xxd -i web.glade ../src/web.h; make -C ../src clean: make -C ../src clean




これで、ヘッダーファイルにインターフェイスを持つ配列ができました。 メインループは標準です。 



 int main(int argc, char *argv[]) { gtk_init(&argc, &argv); GtkWidget *main_window = glade_init( ); gtk_widget_show(main_window); gtk_main( ); }




返品について事前に回答します-std = c99で行っています

次に、このタイプまたはそのタイプのインターフェースをいつ使用するかを理解する必要があります。

ここで、Broadwayが使用されている場合、GDK画面は「ブラウザ」と呼ばれます。 



 #ifdef __WIN32 G_MODULE_EXPORT #endif gboolean is_run_in_a_browser (void) { GdkScreen *current_screen = gdk_screen_get_default(); char *screen_name= gdk_screen_make_display_name(current_screen); gboolean is_browser = !strcmp(screen_name,"browser"); free(screen_name); return is_browser; }




ここでは、必要に応じてWin32で適切に動作するためにifdefが必要です。

この関数では、GDKウィンドウの名前を確認し、それがブラウザである場合は、同時にウィンドウのデザイン（フレーム）を削除します。 次に、アプリケーションがブロードウェイを介して実行されているかどうかに応じて、インターフェイスをロードします

幅広いカスタマイズ













幸いなことに、GTK3はオープンソースであるため、さらに深く進んで何かを変更できます。



ウィンドウのタイトルから始めましょう。


デフォルトのテキストである「broadway 2.0」は、だれにも適しているとは考えにくいため、これを変更します。 HTML5サーバーページは、テンプレートとJSファイルで構成されています。 ページタイトルは、標準のHTMLヘッダーに書き込まれます。 

 <title>broadway 2.0</title>


アセンブリ中に、ページはperlスクリプトによって単純なC配列に変換され、gtk + -3.10.7 / gdk / broadway / clienthtml.h-static const char client_html []に保存されます。

その後、すべてが簡単です、gtk + -3.10.7 / gdk / broadway / broadway-server.c： 



 static void got_request (HttpRequest *request) ............... if (strcmp (escaped, "/client.html") == 0 || strcmp (escaped, "/") == 0) send_data (request, "text/html", client_html, G_N_ELEMENTS(client_html) - 1);


操作アルゴリズムを少し変更します。ヘッダーをsprintfの形式として使用します。 そのため、gtk + -3.10.7 / gdk / broadway / clienthtml.hを少し修正する必要があります。たとえば、次のようにパーセント記号のスクリーニングを追加します。 

 background-image: -moz-linear-gradient(#D1D2D2 0%%, #BABBBC 65%%, #D4D4D5 100%%);\n"


の代わりに 

 background-image: -moz-linear-gradient(#D1D2D2 0%, #BABBBC 65%, #D4D4D5 100%);\n"


そしてタイトルのテキストを置き換えます 

 <title>broadway 2.0</title>


指定子について 

 <title>%s</title>


サーバー自体のアルゴリズムを変更します。 



 char *http_title = getenv("GTK_HTTP_TITLE"); if (NULL == http_title) { http_title = "Default"; } size_t total_html_size = sizeof client_html + strlen(http_title) + 1; char *_client_html = malloc (total_html_size); snprintf(_client_html, total_html_size, client_html, http_title); if (strcmp (escaped, "/client.html") == 0 || strcmp (escaped, "/") == 0) send_data (request, "text/html", _client_html, strlen(_client_html) - 1); ........... g_free (_client_html);


asprintfを使用することもできますが、Makefileをアップグレードする必要がありますが、これは望みません。

broadwaydを再構築し、グローバル変数を設定してサーバーを起動します 

 export GTK_HTTP_TITLE="PAGE TITLE"












デフォルトの画面サイズは変更できますが、モバイルデバイスで使用する場合に関係があります。 



 static void gdk_broadway_screen_init (GdkBroadwayScreen *screen) { screen->width = 1024; screen->height = 768; }




GDK画面の名前を変更します。たとえば、broadwaydの特定のバージョンをいくつか作成し、この方法でそれらを識別することができます。 



 static gchar * gdk_broadway_screen_make_display_name (GdkScreen *screen) { return g_strdup ("browser"); } static gchar * gdk_broadway_screen_get_monitor_plug_name (GdkScreen *screen, gint monitor_num) { return g_strdup ("browser"); }
