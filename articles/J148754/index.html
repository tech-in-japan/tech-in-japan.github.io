Androidストリーミングビデオ

この記事では、 Androidアプリケーションでストリーミングビデオを操作するときに発生する可能性のある落とし穴について説明します。 具体的には、ビデオ変換とビデオ配信/再生プロトコルに焦点を当てます。

この分野の専門家ではないことをすぐに予約しますが、最近の経験のみを共有したいと思います。







ユーザーがサーバーにアップロードした多くのファイルを再生できるAndroidアプリケーションを実装するタスクに直面していると想像してください。 YouTubeをブラックジャックとコーデックで作成します。 これを行うには、少なくとも2つの問題を解決する必要があります。ビデオをAndroidでサポートされている形式に変換し、リモートソースからビデオを再生します。 これらのタスクの両方をさらに詳しく検討しましょう。





ビデオ変換



したがって、 Androidデバイスで動画を再生する前に、この動画をサポートされている形式にトランスコードする必要があります。 Androidのドキュメントには、これらの同じ形式のリストが明確に示されています。



ユーザーがサービスにアップロードしたファイルをトランスコードしたり、TVチューナーからストリームを記録するには、業界の事実上の標準である特別なユーティリティffmpegの助けが必要です。 それをインストールするための詳細な手順は、同じ名前のプロジェクトのウェブサイトで見つけることができます。





現在（私の意見では）ビデオを保存する最も一般的な方法は、 H.264 AVCコーデックを使用するMP4コンテナーです。 実際、私たちはそれらを検討します。





まず、 AndroidはH.264コーデックのすべての機能をサポートしているのではなく、特定のセット（ ベースラインプロファイル （BP）と呼ばれるプロファイル）のみをサポートしていることに注意してください。 したがって、たとえば、 BPにはCABACやB-Framesなどの便利なH.264機能は含まれていません。





私たちにとって、これは、ビデオをエンコードするときにこれらの機能を使用する場合、このビデオを再生するためにAndroidは必要ないことを意味します。 お使いの携帯電話が十分に強力であり、ベンダーが追加のコーデックのインストールとサポートを担当している場合は可能ですが、 たとえば、 メインプロファイルのビデオは、 Samsung Galaxy SIIで問題なく再生できます。 通常のクラスの電話（ Samsung Galaxy Aceなど ）では、ビデオを再生できないことに関するメッセージと、 logcatの誤ったコーデックのコーデックのエラーが表示されます 。





しかし、理論から実践に移りましょう。 ビデオを圧縮するには、次のコマンドを実行する必要があります。



ffmpeg -i in.3gp -f mp4

-vcodec libx264 -vprofileベースライン-b：v 1500K

-acodec libfaac -b：128k -ar 44100 -ac 2

-y out.mp4





各パラメーターをさらに詳しく検討してみましょう。

  • -i src入力（トランスコード）ファイル。
  • -f mp4はビデオコンテナを使用しました。
  • -vcodec libx264はビデオコーデックを使用しました。
  • -vprofile使用されるベースラインプロファイル。
  • -b：v 1500Kビットレート;
  • -acodec libfaacはオーディオコーデックを使用しました。
  • -b：128kオーディオビットレート。
  • -ar 44100の音の周波数;
  • -ac 2つのオーディオストリーム数。
  • 出力ファイルを上書きする-yフラグ。




また、プロファイルを指定せずに、 -x264optsパラメーターを使用して必要なH.264コーデックオプションを明示的に有効/無効にして、 BP条件を満たすようにすることもできます。 しかし、これは恋人のための教訓です。







ビデオ配信



リモートサーバーからビデオを再生する最も簡単な方法は、一時ストレージにダウンロードしてローカルで再生することです。 しかし、現代のビデオ録画のサイズを考慮すると、これは選択肢ではないことを誰もが理解していると思います。





どうする？ Androidプラットフォームは、次のテクノロジー/プロトコルのネイティブサポートを提供します。

  • HTTP / HTTPSプログレッシブストリーミング;
  • HTTP / HTTPSライブストリーミング;
  • RTSP（RTP、SDP）;




それらを順番に考えてみましょう。





プログレッシブストリーミング



従来のWebサーバーを使用してビデオを配信する最も簡単な方法は、基本的には、 HTTP（S）プロトコルを使用して事前に準備されたファイルをダウンロードすることです。 この場合のすべてのソルトは、ダウンロードの終了時にファイルの再生が開始されないことです。ただし、十分なデータがダウンロードされるとすぐに（一部のバッファーがいっぱいになります）。





ここで、 MP4コンテナを使用する場合、ファイルを作成して、ビデオストリームに関するメタデータ（ moov atoms ）がファイルの先頭（ ftyp atomの後）、ビデオデータ（ mdat atom）の前に配置されるようにする必要があります。 これは、 qt-faststartユーティリティでファイルを処理することで実行できます。



qt-faststart output.mp4 result.mp4





プログレッシブストリーミングの主な問題は、ビデオを思いがけない瞬間に巻き戻せないこと、デバイスに十分な空き容量があること、Webサーバーにビデオをダウンロードする多数の「脂肪」クライアントをサポートする必要があることです。





このテクノロジーを使用した再生は、Androidプラットフォームでネイティブにサポートされています。 標準のMediaPlayerクラスを使用して、リモートファイルを簡単に再生できます（ただし、通信チャネル、デバイスのパワー、および空き領域の可用性は例外です）。





擬似ストリーミング



このテクノロジーは、 プログレッシブストリーミングの論理的な拡張機能であり、その主な問題の1つである、まだダウンロードされていないフラグメントへの巻き戻しを解決できます。 H.264 / AACコーデックを備えたMP4 / FLVコンテナに適用可能。





この場合のプログレッシブストリーミングとの唯一の違いは、GET要求のタイムスタンプを考慮して、必要なビデオファイルのフラグメントを返す特別なWebサーバーが必要であるという事実です。 このようなWebサーバーの例としては、当然ngx_http_mp4_moduleを持つOrthodox NGINXがあります 。





Androidでのこの標準のサポートに関する公式情報は見つかりませんでした。 ただし、経験的には、少なくともHTC DesireおよびSamsung Galaxy SIIデバイスに存在することがわかりました。 ただし、お使いのデバイスにネイティブサポートがなくても、 MX Playerなどのサードパーティプレーヤーをいつでも使用できるという事実に注意を喚起したいと思います。MXPlayerは、希望するタイムスタンプでビデオフラグメントをダウンロードおよび再生するロジックを独立して実装し、巻き戻しを整理できるようにします。





ライブストリーミング



Appleの非常に非標準のデータ転送プロトコル。 その本質は、 M3U8形式の特別なファイルプレイリストによって、分散ファイルが多くの小さな部分に「生成」されるという事実に要約されます。 データ転送はHTTP（S）を介して行われます 。





この場合、デバイスの巻き戻しと空き領域に問題はありません。 さらに、特定の条件下では、再生中のビデオの品質を選択する機会があります。





ただし、問題も発生します。 ファイルを「カット」してプレイリストを作成するには、サーバー上のプロセッサリソース、時間、およびスペースが必要です。 前の例のように、ファイルをネットワークにブロードキャストするには、HTTPサーバーが必要です（追加モジュールなし）。



VLCを使用して、ビデオファイルを「 見通す 」ことができます。



vlc -I dummy /path/to/pornofilm.mpg vlc：// quit --sout '#transcode {width = 320、height = 240、fps = 25、vcodec = h264、vb = 256、venc = x264 {aud、プロファイル=ベースライン、レベル= 30、keyint = 30、ref = 1}、acodec = mp3、ab = 96}：std {access = livehttp {seglen = 10、delsegs = false、numsegs = 0、index = / path / to /web/server/root/pornofilm.m3u8,index-url=http://localhost/pornofilm/stream-########.tsasket,mux=ts{use-key-frames},dst= /path/to/web/server/root/pornofilm/stream-#########.ts} '





このようなファイルは、 localhost / pornofilm.m3u8 URLで再生できます 。





バージョン3.0以降、AndroidのネイティブレベルでのHTTPライブストリーミングのサポートが提供されています 。 サードパーティのプレーヤー（ DicePlayer 、 MX Player ）を使用して、wikiから判断すると、バージョン2.2からサポートを受けることができます。





リアルタイムストリーミングプロトコル（RTSP）



ビデオ伝送専用に設計されたステートフルアプリケーション層プロトコル 。 コマンド形式はHTTPに非常に似ています 。 コマンド自体は、従来のカセットレコーダーのボタンに似ています：PLAY、PAUSE、RECORDなど。





HTTPライブストリーミングとは異なり、 RTSPではファイルを小さな部分に分割してプレイリストをコンパイルする必要はありません。 ファイルの必要な部分が生成され、その場でクライアントに渡されます。 VLCをRTSPサーバーとして使用できます。





RTSPプロトコル自体はデータ転送の方法を決定するのではなく、これを他のプロトコルに委任することに注意してください。 たとえば、 RTPです。 RTPを介してファイルをブロードキャストするには、次のパラメーターを使用してVLCを開始する必要があります。



vlc -vvv /path/to/pornofilm.mp4 --sout '#rtp {dst = localhost、port = 1234、sdp = rtsp：// localhost：8080 / pornofilm.sdp}'





ただし、表示するユーザーの可用性に関係なく、個別のポートを使用して各ファイルのプロセスを上げるのは愚かなことです。





RTSPプロトコルに戻り、ビデオをオンデマンドで再生します（Vidoe On Demand）。 VLCをRTSPサーバーとして使用してVODを再生するには、最初にRTSPサーバーとTelnetインターフェイスの属性を指定してVLCを開始する必要があります。



vlc -vvv -I telnet --telnet-password 123 --rtsp-host 127.0.0.1 --rtsp-port 5554





その後、サーバーの実行中に、サーバーを構成する必要があります。 これは、 telnet 'aを使用して最も便利に実行されます。このアプローチにより、オンザフライで構成できるためです。



新しいポルノVODが有効になりました



ポルノ入力のセットアップ/path/to/pornofilm.mpg





ビデオ（Androidプラットフォームを含む）を再生するには、URL rtsp：// localhost：5554 / pornofilmで要求する必要があります 。





欠点の1つは、 HTTPがすべてのファイアウォールおよびプロキシでしばしば開かれるという事実です... RTSPでは 、拒否、許可ポリシーの場合、すべてが異なります。



さらに、 RTSPサーバーを使用してサーバー上のファイルを追加/削除する場合、その構成（vodのリスト）を更新する必要があります。 はい、これにはtelnetがありますが、Webサーバーのディレクトリからファイルを単にアップロードまたは削除するよりもさらに困難です。





このテクノロジーを使用した再生は、Androidプラットフォームでネイティブにサポートされています。 たとえば、同じ標準MediaPlayerクラスを使用します 。





マルチキャスト



多くの人は、 マルチキャストはAndroidでは機能しないと考えています。 これは完全に真実ではありません。





まず、ほとんどの場合、不必要な作業でデバイスリソースをロードしないように、デフォルトで無効になっています。 単にオンにすることができます。





第二に、はい-かなり印象的な数のデバイスでは、すべてが無効になっているか、正しく動作しません。 インターネット上で、あなたは多くの涙といくつかの解決策さえ見つけることができます 。





ただし、実践が示すように、 Androidで マルチキャストビデオを再生できます。 私の場合、最近リリースされたAndroid向け VLCベータ版はこのタスクにうまく対処しました。





さらに、 VLCサーバーを使用すると、 マルチキャストの再生を常にHLSに減らすことができます 。



vlc -I dummy udp：//@192.168.20.1：1234 vlc：// quit --sout '#transcode {width = 320、height = 240、fps = 25、vcodec = h264、vb = 256、venc = x264 { aud、プロファイル=ベースライン、レベル= 30、keyint = 30、ref = 1}、acodec = mp3、ab = 96}：std {access = livehttp {seglen = 10、delsegs = false、numsegs = 0、index = / path /to/web/server/root/multicast-porno.m3u8,index-url=http://localhost/multicast-porno/stream-#########.tsasket,mux=ts{use-key -frames}、dst = / path / to / web / server / root / multicast-porno / stream-########。ts} '



またはRTSP：

新しいマルチキャストポルノVODが有効



マルチキャストポルノ入力udpのセットアップ：//@192.168.20.1：1234





プレイヤーにudp：//@192.168.20.1：1234の形式のURLを渡すことで、デバイスでマルチキャストを再生することができます。





何を選ぶか



ビデオ形式（ H.264 BP / MP4 ）ですべてが明らかな場合、配信方法では問題が解決しません。 それぞれに長所と短所があります。





まず、通常のプログレッシブストリーミングを削除します 。 はい、それはいつでもどこでも動作しますが、ファイル全体を巻き戻したりダウンロードしたりすることの欠如はすでに多すぎます。





次に飛び立つ候補はライブストリーミングです。 その主な欠点は、バージョン3.0以降のAndroidのネイティブサポートです。 また、バージョン2.x のユーザーの80％以上を無視することは選択肢ではありません。 ここでは、サードパーティのプレーヤーを見ることができますが、独自の実装を行うこともできます（残念ながら、 HLSをサポートするための無料の開発は見つかりませんでした）。





そして、私が最後に消すのはRTSPです。 はい、ビデオ専用に設計されたプロトコルです。 はい、その使用はイデオロギー的に真実です。 しかし、2つのポイントがあります。 まず、サーバー構成を常に更新する必要があります。 第二に、 HTTPはいつでもどこでもオープンであり、 RTSP / RTPについては言えません。





個人的には、 擬似ストリーミングに焦点を当てます 。 ファイル全体をダウンロードせずに巻き戻すことができます。 必要なのは、Webサーバーを少し構成することだけです。
