同じ名前の複数のサービスインスタンスを使用する方法



負荷分散と耐障害性に適しています。

実際、DEALER-to-ROUTERモードで同じ接続名を使用すると、最後の接続のみが使用されます。

ソリューションは、ブローカーに接続し、DEALER-to-DEALERモードで個別の接続ポイントを作成するサービスごとに個別のロードバランサーです。 この場合、サービスの場合、接続アドレスをブローカーではなくバランサーに変更するだけで十分です。



パラメータ 
使用法： 

    waha-proxy-balance [-b <zmq bind>] [-v] -n <name> [-u <zmq bind>] [-]
                        [-バージョン] [-h]


どこで： 

    -b <zmq bind>、-backend <zmq bind>
     プロキシのバックエンドバインディング

    -v、-verbose
     より多くの出力を表示する

    -n <名前>、-名前<名前>
      （必須）バランシングサービス名

    -u <zmq bind>、-url <zmq bind>
     ブローカーURL

    -、-ignore_rest
     このフラグに続くラベル付き引数の残りを無視します。

    -バージョン
     バージョン情報を表示して終了します。

    -h、-help
     使用情報を表示して終了します。

HTTP経由でサービスへのアクセスを提供する方法は？





少なくともスマートトースターを介して作業する能力を考えれば、HTTPインターフェースに対する人々の大いなる愛を考えれば、それは無意味ではありません。正しく実行する必要があります。json/プレーン出力、GETおよびPOSTリクエストの両方のサポート、すべての消防士のJSONPです。

Pocoライブラリを使用すると、別のデーモンとして実装されることがわかりました。



  • GET要求またはPOST本体のargsパラメーターには、JSON配列が含まれている必要があります。
  • オプションのタイムアウトパラメータは、ミリ秒単位の最大応答時間です。
  • オプションのパラメータータイプ-戻りデータタイプ-プレーンまたはjson
  • オプションのjsonpまたはcallabackパラメーター-JSON-Pリクエストの関数名

    <string>（複数回受け入れられます）

使用例





読み込みの負荷が大きく、書き込みが少ないユーザーの承認。





ブローカーの起動（デフォルトポートは10000） 

 $ワハブローカー




読み取りのためのバランサーの開始（デフォルトでは、ポート10001） 

 $ waha-proxy-balance -n system.user.check




httpasswdを起動して、ログイン/パスワードを確認します（負荷を分散するために必要な回数だけ実行します）。 シャットダウンコードのみを返します。 後に来るものはすべて、引数としてスクリプトに渡されます。 

 $ waha-service-script -u tcp：//127.0.0.1：10001 -n system.user.check -s htpasswd --ret-code --no-stdout--bv users.passwd




出力とともにユーザーを追加するためのスクリプトを実行する（バランスをとることなく） 

 $ waha-service-script -n system.user.add -s htpasswd --ret-code --stderr--b users.passwd




HTTPインターフェースの開始（デフォルトのポート9001） 



 $ waha-proxy-http




リクエスト例 



 http://127.0.0.1:9001/system.user.check?args=["admin","admin "]
 http://127.0.0.1:9001/system.user.check?args=["admin","admin"†&callback=func123




コマンドインターフェイスの使用





フォーマットなしの出力： 
  $ waha-cli -n system.user.check admin admin


JSONでの出力： 
  $ waha-cli -o json -n system.user.check admin admin
