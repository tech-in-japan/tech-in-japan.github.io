前回の記事では、テーブルの作成、データのロード、単純なSELECTクエリの実行など、基本的なハイブ機能について検討しました。 次に、Hiveを最大限に活用する高度な機能について説明します。







ユーザー定義関数



Hiveを使用する際の主な障害の1つは、標準SQLフレームワークの制約です。 この問題は、言語拡張機能、いわゆる「ユーザー定義関数」を使用することで解決できます。 かなり便利な機能がHive言語に組み込まれています。 私の意見で最も興味深いもののいくつかを以下に示します（ 公式文書から得た情報）。



ジョンソン



大きなデータを扱うときのかなり一般的なタスクは、json形式で保存された非構造化データの処理です。 jsonハイブを使用するには、jsonドキュメントから値を抽出できる特別なget_json_objectメソッドをサポートします。 JSONPath表記の限定バージョンは、オブジェクトから値を抽出するために使用されます。 次の操作がサポートされています。



  • $：ルートオブジェクトを返します
  • 。：子オブジェクトを返します
  • []：配列のインデックスルックアップ
  • *：のワイルドカード


公式ドキュメントからJsonを操作する例：



1つの列（json）と1つの行で構成されるテーブルsrc_jsonがあるとします。 



{"store": {"fruit":\[{"weight":8,"type":"apple"},{"weight":9,"type":"pear"}], "bicycle":{"price":19.95,"color":"red"} }, "email":"amy@only_for_json_udf_test.net", "owner":"amy" }


テーブルクエリの例： 



 hive> SELECT get_json_object(src_json.json, '$.owner') FROM src_json; amy hive> SELECT get_json_object(src_json.json, '$.store.fruit\[0]') FROM src_json; {"weight":8,"type":"apple"} hive> SELECT get_json_object(src_json.json, '$.non_exist_key') FROM src_json; NULL


Xpath



同様に、ハイブを使用して処理する必要があるデータがjsonではなくXMLに格納されている場合、 xpath関数を使用して処理できます。これにより、 適切な言語を使用してXMLを解析できます。 xpathを使用してxmlデータを解析する例： 



 hive> select xpath('<a><b>b1</b><b>b2</b></a>','a/*/text()') from sample_table limit 1 ; ["b1","b2"]


その他の便利な組み込み関数：



組み込みライブラリには、かなり豊富な組み込み関数のセットが含まれています。 いくつかのグループを区別できます。



  • 数学関数（sin、cos、log、...）
  • 時刻を操作するための関数（from_unix_timestamp、to_date、現在の日付、時間（文字列日付）、timediffなど）-日付と時刻を変換するための非常に豊富な関数の選択
  • 文字列を操作するための関数。 lengh、reverse、regexpなどの一般的に適用可能な関数と、すでに考慮されているparse_urlやget_json_objectなどの特定の関数の両方がサポートされています）
  • 多くの異なるシステム関数-current_user、current_database、...
  • 暗号化関数-sha、md5、aes_encrypt、aes_decrypt ...

独自のUDFの作成



問題を解決するために、ハイブに組み込まれた機能が常に十分とは限りません。 組み込み関数がない場合は、独自のUDFを作成できます。 これは、Java言語で行われます。



文字列を小文字に変換する簡単な関数の例を使用して、独自のUDFの作成を調べてみましょう。



1. com / example / hive / udfパッケージを作成し、その中にLower.javaクラスを作成します。 



 mkdir -p com/example/hive/udf edit com/example/hive/udf/Lower.java


2. Lowerクラス自体を実装します。 



 package com.example.hive.udf; import org.apache.hadoop.hive.ql.exec.UDF; import org.apache.hadoop.io.Text; public final class Lower extends UDF { public Text evaluate(final Text s) { if (s == null) { return null; } return new Text(s.toString().toLowerCase()); } }


3.必要なライブラリをCLASSPATHに追加します（hadoopディストリビューションでは、jarファイルへのリンクが若干異なる場合があります）。 



 export CLASSPATH=/opt/cloudera/parcels/CDH/lib/hive/lib/hive-exec.jar:/opt/cloudera/parcels/CDH/lib/hadoop/hadoop-common.jar


4. UDF-kuをコンパイルし、jarアーカイブを収集します。 



 javac com/example/hive/udf/Lower.java jar cvf my_udf.jar *


5.関数をハイブで使用するには、明示的に宣言する必要があります。 



 hive> ADD JAR my_udf.jar; hive> create temporary function my_lower as 'com.example.hive.udf.Lower'; hive> select my_lower('HELLO') from sample_table limit 1; hello


スクリプトを使用してテーブルを変換する



標準のHIVE機能を拡張するもう1つの方法は、TRANSFORMメソッドを使用することです。これにより、任意のプログラミング言語のカスタムスクリプトを使用してデータを変換できます（これは、Javaが気に入らず、udfsを書きたくない場合に特に適しています）。



コマンドを使用するための構文は次のとおりです。 



 SELECT TRANSFORM(<columns>) USING <script> as <new_columns>


<script>-この場合、stdinでデータを受信し、それらを変換し、変換されたデータをstdoutに提供するプログラムです。 実際、これはビッグデータの記事AからZで説明した map-reduceタスクを起動するためのストリーミングインターフェイスに非常に似ています。 パート2：Hadoop



例：



異なる通貨で給与を受け取るユーザーの給与を含むテーブルがあるとします： 



 +-------------------+---------------------+-----------------------+ | user_salary.name  | user_salary.salary  | user_salary.currency  | +-------------------+---------------------+-----------------------+ | alexander         | 100000              | RUB                   | | evgeniy           | 4000                | EUR                   | | alla              | 50000               | RUB                   | | elena             | 1500                | EUR                   | +-------------------+---------------------+-----------------------+


私たちは、すべてのユーザーにルーブル給与が支払われるようなプレートを手に入れたいと思っています。 これを行うには、データ変換を実行するPythonスクリプトを作成します。 



 import sys EXCHANGE_RATE = 75 for line in sys.stdin: name, salary, currency = line.rstrip("\n").split('\t') if currency == 'EUR': print name + "\t" + str(int(salary) * EXCHANGE_RATE) else: print name + "\t" + salary


スクリプトは、入力データがtsv形式であることを暗示しています（列はタブで区切られています）。 テーブルでNULL値が検出されると、値「\ N」がスクリプト入力になります



次に、このスクリプトを使用してテーブルを変換します。 



 0: jdbc:hive2://localhost:10000/default> select transform(name, salary, currency) using 'python transform_to_rub.py' as (name, rub_salary) from user_salary; +------------+-------------+ | name | rub_salary | +------------+-------------+ | alexander | 100000 | | evgeniy | 300000 | | alla | 50000 | | elena | 112500 | +------------+-------------+


実際、TRANSFORM操作を使用すると、従来のMapReduceをハイブに完全に置き換えることができます。



Mapjoin





MapReduceを操作するためのテクニックと戦略についての記事で書いたように、2つのテーブルJOINを実装するには、一般に、いくつかのMapReduceタスクが必要です。 ハイブはMapReduceでのみ機能するため、JOINはハイブにとっても高価な操作です。

ただし、結合する必要がある2つのテーブルのいずれかが各ノードのRAMに完全に収まる場合、プレートをメモリにロードすることで1つのMapReduceで対応できます。 このパターンはMapJoinと呼ばれます。 HiveでMapJoinを具体的に使用するには、ヒント（Hiveの用語では「ヒント」）を指定する必要があります。



例： 



 SELECT /*+ MAPJOIN(time_dim) */ COUNT(*) from store_sales JOIN time_dim on (ss_sold_time_sk = t_time_sk)


この例では、store_salesテーブルが大きく、time_dimテーブルが小さく、メモリに収まると想定しています。 / * + MAPJOIN（time_dim）* /-これは、MAPJOINタスクの開始に関するHIVEのヒントです。



トランザクションモデル



トランザクションACIDモデルは、4つの主要なプロパティのサポートを意味します。



  • アトミック性 -操作が完全に実行されてデータが完全に変更されるか、または落ちて痕跡が残りません。



  • 一貫性 -アプリケーションが操作を実行した後、その結果は後続のすべての操作で使用可能になります。



  • 分離 -一部のユーザーの操作には、他のユーザーに副作用はありません。



  • 耐久性 -成功したオプションの結果として行われた変更は、システム障害が発生した場合でも結果を保持します。



一般的に、Hiveはデータの変更を扱うにはあまり適していませんが、データの変更のサポートが必要な場合がいくつかあります。 まず第一に、これらは次のとおりです。



  • ストリーミングモードで追加されたデータ（ flume 、 kafkaなどのシステムから）。 データが到着したらすぐにハイブで分析できるようにしたい



  • スキーマの更新-たとえば、ハイブテーブルに新しい列を追加します。 列を各レコードに正常に追加するか、いずれかの列に追加して、いずれにも追加しないでください。



  • 個々のレコードを更新する必要がある場合があります。



これらの目的のために、バージョン0.14から、ハイブはトランザクションモデルのサポートを実装し、INSERT、UPDATE、DELETEの3つの操作で実装されました。



これらの操作のサポートは非​​常に制限されています。



  • 現在サポートされているのはORCファイルのみです。



  • デフォルトでは、トランザクションのサポートは無効になっています。 有効にするには、ハイブ構成ファイルに適切な変更を加える必要があります。



  • リレーショナルデータベースでおなじみのBEGIN、COMMIT、ROLLBACKコマンドはサポートされていません。


トランザクションサポートは、デルタファイルを使用して実装されます。 つまり、データ更新操作が実行されると、ソースファイル内のデータは更新されませんが、変更された行を示す新しいファイルが作成されます。 後で、ハイブは圧縮操作を使用してそれらをマージします（hbaseで同様の操作が使用されます）。



一般に、トランザクションサポートは非​​常に限られているため、Hiveでこの機能を使用する前に非常に真剣に検討する価値があります。 HBaseまたは従来のリレーショナルデータベースの方向を調べる価値があるかもしれません。



おわりに



このシリーズおよび前回の記事では、MapReduceタスクの操作を容易にする強力なツールであるHiveの主な機能について検討しました。 Hiveは、SQLの操作に慣れているアナリストに最適であり、JDBCドライバーのサポートを使用して既存のインフラストラクチャに簡単に統合でき、ユーザー定義関数とカスタム変換のサポートを考慮して、従来のMapReduceから自分にデータ処理を完全に転送できます。 ただし、ハイブは「銀の丸薬」ではありません。頻繁に更新されるデータについては、Hbaseや従来のリレーショナルデータベースなどのツールの方向を見ることができます。
