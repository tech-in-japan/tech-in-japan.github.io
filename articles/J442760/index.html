CommVaultによるPostgreSQLのバックアップ方法に関する記事

この記事では、PostgreSQLバックアップ用のCommVaultの使用経験を確認します。 これを行うために、クライアントからPostgreSQLデータベースのバックアップをセットアップする過去のプロジェクトの1つの小さな部分を分析します。







CommVaultについて



CommVaultは、統合されたデータ保護とコンテンツ管理を提供する単一のスケーラブルなプラットフォームです。 プラットフォームは、データのバックアップと復元、アーカイブ、重複排除、複製、メディアの階層的配布、暗号化の機能を備えたソフトウェアモジュールをサポートします。 プラットフォームモジュールは、さまざまなソースからの企業コンテンツと連携し、企業環境でのエンドツーエンドの情報検索を提供し、仮想リポジトリ内のドキュメントの単一の知的インデックス作成により、アーカイブからも一定の可用性を提供します。 このプラットフォームには、ユーザーとアプリケーションのアクションおよびインフラストラクチャの機能に関するレポートを生成する高度な分析ツールも装備されています。



CommVaultは、物理環境および仮想環境でデータとデータへのアクセスを保護、復元、および管理します。



PostgreSQLバックアップについて



PostgreSQLデータベースのバックアップを実行するには、データベースが実行されているサーバーにインストールされているエージェント（iDataAgent）が使用されます。 このエージェントは、PostgreSQLデータベース内の重要なビジネスデータを効果的に管理および保護するように設計されています。 このエージェントを使用して、PostgreSQLサーバー全体または個々のデータベースをバックアップおよび復元できます。 必要に応じて、個々のテーブルを復元することもできます。



主な機能：



PostgreSQL iDataAgentは、さまざまなモードでデータベースをバックアップし、最小限の時間でそれらを復元する柔軟性を提供します。 PostgreSQLサーバー全体、個々のデータベース、またはアーカイブログをいつでもバックアップまたはバックアップできます。



さまざまなモードで実行できるバックアップおよび回復機能：



  • iDataAgentは、PostgreSQLサーバー全体を復元する機能を提供します。 移行元サーバーにあるすべてのデータベースは、移行先サーバーに復元できます。
  • 単一のデータベースまたはデータベースのグループをサブクライアントデータとして定義し、それをバックアップおよび復元します。
  • PostgreSQLサーバー上のログのみをバックアップします。 これらのログファイルは、オペレーティングシステムまたはディスクの障害が原因で失われたデータベーストランザクションを回復するために使用できます。
  • ファイルシステムベースのバックアップのために、特定の時点でPostgreSQLサーバー全体を復元します。
  • CommCellコンソールのジョブコントローラーとイベントビューアーから、バックアップおよび回復操作の状態を表示および確認します。 保存して配布できるレポートを使用して、作業のステータスを追跡します。
  • バックアップはエクステント（またはデータベースの変更部分）のみであり、PostgreSQLデータベース全体ではないため、データをより高速にバックアップする方法としてブロックレベルのバックアップを使用します。
  • ブロックレベルの重複排除 重複排除は、データ保護操作で重複を識別して削除することにより、データをよりスマートに保存する方法を提供します。


建築



スキーム







仕組み：



CommVaultプラットフォームは、CommServe管理サーバーと別個のMediaAgentサーバーの一部としてネットワーク上に展開されます（物理サーバーを使用することをお勧めします）。



エージェント（iDataAgent）がPostgreSQLデータベースと共にサーバーにインストールされ、そのバックアップポリシーが要件に従って構成されます。 iDataAgentは必要なデータを収集し、必要に応じて圧縮、重複排除を行い、暗号化してMediaAgentに転送します。



次に、データはストレージシステム、テープライブラリ、またはクラウドストレージに配置されます。

回復のために、データはストレージから抽出され、PostgreSQLを使用してサーバーにコピーされます。



CommVaultコンソールでのセットアップ



次に、管理コンソールでこれを行う方法を検討します。



1.現時点でデータベースのバックアップを開始するには、CommCell Browserコンソールで次を選択します。



クライアントコンピューター| | PostgreSQL | | DumpBasedBackupSet。



サブクライアントのデフォルトフォルダを右クリックし、 バックアップを選択します。







2.バックアップタイプとして[ 完全]を選択し、[ 即時]を選択します。



3. OKをクリックします。 PostgreSQLバックアップが開始されます。







4.ジョブの実行中、そのステータスはCommCellコンソールの[ ジョブ]ウィンドウで監視できます。







5.タスクが完了すると、[ バックアップ履歴]ウィンドウから完了したタスクの詳細を確認できます。 サブクライアントのデフォルトフォルダを選択し、 バックアップ履歴を選択します。







6. [ バックアップ履歴]ウィンドウで、完了したタスクに関する次のデータを確認できます。



-タスク完了時のバックアップエラー。

-正常にバックアップされたアイテム。

-割り当ての詳細。

-イベント。

-ログファイル。

-データが保存されるメディア。

回復



バックアップは良いです。 当然、私たちはその創造のプロセスだけでなく、復元にも興味を持っています。 それがすべて行われたために。



この状況では、私たちの経験に基づいて、クライアントが2つのケースで修復が必要になる場合があります。



  1. たとえば、データベースから削除できるデータにアクセスするために、特定の時点でデータベースを復元する。
  2. PostgreSQLデータベースクラスター全体が失われた場合。


データベースを復元するには、次のリンクのドキュメントを読んでください： documentation.commvault.com/commvault/v11_sp14/article?p=21502.htm



また、リカバリ中の次の機能と手順にも注意を向けます。



  1. 常に、PostgreSQL DBAを使用してリカバリ手順を実行します。 これにより、誤ったアクションを回避し、リカバリプロセス中に発生する問題を迅速に解決できます。
  2. 復旧は、マスターの役割を持つノードで実行する必要があります。
  3. 復元するときは、操作の完了後にPostgreSQLサービスが開始されないことを確認してください。
  4. 復元されたノードで、設定をマスターロールに変更します。 この例では、スタンバイノードをバックアップしました。
  5. スタンバイノードでサービスを無効にし、マスターノードでサービスを有効にしてから、スタンバイノードでサービスを有効にして、レプリケーションを再構成します。
