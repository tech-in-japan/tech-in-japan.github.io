前に書いたように、開発者はこのデバイスの主な意味である「作曲家」を英語からの翻訳に入れようとしました-作曲家（まだ意味があります-作者-ライターおよびシンセサイザー）。 そのため、Synergy Composerは小さなブレードサーバーであり、シャーシにも搭載されており、一部のOSに加えて、ボード上のすべてのリソース（コンピューティング、データ転送、ストレージ）を管理する拡張HPE OneView機能モジュールを備えています。 さらに、「Microsoft SystemsCenterやVMWare vCenterなどの一般的な管理ツールや、Chef、Docker、OpenStackなどのオープンソース自動化およびDevOpsツールに統合される完全にプログラム可能なインターフェイスです」



Composerサーバーとシャーシの観点から：



  • 接続されたシャーシ（複数）およびこれらのシャーシ内のサーバーを管理します（インストールされた機器の自動検出を含む）。
  • サーバーへの後続のダウンロードのために、サーバーファームウェアオプションを保存します。
  • 詳細に説明されていない特定の機能的な「論理エンクロージャ」を使用します。 いくつかのフレームの鉄に基づいて論理コンピューティングエンティティを作成できるようにすることを提案します。
  • 後続のダウンロードのために、オペレーティングシステムのドライバーのオプションをそれ自体に保存します。
  • また、システム全体とそのコンポーネントの監視とログ記録が可能です。
  • これには、CSVまたはMS Excelにエクスポートできる標準レポートの特定のセットを持つレポートモジュールが含まれています。
  • 外部管理コマンドを処理するためのHPE Unified REST API要素が含まれています
  • SY Image Streamerと対話するためのモジュールが含まれています。 サーバーにOSを迅速に展開するために、ソフトウェアがプリインストールされたオペレーティングシステムディスクのイメージファイルが格納されるモジュール（コントロールモジュールは、シャーシの他の部分とは別の10ギガバイトネットワークで接続されていることを思い出します）。


ネットワークに関して：



  • HPE VirtualConnect、スイッチ、パススルーモジュールの構成をサポート。
  • マスターモジュールとサテライト間の接続を管理します。
  • シャーシ間のリンク集約を制御します（マルチモジュールリンク集約、MLAG）。
  • ネットワーク機器の発見と既存のインフラストラクチャへの拡張を管理します。
  • トラフィック転送を停止せずに論理接続の更新をサポートします。


情報ストレージの観点から：



  • DASまたはSANストレージの使用を制御します。 一部のストレージはComposerを介して管理されますが、最初はストレージとは独立して構成する必要があることが個別に示されています。
  • VASが「プル」されたシャーシの場合、DAS 3940ディスクモジュールに基づいて容量と階層レベルを制御します（未処理のボリューム制限は最大614 TBで、異なるシャーシ間でボリュームを分散できます）。
  • ファイバーチャネル（FC）のBoot-from-SANを管理します。
  • ストレージエリアネットワークのゾーニングポリシーを管理します
  • 一般的に、コンポーネントレベルでSANシステムとストレージシステムを監視および記録します。


サポートされている外部機器は個別に示されています。



  • ストレージ-StoreServe 3PAR（FCスイッチ）、StoreVirtual VSAおよびP4000（iSCSI）。
  • SANネットワーク-Brocade SANスイッチ、HPE 5900cp / afおよび5930 SANスイッチ、Cisco Nexus 5500/6000およびMDS SANスイッチ（FC / FCoE）。


サードパーティのソフトウェア製品との統合に関して：



  • HPE OneView for VMware vCenter 8.1はSynergyシステムをサポートしています。
  • HPE OneView for Microsoft System Center 8.1はSynergyシステムをサポートしています。
  • HPE Helion CloudSystem 10はSynergyシステムをサポートしています。


主な質問は、なぜこの庭全体をフェンスで囲む必要があったのですか？ HPEは、これが「Infrastructure-as-code」の形式で機能することを約束します。つまり、ソフトウェアを記述するレベルで、このソフトウェアを展開するために、すべての対象パラメーターとそのバリエーションを設定できます。 たとえば、ソフトウェアでコードを直接記述します。10台のWebサーバー、プロセッサーなど、メモリー、ディスク、ディスク、ファームウェア、およびOS、およびOSなどのバージョンのファームを展開しますパッチ（Image Streamerから取得）、およびそのようなドライバー。 その後、Synergy Composer APIを介して、これらすべてが正しくかつ迅速に機能するはずです。 このアイデアは興味深いものです。すでに多くの「Software-defined-something」プロジェクトが存在し、すべてが成功したわけではないため、すべて実装に依存しています。 実装が成功した場合、たとえばHelion CloudSystemを使用して穏やかに言えば、ブレークスルーになる可能性があります。ソリューションは「すべての人に」ではありません。



Composerについて他に何を追加できますか？ フォールトトレランスのために、2台目のComposerを使用することをお勧めします（1つのシャーシであっても、Image Streamerをインストールできません）。 iLOのシングルサインオン（SSO）がサポートされています。 作曲者をバックアップできます（？）。 サポートは24x7以上のレベルでのみ見られます。



RESTベースの統合APIのSDKは、Java、PowerShell、およびPythonで使用できます。



最初のモジュール、つまりSynergy Image Streamerとして知られる2番目のモジュールの追加にスムーズに進みます。 また、シャーシに取り付けられている小さなブレードサーバーでもあります。 船上にあるものはまだ知られていない。 外観はComposerと同じですが、情報ステッカーのみが異なります。



Image Streamerの3Dモデルによる外観




リポジトリと同様に、Image Streamerには、次のものが保存されます。



  • プロファイル -BIOS設定のさまざまなバージョンとファームウェアのさまざまなバージョンを含むサーバープロファイル。
  • ゴールデンイメージ -OSの「ゴールデンスナップショット」-I / Oドライバーを備えたOSの基本的な起動可能バージョン。 別の注記があります-「ゴールデンイメージがアプリケーションスタックをキャプチャする場合、アプリケーションスタックは、HPE Image Streamerで展開および/または更新することもできます。」
  • パーソナリティ -アプリケーションと構成（ホスト名、IP構成など）を含むパーソナライズされたOSスナップショット。


一般に、モジュールの機能的役割は、OSおよびソフトウェアリポジトリとiSCSI-boot-from機能の役割として現れます（ここでは個別に引用します-「従来のSANからのブート環境とは異なり、HPE Image Streamerは追加の手動セットアップまたは構成を必要としません（マルチパスサポート、一般的なSAN環境でのアダプター構成、アクセス制御、およびSANアレイ構成）」）。 図は次のとおりです。ソフトウェアファームの展​​開計画をコーディングするソフトウェアでは、Composerにプッシュします。Composerはソフトウェア自体の存在、OS、Image Streamerの設定を決定し、このデータをそこからサーバーにプッシュします。



シャーシ相互作用図のComposer、Image Streamer、およびサーバー






インストールには複数のモジュールが存在する可能性があり、21シャーシで最大のインストールが約束されています。 現時点では、10/20 Gbit / sの制御モジュール（他のシャーシモジュールとは別）間の接続は、数百にオーバークロックされることが約束されています。



次はSynergy Frame Link Module（FLM）です。 良い点として、シャーシの一部として2番目の部分で考慮することは理にかなっていますが、この場合、セクションは論理的に不均衡であることが判明したので、ここで見てみましょう。

3DモデルFLMに応じた外観




FLMタスクは、これらのモジュールのセット（FLM）に基づいて、さまざまなシャーシ間でシャーシ制御モジュールSY ComposerおよびSY Image Streamerの10GbEプライベート管理ネットワークリングのトポロジで管理ネットワークを作成することです。 1つのラック内のシャーシをCAT6ケーブルで接続できます。ラック間にはCAT7が必要です。 この図は、モジュールのフロントパネルに2つのRJ-45ポートがあることを示しています。上のポートはMGMT（制御）、下のポートはLINK（モジュール間の通信）です。 各シャーシには1つのモジュールが含まれており、2つ以上を接続するには、それぞれ2番目のFLMモジュールが必要です。



最後に、このサイクルの執筆/読書を始めたときは知らなかったが、HPE Synergyを使用したインフラストラクチャの学習に役立つ3つの興味深いドキュメントについて説明します。
