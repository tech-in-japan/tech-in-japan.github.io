iOS開発者の生活を簡素化する方法

おそらく、すべての開発者は、新しい技術を習得し始めると、すべてを自分で試して、最低レベルからすべてを実現し、その後、彼の仕事の結果を楽しみたいと思うでしょう。 しかし、経験の蓄積により、多くのタスクは非常に退屈になり、この必要な、しかし面白くないルーチンから自分を救いたいです。 この記事では、開発者の生活を簡素化し、神経を節約するのに役立つ方法とツールを共有したいと思います。



プロジェクト、開発環境のセットアップ、ライブラリ、スニペットの接続



これは最初で最も簡単ですが、まだ重要なアクションの1つですが、 this 、 this 、 thisなど、 これについて多くの記事が既に書かれており、これを知らない開発者はほとんどいないので、ここでやめません。 また、Xcode用の非常に多くの便利なプラグインが作成されています。Xcodeの概要については、 こちらをご覧ください 。



カスタムCocoaPodsリポジトリ



CocoaPodsを介してこのプロジェクトのお気に入りまたは必要なライブラリを接続することに加えて、最も一般的な「ヘルパー関数」を使用して独自のリポジトリを追加できます。 たとえば、次のようなコードを書くことにうんざりしていない人： 



[self.subviews makeObjectsPerformSelector:@selector(removeFromSuperview)];


または 



 NSData *data = [responseString dataUsingEncoding:NSUTF8StringEncoding]; return [NSJSONSerialization JSONObjectWithData:data options:NSJSONReadingMutableContainers error:nil];


これをなくすために、 GitHubのリポジトリを使用して、コードの記述を大幅に削減する便利なカテゴリを使用します。 これを使用して、これらすべてを次のように書き換えることができます。 



 [self removeAllSubviews]; return responseString.JSON;


アプリケーションのローカリゼーション



ほとんどの場合、少なくとも2つまたは3つのプロジェクトロケーションを作成する必要があります。 Xcodeにはこのための独自のツールがあり、それらを改善するためにAppleの開発者に信用を与える必要がありますが、特にアプリケーションがすでにAppStoreにあり、アップデートをリリースする必要がある場合、これは最も便利な方法ではありません-Base Internationalizationなどの十分な問題があります、 Localizable.Stringsを使用します。 既にこれに遭遇した人は、XcodeがStoryboardのローカライズされた* .stringsファイルの値を追加または更新できず、genstringsユーティリティがすべてのエントリを再生成する場合を除き、新しい追加行のみを入力することを知っています。



Localizable.Stringsから始めましょう。 NSLocalizedStringの個別のhファイル、たとえばStringConstants.hを強調表示します。このファイルでは、ローカライズされた文字列が#defineによって定義されます。 このアプローチでは、1つの石で2羽の鳥を殺すことができます。
  1. 「Cancel」、「Logout」、「Invalid username or password」など、プロジェクトに存在する可能性のある一般的なフレーズと単語を収集し、既製の行で持ち運ぶことができるようにします
  2. 長くて不快なNSLocalizedStringマクロを毎回入力せずに、プロジェクト内の既存の行を再利用します
したがって、エラーメッセージを含む標準のUIAlertViewを表示すると、次のようになります。 



 [[[UIAlertView alloc] initWithTitle:kErrorString message:kInvalidLoginOrPasswordString delegate:nil cancelButtonTitle:@"OK" otherButtonTitles:nil] show];


ローカライズされたファイルの生成に関しては、このために、2番目のhファイルであるNewStringConstants.hを使用します。このファイルでは、ローカライズする必要があるエントリを追加してから削除します。 Localizable.Stringsに直接、ビルドフェーズのスクリプトによって新しいエントリが追加されます。 



 cd ${PROJECT_DIR}/MyProject genstrings -a -o Base.lproj NewStringConstants.h genstrings -a -o ru.lproj NewStringConstants.h genstrings -a -o pt-PT.lproj NewStringConstants.h genstrings -a -o es.lproj NewStringConstants.h


これが行われた後、Localizable.Stringsファイルにローカル文字列値を書き込むだけです。



Storyboardを使用すると、タスクもスクリプトで解決されます。スクリプトについては、 この記事で詳しく説明します。 ここで、スクリプトの本質は、ibtoolユーティリティを使用してStoryboardから行を抽出し、呼び出されたときに変更が発生したかどうかを確認することであることを簡単に述べたいと思います。 後者が存在する場合、ローカライズされた文字列のエクスポートが行われ、その後ローカライズされたファイルがマージされます。 繰り返しますが、ローカライズされたファイルを修正するために残っています。 AppleGlotツールについても言及したいと思います。これは上記の記事で言及されていますが、ローカライズの反復的な更新では異なるアプローチを取ります。 特定のツールの選択は、開発者の好み次第です。



アプリケーションテスト



ここでは、限られた数のデバイスでのアプリケーションの手動テストに焦点を当てます。 単体テストでカバーされるコードの量に関係なく、手動テストを除外することはできません。 基本的に、開発者自身がアプリケーションの作成中にこれを行いますが、準備ができたら、目が「ぼやけていない」、作業のすべての詳細で平均的なユーザーとしてアプリケーションを使用する一般ユーザーにアプリケーションを転送する必要があります。 サポートされているすべてのオペレーティングシステムを搭載したデバイスセット全体があれば良いのですが、基本的にそうではありません。 デバイスのライン全体が存在する場合でも、原則として、オペレーティングシステムの種類に違いはありません。これは基本的にはもちろん必要ではありませんが、確認するのに害はありません。 iOSシミュレーターが助けになります。これは、ドロップダウンメニューから目的の項目を選択することで構成できます。 ただし、追加のアクションがなければ、テスターはソースコードを転送する必要がありますが、これは常に受け入れられるとは限りません。 次に、iOSシミュレーターでのテスト用にアセンブルされたアプリケーションを転送する方法と、このために追加でインストールする必要があるソフトウェアについて説明します。



sim_launcherとios-simをインストールします 



 gem install sim_launcher brew install ios-sim


sim_launcherを使用すると、シミュレータでHTTP GETリクエストを介してiOSアプリケーションを実行できます。 これは、ユーザーに端末コマンドを記憶させることなく、テストのためのすべての編成作業をブラウザーで編成できるため便利です。 ios-simは、ターミナルコマンドの一種のラッパーであり、シミュレーターでの作業を簡素化します。 sim_launcherはしばらく更新されていないため、ios-simの最新バージョンと互換性がないようです。 sim_launcherインストールフォルダーにあるsimulator.rbファイルの以下の機能を修正する必要があります。



simulator.rbを修正 
 def start_simulator(sdk_version, device_family) sdk_version ||= SdkDetector.new(self).latest_sdk_version run_synchronous_command( :start, '--devicetypeid', sdk_version, '--exit' ) end def launch_ios_app(app_path, sdk_version, device_family, app_args = nil) if problem = SimLauncher.check_app_path( app_path ) bangs = '!'*80 raise "\n#{bangs}\nENCOUNTERED A PROBLEM WITH THE SPECIFIED APP PATH:\n\n#{problem}\n#{bangs}" end sdk_version ||= SdkDetector.new(self).latest_sdk_version args = ["--args"] + app_args.flatten if app_args run_synchronous_command( :launch, app_path, '--devicetypeid', sdk_version, '--exit', *args ) end


シミュレーターでビルドを実行する 



 sim_launcher


このコマンドは、ポート8881のWebサービスを解除します。 <path-to-app>の代わりに.appファイルへのフルパスを指定して、 http：// localhost：8881 / launch_iphone_app？Sdk = com.apple.CoreSimulator.SimDeviceType.iPhone-6、％208.1＆app_pathのブラウザーに移動します= <path-to-app> 。 iOS 8.1でiPhone 6シミュレーターが開く場合、すべて正常に機能しています。 ブラウザウィンドウにエラーが表示される場合は、sim_launcherのソースコードに問題があるか、シミュレータでの作業が.appファイルでサポートされていない可能性があります。 次のコマンドを使用して、シミュレーターのアセンブリを作成できます。 



 xcodebuild -workspace MyProject.xcworkspace -scheme MyProject -arch i386 -sdk iphonesimulator

アプリケーションを公開



AppStoreのアプリケーションページを適切に設計するには、適切なスクリーンショットを提供する必要があります。 サポートされているデバイスごとに作成する必要があることに加えて、すべてのローカライズにも注意する必要があります。 自動化を行わない場合、このプロセスには多くの日常的な作業の痛みも伴います。 Fastlane.toolsは、テスト、継続的インテグレーション、アプリケーションの公開の両方に関連する多くのプロセスの自動化の図を提供します。 すべてのソースコードはパブリックドメインに投稿されており、それらに対する指示もあります。 ここでは、スクリーンショットの操作に関する部分についてコメントしたいだけです。 UIオートメーションを使用して、スクリーンショット作成コマンドを追加する必要がある適切な場所で、シミュレーターが自動モードで実行するスクリプトが準備されます。 電話とタブレットでは、個別のスクリプトがサポートされています。 すべてのスクリーンショットはローカライズ別にソートされ、個別のフォルダーにスタックされます。また、概要htmlページも生成されます。 別のスクリプトを使用すると、ユーザーの介入なしでも、すべてのスクリーンショットをiTunesConnectにアップロードできます。 したがって、パラメータを構成するためだけに残ります。
