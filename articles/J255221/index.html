IO Ninja-プログラマブルターミナルエミュレーター/スニファー（パート2）

この記事は、 IO Ninjaターミナル/スニファーの動機と歴史を扱った以前の入門記事の続きであり、製品の組み込み機能について少し説明しました。 「箱から出してすぐに」利用できるものの話を続けますが、より実用的なバイアスがあります。



IO Ninjaはもともとオールインワンユーティリティとして考えられており、パッケージにはさまざまなモードでさまざまな車両を操作するための多数の組み込みプラグインが含まれています。 ただし、プラグインのリストとその機能のドライリストの代わりに、私たちの会社が実際に直面した人生のタスクの小さなサンプルを実証することを決めました。また、よく知られている端末とモニターは、IO Ninjaよりもうまく対処しません（そして、まったく対処しません）。


シリアルへのTCP転送



Unixライクシステムのユーザーは、あるプログラムの出力を別のプログラムの入力にリダイレクトできることをよく知っています。 IO Ninjaのセッションリンクは、同様の機能を提供します。 違いは、Unixパイプの場合のように、単方向のデータカスケードではなく、2つのセッションの接続により、一方の出力を他方の入力に「短縮」し、逆も同様です（この場合、すべての送信データがログに記録されます）。 これにより、IO Ninjaをリダイレクトおよび/またはリッスンのユニバーサル仲介として使用することが可能になります。



デバイスがシリアルインターフェイス（シリアル）を介してコンピューターに接続されているとします。 IO Ninjaを開き、シリアルセッションを開始します。 シリアルポートを開いて設定します（おそらく、デバイスが活発であることを確認するために、いくつかのテストコマンドをデバイスに送信します）。



次に、TCPリスナーセッションを開始し、リッスンするインターフェイスとTCPポートを選択します。 設定ダイアログを介して、2つのセッションSerialとTCP Listenerを接続します。 これで、他のコンピューターからTCP経由でシリアルデバイスに接続できます！






UDPブロードキャストを使用してローカルセグメントのデバイスを検索する



ネットワーク上のデバイスを自動検出する一般的な方法は、エコー要求を送信してから応答パケットを収集することによりUDPをブロードキャストすることです。これは特に、当社が製造するすべての組み込みモジュールの自動検出で機能します。 IO Ninjaは、1対多通信のブロードキャストUDPターミナルとして使用できます。



UDPソケットを開き、255.255.255.255（またはアドレスとしてサブネットブロードキャストタイプ192.168.1.255）を登録し、エコー要求を送信します。 その結果、ローカルセグメントのすべてのデバイスのリストが表示されます。 さらに、必要に応じて、特定のデバイスとの通信を継続できます。







ちなみに、コンパスでボタンに注意してください。 UDP（特にサーバー側）で作業する場合、パケットの送信元のアドレスを覚えて、そこに正確に応答する必要があることがよくあります。接続はありません！ 原則として、ログからアドレスをコピーして「リモートアドレス」入力フィールドに貼り付けることを妨げるものはありません。 しかし、なぜあなたは手で車に何を置くことができますか？ コンパスを押すと、必要な自動化が行われます-UDPセッションはリモートアドレスを自動的に再構成します。



入力ストリームの一部をファイルに保存する



デバッグ中に、受信したデータをファイルに保存する必要がある場合があります。 たとえば、このセッションまたは他のセッションで変更せずに送信する必要があるブロックである可能性があります。リモートホストによって送信されたファイルである可能性があります。



標準端末は、このかなり単純なタスクで停止することがよくありますが、これは次の点によるものです。

  • データはバイナリで、印刷できない文字を含むことができます。
  • データは複数のパケットに分散できます。
  • 原則として、入力ストリーム全体ではなく、厳密に定義されたブロックが必要です（つまり、単にストリームをファイルにリダイレクトするだけでは、問題は完全には解決されません）。


IO Ninjaは、入ってくるパッケージを接着する「16進シート」形式のログの存在と比較して有利です。これにより、必要なことを正確に達成できます。







目的のデータブロックを選択し（必要に応じて、ステータスバーを見てください-選択したブロックのオフセットと長さが表示されます）、ファイルに保存してから使用します。 たとえば、別のセッションに転送します。







TCPリスナーを使用したTCP接続の受信



上記で、TCPリスナーがリダイレクトを整理して、TCP経由でシリアルデバイスに接続できるようにする方法を説明しました。 ただし、TCPリスナープラグインの最初の目的はこれではなく、サーバー側のTCP端末として（特に、TCP接続のクライアント側をデバッグするため）です。



新しいTCPリスナーセッションを作成し、インターフェイスとポートを選択して、サーバーを起動します。 クライアントから端末を使用して着信接続を受信した後、通常どおり作業できます。







TCPリスナーには、接続が既に使用可能な場合に新しい着信接続を受け入れるかどうかも選択できることに注意してください。 このオプションは、クライアントが同じアドレスへの多くのセカンダリ接続を確立する状況で非常に役立ちます（たとえば、多くのWebブラウザーなど）



TCPプロキシを使用したTCP接続のリッスン



IO Ninjaには、ネットワーク接続をリッスンするためのPCapベースのネットワークスニファーであるNetwork Snifferプラグインが含まれています。 ただし、場合によっては、別の組み込みプラグイン、つまりTCPプロキシを使用する方がはるかに便利です。



TCPプロキシはTCP接続とTCPリスナーの組み合わせであり、TCP接続の仲介者を編成できます。 このアプローチの利便性は、データストリームのクリーンなログをリアルタイムで即座に提供するという事実にあります（データストリームを後で復元する必要があるパケット形式のログの代わりに）。 さらに、このモデルは、スニファーでのリスニングが困難な場合にも機能します（ネットワークスイッチ、Wi-Fiカードなどの特定のモデルで）



TCPプロキシセッションを作成し、サーバー側とクライアント側を構成して、プロキシサーバーを実行します。 このサーバーへの接続はプロキシのクライアント側からセカンダリ接続を生成しますが、クライアント側で受信したすべてのデータはサーバー側に転送され、その逆も同様です。 この意味で、TCPプロキシはセッションバインディングに似ていますが、TCP向けに特別に強化されているため、より便利です。







Windows名前付きパイプのデバッグ



IO Ninjaは、WindowsのIPC（プロセス間通信）へのかなり一般的なアプローチ（名前付きパイプ）をデバッグするためのツールとして使用できます。 特に、Windowsドライバーの一部は、関連するサービスやアプリケーションと通信します。



アプリケーションのサーバー側（ConnectNamedPipeが呼び出される側）をエミュレートするには、名前付きパイプリスナーを実行します。 このプラグインを使用すると、クライアントアプリケーション/サービスからの着信名前付きパイプ接続を受け入れることができます。

接続を確立した後、クライアントと通信し、受信した要求のログを分析して、応答パッケージ（パッケージビルダーなどで作成）を送信することができます。



クライアント側（CreateFileが呼び出される側）をエミュレートするには、Generic Fileプラグインが使用されます。







名前付きパイプ（または* nixのFIFO）のデバッグに加えて、このプラグインは、非標準のデバイスドライバーの読み取りおよび書き込みにも使用できます。



名前付きパイプをリッスンする問題について簡単に触れます。 IPCにパイプを使用するアプリケーションまたはドライバーがあるとします。 クライアント側で名前付きパイプアドレスを何らかの方法で構成できる場合は、中間（TCPプロキシと同様）を使用してパイプ接続をリッスンできます。

どのくらい正確に？
デバッグ対象のシステムに、\\。\ Pipe \ debuggee_pipe_serverへのパイプ接続を確立させます。 \\。\ Pipe \ my_pipe_proxyなど、未使用のパイプ名を選択します。 この\\。\ Pipe \ my_pipe_proxyでデバッグ対象システムのクライアント側を構成します。 IO Ninjaで、Pipe ListenerとGeneric Fileのペアを開き、リンクします。 Pipe Listenerで、\\。\ Pipe \ my_pipe_proxyと入力して、Listenをクリックします。 汎用ファイルに、\\。\ Pipe \ debuggee_pipe_serverと入力して、[開く]をクリックします。 メディエーターの準備ができました。





IO Ninjaの今後のバージョンでは、中間者を編成することなく、名前付きパイプの完全なリッスンも追加します（最初の近似として、おそらくユーザーモードAPIフックを使用し、後でファイルシステムフィルターを使用します）。



テストのためにビートフレームを送信する



「壊れた」または特別に形成されたイーサネットフレームを送信する必要性はそれほど一般的ではなく、エキゾチックである可能性が高いようです。 それにもかかわらず、この機能は非常に多くの実際のシナリオで需要があります。 たとえば、私たちの実践では、これは、当社が製造した組み込みモジュールのTCP / IPスタックの実装をデバッグするときに必要でした。 別の現実的なアプリケーション領域は、ネットワークセキュリティの専門家、ファイアウォール開発者などが直面するタスクです。



ネットワークスニファーセッションを開き、インターフェイスとキャプチャフィルターを選択して、キャプチャを開始します。 パッケージビルダーでは、必要なフレームを準備します。たとえば、TCP / IPパケットテンプレートの組み込みライブラリを使用したり、キャプチャしたフレームの一部を16進エディターにコピーして必要なフィールドを修正したりできます。







[送信]をクリックすると、愛情を込めてコンパイルしたフレームが出発し、戦闘ミッションが完了します。
