モバイルサービス、ブロブ、Windows8。クラウドにデータを保存する





こんにちは



私たちは、現実世界と仮想世界の両方で、完全なグローバル化の世界に住んでいます。 つまり、異なるデバイスで同じデータ、同じ設定、同じプログラムの動作を実現したいのです。 私たちの時代のすべてのものとすべてのもののそのような同期は、クラウドの助けを借りて簡単に達成されます。 これについては、この記事で説明します。つまり、Windows 8アプリケーション（WinRT）とWindows Azureクラウドを交差させて、データとメタデータを保存します。



入門



したがって、Windows 8用のアプリケーション（テキストエントリを保存するためのサービスとする）があるとします。

1）各レコードに関する最小限の情報（たとえば、タイトルや作成日）を含むレコードのリストを表示する必要があります。 それをメタデータと呼びます。

2）特定の要素に移動するときに、レコード（コンテンツ）に関する詳細情報を表示できる必要があります。 それをデータと呼びます。

3）Windows 8を実行しているさまざまなデバイス間でデータを同期する必要があります（自宅でレコードを作成し、職場で見ました）。



私が見る最も正しいアプローチは、データとメタデータを分割し、それらを異なる場所に保存することです。 これにより、リストに表示する小さなメタデータをすばやく取得し、ユーザーの要求に応じて「重い」データを読み込むことができます。



この記事では、メタデータを保存するためのWindows Azureモバイルサービスと、データを保存するためのWindows Azure Blob Storageを紹介します。 前方には多くのテキストと写真があります。



ステップ1.クラウドストレージを使用する



Windows Azure Blob Storageを使用したより柔軟でクロスプラットフォームな作業のために、モバイルクライアントからの要求を受信および処理するWCFサービスの形式で別のレイヤーを導入することをお勧めします。 したがって、このガイドの最初のステップはいくつかのステップで構成されます。



必要なWindows Azureサービスを作成する


まず、Windows Azureに2つのサービス（クラウドサービスとストレージ）を作成して、クラウド環境を準備する必要があります。 2番目のものから始めましょう。 Windows Azure管理ポータルにアクセスします 。 ページの下部には、プラスの付いた大きなボタンがあります。 クリックすると、作成するサービスの選択が開きます。 データサービス->ストレージが必要です。





便利な新しいサービスの名前を入力し、下の[daw]ボタンをクリックして、サービスがオンラインステータスを受け取るまで待ちます。 その後、ストレージの設定に入り、目とマウスで「キーの管理」ボタンを探し、文字iアイコンがボードに表示されます。 クリックすると、目の前に3つのフィールドが表示されます。 最初のものには、作成中に割り当てたこのサービスの名前が含まれ、2番目と3番目にはアクセスキーが格納されます。 それらの最初のものを覚えておいてください、それはすぐに私たちに役立ちます。



WCFサービスの作成


Visual Studio 2012を使用してWindows AzureでWCFサービスと対応するサービスを作成します。例としてUltimateバージョンを使用しますが、Webバージョンは無料での使用に適しています。

そのため、スタジオを開き、クラウドタイプの新しいプロジェクトを作成します。





表示されるウィンドウで、クラウドに配置する必要があるプロジェクトのタイプを選択するように求められます。 WCFサービスWebロールに興味があります。 それを選択し、作成されたもののリストに追加します（名前を変更することを忘れないでください）：





Visual Studioは、ある程度の検討の後、2つのプロジェクトを作成します。1つはWCFサービスプロジェクトで、もう1つはWindows Azureクラウドで公開するための特別なプロジェクトです。 それらを少し調整しましょう。



サービスをクラウドに簡単に公開するには、Visual Studioがこれを行う必要があるクラウドを認識し、また、それを行う権利があることを理解する必要があります。 彼女を説得するには、クラウドプロジェクトを右クリックして、[公開]メニュー項目を選択します。





以前にクラウドに何も公開していない場合、値WindowsAzureMSDNの代わりに空になります。 厚くするには、リンクをクリックして 、パブリケーションプロファイルをダウンロードし、Visual Studioの[インポート]ボタンを使用してインポートします。 これにより、クラウドサービスの操作をIDEから直接実行できるようになります。



VS2012からクラウドにアクセスしたら、次のステップに進み、WCFサービスをホストするためのクラウドサービスを直接作成します。 [共通設定​​]タブで、[ クラウドサービス ] -> [新規作成 ]を選択し、[詳細設定]タブで、既存のストレージアカウントを指定します（リストで既に選択可能になっているはずです）。 クラウドサービスとストレージアカウントの両方を同じゾーンに維持するようにしてください。







これらの手順を完了したら、[公開]ボタンをクリックするだけで、プロジェクトがネットワークにリークします。



ステップ2.ストレージを使用する



最初のステップは、接続文字列をサービスのweb.configファイルに追加することです。 行は次のとおりです。

DefaultEndpointsProtocol = https; AccountName = service_name; AccountKey = that_the_storage_key_of_Storage



クラウドストレージを操作するには、REST APIを使用する方法と.NETラッパーを使用する方法の2つがあります。 この記事では、2番目のオプションを最も単純な方法として使用します。 さらに、独自のサービスを作成したのも無駄ではありませんでした。



したがって、この機能を使用するには、Windows AzureストレージパッケージをNuGetプロジェクトに接続する必要があります。





BLOBコンテナーの接続と管理は、クライアントを使用して行われます。 

using Microsoft.WindowsAzure.Storage; using Microsoft.WindowsAzure.Storage.Blob; CloudStorageAccount storageAccount = CloudStorageAccount.Parse(_connectionString); _blobClient = storageAccount.CreateCloudBlobClient();


リポジトリのさらなる作業は非常に簡単です。

同様に、データを取得できます： 

 //   CloudBlobContainer container = BlobClient.GetContainerReference(ContainerName); //    -  null if (!container.Exists()) return null; //      var blockBlob = container.GetBlockBlobReference(BlobNamePrefix + id); if (!blockBlob.Exists()) return null; //     var serializer = new XmlSerializer(typeof(CompositeType)); CompositeType result = null; using (var s = new MemoryStream()) { //     blockBlob.DownloadToStream(s); s.Seek(0, SeekOrigin.Begin); result = serializer.Deserialize(s) as CompositeType; } return result;


上記のメソッドはラッパークラスに配置され、WCFサービス自体には2つのアクションの単純なコントラクトがあります。 

 [ServiceContract] public interface IBlobService { [OperationContract] CompositeType GetData(int value); [OperationContract] void PutData(CompositeType data); }


!!! ここでは、重要だが絶対に自明ではないことを述べる必要があります。 BLOBコンテナーとブロックの名前は、小文字にする必要があります。 つまり、 BlobContainerという名前を作成しようとするとエラーが発生し、たとえば、 blob-containerは問題なく作成されます。



サービスを作成したら、クラウドに再度公開し、この手順を完了するとyour_service_name.cloudapp.net / BlobService.svcで利用できるようになります。



ステップ3.モバイルサービス



前の2つのステップは、「データ」を保存するためのインフラストラクチャを作成することを目的としていました。これは、常に必要とは限らず、オンデマンドで必要となる大量の情報です。 それでは、「メタデータ」の保存に移りましょう。 これを行うために、Windows Azureの最新のモバイルサービスを利用します。



新しいモバイルサービスの作成は、Windows Azureの他のサービスと同じくらい簡単です。 もう一度、 管理ポータルに移動します。作成メニューで、[ 計算 ] -> [モバイルサービス ]を選択すると、目の前にウィザードが表示されます。





その中に、将来のサービスの名前、ロケーション領域（1か所のどこかに位置することを忘れないでください）を示し、データを保存するストレージを示します（既存のものを使用するか、新しいデータを作成できます）。ウィザードを完了します。



サービスが作成されたら、雷の付いたクラウドアイコンでマークされたセクションのページに移動します（名前がわかりません）。 このページには、新しいWindowsストアアプリの作成と既存のWindowsストアアプリの接続の 2つのオプションがあります。 1つ目は、モバイルサービスで動作するように構成されたアプリケーションを使用してアーカイブをダウンロードすることを可能にし、2つ目は、既存のアプリケーションに埋め込む必要があるコードを提供します。 原則として、どのオプションを選択するかは問題ではありません。 2番目のものを使用し、アプリケーションを手動で作成することをお勧めします。



それまでは、まだそうしていません。モバイルサービスを使用する準備をしましょう。 これを行うには、最初に（トップメニューの）[データ]セクションに移動し、レコードを保存するための新しいテーブルを追加します。 好きなように呼び出すことができます。TestItemがあります。





このテーブルは、Windows 8アプリケーションで入力されたメタデータエントリをホストします。



ステップ4. Windows 8



それはそれがすべてだったものをする時間です。 Windows 8（WinRT）のアプリケーションは、Visual Studio 2012のExpressバージョンで無料で作成できます（WCFサービスで使用していたものの、現在はWin8アプリケーション用）。

ファイル->新規プロジェクト-> Windowsストア：





まず、モバイルサービスのメリットを享受できるように、必要なライブラリリンクを追加する必要があります。 お気付きの方は、前のステップで、Windows AzureからMobile Services SDKのダウンロードとインストールを提供されました。 これを行う必要があります。モバイルデバイス用のクラウドサービスを使用するためです。



SDKがインストールされると、標準の参照の追加手順を使用してプロジェクトに接続できます。





次に、WCFサービスへのリンクも接続します。 これも、標準の[ サービス参照の追加 ]ダイアログを使用して簡単に実行できます。





サービス参照を接続した結果、Visual Studioは特別なラッパークラスを作成します。これは、サービスコントラクトを持つ通常のクラスのように見えますが、実際には指定されたアドレスでリモートWCFにアクセスします。



最後に、モバイルサービスを使用するためのコードを追加します。これは、前述したように、Windows Azureのクラウドと稲妻を含むページで入手できます。 次のようになります。 

 public static MobileServiceClient MobileService = new MobileServiceClient( "https://***.azure-mobile.net/", "secret_key");


このフィールドをApp.xamlページに追加して、アプリケーションのどこからでもモバイルサービスにアクセスできるようにしました。

ここでは、いくつかのことに注意する価値があります。 まず、何らかの方法でネットワークで機能するすべてのメソッドは非同期です。 自動生成されたWCFサービスのクライアントクラスにも、非同期メソッド（PutDataAsync）が含まれています。 これは、アプリケーションがインターネットに接続しようとしているときに、ユーザーにブロックされたインターフェイスを強制的に表示させることなく、アプリケーションの応答性を高めることができるため、素晴らしいです。

さらに、両方のサービスの操作が非常にシンプルで透過的になりました。 WCFで動作するもの、モバイルサービスで動作するものは、どちらも.Netコードへの単純な呼び出しのように見え、すべての詳細を「どこかに」隠しています。 ここでは抽象的なデータアクセスレイヤーを記述しませんでしたが、混乱してすべてを正しく行うと、アプリケーションをテストするためのモックプラグインを簡単に作成できます。
