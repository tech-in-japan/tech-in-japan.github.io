Apache Hadoop 3.0.0 alphaを停止します



Apache Hadoop 3.0.0 alphaは、 さらなる悪用の可能性の停止要因となるエラーに遭遇した最初の兆候でした。 これはアルファ版であり、その運用におけるリスクの責任は個人的なドラマであると言えます。 私は同意しますが、どうやらロマンスの精神がリスクの本当の評価を覆い隠したようです。 2015年11月以来、Apache Tomcat 9を使用しており、これがアルファ版であるという事実に大きく後押しされました。 最も極端なバージョンを使用し、それに対応して最新の機能を取得しているという感覚は、特にその操作中に深刻な問題を思い出すことが不可能だったため、そのような選択の危険性に関する疑念のシェアを減らします。



Apache Tomcat 9.0.0.M22およびJetBrains IntelliJ IDEA 2017.1を停止します



平均して、月に1回、またはさらに頻繁に、Apache Tomcat 9（アルファ）の新しいバージョンがリリースされます。 リリース後、すぐにスタンドのバージョンを更新しました。最初は開発者向けのテスト版、次に戦闘版のバージョンです。 しかし、Apache Tomcat 9.0.0.M22アプリサーバーがJavaEEプロジェクトを再起動すると、JetBrains IntelliJ IDEA 2017.1.2がフリーズし始めましたが、OSスナップインからのサービスの再起動は問題なく進み、 コミュニティはケースを開始することを推奨し 、 .k。 Apache Tomcat 9.0.0.M19で問題が発生しました 。 7月、JetBrains IntelliJ IDEA 2017.2の新しいバージョンがリリースされましたが、Apache Tomcat 9.0.0.M22の問題は残りました。



その結果、Apache Tomcat 9を放棄してApache Tomcat 8.5.16にアップグレードする必要がありました。



ハードウェアが更新されていない場合、ソフトウェアの新しいバージョンをインストールする必要がありますか？



1か月前、ある都市から別の都市へのかなり長い移動（1200 km以上）の間に、私はWEBアプリケーション（フロントエンド-Java Script / HTML5、バックエンドJava SE / Java EE / ORM、Oracle DBMS 、MySQL、SQLite）。 旅行の1週間前に新しいバージョンのJetBrains IntelliJ IDEA 2017.1.2がリリースされたので、このバージョンに更新しました。 バージョンの印象から、私は豊富なコード分析を行いました。これにより、コンパイル前のエラーの数を排除でき、開発環境はさらに多くのヒントを提供します。 すべてがうまくいきますが、かなり深刻な問題が明らかになりました：コードを編集するとき（そのプロジェクトではJava Script / HTML5およびJava SE / Java EEのコンテキストでした）ラップトップのCPU使用率は非常に深刻な鉄の特性（Inter Core i5-2520、Memory DDR3- 16 GB、SSD）は20〜30秒間100％のキャラクターを獲得しました。 これは実際にバッテリー充電の災害になりました。 予想される3〜3.5時間ではなく、40分ですべて終了しました。 すなわち 以前のJetBrains IntelliJ IDEAは、原則として追加のメモリバーによって決定されたプロジェクトの読み込み時に1.5 GBのメモリを使用していましたが、CPU使用率が追加されました。



IntelliJ IDEAには、「知的部分」を無効にして、3年前にサンプルの機器を操作する機能を提供する機能があるかもしれません。 コミュニティの応答を得て、 ファイルをアクティブにします| 省電力モード 。 IntelliJ IDEAでこのオプションを有効にしても、バッテリーの寿命は延びませんでした。 ハードウェアレベルで本当に質問がありますか？



DockerはCisco Any Connect 4.5.00058にどのように影響しますか？



今週、自宅のPCにDockerをインストールしたいという思いから、Windows 10を再インストールすることにしました。 Dockerはカーネルの現在のバージョンが気に入らなかった。



職務を遂行するには、VPNを介してサーバーの特定のランドスケープに接続する必要があります。 これらの目的のために、Cisco Any Connectを使用します。 驚いたことに、VPNトンネルをアクティブにすると、ローカルホストにアクセスできなくなるという事実に直面しました。 ネットワークルートを確認すると、VPNインターフェイスメトリックの優先度が高いことがわかりました。 イーサネットとVPNインターフェースの優先度を決定して、手動で調整することにしました。 このCisco Any Connectで驚くことはありませんでした。VPNトンネルがアクティブになると、ネットワークインターフェイスの優先順位が以前と同じように再び選択されました。 一般に、Cisco Any Connect 4.5.00058はOS管理者の設定を考慮せず、ネットワークルーティングの優先順位を選択するという認識に基づいて機能します。 全体として、OS管理者の要望を考慮に入れることができるOpenConnectを支持して拒否する必要がありました。
