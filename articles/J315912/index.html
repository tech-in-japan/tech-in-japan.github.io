1.コンソールの接続



ボードを操作するには、コンソールをローカルに接続する必要があります（ただし、ブート後のほとんどの操作はSSHでも実行できます）。 これは単純に行われます。インターネットには十分な情報があります。たとえば、 WD MyBook Live UART Portはここにあります。



ピン配列：




RS232-UARTアダプターを介してコンピューターのRS232ポートに接続されます。 3.3V以内のボード上のロジックレベル。 デフォルトのボーレートは115200です。



接続して電源を入れると、コンソールにu-bootブートローダーが表示されます。 
U-Boot 2009.08-svn65645 (Oct 08 2012 - 14:36:50), Build: 0.2.5 CPU: AMCC PowerPC UNKNOWN (PVR=12c41c83) at 800 MHz (PLB=200, OPB=100, EBC=100 MHz) Bootstrap Option E - Boot ROM Location NOR/SRAM (8 bits) 32 kB I-Cache 32 kB D-Cache Board: Apollo-3G - APM82181 Board, 2*SATA, 1*USB I2C: ready DRAM: Auto calibration 256 MB FLASH: 512 kB DTT: 1 FAILED INIT Net: PHY EC1 Register: 0x2c8c ppc_4xx_eth0 Type run flash_nfs to mount root filesystem over NFS Hit any key to stop autoboot: 0




「Apollo-3G-APM82181ボード、2 * SATA、1 * USB」というメッセージは、USBがなく、SATAが1つしかないため、少し間違っています。 これは、ブートローダーがMyBook Liveと2つのディスクを搭載したDUOモデルに共通しているためです。



コンソールで何もクリックしないと、ハードドライブからの起動が試行されます。 私たちの場合、そこにはないので、いずれかのボタンを押すとダウンロードが停止します。



2.ディスクなしでダウンロード



u-bootブートローダーには、機器を操作するための多くの機能があります。



たとえば、3色LEDのすべての色を見ることができます。
LEDをオフにする（ブートローダーの青色に点灯）： 

 ledtestoff


LEDの3色を交互にテストします。 

 ledtest




コマンドのリストを表示するには、helpと入力します。 ブートローダーの詳細については、 こちらをご覧ください。 コンソールでは、ロード時に、u-bootがnfsファイルシステムをマウントするオプションを提供することがわかります。「run flash_nfsと入力して、NFSを介してルートファイルシステムをマウントします」。 この場合、別のオプションを使用します-TFTPサーバーからRAMイメージをロードします。



TFTPD32（ 公式サイト ）などのサーバーがサーバー上に立ち上がり、ボードを構成します。 ローカルIPアドレスとTFTPサーバー（コンピューター）のアドレスを設定します。 それらは同じサブネットからのものでなければなりません： 



 setenv ipaddr '192.168.1.1' setenv serverip '192.168.1.10'


TFTPサーバーから要求されるファイルの名前を表示します。 



 => printenv fdt_file bootfile fdt_file=apollo3g.dtb bootfile=uImage


変数fdt_fileにはデバイスツリーファイルの名前が含まれ、ブートファイルにはオペレーティングシステムのRAMディスクイメージファイルの名前が含まれていることに注意してください。



setenvコマンドでのアドレスと同様に、名前はより便利な名前に変更できます。 調整後、変数の新しい値をROMに保存することをお勧めします。

3. LEDEでのコンパイル



ボードのオペレーティングシステムのソースとしてLEDEを使用します。 これはOpenWRTのフォークであり、MyBook Liveボードをサポートしています。 ここからダウンロード： apm82181-lede 手順に書かれているとおりにパッケージをダウンロードします。 



 ./scripts/feeds update -a ./scripts/feeds install -a


コンフィギュレーターを起動します。 



 make menuconfig


システム（「ターゲットシステム」）として「AppliedMicro APM821xx」、サブシステム（「サブターゲット」）として「SATA（NAS）から起動するデバイス」、プロファイル（「ターゲットシステム」）として「Western Digital My Book Live」を選択します。 最終イメージ（「ターゲットイメージ」）で、ramdiskをマークする必要があります。



私たちの裁量でパッケージを選択します。 コンフィギュレーターで設定を保存して終了したら、makeコマンドでコンパイルを開始できます。



プロセスの終了後、エラーが発生しなかった場合、必要なファイルをピックアップしてTFTPサーバーに配置できます。



  • ファイルbin / targets / apm821xx / sata / lede-apm821xx-sata-MyBookLiveSingle-initramfs-kernel.binの名前をuImageに変更します
  • ファイルの名前をbuild_dir / target-powerpc_464fp_musl-1.1.15 / linux-apm821xx_sata / tmp / lede-apm821xx-sata-MyBookLiveSingle-initramfs-kernel.bin.dtbからapollo3g.dtbに変更します


したがって、ディスクなしでシステムをブートする段階を経ました。



4.ポート管理（LuCIおよびコンソール経由）



このボードの回路と、一般的にインターネット上のAPM82181プロセッサに基づくデバイスの回路は見つかりませんでした。 したがって、部分的にリバースエンジニアリング（リバースエンジニア）します。つまり、ボード上の接続を推測し、これを検証します。 接続されたポートはデバイスツリーで確認したり、カーネルからの情報を確認したりできるため、LEDE開発者はこれを部分的に行ったようです。



入出力ポート 
 root@lede: cat /sys/kernel/debug/gpio GPIOs 496-503, platform/4e0100000.gpio1, 4e0100000.gpio1: gpio-498 ( |Reset button ) in hi GPIOs 504-511, platform/4e0000000.gpio0, 4e0000000.gpio0: gpio-504 ( |enable EMAC PHY ) out hi gpio-505 ( |Enable Reset Button,) out lo gpio-506 ( |Power USB Core ) out hi gpio-507 ( |Power Drive Port 1 ) out hi gpio-508 ( |? ) out lo gpio-509 ( |? ) out hi gpio-510 ( |? ) out lo gpio-511 ( |Power Drive Port 0 ) out hi




ご覧のように、1つのポートはボタンを押すために使用され、残りは制御のための出力に使用されます。 デバイスツリーファイル（拡張子が.dtsのファイル）を編集すると、オペレーティングシステムのポートを機能から解放できることは明らかですが、ボードの機能ブロックの機能に影響する可能性があります。 LEDを除く。 このインジケータに接続されているポートは、独自の目的に使用できます。 疑問符で示されたポートはそれに接続されています。 また、LED制御は、たとえばコンソールから利用できます。



デフォルトでは、ロード後、LEDEインジケータが緑色に点灯します。 だからあなたはそれをオフにすることができます： 



 echo 0 > /sys/class/leds/mbl\:green\:power/brightness


そして、青をオンにします。 



 echo 1 > /sys/class/leds/mbl\:blue\:power/brightness


LEDEがLuCI Webシェルで構築されている場合、LEDはブラウザーから制御できます。



ネットワークカードのIPアドレスに移動します




[システム]メニューで、[LED構成]に移動し、特定の各LED色がトリガーするイベントを構成します。 たとえば、これは着信および発信ネットワークパケットで緑色のLEDが点滅する方法です。



ネットワーク活動




ニーズに応じてLEDポートを使用する場合、LEDを制御するサービス（/etc/init.d/led）もドライバーを必要としないことは明らかです。 そして、次のようなスクリプトから制御を行うことができます。 



 echo 1 > /sys/class/gpio/gpio508/value


5. I2Cバスへの接続



したがって、少なくとも3つのポートを使用できます。 もちろん、もっとリソースが欲しいです。 これらのポートにポートエクスパンダーを配置し、I2Cソフトウェアポートに接続できます。 その後、低速でありながら、ほぼ無制限の数のポートにダイヤルできます。



ボードにはたくさんの空いている席があり、何かが彼らから離婚しているかどうかを確認したいという願望がありました。



たとえば、プロセッサの近くにある、8本の脚を持つチップのU12とマークされたシート：



はんだ付けケーブル




すべての接点が3.3Vを超えないことを確認した後、デジタルアナライザーを接続しました。

ブートプロセス中に、信号は検出されませんでした。 次に、I2Cバスの説明が最初は存在しなかったため、デバイスツリーに追加されました。 



 IIC0: i2c@ef600700 { compatible = "ibm,iic"; reg = <0xef600700 0x00000014>; interrupt-parent = <&UIC0>; interrupts = <0x2 0x4>; fast-mode; #address-cells = <1>; #size-cells = <0>; };


LEDE構成では、コア（kmod-i2c-core）でI2Cサポートを有効にする必要があることは明らかです。



コンパイルし、TFTPサーバー上のファイルを更新して再起動すると、デバイス/ dev / i2c-0が表示されます。 



 root@lede: dmesg | grep i2c [ 4.816060] i2c /dev entries driver [ 4.819878] ibm-iic 4ef600700.i2c: using standard (100 kHz) mode


echoコマンドで文字を送信することにより： 



 root@lede: echo 11 > /dev/i2c-0 ash: write error: Remote I/O error


デジタルアナライザーによって信号が検出されました。




したがって、このボードのI / O機能を大幅に拡張したI2Cハードウェアバスが見つかりました。



U12の信号の位置（5-GND、7-SDA、8-SCL）から判断すると、これはおそらくADM1032ARMZ-温度センサーと見なされます。



6. PCF8574ポートエクスパンダーの接続



I2Cバスに何かを接続します。 在庫には、2x16の親しみやすさを測定する液晶インジケータがあり、それぞれ8x5ピクセルでした。 HD44780コントローラークローンとPCF8574ポートエクスパンダーに基づいて中国製。



LCD HD44780 16x2 I2C






説明によれば、5Vと3.3Vの両方の電力と信号レベルが可能ですが、確認すると、3.3Vの電力では画像のコントラストが非常に低いことがわかりました。 5Vの電力は0〜5 Vの信号レベルにもつながり、ボードは3.3向けに設計されているため、プリミティブな信号レベルコンバータを作成する必要がありました。 I2Cバスは双方向であるため、デバイスから確認信号を受信する必要があります。したがって、少なくともSDAラインに双方向コンバーターが必要です。 非常に適切な回路オプションについては、5Vおよび3.3Vデバイスのロジックレベルの一致で説明します 。 私たちの場合、それが使用されました



絶縁ゲート電界効果トランジスタ回路、




確かに、トランジスタは似たようなものから選択されています-BSS123。



ブレッドボードに組み立てる：






したがって、5Vの電力がLCDインジケータに供給され（SATA電源コネクタからボード上で取得）、両方の電源がレベルコンバータに供給されます-3.3および5V。



次に、ポートエクスパンダーをLEDEに追加します。 カーネルサポート（カーネルモジュール-その他のモジュール-kmod-gpio-pcf857x ... PCX857x、PCA967xおよびMAX732X I2C GPIOエクスパンダー）、再コンパイル、サーバーへのアップロード、および再起動を有効にします。



ドライバーの存在を探します。 



 root@lede: lsmod |grep pcf gpio_pcf857x 6128 0


接続します（0x27-I2Cバス上のエキスパンダーのアドレス）： 



 root@lede: echo pcf8574 0x27 > /sys/bus/i2c/devices/i2c-0/new_device [ 218.957888] pcf857x 0-0027: probed [ 218.961321] i2c i2c-0: new_device: Instantiated device pcf8574 at 0x27


ポートの外観を確認します。

ご覧のとおり、アドレス0x27のgpiochip488ポートのグループがI2Cバスに現れました。 これで、ポートエクスパンダーを介して機能するという事実にもかかわらず、システムに表示されるポートと、プロセッサにあるポートを追加できます。

作業の方向を結論に設定します。 



 root@lede: echo out > /sys/class/gpio/gpio488/direction root@lede: echo out > /sys/class/gpio/gpio489/direction root@lede: echo out > /sys/class/gpio/gpio490/direction root@lede: echo out > /sys/class/gpio/gpio491/direction


入力する必要がある場合は、コマンド「in」を指定する必要があります。 たとえば、テスターまたはデジタルアナライザーで出力状態を測定するなどして、出力の動作を確認します。 HD447880 LCDがエキスパンダーに接続されている場合、エキスパンダーの3番目のビットがインジケーターのバックライトを制御するという事実を活用できます。 



 root@lede: echo 1 > /sys/class/gpio/gpio491/value root@lede: echo 0 > /sys/class/gpio/gpio491/value


最初のコマンドはバックライトをオンにし、2番目のコマンドはバックライトをオフにします。 起動プロセス中にデバイスを追加するには、デバイスツリーに追加できます。 



  IIC0: i2c@ef600700 { compatible = "ibm,iic"; reg = <0xef600700 0x00000014>; interrupt-parent = <&UIC0>; interrupts = <0x2 0x4>; fast-mode; #address-cells = <1>; #size-cells = <0>; /* Boot ROM is at 0x52-0x53, do not touch */ /* Unknown chip at 0x6e, not sure what it is */ i2cled: led@27{ #gpio-cells = <2>; compatible = "nxp,pcf8574"; reg = <0x27>; gpio-controller; }; };


i2cledラベルの後の行は、コマンド「echo pcf8574 0x27> / sys / bus / i2c / devices / i2c-0 / new_device」を置き換えます。



そのため、エキスパンダーを介してI2Cバスに接続されたI / Oポートを透過的に使用する機会が現れ、実際に実装されています。



最初の部分については、おそらく十分でしょう。 2番目の部分では、当然のことながら、コメントに明らかな対立がない限り、インジケーターを使用してシステムの状態を表示します。







このために、HD44780ドライバーが使用され、ファイナライズされました。 LCD4Linuxパッケージが接続され、既存のドライバーが適合しなかったため、VT100ターミナルコマンドの表示を制御する新しいディスプレイドライバーも書き込まれました。
