完璧さに制限はありません



私の以前のゲームは、サーバーからのphpファイルによって起動されました。 今回はさらに先へ進むことにしました。 私はすべてがインターネット上で動作するだけでなく、ファイルサーバーからも動作することを望んでいました。 第一に、オフラインのバージョンは、競争の条件で必要とされ、第二に、たとえばインターネットがない場所で自分でプレイできるようにしました。 （一般に、私がすることはすべて自分で行います。そして今、ゲームをプレイします。）しかし、必要であれば、サーバーからも、AIではなく人と、つまり、このオプションも存在する必要があります。 一般に、タスクはすべてのスクリプト、3Dモデル、テクスチャ、およびサウンドを含むユニバーサルアセンブリを作成し、インターネット上のページ上、およびダウンロード、解凍、およびインデックスによってゲームを実行するだけのアーカイブ形式で配布することでした。 html もちろん、後者のオプションは、シングルユーザーバージョンでのみ意味があります。



基本的に、前述のローカルバリアントは、相対パスを含む保存されたWebページです。 キャッチは、写真付きのテキストだけでなく、3Dエンジン、およびテクスチャ付きモデルも含まれていることです。 ただし、エンジン自体に問題はありません。 これはTHREE.JSで、JavaScriptで記述されており、重量が500 KBで、単純にSCRIPTタグで接続されています。 ただし、エンジンはローカルファイルシステムから3Dモデルをロードする方法を知りませんが、サーバーからajaxを使用してそれらをプルすることを望んでいます。 モデルがスクリプトファイルからも接続されるように、このためにブートローダーを終了する必要がありました。 アドレスバーに注意してください。







Webバージョン用に最初に作成した3Dモデルはmodel.jsonタイプのファイルに保存されていましたが、SCRIPTタグで接続すると、Firefoxですべてが機能しますが、Chromeではセキュリティポリシーに関連するエラーが発生します。 解決策は簡単でした。jsonの名前をjsに変更します-問題ありません。 おそらく、目的のコンテンツタイプを強制することは可能でしょうが、誰が何を知っているのか、将来、ブラウザにはまだセキュリティポリシーが存在するでしょう。 突然、ローカルファイルシステムから* .js以外の方法で呼び出されたファイルからスクリプトをダウンロードすることはできなくなります。 そして、js-そして質問はありません。 スクリプトをダウンロードしてください。 3Dモデルの読み込み： 



var el=document.createElement("script"); el.type="text/javascript"; el.src=url+'.json.js'; el.async=true; document.getElementsByTagName("head")[0].appendChild(el);


また、ブラウザのセキュリティポリシーにより、サーバーバージョンで完全に動作するモデルのテクスチャは、pngおよびjpgファイルからローカルに接続することを拒否したため、テクスチャタイプのファイル名でスクリプトをbase64に変換して実行する必要がありました.png.js。 さて、これらのスクリプトを、動的に作成された古き良きSCRIPTタグに接続してください。 とても簡単です！ 



 var image = document.createElement( 'img' ); image.onload = function() { var tex = new THREE.Texture( image ); tex.needsUpdate = true; tex.wrapS = tex.wrapT = THREE.RepeatWrapping; tex.anisotropy = 5; }; image.src="data:image/gif;base64,"+imagedata.replace(/^data:image\/(png|jpg);base64,/, "");


これは次のように機能します：私のモデルパーサーはファイル内の画像へのリンクを見つけ、これらのリンクは.jsの末尾で到達し、テクスチャ画像のスクリプトは動的に接続され、画像はbase64から転送され、モデルにスーパーインポーズされます。 そして、スクリプトの読み込みが完了した瞬間を確実に判断するために、最後にそのような各スクリプトに、メインプログラムに読み込まれたことを知らせる関数の呼び出しを追加しました。これで、それに含まれるイメージまたはモデルを操作できます。

まず、もちろん、競争。 その過程で、コンテストの主催者はhtmlファイルを使用したゲームの開始に満足せず、確かにexeが必要であることが判明しました。 jsからの実行可能ファイルのアセンブリには何がありますか？ 最初に試したのはElectronです。 しかし、そこにはすべてが私にとって複雑すぎることが判明しました。 githubから何人かのマネージャーをインストールし、これらすべてを構成して組み立てるために何十もの操作をインストールする必要がありました。 Electronは、標準的なWeb開発の道を私に導きたいと思っていることを理解しています。 競争ゲームの作成に割り当てられた最後の日だったので、これに対処する時間はありませんでした。 いいえ、それは良くありません。 そして、web2exeユーティリティに出会いました。 すべてが非常に簡単です-起動して、htmlファイルに挿入し、node-webkitを単独でダウンロードし、exeファイルが出力されたアセンブリを取得します。 私が気に入らなかったのは、ファイルが105 MBという非常に大きいことが判明したという事実です。 うーん...ゲーム自体の重量はわずか15 MBです。 しかし、最大のnw.dllファイルを圧縮するパッケージングユーティリティが見つかりました。 その結果、アセンブリ全体の重量は68 MBになり、zipアーカイブでは43 MBになりました。

私は競争に勝つことはできませんでしたが、解決には非常に貴重な経験がありました。 今、私は不可能はないと確信しています。
