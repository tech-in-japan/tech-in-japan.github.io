3D PDFとeDrawingsの何が問題になっていますか。 アプリケーションで3Dモデルビューアーを置き換えた方法

これは、新しいC3DビューアーをどのようにLOTSMANに統合したかというストーリーです。PLM製品ライフサイクル管理システム、その理由、そして何をしたかです。







.1「二次提出」とは何ですか？



注目に値するPLMソリューションには、PDMメカニズム/サブシステム（製品データ管理）が含まれます。



PDMシステムデータベースには、3Dモデル、図面、仕様、計算など、さまざまなCADシステムで作成されたドキュメントとファイルが保存されます。



適切なCADアプリケーションを職場にインストールしていない（および非常に高価な）ユーザーがこれらのドキュメントのコンテンツにアクセスできるようにするために、PDMシステムは「ドキュメントのセカンダリプレゼンテーション」（一部のドキュメントのコピー）を生成します「ニュートラル」公開形式。



「セカンダリビュー」がシステムにロードされた後、それを使用して、PDMシステムのクライアントインターフェースでドキュメントの内容のプレビューを直接作成できます。 調整プロセスで外部ユーザーと情報を交換するとき、およびメモと注釈を使用してコメントを交換するときの媒体として使用します。



.2パイロットでの二次プレゼンテーション：PLM



VRML、eDrawings、3D PDFは、LOTSMAN：PLMの3Dモデルの「二次表現」として交互に機能していました。



私はVRMLについては話しません-若者の間違い...それが起こらない人と。



eDrawingsは悪くありませんでしたが、そのためのアダプターとサポートサービスを開発するために非常に控えめなお金を払わなければなりませんでした。 さらに、2014年から、eDrawingsはx64バージョンでのみ利用可能になりました。 すべての要望がありましたが、32ビットクライアントのLOTSMAN：ActiveXとしてのPLMにそれを埋め込むことはできませんでした。



最初は3D PDFは贈り物のように思えました-無料で、すべての最初のコンピューターに既にインストールされており、銀河のスリーブの事実上の標準、埋め込みに適した既製のActiveXライブラリが、時間が経つにつれて、その暗黒面が明らかになり始めました：



  1. 当社は、配布物にAcrobat Readerを配布する権利を留保します。 Acrobat Readerが最初のすべてのコンピューターにインストールされているわけではないことが判明しました。 Acrobat Readerの互換性のないバージョンがそこにインストールされることもあれば、まったく異なるPDFビューアーアプリケーションがAcrobat Readerと互換性がないこともありますが、同時にユーザーの心にとてつもなく重要です。
  2. Acrobat Readerの開発ベクトルは予測不能です。 各リリースは驚きに満ちており、不要なメッセージのウィンドウを自動的に閉じ、迷惑なパネルを最小限に抑え、霧に覆われたオプションをインストールするように設計された豊富なツールを補充します。
  3. Acrobat Readerの更新リリースは突然であり、避けられません。 「セカンダリプレゼンテーション」が機能しなくなったことをユーザーメッセージから確認します。
  4. 開発者からのフィードバックはおそらく可能ですが、悲しいフォーラムの定期的な研究が示すように、無駄です。
  5. ActiveX Acrobat Readerを使用すると、アプリケーションがクラッシュする場合があります。 最終的にAcrobat ReaderでSafeModeを使用しました。 このモードでは、非表示のAcrobat Reader子プロセスのみが定期的にクラッシュし、アプリケーションは引き続き動作します。
  6. ActiveX Acrobat Readerは、オプションのない32ビットバージョンでのみ使用できます。


そして最も重要なことは、大きなモデル（アセンブリ）で作業するときのパフォーマンスが不十分であるということです。



  • KOMPAS-3Dから3D PDFへのエクスポート速度が遅い。

    おそらく、責任の一部はKOMPAS-3Dにありますが、それは-です。
  • 大規模な3D PDFファイルサイズ。ファイルシステムとネットワークの負荷を増加させ、より多くのRAMを必要とします。

    モデル出力の精度とファイルサイズの相関関係の観点から最適な設定を検索しても、許容できる結果は得られませんでした。
  • 大規模な3D PDFモデルのダウンロード速度が遅い。
  • 大きなファイルのレンダリングの速度/品質が不十分です。


.3 C3Dビューアーの紹介



私たちは常にC3D Labsの成功と革新に関心を持っていました（結局、ASCONの子会社、同僚）。 新しいC3Dビューアーの出現は見過ごされませんでした。 製品の初期ベータ版にアクセスした後、機能とパフォーマンスの比較調査を実施しました。

この研究の結果、プロジェクトを立ち上げるきっかけになりました。その結果は、Pilot：PLMで3Dモデルの2次表現を表示および注釈する手段としてC3Dビューアーコンポーネントを組み込むことでした。



  • PDFと比較してC3D形式への保存を高速化-6から18倍に！

    モデルのサイズとコンポーネントの数との相関は見つかりませんでした。
  • C3DファイルはPDFファイルよりも小さく、2〜39倍です。
  • C3Dファイルのダウンロード速度は、PDFのダウンロード速度よりも高速です（6〜264倍）。

    モデルのサイズとコンポーネントの数との相関は見つかりませんでした。
  • C3Dビューアーの回転表示の品質と滑らかさは、Adobe Readerの品質を大幅に上回ります。

    不明な理由により、Adobe ReaderでFPSを測定することは必ずしも機能しませんでした。 測定が成功した場合、FPSは値「5」を超えませんでした。 C3D Viewerは、さまざまなモデルで30〜100 ++ FPSの速度で「ねじれ」ます。
このプロジェクトへの関心は相互のものであり、開発者とのやり取りの質とさまざまな問題の解決速度に最高の効果をもたらしました。

リクエストに応じて、C3D Labsチームは3Dモデルのアノテーション機能を開発し、ロシア語にローカライズしました。



API C3D Viewerのいくつかの改善は、アプリケーション内のコンポーネントの特定のアプリケーションによるものでした。 その結果、新しいメソッドとイベントがAPIに登場しました。これにより、モデルのロードの進行状況と緊急割り込み/ダウンロードのキャンセルを示す独自のメカニズムを実装できるようになりました。



また、eDrawingsの高コストについて述べたので、C3Dビューアーも無料ではないが、その使用条件ははるかに民主的であると言っても過言ではありません。



.4埋め込み



最初に、C3Dビューアー、LOTSMAN：PLMクライアントアプリケーションを組み込む必要があったシステムの簡単な技術的な説明を行う必要があります。



これは、Delphi 2006で記述されたMDIインターフェイスを備えたWin32デスクトップアプリケーションです。

MDI子ウィンドウのレイアウトは、相互に接続された一連の「パネル」の説明に基づいて動的であり、それぞれが指定されたオブジェクトに関する情報の特定の側面を表します。

C3D Viewerソフトウェアコンポーネントは、ActiveX COMライブラリ（C ++、Qt）です。



APIはシンプルでコンパクトであり、主要な操作は非同期であり、別のスレッドで実行されるため、コンポーネントとの対話はメソッド呼び出しとイベント処理に基づいています。



数時間でセカンダリビュー表示モジュールのプロトタイプを組み立ててLOTSMAN：PLMに統合することができましたが、他の「ささいなこと」にはかなり長い時間がかかりました。



独自のツールバー、コンテキストメニュー、モデルの構造を表示するパネル、および注釈のリストを含むパネルを、アプリケーションインターフェースの一般的な視覚スタイルに従って、必要なボリュームと形式で実装しました。



このコンポーネントは、組み込みビュー、フルスクリーンビュー、および注釈モードの2次ビューを操作するいくつかのモードで使用します。











問題は何でしたか：



タイプライブラリの説明をインポートする



最初の問題の1つは、Delphi 2006でActiveX C3Dビューアーからタイプライブラリの説明が正しくインポートされなかったことです。これはQtを使用したC ++で記述されています。



Delphi 2010タイプライブラリのインポートユーティリティを使用しましたが、100％正しい結果が得られませんでした-私はまだ「手動で」12行を修正する必要がありました。



「採用」の難しさ



このアプリケーションでは、ユーザーはWYSIWYGレイアウトエディターにアクセスできます。このエディターを使用すると、情報付きの「パネル」を追加および移動できます。 これらのパネルの1つにActiveX C3Dビューアーがあります。



パネルをActiveX C3Dビューアーから別の「コンテナー」に移動すると、その部分的な「破壊」が発生することが判明しました。



その理由は、VCL Delphiの親ウィンドウ変更メカニズムの特定の実装にあることがわかりました。 要するに、winapiはDestroyWindow ActiveX C3Dビューアー関数を呼び出し、その後、Delphiはそれを「生きている」と見なし続けながら、自身を「殺した」と見なし始めます。



親ウィンドウが変更されているかどうかを判断する方法を知っている特別なコンテナにActiveXを配置することでこの問題を回避し、その時点でActiveXが正しく解放されて再作成されます。



メモリの問題



32ビットアプリケーションがあり、「メモリ不足」はキャンセルされていません。 限られた2 GBのRAMの状態で巨大なアセンブリが発生する可能性があるため、必然的に問題がより深刻になりました。



モデルのC3Dビューアーへの読み込み中にメモリが不足すると、アプリケーション全体が予期せずクラッシュするという事実が特に鮮明でした。

理由をすぐに判断できませんでした。 JDBGデバッグ情報を使用してスタックを復元するサンプリングプロファイラーに基づいて、独自のPostMortemデバッガーを開発する必要がありました。



理由を理解したので、使用可能なRAMの制限を3.5 GBに増やし、IMAGE_FILE_LARGE_ADDRESS_AWAREフラグをEXEヘッダーに追加することから始め、アプリケーションのメモリ消費をわずかに減らしましたが、結果は良くありませんでした。



実際には、32ビットMDIアプリケーション（マルチウィンドウインターフェイスを使用）がある場合、ユーザーは原則として常識により無制限にウィンドウを開くことができます。 そして彼は間違いなくそれを行います。 権利があります。



メモリの問題の可能性を減らすために、メモリの使用を監視および最適化するメカニズムをアプリケーションに組み込みました。 特定の制限に達すると、現在非アクティブなMDIウィンドウが「スリープモード」に浸され、リソースが解放されます。



仮想マシンの問題



C3D Viewerには、OGLを表示するために少なくとも2.1が必要です。 さまざまな仮想マシンを試しました。これについての情報を次に示します。



Hyper-V-C3Dビューアーが機能しません。

Virtual Box 4.1.44 + Window 7-C3Dビューアーが機能します。

VMWare Player v14（設定で3Dグラフィックス+ OGL v3を高速化）-C3Dビューアーが機能します。



その結果、新しいビューアの開発とデバッグのアクティブフェーズには約2か月かかりました。 さらに2〜3週間かけてバグを修正しました。
