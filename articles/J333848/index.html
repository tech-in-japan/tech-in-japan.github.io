Reduxの力とは何ですか？







これは、記事「 Reduxの素晴らしいところ 」（ ジャスティンファルコーネ著）の翻訳です。







Reduxは、Reactコンポーネントの状態を使用して伝えるのが難しい複雑な状態の相互作用を巧みに処理します。 実際、これはオブジェクト指向プログラミングにも見られるメッセージングシステムですが、言語に直接組み込まれているのではなく、ライブラリとして実装されています。 OOPと同様に、Reduxは制御を呼び出し側から受信側に転送します。インターフェースは状態を直接制御しませんが、処理のためにメッセージを渡します。







この点で、Reduxのストレージはオブジェクトであり、レデューサーはメソッドハンドラーであり、アクションはメッセージです。 store.dispatch({ type:"foo", payload:"bar" })

このアプローチの利点は何ですか？







  • 上記の制御の反転により、状態変更の実装が変更されるとすぐに、ユーザーインターフェイスを更新する必要がなくなります。 ロギング、取り消し、タイムトラベルデバッグなどの複雑な機能を追加するのは簡単です。 統合テストは、正しいアクションが送信されたかどうかを確認するためだけに削減され、残りについては単体テストで十分です。







  • Reactのコンポーネントの状態は、ユーザー情報やアラートなど、多くのアプリケーションモジュールに影響を与えるエンドツーエンドの機能と連携するには遅すぎます。 Reduxには、このためだけに、ユーザーインターフェイスに依存しない状態ツリーがあります。 さらに、インターフェイスの外部で状態を処理する場合、localStorageまたはURLでのシリアル化が1か所で実行されるため、一貫性を維持しやすくなります。







  • method_missing

投稿の途中でモナドを説明することに真剣ですか？







モナドはHaskellで一般的な抽象化であり、リストの操作やエラー、状態、時間、I / Oの処理など、さまざまな計算に使用されます。 do-notationの形式の構文糖は、命令コードに似た形式でモナドの操作のシーケンスを表すことができます。これは、非同期コードを同期のように見えるJavaScriptのジェネレーターにほぼ似ています。







最初の問題は、アプリケーションの観点からモナドを記述することは完全に正しいとは限らないことです。 修道士は、副作用と逐次計算を扱うためにHaskellに登場しましたが、抽象的な意味では、それらはそれとは何の関係もありません：彼らは単に2つの関数の相互作用に関する一連のルールであり、特別な意味はありません。 結合性は、算術演算、集合の演算、リストの結合、およびヌル伝播にも同様に適用できますが、それらとは独立して存在します。







2番目の問題は冗長性です。これは、少なくとも、命令型アプローチと比較したモナドの視覚的な複雑さを意味します。 Maybe



ような明確に定義されたオプション型は、落とし穴をnull



として見つけるよりも安全ですが、それらのコードは長くて扱いにくいです。 Either



タイプを使用したエラー処理は、例外をどこにでもスローできるコードよりも理解しやすいように見えますが、同意するように、例外を含むコードは、 Either



で値を定数で返すよりも簡潔です。 状態、入力/出力などの副作用については、命令型言語ではまったく些細なことです。 関数型プログラミングのファン（私を含む）は、関数型言語の副作用を扱うのは非常に簡単だと主張するでしょうが、どんな種類のプログラミングも簡単だと納得できる人はいないでしょう。







これらの例を広く見てみると、その利点は本当に顕著です。モナドの法則に従うだけでなく、同じ法則です。 1つのケースで機能する一連の操作が、他のケースで機能する場合があります。 リストのペアをペアのリストに変えることは、条件でPromise

私の意見では、ReactとReduxを比較するのは興味深いです。ReactはReduxよりもはるかに複雑で、APIははるかに広く、奇妙なことに、学習と使用が簡単です。 React APIで本当に必要なのは、 React.createElement

「原子状態」は抽象的な概念であり、実際に利益をもたらすのは、それを理解した後です。 一方、Reactでは、どのように動作するのか理解していなくても、 setState

Redux開発チームとそのユーザーコミュニティはどちらもAPI拡張に積極的に反対していますが、現在行われているように、多数の小さなライブラリをまとめることは、初心者にとっても厄介な作業であり、退屈な作業です。 Reduxが単純なケースの組み込みサポートのレベルまで成長できない場合、このニッチを占める「救世主」フレームワークが必要になります。 ジャンプスーツは良い候補になる可能性があります。多対多の関係の性質を維持しながら、特定の機能の形でアクションと条件のアイデアを具体化します。 しかし、救い主の選択は救いの事実ほど重要ではありません。
