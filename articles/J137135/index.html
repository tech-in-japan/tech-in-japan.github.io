Nokia向けに開発する価値はありますか？

こんにちは、愛するhabravchane！



Nokiaスマートフォンの開発と公開の経験を共有したいと思います。 私は、すべてが良いか悪いかを示す目標を設定していません。また、他のモバイルプラットフォームの開発と比較するつもりもありません。 この投稿は、Nokia向けのソフトウェアを書くべきか、それとも始めたばかりか、と考えている他の人たちと経験を共有しようとする試みです。 いずれにせよ、この情報が誰かが私の間違いを避け、時間を節約し、モバイルアプリケーションの分野で成功するのに役立つことを願っています。 それから、興味のある多くのテキストがあります-catへようこそ。



物語。 スタート。


ノキアのSymbianモバイルプラットフォームの開発について知り合いになったのは、2009年初頭で、最初のiPhoneとNokia 5800の後に君臨するある程度の興奮に屈したときから始まりました。それ以前は、スマートフォンを持っていませんでした—私は大学3年生でした。余分なお金を稼ぎ始めました-一般的に、典型的な学生は半飢えた日常生活です。 その結果、彼はお金を節約し、新しいNokia 5800を手に入れました。以前はこのブランドの携帯電話しかなく、その間機器は非常に高レベルだったため、選択はNokiaに委ねられました。 ITの専門分野を勉強していたので、すぐに開発者の観点からこの電話に触れたいと思いました。



Symbian C ++


当時、Symbianでは、JavaまたはC ++のいずれかを作成しました。 当時、C ++は非常に馴染みがあったので、Symbianの「方言」を勉強することにしました。 率直に言って、新しいプラットフォームの研究は多少の困難を伴いました。

  • Leave例外の新しい概念（C ++例外の軽量アナログ）;
  • 2フェーズオブジェクトコンストラクター、Leave関数（例外をスローする可能性のある関数、およびメモリをクリーニングする必要がある関数）のオブジェクトとのスタック。
  • 証明書の取得とパッケージの署名に手間がかかります。
  • アセンブリルールとパッケージアセンブリシステム自体の記述。
  • リソースファイルを介してダイアログボックスを記述し、これらのファイルを接続します。
  • 3つの識別子（UID）とさまざまなアクセス権（機能）の組み合わせ。
  • 特にこれらのSDKがいくつかある場合、SDKのセットアップはかなり混乱していました。
  • いくつかの例と貧弱なドキュメント。


一般に、Symbian C ++で作業した人は、当時の恐怖をよく覚えています。 はい、時間の経過とともに、もちろん慣れますが、それでも開発プロセスの中で、多くの場合、これらの小さなものだけでなく、これらの小さなことに気を取られなければなりませんでした。 実際、私はその時点で深刻なことは書きませんでした。たとえば、試験やその他の「Hello、World」のためのあらゆる種類のチートシートです。 当時はOVIストアはなく、アナウンスしかありませんでした。 それにもかかわらず、私は非常に良い経験を得たので、将来役に立ちました。



Qtの外観


2009年の終わりまでに、Symbian C ++での開発を何とか忘れていました。モチベーションはあまりなく、パートタイムはますます多くの時間を費やしていました。 一方、ノキアはOVIストアを立ち上げ、新しいiPhone 3GSが登場し、モバイルアプリ市場は活況を呈していました。 しかし、今年の終わりには、ノキアの世界で新しいイベントが開催されました-SymbianおよびMaemoをサポートするQt 4.6フレームワークのリリースです。 たまたまQtの開発に積極的に取り組んでいたので、過去を思い出して何が変わったのかを見ないのは罪でした。 多くが変更されました：

  • 新しいQt Creator IDEがCarbide.c ++に置き換わるようになり、Qt、便利なヘルプシステム、自動ビルドシステムとの統合が改善されました。今では、通常のC ++と強力なグラフィカルコンポーネントセットでSymbianを開発するのがより簡単になりましたSymbianの下で、スピードで輝きませんでした。
  • Symbian用の改善されたデバッガが登場しました。これはQtクラスを「デプロイ」できました。
  • パッケージの署名と作成のシステムを簡素化しました。
  • 新しいSymbianエミュレータが登場しました。


一般的に、Nokiaの進歩はSymbianの開発を単純化する方向で顕著でしたが、これは容易ではありませんでした。 しかし、古い問題は残り、どこかに新しい問題が追加されました。

  • Qt Creatorは非常に生々しく、多くの場合、プロジェクトの再構築後にクラッシュしました（彼は今日までこれを犯しています）。
  • 組み立て速度が非常に低かった-プロセスを開始して、お茶を飲んだり、インターネットで記事を読んだりすることができました。完全な組み立てには、特に時間がかかり、苦労しました。
  • 5800デバッガーは非常に頻繁にバグがあり、動作を停止し、Qt Creatorは電話をまったく認識しませんでした。
  • リリースされたエミュレーターはWindowsで動作しました。つまり、エミュレーターでアプリケーションをテストするには、Windowsでアプリケーションをビルドする必要がありました。それは素晴らしいですが、Windowsでビルドされない場合はどうでしょうか。
  • 一部のQtダイアログは非常に貧弱に見えました（たとえば、QInputDialog）が、古い形式でリソースファイルに追加するのはそれほど簡単ではありませんでした。
  • アセンブリルールは手動で記述する必要がありました。


それとは別に、登場したOVIストアに注目します。 最初は、登録料が支払われ（50ユーロ）、必要に応じて、証明書にお金を費やす必要もありました（現在の登録費用は1ユーロです）。 しかし、出版の準備をしていて、店の仕事の過程で、「楽しい」ささいなことがわかりました。

  • 何らかの理由で、アプリケーションを更新すると、最初に説明が更新される場合がありますが、それはプログラム自体の数日後です。 しかし、ユーザーが説明と一致しないアプリを取得した場合、素晴らしいですか？
  • 通常のインターフェイスマニュアルはすぐに表示されなかったため、いわばゴミ捨て場が作成されました。
  • QAが公開前にアプリケーションをチェックするために割り当てられた時間は、どこでも明確に規制されていなかったため、誰かが1か月間チェックしました。 そして今、彼らはあなたのアプリケーションの2週間（！）の更新をチェックすることができます。 そして、結果がポジティブになるという事実ではありません。 フォーラムには、さまざまな人がアプリケーションをチェックし、同時にさまざまな評決を出すというメッセージがいっぱいです。
  • アプリに関するお客様からのフィードバックは一切ありません。 愚かな評価システムしかありません-間違いを修正するために間違いの詳細を人に尋ねることさえできません。 さらに、低い評価の最大のシェアは、ユーザーがアプリケーションをダウンロードしていないという事実に由来しています（ノキア、それは本当に開発者の問題ですか？）。
  • 店自体は非常に奇妙です。 たとえば、ユーザー評価を持たない一部のアプリケーションには、すでに5つ星が付いているのはなぜですか（これが最大値ですか）。 どんなメリットがありますか？ または、スクリーンショットのみを配置できるのはなぜですか（注意！）256x256以下のサイズで？ この作品に表示できるのは、画面の断片または何も見えない写真で、電話の画面は正方形ではありません！ 私はまだこの狂気を理解できません。
  • Nokiaはこのストアについて気にしなかったようです。なぜなら、Nokiaはアプリケーションの公開準備のすべての「落とし穴」を説明することができなかったからです。 例で説明しましょう。Symbianは通常SVGを受け入れません。SVGTinyのモバイルバージョンが使用されます。 しかし、明らかに、彼女には問題があります。 SVGでアプリケーションアイコンを作成し、Adobe Illustratorでオブジェクトに影を追加すると、Tiny SVGでNokiaの独自プログラムでそれを追い越し、次に電話の「スミア」アイコンで追い越します。 どうやら、影はサポートされていません。 そして、Nokiaのサイトでアイコンを作成する例のビデオを偶然見つけたときだけ、Illustartorが影を作成するための特別なアクションを持っていることがわかりました。 つまり、彼らはこの横棒については完全に知っていましたが、ウィキにはそれについて何も書いていませんでした！ 誰もが一見明らかな10分間のビデオを見るわけではないので、なぜそれについて書くことができなかったのですか？
  • Nokiaにはさまざまなデバイスがあり、多くを確認する必要があり、さまざまなシステム（現在はS60v5、Anna、Belle）にアセンブルする必要があります。多くの時間がかかります。




上記のすべてをまとめると、アプリケーションや出版物の平凡な準備に開発自体よりも時間がかかった時期があったこともありますが、それはあまり甘くありませんでした。 確かに、Nokiaは開発者がQtに切り替えるために積極的にキャンペーンを展開しており、彼にとって素晴らしい未来を予測しています。



HarmattanとQt Quickの登場


そのため、Qtは積極的に開発され、生産性は向上し、Qt Creatorは向上していました（Symbian ^ 3で電話が落ちなくなり、その間にN8デバイスに切り替えました）。 アセンブリは非常に加速し、今は空の待ち時間で眠れぬ夜を過ごしませんでした。 Symbianの伝統に関する他のすべての恐怖は保存されており、どこにも行くとは考えていませんでした。 私は特に、Qt Creatorですべてができるとは限らないという事実を特に「気に入っています」。 たとえば、特定のMIMEタイプのレコグナイザーを作成します（アプリケーションが指定されたファイルを自動的に開くことができるように）。 独自のUIDマジックを使用して、別個のライブラリとして別個のSDK（最大3ピース）でそれをアセンブルし、次にアセンブリルールを手動でQt Creatorに追加する必要がありました。 また、このMIMEタイプのサポートをQtアプリケーションに追加することも、別の叙事詩です。



しかし、しばしば起こるように、敵は気付かれずに忍び寄った。 現在、ノキアは新しいQt Quick（JSベースの言語）でアプリケーションを書くことを推奨し始めていますが、その機能はやや劣ります（ファイルを開くダイアログが必要な場合は自分で作成してください！）。QWidgetに基づく本格的なQtと比較して。 ノキアは、QWidgetに基づいたモバイルプラットフォームの開発を発表しました。 彼らが言うように、1年も経っていない。 いいえ、ノキアはアプリケーションをさらに書くことを禁止していませんでしたが、開発者は待ち望んでいました。 QtアプリケーションをMeeGo（実際にはLinux）で問題なくアセンブルすると、電話機を回転させるときにウィンドウの向きを変更できなくなります（ランドスケープモードのみ）。 すべての感嘆符には、Qt Quickへの書き換えという1つの答えがありました。 そのように。 彼らは、MeeGoプラットフォームが完全に構築されたアプリケーションを取得できない可能性があることを気にしませんでした。開発者は市場を失うでしょう。



そしていつものように、MeeGo自体のアプリケーションの準備には驚きがありました。 たとえば、一般的なタスクは、メニューにアプリケーションアイコンを表示することです。 しかし、何らかの理由で、Qt Creatorのデフォルトのビルドルールは、表示されないという事実につながります。 インターネットを検索し、Nokia wikiで問題を解決する方法を見つけます-アイコンを別のディレクトリに配置する必要があります。 まあ、すべてがエミュレータに表示され始めましたが、デバイス自体には再び何もありません。 どんなトリックですか？

特に、MeeGoのエミュレータに注目したいと思います。 これは、ロードに時間がかかり、動作が非常に遅く、ボタンが2つしかありません（つまり、加速度計などで失敗する）、400メートルの死体です。 しかし、実際には、タイトな「顔」とファームウェアイメージを持つ通常のQEMUです。 Nokiaの紳士Bravoは、デバイス自体を使用せずにMeeGoで開発するのは、単純に非現実的です。

その結果、この時点で、主要なNokia製の携帯電話で開発するには、S60v5、Symbian ^ 3（Anna、Belle）、およびMeeGoに基づくデバイスが必要です。 悪くないよね？ さらに、さまざまなデバイスに対して、独自のQt QuickコンポーネントとQtウィジェットのセット。 四方を引き裂くのは苦手ですか？ その結果、私はMeeGo向けの開発を続けず、S60v5とSymbian ^ 3のみに限定しました。



最後の行為。 Windows Phone


明らかに、ノキアの開発者はすでにこのように蹴られることに慣れていますが、昨年何が起こるかは誰も予想していませんでした。 Nokiaは、Windows Phoneに切り替える予定であり、Symbianは徐々に取り上げられると述べています。 素晴らしい動き。 今日、彼らはあなたに-明日Qtで、明日-クイックで、明後日-あなたが知っていたすべてを忘れて-Windowsに切り替えます。 最初は、QtがWindows Phoneに移植され、すべてがうまくいくと多くの人が考えていましたが、公式の神話はこれらの神話に反論しました。 はい、多くの人はこの出来事の転換と、開発者に対するそのような態度にショックを受けました。



現時点では、プラットフォームは非常に新しくて若いので、時間の不足と十分な欲求のために試してみる時間がありませんでした。 私は興味があると認めていますが。



エピローグとして


Nokiaデバイスの開発中に、プログラミングスキルとモバイルプラットフォームの開発経験の両方を習得しました。 たくさんの前向きな感情を受け取ったとは言えません。 私自身、ノキアは開発者にとって非常に軽視されており、プラットフォームの飛躍を実現し、実際に顔を直接だましていると結論付けました。 はい、開発プロセスを簡素化する試みがありましたが、明らかに、Symbianが遠い過去に根ざしているという事実は、ある程度、計画を実行することを許可しませんでした。 同社には明確に定義され指示された開発ベクトルがなく、開発者も同様に支払いました。 SymbianおよびMeeGoの開発に取り掛かると、Qt Quickを使用してプロセスを簡素化できますが、ストアでのすべての遅延を節約できるわけではありません。 開発自体以外に時間を無駄にする準備をしてください。



Windows Phoneの登場により、会社のポリシーが変更される可能性がありますが、時間のみが表示されます。



PSノキアを開発して大金を稼ぐという目標を設定しなかったので、アプリケーションに集中したくないので、電話しません。 通常、彼らはここで興味を持っていますが、あなたはなんとか稼ぐことができましたか？ 好奇心の強い人のために、私は事前に答えます。このアプリケーションは年間約1,500ユーロをもたらし、そのうち30％がNokia自身に支払われました。 私はプロモーションを実施せず、広告も行いませんでした。 一般的に、デバイスのコストは支払われ、それは良いです:)
