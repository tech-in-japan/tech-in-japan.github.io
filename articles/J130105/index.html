スパムの送信方法

はじめに



この短い記事では、Oracleからのメール送信の設定方法と終了方法について説明します。 すぐに2つのことを明確にしたいと思います。1つ目は、スパミングのためにこれをしなかったことです。2つ目は、メールサーバーのセットアップに取り組んでいませんでした。







Oracleセットアップ



私のタスクの1つを解決するために、ユーザー認証データを送信する必要に直面しました。 テーブルにあるログインとパスワード。 率直に言って、現時点では数百人や数千人のユーザーがいないので、誰もが手で手紙を送ることができますが、これは数時間の手作業です。



作業の最後の場所では、毎月レポートをユーザーにメールで送信する必要があったため、タスクは私にとってエキゾチックに見えませんでした。また、手作業に見合った時間を処理できると期待していました。 一方、必要なインフラストラクチャはすべて手元にあり、必要なトピックの専門家がいて、彼らと話すことができました。 ここには何もありませんでした-自分で理解する必要があります。



Oracleには、データベースからメールを送信するための多くの方法があります。必要な機能を備えたパッケージの一部を次に示します。
  • UTL_MAIL
  • UTL_SMTP
  • UTL_TCP
  • UTL_HTTP
これらのパッケージには、必要なロジックがすべて含まれています。 最新のものはUTL_MAILパッケージで、これを使用することにしました。 このパッケージは、デフォルトではOracleアセンブリに含まれておらず、手動でインストールする必要があるとすぐに言います。 パッケージの一部は2つのファイルにあります。

1.パッケージ仕様は、ファイル@ $ ORACLE_HOME \ rdbms \ admin \ utlmail.sqlにあります。

2.パッケージ本体は暗号化され、ファイル@ $ ORACLE_HOME \ rdbms \ admin \ prvtmail.sqlにあります。



これはSYSユーザーの下からロールアウトする必要があり、 コマンドウィンドウPL / SQL Developerからパッケージを正常にインストールできませんでした インストールされましたが、コンパイルエラーが生成されました。 sqlplusからすべてが完全にインストールされました。



ロールアップ後、パッケージを使用するための許可を与える必要があります。 一部の著者は、 Publicに許可を与えることを提案していますが、これはセキュリティ上の理由から良い考えではありません。 適切なユーザーに許可が与えられた後、 SYSユーザーの下での作業は完了しました（ただし、 alter systemを実行する必要がある状況が依然として必要です ）。



メールサーバー



結局のところ、メールを送信するためのインフラストラクチャ、つまり Oracleを除いて、私が持っていないメールサーバーが必要です。インストールと構成の方法も想像できませんでした。必要ですか？



短い検索の後、次の記事http://jiri.wordpress.com/2010/03/24/send-emails-using-utl_mail-and-google-gmail-smtp-server/を見つけて、GMAIL経由でメールを送信する方法を説明しました。 記事の著者は私のような状況に陥り、簡単な解決策を提案しました。それを使用することにしました。 GMAILで新しいメールボックスを開始しました。このメールボックスは、 E-MailRelayのダウンロードとインストールを自動的に行うために使用されます。 すぐにそれはうまくいきませんでした。ドキュメントを少し読んで、このサーバーとの相互作用を説明する記事をもっと探す必要がありました。 その結果、必要なすべてのパラメーターがbatファイルに入力され、メールボックスデータがemailrelay.authファイルに入力され、サーバーが正常に起動しました。



手紙を送る



メールサーバー（またはこの場合のようにSMTPプロキシ）をインストールした後、Oracleのアドレスを指定する必要があります。 
alter system set smtp_out_server = 'ip-address:port' scope=Both;
私の場合、OracleとSMTPプロキシは同じマシン上にあるため、 アドレスは127.0.0.1、ポート25です。



utl_mail.sendを使用してメールを送信してみましょう。このため、設定で少し遊ぶ必要がありました。たとえば、送信者フィールドには次の形式のアドレスが必要であることがわかりました。

"<any_mail@any_mail_server.com>" 、同じ場所で送信者の名前を指定することもできます。

「送信者名<my_mail@any_mail_server.com>」ですが、メッセージは実際のGmailメールボックスからとにかく届き 、次のようになります。

「送信者名<my_mail@any_mail_server.com>」<real_mailbox@gmail.com>



mime_typeを使用してさまざまなメールサーバーに送信する少しの実験により、ロシア語のテキストには'text / htmlを使用することが望ましいことが示されました。 charset = "UTF-8" '



コントロールが自分に手紙を送信した後、必要なデータを選択し、各ユーザーにユーザー名/パスワードを送信する手順が作成されました。 最初の20人のユーザーにデータを送信するためのこの手順を開始したら、昼食に出かけました。



約40分後、最初の4文字だけが消え、残りはどこかに留まっていることがわかりました。 デバッグの準備として、同じ手順を使用してメールボックスに10文字を送信しました。 新しいパッケージは失われた手紙を「押し通し」、それらはすべて正常に放置されました。 一般に、SMTPプロキシは機能しましたが、多少の矛盾はありました。



物語の続き



1日後、サーバーで自分のビジネスを行うと、SMTPプロキシが約80文字を送信していることがわかりましたが、最初のメール送信後は何も送信しませんでした。 私はgmail'eのボックスをチェックすることにしました。 受信トレイには、3つのレターパックがありました。





3つのチェーンすべてに、存在しないメールボックスへのメッセージ送信に関するメールサーバーの応答が含まれていました。 次の写真が送信されたメッセージで観察されました。





メールボックスからvbibiorm@gmail.comとw852@ymail.comのメールボックスに2つの文字が残っており、件名の各文字には、SMTPプロキシがインストールされているマシンのIPアドレスが含まれ、文字の本文は空でしたが、名前がnonameでサイズが0バイトの添付ファイルが1つありました。





明らかに、SMTPプロキシには脆弱性があり、添付ファイル付きのメールを送信するときにのみ表示されるのか、添付ファイルにマシンの脆弱性/特性に関する追加情報を含める必要があるのか​​わかりません。 両方のメールボックスにメッセージを送信する時間は同じであるため、それらは同じ人物に属していると結論付けます。 発見された脆弱性の結果、次のタイプの文字がSMTPプロキシを介して送信されました。





Google翻訳では、文字の言語を中国語と定義しています。 これらの電子メールに関する情報をGoogleが検索したところ、これらのメールボックスへの手紙の送信に関連するメールサーバーの不可解なアクティビティについてユーザーが不満を言うことがよくありました 。 一般に、これは私の辞書に「オープンリレー」というフレーズが現れる方法です。 SMTPプロキシをオフにしましたが、質問は残ります。次に何をすればよいですか？ 私は3つのオプションを見ました：
  1. メールを頻繁にパッケージ全体で送信することは計画されていなかったため、すべてをそのままにしておくことができます。 手紙の送信時にのみメールプロキシを有効にします-時間内で最も安価なソリューションです。
  2. メールサーバーのトピックを完全に理解し、これを正しく構成するか、より信頼性の高いものに変更することができます-時間内にかなり高価なオプションであり、さらに、この知識は将来私にとってまったく役に立ちません。
  3. 既存の知識を使用して回避策を見つけるには、時間のコストは不明です。
  4. メールサーバーに精通している友人を見つけます。
元の問題を考えると、3番目のオプションは、ソリューションプロセスから「楽しい」という条件で最も有望に見えました。 さらに、オプション1と4はほぼ同等です。



オプション



最も簡単で理想的なオプションは、サーバーを追加せずに、Gmail経由ですぐにメールを送信することです。 試してみましょう： 
 alter system set smtp_out_server = 'smtp.gmail.com:587' scope=Both;
メッセージを送信しようとすると、次のエラーが返されます。「ORA-29279：SMTPパーマネントエラー：530 5.7.0最初にSTARTTLSコマンドを発行する必要があります。 m29sm5336584poh.20。」 私はGmailのドキュメントに行き、他のポートを試し、httpsではなくhttpを介してメールボックス設定で作業を設定しようとしました。 その結果、エラーメッセージは変わりましたが、メールは消えませんでした。 インターネットでの検索では、いくつかのポイントが示されました。
  1. GmailとYahooでは、STARTTLS（SSLの別名）を使用する必要があります。
  2. UTL_MAILはSTARTTLSをサポートしていません。
  3. 特に頑固な仲間はutl_smtpパッケージとutl_smtp.command（conn、 'STARTTLS'）コマンドを使用しようとしましたが、失敗しました。 ドキュメントのどこかに、STARTTLSがバージョン11.2からOracleによってサポートされていることがわかりました。
  4. コミュニティの一般的な意見は次のとおりです。
  • GmailやYahooを使わず、自分をだまさないでください。
  • Gmailを使用する場合は、SMTPプロキシを自分で設定します（オプション。記事の冒頭で説明します）。
  • 何をしているのか理解できたら、utl_smtpパッケージを使用してメールサーバーへのログインを試みることができます。
ポイント4.aの実装を試みることにしました。別のメールサーバーを探しています。そこでSMTPの設定を見つけ、 alter systemを実行します 。 手紙を送ろうとすると、「ORA-29278：一時的なSMTPエラー：421サービスが利用できません。」というエラーが表示されます。 設定を再読み取りし、utl_smtpパッケージを介して認証を使用してオプションを接続しようとします（このオプションの説明は、たとえばここにあります ）。 別のメーラーでもGmailでも機能しません。 私は既に知っていることの説明でブラウザのタブを常に閉じていますが、潜在的に有用なものの数は無限にある傾向があります：





これを終了する時が来たことを理解しています。次の検索中に、単純な考えを見つけました：私のポート25は閉じられていますか？ チェックします（判明しましたが、その方法がわかりました）-閉じました。 代替メールサーバーとポート587を示す代替システムを作成します。UTL_MAILを使用して電子メールを送信しようとすると、「ORA-29279：Permanent SMTPエラー：501 sender address must must be domain」というエラーが表示されます。 それはほとんど起こりました。送信者の名前に含まれる必要なGmail'y "<>"を取り除き、手紙を残しました。



おわりに



この記事では、現場でメールを送信した経験と、メールサーバーの操作に関する必要な知識がない場合について説明しました。 以前にインストールしたSMTPプロキシと同じ「漏れやすい」メールサーバーが見つかったため、結果はあまり満足のいくものではありません。 さらに1分を費やした後、たとえば次のように指定するだけで、メールサーバーに存在するメールボックスに代わってメールを送信できることがわかりました。 
 v_sender varchar2(200):='  <admin@mail.******>'
ここで、mail ******は、見つかったメールサーバーの名前です。 しかし、問題を解決し、漏れやすいプロキシとその設定を理解する必要性を取り除き、UTL_MAILおよびUTL_SMTPパケットについてもう少し学びました。 オプション4.cも扱う予定でしたが、うまくいきませんでした。時間をかけないことにしました。誰かがGmail経由でOracleから直接手紙を送信し終えたら、それについて読むのは面白いでしょう。 （スパムを検出した後）メールの送信を再構成し、並行して記事を書くと、どこかで6〜7時間かかりました。
