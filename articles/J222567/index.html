Androidアプリケーション内のマークアップの視覚的な編集

前文


EclipseとIdeaには、Androidアプリケーション用の独自の視覚編集マークアップツールがあります。 NetBeansにはこの幸福がありません。 おなじみのNetBeansに似たようなものを比較的簡単な手段で作成したいという願望から、ビジュアル編集のプロセスをアプリケーション自体に移すというアイデアが生まれました。 いくつかの理由があります。

  • Android自体によるNatural Previewマークアップ。
  • デスクトップIDEなしでオフラインで作業する機能（まず、デザイナーにとって興味深いかもしれません）。
  • 「神は神であり、シーザーはシーザーです。」 実装する場合、Viewオブジェクト、クラス定数などの既存のデータ構造を使用すると便利です。




どのように機能しますか？


ミラーツリーマークアップ。 主なアイデアは、Androidがレイアウトをロードするときに生成するマークアップオブジェクト（ビュー）のツリーと並行しています。プログラムは、独自のオブジェクト（プレフィックスZ-Zview、ZButton、...を持つ同じ名前のクラス）のミラーツリーを作成します。 それぞれにオブジェクトへのリンクがあります-オリジナルです。 クラスにはViewと同様の継承があり、たとえば、xmlマークアップファイルの生成時、Viewパラメーターリストの作成時など、ツリーのトラバースに関連するすべての機能をサポートします。 さらに、ミラーツリーは、開いているレイアウトを記述するための主要なデータ構造です。各オブジェクトには、ビューのマークアップパラメーターのベクトルとLayoutParamsのミラー記述子オブジェクトの2つの重要なパラメーターが含まれます。



表示オプション。 元のさまざまなViewパラメーターは、ZParam後継クラスのグループによってサポートされています。各クラスは、特定のタイプのパラメーターの外部および内部表現、変換、編集ダイアログの呼び出し、コンパイルされたファイルの値のタイプのコンプライアンスの決定のすべての詳細を実装します。 現在実装されている型は、int、hex（16進int）、boolean、float、string、charSequence、dimen（dimensions）、id（リソース識別子）、image、color、enumInt（名前付き整数定数のリスト）、enumString、enumClassです。 （名前付き整数クラスのリスト）。 後者の場合、列挙クラスごとに、独自の派生クラスが作成されます。



各ビューのパラメーターベクトルは、アセットにある対応するxmlファイルから逆シリアル化されます。 このようなファイルを作成する手順を簡素化するために、アプリケーションには、すべてのビューのゲッター/セッターを検索し、それらから提案されたパラメーターの説明を作成するコンポーネントがあります。 次に、これらのファイルはマニュアルに従って手動で編集されます。



LayoutParamsレイアウトオプション。 マークアップオプションの場合、同様の記述ファイルは存在しません。 代わりに、各ビューのLayoutParamsが読み取られ（そのタイプは祖先のタイプに依存します）、リフレクションの助けを借りて、同じ名前のクラスのハンドルが生成されます-ZLPとその派生物、各クラスはパラメーターのベクトルを作成し、対応するLayoutParamsから読み取られた値を書き込みます



コンパイル済みレイアウトxml。 Androidアプリケーションファイル（apk）には、レイアウト記述のxmlファイルがコンパイルされた形式で含まれています。 形式の詳細な説明は、 justanapplication.wordpress.com/category/android/android-binary-xmlで入手できます。 この形式の疑いのない利点は、パラメータの内部データ型の数が限られていることです。これは、独自の手段で解析するのに十分単純です（たとえば、ミラーマークアップツリーでダンプを表示したり、パラメータを取得したりする）。 さらに、内部Androidパーサーはandroid.contentクラスです。 res.XmlBlockは、同様の形式のデータを含むバイト配列からビューを作成できます。



備考 apkファイルでは、コンパイルされたファイルは圧縮された形式で含まれています。 ただし、この場合、これは重要ではありません。 XmlBlockパーサーはそれらを既に解凍します。



どのように機能しますか？


起動時に、アプリケーションはリフレクションを使用してAndroidクラスに必要な定数のコレクションをポーリングし、<name-value>ペアの形式でテーブルを生成します。

  • R.javaのリソース識別子（異なるタイプ-サブクラスごとに）;
  • クラス定数を表示する
  • android.R.attr-すべてのパラメーターの内部名定数。
  • android.R.styleable-パラメーターで使用される定数の記号による指定。


アセットに配置されたxmlファイルからのすべてのタイプのビューでは、パラメーター記述子オブジェクト（ZParamクラス）のベクトルが逆シリアル化されます。



この方法でアウトラインを編集します。 元の未編集のレイアウトは、標準のLayoutInflaterによってリソースIDを通じてロードされます。 ロードされたビューは、オブジェクトのミラーツリーを作成します。各ビューに対して、パラメーターの元のベクトルがコピーされます。 次に、標準のandroid.content.res.XmlResourceParserに基づいて構築された独自のCXmlResourceParserが、すべてのViewのパラメーターを読み取り、ZParamオブジェクトに書き込みます。 値がマークアップファイルから読み取られたパラメーターオブジェクトはマークされているため、パラメーターの一般リストで常に強調表示されます。 すべてのビューのイベントハンドラーは、編集ダイアログのプログラムコードに対して構成されます。



パラメーターを変更すると、ネイティブCXmlParserはコンパイル済みxmlファイルの形式でバイト配列を書き込み、そこからXmlBlockパーサーが変更されたビューを作成し、プレビュー画像を変更します。 CXmlParserは、バイナリファイルへの変更を長時間保存するときにも同じことを行います。 変更されたレイアウトをロードする場合、同じスキームが機能しますが、異なるペアのパーサーである内部android.contentのみです。 ビューを読み込むためのres.XmlBlockと、パラメーターを読み込むための独自のCXmlParser。

外部で使用するために、マークアップを通常のxmlファイルにエクスポートできます。 このために、ZViewおよびZParamクラスグループの独自のツールを使用します。



データの配置 。 すべてのデータは、SDカードのGUIWizardディレクトリにあります。 アプリケーションは、独自のパッケージの名前でディレクトリを作成します。 編集されたマークアップの生成されたxmlファイルには、レイアウトの名前が書き込まれます。 cxml拡張子を持つ補助コンパイル済みマークアップバイナリは、データサブディレクトリに書き込まれます。 ビュー用に生成されたパラメーターファイルは、独自のクラスの名前（android.widget.ImageView.xmlなど）でプロジェクトディレクトリに書き込まれます。



備考 最初は、変更されたパラメータを編集するときに、セッターによって対応するビューにすぐに書き込まれ、存在しない場合はプライベートビューフィールドに直接書き込まれると想定されていました。 同様に、フィールドの現在の値を取得するには、ゲッターを呼び出す必要がありました。 ただし、将来的には、「parameter-field-setter-getter」という明確な例外にはかなりの数の例外があることが判明しました。 たとえば、複数のセッターが1つのパラメーターに対応し、1つのセッターが複数のパラメーターを取ることができます。 したがって、ZView-Viewオブジェクトの相互作用を完全に放棄することが決定されました。 代わりに、各パラメーターの変更後、コンパイルされたxmlマークアップファイルが生成され、バイトストリームに書き込まれます（つまり、これは文字通りの意味でのファイルではありません）。 次に、XmlBlockパーサーはそこから変更されたビューを作成します。



反射、どのようにそれなしで？


アプリケーションは、以下を含むリフレクションを集中的に使用します。

  • Androidパラメータ名定数、アプリケーションリソース、Viewクラス定数の読み取り。
  • 現在のオブジェクトのクラス名に応じて、クラスZview、Zparam、およびZlayoutのグループのオブジェクトを生成します。
  • Androidの一部のクラスでアクセス制限をバイパスします。


NetBeansの統合


NetBeansに組み込まれたモジュールは簡単な仕事をします。 追加のボタンが押されると、ADB呼び出しを通じて、SDCard / GUIWizard / <Androidアプリケーションのパッケージ名>ディレクトリの内容がスキャンされます。 パッケージ名は、プロジェクトウィンドウで選択した現在のプロジェクトのマニフェストファイルから抽出されます。 拡張子が.xmlのすべてのファイルが/ res / layoutにコピーされ、削除されます。 古いバージョンは、拡張子.origに変更されます。 モジュールソース-bitbucket.org/solus_rex/netbeans_guiwizard_plugin



現状


プロジェクトのソースは、 bitbucket.org / solus_rex / android_guiwizardで入手できます。



コンパイルされたマークアップファイルの解析をデバッグするには、アプリケーション自体に含まれるバイナリファイルを使用できます。 これを行うには、apkファイルを解凍し、xmlファイルをres / layoutからGUIWizard / <application package> / dataサブディレクトリにコピーする必要があります（これらはコンパイル済みバイナリ形式で格納されていることを思い出してください）。 レイアウトでは、ソースバイナリマークアップファイルと、サイズが一致したときにプログラムによって生成されたファイルのバイト比較を実行できます。 これを行うために、アプリケーションは、出力ファイルを作成するときに、名前とパラメーターリストのテーブルをソースファイル内の順序で並べ替えます。



現在の段階の目的は、主要なアイデアをテストし、レイアウトに実装することでした。 レイアウトでは次のことができます。

  • 定数のリストを表示します。
  • 独自のデータログと例外スタックを読み取ります。
  • ビューパラメータ記述ファイルを生成します。
  • 新しい空のレイアウトを作成します。
  • レイアウトリストを表示します（すべてのリストで、短いクリックでメインコマンドが実行され、長いクリックでコマンドのコンテキストメニューが表示されます）。
レイアウトリスト内の短いクリックでプレビューが呼び出され、バイナリファイルに最後に保存されたファイルが開かれます。ファイルがない場合は、アプリケーション自体のリソースです。









ロングクリックすると、レイアウトで実行されるコマンドのメニューが表示されます。 プレビューでは、任意の要素を短くクリックするとそのパラメーターのリストが表示され、生成されたxmlタグを含むメッセージを長くクリックします。 コマンドメニューの[リストの表示]項目には、ビューの名前とタイプがネストされた形式でリスト表示されます。 このリスト内を短くクリックすると、すでに説明したパラメーターのリストが表示され、長いクリックでは、選択したビューに子として新しいビューを追加するコマンドがあるコンテキストメニューが表示されます。









パラメータリストを短くクリックすると、そのタイプに対応するダイアログが表示され、長時間クリックすると、それに割り当てることができるリソースのリストが表示されます。









新しいリソースを追加する問題は未解決のままです-それらは元のプロジェクトに結び付けられなければなりません。
