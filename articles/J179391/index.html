膝上監視-Cactiを使用してJavaアプリケーションを制御する

この記事では、大規模なJavaアプリケーションのリソース消費を分析および制御するタスクの例を使用して、 Cactiを使用した監視ソリューションについて説明します。



私のタスクは、メモリ消費とパフォーマンスの問題を抱えた大規模な3層Javaアプリケーションを短時間で安定させるための手段を提案することでした。 時間は、いつものように短いです：すべてのための1-2週間。 会社には適切なアプリケーション監視インフラストラクチャがなく、それを作成するのは私の仕事ではありませんでした。 JConsoleを使用するオプションは、長時間にわたる消費を分析し、アプリケーションの突然の再起動の可能性を監視する必要があるため、適切ではありませんでした。



私が働いていた会社の1 つでは 、 RRDツールに基づいてJavaアプリケーションを監視するためのソリューションは、利便性とシンプルさの点で印象的でした。 これは、HTTPを介したデータ収集と表示を提供する単純なアドオンperlスクリプトと、アプリケーション自体の多くの改善データ収集エージェントで構成されていました。 私にとってこれは解決策のアイデアでしたが、RRDを介してバインディングを作成する時間はありませんでした。



慎重に検索した結果、必要なアドオンを実装する無料ツール-Cactiを見つけました。 Cactiは、Apache-PHP-MySqlフレームワークで記述されたアプリケーションで、Webインターフェイスに基づいて監視データの収集と表示を設定できます。 インフラストラクチャを強化し、データ収集エージェントとすべてをセットアップして追加するのに数日かかりました。



さらに、この記事では、私の問題を解決し、最終的には会社のアプリケーションを正常に安定させることができた解決策について詳しく説明しています。







仕事に必要なもの：

  • Cacti 0.8.7i（実装時の最新バージョン）
  • Apache 2.2.21（Cactiのエンジン）
  • PHP 5.3.8（Cactiが記述されているプラ​​ットフォーム）
  • MySQL 5.5（Cacti設定の保存）
  • RRDTool 1.2.30（図の描画と監視データの保存）


（監視が開始されたバージョンが示されています。Apache-PHP-MySqlバンドルが動作します）



Cactiでの監視の原則について説明します。Windows（またはUNIXのcron）の「スケジュールされたタスク」を使用して、データ収集エージェントのポーリングが定期的に開始されます。 収集された情報は、RRDデータベースに格納されます。 循環バッファ内にファイルとして。 さらに、RRDから蓄積されたデータは、分、時間、日、月などのさまざまなスケールでCacti Webインターフェースを介して表示できます。



作業計画は、必要なすべてのインフラストラクチャを引き上げ、分析したアプリケーションをデータ収集に適合させ、Cactiでデータの収集と出力を構成することです。

データ収集を選択して構成する




データを収集する2つの方法-SNMPとJVMおよびアプリケーションの一部としてのJMXサーバーのポーリングを試しました。 SNMPはCactiでサポートされており、JVMメモリの使用量のみを監視する必要があり、これを非常に迅速に行う必要がある場合に使用するのが妥当です。 私はSNMPから始めましたが、最初の成功の後、JMXに切り替えました。 CactiはJMXをサポートしていないため、追加の柔軟性を支払う必要があります。レシーバーと応答をJavaで記述するには手間がかかります。



以下は対応するコードです。



メモリステータスについてJVMをポーリングするためのコード（同様に、コードは任意のJMXサーバー用に記述されています）： 



import java.io.Closeable; import java.io.IOException; import java.lang.management.MemoryMXBean; import java.lang.management.MemoryUsage; import javax.management.JMX; import javax.management.MBeanServerConnection; import javax.management.ObjectName; import javax.management.remote.JMXConnector; import javax.management.remote.JMXConnectorFactory; import javax.management.remote.JMXServiceURL; public class JvmHeapJmxClient { public static void main(String[] args) { JMXConnector jmxc = null; try { String jmxHost = args[0]; String jmxPort = args[1]; String jmxConnectionString = "service:jmx:rmi:///jndi/rmi://" + jmxHost + ":" + jmxPort + "/jmxrmi"; JMXServiceURL url = new JMXServiceURL(jmxConnectionString); jmxc = JMXConnectorFactory.connect(url, null); MBeanServerConnection mbsc = jmxc.getMBeanServerConnection(); ObjectName memoryMBeanName = new ObjectName("java.lang:type=Memory"); MemoryMXBean memoryMBeanProxy = JMX.newMXBeanProxy(mbsc, memoryMBeanName, MemoryMXBean.class, true); MemoryUsage memoryUsage = memoryMBeanProxy.getHeapMemoryUsage(); echo( "used:" + memoryUsage.getUsed() + " " + "committed:" + memoryUsage.getCommitted() + " " + "init:" + memoryUsage.getInit() + " " + "max:" + memoryUsage.getMax()); } catch(Exception e) { e.printStackTrace(); } finally { closeStream(jmxc); } } private static void echo(String msg) { System.out.println(msg); } private static void closeStream(Closeable stream) { try { if (stream != null) { stream.close(); } } catch (IOException e) { e.printStackTrace(); } } }




JMXを実行するJavaコード。 これらの困難はすべて、ファイアウォールで必要なポートとホスト名を修正するために必要です。 コードからわかるように、各サービスには2つのポートが必要です。1つはhttpアクセス用、もう1つはRMI用です。 両方のポートを明示的に指定することを含め、ポートの割り当ては明らかに異なる方法で実行できます（この場合、RMIポート= httpポート+ 1）。 次の単語をアプリケーションの起動行に追加する必要があります（jmxagent-エージェントコードを含むjarファイル）： 



 -Djmxagent.port = <SERVER_JMX_PORT> -Djmxagent.host = <SERVER_HOST> -javaagent：jmxagent.jar





 import java.io.IOException; import java.lang.management.ManagementFactory; import java.rmi.registry.LocateRegistry; import java.util.HashMap; import javax.management.MBeanServer; import javax.management.remote.JMXConnectorServer; import javax.management.remote.JMXConnectorServerFactory; import javax.management.remote.JMXServiceURL; public class JmxFirewallAgent { private JmxFirewallAgent() { } public static void premain(String agentArgs) throws IOException { try { final int rmiRegistryPort = Integer.parseInt(System.getProperty("jmxagent.port")); LocateRegistry.createRegistry(rmiRegistryPort); final int rmiServerPort = rmiRegistryPort + 1; MBeanServer mbs = ManagementFactory.getPlatformMBeanServer(); HashMap<String,Object> env = new HashMap<String,Object>(); final String hostname = System.getProperty("jmxagent.host"); JMXServiceURL url = new JMXServiceURL("service:jmx:rmi://" + hostname + ":" + rmiServerPort + "/jndi/rmi://" + hostname + ":" + rmiRegistryPort + "/jmxrmi"); JMXConnectorServer cs = JMXConnectorServerFactory.newJMXConnectorServer(url, env, mbs); cs.start(); } catch (Exception e) { e.printStackTrace(); } } }

次に、データテンプレート-データソースを指定するためのテンプレートを作成します。 私の意見では、Cactiで最も重要な設定です。 テンプレートは、RRDデータベースの構成に関連付けられ、データベース（データソースアイテム）に保存するためのパラメーターと追加設定を示します。この場合、jmxホストとjmxポートです。最初に説明したように、最初は自動的に入力され、2番目は各データソースで入力されます、つまり データソースは単一のJavaアプリケーションに対応します。







テンプレートに基づいて、データソースが作成されます。これは、グラフチャートに表示できるデータソースです。 ソースは、データを収集するために接続するデバイス、ソーステンプレート、RRDデータベースの名前と場所を示します。 私の場合、追加のパラメーターであるJMXポートも指定する必要があります。 重要な点は、データソースまたはデータテンプレートのデータ収集設定を大幅に更新すると（たとえば、データソースアイテムパラメーターの削除と追加）、RRDデータベースを再作成する必要があることです。 Cactiの外でこれを行うことができます（まだこれにアクセスしていません）、または以前のすべてのデータを失ってデータソースを再作成します。 これは明らかに、私が遭遇したCacti-RRDバンドルの最も不快な機能です。







グラフテンプレートを定義して、データの表示を開始します。 設定は、主にデータの視覚化方法を決定します。 チャートに表示するデータテンプレートのパラメーターを、どのような形式で指定する必要があります。 私の知る限り、これらの設定はすべて、RRDコマンドインターフェイスを介した直接シェルです。
