Windowsの履歴

Windows NTの開発が始まったのは20年以上前の1988年11月です。 噂では、Microsoft が Windows NTを購入/盗んだという 。 もちろん、そうではありません。 新しいオペレーティングシステムで作業するために、MicrosoftはDECに勤務し、 DEC VAX VMSオペレーティングシステムの開発者の1人であるDave Cutlerを招待しました。 1988年には、新しいオペレーティングシステム開発チームはわずか12人で、その中にはDave Cutlerとともに、DEC VAX VMSの開発に参加した元DEC従業員もいました。 そして、新しいWindows NTのすべてのコードはMicrosoftによって書かれました。

「NT」という略語は「New Technologies」としてマーケティングを表していますが、設計ドキュメントではまったく異なるものを意味します。 実際、Windows NTは新しいIntel i860プロセッサ用に開発されたもので、1988年にはまだリリースされていません。 コード名は「N10」（ NT en）でした。

最初のバージョン-Windows NT 3.1は、5年後の1993年にリリースされました。 この時点で、チームにはすでに250人の開発者がいました。

仕様→設計→実装→テスト→配信。

しかし、そのような方法論は小さなプロジェクトでのみ機能します。 現在のWindowsのような製品には、他の方法論が必要です。
  • 製品サイクルモデル

これらすべての方法には、長所と短所の両方があります。 チームの規模とコンポーネントの開発段階に応じて、異なるWindows開発チームが異なる開発方法論を適用します。

Windowsでは、一般に製品として、製品サイクルモデルが使用されます。

昔々、ソースコードブランチは1つしかなく、すべての開発者がそれに直接変更を加えました。 現在、開発チームは非常に大きいため、機能しません。 多くのブランチがサポートされており、その中にはメインブランチであるWinMainがあります。 各ラボには、変更が統合される独自のローカル開発ブランチがあります。 時間の経過とともに検証された変更は、WinMainに統合されます。

Windows 7カーネルの変更

Windows 7カーネルには、次の変更が加えられています。
  • リファクタリング

    Windowsでグラフィックサブシステムを削除できないのはなぜですか？

    技術的な観点からこの質問に対する答えは、Windowsのグラフィックスサブシステムは独立しておらず、Win32サブシステムの一部であるということです。

    Windows 7は、依存関係を解消するために多くの低レベルコンポーネントをリファクタリングしました。 これはユーザーには目立たず、新しいDllのみが表示されます。たとえば、 kernel32.dllは kernel32.dllとkernelbase.dllに分割されます。

    このパーティションにより、 MinWin （ディスクあたり20メガバイト）と呼ばれる最小のコアを割り当てることができました。
  • x86およびx64のEFIサポート（Vista SP1と同様）

    多くのメーカーは、EFIを優先してBIOSを削除しようとしています。
  • VHD（仮想ハードディスク）からの起動
  • 並列デバイスの初期化とサービスの開始

    Windowsをロードするとき、デバイスツリーの構築にはかなり時間がかかります。 PNPマネージャーは、バスドライバー（PCI、USB、FireWireなど）をポーリングして、使用しているデバイスを調べます。 そして、ほとんどの場合、プロセッサはデバイスが応答する（または応答しない）のを待ちます。 実際、バス上のデバイスを特定するには、それらをインタビューする必要があります。 もしそうであれば、彼らは答えますが、そうでなければ、あなたは待たなければならず、プロセッサはアイドル状態です。 これらのタスクを並行して実行すると、ロード時間が短縮されます。
  • スケジューラからDispatcherロックを削除し、メモリマネージャーからPFNロックを削除する

    過去数年間、プロセッサのクロック速度は向上しておらず、開発は1つのコアレベルとシステムレベル（マルチコア）の両方で並列実行命令の数を増やす方向に進んでいます。 この点で、スケーリングを改善するために多くの作業が行われました。

    カーネル内にあった2つの最もホットなロック、このDispatcherロックとPFNロックは削除されました。

    スレッドの状態を変更するときに、スケジューラがディスパッチャロックを使用しました。 このロックは削除され、「待機」スレッドのステータスはいくつかに分割されました。
    • 待機中：進行中
    • 待機中：完了
    • 保留中：キャンセル済み
    物理メモリページの属性を変更するときにPFNロックが使用されました。 マルチプロセッサシステムでは、各プロセッサがこのロックへのアクセスを要求したため、多大な時間を費やしました。
  • 256個の論理プロセッサーのサポート

    以前は、Windowsではマシンワードがアフィニティマスクとして使用されていました。 これは、無料のプロセッサを見つけるのが非常に簡単だったためです。各ビットはプロセッサを表します。 したがって、32ビットシステムでは32個の論理プロセッサ、64ビットシステムでは64個の論理プロセッサがサポートされていました。

    Windows 7では、アフィニティマスクセグメントモデルへの移行の結果、256個の論理プロセッサのサポートが可能になりました。 プロセッサはグループ/セグメントにグループ化され始めました。 各グループには最大64個のプロセッサを含めることができます。 その結果、下位互換性が得られ、古いプログラムは1つのグループのプロセッサのみを「認識」し、新しいインターフェイスを使用する新しいプログラムはシステム内のすべてのプロセッサで動作します。
  • 省エネの改善：プロセッサソケットの無効化今日、ラップトップの所有者だけでなく、データセンターの所有者にとっても省エネの深刻な問題があります。 米国では、電力の2％がコンピューターデータセンターで消費されています。 それらの多くは、低いユーザーアクティビティ（週末）の間、サーバーの一部をシャットダウンします。

    プロセッサソケット全体を切断するほうが、数個ごとに1コアよりもはるかに有益であることがわかった。 この場合、ソケットサポートインフラストラクチャ全体（メモリコントローラー）を無効にできます。


Windowsメンテナンスアップデート

以前は、更新は多くの場合累積的（累積的）でした。 これは、誤ったコードがコンポーネントの早期更新に含まれていた場合、後のバージョンにもこのコードが含まれることを意味していました。 ただし、すべてのユーザーがすべての更新を必要とするわけではなく、構成が異なります。

現在、Windowsでのリリース（RTM）後、ソースコードには2つのバージョンがあります。
  • RTM GDR（一般配布リリース）

    皆のためのそれらのいくつかの変更が含まれています。 主にセキュリティの修正。
  • RTM LDR（限定配布リリース）

    更新プログラムのインストール中に、Windows Updateクライアントは必要なブランチを選択し、そこからコードをインストールします。


セキュリティ更新プログラムを作成する

セキュリティ更新プログラムの作成作業は、脆弱性の発見から始まります。 内部のセキュリティチーム、セキュリティパートナー、開発者など、さまざまな検出方法があります。 脆弱性が発見されると、2つの並列プロセスが開始されます。
  • すべてのプラットフォームのパッチ開発
  • 「オプション」を検索

    すべてのプラットフォームで同様の脆弱性オプションの大規模な検索。 同一のコードではなく、類似のコードを検索します。
パッチの開発後、そのコードの検証が開始されます。 それらが完了すると、修正がアセンブリに統合され、アセンブリがテストのために送信されます。
  • 手動および自動コンポーネントテスト
  • ファイル形式の歪み、ネットワークコンポーネントなどの自動テスト （100万を超えるオプション）
  • 下位互換性テストを含むシステム全体のテスト
すべての品質基準を満たすパッチのみが、Windows Updateおよびダウンロードセンターでのリリースが許可されています。
