単一インターフェースの原理。 APIを介したコールトラッキングと企業CRMの統合

これはMegamindの最初のRingostatサービス投稿です。 これはHabrの「弟」であるため、デビュー資料で100500％のROIと広告予算について話すのではなく、APIを介して他のシステムとコールトラッキングサービスを統合する方法に関する技術記事から始めることにしました。



近年、ビジネス活動を最適化するように設計されたさまざまな種類のソフトウェアの数が大幅に増加しています。 これで、最も多様な情報を収集し、金融および経済指標を分析し、気分、消費者の需要と好み、および他の多くの有用な指標を測定できます。 これらはすべてクールな「モノ」ですが、これらの情報がすべて1つのウィンドウ、シンプルで直感的なインターフェイスで収集される場合に限ります。 毎朝5〜10の仮想ルームにログインして、更新されたデータと分析を表示することは、一方では便利ではなく、他方では時間がかかるためです。



サービス会社もこれを理解するようになります。 企業のコミュニケーション、多機能システム、ポータルを管理するために、単一の取引および調達プラットフォームが開発されています。 オンラインプロモーションとパフォーマンスインジケータの監視には、ダッシュボードと統計アグリゲーターが使用されます。 一方で、ほとんどのサービスは、APIを介した相互統合の可能性を残そうとしているか、より密接な相互作用または完全統合を準備しています。



API統合の仕組み



APIが何であるかを経験で説明しようと思います。 Ringostatサービスは、 WebHookを使用してCRMシステムまたは他の外部システムおよびサードパーティWebアプリケーションと統合できます。WebHookは、コール追跡サービスによって受信および記録される特定のイベントに関する通知を送信するメカニズムです。



システムには、このようなWebHookが5つあります。

  1. 通話開始;
  2. リダイレクトのすべてのステップ。
  3. 呼び出しに応答します
  4. 呼び出しが完了しました。
  5. 完了した通話評価メニュー。


このAPIを使用すると、サービスを相互に接続することも、特定のデータベースに接続することもできます（最大でテーブルまで）。 APIのおかげで、CRMシステムで次のことができます。



  • 発信者の番号で連絡先を自動的に作成します。
  • この番号の連絡先が既に存在する場合、電話をかけるときに顧客カードを自動的に表示します。
  • コールが不在の場合、マネージャのタスクを設定します。
  • 会話記録をCRMにアップロードします。
  • 通話の発信元に関するデータで顧客カードを補足します。
  • などなど。


サービスからCRMまたは他のサードパーティWebサービスへのイベントの転送を構成するには、対応する通知（CRMシステムまたは仲介サービスのアドレス）を送信するためのリクエストの送信先URL（Webアドレス）を指定する必要があります。







どの情報を記入する必要がありますか？ たとえば、このサービスでは、編集フォームは4つのフィールドと渡されたパラメーターのリストで構成されています。



  • name-CRM / Webサービスの名前、たとえば「ExampleCRM」。 これは、個人アカウントでこの統合を認識するのに便利です。
  • URL-HTTPリクエストを行うWebアドレス（たとえば、 examplecrm.com / index.php？param1 =abc¶m2= def ）：必要なURLは、システムを外部アプリケーションと統合するか、専門家に連絡するためのドキュメントに記載されています。どのセットアップ;
  • データ送信の瞬間-要求が送信される瞬間。
  • データ送信方法-CRMを統合するためのドキュメントで指定された方法に従って設定されます。






コールトラッキングの場合、情報が送信されるタイミングを予測することもできます。



  1. コールの前-今後のコールに関する予備データを取得するために使用されます。 たとえば、ユーザーがクリックしたキーワードまたは広告チャネル。
  2. 呼び出し後-呼び出し先に関するほとんどすべてのパラメーターと、呼び出し自体に関する情報が送信されます。 たとえば、発信者の番号、地域、通話時間、通話ステータス、通話録音など。
  3. 通話評価メニューの後-通話のカテゴリと値、および通話を受信したマネージャーの名前/ IDの転送に使用
  4. 受話器を取り上げるとき-責任のあるマネージャーのみにコールを通知するために使用されます。
  5. 着信転送ステップ-特定の着信分配シナリオに使用される設定済みの着信転送スキームのブロックを転送します。


送信されたパラメーター：

  • パラメーター名-ここでは、URLに追加されるパラメーター名を指定する必要があります。 システムで使用できるパラメーターの正しい名前は、関連するドキュメントに記載されているか、サポートサービスからの説明が必要です。
  • パラメータ値-特定のイベントで転送したい、システムでサポートされているデータをドロップダウンリストから選択します。


APIは、多数の異なるタイプのデータを転送できます。 具体的には、当社のサービスは最大25個の異なるパラメーターを転送できます。 統合に関する情報は、Ringostat Knowledge Baseにもあります。







接続のセキュリティを忘れないでください



コールトラッキングの場合のように、サードパーティアプリケーションがデータを送信するのは時々接続しているサービスであることを確認するために、URLは特別なキーで保護されています。 さらに保護するために、WebHookを介して通知を送信できるIPアドレスのリストを制限することができます。



CRM / Webアプリケーションへの通知の送信に失敗した場合、適切なメッセージが電子メールアドレスに送信されるため、接続の確認、再設定、またはテクニカルサポートサービスに連絡できます。



統合がなぜそれほど重要なのですか？



サービス間でデータを交換する機能は非常に重要な機能です。 IPテレフォニーシステムに特化した開発者が、市場で大きな需要がある優れた製品を作成したとします。 しかし、彼の顧客は電話を受けるだけでなく、ソースを分析したいと考えています。 コールトラッキングが必要です。 自分に合ったインターフェイスを備えたサービスを探す代わりに、APIを使用してデータ交換を簡単に構成できます。



次の段階では、特定のクライアントとの仕事の履歴、オンラインフォーム、購入、およびトランザクションを介した彼の要求と、呼び出しを組み合わせる必要があります。 もちろん、このためには、企業CRMとの統合を構成する必要があります。



おそらく、これらすべてが提供される1つのサービスを見つけることができますが、それは有益ですか？ 結果の効率と制御の観点から、異なるサースサービス間の分業はより魅力的です。 それらのそれぞれは、唯一の唯一の配信サービスを完成させることを目指しており、他の人による一部の機能の欠点を補うためではありません。 API-システムを接着できる接着剤。 それらをすべて組み合わせると、個別よりも効果的です。
