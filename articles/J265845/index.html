RESTful APIは大きな嘘です

翻訳者から：

私は最初にそのような量の記事とIT主題を翻訳しようとしました、私は喜んであなたのコメントとコメントを読みます。 記事自体に関しては、少なくとも実際にはRESTを... REST（!!!）に置き換えていますが、フレームが少し異なるため、著者には同意しません。 しかし、この記事は多くの明白なことを示しているという事実にもかかわらず、Habréについての議論に値するように思えました。



この人気のあるテクノロジーを埋める理由







RESTful APIは素晴らしいですよね？



過去10年間にウェブ開発者の概要を読んだ場合、RESTful APIが天から私たちに降りてきたある種の神の贈り物であると考えるのは許されません。 REST APIはどこでも使用され、マーケティング担当者でさえ、管理者またはスタッフ専用の資料で絶えず言及しています。



それでは、REST APIのアイデアはどれほど良いのでしょうか？ この問題を扱う前に、ルーツがどこで成長するかを見てみましょう...



RESTはどこから来たのですか？



このテクノロジーは、2000年にRoy Fieldingが「 建築スタイルとネットワークベースのソフトウェアアーキテクチャの設計 」という博士論文で詳細に説明し、紹介したときに人気を博しました。 Royは、特にHTTPのWeb開発への貢献で知られています。



では、RESTful APIとは何ですか？



RESTは、分散されたスケーラブルなWebサービスを構築するためのソフトウェアアーキテクチャのスタイルです。 Royは、リクエストに特定の意味を与えるために標準のHTTPメソッドを使用することを主張しました。



したがって、これらのHTTP要求には、RESTで異なるセマンティックロードがあります。

  • 取得/オブジェクト/リスト
  • POST /オブジェクト/リスト
  • PUT /オブジェクト/リスト


上記はクエリのいくつかのタイプですが、それらのリスト全体は、CONNECT、DELETE、GET、HEAD、OPTIONS、PATCH、POST、PUT、TRACEです。 それらの一部について聞いたことがない場合でも、クライアントアプリケーションまたはサーバーアプリケーションでサポートされることはほとんどないメソッドがあるため、問題ではありません。



Royはまた、HTTP応答コードが応答自体の意味を判断するのに役立つと主張しました。 約38の応答コードがあり、以下にそれらのリストを見ることができます。 便宜上、いくつかの名前を短縮しました。







したがって、このようなAPIの1つのトランザクションは、少なくとも次のもので構成されます。

  • GET メソッドなどのリクエストメソッド
  • リクエストパス 、例えば/オブジェクト/リスト
  • リクエスト本文 、例：フォーム
  • 応答コード 、たとえば200 OK
  • 応答本文 、たとえばJSONデータ


多くの人がこのパラダイムに積極的に反応し、HTTPを使用したWebサービスの開発にこのパラダイムを使用し始めました。 これをRESTful APIと呼びます 。



ところで、トランザクションはより複雑になる可能性があるため、この理由について説明します。 ただし、これらの問題は他の技術に関連するため、ネットワークとキャッシングに関連する複雑な要素は省略します。



RESTful APIは実際にはひどいです



RESTは、たとえばコンテンツの受信など、多くのことを行うための優れたメカニズムであり、ほぼ20年間忠実に役立ってきました。 しかし、目を開いて、RESTful APIの概念がWeb開発で存在した最悪のアイデアの1つであることを認めるときが来ました。 いいえ、私は主張しません、ロイは素晴らしい人であり、もちろん、彼は多くのクールなアイデアを持っていました...しかし、RESTful APIが彼らのリストに載っているかどうかはわかりません。



APIを構築するためのより適切な別のソリューションをすぐに検討しますが、これを行う前に、RESTful APIの5つの主要な問題を理解する必要があります。 さあ始めましょう！



問題＃1：RESTful APIについてはまだ一般的な合意がありません。



この技術が「RESTpure」ではなく「RESTful」と呼ばれる理由について考えている人はいないでしょうか？ （ 翻訳者注：純粋-明確でわかりやすい ）そして、すべての要求メソッド、応答コード、本文などが誰であるかを誰も決定できないためです。



たとえば、 200 OKコードはいつ使用する必要がありますか？ レコードの更新が成功したことを確認するために使用できますか、それとも201 Createdコードを使用する必要がありますか？ どうやら、 250の更新されたコードを使用する必要がありますが、存在しません。 それでも、 417 Expectation failedコードの意味を誰でも説明できますか？！ もちろん、ロイ以外の人。



HTTPメソッドとHTTPコードの辞書は曖昧で不完全であるため、最終的に統一された定義になりません。 私が間違っていなければ、誰も単一の一般的な秩序を見つけ、それを観察するように他の人に促したことはありません。 ある会社で200 OKが意味することは、別の会社ではまったく異なる情報を示している可能性があり、そのため議論中のテクノロジーは予測不可能です。



これが唯一の問題であれば、おそらく私は辞任し、今日までRESTful APIを書き続けていたでしょう。 しかし、私たちのリストは明らかにされているだけです...



問題＃2：REST辞書は完全にはサポートされていません



最初の問題を解決したとしても、実際の問題は次のようになります。ほとんどのクライアントおよびサーバーアプリケーションは、すべての応答コードと、実際にはHTTPメソッドを意味する動詞をサポートしていません。 たとえば、ほとんどのブラウザでは、PUTおよびDELETEのサポートが制限されています。



これにどう対処しますか？ 1つの方法は、送信するフォームに目的のメソッドを示す動詞を挿入することです。 つまり、この場合、リクエストには次のものが含まれます。



  • HTTPリクエストメソッド （POSTなど）
  • リクエストアドレス 、例えば/オブジェクト/リスト
  • 実際に意味するメソッド 、たとえばDELETE
  • リクエスト本文 、たとえばフォームからのデータ


応答コードのある状況はましです。 異なるブラウザ（およびサーバーアプリケーションも）は、これらのコードを異なる方法で理解することがよくあります。 たとえば、 307一時リダイレクトコードを受け取った場合、1つのブラウザーでユーザースクリプトがこの応答を確認し、実行前にアクションをキャンセルできる場合があります。 別のブラウザは、スクリプトが何もしないようにするだけの場合があります。 実際、処理を恐れてはいけないコードは200 OKと500 Internal server errorだけです。 残りのケースでは、回答のサポートは「かなり良い」から「ひどい」までさまざまです。 それが、しばしば私たちが実際に意味するコードで応答本体を補わなければならない理由です。



上記のすべてを調整し、インターネットに接続されているすべてを魔法のように修正できたが、RESTに対応していない場合でも、別の問題に直面します。



問題＃3：RESTディクショナリが十分ではありません



HTTPメソッドと応答コードのみで構成される辞書は、すべてのアプリケーションで必要なさまざまな情報の効率的な送信と受信には制限されすぎています。 「レンダリング完了」応答をクライアントに送信するアプリケーションを作成したとします。 残念ながら、HTTPコードを使用してこれを行うことはできません。なぜなら、 そのようなコードは存在しないため 、そして次に、 HTTPが拡張可能でないため、作成できないためです。 失望の瞬間。 答えの本文に意味を挿入する必要があると思います。



また、問題は、複数の辞書があり、そのうちの3つがあることです！ 応答コードは数値（200、201、500）であり、要求メソッド （GET、POST、PUTなど）の表現とは異なり、応答本文はJSON形式です。 RESTトランザクションを実行することは、中国に英語の手紙を送り、そこからモールス信号から応答を受け取るようなものです。 これらの困難はすべて、混乱とエラーの主な原因です。 そこで、次のグローバルな問題であるデバッグに移りました。



問題＃4：RESTful APIはデバッグが非常に難しい



REST APIを使用したことがある場合、おそらくそれをデバッグすることはほぼ不可能であることをご存じでしょう。 トランザクション中に何が起こるかを理解するには、一度に7つの場所を調べる必要があります。



  • HTTPリクエストメソッド （POSTなど）
  • リクエストアドレス 、例えば/オブジェクト/リスト
  • 実際に意味するメソッド（リクエスト本文内） 、たとえばDELETE
  • 実際には、リクエスト本文 、たとえばフォームからのデータ
  • 応答コード 、たとえば200 OK
  • （応答の本文で）意味するコード、たとえば206 Partial Content
  • 実際には、応答本文


したがって、2つの非常に限られた辞書だけでなく、エラーが存在する可能性のある7つの異なるポイントもあります。 さらに状況を悪化させる可能性があるのは、RESTテクノロジーが1つのプロトコルに完全に結び付けられており、他の通信チャネルを使用できない場合です。 実際、そうであり、これが私たちの次の大きな問題です！



問題＃5：通常、RESTful APIはHTTPプロトコルにバインドされています。



RESTful APIは、優れたコミュニケーションの基本的な法則の1つを打ち破ります。メッセージの内容は、伝送チャネルから完全に独立している必要があります 。 それらを混ぜることは混乱を完全にする方法です。



HTTPプロトコルと送信される情報が絶え間なく織り交ぜられているため、RESTful APIを他の通信チャネルに転送する可能性が完全に奪われています。 RESTfulAPIをHTTPから他のデータ転送プロトコルに移植するには、先ほど説明した7つの異なるポイントから情報を完全に解体して再構築する必要があります。



幸いなことに、RESTful APIのすべての問題を回避または最小化する優れたソリューションがあります。 会いましょう！



ステップフォワード：JSON純粋なAPI



JSONで純粋なAPIは、先ほど説明したほとんどの問題を処理します 。



  • データの送信に1つのメソッドのみを使用します。通常、Webソケットを使用する場合はHTTPおよびSENDの POST
  • 送信メカニズムとリクエストの内容は完全に独立しています。 すべてのエラー、警告、およびデータは、リクエスト本文でJSON形式で送信されます
  • 正常な送信を確認するために使用される応答コードは1つだけで、通常は200 OK
  • 送信メカニズムと応答の内容は完全に独立しています。 すべてのエラー、警告、およびデータは、応答本文でJSON形式で送信されます
  • すべてのデータが読みやすいJSON形式で1か所にあるため、デバッグがはるかに簡単
  • HTTP / S、WebSocket、XMPP、telnet、SFTP、SCP、SSHなどのあらゆる通信チャネルに簡単に移行


JSON-pure APIは、RESTful APIがブラウザーや開発者自身にとって特に友好的ではないという事実を開発者が認識した結果として登場しました。 メッセージと送信方法を分離することで、JSON-pure APIを高速で信頼性が高く、使いやすく、移植してエラーを検出できます。 今日、たとえばTwitter APIを使用する必要がある場合、マゾヒストはRESTful APIを選択します。 残りはJSON-pure API、または「Web API」とも呼ばれます。



過去10年間、JSON純粋の代わりにRESTfulを使用するように何度も依頼されました。 RESTful APIをほとんどサポートしなければならなかったのは2011年でした。 幸いなことに、バックエンドチームは、すべてのメソッドとコードをJSONに転送するだけで、JSON純粋なAPIをRESTfulと並行して実行することに同意しました。

数ヵ月後、以前RESTfulを使用していたすべての友人がJSONピュアに切り替え、それがはるかに便利であることを認識しました。
