非常に無作法かつ簡潔に言えば、これは調整された独立した方法で機能する一連のノード（ノード）です。 管理者は、各ノードで、コミッションとブロックの生成の両方について、このノードのマイナーに対してのみ有効な独自のパラメーターを設定する権利を持っています。 同時に、1つのノードで見つかったブロックは、共通の原因への貢献に比例して、プール内のすべての参加者に配布されます。 これらはすべて修正され、複雑さが軽減された独自のボールとブロックチェーンp2pool（sharechain）によって考慮されます。 統一戦線として機能する独立したノードのこのような軍事政治的な同盟。 P2Poolについて、 優しく詳細に 。



ポイントに戻りましょう。 手数料ゼロで（ノードは私たち自身のものです-なぜ私たちは自分でコミッションを設定してから取得する必要があるのですか？）トランザクションによって直接生成された見つかったブロックからの貢献に比例するコインの量を取得します。 さらに、ブロック内のコインの基本値に加えて、ブロックに含まれる手数料も分配されます。 すべてがソロマイニングのようなもので、量は少ないですが、より頻繁に行われます。



ちなみに、 BitcoinやLitecoinなどのライナーがすでに底を打っている分散化の基礎について話し、1つのプールが処理するトランザクションと処理しないトランザクションを単独で決定できる場合、かなり広い範囲で発言することができます-パニックなヒステリックから、そして少し中立で終わる。 しかし、これが「良くない」という事実は誰にとっても明らかです。 そして過剰はなく、大きなプールは「51％の攻撃」を防ぐことについて述べていますが、警戒の理由があります-アディアおじさんも多くのことを話し、手遅れになるまで信じられていました。 P2Poolがネットワークハッシュの51％、さらには70％を占める場合-関係なく、トランザクション処理の選択性に大きな影響はありません-P2Poolが独立したノードで構成され、それぞれがルールを設定することを忘れないでくださいトランザクション処理の基準。

マージされたマイニング：はじめに



マージされたマイニングの基礎の作成を始めましょう。 要するに、これにより、複数の通貨のブロックを作成するときに使用する複数の通貨を同時にマイニングし、メインと同じデータを使用して1つのメイン通貨を取得し、その他をボーナスとして取得できます。 もちろん、メイン通貨として自分でマージされた通貨をマイニングできますが、経済的な観点からは無駄です。

すべての通貨から遠く離れた場所でもこの方法でマイニングできます-メイン通貨があり、メイン通貨と一緒にマイニングできるマージされた通貨があります。 この関係は対称ではありません。 もちろん、統合された通貨がその兄の影を決して残すことはないので、彼らの将来の実現可能性が疑問視されています。



この例では、メイン通貨はvertcoinで、それに結合される通貨はMonocle（ ウェブサイトとフォーラム ）とParallaxcoin（ フォーラム ）です。

それらについてのいくつかの言葉：

  • モノクル-vertcoin開発者からのコイン。 イノベーションを導入する前に、イノベーションをテストするためのテストの場として考案されました。
  • 視差は、 他のたわごとフォークを含むいくつかの他のコインと同時にマイニングできるたわごとフォークです。 最近、私は完全に追いかけられており、純粋に学術的な価値があります。


また、共同マイニングの例は、ビットコインとネームコインのペアです（後者はBTCと並行してマイニングできます）。 Litecoinには独自の統合通貨もあります。



上記の各コインの組み立て手順と設定については説明しませんが、手順を簡単に説明します。

  1. GitHub- monocle / parallaxからソースをダウンロードします。
  2. バイナリを収集します。
  3. それらを「正しい」場所- / usr / bin /に転送します。
  4. 両方のコインのサービスディレクトリと設定ファイルを作成します-.monocle＆monocle.confとparallaxcoin＆parallaxcoin.conf 、すべて同じディレクトリ/ home / vtcにあります。 以下は、これらのファイルの内容の例です。 構成ファイルにアクセス許可を設定し、新しいファイルとディレクトリのために、ディレクトリ/ home / vtc /のコンテンツの所有者をもう一度更新します。

ノードの「パフォーマンス」の改善



正式には暗号通貨であるピアツーピアネットワークには、メインレギュレーターもレギュレーターもありません。 これによれば、彼の「正しさ」「以前の」「騒々しい」について叫ぶ「権利」があるでしょう。 はい、はい、2つのノードが同時に同じブロックを見つけた場合について正確に話します-これについて誰に早く早く誰が通知したかが勝者です。 敗者は何も残されません（そして完全に正しくなるために、 孤児となります ）。 ミリ秒ごとにカウントされる、このような戦い。 したがって、ノードに自身の成功を迅速かつ他のノードの最大数まで報告するように教え、それによりネットワーク内のノードの重みを増やします。



「スピード」



1）。 アセンブリの速度およびネットワークを介したブロックのさらなる伝送に影響する基準の1つは、そのサイズです。 ノードが生成できる標準の最大ブロックサイズは250,000バイトです。 上記のプロセスを加速するために、サイズを半分に縮小します。 理論的には、これはもちろん、ブロックにトランザクション（およびそのためのコミッション）を含めないため、利益を減らすことができますが、私の個人的な意見では、速度を上げることで得られる利益は大きくなります。 これは、3つのコインすべて（ * .confの 1つ）のクライアント構成ファイルに次の行を追加することによって行われます 

 blockmaxsize=125000


2）。 速度に影響するもう1つの重要なパラメーターは、核間のプロセスの分布です。 これについては、デーモン構成セクションで後述します。



「ボリューム」



1）。 クライアントに対して可能な最大接続数を増やし、それにより、クライアントが検出されたブロックについてより多くの他のノードに報告できるようにします。 これを行うために、クライアントの構成ファイルに次の行を再度追加します。 

 maxconnections=500


2）。 ファイアウォールの制限を削除します（多くの場合、 iptablesが機能します）。 すぐに言います-OpenVZのDebian OSでは、この機能は1つの簡単な理由で利用できません-少なくとも私のホスティング事業者とフォーラムの一部の声明によると、原則として制限はありません。 したがって、このパートは、何をどのように行うかについての明確な指示なしに、やや抽象的になります。

ファイアウォールに次のルールを設定する必要があります。 

 #  8     /24 -A INPUT -p tcp --syn --dport PORT -m connlimit --connlimit-above 8 --connlimit-mask 24 -j REJECT --reject-with tcp-reset #      IP -A INPUT -p tcp --syn --dport PORT -m connlimit --connlimit-above 2 -j REJECT --reject-with tcp-reset #  TCP        -A INPUT -m state --state NEW -m tcp -p tcp --dport PORT -j ACCEPT


これらのルールは、暗号通貨クライアントが使用するポートとp2poolスクリプトの両方に対して、ノードが機能するすべてのポートに対して設定する必要があります。 コマンドを使用して、誰がどのポートをリッスンしているかを確認できます 

 netstat -tunap

Q、s、およびEパラメーターの値はp2poolに推奨され、ノードとファーム間のより「活発な」ジョブ交換に貢献します。

  • -Q | --queue-要求されたジョブのキューの長さ。デフォルト値は1です。推奨値は0です。
  • -s | --scan-time-現在のジョブをスキャンする時間制限。デフォルトは60です。推奨値は1です。
  • -E | --expiry-作業を受け取った後に許容される、その「遅延」（古い）を考慮する上限時間。デフォルト値は120です。推奨値は1です。
