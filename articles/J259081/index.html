  • 明らかな事実：テーブルがMemory_Optimizedとして宣言されている場合、電源がオフになったときにデータが失われるという意味ではありません。データはログファイルにも書き込まれるためです。 オフにすることができ、これにより速度が大幅に向上します。

  • 新しいハッシュおよび範囲インデックスタイプ

  • InMemが役立つその他のシナリオ：複数のストリームからの同時挿入、ETLのステージングテーブル、集中的な読み取り操作。

レポートノート
簡単に言い直すことは不可能です、私は材料のスケールのアイデアを与えるためにのみリストします）：

  • よく知られている内部結合、左外部結合、および完全結合に加えて、他の要素、たとえば、左半結合があります。 このテーブル結合は、オプティマイザーによって次の形式のクエリで実行されます。
  • セットを使用した操作、たとえば、EXCEPTはテーブルの非表示結合でもあります
  • PREDICATEはスカラー演算子です。 例：CScaOp_AggFunc、CScaOp_Arithmetic、CScaOp_Assign、CScaOp_Collat​​e ...
  • PROBEは、SELECT CASE WHEN WHEN EXISTS（SELECT ..）THEN 10 ELSE 20 END ...という形式のクエリの演算子です。
  • PASS THROUGHは、SELECT CASE WHEN a = 1 THEN（SELECT TOP（1）..）ELSE 0 END ...の形式のクエリの演算子です。
  • 論理演算子のツリーが構築されます。これらは次のようなオブジェクトです。
  • LogOp_Get-テーブルを取得する
  • LogOp_Select-フィルター（「選択」、where、on、having、...）
  • LogOp_LeftSemiJoin、LogOp_RightSemiJoin-準結合
  • 単純化/置換ルールがツリーに適用されます。
  • 論理的最適化「ルールの簡素化」：空のセットの除去、冗長性の除去、述語のプッシュ、サブクエリの拡張、化合物の線形化-約150あり、レポートには良い例が含まれています
  • 例：未使用のテーブル結合は破棄され、WHEREなどに条件がある場合、LEFT JOINはINNER JOINに変換されます。
  • 「実装」ルールを適用する過程で、論理演算子は物理演算子に変換されます
  • 物理的最適化「探索ルール」：接続の可換性、参加前のグループ化、インデックス付きビューのマッチング、完全外側->左外側+左半結合（その他130ルール）
  • 物理的最適化「実装ルール」：論理演算子、コスト、ヒントに依存（たとえば、LOOP JOINまたはHASH JOINを使用）
  • テーブルの結合順序を選択するためのヒューリスティックアルゴリズムがあり、オプションが多すぎます。10個のテーブルの場合、Left Deep Treeメソッドでも3,628,800のオプションがあります。

物理化合物の主な特性：
  • ネストされたループの結合。 用途：ユニバーサル（ノンブロッキング）
  • ネストされたループの適用（ループ内の関数呼び出し）。 用途：小規模な外部ダイヤルとインデックス付き内部ダイヤル。 少量のデータ（TOP、FAST N、EXISTS）の迅速な受信
  • 結合1対多の結合：対象：結合キーインデックスと等値述語を持つ中規模および大規模なセット
  • 結合結合多対多：同じですが、tempdbを使用します（したがって、どの列が一意であるかをオプティマイザーが知っていることが重要です）
  • ハッシュマッチ。 適しているのは：インデックスなしの中規模および大規模セット。 並列実行でスケーリング




そして、いくつかの実用的な答え：
  • サブクエリまたは結合はどれくらい高速ですか？ -問題ではありません。サーバーは両方のリクエストを1つのプランに減らします（複雑な述語を使用しない場合）
  • 条件をどこに書くか、どこに？ -INNER JOINの場合、問題ではありません。
  • リクエストに接続を書き込む順序は重要ですか？ -いいえ
  • 結合後のグループ化とグループ化された値のどちらが良いですか？ -オプティマイザーは、グループ化自体を結合にプッシュします。
