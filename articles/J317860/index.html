ストーブを離れずにORMを調理する





この記事は、 過激派のバイクの設計を求めるものではありません。 投稿の目的はメカニズムをよく理解することです。多くの場合、最初から作成する必要があります。 これは、ORMなどの不安定なトピックの場合に特に当てはまります。



なぜこれだけなのですか？



MS Entity Framework、NHibernateなどの工業製品は複雑で、優れた機能を備えており、実際にはそれ自体が別個のものです。 多くの場合、このようなORMから目的の動作を取得するには、そのようなシステムの複雑な操作に精通している別の人が必要です。これはチーム開発には適していません。



投稿の主な理論的情報源は、 Martin Fowlerの著書 『Patterns of Enterprise Application Architecture（P of EAA）』です。



実際にORM、DDD、TTD、 交通警察とは何かについて多くの情報があります-私はここで止まらないようにします、私の目標は実用的です。 記事の2つの部分が想定されています。最後に、gitハブの完全なソースコードへのリンクがあります。



何を取得したいですか？



すぐに用語を定義します：追跡とは、保存されたデータをさらに同期するために、単一のトランザクション内でビジネスオブジェクトの変更を追跡することを意味します

ORMの主な種類は何ですか？



  • .NETの世界でトラッキングを使用するためのORMには、変更を追跡する場合と変更を追跡しない場合の2種類があります。
  • SQLクエリを生成するアプローチについて：クエリを明示的に使用し、オブジェクトモデルのクエリジェネレーターに基づきます。


たとえば、クエリを使用せず、クエリを明示的に使用する有名なORMの中で、最も顕著な例はStack OverflowのDapper ORMです。 このようなORMの主な機能は、リレーショナルデータベースモデルをオブジェクトモデルにマップすることです。一方、クライアントはデータベースへのクエリの外観を明確に決定します。



オブジェクトモデルの追跡およびクエリジェネレーターの使用におけるMS Entity FrameworkおよびNHibernateの基本概念。 これらのアプローチの長所と短所は考慮しません。 すべてのヨーグルトは同様に有用であり、真実はアプローチを組み合わせることにあります。



顧客（つまり、私）は、追跡を使用して（オフにする機能を備えた）、オブジェクトモデルのクエリジェネレーターに基づいてORMを作成したいと考えていました。 Linq2Sql、LinqToEntitiesを使用せずに、ラムダ-C＃言語式、ゼロからSQLクエリを生成します（はい、ハードコアのみ！）。



すぐに使用できるMS Entity Frameworkには、データのバッチ更新と削除に関する厄介な問題があります。最初にデータベースからすべてのオブジェクトを取得し、次にサイクルで更新/削除してから、データベースに変更を適用する必要があります。 その結果、データベースへの呼び出しが必要以上に多くなります。 ここで問題を説明します 。 この問題は、記事の後半で解決します。

料理を始める



前に説明したことはすべて、公開部分に関するものでした。 次に、内部の部分を検討します。 さらに開発するには、追跡するオブジェクトの構造を決定する必要があります。



ビジネスオブジェクトと追跡オブジェクトを混同しないでください。



  • ビジネスオブジェクトには独自のタイプがありますが、トラッキングオブジェクトは多くのビジネスオブジェクトの大量操作に適している必要があります。つまり、特定のタイプに依存するべきではありません。
  • 追跡オブジェクトは、特定のビジネストランザクション内に存在するエンティティです。多くのビジネスオブジェクトの中で、その一意性はこのトランザクションのフレームワーク内で決定する必要があります


そのようなオブジェクトには、プロパティが必要であることになります。



  • タイプ内で一意
  • 変わらない


本質的に、追跡用のオブジェクトは、ビジネスオブジェクトを格納するためのコンテナです。 前述したように、すべてのクライアントビジネスオブジェクトはEntityBaseスーパークラスの祖先である必要があり、それらのオブジェクト識別子を再定義する必要があります。 識別子は、型、つまりデータベース内のテーブル内で一意性を提供します。

ビジネスオブジェクトの追跡



追跡用のオブジェクトの登録は、リポジトリからこれらのオブジェクトを受信する段階で行われます。



データベースからビジネスオブジェクトを受け取ったら、追跡用のオブジェクトとして登録する必要があります。

このようなオブジェクトには、データベースからの受信直後と、変更をコミットする前の2つの状態があります。



最初のオブジェクトは「クリーン」、2番目のオブジェクトは「ダーティ」オブジェクトと呼ばれます。



例 
  var uow = new UnitOfWork(); using (Session.Create(uow)) { var profileRepo = new ProfileRepository(); // ""      , //  "" var profiles = profileRepo.Get(x=>x.Info = " "); // ""  foreach (var profile in profiles) { profile.Info = " "; } //  uow.Commit(); }




重要な点は、「クリーンな」オブジェクトを保持するには、コピー操作が必要であり、パフォーマンスに悪影響を及ぼす可能性があることです。



一般に、追跡オブジェクトは操作の種類ごとに登録する必要があるため、更新、削除、挿入の操作用のオブジェクトが必要です。



値による比較操作が必要な、実際に変更されたオブジェクトのみを登録する必要があることを考慮する必要があります（オーバーライドされたEqualsメソッドを使用したEntityStruct構造の実装は上記で提供されています）。 最終的に、比較操作はハッシュの比較に限定されます。

パブリッククラスがどのように見えるかを見てみましょう。
