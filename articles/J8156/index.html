ボックスバックアップ-ホットバックアップ。

バックアップの問題が関連性を失うことはありません。 これは、情報構造を想像することが不可能なものです。 今日は、オンラインバックアップ方法について説明します。





彼らはこれをどのように「通常」行うのでしょうか？

既存の情報構造のバックアップの作成について話すと、cronのスクリプト、増分アーカイブ、巨大なディスクアレイを備えた強力なサーバー、ストリーマー、外部メディアに書き込むためのその他のデバイスがすぐに思い浮かびます。 これらすべてがなければ、完全なバックアップシステムを想像することは非常に困難ですが、多くの場合、何とか何らかの形でバックアップなしで機能します。 誰もがこれを知っているので、「強力な後部」を持つ必要性については説明しません。



上記のソフトウェアおよびハードウェアコンポーネントは、何らかの理由でファイルが失われた場合にファイルを復元する機能を提供するように設計されています。 原則として、増分バックアップの必要性は、絶えず変化し、バックアップが必要な膨大な量の情報によって説明されます。



冗長性スキームの1つを基本とし、意見に必要な冗長性を追加し、結果のソリューションを運用します。 以下に示す回路は最悪のオプションではありません。



•月に一度、それを必要とするすべてのファイルの完全なバックアップが作成されます。

•毎日、現在の日に変更されたデータがコピーされます。

•毎週、その週に変更されたすべてのデータがコピーされます。

•2週間に1回、この期間中に変更されたすべてのファイルがコピーされるか、アーカイブが2週間でマージされます。

•新しい月が来るまで、パターンは再び繰り返されます。



好みや会社の基準に応じて、週に2回または一度に1つずつ保存できます。 ファイルを復元する手順は、ファイルをアーカイブに記録する手順の逆です。



•最後の毎月のコピーを復元します。

•毎週2つのコピーを復元します。

•毎週のコピーを復元します。

•その日のコピーを逆の順序で復元します。



説明したような回路は、多くの企業で使用されています。 このスキームを使用すると、最後の「毎日」のコピーの作成時にすべてのファイルが確実に復元されます。 十分な量のデータ（もちろん、珍しいことではありません）を使用すると、外部ストレージメディアを使用し、ハードドライブに少なくとも1つのフルコピーを作成するために必要なスペースを確保する必要があります。 （条件は必須ではありませんが、生活を大幅に簡素化します。）



このスキームおよび類似のスキームは、多くの企業で非常に長い間機能しています。 ただし、それらには多くの重大な欠点があります。 その中には：



•「外部」ユーザーのバックアップ時の既知の問題。 ラップトップユーザーが毎日オフィスでコンピューターの電源を入れたままにすることはできません。

•バックアップのセキュリティ問題。

•単一の失われたファイルと大きなアレイの両方を回復するための比較的長い時間。

•アーカイブの操作の不便さ-バックアップデータベースの検索時間が非常に長くなる可能性があります。

•アーカイブを操作するには、システム管理者の支援が必要です。



すべての既存の問題がリストされているわけではなく、これらは私が直面しなければならなかった問題のみです。 おそらく、使用しているソリューションにはこれらの欠点がありません。 バックアップソフトウェアパッケージが購入され、すべての要件を満たしている可能性があります。 おそらく、あなたは物事の現在の順序に満足しています。 しかし、バックアップを作成する別の原則に注意を払うように、まだ説得しようとしています。



オンラインバックアップBoxバックアップ。

システムはオンラインバックアップを指します。 なぜそのような名前を持っているのですか？



上記のスキームとは対照的に、「ホット」バックアップが行われます...常に。 故障したサーバーの電源をシャットダウンせずに交換するようなものです。 （私はあなたがそのようなコンピューターに出くわしたか、少なくとも聞いたことがあると思います。）ユーザーファイルのバックアップコピーの作成は、ユーザーファイルが変更されるとインタラクティブに行われます。 作業のスキームは単純です：



•最初の起動時に、システムは指定されたディレクトリをスキャンし、アーカイブに転送します。 次に、ファイルシステムで何が起こっているかを監視します。

•変更されたファイルを検出すると、それらをアーカイブに転送します。



つまり、最新のファイルバージョンは常にバックアップ（！）にあります。 ソースシステムでデータが失われた場合、安全かつ迅速に完全に復元できます。 魅力的ではないですか？

•Cygwin（Windows）（クライアントのみ）;

このホットバックアップシステムに初めて慣れるときに生じる、ごく自然な質問です。 私はそれらの最も明白なものに答えようとします。



•ユーザーがファイルを削除した場合、バックアップはどうなりますか？

•ファイルの永続的なコピーは、膨大なネットワークトラフィックとバックアップサーバーの重い負荷を意味します。 この問題を解決するには？

•コピーのセキュリティとは何ですか？

•また、ファイルが多数ある場合、プログラムは変更をチェックするためにいくつのリソースを必要としますか？



取り外し手順。

システムから削除されたファイルは、実際にはどこにも消えません。 （もちろん、クライアントコンピューターについてではありません。）バックアップでは、それらは削除済みとしてのみマークされます。 ユーザー（はい、システム管理者ではなく、後で詳しく説明します）は、いつでも落ち着いて、誤って削除したファイルを返すようにシステムに要求できます。 ラベル付きデータは、バックアップ用に予約された許容量が、マークされたファイルを消去するか、新しいファイルをコピーする場所がないというマークに近づくと、アーカイブから消えます。



リソースを要求します。

予想されるように、大きなネットワーク負荷は、システムが一度に大量のデータを転送する必要がある最初のバックアップを作成するときにのみ実際に観察できます。 ただし、Box Backupクライアントのプロセスの優先度（デフォルト）はかなり低く、コンピューターの全体的な動作を妨げないように設計されています。 バックグラウンドで機能し、システム全体の「思慮深さ」に不快感を与えません。 その後、システムは新しいファイルまたは変更されたファイルのみを転送する必要があり、これはそれほどトラフィックではありません。



バックアップセキュリティ。

セキュリティは多くの注目を集めています。



•バックアップサーバー上のクライアントの承認。

•データ送信のセキュリティ。

•データの暗号化。

•データ構造の保護。

•データストレージのセキュリティ。



SSLは、Box Backupシステムのサーバー側でクライアントを認証するために使用されます。 データのバックアップが必要な各コンピューターで、サーバーで署名されたSSL証明書が作成されます。



TLS暗号化は、安全な通信環境を提供するために使用されます。 クライアントとサーバー間の操作は、暗号化されたチャネルを介して行われます。 これにより、データが傍受される恐れがないため、安全でない環境でもバックアップシステムを使用できます。



暗号化により、バックアップファイルのセキュリティが確保されます。 Box Backupシステムの各クライアントは、構成中に一意のキーを作成します。 このキーを使用すると、送信されるすべてのファイルが暗号化されます。 キーを使用してのみ、ファイルを表示または復元するためにアーカイブにアクセスできます。



さらに、バックアップサーバーでは、ファイルは名前ではなく一意の番号で保存されます。 ファイルの場所がわからない-それが何であるかを推測することはできません。 しかし、あなたの目の前にどんな種類のファイルがあるかを知っていても、何も達成できません。 結局のところ、それらは暗号化されており、アクセスにはキーが必要です。 パスワードは必要ありません。



Box Backupのサーバー側をインストールします。

例としてgentooシステムを使用したBox Backupのインストールを見ていきます。 他のシステムへのインストールは、いくつかの詳細のみが異なりますが、これについては後で説明します。



そのため、最初にシステムのサーバー側をインストールします。 ebuild Box Backupは標準ポートツリーにあるため、gentooシステムで標準パッケージのインストール手順を使用するだけです。



#emerge -av boxbackup



デフォルトでは、システムの古いバージョン（0.09）をインストールするように求められることに注意してください。 portsツリーをたどって目的のブランチに移動し、多くの修正が加えられた新しいバージョンをインストールすることをお勧めします。



#cd / usr / portage / app-backup / boxbackup

#emerge -av boxbackup-0.10.ebuild



もちろん、/ etc / portage / package.keywordsを修正することはできますが、1つのパッケージのためにこれを行う理由はありません（システムはその操作に他のパッケージを必要としません）。



パッケージのアセンブリが完了したら、構成を続行できます。 ここで1つの点に注意することが重要です。 システムにバックアップ用のRAIDコントローラがなく、複数のディスクを使用する場合、Box Backupは、ディスクにエラーが発生した場合に冗長性を提供する複数のディレクトリを作成することをお勧めします。



RAIDがインストールされている場合、この構造の使用を安全に拒否できます。



Box Backupの「raid」を使用するには、次のコマンドを実行します。



＃raidfile-config / etc / boxbackup 2048 /raid/0.0 /raid/0.1 /raid/0.2



または、異なる種類のコマンドで使用することを拒否します。



＃raidfile-config / etc / boxbackup 2048 /raid/0.0



このコマンドのオプションのリスト：

•/ etc / boxbackup-構成ファイルが作成されるディレクトリ。 （システムをgentoo linuxにインストールしない場合、デフォルトで/ etc / boxディレクトリを使用します）;

•2048-ファイルブロックサイズ。

•/raid/0.0およびその他は、バックアップが直接保存されるディレクトリです。 これらのディレクトリは、ユーザーが作成する必要があります。 それらは自動的に作成されません。



構成ファイルの準備ができたら、Box Backupを実行するユーザーを作成します。



#useradd _bbstored



メインシステム構成ファイルを作成します。



＃bbstored-config / etc / boxbackup hostname _bbstored



どこで：

•/ etc / boxbackup-設定ファイルのディレクトリ。

•ホスト名-コンピューター名。

•_bbstored-システムを実行しているユーザーの名前。



別のシステムでパッケージを自分でビルドした場合は、構成ファイルに/ etc / boxディレクトリを使用します。



このコマンドを実行すると、Box Backupシステムのメイン構成ファイル/etc/boxbackup/bbstored.confが作成され、そこにメインパラメーターが書き込まれます。



ユーザーに_bbstoredディレクトリ権限を割り当てます



#chown -R _bbstored / etc / boxbackup / bbstored

#chmod -R go-rwx / etc / boxbackup / bbstored



これで、Box Backupシステムのサーバー側の構成が完了しました。 あとは、システム用のSSL証明書を作成して、顧客アカウントを追加するだけです。 これは、バックアップが実行されるマシンにクライアント部分をインストールした後に行います。



LinuxでのBox Backupのクライアント部分のインストール。

gentoo linuxについて、およびコマンドの前の出力で話しているので



#emerge -av boxbackup-0.10.ebuild



おそらく-client-onlyフラグに気付いたので、それを使用します。 バックアップが必要なシステムで次のコマンドを実行します。



#cd / usr / portage / app-backup / boxbackup

これにより、システムにシステムクライアントがインストールされます。



クライアントがインストールされたら、サーバーで動作するように構成します。 ここでは2つのオプションが可能です。



•lazy-このモードでは、bbackupdバックアップデーモンはファイルの変更を常に監視し、変更されたファイルが閉じられるとすぐにバックアップに転送します。

•スナップショット-このモードの操作は、その名前で理解できます。 設定した期間に一度、bbackupctl syncコマンドが起動され、その後、bbackupdデーモンが構成ファイルで指定されたすべてのディレクトリのスキャンを開始します。 最後の変更の日付に矛盾がある場合、ファイルはバックアップサーバーに転送されます。



どのモードを優先するかは、バックアップする必要があるファイルの数、コンピューターの負荷、およびネットワークトラフィックに直接依存します。 依存関係は非常に明白だと思います。 バックアップサーバーがあるローカルネットワークの外部で動作することが多いラップトップにBox Backupシステムをインストールする場合は、おそらくスナップショットモードを選択する必要があります。



もちろん、外部からアクセスできるようにBox Backupサーバーをインストールする必要はありません。 ただし、この場合、バックアップ中にコンピューターによって作成されるトラフィックの量を考慮する必要があります。



コマンドを実行します。これにより、リモートサーバーで認証用のメイン構成ファイル、システムキー、およびSSL証明書が作成されます。



＃bbackupd-config / etc / boxbackup lazy account_name hostname / var / bbackupd / home



どこで：

•/ etc / boxbackup-構成ファイルの保存場所。

•lazy-バ​​ックアップ方法。

•99-バックアップサーバー上のアカウントの名前（通常、コンピューターのIPアドレスの最後の数字を使用します）。

•ホスト名-バックアップサーバーのネットワーク名（このパラメーターで誤ってコンピューターの名前を指定しないでください！）;

•/ var / bbackupd-Box Backupシステムの一時ファイルが保存される場所（そこに十分な空き容量があることを確認してください）;

/ home-バックアップのパス（フォルダーは構成ファイルで簡単に構成されます。これについては後ほど説明します）。



ここで、バックアップするフォルダとそれらのルールを指定するだけです。 Box Backupクライアントのメイン構成ファイルは/ etc / boxbackupディレクトリにあり、bbackupd.confと呼ばれます。



このファイルに設定されているパラメーターには、必要かつ非常に透過的な説明が含まれているため、バックアップのパスの説明のみに焦点を当てます。

Box Backupシステムのクライアント側とサーバー側を構成したので、承認のためにSSL証明書を処理します。



サーバー側のSSL証明書。

基本的な証明書構造を作成するには、次のコマンドを実行します。



＃bbstored-certs ca init



このコマンドの実行中にエラーを受け取った場合は、OpenSSLパッケージが最初にインストールされていることを確認してください。



証明書を生成した後、署名します。



＃bbstored-certs ca sign-server hostname-csr.pem



これで、Box Backupサーバー側の証明書の準備が完了しました。



クライアント側のSSL証明書。

既にクライアントSSL証明書を持っているので、サーバーで署名するだけで、同時にサーバーからクライアントに証明書をコピーする必要があります。 このためには、ファイル/etc/boxbackup/bbackupd/account_name-csr.pemをサーバーに転送（scp、samba、ftpまたは別の方法）する必要があります。



クライアント証明書をサーバーのフォルダー/ etc / boxbackup / bbackupd /に配置するのが最適です



その後、サーバーでコマンドを実行します。



＃bbstored-certs / ca sign /etc/boxbackup/bbackupd/account_name-csr.pem



Yesリクエストに答えた後、証明書に署名し、2つのファイルをクライアントにコピーする必要があるという指示を確認します。



/ca/clients/account_name-cert.pem

/ca/roots/serverCA.pem



クライアント側では、これらのファイルを/ etc / boxbackup / bbackupdに配置します。



今では、予約のためにクライアントのアカウントを作成するだけです。



バックアップシステムでアカウントを作成します。

アカウントを作成するには、次のコマンドを使用します。



#bbstoreaccounts create account_name 0 10000M 15000M



どこで：

•account_name-数値形式のアカウント名（たとえば、前述したように、クライアントのIPアドレスの最後の数字）。

•10000M-ソフトバックアップサイズの制限。 この場合、ユーザーごとに10ギガバイトを割り当てました（10Gを指定することもできます）。

•15000M-バックアップのサイズの厳しい制限。



その後、Box Backupシステムのクライアントとサーバーのサービスを安全に開始できます。 バックアップシステムの準備ができました！

1. bbackupquery.



このユーティリティは、リモートシェルのように機能します。動作するbbackupdデーモンは必要なく、入力したコマンドを使用してバックアップサーバーと通信できます。サーバーへの接続は、同じSSL証明書を使用して確立され、TLSチャネル暗号化を使用して引き続き通過します。



図1は、helpコマンドの出力を示しています。ご覧のように、かなり多数のコマンドが使用できますが、その中には比較もあり、ローカルハードドライブ上のファイルとバックアップ内のファイルのバージョンを比較できます。 bbackupqueryでの作業は、コンソールクライアントを介したftpサーバーとの通信に多少似ているため、詳しくは説明しません。

このサイトには、少なくともWindows 2000およびWindows XPファミリのシステムで動作するWindowsネイティブクライアントがあります。アーカイブには、ディレクトリc：\ Program Files \ Box Backup \にコピーすることが推奨されるすべての必要なファイルがあります。



残っているのは、クライアント証明書を生成し、サーバーで署名し、署名されたサーバー証明書を転送することです。そして、ここでいくつかの困難が生じるかもしれません。実際、証明書の生成を担当するスクリプトはperl言語で書かれており、それがWindowsシステムにあるという保証はありません。長い苦痛によって望ましい結果を達成することは可能ですが、はるかに高速で便利な方法があります。
