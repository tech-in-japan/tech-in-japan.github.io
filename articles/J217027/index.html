Oracle Database Appliance-使用後の概要





最初に、ODAについて少し説明します。 これはエンジニアリングシステムであり、ストレージシステム、サーバー、およびソフトウェアの既製のスタックであり、最大のパフォーマンスを実現するために互いに適合しています。 他のメーカーには類似物がありません。 スタック全体は、パフォーマンスのボトルネックを特定するために、機器とソフトウェアのメーカーによってテストされました。 その結果、ODEにはさまざまな構成のファンはなく、ディスクで追加のシェルフを追加するだけで済みます。 システムには標準量のメモリ、標準量のディスク、標準プロセッサがあります-特別な追加や設定オプションはありません。



レビューでは、サイズが4ユニットで、それぞれに次の2つのサーバーが含まれる第1世代のODEを使用しました。

96 Gb RAM

1x 4ポート1GbE NIC



1x2ポート10GbE NIC



ストレージシステム：

展開



私はODEを数回（展示会、パートナー、およびテスト用に）展開したので、これはLinuxおよびデータベースのインストールの分野で最低限の知識を必要とするかなり簡単な手順であると言えます。



新しい機器が届き、開梱し、ケーブルを接続してラックに挿入すると、すぐにシステムの電源を入れて展開できます。 Odaには、インストール済みのOSと購入時の最新のパッチが付属しています。あとは、インターネットから数GBのバンドルをダウンロードするだけです。 エンドユーザーGI / RDBMSクローンファイルと呼ばれ、Oda用にコンパイルされた最新のパッチデータベースが含まれています。 次に、それをエンジニアリングシステムのノードの1つにアップロードし、最も簡単なウィザードを実行する必要があります。1〜1.5時間で、 Oracleデータベースが構成され、RAC、RAC 1ノードまたは失敗のいずれかのモードで動作する準備ができた2つのサーバー+ストレージができます以上。 1Cを使用する場合、またはデータベース自体の非標準設定が必要な場合は、dbcaコンソールを使用して作成する必要があります。



いくつかのサーバー、他のストレージシステム、3番目のOS、コンポーネントの4番目のデータベースなど、さまざまなメーカーのものを使用できます。 クラスター形式の2台のサーバーでOSとベースを構成し、ストレージシステムのセットアップに時間を費やし、システム全体のペアリングとテストに多くの時間を費やす必要があります。 ここでは、時間の合計無駄は3時間であり、その結果、データベースマシンは完全に動作します。 また、構成に必要な人数も重要です。 通常の場合、テストには鉄の専門家+ DBの専門家+が必要です。ここでは1人と数時間です。 まあ、ラックに入れる最大のアシスタント、重量はかなり大きいです:)。

免許



Oracleデータベースでは、すべてのライセンスはコアを経由します。コアは、奇妙なことに、プロセッサライセンスと呼ばれます。 4コアの2つのプロセッサ（8コアのみ）を搭載した最新のサーバーさえも使用しない場合、ライセンスポリシーによると、これらは8つのプロセッサライセンスであり、それぞれにコストがかかります。 ライセンスを節約するために仮想マシンを実行する場合、非常に限られた数の製品がこれに適しています（詳細はこちら ）



ODAには大きなユニークな利点が1つあります。これは、必要なカーネルに対してのみデータベースのライセンスをすぐに使用できることを意味し、これはキャパシティオンデマンドと呼ばれます。 これにより、2つのコアからライセンスを開始し、16に拡大できます。このアプローチの主なポイントは、ライセンスを節約することです。



ライセンスの数を増やして生産性を上げる必要がある場合は、追加のライセンスを購入し、テクニカルサポートサイトでキーを生成し、システムにスクリプトを「フィード」して、追加されたカーネルのハードウェアパフォーマンスをすぐに受け取る必要があります。 別のシステムでは、たとえば、データベースを新しい、より強力なシステムに停止、バックアップ、転送するなどの追加作業が必要になります。

展開オプション



バージョン2.5から、ODAを仮想構成と「鉄」構成の2つの方法で展開できるようになりました。



最初のオプション-「Iron」は、Linuxとデータベースをサーバーに直接インストールします。最大負荷がない場合、リソースの使用という点ではあまり有益ではありません。 その結果、データベースで2つのコアを使用する場合、残りはビジーではないため、アイドル状態になります。



2番目のオプションは仮想です。 これを行うには、 Oracle VMハイパーバイザーを両方のサーバーにインストールし、ODE（つまりOS + DB）の形式でテンプレートをそれらにロードします。これにより、デプロイ後にストレージシステムの使用が開始されます。 仮想システムの展開は、実質的に鉄の展開と変わりませんが、その結果、残りのリソースを他の仮想マシンに使用できます。 しかし、ここにはマイナスがあります。つまり、特定の形式のテンプレートをtgz拡張子で排他的にサポートします。 興味のある方は、 こちらからダウンロードできます 。



試してみると、独自のテンプレートを作成できます。 また、軟膏の小さなハエにより、サーバーが使用できるディスク容量の制限が追加されます。 仮想マシンはストレージシステムではなく内部ディスクを使用します。











このエンジニアリングソリューションについて、公式および非公式の多くのレビューを読んだ後、 彼らが時々語らないことに焦点を当てる価値があると判断しました。



1.バンドルは個別にダウンロードする必要があります



2.オラクルは、受領時にソフトウェアの最新バージョンがあることを常に確認しようとしますが、配信中に新しいバージョンがリリースされる場合があります。



3.鉄の異なる構成の欠如。



4.仮想化に組み込みストレージを使用できない。



5.最も便利な仮想マシン管理ではない



6.現在のデータベースの最新バージョンは11.2.0.4.0です



ODEの使用経験後の個人的および主観的な結論：





1.展開時間の節約。 簡単で迅速なインストール。 通常、クラスターのセットアップには多くの時間がかかります。 ここでは、15ステップで小さなウィザードを起動した後、クラスターの準備ができています。



2.システム全体を保守するには、1人が必要です。鉄、DB、およびストレージの個別の管理者は必要ありません。



3.インストール用のスペースを節約します。 Odaは、ラックで4ユニットしか占有しません。 これは、データセンターなどでの配置に非常に良い影響を与える可能性があります。



4.すべての機器が事前にテストされているため、1つのベンダーのすべてがすべての人に馴染みのないものであり、不適合の形で不愉快な驚きがあるためです。



5.機器の陳腐化はありません。 古いODEであっても、たとえそれが販売されなくなったとしても、最新バージョンより劣らない機能を備えたアップデートがまだあります。



6.すぐに成長する可能性があるライセンスの節約（上記のとおり）。



7.機器自体の非常に便利な監視と制御。 ODEは、全体の再構成、再起動、さらには再インストールが非常に簡単です。



8.フォールトトレランス、tk。 すべてが二重に、さらには三重に複製されます。



9. ODEは、使用可能な最小のExadataのコアに近い十分に多くのコアに拡張できますが、これははるかに深刻なレベルの決定であり、トピックはまったく異なる記事です。
