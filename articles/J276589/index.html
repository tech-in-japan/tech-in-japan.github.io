IL2CPPとは何ですか？



IL2CPPテクノロジーは、2つの部分で構成されています。



•Ahead-of-time（AOT）コンパイラ。

•仮想マシンをサポートする実行可能ライブラリ。



AOTコンパイラは、.NETコンパイラからの中間言語（IL）をC ++ソースコードに変換します。 実行可能ライブラリは、サービスと抽象化（ガベージコレクターなど）、ストリームとファイルへのクロスプラットフォームアクセス、および内部呼び出し（マネージデータ構造を直接変更するアンマネージコード）を実装する方法を提供します。



AOTコンパイラ



IL2CPP AOTコンパイラは、il2cpp.exeと呼ばれます。 WindowsではEditor \ Data \ il2cppディレクトリに、OS XではUnityインストール場所のContents / Frameworks / il2cpp / buildディレクトリにあります。 il2cpp.exeユーティリティは完全にC＃で記述され、.NETおよびMonoコンパイラを使用してコンパイルされます。



このユーティリティは、Unityに同梱されているMonoコンパイラによってコンパイルされたマネージアセンブリを受け取り、特定のプラットフォームのC ++コンパイラに渡すC ++コードを生成します。

実行可能ライブラリ



IL2CPPテクノロジーの2番目の部分は、仮想マシンをサポートする実行可能ライブラリです。 このライブラリはほぼ完全にC ++で記述され（プラットフォームコードはいくらかありますが、私たちの間にはそのままにしておきます）、libil2cppと呼ばれます。 プレーヤーの実行可能ファイルに関連付けられた静的ライブラリの形式で提供され、そのシンプルさと移植性はIL2CPPテクノロジーの主な利点の一部です。



Unityに付属のヘッダーファイルを見ると、libil2cppコードを整理するより明確なアイデアを得ることができます（エディターの\ Data \ PlaybackEngines \ webglsupport \ BuildTools \ Libraries \ libil2cpp \ includeディレクトリ、またはContents / Frameworks /ディレクトリにあります） il2cpp / libil2cpp-OS Xで）。 たとえば、il2cpp.exeによって生成されたC ++コードとlibil2cpp環境の間のインターフェイスは、codegen / il2cpp-codegen.hヘッダーファイルにあります。



環境の重要な要素の1つは、ガベージコレクターです。 Unity 5には、Boehm-Demers-Weiserガベージコレクターであるlibgcが付属しています。 ただし、libil2cppは他のビルダーもサポートしています。 たとえば、CoreCLRスイートのオープンソースコンパイラであるMicrosoft GCとの統合を検討しています。 次の記事のいずれかで、それについてさらに詳しく説明します。



il2cpp.exeの実行方法



例を見てみましょう。 これを行うには、Windows上のUnity 5.0.1で新しい空のプロジェクトを作成します。 少なくとも1つのユーザースクリプトがあるように、メインカメラに単純なMonoBehaviourコンポーネントを追加します。







WebGLプラットフォーム用にビルドする場合、Process Explorerを使用して、il2cpp.exeを実行するコマンドを確認できます。



"C:\Program Files\Unity\Editor\Data\MonoBleedingEdge\bin\mono.exe" "C:\Program Files\Unity\Editor\Data\il2cpp/il2cpp.exe" --copy-level=None --enable-generic-sharing --enable-unity-event-support --output-format=Compact --extra-types.file="C:\Program Files\Unity\Editor\Data\il2cpp\il2cpp_default_extra_types.txt" "C:\Users\Josh Peterson\Documents\IL2CPP Blog Example\Temp\StagingArea\Data\Managed\Assembly-CSharp.dll" "C:\Users\Josh Peterson\Documents\IL2CPP Blog Example\Temp\StagingArea\Data\Managed\UnityEngine.UI.dll" "C:\Users\Josh Peterson\Documents\IL2CPP Blog Example\Temp\StagingArea\Data\il2cppOutput"









これはかなり長い行なので、部分的に解析してみましょう。 Unityは最初にこの実行可能ファイルを実行します。



「C：\ Program Files \ Unity \ Editor \ Data \ MonoBleedingEdge \ bin \ mono.exe」



次の引数は、il2cpp.exeユーティリティ自体です。



「C：\ Program Files \ Unity \ Editor \ Data \ il2cpp / il2cpp.exe」



残りの引数は、mono.exeではなくil2cpp.exeに渡されます。 それらも見てみましょう。 まず、Unityは5つのil2cpp.exeフラグを渡します。



•-copy-level =なし

il2cpp.exeが、生成されたC ++コードファイルの特別なコピーを実行しないことを示します。



•-enable-generic-sharing

コードとバイナリファイルのサイズを縮小します。 そのうち、IL2CPPでジェネリックメソッドを実行するプロセスを公開します。



•-enable-unity-event-support

リフレクションによってトリガーされるUnityイベントの正しいコード生成を提供します。



•-output-format =コンパクト

型名とメソッドの文字数が少ない形式でC ++コードを生成します。 中間言語の名前が保存されていない場合、そのようなコードはデバッグが難しくなりますが、C ++コンパイラが解析するコードが少なくて済むため、コンパイルが高速になります。



•-extra-types.file = "C：\ Program Files \ Unity \ Editor \ Data \ il2cpp \ il2cpp_default_extra_types.txt"

追加のタイプに標準の空のファイルを使用します。 Unityプロジェクトに追加できるので、il2cpp.exeは実行時に作成されるジェネリック型またはインデックス付きの型を認識しますが、ILコードでは指定されません。



il2cpp.exeのこの一連のコマンドライン引数はまだ不安定であり、将来のバージョンで変更される可能性が高いことに注意してください。



したがって、コマンドラインには2つのファイルと1つのディレクトリがあります。



•「C：\ Users \ Josh Peterson \ Documents \ IL2CPP Blog Example \ Temp \ StagingArea \ Data \ Managed \ Assembly-CSharp.dll」

•「C：\ Users \ Josh Peterson \ Documents \ IL2CPP Blog Example \ Temp \ StagingArea \ Data \ Managed \ UnityEngine.UI.dll」

•「C：\ Users \ Josh Peterson \ Documents \ IL2CPP Blog Example \ Temp \ StagingArea \ Data \ il2cppOutput」



il2cpp.exeユーティリティは、変換されるすべてのILアセンブリのリストを受け入れます。 私たちの場合、これには単純なMonoBehaviourスクリプト、Assembly-CSharp.dllおよびUnityEngine.UI.dllが含まれます。 ここでは、いくつかのアセンブリが明らかに欠落していることに注意してください。 明らかに、私のスクリプトはUnityEngine.dllを参照しています。つまり、少なくともmscorlib.dllと、おそらく他のアセンブリが必要です。 しかし、彼らはどこですか？ 実際、アセンブリの解決はil2cpp.exe内で行われます。 これらはコマンドラインで言及できますが、これはオプションです。 Unityは、ルートアセンブリ（他のアセンブリが参照しない）を指定するだけでよいことがわかりました。



il2cpp.exeの最後のコマンドライン引数は、C ++ファイルが作成されるディレクトリです。 興味がある場合は、生成されたファイルに慣れることができますが、次の記事でそれらについて詳しく説明します。 ディレクトリを開く前に、WebGLビルド設定でDevelopment Playerオプションを選択する必要があります。 これにより、-output-format = Compact引数が削除され、C ++コードでの型名とメソッド名の表示が改善されます。



Player Settings WebGLまたはiOSで遊んでみてください。 il2cpp.exeに渡され、コード生成のさまざまな段階を担当するさまざまなコマンドラインパラメーターが表示されます。 たとえば、WebGL Player SettingsのEnable ExceptionsパラメーターをFullに設定すると、-emit-null-checks、-enable-stacktrace、および-enable-array-bounds-check引数がコマンドラインに追加されます。



IL2CPPは何をしないのですか？



バイパスすることで多くの問題を回避することにした1つの複雑さに注意を喚起したいと思います。 IL2CPPの下で標準C＃ライブラリを書き換えようとしませんでした。 IL2CPPを使用してUnityプロジェクトを作成すると、標準C＃ライブラリ（mscorlib.dll、System.dllなど）のすべてのコードはMonoスクリプトコードの正確なコピーです。



多くのプロジェクトでうまく機能しているため、標準C＃ライブラリのコードを使用します。 したがって、どこかでエラーが発生した場合、問題はAOTコンパイラまたは実行可能ライブラリにあると確信できます。



IL2CPPの開発、テスト、およびリリース



1月のIL2CPPバージョン4.6.1p5の最初の公式リリース以来、6つの完全なアップデートと7つのパッチ（Unity 4.6および5.0用）をリリースし、100以上のエラーを修正しました。



継続的な最適化を保証するために、1つのバージョンのコードに基づいて更新を開発しています。 各リリースの前に、すべての変更を特定のリリースブランチに転送し、テストを実施して、すべてのエラーが正常に修正されたかどうかを確認します。 QAチームとサステナブルエンジニアリングチームは、そのペースで開発を続けるためにあらゆる努力を払い、エラーは毎週修正されます。



私たちのコミュニティは、多くの有用なエラーメッセージを提供することにより、非常に貴重な支援を提供してきました。 IL2CPPの継続的な改善に役立つすべてのレビューについて、ユーザーに非常に感謝しています。



IL2CPP開発者にとっては、テストが最初になります。 私たちはしばしば、テストを通じて開発技術を基礎としており、コードが完全にテストされていない場合、含めるリクエストを送信することはほとんどありません。 この戦略は、明確な入力と出力があるIL2CPPのようなテクノロジーに適しています。 つまり、発生するエラーの大部分は、予期しないシステム動作よりもむしろ特殊なケースです（ たとえば 、64ビットIntPtrを32ビット配列インデックスとして使用すると、C ++コンパイラエラーが発生する場合があります）。 したがって、バグをすばやく特定して修正できます。

コミュニティの支援により、IL2CPPの安定性と生産性を常に向上させています。
