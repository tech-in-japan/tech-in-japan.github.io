  1. アプリケーション全体に対して1つまたは2つのコンポーネントを使用します。

ダガーの研究の最初に欠けていた、少し抽象化してみましょう。



Daggerのインターネットでの使用例のほとんどには、コンテキストルート依存関係（または本質的にはコンテキストでもあるアプリケーションを拡張するクラス）を備えたAppModuleを使用したいわゆるAppComponentの作成が必ず含まれます。



理由を理解しましょう。



「最初は言葉でした...」



コンテキストがあると、他の依存関係を取得できます。たとえば、SharedPreferences、LayoutInflater、何らかのシステムサービスなどです。 したがって、SharedPreferencesを持つことで、PreferenceHelperを取得できます。これは、設定を操作するためのユーティリティクラスです。 LayoutInflaterを使用すると、ViewFactoryを取得できます。 これらの「高レベル」依存関係から、さらに複雑で複雑な依存関係を取得することもできます。 そして、この多様性はすべて、1つのオブジェクト、つまりコンテキストに由来していました。 この場合、AppComponentのコアと呼ぶことができます。









そして、上記のすべては、アプリケーション全体、つまり シングルトン。 そのため、私たちが持っているコアは、常に存在するオブジェクト-アプリケーションコンテキストオブジェクトです。



この考えを続けて、アダプターがどれくらいの期間存在するかを考えてみましょう。 明らかに、このアダプタが機能する画面があります。



ViewHolderFactoryをアダプタに提供します。アダプタは、アダプタが存在する限り存在する必要があります。 アダプタに加えて、フラグメントにViewControllerを提供します。フラグメントも存在する場合にのみ存在する必要があります。



見ると、この画面が「生きている」間に排他的に使用されるすべての依存関係は、この画面に依存しています。 T.O. 私たちのフラグメント（またはアクティビティ）は、画面が存在する限り存在するローカルコンポーネント-コンポーネントのコアになると言えます。









依存関係のこのすべてのローカルバンチ（列）に対して明確に定義されたライフタイムを実装するには、サブコンポーネントを使用します。



「どうやって？」と聞いて



とりあえず、サブプレフィックスを忘れて、コンポーネントだけを実装していると想像してみましょう。 簡単な場合は、画面がアプリケーション全体であることを想像してください。



まず、基本的なモジュールが必要です。 なぜなら リスト画面、これをListModuleと呼びます。 



 @Module public class ListModule { }


今、私たちはまさにコア、つまり他の誰もが行く基本的な依存関係を必要としています。 前述のように、ListFragmentなどの「画面オブジェクト」自体は、画面の基本的な依存関係です。 それをモジュールのコンストラクターに渡しましょう。 



 @Module public class ListModule { private final ListFragment fragment; public ListModule(ListFragment fragment) { this.fragment = fragment; } }


基礎、そして創造性があります。



アダプターを提供します。 



 @Provides public Adapter provideAdapter(Context context) { return new Adapter(fragment, context, fragment.initList()); }


注：コンテキストはありますが、明らかに、このモジュールまたはコンポーネントの他のモジュールのいずれにも提供していません。 これについては後で詳しく説明します。



データ自体のリストを個別に提供することもできます（これは冗長ですが、例では機能します）。 



 @Provides public List<Model> provideListOfModels() { return fragment.initList(); } @Provides public Adapter provideAdapter(Context context, List<Model> list) { return new Adapter(fragment, context, list); }


さて、それを正常に機能させるために、いくつかの設定を行います。



ダガーに伝えるために：



  1. コンポーネントのすべての依存関係は、メインのグラフとは別の単一のグラフです。
  2. 毎回新しい依存関係を作成するのではなく、唯一の依存関係をキャッシュする必要があります。


いわゆるスコープ注釈があります。 各スコープ注釈は次のようになります。 



 @Scope @Retention(RetentionPolicy.Runtime) public @interface Singleton {}


シングルトンは、Daggerが提供する基本的な注釈です。 それはあなたが何かを始めるためだけに提供されています。 「シングルトン」自体は、AppComponent（Appを拡張するクラス）にAppComponentを保存しない限り、魔法のようには発生しません。 すなわち Daggerは、Componentのこのインスタンスに対して依存関係の単一のインスタンスが作成されることを保証します。 ただし、コンポーネントインスタンスの一意性はユーザーの責任です。



同様に、スコープアノテーションを作成します。 



 @Scope @Retention(RetentionPolicy.Runtime) public @interface ListScope {}


私たちの注釈は、Singleton注釈よりも決して劣っていません。全体のポイントは、それらの使用方法です。



スコープ注釈を使用して、提供メソッドとモジュールを含むコンポーネントをマークします。



重要：特定のスコープによって署名された1つのコンポーネントには、同じスコープによって提供メソッドが署名されたモジュールのみが存在できます。 T.O. 2つの異なる依存関係グラフは交差しません。


ListModuleの最終ビュー： 



 @Module public class ListModule { private final ListFragment fragment; public ListModule(ListFragment fragment) { this.fragment = fragment; } @ListScope @Provides public List<Model> provideListOfModels() { return fragment.initList(); } @ListScope @Provides public Adapter provideAdapter(Context contex, List<Model> list) { return new Adapter(fragment, context, list); } }


コンポーネント： 



 @ListScope @Subcomponent(modules = ListModule.class) public interface ListComponent { void inject(ListFragment fragment); }


ここで重要なのは@Subcomponentアノテーションです。 したがって、親コンポーネントのすべての依存関係にアクセスしたいと言いますが、ここでは親を指定しないことに注意してください。 この例では、親はAppComponentです。



* Contextを取得してアダプターを初期化するのはAppComponentからです。



サブコンポーネントを取得するには、サブコンポーネントのすべてのモジュールを引数として渡すことで、親コンポーネントでの受信方法を記述する必要があります（この場合、1つのモジュールのみ）。



それはどのように見えますか： 



 @Singleton @Component(modules = AppModule.class) public interface AppComponent { ListComponent listComponent(ListModule listModule); }


Daggerはこのメソッドの実装を処理します。



人生の時間を整理します



既に述べたように、AppComponentにはその唯一のインスタンスが格納されているため、AppComponentはシングルトンです。 親の助けを借りてのみサブコンポーネントのインスタンスを作成できます。したがって、Appクラスでサブコンポーネントを受信および保存するすべてのロジックを削除しますが、1つの重要な違いがあります：いつでもサブコンポーネントを作成し、いつでも破棄する機能を追加します。



Appクラスでは、次のロジックを記述します。 



 public class App extends Application { private ListComponent listComponent; public ListComponent initListComponent(ListFragment fragment) { listComponent = appComponent.listComponent(new ListModule(fragment)); return listComponent } public ListComponent getListComponent() { return listComponent; } public void destroyListComponent() { listComponent = null; } }


注：大規模なプロジェクトでは、構成を使用してDaggerを操作するロジックをAppクラスからヘルパークラスに移動することは理にかなっています。



さて、フラグメント内でこれらすべてを使用することを説明することは残っています。 



 public class ListFragment extends Fragment { @Inject Adapter adapter; @Override public void onCreate(Bundle savedInstanceState) { super.onCreate(savedInstanceState); App.getInstance().initListComponent(this).inject(this); init(); } private void init() { recyclerView.setAdapter(adapter); } @Override public void onDestroy() { super.onDestroy(); App.getInstance.destroyListComponent(); } }


したがって、グラフのライフタイムをフラグメントのライフサイクルにリンクしました。



これは、1つの依存関係の場合には冗長に見えるかもしれません（1つの依存関係があったとしても、そのようなロジックを取り除くとコードがよりきれいになり、 フックが少なくなります ）。 ほとんどの作業は、アーキテクチャを作成することです。 したがって、新しい依存関係を提供する必要がある場合、1つの提供メソッドの実装になります。







ボーナス



すべての依存関係が提供メソッドで強調表示されている場合、依存関係をスローするのをやめるなどの素晴らしいことがあります。 アダプターを使用した例を再度検討してください。



ListFragmentは、リストオブジェクトのViewHoldersに関連するイベントのリスナーを実装します。 したがって、各ViewHolderにリスナーを配信するには、アダプターへのリスナーへのリンクを保存する必要があります。



仲介者を取り除きます。











ViewHolderFactoryでViewHoldersの作成を削除することをお勧めします。 だから私たちはやる： 



 public class ListItemViewHolderFactory { private final Listener listener; private final LayoutInflater layoutInflater; public ListItemViewHolderFactory(LayoutInflater layoutInflater, Listener listener) { this.layoutInflater = layoutInflater; this.listener = listener; } public ListItemViewHolder createViewHolder(ViewGroup parent) { View view = layoutInflater.inflate(R.layout.item, parent, false); return new ListItemViewHolder(view, listener); } }


モジュールは次の形式に変換されます。 



 @Module public class ListModule { private final ListFragment fragment; public ListModule(ListFragment fragment) { this.fragment = fragment; } @ListScope @Provides public List<Model> provideListOfModels() { return fragment.initList(); } @ListScope @Provides public Adapter provideAdapter(ListItemViewHolderFactory factory, Context context, List<Model> list) { return new Adapter(factory, context, list); } @ListScope @Provides public ListItemViewHolderFactory provideVhFactory(LayoutInflater layoutInflater) { return new ListItemViewHolderFactory (layoutInflater, fragment); } }


注：AppModuleでLayoutInflaterを提供することを忘れないでください。



この例は、依存関係の柔軟な作業がいかにうまくいくかを示しているように思えます。
