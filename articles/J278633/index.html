    4.2ドメインクラスの作成


    JustBlog.Coreプロジェクトで、 Objectsという名前の新しいフォルダーを作成して、ドメインクラスをその中に配置します。 ブログの場合、3つのクラス、 Post 、 Category 、およびTagを作成する必要があります。 各投稿は1つのカテゴリに属しますが、多くのタグでマークすることができます。 投稿とカテゴリの 多対1の関係間および投稿とタグの 多対多の関係。



    クラス間の関係は、以下のスクリーンショットに示されています。







    以下はPostクラスです 



    namespace JustBlog.Core.Objects { public class Post { public virtual int Id { get; set; } public virtual string Title { get; set; } public virtual string ShortDescription { get; set; } public virtual string Description { get; set; } public virtual string Meta { get; set; } public virtual string UrlSlug { get; set; } public virtual bool Published { get; set; } public virtual DateTime PostedOn { get; set; } public virtual DateTime? Modified { get; set; } public virtual Category Category { get; set; } public virtual IList<Tag> Tags { get; set; } } }




    ほとんどのプロパティは一目瞭然です。 UrlSlugプロパティはTitleプロパティの代替であり、アドレス指定に使用されます。



    ナメクジは新聞の用語です。 「スラッグ」は、文字、数字、アンダースコア、またはハイフンのみを含む短いタグ名です。 主にURLで使用されます。





    たとえば、2010年8月に公開された「Advanced Linq in C＃」というタイトルの投稿があり、そのためにURL http // localhost / archive / 2010/8 / Advanced Linq in C＃を作成します。 投稿タイトルには特殊文字（この例では「＃」）が含まれている場合があり、すべてのサーバーが特殊文字を含むリクエストを処理できるわけではありません。 URLでTitleプロパティを直接使用する代わりに、投稿のタイトルのように見える代替テキストを使用し、 Slug URLプロパティから取得します。



    上記の場合、URLで「Advanced Linq in C＃」を使用する代わりに、フレンドリーテキスト（slug） 「advanced_linq_in_csharp」を使用するため、アドレスはhttp // localhost / archive / 2010/8 / advanced_linq_in_csharpになります。 パート3では、投稿タイトルからスラッグを自動的に作成する方法を学習します。

    4.3 Fluent NHibernateおよびNHibernateの構成


    データベースにアクセスするには、 NHibernateとFluent NHibernateを使用します。 NHibernateはEntity FrameworkのようなORMツールであり、クラスとデータベーステーブル間の関係はXMLファイルを介してマッピングされます。 Fluent NHibernateはNHibernateの拡張機能で、XMLファイルによるマッピングを特別なクラスを使用したマッピングに置き換えます。 クラスを介したマッチングは、XMLファイルを使用するよりもずっと簡単です。



    Nuget Package Manager Consoleを使用して、NHibernateおよびFluent NHibernateアセンブリへの参照を簡単に追加できます。 [ ツール]-> [ライブラリパッケージマネージャー]-> [パッケージマネージャーコンソール ]メニューからパッケージマネージャーを開きます。

    4.4マッピングクラス、データアクセスクラス、およびメソッドの作成


    次に行う必要があるのは、必要なマッピングクラスを作成することです。 マッピングクラスは、クラスとそのプロパティをテーブルとその列にマップするために使用されます。 Mappingsと呼ばれるJustBlog.Coreプロジェクトに新しいフォルダーを作成して、すべてのマッピングクラスを保存します。



    マッピングクラスを作成するには、Fluent NHibernateフレームワークの汎用ClassMapクラスからマッピングクラスを継承する必要があります 。 すべてのマッピングはコンストラクターで実行する必要があります。

    Postクラスのマッピングクラス。 



     using FluentNHibernate.Mapping; using JustBlog.Core.Objects; namespace JustBlog.Core.Mappings { public class PostMap: ClassMap<Post> { public PostMap() { Id(x => x.Id); Map(x => x.Title).Length(500).Not.Nullable(); Map(x => x.ShortDescription).Length(5000).Not.Nullable(); Map(x => x.Description).Length(5000).Not.Nullable(); Map(x => x.Meta).Length(1000).Not.Nullable(); Map(x => x.UrlSlug).Length(200).Not.Nullable(); Map(x => x.Published).Not.Nullable(); Map(x => x.PostedOn).Not.Nullable(); Map(x => x.Modified); References(x => x.Category).Column("Category").Not.Nullable(); HasManyToMany(x => x.Tags).Table("PostTagMap"); } } }




    拡張メソッドIdは、テーブル列の主キーとして設定されるプロパティの名前を表します。



    Map拡張メソッドは、プロパティをテーブル列にマップするために使用されます。 プロパティのマッピング中に、列のサイズを指定したり、NULL値の格納を許可するかどうかを指定したり、他の仕様を指定したりできます。



    例： 



     Map(x => x.ShortDescription).Length(5000).Not.Nullable();




    生成された列名がプロパティ名と異なる場合、 列拡張メソッドを使用して列名を渡す必要があります。 



     Map(x => x.Title).Column("post_title")




    Referencesメソッドは、 Postテーブルの外部Categoryキー列を使用して、 PostとCategoryの多対1の関係を表すために使用されます。 HasManyToManyメソッドは 、 PostTagMapステージングテーブルを使用して、 PostとTagの多対多の関係を表すために使用されます。 Fluent NHibernateとその拡張メソッドの詳細については、こちらをご覧ください 。



    デフォルトでは、 Fluent NHibernateはテーブル名がクラス名とまったく同じであり、列名がプロパティ名とまったく同じであると想定しています。 テーブル名が異なる場合、テーブル拡張メソッドを使用してテーブルをクラスにマップする必要があります。



    例： 



     Table("tbl_posts");




    CategoryとTagのマッピングクラスは、 Postとの関係を除いてまったく同じです。 カテゴリは1 対多の投稿に関連していますが、 タグは多対多の 投稿に関連しています。 



     namespace JustBlog.Core.Mappings { public class CategoryMap: ClassMap<Category> { public CategoryMap() { Id(x => x.Id); Map(x => x.Name).Length(50).Not.Nullable(); Map(x => x.UrlSlug).Length(50).Not.Nullable(); Map(x => x.Description).Length(200); HasMany(x => x.Posts).Inverse().Cascade.All().KeyColumn("Category"); } } }




    Inverse（）拡張メソッドは、リレーションシップの反対側を保存の責任にします。



    拡張メソッドCascade.All（）は、コレクションエンティティでカスケードイベント呼び出し（階層を下る）を開始します（したがって、カテゴリを保存すると、投稿も保存されます）。 



     namespace JustBlog.Core.Mappings { public class TagMap: ClassMap<Tag> { public TagMap() { Id(x => x.Id); Map(x => x.Name).Length(50).Not.Nullable(); Map(x => x.UrlSlug).Length(50).Not.Nullable(); Map(x => x.Description).Length(200); HasManyToMany(x => x.Posts).Cascade.All().Inverse().Table("PostTagMap"); } } }




    リポジトリテンプレートを使用してデータベースにアクセスします。 このテンプレートを使用して、データアクセスコードをコントローラーから分離します。これにより、コントローラーの単体テストを簡素化できます。 リポジトリテンプレートのコアは、すべてのデータアクセス方法の説明を含むインターフェイスです。



    JustBlog.Coreプロジェクトで、 2つのメソッドを使用してIBlogRepositoryインターフェイスを作成します。 



     using JustBlog.Core.Objects; namespace JustBlog.Core { public interface IBlogRepository { IList<Post> Posts(int pageNo, int pageSize); int TotalPosts(); } }




    Postsメソッドは、渡された値に従って最後に公開されたページ分割された投稿を返すために使用されます。 TotalPostsメソッドは、公開された投稿の総数を返します。 次に、インターフェイスに他のメソッドを入力します。



    JustBlog.Coreプロジェクトでは、 BlogRepositoryと呼ばれるクラスでインターフェースを実装します。 



     using JustBlog.Core.Objects; using NHibernate; using NHibernate.Criterion; using NHibernate.Linq; using NHibernate.Transform; using System.Collections.Generic; using System.Linq; namespace JustBlog.Core { public class BlogRepository: IBlogRepository { // NHibernate object private readonly ISession _session; public BlogRepository(ISession session) { _session = session; } public IList<Post> Posts(int pageNo, int pageSize) { var posts = _session.Query<Post>() .Where(p => p.Published) .OrderByDescending(p => p.PostedOn) .Skip(pageNo * pageSize) .Take(pageSize) .Fetch(p => p.Category) .ToList(); } var postIds = posts.Select(p => p.Id).ToList(); return _session.Query<Post>() .Where(p => postIds.Contains(p.Id)) .OrderByDescending(p => p.PostedOn) .FetchMany(p => p.Tags) .ToList(); } public int TotalPosts() { return _session.Query<Post>().Where(p => p.Published).Count(); } } }




    すべてのデータベースアクセスは、NHibernate ISessionオブジェクトを介して行う必要があります。 ISessionを使用して投稿のコレクションを読み取る場合、 デフォルトではカテゴリとタグの依存関係は読み込まれません。 FetchおよびFetchManyメソッドは、NHibernateにそれらをすぐに取り込むように指示するために使用されます。



    Postsメソッドでは、LinqクエリでFetchManyをSkipおよびTakeメソッドと一緒に使用できないため、投稿を取得するためにデータベースを2回クエリします。 そのため、まずすべての投稿を選択し、次にそれらのIDを使用して、投稿を再度選択してタグを取得します。 この問題の詳細については、 このディスカッションスレッドを参照してください。



    NHibernate ISession



    ISessionは、データベースからオブジェクトを保存および取得するために使用される永続性マネージャーインターフェイスです。



    永続的とは、このインスタンスを、それを生成したプロセスの完了後に保持されるように実装する必要があることを意味します。 一時的とは異なり、これはオブジェクトのこのインスタンスが一時的であることを意味します-オブジェクトを生成したプロセスの完了後に存在しなくなります。 （コメント翻訳者）。





    4.5 JustBlog.CoreプロジェクトのNinject設定


    依存性注入（DI）は、クラス内の特定の依存関係実装のインスタンス化を回避します。 これらの依存関係は通常、コンストラクターを介してクラスに導入されますが、プロパティを介して導入されることもあります。 依存性注入の主な利点の1つは単体テストであり、パート3でコントローラーの単体テストを作成するときにこれを確認します。



    Castle Windsor、Unity、Autofac、StructureMap、Ninjectなど、依存関係の注入を単純化するフレームワークが多数あります。Ninjectは使いやすいため選択しました。



    Package Managerコンソールで次のコマンドを実行して、 NinjectをJustBlog.Coreプロジェクトにインストールします。



    PM> Install-Package Ninject

    PM> Install-Package Ninject.Web.Common









    コマンドを正常に実行すると、NinjectおよびNinjectWeb.Commonアセンブリがプロジェクトに追加されたことがわかります。 アセンブリと共に、 NinjectWebCommon.csファイルもApp_Startフォルダーに追加されました。 近い将来、Ninject.Web.Common拡張機能をインストールする必要がある理由を説明します。



    Global.asax.csを使用するか、 App_Startファイルを使用して、 Ninjectを 2つの方法で構成できます。 最初のアプローチを使用するため、 App_StartフォルダーからNinjectWebCommon.csファイルを削除し、プロジェクトからWebActivatorおよびMicrosoft.Web.Infrastructureアセンブリへの不要な参照を削除してください 。



    DIフレームワークの主な機能は、これらのインターフェイスの特定の実装へのインターフェイスのバインドです。 インターフェイスを特定の実装に関連付けることをバインディングと呼びます。 Ninjectモジュールを使用して、特定のNinjectモジュールに関連付けられたバインディングのセットをグループ化できます。 すべてのバインディングとモジュールは、カーネルカーネルのメインNinjectコンポーネントにロードされます。 アプリケーションがインターフェイスを実装する特定のクラスのインスタンスを必要とするときはいつでも、カーネルはそれをアプリケーションに提供します。



    Ninjectモジュール



    Ninjectモジュールは、特定のモジュールに関連する一致/バインディングを1つのクラスにグループ化するために使用されます。





    BlogRepositoryクラスには、 Nibernate ISessionとの依存関係があります 。 ISessionのインスタンスを作成するには、 ISessionFactoryという別のNibernateインターフェイスが必要です 。 JustBlog.Coreプロジェクトで、これら両方のインターフェースを接続するRepositoryModuleというクラスのninjectモジュールを作成しましょう。



    作成済みq Ninjectモジュールは、抽象NinjectModuleクラスから継承し、 Loadメソッドを実装する必要があります。 Loadメソッドでは、 Bindメソッドを使用して両方のインターフェイスをメソッドにバインドします。



    Bindメソッドは、インターフェイスを実装するクラスにインターフェイスをバインドするために使用されます。



    例： 



     Bind<IFoo>().To<Foo>();




    また、インターフェイスの実装を作成して返すメソッドにインターフェイスをマッピングすることもできます。 



     Bind<IFoo>().ToMethod(c => { var foo = new Foo(); return foo; });




    NHibernate ISessionFactory



    ISessionとは異なり、アプリケーション全体でISessionFactoryの1つのインスタンスが必要です。 





     using FluentNHibernate.Cfg; using FluentNHibernate.Cfg.Db; using JustBlog.Core.Objects; using NHibernate; using NHibernate.Cache; using NHibernate.Tool.hbm2ddl; using Ninject; using Ninject.Modules; using Ninject.Web.Common; namespace JustBlog.Core { public class RepositoryModule: NinjectModule { public override void Load() { Bind<ISessionFactory>() .ToMethod ( e => Fluently.Configure() .Database(MsSqlConfiguration.MsSql2008.ConnectionString(c => c.FromConnectionStringWithKey("JustBlogDbConnString"))) .Cache(c => c.UseQueryCache().ProviderClass<HashtableCacheProvider>()) .Mappings(m => m.FluentMappings.AddFromAssemblyOf<Post>()) .ExposeConfiguration(cfg => new SchemaExport(cfg).Execute(true, true, false)) .BuildConfiguration() .BuildSessionFactory() ) .InSingletonScope(); Bind<ISession>() .ToMethod((ctx) => ctx.Kernel.Get<ISessionFactory>().OpenSession()) .InRequestScope(); } } }


    Fluent NHibernate Wikiが一般的に言っているように、Fluent NHibernate APIを使用したNHibernateのセットアップは5つの主要なメソッドを使用して行われ、そのうち3つが必要です。 



     Fluently.Configure() .Database(/* your database settings */) .Mappings(/* your mappings */) .ExposeConfiguration(/* alter Configuration */) // optional .BuildSessionFactory();




    Fluently.Configureはセットアッププロセスを開始します

    Database 、データベース接続文字列を構成します

    マッピングは、エンティティクラスに基づいてマッピングを設定します

    ExposeConfigurationはオプションですが、マッピングからデータベーススキーマを作成できます（クラスからテーブルを作成します）

    BuildSessionFactoryは最後のメソッドであり、以前のメソッドの設定に基づいてNHibernate SessionFactoryのインスタンスを作成します。







    Cacheメソッドはオプションであり、要求をキャッシュするようにプロバイダーを構成します。



    Ninjectには多くの拡張機能があり、そのうちの1つはNinject.Web.Commonです。これには、WebFormsおよびMVCアプリケーションに必要ないくつかの共通機能が含まれています。 クエリの実行中に、NinjectがISessionの同じインスタンスを返すことを確認する必要があります。 Ninject.Web.Common名前空間で定義されたInRequestScope（）拡張メソッドは、ASP.NETが受信したHTTP要求ごとに、クラスのインスタンスを1つだけ作成する必要があることをNinjectに伝えます。 InSingletonScope（）拡張メソッドは、アプリケーション全体で使用可能なシングルトンを作成します。

    App_StartフォルダーからNinjectWebCommon.csファイルを削除します。 NinjectHttpApplicationクラスからグローバルアプリケーションクラスを継承し、 CreateKernelメソッドをオーバーライドします。

    以下に示すように、Global.asax.csファイルを変更します。 



     using Ninject; using Ninject.Web.Common; using System.Web.Mvc; using System.Web.Routing; namespace JustBlog { public class MvcApplication : NinjectHttpApplication { protected override IKernel CreateKernel() { var kernel = new StandardKernel(); kernel.Load(new RepositoryModule()); kernel.Bind<IBlogRepository>().To<BlogRepository>(); return kernel; } protected override void OnApplicationStarted() { RouteConfig.RegisterRoutes(RouteTable.Routes); base.OnApplicationStarted(); } } }




    CreateKernelメソッドでは、タイプIKernelの StandardKernelインスタンスを作成して返します。 IKernelは、バインディングを指定するアプリケーションのコアであり、インスタンスをインターフェイスにマッピングする必要がある場合、それを提供します。



    StandardKernelクラスのインスタンスを使用して、いくつかの重要なことを行います。 まず、 NHibernateインターフェースに関連付けられたすべてのバインディングを含むRepositoryモジュールのインスタンスをロードし、次にIBlogRepositoryインターフェースをそのBlogRepository実装に直接バインドします。 そして最後に、例の最後で、アプリケーションの起動コードがOnApplicationStartedメソッドに移動しました。



    これがアプリでのNinjectの設定です。 次のパートでは、コントローラーとビューの作業を開始します。
