COMポートからWebへのデータのリダイレクト

最近、ハブでArduinからSerialに送信されたデータを美しく表示する方法についての記事「Chromeアプリケーションでのシリアルからのデータの表示」がありました。 私の意見では、彼らは非常に美しい解決策を提案しました。それは一方では非常にシンプルに見え、他方では最小限の労力で素晴らしい結果を得ることができます。



記事へのコメントでは、このようなソリューションがFirefoxで機能しないことを後悔し、「このことを基にしたHTML出力を使用して単純なWebサーバーを作成できる」という考えが表明されました。 このアイデアは私を「夢中にさせた」、グーグルでの簡単な検索では既成のソリューションを提供できなかったので、自分でアイデアを実装することにしました。 そして、ここからそれが生まれました。



警告！ いかなる場合でも、提案されたソリューションが完全であると見なされるべきではありません。 AmperkaのSerial Projectorとは異なり、これはコンセプトであり、可能なアプローチのデモンストレーションであり、試作品であり、それ以上のものではありません。



しばらく前に、Androidスマートフォンに組み込まれた加速度計を使用して、Arduinoに接続されているサーバーを制御するプロジェクトを実行していました。 次に、これらの目的のために、プロジェクトScripting Layer for Android（SL4A）およびRemoteSensorsを利用しました 。 BaseHTTPServerパッケージは標準のpythonライブラリに含まれており、これを使用してpythonでWebサービスを上げることができます。これは数行のコードのタスクです。



Arduinoには手元にセンサーがなかったので、表示される情報のソースとしてArduino Unoに内蔵された内部温度計を使用しました。 私の知る限り、それはあまり正確ではなく、周囲温度を測定することを意図していませんが、プロトタイピングには有効です。



短いグーグルの後に、このスケッチはArduinkaに登場しました。 



// source: https://code.google.com/p/tinkerit/wiki/SecretThermometer long readTemp() { long result; // Read temperature sensor against 1.1V reference ADMUX = _BV(REFS1) | _BV(REFS0) | _BV(MUX3); delay(2); // Wait for Vref to settle ADCSRA |= _BV(ADSC); // Convert while (bit_is_set(ADCSRA,ADSC)); result = ADCL; result |= ADCH<<8; result = (result - 125) * 1075; return result; } void setup() { Serial.begin(115200); } int count = 0; void loop() { String s = String(count++, DEC) + ": " + String( readTemp(), DEC ); Serial.println(s) delay(1000); }


このスケッチはCOMポートを開き、115200ボーレートに設定してから、内蔵温度計の現在の値を毎秒書き込みます。 （温度が発行される単位を尋ねないでください-これは、説明されている問題にとって重要ではありません）。 値はあまりアクティブに変化しないため、データの変化を見やすくするために、温度の前に行番号が表示されます。



WebサーバーがCOMポートから読み取る行の一部ではなく、行全体のみを出力することを確認するには、行

に置き換えられました 

  for(int i=0; i < s.length(); i++ ){ Serial.print( s.charAt(i) ); delay( 200 ); } Serial.println("");


つまり 生成された文字列は完全にシリアルポートに出力されるのではなく、文字ごとに200ミリ秒の休止で出力されます。

 class Handler(BaseHTTPServer.BaseHTTPRequestHandler):


これはBaseHTTPServerモジュールに組み込まれたクラスの子孫であり、do_GETメソッドのみをオーバーライドするのに十分です。



これはまだプロトタイプであるため、サーバーはあらゆる要求に「満足」します。どのURLから要求された場合でも、COMポートから読み取ったすべてのデータをクライアントに提供します。 したがって、Handler.do_GETでは、成功コードと必要なヘッダーですぐに応答します。

その後、無限ループが開始され、COMポートから行全体の読み取りが試行され、この試行が成功した場合、Webクライアントに渡されます。

 class Handler(BaseHTTPServer.BaseHTTPRequestHandler): # Disable logging DNS lookups def address_string(self): return str(self.client_address[0]) def do_GET(self): if self.path == '/' or self.path == '/index.html': self.process_index() elif self.path == '/get_serial': self.process_get_serial() else: self.process_unknown() def process_index(self): self.send_response(200) self.send_header("Content-type", "text/html; charset=utf-8") self.end_headers() self.wfile.write(open('index.html').read()) self.wfile.write("\n\n") self.wfile.flush() def process_get_serial(self): self.send_response(200) self.send_header("Content-type", "text/event-stream") self.end_headers() try: while True: new_serial_line = get_full_line_from_serial() if new_serial_line is not None: self.wfile.write('data: ' + new_serial_line) self.wfile.write("\n\n") self.wfile.flush() except socket.error, e: print "Client disconnected.\n" def process_unknown(self): self.send_response(404)


このオプションでは、Webサーバーが「/index.html」（ページのhtmlコードを与える）と「/ get_serial」（COMポートから読み取った行の無限ストリームを与える）の2つの要求のみに応答することを前提としています。 彼は他のすべてのリクエストに404のコードで応答します。



index.htmlはPython Webサーバーによって提供されるため、COMポートからの行のストリームの絶対アドレスの代わりに指定することで、わずかに変更できます。

ここで他にできること：

  • まず、PythonスクリプトのCOMポートからデータを読み取るのは非常に「面倒」です。実際、「何か新鮮なものはありますか？」という絶え間ないポーリングがあります。 もちろん、このアプローチはプロセッサーをロードし、コンピューター上の1つのコアが100％占有されます。

    解決策として、タイムアウト付きのブロック読み取りを使用できます。 これを行うには、COMポートを開くときに、タイムアウトとしてゼロ以外の値（秒単位）を指定するだけで十分です。次に例を示します。 

      ser = serial.Serial(SERIAL_PORT_NAME, SERIAL_PORT_SPEED, timeout=0.03)


    さらに、pySerialモジュールの説明には、「TCP / IP-シリアルブリッジ」、「シングルポートTCP / IP-シリアルブリッジ（RFC 2217）」、および「マルチポートTCP / IP-シリアルブリッジ（RFC） 2217）「-専門家がそのような問題をどのように解決するかを見ることができます。
  • 次に、1つのクライアントのみがデータを受信できます。 最初のクライアントでページが閉じられるまで、このサーバーに接続して2番目のコンピューターで値を受け取ることはできません。 一方で、これはおそらく正しいです。COMポートは1つだけで、複数のコンシューマがあります。そのうちのどれが読み取り行を提供する必要がありますか？ この質問への答えが「すべて」であるべきだと思うなら、この問題に関する私の考えがここにあります。 この質問は、複数のWebクライアントの要求を同時に処理できる「正直な」マルチスレッドWebサーバー（たとえば、TornadoやFlaskなど）を使用するだけでは解決できないように思えます。 各スレッドからCOMポートを開いて読み取ることはできないため、この場合、COMポートからのデータは1つのスレッド/プロセスのみに送られます。 したがって、私の意見では、サーバー部分を2つの部分に分ける必要があります。

    • COMポートで動作するzmqサーバーは、そこから行を読み取り、PUBソケットを介してすべての関心のあるコンシューマーに送信します
    • COMポートに接続する代わりにPython Webサーバーがzmqサーバーに接続し、そこからデータを受信する


    ZMQライブラリ（ZeroMQ）に慣れていない場合は、代わりに通常のTCP / IPまたはUDPソケットを使用できますが、このライブラリを使用するとこのような問題を簡単に解決できるため、ZMQについて知ることを強くお勧めします。 ZMQの助けを借りて、ソリューションは最大20行に収まるように思えます。 （私は助けることはできませんが、記述された問題を解決する予定はありませんが、あなたの仕事はスレッド/プロセス間でデータを交換するマルチスレッド/マルチプロセスプログラミングに関連している場合でも、このライブラリを見てください-多分これはあなたがずっと話してきたことです夢見た）
  • これまでのデータストリームは、COMポートからWebブラウザまでの単方向です。 ブラウザからArduinoにまだデータを送信することはできません。 このタスクもそれほど難しくはないようで、前のタスクとは異なり、解決することしかできません

    • マルチスレッドサーバーを使用する
    • Handler.do_GETメソッドをファイナライズして、パラメーター付きのGETリクエストを受け入れ、特定の値をCOMポートに送信する


    私の意見では、Arduino IDEに組み込まれたWebベースのシリアルポートモニターの本格的なアナログを書きたいのであれば、それほど難しくありません。 私にとっては、通常のフロントエンドを作成することだけが難しいと思います。

  • ブラウザを介して、COMポートの名前とその操作パラメーターを設定することはまだできません。 一方で、これは論理的に思えます。私たちの惑星の反対側のユーザーは、どのCOMポートとどの速度でArduinが接続されているかをどのようにして知ることができますか？ しかし、同じコンピューターで実行されているPython Webサーバーを正確に認識しています。 ただし、COMポートの名前や作業のパラメーターを変更する機会をユーザーに与えることが依然として望ましい場合は、Handler.do_GETメソッドをファイナライズすることで簡単に解決できます。
  • サーバーを起動するには、pythonをインストールする必要があります。 これは一般的に難しいことではありませんが、何らかの理由でこれを実行できない場合や望まない場合は、pyInstallerが助けになります。 これを使用して、Pythonスクリプトを1つの実行可能ファイル（Windowsの場合は.exe）に結合し、arduinkaが接続されているコンピューターに簡単にコピーできます。

    おそらく最善の解決策は、この場合にGo言語を使用することです。 私の知る限り、その中で「配布」用のファイルを作成するタスクはよりよく解決されます。
