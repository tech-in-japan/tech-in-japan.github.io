 $r->get("/a/b/c/d") ->to(text=>"1"); $ab = $r->route("/a")->route("/b"); $ab->get("/c") ->to(text=>"2-1"); $ab->get("/c/d") ->to(text=>"2-2"); $r->get("/a/b/c/d/e") ->to(text=>"3");


その結果、そのようなツリーが構築されます：

HTTPリクエストパラメータ



リクエストパラメータにアクセスする方法は多くありませんが、たくさんあります。 さらに、それらすべてを使用する価値はありません-場合によっては、異なる場所から取得されたパラメーターが混合され、どこから来たのかを理解することは非現実的になります。



Mojoliciousには4種類のパラメーターがあります。

  1. GET -URLのクエリ文字列から取得し、HTTP要求メソッドはGET、POSTなど、何でもかまいません。
  2. POST-タイプapplication/x-www-form-urlencoded



    またはタイプmultipart/form-data



    POST要求の本体から受信しmultipart/form-data



    -ただし、この場合、ファイルを除き、通常のパラメーターのみが取得されます
  3. UPLOAD-タイプmultipart/form-data



    POST要求の本文から受信したファイル
  4. ROUTE-スタッシュ 用に予約されているものを除き、ルーティングのプレースホルダーを使用してURLパスから切り取られた値


さらに、同じパラメーターを複数回渡すことができ、さらに、GET、POST、UPLOADの各方法で複数回渡すことができ、ルーティングを決定するときに同じプレースホルダーを数回言及できることに注意してください。 GET、POST、およびUPLOADの場合、1つのパラメーターの渡された値はすべて保存されますが、ROUTEの場合、1つのプレースホルダーが複数回指定されると、 最後の値のみが使用されます。



ほとんどの場合、例で$c->param



が言及されています-この関数によって返される値がどこから来たのか見てみましょう（Mojoliciousでは5.47まで） ：

  • scalar $c->param()



    -undefを返します
  • @names = $c->param()



    -すべてのGET、POST、UPLOADおよびROUTEパラメーターの名前を返します
  • $value = $c->param("name")



    -戻り値：

    1. 最後の値はプレースホルダー「名前」であり、そうでない場合は
    2. UPLOAD "name"の最初の値である場合、そうでない場合
    3. 最初のPOST値 "name"（ある場合）、そうでない場合
    4. 最初の値GET "name"である場合、そうでない場合
    5. 未定義


  • @values = $c->param("name")



    -戻り値：

    1. 最後の値はプレースホルダー「名前」であり、そうでない場合は
    2. すべてのUPLOAD "name"値（存在する場合）、それ以外の場合
    3. すべてのPOST "name"値、その後にすべてのGET "name"値（存在する場合）
    4. （）


個人的には、できるだけ明確なコードを書くことを好み、パラメーターが返されるような「マジック」関数は好きではありませんが、それがどこから来たのかは完全に不明です。 もちろん、場合によっては、パラメーターがGETまたはPOSTによって渡されるかどうかに注意を払う必要はありませんが、 $c->param



は、善悪の境界を超えています（たとえば、GET / POSTパラメーターを期待した場合、およびUPLOADを取得し、 文字列値の代わりにMojo :: Upload オブジェクトを取得します ）。 すべてが節度に優れており、魔法のような怠lazな人でさえ、Mojoliciousで実装したいすごい要素を作り出します。



HTTPリクエストのパラメーターにアクセスするために制限することが推奨される機能のリストは次のとおりです。

  • URLパスからプレースホルダー値を取得します。 

     $value = $c->stash( "name" ) # ROUTE


  • すべてのパラメーターの名前とすべての値を取得します。 

     # HASHREF:     : # -  (    ) # -     (    ) $params = $c->req->params->to_hash #  POST,  GET $params = $c->req->query_params->to_hash # GET $params = $c->req->body_params->to_hash # POST # ARRAYREF:    ,  Mojo::Upload $uploads = $c->req->uploads # UPLOAD


  • すべてのパラメーターの名前を取得します。 

     @names = @{ $c->req->params->names } # POST  GET @names = @{ $c->req->query_params->names } # GET @names = @{ $c->req->body_params->names } # POST @names = keys %{{ map { $_->name => 1 } @{ $c->req->uploads } }} # UPLOAD


  • 1つのパラメーターの最後の値を取得します。 

     $c->req->params->param( "name" ) #  POST,  GET $c->req->query_params->param( "name" ) # GET $c->req->body_params->param( "name" ) # POST $c->req->upload( "name" ) # UPLOAD


  • 1つのパラメーターのすべての値を取得します。 

     $c->req->params->every_param( "name" ) #  POST,  GET $c->req->query_params->every_param( "name" ) # GET $c->req->body_params->every_param( "name" ) # POST $c->req->every_upload( "name" ) # UPLOAD


このアプローチにより、コードの明快さと均一性が確保されます。 ただし、完全を期すために、残りの関数のリストを以下に示します。
  • 上記の関数を呼び出すための代替オプション：

    • $c->req->param



      は、 $c->req->params->param



      と同じです
    • $c->req->url->query



      は$c->req->query_params



      と同じ$c->req->query_params





  • to_hash



    代わりにto_hash



    pairs



    を使用できます。すべてのパラメーターの名前と値が順番に移動する配列への参照を返します。一方、1つの名前は複数回出現できますが、すべての値はスカラーです。
