Firebase：幻想への別れ

マーケティングは開発の世界の一部になりました。 GitHubの星の数によって、類似のソリューションのどれがよりクールであるかを決定し、ツイートの数によって、今後6か月でどのテクノロジーが開発されるかを予測できます。 このような状況では、 Live Typingで行った誇大広告の被害者になるリスクがあります。統計情報の収集、チャットの統合、データベースの選択、MVPの迅速な開発など、すべての問題を一度に解決できる聖杯のためにFirebaseを採用します。 戦闘でこのサービスに出くわしたとき、Firebaseのアイデアが現実とは大きく異なり、テクノロジーの範囲を理解することは私にとって本当の啓示であることに気付きました。 とにかく、この理解とFirebaseを適切に使用する方法を共有したいと思います。







私はずっと前にFirebaseで働きたいと思っていましたが、適切なプロジェクトを待っていました。 そして私は待った：MVPオフィス予約システム。 これはMVPであるため、バックエンドのビジネスロジックはかなり原始的です。 さらに、iOS上のモバイルアプリケーションはFirebaseに接続されます。 サービスを使用するための理想的なケースのように見えますが、実装中にいくつかの問題に直面する必要がありました。これについては後で説明します。



しかし、最初に、すべての誤解を排除したいと思います。 Firebaseを使用するために学習する必要がある2つのことを次に示します。



  1. これはバックエンドではなく、データベースです。 サーバー側のないFirebaseのアプリケーションのすばらしい例は忘れてください。これは今のところ達成不可能です。
  2. それはすべての長所と短所を持つNoSQLです。


データ自体の性質に基づいて、データを保存するソリューションを選択する必要があります。 Firebaseには欠点があります。



  • その範囲はNoSQLソリューションよりもはるかに小さいです。
  • Firebaseは、データをフェッチするときに大幅に制限し、必要に応じて同時に複数の場所にデータを書き込みます。
  • すべてのデータ構造からはほど遠く、Firebaseで作業するのが便利です。


しかし、MVPの開発を迅速に開始し、さらに多くの利点を得ることができるツールは、ダンプに捨てるのが残念です。 また、弱点はあなたに干渉する場合に修正できます。



前文



ホテルチェーンの予約システムを開発しているとします。







そのようなエンティティがあります：



  • ホテル
  • 数
  • クライアント
  • 予約


これをSQLデータベースに実装する方法は理解できます。リンク付きの4つのテーブルがあり、ポイントがあります。



これをNoSQL（Firebase）に実装する方法は？ エンティティを相互にネストしてみることができます： 



{ “”: { … “”: { “”: { ??? }, ... } } }


ここで疑問が生じ始めます：部屋のすべての予約に投資する価値はありますか？ そしてどこに顧客を投資しますか？ 等 NoSQLの問題は、多くの場合、データを複製する必要があることです。



2番目のオプションがあります。SQLと同様の方法でNoSQLを使用し、各エンティティのルートオブジェクトを作成し、他のオブジェクトのIDを保存することで接続を維持します。



おそらく、他のNoSQLデータベースでこれらの問題に対処する方が簡単ですが、Firebaseでタスクの解決策を見つけることができませんでした。



どのオプションを選択しても、同じ問題があります。データの複雑な選択ができないことです。 特定のクライアントの予約リストを取得したい場合はどうしますか？ これらの予約は異なる部屋やホテルに埋め込まれている可能性があり、構造が平坦な場合、Firebaseはいくつかのパラメーターに従ってデータをフィルター処理できません（この問題はStackOverflowでも議論されました）。 一般に、顧客と予約日で選択したい場合、Firebase SDKは役に立ちません。



バックエンドでこの問題を解決しようとすることができますが、1つのパラメーターでフィルター処理されたデータの選択をポンプアウトし、さらに自分でフィルター処理する必要があります。 これは受け入れられません。







どうする



Firebaseを複雑なデータサンプリングに使用しないでください。 Node.jsのバックエンドと、以下で説明するツールのいずれかがこれに役立ちます。



Elasticsearch







これは、 Luceneを使用し、Javaで記述されたJSON REST APIを備えた検索エンジンです。 詳細は公式ウェブサイトで読むことができ、Firebaseと組み合わせてすぐに検討を開始します。



設置



サーバーにElasticSearchを配置する必要があります（指示に従って簡単に実行できます）。 Firebaseと統合する必要がある場合、つまりFirebaseデータベースから検索インデックスを作成する必要があります。 Firebaseの公式統合を使用しました。 開始するには、リポジトリをダウンロードし、依存関係をインストールして、Firebaseのキーを構成に追加する必要があります。



このソリューションでは、いくつかの短所が見つかりました。



  1. これはNode.js上のスタンドアロンアプリケーションであり、バックエンドに関連付けるのは困難です。
  2. ElasticSearchの適切なインデックスを作成するのは簡単ではなく、Firebaseデータベースとのデータ同期は十分ではありません。
  3. フィールドタイプは自動的に割り当てられます。


このような単純な例でそれらを感じてください。 建物のリストとその説明と座標があります。 ElasticSearchでは、検索インデックスを作成する段階で、建物の地理座標を担当するフィールドを事前に宣言する必要があります。 エンジンをFirebaseと統合しても、このプロセスを制御することはできませんが、すべてのデータを単に同期して、データタイプを自動的に決定します。



ElasticSearchは無料で、その制御されたサービスにデプロイされます-これはプラスです。 しかし、同時に、事前に検討する必要がある展開とセキュリティの問題がいくつかあります。



例：Elastic Searchが外部クエリに使用するポートが開いています。 同じポートが検索インデックスの記録と管理に使用されるため、これにより脆弱性が生じます。 このような監視の結果として考えられるのは、検索インデックスの削除またはデータへのデータの入力です。 したがって、最初は、このポートはElasticSearchがインストールされている同じマシンからのクエリに対してのみ開いています。



最後に、ユーザー、ElasticSearch、およびバックエンド間の相互作用を実装する方法の問題は開発者にかかっています。



アルゴリア





SaaS検索ソリューション。 有料ですが、無料プランがあります。 価格表およびその他の詳細は、 公式ウェブサイトで見つけることができます。



Firebaseとの統合は、 公式のjsライブラリを使用して実装されます 。 インストールおよび起動プロセスの詳細はreadmeで説明されており、最初の試行でうまくいきました。



統合は次のようになります。 



 var algoliasearch = require('algoliasearch'); … var client = algoliasearch(config.algolia.applicationID, config.algolia.apiKey); //  Algolia var indexRooms = client.initIndex('rooms'); //    Algolia rooms.once('value', initInde); // rooms —  reference    Firebase function initIndex(dataSnapshot) { var objectsToIndex = []; //      Algolia var values = dataSnapshot.val(); //   snapshot'a for (var key in values) { //   room if (values.hasOwnProperty(key)) { var firebaseObject = values[key]; firebaseObject.objectID = key; // id   Algolia      Firebase objectsToIndex.push(firebaseObject); //    } } indexRooms.saveObjects(objectsToIndex, function(err, content) { if (err) { console.log('error'); return; } console.log('success'); return; }); }


その結果、Firebaseからすべての部屋オブジェクトを含む検索インデックスをアルゴリアで取得します。 インポートプロセス中に、データベース内の別のオブジェクトからホテル名を取得するなど、データを追加処理できることに注意してください。



インデックスを作成した後は、インデックス全体を更新する予定はありません。今後、Firebaseでイベントを監視して処理します。 



 rooms.on('child_added', addOrUpdateObjectRooms); rooms.on('child_changed', addOrUpdateObjectRooms); rooms.on('child_removed', removeIndexRooms);


Algoliaを使用する唯一の欠点は、SaaSの支払いが必要になることです。 しかし、MVPの場合、無料の関税で十分であり、Firebaseで大規模なプロジェクトを行うことを考える人はほとんどいません（私は願っています）。



この疑わしいマイナスとは対照的に、分析、検索インデックス、検索クエリのニュアンスにアクセスできる便利な管理パネルが用意されています。

重要なプラスは、モバイルプラットフォームからバックエンドフレームワークまで、あらゆるものに対応するSDKの存在です。 私は本質を掘り下げませんでしたが、iOS開発者は、RESTよりも便利だと言いました。



Algoliaを試してみることをお勧めします。Firebaseとの統合が優れており、インストールが簡単で、最終的には分析とSDKを備えたコンソールを入手できます。 私は技術的な詳細を無視し、パフォーマンスと速度を分析しませんでした。これは複雑で別個のトピックです。



まとめ



このかなり単純なシステムの利点は明白です。 取得するもの：



  • データ、すべての読み取り操作、および単純な非競合要求を格納するためのFirebase。
  • すべての競合クエリと複雑なビジネスロジック+ Algolia / ElasticSearchサービス用のNode.js。
  • 検索および複雑なデータ取得のためのAlgolia / ElasticSearch。


Firebaseの利点はすべてありますが、Node.jsでデータを複製したり、複雑で遅い選択を整理したりする必要があるという欠点はありません。 この状況では、予約システム、またはトランザクションと複雑なデータサンプルを必要とするその他のタスクを簡単に実装できます。 たとえば、エアコンとバルコニーを備えた2人用の特定の日の部屋を最初に選択してから予約すると、部屋がすでに占有されているか、再び占有されることを恐れることはありません。 確かに、一部のデータは複製する必要がありますが、データベースではなく検索インデックス自体にのみ含まれます。



Firebaseを適切に使用することで、データのアクセスと保存に完全に受け入れられるソリューションになります。 データはプライマリであり、間違ったデータ構造またはデータの操作方法を選択すると、重大な開発上の問題に直面することを常に覚えておいてください。
