Veeam Backup＆Replication：10初心者向けのヒント

かなり前に、私たちは主力製品について書きませんでした。 特に、Veeam Support Teamのメンバーから役に立つヒントの別の部分が到着したので、修正しています。 今日はブカレスト出身の同僚のエフゲニー・イワノフです。ルーマニアのテクニカルサポートチームのメンターとして招かれました。



作業の過程で、ユージーンと彼の仲間はまともな「レーキ」のコレクションをまとめました。これは、ほとんどの場合、初心者ユーザーがVeeam Backup＆Replicationを展開および構成するときに攻撃されます。 そして、あなたは彼らの間違いを繰り返さないで、ユージンはそれを正しくする方法を説明します。

猫へようこそ。







＃1：最適なバックアップ方法を選択する



「直接増分」または「無限増分」は、最も高速であるため、一般的に推奨されます。 無限に増分するチェーン（定期的な完全バックアップなし）は、占有するスペースが少なく、非常に迅速に処理されます。 通常のスペースのインクリメンタルチェーンはより多くの容量を必要としますが、インクリメンタルバックアップだけでなく、定期的に作成されるフルバックアップも含まれるため、いわば「復元力」があります。



リバース（リバース）インクリメンタル方式は最も古く、当然、最も低速です。 ストレージシステムによっては、他の3倍以上遅くなる場合があります。 それにもかかわらず、それを使用することにはいくつかの利点があります。チェーンの最後は常に完全バックアップであるため、他のタイプのチェーンよりも速く回復できます。 ただし、通常のインクリメンタルチェーンとの違いはそれほど重要ではありません（このようなチェーンを不当に長く、つまり30日以上保持しない限り）。



メソッドの詳細については、 ここで説明します 。



＃2：合成フルバックアップ設定について考える



合成完全バックアップ操作では、リポジトリに保存されている復元ポイントを使用します。 ただし、すべてのストレージシステムがこの操作に対して十分なパフォーマンスを提供できるわけではないことに注意する必要があります。 したがって、代わりにアクティブなフルバックアップを作成することをお勧めします。



合成完全バックアップを作成するための設定を指定する場合、「以前のバックアップチェーンをロールバックに変換する」オプションに注意してください（以前のチェーンをロールバックポイントに変換します）。 その使用により、増分バックアップ（.VIB）をロールバックポイント（.VRB）に変換するタスクが開始されます（ただし、リポジトリストレージリソースのかなりの部分を消費します）。 たとえば、このオプションを使用すると、特にアーカイブストレージ用に、現在のチェーンを逆増分に変換できます。



ただし、このオプションをバックアップ方法として使用すると、最終的に、完全バックアップファイルと増分および逆増分バックアップファイルから非常に独特なチェーンが作成されます。







＃3：ゲストOS処理を設定する



ゲストOSの処理により、仮想マシンの一貫したバックアップを作成できます。 また、VMがMicrosoft Exchange、Active Directory、SharePoint、SQL Server、Oracleなどのアプリケーションを実行している場合、Veeam Explorerツールを使用してそれらをきめ細かく復元できます。 ゲストOSの操作は、VSS機能（Windowsでサポート）に基づいており、正しく構成する必要があります。そうしないと、バックアップジョブが正常に完了できません。



ゲストOS処理設定をアクティブにするには：



  1. バックアップジョブのプロパティで、 Guest Processingステップに進みます。
  2. アプリケーション対応の処理オプションをオンにします。
  3. [ ゲストOS資格情報]セクションで、ゲストOSにアクセスするための管理者権限を持つアカウントを指定します。





  4. バックアップタスクのVMに個別のアカウントが必要な場合は、[ 資格情報 ]ボタンをクリックします。 次に、 Set User ...をクリックして、必要なデータを指定します。





  5. 特定のアプリケーションを処理するための設定は、[ アプリケーション... ]をクリックすると開く別のダイアログボックスで設定されます。 必要に応じて、個々のマシンのゲストOS処理を無効にすることもできます。







＃4：不必要にファイルのインデックスを作成しない



バックアップ設定でVMゲストファイルシステムのインデックスオプションを有効にすると、Veeam Backup＆ReplicationはVMファイルディレクトリを作成します。 これにより、Veeam Backup Enterprise Manager Webインターフェイスを介してファイル検索と1クリックのリカバリを実行できます。



Enterprise Managerを使用していない場合は、このオプションを有効にしないことをお勧めします。これにより、バックアップウィンドウが（かなり大幅に）短縮され、 C： Veeamバックアップサーバーのディスク領域が節約されます。 これは、Veeam Backup＆Replicationコンソールを介したVMファイルのリカバリには影響しません。



＃5：追加のバックアップを行う



絶対的なデータ整合性を保証するストレージベンダーはありません。 もちろん、Veeamはディスクへの書き込み時にバックアップファイルをチェックしますが、ストレージシステムで何百万もの操作が実行されているにもかかわらず、ビットのランダムな順列は依然として可能であり、結果は「目に見えない」損傷です。 Veeam Backup＆Replicationは、そのような損傷を早期に検出するためのSureBackupとヘルスチェックを提供します。 ただし、これは万能薬ではありません。したがって、バックアップにさまざまな種類のメディアを使用し、少なくとも2つのサイトに保存する必要がある3-2-1ルールを採用することをお勧めします。



これを行うには、メインバックアップジョブを作成した後、バックアップ転送ジョブを構成することをお勧めします。 このようなタスクでは、バックアップストレージまたはクラウドストレージをターゲットストレージの場所として使用できます。 バックアップを磁気テープにアーカイブすることもできます。



＃6：インスタントリカバリの完了



インスタントリカバリVMの機能インスタントVMリカバリを使用すると、バックアップから最短時間で直接マシンを起動できます。 ただし、このマシンはリポジトリでホストされ、本番環境に転送するまでリソースを消費することに注意する必要があります。 この重要な最終ステップを忘れないでください。Veeamテクニカルサポートで長年働いてきたので、実稼働環境に移行せずに数週間バックアップモードで機能していたVMで多くのケースを見てきました。 その結果は、通常はかなり嘆かわしいものでした。ストレージのオーバーフローとデータの損失です。



インスタントリカバリを適切に実行する方法の詳細については、 こちらをご覧ください 。







＃7：リポジトリの配置場所について考える



Veeamは、さまざまなリポジトリをリポジトリとしてサポートしています。 多くの場合、これにより最大のパフォーマンスが得られるため、毎年多くのユーザーがこの目的に物理的なWindowsまたはLinuxサーバーを使用することを好みます。 私たちのフォーラムでそれについて読むことができます。



CIFS共有のリポジ​​トリも、他のリポジトリと比較してパフォーマンスが最も低いにもかかわらず、非常に人気があります。



多くの最新のNASデバイスはiSCSIをサポートしているため、iSCSIドライブを構成し、それをVeeamバックアップサーバー（またはプロキシ）で使用できるようにすることをお勧めします。 このようなシナリオ（NASのリポジトリを使用）では、バックインクリメンタルバックアップ方式を使用することはお勧めできません。 読み取り/書き込みの強度により、ストレージに大きな負荷がかかります。



＃8：複製時にプロキシを使用する



WAN経由でレプリケーションを実行する場合は、リモートサイトでバックアッププロキシサーバーを構成し、レプリケーションジョブ設定で指定することをお勧めします。 これにより、2つのサイト間で信頼できるチャネルを取得できます。 レプリケーション中に仮想アプライアンス （ホットアド）モードで作業するとスナップショットが「失われる」可能性があるため、 ネットワーク （NBD）モードでこのプロキシを有効にすることをお勧めします。







WANアクセラレータを介して作業する場合は、リモートサイトでプロキシを使用することをお勧めします。 WANアクセラレータとプロキシを別のマシンまたは1つのマシンに展開できます（もちろん、十分なリソースがある場合）。







＃9：テープにアーカイブする際に重要なニュアンスを考慮する



データをテープデバイスに転送するために、Veeamは補助サーバー（テープサーバー）を使用します。 このテープデバイスが接続されている物理サーバーに配置されます。



重要！ ESXiホストを介した転送によるVMへの接続はサポートされていません！



Veeam Backup＆Replicationは、オペレーティングシステムからテープライブラリに関する情報を受信するため、最新のドライバーがインストールされていること、およびデバイスマネージャーコンソールにテープデバイスが正しく表示されていることを確認してください。



この記事では 、テープの操作に関するさらに役立つヒントを読むことができます。



＃10：とにかく何かがうまくいかなかった場合



何もする必要はありません。テクニカルサポートポータルでアプリケーションを作成する必要があります。 いくつかの簡単なことを行うことをお勧めします。



  1. アプリケーションに設定した重大度レベルを確認します-このため、関連する基準と相関させます。 すべての問題はユーザーにとって深刻であり（そうでなければユーザーから連絡がなかったはずです）、私たちの責任はできるだけ早くアプリケーションを検討することです。 ただし、レベルを重大度1に設定し、アプリケーションがこのレベルの基準を満たさない場合、貴重な時間を失う危険があります。貴重な時間は、アプリケーションを再認定して適切なキューにリダイレクトするために必要です。
  2. 問題の原因を簡単かつ迅速に見つけるには、ログを準備してください。 正しく行う方法はここに書かれています。 場合によっては、エンジニアがインフラストラクチャの他のコンポーネントのログを要求することがありますので、準備が必要です。







Veeam Backup＆Replicationを展開および構成する際に、この推奨事項が一般的な間違いを回避するのに役立つことを願っています。 今日は以上です。 Veeamサポートチーム、よろしくお願いします。
