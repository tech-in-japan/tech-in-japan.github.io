Unity3DでRTSの動的に変化するランドスケープを作成する

むかしむかし、国内の開発者KD Labsの「Perimeter：Geometry Warriors」と呼ばれる素晴らしいRTSをプレイする喜びがありました。 これは、「フレーム」と呼ばれる巨大な飛行都市が、相互に接続された世界のチェーンである「スポンジ」のオープンスペースをどのように耕すかについてのゲームです。 プロットはかなり奇妙で抽象的なものですが、ゲームのはるかに興味深い革新的なコンポーネントは、プロットではなくその技術的特徴の1つでした。 静的な地形で戦闘が行われるほとんどのRTSとは異なり、ペリメーターでは、主要なゲームメカニズムの1つがテラフォーミングでした。 プレイヤーは、その上に自分の建造物を建てるために景観を操作する手段と、この景観をひび割れ、泳ぎ、赤熱した石/厄介な昆虫ホラーに変えることができる軍隊の全兵器庫を持っています。



ご存知のとおり、RTSの世界は今や衰退しています。 独立系開発者は、レトロなプラットフォーマーや非常に複雑なルージュのようなゲームをリベットするのに忙しすぎているため、しばらく前に「境界」でプレイした後、私自身も似たようなものを実装しようと決めました-アイデアは面白かったです技術的およびゲームプレイの観点。 ゲーム開発の実務経験がある（以前XNAで何かをやろうとしていた）ので、少なくともある程度の成功を収めるには、より高レベルでシンプルなものを使用する必要があると思いました。 私が選んだのはUnity 3Dでした。Unity3Dの5番目のバージョンは報道されていません。

最初の試行の結果の分析



Tech Demoを見ると、変形メカニズムに特定の問題があることにすぐ気付くでしょう。 あなたがそれを見なかったなら（私はあなたを責めない）、私はあなたに何が間違っていたか教えます。 主な問題はパフォーマンスでした。 より正確には、その完全な不在。 変形が始まると、フレームレートは非常に小さな値になりました（一部のマシンでは一意）。これは、ゲーム内にグラフィックスが本質的になかったため、受け入れられませんでした。 SetHeights（）メソッド自体がランドスケープに対して非常に複雑な一連のLOD計算を引き起こすため、リアルタイムの地形変形には適していないことがわかりました。 私の希望は崩壊し、Unityでのリアルタイムデフォメーションの実装は不可能に思えますが、私はあきらめずに、LOD再計算メカニズムの明らかだが非常に重要な機能を見つけました。



地形標高マップの解像度が低いほど、SetHeights（）を使用したときのパフォーマンスへの影響は小さくなります。



高さマップの解像度は、ランドスケープ表示の品質を特徴付けるパラメーターです。 （明らかに）整数であるため、上のスニペットで整数を使用して地図上の座標を示しています。 また、たとえば256x256のランドスケープの場合、ランドスケープのサイズより大きくすることができます。これにより、高さマップの解像度を513に設定できます。これにより、ランドスケープの精度と角度のない輪郭が得られます。 なぜ512ではなく513なのか、次のセクションで説明します。



高さマップの解像度を持つゲームでは、構成に最適なサイズを多少見つけることができましたが、結果には非常に失望しました。 このようなランドスケープをRTSで正常に適用するには、少なくとも2人のプレーヤーがしばらくの間共存できるように、そのサイズが十分に大きくなければなりません。 私の最初の見積もりによると、2x2km（または2048x2048 Unity Units）のカードがちょうどよかったはずです。 リアルタイムで変形のフレームレートへの影響に気付かないために、ランドスケープのサイズは512x512ユニット以下でなければなりません。 さらに、高さマップの単一の精度は、視覚的な品質に関しては最も印象的な結果ではありませんでした。 風景は角張っており、場所によって曲がっていたため、高さマップの精度を2倍にする必要がありました。



一般的に、物事はあまり良くありませんでした。



スーパーテレイン-コンセプトと理論 



  :         Super Terrain.        .


その後、次のような考えが私を訪ね始めました。「1つの大きな景観を作ることはできず、変形のパフォーマンスがまだ十分にあるので、小さなものをたくさん作って並べてみませんか？ 「Minecraftのチャンクはどうですか？」という考えは悪くありませんでしたが、いくつかの問題がありました。



  • 「チャンク」のサイズを選択する方法
  • チャンクのジョイントに目立つ継ぎ目がないことを確認する方法
  • 隣接するchunk'ovのジョイントで発生する変形を適用する方法


最初の問題


最初の問題はかなり些細なものでした。256x256のチャンクのサイズを倍精度で選択しました（高さマップの解像度= 513）。 このセットアップは、マシンのパフォーマンスの問題を引き起こしませんでした。 おそらく将来的にはチャンクのサイズを再検討する必要がありますが、現在の段階では、このようなソリューションは私に適しています。



第二の問題


2番目の問題に関しては、2つのコンポーネントがありました。 最初の方法は、明らかに、隣接するチャンクの高さマップの隣接する「ピクセル」の高さを揃えることです。 この問題の解決中に、高さマップの解像度が2のべき乗+ 1である理由を理解しました。図で説明します。







明らかに、隣接する風景の高さの平等を維持するために、最初の風景の高さマップの「最後の」ピクセルは、次の「最初の」ピクセルと高さが等しくなければなりません。







明らかに、「スーパーテレイン」-これは、Unity Terrain'ovのマトリックスであり、ハイトマップと変形を適用するメカニズムによって結合されています。



ランドスケープを結合するためのコードの実装が完了した後（小さなサイズのローカルデフォメーションの使用は後で残されました-テレインマトリックスを作成し、標高マップの初期初期化のためのメカニズムを開発する必要がありました） ）、幸いなことに簡単に解決されました。 問題は、そのような風景の「組み合わせ」のために、それらが共有され、隣接していることをUnityに説明する必要があることでした。 これを行うために、開発者はSetHeighborsメソッドを提供しました 。 私はまだそれがどのように機能するかをよく理解していませんが、それなしでは、風景の交差点に影と暗い縞模様のアーティファクトが現れます。



第三の問題


3つの中で最も面白くて困難なこの問題は、少なくとも1週間は休息を与えませんでした。 最後の実装に至るまで、4つの異なる実装を削除しました。これについて説明します。 すぐに、実装の1つの重要な制限について簡単に説明します。ローカル変形は1チャンクより大きくできないことを前提としています。 変形はまだ接合部にある可能性がありますが、変形マトリックスの側面はチャンクの高さマップの解像度を超えてはならず、それらはすべて正方形でなければなりません（高さマップ、チャンク、歪み自体）。 一般に、これは大きな制限ではありません。大きな変形は、いくつかの小さな変形を順番に適用することで取得できるためです。 「直角度」に関しては、これは高さマップの制限です。 つまり、その高さマップのみが正方形である必要があり、その上に加算アプリケーションの「ゼロ」セクションまたは乗算アプリケーションの「単一」セクションが存在する場合があります。



変形自体の普遍的な適用のためのアルゴリズムのアイデアは次のとおりでした：



  1. デフォメーションハイトマップを9つの部分に分割します。各部分は、デフォメーションが影響する可能性があるチャンクごとに1つです。 そのため、中央部分は、変形によって直接ヒットされるチャンクの変形を担当し、当事者は、チャンクの左、右、または上/下などを担当します。 変形によってチャンクが変更されない場合、そのコンポーネントはnullに等しくなります。
  2. 部分的な高さマップを対応するチャンクに適用するか、部分的な高さマップがnullの場合、変更を無視します。


このアプローチにより、チャンクの中心に位置し、他のチャンクに影響を与えない変形、およびそれらの境界またはコーナーにある変形のユニバーサルアプリケーションが可能になります。 変形がチャンク境界で開始または終了する場合、それに隣接するピクセルの境界ピクセルも変更する必要があるため、正確に9つの部分（4つではなく）に分割する必要があります。 （目に見える継ぎ目がないように-問題番号2の解決策を参照してください）。



スーパーテレイン-練習



SuperTerrainを作成


2番目の問題の一部として、複数のTerrain'ovを1つに結合し、ランドスケープ全体に適合する「グローバル」高さマップを適用できるメサニズムが開発されました。



高さマップをグローバルに使用する可能性が必要だったのは、ランドスケープを作成するための手続き型であり、その使用にSquare-Diamondアルゴリズムが使用されたためです。



一般的に、SuperTerrainの作成は、かなりシンプルで直感的なプロセスです。



こっち 
 /// <summary> /// Compound terrain object. /// </summary> public class SuperTerrain { /// <summary> /// Contains the array of subterrain objects /// </summary> private Terrain[,] subterrains; /// <summary> /// Superterrain detail. The resulting superterrain is 2^detail terrains. /// </summary> /// <value>The detail.</value> public int Detail { get; private set; } /// <summary> /// Parent gameobject to nest created terrains into. /// </summary> /// <value>The parent.</value> public Transform Parent { get; private set; } /// <summary> /// Builds the new terrain object. /// </summary> /// <returns>The new terrain.</returns> private Terrain BuildNewTerrain() { // Using this divisor because of internal workings of the engine. // The resulting terrain is still going to be subterrain size. var divisor = GameplayConstants.SuperTerrainHeightmapResolution / GameplayConstants.SubterrainSize * 2; var terrainData = new TerrainData { size = new Vector3 (GameplayConstants.SubterrainSize / divisor, GameplayConstants.WorldHeight, GameplayConstants.SubterrainSize / divisor), heightmapResolution = GameplayConstants.SuperTerrainHeightmapResolution }; var newTerrain = Terrain.CreateTerrainGameObject(terrainData).GetComponent<Terrain>(); newTerrain.transform.parent = Parent; newTerrain.transform.gameObject.layer = GameplayConstants.TerrainLayer; newTerrain.heightmapPixelError = GameplayConstants.SuperTerrainPixelError; return newTerrain; } /// <summary> /// Initializes the terrain array and moves the terrain transforms to match their position in the array. /// </summary> private void InitializeTerrainArray() { subterrains = new Terrain[Detail, Detail]; for (int x = 0; x < Detail; x++) { for (int y = 0; y < Detail; y++) { subterrains[y, x] = BuildNewTerrain(); subterrains[y, x].transform.Translate(new Vector3(x * GameplayConstants.SubterrainSize, 0f, y * GameplayConstants.SubterrainSize)); } } } /// <summary> /// Initializes a new instance of the <see cref="SuperTerrain"/> class. /// </summary> /// <param name="detail">Superterrain detail. The resultsing superterrain is 2^detail terrains.</param> /// <param name="parent">Parent gameobject to nest created terrains into.</param> public SuperTerrain(int detail, Transform parent) { Detail = detail; Parent = parent; InitializeTerrainArray(); SetNeighbors(); } /// <summary> /// Iterates through the terrain object and sets the neightbours to match LOD settings. /// </summary> private void SetNeighbors() { ForEachSubterrain ((x, y, subterrain) => { subterrain.SetNeighbors(SafeGetTerrain(x - 1, y), SafeGetTerrain(x, y + 1), SafeGetTerrain(x + 1, y), SafeGetTerrain(x, y - 1)); }); } #region [ Array Helpers ] /// <summary> /// Safely retrieves the terrain object from the array. /// </summary> /// <param name="x">The x coordinate.</param> /// <param name="y">The y coordinate.</param> private Terrain SafeGetTerrain(int x, int y) { if (x < 0 || y < 0 || x >= Detail || y >= Detail) return null; return subterrains[y, x]; } /// <summary> /// Iterates over terrain object and executes the given action /// </summary> /// <param name="lambda">Lambda.</param> private void ForEachSubterrain(Action<int, int, Terrain> lambda) { for (int x = 0; x < Detail; x++) { for (int y = 0; y < Detail; y++) { lambda (x, y, SafeGetTerrain(x, y)); } } } #endregion }




実際、ランドスケープの作成は、InitializeTerrainArray（）メソッドで行われます。このメソッドは、テレイン配列を新しいインスタンスで満たし、ゲームワールドの適切な場所に移動します。 BuildNewTerrain（）メソッドは、次のインスタンスを作成し、必要なパラメーターで初期化し、GameObject'a内に「親」を配置します（不必要なゲームでインスペクターを汚染しないように、SuperTerrainチャンクを含むシーンでゲームオブジェクトが事前に作成されることを想定しています）オブジェクトと必要に応じてクリーンアップを簡素化します。）



ここでは、ランドスケープの境界にある黒いストライプの問題の1つであるSetNeighbors（）メソッドが処理されます。このメソッドは、作成されたランドスケープを反復処理し、近隣に配置します。 重要： TerrainData.SetNeighbors（）メソッドは、グループ内のすべての風景に適用する必要があります。 つまり、ランドスケープAがランドスケープBの上の隣人であることを示した場合、ランドスケープBがランドスケープAの下の隣人であることも示す必要があります。この冗長性は完全に明確ではありませんが、この場合のように、メソッドの反復適用が大幅に簡素化されます。



上記のコードには、次のランドスケープを作成するときに除数を使用するなど、いくつかの興味深い点があります。 正直に言うと、なぜこれが必要なのかわかりません-通常の方法で（除数なしで）ランドスケープを作成すると、間違ったサイズのランドスケープが作成されます（これはバグの可能性があります。 この修正は経験的に得られたものであり、まだ失敗していないので、そのままにしておくことにしました。



また、リストの下部に2つの不審なヘルパーメソッドがあることに気付くかもしれません。 実際、これはリファクタリングの結果にすぎません（いくつかのリファクタリングが行われたがまだ完全ではない、多少安定したバージョンをリストしているため）。 これらの方法は、ローカルおよびグローバル変形を適用するときにさらに使用されます。 彼らの名前から、彼らが何をしているのか簡単に推測できます。

私は同意します、このメソッドのペアはあまり良く見えませんが、すべてを説明しようとします。 したがって、SetGlobalHeightmapメソッドの名前はそれ自体を表しています。 彼が行うことは、すべてのチャンク（ここでは地下と呼ばれます）を反復処理し、その座標に対応する高さマップのその部分に正確に適用することです。 ここでは、運の悪いSetHeightsが使用されます。そのパフォーマンスにより、これらすべての倒錯に追い込まれます。 コードからわかるように、SuperTerrainHeightmapResolution定数は、高さマップの1解像度と2のべき乗（前のセクションで存在が正当化されている）の違いを考慮していません。 その名前と混同しないでください-この定数は、SuperTerrain全体ではなく、チャンクの高さマップの解像度を格納します。 SuperTerrainコードはさまざまな定数を広範囲に使用するため、GameplayConstantsクラスをすぐに紹介します。 おそらく、何が起こっているかがより明確になるでしょう。 このクラスからSuperTerrainに関係のないものをすべて削除しました。

 namespace Habitat.Game { /// <summary> /// Contains the gameplay constants. /// </summary> public static class GameplayConstants { /// <summary> /// The height of the world. GetSubHeightMapメソッドに関しては、これは転送されたマトリックスの一部の一部をマイナーマトリックスにコピーする別のヘルパーです。 これは、SetHeightsがマトリックスの一部を適用できないためです。 この制限により、大量の追加メモリ割り当てが発生しますが、それについては何もできません。 残念ながら、Unity開発者はリアルタイムのランドスケープ変更シナリオを提供しませんでした。

おそらく既に推測したように、リストにある唯一のパブリックメソッドが最も重要なメソッドです。 ApplyDeformation（）メソッドを使用すると、指定した変形を指定した座標の地形に適用できます。まず、呼び出されると、地形の座標が標高マップの座標に変換されます（地形の寸法が標高マップの解像度と異なる場合、これを考慮する必要があります）。変形の適用に関するすべての作業は、変形から対応するチャンクに高さマップのチャンクを適用する9つのApplyPartialHeightmap呼び出し内で行われます。前述したように、可能性のあるすべての境界および角度の場合を考慮するために、4つではなく、正確に9つの部分が必要です。







GetXXXSubmap（）メソッドが関与するのはこの区分です-さまざまなチャンクの変形位置と境界のデータに基づいて、必要な変形マイナーを取得します。変形が対応するチャンクに影響を及ぼさず、これらの同じマイナーを適用するメソッド（ApplyPartialHeightmap（））が入力でnullを受け取った場合、各メソッドはnullを返します。



結果と結論



結果として生じるメカニズムはもちろん理想とはほど遠いですが、すでに機能しており、パフォーマンス設定に関してある程度の柔軟性を実現するために重要な地形パラメーターを調整することができます。主な改善点には次のものがあります。



  • ハードワークは別のプロセスで行われ、特に激しいシーンでのフレームレートへの影響を軽減します。
  • キャッシングなどにより、毎回メモリ割り当てを取り除くことにより、マイナー化を最適化します。これまでのところ、そのようなものをキャッシュする方法を想像するのは困難です。まず第一に、最も頻繁なケースに限定することができます-チャンクの真ん中のわずかな変形。
  • ランドスケープジオメトリだけでなく、そのテクスチャ（スプラットマップの変更による変形）にも影響を与える機能を追加します。
  • 単一のフレームに複数の設定を適用するための最適化。たとえば、いくつかのバッファにチャンクのデフォメーションを蓄積し、ロジックを何らかの方法で結合して適用するロジックの処理の最後に、複数のデフォメーションがあったとしても、チャンクでSetHeightsを1回呼び出します。
