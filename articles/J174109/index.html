gSOAPを使用してC ++でSOAPクライアントを作成する

そのため、ハブ上でgSOAPを使用することは非常に不十分にしか説明されていません。 正直に言うと、たった1つの投稿です。 しかし、Webサービスの作成について説明していますが、クライアントアプリケーションについてはどうですか？ 少し前まで、SOAPを使用してリモートサーバーとの作業を整理する作業に直面していました。これについて短い記事を書くことにしました。

なぜなら 私が働いていたwsdlファイル（NDAなど）を提供できなかったため、テストに適したサービスを探し始めました。 2つは私にとって面白そうだった：



http://www.webservicex.net/ValidateEmail.asmx?WSDL

http://www.webservicex.net/country.asmx?WSDL



それでもwsdlファイルをダウンロードする場所が見つからなかったので、それらのコンテンツをコピーし、ValidateEmail.wsdlおよびcountry.wsdlという名前で保存しました

gSOAPはhttp://www.cs.fsu.edu/~engelen/soap.htmlからダウンロードできます。 同じアドレスでgSOAPについて読むことができます。



執筆時点での最新バージョンは2.8.14です。



仕事に取り掛かりましょう。 2つの非常に重要なユーティリティがgsoap \ bin \ win32 \フォルダーにあります。 まず、wsdl2h.exeに興味があります。 ヘルプを使用して、詳細を確認できます。

> wsdlh2.exe -h

次のパラメーターを使用して実行します。

wsdl2h.exe -o emailAndCountry.h ValidateEmail.wsdl country.wsdl

すべてが非常に単純です。出力ファイルの名前と、処理するwsdlファイルのリストのみを指定しました。

その後、C ++クラスのコードを直接生成します。

soapcpp2.exe -C -dgSoap -j -L -x -I "ADDRESS_TO_GSOAP \ gsoap-2.8 \ gsoap \ import" emailAndCountry.h

-Cスイッチは、クライアントコードのみを生成する必要があることを示します。

-dgSoapは、すべてのファイルをgSoapフォルダーに入れるように要求します（最初に作成する必要があります）。 libファイルを生成せず、soap構造から継承しません。 -xスイッチは、サンプルメッセージを含むXMLファイルを生成しないように要求します。 gSOAPを使用してフォルダーのアドレスを指定し、解析するファイルを指定して、そのファイルに基づいてコードを生成します。

異なるバージョンのgSOAPは異なるコード（一般に論理的）を生成でき、ファイルの構成も異なります。 これは、ビルドサーバー上でgSOAPファイルを突然生成し、バージョン管理システムに保存しない場合に覚えておくことが重要です。

gSoapフォルダーでのすべての操作の後、多くの新しいファイルが表示されます。 これらは、-hおよび-cppファイルに加えて、countrySoap.nsmapおよびValidateEmailSoap.nsmapです。 それらは一致し、その内容を1つ（たとえば、namespaces.nsmap）に保存して削除できます。 namespaces.nsmapをプロジェクトに含める必要があります。 通常、これはgSOAPで動作する何らかの種類のヘルパークラスで行われます。 はい、そのようなクラスは確かに存在します。



その後、stdsoap2.hとstdsoap2.cppをgSoapフォルダーに追加します-これらはsoapStub.hに含まれます



フォルダー全体をプロジェクトに追加し、作業を開始します:)

クラスsoapcountrySoapProxy.cppおよびsoapValidateEmailSoapProxy.cppを見てください。 in * _initメソッド（soap_mode imode、soap_mode omode）-名前空間を削除します（namespaces.nsmapを意図的に含めました）。



電子メール検証機能から始めましょう-gSoap / soapValidateEmailSoapProxy.h

このアプローチは、特定のアドレスにあり、承認を必要としない1つのサービスのみを使用する場合に最適です。

しかし、いつでもどこでもWebサービスをデプロイできます！

実際、gSoapは多くのことができます。 したがって、たとえば、soap構造を介してログインパスワードを設定できます。 最も単純な基本認証の例： 

 soap.userid = login; soap.passwd =password;




soap構造の説明はstdsoap2.hにあります。 

  const char *userid; /* HTTP Basic authorization userid */ const char *passwd; /* HTTP Basic authorization passwd */




リクエストの送信先アドレスは、2つの方法で変更できます。リクエスト自体で指定するか、SOAPプロキシを作成するときに指定します。

これはすべて読者自身から明らかです。 

 virtual int GetISD(_ns2__GetISD *ns2__GetISD, _ns2__GetISDResponse *ns2__GetISDResponse) { return this->GetISD(NULL, NULL, ns2__GetISD, ns2__GetISDResponse); } virtual int GetISD(const char *endpoint, const char *soap_action, _ns2__GetISD *ns2__GetISD, _ns2__GetISDResponse *ns2__GetISDResponse);




2番目の方法の場合、次のようなオブジェクトを作成する必要があります。 

 countrySoapProxy countryProxy("http://www.webservicex.net/country.asmx");




複雑すぎることはありませんか？ ここで、ポートで通常のIPアドレスを設定できます。



また、自分で名前空間を設定する方法も考慮しませんでした。 このアプローチにより、ns1__IsValidEmailをクラス名としてではなく、email__IsValidEmailのようなものとして受信できます。 多くのクラスがある場合、混乱する可能性があります。 おそらく。 これも問題なく行われます。 次の形式の内容でファイルtypeMap.datを作成します。

myCustomNamespace = " www.webservicex.net "

つまり 簡単です。名前空間とそのアドレスを指定します。



一般に、gSoapの操作は特に難しくありません。 しかし、それを操作するときに浮かぶいくつかのポイントがあります。 そのため、たとえば、ユーザー名とパスワードを指定する場合、リクエスト後に変更できます。 つまり コードは次のようになります。 

 countrySoapProxy countryProxy("http://www.webservicex.net/country.asmx"); countryProxy.soap->userid = "login"; countryProxy.soap->passwd = "password"; countryProxy.GetISD(&getISDRequest, &getISDResponse); countryProxy.soap->userid = "login"; countryProxy.soap->passwd = "password"; countryProxy.GetISD(&anotherGetISDRequest, &anotherGetISDResponse);




どうやら疑わしいようです。

たぶん私は何かを考慮しなかったかもしれませんが、gSOAPの短い経験は複雑な印象を残します：開発速度が速く、コードの量が比較的少ない-これはすべてプラスです。 しかし、これはごちゃ混ぜのコードです。CとC ++の混合物はマイナスです。 しかし、私はこれがすべての人にとってマイナスではないことを認めていますが、さらに多くのプラスがあります。 いずれにせよ、gSOAPに代わる価値のある代替品は見つかりませんでしたが、私たちの会社でテストされたソリューションを信頼して、あまり慎重に検索しませんでした。

この記事は、gSOAPの使用を開始することについて簡単に説明した入門記事です。 ここではさまざまな認証方法を提供しませんでした。SOAP構造からの継承の問題については調査しませんでした（これは何かに必要ですか？）。 多くの未解決の質問がありますが、それでも私はgSOAPについて簡単に説明しました。 良い仕事をしてください！
