2回目の API ＆Clean Code 会議 （4月25日）には、プログラムを行うすべての人に影響を与える幅広いトピックがあります。 なんで？



私たち全員がAPIを使用しているため、パブリックアプリケーションコントラクトです。 これは.Net Frameworkそのものであり、私たちが使用する膨大な数のライブラリです。 一部のライブラリが他のライブラリよりも人気がある理由を疑問に思ったことはありませんか？ 彼らはどのように前面に来ますか？ 操作の速度、完全性、および絶望は重要ですが、使いやすさも重要な問題になります。



世界には毎日、新しいユーザーサービス、アプリケーションが増えているためです。 それらはすべて自己完結型ではありません-それらは、公共の契約を通じて互いに密接に関連しています。 多くのサービスは、グラフィカルシェルを提供せず、異なるAPIのセットのみを提供します。



私たちの意見では、サービスでの作業の成功と実りを決定するのはAPIです 。 しかし、ユーザーと開発者の両方にとって便利なように、そのような契約をどのように構築するのでしょうか？ APIの使用を明確かつ安全にする方法は？ APIを管理する必要がありますか？ 長寿命で人気のあるサービスを作成したり、マイクロサービスアーキテクチャを使用したり、フレームワークを作成したりする場合、何千もの質問が生じます。 コードの外部の美しさに加えて、内部の順序があるはずです-きれいなコード、これは会議で議論するための追加のトピックにもなります。 たとえば、興味深いストーリーを持つFxCopの第一人者を見つけるか、同様の機能を持つ他のツールについて議論します。 実践的な解決策である自己規律だけが、創造的な人々、つまりプログラマーに苦しむことがよくあります。 =）



会議で提起された問題は、開発者のすべてのグループにとって興味深いものであり、ウェブサイト、ポータル、サービス、フレームワーク、企業アプリケーション、デスクトップアプリケーションを作成するかどうかは関係ありません。 アプリケーションがネットワークと通信しない場合でも、作業する内部APIがあり、外部アプリケーションのルールに従ってそれを実行しようとします。 痛みを伝えたり共有したりすることがある場合は、レポートをリクエストしてください。



レポートの主題：

  • API（REST、SOAP、RPC）を作成するための最新のアプローチの使用と比較。
  • セキュリティの問題：さまざまなユーザーグループに対して認可され、アクセス可能なAPI。
  • バージョン管理と利用可能なAPIの一般管理のサポート。
  • 明白でシンプルなAPIの作成：これを実現する方法、どのプラクティスとツールが役立つか。
  • ソフトウェアツールを使用して、チーム内のコードの品質を制御することは可能ですか？ FxCopおよびその他のプログラムの機能の使用と拡張。
