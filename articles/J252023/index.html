AzureでTeamCityをセットアップする-クラウドでチームワークを行うための常時利用可能なシステム

この記事では、Kaspersky Labの友人が開始したサイクルを継続し、Microsoftのテストツールを使用した実際の経験（およびそれだけではありません）を推奨事項と結論とともに説明します。 著者は、テストエンジニアであるIgor Shcheglovitov（@ the ins4n3 on the hub）です。 テスト記事には#mstestingのタグが付いています。


この記事では、Azureの仮想マシン上で、人気のあるTeamCityシステムを使用して継続的インテグレーションプロセスを設定した最初の経験を共有したいと思います。 私たちのシステムはいつでも便利にアクセスでき、リソースを制御できるため、これは良いことです。







以下のアクションについて説明します。

  • TeamCityサーバーの構成。
  • クラウドエージェントのイメージを作成します。
  • ローカルgitのセットアップ。
  • AzureでWebサイトを作成する
  • ビルド設定
  • クラウドエージェントの追加と実行
  • 作業チェック




順番に始めましょう。



TeamCityサーバーの構成





TeamCity-serverを新しい仮想マシンに展開します。



Azureで新しいVMを作成します。







ポート80を開きます（TeamCityサーバーで使用されます）。







Azureコンソール経由でVMに新しいディスクを追加します（TeamCityサーバーを使用してデータを保存します）。







RDPを介して作成されたVMに接続し、 TeamCityディストリビューションをダウンロードします







ディスクの管理を開き、新しいディスクを初期化します。



サーバーコンポーネントのみを選択するプロセスで、TeamCityをインストールします。







次に、TeamCityデータディレクトリとして、新しいディスクを指定します。







TeamCityサーバーのポートを設定します（80番目のポートを使用します）







そして、TeamCityサービスが動作するユーザーを選択します（システムを指定しました）







インストール後、 http：// localhost / mntを開き、自動サーバー構成のプロセスを開始します。

[続行]ボタンをクリックした後、TeamCityデータベースのタイプを選択します（この例では、内部データベースを使用します）。







次に、データベースの初期化が開始されます。その後、TeamCityサーバー管理者を作成する必要があります。







FWでポート80を開きます。



クラウドエージェントイメージの作成





JetBrainsは、少し前に、AzureでTeamCityクラウドエージェントを自動的に作成して実行できるプラグインを発表しました。 このプラグインを使用してエージェントが起動すると、新しい仮想マシンが作成されます。 いつでもクラウドエージェントを停止できます。これにより、リソースが解放され、それに応じてコストが削減されます:)クラウドエージェントを使用します。 クラウドエージェントは、仮想マシンの事前に準備されたイメージに基づいて作成されます。



だから：

Azureで新しい仮想マシンを作成します。 Visual Studio 2013で作成されるASP.NETサイトのビルドを作成し、MsTestを使用してテストを実行します。 他のツールも使用できますが。 これは重要ではありません。 エージェントにVisual Studioをインストールする必要があります（Community 2013をインストールしました。無料のサブスクリプションをお持ちの場合、仮想マシンギャラリーでは、スタジオがプレインストールされたイメージが見つからないため、手動でインストールする必要があります）。











ポート9090も開く必要があります。これは、エージェントがリッスンするデフォルトのポートです。







RDPを介して、作成されたVMに接続します。 Git for Windowsをダウンロードしてインストールします 。 インストール中に、Windowsコマンドプロンプトの[Gitを使用する]チェックボックスをオンにします。







TeamCityサーバーのポートを指定します







エージェントパラメータを設定します。

serverUrl -TeamCityサーバーのアドレス

workDir および

tempDirはデフォルトで残すことができますが、これらの変数には通常、VMで特別なディスク（一時ストレージ）を使用します。







FWまたは次のコマンドを使用して、着信接続用にポート9090を開きます。

netsh advfirewall firewall add rule name = " TeamCityAgent In" dir = in action = allow protocol = TCP localport = 9090



クラウドエージェントを作成した後（VMが起動した瞬間から）、初期化段階でハングし、TeamCityサーバーに接続しません。 おそらく、これはJetBrainsプラグイン自体の技術的な実装のバグであるか、まだ問題がある可能性があります。 ただし、RDPを介してエージェントに接続し、エージェントサービスを手動で再起動すると、エージェントは機能します。 このプロセスを自動化することにしました。 これを非常に簡単に行う方法の例を次に示します。

ローカルグループポリシーエディタースナップイン（gpedit.msc）を開きます

新しいスタートアップPowerShellスクリプトを追加する



get-service "TeamCity Build Agent" | restart-service -PassThru







次に、ポータルを介してVMを停止し、それに基づいて仮想マシンのイメージを作成する必要があります。







その後、VMを削除します。



ローカルGITを構成する





GitHubで新しいリポジトリを作成します。











それにリンクをコピーします：







ローカルコンピューターにGITをインストールします（そこからチェックが生成されます）。 コードリポジトリのローカルコピーとして機能する新しいフォルダーを作成します。 その後、ローカルフォルダーをGitHubに接続します。コマンドラインを使用して、作成したフォルダーに移動し、git clone {gitRepositoryUrl}コマンドを実行します。







企業ネットワークから作業している場合、プロキシの問題が原因でクローンコマンドが失敗することがあります。 企業のプロキシリクエストをリダイレクトするローカルNTLMプロキシサーバーをインストールするだけで、この通信の問題を構成および回避することができました。 プロキシ配布キットはここにあります 。

構成が正常に完了したら、Gitのプロキシを構成する必要があります。これを行うには、次のコマンドを実行します。



git config --global http.proxy localhost ：{cntlm-proxy-port}

git config --global https.proxy localhost ：{cntlm-proxy-port}



プロジェクトフォルダー（GIT）でリポジトリを複製した後、Visual Studioで新しいソリューションを作成する必要があります（ASP.NET Webフォームプロジェクト+ UnitTestProject）

スタジオでコミットとプッシュの変更を実行します。







その後、新しいチェックインがGitHubに表示されます。







注：コマンドラインでgitコマンドを使用して変更をコミットすることもできます

git add。 （Gitリポジトリにファイルを追加します）

git commit –am「コミットのテスト」（変更のコミット）

git push {git-url}（変更をローカルストレージからGitHubにプッシュ）



AzureでWebサイトを構築する





Azureコンソールから新しい空のWebサイトを作成する







サイトを作成したら、[バージョン管理システムで展開を構成]リンクをクリックし、GitHubリポジトリを新しいサイトにリンクします。

サイトの設定を開きます。 [設定]タブでは、次のことに興味があります。

URL git アドレス

展開を開始する URL アドレス







最初のURLでdeployをcifromgittestsite.gitに置き換え、受け取ったリンクをgit pushコマンドに渡すと、Webサイトのデプロイメントを起動するコマンドが取得されます。



git.exe push -f address



次に、GitHubからの自動展開をオフにします。 これが行われない場合、各チェックはサイトの自動展開につながります。 そして、私たちの目標はCIを実装することです。コードが正常にアセンブルされ、ユニットテストが実行された場合にのみ展開を実行します。



ビルド設定





TeamCity UI（http：// TeamCityServerDns）を開き、新しいプロジェクトを作成します（プロジェクトの作成）。







プロジェクト名を設定します。 新しいビルド構成を追加する







構成名を指定します







VCS（バージョン管理システム）を構成します。 ここで、VCS（Git）タイプ、名前、フェッチURL（リポジトリのURLのみを読み取り、リポジトリアクセスリンクからgitのhttpsまたはhttpプロトコルに注意してください）を入力し、ブランチ（マスターのみを作成しました）を指定し、パスをgitに設定する必要があります。エージェントのexe







構成後、TestConnectionをクリックして、GitHubリポジトリーへの接続を確認できます

CheckOutオプションでVCSを正常に構成した後、VCSチェックアウトモード-エージェントで自動的に指定します。

[ビルドステップ]を選択し、ビルドのステップを追加（ビルドステップの追加）します。







最初のステップは、ソリウシンを組み立てることです。 ここでは、画面上のようにすべてを入力する必要があります。







2番目のステップは、テストを実行することです。







3番目の手順は展開です（展開チームはサイトのセットアッププロセスで形成されました）







次のようなものが得られるはずです。







新しいトリガーを作成します。 ここでは、チェックインごとにビルドをトリガーするボックスをオンにします。



クラウドエージェントの追加と起動





TeamCityサーバーを使用して、RDP経由で仮想マシンに接続します。 クラウドエージェントを管理するためのプラグインをダウンロードします 。

プラグインをF：\ pluginsフォルダー（FドライブはTeamCityDataDirectory）に入れ、TeamCity Serverサービスを再起動します。







TeamCity UIを開きます（メニュー[管理] => [プラグインリスト]）。

新しいAzure 統合プラグインがプラグインのリストに表示されるはずです







次に、Azureコンソールを介して新しい空のクラウドサービスを作成します。これは、TeamCityエージェントインスタンスを格納するためのリポジトリとして機能します。







[管理] => [グローバル設定]に移動し、正しい完全なTeamCity DNSサーバーを記述します。







新しいクラウドエージェントプロファイルの作成に移りましょう。管理=>エージェントクラウド=>新しいプロファイルの作成







ここで、Azureサブスクリプションからプロファイル名、クラウドタイプ-Azure、管理証明書、およびSubscriptionIdを設定する必要があります（このデータはhttps://manage.windowsazure.com/publishsettingsからダウンロードできます ）







メインプロファイル設定を入力したら、[イメージの追加]をクリックして、クラウドエージェント仮想マシンのイメージの名前、空のクラウドサービスの名前、エージェントの最大数、エージェントのサイズ、作成されたエージェントの名前のプレフィックスを指定します







ご注意 エージェントインスタンスの数 は、仮想エージェントのクラウドイメージ上の ファイアウォール で 開いている ポート の数に依存します 。

例 ： デフォルトのポートがエージェントに指定されている場合、つまり 9090、 ファイアウォール でポート9090のみを開い た 場合 、作成できるエージェントは1つだけですが、イメージに9090-9099の範囲がある場合、 10個のエージェントを作成できます。



設定を保存すると、TeamCityは新しいクラウドエージェントを自動的に起動します。







エージェント管理メニューエージェント=>クラウドエージェント、エージェントを管理できます。







エージェントは、起動時に仮想マシンを自動的に作成します。







作業確認





すべてが作成および構成されます。 これで、ビルドのチェックを開始できます。

これを行うには、「プロジェクト」メニューで「実行」をクリックします







ビルドのログを見ると、コードがGitHubからダウンロードされ、ビルドされ、ユニットテストが開始されます。すべてが成功すると、デプロイが開始されます。

ビルドトリガー、つまり githubに新しい変更をコミットすると、







その後、ビルドは数分で自動的に開始されます-ビルドは自動的にビルドされ、テストが実行され、Azureのサイトが自動的にシャットダウンされます
