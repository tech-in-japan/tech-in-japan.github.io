プログラムのデバッグ中に時間を節約するための11のヒント

プログラミングは、コードを書いて実行し、その完璧なパフォーマンスを満足して観察するときだけではありません。 アプリケーションの効果的なデバッグの必要性がありますが、これは完全な技術です！ この記事では、ネイティブコードのデバッグに役立つヒントのリストを提供します。

















元の記事はBartlomiej Filipekによって書かれ、彼のブログに公開されました。



エイズ



すべてのプログラマーは、デバッガーの起動、ブレークポイントの設定、コード実行の再開、関数の開始と終了（キーボードを使用）の方法を知っている必要があります。 基本的なトリックに基づいてデバッグプロセスを簡単にするための簡単なヒントを次に示します。



1.エラーの場所情報をデバッガーレポート（LinePos）に追加します



スキルレベルに関係なく、おそらく主なデバッグ方法の1つを引き続き使用します。関数とマクロprintf 、 TRACE 、 outputDebugStringなどを使用して特定の値を追跡します。 デバッガーレポートを確認します。 Visual Studioでは、出力ウィンドウから特定のコード行にすばやくジャンプできる興味深いトリックを1つ実行できます。



これを行うには、次の出力形式を設定するだけです。 



"%s(%d): %s", file, line, message


ただし、 fileおよびlineの値は、register関数ではなく、ソースファイル内の実際の位置に従って取得する必要があるため、おそらく次のようなマクロが必要であることを忘れないでください。 

 #define MY_TRACE(msg, ...) MyTrace(__LINE__, __FILE__, msg, __VA_ARGS__) // usage: MY_TRACE("hello world %d", 5);


__LINE__および__FILE__は、コンパイラが認識できる標準のANSI互換の定義済みプリプロセッサマクロであることに注意してください。 定義済みマクロ、MSDNを参照してください。



また、 OutputDebugString関数を使用して、コンソールではなく出力ウィンドウにメッセージを表示することを忘れないでください。



これで、VS出力ウィンドウにメッセージが表示されたら、それをダブルクリックして、指定したファイルと行に移動できます。 同様に、コンパイル時に警告またはエラーメッセージを処理できます。 エラーメッセージを受け取ったが、コード内の正確な場所を特定できなかったため、多くの時間を失いました。 当時、私は弦を見つけなければなりませんでしたが、これは長くて面倒なプロセスです。 説明されているコードへのダブルクリック遷移により、これを数ミリ秒で実行できます。



ところで、別のIDE（Visual Studioではない）がある場合、そのような機会はありますか？ コメントの購読を中止し、知りたいと思います。



そして、ここに記載されているテクニックを試すことができる簡単なコード例： github.com/fenbf/DebuggingTipsSamples 。



更新： jgalowiczがコメントで指摘したように、ファイル名を短縮形で表示したい場合は、 __ SHORT_FILE__マクロを作成します。実装の詳細についてはブログ投稿を参照してください。 ただし、Visual Studio では、/ FCコンパイラオプションはデフォルトで無効になっているため、通常、省略されたファイル名（ソリューションディレクトリのみに関連する）が通常発行されます。



2.単純な静的変数を作成して、このオプションまたはそのオプションを制御します 



 // change while debugging if needed static bool bEnableMyNewFeature = true;


Visual Studioの編集および続行機能は非常に強力なツールですが、簡略化された「手動」バージョンで対応できます。 彼女はそれほどエレガントではありませんが、彼女は仕事を果たします。 プログラムの1つまたは別のオプションを制御する静的変数を取得する必要があります。 これを行うには、通常のブールフラグまたは整数型変数を使用できます。 デバッグ中に、この変数の値を変更できます。これにより、プログラムを再起動または再構築せずにオプションの動作を調べることができます。



デバッグ中に値を変更する方法は？ [ ウォッチ]ウィンドウに移動するか、カーソルを変数の上に移動します。目的の値を設定できる編集ボックスが表示されます。



そして、最終的なビルドとコミットでこの恐ろしい変数を無効化/削除することを忘れないでください！



3.条件付きブレークポイント



条件付きコントロールポイントを既に使用していることを望みますが、それらを使用する主な方法について簡単に説明したいと思います。 名前が示すように、これらのポイントでのデバッガーの作業は、特定の条件（非常に単純）が満たされると中断されます。















1つの小さなアドバイス：コードがどのように機能するかを詳しく調べる必要がある場合は、独自のブレークポイントを作成します。



条件で使用される式のリストを参照してください。msdn：デバッガーの式



しかし、それだけではありません。



スクリーンショットでお気づきのはずですが、ブレークポイントには「ヒットカウント」という別の便利な条件があります。 これにより、チェックポイントがアクティブになるイベントの数を設定できます。 このオプションは、何らかの動的イベントまたは多くのオブジェクトを追跡する必要がある場合に非常に便利です。

5.デバッグモードでオブジェクトのヘルパー変数を追加します



データが多いほど良い！ 不要なメッセージはいつでも除外できますが、何からでもデータを取得することはできません。 特定の目標によっては、オブジェクトにいくつかの補助変数を作成すると便利な場合があります。デバッグ中に、これらの変数は非常に貴重な情報を提供したり、単に作業プロセスを促進したりできます。



たとえば、ツリー構造で作業する場合、 pNext要素とpPrev要素を確認する必要がある場合があります 。 これらのポインターは、通常TreeNodeなどの基本クラスに配置されます。クラス階層の3レベル下にあるMyTreeNodeに関心がある場合 、毎回pNextをチェックするのは非常に面倒です。 しかし、 pNextからデータを追加してMyTreeNodeを更新するとどうなりますか？ この場合、この要素を確認するために毎回階層のすべてのレベルを確認する必要はありません。 ただし、説明した方法には1つの欠点があります。この更新された状態を何らかの方法で修正する必要があります。 pNextの値は簡単に変更できるため、これらの変更を同期するには追加のロジックを実装する必要があります。 ただし、これはほとんどの場合に当てはまりますが、タスクのデバッグにはあまり洗練されていない効果的なソリューションが機能します。



例を挙げましょう。



テキストオブジェクトを表すツリー構造で作業しなければならないことがよくあります。 テキストオブジェクトには文字列が含まれ、文字列には文字が含まれます。 行の最初の文字を返し、 pNextポインターを使用して2番目の文字を見つけ、それに基づいて値を調べる必要があるため、現在の行（つまり、含まれているテキスト）を毎回確認するのは非常に面倒ですこれら2つの文字を使用して、どのような文字列かを確認します。 このプロセスを簡素化する方法は？ strLine変数を開始し、 Lineに追加して定期的に更新しました。 これは情報の不完全なソースでもありますが（あるフレームでキャラクターが追加または削除されるとデータは失われますが、次のフレームでも表示されます）、少なくとも現在どの行にいるかをすぐに見つけることができます。 とても簡単！ そして、多くの時間を節約できます。

7.多くのオブジェクトをチェックする必要がありますか？



いくつかのコードが多くのオブジェクトに対して呼び出される場合、これらすべてのオブジェクトを調べて、それらを行ごとにチェックすることは非常に問題です。 しかし、エラーを探す場所についてのヒントとしてフィールドの一意の値を使用するとどうなりますか？ これを行うには、この値が特定の範囲と比較される条件付き制御点を作成できます。 この範囲が小さいほど優れています。



例：ドキュメント内のすべての文字をソートするコードをデバッグする必要がよくありました。 1つの（特殊な）文字に問題がありました。 すべての文字を個別にデバッグすることは不可能ですが、この特定の文字は境界ボックスのサイズが他の文字と異なることを知っていたため、 幅の値がチェックされる条件付き制御ポイントを設定して、この文字にアクセスできるようにしました（ 幅>通常の文字幅の条件 ） この条件を満たすのは2つまたは3つの要素だけなので、すぐにエラーを見つけました。



一般的に言えば、範囲が狭いほど良い：数十または数百の場所をチェックする必要はありません。



8.マウスイベント



これらのイベントのほとんどは、デバッガーが停止すると消えるため、マウスイベントのデバッグは特に困難です。



通常、マウスクリックのデバッグは簡単です。 たとえば、あるオブジェクトをマウスでクリックして呼び出されたコードを確認する必要がある場合は、 OnClick / onMouseDownメソッドの入り口にコントロールポイントを作成するだけです。



オブジェクトのドラッグアンドドロップについてはどうですか？ デバッガーが停止すると、ドラッグアンドドロップ状態が失われます。 このような状況では、次のトリックを試すことができます。



  • 古き良きtrace / printfを使用してください。 コードの実行を中断することなく何が起こっているかを理解するのに役立つ多くのメッセージをドラッグアンドドロップするとき。 この方法は、短時間のドラッグアンドドロップ操作に適しています。そうしないと、出力が大量のデータになるためです。 この情報を使用して、問題を特定し、後で問題に戻すことができます。



  • 検証が必要な場所に条件付き制御ポイントを確立します。 たとえば、オブジェクトを回転させると、オブジェクトの位置が突然変化する可能性があり、その理由を理解する必要があります。 これを行うには、オブジェクトの位置を担当するメンバーの入り口にコントロールポイントを作成します。これにより、メンバーを学習する機会が与えられます。 デバッガーが停止すると、回転状態は失われますが、これが発生する前に、コード内の問題の潜在的な場所を特定できます。 別の方法は、 obj_rot> some_meaningful_value条件を使用することです。



  • オブジェクトのドラッグアンドドロップは、コピー中に頻繁に発生します。 この場合、コピー操作が完了すると、元のオブジェクトに対応するステータスが割り当てられます。 元のオブジェクトのみがスキャンされるチェックポイントを設定するとどうなりますか？ または、 ドラッグ操作を通知する特別な状態を実装しますか？ この場合、デバッガーはこの操作が完了すると停止します。

このヒントでは、単純な補助変数を作成して、この手法をさらに発展させることを提案しています。 複雑なオブジェクトを扱う場合、データをより効果的に追跡できるツールがあると便利です。 このようなツールの基本セットはVisual Studioおよびその他のIDEまたはデバッガーにありますが、すべてのプロジェクトが異なるため、独自のソリューションを開発することをお勧めします。



この状況は、私には思えますが、ゲームの典型です。 たとえば、インターフェイスの追加のレイヤーを作成し、ゲーム中にオンにして、ゲームの統計、パフォーマンス、およびメモリ消費情報を確認できます。 表示される情報の量は、ニーズによってのみ制限されます。 そのため、時間をかけてそのようなツールを作成することを強くお勧めします。

10.リリースモードでのデバッグ



ほとんどすべてのコード最適化がすでに含まれているため、リリースビルドは高速です。 ただし、この場合もデバッグを妨げるものはありません。 これはどのくらい正確に実装できますか？ 以下を行う必要があります（VS 2013またはVS 2015の場合）：



  • [ デバッグ情報の形式 ]オプションを/ Z7（C7互換形式）または/ Zi （プログラムデータベース形式）に設定します。
  • [ 増分リンクを有効にする]を[いいえ]に設定します
  • [ デバッグ 情報の 生成 ]オプションを[はい]に設定します。
  • Referencesオプションを/ OPT：REFに設定し、 COMDAT Foldingを/ OPT：ICFに 有効にします


11.デバッグアセンブリの高速化！



  • デバッガーが遅い場合は、オプションツール→オプション→デバッグ→全般→「元のバージョンと完全に一致するソースファイルが必要」を確認してください。 ここで彼女について知りました。



  • デバッグヒープ（ Debug Heap ）を無効にします-VS 2015より前のバージョンに関連します。以前の記事のいずれかで、 Visual Studioの低速デバッグと_NO_DEBUG_HEAPについて説明しています。 幸いなことに、VS2015では、デバッグヒープはデフォルトで無効になっているため、この問題は発生しません。



  • デバッグシンボルファイルのダウンロードを制限します。 この手法は、ダウンロードされるシンボルファイルの数を減らし、プログラムの起動を高速化します。 詳細については、こちらをご覧ください： シンボルファイルとVisual Studioのシンボル設定について
