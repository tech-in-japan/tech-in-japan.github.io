残念ながら、独創的なアイデアは私を訪問していないので、これらの記事の例は本質的にやや抽象的で理論的です。 役に立つ何かを書くためにこの情報を使用することはあなたに任せます。



これらの記事を理解するには、OOPの原則とPython（以下-Python）でのGUIアプリケーションの作成の原則に関する十分な知識が必要です。



一般的な情報



XBMC Python APIの一部であるxbmcguiモジュールは、インターフェースの操作を担当します。 このモジュールには、一般的なコンテナクラスと、さまざまなインターフェイス要素を担当するクラスが含まれています。 残念ながら、インターフェースを担当するAPIの部分は、GUIの従来のPythonライブラリよりも便利ではなく、ドキュメントも不正確です。 以前の記事で書いたスタブモジュールの使用をお勧めします 。 開発の利便性に加えて、クラス、メソッド、関数のセット、およびそれらの使用（ドックストリング）の両方に関する関連情報も含まれています。



プラグインインターフェイスコンポーネント



プラグインインターフェイスを作成するために、xbmcguiモジュールはさまざまなコンポーネントを提供します。

-コンテナクラス（Windowクラスとその子孫）;

-ウィジェット、またはXBMC用語では、さまざまなインターフェイス要素（コントロールの下位クラス）を担当するコントロール。

-情報を表示し、ユーザーと対話するために使用されるダイアログ（Dialogクラスなどのメソッド）。



最初の2つのポイントをより詳細に検討してください。



WindowXMLおよびWindowXMLDialogクラス



名前が示すように、これらのコンテナクラスは、XBMCスキンと同様の構造を持つXMLスケルトンインターフェイススケルトンを使用します。 つまり、実際には、これらのクラスに基づいたインターフェイスには独自のミニスキンがあります。 私はスキンライティングのジャングル（私はほとんど知らないことを認めます）を掘り下げたくないので、これらのクラスにこだわることはありません。



WindowおよびWindowDialogクラス



WindowおよびWindowDialogクラスは、他のインターフェイス要素（コントロール）をホストする親コンテナーです。 コントロールを配置するために、これらのクラスは画面の左上隅から始まる座標グリッドを提供します。 デフォルトでは、グリッドの表示領域の解像度は1280 x 720ピクセルです。 この解像度は変更できますが、個人的にはあまり意味がありませんので、例では可視領域の標準解像度を使用します。 「可視領域」という用語を使用したのは何の理由でもありません。 コントロールの座標は、負の値を含む任意の値を持つことができますが、画面に表示されるのは表示領域に収まるものだけです。 これは、たとえば、明らかに可視領域を超える座標（たとえば、-5000、-5000）を設定することにより、ある種のコントロールを一時的に隠すために使用できます。

注意：座標グリッドの可視領域の解像度と画面解像度を混同しないでください。 表示領域は、実際の画面解像度と同じ解像度を持っているため、実際の解像度に関係なく、インターフェイス要素は常に同じ縮尺を持ちます。



WindowクラスとWindowDialogクラスの違いは、Windowクラスの背景が黒く不透明であり、ビデオ再生ウィンドウまたは音楽視覚化ウィンドウの下で非表示にできることです。 WindowDialogの背景は透明になり、そのコントロールは常にXBMCインターフェイスの他の要素の上に表示されます。 したがって、プラグインがビデオを再生し、音楽を再生する場合、そのためにWindowクラスを使用することをお勧めします。他の場合、クラスの選択はプラグインの個人的な好みとタスクに依存します。



インターフェイスイベントを処理するために、両方のクラスにonAction（）およびonControl（）メソッドがあります。 1つ目はキーボードコマンドをインターセプトし、2つ目はアクティブ化されたコントロールです。 したがって、ユーザーとの対話を保証するために、これらのメソッドは、適切なイベントハンドラを記述して、プラグインのインターフェイスを実装する子クラスで定義する必要があります。

注意！ デフォルトでは、onActionメソッドはBACKSPACEおよびESCキーに対応するキーボードコマンドを「キャッチ」し、closeメソッドを呼び出します（プラグインを終了します）。 したがって、このメソッドが子クラスでオーバーライドされる場合、このクラスのプラグインを終了するためのコマンドを少なくとも登録する必要があります。 それ以外の場合、XBMCを強制的に停止（「強制終了」）することによってのみプラグインを終了できます。これは、組み込みシステム（OpenELECなど）で問題になる可能性があります。



コントロール



次に、コントロールについて詳しく説明します。 すべてのコントロールについては説明しませんが、私が個人的に研究したものについてのみ説明しますが、ほとんどのタスクには十分だと思います。 コントロールは、プラグインのインターフェース要素（ウィジェット）です。 ただし、汎用GUIフレームワーク（Tkinter、PyQt）のウィジェットとは異なり、コントロールはインターフェイス要素の「スケルトン」のようなものです。 ビジュアルデザインのコントロールには、画像（テクスチャ）を含むグラフィックファイルが必要です。



フォトジャックグラフィックプログラムに精通している人は自分でテクスチャを描画でき、他のプラグインはXBMCスキンリソースに必要なテクスチャファイルを取り込むことができます。 残念ながら、完成したスキンでは、テクスチャが特別な形式でパックされているため、スキンのソースコードが必要です。 画像を含む標準のXBMCスキン、Confluenceのソースは、 ここにあります 。

注：一部のコントロール（ボタンなど）は、現在のスキンの必要なテクスチャを自動的に使用できますが、この動作に依存することはできません。 第一に、すべてのコントロールがこれを実行できるわけではありません。第二に、別のスキンに切り替えると、プラグインの外観に違反する可能性があります。 したがって、コントロールのテクスチャファイルを明示的に指定することをお勧めします。



上記から推測できるように、独自のインターフェイスでプラグインを作成する場合、2つのアプローチを使用できます：シンプル、プラグインの外観がすべてのスキンで同じ外観を持つ、および複雑な、ウィジェットのテクスチャファイルが現在のスキンに応じて選択される、プラグインインターフェイスがスキンと同じスタイルでスタイル設定されるようにします。 広大さを把握できないことは明らかなので、2番目のアプローチでは、いくつかの人気のあるスキンに制限し、残りの部分にはデフォルトのレイアウトを使用する必要があります。

背景が透明なシンプルな碑文。 テクスチャは使用しません。 これは、TkinterのLabelクラスまたはPyQtのQLabelの完全な類似物です。 テキストの配置は、次の数値定数のいずれかで指定されます。

制御リスト


リスト。 単純なテキスト文字列をリストに追加できますが、実際には、リスト内の各アイテムはListItemクラスのインスタンスです。 プラグインコンテンツソースのリストは同じ要素で構成されていることを思い出させてください。 ただし、ソースプラグインとは異なり、ControlListのリストアイテムの機能は制限されています。 実際、テキストラベルと小さなアイコン（thumbnailImage）のみがサポートされています。 リストアイテムを選択するときのアクションなど、残りのロジックはすべて、自分で実装する必要があります。 リストをスタイルするには、コントロールにテクスチャが必要です。



すべてのリスト項目が指定されたスペースに収まらない場合、リストがスクロールします。

プラグインインターフェイスの基本的なコントロール






ControlButton、ControlRadioButton、ControlEdit、ControlList、およびControlSliderは、選択すると外観が変化するインタラクティブな要素です。 理論的には、ControlLabelなどの任意のコントロールを選択できますが、視覚的なフィードバックはありません。



キーボードまたはリモコンの矢印キーを使用してプラグインインターフェイスをナビゲートするには、対応する矢印キーを押したときにフォーカスが移動するネイバーを各コントロールに割り当てる必要があります。 これを行うには、setNavigationメソッド、およびControlから継承されたcontrolUp、controlDown、controlLeft、controlRightを使用します。

また、ナビゲーションを機能させるには、インターフェイスの初期表示中に、いずれかのコントロールにフォーカスを設定することを忘れないでください。



コントロールは、コンテナクラスのaddControlメソッドを使用して追加されます。 同時に、後で追加されたコントロールは、以前に追加されたコントロールの上に表示されます。 たとえば、背景として機能する画像を最初に表示し、他のすべてのコントロールをその上に配置できます。



注意：ほとんどすべてのコントロールには、プラグインの実行中にプロパティ（テキスト、画像など）を変更できるメソッドがあります。 したがって、addControlメソッドを使用してこのコントロールをコンテナクラスに追加した後は、常にコントロールプロパティ（たとえば、setLabel（ 'Some Text'）を使用してControlLabel碑文を変更します。「非バインド」コントロールのプロパティを変更しようとしても、何も起こりません。 、最悪の場合、XBMCインターフェースのあらゆる種類の興味深い不具合が発生する可能性があります。
