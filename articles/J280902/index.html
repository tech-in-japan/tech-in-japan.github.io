演算子、その過負荷および簡潔さの考え



Swiftのオペレーターに連絡しました。 最初の質問：演算子は特別な構文ですか、それともプロトコルの砂糖ですか？ 私が使用している、またはこれまでに使用したすべての現代言語（Python、Ruby、Io、Elixir、Rust-さまざまな時代とスタイルのいくつかの例）は、他の言語構成の砂糖として単純に実装しています。



整理してみると、演算子は（十分に）定義された関数であることがわかりました グローバルスペース Swiftモジュール。 説明がこれであるため、「良い」の代わりに「大丈夫」と言います。これは、既存の型の間で演算子をバイナリ演算子のように動作させる唯一の方法です。 この場合、理由は言語の構造にあるという事実を見逃しています。 プロトコルはここに完全に収まるように見えますが、おそらく、Rustの特徴とは異なり、問題に十分に対処できないでしょうか？ これは未解決の質問であり、どのように答えればよいかわかりません。



興味深いことに、Rustのオペレーターの数は、前の投稿で言及したものを除いても、Swiftよりもわずかに少ないです。 RustはPythonと同様に、他のより簡単な方法で同じ結果を得ることができるため、プレフィックス演算子とポストフィックス演算子を完全に欠いています。 Swiftでは、これらのステートメントは部分的に保存されています。間違いなく、（Objective）Cを扱ったほとんどのプログラマーはそれらとそのイディオムに精通しているからです。

翻訳者のメモ：Swift 2.2では、元々Rustになかったインクリメント/デクリメント演算子が削除されました。 一方、Swiftでは、独自のステートメントを作成できます。



また、Rustの演算子について何か新しいことを学びました。ブール演算子||

Rustには、3項演算子がありません。これは、言語が式とステートメントを処理する方法に関連しています。 Swiftではそうです。 言語設計の違いに関する興味深い考え： if

比較された言語のブール演算子にも違いはなく、驚くことでもありません。



Swiftに関する本で「短い」という言葉またはその同義語が何回使われているのだろうか。 簡潔さがこの言語の主な目標の1つであることをより明確に理解しています。 たぶんそれはそのように思えますが、それでも少し奇妙です。 簡潔さは良いですが、読みやすさははるかに優れています。









文字列操作を使用した言語構造のトレードオフ



SwiftとRustは、セキュリティとメモリ管理の問題を解決しますが、アプローチは異なります。自動参照カウントによるSwiftと、所有権の概念によるRustです。 毎日の作業では、SwiftアプローチはPythonやRubyと同じ理由でより有利に思えます。誰もがあなたのためにそれを行うと便利です。 Rustは多くの機会を提供しますが、同時に所有権について常に考えさせます。



つまり、すべての言語にトレードオフがあります。 私はこれまでSwiftよりRustの方が好きですが、Swiftをより良くする多くのポイントを見つけることは間違いないでしょう。 すべてを手に入れることはできません。



文字列（または値によって渡される他の型）の操作は、RustよりもSwiftの方がはるかに簡単であるため、このことに注意を向けました。 結果は実質的に同じですが、Swiftではすべての行が値によって渡され、参照によって渡されることはないため、変更される内容について考える必要はありません。



もちろん、この目的のためにRustにはコピー特性があります-私は、Swiftが少し「人間工学的」であることを意味します。



Swiftの文字列補間は非常に便利です。 これがRustで見逃している唯一のものです。 文字列をフォーマットするためのPythonスタイルのマクロは優れていますが、値（ strings with \(variables)



またはembedded expressions like \(2 + 4)



strings with \(variables)



補間は非常に優れています。



一般に、文字列に対するSwiftのアプローチはよく考えられており、詳細に十分な注意を払っています。これにより、複雑な言語または非西洋言語での作業が大幅に促進されます。 私は、印刷に夢中なので、本当に感謝しています。



同時に、Swift文字列はUnicodeの同様の境界ケースをすべて処理するため、文字列にアクセスするためのいくつかの標準パターンが失われ、文字列の内部構造を理解することが困難（または不可能）になります。 状況に応じて、これは利点と欠点の両方になります。 私が言ったように、妥協はどこにでもあります。



実際、さらに読み進めてみると、SwiftはUnicode文字列を非常にエレガントに処理し、さまざまな実装に個別のメソッドを使用してプロセスを理解できることがわかりました。 特に、 String



型を操作し、「コードポイント」への直接アクセスを取得できることを感謝します。いずれかではなく、UTF8、UTF16、またはUTF32のいずれかにアクセスできます。 Appleの経験を信頼してください。テキストに非常に注意する必要があります。



Rustの文字列は悪くありませんが、それほど複雑ではありません（おそらく、これはメモリ内の文字列の表現を単純化するために行われたと思われます）。 この言語では、 String



またはstr



常にUTF8バイトのシーケンスとしてエンコードされたスカラーUnicode値（UTF32）で構成されます。 Swiftとは異なり、他の実装を取得するための便利な方法はありません。 同時に、日常のプログラミングでは、これはめったに発生しないか、おそらくまったく発生しないと考えています。 重要なことは、両方の言語がスカラー値を格納することです。



これは、RustのSwiftに対する明確な優位性を感じなかった最初の部分です。 これらの言語の構造におけるトレードオフのいくつかはここでより明確になり、Swiftの「人間工学」がそれらの中にあることを感謝しています。
