ローカリゼーション



翻訳されたテキストは確かにクールですが、ゲームに含まれていない場合のポイントは何ですか？ Unity / Androidでの名前の翻訳では、すべてが簡単です。プラグインフォルダーにOSの文字列リソースファイルを作成するだけです。

\ Assets \ Plugins \ Android \ res \ values-xx （英語の値のみ）： 



 <?xml version="1.0" encoding="utf-8"?> <resources> <string name="app_name">Hello World</string> </resources>


Unityのゲーム自体のテキストを翻訳するための標準ツールはありませんが、この問題を解決するための有料と無料の両方の多くのカスタムアセットがあります。 しかし、どのプログラマーが高速バイクを好まないのでしょうか？ （回答：良い）ローカライズとして、Androidスタイルのローカライズを含む小さなスクリプトを使用することにしました。

  • 文字列リソースを含むxmlファイル
  • xmlを解析し、文字列の列挙型IDを生成するエディタースクリプト
  • 列挙型IDでXMLリソースから文字列を取得するシングルトン
  • LocalizationControllerは、テキストコンポーネントの配列を持ち、所有者の名前で、シングルトンからのローカライズされたテキストを要求します


一般に、これはenumの自動生成を除き、一般的なローカライズスクリプトです。 残念ながら、著者を見つけることができなかったため、 Gistにコードをアップロードしました。 StrokeTextのサポートを追加しました。

最適化



最初は、そのようなプロジェクトは最適化を必要としないが、人生は別の方法で決定したことを笑い飛ばしたかっただけです。 ゲームのサイズが20 MBを超えるとは思っていませんでしたが、実際には34 MBになりました。 そのため、最適化の一環として、最終アセンブリのサイズを小さくすることに興味がありました。 読む価値のある資料が2つあります。 これは少し古いですが、それにもかかわらず、 フィッシャーと公式ドキュメントからの非常に関連性の高いドロップデッドの記事 （！）です。



まず第一に、習慣から、私はすべてのグラフィックをpngquantで運転しました 。 （エラーのある最適化）もちろん、アセンブリプロセス中にUnity3dがすべてのグラフィックを中間形式に変換するため、これは役に立ちませんでした。これはドキュメントに明確に書かれています。 ところで、私はすぐに問題を特定するのに役立った編集者ログのような素晴らしいことを発見しました：

9.0 mb 26.2％Assets / Pcs / BG.pngは予想される1.3 mbではなく。



容認できないほど多くのスペースが背景で占められ、同時にゲームの最大の画像で占められていました。 サイズを2のべき乗に変換すると、アプリケーションのサイズが7MB縮小されました。 （実際、私はパネルの背景を減らしました-食物ファイル）

ストリッピングレベルを設定して獲得した約1メガバイト。



デフォルトでサブセットNET2.0を使用しましたが、Universal Assemblyを完全に忘れていました。 ARM7およびx86アーキテクチャ用の個別のアセンブリにより、apkファイルのサイズを目的の18MBに縮小できました。 ハッピーエンド。 ここで最適化は必要ないという私の言葉の後に、アニメーションによって誰かが遅くなると恥ずかしいでしょうが。
