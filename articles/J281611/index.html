C ++の依存関係マネージャーとしてのNix







最近、C ++にはpip、npm、maven、cargoなどの独自のパッケージマネージャーが必要であるという多くの話があります。 すべての競合他社には、非標準ライブラリを接続するためのシンプルで標準化されたメカニズムがあります。 C ++では、誰もができる限り行動します：誰かがUbuntu、CentOS、その他のディストリビューション用のパッケージのリストをREADMEに書き込み、誰かがgitサブモジュールとスクリプトを使用してそれらを構築し、誰かがCMake ExternalProjectを使用し、誰かがすべてのソースをコピーします1つの巨大なリポジトリ、誰かがDockerまたはVagrantのイメージを作成します。







この問題を解決するために、スタートアップ-biicodeも作成されましたが、破産し、将来は不明です。 代わりにコナンが登場し、競合他社の動物園の動物園、 cget 、 ハンター 、 cpm 、 qpm 、 cppget 、 pacm 、さらにはgradle for c ++を 補完しました。







上記のいずれの方法にも満足できませんでした。 Conanのパッケージを書き始めましたが、多数のハック、未開発のAPI、ガイドラインの欠如、そしてその結果、他の人のパッケージを再利用する可能性が低いことに遭遇しました。 そして、私はNixOSのパッケージマネージャーのアイデアが本当に好きだったことを思い出しました 。 そして、私は考えました-通常のパッケージマネージャーが同じ問題を解決するのであれば、なぜC ++専用のパッケージマネージャーを作成するのですか？ パッケージの説明に関しては、十分に柔軟でシンプルであることが必要です。 そして、Nixはこの役割にぴったりでした。







それで、Nixが私たちに与えたもの：







  • 1つのコマンドnix-shell



    プロジェクト環境を構築する準備を整える機能。
  • nixpkgs



    既製およびサポートパッケージ。
  • リポジトリーのパッケージから派生パッケージを作成する機能（コードをコピーせずに）。
  • 依存関係のC / C ++ライブラリだけでなく、必要なツール（CMake、GCC）、他のエコシステムのプロジェクト（npm、pip）、サービス（redis）も指定する機能。
  • コミットする環境をバインドする機能。 これは、たとえば、masterブランチがboost 1.55とdevel-1.60を使用できることを意味します。 ブランチからブランチに切り替えると、Nixは目的のバージョンの環境を自動的に構成します。これには1秒もかかりません（アセンブリが既にキャッシュにある場合）。
  • 非侵入性-プロジェクトはNixに依存せず、その使用は誰にとっても個人的な問題です。 cmakeのすべての正しいオプションを指定して、すべての依存関係を手動で（またはお気に入りのパッケージマネージャーによって）収集できます。


ニックスとは



Nixは、パッケージマネージャーのニーズに合わせて調整された関数型プログラミング言語です（Haskellコミュニティで人気を博したのは驚くことではありません）。 パッケージアセンブリは、Nixでの関数計算です。 関数型プログラミング言語の場合と同様に、同じ引数を使用して関数を繰り返し呼び出すと、同じ結果が生成されます（バイナリパッケージ）。 これは、パッケージをキャッシュできることを意味します。これは、Nixが行うことです。すべてのアセンブリは、 /nix/store/$HASH-$PKGNAME



格納されます。 さらに、ネットワーク上の他の誰かが同じハッシュを持つパッケージを持っているかどうかを確認できます。持っている場合は、その人からバイナリパッケージをダウンロードします。







したがって、Nixの「パッケージ」（ここでは派生）は関数であり、「依存関係」はこの関数の引数です。 リポジトリ（ NixPkgs



）とは何ですか？ これも引数を持たない関数であり、多くのパッケージを返します。 リポジトリを使用するには、すべての7344パッケージを収集する必要がありますか？ いや！ Nixは遅延言語です。つまり、明らかに必要になるまで何も計算されません。 また、ユーティリティを使用してパッケージを「必要」にすることができます。
