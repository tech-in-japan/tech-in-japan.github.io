Visual Studio Team Foundation Server 2010を使用する







この記事は、Visual Studio Team Foundation Serverを以前にインストールまたは使用したことがない人に役立ちます。 TFSは、レポート、SharePointとの統合、複数のドメイン、分散データベースなどを含む非常に複雑なインフラストラクチャの一部になることができますが、これらの領域については説明しません。 私の主な仕事は、TFSの基本要素（バージョン管理システム、バグおよびタスク追跡システム、自動アセンブリシステム）の処理を支援し、このシステムを使い始めることです。





まえがき



まず、なぜTeam Foundation Serverなのか見てみましょう。 TFSは、開発ツールを統合して、より速く快適な作業を行うために作成されました。 異なるシステムを一緒に統合できます。







この場合、各システムには独自のデータウェアハウス、独自の識別データセット、独自のコマンドおよびツールがあります。 もちろん、これは可能ですが、そのようなシステムで作業する場合、コンポーネントとサポートを切り替えるのに多くの時間がかかります。



TFSは、必要なすべてのコンポーネントを統合するシステムです。







このような統合は、最も一般的なシナリオをカバーしています。 通常は、ソースコードを追加および編集し、アセンブリを作成し、テストし、見つかったエラーをログに記録して修正します。 ワークフロー全体が1つの統合環境で発生すると、プロセスの各要素を接続できます。 たとえば、エラーを修正したリポジトリにファイルをコピーした場合、すぐにレポートでこれについてメモしたいと思います。 「基本」構成でインストールされたTFSでは、上記のすべてを実行できます。 TFSのフルバージョンには、自動テスト、仮想実験室の展開、およびアプリケーションアーキテクチャの検証が含まれます。







必要に応じて、コンポーネントの一部のみを使用できます。



TFS機能にアクセスするには多くの方法があります。 あなたがプログラマーなら、おそらくVisual Studioを使用して快適に感じるでしょう。 テスターの場合、Visual Studioをインストールすることなく、新しいチームエクスプローラーをクライアントとして使用できます。 プロジェクトマネージャーの場合、Webブラウザー、Excel、Microsoft Project、またはMOSSを介して情報にアクセスできます。



TFS 2010をインストールする



これからは、このプロセスがかつてないほど単純になったと言えます。 そのため、詳細なインストール手順を公開せず（インストールで問題が発生した場合は、 Visual Studio Team Foundation Server 2010のインストールに関する記事を読むことをお勧めします ）、理論的な知識のみに限定します。

TFSの新しいバージョンが提供する主な利点を考慮してください。



1. 要件 -バージョンからバージョンへと歴史的に渡されてきた最も重要な制限は削除されました。

  • TFS 2010はドメインコントローラーにインストールできます。 TFS開発者は、多くの小規模組織には専用サーバーを使用する能力がないことに気付きました。 これで、ドメインコントローラー、電子メールサーバーなどのサーバーが1つしかない場合、このサーバーにTFS 2010をインストールできるようになりました。

  • TFS 2010はパーソナルオペレーティングシステムにインストールできます-TFS 2010はVistaおよびWindows 7 Home Premium以降にインストールされます。 そしてもちろん、サーバーオペレーティングシステムがサポートされています（Windows 2003、Windows 2008、およびWindows 2008 R2）。 これで、職場のラップトップでバージョン管理サーバーを直接実行できるようになりました。

  • TFS 2010は、32ビットと64ビットの両方のオペレーティングシステムで実行できます。





2. インストール 。 TFSをインストールすることは、長年の悩みの種でした。 今、この欠陥は美徳になっています。 TFS 2010インストーラーには、基本、標準、および詳細の3つのインストールモードがあります。 「基本」モードで特に大きなブレークスルーが行われました。 実際、このモードは、Next-> Next-> Nextの原則に基づいたインストールに変わりました。これにより、TFS 2010を20分以内でインストールおよび構成できます（コンピューターに.NET 4.0およびMSSQL Expressが既にインストールされている場合）。 「基本」モードでは、IIS（まだインストールされていない場合）、MSSQL Server（まだインストールされていない場合）、およびTFS 2010を直接インストールして構成することができます。 4.0、コンピューターを再起動する必要があります。 しかし、私を信じてください、結果は価値があります！



「基本」モードでTFSをインストールすると、バージョン管理システム、バグ追跡システム、およびアセンブリ自動化システムが得られます（継続的な統合は簡単です！）。 完全なセットには、SharePointとレポートシステムのコンポーネントがありません。 これらの要素は、「標準」モードに存在します。 もう1つの良いニュースは、TFS 2010“ Basic”が既にインストールされており、システムを完全に再インストールせずに、必要に応じてコンポーネントを簡単に追加できることです。



TFS 2010のバージョン管理システム



そして、十分な理論的知識を得てTFS 2010をインストールしたら、始めましょう。 この章では、TFS 2010が提供するバージョン管理システムの使用の基本について説明します。



コンピューターにTFS 2010が既にインストールされており、Visual Studio 2010がインストールされていることを前提としています。



したがって、最初に行う必要があるのは、Visual StudioをTFSに接続することです。 これを行うには、メインメニュー（チーム）またはホームページのリンクを使用します。







システムは、既存のTFSのアドレスを指定するよう要求します。 私の場合、私のWindows 7ラップトップはdionnis-pcと呼ばれています。











その後、チームエクスプローラーウィンドウにサーバーへの接続の名前とDefaultCollectionが表示されます。 しかし、現時点では、追加されたプロジェクトは複数ありません。







この場合、たとえば、エンタープライズライブラリコードを使用しますが、実際には、標準アプリケーション（ファイル、新規プロジェクト、Windowsフォーム）に制限することができました。 プロジェクトをバージョン管理システムのリポジトリに追加しようとすると、Visual Studioはエラーをスローします。











このエラーは、TFSでプロジェクトを作成する必要があることを意味します。これは、作業に必要なすべてのコンポーネントになります。 そのため、最初に新しいプロジェクトを作成する必要があります。







次のダイアログボックスで、プロジェクトの名前と簡単な説明を指定する必要があります。







Visual Studioは、アプリケーションの開発時に使用する方法を指定するように求めます。 デフォルトはアジャイル（アジャイル開発手法）ですが、CMMIを選択することもできます。 方法論に関する追加情報については、MSDNを読むことをお勧めします。 自分に何が適しているかわからない場合や、柔軟な開発方法論（TDDなど）を使用する場合は、アジャイルにとどまることをお勧めします。



















そして最後に、チームエクスプローラーには、現在のプロジェクトの要素（作業項目、ビルド、およびソース管理）が表示されます。







これで、プロジェクトをリポジトリに追加できます。







次に、データを保存するフォルダーを指定する必要があります。







正常に完了すると、ソリューションエクスプローラーは、リポジトリに配置されると主張するファイルに「+」記号を付けます。 また、アプリケーションをリポジトリに配置するために実行する必要があるアクションのリストも表示されます。 コメントを追加して、[チェックイン]をクリックするだけです。







ファイルをリポジトリに追加するプロセスを確認する必要があります。







そして、おめでとうございます！ プロジェクトをリポジトリに追加しました！



TFS 2010のエラー追跡システム



バージョン管理システムの使用方法を理解したら、バグ追跡システムの原理を検討します。



Visual Studioのエラーおよびタスクエントリは、作業項目に関連しています。 Visual Studioの[チームエクスプローラー]パネルから、作業項目の種類の1つを直接作成できます。 WebインターフェイスまたはVisual Studioのテストおよびラボ管理ツールを使用して、同じことができます。 私たちの場合、チームエクスプローラーパネルを使用します。







プロジェクトの作業を開始したばかりなので、リストにはエントリがありません。







エラーメッセージを追加しましょう。











エラーのリストを更新すると、作成したばかりのレコードを見ることができます。







もっとリアルなエラーメッセージを追加しましょう。







次に、エラーを修正する必要があります。 これを行うには、リポジトリからファイルをダウンロードして編集します。







すべてが順調に進んだ場合、ファイルには編集のためのアクセシビリティに関するマークが含まれます。











編集後、Visual Studio自体の[保留中の変更]パネルには、変更されたファイルのリストが表示されます。







エラーを修正したため、これを記録する必要があります。







修正したエラーを書き留めてコメントを追加したら、ファイルをリポジトリに安全にコピーできます。











これで、エラーのステータスが変更されたことを確認し、修正の詳細に関する詳細情報を取得できます。







TFSを使用する別の方法


Webインターフェースを使用して、プロジェクトの作業項目にアクセスできます。 これを行うには、単にサーバーのアドレスを使用します。















この方法は、Visual Studioの操作に慣れていない人に最も効果的です。 ExcelとMicrosoft Projectを使用することもできます。



TFS 2010のビルド



完全な（最小限の）キットの場合、アセンブリの操作方法を学ぶだけでは十分ではありません。 この章は、このギャップと戦うことを求められています。



最初に、アセンブリパラメータを決定する必要があります。 これを行うには、Visual Studioで既に使い慣れたチームエクスプローラーパネルを使用します。











ここで、可能性のあるパラメーターの一部を検討します。



特に興味深いのは、[トリガー]タブです。 このタブでは、どのアセンブリがアセンブルされるかに基づいてイベントを設定できます。







  • 手動 -アセンブリは要求に応じて手動で設定されます。
  • 継続的インテグレーション -アセンブリは、チェックインの直後（ファイルをリポジトリにコピーした後）に発生します。 この方法は、変更のマージを待たずにアセンブリを作成する場合に非常に効果的です。
  • ローリングビルド -前のビルドが実行されている間にすべての変更が収集される方法。 この方法は、プロジェクトが非常に大きく、アセンブリに時間がかかる場合（変更が行われる速度よりも長い場合）に使用することをお勧めします。
  • ゲートチェックイン -この方法により、ファイルがメインリポジトリに入る前に、すべての変更が正しくコンパイルされていることを確認できます。
  • スケジュール済み -アセンブリを実行するスケジュールを設定する方法。 たとえば、多くの企業では、毎日のアセンブリを作成することをお勧めします。




このような豊富なオプションセットを使用すると、ニーズに基づいてあらゆる種類のアセンブリを作成できます。



アセンブリを設定するときの次の重要なタブは、タブ-Build Defaultsです。 ここでは、アセンブリ後に結果が配置されるフォルダーを指定する必要があります。







これで、アセンブリパラメータを保存し、Team Explorerパネルで使用可能になったことを確認できます。 実行キューに新しいアセンブリを追加しましょう。















キュー内のアセンブリをダブルクリックすると、実行に関する詳細情報が表示されます。







しばらくすると、結果が表示されます。







私の場合、結果は安らぎではありませんでしたが、今では重要ではありません。 あなたが大丈夫であることを願っています！ このレポートは、検出されたすべてのエラーと警告に関する詳細情報を提供するため、このレポートから、エラーの原因となったコードにすぐに移動できます。



それで、アセンブリを構築するためにTFSが提供するツールに注目しました。 これで、TFSを使用して製品の最小ライフサイクルを提供する準備が整いました。



これでTFSに関する記事を締めくくり、さらに興味深いプロジェクトをお祈りします！

そして最も重要なこと-プログラミングを楽しむことを忘れないでください！
