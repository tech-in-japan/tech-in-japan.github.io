PDFドキュメントを表示するためのMonotouch OpenSourceライブラリ

さて、最後に、自分のプロジェクトの1つを公開し、一般に公開することにしました。 それをオープンソースに変えましたが、このイベントについてHabréで話したかったのです。

次に、MonotouchプラットフォームとiOSでのPDFドキュメントの表示について説明します。 このトピックでは、ライブラリのソースコードの束を提供しません;それを必要とする人はGitHubでそれらを見つけますが、ライブラリを構築するときに生じる複雑な問題に集中しようとします。





確かに、Monotouchを使用する多くのiOS開発者は、プログラムでPDFドキュメントを表示するという課題に直面していました。 しばらく前に、プログラムを開発するときにこのような問題に直面しました。 iOSでPDF文書を表示するために提供される標準オプションを調べた後（以下の比較表を参照）、ライブラリを書くことが不可欠であることに気付きました。 ライブラリを作成したとき、iOS 5はまだリリースされておらず、新しいPageControlコンポーネントは手元になかったとすぐに言いたいです。

  • UIWebViewは間違いなく良いことですが、デモプロジェクトにのみ適しています。 コントロールはさまざまなドキュメント形式を表示できますが、表示の品質はドキュメントの内容に大きく依存します。 Word文書、Excelは良心のtwinなしに切断され、一部のPDF文書はまったく開かず、InnerExceptionが報告されます。

  • QLPreviewController-名前によって判断すると、ドキュメントの予備表示を目的としたものがMacOSで使用されます。 MS OfficeドキュメントでもUIWebViewと同じ問題があります。 PDFに関しては、すべて（たとえば-99％）が開きますが、視覚的な調整や新しい機能のねじ込みは行われません。 通常、プログラムに統合しても機能しません。

  • PDFドキュメントを表示するためのサードパーティのObjective-Cライブラリは、iOSのPDFビューアに必要なほぼすべての機能を実装する強力なライブラリです。 しかし、ここでも、カスタマイズ、非常に高いコスト（> 500 EUR）、およびMonotouchで使用するラッパーを作成する必要があるという問題があります。また、オープンクラスが多数あるライブラリの場合、これは最も簡単なタスクではありません。



結局、私は忍耐（たくさんのコーヒー）を買いだめすることに決めて、私の傑作自転車を書き始めました。 ライブラリを作成する前に、その機能のリストを定義しました。

  • ページをめくるための2つのオプション。
  • 自動ページスケーリング。
  • 品質ページの表示。
  • 詳細なページスケーリング。
  • しおり
  • 注。
  • サムネイル。
  • オプションを設定します。
  • リンクの定義。
  • 検索。
  • 文書の内容または構造を定義します。
  • Cocoa Touchクラスのみを使用してください。
  • 100Mb以上の「大きな」ドキュメントの表示。


もちろん、彼には時間がないか、すべてのアイデアを実現できませんでした。おそらくいつか彼の手が届くでしょう。 リンクの定義とドキュメントのコンテンツ、およびドキュメント内の検索はまだ実装されていません。



ページめくり



最初の、そしておそらく主な質問は、視覚的なページめくりを実装する方法でしたか？ 前述のとおり、PageControlコンポーネントはまだ存在していなかったため、独自のソリューションを考案する必要がありました。 考えた後、私は大きなコンテンツ領域とその中のいくつかのビューで標準のScrollViewを使用することにしました。 ScrollViewを使用すると、複数のビューを追加できます。移動すると、視覚的に置き換えられます（裏返しの効果です）。 いくつのビューが必要でしたか？ 答えは簡単です-3つ（前のページに1つ、現在のページに2つ、次のページに3つ）。 この場合のコンテンツ領域は、3 * View.Width、つまり トリプルページビューの幅。







水平および垂直のページめくりを実装しました。 実際のスクロールは、現在のビューを左または上（前方にスクロールする）および右または下（後方にスクロールする）にドラッグすると発生します。



アルゴリズムは簡単です。 このビジネスはすべて、UIScrollViewControllerの後継であるDocumentViewControllerに実装されています。 ビューを格納するためにリスト（配列ではなく）を使用します。 現在のビューを移動すると（次のページが最初または最後でない場合）、新しいビューが作成されてリストに追加され、目的のPDFページが新しいビューにロードされ、未使用のビュー（リストの最初）がリストから削除されます。 さらに、SrollView.Positionは中央の表示位置にシフトされます。 ビューリストの2番目。 



public virtual void OpenDocumentPage(int pageNumber) { if (PDFDocument.DocumentHasLoaded && (pageNumber != PDFDocument.CurrentPageNumber)) { if ((pageNumber < 1) || (pageNumber > PDFDocument.PageCount)) { return; } // Set current page PDFDocument.CurrentPageNumber = pageNumber; // Calc min, max page int minValue; int maxValue; if (PDFDocument.PageCount <= MaxPageViewsCount) { minValue = 1; maxValue = PDFDocument.PageCount; } else { minValue = PDFDocument.CurrentPageNumber - 1; maxValue = PDFDocument.CurrentPageNumber + 1; if (minValue < 1) { minValue++; maxValue++; } else if (maxValue > PDFDocument.PageCount) { minValue--; maxValue--; } } // Create/update page views for displayed pages var unusedPageViews = new List<PageView>(mPageViews); RectangleF viewRect = GetScrollViewSubViewFrame(); for (int i = minValue, j = 0; i <= maxValue; i++,j++) { PageView pageView = mPageViews.FirstOrDefault(v => v.PageNumber == i); if (pageView == null) { pageView = new PageView(viewRect, i); mScrollView.AddSubview(pageView); mPageViews.Add(pageView); } else { pageView.Frame = viewRect; pageView.PageNumber = i; pageView.ZoomReset(); unusedPageViews.Remove(pageView); } viewRect = CalcFrameForNextPage(viewRect); if (i == PDFDocument.CurrentPageNumber) { mCurrentPageView = pageView; } } // Clear unused page views foreach (var view in unusedPageViews) { view.RemoveFromSuperview(); mPageViews.Remove(view); } // Update scroll view content offset UpdateScrollViewContentOffset(); } }




ページの高品質な表示とスケーリング







ページを表示するためのビューは、実際には3つのビューで構成されています。

  • PageView（UIScrollViewから継承）がDocumentViewControllerに追加されます。
  • PageContentView（UIViewから継承）がPageViewに追加されます。
  • ThumbView（UIPageViewから継承）がPageViewに追加されます。


最初のビューはページを拡大縮小するために必要であり、2番目は実際にページの高品質表示のため、3番目はページの一時的な画像のために必要です。



スケーリング時のページの高品質表示のために、CATiledLayerクラスの子孫が使用されました。 このクラスを使用すると、ビューを拡大しながら詳細なコンテンツを表示できます。 しかし、すべての費用を支払う必要があります-CATiledLayerを使用すると、ページの非常に長い初期レンダリング（数秒）が発生し、不快なちらつきが発生し、古い画像が新しい画像に段階的に置き換わりました。 この悪影響を回避するために、別のビュー-ThumbViewを追加しました。これにより、低解像度PDFページのプライマリ出力が作成され、次にページのメイン画像の形成と出力が行われました。



実際にPDFページ自体を表示するのは非常に簡単です。 



 private void Draw(CGContext context) { if (!PDFDocument.DocumentHasLoaded) { return; } // Draw page context.SetFillColor(1.0f, 1.0f, 1.0f, 1.0f); using (CGPDFPage pdfPage = PDFDocument.GetPage(mPageNumber)) { context.TranslateCTM(0, Bounds.Height); context.ScaleCTM(1.0f, -1.0f); context.ConcatCTM(pdfPage.GetDrawingTransform(CGPDFBox.Crop, Bounds, 0, true)); context.SetRenderingIntent(CGColorRenderingIntent.Default); context.InterpolationQuality = CGInterpolationQuality.Default; context.DrawPDFPage(pdfPage); } }




ブックマークとメモ





ドキュメントのブックマークとメモを操作するためのクラス（マネージャー）を設計するとき、開発者がデータを保存する独自のメカニズムを実装する機会を提供したいと考えました：XML、データベースなどに。ライブラリを使用する私のアプリケーションでは、ブックマークとメモに関するデータをSqliteデータベースに保存します。 デモアプリケーションでは、データはアプリケーションの実行中にメモリに保存されます。



実際、両方のマネージャーは互いに似ています。 データをロード、保存、削除するメソッドがあります。 ほとんどのメソッドは仮想です。 メソッドは、それぞれDocumentBookmarkとDocumentNoteのデータオブジェクトを操作します。



なぜなら、Monotouchには、マネージャーをアクティブにするメカニズムであるジェネリック型のような素晴らしいものはないからです。私はそれがあまり好きではありません。 アプリケーションの起動時にマネージャーをインスタンス化するメソッドを持つObjectActivatorクラスを作成しました。 したがって、マネージャークラスの継承の場合、新しいインスタンスをObjectActivatorに登録する必要があります。 



 public class MyObjectsActivator : ObjectsActivator { /// <summary> /// Returns DocumentNoteManager instance /// </summary> /// <returns></returns> protected override DocumentNoteManager CreateDocumentNoteManager() { return new MyDocumentNoteManager(); } /// <summary> /// Returns DocumentBookmarkManager instance /// </summary> /// <returns></returns> protected override DocumentBookmarkManager CreateDocumentBookmarkManager() { return new MyDocumentBookmarkManager(); } }




ページのサムネイル







ページのサムネイルを表示するには、ページ自体を表示するのと同じアプローチが使用されます。ただし、サムネイルの場合は、一度に複数のページが表示され、新しいビューを作成するためのアルゴリズムが少し異なります。 また、サムネイルのScrollViewコンテンツ領域の幅は、すべてのページのサムネイルの幅とパディングの合計です。



正直なところ、サムネイルの表示をいじる必要がありました。 最初に、サムネイルの数が表示されるサムネイルの数に等しくなるアルゴリズムに、左側に1つ、右側に1つを加えたアルゴリズム（つまり、ページ表示とまったく同様のアルゴリズム）を実装しようとしました。 同時に、レンダリングを高速化するために、移動の反対側にあるより多くの不要なサムネイル（非表示）は削除されませんが、移動の方向に移動されます。 つまり それはある種のサイクルになります。 私はこの考えを捨てなければなりませんでした、なぜなら 高速サムネイルがスクロールされると、Scrolledイベントが100ピクセル以上のギャップで発生し、サムネイル間に「ギャップ」が生じました。 間違った場所に。



その結果、現在表示されているサムネイルは常にその場所に残り、移動する場合はバッファーがいっぱいになったときにのみ置き換えられる「バッファー」アルゴリズムに決めました。 バッファのサイズはオプションで設定されているため、パフォーマンスを向上させるには、当然のことながら、メモリの消費量を増やす必要があります。
