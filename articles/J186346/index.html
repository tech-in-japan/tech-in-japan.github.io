メール管理者の理論上の最小値。 Communigate Proサーバーでメールを設定する





この記事では、メールサーバーの管理を開始するために知っておく必要のある基本情報を提供します。一部のテキスト（〜30％）は標準プロトコルまたは広く使用されている慣習を示しているため、 Communigate Proだけでなく関連しています。





管理者

アカウントを作成する



前の記事で中断したところからセットアップを続行します。 サーバーは既にインストールされており、メインドメインの名前が選択されています。 ユーザーを始めましょう。 [ユーザー]- > [ドメイン]管理ページに移動して、メインドメインを選択します。







ドメインにはすでに2人のサービスユーザー（postmaster-メイン管理者とpbx-に代わって、デフォルトの音声アプリケーションが起動される設定を持つ技術アカウント）があります（音声機能の詳細については、以下の記事で説明します）



新しいユーザーの作成は簡単です。たとえば、[アカウントの作成]ボタンの横にあるテキストボックスに受信者などの名前を入力してクリックします。 アカウントのパスワードを入力できる新しいユーザー設定ページが表示されます-[パスワードの通信]フィールド：







レセプション



これは設定するのに最も時間のかかるメールセクションです。これは主にスパムとの戦いによるものです。 一方では、不要なメールを受け入れたくない（もしあれば、レキシカルスパムフィルターを介してあまりにも多くのメールを送信せず、単にゴミをサーバーにロードしないようにするため）、一方、通常の送信者を拒否したくない



DNS


新しく作成されたアカウントへの手紙は既に受け入れることができますが、受信者の名前のドメイン部分でIPアドレスを使用する必要があります。 これはユーザーにとって非常に不便なので、すべてのメールプロトコルはDNSを使用します。



メール配信の主なプロトコルはSMTPで、サーバー間およびクライアントからサーバーの両方で使用されます（その逆は使用されません）。

この場合、さらに送信するためにメールクライアントからサーバーにレターを配信するプロセスは、レターの登録（送信）を呼び出します。送信は、宛先にレターを配信するプロセスです（SMTPの場合、アドレスは他のサーバー上にあります）。



このプロトコルを完全に動作させるには、MXタイプのDNSレコードが必要です。 これらには、メールドメインの名前、優先度、ドメインを提供するサーバーの名前の3つのフィールドが含まれます。 サーバー名ごとにタイプAのDNSレコードが存在する必要があります。



最も優先度の高いレコードがメインメールサーバーと見なされ、残りはバックアップサーバーです。



例： 

>nslookup -type=MX google.com google.com MX preference = 50, mail exchanger = alt4.aspmx.l.google.com google.com MX preference = 10, mail exchanger = aspmx.l.google.com google.com MX preference = 20, mail exchanger = alt1.aspmx.l.google.com google.com MX preference = 30, mail exchanger = alt2.aspmx.l.google.com google.com MX preference = 40, mail exchanger = alt3.aspmx.l.google.com aspmx.l.google.com internet address = 74.125.143.26 alt1.aspmx.l.google.com internet address = 173.194.64.26 alt2.aspmx.l.google.com internet address = 74.125.142.26 alt3.aspmx.l.google.com internet address = 74.125.140.26 alt4.aspmx.l.google.com internet address = 173.194.74.26




クライアントIP（クライアントIP）


Communigate Proには、クライアントアドレスという概念があります。 実際、これらは信頼できるIPアドレスです。通常、通常のIPアドレスと比較して多くの特権があり、セキュリティを担当する一部の設定では、「クライアントのみ」と「クライアント以外の全員」の値を選択できます。



CGPro管理パネルでは、クライアントアドレスは「アドレスリスト」タイプフィールドによって設定されます。このタイプのフィールドは他の設定でも積極的に使用されます（設定->ネットワーク->クライアントIPページ）：







受信機




Communigate Proが接続を受け入れることができる各プロトコルには、SMTP（設定->メール-> SMTP->受信->「リスナー」）などの受信者（ソケットを作成するサーバーオブジェクト）の独自のリストがあります。







各レシーバーは、特定のポートと特定のIPアドレスでソケットを開きます（これは、CGProがWebサーバーと同じマシン上に立つために必要です-Webサーバーは、一方のIPで80番目のポートを占有し、他方でCGProを占有します）。



デフォルトでは、SMTPモジュールでは25番目のポートのみが構成され、さらに2つ追加するだけで、デフォルトで設定されます。







SMTPの最新バージョンでは、ポート25は主にサーバー向けですが、クライアントはSMTP AUTHコマンドによる認証の必須使用の違いである587番目を使用する必要があります。




保護




SMTPに加えて、サーバーにレターを登録できる多くのプロトコルがありますが、いずれもアカウントの1つに代わって認証で機能します。 SMTPは、任意のサーバーから文字を受信するように設計されているため、場合によってはそのような贅沢をする余裕はありません。



このため、スパマーの生活を少なくともわずかに複雑にするためにとるべき多くの対策があります。



中継


リレーとは、サーバーが受信者にさらに転送するためにレターを受信することです。



SMTPは、原則として、サードパーティサーバー向けのメールを登録できますが、実際には、信頼できる送信者を除くすべての送信者に対してこのオプションを閉じることをお勧めします。 。



リレー設定は、ページ<nobr /> Settings-> Mail-> SMTP-> Relayingにあり、デフォルト値は非常に最適です。



主なものは、偶然または特定の理由なしにそれらを変更しないことです。



ヘッダーチェック


SMTP文字を受信するときのメインヘッダーは、送信者と受信者です。 SMTPセッションの例を考えてみましょう。 



 220 mycgpro.com ESMTP CommuniGate Pro 6.0.4 is glad to see you! helo 250 mycgpro.com domain name should be qualified Mail from:sender@gmail.com 250 sender@gmail.com sender accepted rcpt to: recipient@mycgpro.com 250 recipient@mycgpro.com will leave the Internet data: 354 Enter mail, end with "." on a line by itself From: Name That Everybody See <tvoya@mama.com> Tra ta ta Prishli mne deneg na telephon! . 250 90001 message accepted for delivery




ここでは、送信者はMAIL FROMコマンドで指定され、受信者はRCPT TOです。 管理者が知っておく必要がある最も重要なことは、すべてのメールクライアントによって「From who」（レターの「From」見出し）として表示されるフィールドは、実際には送信者ではなく、単にメッセージ本文の一部であるということです。 ほとんどのヘッダーチェック（一般的なリモートブラックリストを含む）は、プロトコルコマンドセットの送信者で特に機能しますが、メッセージの本文についてはわかりません。 MIME形式では、「Return-path」フィールドは送信者に対応しています。





（MIME形式とインターフェイスで上記のSMTPセッションで受信した手紙）



「Return-path」チェックの本質は、Mail Fromコマンドを受信すると、サーバーがアドレスのドメイン部分を取得し、DNSにこのドメインが存在するかどうかをチェックすることです。 このチェックの拡張バージョンもあります-逆接続、このチェックを使用すると、CGProは送信者サーバーに接続し、このサーバーがMail Fromコマンドからの名前を持つ送信者の文字を受け入れるかどうかを確認します。



かなり一般的なリターンパスチェックは、SPFチェックです。 特殊な形式のTXTタイプのDNSレコードが必要です。これらのレコードはSPFレコードと呼ばれます。 これらには、メールドメインの名前と、このドメイン部分を持つ文字の正当な送信者であるIPアドレスのリストが含まれます。 例： 



 >nslookup -type=TXT google.com google.com text = "v=spf1 include:_spf.google.com ip4:216.73.93.70/31 ip4:216.73.93.72/31~all"




この方法の主な欠点は、サーバー管理者がこのチェックを認識して使用している送信者に対してのみ機能することです。 WebAdminでは、メール受信設定は1ページの設定->メール-> SMTP->受信で収集されます：







これらのタイプのチェックはかなりよく知られているため（主にロシアとCISからの多くの人々が定期的に自分自身を見つけているため）、RBLと通常のブラックリストは考慮しません。



レター処理



手紙を受け取ると、すぐにベースディレクトリのキューフォルダ内のハードドライブに拡張子.tmpが書き込まれます。 受信プロセスが完了するとすぐに、拡張子が.msgに変更され、メッセージを受信するモジュールがそれを一般キューに入れます。



サーバーの突然の再起動またはシャットダウンの場合、キューからの文字（Queueフォルダーの.msgファイル）がサーバーに再び入れられます。



キュー




キューでは、1つのモジュールによって登録されたレターが、他のモジュールやユーザーボックスへの配信を待っています。



メッセージキューで次のイベントが発生します。



  • 受信者アドレスの分析と変換。
  • サーバー全体のルールを適用します。
  • キューファイルは1つ以上の論理キューに配置されます（どのモジュールがレターの受信者であるかに応じて）
  • ドメインまたはユーザールールを適用する
  • 受信者モジュールでレターを処理します。




ルーター


Communigate Proは、郵送先住所を検出するたびに、ルーターモジュールを介してその住所を渡します。 管理者は、メッセージルーティングを管理するための補助ツール（ルール、ドメイン、ユーザーエイリアスなど）に加えて、アドレス処理プロセス（ルーティングテーブル）に直接アクセスできる便利なツールを備えています。







この表の各行の形式は次のとおりです。



[リレープレフィックス：] [レコードタイププレフィックス：] left = right [; コメント]



テーブルのスキームは次のとおりです。レコードは、先頭から順に1つずつ表示され、次のレコードがアドレスの変換に使用された場合、新しいアドレスが最初からルーティングモジュールへの入力に送られます。



デフォルトでは、テーブル内のエントリは、すべてのサーバー操作および機能で見つかったアドレスに対して機能します。 録音は、対応するプレフィックス（「メール」、「アクセス」、「信号」）が付いている場合、3種類の操作のいずれかでのみ機能します。



プレフィックスには別の種類の「リレー」があります。 このような接頭辞を持つレコードが受信者の一部に対して機能した場合、レターは特別なメモを受け取り、信頼できるソース（クライアントIPまたは認証されたユーザー）から来たかどうかに関係なく送信されます。 これは、スパマーがリダイレクト先のサーバーに自由に手紙を送信できるため、かなり危険な設定です。



論理キュー




レターを送信するためのキューには、他のサーバー（SMTP）、ローカルメールボックス（LOCAL）、サードパーティプログラム（ PIPE ）、およびメーリングリスト（LIST）の4種類があります。



これらの各行は、いくつかの他の行に分割されます。 たとえば、各受信者ドメインにSMTPキューがあり、各アカウントにローカルキューがあります。 これにより、1つの接続またはメールボックスを開くときに多くの文字を、大量に配信できます。



ルール




ルールは2つのアプローチで適用されます。最初は、すべてのレターのサーバールール（配信キューに配置される前）、ドメインルールとユーザールール、ただしローカルローカルメールキューからのレターのみです。 内部デバイスの観点から見たドメインルールは、ユーザーのルールとはまったく区別できませんが、すべてのドメインアカウントに適用されます。



このようなルールを適用するシステムは、ドメインルールとユーザールールが受信メッセージにのみ影響することを意味します（送信メッセージはLOCALキューに分類されません）。 つまり、サーバールールのみが送信メッセージの操作の実行に適しています。



受け取りボックス





送信の場合と同様に、POP、IMAP、XIMSS、HTTPサーバー（AirSync、WebUser）に保存されているメールボックスのコンテンツを取得できる多くのプロトコルがあります。 しかし、ほとんどの場合、メールボックスの受信には、情報を要求するクライアントの認証が含まれます。



これらのプロトコルには特別な設定はありません-すべてがすぐに動作するはずです。



ドメイン設定を含めて、クライアント設定でユーザー名を完全に指定することをお勧めします（Outlookの一部のバージョンのように、クライアントが「@」記号でドメイン部分を自動的にカットする場合は、代わりに「％」を使用できます）。



まとめると



Communigate Proの初心者メール管理者が遭遇する最も重要な設定を調べました。 この記事は、レターの登録と送信の設定に対してわずかな偏りがあります。これは、ルーターやルールなどの広範なトピックが小さな記事のフレームワークに収まりにくいためです。 オンラインマニュアルで、さらにそれらに精通することをお勧めします。
