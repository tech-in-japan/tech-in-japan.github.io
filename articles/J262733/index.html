HP Superdome Xの保護具

こんにちは、Habr！ この出版物では、 HP Superdome Xについて、またはそれが持つ保護メカニズムのいくつかについて再度説明します。







高度なエラー回復エンジン



Linux OSの例でのメモリ内のファームウェアHP Superdome Xの修正不可能なエラーの開発を示します。 メモリ内の修正不可能なエラーを処理するメカニズムを検討してください。



ユーザーアプリケーションでメモリエラーが発生し、HP Superdome X Serverファームウェアがこのエラーを検出し、ファームウェア-MCAリカバリメカニズムがエラーを含むメモリ領域を特定します。 サーバーでメモリエラーが発生する頻度は？ あなたはどれくらいの頻度で驚くでしょう！ データセンターで実施された Googleの調査では、メモリ内の修正不可能なエラーが非常に一般的であることが示されました（原因は、アプリケーションエラーと宇宙放射線などの外部要因の両方である可能性があります）。



同時に、モジュールで修正不可能なエラーが発生する確率は、ECCまたはChipkillによって修正された修正済みエラーでも少なくとも1回は生き残っているため、サーバーパーク内のそのようなモジュールを詳しく調べて、可能であれば重要なアプリケーションをそれらに置かないでください） Googleデータセンターのメモリモジュールの約8％でRAMの誤動作が発生しています。 ちなみに、このレポートは、「老化効果」への記憶暴露のような興味深い事実を明らかにしています。







HPの調査により、このデータが確認されています-パレートの原則に基づくサーバーコンポーネント障害の年間統計（ACR、年間クラッシュ率）を以下に示します（メモリ分析は分析されたサーバーでは使用されず、標準で最も一般的なSDDC + 1修正モードが顧客によって使用されました） 4チップごとに1チップが修正されます）：







したがって、このクラスのサーバーには、特にビジネスクリティカルなタスクの場合、アプリケーション、OS、およびサーバーに障害を発生させることなく、メモリエラーを分離できるメカニズムが非常に重要です。



エラーの検索は、常時メモリチェックのパトロールスクラバーメカニズムによって実行されます。 エラーを検出した後、ハードウェアサーバーはエラーの修正を試みます（ECCまたはChipkillを使用）。 ハードウェアがエラーを修正できない場合、OSアラートが発行されます。 その後、HPメモリ検疫メカニズムは、データを破損する前にエラーを分離します。これにより、メモリエラーが原因でアプリケーションがクラッシュする頻度が減少します。 さらに、それ以降の呼び出しを避けるために、障害のあるセクションは分離され、メモリモジュールは次のスケジュールされた診断手順中に交換できます。



このプロセスと並行して、Linuxカーネルは失敗したメモリアドレスを「記憶」し、このメモリロケーションを使用してアプリケーションにSIGBUSシグナルを送信します。 このような信号を受信するアプリケーションは、作業を停止することなく、影響を受ける領域を別のメモリアドレスに移動し始めます。



HP Superdome Xに加えて、このメカニズムは4ソケットHP DL580 Gen9サーバーでも使用されます。

HP Superdome Xのライブエラー回復エンジン



Linux OSの例でのファームウェアHP Superdome X I / Oエラーの発生を「オンザフライ」で示します。 ご存じのように、PCIバスはシリアルバスアーキテクチャのアーキテクチャに基づいて構築されています。つまり、このバスで発生したエラーは、このバスで動作する他のデバイスに伝播する可能性があり、データの一貫性が失われる可能性があります。 18を超えるI / Oエラーの可能性が文書化されており、PCIデバイスを追加すると、このようなエラーが発生する可能性が高くなります。 これを知って、Intel Live Error Recoveryと対話するメカニズムがHPファームウェアに追加されました-このようなエラーが発生した場合のI / O分離。



I / Oエラーが発生すると、Intel Live Error Recoveryはエラーを分離し、OSまたはアプリケーションのクラッシュを防ぎます。 同時に、Intel Live Error Recoveryは、HPマイクロコードにそのようなエラーの発生を通知し、その後、I / Oを停止して、破損したデータがサーバーの外部に漏れないようにします。 次に、HPファームウェアは、I / Oデバイスの上位ドライバーとOSにエラーを通知します。



ファームウェア機能により、Linuxは管理者またはサポートサービスによるI / Oエラーの詳細な調査のための拡張レポート（syslog）を作成できます。



さらに、エラー分析エンジンはファームウェアに含まれており、I / Oエラーを分析し、考えられる原因についてメンテナンススタッフに推奨事項を発行します。 デモビデオでは、標準サーバーとHP Superdome Xのネットワークカードでのエラー処理の比較と、これらのサーバーのLOGファイルの比較を見ることができます。



このエラーの処理のビデオはHP Superdome Xです。HPSuperdome Xに実装されている他のRAS機能の詳細については、「 HP Superdome XシステムアーキテクチャとRAS 」を参照してください。



したがって、HP Superdome XマルチプロセッサシステムのHPファームウェアの独自性は、信頼性、可用性、使いやすさ（RAS機能）の観点から、サーバーコンポーネント（プロセッサ、メモリ、デバイス）のすべての機能を実現できることです。 HP Superdome Xとミッションクリティカルなタスクへの適用性を分析したIDCレポート 。



ブレードサーバーの電気的分離のためのパーティションおよびエラー分離（パッシブミッドプレーン）メカニズム



新しいHP Superdome Xの重要な機能は、ブレードパーティションの電気的分離です。 パーティショニングにより、HP Superdome Xを、複数のブレードサーバーで構成される1つの大きなシステムと、独立した独立した複数の小さなシステムとして構成できます。 各セクションには、CPU、メモリ、I / Oの独自の独立したセットがあり、共通のPCIバスを備えたマルチプロセッサシステムとは対照的に、ブレードサーバー全体に障害が発生した場合でもシステムは動作し続けます。







CPU間に共通のミッドプレーンを備えたシステム（A）は、ノード間のエラー伝播に対して潜在的に脆弱であり、共有バスに依存しているため、システム全体のパフォーマンスも制限され、電気的に独立したパーティションとは異なり、大量のCPU-CPUアクセスを迅速に処理できませんHP Superdome XのnPar（B）。これらの欠点はありません。



この機能はSuperdome Integrityプラットフォームから移行されたもので、さまざまなタスクに合わせてHP Superdome Xバスケットのリソースを柔軟に区別できます。 たとえば、1つのHP Superdome X上のデータベースの場合、複数の環境（生産、テスト、開発）を一度に実行し、仮想化を追加して、同じパーティション内に複数のデータベースコンテナーを配置できます。 このアプローチは、コンポーネントの物理的な移動を必要とせず、管理者コンソールから実行できます。
