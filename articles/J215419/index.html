私はそれが正しいと信じています：

  • 要求がWebサーバー（GETメソッド）からゲートウェイに届いた場合、ゲートウェイはこの要求をパラメーターに解析し、要求をオンエアでブロードキャストします。 その後、iBoardは応答を受信（または受信しない）し、応答（ JSON ）を生成します 。


答えとしてサーバーにXMLを提供することは可能ですが、控えめなatmega328リソース（iBoardはこのMK上に構築されています）に実装するのは簡単ではありませんが、より長い行で操作する必要があります。



原理が説明されている、問題は特定です。



LANゲートウェイ<=> RF24



まず、ゲートウェイへのGETリクエストを決定します（IP 192.168.1.2にします）。 次の形式のクエリを使用します。 

http://192.168.1.2/?sid=701&cmd=2&pid=1&val=0


要求は、アクセスされているモジュール（sidパラメーター）、コマンド（cmd：1-読み取り、2-設定）、モジュールパラメーター（pid）、および必要な値（val）に関する情報を送信します。

「人間が理解できる」、このコマンドは次のように聞こえます：「浴室のスイッチ、ライトを消す」（ 前の投稿を参照）。


操作に必要なパラメーターの最小数は3（sid、cmd、pid）です。受信するパラメーターが少ない場合、モジュールは報告および報告する必要があります。 

 { "message": "Bad request" }


モジュールが要求への応答で「報告」した場合、ゲートウェイは応答を発行する必要があります。 

 {"message": "OK","sensor": 701,"parameter": {"pid": 1,"pval": 0.000,"st": 1,"note": "Ch.1 (Light)"}}


回答には、対応するパラメーターに関する完全な情報が含まれます（回答は、無線通信で使用される構造に可能な限り近い）。



回答が届かない場合： 

 { "message": "No answer" }


ゲートウェイはどういうわけかいくつかのタスクを取得したようです。そのため、NTPを介して正確な時間を取得する機能を使用して「わずかにロード」します。



つまり ゲートウェイは、「日付」と「時刻」という2つのパラメーターを持つ「センサー」にもなります。 これを行う方法についてはすでに説明しました（ 「スマートホーム」のワイヤレス通信を参照してください。 実用的な第2部 ）。



繰り返しますが、システムのこの非常に重要なコンポーネントの「番犬」を忘れないでください。



システムのすべての機能を説明するとき、 適切なスケッチを書くことは技術の問題です。



iBoardをフラッシュし、LANケーブルを接続し、電源を投入します。すべてが正常であれば、GETリクエストを介してステータスを取得し、ワイヤレスモジュールを制御することがすでに可能です。 問題の一部はすでに部分的に解決されていますが、まだ便利ではありません（すぐに修正します）。



すべてのワイヤレスモジュールまでの距離が最小になるように、ゲートウェイを中央（アパート、家）に配置することをお勧めします。

サーバー



明確にするため、サーバーのIPアドレスは192.168.1.10です。 このアドレスは上記のスケッチに表示されます。ゲートウェイからデータを受信するページのアドレスもあります： 192.168.1.10/sensors-このページから始めます。



データベース


ただし、ページの開発を開始する前に、データベースを準備します。 最も簡単な方法は、phpMyAdminでこれを行うことです。



db_sensorsというデータベースがあり、sensor_listとsensor_valueの2つのテーブルのみが含まれています（テーブルの目的は名前から完全に推測されます）。

テーブルを作成するスクリプト 
 -- --   `sensor_list` -- CREATE TABLE IF NOT EXISTS `sensor_list` ( `id` int(11) NOT NULL AUTO_INCREMENT, `name` varchar(16) CHARACTER SET utf8 COLLATE utf8_unicode_ci NOT NULL, `unit` varchar(16) CHARACTER SET utf8 COLLATE utf8_unicode_ci DEFAULT NULL, `comment` varchar(100) CHARACTER SET utf8 COLLATE utf8_unicode_ci DEFAULT NULL, `SensorID` int(11) NOT NULL, `ParamID` int(11) NOT NULL, `created` timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP, `value` float NOT NULL COMMENT ' ', `last` datetime NOT NULL COMMENT '  ', PRIMARY KEY (`id`) ) ; -- -------------------------------------------------------- -- --   `sensor_value` -- CREATE TABLE IF NOT EXISTS `sensor_value` ( `id` bigint(11) NOT NULL AUTO_INCREMENT, `sensor_id` int(11) NOT NULL, `value` float NOT NULL, `created` timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP, PRIMARY KEY (`id`) ) ;


データベースの準備ができたら、データを受け入れてデータベースに保存するページを作成します。



データロギング


デバッグを簡単にするために、このページを、データを自分自身に投稿し、その有効性を確認するフォームの形式にします。



さらに、「私たち」からのリクエストのみを受け入れるために、必要なリクエストデータのリストに正しいキー（ゲートウェイスケッチで明示的に設定）を入力します。 すべてが安全すぎるわけではないことは明らかですが、免責事項に基づいて、それはまったく受け入れられます。



スクリプトのロジックは次のとおりです。

  • 必要なすべてのパラメーターの初期チェックと、キーの有効性のチェック。
  • sensor_listテーブルにそのような「センサー」に関するエントリがあるかどうかを確認します。 そのようなレコードがない場合、レコードを作成してレコード識別子を取得します。それ以外の場合は、既存の識別子を取得します。
  • テーブルsensor_valueに、「センサー」の対応する値を書き込みます。
  • さらに、sensor_listテーブル内の対応するセンサーの最後の値を更新し、最後の連絡の時刻を修正します。


後者は、最後の値をすばやく取得できるように（最初のテーブルからのサンプリングがはるかに高速）、このデータの関連性をすばやく評価できるように行われます。



ところで、 ユニバーサルウィジェットのデータとして現在のデータを使用してこのテーブルの選択を使用すると便利です（非常に長い間プロジェクトが更新されていないのは残念です）。


スクリプトは準備され、ファイルはindex.phpという名前で、\ sensor \フォルダーに配置されました（サイトのルートに対して）。



Webサーバーのすべてのファイルを含むアーカイブは、リンクから入手できます。


今、あなたは彼に連絡することができます： 192.168.1.10/sensors



すべてが正常に完了すると、フォームがこのアドレスで開き、テスト値を手動で挿入し、（たとえば、phpMyAdminを介して）データがチェックされ、データベースに正しく配置されていることを確認できます。



すでに無線スイッチとゲートウェイがある場合、このデータもデータベースに送信されます。









ロギングのタスクを解決しました。ワイヤレスモジュールのパラメーターを読み取り、変更するより便利な方法を実装しています。



モジュールパラメータの読み取りと変更


別のスクリプト（ 192.168.1.10/tests ）を作成してみましょう。これは、サーバーがゲートウェイへの要求を形成し、応答を受信し、同じ形式で表示するためのフォームでもあります。









上の画像は、フォームの処理結果をすでに示しています。2つの上部フィールドとセレクターがリクエストのデータを形成します。「OK」は応答のメッセージパラメーターの値、残りのパラメーターは応答の対応する部分です。



応答の解析はJavaScriptを使用して行われます。



このスクリプトにはちょっとしたトリックがあります。 フォームデータはゲートウェイに直接送信されませんが、json-proxy.phpファイルを通過します



このアプローチにより、外部ネットワークから「メイン」スクリプトにアクセスできますが、ゲートウェイは内部ネットワークにとどまることができます。



この中間ファイルでは、SQLインジェクションから身を守るために、完全なデータチェックを行う必要があります。


このページは既に操作するのにはるかに便利ですが、これまでのところ、すべては特定のセンサーの1つのパラメーターのレベルにあります。



コントロールセンターホーム


現在、私たちの成果を組み合わせて、AJAXについて少し読んで、そのようなページを作成することが既に可能です（ 192.168.1.10/tests/switch.phpで ）：







特定のパラメーターのデータを更新するには、関数chekIt（701、1、 'ch1'）が呼び出されます。 （実際には、モジュール番号、パラメーター番号、div識別子の3つの引数があり、その内容を更新する必要があります）。



このページには、モジュールの時間パラメーターが表示されます（脆弱なリソースゲートウェイに負担をかけないようにするため）-このデータは5分ごとにリクエストします。



照明および換気チャネルの状態に関するデータは、5秒に1回（より頻繁に更新されます（多くの場合、対応するオンまたはオフボタンを押した後、状態の変化を長時間待つ必要がないように十分です））。



doIt（）関数の呼び出しはボタンで「ハング」します： 

 <input type="button" value="" onclick="doIt(701,1,1);">


関数パラメーター：モジュール番号、パラメーター番号、および新しい値。



ページの一番下に、ゲートウェイの応答が表示されます。



ページ上のデータは再起動せずに表示され、すべてが「単独で」行われます。



結果



  • 現在、無線制御スイッチは単独で機能するだけでなく、Webページから簡単に制御できます。
  • ユニバーサルLANゲートウェイ<=> RF24が作成されました。これにより、さまざまな家庭用ワイヤレスデバイスを操作できます。 同じ無線プロトコルで新しいデバイスを追加するだけです。ゲートウェイコードを変更する必要はありません。
  • 後処理（たとえば、分析や美しいグラフの作成など）のためのセンサー値の永続的なログが構成されます。
