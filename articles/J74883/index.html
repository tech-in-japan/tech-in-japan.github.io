ビジュアルスタジオの拡張性



ご存知のように、 Microsoft Visual Studioは開発環境（ IDE ）です。 さらに、拡張可能な環境。



環境は単純ではなく、長い歴史があり、それに基づいて深刻なものを実装できるという事実のために、Visual Studio Extensibility（ VSX ）という用語が登場しました。



スタジオには10年の歴史があります。 COM開発のために、 COM時間で作成されました（彼らが合格しなかったことは知っています）。 その後、彼女は主な目的を.NET開発に変更しました。 ただし、COMベースのアプリケーションのままであったため、主な拡張性はそのCOMの性質に直接関係しています。 そのために、拡張性、.NETおよびCOMの相互運用性が使用されます。 これは間違いなく、拡張プロセスに魅力と特定の魅力をもたらします（皮肉のほんの一部）。



バージョンからバージョンへのスタジオコアは基本的に2002バージョンと同じであり、新機能のほとんど（VS2005、VS2008、VS2010）はVSXのパワーを使用して実装されています。



残念ながら、VSXで作業する開発者は、長年にわたってさまざまな人々によって開発され、何度もやり直されてきた「魅力的なコード」に対処する必要があります。 また、スタジオの性質と、場合によっては開発のスケジュールが厳しいことを考慮すると、開発者がどれほど多くの興味深い驚きを見ることができるか想像できます。 残念ながら、エンティティに名前を付けることもやや不自由です。



そして最後に、広範で理解しやすいドキュメントの入手可能性、またはむしろその不在などの側面は、プロの開発者によるVSXの使用に不可解で魔法のような魅力をもたらします。 既に実装されているソリューションを研究し、それらに基づいて新しいソリューションを発明することよりも、何がエキサイティングなことでしょうか？



スタジオを拡張する理由



実際、なぜスタジオ拡張が必要な​​のですか？ 考えてみると、私たち（つまりドナー）が日常生活で使用している多くのVSXベースのソリューションを見つけることができます。 WPF 、 SilverlightおよびXAML 、 WCF 、 LINQ 、 Office用ツール 、 Crystal Reports 、 Workflow Foundation 、テストプロジェクト-MSの開発。以前のバージョンのスタジオにはありませんでしたが、進歩のために登場しました。 次に-実際にはVSXは、それ自体に基づいており、統合パッケージ、DSLツールなどが含まれています。 有名なReSharper 、 AnkhSVN 、 XMLSpy 、 GhostDoc 。 おそらくあまり知られていないCodeRush 、 TestDriven.NET 、 Smart Paster 。 .NETの非基本言語はF＃ 、 IronPython 、 IronRuby 、PHPです。 これらすべては、ある程度、VSXに基づいています。



すぐに注目する価値があります。スタジオ拡張には、分離モードと統合モードの2つのバージョンがあります。 ここでの違いは、スタジオコア自体とそのAPIが常に利用可能であることですが、通常のスタジオに存在する機能は統合モードでのみ利用可能です（たとえば、sysharpとvisual-basic、Webプロジェクト、あらゆる種類のウィンドウとツール）。 孤立した状態では、シェル、シェルのみを取得します。



XNA Game Studio 、 Microsoft Robotics Studio 、 SQL Server Management Studio 、 Microsoft Visual Studio Macrosは、分離されたシェルに基づいて頭に浮かんだ最初の製品です。 また、Express-studiosの全ライン：Visual Basic 2008 Express Edition、Visual Web Developer 2008 Express Edition、Visual C ++ 2008 Express Edition、 Visual C＃2008 Express Edition-同じルートを持っていると思います。



ところで、Habréには、VSXの使用例に関する出版物がありました。

  • Project Navigator for Visual Studio ;
  • Visual Studio-デバッガービジュアライザーのデバッグ支援 。
  • VS 2010および.NET 4.0のScottGu：VS 2010のプロジェクトテンプレート 。
さて、スタジオを拡張する理由をもう少し明確にしたいと思います。 VSXは、特にVSX自体が開発中であるという事実を考慮すると、有用なものであるように思えます。 新しいVisual Studio 2010は、拡張機能を実行する予定のユーザーにとってより使いやすいはずです。 少なくとも、.NET Framework 4.0にはManaged Extensibility Frameworkが含まれるという事実のため（Vladek habrayuzerが思い出したように ）。



統合モードの拡張機能は、非エクスプレスバージョンでのみ使用できることに注意することが重要です。 つまり、無料版では拡張機能をインストールすることは不可能です。 しかし、この統合モードを使用して、新しいプログラミング言語、追加の開発ツールを作成して実装し、既に取得してインストールしたスタジオで使用できます。 言語と技術の例はもっと高かった。



Visual Studio Standard、Professional、またはTeam Systemバージョンの高価なコピーを取得せずに実行したい場合は、特定のエディターなどのスタンドアロンの製品を作成するために使用できる無料のVisual Studio分離シェルがあります。



VSXの機能



2008年版と2005年版の拡張オプションには、3つの主なオプションが含まれていました。

  • マクロ（マクロ）、
  • アドイン
  • パッキング（統合パッケージ）。
私が理解しているように、2010thスタジオと新しいSDKはこのリストに何も追加していません。



拡張機能に戻ります。 マクロとアドオンは常に使用可能ですが、パッケージを作成するにはVisual Studio SDKが必要です。



VS SDKを最大限に活用するには、まずVS 2008 SDK 1.0をインストールしてから、 VS 2008 SDK 1.1をインストールすることをお勧めし ます 。 1.1では、何らかの理由でプロジェクトベースがありません。これについては後で説明します。



一般に、VS SDKをインストールした後、アドオンとパッケージを作成し、 Visual Studio 2008 Shell（分離モード）SP1 -VS Shellベースのソリューションをインストールした後、可能になります。 隔離されたシェルは10か所に存在します。



3つのオプションVSXのみが技術的に制限されています。 そのため、ビジュアライザー、ツールウィンドウ、プロジェクトテンプレート、 MSBuild拡張機能、 DSLツール、および前述のVS Shell Isolatedもあります。



順番に：

  • マクロ 。 これらを使用して、アクションをさらに繰り返して記録することができます。つまり、プログラムでアクションを自動化できます。 マクロの作成は、ビジュアルベーシック（VB.NET）に制限されています。
  • アドオン アドインは単なるコードではありません。 アドインは、コンパイルされたCOMコンポーネント（通常の.NETアセンブリ）としてシェルに統合され、環境とそのAPIにほぼ完全にアクセスできます。
  • パッケージ アドオンのようにシェルに完全に統合できますが、さらに独自のタイプのプロジェクトを作成し、新しいプログラミング言語を統合する機能も提供します。

MPF



VSXは、Managed Package Framework（MPF）に基づいています。 マネージコードを使用してパッケージを作成できます。 MPFには、複数のパッケージインターフェイスのカプセル化された実装が含まれているため、少ないコードで独自のパッケージを作成できます。

VSXtra



このツールキット（ VSXtra ）の作成者によると、パッケージの開発は以前よりもずっと簡単になりました。 多くのラッパー、他のパターンなどを使用して、彼はスタジオのCOMの性質と、.NETを介してCOMで作業するために必要な開発者の「統合失調症状態」を隠す努力をしました。



VSXtraによって解決される問題：

  • MPFの主要なスタジオサービスの不完全なカバレッジ。
  • 膨大な量の追加コード（実際の拡張機能を実装するために数千行）を書く必要性。
  • 時代遅れのアプローチを使用して、ジェネリックとLINQの欠如。
一般に、彼はMPFをラップし、必要なものを実現しましたが、これは実装されていませんでした。このラッパーは非常にモダンで調和がとれています。



この製品の主なタスクは、スタジオ開発チームに新しい最新のMPFを作成するようにプッシュすることです。 さて、サイド1は、開発者を支援し、著者を楽しませることです。



Visual Studio 2010 SDK



Visual Studio 2010 SDKには、拡張機能の開発を簡素化するいくつかの新しい要素が含まれています。

  • ウィンドウとコマンドを作成するためのテンプレート。
  • エディター拡張テンプレートの新しいセット。
  • VSIXマニフェストとVSIXコンテナを作成するための新しいビルドタスク。VisualStudioギャラリーにロードでき、拡張機能を共有できます。
  • VSIXマニフェストエディター
  • VSIXコンテナで既存の拡張プロジェクトをラップできる空のVSIXプロジェクト。
変更点：

  • DSLツールはSDKの外側に移動し、スタジオ自体で利用できるようになりました。
  • SDKの例はオンラインでのみ利用できます。
  • SDKドキュメントはオンラインでのみ入手可能です。
  • 拡張機能の開発プロセスの改善。
  • 拡張用の追加プロジェクトテンプレートを追加します。
  • 拡張機能を作成およびデバッグするための追加ツールの作成。
  • 10番目のスタジオ用の拡張機能を作成する追加の例があります。
Visual Studio 2010 SDKは、以前のバージョンに加えて、いくつかの新しいタイプの拡張機能を実装する機能を提供します。
  • Managed Extensibility Frameworkと組み合わせてパッケージを使用する
  • データデザイナーの機能を拡張する。
  • バージョン管理システムとの統合。
  • デバッガーのカスタマイズまたは新しいデバッガーの作成。
  • テスト用のツールの作成。

パッケージとMEFを使用してStudioを強化する



スタジオエディターは、パッキングに基づいて作成され、いくつかのMEF拡張が追加されました。 エディターの機能を拡張するには、これらの拡張機能を自分で作成できます。



Visual Studio SDKには、C＃、Visual Basic、またはVisual C ++を使用してパッケージとMEF拡張機能を作成するのに役立つツールとドキュメントが含まれています。

外部ソースを操作するためのデータデザイナーの機能の拡張



データデザイナーの機能（Data Designer Extensibility、DDEX）を拡張すると、多くの外部データソースをデザインモードで操作できます。 これらのソースの階層は、サーバーエクスプローラー（サーバーエクスプローラー）に表示されます。 DDEXを使用すると、外部データソースに関連するドラッグアンドドロップ機能を利用したり、スタジオデザイナーのプロパティやその他の機能の可視性を操作したりできます。



DDEX SDKは、独自のDDEXプロバイダーの作成に役立つドキュメント、例、およびその他の情報を提供します。



追加情報： Data Designer Extensibility Architecture 、 Data Designer Extensibility（DDEX）SDK

デバッガーのカスタマイズまたは作成



SDKを使用すると、Studioコンポーネントをカスタマイズしてデバッグしたり、IDEに統合する独自のコンポーネントを作成したりできます。



スタジオでは、デバッグはデバッグされたアプリケーションのプロセスの外側で行われます。 これで、デバッガーと対話する独自のコンポーネントを作成できますが、デバッグされたアプリケーションには影響しません。



Visual StudioデバッグSDKを使用するには、次のことを理解する必要があります。
  • スタジオ環境自体（Visual Studio IDE）;
  • C ++ ;
  • ATL COM。

テストツールの作成



スタジオには、環境の他の機能と緊密に統合されたテストツールが含まれています。 このツールキットは、独自のテストフレームワークだけでなく、ソフトウェアライフサイクルツールキットフレームワークとも連携します。



これらのツールを使用すると、手動テストと自動テストを作成、管理、編集、実行して、テスト結果を保存できます。 独自のタイプのテストを作成することは可能です。
