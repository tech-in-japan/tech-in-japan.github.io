Ludum Dareメンバーのための35の役立つヒント





より最近では、5月9日にLudum Dare 35カウントが完了しました。 Ludum Dare-国際的なゲームジャム（簡単な方法-48時間または72時間で完全にゲームを作成する必要がある競争）。 私たちはこれに何度も参加しました（私たちの中にはこれが4番目のLDである）が、今では新しい人がいます。 過去の出来事からの熱意の波で、この記事は書かれました。



Ludum Dare 35の最後のプロジェクトを完了した後、私たちは事後分析を書きたいと思いました。 しかし、最終的には、このコンテストに参加したい人にとって、自分の経験に基づいた一連のヒントのようなものであることが判明しました。 初心者にもジャム向けのゲームを初めて作る人にも役立つことを願っています。



これらのヒントの多くは明白に思えるかもしれませんが、それらのそれぞれは、不必要な熊手から誰かのための有用なヘルメットであると証明するかもしれません。 特にこのようなことにまだ参加していない人のために。





初期トレーニング





1.一人にならない



48時間ソロモードに参加する予定がない場合は、志を同じくする人々を見つけてください。 実生活、職場、ソーシャルネットワーク、フォーラム、どこでも。 少なくとも1か月前に事前に実行してください。 さまざまな時期に、私たちは3〜8人で、すべての作曲で相互作用に問題はありませんでした。







2.参加の詳細について話し合う



ジャムの1週間前に集まり、すべての重要な問題を解決します。 あなたの唯一のプログラマがより重要なことを見つけたとき、またはあなたのアーティストの祖母が突然の誕生日を迎え、セルタワーさえない場所に行く必要があることが判明した場合、それは非常に不快です。 さらに、信頼できる連絡先を交換します。 ジャムの最中に突然Skypeを取得し、誰もあなたの電話を知らない場合、貴重な時間を失うことになります。



3.ローカルで作業する



そのような機会があれば、ジャム全体のために一か所に集まる方が良いでしょう。 Skype / Slackおよびオンラインサービスよりも効果的です。 ただし、この場合は、事前に世帯の問題を解決する必要があります。 誰もが必要な機器をすべて持っていますか？ 十分なアウトレットがありますか？ 最後に何を食べ、どこで寝ますか？ 準備段階でこれを解決します。



4.テクノロジーを決定する



知っていること、確信していることだけを使用してください。 渋滞中に新しいエンジンを習得するのは悪い考えです。 Unreal Engineの使用を開始したチーム（不安定な時代に遡ります）を知っていましたが、ジャムが終わる1時間前にビルドの問題が解決しました。 また、必要なすべてのソフトウェアを決定し、事前に構成します。



開始する





5.ウェイクアップ



これは簡単な作業ではありません。 私たちの地域では、土曜日の午前7時にジャムが始まるため、事前に参加者に電話する必要があります。そうしないと、誰かが確実に眠ります。



6.トピックをフォローする準備をします



はい、現在のジャムのテーマに従うことはできません。 しかし、ゲームを評価するためのカテゴリとしてトピックを指定しなくても、低い評価が得られる可能性があります。



7.ブレーンストーミング



ゲームの興味深いアイデアを得る最良の方法は、ブレインストーミングすることです。 頭に浮かぶすべてのアイデアを書き出す限られた期間で、決してそれらを批判するものではありません。







8.結果をフィルタリングしますが、気にしないでください



既にたくさんの生のアイデアが山積みになっているとき-それを準備する時が来ました。 それらを捨てて、フランケンシュタインを集めて、何が起こるか話し合うことができます。 主なことは、将来のプロジェクトのコンセプトを取得することです。 このプロセスに関与することはできません。ゲームのアイデアが最初の日の終わりにのみ形成され、それを実現する必要がある場合がありました。 制限時間を設定して、それを超えるとアイデアを議論できなくなり、その後は現在の結果を得ることができます。 通常、4時間で十分です。



作業プロセス





9.タスクの記録を保管する



完了するために必要なすべてのタスクを念頭に置いて、それに同意することはありません。 タスクを記録し、完了をマークします。 通常のボード（ただし、おそらく十分ではないでしょう）、または最新のタスクマネージャーのいずれかです。 私たちの選択はWunderlistです。 その中で、異なるリストでタスクを作成し、他のチームメンバーと共有できます。 このようなプロジェクトでは、JiraやRedmineは必要ありません。複雑にしないでください。



10.重要なポイントを特定する



ゲームの重要な部分の準備ができる時期を事前に決定します。 これは、たとえば、次の計画（非常に簡略化された）である可能性があります。「1日-動作中のプロトタイプ、2日-それを表示する完成版、3日-最終仕上げ版」。 そして、予想された瞬間に彼らが準備ができていないなら-時間通りになるように何かをカットまたは変更する準備をしてください。 したがって、プロジェクトを完了する時間があるかどうかを確認し、タスクを評価する方法を学びます。



11.タスクを評価する



実行するすべてのタスクには優先順位が必要です。 そうすることで、そもそも何をする価値があるのか​​を簡単に理解でき、オーバーレイが問題なくスローされる場合は除外できます。 私たちはこのtrapに何度も陥り、最初に二次的なことをしました。 そうしないでください。



12.使い慣れたシンプルな



問題をできるだけ簡単に解決し、同様の問題に対して以前に使用した解決策を知っている場合は、それを実行します。 チームの1人だけが理解できる難しい決定を下すのに問題がありました。 そうしないでください。



13.一生懸命働きますが、寝ることを忘れないでください。



2-3日で面白いゲームを作成するには、多くの時間が必要です。 それを賢く使用し、それからあなたは成功するでしょう。 しかし、あなたは休憩なしで48-72時間座ることができず、野菜にならないように、そして考えないように必要なだけ休むことができません。



14.フレンドリーな管理を行う



ゲームを操作するのに努力が必要ないことを確認してください。 単純な例-現在、多くの人がタッチスクリーン付きのラップトップを使用しています。タッチスクリーンを使用すると、カーソルや一般的なモバイルデバイスを正確に制御することが困難になります。 マウスを使用せずに、またはマウスを交換して制御する機会がある場合は、実行してください。 キーボードの矢印に沿ってメニュー項目を切り替えるだけでも便利です。 そして、はい、まだ少し苦しんでいます。



15.結果を表示する



投稿とストリームで進捗状況を必ず確認してください。 これには時間がかかることは明らかですが、ゲームについての意見が前もって得られれば得られるほど、より良い結果になります。 単独で参加する人でさえ、なんとか意味のある投稿を書くことができます-それはクールです。 しかし、たとえば、私たちはTwitchを適切にマスターしていませんが、今後改善する予定です。



16.パブリケーションに動的コンテンツを使用する



素晴らしいアイデアは、投稿でテキストやスクリーンショットの代わりにビデオ、GIFアニメーション、さらには再生可能なプロトタイプを使用することです。これらの形式はより注目を集めます。





GIF（Breaking Fatゲーム）の適切な使用例



17.出場者とのコミュニケーション



他の人の投稿であなたの意見を表明し、あなたの投稿のコメントに返信してください。 あなたがアクティブであればあるほど、あなたのゲームは最初はより多くの人々にとって興味深いものになります。



18.スタイルを見つける



LDのゲームがピクセルグラフィックスを備えたインディーズであると想定しないでください。 あなたのスタイルを見つけて、実験することを恐れないでください、結局、これのためのジャムが作成されます。 スタイルがまったくない場合や、既に作成されているものを繰り返す場合はさらに悪化します。











19.キャラクターを見つける



これはデザイナーにとって非常に創造的な作業であり、既成のソリューションはありません。 しかし、ゲームで認識できる画像を見つけた場合-これは大きなプラスになります-観客とコミュニケーションをとるときの「ブランド」になります。 私たちのOurのあるナスは偶然発見されましたが、集合的な心でそれが面白いかもしれないことに気付きました。







20.競争力のあるアイテムを追加する



可能であれば-プレーヤーに結果を測定する機会を与えます。 ただし、慎重に行ってください。 最後のジャムで、私たちはこのタスクに夢中になり、結果テーブルに多くの時間を費やしました。 これをしないでください、これは副次的な作業です。



21.おそらく分析を追加する



時間があれば（まれな状況）-ちょっとした分析をプロジェクトに追加して、レビューだけでなく、人々があなたのゲームをどのようにプレイしているかを理解してください。 幸いなことに、Unity Analyticsなどのサービスを使用すると非常に簡単になりました。



発売日





22. WebGLバージョンを作成する



エンジンがWebでのビルドをサポートしていない場合、またはこれに煩わされたくない場合は、ジャムオーディエンスの大部分がプロジェクトを通過することに備えてください。 PC用のビルドを投稿したとしても、それらのダウンロードに興味を持つ人はほとんどいません。 Unity3D WebPlayerの場合、さらに悪いソリューションであることに注意してください、それはすでに完全に時代遅れです。 WebGLを使用すると、すべてがはるかに簡単になります-クリックして少し待ちました。 KongregateまたはItch.ioを使用できます。後者は最近、LDのゲームページにビルドを直接挿入する機会がありました。







23.適切なホスティングを使用する



Webは事実上の標準ですが、Windows / Mac OS / Linuxも努力する価値があります。 すべてのホスティングに馴染みのあるこのために使用します。 最良の選択肢は、2位のDropboxであるGoogleドライブです。 プレミアム機能を備えた速度制限ホスティングを使用しないでください。 計算にはzipアーカイブのみを使用し、すぐに使用できるすべてのOSで開きます。



24.早ければ早いほど良い



競争が終わる1時間前にビルドを配置するようにしてください。 現在、LDサイトははるかに安定しており、ほとんど落ちませんが、以前のジャムの1つでは、15〜20回の試行でビルドを埋めることができました。 そのような状況に備えてください。



25.みんなに知らせる



ビルドをアップロードしてゲームページを準備したら、利用可能なすべてのチャンネルで報告します。 あなたがどれだけうまくやったかをみんなに知らせましょう！ そして今、あなたは最終的に十分な睡眠を得ることができます。



完了





26.公開



追加の発行サイトにアクセスできる場合は、そこにもゲームを送信してみてください。 indiedb.comのようなサービスとGooglePlay / AppStoreの両方のサービスになります。 しかし、これからの特別な結果を期待しないでください。おそらくそうではありません。







27.レート



LDにゲームを送信するだけでは、目的の結果を得るには不十分です。 他のプロジェクトを評価するほど、より良い結果が得られます。 約3000のゲームを評価することは不可能であることは明らかですが、これを徹底的に検討し、少なくとも100〜200を評価してください。 最終的に受け取るレビューの数は、とりわけこれに依存します。



28.良いゲームのリストを保持する



現在、LDでは、面白いゲームのリストを投稿することが一般的になっています。 コミュニティの感謝に加えて、これを使用してゲームに興味を引くことができます。



29.レビューを読む



ゲームページにコメントが残されますので、必ず読んでください。 特に否定的または示唆的な問題。 だからあなたはあなたのゲームについてより多くを理解し、新鮮な外観は常にあなたが見逃したものを見つけます。 そして、次回は間違いを考えてください。







30.事後分析を書く



あなたがより良くできることを見つける別の方法は、あなたがジャムを通して何をしたかについての投稿を書くことです。 あなたの行動を分析するとき、避けられたかもしれない間違い、そしてもっと良くできたはずの間違いを必然的に見つけるでしょう。 それについて書いて、他の人が同じtrapに陥らないようにしてください。



31.統計を見る



競争後のプロジェクトを忘れずに、その結​​果を見てください。 たとえば、現在のジャムでは、サイトで121票、多くの有益なコメント、Itch.ioで44ビュー、GooglePlayで5つのダウンロードがありました。







32.分析から結論を引き出す



分析システムを組み込む時間があれば、期待を確認してください。 結果は、予想とは異なる可能性があります。 結果管理システムのデータから判断すると、合計175人のユニークユーザーがゲームで877回プレイし、そのうち77人が1つ以上のドアを通過しました。 ゲームが少し簡単になると期待していました。

Unity Analyticsのデータに基づいて、当初の期待を満たさない結論をさらに引き出すことができます。







33.進捗を評価する



以前にLDに参加したことがある場合、比較的客観的に、以前の経験と比べて自分が良くなったかどうかを評価できます。

たとえば、以前のゲームと最後のゲームを見てみましょう。

34.結論を出す



ゲームの結果が得られたら、低評価があったとしても怒らないでください。 これがなぜ起こったのか、そして将来の間違いを修正するために何をすべきかをよりよく理解できます。 私たちは何を理解しましたか？ 自分の強み（グラフィック、サウンド、ユーモア）に取り組み続けるために必要なもの、そして次回はゲームプロセスと初期のプロトタイピングにより多くの時間を割くために。
