プログラミング言語のタプル。 パート2

前のパートでは、さまざまなプログラミング言語でのタプルの実装について検討しました（さらに、静的型付けと古典的なCのような構文を備えたコンパイル済みの非スクリプト言語を検討しました。 このパートでは、既存のものを超えて、基本的にプログラミング言語の設計を扱うことを提案します。 初期データは同じです。静的型付けと命令型パラダイムを含むCライクな構文を備えたコンパイルされた非スクリプト言語（もちろんこれに限定されません）。



このパートでは、夢と実験を試みますが、タプルで何ができるのでしょうか？ それらを最大限に活用するには？ 彼らの助けを借りてプログラミング言語をより強力で表現力豊かにする方法、真のコードハッカーへの賞賛を喚起する方法、そして同時に普通のプログラマーをあまり混乱させない方法？ さまざまな方向でタプルのセマンティクスを正確かつ正しく外挿すると、言語にはどのような予想外の可能性が現れますか？



そのため、プログラミング言語の設計に関するトピックの不鮮明さやホリバーが好きなら、猫の下でお願いします。



構文

まず、構文を決定しましょう。 コードでタプルを設計するためのさまざまなオプションを検討するのに長い時間がかかる可能性があります-中括弧、丸括弧、一般に括弧なし... したがって、タプルは名前または式のシーケンスであり、コンマで区切られ、中括弧で囲まれています。 C ++の統合初期化と同様。 

{1,2,3} {i, j, k}


おそらく彼らには存在する権利と他の選択肢がありますが、私たちの目的にとっては今のところこれで十分でしょう。



また、範囲と繰り返しからタプルを作成するという簡単なアイデアに注目したいと思います。 範囲-列挙要素（整数および列挙要素を含む）のタプルを指定する方法 

 {1..10}


繰り返し-アセンブラーから取得したメソッドで、同じ値でタプルを埋めることができます 

 {10 dup 'A'}


この記事では、これらの方法は使用せず、美しいアイデアとして、ちょうどそのように言及しました。



一般的なアイデア

最初の部分では、誤解を引き起こす特定の考えに言及しました-タプルは「まったく型ではありません」。 同じウィキペディアを開くと、 明確に言う
PythonやMLなどの一部のプログラミング言語では、タプルは特別なデータ型です。

静的型付けを使用するプログラミング言語では、タプルの要素が異なる型に属することができ、そのような型のセットがタプルの型によって事前定義されるという点で、タプルはリストと異なります。つまり、タプルのサイズも決定されます 一方、コレクション（リスト、配列）には、保存されるアイテムのタイプに制限がありますが、長さの制限はありません。


しかし、「まったくタイプではない」と言ったとき、私はどういう意味でしたか？ コンパイル段階で任意のオブジェクトをローカルにグループ化できるような言語構成が必要な場合があります（かなり頻繁に）。 追加のエンティティを導入せずに、単に構成部分と同等になるグループエイリアス。 実際、これはタプルをわずかに異なる視点から見るための単なる方法です。 たとえば、関数からの複数の戻り。 これは、「タプル」タイプの1つのオブジェクトの戻りとして完全に考えることができます。 しかし一方で、複数の戻り値の割り当て操作は、多くの操作の1つにすぎません。 Rustの例では、タプルに対する別の操作の可能性、つまり同等性の比較を見ました。 しかし、すべての操作をタプルで実行できるようにしたらどうでしょうか？ さまざまな質問がすぐに発生します-これらはどのような種類の操作になりますか？長さの異なるタプル、タプル、単一の値に対して実行できますか？ タプルを関数に転送するセマンティクスを考慮することもできます（たとえば、関数に渡すときにタプルを「拡張する」、タプルのすべての要素で関数を実行するなど）。 もちろん、タプルを作成および分解する便利な手段、タプルの要素へのアクセスが必要になります。 おそらく他の何か？..



関数とタイプ無効からの複数の戻り値

最も単純なものから始めましょう。すでに多くの言語で実装されています-関数から複数のリターンがあります。 関数は複数の値を返すことができ、関数が複数の引数を取ることができるように、いくつかの値のタプルを返すと呼びます。 次の一般化は、それ自体を示唆しています。単一の値を返す関数は、単一の値のタプルを返します。 少なくとも、 ある値からのタプルがこれらの同じ値に自由に変換され、その逆も同じであるという願いとしてとらえることができます。



さらに外挿すると、void型になります。 ご存知のように、これは、関数が結果を返さないことを示すために、ほとんどのプログラミング言語で使用される特殊なタイプです。 このタイプのオブジェクトの作成は禁止されています。 voidが本格的なタイプではなく、長さゼロのタプルの指定であると推測することはまったく困難ではありません。 通常の意味でこのタイプのオブジェクトを作成することは本当に不可能です。 しかし、コンパイラーがより高度な方法でタプルを処理できる場合、空のタプル{}の形式でvoid型の「疑似オブジェクト」を導入し、それで何かを行うことを妨げるものは何もありません。 質問は何ですか？ また、どのような場合に役立ちますか？ 今のところ、これに注意して、次の外挿に進みます。



複数の操作

関数からの複数の戻り値と、関連する複数の割り当てを調べました。 ただし、割り当ては多くの可能な操作の1つにすぎません。 割り当て（すべての開始元）との類推により、タプルで他の操作を構築しようとします。 

 {x,y,z} = foo(); //   ,   foo()  3  {x,y,z} += foo(); //   ,   ? {x,y,z} ++; //    {x,y,z} = {1,2,3} + {a,b,c}; //


これまでのところ、見た目は良さそうですが、おそらく、多数の操作を伴うそのような式がどのように見えるのかという考えは少し警戒心が強いものです。 式の各オブジェクトが同じ要素数のタプルであることが重要です。 この規則からの逸脱は、プログラミング言語を設計する際に避けるべきさまざまなあいまいさを生じさせます。 それにもかかわらず、それらを考慮する必要があります（そして可能であれば、それらを許可するために）。



グループ運営

複数の操作に加えて、タプルと単一の値に対するグループ操作は依然として非常に魅力的です。 実際、これは、式の各オペランドの同じ要素数の規則からの最初の逸脱です。 しかし、私はそれを美しく見せたいので、私たちは試してみます： 

 {i, j, k} = 0; //    ...  ,    ,     ? ++{i, j, k}; //    ... {i, j, k} += 100; //    rec.{x,y,z}.At(i).flag = true; //


このような二項演算の一般的な形式は、 「タプル演算子値」または「値演算子タプル」です （ほとんどの演算は可換です）。 また、 「値演算子タプル」という形式では、1つの変数をタプル全体に割り当てます。特に、マルチプルリターン関数の結果です。 

 x = {1, 2, 3}; //     x   ? x = foo(); //  foo()   ,    x ?


関数から複数の戻り値を割り当てる状況では、明らかに、最初の戻り値を「x」に書き込み、次の戻り値は無視する必要があります。 これは予想外のあいまいさです。 しかし、どういうわけかそれを解決する必要があります-私は本当にそのような構文を持ちたいです。 グループの割り当てだけでなく、多くの興味深い場合にも役立ちます。 たとえば、配列にインデックスを付けたり、名前で構造体フィールドにアクセスしたりすることも操作です。 

 arr[ {1,2,3} ] = { 10, 20, 30 }; //     obj.{x,y,z} = {10,20,30}; //     ... {obj1, obj2, obj3}.x = { 10, 20, 30 }; //  ... {arr1, arr2, arr3} [ 0 ] = {10, 20, 30 }; //


式の明らかな明白さにもかかわらず、正式にarr [{1,2,3}]は「値演算子タプル」という形式です。ここで、「値」はarr、「タプル」は{1,2,3}、および「演算子」です。 -角括弧。 タプルを単一の値に割り当てるのとは異なり、ここでは問題は発生しません。そのような操作の結果はタプル{arr [1]、arr [2]、arr [3]}でなければなりません。 しかし、コンパイラにとっては、その割り当て、そのインデックス付けは単なるバイナリ操作です。 つまり、x = {1,2,3}という形式の式は{x = 1、x = 2、x = 3}に展開する必要があります。つまり、変数xにはタプルのすべての値が順番に割り当てられ、結果もタプルになります（そのようなプログラミング言語は現実にありました-それはどんなインタビューにとっても興味深い埋め戻しの質問でしょう：その要素は何に等しいでしょうか？{1,2,3}または{3,3,3}？）

したがって、タプルと唯一の要素に対する操作の適切な編成の問題は未解決のままです。



さまざまなサイズのターゲットに対するバイナリ操作

より一般的なケース-タプルのサイズ-バイナリ演算のオペランドが一致しない場合を考えてみましょう。 この場合の対処方法 いつものように、最も簡単な解決策は拒否することです:)しかし、それを理解してみましょう... GoとRustは特別なメタ変数「_」（アンダースコア）を持っています。 構文では、これは次のようになります。 

 {x, _ ,z} = foo(); {x, _, z} = {1, 2, 3};


タプルの2番目のコンポーネントを使用した操作は、単に無視されます。 Go and Rustでメタ変数「_」を使用することは、返されたタプルの結果が必要ない場合の複数の割り当てに必須です。 したがって、これらの言語では、タプルのサイズを強制的に一致させる必要があります。 しかし、これらの言語では、割り当て以外に複数の操作はありません。 メタ変数「_」を他の操作に外挿しようとすると、非常に興味深い結果が得られるため、別の章で検討する必要があります。



一般的なケースを考えてみましょう：そのような式が書かれた場合の対処方法（@は一般化された二項演算です） 

 {a, b} @ {x, y, z}


オプションは何ですか？

  • 小さいサイズのタプルのサイズで操作を実行します。 つまり、結果は{a @ x、b @ y}です。 グループ操作と矛盾します（単一の値が1つの要素を持つタプルと同等と見なされる場合）-グループ操作では、単一のオペランドがタプルの各要素と対話する必要があります。
  • 大きなサイズのタプルのサイズで操作を実行します。 さらに、小さなタプルの要素が十分ではなかった大きなタプルの要素は変更されません。 {a @ x、b @ y、z}。 同じ理由でグループ操作にも反します。
  • 大きなサイズのタプルのサイズで操作を実行します。 小さいタプルが終了したら、その先頭まで循環します。 {a @ x、b @ y、a @ z}。 これはグループ操作に対応しますが、難解です。
  • 大きいサイズのタプルのサイズに対して操作を実行し、小さいタプルが終了したら、小さいタプルの最後の要素に対して操作を実行します。 {a @ x、b @ y、b @ z}。 また、グループ操作に対応し、難解です。
  • 「1つのタプルの各要素と別のタプルの各要素」（「デカルト積」）のペアで操作を実行します。 {a @ x、a @ y、a @ z、b @ x、b @ y、b @ z}。 また、グループ操作に対応し、難解です。 ところで、ペアをグループ化する方法の問題が発生します-最初のタプルまたは2番目のタプル、またはその他の方法。


おそらく他のオプションがあります。 しかし今、1つのことが明らかです。この機会をプログラマに与える場合、タプルがどのように相互作用するかを明示的に示す必要があります 。 この場合、これはコードの明快さを維持するためのほとんどの前提条件です。 このため、この表現をカバーするいくつかのキーワードを使用できます。 そのような単語のリストは指定しませんが（以前は、オプションごとにキーワードがあると仮定できます）



要素へのアクセス

そして今、私たちは別の重要な機会に目を向けるべきです-タプルの要素へのアクセス。

従来、インデックス作成には角括弧が使用され、名前によるアクセスにはピリオドが使用されます。 ドットを介したインデックス付きのSwiftオプションはそれほど悪くはありませんが、数値の代わりに名前付きの数値定数を使用する場合はあいまいであり、通常は珍しいです。 インデックス（重要-定数インデックス）によるアクセスには角括弧を使用し、名前（ある場合）によるアクセスにはポイントを使用することを好みます 

 {10,20,30}[1]; // 20 {x:10,y:20,z:30}.z; // 30


すべてがシンプルに見えますか？ 実際にはもうありません。 「[]」のインデックス付けや構造「。」の名前付きメンバーの参照など、タプルに複数のグループ操作を導入したため、これらの操作を要素が複雑なオブジェクトであるタプルで使用する場合「ドット」）-何をすべきかが明確ではありません：タプルの要素にアクセスするか、タプルのすべての要素でグループ操作を実行しますか？ 

 {arr1, arr2, arr3} [1] // "arr2"  "{arr1[1], arr2[1], arr3[1]} " ? {x:obj1, y:obj2, z:obj3}.z // "obj3"  "{obj1.z, obj2.z, obj3.z}" ?


別の興味深い側面は、タプルの取得（構築）です。 慣例により、記事の冒頭で取り上げたように、中括弧を使用してタプルオブジェクトを簡単に構築します。 ただし、場合によっては、要素を削除または追加して別のタプルからタプルを作成する必要があります。 構文的には、これは「複数のインデックス付け」を使用して実行でき、配列または構造体と本質的に同じルールを適用します。

タプルを取得するには、複数のインデックスまたは範囲を使用できます。 

 {a,b,c,d,e}[1,4,0] // {b,e,a} {a,b,c,d,e}[1..4] // {b,c,d,e} {a,b,c,d,e}[1..3,0] // {b,c,d,a}


（インデックス操作自体には角かっこが含まれているため、中かっこ内にもう1つ中かっこを書くことはできません）

要素に複合オブジェクト名がある場合、名前で連絡できます。 

 obj.{a, d, e} //   obj.{b .. f} //  -


興味深い結果に注意する必要があります-タプルは定数インデックスによってインデックス付けされるため、アイデアはフィールド名が定数インデックスに変換されることを示唆しています（逆も可能です）



したがって、少なくとも2つの「特別な」操作、「ドット」と「角括弧」があり、これらはタプル自体を積分オブジェクトとして機能させることができます。 タプルの残りの操作は何となく定義されていませんが、たとえば、タプルの連結（2つ以上のタプルを1つの長いタプルにフラットグルーする）が必要であると想定できます。 したがって、質問が発生します： タプルの要素にアクセス操作を直接選択する必要がありますか？ または、タプルの各要素の操作を強調表示する方が正しいですか？



操作から機能まで

すべての操作は、いくつかの機能と同等です。 たとえば、単項ビット反転演算〜xはneg（x）として表現でき、2進加算x + yはsum（x、y）として表現できます。

したがって、タプルの操作を複数の操作として考えると、疑問が生じます-関数呼び出しがそのような式に関与している場合はどうなりますか？

開始するには、単項演算： 

 ~{1,2,3}; //    neg({1,2,3}); //


「グループ割り当て」と同様に、タプルを次のように展開する必要があります。 

 {neg(1), neg(2), neg(3)}


一見、これは非常に論理的に思えます。 関数自体は単一の値を取り、単一の値を返します。 タプルを構成できるように、単一の値を返す必要があります。

おそらく、2つの引数を持つ関数も同様に展開できます。 例えば 

 {1,2,3} + {4,5,6} sum( {1,2,3}, {4,5,6} )


で 

 { sum(1,4), sum(2,5), sum(3,6) }


タプルの暗黙的な複数関数呼び出しの構文は暗黙的であり、操作と同様に、そのような呼び出しを指定する明示的な方法は、それ自体を示唆しています（そのような呼び出しの可能性自体は悪くはありませんが）。



一方、Go、D、C∀の構文を思い出してください。関数に渡されたタプルは引数リスト内で展開され、対応する引数の数を置き換えます。 そして一般的に、これは非常に論理的で予想されるものですが、やはり「グループ」セマンティクスとは互換性がありません！ どういうわけかこの矛盾を解決することは可能ですか？ 引数タプルの次元が一致しない場合、タプルと単一の値が混在する場合、タプル要素の操作の結果からデカルト積を取得する場合など、複雑なオプションはまだ検討していません。



かなり良いと思われる解決策は、C ++から奇妙なことに十分に来ました。 可変数のパラメーターを持つテンプレートなどの機会があり、省略記号を使用した構文は、パラメーターのパッケージ（ところで、タプル）を別のテンプレートに転送するために使用されます。 省略記号は、このコンテキストでこの引数が「明らかに」されていることを視覚的に示しています（これはコードの認識にとって非常に重要です！）。 これがコードですぐに見えることが重要です。 目に見えない唯一のものは、それが明らかにする引数の数です。 しかし、詳細に指定したい場合-タプルの個々の要素へのアクセスを妨げるものは何もありません 

 foo(t..., 10, 20); //  foo(t[0], t[1], t[2], 10, 20); //  foo(tx, ty, tz, 10, 20); //


最後に、最も難しい状況：複数のパラメーターと異なる長さのタプルを持つ関数がいくつかの（またはすべての）パラメーターに転送されます。 バイナリ演算の状況と同様に、そのような呼び出しを開示する多くの可能性があります：最小要素数による複数の関数呼び出し、短いタプルの異なるタイプの追加による最大、デカルト積など。 これらのすべてのメソッドを理解するのは非常に難しいため、これらのメソッドはいずれも明示的に宣言する必要があります。たとえば、関数を呼び出す前に適切なキーワードを使用します。



META変数「_」

タプルを割り当てるときにソースタプルの要素を無視するために一部の言語で使用されるメタ変数「_」の動作に戻ります。 このメタ変数をより複雑なケースに外挿できるかどうかを見てみましょう（結局、代入は単なるバイナリ操作です）。 

 {x,y,z} = {1,2,3} + {a,_,c};


同様に、「_」に数値2を追加する操作は無視されますが、結果はどうなりますか？ 一般に、2つの可能性があります。結果のタプルに数値2を残すか、そこに「_」を配布します。 最初のケースでは、「_」は「 中立要素 」と見なすことができます。 操作（つまり、操作「@」および引数「x」の場合、x @ _ == _ @ x == xはtrue）。 たとえば、式x = y *〜（_ + z）はx = y *〜zに変換できます。

ただし、ここではすべてが明確ではありません。 たとえば、単項符号変更演算「-x」は、ゼロ「0-x」から数値を減算する二項演算として記述できます。 「x」の代わりに「_」を入力すると、この式の意味は録音方法によって異なります。 

 z = y * (0 - _) // z = y*0,   z = 0 z = y * (- _ ) // z = y


2番目のケースでは、タプルの位置に「_」が現れると、「_」を含む構文ツリーノードからツリーのルート（つまり、式の最後、通常セミコロン）までのこの位置のすべての計算は破棄されます（つまり、本当ですx @ _ == _ @ x == _）。 つまり、タプルのi番目の要素に少なくとも1つの「_」が存在するということは、式全体のすべてのタプルのi番目の要素を含むすべての計算がスローされることを意味します。
