食料品ロードマップの使用をやめてGISTを試すべき理由

戦略的製品開発計画へのアプローチに関する、成長と戦略のコンサルタントであり、元GoogleプロダクトマネージャーであるItamar Giladによる記事の無料翻訳。











長年にわたり、私はかなりの数の製品戦略、ロードマップ、プロジェクトのガントチャートを開発してきました。 しかし、私はそれらをもうしません。 より良い代替案を見つけたので、これから説明します。







私はこれを行うために使用します：









戦略-ロードマップ-プロジェクト計画-実装（アジャイル）。







このような計画へのアプローチには膨大な量の作業が必要であり、すべての利害関係者から計画の確認を得る必要があるのは簡単なことではありませんが、そのようなアクションのROIは非常に小さいです。 さらに、計画の範囲が大きくなればなるほど、どの計画もすぐに現実から逸脱し始めます。 ロードマップとガントチャートは、公開するまでに関連性が失われていることに気付くまでに時間がかかりました。







さらに、これはウォーターフォールモデル（開発中のウォーターフォールとは異なります）であり、柔軟性がほぼ完全に欠如していることを意味します。スキームの上部での変更は、スキームの下部でのプロジェクトの再計画やキャンセルさえも引き起こします。







アジャイル開発アプローチはプロジェクトのウォーターフォールにうまく対処できますが、ウォーターフォールの計画はできません。 同時に、それらは会社の革新と文化に影響を与えます。







ロードマップはいくつかの大規模プロジェクト向けに設計されており、これは優先順位付けを意味し、成功する可能性のある多くのアイデアを殺します。 トップダウンの組織では、勝利のアイデアは経営陣からもたらされます。 アイデアの勝利を達成するための「ボトムアップ」の組織では、困難な作業であるため、イノベーションに関する誇大広告を作成し、セールスマンのスキルを使用することが必要になることがよくあります。 私の意見では、これは最後の世紀です。







それでは、代替手段は何ですか？







要点



これは、Googleでの作業中に使用を開始した計画システムであり、今後数年間はリーンスタートアップとアジャイルの原則に適合させました。 いくつかの企業にGISTを実装しましたが、結果は非常によく似ています-途中で、変更と迅速な更新に対応するシンプルな軽量プラン、管理者の作業負荷の削減、開発チームの自律性と作業速度の向上、社内でのやり取りの簡素化、結果としてはるかに優れた製品および管理上の決定。







私はこのブロックをメインブロックの名前でGISTと呼びました：目標（目標）、アイデア（アイデア）、ステッププロジェクト（ステッププロジェクト）、タスク（タスク）。 これらの各ブロックには独自の計画期間があり、独自の変更頻度が含まれます。 ブロックごとに異なる管理ツールを使用できますが、それらはすべて会社全体の計画を形成し、チームのアクションを決定します。

目標



「どこに行くべきかを伝えても、そこに着く方法を言わないなら、その結果に驚くでしょう」



ジョージ・S・パットン


ほとんどの戦略的計画は、目標ではなく決定を定義することによって罪を犯します（テクノロジーXを使用し、Y社と提携し、Z国で立ち上げます）。 現代の軍隊の将軍は、このアプローチは過去に残るべきだと言うでしょう-あなたは兵士に目標を設定し、自分でそれを達成する機会を提供する必要があります（ ミッションコマンドの原則）。 このアプローチはより多くの権限を意味し、より意味があり、管理者の負担を軽減します-現場の状況に応じて決定が変わる可能性があり、目標は変わりません。







GISTのフレームワーク内の目標は、これらの原則を具体化したものです。目標は、望ましい結果、つまり、どこに来たいか、いつ、どのように来たかを理解するという観点から会社の戦略を説明します。 いつでも、目標は「なぜこのプロジェクトをやっているのか」という質問に答えます。







Googleで目標をより密接に策定するプロセスを知ることができました。四半期ごとにOKRの形で目標を綿密に表明しました 。 一部の人々は、OKRが企業としてのGoogleがこれほど成功している理由の1つであると信じています。









OKRレコード形式の目標の例：







Q：2018年末までに損益分岐点に達します。

KR：少なくとも5つの新しいエンタープライズクライアント。

KR：流出を2.4％以下に減らします。







Q：製品をモバイルプラットフォームで使用する機会を提供します。

KR：モバイルでのMAU> 20,000。

KR：>キーアクションの25％はモバイルで実行されます。







Q：バグのない快適なユーザーエクスペリエンス。

KR：ユーザーが修正した上位10個のバグを修復します。

KR：テクニカルサポートの応答時間を40％短縮します。







アイデア



「良いアイデアを持ちたいなら、たくさんのアイデアが必要です。 それらのほとんどは間違っているので、どれを落とすかを理解する必要があります。



ライナス・ポーリング


アイデアは、目標を達成するための仮想的な方法を説明しています。 ここでのキーワードは架空のものです。目標を達成する方法について多くのアイデアを得ることができますが、多くの場合、3つのうち1つだけが肯定的な結果をもたらします（通常はさらに少ない）。 上司、経験豊富な製品マネージャー、デザイナーのアイデアは、平均よりもはるかに成功していません。







このため、GISTで作業する場合、アイデアを時期尚早に殺すことはありません。優先順位付け中に致命的な戦いに遭遇することはありません。リーダーシップアイデア、誇大宣伝/含浸/政治化されたアイデアは、他のアイデアよりも多くのサポートがありません。







代わりに、次のことを行います。







  1. アイデアバンクですべてのアイデアを収集します。基本的に、これはExcelの個別のデータベースまたはテーブルです。 どんなアイデアでも歓迎です。世銀には何百ものアイデアがあります。
  2. 証拠に基づいた優先順位付け - ショーン・エリスのICE優先順位付けが好きですが、どの方法でも使用できます（ 人件費と収益を評価するには十分ではありません ！）。
  3. 可能な限り多くのアイデアをテストする予定です-それらはステッププロジェクトの一部として実装されます。


ステッププロジェクト



「大きく考えるが、小さく始める」



Googleの革新の8つの柱の 1つ


有望なアイデアを1つ選択し、9〜18か月間プロジェクトに変えて実装に着手するのは非常に魅力的です。 これはよくある非常に費用のかかる間違いです-まだ未確認のアイデアに四分の一、時には数年も費やすことができます。 これです。 ほとんどのアイデアは投資の価値がないためです。







代わりに、大規模なプロジェクトを小さな段階的なプロジェクトに分割し、それぞれを10週間以内の段階的なプロジェクトに分けて、一度に1つずつ実装します。







プロジェクト手順の例：mocap-プロトタイプ-MVP-dogfood-beta-起動（リリース）。







Lean Startup Build-Measure-Learnの原則に従って 、各プロジェクトのステップは、実際にはアイデアをテストする際の独立した実験です。 正しい順序で、各プロジェクトのステップには、アイデアの改善された拡張バージョンが含まれます。これは、新しいステージごとに増加するユーザーが利用できます。









ステッププロジェクトに分解されたプロジェクトの実際の例：

初期計画：プロジェクトの開始→mocap v.1→mocap v。 2→作業中のプロトタイプ→ドラフト版→社内のドッグフード→アルファ→ベータ→X

実際のアクション：プロジェクトの開始→mocap v.1、予備調査→mocap v。 2、最初のユーザー調査→作業プロトタイプ、2番目のユーザー調査→ドラフト、メインチームでのドッグフーディング→会社でのドッグフーディング→ドッグフーディングの結果の調査→アルファ→アルファユーザーとのインタビュー→ベータ→ポーリングベータユーザー→使用統計→Y







結果として、最終製品は最初に提示したものよりもはるかに優れています。 理由はここにあります 。







各プロジェクトのステップは非常に小さいため、長いプロジェクトに内在するすべてのマイナスの副作用を回避します。したがって、投資を減らして学習を高速化しながら、より多くのアイデアを並行してテストする機会があります。 政治を売り込んだり育てたりするために何も必要ありません。







あなたのアイデアが数週間でどのようにテストされ、実現されるかを見るのは、プロセスのすべての参加者にとって信じられないほど楽しいです。 その後、あなたは二度と死ぬ運命にあるプロジェクトで働きたくないでしょう。







タスク



最終的に、各プロジェクトはタスクと呼ばれる「1バイト」アクションに分割されます。 GISTのこの部分は、アジャイルツール、かんばんボード、およびその他の高度な開発プロジェクト管理技術で適切に実装されています。 ここで何も変更する必要はありません。唯一の違いは、上位レイヤーもアジャイルマニフェストの値に対応し、変更の準備ができていることです。







計画サイクル



マルチレベルおよび反復のGISTを使用した計画：







  1. 通常、目標には1年または数年の計画期間があります。ここでは、はるか先を考える能力が推奨されます。 これらは年の初めに決定され、四半期ごとに再評価および調整されます。古い目標を追求したくありません。
  2. アイデアは常に収集され、優先順位付けされます。 新しいアイデアを探すプロセスを止めることはありません。
  3. ステッププロジェクトは、各四半期の初めに定義されます。 チームは、この四半期に実装する目標とアイデアを選択し、それに応じてステッププロジェクトを決定します。 四半期ごとのステッププロジェクトのリストは、タスクの計画を考慮に入れて、1〜2週間ごとに再評価および再優先順位付けされます。


タスクは、チームが推奨する方法（ スクラムがある場合はスプリント計画など ）によって1〜2週間の反復で計画され、毎日調整されます。









この例では、チームは2つの目標に対して4つのアイデアを並行して実行します。 アイデアNo. 2の場合、最初の2つのプロジェクト手順はすでに正常に完了しています。 また、アイデア3は、最初のステッププロジェクトの段階でそれ自体を正当化せず、破棄され、残りの3つのアイデアのためにより多くの作業を行うためのリソースを解放しました。







まだロードマップが必要ですか？



私はそうは思いません。 ロードマップは、一般的に次の目的で使用されます。







  1. 作業計画-ロードマップはこのために必要ではないことをあなたが確信していることを望みます。
  2. 内部コミュニケーション-私の経験では、同僚や役員がGIST言語を喜んで理解し、受け入れていることを示しています-移行はそれほど難しくなく、誰もがタスクの現実性と信ability性を高く評価しています。 もちろん、会社の従業員と取締役会のメンバーが計画システム全体を利用できるようにする必要があります。
  3. 外部とのコミュニケーション-顧客やパートナーとの「正式なロードマップ」からの期待は非常に高くなります。 いつものように、議論の焦点を特定の機能からニーズに移すのが私たちの仕事です。
  4. GISTを使用すると、「第3四半期に製品内でやり取りを行うことを目標としています。 これがどのように機能するか正確には言えません。 いくつかのアイデアがあり、柔軟に作業します。 おそらく、第2四半期にMVPを導入する予定です。 ベータテスターに​​参加してフィードバックを送りたいですか？」
  5. あなたが幸運で、アイデアが仕事に取り入れられた場合、これはどのロードマップスケジュールよりもうまく機能します。


おわりに



GISTは根本的に新しいアプローチではありません-それは、近年別々に存在したアイデアと方法の混合物です。 これは、計画のすべての層をまとめ、実行可能な計画を作成し、変更に備えようとする試みです。







基本原則：







  • アイデア、計画タスク、およびそれらの実装の分離はありません-これらはすべて定期的かつ頻繁に同時に発生します。
  • 決定や曖昧な戦略策定の代わりに、目標を策定する必要があります。
  • 食料品のバックログの代わりに、一連のアイデアが使用されます。
  • マルチクォータ/マルチ年の長いプロジェクトの代わりに、四半期内に短いステップのプロジェクトで移動する必要があります。
  • 永遠にかかるいくつかの大きなアイデアに賭ける必要はありません。多くのアイデアを迅速にテストし、機能するものだけをリリースします。
  • 反復が適用されます-計画の各部分の改訂は頻繁かつ体系的に実行され、すべてのレベルでアジャイルのままです。




全世界が開発中であり、クールな製品を作成するには、クールな製品マネージャーが必要です。 団結した会社「ホイール| 屋根| 市場」も必要であるため、製品 マネージャーおよびプロジェクトマネージャー 向けの空席があります 。
