こんにちは、Habr！



すぐに私はトピックから小さな余談をしたいと思います：私はプロのアンドロイド開発者ではなく、ただ学習しています。 初心者向けに設計されたこの投稿では、特にAction Libraryとサポートライブラリを使用してSlidingMenuを添付することで、想定されるアプリケーションを作成するときに発生した問題の解決策を組み合わせたいと思います。



私にとっての品質の例は、VK、Forsquare、およびInstagramです。 実際、ActionBarとSlidingMenuを使用するためのアイデアはすぐにそれらから取り入れられました。 ActionBarは3.0でのみサポートされているため、ここでは、Webサイトのレイアウト/開発（私のメインビジネス）とInternet Explorerでの適切な作業と同様に、古いバージョンのAndroidとの互換性について最初に考えました。 検索のおかげで、私はすぐに解決策を見つけました- バージョン2.1+用のActionBar 。





始めましょう



1. Eclipseで新しいプロジェクトを作成します。 これを行う方法は、Habrahabrの検索を使用して見つけることができます。 サポートされる最小バージョンはAPI 7です。できれば、テーマなし（テーマ：なし）。

2.サポートライブラリを接続します。これを行う方法は、上記のリンクに記載されています。 ここでも、念のため。 トピックを書きます。

3. これは、私が直面しなければならなかった最初の問題です。3.0より前のAndroidではすべて正常に動作します。 上記の記事で1つのポイントが指定されていないことがわかります 。トピックはres / values / styles.xmlファイルで継承されることが提案されていますが、これは最も正しいオプションではないことがわかりました。 プロジェクトを作成すると、EclipseはすぐにAPIのさまざまなバージョンのいくつかのスタイルファイルを作成します。



  • res / values / styles.xml-標準スタイルファイル。
  • res / values-v11 / styles.xml -API 11+のスタイルシート。
  • およびres / values-v14 / styles.xml -API 14+のスタイルシート。


上記のファイルのうち最初のファイルでのみテーマを継承するため、バージョンAndroid 3.0（API11）までしか機能しません。

これは、すべてのファイルでテーマを継承するか、プロジェクトマニフェストで直接解決します。 これを行うには、AndroidManifest.xmlに移動して次の行を見つけます。 

<application android:allowBackup="true" android:icon="@drawable/ic_launcher" android:label="@string/app_name" android:theme="@style/AppTheme" >


交換する場所： 

 android:theme="@style/AppTheme"


に： 

 android:theme="@style/Theme.AppCompat.Light"




行Android：テーマがない場合は追加します。

この場合、トピックをマニフェストで直接継承する場合、配色を使用できないため、各ファイルで個別に継承する最初のオプションを使用することをお勧めします。



この問題には2日かかりました。その解決策は、まったく異なるトピックに関する情報を検索したときに見つかりました。



実行して確認してください。 すべてが機能します。





SlidingMenuを接続する


このようなメニューを検索する過程で、すぐにDrawerに出会いました。 ただし、いくつかの理由でこれは必要ありませんでした。ここでは説明しません。

選択はSlidingMenuにかかった。 ライブラリは完全に無料で、GitHubで入手できます 。



1.すべてのファイルをダウンロードし、Eclipseへの接続を開始します。このためには、 [ファイル]→[新規]→[その他]→[既存のコードからAndroidプロジェクト]を開きます。開くウィンドウで、以前にダウンロードしたSlidingMenuのライブラリフォルダーへのパスを指定します ファイルがコピーされたら、[OK]をクリックします。 ライブラリが接続されています。

2.以前に作成したプロジェクトに接続します。 接続手順Support Liraryを繰り返しますが、今回は「library」を選択します。

3. そして、2番目の問題があります -コンソールは2つのファイルandroid-support-v4.jarの競合を報告します。 このファイルは小さく、プロジェクト自体に含まれているため、サポートライブラリとSlidingMenuも接続されています。 ソリューションは簡単であることが判明しました。StackOverflow.comで見つかりました。このファイルをSlidingMenuライブラリとプロジェクトから削除します（どちらも「lib」フォルダーにあります）。

新しい問題が発生しました-SlidingMenuライブラリは多くのエラーを報告します。 これは、削除したファイルが不足しているためです。 この回避策はStackOverflowでも見つかりました。プロジェクトからSupport Libライブラリを切断して、SlidingMenuライブラリに接続します。 この場合、3つすべてに必要なファイルは最初にSlidingMenuライブラリに接続され、SlidingMenuはその機能と共にプロジェクトにSupport Libを接続します。



コメントでは、問題はより専門的な言語で説明され、簡単な解決策を提案しました。「SlidingMenuの古いバージョンのサポートライブラリに問題があり、コンソールに異なるハッシュ合計が表示されるというメッセージがありました。メインプロジェクトからlibをコピーしてドロップするだけで十分でした。 SlidingMenuで。」



コンソールのエラーに注意を払わなかったからこそ、この問題を解決するのに3日かかりました（そう、私は自分の人生を台無しにするのが好きです）。

難しいですが、これらのニュアンスを理解すれば、すべてが非常に明確で論理的になります。

結論として



読者から生じる可能性のある論理的な質問に答えます。「なぜActionBar Sherlockでないのか」（確かにそれが何かを知っている場合）？

答えは簡単です。作業の最初にその存在を知りませんでした。 SlidingMenuが起動したときに、推奨されるシャーロックについて知りましたが、サポートライブラリとそれを使用するコードの半分を変更する必要はありませんでした。



結局のところ、問題の半分はこのプラットフォームにまったく関係しておらず、そのためのツールに関係しているため、Androidのプログラミングを開始するのはそれほど簡単ではありません。

しかし、上で言ったように、少し理解するだけで、すべてがより明確になります。
