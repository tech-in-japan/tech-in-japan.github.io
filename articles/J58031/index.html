管理されていない環境での.NET-使用と一般的な問題

マネージコードと.NET Frameworkは、最も安定したプログラムを鼻に届ける必要があるプログラマーの観点から見ると、非常に素晴らしいものです。 .NETを使用すると、特にC ++やDelphiと比較して、ソフトウェア製品の開発、テスト、保守のコストを大幅に削減できます。



ただし、マネージコードには、その利点に直接由来する非常に深刻な一般的なトラウマが1つあります。最初は、マネージコードは、動作を強制されるアンマネージ環境と互換性がありません。 プログラマーの生活を楽にするボクシング、メモリフィールド、直接アドレス指定の欠如、およびその他のトリックは、マネージコードとアンマネージコードの相互作用が問題になるという事実につながります。



ただし、解決できない問題はありません（aとクローバーでも）。 今日は、マネージドコードとアンマネージドコード間の相互作用を整理する可能性の概要を説明しました。 多くのC＃および特にVB.NETプログラマーはこれを恐れていますが、実際には心配する必要はありません。 初心者だけが興味を持つ最も原始的な方法から始めます（したがって、経験豊富な.NETオオカミは明確な良心を持って記事の最初の部分をスキップできます）。これも起こります）。 当然、それぞれの場合に具体的な例が示されます。おそらく、野ウサギは私自身のサイクリングについて教えてくれます。 並行して、VSTOとWindows Shellを使用する際の落とし穴についていくつか説明します。



マネージドからアンマネージコードを操作します。



.NETの観点から見たこの状況の標準的なメカニズムは、Microsoftが提供する3つのメカニズムです。これらは、プラットフォーム呼び出し（P /呼び出し）、COM相互運用、および安全ではありません。



一般的な警告



アンマネージコードを操作するすべての方法には、十分に深刻な一連の共通の欠点があり、注意を払う価値があり、検討する価値があります。

  1. アンマネージコードの呼び出しは、アプリケーションマニフェストで操作の有効性を判断する必要がある危険な操作です。 これにより、特にClickOnceメソッドの場合、デプロイ時に問題が発生することがあります。
  2. アンマネージコードは安全ではなく、CLRチェックに合格せず、.NETビルトインセキュリティシステムのルールに従いません。 メソッドで「ファイルへのアクセスを制限する」セキュリティルールを定義し、ファイルで機能するアンマネージ機能を使用している場合、問題なく機能します。


この情報を念頭に置いて、それを操作する方法の検討に移ります。

最後の2つのポイントについて詳しく説明します。 アウトプロセスCOMを作成できないのはなぜですか？



簡単に言えば、事実は.NETアプリケーションもアセンブリであり、main関数に直接登録されているときに実行する必要なく、単純なアセンブリDLLとして自分自身を接続できることです。 また、C ++などでアウトプロセスCOMを記述する場合、システムの一般的な動作は、アプリケーションがロードされるのを待ってから、エクスポートされたオブジェクトをROTに登録するまで待機し、次にクラスファクトリの要求でそれを有効にすることです..NETの場合、アプリケーション開始しません-アセンブリは、単に中間を介して現在のプロセスをDLLとして結合します。 したがって、アセンブリでシングルトンまたは静的クラスを定義しても、win32のプロセスは互いに分離されているため、プロセスごとに異なります。



私の知る限り、DCOMを使用してこの問題を解決できますが、.NETでそのようなコンポーネントを作成することはかなりの困難を伴い、他の多くの問題が発生するため、私は個人的にこのアプローチを使用しませんでした。



しかし、プロセスのアドレス空間にCLRをロードすることの問題は何ですか？ 結局、.NETはアセンブリのバージョン管理をサポートしており、いずれにしても、ライブラリに対応するmscorlibのバージョンが使用されます。



それほど単純ではありません。 .NET Framework 3.5で記述されたアプリケーションで.NET Framework 1.1を使用して記述されたCOMをロードしようとすると、問題が発生します。 同じライブラリの2つのバージョンが同じアドレスプロセスに同時に存在することはできません。 ほとんどの場合、すべてが落ちます。



特に、この時点で、.NETを使用したシェル拡張や名前空間拡張などの記述を直接禁止します。 結局のところ、ファイルを保存または開くためのダイアログボックスは、アンマネージコードまたはマネージコードのいずれのコードからでも呼び出すことができます。
