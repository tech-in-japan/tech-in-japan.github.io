インターフェース設計のジョブストーリー





ターゲットオーディエンス（キャラクター）のモデリングと「ユーザーストーリー」（ユーザーストーリー）の作成は、顧客と製品開発者が離れていたときに効果的でした。 今、すべてが異なっています。 この投稿では、ある開発チームがジョブストーリーメソッドを使用してユーザープロファイルページを作成した方法について説明します。



従来、サイトのユーザーとその好みに関する情報の取得は、マーケティング、事業開発、または営業部門のタスクを指します。 この情報を受け取った後、コンパイルして製品の作成に関与する開発チームに送信します。



このようなカスケードモデルの欠点は、高品質の製品を作成するために理解しなければならないニュアンスに関連しています。これらは、ユーザーの行動の理由、それらを刺激し、動機を与えるものです。 開発チームがユーザーに「近づく」必要性を認識した後の2番目のステップは、製品とユーザーの関係を評価する最適な方法を見つけることです。



振る舞い、ユーザーの問題、およびその動機の原因を修正した同様のアプローチは、 Jobs To Be Done （製品が実行する「作業」の概念）と呼ばれる方法の本質であり、それを適用する効果的な方法の1つは、Job Storiesメソッドを使用することですサイト開発、ユーザーインターフェース、インタラクションエクスペリエンス。



キャラクターモデリング問題



キャラクターモデリング手法を使用するときに発生する最も基本的で最も一般的な問題は次のとおりです。キャラクターは架空のものであり、ユーザーの行動の実際の原因を考慮しない特性を含んでいます。 主に人口統計データであるこれらの特性は、開発者が製品に関する顧客の好みを理解するのに近づけません。



キャラクターの特性（年齢、性別、人種、または週末の過ごし方）では、クライアントがたとえばスニッカーズバーを食べた理由を説明していません。 しかし、購入するのに30秒かかったという事実と、バーを食べて、彼は30分空腹をかき消すことに成功しました、それどころか、説明を与えます。







タイトル：「ピーターはなぜスニッカーズを購入したのですか？」



左の列：「空腹を満たすために」。



右側のコラム：「彼は35歳で、マーケティングの卒業証書を持っています。ピーナッツ、チョコレート、ヌガー、キャラメルが大好きで、スニッカーズバーが大好きで、毎日1つ食べています。 彼はアクティブなライフスタイルをリードし、2匹の犬を飼っています。 彼は箱の中のジャックで休みますが、タコベルで食べ物を買います。 彼はつま先を脱臼しました。」



ユーザーストーリーメソッドの問題



ユーザーとして、保存する必要のないファイルでバックアップドライブがいっぱいにならないように、フォルダーをパーティション分割できます。 上記の例のように、ユーザーストーリーには3つの大きな問題があります。



  1. それらはキャラクターに基づいています。
  2. アクションの実行、動機付け、アクションの結果の違いを考慮していません。
  3. ユーザーの状況、状況、問題のコンテキストを考慮していません。


多くの場合、一部の機能またはインターフェース要素は無効です。 それらがUser Storyメソッドを使用して開発された場合、実装プロセス、ユーザーの動機、およびアクションの結果が「1つのヒープに積み上げられた」ため、理由を見つけることは困難です。 この状況で、何が間違っていたのかを理解するにはどうすればよいですか？ 実装の会計処理に誤りがありましたか、または動機付けに関する仮定が間違っていましたか？ ユーザーストーリーメソッドの使用の複雑さについては、 こちらをご覧ください 。







ジョブストーリーとは







この用語はポールアダムスによって最初に言及されて、 ここでより詳細に説明されます 。 Job Storiesメソッドは、製品の機能、ユーザーインターフェイス、およびインタラクションエクスペリエンスを操作する別の方法です。 しかし、開発チームはどのように作業にそれを適用できますか？



1つのアプローチを次に示します。



  1. 最も重要なタスクを完了することから始めます。
  2. 二次的なタスク、またはより重要なタスクを達成するのに役立つタスクを特定します。
  3. ユーザーが現在どのように問題を解決しているかを分析します（現在どのような仕事をしているのか）
  4. 行動の客観的な原因、ユーザーアクションの動機、およびそれらに伴う困難を識別する1つ以上のジョブストーリーを作成します。
  5. 結果の「履歴」に基づいてソリューション（通常は機能またはユーザーインターフェイスの変更）を開発します。


これがどのように機能するかを示すために、ある開発チームが、自動車を販売する人々が自動車を購入したい顧客に融資を提供するのに役立つアプリケーションでUXおよびUIユーザープロファイルページを準備した方法を見てみましょう。



プロファイルページの開発







左から上へのピラミッド：「顧客プロファイル、動機、不安、状況」。 右側の小さな括弧のテキスト：「相互作用の経験」。 右側の大きな括弧のテキスト：「コンテキストと状況によって定義されるタスク」。



それがプロジェクトが初期段階にあったものです。 開発者は、ツールバーとスタートページの外観と、それらに含まれる要素について説明しました。 ある時点で、開発者の1人であるJoeが立ち上がり、ボード上で最も単純な図を描きました。 彼はブロックの1つを指して言った：



これは自動車販売者のプロフィールです。


チームは、プロフィールページの外観についての彼の議論に耳を傾けましたが、すぐに納得させることはできませんでした。 彼らは、図の各要素とプロファイルページの外観に関する一連の質問を彼に尋ねました。 しかし、すべての質問が尋ねられた後でも、チームはジョーのアイデアを支持するかどうかを判断できませんでした。 そして、次の質問が提起されました。



アプリケーションにプロファイルページが必要なのはなぜですか？ なぜ彼女はこの場所または他のどこかにいる必要がありますか？ どの情報を表示する必要がありますか？ どのような状況を考慮し、どのタスクを実行しますか？


これらの問題を解決するために、ジョブストーリーメソッドを使用して機能を調整しました。



注：簡潔にするために、この記事では主にプロファイルページの「ストーリー」を1つだけ扱います。 実際、いくつかのジョブストーリーが用意されています。



1.より高いレベルのタスクから始めます



問題の製品の最高レベルのタスクは、自動車販売者が顧客が自動車を購入するために使用するローンを確保するのを支援することです。 伝統的に、買い手と売り手は多くの深刻な文書に記入する必要があります。



2.マイナータスクを定義する



ローン文書が正しく記入されるためには、売り手と買い手は車とローンを発行するための条件、および買い手の機密財務データに関する大量の情報を入力する必要があります。 この情報は機密情報であるため、買い手は特定の売り手と仕事をするとき、彼の個人データが安全に保たれていることを知る必要があります。



3.プロセスを分析する



（ユーザーが現時点で問題を解決する方法-現在どのような作業を行っているか）



通常、このタイプのデータを入力する場合、買い手は売り手が誰で、ディーラー代理店が何であるかを（ほとんどの場合視覚的に）分析し、その能力と信頼できるかどうかについて結論を出します。 また、通常、機密情報を指定する場合、買い手は売り手と個人的に紙の書類を記入し、権限のない人に書類を信頼しません。



これにより、彼はデータが正しく入力され、データを表示してはならない人はデータを受信しないと確信できます。



4. 1つ以上の「ストーリー」を作成する



（客観的な原因、ユーザーアクションの動機、およびそれらに伴う困難を明らかにする）



売り手と彼のクライアントがこのアプリケーションを介して互いに対話するとき...



  1. ...顧客は、組織、購入プロセス、販売者を信頼できると感じる必要があります。
  2. 売り手は、車を売るプロセスが買い手にとって快適であることを確認する必要があります...
  3. ...顧客が購入プロセス中に財務データを安全に転送できると感じられるようにします。


上記は、特定の状況のコンテキストで「履歴」を形成します。 詳細は、購入の状況、たとえば、ドキュメントがどこでいつ記入されるか（自宅またはディーラー）、およびプロファイルを作成および表示する際に両方の当事者が経験する困難に関する情報を追加することにより、詳細にすることができます。



5.ソリューションを開発する



（通常、「履歴」に反映された問題を考慮に入れるのは、機能またはインターフェースの変更です）



上記のタスクを完了するために、チームはプロファイルページの機能と外観を決定します。 ページに情報がないため、指定したタスクを完了することはできませんが、それを超えると悪影響が生じる可能性があります。 これが最終的に決定したものです。



  1. 製品を使用する過程で、売り手のプロファイルは常に利用可能である必要があります（顧客が売り手の近くにいないという顧客の懸念を減らすため）。
  2. そのようなプロファイルには、売り手、位置、販売された車の数、および会社での勤務期間の写真を含める必要があります（売り手が有能かどうか、信頼できるかどうかについての顧客の不安を軽減するため）。
  3. このプロファイルは、売り手と簡単に連絡できるようにする必要があります。 たとえば、電話番号、メールアドレス、「質問[販売者名]を尋ねる」ボタンを含めることができます（これにより、お客様がサポートなしで誤ってフォームに入力できるという不安が軽減されます）。


ソリューションの例を次に示します。







以下は、ユーザーインターフェイス、インターフェイスの各要素が実行するタスク、およびそれが解決する問題の解析です。







図：



写真のキャプション（「セールスマネージャー」）、プロフィール写真、売り手の名前：これらはすべて、ジョーイ（フィルグッド）が誰であるかを顧客に思い出させ、売り手の隣にいないという不安を軽減します。



売り手の地位：販売のゼネラルマネージャー、販売された車の数（500）および解説（「素晴らしい販売で8年」）。 これは、ジョーイが有能であり、他の人々が彼を信頼していることを確認します。



電話番号、住所、「質問する」ボタンは簡単にジョーイに連絡できます。これにより、クライアントが自分でフォームに記入できないという不安を軽減できます。



ユーザーインタラクションを設計する過程で、すべての要素が1つのタスクを遂行する必要があります。つまり、個人情報を送信するときにクライアントが安心できるようにすることです。



実際の使用条件を考慮したアプリケーションの設計



成功する製品を作成するには、実際の人々が現在どのように問題を解決しているかを分析し、彼らがいる状況の状況を研究し、彼らの行動、疑念、動機の客観的な理由を理解する必要があります。



ユーザーの特性を抽象化し、実装、動機付け、およびアクションの結果の概念を組み合わせます。これらすべてが、開発チームを作業の本質からそらします。 チームがさらに進んで、「ユーザーが「製品」を「雇う」タスク」を調査すると、より良い決定を下す機会が得られます。
