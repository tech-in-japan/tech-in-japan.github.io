StringBuilderの過去と現在

エントリー



私の最後の記事は、.NETのStringデータ型Stringの機能についてでした。 この記事では伝統を続けていますが、今回はStringBuilderクラスを取り上げます。



ご存知のように、.NETの文字列は（安全ではない）不変であるため、それらに対して大規模な連結操作を実行することはお勧めできません。 これは、次のコードに非常に深刻なメモリ負荷の問題があることを意味します。 



string s = string.Empty; for (int i = 0; i < 100; i++) { s += "T"; }


このコードの何がそんなに悪いのですか？ そして、各反復で、前のステップよりも1単位長い行が作成され、その後に古い行から文字がコピーされるという事実。 したがって、関係する文字の総数は次と等しくなります。





この式は、算術数列の合計に過ぎません：



つまり、このような連結シナリオでは、O（n 2 ）に比例したメモリが必要です。nは文字列の長さです。



このようなコードの問題を修正するには、StringBuilderクラスを使用します。このクラスを使用した操作は、Stringのようなメモリの浪費につながらないことを認識しています。 実際、StringBuilderは変更可能な文字列です。 



 var strB = new StringBuilder(); for (int i = 0; i < 100; i++) { strB.Append("T"); } string str = strB.ToString();


そのようなコードは、すべての欠陥がないわけではありませんが、風の中でいくらかのメモリを浪費しますが、以前のコードに比べてより抑制されています。



StringBuilderクラスの実装は、以前のバージョンと比較して.NET 4.0で劇的に変更されたため、何が起こったのかを書くことは興味深いと思います。



.NET 2.0のStringBuilder



.NET 2.0のStringBuilderクラスには、次のフィールドがあります。 



 public sealed class StringBuilder : ISerializable { internal const int DefaultCapacity = 16; private const string CapacityField = "Capacity"; private const string MaxCapacityField = "m_MaxCapacity"; private const string StringValueField = "m_StringValue"; private const string ThreadIDField = "m_currentThread"; internal IntPtr m_currentThread; internal int m_MaxCapacity; internal volatile string m_StringValue; <------------------------------------- }


m_currentThread-オブジェクトのインスタンスが作成されたスレッドの識別子。

m_MaxCapacity-このインスタンスの最大容量。

m_StringValueは、文字を含む文字列です。



実際、StringBuilderクラスは内部的にStringデータ型Stringで動作します。 mscorlib.dll側の文字列は可変であるため、StringBuilderはm_StringValueにある文字列を変更するための費用はかかりません。



初期の長さは16文字です。新しい文字を追加するのに十分なスペースがない場合、StringBuilderは内部の文字列を2倍の長さの文字列に置き換え、前の+新しい文字列から新しく作成されたすべての文字にコピーします。 文字列の長さを2倍にすると、通常の文字列に固有の2次とは異なり、メモリ内で線形の複雑さ（O（n））が生じます。



パフォーマンスを向上させるための重要な追加は、StringBuilderクラスのインスタンスを作成するときに必要な容量を設定することです。 したがって、サイズをさらに2倍にし、メモリ不足でデータをコピーする必要はありません。 ただし、これは、結果の文字列のサイズが事前にわかっている場合にのみ可能です。



最も一般的に使用される方法の実装を検討してください。 コードを読みやすくするために、入力パラメーターをチェックするための条件を削除しました。

一般に、.NET 2.0のStringBuilderクラスは非常に簡単に実装されます。 可変文字列を使用し、十分なスペースがない場合は、新しい文字列を作成します。その長さは前の文字列の2倍です。 このような長さの2倍のシナリオでは、メモリが線形に複雑になりますが、これは2次よりも1桁優れています。 ただし、行の長さが長い場合、これは効果的ではありません。 ちなみに、文字列はサイズが大きいため、大きなオブジェクト（LOH）のヒープに配置されることがよくありますが、これも優れています。



.NET 4.0のStringBuilder



前述したように、.NET 4.0では、StringBuilderクラスの実装が変更されました。 現在、文字を格納するためにStringの代わりにChar []が使用されており、クラス自体はRopeStringのようなStringBuilderのリンクリストです。



この変更の理由は非常に明白です。このような実装では、メモリが不足しているときにメモリを再割り当てする必要はありません。これは以前の実装に固有のものです。 これは、最後の文字列を最初に形成する必要があるため、ToString（）メソッドの動作が少し遅くなり、コピーが不要なためAppend（）メソッドの動作が速くなることも意味します。 ただし、これはStringBuilderの一般的なユースケースに適合します。Append（）を何度も呼び出し、次にToString（）を1回呼び出します。



.NET 4.0のStringBuilderクラスには、次のフィールドがあります。 



 public sealed class StringBuilder : ISerializable { internal const int DefaultCapacity = 16; internal const int MaxChunkSize = 8000; internal char[] m_ChunkChars; <------------------------------------- internal StringBuilder m_ChunkPrevious; <------------------------------------- internal int m_ChunkLength; internal int m_ChunkOffset; internal int m_MaxCapacity; private const string CapacityField = "Capacity"; private const string MaxCapacityField = "m_MaxCapacity"; private const string StringValueField = "m_StringValue"; private const string ThreadIDField = "m_currentThread"; }


m_ChunkChars-リンクリストの現在の要素（文字列）の文字を含む配列。

m_ChunkPrevious-リスト内の前の要素（StringBuilder）へのリンク。

m_ChunkLength-現在のリスト項目の実際の長さ（使用される文字数）;

m_ChunkOffset-ストリングが使用する文字の総数（論理長）。

m_MaxCapacity -StringBuilderの現在のインスタンスの最大容量。



.NET Framework 4および.NET Framework 4.5では、StringBuilderコンストラクター（Int32、Int32）を呼び出してStringBuilderオブジェクトをインスタンス化すると、StringBuilderインスタンスの長さと容量がMaxCapacityプロパティの値を超えて増加する可能性があります。 これは、特にAppendおよびAppendFormatメソッドを呼び出すときに発生する可能性があります。



MaxChunkSizeリストアイテムの最大長は8000です。ご存知のように、これには理由があります。 クラス開発者からのコメントは次のとおりです。



確実にするために、チャンク配列を大きなオブジェクトヒープ（<85Kバイト〜40K文字）に入れないようにします。 最大チャンクサイズを大きくすると、呼び出される割り当てコードが少なくなりますが、未使用文字の無駄が増え、挿入/置換が遅くなります。



大きなオブジェクトの場合、文字配列がヒープに落ちないようにします。 リストアイテム（スライス）の最大サイズを大きくすると、必要なメモリ割り当てが少なくなりますが、使用されない文字が多くなり、挿入/置換操作が遅くなります。

性能比較



おそらく最も興味深い部分は、クラスの2つのバージョン間のパフォーマンスの比較です。



テスト1.指定された長さの文字列を保存するために必要なメモリ量。







ご覧のとおり、文字列の長さが短いと、新しい実装は古いものを失います。 リスト（StringBuilder）の各要素について、長さ、容量、オーバーヘッド文字の配列の行の先頭からのオフセット+に関する情報が必要であるため、これは理解できます。 しかし、行の長さが16384を超えるとすぐに、古い実装は失われ始めます（行のサイズが2倍になるため、多くの未使用文字が含まれます）。



テスト2. Append（）メソッド







おそらく、これは新しい実装が勝つまさにその方法です。 十分なメモリがないときに文字列の長さを2倍にして文字列をコピーする必要がなくなったため、このメソッドはほぼ2倍（より正確には1.8倍）速く実行されます。

新しい実装は、ほとんどすべての場所から削除されたときに勝ちます。これは、残りの行の文字を左にシフトする必要がないためです（より正確には、必要ですが、以前ほど頻繁ではありません）。



テスト5.メソッドToString（）



前述のように、このメソッドは以前の実装を失います。 以前の実装では、（最初​​の呼び出しで）動作する行へのリンクのみが返され、新しい実装では、リンクされたリストの各要素をバイパスして、結果の行を断片的に収集する必要がありました。







リストは16文字の長さの多くの要素（StringBuilders）で構成されるため、Insert（）メソッドを使用して文字列が形成された場合、新しい実装は非常に遅くなります。



テスト6.特定のインデックスのアドレス指定



StringBuilderがリンクリストになったことを考えると、特定のインデックスで文字列にアクセスする操作は高価になります。 特に、Insertメソッドを使用して形成された場合。







テスト7.典型的なシナリオ：Append（）への多くの呼び出し、そしてToString（）への呼び出し



原則として、特定のシナリオに従ってこのクラスを操作します。Append（）メソッドへの複数の呼び出しと、それに続くToString（）への1つの呼び出しです。 このクラスの実装は、このシナリオの計算で正確に変更されました。







おわりに



見てきたように、.NET 2.0のStringBuilderクラスはToString（）メソッドの速度に対して最適化されましたが、.NET 4.0ではAppend（）メソッドの速度に対して最適化されました。 Append（）メソッドの新しい実装はほぼ2倍高速ですが、Insert（）およびToString（）メソッドは低速です。 ただし、特定のシナリオでこのクラスを操作するため、Append（）メソッドを複数回呼び出し、続いてToString（）メソッドを1回呼び出すと、パフォーマンスが向上します。
