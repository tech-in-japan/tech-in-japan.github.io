火災でさえ障害ではなく、災害後のZimbra Speed Recoveryでもありません

前回の記事で、読者は、サーバーハードウェアの故障や物理的な破壊の後にZimbraがどのように復元されるかについて、詳しく教えてほしいと頼んでいました。 したがって、このプロセスがどのように発生するかについて説明し、いくつかのトリックを共有します。そのおかげで、プロセス全体で数分かかります。







そのため、Zimbra Collaboration Suiteを実行しているサーバーで非常に悪いことが起こりました。 火災か洪水かはそれほど重要ではありません。 重要なことは、企業での仕事が増え、このダウンタイムのために数分以内に企業がお金を失い始めるということです。 これは、Zimbra Collaboration Suiteの機能を早急に復元するときが来たことを意味します。



緊急時の復旧は3つの段階で構成されています：最初にZimbraを展開する適切な物理または仮想サーバーを見つける必要があります。次に、オペレーティングシステムをインストールし、Zimbra Collaboration Suite自体をインストールし、インストールされたZCSをバックアップする必要があります。



経験豊富なITマネージャーは、最初の2つのステップは時間がかかるように見えることにすぐに気付くでしょうが、実際、Linux、Zimbra、およびZextras Suiteを搭載したサーバーを試運転する準備ができている場合は、簡単に最適化できます この場合、最初の2つのステップには数分しかかからず、サーバーの電源を入れてDNSレコードを変更するのに必要です。 いずれの場合でも、Zimbra Collaboration SuiteをUbuntuにインストールする詳細な手順については、以前の記事のいずれかをご覧ください。



第3段階を最適化するには、Zextras Suiteの一部であるZextras Backup zimletが役立ちます。 これにより、バックアップからデータをすばやく復元できます。 そのため、たとえば、古いサーバーでリアルタイムスキャナーによって作成されたバックアップが別のディスクまたはクラウドに保存されている場合、バックアップサーバーにマウントして、そこからすべてのデータを復元できます。 リアルタイムスキャナーはZextras Backupの主要な機能の1つであり、Zimbraとの緊密な統合により、サーバーログの変更を追跡し、すべての変更されたファイルを即座にバックアップし、増分バックアップを作成し、最も一般的なrsyncを使用してバックアップデータを同期できることを思い出してください安全な場所に保管してください。 このコピーメカニズムのおかげで、Zimbraを使用したサーバーでの緊急事態の結果として、データがまったく失われることはほとんどないでしょう。







バックアップを作成するとき、Zextras Backupはサーバー設定だけでなくグローバルZimbra設定も予約することに注意してください。 ただし、バックアップから自動的に復元する場合、これらのオプションは適用されません。 つまり、突然古い北の完全なクローンを作成する必要がある場合、または古いサーバーと同じ設定を新しいサーバーに適用する必要がある場合は、Zextrasコマンドインターフェイスとmagic getServerConfigコマンドを使用する必要があります。 たとえば、 zxsuite backup getserverconfig standard date lastコマンドは、最後に保存されたグローバル設定とサーバー設定を表示し、ITマネージャーはそれらを完全に復元するか、必要な設定のみを適用する必要があります。



企業のIT予算により、Zimbraのバックアップサーバーを準備できない場合、いつでも仮想化のメリットと、いつでも支援を提供できるさまざまなホスティングサービスのサービスを利用できます。 すでにLinuxがインストールされている仮想マシンにロードするためのコンテナーと、障害が発生した「バトルサーバー」と同じバージョンのZimbra Collaboration Suiteが必要です。 もちろん、これはバックアップサーバーをオンにするよりも時間がかかりますが、はるかに安価です。 この場合、それはすべて、Zimbraでのサーバーのダウンタイムが組織で許容される時間に依存します。 回復の第3段階では、前述のアルゴリズムを完全に繰り返します。



良いアイデアは、一種の運動を実施することです。 バックアップマシンにZimbraを展開するために「バトル」サーバーを停止する必要はなく、さらに緊急事態を待たないために、営業日のいずれかでそれを試すことができます。 そのような演習では、結果を排除するのにかかる時間を評価することができ、何らかの理由で結果があなたやあなたのリーダーシップに合わない場合、そのような事件の場合により最適なアルゴリズムを開発することを考えることができます。



したがって、多くの条件と予防措置を条件として、Zimbrからサーバーを物理的に破壊しても、企業の重要なデータ、通信、連絡先リストが失われることはなく、長時間のダウンタイムも回避されます。
