ZeroTurnaroundでの統合テスト



ZeroTurnaround製品は、Javaエコシステムの多くと統合されます。 特に、JRebelとXRebelはJavaエージェントテクノロジーに基づいており、Javaアプリケーション、フレームワーク、アプリケーションサーバーなどと統合されています。







Javaエージェントを使用すると、必要な追加機能を追加するようJavaコードに指示できます。 パッチの適用後、アプリケーションの動作をテストするには、事前に構成されたJavaエージェントを介して実行する必要があります。 アプリケーションが起動して実行されると、HTTP要求をアプリケーションに送信して、目的の動作を再現できます。







このようなテストを大規模に実行するには、アプリケーションサーバーやアプリケーションが依存する他の外部依存関係など、ランタイムを開始および停止できる自動システムが必要です。 継続的統合環境と開発者のコ​​ンピューターの両方で、ほぼ同じことを実行できるはずです。







その結果、多数のテストが行​​われ、それらはまったく迅速に機能しません。したがって、それらを並行して実行することは間違いありません。 これは、リソースの競合がないようにテストを分離する必要があることを自動的に意味します。 たとえば、同じホスト上でTomcatの複数のインスタンスを実行する場合、ポート使用の競合を回避したいと思います。







このような統合テストでは、小さくて美しいTestContainersライブラリが役に立ちます。 上記の要件に近づいただけでなく、実装後、生産性が大幅に向上しました。







テストコンテナ



TestContainersの公式ドキュメントには次のように書かれています：







「TestContainersは、JUnitテストをサポートするJavaライブラリであり、コアデータベースの軽量の一時インスタンス、SeleniumのWebブラウザー、またはDockerコンテナーで実行できる他のものを提供します。」

TestContainersは、環境のカスタマイズを自動化するためのAPIを提供します。 テストの期間中に必要なDockerコンテナを正確に起動し、テストが完了するとすぐにそれらを消します。 さらに、GitHubのリポジトリにある公式例に基づいたいくつかのデモを見ていきます。







GenericContainer



TestContainersを使用する場合、GenericContainerクラスが非常に頻繁に使用されます。 







 public class RedisBackedCacheTest { @Rule public GenericContainer redis = new GenericContainer("redis:3.0.6") .withExposedPorts(6379);


そのコンストラクターは、Dockerイメージを示す文字列をパラメーターとして受け取ります。これは今後使用します。 起動時に、TestContainersは適切なイメージを自動的にダウンロードします（以前にダウンロードしていない場合）。







重要な注意： withExposedPorts(6379)



メソッドでは、6379はコンテナがハングするポートです。 次に、コンテナインスタンスでgetMappedPort(6379)



メソッドを呼び出すことにより、対応する関連ポートを見つけることができます。 これをgetContainerIpAddress()



と組み合わせると、コンテナで実行されているサービスの完全なURLを取得できます。 







 String redisUrl = redis.getContainerIpAddres() + “:” + redis.getMappedPort(6379);


この例のフィールドには@Rule



注釈が付いていることに気付くかもしれません。 JUnitの@Ruleアノテーションは 、このクラスの各テストメソッドでGenericContainer



新しいインスタンスを受け取ることを決定します。 コンテナインスタンスを再利用する場合は、 @ClassRule



アノテーションが存在します。







タスクコンテナ



GenericContainer



継承者は、 GenericContainer



固有のコンテナです。 データアクセスのレベルをテストするために、MySQL、PostgreSQL、およびOracleのコンテナ化されたイメージがすぐに使用できます。 







 PostgreSQLContainer postgreSQLContainer = new PostgreSQLContainer("postgres:9.6.2") .withUsername(POSTGRES_USERNAME) .withPassword(POSTGRES_PASSWORD);


この1行だけで、テスト中ずっと残っているコンテナのコピーを取得できます。 テストを実行するマシンでは、データベースを手動でインストールする必要はありません。 同じデータベースの複数のバージョンでテストしたい場合、これは特に大きな利益をもたらします。







自分のコンテナ



GenericContainer



から継承すると、新しいタイプのコンテナを作成できます。 これは、対応するサービスとロジックをカプセル化する場合に非常に便利です。 たとえば、 MockServerを使用して、アプリケーションがHTTPを介して相互に通信する分散システムの依存関係をロックできます。 







 public class MockServerContainer extends BaseContainer<MockServerContainer> { MockServerClient client; public MockServerContainer() { super("jamesdbloom/mockserver:latest"); withCommand("/opt/mockserver/run_mockserver.sh -logLevel INFO -serverPort 80"); addExposedPorts(80); } @Override protected void containerIsStarted(InspectContainerResponse containerInfo) { client = new MockServerClient(getContainerIpAddress(), getMappedPort(80)); } }


この例では、コンテナーが初期化された直後にcontainerIsStarted(...)



コールバックが使用され、 MockServerClient



インスタンスが初期化されます。 したがって、独自のコンテナタイプ内のコンテナに固有の実装の詳細をすべて隠しました。 これにより、テスト用のよりクリーンなクライアントコードとよりきれいなAPIが得られました。







さらに、手動で定義されたコンテナが、Javaエージェントをテストするための環境の構築に役立つことがわかります。

デモアプリケーション



テストアプリケーションから始めましょう。 HTTP GETリクエストに応答するWebアプリケーションが必要です。 脂肪の多いフレームワークは必要ありません。SparkJavaを使用してみませんか？ 楽しさを追加するために、すぐにGroovyでコーディングを始めましょう ここで、このアプリケーションをテストします。 







 //app.groovy @Grab("com.sparkjava:spark-core:2.1") import static spark.Spark.* get("/hello/") { req, res -> "Hello!" }


これは、 Grapeを使用してSparkJavaの依存関係をロードし、「Hello！」メッセージで応答する単一のHTTPエンドポイントを定義する単純なGroovyスクリプトです。







Javaエージェント



確認しようとしたエージェントは、Jettyサーバーにパッチを適用し、HTTP応答に追加のヘッダーを追加します。 







 public class Agent { public static void premain(String args, Instrumentation instrumentation) { instrumentation.addTransformer( (loader, className, clazz, domain, buffer) -> { if ("spark/webserver/JettyHandler".equals(className)) { try { ClassPool cp = new ClassPool(); cp.appendClassPath(new LoaderClassPath(loader)); CtClass ct = cp.makeClass(new ByteArrayInputStream(buffer)); CtMethod ctMethod = ct.getDeclaredMethod("doHandle"); ctMethod.insertBefore("{ $4.setHeader(\"X-My-Super-Header\", \"42\"); }"); return ct.toBytecode(); } catch (Throwable e) { e.printStackTrace(); } } return buffer; }); } }

実際に、テスト！



テストは非常に簡単です。アプリケーションにリクエストを行い、特別なヘッダーの応答を確認する必要があります。特別なヘッダーは、理論的にはそこに追加します。 タイトルが見つかり、タイトルの値が期待値と一致する場合、テストは正常に合格しました。 コードを見てください： 







 @Test public void testIt() throws Exception { // Using Feign client to execute the request Response response = app.getClient().getHello(); assertThat(response.headers().get("X-My-Super-Header")) .isNotNull() .hasSize(1) .containsExactly("42"); }


IDEから直接実行することも、コマンドラインから実行することも、継続的な統合環境で実行することもできます。 TestContainersは、エージェントが隔離された環境、Dockerコンテナーにあるように、アプリケーションを起動するのに役立ちます。







アプリケーションを実行するには、GroovyをサポートしたDockerイメージが必要です。 快適にするために、Dockerイメージzeroturnaround / groovyを取得しました。これはDocker Hubにあります。 GenericContainer



から継承して使用する方法は次のとおりです。 







 public class GroovyTestApp<SELF extends GroovyTestApp<SELF>> extends GenericContainer<SELF> { public GroovyTestApp(String script) { super("zeroturnaround/groovy:2.4.5"); withClasspathResourceMapping("agent.jar", "/agent.jar", BindMode.READ_ONLY); withClasspathResourceMapping(script, "/app/app.groovy", BindMode.READ_ONLY); withEnv("JAVA_OPTS", "-javaagent:/agent.jar"); withCommand("/opt/groovy/bin/groovy /app/app.groovy"); withLogConsumer(new Slf4jLogConsumer(LoggerFactory.getLogger(script))); } public String getURL() { return "http://" + getContainerIpAddress() + ":" + getMappedPort(getExposedPorts().get(0)); } }


APIがコンテナのIPアドレスと関連ポート（実際にはランダム化されている）を取得するメソッドをどのように提供しているかをご覧ください。 つまり、テストが実行されるたびにポートが異なります。 したがって、すべてのテストを同時に実行する場合、ポート間で競合は発生せず、テストは崩れません。







これで、Groovyでスクリプトを簡単に作成できる特別なGroovyTestApp



クラスができました。この場合、デモアプリケーションをテストするためのものです。 







 GroovyTestApp app = new GroovyTestApp(“app.groovy”) .withExposedPorts(4567); //the default port for SparkJava .setWaitStrategy(new HttpWaitStrategy().forPath("/hello/"));
