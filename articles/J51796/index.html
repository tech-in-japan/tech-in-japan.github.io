WindowsにErlangをインストールする



Erlangのインストールは非常に簡単です。 最初に、 公式サイトからディストリビューションをダウンロードします 。 R12B-5の最新バージョンであるWindowsバイナリ列（ドキュメントを含む）が必要です。 「スタート」-「すべてのプログラム」をインストールすると、Erlang OTP R12Bアイテムが表示され、その中にErlangサブアイテムが表示されます。 これを選択すると、名前のないノードがシェルとともに起動します。このシェルでは、私の例からのように、すでにErlangianコードを実行できます。 「ノード」という言葉は「ノード」と訳されていますが、ロシア語の正式名称はまだ確立されていないため、「ノード」という名前を守ります。



Mac OS XおよびLinux用のErlangをインストールするためのリンクを以下に示します。



インストールを完了するには、Windows環境変数に次の変数を登録する必要があります。

1. Erlangがインストールされたディレクトリへのパス：

ERL_TOP = C：\ Program Files \ erl5.6.5

同じパスですが、 \ binが追加されています。 C：\ Program Files \ erl5.6.5 \ binもPATH変数に書き込む必要があります（まだない場合）



2.次の例のように、HOME変数がまだ登録されていない場合：

HOME = C：\ Documents and Settings \ tolik



3.サードパーティのErlangライブラリへのパスをすぐに登録することをお勧めします。

ERL_LIBS = C：\ Work \ erl_libs

将来的には、Erlang / OTPディストリビューションに含まれていないすべてのサードパーティライブラリをこのディレクトリに配置してください。 これにより、新しいバージョンのErlang / OTPを簡単にインストールし、サードパーティのライブラリを再配置せずに古いバージョンを削除できます。 ところで、Erlang / OTPの新しいバージョンは年に2回リリースされます。



最後に、HOMEディレクトリにファイル「.erlang.cookie」を作成します。 これは、たとえば「AMMVFJKDTENSFHDJDGDERSW」のように、テキストCookieを1行で保存する通常のテキストファイルです。 このCookieは、ノードが相互作用するときに使用されます。 実際には、2つのノードは、同じCookieを持っている場合にのみ相互に対話できます。 いくつかのノードの相互作用にまだ関心がない場合は、この手順をスキップしてください。



同じHOMEディレクトリで、Erlangにコードを記述できる ".erlang"ファイルを作成すると便利です。このファイルは、このコンピューターでノードが起動したときに自動的に実行されます。 通常、コードはこのファイルに書き込まれ、アプリケーションの実行可能ファイルへのパスを検索リストに追加します。



code:add_pathz(" ebin ").







これで、「erl」と入力するだけで、コマンドラインから直接ノードを起動できます。 シェルなしで、またはデーモンとしてノードを実行できます。 シェルなしでノードを起動した場合、2番目のノード（別のコンピューター上で実行可能）を実行することで接続できます：



erl -name test@192.168.0.11 -remsh main@192.168.0.11







ここで、 main @ 192.168.0.11は接続先のノードの名前であり、 test @ 192.168.0.11は新しいノードの名前です。 接続後、最初のノードにいるかのようにすべてのコマンドを実行できます。



起動パラメーターの詳細については、 erlモジュールを参照してください。

Emacsを置き換えるために、新しいerlIDE開発環境が開発されています。 現在のバージョン0.4.3は既に十分に安定しており、以前のバージョンのように落ちません。 デバッガ、コンテキストヘルプ、シンプルなコードナビゲーション、構文の強調表示など、最新のIDEのほとんどすべての通常のツールが含まれています。 インストール手順は十分に詳細に書かれており、通常は質問をしません。 erlIDEはEclipseのプラグインとして機能するため、まずコンピューターにJavaをインストールし（バージョン6以上が望ましい）、次にEclipseをインストールする必要があります。 Java開発者向けのEclipse IDE（85 MB）で十分です。 Windowsでは、Emacsに慣れないように、すぐにerlIDEから始めることをお勧めします。

最初のアプリケーションを作成する



最初に、プロジェクトがあるフォルダーを作成します。 その中に、次のサブフォルダーを作成する必要があります。

  • ebin-コンパイルされたコード（.beam）はここに置かれます
  • src-アプリケーションソース（.erl）
  • include-ヘッダーファイル（.hrl）
  • doc-ドキュメント
  • priv-アプリケーションが動作する必要があるが、他のフォルダーには含まれていないその他すべて


これは、OTPが推奨する標準アプリケーション構造です。 アプリケーションのルートフォルダーで、次の内容のemakefileを作成します。



{"src/*", [debug_info, {i, "include"}, {outdir, "ebin"}]}.







次に、アプリケーションのルートフォルダーにあるノードシェルで、コマンドmake：all（）を実行するだけで、すべてのアプリケーションモジュールを再コンパイルできます。



「ebin」サブフォルダーで、おおよそ次の内容のyour_application.appファイルを作成する必要があります。

{application, your_application.app,

[{description, "Test application"},

{vsn, "1"},

{modules, [you_application, you_application_sup]},

{registered, []},

{applications, [kernel, stdlib, sasl]},

{mod, {your_application,[]}}

]}.







最後に、「src」フォルダーに、最小限のコンテンツを含むメインプロジェクトファイルyou_application.erlを作成する必要があります。

-module(you_application).

-behaviour(application).



-export([start/2, stop/1]).



start(_Type, _Args) ->

your_application_sup:start_link().



stop(_State) ->

ok.







したがって、Erlangで最小限のアプリケーションを作成しましたが、これまでのところ何もしていません。 また、メインアプリケーションスーパーバイザーはまだ実装されていません（これについては後で説明します）。 アプリケーションの作成の詳細については、OTPのドキュメントを参照してください。



ホットスワップコード



Erlang / OTPの非常に便利な機能は、システムを停止せずにコードをホットスワッピングすることです。 これは、1つのモジュールの例でどのように機能しますか？ たとえば、アプリケーションモジュールの1つでコードを変更し、実行中のシステムでその場でコードを置き換えたいとします。 Erlangのこの状況では、アプリケーション全体を停止する必要はありません。 ノードシェルに入力するだけで十分です： l(your_module).



このコマンドを実行すると、新しいコードが機能します。 より詳細には、Erlang仮想マシンは、各モジュールの2つのコピー（現在のモジュールと以前のモジュール）をメモリに保存します。 以前のバージョンを保持する理由 アプリケーションを停止せずに新しいコードが正しく動作しない場合に、以前のバージョンにすばやくロールバックするため。



もちろん、個別のモジュールだけでなく、アプリケーション全体をオンザフライで置き換えることができます。 これを行うには：

  • アプリケーションアップグレードファイルを作成します。このファイルでは、古いバージョンを新しいバージョンに置き換えるためのルールと、バージョンの置き換え中のモジュール間の依存関係を規定しています。
  • リリースアップグレードファイルを作成する
  • 古いバージョンの上に新しいバージョンのアプリケーションをインストールします


記事のサイズを小さくするために、これらの手順を詳細に説明しているドキュメントに送信して詳細を確認します。



しかし、複数のサーバーで構成される分散システム全体があり、システム全体を停止せずにそれらすべてを同時に交換する必要がある場合の対処方法。 残念ながら、Erlang / OTPは既製の便利なソリューションをここで提供していません。 それでも、OTPモジュールにはerl_boot_serverが含まれています。これにより、ノードの開始時に、別のノードから構成ファイルを含むアプリケーションコードをロードできます。 次に、システム全体のオーバーロードを次のように実装できます。最初に、erl_boot_serverモジュールを使用して、起動時に他のノードがロードされたコードであるメインサーバーでコードがホットスワップされます。 そして、メッセージをすべてのノードに送信して、ノードを過負荷にします。 そのようなメッセージを受信すると、ノードでinitコマンドが実行されます：restart / 0は、ノードを完全に停止せずにオーバーロードします。 過負荷の後、すべてのノードはメインサーバーから既に更新されたコードを取得します。 このアプローチについては、Amazon EC2でのErlangのセットアップ （記事の2番目の部分）とAmazon EC2でのErlangクラスターのアップグレードの 2つの記事で説明されています 。

プロセスを死なせる



最初の記事ですでに述べたように、Erlangで構築されたシステムの高い信頼性は、すべてのコードがクラッシュしないように書かれているという事実に基づいていませんが、ワークフローにエラーがあった場合、常にそれを監視している人がいます状況を修正してシステムを復元するプロセス。 実際、ほとんどの場合、ワークフローは通常、エラーが発生した場合の対処方法を正確に把握していないため、単純に死んで観察プロセスに状況を理解させる方がより正確です。 これは次のように行われます。



  • まず、観察プロセスはアーランで有名なコマンドを実行します。

    process_flag(trap_exit, true).





    これは、このプロセスが関連プロセスの落下に関するメッセージをキャッチすることを意味します。 このコマンドを実行した後、ワークフローがクラッシュすると、次の形式のメッセージが表示されます： {'EXIT', From, Reason}



    は観察プロセスに到着します。 process_flag（trap_exit、true）を実行するプロセスが途中で見つかるまで処理します。
  • 次に、監視プロセスは関連するワークフローを作成します。

    spawn_link(?MODULE, worked_function, [Arguments]).



  • 3番目に、監視プロセスはメッセージのリッスンを開始します。

    receive

    {'EXIT', From, Reason} ->

    supervising_function() % (. )

    end







合計すると、監視プロセスの最小コードは次のようになります。

supervising_function() ->

process_flag(trap_exit, true).

spawn_link(?MODULE, worked_function, [Arguments]).

receive

{'EXIT', From, Reason} ->

supervising_function()

end

end.







建築用レンガErlang / OTP



ただし、監視プロセスを最初から記述したり、共有リソースにアクセスするプロセスなどを記述する必要はまったくありません。 Erlang / OTPでの20年未満の言語開発では、Erlangで書かれたアプリケーションが通常構築される、信頼性の高いデバッグされた基本コンポーネントのセットが開発されました。 ここにあります：

  • gen_server-システムでのクライアントとサーバーの相互作用をサポートする基本的なサーバー実装。 ここで、「サーバー」の概念は一般的に受け入れられている概念と一致するものではなく、特定のErlang / OTPサーバーを意味することに注意してください。 通常、システムでは、彼はシステムの共有リソースにアクセスする責任があります。

  • gen_fsm-ステートマシンを実装するベースモジュール
  • gen_event-イベント処理機能を実装する基本モジュール
  • そしてもちろん、 スーパーバイザー -モジュールは、その子プロセスを開始、停止、および監視するように設計されています。 多くの理論を書かないために、スーパーバイザー初期化関数の簡単な例を示します。

    init() ->

    RestartStrategy = one_for_one

    MaxRestarts = 10,

    MaxTimeBetRestarts = 3600,

    SupFlags = {RestartStrategy, MaxRestarts, MaxTimeBetRestarts},

    ChildSpecs = [{Some_Module, {Some_Module, Function, []}, permanent, TimeOut,

    worker, %

    [Some_Module]}],

    {ok, {SupFlags, ChildSpecs}}.





    このコードは1つの子プロセスを開始します。子プロセスの再起動の最大数は10で、頻度は3600ミリ秒以下です。




上記のすべての基本コンポーネント（スーパーバイザーを除く）には、アプリケーションがホットリロードされたときに自動的に実行されるcode_change関数があります。 この関数では、アプリケーションのあるバージョンから別のバージョンへのスムーズな移行を提供するコードを作成できます。



残念ながら、各モジュールの詳細な説明とそれらの間の関係については、いくつかの記事が必要になります。さらに、それらは「 設計の原則」で詳しく説明されています。 また、使用例に目を向ける必要はありません。Erlangで作成された、よく知られているアプリケーションやライブラリを使用して確認してください。 それらのほとんどすべては、これらの基本的なコンポーネントを使用して書かれています。
