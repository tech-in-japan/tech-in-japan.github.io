企業システムでの印刷の自動化、またはプリンターと友達を作る方法

こんにちは、ハラジテル！ 私はあなたと私の苦痛な経験を共有したい、それが役に立つことを願っています。 今日は、企業システムでドキュメント印刷システムをどのように開発したかについてお話します。



それがすべて始まった方法



それはすべて、約2年前に1つの商社でERPプラットフォームを開発することから始まりました。 Linuxが選択されました。C++ / Qtスタック、PostgreSql、およびWebの前面です。 アプリケーションサーバーはC ++ / Qtで実装され、同じ場所で、ビジネスロジックはJSインタープリターレイヤーを介して記述されました。 なぜ別の話なのか、ここでは印刷システムがどのように開発されたかを検討します。



ペンの最初のサンプル



HTML



最初は、すべてのドキュメントが1Cから印刷されていましたが（1Cの束と社内の管理会計に独自の「ERP」を使用します）、すべては順調でしたが、計画はシステムに切り替える予定でした。 その後、システムにシールを固定する必要がありました。



最初のアイデアは、HTMLフォームをレイアウトし、プログラムでデータを入力し、ユーザーにブラウザーにhtmlして、そこから印刷できるようにすることでした。



すぐにいくつかのニュアンスを見つけました：



  1. すべてのドキュメントの下で、0からテンプレートを作成する必要があります
  2. エンドユーザーはhtmlを編集できません
  3. マネージャーはExcelでいくつかの分析を主導しました
  4. 複数ページのファイルを印刷するときの行の折り返しの問題


その結果、ロジスティクス（単一ページフォーム）のアプリケーションを印刷するためのテンプレートが1つだけ作成されましたが、これは引き続き正常に使用されています



Xlsx



2番目のアイデアは、XLSXドキュメントで作業することでした。 GoogleはすぐにQtXlsxWriterライブラリについてプロンプトを出しました。 まだオプションがありましたが 、最終的にはQtXlsxWriterに落ち着きました。



ライブラリでできること：



  • xlsxを開いて作成し、セル値を読み取る
  • セル値の変更、ファイルの保存
  • セル形式（境界線を含む）で作業する
  • 行/列の高さ/幅
  • セルユニオン
  • 行/列のグループ化
  • 画像挿入


これにより、他のシステム（Hi 1C）からxlsx形式のドキュメントテンプレートを取得するコストを即座に削減し、必要なデータと統計情報を入力して、ユーザーが好きなように作業できるxlsxファイルをユーザーに提供できました。



落とし穴がすぐに現れ、QtXlsxWriterはテンプレートからのドキュメントの読み込み、セル形式の喪失（ラップ/アリングなど）を「不十分」に処理しました。 「xlsx形式（誰も知らない場合、xlsxはxmlドキュメントのセットを含むzipアーカイブ）を数時間掘り下げた後、ブールの異なるバージョンでは、xmlファイルの属性が異なる方法で格納されていることがわかりました。 



<ta="0" b="1" />


どちらか 



 <ta="true" b="false" />


parsilの場所のQtXlsxWriterは1/0のみ、true / falseの場所のみ、およびthisとthatの場所もあります。 しかし、何も、固定された手があります。



また、QtXlsxWriterを介してxlsxファイルが形成された後、MS Officeで開くと、それが宣誓され始めて不快な瞬間がありました。

本「test1.xlsx」で、読むことができないコンテンツが見つかりました。 本の内容を復元してみてください？ ソースを信頼する場合...


同時に、視覚的に開いた後、すべてのデータが配置されました。 ただし、価格表を開くと、多くの一般ユーザー（顧客）がこのようなメッセージを恐れることがあります。



MS OfficeとQtXlsxWriterをxmlファイルに何時間も貼り付けて検索した後、 なぜ MSがそれを気に入らなかったのかわかりません 。松葉杖が発明されました。 QtXlsxWriterによって生成されたファイルを取得し、 LibreOfficeを使用して処理する場合、元のすべてのデータを含む有効なxlsxファイルを取得しますが、MS Officeはそれを誓いません： 



 libreoffice --headless --invisible --quickstart --convert-to xlsx test.xlsx --outdir valid_xlsx


そして、生きていくのは良いことでした。xlsxドキュメントの形成に関する必要なレポート用の小さなコードが作成され、ユーザーにアップロードされ、必要に応じて作業し、MS Officeはそれらを怖がらなくなりました。 彼らはグループと遊女でxlsxのOLAPレポートをアンロードすることさえできました。



自動化



会社が成長し、顧客が増え、ドキュメント（アプリケーション、販売、請求書など）が増え、印刷に多くの作業時間がかかり始めました。 さらに、ドキュメントの一部は、システムの1C部分から印刷されました。 この問題を何らかの形で自動化することにしました。 これに先立って（約5〜7年前）、Windows OLEコンテナー（コンテナーはExcelで作成され、ファイルが開かれ、印刷設定が設定されて印刷に送信されました）を介して印刷した経験がありましたが、実際にはLinuxで回転してドラッグしているプラ​​ットフォームではありませんここではWindowsモジュールは必要ありませんでした（ただし、Windows上のプリントサーバーは極端なオプションと見なされていました）。



すべてPDFで



LinuxにはCUPSがあり、lprコマンドを使用すると、印刷用のpdfファイルを簡単に送信できます。 これはただのpdfであり、生成方法がわかりません。 解決策はすぐに見つかりました。 



 libreoffice --convert-to pdf 1.xlsx --headless


しかし、それはそれほど単純ではありませんでした。 ファイルは100％スケールで変換され、ページサイズ（A4 / A3、ポートレート/ランドスケープ、インデント）に適合しなかったか、むしろすべてが標準パラメーター（A4、ポートレート）に従って調整されました。 LibreOffice（LibreOffice Calcを手で開く）でこれらの設定を設定し、xlsxで保存し、libreoffice --convert-to pdfで変換すると、すべてがほぼ完璧に機能することがわかりました。



  1. インデントとページ設定が正しく処理されました。
  2. スケールを調整する必要がある場合、このパラメーターは無視され、100％のスケールで変換されました。
  3. サイズ/ページ数に合う設定があれば、すべてが機能しました


ポイント2について、私はLibreOfficeのサポートを停止しました。彼らからの連絡を楽しみにしています。



パラグラフ3の利点は正しく機能することであり、それを基に構築することにしました。 次に、QtXlsxWriterにページ設定を操作する方法を教える必要があります。 xlsxドキュメント内のxmlファイルを発見すると、このビジネスに責任のある場所が見つかりました



xl / worksheets / sheet1.xml 



 <worksheet> <sheetPr filterMode="false"> <pageSetUpPr fitToPage="false"/> </sheetPr> ... ... <pageMargins left="0.7875" right="0.7875" top="1.05277777777778" bottom="1.05277777777778" header="0.7875" footer="0.7875"/> <pageSetup paperSize="9" scale="50" firstPageNumber="0" fitToWidth="1" fitToHeight="1" pageOrder="downThenOver" orientation="portrait" usePrinterDefaults="false" blackAndWhite="false" draft="false" cellComments="none" useFirstPageNumber="false" horizontalDpi="300" verticalDpi="300" copies="1"/> ... </worksheet>


ここで興味深いのは：

pageMargins-これですべてが明確になったと思う

fitToPage-サイズ/ページ数に合わせる、またはスケールを使用する

fitToWidth-幅のページ数

fitToHeight-高さのページ数

スケール-スケール（％）

paperSize-シートサイズ（9 = A4）

向き-縦/横



QtXlsxWriterでこれらのパラメーターを使用した作業を追加しました。 不完全なコンテンツの断片が別のシートに印刷されないように、適切な場所にインデントを使用してxlsxドキュメントを形成するだけです。 これにより、すべてが非常に単純であるとは限りませんでした。



印刷する



インデントなしで、A4ブックの向きのシートにルートシートを印刷する場合の状況を考慮してください。



ルートシートの例




この場合、ドキュメントの幅が1ページに収まる必要があります。 設定を行います：



fitToPage = false

fitToWidth = 1

fitToHeight = 100

pageMargins-すべて0



これらの条件下では、fitToHeightは、印刷時に予想されるページ数よりも大きいことがわかっている必要があります。



ルートリストは、ルートと内線番号を持つ顧客のリストを示す見出しです。 どの配信が行われるかに関する情報。



そのままにしておくと、シートの最後にある顧客情報を含むブロックの一部が破損し、一部が最初の最後にあり、一部が2番目の先頭にある可能性があり、これは受け入れられません。



その結果、次のアプローチが生まれました（おそらく松葉杖）。



最初に、A4シートサイズを知っています。

幅21cm

高さ29.7 cm



そして、コンテンツがシートの幅に合わせて調整されることを知っています。 コンテンツの相対圧縮率を計算できます：



スケール=シートの幅/コンテンツの幅



ここで、コンテンツの幅を計算するために、すべての列の幅を追加する必要があります。これは難しくありません。 



 double QXlsx::Document::columnWidth(int column);


どの測定単位で結果が得られるかはまったく理解できませんでした。 おそらくここで正しい解決策を見つけることができますが、最終的には、マジック番号5.10238を経験的に見つけることはできませんでした

1 cm = 5.10238 e.sh. （列幅の単位） 



 scale = 4_ * 5.10238 / sum(columnWidth)


次に、シートの幅全体に合わせることができるコンテンツのサイズを計算します 



 height=4_ * 28.3464567 / scale


行の高さをe.i.v.s（行の高さを測定するための単位、インターネット上でこの情報が見つかりました„ r.Height = ht * 28.3464567 // CMをポストスクリプトポイントに変換 "）



行の高さは次の方法で確認できます。 



 double QXlsx::Document::rowHeight(int column);


heightパラメータを使用して、コンテンツの高さが<= heightである間に、コンテンツをxlsxファイルにハンマーで打ち込みます。 新しいブロックBを追加するときに、高さの境界を超える場合、Bの前に必要な高さの空の行を挿入して、ブロックBが新しい行から印刷されるようにします。 空行の高さは、ブロックBの前に挿入されたコンテンツの高さ（sum（rowHeight））を知ることで計算できます。



ここでは、インデント（pageMargins）を使用したページネーションの計算は考慮していません。xmlデータにはこれらの変数の値がインチ（1インチ= 2.54 cm）で格納されているとしか言えません。



したがって、既製の設定と印刷用の行ごとの内訳を含むxlsxファイルが取得されます。 次に、libreoffice --convert-to pdfを使用してpdfに変換すると、ドキュメントを印刷する準備ができました。



印刷するままです： 



 lpr -pFS-4300DN test.pdf


現在、フィニッシャー（ステープル留め）を備えた多機能デバイスでの印刷の自動化を行っています。 すでにテスト装置で少し遊んでいて、Linuxのもとでは、それはすべてステープリングのためだけでした。



左上隅にクリップが1つあるステープル印刷： 



 lpr -P printer_name -o StapleLocation="UpperLeft" order.pdf
