バックアップ/システムリカバリ用のライブUSBの作成

システムのバックアップを作成するたびに、空いているUSBフラッシュドライブを探し、その上にArchLinuxのインストールイメージをダウンロードしてロードしました。 この方法には非常に不快な瞬間がありました- サイズが512 MiB以上の空きUSBフラッシュドライブを探す必要がありましたが、常に64 MBのSDカードが手元にあり、バックアップと復元の目的には十分でした。 しかし、SDカードには1つの問題がありました。まれなディストリビューションキットがEee PC 900から起動する可能性がありました。



準備する



必要なもの：



  • USBフラッシュドライブまたはSD / MMC / CF / ...メモリカード
  • 動作するLinuxシステム。できればコンパイル用の完全なセットまたはパッケージマネージャーが必要です。
  • 動作するために追加のモジュールを必要としないLinuxカーネル
メディアのサイズは重要ではありません-私が見つけることができる最小のメモリカードは64メガバイトのサイズであり、これはバックアップ/リカバリのための完全なシステムを作成するのに十分以上です。

カーネルに関しては、必要なモジュールがコンパイルされ、initrdサポートが無効になっているカーネルを使用しています。 ディスクを操作するために追加のモジュールを必要としないカーネルが既にある場合は、それを使用できます。 そのようなカーネルがない場合は、 kernel.orgからカーネルソースコードをダウンロードし、手動でビルドします。 この記事では、カーネルの構築方法については説明しません。

フラッシュドライブが/ dev / sdcであることに同意しましょう。



ビジーボックス



私はBusyBoxを使用したことをすぐに言いたいと思います-BusyBoxは、わずかに切り捨てられた機能ではありますが、他のユーティリティの非常に大きなセットを置き換えるユーティリティです。 その焦点は、実行方法に応じて、これがどのように動作するかです。 たとえば、lnという名前でbusyboxへのシンボリックリンクを作成すると、lnになります。 リンクの名前がddの場合、ddが取得されます。 1.5メガバイトの重みと依存関係なし-これは奇跡です。

パラメーターなしでbusyboxを起動すると、使用中のバージョンで使用可能なユーティリティのリストが表示され、その機能はbusyboxに置き換えることができます。 サイズが非常に重要な場合は、ソースからBusyBoxを構築し、不要な機能を無効にして、バイナリファイルのサイズを小さくすることができます。 リポジトリから既製のパッケージを使用しました。



デバイスの準備：



fdiskを使用して、デバイスのフラッシュドライブに1つのパーティションを作成し、起動可能にします。

fdisk /dev/sdc





まず、このデバイス上のすべてのパーティションを削除します 。 ほとんどの場合、セクションは1つだけなので、dを1回入力します。 次に、新しいプライマリパーティションを作成します 。 n 、 [Enter] 、 p 、 [Enter] 、 1 、 [Enter] 、 [Enter] 、 [Enter]を押します。 このセクションを起動可能にします-を入力します。 デバイスに新しいパーティションテーブルを書き込むには、 wを押します。

次に、フラッシュドライブの最初のセクションに新しいファイルシステムを作成します。 ファイルシステムとしてExt2を選択しました-そのサポートはどこにでもあり、この種のシステムでのジャーナリングはあまり意味がありません。 Ext2でパーティションをフォーマットするには、次のコマンドを実行します。

mke2fs -m 0 -N 2000 /dev/sdc1





-m 0は、ルートスペースの予約を無効にします（それが意味するものは何でも）

-N 2000は十分なinodを提供します。



システムをまとめる



新しいパーティションを別のディレクトリ（たとえば、/ newroot）にマウントします。

mkdir /newroot

mount /dev/sdc1 /newroot





各Linuxディストリビューションには、必要なディレクトリのセットがあります。 次のディレクトリを作成します。

/bin

/boot

/dev

/etc

/proc

/sbin

/var





オプションは、/ sys、/ lib、/ var、/ mnt、/ usr、および/ homeです。必要に応じて後で作成できます。



/ dev



まず、必要なデバイスを/ devディレクトリに入力します。 それらは、console、kmem、mem、null、ram0、tty1です。 cp -dpR /dev/DEVICE /newroot/dev



を使用して作業システムから直接コピーしcp -dpR /dev/DEVICE /newroot/dev



は上記の各デバイスです。 -dpRオプションを使用すると、ファイルの内容ではなく、ファイル自体がコピーされ、すべてのアクセス権が保持されます。 この方法があなたの好みに合わない場合-より時間がかかりますが、おそらくより正確にすることができます-ls -lを使用して各デバイスのメイン番号と低い番号を見つけ、 mknodを使用して作成します。

前述の必要なデバイスに加えて、ハードディスクデバイス（ hd *およびsd * ）をコピーし、1ダースの端末デバイス（ tty * ）からもコピーします。 loop *も必要になる場合があります 。 必要に応じて、残りのデバイスファイルをコピーします。



/など



このディレクトリにはシステム設定が含まれている必要があります。 ここでの必須ファイルは、rc（システムによって呼び出し方法が異なる）、inittab、fstab、passwd、group、shadowです。 グループは必要ないかもしれませんが、私はこれを確認しようとしませんでした。

これらのファイルを新しいシステムにコピーしたら、次のファイルを編集する必要があります。



inittab- / bin / initはそれを起動します。 さまざまな実行レベルでのシステムの動作を決定します。 man inittab



inittabファイルのエントリの形式について詳しく読むことができますが、非常に重要な点があります：標準のinittabとは異なり、busyboxのバージョンにはidフィールドとrunlevelフィールドを含めるべきではありません。 私のプリミティブなinittabは次のようになります。

fstab-ここではすべてが通常通りです。 ルートシステムとして、ハードドライブではなく、フラッシュドライブとして指定するだけです。



passwdおよびshadow-これらのファイルには、ルートのエントリのみが残されます。 残りは必要ありません。 さらに、デフォルトのホームディレクトリとコマンドラインシェルを変更することもできます。 通常、/ bin / bashまたは/ bin / zshがシェルとして指定されますが、busyboxにはプリミティブなshしかありません。 もちろん、誰もシンボリックリンクbashを作成することを気にしません-> sh。



rcは、inittabで指定されたブートスクリプトです。 それは非常に複雑で他のrc。スクリプトを含むことができますが、単純なシステムの場合は大幅に単純化できます（私のrcの例）：

/ビン



Busyboxはこのディレクトリにインストールされます。 これは多くの方法で実現できます。 たとえば、手動でビルドする場合は、構成中に--prefix = / newrootを追加します。 パッケージマネージャを使用する場合は、ヘルプを読んでください。インストールするパッケージのルートディレクトリを指定できるオプションが必要です。 たとえば、ArchLinuxでは、pacmanでこれは-rスイッチです。

pacman -S busybox -r /newroot -b /var/lib/pacman



/ pacman pacmanは/ newroot内のパッケージデータベースを見つけることができないため、-bオプションが必要です。

インストール後、/ newroot / binディレクトリに移動します。 busyboxを起動し、必要な組み込みユーティリティと不要な組み込みユーティリティを決定します。 各ユーティリティについて、対応するユーティリティ名でbusyboxへのシンボリックリンクを作成します。 例：
ln -s ./busybox ./cd
必ずsh、login、mountという名前のリンクを作成してください！



/ sbin



このディレクトリに移動して、.. / bin / busyboxへの必要なリンクを作成します。 一般に、このシステムでは2つのディレクトリ-/ binと/ sbin-に分けることはあまり意味がありません。ユーザーとルートが1人しかいないからです。

init、getty、poweroff、shutfown、reboot、haltという名前のリンクを作成してください！



/ブーツ



このディレクトリに、カーネルのみをコピーし、vmlinuzという名前を付けます。

GRUBをインストールする



GRUBブートローダーのインストールは、grub-installコマンドを使用して行います。

grub-install --root-directory=/newroot /dev/sdc1





インストールが完了したら、簡単なGRUBメニューファイル（/boot/grub/menu.lst）を作成します。

ルート行では、デバイスの変更が必要な場合がありますが、Eee PC 900では、BIOSを選択してメモリカードから起動すると、GRUBはそのように定義しました。

さらに、rootdelayは重要なパラメーターです。これがないと、カーネルはカードリーダーデバイスを見つけることができず、次のように受け取りました。

おわりに



それだけです。 これで、再起動して新しいシステムを試すことができます。 もちろん、BusyBoxを使用すると、非常に機能的なシステムを取得する可能性は低くなりますが、システムは稼働中のシステムのサービスという観点からは多くの能力を発揮します。 さらに、live-OSのこのオプションはファイルシステムの圧縮イメージを使用しないため、システムに加えたすべての変更は再起動後も残ります。そのため、使用中にスクリプトで機能を安全に補完でき、業績を失うことを恐れません。

もちろん、最適化できるものはまだたくさんあります。私は徐々にこれを行いますが、ディスクバックアップの作成と復元の基本的なニーズに合わせて、システムの準備は整っています。
