LinuxでCubieboardをセットアップする際のハイライト

Cubieboardをセットアップするときにどのような困難に遭遇したかを説明します。 私は何か新しいことを明らかにするふりをしません。おそらく、私が明らかにする多くのことは明らかであり、他の人にとってはCubieboardをセットアップするときに役立つでしょう。



そのため、カスタマイズの主なポイントはCubietruckで記述されています。 旅行ミニサーバー。 パート2 "および" Cubietruck。 居心地の良いホームサーバー 。 」 キューブをHDD付きのルーターとして設定するときに発生した問題についてのみ説明します。



順番に始めましょう。



最初の問題は、平均負荷です。 重いプロセスがない場合、このパラメーターは1を下回らず、CPU負荷は1％でした。 実際には、誤った構成がデフォルトでキューブの[usbc0]セクションのscript.binファイルに登録されています。 変更する必要があります。 システムをnandに配置しました。したがって、 



sudo -i mount /dev/nanda /mnt cd /mnt bin2fex script.bin script.fex rm script.bin


vim script.fexを開き、usbc0でusb_detect_type = 1オプションを探し、usb_detect_type = 0にします。 保存： 



 fex2bin script.fex script.bin


再起動し、la> = 0であることを確認します。



2番目の問題は、システムをnandからHDDに転送することでした。 「ルートフラッシュをNandflashからハードドライブに移動する」というガイドに従っても、私はうまくいきませんでした。 キューブはロードを拒否し、無限に再起動しました。軸をもう一度回転させる必要がありました。 私はさまざまなことを試みました。シデをつけて、標準のUブートローダーを編集してみてください。 ただし、古いソリューションはより正確であることが判明しました。 記事Cubietruckの指示に従ってください。 居心地の良い、ホームサーバー」 HDDからブートすることができました。 その結果、uEnv.txtファイルは次のようになりました。 



 console=tty0 extraargs=console=ttyS0,115200 hdmi.audio=EDID:0 disp.screen0_output_mode=EDID:1280x720p50 rootwait panic=10 rootfstype=ext4 rootflags=discard nand_root=/dev/sda1


3番目の問題は、lubuntuアームで404と応答するリポジトリにありました。 



 nano /etc/apt/sources.list


そして次のものに置き換えてください：



sources.list
deb old-releases.ubuntu.com/ubuntuメインユニバース制限マルチバース

deb-src old-releases.ubuntu.com/ubuntuメインユニバースの制限されたマルチバースのラリング

deb old-releases.ubuntu.com/ubuntu raring -securityメインユニバース制限されたマルチバース

deb-src old-releases.ubuntu.com/ubuntu raring -securityメインユニバース制限されたマルチバース

deb old-releases.ubuntu.com/ubuntu raring -updatesメインユニバース制限マルチバース

deb-src old-releases.ubuntu.com/ubuntu raring -updatesメインユニバース制限されたマルチバース

deb old-releases.ubuntu.com/ubuntu raring -backportsメイン制限ユニバースマルチバース

deb-src old-releases.ubuntu.com/ubuntu raring -backportsメイン制限ユニバースマルチバース

deb old-releases.ubuntu.com/ubuntu raring -proposed main limited universe multiverse

deb-src old-releases.ubuntu.com/ubuntu raring -proposedメイン制限ユニバースマルチバース





4番目の問題は、キューブをアクセスポイントとして設定することに関連していました。 DNSは解決を断固として拒否し、プロバイダーdnsovを/etc/resolv.confから/etc/dnsmasq.confに直接追加することで助けました。



dnsmasq.conf
バインドインターフェイス

インターフェイス= wlan0

dhcp-range = 192.168.1.10,192.168.1.100

サーバー= xxxx（プロバイダーのDNS）





また、小さなささいなこと。 / etc / network / interfacesにallow-hotplug wlan0行を追加します。これにより、再起動後にwi-fiが有効になります。



現在の問題は、トレントの不安定な動作です。 キューブは非常に奇妙な動作をし、定期的にWi-Fiをオフにし、速度を低下させます。 私は10個のスレッドに座ってダウンロード速度を制限していますが、これは松葉杖です。
