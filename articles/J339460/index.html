フェニックスはいつRailsを殺しましたか？







通常のプログラミング言語では問題を解決できない場合があります。 大量のトラフィックを伴うリアルタイムメッセージングを実現するタスクに直面しているとします。 最適な方法 明らかな理由から、Rubyはこれに最適なオプションではないため、代替の検索を開始する必要があります。 多くの方法がありますが、ルビーに関しては、Elixirを使用するのが最善の選択です。







Elixirは、Erlang仮想マシンで実行されるRuby構文を備えた関数型プログラミング言語です。 したがって、Rubyの世界の人にとっては、この言語に精通することは非常に簡単です。 完全に理解するためには、ドキュメントをもう一度読み直してみてください。







タイトルからの質問への回答と、すぐ下にあるエリクシールに関する多くの興味深いこと。







絶対に！ PhoenixはキラーRailsではなく、パートナーです




両方のプラットフォームのWebアプリケーションの構造は非常に似ています。 これは驚くべきことではありません。Phoenixコミュニティの最も重要なメンバーの1人は、Railsのコアチームの元メンバーであり、フレームワークの開発に携わる人々のほとんどはルビーです。 まだいくつかの違いがあります。







  • アプリケーションを構築するためのla "Clean Architecture"アプローチの実装。 最も重要な変更は、Phoenix 1.3の最新バージョンで行われました。 詳細については、 この記事をご覧ください 。 Phoenixの部分をアプリケーションの部分から分離し、ビジネスロジックをコンテキストに制限し、リポジトリを使用してデータベースを操作します。これらはすべてRailsの世界では一般的ではありません。
  • Umbrellaアプリケーション（ Umbrellaアプリ）-マイクロサービスの早すぎる使用を放棄し、膨大なビジネスロジックを1つの大きなアプリケーション内のパーツに分割するというアイデア。 このElixir機能の詳細については、 この記事を参照してください 。
  • フェニックスの表現はテンプレートと呼ばれ、デコレーターに似ていますが、理論的にはこれは特別な困難を引き起こすことはありませんが、相互作用の概念を変え、 現在人気のある花見のようになります 。
  • Phoenixのもう1つの機能-チャンネル。 チャネルに接続するたびに、新しいプロセスが作成されます。その後、エラーが発生しても、他の接続には影響しません。 さらに詳細にこのトピックを検討します。 さらに、Phoenixは、非構造化Webソケットへの接続のための優れた抽象化を備えたフロントエンドライブラリを提供します。
  • エリクサー言語のすべての楽しみ。これについては後で説明します。


フロントエンドで作業するための組み込みツールから、Railsとは異なるアイデアや哲学に至るまで、さまざまな小さな違いもあります。







Erlang仮想マシンとそれが食べるもの



Rubyはオブジェクト指向プログラミング言語です。 これは、生産性とシンプルさを期待して松本幸弘が作成したインタプリタ言語です。 一方、エリクサーは、Jose Valimによって作成され、 BEAM



と呼ばれるErlang仮想マシンで実行される関数型言語です。 プログラムはBEAM



バイトコードにコンパイルされます。これはオペレーティングシステムの単一プロセスのように見えますが、実際には宇宙全体を運んでいます。







BEAM



はサーバー上で継続的に実行されます。 Ruby（およびRails）プロセスは、サーバーがリクエストの処理を開始したときにのみ開始されます。 したがって、たとえば、1時間ごとに何らかのアクションを実行する必要がある場合、Rails環境でCronJob



を使用する必要がありますが、フェニックスでは、 BEAM



内の小さなElixirプログラムがこれを処理します。 







 defmodule Stack do use GenServer # Callbacks def handle_call(:pop, _from, [h | t]) do {:reply, h, t} end def handle_cast({:push, item}, state) do {:noreply, [item | state]} end end # Start the server {:ok, pid} = GenServer.start_link(Stack, [:hello]) # This is the client GenServer.call(pid, :pop) #=> :hello GenServer.cast(pid, {:push, :world}) #=> :ok GenServer.call(pid, :pop) #=> :world


Elixirでタスクを実行するには、新しいプロセス（Erlangの魔法の世界ではスレッドのように動作しますが、実際にはどちらでもありません）を作成できます。一方、Rubyでは、バックグラウンドタスクを別のキューに配置する必要があります（たとえば、 、 Sidekiq



を介して）、Webアプリケーションとは独立してスケーリングできるようにします。







プロセスが分離され、メッセージの交換中にデータの新しいコピーが作成されるため、システムの反対側でデータの破損や変更を心配する必要がなくなります。 プロセス内の操作は同期的であるため、その実行順序は同じノードよりもはるかに適切です。 別の利点は、効率的なガベージコレクターです。 プロセスが「独自に動作」するとすぐに、OSのメモリが解放されます。







Rubyストリーム（ JRuby



、 Rubinius



）には、他のストリームからのデータとやり取りする機能があります。これは、その後デバッグするのに非常に問題があります。 このような場合、競合状態はかなり急速に進行します。

関数型プログラミングとOOP



エリクサーはRubyに少し似ていることがあります。 rubyはオブジェクト指向プログラミングの世界に属し、Elixirは機能の原則に基づいて構築されているため、これは非常に紛らわしいです。 これらは2つの平行した宇宙です！ あなたとRubyが「あなた」に長い間切り替えた場合、Elixirを理解するにはまだ時間がかかります。







PhoenixはElixirで書かれた軽快なRailsだけではありません




次のモジュールを使用して、Elixirでメッセージを送信するプロセスを構成するものを見てみましょう。 







 defmodule Example do def start_link do spawn(fn() -> loop() end) end def loop do receive do {:hello, pid} -> IO.inspect("Got hello from #{inspect pid}") end loop() end end


それではIEx



開いてみましょう： 







 iex(1)> pid = Example.start_link() #PID<0.120.0> iex(2)> send pid, {:hello, self()} "Got hello from "#PID<0.118.0>" "#PID<0.118.0>" {:hello, #PID<0.118.0>}


OOPのようですね。 Erlang / Elixirコミュニティは、ユーモアがあっても、これらの2つの言語は、他の一般的なプログラミング言語よりもOOPの概念と一貫性があると指摘しています。 それで、Elixirはオブジェクト指向言語ですか？ 種類はありません。 しかし、それについては後で。







メッセージングの利点は、送信者が事前にわかっており、プロセスの相互作用を制御できることです。 視覚化とデバッグのプロセスは、ロックやミューテックスなどの並列フローを制御するための従来の方法で起こることと比較して、はるかに簡単です。







オブジェクトRubyとは異なり、機能的なElixirのコードはモジュールで構成され、各モジュールには通常、特定の種類の問題を解決することを目的とした一連の関数が含まれています。 Elixirの大きな利点は、特定のアクションの結果が予測可能であることです。







各関数の引数は事前に記述されているため、開発者は、インスタンス変数などで発生するさらなる困難から解放されます。 また、システムの一部が意図していないデータと対話する場合、上記の問題を回避するのにも役立ちます。
