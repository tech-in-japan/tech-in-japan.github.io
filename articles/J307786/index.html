Linuxバージョンの待機：Inkscapeイメージエディターコードの確認

この記事では、よく知られている別のオープンソースプロジェクトであるInkscape 0.92ベクトルグラフィックエディターのテストに焦点を当てます。 このプロジェクトは12年以上にわたって開発されており、さまざまな形式のベクターイラストを扱う多くの機会を提供します。 この間、そのコードベースは60万行に成長し、PVS-Studio静的アナライザーを使用してテストする時間になりました。





はじめに



Inkscapeは、クロスプラットフォームの無料のベクターグラフィックエディターです。 世界中のアマチュアやプロがイラスト、アイコン、ロゴ、図、地図、Webグラフィックを作成するために広く使用しています。 Inkscapeは、この分野で最も人気のあるエディターの1つになりました。 このプロジェクトは2003年にSodipodiプロジェクトの分岐として作成され、現在も開発中です。 Inkscapeの詳細については、 公式Webサイトをご覧ください 。



最新バージョンのInkscapeが検証に使用されました-0.92のコードは、GitHubのリポジトリおよびPVS-Studio 6.07静的アナライザーで利用できます。 リンクからダウンロードできます。 ただし、執筆時点では、ダウンロードできるのはPVS-Studio for Windowsのみです。 しかし、状況はすぐに変わります。 また、すでにPVS-Studio for Linuxのベータ版をテストするボランティアにサインアップできます。 詳細は、記事「 PVS-StudioがLinuxに対する愛を宣言しています」を参照してください。







しかし、間違いに戻ります。 この記事では、最も興味深いアナライザーメッセージを選択して説明したことに注意してください。 より徹底的な検証のために、プロジェクトの作成者は、PVS-Studioの一時キーとレポートを入手できます。 パブリックPVS-Studioがまだないため、Windowsで実行されているPVS-Studio Standaloneツールを使用してレポートを表示できます。 はい、それは便利ではありません。 しかし、PVS-Studio for Linuxのリリースの幸福な瞬間の少し前に、皆さんに辛抱強くお願いします。

新しい後のnullへのポインタを確認する



PVS-Studio警告： V668メモリが「new」演算子を使用して割り当てられたため、「outputBuf」ポインターをnullに対してテストしても意味がありません。 メモリ割り当てエラーの場合、例外が生成されます。 gzipstream.cpp 180 

bool GzipInputStream::load() { .... outputBuf = new unsigned char [OUT_SIZE]; if ( !outputBuf ) { // <= delete[] srcBuf; srcBuf = NULL; return false; } .... }


最新のC ++標準によれば、メモリを割り当てることができない場合、 new演算子は例外std :: bad_alloc（）をスローし、 nullptrを返しません。 システムがメモリの割り当てに失敗すると、例外がスローされ、関数の実行が停止するため、プログラムは条件の後にブロックに入ることはありません。



この場合、これによりメモリリークが発生する可能性があります。 この問題の最も明らかな解決策は、 try {....} block catch（const std :: bad_alloc＆）{....}を使用することですが、メモリを明示的に解放する代わりにスマートポインターを使用することをお勧めします。

監査中に、不注意によって行われた多くのエラーが発見されました。 PVS-Studio静的アナライザーは、このようなエラーを効果的に検出できるため、プログラマーの時間と神経を節約できます。 主なことは、コード分析を定期的に実行して、タイプミスやその他の欠陥をすぐに特定することです。 このような1回限りのチェックは、PVS-Studioを適切にアドバタイズしますが、効果はありません。 静的アナライザーからのメッセージを、コンパイラーからの拡張警告として扱います。 また、コンパイラメッセージを使用すると、リリース前に一度だけではなく、常に作業する必要があります。 この類推が、コードの品質を心配しているプログラマーの魂に近づき、理解できることを願っています。
