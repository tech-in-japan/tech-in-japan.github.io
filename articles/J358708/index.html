フロントエンド2018：さまざまなフレームワークとミドルの欠如

フロントエンドは非常に競争の激しい環境です。 ここでキャリアを始めるのは簡単ですが、真ん中に行くのは難しいです。 さらに、ホリバー向けの新しいフレームワークとテーマが毎日登場する場合、どの方向に開発するのかという疑問が生じますか？



5月末にモスクワでRIT ++の一部として開催されるFrontendConfのプログラム委員会のメンバーであるSergey Popovに、現代のフロントエンドの外観と動きについて尋ねました。 途中で、レポートがどのように選択され、ここでどのような困難が生じるかについて話しました。







-あなたにとってのフロントエンドとは何ですか？



-私にとって、フロントエンドはJavaScriptだけでなく、インターフェースのレイアウトでもあります。 筋金入りのプログラミングに関係のないものすべてに対して、やや見下すような態度がしばしば形成されます。 ただし、このような問題を解決しないと、サイトは存在しません（バックエンドもありません）。 私は、特にハードコアJavaScriptではなく、レイアウト部分の責任を負っています。



-現在、このセグメントで何が起こっていますか？



-フロントエンドのすべてのコンポーネントは、今日非常に活発に開発されています。 問題を解決するための非常に多くのフレームワークとアプローチがあります。 近年、この「動物園」はすでに不条理のポイントに達しました。GitHubにスターが増えているという理由だけで、人々はテクノロジーを選択し始めています。 技術的な問題を解決する上で、ファッションの概念（まず同じフレームワークの名前）が重要な役割を果たし始めます。 しかし、それは業界です。 フロントエンドには、JavaScriptという1つの言語しかありません。 バックエンドとは異なり、他の言語を使用することはできません（PHP、Python、Rubyがあり、Node.jsをサーバー側のプログラミング言語などとして使用できます）。 そして、アドオンを開発する必要があります。これは、JavaScriptでの記述をより簡単かつ優れたものにするフレームワークです。



-さまざまなフレームワークがフロントエンドの基本機能の1つですか？



-そう思う。 しかし、それが良いか悪いかを言うのは難しいです。 私たちは、複数の言語または1つのフレームワークのみが存在するフロントエンドを使用したことはありませんでした。 おそらく選択肢が少なければ、セグメント全体が1つの「副詞」で通信するため、開発が容易になります。 しかし、コミュニティの観点から見ると、さまざまなフレームワークに関する多くの興味深い意見があります。 話すべきことがあります。



フレームワークが表示され、人々がファッションからではなく自分のタスクから始めて、あるフレームワークから別のフレームワークに移動すると（ビジネスは常にそのような変換を正当化する必要があるため）、このすべての多様性が誰かに本当に必要であることを意味します。



-そして、あなた自身がフレームワークを切り替える必要に直面しましたか？



-ほぼ。 以前の作業場所では、ある時点で生産性が低下しました。 当初は、すべてがネイティブJavaScriptで記述されていましたが、別の機能のリリースにより、このコアで製品をさらに開発することはできないことに気付きました。 彼らは、リーダーシップとの対話を開始し、彼らが何に基づいているかを説明しました。 問題を解決するために何をすべきか、それがどの程度正確に役立つか。 その結果、コア全体がReactに書き換えられました。 私と一緒に約3から4か月かかり、私が去ってから約6か月かかり、最終的には努力する価値がありました。



-一般的なフレームワークの開発に傾向はありますか？



-正直なところ、私は小さなフレームワークに従っていません。 大きなもののうち、AngularとReactがありました。 Vueが登場しました。 最初は彼は懐疑的だったが、それから彼は人気を得始めた。 それを研究したすべての人がVueに特化して書けば、このフレームワークは最大の市場シェアを持つことになります。



しかし、他のフレームワークもまだ立っていません-新しいバージョンが出てきています。 さらに、市場には、依然として使用されている膨大な数の古いフレームワークとライブラリが残っています。



近い将来、3人のリーダーが市場のために互いに戦うように思えます。 ただし、これは明日、状況がまったくないことを意味するものではなく、その結果、新しいフレームワークが表示されるか、古いフレームワークのいずれかを放棄する必要があります。 フロントエンドはかなり不安定なフィールドです。



-そして、あなたはどのフレームワークが好きですか？



-少なくともVue、React、Angularを試すことで、特定のフレームワークの使用に関する個人的な好みを表現できます。 Reactでしか書いていないので、客観的に話すことができません。



-セグメントは人事に関してどのように発展していますか？ 相対的に言えば、どの方向に風が吹いていますか？



-レイアウト全体を積極的に開発しています。 スペシャリストはより良くなり、UI / UXに深く入り込んでいます-この意味で、私たちは西洋に向かっています。多くの場合、専用のレイアウトはありません。UI/ UXの観点からインターフェースを開発するデザイナーと、ロジックを作成するフロントエンド開発者がいます。



また、これらの機能が1人で組み合わされることもあります。 しかし、ほとんどの部分に別の部門があります-市場には非常に多くのタイプセッターとフロントエンド開発者がいますが、UI / UXに関与している人はほとんどいません（3つすべての分野で有能な人は少ないため、非常に高価です）。



私の意見では、レイアウトの専門家はUI / UXをさらに深く掘り下げる必要があります。これは、これがJavaScriptのようなインターフェースの重要な部分でもあるからです。 個人的には、ある時点で不均衡がありましたが、今は避けることをお勧めします。 ほぼ5年間、私は実際にはレイアウトのみに従事していました-既製のJQueryプラグインを接続して使用するレベルでJavaScriptを学びました。これで十分でした。 そのため、フロントエンドとの接触をほとんど失い、シンプルなタイプセッターになりました。



これを実現して、私はいくつかの方向に並行して開発を始めました。 そして今、私は自由時間があるとき、UI / UXに従事し続けます-UXと組み合わせたフロントエンドを必要とする市場にはまだ少数の企業がありますが。 率直に言って、開発は以前に対処されなければなりませんでした。







-資格のあるフロントエンド開発者を検索することは、バックエンダーよりもほとんど難しいと考えられています。 そうですか？



-スタッフの資格により、すべてが本当に複雑です。 教育に直接関係する人として、今日の主な問題は、中級以上の優秀な開発者の不足であると言えます。 すでに今では十分な高齢者がいないと感じられており、毎年より顕著になります。



しかし、問題の原因は最初のステップにあります。 勉強を終えた人は仕事を見つけることができず、経験を積むことができません。 それらの多くは潜在的に優秀な開発者であり、2〜4年で中程度に成長し、5〜7年で上級に成長できます。 しかし、経験のない後輩を望んでいる人はいません。 誰もが少なくとも中級者を必要としています。資格の低い専門家に対する信頼の欠如によるものです。 そして、後輩が成長し始めるまで、中年の男性も年配の男性もいません（彼らだけでは、彼らはまったく正しく成長できません）。 ある時点で、私たちはちょうど天井にぶつかります-ジュニアは成長しておらず、新しいミドルはありません、そしてすべての良いミドルはその時までにシニア男性にすでに成長しています。



私は、問題を否定する人々はまだ自分でそれを感じていないと信じています。 私たちの採用は、ミドルおよびシニア開発者を検索する際に常にこれに直面しています。 ミドルの地位は、多くの場合、スキルがジュニアとより一貫している人になります。 そして、遠くなるほど悪化します。



-そして何をすべきか？



-私たちは会社のレベルでいくつかの措置を講じようとしています。 たとえば、レイアウトのアウトソーシングを行いました。 私たちの集中的に卒業した男たちは私たちからタスクを取得し、経験を積む。 そしてその後、彼らは雇われる可能性が高くなります。



しかし、このソリューションは、私たち自身のプロジェクトのフレームワーク内でローカルであり、ジュニアの流れ全体ではありません。 結局、サービスも販売する必要があり、プロジェクトを後輩が行うことを顧客に伝えると、疑問が生じます。 業界の問題だけを解決するわけではありません。



-何らかの形で業界のイベントを使用しているのでしょうか？



-残念ながら、会議で起こることは、独白を呼ぶ方が適切です。 あなたの視点を伝え、個々の愛好家を見つけ、彼らとチャットすることができます。 しかし、コミュニティ全体にアイデアを伝えて、何らかの形で変更を開始することはほとんどありません。



私はかつて新人について報告しました-チャンスを与えられるべきだと。 そして、新人が仕事を見つけられるようにするプロジェクトを立ち上げることで、後輩になった人を数人見つけました。 しかし、1年で30〜40人の専門家の問題を解決しました。 つまり、業界の規模ではなく、非常にローカルなソリューションが得られました。



しかし、あなたはこれについて話すことができ、また聞くべきです。



-ジュニアの前にこのような「ホラー」の理由は何ですか？



-専門家の育成には費用がかかります。 どの程度うまく発展するかは明らかではありません。 多くの人は、ジュニアが成長するとすぐに彼が去ることを恐れています。 しかし、私の意見では、フロントエンドの入札は常に仕事を変えているため、これは少し奇妙です。 これはすでに標準になっています。



さらに、潜在的な品質の問題を恐れています。 しかし、率直に言って、インターネット上のリソースの半分はすでにかなり低品質です。 これが私たちの共通の問題です。ビジネスと開発者の両方です。 品質について考える必要があることを忘れたとき、時にはビジネスのためにではなく、ウェブを下に引っ張ります。



毎年、より多くの専門家がいます（客観的に、フロントエンドに入るためのしきい値は同じバックエンドよりも低い）ので、ジュニアが信頼されるべきであるというトピックについて、コミュニティとの対話を複数回行いました。



-フロントエンド開発者は、今日JavaScript以外に何を学ぶ必要がありますか？



-JS開発者は、組版できる必要があることを理解する必要があります。 何らかの理由で、多くの人々はこれを好まない、これはフロントエンド開発者の責任ではないと考える人もいますが、これは真実ではありません（専門職の名前に応じて、フロントエンドはインターフェース全体を扱い、レイアウトとUXの両方を含みます）。



私の意見では、JS開発者にとって最も重要なことは、1つのフレームワークにとらわれず、フレームワークからJSを学び始めることができないことを理解することです。 現在、React、Angular、Vueの各開発者がいますが、JSやフロントエンドの開発者はますます少なくなっています。 しかし、これは行き止まりです。 私たちはすでにこれを経験しました-かつてjQueryで1万行のコードを書くことができる多くのjQuery開発者がいましたが、JavaScriptで同じことをする準備はできていませんでした（実際、jQueryはJS）。



JavaScript自体を学習する必要があります。そうすれば、フレームワークまたはライブラリはコードを記述するための単なるツールになり、開発者は異なるスタックを持つチームからチームに簡単に切り替えることができます。



-そして、フルスタック開発の方向でのスペシャリストの開発についてどう思いますか？



-正直、懐疑的。 決して、フルスタックの開発者を怒らせたくはありません（多くの経験を持つ本当にクールな専門家を知っています）が、私は誰もがそれぞれの分野でプロフェッショナルであるべきだと信じています。



私はほぼ10年間レイアウトを行ってきました。 現時点では、この分野のほぼすべてを知っています（もちろん、試したことがないこともありますが、それらが存在し、どのように適用できるかは知っています）。 そして、3年の仕事を持つ人が自分をフルスタックと見なすのを見るのは少し奇妙です。3年ですべての技術を学ぶことができるとは思えません。 15年以上これを行っている人の資格は問題になりません。 しかし、一つのことで開発する方が良いと思います。なぜなら、あなたが先に進むほど、この分野の専門家としてより価値があるからです。 市場にはフルスタックをもっと好む人もいますが、 そして、そのような専門家の需要があります。



-そして、昨年のフロントエンド技術計画で何が変わったのですか？



-もちろん、新しい技術が登場しました。 前回の会議で、私はCSSグリッドレイアウトについて話していましたが、今ではすでに完全に受け入れられています。 CSS Houdiniによって開発されました。



Vueは非常に大きなブレークスルーをもたらしました。 しかし、明確ではありませんが、これは人気や品質の突破口です。 1年前、AngularまたはReactで書く人、Vueを研究する人は条件付きで市場で際立っていましたが、現在Vueはその市場シェアを獲得しており、1年でそれを完全に吸収すると信じられています（知識人はこれが最高だと言いますフレームワーク）。



今年の重要なトレンドは、古いブラウザのサポートをさらに拒否することでした。 今年の初め、マイクロソフトは、Internet Explorerが唯一のブラウザであった最新のOSであるWindows Phone 8.1を「埋め」ました。 現在、公式には、たとえばEdgeが存在しないOSはありません。 これは、誰もがIEの使用をやめ、最新のブラウザーに切り替えたことを意味しません。 ただし、これは実際に業界を減速させる古いブラウザを殺すための重要なステップです。

発生する問題のほとんどは、クロスブラウジングに残ります。 それらは解決可能ですが、古いシステムをサポートしているという理由だけで、100％最新のテクノロジーを使用することはできません。







-あなたのプログラム委員会は、ファッションとテクノロジーの闘争の中でどのようにレポートを選択しますか？



-各レポートの決定の前に、トピックに関するかなり長い議論が行われます-まず第一に、スピーカーと。 私たちは最終的に何を聞くかを理解しようとしています：実際の経験または抽象的な推論。 そして、問題がそのように解決された理由。



たとえば、変化するフレームワークに関するレポート。 特定の問題を解決する場合、人はフレームワークを変更する必要があります。 彼は彼を引き継いで、なぜ、何のために、そしてこれが彼の問題をどのように解決したかを伝えます。 プログラム委員会の側では、スピーカーから真実を得ようとしています。スピーカーとの会話とプログラム委員会内での議論を通して、レポートのトピックを最大化しようとしています。 この方法では、ナプキンに関する議論だけでなく、実際に一般の関心を引くことができるレポートのみが選択されます。



-どのレポートがより可能性が高いですか？



-今年、「バックグラウンド」レポートを拒否しようとしています。 フロントエンド開発者は、ドキュメントセクションのライブラリWebサイトに個別にアクセスし、そこに記載されている内容を理解し、作業を開始できる必要があります。 この会議では、特定の問題とその解決方法に集中しています。 それはあなたが志を同じくする人々を捜して、実際に助けるのを許します。 これは本質的に経験を通じて学ぶことです。



一般に、プログラムは非常に異種です。 私たちの仕事は、フロントエンドに関係するすべての人にとって可能な限り興味深いものにすることです。 そして、毎年、プログラム委員会はより調和して機能しており、レポートの選択はますます厳しくなっています。



-多くのレポートがすでに会議に提出されています。 業界に最も関係する問題を特定できますか？



-トレンドを特定することは困難です。 すべてのレポートは何か新しいことについて語っています。 JavaScriptレポートの数値的優位性はあまり好きではありませんでした。 はい、それらの中には本当に筋金入りの物語がたくさんありますが、それでも私自身はプログラミングよりもフロントエンドに近く、業界はすべての要素の相互作用であると考えています。 したがって、CSS、UI / UX、ツールなどに関するレポートが非常に少ないことは好きではありません。



リファクタリングに関するいくつかのレポートがありますが、これは原則として珍しいことです。 誰もこのトピックについて話すことを好みません。 しかし、そのようなトピックが存在するため、プロジェクトは非常に発展しており、リファクタリングする必要があります。 そして、これはコミュニティの発展のレベルを示しています。



西洋では、インターフェースのアクセシビリティのトピックが非常に人気があります。 彼らはこれについて私たちと話します。 自国のオープンソースプロジェクトの作り方に関する報告書が提出されましたが、一般的に私たちの国ではオープンソースプロジェクトの文化はあまり発達していません。



-FrontendConfはどの方向にさらに発展すると思いますか？



-私は英語の方向に進む必要があるように思えます。 私たちのコミュニティが国際標準に従って発展するように、英語のプレゼンテーションのストリームを持ち、そこでヨーロッパの会議から講演者を招待したいと思います。 しかし、RIT ++は常に国内市場により焦点を当ててきたため、会議自体のそのような開発が必要かどうかを判断することは困難です。
