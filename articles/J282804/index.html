FB2形式の電子書籍のサポートを実装するSDK





サイエンスフィクションの「ノーベル」は 、中国の作家Liu Cixin（劉慈欣）が作品The Three-Body Problem （三體）で受け取ったことを知っています。 バラク・オバマ（ 証明 ）とマーク・ザッカーバーグ（ 証明 ）はこの本に注意を払った。





Olga Braathenは彼女自身のイニシアチブでこの本をロシア語に翻訳しました（ ここではfb2をダウンロードできます ）。



2016年の「ノーベル賞」のもう1つの候補者は、 Neven Stevenson （雪崩とクリプトノミコンを書いた）で、 Sevenevesの作品（ ここから英語でダウンロードできます 。誰もロシア語に翻訳しなかったのは残念です）。



EDISONの開発者は、数年前に書いたElectronic Document Access Controlプログラムを作成しました。今日は、FB2形式の電子書籍のサポートを実装するSDKについて説明します。




はじめに



図書館分野での情報技術の使用は、読者にフィクション、科学および技術文献の豊富なセットへのリモートアクセスを提供するインターネットサービスの出現をもたらしました。 そのようなサービスは、司書職を新しいレベルに引き上げます。 図書館は単一のネットワークに統合され、デジタル化されたコンテンツの地理的に分散した巨大なデータベースを形成します。インターネットでの図書館サービスの提供は、対象読者を拡大し、追加収入を提供します。 主要なユーザーにもメリットがあります。時間をかけて図書館を訪れたり、本を個人的に使用したり、タイムリーに配達したりする必要がありません。 特定の地域では利用できない珍しい文献へのアクセスが可能です。 権利者は、相互に有益な条件でコンテンツを提供することにより、ロイヤリティを受け取ることができます。



図書館サービスでは、著作権の遵守とコピー保護に特に注意が払われます。 独自の形式はトランスポートレベルで使用され、クライアント側のソフトウェアは受信したコンテンツをディスクに保存しないでください。



コンテンツの大部分は、スキャナーを使用して物理メディアを手動でデジタル化し、テキストを認識して検索できるようにすることで生成されますが、これが唯一のソースではありません。 電子書籍を保存するために設計された多くのデジタル形式があり、これらの形式で既にデジタル化された本は少なくありません。 したがって、図書館サービスではさまざまな電子形式のサポートが必要です。 サービスの1つでフィクションブック形式の電子書籍のサポートを導入するためのSDK（開発者ツールキット）の開発についてお話します。



挑戦する



コンテンツ配信サービスは、リモートリポジトリ内の文献へのページ単位のアクセスをユーザーに提供します。 各ページへのアクセスは固定され、課金されます。 このサービスは全文検索を実装し、その結果は半透明の長方形で強調表示されます。 サーバーとクライアント間のトラフィックは保護されており、独自のバイナリ形式です。



EDISONプログラマーは、ソリューションのサーバー部分とクライアント部分を構築するときに共通のコードベースを使用するだけでなく、FB2形式の電子書籍のサポートの実装を簡素化するすぐに使用できる一連の機能を最終開発者に提供するSDKを作成するタスクに直面しました。



サービスの機能に基づいて、SDK関数のセットの要件が事前に決定されました。

  1. 書誌情報の取得。
  2. 電子書籍のページ数を取得する。
  3. 関連ページへのリンクを含む全文検索結果を取得する。
  4. フルテキスト検索の結果を強調表示するために、四角形の座標を取得します。
  5. 任意のページコンテンツをバイナリ形式で受信し、ページをレンダリングします。


実装と技術：C ++ / Qt



解決策



FB2の電子書籍は1ページのドキュメントです。 この形式は、ドキュメントの外観に関する情報を提供しません。 まず、ドキュメントの内容を複製せずに、FB2ドキュメントをページに分割する問題を解決する必要がありました。 その結果、元のドキュメントの解析、ドキュメントのレンダリングおよびページングの結果として取得された、元のFB2ドキュメントに関するメタデータを格納するインデックスファイル形式が設計されました。

インデックスファイルには、XMLドキュメントのフラグメントの場所がドキュメントの先頭からのオフセットの形式で、フラグメントの長さが文字数で、さらにXMLプレフィックスが含まれています。



インデックスファイルの構造には、3つのセクションが含まれます。

  • description-ドキュメントの説明を含むフラグメント。
  • バイナリ-元の文書に写真が含まれるフラグメント。
  • page-ページサイズ、インデント、およびフォントの開始と終了に指定されたパラメーターを使用したレンダリングの結果としてページが取得されたドキュメントのフラグメント。




XMLドキュメントのフラグメントの場所に関する情報を使用すると、解析する必要なく元のドキュメントから目的の情報を差し引くことができます。XMLプレフィックスを使用すると、目的のページのみのマークアップを含むミニチュアXMLドキュメントを作成し、解析して、ユーザーの要求に応じて目的のページをすぐにレンダリングできます。



ドキュメントのページネーションを含むインデックスファイルの例。 

<document> <description> <fragment> <offset>418</offset> <length>5230</length> <prefix><![CDATA[<FictionBook xmlns="http://www.gribuser.ru/xml/fictionbook/2.0" xmlns:xlink="http://www.w3.org/1999/xlink">]]></prefix> </fragment> </description> <binary id="cover.jpg" > <fragment> <offset>43034</offset> <length>48151</length> </fragment> </binary> <page number="1" > <fragment> <offset>5657</offset> <length>1779</length> <prefix><![CDATA[<FictionBook xmlns="http://www.gribuser.ru/xml/fictionbook/2.0" xmlns:xlink="http://www.w3.org/1999/xlink"><body>]]></prefix> </fragment> </page> <page number="2" > <fragment> <offset>7436</offset> <length>2366</length> <prefix><![CDATA[<FictionBook xmlns="http://www.gribuser.ru/xml/fictionbook/2.0" xmlns:xlink="http://www.w3.org/1999/xlink"><body><section><section><p>]]></prefix> </fragment> </page> </document>




ドキュメントをページに分割するアルゴリズムのアイデアがあったとき、私たちはそれを実装し始めました。 インデックスを作成するために、Qtライブラリの標準クラスを使用して元のドキュメントのストリーミング解析の方法を選択しました。これは、QXmlStreamReader :: characterOffsetメソッドを使用してXMLファイルを連続して読み取り、ファイル内のオフセットに関する情報を文字数で格納できるためです。



FB2ドキュメントを解析するプロセスでは、タグからタグへ移動するときに、ドキュメントの段落が単語のセットに解析され、それが再び行に組み立てられます。 設定ファイルに従って、指定されたページ余白の幅と段落のインデントを考慮して、各行が最大幅に設定されます。 行の場合、設定ファイルで指定された行間隔も指定されます。 XMLドキュメントのタグに応じて、フォントパラメーター、サイズ、スタイル、および配置が設定されます。 見出しと小見出しの場合は中央揃えが設定され、エピグラフの場合は右揃えが設定され、デフォルトでは左揃えが設定されます。 文字列に単語を追加すると、追加されたすべての単語の長さを合計することにより、文字列の長さが再計算されます。 行の長さが指定されたページ幅を超える場合、行はページオブジェクトに追加されます。 行に収まらない単語は次の行に追加されます。 ページに行を追加すると、行間隔を考慮してすべての行の高さが再計算されます。 画像を埋め込むとき、ラインの高さは、ラインに追加されるオブジェクトの最大の高さと見なされる必要がありました。 インデントを考慮して、追加されたすべての行の高さが指定されたページの高さを超える場合、次のフラグメントがインデックスファイルに追加されます。 説明されているアルゴリズムは、FB2ドキュメントをページに解析するとき、およびインデックスファイルを使用してページにランダムにアクセスするときに使用されます。



QXmlStreamReader :: characterOffsetメソッドは、ドキュメントのページへのランダムアクセスで、バイト数ではなく文字数でオフセットを返すため、元のファイルの先頭を読み取り、ドキュメントの興味深い部分だけを差し引く必要があります。 seekメソッドを使用したバイト単位のファイルオフセットは、必然的にエラーにつながります。 

 QString Document::documentFragment(uint offset, uint length) { QString fragment; QFile file(m_fileName); if (!file.open(QIODevice::ReadOnly)) { m_error = IOError; return fragment; } QTextStream fileStream(&file); fileStream.setCodec("UTF-8"); fileStream.setAutoDetectUnicode(true); fileStream.seek(0); fileStream.read(offset); fragment = fileStream.read(length); file.close(); if ((uint) fragment.size() < length) { m_error = IOError; fragment = QString(); return fragment; } return fragment; }




これにもかかわらず、パフォーマンスの低下はありません;ドキュメントの最後のページへのアクセスは、最初のページへのアクセスと同じくらい速く、1秒もかかりません。 実際には、FB2形式の写真のない本の平均ボリュームが10 MBを超えることはめったにありません。



7 MBファイルを998ページに分割し、インデックスを準備するには、約10秒かかります。 9 MBファイルを1576ページに分割するには、約15秒かかります。 平均して、1秒間に約100ページがレンダリングされます。 インデックスがある場合、ドキュメントは50ミリ秒で開きます。



次に、ドキュメントのページを参照して全文検索の問題を解決する必要がありました。 そして、ここでは、速度を確保するために、ドキュメントのコンテンツを複製する必要がありましたが、XMLマークアップはなく、通常のテキストファイルの形式でした。 ページの開始マーカーと終了マーカーは、ゼロバイトとしてテキストインデックスに挿入されます。 検索結果をページにバインドし、長方形の座標を保存するには、2つの補助インデックスをバイナリ形式で整理する必要がありました。 検索結果をページにリンクするための補助インデックスには、ページ番号、テキストインデックスのページマーカーの開始バイトと終了バイトのシリアル番号が格納されます。



全文検索は、顧客が提供するライブラリを使用して実行され、特定の単語のすべての単語形式が返されます。 検索クエリは単語のセットに分割され、単語ごとにすべての単語形式が検索され、検索された単語形式の生成された配列のテキストインデックスで検索が実行されます。 テキストインデックスの補助インデックスとページ開始/終了マーカーを使用して、検索結果が属するページ番号が決定されます。 長方形の座標を持つ補助インデックスは、ページが要求されたときにのみ形成されます。 各ページに個別のインデックスファイルが生成されます。 四角形の座標を取得するプロセスでは、行を単語に分割し、各単語の境界を計算するため、元のドキュメントをページに分割するための同じアルゴリズムが使用されます。インデックスは、同じページに切り替えるときにこのアルゴリズムが再度呼び出されるのを防ぐのに役立ちます。



全文検索用のインデックスの作成には、約10 MBの容量のドキュメントで約1分かかります。 インデックスがある場合、検索は1,576ページのドキュメントで約1秒間機能します。



もう1つの驚きは、見つかったテキストの断片の上に半透明の長方形が表示されたことです。 単語の境界を計算するための数学はもともとピクセル単位であったため、ドキュメントのページを拡大縮小するときにいくつかのピクセルの不正確さが生じていました。 解決策が見つかりました。整数の代わりに浮動小数点値を使用し、コードの大部分を修正して、DPI出力デバイスを考慮して、すべての計算をインチに変換する必要がありました。 

  m_dpiX = (qreal) QApplication::desktop()->physicalDpiX(); m_dpiY = (qreal) QApplication::desktop()->physicalDpiY(); QFontMetricsF fm(m_font); m_rect = fm.boundingRect(m_text); m_textDescent = fm.descent() / m_dpiY; qreal width = m_rect.width() / m_dpiX; qreal height = m_rect.height() / m_dpiY; m_rect.setSize(QSizeF(width, height));


フィニッシュラインでは、一連の長方形を含むページビューをバイナリ形式にシリアル化し、そこから読み戻してページのコンテンツをクライアントに転送し、同じSDKを使用してレンダリングする問題を解決するために残りました。 ここではすべてが非常にシンプルであることが判明しました。標準のQTライブラリクラスであるQDataStreamが助けになりました。



問題を解決する過程で、次のクラスが特定されました。

  • Fb2Document-メインクラスであるFB2ドキュメントは、ドキュメントの解析、ページネーション、インデックス形成のロジックをカプセル化し、生成されたインデックスと全文検索を使用して任意のページへのアクセスを提供します。
  • Fb2Page -FB2ドキュメントのページ。ページに一連のドキュメント行を入力し、ページをレンダリングするロジックをカプセル化し、ページの終わりのサインを決定します。 ページの幅と高さ、およびページの端からのインデントをインチ単位で設定するためのインターフェイスを提供します。
  • Fb2Word-指定されたフォントパラメーターに従って、ドキュメントのキャンバス上の単語境界をインチ単位で計算するロジックをカプセル化する単語。ドキュメントページの単語をバイナリ形式でシリアル化し、バイナリ形式から単語を読み取ります。
  • Fb2String-単語のセット（Fb2Word）からの行で、行を単語のリストで埋めるロジックをカプセル化し、行末の符号を決定し、設定ファイルで指定された行間隔を考慮して、行を左、右端、中央に揃え、バイナリ形式でドキュメントページの行をシリアル化し、読み取りバイナリ形式の文字列から。
  • Fb2Image-ドキュメント画像のレンダリング、画像のバイナリ形式へのシリアル化、バイナリ形式からの画像の読み取りのロジックをカプセル化した画像。
  • Fb2Index-インデックス。インデックスファイルの形成と読み取りのロジックをカプセル化します。
  • Fb2Fragment -FB2ドキュメントのフラグメントは、インデックスファイルの基本構造を表します。
  • Fb2Settings-設定ファイル。読み取り/書き込み設定ファイルを操作するロジックをカプセル化します。
  • Fb2Func-クラスラッパーは、タスクの設定時に指定されたインターフェイスに従ってSDK関数のセットを提供します。
  • 辞書 -形態学的辞書のクラスラッパー。








すべてのSDKクラスのメソッドは単体テストでカバーされ、FB2ドキュメントが正しくページ付けされていること、およびクライアントプログラムのユーザーがページを要求するときにまったく同じ画像が表示されることを確認します。ページはインデックスファイルを準備するときにサーバー側で最初にレンダリングされます。



その結果、最終目標が達成されました。 箱入りソリューションよりもSDKを開発する利点は柔軟性です。 最も難しい部分は単純な関数の呼び出しの背後に隠されており、SDKを使用する開発者は、その使用方法を自分で決定できます。たとえば、システム関数の速度とより快適なユーザーエクスペリエンスを確保するために事前にインデックスを構築するか、ドキュメントへの最初のアクセスと最初の検索でインデックスを構築するかなどですディスク領域を節約するためにまれに使用してそれらを要求して削除します。



SDKのパフォーマンスをお客様に示すために、2つのデスクトップアプリケーションがそのベースで実装されました。 FictionBookReaderは、FB2ドキュメントの基本的なリーダーの機能に、検索結果の強調表示によるページ分割および全文検索機能を提供します。







FB2SDKデモは、SDKのサーバーおよびクライアント部分の機能を示します。 サーバーパーツの機能は[サーバー]タブで強調表示されます。このタブでは、ドキュメントの解析とマルチページインデックスの作成、および四角形とフルテキストインデックスを使用したファイルの作成を示します。 クライアントパーツの機能は、生成されたバイナリファイルに従ってドキュメントページのレンダリングを示す[クライアント]タブで強調表示されます。
