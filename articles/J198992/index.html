OpenStackクラスターのハードウェア計画：質問への回答

投稿者：グレッグ・エルキンバード



私の同僚であるAnne Friendと私は最近、 「OpenStackクラウドのハードウェア計画に対処する方法」に関するウェビナーを紹介しました。 ウェビナー中に、私たちはライブで話す時間がない質問に答えることを約束しました。 この記事では、これらの質問に対する答えに専念します。



過負荷のスイッチがあるラックにストレージを追加することについて言及しました。 これを設定する方法について話してもらえますか？



典型的な中央スイッチは、ダウンリンク帯域幅と比較して同じアップリンク帯域幅を持ちません。 たとえば、通常、トライデント+スイッチにはダウンリンク用に48個の10ギガビットポートがあり、合計スループットは960 Gb / sですが、アップストリーム方向の帯域幅として4×40ギガビットポートまたは320ギガバイトのみであるため、約3/1の比率。



これは、通信チャネルを通過するトラフィックを制限する必要があることを意味します。 これを行うには2つの方法があります。 1つは、エッジスイッチのドメイン（L2セグメント）でユーザーVMを実行して、アウトバウンド接続間のトラフィックを削減することです。



トラフィックの2番目の主なソースは、CinderノードとComputeノード間のCinderトラフィックです。 このトラフィックを1つのスイッチに集中させると、アップリンクもオフロードされます。 たとえば、Cinder iSCSIストレージを使用している場合、ラックごとに1つまたは2つのスイッチを提供し、CinderスケジューラーがComputeリソースと同じラックにあるストレージからボリュームを作成することを確認できます。 これらのフィルターは両方ともカスタムであり、NovaおよびCinderスケジューラー用に作成する必要があります。 これはまさに「ターンキーソリューション」ではありませんが、簡単な変更です。



あなたが私的な言葉で説明したトレードオフのいくつかをどのように適用できるかを理解しようとしています。 2つの異なるケースにvCPU / VRAMを割り当てる際の妥協の数値例を挙げることができますか？



詳細に調べるにはユースケースが多すぎますが、実際の計算を見てみましょう。



CPU要件

-100台の仮想マシン

-平均で2つのEC2計算ノード

-最大16のEC2計算ノード

-制限を超えていない



これは次のものに対応します。

-200 GHz CPU容量（100ユーザーx 2 GHz /ユーザー）

-コアの最大数-5（16 GHz /コアあたり2.4 GHz）



計算に基づく：

ハイパースレッディング1.3のオッズ

-10-11コアE5 2640（200 GHz / 2.4 GHz CPU / 6コア）

-5-6デュアルコアサーバー（11コア/サーバーあたり2コア）

サーバーあたり-17 VM（100 VM / 6サーバー）



メモリ要件

-100台の仮想マシン

仮想マシンあたり-4 GB

-最小512 MB、最大32 GB



これは次のものに対応します。

-400 GB合計（100 VM * VMあたり4 GB）



次の計算に基づいています。

-4台の128GBマシン（400GB / 128GB）が必要

-CPUとのバランスを取るには、合計CPU容量に6台のマシンが必要です

-サーバーのメモリを削減し、64または96 GBの6台のマシンで動作（6x64 GBは384 GB、6×96は596 GB）



もう少しメモリが必要な場合は、96 GBのマシンが必要です。



VLANが小規模ネットワークに適していると言うとき、ネットワークはどのくらい小さいのですか？



小規模ネットワークには、4,000未満の仮想ネットワークがあります。 ただし、Neutronでは各ユーザーが複数のネットワークを持つことができるため、4,000人のユーザーに対応できると想定することはできません。 また、静的インフラストラクチャが必要であることを忘れないでください。 これらのネットワークのタグを忘れずに保存してください。



Fuelはネットワーク設定の自動化にどのように役立ちますか？



Fuelは、ネットワーク構成をチェックして、ノードが正しく接続され、適切なVLANタグがすべてスイッチでロック解除されていることを確認できます。



Dell、HPなどの有名なメーカーのハードウェアを使用する方が良いと思いますか、または作成したソフトウェアを使用して同じパフォーマンスを達成できますか？ Open Compute Platformを使用することをお勧めしますか？



簡単な答えは、自社のハードウェアをサポートするのに十分な規模の会社、またはハードウェア障害時のダウンタイムを心配しないほど規模の小さい会社がある場合、仮想データストレージまたは独自のアセンブリのコンピューターを使用できます。 中規模の会社の場合、最適なサービスレベル契約を取得するため、有名なメーカーの機器を使用することをお勧めします。



Open Computeは有望なプラットフォームですが、近日中に提供される広範なハードウェアサポートに依存しています。



別々のnovaサービスを実行しているノードに別々のソフトウェアを推奨しますか？ たとえば、nova-apiを実行しているノードには、glance-apiを実行しているノードよりも多くのメモリが必要ですか？



Mirantisでは、すべてのOpenStackインフラストラクチャサービスをコントローラーと呼ばれる専用ノードに統合することをお勧めします。 このタイプのアーキテクチャは、高可用性を促進します。



ARM（またはAtom）ベースのマイクロサーバーはどうですか？



汎用クラウドを使用している場合、ARMまたはAtomに基づくマイクロサーバーに大きなCPU負荷をかけることは困難です。 ARMでMsSQLまたはOracleサーバーを実行してみてください。 あなたはあまり達成しません。 これらのCPUの制限内に収まる特別なクラウドがある場合は、とにかく使用してください。 クラウドはCPUに完全に依存しているわけではなく、多くのARM / Atomベースのプロセッサのアーキテクチャは、優れたプラットフォームになるための十分な帯域幅やディスクスペースを意味しません。



ブレードサーバーはどうですか？



シェービングブレードはそのままにします。 クラウドには通常のサーバーを使用します。 高密度が必要な場合は、ブレードサーバーの代わりにスレッドフォームファクタサーバー（Dell Cクラス、HP SLクラス）を使用してください。 中央ブレードサーバーモジュールには通常、クラウドで適切に動作するための十分な帯域幅がなく、十分なローカルストレージスペースがないため、シャーシの帯域幅要件に二重の負荷がかかります。 さらに、そのようなサーバーにはプレミアムを支払います。 1つまたは2つのブレードデバイスの設計により、少なくともネットワークのボトルネックが解消され始めていますが、他の疑問も残っています。



共有ストレージなしでリアルタイム移行を提供できますか？



共有ストレージなしでリアルタイム移行を実行できます。 もっと時間がかかります。



小規模なプライベートクラウドの場合、計算ノードの共有ストレージ用のファイバーチャネルまたは1ギガビットの共有ファイルシステムを推奨しますか？



どちらもありません。 10ギグとCephまたは他のブロックストレージを使用します。 FSやファイバーのコストを共有する必要はありません。



swift 6.5xの要件についてもう少し話していただけますか？



これは、記録されたウェビナーのより詳細な回答を含む別の質問ですが、簡単な計算は次のとおりです。



複製係数3を受け入れます。



2つのハンドヘルドデバイスを追加します（クラッシュのための追加スペースが必要です）



さらに、XFSディスク容量の75％を超えると、問題が発生し、次の計算が得られます。



（3 + 2）/。75 = 6.7



展開後、CPUとハードウェアの使用率を確認するためにどのツールを使用または使用していますか？



Mirantisでは、NagiosとZabbixを使用しました（そして成功しました）。



OpenStackをOCP（Open Compute Platform）にデプロイできますか？



はい Mirantis Fuelは通常、ハードウェアアーキテクチャから独立しています。



ディスクレスハイパーバイザーは、ストレージの方程式「ローカルvs共有vsオブジェクト」にどのように適合しますか？ CinderがiSCSIターゲットに接続する機能を損なうことなく、計算ノードをディスクなしでiSCSIクライアントとして管理することは可能ですか、それともハードウェアに別のSANソリューションが必要ですか？



質問を少し変えて、なぜそのような困難が必要なのか尋ねましょう。 Mirantis Fuelには既にオペレーティングシステムが展開されています。 OS用にいくつかの小さなディスクがあると、セットアップが簡単になります。 以前にこれを試しましたが、1つのノードからのOSとCinderの複数のイニシエーターが同じ目標に対処したい場合、アレイに問題があります。 それは価値がありません。



Fuelはインターフェイスボンディングをサポートしていますか？



はい。ただし、Webインターフェイスではなく、コマンドラインインターフェイスを使用する必要があります。



IllumosベースのハイパーバイザーやIllumosを使用して何か作業したことがありますか、それともLinuxでのみ実行されましたか？



ZFSは、Solarisなどのサイドオペレーティングシステムに注意を払うほど包括的ではありません。 はい、警告と制限付きでXENとKVMを実行できます。 独自のオペレーティングシステム開発チームをサポートするのに十分な金持ちであれば、それを行うことができますが、機能は常に遅れます。 ゼロから、さまざまな企業向けに複数のOS開発チームを作成しました。これがあなたの専門分野である場合は、先に進みましょう。 それ以外の場合は、人里離れた道に沿って行く方が良いでしょう。ジャングルを抜けるよりもあなたにとって便利です。
