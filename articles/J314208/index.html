Dart Sassの発表

過去数ヶ月間、私たちは静かに新しいプロジェクトに取り組んできました。 今日、私はダート・サスを全世界に発表する準備ができています。 これは、インストールと開発が迅速かつ簡単になるように設計された、まったく新しいSass実装です。 まだ完成していません-私はそれをSass仕様に合わせようと取り組んでいます-だから今日はバージョン1.0.0-alpha.1を公開しています。 しかし、ダウンロードして、それを使って、 バグの報告を開始できるほど信頼性があります。









リリースページからアーカイブをダウンロードできます-単に解凍し、ディレクトリをパスに追加し、 dart-sass



を実行します。 DartもJavaScriptでコンパイルされているため、 npmがインストールされている場合は、 npm install -g dart-sass



実行してJSバージョンをインストールできます。 あなた自身がすでにDartユーザーである場合は、 pub global dart-sass



からインストールできます。







Sassを書き換える理由



過去数年にわたって、Sassの2つの主要な実装がありました。 Ruby Sassは、主にChrisの多大なサポートを受けて私が書いた最初のものです。 高レベルで簡単に開発でき、新しい機能を導入してリリースした場所でした。 それから、もともとアーロンとハンプトンによって作成され、現在はマルセルとマイケルによってサポートされているC ++実装であるLibSassが登場しました。 これは低レベルのバージョンであり、他の言語にすばやく簡単にインストールして埋め込むことができます。 特に、 Node.jsへのバインドは、JavaScriptの世界でSassを使用する非常に一般的な方法です。







各バージョンのメリットは、他のバージョンの弱点を補完します。 LibSassが高速で移植性がある場合、Ruby Sassは遅く、Ruby以外のユーザーがインストールするのは困難です。 しかし、Ruby Sassは開発が容易ですが、LibSassでは低レベル言語のために新しい機能を追加することははるかに困難です。 補完的な関係は健全である可能性がありますが、それは、どのソリューションも本来あるべきほど良いものではないことも意味します。 5月にMarcelがLibSassチームを正式に辞任したとき、これを見つけました。 （ 著者のメモ：私は「公式に」と言います。なぜなら彼は可能な限りプロジェクトに参加しているからです。しかし、彼が公式の開発者である限りではありません）。







2人の努力を奪われて、LibSassがペースを維持できるかどうか確信が持てなかったので、クリスと私は使用する言語を変更したかったのです。 長い間、大きなスタイルシートを処理する場合、Ruby Sassの方がはるかに遅いことが明らかでした。 CSS をすばやく生成し、新しい機能をすばやく追加できる新しい実装が必要でした。







なぜダーツ？



考えられるいくつかの言語を調べ、いくつかの理由でDartに決めました。 まず、それは非常に高速です -Dart-VMは基本的にJavaScript VMよりもはるかに高速であり、 予備のベンチマークでは、大きなDart Sassファイルの場合、Ruby Sassの5-10倍、LibSassの1.5倍しか遅いことが示されています。 （ 著者注：私はベンチマークの専門家ではなく、テストは選択されました（アドホック- トランス ）。代表的ではないソースファイル。科学的な測定に興味がある方はお知らせください）。 専用のJS実装より1.5〜2倍高速になると慎重に想定していますが、確実には言えません。 さらに、Dartのパフォーマンスは時間とともに向上します。







Dartは簡単に操作できますが、このような大規模なプロジェクトでは、C ++よりもはるかに簡単で、Rubyよりもやや簡単です。 もちろん、JavaScriptのようにこれに精通している人は多くありませんが、言語の実装に関する作業は、いずれにしても外部の参加者が多いことを意味するものではありません。 私は新しい実装でほとんどの作業を行いますが、Dartは現時点で個人的に安心できる言語です（Sassに取り組んでいないときは、Dartチームに所属しています）。 Dartを使用すると、速度が向上します。







RubyやJavaScriptとは異なり、Dartは静的に型付けされるため、コード自体を実行することなく各値の型を推測できます。 また、C ++とは異なり、ガベージコレクションを使用します 。つまり、自分で整理整頓することを心配する必要はありません。 これにより、作成、編集、および保守が簡単になります。 おそらくさらに重要なことは、他のプログラミング言語に簡単に翻訳できるため、LibSassが新しい機能をより高速に取得できるようになることです。







Dartを選択する最後の理由は、少数の言語が誇るもの、つまりJavaScriptの互換性です。 DartはJavaScriptにコンパイルでき、Node.jsで直接使用することも、ブラウザで使用することもできます。 Sassエコシステムの大部分はnode-sassで構築されており、既存のビルドツールとシステムを新しいバージョンに簡単に移行できるように、JSバージョンのDart Sassを可能な限りnode-sassとAPI互換にする予定です。







ここでの唯一の欠点は速度の低下です。V8での実行時のDart Sassの速度は、Dart VMの約2倍です。 ただし、Ruby Sassに比べて3〜4倍の安定した増加が見られます。 最終的には、JSバージョンのユーザーにできるだけ簡単にDart VMへの移行パスを提供できることを願っています。







他の実装はどうなりますか？



LibSassの開発に変更はありません。 マイケルは、 Sass 3.5から新しい機能を追加するために一生懸命に取り組んでおり、新しい機能が追加されてもこのプロセスが継続することを期待しています。 唯一の違いは、LibSassが妥当なパフォーマンスを持つ唯一のバージョンではなくなるため、このバージョン自体を起動する前に、LibSassが最新バージョンの言語と厳密に互換性を持つ必要がなくなることです。







柔軟性が高まると、LibSassのリリースが速くなり、ユーザーが最も望む機能が強調されます。 厳密な互換性は、たとえばカスタムCSSプロパティのサポートなどの重要な革新が、Ruby Sassの対応するバージョンに含まれていたすべての小さくてトリッキーな境界ケース、たとえば :root



が処理されるまでリリースできないことを意味しました :root



可能な限り最大限の互換性を実現しようとしていますが、速度を妨げることはできません。







彼の新しい開発者が来ない限り、Ruby Sassは最終的に完全に終了します。 エコシステムを破壊するリスクがあるため、移行を突然行いたくありません。クリスと私は1年間サポートします。これには、Dart Sassに追加されるのと同じレベルの機能のサポートが含まれます。 この期間を過ぎて開発者として参加することに興味がある人は、来年のコードを紹介させていただきます。 そして誰も話さない場合、Ruby Sassは廃止されサポートされていないものとして公式に認識されます。







私は、Ruby Sassの開発を無料で停止する決定を下していないことを強調したいと思います。 これは私にとって大きな変更であり、簡単な変更ではありません。ほぼ10年間継続的に取り組んでおり、この話を手放すのは難しいです。 しかし、クリスと私はこれについて徹底的に議論し、これが正しい動きであると信じる傾向がありました。 Sassに多くの時間を費やしているため、この時間を実装に費やすことはもはや意味をなさないため、非常に遅いため、最大のユーザーには受け入れられません。







次は何ですか



Dart Sassの最初の安定バージョンをリリースする前に、リストにいくつかの素晴らしいものがあります。







  • Sass仕様との完全な互換性。 特に@extend



    に関して、Dart Sassが正しく動作しない瞬間がまだあります。 Sassの仕様は非常に包括的であるため、解決するのが難しい特定の非互換性を期待していません。したがって、これは失敗したテストの数をゼロに減らすだけです。







  • node-sassのrender()



    とかなり互換性がありrender()



    。 このAPIは、JavaScriptの世界でのLibSassへの主要なエントリポイントです。 これにより、ビルドシステムがSassを起動し、ユーザーが独自のSass機能を定義し、 EyeglassがモジュールをSassに渡します。 既存のエコシステムがDart Sassと連携する範囲でAPIをサポートしたいと考えています。







  • 互換性のあるDartとRuby Sass。 Dart機能が意図的にRubyと異なる場合、特にRubyの動作がバグと見なされる場合がいくつかあります。 これに関する警告をRuby Sassに追加するか、最小限の変更でこれを行うことができる場合は、Rubyに新しい動作を導入する必要があります。


ブラウザーでSassをサポートし、Dart VMでSassにノードと互換性のあるラッパーを提供するなど、最終的に作成したいポイントがいくつかありますが、初期リリースはブロックされません。







未来の未来



今後数か月にわたって、Dart Sassの安定性と互換性を高めるとともに、LibSassにSass 3.5の機能を追加するために多くの作業を行う予定です。 2017年初頭には、Dart Sassの安定リリースとLibSass 3.5のバージョンがリリースされる可能性が高いと思います。 現時点では、主要な機能に関する見解を決定し、Sass 4.0とその完全に新しいモジュールシステムでの作業を開始します。







Dart Sassは大きな変化ではありませんが、同時に刺激的です。 これにより、ユーザーに新しい機能をより迅速に提供できるだけでなく、この機能をより高速に動作させることができます。 さらに、ユーザーはリファレンス実装を簡単にインストールして実行できます。 そして最後に、これにより、Sassを純粋なJavaScriptで実行する生産的な方法が初めて提供されます。 メリットは非常に大きく具体的であるため、コストに見合うだけの価値があると確信しています。
