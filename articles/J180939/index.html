PHPのデコレータ。 拡張機能の実装

最初の記事の調査によると、拡張機能の実装を確認することが決定されました。 この時点で、既存のIDEを満足させるために、構文が少し変更されました。

これはまだ別のhello-world-extension記事ではありません。 基本を理解したい人は 、 Habré自体とロシア語の RTFGの両方で簡単に多くの資料を見つけることができます。

前提条件、販売、および落とし穴に関する記事。 PHPはほとんどなく、ほとんどがCです。





背景



tl; drを読むことに興味がない場合は、すぐにImplementationにアクセスできます。



遠くから始めます
私はPythonが好きで、特に構文のいくつかの点が好きです。 そして、私は最近主にPHPで書いているので、作業ツールをより便利で機能的にしたいです。 PHPは最近のバージョンでは非常によく開発されていますが、同じデコレーターがまだありません。 そのため、すべてを自分の手で取り扱わなければなりません。 最初に、名前とパラメーターをコア関数に渡すという統一のアイデアがありました（ 私のアイデアの別の拡張で 、まだアイデアを形成している段階です。必要に応じて、その作成について書くことができます ）。



関数デコレータ（メソッドはデコレーションと同じであるため、以下では言及しません）は、後者の動作を変更し、追加のロジック層で呼び出しをラップすることができます。 宣言形式は、最終的に元の関数を置き換えるものを返す必要があるラッパーのリストを記述します。 Python構文では、 次のようになります 。 

@decomaker(argA, argB, ...) def func(arg1, arg2, ...): # ... # : func = decomaker(argA, argB, ...)(func)


PHPにはそのような可能性はありません。 最初に、この構文をそのまま使用し、変更せずに転送することにしました（呼び出されたときにデコレーターパラメーターを渡すことを除きます。詳細は以下を参照）。 最初の記事では、まさにこの構文について説明しています。 しかし、構文チェックIDEと2つのコメンテーターキャンプの1つは、私を驚かせました。 その結果、構文の移植性が向上します。 デコレータの説明は、1行のコメント＃で説明する必要があります。

  • ＃は部分的に廃止されているため、その使用は通常の//、/ *、/ **よりも顕著になります。
  • 単一行のコメントは崩壊せず、見失うのが難しくなります。
  • 突然、PHP 5.xでデコレータが表示され、奇妙な構文を持つこの拡張機能の必要性がなくなります。


記述形式を決定したら、デコレータ自体の実装方法を決定する必要があります。 デコレータ関数は、元のデコレータを置き換える関数を返す必要があります。 これは、匿名関数とクロージャーが便利な場所です。

いくつかのPHPコード 
 <?php function decor($func) { echo "Decorator!\n"; return function () use($func) { return call_user_func_array($func, func_get_args()); }; } function a($v) { var_dump($v); } $b = decor('a'); $b(42); /* : Decorator! int(42) */


PHPでは、もちろん、パラメータを渡す間接的な関数呼び出しは冗長であり、これを取り除くことはできません。



結果は、記事の冒頭にある画像の構文です。 

 <?php function decor($func) { return function(){} } # @decor function original() { // ... }


そして、 Zendのレクサーを書き換えずにこれを取得して、PHP自体を再構築する必要がないようにします（ 動作します-触れないでください ）。





実装



計画を満たすために、2つのオプションがあります。

  • PHP lexerに到達する前にソースコードを変更します。
  • 既製のオペコードを変更しますが、構文は既存のものと互換性があるはずです。


2番目のオプションは、あらゆる種類のオペコードキャッシュおよびオプティマイザーとの互換性に関して疑わしいものでした。 この場合、デコレータ構文の元のバージョン（＃コメントなし）は機能しません。

最初のオプションが選択されました。

Zendには、ソースコード収益の2つのソースがあります。

  • コード（evalのアナログ）を含む行： zend_eval_string [l] [_ ex] 、最終的にはすべてzend_compile_stringの呼び出しになります。
  • さまざまなソース（stdin、streamなど）から取得したファイル。すべてのオプションはzend_stream_type列挙にリストされています。 この場合、 zend_compile_fileに興味があります。


どちらの場合も、特定の実装を持つ関数へのポインターがあります。 zend_startupの初期化ポインターを調べると、標準実装を見つけることができます。

  • zend_compile_fileはcompile_fileに実装されています。
  • zend_compile_string-同様に明らかなcompile_string 。


どちらの関数も、 何らかの形式でソースコードを受け入れ、 _zend_op_array構造の形式でオペコードの配列を発行します。 残念ながら、実行されるタスクの類似性にもかかわらず、それらの実装は異なります。 したがって、両方に影響を与えます。



ZendおよびPHP拡張機能のこのような関数ポインターへの影響はストリームに置かれます。 たとえば、同じzend_compile_fileがZendAcceleratorとpharで置き換えられます。 これは、サードパーティの拡張機能をカウントしていません。



代替として、アナログを実装し、ポインターを置き換えて、元のままにするだけです。 すべてがいつものようです。

プリプロセッサのタスクは、デコレータの説明を見つけて、デコレータが影響する関数のコードを変更することです。 このためには、ソーステキストのトークンを使用するのが最適です。 車輪を再発明しないために、ネイティブのZendのlex_scan字句スキャナーが使用されました 。その例は、 token_get_all内で呼び出されるtoken_get_allおよびtokenizeの実装で独自の目的に使用できます。

  1. コードが機能するスキャナーの現在の環境を保存します。

    zend_lex_state original_lex_state;

    zend_save_lexical_state（＆original_lex_state TSRMLS_CC）;



  2. 解析用のソース行を準備します。

    zend_prepare_string_for_scanning（＆source_z、 "" TSRMLS_CC）


  3. レクサーの初期状態を設定します（すべてのオプションはここにあります ）。

    LANG_SCNG（yy_state）= yycST_IN_SCRIPTING;


    token_get_allとは異なり、PHPコードはすでに解析されているため、開始タグは必要ありません。 したがって、初期状態はyycINITIALではなく、yycST_IN_SCRIPTINGです。

  4. ループでは、ソース文字列のすべてのトークンを取得します。

    zval token_zv;

    int token_type;

    while（token_type = lex_scan（＆token_zv TSRMLS_CC））{

    // ...

    }



    token_type-トークンタイプ：

    • <256-1文字のトークンの文字コード。
    • > = 256は定数T_ *の値です。 token_typeのストリング記述は、PHP_FUNCTION（token_name）/ get_token_type_nameを介して取得できます。


    token_zvには、トークンの意味が含まれています。 ただし、代わりに、現在のトークンの最初のバイトのアドレスとその長さをそれぞれ格納するzend_lex_state構造体のyy_textフィールドとyy_lengフィールドを使用できます。 Zendと同様、これらのフィールドへのアクセスは、対応するマクロを介して実装されます。

    #define zendtext LANG_SCNG（yy_text）

    #define zendleng LANG_SCNG（yy_leng）



    ここで、char * zendtextとunsigned int zendlengを使用します。



    メモリリークを回避するには、token_zvの値が元のバッファからそのまま使用されたり、メモリが割り当てられたりすることを考慮する必要があります。 解放する必要があります。 興味のある方はlex_scan（）コードを見て、token_get_allから必要なロジックを取り出すことができます。

  5. コードが機能するスキャナーの環境を復元します。

    zend_restore_lexical_state（＆original_lex_state TSRMLS_CC）;




以上で、ソースコードの字句解析ができました。 しかし、解析の問題をさらに強調したいと思います。



PHPがエラーを解析する場合、ハンドラーはエラーまたは例外を生成し、テキスト内のファイル名と行番号は_zend_compiler_globals状態から取得されます。 たとえば、ファイル名は、compiled_filenameフィールドから取得されます。 zend_prepare_string_for_scanning（）が呼び出されたときに設定されます。 そして、これはzend_error内で使用されます （あらゆる種類のE_ *エラーを生成するために使用されます。E_PARSEを生成するためにこの拡張機能でも使用されます）。 ただし、zend_error（）のcompile_filenameは、Zendがコンパイル状態（zend_bool in_compilation、すべて同じ_zend_compiler_globals）の場合にのみ使用されます。 ソースを解析すると、それ自体はアクティブになりません。

解析する前に、「コンパイル」に切り替えます。

zend_bool original_in_compilation = CG（in_compilation）;

CG（in_compilation）= 1;



そして最後にすべてを返します：

CG（in_compilation）= original_in_compilation;



これで、正しいファイル名をzend_prepare_string_for_scanningに渡すと、発生する可能性のあるエラーがさらにわかりやすくなります。 現在のファイル名はzend_get_compiled_filename（）で取得できますが、これはNULLを返すことができ、そこからPHP（NULLがzend_prepare_string_for_scanningに渡される場合）はsegfaultに分類されます。

最後に小さなPHP



デコレータを処理するとき、装飾された関数のソースコードはわずかに変化します。関数の本体は匿名関数でラップされ、パラメータによって最も近いデコレータに渡されます。 つまり コード用 

 // comment @a(A) @b @c(C, D) /** * yet another comment */ function x(X) { Y }


前処理の結果、次のコードが取得されます。 

 // comment /** * yet another comment */ function x(X) { return call_user_func_array(a(b(c(function(X) { Y }, C, D)), A), func_get_args());}


A、C、D、Xとは、そのままコピーされる任意のコードを意味します。

これにより、次の結果が生じます。

  • デコレートする関数がデフォルト値を持つパラメーターで宣言されている場合、匿名関数ではすべてが同じになります。

    関数foo（$ a、$ b = 42、$ c =配列（100、500））

    {...

    =>

    関数foo（$ a、$ b = 42、$ c =配列（100、500））

    {return call_user_func_array（...（function（$ a、$ b = 42、$ c = array（100、500））{...



  • デコレータ名の説明にエラーがある場合、エラーを説明するときのコード行は、本文を開く関数「{」を含む行になります。 同様に、デコレーターパラメーターについては、関数 '}'が本文を閉じる行にあります。
  • デコレータパラメータは、装飾される関数の変数名に設定できます。 パラメータの読み取り/書き込み制御が判明します。
  • なぜなら func_get_args（）を使用してパラメーターを渡すと、装飾される関数への参照によるパラメーターの受け渡しは現在機能しません。
