子供時代はどこへ行くのですか？

どういうわけか、私の弟は、普通の人のように、家族と一緒に休暇に出かけていました。 出発の前夜に、彼は両親の家に外付けハードドライブを持って現れ、「彼をここに寝かせて」と言いました。 「これは何で、なぜ必要なのか」という質問（ミサイルガイダンスコードや未公開のWikileaks資料がある場合）に対して、彼はこれが彼の子供の写真のいくつかのバックアップコピーの1つであり、火災や空のアパートの強盗の場合、これらは彼は写真を失いたくない。 結局のところ、お金と装備は稼ぐ方法によって回復することができます。 子どもの失われた写真は、熱力学の卑劣な第二法則のために、いかなる方法でも元に戻すことはできません-私たちの世界の時間は取り返しのつかないほどに残されています。 これがバックアップの主な機能です-リーク時間を停止します。 これは、バックアップの値-凍結時間です。 しかし、ここで興味深い点が生じます。価値があるため、バックアップ自体にバックアップが必要です。 これがバックアップ複製シナリオです。 しかし、彼らが言うように、すべての卵を1つのバスケットに入れることは意味がありません。 したがって、バックアップは空間に分散する必要があります。 親に持って行ってもいいし、クラウドに持って行ってもいい。 私の兄弟は、オフサイトバックアップの最初の例を見せてくれました。



肉から肉、バックアップからバックアップ

したがって、データを本当に重視するのであれば、バックアップのバックアップコピーが必要です。 ただし、コピー数の増加に伴い、悪意のあるエンコーダーコピーペーストが遭遇する問題-調整の問題に直面しています。 たとえば、私は何らかの方法で写真のバックアップを備えたWestern Digitalの外付けドライブを使用しました（その後、WDに対する激しい不信があります）。 それ以来、写真を2つのディスクにコピーしていますが、これらのコピーを同期するための追加の負荷が発生します。 USBドライブを貼り付けることは、これからの人生でやりたいことではありません。 誰かが私のためにバックアップをバックアップしてくれるといいですね。

Acronis Backup＆Recovery製品では、これは製品自体であり、作成されたバックアップチェーンを追加のストレージに複製できます。 たとえば、これはD2D2T（ディスクからディスクへのテープ）またはD2D2C（ディスクからディスクへのクラウド）スクリプトの解決方法です。

Acronis True Imageでは、バックアップの冗長性はクラウドへのバックアップによって決定されます。 実際、Acronisクラウドには、複数のストレージノード間でデータをコピーするReedとSolomonの 2人が既にレプリケーションを行っているため、1つのノードに障害が発生してもデータは失われません。 クラウドの実装に関する詳細については、別の投稿で詳しく説明します。

技術的要因
前の要因がクラウドへのバックアップのビジネスロジックに影響を及ぼす可能性が高く、ユーザーが特定のデータセンターを選択できるかどうかは、次の5つの要因が関連するテクノロジに直接影響します。 この要因またはその要因に応じて、さまざまなボトルネックが発生します。これらの要因に対して、1つまたは別のテクノロジが戦う必要があります。これについては、要因を簡単に検討した後で検討します。



コピーするデータの量
ボリュームが大きいほど、どこでも問題が発生します。 コピープロセス自体は時間的に遅れており、再起動せずにデータ変更の頻度、データの一貫性、およびコンピューターランタイムの効果を高めます。 したがって、操作の自由のために、小さなボリュームで管理するよう努めなければなりません。 もちろん、これが可能でない限り。



データ頻度
データが変更されなかった場合、バックアップは特に必要ありませんでした（これは、変更のいずれかの形式のデータの破壊を検討する場合です）。 変更の頻度に応じて、疑問も発生します。ユーザーが各変更を追跡することは重要ですか、それとも一部の変更は無視できますか。 たとえば、DOCファイルを編集する場合、ユーザーがドキュメントのすべての保存されたバージョン（一定の高レベルのアンドゥスタックの一種）を追跡することは興味深いかもしれませんが、データベースファイルの各ブロックの変更を追跡することに関心を持つ人はほとんどいません。 データ変更の頻度は、その一貫性の維持に直接影響します。これは、スナップショットが作成されたビジーな実稼働システムでは特に痛く、I / O書き込み操作の重複につながります。



チャンネル幅
チャネルが広い場合（プロセッサバスや高速ローカルエリアネットワークなど）、ディスク自体からの読み取りは薄いスポットです。 最適化は、シングルパスディスクコピーまたはイメージングによって達成されるI / Oディスク操作の最小化で構成されます。 ただし、ボトルネックが通信チャネルである場合、I / Oディスク操作が重要な役割を果たしなくなり、他の要因が前面に出てきます。



再起動なしのランタイム
大量のデータと遅いチャネルでは、バックアップにかなり時間がかかる場合があります。 真のBigDataを扱うある知り合いは、カセットの四半期の分析データのバックアップには1か月半かかったと言いました。 クラウドにダウンロードする場合、12時間のプロセスは決して珍しいことではなく、この期間中のホームユーザーはコンピューターの電源を切ることをお勧めします。 したがって、特別な措置を講じなければ、データの全量をコピーするプロセスは不可能になります。



データの一貫性
システムとアプリケーションをバックアップするとき、このシステムまたはアプリケーションを構成するファイルの一貫性は特に重要です。 コピープロセスが1時間続くとしましょう。 単純な列挙でファイルをコピーすると、最初のファイルが1つの状態になり、最後のファイルが1時間後に状態になるという状況になります。 そして、この時間内に最後のファイルが変更された場合（これは、ロードされたシステムでは非常に正常です）、バックアップには、自然界には存在しなかったため、動作する必要はありません。 たとえば、レジ​​ストリには、存在しないファイルや分散システムの他の恐怖へのリンクが含まれている場合があります。 ただし、データの一貫性が重要でない場合のバックアップシナリオがあります。 たとえば、各写真には自給自足の価値があり、隣接する写真の状態に依存しません。



クラウドシナリオ

使用されるテクノロジーとシナリオに影響する要因を検討したので、それらがどのように結合し、何に変換されるかを考えてみましょう。



同期する
データの一貫性が重要でない場合、それを保護するための最も便利なソリューションは同期です。 高レベルの同期スキームは次のとおりです。
  1. 同期するすべてのノードにエージェントがインストールされます。
  2. エージェントには、追跡用の特定の領域（フォルダー、フォルダーのセット、ボリューム全体）が割り当てられます。
  3. エージェントは変更を監視し、利用可能な場合はすぐにクラウドにアップロードするか、接続が表示されるまで延期します
  4. エージェントはクラウド内のイベントにサブスクライブし、ローカルでイベントを再生します
  5. バックアップによるファイルのバージョン管理は、クラウド自体で行われます。




同期は、主にドキュメントや写真などの独立したデータを扱うため、ほとんどのホームユーザーに適しています。 これらのファイルに加えて他のすべてのファイルも大きくないため、クラウドへのダウンロードに成功したり、コンピューターとのユーザーセッションのためにそこからポンプでくみ出されたりします。 そのため、ユーザーセグメント内の個々のファイルを保護する市場は、徐々に同期に移行しています。



システムのバックアップ
システムのセットアップ、お気に入りのアプリケーションのインストールに1時間以上費やしたユーザーは、時は金なりだと理解しています。 原則として、これらすべてのアプリケーションが配布ディスクから回復可能であっても、構成自体に費やされた時間を元に戻すことはできません。 したがって、システムを保護する必要があります-あなたの努力のバックアップを作成します。 この観点では、システム構成はドキュメントの作成に例えられます。 より複雑で大きくなります。 このシナリオは、回復時間が経済的損失に直接影響する事業セグメントで特に鮮明です。 そのため、障害発生後の復旧シナリオが得られます。これは、システムイメージを正常な状態にすばやく戻す機能です。 このシナリオでは、データの一貫性が重要な役割を果たし、ボリュームは同期の場合よりも大きくなります。 たとえば、ローカルディスクへのバックアップのようにチャネルが広い場合、スナップショットからのイメージのセクターごとの削除が最も簡単で安価なソリューションになります。 ただし、クラウドへのチャネルは広くなく、信頼できません。



このようなチャネルでは、数百ギガバイトを埋めるのに数時間、さらには数日かかります。 大容量のボリュームでは、増分コピーを使用できますが、プライマリの大容量ボリュームからは逃れられません。 それでも企業クライアントがこのような時間に耐えられる場合、ホームユーザーは最初の日にコンピューターの電源をオフにします。 忍耐強い企業のクライアントも、I / O操作を2倍にするシステムのアクティブスナップショット（ロシア語に「スナップショット」という言葉を持ち込んでください！） この問題には2つの解決策があります。

一次ダウンロード
このソリューションの本質は、この主要な量のデータを、低速で信頼性の低いHTTPチャネルではなく、実際の物理メール（ロシアの郵便でない場合に信頼性の高い）などの高速（大容量の）信頼性の高いチャネルを介してデータセンターに転送することです。 つまり、ユーザーは自分のバックアップを実際のディスクにコピーし、DHLなどのメールでデータセンターに送信します。 そこで展開し、その後、ユーザーは通常のWANを使用して、データの小さな変更をクラウドに段階的にコピーします。 物理チャネルが非常に高速であるという事実は、KamAZにハードドライブをロードし、60 km / hに分散させることでチャネルのクレイジーな帯域幅を示すよく知られたジョークによって確認されます。 ちなみに、私の弟が両親の家にディスクを持ってきたとき、彼は本質的にこの最初の種まきをしました。 ディスクの送信と追跡は非常にエネルギーを消費するタスクであり、多くのホームユーザー（Acronisには500万人以上）がいるため、コンシューマセグメントのこのソリューションは、純粋に組織的にはAcronisに適合しませんでした。 企業セグメントでは、それが正常に適用された場所に適しています。





再開
再開は、バックアップを半分に中断し、深刻な追加オーバーヘッドなしで後で続行する機会です。 これは、セクターごとのコピーが失敗する場所です。 結局、半分ロードされたディスクは半分妊娠とほとんど同じです。 ディスクは完全にダウンロードされるか、まったくダウンロードされません。 コピーされたディスクの一貫した状態に使用されるスナップショットはコンピューターの再起動に耐えられないため、マシンイメージをベアアイロンに転がすことなく、よりきめ細かなデータコピーを行うメカニズムが必要です。



ハイブリッド画像キャプチャ

True Image 2014では、Acronis（そして世界でも！）で初めてハイブリッドイメージングが実装されました。 アルゴリズムは次のとおりです。
  1. ファイルシステムのスナップショットを作成します
  2. イメージキャプチャテクノロジは、空のボリュームに基本的なブート情報をコピーします（このボリュームは200 KBを超えません）
  3. イメージからのファイルは、クラウドに増分的にコピーされます。


接続が途中で中断された場合、バックアップは不完全であると見なされますが、クラウドにすでにアップロードされているファイルは、ダウンロードおよびリカバリに使用できます。 次のレジュームで、ファイルが変更されていない場合（およびハッシュはそれについて教えてくれます）、それらはアップロードされず、増分バックアップを作成するときに考慮されます。 履歴書が実装されました！



ところで、ブータブル情報を含むボリュームのイメージがあるという事実により、個々のファイルを復元することに加えて、ブータブルメディアからベアメタル上のバックアップコピーをロールすることができます。 彼のラインでアクロニス！
