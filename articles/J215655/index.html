LSMW（Legacy System Migration Workbench）を使用して履歴データをSAPにアップロードします

SAP ERPシステムを実装する場合、最も重要な問題の1つは、履歴システム（SAPの導入前に企業で稼働していたシステム）のデータを新しいERPにロードする問題です。 これを行うために、SAP内に強力なツールがあります-履歴システム（LSMW）からのデータ転送システム。 8年以上にわたってSAPで仕事をしてきましたが、このシステムを使ってかなりの経験を積んでおり、LSMWにはわかりやすいマニュアル（特にロシア語）がなく、ショップの同僚が常にアドバイスを求め、ロシアのSAPフォーラムで私を見つけていますこの記事は、インターンとこれまでLSWMの経験がない経験豊富なコンサルタントの両方にとって非常に有益であると信じています。



LSMWトランザクションを開始しました。見よ、すでに歓迎されています。LegacySystem Migration Workbenchへようこそ！



初期画面では、次の画像が表示されます







まず最初に、プロジェクト（プロジェクト）を作成する必要があります。 プロジェクトは、たとえば、トップレベルの「プロジェクト」（たとえば、メンテナンスグループ全体（メンテナンスおよび機器の修理）で作業するための名前「PM」）、またはよりローカライズされたプロジェクト、たとえば、テクニカルオブジェクトをシステムにロードするプロジェクト（呼び出します）彼の「Tech Objects」）。 本質を生み出さないこと、おじさんの戒めを覚えて、同じ上位レベルを始めるプロジェクトとしてお勧めします。 すべてのプロジェクトチームにとって非常に便利です。LSMWでは、モジュールをプロジェクトとして見ることができます。



Projectフィールドのカーソルになり、白いシートを押します。 プロジェクトを作成します。 私の例の場合、「PM」になります。



次に、サブプロジェクトを作成します。 これもプロジェクトの構造化にすぎないため、プロジェクトを選択する決定に応じて、サブプロジェクトを作成します。 私の例の場合、サブプロジェクトは単なる「Tech Objects」です。



さて、オブジェクト（Object）。 ここでは、LSMWを使用してシステムにロード/変更する内容を詳細に説明しています。 たとえば、Equipment Unitデータをダウンロードする必要があります。 この場合、「PoEアップロード」オブジェクトを作成します







プロジェクト、サブプロジェクト、オブジェクトを作成し、開始します。



20ステップの画面が表示されます

（メニュー項目[編集]で、[ナンバリングオン]を有効にして、各ステップの反対側にシリアル番号が表示されるようにします）







1）ステップ1-オブジェクトの属性を維持する



このステップでは、データのロード方法を示す必要があります。 私が最も頻繁に使用し、原則として一般的な方法の2つに焦点を当てます。



A）標準オブジェクト（標準バッチ/直接入力）



LSMWは事前定義済みオブジェクトのリストを提供しますが、その中に必要なものが最も多く含まれている可能性があります。



私の場合、これは機器です







「方法」フィールドで、0001-PakVvの作成（つまり、「バッチ入力」）を選択します。







保存して終了します。



B）バッチインプット記録



標準オブジェクトが十分にない場合、またはニーズに合った標準オブジェクトがない場合は、原則として、この場合、「マクロ」を作成する必要があります（私はそれを呼びます）。 つまり、ダウンロードする必要があるデータをトランスクリプションと呼び、アクションのシーケンスを記録してから、次のステップで記録された「マクロ」と一致させて、ダウンロードテンプレートファイルの構造を作成します。



トピックがリーダーを見つけた場合に、別のトピック（これは大きな別のトピックです）に時間があれば、バッチ入力記録メソッドを記述します。



2）ステップ2-ソース構造を維持する



このステップでは、ロードされたデータの構造を決定する必要があります。 設備ユニットの場合、構造は単一レベル（「フラット」）であり、下位のサブ構造はありません。 後者は、たとえば、メンテナンスオーダーを読み込む場合に、1つのオーダーヘッダーに対してN個の操作があり、それぞれがM個のコンポーネントを持つことができる場合に役立ちます。 また、機器には、ネストなしで番号とデータセットがあります。



構造が平坦な場合、この手順は形式的です。 「Main」などの構造を作成します。 保存して終了します。







3）ステップ3-ソースフィールドを維持する



ステップ3では、ロードする予定のフィールドを決定し、メイン構造にカーソルを置き、白いシートアイコンをクリックして、フィールドを作成します。



フィールドは、オブジェクトにロードするフィールドです。 私の場合、たとえば、機器ユニットにロードする必要があります。



-番号

-SWの名前

-オブジェクトのタイプ



これはもちろん一例です。 任意のフィールドを出荷できます。 簡潔にするために、これら3つだけに制限します。



将来、ステップ5で、独自のフィールド名とシステムフィールドを比較することに煩わされることはありません。明確にするために、システムで呼び出されるフィールドに名前を付けることをお勧めします。 このフィールドまたはそのフィールドの名前、タイプ、ビット深度を確認するには、このフィールドにカーソルを合わせて「F1」を押します。







表示されるウィンドウで、エントリ「フィールド名」をダブルクリック（ドリルダウン）します







興味のあるフィールド（この場合、「装置のユニット番号」はEQNRフィールドです）、そのタイプ（CHAR）、および長さ（18文字）を見つけます。 そして、それに応じて、このデータを3番目のステップで作成されたフィールドに書き込みます









ロード可能なフィールド、保存、終了として興味深いセットを作成しました。



4）ステップ4-構造関係の維持



このステップでは、構造を標準構造に接続する必要があります。 私のシンプルでフラットな構造の場合、すべてが非常にシンプルです。カーソルをIBIP（機器のユニット）に置き、白いシートを押します。 唯一のメイン構造は、自動的に標準EO構造に結び付けられます。 保存して終了します。







5）ステップ5-フィールドマッピングと変換ルールを維持する



ここで、システムに認識されている対応するSWフィールドにロードされることが予想される作成済みフィールドを関連付ける必要があります。







これは、ステップ3でシステムがフィールドを認識するのと同じ方法でフィールドを呼び出した場合に便利です。 これで、[エクストラ]メニュー項目で[自動フィールドマッピング]をクリックすると、システムが自動的にフィールドをマッピングします。 さらに表示されるすべてのウィンドウで、「OK」をクリックする必要があります。作成したフィールドの名前にエラーがない場合は、システムがすべて自動的に作成します。







すべてのフィールドがマップされたら、保存して終了します。

7）ステップ7-ファイルを指定する

このステップでは、ダウンロードするデータを含むテンプレートファイルを指定する必要があります。



次の2つのオプションがあります。



a）PC上のレガシーデータ（フロントエンド）

b）R / 3サーバー（アプリケーションサーバー）のレガシーデータ







オプション「a」は、コンピューターに直接配置されているテンプレートファイルの選択であり、「b」はSAPサーバー上にあります。 どういうわけか私は2番目のオプションに出くわしませんでした。 便宜上、ファイルシステムに直接配置されたファイルを常に使用しています。 Legacy Dataに配置します（赤い領域で、「Create」リーフレットをクリックします）



実際、ハードドライブにテキストファイルを作成します。 私たちは3つの分野で失礼なことを思い出します：



SW番号

SWの名前

SWのタイプ



テンプレートファイルでは、TABを通る最初の行で、明確にするためにシステムフィールド名を指定できます。 2行目以降にも、TABを使用して、ロードする値を書き込みます







表示される画面で、コンピューターのドライブ上のテンプレートファイルの場所を選択し、デリミターサブ画面（区切り記号）の[タブ]ラジオボタンを設定し、[ファイルの先頭のフィールド名]ボックスをオンにします（ファイルの最初の行がフィールドの技術名であることを示します） 。

テンプレートファイルのフィールドのシーケンスが、ステップ3-ソースフィールドのメンテナンスの対応するシーケンスに対応していない場合は、「フィールドの順序がソース構造の定義に一致する」のチェックを外す必要があります。 画面上の残りの部分には触れず、閉じて終了します。
