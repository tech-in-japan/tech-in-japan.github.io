クラウド内のSQL Server Reporting Services

PaaSモデルとしてのWindows Azureクラウドプラットフォームには、 Windows Azure SQLデータベースサービス（SQL Azureと呼ばれる）だけでなく、Windows Azure SQLレポートレポートサービスも含まれています。 ご存知のように、クラウドアプローチの利点の1つは（クラウドサービスプロバイダーに関係なく）弾力性と有料です。 必要に応じてリソースを引き付け、実際に消費されたリソースに対する支払い。これにより、レポート期間が閉じられたときにのみアイドル状態でロードされる洗練された高価なサーバーを組織が購入する必要がなくなります。 重量レポートは、クラウドへの移行に適した候補です。 クラウドレポートの技術的な利点は、オンプレミステクノロジーとの相互運用性です。 SQL Server Reporting Servicesに詳しい開発者にとって、クラウドのレポートプロセスは従来のレポートと変わりません。

前の記事で作成されたレポートは非​​常に簡単ですが、その例では、クラウドへのレポートの移行プロセスの概要を示すことができます。 まず、クラウドレポートサーバーを作成します。 古いスタイルのWindows Azure管理ポータルに移動します。 2012年6月上旬にプレビューとして発表されたものでは、レポートもデータ同期もまだ実装されていません。 新しいポータルを試すという申し出に急いでいる場合は、上部の緑色の[プレビュー]ボタンをクリックして、[以前のポータルに移動する]を選択します。 左下の[レポート]をクリックします。





図1



現在のサブスクリプションにはクラウドレポートサーバーがないため、新しいSQL Azureレポートサーバーを作成することを提案します。





図2



レポートサービスは、既存のすべてのMicrosoftクラウドデータセンターで利用できます。 レポートの基になるデータを保存するクラウドベースのSQL Serverもあり、データセンター内のトラフィックは無料なので、西ヨーロッパセンター（アムステルダム）で作成します。 SQL Serverを作成する場合と同様に、名前は自動的に生成され、変更できません。





図3



サーバー管理者の名前とパスワードを設定するためにも提供されます。





図4



その後、サーバーはレポートを受け入れて実行する準備が整います。





図5



クラウドレポートは、ローカルツールと同じツール（ SSDTの一部としてのレポートデザイナー、 レポートビルダー ）によって作成されます。 主な機能は、そのデータソースがWindows Azure SQLデータベース（SQL Azure）であり、ローカルレポートが組み込みの構成可能なデータソースをサポートしていることです。





図6



ご覧のとおり、SQL Azureがそれらの間に存在します。これにより、クラウドからのデータを使用して、ローカルレポートサーバーでレポートを実行する場合のハイブリッドシナリオ、または必要に応じて中間シナリオを実装できます。

クラウドとオンプレミスのSQL Serverの機能には大きな共通点がありますが、データがSQL Azureから取得される場合、ソースはSQL ServerではなくSQL Azureである必要があります。 そうしないと、デプロイメントでエラーが発生します

レポート処理中にエラーが発生しました。 トレースIDはf8806086-edd3-4a68-89b6-26bd33504f82です。 （rsProcessingAborted）

共有データセット 'DataSet1'の実行に失敗しました。 （rsDataSetExecutionError）

このレポートサーバーに登録されていないか、このエディションのReporting Servicesでサポートされていないデータ拡張機能 'SQL'を使用しようとしました。 （rsDataExtensionNotFound）



まず、レポートが作成されるデータをローカルデータベースからSQL Azureに転送する必要があります。 これは、 bcp 、 SSIS 、 DACPAC 、 BACPAC 、 SQL Azure Data Sync 、およびそれらの組み合わせの5つを分析した多くの方法を使用して実行できます。

簡単にするため、ローカルデータベースをBACPACにエクスポートし、指定されたAzure SQLサーバーにすぐにインポートする、SQL Azureにデータベースをデプロイする方法を選択します。

図8



データがクラウドに移動した後、レポートのDataSource1接続をローカルのSQL Serverからクラウドに再実行する必要があります。





図9



これで、以前のローカルレポートと同様に、ローカルレポートがクラウドデータで実行されます。 サーバーへの接続が正常に確立されたというエラーがポップアップしたが、証明書のCN名が渡された値と一致しない場合は、接続文字列を追加します図9; TrustServerCertificate = True。

ローカルのレポートサーバーからクラウドにレポートを移動します。 これを行う最も簡単な方法は、開発ツール（この場合はSSDT）から直接インストールすることです。 プロジェクトプロパティ（プロジェクト-> <プロジェクト名>プロパティ）で、ターゲットサーバーのURLとして、Azure管理ポータルで確認したクラウドレポートサーバーの名前を指定します（図5）。 通常、共有データソースとデータセットが配置されるフォルダーの名前を指定します。 OverwriteDatasetsおよびOverwriteDatasourcesプロパティをtrueに変更します。 それ以外の場合、同じ名前のデータセット/データソースがこのフォルダーに既に存在する場合、それは完了しません。

-展開の開始：プロジェクト：レポートProject1、構成：デバッグ-iijsstvk71.reporting.windows.net/ReportServerへの展開

データソース '/ Data Sources / DataSource1'を展開しています。

警告：データソースDataSource1は既に存在し、OverwriteDataSourcesが指定されていないため、サーバーに展開できません。

データセット '/ Datasets / DataSet1'を展開しています。

警告：データセットDataSet1は既に存在し、OverwriteDatasetsが指定されていないため、サーバーに展開できません。





図10



SSDTメニューバーで、[ビルド]項目に移動し、[<プロジェクト名>の展開]を選択します。





図11



Azureポータル->レポートに移動すると、図5と比較して、図10の順序でレポートサーバーにデータソース、データセット、およびレポートProject1フォルダーが形成されていることがわかります。





図12

図13



トップメニューの[アップロード]ボタンに注意してください。 レポートを「手動」で完了することができます。 SSDT / Report Builderに頼る必要はありません。 ポータルでデータソースを直接作成することもできます-[データソースの作成]メニューボタン。 レポートをクリックして、レポートサーバーでの認証用のログインとパスワードを入力します。





図14

図15

図16
