ファイナンスとプログラミング：概要



従来、金融業界では金融言語が需要があるため、さまざまなハードウェアと相互作用する高負荷のデータ処理および伝送システムを作成できます。



証券会社と取引所のインフラストラクチャの大部分は、 C ++言語を使用して作成されました。その助けにより、デリバティブの価格モデルを計算するライブラリからデータストリームを処理するモジュールまで、さまざまな製品が作成されます。 この言語を知っている専門家が金融の分野で非常に需要があることは驚くことではありません。



さらに、 C＃とJavaは非常に人気のあるプログラミング言語です。 これらの言語は、インフラストラクチャの問題を直接解決するためにあまり使用されませんが、市場では、これらの言語でモジュールが実装された取引ロボットを見つけることができます。 さらに、金融会社はそれらを使用して、さまざまな最前線のサービスを実装できます。 C＃とJavaのスペシャリストもそのような企業に需要があります。



Python、MATLAB、Rなどのスクリプト言語は別のレイヤーを表し、取引戦略やプロトタイプの量子モデルを記述するためによく使用されます。 トレーダーは多くの場合、プロトタイプ用のコードをスクリプト言語で記述し、後でC ++などの生産性の高い言語を使用して記述します。



ただし、場合によっては、機械取引システムをスクリプト言語で完全に実装できます。 TradeScript言語などのこれらのツールの一部はトレーディングターミナルにも組み込まれているため、ユーザーは相場やチャートの分析から気を散らすことなく、トレーディング戦略をすばやく「スケッチ」できます。



以下は、放物線SARインジケーターと指数移動平均（EMA）を使用して信号を生成する取引戦略を説明するTradeScriptコードです。 この場合、20周期と60周期のEMAを使用してポジションを入力し、放物線システムを使用してポジションを終了します。



TradeScriptの簡単な取引システムコード 
Buy Signals # 20- EMA  - 60- EMA CROSSOVER(EMA(CLOSE, 20), EMA(CLOSE, 60)) Sell Signals # 20- EMA    60- EMA CROSSOVER(EMA(CLOSE, 60), EMA(CLOSE, 20)) Exit Long #    - Parabolic SAR CROSSOVER(CLOSE, PSAR(CLOSE, 0.02, 0.2)) Exit Short #      Parabolic SAR CROSSOVER(PSAR(CLOSE, 0.02, 0.2), CLOSE)


トレーディングロボットの作成



近年、金融業界で最もダイナミックに発展している分野の1つは、アルゴリズム取引と取引ロボットの作成です。 この場合にどのプログラミング言語が使用され、なぜ使用されるのかについて話しましょう。



まず、ソフトウェア製品の場合と同様に、取引ロボットの開発者は何らかの形で、市場の成功に影響を与える主な要因を考慮に入れることができるさまざまな妥協をしなければなりません。



  • 取引速度、すなわち 取引操作自体の実行-アプリケーションの速度、取引所からの応答の受信速度、ロボット自体の処理速度
  • 新しい市場、金融商品、アルゴリズムオプションに対応するロボットの汎用性とカスタマイズ性
  • 開発速度と大幅な変更。


したがって、速度が重要な場合（たとえば、HFT取引の場合）、効果的な低レベル言語（C ++および純粋なC）が使用されます。



多数のトランザクションを伴わない戦略に従って機能するトレーディングシステムをプログラムする場合、一見すると作業速度ではなく、開発自体の利便性と速度です。この場合、たとえばプラットフォームで最速の言語は使用されません.NET（同じC＃またはVisal Basic）。



もう1つのオプションは、重要な計算が必要な状況ですが、必ずしもリアルタイムではありません。 このような場合、解釈されたPython、R、またはTradeScript言語の使用が保証される場合があります。



言い換えれば、対応するプログラミング言語は、取引ロボットのさまざまなモジュールに使用されます。



  • 取引エンジンは、C、C ++、およびさまざまなレベルのアセンブラーで作成されるルールとして、単純な一連の操作を実行するシンプルで最速のロボットです。
  • パラメーター管理システム-このようなモジュールは、アルゴリズムの操作を制御する機能とユーザーインターフェイスを組み合わせたもので、取引の結果を表示するための制御パラメーターとメカニズムを含む場合があります。 C ++、C＃、Javaなどをここで使用できます。
  • バックテストと取引パラメータの選択のシステム-モジュールは履歴データで新しいアルゴリズムをテストし、現在のアルゴリズムの新しいパラメータを選択するために使用されます。 このモジュールを実装するには、Pythonおよびその他のスクリプト言語を使用できます。 （私たちのブログでは、Pythonでイベント指向のバックテストシステムを作成するストーリーを公開しました）。

StockSharpプラットフォームの創設者であるミハイル・スホーフは、さまざまなプログラミング言語のアプリケーション分野について次のように説明しています。



取引の言語は、2つのサブグループに分けることができます。



1）プラットフォーム言語。 R、Matlabなど。 言語の品質はプラットフォーム自体と密接に関連しているため、言語を個別に解析してアナログと比較することはあまり意味がありません。 一部のプラットフォームが最適であるが、言語が非常に不便な場合、すべての欠点に関係なく使用する必要があります。 また、逆に、不便なプラットフォームで完全に便利な言語の場合、この言語はまだ開発用に選択されません。



2）プラットフォームに依存しない言語。 C＃、Java、C ++、Pythonなど...



a）JavaとC＃（= .NET）はほぼ同一のプラットフォームです。 Javaは主にnixライクなシステムで使用されます。 WindowsのC＃。 他のシステムのトレーダーよりもWindowsのトレーダーの方が多いため、 StockSharpプラットフォームの基礎としてC＃を選択しました（モバイルプラットフォームは基本的ではなく補完的であるため、Androidはカウントしません）。



b）C ++に関しては、その複雑さのため、トレーダーの間では実際には開発されていません。 しかし、それはプロのプログラマーによって使用され、重い負荷と迅速な応答を備えたシステムを作成します。



c）Python（List、Haskellなどの言語はこのカテゴリに追加できます）は、アルゴリズムの問​​題を解決するためのシンプルなツールです。 トレーダーはより頻繁にこの言語を選択します。 しかし、そのような言語は、オペレーティングシステムとの貧弱な統合、貧弱なパフォーマンス、および優れたグラフィックスライブラリの欠如により、プラットフォームの構築に制限されています。



一番下の行では、あらゆるタスクやユーザーのカテゴリーに普遍的に便利な言語は存在しないと言えます。 したがって、言語を選択する前に、1つまたは別のツールの利点ではなく、独自のタスクと能力を理解することが重要です。


おわりに



金融市場で動作するトレーディングロボットまたはその他のシステムを作成するためのプログラミング言語を選択するのは簡単な作業ではありません。 適切な開発ツールを選択するには、将来のシステム全体のアーキテクチャを徹底的に分析する必要があります。 パフォーマンス、開発の容易さ、テスト、関連ライブラリのソースコードの可用性と安定性など、さまざまな要因を考慮することが重要です。



トレーディングシステムの性質は、彼女の人生のさまざまな段階でさまざまなテクノロジーを使用するのに役立ちます。長年にわたって利益をもたらすアルゴリズムをプログラムすることはできません。 このプロジェクトは絶えず開発および近代化され、その生涯を通じて特定の問題を解決するためにさまざまなプログラミング言語が適している場合があります。



その結果、C ++またはCUDAを使用してプロセッサをすばやく操作し、Javaを使用してデータを処理し、Pythonを使用して履歴データをテストし、Excel / VBAを使用してレポートを作成するシステムがあります。
