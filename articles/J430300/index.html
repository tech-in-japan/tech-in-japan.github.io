Goキットを使用したGoのマイクロサービス：はじめに

この記事では、Goでマイクロサービスを作成するためのツールとライブラリのセットであるGoキットの使用について説明します。 この記事はGoキットの紹介です。 私のブログの最初の部分である例のソースコードは、 ここから入手できます 。







Goは、最近の分散システムの開発にますます選ばれています。 クラウドベースの分散システムを開発している場合、サービスのさまざまな特定の機能をサポートする必要がある場合があります。さまざまなトランスポートプロトコル（ transl。HTTP、gRPCなど ）およびそれらのメッセージエンコーディング形式、RPC信頼性、ロギング、トレース、メトリックとプロファイリング、リクエストの中断、リクエストの数の制限、インフラストラクチャへの統合、さらにはアーキテクチャの説明も含まれます。 Goは、そのシンプルさと「魔法のない」アプローチのために人気のある言語です。したがって、Goのパッケージ（標準ライブラリなど）は、多くの「内部の魔法」を備えた本格的なフレームワークを使用するよりも、分散システムの開発にすでに適しています。 個人的に、私[ 約。 あたり Shiju Varghese ]私は本格的なフレームワークの使用をサポートしていません。開発者により多くの自由を与えるライブラリを使用することを好みます。 GoキットはGoエコシステムのギャップを埋め、マイクロサービスの作成時にライブラリとパッケージのセットを使用できるようにしました。これにより、分散システムで個々のサービスを設計するための優れた原則を使用できます。













Goキットの概要



Goキットは、信頼できるサポートされたマイクロサービスを簡単に構築できるGoパッケージのセットです。 Goキットは、製品でマイクロサービスを実行するために必要なロギング、メトリック、トレース、制限、割り込み要求などのレイヤーを使用して、透過的で信頼できるアプリケーションアーキテクチャのさまざまなコンポーネントを実装するためのライブラリを提供します。 Goキットは、さまざまなインフラストラクチャ、メッセージエンコード形式、さまざまなトランスポートレイヤーと対話するためのツールが実装されているため、優れています。







ワールドサービスを開発するためのライブラリセットに加えて、サービスのアーキテクチャを設計するための優れた原則の使用を提供し、奨励しています。 Goキットは、 Alistair Cockburnが提案するSOLID原則、サブジェクト指向アプローチ（DDD）、 六角形アーキテクチャ 、またはJeffrey Palermoが 「 タマネギ アーキテクチャ 」、 Robert C. Martinが 「 クリーンアーキテクチャ 」として知られるその他のアーキテクチャアプローチを順守するのに役立ちます。 Goキットは、マイクロサービスを開発するための一連のパッケージとして設計されましたが、エレガントなモノリスの開発にも適しています。







Architecture Goキット



Goキットを使用して開発されたアプリケーションのアーキテクチャの3つの主要なレベルは次のとおりです。







  • 輸送レベル
  • エンドポイントレベル
  • サービスレベル


輸送レベル



分散システム用のマイクロサービスを作成する場合、それらのサービスは多くの場合、HTTPやgRPCなどのさまざまなトランスポートプロトコルを使用して相互に通信するか、NATSなどのpub / subシステムを使用する必要があります。 Goキットのトランスポート層は、特定のトランスポートプロトコル（以下、トランスポート）に関連付けられています。 Goキットは、HTTP、gRPC、NATS、AMQP、Thirftなど、サービスのさまざまなトランスポートをサポートします（おおよそ、 プロトコル用に独自のトランスポートを開発することもできます ）。 したがって、Goキットを使用して記述されたサービスは、使用されるトランスポートについて何も知らない特定のビジネスロジックの実装に焦点を当てることがよくあり、同じサービスに対して異なるトランスポートを自由に使用できます。 たとえば、Goキットで記述された1つのサービスは、HTTPおよびgRPCを介して同時にアクセスできます。







エンドポイント



エンドポイントまたはエンドポイントは、サービスと顧客の基本的な構成要素です。 Goキットの主な通信パターンはRPCです。 エンドポイントは、別個のRPCメソッドとして提示されます。 Goキットの各サービスメソッドはエンドポイントに変換されるため、サーバーとクライアント間でRCPスタイルで通信できます。 各エンドポイントは、トランスポート層を使用してサービスメソッドを公開します。トランスポート層は、HTTPやgRPCなどのさまざまなトランスポートプロトコルを使用します。 個別のエンドポイントは、複数のトランスポート（ 異なるポートで約Per HTTPおよびgRPC ）を使用して、サービスの外部に同時に公開できます。







サービス



ビジネスロジックは、サービスレイヤーに実装されます。 Goキットで作成されたサービスは、インターフェースとして設計されています。 サービスレイヤーのビジネスロジックには、ビジネスロジックのメインコアが含まれます。これには、使用されているエンドポイントや、HTTPやgRPCなどの特定のトランスポートプロトコルについて、またはさまざまな種類のメッセージの要求と応答のエンコードまたはデコードについての知識は必要ありません。 これにより、Goキットを使用して記述されたサービスのクリーンなアーキテクチャを順守できます。 各サービスメソッドは、アダプターを使用してエンドポイントに変換され、特定のトランスポートを使用して外部に公開されます。 クリーンなアーキテクチャを使用することにより、複数のトランスポートを同時に使用して単一のメソッドを設定できます。

Goキットのエンドポイントは次のようになります。 







 type Endpoint func(ctx context.Context, request interface{}) (response interface{}, err error)


前述したように、エンドポイントは別のRPCメソッドを表します。 各サービスメソッドは、アダプターを使用してendpoint.Endpoint

 import ( "context" "github.com/go-kit/kit/endpoint" "github.com/shijuvar/gokit-examples/services/order" ) // Endpoints holds all Go kit endpoints for the Order service. type Endpoints struct { Create endpoint.Endpoint GetByID endpoint.Endpoint ChangeStatus endpoint.Endpoint } // MakeEndpoints initializes all Go kit endpoints for the Order service. func MakeEndpoints(s order.Service) Endpoints { return Endpoints{ Create: makeCreateEndpoint(s), GetByID: makeGetByIDEndpoint(s), ChangeStatus: makeChangeStatusEndpoint(s), } } func makeCreateEndpoint(s order.Service) endpoint.Endpoint { return func(ctx context.Context, request interface{}) (interface{}, error) { req := request.(CreateRequest) id, err := s.Create(ctx, req.Order) return CreateResponse{ID: id, Err: err}, nil } } func makeGetByIDEndpoint(s order.Service) endpoint.Endpoint { return func(ctx context.Context, request interface{}) (interface{}, error) { req := request.(GetByIDRequest) orderRes, err := s.GetByID(ctx, req.ID) return GetByIDResponse{Order: orderRes, Err: err}, nil } } func makeChangeStatusEndpoint(s order.Service) endpoint.Endpoint { return func(ctx context.Context, request interface{}) (interface{}, error) { req := request.(ChangeStatusRequest) err := s.ChangeStatus(ctx, req.ID, req.Status) return ChangeStatusResponse{Err: err}, nil } }

Goキットのミドルウェア

Spring Boot Microservices Syndromeに注意してください



Goキットと同様に、Spring BootはJavaの世界におけるマイクロサービスツールキットです。 ただし、Goキットとは異なり、Spring Bootは非常に成熟したフレームワークです。 また、多くのJava開発者は、Spring Bootを使用して、使用からの肯定的なフィードバックを伴うJavaスタックを使用してワールドサービスを作成します。 マイクロサービスの使用を誤って解釈する多くの開発チームがいます。それらはSpring BootとOSS Netflixを使用してのみ開発でき、分散システムを開発する際のパターンとしてマイクロサービスを認識しません。







したがって、Goキットや何らかのフレームワークなどのツールセットを使用すると、設計テンプレートとして開発をマイクロセキュリティに向けることができます。 マイクロサービスは、コマンドとシステムの両方のスケーリングの問題を多く解決しますが、マイクロサービスベースのシステムのデータはさまざまなデータベースに散在しているため、多くの問題も発生します。 それはすべて、サブジェクト領域の問題とシステムのコンテキストに依存します。 クールなのは、マイクロサービスを作成するためのツールとして設計されたGoキットが、システムの優れたアーキテクチャ設計で作成されたエレガントなモノリスの作成にも適していることです。







また、要求の中断や制限などのGoキットの機能は、Istioなどのサービスメッシュプラットフォームでも利用できます。 したがって、Istioのようなものを使用してマイクロセキュリティを実行する場合、Goキットの一部は必要ないかもしれませんが、サービスメッシュを使用してサービス間通信を作成するのに十分なチャンネル幅があるとは限りません1つのレベルと追加の複雑さ。
