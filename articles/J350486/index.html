本日、WebテクノロジーとJavaScriptの使用に関する一連の記事の第9部の翻訳を公開します。 この記事では、Webプッシュ通知について説明します。 つまり、それらの基礎となるメカニズム、および通知のサブスクリプションが実行される方法、通知の送受信プロセスがどのように配置されるかについて説明します。







多くの開発者が使用したい機能の1つですが、モバイルアプリケーションの世界では非常に一般的なプッシュ通知がかなり遅れてWebに届きました。



復習



ここで説明しているテクノロジーを使用すると、ユーザーは、新しい素材の出現、または興味のあるイベントの発生についてサブスクライバーに通知することを目的としたWebアプリケーションの定期的な通知をサブスクライブできます。 Webリソース自体の観点から、これは、プッシュ通知にサブスクライブするユーザーをこのリソースにアクセスするよう招待する理由と機会があることを意味します。



プッシュ通知を提供するメカニズムの1つは、 サービスワーカーです。 プッシュ通知を処理するサービスワーカーは、特定のサービスワーカーが担当する新しい通知がブラウザに到着したときにのみコードが実行されるため、システムリソースを経済的に消費します。



プッシュAPIと通知API



ここで「Webプッシュ通知」と呼ぶものは、実際には2つのテクノロジーで表されます。 これは、サーバーがService Workerにメッセージを送信するときに使用されるPush APIと、Service WorkerまたはWebアプリケーション自体のスクリプトがユーザーに通知を表示するときに使用されるNotifications APIです。



プッシュAPI



プッシュ通知メカニズムを実装するには、次の3つの手順を実行する必要があります。



  • ユーザーインターフェイスの準備。 このステップでは、ユーザーが通知をサブスクライブできるメカニズムを準備します。

  • プッシュメッセージを送信します。 このステップでは、Webアプリケーション開発者のサーバーで、対応するAPIにアクセスします。これにより、ユーザーのデバイスに通知が送信されます。

  • プッシュメッセージを受信します。 このステップでは、プッシュメッセージはブラウザに配信された後に処理されます。



これをすべて詳細に検討してください。



ブラウザの機能を確認する



まず、現在のブラウザがプッシュ通知をサポートしているかどうかを確認する必要があります。 これは、次のチェックを実行することで実行できます。



  • navigator



    オブジェクトでserviceWorker



    を確認します。
  • window



    オブジェクト内のPushManager



    の存在を確認します。


これは次のようなものです。 



 if (!('serviceWorker' in navigator)) { //    -. return; } if (!('PushManager' in window)) { //    push-. return; }


ユーザーのブラウザーがWebプッシュ通知の操作に必要なテクノロジーをサポートしていない場合、ユーザーにそれらのサブスクライブを申し出ても意味がありません。



サービスワーカー登録



ブラウザを確認した後、必要なものをサポートしていることが判明した場合は、Service Workerの登録に進むことができます。 これを行う方法についてはすでに説明しました。



ユーザー許可の取得



Service Workerの登録後、ユーザーを通知にサブスクライブするプロセスを開始できます。 これを行うには、ユーザーからプッシュメッセージを送信する許可を取得する必要があります。



許可APIは比較的単純です。 注意する必要があるのは、 このAPIの 2つのバージョンが使用されることだけです。



以前のバージョンでは、コールバック関数を受け入れていましたが、現在ではプロミスを返します。 現在のブラウザに実装されているAPIのバージョンを事前に知ることはできないため、これが問題の原因です。 したがって、これらのアプローチの両方をサポートする必要があります。



次のようになります。

プッシュメッセージを送信する



ユーザーにプッシュメッセージを送信する場合、送信するデータ、送信先、メッセージに関する追加情報について、プッシュサービスに（適切なAPIメソッドを呼び出して）通知する必要があります。 この呼び出しは通常、Webアプリケーションサーバーから行われます。



プッシュサービス



プッシュサービスは、プッシュ通知を送信する要求を受信し、それらをチェックして、適切なブラウザーに通知を配信するシステムです。



プッシュサービスは、Webアプリケーション開発者が管理していないサードパーティのサービスであることに注意してください。 サーバーは、APIを介してプッシュサービスとやり取りするサーバーです。 プッシュサービスの例はGoogle FCMです。



プッシュサービスは、多くの複雑なタスクを処理します。 たとえば、ブラウザがオフラインの場合、プッシュサービスはメッセージをキューに入れ、メッセージが送信されるまでメッセージが送信されるまで待機します。



各ブラウザーは任意のプッシュサービスを使用できますが、Webアプリケーション開発者はプッシュサービスの選択に影響しません。 ただし、すべてのプッシュサービスは同じAPIを使用しているため、これらのサービスの多様性によってプッシュ通知メカニズムの実装に問題が生じることはありません。 プッシュメッセージを送信する要求を処理するURLを取得するには、 PushSubscription



オブジェクトのendpoint



パラメーターの以前に保存された値を参照する必要があります。



プッシュサービスAPI



プッシュサービスAPIは、ユーザーにメッセージを送信するためのツールを提供します。 これは、 Web Push Protocolで表されます。これは、プッシュサービスを操作する手順を定義するIETF標準です。



プッシュメッセージで送信されるデータは暗号化する必要があります。 そのため、開発者はプッシュサービスがユーザーに送信されたデータを表示することを許可しません。 使用するプッシュサービスを決定するのはブラウザーであるため、これは重要です（プッシュサービスが十分に安全でない可能性があります）。



次のプロパティは、プッシュメッセージごとに設定されます。



  • TTL



    未配信のプッシュ通知が削除される前にキューで費やすことができる期間を決定します。

  • Priority



    -メッセージの優先度を設定します。これにより、ユーザーのデバイスのバッテリー電力を節約する必要がある場合に、プッシュサービスは優先度の高いメッセージのみを送信できます。

  • Topic



    -プッシュ通知にトピック名を割り当てます。これにより、同じトピックで配信を待機しているメッセージが置き換えられます。 その結果、ユーザーのデバイスがアクティブになるとすぐに、ユーザーは最新のメッセージを受け取ります。









Webアプリケーション開発者サーバー、プッシュサーバー、およびメッセージの到着先のブラウザー



ブラウザプッシュイベント



メッセージがプッシュサービスに送信されるとすぐに、次のいずれかのイベントが発生するまで待機状態になります。



  • デバイスはオンラインです。

  • TTLパラメーターで指定されたメッセージ保持時間が切れました。



プッシュサービスがメッセージを配信すると、ブラウザーはそれを受信し、解読し、以前に登録されたサービスワーカーでプッシュイベントをトリガーします。



最も興味深いのは、対応するWebページが開いていなくても、ブラウザーがサービスワーカーを呼び出すことができることです。 ここで次のことが起こります。



  • プッシュ通知がブラウザに届き、ブラウザはそれを解読します。

  • ブラウザはService Workerをアクティブにします。

  • プッシュイベントはサービスワーカーにディスパッチされます。



プッシュイベントハンドラーの構成に使用されるコードは、他のJavaScriptイベントハンドラーのコードのように見えます。 



 self.addEventListener('push', function(event) { if (event.data) {   console.log('This push event has data: ', event.data.text()); } else {   console.log('This push event has no data.'); } });


サービスワーカーの機能について説明する場合、サービスワーカーをアクティブにするタイミングと停止するタイミングを決定するのはブラウザーであるため、開発者はサービスワーカーの実行時間を最小限のレベルで制御できることに注意してください。

まとめ



プッシュ通知は、使用されるリソースとこれらのリソースのユーザーの両方に利益をもたらすことができます。 おそらく、Webリソースの所有者が覚えておくべき主なことは、プッシュ通知にはユーザーにとって本当に興味深く必要なものが含まれていることです。
