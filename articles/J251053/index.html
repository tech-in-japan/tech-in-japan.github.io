GDB-gdbserverバンドルを使用したLinuxでのリモートデバッグ

ご存じのとおり、デバッグプロセスはそのようなものであり、その重要性を過大評価することは困難です。 さらに、LEDの点滅やポートへのデバッグメッセージの出力などの方法の重要性を理解すると、ステップバイステップのデバッグに関してはこれまで何も発明されていないという意見が残ります。 ただし、Linuxで組み込みシステム（rasbery pi、virt2real、または産業用プロセッサモジュールなど）でプログラミングする場合、ステップバイステップのデバッグタスクはそれほど簡単ではありません。



Linuxでのこのタスクは、 GDBおよびgdbserverプログラムの標準バンドルによって解決されることを目的としています。 アイデアは、コンピューター（GDB用語ではホスト）でプログラムを作成し、それをコンパイルして、ターゲットデバイス（ target ）にアップロードすることです。 次に、デバッグされたファイルとgdbserverをターゲットデバイス（ target ）で実行し、 GDBホストで実行します。



このようなデバッグの相互作用スキームは、次のように簡単に表すことができます。







実行可能ファイルは、リモートでデバッグするプログラムです。 正常に動作させるには、プログラムインスタンスをターゲットデバイスだけでなく、デバッグが進行中のコンピューター（ホスト）にも配置する必要があります。

GDBは、デバッグプロセスを直接実行するプログラムです。 通常、これはGCCツールチェーンの一部であり、コンパイラーと同じ場所にあります。

gdbserverはGDBの相互部分であり、デバッグモードで実行可能ファイルを実行します。 gdbserverはリモート側（ターゲット）で実行されるため、クロスコンパイラを使用してターゲットデバイス用に構築する必要があります。 実際、この記事は主にソースからのgdbserverのアセンブリに焦点を当てています。



私が自由に使えるのは、 virt2realボードと、TI AM335xシリーズのプロセッサをベースにしたプロセッサモジュールです。 以下にvirt2realの例を使用した一連のアクションを示しますが、 AM335xチップで私が成功した（そして重要なこと-同様に）成功した​​ことはすべて同じです。



注：ホストにインストールされているオペレーティングシステムはUbuntu.12.04です。



準備する



ホームディレクトリにgdbフォルダを作成し、すべての操作を実行します。 内部で、 ダウンロードサブフォルダーを作成します。 



mkdir -p ~/gdb/downloads cd ~/gdb/downloads


必要なソースをダウンロードして展開します。 GDB自体とtermcapライブラリ （GDBが使用）が必要です。 



 wget ftp://ftp.gnu.org/gnu/termcap/termcap-1.3.1.tar.gz wget ftp://ftp.gnu.org/gnu/gdb/gdb-7.8.tar.gz tar xvzf termcap-1.3.1.tar.gz tar xvzf gdb-7.8.tar.gz


通常、ソースフォルダーに直接収集されますが、ソースをそのまま残すためにこれを行いません。 これは、ターゲットプラットフォームが1つではなく複数種類ある場合に便利です。 その後、同じダウンロードされたソースを複数回使用できます。



termcapライブラリを構築する



したがって、後でgdbserverをビルドするときに必要になるため、 termcapライブラリから始めます。 プログラムを収集するbuildsフォルダーを作成します。 内部で、 v2rフォルダーを作成し、 virt2realプラットフォームのすべての結果をそこに入れます。 まあ、すでにtermcapライブラリを構築するためのtermcapフォルダーがあります。 作成されたディレクトリに移動します。 



 mkdir -p ~/gdb/builds/v2r/termcap cd ~/gdb/builds/v2r/termcap


使用するコンパイラとranlibをシステムに示します。 私の場合、これは次のとおりです。 



 export CC=/opt/virt2real-sdk/codesourcery/arm-2013.05/bin/arm-none-linux-gnueabi-gcc export RANLIB=/opt/virt2real-sdk/codesourcery/arm-2013.05/bin/arm-none-linux-gnueabi-ranlib


次に構成します。 --hostおよび--prefixの2つのパラメーターを指定する必要があります。

パラメーターとして指定するもの
  • --hostで何を指定するかが重要かどうかはまだわかりません。 いいえという印象を受けました。 いずれにせよ、このパラメーターに正確に何を指定する必要があるかについての明確な説明が見つからなかったため、コンパイラーのプレフィックスを示しました。 つまり コンパイラはarm-none-linux-gnueabi-gccと呼ばれるため、-hostとしてarm-none-linux-gnueabiを指定しました。
  • --prefixパラメーターとして、作業（構成）の結果を保存するフォルダー、つまり 現在のフォルダ。 


 ../../../downloads/termcap-1.3.1/configure --host=arm-none-linux-gnueabi --prefix=~/gdb/builds/v2r/termcap


すべてが正しく、エラーなく実行されると、Makefileが〜/ gdb / builds / v2r / termcapフォルダーに作成されます。

Makefileを確認する
安全にプレイし、Makefileを開いて、変数CCとRANLIBが上で示したとおりであることを確認することをお勧めします。 実際、gdbserverを最初に構築しようとしたときに、すべてがうまくいくように見えたが、結果のファイルはターゲットボード上で実行されたくないため、この点で長い間立ち往生していました。 未知の理由で、Makefileで標準コンパイラが永続的に使用されていることが判明しました。 Makefile内のコンパイラーが同じでないことがわかった場合は、単純に手で置き換えることができます（私を助けてくれました）。 しかし、記事を書く前にすべての手順を繰り返したとき、そのような問題はありませんでした。


次に、ライブラリを収集します。 



 make make install




gdbserverの収集



サーバーを構築するフォルダーを作成し、そこに移動します。 



 mkdir -p ~/gdb/builds/v2r/gdbserver cd ~/gdb/builds/v2r/gdbserver


termcapライブラリの入手先を指定します。 



 export LDFLAGS="-static -L~/gdb/builds/v2r/termcap/lib" export CFLAGS="-g -O2 -I~/gdb/builds/v2r/termcap/include"


termcapと同様に設定します。 ここで、GDBではなくgdbserverを作成していることに注意することが重要です。したがって、/ gdb-7.8 / gdb / gdbserverフォルダーから構成ファイルを指定します。 



 ../../../downloads/gdb-7.8/gdb/gdbserver/configure --host=arm-none-linux-gnueabi --prefix=/home/den1s/gdb/builds/v2r/gdbserver --disable-werror # &&


ご注意
どのように絶対パスをどこに示しようとしなかったとしても、命令を自分で調整することなく使用できるように、私は成功しませんでした。 なぜなら 〜で始まるパスを--prefixとして指定しようとすると、configureファイルは誓います。 したがって、ここでユーザー名を入力する必要があります。


すべてが正しければ、Makefileが作成されます。 さらに標準： 



 make make install




お試しください



デバッグプロセスをテストするには、短いHello worldを作成し、ターゲットプラットフォーム用にコンパイルします。 



 mkdir -p ~/virt2real/hello_test/ cd ~/virt2real/hello_test/ /opt/virt2real-sdk/codesourcery/arm-2013.05/bin/arm-none-linux-gnueabi-g++ hello.cpp -O0 -g -o hello




ソースパッケージhello.cppをダウンロードします 。 



 #include <iostream> int main(int argc, char* argv[]) { std::cout << "Hello,\n" << std::endl; std::cout << "debugged\n" << std::endl; std::cout << "World!!!" << std::endl; return 0; }


hello実行可能ファイルとgdbserverをターゲットボードに入力します。

SCPを使用して/ usr /フォルダーにアップロードします。 



 scp ~/gdb/builds/v2r/gdbserver/bin/gdbserver root@192.168.3.1:/usr/ scp ~/virt2real/hello_test/hello root@192.168.3.1:/usr/


次に、ターミナルの2番目のインスタンスを実行し、sshを介してターゲットボードに接続し、 / usr /フォルダーに移動します。 



 ssh root@192.168.3.1 cd /usr/


ターゲットボード上でgdbserverを実行し、それと共にhello実行可能（デバッグ済み）ファイルを実行します。 次に、気に入ったポートでデビューセッションを開きます。 



 [root@virt2real usr]# ./gdbserver :123 hello Process hello created; pid = 5680; listening on port 123


ホストに戻り、 GDBを使用してデバッグされたhelloファイルを実行します 



 /opt/virt2real-sdk/codesourcery/arm-2013.05/bin/arm-none-linux-gnueabi-gdb hello

Sysrootインストール



GDBを正しく動作させるには、リモートOSライブラリ（ターゲット）から読み取ることができる、いわゆるデバッグシンボルが必要です。 それらの不在は、例えば、そのようなメッセージの理由です： 



 (gdb) next Cannot access memory at address 0x0 Cannot access memory at address 0x0 0x36d1f040 in ?? () Use the "info sharedlibrary" command to see the complete listing. Do you need "set solib-search-path" or "set sysroot"?


また、他の不快なデバッグ問題を引き起こす可能性があります。

ホスト上のGDBの問題を修正するには、次のコマンドを使用して、これらのまさにライブラリを取得する場所を指定する必要があります。 



 set sysroot [Directory]


ホストのどこかにターゲットLinuxのイメージがオーバーロードしている場合、ライブラリフォルダーへのパスを指定する必要があります。

または、これらのライブラリをホストにプリロードします。 



 mkdir -p ~/virt2real/sysroot cd ~/virt2real/sysroot scp -r root@192.168.3.1:/lib lib


プロジェクトのあるフォルダーに戻り、GDBを再度実行して、ライブラリーへのパスを示します。 



 cd ~/virt2real/hello_test/ arm-none-linux-gnueabi-gdb hello (gdb) set sysroot ~/virt2real/hello_test/


現在、次のようなメッセージ： 



 Reading symbols from ~/virt2real/sysroot/lib/libgcc_s.s...done.


現在使用されているライブラリのリストは、次のように表示できます。 



 (gdb) info sharedlibrary From To Syms Read Shared Object Library 0x36fd77a0 0x36ff24d0 Yes (*) ~/virt2real/sysroot/lib/ld-linux.so.3 No /usr/lib/libstdc++.so.6 0x36e70e00 0x36ed107c Yes (*) ~/virt2real/sysroot/lib/libm.so.6 0x36e51028 0x36e63154 Yes (*) ~/virt2real/sysroot/lib/libgcc_s.so.1 0x36d1cb00 0x36e11ae8 Yes (*) ~/virt2real/sysroot/lib/libc.so.6 (*): Shared library is missing debugging information.
