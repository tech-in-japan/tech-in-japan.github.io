開発環境を作成する



プラグインをビルドするには、最初に仮想開発環境を作成する必要があります。ディレクトリを作成し、アクティブにして、 Fuel plugin builderをインストールします 。 



$ mkdir plugins $ cd plugins $ virtualenv .venv $ source .venv/bin/activate $ pip install fuel-plugin-builder




秘Theは、この仮想開発環境で新しいプラグインを構築するためにサーバーでrootになる必要がないことです。



プラグインの作成



ここで、Fuelプラグインビルダーを使用し、次のコマンドでプラグインの作業を開始して、将来のプラグインの「スケルトン」を生成します。 



 $ fpb -create external_nfs




その結果、開発者が特定のフィールドの値を修正できる一連のテンプレートを取得できます。 Fuelプラグインビルダーの動作の詳細については、 Fuel Plugins wikiをご覧になることをお勧めします。



プラグインバックエンド



以下に示すように、metadata.yamlでプラグイン定義を定義します。 ここでは、HAおよびマルチノードモードのプラグインでFuel 6.0を使用します。

これを行う方法に関する最も詳細な情報は、 OpenStack Cloud Administrator Guideと呼ばれるドキュメントで入手できます。



重要！ Mirantis OpenStack 6.0以降で作成されたプラグインのみを検証できます。 検証とは何か、なぜ検証が必要なのかについては、この画期的な話の最初の部分で書きました。 検証に合格する方法-このテキストの最後に説明します。



そして今、件名にガイドのない人のためのいくつかの段落： 



 $ Plugin name name: external_nfs title: External NFS backend for Cinder $ Plugin version version: 1.0.0 $ Description description: External NFS backend for Cinder $ Required fuel version fuel_version: ['6.0'] $ The plugin is compatible with releases in the list releases: - os: ubuntu version: 2014.2-6.0 mode: ['ha', 'multinode'] deployment_scripts_path: deployment_scripts/ repository_path: repositories/ubuntu - os: centos version: 2014.2-6.0 mode: ['ha', 'multinode'] deployment_scripts_path: deployment_scripts/ repository_path: repositories/centos $ Version of plugin package package_version: '1.0.0'




シェルタスクの追加



NFS UbuntuとCentOSに必要なパッケージをダウンロードし、それぞれexternal_nfs / repositories / ubuntuとexternal_nfs / repositories / centosディレクトリに移動して、パッケージをインストールするタスクを追加します。

deployment_scripts / install_packages.shにパッケージを保存し、スクリプト「chmod + x install_packages.sh」の実行可能権限を追加します。 



 $ !/bin/bash OS_NAME='' if grep -i CentOS /etc/issue.net >/dev/null; then OS_NAME='centos'; elif grep -i Ubuntu /etc/issue.net >/dev/null; then OS_NAME='ubuntu'; fi function install_package { if [ $OS_NAME == 'ubuntu' ]; then apt-get install -y rpcbind nfs-common libevent-2.0 \ libgssglue1 libnfsidmap2 libtirpc1 elif [ $OS_NAME == 'centos' ]; then yum install -y rpcbind nfs-utils nfs-utils-lib libevent \ key-utils libtirpc libgssglue fi } install_package




次に、パッケージをインストールするシェルタスクを追加します。 Cinderノードへのデプロイ後に起動されるinstall_packages.shスクリプトを含むtasks.yamlファイルは、次のようになります。 



 - role: ['cinder'] stage: post_deployment type: shell parameters: cmd: ./install_packages.sh timeout: 180




フロントエンドプラグイン



次に、Fuelインターフェイスの[設定]タブにチェックボックスと特定の項目を追加します。 これを行うには、 次のパラメーターに従ってenvironment_config.yamlでUIを記述します 。



1. NFS共有ファイル

2. NFSマウントオプション

3. NFSスパースボリュームが使用されている



次の例に示すように、プラグインインターフェイスの説明を追加します。 



 attributes: endpoint: value: '' label: 'NFS endpoints' description: 'comma separated HOST:SHARE values' weight: 25 type: "text" mount_options: value: '' label: 'NFS mount options' description: 'optional NFS mount parameters' weight: 25 type: "text" nfs_sparsed_volumes: type: "checkbox" weight: 30 value: false label: "NFS sparsed volumes" description: ""




パペットタスクを追加する



Puppetマニフェストは、 OpenStack Cloud管理者ガイドで説明されているタスクを提示し、GitHubのdeployment_scripts / puppetディレクトリに送信します。



パペットマニフェストを適用した後、/ etc / nfssharesファイルのアクセス許可に注意し、次のコマンドを実行してCinderサービスを再起動します。



Puppetはexternal_nfs / deployment_scripts / puppet / modules / cindernfs / manifests / backend / nfs.ppを適用します。



次に、tasks.yamlにマニフェストを適用するタスクを追加します。 



 - role: ['cinder'] stage: post_deployment type: puppet parameters: puppet_manifest: puppet/site.pp puppet_modules: puppet/modules/:/etc/puppet/modules timeout: 360




プラグインアセンブリ



pre_build_hookをfuel-libraryの対応するバージョンと互換性があります。 この場合、次のようになりました。 



 #!/bin/bash set -eux ROOT="$(dirname `readlink -f $0`)" MODULES="${ROOT}"/deployment_scripts/puppet/modules TMP_DIR="${ROOT}"/tmp mkdir -p "${MODULES}" mkdir -p "${TMP_DIR}" REPO_PATH='https://github.com/stackforge/fuel-library/tarball/f43d885914d74fbd062096763222f350f47480e1' wget -qO- "${REPO_PATH}" | \ tar -C "${MODULES}" --strip-components=3 -zxvf - \ stackforge-fuel-library-f43d885/deployment/puppet/{inifile,stdlib}




これで、次のようにパッケージをビルドできます。 



 fpb --build external_nfs




パッケージはexternal_nfs / external_nfs-1.0.0.fpとして保存され、 Mirantisクイックスタートマニュアルに示されているように、VirtualBoxを使用してモジュールをデプロイできます。



重要！ OpenStackのデプロイ後に仮想マシンのスナップショットを作成することを忘れないようにしてください。



プラグインの展開



そのため、プラグインが作成されます。 これをFuel Masterノードにコピーし、展開用にインストールします。 



 scp external_nfs/external_nfs-1.0.0.fp root@master-node: ssh root@master-node fuel plugins --install external_nfs-1.0.0.fp




プラグインがインストールされていることを確認してください： 



 $ fuel plugins




インストール後、環境をFuelインターフェースに追加し、[設定]タブでNFSプラグインを構成できます。 ノードを追加してプラグインを確認することもできます。 テストに移る前に、考えられる問題を診断して解決する手順を見てみましょう。



トラブルシューティング



たとえば、一般的な状況：ファイルは存在しますが、見つかりません。 たとえば、install_packages.shスクリプトのパスを指定しなかったと仮定します。 



 - role: ['cinder'] stage: post_deployment type: shell parameters: cmd: install_packages.sh timeout: 180




この場合、クラウドのデプロイは失敗し、システムは次のエラーを表示します。 



 Deployment has failed. Method deploy. Failed to deploy plugin external_nfs-1.0.0. Inspect Astute logs for the details.




この場合、Fuelマスターノードの/var/log/docker-logs/astute.logにあるログを確認したいことが想定されており、特定のケースでは次のようになります。 



 2014-12-23T16:31:54 debug: [395] c5d1e2b5-56bb-4428-9666-8c5574cb06a4: MC agent 'execute_shell_command', method 'execute', results: {:sender=>"3", :statuscode=>0, :statusmsg=>"OK", :data=>{:stdout=>"", :exit_code=>127, :stderr=>"sh: 1: install_packages.sh: not found\n"}} 2014-12-23T16:31:54 debug: [395] c5d1e2b5-56bb-4428-9666-8c5574cb06a4: cmd: cd /etc/fuel/plugins/external_nfs-1.0.0/ && install_packages.sh cwd: /etc/fuel/plugins/external_nfs-1.0.0/ stdout: stderr: sh: 1: install_packages.sh: not found




これは、tasks.yamlのinstall_packages.shパスが正しくないことを示しています。 次のように修正できます。 



 - role: ['cinder'] stage: post_deployment type: shell parameters: cmd: ./install_packages.sh timeout: 180




修正後、パッケージを再構築し、Fuel Masterノードで更新します。 



 fpb --build external_nfs scp external_nfs/external_nfs-1.0.0.fp root@master-node: ssh root@master-node fuel plugins -- install --force external_nfs-1.0.0.fp




結果を確認する



この問題を解決したら、新しい環境を作成し、クラウドを構成して展開できます。 展開が成功したら、CinderがNFSバックエンドで適切に動作していることを確認します。 簡単にする：



1.コントローラーにログインして、Cinderボリュームを作成します。 



 ssh root@master-node ssh controller-node source openrc cinder create 1




2. Fuel CLIでボリュームの状態を確認します。 詳細については、 Mirantis OpenStackユーザーガイドを参照してください。



燃えがらリスト







3. NFSで作成されたcinderリストの出力からのIDが、volume-Id値に対応する名前のファイルとして手探りされていることを確認します。 



 nfs-server$ ls volume-d9382a2b-fd13-424b-9eba-3a77a1d16008




CinderノードでPuppetマニフェストを修正する場合は、/ etc / fuel / plugins / external_nfs-1.0.0 / puppetディレクトリで見つけることができます。 



 ssh root@master-node ssh cinder-node cd /etc/fuel/plugins/external_nfs-1.0.0/ puppet apply --modulepath=/etc/fuel/plugins/external_nfs-1.0.0/puppet/modules site.pp




詳細については、 こちらをご覧ください 。



プラグインリリース



プラグインを作成すると、次のオプションのいずれかを使用できます。



1. Mirantisチームに連絡せずに、プラグインを自分でデプロイして使用します。 このシナリオには2つのマイナス点があります。クラウドは当社のサポートによってサポートされないため、お客様はご自身の責任とリスクで行動してください。 プラグインはカタログに公開されないため、コミュニティはあなたの偉業を知らず、あなたの仕事の結果を利用できず、作成されたプラグインの機能を評価できません。



2.検証なしでMOSのプラグインを配置します。 つまり、Mirantis OpenStackをダウンロードする他の人にもプラグインをダウンロードすることを許可しますが、そのようなインストールはMirantisチームによってサポートされません。 これを行うには、driverlogにエントリプラグインを置くだけです 。



3.プラグインを配置し、 Mirantisパートナーになり（つまり、Mirantis Unlockedプロジェクトの特定の要件を満たし）、検証のリクエストを送信し、それを認証します。検証手順でプラグインの品質が確認された場合、カタログで公開し、MOSクラウドをサポートしますこのプラグイン。



検証プロセスはオプションであり、時間がかかりますが、検証済みのプラグインはFuelプラグインカタログに追加され、Mirantisサポートによってサポートされます。 これは、特にスタッフに多数のOpenStackスペシャリストがいない大企業にとって重要です。



プラグインの検証



それでもプラグインを検証することに決めた場合は、手順について簡単に説明します。 しかしその前に、私たちはあなたがミランティスのパートナーでなければならないことをもう一度思い出します。 プラグインの統合（および検証はMOSとの統合手順の1つ）は、アフィリエイトプログラムのフレームワーク内でのみ可能です。



まず、Mirantisに特定のドキュメントセットを提供する必要があります。



-プロジェクトの仕様（または「仕様」。このドキュメントはパーティーで呼び出されるため）、

-プラグインのインストールとデプロイのガイド、

-ユーザーマニュアル

-テスト計画、

-テスト結果。



マニュアルの構成方法の詳細については、 こちらをご覧ください 。



第二に、認証プロセス自体は次の手順で構成されます。



-プロジェクトの仕様のレビュー、

-コードレビュー、

-テスト計画のレビュー、

-プラグインの展開手順のレビュー、

-ユーザーマニュアルのレビュー、

-開発者が提出したテスト結果の分析、

-Mirantisチームによるネイティブプラグインのテスト。



重要： Fuelなしでプラグインを使用できます。また、認証は必須の手順ではありません。 Mirantisチームによる著作権プラグインを使用してMOSクラウドのサポートを保証する場合に必要です。 しかし、私たちは自分自身を繰り返しているようです。 しかし、それは本当に重要です！



結論の代わりに：プラグインでOpenStackの機能を拡張する



例としてNFSを使用してプラグインを作成する方法を示しましたが、独自のニーズを満たすプラグインを作成できます。 Fuelの柔軟性により、クラウドを快適に展開および管理できます。 さらに、認定手続きを怠る必要がない場合は、プラグインのMirantisスペシャリストのサポートを受けられます。



サポートについて話すとき、プラグイン自体-そのコード-は開発者によってサポートされ、Mirantisチームは失敗したプラグインでMOSクラウドプラグインをサポートできるという事実について話します。



Fuelのプラグインを作成することで、OpenStackエコシステムの機能を拡張できます。 そして、慎重に食べ物を噛む-社会を助けて！
