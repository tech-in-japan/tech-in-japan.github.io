Firefox拡張機能を自動的に更新する

記事「 Firefox拡張機能の自動更新 」の翻訳に注目してください。



この記事は古いですが、すべての情報はこの日に関連しています。



もちろん、Firefox拡張機能の開発者は、 公式ストアを通じて拡張機能を配布する場合、拡張機能を無料で自動的に更新できることを知っています。 しかし、拡張機能を自分のサイトでホストしたい場合はどうでしょうか？ 自動更新のサポートをどのように実装しますか？



拡張機能に署名する



Firefoxの3番目のバージョン以降、すべての拡張機能の更新は信頼できるチャネルを通じて提供される必要があります。 拡張機能を独自にホストすることに決めたため、この要件を満たすための2つのオプションがあります。



  • セキュア（https）接続を介して更新を提供する
  • 拡張機能に電子署名する


最初のオプションは考慮しません-ほとんどの場合、これには追加費用が必要です（SSL証明書を購入し、静的IPを所有する必要があります（私はそれが何であるか理解できませんでしたが、曲から単語を消去しません[おおまかです。Transl。 ）））無料で使いやすく、実装が高速です。 さて、それを使用する方法を学びましょう！



秘密/公開キーのペアを作成する



拡張機能に署名する最初のステップは、秘密/公開キーのペアを作成することです。 Mozillaは、この目的のためにMcCoyユーティリティを提供しています。 世界で最も簡単なユーティリティではないため、以下にその調理方法を示します。



  • McCoy Webサイトにアクセスして、適切なパッケージをダウンロードします（Windows、Linux、およびMac OS Xのバージョンがあります）
  • 便利なディレクトリにパッケージを展開します
  • アプリケーションを実行します。 最初の起動時に、キーを保護するためのマスターパスワードの作成が提案されます。 そのようなパスワードを作成することを強くお勧めします！ McCoyは、再起動するたびに、このパスワードを要求します。
  • アプリケーションが実行されている場合は、メニューの[キー]»[新しいキーの作成]を選択します。 キーの正しい名前を入力し、[OK]をクリックします。 サイトで複数の拡張機能をホストする場合でも、1つのキーで十分です。


さて、キーが作成されました。次に、拡張マニフェストを更新する必要があります。

update.rdfを作成します



拡張機能自体が完成したので、更新マニフェストを作成します。 このファイル（update.rdf）はサーバー上に存在し、実際、更新を確認するときにユーザーに表示される内容を決定します。 まず、作成するファイルの最終バージョンを簡単に見てみましょう。 



 <?xml version="1.0"?> <r:RDF xmlns:r="http://www.w3.org/1999/02/22-rdf-syntax-ns#" xmlns="http://www.mozilla.org/2004/em-rdf#"> <r:Description about="urn:mozilla:extension:tuttoolbar@borngeek.com"> <updates> <r:Seq> <r:li> <r:Description> <version>1.0.2</version> <targetApplication> <r:Description> <id>{ec8030f7-c20a-464f-9b0e-13a3a9e97384}</id> <minVersion>4.0</minVersion> <maxVersion>30.*</maxVersion> <updateLink>http://www.example.com/downloads/tuttoolbar_1_0_2.xpi</updateLink> <updateHash> sha256:c22ad513c1243959a6d8e6b3cfad18a2a9141306f0165da6b05b008b2042e502 </updateHash> </r:Description> </targetApplication> </r:Description> </r:li> </r:Seq> </updates> </r:Description> </r:RDF>


行ごとに見て、ここで何が起こっているのかを議論しましょう。 最初に、r：RDF要素が続く標準XMLヘッダーがあります。これは、RDF（リソース記述フレームワーク）ファイルがあることを示しています。 主にr：Description要素に興味があります。 update.rdfで記述する拡張機能ごとに1つの要素が必要です。



（ はい、はい、1つのupdate.rdfで、多くの拡張機能の更新、 拡張機能のバージョン管理、更新と互換性を説明できます[翻訳に注意してください。] ）

マニフェストに署名する



マニフェストに署名するには、次の簡単な手順を順番に実行します。



  • マニフェストの開発者バージョンをコピーし、update.rdfという名前を付けます。 署名するのはこのファイルです
  • McCoyがまだ実行されていない場合は起動します
  • 署名するキーを選択し、[更新]メニューで[更新マニフェストに署名]を選択します
  • 開いたメニューで、作成したupdate.rdfを選択し、[開く]をクリックします


何も起きていないように見えますが、そうではありません！ 私はマッコイにそのような主張をしている-彼は仕事の終わりに何も言わない。 ただし、マニフェストを開くと、少し変更されていることに気付くかもしれません。 実際、これは拡張機能の作業が終了する場所です。今度はサーバーにファイルをアップロードします。



ホストupdate.rdf



ファイルをアップロードする前に、サーバーがrdfおよびxpiファイルをホストできる状態であることを確認する必要があります。 Apacheでは、.htaccessルールを使用してこれを実行できます（IISまたはTomCatでどのように機能するかわかりません。これはApache専用の命令であることに注意してください）。 通常、これらのルールはサイトのルートの.htaccessに配置します-xpiとrdfを移動したい場合に備えて。 ルールは簡単です： 



 AddType application/x-xpinstall .xpi AddType text/xml .rdf


（ 今ではすでにファッショナブルなnginxです。これを行うために： 



 types { application/x-xpinstall xpi; text/xml xml, rdf; }


これを一般的な/etc/nginx/mime.types



またはサーバーの構成に配置します。 [約。 翻訳。] ）



これは、サーバーによるファイルの正しいアップロードに必要です。 これは非常に重要なポイントです！ そうしないと、拡張機能がインストールされず、更新も機能しません。 また、覚えておいてください：何らかの種類のCMS（WordPressなど）を使用する場合は、念のためサイトのルートにこれらのルールを配置することをお勧めします。
