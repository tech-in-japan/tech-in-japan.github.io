MonoとMS.NETの違いについて話しましょう





.NETのクロスプラットフォーム開発は、日々ますます現実的になっています。 そして、最近の公式のLinux / MacOSサポートの発表後、幸せな未来は少し近くなりました。 ソースコードは現在MITの下にあるため、上の図は以前の関連性を失っています。 ただし、クロスプラットフォームの.NETアプリケーションは長期間作成できます。Monoはこれを支援します。 しかし、コミュニティでの彼に対する態度はかなりあいまいです。 「モノは愚かで、すべてが3倍遅くなる」や「モノの下では何も正常に起動しない」などの言葉をよく耳にします。 さらに、これらの人々から特定の事実を聞くことは非常にまれです。 質問「何が本当に愚かですか？」または「何が正確に機能しませんか？」それらをst迷に陥れます。 すべてではありません（建設的な議論が可能なものもあります）が、ほとんどです。 ほとんどの場合、inする答えは「はい、何も機能しません！」の精神で始まります。 そして、それが機能する場合、非常に遅いです！」 会話の終わりに、Monoでの最後の各マシンコマンドは数倍遅くなり、ソースコードの半分にthrow new Exception()



です。



この投稿では、少し経験を共有したいと思います。 少し前に、 PassportVision製品（ Habréでの発表 ）をLinuxに移植しました。 かなり正常に機能していると言えます。 はい、Microsoftの従来の.NET（以降-MS.NET）上のWindowsよりも少し遅いです。 しかし、それは非常に安定して機能し、パフォーマンスの低下は根本的なものではありません。 同時に、当社の製品は非常に大きく、エンタープライズのカテゴリーに分類され、C＃/.NET機能を最大限に活用しています。 したがって、.NET用の大規模なサーバーアプリケーションを取得するのは現実的です-要望があります。 また、Monoで何かを書いているさまざまな開発者と話す機会がありました-主に成功した物語です。



しかし、なぜモノに対してそれほど否定的なのですか？ 問題は、人々がランタイムの違いを本当に理解したくないということだと思います。 Mono 2.4でLinux用の.NETアプリケーションを起動したが、すぐには起動しませんでした。つまり、Monoは完全に悪いので、使用しません。 そして最後に、唯一の方法は非難することであり、その実装はMS.NETとはわずかに異なります。 新しいMonoバージョンは2ヶ月に1回リリースされ、実装は長い間修正されていますが、人々は詳細を理解することを望まず、貧しいMonoを歩き続け、大切にし続けています。



今日は、ランタイムが異なる場合の例をいくつか示します。 もちろん、すべての違いを持ち出すことはできません-あなたはそれについて本全体を書くことができます。 はい。また、非常にクールな問題を製品コードから引き出すこともできません。ソフトウェアアーキテクチャのコンテキストからそれらを分離して、理解しやすい小さなコードの形にすることは非常に困難です。 今日の私の仕事は、MonoとMS.NETの違いを伝えることです。 これが初心者のMonoプログラマーが、「Mono dumb」という言葉を振り返るのではなく、発生する問題をより広く見て対処するのに役立つことを願っています。



例1.異なるバージョンのコンパイラー



非常に簡単な例から始めましょう。 C＃プログラムがあり、それをコンパイルしたいとします。 知っているように、標準コンパイラは単にC＃コードを対応するILコードに変換し、アセンブリの形式で実行します。 さらに、翻訳プロセスには特に巧妙な最適化が含まれていません。JITが最適化を担当しているためです。 翻訳は非常に簡単であり、経験豊富な.NET開発者は多くの場合、どのようなILコマンドが発生するかを把握できます。 しかし、何らかの理由で、多くの開発者はこのプロセスが明確であることを確信しています。 「ソースプログラムをILにマップする方法はまさに1つである」ということを証明しようと、かなり熱烈な議論がいくつかありました。 通常、人々は「仕様があります！」という素晴らしい議論でこれを動機付けます。 仕様では多くの有用なことが述べられていることを保証したいのですが、任意の番組を放送するための唯一のオプションについては何も言われていません。 さらに、コンパイラのバージョンが異なると、翻訳ロジックが異なる場合があります。 これを説明する非常に簡単な例を取り上げました。 次のコードを検討してください。

例3. Monoのバグ



はい、Monoにはバグがあります。 そして、はい、彼らは非常に少なくありません、私たちはそれと共に生きなければなりません。 幸いなことに、それらをヒットする必要はめったにありません。また、オープンなバグのうち、重大なバグはあまりありません。 1つの話をしましょう。PassportVisionをMonoに移植するとき、あまり気に入らない瞬間に出会いました。 多くの条件に応じて、コード生成を処理し、その場でロジックの一部を作成する必要がありました。 TypeBuilderを使用して新しい型を作成し、これらの型は特定のインターフェイスを実装しました。 Monoでのコード生成について多くの新しいことを学びましたが 、その大部分はコンテキストから簡単に伝えることは非常に困難です。 しかし、1つのバグが非常に簡単に再現されましたTypeBuilder.CreateType()

 private interface IFoo {} // ... void Main() { var assemblyBuilder = Thread.GetDomain().DefineDynamicAssembly( new AssemblyName("FooAssembly"), AssemblyBuilderAccess.Run); var moduleBuilder = assemblyBuilder.DefineDynamicModule("FooAssembly"); TypeBuilder typeBuilder = moduleBuilder.DefineType("Foo", TypeAttributes.Public, typeof(object), new[] { typeof(IFoo) }); typeBuilder.CreateType(); }

そのような会話の後、石は通常モノの方向に飛び始めます。彼らはそれは一種の悪いことであり、それにバグがあると言います。 そして、誰もがマイクロソフトがバグを行っていることをどういうわけか忘れています。 次の話では、私の友人に一定数の神経細胞がかかりました。 次のコミットの後、ビルドサーバーはテストが失敗したことを報告しました。 さらに、稼働中のマシンではすばらしい仕事をしましたが、ビルドサーバーに落ちました。 バグの検索方法に関する魅力的な説明は省略し、要点を説明します。ビルドサーバーと作業マシンに異なるバージョンの.NET Frameworkがインストールされました：4.0および4.5。 しかし、バグ自体はその行でした 



 new Uri("http://localhost/%2F1").AbsoluteUri

例5.標準クラスの実装の微妙さ



別の有益な話をさせてください。 ある優れた会社の社員は、構造をハッシュテーブルのキーとして使用することを決定しました。 悪いことは何もありませんでした。アプリケーションは正常に動作しました。 少し遅くなりましたが、それほど重大ではありません。 そして、ある日、連中はMonoでアプリケーションを起動することにしました。 そして、見よ：それは何倍も速く動き始めた。 「ああ、なんて良いモノなのか、その下ですべてがどれだけ速く機能するのか」と彼らは喜んだ。 私を悩ませた唯一のことは、そうではなく、どこかが間違っていたという考えでした。



この例を理解するには、MS.NETの構造に対するGetHashCodeの実装について少し読む必要があります（ 優れたハブポスト）。 つまり、ハッシュ関数には2つのバージョンがあります。1つは参照フィールドとフィールド間の空きスペースのない構造用で、もう1つは他のすべてのバージョン用です。 思慮深い読者が推測したかもしれませんが、私たちの仲間はキー構造でうまくいきませんでした（つまり、フィールド間に「穴」がありました;明示的なGetHashCode（）を登録することを誰も気にしませんでした）。 そしてこの場合、構造の最初のフィールドに基づいたハッシュ関数の2番目のバージョンが使用されます。 例を考えてみましょう： 



 var a1 = new KeyValuePair<int, int>(1, 2); var a2 = new KeyValuePair<int, int>(1, 3); Console.WriteLine(a1.GetHashCode() == a2.GetHashCode()); var b1 = new KeyValuePair<int, string>(1, "x"); var b2 = new KeyValuePair<int, string>(1, "y"); Console.WriteLine(b1.GetHashCode() == b2.GetHashCode());


このコードは、MS.NETでFalse True



を出力します。 ハッシュa1



とa2



は異なり（すべてのフィールドに基づいて計算されます）、 b1



とb2



ハッシュは一致します（最初のフィールドに基づいてのみ計算されます）。 Mono開発者は、さまざまなバージョンのハッシュ関数に煩わされないことに決めました。彼らは、すべてのフィールドに基づいて機能するものを作成しました（ GetHashCodeおよびInternalGetHashCodeを参照）。 したがって、上記のコードはすべてのハッシュが異なるため、モノラルの下でFalse False



を出力します。



ハッシュテーブルを使用して、楽しい仲間に話を戻しましょう。 状況を正しい角度から見ると、そのアプリケーションはMonoでは動作しませんでしたが、MS.NETでは動作が遅くなりました。 そして、最初のフィールドが一致するため、多くのキーが同じハッシュを持っているという事実のため、速度が低下しました。 些細なことのように思えますが、パフォーマンスにはかなり強い影響がありました。



ご存知のとおり、.NETの内部構造について誰かに話すと、彼らはしばしばフレーズで私を批判します。「なぜこれをすべて知る必要があるのですか。 このような内部実装の詳細は、現実の世界では決して役に立ちません 。」 そして、私は答えます：「まあ、もちろん、彼らは誰にも役に立たないでしょう。」
