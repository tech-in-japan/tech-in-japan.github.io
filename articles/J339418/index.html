DevOops 2017：レビューレポート

以前、 DevOops会議を発表し、プログラム委員会とのインタビューを公​​開しました 。 そして今、会議の2週間前にプログラムが結成されたら、特定の報告について話し合うときです。 10月20日にサンクトペテルブルクで観客は何を期待しますか？ Kubernetesのニュアンスからギリシャの悲劇まで。 以下にトピックのリストを示します。対応するレポートはカットの下に描かれています。

クベルネテス





数年前、貝殻を耳に当てると、「Docker」という言葉が聞こえました。それはどこからでも聞こえたからです。 コンテナ自体はもはや目新しいものではありませんが、そのオーケストレーションは非常に議論されたトピックになっており、Kubernetesはどこにでも聞こえます。 したがって、DevOopsでは、いくつかのレポートが一度に彼に関連付けられます。





Ivan Glushkov-k8を使用する[しない] 10の理由



一方で、このレポートは、Kubernetesをまだ使用していない人がこれを決定するのを支援することを目的としています。 その一方で、すでにそれを最大限に使用している人は、自分にとって興味深いものを見つけるでしょう。彼らは、困難に対処するのに役立つ特定の「集められた熊手」と開発を検討します。





ニコライ・リジコフ-k8の拡張



ここでは、前のケースと同様に、Kubernetesに関連するいくつかの問題とその修正方法について説明します。 しかし、問題はこのツール自体に限定されるものではなく、ニコライは全体像でそれを見る-「Kubernetes、ThirdPartyResources、およびOperators（Custom Controllers）プリミティブを使用して最終的なソリューションを構築する慣用的な方法」を示すつもりです。





Craig Box-KubernetesとIstioを使用してマイクロサービスを管理する



モノリスからマイクロサービスへの移行をどのように賞賛しても、否定することは不可能です。人生を単純化するだけでなく、複雑にします。 システムに多数の独立したコンポーネントがある場合、それらの相互作用のすべてを理解することさえ容易ではありませんが、それらを効果的に管理する必要もあります。 クレイグボクシングは、KubernetesとIstioの共有がこれにどのように役立つかについて話します。 このレポートは、DevOpsについて共通の理解を持っているすべてのユーザーが利用できますが、主にコンテナーとKubernetesをすでに使用したことがあるユーザーを対象としています。







アプリケーションの監視と監査





Adrian Cole-遅延の原因を適切に責める方法：分散トレースとZipkinの紹介



応答時間分析は、有罪の検索です。どのコンポーネントがユーザーに顕著な遅延を引き起こしましたか？ マイクロサービスの台頭により、容疑者の輪が拡大し、「調査員」になることがより困難になりましたが、ジプキンは助けることができます。 エイドリアンとコールは理論的な部分を説明し、小さなデモを見せます。





Sergey Sharapov-SmartMonitoring-Odnoklassnikiのビジネスロジックの監視



Odnoklassnikiは困難な状況に直面しています。そのような場合、よく知られたソリューションに満足せず、独自のソリューションを作成します。 その結果、ソーシャルネットワークにはそのような問題について多くの専門知識があり、すぐに共有することができます。 この場合、彼らはポータルで異常を見つけるだけでなく、それらの間の接続を表示する独自の監視システムの作成について話します-管理者と開発者の生活を楽にします。





継続的デリバリー





Vitaliy Khabarov-Windows環境での継続的デリバリー



WindowsでMarsとDevOpsに生命はありますか？ 結果に満足し続けるために、人々はこの環境で継続的デリバリーを構築しますか？ もしそうなら、どのくらい正確に？ 最初の質問に対する答えはイエスです。Vitalyのストーリーは、Windowsアプリケーションサーバーを使用したテスト環境の起動を正常に自動化した経験に基づいています。 そして、彼はレポートの3番目の質問に詳細に答えます。





Paul Stack-インフラストラクチャをクラウドに継続的に提供



Paulは現在Joyentで働いていますが、最近までHashiCorpの従業員でした-この会社のPackerおよびTerraformツールをよく知っているのは驚くことではありません。 また、DevOopsでは、「インフラストラクチャをクラウドに継続的に提供する」というテーマが、その使用例によって明らかになります。 予備調査では、多くの人がこのレポートを待っていることがわかりました。



小さな免責事項：ポールは現在、ビザに関して少し困難を抱えていますが、近い将来に首尾よく解決されると信じています。 突然会議に影響が出た場合、変更はすぐにサイトに反映されます。







構成管理







アンドレイ・フィラトフ-塩辛いパペットシェフのアンサンブル：Ansible、SaltStack、シェフ、パペットの比較



IT担当者は永遠に何ができるでしょうか？ どのツールが優れているかを議論する。 しかし、これはしばしば無意味で容赦ないホリバーに変わります。 このレポートでは、同様のツールの比較もありますが、「真空状態」ではなく、特定のタスクに関連して、はるかに建設的でバランスが取れています。







雲







Radek Simko-複数のクラウドFTWの管理



AWSとGoogle Cloudのどちらが良いですか？ もちろん、これはそれらが正確に何に使用されるかに依存します。異なるシナリオでは、答えは異なる場合があります。 さらに、これらのさまざまなシナリオは1つの大きなプロジェクトに共存できます。これにより、さまざまなクラウド間で負荷を共有することを考えるようになります。 しかし、2つの雲の上に座るのは2つの椅子に座るよりも難しい場合があります。これらすべてが調和して共存するためには、多くの問題を解決する必要があります。 HashiCorpのTerraformはこれらの問題の解決に役立ちます-Radek Simkoが正確にその方法を教えてくれます。





Oleg Anastasiev-ワンクラウド：OdnoklassnikiのデータセンターレベルのOS



すでに上で述べたように、Odnoklassnikiは、他の人が標準的なもので管理する場合に、しばしば独自の決定を行います。 だからここにある：誰もがKubernetesとMesosを使用しているが、彼らはニーズに合うように完成品をカスタマイズするよりも自分でやることのほうが良いと判断した。 Oleg m0nstermind Anastasievはすでにインタビューでこれについて簡単に語ってくれました。カンファレンスでは、彼はもっと技術的な詳細を説明します。 同様のプロジェクトが明らかに開始されないであろう中小企業の人々に耳を傾けることは興味深いでしょうか？ レポートの説明は、「Dockerが落ちないようにすることができますか」という質問に対する答えを約束します-そして、誰もが少なくともこれについて理論的に知りたいです。







一般的なDevOps



会議レポートのほとんどは、特定の技術的なトピックを詳細に分析します。これは、他の会議の訪問者にはすでに馴染みのあるアプローチです。 しかし、DevOpsの場合、すべてが多少異なります。この概念を特定のツールとプラクティスのセットに単純に分解することはできません。 したがって、「全体像」を示すパフォーマンスがあります。





Corey Quinn-私と一緒にスケールアウトしてください：あなたが持っていない問題を解決する



AWSニュースレターの先週で知られるCorey Quinnは、「私たちのインフラストラクチャは暗い領域の光線であり、あなたとあなたの会計士として再訓練される」という精神で多くのDevOpsレポートを見ました。 これらの「光線」の一部を内側から見ると、コーリーはこのアプローチに強く反対していることに気づきました-そして、会議のオープニング基調講演で、一般的にすべてがひどいのはなぜですか、同時にあなたは大丈夫です。





Alexander Titov-システム管理者から人へ



通常、DevOpsについて話すとき、2つの相反する関係者が思い出します-異なるタスクとアプローチを持つ管理者と開発者。 アレクサンダー・ティトフは、一般にビジネスの側面もあることを思い出し、一般に、最終的な分析では、すべてがそのタスクに良い影響を与えるはずです。 したがって、彼のレポートは「システムとしてのDevOpsについて：それがビジネスにどのように役立つか、エンジニアの能力がDevOpsにどのように見えるか、DevOpsのソフトウェア生産方法によって解決できるビジネスタスク、DevOps生産への道でどのようなエラーが起こり得るか」それらを避けたり、止めたりしてください。」





Sergey Berdnikov、Artyom Kalichkin-サクセスストーリー、または「Dev + DevOps + Ops」



「ロードマップではスムーズでしたが、渓谷については忘れていました」という状況があります。すべてを明確にしようとするアイデアは、それらを実装しようとすると、多くの説明のつかない機能に出会います。 もちろん、DevOpsでもこれは事実です。確立されたルールと習慣を備えた企業で使用され始めると、多くの困難が突然発生します。 このような状況でDevOpsプラクティスを正常に実装するためにDevOpsプラクティスをどれだけ歪めれば、誰もが気分が良くなりますか？ セルゲイとアーティオムは、彼ら自身の経験について話します。







バルーク・サドグルスキー





映画評論家のジョークとして、トムクルーズの参加は、あらゆる映画（少なくともコメディ、SF）を「トムクルーズのある映画」という特別なジャンルに変えます。 多くの有名なBaruch jbaruch Sadogursky （JFrog）についても同様です。彼のカリスマ性のために、「Baruchのレポート」は別のジャンルでもあります。 また、BaruchはDevOpsの熱烈な支持者であるため、この会議ではStakhanoviteになり、3つのレポートを作成し、円卓会議を開催します。 彼のすべてのスピーチの説明をグループ化しました。





バルーク・サドグルスキー、レオニード・イゴルニク-シリコンチェーンソー大虐殺（先週末の勤務中）



DevOopsに関する他のレポートと同様に、Baruchは一緒に話します。この場合、 Leonid Igolnik （CA Technologiesの開発者であり、ベンチャー投資家）と話します。 一緒になって、「血の雰囲気の中で、時計が手足や腸の汚れを飛ばさないようにするための監視プロセスを構築する方法」について話し合います。





Ray Tsang、Baruch Sadogursky-Kubernetesでの運用アプリケーションのトラブルシューティングとデバッグ（別名The Failing Demo Talk）



BaruchはJFrogの開発者支持者であり、Google Cloud Platformの開発者支持者が参加します。 当然のことながら、Kubernetesアプリケーションの実稼働デバッグでは、Google Cloud PlatformとJFrog Artifactoryの両方が関連する可能性がある状況を考慮します。 ただし、これらのプロジェクトについてだけでなく、Kubernetes自体の組み込み機能についても説明します。





Baruch Sadogursky、Alexander Titov-ラウンドテーブル：DevOpsを同僚、上司、その他の無関心な人に「売る」方法は？



これは標準的なレポートではなく、多数の会議の講演者が参加するラウンドテーブルです。 参加者の輪は現在明確にされており、これは一瞬のリハーサルされたパフォーマンスではないので、私たちはそれがどのように見えるかを完全には知りません。 しかし、バルークをリードする役目を果たすと、すぐに明白になります。鼻の下での退屈な泡立ち以外は期待できません。





バルーク・サドグルスキー、レオニード・イゴルニク-スケールでのDevOps：3つの行為におけるギリシャの悲劇



最後に、会議はBaruchとLeonid Igolnikの基調講演で終了します。これは、会社がさまざまな段階で成長するにつれてDevOpsの問題が生じることについてです。 このような3つの段階があり、結果は「3つの行為におけるギリシャの悲劇」です。



すべてがどのように感じるかについてのアイデアを与える2つの組織の詳細があります。 第一に、講演者はトガスやその他の環境を要求し、ギリシャの悲劇の雰囲気を適切に再現するのを助けました。 第二に、彼らは聴衆に、この公演の前の休憩中であってもパーティーのためにアルコールを準備するように頼み、ある意味でパーティーを開いた。 一般的に、一日の終わりに、数字やグラフを知覚することが難しくなると、そのような問題が明らかに発生しないパフォーマンスがあります。







会議プログラムはすでに完全に知られているため、「それに行くかどうか」という決定を延期する理由はもうありません。 判断する時が来ました。説明に興味がある場合は、 DevOops Webサイトでチケットを リクエストして ください 。 10月20日にサンクトペテルブルクで皆さんを待っています（そしてサンクトペテルブルクに決していない人は、聴衆の間でオンライン放送を待っています）。
