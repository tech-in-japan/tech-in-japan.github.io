こんにちは、ハブロフチャン。

この記事は条件付きで3つの部分に分けることができます。 最初に、RS232 /イーサネットインターフェイスコンバーターについて説明します。RS232/イーサネットインターフェースコンバーターとは、その使用方法です。

2番目では、ModBus ASCII_RTU / ModBus TCPゲートウェイを作成する例を使用して、Tibboコンバーターのプログラミングの可能性を示します。

最後に、第3部では、SMSメッセージ、Webベースの管理インターフェイスを使用してコマンドを送信し、MySQLにデータをアップロードする機能を備えた、わずかに変更されたTibboインターフェイスコンバーターから自動加熱制御システムを作成しようとします。









1.デバイスサーバー。



1.1。 Device Serverまたは一般的なものとは何ですか。


西洋用語のシリアルデバイスサーバーとは、シリアルインターフェイス（RS232 / 422/485）からイーサネットへの変換器を意味します。 そのようなデバイスの必要性は、単純な理由で発生しました：製造業者に関係なく、ほとんどすべてのマイクロコントローラーがUARTインターフェースを使用し（これはこのインターフェースを実装する技術的な単純さによる）、TTLまたはCMOSロジックを使用して「外界」と通信します。 したがって、RS232 / 485インターフェースを介してデータ伝送チャネルを受信するには、開発者がトランシーバーのマイクロ回路をコントローラーに接続するだけで十分です。 しかし、シリアルインターフェイスは、最新の通信システムを作成するのに十分な機能を提供しません。 一方、ローカルサブネットを作成し、それらを組み合わせて、世界中の地理的に分散したポイント間のデータチャネルを整理できる高度なイーサネットテクノロジーがあります。 通信機器のメーカーがシリアルインターフェイスとイーサネットテクノロジを「組み合わせる」ことを決定したのは論理的です。 したがって、シリアルデバイスサーバーと呼ばれるハードウェアゲートウェイが登場しました。



台湾の会社Tibbo Technologyの RS232 /イーサネットインターフェイスコンバーターは、10年間世界中で使用されています。 この間、同社のエンジニアはこの機器を使用したプロジェクトの実装で膨大な経験を蓄積し、重要なタスクは解決され、デバイスの機能は「通常の」コンバーターの範囲をはるかに超えて新しいレベルに拡張されました。 デバイスのこのラインは、 Tibbo Device Server- プログラマブルインターフェイスコンバーターと呼ばれます。 この定義のキーワードは、「プログラム可能」という言葉です。これは、デバイスの最終的な機能が、作成されてロードされたファームウェアに完全に依存するためです。

しかし、簡単なものから始めましょう。







1.2。 シリアルオーバーIP。


構造的に、Tibbo Device Server（DS）デバイスは、タバコのパックのサイズの小さな箱です。 一方には、シリアルインターフェイス用の標準コネクタ（または複数のコネクタ）があり、他方には、イーサネットテクノロジを使用してツイストペアケーブルを接続するためのソケットがあります。 デフォルトでは、Tibbo Device Serverには「Serial Over IP」ファームウェアが付属しており、RS232 / 485インターフェースからイーサネットへのデータ転送が可能です。

物理的な切り替え後、デバイスの設定を構成する必要があります。 このため、メーカーのユーティリティセットが使用されます。DSManager-デバイスパラメータの構成に使用され、VSP Manager-仮想COMポートパラメータ、仮想COMポートドライバ、およびいくつかの診断ユーティリティの構成に使用されます。 設定の説明については触れませんが、これらのユーティリティのいくつかの機能に注意を向けます。

  • デバイス設定は、ローカルネットワークを介して直接構成されます。
  • ソフトウェアはどのWindows OSでも動作します。
  • 仮想COMポートドライバーは適切に作成されており、 ここなどの多くの問題を回避できます。 また、Linux用のドライバーバージョンが作成されています。
  • ソフトウェアのカスタマイズの可能性があります。


インターフェイスコンバータは、3つのモードのいずれかで構成できます。 クライアントモードは、シリアルデバイスがイーサネット接続を開始するときです。 このモードは、パッシブモニタリングに使用されます。 たとえば、特定の間隔のPBXがサーバーの統計情報をドロップする場合。 または、バーコードスキャナーがデータを1Cに送信します。 このモードでは、データの送信先のIPアドレスを指定します。サーバーの仮想COMポートは、着信データストリームをTCP / IPポートでリッスンします。

サーバーモード—シリアルデバイスは、着信要求に応答します。 このモードは、I / Oモジュール、PLC、カウンターなどのポーリングに使用できます。



サーバー/クライアントモード-上記の両方のオプションを組み合わせます。 この場合、サーバーの仮想COMポートは、着信データストリームをリッスンし、指定されたIPアドレスでデバイスとの接続を確立できます。 そのようなソリューションはどこで使用できますか？ これは多数の垂直市場とアプリケーションです。 例：

  • 自動エネルギー計量システムのシステムにおける電気メーターの尋問と監視。
  • 住宅および共同サービスにおける熱量計または水道量計の調査。
  • バーコードスキャナーまたはRFIDリーダーからのデータ収集と、仮想ポートを介した1Cへの転送。
  • 入力/出力モジュールの尋問;
  • PLCモジュールのリモート監視および制御。
  • 電話交換またはビルエンジニアリングシステムの監視。
  • 火災安全またはアクセス制御システム。
  • 独自のマイクロコントローラーのリモート制御。
  • 言い換えれば、シリアルインターフェースを使用したリモート作業が必要な場合は、インターフェースコンバーターを使用できます。


多くの人が質問します。なぜTibboデバイスはアナログよりも優れているのですか（このセグメントではこれで十分です）。 ほとんどの同様のデバイスで同一の技術仕様を詳細に比較することはしません。 Tibboコンバーターの重要な機能のみに言及します。

  • WindowsだけでなくLinuxでも動作する、便利で高品質で信頼性の高いソフトウェア。
  • 信頼性の高い仮想COMポートドライバー。
  • PCの参加なしで自律システム（デバイス-DS-DS-デバイス）を作成する機能。
  • Webベースの管理インターフェイスを介してコンバーターを構成する機能。
  • オプションとして、WiFiネットワーク802.11b / gを使用する機能。 したがって、ワイヤレスデータ送信によってシリアルポートからデータを送信できます。
  • オプションとして、Power over Ethernetテクノロジーの使用。
  • シリアルポートのオプトアイソレーション、IP68保護証明書、GPRSを介したデータ転送機能を備えた保護された機器ラインDS10xx 。
  • UDP、TCP、ARP、ICMP（PING）、DHCP、PPPoE、LCPおよびHTTPのサポート。
  • Telnetを介して構成する機能。
  • ADSLモデムの直接制御。
  • シリアルインターフェイスのマルチチャネル操作の可能性が実装されています（RS232インターフェイスを備えた最大3つのデバイスを1つのDB9コネクタで接続できます）。
  • コンバータをAggreGate Scada / HMIシステムに簡単に接続できます 。
  • 同業他社と比較して有利な価格設定ポリシーは言うまでもありません。


しかし、競合他社との主な違いは、Tibboコンバーターがプログラム可能であることです！ 指定された機能が十分ではなく、何か特別なものが必要な場合はどうなりますか？ これを実現する方法と、Tibbo TechnologyのDevice Serverの全機能を明らかにする方法を見てみましょう。



2. Device Serverをプログラミングするか、ModBusゲートウェイを作成します。



ModBus TCPプロトコルを使用してデバイスをポーリングする施設にScadaシステムがインストールされている場合はどうなりますか。 また、ボード上のModBus RTUプロトコルとRS232シリアルチャネルを使用してデバイスに問い合わせる必要があります。 従来のRS232インターフェイスコンバーターを接続することにより、ローカルネットワーク、さらにはインターネットにアクセスできますが、ScadaはModBusプロトコルを実装しないとデータを表示しません。

これは、Tibboコンバーターの主な機能が助けになる場所です-プログラムする機能。

プログラムは、イベントオブジェクト指向言語Tibbo Basicでエンコードされます。 簡単に言うと、 Tibbo Project Systemに関する記事でこれについて書いています 。 現在、「C」言語のコンパイラは準備ができており、まもなくリリースされます。 したがって、プログラミング言語-機能的で深い「C」またはよりシンプルで理解しやすいBasicを選択できます。

Tibbo IDEプログラミング環境は無料です。 プログラムやファームウェアのデバッグにプログラマは必要ありません。ステップバイステップのプログラム通過モードやブレークポイントなど、すべてがローカルイーサネットネットワークを介して直接行われます。

したがって、タスク（ModBus RTUを使用してデバイスをポーリングする）を実行するには、Tibbo機器用のModBusプロトコルを実装する必要があります。 幸いなことに、プロトコルはすでに作成されており、 パブリックドメインにあります 。 サイトからそれをダウンロードし、プロジェクトに接続し、デバイスをコンパイルしてフラッシュするだけで、ModBus TCPにModBus RTUゲートウェイを取得できます。

以下は、ライブラリを接続した後のソースコードでのデータ転送の様子です。

ソースコード： 
sub on_ser_data_arrival() ' ,       . dim x as string #if modbus_serial_mode = 0 'ASCII  x = ascii_to_tcp(ser.getdata(255)) '   ASCII  tcp #else 'RTU  x = rtu_to_tcp(ser.getdata(255)) '   RTU  tcp #endif sock.num = 0 sock.setdata(x) '       sock.send ' . end sub '-------------------------------------------------------------------- sub on_sock_data_arrival() ' ,      . dim x as string #if modbus_serial_mode = 0 'ASCII  x = tcp_to_ascii(sock.getdata(255)) '     ASCII #else 'RTU  x = tcp_to_rtu(sock.getdata(255)) '     RTU #endif ser.num = 0 ser.setdata(x) '       ser.send '  end sub ' ip ,     sub on_sys_init() ' ,       net.ip = "192.168.1.222" net.netmask = "255.255.255.0" net.gatewayip = "192.168.1.1" 'setup the Ethernet socket sock.num = 0 sock.protocol = PL_SOCK_PROTOCOL_TCP sock.inconmode = PL_SOCK_INCONMODE_ANY_IP_ANY_PORT sock.reconmode = PL_SOCK_RECONMODE_1 sock.localportlist = "502" sock.connectiontout=50 sock.rxbuffrq(2) sock.txbuffrq(2) 'setup the serial port ser.num = 0 ser.parity = PL_SER_PR_NONE ser.bits = PL_SER_BB_8 ser.flowcontrol = DISABLED ser.baudrate = ser.div9600 ser.interface = PL_SER_SI_FULLDUPLEX ser.interchardelay = 5 ser.rxbuffrq(3) ser.txbuffrq(2) sys.buffalloc ser.enabled = YES end sub




問題を拡大します。 シリアルデバイスはModBusプロトコルをまったくサポートしていないため、Scadaシステムに接続したいと思います。 次に、標準のRS232インターフェイスを使用して必要なデバイスメトリックの調査をプログラムし、ModBus TCPコードを取得して、デバイス専用のハードウェアドライバーを取得します。

タスクをさらに拡張できます-接続が切断された場合にデータをバックアップする必要がありますか？ 1 MBの内蔵フラッシュメモリを使用し、接続が復元されるまでデータをオフラインで保存します。 Webベースの構成が必要ですか？ 組み込みのWebサーバーをプログラムし、指定された機能を取得します。

したがって、独自のハードウェアデバイスドライバーなどを作成することで、任意のプロトコルを実装できます。 独自の小さなPLCを取得して、ロジックをプログラミングできます。



3.おでこを少し練習したり加熱したりします。



かなり興味深い仕事に出くわしました。 民家では、暖房システムが設置されています-2つの電気温水ボイラー。 ボイラー出口とパイプラインの逆分岐には、水温センサーが設置されており、システムには過圧センサーと屋外温度センサーもあります。 すべてのセンサーはMB485アナログ入力モジュール（ARIES）に接続され、RS485インターフェースを介して「世界と通信」します。 タスクは、センサー読み取り値のリモート監視です。 理想的には、システムはMySQLデータベースに値を書き込んで、顧客の個人Webサイトにさらに情報を表示する必要があります。 DS1102デバイスを選択しました 。

最初にすることは、MBA8モジュールのデータ転送プロトコルを実装することでした。 プロトコルは非常にシンプルで、コーディングに数時間を費やしました。その結果、get_data_sensor（）を含むアナログ入力モジュールを操作するためのいくつかの便利な機能が得られました。 次のステップは、MySqlプロトコルを実装することでした。 それからもう少し時間を過ごしました。 その結果、このタスクは数時間で解決されました。 完成したMySqlライブラリは 、製造元のWebサイトで公開されています。

これで、DS1102はMySqlデータベースにアクセスできるようになりました。MBA8をポーリングできます。センサーをポーリングしてデータベースに値を書き込むための最も簡単なロジックを使用します。

すべてが機能し、データがMySqlテーブルに正常に転送されます。 顧客は満足し、実装が成功したことに気付いたとき、突然「部屋の温度をリモートで制御できますか？」と尋ねました。 必要に応じて、それを行います。



リレーを使用してボイラーを制御します。


ボイラー電源回路では、メーカーは外部の低電力リレーを使用してヒーターをオンにする可能性を提供しました。 この方法は、負荷電力制御を意味しないため、実装が非常に簡単です。 これは、温度制御がPIDコントローラーを使用せずに「額で」行われることを意味します。 しかし、同時に、大きな規制上の誤差が蓄積します（過熱、その後冷却）。 これは顧客に適していました。 しかし、別の問題がありました。 DS1102モジュールでは、リレーはありません。 ただし、リレーは「変更された」デバイスサーバーデバイスDS1014にあります。 このモデルにはリレーだけでなく、アナログ入力、ビルトインGPRSおよびWiFiもあります。



少し気が散る。 実際、Tibbo Device Serverデバイスは、いわゆる埋め込みモジュールに基づいています。これについては、以降の記事で説明します。 これらのモジュールは、プログラム可能なマイクロコントローラーと見なすことができます。 COMポートを備えたコンストラクトに埋め込みを使用すると、インターフェイスコンバーターが得られます。 ただし、単純な入力/出力モジュールをすでに受け取っているため、ADCとDACをボードに追加する価値があります。 これが、DS1014モデルの作成方法です。 Tibboの「異なる」ハードウェア用に作成されたプログラムは、実際にはそれらの間で簡単に移植できるという結論を出すのは難しいことではありません。 したがって、1つのTibboデバイスを別のデバイスに交換するのは簡単です。



また、顧客はこの変更に同意しました（すべての顧客がそのような顧客を持っているでしょう？）。 さて、既存のコードを書き換える必要はありません。 仕事に取り掛かる。 屋外温度と特定の値との不一致を調べ、許容されるデルタよりも大きい場合は、ヒーターのオンとオフを決定します。 回路内の水圧が臨界レベルを下回りました-ボイラーをオフにします。

ソースコード： 
 sub on_sys_timer …... If control = true then '   ,   ,  ,    ( ) If mva8.temperature_air>temperature_set+delta then boiler(on) If mva8.temperature_air<temperature_set-delta then boiler_on(off) End if If mva8.pressure<pressure_min then '     ,      . boiler(off) Control = false Else Control = true End if End sub public sub boiler (on_off as string[3]) '  ,    2 ,   . select case on_off case “on”: io.num=PL_IO_NUM_13_TX2 io.state=LOW io.enabled=Yes io.num=PL_IO_NUM_3 io.state=LOW io.enabled=Yes case “off”: io.num=PL_IO_NUM_13_TX2 io.enabled=NO io.num=PL_IO_NUM_3 io.enabled=NO end select end sub




リレーがクリックすると、部屋は暖かく快適です。 確かに、実際のアルゴリズムでは、温度が境界状態にあるときの「バウンス」を回避するために、ボイラーのオン/オフ間の遅延も提供します。 オブジェクトでの重要なイベントのデータはMySQLに転送されます。 しかし、ユーザーに部屋の必要な温度値を迅速に設定する機会をまだ与えていません。 最新のTibboモジュールにはすべてWebサーバーが組み込まれているため、Webインターフェイスを使用してこれを簡単に実装しました。 同時に、メーターのオンライン監視を提供しました。

SMS管理


素晴らしい、すべてがブラウザを介して設定されます。 顧客は満足しています。 しかし、私は止められません。 SMS経由で温度を設定すると便利です。 たとえば、忙しい一日を過ごして帰宅したら、SMSをシステムに書き込んで、家の中の温度を25°Cに上げます。 ただし、DS1014モデルにはGPRSモデムが組み込まれており、コントローラーの4つのシリアルポートの1つを介して接続されています。 AT＆Tチームと少し協力しましょう。

ソースコード： 
 sub on_ser_data_arrival() … Select case ser.num … case 1: '-----        gsm ----- gsm_extract(ser.getdata(ser.rxbuffsize)) ... end select End sub public sub gsm_extract(buff as string) … select case prefix … case "+CMGR": i=instr(1,buff,chr(44),1) j=instr(1,buff,chr(44),2) cmpphone=mid(buff,i+2,ji-3) if cmpphone=phone then j=instr(9,buff,chr(13)+chr(10),1) data=mid(buff,j+2,len_buf-j-2-3) gsm_get_command(data) '     . end if in ="AT+CMGD="+num+chr(13) if val(num)>10 then in="AT+CMGDA="+chr(34)+"DEL ALL"+chr(34)+chr(13) '     10   ,    SIM . ser.num=ser_gsm ser.setdata(in) ser.send case "error": …. end select End sub




ご覧のように、コードでは、コマンドの発信元の電話番号と指定された電話番号を比較しています。 gsm_get_command（）関数-入力でSMSテキストを受け取り、そこからコマンドを抽出します。 たとえば、テキスト「SET TEMP 25」を含むSMSは、25℃の領域で温度維持を設定するコマンドを提供します。

しかし、パイプラインがブレークスルー（圧力が急激に低下）したらどうなるでしょうか？ この場合、システムはヒーターをオフにする方法をすでに知っていますが、所有者に警告する必要もあります。

ソースコード：
