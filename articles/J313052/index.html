OpenVPNを準備しています。 パート1.静的キー

OpenVPN接続の確立中に理解できないエラーが発生し、内部での配置方法についての理解が不足していると感じました。 猫の下には、プロトコルの暗号化部分がどのように配置されているか、実際にどのように見えるか（つまりWiresharkで）、VPNの内部を見る方法、つまり トラフィックを自分の手で解読します（キーがある場合はもちろんです;-)）。 この部分では、静的キーを持つモードのみを考慮します。









動作モード



プラグインなしのOpenVPN自体は、接続を確立するための2つのオプションをサポートしています。







  • PSK （PreSharedKeyまたは静的キーモード ）-VPNセットアッププロセス中に生成され、すべてのマシンに書き込まれる対称キーを使用します。 記事の最初の部分では、このモードのみを扱います。
  • TLSは公開鍵インフラストラクチャ（PKI）に基づいています 。 最初に、TLS接続が確立されます。TLS接続は、証明書による両当事者の認証を使用して確立され、それを介して鍵素材が交換されます。 対称キーはこのキーマテリアルから生成され、VPN内部のトラフィックはPSKモードと同じ方法でこれらのキーで暗号化されます。 TLSチャネル内にはユーザートラフィックはありません。OpenVPN自体のサービス情報のみです。 このモードでのOpenVPNの準備に関する第2部を書く予定です。


PSKモードの詳細



PSKモードは、 シークレットオプションによって有効になります。その最初のパラメーターは、VPNセットアップフェーズ中に生成される静的キーを持つファイルの名前です。 このファイルには、2組の512ビットランダムキー（各ペアで暗号化用のキーとHMAC用のキー）が含まれ、次々に（ヘッダーなどなしで）書き込まれ、HEXでエンコードされます。 VPNで接続されているすべてのコンピューターに同じキーファイルが存在する必要があります。 2番目のオプションパラメーターであるdirectionがsecretオプションに設定されていない場合、最初のキーペアのみが使用されます。つまり、トラフィックは両側で同じキーで署名および暗号化されます。







静的キーモードでは、 CBCおよび暗号化なしのデータパケット暗号化形式のみを使用できます。 次に、 CBC形式を検討します。 暗号化はそれほど興味深いものではありません 。







デフォルトでは、 BlowFishは 128ビットキーを使用したCBCモードでの暗号化に使用され、 HMACにはSHA1および160ビットキーが使用されます。 キーは、静的キーを持つファイルの512ビットキーの最初の128ビットと160ビットです。







すべてのキー情報と設定は事前に設定されているため、リモートホストのアドレス（ リモートオプション）を指定すると、カプセル化されたトラフィックは、制御パケットの交換前でもOpenVPNを起動した直後に送信できます（詳細は以下を参照）。

準備中です



送信されるパケットの形式は次のとおりです。 







 [Len (2 )] | HMAC (20 ) | IV (8 ) | packet ID (4 ) | timestamp (4 ) | packet payload


Len-最初の2バイトを除くパケット全体の長さ。 TCP経由で作業する場合にのみ使用されます。 UDPを使用する場合、OpenVPNパケットの長さはUDPパケットの長さから計算されますが、OpenVPNパケットのLenフィールドは計算されません。

HMAC -HMACの後に来るすべてのHMAC-SHA1（IVおよび暗号化されたデータ）

IV -CBCモードの初期化ベクトル。

IVの後に来るすべてが暗号化されて送信されます。

パケットID-パケット番号。 ここには繰り返しを防ぐために含まれており、保証されたデータ配信を整理するためには使用されません。

タイムスタンプ -繰り返しに対する保護にも使用されます。

パケットペイロード -パケットが運ぶデータ。 これは、OpenVPNオーバーヘッドまたはカプセル化されたトラフィック自体です。







HMACとは何ですか？

HMAC （キー、メッセージ）-認証コード（メッセージの署名として機能するビット列）を生成する暗号化ハッシュ関数（この場合、SHA1に基づく）に基づく関数。 秘密鍵 、メッセージ、および認証コードを使用して、誰でも認証コードが正しいことを確認できます。 コードが正しい場合、 同じキーを持つ誰かによって署名された時点からメッセージ（この場合はパッケージ）が変更されていないことを意味します。 つまり HDSをチェックするには、EDSとは対照的に、両方の当事者が同じ秘密鍵を持っている必要があります。

備考



OpenVPN Ping



pingパケットもあります-最後のパケットが別のホストに送信されてから時間が経過した場合に接続を確認するためにOpenVPNが送信するサービスパケット（キープアライブ、非アクティブ、ping *オプションで設定）。 通常のicmp pingとは異なり、OpenVPNは受信したパケットに応答しませんが、接続が生きていることを単に覚えています。







nmapとOpenVPN



OpenVPNがPSKモード（およびTLS-AUTHオプションを有効にしたTLSモード）でリッスンするUDPポート。秘密鍵を知らないすべての人にとって、その上のすべてのパケットがドロップしているように見えます。 nmapは、このポートのステータスを「open | filters」と表示します。 OpenVPNログ（verbオプションの値が十分に高い）で、ポートをスキャンすると、「パケットの認証/復号化エラー：パケットHMAC認証に失敗しました」というエントリがあります。







OpenVPNトラフィック検出



PSKモードでは、UDPを使用して作業する場合、サードパーティのオブザーバー（秘密鍵を持たない）のOpenVPNパケットのコンテンツ全体がランダムに見えます。 これらのパッケージをOpenVPNに起因させる署名はありません。 ただし、パケット長はランダムデータイメージから外れます-使用されるハッシュ関数と、使用される暗号のブロックサイズに等しい増分の変化、および固定長サービスパケット（OCC_REQUESTおよびOpenVPN PINGパケット）の存在に依存します。







さらに、TCPを使用して作業する場合、各OpenVPNパケットの先頭には、暗号化されていないパケット長の2バイトフィールドがあります。







おわりに



PSKモードでは、すべてが非常に単純に配置されていることがわかります。 実際にはクライアント/サーバーの分離さえありません。
