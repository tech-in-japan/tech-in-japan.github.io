マイクロサービスアーキテクチャ用のLAppSアプリケーションサーバー







背景



昨年の12月20日、私は2週間も休暇を取りました。 休暇中にすべきこと そうです-コード付き。 営業時間中に行う時間がないコード。 ここ数年、私はコーディングをほとんどしていません。 手をつないでいた。 彼らは休暇中にどのようなコードを書きますか？ あなたのことは知りませんが、自転車を書きます。 なんで？ 多くの理由があるかもしれませんが、主な理由は私にとって興味深いものです。 C ++とLuaが大好きです。 bashとawkも大好きです。 石を投げないでください、それは個人的なものです、それは起こりました。 私はJavaScriptがあまり好きではありません（ただし、JSで何かがエンコードされた場合、過去2年間）、これも個人的なものです。







最終的に何が起こったのか



休暇中の退屈の結果は、 LAppS-Lua Application Serverでした。 この休暇のコーディングは6か月間続きました（もちろん、12月以降、コードにあまり時間をかけませんでした。これはgithubのコミットからわかります）。 しかし、過去2週間で、何かを機能させるために多くの時間を奪いました。







これは何ですか



名前が示すように、 LAppSは Luaアプリケーションサーバーです。 Luaは非常に人気のある言語であり、Nginx + LuaとOpenRestyライブラリの組み合わせが世界中で積極的に使用されており、Cloudflareには価値があります。 開発の初期には、前述のNginxおよびTarantool用のLuaモジュールであるluvit.ioがすでに存在していました。 しかし、それらのどれもWebSocketをサポートしていませんでした。 LAppSはHTTPをサポートしていません。 しかし今では、 LAppSはuWebSocketsよりもパフォーマンスに優れています（大規模なコンピューティングリソースを消費します）。







主なアイデアは、マイクロサービスの開発サイクルを最小限に抑えることでした。 Luaのエントリのしきい値はJavaScriptよりも低くなっています。 中学生でも簡単にLuaでプログラミングを始めることができると確信しています。 しかし、Lua用のWebアプリケーションを開発するために利用できるすべてのツールは次のとおりであり、そのような最小エントリーしきい値はありません。







したがって、開発における主な重点は、最小限のAPIとアプリケーション作成の容易さ、およびアプリケーションサーバーとサービスの構成の容易さでした。







詳細



LAppSの Luaアプリケーション（サービス）はI / Oをブロックしません。 これはおそらく、Luaでスクリプトを作成するWebサーバーとの最大の違いです。 LAppSは2つの構成ファイルを使用して、WebSocketsサーバーの動作を構成し、アプリケーションを展開します。







デフォルトでは、構成ファイルがない場合、LAppSはディストリビューションで実行されているデモアプリケーションの実行を試みます。







ws.json-WebSocketsサーバー構成ファイル 



{ "listeners" : 1, "connection_weight": 0.7, "ip" : "0.0.0.0", "port" : 5083, "workers": { "workers" : 3, "max_connections" : 1000 }, "tls" : true, "tls_certificates" : { "ca" : "/opt/lapps/etc/ssl/cert.pem", "cert" : "/opt/lapps/conf/ssl/cert.pem", "key" : "/opt/lapps/conf/ssl/key.pem" }, "auto_fragment" : true, "max_inbound_message_size" : 300000 }


  • listeners-並行して起動されるリスナーの数。このパラメーターは、 LAppSが着信接続を受け入れる速度に影響します
  • connection_weight -IOWorkersのI / Oキューの深さと接続数を比較する内部バランサーのパラメーター。 同時接続が多いほど、このパラメーターは少なくなります。 接続は、異なる負荷プロファイルで異なるサービスを使用できます。 このような状況では、接続数の少ないIOWorkerよりも、新しい接続に負荷の少ないIOWorkerを使用することをお勧めします。
  • ip-サーバーが着信接続を待機するインターフェースのIPアドレス。 デフォルトでは、すべてのインターフェース。
  • ポート -ポート。 デフォルトは5083です。
  • workers.workers-同時に動作するIOWorkersの数。 デフォルトは3です。
  • workers.max_connections-まだ使用されていません。 将来的には、最初のIOWorkerのアクティブな接続の制限を設定します。

    • tls -TLSを使用しますか？ デフォルトはyesです。 一般的に、このパラメーターは無意味です。 TLSがビルドパラメータによって決定されるかどうかを使用します。 デフォルトでは、サーバーはLibreSSLおよびTLS 1.2を使用して構築されます。
  • tls_certificates.ca -LibreSSLに同梱されている証明書へのパス。
  • tls_certificates.cert-証明書パス（デフォルトはlocalhost自己署名証明書です）
  • tls_certificates.key-証明書キー
  • auto_fragment-送信メッセージに自動断片化を使用するかどうか（まだ有効ではありません）
  • max_inbound_message_size-着信メッセージのサイズ制限（まだ有効ではありません）


lapps.json-サービス構成ファイル 



 { "directories" : { "applications" : "apps", "app_conf_dir" : "etc", "tmp": "tmp", "workdir": "workdir" }, "services" : { "echo" : { "internal" : false, "request_target" : "/echo", "protocol" : "raw", "instances" : 3 }, "echo_lapps" : { "internal" : false, "request_target" : "/echo_lapps", "protocol" : "LAppS", "instances" : 3 } } }


  • directory.applications-環境変数LAPPS_HOMEに関連するアプリケーションディレクトリへのパス（デフォルト/ opt / lapps）、-デフォルトアプリ
  • directory.app_conf_dir-サービス構成ファイルへのパス（まだ使用されていません）
  • *ディレクトリ{tmp、workdir}-使用されません
  • services-サービス設定のマップ


上記の例では、echoとecho_lappsの2つのデモサービスが構成されています。 サービスの名前はデフォルトであり、Luaアプリケーションモジュールの検索パスです。 本質的に、 LAppSのアプリケーションは、特定のインターフェイスに従うモジュールです。







内部パラメーターに加えて、他のすべてのサービス設定が必要です。







  • services。{name} -例のservices.echoおよびservices.echo_lappsでは、サービスの名前。
  • services。{name} .request_target -WebSockets URLのターゲット、例wss：//127.0.0.1：5083 / echowss：//127.0.0.1：5083 / echoまたはwss：//127.0.0.1：5083 / echo_lapps。 ハンドシェイク後のIOWorkersは、この目標に基づいてソケットを特定のアプリケーションに関連付けます。 サービスマップにクライアントが要求した目標がない場合、ハンドシェイクはコード403で中断します。
  • services。{name} .protocol-アプリケーションプロトコルのタイプ。 現在、2つのタイプのプロトコルのみが使用されています：rawおよびLAppS（それらについては以下）
  • services。{name} .instances-同時に実行されているアプリケーションインスタンスの数。 各接続は、独自のアプリケーションインスタンスに割り当てられます。

アプリケーション/サービス



アプリケーションは実際にはLuaモジュールであり、動作が事前定義されたいくつかの事前定義メソッドが必要です。







  • onStart-アプリケーションが開始する前に1回実行されるonStartメソッド。 ここで、初期設定を実行し、アプリケーションの初期化に重要なコードを実行できます。 この方法では、Lua仮想マシンを無期限にブロックしないでください。 つまり このメソッドでwhile（1）ループを実行すると、サービスは着信要求に応答しなくなります。
  • onShutdownは onStartに似ており、アプリケーションが停止したときにメソッドが1回実行されます。 また、Luaマシンの実行スレッドを際限なくブロックしてはなりません。 そうでない場合、サービスはそれ自体を停止せず、アプリケーションサーバーの停止を許可しません。
  • onMessage -rawプロトコルとLAppSプロトコルに対して別々に実装されます。 構成で指定されたパラメーターに応じて、このメソッドはさまざまなタイプのデータを受け取ります。

    • raw： bool onMessage（handler、opcode、message） -ここで、 handlerは接続の一意の識別子です。 opcode -WebSockets OpCodeは、1（TEXT）または2（BINARY）の2つの値のみを取ります。他のオペコードを持つすべてのフレームはサーバーによって処理され、アプリケーションには転送されません。 message-バイナリまたはテキストメッセージを含むLua文字列値（フレームの送信に使用されたオペコードによって異なります）。
    • LAppS： bool onMessage（handler、msg_type、message） - ハンドラーは一意の接続識別子です。 message -LAppS 1.0仕様に準拠する実際のメッセージ。これは、nljson型のユーザーデータオブジェクトです。 msg_type-補助パラメーター、メッセージタイプは、1〜4の4つの値を取ります。

      • 1-クライアント通知（CN）
      • 2-params配列に追加パラメーターを持つCN
      • 3-パラメーターなしのメソッドの実行の要求
      • 4-params配列のパラメーターを持つメソッドの実行の要求
  • onDisconnect（handler） - クライアントが切断する（ RFC 6455に従って正しくリセットまたは閉じる）ときに呼び出されるメソッド。 ハンドラ-接続の一意の識別子。


注： onMessageメソッドはブール値を返す必要があります。 値を返すとき、それはfalseであり、メソッドが呼び出された接続は壊れています（コード1000を閉じます）。







エコーサーバー（rawプロトコル）を実装する最も単純なスケルトンアプリケーション。 



 myapp = {} myapp.__index = myapp; myapp["onStart"]=function() -- do something on start end myapp["onDisconnect"]=function(handler) -- handler - is a unique client identifier -- react on client disconnect end myapp["onShutdown"]=function() -- do something on shutdown end myapp["onMessage"]=function(handler,opcode, message) -- it is an echo, - we return back the same message local result, errmsg=ws:send(handler,opcode,message); if(not result) then print("myapp::OnMessage(): "..(errmsg or "none")); end return result; end return myapp;


このサービスの構成： 







  "myapp" : { "internal" : false, "request_target" : "/myapp", "instances" : 1, "protocol": "raw" }


LAppSプロトコル



LAppSプロトコル仕様は、Google JSON-RPC仕様に基づいており、次の重要な違いがあります。







  • CBOR形式のバイナリメッセージング
  • プロトコルは、サーバーによって開始されたメッセージのタイプである「Out of Order Notifications」（OON）を指定します。
  • プロトコルはメッセージチャネルを指定します。 要求/応答メッセージの必要性、およびサーバー側の通知の必要性を考慮して、チャネルを使用すると、クライアントアプリケーションは同じ接続の異なるチャネルで並列データストリームを使用できます。 チャネル0（CCH）は、要求/応答メッセージ用に予約されており、他のすべてのチャネルはアプリケーションによって合意されています。


仕様はgithubで入手できます

自動ロードモジュール



LAppSは、サービスを開始する前に、 nljson 、 ws 、 bcastのいくつかのモジュールをロードします。 モジュールの仕様の詳細は、プロジェクトwikiにあります。







簡単に：







  • nljson -JSON（ecode / decode / cbor）を操作するためのモジュール。


モジュールでの作業は、ネイティブLuaテーブルでの作業と大差ありません。 さらに、Luaテーブルは、単純な割り当てでnljsonユーザーデータに変換されます。 ただし、Luaはオブジェクト（キー値）と配列を区別しないため、たとえば、空のLuaテーブルの場合、それらをnljsonに変換することは障害です。 例えば 







  local object=nljson.decode({})


これにより、オブジェクトという名前のJSON配列が作成されます。 したがって、この初期化を使用することをお勧めします。 







  local object=nljson.decode('{}')


この定義により、JSONオブジェクトが一意に作成されます。







さらに、このオブジェクトはネイティブluaテーブルとして使用できます。 







  object["test"]=""; print(object.test) object["map"]={ ["key1"] = "value", ["key2"] = 33 } print(object)


nljsonオブジェクトの速度は、ネイティブのLuaテーブルとほとんど異なりません。







  • wsはWebSocketメッセージを送信するためのモジュールで、send、closeの2つのメソッドしかありません。
  • bcast-ブロードキャストメッセージのモジュール、利用可能なメソッド：サブスクライブ、サブスクライブ解除、作成、送信。
