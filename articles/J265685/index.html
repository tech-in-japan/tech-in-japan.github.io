Rebolの後継者であるRedについて、Nenad Rakocevicにインタビュー

これはモナドのチュートリアルではないという ブライアンマッケナとの最後のインタビューの後、 Redプログラミング言語の作成者であるNenad Rakocevicにインタビューしました。



私の主観的な観点から、赤とレボルはかなり奇妙な生き物です！ 誤解しないでください、悪いことを意味するものではありません。 たとえば、標準的な低レベルプログラミング用の埋め込みDSLや50種類の組み込み型などの機能を備えた高レベル言語はあまり知りません。 試してみてください。Redでのプログラミングには多くの興味深いアイデアがあります。



@unbalancedparenにツイートしてください。これはMonadのチュートリアルではありません。 私たちと一緒にいてください！











赤の出現について少し教えてください。 それは何のために作成されたのですか？





私の場合、若い頃に、Amigaというマイクロコンピューターのプログラミングを始めました。 私は30年以上プログラミングを続けています。 最初の経験の後、私は既存のプログラミング言語とツールに不満を感じました。 主に、彼らが私にとって十分に生産的でも友好的でもなかったという事実による。 それで、1999年にRebolに出くわしたとき、彼はいわゆる「現代の」開発慣行のどこが悪かったのかに目を開いた。 （今日はさらに悪いです）。 ソフトウェアのあらゆる面で複雑さを克服することは、論理的な行動方針になっています。



2010年、Rebolソースコードは閉鎖されました。 彼のアプローチはもっと多くのことができると強く感じましたが、Rebolは実際には発展しませんでした。 これは、オープンソースプログラムであるRebolの親onで作業を開始する合図でした。



赤の主な議論は何ですか？



  • これは最初のフルスタックプログラミングソリューションです。1つのツールに統合され、高レベルのコード（GUIアプリケーション、スクリプト、DSL）と高速の低レベルのコード（デバイスドライバー、オペレーティングシステム、ネイティブインターフェイスなど）を記述できます。 さらに、Redは双方向のテクノロジーでもあります（クライアントおよびサーバー用）。

  • ネイティブコードへのクロスプラットフォームコンパイラ：ツールチェーンをサポートするプラットフォームでは、単純なコマンドラインオプション（-t Windows、-t Linux、-t Darwin、-t RPi、...）を使用して15以上のプラットフォーム用にコンパイルできます。

  • 非常に軽量：赤は1 MBの単一ファイルであり、インストール、構成、ツールチェーン全体ではありません。 ほんの数秒でダウンロードでき、すぐにコードの記述と実行を開始できます。設定するものはまったくありません （これは例外であり、標準ではないのは恐ろしいことです...）。

  • バッテリーソリューション：小さなサイズでありながら、通常のタスクに必要なほぼすべてをカバーする、非常に豊富なランタイムライブラリが含まれています。

  • DSL中心の環境：Redには、特定のニーズ（GUIやシステムプログラミングなど）向けに設計された多くの組み込みDSLが付属しています。 DSLは、生産性を劇的に向上させながら、フレームワークまたはAPIに起因する複雑さを軽減する非常に強力な方法です。 赤には、DSLを作成するためのDSL（Parseと呼ばれる）が含まれています。

  • 赤（Rebolのような）はLispの派生物ですが、人間の構文（括弧地獄なし）を備えています。 赤は、それ自体のデータ形式です。 すべてのコードは、実行を開始するまでデータと見なされ、シリアル化コード/データは無料で提供されます。

  • Red（およびRebol）の背後にある哲学は、単純なものを可能にし、困難なものにすることです。


RebolがRedの主なインスピレーションになったのはなぜですか？



Rebolは、過去20年間に作成された最も革新的なプログラミング言語の1つです。 残念ながら、Perl、Python、Rubyなどのオープンソース言語が業界に進出している一方で、ソースコードを閉じていたため、範囲外のままでした。 Rebolアプローチは、プログラマーが開発において「シンプル」または「効果的」と考えるもののまさに基礎を損ないます。 一般に、他の言語では「シンプル」と呼ばれるAPIは、Rebolのメガネを通して世界を見ることに慣れていると無意味に複雑に見えます。 ;-)例のいくつかのワンライナーがあります（Rebol2 REPLの場合）：



コンソールに「Hello」をクリックするボタンを持つグラフィックウィンドウを作成します。 



view layout [button “Click Me” [print “Hello”]]




Webページのコンテンツをコンソールに印刷します。 



 print read http://rebol.com




Webページの名前を取得します。 



 parse read http://rebol.com [thru <title> copy text to </title> (print text)]




現在のディレクトリのファイルのリストを電子メールで送信します。 



 send user@domain.com mold read %.




MySQLからすべてのレコードを取得して印刷します。 



 foreach row read/custom mysql://root@localhost/books [“SELECT * FROM authors”] [print row]




Rebolコードを見たことがない場合でも、それを読んで、そのほとんどが何をするのかを理解できることに注意してください。



RebolとRedの主な違いは何ですか？



  • 赤は（クロス）ネイティブコードにコンパイルできます。Rebolは解釈されるだけです。 コンパイルされたコードは、解釈されるよりもはるかに高速に実行されます。
  • 赤は、システムのプログラミングと高速で低レベルのコードの記述を可能にし、Rebolはスクリプトレベルでスタックしています。
  • RedはネイティブウィジェットとネイティブGUIサポートに依存しており、Rebolには独自のエンジンがあります。 Redを使用すると、グラフィックスアプリケーションはより自然に見え、ユーザーのOSにより良く統合されます。


さらに、言語自体は非常によく似ており、約95％です。 Rebolを知っていれば、Redを知っています。



Redは低レベルと高レベルのプログラミング間の抽象化のクラス全体をカバーし、CのようなセマンティクスとRebolのような構文を持つRed / System方言を提供します。 元のデザインに赤と赤/システムの違いはありましたか？ Red / Systemを使用してどのようなメリットがありましたか？



もちろん、Red / Systemは、Rebol実装を複製するのではなく、新しい言語スタックを作成する主な理由の1つでした。 Red / Systemは、ネイティブコード（Cなど）用に設計された静的にコンパイルされた言語です。 赤/システムには、主に2つの目的があります。



  • Redユーザーに組み込みDSLを提供して、高速コードとシステムプログラミングのニーズをサポート
  • 赤でコンパイルされたコードの中間表現言語である


私の知る限り、Redはユニバーサル低レベルプログラミング用の組み込みDSLなどの機能を備えた最初の高レベル言語です。



Red / Rebolを使用しているのは誰ですか？



Rebolコミュニティは2000年代前半には比較的大きかったが、開発の停止により劇的に減少した。 多くのユーザーは、スキルスケールで驚くほど多様でした。 多くの人は、シンプルなタスクのシンプルさとGUIのクロスプラットフォームエンジンが好きで、他の人は深さに興味がありました（動的リンク、DSLの簡単な作成、本格的なメタプログラミング機能など）。



それ以来、システムを構築してきた筋金入りのファンや企業のみが、それを引き続き使用および宣伝しています。 多くの点で、これらは同じ人々であり、Rebolに興味を持っているがその所有権の性質を好まなかった他の多くの人々とともにRedの最初の支持者を形成しました。 それらの一部は、Red用のライブラリを作成し、その他は小さなおもちゃ、またはWindowsデバイス用のドライバーを作成します。 :-) Redの生産準備が整い次第、より多くの人々がRedに参加してRedの使用を楽しんでくれることを確信しています。



Redはどのようなタスクに適していると思いますか？



Redは、 あらゆるプログラミングタスクに十分な汎用ソフトウェアソリューションです。 実際には、これは利用可能なフレームワークとライブラリに（のみ）制限されています。 したがって、Redはこれらのタスクに非常に適しています。



  • スクリプティング/コードバインディング
  • グラフィカルインターフェイス（今後のv0.6.0で）
  • Androidアプリケーション（v0.6.1）
  • データ処理
  • 文法パーサー/ DSL作成
  • システムプログラミング
  • デバイスドライバー
  • モノのインターネット用のプログラミングデバイス（IntelまたはARMを搭載）


1.0（来年）に達すると、Redは以下にも非常に役立ちます。



  • Webアプリケーション
  • サーバー作成
  • 2Dゲーム
  • ロボティクス


RebolとRedは、実際のアプリケーション向けに多種多様な組み込みタイプを提供します。 一部の人々は、言語機能の小さなコアを持つ方が良いと考えています。 これについてどう思いますか？



RebolとRedには50を超えるデータ型があり、ランタイムは500 KBを超えています。 そのうち、3分の2には特別なリテラル形式（例：お金、メール、URL、時間、日付、色など）があり、すぐに組み込みDSLを作成するための豊富なリテラルセットを提供します。



もう1つの利点は、日常のタスクに必要な機能のほとんどが既にここにあることです。他の機能と完全に統合された一流の施設は、サポートされる各プラットフォームでまったく同じように機能します。 これにより、生産性が大幅に向上し、学習/言語の使用がさらに楽しくなります（単純なタスクごとに「インポート」する必要はありません）。 このような言語は実用的であり、ソフトウェアの作成コストを削減することを目的としています。



Redは一般的にどのように実装されていますか？ すべてのコンポーネント（パーサー、コードジェネレーター、ガベージコレクターなど）は手で書かれていますか？ Redの依存関係は何ですか？



スクリプトによって生成された単体テストの大部分を除いて、他のすべては手で書かれています。 Rebolを使用してRedをデプロイするため、ツールチェーン（コンパイラー、リンカー、ビルダー）はRebol2で記述されています。 Rebolには解析用のDSLがあります。これは非常に効率的であり、強力なメタプログラミング機能と組み合わせることで、他のツールを使用する必要がなくなります。 Redスクリプトは、REPLから解釈するか、Red / Systemを中間表現として使用してネイティブコードにコンパイルできます。 ランタイムライブラリは、RedとRed / Systemのコードの混合で構成されています。



赤の実行可能ファイルは通常、サイズが約0.5MBであり、依存関係はありません。



2015年半ばのレッドはどれくらいいっぱいですか？



すでに多くが実装されているので、まだ足りないものを説明した方が良いでしょう。 現在、Windows用の最初のバックエンドでクロスプラットフォームGUIのサポートを完了しています。 Android、Linux、OS Xのバックエンドは次のとおりです。 I / Oは、これまでのところ単純なファイル操作とHTTPクライアントに限定されています。 モジュラーコンパイル、本格的なガベージコレクター、および競争力のサポートは、リリース1.0以前の重要な目標です。 2016年に1.0をリリースする予定です。



レッドの未来をどう見ていますか？



Redには、既存のツール（いわゆる「単純な」ツールでさえも）に満足していない多くの開発者（特に選択の自由がある無料の開発者）を誘惑する可能性があります。 Redは今後2、3年で広く普及し、プログラマーがさまざまな問題を喜んで解決し、人生を楽にするのに役立つと期待しています。 Redは、複数のドメインにわたって多くの強力なDSLを使用して拡張し、既存のライブラリを直接使用するための優れた代替品を提供します。 たとえば、ロボットやAIの分野で推進していきます。



レッドの開発から得た最も重要な結論は何ですか？



  1. オープンソースは、高品質のソフトウェアを作成する最良の方法です（レッドはこの事実を再確認しました）。
  2. 「オープン」で作業することは必ずしも良いことではありません。時には、困難なタスクを完了するために、周囲の「ノイズ」から自分を隔離する必要があります（主に設計上の問題）。 プロジェクトが成長するにつれて、それはますます困難になります。
  3. 成長しているユーザーのコミュニティと対話する必要性は、 多くの時間を消費します。 このタスクのための人々を見つけることは非常に重要です。
  4. 優れた構文の設計は、優れたセマンティクスの設計よりもはるかに複雑です。 多くの言語作成者はこの点を見逃しているため、優れたセマンティクスですが、ひどい構文になります。
  5. 静的に型付けされた言語のネイティブコードにコンパイラを書くことは実際難しくありません。コンピューターサイエンスの知識がほとんどないプログラマーのほとんどはこれが可能であり、何ができるかわかりません。
  6. 時期尚早の最適化は（多くの場合）背後から攻撃される可能性があります。 最適化が時期尚早であることを知ることは、ブラックマジックのカテゴリーに由来します。
  7. 大規模なプロジェクトは、少なくとも2人の非常に意欲的で同等の資格を持つ開発者のチームで開始する必要があります。 大規模なプロジェクトで単独で作業するのはおかしく、より良い結果を保証するものではありません。
  8. 十分に魅力的でフォロワーを獲得できるオープンソースプロジェクトに取り組んでいる場合、ユーザーからの寄付で生活することができます（基本的な生活費はすべてこのように2年間住んでいます）。 私は始めたときにこれが可能だとは思わなかったし、まったく期待していなかった。 たぶん、Redがこんなに素晴らしいコミュニティを持っているのは幸運だと思う。


最初のプログラミング言語を書くために読むことをお勧めしますか？



すべての必要なものの概要は、 Modern Programming Languages：A Practical Introductionのような本にあります。 より深く、より抽象的なカテゴリーに飛び込みたい場合は、 The Dragon Bookが最高です（ ウィリアムズは最近、最新版の翻訳を翻訳者のメモとしてリリースしました） 。



しかし、最も便利な方法は、いくつかの小さな言語の実装を学習することです。これにより、これを自分で行う方法をよりよく理解できます。 たとえば、Red 0.1.0のリリースはzipアーカイブでは24 KBのみですが、Red / System用の有効なコンパイラー/リンカーと多くの機能（FFIを含む）があります。 ここからダウンロードしてください： https : //github.com/red/red/releases/tag/v0.1.0



観察している言語と技術は何ですか？



  • Go：これは近年最も急速に成長している言語です。この理由を理解することは、Redをより速く成長させるための鍵かもしれません。 Goの競争モデルはユーザーを引き付けるようですので、一見の価値があります。
  • Lua：どこで動いていて、どのように成長しているかを理解しようとしています。
  • Python 3.x：どこに向かっているのかを理解しようとしていますが、彼の計画を理解しているかどうかはわかりません。
  • Webassembly：Webプログラミングの将来の基盤。
  • MagicLeap：ヒューマンマシンインターフェイスの未来！

Carl Sassenrath -Amiga OSの作成者。パーソナルコンピュータ、Ada、Pascal、Smalltalk、Lisp、Forth、SPLなどの言語のプログラムにマルチタスクをもたらしました。 アランケイ、ラリーテスラー、ダンインガルス、ビルアトキンソンなど、多くのITの伝説でAppleの研究部門に勤務し、言語とシステムの理解を深めました。 1996年、Java、Perl、Pythonが人気を博し始めたとき、彼は自分のアイデアを世界と共有することを決定し、結果としてREBOL（相対式ベースのオブジェクト言語）になりました。 しかし、インタビューで述べたように、この言語は独自のものであり、さまざまな「構成」で配布されており、その一部はライセンスの下でのみ利用可能でした。 その後、CarlはREBOL3を作成しました。REBOL3は最初はオープンソースでしたが、REBOL2の機能とはほど遠いものであり、現在のところ開発されていません。 現時点では、赤はすでに積極的に開発されています。
