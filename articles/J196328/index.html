大規模な産業施設を設計するためのCAD

この記事では、今日の大規模な工業企業の設計のためのCAD開発段階のビジョン、それを実装する際に直面する困難、およびこの業界で概説されている傾向について説明します。

私の経験について少し。 工科大学の後ろで、設計エンジニアとして働き、CADスペシャリストとして再訓練し、C＃プログラマーとして働き、5つの設計機関で複雑な3次元設計システム「AVEVA PDMS」の実装、サポート、適応に参加します。 （石油/ガス処理企業とエネルギー施設の設計）とウクライナの1つ（エネルギー施設の設計）。 一般に、状況の何らかの分析を行うには経験が十分です。

各プロジェクト組織には独自の作業仕様があり、長年にわたって独自のプラクティスが開発されているという事実にもかかわらず、一般的な機能は依然として見えています。

1）すべての設計機関で、設計プロセスの自動化に対する管理の関心が高まっています。

2）自動化の最初の波が過ぎました。

この期間中、個々の部門のエンジニアの作業は、特殊なソフトウェアを購入することにより自動化されました。 さらに、多くの場合、部門間の情報交換は依然として「紙」形式で行われました。 部門が互いに分離されているため、最も高価な設計エラーの1つである衝突が発生しました。 この問題は、ネットワーク（たとえば、AVEVA PDMS、Intergraph SmartPlant 3D、Microstation AutoPlant）を介した単一の3Dモデルの主要な設計部門の共同作業のために、組織に統合設計システムを導入することで解決しました。

3）今日、自動化の第二の波があります。

統合された3D設計システムの実装中に最初の段階で得られた経験が分析され、そのようなシステムの実装とサポートのプロセスの複雑さに対する明確な認識があります。 実装に失敗した場合は、別のベンダーのソフトウェアを使用して2回目の試行が行われます。 組織内では、通常のプログラマーを使用して、組織の仕様に従って購入したプログラムの機能を拡張するプログラムが作成されます。 使用するソフトウェア間のさまざまなデータ交換インターフェースを独自に作成しました。 電子図面ワークフローと組み合わせて、電子プロジェクトワークフローシステムが導入されています。 これらのすべてのステップは、設計プロセスを編成する新しい方法-単一の情報スペース（SIP）での設計-への移行に徐々につながります。

組織のEIPを、設計された施設のEIPから分離する必要があります。 組織のEIPは特定のプロジェクトには適用されず、利用できなくても存在します。

次のシステムは、組織レベルのEIPに起因する可能性があります。

1）電子文書管理会社。

2）従業員のディレクトリ、時間管理システム、タスク管理システムを含む会社の企業ポータル。

3）設計見積もりの​​電子アーカイブ。

4）ソフトウェア、技術文書、組織文書、規制文書など、従業員の情報交換の手段。

5）ユーザーテクニカルサポートサービス（HelpDesk）。

もう1つは、設計された施設のEIPであり、プロジェクトに関する情報が蓄積されて保存されます。 この情報スペースは、次のコンポーネントで構成されています。

1）3Dモデル;

2）2Dドキュメント。

3）プロジェクトデータベース。

各項目について詳しく説明します。 3Dモデルは、1つのプログラム（PDMS、SmartPlant 3D）の両方で実行でき、さまざまな3Dモデリングプログラム（PDMS、オートデスクの製品など）で作成されたプロジェクトのさまざまな部分の3Dモデルのアセンブリです。 Autodesk Navisworksを使用すると、3Dモデル全体を組み立てて衝突をチェックすることができます。 より複雑な組み合わせも可能です。 たとえば、Inventor、Revit、Civil 3Dなどの特殊なプログラムで作成された3DモデルをPDMSにロードする場合、これらのモデルをロードするには2つの方法しかありません。機械装置インターフェイスを使用するか、自作プログラムを使用して3Dモデルを一連の標準プリミティブに変換し、 PDMS表面。

主に図面とテキストドキュメントで構成され、PDF形式のドキュメント（顧客に送信）と編集可能なドキュメント（主にDWG、Word、Excel）に分かれている2Dドキュメントに目を向けます。 EIPの組織の基本原則は、プロジェクトの要素に関する情報の一貫性です。 つまり 同じオブジェクトが3Dモデル、図面、Excelスプレッドシート（​​および/またはWord文書）に存在する場合、これらの要素はリンクでリンクする必要があります。 当然、要素ごとのコーディングシステムがプロジェクトごとに作成され、プロジェクト内での一意性が確保されます（KKS）。

そのため、プロジェクトのすべての要素をリンクするには、プロジェクトEIPの3番目（およびメイン）のコンポーネントであるプロジェクトデータベースが必要です。 Microsoft SQL Serverに基づいて行うのが（私の意見では）最も簡単です。 プロジェクトデータベースを作成することにより、すべての関連ドキュメント内のオブジェクトに関する情報の変更を同期できます。 設計組織ごとに異なるアプローチを使用して設計し、異なるソフトウェアを使用しているという事実により、誰にも平等に適合する既製のソリューションは市場にありません。 したがって、設計組織でEIPを作成する唯一の方法は、（フルタイムのプログラマーの助けを借りて）EIP自体を作成することです。

それとは別に、CAD部門のプログラマーチームについてお話ししたいと思います。 このようなチーム（通常3〜5人）は今日、CAD部門の不可欠な部分になっています。 このようなチームのさまざまなタスクは非常に大規模です。 最も特徴的なものは次のとおりです。

1）専用ソフトウェア間のデータ交換の自動化（計算、推定および描画プログラム）;

2）設計部門間のタスク交換プロセスの自動化。

3）既にインストールされているプログラムのユーザーインターフェイスのカスタマイズ。

4）さまざまなレポートを取得するためのプログラムの開発。

開発プロセスが制御不能にならず、ソースコードが常に正常な状態にあることを保証するには、共同開発とバージョン管理にさまざまなシステムを使用する必要があります。 Atlassian Jira + SVNやTeam Foundation Server 2012などのシステムを使用してきましたが、プロジェクト管理システムとバージョン管理システムが完全に接続されており、すべてがVisual Studioに統合されているため、TFSを選択するのは間違いありません。

共同3D設計システムを実装するプロセスについてのいくつかの言葉。 組織のすべての主要な設計部門が関与しているため、これは非常に複雑で時間がかかる作業です。 人々は新しいソフトウェア製品を学び、長年にわたって開発された習慣を変えなければなりません。 私自身の経験から、従業員の大多数は常に否定的に変化に直面していると確信しました。 組織が実装を独自に実装することを決定した場合（これに経験のある専門家を関与させずに）、資金と時間を費やして結果が達成されなかった場合に実装が失敗する可能性が高くなります。 したがって、組織内のこのようなグローバルな変更を決定する際には、「7回測定し、1回カットする」という民族の知恵が非常に適切です。
