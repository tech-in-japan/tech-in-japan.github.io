従来の方法による通常のVPN接続ではない

私は、Habrohabrに招待されるのに注目に値する興味深いトピックを探していましたが、今ではそれを見つけました。 最近、そのような特別なケースを実装する必要がありました。



問題ステートメント：リモートネットワークのノードへのアクセスを取得します。



ここでは、結合する必要がある2つのネットワークについて説明します。1つは「オフィスネットワーク」、もう1つは「リモートネットワーク」と呼びます。

リモートネットワークのシステム管理者は、最小限の変更を行うことを拒否します。接続するためにできることは、リモートネットワークに機器を置くことだけです。 このnetworkv4からのインターネットアクセスは、世界に入るゲートウェイを介して行われます。 2つのオフィス間にトンネルを構築して、オフィスネットワークのノードがリモートネットワークのノードにアクセスできるように、両側での変更を最小限に抑える必要があります。



2つのネットワークを組み合わせて仮想トンネルを構築するタスクを実行するには、仮想プライベートネットワークを使用する必要があります。 適切な接続オプションを検索する過程で、私はVPNをクライアントサーバーバージョンとピアの2つのタイプに分けました。 次の点が根本的な違いをもたらします。

  • グローバルインターネットを使用したピアツーピアVPNでは、各ノード（少なくとも2つのノード）に1つの実際のIPアドレスが必要です。 ここでは、接続は各当事者によって開始できます（これが私がそれを同等と呼んだ理由です）。
  • グローバルインターネットを使用するクライアントサーバーバージョンでは、サーバーに必要な実際のIPアドレスは1つだけです。 ここでの接続はクライアントの要求で発生し、サーバーは常に予期します。複数のクライアントが存在する可能性があります。


注1：どちらの場合も、条件の1つが満たされている必要があります（クライアントサーバーバージョンの場合、サーバーの場合のみ）。

  • A. VPNピアは、ゲートウェイ上に直接配置する必要があります（追加のソフトウェアをインストールするか、デバイスが目的のタイプのVPN接続を確立できる必要があります）。
  • B.ゲートウェイでVPNピアを直接起動できない場合は、VPNピアとして構成された別のデバイスにポートを渡すことができるように構成する必要があります。

したがって、選択が明らかになります - クライアントサーバーVPN 。 このオプションではクライアント側から何も変更する必要がないためです。



選択するクライアントサーバーオプションはどれですか？



そして最初に頭に浮かぶのは、古き良きPPTPです。 彼にはセキュリティ上の欠陥がたくさんありますが、より高度で新しいバージョンがあります。 セキュリティの欠陥にもかかわらず、PPTPには別の非常に不快な機能があります。これは、PPTPクライアントがサーバーへの接続を確立するために使用するGREプロトコルです。 注1.Bで指定されたスキームに従って通信を確立する場合、ポートフォワーディング（ポートフォワーディング）を実行する機器は、PPTP（多くの場合アプリケーションレベルゲートウェイセクションにあります）を介して接続を通過できる必要があります（この機能は多くの場合、PPTP経由と呼ばれます） PPTP自体を通過させます）。 これは、プロトコルの機能と、最初はPPTPがNATを介して動作するように設計されていなかったためです。 そのような機能がない場合、このスキームでは、サーバーが着信接続要求を認識しないという単純な理由により、VPNチャネルはインストールされません。

PPPoE、L2TP、およびPPTPは、完全にはほど遠いpap、chap、mscap1、およびmscap2認証プロトコルを使用します。



幸い、OpenVPNプロトコルには上記の欠点のほとんどがなく、GREプロトコルを使用せず、NATを使用するネットワークの機能を考慮して特別に設計されています。

Open VPNについて少し：ポート1194で動作し、TCPプロトコルとUDPプロトコルの両方を使用できます。 アプリケーション層プロトコル、OSIモデル。

したがって、これらのプロトコルの選択も明らかになりました-OpenVPN 。 もちろん、完璧ではなく万能薬ではありませんが、この場合には最適です。



少し要約してください。


OpenVPNサーバーをオフィスネットワークの側から使用します（何かを変更できる場合、全般的にはすべてを変更できますが、単に変更したくないだけです）。リモートネットワークの側からクライアントがあります。 ファイアウォールがハードウェアデバイスである（そしてOpenVPNサーバーが組み込まれていない）オフィスネットワークのスキームをあまり変更しないようにするには、OpenVPNサーバーが背後にあるデバイスをインストールし、ポート1194を転送する必要があります。

ここで、クライアントとサーバーの間で接続がどのように確立されるかを想像しますが、オフィスのネットワークノードはどのようにしてリモートネットワークのノードを見ることができますか？

リモートネットワークのルーターを変更しないように、オフィスのネットワークをNATで非表示にします。 オフィスの側のルーターで、VPNトンネルを介したリモートネットワークへのルートを書き留めます。

この場合のマスキングの欠点は、リモートネットワークのノードからオフィスのノードにアクセスできないことです。この場合、これはむしろ利点です。

叙情的な余談



理論的には、単一の実際のIPアドレスがなくてもVPN接続を確立できることに注意してください。 このような接続の類似物は、Skypeなどの音声インターネットメッセンジャーによって確立されます。 このような接続の本質は、複数のノードと1つのサーバーがあることです。 各ノードはサーバーに接続されますが、node1はnode2との通信セッションを確立する場合、サーバーからnode2のサーバーへの確立された接続のパラメーターを要求し、その後、確立された接続が切断されて直接確立されるまで、サーバーの代わりに「ウェッジイン」します。 これは、SIP（セッション開始プロトコル）を使用して実装されます。 そのため、Skypeサーバーには、クライアントによって生成された中間トラフィックが読み込まれません。 SIPコントロールを使用したVPN VPN接続のインストールに関するこのような調査の結果は、 www.ietf.org / mail-archive / web / sipping / current / msg10092.htmlで提供されました 。 残念ながら、 IETFからのドラフトVPN-SIPがあり、 ETSI TS 185 010 V2.1.1（2009-07）からの説明もあるにもかかわらず、このアイデアは広まりませんでした（ 「TS 185 010」と入力する必要がありますETSIの登録メンバーがドキュメントを読む）。
