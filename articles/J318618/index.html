配列、ベクトル、およびリスト



Cでは、配列は、同じタイプの変数の（明確に定義された）量の値を配置できる順次データブロックです。 int [5]は、int型の5つの変数を次々に直接格納するように設計された配列を記述します。



C ++は、ベクトルの概念を、現在のニーズに合わせて自動的にサイズ変更できる配列の類似物として導入します。 リストの標準タイプもあります。この場合、二重リンクリストを参照します（実際、標準は特定の実装に要件を提示していませんが、機能要件により、配列に基づくベクトル、および二重にリンクされたリストに基づくリストを論理的に実装します）。 しかし、ちょっと待ってください！ C ++ 11では、「initializer_list」という用語が導入されており、その名前には「list」（リスト）という単語が含まれていますが、実際は配列です。



もちろん、Lispリストはリンクリストであるため、頭と尾へのアクセスの観点からリストを簡単に処理できます。 Haskellも動作します。さらに、インデックスによってアイテムにすばやくアクセスするためのData.Arrayがあります。



Perlでは、シーケンス型は配列ですが、ここで使用するのに「型」という言葉はあまり適切ではありませんが、むしろ変数の形式です。 Perlには「リスト」という概念もありますが、これは式の評価の一部として存在する一時的なオブジェクトにすぎず、従来のコンテナーデータ型ではありません。 これはかなり奇妙なことであり、その説明には複数の段落が必要なので、ここからは始めません。



Pythonでは、リストはC ++のベクトルに似たプロパティを持つ基本データ型であり、（CPythonでは）C配列に基づいて実装されます。 標準ライブラリは、ほとんど使用されないデータ型配列も提供します。これは、C配列に数値をパックしてスペースを節約し、Cを介してPythonにアクセスするプログラマーを混乱させます。 ああ、組み込みのバイト配列型がまだありますが、これはバイトを格納する配列とは異なります。



Javascriptには配列型がありますが、文字列（！）キーを持つハッシュテーブルの上に構築されます。 C配列に数値を格納するためのArrayBufferもあります（Pythonの配列タイプに非常に似ています）。



PHPでは、配列と呼ばれるデータ型は、実際には文字列（！）キーを持つ順序付きハッシュテーブルです。 PHPにもリストがありますが、これはデータ型ではなく、単なる構文上の砂糖です。 PHPから他の言語に切り替えた人は、古典的なハッシュテーブルが整然としていないように見えることに時々驚かされます。



Lua言語は、配列、ベクトル、またはリストの用語をまったく使用せずに、あらゆる種類の伝統を拒否します。 そこにある唯一のデータ型はテーブルと呼ばれます。

C ++ ：マップ（ただし、これは実際にはバイナリツリーです。C++ 11はハッシュテーブルであるunordered_mapを追加します）

ポインター、リンク、エイリアス



Cには、一部のデータをメモリに保存するためのアドレスであるポインターがあります。 Cの場合、これは自然なことです。Cのすべてがメモリ内のデータを管理し、1つの大きなデータブロック内のアドレスとしてすべてのデータを表現することに関するものであるためです。 ポインタは、この大きなデータブロックの単なるインデックスです。



C ++は、Cからポインターを継承し、それらの乱用に対して直ちに警告します。 別の方法として、ポインタとまったく同じように見えるが、「*」演算子を使用する必要のない値にアクセスするためのリンクが提案されます。 これにより、Cにはなかった新しい（非常に奇妙な）機会がすぐに作成されます。2つのローカル変数がメモリ内の同じデータブロックを指すことができるため、行a = 5; 変数bの値を完全に変更できます。



Rustにはリンクがあり、C ++構文も使用しますが、実際には「借用ポインター」です（つまり、ポインターですが、透過的です）。 Cポインターの構文を使用する言語には、あまり一般的ではない「純粋なポインター」もあります。



Perlにはリンクがあります。 2つの別個のタイプのリンクです。 ハードリンク（アドレスが利用できず、アドレスを直接使用しないことを意味する唯一の例外を除き、Cのポインターの類似物）およびソフトリンク。変数の内容を別の変数の名前として使用します。 また、Perlには、C ++の参照と同様に機能するエイリアスがありますが、ローカル変数では機能せず、実際にはデータ型ではなく、単なるシンボルテーブルの操作です。



PHPにはリンクがありますが、Perlの影響にもかかわらず、リンク構文はC ++から取られました。 C ++は、それが参照する変数のタイプによってリンクを定義します。 しかし、PHPには変数の宣言がないため、変数は、＆演算子を含む特定の操作セットに参加した瞬間からリンクと見なされ始めます。 この魔法のシ​​ンボルは、変数に「参照性」を「感染」させます。



Python、Ruby、JavaScript、Lua、Java、および他の多くの言語には、ポインター、リンク、またはエイリアスがありません。 これは、特定の高レベルのことを説明する過程で、「これが示す...」、「これは...を参照する」などのフレーズをしばしば言わなければならないため、CおよびC ++の世界から来た人々にとってこれらの言語を理解することを困難にしますこれは人々を誤解させ、メモリ内の特定の領域へのポインタまたはリンクが実際にあり、その内容に直接アクセスできるという印象を与えます。 この理由から、私はC ++エイリアシングのリンクの動作を呼び出します。これは、何が起こっているかという本質をより明確に反映し、「リンク」という単語をより一般的に使用するためです。



参照渡しおよび値渡し



ところで、リンクについて。 Python についてはすでにこれを説明しましたが、ここで短いバージョンをもう一度書きます。 ここで重要な質問は言語Cが意味と見なすものであり、この質問はCに関連する言語ファミリー以外では意味をなさないため、この二分法はほとんどの言語で意味をなさない。



ここでの基本的な問題は、Cが構造を記述するための構文を持っていることですが、構造の言語のセマンティクスはコードでは見られません-単なるバイトのセットです。 構造はコンテナのように見えます。このような信頼性の高いコンテナです。内容は中括弧で囲まれているため、演算子「。」を使用する必要があります。 内部メンバーにアクセスします。 しかし、Cの場合、構造はバイナリデータの単なるブロックであり、intとそれほど変わらず、まあ、少し大きいかもしれません。 ああ、はい、あなたはまだいくつかの別々のデータを見ることができます。 ある構造を別の構造内に配置すると、C言語は外部構造内の内部ブロックのデータブロックを愚かに分離します。 ある構造を別の構造に割り当てると、たとえばdouble型の変数を割り当てる場合と同様に、通常のバイトコピーが発生します。 線は幻想的です。 その結果、Cで唯一の「本当の」コンテナはポインタです！



関数に構造体を渡すと、Cは他の型の変数のように、構造体を完全にコピーします。 関数で構造を変更する場合は、関数で関数にポインターを渡す必要があります。 非常に大きな構造体を関数に渡したい場合は、パフォーマンスを向上させるために再びポインターを使用する必要があります。



C ++がリンクの概念を導入しました。これは、Cのすべてのポインタが簡単すぎてわかりにくいのに突然気づいた場合に備えてです。 これで、以前と同様に、構造を「値で」渡すことができますが、呼び出された関数がリンクを受け入れる場合は、構造を「参照で」渡し、関数はそれを変更できます。 関数の引数は、渡される変数のエイリアスになるため、intのような単純な型でも書き換えることができます。 この「参照による転送」は、「エイリアスによる転送」と呼ばれます。



Java、Python、Ruby、Lua、JavaScript、および他の多くの言語は、個別のエンティティとしてコンテナ上で動作します。 内部に構造がある変数があり、この変数を別の変数に割り当てると、実際にはコピーは行われません。 両方の変数が参照するようになりました...いいえ、参照しない、示す...（いいえ、示さない）...



そして、彼女はここにいます-用語の問題です！ 言語Xがパラメーターを値で渡すのか参照で渡すのかを尋ねると、おそらくこの人はC言語モデルの観点で考え、他のすべての言語をこの基本モデルに何らかの形で当てはまるものとして提示します。 「両方の変数が参照されている」と言えば、C ++リンク（エイリアス）について話していると思うかもしれません。 「両方の変数が示す」と言えば、Cスタイルのポインターについて話していると判断できます多くの場合、言語には最初も2番目もありません。 しかし、英語では、私たちが言いたいことを表現する他の言葉はありません。



意味的には、言語は、変数の内容（その意味）が抽象的な世界で独自に存在しているかのように動作し、変数は単なる名前です。 割り当ては、名前と値を関連付けます。 これを初心者に「今はbを指している」または「今では同じオブジェクトを参照している」と説明するのは魅力的ですが、これらの説明は実際には言語に存在しない間接性を追加します。 aとbは両方とも同じオブジェクトを参照します。



この場合の関数呼び出しは、関数内の引数が呼び出しコードが関数に渡したのと同じ値を指定するため、割り当てフォームです。 それらを変更することができます-また、呼び出しコードはこれらの変更の結果を見ることができます。 呼び出された関数内で変数を再割り当てすることはできません。この場合の変数はエイリアスではありません。値を割り当てると、名前（関数内）が新しい値にバインドされますが、呼び出し元のコードの変数には影響しません。 これはすべて、古典的な「参照渡し」や「値渡し」をやや超えています。 確立された用語はまったくありません。オブジェクトの転送、名前による転送、転送の共有と呼ばれるものを聞きました。



原則として、C ++スタイルの参照渡しは、他の言語でも実装できます（前述したように、PHPはC ++リンク構文を使用してエイリアスで転送できます）。 ただし、エイリアスによる送信は、値による送信の代替としてのみ存在し、値による送信は存在します。半世紀前の低レベルCでは、他のものを実装することは不可能だったからです。



値で渡すことでできることはすべて、名前で渡すことで明示的なコピーを行うことでもできます。 さらに多くの場合、そのようなことは、関数からいくつかの値を返すことができるようにするために行われます。これは、高レベル言語では他の多くの方法で行うことができます。



無料タイピング



もちろんこれは解釈の問題ですが、個人的には「フリータイピング」などは存在しないと確信しています。 少なくとも、この用語の特定の定義を聞いたことはありません。

私は思い出させます：



  • 強力な型指定があります。つまり、変数がその型を変更せずに、コードが実行したい操作に「適応」します。 Rustは厳密に型指定された言語であるため、32ビットと64ビットの整数値を比較するとエラーが発生します。
  • 弱い型付けがあります。つまり、計算式に近づくために変数の型を変更できます。 JavaScriptは弱い型付けの言語であり、5 +“ 3”は暗黙的に文字列を数値に変換し、式は結果8を返します（冗談です。当然、結果は“ 53”になります）。 Cの型も弱くなります。int型の変数を取得して「3」に設定するだけで、エキセントリックですが、コンパイルされたコードを取得できます。
  • 静的型付けがあります。つまり、変数の型はコンパイル段階で既知です。 Javaは静的型付けを行う言語です。 Javaコードを見るだけです-使用されている型の名前の70％で構成されているようです。
  • 動的型付けがあります。つまり、プログラムの実行時に変数の型が決定されます。 Rubyは動的型付け言語であり、型は実行時に定義されます。


「強い」タイピングと「弱い」タイピングの概念は、世界の調和のとれた図を作成します。 「静的」および「動的」な類型化も理解可能であり、補完的です。 言語には静的および動的だけでなく、強い型と弱い型の両方の要素を含めることができますが、一部の位置はまだ一般的です。 たとえば、Goは静的に型付けされていると見なされますが、Goのインターフェイス{}には動的な型付けの兆候があります。 逆に、Pythonは正式に静的に型指定されており、各変数はオブジェクト型ですが、幸運を祈ります。



「強い」/「弱い」タイピングの関係は変数の値を指し、「静的」/「動的」はそれらの名前を指すため、4つの組み合わせすべてが存在します。 Haskellは強力で静的です。弱いと静的では、Pythonは強力で動的です。Shellは脆弱で動的です。



それでは、「無料入力」とは何ですか？ これは「弱い」の類似物であると誰かが言いますが、多くの人は「自由に型付けされた」Pythonと呼んでいますが、Pythonは強い型付けの言語を指します。 （少なくともCより強い！）。



また、「自由に型付けされた」という用語はほとんど軽rog的な意味で出くわすので、人々は「C ++のように型付けされていない」ことを意味すると推測できます。 ここでは、だれの牛が動き、C ++が沈黙することに注意する必要があります。 C ++型システムには欠陥はありません。 たとえば、タイプTへのポインターのタイプはどうなりますか？ いいえ、これはT *ではありません。ヌルポインター（およびこれはT型の変数へのポインターではない）またはランダムガベージ（T型の変数へのポインターではない可能性が高い）を割り当てることができるためです。 特定の型の変数に実際にこの型の値が含まれていない可能性がある場合、静的型付けを誇りに思うポイントは何ですか？



キャッシング



キャッシングで最もおもしろいことは、これはどの言語の機能でもなく、よく知られた概念です。 キャッシングにより、一部の計算結果が保存され、後でそれらを繰り返す必要がなくなります。 古典的な最適化、または速度のためのメモリの交換。 キャッシュの最も重要な特性は、キャッシュをクリアするか、キャッシュを破棄するか、他の理由でキャッシュ内のデータにアクセスできない場合、すべてが以前と同じように動作し続け、少し遅くなることです。



そして、一般的に再利用するためにキャッシュをデータストレージと呼ぶプログラマーとコードをどこでも見ます。 これは非常に紛らわしいです。 良い例は、Pythonプロジェクトでよく目にするコードの一例です。 彼に最初に気付いたのは、この機能がreifyと呼ばれるPyramidプロジェクトです。 彼女は、次のようなオブジェクト属性の遅延初期化を実行しました。 



class Monster: def think(self): # do something smart @reify def inventory(self): return []


ここでは、monster.inventoryは、実際に読んでみるまで存在しません。 この時点で、reifyが呼び出され（一度だけ）、返されるリストが属性になります。 すべてが完全に透過的です。値が作成されるとすぐに、間接アクセスのオーバーヘッドが発生しない共通の属性になります。 それに何かを追加でき、すべてのアクセスで同じ結果が表示されます。 この属性は、見る試みによって命を呼ぶまで存在しませんでした。
