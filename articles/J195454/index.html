ディストリビューションデータベースのストレージ形式の説明


データベース内の配布数：1411636



2つの保管場所があります：テーブルと説明ベース。



このテーブルには、ルートトラッカーの配布番号、配布の名前、バイト単位の概算サイズ、シードの数、ピアの数、base32形式のハッシュ、ダウンロードの数、更新が配布された日付が含まれます。 分布のサイズは「2.05 GB」の形式の文字列を解析することで取得されたため、おおよそのサイズです。 残念ながら、配布ページのソースコードから正確なサイズを見つける方法は見つかりませんでした。 ディストリビューション名はUTF-8でエンコードされているため、このエンコードが標準のシステムでは、追加の操作を行わなくてもファイルを表示できます。 使用するスペースを少なくするための配布のBase32ハッシュ。 ベースを表示するためのグラフィカルプログラムでは、（マグネットリンクを含む）ハッシュの表示をHEXに切り替えることができます。 フィールド区切り：TAB。 ディストリビューション名のすべての空白はスペースに置き換えられました。 名前のすべてのHTMLコンストラクトは、対応するUnicode文字に置き換えられました。これは、cp1251がUTF-8を支持して放棄されたもう1つの理由です。 日付は「16-Jul-11 06:23」の形式でエンコードされます。 月の英語名は、解析のトラブルが少なくなるように選択されます。

説明のデータベースはtar.gzファイルのコレクションであり、各ファイルには1000ずつ増加する分布が含まれています。RAMの速度と気取らない理由からgzipが選択されています。 アーカイブファイルは、100個ごとにフォルダーにグループ化されます。 番号1234567の配布の説明は、UTF-8エンコードのファイル012 / 01234.tar.gz / 01234567にあります。



プログラム


ソース GNU GPL v2ライセンス。 プルリクエストを送信します。



プログラムは、Qtおよびkdelibsライブラリを使用してC ++で記述されています （アーカイブの操作用）。 プログラムの主要な部分は、分布が表示されるテーブルです（ QTableWidgetを使用）。 上部には、検索フレーズを入力するためのフィールドがあります。 検索（テーブルを含むファイルの読み取りと適切な行の選択）は別のスレッド（スレッド）で行われ、結果は部分的にメインストリームに送信され、テーブルに新しい行が追加されます。 タイプQt :: QueuedConnectionの接続は、スレッド間で結果を転送するために使用されます。 ファイルが最後まで読み取られるか、必要な数の結果が選択されると、検索が完了したことを示すメッセージがメインストリームに送信されます。 その後、テーブルは再ソートされます。 検索中に上部にある[停止]ボタンを使用して、検索を中断できます。



テーブルを含むファイルは、gzip、bzip2、またはlzma / xzで圧縮できます（残念ながら、Windowsでは、後者のオプションはアセンブリでサポートされていません）。 ファイルは解凍され、その場で表示されます。一時ファイルを完全に解凍して作成することはありません。 これは、 kdelibsライブラリのKFilterDevクラスを使用して実装されます。 gzipおよびxzは、bzip2よりもはるかに優れたアンパック速度を提供することがわかったため、bzip2は、データベースが配布される形式を選択するときに放棄されました。 Gzipはxzの何倍もの速度を示し、使用されているバージョンのkdelibsライブラリのウィンドウに存在していました。 したがって、圧縮が1.5倍失われたにもかかわらず、選択はgzipで行われました。 ユーザーはテーブルを個別に解凍するか、対応するメニューオプションを使用して、テーブルを圧縮せずにディスクに保存できます。 ちなみに、検索中に大量のデータがハードドライブから読み取られ、ハードドライブからの読み取りはgzipの解凍よりも遅くなる可能性があるため、これにより検索が高速化されることはありません。



テーブルを考えてみましょう。 列の意味を説明する必要はないと思います。 すべての列で並べ替えることができます。デフォルトでは、結果はダウンロード数で並べ替えられます。 ソートを実装するには、QTableWidgetItemから継承して比較操作を定義する必要がありました。



セルをダブルクリックすると、その値が強調表示され、コピーに適した状態になります。



分布の説明を表示するには-分布番号とハッシュを除く任意のフィールドを左クリックします。 説明が下に表示されます（ QWebViewを使用）。

ディストリビューションを含むページをロードして下に表示するには、ディストリビューション番号をクリックします。 配布URLをコピーするには、その番号を右クリックします。

分布の番号とハッシュを含むセルを右クリックすると、オプション「リンクのコピー」を含むコンテキストメニューが表示されるようになります。 たぶん、読者の1人はQTableViewからこれを達成する方法を知っています。 ただし、マウスの右ボタンをクリックする方がコンテキストメニューから項目を選択するよりも速いため、そのままにしておくことができます。



セルでのマウスイベントのインターセプトの実装は、 QItemDelegateから継承し、 editorEventを定義することによって行われます。 対応するtar.gzから説明を取得するには、 kdelibsライブラリのKTarクラスを使用します。



このプログラムは、配布の説明のデータベースがなくても使用できます。配布番号をクリックすると、サイトを通じてのみ説明を表示できます。



プログラムは、プログラムフォルダーにあるdump_viewer.iniファイルに設定を保存します。

Debian GNU / LinuxおよびWindows用のプログラムをビルドする手順は、 INSTALLファイルにあります。



プログラムの開発中に、日付の解析で面白い事件が発生しました。 日付形式「16-Jul-11 06:23」は非標準ですが、かなり短く、読みやすく、rutrackerが出力で使用するものと類似しているため、廃止されました。 QDateTime :: fromStringはローカライズされた月の指定（ロシア語を話す環境ではJanではなくJan）を想定していることが判明しました。 したがって、月のテキスト指定を数値に変換する松葉杖を作成する必要がありました（ Jan- > 01）。



なぜこれをしたのですか？


データベースは、トラッカーサイトのアクセシビリティに問題がある場合に、ディストリビューションへのユーザーアクセスを容易にするために準備されました。 たとえば、「フォーラムが一時的に無効になっています」というメッセージが表示された場合。 さらに、この配布は、トラッカーがブロックされたサイトのリストに含まれている場合に役立ちます。 私たちが長年にわたって一緒にやったことのすべてが、例えば役人の気まぐれやサーバーの故障などで失われた可能性が最小になることは望みません。 このディストリビューションはまだ生きていますが、トラッカーのすべてのディストリビューションも生きています。 おそらく月に一度、このディストリビューションを更新する必要があります。



rutrackerは、トラッカー上の暗号化された配布の方が優れていると書いています。

回答：（詳細はこちらとこちら ）

a）分布の説明があります。 たとえば、説明を見ずにBDRipを選択することは困難です。 すべての記述のベースは、最大2ギガバイトに圧縮されています。 より強く絞ることは可能でしたが、「銃口」の速度を犠牲にして節約しないことに決めました。 （実際、最適化についてはまだいくつかの考えがありますが、これまでのところ、最良の敵は良いと判断しました。しかし、アイデアとコミットは大歓迎です！）

b）パスワードを知っている人々のグループが世界中に分散している場合でも、これは計算でき、購入または脅迫するのに必要なリソースを備えた最終的な人々のグループです。

c）個人的にrutrackerと知性の管理は、疑う余地なく無限に正直な人々ですが、私自身がその分布がrutrackerのベースであり、暗号化されたホワイトノイズではないことを知るまで、誰も信じません。 ごめんなさい

d）偽サイトおよび偽磁気リンクには問題ありません。 データベースは管理者だけでなく実行できます（データベースはこの例です）。そのため、ルートキーパー上のデータベースの暗号化は保存されません。 また、データベース内のハッシュの有効性は、チェックサム（GPG署名付き）、またはrutracker自体との通常の比較（使用可能な場合）によって確認されます。

e）データベースに実際の分布を持たせるには、データベースを簡単に更新する必要があります。 より頻繁に、より良い。 そして、ルートキーパーの管理者がユーザーが関連情報を受け取ることを本当に気にしているのであれば、私たちがデータベースの更新を妨げないことを望みます。 そして、彼らは助けます、地獄が冗談ではないもの。

次の論理的なステップは、HTML [PHP]ジェネレーター（プログラムとデータベースの機能を複製するサイト ）を作成することです。 その後、サイトのすべての部分、つまり、純粋なHTML / CSS / JSの静的な実装に取り​​組み、PHPまたは同様のサーバーロジックを使用しません。 これにより、無料ホスティングを含むほぼすべてのホスティングにサイトをアップロードできるようになります。これにより、このデータベースをネットワークから根絶することが原則的に不可能になります。 JavaScriptでの検索の実装に関するトピックについては既にアイデアがあります（たとえば、単語ごとの分布のインデックスを作成し、それを個別のファイルに分割し、1つのファイルの平均サイズとファイルの総数の間でバランスを取ります）。 本格的なサーバー側の検索実装を追加できます。 残念ながら、賢明なWeb開発者はいませんが、希望する人は探しています。
