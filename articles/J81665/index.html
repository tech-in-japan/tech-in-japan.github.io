ユーザーが好むアプリケーションを作成する方法

このhabratopikeでは、私が思うに、初心者（だけでなく）プログラマーがユーザーが喜んで愛用するアプリケーションを作成するのに役立ついくつかのポイントを説明したいと思います。



まえがき



残念ながら、他の場所と同様、 本当に便利で便利なプログラムはめったに見つかりません。プログラマーとしては、個人的には非常にイライラします。 したがって、少なくともソフトウェア開発に携わっているHabrahabrの読者の間で、アプリケーションの品質が向上することを期待して、この記事を書くことにしました。 私は商業的利益を得るという観点からソフトウェアを作成することを考えていません、私はこれにあまり強くありません:)。 したがって、ここで述べられているヒントの一部は、市販のソフトウェアに適用すると有害な場合があります。 それでも、オープンソースまたはフリーソフトウェアを開発している場合、音声付きのヒントは製品の品​​質を向上させ、プログラマーとしてのスキルを向上させるのに役立ちます。



「良い」プログラムの特徴：



  1. このアプリケーションは、ユーザーのいくつかの非常に具体的なタスクと問題を解決し、このプログラムなしで同じ問題とタスクを解決するよりも効率を上げることができます。
  2. 簡単なインストール-ユーザーはアプリケーションのインストールに最小限の障害を抱えている必要があり、プログラムは含まれていないサードパーティのライブラリ/ユーティリティに依存してはいけません
  3. シンプルでレスポンシブで高速に動作するインターフェース-ユーザーは、プログラムが「フリーズ」したり、新しいウィンドウをほとんど読み込まないなど、本当に嫌いです。 彼らにはこれに対する権利があります。今日のコンピューターは本当に優れたコンピューティングリソースを備えており、適切なアプリケーション設計により、複雑な視覚効果を使用する場合でも、プログラムのGUIの「ブレーキ」を完全に排除できます
  4. 作業の正確さ-プログラムは、ユーザーが期待するアクションを正確に実行する必要があります。 何らかの操作を完了できない場合、プログラムは、ドキュメントに登る必要なく、問題の理由とその解決方法を人間の言葉で説明しようとする必要があります
  5. プログラムの高速-これはすべてに適用されます：迅速にロードし、その義務を迅速に実行し、その閉鎖に抵抗してはなりません（uTorrentのような一部のアプリケーションはそうしますが、uTorrentは正当な理由で行う）


例



「良い」プログラムの例は、多くを導くことができます。 何年も前に、私はHieroglyphプログラムに非常に感銘を受けました。これは完璧に機能することに加えて、たとえば、そのインターフェイスの言語をその場で切り替えることができます（Windows Vista / 7で起動するかどうかはわかりませんが、古いWindowsでは作品）。 これは一見些細なことです（初心者向け:)）、プログラム自体を複雑にすることで不要なアクションを実行する必要性からユーザーを守ります。 ロシア語のテキストを編集するタスクの実行を保証するために、Hieroglyphは独自のフォントレンダリングエンジンを使用して、Windowsユーザーを悩ませる「crook」の問題を完全に排除する機会さえ得ました。



Mac OS Xのプログラムの中で、私はTextMateが本当に好きです。Komodo 、 Eclipse 、 NetBeansなどの最新の非常に重いIDEやテキストエディターとは対照的に、TextMateは必要な機能と拡張の容易さを持ちながら、スペースをほとんど取りません（ディストリビューションで10 MB）非常に高速にロードされ、多くのシステムリソースを必要としません。



純粋にLinuxアプリケーションから、おそらくEnlightenmentをウィンドウ環境と呼ぶことができます-それはまた、ミニマリズムと操作の容易さを公言します。



開発アプローチ



明らかに、 うまく機能しないアプリケーションを書くのは簡単です。 あなたがプログラマーである場合（そうでない場合、なぜこの記事を読んでいるのですか？）？）そして、あなたはあなた自身がそのようなプログラムをたくさん書いて、毎日書いているので、あなたはこれをよく知っています。 また、私はそれらを書きます。特にプログラムが「自分用に」書かれている場合、結果を得るのに必要な時間よりも多くの時間を費やしたくありません。 それにもかかわらず、あなた自身のためではなく他のユーザーのためにプログラムを書いている場合、開発アプローチと品質要件の両方も異なるはずです：

  1. 開発の最初の段階で、たとえば「ファッショナブルな」ライブラリ/フレームワーク/パラダイムを使用するかどうかを選択する場合は、必要な量とその機能をどれだけ使用するかを理解し、評価しようとします。アプリケーションのサイズ、ダウンロード速度、インストールの容易さへの影響。 実践では、小さなプログラムの場合、大きなフレームワークまたは純粋に独自の技術を使用しても、ユーザーの利便性に対するアプリケーションでの正当性は示されません。 （ Haberで最近公開されたCodeCraftプログラムを例に挙げることができます。これは、数メガバイト（そのほとんどは背景画像です）の重量で、100 MBを超える合計重量のフレームワークを個別にダウンロードおよびインストールするために最も人気のあるWindows XP OSが必要です、独自のロボットを作成するには-再度VisualStudioをインストールし、少なくともMicrosoft Word / OpenOfficeをインストールして指示を読んでください））
  2. アプリケーションにグラフィカルインターフェイスが含まれている場合は、少なくとも、対応するOSまたはDEのGUIガイドラインを垣間見ることができます。 Webアプリケーションを開発している場合は、標準コントロールを独自のものに置き換えないようにし、非標準フォントを使用しないようにし、すべてのブラウザーで利用できない機能を乱用しないようにしてください。 確立された標準に従うことで、ユーザー間のコミュニケーションがより簡単で直感的になります。
  3. インターフェイスの応答性を確保するには、イベントハンドラーでリソースを集中的に使用する操作、特にマウスクリックやウィンドウのサイズ変更を実行しないようにしてください。 イベントハンドラーでは、すべてのイベントを処理するアプリケーションのメインスレッドに大きなキューがないように、バックグラウンドで同じ操作を実行する最小限のコードのみが必要です。そうでない場合、ユーザーはプログラムが「フリーズ」していると思います。 この推奨事項を無視する非常に良い例は、ネットワークを操作するときにWindowsの古いバージョンのエクスプローラーです。イベント処理スレッドでブロック呼び出しを使用することが多く、よく知られている結果につながりました:)。 Webアプリケーションを作成する場合は、少なくとも長時間、クライアントでリソース集約型の操作を実行しないようにしてください。これは、Javascriptにはマルチスレッドを作成するための単一の標準がなく、すべてのコードがシングルスレッドモードで実行されるためです。イベント処理遅延
  4. アプリケーションの作成時に設定された目標を忘れないでください。プログラムは、本来意図されていたとおりに動作する必要があり、それ以外は何もしません。 さらに、プログラムは正しい初期データで誤った結果を与えてはならないことに注意してください。 つまり、サブタスクを解決する方法がいくつかある場合は、このためにパフォーマンスを犠牲にする必要がある場合でも、初期データに対して正しい答えを与えるものを選択します（プログラムを修正する必要がない場合は同意します）作業したら、ルート計算関数をdouble sqrt（double x）{return 0.0;}として書くことができます-これは数値のルートを計算するための最速の関数になりますが、それは意味がありません:)）
  5. 良いプログラムはすぐに動作するという事実にもかかわらず、これはプログラムを書く前またはプロセスのすぐ前にプログラムを最初から最適化する必要があるという意味ではありません-Donald Knuthが言ったように、「早すぎる最適化はすべての悪の根源です」（ en.wikipedia .org / wiki / Program_optimization＃Quotes ）。 それでも、アプリケーションはプログラムを使用する人のハードウェアよりも強力なハードウェアでテストされる可能性が最も高いことを忘れないでください。したがって、マシンのわずかな「ブレーキ」で問題を見つけ、対応するものを高速化してください作品。 また、平均してユーザーから予想されるボリュームを大幅に超えるアプリケーションデータボリュームに「フィード」しようとすると、予想されるデータボリュームを桁違いに超えてもアプリケーションが「ハング」せず、「スローダウン」しないことを確認できます。 （ほとんどの場合、ユーザーは自分のハードウェアを「最大限に」使用したい場合があり、アプリケーションは、何らかのデータ処理に従事している場合にこれを可能にする必要があります）
  6. Webアプリケーションを作成する場合は、狭いチャネルまたは不安定なチャネルを持つユーザーでも不便なく使用できることを確認してください。ドキュメントとそのコンテンツのリストを表示するために5分間待つ必要はありません（はい、これGoogleドキュメントの庭の石！）。 私たちはかつてダイヤルアップを使用してインターネットにアクセスしていましたが、最近個人的には64 kbit / sのチャンネルがあり、遠隔地に住んでいます
  7. そして最後に、製品のこれらのリリース（および中間バージョンも）をさまざまな状況で徹底的にテストすることを忘れないでください。 「遅くなる」プログラムよりも悪いのは、不安定に動作するプログラムだけです！

文学



数冊の本に言及せざるを得ませんが、それは読むことを義務付けているか、少なくとも非常に有用で、プログラムの書き方を学ぶのに役立つと考えています

  1. Brian Kernighan、Dennis Ritchie「 ANSI C 」は非常に説明的なC言語ガイドで、Cを学ぶだけでなく、プログラミング（あらゆる言語）を教えるのに役立ちます。
  2. Brian Kernigan、Rob Pike、 The Practice of Programming 、わずか10年前の本（1978年に最初のバージョンがリリースされた前の本と比較して）には、Cプログラミングだけでなく、例も含まれています。 C ++、AWK、Perl、Javaで、実際の問題の解決に適用されるように、純粋に実用的な側面でプログラミングのほぼすべての領域に影響を与えます
  3. Marc J. Rochkind“ Advanced UNIX programming ”（ロシア語版では単に“ Programming under UNIX ”と訳されています ）は、UNIXでのプログラミングについて詳細に説明する優れた本であり、私のようなアプリケーションを書くための膨大な数の一般的な推奨事項も含まれていますUNIXおよびUNIXに似たシステムのさまざまなバージョンのシステムコールの記述と関連する落とし穴よりも価値があるようです
