イベントフラググループヘルパーサービス



Nucleus RTOSには、イベントフラググループのヘルパー機能を提供する3つのAPIコールがあります。グループ情報の取得、アプリケーション内のイベントフラググループの数に関する情報の取得、およびすべてのイベントフラググループへのポインタの取得です。 最初の2つの課題はNucleus SEに実装されています。



イベントフラググループ情報の取得



このユーティリティ呼び出しは、イベントフラグのグループに関する情報を返します。 Nucleus SEでのこの呼び出しの実装は、オブジェクトの命名とタスクの一時停止の順序がサポートされていないため、返される情報が少ないという点でNucleus RTOSの実装とは異なります。



Nucleus RTOSでのイベントグループ情報の呼び出し

サービスコールのプロトタイプ：

STATUS NU_Event_Group_Information（NU_EVENT_GROUP * group、CHAR * name、UNSIGNED * event_flags、UNSIGNED * tasks_waiting、NU_TASK ** first_task）;



パラメータ：

group-イベントフラグのグループに対してユーザーが提供する制御ブロックへのポインター。

name-イベントフラグのグループの名前を表す8文字の領域へのポインター。これには終了ゼロも含まれます。

event_flags-指定されたイベントフラグのグループの現在の値を取得する変数へのポインタ。

tasks_waiting-このイベントフラググループ内の一時停止タスクの数を取得する変数へのポインタ。

first_task - NU_TASK型の変数へのポインター。これは、最初に中断されたタスクへのポインターを取ります。



戻り値：

NU_SUCCESS-呼び出しは正常に完了しました。

NU_INVALID_GROUP-イベントフラグのグループへの無効なポインター。



Nucleus SEでイベントグループ情報を取得するための呼び出し

この呼び出しは、Nucleus RTOS APIのコア機能をサポートします。



サービスコールのプロトタイプ：

STATUS NUSE_Event_Group_Information（NUSE_EVENT_GROUPグループ、U8 * event_flags、U8 * tasks_waiting、NUSE_TASK * first_task）;



パラメータ：

group-情報が要求されるイベントフラグのグループのインデックス。

event_flags-指定されたイベントフラグのグループの現在の値を取得する変数へのポインタ。

tasks_waiting-このイベントフラグのグループ内の中断されたタスクの数を取る変数へのポインター（タスクの中断が非アクティブ化された場合、何も返されません）。

first_task - NUSE_TASK型の変数へのポインター。これは、最初の中断されたタスクのインデックスを取得します（タスクの中断が非アクティブ化された場合、何も返されません）。



戻り値：

NUSE_SUCCESS-呼び出しは正常に完了しました。

NUSE_INVALID_GROUP-イベントフラググループのインデックスが無効です。



Nucleus SEでのイベントグループ情報の実装

このAPI呼び出しの実装は非常に簡単です。 



*event_flags = NUSE_Event_Group_Data[group]; #if NUSE_BLOCKING_ENABLE *tasks_waiting = NUSE_Event_Group_Blocking_Count[group]; if (NUSE_Event_Group_Blocking_Count[group] != 0) { U8 index; for (index=0; index<NUSE_TASK_NUMBER; index++) { if ((LONIB(NUSE_Task_Status[index]) == NUSE_EVENT_SUSPEND) && (HINIB(NUSE_Task_Status[index]) == group)) { *first_task = index; break; } } } else { *first_task = 0; } #else *tasks_waiting = 0; *first_task = 0; #endif return NUSE_SUCCESS;


この関数は、イベントフラグのグループの値を返します。 次に、タスクブロッキングAPI呼び出しがアクティブになっている場合、保留中のタスクの数と最初のタスクのインデックスが返されます（それ以外の場合、これら2つのパラメーターはゼロに設定されます）。



イベントフラググループの数の取得



このユーティリティ呼び出しは、アプリケーション内のイベントフラググループの数を返します。 Nucleus RTOSでは、この値は時間とともに変化し、戻り値は現在のグループ数を示します。NucleusSEでは、この値はアセンブリ中に決定され、時間とともに変化しません。



Nucleus RTOSでイベントフラググループのカウンターを呼び出す

サービスコールのプロトタイプ：

UNSIGNED NU_Establised_Event_Groups（VOID）;



パラメータ：

欠席しています。



戻り値：

作成されたイベントフラググループの現在の数。



Nucleus SEでイベントフラググループのカウンターを呼び出す

サービスコールのプロトタイプ：

U8 NUSE_Event_Group_Count（void）;



パラメータ：

欠席しています。



戻り値：

設定されたイベントフラググループの数。



Nucleus SEでイベントフラググループカウンターを実装する

このAPI呼び出しの実装は非常に簡単です。 #define NUSE_EVENT_GROUP_NUMBERシンボルの値が返されます。



データ構造



他のすべてのNucleus SEオブジェクトと同様に、イベントフラググループは1つまたは2つのデータ構造の配列を使用し（両方ともRAMに配置されます）、配列のサイズは設定で定義されたグループの数によって異なります。



アプリケーションコードはこれらのデータ構造への直接アクセスを使用せず、提供されたAPI関数を介してそれらを参照することを強くお勧めします。 これにより、Nucleus SEの将来のバージョンとの非互換性および不要な副作用を回避し、Nucleus RTOSへのアプリケーションの移植を簡素化します。 サービス呼び出しコードとデバッグの理解を深めるために、データ構造の詳細な概要を以下に示します。



RAMデータ



このデータの構造は次のとおりです。

NUSE_Event_Group_Data [] -構成されたフラグのグループごとに1つのレコードを持つ、タイプU8のデータの配列。 イベントフラグデータを格納します。

NUSE_Event_Group_Blocking_Count [] -イベントフラグの各グループでブロックされたタスクのカウンターを含むU8型の配列。 この配列は、APIのロック機能が有効になっている場合にのみ存在します。



これらのデータ構造は、Nucleus SEの起動時にNUSE_Init_Event_Group（）関数でゼロで初期化されます。 次の記事のいずれかで、Nucleus SEの起動手順の完全な説明が提供されます。



以下は、 nuse_init.cファイルのこれらのデータ構造の説明です。 



 RAM U8 NUSE_Event_Group_Data[NUSE_EVENT_GROUP_NUMBER]; #if NUSE_BLOCKING_ENABLE RAM U8 NUSE_Event_Group_Blocking_Count[NUSE_EVENT_GROUP_NUMBER]; #endif


ROMデータ



イベントフラグのグループを実装するために、ROM内のデータは使用されません。



イベントフラググループのメモリ容量



すべてのNucleus SEカーネルオブジェクトと同様に、イベントフラググループに必要なメモリ量は予測可能です。



アプリケーションのイベントフラグのすべてのグループのROMのデータ量は0です。



APIロック機能がアクティブになっているイベントフラグのすべてのグループのRAMのメモリ量は、 NUSE_EVENT_GROUP_NUMBER * 2です。



それ以外の場合は、 NUSE_EVENT_GROUP_NUMBERです。



未実現のAPI呼び出し



Nucleus RTOSにあるイベントフラググループの3つのAPI呼び出しは、Nucleus SEには実装されていません。



イベントフラググループの作成



このAPI呼び出しは、イベントフラグのグループを作成します。 イベントフラググループは静的に作成されるため、Nucleus SEはこの呼び出しを必要としません。



サービスコールのプロトタイプ：

STATUS NU_Create_Event_Group（NU_EVENT_GROUP * group、CHAR * name）;



パラメータ：



group-イベントフラグのグループに対してユーザーが提供する制御ブロックへのポインター。 他のAPI呼び出しでイベントフラグのグループを管理するための記述子として使用されます

name-この領域に含まれる終了ヌルバイトを持つイベントフラグのグループの8文字の名前へのポインタ。



戻り値：



NU_SUCCESS-呼び出しは正常に完了しました。

NU_INVALID_GROUP-イベントフラググループのコントロールユニットへのNULLポインター（ NULL ）または既に使用中。



イベントフラグのグループを削除する



このAPI呼び出しは、以前に作成されたイベントフラググループを削除します。 イベントフラググループは静的に作成され、削除できないため、Nucleus SEはこの呼び出しを必要としません。



サービスコールのプロトタイプ：



STATUS NU_Delete_Event_Group（NU_EVENT_GROUP * group）;



パラメータ：



group-イベントフラグのグループの制御ブロックへのポインター。



戻り値：



NU_SUCCESS-呼び出しは正常に完了しました。

NU_INVALID_GROUP-イベントフラグのグループへの無効なポインター。



イベントフラググループポインター



このAPI呼び出しは、システム内のイベントフラグのすべてのグループへのポインターの順次リストを構成します。 イベントフラググループにはポインタではなく単純なインデックスがあるため、Nucleus SEはこの呼び出しを必要としません。



サービスコールのプロトタイプ：



UNSIGNED NU_Event_Group_Pointers（NU_EVENT_GROUP * pointer_list、UNSIGNED maximum_pointers）;



パラメータ：



pointer_list-ポインターの配列NU_EVENT_GROUPへのポインター。この配列には、システムで作成されたイベントフラグのグループへのポインターが格納されます。

maximum_pointers-配列内のポインターの最大数。



戻り値：



配列内のNU_EVENT_GROUPポインターの数。



Nucleus RTOS互換



Nucleus SEを開発するときの私の目標は、Nucleus RTOSとのコード互換性の最大レベルを確保することでした。 イベントフラグのグループも例外ではなく、開発者の観点から見ると、Nucleus RTOSとほぼ同じ方法で実装されています。 最終的なコードが必要なメモリ量の点でより理解しやすく、効率的になることを考えると、有効だと考えた非互換性がいくつかあります。 それ以外の場合、Nucleus RTOS API呼び出しは、Nucleus SE呼び出しとしてほぼ直接使用できます。



オブジェクト識別子



Nucleus RTOSでは、すべてのオブジェクトは特定のタイプのデータ構造（コントロールユニット）によって記述されます。 この制御ユニットへのポインタは、イベントフラグのグループの識別子です。 Nucleus SEでは、メモリを効率的に使用するには別のアプローチが必要であると判断しました。すべてのカーネルオブジェクトは、RAMおよび/またはROMのいくつかのテーブルで記述されます。 これらのテーブルのサイズは、各タイプの設定済みオブジェクトの数によって決まります。 特定のオブジェクトの識別子は、このテーブルのインデックスです。 したがって、 NUSE_EVENT_GROUPをU8と同等のものとして定義しました。このタイプの変数（ポインターではない）は、イベントフラグのグループの識別子として機能します。 コードをNucleus SEからNucleus RTOSに、またはその逆に移植する場合、このわずかな非互換性は簡単に処理できます。 通常、移動と保存以外のオブジェクト識別子に対する操作は実行されません。



Nucleus RTOSは、イベントフラググループの命名もサポートしています。 これらの名前は、デバッグにのみ使用されます。 メモリを節約するために、Nucleus SEからそれらを除外しました。



グループ内のフラグの数



Nucleus RTOSでは、イベントフラググループにはそれぞれ32個のフラグが含まれます。NucleusSEでは、これを単純なアプリケーションに十分であり、RAMを節約できるため、8個に減らしました。 イベントフラグのより大きなグループが必要な場合、Nucleus SEは簡単に変更できます。



フラッグ吸収機能



Nucleus RTOSには、イベントフラグを読み取った後にクリア（吸収）する機能があります。 すべてのブロックされたタスクが読み取り用のフラグを受け取ったときにフラグの吸収（削除）が発生するため、システムを簡素化するためにこの機能をNucleus SEから除外することにしました。 必要に応じて、フラグ読み取りタスクは、別のAPI呼び出しを使用して常にそれらをクリアできます。



未実現のAPI呼び出し



Nucleus RTOSは、イベントフラググループを操作するための7つのユーティリティコールをサポートしています。 これらのうち、3つはNucleus SEには実装されていません。 これらの課題の詳細、およびそれらをNucleus SEから除外する決定については、上記で説明しています。
