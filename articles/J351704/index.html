ES6モジュールの歴史

Developer Softのソフトウェアエンジニアで、 Netologyコースの教師であるVladislav Vlasovは、ブログ専用のEcmaScript6に関する一連の記事を執筆しました。 前半では 、例ではIroh.jsを使用してEcmaScriptでの動的コード分析を検討し、 後半ではキャンセルされたPromiseの実装に焦点を当てました。 この記事では、ES6モジュールの歴史について説明します。







EcmaScript言語の歴史は、ブラウザーの単純なスクリプト言語から、さまざまなホスト環境で動作する最新の汎用言語にまで及びます。 言語の複雑さとともに、モジュール構造を整理し、ライブラリに配置してコードを再利用する必要性が生じました。 最初のライブラリは、対応するJSファイルをプロバイダーのホストまたはCDNから読み込むことによってインポートされ、相互作用は、原則として、既知の名前を持つ関数とクラスをグローバルスペース（ウィンドウオブジェクト）にエクスポートすることによって実行されました。



このスキームは長い間使用されており、単純なケースでは非常にうまく機能します。

困難は、ライブラリが多すぎてライブラリ間の関係が始まったときに始まります。
まず、グローバルホストオブジェクトが詰まっており、接続されているすべてのライブラリは、競合しない一意の名前をインポートする必要があります。 第二に、ライブラリ間の相互運用性と再利用を保証する明示的な方法はありません。

トランスパイルおよびバンドルビルダーによるES6モジュールのサポート



Ecmascript 262バージョン6仕様とその後のエディションの出現により、多くの新しい構文構造とネイティブ関数が言語に追加されました。 原則として、それらのほとんどは、構文構築のための予備的なトランスパイル、および欠落している機能のためのポリフィルの追加により、古いバージョンのJSエンジンで簡単に起動できます。



ES6-モジュールは、ロードの同期ノンブロッキングセマンティクス、エクスポート/インポートされたエンティティのバインディングバインディング、モジュラースコープ、および通常のトランスパイルで提供するのが容易ではない他の側面を提供しました。

ES6モジュールのネイティブサポート



依存関係のバンドルを使用したスキームは、その時代に関連していましたが、現在、すべての最新のブラウザーはES6モジュールをネイティブにサポートしています。





これには、ブラウザーの不完全さと必要な機能の不足のためにbundle



が必要だったため、最新のWebアプリケーションのアセンブリに関するビューの改訂が必要です。 外観後、ネイティブデザインを使用すると、はるかに優れた結果が得られます。



第一に、構文構造の過度のtranspile



と、コードのエミュレートによる置換は、最適化のtranspile



低下と困難につながります。 これは、非同期およびジェネレーター関数に適用され、 リジェネレータランタイムに置き換えられ、レキシカル変数let



/ const



最適でないvar-declarationに変換されます。



もちろん、これはES6モジュール自体に直接関係するものではありませんが、通常はアセンブリとアプリケーションクライアントへの配信によって決定されます。 この意味で、これらは相互接続されたものです。



第二に、モジュールはパフォーマンスの点で効率的です。 ES6モジュールは、デフォルトで遅延モードでロードおよび実行されます。 つまり、誤ってブロッキングモジュールをWebアプリケーションに追加することは不可能であるため、すぐに使用できるSPOFの問題はありません。



ES6モジュールをサポートしていない古いブラウザーで操作性を維持するために、 bundle



して古いエージェントに提供することができます。 同時に、ES6モジュールのインポートの設計機能により、ブラウザー行のユーザーエージェントまたは機能検出ツールに応じて、提供されたコードの分離で条件付きでwebpackを構成する必要はありません。



区別するには、次のコードで十分です。 



 <html> <head> <script src="app/index.js" type="module"></script> <script src="dist/bundle.js" defer nomodule></script> </head> <!-- … --> </html>


ES6モジュールをサポートしないブラウザーは、単にdist / bundle.jsをロードし、古いスキームに従って動作します。 最新のブラウザはapp / index.jsをエントリポイントとして使用し、依存リソースを自動的にロードします。



上記のスキーム、モジュールの非同期および遅延ロード、依存関係キャッシュ、インラインモジュール、およびそれらのCORSポリシーのためにwebpackを効率的に構成する問題については、「 ES6モジュールがブラウザーのランドをサポート：バンドルを再考する時ですか？」 「および」 ブラウザのECMAScriptモジュール 」。



まとめ



EcmaScriptは長い道のりを歩んでおり、今日まで進化を続けています。 多くのソリューションはその時代に関連しており、クライアントエージェントにまだ組み込まれていない機能のプロアクティブなサポートなど、問題の解決を可能にしました。 現在、ブラウザーとNode.jsサーバーはバージョン更新を頻繁にリリースし、最新のEcmaScript機能を追加しています。



その結果、過去に一般的なバージョンのブラウザの新機能のサポートをエミュレートできるソリューションは、旧式のエージェントに適用できます。タスクによっては、個別に保守するか完全に除外するのが理にかなっています。



モジュールの事前認証とリンク、およびその後のバンドルは、最近までほとんどのブラウザーでES6モジュールをサポートする主な方法でしたが、現在ではそれらに悪影響を及ぼし、最適化ツールとキャッシュツールを妨げています。



したがって、Webアプリケーションのアセンブリを設定する際には、構文要素やモジュールのインポート/エクスポートなど、最新のエージェントに最新のEcmaScriptコードを提供することをお勧めします。
