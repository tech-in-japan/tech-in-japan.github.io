Web開発からゲーム開発への移行方法

こんにちは、Habr！ ほぼ10年半の間、Webサービスを作成および開発してきました。 あなたはそれらのいくつかを知っているか、使用の経験を持っているか、情熱的に愛している、または複雑な感情を持っているかもしれませんが、これは今ではありません。







そのため、2つのWebサイトビルダーが市場で知られています。uCozとuKit、 90 ％ 以上の登録が独自のサイトの作成に変換されなかった、少なくとも5 ％のオーディエンスを節約したいという意欲、そしてゲーム開発の経験はありません。 Web業界に関するゲームのリリースに必要なものではなかったわけではありませんが...そうですね。

ReactおよびReduxからVueおよびVuexへの移行



私たちは今、可能性のあるホリバーの薄い氷の上を歩いていることを理解しているので、私は私たちが何かを課したり主張したりしないことをすぐに予約します。 これは私たちの道、選択の理由と経験の単なる説明です。







ゲームを作成するプロセスの最初の革命は、非常にソフトではありますが、ReactとReduxからVueとVuexへの移行でした。







当社では、ほぼ同じ技術スタックですべての製品を開発しようとしています。 これは主に蓄積された専門知識の問題であり、その場合、開発チーム間での簡単な移動です。 今日の基本：NodeJS、React、MongoDB。







大量のデータと接続を使用してゲームを作成することは、最初はNoSQLでは馬鹿げていました。 その結果、私たち全員が移行したのは1週間流血でしたが、最初に最初にしたことです。







途中でReactからVueに変わったのはなぜですか？



新しいゲームデザイナーの登場により、ゲームの中心的な仕組みが変わり、ゲーム内のほとんどのインターフェイスが大幅に変更されました。 新しいメカニズムはVueで迅速にプロトタイプ化されました。この選択の基準は、技術を導入するための低しきい値でした。 同じ時期に、トラフィックの成長、利益、エネルギーのベクターシステムを思いついて導入し始めました。 それ以前は、データを時間内に書き込むだけでしたが、変化の速度には影響しませんでした。







最初は、ReactとともにReduxを使用しました。 スプレッドは非常に急速に成長しました。ゲーム内のWebサイトでのユーザーアクションごとに新しいエントリが作成されました。 したがって、ストーリー全体が変異し、ゲッターが再集計され、結果としてトラフィックと利益の複雑な計算が行われました。これはすべて非常に遅くなりました。 もちろん、MobXを使用して計算のロジックをやり直すこともできますが、この瞬間はセンターメカニクスの重大な変更と一致しました。 または星が一致しました。 彼らが複数の関係者をつないでReduxの問題を解決しようとしたところ、Vuexではすべてが箱から出して機能し、不必要なジェスチャーなしで任意の数のサブモジュールに分割されました。







Vueの構文シュガーと柔軟性は、本当に「ヒット」します。 たとえば、現在、コンポーネントの値を定期的に更新するために、計算されたプロパティの代わりに 







foo() { return bar + baz; }


書きます 







 foo() { return (this.oneTick, bar + baz); }


プロパティthis.oneTickには小さな魔法が隠されています。これはリアクティブで、1秒間に1回更新され、式bar + bazの結果が変わるとコンポーネントがレンダリングされます。







データベースの移行



バックエンドから、1つの小さな移行と1つの大きな移行がありました。 最初は、プロジェクトはMySQLで行われました。 それは迅速だったので、単純にリレーやその他の喜びが必要だと考えました。 少し成熟した後、かなり簡単にPostgreSQLに切り替えました。





より広範で複雑なのは、MongoDBへの移行です。 この決定は、容易なスケーラビリティと比較的高いパフォーマンスによるものでした。 ORMの存在にも関わらず、2番目の移動ではさらに多くの問題がありました。 しかし、標準のReplicaSetとAutoFailOverのセットアップには1時間しかかかりませんでした。
