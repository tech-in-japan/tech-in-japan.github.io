HL7開発ツール（HL7v2、HL7v3、CDA）

この短い記事は包括的なものではありませんが、HL7およびCDAのさまざまなバージョン用の開発ツールをいくつかリストしています。 ライブラリ、フレームワーク、または関連ツールがこのリストに含まれる基準は次のとおりです。ツールは、フリーまたはシェアウェアであり、組織または開発者の確立されたコミュニティ、好ましくはオープンソースによってサポートされている必要があります。 このため、TrifoliaやMirth CDAPIのようなあらゆる種類の良いものはリストされていません。多くの読者はほとんど聞いたことも出会うこともありません。 たとえば、HL7SDK、多くのHL7エディター、およびDICOM用の開発ツールはヒットしませんでした。



そして、非常に簡単な説明で順番に。



HL7v2ソフトウェア開発フレームワーク



  • HAPI（Java）


「HAPI（HL7アプリケーションプログラミングインターフェイス）は、Java用のオープンソースのオブジェクト指向HL7 2.xパーサーです 。 」 このライブラリのセットは、JavaでHL7v2アプリケーションを開発するための事実上の標準と考えることができます。 HAPIは、HL7プロファイルを生成することで拡張できます。 で利用可能-hl7api.sourceforge.net



  • メッセージングワークベンチ（MWB）


「 MWBは、Health Level Seven（HL7）V2.x実装者向けの多目的生産性ソフトウェアです。仕様とレポートの迅速な開発を促進します。 」

MWBは純粋な開発ツールではありません。 むしろ、それはHL7メッセージモデルエディターであり、これらのモデルを使用してHL7v2メッセージを検証する手段です。



リストに追加した理由は、HAPIでクラスを生成するために使用するプロファイルを作成できるためです。 たとえば、公式リリースがHL7v2.7をサポートしておらず、本当に実装する必要がある場合、MWBでデータ型とセグメントを複製および変更し、プロファイルをエクスポートして、それらを使用してHAPIでクラスを生成できます。



この製品は1人のプログラマのみがサポートしていました。 彼が別の世界に行き、いくつかのソースが保存されているディスクの暗号化された部分からパスワードを彼に持って行った後（そして、彼はもはや利用できない商用ライブラリを使用したことを考慮して）、このプロジェクトの開発は行き詰まりました。 機能が不明なEmbarcadero CodeGearに移植されたバージョンがあります。 誰かがCodeGearのエンタープライズバージョンのライセンスを所有しており、この製品の復活に費やす時間と希望がある場合は、 sourceforge.net / projects / mwbenchに直接お問い合わせください。



この開発ツールHL7v2のリストで、終了します。 追加するものがある場合は、コメント内で、上記の基準に従ってください。



HL7v3ソフトウェア開発フレームワーク



  • MARC-HI Everest Framework（Java、.NET、.NET CF）


「 エベレストフレームワークは、リモートシステムでHL7v3構造の作成、フォーマット、および送信を容易にするように設計されています。 」

エベレストフレームワークの機能の1つは、おそらく、MIF（Model Interchange Format）ファイルをサポートする数少ない開発ツールの1つであり、おそらくMIF2をサポートする唯一のツールであることです。 MIFが何であり、XMLスキーマとどのように異なるのかわからない場合は、Grahame Grieve氏によると、「 プログラマーを除き、人間はMIFを見るべきではありません 。

利用可能なフレームワーク-te.marc-hi.ca



説明の翻訳なしのさらにいくつかの手段。 すべてが明確であることを願っていますが、Javaはおそらく多くのCAPSを使用したでしょう。



  • Open eHealth Integration Platform（Java）


「 Open eHealth Integration Platform（IPF）は、Apache Camelルーティングおよびメディエーションエンジンの拡張機能です。 Groovyプログラミング言語に基づくアプリケーションプログラミングレイヤーがあり、eHealthドメインのメッセージ処理および接続システムを包括的にサポートしています。 IPFは、HL7固有の統合ソリューションだけでなく、汎用のエンタープライズ統合パターンを実装するためのドメイン固有言語（DSL）を提供します。 」

www.openehealth.org



  • Eclipse OHF HL7 v3 Tooling（Eclipse OHF H3ET）プロジェクト（Java）


“ 階層デザインビューとソースビューの両方でHL7 v3メッセージインスタンスを作成、編集、表示するためのツール。 Eclipse OHFプロジェクトは、コミュニティの関心がThe Open Health Toolsプロジェクトに転用された後、廃止されました。 」

wiki.eclipse.org/OHF_H3ET



  • HL7用のJava CAPSアダプター（Java）


「 TCP / IP HL7用Oracle Java CAPSアダプターは、Java CAPS ESBシステムがHL7データプロトコルを使用して外部TCP / IPアプリケーションとデータを交換できるようにするOracle Java Composite Application Suite（Java CAPS）のコンポーネントです。 」

この開発ツールではある種の跳躍が進行しているため、それが生き残るかどうかは明らかではありません。

docs.oracle.com/cd/E21454_01/index.html



  • Infoway Message Builder（Java、.NET）


Infoway Message Builderを使用すると、開発者は、汎カナダHL7メッセージングの異なるバージョン間の違いを抽象化し、現在の実装の制約をサポートすることにより、ソリューションを各電子健康記録実装に統合するビジネス上の課題に集中できます。 開発者は、好みのプログラミング言語を使用して、使い慣れた開発環境でインターフェイスを構築できます。一方、メッセージビルダーAPIは、HL7v3リクエストおよびレスポンスへの迅速かつ簡単な作成、取り込み、アクセスを促進します。 」

これとInfowayのすべてのツールは、実際の作業ではなく一般的な知識のために含まれています。 それらの使用は、ローカライズされた、各州ごとに異なるカナダのHL7v3バージョンの実装に意味があります。 どのように見えるかを見るのは理にかなっていますが。

infocentral.infoway-inforoute.ca/3_Tools_and_solutions



  • Infoway Message Remixer（Webベース）


Infoway Message ReMixerは、元の標準メッセージの整合性を維持しながら、管轄の要件を満たす汎カナダ標準（pCS）メッセージのローカライズを可能にするWebベースのアプリケーションです。 」

infocentral.infoway-inforoute.ca/3_Tools_and_solutions/Message_ReMixer

HL7v3メッセージ検証ツール



このセクションでは、HL7v3メッセージ検証ツール。



  • TestLevel 7（ウェブベース、30日間トライアル）


「 Intelliwareのテストレベル7（TL7）は、信頼できる医療相互運用性テストプラットフォームです。 このツールは、HL7 v3メッセージを迅速に検証し、Canada Health Infoway（CHI）が後援するフォーマット標準に準拠しているかどうかを確認します。 HL7v3 e-healthシステムの開発者およびメッセージモデラーは、独自のクライアントレジストリ、プロバイダーレジストリ、およびCeRxメッセージをツールにカットアンドペーストして、メッセージの有効性と公開CHI標準への適合に関するフィードバックを即座に取得できます。 」



繰り返しますが、これはいくつかのタイプのHL7v3メッセージのローカルバージョンを検証するためのカナダのツールです。 ただし、慣れるには適切です。

tl7.intelliware.ca/validateMessage.faces



  • HL7v3テストハーネス（.NET）


「 HL7テストハーネスシステムは、クライアントシステムやサーバーシステムのメッセージ処理アクションをシミュレートする機能も提供するHL7 v3メッセージ適合性テストアプリケーションです。 このアプリは、メッセージワークフロー、形式、およびコンテンツのテストを提供します。 」



正直に言うと、私はそれを使ったことがないので、この治療法がどのような状態にあるかを言うことはできません。



sourceforge.net/projects/hl7-testharness
