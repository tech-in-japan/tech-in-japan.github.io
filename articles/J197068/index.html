Audiofocus-オーディオサブシステムへのアクセス制御

これは、Google Developer RelationsのKristan Uccelloによるオーディオフォーカスの尊重に関する記事の翻訳です。



報告中に中断するのは失礼と見なされます;これは話者に対する無礼を示し、聴衆を悩ます。 アプリケーションがオーディオフォーカスを操作するためのルールを考慮していない場合、他のアプリケーションを尊重せず、ユーザーを悩ますことを意味します。 オーディオフォーカスについて聞いたことがない場合は、 Android開発者向けトレーニング資料のドキュメントをご覧ください。

複数のアプリケーションがオーディオを再生できる場合、それらがどのように相互作用するかを考えることが重要です。 すべてのプレーヤーが同時に再生される状況を回避するために、Androidはオーディオフォーカスの概念を使用してサウンドの再生を制御します。アプリケーションは、オーディオフォーカスを受け取ったときにのみオーディオを再生する必要があります。 この記事では、ユーザーのオーディオフォーカス状態の変化を適切かつ最適に処理するためのいくつかのヒントについて説明します。



オーディオフォーカスリクエスト



アプリケーションの開始時に欲張りになってオーディオフォーカスを要求する必要はありません。 アプリケーションがオーディオストリームで何かを開始するまで待つ方が良いでしょう。 AudioManagerサービスを介してオーディオフォーカスを受信する場合、 AUDIOFOCUS_GAIN *定数を使用して、目的のフォーカスモードを示すことができます。



フォーカスリクエストの例 
AudioManager am = (AudioManager) mContext.getSystemService(Context.AUDIO_SERVICE); int result = am.requestAudioFocus(mOnAudioFocusChangeListener, // Hint: the music stream. AudioManager.STREAM_MUSIC, // Request permanent focus. AudioManager.AUDIOFOCUS_GAIN); if (result == AudioManager.AUDIOFOCUS_REQUEST_GRANTED) {  mState.audioFocusGranted = true; } else if (result == AudioManager.AUDIOFOCUS_REQUEST_FAILED) {  mState.audioFocusGranted = false; }


この例では、一定のオーディオフォーカスを要求しています。 代わりに、一時的（ AUDIOFOCUS_GAIN_TRANSIENT ）フォーカスを要求できます。これは、最大45秒続くサウンドの再生に適しています。

別のアプリケーションは、オーディオサブシステムを他のアプリケーションと共有できる状況に「クワッキング」モード（ AUDIOFOCUS_GAIN_TRANSIENT_MAY_DUCK ）を使用できます（たとえば、フィットネスアプリケーションの「もっと燃やす」というフレーズで、バックグラウンドミュージックが中断されないことを期待します）。 「quack」モードでフォーカスを要求するアプリケーションは、オーディオサブシステムを15秒以上連続して使用しないでください。



オーディオフォーカスのステータスの変更を処理します



オーディオフォーカス状態変更イベントを処理するには、アプリケーションはOnAudioFocusChangeListenerインスタンスを作成する必要があります。 このハンドラーでは、 AUDIOFOCUS_GAIN *およびAUDIOFOCUS_LOSS *イベントを処理する必要があります。 AUDIOFOCUS_GAINイベントには、2番目の例で説明したいくつかの機能があることに注意してください。



イベント処理の例 
 mOnAudioFocusChangeListener = new AudioManager.OnAudioFocusChangeListener() { @Override public void onAudioFocusChange(int focusChange) { switch (focusChange) { case AudioManager.AUDIOFOCUS_GAIN:  mState.audioFocusGranted = true;     if(mState.released) {   initializeMediaPlayer();  }  switch(mState.lastKnownAudioFocusState) {  case UNKNOWN:   if(mState.state == PlayState.PLAY && !mPlayer.isPlaying()) {    mPlayer.start();   }   break;  case AudioManager.AUDIOFOCUS_LOSS_TRANSIENT:   if(mState.wasPlayingWhenTransientLoss) {    mPlayer.start();   }   break;  case AudioManager.AUDIOFOCUS_LOSS_TRANSIENT_CAN_DUCK:   restoreVolume();   break;  }     break; case AudioManager.AUDIOFOCUS_LOSS:  mState.userInitiatedState = false;  mState.audioFocusGranted = false;  teardown();  break; case AudioManager.AUDIOFOCUS_LOSS_TRANSIENT:  mState.userInitiatedState = false;  mState.audioFocusGranted = false;  mState.wasPlayingWhenTransientLoss = mPlayer.isPlaying();  mPlayer.pause();  break; case AudioManager.AUDIOFOCUS_LOSS_TRANSIENT_CAN_DUCK:  mState.userInitiatedState = false;  mState.audioFocusGranted = false;  lowerVolume();  break; } mState.lastKnownAudioFocusState = focusChange; } };


AUDIOFOCUS_GAIN定数は、コード内で2つの異なる役割で使用されます。 まず、例1のようにオーディオフォーカスを取得します。同時に、 OnAudioFocusChangeListenerイベントは発生しません。つまり、オーディオフォーカスの要求（および受信）が成功すると、ハンドラーは対応するAUDIOFOCUS_GAINイベントを受信しません。

AUDIOFOCUS_GAINは、 OnAudioFocusChangeListener実装でもイベントバリアントとして使用されます。 前述のように、オーディオフォーカス要求が行われたときにAUDIOFOCUS_GAINイベントは発生しません。 それどころか、対応するイベントAUDIOFOCUS_LOSS *が発生した後にのみ発生します 。 AUDIOFOCUS_GAINは、両方の状況で使用される唯一の定数です。

オーディオフォーカス状態変更イベントのイベントハンドラーで考慮する必要がある4つの状況があります。 アプリケーションがAUDIOFOCUS_LOSSイベントを受け取ったとき、これは通常、オーディオフォーカスを受け取らないことを意味します。 この場合、アプリケーションはオーディオサブシステムに関連付けられたリソースを解放し、再生を停止する必要があります。 例として、ユーザーがアプリケーションを介して音楽を聴き、オーディオプレーヤーからオーディオフォーカスを奪うゲームを起動するとします。 ユーザーがゲームを閉じる時間を予測することは不可能です。 ほとんどの場合、彼はメイン画面に移動し（ゲームをバックグラウンドに残して）、別のアプリケーションを起動します。 または、音楽プレーヤーに戻り、作業を再開します。これには、onResumeでのオーディオフォーカスの新しい要求が必要になります。

しかし、議論に値する別のケースがあります。 オーディオフォーカスを永久に失う（上記の例のように）か、一時的に失うかには違いがあります。 アプリケーションがAUDIOFOCUS_LOSS_TRANSIENTイベントを受信すると、アプリケーションはAUDIOFOCUS_GAINイベントを受信するまでオーディオの使用を一時停止することが予想されます。 AUDIOFOCUS_LOSS_TRANSIENTイベントが発生すると、フォーカスが戻ったときにどの動作が正しいかを判断するために、アプリケーションはフォーカスの喪失が一時的なものであることを覚えておく必要があります。 （例2を参照）。

時々、アプリケーションはオーディオフォーカスを失い（つまり、AUDIOFOCUS_LOSSを受信し）、中断されたアプリケーションが終了するか、他の方法でオーディオフォーカスを失います。 この状況では、オーディオフォーカスがあった最後のアプリケーションがAUDIOFOCUS_GAINイベントを受け取る場合があります。

後続のAUDIOFOCUS_GAINイベントでは、アプリケーションは一時的な損失後にオーディオフォーカスを受信したかどうかを理解する必要があり、単に再生を再開するか、フォーカスが完全に失われた後に再生を復元して調整する必要があります。

アプリケーションが短時間（45秒以内）だけオーディオを使用する場合、 AUDIOFOCUS_GAIN_TRANSIENTモードでオーディオフォーカスを要求し、再生または録音が完了したらすぐにリリースする必要があります。 システムのオーディオフォーカスはスタックのように扱われます。フォーカスは、それを最後に所有したアプリケーションに与えられます。

オーディオフォーカスを受信したら、MediaPlayerまたはMediaRecorderを作成し、リソースを予約します。 また、アプリケーションがAUDIOFOCUS_LOSSを受け取った場合、予約されているすべてのリソースを解放することをお勧めします。 オーディオフォーカスを取得するには、フォーカスを失うためのさまざまなオプションに対応する3つのオプションがあります。 OnAudioFocusChangeListenerハンドラーでフォーカスを失うためのすべてのオプションを明示的に処理すると便利です。

アプリケーションはAudioManagerにオーディオフォーカスを要求します（記事の最後にあるリンクの例のように）。 パラメーターは、オプションのハンドラー、オーディオチャネルのタイプ（表2）、および表1のオーディオフォーカスのタイプのヒントです。オーディオの初期化は、システムがオーディオフォーカスの受信を許可した場合にのみ行う必要があります（ AudioManager.AUDIOFOCUS_REQUEST_GRANTED ）。

注：電話での会話が発生した場合、システムはオーディオフォーカス（ AUDIOFOCUS_REQUEST_FAILED ）の受信を許可せず、通話の終了後にアプリケーションにAUDIOFOCUS_GAINイベントを送信しません。

OnAudioFocusChange（）イベントに対するアプリケーションの応答の簡単な説明を表3に示します。

オーディオフォーカスが失われた場合、フォーカスが完全に失われることを確認する必要があります。 アプリケーションがイベントAUDIOFOCUS_LOSS_TRANSIENTまたはAUDIOFOCUS_LOSS_TRANSIENT_CAN_DUCKを受信した場合、予約されたリソースを保持できます（release（）を呼び出さないでください）。 ほとんどの場合、オーディオフォーカスを変更する新しいイベントがすぐに発生します。 何らかのフラグで、または状態グラフの別の頂点に移動して、一時的にフォーカスが失われたという情報を保存する価値があります。

アプリケーションがAUDIOFOCUS_GAINモードで一定のオーディオフォーカスを要求し、AUDIOFOCUS_LOSS_TRANSIENT_CAN_DUCKイベントを受信した場合、適切な応答は、ボリュームを下げ（古いボリューム値を保持することを忘れないで）、AUDIOFOCUS_GAINイベントを受信したときにボリュームを返すことです（図を参照）。







表3 。 オーディオフォーカスの状態が変化したときのアプリケーションの反応。

フォーカス変更のタイプ

反応

AUDIOFOCUS_GAIN

フォーカスの喪失イベント後にイベントを受信する ：アプリケーションの状態がこれに矛盾しない場合、メディアの再生を再開します。 たとえば、ユーザーはフォーカス損失イベントの前に一時停止しました。

AUDIOFOCUS_LOSS

再生を停止し、リソースを解放する

AUDIOFOCUS_LOSS_TRANSIENT

再生を一時停止し、フォーカス損失が一時的であることを示すフラグを保存します。これにより、AUDIOFOCUS_GAINを処理するときに、必要に応じて再生を再開できます。 リソースを解放しないでください。

AUDIOFOCUS_LOSS_TRANSIENT_CAN_DUCK

AUDIOFOCUS_LOSS_TRANSIENTの場合のようにステータスを監視することを忘れないで、ボリュームを静かにするか、再生を一時停止します。 リソースを解放しないでください。
