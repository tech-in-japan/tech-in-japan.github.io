私たちには2つの正反対があります。1つ目は左足で作られたプロジェクトですが、予算と親会社からのトラフィックという形で堅実なリソースを持っています。 このようなプロジェクトは、管理の不備、および大規模なオフィスでの官僚主義と分離のために、数か月間行われます。 その過程で、プロジェクトの開始後、数十人の異なる人々が参加し、寄り添うマネージャーとテトリスは変化する可能性があり、最終的にプロジェクトは「母」を犠牲にしてバナーを外し、不運な子供にトラフィックをマージします。



2つ目の極端な例は、同様のプロジェクトです。サイトを分散し、できるだけ早く戦利品のハッキングを開始するために1-2人の開発者が行います（繰り返します：できるだけ早く）。 考えられるすべてのパラメーターが考慮され、最適なリーフィング、パッシブプロモーションを実現するための手段が取られます（「受精」ボタン、「ビジネスに投稿」ボタン、「友人にリンクを送信」サービスなど）。



2番目のケースでは、最初のケースとは異なり、予算も肥大化が少ないことが多いため、すべてのトラフィック、ギガバイトのトラフィック、余分なメガバイトのメモリ、サーバー上のディスクスペースをすべてカウントする必要があります。

鉄とプラットフォーム

ウェブサイトのプラットフォームについて空虚な議論に費やした時間は1時間ではありませんでした。 すべてのシギは彼の沼を賞賛し、私は私の賞賛します。 「所有コスト」というものがあります。 これらの2つの単語を覚えて、学生がJavaサーブレットのサイトを作成するか、オリジナルではない何かをする提案をしてあなたに次に来るときを思い出してください。 最も適切なプラットフォームを決定する最も簡単な方法は、ホスティングプロバイダーの統計を調べることです。 プロバイダーの99％は、Linuxプラットフォーム（FreeBSD）+ Apache + MySQL + PHP + Perlで仮想ホスティングを提供しています。



ここからダンス。 通常の仮想ホストを購入し、その上に新しいサイトを構築します。 市場には数百のPerlまたはPHPエンコーダーがありますが、1つまたは2つは間違いなく適切な価格で見つかります。 また、エンコーダが変更されている場合、エキゾチックなプラットフォームと環境が使用される状況とは異なり、同様のエンコーダを見つけるのに大きな問題はありません。



もう1つ良い点があります。プロジェクトを購入した仮想ホスト（VDSでさえも）で開発を開始すると、最初はサーバーリソースに依存しないため、多少最適なコードを作成し、ディスクスペースを慎重に消費します（そうでない場合、プロバイダーの管理者は曲がったスクリプトの食欲について迅速に報告します）。 このような経済的なプロジェクトがある時点で、適切に調整された別個のサーバー（またはクラスター）の形で「大邸宅」に落ちた場合、最適に設計されエンコードされたWebサイトはとにかく時計のように機能するため、あなたを称賛することは名誉なことです。
