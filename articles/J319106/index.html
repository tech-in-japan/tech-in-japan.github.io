Nuklearはマイクロプロジェクトに最適なGUIですか？

Nuklearは、即時モードのユーザーインターフェイスを作成するためのライブラリです。 ライブラリには依存関係はありません（C89のみ！ハードコアのみ！）が、オペレーティングシステムのウィンドウの作成方法や実際のレンダリングの実行方法もわかりません。 Nuklearは、実装されたアプリケーションを使用してレンダリングするための便利なインターフェイスを提供する埋め込み可能なライブラリです。 WinAPI、X11、SDL、Allegro、GLFW、OpenGL、DirectXの例があります。 コンセプトの親はImGUIライブラリでした。







Nuklearはなぜ美しいのですか？ 小さいサイズ（約15,000行のコード）で、1つのヘッダーファイルに完全に含まれており、移植性と使いやすさを重視して作成されました。 パブリックドメインライセンス。







Nuklear Webデモ







問題の声明



私はしばしば、数百行のコードで小さなユーティリティを作成しなければならない実装のためのタスクを持っています。 通常、結果は他の誰も実際に使用できないコンソールアプリケーションです。 おそらく、シンプルなGUIでこれらのユーティリティをより便利にできるでしょうか？







したがって、結果の要件：







  1. 数百キロバイトまでの小さいサイズ。
  2. クロスプラットフォーム、少なくともWindowsとLinux。
  3. Windowsの外部ライブラリに依存しないため、すべてが1つのEXEファイルに含まれている必要があります。
  4. まともな/美しい外観。
  5. JPGおよびPNG形式の画像のサポート。
  6. 開発の容易さ、WindowsおよびLinuxでの開発能力。

    Nuklearはそれを行いますか？








たとえば、dxBin2hユーティリティ（ GitHub ）の作成を検討してください-ファイルをバイト単位で読み取り、C配列の形式で書き込みます。 主な機能に加えて、プログラムには、不要な文字の削除など、あらゆる種類の「グッズ」があります。 通常、サードパーティの機能のために、独自の小さなユーティリティが作成されます。 たとえば、dxBin2hは、 Winter Novel用に作成され、ASCIIファイルを前処理します。







開発のしやすさ、クロスプラットフォーム



まあ、何で、そして問題の開発の単純さではいけません。 結局のところ、それに目を向けると、ライブラリが作成されましたよね？ GitHubのReadmeに例があります 。 絶対に明確で簡潔な20行のコードは、美しく明確な結果をもたらします。

Windowsへの依存なし



さて、OpenGLでSDL2レンダラーを使用して、Windows、Linux、Mac OS X、Android、iOSなどのアプリケーションを作成してください！ すべてがスーパーですが、標準のWindowsパッケージにはSDLライブラリはありません。 だから、あなたは一緒にドラッグする必要があります。 そして、これは最初の要件（小さなサイズ）に違反しています。 SDL自体の重量は約1メガバイトです。







しかし、例のリストにはGDI +があります。これはXP以降のWindowsにあります。 GDI +はttfフォント、PNGおよびJPGイメージを実行でき、これらはすべてメモリから直接ロードできます。 最終的に2つのレンダリングが可能になります。Windowsの場合はGDI +、その他の場合はすべてSDLです。 レンダリングに応じてコードの一部を別のCファイル（ nuklear_cross.c ）に移動できます。 そうすれば、メインコードはオーバーロードされず、インターフェイスに集中できるようになり、開発が大幅に簡素化されます。 追加のプラスはコンパイルの高速化です-Nuklear全体が個別のオブジェクトファイルにコンパイルされ、ほとんど変更されません。

フォント



すべてのオペレーティングシステムでアプリケーションを同じように表示するには、同じフォントを使用する必要があります。 どこかにあることが保証されているシステムフォントを使用できます。 しかし、そのようなフォントはありません。 したがって、アプリケーションにフォントを含める必要があります。 ttfフォントの重量は通常数百キロバイトですが、必要な文字を含むサブセットはそれらからうまく作成されます。 たとえば、 FontSquirrel Webサービスを使用します 。 DejaVu Serifは40kbに縮小しましたが、キリル文字、ポーランド語、および多数の言語が含まれています。







すべて問題ありませんが、NuklearのGDI +ドライバーは、ファイルからのみフォントをメモリからロードできませんでした。 修正する必要がありました ...ところで、同じdxBin2hを使用してアプリケーションにフォントを含めることができます。

画像：PNG、JPG



SDL2とGDI +の両方で画像をロードできます。 ただし、SDLの場合、JPGおよびPNGを読み込むと、SDL_imageという追加の依存関係が表示されます。 プロジェクトをSDLでビルドする場合は、 stb_image.hを使用してください 。







GDI +では、すべてがうまくいくわけでもありません。 つまり、NuklearのGDI +ドライバーは、GDI +を使用して画像を描画できませんでした。 画像を使って作業を掘り下げ、自分で実装する必要がありました（ Pull Request ）。 これですべてが修正され、コードが公式リポジトリに追加されました。







OpenGLのstb_imageを介した画像アップロードコード 
 struct nk_image dxNkLoadImageFromMem(const void* buf, int bufSize){ int x,y,n; GLuint tex; unsigned char *data = stbi_load_from_memory(buf, bufSize, &x, &y, &n, 0); glGenTextures(1, &tex); glBindTexture(GL_TEXTURE_2D, tex); glTexParameterf(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_LINEAR_MIPMAP_NEAREST); glTexParameterf(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_LINEAR_MIPMAP_NEAREST); glTexParameterf(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_CLAMP_TO_EDGE); glTexParameterf(GL_TEXTURE_2D, GL_TEXTURE_WRAP_T, GL_CLAMP_TO_EDGE); glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA8, x, y, 0, GL_RGBA, GL_UNSIGNED_BYTE, data); glGenerateMipmap(GL_TEXTURE_2D); return nk_image_id((int)tex); }

結果は何ですか？



  1. UPX 90kbを圧縮した後、200kbをコンパイルした後のWindows EXE。 Linuxでは、stb_imageを使用するため、アプリケーションのサイズは平均で100kb大きくなります。
  2. WindowsおよびLinuxで動作を確認しました。
  3. フォントと画像は、配列としてアプリケーションメモリに保存されます。 WindowsのWinAPIに依存しない依存関係はありません。
  4. アプリケーションスタイル変更エンジンが機能します。
  5. PNGとJPGは、GDI +とstb_imageを使用してロードされます。
  6. ダーティなプラットフォーム固有のコードはすべて別のファイルに移動されます。 開発者は、アプリケーションの作成に集中します。


既知の問題



  • オペレーティングシステムごとに異なるフォントスムージング
  • 異なるサイズのチェックボックス
  • 異なる画像のサポート（stb_imageを使用する場合、問題のある画像は避けてください）
  • 切り捨てられたフォントでの完全なUnicodeサポートではありません
  • Mac OS Xテクノロジーの例はありません


ベストプラクティスの使用方法



  1. リポジトリhttps://github.com/DeXP/nuklear_crossをプロジェクトにクローンします
  2. 「nuklear_cross / nuklear_cross.h」を接続し、そこから関数を使用します

アプリケーションは、オペレーティングシステムによって少し異なります。 しかし、違いはわずかであり、得られた結果は私を満足させました。 Nuklearは、「テストを行わなくてもどこでも機能するはずです」カテゴリにはありません。 ただし、「何か必要な場合-簡単に追加します」というカテゴリに分類されます。
