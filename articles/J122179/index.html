Linuxをゼロから作るユニバーサルディストリビューション

数年前、私が所属する開発チームに勤務していたので、予期しないタスクが落ちました-Linux用のコントロールPCを使用した機器管理システムの開発（これは驚きではありません）

Linux部分は、UbuntuのCode :: Blocks環境で開発されました（そして現在進行中です）。 しかし、実践が示しているように、質の高い作業を行うには、応答時間を保証した、もっと簡単なものが必要です。 ユーザーインターフェイスを整理するタスクは、TCP / IPを介して接続されたリモートコンピューターで解決されたため、コンソールは作業に十分でした。

それから、私たち自身のアセンブリのLinuxディストリビューションを使用するというアイデアが浮上しました。実際、これは（ディストリビューションアセンブリ）私の自由時間に行いました。 選択はLFSにかかった。 このようなLFSがHabréでも繰り返し記述されているという事実については、特定のケースで目の前で発生したいくつかの追加のタスク（単純なLinuxを除く）のソリューションについて説明します。

最初は、リアルタイムコアを使用するというようなタスクが1つだけありました。

しかし、さらに、配布キットを備えたUSBフラッシュドライブのアイデアがみんなの好みになったとき、フラッシュドライブを増やし、さまざまなコンピューターでシステムを起動するタスクがありました（多くのテストスタンドがあり、フラッシュドライブをポケットに入れていずれかに行きます）。 ここで問題が発生しました-LFSには、あるコンピューターから別のコンピューターへの100％の移植性がありません。 特定のコンピューターに適合させるには、いくつかのスクリプトを編集する必要があります。これは、昨日のWindowsエンコーダーのチームでは問題があります（一部はUbuntu仮想マシンに転送されましたが、コンソールとスクリプトは災害です）。 システムの複製には、アセンブリプロセス中に実行されるいくつかの操作の繰り返しも必要です（同じGRUBをインストールします）。



当然のことながら、すべての問題の解決策はインターネット上にありますが、一部の情報を1か所で収集しても、だれも傷つけることはないと思います。

したがって、特定のタスクは次のとおりでした...



1.リアルタイムパッチでLinuxカーネルを使用する


これは最も簡単なタスクの1つでした。 ビルドプロセスは1つの例外を除きLFSブックを通過しました。本の通常のカーネルの代わりに、2.6.33.9とそのためのRTパッチが採用されました。 カーネルが操作された場合（Linuxヘッダーとカーネルを直接インストールする場合）、パッチを適用したバージョンを使用します。

また、ディストリビューションキットはスワップパーティションを接続せずに構築されたと言えます（この場合、2GBのRAMは屋根の上にあり、スワップは保証された応答時間への悪影響のために望ましくなく、フラッシュドライブにとっては非常に破壊的です）単一のext2fsパーティション。



2.システムへの自動入力（開発環境では、セキュリティは重要ではありません。多くの理由により、システムをルートで実行できます）


自動ログインのアイデアは、ここから取り入れられました 。

autologin.cファイルは、次の内容で作成されました。 

int main() { execlp("login", "login", "-f", "root", 0); }


次に、ファイルは次のコマンドでコンパイルされました。



gcc autologin.c -o /usr/local/sbin/autologin







さらに、自動ログイン機能を備えた2台のコンソールで十分であることが決定されました（1台はシステムを起動し、もう1台は必要に応じて他のすべてを起動します）。

/ etc / inittabファイルでは、次の行：



1:2345:respawn:/sbin/agetty tty1 9600

2:2345:respawn:/sbin/agetty tty2 9600







次のものに置き換えられました：



1:2345:respawn:/sbin/agetty -n -l /usr/local/sbin/autologin tty1 9600

2:2345:respawn:/sbin/agetty -n -l /usr/local/sbin/autologin tty2 9600







その後、自動ログインに問題はなかったため、追加のスクリプト操作はありませんでした。



3.ドライブを接続する順序からシステムを解く



BIOSごとに問題があります。 たとえば、最初のドライブは起動元のドライブであると考える人もいます。 この場合、フラッシュドライブはsdaになります。 他の人は、最初にハードドライブを使用し、次に他のデバイスを使用する必要があると考えています。 この場合、フラッシュドライブの名前はsdb、sdcなどになります。

その結果、存在しないディスクからシステムを起動できず、/ etc / fstabで間違ったドライブが指定されているため、ルートディレクトリをマウントできません。

すべては、特定のマシンの/boot/grub/grub.cfgおよび/ etc / fstabを修正するか、ドライブ名（sda、sdbなど）を使用してダウンロードおよびマウントするのではなく、このファイルシステムのファイルシステムのUUIDを使用することで解決されますフラッシュドライブ上の一意であり、最も重要なことは、永続的です。

問題は、GRUBはUUIDで動作するが、カーネルではできないということです。つまり、UUIDで（システム名を知らずに）ルートシステムを直接マウントすることはできません。 これはバグではなく、Linus Torvaldsのイデオロギー的考慮の結果であるため、将来このような機会を期待するべきではありません。 それにもかかわらず、回避策があります-これはinitramfsです。

Initramfsは、実際のシステムのファイルシステムのダウンロードとマウントに役立つ一時ファイルシステムです。

Initramfsは標準のLFSアセンブリに含まれていないため、 Gentoo Wikiの推奨事項と独自の考慮事項を使用してビルドします（Gentoo Wikiのバージョンでは、ディスク名の問題は私のケースでは変更せずに解決せず、機能しませんでした）。

メインUUIDをマウントするシンプルなinitramfsシステムを作成するには、シンプルなシェルとinitスクリプトが必要です。 コマンドラインユーティリティの完全なセットはinitramfsにとって扱いにくいため、busyboxはこの目的によく使用されます。サイズと要件は控えめで、最も一般的に使用されるユーティリティのいくつかを実装します。

busyboxの最新バージョンを選択します。 



 cd sources 
      

        
        
        
      

     wget http://busybox.net/downloads/busybox-1.18.4.tar.bz2




開梱して構成：



tar jxf busybox-1.18.4.tar.bz2

cd busybox-1.18.4

make menuconfig







構成はメニュー（Linuxカーネルなど）を使用して行われます。 原則として、私たちのニーズには標準構成で十分ですが、念のため、次の機能が接続されていることを確認する価値があります。

devfsのサポート-/ devで作業するためのdevfsのサポート。

Busyboxを静的ライブラリ（共有ライブラリなし）-静的レイアウトとしてビルドします。これにより、多数のライブラリをドラッグしないようにします。

バージョン2.6.x Linuxカーネルのサポート-2.6カーネルコアのサポート。

ユーティリティの機能のサポート：sh、cat、cut、findfs、mount、umount、sleep、echo、switch_root。



コンパイルします：



make







次に、ファイルシステムのディレクトリツリーを収集します。



mkdir /usr/src/initramfs

cd /usr/src/initramfs

mkdir -p bin lib dev etc mnt/root proc root sbin sys

cp -a /dev/{null,console} /usr/src/initramfs/dev/







busyboxをコピーし、ユーティリティへのリンクを作成します。



cp /sources/busybox-1.18.4/busybox ./bin/

cd bin

ln -s busybox sh

ln -s busybox cat

ln -s busybox cut

ln -s busybox findfs

ln -s busybox mount

ln -s busybox umount

ln -s busybox sleep

ln -s busybox switch_root

cd ..







initスクリプトを記述することは残ります。 

 #!/bin/sh #    proc  sysfs mount -t proc none /proc mount -t sysfs none /sys # USB-     #     10 ,       sleep 10 #     mount -t devtmpfs none /dev #   UUID       for cmd in $(cat /proc/cmdline) ; do case $cmd in root=*) uuid=$(echo $cmd | cut -d= -f3) mount -o ro $(findfs UUID="$uuid") /mnt/root ;; esac done #      umount /dev umount /proc umount /sys #    exec switch_root /mnt/root /sbin/init




スクリプトを実行可能にします。



chmod +x /usr/src/initramfs/init







一時ファイルシステムをアーカイブに配置します。



cd /usr/src/initramfs

find . -print0 | cpio --null -ov --format=newc | gzip -9 > /boot/initrd.img-2.6.33-rt31







ファイル名に注意しましょう-これはカーネル名に対応している必要があります（これはGRUBがファイルを正しくピックアップするために必要です）。 つまり、カーネルの名前がvmlinux- 2.6.33-rt31の場合、initramfsの名前はinitrd.img- 2.6.33-rt31でなければなりません。



これでgrub-mkconfigが実行されると、GRUBはinitramfsを検出し、構成にルートシステムのUUIDも含めます。 確認するには、手で/boot/grub/grub.cfgを修正します。 たとえば、構成：



menuentry "Linux 2.6.33-rt31" --class gnu-linux --class gnu --class os {

insmod ext2

set root='(hd0,1)'

search --no-floppy --fs-uuid --set 47029df8-8567-417d-b813-eedfe1ff8b0f

linux /boot/vmlinux-2.6.33-rt31 root=/dev/sda1 ro

}







修正する：



menuentry "Linux 2.6.33-rt31" --class gnu-linux --class gnu --class os {

insmod ext2

set root='(hd0,1)'

search --no-floppy --fs-uuid --set 47029df8-8567-417d-b813-eedfe1ff8b0f

linux /boot/vmlinux-2.6.33-rt31 root=UUID=47029df8-8567-417d-b813-eedfe1ff8b0f ro

initrd /boot/initrd.img-2.6.33-rt31

}







ファイルシステムのUUIDは、次のように確認できます（たとえば、/ dev / sdb1の場合）。



blkid -p -o udev /dev/sdb1







/ etc / fstabを修正し、次の行を置き換えます：



/dev/sda1 / ext2 defaults 1 1







に



UUID= 47029df8-8567-417d-b813-eedfe1ff8b0f / ext2 defaults 1 1







上記のすべての操作では、カーネルがdevtmpfs（CONFIG_DEVTMPFS = y）およびinitramfs（CONFIG_BLK_DEV_INITRD = y）をサポートする必要があることにも注意する必要があります。



4.ネットワークカードからシステムを解除する


コンピューターに複数のネットワークアダプターがインストールされている場合、カーネルモジュールの並列読み込みは、これらのボードへの名前の一定の割り当てを保証しません。 たとえば、2つのボードがあります。 Board_1にはeth0システムのインターフェース名があり、Board_2にはeth1があります。 次回の再起動時に、Board_1がeth1になり、Board_2がeth0になることがあります。

このため、LFSは名前を特定のボードにバインドします。 別のコンピューターで起動する場合、ネットワークが立ち上がらない可能性が非常に高くなります。

私の特定のケースでは、すべてのコンピューターのボードは同じであり、IPは静的です（ターミナルコンピューターとの直接通信のみ）。

LFSでのネットワークインターフェイスの引き上げは、スクリプト/etc/rc.d/init.d/networkによって実行されます。 スクリプトを追加して、構成ファイル/etc/udev/rules.d/70-persistent-net.rulesが生成され、作業が完了するとこのファイルが削除されるようにします。 もっと簡単な方法があると思いますが、見つかった方法は機能しており、システムの組み立て時にUdev操作の原理を掘り下げる特別な欲求はありませんでした。

caseコマンドの開始セクションの先頭に、次を追加します。 



 for NIC in /sys/class/net/* do INTERFACE=${NIC##*/} udevadm test --action=add $NIC done




そして、停止セクションの最後（;;の直前）に追加します： 



 rm /etc/udev/rules.d/70-persistent-net.rules




これで、任意のシステムで起動すると、ネットワークインターフェイスの名前がeth0になり（最もエキゾチックな場合を除く）、ネットワークが上昇します。 もちろん、ipv4ファイルがあるディレクトリ/etc/sysconfig/network-devices/ifconfig.eth0が存在する必要があります。 このファイルの内容は、LFSブックに記載されています。



5.任意のフラッシュドライブにLFSをインストールするスクリプトを作成する


最後に残っているのは、システムのアーカイブと、任意のメディアにインストールするスクリプトを作成することです。

（USBフラッシュドライブからではなく）別のシステムを起動し、たとえば/ mnt / usb-osにUSBフラッシュドライブをマウントします。 コンテンツをアーカイブします。



cd /mnt/usb-os

tar -cvjf ~/pack.tar.bz2 *







install_usb-os.shをインストールするスクリプトを作成しています。 パラメーターとして、スクリプトは、システムを展開するデバイスの名前（/ dev / sdbなど）を取ります。 スクリプト自体が必要なパーティションとファイルシステム（名前/ dev / sdbが指定されている場合は/ dev / sdb1）を作成し、アーカイブを解凍してGRUBをインストールします。



ルート権限で実行する必要があり、実際、非常に危険です。
