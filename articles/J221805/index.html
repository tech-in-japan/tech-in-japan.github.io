このため、HTTP 1.1は概して開始され、 HTTP 2.0 （ spdyで表される）もこの方向で開発されています。 GoogleとのIETFは現在、現在のネットワークアーキテクチャを最大限に活用するために、それを破壊することなく最善を尽くしています。 そして、これを行うことができます...まあ、はい、TCP接続を可能な限り効率的に使用し、 多重化 、パケット損失からの回復などを通じて帯域幅を可能な限り密に使用します

システム管理者にとって、これは獣ではなく、夜の死体です。 HTTP 1.1アップグレードを通じて「巧妙な」方法で、ブラウザはHTTP接続の「タイプ」を変更し、開いたままにします。

開発者の観点から-もちろんこれは便利ですが、ブラウザとサーバー上のWebアプリケーションの間に二重チャネルが表示されます。 ストリーミングが必要ですか、メッセージが必要ですか。 しかし：

  4. 多重化なし。その結果、各接続の各帯域幅が非効率的に使用されます。
  5. サーバーには、ブラウザからの直接のTCP接続が多数あり、「データベースにとって厄介な」ことをしています

設定ポイントを測定して対応する方法を学びます。 Webアプリケーションのログを処理し、それらの遅いクエリを処理します。 ナビゲーションタイミングAPIのおかげで、クライアント側の速度も測定できるようになりました。ブラウザでパフォーマンスデータを収集し、JavaScript経由でクラウドに送信し、ピンバで集計し、逸脱に対応します。 非常に便利なAPIです。必ず使用してください。

  • データベースのクエリが遅い。 解決策は、クエリの最適化、緊急時の非正規化です。
  • アプリケーションコードの実行が遅い。 解決策は、アルゴリズムの最適化、キャッシュです。

  • クライアントへの静的リソースの戻りが遅い。 ソリューションはCDNです。
  • ブラウザでサーバーへの保留中の接続をロックします。 解決策はドメインシャーディングです。
  • ロングポーリングは、顧客がヒットするよりもサーバーに大きな圧力をかけます。 ソリューションは、サーバー送信イベント、Webソケットです。
  • 遅い、Webソケットは不安定です。 解決策は、それらのTLS（wss）です。
