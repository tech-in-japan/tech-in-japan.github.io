サーバー送信イベントを使用したリアルタイムアプリケーションの作成

つい最近、 Firefox 6がSSE （Opera 10.6 +、Chrome、WebKit 5 +、iOS Safari 4 +、Opera Mobile 10+で既に利用可能） を受信することがわかったため、すべてのブラウザーの半分以上（ユーザーにリーチする）のサポートはもはや有利ではありません山。 この技術を詳しく見てみましょう。 SSEは7年以上前にIan Hickson によって提供されましたが、1年前にブラウザに表示されるようになりました。 WebSocketがあるのに、なぜ別のプロトコルが必要なのですか？！ しかし、すべてに長所と短所があります。SSEが何に役立つか見てみましょう。



SSEの概念は単純です。クライアントはサーバーイベントにサブスクライブし、イベントが発生するとすぐに、クライアントは通知とこのイベントに関連するデータをすぐに受信します。 SSEプロトコルの有用性を理解するには、イベントを受信するための通常の方法と比較する必要があります。その本質を簡単に説明します。



ポーリング



最も単純ですが、最も効果的な方法ではありません。クライアントは数秒ごとにイベントを求めてサーバーをポーリングします。 何も存在しない場合でも、クライアントはすべて同じように要求を行いますが、何が来るかわかりません。

長所：

-ただ

-データを振ることができます

短所：

-多くの追加リクエスト

-イベントは常に遅れる

-サーバーは、クライアントがイベントを取得するか、イベントが期限切れになるまでイベントを保存する必要があります



長いポーリング



以前の方法の改良版。 クライアントはサーバーにリクエストを送信し、サーバーはデータが到着するかクライアントが自動的に切断するまで接続を開いたままにします。 データが到着するとすぐに、応答が送信され、接続が閉じられ、次の接続が開かれます。

ポーリングと比較した長所：

-リクエストの最小数

-イベントの高い時間的精度

-サーバーは再接続中にのみイベントを保存します

ポーリングと比較した短所：

-より複雑なスキーム



Websocket



これは、クライアントとサーバーが同等の条件で通信できるようにするバイナリ二重プロトコルです。 このプロトコルは、ゲーム、チャット、およびリアルタイムに近い非常に正確なイベントが必要なすべてのアプリケーションに使用できます。

ロングポーリングと比較した長所：

-1つの接続が上昇する

-イベントの非常に高い時間的精度

-ブラウザーによって制御されるネットワーク障害管理

ロングポーリングと比較した短所：

-HTTPは互換性のあるプロトコルではないため、独自のサーバーが必要です。デバッグは複雑です



私たちはこのような美しいWebSocketプロトコルを持っているので、なぜSSEを使用する必要がありますか？！ まず、すべてのWebアプリケーションが双方向通信を必要とするわけではありません-SSEも適しています。 次に、SSEはHTTP互換プロトコルであり、任意のWebサーバーでイベントブロードキャストを実装できます。



サーバー送信イベントプロトコル



クライアントはサーバーに要求を送信し、サーバーは応答として次のヘッダーを送信します。 

Content-Type: text/event-stream


また、接続を閉じません（phpでは、無限ループを作成できます。node.jsで行う方法については、サンプル記事で説明します）。 それだけです-SSEは動作します！ クライアントにデータを送信するために、サーバーは次の形式の行をソケットに書き込むだけです。 

 data: My message\n\n


複数行のデータを送信する必要がある場合、形式は次のようになります。 

 data: {\n data: "msg": "hello world",\n data: "id": 12345\n data: }\n\n


ここでは、原則として、プロトコルベース全体です。 さらに、サーバーはメッセージIDを送信できます。これは、接続が切断された場合に必要です。 接続が切断された場合、クライアントは接続しようとすると、特別なヘッダー（Last-Event-ID）を送信して、失われたイベントを復元します。 

 id: 12345\n data: GOOG\n data: 556\n\n


エラーが発生した場合の再試行時間： 

 retry: 10000\n data: hello world\n\n


idおよびretryフィールドはオプションです。



クライアントでは、すべてが次のようになります。 

 var source = new EventSource('http://localhost/stream.php'); source.addEventListener('message', function(e) { //  -  console.log(e.data); }, false); source.addEventListener('open', function(e) { //    }, false); source.addEventListener('error', function(e) { if (e.eventPhase == EventSource.CLOSED) { //   } }, false);


すべてが非常に簡単です。 SSEプロトコルに基づいてアプリケーションを構築しましょう。 いつものように、これはチャットになります。



マルチパートXMLHTTPRequest



マルチパートストリーミングとも呼ばれます（Firefoxのみをサポート）。 SSEプロトコルに非常に似ています。

ヘッダーの形式は次のとおりです。 

 Content-type: multipart/x-mixed-replace;boundary=smthing


そして、パーツは次の形式で送信されます。 

 Content-type: text/html\r\n\r\n --smthing\n Message\n --smthing\n




通常のXHRがクライアントで作成されますが、リクエストを送信する前にフラグreq.multipart = true;



設定する必要がありreq.multipart = true;

SSEに関する一連のイベント（オフラインになる、オンラインになる、メッセージ）を受け入れます。 なぜなら SSEはメッセージを送信できないため、HTTPを介して送信します。



作業のスキームは次のとおりです。

-チャットに入るとき、名前が要求されます

-クライアントはチャットサーバーに接続します。 イベントストリームが作成されます。

-クライアントが接続すると、チャットは全員にイベントを送信します：％username％online

-クライアントを切断すると、チャットは全員にイベントを送信します：％username％offline

-クライアントはHTTP「POST /メッセージ」を介してメッセージを送信でき、サーバーはこのメッセージを受信し、SSEを介してすべてのクライアントに受信したメッセージを送信します。
