問題履歴



さまざまな企業の同僚と話をした後、私は多くの人が会社の従業員のワークステーションをバックアップすることを怠り、本番アプリケーションのコピーのみに限定していると結論付けました。



一方では、これは理解できる-「戦闘」サービスであっても、「大衆に行った」確立された慣行はまだ多くない。 したがって、重要なサービスに対して優れたRTOおよびRPOパフォーマンスを確保することは、大企業および中規模企業のIT部門の主な頭痛の種の1つです。 カスタムマシンのバックアップなど、些細なことに時間を費やすことは許されない贅沢です。



ただし、実際には、ワークステーションの障害により、古いデバイスから新しいデバイスへのハードドライブの転送（その後のインベントリおよびアカウンティングの困難性）、またはユーザーデータの完全な損失が発生することを意味します。



もちろん、重要なデータをNASサーバーや企業のファイルホスティングサービスに保存することは、長い間良い形と考えられてきました。 この場合、すべての重要な情報は保存されますが、一時的/重要ではないデータは失われます。 さらに、アイロンを交換した後、ユーザーは自分の職場をパーソナライズし直す必要がありますが、それは彼の気分を高めません。



おそらく、これはビジネススケールでは重要ではありませんが、実際には運用上の遅延につながり、故障が発生した場合のダウンタイムを増加させ、IT部門と内部顧客の間の緊張をわずかに増加させます。 そして後者は、多くのITマネージャーが不当に無視する重要な要素です。



解決策


この問題は、いくつかの方法で解決できます。

1.すべてをそのままにします。

2.従業員の職場を仮想化します。

3.ジョブのバックアップサービスを実装します。



もちろん、2番目のシナリオは、さまざまなソースで繰り返し説明され、実証されている最大の利点を提供します。 ただし、これは実装するのにかなり高価なソリューションです。 そして、職場を組織化する「古典的な」スキームから仮想化されたものへの移行は、技術的および運用上の多くの困難に関連しています。



一方、一元化されたバックアップツールは実装が簡単で、インフラストラクチャの大規模な再構築を必要とせず、多くの場合より予算がかかります。

アバマール

どのように機能しますか？



Avamarは、従来のサーバー/クライアントスキームに従って動作します。 クライアントアプリケーションは、保護されたマシンにインストールされ、データを収集、重複排除（重複データを除く）、および送信します。

したがって、コピーは常に増分です-つまり 最初に、システムはデータの完全なコピーを作成し、次にブロックのみを変更します。 これにより、チャネルの負荷とストレージ容量のコストを大幅に削減できます。 増分バックアップと混同しないように、Avamarは合成完全バックアップを保持しています。



そして、Avamarの仕事の重要な原則は、まさに重複排除アルゴリズムです。

一言で言えば、このテクノロジーは、特定のコンピューターからだけでなく、バ​​ックアップがすでに作成されているすべてのソースからの不要なデータ転送を排除します。



以下は、Avamarで使用される重複排除アルゴリズムのもう少し詳細な説明です。
Avamarでは、データの「ソース」で、いわばクライアントで重複排除が行われます。

たとえば、誰もバックアップしたことがないまったく新しいファイルをバックアップするタスクを分析してみましょう。

1）バックアップのタスクが来ました。

2）クライアントAvamar'aは、プロセスavtar.exeを開始します。これは、重複排除のさらなるプロセスに関与しており、最初にすべての処理がファイルレベルで行われます。

3）SHA-1アルゴリズムを使用して、ファイルメタデータの160ビットハッシュ値が計算されます。

4）Avtarは、「ファイルハッシュ」のデータベース-F_cache.datで受信したハッシュを探します。 クライアントのC：\ Program Files \ avs \ varなどにあり、バックアップ中にF_cache.datはRAMに完全にアンロードされます。

5）ファイルが新しいため、そのハッシュはクライアントのF_cache.datで検出されず、同じF_cache.datのAvamarのサーバー側で検索が実行されますが、すべての以前のバックアップからハッシュデータベースが既に収集されています。ワークステーション;

6）しかし、ハッシュ値はありません。 その後、avtarプロセスはブロックレベルに移行し、すでにデータブロックのバックアップについて説明しています。

7）データは、1 Bから64 KB、平均25 KBのさまざまな長さのセグメントに分割されます。

8）その後、セグメントは30〜50％圧縮されますが、セグメントが25％を超えて圧縮されない場合、圧縮せずにさらに処理が行われます。

9）160ビットのハッシュは、圧縮/非圧縮セグメントごとに計算されます。

10）ハッシュは、8,000〜30,000ハッシュの複合に結合されます。

11）次に、ハッシュ合成のハッシュが計算されます。

12）複合材料のハッシュハッシュは再び複合材料に結合されます。

13）そして大きなルートハッシュが計算されます-これは複合ハッシュの複合ハッシュのハッシュです

14）Avtarは、すでに「ブロックハッシュ」のデータベース（P_cache.dat）で受信したルートハッシュ値を探します。

15）クライアントでルートハッシュが見つからない場合、検索はAvamarのサーバー側で行われます。

16）Avamarサーバーでルートハッシュが見つからない場合、検索はハッシュコンポジットのハッシュによって既に実行されており、まずクライアントで、次にサーバーで実行されます。

17）少なくとも何らかのハッシュが見つかるまで続きます。

18）すべての一意のデータブロックが見つかり、そのハッシュがどこにも記録されていない場合にのみ、ワークステーションからavamarサーバーへのデータ転送が開始されます。

19）avamarのサーバーでは、保存されたばかりの「データブロック-ハッシュ」の束が一意のインデックスの下に記録されます。

20）バックアップマップが作成されます-これらはファイルおよびブロックへのリンクです。

21）そして、これらはすべてPostgreSQLデータベースで行われます。

22）バックアップタスクの番号が記録され、それに応じて、どのクライアントからバックアップが作成されたか、バックアップを保存する時間、インデックスツリー、バックアップカードなどが決定されます。 など



このようなアルゴリズムには、コインの2つの側面があります。



1.これにより、データ伝送チャネルの幅を大幅に節約でき、リモートサイト（例：ブランチネットワークのワークステーション）のソースからでもバックアップできます。

2.ただし、ソースでの重複排除には、プロセッサパフォーマンスとRAM容量の約5〜10％が必要です。 サーバーにとっては、これは非常に重要な数字になりますが、ワークステーションにとっては目に見えません。



EMC Data Domainと統合する場合、サーバーはローカルストレージにメタデータのみを保存し、データ自体は重複排除後にすでに圧縮され、Data Domainに配置されます。 このようなスキームは、DDで使用されるブーストテクノロジーで機能します。

Avamar仮想アプライアンスのインストールプロセス



Avamar仮想アプライアンスのインストールプロセス
VMwareクラスターが高くなっているため、インストールは非常に簡単です。

1.パートナーリソースEMC .ovfからダウンロードした仮想マシンのイメージと.vmdkディスクをインストールパッケージと共に展開します。

2. AVEのバージョンに応じて、必要な数の仮想ディスクを作成し、AVEでVMに接続します。 私たちの場合、これらは3つの250 GBディスクです。

3. VMを展開して起動すると、必要なすべてのリポジトリが内部にある、裸のSUSE Linux Enterpriseが取得されます。 接続は仮想コンソールvSphereを介して実行され、管理はコンソールを介して実行されます。 スクリプト/usr/local/avamar/bin/ave-part.plを使用して、AvamarをインストールするOS を準備します。

4.ネットワークインターフェイスを設定します。 これを行う最も簡単な方法はdpnnetutilユーティリティを使用することです。これはすでに仮想マシンの一部ですが、 YaST2を使用することもできます。 ネットワークインターフェイスの構成が完了したら、VMを再起動する必要があります。

5. ./usr/local/avamar/src/avinstaller-bootstrap-version.sles11_64.x86_64.runコマンドでAVEインストールウィザードを展開します。 その後、起動します。 インストールウィザードには、 Avamar_Server_IP ：8543 / avi / avigui.htmlという便利なWebベースのインターフェイスがあります。







Avamarのインストールおよび構成プロセスには時間がかかり、ユーザーの参加が必要になります。







8. Webインターフェースでインストールが完了したら、コンソールで最終スクリプトを実行します

./usr/local/avamar/bin/ave-post.sh



9.サーバーのインストールの最後に、管理コンソールをダウンロードしてインストールする必要があります。 これを行うには、ブラウザーでサーバーのブラウザーIPアドレス（ステップ4で設定したもの）を開くだけです。 Webインターフェースを使用して、さまざまなオペレーティングシステム用のクライアントと、Windows用のベアメタルユーティリティをダウンロードできます。 また、Avamarでできることすべてに必要なすべてのドキュメントがあります。

ドメイン


集中管理の概念には、階層的な「ドメイン」が含まれます（Windows ADと混同しないでください）。 実際には、クライアント、コピーポリシー、ユーザー、および委任された権限のリストを含む、ツリーのようなフォルダー構造にすぎません。



したがって、コピーポリシーを使用して、保護されたデバイスをさまざまなグループに結合し、さまざまなユーザーのアクセスをそれらに制限できます。

VMwareとの統合


Avamarでは、VMware vSphere ESXiハイパーバイザーで実行されている仮想マシンをコピーできます。 このため、各VMにクライアントをインストールする必要はありません。 ハイパーバイザー全体のクライアントとして機能するクラスターに仮想マシンを展開するだけで十分です。 これはAvamarプロキシと呼ばれ、バックアップにはVADP（データ保護用のvStorage API）テクノロジーを使用します。 AvamarをESXiと統合するには、vCenterが必要です。

インストールプロセス
Avamarプロキシ配布はWebインターフェースからアクセスでき、.ovaファイルです。

.ovaをデプロイしてシステムをロードすると、仮想マシンのネットワークインターフェイスを構成するためのスクリプトがすぐに開始されます

次に、AvamarサーバーのあるVMの/usr/local/avamar/var/mc/server_data/prefs/mcserver.xmlファイルで、次の行にTRUEを書き込みます。entry key = "allow_duplicate_client_names" value = "true"



VMware仮想マシンのみをコピーする場合は、VDP-A製品-vSphere Data Protection Advancedを購入できます。 この製品もVADPテクノロジーに基づいています。 本質的に、VDP-AはAvamar Virtual Editionであり、VMwareの下で「剥ぎ取られ」、基盤となっています。 仮想マシンと一部のMicrosoftアプリケーションをバックアップする機能のみを保持しているため、EMC AVEよりも安価です。

その結果、追加されたシステムが表示されます。
