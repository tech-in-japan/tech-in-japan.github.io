ただ何？ 最初に、顔検出について説明しましょう。 インターネットには、画像の分類に関する記事がたくさんあります。 しかし、まず、分類は必要ありませんが、検出は必要です！ そして、それははるかに複雑です。 幸いなことに、最初の重要な洞察がここに表示されます。検出を分類に減らすことができます。 ネットワークを取得して、画像全体ではなく、この画像内のすべての正方形のウィンドウに適用してみましょう。 より正確には、すべてではなく、（x、y）に沿ったオフセット（4 i、4 j）を持ついくつかのサイズのウィンドウで、i、jは（0、ceil（w / 4））、（0、ceil（h / 4））。

アルゴリズム（ハーフコード）が判明します。 



def windows(img): window_size = min(img.height, img.width) while not_too_small(window_size): y = 0 while y < img.height: while x < img.width: yield (x, y, x + window_size, y + window_size) x += dx if pixels_x_left_unyilded(): yield (img.width - window_size, y, img.width, y + window_size) y += dy if pixels_y_left_unyilded(): while x < img.width: yield (x, img.hight - window_size, x + window_size, img.hight + window_size) x += dx window_size /= resizing_factor dx /= resizing_factor dy /= resizing_factor


合計で、いくつかの異なる縮尺で画像のサイズを変更し、各縮尺に対して、4ピクセルのステップで固定サイズのウィンドウを駆動します。 そして、各正方形の顔を分類します。 その後、もちろん、各人にライトアップされた多数の正方形を取得するので、それらを何らかの形でマージする必要があります。 アルゴリズムは異なると考えられますが、賢くはありませんが、強く交差する中で最も照らされた正方形を選択します：

増強



さて、私はより多くのデータが必要だと考えました-もっとデータをやりましょう！ そして、彼が後で学んだように、データ拡張と呼ばれるプロセスを実装しました：これは、変換が元のデータに適用されるときに行われ、トレーニングサンプルのサイズを効果的に数百倍、さらには数千倍にします。



したがって、各例では、0.5の確率で、顔/顔ではないクラスがこれから変化しないため、水平に反映します。 また、各人について、データセットに存在する楕円から取得した元の正方形を取得しませんでしたが、最初の正方形は特定の間隔でランダムにわずかに増加（または減少）し、次に、誤ってxにわずかにシフトし、一定の間隔でy。

さらに、後でこの正方形自体を取得しないように思いつきましたが、xまたはy（50/50）でランダムにわずかに引き伸ばされました。 その結果、顔から写真から長方形が切り取られ、圧縮によって正方形に変換する必要があります。 この変換の違いは、前の変換は正方形のサイズと位置を変更しますが、顔自体は変形しないことです。この変換は顔を変形します。垂直または水平のいずれかにわずかに伸縮します。

最適化i



確かに、少量のトレーニングデータの見返りに、別の問題がありました。潜在的に無限の量のデータには、潜在的に無限に強力なコンピューターが必要です！ またはスマートなトリック。 私はソフトウェアをより専門にしているので、次回は無限に強力なコンピューターを構築することを決定し、現在の問題をスマートトリックで解決します。トレーニング用にデータセットを事前生成せず、完全にメモリにロードしますが、特定のインデックスを作成します最初の不完全なデータセット。実行時に直接補足します。 実際、反復中、ミニバッチ内の各オブジェクトに対して、上記のアルゴリズムを呼び出し、更新されたデータをその場で生成します。



ちなみに、面白い話はこれに関連しています。 CPUでこれをすべてデバッグし、GPUで実行を開始すると、パフォーマンスがおかしくなりました。自分で判断すると、増強アルゴリズムには多数のロジックが含まれており、その後、画像のリフレクションとサイズ変更も行われます！ したがって、ミニバッチの1024要素すべてに対して、反復ごとに数十のミニバッチを掛けます！



一般的に、もちろん、これはカテゴリ的に私に合わなかったので、整理しに行きました。 そして、すべてが非常に論理的であることが判明しました。シングルスレッドpythonプログラムでは、GPUは単純にスリープしますが、CPUはゆっくりとミニバッチを生成します。 怠azineは悪いです、私たちはそれと戦わなければなりません、そして私はGPUが、私が利用できるあらゆる瞬間、ギャレーの奴隷のように動作するべきだと決めました！



解決策？ GPUをブロックしないでください！ さあ、GPUが現在のネットワークでネットワークを学習している間に、CPUは次のバッチを非同期で準備します。 だから私はやった。 そして、私はこの手術の成功を確信していましたが、がっかりしました。それはほとんど助けになりませんでした。 特に小規模ネットワークでは、CPUがGPUよりもはるかに長く実行され、GPUはほとんどの時間スリープすることが判明しました。



さて、私はこれらのすべてのWebサービスを設計しましたか？ 大量のオブジェクトの並列読み取り専用処理を最適化する方法は？ シャーディング（別名MapReduce）！ たくさんのプロセスを開始して、それぞれが処理するミニバッチの一部を与え、結果をキューに入れて、待機せずにすぐに次のミニバッチの一部を処理します。 さらに、このキューをリッスンする別のプロセスを開始し、ピースがどのミニバッチに属しているかを理解し、ピースからミニバッチを組み立てて、2番目のターンでスタックします。GPUで動作するメインプロセスは既にデータを取得しています。



そして最後に、32個のコンピューティングコアを備えたサーバーで武装し、32個のプロセスで拡張データの生成を開始し（2つはほとんど常にインラインまたはGPUで待機しています）、2番目の時代から始まって、GPUはほとんど不要にダウンロードしました。



ほら、教えて！



そのため、通常のサイズのおもちゃ以外のモデルを既に教えることができ、成功しました！、適切な設定で、再訓練せず、低域の外に出かけることはありません。
