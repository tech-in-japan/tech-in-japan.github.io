ビッグスリーのサーバーのエラーと問題：パート3。 Ibm





こんにちは、Habr！ 以前の記事では、 DellおよびHPサーバーのエラーと問題に対処しました。 再生されたサーバーエラーに関するストーリーは、3番目のビッグスリーベンダーであるIBMの製品について言及しなければ完了しません。 この栄光の企業はすでにサーバーの生産から離れていますが、その製品は依然として積極的に使用されています。 そのため、IBMサーバーを「飼いならす」ことの蓄積された経験を急いで共有します。 これは問題の完全なリストではありませんが、それでも誰かにとって有用な場合があります。



RAM



IBMサーバーは、メモリモジュールの構成に敏感です。 多くの場合、独立したアップグレード（メモリの追加または交換）の後、サーバーがロードされないか、実際にインストールされているよりも少ないメモリが表示されます。 幸いなことに、このような状況では、障害の原因について長い間推測する必要はありません。診断パネル（ある場合）で、2つのConfigおよびMemoryインジケーターが点灯します。







したがって、メモリをアップグレードする前に、仕様、サーバーでサポートされているメモリの種類とサイズを必ず確認してください。 サーバー内のプロセッサの数も非常に重要です-モジュールがスロットに配置される順序はこれに依存します。 これも仕様で明確にする必要があります。



一般に、メモリに関する状況は、たとえばHPに関する記事で説明されている状況とまったく同じです。 要するに：



  • メモリチャネルを観察します。
  • ECC REG 1（2）Rx4メモリをデュアルプロセッサシステムに挿入し、UDIMMをシングルプロセッサシステムに挿入します。
  • 各プロセッサに同じ量のメモリを配置します。






そして、指示に従ってメモリを挿入しても、サーバーがまだ動作せず、メモリインジケータが裏切りに点灯した場合はどうなりますか？ この場合、異なるオプションを確認する必要があります。



  • このタイプのメモリはサーバーでサポートされていません 。 仕様を注意深く確認してください。
  • 記憶は "バット"であることが判明した 。 行をまったく同じものに置き換え、サーバーが起動するかどうかを確認します。
  • マザーボードのスロットにほこりが詰まっています 。 これは、サーバーが数年間稼働している場合はかなり一般的な理由であり、最初の所有者でない場合はさらにそうです。 圧縮空気でスロットを吹き飛ばします。
  • ソケットの曲がったピン 。 これは非常にまれにしか発生しませんが、実際には発生します。プロセッサソケットの接触が曲がっているためにメモリが機能しなくなります。 前のオプションで障害の原因を見つけられなかった場合は、プロセッサーを取り外して、ソケットを慎重に検査してください。 あなたが少数の「幸運な人」の中にいるなら、あなたは曲がった接触を注意深く真っ直ぐにすることを試みることができます、しかしこれは完全にあなた自身の危険と危険にさらされています。


多くのシステム管理者は、MemTest86を使用してRAMをチェックすると、明らかに動作しているモジュールまたは同じトラック上でもエラーメッセージを受信するという事実に直面しています。 これは、M4世代のサーバーで特に一般的です。 これは、マシンまたはメモリの問題ではありません。MemTest86は、サーバーメモリのテストには推奨されません。 メモリが故障し始めると、サーバーは診断パネルを通じてこれを報告します。 標準の自己診断ツールを使用して、IBMサーバーのメモリを確認することをお勧めします。



ドライブ



サーバーに「ネイティブ」ドライブをインストールする必要がないことを繰り返し述べました。 IBMも他のベンダーもそれらを生産せず、有名なメーカーからのみ購入し、ロゴを再フラッシュして接着します。 したがって、「ネイティブ」ドライブの代わりにアナログを選択することにより、ディスクアレイのアップグレードまたは復元を簡単に保存できます。 これは、特に改装されたサーバーに関しては、価格の2〜3倍の差を正当化します 。 ネットワークでは、次のようなモデル一致テーブルを簡単に見つけることができます。

IBMモデル オリジナル
IBM 49Y2003 シーゲイトST9600204SS
IBM 90Y8872 シーゲイトST9600205SS
IBM 90Y8908 シーゲイトST9600105SS
IBM 81Y9650 シーゲイトST900MM0006


それでも、ネイティブでないドライブとサーバーとの非互換性の状況は依然として可能です。 この場合、サーバーは正常にロードされないか、ドライブを認識しません。 これは通常、新しいRAIDコントローラーファームウェアをインストールすることで解決します。 ちなみに、ファームウェアとバックプレーン/エキスパンダーを更新することをお勧めします。IBMBootable Media Creator （BoMC）アプリケーションがこれを支援します。



サーバーの電源を入れてPOSTチェックに合格すると、エラーが発生する場合があります。



検出エラーが発生しました。システムとこのシステムに接続されているすべてのエンクロージャーの電源を入れ直してください。



これは、ドライブの1つに問題があることを示しています。 計算は簡単です。他のすべてのメディアがテストに合格し、点滅が止まった場合でも、スライド上のインジケータは常に点滅します。



ディスクサブシステムには、よりエキゾチックな問題があります。 たとえば、 MegaRAID Storage Manager独自のアプリケーションでRAID-1を使用すると、次のようなエラーが表示される場合があります。



ID = 63

シーケンス番号= 48442

時間= 2016年1月24日17:03:59

LOCALIZED MESSAGE = Controller ID：0 Consistency Check found found inconsistent parity on VD strip：（VD = 0、strip = 637679）



ほとんどの場合、これはディスクが死んでいるという意味ではなく、パリティエラー、つまりプライマリディスクとセカンダリディスクのデータの不一致に関するものです。 考えられる理由：



  • 多くの場合、このようなエラーは、新しいアレイを構成した後、またはディスクの1つを交換した直後に表示されます。
  • パンケーキ表面診断セッション中に、ディスクが初期化され、I / O操作が実行されます。 RAID-1では、これにより一時的なボリュームの不一致が発生する可能性があり、これは次のコンプライアンスチェック中に自動的に修正されます。 これは診断セッション中には発生しませんが、星が収束するとき：

    • oキャッシュなしでRAIDコントローラーを使用するか、ライトスルーモードがアクティブになります。
    • o RAMの不足。ディスクからアクティブにページングします。
    • oディスクの非常に頻繁な使用。


この問題を解決するには、ディスクからのスワップアクティビティを減らすことをお勧めします。キャッシュ付きのRAIDコントローラを使用し、RAMの量を増やします。



ファームウェアおよびソフトウェアの更新



興味深い問題は、Windows 2012またはWindows 2012 R2をゼロからインストールするときに待機する場合があります。新しくインストールしたOSにはドライブが表示されません。 そして、これはIBMサーバーだけで起こりません。 実際、サーバー内のすべてのドライブはRAIDで接続されており、前述のOSバージョンにはRAIDを操作するための組み込みドライバーがありません。 そして、彼らはただそれらを無視します。 になる方法 最も信頼できる方法： IBM ServerGuideユーティリティーを使用します。 OSをインストールすると、このモデルとオペレーティングシステムのバージョンに必要なすべてのドライバーが強制されます。 OSイメージは、フラッシュドライブではなくディスクからインストールする必要があることに注意してください。ServerGuideは、起動元と同じUSBドライブ上のイメージでは動作しません。







サーバーを購入するとき、最初にすべてのファームウェアを更新してからシステムをロールする必要がある場合があります。 これは、前述のIBM Bootable Media Creatorを使用して実行できます。



  1. 起動可能なフラッシュドライブまたはディスクから起動します。
  2. BoMCを管理者として起動します。
  3. あなたがしたいことを選択してください：診断を更新および/または実行します。
  4. プログラムは、ドライバーを入手する場所を尋ねます。ダウンロードするか、指定したアーカイブからプルします。
  5. ブートイメージを書き込むメディア（フラッシュドライブまたはディスク）を選択します。 録音は数時間継続できますが、心配する必要はありません。プログラムはハングしません。
  6. 記録の最後に、このメディアから起動し、指示に従います。


この手順は、多くの問題状況でも役立ちます。 たとえば、統合管理モジュールの更新が完了するのを待たずに「キャンセル」ボタンをクリックした場合、次回のダウンロード時にサーバーはIMMをロードできず、デフォルト設定を使用する場合があります。 ファームウェアIMMイメージがロードされているおかげで、最初にマザーボードの「UEFI＆IMMリカバリジャンパー」ジャンパーを使用して復元を試みることができます。







それでも解決しない場合は、BoMCを介して更新手順を使用します。



意地悪の法則に従って、より新しいバージョンのBIOSのインストール中に電源障害が発生すると、さらに不快な状況が発生します。







その後、サーバーはメインファームウェアをダウンロードできなくなり、バックアップを使用します。 通常のBIOSリカバリ手順が役に立たない場合は、次のことを行ってください...ダウングレード：停電前のファームウェアより古いファームウェアをインストールします。 これは通常役立ちます。 その後、最新のBIOSバージョンのインストールを既に再試行できます。 彼らが言うように、一歩後退-2歩前進。



その他の問題



サーバーをリモートで制御しようとすると、「ログインがアクセス拒否エラーで失敗しました」というエラーが発生する場合があります。また、ブラウザでも発生します。 サーバーとクライアントを再起動しても解決しない場合は、IMMを工場出荷時の設定にリセットすることをお勧めします。



HPサーバーエラーに関する記事で、冷却システムの問題について言及しました。サーバーを起動した直後に、ファンは高速になり、それ以上は減少しませんでした。 この種の病気はIBMサーバーでも起こります。 サーバーは離陸時にジェット旅客機のようにうなります。 このような失敗の原因を見つけることはできませんでしたが、次のことをアドバイスできます。



  1. 電源コネクタの締め付けを確認してください。
  2. すべてのファンをオフにして、バスケットを取り外します。
  3. 異なるサーバーの各ファンを確認してください。
  4. ファンを交換して、バスケットを再び組み立てます。 または完全に交換してください。


私たちのプラクティスにはこのような興味深い失敗がありました。サーバーが起動すると、IMMが定期的に初期化され、その後UEFIの初期化が始まり、それだけです。 さらに、サーバーは説明なしでロードされません。 ネットワークからの切断、完全な電源切断、さまざまなコンポーネントの切断などの操作は役に立ちませんでした。 マザーボードのジャンパーを使用してUEFIバックアップをダウンロードすることも助けにはなりませんでした。 経験的に、20分ほど待っても、サーバーがロードされるのを待つことができます。 そのため、それ以来動作しています-毎回ロードに20分かかります。 失敗の原因を見つけることはできませんでした。
