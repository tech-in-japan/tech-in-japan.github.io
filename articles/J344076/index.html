C ++言語のサポート



グローバルな改善



おそらくほとんどの人は、C ++用の独自のパーサーを作成していることを知っているでしょう。 これにより、ユーザー間（この議論はHabréやredditなどに関するここでの投稿のいずれかで発生します）とチーム内で、さまざまな議論を引き起こします。 さらに興味深いのは、現在のソリューションを改善し、可能な代替オプションを考えることです（ところで、完全に除外するわけではありません！）。 そして最近、私たちはソリューションをグローバルに再設計することを決定しました。一番下にあるサブシステムから始めて、徐々に上昇します。 バージョンCLion 2017.3は、 リストの初期化と名前検索に影響しました。



リストの初期化の場合、修正により、基本的にCLionは「引数が多すぎます」 、 「引数が少なすぎます」 、 「一致するコンストラクタがありません 」などの誤った警告を表示しなくなりました。 そして、それどころか、今ではそれが必要な場所を示しています：





さらに、CLionは均一な初期化の場合に変数を正しく理解できるようになりました。





このリリースで修正されたもう1つの分野は、 名前検索です。 変数名とその宣言を正しく関連付けることは、言語サポートの基本的なタスクの1つです。 この段階での問題は、不正なコードアナライザーメッセージ、不正なコード生成などにつながります。主な修正は、 using

MSVC拡張機能のサポート



マシンにMicrosoft Visual C ++コンパイラがインストールされている場合、CLionで実験的なMSVCサポートを有効にし、GCCおよびClangの代わりにこのコンパイラを使用できます。 実験的サポートには、いくつかの理由で名前が付けられています。



  1. 当初、CLionは特別なC ++ MSVC拡張機能をサポートしていませんでした。
  2. CLion for MSVCで利用可能なデバッガはありません


CLion 2017.3には、最初のアイテムの修正が含まれています。 サポートされている拡張機能は、 __unaligned



__uuidof



、 __unaligned



/ __alignof



、 __ptr32



/ 64



などです。 完全なリストは、 ブログで入手できます。



残念ながら、MSVCのデバッガーに関する具体的な計画はまだありません（ただし、MSVCのさまざまなデバッガーとそのライセンスのトピックに関する興味深い議論があります）。



C ++サポートのその他の便利な改善



CLionの言語サポートチームは成長しています！ したがって、トラッカーからのより重要なリクエストを徐々にカバーしています。 このリリースでは、彼らはまだ次のことができました。



  • マクロサポート__COUNTER__ 。 その値は現在、正直に計算されています。つまり、CLionは、たとえば、 「重複宣言」に関する誤った警告を表示しなくなります。 また、クイックドキュメントウィンドウ（Linux / WindowsではCtrl + Q、macOSではF1）で、マクロ置換の結果を確認できます。



  • 関数の定義生成（Linux / WindowsではShift + Ctrl + D、macOSでは⇧⌘D）は、関数テンプレートの場合に正しく機能するようになりました。
  • 人気のリファクタリングテンプレートのサポートが追加されました-if if-else



    ブロックの反転。 このパターンは、多くのネストされたifブロックを使用するコードを簡素化するためによく使用されます。


オーディオアプリケーション開発者向け



また、マルチプラットフォームオーディオアプリケーションを開発する人たちの改善点についても話したいと思います。 そのような開発のために、非常によく知られているC ++ライブラリJUCEがあります。 CLion 2017.3では、JUCEの使用時に発生したいくつかの言語サポートの問題を修正しました。



しかし、JUCEユーザーにとっての主なニュースはJUCE 5.2のリリースで、Projucerの新しいバージョンはプロジェクトをCMakeにエクスポートする機能を追加しました。 そして、それはCLionでのさらなる開発のためにそれを開くことを意味します！ オーディオプラグインのデモを作成し、CMakeにエクスポートし、CLionで開き、Ableton Liveでプラグインを実行し、CLionでデバッグする短いデモも記録しました。



Valgrind Memcheckとの統合



トラッカーで最も人気のあるリクエストの1つは、プロファイリングツール、特にValgrindとの統合です。 このリリースでは、最も重要なものから始めました-Valgrind Memcheckを使用してメモリのプロファイルを作成する機能を追加しました。



このタスクでよく使用されるGoogle Sanitizersとは異なり、Valgrind Memcheckは多くのプラットフォームをサポートしています（ただし、残念ながら、Sanitizersと同様に、Windowsでは使用できません）。 プログラムを再コンパイルする必要はなく、ClangとGCCの両方を使用してコンパイルされた実行可能ファイルで動作します。 さらに、このツールを使用すると、メモリを操作する際のさまざまな問題を特定できます。



  • メモリリーク
  • たとえば、freeを呼び出した後の不正なメモリアクセス
  • メモリ割り当てと割り当て解除操作の不一致（malloc / free、new / delete）
  • 初期化されていない変数を使用する
  • 交差するメモリ領域をmemcpy型関数の関数として渡す
  • 負のサイズなど、間違ったサイズのメモリを割り当てる機能を渡す


CLionでは、Valgrind Memcheckを使用して構成を実行でき、起動結果ウィンドウで、問題に対応するスタックの一部や、問題が発生したコード内の場所のプレビューなど、潜在的な問題に関する情報を取得できます。 設定を使用すると、いわゆる抑制ファイル（ライブラリでの分析を無効にする場合など）とValgrind Memcheckを起動するためのオプションを指定できます。





単体テストの機能強化



CLionでの単体テストのフレームワークとの統合には、結果の出力（テストランナー）を備えた組み込みウィンドウでのテストの実行が含まれます。 Google Test and Catchが以前にサポートされていた場合、今でもBoost.Testが追加されます。





ちなみに、左側の列（「gatter」）には、新しいアイコンがあります。 サポートされている3つのフレームワークすべてで機能し、テストの実行に役立ちます-それらをクリックすると、Valgrind Memcheckで実行、デバッグ、または実行を選択できます。 また、最初のテスト実行後、アイコンの外観が変わり、テストのステータス（合格または不合格）の表示が開始されます。 最近起動したテストでファイルを開き、何をどのようにすばやく理解したい場合に非常に便利です。



同様に、左のガターのアイコンを使用して、CMakeターゲットがあるメイン関数を実行できます（ターゲットなしでは、何をどのように実行するかを正確に理解することは非常に困難です）。 ただし、実行構成は作成する必要はありません-アイコンをクリックすると、CLionは自動的に作成します。



ツールチェーン、コンパイラ、CMakeの設定



CMakeの分野でのほぼ伝統的な秋の変更、およびツールチェーンと設定の設定は、このリリースでは通過しませんでした。 主なもの：



  1. IDEで複数のツールチェーンを構成し、各プロジェクトに必要なツールチェーンを選択するか、同じプロジェクトの異なるターゲットに複数の異なるツールチェーンを使用することもできます。 これは特に、Cygwin / MinGW / MSVC環境を簡単かつ迅速に切り替えたいWindowsのユーザーに当てはまります。



  2. Makeと同様に、CコンパイラとC ++コンパイラを簡単に切り替えることができるようになりました。 これは、上記のツールチェーン設定で行うことができます。
  3. これで、同じビルドタイプ（デバッグ、リリースなど）で複数のCMakeプロファイルを作成できますが、異なるオプション、ツールチェーン（コンパイラなど）で使用される環境設定などがあります。



  4. CMakeプロファイルは、開始された実行/デバッグ構成に直接関連しなくなりましたが、個別に選択されます。





デバッガーとCMakeの新しいバージョン



CLion 2017.3には、次のツールの更新バージョンが含まれています。



  • CMakeが3.9に更新されました
  • GDBを8.0に更新
  • LLDBを5.0に更新


CLionに付属するGDB 8.0は、いくつかのアーキテクチャをサポートして構築されていることに注意してください。 つまり、異なるプラットフォーム間（WindowsからLinuxなど）でのリモートデバッグを使用すると、GDBを再構築する必要はなくなりますが、組み込みを使用できます。



コトリン/ネイティブ



JetBrainsが新しいプログラミング言語であるKotlinを作成し、積極的に開発していることをご存じの方はご存知でしょう。今年、Google I / OでAndroidアプリケーションを開発するための言語として公式に認められました 。 また、iOS、Linux、Windows、macOS、WebAssemblyなどのプラットフォーム用のマシンコードにKotlinを直接コンパイルできるKotlin / Nativeプロジェクトもあります。



CおよびC ++でのクロスプラットフォーム開発用のIDEであるCLionはどこにありますか？ 実際、Kotlin / Nativeは、ClangやLLDBなど、ネイティブ開発の世界のテクノロジーと統合されています。 そして、CLionは、ネイティブ開発のIDEであると見なされています。この言語では、CおよびC ++が主要言語として、Python、Swift、Rustが追加言語としてサポートされています（これまではすべてCMakeのフレームワーク内）。 したがって、開発用のプラグインをリリースするのは当然のことで、特にCLion向けにKotlin / Nativeで開発しました。 詳細については、Kotlinブログをご覧ください。 そして将来、他のツールにKotlin / Nativeが登場することを期待できます。



これで、CLion 2017.3の新機能の概要で終わります。 もちろん、IntelliJプラットフォームから来たバージョン管理システムと他の機能をサポートするアップデートもあります。 しかし、私は自分自身を繰り返さず、 ここHabrのIntelliJ IDEA 2017.3に関するブログ投稿を参照します。
