サーバーを使用しないAdobe Flash Playerとストリーミングデータ、パート1：ピアツーピアブロードキャストの整理

ご存知のように、 Adobe Flash Player製品を10.1に 、 Adobe AIRを1.5バージョンに最近アップデートしたことで、新しいリアルタイムメディアフロープロトコル（RTMFP）が導入されました。 事前に、サービングサーバー側で必要なAdobe Flash Media Server（FMS）の使用について、 リアルタイムメッセージングプロトコル（RTMP）と混同しないようにお願いします。

開発者は、パケットの送信時にプロトコルの遅延時間を短くすることを約束しますが、最も重要なことは、プロトコルがピアツーピアネットワークの編成に焦点を合わせていることです 。

Flash Playerの以前のバージョンでは、RTMPを介したリアルタイムのデータ転送が使用され、多大な金銭的費用が必要でした。 Adobe Flash Media Server（FMS）の機能には、非常に高価で気まぐれなホスティングが必要です（このビジネス全体には多くのリソースがあり、サーバーシステムに個別のソフトウェアモードとしてインストールされるため、物理サーバーをレンタルするまで）。

RTMPは、メディアのストリーミング、さまざまなデータの共有、またはリモート/同時使用に最適です。

そのため、新しいプロトコル（RTMFP）には既に「ピアツーピア」というニックネームが付けられています。 AIRおよびFlexアプリケーションを実行するためのサードパーティのメディアサーバーの厳密な必要性はありません。 サーバーは存在するが、マルチメディアやその他のデータを処理/保存するサーバーではなく、クライアント（ピア）間のセッションのみを制御し、トラフィックを分散するため、「厳格」なのはなぜですか。

Jozsef Vass（ Joseph Wass、Adobe / Flash Runtime Foundationの主任研究員、以前は革新的なVoIPテクノロジーの会社で働いていた）のメッセージは、 「Flash Player 10でRTMFPを使用してエンドツーエンドアプリケーションを開発するためのCirrusサービス」 。「Flash Player 10でRTMFPを使用してエンドアプリケーションを開発するためのCirrusサービス」 ）RTMFPを使用するには、Flash PlayerピアはCirrus （以前のコードネームStratus ）やFlash Media ServerなどのRTMFP互換サーバーに接続する必要があります4.記事では、Cirrusはインストールに役立つネットワークサービスであると述べています。 彼らのフラッシュプレーヤーを通じてクライアント（ピア）の間のIDE。 記事の資料とアプリケーションに基づいて、同様のサーバーをWebリソース上で編成できます（ アプリケーションのソース、ファイル「reg.cgi」を参照）。



RTMFPの利点について

-プロトコルは、失われたデータパケットを再送信する習慣があるため、 伝送制御プロトコル（TCP）によって制御されませんが、クライアント（ピア）とサーバー間の応答時間を短縮するユーザーデータグラムプロトコル（UDP）に基づいています。 さらに、ほとんどの場合、UDPはファイアウォールによってブロックされないため、ファイアウォールを再構成する必要がありません。

-データは、中央サーバーを経由せずにリアルタイムで2つのFlash Playerクライアント間で直接送信されます。

-2つのクライアント（ピア）間のインターネットチャネルの幅に見合うだけの高品質のマルチメディアの送信をサポートします。

-その存在、すなわち Cirrus adobeフロー制御サービス：rtmfp：//p2p.rtmfp.net。開発者キーを使用してプロトコルを使用する可能性を確認します。 繰り返しますが、Flash Media Serverとは何の関係もありません。 Cirrusは、Flash Playerインスタンスがオンラインでお互いを見つけるためにのみ使用されます。

-Diffie-Hellmann鍵交換方式を使用する128ビットAESにより、安全な接続が実現されます。 SSLまたはRTMPSのサポート。

-ActionScript 3.0の個別のクラスの存在。



ピアツーピア放送の組織

Adobe Cirrusを使用する前に、Adobe Labsから開発者キーを取得する必要があります。 ピアは、Cirrusを使用せずに相互に通信できます。 これには、クライアントが同じローカルネットワーク上にあることが必要ですが、これは非常にまれです。

Adobe Cirrusサービスに接続するには、 NetConnectionのインスタンスを作成する必要があります。 接続時には、以下に示すように、Adobe Cirrus URLと開発者キーの2つの文字列パラメーターを渡す必要があります。



private const CirrusAddress:String = "rtmfp://p2p.rtmfp.net";

private const DeveloperKey:String = "your-developer-key";

private var netConnection:NetConnection;

netConnection = new NetConnection();

// (listener)

netConnection.addEventListener(NetStatusEvent.NET_STATUS, netConnectionHandler);

// Cirrus

netConnection.connect(CirrusAddress, DeveloperKey);







Cirrusとの接続が成功した場合、文字列「NetConnection.Connect.Success」がNetStatusEvent.NET_STATUSのinfo.codeプロパティに割り当てられます。

注：現在、Cirrusを使用する場合、 NetConnection.call（）メソッドはサポートされていません。 NetConnection.call（）の呼び出しは、Cirrusによって無視されます。 開発者は、CirrusでRTMFPを使用する場合、 NetConnection.call（）メソッドを使用しないことをお勧めします。これにより、必要な機能効果なしでクライアントのメモリ消費が増加します。

Flash PlayerインスタンスがCirrusサービスに正常に接続されると、他のFlash Playerクライアントがアクセスできるように、CirrusサービスがクライアントのIPアドレスとポート番号にバインドするために使用する一意の256ビットのpeer_IDを受け取ります。

新しいCirrus 2サーバーチャネルとFlash Player 10.1を使用して、手動でpeer_IDを交換することなく、自己管理のP2Pネットワークを形成できるようになりました。 これにより、複数のFlash Playerクライアントが最も効率的な方法で互いに直接通信できます。

次に、クライアント（ピア）との接続が確立されます。 直接ストリーミングは、単方向のNetStreamチャネルを使用して実現されます 。 双方向のデータ転送が必要な場合、各Flash Playerクライアントは、送信するNetStreamとデータを受信するNetStreamを作成する必要があります。



//

private var sendStream:NetStream;

sendStream = new NetStream(netConnection, NetStream.DIRECT_CONNECTIONS);

sendStream.addEventListener(NetStatusEvent.NET_STATUS, netStreamHandler);

sendStream.publish("media");

sendStream.attachAudio(Microphone.getMicrophone());

sendStream.attachCamera(Camera.getCamera());

private var recvStream:NetStream;

recvStream = new NetStream(netConnection, id_of_publishing_client);

recvStream.addEventListener(NetStatusEvent.NET_STATUS, netStreamHandler);

recvStream.play("media");

放送局側では 、 NetStream.peerStreamsはすべての署名済みピアに共通のプロパティです。 たとえば、 sendStream.send（）メソッドを使用すると、すべてのピアに対してストリームが送信されますが、特定の受信者にストリームを送信することもできます。

sendStream.peerStreams[i].send();







他に2つのプロパティに言及する必要があります。これらはNetConnection.maxPeerConnectionsとNetConnection.unconnectedPeerStreamsです。

NetConnection.maxPeerConnections-ブロードキャスターへの接続を許可されているピアの数を決定します。 デフォルトでは、このプロパティの値は8ですが、実際にはさらに多くの値があります。 各Flash Playerクライアントは2つのストリームを送受信するため、完全なネットワークが作成されます。 通常、ダウンロードによるインターネットチャネルの幅はアップロードよりもはるかに大きいため、署名されたピアからの着信チャネルを過負荷にしないように非常に注意する必要があります。

ブロードキャストデータストリームにまだサブスクライブしていないピアの配列に含まれるNetConnection.unconnectedPeerStreamsプロパティ。 糖蜜放送にサインアップすると、ピア名はこの配列からNetStream.peerStreams配列になります 。
