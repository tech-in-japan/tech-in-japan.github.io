Androidの継続的統合

Android向けにいくつかのプロジェクトを実装した後、サイズが小さく見かけのシンプルさにもかかわらず、自動ビルドとテストを実行できないことが明らかになりました。 次の機能を取得したいという思いが生まれました。

  • ユニットテスト、
  • 統合テスト、
  • インターフェイステスト
  • Androidおよびデバイス構成のさまざまなバージョンでこれらのタイプのテストをテストします。
  • ビルドが壊れた場合のメール送信;
  • 自動リリースビルドでも問題はありません。


ほとんどすぐに、サイトtestdroid.comが見つかりました。ほとんどすべてが同じで、さらに多く（実際のデバイスでのテスト）を提供しますが、もちろんdenyuzhku向けです。

この点で、2週間の調査の後、Ant上のJenkins + buildスクリプト（Windowsの場合）の選択が停止されました。



それで、どうしてこれをすべて上げることができますか？！



前提条件：



  1. 何らかの方法でMyProjectプロジェクトをすでに作成しているとします。 そして彼にはMyProjectTestをテストするためのプロジェクトがあります（テストプロジェクトには少なくとも1つのテストがあります）。 そして、プロジェクトのあるフォルダーをd：\ projects \ MyProject \ trunk \ディレクトリに置きます。 プロジェクトのソースコードはSVNに保存されます。
  2. このマニュアルは、Android SDKバージョン16（http://developer.android.com/sdk/index.html）を使用して作成されました。 別のバージョンを使用している場合は、何か別の方法で作業する必要があります。


手順1.環境変数の設定（オプション、ただし簡単）。



システム環境（システム変数）にANDROID_HOME変数を作成し、その値にAndroid SDKフォルダーへのパスを指定します。 私はこのパスDを持っています：\ android \ android-sdk-windows。

また、値;％ANDROID_HOME％\ platform-toolsを Path変数に追加します。



ステップ2. Antのインストール



http://ant.apache.org/bindownload.cgiからzipアーカイブをダウンロードして、いくつかのフォルダー（たとえば、d：\ server \ android \ ant）に解凍します。 ant \ binフォルダーもPathシステム環境変数に追加する必要があります。



ステップ3. ant用のビルドスクリプトを準備します。



Googleは、Android SDKで、プロジェクトをビルドするためのAntスクリプトを生成できるようにしました。このために、コマンドを実行するために必要なプロジェクトのトランクフォルダーに：

  1. メインプロジェクトのビルドスクリプトを作成するには：

    1. android update project -p MyProject





  2. テストプロジェクトのビルドスクリプトを作成するには：

    1. android update test-project -m .. \ MyProject -p MyProjectTest


Android SDK <14バージョンのオプションについては、 こちらをご覧ください 。



手順4.プロジェクトのアセンブリを確認します（これもオプションです）。



  1. trunk \ MyProject \フォルダーに移動して、コマンドを実行します。

    1. ant cleanデバッグ
    実行の終わりに何かのような

    1. 成功する
    2. 合計時間： 5秒
    これは、プロジェクトが正常に組み立てられたことを意味します。

  2. Androidエミュレーターを実行し、trunk \ MyProjectTestフォルダーで

    1. ant all clean emma debug install test
    ビルドは少し長くなりますが、私の場合は35秒でした。 このプロセスでは、デバッグアセンブリMyProjestとMyProjectTestがアセンブルされ、実行中のエミュレーターにインストールされ、テストが実行されます。



ステップ5. Jenkinsをインストールします。



http://jenkins-ci.org/にアクセスすると、右側のディストリビューションへのリンクがあり、ダウンロードしてインストールできます。 デフォルトでは、http：//127.0.0.1：8080 /にインストールされ、Webインターフェースが提供されます。



小さな問題があります。 デフォルトでは、JenkinsサービスはSystemユーザーアカウントから開始します。これにより、エミュレーターを制御するプラグインがアカウントのホームフォルダーにAVDを作成できなくなります。 ユーザーをアカウントに交換するか、Jenkins用の特別なアカウントを作成する必要があります。



誰かが突然わからない場合は、コンピューターの管理（最も簡単な方法：スタート>コンピューター（右クリック）>管理（管理））に移動する必要があります。そこで、サービスとアプリケーション>サービスを選択し、Jenkinsサービス>プロパティ>ログオンタブを見つけます。 アカウントを変更した後、サービスを再起動する必要があります。



手順6. Jenkinsを構成します。



そのため、http：//127.0.0.1：8080 /に移動し、 [Jenkinsの 管理]を選択して、 [プラグインの管理]を選択します。 [ 利用可能なプラグイン ]タブをクリックします。 空の場合、しばらく待ってください。ジェンキンスはまだロードしていません。

次のものが必要です。

  • Jenkins emmaプラグイン
  • Android Emulatorプラグイン
  • ハドソンポートアロケータープラグイン
  • アリ


おそらくそれらのいくつかは、すでに最初にインストールされます。

私の場合、私のプロジェクトはSVNにあったため、Subversionプラグインをインストールする必要がありました。 プロジェクトが別のSCMの下にある場合は、そのプラグインをインストールすることを忘れないでください。

次に、下部にある「 再起動せずにインストール 」ボタンをクリックします。このプロセスでは、 「インストールが完了し、ジョブが実行されていないときにJenkinsを再起動する」ボックスをオンにして、Jenkinsが自動的に再起動し、プラグインが機能するようにします。

次に、 Jenkinsの管理>システムの構成に進みます 。 ここに

  1. Androidセクションで、Android SDKルートを登録する必要があります（前述のとおり、この場合はD：\ android \ android-sdk-windowsです）
  2. ビルドが中断したときに電子メールを送信するには、「電子メール通知」セクションに記入する価値もあります。


次に、Jenkinsのジョブを作成します。 メインページで、[ 新しいジョブ ]をクリックし、名前を入力して、 ジョブの種類を選択しますフリースタイルソフトウェアプロジェクトを構築します。



ここでは、次のセクションに記入する必要があります。

  1. ソースコード管理 -バージョン管理システムのタイプを選択し、必要なパラメーターを入力します。 注意！ トランクフォルダーのみをリポジトリからプルする必要があります。そうしないと、Jenkinsのジョブ設定に追加されるすべてのパスを修正する必要があります。
  2. トリガーの作成-SCMのポーリングを選択し、* * * * *と書いてください。 これは、Jenknsが1分ごとにリポジトリをチェックし、新しいコミットがある場合はビルドを実行することを意味します。 5分ごとのアセンブリの場合、* / 5 * * * *を指定できます。
  3. ビルド環境 -ここでは、衝突を防ぐために一意のTCPポートを割り当てるチェックボックスをオンにすることをお勧めします。
  4. ビルド中にAndroidエミュレーターを実行します 。 エミュレータのパラメータを入力します。たとえば

    • Android OSバージョン：2.1（このバージョンがSDKからダウンロードされていることを確認してください）
    • 画面密度：240
    • 画面解像度：WVGA
    • デバイスロケール：en_US
    • SDカードのサイズ：16M
  5. ビルドセクション、Invoke Antの追加、ここ

    • ターゲット：すべてのクリーンエマデバッグインストールテスト
    • ビルドファイル：QuickMeetingTest \ build.xml
    • プロパティ：sdk.dir = D：\\ android \\ android-sdk-windows（正確に、二重スラッシュを使用）


  6. ビルド後のアクションセクション

    • アーティファクトをアーカイブします：** / * test-TEST.xml
    • JUnitテスト結果レポートの公開：** / * test-TEST.xml
    • レコードエマカバレッジレポート：** / coverage.xml
    • ファイルのフィンガープリントを記録して使用状況を追跡します：** / * test-TEST.xml


  7. 電子メール通知-お好みに合わせてカスタマイズします。


実際、ジェンキンスのセットアップはほぼ完了しましたが、準備が整うまで急がないでください。



2つの問題があります。



問題1。


Standard InstrumentationTestRunnerは、テスト結果を含むXMLを生成しません。 したがって、MyProjectTestを少し強化する必要があります。

  1. http://code.google.com/p/the-missing-android-xml-junit-test-runner/にアクセスし、そこからpolidea_test_runner_1.1.jarをダウンロードし、trunk \ MyProjectTest \ libsに入れて、Java Build Path \ Librariesを追加しますプロジェクトのプロパティで。
  2. AndroidManifest.xmlで、<instrumetation>を次のように変更します。

    1. <計装
    2. android：name = "pl.polidea.instrumentation.PolideaInstrumentationTestRunner"
    3. android：targetPackage = "com.example.MyProject" />


  3. project.propertiesファイルに次の行を追加します

    1. test.runner = pl.polidea.instrumentation.PolideaInstrumentationTestRunner


  4. そして並んで
    1. <！-バージョンタグ：1->
    1をカスタムに変更します（このファイルの変更は、次回のAndroid更新中にほつれないようにする必要があります）。


問題2


テスト実行の結果を含むxmlファイルとカバレッジテストの結果を含むファイルがデバイス上に生成されます。 ジェンキンズがそれらを拾うには、そこから出して行く必要があります。

プロジェクトを実際にビルドするメインantスクリプトは$ {sdk.dir} /tools/ant/build.xmlにあり、次の行でプロジェクトビルドスクリプトにインポートされます。

  1. <インポート ファイル = "$ {sdk.dir} /tools/ant/build.xml" />


  1. $ {sdk.dir} /tools/ant/build.xmlに移動し、セクション<target name = "test" ../>全体を取得して、上記のメインスクリプトのインポートの前にテストプロジェクトのbuild.xmlに貼り付けます。

    これにより、テストターゲットへの呼び出しがオーバーライドされ、変更すると、変更が機能します。

  2. build.xmlの<target name = "test" ../>で

    1. <html outfile = "coverage / coverage.html" />


    次を追加

    1. <xml outfile = "coverage / coverage.xml" />


  3. 最後に、 </ target>の前に、エミュレータからテスト結果を取得するコードを挿入します。

    1. <mkdir dir = "$ {basedir} / junit-results" />
    2. <exec 実行可能ファイル = "$ {adb}" failonerror = "true" dir = "$ {basedir} / junit-results" >
    3. <arg line = "$ {adb.device.arg}" />
    4. <arg value = "pull" />
    5. <arg 値 = "/data/data/${tested.manifest.packageasket/files/" />
    6. </ exec >

ステップ8.複数のエミュレーターでテストします。



  1. Jenkinsのジョブを作成するときに、Build multi-configurationプロジェクトのジョブタイプを選択できます。
  2. [構成マトリックス]セクションで、各変更パラメーター（たとえば、Androidの解像度とバージョン）の軸を作成し、Run emulatorを$ {resolution}の形式のプロパティに置き換えます。 上記のリンクの写真を見ると、すべてが明確になります。
  3. 残りのパラメーターは、通常のタスクと同様に指定されます。

ステップ9.リリースビルドの準備。



リリースビルドの構築は、テストの実行よりも簡単です。

  1. JenkinsでMyProjectReleaseタスクを作成します（フリースタイルのソフトウェアプロジェクトのビルドタイプ）。
  2. 手順6と同様に、ソースのソースを指定します。
  3. パラメーターを使用して、Invoke Antビルドステップを作成します。

    1. 対象：
    2. 解放する
    3. ビルドファイル：
    4. MyProject \ build.xml
    5. プロパティ：
    6. sdk.dir = D：\\ android \\ android-sdk-windows
    7. key.store = .. \\ myproject.keystore（キーストレージファイルへの相対パス、トランクにキーを置きます\）
    8. key.alias = <エイリアス>
    9. key.store.password = <pass>
    10. key.alias.password = <pass>
  4. アーカイブするファイルを使用してアーティファクトをアーカイブする= ** \ *-release.apkを構成することもできます（すべてのリリースを注意深く保存し、Jenkins Webインターフェースからダウンロードできるようにする場合）。
  5. ProGuardを使用してリリースを難読化するには、proguard.config = proguard.cfgの行をproject.properties（またはJenkinsのInvokeAntタスクのプロパティ）に追加するだけです。

    • proguard.config-変数の名前。ProGuardの設定ファイルの名前が含まれている必要があります。
    • また、proguard.cfgファイルは、プロジェクトの作成時にEclipseによって自動的に作成されます。
