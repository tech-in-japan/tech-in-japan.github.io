ブロックチェーン上に電子投票システムを作成する方法は？

前の記事で、ブロックチェーン技術を使用して構築されたデータ交換システムについて説明しました。 成功した経験から、このテクノロジーを使用して投票システムという別の製品を作成することになりました。 この記事では、システムの実装の詳細について説明します。



投票システムのブロックチェーンとは何ですか？



投票は、社会にとって重要な問題に関する意思決定プロセスとして、数千年にわたって人類に知られています。 特別なアンフォラに色のついたボールを投げたり、手でジェスチャーしたりして、権限のある人の狭いサークルによって決定が下された時代がありました。 しかし、人口は増加し、社会は発展し、ますます多くの人々が意思決定プロセスに参加し始めました-数十万、さらには数百万人。 次に、投票をカウントするこのような単純な方法は、より複雑な方法に置き換えられました。投票所が作成され始め、アンフォラが投票箱に、ボールが投票に置き換えられました。 投票へのこのアプローチは、人口の巨大な大衆の意見を考慮することを可能にしましたが、いくつかの欠点があります：



  • 技術：投票用紙を詰める可能性。 結果を偽造するための領域での退色インクの使用。 プロトコル改ざん 投票の秘密の侵害; 計算の意図的なエラー。 結果の計算が遅い。
  • ソーシャル：贈収賄（カルーセルなど）; 有権者への行政的圧力。
  • 経済的：州予算のコストが高い。 たとえば、ロシアでの2015年の選挙の費用はほぼ1億ドルです。




説明されている問題の一部は、電子投票システム（インド、ブラジル、エストニア、オランダ、米国、ドイツなどで使用されています）によって解決されますが、その使用にはいくつかの欠点もあります。



  1. セキュリティの問題（電子システムがハッキングされる可能性があり、多くの場合スキャンダルがこれに関連付けられます）
  2. 選挙結果の検証に関する問題（数えることができる紙の投票とは対照的に）
  3. ソフトウェアエラーによるシステムの誤動作の可能性


この点で、電子投票装置および電子投票自体は世界的な慣行になっていない。 さらに、電子投票が当初普及していた多くの国（オランダ、イギリス、ドイツ）では、技術の不完全性により最終的に使用が制限され、より信頼性の高いアナログ投票方法に戻りました。



しかし、技術と社会の発展により、人類はより安価で信頼性の高い投票方法を探すようになっています。 投票にブロックチェーン技術を使用すると、既存の選挙制度の問題のほとんどを解決できます。



ブロックチェーン技術は、トランザクションモデルに基づいています。各ユーザーは、固有の公開キーと秘密キーを持つウォレットを持ち、それを使用してデータの変更を確認します。 すべてのトランザクション情報は連続して書き込まれたブロックに保存されるため、前のブロックのデータハッシュは次のブロックのデータに含まれます。 これにより、データの不変性が保証されます。ブロックを変更すると、後続のすべてのブロックが自動的に無効になります。 ブロックチェーンは、すべてのノードのすべてのトランザクションに関するすべての情報を同時に完全に保存し、変更または削除することはできません。 ブロックチェーンは、特定のデジタルオブジェクトへの財産権の移動に関するデータを登録する分野で最も広く使用されています。最新の暗号通貨はすべてこの考えに基づいています。 通常「コイン」または「トークン」と呼ばれるこのようなオブジェクトは、システムの操作中に所定のアルゴリズムに従って自動的に作成（「鉱山」）するか、そうする権限を持つシステムの参加者によって発行されます。



投票システムでブロックチェーンを使用するという考え方は、ブロックチェーンを使用すると、物理オブジェクト（目的の色のボール、紙の投票用紙など）で表される誰かの声をデジタルトークンに転送することで、古代の投票技術を置き換えることができます。 同時に、ビジネスプロセスが物理的な世界からデジタルに移行する他の多くの場合と同様に、トランザクションコストが大幅に削減され、システムの可用性が向上します。 さらに、ブロックチェーンを使用すると、追加の利点が得られます。



  1. 結果の信頼性。 ブロックチェーン技術を使用して編成された投票の結果は偽造できません。 投票の開始時に発行された投票数、ウォレット間での投票方法、取引が行われた時間をいつでも確認できます。
  2. プロセスの透明性。 ブロックチェーンを使用すると、投票プロセスを制御できます。関心のある人はすべてのデータの完全なコピーを使用してノードを展開し、ブロックチェーンレベルで独立して分析できるためです。
  3. 匿名。 各投票者は、ローカルマシン上で公開鍵と秘密鍵のペアを作成する機会があり、特定のウォレットが自分に属していることを彼以外の誰も知りません。 したがって、参加者自身がこのウォレットが自分に属していることを発表しない限り、このメンバーがどのように投票したかを誰も知ることができません。
  4. データ処理速度。 異なる国にオフィスを持つ都市、地域、国、または企業内での投票は、投票とデータ処理の両方に多大なコスト、組織上の困難、および時間の損失を伴う可能性があります。 地方分権化により、各地域/市/地区が独自のシステムノードを操作して負荷を分散できるにもかかわらず、全国の投票結果を見ることができます。


私たちの前に、ブロックチェーンに投票システムを実装する試みがありましたが、それらは広く配布されませんでした。 私たちの意見では、これにはいくつかの理由があります。



  1. 既存のプロジェクトの多くは、イーサリアムやビットコインなどの開発されたプラットフォームに基づいています。 暗号通貨のレートは最近非常に不安定になっていますが、同じEtherが年の初めに比べて印象的に成長しているため、このプラットフォームに基づく投票は非常に高価になります。
  2. これらのシステムのほとんどは、Proof of Workアルゴリズムを使用しているため、大規模な投票の結果の計算が非常に遅くなります（上記の問題については、以下で詳しく説明します 。

  3. これらは、独立した投票プラットフォームとして独占的に宣伝されます（つまり、投票または参加するには、プラットフォームで明示的な登録プロセスを実行する必要があります）。
  4. それらのほとんどは、アイデアレベルのままであるか、開発を停止しました


投票プロセスについて



システムでの投票はトランザクションです。 投票できる各オプションには独自のウォレットがあり、投票者（それぞれにウォレットもある）が投票トークンを転送します。 投票ごとに一意のトークンが発行されるため、たとえば、お気に入りのお茶に投票するために発行されるトークンは、遊び場ではなく駐車場の建設に投票することはできません。



アプリケーションの最終バージョンでは、さまざまな種類の投票を実装します。多数決（単純多数決によって決定されます）、代替案（提案されたものからいくつかのオプションが選択されます）、評価（各オプションには、好みに応じて特定の「重み」が割り当てられます）。 それらはすべてオープン（誰でも接続可能）およびクローズ（参加するには招待が必要）できます。 そのような投票が役立つ可能性のある分野は非常に異なっている可能性があります：HOAの長の選出、任意の競争での投票、株式会社の枠組み内での決定の投票など。



プロトタイプ段階で、オープンスキームに従って投票の多数決シナリオに限定することにしました。 これは、プラットフォームの機能を実証するとともに、このトピックが公共およびビジネスにとってどのように興味深いかを理解するために行われました。



プロトタイプでは、投票を作成するプロセスはいくつかの段階で構成されています。



  1. ユーザーはシステムにログインし、ローカルマシンでキーウォレットのペアを生成します。 システムはユーザーにコードワードを要求し、それを使用してウォレットデータを暗号化します。その後、暗号化されたウォレットを同じブロックチェーンに保存して、ユーザーを特殊なウォレットソフトウェアをインストールする必要をなくします。 同時に、オープンフォームのウォレットはクライアント側でのみ処理されます。これにより、所有者のみがウォレットにアクセスできるようになります。 ただし、コードワードを紛失すると、ユーザーはウォレットにもアクセスできなくなり、その場合、彼を支援することは不可能になることを理解することが重要です。
  2. プロトタイプ段階で、すべての登録ユーザーに、システムでの投票の作成に対して支払うように設計された10個のIDVトークンを提供することにしました。 彼らの助けを借りて、登録した参加者は投票を整理し、プロセス全体がどのように機能するかを見ることができます。 したがって、ウォレットを作成した直後に、処理サーバーは10個のIDVをユーザーのウォレットにクレジットします。
  3. システムで投票を作成するコストは1IDVです。 したがって、ユーザーが投票を作成すると、システムは残高をチェックし、十分なお金がある場合はトークンを1つ書き落とし、投票を準備します。投票オプション用のウォレットを作成し、いくつかの一意の投票トークンを発行します。 投票の作成の結果（作成に成功したかどうか、そうでない場合はその理由）は、特別なログに記録され、これもブロックチェーンに保存されます。


ユーザーが投票したい場合、以下が発生します。



  1. 彼は一般リストから必要な投票を選択し、それに参加する許可を要求します。
  2. システムは、ユーザーが投票に参加できるかどうかを確認し、参加できる場合は、この投票の一意のトークンをユーザーに請求します。
  3. ユーザーは提案されたオプションの1つに投票します。その後、システムはユーザーが投票できるかどうかを確認し、投票できる場合は投票トークンがウォレットから選択した投票オプションのウォレットに転送されます。


投票結果の確認は、いくつかの方法で可能です。 まず、各参加者のプロファイルには、「マイトランザクション」というオプションがあり、ウォレットの作成から始めて、すべてのトランザクションを確認できます。 次に、投票リストで特定の投票の画面に移動し、投票の詳細で選択したオプションのウォレットからウォレットへのトランザクションを見つけることができます。 第三に、読み取り専用ノードを展開し、結果のアカウンティングとブロックチェーンレベルでのシステムの動作を確認する機会が常にあります。



選択したテクノロジーについて少し



プロジェクトの技術コンポーネントについて説明する前に、次の点に注意してください。プロジェクトの主なタスクは、サードパーティアプリケーションのブロックチェーンプラットフォームに信頼できる票を埋め込むための便利な手段を提供することです。 そのため、便利なAPIの開発に特別な注意を払っています。 もちろん、投票のために顧客を提供しますが、主に投票プラットフォームの機能を実証するために顧客を開発しています。 以下では、プロジェクトの技術面について検討します。

以前のプロジェクトと同様に、Mulitchainブロックチェーンを基盤として使用しました。 次の考慮事項から進めました。



  1. セキュリティ。 ほとんどのブロックチェーンシステムの脆弱性の1つは、匿名性という独自の利点にあります。 誰でも任意の数の書き込みノードを拡張できる場合、変更された情報を含むノードの合計数の51％を拡張し、残りのノードが書き込みのためにそれを受け入れるというリスクがあります。これは、ブロックの最長チェーンが続くためです。 これは51％攻撃と呼ばれます。 このような攻撃を防ぐために、ほとんどの公開アルゴリズムは、Proof of Work確認アルゴリズムを使用します。各ノードは、特定の報酬（「鉱山コイン」）を受け取る複雑な数学的問題を解決します。 問題の解決には、時間とかなりの計算能力が必要です。 これにより、この種の攻撃は非常に費用がかかり、実用的ではありません。 マルチチェーンの主な機能は、コンソーシアムの原則に従ってブロックチェーンを整理するために特別に作成されていることです。 お互いを知っている人々の限られた輪、すなわち 匿名ではありません。 実装では、ブロックチェーンに接続するすべてのノードはデフォルトで読み取り専用ノードです。 つまり、すべてのデータを保存しますが、新しいデータを書き込む権利はありません。 書き込み許可を取得するには、ノードはマスターノードから対応する権利を取得する必要があります。 したがって、認証されたノードによってのみ新しいデータがシステムに書き込まれるようにし、不正のリスクを減らします。 同時に、読み取り専用ノードはすべてのブロックチェーンデータの完全なコピーを受信し、一貫性のために投票データを個別に分析できます。
  2. リソースの強度。 前述したように、作業の証明アルゴリズムを適用する場合、計算量が多く、ユーザー数が増加すると、一時的なリソースも必要になります。 この問題を解決するために、Proof of Authorityを優先してProof of Workアルゴリズムを放棄します。この場合、自身を識別したノードにのみ情報を記録する権利を与えます。 したがって、新しい情報はより迅速にブロックチェーンに分類され、「記録価格」は最小限になります。
  3. 経験。 過去のプロジェクトでこのブロックチェーンプラットフォームを使用した経験があり、発生する問題を解決する方法を熟知しています。


技術部分をさらに詳しく見てみましょう。 投票構造は次のとおりです。



システムの構造は、3つのセグメントに分けられます。



UI-パブリックREST API（共通領域セグメント）に接続するか、弾頭の読み取りノード（図には示されていません）に直接接続するエンドユーザーアプリケーション。 プロトタイプでは、機能を実証するために、UI WebサイトをAngularのSPAとして実装しました。 誰でも、REST APIまたは読み取り専用マルチチェーンノードをデータソースとして使用して、利用可能な任意のテクノロジーにUIを実装できます。



共通領域 -これらは、マルチチェーンが記録モードでデプロイされるシステムのNノードと、共有アクセス用のAPIを備えたWebサーバーです。



ASP.NET Core 1.1で記述されたREST API Webサイト。 投票メタデータ、メタデータアーカイブ、投票プロセス自体（オプションを選択して投票する）、およびユーザーのウォレットを使用して作業を実装しました。 APIの詳細な説明はこちらにあります 。



プライベートエリア -一般的なアクセス（記録）のために閉じられているシステムの一部。処理サーバーとユーザー認証サーバーが含まれます。

独自の部分では、処理サーバーとユーザー認証サーバーを取り出しました。



処理サーバー



APS.NET Core 1.1のCommon REST APIと同様に実装されています。 プロトタイプでは次のことを担当します。

  • ユーザー登録中のIDVトークンの計算（作成された投票の支払いのため）
  • 投票を作成
  • 投票ステータスの管理（指定された条件に応じた開閉）
  • ユーザーへの投票トークンの発生。


処理サーバーによって実行されるすべてのアクションは、マルチチェーンに記録されます。 したがって、すべてのノード（リーダーを含む）はこれらのアクションを見ることができます。



アイデンティティサーバー



システムユーザー認証サーバー。 IdentityServer 4コンポーネントを使用してASP.NET Core 1.1に実装されています。



機能：



  • ユーザー登録;
  • ユーザー認証
  • ユーザー認証;


ユーザープロファイルも暗号化された形式でマルチチェーンに保存されます。



プロトタイプの制限



私たちのシステムは、さまざまな種類の投票を実施するように設計されています。 ただし、プロトタイプの機能を実証するために、参加者のオープンリストを使用した単純な多数決のシナリオに限定することにしました。 誰でもそのような投票に参加できます。 したがって、この記事では、私的な投票、評価、および代替投票に参加するように人々を招待するシナリオは考慮しません。 また、プロトタイプでは、厳密な識別メカニズムを実装していません。 これは意図的に行われたため、誰でも自分で試すことができます。
