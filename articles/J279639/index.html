なぜ彼らはデジタル時代に鞭を打つことができるのでしょうか？ GitHubはDMCA、RFC 7725、およびHTTP 451とはどのように準拠していますか？

2012年に、Habréは「制限者」、「仕切り」、「検閲」、「検閲」、より正確にはアクセスが制限されているリソースに対して新しいHTTPステータスが提案されたという事実に関する記事で「虐待に抵抗する」プロバイダーについて議論しました法律の問題が原因です。



実際、 HTTP 451のステータスは GoogleのTim Brayによって提案され、アラームの犯人（文字通りおよび比fig的に）は再びブロックされ、恥ずかしく、検閲され、偉大なThe Pirate Bayでした。



なぜこれが重要なのですか？ あなたは、おそらく私のように、GitHubを使用しているのは、魂のためだけでなく、おそらくは有益なインターネットリソースを所有しているからです。



あなたはすでに著作権分野にいます。



この記事の執筆時、Habréでの2年間の沈黙の最初に、私は最近私に起こった物語によって促されました。 興味があれば、彼女は記事の一番下にいます。今はRFC 7725、GitHub、DMCAについてです。



451法的理由により利用不可



ごく最近、今年の2月に、Googleの従業員の上記の提案は、 法的障害を報告するためのRFC 7725ステータスコードの形式でIETF によって批准されました （法的制限を報告するステータスコード） 。



これは、近い将来、インターネットプロバイダー、ホスティング、検索エンジン、およびさまざまなサービスが、 404 Not Found 、 403 Forbiddenの代わりに、またインターネットにアクセスしようとしたときに最初に発生したものの代わりに、悲しい理由である 「間違い」を表示し始めることを意味します-法定制限の対象となるリソース。



Githubはすでにサポートしています



昨日、GitHubブログにGitHubがRFC 7725をサポートするというメモが掲載されました。



このRFCの言及に加えて、この投稿では、著作権侵害の申し立てをGitHubに適切に送信する方法、およびGitHubがDMCA要件に準拠する方法を示しています。



RFC 7725は何と言っていますか？
仕様で示されているように、このステータスコードは、法律または公共政策の問題がサーバーの動作に影響を与える状況で透明性を提供するために使用でき、そのような透明性はオペレーターとエンドユーザーの両方に役立つことがあります。



HTTP応答コード451は、要求のキャッシュに影響する他のHTTPヘッダーが指定されていない限り、デフォルトでキャッシュされます。



要求に応答するサーバーは、宛先サーバーではない可能性があります。 このタイプの法的制限は、インターネットプロバイダーと検索エンジンの作業に関連しています。



さらに、HTTP応答の本文には、制限の詳細を記載した説明を含める必要があると言われています。そのような制限を課す人、制限の根拠、どの人に対してそのような制限が有効か。



また、HTTP応答のLinkヘッダーには、ソースリソースへのアクセスをブロックしたサーバーのURIが含まれている必要があります。



HTTPステータス451の応答は、リソースの有無を示しませんが、制限の有無のみを示します。



セキュリティ上の理由から、クライアントは次のように、451の回答に依存しないでください。 一部の法的機関がアクセス制限を要求するだけでなく、制限を課すことへの関与を隠したい場合にも、状況は可能です。



DMCAとは何ですか？
DMCAは、デジタル時代の米国著作権法です。 なぜ著作権だけではないのですか？ もともと、米国の著作権は、文学、音楽、映画の制作と流通におけるさまざまな法的関係を説明し、デジタル技術の出現と発展を考慮していなかったため、デジタル技術の特徴を考慮せずに「そのまま」著作権の適用と使用に特定の困難をもたらしました。





GitHubはDMCAに準拠しています



GitHubには、GitHub に対する著作権所有者の訴えのテキストを含む独自のリポジトリがあることがわかりました。 このリポジトリには、本質的に特定の前例（現時点では1139）、GitHubでコードを公開するときに発生する可能性のある特定の状況、適切なライセンス条項を保持せずに誰かのソースコードを使用する、または著者の権利を著しく侵害することが含まれます。



誰かのリポジトリへのアクセスをブロックするリクエストでGitHubに連絡した法人の中には、Oracle、Cisco、Sony、Nintendo、Instagram、Microsoft、Apple、Adobe Systems、IEEEなどの組織が現れます。



結論



DMCAについて考える時が来ました。 人々に光をもたらすのではなく 、仕事をするよりも頻繁にそれをしなければなりません。



たとえば、GitHubをホスティングとして使用した無害なアプリケーションGBA4iOS（GameBoy Advanceエミュレーターfor iPhone）のサイトは、 Nintendo Of USAのリクエストにより削除されました 。



たとえば、写真に関する人気のあるモバイルアプリケーションを突然「ピックアップ」し、そこからシェーダーを使用して、プロジェクトをGitHubに投稿すると、 ホイップできます 。



誰かのソースコード内のメソッドまたはクラスの名前を変更して、著者を「隠蔽」しようとしないでください。計算されて投獄されます。



BitCoinフォークで作業している場合でも、MITライセンスに関するいくつかの言葉でコメントのフットクロスを削除しようとしないでください- 首に砂が埋まっている可能性があります 。



さて、あなたは理解する、 悪いアイデアの約1135の例 。



どうする？





著作権を侵害したくない
  1. 使用しているコードがどのライセンスの下にあるかを確認します
  2. ソースコードの作成者と電子メールでやり取りし、目的のライセンスの属性を追加するか、そのコードを使用するための承認を要求する
  3. 著者からメールで返信が得られない場合は、Twitterで手紙を送ったことを書いてください。
  4. 著作権ニュース、ユースケース、法律を読む
  5. 人生を楽にして




著作権を侵害しています
  1. みんなとみんなにそれについて話さないでください。
  2. 「他の人の権利を侵害したくない」のセクションに従って、少なくとも一部の権利の侵害をやめましょう
  3. あなたが誰であるか自分自身を受け入れる
  4. 事前に最高の弁護士を見つけ、その費用を調べてください




私の権利が侵害されました
  1. 気分を害したり、重大な損失を被ったりしていますか？
  2. 指示に従って GitHubに連絡し、 これらの手続きを忘れずに追加してください。
  3. 他の人が訴訟を続けてあなたの権利を侵害できないように行動を起こす
  4. あなたの作品を公開することを考えるべきかもしれません。
  5. 問題のある当事者を訴える
  6. この世界の不正を受け入れ、あなたの経験を共有してください。
