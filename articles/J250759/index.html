Linq to Sqlの興味深い点

同様のトピックに関する前回の投稿から1年以上が経過しました。 この間、なんらかの形でEntity Frameworkへの切り替えに近づきませんでした（現在の凡例によれば、EF 7の安定バージョンが表示されたときに切り替えます）が、共有したい経験がいくつかあります。 この記事は、私たちのように、この一般的に優れているが、 Microsoftテクノロジーを忘れてしまった人にとっては興味深いものになると思います。



Dbtype



Linq 2 Sqlのエンティティプロパティには、 DbTypeヒントの指定（列挙型を除く、詳細は以下を参照）は必要ありません。 そして確かに、間違ったDbTypeを指定すべきではありません。 たとえば、ベースの列のタイプがnvarchar（50）である場合、その列のタイプがnchar（50）であることをLinq 2 Sqlに示す必要はありません。 次の例のように、このフィールドが弁別子である場合、特にこれを行う価値はありません 



[Table(Name = "directcrm.OperationSteps")] [InheritanceMapping(Code = "", Type = typeof(OperationStep), IsDefault = true)] // ... [InheritanceMapping(Code = "ApplySegment", Type = typeof(ApplySegmentOperationStep))] public class OperationStep : INotifyPropertyChanging, INotifyPropertyChanged, IValidatable { //    ... [Column(Storage = "type", DbType = "nchar(50) NOT NULL", CanBeNull = false, IsDiscriminator = true)] public string Type { get { return type; } set { if ((type != value)) { SendPropertyChanging(); type = value; SendPropertyChanged(); } } } }




それでは、データベースからOperationStep型のエンティティを読み取って、継承マッピングで処理できるかどうかを確認してみましょう。

原則として、パラメータの値は想定されていましたが、そのようなバグを検出することはそれほど簡単ではありませんでした。 注意してください。 バグは現在出現していることは明らかですが、一般的に言えば、エンティティが正しく作成されてデータベースから削除されるため、システムに長期間残る可能性があります。 弁別器の値の奇妙な尾に注意を払うか、弁別器を更新するスクリプトの後にすべてが落ち始めるまで。



linq to sqlエンティティにenumを保存することについて少し説明します。



デフォルトでLinq to sql（ DbTypeが指定されていない場合）は、Enumの列タイプがIntであると見なします。 したがって、次のエンティティを操作することはできません（ directcrm.CustomersテーブルのSexフィールドはnvarchar（15）タイプです）： 

  [Table(Name = "directcrm.Customers")] public sealed class Customer : INotifyPropertyChanging, INotifyPropertyChanged, IValidatable { //    [Column(Storage = "sex", CanBeNull = true)] public Sex? Sex { get { return sex; } set { if (sex != value) { SendPropertyChanging(); sex = value; SendPropertyChanged(); } } } }




ベースから減算しようとすると、Customerエンティティ（Sexフィールドには文字列「female」が入力されます）は、 System.InvalidCastExceptionから落ちます。 指定された性別で消費者を保存するとき、このリクエストを取得します： 



 DECLARE @p20 Int = 1 INSERT INTO [directcrm].[Customers](..., [Sex], ...) VALUES (..., @p7, ...)




注目に値するのは、そのようなタプルをテーブルから減算しても機能しないことです。同じサイレントSystem.InvalidCastExceptionが発生します。 したがって、linq to sqlを使用してデータベースに列挙文字列を保存する場合は、DbTypeを指定することを忘れないでください。

ちなみに、Entity Frameworkは列挙型を行に格納できないため、使用することを決めたプロジェクトでは、ハックを使用する必要がありました：列挙型自体が解析した列挙型フィールドごとに追加のゲッター（列挙型の値が想定されます文字列型のプロパティに保存します）。

LeftOuterJoin



ご存じのように、Linqは、コレクションの1つに値がない可能性があるコレクションを接続するための拡張メソッドをまったく提供していません。 しかし、時々linq to sqlで作業する場合、たとえばsqlで左外部結合を取得する必要があり、そのような状況ではlinqメソッドの組み合わせを使用し、最終的に左外部結合に変換します。 左外部結合を取得する2つの方法を知っています。

最初のオプション：

 CustomerActions .SelectMany(ca => CustomerBalanceChanges .Where(cbch => cbch.CustomerAction == ca) .DefaultIfEmpty(), (ca, cbch) => new { ca, cbch}) .Dump();

さらに、単純な左外部結合に対して上記の構造のいずれかを常に使用することはいくぶん不利です。LeftOuterJoin拡張メソッドを自分で記述してクエリで使用する方がはるかに簡単です。 このような拡張機能がどのように表示されるかを次に示します。 



 public static IQueryable<TResult> LeftOuterJoin<TOuter, TInner, TKey, TResult>( this IQueryable<TOuter> outerValues, IQueryable<TInner> innerValues, Expression<Func<TOuter, TKey>> outerKeySelector, Expression<Func<TInner, TKey>> innerKeySelector, Expression<Func<TOuter, TInner, TResult>> fullResultSelector, Expression<Func<TOuter, TResult>> partialResultSelector) { Expression<Func<TOuter, IEnumerable<TInner>, IEnumerable<TResult>>> resultSelector = (outerValue, groupedInnerValues) => groupedInnerValues.DefaultIfEmpty().Select( innerValue => Equals(innerValue, default(TInner)) ? partialResultSelector.Evaluate(outerValue) : fullResultSelector.Evaluate(outerValue, innerValue)); return outerValues .GroupJoin(innerValues, outerKeySelector, innerKeySelector, resultSelector.ExpandExpressions()) .SelectMany(result => result); }




この拡張機能は常に変換されますが、SQL側でnullチェックを使用します。 以下の使用法が意図されています。

拡張メソッド自体がEvaluateおよびExpandExpressionsメソッドを使用していることに気付くかもしれません。 これらは、 Mindbox.Expressionsライブラリの拡張メソッドです。 ExpandExpressionsメソッドは、呼び出された式ツリー全体を再帰的に走査し、Evaluate呼び出しをEvaluateが呼び出された式で再帰的に置き換えます。 ExpandExpressionsメソッドは、ExpressionオブジェクトとIQueryableの両方で呼び出すことができます。これは、より便利な場合があります（たとえば、クエリが複数の場所に構築されている場合）。 このライブラリには、コードのリフレクション作業のための興味深い関数も多数あります。 おそらく、ライブラリは誰かに役立つでしょう。
