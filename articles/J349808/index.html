どうやってAB-DOCをしますか

この記事では、私たちと、Amazon S3ストレージへのインターフェースを提供するサーバーレスWebアプリケーションであるAB-DOCについて説明します。 以前の記事でAB-DOCの詳細を読むことができます。 次に、AB-DOCの基礎となる技術ソリューションに焦点を当てます。 開発および展開プロセスをどのように構築したかを説明します。







写真では、自宅のデスクにいて、90％の時間働いています。





私たちには小さなチームがあり 、ビジネス自動化のためのWebアプリケーションを開発しています。 カスタムプロジェクトの実行に加えて、2つのオンラインサービスを作成および開発しています。

  1. AB-DOC情報の保管およびシステム化サービス
  2. タスクトラッカーAB-TASKS


チーム全員がAB-DOCに取り組んでいるわけではありませんが、私とIgorBBの 2人だけです 。







それでは、AB-DOCとは何ですか？



サーバーレスアーキテクチャ



サーバーレスアプリケーションは、現在、Web開発の世界で強力なトレンドとなっています。 簡単に言えば、これらは完全にクラウドプロバイダーのサービスに基づいて動作するアプリケーションです。 それらの利点は、それらを機能させるクラウドサービスの特性（フォールトトレランスとスケーラビリティ）に由来します。 さらに、クラウドサービスと同様に、それらのコストは負荷に直接依存します。 負荷がない場合、コストはゼロです。 それは非常に快適で素晴らしいです。



しかし、コインには裏返しがあります。 サーバーレスアプリケーションの最初の欠点は、複雑な開発プロセスです。 つまり、インフラストラクチャの初期構成および開発プロセスの構築の段階では、サーバーを備えた通常のWebアプリケーションよりも多くの労力が必要です。 2番目の欠点は、このようなアプリケーションは通常ベンダーに依存しているため、別のクラウドプロバイダーに移行するのは困難な作業になる可能性があることです。



Amazon Web Servicesでアプリケーションを作成しているため、この記事のすべてはこのクラウドプロバイダーを参照します。



通常、サーバーレスアプリケーションにはバックエンドがあります。 バックエンドは、サービスDynamoDB（NOSQL DBMS）またはRDS（リレーショナルDBMS）の形式のデータベースです。 バックエンドのコードはLambda関数として実装され、API Gatewayを介してアクセスされます。



AB-DOCにはこれはありません。 通常、バックエンドとサーバーコードはありません（少なくともまだ）。



AB-DOCアーキテクチャは次のようになります







アプリケーションコード（HTML、CSS、JavaScriptなど）は別のS3バケットにあり、CDN CloudFrontを介して提供されます。 原則として、バケットでは静的コンテンツをホストする機能を有効にできるため、S3から直接提供できます。



CloudFrontが必要な理由は、コンテンツの読み込みを高速化するだけでなく、すべてのリンクをindex.htmlにリダイレクトする方法が必要だったためでもありました。 AB-DOCは単一ページアプリケーション（SPA）であるため、URLを要求する場合、ユーザーを読み込むためにindex.htmlが必要です。 さらに、リクエストされたURLに基​​づいて、JavaScriptはajaxを使用してユーザーに必要なコンテンツを読み込みます。 フロントエンドフレームワークは使用しないため、独自の小さなルーターを作成しました。



したがって、すべてのURLをindex.htmlにリダイレクトするには、CloudFrontで404エラー処理ルールを設定します（ページが見つかりません）。







このルールのおかげで、CloudFrontは、リソースに存在しないURLのリクエストに応じて、200の応答コードでindex.htmlを提供します。これは、S3でコードをホストするSPAを実装するための魔法です。



ユーザーコンテンツもS3の別のバケットでホストされ、CloudFrontを通じても提供されます。 カスタムコンテンツには、ツリー構造（Json）、ドキュメントコンテンツ（HTML）、ドキュメントに埋め込まれた画像、および添付ファイル（さまざまな形式）が含まれます。



さらに、私の意見で興味深い個人、アプリケーションの実装のニュアンスについてお話します。



認証と承認



コードをユーザーのブラウザにダウンロードした後、AB-DOCは最初にユーザーを認証および承認します。 このために、Cognitoを使用します。



認証はユーザーを認証するプロセスであり、承認はこのユーザーにプログラムで作業する特定の権利を与えるプロセスであることを思い出させてください。



Cognitoは3つのサービスで構成されています。



  1. ユーザープール
  2. フェデレーションID
  3. 同期する




Cognitoサービスの経験がなかった場合、各サービスが必要な理由を理解することは困難です。 これを理解する方法を書いています。



ユーザープール



これは、ユーザーの登録、認証（認証）、およびアカウントのストレージを提供するマネージドサービスです。 各ユーザーのデータフィールド（格納するもの）、パスワードの複雑さのポリシー、MFAを使用するかどうか、登録中に起動できるさまざまなトリガー、ログインなどを構成できます。 ここで、ユーザーを登録するときにサービスが電子メール検証のために送信するメッセージを構成できます。



外部IDプロバイダー（Facebook、Google、Amazon、またはSAML）をユーザープールに添付できます。 外部プロバイダーを介してログインすると、ユーザープールにアカウントが作成されます。 これは基本的に、Cognitoサービス自体がホストするログインページを使用する場合に機能します。 つまり、ユーザーをリダイレクトして、your-app.auth。[Region] .amazoncognito.comという形式の特別なURLを入力する必要があります。 このページの外観は、アプリケーションの外観のようにカスタマイズできます。 承認後、ユーザーはアプリケーションに戻ります。 私はこの実装オプションが好きではありませんでした。



JavaScript用Amazon Cognito Identity SDKを使用して、ユーザープールを外部プロバイダーと統合することをかなり長い間試みました。 理論的には、これも可能ですが、このテーマに関するドキュメントを見つけることができず、私はあきらめました。 それは2〜3ヶ月前でした。



フェデレーションID



ユーザープールとは異なり、このサービスはユーザーを承認する責任があります。つまり、AWSサービス（S3など）にアクセスする特定の特権をユーザーに付与します。



このサービスが機能するには、IDプールを作成し、それが機能する認証プロバイダーを構成する必要があります。 ユーザープールは、認証プロバイダーとしてだけでなく、Amazon、Facebook、Google、Twitter、OpenID、SAMLなどの多くの外部プロバイダー、または自分で作成した認証プロバイダーとしても機能します。



AB-DOCでは、ユーザープールと1つの外部プロバイダーであるGoogleを認証プロバイダーとして使用します。 したがって、ユーザーには2つのオプションがあります。



  1. 新しいアカウントを作成します（ログインパスワードを作成し、メールを確認します）。 この場合、ユーザープールにアカウントが作成され、次にIDプールにアカウントが作成されます。これにより、彼は必要な権限を取得します。
  2. Googleアカウントでサインインします。 その後、ユーザープールのアカウントは作成されず、ユーザーはIDプールにアカウントを作成するだけで、アプリケーションを操作するために必要な権限が与えられます。




IDプールでの権限付与のプロセスは非常に簡単です。 IDプールの場合、AWS Identity and Access Management（IAM）から2つのロールが選択されます：認証済みユーザー用と非認証ユーザー用。



IAMの各ロールに対して、ユーザーに必要な権限を任意に付与するポリシーを添付できます。 ポリシーで変数を使用することで、個別化が可能です。 たとえば、S3バケットのフォルダー内でのみユーザーに書き込みアクセスを許可するには、リソースの説明でこのような変数を使用します 

... "Resource": [ "arn:aws:s3:::ab-doc-storage/${cognito-identity.amazonaws.com:sub}", "arn:aws:s3:::ab-doc-storage/${cognito-identity.amazonaws.com:sub}/*", ] ...




したがって、各ユーザーの$ {cognito-identity.amazonaws.com:sub}の代わりに、アイデンティティプール内のユーザーの識別子がポリシーに代入されます。 各ユーザーは、バケット内の自分のフォルダー内で作業します。この名前は、IDプールのIDに対応しています。



S3には実際にはフォルダーがないため、ここのフォルダーという言葉は引用符で囲むことができます。 これはフラットファイルシステムです。 各ファイルには単純にキー（キー）があり、条件付きで「フォルダー」に分割されます。



同期する



このサービスは、アプリケーションのユーザーデータのストレージを提供します。 Syncでホストされるデータは、IDプールのユーザーIDにバインドされたキーと値のセット（データセット）として保存されます。 したがって、Syncを使用すると、ユーザープールまたは外部プロバイダーであるかどうかにかかわらず、認証されないすべてのユーザーの任意の情報を保存できます。 さらに、Syncは、ユーザーがアプリケーションで作業しているすべてのデバイス間でデータセットの同期を提供します。



一般に、認可はトークンに基づいて構築されます。 それを担当するJavaScriptコードは、現在、アプリケーションコード全体の1/3以上を占めています。 つまり、これはかなり膨大なトピックであり、これについては今後別の記事を書くかもしれません。



変更とイベントキューを追跡する



AB-DOC自体は、ツリーノードまたはドキュメントのコンテンツを編集するときにユーザーが加えた変更を追跡して保存します。



このメカニズムの動作は、JavaScriptタイマーsetInterval（）に基づいています。 ツリー用とドキュメント用に別々のタイマーが作成され、3秒ごとに変更が発生しているかどうかが確認され、発生した場合はS3に保存されます。 アプリケーションのタイマーは、TIMERSオブジェクトを介して中央で作成されます。



一元化された変更追跡のために、イベントキューを管理するACTIVITYオブジェクトを作成しました。 キューは、ユーザーが作成できるコンテンツのオプションに従って形成されます。 ツリーには独自のキューがあり、ドキュメントには独自のキューがあり、各キューには独自のキューがあります。



キューに記録されるイベントには、保留または保存の2種類があります。 キューのロジックは次のとおりです。



ユーザーが何らかの編集を行うと、待機イベントが目的のキューに追加されます。 ツリーでは、イベントは変更を処理するコードから直接追加され、ドキュメントの変更はMutation Observerのおかげで追跡されます。



さらに、対応するタイマーがトリガーされると、 保存イベントがキューに追加され、S3に変更をロードするプロセスが開始されます。 ダウンロードが成功すると、このキューは最後の保存イベントまでクリアされます。 クリアの時点で、キューには次の3秒サイクルを待機する新しい待機イベントが既にある場合があります。



これは、ドキュメントの変更を担当するドキュメント変更キューの例でどのように見えるかです。







ヘッダーの変更インジケータは、イベントキューのステータスを反映しています。



何も起こらず、すべてのキューが空の場合、インジケータは表示されません。



キューに待機中のイベントがある場合、オレンジ色の鉛筆で表されます。



保存中のキューがある場合、インジケータは白に変わります。





キューが空ではなく、ユーザーがブラウザを閉じようとするか、単にページを離れようとすると、window.onbeforeunload（）関数が機能します。 彼女は、編集内容がまだ保存されていないことをユーザーに警告します。 彼は、保存が完了するまで待つか、最後の変更を失ってページを離れることができます。



インジケーターは、$更新変数に書き込まれます。 

 $update = $('#update');

S3でのデータのロードと保存



S3の各ユーザーには、IDプールのIDと一致する名前の個別のフォルダーが割り当てられます。 このフォルダのルートで、AB-DOCはユーザーツリー構造を格納するtree.jsonファイルを保存します。



tree.jsonファイル形式は次のとおりです。 

 [ { "id": "GUID  ()", "name": "  ()", "children": [ { "id": "...", "name": "...", "children": [...] }, { "id": "...", "name": "...", "children": [...] }, ... ] }, { "id": "GUID  ()", "name": "  ()", "children": [ { "id": "...", "name": "...", "children": [...] }, { "id": "...", "name": "...", "children": [...] }, ... ] }, ... ]


この構造は、ツリーをレンダリングするときにzTreeのデータソースとして変更なしで使用されます。



各ドキュメントは個別のサブフォルダに保存され、その名前はドキュメントのGUIDと一致します。 このフォルダー内に、実際にはドキュメントであるindex.htmlファイルが保存されます。 このファイルに加えて、ドキュメントに挿入された画像は個別のオブジェクトとしてドキュメントフォルダーに保存され、添付ファイルのサブフォルダーにはドキュメントに添付されたすべてのファイルが保存されます。



すべてのデータは、AWS JavaScript SDKのupload（）関数を使用して、ユーザーのブラウザーからS3に直接ダウンロードされます。 彼女は、S3にファイルをアップロードし、それらを部分に分割（マルチパートアップロード）し、いくつかのストリームでこれを行う方法を知っています。



たとえば、これはドキュメントに添付されたファイルが4つのストリームにロードされ、それぞれ6 MBの部分に分割される方法です。 このコードはabDocオブジェクト（ドキュメント）内で動作します。この場合はself =このオブジェクトのthisです。

httpUploadProgressイベントを使用すると、ダウンロードの進行状況を追跡し、ファイルの進行状況バーを更新できます。



s3オブジェクト（AWS JavaScript SDK）は、ID IDプール、ユーザーのトークンID、認証プロバイダーの名前を含む資格情報を使用します。 S3サービスへのすべての呼び出しは、ユーザーIDトークンの転送で行われます。 ユーザーには、ロールに関連付けられたポリシーで規定されている権利が付与されます。このポリシーは、アイデンティティプールで設定されます。 同じ原則がJavaScript SDKを介して他のAWSサービスで機能します。 IDトークンの有効期間は短い（1時間）後、更新トークンを使用して新しいIDトークンを取得する必要があります。 リフレッシュトークンの有効期間は365日です。



空き容量インジケータ



一見、これは非常に単純なことのように思えるかもしれませんが、完全に真実ではありません。







空きスペースのインジケータをバスケットの形で作成しました。下部が狭く、上方に向かって広がっています。 視覚的に充填可能なバスケットを、そのようなバスケットが物理的な世界でどのように満たされるかと対応させたいと考えました。 つまり、下から狭くなるため、最初はすぐにいっぱいになります。 そして、それがいっぱいになると、レベルは上向きに拡大するため、ゆっくりと上昇します。



これを実現するために、四角形のガウス面積式を使用しました 。







この式を使用して、バスケット上のポイントの座標からバスケット全体の面積を計算できます。これは、1 GBの最大空き領域制限に対応します。 さらに、ユーザーが満たしたボリュームの値に基づいて、単純な比率を使用して、ユーザーが占有するバスケットの部分の面積を計算します。



ユーザーが領域を占有すると、上記の式に基づいて、塗りつぶされたスペースの上部境界のY座標を計算します。 占有スペースの上部ポイントのX座標は、バスケットのコーナーの接線を介して表示されます。 一般に、Yの導出の結果、ひどい見た目ですが、普通の2次方程式が得られました。 ボード上の右下隅にあります。

開発と展開



プロジェクトに取り組んでいるのは2人だけであるため、開発プロセスは非常に簡単です。 BitBucketでgitとプライベートリポジトリを使用します。 マスターブランチは製品コードです。 それからのみデプロイします。 私たちはすべての機能/エラーを別々のブランチで処理します。 各ブランチには、独自のタスクトラッカーAB-TASKSに個別のタスクがあります。 ブランチは、できるだけ頻繁にマスターとマージしてデプロイしようとします。



展開プロセスでは、アプリケーションコードをS3バケットにロードする必要があります。 ここにはいくつかのニュアンスがあります。



上記で書いたように、CloudFrontを使用してコンテンツを配信します。 コンテンツがほとんど常にCloudFrontキャッシュから配信されるメカニズムを提供する必要がありますが、ファイルの更新バージョン（picture、css、js、htmlまたはその他のファイル）をダウンロードすると、CloudFrontはキャッシュ内のファイルを更新する必要があります。 そうしないと、ユーザーがキャッシュからファイルの一部を受け取ったり、ソースから更新されたファイルを受け取ったりするという不快な状況が発生する可能性があります（S3）。 これは予測不可能な結果につながる可能性があります。



アプリケーションコードを返すには、AB-DOCソースコードを含むS3バケットの形式の1つのソースと、2つのキャッシュルール（キャッシュの動作）を備えた個別のCloudFrontディストリビューションを使用します。



キャッシュルールは、CloudFrontがファイルをキャッシュする方法を定義します。 ルールを構成するには、要求パターン（* .htmlまたはimages / *。Jpgなど）と、要求が指定されたパターンと一致したときに適用される設定が設定されます。 設定には、キャッシングがアクティブなHTTPメソッド、TTLパラメーターまたはキャッシュ内のオブジェクトの存続期間（最小、最大、デフォルト）、圧縮、および実際にキャッシングが実行される基準（パラメーターが考慮されるかどうか、Cookie ...）、およびその他の多くが含まれますパラメータ。



2つのキャッシュルールを構成しました。

  • 最初のルールはHTMLファイル用です。 キャッシュ設定では、小さいTTLが設定されています（最小TTL = 100秒）。 これらのファイルにcache-control max-age = 0を設定すると、最小TTLが適用されます。つまり、HTMLファイルは100秒後に期限切れと見なされ、CloudFrontはソースを呼び出して、ファイルの最新バージョンがそこに表示されるかどうかを確認します。
  • 他のすべてのファイルの2番目のルール。 これらのファイルでは、cache-control max-ageファイルのヘッダーを設定しないため、最小TTLとデフォルトTTLの2つの最大値が使用されます。 私たちの場合、これは86400秒または24時間です。つまり、長い時間キャッシュされます。 ただし、これに加えて、ルール設定では、オブジェクトをキャッシュするときにCloudFrontがリクエストパラメーターを「見る」ことを示しています。




これは、 AWSのドキュメントで 、さまざまなTTLとキャッシュ制御設定がどのように組み合わされるかを説明している場所です。



クエリパラメータを使用すると、キャッシュを簡単に管理できます。 すべてのファイルについて、パラメータとしてファイルの条件付きバージョンを追加します： filename.extension？V = 123 。 ファイルが変更されている場合、パラメーター値を変更するだけで、CloudFrontはキャッシュからではなくソースにファイルを返します。



ただし、ファイルの変更を追跡し、それらへのリンクを手動で更新するのは非常に面倒であり、概して不可能です。 私たちはそれをすることを常に忘れていました。 そのため、これを自動的に行うbashスクリプトを作成しました。 同時に、S3でのアプリケーションコードの本格的な展開のために多くの機能を実行します。
