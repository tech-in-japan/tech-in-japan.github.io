Ivan Chukicは、KDEの開発者の1人であり、ベオグラード大学のプログラミング言語設計の教師および研究者です。







Alexander Granin （ graninas ）は、ノボシビルスクFPコミュニティLambdaNskの主催者である、FPを専門とする有名なスピーカー兼開発者です。

アレクサンダー： AFの「正しい」定義は1つもないと思いますが、個人的に言えば、AFは機能的な構成と一流の機能を備えたものです。







Ivan：同意しますが、より高次の関数を追加します。他の関数を引数として取り、結果として返すことができる関数です。







Sergey ：Cの関数へのリンク-それは考慮されますか？







Ivan：いいえ、Cは関数型プログラミング言語ではありません:-)







セルゲイ：理由を教えてください。







Ivan：関数ポインタの組み合わせから新しい関数を作成することはできないため、既存の関数ポインタのみを指すことができます。 もちろん、一部のアセンブラーハックが使用されなかった場合。







Sergey：素晴らしい、今公式の答えがあります！ 本格的な機能があるため、人々は常にCが関数型プログラミング言語ではない理由を常に質問します。 そして、ここにC ++がより理解しやすい関数型言語である理由があります...







Alexander： C ++が本当に機能的なプログラミング言語であるとは言いません。1つの言語に注ぎ込まれたパラダイムの束をサポートします。







Sergey：つまり、C ++は機能的なパラダイムをサポートしています。 ところで、なぜですか？ 高階関数を操作できるので、彼はそれをサポートしますか？







Ivan：そうですね、C ++ 98でも、STLでも高階関数が既に存在しているので、彼はいつもそうでした。 これは最も便利な関数型言語ではありません-FP以上を実装する言語があります。 しかし、私のニーズのために、それは常にかなり機能的でした。







セルゲイ：しかし、この場所からより詳細に。 あなたのニーズは何ですか？







Ivan：複雑です。 さあ、話をしてください。 私が大学にいたとき、私たちはLISPを使いましたが、このLISPはISPいので誰もが嫌いでした。 しかし、それから私が理解したのは、LISPコードでC構造を直接シミュレートする方法です。 そして、ある日、組み込みのforeach



などのためにリリースされたJavaの新しいバージョンがリリースされました。私は、新しいコンパイラ、言語の新しいバージョン、そして大学でLISPで行ったことを実装するために新しいものがすべて必要だと考えました、通常はループもサポートしていません。 その瞬間、私はFPが高レベルの抽象化を構築するのに非常に良いことであることに気づきました、そしてそれが私が2019年に機能的なC ++を使用する理由です。







Sergey：実際、高レベルの設計にはFPを使用しています。







アイヴァン：そのとおりです。







Alexander：現在、私はC ++では動作しませんが、データの操作に使用することを拒否しません。これは命令型アプローチよりもはるかに快適です。 それとは対照的に、OOPと比較した場合でも、ここでは変換のみが許容され、これは便利です。







Sergey： FPはC ++だけでなく使用できますよね？ では、次の質問：C ++のどの部分を使用していますか？ 設計上の問題のみが重要な場合は、FPに適した部分のみを選択できます。これは実際に使用する部分です。







Ivan：これらは間違いなくラムダ関数です。 さらに重要なのは、テンプレート。他の関数を引数などに渡すことができるためです。ラムダは、機能オブジェクトを作成するための優れた構文です。







セルゲイ：はい、あなたはラムダが本当に好きだとすでに気付きました:-)







Ivan：彼らが本当に気に入ったというわけではありませんが、これは明らかにC ++ 98で私たちが持っていた最高のものであり、彼らと一緒に仕事をする方が便利です。







Alexander：はい、私もラムダが好きです-この機能は非常に普遍的であるため、それらだけで書くことができます。 これは、任意のロジックを構築できるユニバーサルコンビネータのようなものです。他の言語ほど美しくはないかもしれませんが、それほど有用ではありません。







Sergey： Ivan、C ++ 11より前の標準、たとえば非常に一般的なC ++ 03などがあり、そこにはすでに機能的な機能があることに注意してください。 そして、新しい標準にはより機能的な機能があります... C ++がFPに向かっていると言うことは可能ですか？ この動きは続くか、止まりますか？ そして、それは何につながりますか？







Ivan：プログラミング言語全般に関するSimon Peython Jonesの優れた報告があり、彼はそこに多くの安全な言語と使用されている言語を示すグラフを描きました。 Haskellは、何もできない完全に安全な言語として物語を始めました-I / Oがなく、まったく何もないからです。 SPJは、C言語とアセンブリ言語を非常に有用であると同時に非常に危険なものとして分類しました。 それ以来、Haskellはセキュリティの強化に向けて動き始めました。 一方、C ++に表示されるこれらの機能は、主にセキュリティを向上させ、正しいプログラムをより簡単に作成できるように見えます。 そのため、これらのほとんどは関数型プログラミング言語に由来しています。







セルゲイ：どうしてそう思いますか？ AFの性質のためですか？







Ivan：はい、たぶん本質的には…でも、私にはわかりません。







アレクサンダー：テンプレートに苦労したり、バイトを再配置したり、何らかの低レベルのかすみにうんざりしているからこそ、AFがすべての人を魅了していると思います。







セルゲイ：つまり、あなたにとってこれは一種の知的運動ですか？







アレクサンダー：はい、そうです。







セルゲイ：わかりました。 馬鹿げた標準機能を実装するよりも、開発者が高レベルの抽象化について考えるのがどれほど面白いか-これは不快です。 次に、これが質問Cです。C++でのFPの実用的なアプリケーションの経験はありますか？ 本番プロジェクトはありますか？







アイヴァン：もちろんです。 世界最大のプロジェクトの1つであるKDEには、機能スタイルを集中的に使用するいくつかの部分があります。 もちろん、これは、従来のオブジェクト指向のC ++と一連の機能概念を組み合わせたものです。 私は決して純粋主義者やそのようなものになるつもりはありませんでした。 私は常に最高の異なる世界を組み合わせようとします。







Sergey： HaskellやScalaはどうですか？ 結局のところ、それらは本番環境で広く使用されています。 Haskellは今や関数型言語の標準と見なされているという考えをどのように気に入っていますか？ これは特に純粋主義者によって注目されています。







アイヴァン：はい、今日のHaskellはFPの同義語であることに同意します。 実際、Haskellの機能はすべて、FPに関連するものとして人々に認識されています。 これは必ずしも真実ではありませんが、Haskellは本当に最も人気のあるアカデミック関数型プログラミング言語になったと思います。 ロンドンと北ヨーロッパのいくつかの銀行がHaskellを広範に使用していることは知っていますが、現在Scalaははるかに人気があります。







アレクサンダー： Scalaの方が人気があることに同意しますが、Haskellはより機能的な言語のようで、その機能のほとんどはより正しく実装されています。 つまり、簡単なカレーを使用する場合、作曲を簡単に行う方法がある場合、プログラミングは簡単でシンプルになります。まるで森の中を歩いて景色を楽しむようなものです。







アイヴァン：しかし、時々森の中にクマがいます。 あなたがロシアにいるなら。







Sergey： C ++は主にHaskellに触発されていると思いますか？ それは価値がありますか？







Ivan：特定の委員会のメンバーについてほのめかしていますか？ :-)誰かがクラスクラスを理解した結果としてコンセプトが生まれたとほのめかしましたが、ビョルンはうわさを止め、コンセプトがクラスクラスとどのように異なるかについての文書を書きました。 私の観点からは、それらはすべてが異なりますが、同じ目的を果たします。 異なるアプローチ。

Alexander：もちろん、これはC ++を本当に良い言語に変えることができる最も重要な機能です。







Ivan：ところで、私たちはカンファレンスに参加しているので、質問もさせてください、セルゲイ。 会議は刺激的な仕事だとおっしゃいました。 それについてとても興味深いことを共有してください。世界の他の地域で独立して会議を開催するように、私またはグラニン同志にアドバイスしてもらえますか？

それでは、トピックに戻りましょう。 関数型プログラミングについて話しましたが、あなたの報告が私を納得させるという意味で、あなたは私をほぼ納得させました。 C ++の機能的なアプローチは、さまざまなものを同期する必要があるときにマルチスレッド化に役立つと疑われています。 本当に役立つのでしょうか？







アイヴァン：もちろんです。







アレクサンダー： C ++での機能的なマルチスレッドの使用経験は限られていますが、純粋な関数の世界がある場合、マルチスレッド環境でそれらについて話すのははるかに簡単です。 たとえば、競争力のあるロジックを作成する場合、これらすべてを同期すること、ミューテックス、クリティカルセクション、何かについて考える必要はありません。 コードは通常のシーケンシャルコードと見なされ、すべてのマルチスレッドと同期は内部のどこかに隠されているため、これはすべて消えます。 マルチスレッドプログラミング、および関数型には多くのアプローチがあります。 これが絶対にすべてのアプローチで起こるかどうかはわかりませんが、たとえば、ソフトウェアトランザクションメモリは、競合するアプリケーションの複雑さを軽減するための優れたアプローチです。 残念ながら、適切な妥協案を選択することです。







Ivan：コンパイラーがすべての作業を行うとき、効率的にそれを支払う必要があります。







アレクサンダー：まあ、さまざまな問題があります。 開始するには、STMのようなこれらすべてのことを理解してから、この秘密の知識を同僚に伝える必要があります。 そして、特定の実装および場所にエラーがあり、手動のスレッド制御を使用すると、はるかにうまく機能します。 しかし、そのような手動制御よりも速く簡単に書くことができます。 実行は遅くなりますが、コードのエラーは少なくなります。 ところで、Ivan、この問題がカンファレンスでどれだけうまくカバーされているかについてどう思いますか？







アイヴァン：このトピックは活況を呈しています。 近年、すべての主要な会議-CPPConf、C ++ Russia、Meeting C ++など。 -FP、代数的データ構造、またはそのようなもののいずれかで直接受信したレポート。 時には、講演者は、彼らのレポートでFPからの概念について話しているとは思わないことさえあります。 C ++では、物事は多くの異なる場所から来ます...人々は通常1つの言語だけで書くことはありません。 イラン・イワノフがErlangとC ++で書かれたプロジェクトに取り組んでいると想像してください。 その後、Tatyana Petrovnaが到着し、彼女はすでにHaskellできれいな機能を使って働いており、彼らはお気に入りのメカニズムをC ++に移植し、その結果、さまざまなコミュニティの膨大な数の人々がC ++にますます多くのものをもたらします。 これはすべて目の前で起こります。 少なくとも、これはC ++開発者コミュニティで発生しますが、C ++企業に関しては、私にはよくわかりません。 ただし、C ++コミュニティの多くの人々が現在、機能概念に取り組んでいます。







Alexander：多くのトップC ++開発者がHaskellを研究するのは、C ++で何が起こっているのかを理解するためだけであると正しく理解しましたか？







Ivan：この理由で彼らがHaskellを教えるかどうかはわかりません。 C ++開発者はとても利己的だと思います。 複雑だからといって、C ++をとてもよく学んだ。 そして、あなたが本当に新しい何かを学びたいのであれば、あなたの道は明らかに単純であるように特別に作成されたJavaにはありません。 最も奇妙な奇妙で奇妙な言語の分野を調べる必要があり、Haskellは自動的に最も人気のある答えの1つになります。 人はそれを見て理解します。ああ、これはC ++よりも複雑なものです。あなたは学ぶ必要があります。 私がHaskellを勉強したとき、それは私と同じでした。そして、私はまったく同じ推論の道筋をたどってきた友人がいます。







アレクサンダー：エリック・ニブラーがシベリアで私たちと一緒にいて、彼のライブラリーの範囲を示したとき、彼はしばしばインスピレーションの源は何かと尋ねられました。 彼はHaskellだと答えた。 行のすべての機能を使用することはできませんが、コミュニティには明らかに必要な機能があります。







アイヴァン：少し進化しています。 遺伝物質。 また、HaskellはC ++から何かを取得することでも改善できます。 ほとんどの言語はこのような進化を遂げています。 JavaはC＃のLINQをそれ自体に適合させようとし、C＃のLINQ作成者はHaskellなどからインスピレーションを引き出しました。 異なる言語間で相互に影響を与えるこのような美しい絡み合ったネットワークが判明しました。







Alexander：しかし、C ++はまだ低レベル言語ですか？







アイヴァン：ほとんどの人はそう思います。







Sergey：どんな「低レベル」について話しているのですか？

Sergey：範囲自体はブレーキをかけていませんが、デバッグモードの範囲-これは議論全体の本質です。 リリースモードでは、正常に機能します。







アイヴァン：これは正常です。







セルゲイ：誰もがこれに同意するわけではありません:-)

Sergey：私は通常、インタビューでそのような質問をします:-)これは非常によくある質問です。







Ivan：はい、よくある質問はソートの実装方法です。 次に、クイックソートの基本バージョンについて説明します。これは、実際には世界の他のどこでも使用されていません。 標準ライブラリでは、N log Nが保証されている必要があり、高速ソートができないため、標準ライブラリの使用は許可されていません。 ほとんどの場合、それはできますが、この場所の標準は非常に厳密であり、累積N log Nを意味するものではなく、純粋なN log Nである必要があり、そのようなアルゴリズムをどのように実装しますか？ libstdc ++で行われているように、すばやくソートするためのさまざまな最適化を調査し、少なくとも3つの異なるアルゴリズムで構成される1つのアルゴリズムにマージする必要があります。 これは標準ライブラリの意味です-プログラムするためにこれらすべてを知る必要はありません。 最も効率的な方法ですべてを実装する方法を理解する必要はありません。他の誰かがすでにこれを処理してくれています。 したがって、「STLは非常に複雑です。STLを使用せずに、すべてを手作業で最初から記述してください」と言うとき、私はこのアプローチが好きではありません。
