NimとRustを比較する

Habrahabrの読者に、記事「Nimと さび 。 」 私のコメントは斜体で表示されます。



RustとNimは、私がフォローしている2つの新しいプログラミング言語です。 Rustに関する最初の投稿の直後に、 Nim 0.10.2がリリースされました 。 これにより、Nimをより密接に知り、自然にRustと比較するようになりました。



この投稿では、NimとRustで作成された2つの簡単なプログラムと、実行時間の大まかな比較を示し、これらの言語でのプログラミングの主観的な印象を表現します。







例1：単語のカウント（wordcount）





この例では、ファイルI / O、正規表現、ハッシュテーブル（連想配列）、およびコマンドに渡された解析引数を使用します。 名前が示すように、プログラムはfilesまたはstdin内の単語の数をカウントします。



使用例： 



Usage: wordcount [OPTIONS] [FILES] Options: -o:NAME set output file name -i --ignore-case ignore case -h --help print this help menu




-i引数を渡すと、結果は次のようになります。 



 2 case 1 file 1 files 1 h 2 help ...




Nimバージョン





Nimのプログラムは非常に単純です。 テーブルを使用します。 単語カウント用のCountTable parseopt2 。 コマンド引数とsequtilsを解析するためのgetopt 。 機能的なマッピング操作のためのmapIt 。 正規表現については、 reではなくNimのドキュメントで推奨されているpegsモジュールを選択しました。



ディレクティブ{.raises：[IOError]。} 3行目では、 doWorkプロシージャがIOError例外のみをスローするようにしています。 これを行うには、 input.findAll（peg "\ w +"）を21行目のtry式内に配置して、理論的に発生する可能性のある例外をキャッチします。



wordcount.nimコードの一部： 



 proc doWork(inFilenames: seq[string] = nil, outFilename: string = nil, ignoreCase: bool = false) {.raises: [IOError].} = # Open files var infiles: seq[File] = @[stdin] outfile: File = stdout if inFilenames != nil and inFilenames.len > 0: infiles = inFilenames.mapIt(File, (proc (filename: string): File = if not open(result, filename): raise newException(IOError, "Failed to open file: " & filename) )(it)) if outFilename != nil and outFilename.len > 0 and not open(outfile, outFilename, fmWrite): raise newException(IOError, "Failed to open file: " & outFilename) # Parse words var counts = initCountTable[string]() for infile in infiles: for line in infile.lines: let input = if ignoreCase: line.tolower() else: line let words = try: input.findAll(peg"\w+") except: @[] for word in words: counts.inc(word) # Write counts var words = toSeq(counts.keys) sort(words, cmp) for word in words: outfile.writeln(counts[word], '\t', word)




錆バージョン





Rustをよりよく理解するために、 コレクション:: BTreeMapに似たシンプルなBTreeMap構造を実装しましたが、最終的にNimとの公正な比較のためにコレクション:: HashMapを使用しました（レビューのためにBTreeMapコードはリポジトリに残りました）。 getoptsパッケージは、コマンド引数を解析してConfig構造体にするために使用されます。 さらに、すべてが明確でなければなりません。



私の錆ワードカウントプロジェクトのコードの一部： 



 fn do_work(cfg: &config::Config) -> io::Result<()> { // Open input and output files let mut readers = Vec::with_capacity(std::cmp::max(1, cfg.input.len())); if cfg.input.is_empty() { readers.push(BufReader::new(Box::new(io::stdin()) as Box<Read>)); } else { for name in &cfg.input { let file = try!(File::open(name)); readers.push(BufReader::new(Box::new(file) as Box<Read>)); } } let mut writer = match cfg.output { Some(ref name) => { let file = try!(File::create(name)); Box::new(BufWriter::new(file)) as Box<Write> } None => { Box::new(io::stdout()) as Box<Write> } }; // Parse words let mut map = collections::HashMap::<String, u32>::new(); let re = regex!(r"\w+"); // let re = Regex::new(r"\w+").unwrap(); // let re = regex!(r"[a-zA-Z0-9_]+"); // let re = Regex::new(r"[a-zA-Z0-9_]+").unwrap(); for reader in &mut readers { for line in reader.lines() { for caps in re.captures_iter(&line.unwrap()) { if let Some(cap) = caps.at(0) { let word = match cfg.ignore_case { true => cap.to_ascii_lowercase(), false => cap.to_string(), }; match map.entry(word) { Occupied(mut view) => { *view.get_mut() += 1; } Vacant(view) => { view.insert(1); } } } } } } // Write counts let mut words: Vec<&String> = map.keys().collect(); words.sort(); for &word in &words { if let Some(count) = map.get(word) { try!(writeln!(writer, "{}\t{}", count, word)); } } Ok(()) }




Zachary Dremannは、find_iterを使用するプルリクエストを提案しました 。 Nimバージョンとの一貫性のためにcaptures_iterを残しましたが 、コードを少し改善しました。

ニムまたは錆





NimとRustは優れたパフォーマンスを期待してコンパイルされた言語ですが、それらは非常に異なっています。 私にとって、それらの類似点は次のとおりです。



  • コンパイルおよび静的型付け

  • 良好なパフォーマンスを得るための計算（各プログラムは、プログラムの実装とその最適化に応じてより高速に動作します）

  • 継承ではなく構成（新しい言語の傾向に似ていますか？）

  • Cの単純な束

  • 一般的な言語の利点：ジェネリック、クロージャ、機能的アプローチ、型推論、マクロ、ステートメント演算子など





しかし、それらの違いはより興味深いものです。



哲学：自由または規律





Nimでプログラミングすると、スクリプト言語で書いているような感覚が得られます。 彼は本当に行を消去します。 Nimは、コード内のノイズを可能な限り取り除き、それを喜んでプログラミングしようとします。



ただし、そのような自由には裏返しがあります。明確さ、純度、持続可能性が損なわれる可能性があります。 小さな例を次に示します。Nimのインポートでは、すべてのモジュール名がネームスペースにインポートされます。 インポートされたモジュールからの名前は、構文module.symbolを使用するか、名前の制御されたインポートのためにfrom module import nilを使用して制限できますが、これを使用するのは誰ですか？ さらに、このアプローチはNimには一般的ではありません。 その結果、他の誰かのコード（または自分のコード）を読み取るときに、どのモジュールからどの名前が付けられたのかを理解できません（残念ながら、このような場合、Nimはハエをカツレツから分離するため、名前の競合は発生しません）。



その他の例： UFCSでは、 len （x） 、 len x 、 x.len（）またはx.lenを必要に応じて使用できます。 名前をアンダースコアと異なる大文字で区切らないので、 mapWidth 、 mapwidthおよびmap_widthは同じ名前に変換されます（バージョン0.10.2に「部分的な大文字と小文字の区別」のルールが含まれていて、 Fooとfooは異なる名前と見なされます） ; 物事の順序で初期化されていない変数の使用です。 理論的には、厳密なコーディング原則に従うことができますが、Nimでプログラミングするときは、よりリラックスした気分になります。



一方、錆は規律を尊重します。 そのコンパイラは非常に厳格です。 すべてが非常に明確でなければなりません。 事前に適切なアプローチを取得します。 あいまいさはRustコードに関するものではありません...

このアプローチは、原則として、長命のプロジェクトと持続可能性に適していますが、Rustでプログラミングを行う場合は、まったく関心のないような詳細の処理を開始します。 タスクの優先事項ではない場合でも、メモリの使用やパフォーマンスの向上について考えるようになります。 Rustにより、規律が向上します。



どちらにも長所と短所があります。 プログラマーとして、私はNimをもっと楽しんでいます。 メンテナーとして、Rustで書かれた製品の方が良いでしょう。



ビジュアルスタイル：PythonまたはC ++





Pythonと同様に、Nimはインデントを使用してコードブロックを分離し、 文字数を減らしています 。 RustはC ++に似ています。 {} 、 :: 、 <> 、および＆は、C ++プログラマーにはおなじみです。さらに、Rustは'aのような新しいものを追加します。



時々、ニムは文字通りすぎるかもしれません。 たとえば、Rustの一致構文を考えます。 



 match key.cmp(&node.key) { Less => return insert(&mut node.left, key, value), Greater => return insert(&mut node.right, key, value), Equal => node.value = value, }




Nimのcase式よりもきれいに見えます。 



 case key of "help", "h": echo usageString of "ignore-case", "i": ignoreCase = true of "o": outFilename = val else: discard




しかし、全体的に、Nimコードはノイズが少ないです。 私の意見では、Rustの特別な混乱は存続期間パラメーターによって導入され、これは変更されません。



メモリ管理：ガベージコレクターまたは手動管理





Nimは安全でないメモリ管理を可能にし、より予測可能な動作のためにランタイムでガベージコレクタを管理するためのサポートを提供しますが。 それはまだガベージコレクターを備えた言語であり、それに長所と短所があります。 Nimのオブジェクトには、値のコピーが割り当てられます。 ガベージコレクターがタスクに干渉しない場合、Nimでメモリを管理しても問題は発生しません。



Rustはガベージコレクションの限定的なサポートを提供しますが、多くの場合、メモリ管理ではテニュアシステムに依存しています。 Rustプログラマーとして、初心者にとって最初の障壁となるプログラムを効率的に作成する前に、彼のメモリ管理モデル（所有権、借用、寿命）を完全に理解する必要があります。



一方、Rustの強みはこれにあります。ガベージコレクターを使用しない安全なメモリ管理です。 Rustはこれについて素晴らしい仕事をします。 共有リソースのセキュリティ、競合するデータアクセスのセキュリティ、およびNULLポインターの削除に加えて、Rustは実行時のリソース消費のオーバーヘッドのない非常に信頼性の高いプログラミング言語です。



要件に応じて、Nimガベージコレクターで十分であるか、選択がRustになります。



その他の違い





ニムの強み：



  1. 生産性：同じ時間枠で、Nimでより多くの機能を利用できます
  2. 習得が簡単
  3. スクリプト言語としてコンパイルされた言語は、タイピング、インタラクティブな調査、バッチ処理などに適しています。
  4. チップ：

    • メソッドのオーバーライド
    • 新しい演算子の定義
    • 名前付き引数とデフォルト値
    • 強力なマクロ






さびの強さ：



  1. 実際のシステムプログラミング言語：組み込み、ガベージコレクターなし、 ハードウェアに近い
  2. 安全、規律、信頼性
  3. 強力なカーネルチームと活発なコミュニティ
  4. チップ：

    • 優れたパターンマッチングの実装
    • enums 、ただしNimではenumも良い
    • varの代わりにmutをしましょう （小さいが重要なこと）
    • 構文を逆参照する強力な構造






エラー処理：Nimは一般的な例外メカニズムを使用し、Rustは戻り値型Result （およびパニックマクロ！ ）を使用します。 私はこれを好みませんが、この違いに言及することが重要であることがわかりました。



途中リリース1.0





NimとRustは今年リリースされる予定です（Rustはリリースされました ） 。 これはとてもクールです！ Rustはすでに多くの注目を集めていますが、ニムはより有名になっています。 味はまったく異なりますが、どちらも素晴らしい新しいプログラミング言語です。 Rustはパフォーマンスとセキュリティの点で最善を尽くします。 Nimはアジャイル（しゃれ：Nimは機敏です）で表現力豊かで、スクリプト言語とコンパイル言語の長所を実現しています。 どちらもツールキットに追加するのに最適です。
