デバイスとシミュレーターのコンソールからiOSアプリケーションを起動します

この記事では、コンソールコマンドを使用してアプリケーションをアセンブルし、Xcodeを開かずに実際のデバイスとシミュレーターで実行する方法について簡単に説明します。



一般に、私の会社には、アプリケーションのバンドルを扱う10人以上のios開発部門があるという事実から始める価値があります。 ルーチン作業を自動化するために、CIサーバーを展開する必要がありました（特別な歴史的理由により（他のすべての人と同様に）自己記述で、jenkinsに移行する予定です）。 私は日常的なことを考えます-プロジェクトの組み立て、テストの実行（もちろん、あなたが書いていない限り）、テスター用とアプリストアでのレイアウト用のipashkaの作成。 一般的に、gitaでボタンまたはフックを押すことで、これらすべてが機能することを望んでいました。 開発者がボタンをクリックするだけですべてが機能しますが、プログラムはこれまでのところ計画内にあります。 この記事では、プロジェクトをコンパイルし、それをipaファイルにパックするというトピックについてのみ触れます。 CIサーバー側のデバイスでプロジェクトを起動する機能はまだ開発中ですが、パッケージングプロセス自体は長い間「戦い」にあり、誰かに役立つ可能性があります。



私たちは何を計画していますか：



  • アプリケーションをアセンブルし、シミュレーターで実行します
  • IPAを収集して、実際のデバイスで実行しましょう
  • 市場にアップロードするためにIPAを収集しましょう
  • わずかな組み立てエラー


私たちが持っています：



  • OS X 10.10.5
  • Xcode 7.1


実験の頻度については、サードパーティのアプリケーションを使用して、親愛なる読者が記事で説明されていることを繰り返すことができるようにします。 オープンソースのresearchKit apple-ovフレームワークのテストアプリケーションを使用します。 



git clone https://github.com/ResearchKit/ResearchKit.git ~/Downloads/researchkit
      
      

        
        
        
      

    
        
        
        
      
      

        
        
        
      

    
     


 cd ~/Downloads/researchkit


プロジェクトをコンパイルする前に、apple doc generator github.com/tomaz/appledocをインストールする必要があります（インストール方法の説明があります）。

ダウンロードしたプロジェクトに渡します。 ルートプロジェクトはResearchKitフレームワーク自体であり、テストアプリケーション自体は別の場所にあります。 



 cd samples/ORKCatalog/


アプリケーションをコンパイルしようとしています。 シミュレータ用のプロジェクトのアセンブリには証明書が必要ないため、シミュレータ用に収集します/ファイルが提供されます。 



 xcodebuild -project ORKCatalog.xcodeproj -scheme ORKCatalog -arch x86_64 VALID_ARCHS=x86_64 -sdk iphonesimulator


エラーが発生します：







ORKCatalogスキーマがないため、ビルドがクラッシュします。 元のプロジェクトのスキームは共有としてマークされていなかったため、gitからの「ドレイン」の後、xcodeはまだこのスキームについて何も知らないので、雪が降るので、プロジェクトを開くだけです。 したがって、プロジェクトを開くだけです： 



 open ORKCatalog.xcodeproj


そして、すぐに、スキームが表示されたら、 xcodebuild -listを使用してスキームを確認できます 。 再度収集を試みます。 



 xcodebuild -project ORKCatalog.xcodeproj -scheme ORKCatalog -arch x86_64 VALID_ARCHS=x86_64 -sdk iphonesimulator


待ちに待った** BUILD SUCCEEDED **を確認した後。 素晴らしい、すべてが機能します。



シミュレーターでプロジェクトを実行する



シミュレーターでコンパイルします。アーキテクチャーはi386 / x86_64です。 SYMROOTを使用して、ビルド結果のパスを設定します。 



  xcodebuild -project ORKCatalog.xcodeproj -scheme ORKCatalog ARCHS='x86_64 i386' VALID_ARCHS='x86_64 i386' -sdk iphonesimulator -configuration Debug SYMROOT=$(pwd)/build build


（考えを声に出してください：記事を書いたとき、リリースモードでのコンパイルは機能していましたが、記事を公開する前にすべてのステップを再度確認し、このモードでのコンパイルが機能しなくなったため、デバッグで収集しているため、最後のコミットのいくつかがこれを破りました。）



ビルドが成功した後、 build / Debug-iphonesimulator /に ORKCatalog.appファイルを受け取りました。 シミュレータでこれを実行することは残っています。 これには、 ios-simユーティリティgithub.com/phonegap/ios-simを使用します。 使い方はとても簡単です。



利用可能なシミュレーターのリストを取得します。 



 ios-sim showdevicetypes






選択したリストから、「iPhone-6-Plus」。 その上でアプリケーションを実行します。 



 ios-sim --devicetypeid iPhone-6-Plus launch build/Debug-iphonesimulator/ORKCatalog.app


すべてが正しく行われた場合、アプリケーションを備えたシミュレーターが開始するはずです（ctrl + Cを使用して、コンソールで入力モードに入ります）。







IPAファイルを作成し、実際のデバイスで起動する



ここではもう少し複雑です。開発用のmobileprovisionファイル（開発者）とマシン上の証明書（p12ファイル）が必要であり、xcodeのアカウントは必要ありません。 コンソールからアプリケーションに署名/パッケージ化する場合、xcodeでアカウントを追加する必要はありません。これは、たとえば、p12ファイルのみを保存できるCIサーバーで非常に役立ちます。



マシンには対応する証明書があると想定しています。 アカウントのdeveloper.apple.comで開発者mobileprovisionを生成し、マシンにダウンロードしたら（test.mobileprovisionという名前を付け、バンドルIDはru.habrahabr.testになります）。 xcodeが取得できるディレクトリにコピーした後： 



 cp test.mobileprovision ~/Library/MobileDevice/Provisioning Profiles/


デバイスのアーカイブを収集します（これはarm64 / armv7のアーキテクチャです）。 



 xcodebuild -project ORKCatalog.xcodeproj -scheme ORKCatalog ARCHS='arm64 armv7' VALID_ARCHS='arm64 armv7' -sdk iphoneos -configuration Debug archive -archivePath build/archive


バンドルIDをアプリケーションにまだ登録しておらず、mobileprovisionをリンクしていないため、コンパイルは失敗します。エラーは次のようになります。







コンソールからバンドルIDを渡し、Info.plistファイルに登録する必要があります。 また、コンソールを介してリンクをmobileprovisionに転送します。 パスORKCatalog / Supporting Files / Info.plistに沿ってPlistファイルを見つけます。CFBundleIdentifierキーの場合、値をru.habrahabr.testに設定します。 バンドルIDは、ID PRODUCT_BUNDLE_IDENTIFIER = en.habrahabr.testの値を持つキーを介して渡されます。 mobileprovisionに登録されているUUIDを使用して、PROVISIONIG_PROFILEキーを介してmobileprovisionへのリンクを渡します。



UUIDを取得します。 



 security cms -D -i test.mobileprovision | grep -o "\w\{8\}-\w\{4\}-\w\{4\}-\w\{4\}-\w\{12\}"


値は87b0df89-793a-4a0f-92bf-c5f9c35f1405に似ています。 再度収集します。 



 xcodebuild -project ORKCatalog.xcodeproj -scheme ORKCatalog ARCHS='arm64 armv7' VALID_ARCHS='arm64 armv7' -sdk iphoneos -configuration Debug archive -archivePath build/archive PRODUCT_BUNDLE_IDENTIFIER=ru.habrahabr.test PROVISIONING_PROFILE=87b0df89-793a-4a0f-92bf-c5f9c35f1405


その結果、 build / archive.xcarchiveアーカイブを取得しますが、これはipaにパッケージ化されたままです。 Xcode 7は新しいパッケージング方法を導入し、それを使用します。 これの前に、次の内容で構成ファイルoptions.plistを作成します。 



 <?xml version="1.0" encoding="UTF-8"?> <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd"> <plist version="1.0"> <dict> <key>method</key> <string>development</string> <key>uploadSymbols</key> <false/> </dict> </plist>


IPAを構築しようとしています： 



 xcodebuild -exportArchive -exportOptionsPlist options.plist -archivePath build/archive.xcarchive -exportPath build/dev-ipa/


ログによると、アセンブリがクラッシュし、何かに資格があることを理解できます。







誤って、アプリケーションがエンタイトルメントに署名していることは明らかです。エンタイトルメントの値は、モバイルプロビジョニングの値、つまりcom.apple.developer.healthkitに対応していません。 理由を探しています。 アプリケーションが署名されている資格を確認します。 



 codesign -d --entitlements - build/archive.xcarchive/Products/Applications/ORKCatalog.app


取得するもの： 



 <dict> <key>application-identifier</key> <string>XXXXX.ru.habrahabr.test</string> <key>beta-reports-active</key> <true/> <key>com.apple.developer.healthkit</key> <true/> <key>com.apple.developer.team-identifier</key> <string>XXXXX</string> <key>get-task-allow</key> <false/> </dict> </plist>


これを除いて、すべてが正常であることがわかります。 



 <key>com.apple.developer.healthkit</key> <true/>


mobileprovisionにはこのオプションがありません。それがどこから来たのかを知る必要があります。 



 find ORKCatalog/ -name "*.entitlements" -type f


検索により、 ORKCatalog / Supporitng Files / ORKCatalog.entitlementsが得られました 。 私たちは内部を見ます： 



 cat ORKCatalog/Supporting Files/ORKCatalog.entitlements


値は1つだけです。 



 <plist version="1.0"> <dict> <key>com.apple.developer.healthkit</key> <true/> </dict> </plist>


論理的には、この値を追加する必要があるmobileprovisionを再割り当てする必要がありますが、私たちは怠zyであり、テストのためにこの値のない資格でアプリケーションに再署名する必要があります。



少なくとも2つのオプションがあります。



1.既存のエンタイトルメント（ORKCatalog / Supporitng Files / ORKCatalog.entitlements）を編集して、再構築します（アーカイブ経由）。

2.再構築せずに、私たち自身が必要な資格でORKCatalog.appに再署名します。



最初のオプションをよりシンプルに選択します。 したがって、ファイルORKCatalog / Supporitng Files / ORKCatalog.entitlementsから次の行を削除するだけです 。 



  <key>com.apple.developer.healthkit</key> <true/>


アーカイブを再度再構築します。 



 xcodebuild -project ORKCatalog.xcodeproj -scheme ORKCatalog ARCHS='arm64 armv7' VALID_ARCHS='arm64 armv7' -sdk iphoneos -configuration Debug archive -archivePath build/archive PRODUCT_BUNDLE_IDENTIFIER=ru.habrahabr.test PROVISIONING_PROFILE=87b0df89-793a-4a0f-92bf-c5f9c35f1405


ipaを作成した後： 



 xcodebuild -exportArchive -exportOptionsPlist options.plist -archivePath build/archive.xcarchive -exportPath build/dev-ipa/


待望のメッセージ** EXPORT SUCCEEDED **が表示されます。 build / dev-ipa /の下に、デバイスにインストールするipaファイルが表示されます。 ios-deploy github.com/phonegap/ios-deployを使用してデバイスにインストールします。 デバイスを車にくっつけ、デバイスIDを取得します： 



 ios-deploy -c


デバイスに展開します。 



 ios-deploy -i <device id> -b build/dev-ipa/ORKCatalog.ipa


市場のIPAファイルを収集します



すべては開発バージョンと同じです。リリースmobileprovisionへのリンクを変更するだけで、 options.plistで開発の代わりにアプリストア を配置します（他のオプションについては、xcodebuild -helpヘルプを参照してください）。



まとめ



xcodeを使用しなくても、テスト用およびアプリストアでのレイアウト用にipaファイルを収集できました。 これはすべて、CIサーバーで簡単に自動化できるため、開発者の作業が楽になります。



PS：複雑な構造のアプリケーションをコンパイル/署名するプロセスでは、複数のターゲットがそれぞれ独自のモバイルプロビジョニングファイル（拡張機能、クロック、組み込みフレームワークを備えたアプリケーション）を必要とする場合、上記のプロセスはファイルなしでは機能しないことに注意してください。
