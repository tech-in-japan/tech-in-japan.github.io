Flex 3 vs. エンタープライズ開発におけるSilverlight 3



FlexとSilverlightを比較する記事は豊富にありますが、両方のテクノロジーを比較するエンタープライズ開発について一言も見たことがありません。 Flexのプロである多くの人々は基本を引用していますが、企業環境ではこれは主な議論ではなく、あなたの決定に影響する事実の1つにすぎません。



最初から.NETプラットフォームでエンタープライズソフトウェアを開発した経験があり、最近休憩を取り、数か月かけて非常に大きなFlexアプリケーションを操作しました。



リッチインターネットアプリケーション（RIA）は成長しています。 Adobeは、より開発者指向のFlexであるFlash製品を作成しました。 マイクロソフトは最近、Silverlightで市場に参入しました。



私の意見では、アドビには見栄えの良いアプリを備えた強力な設計ポートフォリオがあります。 Microsoftと.NETプラットフォームは、ビジネスアプリケーションの開発におけるリーダーであり、Silverlightにマークを残しました。Silverlightでは、ビジネス機能が外観よりも重要です。



この記事では、2つのテクノロジーの比較を通じて、Silverlight統合ビジネス機能を提供する価格がFlexの価格よりも大幅に低いことを示すよう努めています。



言語-C＃vs. Actionscript



C＃は完全にオブジェクト指向のプログラミング言語であり、ビジネスアプリケーションの開発において主導的な地位を占めています。開発者は当たり前だと思っています。 リストするいくつかの違い：



C＃

  • 10進データ型
  • スレッドのサポート
  • ジェネリック
  • メソッドのオーバーロード
  • オペレーターの過負荷
  • 構造と列挙
  • プライベートコンストラクター
  • 抽象キーワード
  • ラムダスとリンク
  • パブリックゲッター/プライベートセッター
  • 自動プロパティ
  • オブジェクト初期化子
  • 変数の無条件型付け


Actionscript：

  • 動的なクラスとタイプ
  • Actionscriptのイベントは文字列を介してリンクされているため、リファクタリングで問題が発生する可能性があります。
  • 内部アクセス修飾子は、c＃での実装と比較すると、少し無意味なようです。


これらの欠落している機能は、ダーティーで、オプションで乱雑なオブジェクト指向コードの一貫した結果です。 タイプの強制は、たとえばコレクションを参照する場合などに最も頻繁に必要になります。 しかし、Actionscriptで必要なことを行うことは依然として技術的に可能です。その結果、コードは必然的にアマチュアになり、タイプセーフが少なくなり、保守が難しくなります。



Flexには10進数型のサポートがないことが知られていますが、この問題は金融アプリケーションを開発する際に特に深刻です。 浮動小数点を使用する数学は、ほとんどの場合、財務計算には十分ではありません。詳細については、 こちらをご覧ください 。 FXCompsは10進数の実装をFlexにもたらしましたが、パフォーマンステストを実行すると、10進数の除算が浮動小数点の除算より3,000倍遅いことがわかりました。 10進数の乗算は、浮動小数点の乗算よりも650倍遅いです。 正直なところ、10進数の計算はC＃のfloatより遅いですが、 ここと同じスケールではありません。 この項目は、金融アプリケーションの作成を検討している人々にとって、Flexの適合性について深刻な疑念を抱くのに十分なはずです。



Actionscriptのこの欠点およびその他の欠点の結果として、コンパイラはコンパイル時に問題を検出する効率が低下します。 コンパイルはC＃よりも遅くなります（Flexはコード100,000行ごとに2分必要です）。 簡単に言えば、ActionscriptはC＃よりもはるかに遅いです。



言語自体に触れることなく、C＃に.NET Frameworkがあり、その多くがSilverlightで利用できることに気付くでしょう。 この点で、Flex / Actionscriptをまだ実装する必要があります。



MXML対 Xaml



MXMLとXAMLは、同様のことを達成しようとしています。 どちらもUIとアニメーションを見つけるためのマークアップ言語です。 MXMLは比較的単純で直感的であるため、経験豊富な開発者はMXMLをすばやく習得できます。 一方、XAMLはそれほど厳密ではなく、より強力ですが、学ぶことも困難です。



Silverlightは、コードとマークアップの分離により貢献し、部分クラスがよりきれいな分離コードモデルを提供します。 Flexでは、スクリプトブロックをMXMLファイルに配置できます。これにより、プレゼンテーションコードモデルでUIマークアップファイルが膨れ上がります。



フレームワークコンポーネント



すぐに使用できるFlexには、さらに包括的なコンポーネントセットがあります。 これらのコンポーネントのより複雑な機能が必要な場合、品質に常に互換性があるとは限りません。



Advanced DataGridの内部で作業したFlex開発者に質問してください。 （ 問題の説明 ）。 私は、人々が手動でコントロールの焦点を合わせ、タブイベントを処理し、グリッドのコンボボックスのスクロールバーのコードにコードを実装しているのを見ました。 機能のわずかな複雑さが必要になるとすぐに、Flexのコードの複雑さが指数関数的に増大することがわかりました。



マイクロソフトは、Flexのいくつかの機能をうまく吸収しました。 ただし、Silverlightには欠けているものがあります：ビデオとオーディオの録音、印刷。



サードパーティのコンポーネント



Microsoftはここで特に優れています。 Infragistics 、 Syncfusion 、 Telerikからのオファーをご覧ください。 これらの提案を使用すると、ほとんどコードを記述することなく、強力なUIをすばやく作成できます。



Flexには、 IBMのILOG Elixirなどの商用およびオープンソースのオファーもあります。 しかし、Microsoft向けのコンポーネントを何年も開発している企業があります。 Flexはより限定的なオファーを提供します。



テンプレートとプラクティス、フレームワークと管理の反転（IoC）



Silverlightには、Composite WPF、CLSA.net、Spring.netなど、すぐに使用できるフレームワークが多数あります。 そのようなフレームワークを使用すると、一貫性のあるサポートされたコードを記述するための反復パターンがインストールされるため、開発者の生産性が大幅に向上するだけでなく、すでに実装されているプロジェクトで記述する必要がある最も複雑なコードの多くが含まれます。



繰り返しますが、.NETには、コードの依存関係を減らすことができる優れたIoCフレームワークがいくつかあります。 私のお気に入りのフレームワークの1つはGoogle Autofacです。 そして、Actionscriptはその方向に動いていますが、それに関してはまだC＃に遅れをとっています。 SpicefactoryのParsleyは優れていますが、Springのような大規模な構成ファイルが必要です。 AdobeにはLifeCycle ESがあり 、これには多くの機能がありますが、特定の制限が課せられることもあります。



IDE /リファクタリングツール



大規模なプロジェクトでは、IDEにより開発者の時間を大幅に節約でき、お金も節約できます。 逆に、機能が制限された不安定なIDEは、プロジェクトの遅延につながります。



これは、Silverlightがリーダーである分野の1つです。 Visual Studio 2008は、特にResharperなどのリファクタリングツールと組み合わせると、優れたIDEです。



FlexBuilderおよびEclipseには、次のような特定の欠点があります。

  • 実質的にリファクタリングのサポートはありません。 バルクコードでは、検索や名前変更の便利さをお試しください。IDEを強制終了して再起動する必要がありますが、メモリオーバーフローが発生するまで10分間待つことができます。
  • デバッガー Flex Builder 3には条件付きブレークポイントなどの基本的なものはなく、単純な式のみを監視できます。 条件付きブレークポイントはFlex Builder 4にありますが、これは再びVSとFBの間のギャップを強調します。 VS2010はヒストリカルデバッグなどの新世代の機能を提供しますが、FB4は最終的に必要な機能を追加します。これはすべてのIDEバージョン1にあるはずです。
  • MS TestまたはResharperの提供と比較して、統合されたユニットテスターの欠如。




単体テスト/モック



フレックスユニットのテスト能力は十分です。 個人的には、この分野でこのようなツールを選択すると、プロジェクトが行き詰まる可能性があると思います。 ただし、C＃にはmoqライブラリの最良の選択があります。



Visual Studioを使用したC＃での単体テストは、MS TestやResharperなどの統合ツールを使用するとより効率的になります。テストするメソッドまたはクラスを選択するだけで、IDEに結果が表示されます。 テストが失敗した場合は、クリックするだけで、失敗したテストメソッドが開きます。 Flexアプリケーションのごく一部のみをテストする必要がある場合は、別のテストクラスを作成して実行し、テスト条件を定義し、プロジェクトを再コンパイルして実行する必要があります。 Flexユニットでテストが失敗した場合、失敗したテストにフラグを付けて、IDEで手動で見つける必要があります。



はい、Flexでテストできますが、より複雑で時間がかかります。



プロファイリング



Flex Builderのプロファイリングツールは非常に少なく、サードパーティの代替手段はありません。 パフォーマンス統計は、スナップショットからの2回ではなく、アプリケーションの開始から取得されます。



私の知る限り、現在、Silverlightのプロファイラーはありませんが、コードをWPFに変換するのは非常に簡単で、プロファイリングツールを使用した後、私のお気に入りはdotTraceです。 AqtimeとDevPartnerを使用できます。



コード例とドキュメント



C＃でやりたいことの99％がすでに実装されており、90％以上がインターネットで説明されています。 .NETコミュニティは活況を呈しています。 立ち往生している場合は、Googleで5分間過ごすか、MSDNを表示した後、答えが見つかります。 Flex Livedocsには、MSDNよりもはるかに少ない素材のセットもあります。



開発者の経験



一般に、Flash / Flexの開発者は長年にわたって優れたインターフェイスを備えたアプリケーションを作成した経験がありますが、サポートされる信頼できるコードを常に作成するとは限りません。



また、エンタープライズアプリケーションの作成に長年の経験を持つC＃開発者も多数いますが、ユーザーインターフェイスのほとんどは貧弱です。



当然、両方の場合に例外があります。 私はいくつかの優れたFlex開発者に精通しています。



設置ベース



Flash Playerは99％のコンピューターにインストールされます。 軽量アプリケーションを使用してできるだけ多くのユーザーにリーチしたい場合は、Flex / Flashが当然の選択です。



エンタープライズアプリケーションの場合、これは重要な議論ではありません。 一般に、顧客がアプリケーションを購入する場合、Silverlightを要求するのはそれほど複雑でも要求も厳しいことではありません。 多くの企業は現在、標準インストーラーの一部としてSilverlightを含めています。



まとめ



最終的に、私にとって、選択はあなたが達成しようとしているものになります。 A.現時点では、アプリケーションの外観が最も懸念されるものである場合、Flexが最も適している可能性があります。 ただし、複雑なビジネス機能を開発する予定で、お金の種類の使用にも関心がある場合は、Silverlightが最適です。 これは主に、Siverlightプロジェクトで取得したコードの結果、シンプルで高レベル、より保守性が高く、クリーンなためです。 複雑さが増すにつれて、構造化された基本コードを持つことが非常に重要になりますが、これがなければ、エラーとリグレッションの数が増え、追加コストになります。



私は何年もの間C＃で開発していて、楽しんでいます。 フレックスに切り替えたとき、背中の後ろに手を縛られたような気がしました。 環境が実現できるもののほんの一部しか提供しないのはイライラすることです。 これを言って、Flexアプリは私が見た中で最も見栄えの良いアプリであることを認めます。
