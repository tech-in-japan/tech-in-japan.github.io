面接を受けるべきではない... [ネタバレ-あなた自身は面接に行かない]

論文： 過去6か月間に求職者として少なくとも3回の面接に参加していない場合、面接を行うべきではありません 。







さらに、あなたはあなた自身がこれを意識的または無意識的に非常によく知っていますが、企業倫理はあなたの疑問を直接表現することを難しくします。







注目を集めるために、写真を見せて続行します。















多くの場合、インタビュー中に、人々はインタビュープロセス自体をかなり一方的に見ます。 しかし、ほとんどの経験豊富な求職者は、少なくとも一度は面接を行ったか、最近彼らに興味を持っていました（自分自身の生活を楽にするため）。 しかし、テーブルの反対側には、仕事の途中で会議に出席した人（常にではありませんが、非常に頻繁に）がいます。プロジェクトはまだ頭の中で回転しています。 このことから、同じ質問と回答、 申請者と雇用者は異なって見えることがわかります。













上記で述べたように、 過去6か月間に求職者として少なくとも3回の面接に参加していない場合は、面接を行うべきではありません 。 そして最終的に、あなたは必要なものを雇っていない。 ただし、企業倫理では、「すべてが正しく行われ、チームが大きくなったため、間違いなく間に合うようになりました」と報告する必要があります。







有能な人を雇うのを妨げる少なくとも3つの標準的なトピックを見つけました。 それらはすべて一つのことで結ばれています-論文のアドバイスに従えば、あなたはチームの将来のメンバーと話をするときに彼らがあなたをノックダウンするだけの理由を明確に理解できます。 しかし、論文には裏返しがあることを覚えておくことが重要です ： 面接をうまく行っていないことを知っているが、何も改善したくない場合、有能な人を雇いたくない場合、あなたはあなたの散歩に時間を費やすべきではない理由を見つけるでしょうインタビュー用 。







すべてのトピックは、プログラマーの立場（彼ら自身と両当事者のストーリーを含む）のインタビューの編集です。







愚かなトピック＃1-哲学について話す



それで、あなたは会社に雇われるようになりました。 あなたはあなたの分野で優秀なプラウマンであり、経験があります。 あなたは超天才ではなく、ノーベル賞受賞者でもありません。 あなたは普通の資格のある専門家です。







一方、チームリーダー（またはプロジェクトマネージャー、または中級レベルのマネージャー）がいます。 彼のプロジェクトは燃え上がっており、誰が彼と話しに来たのか全く知りません。 彼が覚えている唯一のことは、「まあ、ある種のプログラマー、まあ、ある種.NET Coreで働いていた」ということです。 そしてその後、質問が尋ねられます：「そしてソリッドの原理について教えてください？」。







そのようなトピックに関する質問は非常に美しいようです。 彼は申請者の経験を強調し、緑の新参者と経験豊富な戦士の違いを示さなければなりません。 一つだけあります- この質問に対する答えは全く何も示していません 。 人が一度に5文字すべてを解読した場合、それは彼が最近それについての記事を最近読んだことを意味します。 そして最近-それは数週間以内にあります。 さらに、人が文字の意味をまったく覚えていない場合、それは何も意味しません。 結局のところ、あなたはエンジニアであり、哲学者ではなく、科学者ではなく、エンジニアです。 彼はコードを書き、あなた自身のバグを修正し、高いことについては話さないでしょう。 彼は、なぜオラクルがインターフェイスをそれほど奇妙にしたのかを理解し、プログラムが「正規の200ルールに準拠していないために正規ではない」理由を理解しないでしょう。







ただし、 インタビューで同様の質問を聞いた場合は、目の前にいくつかの選択肢があります。 最も可能性が高いのは、コードを時々見る中間管理者です。 または、少なくともいくつかの問題について対話を開始することを決めた練習エンジニア。 したがって、申請者の正解には、彼自身の質問も含まれます。「SOLID原則について話し始めたので、これらの原則に基づいて認可サービスをどのように完全に行うのですか？あなたと私は、サービスが多くの重要なことに責任を負うべきであるが、ログイン/パスワードを確認し、トークンを発行し、同時にこのトークンを確認します。







上記の質問は、あなたが誰に対処しているのかを素早く（そして高い確率で）理解するので良いです。







  • 技術者はすぐにギアを逆にして、これがすべて哲学であり、会話を始めるために必要であることを認めますが、これはすべて重要ではありません。 そして、会話を続けます。
  • 中間マネージャーは、「 クラスローダーを使用して非ブロッキングキューとIoCコンテナーを使用する必要がある」というスタイルの哲学的な質問に答えます（ポイントが明確になるように意図的に答えを誇張しました）。 これは、プログラマーでない人にとっては非常に賢いことのように思えます。つまり、それは不注意な答えのようにも見え、だれでもさらなる解決策を理解できるでしょう。 ただし、ソフトウェア開発者の役割について話しているのは、テーブルの反対側であなたはすぐに技術専門家ではないことを推測するからです。 したがって、彼が好きになるように答えを選ぶことができます 。


小さな発言は、彼が専門家であると信じている非専門家と対話する方法です



ほとんどの場合、そのような人はプログラマーから成長しますが、5年前に最後に何か深刻なことをしたときです。 ただし、当社のパートナーはまだ自分がトレンドにあると信じているため、技術的な詳細について誇らしげに語っています。 これはプッシュされます。







データベース開発者にOracle / MS SQl自体がスマートであり、個別のチューニングが必要ないことを伝えると、スペシャリストがすぐにあなたを送り返します。 しかし同時に、彼は怒り、そうでないときは人生から多くの議論と例をもたらします。 しかし、マネージャーと話をするとき、「データベース開発には最も重要なルールがあります-すべてのクエリは単純である必要があります。クエリのチューニングと高速化（つまりヒントの追加）は、チームの時間を要し、決定を複雑にします」というフレーズを簡単に伝えることができます排気は最小限になります。」 このようなフレーズは、実際にはデータベースに精通していないが、理論と哲学を知っている人の魂の香油のようになります。 彼はまた、上司への約束で想定していたように、タスクを迅速に完了することが不可能である理由を同僚から説明しました。







哲学者マネジャーに圧力の別のスキームがあります（技術的に読み書きができるようにしようとする、これは非常に重要です）。 多くの場合、彼らは開発者として彼らの激動の若者のテクノロジーと共に生きているので、あなたはまだ時間までにテストされていない最新のソリューションを導入することに反対すると言うことは有益でしょう 。 彼はCのような構文に慣れていて、人々はKotlin / Swift / Scalaでコーディングするつもりです。 そしてここに、私たちの最愛のマネージャーがいつでも伝えることができる、マネージャーが理解できる古いテクノロジーを使用する準備が完全に整った（そしてすべての手で）人が来ます。 さて、ここでどうやって抵抗できますか？







免責事項



上記のヒントは、管理が不十分なマネージャーによる操作にのみ有用であるため、多くの場合、技術的なヒントを使用して登ります（ヘルプは使用しません）。 そのような推奨事項は、次のような話をするときは何の助けにもなりません（そして害を及ぼしません）。







  • 有能なマネージャー。 彼らは単に技術的なフレーズをスキップします。 ソリューションが多くの便利で便利な機能を備えているように、彼らが安定して予測可能に動作することが重要です。 彼らは良くて賢い、彼らはすぐに操作を感じるでしょう。
  • 自分でプログラミングすることを忘れないプロジェクトマネージャー（そしてコードを時々見ているだけではありません）。 これらの人々はすぐにきれいな水に連れて行ってくれます。


哲学的推論に関する結論



哲学について質問しないでください。 あなたは自分自身を置き換えるだけでなく、同時に申請者を心配させ、間違いを犯します。 実際には、従業員がプログラムについてどのように理論​​化するかは、あなたにとって絶対に重要ではありません。







ただし、次のトピックは哲学の話には当てはまりません。







  • 基本構造がどのように、なぜ構造化されるかについてのアイデア。 これらは離散数学についての単なる質問であり、経験豊富な従業員のほとんどが構造の誤用を見てきました（めったにありませんが）。
  • クラスXXXをどのように改善できますか。 これは非常に実用的な質問であり、あなた自身がコードを見て答えることがよくあります。


愚かなトピック＃2-仕事以外の会話



決まり文句-献身



非常に重要な発言：有毒になるためには、これらの質問はすべて、従業員の少なくとも半分に関係するものではありません。 または、別の方法で：従業員自身が「間違った」回答をすることができ、これは絶対に正常と見なされます。







決まり文句は古いですが、使われています：そして、5年後に誰が私たちの会社にいると思いますか？ さらに、ほとんどの場合、この質問は、会社に数年しかいなかった人、または会社の同じ場所に約15年間座っていた人のいずれかによって尋ねられます。







ただし、上記の質問は変化しています。 たとえば、2018年の傾向の1つ： 1年間の自由時間がある場合、給与などが支払われます。 質問：あなたは何をしますか？ 残念ながら、雇用主の少なくとも80％を満足させるこの質問に対する答えは知りません。 この質問は、インタビューを害するだけです。







  • 申請者は、ケースに関する質問を使い果たした彼の前にバカをすぐに見ますが、彼は本当に仕事に戻りたくありません。 雇用主は頭の中でまったく違うものを持っているかもしれませんが、それは彼が外から見る方法です-ばかです。
  • この質問には、真実で誠実な答えは決してありません 。 誰もが他の人の目に、トラブルに巻き込まれない人として現れたいと思っているからこそ、私たちの時代のマザー・テレサ。 これは慣例ではありませんが、私は本当に思われたいです。 したがって、申請者は可能な限りほとんどすべてを誇張する可能性が非常に高いです。
  • この質問にはすでに、誠実で検証済みの有名な答えがあります。 40歳のプログラマーは、彼の人生で約20年の仕事、つまり約20か月の休暇、2,000日間の休暇を過ごしました。 さらに、まだ多くの夜、休日などがあります。 私たちは重要な結果を得る - 理論的な年に人ができることはすべて、彼は人生ですでにやったことだ。 さらに、概要には、最もおいしい詳細、すべてのオープンソースプロジェクト、すべての成果が記載されています。 すべてがすでにそこにあります。
  • そして、質問が有害である主な理由： 私たちは従業員についての真実が好きではありません、私たちは彼が嘘をついて空想するように頼みます。


自由時間の有用性について話す



インタビューで自分自身を混同する方法の別の例。 これらは「自由時間をどのように過ごしますか」というスタイルの質問です。 繰り返しますが、このフレーズは非常に賢明に見えますが、実際には有害です。







この質問のマイナスは何ですか？ 結局のところ、私たちはその人にどれほど頭が良くて良いか、 新しい知識をどれだけ簡単に習得できるかを私たちに尋ねます（ イタリック体で強調されているのは詐欺とcです）。 ただし、実際には：







  • 人が10-12時間真剣に働く場合、彼はしばしばその仕事自体で十分に勉強します。 彼は同僚のプレゼンテーションを見たり、プロフィール記事を読んだり、仕事に直接関係する多くのことを分析したりしています。 その結果、 人が真にプロフェッショナルである場合、彼は自由時間について正直かつ美しく答えることができなくなります 。
  • 私たちの求職者が細いマニピュレーターである場合、類似の質問をすると、彼はすでに実際に彼を連れて行ったことをすぐに理解できます。 雇用主は、対談をしないことを既に決めていた場合、職業について質問しないでしょうか？ そのため、 有能な交渉担当者が精神で何かに答えます 。「私たちはWindows上の.Netテクノロジーについて話しました。自宅でLinux上の.Net Coreの記事を読んでいます（まあ、MacでもiOS / Androidで可能です。しかし 、これは偶然ではありませんでした 。 ）これは業界の未来だと思います。したがって、新しいソリューションにすぐに取り組むため、新しいソリューションにはもっと注意を払おうとしています。」 私はそのような答えを繰り返し聞いてきましたが、そのたびに申請者は理論のために技術的な説明をすぐに残しました。 ただし、回答テンプレートは独自の方法でシンプルで独創的です。 結局のところ、新しい従業員は、対談者が扱っているテクノロジーに向かって通路を作ります。 したがって、彼はいくつかの真実の（よく知られている）事実を与えます。 そして最後に、雇用主が確認するのは難しいので、彼はトピックを解体して率直に嘘をつくことができる場所に移します（彼自身はほとんど勉強していません）。
  • そして最も重要な結果：応募者がどのように答えても、これはすべて重要ではありません。 あなたは自分で会社に連絡して、自由時間に関する答えが仕事の質と決して相関しないことを確認することができます。


仕事以外の会話の結論



そのような質問をしないでください。 すべての企業は多かれ少なかれ同じであり、候補者に忠誠を認めるよう求めることは意味がありません。 確認するのがさらに難しいことを尋ねるのは意味がありません-嘘つきになり、正直な働き者を追放するだけです。







チームメンバーでさえ間違って回答するような質問はしないでください。

決まり文句-これはここでは発生しませんが、話しましょう



有毒な質問の例：「大量のデータはありませんが、100 Pbのデータをどのように保存し、すぐにアクセスできるようにしますか？」







このような質問は、インタビューした同僚と潜在的な雇用主の両方から聞いた。 すべての場合において、私は自分自身に質問をしました。「では、なぜこれを尋ねるのですか。私たちはそのような仕事を持っていません。私たちはこの分野の専門家ではありません。はい、そして応募者は他のスキルを持って来ました。 」







この質問は哲学的な質問に非常に似ていますが、 それは非常に具体的で現実的で、最も重要なことに見えます-それは会社の規模について語っています。







また、次の理由により、非常に問題が多くなります。







  • その人の応答を確認することはできません。 適切なソフトウェアの哲学に似ています-その人は答えますが、これは彼が本当にできることに関するデータを私たちに与えません 。
  • タスクは非常に抽象的に設定されています。 オブジェクトのサイズも記録頻度も言われていませんが、システムの信頼性に関する要件さえありません。 ここで、有能なスペシャリストは、「私は最も効率的な分散ストレージシステムの1つを取ります」のような答えを単に却下することができます。 または、彼は彼の無知を恐れて、それゆえに、混乱した理解できない答えを与えるでしょう。
  • 多くの場合、あなたの同僚でさえ、この質問に対して明確で正しい答えを出すことができません（これは悪いことではなく、これは正常です;検眼医は、どの薬がまれな形の癌を治療するかを単に知らないかもしれません）。
  • 同じ結果が得られます。この質問は、候補者の評価の誤りを増やすだけです。 この質問により、人がどれだけチームを助けるかではなく、人が他の部門（または別の会社）の問題をどれだけうまく解決できるかがわかります。
