パーサーの紹介
パーサーは、一連の要素を取り、成功した場合、処理結果と残りのシーケンスを返す関数です。

ただし、失敗した結果には、「失敗」と「エラー」の2つの形式があります。

比Fig的に言えば、パーサーはシーケンスの一部を最初から噛み、噛まれたものを特定のタイプのオブジェクトに変換します。



最も単純なパーサーは、シーケンスの最初の要素が所定の要素と等しいことを確認し、この要素を成功した結果として返すパーサーです。 残りとして、この要素のないシーケンスがあります。



要素からそのようなパーサーを作成するには、要素を受け入れるacceptメソッドを使用します。 このメソッドは暗黙的として定義されており、コンパイラがパーサーに会うと予想される要素に遭遇した場合、このメソッドのアプリケーションを要素に追加します。

文字のシーケンスを解析するとしましょう： 

 def elementParser: Parser[Char] = 'c' //  def elementParser: Parser[Char] = accept('c') //


したがって、パーサーを組み合わせるときに、パーサーがあるべき場所に要素がある場合、そのような基本的なパーサーがそこにあることを知っています。



一般に、これは明示的に定義された唯一のパーサーです。

他のすべてのパーサーは、他のパーサーと結果変換の組み合わせによって取得されます。



パーサーを組み合わせる


善のためにある
実際、scalaには演算子はありませんが、これを知っていれば、おそらくパーサーについて話す必要はありません。


二項演算子は「〜」です。 「and」の原則に従って2つのパーサーを組み合わせます。 最初のパーサーが成功した場合にのみ成功し、その後、最初のパーサーが解析した残りの2番目のパーサーが成功します。

比ur的に言えば、最初のパーサーが最初に彼に合ったものを噛み、次に2番目のパーサーが残り物を食べます。

両方のパーサーの結果を含むコンテナーが結果として返されます。 

 parser1 ~ parser2


この方法で、パーサーの任意のセットを組み合わせることができます。

このコンビネータには、「〜>」と「<〜」の2つの兄弟があります。 これらは同じように機能しますが、組み合わせたパーサーの1つだけの結果を返します。



二項演算子「|」。 「または」に基づく関連付け。 少なくとも1つの結果が初期入力で成功した場合は成功です。 最初のパーサーが「失敗」を返した場合（エラーではない）、同じ入力を2番目のパーサーにフィードしようとします。



担当者 シーケンス。 パーサー「myParser」がある場合、「rep（myParser）」を使用して作成されたパーサーは、入力から最初の失敗したアプリケーションへの「myParser」の助けを借りて「バイト」します。 すべての「バイト」の結果は、コレクションに結合されます。

空でない結果コレクション（rep1）および区切りシーケンス（repsep）には、関連する変換があります



結果を変換する


解析の結果に対して変換を実行する必要がある場合は、^^^や^^などの演算子が役立ちます

^^^は結果を指定された定数に変更し、^^は指定された関数を使用して結果を変換します。





パーサーの組み合わせ（およびw3c仕様のリテラシー）により、ためらうことなくパーサーを作成できます。

実際、仕様を2回書き換えています。 唯一の大きな違いは、再帰的な定義を「循環」定義（repおよびrepsep）に置き換えたことです。



例：



仕様： 

  [15] PrimaryExpr :: = VariableReference	
                                       |  '（' Expr '）'	
                                       | リテラル	
                                       | 数	
                                       | 関数呼び出し


パーサー ： 

  def primaryExpr: Parser[PrimaryExpr] = variableReference | `(` ~> expr <~ `)` ^^ { GroupedExpr } | functionCall | number | literal


唯一の条件は、最も「厳密な」パーサーが「|」を介してユニオンに入ることを保証する必要があることです。 残りの前に。 この例では、関数の名前の解析に成功しただけで、functionCallが成功した場合にリテラルが明らかに成功します。したがって、リテラルを先に配置すると、単にfunctionCallに到達しません。

パーサーのセット全体が150行に収まり、オブジェクトモデルの定義よりも大幅に短くなります。

メカニズムは非常に簡単です。有効なメソッド名である前の行（スペースなし）に遭遇すると、コンパイラは単純な変換を実行します。 

 t"strinf $x interpolation ${ obj.name.toString } " StringContext("strinf ", " interpolation ", " ").t(x, { obj.name.toString })


文字列は、変数と式の出現によって断片に分割され、StringContextファクトリメソッドに渡されます。 文字列の前の名前はメソッドの名前として使用され、すべての変数と式はパラメーターとしてこのメ​​ソッドに渡されます。

これが「s」や「f」などのメソッドで終わる場合、StringContextにないメソッドについて、コンパイラは暗黙的なクラス、つまり目的のメソッドを含むStringContextのラッパーを探します。 このような検索はscalaの一般的なメカニズムであり、文字列補間には直接適用されません。
