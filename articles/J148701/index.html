プレースホルダー-データ置換


原則として、すべてがシンプルです。データはリクエストに直接入力されるべきではなく、何らかの代表的なワイルドカード表現を介して取得される必要があります。

リクエストは、たとえば、 
SELECT * FROM table WHERE id > ? LIMIT ?
データは個別に追加および処理されます。

しかし、「通常のエスケープ」よりも優れているのは何ですか？ そして皆に：

  • まず、「エスケープ」はセキュリティとはまったく関係ありません。 想像してみて！
  • 第二に、コードが短くなります。 mysql_real_escape_string（）もintval（）もありません-すべての処理は内部に隠されています。
  • 第三に、コードが簡単になっています。 リクエストの異なる部分をフォーマットするための異なるルールを覚える必要はありません
  • 第4に、プレースホルダーを使用すると（プレースホルダーが正しく処理される場合）、データを介したインジェクションが保証されます。 「これは明らかです！」-あなたは言う-「別のポイントをフェンスすることは価値がありましたか？」 同じエスケープ保護は保証しません。 さらに、フォーマット規則を1つに減らす-ここで何かを混乱させるのは難しいだろう
  • 5番目、そして最も重要なことは、データを必要な場所で正確に処理することです！ これは多くの人が理解していない非常に重要なポイントです。 古典的な「楽しい」教科書では、SQLのデータ形式はコード全体に散らばっています。 また、PHPの古いバージョンでは、コード実行の開始前でさえ開始されていました。これは、まったくゲートには入りません！ この状況では、必要に応じて、わずかな利点なしに、一部のデータが2回フォーマットされ、他のデータが半分だけ、さらには他のデータがまったくまたはまったくフォーマットされないという事実につながります。 同時に、SQL用にフォーマットされたデータがHTMLまたはCookieに突然表示されますが、これもユーザーや開発者に喜びを与えません。


したがって、最適なオプションは、リクエストを実行する直前にデータをフォーマットすることです-このようにして、データが正しくフォーマットされていることを常に確認します、これは一度だけ行われ、フォーマットされたデータは意図したとおりになります-データベースにどこにも行かない

そして、それはまさにそのような-タイムリーで安全で正しい-データ処理の目標であり、プレースホルダーが提供することで、セキュリティを保証すると同時にコードを簡素化します。



使用例：


マニュアルから： 

 $stmt = $dbh->prepare("SELECT * FROM REGISTRY where name LIKE ?"); $stmt->execute(array("%$_GET[name]%")); $data = $stmt->fetchAll();


努力すべき例： 

 $ban = $db->getRow("SELECT 1 FROM ban WHERE ip = inet_aton(s:)", $ip);


ご覧のとおり、難しいことではありません。また、巧みに使用すれば、手でリクエストを行うよりもはるかに短くなります。 あなたはまだ昔ながらの方法で書きたいですか？



重要な注意：もちろん、データソースやその他の条件に関係なく、プレースホルダーを介したデータ置換は常に実行する必要があります。



識別子とキーワード-ホワイトリスト


注射に関する記事の大部分は、この点を完全に見逃しています。 しかし現実には、クエリではデータだけでなく、他の要素（識別子（フィールドとテーブルの名前）、さらには構文要素、キーワード）を置換する必要があります。 DESCやANDのように重要ではない場合でも、そのような置換のセキュリティ要件は、それほど厳しくはないはずです！



かなり平凡なケースを調べてみましょう。

製品のデータベースがあり、それはHTMLテーブルの形式でユーザーに表示されます。 ユーザーは、このテーブルをいずれかのフィールドで任意の方向に並べ替えることができます。

つまり、少なくともユーザー側からは、列名と並べ替え方向を取得します。

それらを要求に直接代入することは、保証された注入です。 通常のフォーマット方法はここでは役に立ちません。 識別子もキーワードもない準備された式は、エラーメッセージ以外は何も導きません。

唯一の解決策はホワイトリストです。

もちろん、これはニュートンの二項式ではなく、多くの開発者は外出先でこのパラダイムを簡単に実装します。最初にクエリのフィールド名を置き換える必要がありました。 ただし、この規則のない注入保護に関する記事は不完全であり、保護自体は漏れやすいものです。



このメソッドの本質は、 すべての可能な選択肢をコードにしっかりと記述し、ユーザーの入力に基づいてそれらのみをリクエストに含める必要があることです。



適用例： 


 $order = isset($_GET['order']) ? $_GET['order'] : ''; //     $sort = isset($_GET['sort']) ? $_GET['sort'] : ''; $allowed = array("name", "price", "qty"); //  $key = array_search($sort,$allowed); //      $orderby = $allowed[$key]; //  (,     - ) . $order = ($order == 'DESC') ? 'DESC' : 'ASC'; //    $query = "SELECT * FROM `table` ORDER BY $orderby $order"; // 100%


以前は、プレースホルダで識別子に十分であると想定していました。 しかし、時間が経つにつれて、この方法の欠点が理解されるようになりました。

  • まず、無効なフィールド名の場合、リクエストによりエラーが発生します。 そして間違いは常に悪い
  • 第二に、はるかに重要です：予備フィルタリングなしでフィールド名を自動的に置換する場合、自動的にエスケープするだけで、異なる種類のインジェクションを取得できます-結局、ユーザーは変更しないはずのフィールド名を入力できます！ たとえば、$ _POST配列に基づいてSQLクエリを自動的に生成する場合（フィールド名を正しくフォーマットしている間に！）、ハッカーは、登録時に値「1」のadminフィールドをフォームに追加し、管理者になります


だから今、私は両方の方法を使用します：

最初に、ホワイトリストから識別子を取得します。

そして、プレースホルダーを介して追加します-手動の書式設定を処理しないようにするためです。 そして均一性のために。 この場合、最後の行は次のようになります 

 $query = "SELECT * FROM `table` ORDER BY n: $order";


基本的に、この情報は完全に安全なクエリを書き始めるのに十分です。 しかし、いつものように、実際の生活にはさまざまなニュアンスがあり、プレースホルダーのメカニズムをより詳細に理解したいと思います。



プレースホルダーを使用する



最初に、プレースホルダーを実装するための2つのオプション（サーバーとクライアント）があることを理解する必要があります。

  • 最初のケースでは、リクエストはプレースホルダーとともにサーバーに送信され、データはそれとは別に送信されます。 英語では、「ネイティブ」準備済みステートメント-「ネイティブ」準備済みステートメントと呼ばれます。つまり、プレースホルダーの処理は、サーバー上のDBMS自体によって実行されます。 簡潔にするために、ネームサーバーサーバーのプレースホルダーを使用します。
  • 2番目の場合、データはフォーマットされ、クライアント上の直接のプレースホルダーの代わりにクエリ文字列に置換され、従来のSQLクエリを形成し、通常の方法でデータベースに送信されます。
それぞれの方法には長所と短所があり、以下で検討します。

また、PDOはこれらの2つの椅子に同時に座っていることを覚えておく必要があります。デフォルトでは、2番目のオプションに従って動作し、最初のオプションのみをエミュレートします。 この機能を無効にするには、PDOにリクエストとは別にサーバーにデータを送信させる必要があります。

サーバーのプレースホルダー


まず始めに、なぜ注射が可能なのでしょうか？

実際、SQLクエリはプログラムです。 完全なプログラム-演算子、変数、および文字列リテラル。 問題は、外出先でこのプログラムを動的に収集することです。 一度だけ記述され、着信データに基づいて変化しないPHPスクリプトとは異なり、SQLクエリは毎回動的に再生成されます。 そして、その結果、誤ってフォーマットされたデータはリクエストを台無しにしたり、場合によっては変更したりして、当社が提供していない演算子を置き換えます。 実際、これはまさに注射の本質です。



プレースホルダーのサーバー処理は何を提供しますか？

非常に簡単なこと：... 変数のようなものをプログラムに導入します ！ はい、プレースホルダーは通常の変数であり、SQLの「スクリプト」にハードコーディングされており、データによっては変化しません。 また、データ自体はリクエストとは別にサーバーに送られ、決して交差することはありません。 リクエストが解釈された後にのみ、データは実行段階で直接使用されます。

実際には、次のようになります：prepare（）を呼び出すと、リクエストはこの形式でサーバーに直接送られます-プレースホルダー/変数を使用して、サーバーはそれを解析し、「すべてが正常で、データを受信する準備ができている」ことを通知します（まあ、またはエラーを報告します）。 そして、execute（）が実行されると、データはすでにサーバーに送られ（テキスト形式ではなく、バイナリパッケージで、クエリ結果が返されるものと構造が似ています）、すでに実行に直接関与しています。



理論的には、非常に魅力的です。

ただし、実際には、残念ながら、PHPでMysqlを操作するための既存のライブラリでは、準備された式を操作する実装はまだ理想からはほど遠いものです。

たとえば、次の事実を引用するだけで十分です。

  • 準備された式を使用したクエリの結果文字列を配列（mysql_fetch_array（）のアナログ）に入れるなどのありふれた機能は、バージョン5.3（！）でのみmysqliに追加されました。 このバージョンより前は、ライブラリは実際には不安定で、 地獄のような倒錯がありませんでした。
  • 同じことがPDOでの接続のエンコーディングの設定にも適用されます-これは同じ5.3でのみ可能になりました。 以前のすべてのバージョンでは、PDOで接続のエンコードを指定することは単に不可能でした（公平性を考慮して、互換性モードがオフになっている場合、これは大きな問題ではありませんが-穴！）。
  • mysqliは、フィールドがデータベースにできるだけ収まるように、できるだけ多くのメモリを予約しようとする場合があります。中程度のテキストには注意してください。
これらの事実は、両方のライブラリがまだかなり未加工であり、他の必要な機能がないことを期待できることを示しています。



既存のライブラリの主な欠点をリストします。

  • いくつかの有用なプレースホルダーの欠如
  • デバッグ目的で従来のSQLクエリを取得できない

より詳細に分析します



冗長性。


たとえば、クエリ結果のすべての行を2次元配列に入れるなどの一般的な操作を考えてみましょう。 mysqliにはそのような関数はまだありません。 または、たとえば、変数をプレースホルダーにバインドできるのは、別個の関数のみです。

その結果、1つのリクエストのデータを配列に取得するには、少なくとも9行（！）のコードが必要です。 

 $data = array(); $query = "SELECT Name, Population, Continent FROM Country WHERE Continent=? ORDER BY Name LIMIT 1"; $stmt->prepare($query); $stmt->bind_param("s", $continent); $stmt->execute(); $result = $stmt->get_result(); while ($row = $result->fetch_array(MYSQLI_NUM)) { $data[] = $row; }


さらに、このコードのほとんどはセマンティックロードを持たず、実行されたすべてのクエリでまったく同じであり、何百ものスクリプトで何度も繰り返されます。

データを取得するには2行だけで十分であるという事実にもかかわらず： 

 $query = "SELECT Name, Population, Continent FROM Country WHERE Continent=? ORDER BY Name LIMIT 1"; $data = $db->getAll($query,$continent);


PDOの方がこれに少し優れています-配列を実行（）に渡すことができ、fetchAll（）メソッドがあるかどうかがわかりました。 しかし、最も単純な操作のために、不必要な単調なコードをたくさん書く必要があります。

特に、記事のトピックであるバインディングに戻ります。 ここに変数$ _GET ['id'];があります。 プレースホルダーに添付します。 すばらしい、execute（）で直接行うことができますが、配列にすることによってのみ可能です。 なんで？ ドライバー自体が私のためにこれを行うことはできませんか？

 $conts = array('Europe','Africa','Asia','North America'); $query = "SELECT * FROM Country WHERE Continent IN(?) ORDER BY Name LIMIT 1"; $data = $db->getAll($query,$conts);


数十行の不自然なコードの代わりに。

性能


通常、サーバー側で準備された式の謝罪者は、リクエストの解析が一度だけ行われるという事実を強調しています。

残念ながら、Webアプリケーションの場合、これは機能しません。 prepare（）を実行したスクリプトのコピーは、このリクエストに対してexecute（）を正確に1回実行し、安全に終了します。 新しいコピーが再度準備されます。 その結果、保存したい場所により多くの作業が発生します。

最近の記事に対するコメントで、別の潜在的な速度向上-クエリプランのキャッシュ-が指摘されました。 実際、各クエリを準備しても、データベースは異なるデータに対して同じクエリをキャッシュできます。 解析せずに、単純な文字列比較で実行計画を取得します！

残念ながら、私はMySQLの内部構造において、メカニズム自体の存在とその実用的な有効性の両方を確認または否定するほど強くはありません。

同時に、私が知る限り、実際の負荷が大きいと、サーバーが準備した式は標準のSQLクエリの速度を失います。

一般に、DBMSとそれを操作するためのドライバーが提供するツールは、広告に見られるほど良くないことがわかりました。 そして疑問が生じます-自分でプレースホルダーとの作業を実装できますか？ そして答えが生まれます-私たちはできます！



プレースホルダーの独立した実装



実際、プレースホルダーに煩わされることなく、既存のライブラリの使いやすさを改善することを妨げるものは誰もいません。 たとえば、独自のPDOプレースホルダー（または1年前にHabréで公開されたライブラリで行われたmysqli）を使用して、不足している機能を実装するPDOのラッパーを作成します。



ただし、自家製のプレースホルダーを検討する理由はいくつかあります。

まず、すでに見たように、標準ライブラリで利用可能なプレースホルダーのセットでは明らかに十分ではありません。

第二に、何らかの理由でサーバーのプレースホルダーが適切でない場合があります。

第三に、プレースホルダーの独立した処理の例では、SQLクエリを正しくフォーマットすることのニュアンスを考慮します。



SQLクエリのさまざまな要素の書式設定の原則


自家製のプレースホルダーを実装するのは簡単です。 プリミティブパーサーは既にPHPに組み込まれています。 必要なのは、リクエストのさまざまな要素を区別する方法を学ぶことだけです。 しかし、これは非常に重要なポイントであり、より詳細に説明する必要があります。

フォーマット規則は要素のタイプに依存するため、まず、どの特定の要求要素に置換するのかを明確に理解する必要があります。 次に、この情報を何らかの方法でプレースホルダープロセッサに伝える必要があります。



まず、クエリを構成する要素を決定しますか？

たとえば、次のようなSQLを考えます。 

 INSERT INTO `db`.`table` as `t1` VALUES('string',1,1.5,NOW());
要素の3つの主要なグループを区別できます。

  • SQL言語要素自体-演算子、組み込み関数、変数など
  • 識別子（データベース、テーブル、およびフィールドの名前）
  • いくつかの異なるタイプのリテラル（クエリに直接代入されるデータ）。
