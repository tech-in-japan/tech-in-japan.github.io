単体テスト：何を、どのように、いつテストするのか？

ソフトウェアコードのテストは骨の折れる複雑なプロセスです。 その中の作業の大部分は、単体テストによって行われます。 それらが「緑色になる」まで、さらにテストする意味はありません。



ユニットテストを正しく書く方法は？ 100％のカバレッジを追いかける価値はありますか？ エンジニアは実際にどのような困難に直面していますか？ Marc PhilippとVsevolod Brekelovは彼らの経験を共有しています。









Marc Philippは、Javaテスター向けのツールであるJUnit 5フレームワークの主要な開発者の1人です。 現在、彼はドイツの会社LogMeInでクラウドSaaSソリューションのエンジニアとして働いています。



Vsevolod Brekelov-Grid DynamicsのシニアQAエンジニアは、5年以上テストを行っており、テストの自動化をゼロから構築した経験があります。



-単体テストに関する記事では、通常、テスト方法と計算機のクラスが例として挙げられています。 そのような例は、実際のタスクの複雑さを示すことができますか？ フル機能のプログラムのテスターは何に直面しますか？



Marc Philipp：実際、計算機を使用した例では、実際のタスクの複雑さを示すことは不可能です。 記事で選択されているため、読者は複雑なコードを解析することなく単体テストのアプローチを理解することに集中できます。 これらの例は非常に単純ですが、ユニットテストの基本的な考え方と原則をよく示しています。 実際には、単体テストが実行されることを考慮して、テスト済みのコードを最初に記述する必要があります。 これを確実にする1つの方法は、コードを記述する前に、またはほぼ同時にテストを記述することです。 テストに適したコードがある場合、単体テストの作成は電卓の場合ほど複雑ではありません。



Vsevolod Brekelov：実際のタスクの複雑さは実際の問題でしか理解できないと思います。 真剣に、非常に重要な例が詳細に検討されている良い記事があります。 現実に近づくのに役立つと思います。



たとえば、「Javaの単体テスト」というリクエストでは、 Habréに関する記事をすばやく見つけることができます 。 それはずっと前に公開されましたが、その関連性を失っていません。



作業の機能に関しては、次のテスターのグループを選び出します（誰かを怒らせないでください）。



  • バックエンド-システム、統合、コンポーネント、単体テストを作成できます（なぜですか？）。
  • フロントエンド-e2eテストとコンポーネントおよびユニットテストの両方を作成できます。
  • DB-彼らはデータ/データベース自体のテストに従事しています。
  • パフォーマンス-ここでは明らかなようです。
  • インフラストラクチャ-「near-devobsnye」問題の詳細。
  • モバイルテスト（iOS、Androind、IoT）-今では、このようなエンジニアを分離することは非常にファッショナブルになりましたが、私の意見では、すべて同じバックエンド/フロントエンドです。


これらの人々は通常、ソフトウェア自体をテストするだけでなく、要件とプロセス自体もテストします。 確かに、彼らはこれに最も頻繁に正式にアプローチしますが、これは私の意見では間違っています。



プロセスに注意を喚起したいと思います。 私の練習では、バグと、不要なものを実装するための時間の主な無駄がそこから成長するため、各テスターは開発プロセスの構築に精通している必要があると考えています。



-各テストで1つのことを確認する必要があります。 実際には、この条件をどの程度完全に満たすことができますか？ どのフレームワークを使用しますか？



Marc Philipp：単体テストを作成するとき、通常、テスト対象の問題領域の等価クラスからの入力のサンプルが1つ取得されます。 もちろん、最初にこれらの同じ等価クラスを定義する必要があります。 各テストでは、テストに関連するプロパティに対してのみアサーションを追加します。 同じアサーションを新しい各テストにコピーして貼り付けないでください。 ユニットのパフォーマンスに影響を与える依存関係がある場合は、スタブまたはモブを使用してテストの独立性を維持することを検討してください。



JUnit 5の単体テストの多くは、モックフレームワーク（この場合はMockito）によって作成されたモックを使用します。 上で言ったように、それらは分離されたコードをテストするのに非常に便利です。 この場合の主なタスクは、mokが実際のコードと同様に動作することを確認することです。 そうしないと、テストは無意味になります。



Vsevolod Brekelov：はい、意見があります。1つのユニットテスト-1つのアサーションです。 実際には、これはめったに見ませんでした。 これがチームの哲学だと思います。 複数のアサーションには場所があります。



コンポーネントテストではなくユニットテストを実施する場合、すべての依存関係（moki、スタブ-すべてはあなたの手にあります）を分離します。 私の意見に問題はありません。 もしそうなら、StackOverflowは間違いなく役立ちます。



Java / JavaScript（Angular）で記述しているため、通常の一般的なツールを使用します。

Javaで-Mockito / EasyMock。 コンポーネントテストの場合、レスポンシブモックを作成することもお勧めです。 皆さんにアドバイスします。



JavaScript-ngMock。 ところで、コンポーネントテストの非常にクールなトピックはAngularPlaygroundです。
