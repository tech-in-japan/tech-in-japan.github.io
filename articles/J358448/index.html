Gitlab CIを使用してAndroidおよびiOS用のUnityプロジェクトを自動的にビルドする

この記事では、macOSを使用したネイティブビルドでGitlabを介してAndroidおよびiOSでUnityプロジェクトをビルドする方法について説明します。







私は小さなゲーム開発会社で働いていますが、次の問題が原因でアセンブリを自動化するタスクが現れました。







  • 5つの分散チームは、世界中のどこからでもプロジェクトを収集する必要があります
  • 異なるユニットバージョンをサポートする必要があります
  • アセンブラーは各チームから少なくとも週に5つのアセンブリを提供する必要があります
  • 証明書は開発者ではなく中央に保存する必要があります
  • 収集されたビルドは、世界中のどこからでも参照によりアクセス可能でなければなりません
  • プロジェクトで必要なライブラリの存在を確認する必要があります（広告SDKとコード、ローカリゼーション、保存）
  • コマンドのアセンブリ構成は1か所で行う必要があります


これらの問題を解決するために、Unity Cloud Build、TeamCity、Jenkins、Gitlab CI、Bitbucket Pipelinesなどの既製のソリューションが既に作成されています。







最初のものは、Unityプロジェクトのアセンブリ用に準備されていますが、証明書を使用した作業の自動化は許可されておらず、プロジェクトごとに手動で入力する必要があります。 TeamCityとJenkinsでは、管理領域でプロジェクトを設定し（これにより開発者の構成が少し複雑になります）、別のサーバーに追加のソフトウェアをインストールし、そのサポートを行う必要があります。 その結果、GitlabとBitbucketの2つのオプションを実装するのが最も簡単で最速でした。

問題が解決した時点では、Bitbucket Pipelinesはまだ発表されていなかったため、Gitlabを使用することが決定されました。







このアプローチを実装するために、次の手順が実行されました。







  • プロジェクトのセットアップ
  • ランナー設定
  • ビルドスクリプトの作成


1.プロジェクトのセットアップ



コレクターで収集されたプロジェクトは、Gitlabに保存されます。 サービスの無料版は、リポジトリ自体とその数を制限しません。

各プロジェクトには、ポピーで実行されるランナー（gitlabサーバーからコマンドを実行するサービス）が含まれています。

コレクターの構成は、.gitlab-ci.ymlファイルとしてプロジェクトのルートにあります。 アプリケーションID、必要な署名ID（AndroidのキーストアとiOSのアカウント名）、必要なUnityバージョン、ブランチ、起動モード：手動または自動、およびアセンブリを開始するコマンド（必要に応じて、gitlabはより多くのパラメーター、 ドキュメントをサポート） について説明します 。







サンプル.gitlab-ci.yml構成ファイル 
variables: BUNDLE: com.banana4apps.evolution SIGNING: banana4apps UNITY_VERSION: 2017.1 build:android: script: - buildAndroid.sh $BUNDLE $SIGNING $UNITY_VERSION only: - releaseAndroid when: manual build:ios: script: - buildIOS.sh $BUNDLE $SIGNING $UNITY_VERSION only: - releaseIOS when: manual


2.ランナーのセットアップ



Gitlab CIは、共有ランナーと独自のランナーで動作します （ ドキュメント ）。 無料版では、共有ランナーの使用時間に制限がありますが、独自のランナーを無制限に使用できます。 共有ランナーはLinux上で実行されるため、iOSアプリケーションをそれらでアセンブルすることはできません（ただし、Unityは成功します。ハブに関する記事がありました）。 このため、私は自分のポピーでランナーを上げる必要がありました。 上記の例では、ランナーはbuildAndroid.shまたはbuildIOS.shスクリプト（ブランチによって異なる）を実行します。これは、準備手順、Unityの起動、ビルド結果の通知を記述しています。

ランナーの設定プロセスはドキュメントにgitlab-runner install



説明されており、 gitlab-runner install



およびgitlab-runner start



ます。

その後、必要なUnityバージョンがポピーにインストールされます。







3.ビルドスクリプトの作成



プラットフォームごとに、ビルドプロセスの違いにより、独自のスクリプトを作成する必要がありました。 ただし、アルゴリズムは同じです。







  • プロジェクトIDの正確性、目的の署名IDの証明書の可用性を確認します
  • SDK、Javaへのパスを定義する
  • アセンブリを開始するメソッドを使用してC＃プロジェクトにクラスを作成します
  • Unityの必要なバージョンの可用性を確認し、アセンブリを実行します。 そうでない場合は、デフォルトバージョンでビルドしてみてください
  • apkまたはXcodeプロジェクトの可用性を確認し、存在しない場合はSlackでエラーを通知します
  • iOSの場合：Xcodeプロジェクトの構築
  • アーティファクト（apk、ipa）をサーバー（Amazon S3など）にアップロードします
  • Slackでビルドが成功したことを通知し、アーティファクトをダウンロードするためのリンクを送信します


Unityプロジェクトをビルドする特性は、Unityをバッチモードで使用すると、プロジェクトで使用可能なクラスの静的メソッドのみを実行できることです。 したがって、アセンブリスクリプトは、アセンブリを開始するためのメソッドを含むクラスをプロジェクトに「スロー」します。







CustomBuild.cs 
 public class CustomBuild { static string outputProjectsFolder = Environment.GetEnvironmentVariable("OutputDirectory"); static string xcodeProjectsFolder = Environment.GetEnvironmentVariable("XcodeDirectory"); static void BuildAndroid() { BuildTarget target = BuildTarget.Android; EditorUserBuildSettings.SwitchActiveBuildTarget(target); PlayerSettings.applicationIdentifier = Environment.GetEnvironmentVariable("AppBundle"); PlayerSettings.Android.keystoreName = Environment.GetEnvironmentVariable("KeystoreName"); PlayerSettings.Android.keystorePass = Environment.GetEnvironmentVariable("KeystorePassword"); PlayerSettings.Android.keyaliasName = Environment.GetEnvironmentVariable("KeyAlias"); PlayerSettings.Android.keyaliasPass = Environment.GetEnvironmentVariable("KeyPassword"); BuildPipeline.BuildPlayer(GetScenes(), string.Format("{0}/{1}.apk" , outputProjectsFolder, PlayerSettings.applicationIdentifier), target, options); } static void BuildIOS() { BuildTarget target = BuildTarget.iOS; EditorUserBuildSettings.SwitchActiveBuildTarget(target); PlayerSettings.applicationIdentifier = Environment.GetEnvironmentVariable("AppBundle"); PlayerSettings.iOS.appleDeveloperTeamID = Environment.GetEnvironmentVariable("GymTeamId"); BuildPipeline.BuildPlayer(GetScenes(), xcodeProjectsFolder, target, options); } //        static string[] GetScenes() { var projectScenes = EditorBuildSettings.scenes; List<string> scenesToBuild = new List<string>(); for (int i = 0; i < projectScenes.Length; i++) { if (projectScenes[i].enabled) { scenesToBuild.Add(projectScenes[i].path); } } return scenesToBuild.ToArray(); } }


Environment.GetEnvironmentVariableメソッドは、bashスクリプトで以前に指定された環境変数の値を取得します。

iOS用のサンプルビルドスクリプト

プロジェクトのアセンブリは、UnityからのXcodeプロジェクトの形成とXcodeプロジェクトのアセンブリという2つのステップで実行されます。 開発者はXcodeプロジェクトに直接影響を与えることはできません。これは制限をもたらします。プロジェクト設定、アセンブリ情報を直接変更することはできません。

また、iOSのビルド機能では、テストデバイスをアプリケーションのプロビジョニングプロファイルに登録する必要があります。 Xcodeプロジェクトをビルドするには、ビルドする前にAppleの開発者コンソールで証明書、プロビジョニングプロファイル、およびアプリケーションIDを作成する必要があります。

Fastlaneは、このプロセスを自動化するために使用されます。 このツールは、証明書とプロファイルを作成および同期し、ビルドとメタ情報をiTunes Connectにアップロードできるようにします。







XcodeにアクセスせずにUnityプロジェクトを構築する場合、微妙な違いがあります。







  • Unityでは、プロジェクトをビルドする前に、開発者のコ​​ンソールにあるTeamIdを指定する必要があります-これはPlayerSettings.iOS.appleDeveloperTeamIDを介して行われます
  • 後処理プロジェクトスクリプトでは、Xcodeプロジェクトを前処理する必要があります。info.plistを構成し、設定をビルドします

    リリースとアドホックビルドには、結果の形成が異なるスクリプトもあります。リリースは、iTunes Connectでアーカイブをダウンロードし、IPAをアドホックダウンロードし、マニフェストとダウンロードページを無線で作成し、すべての関係者にリンクが送信されます。

他の人がこのシステムでどのように働くか



  • 開発者は、テンプレートファイル.gitlab-ci.ymlをプロジェクトルートに追加し、gitlab.comのプロジェクト設定にランナーを含め、コードを目的のブランチにプッシュします。
  • ゲームデザイナーとテスターは、アセンブリの成功に関する通知と、Slackでapkおよびipaアーカイブをダウンロードするためのリンクを受け取ります。
  • 私はビルドを追跡し、ログを確認し、開発者がバグに対処するのを助けることができます。 ログと実行中のビルドは、gitlabで直接確認できます。 マイナスの-今、インターフェイスでは、特定のランナーのビルドキューを見ることができません。
