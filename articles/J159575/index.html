Djangoアプリケーション用バトルサーバー：Ubuntu Server 10.04 LTS + django 1.4 + nginx + gunicorn

多くのDjango開発チュートリアルでは、機能するデバッグサーバー（python manage.py runserver）を迅速に取得する方法が明らかにされており、戦闘モードでのデプロイの問題は未解決のままであるか、最も単純で最も効果的な方法とはかけ離れています。

以下では、ホスティングの選択からWebサーバーのデプロイまで、戦闘モードでDjangoにサイトをデプロイする方法の1つについて説明します。 したがって、この記事は、Djangoに基づいた開発をマスターしたが、サーバーの展開の経験がない人には役立つかもしれません。 私の方法は数多くありますが、非常にシンプルで、操作が効率的で、保守が容易です。 VPSホスティング、Ubuntu 10.04、nginx、gunicornを使用します。

ホスティング



私にとって、djangoの最も便利なホスティングはVPSホスティングであることがずっと前にわかりました。 いくつかのVPSホスティングを試した後、価格/品質比で最も受け入れられるFirstVDSに決めました 。 私は月に400ルーブルの料金を使用します-1500 MB RAM、24000 MB HDD。 ただし、最初は150ルーブルの関税がかかります（この場合、一部のアプリケーションに、より少ないリソースを消費するように教える必要があります）。

したがって、開始するには、VPSを注文して支払い、仮想化OpenVZの種類を選択する必要があります-物理マシンの不完全なシミュレーションですが、タスクには十分です。同じお金で仮想化を行うと、memcachedに役立つメモリが増えます。 。 オペレーティングシステムとして、Ubuntu Server 10.04 LTS＃を選択することをお勧めします。 Ubuntu Server 12.04 LTSも利用可能です、私はそれをそれほど難なく動作させることができましたが、最近ホスティングにテンプレートが現れました、そしてホスティングプロバイダーのサポートスペシャリストはまだバトルサーバーにそれをインストールすることを推奨していません。 現在は、テストサーバーでのみ使用しています。



代替のUbuntuサーバーホスティングオペレーティングシステム
確かに、より安定したホスティングシステムの支持者がいます：FreeBSD、Debian、RedHat、CentOS。 ホリバーには関与しません。 私の経験では、Djangoでの開発には、かなり新しいバージョンのパッケージとアプリケーションが必要になることが多いことに注意してください。 これらのパッケージは、多くの松葉杖に支えられてリストに記載されたシステムに入ります。その結果、信頼性が失われます。 Ubutnuでは、パッケージの新しいバージョンがリポジトリでより速く受け入れられます。





メールを注文して支払いを済ませると、SSHアクセスが可能になり、オペレーティングシステムの設定を開始できます。



Ubuntuサーバー



アクセスを取得したら、まずシステムを更新してから、SSHルートアクセスを無効にします。

SSHは、よく知られているPuttyプログラムを使用してWindowsに接続できますが、私はより「ねじれた」修正-kittyを使用します。 サーバーのIPアドレスとして、ホストによって送信されたアドレスを入力し、接続し、プロンプトが表示されたらユーザーおよびパスワードとしてrootを入力します。

Linuxに似たシステムに接続するには、単純なコマンドを入力してネイティブターミナル（ターミナル、Konsoleなど）を使用します。 

 ssh root@1.2.3.4


1.2.3.4はホストによって送信されたIPアドレスです。プロンプトが表示されたら、ホスティングプロバイダーによって送信されたパスワードを入力します。

まず、ルートパスワードを変更します。 

 passwd


リクエストに応じて、新しいパスワードを2回入力します。生成してKeePassXに保存することをお勧めします。より複雑にすることをheしないでください（20文字以上）。ほとんど使用する必要はありません。

システム内のパッケージの更新： 

 apt-get update apt-get upgrade


サーバーをデプロイするユーザーに代わってユーザーを追加します。 

 adduser myuser


パスワード以外は何も入力できませんが、より複雑なものを考え出す方が良いです

今、あなたは設定を編集するためのエディタが必要です、私はナノを好む： 

 apt-get install nano


ユーザーにルート（sudo）としてコマンドを実行する権限を追加します。 

 nano /etc/sudoers #   root ALL=(ALL:ALL) ALL #     myuser ALL=(ALL:ALL) ALL


次に、rootユーザーがSSH経由でログインする機能を無効にします（オプション、セキュリティ対策）。 

 nano /etc/ssh/sshd_config #  PermitRootLogin yes #   PermitRootLogin no


その後、マシンを再起動して、すべてが正常に実行されたことを確認します。 

 reboot


1分後、myuserとしてログインできます（または、ログインが機能しないことを事前に確認できます）。 

 ssh myuser@1.2.3.4




次に、Djangoアプリケーションとインフラストラクチャが機能するために必要なシステムパッケージをいくつか配置します。 

 sudo apt-get install gcc mysql-server python-mysqldb memcached mercurial python-profiler w3m python-setuptools libmysqlclient-dev git-core python-dev rabbitmq-server supervisor nginx


何が何なのか見てみましょう：

  • gcc-C、C ++、その他のヒープの前のコンパイラ、いくつかのパッケージをビルドする必要があります。
  • python-setuptools-easy_installコマンドをインストールする場合、virtualenvをインストールするだけです。事実、virtualenvはシステムパッケージの最新バージョンではないため、多くの厄介な問題が発生します。
  • mysql-server-名前から、これがMySQLサーバーであることは明らかです。Webアプリケーションホスト上でデータベースサーバーを上げるため、これが必要になります。
  • libmysqlclient-dev-このパッケージは、Python用のMySQLアクセスインターフェイスをアセンブルするために必要です。
  • python-dev-いくつかのパッケージをビルドするために必ず必要なpython API。
  • スーパーバイザはプログラムです。デーモンコントローラです。Webサーバーと、場合によっては他の多くのプログラムを飼いならす必要があります。
  • nginx-nginx Webサーバー、 Igor Sysoevに感謝します。
  • rabbitmq-server-オプションのAMQPサーバー。セロリを使用する必要がある場合に役立ちます。
  • memcached-オプションで、RAM内のキャッシュの既知の実装。
  • git-core-オプション、gitバージョン管理システムのクライアント。必要なパッケージがgitリポジトリに保存されている場合は必須です。
  • mercurial-オプション、mercurial（hg）バージョン管理システムのクライアント。必要なパッケージがmercurialリポジトリに保存されている場合に必要です。




プロジェクト展開



デプロイには、virtualenvツールを使用して仮想環境を使用します。

前の章では、便利なeasy_installユーティリティをインストールしました。これを使用して、virtualenv最新バージョンをインストールします。 

 sudo easy_install virtualenv


ご注意 仮想環境のpipが偶発的に競合しないように、意図的にpipをグローバルにインストールしませんでした。

これで、プロジェクトの展開を開始できます。 まず、ホームディレクトリのどこかに配置する必要があります。 

 /home/myuser/web/myproject


gitバージョン管理システムを使用しています。 ただし、SCPに沿ってコピーするだけでこれを実現できます。 Windowsでは、 WinSCPを使用できます。Linuxでは、ファイルマネージャーはアドレス形式sftpをサポートします：//myuser@1.2.3.4、おそらく推測されるように、認証データはSSH経由のアクセスに使用されます。

私のプロジェクトでは、仮想環境を構築するためのルートファイルはbuild_env.shです。 

 #!/bin/bash echo $0: Creating virtual environment virtualenv --prompt="<myenv>" ./env mkdir ./logs mkdir ./pids mkdir ./db mkdir ./static_content mkdir ./static_content/media echo $0: Installing dependencies source ./env/bin/activate export PIP_REQUIRE_VIRTUALENV=true ./env/bin/pip install --requirement=./requirements.conf --log=./logs/build_pip_packages.log echo $0: Making virtual environment relocatable virtualenv --relocatable ./env echo $0: Creating virtual environment finished.


また、requirements.confファイルには、アプリケーションが機能するために必要なパッケージが含まれています。 

 django git+git://github.com/sehmaschine/django-grappelli.git#egg=django-grappelli git+git://github.com/django-mptt/django-mptt.git#egg=django-mptt git+git://github.com/krvss/django-social-auth.git#django-social-auth git+git://github.com/gabrielhurley/django-wymeditor.git#django-wymeditor git+git://github.com/jtauber/django-mailer.git#django-mailer git+git://github.com/tweepy/tweepy.git#tweepy django-celery django-debug-toolbar django-pdb python-memcached MySQL-python xlrd unidecode anyjson gunicorn pillow south fabric requests xlwt


これは可能なパッケージのセットです。いくつかのパッケージについて詳しく説明しますが、このトピックの開発面については説明しません。

そのため、プロジェクトフォルダー/ home / myuser / web / myprojectで開始した後： 

 ./build_env.sh


仮想環境は、サーバーの操作のためにインストールされます。

ここで、データベースの作成に注意する必要があります。 プロジェクトでは、MySQLを使用します。 MySQLコンソールを起動します。 

 mysql -uroot -pROOTPASSWORD


データベースとユーザーを作成する場所： 

 CREATE DATABASE myproject CHARACTER SET utf8 COLLATE utf8_general_ci; CREATE USER 'myproject'@'localhost' IDENTIFIED BY 'USERPASSWORD'; GRANT ALL PRIVILEGES ON myproject.* TO 'myproject'@'localhost';


これで、アクセス設定をプロジェクト設定ファイル（settings.pyまたはlocal_settings.py）に入力できます。

ベースダンプがある場合は、何をすべきかを知っています。 そうでない場合は、その中に図と必要なデータを作成します。 

 cd ~/web/myproject #     source env/bin/activate #    python manage.py syncdb #
次に、静的ファイルを収集します。 

 python manage.py collectstatic


このアプリケーションで実行する準備が整いました。



Nginx Webサーバーを構成する



Nginx Webサーバーを構成するには、myproject.conf構成（接尾辞は.confでなければなりません）を/ etc / nginx / sites-available /フォルダーに配置する必要があります。

nanoを介してSSHコンソールでファイルを作成および編集できます。 

 sudo nano /etc/nginx/sites-available/myproject.conf


次に、以下の例の内容を貼り付け（Shift + Insert）、必要に応じて修正し、保存（Ctrl + O）、終了（Ctrl + X）します。

myproject.confファイルの内容の例： 

 upstream myproject.ru { server localhost:12345 fail_timeout=0; } server { listen 80; server_name www.myproject.ru; rewrite ^/(.*) http://myproject.ru/$1 permanent; } server { listen 80; client_max_body_size 4G; server_name myproject.ru; access_log /home/myuser/web/myproject/logs/myproject.access.log; keepalive_timeout 5; root /home/myuser/web/myproject/static_content; location / { proxy_pass http://myproject.ru; } error_page 500 502 503 504 /500.html; location = /500.html { root /home/myuser/web/myproject/static_content/static/html; } location ~ ^/(static|media)/ { proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header Host $http_host; proxy_redirect off; if (!-f $request_filename) { proxy_pass http://myproject.ru; break; } } }
一般的な言葉を理解してみましょう、何が何です。

この例では、アップストリームはlocalhost：12345ソケットであり、次のURLを満たさないすべての要求をそこに転送します。

/ static / *-静的サイトファイル。

/ media / *-メディアファイル、サイトの運用中に作成されるファイル。

このソケットに、リスニングのために記事の次の部分でgunicornを掛けます。 理論的には、アップストリームの名前は任意に選択されますが、アップストリームの名前がサイトのドメイン名と異なる場合、問題に遭遇しました（詳細を説明するDjangoエラーのテキストは覚えていません）。



構成のさらに興味深い部分：

www.myproject.ruサブドメインからmyproject.ruにリダイレクトします。必要に応じて反対のことができます。

5XXエラー処理は、静的な500.htmlファイルの出力によって編成されています。 このようなファイルの取得は非常に簡単です。サイトに404ページを表示し、ブラウザに500.htmlとして保存し、エラーテキストを編集します。

静的ファイルが見つからない場合、要求はgunicornに送信され、標準の404エラーをスローできるようになります。



構成ファイルが存在する場所に配置されたら（コマンドcat /etc/nginx/sites-available/myproject.confで確認できます）、シンボリックリンクを配置する必要があります。 

 sudo ln -s /etc/nginx/sites-available/myproject.conf /etc/nginx/sites-enabled/


nginxを再起動します。 

 sudo service nginx restart




gunicornとスーパーバイザーを構成する



プロジェクトをデプロイするときに、gunicornパッケージをインストールしました。これを使用して、動的HTTP応答（HTTP応答）Djangoアプリケーションを生成します。 Gunicorn自体はリクエストを受け入れてそれらに対する回答を生成できますが、gunicornを実行し続ける必要があります。たとえば、システムの再起動時に起動します。これをこのプログラムに特化したスーパーバイザーに任せます。

gunicorn開発者は、最前線に配置することを推奨していないことに注意してください。むしろ、nginxなどの専用サーバーにユーザーからのリクエストを受信するよう指示します。

したがって、/ etc / supervisor / conf.d /フォルダーで、次の内容のmyproject.confファイルを作成します。 

 [program:myproject] command=/home/myuser/web/myproject/env/bin/python /home/myuser/web/myproject/src/myproject/manage.py run_gunicorn --bind=localhost:12345 --workers=3 --pid=/home/myuser/web/myproject/pids/gunicorn.pid --log-file /home/myuser/web/myproject/logs/gunicorn.log directory=/home/myuser/web/myproject/src/myproject umask=022 autostart=true autorestart=true startsecs=10 startretries=3 exitcodes=0,2 stopsignal=TERM stopwaitsecs=10 user=myuser


gunicornのパラメーターでは、盗聴用の同じポート（この例では12345）を渡すことに注意してください。これはnginx構成で示されています。 ワークプロセスの数は3に設定されています。ワークロードに基づいて、サイトの1つに対してこのパラメーターを主観的に選択しますが、これまでのところ、値を推奨することはできません。

構成が正しいフォルダーに配置されたら（cat /etc/supervisor/conf.d/myproject.confを実行することで確認できます）、ファイルを再読み取りするか、スーパーバイザーを再起動する必要があります。信頼性のために2番目のオプションを使用します。 

 sudo supervisorctl reload


次のコマンドを使用して、起動の成功を確認できます。 

 sudo supervisorctl status


成功した場合、myprojectプロセスのステータスは数秒で実行中になります。

最初から、コマンドを入力すると、システムは警告の発行を開始します。 

 perl: warning: Setting locale failed. perl: warning: Please check that your locale settings:
正しいロケールを追加して解決しました： 

 locale-gen ru_RU.UTF8

sudo：setuidルートである必要があります
新しいユーザーの下からsudoを初めて使用するとき、エラーが発生する可能性があります。 

 sudo: must be setuid root


たとえば次のように解決しました： 

 chmod 4755 /usr/bin/sudo






E：問題を修正できません。壊れたパッケージを保持しています。
Ubuntu 12.04にいくつかのパッケージ（たとえば、python-devまたはlibmysql-dev）をインストールすると、エラーが発生する可能性があります。 

 libc6-dev : Depends: libc6 (= 2.15-0ubuntu10.2) but 2.15-0ubuntu10+openvz0 is to be installed E: Unable to correct problems, you have held broken packages.
これを解決するには、/ etc / apt / preferences.d / 99ovz-libc-pinファイルを変更するだけです：

交換 

 libc-bin libc6


に 

 libc-bin libc6 libc6-dev libc-dev-bin






/ dev / nullを開けませんでした：許可が拒否されました
ユーザーを作成するときに、エラーが発生する場合があります。 

 Couldn't open /dev/null: Permission denied


このように決めた： 

 sudo chmod 666 /dev/null


これがどれほど正しいかはわかりませんが、危険は見ませんでした。





おわりに



この展開の肯定的な側面に注意してください。

  • VPSにソリューションを展開する仮想マシンは、実質的に無制限の管理機能を提供します。
  • たとえば、Apacheやmod_pythonを使用する場合によく示唆されるように、Guninornのnginxの独立性。 特に、デバッグ用に開発サーバーを備えたgunicornに置き換えるのは非常に簡単です。 

     sudo supervisorctl stop myproject python manage.py runserver localhost:12345 #


    これで、nginxは変更について学習することなく、開発者のサーバーにリクエストを送信します。
  • 静的ファイルのディストリビューターとしてのNginxは、最高の、おそらく最高の1つとして認識されています。
  • virtualenvを使用すると、異なるプロジェクトの仮想環境を簡単かつ自然に分離できます。 現在、ディスクスペースのオーバーヘッドは、パッケージバージョンの競合よりもはるかに安価です。
  • スーパーバイザーによって制御されるgunicornは、/ etc / init.d /のスクリプトおよび他のいくつかの起動方法によって制御されるgunicornを実行する方法よりもはるかに便利で信頼性があります。


Djangoアプリケーションをデプロイするための多くのオプションの1つだけがsatyaで考慮されています;私は多くの試行錯誤を通してそれに取り組みました。 私の推奨事項は、究極の真実であると主張していません。 さらに、建設的な批判を喜んで受けます。
