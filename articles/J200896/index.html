ATtiny85用のコンパクトなUSB HIDブートローダー



AVR Tiny 45/85用の新しいコンパクトブートローダーを紹介します。 このオートローダーは、V-USBライセンスの必要に応じてGNU GPLの下でライセンスされています。 このオートローダーの基盤は、Embedded CreationsのATtiny85 USBブートローダーと、Digisparkボードで使用されるその子孫のmicronucleus-t85でした。

上記と同様に、TinyHID LoaderはV-USBライブラリに基づいています。



主な機能：



  • ATtiny45およびATtiny85で動作します
  • コンパクト（基本機能を備えた2kb）
  • 拡張可能（追加機能がいくつかあり、それらを含めるとサイズが大きくなります）
  • ドライバーは必要ありません（Windowsの新しいバージョンでは特に重要です）
  • AVRdudeでは動作せず、代わりにC＃でクロスプラットフォームAPIを使用します
  • 自分自身を更新できる




そして、なぜAVR Tiny 85なのか？



経済は、経済的な紳士でなければなりません。 また、AVR Tiny x5を使用すると、2回、さらに2つの方法で節約できます。 1つは価格の問題です。tinkyはmegよりも安いです。 しかし個人的には、Tiny x5にはPLLジェネレーターがあり、水晶発振器を放棄することもできます。 そのため、さらに2つのコンデンサを節約できます。 さらに、ボード上のスペースの問題もあります。 QNF本体では、ゴキブリのサイズは2.5 x 2.5 mmです。 また、共振器は必要ありません。 回路全体は、とんでもない12x5 mmに簡単に収まります。 要約すると、AVR Tiny 85の使用は常に適切です。特に、このゴキブリがサポートしない機能が必要になるまで（たとえば、多くのIOポートが必要な場合）、USBデバイスに適しています。



なぜ特別なオートローダーが必要なのですか？



AVR Megaのオートローダーは1ダースですが、Tiny 85ではそれほど単純ではありません。 これらの困難を理解するために、古典アルゴリズムの概要を説明します。



  • BOOTRST構成ビットが0に設定されている場合、再起動後、コントローラーはゼロアドレスからではなく、ブートリセットアドレス（異なるモデルに固有で構成可能な）から実行を開始します。
  • ブートリセットアドレスに従って、オートローダー自体が配置され、まず、割り込みテーブルをブートリセットアドレスに移動します。
  • オートローダーは、それ自体でロードする必要があるかどうかを判断します。 必要に応じて、通常のファームウェアとして起動しますが、ゼロアドレスにはありません。
  • オートローダをロードする必要がない場合は、割り込みテーブルをゼロアドレスに戻し、そこに制御を移して、アプリケーションファームウェアをロードします。
  • フラッシュにデータを書き込んでいる間、オートローダーは引き続きUSBプロトコルで動作し、ホストの要求に応答します。
  • また、オートローダーのメモリ領域は記録できないため、オートローダーが損傷することはありません。




そして今、85番目のタイニーのニュアンス：


  • ATtinyは常にゼロアドレスから起動します;別のアドレスに設定することはできません。
  • 割り込みベクターもどこにも転送されません。
  • 録音中、コントローラーは4.5ミリ秒間停止し、何にも応答できません。
  • まあ、オートローダーには自己記録に対するハードウェア保護もありません。




および回避策。


  • クリアされたメモリは、コントローラによってNOPとして扱われるユニットでいっぱいになります。 そしてこれは、リセット後、コントロールがオートローダーに到達することを意味します。 これまでに少なくとも1,500のNOPが完了します。
  • しかし、USBプロトコルが機能するには、これだけでは不十分です。 PCINT0割り込み応答も必要です。 割り込みベクターテーブルは移動できませんが、このテーブルにオートローダーのアドレスを書き込むことができます。 したがって、最初のブート直後に、オートローダーはハンドラーのアドレスベクトルをRESETおよびPCINT0に書き込みます。
  • アプリケーションファームウェアを記録する場合、オートローダーはRESETおよびPCINT0ベクトルも置き換えます。
  • オートローダーは、適用されたファームウェアの初期RESETとPCINTをオートローダーの直前のアドレスに書き込みます。
  • PCINT0アプリケーションファームウェアプロセッサを呼び出すことができるように、V-USBプロセッサPCINT0を変更します。 オートローダーハンドラーが機能するための条件は、TCCR1 == 0 && TCNT1 == 0xffです。 それ以外の場合は、アプリケーションハンドラーを実行します。
  • メモリを記録またはクリアする前の遅延。 この時点で、コントローラーはホストに操作の成功を通知し、ホストはそれを失いません。
  • メモリクリーンアップ操作の後、オートローダーは常にRESETおよびPCINT0ベクトルを書き込みます。
  • アプリケーションファームウェアを書き込む前に、オートローダーはFLASHを最初から最後までフラッシュします。 このタイプのクリーニングにより、予期しない停電によってクリーニング/記録プロセスが中断された場合でも、オートローダーは動作し続けます。 または、FLASHの最初のページはまだ消去されていないか、すでに書き込まれており、オートローダーの制御はベクター上で移動します。 すべてのメモリがクリアされ、NOPのチェーンを介してオートローダーに制御が転送されます。
  • ハードウェアの代わりにオートローダーのエントリのソフトウェアチェックが実行されます。




オートローダーの機能の詳細については、 Embedded Creations WebサイトのAVR Tiny 85をご覧ください。



なぜ車輪を再発明するのですか？



前の章で説明した理由により、AVR tinyで従来のMEGAベースのオートローダーを使用することはできません。 しかし、それらがなくても、ATtiny85 USBブートローダーとmicronucleus-t85の2つのオートローダー実装があり、どちらもかつて人気のあったUSBaspプログラマーをエミュレートします。 また、前述のプログラマーはlibusbライブラリーを介してホスト上で実行されます。 これは素晴らしい多機能ライブラリであり、Mac OSまたはLinuxでの使用は2x2の基本です。 自分で作成したものを自分で使用する場合、TinyHID Loaderは、よりコンパクトなサイズを除いて魅力的かもしれません。 しかし、Windows 8.1 x64では、libusbドライバーのインストールは簡単な作業ではなくなりました。 設定の深いところには、ドライバーの電子署名検証を無効にしてコンピューターを再起動できる項目があります。 そして、この方法を一般的なユーザーに説明する人は誰にとっても悲惨です。 したがって、主なアイデアは、HID機能レポートを介して機能するオートローダーを作成することでした。 すべてのHIDデバイスでは、標準のOSドライバーが使用され、追加のドライバーをインストールする必要はありません。 また、avrdudeのサポートに失敗すると、ブートローダーの重量を減らすことができることがわかりました。 FLASHの書き込み/クリアおよびアプリケーションファームウェアへのプログラムアクセスを除くすべてのオプションを無効にすると、オートローダーの重量は2kbになり、競合他社よりも少なくなります。



使い方は？



まず、ファームウェアを構成する必要があります。 これを行うには、スキーム/ニーズに合わせてファームウェア/usbloader/usbloader.hファイルを調整する必要があります。 



//  ATtiny,    USB D- #define USB_CFG_DMINUS_BIT 2 //  ATtiny,    USB D+ #define USB_CFG_DPLUS_BIT 1 //  ATtiny,     // ( ,   ) // #define LED_PIN 4 //  ATtiny,       (  ) // (    ) #define START_JUMPER_PIN 0 //   1,     EEPROM (    ) #define CAN_ERASE_EEPROM 0 //   1,     FLASH (    ) #define CAN_READ_FLASH 0 //   0,         (   ) #define CAN_LEAVE_LOADER 1




同時に、PCINT0割り込みは任意のレッグで構成可能なオートローダーによって使用されるため、USB_CFG_DPLUS_BITがINT0レッグを取得することに注意する必要はありません。

上記のバージョンでは、ブートローダーはエンドツーエンドですが、2kbに適合します。 追加のオプションを有効にすると、これらの制限を超えて彼を連れて行くと同時に、あなたはメモリ内の彼の場所を変更する必要があります。 これは、AtmelStudioを使用するかMakefileを使用するかによって異なります。



Atmelスタジオ


リリースが選択されていることを確認します。

プロジェクトプロパティを開き、[ツールチェーン]タブに移動し、[AVR / GNUリンカ/メモリ設定]項目で、値 ".text = 0xc00"を減らします。 32ワードの部分でのみ減らすことができます。 つまり、0xbe0、0xbc0、0xba0です。 0xba0は使用可能なすべてのオプションが機能するのに十分であり、それ以上の削減は必要ありません。 これに加えて、定数BOOTLOADER_WADDRESSの値も減らす必要があります。 これは、AVR / GNU Cコンパイラ/シンボルページで実行できます。 値は値 ".text ="と等しくなければなりません。

構成後、プロジェクト（F7）をコンパイルする必要があり、外部プログラマーを使用してコントローラーにアップロードできます。



メイクファイル


Makefileを開き、定数BOOTLOADER_ADDRESSを目的の値に減らします。 64バイト部分のみを削減できます（AtmelStudioは単語を使用し、Makefileはバイトを使用します）。 つまり、17c0、1780、1740。1740は使用可能なすべてのオプションが機能するのに十分であり、それ以上の削減はありません。

構成後、コマンドラインにmakeと入力してプロジェクトをコンパイルする必要があり、外部プログラマを使用してコントローラーにアップロードできます。

プログラマーは外部プログラマーを必要としません。コンピューターのソフトウェアにアクセスしてください。



ソフトウェア


ファームウェアをコントローラーにアップロードするユーティリティはC＃で記述されており、プロジェクトはVisualStudio 2012で開発されましたが、MONOでさらに作業するためにXamarinで開くこともできます。 HIDとの通信には、HidSharpクロスプラットフォームライブラリが使用されます。これにより、MONOプロジェクトが真のクロスプラットフォームになります。

ダウンロードユーティリティは非常に使いやすいです。 

 TinyLoaderCmd.exe firmware.hex


そして3秒後に、ゴキブリは縫われて行く準備ができました。 ファームウェアがデフォルトの構成で作成された場合、ユーティリティを起動するとコントローラーがフラッシュされるだけでなく、準備ができたら記録されたファームウェアが起動します。

または、C＃プロジェクトからTinyHID Loader APIを使用できます。 

 void UploadNewFirmware(string file) { HexFile file = new HexFile(args[0]); Loader ldr = Loader.TryGetLoader(40); byte[] programm = new byte[Loader.LOADERSTART]; for (int i = 0; i < programm.Length; i++) programm[i] = 0xff; file.Fill(programm); ldr.WriteFlash(programm, 0); ldr.LeaveBootloader(); }


ファームウェアにオートローダーブート機能を追加する場合： 

 void runBootloader() { cli(); TCCR1 = 0; TCNT1 = 0xff; asm volatile ("rjmp __vectors"); }


ボタンをクリックするだけで、ユーザーに向けて点滅します。 そして、その時点でのプロジェクト：



  • ファームウェアにオートローダーをロードするよう指示します。
  • システムがオートローダーをピックアップするまでしばらく待機します（Loader.TryGetLoader）。
  • 新しいファームウェアを記録します。
  • ファームウェアをダウンロードするようオートローダーに指示します。
  • システムが新しいファームウェアを取得するまで待機します。




また、別のオプションセットが必要な場合はどうしますか？



オプションのフルセットを提供しましたが、新しいファームウェアは適合しませんか？ または、メモリを読み取る機能を追加しなかったことを後悔していますか？ さて、またはTinyHIDローダーはまったくあなたに合わず、avrdudeなしでの生活は良くありませんか？ しかし、マイクロコントローラーはステッチされているだけでなく、密閉されています。 そして、私はそれをまったくはんだ付けしたくない。 質問なし-TinyHID Loaderはそれ自体を再構築できます！

自己点滅アルゴリズムは次のとおりです。



  • コマンドラインから、ユーザーはTinyLoaderCmd.exe reload bootloader.hexを入力します（オプションとして-誰も何も入力せず、ホスト上のソフトウェアからAPIを介してプロセスを実装します）
  • ユーティリティは、リロード用のファームウェアを作成します。 これには、新しいオートローダーのイメージ、それに関する概要情報（FLASHおよびCRC16の宛先アドレス）、および特別な更新ユーティリティ（リローダー）が含まれます。
  • ファームウェアは、他のファームウェアと同様に、通常の方法でマイクロコントローラーにアップロードされます。
  • デバイスが再起動し（このオプションが無効になっていない場合は、プログラムで実行される場合があります）、制御がリローダーに転送されます。
  • リローダーは、CRC16のサイズとデバイスに適合する能力をチェックすることにより、新しいオートローダーの正確性をチェックします。
  • すべてのチェックが成功すると、リローダーはメモリの最初のページをクリアします。 この瞬間から後戻りはありません。 また、電源が切れた場合、制御は古いブートローダーに転送されません。
  • 主な段階は、新しいオートローダーをイメージから実際のアドレスにコピーすることと、オートローダーの前のページを直接クリアすることです（すべてのオートローダーが使用するため）
  • そして最後に-新しいオートローダーのRESETベクトルに沿った移行-オートローダーがFLASHを再初期化して作業を開始します


リローダのセキュリティ保証は、CRC検証とプロセスのロジックであり、予期しない終了の場合に再び開始されます。 しかし、誤った新しいファームウェアはゴキブリをレンガに変え、プログラマーのみがそれを復活させることができます。 したがって、注意してください-設定が正しいことを確認してください。

また、リローダーを使用すると、別のプログラマーでデバイス上のTinyHIDローダーをフラッシュできます。 つまり、一般的なAVR Tiny 85オートローダーを他のものに置き換えることです。
