クロスデバイス広告の仕組み：技術開発の難しさと展望

はじめに



さまざまなデバイスを使用してネットワークにアクセスするユーザーが増えています。 同時に、多くの広告チャネルを使用して、広告主と潜在的な購入者とのやり取りが発生します。 多くの場合、個人がコンテンツを消費し、この対話チャネルを決定するデバイス。 ユーザーは、テレビ広告に興味があるかもしれませんし、その逆もあります-携帯電話やパーソナルコンピューター上のソーシャルネットワークでのコミュニケーションのために、その表示中に気を散らされるかもしれません。 ブランドまたは製品との最初の接触から購入時までの途中の潜在的な買い手は、複数のデバイスを変更できますが、同時にそれが常に個人的であるとは限りません。











[1]のデータによると、ロシア人の95％が携帯電話を所有しており、ラップトップまたはパーソナルコンピューターを使用しているのは80％のみです。 2014年のGoogle Russia [2]によると、62％がモバイルデバイスを使用して製品情報を検索し、国内ユーザーの39％が少なくとも一度はスマートフォンから購入しました。 また、たとえば、モバイルデバイスでの検索から始まった小売セグメントでの購入までの経路は、同じデバイスでの購入で3％のケースでしか終了しなかったことにも注目しています。



また、Criteoの専門家は、2016年にモバイルデバイスからRuNetで購入した数の増加が50％を超えると予測しました[3]。

上記は、使用されるすべてのチャネルおよび広告主が購入者とやり取りするすべてのデバイスでの広告キャンペーンの有効性を評価する必要性を決定します。



さまざまなデバイスに関連する広告を作成する方法を使用すると、特に、ネットワークへの複数のアクセスソースを持つユーザーの広告提案を一度に擬人化できます。 いわゆる関連広告には多くの英語の同義語があり、次のような多くの販売フレーズで人気があります。



  • Tapad ：複数の画面にわたってコンテンツをパーソナライズします。
  • BlueCava ：任意の画面で価値の高いターゲットに到達します。
  • Adelphic ：デバイスではなく人をターゲットにします。
  • Tactads / MediaMath ：適切なメッセージを適切な人に適切なデバイスで送信します。
  • Drawbridge ：デスクトップリターゲティングキャンペーンをモバイルに正確にスケーリングします。


今日は、これがどのように機能し、このアプローチが広く採用されるのを妨げるものについて説明します。



広告の自動調達のためのエコシステムの出現により、広告主は特定のサイト内だけでなく、広告キャンペーンに参加しているすべてのサイト内でもユーザーを識別することができます。 これにより、広告キャンペーンのカバレッジを確実に評価し、ユニークユーザーに対する広告の頻度を調整できます。 ただし、このようなサービスを提供するために、広告プラットフォームは、ユーザー識別子のセットを1つのユニバーサル識別子と比較するきちんとした作業を実行する必要があります。 広告キャンペーンの効果を監視します。



グローバルに、次のタスクは、多くの異なるユーザーIDを1つのユニバーサルIDに一致させる作業に関連する何らかの方法で区別できます。



  • さまざまなSSP（ 供給/販売側プラットフォーム ）でのユーザー識別。
  • ブラウザ間のユーザー識別。
  • ブラウザとモバイルデバイス上のアプリケーション間のユーザー識別。
  • さまざまなデバイス間のユーザー識別-広告技術の進化的開発の頂点として。

さまざまなSSPでのユーザー認証



1つのDSP（ デマンドサイドプラットフォーム ）は常に複数のSSPに接続されており、同じユーザー（異なるSSPで観察される）は、1つのデバイスからでも、広告キャンペーンのフレームワーク内の複数の異なるユーザーと見なすことができます。 これにより、一意のユーザーごとに広告を表示する頻度を確実に監視することはできません。



User



オブジェクトのIABからのOpen RTBプロトコル[4]は、SSPからDSPへのリクエスト（ bid ）時にユーザーを識別するための2つのフィールドを提供しますbuyerid



は特定のSSP内のユーザー識別子で、 buyerid



はDSPユーザー識別子です。 SSPがバイヤーIDに関する情報を送信できるようにするには、たとえば[5]のように、SSP側で識別子照合技術を実装する必要があります。 ただし、すべてのSSPが非常に大きなマッチテーブルをサイドに保存し、これに技術的なリソースを費やす準備ができているとは限りません。



実際には、ほとんどの場合そうではありません-SSPは、識別子照合プロセスの一部として、ユーザーIDをDSP側に転送します。 この場合、SSPはUser



オブジェクトのidフィールドに入力するのに十分であり、DSPは潜在的な表示が提供されるユーザーを個別に認識します。 あまり一般的に使用されないのは、識別子の双方向の交換です。



さまざまなSSPを介して受信したユーザーの相互識別に関するDSPの注意深い作業により、広告主が必要とするインプレッションの頻度に確実に準拠し、広告キャンペーンの範囲を広げることができます。



ブラウザーとアプリケーションでのユーザー認証



ユーザーが単一のブラウザー内でユーザーを識別できないことは珍しいことではありません。これは、ユーザーが匿名化のために特別なブラウザー拡張機能を使用し、特定の種類のCookieを設定できないことが原因である可能性があります（特にモバイルデバイスのSafariに当てはまります）。 これにより、そのようなユーザーからの広告トラフィックは、従来のリターゲティングのニーズにも使用できず、そのようなユーザーの広告の表示頻度の制御は機能しなくなります。 記載された問題を解決し、これらのユーザーを識別するために、指紋技術が通常使用されます（たとえば、 Panopticlick 、[6]）。[7]でさまざまな実装のかなり完全な比較が提示されます。 また、特定の匿名化技術を使用するユーザーは、その逆であることがわかります-ネットワーク上の匿名性の点でより脆弱です。



DSP Exebid.DCAの開発では、実験を停止して、現時点で明らかに明らかに反対しているユーザーを意図的に識別することで、はるかに簡単になりました。 そのようなユーザーのセグメントを選び出し、強力な識別が重要である広告キャンペーンを排除することができました。 より単純なケースでは、いわゆる サードパーティのCookieを使用すると、ローカルストレージ（ localStorage



）やpostMessage



[8]などの最新のブラウザーの機能を使用できます。



また、デスクトップデバイスの場合、ユーザーがさまざまなタスクに2つの異なるブラウザーを使用すると、かなり頻繁に発生する状況になります。 特別な識別手段を講じない場合、そのようなユーザーは、広告プラットフォームによって、互いに接続されていない複数の異なるユーザーと見なされます。 ここでは、ブラウザに依存しない機能が、たとえば[9]からの識別に使用されます。IPアドレス、インストールされているフォントに関する情報、タイムゾーン、画面解像度など。



別の問題は、モバイルアプリケーション内とWebサイトでのユーザーの行動の比較です。 一般的な場合、この問題には解決策がありませんが、モバイルアプリケーションでの広告の有効性を監視するために、「ネイティブクリック」のテクノロジーを使用できます（ ネイティブブラウザークリックなど、[10]）。 この場合、アプリケーションのIDFA（Advertizerの識別子 ）をモバイルデバイスのブラウザー内のユーザーのCookieと相関させることが可能になります。



電話で選択して、パソコンから購入する



現代のインターネット購入者はタブレットやスマートフォンからの広告提供に精通する傾向があり、Google Russia Stanislav Vidyaevの上級アナリストは、後でパソコンから商品を購入することを指摘しました。 「スマートフォンやタブレットの小さな画面からサービスを購入したり注文したりすることは、デスクトップから行うよりも難しい」と彼はGoogle Think Performanceカンファレンス[11]で語り、聴衆をユニークユーザーのクロスデバイストラッキングを導入することは避けられないという考えに導きました。



一部のデバイスは他のデバイスの変換に役立ちます



スマートフォン/タブレットとPCからユーザー行動モデルを接続し、それが同じ人物であると判断する機能により、広告主の生活は大幅に簡素化されます。 新しいテクノロジーを使用して、彼らはサイトまたはモバイルアプリケーションに優柔不断な訪問者を押して別のデバイスから再度アクセスして購入し（クロスデバイスユーザーIDに基づいたリマーケティング）、タブレットで既に無視しているユーザーに広告を表示しないことで予算を節約できますスマートフォンまたはデスクトップコンピューター、またはその逆。



クロスデバイス広告モデルの導入によるプラス面では、インターネットユーザー自身が留まる必要があります。 彼らは興味深く、パーソナライズされたオファーのみを受け取り始め、すべてのデバイスで同じ無関係な広告の退屈な流れを取り除きます。



クロスデバイス広告は、多くの技術的な困難のためにまだ客観的な現実にはなっていませんが、一部の企業はすでに新しい方法を試して最初の結果を得ています。 そのため、Adweekの広告に関する米国のプロファイル出版物のジャーナリストは、スマートフォン、タブレット、パソコンの3つの異なるデバイスで広告を表示したユーザーにリターゲティング技術を使用した高級車メーカーの広告キャンペーンの結果について学びました[12]。 自動車メーカーは、1つのデバイス上の連絡先と比較して、コンバージョンを15％増加させたことが判明しました。



クロスユーザーアプローチ



しかし、スマートフォン、タブレット、PCのホストが同じであることをどのように判断するのでしょうか？ Googleアナリティクスサービスでは、パソコンやIDモバイルデバイスのブラウザのCookieを追跡するだけでなく、特定の店舗の顧客IDを使用して一意のユーザーを識別できます（これは、このクロスデバイスマッチングテクノロジーがサイトの登録ユーザーに対してのみ機能するという事実につながります）。 システムは、承認後に異なるデバイスに座っている同じユーザーを認識します。 ロシアのGoogleの[11] Stanislav Vidyaevによると、これは広告主がユーザーの購入経路を追跡し、 「すべての取引がパソコンを使用して行われるためタブレットの広告は非効率的」などの誤った判断を回避するのに役立ちます。



このような決定論的追跡 （ 決定論的追跡 ）は、実際には、インターネットシステムでのユーザーログインの比較に基づいており、特定のターゲットサイトストアだけでなく、ソーシャルネットワークやブラウザー（たとえば、Google Chromeブラウザーでの設定とブラウジング履歴の同期） 2012年バージョン18以降のデバイス[13]。 したがって、このアプローチはより早く実装および普及することができます。



決定論的追跡は、ユーザー資格情報を収集するすべてのプラットフォームおよび発行元が使用できます。 ただし、すべてのユーザーを対象としたわけではありませんが、購入への道は2つの簡単なステップで構成されています。 誰もがサイトに登録したいわけではありません。



クロスデバイスマッチングへの2番目のより複雑なアプローチは確率的追跡で、確率的アルゴリズムを使用してさまざまなデバイスでのユーザーの行動を分析します。 BlueCava、Adelphic、Tapad and Drawbridge、またはロシアのDCAなどの大規模なテクノロジー企業は、多くのCookieのデータを収集し、検索エンジンの使用パターンの類似パターンを分析し、同じユーザープロファイルを持つ異なるデバイスの接続を決定するために特別なテストを使用します。



クロスデバイステクノロジーを実装するための最も明白なアプローチは、すべてのユーザーのすべてのデバイスの識別子のセット全体を、頂点がこれらの識別子であるグラフと見なすことです。エッジを持つ2つの識別子の重み付き接続は、特定の確率でこれらの識別子が1人の人物、たとえば特許に属することを意味します[14]。 このアプローチにより、ソーシャルネットワークでクラスターを検索するためのよく知られた方法を適用し、異なるデバイスで同じユーザーを識別することができます。



Drawbridgeによると、確率的追跡は97.3％の精度を保証できます[15]。 2013年、Expedia（チケットとホテルのオンライン予約サイト）はこのシステムのテストを実施しました。その枠組みの中で、Expediaアプリケーションのインストールを提案する広告がサイトのデスクトップオーディオサイトに提供されました[16]。 テスト結果によると、桁違いの変換の増加が認められました。



結論と開発の見通し



ネットワークに接続された複数のデバイスをユーザー購入者と接続できる、クロスデバイスアドバタイジングの高度なテクノロジーでさえ、デジタルツールの開発の限界ではありません。 クロスデバイス広告は、完全なクロスチャネルコンバージェンスに向けた最初のステップです。これには、インターネットサイト、メールニュースレター、テレビ、ラジオ、ソーシャルネットワーク、コールセンターなど、複数のチャネルを介して同じ広告のフォームを一度に関係者にブロードキャストすることが含まれます企業。 たとえば、コマーシャルを見て、カスタマーサービススペシャリストに電話して製品の詳細を調べることにした視聴者は、割引価格で商品を購入する旨のメールを送信したり、購入のボーナスを受け取ることができます。



効率的に機能するクロスチャネルモデルの導入はマーケティング革命となりますが、スペシャリストは、クロスデバイスモデルが問題なく機能する場合にのみ密接に開発できます。 現在、1つのチャネル（インターネット）のフレームワーク内であっても、デバイスの統合に問題が生じています。 2つのブラウザーを使用してネットワークにアクセスするユーザーは、システムによって2人の異なるユーザーとして認識されます。 さらに、モバイルデバイスでは、アプリケーション内およびブラウザ（そのうち複数の場合もあります）内のユーザーは、複数のユーザーとしてカウントされます。 したがって、モバイルデバイスのデフォルトブラウザーのCookieを使用して、アプリケーション内のユーザーを識別するために使用されるIDFAマッチングテクノロジーの開発が急務です。



ロシアでは、これにさらに困難が加わります。多くの家族ではスマートフォンのみが「個人」であり、複数の家族がタブレットとPCを一度に使用することが多く、この場合の広告は目的地に届かない場合があります。



明らかな困難にもかかわらず、クロスデバイス広告に未来があることは明らかです。 そのため、ビジネスでは、潜在的な顧客とそのニーズに関する情報を収集および処理する方法を開発する必要があります。
