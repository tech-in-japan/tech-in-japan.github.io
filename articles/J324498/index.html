ownCloudおよびNextcloudとの統合方法：落とし穴とそれらを克服する方法

ユーザーの1人がONLYOFFICEとownCloudは天国で作られた結婚であると書きました（申し訳ありませんが、オリジナルは天国で作られたものでした）。 私たちは彼が正しいと考えて考えました。 さらに、私たちは統合が大好きです！ それに関しては、もちろん、いつでもどこでも便利な編集者を求めています。



一般に、ドキュメントエディタをownCloud（およびNextcloud）と統合するためのアプリケーションを作成しました。この記事では、それについて少し説明します。 私たちの道には乗り越えられない障害はありませんでしたが、いくつかの困難が生じ、それらを克服した経験を共有します。 第一に、それはあなたにとって有用かもしれません、そして第二に、私たちはもはやそれを自分自身に保つことができません！ つまり、長い間ブログに何も書いていませんでした。









アプリケーションの作成は簡単で、ownCloudで動作します-いいえ



したがって、統合アプリケーションはコネクタであり、私たちの恋人は、ownCloudドキュメント管理WebサービスとDocument Server（ドキュメントサーバー、一般的にエディター）の間の単なる橋渡し役です。 ownCloudは、アプリケーションを作成してワークスペースに埋め込むためのAPIを丁寧に提供し、これを利用しました。



アプリケーションの作成において、ownCloudの公式ドキュメントは私たちを大いに助けてくれました。 そこで示されている手順を完了すると、既製のアプリケーションテンプレートが作成されました。 それは簡単でしたが、さらなる困難が始まりました。 内部APIは、控えめに言っても文書化されていますが、あまり詳細ではありません。



たとえば、ファイルを保存するには、特定のユーザーとして承認を取得する必要がありました。 許可アクションを実行するには、特定のコマンドを実行する必要があります。どのコマンドですか？ それらはどこにありますか？ それらを実現する方法は？ ドキュメンテーションにはこれについて何と書かれていると思いますか？ 正解は、「なし」です。



創造的な方法



文書化の難しさは、最初の段階から始まりました。 あるサーバー上のownCloudと別のサーバー上のDocument Serverへのバインディングを作成し、それらの間のデータ交換を整理するには、まずDocument ServerのWebアドレスを指定する必要があります。 このアドレスは、アプリケーションのデータベースに保存され、エディターを開いたり文書を変換するときに使用されます。



一般に、アドレスを指定する必要がありますが、どこですか？ 彼をどこに向けますか？ 最初に、ownCloud管理パネルにONLYOFFICE設定ページが必要ですが、何らかの理由でそれ自体は表示されませんでした（実際には必要でしたが）。 一般に、このページは何らかの形で書く必要があります。



ownCloudの他のオープンソースアプリが救いに来ました。 彼らがこのタスクにどのように対処したかを覗かなければなりませんでした。 具体的には、設定ページでは、ファイルアプリケーション用の独自のownCloud Antivirusアプリが役に立ちました。



このアプリケーションの開発者の経験に基づいて、追加したものと場所を次に示します。

ODT、ODS、ODP、DOC、XLS、PPT、PPS、EPUB、RTF、MHTのアクションを選択すると、HTMLファイルが適切なOOXML形式への変換のために最初に送信されます。 DOCX、XLSX、PPTX、PPSX、TXTの場合、エディターの新しいタブが開き、CSV、XPS、PDF、DJVUの場合、エディターの表示モードのタブが開きます。



Nextcloudについて



自分たちのownCloudで作業していることを世界に知らせるとすぐに、世界はすぐにNextcloudでも作業することを要求しました。 これらの要件は公正であると考えましたが、集会はこのシステムで機能し始めませんでした。



生じた困難は次のとおりです。



  1. 行くアイコン（もちろん、スタイルを修正することは問題ではありません）。
  2. Nextcloudはコンテンツセキュリティポリシーを追加しました（インラインエディターの初期化スクリプトに署名が必要です）。
  3. [設定]ページの新しい要件 。


しばらくの間、関連システム用に2つの個別のアプリケーションを作成する必要があると考えた3番目のポイントのためでした。 それは次のとおりです。NextCloudで新しいインターフェイスを実装するには、アプリケーションの説明に設定のクラスを登録する必要がありました。



/appinfo/inxo.xml 

 <settings> <admin>OCA\Onlyoffice\Controller\SettingsController</admin> </settings>




そして、クラスに必要なISettingsインターフェイスを実装します。



/controller/settingscontroller.php 

 class SettingsController extends Controller implements ISettings { public function getForm() { return $this-index(); } public function getSection() { return "server"; } public function getPriority() { return 50; } ...




しかし、ここには小さな問題があります。 この行にぶつかり、ownCloudがクラッシュしました。その環境にはISettingsインターフェイスがないためです。 



 class SettingsController extends Controller implements ISettings




ISettingsインターフェースの実装を別のクラスに作り、ownCloudとNextcloudレイアウトのレイアウトを統一したままにして、トリッキーな方法ですべてを行う必要がありました。



/appinfo/inxo.xml 

 <settings> <admin>OCA\Onlyoffice\AdminSettings</admin> </settings>




/lib/adminsettings.php 

 class AdminSettings implements ISettings { public function getForm() { $app = new Application(); $container = $app->getContainer(); $response = $container->query("\OCA\Onlyoffice\Controller\SettingsController")->index(); return $response; } public function getSection() { return "server"; } public function getPriority() { return 50; } }




したがって、ownCloudの設定ページは古いアルゴリズムに従って接続され、Nexcloudの設定ページは新しいアルゴリズムに従って接続されます。 ちなみに、私たちのアプリケーションからフォークを持ち込み、自分でNextcloudにドッピングしたイニシアチブの人たちがいました。 彼らはフォーラムでこれがうまくいくと書いており、私たちはそこですべてがどのように行われるかに自然に興味を持った。 しかし、彼らは設定に関する問題の解決策を見つけられず、すべてを独自の方法で行いました。



どうしたの？



人気のあるownCloudとNextCloudのインターフェースでONLYOFFICEエディターを使用してドキュメントを操作できるアプリケーションを入手しました。 公式には、両社はCollaboraと協力していますが、ご存じのように、誰もがCollaboraの展開に成功しているわけではなく、本格的な共同編集はありません。 さらに、主なものとしてさまざまな形式を選択しており、「X」がさらに好きな場合は、ONLYOFFICEがそれらに合わせて調整されます。 コネクタを使用して、ownCloudまたはNextCloudを無料版のコミュニティとエンタープライズの両方に接続できます。
