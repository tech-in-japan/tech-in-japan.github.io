Unity3Dの小規模および中規模プロジェクト用の代替サウンドマネージャー

小さなプロジェクトでの使用に適したサウンドマネージャに関する別の記事が、この記事を書くきっかけになりました。 この投稿では、著者がリストしていない欠点のいくつかを説明し、私の意見では、それらを修正する独自の実装オプションを提供します。



この記事は、初心者の開発者が経験を積んでベストプラクティスを習得するのに役立つだけでなく、熱心なアーキテクトにとっても役立ちます。 私が提案した解決策は完全に普遍的ではなく、欠点があると確信していますが、私にとって重要で楽しい要素は、興味のある各habrayuzerが自分に役立つ何かを学び、私のアドバイスを使用して自分のモジュールを改善することです。



問題



怒っている孤独



多くの人が私に反対するかもしれませんが、特に音の再現などの側面でのシングルトーンの使用は、あらゆる規模のプロジェクトでは受け入れられないと考えています。 このアンチパターンを使用すると、コードのすべてのセクションが緊密に接続され、タイプが直接示され、一度に複数の方向に手を結びます。 可能であれば、シングルトンのテストを書くことは非常に難しく、見苦しく、決定論できらめきません。 また、このサウンドマネージャーが使用するモジュールのテストをエレガントに作成することはできません。 制御されていないライフサイクルで同じインスタンスを使用しているという事実により、手を自分自身にバインドし、お互いのことをまったく知らないコードの別々のセクションに論理的に依存関係を作成します。



例： 



static void PlayMusic(string name); static void PlaySound(string name, bool pausable = true);


このメソッドは、サードパーティのコードが特定のメロディ名を認識していることを主張しています。 引数を正しく渡すのは、サウンドを担当する各モジュールのプログラマーの責任です。 また、プロジェクトには、さまざまなUI要素、射撃/死にかけているユニット、環境など、このような場所がたくさんあります。 ref記事へのコメントで、読者の1人は、引数のサウンドに異なるチャネルを使用することを提案しています。これにより、コードのセクションも論理的に接続されます。 



 public void PlayFX (AudioClip clip, SoundFXChannel channel = SoundFXChannel.First, bool forceInterrupt = false) { } public void StopFX (SoundFXChannel channel) { }


現在、たとえば、メソッドを使用するボタン（または、必要に応じてUIManager）は、どのチャンネルに属しているかを考慮に入れる必要があります。実際、これはプログラマーの責任です。



アクセスが多すぎる



私が別のコードでメソッドを呼び出すと、MonoBehaviourから型継承を返すことは常に奇妙でした。 コルーチンを起動しても安全ですか？ 開発者はそれをDestroy（）から保護しましたか？ または、将来コードで「UnityEngineを使用する」ことを確認したいですか、MonoBehaviourは必要ありませんか？ この問題は、シングルトンに関する前の段落に部分的に適用されます。インスタンス自体へのリンクは必要ありません。それを使用するのに十分なAPIがあります。 おかしいが、この方法で静的呼び出しを実装する場合でも： 



 private static SoundManager instance; public static ISoundManager Instance { get{ return (instance as ISoundManager) }}


抽象化を受け取るとき、特定のタイプを使用する必要があります： 



 ISoundManager sm = SoundManager.Instance;


問題を部分的にしか解決しません。



ステッチパスと直接ダウンロード 



 private AudioClip LoadClip(string name) { string path = "Sounds/" + name; AudioClip clip = Resources.Load<AudioClip>(path); return clip; }


私の意見では、音の読み込みの遅延は必ずしも意味をなさない。 まず、サウンドをユニットにインポートするための設定で、サウンドを保存する方法を構成できます：すぐにRAMに、ディスクからストリーミングまたはメモリにロードし、再生の直前に変換します。 インポート設定の詳細 。 第二に、ユニットアセンブリログを分析した経験から、全体的なサイズでの健全なリソースは平均3位以下であることが示唆されます。 また、メモリの最適化は、開始した場合、サウンドでは明確ではありません。 （もちろん、これはゲームプレイがサウンドに関連付けられているプロジェクトには適用されない可能性があります）。 ログの詳細 。



さて、コードに縫い付けられたパスに関して：再び、プログラマーの責任は、プロジェクトからプロジェクトへこのモジュールを転送するとき、パスの対応を監視することです。 本当のダンスは、「考えてみると、すべてのプロジェクトで必要に応じてこのモジュールの最新バージョンが存在するように、gitサブモジュールを作成し、そこにオーディオマネージャーを配置しないのはなぜか」と考えたときに始まります。 パスはコードに埋め込まれているため、他のプロジェクトではエラーになるため、変更することはできません。 一方、ローカルでのみパスを変更した場合、gitは常にこの変更を反映します。

それでは、最初にアーキテクチャについて説明しましょう。



このオーディオモジュールは、すべてのアーキテクチャの依存関係グラフの最終シートと見なされ、グラフの下に依存関係は必要ありません。作成者は関係ありませんが、制限があります。このモジュールは「シングルトン」ライフスタイル（シングルトンデザインパターンと混同しないでください）詳細については、Mark Siman著の「.NETでの依存性注入」を参照してください）。 これは、アプリケーションごとに1つのAudioListenerのみのUnity3D要件によるものです。 プロジェクトで依存性注入を使用している場合、バインダーは次のようになります（例としてNinjectを使用）。 



 binder.Bind<IAudioController, IAudioPlayer, IMusicPlayer>().To<AudioController>().InSingletonScope();


このクラスを作成してプロジェクトで使用するだけの場合は、オーディオ再生呼び出しのすべてのソースに同じインスタンスの抽象化が提供されていることを確認してください。



例として： 



 var ac = new AudioController(); IAudioController iac = ac; IAudioPlayer iap = ac; IMusicPlayer imp = ac;


そして、将来、すべてのソースへの作業と配信は、抽象化iac、iap、impでのみ実行されます。

結果として何が起こったのか



プロジェクトでシングルトンを使用せずに、便利な継ぎ目を作成します。将来、必要に応じて抽象化を置き換えることができます。 これで、抽象モックを使用して、あるクラスのサウンドを再現するためのテストを作成できます。 



 IAudioPlayer mock = Substitute.For<IAudioPlayer >(); var testClass = new Class(mock);


クラスへのアクセスはインターフェースによって制限されており、それ以上は何もできません（誤ったaudioCodeでabusを考慮しない場合）。 名前空間HexGrimmDev.Audio以外の不要な依存関係は拡大しません。 Mark Simonの推奨事項のように、すべての不必要な責任はクラスに移され、必要に応じてコンストラクターを介して移すことができます。 外部の論理接続はなく、モジュールをgit-submoduleとして配布できます。



すべての断熱材が同じように役立つわけではないことを理解していますが、この場合、余分な時間の継ぎ目を作るのにそれほど時間はかかりませんでした。 さらにインスピレーションを得るために、トピック「 Unityプロジェクトがコンソールで機能する理由 」についてのOleg Chumakovの講義を読むことを提案します。 」



また、コンストラクタを介してモジュールによるリンクを渡すことを強くお勧めします。これはもちろん消費者にとってより明確であり、さらに、規律を気にします。 そして最も重要なこととして、私は完全な普遍化を追求しないことを提案します。 このトピックに関する優れた講義があります 。「 コンポーネントのユニバーサル化の追求に夢中にならないように 」。
