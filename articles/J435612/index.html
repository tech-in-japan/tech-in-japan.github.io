ノイシュヴァンシュタイン城



2020年6月、正確に50年間の表形式のデータウェアハウス、または正式にはリレーショナルデータモデル。 これは公式文書です- 同じ有名な記事です。 エドガー・フランク・コッド博士に感謝します。 ところで、リレーショナルデータモデルは、過去100年間で最も重要なグローバルイノベーションのフォーブスリストに載っています。



一方、奇妙なことに、CoddはリレーショナルデータベースとSQL言語を彼の理論の歪んだ実装と見なしていました。 ガイドラインとして、彼はすべてのリレーショナルデータベース管理システムが満たす必要のある12のルール（実際には13のルール）を開発しました。 実際、今日、世界では少なくともCoddの「ルール0」を満たすDBMSが見つからないため、DBMSを100％リレーショナルと呼ぶことはできません:)例外があるかもしれません。



リレーショナルモデルはそれほど複雑ではなく、全体にわたって研究されています。 さらに深く研究されているかもしれません。 一方、2019年の今年はまた別の記念日を祝います-ちょうど10年前、＃NoSQLハッシュタグ、彼は後に「SQLだけでなく」Twitterに登場し、データベースモデル開発の実践に急速に浸透し始めました。



なぜそんなに長い紹介ですか？ プログラミングの世界はデータ（およびもちろんアルゴリズム）で構成されているという事実と、リレーショナルモデルの独占的地位は、より丁寧であるため、プログラマーの分裂意識につながります。 開発者の頭にある作業オブジェクト（OOPも合計ですか？）のため、これらのリスト、キュー、ツリー、ヒープ、辞書、スレッドなどはすべてテーブルではありません。



続けて、最新のDBMSのストレージアーキテクチャについて覚えていますか？ 率直に言って、彼らの正しい考えの誰もテーブルの形でデータを保存しません。 DBMS開発者は、ほとんどの場合、さまざまなBツリー（たとえば、PostrgeSQL）を使用します。 一方、DBMSを使用して格納するバリケードも、非テーブルで動作します。 そしてこれにより、プログラマーはセマンティックなギャップと不器用な中間データレイヤーを常につなぐようになります。 そして、それにより、内部の二分緊張を引き起こし、全身の不快感と不眠症をデバッグします。



つまり、要するに矛盾があります。データをタスクに適したオブジェクトにパックし、保存するときにいくつかのテーブルを処理する必要があります。



絶望？ いいえ:)しかし、オブジェクトリレーショナルマッピングについてはどうでしょうか。それはORMの一般的な話ですか？ この聖戦を仲間と共にイゴール・ブガエンコに任せましょう。 そして、 ボブおじさんによると 、前世紀のこの全体の物語は私たちを心配するべきではありません 。



もちろん、「バイト付きバッグ」（Robert Martin "Clean Architecture"）は、シリアル化してファイルにドロップしたり、他の適切なストリームにプッシュしたりできることに言及する価値があります。 しかし、第一に、言語ですぐに制限され、第二に、DBMSに格納することだけを心配するようになります。



バイトバッグを使用したこれらの浮き沈みには、優れた例外があります。インターシステムズCachéDBMS（および現在はインターシステムズIRISデータプラットフォーム）です。 これはおそらく、開発者から自明なことを隠さず、さらに先へ進む世界で唯一のDBMSです。つまり、「それを正しく保存する方法」を考えることから解放されます。 クラスがPersistent属を継続し、要点は帽子、 つまりグローバル （グローバル変数と混同しないでください！）にあると言うだけで十分です。



文字ストリームやバイナリストリームなど、あらゆる種類のデータを保存できます。 以下に簡単な例を示します。 



//       - // :      ,     ,     , ,   Class FW.Events Extends %Persistent { Property "My name" As %String; } //      //   «»  set haJS = ##class(FW.Events).%New() //   write haJS.%Id()


さらに、これは素晴らしいことです。CachéネイティブのObjectScriptだけでなく、Python、Java、JavaScript、C ++、C＃、Perlで直接オブジェクトを取得および保存することで、格納されたオブジェクトと通信できます。 そして、さらに恐ろしい:)。 SQLクエリを介して同じオブジェクトから直接情報を取得することもできます。また、オブジェクトで独自のメソッドを呼び出すこともできます。 より正確には、この場合のメソッド自体（およびマジックワードSqlProc）はストアドプロシージャになります。 すべての魔法はすでにCachéDBMSの内部にあります。

システム間CachéDBMSへの無料のテストアクセスを取得する方法

悪の言語が何と言っても、これは絶対に本当です！ :) ここで、 Cachéのフル機能バージョンのシングルユーザーをダウンロードしてインストールできます（無料で登録する必要があります）。 ビルドは、MacOS、Windows、およびLinuxで利用可能です。
Eclipseに基づくAtelier IDEを使用して、ObjectScriptコードを操作し、CachéDBMSサーバー（およびInterSystems IRISプラットフォーム）に直接アクセスするのが最も便利です。 すべてのダウンロードおよびインストール手順はこちらです。



より便利で馴染みのある人にとっては、快適でシンプルなVisual Studio Codeを使用でき、コミュニティが開発したObjectScriptプラグインで補完できます。



そして今、いくつかの実用的な例。 いくつかの関連オブジェクトを作成して、ObjectScriptとPythonでそれらを操作してみましょう。 他の言語との統合も同様に実装されます。 Pythonは、ObjectScriptとの「最大の親和性」の理由で選択されます-両方の言語はスクリプト可能で、OOPをサポートし、強い型付けはありません:)



例のアイデアについては、活発なハバロフスク（ハブロフと混同しないでください！）「シットアンドゴーパーティー」フレームワークのプロジェクトに目を向けます。 イデオロギーのソースコードはgithub.com/Hajsru/framework-weekendにあります。ソースコードはテキストが低くなっています。

macOSユーザーにとって重要なニュアンス。 Pythonのサポートモジュールを開始する場合、CachéをインストールしたディレクトリへのパスDYLD_LIBRARY_PATHを指定する必要があることを覚えておく必要があります。 たとえば、次のように：

export DYLD_LIBRARY_PATH = / application / Cache / bin：$ DYLD_LIBRARY_PATH

これは 、ドキュメントに具体的に示されています。

ObjectScriptでストアドクラスを作成します



行きましょう。 Cachéのクラスは非常に単純です。 IDEなしでも実行できます。Cachéプラットフォームインスタンスのポータルからクラスコードを直接コピーします（はい、CachéDBMSは単なるDBMSにはほど遠いです）：システムブラウザ>クラス>インポート（ネームスペースUSER）。



保存後、オブジェクトは対応するクラスの名前と一致する名前でグローブに表示されます。 また、Caché管理ポータルの[システムブラウザ]> [グローバル（ネームスペースUSER）]を確認します。 



 //    , ,      Class FW.Event Extends %Persistent { Property title as %String; Property description as %String; Property date as %Date; Property visitors as list of FW.Attendee; } //    / Class FW.Attendee Extends %Persistent { Property name As %String; }


PythonからCachéのオブジェクトにアクセスする



最初に、CachéDBMSデータベースに接続します。 ドキュメンテーションのように繰り返します 。

役に立つ有用な事実。 無料で、CachéDBMSのトレーニングバージョンでもあります。これにより、完全に機能するバージョンで利用可能なすべてを実行できますが、アクティブな接続は2つしか許可されません。 したがって、同時に、IDEからの接続を維持し、サーバーと対話するために他のコードを実行しようとすることはできません。 最も簡単な解決策は、Pythonコードの実行中にIDEを閉じることです。 
 #   Caché    Python3 import intersys.pythonbind3 #    conn = intersys.pythonbind3.connection() conn.connect_now("localhost[1972]:USER","_SYSTEM","SYS", None) #    print ("conn = %d " % conn.handle) #     database = intersys.pythonbind3.database(conn)


そして、ITイベントとその参加者のオブジェクトデータベースを作成します。 とても簡単です。 1つ目は、イベント参加者に関する情報を登録および保存するためのクラスを作成することです。 クラスを単純化するため、メンバー名のみ。 



 #         class Attendee: #        def __init__ (self): self.att = database.create_new("FW.Attendee", None) #             id def new (self, name): self.att.set("name", name) self.att.run_obj_method("%Save",[]) #    id     def use (self, id): self.att = database.openid("FW.Attendee",str(id),-1,-1) #       def clean (self): id = self.att.run_obj_method("%Id",[]) self.att.run_obj_method("%DeleteId", [id])


ご覧のように、Cachéのオブジェクトのフィールドにアクセスするメソッドに既製のラッパー関数を使用します。setパラメータで、パッケージとクラスの名前とともに引用符とopenidでプロパティ名を渡します。 同様のget関数については、以下の例があります。 先祖からクラスによって継承されたメソッドを含む他のメソッドにアクセスするには、必要に応じてメソッド名と呼び出しパラメーターでrun_obj_method（）関数を使用します。



行の中で最も重要な魔法：self.att.run_obj_method（ "％Save"、[]）

これにより、ユビキタスで見苦しいORMなどの追加のライブラリやフレームワークを使用せずに、オブジェクトを直接参照して保存することができます。



さらに、ObjectScriptのオブジェクト指向の性質とそのクラスのメソッド（この例では実行しませんでした）を考えると、Persistentクラスとその祖先から継承されたメソッドセット全体へのPythonアクセスからボーナスが得られます。 完全なリストは次のとおりです。



最初のメンバーを作成します。 



 att = Attendee() att.new("")


このコードを実行すると、FW.AttendeeDという名前のグローバルがデータベースに表示され、スクリーンショットのように保存したばかりのオブジェクトの内容が表示されます。







保存後、このオブジェクトは独自のID（番号1）を持ちます。 したがって、このIDを使用してプログラムに読み込むことができます。 



 att = Attendee() att.use(1) print (att.att.get("name"))


そして、必要に応じて、再びidを知って、参加者データベースからオブジェクトを削除できます。 



 att = Attendee() att.use(1) att.clean()


この例を実行した後、オブジェクトに関するレコードがグローバルに表示されなくなることを確認してください。 ダウンロードされたデータは、プログラムが終了するまでオブジェクトの「メモリ内」に残ります。



次のステップに進みましょう。 実際のイベントレコードを作成します。 



 #        class Event: #        def __init__ (self): self.event = database.create_new("FW.Event", None) #            id def new (self, title, desc, date): self.event.set("title", title) self.event.set("description", desc) self.event.set("date", date) self.event.run_obj_method("%Save",[]) #    id    def use (self, id): self.event = database.openid("FW.Event",str(id),-1,-1) #       def addAttendee (self, att): eventAtt = self.event.get("visitors") eventAtt.run_obj_method("Insert", [att]) self.event.set("visitors", eventAtt) self.event.run_obj_method("%Save",[]) #      def clean (self): id = self.event.run_obj_method("%Id",[]) self.event.run_obj_method("%DeleteId", [id])


クラス構造は、参加者の上記のクラスとほぼ同じです。 最も重要なことは、このイベントaddAttendee（att）の参加者のリストに参加者を追加する方法が登場したことです。



新しいイベントに関するオブジェクトレコードを作成し、データベースに保存しようとします。 



 haJS = Event() haJS.new("haJS", " ", "2019-01-19")


次のような結果になるはずです（日付は自動的にObjectScript形式に変換され、Pythonオブジェクトに再度読み込まれると元の形式に戻ります）。







参加者をイベントに追加することは残ります。 



 #     haJS = Event() haJS.use(1) #    att = Attendee() att.new("") #      haJS.addAttendee(att.att)


したがって、これらの例は、データモデルとその表形式のストレージスキームについて同時に考える必要がないことを示しています。 より明確なツールをタスクに使用できます。
