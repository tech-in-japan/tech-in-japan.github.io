小麦粉：JMS vs RMI。









サービスバスとモンスターの統合は、Camelがサポートする多くのコンポーネントの1つを使用して実装できます。 上記のタスクに基づいて、要件を形成しました。通信は安定し、メッセージ配信を保証する必要があります。 最初の2つは同期RMIとHTTPで、1つは非同期JMSでした。 3つのうち、Javaプロジェクトに適した最も単純な2つのバージョン、JMS、RMIに決めました。 JMS（Javaメッセージサービス）は、メッセージを送信するための標準であり、メッセージの送信、受信、作成、表示のルールを規制します。 2番目のRMI（Remote Method Invocation）は、リモートプロシージャを呼び出すためのプログラミングインターフェイスであり、あるJVMのメソッドを別のJVMのコンテキストで呼び出すことができます。 標準のリモート呼び出し手順には、Javaオブジェクトのパックと受け渡しが含まれます。 公平に言うと、JMSはトランスポートであり、RMIの不可欠な部分になる可能性があるため、JMSとRMIの対比は正しくないことに注意してください。 標準のRMI実装とJMS実装であるActiveMQを比較します。 以前は、RMIを使用して2つのアプリケーションを統合しました。 なぜ彼は選ばれたのですか？ 両方のアプリケーションがJavaである場合、RMIほど簡単なものはありません。 それを使用するには、インターフェースを記述し、このインターフェースを実装するオブジェクトを登録するだけで十分です。 しかし、アプリケーション間で大量のデータを転送するときにRMIで発生する問題を解決する機会がありました。メモリが詰まり、アプリケーションが「スタック」しました。 この問題の解決策を探し、JavaOne JVM開発者と議論しました。 仮想マシンのガベージコレクターと分散コレクターは2つの異なるものであることが判明しました。 それはすべて、標準のガベージコレクターではそのタイプを選択して最適なパラメーターを構成することが可能であるという事実にありますが、分散型ではそのような可能性はありませんでした。 他の違いについて話すと、RMIはJVM上で実行されるアプリケーションへの統合を制限していましたが、JMSはそうではありませんでした。 記載された困難に加えて、RMIを放棄し、代替ソリューションを使用するという新しいことを学びたいという要望がありました。



キャメルの最初のプロトタイプ



プロトタイプの作成に戻りましょう。 サービスバスの最初の実用的なタスクはこれでした。ユーザーがデータをアップロードするプロセスを開始し、システムがそれを準備し、サービスバスに送信します。 すべてのデータ配信作業は彼女にかかっています。 この図は、エンタープライズアプリケーション統合パターン（EIP）のシンボル内の問題ステートメントの例を示しています。



サービスバスは、JMSチャネルからのメッセージの受信、変換、およびHTTP経由の送信を組み合わせます。 図にマークされているJMSチャネルおよびHTMLメッセージ送信チャネルは、Camel JMSおよびJettyコンポーネントを使用して実装されることになっています。 データ変換プロセスは、Javaおよび/またはテンプレートエンジン（VM（Apache Velocity）など）を使用して実装できます。 提案されたデータ転送スキームは、Java DSLで1行で実装されます。 例： 

from("jms:queue:se.export") .setHeader(Exchange.HTTP_METHOD,constant(org.apache.camel.component.http.HttpMethods.POST)) .process( new JmsToHttpMessageConvertor() ) .inOnly("jetty:{{externalsystem.uri}}");


上記の例は、Java DSLルートを示しています。 ルートは、メッセージ送信ルートの説明です。 Camelには、Java DSLとXML DSLの2種類の記述があります。 このようなルートの特性は、開始点と1つ以上の終了点であり、それぞれfromとtokenで示されます。 ルートは、開始点から終了点までのメッセージパスを記述します。 エンドポイントが順番に示されている場合、メッセージは最初のエンドポイントに送信され、サービスバスは応答を待ってから、次のポイントに送信します。 目的のエンドポイントを選択するルート（ Dynamic Router ）、または複数のポイントにメッセージを一度に送信するルート（ Recipient List ）が存在する場合があります。 fromおよびtokensパラメーターは、URIを含む文字列です。 URIは、Camelコンポーネントの名前、リソース識別子、および接続パラメーターで構成される3つのパラメーターです。 例を見てみましょう： 

 from(“jms:queue:se.export?timeToLife=10000”)


これは、JMSコンポーネントを使用するエントリポイントの説明です。 JMSコンポーネントは、キューからデータを取得する機能を提供します：se.exportリソース。 キューはメッセージチャネルのタイプであり、キューまたはトピックのいずれかです。 次に、チャネル名「se.export」が来ます。 この名前のキューは、メッセージブローカーによって作成されます。 URIの最後の部分であるエンドポイントパラメータ「timeToLife = 10000」は、パケットのライフタイムが10秒であることを示します。

この例から、データの転送をどのように編成するかが明確になりました。次の記事では、より実際のコードと例があります。

そこで、データ転送の問題を解決し、Camelで構成され、実装の準備がほぼ整った統合バスのプロトタイプを作成しました。 正しく便利な構成の問題を解決するために残った。



プロトタイプのセットアップ



実際のアドバイスや実装例を見つけるのは難しいため、このトピックには非常に感銘を受けました。

スタンドの構造は次のとおりです。

各開発者はソフトウェアの独自のコピーを所有しており、それを起動する前に完全に構​​成します。 手動テスト用に2つのテストベンチがあり、もちろん動作するシステムがあります。 問題の説明は次のとおりです。

  • 各スタンドの微調整は、システム構成ファイル内のパラメーターの最小数を編集するだけで行う必要がありました。
  • すべてのスタンドで大規模なカスタマイズ（新機能の追加）がシームレスに実行されました

このタスクには多くの困難があります。CamelはJMSブローカーと接続されているため、異なるスタンドまたは異なるメッセージブローカーに異なるチャネルを使用する必要があります。 バスに統合されたActiveMQブローカーを起動することにより、最も簡単な方法を実行しました。 この場合、異なるサーバーに接続設定を提供することは残ります。



キャメル設定でパラメーターを使用する例に移りましょう。

  1. camel-config.xmlの例 

     <bean class="org.springframework.beans.factory.config.PropertyPlaceholderConfigurer"> <property name="locations"> <list> <value>classpath:system.properties</value> <value>classpath:smtp.properties</value> </list> </property> </bean>


    この例では、Springの2つの構成ファイルを設定します。
  2. 前の例の2つのファイルで指定された設定を使用する例。 

     <bean id="someServiceStartPolicy" class="org.apache.camel.routepolicy.quartz.CronScheduledRoutePolicy"> <property name="routeStartTime" value="${config.someService.routeStartTime}"/> <property name="routeStopTime" value="${config.someService.routeStopTime}"/> </bean>


    プロパティ自体は、「$ {config.someService.routeStartTime}」行で設定されます

  3. 複数の設定ファイルがCamelコンテキストに転送される例 

     <camelContext id="rootCamelRoute" xmlns="http://camel.apache.org/schema/spring"> <propertyPlaceholder id="properties" location="smtp.properties, system.properties"/> … </propertyPlaceholder> </camelContext>


  4. Java DSLのCamelルートでパラメーターを使用する例 

     from("jms:queue:email.send") .setHeader("to", simple("${headers.email}")) .setHeader("from", simple("${properties:config.smtp.from}")) .to("{{config.smtps.server}}?username={{config.smtps.user}}&password={{config.smtps.password}}&contentType=text/html&consumer.delay=60000")


    ここでは、パラメータをアドレス指定するいくつかの方法が一度に使用されます。 ここにあります：

    -単純な方言では、文字列「$ {properties：config.smtp.to}」

    -文字列「{{config.smtps.server}}」を持つエンドポイントURI

    パラメーター名は任意です。行は上記の例から取られています。





本当の問題を想像してみましょう。

サービスバスを介してSMTPサーバーにレターを送信するサービスがあります。 このようなサービスは、稼働中のシステムのユーザーにメッセージを送信し、テストシステムの場合、すべてのレターを1つのメールボックスに送信する必要があります。

次に、このタスクは、テストシステムと実稼働システムで実行されるいくつかのルートに異なるロジックを追加するように変換されます。



キャメルルートのパラメーターを使用して、このような問題を解決する方法の例を次に示します。 

 from("jms:queue:event.recoverypass") .setHeader("to", isDebug() ? simple("${properties:config.smtp.to}") : simple("${headers.email}")) .setHeader("from", simple("${properties:config.smtp.from}")) … // some other headers .choice() .when( header("password").isNotNull() ) .setHeader("subject", simple("${properties:config.passwordNotify}")) .to("velocity:vm/email/newPasswordNotify.vm") .otherwise() .setHeader("subject", simple("${properties:config.recoverypass}")) .to("velocity:vm/email/recoveryPassword.vm") .end() .to("{{config.smtps.server}}?username={{config.smtps.user}}&password={{config.smtps.password}}&contentType=text/html&consumer.delay=60000")


上記のサンプルコードは、パスワード回復サービスに必要です。 次のロジックが実装されます：ユーザーがパスワードのリセットボタンをクリックし、新しいパスワードを生成するための一時リンクを含む電子メールがユーザーに送信され、ユーザーがリンクをクリックし、同じルートがユーザーに新しいパスワードをメールで送信します。 それはおそらく最初の部分が完了するべき場所であり、それは在庫を取るためだけに残っています。
