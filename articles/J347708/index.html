FacebookまたはTelegram？ ウクライナの.NETコアコミュニティの歴史

こんにちは、Habr！ ある日、.NET Coreに強い関心を持つ人がブログを始めることにしました。 そして、「どのリソースを使用すればよいですか？」という疑問が生じました。 既存のFacebookとTelegramは、出版物の構造化にはまったく不適切でした。 その結果、同じ.Net Coreを使用して、ブラックジャックと構造化を使用してサイトを作成することが決定されました。







著者に発言権を与えます。



.NET Coreの発表以来、私はこの技術を注意深く監視し始めました。 最初のバージョンがリリースされたとき、私のプロジェクトのいくつかはこのプラットフォームに移されました。 結果は満足でした-すべてが安定して機能し、フレームワークのクラシックバージョンよりもさらに高速でした。 しばらくして、.NET Coreの開発に興味のあるプログラマのインターネットコミュニティを組織することにしました。 コミュニケーションの最初のプラットフォームはFacebookでした。



しばらくして、Telegramでチャンネルを作成することにしました。そこで、.NET Coreの開発に関連するさまざまなニュースへのリンクを公開し始めました。 約6か月後、私はTelegramはニュースレターには非常に便利ですが、出版物の検索と構造化には絶対に適さないと結論付けました。 時間が経つにつれて、チャンネルのニュースが十分に頻繁に公開される場合、フィードでは少なくとも先週読んだ出版物をすばやく見つけることはほとんど不可能です。



Facebookグループもこのタスクであまりうまくいっていませんでした。 したがって、情報を構造化し、出版物のナビゲーションをより便利にするために、最終的に// devdigestと呼ばれる集約サイトを作成することにしました。



当然、.NET Coreは開発プラットフォームとして採用されました。 プロジェクト自体はGitHubの公開リポジトリでホストされており、誰でもその内部構造とソースコードを知ることができます。



この記事で// devdigestと話す方法について説明します。



Webページ用のウィジェットを作成する



サイトの主なタスクは、すべての出版物を便利な形式で保存することです。 サンプルでは、​​Facebook外部リンクの説明がどのように生成されるかを取り上げました。リンクごとに、タイトル、写真、短い説明から小さなウィジェットが形成されます。



リンクの説明を作成するために、小さなライブラリを作成しました（そのソースコードはGitHubで入手でき、ライブラリ自体はnuget.orgで公開されています ）。 ライブラリのロジックは非常にシンプルで、リンクを受け取った後、 HtmlAgilityPackを使用してドキュメントのドキュメントモデル（同じDOMツリー ）が形成され、このモデルから抽出されます。



  • 標準のメタタグ（タイトル、キーワード、説明）、
  • プロトコルのOpenGraphメタタグ（このプロトコルは、たとえばFacebookを使用します）：og：title、og：image、og：description
  • 画像リンク
  • 次に、受信したデータから、メタデータクラスのインスタンスが形成され、Webアプリケーションが動作します。

ホスティング



プロジェクトを配置する場所には、商用プロジェクトに使用するサーバーの1つ、またはMicrosoft Azureクラウドの2つのオプションがありました。 クラウド配置オプションは、いくつかの理由で望ましいものでした。



  • 連続配信を構成する機能-ご存知のように、Azure Web Appsでは、GitHub、Bitbucket、Visual Studio Team Services、独自のgitリポジトリ、さらにはDropboxやOneDriveからも自動発行を構成できます。 印象的なリストですね。 devdigestはもともとオープンソースプロジェクトとして考案され、コミュニティの公開リポジトリに配置されていたため、この場合の継続的配信設定は文字通り10秒になります。
  • Applications Insight'sとの統合-アプリケーションのすべてのアクティビティを追跡し、さまざまなメトリックを収集し、何か問題が発生した場合にレターを送信できるサービス


認知サービス



時間が経つにつれて、ウェブサイトの出版物のメインフィードに英語のみを表示することが決定されました。それらのほとんどと、異なる言語でのコンテンツの交互表示がかなり奇妙に見えたためです。 もちろん、このようなフィルタリングを実装するために、サイト管理者のインターフェースで、出版言語を選択して別のコンボボックスを作成することができました。 しかし、少なくともそれは平凡で面白くありません。 特に、現在利用可能な既製の機械学習ベースのサービスの数を考慮してください。 そのため、Microsoft Azure Cognitive Servicesの一部であるText Analytics APIを使用することにしました。 Text Analytics APIを使用すると、テキストからキーワードを抽出したり、送信されたテキストの言語を決定したりできます。 これはまさに私が必要としたものです。



サービスの価格は非常に人道的です-ご覧のとおり、無料の料金プランでさえ、テキスト分析のための非常に豊富な機会を提供します：キーフレーズの抽出とテキストの言語の決定。 これはまさに私たちが必要とするものです！
