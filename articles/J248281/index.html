CRMとは



まず、CRMの概念を理解する価値があります。 CRMは、Customer Relationship Management（顧客関係管理システム）の略語です。 CRMの定義には多くの定義があります。 ここでは、私にとって最も正確なものを提供します。 ギネスの記事「ビールの理想的なパイント」で先ほど引用しました。 優れたCRMソリューションの事例。

顧客管理と呼ばれることの多いCRMシステムは、ビジネスアプローチです。 その目標は、慎重に選択された顧客との関係を作成、開発、強化し、顧客の利益を増やし、企業の利益を増やし、投資家の利益を最大化することです。
（Adrian Payne著「CRM Handbook。The Path to Improving Customer Management」から）。



したがって、CRMは、顧客との関係を構築、開発、強化するという目標を達成することを目的とした、さまざまなツール、イベント、情報技術です。 Zoho CRMは、この複雑なITコンポーネントの一部です。 Zoho CRMは、顧客に彼が最もよく知っていること、つまり営業を行う機会を提供します。 Zohoはワークフローの自動化を担当します。



ライセンス



Zoho CRMはsaasソリューションです。つまり、プログラムの作業はWebブラウザーを介してのみ実行され、ユーザーのコンピューターにはプログラムはインストールされません。



UPD： リンクを使用して登録すると、Zoho CRMを無料で試すことができます。



Zoho CRMの使用を開始するには、ライセンスを購入する必要があります。 1か月または1年間システムにアクセスするには、ライセンスが必要です。 各ライセンスは、たとえばコンピューター、ラップトップ、スマートフォンなどから、必要な回数だけログインとパスワードで接続する権利を与えます。 ライセンスはユーザーによって行われます。 1つのライセンスは、CRMに登録された1人のユーザーに対応します。 1人のユーザーと同じ数の同時セッションを実行できます。 これは、購入する必要があるライセンスの数を判断するために理解することが重要です。 たとえば、CRMシステムでレポートのみを使用するマネージャーが3人いる場合、3つのライセンスを購入する必要はなく、1人で十分です。 ただし、Zoho CRMに記録されるユーザーの数はライセンスの数を超えることはできません。これは理解する必要があります。 つまり、新しいユーザーにアクセスを許可するには、別のユーザーを非アクティブ化するか、新しいライセンスを購入する必要があります。

コンポーネントシステム



管理者とユーザー


ユーザーは、システムにアクセスできるユーザーです。 一意のログイン名とパスワードを使用して、ユーザーは異なるデバイスから何度でもZoho CRMにログインできます。 ユーザーには2つのタイプがあります。

  • 管理者


管理者は、Zoho CRMへのフルアクセス権を持ち、アクセス権を設定したり、他のユーザーを制御したり、モジュールやシステムの外観を設定したり、プラグインやサードパーティシステムを接続したりできます。 管理者権限には、システムを構成するユーザーが必要です。 システムへのアクセスを決して失わないことが非常に重要です。 管理者が誤って管理者権限を設定せず、システムにアクセスできないことがあります。 これはZoho CRMの場合には当てはまりません。ここでは、管理者がシステムへのフルアクセスを削除することはできません。 Zohoには常に1人の管理者がいます。 システムがこれを制御します。

  • 他のユーザー。
システムの使用に制限がある場合とない場合があります。 コミットされたすべてのアクションはユーザーに関連付けられていることを理解する必要があります。これは、データの記録と分析、プロセスの予測、従業員に関するレポートの生成、およびその有効性の判断に非常に便利です。



モジュール


モジュールは、Zoho CRMシステムでのすべての作業の実行に不可欠な要素です。 システムを構成するレンガのようなものです。 モジュールはシステムにタブとして表示され、その外観はカスタマイズ可能です。 モジュールにはフィールドがあります。 フィールドには、「名」、「姓」、「電話番号」、「電子メール」、「ステータス」、「取引ステージ」などが表示されます。 Zohoでは、独自のフィールドを追加できます。 フィールドには、可能な限り幅広い構成と編成があります。 しかし、モジュールに戻ります。 Zoho CRMには2種類のモジュールがあります：

  • プリセット（基本）モジュール。
これらは、システム自体によってインストールされるモジュールであり、システムを操作する独自のロジックを備えています。 基本モジュールの機能には制限があります。基本モジュールの中には、変更できないものもあります。 基本モジュールの中には、たとえば、リード（リード）、連絡先（コンタクト）、アカウント（取引先）、タスク（タスク）などがあります。

  • カスタムモジュール
これらはユーザーが作成したモジュールです。 編集制限なしで自由に変更できます。 モジュールを作成し、タブの形式で表示できます（たとえば、クライアントに送信されたすべてのSMSを含むSMSメッセージ用の独自のモジュールを作成しました）。 モジュールは、Zoho CRMと統合する特別なアプリケーションZoho Creatorで作成されますが、これについては後で詳しく説明します。 システム内に作成できるモジュールの数に制限はありません。

4つの基本モジュール



リード


-これはあなたの会社に興味を示した潜在的な顧客です。 これは仕事の非常に最初の段階です。 これはまだクライアントではありませんが、彼との作業はすでに進行中です。 連絡を取るには早すぎます。 リードはさまざまなソースから生成されます。 ソースのリストには、展示会、電子メールキャンペーン、Webフォーム、コールドコール、ダイレクトマーケティングなどが含まれます。 リードは、手動または自動で作成できます（たとえば、埋め込みWebフォームを介してサイトから申請する場合、または未定義の番号から呼び出す場合）。 このリード（たとえば、「リードを呼び出す」）にタスクを自動的に添付し、このタスクを特定のマネージャーに移動させることができます。 リーダーとの作業が行われた後、連絡先または取引先および潜在的なトランザクションに変換できます。



連絡先


-これは、ユーザーが直接やり取りする人です。 連絡先はシステム全体のコアモジュールであり、クライアントとのやり取りを担当します。 連絡先は、あらゆるビジネスの強みの源です。 したがって、クライアントに関する情報が多いほど、ビジネスは成功します。 Zoho CRMの連絡先モジュールは、必要な連絡先情報をすべて1つのウィンドウに保存する強力でパーソナライズされた方法です。 連絡先モジュールには、クライアントについて知る必要があるすべてのものが含まれています。



  • 電話番号
  • メール 手紙
  • タスク
  • リマインダー
  • 予定リスト
  • ドキュメント
  • その他


どのように機能しますか？ 連絡先がユーザーを呼び出していると想像してください。 連絡先の名前はユーザーの画面に表示され、ユーザーはこの連絡先のカードを開くことができます。クライアントのすべての情報が1つのウィンドウに表示されます。最後の注文、最近のやり取り、連絡先のタスクを開く したがって、会話中のユーザーは、この連絡先との作業のロジックを構築できます。 着信コールが完了すると、システムはこのコールをインタラクションのリストに保存し、リマインダー付きのタスクの作成を提案します。



取引先（アカウント）


組織、連絡先に関連付けられている会社です。 1つの取引相手が複数の連絡先の対象となる場合があります。 基本的に、クライアントとのすべての作業は連絡先モジュールで実行されますが、会社全体の作業をグループ化するには、カウンターパーティモジュールが必要です。 Counterpartiesモジュールもカスタムフィールドで構成されていますが、ここでは、連絡先とは異なり、特定の組織の法的情報が含まれています。



潜在的な取引


-これは、一般的なトランザクションの制御を担当し、販売チェーン内の特定のトランザクションを追跡するモジュールです。 ここでは、取引がどの段階にあるか、取引に勝つ確率、取引履歴、取引額、責任などを記録できます。 他のモジュールと同様に、ここのフィールドも構成されます。



Zoho CRMでトランザクションを操作する



ここで、Zoho CRMでの段階的なトランザクションがどのように見えるかを説明します。



  1. 典型的な販売プロセスは、リード生成キャンペーンから始まります。 これは、電子メールキャンペーン、展示会、Yandexでの広告、またはその他のプロモーションシステムです。
  2. Zoho CRMでは、リードは受信したデータに基づいて自動または手動で作成されます。 これは、サイト上の完成したアプリケーション、受信メールメッセージ、または名刺にすることができます。 Zoho CRMでルールを作成して、リードを作成するときに挨拶付きの電子メールメッセージを受信できるようにすることができます。
  3. リードは異なるユーザーに割り当てられます（各リードには責任のあるユーザーがいます）。 ユーザーはリードを使用してタスクを取得します。 これは、手動または自動で実行できます（対応するルールが綴られている場合）。 タスクは、トピック、期限、実行する必要のある操作の種類、そしてもちろん、このタスクが関連付けられているリードを示します。
  4. ユーザーはリードを操作します。電話をかける、電子メールまたはSMSを送信する、会議を開催する、システム内のすべてのやり取りを記録する。
  5. 次に、リードを認定する必要があります;動作していないリードはシステムから削除する必要があります。
  6. インタラクションが行われた後、リードは関心を示し、その潜在性を明らかにしました-取引相手、連絡先、潜在的取引に変換されます。 これは、1つのキーを押すことで発生します。その後、クライアントはLeadモジュールから選択したモジュールに移動します。
  7. 取引が成立するまで作業が続行されます。 または失われていません。


ご覧のとおり、ワークフローは非常に透過的であり、幅広い自動化機能を備えています。 次に、プロジェクトで最も需要の高いCRMの機能に進みます。



Zoho CRMの機能と機能



データのインポート


データのインポートには特に注意を払いたいです。 Zoho CRMへのデータのインポートは難しくありません。データのインポートに制限はありません。 プロセスは非常に簡単で高速です。 XLSまたはCSV形式でファイルをアップロードし、フィールドをマップしてデータをロードします。 Zoho CRMのフィールド名は、テーブルの列見出しです。 操作をいつでも実行してキャンセルできると非常に便利です。このインポートにあったすべてのデータはシステムから削除されます。



API


APIは、CRMシステムを選択する際に最初に注意したことです。 APIとは何ですか？ これは、あるソフトウェアを別のソフトウェアと統合するためのツールです。 Zoho CRMのAPIの説明が本当に好きで、ドキュメント、彼らが提供するサービスがとても気に入りました。 CRMシステムはそれ自体のものではなく、単一の情報フィールド内に存在する必要があり、十分に統合されている必要があります。 また、システム自体は優れているが統合が不十分な場合、このシステムは機能しません。 Zoho CRM APIは非常に使いやすく、さまざまなサービスとの優れた統合を提供します。

APIには一定の制限があります 。 選択したZoho CRMの料金に応じて、アップロードまたはダウンロードできるデータの量は異なります。



エンタープライズバージョンで使用できる最小値は、組織ごとに1日あたり4000要求です。

最大は組織あたり1日あたり25,000リクエスト、またはユーザーあたり500リクエストであり、制限はそれよりも低いレートです。



同時に1つのリクエストが可能です。

  • 最大200エントリの追加または読み取り
  • 最大100エントリを削除または更新します


つまり、3人のユーザーがいる場合、3 * 500 = 1500リクエストを作成できます。 1500は最小の4000よりも少ないため、4000のクエリを作成できます。

10人のユーザーがいる場合、10 * 500 = 5000リクエストを作成できます。 5000は、4000に等しい最小値以上、25000に等しい最大値未満です-5000リクエストを作成できます。

最大で、50人を超えるユーザー数でのみ休息できます。
