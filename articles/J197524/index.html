Symfony CMF パート1、データストレージ

序文の代わりに



私は2年間Yiiでプログラミングをしており、最近Symfonyフレームワーク2を見つめ始めました。一部、よく考え抜かれたアーキテクチャ、コンポーネントの弱い接続性、ビルドされたアプリケーションの柔軟性に魅了されています。 新しいフレームワークのメインデバイスを見つけた直後、その上にCMSを構築することは可能か、完成したものを使用することさえ可能だろうかと考えました。



ボックスソリューションはまだ発明されていませんが、どういうわけか私はSymfony CMFプロジェクトのWebサイトをさまようと、Drupalの設計を引き出すために組立ラインで作業していたときに遭遇した問題を解決するための体系的なアプローチにその場でぶつかりました。 Habréには、CMFに関する出版物はなく、プロジェクト自体はまだ非常に粗雑ですが、将来はすべてがおもしろそうに見えますが、場所によっては不満があります。



symfony cmf



Symfony CMFプロジェクトは、Symfony Framework 2を使用するすべての人のCMSに固有の機能の開発を簡素化するように設計されています。

プロジェクトの主な機能：



  • コンポーネントの接続性が悪い
  • 拡張性
  • 便利さ
  • テスタビリティ


CMFという言葉に注目する価値があります-プロジェクト自体はCMSではなく、 フレームワークです。 すべてのコンポーネントが互いに密接に結び付けられているCMSとは異なり、Symfony CMFでは次のことができます。



  • 好きなものを使う
  • 気に入らないものを置き換える
  • 不要なものを無視する


つまり、ターンキーターンキーアプリケーションではなく、一連のモジュール式開発ツールが提供されていますが、CMS機能を提供する基本的なバンドルはすでに開発されています。



なぜ別のCMFなのか？



有料（1C-Bitrix、UMI）と無料（Drupal、MODx、Wordpress、Joomla）の両方の既製品が市場に出回っているのは秘密ではありません。 したがって、 CMS / CMFの碑文を見ると、疑問が生じるのは当然です。なぜ別のCMSを作成する必要があるのですか？ 彼らはすでにとてもいっぱいです。

そして、私は絶対に同意します。 ユーザーとして。



CMSは本当に1ダースです。 しかし、開発者として、私はしばしば汗、血、涙を流し、それらからもっと何かを得ようとしました。それは基本システムとサードパーティの拡張機能の作者によって定められたものではありませんでした。



十分に考えられていないアーキテクチャのため、既成のソリューションを使用する場合、次のことに対処する必要があります。



  • ロジック、構成、コンテンツ、およびプレゼンテーションの明確な分離の欠如。 Drupalモジュールを思い出すのに十分です-ファイルの束、理解できないグローバル関数のハッシュ、フックなど。 ところで、この問題を議論する良い記事
  • 古いバージョンからの多くのレガシーコードが残っています。 開発者は時々これを修正しようと試み、カーネルやその他の喜びを書き直すことを約束しますが、新しい（書き直された）バージョンが「使用可能」段階に達するまで、多くの時間が経過します。
  • 多くの場合、システムには開発、テストなどの概念はなく、展開用のツールもありません
  • キャッシングの問題。 どこかにない、どこかにあるが、十分な柔軟性を提供していない、障害に問題がある、または単に保存しないなど
  • 大規模（この場合は相対的な用語）データボリュームでの低パフォーマンス
  • 独自のコンポーネントの作成や既存のコンポーネントのオーバーライドが困難
  • 事前定義されたデータ型から選択する必要があります。リレーショナルDBMSの上にEAVリポジトリがあるか、さらに悪い
  • CMSの作者が発明した不快なテンプレートバイク...
  • ...そしてこれはすべてNIH症候群の結果として。


これらのシステムの開発者は欠点を認識しており、申し立てを拒否しませんが、現時点ではこれらすべての問題を解決することは不可能です。 しかし、私たちはまったく誓いません。ユーザーの利便性のためにCMSが解決すべき多くの問題を定式化し、Symfony CMFでこれらの問題がどのように解決されるかを確認します。 問題は次のとおりです。



  • データストレージ
  • テンプレートシステム
  • ルーティング、CNC、およびユーザーがこれをすべて制御する方法
  • メニュー設定
  • コンテンツ管理（ページ上の編集可能なブロック、ライブサイトでのフロントエンド編集、ファイルのアップロード）
  • 国際化
  • 良い管理者

データストレージの問題



CMSの概念の実際の解釈に基づいて、CMSの最も重要なコンポーネントはデータストレージであることが明らかになります。 さらに-CMSは、異なるプロパティを持つデータストレージを提供する必要があります 。 たとえば、BlogPostやNewsItemなどの資料の場合、共通のtitle



フィールドとbody



フィールドを作成できます。そうすると、違いが生じます。ニュースに写真を添付する必要がある場合があります。



オンラインストアを想像してください。 データベースには何が保存されていますか？ 少なくとも、製品の説明と注文履歴。 1つ目とは異なり、2つ目のストレージスキームを設計する方がはるかに簡単ですが、両者なしでは両方が存在できないことは明らかです。 したがって、次の要件： CMSは、CMS内とシステムの他の部分の両方のコンテンツにリンクできる必要があります 。



サイト上のコンテンツ自体は、ほとんどの場合、ファイルシステムを幾分繰り返すツリー構造の形式で編成されています。 サイトの作成者は同時に、ニーズに応じてさまざまな方法でコンテンツを整理し、メニューや資料のアドレスを柔軟に構成したいと考えています。 したがって、 CMSはデータをツリー構造で提示 し、同時に複数の独立したツリーに対応できる必要があります 。



ユーザーがCMSで詰まらせる情報が、理想的な構造になっていることはほとんどありません。 場合によっては、1つ、別の、3番目、10番目のフィールドを追加する必要がある場合があります。CMSは、コンテンツに対して単一のスキームの使用を強制するべきではありません 。



大規模な組織では、ワンクリックで公開するのではなく、サイトに表示される前にいくつかの検証段階を経ることが珍しくありません。CMS はツリー間のコンテンツの移動とエクスポートをサポートする必要があります。 そして、物語のために、いつでも復元できるコンテンツのバージョンを保持しておくといいでしょう。



他の国や地域のユーザーについて覚えておく価値があります。 通常、Webサイト全体を別の言語に翻訳する必要はありませんが、 CMSは、異なる言語でコンテンツを表示する機能を提供し、指定されたルールに従ってオプションのフォールバックを提供する必要があります 。



コンテンツが多すぎる場合は、 全文検索 、 サブツリーへのアクセスを制御するためのルールを決定する機能、および複数の作成者によるドキュメントの公開プロセスの支援が必要になります（ワークフローはすべてのユーザーで異なります）。



コンテンツリポジトリ


1つの「筋肉」を取り除くことができないことが明らかになります。 リレーショナルデータベースは単純にそのようなタスクに対処できませんが、グラフ構造をフラットデータベースに格納できるマテリアライズドパスやネストセットなどのアルゴリズムがあります。 しかし、単一の実装が機能する場合でも、特定のエンジンに厳密に結び付けられている可能性が高く、これはすでに私たちの自由と柔軟性を奪っているので、すでに悪いことです。 RDBMSを非難する必要はありません-それらは完全に異なるタスクのために設計されており、弱く構造化された要素で構成されるツリーではなく、明確に記述されたデータが必要です。



ただし、動揺することはありません。結局、ブルジョア語を音訳すると、コンテンツリポジトリ、またはコンテンツリポジトリを長い間考えてきました。 リポジトリは、データを必要とするアプリケーションに関係なく、データの読み取り、書き込み、および取得にアクセスできるように設計されています。 実際、これは主にデータ処理の論理的な側面に重点を置いたデータウェアハウスです。



JCR-170


ドキュメント指向システムのデータストレージの問題は何年も前に発生したため、2000年代前半に、Day Software（つまりDavidNüscheler）の人々がJava Community Processを介してJavaのContent Repository API（JCR）仕様を受け入れる要求を提出しました。その後、仕様は番号JSR-283（2.0）、JSR-333（2.1、最終ドラフトは8月31日に完成）で行われましたが、これまでは最初のオプションへのリンクがより一般的でした。



仕様によれば、リポジトリは、階層データの保存、検索、検索を提供するオブジェクトデータベースです。 さらに、提供されているAPIを使用すると、データのバージョン管理、トランザクション、変更の追跡、XMLでのインポート/エクスポート、およびバイナリとメタデータの保存を使用できます。



このようなリポジトリは、プロパティを持つノードのツリーとして編成されます。 データはそれらに直接格納され、数値、文字列、または任意の長さのバイナリデータにすることができます。 ノードは、タイプに細分化され、子ノード、特定の動作特性、または単純に隣接ノードを参照できます（特別なプロパティと各ノードが持つ一意の識別子を使用）。



仕様の2番目のバージョンから、リポジトリはSQLクエリに応答できるようになります。これは、最初のエディションのXPathの対応よりも便利です。



そのような幸福の実現の鮮やかな例として、Javaで書かれたオープンソースのリポジトリであるApache Jackrabbitプロジェクトを選び出すことができます。 上記のすべての利点に加えて、このプロジェクト（2004年にJCR APIの初期実装として始まった）は、コンテンツへのアクセスを柔軟に制御できます。 クラスタリング、ロックメカニズムなどもありますが、今はあまり面白くないのでスキップしてください。

重要なプラス-私たちは自転車を発明しません（覚えているように、CMSのデータストレージの問題は既に解決されているため）。

Midgard2 PHPCR


Midgard2は、C、Python、およびPHPのバインダーを備えたオープンソースのコンテンツリポジトリです。



JCRとは少し異なる用語であるMidgard2は、 php5-midgard2拡張を使用してMidgard2 PHPCRを介してコンテンツにアクセスするための同じ機能を提供します。 libgda GNOMEライブラリの上に構築されたMidgard2は、リポジトリをホストできるリレーショナルデータベースの印象的なリストを保持しています。



私はすぐに軟膏のフライについて言います-PHP拡張はかなり少数のオペレーティングシステム用に構築されています：



  • Debian 7 Wheezyの下では、パッケージはまだ不安定なブランチにあります（当然、PHP-FPMをセグメンテーション違反に静かに落とします）。
  • CentOSの場合、パッケージはすべてのアーキテクチャで古くなっているか、すべてではありません（ただし、その場合、動作する可能性が非常に高く、手が届きませんでした）
  • Windows上のビルドは自然には存在しません（おそらく、「gnomes」はMidgard2自体のルートですが、4年前のリポジトリ内のファイルはPHP4でも表示されます）
  • Mac OSが不足しているため、Mac OSでテストすることはできませんでした（ただし、サイトによると、すべてはbrewで設定されています）。


一般的に、すべてがUbuntu Server 12.04に正常にインストールされ、パッケージはそこに新しく、何もクラッシュしません。



しかし、IRCでSymfony CMF開発者と話をしたところ、このバックエンドプロバイダーが数か月間壊れており、テストも無効になっていることが明らかになりました。 その理由は、Midgard2チームのどこかにありますが、Bergieは修正することを約束しました。







Midgard2 PHPCRをSymfony CMFで動作させることができませんでした。 たぶん他の誰かが成功するでしょう。 今ではなく、その後。



ジャッカロープ


名前でウサギのテーマ（ Jackrabbit 、 Jackalope ）を破り続けることで、Jackalopeは3種類のデータウェアハウスへのアクセスを提供します。



  • これはすでに私たちに知られていますApache Jackrabbit
  • Doctrine Database Abstraction Layer。これにより、サポートされているDBALエンジンを使用できます。 これは理論上、実際には、MySQL、PostreSQL、およびSQLiteのみがテストされています（他の何かを使用していますか？）。
  • MongoD （2年間更新されていない、破損または無関係である可能性が高い）


Jackalope（特にjackalope-jackrabbit）は非常に安定しており、機能の面で最も完全なPHPCR API実装としての使用が推奨されます。 彼女と一緒に働きます。 ただし、PHPCR APIの可用性と機能を確認するphpcr-api-testsもjackalope-doctine-dbalに含まれており、時間が経つにつれて追いつく可能性があります。



PHPCRの概要



そのため、JCR API標準に準拠したコンテンツリポジトリにアクセスするための（PHPに適合した）APIがあります。 このAPIのために、データウェアハウスからアプリケーションコードを抽象化するいくつかのライブラリが開発されました。



これまでのところ、2つの主要な質問が発生するはずであり、両方とも回答されます。



PHPCRを使用する場合



  • 階層ナビゲーション構造を使用する場合
  • 相互に関連するデータがある場合

PHPCR ODMは、バージョン管理と多言語主義という2つの非常に重要な機能を既に実装しています。 最初のものから始めましょう。



PHPCRには、simpleVersionableとversionableの2種類のバージョン管理があります。 単純なバージョン管理のために、チェックイン/チェックアウトの方法と変更の線形履歴が提供されます。 Chekinはノードの新しいバージョンを作成し、読み取り専用にします。何かを書き込むには、チェックアウトする必要があります。



これに加えて、完全なバージョン管理は、非線形バージョン履歴（PHPCR-ODMにはまだヘルパーメソッドがありません）およびラベル（Jackalopeがサポートを開始するとすぐに追加される予定です）をサポートします。各ノードについて、バージョンにラベルを追加できますが、各ノードの履歴全体にわたってラベルを2回繰り返すことはできません（つまり、別のバージョンをマークする場合は、まず古いバージョンから削除する必要があります）。
