C ++のAwesomium

Awesomiumは、Chromiumベースのブラウザをアプリケーションに統合するためのライブラリです。 Awesomiumの魅力は、ほぼすべてのタイプのアプリケーションに統合できることです（Unity3Dに基づいた3Dゲームへの統合の例があります）。開発者にとって幅広い機能を備えており、率直に言って、それを使って作業するのは素晴らしいことです。



しかし、残念なことに、ライブラリにはかなり控えめなドキュメントがあり、実際には例やコミュニティはありません（これはC ++での開発に適用され、アフィリエイトはより幸運です）。 さまざまなソースから少しずつ収集する必要がありましたが、他の人は一連の試行錯誤の後に理解するようになりました。



そして、いくつかの知識を蓄積したので、それらを共有することにしました。 私はこの道を行く最初の人でも最後の人でもないことを確信しています。



私はすぐにあなたに警告したい：多分私は誤解または誤解されたいくつかの場所で、あなたは厳密に判断するべきではなく、正しい道を指すだけで、私は非常に感謝します。



窓と窓のないAwesomium





まず第一に、Awesomiumを使用するアプリケーションには2種類あります。



  • オフスクリーン -ブラウザーウィンドウを表示せず（ページ上のスクリプトのすべてのレンダリングと作業はバックグラウンドで行われ、ユーザーには表示されません）、またはブラウザーの表示とともに、プログラマーはメッセージのレンダリングと処理に従事します
  • Windowed-Awesomiumはいくつかのウィンドウにしがみつき、その上に表示されます。 マウスとキーボードのメッセージの描画、処理はライブラリを使用して実行されます




1つ目は、ウィンドウのないブラウザで作業できるという点で興味深いものです。 コンソールアプリケーションを使用して、たとえばページのスクリーンショットを作成することもできます。 良い例があり、私は実際にそれを使用しなかったため、このタイプについては説明しません。 2番目のモード-Windowedに興味がありました。



Windowed Awesomiumは、ウィンドウアプリケーションでフルブラウザが必要な場合に使用されます。 この場合、ライブラリは主なもの、つまりユーザーインタラクションを処理します。 しかし、プログラムに数行追加するだけでフルブラウザが得られるとは思わないでください。そうではありません。 作業の大部分はプログラマーの肩にあります。たとえば、ファイルのアップロード/アップロードダイアログ（はい、基本的にすべてのダイアログ）、ポップアップメニュー、ドロップダウンリスト、新しいタブの作成などです。



ライブラリー構造





ライブラリの構造は非常に単純です（ただし、最初はまったく明らかではありませんでした）。



ブラウザのコアはコア（ WebCore ）です。 各アプリケーションはコアを1つだけ持つことができ、これによりいくつかの制限が課せられます。 たとえば、一部のタブで他のユーザーエージェントからキャンセルしたい場合、ユーザーエージェントの値はカーネル設定に保存されるため、成功しません。



階層の次はWebViewです。これはブラウザウィンドウです。 各WebViewを作成すると、個別のChromiumプロセスが起動します。 WebViewが不要になったら、クリーンアップすることを忘れないでください。そうしないと、WebCoreが破壊されるまでプロセスがハングします。



各WebViewはセッション（ WebSession ）を使用できます。 セッションはユーザーデータ（Cookie、キャッシュ、証明書、ローカルデータベースなど）です。 1つのWebSessionは異なるWebViewに接続できます。 WebSessionは、プロキシなどの重要なブラウザ設定（このセッションを使用するもの）も保存します。



さて、最後の（私が作業しなければならなかった）リスナーは、ブラウザーと対話するためのインターフェースです。 リスナーはいくつかあり、それぞれが特定の問題を解決します。 たとえば、WebViewListener :: Dialogにはダイアログボックスを操作するためのメソッドの実装が含まれ、WebViewListener :: Downloadにはファイルのダウンロードの進行状況を通知するためのメソッドが含まれます。



誕生





理論は終わりました。実践に移りましょう。 まず、カーネルモードとブラウザタブをウィンドウモードで作成および初期化するプロセスを検討します。



まず、カーネルを初期化する必要があります。 

WebCore* core = 0; WebConfig config; core = WebCore::Initialize(config);




次に、セッションを作成します（たとえば、セッション設定では、使用するプロキシを使用します） 

 WebPreferences prefs; prefs.proxy_config = WSLit("198.1.99.26:3128"); WebSession * session = core->CreateWebSession(WSLit(""), prefs);




そしてもちろん、WebViewオブジェクトを作成します。 

 WebView* view = core->CreateWebView(WIN_WIDTH, WIN_HEIGHT, session, kWebViewType_Window); view->set_parent_window(hwnd);




最後のステップでは、さらに詳しく説明します。



  • まず、ブラウザウィンドウの初期の幅と高さを指定する必要があることに注意してください。 後で変更できます（たとえば、親ウィンドウのサイズを変更するとき）
  • 3番目のパラメーターはセッションです。 タブの作業中にセッションを変更することはできませんが、セッションへのアクセスは維持されるため、理論的には（チェックしませんでした）設定を変更でき、それらが有効になります
  • ブラウザータブ作成メソッドの最後のパラメーターはWebViewTypeです。 kWebViewType_Offscreen（デフォルト）またはkWebViewType_Windowのいずれかです。 okkonyブラウザーを作成するときに、2番目を指定します
  • タイプkWebViewType_WindowでWebViewを作成した後の必須手順は、親ウィンドウのハンドルを指定することです。 ハンドルは、Awesomiumが使用されているプロセスと同じプロセスのウィンドウである必要はありません。たとえば、Delphiプログラムにハンドルパネルがあり、すべてが正常に動作します。




人生





Awesomiumの使用目的に応じて、その「寿命」は大きく異なる場合があります。 Windowedタイプのタブを使用する場合の主なポイントについて説明します。



第一に、Windowed Awesomiumは部分的に自分自身の生活を送っていますが、それでも定期的に働くことを奨励する必要があります。 プッシュは、カーネルの更新メソッドによって行われます。 次のように、タイマーで呼び出すことをお勧めします。 



 // Create our WebCore Update timer, renders at a max of 66 FPS UINT myTimer = SetTimer ( hwnd, 0, 15, NULL ); ... case WM_TIMER: { if (core) { core->Update(); } } ...




また、WebViewの破壊後にカーネルの更新を行うことをお勧めします。



次に、ブラウザウィンドウを親ウィンドウとともにサイズ変更する場合、親ウィンドウのWM_SIZEメッセージを処理し、ブラウザに新しいサイズを報告する必要があります。 これは非常に簡単に行われます： 



 case WM_SIZE: { if (view) { int nWidth = LOWORD(lParam); //    int nHeight = HIWORD(lParam); //    view->Resize(nWidth, nHeight); } }




そして第三に、ブラウザが完全に機能するためには、リスナーインターフェイスを実装し、それらを各WebViewに接続する必要があります。 インターフェイスの実装に問題はないはずです、それらは十分に文書化されています。 必要なヒントはすべて、 WebViewListener.hファイルにあります。 まあ、それらを接続することはまったく難しくありません： 



 DialogListener * dialog_lis = new DialogListener(); view->set_dialog_listener(dialog_lis);




前にも言ったように、ウィンドウモードで作業する場合、ライブラリはこれについて多くのことを心配します。親ウィンドウでの読み込まれたページのレンダリング、キーボードとマウスのキーストロークの処理、マウスカーソルの変更など。

死





仕事の後の清掃では、最大の問題が発生しました。 Windowedモードで作業するための公式ドキュメントおよびGoogleのこの主題に関する情報は、ゼロよりもわずかに多くなっています。 試行錯誤によって、私は次のことに気付きました。



1. WebViewを破棄する前に-最初に親ウィンドウを破棄する必要があります（逆の順序はアプリケーションのクラッシュにつながります）。



2.セッションデータを保存する場合（メモリ内セッションではない場合）、セッションインスタンスでReleaseメソッドを呼び出す必要があります。



3. Destroyメソッドを呼び出すとWebViewが破棄されます（他には何もありません）



4. Shutdownメソッドを呼び出すと、カーネルが破棄されます。このメソッドは、誤って破棄されたWebViewをクリーンアップすることもできます（ただし、このプロセス自体を制御することをお勧めします）



大体、次のようになります。 



 // -     ... //    (     ,      ,    ) view->Stop(); //      view->session()->Release(); //  WebView (    Chromium       ) view->Destroy(); //   (      ,      ) core->Update(); //   core->Shutdown();




カーネルの破棄は、プログラムが終了する前にのみ必要です。 1つのブラウザタブを閉じる必要がある場合は、そのタブとこのタブにあったWebViewを削除するだけです。
