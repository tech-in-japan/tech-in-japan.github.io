Windows Azureの最初のプロジェクトのクイックスタート。 Webサイト。 クラウドサービスへのWebサイトの移行

こんにちは、同僚の皆さん。



サイクルの最初の5つの部分は、 ここから入手できます 。 シリーズのこのパートでは、Windows Azureロールモデル、Windows Azureサービスバス、およびストレージサービスを使用する単純なアプリケーションを使用して、単純なWebサイトをクラウドサービス（以前のホストサービス）に移行することを検討します。





次のシナリオを見てみましょう：Windows Azure Webサイトに既にWebサイト（たとえば、後で説明する標準ASP.NET MVC 4 Webサイト）を展開しましたが、しばらくするとWebサイトに追加の機能-ユーザーがあなたのウェブサイトに行き、自分のギャラリーに画像をアップロードできることが必要です。 1つのWebサイト機能を使用してこのようなシナリオを実装しますが、要件のリストには、Webサイトの柔軟なスケーラビリティの条件も含まれています。 Windows Azure Webサイトのみを使用する場合、Webサイト全体のみをスケーリングできますが、これは非効率的なソリューションです。ほとんどの場合、主な負担はWebユーザーインターフェイスではなく画像プロセッサにあります。 Webサイトをクラウドサービスに移行し、Windows Azureクラウドサービスの役割モデルに基づいてアーキテクチャを再検討し、Windows Azure Blobストレージとサービスバスキューを使用すると、このシナリオの効果的な実装を作成できます。 シリーズのこのパートでは、2つのレイヤーで構成されるクラウドアプリケーションを作成します。



1）Webロールとして実装されたWebインターフェイス。 アプリケーションのメインページで、ユーザーは画像をアップロードしたり、ダウンロード済みの画像を表示したりできます。 イメージを読み込んだ後、WebロールはBLOBストレージ内のイメージをBLOBコンテナーに送信します。BLOBコンテナーの名前はタスクを簡素化するために事前定義され、その後、イメージを含むBLOBへのリンクで構成されるメッセージをService Busキューに入れます。



2）Workerロールとして実装された画像プロセッサ。 無限ループでは、ハンドラーはWebロールからのメッセージをService Busキューに照会し、キューにメッセージがある場合、メッセージを取得し、イメージBLOBへのリンクを受信し、コピーソースBLOBメカニズムを使用して新しいBLOBを作成します同じイメージをblobリポジトリに配置します。
