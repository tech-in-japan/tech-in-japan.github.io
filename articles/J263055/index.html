#DockerCon #DockerCon #DockerConコンテナの生産と永続性

2015年6月末に、米国サンフランシスコでDocker開発者向けの会議が開催されました。 EMCエンジニアのRyan Wollner氏がDockerCon 2015がどのように進んだか、会議を中心に発展したパーティとエコシステムについての話をお届けします。







私は5年以上オープンソースコミュニティに参加しており、ネットワークテクノロジー、PaaSインフラストラクチャ、コンテナ化テクノロジーに取り組んでいます。 OpenFlow / SDNなどのプロジェクトへの参加、OpenStackはさまざまな成功を収めましたが、Dockerは客観的に技術コミュニティから最大限の注目を集めています。



世界中の開発者は、組み込みアーキテクチャ、ジャーナリング、監視、データ移行、ネットワーク、およびコンテナ内でのステートフルサービスの実行に多大な注意を払っています。 永続化ボリュームのトピックをさらに深く掘り下げ、チームの結果を示す前に、会議とコミュニティ全体の関連トピックをいくつか強調したいと思います。



DockerCon 2015のテーマの1つは、実稼働環境でのDockerの使用です。 Dockerでの経験について誰に尋ねるか、どのように使用されたかに応じて、答えは異なります。 ブライアン・カントリルが言ったように、マイクロサービスを備えたコンテナを使用した生産は「地獄」である可能性があります（ブライアンがUnixの哲学と実行時のデバッグについて話していないのなら、特にO'Reillyカンファレンスからの彼のレポートのビデオを見ることを強くお勧めします、 DockerConの前夜に行われました）。 ただし、一方で、アプリケーションを高度な制御性を備えた限られたドメインに分割すると、開発チームがCI / CDデバッグ/配信/展開プロセスを効果的に実施する機会が得られます。 Netflixのgithubのオープンソースコードはこれを確認し、Andrian Cockcroftはこの主題を深く掘り下げました。 また、マイクロサービスに関する私の投稿を見ることができます。これは、コンテキストを理解するのに役立ちます。



Dockerは販売に適していますか？ 以下では、本番環境でDockerを使用する前に知っておく必要があるポイントについて説明します。 また、各アイテムはDockerConの大きなトピックであるということを付け加えておきます。



ネットワーク



Docker社とSocketplaneスタートアップの相互作用により、プロジェクトでLibnetworkテクノロジーを使用できるようになりました。これにより、異なるホストからDockerデーモンを接続できるようになり、コンテナーがネットワークのデータリンクレイヤーを介してIPトラフィックを簡単に転送できるようになりました。 Libnetworkは、メインのDockerデーモンの外部からサポートされ、ネットワーク実装に依存せず、最も重要なこととして、実装の詳細をユーザーから隠し、システム全体が「正常に動作する」ようにします。



安全性



私はDockerのEric Windishと話をしました。彼は、Dockerエンジンのセキュリティは常に重要な注意を払っていると述べました。 セキュリティサブシステムのコードを改善するための広範な作業が進行中であり、 AppArmorおよびSELinuxに基づく他のツールも使用されています。 VMWare Lightwaveなどのプロジェクトに関連してDockerセキュリティサブシステムがどのように展開されるかは非常に興味深いです。



ジャーナリング、監視、管理性



コンテナはもちろんクールですが、同時に1,000個または10,000個のコンテナを起動することにより、コンテナを効率的にデバッグ、監視、および管理できるツールの必要性が認識されます。 そして、決定が私たちを待っているようです。 Dockerの創設者であるSolomon Hykesが発表した Project Orcaは、Docker Engine、Docker Swarm、GUI、Docker Compose、およびセキュリティ、インストール、展開、構成のためのツールを接続することを目的としています。 もちろん、Orcaを介してDockerを実行する必要はありませんが、速度と管理性を損なうことなく上記のすべてを取得する方法があるとすばらしいでしょう。 他のツール（loggly、cadvisor、 ruzit 、datalog、log entries）がこのリストの場所を奪い合っていますが、それは素晴らしいことです！



拡張性



Dockerはコミュニティにイニシアチブを提供し、さまざまなプラグインがネットワーク、認証、データウェアハウスと連携できるようにする拡張可能なツールになりたいと述べています。 現在、認証用のVMWare Lightwaveと永続ストレージ用のEMC ScaleIOがあります。 Dockerがたった2歳であることを考えると、それほど悪くない！



内部状態を保持する永続的なサービス



EMCとClusterHQは、 Flockerオープンソースプロジェクトを作成し、共に素晴らしい仕事をしました。 これにより、コンテナボリュームを管理し、コンテナ化されたアプリケーションを復元する必要がある状況で、データの移行とボリュームの高可用性を実現できます。



また、カンファレンスで大きな関心を集めたのは、Dockerのネイティブボリュームの拡張の発表でした。 バージョン1.7.x以降の実験的ステータスにもかかわらず、これらの拡張機能は、内部状態を維持しながらサービスを管理するためのさまざまなオプションを提供します。 Flocker -Dockerプラグインを使用する場合 、Flocker自体は、volume-driver = flockerフラグを使用してDockerボリュームの拡張機能で動作できます 。



ClusterHQとの相互作用の結果は、 EMC ScaleIOおよびEMC XtremIOとの連携を可能にするソースドライバーのGitHubに関する出版物でした 。 サンフランシスコのPivotal Labで、すべてのドライバー機能のデモを行いました。 その結果、人々は非公式の設定でビールと会話のデモを見るために集まりました。 このリンクにはScaleIOデモが含まれています。これは、TwitterからデータをダウンロードしてMongoDBに配置したアプリケーションのMEANスタックを使用してAmazon AWSにデプロイしたFlocker + ScaleIOの作業に捧げた会議で示しました。 会議のスライドもこちらから入手できます 。







DockerConのすべてのゲストが十分な席を持っているわけではなく、セッション全体を通して多くのリスナーが床に立ったり座ったりしていました。 ラップトップを持っている人がデモをハッキングする機会ができるようにもう少し時間があることを望んでいましたが、割り当てられた40分間では明らかに十分ではありませんでした。 彼らはできる限りのことをしました！











RexRayと呼ばれるネイティブのGo実装により、EMCは永続ボリューム管理パスを統合しますが、 Flockerが生成する自動データ移行はありません。 RexRayの可能性は非常に広く、たとえば、EC2 EBS + EMC ScaleIOなど、複数のバックエンドを同時に使用できます。







一般に、永続性とコンテナは引き続き関連しており、これには理由があります。 最初は、 ステートレスプロセスと12の要因の方法論がトレンドでした。 しかし、これはすべて現実と離婚しています。 コミュニティは状態が存在することを理解しており、データベースなどのコンテナーでのステートフルプロセスの起動は、マイクロサービスの世界に本当に関連しています。 各コンテナは、「ステートレス」状態であっても、何らかの状態にあります。 たとえば、メモリには実行可能アプリケーションやオープンソケットなどの状態が存在する場合があります。これらの状態は、データの移行中など、特定の場合に作業する必要があります。



コンテナ環境ではデータが重要になります。 コンテナアーキテクチャの負荷がマッピングされると、データの整合性を追加する必要が生じます。 内部状態を維持するサービスには、整合性と可用性が必要です。 このすべてに産業用ストレージも追加します。 会議では、この問題に対するさまざまな解決策が提案されましたが、その一部を以下に示します。



• Crate.io 。 アプリケーションサーバーにインストールするスケーラブルな分散データベース。

• ClusterHQ /フロッカー。 コンテナボリュームのオープンソースマネージャ。データベースや他のプロセスをコンテナに入れて状態を保持したり、バックエンドを気にせずに移動したりできます。

•Red Hat は 、RHS（Red Hat Storage）を使用したコンテナの永続ストレージの統合を発表しました。

•Nutanix は 、プラットフォーム上のコンテナの永続性を確保するのに役立つボリュームAPIについて話しました。

•Portworx。 これはClusterHQと競合する新しいスタートアップのようです。 この発表は、彼らのプラットフォームの「一部」がどのようにオープンソースになるかを述べています。

•ストレージのKubernetesサポート。 KubernetesはGitHubのドキュメントを更新しました。これは、k8sでGoogle Cloud永続ディスクを使用する機能を反映しています。



全体として、それは素晴らしいDockerConであり、楽しいイベント、優れた人材、革新的なテクノロジーが満載でした。 11月にバルセロナでお会いしましょう！
