ESXiストレージサブシステムの仕組み



何も機能しない理由を理解する前に、メモリ内の何かを更新する必要があります。 ハイパーバイザーのストレージサブシステムの配置方法、構成中に使用されるユーティリティ、エラーの検索場所-これが数行に費やされるものです。





VMware Pluggable Storage Architecture Frameworkは、特定のデバイスクラスのI / O操作を通常担当するマルチパスプラグインをロードします。

ハイパーバイザーには、ネイティブマルチパスプラグイン（NMP）と不要なデバイスを隠す MASK_PATHスタブプラグインの2つのプラグインが含まれています 。 一部のベンダーは、EMC PowerPath / VEまたはSymantec / Veritas DMPなどの独自のMPプラグインを開発しています。



NMPは 、パス選択プラグイン（PSP）およびストレージアレイタイププラグイン（SATP）プラグインを通じてその機能を実装します。



PSPはI / Oを直接処理し、複数のアクティブパス間でトラフィックのバランスをとるポリシーを実装します。 ブランドPSPは3つしかありません（ したがって、政治家 ）= MRU 、 RR 、修正済み。 ベンダーPSPもあります。 I / O操作中にエラーが発生した場合、PSPはそれをSATPに送信し、SATP はそれに応答します 。



SATPは、特定のアレイで作業し、パスの状態を監視するときに、アクティブな（最適な）パスを選択します。 アレイから返されたSCSI Senseコードに応答します （一部のコードは同じで、一部はベンダーによって異なる場合があります）。 RTPGコマンドでALUAアレイをポーリングして最適化されたパスを選択するのはSATPです。 プラグインの動作は、追加オプションで「調整」されます。



デバイスの場合、特定のデバイスを処理するMPとSATPを決定するルールが作成されています。

複雑に聞こえますが、実際には、すべてがはるかに単純です。

ドライブを接続します



ライブラリをSANスイッチに接続し、ライブラリが1つのホストと1つのカードのみを見ることができるようにゾーニングを構成します。 うまくいかない場合は大丈夫です。プラグインでマスクします。



ESXiがストリーマーを見たことを確認します。



フォームnaa。{Your identifier}の識別子をすぐに書き留めます。



[パスの管理]に移動して、デバイスでアレイを操作するためのプラグインが選択されていることを理解します。





この例では、これはVMW_SATP_ALUAです。 同時に、プラグインはストリーマーに対して非常にうまく機能するため、同時にプラグインへのすべてのパスが無効としてマークされます。 ストリーマがALUAとマルチパッシングの両方に対して多少異質であると仮定することは論理的です。 したがって、単純なローカルデバイスに変換します。

注：VMW_SATP_LOCALプラグインが使用されているため、デバイスへの単一パスのみがサポートされます。



VMware、 KB1026157


必要なもの。



テープドライブのNMP設定を見てみましょう。



そうです、VMW_SATP_ALUAを使用します。 esxcli storage nmp satpルールリストルールは、SATPプラグインの選択に責任があることを思い出してください。



デバイスが特定のルールに該当しない場合、「デフォルト」ルールのいずれかによって処理されます。





論理的には、トランスポートは「fc」であるため、デバイスは「ファイバーチャネルデバイス」というルールに該当するはずですが、実際にはプラグインには「ALUAをサポートするアレイ」というルールが割り当てられています。 どこかにキャッチがありますが、それは根本的なものではありません。 私たちのタスクは、VMW_SATP_LOCALプラグイン（通常はusb、ide、blockなどのローカルデバイスを処理）によって処理されるように、ストリーマーにこのようなルールを記述することです。



次のコマンドを使用して、デバイスを注意深く調べます。 

 esxcli storage core device list




ルールを作成するには、次の一意のストリーマープロパティを使用します。

ベンダー ：HP

モデル ：Ultrium 5-SCSI



ルールを書きます： 

 esxcli storage nmp satp rule add --satp=VMW_SATP_LOCAL --vendor="HP" --model="Ultrium 5-SCSI"


次に、テープドライブへの使用可能なすべてのパスを再初期化する必要があります。 

 esxcli storage core claiming reclaim -d naa.{ }


その結果、ホストはデバイスへの1つのパスのみを使用し、1つのHBAのみで使用します。

他のカードでストリーマーをマスクするといいでしょう。



仮想マシンで、新しいデバイスを追加します。





デバイスの種類「テープHP」：





ここで、仮想デバイスノードを慎重に選択する必要があります。 ルールは次のとおりです。

-ターゲット番号は実数と一致する必要があります。 私たちの場合、これはターゲット2です。



-アダプタポートにディスクデバイスがないはずです。 仮想マシンのハードディスクは通常ポート0にマッピングされるため、最初の適切なポートは1：2（1-コントローラーSCSIポート番号、2-ストリーマーターゲット番号）です。 VMware仮想SCSIコントローラーはデュアルポートです。 ポート3または4を指定すると、新しいコントローラーが自動的に追加されます。



Windowsドライバーパッケージ用のHP StoreEverテープドライバーをダウンロードします。 仮想マシンでは、パッケージはインストールされません。





7zipを解凍し、setup.exeを実行します。



ストリーマーはデバイスマネージャーで準備完了です！





ライブラリを接続する





ストリーマーを接続するだけでは十分ではありません;通常の操作では、オペレーティングシステムで「チェンジャー」または「ロボット」とも呼ばれるライブラリ自体を確認する必要があります。



リストからわかるように、ライブラリへのパスがありません：

この未知のデバイスのSCSIターゲットIDはストリーマーと一致しますが、LUN番号は1です。明らかに、ライブラリはそれを認識できません。 NMPはライブラリをVMW_SATP_ALUAプラグインに接続しようとしましたが、何もできませんでした。

この問題は、キャッチオールSATPルールがデバイスを要求し、デバイスがALUAとして登録されているために発生します。

必要なパラメーター：

ベンダー：HP

SATPの新しいルールを追加します。 

 esxcli storage nmp satp rule add --satp=VMW_SATP_LOCAL --vendor="HP" --model="MSL G3 Series"


パスの請求を解除する 

 esxcli storage core claiming unclaim -t location -A vmhba2 -C 0 -T 1 -L 1


デバイスをスキャンする： 

 esxcfg-rescan vmhba


メディアチェンジャーへのパスが正常に追加されました：

他のESXiホストでドライブをマスクする必要がある場合（チェンジャーはオプションであり、既に見たように検出されません）、記事KB1029786のように進めてください。 そして、ごく短時間であれば、MASK_PATHプラグインによってNMPの代わりにデバイスが処理されるようにルールを追加します。 

 esxcli storage core claimrule add -r 192 -t location -A vmhba{N} -C {C} -T {T} -L {0} -P MASK_PATH esxcli storage core claimrule load esxcli storage core claiming reclaim -d naa.{ }
