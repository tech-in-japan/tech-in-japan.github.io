PHP 7の概要：追加されたもの、削除されたもの

2015年にPHPの世界で起こった最も重要なイベントの1つは、PHP 7のリリースでした。10年の間、5番のPHPの最初のリリースのリリースから分離されました。仕事のスピード。

ただし、7番目のバージョンでは、廃止された機能が削除されたため、下位互換性に関する問題が発生し、古いアプリケーションを新しいバージョンに移行することが困難になりました。 この記事は、新しいアプリケーションを作成したり、既存のアプリケーションをPHP 7に変換する場合のクイックガイドとして役立ちます。



待って、PHP 6はどこですか？



PHPをしばらく使用していない場合、PHP 6がどこに行ったのか疑問に思うかもしれません。なぜPHP 5からPHP 7に直接ジャンプしたのですか 簡単に説明しますが、PHP 6バージョンのリリースは行われませんでした。 PHPは主にWeb開発で使用されるため、バージョン6の主な機能はUnicode文字のサポートでしたが、WebはUnicodeサポートを必要とするため、その実装は理にかなっています。



アイデアは、Unicodeサポートをカーネルに直接実装することでした。 このアプローチは、変数の名前と関数としての愚かな絵文字の使用から、国際的な文字列の強力な機能まで、言語の機能を拡張することになっています。 たとえば、別の言語が英語以外の大文字と小文字を使用する場合、または中国語の名前を英語の名前に変換する場合。



PHP6は野心的でしたが、残念です。 したがって、6番目のバージョンをスキップするプロセスでPHP7を開始しました。


残念ながら、この野心的な計画には予想以上に多くの問題がありました。 カーネルの場合と重要な拡張機能の場合の両方で、Unicodeをサポートするためにコードベースのほとんどを移植する必要がありました。 これにより、他の言語機能の開発プロセスが遅くなり、多くのPHP開発者が混乱しました。 他の障壁が現れたため、ネイティブUnicodeサポートの開発への関心が低下し、やがてプロジェクトは完全に放棄されました。



また、書籍や記事などの多くのリソースがPHP 6の名前とUnicodeサポートを使用して記述されているため、誤解を避けるために、新しいバージョンの名前をPHP 7にすることにしました。



可能であれば、暗い過去を掘り下げるのをやめましょう。 PHP 7の新機能を見てみましょう。



生産性戦争、PHP 7対 PHP 5



実質的にすべての更新により、パフォーマンスがわずかに改善されました。 ただし、今回は、以前のバージョンと比較してPHPのパフォーマンスが大幅に向上し、PHP 7の最も魅力的な機能の1つになりました。これは「PHPNG」プロジェクト（「php new generation」または「php new generation 。トランスレータ ）、これはZend Engine自体に影響しました。



内部データ構造をリファクタリングし、コードを抽象構文ツリー（AST）にコンパイルする前に追加のステップを追加すると、優れたパフォーマンスとより効率的なメモリ割り当てが可能になりました。 数値自体は有望に見えます-実際のアプリケーションで実行されたテストは、PHP 7がPHP 5.6の平均2倍の速度であり、クエリ処理中のメモリ使用量が50％少ないことを示しています。 この Zend インフォグラフィックを見てください。これは、いくつかの有名なCMSとフレームワークのパフォーマンスを示しています。



PHP 7は見慣れていますが、パフォーマンスが「調整」されています。 Zend Engineが改善され、生産性が向上したため、Zend Engineと以前のバージョンとの間に大きな違いが生じました。



メモリ消費を削減することで、弱いコンピューターでもリクエストをより適切に処理できるようになり、PHPの周りにマイクロサービスを構築できるようになります。 内部変更、特にASTの実装は、生産性をさらに高めることができる将来の最適化の機会も提供します。 JITコンパイラの新しいネイティブ実装は、将来のバージョンでの開発を意図して開発されました。



PHP 7の構文シュガー



PHP 7では、いくつかの新しい構文が導入されました。 目を楽しませてくれるコードを書くためのより良いまたはより簡単な方法を提供するのであれば、言語自体の機能のいくつかを拡張してみませんか？



インポート宣言のグループ化



これで、同じネームスペースのクラスのインポート宣言を1行でグループ化できます。 これにより、宣言を特定の意味のある方法に揃えたり、コードの数バイトを節約したりできます。 

use Framework\Module\Foo; use Framework\Module\Bar; use Framework\Module\Baz;




PHP 7では、次のことができます。 

 use Framework\Module\{Foo, Bar, Baz};




または、複数行スタイルを好む場合： 

 use Framework\Module{ Foo, Bar, Baz };




ヌル合体演算子



PHPの一般的な問題を解決します。 別の変数に割り当てられている変数に値を割り当てたいが、最後の変数に値が割り当てられていない場合、何らかの明示的な値を割り当てる必要があります。 多くの場合、ユーザー入力を操作するときに現れます。



PHP 7より前： 

 if (isset($foo)) { $bar = $foo; } else { $bar = 'default'; //  $bar  'default'  $foo  NULL }




PHP 7の場合： 

 $bar = $foo ?? 'default';




変数のチェーンで使用できます： 

 $bar = $foo ?? $baz ?? 'default';




宇宙船オペレーター



「スペースシップ」<=>演算子を使用すると、2つの値の3レベルの比較が可能になり、それらの等値または不等式だけでなく、不等式の場合にどちらが大きいかを理解でき、1.0または-1を返します。



この場合、値の違いに応じてさまざまなアクションを実行できます。 

 switch ($bar <=> $foo) { case 0: echo '$bar  $foo '; case -1: echo '$foo '; case 1: echo '$bar '; }




比較される値は、整数型、浮動小数点数、文字列型、さらには配列型です。 異なる値は互いにどのように比較されますか？ ドキュメントを参照してください 。

スカラーパラメータのタイプと返されるタイプのヒント



PHP 7は、可能なパラメーター型として整数（int）、浮動小数点（float）、論理（bool）、および文字列（string）の4つのスカラー型を追加することにより、メソッド（クラス、インターフェイス、および配列）のパラメーターの既存の宣言を拡張しました。



さらに、オプションで、関数またはメソッドによって返される結果のタイプを指定できます。 タイプbool、int、float、string、array、callable、クラスまたはインターフェース名、およびparentがサポートされています（クラスメソッドの場合）。 

 class Calculator { // ,      integer public function addTwoInts(int $x, int $y): int { //  ,     return $x + $y; } }


型を宣言することにより、透過的なアプリケーションを構築し、関数を操作するときに不正な値の転送と戻りを回避できます。 その他の利点は、ドキュメンタリーノートDocBlocksがない場合に、コードのより明確な表示を提供する静的コードアナライザーとIDEの外観です。



PHPは型が弱い言語であるため、コンテキストに基づいていくつかのパラメーター値と戻り値の型が引用されます。 int型の宣言されたパラメーターを持つ関数に値「3」を渡すと、インタープリターはそれを整数と見なし、エラーを生成しません。 この動作に慣れていない場合は、適切なディレクティブを追加することにより、厳格モード- 厳格モード -で作業できます。 

 declare(strict_types=1);


グローバルオプションはコードリポジトリを、オプションをオンにしてファイルが開発されたものと無効にされたものに分割するため、異なる設定の2つのリポジトリからのコードを操作するときの予測できない動作につながるため、ファイルごとに個別にインストールされます。



エンジンの例外



カーネルレベルの例外の出現により、以前はスクリプトの実行が停止する可能性があった致命的なエラーを簡単に傍受して処理できるようになりました。



存在しないメソッドの呼び出しなどのエラーがスクリプトを停止することはなくなり、代わりにtry catchブロックで処理できる例外がスローされ、アプリケーションのエラー処理が明らかに改善されます。 これは、致命的なエラーがなければ再起動が必要になる可能性があるため、一部の種類のアプリケーション、サーバー、デーモンにとって重要です。 致命的なエラーはテストプロジェクト全体をドロップする可能性があるため、PHPUnitのテストも使いやすくなるはずです。 エラーとは異なり、例外はテストごとに個別に処理できます。



PHP 7は見慣れたツールのように見えますが、高いパフォーマンスを目指しています。 Zend Engineの再設計と速度の向上により、以前のバージョンとの大きな違いが生じました。


PHP 7には、発生する可能性のある種類のエラーを処理するように設計された新しい例外クラスがかなりあります。 バージョン間の互換性を確保するために、新しいThrowableインターフェイスが追加されました ;カーネルレベルの例外とユーザーの例外の両方で実装できます。 このアプローチは、カーネル例外による例外の基本クラスの継承を防ぐために実装されます。これにより、以前は存在しなかった以前に記述されたコードに例外が出現します。



PHP 7より前では、このようなコードは致命的なスクリプト実行エラーを引き起こしていました。 

 try { thisFunctionDoesNotEvenExist(); //() } catch (\EngineException $e) { //           echo $e->getMessage(); }




匿名クラス



匿名クラスは、短くて明確なオブジェクトの機能を作成するために使用できる匿名関数のいとこです。 匿名クラスは、通常のオブジェクトと同じ方法で簡単に作成および使用できます。 これはドキュメントの例です。



PHP 7より前： 

 class MyLogger { public function log($msg) { print_r($msg . "\n"); } } $pusher->setLogger( new MyLogger() );




匿名クラスの使用： 

 $pusher->setLogger(new class { public function log($msg) { print_r($msg . "\n"); } });




匿名クラスは、ユニットでテストする場合、特にオブジェクトとサービスをテストする際にモック（実際のオブジェクトの動作をシミュレートする-ほぼトランスレーター）を行う場合に役立ちます。 それらが存在することで、モッキングに使用できるインターフェイスをサポートするシンプルなオブジェクトを作成することで、大きなモッキングライブラリとフレームワークの使用を回避できます。

暗号的に安全な文字列と整数を生成するための2つの新しい機能。 最初は長さ$ lenのランダムな文字列を返します： 

 random_bytes(int $len);




2番目は、$ min ... $ maxの範囲の数値を返します。 

 random_int(int $min, int $max);




Unicodeのエスケープコード構文



PHPバージョン7より前のその他の多くの言語とは異なり、PHPでは、文字列内のUnicode文字のエスケープシーケンスを指定する方法がありませんでした。 これで、エスケープシーケンス\ uを使用して、UTF-8セットのコードを使用してそのような文字を生成できます。 これは、文字を直接挿入するよりも優れています;目に見えない文字や、値以外のグラフィック表示を持つ文字は、より適切に制御されます。 

 echo "\u{1F602}"; //




以前に動作し、文字の\ uペアを使用していたコードは、動作が変更されたため、バージョン7では正しく動作しないことに注意してください。



更新されたジェネレーター



PHPのジェネレーターにもいくつかの素晴らしい新機能があります。 現在、returnステートメントがあり、これを使用して、反復の完了時に関連していた最終値を返すことができます。 ジェネレータの正しい実行を確認するために使用できます。 たとえば、エラーなしで実行されたかどうかを確認するため、発生した状況をジェネレーターを呼び出したコードが正しく処理できるようにします。



さらに、ジェネレーターは他のジェネレーターから式を返したり返したりできます。 したがって、複雑な操作をより単純な操作に分解できます。 



 function genA() { yield 2; yield 3; yield 4; } function genB() { yield 1; yield from genA(); // 'genA'       yield 5; return 'success'; //  ,      } foreach (genB() as $val) { echo "\n $val"; //    1  5 } $genB()->getReturn(); //  'success'


期待



期待-下位互換性を維持しながらassert（）関数を改善します。 これにより、作業コードでゼロコストアサーションを使用でき、ステートメントの処理中にエラーが発生したときにカスタム例外を生成する機能をサポートします。これは開発中に役立ちます。



assert（）関数は、PHP 7の言語構成体になりました。クレームは、デバッグ目的の開発およびテスト中にのみ使用してください。 その動作を設定するには、2つのディレクティブを使用する必要があります。



  • zend.assertions

    1：コードを生成して実行する（開発モード）（デフォルト値）

    0：コードを生成しますが、実行時にバイパスします

    -1：コードを生成せず、ゼロコストのコードにします（作業コードモード）

  • assert.exception

    1：対応する例外オブジェクトを作成するか、そのようなオブジェクトが作成されていない場合はAssertionErrorオブジェクトを生成することにより、アサーションエラーによって生成されます

    0：上記のThrowableを使用または生成しますが、そのオブジェクトに基づいて警告のみが生成され、それを使用して例外はスローされません（PHP 5互換の動作）





PHP 5からPHP 7に移行する準備をする



PHPバージョン7の登場により、廃止された、またはしばらくの間すでに不要であると考えられていた機能を変更/更新、さらには削除することができました。 このような変更は、古いアプリケーションとの後方互換性の問題につながる可能性があります。



このようなバージョンが表示されるときに発生する別の問題は、重要なライブラリとフレームワークが新しいリリースと互換性のある更新プログラムを持たない可能性があることです。 PHP開発チームは、新しいバージョンへの移行にかかる時間ができるだけ短くなるように、下位互換性を維持しながら可能な限り変更を加えようとしました。 比較的新しくタイムリーに更新されたアプリケーションは、おそらく7番目のバージョンに簡単に移行できますが、古いアプリケーションはバランスのとれた決定が必要です。取得した機能に費やした労力、またはまったく更新しない方がよいでしょう。



ほとんどの問題は小さく、簡単に修正できますが、他の問題はより多くの時間と労力を必要とする場合があります。 一般に、PHP 7をインストールする前にレガシー関数の使用に関する警告が表示された場合は、修正されるまでアプリケーションの実行を停止するエラーメッセージが表示される可能性が高くなります。

PHP 7インフラストラクチャの互換性



多くのホスティングサービスがPHP 7のサポートを追加しました。これは、共有ホスティングプロバイダーにとって朗報です。生産性が向上すると、ハードウェアを更新せずにクライアントWebサイトの数を増やし、ランニングコストと収益を削減できるためです。 クライアントに関しては、これらの条件での生産性の急激な増加を期待すべきではなく、率直に言って、パフォーマンス指向アプリケーションの場合、共有ホスティングはいずれの場合もオプションではありません。



一方、プライベート仮想サーバーまたは専用サーバーを提供するサービスは、パフォーマンスの向上の恩恵を受けます。 Herokuなどの一部のPaaSサービスは 、PHP 7を長い間サポートしていますが、AWS BeanstalkやOracle OpenShiftなどのPaaSサービスは遅れています。 PaaSプロバイダーのWebサイトで、PHP 7がサポートされているか、近い将来に計画されているかを確認してください。



IaaSプロバイダーを使用すると、ハードウェアを制御してPHP 7をインストール（または、必要に応じてコンパイル）できます。 PHP 7パッケージは、ほとんどのIaaS環境ですでに利用可能です。



PHP 7とのソフトウェア互換性



インフラストラクチャの互換性に加えて、潜在的なソフトウェア互換性の問題にも注意する必要があります。 WordPress、Joomla、Drupalなどの有名なCMSは、既にPHP 7のサポートを追加しています。SymfonyやLaravelなどの主要なフレームワークもこれを実行しています。



しかし、注意する時が来ました。 このサポートは、CMSまたはフレームワークを参照するアドオン、プラグイン、パッケージなどの形式のサードパーティコードには適用されません。 問題があるかもしれません。あなたの仕事は、すべてがPHP 7で動作する準備ができていることを確認することです。



サポートされているアクティブなリポジトリの場合、これは問題になりません。 ただし、PHP 7をサポートしない放棄されたリポジトリでは、アプリケーション全体が使用できなくなる可能性があります。

ライブラリとフレームワークはPHP 7に移行し、新しいバージョンが登場します。 7つを試して結果を評価することをお勧めします。 あなたのアプリケーションは既に新しいバージョンと互換性があり、PHP 7で使用する準備ができているのかもしれません。
