オブジェクトデータモデルのトピックを続けます。 このパートでは、実際にはORMシステムであるDataモジュールについて説明します。 Dataモジュールの操作を説明するために、それを使用して、単純なサイトのコンテンツが作成されます。 前の記事： CMSアーキテクチャ。 データモデル。 パート1



Dataモジュールは、クラスData、Object、Multy、QueryおよびCond *クラスのセットで構成されています。 モジュール自体は静的なDataクラスであり、残りのクラスはそれが機能するデータ構造を表すために使用されます。 プログラムコードでエンティティを表すには、Objectクラスが使用されます。 すべてのオブジェクトに対して、どのタイプのエンティティ（データクラス、データオブジェクト、またはそれらの間の関係）でもかまいません。 Multyクラスは、特に複数のプロパティを表すために、一連のエンティティに関連付けるために使用されます。 QueryおよびCond *クラスは、柔軟な条件を考慮して（データベース内の）オブジェクトモデルで検索する必要があります。



エンティティ読み取り



次の方法で、識別子からデータベースからエンティティを取得できます。



$obj = Object::Create(1); // 1







new演算子の代わりに、静的なCreate（）メソッドが使用されます。 要求されたエンティティのインスタンスの存在を確認します。おそらく、エンティティはデータベースからすでにロードされています。 ロードされた場合、Create（）メソッドはそのインスタンスへのリンクを返します。そうでない場合、要求されたエンティティに関連付けられたインスタンスが最初に作成され、次にそのエンティティへのリンクが返されます。



エンティティを受け取った後、それを完全に使用できます。たとえば、そのクラスを参照してシステム名を見つけます。



echo $obj->getP( 'class' )->getA( 'sys_name' );





ロードされたエンティティはクラスですが、作業はオブジェクトとして実行されることに注意してください。 エンティティはオンデマンドでデータベースからロードされます。つまり、データベースへの実際のアクセスは、エンティティのプロパティまたは属性が最初にアクセスされたときにのみ発生します。さらに、使用されているプロパティ（通信が実行されるエンティティ）のみがロードされます。



プロパティと属性にアクセスするために設計を短縮することが可能かどうか疑問に思っているかもしれませんか？ 魔法の手法、最適化、様式化は時間の問題であり、微妙な問題の解決策です。


オブジェクトには、プロパティ（属性）と属性（属性）があります。 属性はスカラー値です。 プロパティとは、別のオブジェクトへの接続であり、実際には、接続が確立されるオブジェクトです。 クラスへの関係と継承関係は最適化されています。データベースでは属性として表されますが、データモジュールのロジックによってプロパティになります。

プロパティにアクセスするには、getP（$ name）メソッドを使用します。このメソッドは、接続が確立されたオブジェクトを返します。 getL（$ name）メソッドを使用して接続自体にアクセスできます-接続オブジェクトが返されます。



上記の例では、クラスへの呼び出しが行われますが、クラスへの接続はオブジェクトによって実装されていないため（最適化されているため）、クラスへの接続への呼び出しは無意味です-通信オブジェクトはありません。 getA（$ name）メソッドは、名前で属性値を読み取るために使用されます。



エンティティの読み込みは、その属性またはプロパティへの最初の呼び出しで自動的に開始されます。 ロードはLoad（）メソッドによって自動的に実行されます。 エンティティをロードするには、そのクラスが定義され、クラス継承の階層が決定されます。 クラスは、エンティティ属性がロードされるテーブルを決定します。 プロパティをロードするには、リレーションテーブルを直接見て、属性「primary」がロードされたエンティティの識別子と等しいすべてのリレーションをロードするだけで十分です。 リンクのみがロードされます。 リンクによって参照されるオブジェクトは、属性またはプロパティが使用されている場合、一般的な方法でロードされます。



エンティティ作成



例として、クラス「id」のオブジェクトを作成します。 オブジェクトには、識別子と、それが定義するクラスとの関係のみがあります。



$new = Object::Create();

$new->setP( 'class' , Object::Create(1));

$new->Save();



または省略形で：

$new = Object::Create( null , Object::Create(1));

$new->Save();





すべてが非常に簡単です。空のエンティティインスタンスが作成され、識別子1のクラスへの接続が作成され、保存が実行されます。 その後、新しいレコードが「id」テーブルに表示されます。 エンティティID属性は、「id」テーブルのキーフィールドを自動インクリメントすることで自動的に作成されます。



クラスの作成はまったく同じです。 たとえば、「my」という名前と「value」属性の定義を持つテストクラスを作成します。



$new = Object::Create();

$new->setP( 'class' , Object::Create(2)); // , #2

// ,

// , ( )

$new->setP( 'extend' , Object::Create(1)); //

$new->setA( 'sys_name' , 'my' ); //

$new->setA( 'final' , 0); //

//

$new->setA_cl( 'value' , array( 'type' => 'varchar' , 'max' =>255)); //

$new->Save();





属性の定義は、データベーステーブルのフィールドのパラメーターを示すことに限定されます。 保存後、テーブル「my」がフィールド「id」および「value」で自動的に作成され、エントリがエンティティクラス「my」に関連付けられたテーブル「id」および「class」に追加されます



属性の説明は、setA_cl（$ name、$ params）メソッドによって実行されます。 複雑なことはなく、データベーステーブルフィールドのパラメーターに対応する属性名とパラメーターのみが示されています。



プロパティを定義する必要がある場合、setP_cl（$ name、$ class）メソッドは、クラスエンティティ$ classに名前$ nameで接続を作成します。 この場合、自動的に作成されたエンティティ関係のgetL_cl（$ name）メソッドに連絡することにより、関係の属性を明確にすることができます。



これで、クラス「my」のオブジェクトを作成できます。 作成されたクラス「my」が識別子5を取得したとしましょう。

エンティティを削除



エンティティの削除はさらに簡単です：



$obj = Object::Create(6);

$obj->Delete();





最初に、削除されたエンティティのインスタンスが作成され（エンティティはデータベースからロードされません）、次に呼び出されます

メソッドDelete（）。



エンティティを削除すると、その関係はすべて自動的に削除されます。 接続が「consist」タイプ（種類= 0）の場合、接続によって参照されるエンティティも削除されます。 別のエンティティ（接続のタイプ= "consist"）の不可欠な部分であるエンティティの削除は失敗します。 また、クラスが別のクラスに継承されている場合、またはこのクラスのエンティティがある場合、クラスを削除することはできません。
