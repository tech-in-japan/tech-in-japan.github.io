メジャーおよびマイナーデバイス番号
メジャーおよびマイナーデバイス番号

いくつかのデバイスファイルを見てみましょう。 以下は、プライマリハードドライブの最初の3つのパーティションを表すものです。 

 # ls -l /dev/hda[1-3] brw-rw---- 1 root disk 3, 1 Jul 5 2000 /dev/hda1 brw-rw---- 1 root disk 3, 2 Jul 5 2000 /dev/hda2 brw-rw---- 1 root disk 3, 3 Jul 5 2000 /dev/hda3






コンマ区切りの列に注目してください。 最初の番号は、デバイスの「高い番号」と呼ばれます。 2番目-「ジュニア番号」。 メジャー番号は、ハードウェアの保守に使用されるドライバーを示します。 各ドライバーには、固有のメジャー番号があります。 同じメジャー番号を持つすべてのデバイスファイルは、同じドライバーによって管理されます。 上記のデバイスファイルはすべて、同じドライバーによって制御されるため、3に等しいメジャー番号を持っています。

マイナー番号は、ドライバーが制御するハードウェアを区別するために使用されます。 上記の例に戻ると、3つのデバイスすべてが同じドライバーによって処理されますが、それぞれに固有のマイナー番号があるため、ドライバーはそれらを異なるハードウェアデバイスとして「認識」します。

デバイスは、ブロックとシンボリックという2つの大きなグループに分けられます。 ブロックデバイスとキャラクターデバイスの主な違いは、ブロックデバイスとデータがバイト部分で交換されることです-ブロック。 内部バッファがあり、為替レートが増加します。 ほとんどのUnixシステムでは、1ブロックのサイズは1キロバイトまたは2の累乗である別の数値です。シンボリックデバイスは、データがバイト単位で順番に続く情報転送の単なるチャネルです。 ほとんどのデバイスは、ブロックサイズによって制限されず、バッファリングする必要がないため、文字クラスに属します。 リストの最初の文字がls-l / devによって受信された場合、 'b'の場合、これはブロックデバイスであり、 'c'の場合、それは文字です。 上記の例で示されたデバイスはブロックデバイスです。 以下は、いくつかのキャラクタデバイス（シリアルポート）のリストです。 

 crw-rw---- 1 root dial 4, 64 Feb 18 23:34 /dev/ttyS0 crw-r----- 1 root dial 4, 65 Nov 17 10:26 /dev/ttyS1 crw-rw---- 1 root dial 4, 66 Jul 5 2000 /dev/ttyS2 crw-rw---- 1 root dial 4, 67 Jul 5 2000 /dev/ttyS3






メジャーデバイス番号がどのように割り当てられるかを知りたい場合は、/ usr / src / linux / documentation / devices.txtファイルを調べてください。

すべてのデバイスファイルは、mknodユーティリティを使用してシステムのインストール中に作成されます。 たとえば、「coffee」という名前でメジャー番号12、マイナー番号2の新しいデバイスを作成するには、コマンドmknod / dev / coffee c 12 2.を実行する必要があります。/devディレクトリにデバイスファイルを配置する義務はありませんこれは、受け入れられている契約に準拠しています。 ただし、デバッグのためにデバイスドライバーを開発する場合、デバイスファイルをホームディレクトリに配置することは、おそらくそれほど悪い考えではありません。 唯一のこと-デバッグの完了後にデバイスファイルを配置する場所を修正することを忘れないでください。

議論中のトピックには明らかに関係ないが、私がしたいコメントがいくつかあります。 デバイスファイルにアクセスすると、カーネルは最大のファイル番号を使用して、この呼び出しを処理するドライバーを決定します。 これは、カーネルが実際にマイナー番号を使用せず、さらには何も知らないことを意味します。 これについて心配しているのは、ドライバー自体だけです。 マイナー番号を使用して、異なる物理デバイスを区別します。

ちなみに、「デバイス」と言うときは、手で持つことができるPCIボードよりも抽象的なものを意味します。 次の2つのデバイスファイルを見てください。 

 % ls -l /dev/fd0 /dev/fd0u1680 brwxrwxrwx 1 root floppy 2, 0 Jul 5 2000 /dev/fd0 brw-rw---- 1 root floppy 2, 44 Jul 5 2000 /dev/fd0u1680






現時点では、これらのデバイスファイルについては、どちらも同じドライバー（メジャー番号2）によって提供されるブロックデバイスであると言えます。 ドライブが1つしかない場合でも、両方ともフロッピードライブを表していると主張することさえできます。 しかし、なぜ2つのファイルなのでしょうか？ しかし、問題はこれです。そのうちの1つは1.44 Mbフロッピーディスクドライブです。 もう1つは同じドライブですが、1.68 MBフロッピーディスク用であり、一部の人々が「スーパーフォーマット」ディスクと呼ぶものに対応しています。 このようなフロッピーディスクは、標準のフォーマット済みフロッピーディスクよりも多くのデータを格納できます。 マイナー番号が異なる2つのデバイスファイルが、実際には同じ物理デバイスを表す場合です。 ですから、私たちの議論では、「デバイス」という言葉は、より抽象的なものを意味する場合があります。

 static const struct file_operations lptlcd_fops = { .owner = THIS_MODULE, .read = dev_read, .write = dev_write, };




dev_readとdev_writeは、それぞれデバイスファイルからの読み取りと書き込みを処理する関数（コールバックの登録）へのポインターです。 ここでもまだできますが、おそらく.openおよび.releaseポインターを追加して、デバイスファイルの1回限りの開閉を確認する必要がありますが、今のところ私は怠け者です。



ドライバーを登録するには、この構造を開始します 



 static struct miscdevice lptlcd_dev = { MISC_DYNAMIC_MINOR, "lptlcd", &lptlcd_fops };




ここで、 MISC_DYNAMIC_MINORはダイナミックマイナーのマクロで、行「lptlcd」はデバイスの名前、/ dev /フォルダーでの表示方法（この場合は/ dev / lptlcd）、＆lptlcd_fopsはfile_operations構造体へのポインターです。
