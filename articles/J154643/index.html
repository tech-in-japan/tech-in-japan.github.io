完璧な.htaccessが欲しい！



インターネット上では、.htaccessの使用方法に関する多くの異なる矛盾する情報が判明しています。 何が真実で何が間違っているかを理解するために、長く退屈な方法を試さなければなりませんでした。 ほとんどの場合、ブログのヒントの著者は、特定の条件が機能するためには、このApacheモジュールが必要であることを忘れています。

しかし、十分な言葉、それはビジネスへの時間です。 ほぼ完璧な.htaccessを作成してみましょう



1.最初の行は主なオプションを設定します： 



Options All -ExecCGI -Indexes -Includes +FollowSymLinks


Apacheを構成するアクセス権がない場合、または同じサーバー上の異なるサイトに対して異なるApache設定が必要な場合、オプションを使用すると非常に便利です。



-ExecCGIオプションは、CGIスクリプトの実行を防ぎます。 特定のフォルダのみを許可します。 セキュリティを強化します。

-Indexesオプションは、ディレクトリにインデックスファイルがない場合、ディレクトリの内容が表示されないようにします。 通常、共有ホスティングはデフォルトで有効になっています。 マイナスインデックスをプラスインデックスに変更することで、その逆の場合、ディレクトリの内容を表示できます。

-Includesオプションは、SSIを無効にします。 あなたがそれが何であるかわからない場合-禁止（これについてさらに知りたい場合は、サーバーサイドインクルードのリクエストでYandex（ ！ ）できます）。 IncludesNOEXECオプションを使用すると、スクリプトを実行せずにSSIを使用できます。

+ FollowSymLinksオプションを使用すると、サイトのルート内にないファイルまたはディレクトリへのシンボリックリンクを使用できます。



ご注意
ディレクトリごとに異なる設定でhtaccessを使用できます。 サイトのルートで-Indexesを宣言し、お気に入りのディレクトリに別の.htaccessファイルを作成して、その中に+ Indexesを宣言できます。 htaccessオプションは、別のhtaccessによってキャンセルされるまで深くネストされるまで、ディレクトリツリーの最上部から最下部に広がることに注意してください。



変更されていない場合、子.htaccessのすべてのオプションを完全にリストする必要はありません。 変更するオプションとディレクティブのみを指定（再割り当て）するだけで十分です。 他のオプションも親から継承されます。


例
このパス/サイト/ folder_one /サブフォルダー/その他/があるとしましょう

/site/.htaccessファイルで、次を指定します。 

 Options -Indexes Allow from all


ファイル/site/folder_one/.htaccessで以下を指定します。 

 Deny from all


ファイル/site/folder_one/subfolder/.htaccessで以下を指定します。 

 Options +Indexes Allow from all


結果は次のとおりです。

直接アクセスされない限り、コンテンツのファイルはサイトフォルダーに表示されます。 または、いずれかのファイルに明示的にアクセスしていない場合は、インデックスファイル。 インデックスファイルがない場合、403エラーが受信されます。

folder_oneフォルダーへのアクセスが拒否されました。 ファイル名を知っていてアドレスバーに入力しても、サーバーは応答として403エラーを返します。

サブフォルダーフォルダーは、直接アドレスにアクセスできます。インデックスファイルがない場合は、ディレクトリの内容が表示されます。 同じ権限が他のフォルダーにも適用されます。

フォルダーfolder_oneからファイルhtaccessを削除すると、親サイトから権限を継承します。

5. php.iniにアクセスできない場合



.htaccessを使用すると、多くのPHP設定も管理できます。 共有ホスティングでは、原則として、php.iniの設定を変更する方法はありません。 ほとんどの場合、これは必要ありません。 しかし、私たちにとって役立つかもしれないオプションの制御がいくつかあります。 たとえば、ファイルのダウンロードの制限、またはPOSTメソッドを使用したデータ転送の制限を増やします。 



 <ifModule mod_php.c> php_value upload_max_filesize 32M php_value post_max_size 10M php_value default_charset utf-8 php_value max_execution_time 200 </ifModule>


最初の行では、最大32メガバイトのサイズのファイルをアップロードできます。 デフォルトでは、phpでは通常8または16メガバイトです。

2行目は、最大10メガバイトの投稿を許可します。 デフォルトでは、この値は通常2メガバイトです。

3行目は、スクリプトで使用されるエンコーディングを設定します。 コアでは、「AddDefaultCharset UTF-8」という行を複製します。 しかし、私はしばしばPHPを介してエンコーディングを設定することに頼ります。

4行目は、スクリプトに割り当てられた制限時間を変更します。 デフォルトでは、通常は30秒です。 ただし、複雑な処理を完了するのに時間がかかる場合があります。

私の練習では、ソフトウェアプロバイダーによる更新の後、ファイルの種類が飛んだことが何度かありました。 これはまれですが。 10年間、たった2件。 しかし、時々htmlをphpのように動作させる必要がありました。 また、Apacheに知らないファイルの種類を区別するように教える必要がある場合があります。 （判明したように、Apacheはまれなファイルの種類からほとんど知りません。）この状況では、次のコードで節約できます。 



 AddHandler application/x-httpd-php .html AddHandler cgi-script .pl .py .jsp .asp .htm .shtml .sh .cgi AddType application/x-javascript .js AddType text/css .css AddType text/xml .xml AddType application/octet-stream .doc .mov .avi .pdf .xls # ForceType application/x-httpd-php




最初の行では、phpファイルにhtml拡張子を付けることができますが、phpとして実行されます。 多くの場合に役立ちます。 そして、古き良き時代に、検索エンジンがCNCをより良く索引付けしたとき、そのようなラインは常に救助に来ました。

任意のタイプのファイルを便利な許可に再割り当て、追加、または置き換えることができます。

たとえば、私は書きます 

 AddHandler application/x-httpd-php .roosso

2行目は、cgiスクリプトとして実行するファイル拡張子を示しています。



3行目、4行目、5行目は、念のため、解像度css、js、xmlでファイルのMIMEタイプを指定します。 最初の2行で行われているように、サーバーのリクエストへの応答におけるヘッダーの割り当てと混同しないでください。



6行目はサイトです。サイトにユーザー向けのダウンロード可能なコンテンツがある場合に必要です。 この行を追加すると、これらのファイルにアクセスするときにコンピューターにダウンロードするように求められます。最近のブラウザーでは、ブラウザーで単語を開くプラグインも好きではありません。

7.大人と子供は、アーカイブの重量が少ないことを知っています...



ページの読み込みが高速の場合に役立ちます。 したがって、人々はアーカイブファイルを思いついた。 そして、彼らはまだあなたがウェブサイトと写真のページをアーカイブできると考えて、決定しました。 そして、これをユーザーに気付かれずに行うが、トラフィックを大幅に減らし、ページの読み込みを高速化する。 魔法の言葉を聞いたので、これをどのように行うかについて説明します。



Apacheには2つの圧縮モジュールがあります。 どちらもデフォルトのモジュールではないため、必ずしもプロバイダーに存在するとは限りません。 しかし、実践が示しているように、プロバイダーの99％がそのうちの1つを持っています。 最も一般的なmod_deflate。 これを使用してサイト上のすべてのコンテンツを圧縮するには、次の行を.htaccessに追加します。 



 <ifModule mod_deflate.c> AddOutputFilterByType DEFLATE text/html text/plain text/xml application/xml application/xhtml+xml text/css text/javascript application/javascript application/x-javascript </ifModule>


ご覧のとおり、圧縮する必要があるMIMEタイプファイルをリストする必要があります。 これにビデオと写真を追加できますが、ほとんど使用できません。 jpegまたはgif自体は圧縮形式であるためです。 aviやflvと同じです。 実際に指定しても何も勝ちません。



2番目に人気のないモジュールはmod_gzipで、これを使用して圧縮を有効にするには、次の行を追加します。 



 <IfModule mod_gzip.c> mod_gzip_on Yes mod_gzip_dechunk Yes mod_gzip_item_include file \.(html?|txt|css|js|php|pl)$ mod_gzip_item_include mime ^text\.* mod_gzip_item_include mime ^application/x-javascript.* mod_gzip_item_exclude mime ^image\.* mod_gzip_item_exclude rspheader ^Content-Encoding:.*gzip.* </IfModule>


このモジュールはマスクを使用できますが、これは間違いなく大きなプラスです。 そして、構文は以前のものよりもはるかに柔軟です。 しかし、彼らはそれをあまり使用しません。 また、圧縮によって、どのモジュールが優れているか判断することさえできません。 私はテスト中に大きな違いに気付きませんでした。



8.さらに高速ですか？



できます。 ページキャッシュを適用する場合。 キャッシングには長所と短所があるため、この問題に備える必要があります。 人気のあるフォーラムなど、2〜3分ごとに動的に更新されるサイトの場合、ユーザーが関連情報を表示する必要があることを考慮する必要があります。 しかし、どのサイトにもほぼ静的なコンテンツがあります。 たとえば、同じ写真、またはスタイルファイル。 したがって、サイト上のさまざまなコンテンツの異なるキャッシュを使用する必要があります。 HTMLマークアップでは、常にメタタグを使用できます。 また、phpを使用して、サーバーの応答ヘッダーを設定できます。 問題は、CSS、JS、画像などをどう処理するかです。 など



これに役立つ2つのモジュールがあります。mod_headersとmod_expiresは、サーバー応答にヘッダーを設定し、ブラウザーにキャッシュする方法と方法を指示できます。 通常、モジュールの1つはプロバイダーによって提供されますが、標準のApacheアセンブリに含まれていないモジュールと同様に、100％の保証はありません。 したがって、500番目のエラーを回避するために、各モジュールの状態を示しています。 



 <ifModule mod_headers.c> # html  htm     <FilesMatch "\.(html|htm)$"> Header set Cache-Control "max-age=43200" </FilesMatch> # css, javascript       <FilesMatch "\.(js|css|txt)$"> Header set Cache-Control "max-age=604800" </FilesMatch> #      <FilesMatch "\.(flv|swf|ico|gif|jpg|jpeg|png)$"> Header set Cache-Control "max-age=2592000" </FilesMatch> #  <FilesMatch "\.(pl|php|cgi|spl|scgi|fcgi)$"> Header unset Cache-Control </FilesMatch> </IfModule>


これはmod_headersの構文です。 コメントから、何が何であるかは明らかだと思います。

このセクションでは、phpファイルのキャッシュをオフにしました。 私の意見では、短いキャッシュ間隔はそれらを傷つけませんが。 5〜30秒。これは、ほとんど変化しない時間間隔です。 そして、多くのユーザーはバックキーを使用することを好みます。 それらのページを再度ロードするのではなく、キャッシュから取得するために、妥当なキャッシュ間隔が適切です。



mod_expiresの条件が適用される2番目のセクションでは、まさにそれを行います。phpの場合は、小さなキャッシュ間隔を設定します。 



 <ifModule mod_expires.c> ExpiresActive On #    5  ExpiresDefault "access plus 5 seconds" #      ExpiresByType image/x-icon "access plus 2592000 seconds" ExpiresByType image/jpeg "access plus 2592000 seconds" ExpiresByType image/png "access plus 2592000 seconds" ExpiresByType image/gif "access plus 2592000 seconds" ExpiresByType application/x-shockwave-flash "access plus 2592000 seconds" # css, javascript       ExpiresByType text/css "access plus 604800 seconds" ExpiresByType text/javascript "access plus 604800 seconds" ExpiresByType application/javascript "access plus 604800 seconds" ExpiresByType application/x-javascript "access plus 604800 seconds" # html  htm     ExpiresByType text/html "access plus 43200 seconds" # xml     ExpiresByType application/xhtml+xml "access plus 600 seconds" </ifModule>




9.丁寧な口調の規則...



サイトをアップグレードする過程で、削除したページにつながるリンクの形でテールを残すことがよくあります。 これらのリンクは、他のサイトまたは検索エンジンからのものである場合があります。 ユーザーを失うことがないように、さまざまな種類のエラーに対して独自のページを用意することが丁寧であると考えられています。 サーバーエラーも含めます。 たとえば、500番目のエラーに関する標準メッセージは非常に退屈で悲観的であり、ユーザーが再びページにアクセスしたくなることはほとんどありません。 しかし、htaccessには、このページをより良くするためのトリックがあります。 



 # Bad Rquest ErrorDocument 400 /400.html # Authorization Required ErrorDocument 401 /401.html # Forbidden ErrorDocument 403 /403.html # Not found ErrorDocument 404 /404.html # Method Not Allowed ErrorDocument 405 /405.html # Request Timed Out ErrorDocument 408 /408.html # Request URI Too Long ErrorDocument 414 /414.html # Internal Server Error ErrorDocument 500 /500.html # Not Implemented ErrorDocument 501 /501.html # Bad Gateway ErrorDocument 502 /502.html # Service Unavailable ErrorDocument 503 /503.html # Gateway Timeout ErrorDocument 504 /504.html


400エラーの場合、PHPで動的ページを使用できます。 しかし、500の場合は、htmlとjsで行う方が適切です。 通常、エラーのこの部分はサーバーエラーに関連しており（ほとんどの場合）、この状況では通常phpまたはcgiは機能しません。



あまりにも多くのページを実行するのが面倒な場合は、エラーページをサイトまたはサイトマップのメインページに設定します。



このすべては、あなただけの人々のために行います。 検索エンジンは、エラーページがあるかどうかは気にしません。 エラーコードを含むサーバーの応答を確認し、検索でページをブロックします。 したがって、これは万能薬ではありません。 これは、視聴者とサイトの画像に対する礼儀にすぎません。
