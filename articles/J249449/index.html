Goでのクロスコンパイル

クロスプラットフォームは、ほぼすべての最新の言語およびライブラリの実質的に標準的な属性になっていますが、真のクロスプラットフォーム製品を作成することはまだ容易ではありませんでした。 コンパイルされた言語と関連ライブラリーには、ビルド環境とライブラリーの複雑なインストールと構成が必要であり、インタープリター言語には、インタープリターの必要なバージョンの存在またはデプロイメントが必要でした。 このプロセスを少し単純化しようとする多くのプロジェクトがありますが、多くの場合、唯一の解決策は別個のサーバーをインストールしてネイティブにコンパイルすることでした。



Goで、クロスプラットフォームは、コンパイルファーム、特別に構成された開発環境、アセンブリ用の仮想マシン、またはchroot / docker-devソリューションを初めて安全に放棄できるレベルに達しました。 そして、これはもう一つの重大なゲームチェンジャーです。これについて、例を挙げて話をして、見せたい

行こう







ご存知のように、Goは動的リンクを意図的に拒否しました-その主な理由は、Goのほぼすべての側面の設計に関する通常の説明と非常によく似ています-「[動的リンク]の利点は、その欠点とアーキテクチャにもたらす複雑さよりもはるかに少ないです」 さて、動的リンクの出現の主な理由は、リソースだけでなく、主にディスクスペースとメモリを節約したいという欲求でした。これは、サーバーだけでなく、組み込みデバイス（ヘリコプター、 GB RAM！）。 一般に、個別のリンク方法の長所と短所をリストすると、別の投稿に移動するため、現時点ではそのまま受け入れます。Goでは、出力に常に静的バイナリがあります。



現在、Go 1.4.1の現在のバージョンでは、次のプラットフォームがサポートされています。

  • Linux 2.6 1以降-amd64、386、arm
  • MacOS X 10.6以降-amd64,386
  • Windows XP以降 -amd64、386
  • FreeBSD 8以降-amd64、386、arm
  • NetBSD -amd64、386、アーム
  • OpenBSD -amd64、386
  • DragonFly BSD -amd64、386
  • プラン9 -amd64、386
  • Google Native Client -amd64p32、386
  • Android-アーム


1-カーネル2.6.23以降は公式にサポートされていますが、実際にはすべてがブランチ2.6の以前のカーネルで動作します-たとえば、多くの人が2.6.18のRHEL5 / CentOS5でGoを使用しています。



Go 1.5 は iOS をサポートする予定です。

また、最初はGoでWindowsがサポートされていなかったことも注目に値します-チームは小さく、Windowsのコード実装を手に入れる人はいませんでしたが、オープンソース開発のためにプロジェクトが開かれたおかげで、Windowsの移植はサードパーティの人々によって非常に迅速に書かれ、統合されました公式のコードベースに。



以下で説明するプロセスはすべてのプラットフォームで完全に同一ですが（Androidおよびネイティブクライアント（NaCl）は例外で、追加のジェスチャーが必要です）、この記事の残りの部分では、最も人気のある3つのデスクトップのいずれかを使用していると想定しますプラットフォーム-Linux、MacOS XまたはWindows。 さらに、より簡単にするために、Cライブラリにリンクする必要がない（したがって、cgo / gccを使用する必要がない）Goコードのみを記述して使用することを意味します。 別のケースもあります-cgoに関連付けられた標準ライブラリの多数の関数を使用する必要がある場合ですが、これについては最後の別の章で説明します。



ツールチェーンの準備



実行する必要がある最初のステップは、目的のプラットフォーム用のツールチェーンを構築することです。



Goソースコード（$ GOROOT / src、常にマシン上にあります）を使用してディレクトリに移動し、Windows / amd64などの目的のプラットフォーム用に再構築します。 

cd $(go env GOROOT)/src sudo GOOS=windows GOARCH=amd64 CGO_ENABLED=0 ./make.bash --no-clean


このプロセスは、Macbook Air 2012では約26秒かかります。 make.bashスクリプトは、ソースからインストールする場合にGoをインストールする標準のGoビルドスクリプトです。 実際には、Goと標準ライブラリ全体を収集しますが、今回のみ-windows / amd64プラットフォーム用です。

また、上記の理由により、CGOサポートを無効にしました。

例1. Linux for Windowsで作成およびコンパイルされたWebサーバー



コマンドラインを解析するために、いくつかの言語/ライブラリよりもGoで作成する方が簡単な単純なWebサーバーを作成します。 

 package main import ( "log" "net/http" ) func Handler(w http.ResponseWriter, r *http.Request) { w.Write([]byte("Hello, world\n")) } func main() { http.HandleFunc("/", Handler) log.Println("Starting HTTP server on :1234") log.Fatal(http.ListenAndServe(":1234", nil)) }


そして、32ビット版と64ビット版のWindows用にまとめます。 

 GOOS=windows GOARCH=386 go build -o http_example.exe GOOS=windows GOARCH=amd64 go build -o http_example64.exe


私たちはチェックします： 

 $ file http_example*.exe http_example.exe: PE32 executable for MS Windows (console) Intel 80386 32-bit http_example64.exe: PE32+ executable for MS Windows (console) Mono/.Net assembly


両方のバイナリがターゲットWindowsシステムにコピーする準備ができており、動作するということを言う必要はないと思います。

CGOを理解する



デンバーで開催されたGopherCon 2014カンファレンスのビデオを見た人は、おそらくAlan Shreveの 「Goで開発ツールを構築する」という話を覚えているでしょう。 -コンパイル、ネイティブにコンパイルします。 " 次は説明です-理由はCgoです。 cgoを使用する必要がない場合は、すべて問題ありません。 実際、特定のGoコードのごく一部にサードパーティのCライブラリが必要です。 問題は何ですか？



問題は、標準ライブラリの一部の機能がcgoに依存していることです。 つまり、GoをCGO_ENABLED = 0で収集すると、それらは単に使用できなくなり、コンパイル段階でエラーが発生します。 非常に便利で美しい回避策があるという事実にもかかわらず、標準ライブラリの内容がcgoにどのように依存しているかを見てみましょう。



幸いなことに、これは簡単です。 

 # cd $(go env GOROOT)/src/ # grep -re "^// +build.*[^\!]cgo" * crypto/x509/root_cgo_darwin.go:// +build cgo net/cgo_android.go:// +build cgo,!netgo net/cgo_linux.go:// +build !android,cgo,!netgo net/cgo_netbsd.go:// +build cgo,!netgo net/cgo_openbsd.go:// +build cgo,!netgo net/cgo_unix_test.go:// +build cgo,!netgo os/user/lookup_unix.go:// +build cgo runtime/crash_cgo_test.go:// +build cgo

実用化



ここで主なことについて-それを実践する。 これまでに本番環境で使用したスキームは、「darwin / amd64でビルド-> linux / 386にデプロイ」、「linux / amd64-> linux / 386」、「linux / amd64-> windows / amd64」の3つだけです。 これらは、1年以上完全に機能している製品です。 3番目のケース（Windowsでの展開）は、一般的に驚いた。Linuxで正常に実行されているサーバーがあり、その後突然Windowsで実行する必要が生じた。 さらに、「緊急に必要」。 クロスでの眠れない夜の経験を覚えています-はい、Windowsでの展開のためのQtコンパイルだけで何がありますか-60秒間のプロセス「これを行う方法→ツールチェーンを構築→プロジェクトを再コンパイル→Windowsに展開」-ショックでした私は自分の目を信じられませんでした。



しかし、ここで次の瞬間が発生します-クロスコンパイルと展開が非常に簡単かつ高速になるため、構成ファイル、証明書、その他のファイルに依存してバイナリに埋め込むインセンティブがあります。 ただし、これはio.Readerインターフェイスとgo-bindataパッケージの効率的な使用のおかげで、サードパーティのライブラリでもかなり簡単なタスクですが、これは別の記事のトピックです。



主なものを見逃していないことを願っています。

しかし、全体的に、これは実際には以前のすべてのクロスビルドの経験と非常に大きな違いです。 正直に言うと、私はまだこの変更に慣れていません。 開発環境が構成された仮想マシンは不要になり、アセンブリにドッカーイメージは不要になりました。一般的に、開発環境はそのままでは消えません。 これはゲームチェンジャーとしてはあまりにも鋭すぎてすぐに慣れることができません。
