イシンガ2



IcingaはNagiosのフォークで、2番目のバージョンでゼロから書き直されました。 Shinkenとは異なり、この活発で頻繁に更新されるプロジェクト。







拡張性



一般的なアーキテクチャ：













Icinga 2には、適切に設計された分散監視スキームがあります。 テストクラスタを上げるときに見つけた唯一の欠点は、最も単純な分散回路でさえ難しい初期設定でした。







構成管理システムとの統合



統合は非常に良好です。トピックに関する2つのプレゼンテーションがあります。Tomde VylderのIcinga 2とPuppetによる遅延監視への道と 、Walter HeckのIcinga 2とPuppet：自動監視です。 Icingaの重要な機能は、構成をファイルに保存することです。これにより、Puppetを使用して構成を簡単に生成できます。私の場合、 PuppetDBをすべてのホストおよびサービスに関する情報ソースとして使用します。







アクションロギング



私が発見したように、アクションロギングはdirectorモジュールに表示されます。 現在、IcingaWeb2には組み込みの監査サポートはありません。







UI









IcingaWeb2は、さまざまなニーズに対応する多くのアドオンを備えた優れたUIのように見えます。 私が見たところ、最も柔軟で拡張性があり、同時に、あなたが期待できるすべての機能をすぐに使えるように見えます。







短所



私が出会った唯一の欠点は、初期セットアップの複雑さです。 私の場合、Zabbixのように以前とまったく異なるものを使用した場合、Icingaの監視の見方を理解することは容易ではありません。







ザビックス



Zabbixは、安定した開発率を備えた安定した信頼できる監視システムです。 ユーザーの巨大なコミュニティがあり、あなたが尋ねる質問のほとんどはすでにどこかで答えられているので、Zabbixで可能かどうかを心配する必要はありません。







拡張性



サーバーは単一のデータベースで動作し、アクションに関係なく、手元の他のリソース（メモリ、ネットワーク、CPU）で、ある時点でデータベースが使用するディスクのIO制限に遭遇します。 Amazonで 6000 IOPSを使用すると、1秒あたり約2,000 nvpsの新しい値をサポートします。これは悪くはありませんが、まだ多くの要望が残っています。 データベースのプロキシとパーティション化によりパフォーマンスが向上しますが、フォールトトレランスの観点から、データベースは1つだけであり、これがシステム全体の障害点となります。







構成管理システムとの統合



Zabbixは、構成管理システムによって管理される多様な環境に対して十分に準備されていません。 ホストおよびサービスを低レベルで検出するための機能が組み込まれていますが、制限があり、構成システムに関連付けられていません。 このような統合の唯一の可能性は、APIを使用した独自のソリューションです。







アクションロギング



Zabbixは、1つの盲点を除いて、ユーザーアクションを適切にログに記録します。APIを介して行われた変更は、 ほとんどログに記録されません。 私が言及したいもう一つのことは、Zabbixのすべての問題がバグトラッカーのどこかに記録されており、彼らがコミュニティから十分な注目を集めれば、遅かれ早かれ修正されることです。







UI









ZabbixのUIは便利で、多くの機能が含まれています。 反対に、実際には拡張可能ではありません。標準のダッシュボードが提供するという事実を受け入れるか、独自のダッシュボードを作成します。 標準UIの洗練は、その複雑さのために非常に簡単な作業です。







短所



  • 現時点で何が起こっているかについての基本的な分析のみ（現在の問題ではなく、発信元からの頻度および同様の情報）。 3.0の「トップ100の射撃トリガー」の登場により、状況は劇的に改善されました。
  • Nagiosベースのシステムとは異なり、メンテナンス作業のセットアップはトリガーレベルで設定できず、最近3.2を再設計するまで非常に複雑でした。
  • 箱から出してアラートを生成することは、多くの要望が残っています（ただし、これはすべての統合された監視システムにとって問題です）。 私たちの場合、アラートを集約する外部システムを開発する必要がありました（おそらくいつかはオープンソースで公開されるでしょう）。
  • 適切な専門知識がなくてもパフォーマンスの問題を調査することは、診断する必要がある1つの分割できないサーバーがあるため、混乱になります。
