Banana Pi：U-Boot経由でArch Linuxに

Raspberry Piの発売以来4年連続で、非x86アーキテクチャのさまざまなマイクロコンピューターが、メディアセンター、スマートホームコントローラー、Webサーバーとして機能する世界中の市場に配信されました。



2014年から2015年までに、誰もがRaspberryのコンピューティング機能に満足しているわけではなく、より強力なハードウェアを備えた多数のクローンのリリースが開始されました。 それらのほとんどには欠点があります：カーネルとブートローダーの古いバージョンが修正されていること、ディストリビューションの選択が少ないこと。 カットの下で、Banana Piの例を使用して、ベンダーの束縛を打破する方法について説明します。



背景



2014年の後半にホームサーバーが欲しかったのですが、Intel Atomシリーズプロセッサを搭載したクラシックなx86アーキテクチャと、私にとって新しいもの、ARMベースのマイクロコンピュータのどちらかを選択しました。



価格/性能/機能の比率を評価した後、選択肢はバナナPiに落ちました-デュアルコアAllwinner A20、1ギガバイトDDR3、ギガビットLAN、SATAコネクタなどのデバイスのパン粉です。 数日遊んだ後、私はバナナが投獄されたプラスチック部品からケースを作り、長い間あらゆる種類の小さな仕事を務めました。



ディストリビューションはBananianでした-これは、リソースを節約するために多くの点で切り捨てられたDebian 7 Wheezyです。 今日まで新しいDebian 8.xはありません。セキュリティパッチのみを受け取るソフトウェアがどれだけ古いか想像できます。 このすべての不名誉は真夏まで続きました...







最後に、7月に、ペットプロジェクトのためにWebサーバーと関連サービスを上げる必要がありました。 古いDebianを保守的でなく、私にとってより便利なものに変更する時が来ました。



問題の声明



ディストリビューションにはいくつかの要件がありました。すべてのニーズをカバーする「すぐに使える」リポジトリが必要であり、それらのプログラムは利用可能な最新バージョンに最も近いはずです。 私はディストリビューションとそのプログラムの開発を追っていますが、私はすべてのソースをコンパイルするのが面倒なので、Arch Linuxが当然の選択になりました。



メーカーのウェブサイトのダウンロードセクションで、私は有線の未更新カーネル（3.4.100）、同年齢のブートローダー、および別のデバイス（Cubieboard 2）を使用した1年の期限切れの画像を待っていました。 Arch Linuxサイトでは、Banana Piのサポートは発表されていません。 これらすべてに基づいて、残っていることが1つだけありました。ディストリビューションを自分でビルドして実行することです。



マテリエル







Das U-Boot （名前の由来は、潜水艦とドイツ語と英語のしゃれと何らかの関係があります。Uボートを参照 ）

これはオペレーティングシステムのブートローダーであり、MIPS、PowerPC、ARMなどのアーキテクチャを備えたデバイスに焦点を当てています。 彼は多くのことを知っています。NAND、NOR、SPI、MMC、SATA、USB、TFTP、NFSからシステムを起動し、デバイスのROMまたは外部メモリに格納でき、dhcpクライアントとシェルを内蔵し、キーボードから信号を受信し、UART、HDMIに出力します、VGA、アナログビデオ。 当然、これをすべてサポートするには、ドライバーが必要です。



これがx86プラットフォームとの違いの始まりです-ARMにはACPI（高度な構成および電源管理インターフェイス）およびUEFI（オペレーティングシステムとデバイスの「ファームウェア」間の相互作用インターフェイス）がないため、ARMコンピューターには統一されたディストリビューションはありませんが、その動きはUEFIとACPIの側は数年間続いています。 それまでの間、この動きは続いており、私たちは現実の厳しい世界にいます。ローダーはさまざまな種類のハードウェアに合わせて個別にコンパイルする必要があります。 鉄はと呼ばれるデータ構造で記述されます デバイスツリー（dt） DTの詳細については、専用サイトdevicetree.orgをご覧ください 。



ブートローダーのソースコードを完備し、コンパイル用の設定ファイルのセットと1089デバイスのデバイスツリーが付属しています。 残念ながら、Googleはこのリストを見つけられないので、ここから入手できます。 U-boot対応デバイス 。







Linuxカーネル



彼については多くのことが書かれており、多くのことが言われてきたが、今ではいくつかの事実だけが役に立つだろう。

  1. 私が出会ったディストリビューションは、カーネルをいくつかの形式で保存しています。

    • 画像は圧縮画像ではありません。
    • zImageは、圧縮された自己解凍画像です。
    • uImage-U-Bootの情報を含むラッパーを含むイメージ。


    以前は、uImageはU-Bootを備えたデバイスで広く使用されていましたが、近年、開発者はブートローダーにzImageを使用することを教えています。
  2. 良いニュースです！ 同じアーキテクチャ（ARMv5、ARMv6、ARMv7、ARMv8）内にある可能性の高いコアは、デバイスごとにコンパイルする必要はありません。 というバージョンのデバイスツリーを使用します デバイスツリーバイナリ（dtb） これらは、どのモジュールをロードする必要があるかをカーネルに示す小さなファイルです。 dtbの詳細については、 こちらをご覧ください 。



理論から実践へ



あなたが必要になります：

  • Banana PiまたはU-Bootがサポートするデバイスのリストにある別のマイクロコンピューター
  • メモリカードとカードリーダー
  • Linuxコンピューター
  • 目的のアーキテクチャにインストールされたツールチェーン

    入手先
    最も簡単な方法は、次のことを検討することです。

    1. プラットフォーム用の既製のツールチェーンをLinaro Webサイトからダウンロードします。

      Banana Piにはlinaro-toolchain-binaries（リトルエンディアン）が必要です。

    2. ツールバーのbinフォルダーをPATHに追加します。 

      export PATH=~/your_toolchain_path/bin/:$PATH






  • bcおよびdtcユーティリティは、それぞれファイルシステムの作成およびブートローダースクリプトのコンパイルのために、ブートローダー、dosfstoolsおよびuboot-toolsパッケージをコンパイルするために必要でした。



U-Bootの最新バージョンのソースコードをダウンロードして解凍します。 

 wget ftp://ftp.denx.de/pub/u-boot/u-boot-latest.tar.bz2 tar -jxf u-boot-latest.tar.bz2


表示されるu-boot-2015.07 / configsディレクトリに「鉄片」を見つけて、ファイル名を記憶します。 Banana Piの場合、構成ファイルはBananapi_defconfigと呼ばれます 。



Arch Linux ARMディストリビューションのコアはzImageに保存されていることを覚えていますか？ 事前にこれを処理し、コンパイルする前に自己解凍カーネルをロードする機能を追加する必要があります。 これがデフォルトで既に有効になっているかどうかについての正確な情報はありません。したがって、 u-boot-2015.07 / include / configs /ディレクトリで、ご使用のマイクロコンピュータープラットフォームのヘッダーファイルを見つけてください（Banana Piの場合はsun7i.h ）。 

 #ifndef __CONFIG_H #define __CONFIG_H ... #endif /* __CONFIG_H */


行 

 #define CONFIG_CMD_BOOTZ


これにより、ブートローダーでbootzコマンドを使用して、自己解凍圧縮プログラムを実行する機能が追加されます。 出所



これでコンパイル段階です。 すべての操作はu-boot-2015.07ディレクトリで行われます 。

クロスコンパイルしないとどうなりますか？
同じアーキテクチャ（たとえば、ARMv7を搭載したコンピューター）でコンパイルする場合、makeではCROSS_COMPILEフラグを省略できます。 



 make CROSS_COMPILE=arm-linux-gnueabihf- Bananapi_defconfig


ブートローダーを直接コンパイルします。 

 make CROSS_COMPILE=arm-linux-gnueabihf-


その結果、u-bootプレフィックスを持つ多くのファイルが作業フォルダーに表示されるはずです。 u-boot.binにはブートローダー実行可能ファイルが含まれていますが、「with-spl」という接尾辞が付いたファイルが必要です。 SPLは、ブートローダーの直前にある小さなプログラムで、U-Boot自体をRAMにロードします。 Banana Piの場合、これはu-boot-sunxi-with-spl.binファイルです 。 保存しておけば、後で便利になります。



次のステップは、システムが起動するフラッシュカードを準備することです。 レイアウトは次のようになります。







カードリーダーを介してフラッシュカードを接続し、fdiskユーティリティを実行します。 

 fdisk /dev/sdb


カードの名前を調べる方法は？
挿入されたカードに対応するディレクトリ/ dev /に新しいファイルが表示されます。 SCSIデバイス（sda、sdb、sdc ...）またはブロックMMCメモリ（mmcblk0、mmcblk1 ...）として定義されます。



MBRマークアップを追加します。 

 Command (m for help): o Created a new DOS disklabel with disk identifier 0xa53166ce.


そして、少なくとも2つのセクションを確認してください。

  1. 50-100 Mb

    fdisk、最初のマークアップ 
     Command (m for help): n Partition type p primary (0 primary, 0 extended, 4 free) e extended (container for logical partitions) Select (default p): p Partition number (1-4, default 1): 1 First sector (2048-7831551, default 2048): 2048 Last sector, +sectors or +size{K,M,G,T,P} (2048-7831551, default 7831551): +75M Created a new partition 1 of type 'Linux' and of size 75 MiB.




  2. 2 GB以上

    fdisk、2番目のマークアップ 
     Command (m for help): n Partition type p primary (1 primary, 0 extended, 3 free) e extended (container for logical partitions) Select (default p): p Partition number (2-4, default 2): 2 First sector (155648-7831551, default 155648): 155648 Last sector, +sectors or +size{K,M,G,T,P} (155648-7831551, default 7831551): +3G Created a new partition 2 of type 'Linux' and of size 3 GiB.






このすべての後、「w」を入力して、変更をディスクに書き込みます。



なぜそのようなセクションですか？ U-Bootはスクリプトを使用して制御でき、ドライブの最初のパーティションからのみ読み取ります。このセクションはFATファイルシステムである必要があります。 同じセクションで、将来のArch Linuxの/ブートディレクトリがマウントされ、U-Bootスクリプトからカーネルにアクセスしやすくなります。また、出会ったコンピューターからカーネルをすばやく交換できます。 2番目のセクションは、システムのルート用です。 残りはオプションです。



マークした後、ブートローダー用のスペースをクリアする必要があります。 最初の512バイト（MBR）を上書きせず、さらに1024セクターに影響を与えないことが非常に重要です（上記の表に基づく）： 

 dd if=/dev/zero of=/dev/sdb bs=1K count=1023 seek=1


U-Bootを作成します。 

 dd if=u-boot-sunxi-with-spl.bin of=/dev/sdb bs=1024 seek=8


これで、USBフラッシュドライブを宛先デバイスに挿入し、電源を入れてLEDで点滅する方法を確認し、dhcpでIPアドレスを取得し、tftpサーバーに到達して起動を試み、出力またはUARTで表示してシェルに招待します。



しかし、1つのブートローダーでは十分ではなく、U-Bootを管理するスクリプトとArch Linux自体の配布用のコンポーネントがまだいくつかあります。

os.archlinuxarm.orgにアクセスし、アーキテクチャに適したディストリビューションのアーカイブをダウンロードします。 バナナの場合、これはArchLinuxARM-armv7-latest.tar.gzです。 別のディレクトリに展開し、フラッシュカードの操作を続けます。



ファイルシステムを作成し、指定された順序でマウントしましょう。 

 mkfs.vfat /dev/sdb1 mkfs.ext4 /dev/sdb2 mount /dev/sdb2 /mnt mkdir /mnt/boot mount /dev/sdb1 /mnt/boot


解凍したアーカイブの内容全体を/ mntにコピーします。 

 cp -r ~/ArchLinux-generic-armv7/* /mnt


フラッシュドライブの最初のセクション（/ mnt / boot）を見てみましょう。 内部では、dtbsフォルダーと圧縮されたカーネルイメージ-zImageを待機しています。 dtbsはまさに上記で説明したデバイスツリーバイナリです。

ARMv7の場合、301 dtbもあります。 このリストは、 ここで表示するか、コンピューターのディストリビューションのboot / dtbsディレクトリを直接開くことで確認できます。 dtbがある場合、カーネルはほぼ100％の確率で起動します。 Banana Piの場合、目的のファイルはsun7i-a20-bananapi.dtbです。 U-Boot用のスクリプトを作成し、当然のことながら、結果のシステム全体を起動する最終段階と最終段階に必要です。



始めましょう。 boot.cmdファイルを適切な場所（ / mnt / boot ）に作成し、書き込みを開始します。



最初に、カーネルイメージをRAMにロードするようU-Bootに指示します。 

 fatload mmc 0 0x46000000 zImage


次に、目的のデバイスのdtb： 

 fatload mmc 0 0x49000000 dtbs/sun7i-a20-bananapi.dtb


カーネルを起動するためのパラメーターを設定します。 

 setenv bootargs console=ttyS0,115200 earlyprintk root=/dev/mmcblk0p2 rw rootwait panic=10


そして、このメモリ領域からブートするように彼に伝え、それによってLinuxカーネルに制御を移します。 

 bootz 0x46000000 - 0x49000000


各モデルでダウンロードできるメモリ領域は異なりますが、残念ながら自分でそれらを決定する方法はわかりません。 誰かがコメントの中で簡単な言葉でそれについて語ってくれたらとてもうれしいです。



作成したスクリプトはコンパイルのみ可能です。 このために、uboot-toolsパッケージにはmkimageユーティリティがあり、それを使用します。 

 mkimage -C none -A arm -T script -d boot.cmd boot.scr


必要に応じてboot.cmdを削除できます。ブートローダーに必要なのはboot.scrのみです。

boot.scrのようなものはありますか？
はい、そうです。 一部のデバイスのU-Bootは、単純なテキストファイルuEnv.txtで構成できます。これはベンダーによって異なりますが、Banana Piではこのトリックは機能しません。


最後のタッチ-起動時に接続される新しいシステムのetc / fstabにセクションを追加します。 

 /dev/mmcblk0p2 / ext4 rw,relatime,data=ordered 0 1 /dev/mmcblk0p1 /boot vfat rw,relatime,fmask=0022,dmask=0022,codepage=437,iocharset=iso8859-1,shortname=mixed,errors=remount-ro 0 2


フラッシュドライブをデバイスに挿入して、新しいシステムを楽しむことができます。 ネットワークにDHCPサーバーがある場合、dhcpdデーモン自体がアドレスを取得します。 SSHサーバーはデフォルトで自動ロードに追加されます。 Arch Linux ARMのデフォルトイメージのアカウントは常に唯一の1つです：root / root。 必ずパスワードを変更し、制限された権限を持つユーザーを追加してください。







まとめ



何を得たの？ 私のマイクロコンピューターのサポートは、U-Bootでサポートされている1089デバイスのリストにあり、ARMv7のメインラインでサポートされているカーネルの301番目にあります。 したがって、まさに私たちが望んでいたものを手に入れました。システムのカーネル（現在4.1.4）を含むリポジトリから安定的に更新される最新のソフトウェアを備えたシステムです。 安定したブートローダーブランチは3か月ごとに更新され、上記のようにコンパイルしてインストール手順を繰り返すだけで済みます。



すぐに動作しないものは何ですか？
カーネルとディストリビューションには、Maliドライバーのカーネルまたはユーザースペース部分がありません。 これは、完全に機能するビルドが、ARMパートナーにのみ提供される独自のSDKを使用することでのみ可能であるという事実によるものです。
