NoSQLデータベース：本質を理解する

最近、「NoSQL」という用語は非常に流行し人気があり、あらゆる種類のソフトウェアソリューションがこの看板の下で積極的に開発および宣伝されています。 NoSQLは、大量のデータ、線形スケーラビリティ、クラスター、フォールトトレランス、非リレーショナルと同義語になりました。 ただし、NoSQLリポジトリとは何か、用語がどのように表示されるか、どのような共通の特徴があるのか​​を明確に理解している人はほとんどいません。 このギャップを解消してみましょう。







物語。



この用語で最も興味深いのは、90年代後半に最初に使用されたという事実にもかかわらず、2009年の半ばにしか使用されなかったという意味では、本当の意味が得られなかったということです。すべてのデータをASCIIファイルとして使用し、SQLの代わりにシェルスクリプトを使用してデータにアクセスしました。 現在の形式の「NoSQL」では、何の関係もありませんでした。



2009年6月、Johann Oscarssonはサンフランシスコで会議を開催し、ITストレージおよび処理市場の新しいトレンドについて議論しました。 会議の主な刺激は、BigTableやDynamoなどの新しいオープンソース製品でした。 会議の明るい兆候を得るには、Twitterハッシュタグに完全に適合する容量のある簡潔な用語を見つける必要がありました。 そのような用語の1つが、RackSpaceのEric Evansによって提案されました-「NoSQL」。 この用語は1回の会議でのみ計画されており、意味的な負荷はあまりありませんでしたが、たまたまバイラル広告のようにグローバルネットワーク全体に広がり、IT業界全体の事実上の名前になりました。 ちなみに会議では、Voldemort（Amazon Dynamoのクローン）、Cassandra、Hbase（Google BigTableのアナログ）、Hypertable、CouchDB、MongoDBが話しました。



「NoSQL」という用語は完全に自発的な起源を持ち、その背後に普遍的に認識されている定義や科学機関がないことを再度強調する価値があります。 この名前は、リレーショナルデータベースとは別にIT開発のベクトルを特徴付けるものです。 SQLだけではありませんが、No SQLの直接定義の支持者はいますが、これはSQLだけではありません。 Pramod SadalajとMartin Fowlerは、最近の本「NoSQL Distilled」で、NoSQLの世界に関する知識をグループ化し体系化しようとしました。



NoSQLデータベース機能



多くの異なるシステムがNoSQLラベルの下に隠れているため、すべてのNoSQLに共通する特性はほとんどありません（最も完全なリストは、おそらくhttp://nosql-database.org/で見つけることができます）。 多くの特性は、特定のNoSQLデータベースにのみ固有のものです。これについては、リストの際に必ず言及します。



1. SQLは使用されません



これはANSI SQL DMLを指します。これは、多くのデータベースがよく知られているお気に入りの構文に類似したクエリ言語を使用しようとしますが、完全に実装することができず、成功する可能性が低いためです。 SQLを実装しようとしているスタートアップがあるという噂もありますが、たとえば、ブートアップ（ http://www.drawntoscalehq.com/およびhttp://www.hadapt.com/ ）



2.非構造化（スキーマレス）



つまり、NoSQLデータベースでは、リレーショナルデータベースとは異なり、データ構造は規制されていません（または、プログラミング言語との類似性を引き出す場合、型指定が不十分です）-テーブル全体の構造を宣言的に変更することなく、別の行またはドキュメントに任意のフィールドを追加できます。 したがって、データモデルを変更する必要が生じた場合、唯一の十分なアクションは、アプリケーションコードの変更を反映することです。



たとえば、MongoDBのフィールドの名前を変更する場合： 



BasicDBObject order = new BasicDBObject(); order.put(“date”, orderDate); //     order.put(“totalSum”, total); //     “sum”


アプリケーションロジックを変更すると、読み取り時にも新しいフィールドが必要になります。 しかし、データスキームがないため、totalSumフィールドは他の既存のOrderオブジェクトにはありません。 この状況では、さらにアクションを実行するための2つのオプションがあります。 1つ目は、すべてのドキュメントをバイパスし、既存のすべてのドキュメントでこのフィールドを更新することです。 データ量が多いため、このプロセスはロックなしで発生し（alter table rename columnコマンドに相当）、したがって、更新中に既存のデータを他のプロセスで読み取ることができます。 したがって、2番目のオプション（アプリケーションコードでの検証）は避けられません。 



 BasicDBObject order = new BasicDBObject(); Double totalSum = order.getDouble(“sum”); //    if (totalSum == null) totalSum = order.getDouble(“totalSum”); //




そして、再記録するとき、このフィールドを新しいフォーマットでデータベースに書き込みます。



スキーマがないことの良い結果は、スパースデータの効率です。 1つのドキュメントにdate_publishedフィールドがあり、2番目のドキュメントにない場合、空のdate_publishedフィールドは2番目に作成されません。 これは、原則として、論理的ですが、あまり明白ではない例です。つまり、テーブル/列というおなじみの概念を使用する列ファミリNoSQLデータベースです。 ただし、スキームがないため、列は宣言的に宣言されておらず、データベースとのユーザーセッション中に変更/追加できます。 これにより、特に動的列を使用してリストを実装できます。



非構造化スキームには欠点があります-データモデルを変更する際のアプリケーションコードの上記のオーバーヘッドに加えて、ベースからのあらゆる種類の制限（非ヌル、一意、チェック制約など）がなく、さらに構造の理解と制御にさらに困難があります異なるプロジェクトのデータベースとの並列作業中のデータ（データベース側に辞書はありません）。 ただし、急速に変化する現代の世界では、このような柔軟性は依然として利点です。 例としては、Twitterが5年前にツイートとともに追加情報（時間、Twitterハンドル、および数バイトのメタ情報）のみを保持していましたが、現在では、メッセージ自体に加えて、数キロバイトのメタデータがデータベースに保存されています。



（以下、主にキー値データベース、ドキュメントデータベース、列ファミリデータベースに関するものです。グラフデータベースにはこれらのプロパティがない場合があります）。



3.集計形式でのデータの表示。



正規化のためにアプリケーションの論理ビジネスエンティティをさまざまな物理テーブルに格納するリレーショナルモデルとは異なり、NoSQLストアは、統合オブジェクトと同様にこれらのエンティティで動作します。







この例では、eコマースの「概念-注文-注文-支払い-製品」の概念的なリレーショナルモデルの集計を示します。 どちらの場合も、注文は1つの論理オブジェクトの位置と組み合わされ、各位置には製品へのリンクとその属性の一部、たとえば名前が含まれます（注文の取得時に製品オブジェクトを要求しないように非正規化が必要です-分散システムの主なルールは最小ですオブジェクト間の「結合」。 1つのユニットでは、支払いは注文と結合され、オブジェクトの不可欠な部分です。別のユニットでは、支払いは別のオブジェクトに取り出されます。 これは、NoSQLデータベースのデータ構造を設計する主なルールを示しています。アプリケーションの要件に従って、最も一般的なクエリに対して可能な限り最適化する必要があります。 支払いが注文とともに定期的に取得される場合、それらを共通のオブジェクトに含めることは理にかなっています。多くのリクエストが支払いのみで機能する場合は、それらを別のエンティティに配置することをお勧めします。



クエリが集計の構造に収まらない場合、データに対して任意のクエリを試行すると、多くの場合、非正規化された大規模なオブジェクトでの作業には多くの問題が伴うことに注意してください。 注文と広告申込情報を一緒に使用して（これがアプリケーションの仕組みです）、ビジネスから先月販売された特定の製品のユニット数を計算するように求められたらどうなりますか？ この場合、OrderItemテーブルをスキャンする代わりに（リレーショナルモデルの場合）、NoSQLストレージで注文全体を取得する必要がありますが、この情報のほとんどは必要ありません。 残念ながら、これは分散システムで行う必要のある妥協です。従来の単一サーバーシステムのようにデータを正規化することはできません。異なるノードからのデータを結合する必要が生じ、データベースの速度が大幅に低下する可能性があるためです
