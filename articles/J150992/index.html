Subversionを使用したCachéスタジオでの開発



Caché-SVNと呼ばれるCachéスタジオ拡張が開発されました。 この拡張により、Cachéスタジオを離れることなくSVNリポジトリ（リポジトリ）を操作できます。 これにより、チェックアウト、コミット、および更新を実行できます。



Caché-SVNは、個々のファイル（クラス、プログラム、cspページなど）ではなく、プロジェクト全体で機能します。



Caché-SVNは、Cachéがインストールされているサーバー上に一時フォルダを作成し、そこにプロジェクトファイルをアップロードして、svn commitを実行します。 同様に、svn updateを使用すると、結果のxmlファイルの説明がCachéにインポートされます。





Caché-SVNは、Cachéに付属する％Studio.Extension.Baseクラスに基づいており、独自のメニューをCachéスタジオに追加できます。



入手先



Caché-SVNは、 subversion.assembla.com / svn / intersystems / cache-svn / svn-0.26.xml.zipからダウンロードできます。



インストール方法



実際にSVN


（Cachéスタジオが実行されているマシン上ではなく）Cachéが配置されているサーバー上で、svn.exeファイルが使用可能でなければなりません。



コミットのコメントはUTF-8エンコーディングでサーバーに送信されるため、svn設定構成ファイルでパラメーターlog-encoding = UTF-8を設定する必要があります。



Tortoise SVNをインストールする場合、インストール中に「バッチファイルのインストール」ボックスをチェックする必要があります。



Caché-svn



インストール時に、CACHELIBデータベースへの書き込みアクセスを有効にする必要があります。 これは、システム管理ポータルを介して行われます。 [構成]> [システム構成]> [ローカルデータベース]を選択します。 CACHELIBの行の[編集]リンクをクリックし、ドロップダウンリストの[読み取り専用]で[いいえ]を選択します。 Caché-SVNをインストールした後、フラグを返すことができます。



Cachéスタジオで、％SYS領域を選択し、Caché-SVNを使用してファイルをインポート（ "ツール>ローカルにインポート"）します。 すべて、再びCACHELIBへの書き込みを禁止できます。



ここで、Caché-SVNがバージョン管理に使用される領域を確立する必要があります。 これを行うには、システム管理ポータルで、[構成]> [詳細設定]> [バージョン管理システム]を選択します。 必要な領域について、class％SourceControl.SVNをマークし、[OK]をクリックします。 このチュートリアルの例はSAMPLESスコープで動作するため、％SourceControl.SVNクラスをバージョン管理クラスとして選択します。 インストール後、Cachéスタジオで、Caché-SVNがバージョン管理システムとして選択されている領域を選択すると、[バージョン管理システム]メニューが表示されます。



最初のチェックアウトの方法



最初のプロジェクトをSVNにアップロードしましょう。 これを行うには、a）プロジェクトとb）svnリポジトリが必要です。 プロジェクトとして、SAMPLESエリアから映画プロジェクトを取り上げます。 ローカルリポジトリを作成するか、プライベートまたはパブリックsvnリポジトリを提供するサーバーを選択できます。 この例では、assembla.comサイトを使用します（はい、これは広告です）。



そのため、assembla.comにサインアップしてSVNリポジトリを作成します。 Assemblaがトランクフォルダーを作成します。ソースをそこに配置しましょう。 さらに、リポジトリがsubversion.assembla.com/svn/cache-cinema/trunkにあると仮定します。



Cachéスタジオで[サンプル]領域を開きます。 Caché-SVNを正しくインストールして構成した場合、スタジオには「バージョン管理システム」メニューが表示されます。



Cinemaプロジェクトを開き、[プロジェクト]> [設定]メニューの[説明]セクションで、svn-repo = https：//subversion.assembla.com/svn/cache-cinema/trunkの行を追加します。 プロジェクトを保存します（「ファイル>プロジェクトの保存」）。 したがって、リポジトリーのパラメーターはプロジェクトにバインドされます。 これで、プロジェクトをエクスポートするときに（メニュー[ツール]> [エクスポート]）、この行はそれとともにアンロードされます。



次に、ローカル設定を行う必要があります。 メニュー項目「バージョン管理システム>プロジェクト設定」を開き、書き込みます
  • svn.exeファイルへのパス。 svn.exeファイルがsystem％PATH％にない場合、フルパスを指定する必要があります（たとえば、c：\ svn \ bin \ svn.exe）。 さらに、SVN設定が保存されているフォルダーを明示的に指定する必要があります。 これは--config-dirオプションで実行されます。 その結果、行は次のようになります。svn.exe--config-dir c：\ svn \ conf。 このパスはすべてのプロジェクトに使用されます。
  • 一時フォルダー。 このフォルダには、SVNからロードおよびアップロードされるxmlクラスの説明が保存されます。 フォルダー構造は次のとおりです：AreaName \ ProjectName \ project files。 プロジェクトをチェックアウトするたびに、対応するフォルダーの内容がクリアされます。 このフォルダーは、すべてのプロジェクト用です。
  • パラメーター「SVNリポジトリーへのパス」はこのウィンドウでは編集されず、プロジェクトの説明から取得されます。 ただし、ここで正しく指定したかどうかを確認できます。
  • パラメーター「すべてのクラスを1つのフォルダーに入れる」は、all-classes-in-one-folderの行に対応します。これは、svn-repoと同様に、プロジェクトの説明で指定されます。 デフォルトでは、このフラグはチェックされず、すべてのクラスの説明はパッケージに従ってサブフォルダーにグループ化されます。 プロジェクトの説明にall-classes-in-one-folder = 1という行を追加すると、すべてのクラスの説明は同じフォルダーになります。
  • ユーザー名-SVNユーザー名
  • パスワード-SVNユーザーパスワード
  • SVNがパスワードをキャッシュする場合（通常はパスワードをキャッシュします）、オプション「最初の使用後にパスワードを削除する」を設定できます。 この場合、最初の使用（通常はチェックアウト）後のパスワードはサーバーから削除され、将来的にはキャッシュからSVN自体によって取得されます。






ユーザー名、パスワード、パスワード削除フラグは、プロジェクト名とともにノードの^ SVNグローバルの対応する領域に保存されます。



svn.exeへのパスと一時フォルダーは、％SYSエリアの^ SVNグローバルに保存されます。



そのため、すべての準備が完了しました。最後に残っているのは、メニュー項目「バージョン管理システム>チェックアウト」を選択することです。 すべてを正しく実行すると、出力ウィンドウに次の行が表示されます。





コミット方法



メニュー項目「バージョン管理システム>コミット」を選択します。 「初期アップロード」などのコメントを入力し、「はい」をクリックします。



「出力」ウィンドウで行が実行され、すべてがこの図で正常に終了します。





リポジトリに接続する方法



じゃあ プロジェクトをアップロードしました。 今、別の開発者がそれに接続したいことを想像してください。 別のバージョンのCachéを配置しないために、この開発者は、SAMPLESエリアではなく、USERエリアで作業していると想定します。 ％SourceControl.SVNをUSER領域にバージョン管理システムのクラスとしてインストールします（まだ実行していない場合）。



リポジトリに保存されているプロジェクトと同じ名前で新しいプロジェクトを作成し、構成します。リポジトリのアドレス、ログイン、パスワードを指定し、チェックアウトを実行します（「バージョン管理システム>チェックアウト」）。





最後の行に注意してください。リポジトリに保存されているプロジェクトの名前とは異なる名前でプロジェクトを作成した場合、リポジトリのプロジェクトはロードされたままになります。 フィールドでそれを見つけ、それを開き、設定し、すでにチェックアウトする必要があります。



また、SAMPLESエリアでは、cspファイルがcsp / samplesアプリケーションにあったことに注意してください。 USER領域にインポートすると、すべてのcspファイルがUSER領域のデフォルトのcspアプリケーション-csp / userに移動されます。



チームワーク



プロジェクトは両方の領域にロードされています。 2人のユーザーがプロジェクトでどのように連携するかを見てみましょう。



USER領域で、Search.cspファイルを開き、そのタイトルを「Search page title」に変更します。 他の何かを変更するか、プロジェクトに新しいクラス/プログラムを追加する場合。 コミットします。





ほら 1つのファイルのみを変更し、9つが送信されました。 Cinemaプロジェクトに関する情報（Cinema.prj.xmlファイル）が変更されました。これは、サーバーに保存されているプロジェクトの説明で、cspファイルがcsp / samplesアプリケーションにあり、cspファイルがcsp / userアプリケーションにあるためです。



したがって、すべてのcspファイルの説明が送信されます。 xmlの説明には、このファイルが属するcspアプリケーションを示す行があります。 アプリケーションが変更されました-ファイルをサーバーに再度送信する必要があります。



ここで再びページSearch.cspのタイトルを変更してコミットすると、Search.cspのみが送信されます。cspアプリケーションのcsp / samplesからcsp / userへの変更はすでにコミットされています。





更新方法



次に、SAMPLESエリアに変更をアップロードしましょう。 それに移動し、更新を実行します（「バージョン管理システム>更新」）。 Search.cspページのタイトルが変更されました。 Caché-SVNは、起動時にプロジェクトを再コンパイルしません。 あなたは自分でそれをする必要があります。



紛争解決



更新またはコミット中に競合が発生した場合、SVNはファイルに変更を加えたトークンを挿入します。 クラス（またはプログラム、またはcspページ）の内容は、これらのマーカーとともにCachéにロードされます。 場合によっては、マーカーがファイル記述のxmlマークアップを破壊し、これらのファイルのインポートが不可能になります。 次に、xml記述が一時プログラムに完全にロードされます。



例1


SAMPLESエリアで、Cinema.Filmクラスを開き、TopFilmsクエリで、条件「AND（Rating = 'G'）」をWHEREに追加します。 コミット。



次に、USERエリアを開き、同じCinema.Filmクラスで、同じTopFilmsクエリで、WHERE-AND（Rating in（ 'G'、 'PG'））に別の条件を追加します。 更新を実行します。





ファイルに必要な行のみを残した後、メニュー項目「バージョン管理システム>競合を解決済みとしてマーク」を選択します。 開いているファイルの競合は閉じられ、コミットを妨害しません。 コミット。



例2


SAMPLESエリアで、Cinema.Filmクラスを開き、TopCategoryクエリの最初の行で「Top 3」を「Top 6」に変更してコミットします。 USERエリアで、同じCinema.Filmクラスを開き、同じTopCategoryクエリで「Top 3」を「Top 7」に変更して、更新を行います。





この更新によるxml記述のマークアップは壊れ、インポートは失敗します。 この場合、一時プログラムDIFF.Cinema.Film.cls.xmlが作成され、インポートできなかったxmlファイルがそこにロードされます。





Cinema.Film.xlsファイルの競合が解決済みとしてマークされると、この一時プログラムは削除されます。



その他のメニューコマンド



以下のCaché-SVN機能は言及されていません。

  • [変更を元に戻す]メニューを選択すると、いつでも最後の更新後に行われた変更をロールバックできます。
  • [変更履歴]メニューからコミットに関するコメントを表示できます。
  • [リビジョンに更新]メニューを使用して、前のリビジョンに戻ることができます。 リビジョン番号を指定する必要があります。
