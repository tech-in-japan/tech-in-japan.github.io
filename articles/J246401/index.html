実際の影響マッピング





インパクトマッピングを初めて読んだとき、途中でドロップしたいという思いがありました。 そこに書かれていることはすべて明白すぎる。 この本は短く、大きな写真があるので、私は自分自身に強さを見出し、読み終えました。 後で判明したように、すべての塩は、私が私の仕事の本からこれらの明白でシンプルなプラクティスをすべて適用しなかったことでした。



顧客は、プロジェクトの公式文書に目標を書いたことがあります。 時々、私は顧客の目標をすでに理解しているように思われました-それらは絶対に明白です。 明らかなことを明確にする理由 インパクトマッピングを仕事に適用し始めたとき、違いを感じました。







影響マッピング履歴





以前は、プロジェクトの開始時に、技術仕様、システム操作スキーム、そして良いバージョンではインターフェイスプロトタイプがありました。 これらの文書には、プロジェクトの開発のダイナミクスと作業の優先順位に関する理解が欠けていました。



ユーザーストーリーの作成とストーリーマッピングを開始しました。 これらのプラクティスにより、プロジェクトの開発方法、現在の優先事項について理解が深まり、顧客とより効果的にコミュニケーションをとる機会が与えられました。 何が欠けていましたか？ 製品は真空状態では存在しません。システムの使用履歴のどこかにあるグローバルタスクをもっと見る必要があります。 プロジェクトの目標を設定するのに十分な単純なゲームの練習がありませんでした。そこから後のストーリーマッピングとユーザーストーリーリストが表示されます。



Mijo BalicとIngrid Otterstenは2007年にEffect Effect ITという記事を書きました（ Effect Mapsを使用したアジャイル製品管理についての詳細）。 2011年の4年後、 Gojko AdzicはExample Specification by Exampleをリリースしました。「目標からスコープを導出する」の章で、エフェクトマッピングと呼ばれる手法について言及しています。 この手法は、チームがビジネス目標に焦点を合わせ、利害関係者とそのニーズを特定できるように設計されています 。



Gojko Adzicは、目標と効果の優先順位付け、Whatレベルでの技術的詳細からの移行、仮定と実験の周期的な性質など、効果マッピングに時間の経過とともにいくつかの改善を追加します。 個人的には、これらは重要な変更であり、実際の生活に有用性を追加するように思えます。 その後、この手法は新しい方法で呼び出されるようになりました-インパクトマッピング。



1キログラムのコードはいくらですか？





状況を想像してください。 顧客が来ます。 彼はすでに購入する機能のセットを持っています、彼は彼が望むものを知っています。 彼は店のように買い物かごを持ちます。 テクノロジーのリスト、多数のインターフェースプロトタイプ、ソーシャルサービスとの統合を追加します。 ネットワークなど 彼はチェックアウトカウンターに近づき、すべてを計量し、それを認識し、彼に請求します。







顧客は、彼の問題のいくつかに対する既成の解決策であなたのところに来たことがわかります。 このような状況では、開発者の手だけが仕事をし、頭ではありません。 開発者は、私たちが下すすべての決定を批判的に見ることはできません。



そのようなプロジェクトは成功するでしょうか？ クライアントが実際のビジネスを行っており、プロジェクトが「テーブルで」行われていない場合、成功はビジネスに与えられた効果によってのみ評価できます。 提供される機能の数、締め切り、予算、またはビジネスの変化によってではありません。



率直に言って、プロジェクトが成功することを保証することは困難です。 しかし、チームの全員がビジネスの目標を理解し、共有するという事実により、成功の可能性を高める力があります。 その後、ビジネスの実際のニーズに基づいて、コード内の変数の命名からアーキテクチャの選択まで、あらゆる決定が行われます。



問題は、顧客から達成したい実際のビジネス目標をどのように得るかです。 チームに彼らの声を聞いてもらい、受け入れて協力を開始する方法は？



インパクトマッピングの作成





インパクトマッピングは、目標を達成するために顧客のビジネスを後押しする影響のマップを備えたプロジェクト目標のマインドマップです。







なんで？

重要な質問に答えるマップの中心的な要素： なぜこれを行うのか？ これは、ビジネスが達成しようとしている目標です。



誰？

最初のレベルでは、次の質問に答えます： 誰が希望する結果を達成できるか 誰が干渉できますか？ 当社製品のユーザーは誰ですか？ これには、ビジネスの目標に影響を与える可能性のあるすべての関係者が含まれます。



どうやって？

第2レベルでは、ビジネスが目標を達成するために利害関係者が持つ必要のある影響を説明する必要があります。 私たちは、質問に対する答えを探しています。ビジネスが目標を達成するのにどのように役立つのでしょうか？ 彼らはどのようにプロジェクトの成功を妨げることができますか？



なに？

基本的な質問に答えた後、特定のタスクについて話し合うことができます。 3番目のレベルは質問に答えます： 必要な影響を作成するために組織または開発チームとして何ができるか？ 私たちの仕事の最終結果はここで説明されます。



プロセス編成




  1. このイベントに招待するのは10人から15人までです。そうしないと、実施が困難になります。 顧客から3〜4人、チームからも同じように電話することが最適です。
  2. 顧客側では、すべての目標の具体的な実装についてすでに意見を持っている技術専門家だけでなく、ビジネスの代表者を採用することが不可欠です。
  3. ボードまたは壁にマップを作成し、事前に準備します。 6〜8か月続くタスクの影響マッピングは、標準サイズのボードに収まります。

    私はまだこの手法のオンライン版を試していませんが、インタラクティブなツールなら何でもできると思います。 コミュニケーションは同じではありませんが、顧客を個人的に知っているか、優れたファシリテーター/モデレーターである場合、あなたを救うことができます。
  4. マッピングには1時間から2日かかります。 この数字は、参加者の構成とあなたの指揮スキルに大きく依存します。
  5. マップの各ブロックは、マーカーで描画するか、ステッカーで作成できます。 ステッカーのほうがモバイル性が高いため、私はステッカーを好みます。また、プロジェクトに飛び込むと、影響マップが頻繁にソートおよび変更されます。
  6. 開始する前に、マッピングのルールと目標について必ず話し合ってください。 時間がある場合は、準備のためにトピックに関する資料を全員に送信します
  7. 可能であれば、状況が許せば、いくつかの砕氷船を作ります。
  8. そして最も重要なのは、プロセス自体が簡単で楽しいことです。 官僚主義を加えないでください！




事例紹介





最初にインパクトマッピングを行った実際のプロジェクトに非常に近い例を調べてみましょう。 Impact Mappingをコンパイルする際のキーポイントと、アイデア全体を台無しにする可能性のあるエラーについて説明しましょう。



なんで？




マップのルート要素は、ビジネス目標のリストになります。 たとえば、これはユーザー満足度の2倍の増加になります 。 ユーザーの満足度が指標であることが重要です。つまり、 顧客の意見/感情ではなく、CRMから取得できる特定の数値。 機能を提供した後、目標の達成度を測定し、その方向に進んでいるかどうかを理解します。 ユーザーの満足度が数値でない場合、目標を達成したことをどのように知ることができますか？ また、単なる増加ではなく、正確に2回書き込みを行うことも重要です。 良い目標はスマートでなければなりません：







最初の段階はかなり複雑で、マップの残りの部分を作成する衝動を与え、参加者間の信頼を作成する必要があります。 私の練習に基づいて何をアドバイスできますか：

  1. この時点で時間をかけて解決策を提案してください。 お客様の声に耳を傾け、 真に耳を傾けてください。 さらなる議論の過程で、あなたはすべてを修正し、とかし、今のところ彼が頭に持っているものを書き留める時間があります。
  2. 最も一般的な問題は、目標が明確になる前に解決策を課すことです（What？ステップ）。 エンジニアリングの考え方は光の速さで飛びます-顧客は口を開けて、自分の目標について話し始めたばかりで、すべてのテーブルを頭に入れてデータベースを作成し、アーキテクチャを考え出し、コードを配置しました。 すでにすべてを考え出したのに、なぜさらに聞きますか？ 顧客を中断し、解決策を提案し始めるのは間違いです。 このテーマに関するジョークを覚えて、問題を回避しようとします。「ディマは「こんにちは」と言いました、そしてダーシャは精神的に結婚式をし、3人の子供を産みました。」
  3. この時点で顧客を安心させないでください。 最初は、彼のビジネスのすべての微妙な点を知りません。 顧客はITプロフェッショナルとしてあなたを信頼することができ、そのため、あなたの訂正にすぐに同意します。 あなたは、あなたが取締役会に課した目標だけがどのように表示されるかを気づかず、顧客がこれまでずっと住んでいたものに気付かないでしょう。
  4. 目標を測定するのが難しい場合でも、その達成基準を考えてみてください。 プロジェクトの最終段階に精神的に移行し、目標が達成されたかどうかを確認する方法について考えてください。
  5. 目標を策定するプロセスは反復的であり、最初のラウンドで顧客からすべての目標を絞り出す必要はありません。
  6. 人工ターゲットを伸ばす必要はありません。 投資家はソフトウェアクリエーターを演じたいので、単純に存在するプロジェクトがあります。 これに同意し、インパクトマッピングのコンパイル作業を削減する必要があります。




HappyDev 2014では、インパクトマッピングとストーリーマッピングの作成に関するマスタークラスを教えました。 建設申請の処理のプロジェクトマネージャーは、顧客の役割を果たすことに同意しました。 トレーニングに参加した全員が非常に活発で、すぐにプロセスに参加しました。 時間が経つにつれて、顧客の声に耳を傾け、彼の問題を理解することは非常に難しいことがわかりました。 同僚は自分たちのソリューションを提供しようと争いました。 ある時点で、グループの作業を中断して、もっと聞く必要があることを思い出させる必要がありました。 緊張した雰囲気と参加者のプレッシャーのために何度か、顧客は私たちの決定を下し、自分自身を放棄しました。 すべての参加者は、顧客の声に耳を傾けるタイミングとソリューションを提供するタイミングの間に重要なバランスを感じたと思います。





誰？




この段階で、目標に影響を与えるのを助ける人、目標の達成に貢献する人、干渉する人をすべて特定する必要があります。 この例では、これらはマーケティング部門とフォーラムモデレーターになります。 顧客によると、ユーザーの満足度を変更できるのは彼らです。







ここでは、特定の人、部門名、市場セグメントなどを示すことができます。 それがあなたのプロジェクトにとって適切である場合にのみ、抽象化のレベルを選択してください。



どうやって？




次に、目標を達成するために行われる影響を判断する必要があります。 たとえば、フォーラムのモデレーターは1分以内に質問に答えようとします。 これはユーザーの満足度を高めると思いますか？ 増加すると想定しているため、この「影響」を記述します。 残りの役割についても同じことを行います。







いくつかの推奨事項：

  1. オプションですが、影響も測定可能であることが望ましいです。 フォーラムに返信するだけでなく、フォーラムに1分間返信します。
  2. 各役割の考えられるすべての影響を記録しないでください。 目標の達成につながる活動のみが必要です。




なに？




インパクトマッピングで最も重要でないものに到達しました。 マップの最後のノードには、プロジェクトの作業が通常開始されるショッピングバスケットがあります。 違いは、各機能の価値、この機能がなぜここにあるのか、そしてその実装が何につながるのかを理解することです。







いくつかのコメントと推奨事項：

  1. マップのエンドノードで、ユーザーストーリーまたはモジュール/サブシステムの名前を記述できます。
  2. 地図のこの部分を詳細にペイントすることはできません。塗りつぶすこともできませんが、主なポイントを言うだけです。 Story Mappgingですべてのユーザーストーリーの完全なリストを作成できます。
  3. ここでITタスクを説明する必要はありません。 代わりに、何らかの組織変革と、一般的に目標への影響を実装するためのあらゆるアクションを記述できます。
  4. 目標を理解することで、これらの目標を達成するための安価で高速なソリューションを作成する機会が得られます。 このカードにより、開発者の手だけでなくヘッドも使用し始めます。各チームメンバーは情報に基づいた決定を下すことができます。




影響マッピング結果





これで、インパクトマッピングの準備ができました。 各列の優先順位付けは残ります。 すべての目標が等しく重要であるとは限らず、残りのマップノードについても同じことが言えます。 優先順位付けにはさまざまな方法があります。 なぜなら シンプルさと視覚化の道をたどり、星を付けることをお勧めします。 各参加者に5つ星を与えると、好きな場所に置くことができます。 したがって、優先順位が最も高いノードを識別することができます。



作業の結果は、完全な視野でハングアップする必要があります。 チームが分散している場合は、インパクトマッピングを共通のナレッジベースに配置するか、すべての開発参加者に表示される画面の前に配置する必要があります。 主な目標は、この情報の可視性と到達可能性を確保することです。これは、プロジェクトで作業する際に情報に依存しているためです。



AgileClubでのImpact Mappingについて話したとき、同僚は、戦略目標を理解する他の方法があることに気付きました。 たとえば、 リーンキャンバスを使用したり、目標と利害関係者を説明するプロジェクトドキュメントで要件を収集したりできます。 実際、インパクトマッピングは他のアプローチと矛盾せず 、それらで使用できます。 個人的に、私は彼がより好きです：

  • これは、コミュニケーションと相互作用を促進する単純な手法であり、官僚主義はありません。
  • ITおよびソフトウェアの生産を理解していないお客様にとって、このアプローチは非常に簡単に説明でき、数分で十分です。

受信フィルター








誰もがプロジェクトの目標とその達成方法に同意した場合でも、顧客はプロジェクトに自分が本当に気に入っている機能であるペット機能を追加できます。 目標を通してフィルタリングし、この機能が目標を達成するようには決して導かないことを示します。



同様に、開発チームからのシステムアーキテクチャと設計に関するアイデアをフィルタリングします。 アーキテクチャの修正により、目標をより速く、より安く達成できますか？ そうでない場合、なぜこれを行う必要がありますか？



かんばんボードのアップグレード


かんばんボードの最後の列はどれですか？ リリース、デプロイ、完了、またはそのようなものだと思います。 ボードの最後の列は-目標の達成を確認する必要があります。 機能をサーバーにアップロードするだけでは十分ではありません。期待どおりかどうか、目標を達成したかどうかを確認する必要があります。



よくある質問



  • プロジェクトを開始する前に、インパクトマッピングの作成を顧客に販売する方法は？

    問題から出発するのが最善です。 多くの機能が作成され、ビジネスがこれに苦しんだときのケースを思い出すように顧客に依頼します。 なぜこれが起こったのですか？ たぶん、目標を明確に記述する必要がありますか？
  • この仕事は支払われるべきですか？

    はい、間違いなく。 Impact Mappingのコンパイルには数日かかることがあり、ビジネスにとって価値があります。無料で行うことはお勧めしません。
  • 顧客がこれをしたくない場合はどうなりますか？

    あなたは、専門家として、将来の予測を顧客に提供する必要があります。 起こりうる問題について説明し、リスクとその危険性を説明してください。 その後、顧客に選択してもらいます。 問題のある可能性のある未来を伝え、クライアントがそれを受け入れ、結果を完全に明確にしたインパクトマッピングを放棄した場合、これはあなたの問題ではなく、単に機能にするだけです。




インパクトマッピングは、顧客と開発者の両方をより幸せで効率的にする活動の1つです。 適切な目標を正しく設定してください！
