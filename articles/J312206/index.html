VDS（仮想分散スイッチ）がvSphere環境で使用されている場合、追加のアクションなしで自動的に移行されます。 構成と共に構成履歴とパフォーマンスデータ（統計、イベント、タスク）を保存する場合は、このデータを転送できるオプションがあります。 覚えておいてください。これにより、vCenter Serverのデータベースの移行時間とデータ量が増加します。



wiki KB 214620の記事を使用することを強くお勧めします。この記事は、移行プロセスの評価と表示に役立ちます（期間、 所要リソースなど）。 この投稿には、移行の準備時に考慮すべき事項のチェックリストが含まれています。これは、vSphere Single Sign-OnドメインおよびvCenter Server展開モデルに関連しています。 私が強調したい移行ツールについては、2つの主要なポイントがあります。



最初のもの。 移行が実行されるだけでなく、バ​​ージョン5.5（更新あり）からバージョン6.0（更新2）への更新も実行されます。 移行する前に、VMwareのすべての部分とすべてのサードパーティベンダーソリューションがvSphere 6.0をサポートしているかどうかを確認してください。 移行および更新手順は以前のバージョンから変更されていませんが、このプロセスを大幅に簡素化するメカニズムがあります。



2番目の重要な点-Windows vCenter Serverのすべての構成が既に保存されていることを確認する必要があります。 これには、データと設定（IPアドレス、FQDN、UUID、証明書、MoRef ID）が含まれますが、これらに限定されません。 これにより、移行プロセスが非常に簡単になります。これは、構造のすべての部分がUUID認証標準を介してvCenter Serverと通信し、古いバージョンの場合のように移行後に関与するためです。 このチェックリストは、簡単な移行プロセスを準備するための非常に重要な部分です。



vSphereドメイン



移行と更新の準備の黄金律：「ITインフラストラクチャをよく知る必要があります。」 集中する必要がある2つの領域があります。vSphereシングルサインオンドメイン （SSO）とvCenter Serverの展開モデルです。



最初にSSOドメインを扱いましょう。 vSphere 5.5では、必要に応じてvSphere SSOドメインを結合できます。 このようなユニオンを作成する唯一の機会はバージョン5.5です。これは、バージョン6.0ではこれが不可能だからです。



以下の例では、2つの異なるvSphere SSOドメイン（vSphere.local -1およびvSphere.local -2）。 最初の目標は、これら2つのドメインを1つのvSphere.localに結合し、2つのドメインのすべての構成にすることでした。 これは、既存のSSOドメインを選択するこのオプションが、2番目のSSOサーバーへの展開中に適用されなかったために可能でした。



vSphere SSOドメインを結合する場合は、移行の前に次を実行する必要があります。



•新しい外部SSOサーバーを展開する

•新しいSSOサーバーでvCenter Inventory Service、Web Client、vCenter Serverサービスを指定する

•内部展開モデルから外部展開モデルに移行する場合は、内部SSOコンポーネントをインストールする必要があります



プロセスが完了すると、右の図に示すようにSSOドメインがマージされ、移行プロセスを進めることができます。 この統合プロセス中に、 vSphere 6.0 SSOの最大値があることに注意してください。 このガイドには、バージョン5.5でvSphere SSOドメインを結合するときに重要な詳細が含まれています 。 既存のvSphere SSOトポロジに慣れている場合は、vCenter Serverの展開を計画できます。







展開モデル



これは、vCenter Serverの展開における次の重要な瞬間です。 Windows vCenter Server 5.5には、シンプルとカスタムの2つの展開モデルがあります。



シンプルなモデルは、単一の仮想マシン上のすべてのvCenter Serverサービスに含まれています。 一方、ユーザーモデルでは、vCenter Serverサービス（SSO、Inventory Service、Web Client、vCenter Server）を異なる仮想マシンに分割できます。 VSphere 6.0は、Platform Services Controller（PSC）が組み込まれたデプロイメントモデルを簡素化します。 これで、単一のvSphere SSOドメインに分散されたさまざまなサービスを管理できます。 これらのサービスには、シングルサインオン（SSO）、証明書管理、ライセンス、ロール、タグ付けと分類、共有制御が含まれます。



vSphere 6.0には2つの展開モデルもありますが、現在は「内部」と「外部」と呼ばれています。 vSphere 6.0の内部展開モデルは、バージョン5.5の単純なモデルに従います。 これは、すべてのvCenter ServerサービスとPSCコンポーネントが同じ仮想マシンに残ることを意味します。 仮想マシン上の外部vSphere 6.0展開には、PSCコンポーネントとvCenterがすでにあります。 これで、各コンポーネントが責任を持ち、サービスを管理します。 バージョン6.0の推奨トポロジーのリストはこちらです。



外部実装モデルの利点の中で、拡張リンクモードでは、拡張リンクモードとvCenter Serverを追加することで自動スケーリングを実行できます。



重要！ vSphere 6.0 Update 2m-移行プロセス中に展開トポロジを変更することはできません。







外部移行モデルバージョン5.5のルール：



•vCenterインベントリサービスを外部SSOサーバーと同じ仮想マシン上に保持しないでください。 vSphere SSOサービスの移行後に内部SSOサーバーがシャットダウンし、vCenter Inventory Serviceが使用できなくなります。



•すべてのサービス（SSO、vCenter Inventory Service、Web Client、vCenter Server、各仮想マシンで個別）が分離されている場合、移行アシスタントを実行するだけで済みます。



•設定が保存されているすべての拡張機能がvSphere SSOサーバーにインストールされると、移行アシスタントがロックされます。 設定を保存する拡張機能-これは、vCenter Inventory Service、Auto Deploy、vSphere Authentication Proxyなどのサービスのいずれかです。



•設定を保存せずに拡張機能がvSphere SSOサーバーにインストールされると、移行アシスタントがシグナルを送信します。 データを使用するが保存しない設定を保存せずに拡張-vSphere Web ClientおよびDump Collector。



•ロードバランサーをバイパスするvSphere SSOサーバーの移行が可能です。



VMwareには、vCenter Serverの展開モデルを内部から外部に変更するかどうか、および移行モデルを変更するタイミング（移行プロセスの前後）について多くの質問が寄せられました。 SSOドメインを結合する必要がない場合は、最初に組み込みモデルを使用します。 その後、cmssoコマンドを使用して、内部モデルを再構築し、外部モデルに再構成します。 cmssoを使用してトポロジを変更する方法の詳細については、こちらを参照してください 。 SSOドメインを結合する必要がある場合、ドメイン統合プロセスの一部として内部モデルが外部になります。 マージプロセスが完了すると、移行プロセスを開始できます。 外部モデルの移行では、まずすべてのvSphere SSOサーバーがvSphereドメイン内で移行され、次にすべてのvCenterサーバーが移行されます。 これは更新プロセスと変わりません。



上記のように、移行およびアップグレードプロセスは変更されていません。 移行を開始する前に実行する必要があるアクションはまだ多くありますが、移行プロセスをより簡単にするツールが用意されています。



移行に役立つリソースは、 vCenter Server 5.5からvCenter Server Appliance Update 2への移行に関するFAQです。



移行できたかどうかをお知らせください。VMwareの技術サポートが追跡して対応できるように、 ＃migrate2vcsaタグを使用してコメントまたはTwitterで登録を解除できます。
