Qt Quick ComponentsをS60 5thエディションに移植





Symbian用のFutoubraクライアントの実装を開始して、アプリケーションのユーザーインターフェイスを実装するためのフレームワークとしてQt Quickを選択することに疑いの余地はありませんでした。 リアクティブデータバインディングのおかげで、高度なアーキテクチャ設計パターン、特にModel-View-ViewModelを使用してプレゼンテーションレイヤーを実装することが可能になります。 ただし、Qt Quickの「裸の」形式には基本的な要素が非常に少ないため、ボタン、入力フィールドなど、より複雑な要素を組み立てることが提案されています。 以前のプロジェクトでユーザーインターフェースを作成するときにそれらを実装するのにどれだけ時間がかかったかを念頭に置いて、私はすぐに自分で行うという考えを捨て、Qt Quick Componentsライブラリーに注意を向けました。 カッターの下には、S60 5thエディションプラットフォームでの適用範囲を広げて「取得」する方法を説明しています。



Qt Quick Componentsライブラリは、QMLの上に幅広いウィジェットを実装するQt Quickプラグインのコレクションです。 ライブラリの主な機能の1つは、クロスプラットフォームです。 Windows、Mac OS、KDE、GNOME用のデスクトップバージョンと、Symbian、Maemo、MeeGoに代表される小さな兄弟向けのバリエーションがあります。 もちろん、プラットフォームの機能、画面解像度、およびデザインガイドラインにより、バージョンには多少の違いがありますが、WindowsやSymbianなどのQMLコードの約80％を取得することは可能です。 ライブラリの同様に重要な機能は、各プラットフォームのウィジェットのネイティブな外観です。



Symbian用ライブラリのバージョンのドキュメントを調べ始めたところ、すぐに不快なニュースに出会いました。 ドキュメントによると、 Qt Quick Componentsは最新のSymbian ^ 3、Symbian AnnaおよびNokia Belleでのみサポートされています。 私にとって、5800 XpressMusic、5228など、我が国での第一世代のNokiaタッチスマートフォンの人気を考えると、S60 5thエディションプラットフォームをサポートすることも必要でした。 あと1つしかありませんでした-あなた自身のリスクと、このオペレーティングシステムにライブラリを移植するリスクもあります。 この体験は成功したことが判明し、 ここで独自のバージョンのQt Quick Componentsを使用してデモアプリケーションの完全なソースコードを公開しました。 以下は、主にqmakeプログラミングに関連する最も複雑な移植の問題のみを示しています。



アンロード



ここで重要なのは、1.0ブランチである必要があることです-すぐに使用できます。 1.1.xライブラリの以降のバージョンには、S60 5thエディションSDKにはないヘッダーファイルとライブラリの形でSymbian SDKに「重い」依存関係があります。 リンクを使用して、または対応するGitコマンドを使用して、アーカイブの形式でソースを取得できます。 



git clone git://gitorious.org/qt-components/qt-components.git -b 1.0-symbian




シャーマニム



Qtのドキュメントにはプラグイン作成の詳細な例が記載されていますが、括弧の後ろに残っている非常に難しい部分は、プロジェクトの* .proおよび* .pri-files（以下、単にスクリプト）の構成です。 そのため、Qt Creator環境によってデフォルトで作成されるスクリプトには、Qt Quickプラグインを含むインストーラーを作成するメカニズムが含まれていません。 Symbianにアプリケーションをインストールする機能の1つは、すべての実行可能ファイルと動的ライブラリをC：\ sys \ binディレクトリに配置する必要がありますが、Windowsはこれらの場所を規制しないことです。 その結果、エミュレーターとデバイスのアプリケーションのインストールに起因するファイル構造が大幅に異なる状況になりやすく、各プラットフォームに固有のインストーラーのバグを検索することになります。 スクリプトを開発するときの私の主なアイデアは、WindowsとSymbianの両方にアプリケーションを等しくインストールすることでした。 それらを検討します。



/config.pri

config.priファイルは、他のスクリプトで使用されるすべての環境変数を定義します。 $$ PWDを使用して、プロジェクトルートへのパスが決定され、残りのパスはそれに関連して定義されます。 プロジェクトをビルドすると、インストールフォルダーがそのルートに作成されます。これは、デバイスのC：システムドライブのプロトタイプです。 それと同じように、プロジェクトはデバイス上にレイアウトされます。



簡潔にするために、最も単純なコンポーネントであるプラグインスクリプトを検討します。 各プラグインには、同じ名前のpriファイルとproファイルがあります。 最初はインストーラーの作成に使用されるプラグインのプロパティを説明し、2番目はライブラリ自体のビルドに使用されるプロパティを説明します。



/src/3rdparty/qt-components/components/components.pri 
 QT_COMPONENTS_PLUGIN.source_path = \ src/3rdparty/qt-components/components QT_COMPONENTS_PLUGIN.imports_path = \ Qt/labs/components QT_COMPONENTS_PLUGIN.target = \ $$qtLibraryTarget(embedded_qt_components_plugin) QT_COMPONENTS_PLUGIN.qml_files += \ qmldir \ Checkable.qml \ CheckableGroup.qml \ CheckableGroup.js APP_PLUGINS += QT_COMPONENTS_PLUGIN


プラグインを記述する構造内のフィールドのプロパティには、次の意味があります。

  • source_path-プロジェクトのルートに相対的なプラグインへのパス。
  • imports_path-アプリケーションインポートディレクトリのルートからの相対パス。 スラッシュをドットに置き換えると、アプリケーションのqml-filesにインポートの名前が付けられます。たとえば、この場合- 
     import Qt.labs.components;
  • target-構築されるライブラリの名前
  • qml_files -qmlファイルで接続されたときに使用可能なプラグインウィジェットのリスト。


組み込みのQt Quick Componentsの1対1の相対インポートパスは、デバイスで利用可能な元のプラグインのインポートパスと一致します。 使用するプラグインの性質を知ることからqmlファイルを抽象化することで、S60 5thエディションと最新のSymbianのアセンブリでそれらをシームレスに切り替えることができます。 priファイルの最後の行で、コンポーネントプラグインをAPP_PLUGINSに追加します。これは、デバイスにインストールされたプラグインの配列です。



/src/3rdparty/qt-components/components/components.pro 
 include(../../../../config.pri) include(components.pri) TARGETPATH = $$QT_COMPONENTS_PLUGIN.imports_path TEMPLATE = lib TARGET = $$QT_COMPONENTS_PLUGIN.target INCLUDEPATH += $$PWD $$PWD/models CONFIG += qt plugin QT += declarative network script DEFINES += QT_BUILD_COMPONENTS_LIB QML_FILES += $$QT_COMPONENTS_PLUGIN.qml_files symbian { TARGET.EPOCALLOWDLLDATA = 1 TARGET.CAPABILITY += $$APP_CAPABILITY TARGET.UID3 = 0xE1E604E2 } HEADERS += qglobalenums.h SOURCES += plugin.cpp include(kernel/kernel.pri) include(models/models.pri) include(../../../../install.pri)


冒頭のペアのプロファイルにはconfig.priが含まれており、APP_CAPABILITYなどのすべての環境変数を利用できます。 後者の使用は、実行可能ファイルで使用される動的ライブラリのアクセスレベルがファイル自体のアクセスレベルよりも低くないと主張するSymbianセキュリティシステムによるものです。 これに悩まされないために、最も簡単な方法は、すべてのプラグインとexe-shnikに同じ機能を持たせることです。

最後に、アプリケーション自体でインポートがどのように接続されているか。



src / app / gui / application.cpp 
 QDeclarativeView *Application::buildRootView() { QScopedPointer<QDeclarativeView> view(new QDeclarativeView()); QObject::connect(view->engine(), SIGNAL(quit()), view.data(), SLOT(close())); view->setResizeMode(QDeclarativeView::SizeRootObjectToView); view->engine()->addImportPath(QLatin1String("../../resource/demo/imports")); view->setSource(QUrl("qrc:/layout/main.qml")); return view.take(); }


定義した環境変数に従って、アプリケーション自体は/ sys / binにインストールされ、プラグインは同じドライブの/ resource / demo / importsにインストールされました。 ここから、QDeclarativeEngineのaddImportPath関数で指定する必要があるパスを取得します。



結論の代わりに



手作業で組み立てられたQt Quick Componentsを使用して作成されたアプリケーションは、元のバージョンで作成された同様のアプリケーションとほとんど区別できません。 これにより、Nokia StoreのQAチームの側で面白い誤解が生まれました。 彼らは長い間アプリケーションを公開したくなかったので、本当のQt Quickコンポーネントを使用していると非難されましたが、何らかの理由でインストーラーでそれらへの依存を宣言しませんでした。 コメントは次のとおりです。

Jerry（VC-Signing）（Thu、03 May 2012 18:41:30 +0000）

パブリッシャー様、

あなたのアプリはQt Quick Components for Symbianを使用していますが、アプリパッケージには対応するコンポーネント依存関係宣言がありません。 依存関係を追加して、アプリを再構築してください。



オリジナルのプラグインを使用して最新のSymbian用のアプリケーションをアセンブルすることによってのみ、この問題に対処することができました。 S60 5thエディションでは物理的に同じことができないため、Nokia Storeの従業員は、このプラットフォームのアプリケーションインストーラーでの存在を主張しませんでした。 最後に、独自のバージョンのQt Quick Componentsを備えたデモアプリケーションのソースコードがこちらから入手できることを再度お知らせします 。
