急速な開発を妨げるITスタートアップの典型的な問題とそれらを回避する方法

IIDFオンライン会議「技術ベースのビジネスを構築する方法」で、Googleの開発者関係を担当するZviad Kardava氏は、製品の開発、開発、プロセス管理における技術系新興企業の問題と、それらを解決または回避する方法について話しました。







私たちは生き残る時間のないダイナミックな世界に住んでいます。MVPを素早く作り、市場に参入し、コンバージョンとプレゼンスを高めようとしています。 このアプローチは正当化されますが、その過程で、スタートアップの創設者は将来干渉する問題に直面する可能性があります。



スタートアップに問題がある3つの分野：



1）管理とプロセス



短期間でチームが5人から50人に成長した場合、以前のスキームでは機能しません。新しいビジネスプロセスを設定して管理する必要があります。会社と製品に優先順位を付け、成功指標の主要な指標と会社のドローダウンを決定できるようにするため/製品。



プロセスで最もよくある間違いは、CTOがほとんどの時間を開発に直接費やしていることです。 もちろん、CTOには優れた技術スキルが必要です。また、スタートアップに3〜5人しかいない場合、開発への参加は完全に正常です。 しかし、チームにすでに50人がいる場合、サービスステーションは開発プロセスをリードし、開発部門とテスト部門のチームリーダーと密接に連携し、ほとんどの場合、開発自体に対処するべきではありません。 この人物は、すべてのテクノロジーを理解し、チームと話し、適切な開発を構築する方法を知っており、彼の製品と新しいテクノロジーを試しているジェネラリストです。



2）製品開発



機能の優先順位付け、テスト、製品の適合に問題がある可能性があります。 製品の開発を追跡することは、正しいバージョンかどうかを理解するのに役立ちます。新しいバージョンまたは機能がリリースされたとき、パフォーマンスを測定し、反復を移動し、1つのことだけで長く停止しないようにする必要があります。 製品メトリックスの追跡と評価はマーケティングと非常に密接に関連しています。非常に優れた機能を作成できますが、販売は非常に悪く、逆もまた同様です。 A / Bテストや分析などのアプローチは、顧客と製品の理解に役立ちます。







たとえば、Netflixを携帯電話の安価なサブスクリプションで低所得地域の居住者のように見せようとした企業は、分析の助けを借りて、最初の開始時に表示を開始してから1分以内にコンテンツの再生を中断するユーザーがかなり多いことがわかりました。 チームは、新機能のリリース、設計の変更、機械学習の適用を試みましたが、問題ははるかに簡単であることが判明しました。このユーザーグループにはヘッドフォンがありませんでした。 彼らが路上や公共の場所にコンテンツを含めると、彼らはヘッドフォンなしでそれを見るだけで不快でした。



3）開発



多くの場合、開発者はすべてをゼロから構築することを好みます。私たちの国では、プログラマーは技術的な問題を解決できます。 しかし、多くの場合、時間がかかりすぎます。 既製のコンポーネントが公開されている場合や有料である場合は、それらを使用してください。 これにより、製品固有の機能の開発に集中できます。 たとえば、何らかのモバイルアプリケーションを作成している場合、Firebaseを使用するだけで十分な場合がよくあります。 分析、ホスティング、プッシー、データベース、クラウド機能、プロモーションツール、クラッシュレポート、A / Bテスト用ツール、実験など、必要なものはすべて揃っています。 多くの場合、クラウド機能は単純なバックエンドロジックを実装するのに十分です。 このインフラストラクチャを自分で作成または保守する必要はありません。コンポーネントを正しく組み合わせるだけです（このアプローチはサービスとしてのバックエンドと呼ばれ、開発時間と市場参入を大幅に節約できます）。



また、開発用の「ツール」を保存しないでください。プロセスを大幅に高速化するのに役立ちます。 これは、開発環境（IDE）だけでなく、テストツールまたはインフラストラクチャにも適用されます。 たとえば、Github、Gitlab、Bitbucketを使用すると、CTOはコードのレビューを行い、問題を発行し、分散チームで便利に開発できます。 自分でコードを保持することにした場合は、Gitlabなどの適切なリポジトリをセットアップし、適切なハードウェアを用意してください。



自分のチームまたはアウトソーシング？







テクノロジー製品とプラットフォームをターゲットとする多くのスタートアップは、しばしばアウトソーシングを否定的に受け止めています。 ロシアには多くのデベロッパーがいて、平均市場の給与は高いですが、これだけの経験でも、デベロッパーがあなたの期待に応えることを保証するものではありません。チーム。



優秀なエンジニアやプログラマーは、しばしば非常に悪いデザイナーです。 優れたUI / UXは製品にとって非常に重要であるため、UI / UXを簡単に外部委託でき、同時に社内開発を離れることができます。主なことはバランスを維持することです。



アウトソーシングに必要な特定の人々や企業-プライベートデザイナーや専門家、小さなモバイル開発スタジオ、またはアウトソーシングに特化した大企業-を理解することが重要です。 私は大企業を小さなスタートアップに勧めません。原則として、彼らは異なる焦点と仕事の原則を持っています。



製品の何が問題なのか：主な問題領域



1）フロントエンド



慣例により、フロントエンドの概念には、少なくとも何らかの形でユーザーと対話するすべてのものが含まれます。 1つのケースでは、Webまたはモバイルアプリケーションの形式のユーザーインターフェイスであり、もう1つでは、プラットフォーム、アプリケーション、マーケットプレイスのAPIです。 ユーザーフレンドリーなAPIは優れたUX / UIであり、これは非常に重要です。 次は、アーキテクチャ、フレームワーク、言語です。



ある言語、フレームワーク、またはアプローチが他の言語よりも優れていると主張することはできますが、ユーザーはアプリケーションの記述内容を気にしません。 それらの主なものは、美しく直感的なインターフェース、迅速な製品操作です。







UX / UIが悪い場合の対処方法



専門のアウトソーシングスペシャリストに相談し、デザイナーにレイアウトを依頼して、ユーザーが画面を操作してあるページから別のページに移動する方法をテストし、製品を社内で変更して定期的に誰かと相談できます。



友人とのユーザビリティのテストも、UX / UI相談オプションの1つです。 お母さんなどの親relativeにアプリやウェブページをテストしてもらいましょう。 彼女が使用に関する問題を抱えておらず、プロンプトを表示する必要がない場合は、成功に近づいています。







新しいユーザーを適応させる必要があります。 インストール後のアプリケーションが、連絡先、ファイル、カメラへのアクセスをすぐにユーザーに要求した場合、ユーザーはこのアクションが疑わしいと認識します。



何かにアクセスするリクエストなしでアプリケーションが起動し、必要な場合にのみ、ユーザーがしばらく操作した後に送信することをお勧めします。 たとえば、ゲームプラットフォームがあり、インストール後、ユーザーにゲームをプレイすることを提案します。その過程で、ユーザーは友人と競争して結果を共有できることを理解します。 この時点でアプリケーションが連絡先へのアクセスの要求を開始すると、ほとんどの場合、同意します。



別の例は、登録する必要があるプラットフォームです。 私たちは皆、携帯電話と外出先で生活しているので、大きくて不快なフォームに完全に記入するように人々に頼むのは悪い戦略です。 代わりに、会話型UX（ユーザーインタラクション）を追加できます：API.ai（dialogflow.com）またはFacebookメッセンジャーに接続します。 ユーザーが必要とするのは、FacebookまたはGoogleからログインし、回答オプションのボタンをクリックして質問に回答し続けることだけです。 これにより、ユーザーは外出先でフォームに入力できます。



優れたUX / UIは次のとおりです。



  1. 競争上の優位性。 人々が不快であるという理由だけでいくつかのアプリケーションの使用をやめます。
  2. より高い評価：より高いコンバージョンと収益化をもたらします。 ユーザーがこのアプリケーションを気に入って人気を博している場合、アプリケーションストアはそれをより簡単にコレクションに追加します。 プラットフォームが優れているほど、技術サポートを受けるユーザーは少なくなります。


トレンドにあり、常に新しいテクノロジーを監視することが非常に重要です。 たとえば、今まで、多くの人はすでに3年以上前のProgressive Webアプリケーションについて知りません。 これは非常にポンプアップされたWebページで、オフラインでも高速に動作し、モバイルデバイス上のアプリケーションのように見えます。 PWAはアプリケーションとして携帯電話の画面に追加でき、開発が非常に簡単です。 プログレッシブWebアプリケーションには、Web支払い、Web認証情報、Webセンサーなど、さまざまなAPIセットもあります。







Webペイメントは、支払いページにリダイレクトすることなく、便利なサブスクリプションと支払いでWebページを作成するのに役立ちます。 Web Credentialsを使用すると、ユーザーが少なくとも一度ログインすると、システムに自動的にとどまることができます。



もう1つの興味深いテクノロジーは、インストールする必要のないすべての機能を備えた完全にネイティブなアプリケーションであるAndroid InstantAppsです。検索、リンク、ビーコン、Telegramなどから直接取得できます。 リンクをクリックすると、アプリケーションがすぐに開き、ユーザーは何かを購入、購読、実行できます。 そしてその後のみ-必要に応じてアプリケーションをインストールします。 これにより、ユーザーエクスペリエンスが向上し、市場へのリーチが拡大します。



しかし、新しいテクノロジーに遅れずについていくことは、製品にすべてを続けて適用することを意味しません。 最近、誰もが本当にニューラルネットワークを使用するか、これを宣言したいと考えています。 これはPRとマーケティングには良いかもしれませんが、製品がこれらの技術を対象としていない場合、またはそれらの経験がない場合は、時間を無駄にして魔法をかけるべきではありません：あなたが何をしていて、専門家が何をしているのかを理解する必要があります。



音声認識、文字起こし、翻訳、画像分析などの標準的なものが必要な場合は、MLモデルを開発およびトレーニングする代わりに、既製のものをより速く簡単に使用できます。 Google Cloud Platformやその他のプラットフォームには、優れた簡単なAPIを備えた品質モデルがあります。



2）バックエンドとインフラストラクチャ



バックエンドに適したアーキテクチャを構築することは、最初から非常に重要です。 現在、誰もがいわゆるマイクロサービスアーキテクチャに導かれています。 これは、1つのテンプレートだけでなく、バ​​ックエンドを構築するためのアプローチです。したがって、さまざまなバリエーションが可能ですが、一般に、このアプローチは正当化されます。



要するに、マイクロサービスアプローチでは、単一のアプリケーション（バックエンド）が一連の小さなサービスとして構築され、それぞれが独自の環境で動作し、軽量メカニズムを使用して残りと通信します。 HTTP / RESTが通信に使用されることは非常に多くありますが、GRPC-すべてのプラットフォームおよびすべての言語で機能し、HTTPよりもはるかに効率的なリモートプロシージャコール用のオープンフレームワークを探して検討することをお勧めします。 適切なデータベースを選択することも非常に重要です。



これにより、ホスティングを含む多くの労力と費用を節約できます。 仮想マシンの代わりに安価なコンテナを使用でき、バックエンドの柔軟性と拡張性を維持できます。



開発者の1人がこのDBMSを使用した経験があるため、MySQLが選択したスタートアップがあり、彼はそれで十分だと考えました。 それから、JSON形式でデータを保存する必要があり、MySQLで保存する必要があることがわかりましたが、JSONを効果的に使用する方法はわかりません。 次に、MongoDBにはドキュメント指向があるため、誰かがMongoDBを使用することを提案しました。 MySQLはスローダウンし始めました。理由を理解せずに、PostgreSQLを採用しました。これは良いリレーショナルデータベースだと聞いたからです。 そこで、彼らは3つのデータベースを得ました。







しばらくして、3つの基地すべてが「減速」し始めました。 そして、チームは重要なホットデータをRedis（NewSQL）に保持できると判断しました。 彼らはまたelasticsearch検索エンジンを持っていました...合計：4つの異種データベースを持つスタートアップがあります。 それらを維持することは非常に不便で高価です。



このようなスタートアップの通常のソリューションはPostgreSQLであり、SQLデータとJSONの両方で適切に機能します。 理想的なケースでは、チームには個別のDBAスペシャリストが必要です。 または、これらの問題を十分に経験し、そのような技術的な詳細を知っている人。



さらに、フレームワーク、独立した既製のコンポーネント、サービス、および言語があります。



適切なフレームワーク、特に言語の選択に関するトピックでは、多くのコピーが壊れています。

タスクまたは領土属性ごとにプログラミング言語を選択する必要があります：モスクワまたはサンクトペテルブルクにいなくても、あらゆるレベルの開発者を見つけることができる場合は、地域に応じて選択します：地域に強力なNode.jsまたはPythonコミュニティがあり、タスクに適しています-これらのいずれかを選択します。 Erlangのプログラマーがいない場合は、探す価値はありません。



多くは、継続的な統合と継続的な配信の慣行を無視しています。 これは実行する価値がありません。製品の品質を向上させ、バグの数を減らし、システムおよび開発プロセスをすべてのチームメンバーにとってより「透明」にするのに役立ちます。



次の質問は次のとおりです。鉄または雲を選択しますか？ クラウド、既成のサービス、プラットフォームを選択することをお勧めします。最終的には、正しく使用すれば、安価で時間を節約でき、それ自体がコスト削減の点でアクセスしやすく、有利になります。



雲は鉄よりはるかに高価であるという意見は根拠がありません。 サービスプロバイダーのすべての機能を再検討し、考慮すると、それらが相互作用のための優れたインフラストラクチャと環境を提供することを理解できます（スタートアップのホスティングの選択方法については、この記事FRIIを参照してください ）。



ホスティング、データベース、その他のインフラストラクチャコンポーネントを選択する場合、覚えておくことが重要です。ユーザーは、アプリケーションの記述内容やホスト場所を気にしません。 主なものは、それが迅速に動作し、ユーザーがフロントエンドと対話できるようにすることです。



単純なPostgreSQL、MS SQL、またはMySQLを使用しても、ほとんどの問題を解決できます。 膨大な数の開発者が使用するStack Overflowは、データを4つのデータベースに保存します。メイン（すべての場合）、補助、2つのレプリカです。 スタックオーバーフロー全体をサポートするにはこれで十分です。



アーキテクチャ、CI / CD、DevOpsアプローチ、コンテナー、データベース、スケーリングに関する多数のチュートリアルと資料を見つけることができます。 これらのことを学習するのにそれほど時間はかかりませんが、将来的には時間、コスト、リスクを大幅に削減できます。 仮想マシンの代わりにコンテナを設定すると、通常のアーキテクチャで何らかのCI / CDがある場合でも、開発が高速化されます。環境を毎回設定する必要はなく、既成のコンテナをダウンロードするだけで作業できます。その後、同じコンテナでテストし、実稼働環境でリリースしてから、スケーリングします。



API管理は、誰もが忘れている新製品の最も重要な問題の1つです。 複雑なことに煩わされる時間がない開発者にとって最も簡単な解決方法は、APIをコンテナにプッシュすることです。 これにより、APIのバージョン管理が可能になるだけでなく、APIの異なるバージョンをロードするときに簡単に拡張できます。



単一のサービスプロバイダーに依存して高可用性を期待することはできません。



現代の世界では、どのサービスプロバイダーにも問題が発生する可能性があります。 したがって、1つのプロバイダーまたはハードウェアのみを使用すると、自分自身を非常にアクセスしやすいサービスまたはプラットフォームと見なすことができないという事実につながります。



複数のサービスプロバイダーを使用することは、まったく通常のアプローチです。 たとえば、Kubernetesはすべてのクラウド（Googleクラウド、Amazonなど）にデプロイでき、サービスの可用性が高くなります。



事前にスケーリングを検討してください。



もちろん、2台の仮想マシンを使用している場合は、さらに数台展開できます。 しかし、急成長を目指す場合、これはトラブルにつながる可能性があります。モノリシックアプリケーションと多数のユーザーが問題になる可能性があります。







PokemonGoが重い負荷に耐えることができたのは、最初から適切なコンポーネント選択戦略でした。最悪のシナリオでは5倍のトラフィックを期待していましたが、最初の数日間の実際のトラフィックは50倍でした。 KubernetesとCloud Datastoreのおかげで、このような負荷に耐えることができました。Kubernetesはほとんどすべてのクラウドにデプロイでき、膨大な数のユーザーと大規模なコミュニティを持っています。 また、Cloud Datastoreは、アプリケーションの負荷に合わせて自動的にスケーリングされる、非常にスケーラブルなNoSQLデータベースです。 あなたは何もする必要も心配する必要もありません。
