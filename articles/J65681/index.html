CMSを揺るがす17の使いやすさのヒント



あなたの注意を引くのは、パトリックケネディによる便利なコンテンツ管理システムであるべきものに関する記事の翻訳です 。



CMSをそのまま使用すると、CMSに多くのユーザビリティの問題が発生する可能性が高くなります。 ウェブサイト管理システム、イントラネットポータル、ウィキなどのシステムの実装を委託されたプロジェクトに参加して、ユーザーインターフェイスには使いやすさの点で修正が必要ないくつかの重要な領域があることに気付きました。





ここに表示されるユーザビリティに関する推奨事項はすべて、一般的な使用原則に基づいており、ほとんどのCMSソリューションにこのような問題があることは明らかなので、任意のソフトウェアおよびWebアプリケーションに適用できます。



これらのヒントを使用して、既存のCMSを改善するか、新しいCMSを導入します。



1.疑わしい場合は、使用しないでください



ユーザーインターフェイスは、ユーザーが自分のタスクを解決する必要がないすべてのものを奪われるべきです。 ほとんどのCMS製品の可能性は実際に使用されているものを超えているため、ユーザーがそれを必要としない場合は実証しないでください。 多くの製品には多くの追加機能がありますが、ご使用の製品のバージョンでそれに関するすべての鐘を鳴らしてはいけません。 良くも悪くも、一部の製造業者はシステムにモジュールを挿入して、欠落している機能を開発しています（おそらく追加の販売を促進するため）。 ユーザーが必要としない場合は表示しないでください。



CSSを使用して詳細を非表示にし、インターフェイスをきれいにします。 基本的なナビゲーション、リンク、および無関係なシステム要素について説明しています。 ここで、Steve Krugの言葉を思い出す価値があります。「Krugの使いやすさの第3の法則：各ページの半分の単語を取り除き、残りの半分を取り除きます。」 各ページ名、小見出し、ボタン、ナビゲーションアイコン、フォーム、アイコン、グラフィックスは、有用で意味があり、ユーザーと明確にコミュニケーションをとる必要があります。



2.システムの複雑さからユーザーを保護する



CMSは任意に複雑で強力にすることができますが、ユーザーは気にする必要はありません。 ユーザーインターフェイスは、システムの内部機能から抽象化され、ユーザーのタスクとそれらを解決する機能の間のトランスレーターとして機能する必要があります。



ユーザーに「資産モデル」、データ構造、および「地下ストレージ」からの他のものを通知することにより、システムの内部操作に関与させないでください。 これらはすべて、システムがユーザーのタスクを解決する方法にすぎません。 ユーザーはそれらについて知る必要はありません。これは混乱を招くだけです。 インターフェイスが「開発者向け開発者」によって作成されたことは、あなたの秘密のままにしておくべきです。



これの症状は、多くの場合、エラーメッセージとシステム用語で見ることができます。 そしてその過程で、ユーザーは、基本的なタスクを実行する場合でも、不必要な情報の曲がりくねった道を進む必要があります。 たとえば、多くのCMS製品でよくある問題は、ユーザーがシステムのある部分から別の部分に前後に移動してタスクを完了することを余儀なくされる場合、二重の作業または複製を行うことです。



3.ユーザーの言語を話す



ユーザーインターフェイスは通信に基づいています。 ユーザーが理解できる用語を使用し、ユーザーが必要な措置を講じてアクションを続行できるように、明確なコミュニケーションを確保する必要があります。 この作業は、ナビゲーション、通知とメッセージ、ボタン、フォームによって行われます。 専門用語と内部システムの技術的な対話は、ユーザーの目に触れないようにする必要があることは言うまでもありません（特定の組織に固有のビジネス用語を使用することは可能ですが、推奨事項10を参照）。



関連する言語の研究は、使用すべき言語を特定するのに役立ち、 37signalsの出版物Defense Web Design：How to Improve Error Messages、Help、Forms、and Other Critical Pointsは、エラーメッセージ、通知などの作成に関する優れたヒントを提供します。



4.実際のユーザーを知る



一体何のためにシステムを作成しているのかを調べましょう。 エンドユーザーは誰ですか？ ほとんどの場合、これらは技術者でも開発者でもシステム管理者でもありません。 データベースを改善したり、アーキテクチャを拡張したりする方法がわからないので、彼らを恥じてください。 実際、彼らはそれがすべてどのように機能するかを気にするべきではありません。



もちろん、開発者とシステム管理者はシステムのユーザーになることができますが、大部分を占めることはほとんどありません。 エンドユーザーは、コピーライター、プロダクトマネージャー、売り手、編集者...人間になります（まあ、営業担当者を除いて:)。



これらのユーザーがシステムをどのように、いつ、なぜ使用するかを調べるために、少し調査してください。 彼らと話し、彼らを見て、分析します。 これは設計に役立ち、ヒントの適用方法を決定するのに役立ちます。



5.本当の目標を忘れないでください



CMSの略語は3つの単語で構成されており、そのうちの1つは他の2つよりも重要度が低くなっています。S。システムにとらわれないでください。 ユーザーがシステムを使用する理由です！



システムを設計および作成するときは、これを細かく考慮する必要があります。 システムの目的は、スケーラビリティ、データ同期、参照整合性、またはネットワークセキュリティではありません。 これはすべて、目的を達成するための手段に過ぎず、裏に残らなければなりません。 技術者はこれらのことに集中する必要があります。これは彼らの義務であり、存在の意味であるためです。 そのため、プロジェクトで作業する場合、ユーザーの作業がこれらの技術的側面に関係してはならないことを理解している健全な人が必要であり、それはコンテンツの作成と管理の問題です。



6.個々のタスクを本当にうまく実現します。



システムのユーザーを知る過程で、ユーザーが実行する最も重要なタスクを分析して見つける必要があります。 これらの重要なタスクに焦点を合わせて優先順位を設定し、真に高品質な方法でそれらを実装します。 インターフェースを構成し、いくつかの機能を自動化して、プロセスのコストをいくらでも単純化しますが、ユーザーのこれらのタスクのソリューションをできる限り単純化します。 ホームページからすぐに取り組むようにしましょう。



トラフィックが多く、コンテンツが多いWebサイトの場合、主なタスクはこのコンテンツの形成です。 さらに、数万ページの非常に大規模なサイトの場合、重要なアクティビティはコンテンツの検索です。 この場合、おそらく検索の周りにシステムの構造を整理することにより、検索を本当に高品質にします。



7.可能な場合は実際のマッピングを使用する



CMSの場合、自然なマッピングとは、ユーザーインターフェイスが実行中のアクションを模倣することです。 良い例は、Webページ上の編集要素の配置です。 通常、CMSではこれは入力フォームを使用して実現されます（「インプレース編集」を使用しない限り）。 しかし、フォームは完成品とは似ていないため、これは自然な表示ではありません。 簡単なように思えるかもしれませんが、完成したページの同じ部分に従って編集ページのフォームフィールドを再配置すると、作成が簡単になります。 または、各フォームフィールドが実際の世界でどのように接続されているかを示す視覚的なマップを作成します。 または、各フィールドのショートカットを作成するだけでも意味があります:)



優れたユーザーインターフェイスは、システムモデルではなくメンタルユーザーモデルを反映しています。



別の例は、サムネイルの使用です。 これらは、検索時、リスト内、またはプロパティの表示時の写真です。 たとえば、画像の外観、テキストの名前、またはページの名前が明確である場合、最終製品で公開される視覚属性のコンテンツを決定するのがはるかに簡単です。



8.一貫性を保つ

ユーザーインターフェイスは、ページ間、コンポーネント間で一貫している必要があります。 これは、ナビゲーション、ボタン、フォーム、テキストスタイル、リンクスタイル、レイアウト、用語、およびフィードバックメカニズム（通知ウィンドウや黄色のフェードテクノロジなど）に適用されます。



これは、一部のパーツがカスタマイズされており、原則として外観が大幅に異なる場合に特に重要です。 そのような「フランケンシュタイン」を使用に移すことは、システムに混乱を与える信頼できる方法であり、ユーザーによる新しいシステムの認識と採用に貢献しません。 ユーザーは感じる必要があります-これは高品質の製品です。



9.準拠することを忘れないでください。



CMSは、それ自体が一貫しているだけでなく、ユーザーが他の同様のシステムに期待するものと互換性があり、一貫している必要があります。 CMSはWebアプリケーションであり、Webサイトの標準では、検索ボックスの配置、ドロップダウンメニューなどのコントロールの使用、およびリンクのシングルクリックが必要です。 リッチインターネットアプリケーションがより一般的になっている一方で、新しい標準が登場しますが、これがイニシアチブの理由ではありません。 機能は、従来とは異なる相互作用の方法（たとえば、マウスの右ボタンの使用やリンクのダブルクリック）によって失われる可能性があり、ユーザーの期待に合わないため、明らかではありません。



10.特定の条件に合わせてシステムを再設計します。



一貫性の原則は、標準操作環境でユーザーが使用する他のツールやアプリケーションと同様に機能するCMSの作成にも適用されます。 これらのツールとの統合は、テキストエディターからCMSへのコンテンツの簡単なコピーを提供する場合でも、CMSが標準ブラウザーで動作するようにする場合でも、組織の単一ログインを作成してユーザーが個別にCMSにログインする必要がない場合でも、非常に便利です。



CMSの再設計は、独自の設計とブランディングの一貫した実装を意味することもあり、その結果、ユーザーはCMSに対して「自分自身」を感じ、新しいシステムの導入（感情的な投資の創出）に大いに役立ちます。



この適応には、不要な要素の削除（推奨1）、インターフェイスの簡素化（推奨2）、特に重要なタスク（ヒント6）に関連した、コミュニケーション言語の簡素化（推奨3）も含める必要があります。 たとえば、特定の目標がイントラネットポータルのCMSである場合、用語が「そのまま」継承されたWebサイトではなく、イントラネットにユーザーインターフェイスを適合させる必要があります。



11.効果的なスタートページを作成する



CMSのメイン（開始）ページまたは管理パネルは、適切に設計され、有用な要素が含まれている場合に優れたツールです。 「すぐに使える」標準的な要素で過負荷にしないでください。ユーザーが最初にシステムにログインしたときに表示される内容をよく考えてください。



このページを使用して、すべてのシステム機能の優先度順に主要なタスクを解決し、メインタスクに特別な注意を払うための妨げられないアクセスを提供します（ページの上部に、場合によってはアイコン付きで表示する必要があります）。 あまり使用されていない、または重要度の低い機能は、それらをすべて表示することにした場合、小さく表示され、ページの下部に配置される場合があります。



この段階では、ログインプロセスを合理化する必要があります。 CMSで複数のサイトを編集できるが、特定のユーザーがそのうちの1つのみを編集する場合、サイトにログオンするたびにリストから選択させないでください。



さらに、管理パネルまたはメインページを放棄することを検討し、一般的に最も一般的なタスク（存在する場合）にユーザーを直接送信します。必要に応じてメインページに移動することもできます。



12.高品質の形状を作成する



ほとんどのCMS製品の主なデータ入力方法はフォームです。 そして、原則として、それらはあまり便利ではありません。 フォームは、論理構造、わかりやすい指定、デューデリジェンス、明確な行動要請を備えて、適切に設計する必要があります。 推奨事項7で述べたように、最終製品に近いフォームを使用すると理解しやすくなります。 フォームを作成するための優れたヒントは、Luke Wrublewski- Web Form Designを提供します。



13.デザイナーを関与させる



システムを作成してインストールする開発者にユーザーインターフェイスの設計を任せないでください。 インターフェイスを開発するために必要なスキルを持っている場合でも、CMSを開発する際に日常的な詳細に関与しすぎるため、さらなる開発の見通しを把握することはできません。 経験豊富なユーザビリティの実践者でさえ、エンドユーザーを視野に入れるのは難しいと感じています;彼らはシステムを作成する技術的な複雑さにあまりにも集中しています。



ほとんどの企業は、グラフィックデザイナーや使いやすさを最終製品への道のツールとして使用することを考えていません。 しかし、彼らはこのツールの重要性を、特に大規模なサイトに関しては過小評価しています。



14.トレーニングに頼らないでください



システムの使用方法を学習しても、利便性の問題は解決しません。 トレーニングセッションの後、インターフェースが直観的でなく、システムが複雑な場合、問題が発生し続けます。 最悪の場合、教師自身でさえ、システムがどのように機能し、どのように使用するかを説明するのは難しいでしょう。なぜなら、彼らは技術システムとユーザーの間のギャップを埋めようとしているからです。



私の意見では、CMSトレーニングプログラムは生活を少し楽にしますが、ほとんど効果的ではありません。 なぜ時間とお金を費やし、システムを使いやすくするだけではありませんか？ そして、誰もがトレーニングなしでそれを使用することができ、彼らが学んだことを繰り返す必要があります。



15.ユーザー支援を提供する



トレーニングでは、設計が不十分なシステムは保存されませんが、ユーザーに支援を提供する必要があります。 これらは、ユーザーが何をどのように行うべきかを説明する指示かもしれません。 彼らは、トレーニングの助けを借りて、または「初心者」モードで学習プロセスをデモンストレーションできます。これにより、いつかユーザーはCMSを使用して快適に感じることができます。



指示に加えて、コンテンツの操作とユーザーのタスクの解決のプロセスでヘルプが必要です。 通知やエラーメッセージと同様に、ヘルプが役立ちます。 ユーザーインターフェースでユーザーに表示されるものに関するヒント（「この画面にはXYZボタンが含まれる」など）だけでなく、特定の機能の使用方法、それらに期待すること、およびそれらを復元する方法も説明する必要があります。何かがうまくいかない場合。



もちろん、参考資料の必要性を減らす方法は、わかりやすいインターフェイスを開発することです。インターフェイスの使い方は明らかです。ナビゲーションボタンの高品質なショートカット、ページとフォームの明確な視覚的階層。 ページまたはリスト内で最も有利な位置は、最も重要なポイントに与えられます。 明確なアクセントと明確な識別のおかげで、ユーザーがアクションを実行しなければならないシーケンスは、すべてのステップで明確になります。



16.開発者インターフェースを検討する



その過程で、次のような頻繁な異論に出くわすでしょう。「しかし、これは開発者にとって有用です...」 そして、もちろん、これを支持する議論があります。なぜなら、開発者や他の技術専門家は通常のユーザーと一緒にシステムを使用するからです（おそらくそれよりも少ないでしょう）。 通常のユーザーインターフェイスを損なうことなく、CMSと最終製品の管理に必要なすべてのシステム機能を使用するためのアクセスを許可できます。 ユーザーインターフェイスの外観は、ログインによって異なります。



17.準準備状態でリリースしないでください。



多くの場合、何かを最小限に抑え、後でそれをすべて修正したいという要望があります。 つまり、生の製品をリリースするために、利便性は「後で処理されます」。 これは悪い考えです。 まず、最初から何かが機能しない場合、ユーザーは新しいCMSに感謝と信頼を感じることはほとんどありません。 第二に、将来のトラブルシューティングでは、ユーザーがシステムを再調査する必要があります。 第三に、この「後で」は起こりそうにありません。さらに、買い手がすでにCMSの購入/開発に投資している場合、当然、最良の方法で結果を期待します。



これはすべて、ユーザーインターフェイスを含むシステムの継続的な改善の可能性を排除するものではありませんが、これは、CMSがリリースの瞬間から使用可能で便利であることを意味します。 CMSを計画および開発する場合、ユーザーのニーズを考慮する必要があります。ユーザーのニーズを最後まで残すことはできません。そうしないと、ユーザーのニーズを満たすのに十分な時間またはリソースがありません。



おわりに



これはどれほど重要ですか？ ツールが機能する限り、周りの生活は凍結しますよね？ いえいえ使いやすさの欠如は、システムをエンドユーザーに習得するのを難しくし、その結果、CMSの売上の減少、または不正使用の増加につながります。



そして、これらのヒントは単なる基礎であり、明らかなことです。サイトの管理を改善するのに役立ちますが、別のより良いシステムを選択する方が良いかもしれません。行う必要がある重要なことに集中し、「機能」や一度だけ使用する機能にとらわれないようにします。設計されたCMSが、製品評価のシナリオで説明されているキーポイントを考慮していることを確認してください。特にユーザーインターフェイスを構成する必要がある場合は、これが簡単に行われ、将来のアップグレードの可能性に影響しないことを確認してください。したがって、ユーザーインターフェイスとシステムの間で独立性を維持する必要があります。
