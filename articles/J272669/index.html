アプリケーションサーバーとしてのTarantool

こんにちは、Habrauser％。 タランチュラチームは 、高負荷のプロジェクトでデータを効果的に使用するための洞察と専門知識を共有し続けています。 今日は、Tarantoolが「ツーインワン」である理由を解明しようとします。データベースだけでなく、アプリケーションサーバーもそうです。 たぶん、TarantulaをLuaでのレプリケーションとストレージをサポートする超高速で永続的なメモリ内ストレージと聞いた人もいるでしょう。 Redisの断片を取り出し、凍結したNode.jsを追加し、Goで補充してから、 煮沸してから5分間ゆっくりと調理します。 アプリケーションサーバーはそれと何の関係があるのでしょうか？







多くの人が驚くでしょうが、nginx、Go、Node.js、Redis、MongoDB、Tarantoolなどの優れた製品には、アーキテクチャ面で多くの共通点があります。 何らかの方法で高性能ネットワークサーバーを作成するには、非ブロッキング入出力、非同期イベント処理、メモリの操作、エラー処理、ログ、悪魔などを提供する一連のライブラリが必要です。このランタイムエンジンは通常、かなり複雑なことであり、さまざまなシステムの基礎と低レベルのプログラミングスキルを深く理解する必要があります。



Tarantoolチームの私たちは、非同期イベント処理、ノンブロッキングI / O、ユーザー空間（ファイバー）のグリーンスレッド、協調マルチタスク、特殊なメモリアロケーターのファミリーなどを備えた独自のランタイムを作成し、非常に長い道のりを歩みました。 Goと最もアーキテクチャ的に似ています（愚か者も同じ考えを持っています）が、純粋なCのライブラリの形でのみです。この基盤のおかげで、ボンネットの下に「DBMS」を作成することができました。記憶 市場のフットプリント。 ある時点で、アプリケーション開発者に無制限の自由を与え、ストアドプロシージャでデータベースクエリを実行できるだけでなく、ツールキット全体を完全に使用できるようにすることにしました。 データベースから直接クラウドサービスからHTTP経由でJSONを取得および解析するのは簡単です。 DBMSでRESTサービスを直接実行します-お願いします。 多数のサーバーでデータを取得し、クエリを並列処理します-問題ありません！ すべての機能とツールは開発者の手にあります！ 開発者、開発者、開発者！

興味深いことに、3層アーキテクチャがありますか？







「別のアプリケーションサーバー（Node.js、PHP、Pythonなど）とDBMS（Redis、MongoDBなど）ではなく、すべてのビジネスロジックをデータベースに転送することを再度提案しますか？」 いいえ、私たちは宇宙の基礎を侵しているわけではありません。 もう少し実用的に見てみましょう。 ブラウザーでスペースアプリケーションを直接直接書き込むことができる場合、サーバー側はデータの保存と処理のためにほとんどの部分を維持しますが、ブラウザーはAJAXを介して必要なすべての情報を動的に要求および更新できます。 このシナリオでは、アプリケーションサーバー（PHP / Node / Go / Python）は何をしますか？ データベースからの応答を待機していて、すぐにすべてをnginxのJSON形式で提供するのはアイドルですか？ ただし、トランザクションを開いてから、ネットワーク経由でデータベースからデータをダウンロードし、それらの一部のフィールドをアプリケーションサーバーに変更し、更新をデータベースに送り返し、トランザクションを閉じて結果をnginxに返す必要があります。 これに追加のネットワークラウンドトリップおよびユーザー空間<->カーネル<->ユーザー空間の切り替えを何回費やしますか？ それでも、データベースは、たとえばロックやその他の重いメカニズムを使用するコストで、開いている各トランザクションのデータの完全な読み取りビューをサポートする必要があります。 1つのプレートから5つのレコードを選択し、別のプレートの2つのレコードを更新し、結果をRESTサービスに返すために、これらすべてが必要ですか？



この種のマイクロサービスのために、TarantulaはDBMS内でデータ自体のすぐ隣にストレージを書き込むための釣り竿を提供しています。 tarantool-httpモジュールとnginx_tarantool_upstreamは 、TarantulaからRESTサービスを簡単に整理し、サービスアーキテクチャを簡素化し、専用のアプリケーションサーバーの形で不要なリンクを削除します。 同時に、プロジェクトの最も負荷の高い部分のみをマイクロサービスとして選択できるため、この方法でアプリケーション全体を書き直すことを申し出る人はいません。従来のソリューションの生産性マージンは十分ではありません。 残りについては、さまざまなプログラミング言語のコネクタを介して、汎用DBMSと同じTarantoolを使用できます。

 #!/usr/bin/env tarantool box.cfg{} --      box.once('schema', function() box.schema.create_space('hosts') box.space.hosts:create_index('primary', { type = 'hash', parts = {1, 'str'} }) end) --  GET-  / local function handler(self) --  IP-  local ipaddr = self.peer.host --         box.space.hosts:upsert({ ipaddr, 1 }, {{'+', 2, 1}}) --      JSON  return self:render{ json = box.space.hosts:select() } end local httpd = require('http.server') local server = httpd.new('127.0.0.1', 8080) server:route({ path = '/' }, handler) server:start()
