エンタープライズクラウド：接続オプション





ガートナーによると 、クラウドテクノロジーはますます人気が高まっています。 2012年にIaaSの市場シェアが61億ドルに達した場合、2015年にこの数字は160億ドルに成長し、さらにIaaSの人気の成長率は低下しません 。



IaaSにより、多くの問題を解決し、柔軟性と拡張性を提供し、さまざまなクライアントのニーズに対応できます。 IaaSに目を向けると、リソースをオンデマンドで提供する機能と単一の管理システムを備えた（従来のインフラストラクチャと比較して）費用対効果の高いソリューションが得られます。



今日、ますます多くの企業がインフラストラクチャをクラウドに移行しています。 このアプローチは、経済的に正当化され、収益性が高く、便利です。 質問をするかもしれないポイントの1つは適切な接続メソッドの選択です。 原則として、クラウドサービスへの接続には、RDPクライアント、RemoteApp、Webアクセス、リモートアクセスVPN、VPNサイト間、DirectAccess、VDIのソリューションが使用されます。



RDPクライアントまたはリモートデスクトップ接続は、クラウドに展開されたワークステーションへのアクセスを提供する最も便利で汎用性の高いツールの1つです。 独自のRDP（リモートデスクトッププロトコル）プロトコルに基づいています-それは、ターミナルアクセスサービスを実行しているコンピューターへのリモートアクセスをユーザーに提供するものです。 ユーザーは、RDPクライアントを介してターミナルサーバーに接続し、リモートシステムのデスクトップを表示します。



確立されたセッションのフレームワーク内で、ターミナルサーバーに展開されたアプリケーションを実行できます。 この接続のセキュリティは、HTTPSプロトコルを介したRDPを使用するリモートデスクトップゲートウェイによって保証され、強力な暗号化方式を提供します。 現在、Windows、Linux、FreeBSD、Mac OS X、iOS、Android、Symbianファミリーのほぼすべてのオペレーティングシステム用のクライアントがあります。



RemoteAppは、上記のオプションのバリエーションです。 ただし、この場合、ユーザーには実行中のリモートアプリケーションのみが表示されます。 このオプションは、特定のアプリケーションへのアクセスを制限する必要がある場合、またはユーザーが何らかのクラウドアプリケーションを使用してローカルマシンでの作業を結合する場合に役立ちます。 この場合、起動されたリモートターミナルサービスアプリケーションは、ユーザーのシステムで実行されているように見えます。



クラウド内の特定のアプリケーションおよびデスクトップへのアクセスは、ブラウザーを使用して調整できます。 ユーザーはブラウザを起動し、必要なアドレスを入力してログインし、すでにアプリケーションまたはリモートデスクトップで作業しています。



クラウドリソースに接続するための別の便利で安全で頻繁に使用されるツールは、リモートアクセスVPNです。 この場合、クライアントのコンピューター上のアプリケーションと、たとえばホスティングプロバイダーのクラウドにあるVPNコンセントレーターまたはルーターとの間に安全なトンネルが確立されます。



リモートリソースに接続するには、ユーザーはVPNショートカットを起動し、認証に成功すると、目的のリソースにアクセスします。 したがって、ユーザーのコンピューターはクラウド内の仮想リモートオフィスのネットワークに入り、会社のオフィスに直接配置されているかのようにリソースを使用できます。







ただし、このシナリオも可能です。非クラウドインフラストラクチャの従業員は、クラウド内のリソースに接続する必要があります。 これを行うには、サイト間VPNがあります。これは、トンネルが形成される間に2つのデバイスが存在することを意味します。



この場合、ユーザーは「デバイスの背後」にいるため、コンピューターに特別なソフトウェアをインストールする必要はありません。







たとえば、ここでVPN Server 1はVPN Server 2サーバーとのVPN接続を確立します。その後、ユーザーは要求されたリソースの内容を確認します。 クライアント側では、発信VPN接続を作成する必要はありません。



クラウドへのリモート接続に使用できる通常のVPN実装に加えて、DirectAccessという別のテクノロジがあります。 この場合、ユーザーのコンピューターがインターネットに接続するとすぐに、ユーザーはすぐにインターネットリソースおよび企業ネットワーク全体にアクセスします。 つまり、DirectAccessクライアントとして構成されたユーザーコンピューターは、DirectAccessサーバーへの信頼できるトンネルを自動的に構築し、それを介してアクセスを取得します。 この場合、ユーザーは追加のアクションを必要としません。



もう1つの方法は、仮想デスクトップインフラストラクチャ（VDI）です。 企業のIaaSプロバイダーの多くのクラウドプラットフォームに実装されています。 このテクノロジーにより、ユーザーワークステーションを仮想化サーバーに集中化しながら、管理、展開、およびメンテナンスの単一ポイントを作成できます。 プロバイダーのクラウドでは、ハイパーバイザーを備えたサーバーが割り当てられ、その上に個々のVMが展開されます。 ユーザーはクライアントを起動し、インフラストラクチャに接続します。



このタイプの接続は、一見、RDP接続と大差ありません。 しかし、違いは何ですか？ ターミナルサーバーへのRDP接続の場合、これは共有Windowsサーバー上の別個のセッションです。 VDIの場合、クライアントオペレーティングシステムを備えた独立した分離コンテナです。



前述したように、ますます多くの企業がクラウドを使用し始めています。 より大きな数...しかし、すべてではありません。 なぜクラウドはビジネスにとってそれほど怖いのですか？ 1つの理由はプライバシーの問題です。 心配することはありません。落ち着いて眠るには、情報を扱う現代のニュアンスを理解するだけです。 クラウド保護は、基本的に従来のデータセンターと変わりません。 多くの場合、使い慣れたセキュリティツールを使用することも可能です。 特に、IaaSを使用すると、保護メカニズムを実装して、必要なレベルの機密性を取得できます。



2番目の一般的な誤解は、おそらく、これらの新しい技術はまだ未加工であり、深刻な作業には適していないということです。 クラウドインフラストラクチャは、実績のある技術に基づいています。 たとえば、IaaSはVMware vSphereなどの仮想化システムにほぼ完全に依存しています。 このような製品は10年以上にわたって改善されており、サーバーインフラストラクチャの柔軟性を高めることができる信頼できるソリューションと見なすことができます。







上の図は、VMware製品の最初の仮想化システムから、企業や商業データセンター向けの統合ソリューションまでの簡単な進化を示しています。 すべてのソリューションは、10年以上の歴史を持つテクノロジーであるESXiハイパーバイザーに基づいています。



Gartnerの調査によると、クラウドコンピューティングに関する最も一般的な神話の1つは、ビジネスクリティカルなアプリケーションに適用できないことです。 「私たちのアプリケーションはクラウドにとって大きすぎて重要です」-このようなフレーズは頻繁に聞くことができます。 クラウドテクノロジーは仮想化システム上に構築されているため、アプリケーションを仮想サーバー上で実行できれば、クラウドに問題はありません。



仮想およびクラウド環境での「大規模」アプリケーションの作業の実際の例として、製品SAP HANAを提供できます。 SAPは、自動化プロセス制御システムの世界的な大手メーカーとして長い間その地位を確立してきました。 HANAは、要求の厳しいアプリケーション向けの高性能DBMSです。 すべてのユーザーデータベース情報はサーバーのRAMに保存され、最小限の遅延でクエリに使用できます。 メーカーは、仮想VMware環境におけるこのプラットフォームの完全な機能を確認しています。



サーバーインフラストラクチャ全体をどこかに移動する場合、データ移行の問題は最も深刻です。 すべてが簡単でシンプルであるとは言いませんが、過去にすでにこれに遭遇したことがあれば、ほとんどの困難は過ぎ去ったとみなすことができます。 ただし、質問の1つは依然として重要です。「ホスティングプロバイダーのリモートサイトにターミナルサーバーを展開する場合、ターミナルサーバーに割り当てる必要があるリソースの数は？」



ターミナルサーバーのリソースの正確な計算は依然として複雑なので、これを行うのに役立つ方法を検討することをお勧めします。



パイロットは、おそらくテスト仮想マシンがクラウドに展開され、ターミナルサーバーの役割を果たすときに実際に使用される最も簡単な方法です。 サーバーの負荷は徐々に増加していますが、同時に、使用されるリソースが監視され、異なる期間のパフォーマンスインジケータが比較されます。 収集されたデータに基づいて、結論が導き出されます。



ユーザーシステムへの外挿-この方法は、1人のユーザーが使用するシステムから受信したデータに基づいています。 メモリ、プロセッサ、ディスク容量、ネットワークなどのパフォーマンスの側面が評価されます。 将来的には、マルチユーザー環境で容量を計算するために使用されるのはそれらです。



モデリング-このメソッドは、スクリプトのフレームワークで既に収集されたインジケーターに基づいています。 ここではテストサーバーも強調表示されており、特別なツールを使用して、さまざまなレベルの負荷をシミュレートしてサーバーの機能を判断しています。



後者の方法は、システムの可能性を正確に評価する方法の1つです。 動作させるには、最初に明確なユーザーアクションのシーケンスを決定し、ユーザーがデータ（ドキュメント、ファイル、メディアコンテンツなど）をどのように使用するかを理解する必要があります。



たとえば、ユーザーのレビューとアクティビティの監視は、タスクに対処するのに役立ちます。 さらに、テスト環境の展開後、テスト自体が実行されます。このステップの本質は、サーバーに負荷を形成し、システムの実行可能性を判断することです。 ここでパフォーマンステストを実行して、ターミナルサーバーが処理できるユーザーの最大数を決定します。



この段階では、自動化ツールが使用され、複数のユーザーの作業をシミュレートするアプリケーションのインスタンスが同時に起動されます。 人気のある「シミュレーション」ツールの1つは、AutoITスクリプト言語です。 スクリプトを使用すると、マウスとキーボードからのユーザー入力をシミュレートしたり、ネットワーク、レジストリ、クリップボードなどを操作したりできます。



WinBatchおよびAutoITスクリプトを使用すると、コマンドラインオプションをサポートしていないアプリケーションを含むアプリケーションを自動化できます。 モデリング後は、結果を分析し、システムの許容可能な負荷を決定するだけです。 得られた結果に応じて、追加のリソースを増やす必要があるかどうか、または適切な操作に十分であるかどうかを判断します。



ご覧のとおり、ターミナルサーバーをIaaSプロバイダーのクラウドに展開するときに必要なリソースを計算するためのさまざまな方法とツールがあります。 残っている主なことは、考慮されているすべてのメソッドの正しい使用です。
