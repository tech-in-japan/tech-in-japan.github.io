Merlinでプロジェクトの予算計画を作成する方法（Mac OS）

プロジェクトの販売前段階では、常に2つの疑問が生じます。

  1. あなたは何をしますか？
  2. 費用はいくらですか？


それらにすばやく正確に回答するには、プロジェクト予算を作成するのが最も簡単です。







計画と予算編成が比較的簡単なプロジェクトがあります。

  1. インターフェース設計
  2. Content Firstスキームに従ってサイトを構築します。
  3. 企業のスタートアップのアイデア開発の開始。
  4. 簡単な（！）技術的なタスクを作成します。
  5. すべてのストライプのテストと監査。
  6. コースまたはセミナーの実施。


予算計画は 、次の質問に答えません 。

  1. なぜプロジェクトを行う必要があるのですか？
  2. 特定のサブタスクの本質と意味は何ですか？
  3. 誰が具体的にタスクを実行しますか？


各企業には、予算計画を作成するための独自の微妙さと好みがあります。 アプローチについての本が書かれています。

しかし実際には、常に簡単な指示が必要です。それを実行してから実行すると、すべてがうまくいきます。

以下はそのような指示です。Merlinプログラムでプロジェクトの計画予算を段階的に作成する方法。







Merlinは、Mac OSオペレーティングシステム用のMS Projectの類似物です。 選択肢はありますが、この記事ではそれらについては何も書きません。



理論全体を省略しようとしました。 コメントでは、「なぜそうなのか、そうでないのか」という質問について話し合う準備ができています。



この記事は、顧客、営業担当者、およびプロジェクトマネージャーに役立ちます。 やや少ない-アナリスト、デザイナー、プランナー、テクニカルライター、エディター、コンテンツマネージャーなど。 この資料はプログラマーとは関係ありません。



手続き



出力では、プロジェクトの計画予算がガントチャートの形式で取得されます。 結果の形式はPDFファイルです。 言語はロシア語です。 特別な指定の数は最小限です。初心者が5〜10分でドキュメントの本質を簡単に説明できます。



手順の順序は次のとおりです。

  1. タイミングとコストに関する期待を策定します。
  2. プロジェクトをセットアップします。
  3. 作業の段階をリストします。
  4. 各ステージについて：

    1. 作業のシーケンスを記録します。
    2. 中間結果をリストします。
    3. 承認とプレゼンテーションを貼り付けます。
    4. アクションの順序を設定します。
    5. 反復回数を見積もり、それらを計画に追加します。
  5. 各ステップの人件費を見積もります。
  6. 遅いジョブのカレンダーフレームを設定します。
  7. パフォーマーを任命する。
  8. 合計とコストを予想と比較します。
  9. 必要に応じて再確認して修正します。
  10. プロジェクト予算計画を含むPDFファイルを作成します。




ニュアンスを以下に説明します-ニュアンスのみ！ -すべてのステップ。 明らかなアクションはスキップします。それでも、これはダミー向けのトレーニングコースではありません。



目的：最初から20〜40分で、非定型プロジェクト（作業リストの100〜300項目）を計画します。 ドキュメントの作成者はすばやく考え、さらに速く印刷すると想定しています。



ターンベースのベビーベッド



1.タイミングとコストに対する期待を策定する



この手順はMerlinとは関係ありません。 しかし、評価が「天井から」どれほど危険で不正確であるかを理解するのを助けるのは彼です。



計算なしの見積もり：出口で受け取る条件と費用はどのくらいですか？ これらの2つの数字を、ファイル、Evernote、またはステッカーに書き込みます。







迅速かつ静かに評価します。 これは冗談ではありません。自分で決断を下し、他人を拷問しないことが重要です。



2.プロジェクトを設定する



一部のプロジェクトパラメータは、Merlinインターフェイスで掘り下げるのが困難です。 そして、時には彼らが一般的に求められ、変えられる必要があると推測するのは難しいです。







3.作業の段階をリストする



段階として、特定の結果につながる簡単なアクションの短いシーケンスを検討します。 7〜10個のタスク。 大きなブラシストロークプロジェクト計画は、ある段階から別の段階へのパスです。







ステージは次々に続きます（ごくまれに並行します）。

  1. 標準や既製のテンプレートは忘れてください。 他の人の方法に従うのが好きな人はいますか？ カオスから独自の構造を作成することは、はるかに意味のある作業です。
  2. ほとんどの場合、2レベルのネストで十分です。 3-常に。
  3. ほとんどすべてのプロジェクトの2つの必須段階は、開始と終了です。 練習には他にどのような必須のステップがありますか？
  4. 手順を詳しく説明するには早すぎます。 これが次のステップです。




4.各ステージについて



多くの人がタスクのリストを作成し、各段階の予算を把握し、これが計画の終わりであると考えています。

これでは十分ではないと考えています。 各段階の詳細を確認してください。

粒度が細かいほど、実行者との会話で計画エラーを特定しやすくなります。 そして、予算がより正確になります。



4.1。 作業手順を記録する



可能であれば、すべての作業を事前に考慮する必要があります。 そして、それらが特定の中間結果につながるように調整します。







  1. 思い浮かぶすべての仕事を書き留めてください。 重要な質問：「結果に近づくために、小さくて理解できることは何ですか？」
  2. 結果のリスト内の各項目について、「...のために何をすべきか」と「その後に何が起こるか」という2つの質問をします。リストに回答を追加します。
  3. 結果をストーリーとして読み、イベントの展開を想像してください。 このストーリーのギャップを見つけます。 それらを新しい仕事で満たします。
  4. 作品の名前はすべて「バナーのスケッチを描く」、「チーム内の要件を話し合う」という動詞で始まります。
  5. この文言は、「何をすべきか？」という質問に答えます。 つまり、最初は動詞だけでなく、完璧な形です。私がやるのではなく、やるのです。 唯一の例外があります-「何かのプレゼンテーション」。 以下彼について。
  6. ある段階では10作品以下です。 それらの数が15に近づいている場合、それを2つの段階または複数の反復に分割する方法を見つけます。
  7. 会議と議論は仕事です。 そして非常に大きい：総量の約60％。 時には80％。 すべての会議とディスカッションを計画に含めるようにしてください。 同じことが電話にも当てはまります。
  8. 会議や議論から修正することも有効です。




4.2。 中間結果をリストする



結果はどうなりますか？ 何を拾えますか？ チームの創造的な努力を具体化したものは何ですか？ どの中間的な「レンガ」を作成し、調整し、仕上げ、最後に提示して使用しますか？ このことの名前は何ですか？ その結果、ステージが明らかに有用かつ具体的になります。







  1. 結果を短く正確に示します。 「コメントのリスト」は悪いです。 「デザイナーのコメント」はすでに優れています。 「インターフェイスを完成させるための提案」-少し長いですが、適切です。
  2. 条件に同意します。 人生の状況はユーザースクリプトと同じではありませんか？
  3. フォーマットをロックします。 インターフェースのプロトタイプはHTMLファイルのセットですか、それとも画像ですか？
  4. 何かの最終バージョンを呼び出します—最終バージョン：「プレスリリースの最終バージョン」。
  5. 概念を明確にします。 「参照規約」-3つのリーフレットまたは3つのボリュームですか？ 「デザイン」はスケッチですか、それとも100,500のスケッチですか？




4.3。 承認とプレゼンテーションを貼り付ける



顧客とプロジェクトチームとのコミュニケーションは仕事です。 意味のある。 そして非常に難しい。 各ステージには、少なくとも1つの承認、ディスカッション、プレゼンテーションが必要です。







  1. 必要に応じて、「ステージの開始時に先に進む」という作業を行います。
  2. 少なくとも3日ごとにクライアントと通信できるように、会議とディスカッションでワークフローを希釈します。
  3. すべてのステージの結果の最新バージョンは、常に同意する必要があります。 これが作品です：「プロジェクトの理論的根拠に同意します。」
  4. 各段階に適した用量-クライアントとの2回の接触。 一度「結果の最初のバージョンについて話し合う」、2番目の「最終バージョンに同意する」。
  5. 計画のステージごとに、クライアントとの会議が3〜4回行われることがあります。 より多くは計画ミスである可能性が高くなります。ステージを2つに分割します。
  6. プレゼンテーションは、「インターフェイスコンセプトのプレゼンテーション」という動詞なしで記録できます。 はい、ルールの例外です。
  7. プレゼンテーションの準備が必要です。 これは仕事です。 計画に挿入します：「プレゼンテーションスライドの準備」、「プレゼンテーションの本質を意思決定者と事前に話し合う」など。
  8. 会議を整理する必要があります。 参加者がゴキブリのように散らばり、会話がほとんどない場合、これも「約束をする/電話をかける」という仕事です。
  9. 内部の議論も有効です：「上級デザイナーとプロトタイプを議論してください。」
  10. 各ディスカッションの後に、「ディスカッションの結果に基づいて修正を加える」というタスクを挿入します。




4.4。 手順を定義する



ガントチャートでは、プロジェクトの合計期間は小さなタスクの期間で構成されます。 そのため、タスクはチェーンで接続する必要があります。何が背後にあるのか、何が何に平行するのか。







  1. ごくまれに、完了-開始よりも複雑なものが必要です。
  2. [完了-開始]リンクがとんでもないように見える場合は、中断します。 そして、ジョブ間のより細かい接続を個別に調整します。




4.5。 反復回数を見積もり、それらを計画に追加します。



カスケードパターンで実行できるのは非常に小さなプロジェクトのみです。 通常、プロジェクトの中核は、「意味のある」作業の複数の反復です：設計、テスト、要件の収集、テキストの作成など。







  1. 反復は作業の点で同じでなければなりません。 またはほぼ同じ。
  2. 繰り返しの数は、予算計画ですでに見積もられている必要があります。 これは不快ですが、必要です。 やはり必要です。 したくないのですが、する必要があります。
  3. 設定する反復回数はまったくわかりません。3を入力してください。
  4. 大規模で不明瞭な顧客の場合は、17を入力してください 。
  5. 繰り返しの回数は、顧客との交渉の問題です。 賢明な顧客は、2つのイテレーションですべてを迅速に調整することが自分の利益になることにすぐに気付きます。 交渉のためのスペースを確保したことを確認してください。プロジェクトの意味を失うことなく、1つのイテレーションをスローできます。




5.各ステップの人件費を見積もる



驚くべきことに、「天井」評価でも機能します。 推測してください（はい、これはすべてのルールに反しています）。 ただし、計画の各項目の実行者に問い合わせることをお勧めします。







  1. 進行から人件費の見積もりを選択します：1–2–4–8–16。
  2. 16時間以上のすべてをいくつかの作品に分けます。
  3. 最小人件費-1時間。 たとえ実際には5分であっても、潜り込むには小さな時間がかかります。
  4. 評価する時間はわからない-4時間置きます。 カットする時間があります。
  5. オフィス外のすべての会議-4時間。 モスクワの場合-8。
  6. より正確には、2〜4時間の増分よりも、どのような場合でも評価する必要はありません。
  7. 成績をパフォーマーと比較するときは、「この作業は2時間未満ですか、それとも4時間ですか？」と質問します。ゼロから評価するように求めないでください。




6.遅いジョブのカレンダーフレームを設定する



5分間-動作するタスクがあります。 ただし、これらの5分間は、忙しい人のスケジュールで見つける必要があります。 人件費だけでなく、カレンダーの時間もそのような計画項目のために予約する必要があります。これはタイムラインに影響しますが、プロジェクトの費用には影響しません。







  1. ほとんどのタスクは、個別のカレンダー時間を必要としません。
  2. カレンダーの範囲を日単位で書きます。 数時間ではありません。
  3. すべての会議とプレゼンテーションに2暦日を割り当てます。 シフトはほとんど避けられません。
  4. 通話と調整の場合-1日。 同じ理由。 内部会議にも適用されます。
  5. 「統計の収集」タイプのタスクにカレンダー期間を設定します。 5分でデータを確認できますが、数日または数週間も蓄積されます。




7.パフォーマーを割り当てる



珍しいステージ。 通常、計画の最初の段階でプロジェクト全体に割り当てた1時間ごとのレートで「デフォルトの請負業者」で十分です。







  1. 他の人の作業を計画に含めるには（たとえば、顧客の従業員が何かをしなければならない場合）、コストがゼロのリソースを作成します。
  2. 特別なレートの出演者がいる場合、それらのみを明示的に割り当てます。 残りのタスクはデフォルトのエグゼキューターのままにします。
  3. 困難な場合でも、通常、予算を見積もるためにレートの異なる3人のパフォーマーを作成するだけで十分です。
  4. 出演者の任命後、計画のタスクの番号付けが行われます。 これはあなたのせいではなく、単にマーリンの機能です。 「損失タスク」を探しないでください。






8.納期と費用を期待して検証する



最初の段落からステッカーを見つけます。 何が書かれていましたか？ 結果と比較します。







  1. あなたの「間違い」は何も意味しません。
  2. 正確なヒット-も。
  3. 重要なことは1つだけです。時間の経過とともにどのようなエラーが許容可能と見なされますか？






9.必要に応じて、再度確認して修正します。



結果が満たされないことが起こります。 特に顧客。 その後、すべての手順を再度実行する必要があります。 そして、2つの質問を自問してください。

  1. 計画を夢に近づけるために何を変えることができますか？
  2. ここで間違いは何ですか？


作業パラメータを編集する必要がある場合もあれば、プロジェクト設定全体を編集する必要がある場合もあります。







  1. 変更できないことがあります。 または、変更後、状況が悪化しただけです。 それでは、それが真実のようです。
  2. このような計画を不合理な期待に合わせるのは不可能です。 そしてそれは良いことです。
  3. 通常、レビュー後、5〜7個の忘れられたタスクが表示されます...
  4. ...そして、人件費の改善により、価格が20％削減されます。
  5. カレンダーの日付はほとんど削減できません。




10.プロジェクト予算計画を含むPDFファイルを作成する



奇妙なことに、これは重要なインターフェイスタスクです。
