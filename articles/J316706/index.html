ゲームでのマルチプレイヤーの実装。 Game Center、Steamworks、GameSparksの機能の比較

マルチプレイヤーモードのゲームは、マルチプレイヤーモードのないゲームよりもはるかに興味深いものです。 ただし、マルチプレイヤーの実装は、独自のサーバー、ネットワークインタラクションを実装するコード、マッチメイキングなどの存在を意味します。



幸いなことに、多くの既製のソリューションがあります。 ゲームの開発中に、次のオプションを試しました。



  1. Game Center（iOSゲーム）
  2. Steamworks（Steam用ゲーム）
  3. GameSparks（クロスプラットフォームソリューション）


カットの下で、これらの各オプション、主な機能、遭遇した困難、賛否両論についてお話します。 コードの一部は控えます。 すべてがドキュメントにあります。



すべてについて簡単に



リストされているオプションは、機会と柔軟性の高い順に並べられています。 実装の複雑さも増加します。



1. GameCenter

最もシンプルで、同時に最も貧弱なオプションです。 プレーヤーがボタンを押すと、敵の検索ウィンドウが表示されます。 一定数のプレーヤーが蓄積されると、試合が始まります。 この時点で、プレイヤーはメッセージを交換する機会を得ます。 すべてのプレイヤーが試合を去った後、ゲームは終了します。



2. Steamworks

この機能は、マッチメイキングとメッセージングの2つの独立した部分に分かれています。 前半は、プレイヤーが今後の試合について議論できる部屋を作成する機能で構成されます。



後半は、ダイレクトメッセージングの機能です。 両方の部分は完全に独立しています。 プレイヤーがいる部屋に関係なく、いつでもプレイヤーにメッセージを送信できます。



3. GameSparks

実際、GameSparksは本格的なサーバーであり、すべての利点があります。 プレイヤーデータを保存し、彼のすべてのアクションを確認できます。 マッチメイキング、リーダーボード、アチーブメントなどのための組み込みのメカニズムがあります。



最初の2つのオプションは、セキュリティ要件の低いシンプルなゲームに適しています。 そして、大きな真剣なゲームには、もちろん、あなた自身のサーバーが必要です。



私にとって、最後のオプションは開発で最も快適であることが判明しました。多くの松葉杖を取り除くことができたからです。



私たちのゲームについて少し



GalaxyAdmiralsは、ターンベースの宇宙戦略です。 ゲームプレイは、六角形の競技場での短い戦いで構成されています。 プレイヤーは交代で、ランダムな要素はありません。 本質的に、これは宇宙のチェスです。









ゲームには30のミッションのキャンペーンがあります。 マルチプレイヤーモードは、ランダムな対戦相手との1対1の戦いです。







ゲームはUnity3Dで行われます。



ゲームセンター



概要



Game CenterはiOS向けのこのようなアプリケーションです。 それは他のゲームであなたの実績を表示します：実績、ハイスコア。 友達を追加したり、チャレンジしたり、実績を比較したりできます。 機能は素晴らしいものではありませんが、私の意見では、これらすべての使いやすさはかなり疑わしいです。 しかし、最も重要なことは、Game Centerを介してシンプルなマルチプレイヤーを実装できることです。



統合



おそらく、既製のプラグインがいくつかありますが、Objective Cで独自のプラグインを作成しました。多くのコードはなく、非常に簡単です。



実装



Game Centerを介してマルチプレイヤーを実装するには、3つのオプションがあります。



  1. ターンベース。



    これは、チェッカー、チェスなどのターンベースのゲーム用です。 プレイヤーは順番に交代します。 各移動の後、プレイヤーはGame Centerでメッセージを受け取ります。これは今度は自分の番です。 ただし、ゲームを実行する必要はありません。 ゲームのステータスはGame Centerに保存されます。 プレーヤーは互いに通信せず、サーバーと通信してゲームの状態を更新するだけです。



  2. 自己ホスト。



    Game Centerはマッチメイキングのみを提供します。 サーバーを介して独立してマルチプレイヤーを実装する必要があります。



  3. リアルタイム



    前の例のように、Game Centerリレーサーバーを介してメッセージングのみが行われます。 これは私が使用したものです。 以下で彼について詳しく説明します。


次のようになります。 プレイヤーがボタンを押す-敵の検索が開始されます。









マッチメイキングを開始すると、プレーヤーは次のパラメーターを渡します。



  • プレーヤーの最大数、最小数
  • グループ
  • 属性


グループは、一致に関する追加情報がエンコードされる番号です。 Game Centerは、あなたと同じグループを示した対戦相手を選択します。



属性は、プレイしたい役割を意味する32ビットの数値です。 これは本質的にビットマスクです。 Game Centerは、「または」で結合されたビットマスクがユニットで完全に満たされるようにプレーヤーを選択します。



たとえば、白だけのチェスをしたいとします。 次に、マスク0xFFFF0000を渡します。 黒のためにプレーしたいプレイヤーは0x0000FFFFを渡す必要があります。 プレイヤーが誰のためにプレイするかを気にしない場合、ユニットで埋められたマスク0xFFFFFFFFを渡します。 次に、白のためにプレーしたいプレーヤーは、黒のためにプレーしたい相手、または誰のためにプレーしたいかを気にしない相手をピックアップします。



対戦相手が見つかると、試合が始まります。 プレイヤーには、メッセージを送信できる相手のリストが返されます。 すべてのプレイヤーは、対等な立場でゲームに参加します。 プレーヤー番号1とプレーヤー番号2のどちらかに同意する必要があります。 たとえば、IDでプレーヤーを並べ替えることができます。 その後、両方が同じリストを取得します。



「マッチ完了」という特別な状態はありません。 プレイヤーは、まずゲームが終了したことをお互いに同意する必要があり、その後、それぞれが切断します。



友達に招待状を送ることができます。 この招待状は、ゲームが実行されていない場合でも、Game Centerに表示されます。 メッセージをクリックすると、ゲームが開始されます。 招待が正しく機能するためには、アプリケーションの実行中とそうでない場合の2つのオプションを別々に処理する必要があります。



残念ながら、プレーヤーが招待を受け入れなかった場合、それを送信したプレーヤーはそれを知ることができません。 彼は退屈するまで彼を待ちます。



問題



1.試合の完了に関する問題

この機能は、「ゲームオーバーです、私は行った」というメッセージを送信して、すぐに切断できないように機能します。 この場合、敵はあなたのメッセージを受信しませんが、あなたが切断したことを見るだけです。 あなたが落ちたと判断すると、彼は彼自身のために勝利を割り当てるでしょう。



したがって、「ゲームが終了したというメッセージを受け取ったので、そのままにしてください」など、追加の確認を送信する必要があります。



2.プレーヤーの1人の切断の問題

プレーヤー間の接続はピアツーピアで確立されます。 したがって、あるプレイヤーのインターネットが消えると、両方のプレイヤーが敵が倒れたというメッセージを受け取ります。 理論的には、倒れたプレイヤーに敗北を、ゲームに残ったプレイヤーに勝利を与えるべきです。 そして最も不愉快なことは、Game Centerに誰がゲームに留まり、誰が切断したかを尋ねる方法がないということです。



調べるには、松葉杖を挿入する必要があります。 たとえば、対戦相手が切断した直後にGame Centerへの接続を確認します（たとえば、実績のステータスを要求しました）。 Game Centerが応答しない場合、敵ではなく私が倒れたと想定できます。



しかし、これでも機能しない場合があります。 たとえば、プレーヤーはアプリケーションを折りたたみます。 彼がアプリケーションをデプロイすると、プレーヤーとの接続は既に切断されますが、インターネット接続は維持され、プレーヤーは彼ではなく、この対戦相手が落ちたと判断できます。 つまり、追加の松葉杖を挿入する必要があります。



短所



  • マッチメイキング機能が単純すぎる。
  • 不快な松葉杖を作成する必要があります。


Steamworks



概要



Steamworksは、ゲームをSteamストアと統合するためのApiです。 これにより、ゲームのマルチプレイヤー、実績、リーダーボード、購入、クラウド内のユーザーデータのストレージに実装できます。



統合



Steamworks SDKはC ++で記述されています。 したがって、Unity3Dへの統合にはラッパーが必要です。 いくつかのオプションのうち、Steamworks.NETをランダムに選択しました。 原則として、ラッパーは悪くなく、すべてがうまく統合されていますが、後で奇妙な動作に気づきました。



各関数は、ほぼこの種類のコールバックを通じて結果を返します。 



protected void OnResult(LobbyEnter_t pCallback, bool failure)


ご覧のとおり、追加のブール型パラメーターが関数に渡され、失敗した操作が報告されます。 結局のところ、このパラメーターは決してtrueを返しません。 明らかに何か間違ったことをしたとしても。 これがSteamworks SDKなのかSteamworks.NETなのかわかりません。 開発者から明確な答えが得られず、松葉杖の助けを借りて外に出なければなりませんでした。



実装



上で書いたように、Steamworksにはより柔軟なマッチメイキング機能があります。 できることのリストの例を次に示します。



  • 部屋を作成します（ここでは「ロビー」と呼びます）
  • いくつかのパラメーターに部屋を割り当てます（たとえば、部屋の名前と、どのマップでプレイするかを指定します）
  • 1つ以上のパラメーターで部屋のリストをフィルターする
  • 部屋のプレイヤーにパラメーターを割り当てる
  • 部屋の中にメッセージを送る


Steamバージョンではこの機能をすべて使用しました。







スクリーンショットからわかるように、試合開始前にチャットルームでチャットしたり、レースを選択したり、マップや試合のその他のパラメーターを選択したりできます。



すべてのプレイヤーが平等というわけではありません。 ルームを作成したプレイヤーはその所有者です。 所有者が部屋を去った場合、このタイトルは部屋の別のプレイヤーにランダムに転送されます。 このプレーヤーは、物議を醸す状況でアービターとして行動できます。



理論的には、ゲームのすべての機能を部屋の中に実装して、ゲームの状態をその部屋に保存できます。 しかし、それはすべて倒錯のスナックです。



Steamにはプレイヤー間の接続を作成する特別な機能はありません。 プレイヤーにメッセージを送信するだけです。 接続が以前に確立されていない場合は、自動的に確立されます（相手側は、あなたと通信したいことを確認する必要があります）。 接続が突然切断された場合、Steamは次にメッセージを送信したときに接続の復元を試みます。



Game Centerのように、ゲームオーバーのようなものはありません。 つまり ゲームが終了したと見なされた場合、プレイヤーはお互いに同意する必要があります。



インセンティブでは、まだこのような興味深い機能があります。 マルチプレイヤーをSteam経由ではなく、サーバー経由で行うとしましょう。 サーバーをSteamに登録できます。 プレーヤーは、登録されているすべてのサーバーのリストを取得できます。 したがって、Steamは外部サーバーのディレクトリとして機能します。



問題



1.プレイヤーの1人の切断の問題。

残念ながら、Game Centerのように、サーバーへの接続が自分または相手から消えたかどうかを簡単に判断することはできません。 松葉杖を書かなければなりません。



長所



  • より柔軟なマッチメイキング機能。


短所



  • 奇妙なSteamworks.NETエラー動作。
  • プレーヤーの接続解除に関する問題。


ゲームパーク



概要



GameSparksは、ゲームのバックエンドサーバーとして機能するサービスです。 データの保存、スクリプトの実行、マッチメイキング、アチーブメント、リーダーボードなどのための組み込みメカニズムを備えています。



統合



GameSparksは、公式プラグインを使用してUnity3Dに統合します。 すべて正常に動作します。



実装



すべてのスクリプトはJavaScriptで記述されています。 したがって、私が最初にしたことは、通常の強く型付けされた言語で書くための環境をセットアップすることでした。 TypeScriptを使用しました。



サーバーとの対話は非常に簡単です。 サーバーにメッセージを送信します。 このメッセージに対応するスクリプトがサーバーでトリガーされます。 スクリプトはデータベースから情報を読み取り、変更を加え、結果を返します。 また、スクリプトは他のユーザーにメッセージを送信できます。 すべてが非常にシンプルで論理的です。



原則として、GameSparksでは、任意の複雑なマッチメイキングを実装できます。 たとえば、インセンティブとして部屋を作ることができます。 しかし、気にするのではなく、組み込みのマッチメイキングメカニズムを使用することにしました。



マッチメイキングを開始するときは、プレーヤーの強さ（数値パラメーター）を転送する必要があります。 その後、GameSparksは、特定の値（絶対値または相対値）で送信されたものとは強度が異なる相手を返します。



リアルタイムのマルチプレイヤーでは、別々の部屋を作成することができます。 この部屋では、サーバー側で絶えず回転するスクリプトを実行できます。 しかし、私はこの機能を使用しませんでした。 プレーヤー間でメッセージを転送するために、適切なプレーヤーにメッセージを単純に中継する通常のスクリプトを使用しました。



長所



  • サーバーにプレイヤーデータを保存することが可能です
  • Game CenterとSteamで使用されていた松葉杖を取り除くことができました。 サーバーは常に誰がオンラインで、誰がオンラインではないかを常に把握しており、いつゲームを終了したかを常に確実に判断できます。


短所



  • 少し面倒なドキュメント。
  • 彼らは、インディー開発者にとっては無料でゲームを実行できると約束しました。 実際、私は支払わなければなりませんでした。

左側にはあるプレイヤーのログ、右側には別のプレイヤーのログがあります。 矢印は、メッセージが送信された時点、到着したメッセージを示します。
