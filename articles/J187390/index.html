2 SOAPとは何ですか？



一般的に、最初はSOAPとは何かを書くつもりはなかったので、必要な仕様を備えたw3.orgウェブサイトへのリンクとWikipediaへのリンクに限定したかった。 しかし、最後に私はこのプロトコルについて短いメモを書くことにしました。



そして、このデータ交換プロトコルは、いわゆるRPC（リモートプロシージャコール）パラダイムに基づくプロトコルのサブセットを参照するという事実から話を始めます。そのパラポッドはREST（Representational State Transfer、代表状態転送）です。 これについての詳細はウィキペディアで読むことができます。記事へのリンクはトピックの最後にあります。 これらの記事から、次のことを明確にする必要があります。「RPCアプローチを使用すると、多数のメソッドと複雑なプロトコルで少量のネットワークリソースを使用できます。 RESTアプローチでは、メソッドの数とプロトコルの複雑さが厳密に制限されているため、個々のリソースの数が多くなる可能性があります。 つまり、私たちに関しては、RPCアプローチの場合、サイトにはサービスへの入力（リンク）が常に1つあり、RESTアプローチでは、データと一緒に着信データを処理するために呼び出す手順が常に存在するということです。サイトには多くの入力（リンク）があり、それぞれが特定のデータのみを受信して​​処理します。 読者の一人が、これらのアプローチの違いをより簡単に説明する方法を知っているなら、コメントを書いてください！



SOAPについて次に学ぶ必要があるのは、このプロトコルがトランスポートと同じXMLを使用することです。 すぐに、このマークアップ言語に基づいた技術のスタックのすべての力が私たちの武器、つまりXML-Schema-XMLドキュメントの構造を記述するための言語になります（Wikipediaのおかげ！）、クライアントからサーバーに来るデータの自動検証を可能にします。



したがって、SOAPはリモートプロシージャコールの実装に使用されるプロトコルであり、XMLをトランスポートとして使用することがわかりました。 Wikipediaの記事を読むと、そこから、HTTPとのペアリングだけでなく、アプリケーションレベルのプロトコルで使用できることもわかります（残念ながら、このトピックではHTTP経由のSOAPのみを検討します）。 そして、あなたは私がこのすべてについて最も好きなものを知っていますか？ 推測がない場合は、ヒントを表示します-SOAP！...推測が表示されませんでしたか？... Wikipediaの記事を間違いなく読みましたか？...一般的に、これ以上迷惑をかけません。 したがって、私はすぐに答えに行きます：「SOAP（英語から。Simple Object Access Protocol-オブジェクトにアクセスするための単純なプロトコル 。 仕様1.2まで ）。」 この行で最も注目すべきは斜体です！ このすべてからあなたがどのような結論を出したのかわかりませんが、次のように見えます-このプロトコルは決して「シンプル」と呼ぶことはできないため（そして明らかにw3もそれに同意します）、バージョン1.2から一般的に何らかの方法で解読を停止しました！ そして、それはSOAPと呼ばれ、単にSOAPと呼ばれました。



さて、大丈夫、すみません、少し横に滑りました。 前に書いたように、XMLはトランスポートとして使用され、クライアントとサーバー間で実行されるパケットはSOAPエンベロープと呼ばれます。 エンベロープの一般化された構造を考慮すると、非常に馴染みがあるように思えます。 HTMLページの構造に似ています。 メインセクション-Envelopがあり 、これにはHeaderセクションとBodyセクション、またはFaultセクションが含まれます。 データはボディに転送され、エンベロープの必須セクションですが、 ヘッダーはオプションです。 Headerでは、許可、またはWebサービスプロシージャの入力データに直接関連しないその他のデータを転送できます。 エラーについて話をする特別なことはありませんが、エラーが発生した場合にサーバーからクライアントに通知される点が異なります。





これで、SOAPプロトコルの概要が終了し（クライアントとサーバーが相互に実行する方法を最終的に学習するときに、エンベロープ自体とその構造をさらに詳しく調べます）、 WSDL （Webサービス記述言語）と呼ばれるSOAPコンパニオンについて。 はい、はい、これはまさにこのプロトコルでAPIを取得して実装しようとする試みから私たちのほとんどを怖がらせるものです。 その結果、通常、JSONをトランスポートとして使用して自転車を発明します。 それでは、WSDLとは何ですか？ WSDLは、WikipediaのXML（c）言語に基づいたWebサービス記述およびアクセス言語です。 この定義から、このテクノロジーの神聖な意味全体を理解していない場合は、自分の言葉で説明しようと思います！



WSDLは、クライアントがサーバーと正常に通信できるように設計されています。 これを行うために、拡張子が* .wsdlのファイルに次の情報が記述されています。

  • 使用された名前空間、
  • 使用されたデータスキーム、
  • Webサービスが顧客に期待するメッセージの種類は、
  • どのデータがどのWebサービスプロシージャに属しているか、
  • Webサービスにはどのような手順が含まれていますか？
  • クライアントはどのようにWebサービスプロシージャを呼び出す必要がありますか。
  • クライアント呼び出しの送信先アドレス。


ご覧のとおり、このファイルはWebサービス全体です。 クライアントでWSDLファイルのアドレスを指定すると、Webサービスに関するすべてがわかります。 その結果、Webサービス自体の場所について絶対に知る必要はありません。 WSDLファイルの場所アドレスを知るだけで十分です！ SOAPは、ロシアのことわざ（c）で描かれているほどひどくないことがすぐにわかります。



3 XMLスキーマの概要



これで、SOAPが何であるか、SOAPが何であるかについて多くのことを理解し、SOAPがそれを取り巻くテクノロジースタックの概要を把握できました。 まず、SOAPはクライアントとサーバー間のやり取りの方法であり、XMLマークアップ言語がそのトランスポートとして使用されるため、このセクションでは、XMLスキームを介した自動データ検証がどのように発生するかを少し見ていきます。



スキームの主なタスクは、処理するデータの構造を記述することです。 XMLスキーマのすべてのデータは、 単純 （スカラー）型と複雑 （構造）型に分けられます。 単純型には次のような型が含まれます。

  • ひも
  • 数
  • ブール値
  • 日付


内部に拡張機能がない非常に単純なもの。 それらの対掌体は複雑な複合型です。 誰もが思い浮かぶ複雑なタイプの最も単純な例はオブジェクトです。 たとえば、本。 この本は、 著者 、 タイトル 、 価格 、 ISBN番号などのプロパティで構成されています。 そして、これらのプロパティは、単純なタイプでも複雑なタイプでもかまいません。 そして、XMLスキーマのタスクはそれを記述することです。



SMSメッセージ用にXMLスキーマを作成することをお勧めします。 以下は、SMSメッセージのxmlの説明です。 



<message> <phone>71239876543</phone> <text> </text> <date>2013-07-20T12:00:00</date> <type>12</type> </message>


複雑なタイプ図は次のようになります。 



 <element name="message" type="Message" /> <complexType name="Message"> <sequence> <element name="phone" type="string" /> <element name="text" type="string" /> <element name="date" type="dateTime" /> <element name="type" type="decimal" /> </sequence> </complexType>


このエントリは、次のように読み取られます。「 Message 」タイプの変数「 message 」と、「 Message 」という複合タイプがあり、「 string 」タイプの要素「 phone 」、「 string 」タイプの「 text 」、「 dateTime 」タイプの「 date 」 、タイプ10進数の " タイプ "。 これらのタイプは単純で、スキーマの説明ですでに定義されています。 おめでとうございます！ 最初のXMLスキーマを作成しました！



要素「 element 」と「 complexType 」の意味は多かれ少なかれ明確になったと思うので、それらに焦点を合わせず、すぐに要素コンポーザー「 sequence 」に切り替えます。 composer要素「 sequence 」を使用する場合、それに含まれる要素は常に図に示されている順序で配置する必要があり、それらはすべて必須です。 しかし、絶望しないでください！ XMLスキーマには、さらに2つのcomposer要素があります： choiceとallです。 作曲家「 choice 」は、リストされている要素のいずれか1つが存在する必要があることと、作曲家「 all 」-リストされた要素の任意の組み合わせを報告します。



覚えているように、トピックの最初のセクションでは、パケットでsmsメッセージを1から無限に送信できることに同意しました。 したがって、このようなデータがXMLスキーマでどのように宣言されているかを理解することを提案します。 全体的なパッケージ構造は次のようになります。 



 <messageList> <message> <phone>71239876543</phone> <text>  1</text> <date>2013-07-20T12:00:00</date> <type>12</type> </message> <!-- ... --> <message> <phone>71239876543</phone> <text>  N</text> <date>2013-07-20T12:00:00</date> <type>12</type> </message> </messageList>


この複合型のスキーマは次のようになります。 



 <complexType name="Message"> <sequence> <element name="phone" type="string" minOccurs="1" maxOccurs="1" /> <element name="text" type="string" minOccurs="1" maxOccurs="1" /> <element name="date" type="dateTime" minOccurs="1" maxOccurs="1" /> <element name="type" type="decimal" minOccurs="1" maxOccurs="1" /> </sequence> </complexType> <element name="messageList" type="MessageList" /> <complexType name="MessageList"> <sequence> <element minOccurs="1" maxOccurs="unbounded" name="message" type="Message"/> </sequence> </complexType>


最初のブロックには、複合型「 Message 」のよく知られた宣言が含まれています。 お気づきの方は、「 メッセージ 」に含まれる各単純型に、新しい修飾属性「 minOccurs 」と「 maxOccurs 」が追加されました。 名前から推測するのは難しいことではないため、最初（ minOccurs ）は、このシーケンスに、タイプ「 phone 」、「 text 」、「 date 」、および「 type 」の少なくとも1つの要素があることを報告しますが、次（ maxOccurs ）この属性は、シーケンスにそのような要素が最大で1つあることを宣言します。 その結果、任意のデータのスキームを記述するとき、それらをセットアップするための最も幅広い選択肢が与えられます！



回路の2番目のブロックは、タイプMessageListの messageList要素を宣言します 。 MessageListは少なくとも1つのメッセージ要素を含む複合型ですが、そのような要素の最大数は無制限です！



これについては、スキームに従ってLikBezが完了し、別のエキサイティングな冒険が待っていると仮定します。独自のWSDLを作成します。



4 WSDLを作成します



WSDLがWebサービスであることを覚えていますか？ 覚えておいてください！ これを作成すると、小さなWebサービスが浮かび上がります。 したがって、ハッキングしないことを提案します。



一般に、すべてが正しく機能するためには、正しいMIMEタイプのWSDLファイルをクライアントに転送する必要があります。 これを行うには、それに応じてWebサーバーを構成する必要があります。つまり、拡張子が* .wsdlのファイルのMIMEタイプを次の行に設定する必要があります。 



 application/wsdl+xml


しかし実際には、PHPを介して「 text / xml 」HTTPヘッダーを通常送信しました。 



 header("Content-Type: text/xml; charset=utf-8");


そしてすべてがうまくいきました！



すぐに警告したいのですが、シンプルなWebサービスの説明は非常に印象的です。怖がらないでください。 テキストの大部分は水を必要とするものであり、一度書けば、あるWebサービスから別のWebサービスに常にコピーできます。



WSDLはXMLであるため、最初の行でそれについて直接記述する必要があります。 ファイルのルート要素は、常に「 定義 」と呼ばれる必要があります。 



 <?xml version="1.0" encoding="utf-8"?> <definitions> </definitions>


通常、WSDLは4〜5個のメインブロックで構成されます。 最初のブロックは、Webサービスの定義、つまりエントリポイントです。 



 <?xml version="1.0" encoding="utf-8"?> <definitions> <!—  --> <service name="SmsService"> <port name="SmsServicePort" binding="tns:SmsServiceBinding"> <soap:address location="http://localhost:80/smsservice.php" /> </port> </service> </definitions>


ここでは、「 SmsService 」というサービスがあることを示しています。 原則として、WSDLファイル内のすべての名前は、任意の名前に変更できます。 それらはまったく役割を果たしません。



その後、 SmsService Webサービスに「 SmsServicePort 」というエントリポイント（「ポート」）があることを発表します。 このエントリポイントで、クライアントからサーバーへのすべての要求が送信されます。 そして、「 address 」要素で、リクエストを受信するファイルハンドラへのリンクを示します。



Webサービスを決定し、そのエントリポイントを指定したら、サポートされている手順をそれにバインドする必要があります。 



 <?xml version="1.0" encoding="utf-8"?> <definitions> <!—  - --> <binding name="SmsServiceBinding" type="tns:SmsServicePortType"> <soap:binding style=”rpc” transport="http://schemas.xmlsoap.org/soap/http" /> <operation name="sendSms"> <soap:operation soapAction="" /> <input> <soap:body use="literal" /> </input> <output> <soap:body use="literal" /> </output> </operation> </binding> <!—  --> <service name="SmsService"> <port name="SmsServicePort" binding="tns:SmsServiceBinding"> <soap:address location="http://localhost:80/smsservice.php" /> </port> </service> </definitions>


これを行うために、どの操作がどの形式で呼び出されるかをリストします。 つまり SmsServicePortポートの場合、バインディングはSmsServiceBindingという名前で定義されます。この名前はrpc呼び出しタイプを持ち、トランスポート（トランスポート）プロトコルとしてHTTPを使用します。 したがって、ここではHTTP経由でRPC呼び出しを行うことを示しました。 その後、Webサービスでサポートされる手順（ operation ）について説明します。 「 sendSms 」という1つの手順のみをサポートします。 この手順により、すばらしいメッセージがサーバーに送信されます！ 手順が発表された後、データを送信する形式を指定する必要があります。 この場合、標準のSOAPエンベロープが使用されることが示されています。



その後、プロシージャをメッセージにバインドする必要があります。 



 <?xml version="1.0" encoding="utf-8"?> <definitions> <!—     --> <portType name="SmsServicePortType"> <operation name="sendSms"> <input message="tns:sendSmsRequest" /> <output message="tns:sendSmsResponse" /> </operation> </portType> <!—  - --> <binding name="SmsServiceBinding" type="tns:SmsServicePortType"> <soap:binding style=”rpc” transport="http://schemas.xmlsoap.org/soap/http" /> <operation name="sendSms"> <soap:operation soapAction="" /> <input> <soap:body use="literal" /> </input> <output> <soap:body use="literal" /> </output> </operation> </binding> <!—   --> <service name="SmsService"> <port name="SmsServicePort" binding="tns:SmsServiceBinding"> <soap:address location="http://localhost:80/smsservice.php" /> </port> </service> </definitions>


これを行うには、バインディングのタイプが「 SmsServicePortType 」であることを示し、同じタイプ名の「 portType 」要素で、プロシージャをメッセージにバインドすることを示します。 そのため、着信メッセージ（クライアントからサーバーへ）は「 sendSmsRequest 」と呼ばれ、発信メッセージ（サーバーからクライアントへ）は「 sendSmsResponse 」と呼ばれます。 WSDLのすべての名前と同様に、着信メッセージと発信メッセージの名前は任意です。



ここで、メッセージ自体、つまり 着信および発信： 



 <?xml version="1.0" encoding="utf-8"?> <definitions> <!--   sendSms --> <message name="sendSmsRequest"> <part name="Request" element="tns:Request" /> </message> <message name="sendSmsResponse"> <part name="Response" element="tns:Response" /> </message> <!--     --> <portType name="SmsServicePortType"> <operation name="sendSms"> <input message="tns:sendSmsRequest" /> <output message="tns:sendSmsResponse" /> </operation> </portType> <!--   - --> <binding name="SmsServiceBinding" type="tns:SmsServicePortType"> <soap:binding style=”rpc” transport="http://schemas.xmlsoap.org/soap/http" /> <operation name="sendSms"> <soap:operation soapAction="" /> <input> <soap:body use="literal" /> </input> <output> <soap:body use="literal" /> </output> </operation> </binding> <!--   --> <service name="SmsService"> <port name="SmsServicePort" binding="tns:SmsServiceBinding"> <soap:address location="http://localhost:80/smsservice.php" /> </port> </service> </definitions>


これを行うには、それぞれsendSmsRequestおよびsendSmsResponseという名前のメッセージ要素を追加します 。 それらにおいて、エンベロープは入力で受信されるべきであり、その構造は「 リクエスト 」データ型に対応することを示します。 その後、データタイプ「 Response 」を含むエンベロープがサーバーから返されます。
