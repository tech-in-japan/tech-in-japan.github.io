詳細
ファームウェアの開発に伴い、いくつかのVCP制御コマンドが導入されました。 これは、USBデバイスをOSに接続した後、仮想COMポートが作成されたことを意味します。 これを使用して、制御および診断コマンドがユーザー端末から送信されました。 応答として、デバイスは実行ステータスとその現在のステータスを受け取りました。



システムは、サービスの観点から非常に典型的です。シリアルポートと、管理および診断用のコマンドセットがあります。



すべてが短時間で変化しました。 客観的な理由から、必要なチームのセットが増えました。 出力の相互作用も必要でした。ダイナミクスで表示するためにいくつかのパラメーターが必要になりました。 たとえば、強磁性体が通過するときの磁気センサーの読み取り値として。 このため、追加のコマンドが導入されました。これらのコマンドは、 制御シーケンスで動作し、ユーザー端末に高頻度で情報を出力しました。 これにより、必要なリアルタイム観察の感覚が生まれました。 インタラクティブなチームはエンジニアにとって非常に便利だったため、一部のチームは後にコンセプトに従って追加されました。 そして、亀裂がありました。 インタラクティブ、診断、制御コマンドなど、いくつかのグループのコマンドを一度にサポートする必要がありました。 同時に、コードの定期的なリファクタリングは、処理された多数のコマンドの時間のかかる編集に関連付けられていました。 チームのユーザーグループがまだ存在する必要があることが明らかになりました-使用のための指示に従うだけの資格のない人のために。 彼らにとって、ボタンとフラグを備えたクライアント端末を書くというアイデアが生まれました...そして疑問が生じました：機能性にあまり注意を払わずにサービス部分をやっていることが明らかになりました！ ただし、クライアントコンピューターで実行されるユーザープログラムには、クロスプラットフォームおよびLTS（サポート期間）という独自の要件も必要です。

そこで、デバイスを制御するためのWEBインターフェイスを作成するというアイデアが生まれました。 サードパーティのソフトウェアを開発する必要はありません。コントロールページを表示するためのブラウザはすべての必要なOSにあります。 インターフェイス設計の可能性は膨大です。 http / html / js標準に関するサポート期間も疑いの余地がありません。



計画によると、経営陣は次のように機能するはずでした。

1. USBデバイスはネットワークカードで表されます

2.クライアントコンピューター（PCまたはガジェット）は、デバイスのネットワークで作業するためのIPアドレスを受け取ります

これは、考えられる多数のソリューションの選択肢の1つにすぎないことを理解する必要があります。 長所と短所があります。 主に純粋にネットワークデバイスのメーカーがWebインターフェイスの使用に頼るようになったので、モデムとルーターをセットアップしました。 この記事のメッセージ-本当に便利なものを適用しましょう。 そして、途中の困難を恐れないでください。今それらを克服することは、将来的に私たちに多くの時間を節約します！

手順1. USBデバイスを接続します。

前述のように、この時点で、デバイスはホストに「私はネットワークカードです！」と伝えます。

ホスト（クライアントコンピューター）は、クラフトを接続した後、リクエストの送信を開始します。

ホストには次の情報が必要です
-製品とは何ですか

-製品のVIDとPIDは何ですか（製造業者と製品の識別子、 リストを参照 ）

-デバイスが属するクラスとサブクラス

-交換するエンドポイントとブロック

さて、そしていくつかの他の情報。 この場合、構成パケットはエンドポイント0で送信されます。それ自体に関する情報を含むデバイスからの応答パケットは、通常「USBデバイス記述子」と呼ばれます。

ステップ3. IPアドレスの取得

それで、なぜ動的アドレスを取得するためのサービスが必要なのでしょうか。 このサービスはDHCP （動的ホスト構成プロトコル）と呼ばれます 。

ホストがデバイスを初期化すると、ネットワークインターフェイスが作成されます。



ネットワークインターフェイス（誰も知らない場合）
ネットワークインターフェイスは、物理ネットワークまたは仮想ネットワーク上のリソースへのアクセスを提供するソフトウェアエンティティです。

ほとんどの場合、ホストの各ネットワークインターフェイスは特定のネットワークアダプターに対応しています。 しかし、ローカルループや仮想マシンとのやり取りに役立つインターフェイスなど、他にも多くのインターフェイスがあります。 彼らの場合、信号形式では、ホストからは何も出ません-交換はプログラムで実行されます。



各ホストネットワークインターフェイスには、少なくとも1つのIPアドレスが関連付けられている必要があります。 それによると、「ネットワークの居住者」はホストに連絡できます。



複数のネットワークが「ワイヤ上」でアドレス指定されている場合（たとえば、IPアドレス10.4.1.xxおよび192.168.1.xxのデバイス）、2つの「個人用」IPアドレスをインターフェイスに割り当てることができます。 これらは、10.4.1.151および192.168.1.200のようになります。 Windowsでネットワークインターフェースのセットと関連するIPアドレスを確認するには、Linuxでipconfigコマンドとifconfigを使用します。



マスクは、ネットワーク/サブネットを記述するために使用されます。 たとえば、ネットワーク10.4.1.xxの正しい説明は、ネットワーク10.4.1.0、マスク255.255.255.0です。 または、マスクの4バイト数をバイナリ形式で表し、先行ユニットの数をカウントすると、値24が得られます。ネットワークは次のように記述できます：10.4.1.0/24。



関連するソースでこれについて詳しく読むことができます 。



IPアドレスをインターフェイスに割り当てるには、主に2つの戦略があります。静的メソッド（ユーザーがインターフェイスのアドレスを指定する場合）と動的メソッド（DHCPサービスを使用）です。



後者は、ホストでインターフェイスを作成するときに、DHCPクライアントサービスがアクティブになることです。 DHCPサーバーがネットワークに存在することを期待して、 UDPプロトコルを介してネットワーク（構成はまだ不明）にブロードキャストパケットを送信し始めます。







DHCPサーバーの一般的な機能、特にコントローラーの機能は、クライアントに応答することです。 応答として、コントローラーは「言います」：クライアント、あなたはそのようなネットワークにいて、そのようなIPアドレスを保持し、そのようなアドレスを持つDNSサーバーも持っています。



その後、ホストははるかに「感じ」ます。指定されたIPアドレスをインターフェイスに割り当て、DNSサーバーのIPアドレスを記憶します。



初期化が終了しました。ホストブラウザーでページ名（run.stm）を入力できるようになりました。



LRNDISライブラリの動作はカスタマイズ可能であると言わなければなりません。 DHCPサーバーサービスは、アセンブリから除外できます。 その後、ホストは192.168.7。（2-254）の範囲に属するアドレスを登録する必要があります。 このようなネットワークはデフォルトで作成されます。 パラメーター（192.168.7.0/24）も構成可能です。 この例では、クライアントは192.168.7.2 ... 192.168.7.4の範囲のアドレスを24時間のリース時間で受け取ります。

ライブラリのセットアップの詳細については、 前の記事を参照してください 。



ステップ4.ページをロードする

ページをダウンロードするには、ユーザーはデバイス192.168.7.1のアドレスを直接入力できます。

ただし、数字を覚えておく必要はありません。 DHCPサーバーに加えて、DNSサーバーをサポートするライブラリを構築することができます。その機能はネットワーク名を解決することです。 公開された例では、DNSサーバーはリソース名「run.stm」を解決するようにトレーニングされています。



したがって、ブラウザでrun.stmを記述すると、ホストネットワークサービスはDNSサーバーにリクエストを送信し、それに応答して、サーバーが有用に通知します：「run.stm」ドメイン名はIPアドレス192.168.7.1に対応します。 次に、既知のアドレスにあるホストブラウザーは、 HTTP要求を送信してルートページを取得するためにTCP接続を確立します。

履歴から、ルートHTMLドキュメントを読み込んだ後、ブラウザーはコントローラーから他の2つのファイル、discovery.svgとzepto.min.jsもダウンロードすることがわかります。 1つ目は、ディスカバリーボードの画像です。 SVG形式が選択されているのは、ベクターグラフィックスのイメージであるため、マイクロコントローラーのROMのスペースをほとんど占有しないためです。 スクリプトファイルzepto.min.jsが含まれています。 は有名なJQUERYの単純化されたアナログです。 ROMのスペースの細心の節約は行われていないことに注意してください。 すべての静的リソースに対して35 KBの犠牲がありますが、コントローラーのメモリはまだ十分です。 さらに、インターフェイスの複雑さがさらに増すこのサイズは、著しく遅くなることを約束します。 インターフェースが大幅に成長した場合-常に静的リソースを圧縮形式で保存および提供する方法があります-すべての既知のブラウザーは現在、その場で解凍をサポートします。



ブラウザが送信する別のリクエストは、/ state.cgiリクエストです。 これは、1秒あたり5回の頻度で、ルートHTMLドキュメントからのスクリプトによって形成されます。 デバイスの現在の状態のダイナミクスを取得する要求が必要です。

この要求を受信すると、コントローラーはJSON形式で次の行を生成して応答します。 



{ "systick": 9528746, "button": 0, "acc": [54, -288, 936], "leds": { "g": 0, "o": 0, "r": 0 } }


これには、デバイスの現在の状態に関するすべてのデータが含まれ、JavaScriptコードを使用してページに表示されます。



さて、おそらくブラウザと通信する最後の瞬間は、ブラウザを制御する方法です。



この例では、3つのLEDが制御されています。 たとえば、JavaScriptを使用して赤色のLEDフラグをクリックすると、「r」パラメーターが渡され、値が0または1のHTTP GETリクエストが送信されます。完全なリクエストは次のようになります。/ctl.cgi?r=1



何らかの方法で、論理状態0/1、テキストフィールドの値、ボタンクリックの通知など、データセットを転送できます。 このアプローチの利点は、厳密に形式化されたコントロールメッセージを受け取るため、プログラムロジックがコントロールをまったく認識しない可能性があることにあります。 インターフェイスパーツ（HTML + JS）をすべての設備でローカルに変更およびデバッグし、ファームウェアの一部としてコントローラーに一度アップロードすることもできます。 重要ではないローカルWeb開発は、適切な専門家が処理できます。



LWIPスタックについて

ライブラリに組み込まれたこのネットワークスタック用でない場合、ネットワーク交換は手配できません。

ライブラリはベアメタル（OSおよび動的メモリ割り当てなし）で動作するため、ソケットアドインは使用できません。 したがって、未加工のAPIスタックを使用してネットワークアプリケーションを作成します。 幸いなことに、このトピックに関する多くの情報がネット上にあります。



また、パッケージの貢献には、多くの既製のソリューションがあります。 そこから、HTTPサーバーが使用されました。



前の記事で、スタックとその設定について簡単に説明しました。 現時点では、ファイル内のスタックにとって重要な一連の定義が洗練されています。
