QMLエンジンの内部。 パート2：バインディング

この投稿は、 この （ 翻訳された ）記事の続きです。



前の投稿で、QMLエンジンがファイルをロードする方法を見てきました。 最初にQMLファイルが解析され、次に中間バイトコードにコンパイルされ、最後にバイトコード命令に従って、各QMLファイルの各イベントに対してC ++オブジェクトが作成されることを思い出してください。 たとえば、QMLファイルにText要素が含まれている場合、エンジンはQQuickTextクラスのC ++インスタンスを作成することがわかりました。



実際、ファイルのダウンロードはQMLエンジンが行うほとんどすべてのことです。 ダウンロード後、アプリケーションプロセスへの干渉を停止します。 ランタイムのイベント処理およびレンダリング要素は、完全にC ++に依存します。 たとえば、TextInput要素は次のようなイベントを処理します QQuickTextInput :: keyPressEvent（）およびQQuickTextInput :: updatePaintNode （）を更新します。QMLエンジンは関与しません。



ただし、実行時にQMLエンジンが影響を与える2つの重要なことがあります。 バインドされたシグナルハンドラーとプロパティバインディングの更新です。 関連するシグナルハンドラは、 MouseAreaのonClickedハンドラのようなものです 。 今日はバインディングを見ていきます。



例： 

import QtQuick 2.0 Rectangle { width: 300 height: 300 color: "lightsteelblue" Text { anchors.centerIn: parent text: "Window Area: " + (parent.width * parent.height) } }


この例には、2種類のプロパティの割り当てが含まれています。

  1. C ++クラスQQuickRectangleのRectangle要素のwidthプロパティに値300を割り当てるなどの単純なプロパティ割り当てこの場合、VMEはこのコンポーネントを作成するときにSTORE_DOUBLEバイトコード命令を実行するだけです。 VMEは関数QMetaObject :: metacall（QMetaObject :: WriteProperty、...）を呼び出すだけで、最終的にはQQuickRectangle :: SetWidth（）の呼び出しで終了します。 この割り当ての後。 QMLエンジンはこの割り当てを忘れています。
  2. 上記の例のように、 テキストプロパティへの「ウィンドウ領域：」+（parent.width * parent.height）またはcenterInプロパティへの親のバインディング割り当て。 バインディングの魔法のおかげで、Rectangle要素の高さまたは幅が変更されるたびに、 テキストプロパティが自動的に更新されます。 どのように機能しますか？ 魔法はまったくありません。答えを見つけるために読み進めてください。


バインディングを作成



パラメーターQML_COMPILER_DUMP = 1を設定すると、バインダーがSTORE_COMPILED_BINDINGステートメントによって作成されていることがわかります。 



 ... 9 STORE_COMPILED_BINDING 43 1 0 10 FETCH 19 11 STORE_COMPILED_BINDING 17 0 1 ...




コンパイル済みバインディング。これは実際には最適化されたオプションです。 STORE_BINDINGステートメントによって作成された通常のバインディングを見てみましょう。 QQmlVME :: run（）関数は、 QQmlBindingオブジェクトを作成し、フォームの文字列を渡します 
 function $text() { return “Window Area: ” + (parent.width * parent.height) }
これはJS式です。



そうです、すべてのバインディングはJavaScript関数です。 関数「function $ text（）」の一部はQMLコンパイラーによって追加されます。そのため、QMLのJMLエンジンであるV8は、ご想像のとおり、JSコードのみを実行できます。 文字列関数は、v8コンパイラーによってv8 :: Functionオブジェクトにコンパイルされます。 v8エンジンは、組み込みのJITコンパイラーにより、ネイティブコードを作成します。 作成されたv8 :: Functionオブジェクトは実行しませんが、将来のために保存します。



合計：STORE_BINDINGステートメントを実行すると、QQmlBindingオブジェクトが作成されます。QQmlBindingオブジェクトは、渡された関数を文字列としてv8 :: Functionオブジェクトにコンパイルし、組み込みのV8 JSエンジンを使用します。



バインド

バインディング更新



さて、テキストプロパティが元の意味をどのように取得したかがわかりました。 更新はどうですか？ QMLエンジンは、親長方形の高さまたは幅が変更されたときにこのプロパティを更新する必要があることをどのように認識しますか？



この質問に対する答えは、同じオブジェクトラッパーにあります。これは、覚えているように、V8がプロパティにアクセスする必要があるときに呼び出されます。 ラッパーは、プロパティの値を返すだけではなく、アクセスを要求されたすべてのプロパティを記録します。 基本的に、プロパティへのアクセスが発生すると、ラッパーは現在実行中のスナップキャプチャ機能を呼び出します。 この例では、これはQQmlJavaScriptExpression :: GuardCapture :: captureProperty（）です （QQmlBindingはQQmlJavaScriptExpressionクラスのサブクラスです）。



キャプチャー関数では、バインディングは単に、要求されたプロパティのNOTIFYタイプの信号に単純に添付されます。 これで、NOTIFYシグナルが呼び出されると、それに接続されているバインディングスロットが呼び出され、バインディング自体が再起動されます。 NOTIFYシグナルを聞いたことがない場合、心配しないでください、そのロジックは単純です：Q_PROPERTYマクロを使用してプロパティが定義されると、この場所にNOTIFYシグナルが常に存在し、このプロパティが変更されるたびに送信されます。



たとえば、QQuickItemクラスのwidthプロパティの宣言は次のとおりです。 

 Q_PROPERTY(qreal width READ width WRITE setWidth NOTIFY widthChanged)


このシナリオでは、バインドの最初の開始時に初めてwidthプロパティにアクセスすると、プロパティキャプチャ関数はwidthChanged（）信号とバインドの起動スロットを接続します。 これで、アプリケーションの実行中にQQuickItemオブジェクトがwidthChanged（）シグナルを発行すると、それに関連付けられているすべてのバインディングが呼び出され、再起動されます。



自己定義プロパティにNOTIFYシグナルを設定し、プロパティが変更されるたびに送信することが非常に重要です。 これを行うのを忘れると、バインディングは再起動されないため、プロパティのバインディングは正しく機能しません。 一方、プロパティが変更されていないときにNOTIFYシグナルが送信されると、バインディングはアイドル状態になります。



要約すると、プロパティにアクセスすると、ラッパーオブジェクトはバインディングからキャプチャ関数を呼び出します。これにより、このバインディングがプロパティのNOTIFYシグナルにアタッチされ、プロパティが変更されたときにバインディングが再度実行されます...

今日は、QMLでバインディングがどのように機能するかを見ました。 簡単に説明すると、バインディングは、関連するプロパティが変更されるたびに実行されるJS関数です。
