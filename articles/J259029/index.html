Samba4-Pythonスクリプトインターフェイスの使用

Samba4には統合されたPythonインターフェイスがあります。 多くのユーティリティ（たとえば、samba-tool）は、このインターフェイスを使用してPythonで完全に実装されています。



LDAPインターフェースから行われたすべてのことは、Samba 4 Python Scriptingで行うことができます。 利点-ファイルアクセス。これは、LDAPにはない高速でいくつかの機能を意味します。 たとえば、あるデータベースからユーザーパスワードのハッシュを取得して、別のデータベースに転送できます。 また、SID、パスワード、その他すべてを備えたユーザー自身を別のドメインに転送する必要があります（SIDの履歴に問題はありません）。



ドキュメントだけでは十分ではありませんが、 ソースがある場合は<samba-source> / python / sambaディレクトリにあり、それ以外の場合は/usr/lib/python2.7/dist-packages/sambaにあります。



最も興味深いのは、samdb.pyファイルです。これは、ADのほとんどの操作の実装です。



ADドメインコントローラーの構成にSamba4がインストールされているとします。 PythonプログラムからADデータベースに接続してみましょう。 まず、必要なライブラリをインポートします。 



#!/usr/bin/env python # -*- coding: utf-8 -*- import ldb from samba.samdb import SamDB from samba.auth import system_session from samba.ndr import ndr_pack, ndr_unpack from samba.dcerpc import security import samba.param import base64 import binascii

 lp = samba.param.LoadParm() lp.load(samba.param.default_path()) # lp.load("/etc/samba/smb.conf") sam = SamDB(lp=lp,session_info=system_session())

これで、samオブジェクトを使用して、LDAP構文に完全に従ってADデータベースを検索および変更できます。

たとえば、データベース検索（ベースは「CN =ユーザー、DC = myDom、DC = lan」タイプのLDAPツリーノード、式はオプションの選択条件、attrsは必要な属性のリストです）： 



 res = sam.search(base=base, expression=expression, attrs=[*])


ユーザーがOSに横たわることを許可します。 



 base = "OU=myUsers,DC=myDom,DC=lan" #        base = "CN=Users,DC=myDom,DC=lan"


パスワード「secret」でユーザー「tst」を作成します。 SamDBクラスには既製のメソッド-newuser（）がありますが、これも試すことができます。 



 newUsr = "tst" usrPass = "secret" ld = {'dn': 'CN=%s,%s' % (newUsr,base), "sAMAccountName": newUsr, "userPrincipalName": "%s@%s" % (newUsr,"myDom.lan"), "objectClass": "user", "displayName": newUsr, "description": newUsr, "homeDirectory": r"\\%s\users\%s" % ("myHost",newUsr), 'scriptPath': "loginScr.cmd", } sam.transaction_start() try: sam.add(ld) sam.setpassword("(samAccountName=%s)" % ldb.binary_encode(newUsr), usrPass, False) except: sam.transaction_cancel() print '!!!error' else: sam.transaction_commit()


ご覧のとおり、SamDBはトランザクションをサポートしています。

ここで、Win 2003のドメインからSamba4にユーザーを転送する実際の例です。



古いドメインには問題が蓄積されています（間違ったドメイン名で始まっていても）。 DCからSamba4への通常のレプリケーション（反対方向-Samba4 DC-> W2003 DC）は、おそらくドメイン内の問題が原因で開始されませんでした。

タスクはSamba3上のファイルサーバーの存在によって悪化したため、SAM3に既に存在するsAMAccountName <->マッピング（UID、GID）（通常は/var/lib/samba/winbindd_idmap.tdb ）を保存する必要がありました。 実際には、タスクはここで説明したタスクに似ていました 。



すべての実験と最終バージョンは、OpenVZコンテナー（CentOS 6）で実行されているUbuntu 14.04サーバーで行われました

Samba4のインストール、構成は何度も説明されています。 たとえば、すでにここで言及されています 。 rfc2307を使用したスキームでのUnix IDの通常の表示には、 sssdが使用されました。 ちなみに、多くの人がお勧めするsernet Samba4アセンブリは使用しないほうがよい-sssdパッケージと友達になるのは難しい。



ユーザーのパスワードとSIDを保存するには、前述のように、古いADデータベースがプライベートディレクトリSamba4の形で既に存在している必要があります。 詳細（ ここを参照）、「samba-tool domain join samdom.example.com DC -Uadministrator --realm = samdom.example.com」をスキップすると、必要なデータベースが既に作成されているため、sambaサービスを開始せずにそこで停止できます。 。 データベースをさらに更新する必要がある場合は、sambaサービスを開始せずに行うことはできません。



既存のWin ADドメインへの影響は最小限です（ほとんどアイドル状態のコントローラーがもう1つ作成されるため、ログに多くのNTDSレプリケーションエラーが発生します）。スタンドアロンADデータベースを作成した後、リスクなしで仮想環境で実行できます。 MS Winドメインがしばらくの間正常に動作する必要がある場合、この一時的なSamba4を強制終了し、動作中のDCからこのDCに関する情報をクリーンアップすることをお勧めします。



結果のプライベートディレクトリ（通常/ var / lib / samba / privateまたは/ usr / local / samba / private ）は将来のSamba4にどこかにコピーし、/ etc / sambaからsmb.confもコピーする必要があります。 これで、古いドメインに関するすべてのデータが1か所に保存され、ローカルFSでも使用できるようになりました。



Samba3にファイルサーバーがまだある場合、Samba3で開発したidmapを保持したい場合は、Samba3から/ var / lib / sambaディレクトリを置く必要があります（winbindd_idmap.tdbとgroup_mapping.tdbの2つのファイルが必要です）。 。
