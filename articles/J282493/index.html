ストアアンドフォワード通信、UUCP

現在、インターネットの世界では、すべてのサービスの大半がオンラインで機能しています。つまり、インターネットとサーバーへの永続的な接続です。 HTTP、FTP、またはIRCサーバーが利用できない場合、すぐに明示的に通知されます。 永続的なオンラインの贅沢を常に持つことができるとは限りません。 高価な場合もあれば、技術的に不可能な場合もあります。 リソースへのホワイトリストアクセスのみを許可するグレートロシアファイアウォールが表示され、せいぜい場所からしか「本格的な」インターネットが利用できないという危険があります。 送信用のデータが保存され、接続が表示されるまで待機する操作モード。ストアアンドフォワードと呼ばれます。 このモードは、不安定なオンライン状態で作業できるようにするのに便利です。







広く普及しているサービスのうち、電子メール（SMTP）のみがこれに似たものを持っています。メッセージを送信すると、受信サーバーに接続して正常な応答コードを受信できるまで、サーバーに保存されます。 そして、手紙が受取人に届くまでチェーン全体でそうです。 ファイルはメールで転送することもできますが、 Base64エンコードのために非常に非効率的です。 事前にメールサーバーを設定しておくと、実行用のコマンドがある対話型の特別なアドレスにメッセージを送信できます。つまり、インタラクティブな介入と厳しい期限を保証する必要のないコマンドのリモート実行です。



ただし、SMTPサーバーは、ネットワーク上での永続的なプレゼンスをほぼ維持するように設計されています。 リモートサーバーが利用できない場合、転送試行の頻度が減少します。 5分間のネットワーク可用性がある場合、この時点でSMTPサーバーが再試行を試みるという事実ではありません。 原則として、数日中に彼はメッセージがまだ配信されず、すぐに削除されることを思い出させます。



このモードで完全に、 FidoNetネットワークは機能します。 たとえば、人は1日に2、3回上位ノードに接続し、手紙のパケットを受信し、蓄積したものを送信します。 次に、オフラインで読み取り、受信したメッセージに応答します。 FTN（FidoNet Technology Network）テクノロジーを復活させ、再び使用する価値はありますか？ 個人的には、私はそうは思いません。FTNはアマチュアが家庭用コンピューター用に作成したものです。 私は、FTNは、UNIXライクなオペレーティングシステムを持っていない人にとって、「貧しい」人にとってUUCP （UNIX-to-UNIX CoPy）だと言うでしょう。 さらに、すべてのFTNソフトウェアはインターネット技術とは一線を画しています。 UUCPは既存のテクノロジーとうまく統合します。



UUCPは、ストアアンドフォワードサービスを作成する問題を便利に解決できます。 70年代および80年代に非常に人気があり、インターネットが広く使用されるずっと前から、UNIXシステム間の通信の事実上の標準でした。



  • ネットワークに接続されているかオフラインであるかにかかわらず、システムの動作は区別できません。 UUCPを介してファイルまたは電子メールを送信する場合：これは常に成功したリターンコードです。 それでもメールクライアントのドラフトメッセージにメッセージを保存できる場合は、cronデーモンまたはgit-send-emailからのメッセージに同様の機能はありません。 これにより、ソフトウェアが大幅に簡素化されます。
  • すべてのネットワーク参加者には、明確に定義された動作モデルがありません。ポーリングモードでメールまたはファイルを受信できる場合（リモートシステムに時々問い合わせる場合）、明示的なプッシュを行うことができます。 このように、あなたはあなたの操作条件であなたにとってより便利で効率的になります。 追加のプロトコルやソフトウェアサービスを使用せずに、NAT thの背後にある2つのシステムを中間ノードを介して接続できます。
  • 文字とファイルの送受信には同じプロトコルが使用されます-たとえば、SMTPとPOP3に分離されません。 さらに、データの受信と送信は同じ通信チャネル内で並行して行われるため、使用効率が向上します。
  • UUCPを使用すると、ゼロから開始するのではなく、ダングリングデータ転送を続行できます。 これは、劣悪な通信チャネル、低速の通信チャネル、または短命の通信チャネルにとって非常に貴重です。
  • トラフィックの優先順位付けがサポートされています。メッセージの送信時に、重要度（グレード）を示すことができます。 これにより、重い大きなファイルが小さなメールメッセージやリモートコマンドの通過を妨げないことを保証できます。
  • このプロトコルは、双方向バイトストリームを送信する接続の上で機能します。 シリアルCOMポート、モデム、TCP、実行中のプログラムのstdin / stdoutにすることができます。 IPプロトコルの存在は重要ではありません。
  • インターネット電子メールとは異なり、7ビット通信チャネルに関連する制限はありません。 追加のBase64 /変換なしでバイナリファイルを転送できるため、トラフィックを大幅に節約できます。
  • UUCPは、すべての無料オペレーティングシステムで使用できます。 デファクトでは、GNU Taylor UUCP実装を使用しました。 それは長年開発されていません-単にそれを完了するものは何もないため、動作します。
  • sendmail、Exim、Postfixなどの一般的なインターネットメールサーバーはすべて、UUCPと統合できます。 ユーティリティ自体には、UNIXの古典的なアプローチがあります。1つ小さなことをしますが、それはうまくいきます。 FTNネットワークとは異なり、UUCPとインターネットリソースを透過的にリンクできます。




最新の定義を使用すると、UUCPはF2F （友人同士）ネットワークです。 「友人」との関係をすべて自分で明確に規定し、構成します。 さらに、メッセージルーティングを明示的に制御します。 はい、これは大きな責任と労力ですが、検閲を使用できる中央サーバーからのDoS攻撃の可能性に対する優れた防御策です。 F2Fネットワークは自己調整型です。FidoNetで発生したように、攻撃者や悪意のあるユーザーは単にネットワークから追い出されます。 これは非常にうまく機能しますが、実際には、ネットワークはほんの一握りのファンだけでなく、グローバルな規模を持つことができることがすでに示されています。



ただし、UUCPは不明な匿名ユーザーもサポートしています。 つまり、誰でもファイルを配布できるように公開サーバーにアクセスできるようにしたり、ファイルをアップロードしたりできます。



UUCPは、暗号化に関連するものを提供していません。 現在の現実では、電話回線などのモデムを持っている人がほとんどいない場合、少なくともノードの認証には暗号化を使用する必要があります。 幸いなことに、好きなように自由に行うことができます。 物理的に隔離されたエアギャップ、Sneakernet / FloppyNet、またはLAN接続システムの場合は、これに目をつぶることができます。その後、公共通信チャネルを介した通信では、stunnel、SSH、VPNなどをすぐに上げることができます。 UUCPを非表示のTorまたはI2Pサービスとして上げるのは簡単です。

同時に、シリアル接続では、「g」プロトコル（信頼性の低い通信チャネルに使用され、独立して整合性をチェックしてデータを送信します）と、すでに信頼できる通信チャネルであるTCPには「t」プロトコルを使用することを示しました。



この構成では、ベータ版からアルファ版にファイルを既に送信できます。 

 beta％uucp myfile.txt alpha！〜/ myfile.txt




UUCPのデフォルトでは、「〜/」はユーザーのホームディレクトリではなく、全員のパブリックエリアであり、FTPの「pub」ディレクトリに相当します。 私のシステムでは、これはディレクトリ/ var / spool / uucppublic /です。 sysファイルでは、ファイルを送信できるディレクトリまたは要求元のディレクトリをシステムごとに指定できます。 同様に、アルファはファイルを送信できますが、ベータが接続されたときにのみ送信されます。 特定のシステム上の特定のユーザーにファイルを送信できる便利なuutoユーティリティがあります。 

 ％uuto myfile.txt remote！ユーザー名




また、リモートリモートシステムでは、ユーザーはuupickを呼び出して、自分に送信されたすべてのファイルのリストを表示し、たとえば、指定されたディレクトリにファイルを移動することができます。



ベータと接続しているガンマシステムにファイルを送信する必要がある場合は、ファイルのルートを明示的に指定する必要があります。 

 ％uucp myfile.txt beta！gamma！〜/ myfile.txt




UUCPを使用すると、リモートシステムでコマンドを実行するタスクを送信できます。 

 ％uux "remote！service nginx restart"
 ％uux "remote！zfs snap zroot @ backup"




受信システムでは、別のuuxqtデーモンがこの方法で受信したコマンドを開始します。 sysファイルでは、特定のシステムで実行できるコマンドを制御できます。 

 ％grepコマンド/ usr / local / etc / uucp / sys
コマンドrmail /home/uucp/wget.sh




この場合、rmailおよびwget.shコマンドのみを実行できます。 wget.shは、Webページをダウンロードし、UUCPを使用して圧縮された暗号化形式で送信し、優先度を最小限に抑える自己記述型の簡単なスクリプトの例です。 

 ＃！/ bin / sh -e
 WORKDIR = / home / uucp / wget_dir
名前= "$ 1"
 url = "$ 2"
 wget --output-document-"$ url" |  xz -9 |  gpg \
     --homedir /home/uucp/.gnupg --compress-level 0 --encrypt \
     --recipient 0xHIDDEN> $ WORKDIR / "$ name" .xz.gpg
 uuto --grade 9 $ WORKDIR / "$ name" .xz.gpg 'sgtp！vpupkin'




電子メールは、rmailユーティリティを呼び出すuuxコマンドとまったく同じように送信されます。実際には、ローカルのsendmailサーバーと通信し、メッセージ本文を送信します。 たとえば、Postfixがインターネットから受信したメールをUUCPに送信できるようにするには、リモートサーバーで十分です（これについてもここで説明します ）。

  • master.cfのuucp / uuxで行のコメントを解除します
  • UUCPドメインをrelay_domains / relay_domainsに追加します
  • トランスポートでドメインでリレーを生成する方法を指定します。 

     ％cat / usr / local / etc / postfix / transport
 cypherpunks.ru uucp：sgtp
 cryptoparty.ru uucp：sgtp


    ここでは、cypherpunks.ru / cryptoparty.ruドメインについて、UUCPを介して「sgtp」ノードにメールを送信する必要があることがわかります。





ローカルマシンからUUCPゲートウェイを介してメールを送信するには、master.cfのuucp / uux行のコメントを外すだけでなく、main.cfに追加するだけです。 

 default_transport = uucp：uucp-gateway




EximとsendmailでのUUCPサポートも同様に簡単に設定できます 。



したがって、コンピューター間の相互接続が不安定な状況で、すべての最新の無料オペレーティングシステムで便利な仕事を簡単に得ることができます。 実質的に不要なジェスチャーなしで、ファイルを効率的に（たとえば、Base64のオーバーヘッドなしで）転送し、電子メールを交換できます。 さらに、リモートコマンドのバッチを実行することは、電子メールで管理されるサービスを作成するよりもはるかに簡単です。
