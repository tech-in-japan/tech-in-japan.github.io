Rock-Paper-Paperオンラインゲームの例を使用してJavaサーバーを作成するためのReddwarf：Server

RedDwarf-Javaでオンラインゲームを開発するためのサーバープラットフォームの記事で、ゲームサーバーを作成するためのこのプラットフォームの機能について説明しました。 この記事では、RedDwarfを使用してサーバーを作成する方法を例に示します。

例として、ゲーム「Rock-Paper-Scissors」のオンライン実装を作成することが決定されました。

この記事では、サーバーを作成して起動しようとします。 次の記事では、このサーバー用の小さなクライアントを作成し、そのパフォーマンスを確認します。





仕事の準備



まず、ここから sgs-server-dist-0.10.2.zipアーカイブにあるReddwarfサーバーをダウンロードし、sgs-server-dist-0.10.2フォルダーにコンテンツを解凍する必要があります。



プロジェクト作成



お気に入りの開発環境でプロジェクトを作成します。

プロジェクトはシンプルになるため、Mavenは使用しません。

開発には、sgs-server-dist-0.10.2 \ lib \ディレクトリにあるsgs-server-api-0.10.2.jarライブラリが必要です



META-INFフォルダーを作成します。これには、マニフェストファイルMANIFEST.MFが含まれている必要があります。 これがないと、プラットフォームはプロジェクトjarファイルでの作業を拒否します。 私のファイルには1行しか含まれていません：

マニフェストバージョン：1.0



また、META-INFフォルダーにapp.propertiesファイルを作成する必要があります。 このファイルには、サーバーの起動設定が含まれています。 プロジェクトの場合、ファイルには次のプロパティが含まれています。 

#  .        com.sun.sgs.app.name=RockPaperScissors # ,   AppListener      com.sun.sgs.app.listener=hello.reddwarf.server.Server #  ,        com.sun.sgs.app.root=data


これは、最低限必要なオプションのセットです。 開発中に、次のプロパティが引き続き有用な場合があります。

  • com.sun.sgs.impl.transport.tcp.listen.port-サーバーがリッスンするポート（デフォルトでは62964）
  • com.sun.sgs.app.authenticators-認証を担当するクラスの名前（認証プロセスはゲームロジックから削除され、独立したモジュールとして使用できます）
  • com.sun.sgs.impl.service.session.allow.new.login-別のクライアントから接続されたプレーヤーが接続できるようにするかどうか。 trueの場合、ゲーム内にいる人がスローします。 falseの場合、別のクライアントからの接続を許可しません。


その他のプロパティの詳細については、 ドキュメントをご覧ください。



ゲームアーキテクチャ



ゲームには次のエンティティが必要です。

サーバー -プレイヤーのリストをオンラインで保存し、接続の処理を処理するクラス。

プレーヤー - プレーヤーを表します。 プレイヤーには次の属性があります：名前（同じログイン）とポイント数。 戦闘に参加できます。

バトル -バトルです。 このオブジェクトでは、プレーヤーの応答を待って勝者を決定します。 2人のプレーヤーへのリンクが含まれています。

武器は、直接石、はさみ、紙などの武器の単純な列挙です。



クラス図の形式で描かれている場合、次のことがわかります。





すべてのゲームエンティティ（武器を除く）は、サーバーの実行中にトランザクション性を提供する内部データベースに格納され、相互に参照するため、java.io.Serializableおよびcom.sun.sgs.app.ManagedObjectインターフェイスを実装する必要があります。



サーバークラス。 プレーヤーの初期化と接続


Serverクラスはサーバーの起動ポイントであるため、com.sun.sgs.app.AppListenerインターフェイスを実装する必要があります。



void initialize(Properties props)



は、サーバーの最初の起動時に呼び出されます。 作業に必要な初期値で内部データベースを満たします。 重要な機能：サーバーが停止（または強制終了）してから再起動した場合、このメソッドは呼び出されません。 内部データベースはサーバーが起動するまで保存され、停止した瞬間から作業を続けることができます。



ClientSessionListener loggedIn(ClientSession session)



は、認証が成功した後に呼び出され、プレーヤーを表すオブジェクトを返す必要があります。 この例では、これはPlayerになります。



サーバーに接続されているすべてのプレーヤーは、特別なコレクションに保存されます。 Reddwarfには、ゲームエンティティ用のScalableHashMapの特別なコレクションがあります。 このコレクションの利点は、変更されると、完全にではなく部分的にブロックされることです（内部データベースのロックを意味します）。 また、Serverオブジェクトでは、コレクション自体ではなく、コレクションへのリンク（ManagedReference）を保存します。



言葉から行為に移ると、次のコードが得られます。

データベースを操作するには、DataManagerを使用します。これにより、データベースへの書き込み、データベースからの読み取り、ManagedReferenceリンクの作成が可能になります。 データベースはキーと値のストレージであるため、「player」というプレフィックスの付いたプレーヤーの名前がキーとして使用され、Playerオブジェクト全体が値にシリアル化されます。 ベースからプレーヤーをロードする関数を作成します（プレーヤーがデータベースに見つからない場合は作成します）。 



  private Player loadOrRegister(String name) { try { return (Player) AppContext.getDataManager().getBindingForUpdate("player." + name); } catch (NameNotBoundException e) { //       - //   ,       Player player = new Player(name, this); AppContext.getDataManager().setBinding("player." + name, player); return player; } }




プレーヤーのクラスとプロトコル


次はPlayerクラスを作成する番です。 このクラスはプレーヤーを表し、プラットフォームから着信メッセージの通知を受け取ります。 それでは、プロトコルについて話しましょう。 Reddwarfを使用すると、着信および発信メッセージをバイトの配列として処理でき、プロトコルの実装はゲーム開発者の裁量に任されます。 ゲーム「じゃんけん」では、単純なテキストプロトコルを使用します。



（ server- > client） SCORE <number> -サーバーはプレイヤーにポイント数を伝えます

（クライアント->サーバー） PLAY-ゲームを開始するためのプレーヤーのリクエスト

（サーバー->クライアント） BATLE <名前> -指定されたプレイヤーとの戦いが始まりました

（サーバー->クライアント） エラー -戦闘のプレイヤーが見つかりませんでした（サーバーに誰もいないか、すべての戦闘に参加していません）

（クライアント->サーバー） ROCK-プレイヤーは「Stone」と言います

（クライアント->サーバー） SCISSORS-プレイヤーは「はさみ」と言います

（クライアント->サーバー） 紙 -プレイヤーは「紙」と言います

（サーバー->クライアント） DRAW-描画

（サーバー->クライアント） WON-プレーヤーが勝ちました

（サーバー->クライアント） LOST-プレイヤーが失われました



プロトコルから、プレーヤーのアクションと機能のシーケンスを理解できるため、これについて個別に説明することはしません。



このコードを使用して、テキストをバイトにエンコードしたり、その逆を行うことができます。 

 package hello.reddwarf.server; import java.nio.ByteBuffer; public class Messages { public static ByteBuffer encodeString(String s) { return ByteBuffer.wrap(s.getBytes()); } public static String decodeString(ByteBuffer message) { byte[] bytes = new byte[message.remaining()]; message.get(bytes); return new String(bytes); } }

武器および戦闘クラス


武器の列挙は非常に単純であり、コメントは不要です。 

 package hello.reddwarf.server; public enum Weapon { ROCK, PAPER, SCISSORS; boolean beats(Weapon other) { return other != null && this != other && this.ordinal() == (other.ordinal() + 1) % values().length; } }




戦いに移ります。



戦闘には一意の識別子があり、2人のプレイヤーへのリンク、その回答、およびアクティビティフラグが含まれています。



戦闘が作成されると、別のタスクが起動され、5秒で戦闘が完了します。

この時間が経過すると、戦闘の結果が合計されます。 プレイヤーの1人だけが答えを出した場合、両方の場合、勝者と見なされます-勝者は通常の規則「じゃんけん」に従って決定されます。



タスクは、AppContext.getTaskManager（）を使用して取得できるTaskManagerサービスを使用して実行されます。 このマネージャーを使用すると、個別のトランザクションで実行されるタスクをすぐに実行するか、指定した期間の後に実行するか、定期的に実行できます。 予想どおり、すべてのタスクは内部データベースにも保存されます。つまり、サーバーの再起動後に実行されます。



したがって、クラスコードはBattleです。

このコードを読むと、「内部BattleTimeoutクラスが静的になり、バトルへのリンクを明示的に保存するのはなぜですか？ 非静的として宣言し、バトルフィールドに直接アクセスできます。」

実際、非静的内部クラスは親バトルへのリンクを暗黙的な形式で保存し、それを介してバトルにアクセスします。 ただし、Reddwarfプラットフォーム（トランザクション）の機能は、別のトランザクションからManagedObject（Battle）に直接アクセスすることを禁止しています。この場合、例外がスローされます。 別のトランザクション内のオブジェクトへの直接参照は正しくありません。 静的な内部クラスのみを使用するというプラットフォーム作成者の推奨事項は、これに関連しています。



また、参照による管理対象オブジェクトの受信に注意したいと思います。

ManagedReferenceの上記のコードは、get（）メソッドとgetForUpdate（）の両方を使用します。

原則として、get（）のみを使用できます。 getForUpdate（）を使用すると、サーバーは、トランザクションが完了する前にどのオブジェクトが変更されるかを知り、競合するトランザクションが検出された場合、少し前にタスクをキャンセルできます。 これにより、get（）を使用した場合に比べて速度がいくらか向上します。



最後に、サーバーの準備がほぼ整いました。

ロギングを少し追加し（簡単にするためにjava.util.loggingを使用します）、プロジェクトをビルドできます。
