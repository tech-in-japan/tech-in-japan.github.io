レイアウト、依存関係の地獄、後方互換性について

この記事では、レイアウトの概要を説明します。 Linux、BSD *、Mac OS X、Windows上のアプリケーションが依存する共有ライブラリはどこにありますか？ 後方互換性をどうするか？ 中毒地獄に対処する方法？



読者は、「コンパイラ」、「オブジェクトファイル」、「リンカー」、「静的ライブラリ」、「動的ライブラリ」、「動的ローダー」などの文字セットに精通していると想定されるため、何も噛みません。



動的ライブラリの静的ロードの問題：



  1. main.exeはversion-0.3.dllおよびbar.dllに依存しています。 一方、バーはバージョン0.3とバイナリ互換ではないバージョン0.2.dllに依存しています（文字が存在しないだけでなく、名前が一致しますが、引数の数が異なるか、異なる性質のオブジェクトを作成するなど）。 version-0.2.dllの文字はversion-0.3.dllから削除されますか？ ライブラリの静的バージョン（たとえばversion-0.2.lib）と動的バージョン（version-0.3.dll）を使用する場合にも同じ問題が発生します。
  2. ローミングアプリケーションの作成：ダイナミックローダーは、前の段落のアプリケーションのversion-0.?.dllおよびbar.dllをどこで検索しますか？ 別のフォルダーに移動した場合、main.exeの依存関係を検出しますか？ main.exeをアセンブルして、実行可能ファイルに対して依存関係を検索する方法は？
  3. 依存関係の地獄：同じライブラリの2つのバージョン/opt/kde3/lib/libkdecore.soおよび/opt/kde4/lib/libkdecore.so（これによりプラズマが落ちなくなります）、プログラムの半分が最初を必要とし、プログラムの残りの半分が2番目を必要とします。 両方のライブラリを1つのスコープ（1つのディレクトリ）に配置することはできません。 バージョンライブラリの2つのバージョンを同じディレクトリに配置する必要があるため、ポイント1にも同じ問題が存在します。


最初の段落を読んだ後、読者は叫ぶかもしれません。 彼らはそうしません！ ライブラリの1つのバージョンを使用する必要があります！”はい、そうですが、人生ではすべてが起こります。 最も単純な例：アプリケーションはライブラリaとサードパーティのクローズドライブラリ（これは他の誰かの有料製品を強調します ）bを使用します。



巨大なプロジェクトのフレームワークのもう1つの例は、さまざまな部分のリリースには異なる期間があるという事実です（私たちのチームが遭遇し、このテキストが一般的に書かれた原因）。 したがって、主な製品は、パラグラフ1で説明した構成でときどき出てくることがあります。



依存関係の地獄は、システムライブラリとオペレーティングシステムの開発者にとってより重要ですが、アプリケーション領域でも発生する可能性があります。 繰り返しますが、いくつかの実行可能プログラムがある巨大なプロジェクトがあるとします。 それらはすべて1つのライブラリに依存していますが、バージョンが異なります。 （これはステップ1と同じ状況ではありません：1つのプロセスに同じライブラリの2つのバージョンがロードされ、ここでは1つだけが各プロセスにロードされますが、それぞれ独自のバージョンを使用します）。



地獄からの脱出


答えは簡単です。バージョンをライブラリファイル名に追加する必要があります。 これにより、ライブラリファイルを1つのディレクトリに配置できます。 同時に、APIではなくABIバージョンを追加することをお勧めします。これにより、2つの並行バージョンブランチとそれに対応する困難が作成されます。



バージョン管理は非常に日常的な仕事です。 xyzスキームを検討してください。

  • x-重大な問題。 互換性の話はありません。
  • y-軽微な問題。 Libaはソースレベルで互換性がありますが、バイナリ互換性が壊れている可能性があります。
  • zはバグ修正です。 Libaは両方向に互換性があります。


ファイル名にxyを含めることは妥当ですが、マイナーバージョンを増やしながら互換性が維持されている場合は、対応するシンボリックリンクを作成するだけで十分です。

version-1.1.dll

version-1.0.dll→version-1.1.dll



version-1.1.0を使用するアプリケーションとversion-1.0.xを使用するアプリケーションの両方が機能します。



互換性が壊れている場合、システムには2つのファイルがあり、再びすべてが機能します。



何らかの理由でバグ修正中に互換性が失われた場合、マイナーバージョンを増やす必要があります（お気に入りのQtのチームが行ったように、失敗するものは何もありません） [1] ）。



ところで、APIバージョンを含めることを禁止する人はいません。そうすれば、より多くのシンボリックリンクが作成されます。 多くの場合、互換性が維持されます。 しかし、この場合、言及されたQtファイルは簡単に解決され、マイナーバージョンを強制的に増やすことはありません。



これはすべてのプラットフォームに当てはまります。



残りの2つの問題の解決策は、OSによって異なります。



ELF＆GNU ld（Linux、* BSDなど）


ELF形式の共有ライブラリには、いわゆるSONAMEがあります [2] [3] 。 これは、DT_SONAMEセクションのバイナリファイルに書き込まれる文字列です。 ライブラリのSONAMEは、たとえば次のように表示できます。 
$ objdump -p /path/to/file | grep SONAME


fazプログラム/ライブラリがSONAME = baz-0.dllのbazライブラリにリンクされている場合、 baz-0.dll行はDT_NEEDEDセクションのバイナリfazファイルにハードコードされ、開始時にダイナミックローダーはbazという名前のファイルを探します。 -0.dll この場合、ファイルに異なる名前を付けることを誰も禁止していません！



実行可能ファイルが依存するSONAMEを表示できます。 
 $ objdump -x /path/to/file | grep NEEDED


ダイナミックローダーは、次の場所でDT_NEEDEDセクションからこの順序でライブラリを検索します [4] [5] ：

  1. DT_RPATHセクションのディレクトリのリスト。実行可能ファイルにハードコーディングされています。 ほとんどの* nixシステムでサポートされています。 セクションDT_RUNPATHが存在する場合は無視されます。
  2. LD_LIBRARY_PATH-環境変数。ディレクトリのリストも含まれています。
  3. DT_RUNPATH-DT_RPATHと同じ。LD_LIBRARY_PATHの後にのみ表示されます。 最新のUnixライクシステムでのみサポートされています。
  4. /etc/ld.so.conf-ダイナミックローダーld.soの構成ファイル。ライブラリのあるフォルダーのリストが含まれています。
  5. ハードワイヤードパス-通常/ libおよび/ usr / lib。


RPATH、LD_LIBRARY_PATH、およびRUNPATHのデータ形式は、PATHの場合と同じです：コロンで区切られたパスのリスト。 たとえば、次のようにRUNPATHを表示できます。 
 $ objdump -x /path/to/file | egrep 'R(|UN)PATH'


R [UN] PATHには特別なラベル$ ORIGINが含まれている場合があります。これは、ダイナミックローダーがロードされたエンティティへの完全な絶対パスに展開します。 ここでは、一部の開発者がRUNPATHに「。」（期間）を追加することに注意してください。 これは$ ORIGINと同じではありません！ ポイントは現在の作業ディレクトリに展開され、本質へのパスと自然に一致する必要はありません！



書かれている内容を示すために、 p。1のスキーム（githubのストレージへのリンク： github.com/gshep/linking-sample ）に従ってアプリケーションを開発します。 システム全体を構築するには、フォルダーのルートに移動して./linux_make_good.sh



を呼び出す./linux_make_good.sh



で、結果はresult



フォルダーに./linux_make_good.sh



されます。 いくつかの組み立て手順を以下で説明します。



バージョン-0.xライブラリをリンクする段階で、SONAMEが設定されます： 
 $ gcc -shared -Wl,-soname,version-0.3.dll -o version-0.3.dll version.o


システムライブラリのみに依存するため、R [UN] PATHセクションは不要です。



barライブラリはすでにバージョン0.2に依存しているため、RPATHを指定する必要があります。 
 $ gcc -shared -Wl,-rpath-link,/path/to/version-0.2/ -L/path/to/version-0.2/ -l:version-0.2.dll -Wl,-rpath,\$ORIGIN/ -Wl,--enable-new-dtags -Wl,-soname,bar.dll -o bar.dll bar.o


--enable-new-dtags



は、DT_RUNPATHセクションに入力するようリンカーに指示します。



-Wl,-rpath,...



は、R [UN] PATHセクションを埋め-Wl,-rpath,...



。 パスのリストを指定するには、パラメーターを複数回指定するか、コロンですべてのパスをリストします。 
 $ gcc -Wl,-rpath,/path/1/ -Wl,-rpath,/path/2 ... $ gcc -Wl,-rpath,/path/1:/path/2 ...


これで、結果フォルダーのすべてのコンテンツまたはフォルダー全体をファイルシステム内で自由に移動できますが、ダイナミックローダーを起動するとすべての依存関係が検出され、プログラムが実行されます。 
 $ ./result/main.exe Hello World! bar library uses libversion 0.3.0, number = 3 version::get_version result = 0 But I uses liversion 0.3.0 number = 3


そこで、キャラクターをマッシュする問題に取り掛かります！ バーは、get_number（）が2を返すversion-0.2.dllと、同じ関数が既に3を返すアプリケーションversion-0.3.dllを使用します。アプリケーションの出力によると、get_number関数の1つのバージョンは別のバージョンによって上書きされます。



事実は [6; 動的リンクと読み込み、動的リンクのアプローチの比較] GNU ld＆ELFはインポートされた文字の名前空間としてSONAMEまたはファイル名を使用しません：

異なるライブラリが同じ名前のエンティティをエクスポートすると、それらのいくつかは他のものを粉砕し、最良の場合はプログラムがクラッシュします。



ライブラリの1つが静的である場合は単純に解決されます。静的ライブラリのすべてのシンボルを非表示にする必要があります [7、2.2.2グローバル可視性の定義] 。



残念ながら、動的ライブラリの場合、事態はそれほど単純ではありません。 GNU Linker / Loaderには、直接バインディングなどの機能がありません [8] 。 誰かがゲントでこの機会を見ました [9]しかし、すべてが死んだようです。 ディーゼル燃料で彼女は [10] [11]しかし、ソリアルカ自身は死にました...



1つのオプションは、キャラクター自体をバージョン管理することです [7、2.2.5エクスポートマップの使用] 。 これは実際にはキャラクターを飾るようなものです。 （C ++でプログラミングしているリーダーが今叫んでいるものしか想像できない...）

Macの状況はどうですか？


Mac Axisは、実行可能ファイル、および文字の検索に2レベルの名前空間のMach-o形式を使用します [14、2レベルの名前空間] [16] 。 現在はこれがデフォルトですが、フラットな名前空間でビルドするか、プログラムの起動時に完全に無効にすることができます [15、FORCE_FLAT_NAMESPACE] 。 このコマンドは、名前空間をサポートするバイナリが構築されているかどうかを確認するのに役立ちます。 
 $ otool -hv /path/to/binary/file


つまり、名前の追加の装飾を浴びる必要はありません-ファイル名にバージョンを含めるだけです！



しかし、依存関係の検索はどうでしょうか？



マコシでは、ほとんどすべてが似ていますが、異なる方法で呼ばれています。



SONAMEの代わりに、ライブラリIDまたはインストール名があります。 たとえば、次のように表示できます。 
 $ otool -D /usr/lib/libstdc++.dylib
install_name_toolを使用して変更できます。

ライブラリにリンクする場合、そのIDはバイナリで書き込まれます。



バイナリ依存関係は次のように表示できます。 
 $ otool -L /path/to/main.exe
または 
 $ dyldinfo -dylibs /path/to/main.exe


起動時に、dyldは「id」という名前のファイルを開こうとします [15、動的ライブラリのロード] 、つまり、インストール名を依存関係への絶対パスと見なします。 失敗した場合、環境変数DYLD_LIBRARY_PATH（LD_LIBRARY_PATHの完全な類似物）にリストされているディレクトリで、名前/サフィックス "id"を持つファイルを検索します。



DYLD_LIBRARY_PATHでの検索が結果を返さなかった場合、dyldは同様にさらに2、3の環境変数を調べます [15] 、その後、標準カタログでlibを検索します。



このスキームでは、ローミングアプリケーションを収集できません。そのため、id：@ executable_path /で記述できる特別なラベルが導入されました。 起動時のこのマークは、実行可能ファイルへの絶対パスに展開されます。



次に、完成したバイナリの依存関係を変更できます。 
 $ install_name_tool -change /usr/lib/libstdc++.dylib @executable_path/libstdc++.dylib main.exe


これで、ローダーはまずmain.exeと同じフォルダーでこれを検索します。 完成したバイナリを変更しないようにするには、libstdc ++。Dylib lib with id = @ executable_path / libstdc ++。リンク中にDylibをスリップする必要があります。



さらに、1つ、または2つの問題が発生します。 そのような階層があるとします：

  • main.bin
  • ツール/

    • 補助.bin
  • library.dll


main.binはlibrary.dllに依存していますが、tools / auxiliary.binもそれに依存しています。

同時に、id = @ executable_path / library.dllであり、両方のバイナリが単にそれにリンクされていました。 次に、auxiliary.binを起動すると、ブートローダーは/path/to/tools/library.dllを探しますが、当然それを見つけられません！ もちろん、リンクした後にツールでツール/auxiliary.binを修正するか、ソフトリンクをスローすることもできますが、やはり不便です！



さらに良いことに、プラグインに関しては問題が現れます：

  • main.bin
  • プラグイン/

    • 1.プラグイン
    • helper.dylib


1.pluginには@ executable_path / helper.dylibというエントリがありますが、起動時には1.pluginではなくmain.binへの絶対パスに展開されます！



この問題を解決するために、Axis 10.4のYablokoは新しいマーカー@ loader_path /を導入しました。 依存関係の読み込み中、このマーカーは、依存関係をプルするバイナリへの絶対パスに展開されます。



最後の難点は、リンクされたライブラリの2つのバージョンが必要なことです。一部はシステムにインストールされ、id = /usr/lib/libfoo.dylibになりますが、その他はプロジェクトのビルドに使用され、id = @ loader_path / libfoo.dylibになります。 これはinstall_name_toolで簡単に解決できますが、面倒です。 したがって、バージョン10.5から、@ rpath /ラベルを入力しました。 ライブラリはid = @ rpath / libfoo.dylibで構築され、どこにでもコピーされます。 バイナリは依存関係を探すためのパスのリストを使用してコンパイルされ、@ {executable、loader} _path /が許可されます。 
 $ gcc ... -Xlinker -rpath -Xlinker '@executable_path/libs' -Xlinker -rpath -Xlinker '/usr/lib' ...


これは、ELFのRPATH / RUNPATHに似ています。 バイナリが起動すると、行@ rpath / libfoo.dylibは@ executable_path / libs / libfoo.dylibに展開されますが、これはすでに絶対パスに展開されています。 または、/ usr / lib / libfoo.dylibにデプロイします。



次のように、バイナリに配線されたrpathを表示できます。 
 $ otool -l main.bin | grep -A 2 -i lc_rpath


install_name_toolを使用して、rpathを削除、変更、または追加できます。

おわりに


すべての主要なプラットフォームにより、通常のコピーでインストールできるアプリケーションを比較的簡単に作成できます。 ただし、開発者は、依存関係の地獄、下位互換性、およびキャラクターマッシングの問題を自分で解決する必要があります。



主な決定は、正しいバージョン管理を選択し、それを制御することです。
