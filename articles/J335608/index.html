導入部（すべての記事へのリンク付き）



ITIL（v3）で説明されているプロセスには、特に興味深いものが2つあります。「構成管理プロセス」と「変更管理プロセス」で、システム構成の変更を分析および管理します。 話を続けるには、「システム」とは何かを決める必要があります。 この概念には、サービスの提供に（直接的または間接的に）影響を与える膨大な数のコンポーネントが含まれます。



  • サーバー



    • セキュリティ設定（ユーザー、グループ、権利、ファイアウォール）;
    • インストールされたアプリケーションとライブラリ。
    • アプリケーション設定（記述子、メモリ、CPU時間などの制限）;
    • バックアップ


  • アプリケーションおよびシステムソフトウェアの監視システム。
  • 製品自体の構成ファイル、そのコンポーネント、補助システム、およびアプリケーションアプリケーション
  • ...


プロジェクトのシステムの輪郭を最小限にしようとすると（タイプ、システムの機能にはバックアップは適用されません）、自分で穴を掘ることになり、遅かれ早かれ失敗します。





このシステムコンポーネントのリストは、複数のサーバー（さらには1台）に展開されているほとんどのプロジェクトにとって最低限のものです。 さらに、プロジェクトが進むにつれて、その数は変化し、増加します。 会計プロセスがない場合、これが何につながるのかが明らかになります：すべてが完全に機能するテストベンチ（または開発スタンド）がありますが、製品に複製するか、新しいスタンドを作成しようとすると、



  • 非常に長い展開/複製プロセス（障害が発生したときに何がどのように構成されているかを誰もが覚えており、多くの場合、既存のスタンドと設定を比較します）。
  • 予想外に、展開された製品の操作中に問題が発生します（テストベンチで最後の瞬間に行われたが、製品に転送されなかった設定を忘れることがあります）。
  • テストプロセスの速度が低下します（内部テスターであっても、新しいアセンブリの公開は、時間と複雑さの両方で偉業です）。
  • そして最後に、新しいバージョンのリリース前に開発者の間で恐れがあり、その結果、新しいリリースの計算が遅れます（誰も公然と話さないが、その効果は常に感じられます）。


その結果、製品が正しく動作するための環境は制御不能になり、使用可能な自動化およびバージョン管理ツールが使用できなくなります。



やる気



物事は明らかなように見えますが、ほとんどの開発グループでは彼らに目をつぶっています-多くの人は、サーバーが1つ、設定が少なく、労働者の能力があると主張していますリスクなど など



ITILに記載されているものの見かけの複雑さと冗長性（実際には多くの記述があります）にもかかわらず、まず、これらのプロセスの自動化のための要件をそこから取り入れる必要があります。 アセンブリの自動化、テストの自動化、脆弱性の検索の自動化-これらはすべて実装されており、開発中に必要な最小限のものとみなされます。 自動化はプロセスを加速し、自信を植え付け、透明性と保証された結果を提供し、時間と結果に対するパフォーマーからの恐怖を取り除きます。



単一の開発者の動機



前の段落の強力な導入部分を考えると、単一の開発者にとっての利点は、より簡単かつ簡単に説明されます。



  • なぜなら 開発は毎日行われません。その結果、いくつかの展開機能は数日後にスムーズに消えます。
  • ローンの自動化-最も信頼できるパートナー。
  • 自動化によって提供される作業の速度により、「融合」とプロジェクト開発への意欲を失うことがなくなります（これは非常に重要な特定のポイントです）。


ITILに関する重要な免責事項



ITILに記載されているプロセスから、そこに記載されているすべての手順を実行しようとしないでください。状況は以前よりも悪化します。 一部の銀行システムでこのような手順がどのように実装され、それが何をもたらすかを知っています（特に自動化なし）-官僚的な機械は、動的に開発中のプロジェクト/システムを停止させます。

Puppetについて



Puppetは、システム管理者がアクセスできるように開発された独自の構成言語（DSL）を使用します。 Puppet言語は、正式なプログラミング知識を必要としないと位置付けられており、その構文はNagios構成ファイル形式の影響下で形成されました。



Puppet言語の主な目的は、リソース（Puppetのリソース-ファイル、ディレクトリ、サービス、制限、ファイアウォール、ユーザーなど）を定義することです。 言語の他の部分はすべて、リソースの定義方法に柔軟性と利便性を追加するためにのみ存在しています。



リソースグループは、より大きな構成単位であるクラスに編成できます。 リソースは単一のファイル/ディレクトリまたはパッケージを記述することができますが、クラスはサービスまたはアプリケーションの構成に必要なすべてを記述することができます（必要な数のパッケージ、構成ファイル、デーモン/サービス、メンテナンスタスクを含む）。 小さいクラスを組み合わせて、システム全体の役割を記述する大きなクラス（「データベースサーバー」または「クラスター作業ノード」）にすることができます。



内部にリソースがあるクラスの例： 



class apache (String $version = 'latest') { package {'httpd': ensure => $version, # Using the class parameter from above before => File['/etc/httpd.conf'], } file {'/etc/httpd.conf': ensure => file, owner => 'httpd', content => template('apache/httpd.conf.erb'), # Template from a module } service {'httpd': ensure => running, enable => true, subscribe => File['/etc/httpd.conf'], } }


異なる役割を実行するマシンは、一般的な場合、異なるクラスのセットを受け取る必要があります。 どのクラスをどのマシンに適用するかを構成するタスクは、Puppetノードのタスクです。



Puppetノードの定義例 



 node 'www1.example.com', 'www2.example.com', 'www3.example.com' { include common include apache, squid } node /^(foo|bar)\.example\.com$/ { include common }


Hieraの事実とデータベース。 Puppetコードを実行する前に、ノードに関する情報が収集され、収集された情報は事前定義されたファクト（コード内のどこでも使用できる変数）の形式で表示されます。 Hieraは、組み込みの「キー値」データベースです。 デフォルトでは、YAMLまたはJSONファイルがデータソースとして使用されますが、任意のデータソースを使用するように拡張することもできます。 その階層とノードに応じてデータを変更する機能により、その使用はほとんどのモジュール/クラスの作業の不可欠な部分です。



モジュールは、コードとデータの自己完結型ブロックです（クラス、テンプレート、ファイルなど）。 これらの再利用可能で一般にアクセス可能なアイテムは、Puppetの主要な構成要素です。



Puppetの使用を計画している人は、ほとんどの場合クラスと、時にはモジュールを作成する必要があります。



展開オプション



Puppetを実装する場合、集中構成ストレージを使用する場合と使用しない場合の2つのオプションがあります。



  • 一元化された構成ストレージ：多くのサーバーがある場合、その利点は明確に見えます。 この場合、マシンのみに関する情報がクライアントマシンに送信されます。これにより、一定レベルのセキュリティが提供され、トラフィックが最小限に抑えられます。
  • 構成の分散ストレージ：少数のサーバーで正当化されますが、マシンには構成スクリプトとファイルの完全なセットが必要であり、エージェントの起動時にそれらがコンパイルされ、このマシンの一部が実行されます。 15分ごとに実行される通常のcronタスクによって実装されます。

私の場合、もちろん、分散システムが使用されます。なぜなら、 実装が簡単で（インフラストラクチャの編成の観点から）、テストベンチを展開するためのビルドスクリプトが大幅に簡素化されます。テストベンチは1日に数十回作成して実行します。



指定されたスクリプトを実行する場合：



  1. Puppetクライアント自体と必要なパッケージがインストールされている
  2. 必要なPuppetモジュールがインストールされています
  3. 「最新の」ラベルのコミット番号の変更が検証されます（新しいバージョンの統合テストが成功したときに行われます）
  4. 現在の環境のHiera Puppetデータベースの構成（hiera.yaml）が置き換えられます（$ PUPPET_ENV変数）。
  5. Hiera Puppetデータベースのデータを含むYAMLファイルは置き換えられます。
  6. モジュールの説明を置き換えるためにコピーされました。
  7. ノード（私のシステムのサーバー）の構成がコピーされます。
  8. コピー/インストールされたすべての設定のアプリケーションが呼び出されます（$ PUPPET_BIN apply ....）


Puppetクライアントが起動時に実行するタスクのリストは膨大です（手動での検証と必要な実行は単に不可能です）：



  • 開いているファイル、関連するメモリ量、スワップ、接続数に制限が設定されます。
  • ログローテーションが構成されます（システムと、アプリケーションと必要なサービスの両方）。
  • 必要なグループと権限で必要な管理アカウントが作成されます。
  • NTPサーバーがインストールおよび構成されています。
  • SSHサーバーがインストールおよび構成されています。
  • Oracle JDKのインストール
  • ファイアウォールを構成する
  • この特定のノードでプロジェクトまたはそのコンポーネントが機能するために必要な多数のコンポーネントがインストールおよび構成されます。
