1.何となぜ？



定義からわかるように、監視システムは、任意のネットワークノード\マシン\ルーター\の最新情報を取得して、必要な情報をいつでも入力できるシステムです。



なぜこれが必要なのかは明らかです。 何が起こっているかについていくために。



多くの場合、監視システムは、インフラストラクチャ全体とその個々の部分の両方の状態に関する非常に包括的な情報を提供できます。 さらに、アラートを設定すると（監視なしでは価値がない）、緊急事態にタイムリーに対応できるだけでなく、ほとんどの場合、これらの事故を防ぐことができる深刻なツールが手に入ります。

3.3。 ネットワークマップの構築



まず、ネットワークマップをオブジェクトとして作成する必要があります。 通常どおり、 RMBのネットワークマップブランチで、マップタイプを作成、選択、パラメーターを設定すると、システムはすべてを独自に実行します。



カードの種類は次のとおりです。



  1. カスタム これは、ユーザーの要求に応じて、手動で作成するために設計されたカードです。
  2. レイヤー2トポロジー 。 これは、OSIモデルの第2レベルに依存して、ユーザーに対してすべてを自動的に実行できるマップです（実際、デバイスのMACアドレスに基づいてマップを作成します）。
  3. IPトポロジー 、別名レイヤー3 。 レイヤー2の場合と同様に、OSIの3レベルモデル（IPアドレス）に基づいてマップのみが構築されます。


手動マッピングを選択する場合、ノードをツリーからマップレイヤーにドラッグし、それらを接続する必要があります。 通常の描画。



他の2つのケースでは、システムがマップの構築を開始する基準点を選択する必要があります。 たとえば、ルーターのノード。 システムは、接続されたデバイスとそれらが接続されているポートに関するデータをそこから読み取り、その後、次のルーターからデータを読み取り、比較などを行います。 すべてのデバイスを分析した後、NetXMS自体が、ネットワークデバイスと接続先のポートとの間の接続を取得します。 カードプロパティ（ RMB-プロパティ-カードプロパティ ）の対応する項目をタップするだけで、システムがカードにエンドデバイス（プリンター、PC、サーバー）も配置することを示すこともできます。 また、この段落では、必要または必要に応じて「検出半径」を設定できます。

分散ネットワークトポロジを使用すると、エリアのマップにオブジェクトをオーバーレイできます。 また、たとえば、必要なオブジェクトをその上にオーバーレイすることにより、サーバールーム内の機器を備えたラックの写真を使用することもできます。 したがって、ラック内のデバイスのステータスをすばやく判断できます。 私の意見では、これは、状況を説明するために後継者または新しく到着した同僚にケースを転送するために、必要に応じて非常に有用なことです。 視覚的に、便利に、自明に。

3.5。 トリガー



トリガーは、収集されるデータの特定のイベントに応答するように構成できます。 たとえば、ルーターへの特定のピンポン値を超える。 各トリガーの通知方法も構成されます（一部の場合、コンソールのアイコンで十分であり、一部のアラートは可能な限り即座にアラートを送信します）。 イベントへの応答は、アクション構成で事前構成されたものから選択されます。



トリガーを作成するには： ノードまたはテンプレートのデータコレクション構成を入力し、目的のパラメーターをダブルクリックし（そうでない場合は最初に作成する必要があります）、プロパティの[しきい値]タブに移動して、追加ボタンをクリックします。







「 関数 」-条件を計算するために取る値または値。 次の値を取ることができます（検証に使用される値の数は、「サンプル」フィールドに示されています）。

-最後にポーリングされた値：最後にポーリングされた値

-平均値：最後からの平均値

-平均偏差：平均偏差

-前の値との差：前の値との差

-データ収集エラー：データの受信エラー

-値の合計：値の合計

-スクリプト：ユーザースクリプトの結果として取得された値



「 操作 」-トリガー条件をトリガーします。 値を取ることができます（値は対応する[値]フィールドに示されています）。

理論のビット
収集される情報は、いくつかのソースから取得される場合があります。 マシンで実行されているNetXMSエージェント 、サーバー自体が送信する「 単純な要求 」または「 内部 」（たとえば、同じping）、「 SNMP 」、「 チェックポイントSNMP 」、「 プッシュ 」、「 Windowsパフォーマンスカウンター 」 、「 SM-CLP 」、またはカスタム「 script 」。 情報のタイプごとに、受信時にシステムが予期するタイプを選択できます。 たとえば、pingは整数を返し、バイト単位の空きディスク容量も整数になりますが、同じ温度は整数になりません（つまり、実数）。



NetXMSでSNMPを介して情報を収集するには、事前定義されたパラメーターの膨大なリストがあります。つまり、特定のOIDを入力することはできませんが、MIBツリーで見つけることができます。 そのため、メーカーのドキュメントを詳しく調べ、各デバイスのIOD値を個別に探す必要があります。 サポートされていないシステムからのメッセージにぶつかって、デバイスからの情報の読み取りを設定する多くの時間を殺すことができますが、最終的にこの特定のデバイスは要求された情報を提供する方法を知らないことがわかります。 マニュアルと希望を吸うことだけに残っている。



たとえば、ノードの応答時間（ping）に関する情報を収集するには、新しいパラメーター（ RMB-new parameter ）を追加し、[Origin]フィールドで[Internal]を選択して、[Select]ボタンを押し、[Ping Time]を選択します。 Zabbixで私が夢見ていた検索があります。 また、ノードのポーリング期間を変更することは理にかなっています（デフォルトでは60秒です。つまり、監視システムによって問い合わせが行われた直後にルーターが応答を停止した場合、システムに問題があると認識されるまでにほぼ1分かかります）そのため、アラームが鳴ります）。

4.結論



私は自分の記事のタイトルに反論したいと思います。 NetXMSは怠け者向けではありません。 開発者は、 一部の機能などの機能に焦点を合わせなかっただけですが、他のすべてに加えて、インターフェイスも管理しました。



やってみましょう。 ZabbixとNetXMSを使用しました。 このために、これら2つのシステムから自分の気持ちを書きます。



だから。



NetXMSは非常にユーザーフレンドリーなシステムだと思います。 まあ...管理者に...はい、管理者に。 彼女は直感的なインターフェイスと呼ばれるものを持っていますが、それは素晴らしいです。 独自の式を記述する必要はありません。多数のサポートドキュメントを読んで、切り替え前または他の簡単なことの前にping検証を有効にしてください。 しかし。 Zabbixでは、この最も単純なことは本当に複雑なこととそれほど変わりません。つまり、習得すれば残りを行います。 NetXMSにはヒントがあります。ほとんどのパラメーターはインターフェイスにあり、マウスで選択して構成できます。 Zabbixでは、システムがそれだけでこれができるかどうか、もしそうなら、どのように書かれ、どのパラメータが返されるかを調べる必要があります。 Zabbixは、あらゆる場面でコミュニティと多数のテンプレートで勝ちます...このジャンクマウンテンで必要なものを見つけることができれば...一方、NetXMSには、このシステムに作成された多くのファンと追加機能があります。 また、プロジェクトのメインページで説明されているように、システム自体は、すぐに使用できる最も一般的なアプリケーションを監視できます。



私にとってZabbixに対するNetXMSの主な利点は、トリガーに応じて問題のあるノードでスクリプトを実行できることでした。 CPUの温度がしきい値を超えたため、CPU自体がホストを消滅させます。 そして、我々は理解します-問題は何ですか。 おそらくZabbixにもそのようなことがありますが、私はできませんでした。 ルーターのポート監視と同様。 そして、ここにあります...まあ、そのまま...スクリプト言語を習得する必要があり、それだけです...



NetXMSのアクセス権により、システムのさまざまなユーザーの機能を非常に柔軟に区別できます。 ただし、すべてを設定するには、直接調整する必要があります。



一般的に、システムは非常によく似ており、一見したところ、その機能は少なくとも同等のように見えます。 Zabbixはさらに多くのことができると感じていますが、これらの機能は内部にあり、シンプルな定命の管理者にとっては、オールインワンは実現不可能です。 NetXMSは、シンプルでわかりやすく、フレンドリーで楽​​しいものと見なされています。



Zabbixには、優れたドキュメント、Wiki、操作方法、FAQ、ベストプラクティス、さまざまなフォーラムなどがあり、ほぼすべての質問に対する回答を見つけることができます。 そしてそれなら-あなた自身を設定します。



要約すると、NetXMSは、始めたばかりの人や、すべてを監視することに深く潜る必要のない人に推奨したいと思います。 さらに対処すれば、非常に微妙な観察結果を得ることができると確信しています。



しかし、それでも、私が見る主なプラスは、非常に素晴らしいインターフェースとセットアップの容易さです。
