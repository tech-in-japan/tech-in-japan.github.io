Vue.jsの作成者がHabrと出会う





すべて金曜日に！



約束通り、私たちは以前の投稿で長く苦労して集めた質問に対するEvan Yuの回答と、Telegramでのロシア語のVueチャットを公開しています。



Vueについて



Q： Vue.jsの最もユニークなキラー機能は何ですか？

O：プログレッシブ適応性：jQueryの代替として使用したり、CLI、ルーター、Vuexを使用して複雑なアプリケーションを作成したりできます。



Q： Vueの弱点は何ですか？

O：現時点では、おそらくタイピングに対する不親切さがあります。 APIは、型付き言語（TypeScriptなど）のサポートを計画することなく開発されましたが、2.5では大幅に改善されました。



Q： Reactは自分自身をライブラリと呼び、Angularは自分自身をフレームワークと呼びます。 Vueがフレームワークとして位置付けられるのはなぜですか？

O： 「プログレッシブフレームワーク」とは、ライブラリとして、または選択したフレームワークとして使用できることを意味します。 フレームワークの一部（CLI、ルーター、状態管理パターン）がありますが、必要でない場合、誰もそれらを強制的に使用することはありません。 （より詳細なビデオ （eng）を参照してください- 約。 ）



Q： VueはReact / Angularの完全な代替品ですか、それともニッチフレームワークですか？

O：はい、完全に置き換えることができます。 これはニッチフレームワークではありません。



Q： Vue.jsの予想されるライフサイクルは何ですか？ Web Components / VanillaJSは近い将来、JSフレームワークを置き換えることができますか？

O：私たちは急速に成長しており、現在は比較的安定した成熟した状態にあります。 いいえ、Webコンポーネントがフレームワークに取って代わるとは思いません。 それらは低レベルのビルディングブロックにすぎません。 今日使用している場合でも、おそらくPolymerを使用せざるを得ません。Polymerは、Webコンポーネント上のフレームワークです。 独自の追加のAPI、パターン、ライブラリ、およびツールがあります。 Webコンポーネントの利点は、主にフレームワークまたはサードパーティパッケージ（サードパーティの配布）間の互換性にあります。 アプリケーションの内部では、既存のコンポーネントモデルと比較して新しいものは何も提供しません（実際、それよりも少ない場合があります）。 VanillaJSは、プロジェクトの開発を永遠に費やす余裕がある場合にのみ意味があります。



Q：非推奨、互換性のない変更、下位互換性に関するVueのポリシーは何ですか？

O： semverに従います 。 マイナーリリースでは廃止された機能を宣言できますが、実際には削除されないため、アプリケーションは中断しません。 メジャーバージョン/互換性のない変更については非常に慎重であり、近い将来にはそれらを計画しません。



Q： Vueの推奨コード構造は何ですか：OOPまたはFP /宣言的アプローチ？

O： 1番目でも2番目でもありません。 UIコードで高レベルのパラダイムを引き出すことは意味がないと思います。 両方から良いアイデアを選んでください。 しかし、Vue側を本当に選択したい場合は、もう少しOOPがあると思います。



Q：フォームはAngularでは強い特性であり、Reactでは弱い特性です。 Vueは複雑なフォームでどのように機能しますか？

O： Vueには、AngularのngModelと非常によく似た組み込みの双方向バインディング（vモデル）があります。 また、Angularのような検証ライブラリを最上部に構築することもでき、エコシステムにはすでにいくつかのライブラリが存在します。



Q：中国で最大のVueライバルは何ですか？

O： ReactとAngularは中国で非常に人気があります。 実際、それらに加えて、他にはありません。



Q： Vueはベンチマークが実行されていると判断し、テスト結果（hello、VolksWagen！）を特に過大評価していると噂されています。

O：証拠のないうわさはうわさだけです;）Vue.jsコードは公開されているため、そのようなコードがあれば簡単に見つけることができます。



サードパーティツール



Q： Weexの見通しは？ Weexの使用を今すぐ開始すべきですか？ それとも、NativeScript-Vueでの作業に集中する方が良いでしょうか？

O： Weexは信頼できるテクノロジーです。 Alibaba内で広く使用されています。 単純に、他とは異なり、彼らはドキュメント/開発者サポートに多くのリソースを割り当てません。 NativeScript-Vueには、最高のドキュメントと開発者のフィードバックがあります。 チーム/製品/環境によって選択を決定する必要があるため、明確な推奨事項は示しません。 これらの技術の可能性を最大限に引き出すには、両方を試す必要があります。



Q： Nuxtについてどう思いますか？

O： Nuxtは素晴らしいことです。 VueでSSRが必要な場合-すべてを自分で設定したくない場合に使用します（ Razzle- およそTrans。のようなフレームワークに依存しない選択肢がまだあります ）



Q： Vue.jsのQuasar Frameworkは、React.jsのReactNativeの類似物だと思いますか？

O：いいえ。 QuasarはCordova / WebViewに基づいているため、VueのIonicに似ています。



Q： Loguxおよび分散データ型との統合について、一般的にどう思いますか？ 次の革命はここに概説されていますか？

O：残念ながら、私はLoguxに慣れていません。



Q： VueのKotlinプログラミング言語（SFC、プラグインなど）のサポートはありますか？

O：私はそのようなことを知りません。



ドキュメント



Q： Vue.jsの公式ドキュメントの例は、ページで直接実行できないため、必ずしも明確ではありません。 CodePenまたはJSFiddleをVue.js公式ドキュメントに統合する計画は何ですか？

O：良い提案。 時間があれば、おそらく検討するでしょう。



Q：開発を開始する前にページレイアウトをコンポーネントに分割する方法は？ エヴァン・ユーの「コンポーネント基準リスト」のようなものはありますか？

Q： VueアプリケーションをWebコンポーネントに変える方法は？

Q： Vue.jsの公式ドキュメントでは、推奨されるTypeScriptコンパイラオプションは「モジュール」：「es2015」です。 「モジュール」：「amd」、「モジュール」：「システム」の使用はどの程度受け入れられますか。

O：この推奨事項は、webpackやロールアップなどのESモジュール互換パッカーが使用されるという前提に基づいています。 個人的に、私はもはやAMDまたはシステムを使用していないので、この質問に答えることはできません。 私の一般的な提案は、あなたが何をしているかわからない限り、それらを使用しないことです。 あなたはまだこの質問をするので、メインストリームを使用します（つまり、デフォルト設定- 約Per。 ）。



Q：テンプレートのASTビルド仕様はいつリリースされますか？

O：そのような計画はないと思います。 このようなソリューションを検討するには、非常に説得力のあるユースケースが必要です。



機能リクエスト



Q：モジュラーバンドラーなしで単一ファイルコンポーネントコンパイラを使用できますか？

O：はい、SFC（シングルファイルコンポーネント）コンパイラがあります。外部またはブラウザ内コンパイラの作成に使用できることを願っています。



Q：ライブラリを生成するためのテンプレートがvue-cliに追加されますか？

O： vue-cli 3は、 ライブラリの作成にも使用できます 。



Q： Reactバージョン16.2のフラグメントはうまく機能しています。 Vue.jsでも同様のものがありますか？

O：はい、近い将来ではありませんが、 現在の差分アルゴリズムでは多くの変更が必要です。 Reactは、ほぼ2年間、すべてを完全に書き換える必要がありました。



Q： Vuexはアクションとミューテーション間のミドルウェア処理をサポートしますか？

O：ありそうもない。 ほとんどの場合、アクション（変異）の違いを取り除きます。



Q： `npm run lint`コマンドの実行はいつ自動修正をサポートしますか？

O： vue-cli 3はすでにこれをサポートしています。 `--fix`フラグを追加して、コマンドを自分で変更することもできます。



Q： Reactは最近リリースされた別名Fiber非同期レンダリングです。 Vueの答えは何ですか？ Vueは3つのフレームワークの中で最も高速ですか？



O：非同期性はReactを高速化しません。 目に見えるパフォーマンスを向上させるために、更新を優先する特定のシナリオでのみ使用されます。 Vue更新エンジンには、これらのシナリオの一部を既にシミュレートする機能がありますが、使用を簡単にするために対応するAPIを開く必要があります。



要するに、完全な同等のFiberを作成する予定はありませんが、そこに登場したいくつかの革新のためのAPIを提供します。

コミュニティ



Q： Vueは多くのAngularおよびReact機能を吸収しました。 AngularおよびReactコミュニティとVueの関係は何ですか？

O：実際、私はReactコミュニティと頻繁に連絡を取り合っています。 私は彼らの良いアイデアがすべて好きです。 しかし、私はAngularコミュニティのそれほど多くの人々に精通していません。



Q： Vueの開発に貢献したいです。 どの分野で助けが必要ですか？



O：他の人の質問に答え始めましょう！ GitHubでのディスカッション、Discordでのチャット、StackOverflow（またはToaster- 約Per。 ）が可能です。 答えはできるだけ明確でなければならないことを覚えておく必要があります。 通常、問題をよりよく理解するために、ドキュメントまたはソースを読み直す必要がある場合があります。 しばらくこれを行うと、Vueの理解が深まり、Githubでより多くの問題を解決できるようになり、コード例の支援、バグのソースの発見、修正や新機能の作成さえ可能になります。



Q：訪問する価値のある最大のVue会議、会議、イベントは何ですか？

今後の計画



Q： Vue 3.0で何か作業はありますか？ その場合、重要な変更点は何ですか？

O： 3.0の計画はありますが、これまでのところ確実ではありません。 次の大きなリリースは2.x-nextブランチで、2.xと機能的に完全に互換性がありますが、内部反応性システムはES2015 Proxyに書き換えられました。 これにより、パフォーマンスが向上し、既存の実装の制限の一部を取り除くことができます（たとえば、オブジェクトと配列の変更の追跡- 約Per。 ）。 IE11以前のサポートは停止しますが、最新のES2015機能を使用してコードをより効率的にしましょう。



Q： （$ parentを介して）親コンポーネントにアクセスすると、経験の浅い開発者がひどい間違いを犯す可能性があります。 これについてどう思いますか？ そのようなアクセスを削除する予定はありますか（たとえば、バージョン3.0）。

O：特定の状況で役立つ可能性があるため、削除することはほとんどありません。 ESLintのカスタムルールなど、静的分析を使用して、親へのアクセスをいつでも拒否できます。



私自身について



Q： IT関連のトピックから離れましょう。 経験をまとめると、どんな知識を共有できますか？ あなたの興味は何ですか、そしてそれらはどのようにプログラミングに適合しますか？ 一般の人々、特に開発者にどのようなアドバイスをしますか？

O：私は美術教育を受けており、ほとんどの部分でプログラミングを勉強しました。 おそらくこのため、私は実用的で学びやすいソリューションに感謝する傾向があります。 時間が経つにつれて、私は多くの場合、プログラミングは適切な妥協案を選択することであることに気付きました。 私のアドバイスはあまりカテゴリー的ではなく、実用的であり、妥協に備えるべきだと思います。



Q： Vue.jsの作業には1日何時間かかりますか？

O：これが私の主な仕事になって以来、私は通常の平日の1日を9から5に費やしています（まあ、もう少し長いかもしれません）



Q： Google、Meteorなどでの過去の経験について教えてください。 Vueのアイデアはどのように生まれましたか？

O：すぐにdevchat.tvで利用できるポッドキャストのエピソードを終えました。 そこで、私はそれについて詳しく話します。



Q：あなたの意見では、今日のウェブ技術のスタックの中で最も重要な瞬間は何ですか？ どの分野で変更が必要ですか？ 何が整然としていて、何がそうでないのか？

O：フロントエンドについてのみ話すことができます。 主な問題は、Webプラットフォームが提供する標準APIが低すぎることだと思います。 その結果、非常に多くのライブラリが、プラットフォームが解決できないほどのほぼすべての問題を解決します。 これは良い面と悪い面の両方です。 良いことは、これらの革新（競争によって引き起こされる）が素晴らしいアイデアを生み出すことです。 マイナス面は、表示されるほとんどすべてのスタックが、急速に発展する膨大な数の依存関係で構成されていることです。 これは今後数年間続く可能性があります。



個人的には、いつの日かブラウザが非常に効果的になり、アセットの配信をバンドルしたり、最適化することについて心配するのをやめることができることを願っています。 これにより、今日のフロントエンドアーキテクチャで通常見られる一時的な（ストップギャップ）ツールの問題がすべて削除されます。 ただし、これがいつ起こるかはわかりません。



Q： Viewのどの技術ソリューションを最も誇りに思っていますか？

O：簡単に学べると同時に、非常に強力です。



Q：トニー・ホアはヌルを10億ドルのエラーと呼んだ。 あなたのキャリアで最悪の技術的決定は何でしたか？

O： Vue 2.xのコードの書き直しを始めたときでも、最初にTypeScriptを使用するといいと思います。



Q：あなたが取り組んだ最も難しいVue機能は何ですか？

O：おそらく2.4でのサーバーレンダリングの最適化により、レンダラーはコンパイル時に作成されたVDOMノードと文字列表現ノードの混合を処理できました。
