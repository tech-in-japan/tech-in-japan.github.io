IBM Tivoli Storage Managerを使用したIBM SAP HANAデータのバックアップ（RK）

IBM SAP HANAとは何ですか？


SAP HANA-高性能分析アプライアンス。 これは、RAMでクエリを実行するビジネスアプリケーションと分析のリアルタイムプラットフォームです。



IBMは、インメモリコンピューティングテクノロジーを使用して、SAPによって実装されるタスクを解決するためのバランスの取れたプラットフォームを準備しました。 IBM SAP HANAコンプレックスは、規模が柔軟で、高い耐障害性を提供​​する共有ディスクアクセスを備えた高性能ファイルシステムであるGPFSソフトウェアを含むコンピューティングユニットで構成されています。



SAP HANAバックアップが定期的にどのように編成されるか


バックアップは、ディスクを使用する際のエラーからデータを保護するために使用されます。



SAP HANAバックアップには以下が含まれます。

  • RKからファイルシステム（外部マウントドライブ）
  • ERPからのレプリカデータ、シミュレーションデータを含むRK
  • RKは、「SAPインメモリコンピューティングスタジオ」を使用するか、コマンドラインからSQLスクリプトを使用して実行されます。
  • RK構成ファイルの場合、それらは別のストレージデバイスに手動でコピーされます
  • RKログはSAP HANA 1.0では使用できません


カザフスタン共和国のディスク容量の計算は、SAP HANAの使用済みRAM + 10〜50％の予備に基づいています。 使用済みのスペースのみが予約されます。



ボリューム3.5 TBのSAP HANAのバックアップシステムの実装を検討する


図1に示すように、検討中のSAP HANAパッケージは8つのノードで構成されています。 7つの本稼働ノード。そのうちの1つはマスターノードとして割り当てられ、8番目のスタンバイノードは単一クラスター内で高可用性機能を実装します。



図1-SAP HANAクラスター







IBM SAP HANAソリューションはGPRSクラスターを使用して構築されており、コンプレックス自体のディスクアレイのメンテナンスと可用性を必要としないことに注意してください。



SAN用のTivoli Storage ManagerバックアップクライアントとTivoli Storage Managerはマスターノードにインストールされますが、単一障害点を排除するためにスタンバイノードにインストールすることもお勧めします。



初期段階では、データはHANA Data Studioを使用してSAP HANAクラスター外のファイルシステムにアップロードされます。この機能の実装については、SAPのドキュメントで説明されています。



バックアップおよびアーカイブコピーの長期保存を整理するために、ディスクシステムをIBM SAP HANAコンプレックスに接続します。 その結果、ディスクストレージ上のDataStudioを使用して、Tivoliツールを使用してさらにコピーするための「フラット」ファイルを取得します。



バックアップクライアントは、ディスクシステムからデータを取得し、テープへのデータの書き込みを開始します。 バックアッププロセスからの本稼働サーバーからの負荷を減らすために、Lan-Freeクライアントを使用することをお勧めします。 このため、FCアダプターがクラスターのマスターノードとスタンバイノードに追加されます。 IBM Storwize V7000ディスクシステムは、格納されたデータの量に対する顧客の要件を満たし、高速アクセスを提供できるディスクシステムとして提案されました。



ソリューションコンポーネント：

  • テープライブラリ
  • ディスクシステム-Storwize V7000
  • バックアップサーバー
  • SANスイッチ
  • ソフトウェア-Tivoli Storage Manager Extended Edition
  • ソフトウェア-SAN用Tivoli Storage Manager


計算例


次の要件を備えたシステム実装を検討してください。

  • 生産ベースサイズ-3.5 TB
  • 有効なバックアップウィンドウ-5時間
  • 1日のコピー数-1
  • 毎日のコピーの保存期間-90日
  • 一年を通して毎月のコピーを保管する
  • RPO-24時間
  • RTO-5時間


要件に従って、計画されたストレージのデータ量を計算します。

1つのHANAノードのファイルセットのサイズは512GB x 1.5 = 750GBです（SAP HANAのドキュメントでは、カザフスタン共和国のボリューム率が10％から50％に増加しています）。







速度要件を考慮してテープライブラリを計算します







初期要件に基づく合計

カザフスタン共和国のテープでのストレージの年間データ量-617 TB

リボンの数-412 LTO5カートリッジ

ライブラリ内のドライブの数は少なくとも4です



図2-Lan-Free Backupを実装するためのSAP HANAクラスターの切り替え







  1. ユーザーをアプリケーションに接続するためのネットワーク（SAP HANAクライアントネットワーク）-1 Gbイーサネット-サーバーあたり2ポート
  2. GPFSクラスターを編成するためのネットワーク-10 Gbイーサネット-サーバーあたり2ポート
  3. データ複製用のネットワーク（SAP複製）–各ノードに1 Gbイーサネット2ポート
  4. HANAからStorwizeへの接続ネットワーク-各マスターノードとスタンバイノードの8 GB FC 2ポート
  5. マスターノードとスタンバイノードの2つのFCポートのテープライブラリへのネットワーク接続
  6. サーバー管理用ネットワーク（IMMネット）機器管理-1 Gbイーサネット-サーバーごとに1ポート
  7. アプリケーションレベルでのSAP HANAノード間のデータ交換用の高速ネットワーク（SAP HANAネット）-10 Gbイーサネット-サーバーあたり2ポート
  8. SAPアプリケーション管理ネットワーク（SAPアプライアンス管理ネット）。 1GBイーサネット監視システム


おわりに


開発したソリューションにより、データのバックアップコピーとアーカイブコピーをテープメディアに保存する要件を実装できるため、長期的で費用対効果の高いストレージ（1TBの保存データあたりのコスト）を確保し、法律および業界の要件に従ってさまざまなデータストレージポリシーを実装できます。



提案されたシステムを使用すると、IBM Tivoli Storage Managerに基づいて、SAP HANAデータを集中バックアップシステムに統合できます。
