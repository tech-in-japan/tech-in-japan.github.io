最初に、いくつかのコメント：



1. xWikiを開始するには、 ホストとして使用するPCにJavaをインストールする必要があります

2.構成について説明するポータブルバージョンは、WindowsとLinuxを実行するPCの両方で実行されますが、別々に説明するいくつかの点でチューニングが異なります。

3.著者の意見では、彼の経験に基づいて、Intel / AMDファミリーの別個のデュアルコアプロセッサと、少なくとも3〜4 GBのRAMまたは同様に構成された仮想マシンを別々のホストで快適に動作させることが望ましい。 ホストがオフィス作業用のPC /ラップトップの場合、RAMを少なくとも4GBにアップグレードし、i3プロセッサー（またはそのアナログ）以上を搭載することをお勧めします。

4. Wikiエンジンの構成ファイルを正しく変更するには、Notepad ++または同様のものを使用します。

5.変更するたびに、念のためバージョンの中間バックアップを作成することをお勧めします。

6.プログラムは、技術エンジニア/見積もりとして作業結果を構造化するために、それ自身のために調整されました。

開発

I.情報ストレージの構造。
データストレージの場合、Jettiに基づくポータブルバージョンは、HSQLDBデータベースとそのすべての長所と短所を使用します。特に、32 MBを超える投資にはソフトウェアの制限があります（必要に応じて増加できますが、このデータベースは使用できません）。 おそらくあなたの特定のケースではこれで十分でしょうが、HSQLDBに小さなファイル（写真1..30Mb）を4GB以上詰め込むと、システムの動作が遅くなり、ある時点で新しい添付ファイルの受け入れを拒否します。 テキストと添付ファイルを分離するには、次の方法も最適です。つまり、 xWikiはハードドライブ上のフォルダーにファイルを保存することをサポートします。



まず、添付ファイルの保存場所をデータベースからファイルシステムに変更します。このため、xwiki.cfg（.. \ webapps \ xwiki \ WEB-INF \）を変更する必要があります（構成ファイルの文字シーケンス＃-＃および＃はコメントを意味します）



xwiki.store.attachment.hint = file

xwiki.store.attachment.versioning.hint = file

xwiki.store.attachment.recyclebin.hint = file



ここの最初の変数は、添付ファイルがハードドライブに書き込まれることを意味します。 2番目は、添付ファイルのバージョンもハードドライブに保存されることを意味し、3番目は、削除されたファイルを含むバスケットもハードドライブに保存されることを意味します。 このソリューションの欠点は、ファイルシステムに分類されるファイルへのパスの負荷です。



また、サイトのドキュメントには、このファイル構造に手を加える必要はないという発言があります。そうしないと、データとリンクが一致しなくなり、エラーが発生する可能性もあります。



有名なバグ
XWikiには、UIを介してごみ箱から添付ファイルを復元できないバグがあります。空のLiveTableが見つかります。 添付ファイルの削除アクションにより、ドキュメントに新しいバージョンが追加されるため、添付ファイルを復元する正しい方法は、必要なバージョンにロールバックすることです。 添付ファイル付きのドキュメントを削除してから、もう一度[削除]をクリックしてドキュメントを履歴から完全に消去すると、添付ファイルもシステムから一緒に削除されます

このバグが修正されるまで、ファイルシステム内の添付ファイルの混乱を避けるためのオプションの1つは、添付ファイルのごみ箱をオフにするだけです。



おおよその翻訳

XWikiには、ユーザーインターフェイスを介してごみ箱から削除された添付ファイルを復元できないエラーがあります。空のLiveTableが見つかります。 添付ファイルを削除するアクションは、ドキュメントのバージョンのカウンターをノックダウンするため、添付ファイルを復元する正しい方法は、目的のバージョンにロールバックすることです。 アプリケーションでドキュメントを削除し、再度ボタンをクリックしてドキュメントを履歴から削除すると、添付ファイルも一緒にシステムから削除されます。 このエラーが修正されるまで、ファイルシステム内の添付ファイルとの混同を避ける1つの方法は、削除されたすべての添付ファイルのゴミ箱を単に無効にすることです。



なぜなら 削除されたファイルはユーザーインターフェイスのバスケットに表示されません（削除されたページには適用されません）。このバグが修正されるまで、削除されたファイルとそのバージョンのバスケットを無効にすることをお勧めします。

storage.attachment.recyclebin = 0

xwiki.store.attachment.versioning = 0



アクションを有効にするには、XWikiを再起動します



次に、投資のサイズの32MBの制限を変更しました。 それはもはや関係ありません、これのためにそれは必要です：

1.移動： http：// "yourwiki" / xwiki / bin / edit / XWiki / XWikiPreferences？Editor = object

III。 Office統合を構成する
もちろん、 Open Office / Libre Officeを意味します。 最初に、システムにパッケージの1つをインストールする必要があります。 次に、xwiki.properties（.. \ webapps \ xwiki \ WEB-INF \）を変更し、パラメーターを設定する必要があります。

openoffice.autoStart = true



また、インストールされているオフィススイートのディレクトリへのパスを次の変数で指定します。

openoffice.homePath = C：/ユーザー/ {user} /AppData/Roaming/openOffice.org/3

openoffice.profilePath = C：/ Program Files / OpenOffice.org 3



既知のバグと回避策
XWikiバグ7164バグのため 、XWikiはxwiki.propertiesファイル内のオフィスサーバーのopenoffice.homePath変数を無視します。 これを行うには、WindowsのXWikiディレクトリにあるstart_xwiki.batファイルに、次の行を追加して変更する必要があります（インストールされているOfficeパッケージへのパスを考慮して）。

set XWIKI_OPTS =％XWIKI_OPTS％-Doffice.home = "C：\ Program Files \ LibreOffice 3.4"



Linuxの場合、start_xwiki.shファイルを変更します

XWIKI_OPTS =％XWIKI_OPTS％-Doffice.path = / usr / lib / libreoffice /を設定します


XWikiを再起動します。

import com.xpn.xwiki.*;

import com.xpn.xwiki.api.*;

import com.xpn.xwiki.doc.*;



import org.apache.commons.io.FileUtils;



import java.io.File;

import java.net.URL;



class ExportedAttachments {

static attachmentsNo = 0;

}



// Export attachments

def void exportAttachments(XWikiContext context) {

def dirPath = request.dirpath + "/";

def dir = new File(dirPath);



String hqlDocCount = "select distinct count(doc.name) from XWikiDocument as doc";

String hqlDocList = "select doc.fullName from XWikiDocument as doc";



if(request.spaceName && request.spaceName != ''){

hqlDocCount += " where doc.web='" + request.spaceName + "'";

hqlDocList += " where doc.web='" + request.spaceName + "'";

}



XWiki wiki = context.getWiki();



int total = wiki.getStore().search(hqlDocCount, 0, 0, context).get(0);

println "**" + total + "** documents to search for attachments";



List documentNames = null;

int offset = 0;

while (documentNames == null || documentNames.size() == 100) {

documentNames = wiki.getStore().search(hqlDocList, 100, offset, context);



for (docName in documentNames) {

try {

XWikiDocument doc = context.getWiki().getDocument(docName, context);

exportDocAttachments(doc, dir, context);

} catch (Exception e) {

println "{{error}} Error : " + e + "{{/error}}";

}

}



offset += documentNames.size();

}



println "";

println "((({{info}}Export done.{{/info}})))";

println "**" + ExportedAttachments.attachmentsNo + "** attachments exported.";

}



// Export attachments for the given document

def void exportDocAttachments(XWikiDocument doc, File dir, XWikiContext context) {

if(doc.getAttachmentList().size() > 0){

println "* **" + doc.fullName + "**";



for (XWikiAttachment attach in doc.getAttachmentList()) {

println "** " + attach.filename;



try {

// Create file

File spaceDir = new File(dir, java.net.URLEncoder.encode(doc.web));

File docDir = new File(spaceDir, java.net.URLEncoder.encode(doc.name));

File destinationFile = new File(docDir, java.net.URLEncoder.encode(attach.filename));

def attachmentURL = doc.getExternalAttachmentURL(attach.filename, 'download', context);



// Copy attachment from URL

URL url = new URL(attachmentURL);

FileUtils.copyURLToFile(url, destinationFile);



ExportedAttachments.attachmentsNo++;



}catch(Exception e){

println "{{error}} Error: Could not create file: " + e + "{{/error}}";

}

}

}

}



// Handle request

if(request.confirm && request.dirpath){

// Set database

def currentDatabase = xcontext.getDatabase();

def newDatabase = currentDatabase;

if(request.dbName && request.dbName != ''){

newDatabase = request.dbName

}

xcontext.setDatabase(newDatabase);



println "=Exporting attachments from **" + newDatabase + "** wiki=";



// Export

exportAttachments(xcontext.context);



// Reset database

xcontext.setDatabase(currentDatabase);

}
