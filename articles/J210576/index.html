理想的には、設計は会社の最初の製品の発売から体系的に取り扱われるべきです。 しかし、これはさまざまな理由で常に当てはまるわけではありません-できるだけ早く開始することが重要です、開始時にお金がなかった、最速の成長が必要でした、コンセプトは実用的なビジネスモデルの検索で絶えず変化していました 最初のWindows、バージョン4.0までのAndroid、および他の多くの製品を思い出してください。 なぜこれが起こっているのですか？













1985：約50万部。 2013：市場の90％以上。



2009：市場の約2％。 2013：市場の80％以上。 インターフェースの進化 。



製品のオーディエンスの成長率と、企業の初期における市場への浸透の深さは、技術的ソリューションの完成度とデザインの品質よりも重要です。 これは特に、リーンスタートアップとMVP（最小実行可能製品）の概念に当てはまります。まず、適切な市場の適切な対象者に対して適切な問題を解決していることを確認する必要があります。 そして、そうして初めて製品を磨くことができます。



ある時点で、製品を成功させるための設計の重要性を理解することができます。 理由は異なる可能性があります-競合他社が激化している、ユーザーからの悪いフィードバックがある、否定的な世論がある、視聴者の流出がある、流行しているなど） 同社は、ユーザーベースの積極的な蓄積からその保持へと移行しています。



何年も前、当社のMail.Ru Groupも同様のタスクに直面していました。 約40の製品があり、それらの多くにはモバイルバージョンとタブレットバージョンもあります。合計で、数百のプロジェクトが出ています。 それらの重要な部分は彼らのニッチのリーダーです。 総視聴者は1億人以上のユーザーです。 このスケールは、現在の設計を再考するプロセスにその機能を課します。



このシリーズの記事では、製品の大規模なポートフォリオを持つ企業でUXが変化している開発のレベルについて説明します。 私たちの経験、他の大規模な組織の観察、およびUX成熟度の既存モデルの調査に基づいて、3つのレベルの開発を特定しました。



  • 運用 -究極のパフォーマーとしてのデザイナー。 設計成果物を作成するタスクのパフォーマンスを提供します。
  • 戦術 -製品チームの一員としてのデザイナー。 設計作業を他の製品タスクと密接に統合します。
  • Strategic-ビジョンを持った製品マネージャーとしてのデザイナー。 製品開発に関する戦略的意思決定に影響します。






これらの開発レベルは、製品およびビジネス管理の浸透レベルに対応しています。 それぞれに、それを実現するために必要な基本的なコンポーネントがあります。 また、持続可能な開発を提供するもの。 持続可能な開発は、企業の設計を体系的に予測可能な品質で進化させるために重要です。 これを合わせて、設計戦略と呼びます。 これについては、私の一連の記事で説明します-膨大で、あなたにとって役立つことを願っています。



会社の成熟度



UXの成熟度に加えて、会社自体も成熟している必要があります。 彼らは、ビジネスの成功と内部組織の観点から、さまざまな開発レベルにあります。 これにより、目標の選択におけるモビリティ、ビジネスモデルの安定性、市場での現在の位置、製品およびプロセスへの投資資金の可用性、従業員と経営陣の資格と先見性、共通の優先事項などが決まります。 これはUXが開発する環境であり、その機能はUX戦略の実装を成功させるために非常に重要です。 それを実装するには、会社のビジネスを深く理解する必要があります。



私たちのタスクでは、会社の3つの重要な要素（リソース、プロセス、優先順位）の状態を理解することが重要です。



資源



これらは、UX戦略を実行する機能を提供する基本的な要件であり、一般に設計への注意が必要です。 最も単純な意味では、リソースは次のとおりです。



  • 最も基本的なニーズ以外に投資できるお金は、現在の製品の効率を確保し、新しいユーザーを引き付けることです。 会社に一般的なお金があるのか​​、特定の製品またはユニットがあるのか​​を理解することは有益です。
  • 設計およびUXタスクを実行できる人 。 彼らの資格と作業負荷は特に重要です。
  • 開発チームが製品またはそのコンポーネントを起動するまでの時間 。
  • 信用の信用 。これにより、管理の側からのあらゆる小さな事柄について、手動で制御したり直接指示したりすることなく、大胆な決定を下すことができます。


彼らの現在の状態を評価し、戦略を成功させるために必要なものと比較することが重要です。 それらの十分がある場合-素晴らしい、最初に！ リソースがない場合、近い将来にリソースを取得することは可能ですか？これについて誰と合意する必要がありますか？ 戦略の実装をいくつかの段階に分割し、新しい段階を徐々に選択するとどうなりますか？ 必要なリソースがすでに利用可能な場合、だれかがそれらを要求しただけですか？



プロセス



UXの体系的な作業は、まず、設計の作成と改善のプロセスを構築し、一般的な生産に統合することです。 適切な設計プロセスでは、生産チェーン全体を考慮する必要があります。 そして、彼女の本当の状態を覚えて、完璧に持って来てください：



  • 製品の開始と新機能 。 正確に社内の誰が、特定の機能だけでなくサービスの開始と再設計を決定しますか？ 市場、競合他社、ターゲットオーディエンスはどのように実施されていますか？ これらの決定は、デザイナーに届くとどのような形でタスクに変わりますか？
  • 開発 。 会社ではどのような開発モデルが受け入れられていますか？カスケード、反復、リーン？ デザイナーが開発者に渡すアーティファクトと、実装サポートはどのように行われますか？
  • 品質保証 。 製品の品質の実装とは、会社が何を意味するのか-基本的な「落ちない」、機能要件とビジネス要件の順守、使いやすさのレベル、設計ガイドラインと元のレイアウトに従うことですか？ 消費者の質に問題がある場合、設計チームは製品のリリースを遅らせる機会がありますか？
  • マーケティング 。 製品はどのような視聴者にどのように動いていますか？ 約束は彼が実際にできることと一致していますか？ 製品自体の視覚的なデザインの品質は、マーケティング資料のデザインと同等ですか？
  • サポート 。 同社はどの段階でどのような形で製品を使用する際の問題を解決するのにユーザーを支援していますか？ サポートに連絡することでユーザーが満足することは何ですか？


開発段階と品質保証段階では、設計者の参加が重要です。 設計チームが単なるパフォーマー以上のものになりたい場合、開始段階で重要です。 また、サービスの設計の一環として、サポートとマーケティングプロセスの最適化は、設計者にとって独立したタスクになります。 理想的な状況では、UXは各ステップの重要なコンポーネントです。



特定の会社の基本原則を理解することも同様に重要です。



  • 製品チームの各メンバーの権限と責任 。
  • 調整手順は、要件と最終成果物から作業の大規模な段階と製品ソリューションまでです。
  • 有効性の評価と監視 -従業員、部門、製品、製品ソリューション。
  • 組織構造 -マトリックス、機能、部門、または結合。 部門、製品、部門、および役割による分離。


創業当初、創業者と最初の従業員の小さなグループがすべての製品の問題を解決できます。 これらは、すべての責任を自分で閉じているジェネラリストです。 しかし、時間が経つにつれて、専門化の必要性があります。 役割の分離、新しい人材の採用、会社全体の成長があります。 これは、組織構造とワークフローが開始から製品の発売に変わることを意味します。 そして、会社が野心的な場合-これは定期的に行われます。



さらに、どの組織にもポリシーがあります。 必ずしも悪い意味ではありません-映画やニュースで説明されている陰謀やデマです。 まず、ワークフローの他の参加者の関心を考慮しています。 これは彼らの支援を受け、全体として、チームとしてより効果的に行動し、内部の矛盾や矛盾を回避するのに役立ちます。



会社の現在のプロセスは常に効果的ではありません。 どこかでこれは組織の若さ、どこかで-困難な遺産や現在の危機、そしておそらくリーダーシップの慣性によるものです。 そして最初は、会社の条件でこの構造に統合する必要があります。 ただし、何かを真剣に変更する場合は、設計に加えてプロセスの再構築に対処する必要があります。 正直なところ、成功する企業は常に変化しているため、プロセスの効率は通常、成長に追いついていません。 これは、最適な設計プロセスを構築した後、注意する必要があることを意味します。



優先順位



会社とその製品には、常に多くの開発パスがあります。 優先順位は、どのものと彼女が選択するかについて話します。 この選択は、市場の一般的な状態とそのニッチに影響されます。 競合他社の位置と行動。 一般的な技術動向; 製品自体の現在の状態と会社のビジネスなど。 UXストラテジストは、会社の進路を知るだけでなく、その理由を正確に理解する必要があります。現在どのような問題がビジネスに直面しており、中長期的には重要です。 結局のところ、設計の助けを借りて解決する必要があるのはまさにこれらの問題です。



優先順位自体は異なります。



  • 収益化に適した市場、製品ソリューション、ビジネスモデルを検索します。 製品は単に頻繁に変更する義務があり、会社が生き残るためには劇的な場合さえあります。 設計者の役割は、製品仮説の研究、声明、テストへの参加、設計の運用上の磨きです。
  • ユーザーベースの成長および/または利益 。 この製品は新しい機能で大きくなりすぎており、新しい配布方法を取得しています。 設計者の役割は、インターフェースの研究、詳細設計、設計およびテストです。
  • ユーザーの保持 。 消費者の質を向上させ、製品の機能を拡張し、主要な使用シナリオを最適化し、ブランドの重要性を高め、市場の差別化を行うことにより、競合他社から離調する必要があります。 デザイナーの役割は、インターフェイスを調査して最適化することです。
  • 製品ポートフォリオのパフォーマンス 。 設計を含め、サービスの作成と開発のプロセスを高速化し、簡素化することが重要になります。 設計者の役割は、ガイドラインと標準を作成することです。
  • ユーザーベース、利益または市場シェアが低下している危機からの製品の結論 -すべて一度にまたはそれらのいずれか。 次の2つのシナリオが考えられます-段階的な進化または急激な変化。 設計者の役割は、インターフェースの研究、詳細設計、設計およびテストです。


企業は一度に複数の優先順位を持っている場合があります。 さらに、会社の製品ごとに優先順位が異なる場合があります。 そして、UXの専門家は、自分が何ができるかを正確に理解する必要があります。 最初に、現在の設計要求を効率的に処理する方法を学ぶことが重要です。多くの場合、それらは非常に単純であり、適切なテクニックが必要です。 しかし、その後、問題の定式化と優先順位の選択への影響について考えることができます。 ここでは、設計チームが行う成果物やアクティビティにビジネスタスクを接続する方法が役立ちます。

あなたの会社はどうですか？



リソース、プロセス、および優先順位のこれらのパラメーターは、UX戦略を検討する企業を検討する際のチェックリストとして使用できます。 彼らは彼女の「肖像」を作り上げます。 また、ほとんどの組織には独自の特性があるため、UXを実装する目的はほとんど同じですが、戦略の実装はほとんど常に一意です。



誰かが幸運であり、会社は当初、デザインが重要な競争上の優位性であり差別化の方法であり、それが真のビジネスドライバーであるという約束に基づいて構築されています。 どこかで、創業者または最初の従業員の一部がデザイナーであるため、最初はインターフェイスが正しく作成されています。 しかし、ほとんどの組織は、ビジネスとして形成された後のUXへの体系的なアプローチを考えているため、UXストラテジストの主要なタスクの1つは、その現状を調査することです。 チェックリストに加えて、企業および製品の成熟度モデルがこの分析に役立ちます。

  2. 最初のデザイナーが雇われるか、能力の始まりを持つ既存のチームの誰かがデザイナーになります。 おそらく、会社の創設者の1人が設計の経験を持っています。

チームとワークフロー



ワークフローが確立されました。 短期および中期計画があります。 タスクを設定して受け入れるための明確な基準があります。



  1. チームは購入または準備を整えます。 可能な場合は常に、強い人や有望な人が常に取られます。 初期の段階では、中程度のものを使って穴を塞ぐことができます-後で彼らはチームとともに成長するか、去ります。
  2. 特定のタスクを設定して受け入れる手順が決定されます。 設計作業の短期および中期計画の原則が構築されます。
  3. 新しい製品コンセプトの作成から既存のコンセプトのサポートまで、一般的なタスクの主要な作業段階が決定されます。
  4. 知識と設計成果物の設計と設計、保存と交換、計画と効率の追跡に最適なツールが選択され、デバッグされます。
  5. アウトソーシング業者は、非生産的で技術的な問題を解決するため、または手が足りない場合に接続します。 長期的な関係が確立されています。
  6. チーム内の専門家のトレーニングおよび専門能力開発のプロセスが開始されます。
  7. 製品の定期的な更新があります。


戦術チーム



デザイナーは製品チームの一員となり、設計作業を他の製品タスクと密接に統合します。







チーム統合



製品チーム全体と連携した効果的な作業。 デザイナーは開発者とマネージャーを理解し、デザイナーを理解します。



  1. 優れたUXの重要性は、特定の製品マネージャーに伝えられています。 彼らはUXチームを理解しサポートし始めます。 既成の解決策ではなく、問題が発生します。 設計者は設計とUXに責任を負いますが、マネージャーと連携して製品の問題を解決します。 新しいマネージャーが来るとき、彼らはまた訓練される必要があります。
  2. チームは信頼性と信用の信用を受けます。 彼らは彼女をより信頼し、常に耳を傾け、すべての設計上の決定が事前に行われます。
  3. デザイナーは、開発者、レイアウトデザイナー、テスターなどのエンドスペシャリストから信頼されています。 彼らは多くの場合、マネージャーを迂回して、現在の仕事の問題を直接解決します。
  4. 設計作業の中長期計画の原則が構築されています。
  5. 製品には標準化された品質管理メカニズムがあります。 設計実装および設計レビューの品質、使いやすさ、設定されたビジネスタスクへの準拠に関する分析。
  6. 共通のスタイルと仕事の原則によって結ばれた一連の製品があります。


設計原則と標準



最新のサービスを維持し、新しいサービスを開始することがより簡単になります。 製品ポートフォリオには独自の顔があります。



  1. 最初の「モデル」製品が登場しました。これは、将来の統一の基礎となるものです。 彼らのスタイル、仕事の論理、消費者の資質は十分です。 このコンセプトにより、スケーリングすることができます。 真空状態でガイドラインを作成して全員に適用することは難しく、大量使用する前にいくつかの製品で実行する必要があります。
  2. 新製品と既存製品の更新は、モデル製品のイメージで行われます。
  3. 作業の論理とモデル製品のスタイルを説明するガイドラインが作成されます。 ガイドラインは、デザイナー、マネージャー、開発者向けのガイドになりつつあります。 ガイドラインは、仕様とレイアウトの形で存在します。
  4. ガイドラインの技術化。 ガイドラインに基づいて、フレームワーク、統一されたコードベースなどが作成されます。 デザインは、開発者が使用する構築ツールに縫い付けられます。
  5. 100個の製品をいくつかのガイドラインに減らします-それらは管理と開発が簡単です。 何百もの製品を管理する代わりに、いくつかのガイドラインを管理しています。 製品グループにすぐに変更を加えますが、個々には変更を加えません。
  6. 製品の作成と開発において会社を導く設計原則。 設計ソリューションを選択するための基準を暗示する高レベルの「戒め」。
  7. すべての製品の設計を更新しました。 製品ポートフォリオのUXは一貫しています。


先見の明のあるチーム



チームは先見の明を持つようになり、製品マネージャーでさえ、製品開発に関する戦略的決定の採用に影響を与えます。







知識の蓄積と交換



ユーザーについて得られたすべての知識と、インターフェースを使用した作業は、製品ポートフォリオ全体の改善に役立ちます。 会社は間違いを繰り返さない。



  1. ユーザーが製品、競合他社、トレンドなどをどのように扱うかについての単一の知識ベースがあります。 分析、ユーザー調査、市場調査および競合調査、フィードバック、CRMシステムからのデータ。
  2. 製品設計の開発を評価するメトリックが設定および追跡されます。 それらはビジネスのパフォーマンスと目標に結びついています。
  3. 製品開発のアイデアは、デザイナーからプロダクトマネージャーまで、下から上へと進みます。


市場差別化のツールとしての設計



同社には独自の視覚言語と対話言語があり、業界全体に影響を与えています。



  1. デザインは市場の差別化の手段になります。
  2. 製品チームのすべてのメンバーは、少なくとも開発者、レイアウトデザイナー、テスター、マネージャーなどのデザインを作成する基本的な能力を持っています。
  3. 企業は視覚的な言語を定義できます。 文体は最初にトレンドをキャッチするか、それらを定義します。
  4. 製品設計は、業界全体に影響を与えます。 インターフェイスソリューションは競合他社によってコピーされます。
