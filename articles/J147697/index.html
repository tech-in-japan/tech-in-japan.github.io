Windowsストアの経済と開発者の機会

「 Windows 8のアプリケーション開発：理由と方法 」の記事で、Windows 8の開発のアイデアに触れました。ご存知のように、Windowsストア-Windows 8アプリケーションストアは、新しいシステムの重要なコンポーネントの1つであり、成功の重要なコンポーネントですWindows 8用のアプリケーションを作成することを決定した開発者の成功と同様に。



Windows 8が開発者に提供する機会を疑う人、またはWindowsストアの意味を完全に理解していない人のために、ストアが提供するアイデアと可能性を確認することにしました。



Windowsストアデバイス



Windows Application Store自体は、すでにクラシックになっているスキームに従って設計されています。



これは、Windows 8のすべてのインストールで使用できる個別のアプリケーションです。開発者がアプリケーションを公開する中央ディレクトリに接続されています。

Windowsストアにはいくつかのセクションがあります。アプリケーション（ゲーム、ソーシャルアプリケーション、エンターテイメントなど）の各カテゴリに対応するセクションで、最初のセクションは編集プロモーションです。

さらに、各カテゴリには独自のプロモーションがあります。 ここでは、たとえば、ゲームカテゴリでどのように見えるかを示します。



あらゆる種類のトップリストによくあるように、新しいアプリケーションと新星のリストがあることがわかります。

もちろん、別のアプリケーションのページには、説明、スクリーンショットなど、必要なすべての情報とツールがあります。





Windowsストアテクノロジー



Windowsストアを介してアプリケーションを配布する場合は、そこで機能する基本的なルールについて知っておく必要があります。

開発者アカウントの登録から始まります。 個人の開発者には年間49ドル、法人には年間99ドルかかります。 同様の構造と同様に、Windowsストアは特定のルールに従って動作し、ストアに表示される前に認定される公開アプリケーションの要件も設定します。

  • Windowsストアアプリ開発者契約
  • Windowsアプリの認定要件


優れた開発者ツールは、 Windowsアプリ認定キット （WACK）です。 これは、公開前であってもアプリケーションの問題を見つけることができる自動ツールです。



無料/有料/試用版


各開発者には、アプリケーションを販売するための2つの主なスキームがあります。有料と無料です。 同時に、プラットフォーム自体と対応するAPI は、制限時間 （ システムによって自動的に制御される ）と機能 （試用モードでのアプリケーションの機能はそれ自体によって決定されます）の2種類のアプリケーションの試用版をサポートします。



広告


当然、今日、アプリケーションに組み込まれた広告は、開発者の主な収入源の1つです。 ユーザーインターフェイス用の適切なSDKとコントロールを使用して、独自の広告エンジンであるMicrosoft Advertisingをサポートしています。 もう1つのことは、アプリケーションで独自の広告表示メカニズムを構築することを誰も気にしないということです。



アプリ内購入


Windowsストアには、 アプリ内 購入メカニズムと対応するAPI が組み込まれています。 Windowsストアの重要な機能は、アプリケーション内で支払いメカニズムを使用できることです。 これにより、通常のツールで作業できるようになるだけでなく、Windowsストアで収益を分配する必要がなくなります。



収益分配


収益分配といえば。 アプリケーションで受け取った収入は25,000ドルを超えませんが、70/30スキームに従って分割されます。開発者は70％を受け取り、Microsoftは30％を受け取ります。 収入が25万ドルを超えるとすぐに、 開発者はアプリケーションから収入の80％を既に受け取ります 。





Windowsストアオーディエンス



そもそも、Windows 8のエディションに目を向けてみましょう。そのうち3つしかありません。

  • 窓
  • Windowsプロ
  • Windows RT


1つ目と2つ目は、従来のPCアーキテクチャ（x86 / x64）を備えたデバイスを対象としています。3つ目は、ARMアーキテクチャを備えたデバイス専用です。 Windows RTといえば、主にタブレットを意味します。

クラシックデスクトップアプリケーションとメトロアプリケーションの両方がWindowsとWindows Proで動作します。 同時に、「Designed for Windows 7」というラベルが付いているコンピューター、または事実上このステータスに該当するコンピューターでは、Windows 8をインストールできるだけでなく、Windows 8で完全に動作することもできます。

Windows RTは少し異なります。 このシステムはARMアーキテクチャで実行され、既存のデスクトップアプリケーションは実行できません。 もう1つは、Metroアプリケーションの場合、MicrosoftはWindows / Windows ProとWindows RTの両方のタイプのシステムでの実行を保証することです。 C ++で記述され、ネイティブコードにコンパイルされたアプリケーションでも、適切なアセンブリを作成し、Windowsストアに公開するときにアプリケーションパッケージに含めるだけで済みます。 繰り返しますが、これはMetroアプリケーションでのみ機能します。 さらに、Windows RTでは、特定の企業シナリオを除き、Windowsストアからのみアプリケーションをインストールできます。

したがって、Metroスタイルでアプリケーションを開発する場合、デバイスの種類に関係なく、Windows 8をインストールしたり、Windows 8を搭載したデバイスを購入したりする潜在的なユーザーおよび購入者になります。

Windows 8で、マイクロソフトがタブレットから巨大なモニター（もちろんタッチ）を備えたデスクトップまで、あらゆる種類のデバイスでタッチスクリーンを使用するというアイデアを積極的に推進していることは周知の事実です。 このアイデアは、デバイスメーカーと多くのユーザーに非常に受け入れられています。 もちろん、タッチインターフェイスはすべてのシナリオに適しているわけではありません。 そのため、すべてのメトロアプリケーションでキーボードとマウスの操作をサポートすることが非常に重要です。 当然、Windows 8およびWindows 8 Proには通常のデスクトップがあり、通常のアプリケーションはすべて機能します。 ところで、Windows RTでも、デスクトップモードで動作するMS Officeの特別なARMバージョンが提供されます。
