Windowsのスタンプ：Vagrantを使用したHyper-VでのWindows仮想マシンの自動展開（パート2）

前のパートでは、Windows VMの展開の目的とホストシステムの準備方法について検討しました。 今日は、プロセスの最も大規模な部分の技術的な実装、つまりVagrantのベースボックスの準備に焦点を当てたいと思います。 必要なソフトウェアをすべてインストールしたら、モデルボックス（マシンを展開するテンプレート）を準備します。ボックスは、VHDディスクファイルとメタデータを含む通常のtarアーカイブです：virtualka設定とサービス情報。 したがって、スペースを節約して展開を加速するために、新しいVMの作成、更新、構成、圧縮によってイメージの準備を開始する必要があります。 Hyper-Vマネージャーで新しい仮想マシンを作成する方法、システムをインストールして更新をロールする方法については説明しません。 ニュアンスのみに焦点を当てます。
  • 第一世代の車（Gen1）を構築します。 これは、Vagrantの現在のバージョン（1.7.2）がハードディスクのパラメーターを誤って設定しているという事実によるものです（Gen2には「IDE」オプションがなく、「VHD」に置き換えられます）。現在、Gen2には関与しないことをお勧めします。 ただし、本当に必要な場合は、ファイルplugins / providers / hyperv / scripts / import_vm.ps1を修正できます。
  • ハードディスクの1つのファイルのみを使用してください。そうしないと、ボックスが上昇しません（以下を参照）。
  • 不要なソフトウェア（Windowsコンポーネントを含む）をボックスにインストールしないでください。 これにより重くなり、しばらくすると古いバージョンが存在する可能性が高くなります。 プロビジョニングの段階で、これを後で行う方がより正確です。
  • Boxstarterが正常に機能するには、Powershellの制限を無効にする必要があります。
    Set-ExecutionPolicy -ExecutionPolicy Unrestricted
  • デフォルトでは、VagrantはSSH（マシンに後者をインストールする必要があります）を介してマシンにアクセスしますが、最近のバージョンではWindowsネイティブのWinRMプロトコルを使用して動作することを学びました。 必要なことは、WinRMを構成することだけです。
    winrm set winrm / config / client / auth '@ {Basic = "true"}'

    winrm set winrm / config / service / auth '@ {Basic = "true"}'

    winrm set winrm / config / service '@ {AllowUnencrypted = "true"}
  • 私は通常、不要なサービスを無効にし（スタートアップモードを手動に設定します）（印刷スプーラーは同じです）、ディスクインデックス作成と毎週の自動最適化も削除します。ハイパーバイザー自体によってストレージ層を最適化します。
すべての更新プログラムをインストールした後、Windowsは通常12〜15ギガバイトに「膨張」しますが、これは明らかに高速クローン作成には少し大きいです。 これは、システムがパッケージインストーラーとそのコンポーネントの一部のディストリビューションをキャッシュするという事実によるものです。 幸いなことに、これらはすべて簡単にクリーンアップされます。 それで、VHDファイルが占めるボリュームを減らす方法は？
  • スワップファイルのサイズを小さくし、このサイズを固定します。
  • すべてのログを消去します。
    wevtutil el | Foreach-Object {wevtutil cl "$ _"}
  • 不要なWindows Updateコンポーネントを削除します。
    Dism.exe / online / Cleanup-Image / StartComponentCleanup / ResetBase

    Dism.exe /オンライン/ Cleanup-Image / SPSuperseded
  • Windowsコンポーネントのインストーラーを削除します。 Get-WindowsFeatureコマンドを実行すると、一部のコンポーネントがインストール状態=利用可能になっていることがわかります。 これは通常、これらのコンポーネントをインストールできることを意味し、このためには、ローカルディスクに直接格納されたキャッシュで十分です。 したがって、次のコマンドを実行してこのキャッシュを削除することは理にかなっています。
    Get-WindowsFeature |

    ？ {$ _。InstallState -eq 'Available'} |

    Uninstall-WindowsFeature -Remove
  • 後で便利になるコンポーネントがある場合は、そのままにしておくことをお勧めします。それ以外の場合は、後続のインストール時にwim-fileへのパスを指定する必要があります（-Sourceパラメーター）。 ただし、心配する必要はありません。このファイルはシステムのインストール元のISOファイルにあり、ローカルおよびネットワークリソースの両方から使用できます。 さらに、一部のコンポーネントはインターネットから安全にインストールされます-Powershellは必要なファイル自体をダウンロードします。
  • 必要に応じて、ディスククリーンアップ-一時ファイルからシステムを自動的にクリーンアップするプログラムをインストールできます。 ただし、このためには、Desktop-Experienceロールを追加する必要があり、アンインストールによりsysprepで問題が発生します。 そのため、2つのオプションがあります。この機能をそのままにして、それ自体が場所を食い尽くし、レジストリを重くします（インクや手書きのような必須だが余分なジャンクを引っ張るから）、またはsysprepを拒否します。ドメイン内のマシン。
  • これらすべての操作の後、動的VHDファイルのサイズは変わりません。 圧縮するには、まずデフラグして、すべてのファイルをディスクの先頭に移動する必要があります。
    Optimize-Volume -DriveLetter C -Defrag
  • 最適化後、SysInternalsの一部であるsdeleteコマンドを使用してすべての空き領域をリセットすることが望ましいです。
    sdelete -zc：
    これにより、後でより効率的に圧縮できます。
その後、仮想マシンをオフにして、そのVHDでコンパクト操作を開始できます。 最終的なサイズはさまざまですが、通常は5〜7 GBのレベルです。 tarとGZipを使用してさらに圧縮するため、最終的なボックスサイズはさらに小さくなります。 ただし、ドメイン環境でこのマシンを使用する予定の場合は、SIDの重複が発生しないように準備する必要があります。 これを行うには、Sysprepユーティリティを使用します。



Sysprepウォーキング

最も一般的なケースでは、sysprepを使用してクローン作成のために仮想マシンを準備することは、かなり一般的な手順です。 これをGUIとして使用できます。
C：\ Windows \ System32 \ Sysprep \ sysprep


コマンドラインは、私たちの好みにより合っており、追加のパラメーターを指定することもできます。
C：\ Windows \ System32 \ Sysprep \ sysprep / generalize / oobe /モード：vm / shutdown
その後、プログラムはすべての必要な操作を実行し、車を出します。 次回システムの電源を入れると、前述のOOBE（Out-of-Box Experience）になります-最初に電源を入れたときに新しく購入したラップトップが入っている状態です：鉄のインベントリが実行され、ローカル管理者パスワード、ライセンスキー、コンピューター名などが要求されます。 すべては何もありませんが、キーボードからこれらすべてを手動で入力する必要があります。 幸いなことに、これはすべて簡単に自動化されます。 これを行うには、応答ファイルunattend.xmlを準備し、起動されるsysprep.exeファイルの隣（C：\ Windows \ System32 \ Sysprepフォルダー）に配置するか、コマンドラインパラメーターとしてそのパスを指定します。これにより、外部にファイルを配置できます最後のボックスに表示されないようにします。 これについては詳しく説明しません。このためには、Windowsインストールイメージ（より正確には、そこからのinstall.wimファイル）とWindows Assessment and Deployment Kit（ADK）（Windowsシステムを含むDeployment Toolsアイテムのみをインストールする必要があります）画像マネージャー）。 ディストリビューションへのリンク、指示、使用例は、記事の次の最終部分の適切なセクションにあります。

重要なニュアンス：
  • Desktop-Experienceコンポーネントをインストールしてからアンインストールした場合、Sysprepはエラーで失敗します。 上で述べたように、このコンポーネントをインストールしたままにするか、インストールしないかの2つのオプションがあります。 詳細については、 ブログの Matt Rock（Boxstarter著者）を参照してください。
  • ボックスを準備するとき、多くの場合、何かを変更する必要があります。これにより、ボックスがOOBE状態になり、すべての設定が応答ファイルに従って渡され、必要な変更が行われ、システムがSysprepによって再度実行されます。 これが2回しか発生しない（つまり、3回のsysprep実行）ことに注意してください。 これは、Microsoftのライセンス機能によるものです。 原則として、レジストリのunningな編集によってこれを行うことができますが、これは本番環境ではなく、一時的なマシンまたはテストマシンに対してのみ行うことができます。したがって、必要に応じて、さらに作業するためにイメージを単純に保存し、必要に応じてコピーを作成し、新しいバージョンを個別に準備することをお勧めします。 したがって、各仮想マシンは、その存続期間全体にわたって一度に「切り離され」ます。
  • コンピューター名を設定しない方が良い-それはランダムに生成されるので、複数のマシンが同時に発生している場合の混乱と競合を回避できます。 さらに、持ち上げるときにプロビジョニングの段階で、浮浪者自身が簡単にインストールできます。
  • ローカル管理者の複雑で安全なパスワードを設定します。 もちろん、将来的には変更する（またはブロックする）ことをお勧めします。 このアカウントは、プロビジョニング段階でマシンを上げるときに必要になります。Vagrantがソフトウェアをインストールし、システムを更新し、必要なすべての設定を行うのはその下です。
  • ハイパーバイザー自体のマシン名（Hyper-Vマネージャーで呼び出される）を短くシンプルな名前に変更することをお勧めします-これは、拡張ボックスからのマシンの名前です。 同僚が私のトライアルボックスを拾い上げた結果、「Winda 2k12 r2 x64 syspreped with bloody winrm settings version 111 test 222 yo」というタイプのマシンを手に入れたとき、厄介な状況がありました。


箱詰め

Sysprepが車を出した後、仮想マシンをボクシングファイルにパックします。 一般に、Vagrantにはこのための組み込み機能（プロビジョニングコマンド）がありますが、現在のところ、Hyper-Vでは機能しません。 だから私たちは自分の手でボクシングをパックします-恐れることは何もありません、これは非常に簡単に行われます：
  • マシンのハードウェア構成がチェックされます：プロセッサコアの数、メモリサイズ、ブート優先順位。 これらのパラメーターは、新しく展開されたマシンにあります（もちろん、後で変更することを妨げるものはありません）。
  • ハードディスクのすべてのイメージ（スナップショット）が削除されます。
  • 準備完了マシンは、ディスク上の別のフォルダーにエクスポートされます。 これは、ハードディスクファイルとすべてのメタ情報の隣に配置する必要があります。
  • スナップショットファイルのフォルダーは、仮想マシンのあるフォルダーから削除されます。WagrantプロバイダーのHyper-Vは引き続き使用しません。
  • テキストファイルmetadata.jsonもそこに作成されます。 少なくとも、そこに少なくとも1つのプロバイダーを定義する必要があります（この場合、Hyper-V）。 このケースのサンプルファイル：
    {

    「名前」：「Windows2012R2」、

    「説明」：「無期限のプロビジョニングでsysprepされたWindows 2012 R2ボックス」、

    「プロバイダー」：「hyperv」

    }
  • あとは、ディレクトリの内容から.boxファイルを作成するだけです。 このために、tarが使用されます（リポジトリから無料の7-Zipまたはbsdtarを使用できます。Vagrant1.7.2に付属のバージョンは機能しません）。 重要：アーカイブには、マシンがエクスポートされたフォルダーの内容のみが含まれ、親フォルダーは含まれません。そうでない場合、ボックスは認識されません。 つまり 内部のルートには、「仮想ハードドライブ」ディレクトリと「仮想マシン」ディレクトリ、およびmetadata.jsonファイルが必要です。 結果のファイルは、gzip（再び7-Zipを使用できます。主なことは、既に.tarファイルで別の反復で行うこと）またはLZMA（より良い圧縮を提供しますが、コピーに費やされた時間がより長く費やされます） gzip、unzip）、拡張子をboxに変更します。 はい！ ボクシングの準備ができました！


Vagrantがそれを自分自身にコピーして表示するには、ファイルを使用してディレクトリから直接addコマンドを使用して追加する必要があります。
vagrant box add --provider hyperv --name Win2012R2x64 Win2012R2.box
すべてがうまくいくと、ボックスが使用可能になり（vagrantはそれを％VAGRANT_HOME％にコピーして解凍します）、vagrant box listコマンドの出力に表示されます。 同僚がアクセスできるようにするには、共有ネットワークドライブに置くことができます（ボックスを追加するコマンドは、vagrant box add --name Win2012R2x64 D：\ Path \ To \ Win2012R2x64 \\ fileserver.domain.local \ boxs \ win2012r2の形式になります。 box）またはWebサーバー：local（vagrant box add --name Win2012R2x64 http：//devops.local/boxes/win2012r2.box）またはexternal。 たとえば、 http： //www.vagrantbox.es/またはHashicorp Atlasで 。
