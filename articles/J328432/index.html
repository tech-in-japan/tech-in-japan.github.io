iOSビデオチャットアプリをApp Storeで公開した方法







この記事では、iOS用の最初のWebRTCアプリケーションを構築してApp Storeに配置した方法、対処しなければならないこと、およびその結果について説明します。



なぜ



WebRTCメディアサーバーと、サーバーを操作するための3つのSDK（Web、Android、iOS）のソーイングに取り組んでいます。



各SDKにはいくつかの例があります。 たとえば、iOS SDKの場合、ソースがここから入手できるアプリケーションの例が11あります 。



開発者はサンプルを参照し、サンプルが機能することを確認してから、ソースからサンプルを収集し、ストリーミングビデオを操作するためにアプリケーションでSDKの変更/埋め込みを開始します。



「デモを見て-組み立てて-実装する」というようなアプローチは、ほぼすべての場所でうまく機能しますが、Appleではうまくいきません。 App Storeで公開されるまで、アプリケーションをテストしたい人へのリンクをリセットすることはできません。



おそらく、iOSのすべての開発者と発行者が長い間扱ってきた単純な真実をここに書いているのかもしれませんが、私たちにとってこれは小さな驚きでした。



App Storeに公開しないと、デバイス（iPhone）のUDIDがリストされるまで誰もアプリケーションをインストールできません。



つまり 誰かがアプリケーションをテストできるようにするには、まず誰かにデバイスのUDIDを提供するよう依頼し、 developer.apple.comにリストし、 Xcodeでアプリケーションを再構築し、それを出してから、この特定の人にダウンロードしています。



この配布方法はadhocと呼ばれます。 原則として、これは便利です;チーム内で、すべての開発者とテスターを一度入力してください。 しかし、テストデモアプリケーションへのアクセスを全員に提供する必要がある場合、これは非常に不便です。



その結果、リソースから構築する時間を無駄にすることなく、誰もがそれをテストし、それがどのように機能するかを理解する機会を与えるために、App Storeでデモアプリケーションを公開することが決定されました。



アプリ



パイロットアプリケーションとして双方向ストリーミングを採用しました。次のようになります。









この単純なアプリケーションを使用すると、次の3つのことができます。



  • サーバーに接続します。
  • カメラからビデオストリームをキャプチャし、サーバーに送信します。
  • サーバーからビデオストリームを取得し、右側のディスプレイで再生します。


このアプリケーションは偽のビデオチャットで、 iPhoneからビデオストリームを送信して、すぐに別の人のビデオストリームを見ることができます。 つまり 必要に応じて、アプリケーションはChatrouletteのようなビデオチャットに簡単に変換できます。



アプリケーションのソースコードはこのリンクから入手でき、依存関係としてWCS iOS SDKが必要です。 描画リソースを使用したアプリケーションおよび他のすべてのサンプルのビルドプロセスについては、SDKページで説明しています。 したがって、アセンブリに焦点を当てるのではなく、App Storeでの出版物の説明にすぐに進みます。

バッジにアイコンが表示されたのは、 彼らは最大のアイコンを少し異なるものにすることを決めました。このため、私たちのアプリケーションは最初に拒否されましたが、それについては後で詳しく説明します。

フォームへの入力は複雑なプロセスではないため、 アップルのWebサイトでフォームに慣れることをお勧めします。

4. iTunes Connectでアセンブリをダウンロードします



これは、アプリケーションをiTunes ConnectにアップロードするためにApplication Loaderを使用して実行できます。



1）Xcodeメニューを開き、 Open Developer Tool / Application Loaderをクリックします

2）Apple IDでログインします

3）[アプリの配信]をクリックします。

4）ipaアプリケーションを選択し、 「送信」をクリックして、ダウンロードが完了するのを待ちます。



5.ダウンロードしたアプリケーションをiTunes Connectでモデレートするために送信します



ここでは、3つの質問に答える必要があります。 少しグーグルで、3つの答えすべてでNoと入力しました。

アプリは暗号化を使用するように設計されていますか、それとも暗号化が含まれているか組み込まれていますか？ いや
アプリにサードパーティのコンテンツが含まれている、表示されている、またはアクセスしていますか？ いや
このアプリは広告識別子（IDFA）を使用していますか？ いや


実際、このアプリケーションでは暗号化、つまりAESとHTTPS / Websocketを積極的に使用しています。 しかし、Googleはこれはカスタム暗号化に関するものであり、AESの標準技術であるHTTPSを超えていると答えました。 したがって、他の2つの質問に関しては、彼らはNoを設定します。



その結果、アプリケーションは適度になり、ビジネスに取りかかりました。

次のリダイレクトは4日後に発生し、一見するとよりわかりにくいものになりました。



アプリは、Info.plistのUIBackgroundModesキーでオーディオのサポートを宣言していますが、永続的なオーディオを必要とする機能は含まれていません。



オーディオキーは、音楽プレーヤーやストリーミングオーディオアプリなど、バックグラウンドでユーザーに音声コンテンツを提供するアプリで使用するためのものです。 アプリを修正して、アプリがバックグラウンドにある間に音声コンテンツをユーザーに提供するか、UIBackgroundModesキーから「オーディオ」設定を削除してください。



ここでは、アプリケーションがUIBackgroundModesでオーディオサポートを宣言しているが、バックグラウンドでのオーディオ再生を必要とする機能がないことを述べています。 これは、アプリケーションの実行中に別のアプリケーションに切り替えるか、ホームボタンを押すと、オーディオの再生が続行されることを意味します。これはUIBackgroundModesではオーディオと呼ばれます 。 音楽の再生などに関連する機能。



Appleモデレーターと議論せず、双方向ストリーミングアプリケーションがオーディオ/ビデオストリームをバックグラウンドで再生できることを証明しないために、この機能の宣言をオフにしました。



4. UIBackgroundModesでの作業に関する同様の要件は、VoIP機能に関するものでした。 私たちのアプリケーションは、バックグラウンドでVoIPのサポートを宣言し、モデレーターはVoIPがない場合にアプリケーションを疑っていました。



アプリはInfo.plistのUIBackgroundModesキーでVoIPのサポートを宣言していますが、Voice over IPサービスは含まれていません。



この問題を解決するには、アプリを修正してVoIP機能を追加するか、UIBackgroundModesキーから「voip」設定を削除してください。



「アプリを修正してVoIP機能を追加する」という文言は、あまり好きではありませんでした。 VoIP機能があり、UIBackgroundModesで動作する必要があることをモデレーターに証明し、表示する必要があります。 そのため、プロセスを高速化するために、UIBackgroundModesでVoIPをオフにし、アプリケーションを再構築して、モデレートのために送り返しました。
