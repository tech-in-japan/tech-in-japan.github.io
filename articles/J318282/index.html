従来のJavaScriptベンチマークに関する真実





おそらく、JavaScriptはソフトウェア開発で最も重要な最新テクノロジーであると言っても十分でしょう。 プログラミング言語、コンパイラー、および仮想マシンを研究した多くの人々にとって、JavaScript言語の構造という点でその優雅さがすべてコンパイルに関してまだ十分に最適化されておらず、素晴らしい標準ライブラリを誇ることができないという事実は依然として驚くべきことです。 対話者が誰であるかに応じて、JavaScriptの欠陥を数週間にわたってリストしても、聞いたことがないような奇妙なものを見つけることができます。 しかし、明らかな欠陥にもかかわらず、今日、JavaScriptはWebの重要な技術であり、サーバー/クラウド領域での優位性に向かっています（Node.jsのおかげです）。また、モノのインターネットにも浸透しています。



質問は-なぜJavaScriptはそんなに人気があるのですか？ 包括的な答えがないのではないかと心配しています。 今日、この言語を使用する多くの理由がありますが、最も重要なのはおそらく、JavaScriptと無数のリソースを中心に構築された巨大なエコシステムです。 しかし、これはすべてある程度の結果です。 そして、なぜこの言語は最初に普及したのですか？ あなたは言うことができます：なぜなら彼は長い間、ウェブの共通語だったからです。 しかし、それはかなり前のことであり、開発者はJavaScriptを情熱的に嫌っていました。 振り返ってみると、JavaScriptの人気は2000年代後半に始まりました。 ちょうどその頃、JavaScriptエンジンはさまざまな負荷ではるかに高速に動作し始め、おそらくこの言語に対する多くの人々の態度に影響を与えました。



当時、すべてのJSマイクロベンチマークの 先駆けであるApple SunSpiderからMozilla KrakenとGoogle V8まで、いわゆる伝統的なJavaScriptベンチマークが速度の測定に使用されていました。 その後、GoogleベンチマークはOctaneに取って代わり、AppleはJetStreamをリリースしました。 これらの従来のベンチマークは、JavaScriptのパフォーマンスを世紀の変わり目に誰も予想しなかったような高さにするために、信じられないほどの努力をしてきました。 数千の加速が記録され、突然<script>



使用が悪魔とのダンスでなくなり、クライアント側での計算の実行が可能になっただけでなく、奨励されました。







出典： V8およびWeb Assemblyを使用した高度なJSパフォーマンス 、Chrome Developer Summit 2016、 @ s3ththompson



2016年には、すべての（重要な）JSエンジンが信じられないほどのパフォーマンスを達成し、Webアプリケーションはネイティブのものと同じくらい高速になりました（または高速になります）。 エンジンには、高度に最適化されたマシンコードの短いシーケンスを生成する洗練された最適化コンパイラが付属しています。 これは、さまざまなタイプ/フォームで利用可能な統計に応じて、各操作（プロパティ、バイナリ操作、比較、呼び出しなどへのアクセス）のタイプ/フォーム（タイプ/形状）を慎重に選択することで実現されます。 これらの最適化の大部分は、SunSpiderやKrakenなどのマイクロベンチマーク、およびOctaneやJetStreamなどの統計パッケージによって決定されました。 asm.jsやEmscriptenなどのJavaScriptベースのテクノロジーのおかげで、今日ではJavaScriptで大規模なC ++アプリケーションをコンパイルし、ダウンロードやインストールなしでブラウザーで実行できます。 たとえば、 AngryBotsでは簡単にオンラインで簡単にプレイできますが、以前はAdobe FlashやChrome PNaClなどの特別なプラグインが必要でした。



これらすべての成果の大部分は、マイクロベンチマークとパフォーマンス測定パッケージの存在、および従来のJSベンチマーク間で生じた競争のおかげで可能になりました。 SunSpiderについては何でも言うことができますが、それなしではJavaScriptのパフォーマンスが今日とほとんど変わらないことは明らかです。



しかし、十分な賞賛、それはコインの裏を見てみる時間です。 すべての測定テストは、マイクロベンチマークであれ大きなパッケージであれ、時間が経つにつれて無関係になる運命にあります！ なんで？ ベンチマークは、あなたがそれで遊んで始めるまで、あなたに何かを教えることができるからです。 特定のしきい値を超える（または超えない）とすぐに、特定のベンチマークで得られる最適化の一般的な適用性は指数関数的に低下します。



たとえば、Octaneをプロキシとして使用して、実際のWebアプリケーションのパフォーマンスを測定しました。 そしてしばらくの間、彼はこの仕事に非常にうまく対処しました。 しかし、今日では、Octaneと実際のアプリケーションでの時間の分布は大きく異なるため、Octaneをさらに最適化しても、実際のアプリケーション（Node.jsを含む）が大幅に改善されることはほとんどありません。







出典： Real-World JavaScript Performance 、BlinkOn 6 Conference、 @ tverwaes



JetStreamとOctaneの最新バージョンを含むJavaScriptパフォーマンスを測定するための従来のベンチマークのすべてが時代遅れになっているようになったことがますます明らかになったため、プロファイリングとV8およびChrome用の新しいインターセプターを追加して、実際のアプリケーションを測定する新しい方法を探し始めましたトレース。 また、スクリプトの実行、ガベージコレクション、コンパイルなど、サイトの閲覧により多くの時間を費やすことを理解するためにツールを使用しました。 研究結果は非常に興味深く、予想外でした。 前の図からわかるように、Octaneを起動すると、70％以上の時間がJavaScriptとガベージコレクションに費やされますが、JavaScriptを使用したサイトの閲覧にかかる時間は常に30％未満であり、ガベージコレクションにかかる時間は5％以下です。 しかし、解析とコンパイルには多くの時間がかかり、Octaneについては言えません。 そのため、JavaScriptの実行を最適化するための相当な努力により、Octaneのオウムが大幅に増加しますが、サイトの読み込みはそれほど速くありません。 さらに、JavaScriptの実行を最適化する趣味は、コンパイルに時間がかかるため、実際のアプリケーションのパフォーマンスを損なう可能性があります。または、コンパイル、IC、ランタイムを拡張する追加のパラメーターを追跡する必要があります。







JavaScript や DOMを含む、ブラウザ全体のパフォーマンスを測定しようとする別のベンチマークパッケージがあります： Speedometer 。 彼は、さまざまな人気のあるWebフレームワークに実装されたシンプルなTodoMVCアプリケーションを起動することで、より現実的に測定にアプローチしようとします（今日は少し時代遅れですが、新しいバージョンがすでに作成されています）。 パッケージには、新しいテスト（角度、エンバー、反応、バニラ、フライト、バックボーン）が含まれています。 今日まで、スピードメーターは実際のアプリケーションのパフォーマンスを測定するための最も好ましいプロキシの役割のようです。 ただし、これは2016年半ばのデータであり、Webで使用されるパターンが開発されると、すべてが既に変更される可能性があることに注意してください（たとえば、ICシステムをリファクタリングしてコストを大幅に削減し、パーサーを再設計します ）。 上記の状況はサイトの閲覧にのみ関係しているようですが、実際のNode.jsアプリケーションの場合、従来のピークパフォーマンスのベンチマークはプロキシロールに対してあまりうまく機能しないという非常に説得力のある証拠を受け取りました。

ガベージコレクションは有害と見なされます



特定のケースに加えて、SunSpiderには別の基本的な問題があります。それは、合計実行時間です。 現在、適切なIntelハードウェアでは、V8エンジンは約200ミリ秒でベンチマーク全体を実行します（新しいスペースの生きているオブジェクトと古いスペースの断片化に依存します）が、ガベージコレクションの主要な一時停止は簡単に30ミリ秒に達します。 また、インクリメンタルマーキングのコストは考慮されていません。これは、SunSpiderパッケージの合計実行時間の10％以上です。 したがって、ガベージコレクションのためにエンジンの速度が10〜20％低下したくない場合は、SunSpiderの実行中にエンジンが開始されないようにする必要があります。
