コサックのリバースエンジニアリング、パート3：LANの指ぬき







2016年の終わりの庭で、ついにファンの間で熱狂の嵐が起こり 、「コサック」の3番目の部分が出てきました...しかし、私はまだ最初の部分のネットワークコンポーネントの奇妙な間違いに悩まされていました。 奇妙なことは、ローカルネットワーク上でゲームを作成するとき、通常2人しかゲームを起動できないことでした。 プレイヤーが3人いると、負荷インジケータは非常にゆっくりと成長し、4人からは全体で0％のままでした。 さあ、調査を始めましょう！



エラー表示



問題の症状はいくつかあります。 ゲームルームに表示される「最大ping」値が高すぎて、プレーヤーの数に比例して増加します。 すべてのプレイヤーが同じスイッチに接続され、通常のicmp pingが一貫して1ミリ秒未満を生成しますが、ゲームルームでは最大350ミリ秒の遅延が表示されます。 部屋にプレイヤーが2人しかいない場合、「最大ping」は最初に約90ミリ秒で、1秒あたりに約10ミリ秒に低下します。



2番目の症状は、「 プレーヤー名で直接接続が確立されていません」という警告です。 私たち3人は約50％の場所でそれを獲得する機会があり、部屋に4人のプレイヤーがいる場合、常に表示されます。 Ctrlキーを押しながら「開始」をクリックすることでこの警告は無視できますが、これはゲームの側面からの接続品質の「認識の問題」を示しています。



3番目の症状は、ダウンロード速度が遅いことです。 すべてのプレイヤーが[開始]をクリックすると、割合のインジケーターがゲームのホストに表示されます。 〜8％ずつ増加し、100％に達し、その後のみゲームを開始できます。 このインジケータは、ランダムに生成されたカードのファイルをホストからプレーヤーに転送する進行状況を主に示しており、通常のカードのこのファイルのサイズは約3.5 MBであるため、ギガビットLANでの5秒のダウンロードでも問題になります。 また、3人のプレーヤーを「ダウンロード」するのにかかる時間は、ゲームとともにフォルダー全体をコピーできます。



最後から開始



あなたの許可を得て、私はあなたにこの逆転の始まりの詳細と生じた問題を解放します。 ネットワークコンポーネントを分解するのは楽しいだろうと思って間違えたとしか言えません。 ポインターを介した関数の呼び出し。 マルチスレッド。 DirectPlayを使用して、sendto（）とrecvfrom（）を同時に使用します。 デバッグ情報は言うまでもなく、この恐竜に関するわかりやすいドキュメントの欠如。 私のカンフーはここでは明らかに不十分でした。



したがって、通常どおり動作します。 みんなのお気に入りの最も美しいプログラムを開き、Alt + Tを押して「直接接続なし」を探します。 文字列を含むメモリの領域を見つけ、 xrefを介して直接ポインターに移動し、次にサイズが32キロバイトの体積関数に移動します。 とりわけ、ゲームルームのインターフェイスの要素はその中で初期化され、メッセージが処理され、表示される前に行が構成されます。 彼女をLanLobby（）と呼びましょう。 以下に、プレーヤーに警告を表示して「開始」キーを無効にするかどうかを決定するアルゴリズムを示します。



注：記事に示されている逆アセンブラーのリストは、読みやすくするために処理されています。



  1. SomeIteration（）と呼ばれる小さな関数が呼び出されます。 今後、彼女の体内で同様の関数への呼び出しがあり、ImportantIteration（）と呼びます。 後者が少なくとも1回ゼロを返す場合、SomeIteration（）はすぐにループを終了し、ゼロも返します。 この場合、移行は行われず、警告を受けるリスクがあります。



  2. 次に、Ctrlボタンのステータスがチェックされます。 クランプされている場合、「開始」ボタンは無効になりません。



  3. 「開始」ボタンの無効化。 hStartButton変数は、「addVideoButton」という名前の関数の結果で上記で初期化されます。



  4. タイマーは2秒経過したかどうかチェックされます。 PreviousTick変数の上に、現在のGetTickCount（）値が、ルーム内のプレーヤーの数が変わるたびに割り当てられます。 それから2秒未満が経過すると、移行が行われ、警告は表示されません。



  5. 最終的に、警告行は、ゲームルームのステータスバーとして画面に表示するための行にコピーされます。



結論：プレイヤーを部屋に連れて行った後、ゲームは接続を確立するのに2秒かかります。 その後、ImportantIteration（）がまだゼロを返す場合、直接接続の欠如に関する警告が表示されます。



つまずきブロック



リバースのこの段階では、ImportantIteration（）で何が起こっているのかまだ理解できていなかったので、ローディングインジケーターラインのレイアウトを見つけて、そこから作業を続けることにしました。 以前、「最大ping」計算アルゴリズムを分析する無駄な試みで、数値変数を持つすべての文字列が最初にsprintf（）を介してコンパイルされ、次にターゲット文字列に配置されることに気付きました。 フォーマット文字列の構文が与えられたら、テキスト「%%」を探し、LanLobby（）で一致するものを見つけます。 これは、コードの断片がどのように見えるかであり、ダウンロードインジケーターのフィールドにパーセンテージの数字を表示するか、ゲームの「準備」を示すチェックマークを表示するかを決定します。









GetLoadPercentage（）関数の結果は、100未満の場合、1つずつ画面に転送されることがわかります。 2番目のブランチは目盛りを描画する必要があります。 この関数で何が計算されるのだろうか？ GetLoadPercentage（）は、プレーヤーのデータを使用して配列を通過するループで構成されています。



そして、ここで重要な反復（）は問題を解決します。 はい、名前と間違われていません。 算術演算を行うと、ImportantIteration（）は0x00から0x0Cの範囲でプレーヤーのダウンロードステータスを返します。つまり、この「スケール」には12ステップしかありません。 パーセンテージインジケータが8％ずつ増加する理由が明らかになりました。



ImportantIteration（）が何をすべきかを理解したので、それがまだ使用されている場所を見てみましょう。


接続品質をチェックし、関心を計算するときに知っている呼び出しに加えて、ImportantIteration（）は2つの場合に呼び出されます：接続不良についての警告が定式化されるとき-ステータスバーのプレーヤーのリストをコンパイルするために使用され、すべてのプレーヤーが準備ができているかどうかをチェックするために使用されます。 後者は、すべてのプレイヤーを通過し、ロードの度合いを0x0Cと比較する小さなサイクルの関数で実行されます。 少なくとも1人のプレイヤーが少ない場合、サイクルはゼロを返します。 最後の関数の結果は、ホストの[スタート]ボタンのアクティブ化とゲームを開始する機能に直接影響すると想定できます。



解決策



したがって、最も簡単な解決策は、ImportantIteration（）の結果に応じて、ロジックを修正することです。 幸いなことに、すべての場合において、移行は肯定的な結果で実行されます。つまり、すべての移行を条件付きから無条件に変更するだけです。 dmcr.exeファイルのいわゆる「Windows 7バージョン」の場合、パッチ全体は次のように説明できます。 



 |  |  |  ---------+-----------+-----------+----------------------------- 0x00CEEA | 0x7D | 0xEB |      0x098792 | 0x0F 0x8D | 0x90 0xE9 |     100% 0x09C389 | 0x0F 0x85 | 0x90 0xE9 |


そして、すべての変更はローカルネットワーク上のゲームのゲームルームのロジックのみに関係し、ネットワークメッセージ自体は並行して発生し、それに依存しないため、副作用を恐れることはできません。 ランダムに作成されたカードのファイルは、ほぼ瞬時にネットワーク上に配布され、部屋を使用すると、遅滞なくゲームを開始できます。 ここでは手を洗うことができますが...



の中には何がありますか？



結局のところ、ゲームルームがプレイヤーの読み込み状態をどのように判断するのかが不思議です。 今日の記事の主人公、ImportantIteration（）をご覧ください。









ここで、いくつかの興味深いことがわかります。



  • 2つのパラメーターの1つはecxレジスターを介して渡されます。
  • 必要なデータはすべて互いに隣り合ってメモリ内にあるので、なぜそれらのアドレスを保存する必要があるのでしょうか？ ポインターPointer_AとPointer_Bの代わりに、Pointer_Aと（Pointer_A + 4）があります。
  • ポインターパラメーターPlayersDataStructが指すメモリ領域は、並列スレッドで書き込まれ、LanLobby（）で何が起こっているかに依存しません。


示されたメモリ領域で何が起こっているかを理解しようとして、構造の配列に似たものが発見されました。 各要素のサイズは0x84バイトで、内部には、とりわけ、プレーヤーの名前、使用されたランダムカードファイルの名前、ゲームクライアントのバージョン、ping値、ブール型のいくつかの変数、およびいくつかの整数値および/またはポインターが格納されます。 この領域のレコードを追跡しようとするすべての試みは、他のスレッドからのmemcpy（）呼び出しによって制限され、静的に分析するのはかなり困難です-xrefは82個のリンクを生成し、値0x12345678での初期化に加えて、それらすべてが読み取られます。 つまり 構造体のアドレスがレジスタにロードされ、目的の要素のアドレスが計算されます。その後、読み取り、書き込み、変数としてのポインターを使用した他の関数の呼び出し、または上記のすべてが直ちに実行されます。



最後に、この構造の特定の部分に書き込み追跡を設定し、ImportantIteration（）の本文にいくつかの条件付きブレークポイントを追加しました。 実際、デバッガーの停止をオフにし、条件として、メッセージウィンドウにレジスターの内容を表示する小さなIDC関数を指定しました。 



 Message("EDX: %08X\n", EDX), 0


その後、ゲームを開始し、ローカルネットワーク上のホストに数秒間接続しました。 次に、デバッガーを停止し、追跡結果とメッセージボックス（「トレースウィンドウ」と「出力ウィンドウ」）の内容を確認した後、次の結論に達しました。



データの抽出元となるメモリ領域、および重要な反復（）は、常に変化しています。 同時に、新しいデータが書き込まれる前に、いくつかの値が最初にリセットされます。 おそらく、新しい情報を保存する前のネットワークメッセージの処理のどこかで、このメモリ領域が最初に再初期化されます。 また、DirectPlayとマルチスレッドを使用しているため、これらのゼロはサイクルの実行中にのみ存在する可能性があり、記事の冒頭で説明した動作につながります。
