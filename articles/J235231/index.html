== DBusとは==

D-Busは、メッセージを送信するための複数のバスをアプリケーションに提供するプロセス間通信システムです。 デスクトップアプリケーション間のシームレスな接続と、デスクトップアプリケーションとシステムサービス間の接続を提供します。 メッセージ（シグナル）のブロードキャストだけでなく、メソッドのリモート呼び出しもサポートされています。



つまり Awesome、外部アプリケーション、またはウィジェットから、データを処理するためのリクエストを送信し、DBusバスを介して処理結果を取得し、最後にこのイベントの関数ハンドラーを呼び出すことができます。 同時に「フリーズ」もありません。Awesomeに結果を期待させないからです。



Dbusを使用するには、スクリプトまたはシェルからアプリケーションに信号を送信するための標準ユーティリティ「dbus-send」、およびアプリケーションやシステム間で送信されるすべての信号を監視できる「dbus-monitor」を使用できます。 どのアプリケーションがdbusに登録されているか、どのアプリケーションが提供できるかについて、より完全な情報を取得したい場合は、KDEキット「qdbus」のサードパーティユーティリティを使用できます。



dbus-monitorを実行すると、アプリケーションとシステムから送信されるすべての信号を監視します。 ただし、特定の信号に関心がある場合は、出力をフィルタリングできます。 

dbus-monitor "interface='ru.gentoo.kbdd' "


この場合、kbddによって送信されたキーボードレイアウトの変更に関するシグナルを受け取ります。 詳細については、 LORにお問い合わせください。 原則として、dbus-monitorはスクリプトのデバッグに便利です。



しかし、スクリプトとAwesomeの相互作用の可能性に興味があります。



次のコードを使用して、Awesomeに信号を送信できます。 

 dbus-send --session --dest=org.naquadah.awesome.awful /ru/console/mocp ru.console.mocp.songChanged


ここにあるものと何を分析します。

--session-（システムではなく）セッションデータバスが使用されていることを示します。 つまり セッションバスは、ユーザーに代わって起動されたアプリケーションが接続されるユーザーバスであり、システムバスにはほとんどの場合ユーザーがありません（HALサービス、ネットワークスタック、Bluetoothなど）

--dest = org.naquadah.awesome.awful-ここでは、誰が私たちのシグナルの受信者になるかを示します。この場合、それは素晴らしいです

/ ru / console / mocp-オブジェクトの一意の名前（通常はサービスの名前、オブジェクトへのパス、インターフェイス）。この場合、自分で作成します。

ru.console.mocp.songChanged-使用されるメソッド。実際には、アプリケーションが使用される場合に信号を生成する呼び出される関数。



同じ方法を使用して、Awesomeからさまざまなアプリケーションに信号を送信することもできます。たとえば、異なるプレーヤーでトラックを切り替えたり、Pidginでステータスを変更したりできます。 同時に、この方法の利点は、端末のコピーを開始して処理用のコマンドを送信する必要がないことです。それほど多くはありませんが、システムのリソースをロードして端末を作成し、その端末でコマンドを処理してから、この端末を破棄します この方法と例については後ほど説明します。



ご覧のとおり、すべてが非常に簡単です。 しかし、これでは十分ではありません。 そのような関数が機能したことをメッセージで送信することと、結果をシグナルで送信することは別のことです。 このような機会があります。Dbus信号を介してウィジェットまたは機能のデータを送信できます。

たとえば、同じkbddは、信号自体に加えて、選択されたレイアウトも転送します（数字として、別の関数ではレイアウト名、experiment）。 別のレイアウトマネージャーを使用する場合、状況は同じです。 KDEで最も重い信号であり、バイナリを含む多くの情報がそこに送信されます。



サポートされているデータ型：文字列、バイト、ブール、int16、uint16、int32、uint32、int64、uint64、dooble、object_path

次に、データを要求するタイマーを作成します。 

 fs_timer = timer ({timeout = 600}) --  10  fs_timer:add_singal ("timeout", function () awful.util.spawn_with_shell("dbus-send --session --dest=org.naquadah.awesome.awful /ru/console/df ru.console.df.fsValue string:$(df -h --output='pcent' /home | sed '1d;s/ //g' )" ) end ) fs_timer:start()


素晴らしい3.5を使用している場合は、add_singalをconnect_signalに置き換えてください。



そして、信号を受信したときに値を更新します。 

 dbus.request_name("session", "ru.console.df") dbus.add_match("session", "interface='ru.console.df', member='fsValue' " ) dbus.add_singal("ru.console.df", function (...) local data = {...} local dbustext = data[2] fs_root.text = ": " .. dbustext -- 3.5 fs_root:set_text(":" .. dbustext) end )


それだけです、すごい再起動！



この方法の利点は、呼び出されたコードが処理されている間、Awesomeが待機（したがってハング）しないことです。

 --  kbdwidget = widget({type = "textbox", name = "kbdwidget"}) kbdwidget.border_color = beautiful.fg_normal kbdwidget.border_width = 1 kbdwidget.text = '<span color="#F8EC5D"><b> Eng </b></span>' next_layout=1 function changeKeyboardLayout(keyboard_layout) awful.util.spawn( "dbus-send --type=method_call --session --dest=ru.gentoo.KbddService /ru/gentoo/KbddService ru.gentoo.kbdd.set_layout uint32:".. keyboard_layout ) end dbus.request_name("session", "ru.gentoo.kbdd") dbus.add_match("session", "interface='ru.gentoo.kbdd',member='layoutChanged'") dbus.add_signal("ru.gentoo.kbdd", function(...) local data = {...} local layout = data[2] lts = {[0] = '<span color="#F8EC5D"><b> Eng </b></span>', [1] = '<span color="#FF3000"><b>  </b></span>'} kbdwidget.text = " "..lts[layout].." ==目的のアプリケーション信号を検索==

ほとんどのアプリケーションは、dbusを介して直接制御できます。 トラックを切り替えたり、アプリケーションを全画面モードにしたり、ステータスを変更したりできます。 考えられるすべての信号とメソッドのリストを取得するには、アプリケーションを起動し（これがないと、使用可能なイベントの登録は行われません）、qdbusを実行して、目的のインターフェースを見つけます。たとえば、 

 qdbus | grep clementine   : org.mpris.MediaPlayer2.clementine org.mpris.clementine , : qdbus org.mpris.clementine  : / /Player /TrackList /org /org/mpris /org/mpris/MediaPlayer2   : qdbus org.mpris.clementine /Player


そして、このアプリケーションのすべての可能なメソッドとシグナルを取得します。 たとえば、次のトラックに切り替えることに興味があります。この方法は次のとおりです。 

 method void org.freedesktop.MediaPlayer.Next()


この場合、メソッドはパラメーターを必要としないため、単に呼び出します。 

 dbus-send --type=method_call --session --dest=org.mpris.clementine /Player org.freedesktop.MediaPlayer.Next


実際には、それだけです。 さらに実験して、自分を探してください。



==スクリプトからの信号の追跡==



個々のメソッドを呼び出すよりも複雑なタスクを実行する場合は、dbus-sendコマンドを含むシェルスクリプトを作成するか、高レベルの言語を使用してタスクを簡素化できます。 Python、Ruby、およびJava言語用のD-Busバインディングがあります。



次の例では、スクリーンセーバーがアクティブになったときにPidginのステータスを「キーボードから離れる」に変更するPythonスクリプトが表示されます。 D-Busには2つの側面があります。スクリプトはスクリーンセーバーからのシグナルを待ってから、Pidginでメソッドを呼び出します。



すぐに予約してください。スクリプトは私のものではありません。オリジナルへのリンクは以下にありますが、この相互作用の可能性については説明できませんでした。

pidgin_screensaver.py 

 #!/usr/bin/env python def pidgin_status_func(state): obj = bus.get_object("im.pidgin.purple.PurpleService", "/im/pidgin/purple/PurpleObject") pidgin = dbus.Interface(obj, "im.pidgin.purple.PurpleInterface") status = pidgin.PurpleSavedstatusFind("afk") if status == 0: status = pidgin.PurpleSavedstatusNew("afk", 5) if state: pidgin.PurpleSavedstatusSetMessage(status, "Away from keyboard") pidgin.PurpleSavedstatusActivate(status) import dbus, gobject from dbus.mainloop.glib import DBusGMainLoop dbus.mainloop.glib.DBusGMainLoop(set_as_default=True) bus = dbus.SessionBus() bus.add_signal_receiver(pidgin_status_func, dbus_interface="org.gnome.ScreenSaver", signal_name="ActiveChanged") loop = gobject.MainLoop() loop.run()


このスクリプトを見てみましょう。 pidgin_status_func関数は、ステータスをPidginに設定します。 セッションバスからim / pidgin / purple / PurpleObjectオブジェクトとim.pidgin.purple.PurpleInterfaceインターフェースを受け取ります。 次に、インターフェイスメソッドが呼び出されます。 「afk」という名前のステータスタイプの存在を確認した後、新しい「保存されたステータス」タイプを作成します（「afk」は「キーボードから離れる」を意味し、5は「不在」ステータスの一種です）。



次に、関数は状態変数をチェックします。これは、pidgin_status_func関数の引数です（この引数の意味を以下で説明します）。 引数がtrueの場合、「new afk」というメッセージに「Away from keyboard」という値が割り当てられ、ステータスがアクティブになります。 その結果、Pidginは「キーボードから離れました」というメッセージとともに、ステータスを「afk」として表示します。



次に、スクリーンセーバーのアクティブ化とともにこの関数を呼び出す必要があります。 したがって、dbus.mainloopを実行し、セッションバスに接続します。 次に、org.gnome.ScreenSaverインターフェースからActiveChangedシグナルをリッスンするシグナルレシーバーを追加します。 /信号がトリガーされると、pidgin_status_func関数を呼び出します。 ActiveChanged信号にはスクリーンセーバーの現在の状態（1-アクティブ、0-非アクティブ）を示すブール引数があるため、pidgin_status_func関数では1つの引数（状態）のみを使用します。 一定のリスニングを行うには、スクリプトの実行中に機能する無限ループを実行します。



一般に、多くのアプリケーションにはdbusインターフェースがあるため、それらを管理する可能性は非常に広範囲であり、想像力と欲求によってのみ制限されます！
