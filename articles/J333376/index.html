これの準備はできていますか？



特定のDBMSを効果的に使用するには、IDEマネージャーやDBマネージャー、監視、モデリング、スキーマ移行ツールなど、特別なツールが必要であることは周知の事実です。 ただし、これらのツールと環境の大部分は、単一のリレーショナルDBMSがすべてのプロジェクトに十分であり、「 アジャイル 」、「 DevOps 」、「 CI / CD 」などの流行語で栄えた時代に発明され開発されました。現れなかった。 それ以来、ほとんど変わっていない データベースの開発と保守の分野は常に閉鎖的で保守的であり、ほとんどの開発者はそれを古くて複雑で理解できないものに関連付けています。 今日、最も多様なDBMSのいくつかとそのインスタンスの数十/数百がほぼすべてのプロジェクトエコノミーで同時に動作しているとき、開発者とDBAの通常の日常的な問題はさらに深刻になっています。







従来、データベースを操作するための最も一般的で不可欠なツールは、すべての種類のIDEです。 通常、これは、単一の作業環境での開発と管理のための最も多様な機能を主にグラフィカルなインターフェースの下で組み合わせた包括的なソリューションであり、データベースとの作業をより生産的で快適にします-目的のメニュー項目をクリックするだけです。 確かに、 EMS SQL Manager 、 Toad 、 dbForge 、 その他の多くのクールなものなしでやる人はほとんどいません。







このような「ボックス」の主な問題の1つは、新機能と新しいDBMS（市場に出ることを止めない）のプレッシャーの下で、かなり複雑になり、毎回より複雑になり続けることです。 さらに、使用中（多くの場合、特定のDBMSの機能をマスターする必要はありませんが、IDE自体のトリックと機能を習得する必要はありません）および実装（したがって、このようなシステムは通常閉鎖され、非常に有償です）の両方です。 オープンソースDB IDEの旗艦である多くの（私を含む）最愛のDBeaverでさえ 、 そのEnterpriseバージョンでのみ NoSQLソリューション（および現在、それなしで実行しているプロジェクト）をサポートしています。 これらすべてに加えて、非常に必要で興味深いDBMS機能のサポートは、IDE開発者がそれをプッシュする必要があると考えるまで（または、カウントしてプッシュすることすらまったくできない）、期待できないという非例示的なリスクが残っています。







この状況に対応して、ユーザー自身が新しいDBMSに必要な機能やサポートが現れるまで待たずに、自分自身やプロジェクトに最適な方法で、入ってくる問題を個別に解決します。 GitHubは、さまざまな程度の複雑さのソリューションに満ちています。 基本的に、これらはすべての場合のSQLスクリプトのセットです（ dataegret / pg-utils 、 NikolayS / postgres_dba 、 gwenshap / Oracle-DBA-Scripts 、 ktaranov / sqlserver-kit 、 lestatkim / opensql ）。 または、トップのようなpg_activityやpgcenterなどのコンソールユーティリティ。 また、完全に独立したクライアントと監視ツール（ pg_web 、 pg_hero ）から始まり、たとえば、単純にシステムテーブル上のhtmlラッパー（ pg_web_stats 、 pg_stats_viewer ）で終わる、あらゆる種類のWebベースのツール。 その結果、「プログラマーのためのプログラマー」、「DBAのためのDBA」などのソリューションのクラス全体が形成され、その助けを借りて、DBMSユーザーはお互いの経験を直接共有します。
