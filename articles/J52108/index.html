すべてが非常に単純な、通常のMVCです。 プログラムを作成するためのアルゴリズムは次のとおりです（誰もが大好きな「テストによる開発」とは多少異なりますが、原則として、開発前/後/開発中にフォーム、コントローラー、モデルのテストを調整できます）。

  1. まず、特定のユーザーアクション中に発生する何かが必要です。デザイナーがそれについて質問したか、何か他のものがあります。
  2. アクションは常に何らかのURL呼び出しです。 URLを作成します（任意でも構いませんが、/「controller_name」/「action_name」という形式のURLに非常に満足しています。しかし、投稿を取得するには、何らかの/ post / 12321を使用する方が適切ですか？）、決定しますそれがコントローラーによってどのように処理されるか（何も当てはまりませんか？/ application / controllers /に新しいものを作成します）、このコントローラーにアクションを追加します。
  3. /views/scripts/controller_name/action_name.phtmlを作成し、必要なhtmlをそこに描画します。

    • 他のページに表示されていないものを表示する必要がある場合は、次のように記述します。<？= $ This-> shachlo？>、書き留めるか、必要なものを覚えておいてください。
    • たとえば、あるユーザーの名前など、たくさん表示されるものを出力する必要がある場合、<？= $ This-> printUser（$ this-> user）？>;


  4. 次に、必要なすべてのヘルパーを作成します（ヘルパーは、次のように呼び出される関数です：<？= $ This-> printUser（$ this-> user）？>）。 それらを/ views / helpers /で書きます。 アイコンを追加したり、ユーザーの名前に他のスタイルをいつでも適用できるように、それらを個別に使用することをお勧めします。 または、すべてのユーザーを「OK」という言葉に置き換えて、すべてのユーザーが驚いたことに夢中になるようにします。 ％）
  5. 幸せにジャンプして、アクションを書きます。 アクションコード全体を（実際にはそうではないが）3つの部分に分けることができます。

    • 引数を処理します。フォームまたはそこのURLからデータを取得します（URLが/ user / 12345の場合）。
    • アクションに必要なことを行います。ロケットを宇宙に発射するか、ポケットベルでタクシー運転手にメッセージを送信します。 データベースからのデータが必要な場合、またはLast.fmやTwitterからのデータ、またはその他のデータが必要な場合は、モデルについて質問します。 [これまでに]モデルが発行方法を知らない場合、できるかどうかを尋ねます：$ this-> m-> twitter-> getUser（ "va1ka"）; またはそこに$ this-> m-> getUsersTable（）-> getUser（ "va1en0k"）; -必要に応じて、質問してください。
    • 彼がステップ3で要求したすべてを表示します。$ this-> view-> shachlo = "popyachtsa"; など。
  6. 前のステップで彼女に求めたすべてをモデルに追加します。




このアルゴリズムはほぼ普遍的であり（ユーザーアクションで何が起こるかが必要な場合は、少し異なる方法で行動する必要があります）、できるだけ明確に固執することをお勧めします。 なぜ最初に見てから、残りを見るのですか？ さて、2つの理由で、最初に37signalsがアドバイスし、次に、これが私たちが必要とするものと受信する形式を理解する最も合理的な方法です。 すべてのデータリクエストがモデルを介してのみカプセル化されるようにすることが重要です：幸いなことに、その場合、データベースを見ずに手を振ったり、ステップモジュールのリセットなど、プログラムを高速化する素敵なキャッシュを追加したりできます。 すべてのHTML、つまり非常に完全にすべてが、フォルダー内、ヘルパー内、またはそのようなビューまたはレイアウトを通過するようにすることが重要です。

コントローラー


コントローラーはアクションで構成されています。 アクションは、アプリケーションのバックボーンです（統合アプローチに基づく構造機能社会学の場合と同様）。 これは、プログラム設計者からプログラマーへの理想的な要求です。「さあ、このボタンを押すと、これが起こります。」 したがって、「そのような」はアクションです（誰に説明しますか？）。



コントローラーをバッチで作成する前に、トリッキーな動きをすることをお勧めします。独自のフロントコントローラーとアクションコントローラーを作成し、それらを既に継承しています。 次に、その場合、たとえば、サイトにアクセスせずにページにアクセスすることを禁止できます（たとえば、らい病のコロニーのように）、アクションコントローラーに必要なさまざまな変数を設定します。

モデル


実際、このモデルでは、私は経験不足について最も疑問を持っています。 まあ、私は通常個人的にこれを行います：

各データタイプについて、データベースにタブレットを作成します。 そこには、投稿のために、コメントのために、すべてのものがあります。 データ型に配列変数（たとえば、投稿のタグ）がある場合、そのために、切り株は明確で、別個のラベルです。 次に、プレートごとに/ application / models / DbTable /に独自のファイルを作成します。 さて、メインファイルはSuperModel.phpで、このファイルでは、アンジェリーナジョリーがすべてのタブレット（静的変数に格納されている）をメソッドとともに返します。 同様に、ファイルシステムまたは外部データソースを操作するために必要なクラスを突き出し（最近では、まったく緊張せずにZend_Rest_Clientを使用してlast.fmを操作しました）、スーパーモデルに到達するためのメソッドを突き出しました。
