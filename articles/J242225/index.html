ActiveRecordはレーキ、リレーション、インデックスについて少しです

痛みを伴うことについてお話したいと思います。一般的にはARで、特にリレーションで働くことです。 簡単にあなたの人生を台無しにして、コードを遅くて大食いにすることができる標準的な庭の製品に対して警告してください。 物語は、同じ流出のRails 3.2とActiveRecordに基づいています。 もちろん、Rails 4には新しくて便利なものがたくさんありますが、それを切り替える必要があります。とにかく基盤は同じです。



ほとんどの場合、この資料は初心者向けです。ARを使用する場合、作成者がActiveRecordオブジェクトやその他の射撃肢の形でテーブル全体の内容をメモリに抽出するのは非常に苦痛だからです。 Zenを学んだ開発者にとって、このトピックは役に立つとは思われません。彼らは自分の例と教訓でそれを補完することしかできません。






彼らは世界に何回繰り返しましたか...



Relation （および一般的なActiveRecordオブジェクト）で作業を開始した場合、1つのことを明確に提示する必要があります。選択を「具体化」するポイント、つまりSQLクエリの構築を停止するポイントです。 言い換えると、データがフェッチされ、メモリ内の処理に進むときです。 なぜこれが重要なのですか？ はい、それは厄介だからです： 



Product.all.find{|p| p.id == 42}


サーバーをハングアップさせ、すべてのRAMを取得し、さらに多くの汚いトリックを実行できます。 そして同じことですが、言い換えれば： 



 Product.find(42)


結果なしですぐに動作します。 したがって、 findとfindはまったく同じものではありません！ なんで？ はい、最初のケースではProduct.allと言って足で撃ちました。これは、 製品テーブルのコンテンツ全体を抽出し、各行でARオブジェクトを構築することを意味するため、それらから配列を作成し、クラスメソッドであるfindをウォークスルーするためです配列 （一般的に、 findは Enumerableからのものですが、これらはすでに詳細です）。 2番目のケースでは、すべてがはるかに優れています。findはARメソッドであり、 pkで検索するように設計されています。 つまり、リクエストを生成します 



 SELECT * FROM products WHERE products.id = 42;


それを実行し、1行すべてを受け取ります。



良い点と悪い点



ここで、ARを扱うことが大きな責任である理由がわかったので、足元を撃たないようにしましょう。 これは非常に簡単です。ARが提供するメソッドを使用する必要があります。 ここにあります： where、select、pluck、includes、joins、scoped、unscoped、find_eachなど、ドキュメントまたは隣接するハブにあります。 ただし、使用しない方が良いのは列挙が非常に難しく、同時に非常に簡単です。他のすべてのメソッドを使用するのは望ましくありません。残りのほとんどすべてのメソッドが関係を配列に変換するためです。



シンプルなレシピ



今、私はいくつかの標準を提供しますが、生活を楽にするデザインはあまりありませんが、しばしば忘れられます。 しかし、読者に質問する前に、has_many関数を思い出してください。 知っているパラメーターと、積極的に使用しているパラメーターについて考えてください。 あなたの心にそれらをリストし、それらを数えます...そして今質問は次のとおりです：あなたはそれらのうちの何人が実際に知っていますか？



答え
Rails3に24個、Rails4に12個。 12pcsの違いは、 where、groupなどのメソッドと、ハッシュではなくブロックでRails4に渡される純粋なSQLを操作するメソッドによって生じます。


なぜこれを聞いたのですか？ はい、あなたのレベルを非常に大まかに評価し、ほとんどのオプションを知っているなら、以下はあなたに新しい知識をもたらす可能性が低いと言うために。 この評価は非常に条件付きです。したがって、親愛なる読者は、とんでもない/支持できない/奇妙/など（必要なことを強調する）ように思えても、あまり怒らないでください。



レシピ回数



それでは、順番に行きましょう。 update_attributesとupdate_attribute （またはすべてではない？）については誰もが知っています。 最初の-検証とコールバックの呼び出しでフィールドを大量に更新します。 面白くない。 2番目-すべての検証をスキップし、コールバックを起動しますが、選択した1つのフィールドの値のみを更新できます（一部の人にとっては、 保存（検証：false）の方が似ています ） しかし、何らかの理由で、彼らはしばしばupdate_columnとupdate_allを忘れます。 このメソッドは、検証とコールバックの両方をスキップし、前戯なしでデータベースに直接書き込みます。



レシピ番号2



コメントは素晴らしいタッチ方法を思い出させました。 彼らはしばしば彼のことを忘れて、 



 @product.updated_at = DateTime.now @product.save


または 



 @product.update_attribute(:updated_at, DateTime.now)


ただし、この目的のためには、次のようにするほうが簡単です。 



 @product.touch(:updated_at) #


さらに、 touchには独自のafter_touch コールバックと、オプション：touchが belongs_toメソッドにあります。



正しく反復する方法



ハブはすでにfind_eachについて話していましたが、もう一度言及するしかありません。 



 product.documents.map{…}


そしてそれらは同形であり、どこよりも少し多く見つかりました。 Relationで使用される従来のイテレータには、1つだけ問題があります。それらは、データベースからすべてを一度に引き出します。 そしてこれはひどいです。 対照的に、デフォルトでは、 find_eachは一度に1000個を持ち、問題ありません！



UPD：コメントで既に述べたように、raw-sqlに明示的に投影されていないすべてのメソッドはto_aに委任されます。これが、クエリ全体がメモリにフェッチされ、その操作がDB側ではなくRuby側にある理由です。



default_scopeに関するヒント



default_scopeの内容をブロックでラップします。 例： 



 default_scope where(nullified: false) # ! default_scope { where(nullified: false) } #


違いは何ですか？ 最初のオプションはサーバーの起動時に実行され、 無効化されたフィールドがデータベースになかった場合、サーバーは起動しません。 同じことは移行にも当てはまります。フィールドがないために移行されません。おそらく追加するだけです。 2番目のケースでは 、 Rubyが遅延しているという事実により、ブロックはモデルがアクセスされ、移行が正常に実行されるときにのみ実行されます。



Has_many through



別の一般的な患者は 



 product.documents.collect(&:lines).flatten


ここでは、製品には多くの行がある多くのドキュメントがあります。 製品に関連するすべてのドキュメントのすべての行を取得したいことがよくあります。 そして、この場合、上記の構造を作成します。 この場合、リレーションのスルーオプションを呼び出して、製品に対して以下を実行できます。 



 has_many :lines, through: documents


そして実行する 



 product.lines


より明確かつ効率的になります。



JOINについて少し



結合のトピックの続きで、 インクルードについて思い出してください。 何がそんなに特別なのですか？ はい、それはLEFT JOINです。 かなり頻繁に、左/右結合が明示的に記述されていることがわかります 



 joins("LEFT OUTER JOIN wikis ON wiki_pages.wiki_id=wikis.id")


確かに機能しますが、RoRの純粋なSQLは常に好まれません。



また、レジから逸脱することなく、 結合と値を一緒に使用する場合の値の違いを思い出す必要があります。 usersテーブルがあり、 製品などのさまざまなエンティティにauthor_idフィールドとauthorリレーションがあり、その下にusersテーブルがあるとします。 



 has_one :author, class: 'User', foreign_key: 'author_id' #  ,


このような場合、次のコードは機能しません 。 



 products.joins(:author).where(author: {id: 42})


なんで？ 結合では、リレーショナルの名前が示されます。これはjoinimであり、テーブルに条件が課される場所であり、 



 where(users: {id: 42})


これは、結合で明示的に「AS author」を指定することで回避できますが、これも純粋なSQLです。



次に、異なる角度から結合を見てください。 参加するものは何でも、すべてを開始したクラスのオブジェクトになります。 



 Product.joins(:documents, :files, :etc).first


この場合、結合の数に関係なく製品を取得します。 彼らは結合されたテーブルからフィールドを受け取りたいので、この振る舞いはいくらか動揺します。 そして、彼らは反対側から同じ要求をし始めます：ドキュメントを受け取り、それらを製品に参加させ、他のエンティティと通信するための純粋なSQLを書き、一般に、正しい論理コードが最初に書かれたときに自転車を発明します。 したがって、私はまさにその根拠を思い出します： 



 Product.joins(:documents, :files, :etc).where(...).pluck('documents.type')


ここでは、データベースから目的のフィールドを持つ配列を取得します。 長所：最小限の要求; ARオブジェクトは作成されません。 短所：Rails 3では、 pluckは1つのパラメーターのみを受け入れますが、ここでは 



 pluck('documents.type', 'files.filename', 'files.path')


Rails 4でのみ実行できます。



関係を構築する



ここで、関係の構築を扱うことの検討に移りましょう。 一般に、すべては非常に簡単です。 



 product.documencts.build(type: 'article', etc: 'etc').lines.build(content: '...')


product.saveを呼び出した後、すべての関連付けが検証、設定、遊女とともに保存されます。 このすべての楽しいアクションには1つの注意点があります。これは、 製品が 読み取り専用でない場合、および/または他のストレージ制限がない場合にすべて良好です。 そのような場合、多くの人は、上記の例の結合を持つ庭に似た庭を配置します。 つまり、 ドキュメントを作成し、それを製品にバインドして、ドキュメントの行を作成します。 デフォルトの動作は曲がっており、通常はエラー処理に関係していることがわかりますが、 製品は動作しません。 したがって、付属資料では、これはすべて松葉杖ですぐに囲まれ、間違いを投げて、かなり厄介なものになります。 この場合の対処方法 自動保存について覚えて、それがどのように機能するかを理解する必要があります。 詳細に入ることなく、彼はコールバックに取り組んでいると言います 。 したがって、上記の製品の関係を保持する方法があります。 



 product.autosave_associated_records_for_documents


この場合、ドキュメントが保存され、そのコールバックが呼び出されて行が保存されます。



インデックスについて一言



最後に、多くの人はインデックスに基づいた問題のために固体オブジェクトに頭をぶつけたため、インデックスについて言う必要があります。 ActiveRecordとデータベースの機能の多くを妨害してすぐに謝罪しますが、個人的な意見としては、データベース側でその瞬間に何が起こっているのかを理解しないとARをうまく動作させることはできません。



問題1



何らかの理由で、多くの人は、 リレーションの 順序がソートする列に依存しないことを確信しています。 この誤解のバリエーションは、 順序関係と順序配列の違いを理解していないことです。 このため、VARCHARフィールドの順序と精神の質問でdefault_scopeを満たすことができます。「ページの読み込みが遅いのはなぜですか？ そこでは、データベースから取得されるレコードはわずかです！” ここでの問題は、この列にインデックスがない場合、デフォルトのソートが非常に高価になることです。 デフォルトでは、ARはpkでソートします。 私たちがやると起こる 



 Products.first


ただし、 pkにはほぼ常にインデックスがあり、問題はありません。 しかし、モデルへの呼び出しで順序（：名前）がどうなるかを言うと、問題が始まります。

参考 ：「指で」説明すると、インデックス付きの列で並べ替える場合、実際の並べ替えは行われず、データベースに既に存在し、データはすぐに正しい順序で送信されます。
