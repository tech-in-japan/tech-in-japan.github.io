日食



Eclipseは、IDEを作成するための拡張可能なプラットフォームです。 プラットフォームはプラグインで拡張され、それぞれがEclipseの動作を補完できます。 たとえば、新しいエディター、メニュー項目などを追加します。



作業エリア







ワークベンチは、Eclipseプラットフォームのグラフィカル表現のルート要素です。 ユーザーに何らかのモデルを提供する1つまたは複数のウィンドウ（ ワークベンチウィンドウ ）を含めることができます。 ほとんどの場合、これらはワークスペースに保存されているリソースです。 [ウィンドウ]→[新しいウィンドウ]を使用して、新しいウィンドウを開くことができます。

各ウィンドウには、ページのコレクション（ ワークベンチページ ）とアクティブページへのリンクが含まれ、1つのウィンドウでアクティブにできるのは1ページのみです。

エディターとビューは、適切なマネージャー（ ViewFactory、EditorManager ）を使用してページに保存されます。 エディターとビューには共通のインターフェイスがあります-IWorkbenchPart 、次に作業部分。 ワークスペースはエディターとビューへのリンクを保存しませんが、EditorReferenceやViewReference などの中間オブジェクトを使用します。 これにより、 遅延初期化を使用してプラグインをロードできます。 たとえば、次のスクリーンショットでは、階層ビュー、Javadocビュー、および宣言ビューを定義するプラグインはまだ使用されていないため、まだロードされていない可能性があります。







ビューとエディターは互いに非常に似ており、ページの一部であり、メニューとボタンを作成するためのメカニズムは同じです。どちらもいくつかのファイルのコンテンツを表示できますが、概念的な違いはあります。 ビューとは異なり、エディターの複数のインスタンスを1ページに作成できます。 エディターはファイルの変更のステータス（ dirty ）を表示でき、保存せずにエディターを閉じると失われる可能性のある未保存の変更があることを報告できます。



ページには多くの視点も含まれています 。 パースペクティブは、パーツのコレクション、ページ上の相対的な位置、コマンドのキーボードショートカット、メニューの可視性などを含むページテンプレートです。 コマンドウィンドウ→パースペクティブのカスタマイズを使用して、パースペクティブをカスタマイズできます。

ユーザー設定に関する情報は、 [workspace] /。Metadata / .plugins / org.eclipse.ui.workbench /ディレクトリに保存されます 。 これらのファイルを使用すると、さまざまな作業領域でパースペクティブ設定を使用できます。 ただし、ワークスペースを切り替えるときに設定をコピーするオプションを使用することをお勧めします。

プラグイン



プラグインは、ワークスペースのコンテキスト内で特定のサービスを提供するコンポーネントです。 プラグインの例は、開発者がプロ​​グラムをデバッグできるorg.eclipse.debug.coreと、前者のグラフィカルな表現を含むorg.eclipse.debug.uiです。



プラグインの説明は、プラグインフォルダーにある特別なXMLファイルplugin.xmlで行われます。 このファイルは、プラグインがEclipseプラットフォームを拡張する方法（エディター、ビュー、コンテキストメニューを説明する）、およびその方法を「依存関係」セクションで説明します。 プラグインの実装コードは、プラグインの起動時にのみロードされ、再び初期化が遅延します。 このようにして、Eclipseはプラグイン拡張機能を直接ダウンロードせずに表示できます。



plugin.xmlの例は、記事「プラグインマニフェストファイル（plugin.xml）とは？」に記載されています。 



<?xml version="1.0" encoding="UTF-8"?> <?eclipse version="3.4"?> <plugin> <extension point="org.eclipse.ui.views"> <view class="com.xyz.myplugin.ViewPart" id="com.xyz.myplugin.testview" name="Test View"> </view> </extension> </plugin>




ランタイム環境



Eclipseランタイムは、プラグインのライフサイクルを制御する機能を提供します。 プラグインは必要な場合にのみロードされ、同様に無効になります。 同時に、起動されたEclipse内でプラグインのインスタンスが作成され、プラグインのクラス（ org.eclipse.core.runtime.Plugin ）が対話ポイントと見なされ、アクティベーターとも呼ばれます。 アクティベータの主な機能は、プラグインの起動中に追加のアクションを実行することです（画像の初期化、特定の静的リソースのキャッシュなど）。 追加のアクションを実行する必要がない場合、アクティベーターを指定することはできず、標準のアクティベーターが使用されます。 アクティベーターの詳細については、 こちらをご覧ください 。 Eclipseランタイムは、OSGi仕様を実装しています。



OSGi（Open Services Gateway Initiative）は、複雑なシステムをモジュール（ バンドル ）に分割し、システムを再起動せずにこれらのモジュールをリモートでインストール、更新、または削除できるようにする仕様です。 各モジュールには、他のモジュールに対する依存関係のセットと、他のモジュールと対話するための厳密に定義されたAPIがあります。

技術的には、モジュールはMETA-INF / MANIFEST.MFモジュールに格納されている追加のメタ情報を含む.jarファイルであり、このファイルはjarファイルの仕様の一部です。 非OSGiランタイム環境は、これらのファイルを無視します。



OSGiは、バンドル、サービス、ライフサイクル、サービスレジストリなどのコンポーネントについても説明します。 OSGiアーキテクチャは次のとおりです。







Eclipseに加えて、OSGiには、 Apache Felix 、 Knopflerfish 、 Concierge OSGiなどの実装がいくつかあります。 OSGiの詳細については、 公式Webサイト 、 wiki 、およびLarsを参照してください 。



エクイノックス



EclipseでのOSGiの実装は、哲学的にEquinoxと呼ばれます。 したがって、プラグインはOSGiモジュールであり、そのメタ情報は次のとおりです。 



 Manifest-Version: 1.0 Bundle-ManifestVersion: 2 Bundle-Name: Test Plug-in Bundle-SymbolicName: com.xyz.myplugin; singleton:=true Bundle-Version: 1.0.0 Bundle-Activator: com.xyz.MyPlugin Require-Bundle: org.eclipse.ui, org.eclipse.core.runtime Bundle-ActivationPolicy: lazy Bundle-RequiredExecutionEnvironment: JavaSE-1.6


Equinoxは、実行時にモジュール間の依存関係を管理します。 インストール中にMANIFEST.MFモジュールを読み取り、すべての依存モジュールが既にロードされていることを確認してから、要求されたモジュールをアクティブにします。 依存関係がない場合、モジュールはロードされません。 モジュールが依存関係なしにクラスにアクセスしようとすると、ClassNotFoundExceptionがスローされます。

OSGiはプラグインをアクティブにするために使用できます 。プラグインは、個別のプラグインフォルダーeclipse_install_dir / pluginsにあります 。



依存関係



直接的な関係


この役割は、依存する必要なプラグインによって果たされます。 古典的な直接依存関係は、MANIFEST.MFプラグインのメタデータで説明されています。

依存関係は、実行時だけでなく、コンパイル時にも決定されます。 実行時に、Eclipseは必要なすべての依存関係が依存モジュールで利用可能であることを確認し、それをアクティブ化するだけです。 コンパイル時に、Eclipseはそれに応じて依存プラグインのクラスパスを変更します。



延長




Eclipseで最もよく使用されている依存関係の1つは、 拡張ポイントと呼ばれます。 ポイントは、他のプラグインに直接依存せずに、ホストのホスト機能を拡張するために使用されます。 拡張機能と拡張ポイントは、プラグイン記述ファイルplugin.xmlに記述されています。

拡張ポイントを定義するプラグインにより、他のプラグインが拡張機能の説明に基づいて機能を追加できます。 拡張は、コードまたはデータのいずれかです。



拡張ポイントの良い例は、電源ネットワークです。その中の拡張ポイントは、電源の他の要素（ランプ、コンピューター、アイロン）を接続できるソケットで、電源の機能を拡張します。 この場合、各デバイスには電源ネットワークと一致するデバイスが必要です。そうでない場合、接続できません。



例を続けて、 org.eclipse.uiモジュールは、「エディター」、「ビュー」、「メニュー」などの多くの拡張ポイントを記述します。 同時に、別のプラグインorg.eclipse.help.uiがあります。このプラグインは、メニュー項目「Help-> Help Contents」を追加します。 これを行うには、次の拡張機能について説明します 



 <extension point="org.eclipse.ui.actionSets"> <actionSet description="%actionSet.description" id="org.eclipse.help.ui.actions" label="%actionSet.label" visible="true"> <action class="org.eclipse.help.ui.internal.DynamicHelpAction" icon="$nl$/icons/etool16/help.gif" id="org.eclipse.help.ui.dynamicHelp" label="%searchAction.label" style="push" toolbarPath="org.eclipse.ui.workbench.navigate" tooltip="%dynamicHelpAction.tooltip"/> </actionSet> </extension>

プラグインは、既存の拡張ポイントを使用することに加えて、新しい拡張ポイントを定義できます。 ポイントは、* .exsdおよび対応する拡張ポイントタグを使用して、説明ファイルへのリンクを使用して説明されます。
