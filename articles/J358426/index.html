著者のアレクセイ・プロトニコフに床を渡します。



ユーザー識別は、最新のアプリケーション、特にゲームの基礎です。 評価表、実績、氏族などの社会的機能のないゲームを想像することは困難です。 社会的機能のないアプリケーションでは、たとえば、デバイス間の同期プロセスを改善するために、識別が必要です。 ほとんどの場合、この問題に関する私の研究の出発点となったのは、まさに同期の問題でした。



各UWPアプリケーションには専用のクラウドストレージ（ローミングデータ）があり、ユーザーが同じMSアカウントを使用していれば、任意のデバイスからアクセス可能なデータを保存できます。 残念ながら、このストレージにはいくつかの重大な制限があります。



まず、クラウドに配置できるデータのサイズには制限があります。 ほとんどのタスクでは、割り当てられたサイズで十分ですが、これは依然として制限です。 さらに重要な制限は、データをクラウドに配置する原則です。 この機能の内部操作アルゴリズムは、たとえば、省エネや制限されたインターネット接続などの多くの要因を考慮し、最終的にデバイス間の同期プロセスには10分以上かかるだけでなく、まったく保証されません（詳細はこちら ）。



このような制限は私のタスクにはまったく適していません。そのため、同期プロセスは独立して実装することに決定しました。最初の問題は、同じプレーヤーに属するデバイスを識別することでした。



多くのゲームとアプリケーションはFacebookのユーザーアカウントを使用して同期しますが、ほとんどの場合、これは既存のアルゴリズムをMicrosoft Storeに転送するためであり、実装の利便性のためではありません。 他には、プレーヤーが登録する必要のある独自のサービスがありますが、これも不必要な複雑さです。 結果として、最も論理的なソリューションはもちろん、ユーザーが間違いなく持っているMicrosoftアカウントを使用することです。これがないと、Windowsストアからアプリケーションを購入してインストールすることができません。 残っているのは、異なるデバイスでユーザーデータを識別するためにユーザーデータを取得することだけです。



質問を詳しく調べてみると、Microsoftアカウント情報はMicrosoft Live APIを使用して取得されていることがわかりました。 実際、ユーザーに関するデータを受信するには、ユーザーは自分のアカウントにログインする必要があり、アクセストークンを受信する必要があります。これにより、ユーザーはそれらを要求できます。 この手順はセキュリティのために作成されたもので、 OAuth認証標準の一部ですが、初心者向けの独立した実装は複雑で不便に思えます。



幸いなことに、UWPの作成者は、OAuthが初心者の開発者にもたらす可能性のある恐ろしさを理解し、Microsoftアカウント（だけでなく）を操作するエレガントでかなり単純な方法を作成しました。 このツールは「アカウントマネージャーWebアカウント」と呼ばれ、後で検討します。

ただし、初期化する前にパネルにコマンドを入力することは不可能であるため、最初に行う必要があるのは、対応するイベントをサブスクライブすることです。 パネル表示コードの前に次の行を追加し、すぐにイベントによって呼び出されるプロシージャを形成します。

ちなみに、公式ガイドでは、AccountCommandsRequestedイベントへのサブスクリプションは現在のページへの移行時に発生し、それから切り替えるとサブスクリプションが解除されます。 これは、認証プロセス用に別のページを選択した場合に理にかなっていますが、このアプローチには十分な柔軟性がないと思います。

このコードを扱います。 まず、コマンドのリストを取得するのに時間がかかる場合があるため、パネルがいっぱいになるまでパネルの表示を遅らせる必要があります。 これを行うために、遅延オブジェクトが作成されます。 次に、前述のFindAccountProviderAsync関数を使用して、プロバイダーが読み込まれます。 値が「consumers」のパラメーターもこの関数に渡されることに注意してください。 これは、組織のアカウントではなく、標準のMicrosoftアカウントを取得することをプロバイダーに伝えます（「組織」になります）。



取得したプロバイダーを使用して新しいチームを作成したら、チームがクリックされたときに呼び出されるプロシージャを指定し、結果をパネルに追加します。 最後の行は、パネルが形成されて表示できることを通知します。







これで、起動して「ログイン」ボタンをクリックすると、いくつかのログインオプションが表示されたパネルが表示されます。 パネルに1つのコマンドを追加したという事実に戸惑うかもしれませんが、2つのコマンドがあります。 実際、これが私が「アカウントマネージャーWebアカウント」を好むもう1つの理由です。彼は現在のユーザーアカウントを独自に決定し、それをメインログインオプションとして提供し、ユーザーが別のアカウントを使用する場合、別のオプション。 内部ロジックの観点から見ると、両方のオプションは同一であり、その結果、前に作成したGetMsaTokenAsyncプロシージャに送られます。



さらにアクションをいくつかの段階に分けることができます。 まず、GetMsaTokenAsync内で、要求するデータの種類を示すデータプロバイダーへの要求を行い、この要求の結果を生成する必要があります。 



 Dim request As WebTokenRequest = New WebTokenRequest(command.WebAccountProvider, "wl.basic") Dim result As WebTokenRequestResult = Await WebAuthenticationCoreManager.RequestTokenAsync(request)


最初の関数に渡されるパラメーター「wl.basic」に注意してください。 これは、アプリケーションがユーザーデータを操作するときに受け取るアクセス許可を示しています。 異なるアクセス許可を指定することで、アドレス帳、カレンダー、写真、その他の多くのデータを操作できますが、名前、アバター、ユーザーIDのみが必要なので、wl.basicを使用します。これで十分です。 許可の完全なリストは、 ここにあります 。

まず、別のクラスでWebアカウントマネージャーを操作するロジックを完全に取り除きました。 これは、アプリケーション内のどこからでもこのコードにアクセスできるようにするために行われます。 たとえば、「 高度なスプラッシュスクリーン 」という記事で、スプラッシュスクリーン内で実行するのが理にかなっている長い操作について説明しました。BackgroundConnectUserでユーザーデータを取得するプロセスは、まさにそのような操作です。



コードを別のクラスに移動した結果、ページ自体のクラスにはほとんど何も残りませんでした。 ページ内のすべての作業は、2つのボタンを押して（MainPage_Loadedプロシージャで）ページのデータコンテキストを設定する反応に限定されます。 UserManagerクラスのインスタンスが、主要な作業が既に行われているデータコンテキストとして機能することを推測することは難しくありません。 このクラスは、ページレイアウトからフィールドにバインドできるINotifyPropertyChangedインターフェイスを実装します。



移植されたコードはいくつかの変更を受け取り、元のサンプルが作成されたのと同じ順序でそれらを解析しようとします。



  • ConnectUserプロシージャ。 基本的に、「ログイン」ボタンをクリックして以前に呼び出されたコードを複製します。 このプロシージャへの呼び出しは、バックグラウンド接続用のトークンがない場合にのみ必要です。
  • BuildPaneAsyncは、いくつかの重要な変更を受け取りました。 開始するには、AccountCommandsRequestedイベントからサブスクライブを解除します。このイベントは、このプロシージャを呼び出した後に不要になります。 これには、既知のコードが続きます。これは、マネージャーウィンドウで小見出しを設定し、プライバシーポリシーへのリンクを追加することで補完されます。 これらのポイントは両方ともオプションであり、あなたの裁量で完全に実装されます。 さらに、2番目のケースでは、任意のコードを実行するリンクを追加できます。プライバシーポリシーのあるページに移動する必要はありません。







  • [続行]ボタンをクリックすると、GetMsaTokenAsyncプロシージャが開始されるので、最初の行でログオンステータスを設定します。このステータスに応答して、進行状況インジケータの表示またはインターフェイス要素のブロックを行います。 次に、すでにわかっているコードがありますが、入力がキャンセルまたは失敗した状況への反応が追加されています。 私は認めている、私はこの一見普通な場所のために一日中戦った。 キャッチは、入力オプションを選択したウィンドウの直後にこのコードを初めて実行すると、データにアクセスするための許可を求める別のウィンドウが表示されることです。 これら2つのウィンドウの切り替えの間に比較的長い期間（最大2秒）が経過する場合があるため、「ログイン」ボタンを繰り返し押すことを排除するために、この期間中はログオン状態を維持するのが理にかなっています。 ただし、ユーザーはデータへのアクセスの提供を拒否したり、2番目のウィンドウを閉じたりするだけです（モバイルデバイスの[戻る]ボタンをクリックします）。そのため、そのようなアクションの場合、元のステータスをNoneに戻す必要があります。これはElseブロックで行います。
  • 新しいバリエーションのBackgroundConnectUserプロシージャは非同期関数になったため、バックグラウンドでのデータ受信の失敗（または成功）の試行に応じて対応できます。 この関数はさまざまな場所（スプラッシュスクリーンなど）から呼び出すことができるため、ロードされたアクセストークンが空でないことを確認する必要があります。 保存されたマーカーを読み込むために、別のGetWebAccount関数が作成されています。ローカル設定にマーカーがない場合は、Nothingを返します。 WebAccountオブジェクトの取得に失敗した場合はデフォルトの結果（False）を返し、成功した場合はステータスをLogonに設定して手順を続行します。 次に、既知のアクションを繰り返しますが、唯一の違いは、トークンの受信を拒否した場合はNoneステータスを返す必要があり、成功したリクエストではSsuccessfulであるということです。
  • LogOutUserプロシージャには、以前は使用されていなかったコードが含まれています。 アカウントを終了し、保存された設定からマーカーを削除する機能を実装します。 この機会をアプリケーションに実装してください。これにより、他の資格情報を入力する必要性は言うまでもなく、機密性の確保に関してユーザーに完全な自由が与えられます。
  • さて、最後に言及するのは、このクラスで作成したConnectStatusプロパティです。 このプロパティには、独自の列挙子UserConnectStatusEnumのタイプがあり、ログインステータスを設定するために必要です。 これはクラスとの対話の重要な要素です。このプロパティにバインドすることにより、コードの特定のセクションの実行時に押すのが望ましくないボタンをブロックしたり、逆にログイン後にのみ使用可能なボタンをロック解除できるためです。 IsEnabledプロパティのConnectStatusへのバインドを正しく解釈するために、ConnectStatusToEnabledConverterコンバーターも作成されました。
