文字列の変更について



-アレクセイ、パフォーマンスについてよく話していましたが、最近はストリングクラスをたくさんやっています。 この理由は何ですか？



-文字列の最適化を目的としたプロジェクトに参加しているため、パフォーマンスの観点から文字列についても説明します。 この領域には毎日多くの小さな最適化がありますが、2つの大きな変更を行っています。



1つ目はCompact Stringsです。 Javaアプリケーションにあるほとんどの文字列のシンボルは、ほとんどASCIIに適合します。 これは、文字列の各文字に対して、char仕様で要求されているように、2バイトではなく1バイトを使用できることを意味します。 現在、ストリング内、ストレージ内は文字配列です。 高度に最適化されており、人々は高いパフォーマンスを期待しています。 したがって、これらの行を圧縮するには、このStringの2つの表現を作成する必要があります。1つの表現は典型的なchar配列で、2番目の表現はバイト配列です。各バイトは特定の文字に対応します。 そして、リリース基準で書いたように、「非回帰」を提供する多くのパフォーマンス作業が必要です。 そのため、Java 8からJava 9に切り替えるユーザーは痛みを感じるだけでなく、この機能の幸福感とパフォーマンスの向上を体験できます。



また、「ライブラリ」がどのように機能するか、文字列連結で何をどのように行うか、ランタイムがこの全体をどのように処理するかについて、さまざまな感動的なものがあります。 一般的に、多くの小さな詳細があります。

今、私たちはこの変更、正確な測定を慎重にプロトタイピングし、これがパフォーマンスにどのように影響するかを理解し、適切なコード生成戦略を選択するのに6か月を費やしました...このすべての作業の後。 すべての長所と短所を知ることは助けになり、それですべてがうまくいきます。

-そして、「パフォーマンスの低下なし」とは何ですか？ テストのセットはありますか、それを評価するための基準はありますか？



-これには正式で非公式の態度があります。 非公式の態度は、メジャーリリースが少なくとも最後のメジャーリリースと同じくらい遅く機能するという事実に慣れている大きなコミュニティがあるということです。 そうしないと問題が発生するため、新しいバージョンへの移行方法。 したがって、新しい機能を開発するときは、パフォーマンスに失敗するだけでなく、どこかで勝つ必要があることを覚えています。



そして、そのようなワークロードで非常に多くの割合でパフォーマンスを浪費できるかどうかを示す正式な基準があります。 しかし、これは開発チームの内部問題であり、OpenJDKではありません。 むしろ、態度はそれだけです-私たちは、コミュニティがJavaが退行しないことを望んでいることを理解しています。 また、Javaが退行しないようにします。 この暗黙の仮定に基づいて、開発が行われます。

低レベルのタスクの複雑さについて

-主に、非常に多くの可動部品があるため。 知っておくべきことがたくさんありますが、そのために準備する必要があります。 たとえば、コンパイラを作成し、プロセッサのエラーを知る必要があります。 まず、それらが存在すること（そして多くの人にとっては、プロセッサにバグがあるということです）を知るには、これらのバグを調べて、コンパイラが常に非標準の動作に罪があるとは限らないことを理解する必要があります。 あなたが今修正しているバグは、あなたが型の責任を負う部分までに作られた過去のコード変換のある種の大きな干渉に関係しているかもしれないことを知って、可能になり、理解するべきです。 このスタックを詳細に知る必要があります。



Ranthime自体は、コンポーネントを大きなセルに分離できる製品ですが、実際、これらのコンポーネントは互いに非常に密接に関連しています。 バグを修正したい場合、これは多くの場合、異なるコンポーネントのバグを修正する必要があることを意味し、パフォーマンスに取り組んでいる場合、間違いなく同時に多数のコンポーネントに関与します。



パフォーマンスパッチが1つのファイルで5行かかる場合は、子供の頃はうれしいです。これは、パフォーマンスのすばらしい変更であり、明らかに正しくて役立ちます。 しかし、大きな、優れたパフォーマンスの変更には、通常、大きな製品全体の小さな場所すべての変更が必要です。 したがって、この製品を知っておく必要があり、製品は巨大です。

「それはすべて紙の上でうまく機能します。」 実際には、次の2つのことが起こります。まず、パフォーマンスを追求し始めると、抽象化がいくつかの別々の場所で行われることが必要であることがわかります。 次に、バグが発生します。

JMH、ベンチマークおよびナロースペシャライゼーションについて



-それが問題です。 過去3年間、 Java Microbenchmark Harness （JMH）を積極的に開発しています 。 あなたが行うほとんどのコミットは、これがほとんどあなたのプロジェクトであり、現在はOpenJDKの後援の下にあります。 だから？

-JMHの場合、ストーリーは次のとおりです。 あなたは常に自分自身に質問する必要があります-誰が恩恵を受けます。 パフォーマンスチームがJMHを行うのはなぜですか？ 彼女は私たち自身の活動を促進するためにそれを行っています。なぜなら、私たちには必然的にベンチマークにつながる研究があるからです。 この調査を行うにはツールが必要ですが、これはすべてツールに限定されません。 実験から得られる主なものは数字ではないからです。 あなたができる主なことは、数字から知識を引き出すことです。 原則として、多かれ少なかれ信頼できる知識を抽出するためには、理論のシステムが必要です。それぞれの理論はこれらの実験によって確認されなければなりません。 これには新しいものは何もありません-あなたは科学の哲学を取り、それはまったく同じです。



製品に関するパフォーマンスは、自然に関する科学者のようなものです。 高レベルのモデルを構築し、実験を通じてこれを行います。 JMHは実験を行うのに役立ちます。このフレーズの主な言葉は「ヘルプ」です。 しかし、あなたのために実験しないでください。 時間を節約し、実験の非自明な特定のものに対処するために、ベンチマークの明らかな熊手に踏み込まないようにするのに役立ちます。 だから、すぐに書くことができます[開始-約。 赤]ベンチマーク、このベンチマークを正しく記述する方法を理解するために残りの時間を費やします。 そして、例えば、デッドコードの除去で愚かな間違いを修正しようとして何時間も費やさないでください。 それが目的です。



各実験で解決する必要がある光の問題を解決するのに役立ちます。 特定の実験の特定の問題を解決する助けにはなりません。 分析は、このデータからどの洞察を提供できるか、この洞察を信頼できるかどうか、この洞察が知識の一般的なシステムのどこにあるかを理解するために必要です。



あなたがブログを書くとき、あなたはいくつかの機能のベンチマークを作成するので、私たちはしばしばこれを罪にします-そして、私たちは特にこの機能を対象とする実験を示します。 実際、この実験の背後にはこれらのリンクのチェーンがまだあります。 たとえば、このハーネスはこのマシンで正常に動作すること、このマシンを較正したこと、サーマルシャットダウンに陥らないことを知っています。 または、テストへのそのようなアプローチが機能することを知っています-個別に検証しました。 原則として、これはブログに書くこともできますが、それはブログではなく、今行っている実験が信頼できることを確認するために、以前にどの段階の実験を行ったかについての本全体になります。 ベンチマークを配置し、デフォルトでベンチマークが正しいと仮定することはできません。 これは起こりません。 ベンチマークは、あなたが既に知っている他のすべてと結びついて洞察を与える場合にのみ正しいです。



-今、何が起こるか見てみましょう。パフォーマンスについて学ぶ必要がありますが、私はこれであまり大きなドックではありません。 たとえば、特定の当局に言及します。 ベンチマークを実行し、測定し、パフォーマンスモデル、具体的な指示についての仮説を立てています。 アセンブラーのリストを取得し、そこにあるアプリケーションからこれらの指示を見つけます（または見つけません）。 秘trickは次のとおりです。このアセンブラーリストでどのような命令を探すべきかを理解するには、背景が必要であり、JMHを使用せずに同じ問題を解決できる必要があります。 だから？
