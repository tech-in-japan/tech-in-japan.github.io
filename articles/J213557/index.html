シェル-コンソール

KolibriOSオペレーティングシステムの重要な部分（カーネルとほとんどのドライバーとプログラム）はアセンブリ言語で記述されています。 ただし、高水準言語のプログラムも利用できます。 今日はそのうちの一つについてお話します。



Shellコマンドラインプログラムを作成する前に、Hummingbirds用のプログラムを作成した経験がありました。 基本的に、これらはゲームでした-Piton 、 Donkey 、ZX Spectrumエミュレーターのポート、私はe80と命名しました 。コンソールタグのポートとElizaの仮想対話者のポートです。 できる限り楽しみました。







おもちゃに加えて、もっと役に立つものを書く試みがありました。 そのため、たとえば、 cObjユーティリティ（「.obj」を参照）を使用すると、ダイナミックライブラリによってエクスポートされたリストを表示できます。 グローバル変数EXPORTSまたは_EXPORTS関数を含むCOFFファイル。 実際、これは最も切り捨てられたDependency Walkerです。 cObjプログラムはアセンブラーで作成されており、kpackコンプレッサーで圧縮された実行可能ファイルのサイズは256バイトです。 したがって、私にとってはこのプログラムはHummingbird OSの概念に完全に適合しているように思えます。







なぜ私は後で議論する間違ったプログラムに段落全体を捧げたのですか？ 2つの理由があります。 最初：cObj-コンソール（console.obj）を操作するためのライブラリを調査するための素晴らしいツールであることが判明しました。 KolibriOSの多くのものと同様に、ライブラリは十分に文書化されているため、すばらしい追加ツールです。 2番目の理由は、私は怠け者であり、アセンブリ言語は特定の結果を達成するのではなく、コンピューターゲームに時間を費やすのと同じくらい楽しいことです。 主要な作業場所のプロジェクトに関連するコードの低レベルの最適化は考慮されません。 もちろん、cObjとElizaのポートを除く、記事にリストされているすべてのプログラムは、偉大で強力なCで書かれています。



おもちゃのおもちゃ（いまいましい、そして今私はそれらを書いています！）、しかし、私はプログラム開発者だけでなく有用な何かを書きたかったです。 また、高度なコマンドラインがなかったため、実装することにしました。



「ハミングバードOSにはコマンドラインがなかったのですか？！」 はい、コマンドラインが存在しました。 アセンブリ言語で書かれたプログラムはCMDと呼ばれ、原則として、客観的に話せば、非常に機能的でした。







ただし、主観的には、多くの欠点が見られました（小さな印刷など、ウィンドウをスクロールできないなど）。 そのため、「スクリプトとコンソールアプリケーションを使用してコマンドラインを作成します」という目標が設定されました。







2008年7月 彼はHummingbird OSのフォーラムでトピック「非常に機能的なシェル」を開きました。 最初のメッセージは、「console.objを使用してCで機能シェルを作成することにしました。」というメッセージで終了しました。「コマンドとパラメーターを分離する機能を作成し、ヘルプコマンド（これまでパラメーターなし）、ver、exitを実装しました。」



最初のメッセージに添付されたshell.zipファイル（現在180回以上ダウンロードされています）には、サイズ1.3キロバイトのこの技術のバイナリが含まれていました。



翌日、追加コマンドexit、ls、pwd、ps、kill、help（最後のコマンド-パラメーターあり、パラメーターなし）を追加したバージョンを投稿しました。



当初、更新は非常に頻繁に行われていましたが、その後は更新が少なくなりました。 これは、私が自分の開発に興味を失ったということではありません。 ただ「楽しみのために」プログラミングするための時間が残っているわけではないというだけです。



Shellプログラムのバージョン履歴は0.01から始まります。 はい、正確に0.01です。 バージョン0.1では、プログラムを実行することが可能になりました（そのような機会はすぐには実現しないと誰が考えたでしょうか？）。 バージョン0.3以降、シェルは独自の形式のスクリプトを実行できました。 また、このバージョンでは、エイリアスを使用した作業を含むチームの履歴がありました。



バージョン0.4.2以降、Hummingbird OSコミュニティの他のメンバーがプロジェクトの開発に参加しました。 そのため、ダイヤモンド（Hummingbirdsの多くのプログラムの作成者およびconsole.objライブラリの開発者）は、ウィンドウを閉じるときにプログラムの正しい終了を実装し、速度関数を最適化しました。Pteroxはいくつかの便利なコマンドを追加し、 Leencyはスクリプトの相対パスをサポートしました。 さらに、 CleverMouseに助けがなかったのは驚くことではありません。



2009年1月末のHummingbird OS 0.7.5.0のリリースにより、シェルは古いCMDコマンドラインを置き換えました。

この「ほぼ非常に機能的なシェル」の可能性は何ですか？



現在までのプログラムの現在のバージョンは0.7.4です。 はい、1.0には程遠いので、あまり期待しないでください。 ただし、25個のコマンドが実装されています（about、alias、cd、clear、cp、date、echo、exit、free、help、history、kill、ls、mkdir、more、ps、pwd、reboot、rm、rmdir、shutdown、スリープ、タッチ、アップタイム、ver）、スクリプトとコンソールアプリケーション、コマンド履歴、コマンドエイリアスのシステム、制御文字とESCシーケンスの処理のサポートがあり、システム全体のクリップボードをサポートするための最初のステップが取られました。 プログラムは、ロシア語または英語のメッセージでコンパイルできます。







「内部」では、シェルには約80キロバイトのCソースコードのみがあり、そのうち18はlibcに似たライブラリであり、KolibriOS API関数を備えたライブラリです。 独自のlibc（さらに、Hummingbird OSには少なくとも2つの実装があります）を使用すると、プログラムの構築が簡単になり、出力ファイルのサイズがわずかに小さくなりました。 ちなみに、シェルのダイヤモンドバージョンはMSVC用に設計されていますが、MinGWコンパイラはシェルのビルドに必要です。 私はプログラムのソースコードを可能な限り透過的にしようとしました。 初心者プログラマでも理解できる（それがまさに彼らにコメントがほとんどない理由です！コードは理解できる！私は冗談を言っています） シェルのソースコードはモジュールに分割され、各内部コマンドは個別のファイルに配置されます。理論的には、必要なセクションの検索が簡単になります。



プログラムの能力から、私たちは仕事の原則に目を向けます。 インタラクティブモードでは、すべてが明確である（コマンドを入力する、答えを取得するなど、退屈するまで）と思われる場合、シェル用のスクリプトおよびコンソールアプリケーションの作成には微妙な違いがあります。



スクリプトには、対話的に動作するコマンドを含めることができます。 さらに、起動時にシェルは設定ディレクトリ（通常は/ sys / settings /）を検索し、それが起動されたディレクトリで「.shell」ファイル（引用符なし）を検索し、見つかった場合は実行します。 通常は、aboutコマンドと、ヘルプを入力するための提案が含まれています。 ただし、もちろん、独自のコマンドシーケンスを記述できます。







スクリプトでは、ポンド記号（＃）で始まるコメントを使用できます。



スクリプトの最初の行（上のスクリーンショットを参照）は「#SHS」でなければなりません。 これは「SHellScript」を意味し、Shellはスクリプトを実行できるかどうかを決定します。



シェルのコンソールアプリケーションは、コンソールウィンドウを初期化する必要さえない通常のHummingbird OSアプリケーションです。 確かに、このようなアプリケーションの開発者は、シェルとのインターフェイスの正しい初期化に注意する必要があります。



シェルアプリケーションとコンソールアプリケーション間の情報交換は、指定された領域を介して実行されます。 コンソールアプリケーションを起動した後、まず、pid-SHELLという名前の名前付きエリアを作成する必要があります。ここで、pidは大文字のゼロを含まないプロセスの識別子です。例：6、42または204。エリアの最初のバイトはコマンドです（最大255コマンド、十分に）、次にデータを取得します（ただし、データがない場合があります）。



実装されたコマンドのリスト：



  • SC_OK 0何もしない
  • SC_EXIT 1出力
  • SC_PUTC 2ディスプレイキャラクター
  • SC_PUTS 3表示ストリング
  • SC_GETC 4はキーボードから文字を読み取ります
  • SC_GETS 5キーボードから行を読み取ります
  • SC_CLS 6クリアスクリーン


コンソールアプリケーションは、Hummingbird OS用のコードを生成できるコンパイラーのプログラミング言語で作成できます。 ShellディレクトリのKolibriOS OS SVNサーバーでは、コンソールアプリケーションの2つの例を見ることができます-Cとアセンブリ言語（そして最後の例が最初のものよりずっと後で現れたことは注目に値します）。



開発者が知っておくべきいくつかの機能：

  • シェル自体のコンソールプログラムは、プロセッサ時間の合理的な使用に注意する必要があります。
  • Shellのコンソールプログラムは、指定された領域自体を閉じることに注意する必要があります。


つまり 原則として、いくつかの簡単なルールと、特に例のソースコード（Cで2キロバイト、アセンブラーで3キロバイト）を解析した後の残りは明らかです。 例のメイン関数など、複雑ですか？ 



void kol_main() { char string[256]; sc_init(); sc_cls(); sc_puts("This is a test console application for Shell\n\r"); sc_puts("Type a string (255 symbols max): "); sc_gets(string); sc_puts("You typed:\n\r"); sc_puts(string); sc_puts("Press any key: "); string[0] = sc_getc(); sc_puts("\n\rYou pressed: "); sc_putc(string[0]); sc_exit(); }




インターフェイスのシンプルさにもかかわらず、KolibriOSにはシェル用の単一のコンソールプログラムはまだありません！ ただし、スクリプトもあまり一般的ではありませんが、依然として使用されています。



一般的に、Shellプログラムについてはかなり長い間話すことができます。 特にそのさらなる開発について。 結局のところ、今日のことは多くの人には向いていません。 ユーザーは、どのチームを指定せずに、新しいチームの実装を要求します。 はい、シェルはBusyBoxから遠いです。 ただし、Shellは非UNIX系のシステムを対象としており、標準のlibcは使用されていないため、各新しい関数の実装は一種の実験であることは明らかです。 そして、そのような実験はもっとたくさんあります。 したがって、バージョン1.0では、Shellを「非常に機能的なシェル」と安全に呼べることを願っています。
