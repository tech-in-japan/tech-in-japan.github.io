新しい時間またはExtJS 3.0の始まり。 Ext-Coreの最初のベータ

ExtJSフレームワークは長い間メインのお気に入りのツールになってきたため、 ExtJS開発者とユーザー向けの最初の会議での今後の発表を楽しみにしています。 その間、「Ekstovtsy」は、長い間約束されていた製品のリリースに満足しており、優れたライセンスの下でさえ（本当にそれを必要としていた人たちに）喜んでくれました。 そして、 Ext Core 3.0 、どうやら、3行目の一連のリリースからの最初のツバメを待っていました （当然、これはもちろん、完全なフレームワークの新しいバージョン、特に新しいビジュアルコンポーネントのリリースです）。 Ext Core 3.0を詳しく見てみましょう。



そのため、Ext Coreは、最小限のAJAXツールが必要であり、それまでjQueryやDojoなどのツールが機能していた単純なWebサイトと開発の世界に参入しようとする試みです。 ExtJSを使用した以前の開発でライブラリ全体（またはそのカスタムアセンブリ）の接続が必要であり、機能の数パーセントしか使用されていない単純なプロジェクトでは受け入れられなかった場合、メインライブラリと同じAPIと機能を備えた完全に既製のカーネルがあり、多くの必要な機能を実装するのに十分です。 はい、はい、メインライブラリとの類似点に気付くかもしれません。メインライブラリは、ext-core、ext-adapter、およびメインファイルに分けられています。 この場合、アダプタの機能（ext-baseである独自のアダプタのみを使用し、Ext Coreでサードパーティのアダプタを使用することはできません）とライブラリの基本機能を組み合わせて、スタイルを必要とせずにすべてが1つのファイルに結合されますいいえ。ウィジェットとインターフェース要素を操作する方法がないためです。 はい、Ext Coreは、私が利用できる豊富な機会から視覚的コンポーネントのユニークで最高のシステムを削除し、基本的な機能と方法だけを残して、何でも構築できるようにします。



ちなみに、よく見ると、srcディレクトリには、Coreを構成するコンポーネントの完全なセットがあります。これは、その内部を調べるのに非常に便利です。 ただし、通常のExtJSフレームワークを使用した場合、新しいことは何もありませんが、ライブラリをプロジェクトに接続してすぐに作業を開始できます。 プロジェクトが配布されるMITライセンスでは、ExtJSで使用されるGPLとは対照的に、ソースコードを開かなくても、独自のクローズドおよび商用プロジェクトでコードを使用できます。 したがって、ExtJSのすべての機能を本当に必要としないが、たとえば将来的にそれを行う予定がある場合は、ここで使用できます。すでにCoreバージョンを使用しているプロジェクトに他のコンポーネントのサポートを追加することはできません難しいことはありません（ただし、先日テストする必要があります）。











コードのサイズも良い印象を残します-アーカイブせずに、コアは76 KBのみを使用します。これは、jQueryよりも優れていると思います。デバッグバージョンは180 KBほど大きくなりますが、gzip圧縮を使用すると、サイズは25 KBに縮小されます、これは同様の機能を備えた最新のライブラリの最良の指標に匹敵します。



Coreライブラリには何が含まれていますか？

  • ページ要素を操作するためのすべての可能性-DOM、CSS、イベントとの統一作業、ブラウザのニュアンスの隠蔽、ページ要素、アニメーション、および類似の古典的な機能を操作するときに使用される視覚効果を担当するサブシステム。 簡単な検査の後、APIは完全なExtJSと比較して特に変更されていないように思われるため、多くの場合、コードとの互換性が保証されます。 以下は、コアに含まれるクラスのリストです-Ext.CompositeElementLite、Ext.DomHelper、Ext.DomQuery、Ext.Element、Ext.EventManager、Ext.EventObject、Ext.Fx。
  • テンプレートシステムのサポートはExt.Templateですが、メインバージョンのより強力で高度なXTemplateテンプレートシステムが移植されていないのは残念です。 ただし、すべての基本機能が利用可能であるため、サーバー上とほぼ同じスタイルでクライアント上で作業することはかなり可能です（Smartyまたは別のテンプレートエンジンのようなものを意味します）。
  • AJAXサポート。 同じアーキテクチャを維持するために、Coreは、XMLHTTPRequestへの低レベルのアクセスを提供するConnectionコンポーネントが提供されるDataモジュールを導入しました。 残念ながら、コアではこのコンポーネントを介した作業のみが利用可能であるため、スクリプトタグまたはiframeを介して接続するためのプロキシはありません。 ただし、これはそれほど頻繁に使用されることはなく、ほとんどがすでにWebアプリケーションで使用されています。つまり、ExtJS全体を使用する方がよいことを意味します。 従来のAjaxクラスはConnectionコンポーネントを拡張し、リクエスト処理への統合アクセスを提供します。 独自のHTTPヘッダーを指定し、フォームを送信し、データの送信方法を再定義し、グローバルイベントをインターセプトできます。 奇妙で悲しいことは、UpadateManagerコンポーネントがないことです。これにより、任意の要素に対して、1つのupdate / loadメソッドでajaxを介してコンテンツを変更できます。 いいえ、停止しますが、ロードメソッドは利用可能です。ただし、スクリプトの実行に制限があり、多くの場合、これは依然として制限であり、いくつかの工夫が必要です。 個人的には、このライブラリに従来のUpdateManagerを追加しようとします。
  • JavaScript基本クラスの拡張。 JS自体は非常に柔軟な言語であると指摘し、開発者は一部の組み込みオブジェクトの基本機能を補完および拡張しますが、それでも言語自体への影響を最小限に抑えようとしているため、たとえば複数のライブラリを同時に使用することが可能です。 高度な機能の中でも、関数の拡張機能-createCallback、createDelegateおよびdeferを強調する価値があります。要素の値によってインデックスを決定する方法とその削除は、Arrayクラスの機能に追加されます。 文字列プリミティブは、基本的にこのような低レベルのテンプレートエンジンであるformat関数によって拡張されます（この場合、Ajax.NETページで使用するとこのメソッドが競合する可能性があるという特別な注意があります。 これは明らかにライブラリです ）
  • Extグローバルクラスにカプセル化された主な機能（プラットフォームとブラウザーの定義）を含むユーティリティメソッドは、URLデータとJSONデータを処理し、それぞれを通して配列を処理し、変数タイプの定義を処理します。
  • タイマーと定期的なイベントを操作します。 あまり説明されていないが非常に有用なコンポーネントは、定期的なデータ処理（指定された関数の起動）のためのTaskRunnerタイマーの拡張バージョンである遅延タスク（DelayedTask）です。 正直なところ、私はこれらの機会をプロジェクトで頻繁に使用してきたので、これらの機会に非常に満足しています。
  • そして最後に、すべての強力なオブジェクト指向インフラストラクチャがコアに含まれており、これに基づいて、実際にはフレームワーク全体が構築されます。 独自のクラスを自由に作成し、継承を使用して（ライブラリの観点から-拡張）、オーバーライド/オーバーライド、プロトタイプを操作できます。 これには、最も一般的に使用されるパターンのいくつか-Singlton、Observableの実装も含まれます。 さらに、名前空間のサポート。


実際、ExtJSが私たちに喜ばれているのはそれだけです。 メインフレームワークで育ったため、その機能の一部が欠けていると感じることもありますが、Ext Coreとメインフレームワークのスコープが根本的に異なることを忘れないでください。 しかし、たとえば、Webアプリケーションを作成している場合、訪問者にメガバイトのExtJSをダウンロードさせることなく、同じライブラリでサイトを作成することは非常に論理的です。代わりにCoreを使用するだけで、開発者は再トレーニングを必要としません。また、プロジェクトをさらに拡大しても、コードを書き換える必要はありません。 基本的なフロントエンドでは軽量ライブラリを使用し、ウィジェットやコンポーネントに直面してデータを処理するための重火器を使用すると、Coreは開発と開発の架け橋に過ぎないようです。 そして、ここでは、Dojoで行われたものと同様に、新しいコンポーネントの動的ロードの実装は非常に便利です。



コアは、フレームワーク全体の機能の基本的な研究にも最適です。クラスとその複雑さはそれほど多くないため、初心者の開発者は徐々に理解し、すでに大きなプロジェクトで作業を進めることができるためです。 一部の使い慣れたツールが不足していると、最初はプロの開発者を混乱させるかもしれませんが、すぐに慣れてしまい、現在のバージョンのステータスはベータ版に過ぎません。
