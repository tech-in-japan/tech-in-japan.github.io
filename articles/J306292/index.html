HabraKvestはASP.NET CoreとAngular2で作成します

新しいフレームワークがリリースされるたびに、私はそれを試して、その上にアプリケーションを作成したいと思います。 前回 、クエスト形式はうまくいきました。 したがって、ほぼ1年半の間に変化したものを見て、別のクエストを書くことを提案します。フレームワークが表示され、プレイできます。

結果：

-ソースを見ることに興味がある人のためにgithubで並べ替えます

-何が起こったかに興味があるか、別の論理的なクエストに時間を費やす人のためのクエストへのリンク 。



猫は、プロジェクトの作成から展開までの完全なプロセスを説明しています。





前提条件



ほとんどすべてのIDEまたはテキストエディターは、ASP.NET CoreおよびAngular2での作業に適しています。 もちろん、ASP.NETの完全なデバッグにはVisual Studioが必要ですが、開発用の新しいコアバージョンのリリースでは必要ありません。Sublime\ Atom \ ...のコードを操作し、コマンドラインからアプリケーションを実行できます（SDKのインストール後）。 

dotnet run


ソースコード、インストーラー、インターフェイスコマンドラインのドキュメントNETおよびSDKは、次の場所にあります。

dotnet.github.io

さらに、これらはすべてWindowsだけでなく、Max、Linux、Dockerでも使用できるようになりました。



多くの人が見たときにおそらく認知的不協和音を経験するでしょう 

 sudo apt-get install dotnet-dev-1.0.0-preview2-003121




プロジェクトを作成する



新しいASP.NETコアアプリケーションを作成するには、いくつかの方法があります。

1） Visual Studioを使用する

2）コンソールから-dotnet new

3）サードパーティのジェネレーターを使用します。



特に空のアプリケーションは必要ありませんが、Angular2が接続されており、できればビルドスクリプトが構成されているため、プロジェクトの作成と構成のすべての問題を掘り下げることはしません。

ASP.NET Core JavaScriptサービス github.com/aspnet/JavaScriptServicesはこれに理想的です。これにより、Angular 2、React、Knockoutの選択したJavaScriptフレームワークでASP.NET Coreアプリケーションを作成できます。

これを行うには、ASP.NET Core、Node.js、およびaspnet-spaテンプレートを備えたyeomanジェネレーターが必要です。 後者は次を使用してインストールできます。 

 npm install -g yo generator-aspnetcore-spa


新しいプロジェクトの作成は、最終的には 

 cd <__> yo aspnetcore-spa






Angular2テンプレート、プロジェクトの名前、 ジェネレーターを選択し、構造全体、依存関係などを作成します。 （これには数分かかる場合があります）。

最初は、特にあなたが後者のファンではない場合、またはフロントエンドでほとんど作業していない場合、このジェネレーターが使用するライブラリが多すぎるように見えるかもしれません。 しかし、個人的には、これは最もバランスのとれたパターンの1つであるという意見です。 特に、次のプロパティを強調したいと思います。



TypeScript



もちろん、シンプルなJavaScriptを使用してAngular2でアプリケーションを開発できます。TypeScriptを使用すると、OOPの入力構文、オートコンプリート、使い慣れた（ほとんどの）構文を使用できます。 また、Angular2チームはTypeScriptを積極的に使用しており、TypeScriptはAngular2の公式のデフォルト言語ではない（または既に使用されている）ようです。



Webpack



フロントエンドツールの戦いでは 、 webpackがすぐにお気に入りになりつつあるようです。 絶え間ないコード変更を伴う機能に注目することが特に重要です。 実際、開発者は、Hot Module Replacement（ HMR ）を使用してプロジェクトを絶えず再組み立てし、ページを更新することを忘れることがあります。 1台のモニターでフロントエンドを記述しようとした後、2台目のモニターでは目の前で変化しているので、ホットスワップの可能性なしに楽器に戻る気がしません。



その他のボーナス



サーバーの事前レンダリング、遅延読み込み、開発ビルドと本番ビルドでの便利な作業。

プロジェクト構造



JavaScriptServicesプロジェクトの構造はWebパック指向のようです。 つまり、それぞれnpm + webpackであり、gulpとbowerはありません。 私の知る限り、bowerはJavaScriptライブラリーに多くの依存関係がないため、スタートアッププロジェクトに追加しないことに決めました。npmはすべてを処理します。 そして、すべてのgulp （またはうなり声 ）機能がwebpackによって実行されるようになりました 。



ファイル構造に関しては、サーバーファイルに関してはすべて以前と同じですが、フロントエンドでいくつかの違いがあります。 たとえば、静的なwwwrootファイルがあるフォルダーには、実際にはすべてのJavaScriptがコンパイルされるdistフォルダーのみがあります。

Angularの最初のバージョンと比較すると、表現とコントローラーが存在しないことは驚くべきことです。すべてがコンポーネント内にあり、コンポーネントの各HTMLパーツにはTypeScriptパーツがあります。



プロセスの特徴



もちろん、Webpackはモジュールのホットスワップに関して非常に満足しています。このため、フロントエンド部分の開発ははるかに快適です。

また、WebPackエラーはASP.NETレベルですでに完全に表示されており、非常に便利な統合であることに注意してください。 確かに、誤った状態（たとえば、存在しないコンポーネントのプルアップ）の後、ホットスワップは機能しませんでした。そのような場合でも、ページを手動で更新する必要があります。



また、ブラウザでTypeScriptを借方記入できることも非常に便利であり、これらはすべてデフォルトですでに組み込まれています。





まず第一に、必要なものに例を使ってページを書き直しました。 Angular2とTypeScriptについての私の知識がゼロに近いという事実にもかかわらず、それは非常に簡単であることが判明しました。 一般的に、新しいAngularの構文は、少なくとも一見すると非常に読みやすいようです。 また、TypeScriptでインターフェイス、クラス、およびタイプを記述する機能により、非常に理解しやすくなります。

展開



バックエンドは.Netで記述されているため、Azureにアプリケーションをデプロイします。 ちなみに、過去1年半にわたって、ここでたくさんの変更があります。初めて何をどこで理解するかはわかりません。 このアプリケーションでは、データベース（Webアプリ+ SQL）を使用して最も基本的なサイトを作成するだけで十分です。

展開では、いつものように、非常にスムーズではありません。 まず、エラーなしで失敗することはありません。 サイト自体は機能しますが、最終的に、Visual Studioは一部の外部TypeScriptモジュールでエラーを表示します。 これまでのところ、私はそれが何であるかを理解していません。 次に、サイトを開くと、「TaskCanceledException：タスクがキャンセルされました」というエラーが発生します。 これもまだ完全には明らかではありませんが、問題はホスティングレベルのどこかにあるようです。 そしてこれに加えて、私はマイグレーションで少し遊んだ。



概して、私はこのステップに満足していました。 非常に詳細でわかりやすいログ。すべてを構成できます（ベース、移行）。 ただし、最初は、すべてが構築されてコピーされるまで、しばらく（数分）待つ必要があります。 とにかく、私が理解している限り、最初はすべてがローカルにコピーされ、それから既にAzureに転送されているプロセスは非常に複雑であるように見えますが、これらは詳細です。
