internal = javaのパッケージの可視性。 バッチの可視性は必要ありません（そして一般に、2、3のコミット後にこれらのフィールドを削除します）。 すべての場合にすぐにフィールドを初期化できるわけではなく、一般的にそれをnull可能にすることは愚かです（nullセーフのためだけにkotlinに興味があります）。 このために、kotlinにはlateinit修飾子があります。これは、プログラマーが歯に与えるものをコンパイラーに伝えます。このフィールドを使用するときはnullになりません。 これは、いわゆる構文ソルトです。 一般に、自動変換の結果としてではなくこのコードを見る場合、見る方が適切です： 



 private val batch = SpriteBatch() private val img = Texture("badlogic.jpg")

Scene2Dの最初の使用。 起動画面、起動シーン



そこで、Scene2Dを初めて使用しました。 それが何を意図していて、あなたがそれから何を望むことができるかについて簡潔にいくつかの言葉。



Scene2Dは要素のグラフ（ツリー）であり、主にUIの作成を目的としています。 箱から出してすぐに、タイプセット、要素の変換（回転、拡大縮小、シフトなど）を行うことができます。 大きな利点は、タッチの処理です。 さて、ケーキのチェリーはアクションのシステムです。 理解できない定義で、私たちは人間の言語でも同じことを終えました。



シーンがあり、画面全体を占有します。 ステージ上には、テーブル、テーブル内の写真、スクロールバー、1ダースのボタン、さらにはaldげたダッシュ（主なものは彼が俳優であるということです）を配置できます。 上/中央/左/幅などのマジックワードを使用する レイアウトを実装します。 例は、こんにちは世界が明日だけになるよりも複雑であるため、この記事はすばらしいことがわかりました。 さらに、任意の要素にタッチハンドラーを掛けると動作します。 クリックの座標を手動でキャッチしたり、そこにあるもの、z-indexオブジェクトなどを確認したりする必要はありません。 しかし、再び、明日について。 そして今日、最後にいくつかのコードスニペットがあります。 



 class MedievalTycoonGame : Game() { val viewport: FitViewport = FitViewport(AppConstants.APP_WIDTH, AppConstants.APP_HEIGHT) override fun create() { screen = LoadingScreen(viewport) } }


MedievalTycoonGameクラスはGameから継承するようになりました。Gameの全タスクは、画面に作業を配置することです。 ユーザーに表示する最初の画面は、LoadingScreenと呼ばれ、1つのシーン（LoadingStage）が含まれます。 なぜなら これらのクラスは成長するものではないため、1つのファイルLoadingScreen.ktに配置します



LoadingScreen.kt 
 class LoadingScreen(val viewport: Viewport) : ScreenAdapter() { private val loadingStage = LoadingStage(viewport) override fun render(delta: Float) { Gdx.gl.glClearColor(0f, 0f, 0f, 0f) Gdx.gl.glClear(GL20.GL_COLOR_BUFFER_BIT) loadingStage.act() loadingStage.draw() } override fun resize(width: Int, height: Int) { viewport.update(width, height) } } class LoadingStage(viewport: Viewport) : Stage(viewport) { init { val backgroundImage = Image(Texture("backgrounds/loading-logo.png")) addActor(backgroundImage.apply { setFillParent(true) setScaling(Scaling.fill) }) } }




LoadingScreenは、画面を黒で上書きし、LoadingStageのact（）およびdraw（）メソッドを呼び出します。 act（）では、プログラムロジックをハングさせ、データを操作するのが非常に便利です。 描画（）は、シーン内のすべての要素を描画するだけです。

kotlinの場合、要素を常に初期化し、隣接する行に配置します。 これは、拡張機能の適用機能によって実現されます。 kotlinの階層は自動的にインデントされ、視覚的に非常に読みやすくなります。 Javaでは、すべての組版がインデントされます。 多くの場合、要素を初期化して配置することはできません。 階層が3レベル以上の深さで構成されている場合、Javaで要素を美しく（そして安価にサポートして）整理することは不可能です。
