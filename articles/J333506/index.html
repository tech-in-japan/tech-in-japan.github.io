CSSは黒魔術ではありません

すべてのWebプログラマーは、時々CSSを作成する必要があります。 CSSに初めて出会ったとき、CSSを理解するのはナンセンスであることに気付くでしょう。 そして真実は-彼らは境界線を追加し、そこで色を変えました... JavaScriptは本当に複雑なものです。 CSSはそれに比べておもちゃです。







ただし、Web開発が進むにつれて、CSSに対する軽薄な態度は過去のものになります。 なんとも言えない奇妙なことに直面すると、CSSがどのように機能するか、その奥底で何が行われているかを正確に想像することができないことに気付くでしょう。 似たようなことが私にもありました。 勉強してから最初の数年、私はフルサイクルのJavaScript開発に従事し、時にはCSSに触れました。 私はいつも自分のパンがJavaScriptであると考えていました。



たとえば、 JavaScript Jabberに関する議論に参加しました。 昨年、私はフロントエンドに焦点を合わせることにしましたが、そのときから、例えば、JSコードと同じ方法でスタイルシートをデバッグすることができないことに気付きました。



CSSはしばしば冗談を言っていますが、十分に真剣に理解して理解しようとした人はほとんどいません。 ブラウザがCSSを処理する方法を考えて、問題に直面したときに意図的にエラーを検索したのは誰ですか？ 代わりに、Stack Overflowで利用可能な最初の回答からコードを取得し、さまざまな種類のハッキングを軽daしないでください。または、単に問題を無視してください。



その結果、ブラウザがスタイルで奇妙なことをするとき、開発者はあまりにも頻繁に手をすくめるだけです。 CSSは黒魔術に似ていると考える時です。 ただし、プログラマーはコンピューターが人間が作成したコマンドを解析および実行するマシンであることを知っています。この点で、CSSは私たちのお気に入りのJSと違いはありません。 もしそうなら、CSSは有意義かつ生産的に認識され、使用されます。



CSSの内部メカニズムの知識は、さまざまな状況で役立ちます。 たとえば、深刻なデバッグとWebページのパフォーマンスの最適化。 多くの場合、彼らはCSSについて話したり書いたりするとき、一般的な問題の解決に焦点を合わせます。 これがどのように機能するかについてお話したいと思います。



DOMとCSSOM



まず、ブラウザには、他のサブシステムの中でもJavaScriptエンジンとレンダリングエンジンがあることを理解することが重要です。 この場合、後者に興味があります。 たとえば、WebKit（Safari）、Blink（Chrome）、Gecko（Firefox）、およびTrident / EdgeHTML（IE / Edge）に関連する詳細について説明します。



ブラウザは、Webページを表示するプロセスで、ドキュメントオブジェクトモデル（DOM、ドキュメントオブジェクトモデル）およびスタイルシートのオブジェクトモデル（CSSOM、CSSオブジェクトモデル）の構築につながる一連のアクションを実行します。 簡略化されたこの一連のアクションは、次のように表すことができます。



  • 変換 ：HTMLおよびCSSバイトをディスクから読み取るか、ネットワークからダウンロードします。



  • トークン化 ：入力データストリームをフラグメントに分割し（例：開始タグ、終了タグ、属性名、属性値）、スペースや改行などの不要な文字を削除します。



  • Lexing ：この手順はトークン化に似ていますが、ここでは各トークンのタイプが決定されます（たとえば、このトークンは数値であり、文字列リテラルであり、別のトークンは等値演算子です）。



  • 解析 ：ここでは、システムは字句解析後にトークンのストリームを受け入れ、特定の文法を使用してトークンを解釈し、このストリームを抽象構文ツリーに変換します。


CSSおよびHTMLのこのようなツリー構造を作成した後、レンダリングエンジンは、それらに基づいて、レンダリングツリーと呼ばれるものを形成します。 これは、ドキュメントレイアウトを作成するプロセスの一部です。



レンダリングツリーは、ドキュメントの視覚的表現のモデルであり、ドキュメントのグラフィック要素を正しい順序で表示できます。 レンダリングツリーの構築は、次のアルゴリズムに従って実行されます。



  • DOMツリーのルートから開始して、表示されているすべてのノードをトラバースします。

  • 非表示のノードをスキップします。

  • 表示されているノードごとに、CSSOMで対応するルールを見つけて適用します。

  • コンテンツおよびスタイル用に計算された可視ノードを生成します。

  • 最後に、画面に表示されるすべてのページ要素について、コンテンツとスタイルに関する情報の両方を含むレンダリングツリーを作成します。



CSSOMはレンダーツリーに大きな影響を与えることができますが、DOMツリーには影響しません。



レンダリング



レンダリングツリーとページレイアウトが作成された後、ブラウザは画面上に要素の表示を開始できます。 これが、このプロセスの外観です。



  • ページレイアウトを作成します 。 このステップでは、要素のサイズと画面上の要素の位置を計算します。 親要素は子要素に作用できます。 時々、子供は親に影響を与えることができます。



  • レンダリング 。 この段階で、レンダリングツリーは画像に変換されます。これは、画面に表示されるものの基礎となります。 これには、テキスト、色、画像、境界線、影の出力が含まれます。 通常、レンダリングは複数のレイヤーで実行され、ページのJavaScriptコードがDOMに影響する場合、ページを何度も再描画できます。



  • レイヤーを平坦化します 。 この段階で、レイヤーがマージされて最終的な画像が形成され、画面に表示されます。 同時に、ページ要素は異なるレイヤーに表示できるため、レイヤーを正しい順序でまとめる必要があります。



ページのレンダリング時間は、レンダリングツリーの機能によって異なります。 さらに、要素の幅と高さが大きいほど、レンダリング時間が長くなります。

あなたが私のような人なら、おそらくCSSOMよりもDOMに精通しているので、CSSOMに注意を向けましょう。 デフォルトでは、CSSはレンダリングをブロックするリソースとして扱われることに注意することが重要です。 これは、CSSOMの準備ができるまでブラウザがレンダリングしないことを意味します。



さらに、DOMとCSSOMの間に完全な対応関係がないことを理解する必要があります。 これらの構造の違いは、DOMに不可視要素、スクリプト、メタタグ、画面に表示されないサービス情報を含むタグなどが存在するためです。 CSSOMを構築する場合、ページのグラフィック表示には影響しないため、これらすべては考慮されません。
