リポジトリ構造



まず、通常のリポジトリがどのように機能するかを見てみましょう。 リポジトリには2つのタイプがあります-いものと斜めのもの（または元のもの-フラット-フラット、私たちはそれらを考慮しません）と通常の階層的なものです。 ここでは、階層的なものを作成する方法を学びます。 したがって、通常のリポジトリはpoolとdistsの 2つのディレクトリで構成されます。



プール/

階層内のこのディレクトリには、パッケージが存在します。 本当に好きなようにレイアウトできるので、「任意」と言いますが、実際には通常、次の階層が実装されます。

プール/メイン/ q / qutim /


ここで、「main」はパッケージを参照するセクションの名前（セクションについては最初の記事を参照）、srcパッケージの名前の最初の文字、srcパッケージにちなんで名付けられたディレクトリです。 srcパッケージとそこから収集されたすべてのバイナリパッケージもこのディレクトリにあります。 パッケージ名がすべて「l」ではなく「lib」で始まる多数のライブラリの場合、「a」は「lib」に続く名前の最初の文字である「liba」が使用されます。



dists /

このディレクトリは、はるかに複雑に編成されています。 まず第一に、リポジトリの名前でディレクトリを見つける必要があります-Ubuntuではこれらはさまざまな「intrepid」、「intrepid-proposed」、「jaunty」などです。debianでは「etch」、「etch-proposed-updates」です、「レニー」とその同義語-「安定」、「テスト」など-通常、これらは正規名へのシンボリックリンクです。 したがって、パッケージは一度に複数のリポジトリの1つの場所に保存できます。



dists /不安定/ [1]

このディレクトリ内には、次のファイルがあります。

  • Contents-arch-すべてのパッケージのすべてのファイルのリストを含むファイル。 「arch」の代わりに、パッケージアーキテクチャの名前が置き換えられ、最後に「.gz」と「.bz2」もあります（ファイルがそれに応じて圧縮される場合）。 これはオプションであり、主にapt-fileなどのユーティリティによって使用され、必要なファイルが含まれているパッケージを検索します。 サンプル名-Contents-amd64.gz
  • リリースは、リポジトリの簡単な説明とdists /に含まれるすべてのサポートファイルのハッシュを含むファイルです。
  • Release.gpgは、特定の作成者によるファイルの作成を証明するリリースファイルのgpg署名です。 両方のファイルはオプションですが、APTが存在しない場合、認証されていないリポジトリを誓うことが保証されています。


これに加えて、同じディレクトリに、リポジトリセクションの名前に対応する名前のディレクトリがあります。メイン、contrib、non-freeまたはmain、universe、multiverse、restricted、または自分で定義したものです。



dists /不安定/メイン/

（もちろん、メインだけでなく）各セクションでは、次の形式の一連のディレクトリを確認できます。

  • binary-archは、対応するアーキテクチャのパッケージを記述するためのディレクトリです。たとえば、 binary-amd64はamd64アーキテクチャ用です。 すべてのアーキテクチャパッケージ（devパッケージなど）は、各アーキテクチャのディレクトリに記載されます。
  • source -srcパッケージはこのディレクトリに記述されています
  • i18n-このディレクトリはめったに使用されず、主に公式リポジトリでのみ使用されます-パッケージの説明を他の言語に翻訳したファイルが含まれています。 このディレクトリはあまり重要な要素ではないので、説明しません。使用したい場合は、リポジトリのubuntuを常に確認できます。


dists /不安定/メイン/ binary-i386 /

これは、Packagesファイル（およびPackages.gzおよびPackages.bz2 ）が格納される場所であり、対応するアーキテクチャのバイナリパッケージの説明が含まれています。 本質的には、特定のアーキテクチャに接続されたリポジトリのすべてのバイナリパッケージからの制御ファイルです。 多くの場合、dists / unstable /の同じファイルとは異なり、ファイルハッシュは含まれず、このディレクトリのリポジトリの説明のみを保存するリリースファイルを見つけることができます。



dists /不安定/メイン/ソース/

ここでは、パッケージという単語の代わりにソースという単語が使用されていることを除いて、すべてがbinary-archと同じです。



apt-ftparchive



もちろん、これらのファイルはすべてペンで書くことができますが、お勧めしません。 特に、少なくとも3〜4個のパッケージがある場合。 したがって、apt-utilsパッケージのapt-ftparchiveマジックユーティリティと、dpkg-devパッケージのdpkg-scanpackagesを使用します。

したがって、前のシリーズで作成した特定のqutim srcパッケージと、そこから組み立てられたqutimおよびqutim-devバイナリパッケージがあるとします。 2つのアーキテクチャ用にそれらを収集したので、次のファイルセットができました。

qutim_0.1.99.138.orig.tar.gz

qutim_0.1.99.138-1.diff.gz

qutim_0.1.99.138-1.dsc

qutim_0.1.99.138-1_amd64.deb

qutim_0.1.99.138-1_i386.deb

qutim-dev_0.1.99.138-1_all.deb





私たちが思い出すように、最初の3つのファイルはsrc-packageであり、残りはアセンブルされたバイナリパッケージです。 また、パッケージアセンブリ中に生成されたqutim_0.1.99.138-1_i386.changesおよびqutim_0.1.99.138-1_amd64.changesファイルもあります。

それらから最小限のまともなリポジトリをまとめましょう。

したがって、初期構造でディレクトリを作成し、パッケージをそこにコピーします。

$ mkdir -p rep/dists

$ mkdir -p rep/pool/main

$ cp qutim_0.1.99.138.orig.tar.gz qutim_0.1.99.138-1.diff.gz qutim_0.1.99.138-1.dsc qutim_0.1.99.138-1_amd64.deb qutim_0.1.99.138-1_i386.deb qutim-dev_0.1.99.138-1_all.deb rep/pool/main/

$ cd rep





ご覧のとおり、パッケージ自体をリポジトリに配置するのに複雑なことはありません。 これらのパッケージの説明のセットを生成するのはもう少し難しくなります。

$ mkdir -p dists/unstable/main/binary-amd64

$ mkdir dists/unstable/main/binary-i386

$ mkdir dists/unstable/main/source

$ dpkg-scanpackages -a amd64 pool/main >dists/unstable/main/binary-amd64/Packages 2>/dev/null

$ dpkg-scanpackages -a amd64 pool/main | gzip -c9 >dists/unstable/main/binary-amd64/Packages.gz 2>/dev/null

$ dpkg-scanpackages -a amd64 pool/main | bzip2 -c9 >dists/unstable/main/binary-amd64/Packages.bz2 2>/dev/null

$ dpkg-scanpackages -a i386 pool/main >dists/unstable/main/binary-i386/Packages 2>/dev/null

$ dpkg-scanpackages -a i386 pool/main | gzip -c9 > dists/unstable/main/binary-i386/Packages.gz 2>/dev/null

$ dpkg-scanpackages -a i386 pool/main | bzip2 -c9 > dists/unstable/main/binary-i386/Packages.bz2 2>/dev/null

$ apt-ftparchive sources pool > dists/unstable/main/source/Sources 2>/dev/null

$ apt-ftparchive sources pool | gzip -c9 > dists/unstable/main/source/Sources.gz 2>/dev/null

$ apt-ftparchive sources pool | bzip2 -c9 > dists/unstable/main/source/Sources.bz2 2>/dev/null





ここでは、各アーキテクチャとsrcパッケージの説明を生成し、その後、対応するファイルに記述します。 各パッケージには 、 Packages 、 Packages.gz 、 Packages.bz2 （ソースと同様）の3つのファイルが生成されることに注意してください。 原則として、これは必要ではありませんが、私たちは文化人であり、3つすべてを生成します。好きなものを選択してください。 -aオプションは、プール/メインのセット全体から選択するアーキテクチャのパッケージを毎回示します。 先ほど言ったように、qutim-devパッケージは

理論的には、バイナリパッケージの記述の生成はapt-ftparchiveを使用して実行できますが、出力にすべてを表示する代わりに特定のアーキテクチャのパッケージを選択するオプションが見つかりませんでした。

次に、リリースファイルを生成する必要があります。

$ echo "Archive: unstable" > dists/unstable/Release

$ echo "Suite: unstable" >> dists/unstable/Release

$ echo "Components: main" >> dists/unstable/Release

$ echo "Origin: qutim.org" >> dists/unstable/Release

$ echo "Label: qutim.org Debian repository" >> dists/unstable/Release

$ echo "Architectures: amd64 i386" >> dists/unstable/Release

$ echo "Description: Debian qutIM unstable" >> dists/unstable/Release

$ apt-ftparchive release dists/unstable >> dists/unstable/Release





あなたはまだあなたの手で説明を書く必要があります、apt-ftparchiveはこれを行う方法を知りません。

最後のステップは、リポジトリに署名することです。

$ gpg -abs -o dists/unstable/Release.gpg dists/unstable/Release





これで、リポジトリを使用できます。 /etc/apt/sources.listに次の行を追加します。

debファイル：/// path / to / rep unstable main

deb-srcファイル：/// path / to / rep unstable main


パッケージのリストを更新します-リポジトリを使用できます。

もちろん、これはあまり便利ではありません。パッケージのセットを更新するたびに記述ファイルを再生成する必要があります。 もちろん、これらはすべてスクリプトによって自動化されていますが、それでも面白くありません。 したがって、使用する2番目のプログラムに進みます。これはすべての処理を行います。



リプレプロ



このプログラムは、一般的にリポジトリの構造を気にしたくないが、一度設定して使用したい人にとっては楽園です。 したがって、自家製のリポジトリを破壊し、repreproパッケージをインストールします。

$ cd .. && rm -rf rep/*

$ sudo apt-get install reprepro

$ mkdir rep/conf





rep / confディレクトリにディストリビューションファイルを作成します。その中にリポジトリを記述します。

コードネーム：lenny

スイート：不安定

バージョン：不安定

起源：qutim.org

ラベル：qutim.org Debian Repository

説明：qutim.org Debianリポジトリ

アーキテクチャ：ソースi386 amd64

コンポーネント：メイン

SignWith：デフォルト

DebIndices：パッケージリリース。 .gz .bz2

DscIndices：ソースリリース。 .gz .bz2

内容：。 .gz .bz2


コード名には、リポジトリの名前（lenny、hardy、etch、jaunty）、およびSuite-シンボリックリンクを作成するための同義語（安定、不安定）を含める必要があります。 Version、Origin、Label、およびDescriptionパラメーターは特に意味がなく、単にReleaseファイルにコピーされます。 SignWithパラメーターは、R​​elease.gpgファイルを作成してリポジトリに署名する必要があることを示します。 DebIndices、DscIndices、およびContents-パッケージ、ソース、Contents-archファイルを作成する必要があることを、シンプルでアーカイブされた形式で示します。

これで、パッケージを追加できます。

$ reprepro -b rep/ createsymlinks

$ reprepro -b rep/ --ask-passphrase -C main include unstable qutim_0.1.99.138-1_amd64.changes

$ reprepro -b rep/ --ask-passphrase -C main includedeb unstable qutim_0.1.99.138-1_i386.deb





-bスイッチはリポジトリの場所をrepreproに指示します。--ask-passphraseコマンドはgpg-keyがパスワードで保護されていることを報告します。 ここで興味深い点が1つあります。 includeコマンドは、ファイルを含めるリポジトリの名前と、srcパッケージのアセンブリ中に作成された.changesファイルの名前を引数として取ります。 同時に、このアーキテクチャのsrcパッケージとアセンブルされたバイナリパッケージがリポジトリに追加されます。 そして、ここに問題があります-reprepro開発者は、既存のバージョンのリポジトリにパッケージを追加することをきっぱりと拒否します。 そのため、別のアーキテクチャの.changesファイルを指定しようとすると-エラーが発生します-srcパッケージだけでなく、すべてのアーキテクチャ用にコンパイルされたqutim-devパッケージも既に追加されています。 この場合、includedebコマンドを使用すると、異なるアーキテクチャのdebファイルのみを指定できます。

repディレクトリの内容を今すぐ参照します。 confサブディレクトリにはリポジトリ設定が含まれ（詳細は ` man reprepro`にあります ）、dbディレクトリには内部データベースが含まれます。 しかし、poolディレクトリとdistsディレクトリは、apt-ftparchiveを使用して作成したものとは大きく異なり、冒頭で説明した構造と似たものになりました。 sources.listにリポジトリのアドレスがすでにあるので、 ` sudo apt-get update`と入力して:)を使用します



最後に、リポジトリの構造をさらに詳しく調べたい場合は、 公式のUbuntuリポジトリを調べることをお勧めします 。 もちろん、希望する人はDebianリポジトリに精通することもできますが、私はすぐにそれがはるかに複雑であることを警告します。
