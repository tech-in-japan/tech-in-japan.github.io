Visual C ++でWindowsランタイムコンポーネントを作成する

C＃の荒野を通るでこぼこ道と、ある時点でのWindowsランタイム向けのC ++ / CX開発の茂みにより、WinRTおよびCOMアプリケーションとコンポーネントを簡単に作成できるWRLテンプレートライブラリに至りました。 この特定のライブラリを操作するとき、コードがそれ自体の下に隠れるものを見つけたいと思いました。 



#include "pch.h" #include "RAWinRT.WRL.h" using namespace Microsoft::WRL::Wrappers; using namespace Microsoft::WRL; using namespace ABI::RAWinRT::WRL; using namespace ABI::Windows::ApplicationModel::Background; class ABI::RAWinRT::WRL::TestTask : public RuntimeClass < RuntimeClassFlags<WinRt>, IBackgroundTask > { InspectableClass(RuntimeClass_RAWinRT_WRL_TestTask, BaseTrust); public: STDMETHODIMP Run(IBackgroundTaskInstance *taskInstance) override { return S_OK; } }; ActivatableClass(TestTask);


これらの不可解なマクロ、テンプレート、ライブラリ関数。

そして、私は最も単純なものから始めることにしました。 Visual C ++でバックグラウンドタスクの単一の「クラス」を持つWindowsランタイムコンポーネントを記述します。



あなたがそれから来たものに興味があるなら、猫へようこそ。



コンポーネントプロジェクトの作成とカスタマイズ



最初に、Visual Studio 2013 IDEで空のソリューションファイルを作成し、WindowsストアアプリケーションのDLLプロジェクトを追加しました。











プロジェクトでは、NMSPC.TestComponentという名前を選択しました。ここで、NMSPCは名前空間です。 このような命名は、プロジェクトを作成する際のかなり一般的な慣行であるため、デモ目的でこれを行いました。 また、デフォルトのネームスペースをNMSPC_TestComponentから対応するプロジェクト名に変更しました。







ファイルについては、短い名前を好むため、ヘッダーファイルとソースファイルの名前をTetsComponentに変更しました。 コードにコンポーネントを実装する前に、いくつかの追加ファイルを追加しました。 TestComponent.def-動的ライブラリによってエクスポートされる関数を定義するためのファイル、TestComponent.idl-インターフェイスを記述するためのファイル











これらのファイルをプロジェクトに追加したら、構成を開始しました。 各構成の設定を個別に変更しないようにするには、すべての構成とプラットフォームを選択して、パラメーターの編集に進むだけで十分です。 アラートレベル設定が指定され、メタデータ生成オプションが指定され、MIDLコンパイラーによって生成されたヘッダーファイルの名前テンプレートが変更され、runtimeobject.libのレイアウトが追加され、サブシステムが選択されました。



























次に、プロジェクトを構築するための追加ステップを設定しました。 彼についてもう少し説明します。











このステップは、プロジェクトメタデータの正しい生成を目的としています。 コマンドラインは次のように設定されました。 



 del "$(OutDir)$(TargetName).winmd" && mdmerge -partial -i "$(OutDir)." -o "$(OutDir)Output" -metadata_dir "$(WindowsSDK_MetadataPath)" && del "$(OutDir)*.winmd" && copy /y "$(OutDir)Output\*" "$(OutDir)" && rd /q /s "$(OutDir)Output"


少数の連続したステップで構成され、各ステップは特定のタスクを実行します。
  1. 宛先フォルダーからNMSPC.TestComponent.winmdプロジェクトメタデータファイルを削除します。
  2. メタデータファイルを結合します。 結果は、出力フォルダーの$（OutDir）に配置されます。
  3. Ouputフォルダーから$（OutDir）フォルダーにメタデータファイルをコピーします。
  4. コンテンツと共に出力フォルダーを削除します。
これらの準備手順をすべて完了した後、ようやくコードの記述を開始できました。



DEF、MIDL、PCH



自尊心のあるWindowsランタイムコンポーネントライブラリは、ランタイムによって使用される2つの非常に重要な関数、DllGetActivationFactoryおよびDllCanUnloadNowをエクスポートする必要があります。 これらの関数のエクスポートは、TestComponent.defファイルで定義されました（コードで実装する必要がありますが、これについては後で詳しく説明します）。 



 EXPORTS DllGetActivationFactory PRIVATE DllCanUnloadNow PRIVATE


次に、TestComponent.idlファイルでクラスインターフェイスについて説明しました。 



 import "Windows.ApplicationModel.Background.idl"; namespace NMSPC { namespace TestComponent { [version(1.0)] [activatable(1.0)] [marshaling_behavior(agile)] [threading(both)] runtimeclass TestBackgroundTask { [default] interface Windows.ApplicationModel.Background.IBackgroundTask; }; } }


最初のディレクティブは、Windows :: ApplicationModel :: Background :: IBackgroundTaskのバックグラウンドタスクインターフェイスの説明を含むファイルをインポートします。 このファイルはMIDLコンパイラに十分なので、他のインターフェイス記述ファイルをインポートする必要はありません（Windowsストア8.1プラットフォームの場合、インターフェイス記述ファイルとヘッダーファイルはC：\ Program Files（x86）\ Windows Kits \ 8.1 \ Include \ winrtにあります） 。 クラスの名前空間は、プロジェクト名NMSPC :: TestComponentに従って選択されました。 属性を使用して、クラスのバージョン（バージョン）、デフォルトコンストラクターの存在の兆候（起動可能）、スレッドの操作（スレッド化）およびマーシャリング（marshaling_behavior）を設定しました。 MIDLコンパイラを使用してこれをコンパイルすると、TetsComponent.hヘッダーファイルが取得されました。



コンパイル時間を短縮するため、pch.hファイル（プリコンパイル済みヘッダーファイルの生成に使用）にactivation.hおよび新しいヘッダーファイルを含めるためのディレクティブを追加で発行しました。 これらのヘッダーファイルを含める必要があることは、IActivationFactoryインターフェイスの依存関係とstd :: nothrow定数によって説明されています。 



 #pragma once #include "targetver.h" #ifndef WIN32_LEAN_AND_MEAN #define WIN32_LEAN_AND_MEAN // Exclude rarely-used stuff from Windows headers #endif #include <windows.h> #include <activation.h> #include <new>


コードにクラス、ファクトリ、エクスポートされた関数を実装するために残っただけです。

C＃アプリケーション



Blank Appテンプレートを使用してNMSPC.CSTestApppアプリケーションプロジェクトを作成したら、コンポーネントプロジェクトとMicrosoft Visual C ++ 2013ランタイムパッケージへのリンクを追加しました。















残ったのは、アプリケーションマニフェストファイルを編集して、バックグラウンドタスクの定義を追加し、バックグラウンドタスクを登録するコードを書くことだけでした。







コードは、AppクラスのOnLaunchedメソッドに配置されます。 コードは単純です。最初にすべてのタスク登録を削除してから、タスクビルダオブジェクトを作成し、マニフェストで指定されたトリガーを設定して、タスクを登録します。 



 foreach (var pair in BackgroundTaskRegistration.AllTasks) { pair.Value.Unregister(true); } var taskBuilder = new BackgroundTaskBuilder { Name = "TestBackgroundTask", TaskEntryPoint = "NMSPC.TestComponent.TestBackgroundTask" }; taskBuilder.SetTrigger(new SystemTrigger(SystemTriggerType.TimeZoneChange, true)); taskBuilder.Register();


C ++コードのブレークポイントに移動できるようにするために、アプリケーションプロジェクトのデバッグ設定でプロセスタイプMixed（Managed and Native）を設定します。 ところで、この設定はC ++ / CXアプリケーションにも関連しています。







これで、デバッグモードでアプリケーションを起動し、コンポーネント登録コードを実行し、[デバッグの場所]セクションの[ライフサイクルイベント]ボタンを使用してバックグラウンドタスクの起動をテストできました。







これを実行すると、出力ウィンドウに非常に大事な行が表示され、その出力はOutputDebugStringW関数を使用してC ++コードでプログラムされました。 



 Hello from DLL. Hello from background task.
