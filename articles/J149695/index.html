「現在システムで何が起こっているのですか？」 このスクリプトまたはそのスクリプトはどのくらいの頻度で実行されますか？ 最も時間がかかる操作は何ですか？” Zabbixタイプの監視システムは、アプリケーションの状態の一般的な表面的な画像のみを提供します。



これらの質問に対する回答を得るために、Badbaはリアルタイムの監視および統計サービスPinbaを開発しました。 この記事では、システムの監視とプロファイリングに使用した経験について説明します。



Badooについて一言



タスクの難易度を評価するために、Badooがどのようなものかを簡単に説明します。 現在、2つのデータセンターがあり、1つはアメリカに、もう1つはヨーロッパにあります。 何百ものPHPサーバー、MySQLサーバー、C / C ++で書かれた数十のmemcachedおよび「自己作成」サービスがデータセンターにインストールされています。 Badooには独自のサブプロジェクト（たとえば、Webサイト、モバイルデバイスとデスクトップ用のアプリケーション）があり、実装と負荷の性質は大きく異なります。 1つのPHPリクエストで、多数の異なるデータソース（データベース、キャッシュ、サービス）とのやり取りが発生する可能性があります。 これはパフォーマンスには悪いですが、ビジネスタスクの解決には必要です。 また、これに加えて、新しいコードを週に2〜5回投稿します。 このようなシステムのパフォーマンスを監視することは簡単な作業ではありません。 ピンバは、PHPスクリプトのパフォーマンスの監視に関連する問題の大きな層の解決を支援しました。 このサービスは、処理されたすべてのPHPリクエストに関する情報を収集し、このデータに関する詳細なレポートを生成します。



仕組み



Habrahabrには、Pinbaでの作業を開始する方法に関する記事が既にありました。「 Pinba-PHPをリアルタイムで監視する 」。 その中で、著者はPinbaのインストール方法と構成方法を説明したので、今度はもっと興味深いことに集中します。



Pinbaを使用する一般的なスキームを図に示します。







ピンバは、PHPモジュール（英語のPHP拡張機能 ）とサーバーで構成されています。 Pinbaモジュールは、処理されたすべてのリクエストに関するデータをサーバーに自動的に送信します。 ピンバサーバーは、Cで記述されたサービスです。

  • UDP経由で処理された要求に関するデータを収集します。
  • このすべてのデータをメモリに保存し、レポートを作成します。
  • MySQLプロトコルを使用して生データとレポートへのアクセスを提供します。つまり、実際にはMySQLエンジンです。


現在システムで何が起こっているのかを知るには、MySQLクライアントを使用してサーバーに接続し、レポートテーブルからいくつかのクエリを実行します。 たとえば、各PHPスクリプト（Pinbaで最も使用されているレポートの1つ）のリクエストに関する統計を表示できます。 

mysql> SELECT * FROM report_by_script_name;




以下では、Pinbaでレポートを操作する方法について詳しく説明します。



PinbaのPHPモジュールの仕組み



Pinbaサービスの最初のバージョンの計画段階でさえ、PHPコードに変更を加えることなく、一般的な統計情報の収集を整理するというアイデアがすぐに思いつきました。 PHPモジュールの作成経験がある人は、後者が各リクエストの前（フェーズPHP_RINIT）および処理後（フェーズPHP_RSHUTDOWN）にいくつかのアクションを実行できることを知っています。 たとえば、PHP_RINITフェーズでは、モジュールは内部リソースを初期化でき、RSHUTDOWNではリソースを解放できます。 PinbaのPHPモジュールはこれらのフェーズの両方を使用します。 RINITでは、初期化され、RSHUTDOWNでは、要求の処理に費やされたリソースの数が計算され、すべてのデータがUDPパケットによってPinbaサーバーに送信されます。



UDPパケットをCに送信するのは十分に速いため、PinbaモジュールはPHPプロセスに余分な負荷を追加しません。 また、Pinbaはモジュールとして機能するため、インストールと構成を行うだけで十分です。PHPコードを変更することなく、すぐに、各スクリプトのリクエスト数、平均時間などのシステムのパラメーターを見つけることができます。などなど。



残念ながら、複雑なシステムがある場合、共通のパラメーターでは十分ではありません。



タイマー



最初は、Pinbaの最初のバージョンにはタイマーがありませんでした（Pinba v0.1と呼びましょう）。 このバージョンでは、特定のスクリプトが呼び出される頻度、平均応答時間、バイト単位の平均応答サイズなどがわかります。 MySQL、memcached、またはその他のサービスへのクエリの頻度を見つけることは不可能でした。 それほど重要ではないように思えるかもしれませんが、私たちのシステムでは、MySQLとmemcachedに加えて、C / C ++で書かれた多数のサービスがあります。



システムに問題が発生した場合（たとえば、何かが「スローダウン」し始めた場合）、Pinba v0.1は問題があることを示しましたが、それが何であるかをすぐに理解できるとは限りませんでした。 ほとんどの場合、このような状況では、PHPがクエリを実行したサービス（MySQL、memcachedなど）の1つが有罪でした。



そのような問題領域を迅速に見つける方法を学ぶために、ピンバの次のバージョンでタイマーが導入されました。



実際、mysql_connect / mysql_query / memcache_connect呼び出しなどの動作時間を測定し、この時間をPinbaサーバーに送信してタイマーレポートを作成することにしました。 擬似PHPコードでは、次のようになります。 



 pinba_timer_start(array('type'=>'db::connect', 'host'=>'users.sql')); $conn = mysql_connect(...); pinba_timer_stop();


ここでは、mysql_connect呼び出しの実行時間を測定し、2つのタグをタイマーにアタッチします。これにより、後でレポートを作成できます。



すべてのタイマーは、すべてのデータとともにPinbaサーバーに送信され、そこにレポートテーブルが作成されます。 これにより、現在最も時間がかかっている操作を確認できます。 さらに、たとえば、数百のMySQLサーバーがあり、そのうちの1つだけが遅くなる場合、Pinbaは非常に柔軟なタイマーレポートスキームのおかげで同じ問題のサーバーを簡単に計算できます。



テーブルを使い始める



ピンバがアクセスできるテーブルのリストは次のとおりです。 



 mysql> SHOW TABLES FROM pinba; +--------------------------------------+ | Tables_in_pinba | +--------------------------------------+ | info | | report_by_hostname | | report_by_hostname_and_script | | report_by_hostname_and_server | | report_by_hostname_server_and_script | | report_by_script_name | | report_by_server_and_script | | report_by_server_name | | report_by_status | | request | | tag | | timer | | timertag | +--------------------------------------+ 13 rows in set (0.00 sec)


従来、すべてのテーブルは、生データとレポートを含むテーブルの2種類に分類できます。

生データの4つのテーブルがあります：リクエスト、タグ、タイマー、タイマータグ。 これらのテーブルには、最後の数分間のスクリプト、タイマー、およびタグへの各呼び出しの記録が含まれています。 負荷の高いシステムでは、これらのテーブルには何百万ものレコードが含まれている可能性があるため、直接使用することはお勧めしません。 代わりに、システムパフォーマンスを分析するのに十分な有用なデータを含むレポートテーブルを使用することをお勧めします。

タイマーとタグ



上記のレポートは、一般的なシステム分析に役立ちます。 しかし、より詳細な分析のために、タイマーとタグを使用します。



タイマーは、コードの実行時間です。 タイマーには複数のタグを添付できます。 タグにはキーと値があります。 たとえば、上記のタイマーコードでは、 



 pinba_timer_start(array('group'=>'db::connect', 'host'=>'users.sql')); $conn = mysql_connect(...); pinba_timer_stop();


mysql_connect関数の実行時間を測定し、グループとホストの2つのタグを添付します。



デフォルトでは、Pinbaにはタイマーとタグに関するレポートはありませんが、非常に柔軟に作成できます。 以下では、Badooでこれがどのように行われるかを詳しく見ていきます。



まず、タイマーを使用して、MySQL、memcached、C / C ++で記述されたサービスなど、さまざまなデータソースへのリクエストに関する統計を取得します。 グループとホスト（ホストはデータベースまたはサービスのアドレス、グループはサービスの操作）の2つのタグを持つPinbaタイマーを使用して、そのようなすべてのリクエストを「ラップ」しました。



したがって、Pinbaに送信されるタグをシステムに実装した場合、いくつかのレポートを作成できます。 まず、すべてのタグ値に関する完全なレポートになります。 groupタグを使用した例では、この操作またはその操作が呼び出される頻度と、その操作にかかる時間（合計）に関するレポートを作成できます。 これを行うには、テーブルを作成します。 



 CREATE TABLE `tag_info_group` ( `tag_value` varchar(64) DEFAULT NULL, `req_count` int(11) DEFAULT NULL, `req_per_sec` float DEFAULT NULL, `hit_count` int(11) DEFAULT NULL, `hit_per_sec` float DEFAULT NULL, `timer_value` float DEFAULT NULL ) ENGINE=PINBA DEFAULT CHARSET=latin1 COMMENT='tag_info:group'


テーブル名は任意であり、何にも影響しません。 フィールドの順序が例とまったく同じであり、ENGINEおよびCOMMENT = 'tag_info：group'の値であることが重要です。 コメント内のgroupは、レポートを作成するタグの名前です。 また、tag_infoはレポートの一種です（tag_reportもあります。これについては以下で説明します）。



テーブルを作成すると、Pinbaは自動的にデータの入力を開始します。 このレポートは次のようになります。

たとえば、このレポートは、非常に遅いcurl_requestがあることを示しています。



tag_infoタイプのレポートは、システムでの使用の概要を示します。 特定の操作が呼び出される場所をより詳細に調べるには、同じレポートを作成する必要がありますが、スクリプトの名前でグループ化されます。 これを行うには、Pinbaでテーブルを作成します。 



 CREATE TABLE `tag_report_group` ( `script_name` varchar(128) DEFAULT NULL, `tag_value` varchar(64) DEFAULT NULL, `req_count` int(11) DEFAULT NULL, `req_per_sec` float DEFAULT NULL, `hit_count` int(11) DEFAULT NULL, `hit_per_sec` float DEFAULT NULL, `timer_value` float DEFAULT NULL ) ENGINE=PINBA DEFAULT CHARSET=latin1 COMMENT='tag_report:group'


テーブル構造は、以前のtag_info_groupレポートと多くの点で似ています。 script_nameフィールドが表示され、COMMENTフィールドにtag_reportタイプのレポートを作成したことに注意してください。 このテーブルから、前のものと同じデータをすべて取得できますが、 script_nameでグループ化されます 。



特定のスクリプトで実行される操作を確認できます。

この例は、 hit_countフィールドとreq_countフィールドの違いを示しています 。 Req_countは、指定されたtag_valueのタイマーが少なくとも1回呼び出されたHTTP要求の数です。 Hit_count-指定されたtag_valueのタイマーが呼び出された回数。



上記の例は、スクリプト/admin/messages/all.phtmlで、memcache_getが平均30回呼び出されることを示しています。 おそらくこれはプログラマーのエラーであり、ページを最適化する必要があります。

ホストタグの一般的なレポートは、十分な情報を提供しません。 1つまたは別のホストでタイマーを呼び出す頻度を示しますが、実行された操作に関する情報はありません。 幸いなことに、Pinbaは2つのタグをグループ化してtag_infoやtag_reportなどのレポートを作成できます。 



 CREATE TABLE `tag_info_group_server` ( `group_value` varchar(64) DEFAULT NULL, `server_value` varchar(64) DEFAULT NULL, `req_count` int(11) DEFAULT NULL, `req_per_sec` float DEFAULT NULL, `hit_count` int(11) DEFAULT NULL, `hit_per_sec` float DEFAULT NULL, `timer_value` float DEFAULT NULL ) ENGINE=PINBA DEFAULT CHARSET=latin1 COMMENT='tag2_info:group,server'


このレポートを使用すると、各ホストの操作に関する統計を取得できます。 たとえば、特定のホストでの操作の統計を表示できます。

tag2_reportタイプのレポートを作成することもできます。このレポートでは、2つのタグに関する情報がスクリプトごとに分類されます。 



 CREATE TABLE `tag_report_group_server` ( `sc0ript_name` varchar(128) DEFAULT NULL, `tag1_value` varchar(64) DEFAULT NULL, `tag2_value` varchar(64) DEFAULT NULL, `req_count` int(11) DEFAULT NULL, `req_per_sec` float DEFAULT NULL, `hit_count` int(11) DEFAULT NULL, `hit_per_sec` float DEFAULT NULL, `timer_value` float DEFAULT NULL ) ENGINE=PINBA DEFAULT CHARSET=latin1 COMMENT='tag2_report:group,server'


この表が役立つ例を示します。 このシステムでは、すべてのユーザーが数百のMySQLサーバーに分散されています。 これはシャーディングと呼ばれます。 特定のタスクのための特定の中央MySQLサーバーもあります。 理論的には、すべてのWebスクリプトは「シャード」のみを照会し、中央データベースは照会しないでください。 「シャード」のホスト名は、dbs *またはdbb *で始まります。 このリクエストを使用して、他のMySQLサーバーに接続するスクリプトがあるかどうかを確認できます。 



 mysql> SELECT * FROM tag_report_group_server WHERE tag1_value='db::connect' AND tag2_value NOT LIKE 'dbs%' AND tag2_value NOT LIKE 'dbb%';

nginx +ピンバ



しかし、これだけでも十分ではありませんでした。 クライアントがHTTPリクエストを送信すると、後者はまずnginxに入り、次にPHPにプロキシします。 この時点でPHPプロセスがリクエストを受け入れて処理できない場合はどうなりますか？ たとえば、サーバーが過負荷になっている場合、またはネットワークの問題が原因で、nginxがphp-fpmプロセスに接続できませんか？ この場合、ユーザーにはエラーが表示されますが、PHPではエラーがわかりません（nginxログには記録されます）。



クライアントに返す応答（HTTPステータス）を把握するために、Pinba開発者のAnton Dovgalはnginxのモジュールを作成しました。 nginxのモジュールはPHPモジュールと同じように機能しますが、明らかな理由からタイマーはありません。 彼が作成する最も有用なレポートは、 report_by_script_and_statusおよびreport_by_statusです。



プロット



ピンバは最後の数分間のデータを保存します。 この時間は設定で変更できます 。 しかし、これはシステムのパフォーマンスが時間とともにどのように変化するかを監視したい場合には十分ではありません。 これを行うには、Pinbaレポートデータ（必ずしもすべてではない）を長期間保存し、グラフを作成する必要があります。 したがって、システムとその個々のスクリプトの負荷のダイナミクスを観察したり、PHPコードがデータベース、memcached、およびその他のデータソースをロードする方法を監視したりできます。



ピンバは、長期間にわたってデータをプロットおよび保存する方法を知りません。 このためには、個別のユーティリティを使用する必要があります。 ピンバを監視し、その値の一部に基づいてグラフを作成するようにZabbixを設定できます。 RRDtoolやGraphiteなどのリング（サイクリック）データベースを使用できます。 当社では、RRDtoolを使用しています。



RRDtoolは簡単に説明していません。 彼には多くのニュアンスがあり、チャート作成の好みは人によって異なるため、このトピックは別の記事に値します。 インターネットには、このユーティリティセットをゼロから始めるのに役立つ十分なガイドが既にあります。



結果



ピンバは私たちに何をくれましたか？ ほとんどのパフォーマンスの問題は、その実装前よりもずっと早く検出され始めました。 スクリプトがユーザーにより頻繁に要求されるようになった場合、ピンバで簡単に見つけることができます。 サービスの1つ（MySQL、memcachedなど）の動作が突然遅くなった場合、Pinbaの適切なレポートを使用すると、このサービスを簡単にローカライズできます。 また、リクエストの頻度が高いために「スローダウン」し始めた場合、Pinbaのおかげで、特定の問題サービスを頻繁にリクエストするスクリプトをローカライズできる場合があります。



新しいバージョンのコードをサーバーにアップロードした後、ピンバチャートを見ると非常に便利です。 システムの負荷がどのように変化したかを確認できます。 ただし、このためには、どのようなグラフが作成されるかを考えることが重要です。



ただし、サーバー全体の監視の代わりにPinbaを使用しないでください。 たとえば、Zabbixを使用して、すべてのサーバーのCPU、メモリ、ディスク、およびネットワークアクティビティの消費を監視します。 一方、RRDにはチャートを含むPinbaがあります。 これら2つのシステムのグラフを比較することにより、PHPリクエストの「コスト」を簡単に計算し、サーバーが1秒間に処理できるリクエストの数を見積もることができます。 そして、これにより、近い将来に新しいサーバーを購入する必要があるのか​​、コードを最適化する機会があるのか​​を計画できます。



PHPモジュールとは別に、Pinbaは決してPHPに関連付けられておらず、他の言語で書かれたシステムを監視するために使用できます。 GitHubには、既にPythonとRubyでPinbaを使用したプロジェクトがあります。
