Thinstation-バージョン2.5までのシンクライアントで「重量を失う」



こんにちは、Habr！



比較的最近、人気のあるThinstationシンクライアントの新しいバージョン、つまり2.5がリリースされました。 そして、もちろん、それは新しいバンと新しいレーキの両方、さらに新しいバージョンの最小限のドキュメントを持っています。



この記事（および初心者 、特にLinuxを初めて使用する人向け ）で、シンクライアントを迅速に構築し、その使用をかなり安全にする方法について説明します。 スマートカードの habrakat使用下で、 2Xの会社RDPクライアントとハッピーエンド。 ようこそ





問題の声明



だから私たちは持っています：

  • 誇らしげにシンクライアントと呼ばれる鉄片。 たとえば、数十台の新しいHP HSTNC-001L-TC車 。
  • シンクライアントが固執する構成済みのターミナルサーバー。 MS Windows Server 2003または2008とします。


そして今、実際に私たちが望むもの：

  • ネットワーク経由でシンクライアントをダウンロードします（つまり、ディスクレス）。
  • シンクライアントは、MS RDPバージョン6+または7をサポートします。 より安全で涼しい 。
  • また、MS RDPだけでなく、TLS 1.0もサポートしています 。
  • スマートカードを使用したユーザーの承認（つまり、シンクライアントからサーバーへのスマートカードの転送）。
  • 世界を引き継ぐ 。


どこから始めますか？



シンクライアントのディスクレスロード（およびPXEプロトコルを使用してロードされる）の場合、DHCPサーバーとTFTPサーバーを構成する必要があります。 ネットワークブート（PXE）はどのようなもので、どのように構成され、どのように適切に構成され、詳細がここに記述されています 。 Windows用のTFTPサーバーとして、 ここからダウンロードできるtftpd32をお勧めします 。 名前にもかかわらず、x64プラットフォーム用のバージョンがあります。



さらに、必要に応じて、 こちら 、 こちら 、 こちら （ダウンロードするファイルのリストの下）でThinstationについて少し読むことができます。 ロシア語の情報はここで見つけることができますが、すでにいくらか古くなっています。 ここでは、Thinstationイメージバージョン2.2.2の作成と構成を見ることができます。これは2.5にはるかに関連しています。 直接バージョン2.5は、これまで1ページのみに専念しています 。 それでは始めましょう。



最初のビルド



ThinstationはLinuxベースであるため、シンクライアントを構築するにはLinuxがインストールされたコンピューターが必要です（ありがとう、KO！）。 Ubuntu 11.10を使用しました。 また、Gitをインストールし（まだインストールされていない場合）、それを使用してイメージジェネレーターでリポジトリを複製する必要があります。 



sudo apt-get install git-core cd /home/user/ git clone --depth 1 git://thinstation.git.sourceforge.net/gitroot/thinstation/thinstation cd thinstation


イメージジェネレーターをダウンロードしたら、スクリプトを実行する必要があります。 



 ./setup-chroot


最初の起動時に、このスクリプトは必要なパッケージを収集し、ブートイメージをさらに生成するためにインフラストラクチャ全体を展開します。



最初の、今までの「脂肪」画像を収集する時が来ました。 ハードウェアの非常に幅広いリストをサポートするこの大きなイメージは、特定のハードウェアをサポートする小さなプロファイルを生成するために必要です。 これは、新しいバージョンのThinstationの主な利点の1つです。イメージに含める必要があるドライバーのリストを手動でコンパイルする必要はありません。スクリプトによって自動的に生成されます。



開発者がアドバイスしているように、アセンブリはchrootセッション内で行う必要があるため、 setup-chroot.shスクリプトを終了せず（「Q」を押してスクリプトのウェルカムメッセージを非表示にします）、コンソールで次のコマンドを記述します。 



 cd ts/2.5 nano build.conf


build.confファイルで、 「 package extensions 」行のコメントを外します。 プロキシ経由でインターネットを使用している場合は、「 param httpproxy 」行のコメントを解除し、プロキシサーバー設定を指定します（たとえば、「 param httpproxy user ：password @ proxy：port 」）、ファイルを保存してアセンブリを続行します。 



 ./build --allmodules


ビルドスクリプトログの長い足跡を見て、彼が要求した場合は追加のパッケージをダウンロードすることに同意し、プロセスが完了するのを待ちます。 ディレクトリ「 /home/user/thinstation/ts/2.5/boot-images/pxe 」（およびこれがアセンブルされたブートイメージ）の内容をTFTPサーバーのルートにコピーし、ネットワーク経由でシンクライアントを初めてロードしようとします。



そして、ここで最初の待望のレーキに出会うことができます。 シンクライアントに十分なRAMがない場合は、 build.confファイルの編集に戻り、「 #package chrome 」などの重いパッケージをコメントアウトし、アセンブリを繰り返して、概要がほぼ2倍に減少することを確認します。 これでダウンロードが開始されます。



その後でも、100％に近い確率で、シンクライアントは完全にはロードされません。 しかし、これは必要ありません。 ブートローダーが「Thinstation」という碑文とプログレスバーを含む画像を表示するのを待っています。 その後、Ctrl + Alt + F3を押すと、コンソールに入力の招待が表示されます。 次のログインとパスワードのペア「 root-pleasechangeme 」を入力して、スクリプトを実行します。 



 hwlister.sh


このスクリプトは、シンクライアントの特定のハードウェアのプロファイルファイルを生成します。 通常、それらの2つがあります： " module.list "（ハードウェアのドライバーのリスト）および " vbe_modes.list "（グラフィックモード）。 次に、それらをLinuxマシンにコピーする必要があります。 これは、たとえばTFTPサーバーを介して実行できます（記録を許可する必要があります）。 シンクライアントコンソールで、次を入力します。 



 cd / tftp -p -l module.list -r module.list 192.168.0.1 tftp -p -l vbe_modes.list -r vbe_modes.list 192.168.0.1


、ここで192.168.0.1はTFTPサーバーのアドレスです。 Linuxマシンに戻って、「 /home/user/thinstation/ts/2.5/ machine /my_machine 」フォルダーを作成し、受け取った2つのファイルをTFTPサーバーのルートからコピーします。



怖い獣-スマートカード



したがって、ユーザーがスマートカードを使用してシステムにログインできるようにするには、いくつかのことが必要です。

  • 実際、スマートカード自体。 たとえば、 そのような 。
  • スマートカードリーダー（カードリーダー）。 たとえば、 そのような 。
  • そしてもちろん、ターミナルサーバーは、認証にスマートカードを使用するように適切に構成する必要があります。 このサーバーのセットアップは大きなトピックです。 それについては、あなたは別々に見ることができます 。 さらに、特定のスマートカードをサポートするには、製造元のソフトウェアをターミナルサーバーにインストールする必要があります。 例として選択したアラジンカードの場合、 ここにあります 。 この段階では、ターミナルサーバーを既に構成し、ユーザーがスマートカードを使用してログインできるようになっていると想定しています。


次に、Linuxカードリーダー用のドライバーを見つけてビルドする必要があります。 製造元のWebサイトで、ドライバーをここで見つけ、ダウンロードして解凍します。 



 cd /home/user/ wget http://www.athena-scs.com/downloads/asedriveiiie-usb-3.7.tar.bz2 tar -xjf asedriveiiie-usb-3.7.tar.bz2 cd asedriveiiie-usb-3.7


READMEを読んで、アセンブリのためにPCSC Liteパッケージをインストールする必要があることを確認します（その時点でccid-1.4.5の最新バージョンをインストールしました）。また、 libusb-0.1.12ソースコードも必要です（古いバージョンでは使用されません ） 。



PCSC Liteを、カードリーダーのソースドライバーが含まれるフォルダーに入れ、 libusbソースからusb.hファイルをコピーします。 今、通常の実行： 



 ./configure make make install


Thinstationには既にPCSC Liteパッケージが含まれているため、次のようにドライバーをThinstation Builderにコピーするだけです。 



 cp -LR /usr/lib/pcsc/drivers/ifd-ASEDriveIIIe-USB.bundle /home/user/thinstation/ts/2.5/packages/ccidreader/lib/pcsc/drivers cp /etc/udev/rules.d/50-pcscd-asedriveiiie.rules /home/user/thinstation/ts/2.5/packages/ccidreader/etc/udev/rules.d


すべて準備完了です！ これで、シンクライアントをロードすると、カードリーダーが検出され、正常に動作するようになります。 バージョン2.5では、2.2.2のようにスマートカードを操作するためのこのような歪みはもう必要ありません。



RDPクライアント



ターミナルサーバーに接続するクライアントについて少し説明します。

現在、Linuxシステム用のMicrosoft RDPの最も有名なクライアントはrdesktopとそのフォークFreeRDPです。 しかし！ rdesktopはTLS 1.0をサポートしていないため、 FreeRDPはスマートカードを使用できません。 そして、それは完全な悲しみを引き起こします！



長い検索の後、別の2X RDPクライアントが発見されました。 こちらからダウンロードできます。 彼は上記のすべてを行うことができ、無料であり、MS RDPバージョン7.0もサポートし、積極的に開発していることが判明しました。 このクライアントがThinstationの一部であることがわかったとき、私のアカウントは何でしたか！



フィニッシュライン：構成とアセンブル



慎重な設定は大きなトピックですので、 ここで「 設定ファイル 」セクションで各ファイルが必要な理由とその場所を読みます。 この記事では、Thinstationバージョン2.2.2の構成について説明します。 ここでは、新しいバージョンの変更点について説明し、構成ファイルの例を示します： build.conf 、 thinstation.conf.buildtimeおよびthinstation.conf.network 。

したがって、例の構成のパラメーターについてコメントします。



build.conf：

  • machine my_machine-鉄のプロファイルを生成し、それを「 my_machine 」フォルダーに入れたことを覚えていますか？ これだ！
  • パッケージxorg7-vesa -Xorgドライバーを選択します。 ここで問題が発生したのは、SISチップセットのネイティブドライバーが適合せず 、残りのどれが機能するかを実際に見つけなければならなかったからです。 Vesaは私のチップセットでうまく機能します。 おそらくここでは、実際にパラメーターを選択する必要があります。
  • パッケージccidreader-スマートカードを使用できるようにするPCSC Liteパッケージ。
  • パッケージ2x、パッケージalsa-lib-これは私たちの素晴らしいRDPクライアントです。 確かに、実際には、彼が動作するにはAlsaパッケージが必要であることが明らかになったので、それを含めます。
  • param fastboot false-このパラメーターがtrueに設定されている場合 、ブートイメージはメイン部分とロード部分に分割されます。 残念ながら、私のネットワークカードは画像全体の読み込みを拒否したため、画像全体を生成します。
  • param basepath config-クライアント用の構成ファイルが配置されるTFTPサーバー上のフォルダーを示します（ thinstation.conf.networkなど）。
  • #param rootpasswdなど -パスワードを設定するパラメーターについてコメントします。 ルートパスワードがコメントアウトされている場合、Thinstationコンソールへのアクセスをだましたとしても、誰もルートの下でログインできません。 そしてこれは良いです）
  • param 2xurl -2Xクライアントのダウンロード元を設定します。 ビルドスクリプトを初めて実行するときに1回だけダウンロードされます。


thinstation.conf.buildtime：

  • DONT_VT_SWITCH_STATE = TRUE-ユーザーがCtrl + Alt + F3を使用してコンソールに切り替えることを許可しません。
  • DONT_ZAP_STATE = TRUE-ユーザーがCtrl + Alt + Backspaceを使用してグラフィックモードを再初期化して、コンソールに戻ることを許可しません。


最後に、2Xクライアントのセッション記述の例（ thinstation.conf.network ）： 



 SESSION_0_TITLE="2X" SESSION_0_TYPE=2X SESSION_0_2X_OPTIONS="-m MX -C -u user -p password -s ssl://myTerminalServerIp" SESSION_0_AUTOSTART=ON


ここで：

  • -m MX-クライアントモード、MS RDP、フルスクリーン。
  • -C-スマートカードをリダイレクトします。
  • -u user -p password-論理的には、user-password。 しかし！ パスワードではなくスマートカードを使用してログインします。 簡単です：実際には、現在の2Xクライアントはユーザーパラメーターとパスワードなしでは起動しませんが、セグメンテーションエラーが発生します。 そして、これは完全なナンセンスです。 しかし、顧客サポートとの長い会話の後、彼らは次のリリースでこの問題を解決することを約束しました。 それまでの間、存在しないユーザーとパスワードをランダムに書き込み、必要に応じてスマートカードを使用して穏やかにログインします。
  • -s ssl：// myTerminalServerIp-接続するサーバーのアドレス。 sslは、TLS 1.0が使用されることを示します。




設定が完了しました。 クライアントを収集します。setup-chroot.shスクリプトを実行して、 次のように入力します。 



 cd ts/2.5 ./build


結果のイメージをTFTPサーバーのルートに追加します。 また、TFTPサーバーのルートで、「 config 」フォルダー（ build.confで指定したフォルダー ）を作成し、 そこにthinstation.conf.networkファイルをコピーします。

すべて準備完了です！ 開始、確認、ターミナルサーバーのログインウィンドウを確認し、喜んでください！ ハッピーエンド！
