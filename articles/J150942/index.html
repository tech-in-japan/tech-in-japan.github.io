人に優しいURL（CNC）



デフォルトでは、CNCはMODXで無効になっています。 それは非常に簡単にオンになります：システム->システム設定->「friendly_urls」をフィルターに入れてEnterを押します（もちろん、ハンドルを使用して検索できますが、より高速です）->値を「はい」に切り替え、上部の「保存」をクリックします。

一般的に、MODXでは、保存ボタンが右上隅にあることを忘れないでください。 MODXでは自動保存がトリガーされる場合とそうでない場合があるため、設定を忘れてしまうことがあります。

同様に、CNCの自動生成のための関数 "automatic_alias"に対しても実行します。

CNCを正常にオンにしたので、リンクは次のように形成されます。 

 example.com/resourcename.html


最後に.htmlが挿入されていることに注意してください！ 個々のリソースのContent-Typeに依存します（リソース自体のプロパティの変更）。 コンテンツタイプ自体の設定は、システム->コンテンツタイプにあります。

ただし、追加のGETパラメーターはすべて同じままであり、これは次のように機能します。 

 example.com/news.html?date=05092012&nid=1


個人的に、私はそのようなシステムが好きではなかったので、これらのパラメーターをCNCの一部にする方法を探し始めました。 そして見つけた！

手始めに、MODXを拡張できるさまざまな要素について少しお話しする価値があります。 それらのいくつかがあります：

  • テンプレート変数-追加のリソースプロパティフィールド。 後で彼らに戻ります
  • チャンク-他のチャンク、リソース、または一般的にどこにでも挿入できるHTMLコードのブロック（MODXマークアップの可能性あり）。 注意！ ここにはPHPを挿入できません！
  • スニペットは、特定のデータを挿入された場所に出力するPHPコードのブロックです（MODXマークアップを使用）。 チャンクの一種のPHPアナログ。
  • プラグインは、MODX内で発生する特定のイベントに従って実行されるPHPコードのブロックです。 おそらくDrupalフックのようなもの。


この場合、スニペットとプラグインが必要です。

それでは、独自のURL形式でMODXにどのように統合しますか？ 実際、それは非常に簡単です-MODXに指定されたURLでページを見つけさせ、見つからない場合は制御して解析し、その後MODXをあるべき場所に送ります。 これを行うには、 OnPageNotFoundイベントのプラグインを使用します。

高度なCNCのプラグインコード 
 <?php if($modx->request->getResourceMethod()!="alias") return; $uri = $modx->resourceIdentifier; $uriChunks = explode("/", $uri); $paramNum=0; $uriChunksCount = count($uriChunks); $paramDelimiter="-"; //DO NOT use "/"! Set the desired delimiter here. for($i=$uriChunksCount-1;$i>0;$i--) { $parameter=explode($paramDelimiter, $uriChunks[$i], 2); if(count($parameter)!=2) return; $modx->request->parameters['GET'][$parameter[0]]=$parameter[1]; if(empty($modx->request->parameters['POST'][$parameter[0]])) $modx->request->parameters['REQUEST'][$parameter[0]]=$parameter[1]; $paramNum++; unset($uriChunks[$i]); $uri = implode("/", $uriChunks); if (array_key_exists($uri, $modx->aliasMap)) { $modx->sendForward($modx->aliasMap[$uri]); } } if($paramNum==($uriChunksCount-1)) { $parameter=explode($paramDelimiter, $uriChunks[0], 2); if(count($parameter)!=2) return; $modx->request->parameters['GET'][$parameter[0]]=$parameter[1]; if(empty($modx->request->parameters['POST'][$parameter[0]])) $modx->request->parameters['REQUEST'][$parameter[0]]=$parameter[1]; $modx->sendForward($modx->getOption('site_start', null, 1)); }


実際、ここではおそらく次のことをやめる価値があります。 

 $modx->request->getResourceMethod()


CNCがオンになっているかどうかを確認します。 

 $modx->resourceIdentifier


このそれほど明白ではない変数は、要求されたURLを格納します。 

 $paramDelimiter="-";


この変数は、GETパラメーターの名前とその値の間の区切り記号を設定します。 「/」以外に変更できます。 

 $modx->request->parameters['GET'][$parameter[0]]=$parameter[1]; if(empty($modx->request->parameters['POST'][$parameter[0]])) $modx->request->parameters['REQUEST'][$parameter[0]]=$parameter[1];


実際には、MODXは各リクエストでGETおよびPOSTパラメーターに関するすべての情報を独自の配列に保存し、その後、およそ次の方法で共通の配列を形成します。 

 $modx->request->parameters['REQUEST'] = array_merge($modx->request->parameters['GET'], ($modx->request->parameters['POST']);


この構造を保存するために、POST配列のデータがREQUEST配列に保存されているかどうかを毎回確認し、存在しない場合にのみREQUEST配列のデータを置き換えます。 

 if (array_key_exists($uri, $modx->aliasMap)) { $modx->sendForward($modx->aliasMap[$uri]); }


そして、これは実際に目的のリソースへのリダイレクトです。 CNCテーブル全体が$ modx-> aliasMapに保存されます。これを使用して、目的のリソースに移動する時間か、またはアンダーパースされたuriのコンテンツがまだパラメーターで構成されているかを確認します。 $ modx-> sendForwardとこのリダイレクトを実行します。



ただし、おそらく、このURL生成メカニズムを使用して、たとえば現在のページへのリンクをすぐに提供したいと思うでしょう。 これを行うために、現在のページのURLを生成するだけの簡単なスニペットを作成しました。 もちろん、生成や他のリンクのために拡張できますが、これはすでにそれを必要とする人の個人的な宿題になります-それについて複雑なことは何もありません:)

  2. OnHandleRequestイベントのプラグインを作成します（一般的な欠点：もう少しリソースが使用されます-Webコンテキストが常に最初にロードされ、その後に必要なコンテキストがロードされます）：

ここで最後のオプションのみを説明します。 私のプラグインのテキスト（思い出して、 OnHandleRequestイベント）：

 <?php if($modx->context->key!="mgr") { $object = $modx->getObject('modContextSetting', array('key' => 'multisite_http_host', 'value' => $modx->getOption('http_host'))); if($object) $modx->switchContext($object->get('context_key')); }


使用するには、コンテキストプロパティの「multisite_http_host」パラメータに、処理するドメインの値を設定する必要があります。 適切なコンテキストが見つからない場合、プラグインはデフォルトでWebコンテキストを使用します。

ここでは、2つのことに焦点を当てたいと思います。

まず、MODXを正常に使用するには、xPDOの原理を理解する必要があります。 xPDOは、データベースとPHPオブジェクト間の通信のためのメカニズムです。 PHPでオブジェクトを操作すると、データベースにアクセスできます。 MODXのほぼすべてが、xPDOを使用してデータベースと通信します。 

 $object = $modx->getObject('modContextSetting', array('key' => 'multisite_http_host', 'value' => $modx->getOption('http_host')));


このコードは、特定の条件を含むコンテキスト設定オブジェクト（つまり、httpホストの値を持つmultisite_http_host構成キー）を検索するためにデータベースに移動します。 その後、受信したオブジェクトからロードする必要があるコンテキストキーを取得します（$ object-> get（ "context_key"））。

第二に、haberuser XanderBassが正しい質問をしました。ここで私はそれを複製したいと思います。 では、なぜ「http_host」だけでなく「multisite_http_host」というパラメータ名を使用しているのでしょうか？

すべてが非常にシンプルで、システム設定、コンテキスト、ユーザーを組み合わせるメカニズムに基づいています。 実際、システム設定はコンテキスト設定によって上書きされ、それらはユーザー設定になります。

この場合、http_hostはシステム設定です。 コンテキスト設定でmultisite_http_hostの代わりにhttp_hostを使用すると、コンテキスト設定によって消去されます。 もちろん、この特定の例では、これはそれほど怖いものではありません。 ただし、このプラグインを少し書き換えて、特定のドメインのすべてのサブドメインを1つのコンテキストで処理します。 または、いくつかの異なるドメインの1つのコンテキストで処理を実行します（たとえば、ほぼこの形式のパラメーターに駆動する：sub1.example.com; sub2.example.com; sub3.example.com "など）。この場合、 http_hostでは、リクエストに使用されたhttp_hostではなく、この設定になりますが、あなたは決して知らないでしょう、元のhttp_hostがどこか他に必要になるかもしれません...



マルチコンテキスト認証


時には、複数のコンテキストでユーザーを一度に認証したいことがあります。 デフォルトでは、システムにログインする（たとえば、ログインmodを使用して）ユーザーは、1つのコンテキストのみでシステムにログインします。 次のプラグインは、承認中にいくつかのコンテキストに自動的にログインします（ OnWebLoginおよびOnWebLogoutイベント ）：

マルチ認証 
 <?php $currentSiteGroup = $modx->getOption("multisite_site_group"); if(empty($currentSiteGroup)) return; $currentContext = $modx->context->get("key"); $currentContextSettings = $modx->getCollection('modContextSetting', array('key' => "multisite_site_group", "value" => $currentSiteGroup)); foreach($currentContextSettings as $currentContextSetting) { $contextKey = $currentContextSetting->get('context_key'); if($contextKey!="mgr" && $contextKey!=$currentContext) { if($user) { if($modx->event->name=="OnWebLogout") { $modx->user->removeSessionContext($contextKey); } else if($modx->event->name=="OnWebLogin") { $modx->user->addSessionContext($contextKey); $_SESSION['modx.'.$contextKey.'.session.cookie.lifetime']=$attributes["lifetime"]; } } } }




使用するには、パラメータ「multisite_site_group」を、ユーザーが同時にログインするすべてのコンテキストのすべての必要なコンテキストに同じ値を設定する必要があります。

 $modx->user->addSessionContext($contextKey);

 $modx->user->removeSessionContext($contextKey);
