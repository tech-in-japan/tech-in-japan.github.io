データ、データ、そして再びデータ。 銀行業務分析から航空会社の交通渋滞の最適化まで。 「リアルタイム」で状況を研究することは、限られた時間枠で結論を導き、バランスの取れた決定を下す機会です。 素晴らしい例は、F1レースです。



各ステージの前に、ライダーはテストレースを実施します。 火の玉には何百ものセンサーが詰め込まれており 、マシンのすべてのコンポーネントとアセンブリのパフォーマンスを監視できます。



1台の車のセンサーは、レースごとに1 GBを超えるデータを生成します。これは、トラック上の車の位置、速度、ギアボックスの操作、さらにはタイヤの空気圧に関する情報です。 これらのデータを使用すると、車の技術的状態の最も完全な画像を取得し、誤動作を修正できます。



シーズンの終わりまでに、テラバイトのデータがレースチームに蓄積され、慎重に分析されます。 結果は、さらなるレースでのチームの成功または失敗です。 このようなタスクは、ストレージおよび処理プラットフォームによって解決されます。 たとえば、2014年のF1レースシーズンでは、マクラーレンチームはSAP HANAテクノロジーに頼りました。

SAPホスティングとは



SAPホスティングは、プロバイダーのクラウドインフラストラクチャにおけるSAPソリューションのホスティングサービスです。サプライヤーは、SAPアプリケーションの移行、管理、メンテナンス、およびメンテナンスのタスクを担当します。



SAP HANAの場合、SAPは製品の略語に対して次の略語を提供します。HighPerformance Analytical Appliance。 HANAは、トランザクション処理や大きなデータなどのツールを含む高性能分析ソリューションです。 たとえば、SAP HANAを使用すると、さまざまなソーシャルネットワークでメーターの測定値を自動的に読み取ることができます。



IT-GRAD社は 、特別な統合データセンター機器でvSphere上のSAP HANAを提供しています。 同時に、VMware vSphereでの単一インストールに対して、最大1 TBのメモリと32の物理コア（これらは64の仮想コア）を取得します。



SAP HANAをホストするためのクラウド容量の計算



ここで、SAP HANAをホストするためのクラウド容量の計算方法を見てみましょう。 一般に、SAPにはSAP HANAの構成がいくつかあり、サイズに合わせてTシャツを選んでいるかのように文字でそれらを示します。









サイズSのTシャツを試してみて、サイズが小さいことがわかった場合は、サイズを大きくすることができます。M。サイズMはLに置き換える必要があります。 同じことがリソースでも発生します。リソースが十分でない場合は、いつでも追加できます。 表に示されている「ディメンション」は、SAP HANAに基づいたほとんどのビジネスタスクを満たしますが、それらに加えて、さらにユニークで特定の構成がいくつかあります（以下を参照）。









しかし、必要なリソースの評価に進みましょう。 共有メモリから始めましょう。 そのサイズは、保存する必要があるデータの量によって決まります。 この値に基づいて、ディスクサブシステムとプロセッサのパラメーターは既に計算されています。



RAMの要件により、2つのコンポーネントが区別されます。 1つ目は、HANAが使用する静的メモリで、メモリに保存する必要があるデータの量によって決まります。 2つ目はダイナミックRAMです。データとアプリケーションの新しいブロックをロードするときに必要なRAMです。



SAPの推奨によると、同じ量の動的メモリと静的メモリを予約する必要があるため、メモリの合計量は静的RAMの量に2を掛けた値に等しくなります。



RAM d = RAM st

RAM 合計 = 2 x RAM st



次に、ディスクシステム（HDD）のリソースの計算。 同時に、予期しない事故が発生した場合に情報を保存するために、データベースからのデータセットがいわゆるデータボリュームに定期的にコピーされることに注目します。 データボリュームのボリュームは、式に従って計算されます。



DISK pers = 4 x RAM 合計



また、不可抗力の場合に「ロールバック」できるように、データベースの変更に関する情報が保存されているログのボリュームのボリュームを考慮する必要があります。 ボリュームボリュームは、次の式を使用して計算されます。



DISK ログ = RAM 合計



合計合計ディスク容量：



DISK 合計 = DISK ログ + DISK pers



データ量の急激な増加によるトラブルやパフォーマンスの低下を避けるために、主要な結果を定期的に確認する必要があることに注意してください。



次に、CPUリソースが変わります。 必要なプロセッサリソースの量は、ユーザーの数によって異なります。 SAP HANAは、ユーザーごとに300 SAPS（従来型鉄負荷ユニット）をサポートしています。 さらに、システムは1つのCPUで60〜65の同時アクティブユーザーをサポートします。 興味のある方は、SAPSに関する詳細情報をこちらとこちらでご覧ください 。 完全な構成計算の例はここにあります。



SAP HANAのリソースの計算プロセスを簡素化するために、SAPではQuick Sizerユーティリティの使用を推奨しています。 RAM、HDD、CPU、IOPSのボリュームを計算できます。 以下の画像は、Quick Sizerツールの結果を示しています。 この場合、CPU、HDDのパラメーターの値は、前述の「Tシャツのサイズ」に従って決定されます。 ツール自体はここから入手できます 。
