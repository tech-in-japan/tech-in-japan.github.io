本の書き方

私の最大の独立プロジェクト-大量の「Pro Android Web Game Apps」は、電子形式だけでなく、肉体でもリリースされました。 この記事では、出版社とのアイデアや交渉から表紙や印刷物の承認に至るまで、技術書がどのように生まれたかについて話すために、私の経験を共有したいと思います。







最初のステップと出版社との会話



本の考えが最初に思い浮かんだとき、私はそれを表明したくさえなかった。 誰かが私のアイデアの99％をすでに考え出しているので、適切に見る必要があります。 今回はそうではありませんでした-グーグルはブログやスケッチノートだけを返しました-本当に基本的なものは何もありませんでした。 そのため、アイデアにはチャンスがあります。 もしそうなら、試してみませんか？



「新しい著者のための」プロセスは、すべての主要な出版社でほぼ同じです。あなたのアイデアを私たちに説明し、それを実装する理由を説明し、私たちに電話しないでください。 交渉を始めるには、あなたとあなたの本についてフリースタイルで2〜3ページ書く必要があります。 私が書いた5つの出版社のうち3人は、私のトピックは来年の開発戦略に含まれていないと答えました。 他の2つの出版社、ApressとManningは、彼らは興味があると答えた。 Apressはより迅速に回答し、わずか数時間で私は彼らと働き始めました。

この時点で、私の主な問題は言語でした。 私はよく話し、プロジェクトのリリース、サッカー、またはエマ・ワトソンの試合について合理的に議論することができますが、本はまったく異なります。 不正確な場合は、著者が外国語で書いていることを再度思い出させます。





ポールは私を助けてくれました。イギリスの私の良き友人です。 最初に、彼は私が書いたものをすべてチェックしました-出版社への小さな手紙でさえ。 私たちが契約に署名したとき、ポールは章を読み直し、ケンブリッジの英文学教師にとっても読みやすいテキストであることを確認しました。 私は自分のテキストのためにポールの編集を渡しませんでした。 初日から、出版社は私が自分で書いていないことを知っており、そのような「タンデム」を承認しました。



私たちが発行した最初の深刻な文書は、出版社に対する正式な提案でした。 「私にはアイデアがあります。あなたにとってはどうですか」という精神で書かれた最初の手紙とは異なり、正式な提案はビジネスプランのように見える大きく詳細なテキストです。 すべてがここに示されています：各章の構造、タイトル、簡単な説明、イラストの数、他の類似の本の分析（あなたは直接競合他社を知る必要があります）。 この論文は編集委員会によって検討されますが、あなたの契約の運命はそれに依存します。



私は幸運だった-私の申し出は合格し、契約書に署名した。 すぐに申し上げます-契約の条件は非常に厳しいです。少なくとも、出版社が期限を守れなかった場合に「追加」のために本のすでに完成した部分を別の著者に転送する権利を持っているという事実を考慮してください。 ただし、実際にはこれは起こりません。 私は自分のプロジェクトをコスト以上に引きずりましたが、過酷な制裁は適用されませんでした。



誰もが著者の報酬の問題に常に関心を持っています。 この本から利益を得てヨットを購入することは期待できません。ソウルメイトとトルコのどこかでリラックスするのに十分なお金しかありません。 このボリュームの本の場合、料金の合計額は約4,000ドルです。 料金は、将来の販売に対して発行されます。つまり、この金額を超える利益を得るためには、出版社は数百部を販売しなければなりません。 アーティストから税金と手数料を差し引いたり、この金額からその他の費用を差し引いたり...このメモを書いているiPadをまだ獲得していると思います。



下書きから本まで



最も興味深い段階は、契約が承認されたときに始まり、詳細が合意され、あなたが座って書きます。 空白のシート、500ページの計画、半年または1年間の寄付された自由時間です。



本は章に分かれています。 章は、あなたとあなたの出版社が一緒に仕事をするプロジェクトの原子的な部分です。 たとえば、進捗状況を評価するために、テクニカルエディターが承認した章の数、コピーエディターが読んだ章の数などが推定されます。 ページは重要ではありません。 コードの行として-機能がまだ機能していない場合、どのくらい書いたかは関係ありません。



各章にはいくつかの段階があります。 ドラフト（最初のドラフト）から事前に作成され、印刷可能なPDFに至るまで、十数を超えるものがあります。 印刷がPDFで行われるかどうかはわかりませんが、この段階の後、著者の役割はシャンパンを飲み、彼の紙の本を待つことです。 各ステージでは、新しいチームの1人以上の人が責任を負います。 これらの人々をよりよく知りましょう。



最初に会うのはエディター、または単にエディターです。 出版社のほとんどの人は編集者です。 この役職は、普通の従業員から中堅レベルのマネージャーまでを意味します。 それらのほとんどは、編集に直接関与していません。



これから説明するエディターは、プロジェクトコーディネーターです。 サイト運営者のサービスが期待どおりに機能することを確認し、「章を2つの章に分割する必要がある」や「投資する予定のページが見つからない」などの問題も解決します。 質問があまりにも深刻な場合、彼は誰にそれを送るべきかを知っています。 彼はまた、本がどのように書かれているか、プロセスとツールについて説明し、本を書くのを助ける他のチームメンバーを紹介します。 しかし、ここでは入門的なブリーフィングを学んだので、作業を始めましょう。



最初の章を選択しました（コンテンツの最初の章である必要はありません）。テキストとコードを書きました。 コードは機能しているようです。 テキストは読まれているようです。 あまり時間をかけないように一部の詳細を省略できるため、この章を編集する機会は複数あります。 たとえば、「ここに図を追加する」や「章が書かれたら正しい章番号を参照する」などのコメントは完全に適切です。



この章を終えたら、チームの2番目の最も重要な人物である技術編集者について知りましょう。

テクニカルエディターの主なタスクは、本を著者の思考の流れではなく本にすることです。 彼はテキストの構造、わかりやすさ、言語、「人間工学」を監視しています。 テクニカルエディターは、さらに例を追加したり、図を修正したり、スクリーンショットを別の方法で配置することを勧めます。 彼はもっと過激なことをアドバイスできます。 私の場合、編集者はかつてアイソメゲームにもっと注意を払うことを提案しました。 このアドバイスに従って私が書いた2つの章は、本の小さな宝石になりました-私が最も誇りに思っているのはこれらの章です。



テクニカルエディターは、次の段階に向けて章の準備ができていると判断する人です。 言い換えれば、彼は本の進行を最も直接的に決定します。



チームの次のメンバーは技術修正者です（英語-技術レビュアーですが、文字通りの翻訳はなんとなく完全に惨めに聞こえます）。 彼はあなたが書いたものすべてを注意深く、細心の注意を払ってチェックします-各リンクを通過し、数式に従って数値を再計算し、各リストを再入力して実行し、彼の意見を伝えます。 彼は言語の構造や品質のどちらも調べません。技術的な信頼性だけを考えます。 彼は細部に非常に注意を払っています。 私の修正者-チャーリー、かつて私に余弦の正弦が現れた行列乗算の例でエラーを指摘しました。 気づくのはそれほど簡単ではありません。



編集者と校正者が作業を完了するとすぐに、改訂のために章が再び著者に戻ります。 これは、テキストが悪いという意味ではありません。 最初の試行から、印刷可能な素材を書くことは不可能です。 この章はすぐに処理することもできますが、2〜3週間そのままにして、テキストを新鮮な外観で見ることもできます。



著者と編集者がこの章の外観に満足したら、テキストは次のチームメンバーであるコピーエディターに送信されます。 彼は本のトピックについては何も知りませんが、彼の記事を素晴らしく配置し、複雑すぎる文章を見つけ、テキストを読みやすくします。 私の場合、彼にはほとんど仕事がありませんでした-ポールは素晴らしい仕事をし、テキストはとてもきれいでした。



コピーエディターの後、作成者は編集内容を再度確認し、イラストレーターに章を渡します。 Illustratorは、印刷用の画像を準備します。 私のイラストレーターは彼の作品に没頭しなければなりませんでした-写真の色のアクセントはほとんど区別できなくなり、いくつかの場所では圧縮によるアーチファクトが現れました。 二度目には、はるかに良くなった。

テキストと写真の準備ができたら、リストがチェックされます-返品のポイントが来ます。 次のエディターはPDFを構成しますが、編集することは絶対に望ましくありません。



並行して、カバーの準備が行われます。 通常、著者はこれに参加しませんが、私の場合はそうではありませんでした。 私は標準的な「黒い背景のボール」画像を拒否し、才能ある芸術家セルゲイ・レショークに私の本のイラストを作成するように頼みました。 あなたは自分で結果を評価することができます、私は個人的に本当にそれが好きだった。



そして、ここが最後のフロンティアです-すべての章をまとめた完成したPDFを見て、追加または変更したいものをもう一度考えることができます。 今回は、編集ができなくなるため、考えてみてください。 この本は数週間で公開されます。 もちろん、あなたが私と同じように幸運でもなく、カトリーナが出版社の一部とともにマンハッタンを破壊しなければ。



著者の目を通して



これで、出版社の目を通して本を作成するプロセスがどのように見えるかがわかりました。 実際には、すべてがやや複雑です。 執筆を始めたとき、私は単純なロジックに従いました-私はブログを書くのが得意でした、そして技術的な本はそれとあまり違いはありません。 これは根本的に間違っています-「靴ひもを結ぶことができるので、帆を届けることができる」と考えるのと同じです。



章、セクション、および本全体の構造には、ブログ投稿よりもはるかに慎重な態度が必要です。 構造や目的、計画のない章は、著者の考えやアドバイスとして読まれます。 料理のショーのようなもので、プレゼンターが興味深い事実を語り、料理が魔法のように最終的な形になります。



私の良き友人が私に問題を指摘するまで、これらは私の最初の章でした。 「良い本では、各章の意味が最初のセクションにあり、各セクションの意味が最初の段落にあり、段落の意味が最初の文にあることに気付きました。」 この発言は、書く方法に関する私の見方を完全に変えました。



素材そのものだけでなく、素材の配送方法にも集中する必要があることに気付きました。 章は、完全な図を作成するためにレイアウトする必要がある段落、段落、リスト、および図のモザイクのように見えました。 理論と実践、明白で複雑なテキストと図のバランスを探していました。 私は簡単であると同時に基本的な本を書きたかったので、読者はいくつかのページをスキップしたい、時には考えるが、退屈な教科書のようではなく、むしろ少し考えたいと思いました。 それが判明したかどうか-読者を判断する。



完全主義は別の話です。 これは、あなたが何を書いても、コードでも、技術的なテキストでも、プログラマの災難です。 コードの場合、「週末にリファクタリングを行う」ことを常に安心させることができます。 本では、すべてがはるかに複雑です。 テキストが紙に当たると、すぐに引き返すことはできません。 あなたが愚かさを書いたなら-それは永遠に本に残るでしょう、コードが理想から程遠いままなら、これはそうあるべき姿です。 コメントでそれを笑い飛ばし、読者に感謝し、メモのテキストを修正することはありません。



完全主義は、テキストを何度も読み直し、段落を書き直し、イラストを描き直します。 このプロセスに終わりはありません。本をリリースする唯一の方法は、「理想的」を追求するためにそれを乗り越えずに、「十分な」ラインを自分で定義することです。



ツール



作業は、職場の設定から始まります。 Apressのメインツールは、特別な「ブック」テンプレートを備えたMicrosoft Wordです。 私は他のエディターを使用しなかったため、これが良い選択か悪い選択かを判断することを想定していません。 Wordには固有の問題があります。スタイルが頻繁に飛んだり、リストが正しくフォーマットされなかったり、エディターが何度もハングしたりして、数ページのテキストが失われました。 最後に、本のサイズがスタイリングの問題により、予定よりも60ページ大きいことが突然判明しました。



しかし、Wordには発行に関して多くの利点があります。 ハイライト-WYSIWYGのニュースと編集ツール。 これは、初めてキーボードに座ったときの忘れられない経験であり、新しいページは本物のページのように見え、文字は楽しく新鮮なまだにおいのある活版印刷のシートに表示されるようです。



5人のユーザーが同時にドキュメントを操作する場合、「ピアレビューモード」なしで作業することは不可能に思えます。 段落の横の余白で、多くの電子メールから収集することが不可能な小さなマルチカラーのダイアログが成長します。







2番目の重要なツールは、チャートエディターです。 私はあまり独創的ではなく、Visioで働いていました。 それはいつも私には面倒に思えたが、私はより良い代替を見つけることができませんでした。 最初は、図を避けてテキストに集中しました（おやつに写真を残して、すべてを一度に行うため）が、このアプローチは自分を正当化するものではありませんでした。



他のユーティリティは、独自の設定からのみ選択されます。ソースコード用のgitまたはsvn、便利な「スクリーンショット」、マインドマップ用のソフトウェアです。 主なことは、著者が快適に感じ、ツールを効果的に使用する方法を知っていることです。



ヒント



記事の最後の部分では、本を書くことを真剣に考えている人のためにいくつかのヒントを共有します。



あなたの動機を理解してください。 お金が必要な場合は、「WordPressでWebサイトを仕上げる」ことで、はるかに少ない労力でより多くの利益を得ることができます。 使用よりも販売からより多くの利益をもたらす知識は役に立たない。 あなたの動機が名声と認知である場合、過去1年間に出版された本の数を見てください。



「市場」を評価してください-同様の主題の他の本。 あなたの本が競合他社の本とどのように異なるかを決定します。



30〜40ページの1章（ただし、最初ではなく最後でもない）を書いて、本全体の所要時間を見積もってください。 たとえば、私の記録は朝から夜遅くまで1日15ページです。 ドラフトページまで約1時間。



テキストを読むことができるネイティブスピーカーを見つけます。 彼は主題分野を理解するべきではありませんが、正しく書く必要があります。



協力者のことを考えてください。 私は自分の本を書き始めたことを何度も後悔しました。



構造を追跡します。 本全体の計画を立ててから、各章の計画を立てて、最初のセクションで説明します。 目標を定義し、読者を目標に導きます。 セクションの素材がメイントピックから大きく逸脱していないことを確認し、特別なスタイルで「好奇心の強い情報」を強調します。 読者に、この章の終わりに「目標が達成された」と感じさせてください。



複雑なアイデアをシンプルに。 偉大な人の誰かが書いた：「読者への最高の敬意は、彼らを馬鹿と考えることです。」 どんなに失礼に聞こえるかもしれませんが、これらの言葉の背後には正しい考えがあります。 複雑なテキストを書くことは難しくありません-私たちの学校の教科書を見てください。 複雑な思考を理解、記憶、受け入れられるような方法で表現するのが本物の芸術です。 比phor、写真、図、コードを使用してください-言葉に限定されません。 「水を追加するだけ」のレシピは機能しないとすぐに言います。



「十分」の境界を自分で定義し、より重要なタスクがある場合はそれを越えないでください。 そうしないと、テクノロジが廃止されるまで章を書き換える危険があります。 国境は正式である必要はありません。 十分な内的感覚。



賢明な批評家を選び、できるだけ早く彼らの作品を読んでみましょう。 知的な批評家には、本をより良くするという1つの動機しかありません。 彼は「友情によって」著者をspareしんではならず、あなたの過ちを喜んで、あなたの費用で自分を主張しようとしてはいけません。



アドバイスに耳を傾けますが、アドバイスは書くよりもはるかに簡単であることを忘れないでください。 すべての議論の後、あなたがまだ別のことをする必要があるという感覚を持っている場合-あなた自身を信頼してお気軽に。



本が調和し、バランスが取れていることを確認するように努めてください。 これは、素材の複雑さ、セクション内のコードと写真の量、章のサイズなど、あらゆる面に当てはまります。 ここに属していないと思われる場合は、テキストの一部を削除することを恐れないでください。 トピック間の移行がシャープすぎる中間段落を追加し、イラストを変更し、大きな章を分割し、小さすぎる章を「接着」します。 読者を捕まえ、最後のページに移動させ、運んで実験を誘発させないようにしてください。そうすれば、あなたの努力は報われます。



おわりに



それで、まだ本を書くのはどうですか？ 難しいですか？ もちろん。 しかし、大きなプログラムのコードを書いたり、家を建てたりするのと同じくらい難しいことではありません。 これらのプロジェクトの複雑さは、プロジェクトとの関係によって異なります。 この恐怖をすぐに忘れるために、「自分で」または「とどまる」ためにビルドするか、お気に入りのペットプロジェクトのコードを書くか、顧客にそれを渡そうとします。 本は急いで書くこともできます-出版社は材料の品質をあまり要求しません。



しかし、あなた自身が喜んで迎えるに値する何かを書くためには、もっと多くの努力をする必要があります。
