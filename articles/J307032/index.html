WSUSを使用したソフトウェアのインストール-2

ヨセミテの最初の部分に続いて、LUP- WSUS Package Publisherの論理的な継続と開発を紹介したいと思います。 LUPとは異なり、WSUSPPはWindows Server 2012での動作、開発、およびサポートを提供します。このマニュアルでは、更新プログラムの追加、ルールの作成、最も一般的なサードパーティソフトウェアの説明のみを検討します。 すなわち：



1） Adobe Flash PlayerおよびAdobe Reader

2） 7-Zip

3） PDF24作成者

4） X-マウスボタンコントロール

5） Adobe Acrobat Reader DC（Adobe Reader XIからのアップグレード）



注意！ たくさんの写真、リンク、情報！



もちろん、機能するWSUSとWSUSPPが必要です。 WSUSPPの構成は簡単であり、考慮されません。

不要なツール：



- リモートMSIマネージャー -インストールされた製品の表示/削除、リモートコンピューターへのmsiパッケージのインストールのためのソフトウェア。 Produst Codesを検出する必要があります。

-SuperOrca -Orcaの開発、msiパッケージのプロパティの表示/変更。

-いくつかのテスト仮想マシン。その数はジョブのフリートに依存します。





Adobe Flash PlayerおよびAdobe Reader



これらの製品の例はWSUSPP Webサイトにあり、自分で調べることができます。 以下のすべてのリソースへのリンクを残します。

しかし、説明にないのはSCUPディレクトリのドキュメントです。 はい、WSUSPPはサードパーティのソフトウェアカタログをサポートしています。 そして、これらのアドビ製品には、事前定義されたテンプレートがあります。

2）ディレクトリとそのパラメーターを構成する



どこで：

以下を確認する： -カタログの更新を確認する間隔

変更 -選択したディレクトリのパラメーターを保存する

削除 -選択したディレクトリを削除します

テスト接続 - テスト接続

3）新しい更新を追加するには、カタログを選択し、[ このカタログから更新をインポート...]-> [カタログを開く ]をクリックし、関心のある更新をマークして、[ 選択した更新をインポート ]をクリックします。



ダウンロードして公開したら、 「閉じる」をクリックするだけです。 更新の準備ができました。 追加の変更は必要ありません。mspパッケージには必要なパラメーターがすべて含まれています。



同じ方法をAdobe Flash Playerに適用できます。 必要なすべてのルールは既にパッケージレベルにあり、アップデートはプレーヤーが以前にインストールされた場所にのみインストールされます。

7-zipアーカイバ



このソフトウェア用のディレクトリはありませんが、32ビットおよび64ビット用のmsiアセンブリがあります。 したがって、タスクは、どこに何を置くかを正しく決定する場合にのみ削減されます。 ほとんどの場合、更新プログラムがインストールされていることを示すインジケーターは製品コードです。 このコードは、WSUSPP（ 「ツール」->「Msiリーダー」 ）とSuperOrcaなどのサードパーティプログラムの両方で、msiで表示できます。 ただし、バージョン16.00、16.01、および9.20のOSアーキテクチャと製品コードは、インストールの必要性の指標となります。



更新を作成する
x86バージョンの製品コード：

9.20-{23170F69-40C1-2701-0920-000001000000}

16.00-{23170F69-40C1-2701-1600-000001000000}

16.01-{23170F69-40C1-2701-1601-000001000000}

16.02-{23170F69-40C1-2701-1602-000001000000}

Xマウスボタンコントロール



X-Mouse Button Controlという素晴らしい無料プログラムがあります。 マウスボタンにさまざまな機能と組み合わせを割り当てることができます。 5個以上のボタンを持つマウスを使用する場合は、強くお勧めします。 このソフトウェアの更新は、アクティブユーザーのTempディレクトリでインターセプトする必要があるexeファイルの形式で提供されます。 以下の最新のアップデートへのリンク。 また、更新のために、システムでプログラムプロセスが開始されていないことが必要です。 これはすべてサポートされており、msiへの変換は必要ありません。



更新を作成する
このような更新は、 「更新」->「カスタム更新の作成」で作成されます 。

必要なアクションはダブルクリックで追加されます。 シーケンスを間違えた場合は、アクションを右クリックして、ラインアップ/ダウンを移動できます。



1）最初に、完全インストールのすべての手順を説明します。

「自分の名前でプロセスを強制終了」-> XMouseButtonControl->「Ok」 （名前は拡張子なしで指定されます）

「ファイルの実行を許可する」-> XMBCUpdate_2.13.1.exe->「Ok」->「Ok」 （名前は任意ですが、その名前のファイルはポイント2で使用する必要があります）





2）次のウィンドウで、exeファイルを追加する必要があります。 残りのファイルは変更しないでください。

3） 「ベンダー名」 、 「製品名」 、 「タイトル」などのフィールドに入力します。 コマンドラインは変更/削除しません。

Adobe Reader XIをAdobe Acrobat Reader DCに更新する



おそらく多くの人がこの質問をしました。 そして、現在のバージョンにすぐに更新したいと思います。 複雑なことはなく、追加の手順がいくつか必要です。 また、ドキュメント： キーを持つブートストラップ 、 GUID 、 Adobe Properties



準備する
まず、exeを解凍する必要があります。 これは、パラメーター-sfx_o "unpack path" -sfx_neを使用してコマンドラインから実行します（ sfxの代わりにnosを使用できます）。

私の場合、次のようになります。

"C：\ WSUS \ CustomUpdates \ Adob​​e Acrobat Reader \ AcroRdr20151500630033_MUI.exe" -sfx_o "C：\ WSUS \ CustomUpdates \ Adob​​e AcrobatReader \ AcroRdr20151500630033_MUI" -sfx_ne



実際のパッチは、解凍されたコンテンツを含むフォルダーに配置する必要があります（執筆時点では、記事は15.006.30201です）。





最後に、2行追加してSetup.iniファイルを編集します。

[スタートアップ]

CmdLine = / sAll / rs / msi UPDATE_MODE = 0 EULA_ACCEPT = YES



[製品]

PATCH = AcroRdr2015Upd1500630201_MUI.msp







更新を作成する
0） 「更新」->「カスタム更新の作成」 。



1）プロセスを完全に自動化するには、AcroRd32.exeプロセスを完了する必要があります。

「自分の名前でプロセスを強制終了」-> AcroRd32->「OK」 （名前は拡張子なしで指定されます）

「ファイルの実行を許可」-> AcroRdr20151500630033_MUI \ Setup.exe->「Ok」->「Ok」 （手順2でフォルダーが指定されるため、ファイルへのパスはそれを考慮して示されます）。



追加のパラメーターは使用されません。 すべてがSetup.iniで既に指定されています



2）フォルダーを追加する

「フォルダを追加...」-> AcroRdr20151500630033_MUI->「OK」->「次へ」





3） 「ベンダー名」 、 「製品名」 、 「タイトル」などのフィールドに入力します。 コマンドラインは変更/削除しません。





4）製品コードの検証を通じてインストールされた更新を検討します。 新しいパッチをインストールしても変更されません。

「ルールタイプ：インストールされたMsi Produst」->「ルールの追加」-> AC76BA86-7AD7-FFFF-7B44-AE0F06755100->「OK」->「次へ」





5）インストールの必要性も同じ方法でチェックされますが、OSバージョン（DCバージョンはWindows 7以降をサポートします）によってもチェックされます。

「ルールタイプ：インストールされたMsi Produst」->「ルールの追加」-> AC76BA86-7AD7-1049-7B44-AB0000000001->「OK」

「ルールタイプ：Windowsバージョン」->「ルールを追加」->「比較：より大きいか等しい」、「簡単選択：Windows 7」->「OK」->「次へ」





6）経験のあるユーザーはメタデータを修正できます。 [ 発行 ]をクリックします。



PS： CmdLineの内容は、手順1でSetup.iniからパラメーター（オプション）に転送できます 。





結局のところ、WSUSPPはSCCMと競合できる非常に強力な製品です。 もちろん、彼には欠陥がないわけではありません。 これらには、Custom Update Creatorウィザードの最初のステップを修正できないことや、ルールを追加する独特のロジックが含まれます。 それでも、WSUS Package Publisherは、フリートで最新の重要性を理解している多くのシステム管理者にとって便利です。
