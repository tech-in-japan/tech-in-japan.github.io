答えを得るために正しく尋ねる方法についての考察

多くの場合、開発者がアナリストまたはクライアントから必要な情報を見つけようとする方法を観察する必要があります。 その結果、これは手紙や会議の無限のやり取りに変わり、結果として、開発者が必要な答えを受け取るという事実ではありません。



主な理由は、尋ねる人と答える人が異なる「世界の写真」を持っていることです。 同じものに対する異なる見解。 世界の絵は、出来事や現象を知覚し解釈する方法です[1]。 たとえば、開発者が質問をする場合、彼は解決している特定のタスクの観点から質問をします（彼にとって重要ですが、システム全体としては小さい）、質問を読んだ後、アーキテクトはシステム全体のアーキテクチャの観点から彼に答えることができます、および同じ質問のアナリスト-要件の観点から、およびプロジェクト領域の用語で。 つまり、さまざまな活動分野を持つ人々に同じ質問をすることで、さまざまな答えを得ることができます。 彼らは間違っていません、彼らは単に同じオブジェクト/状況（言い換えれば、「世界の絵」）の異なるビジョンを反映します。 原則として、すべての人々の世界観は異なっていると言わなければなりません。

簡単な人生の例：右の隣人に「良い開発者」が彼にとって何を意味するか尋ねると、おそらくあなたはあなたの考えと一致しない答えを得るでしょう。

したがって、質問をする前に、その意味内容（質問の対象）を処理する必要があります。

主題は、研究者によって決定され、その本質的な内容を反映するオブジェクトの概念的なビジョンです[2]。 したがって、質問の意味内容は、質問の主題の表現です。 物事の見通しに強く依存するのは、問題のセマンティックコンテンツです。 したがって、質問するときは、「どのような質問がそのような答えなのか」という単純なルールを覚えておく必要があります。



また、次の種類の質問が存在することを覚えておく価値があります。



1.非公開の質問-明確な回答ができる質問はい/いいえ。 例：.NETはプロジェクトで使用できますか？

2.未解決の質問-明確な回答がなく、対談者から主題に関する意見や意見を聞くことが重要な質問。 例：プロジェクトで.NETを使用することについてのあなたの意見は何ですか？

3.誘導質問-対話者に彼からの期待される答えを促すために策定された質問。 例：サーバーは現在MS Windows Server 2008を実行しています。Javaまたは.Netで機能を実装できます。 あなたの決断は何ですか？

4.セキュリティの質問-目的の理解に到達したことを確認することを目的とする質問。 たとえば、.NETを使用してプロジェクトを実装することを正しく理解しましたか？



正しい答えを得るのに役立つ質問を正しく定式化するために、いくつかのプラクティスを使用できます。



1.質問する前に、質問の主題を明確に定義する必要があります（その内容を定式化するため）。 言い換えれば、あなたが尋ねたいことと見返りに何を期待するか。 これにより、質問に回答できる人を事前に決定できます。

2.人々の世界の写真は異なっていて、同じ質問をさまざまな方法で理解できると仮定します。 したがって、彼に尋ねる前に、状況についてのあなたのビジョン（「世界の絵」）を少なくとも簡単に対談者に概説してみてください。 これにより、主題に関するビューを同期できます。 また、問題の主題を提示するときに使用する用語を明確にすることが必要な場合があることを忘れないでください。

3.「閉じた質問」の使いすぎを避けます。 多くの場合、この質問の対象となる人々

単一の回答はありません。また、決定を行うために追加情報が必要です。 このタイプの質問の乱用は、相手が尋問を受けているという印象をパートナーが持つ可能性があるため、コミュニケーション中に緊張した雰囲気につながります。 非公開の質問は、セキュリティの質問として要約するのに最適です。

4.問題を解決するためのオプションがいくつかあるが、そのうちの1つを選択するのが難しい場合は、まず問題を説明し、長所と短所を使用して可能な解決策を説明し、提案された解決策のどれを​​使用できるか質問します。 この質問の定式化により、決定を下すための情報を提供し、同時に可能な答えを制限します。 原則として、「どのように必要ですか？」という抽象的な質問に答えるよりも、既成のオプションの1つを選択する方が簡単です。

5.質問をする必要がある場合、その質問に応じて、主題に関するいくつかの詳細が通知されると予想される場合は、「一般から特定へ」の戦略を使用します。 最初に一般的な質問をしてから、詳細を明確にします。 逆ではなく、詳細な質問から始めると、主なもの-正確に何をする必要があるのか​​、正確に何を解決しようとしているのかを見逃す可能性があります。 この場合は、未解決の質問から始めてから、結果を明確化して要約した質問に要約することに徐々に進む方がよいでしょう。

6.対話者があなたを理解しなかった、または受け取った回答が完全に明確でないと感じた場合、質問の主題と対話者に関する見解を同期できるようにする明確な質問を使用します。 例：

-プロジェクトの一部として.NETを使用する予定です。 このオプションはあなたに合っていますか？

-ビジネス運営の観点から見ると、私たちにとって最も重要なことは、アプリケーションのパフォーマンスを確保することです。

-.NETを使用できるようにアプリケーションのパフォーマンスが確保されていれば、それは正しく理解できましたか
