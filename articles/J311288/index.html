プログラミング言語の歴史：SQL-生涯標準化





CodingDojoのアナリストによると 、SQLはどんなに奇妙に聞こえても、プログラミング言語の中で最も重要で必要なクエリ言語です。 CodingDojoの評価では、労働市場におけるプログラミング言語の需要に関する統計が考慮されます。



結局のところ、DBMS（MySQL、PostgreSQL、Microsoft SQL Server）はどこにでもあります。大小の企業、病院、銀行、大学などです。 基本的に、SQLはデスクトップデバイスのみに限定されません。SQLiteDBMSは、Apple Androidスマートフォンおよびモバイルデバイスで成功しました。 したがって、SkypeやDropboxなどのアプリケーションは常にアクセスします。



ただし、スマートフォンがない場合もありましたが、この言語はすでに存在していました。 SQLの歴史は数年ではなく、数十年です。 彼らはすぐに彼を信じませんでした。



System RおよびIBM



この言語の最初の言及は1974年に遡ります。 SQLはProject Rの実験的なリレーショナルDBMSシステムRの一部として作成されました。IBMはこのプロジェクトに従事していました。



当初、この言語はSEQUEL（Structured English Query Language）と呼ばれていましたが、このフレーズから「英語」という言葉が消え、略語は私たちが長い間慣れ親しんでいた形になりました。 一方では、SQLはリレーショナルデータベースへのクエリのユーザーフレンドリな表現に向けられていました。 一方、ほぼ最初から、それはいわゆる「フルデータベース言語」でした。 これは、SQLに次が含まれることを意味します。



•データベーススキーマを決定および操作する手段。

•整合性制約とトリガーを決定する手段。

•データベース表現を定義する手段。

•クエリの効率的な実行をサポートする物理層の構造を決定する手段。

•関係とそのフィールドへのアクセスを許可する手段。

•トランザクションセーブポイントを定義し、トランザクションをコミットおよびロールバックする手段。



確かに、並列トランザクションの側からデータベースオブジェクトへのアクセスを同期する手段を実装していませんでした。 実際、開発者は、必要な同期がDBMSによって暗黙的に実行されると最初に計算したということです。



この言語は、リレーショナルおよび非リレーショナルの両方の大部分のDBMSに実装されています。 開発の目的は、プログラミングスキルさえ持たないユーザーでも使用できる、単純な非手続き型言語を作成することでした。



クエリ言語は、Donald D. ChamberlinとRay Boyceによって開発されました。







SEQUELがこの目的の唯一の言語ではありませんでした。 カリフォルニア大学バークレー校では、非営利のDBMSであるIngresが開発されました。これはリレーショナルDBMSでしたが、独自のQUEL言語を使用していましたが、SQLと比較してそれをサポートするDBMSの数に匹敵することはできませんでした。
System Rは、最も複雑で完全なバージョンのSQLを実装しています。 同じIBMのSQL / DSとDB2の機能はわずかに少なかった。 SQLシステムRからは、十分に開発されていない部分（たとえば、セーブポイント）、または実装によって技術的問題が大きすぎる部分（たとえば、整合性制約やトリガー）のみが削除されました。



商業的成功



したがって、IBMが採用したSQLの商用実装への道は、トップダウン運動と呼ばれます。



Oracle、Informix、およびSybaseは異なる方法で行きました。ボトムアップ：市場に導入されたこれらのシステムの最初のバージョンは、SQLシステムRのかなり限られたサブセットを使用しました。 ただし、Oracle DBMSでのSQLの最初の商用実装では、サンプリング演算子でネストされたサブクエリの使用が許可されておらず、複数の関係の化合物でクエリを作成する可能性はありませんでした。





2011年時点の市場シェアの分布



リレーショナルデータベース管理システムへの早期移行に対する市場の関心の高まりにより、上記の企業の開発者は商業的な成功を収めることができました。 これは、DBMSが完璧からはほど遠いという事実に反して、むしろ起こりました。 さて、今ではOracle、Informix、Sybase、Microsoft SQL Serverは非常に強力なSQL方言をサポートしています。



標準化



SQLの多くの方言の出現とその成長は、互換性の問題やその他の矛盾につながると考えられていました。



しかし、SQL言語の標準化は非常に時間通りに始まりました-最初の商用実装の出現とほぼ同時に。 1982年、米国規格協会（ANSI）のデータベース委員会は、リレーショナルデータベースの標準言語の仕様の開発を任されました。







1986年に失敗した一連の規格のバージョンを拒否した後、専門家は共通点を見つけました。 そして1987年、SQL / 86標準は国際標準化機構（ISO）によって承認されました。



SQL System Rを標準の基礎とすることは不可能でしたまず、このバージョンの言語は技術的に十分に機能していませんでした。 第二に、実装するのが難しすぎるでしょう。 したがって、1980年代初期にIBMで開発されたSQL言語の方言が基礎として採用されました。 本質的に、この方言はSQL System Rのサブセットでした。



SQL1標準



1989年までに、SQL / 86標準はわずかに拡張され、その後、ANSI / ISO SQL / 89と呼ばれる次の標準が登場しました。



SQL / 89は、世界で最初に受け入れられたSQL言語標準です。 この言語には多くの欠点があります。多くの重要な概念が定義されておらず、多くは実装に翻弄されています。 この標準は、データベーススキーマや動的SQLの操作などの重要なセクションを完全に排除します。



それにもかかわらず、彼は真に標準化されたリレーショナルデータベース管理システムの開発において役割を果たしました。 さらに、SQL / 89標準の出現により、特定のDBMS製造業者とあまり関係のない情報システムの設計、開発、および保守が可能になりました。 ある意味で、SQL / 89の出現は、オープンシステムへのデータベーステクノロジーの進歩でした。



おそらく、SQL / 89標準の最も重要な成果は、構文の明確な標準化、データ検索とデータ操作演算子の明確な標準化、およびデータベースの整合性を制限する手段の固定です。



この規格では、2つのレベルの言語と、整合性を維持するための個別の手段が定義されています。 レベル2は、整合性ツールを含まない完全なSQLデータベース言語です。 レベル1は、レベル2の指定されたサブセットです。



整合性ツールには、以下を決定する機能が含まれています。



•テーブル間のリンクに必要な制限。

•テーブル行の制約を確認します。

•テーブルに行を入力するときのデフォルトの列値。



外部キーを決定する手段により、データベースのいわゆる参照整合性の要件を簡単に定式化できます。 リレーショナルデータベースで一般的なこの要件は、SQL System Rの整合性制約の一般的なメカニズムに基づいて定式化できますが、外部キーの概念に基づいた定式化はより簡単で理解しやすいものです。





さまざまな標準での結合操作の機能



SQL2標準とその追加



SQL標準の不完全さを認識して、さまざまな企業の専門家がSQL2と呼ばれる次の標準の作業を開始しました。 この作業は数年続き、多くの標準ドラフトが発行されました。最終的に1992年3月に標準の最終ドラフト（SQL / 92）が採用されました。 この標準は、SQL / 89標準よりもはるかに完全であり、アプリケーションの実装に必要なほぼすべての側面を網羅しています：データベーススキーマの操作、トランザクションの管理（保存ポイントが表示されます）およびセッション（セッションは、一時的な関係が格納されるトランザクションのシーケンスです）、データベース接続、動的SQL。 最後に、データベースディレクトリの関係が標準化されました。これは実際には言語に直接関係していませんが、実装に大きく影響します。



1995年に、標準は呼び出しレベルインターフェイス（SQL / CLI）仕様によって補足されました。 SQL / CLIは、動的に定義されたSQLステートメントを実行できるようにする呼び出しを持つプロシージャインターフェイスの仕様のセットです。 基本的に、SQL / CLIは動的SQLの代替です。



SQL / CLI標準は、ユビキタスODBC（Open Database Connectivity）およびJDBC（Java Database Connectivity）インターフェースの作成の基礎として機能しました。



1996年、別のコンポーネントがSQL / 92標準に追加されました-SQL / PSM（Persistent Stored Modules）。 この仕様の主な目的は、ストアドプロシージャを定義および使用する方法を標準化することです。つまり、データベースに格納され、アプリケーションによって呼び出され、DBMS内で実行できるSQLステートメントを含む特別に設計されたプログラムです。



Oracleは最も人気のあるDBMSの1つです。 さらに、SQL / 92標準との互換性が実装されたのは初めてでした。

そして当初、SQL言語をサポートする最初のDBMSは、VAXマシン用に開発されたOracle V2でした。 これは1979年のことです。
Oracleは、Windows、Linux、Max OS X、Sun Solarisなど、さまざまなプラットフォームをサポートしています。



Oracleが開発した手続き型拡張機能のSQLはPL / SQL（手続き型言語/構造化照会言語）と呼ばれ、AdaおよびPascal言語の構文に基づいています。 Oracle DBMSでSQLおよびPL / SQLとともに使用される3番目の主要言語はJavaです。



PL / SQLは、プログラムブロックのほか、数値、文字列、日付、フロー制御演算子（条件付きジャンプとループを含む）および3種類のコンテナー（コレクション）を格納するためのさまざまなデータ型をサポートします-可変長配列、連想配列、ネストされたテーブル。



SQL3標準



当初は、1995年に新しい標準の作業を終了する予定でした。 実際には、新しい標準の作業は1999年にのみ部分的に完了しました。このため、標準はSQL：1999と呼ばれていました。

SQL言語標準の各新しいバージョンは、以前のバージョンよりも大幅に大きくなりました。 したがって、標準のSQL / 89が約600ページを占めていた場合、SQL / 92のボリュームは300ページ以上増えました。
最初のSQL3プロジェクトは約1,500ページを占有しました。



しかし、SQL3の開発者は、このような標準のボリュームにより、その採用とその後のサポートの成功の可能性が大幅に減少するという結論に達しました。 したがって、彼らは標準を比較的独立した部分に分割することを決めました。



1999年に、SQL標準の5つの部分である1999が採用されました。



最初の部分（SQL /フレームワーク）は、標準の概念構造の説明に当てられます。 このパートでは、次の4つのパートの詳細な要約を提供し、標準に準拠していると主張する実装の要件を策定します。



SQLの2番目の部分：1999（SQL / Foundation）は、標準の基礎を形成します。 言語タイプのシステムが導入され、機能の依存関係と可能なキーを決定するための規則が定式化され、主要なSQLステートメントの構文とセマンティクスが決定されます。



•データベーススキーマを決定および操作するための演算子。

•データ操作演算子。

•トランザクション管理オペレーター。

•データベース接続管理オペレータなど







3番目の部分は、SQL / 92と比較して、指定されたSQL / CLI仕様で占められています。 4番目の部分では、SQL / PSM —ストアドプロシージャ定義言語の構文とセマンティクスを指定します。 最後に、5番目の部分（SQL /バインディング）は、プログラミング言語の標準バージョンのSQLバインディングルールを定義します。



SQL：1999には、さらにいくつかの部分が含まれることになっています。 その中で、次のツールの仕様：



•分散トランザクション管理（SQL /トランザクション）。

•テンポラルデータプロパティ（SQL /テンポラル）のサポート。

•外部データ管理（SQL / MED）。

•オブジェクト指向プログラミング言語（SQL / OLB）へのバインド。

•運用分析処理（SQL / OLAP）のサポート。



21世紀のSQL



2003年の終わりに、国際的なSQL：2003標準の新しいバージョンが採用され、公開されました。 多くの専門家は、SQL：1999に続く標準のバージョンでは、SQL：1999の不正確性のみが修正されると信じていました。 しかし実際には、SQL：2003では、いくつかの新しい重要なプロパティが指定されており、2008年の後半にわずかな変更が導入されています。



SQL：2003標準のパート2で指定されているSQL言語の最も重要な変更は、次の側面に関連しています。



•データ型。

•SQLから呼び出されるルーチン。

•CREATE TABLEステートメントの高度な機能。

•新しい回路オブジェクト-シーケンスジェネレータ。

•新しいタイプの列-識別列（ID列）および生成列（生成列）。

•新しい演算子MERGE。



標準の一般的な組織は、いくつかの変更を受けています。 SQL：2003標準は、次の部分で構成されています。



•9075-1、SQL /フレームワーク。

•9075-2、SQL / Foundation;

•9075-3、SQL / CLI。

•9075-4、SQL / PSM。

•9075-9、SQL / MED。

•9075-10、SQL / OLB;

•9075-11、SQL /スキーマ。

•9075-13、SQL / JRT;

•9075-14、SQL / XML。



パート1-4および9-10は、必要な変更を加えて、SQL：1999と同じままでした。 パート5（SQL /バインディング）は存在しなくなりました。 関連する仕様はパート2に含まれています。



情報スキームに特化したSQL：1999パート2のセクションは、別のパート11で強調表示されています。2つの新しいパート-13と14が登場しました。



パート13は、完全に「Javaプログラミング言語を使用したSQLルーチンと型」と呼ばれています。 標準のこの部分の外観は、SQL指向のDBMSの大手メーカーからJava言語への注目が高まっていることによって正当化されています。



最後に、SQLの最後の部分である2003は、SQL環境でXMLドキュメントを操作できるようにする言語ツールの仕様に特化しています。



開発者の努力にもかかわらず、標準化プロセスは明らかに変更に対応していません。





SQL履歴のハイライト



それでも、データベーススキーマの定義、データの選択と操作、データアクセスの許可、プログラミング言語へのSQLの埋め込みのサポート、動的SQLステートメントを含むSQLステートメントの基本セットは、商用実装で十分に確立されており、ほぼ標準に準拠していると言えます。



PS SQLは、従来のプログラミング言語に完全に帰属することはできません。プログラムの実行を制御する従来の演算子、型記述演算子などは含まれていません。データベースに保存されたデータにアクセスするための標準演算子のセットのみが含まれています。 しかし、これらの「欠点」は、ITの歴史における彼の位置を考えると、彼を許しています。
