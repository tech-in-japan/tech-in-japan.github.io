Node.js対Ruby on Rails



ほとんどの開発者とは異なり、私はNode.jsを甘やかし始め、その後、Ruby on Railsに没頭し、すばらしいRails for Zombiesコース、そしてMichael Hartl Ruby on Railsチュートリアルに没頭しました。

以下に書かれているのは、Node.jsとRuby on Railsの非科学的な比較です。



さあ始めましょう！



Node.js-フレームワークではありません！


まず、Node.jsは、非同期性と高速性を目的としてJavascriptで記述された、低レベルのルーティングおよび管理機能を備えたWebサーバーです。 この比較の最初から、Node.jsにはExpress.jsとMongooseの形式の「友人」が必要です（もちろん、データベースとしてMongoDBを選択していない場合）。 一方、RailsはRubyで書かれた最も基本的なフレームワークです。 技術的には、正直な比較のために、ここでは、RailsではなくNode.jsとThinまたはUnicornについて説明する必要がありますが、エンジンコンパートメントだけでなく、完全なバックエンドを開発したいと考えています。



ルビーは黄金の心を持つ不機嫌そうな老人です。


慣習を取り除いたので、言語について話しましょう。 Node.jsはすべてJavascriptで記述されていますが、RailsはRubyで記述されています。 1番目と2番目の両方に長所と短所があります。 一方、Javascriptは構文の学習を容易にし、 多くの人はおそらくフロントエンドで（彼自身のRailsに対して） 



:VARIABLE => “--” def func arg = "   " VARIABLE do |iterator| Module::Class    class Subclass < Class @instance ,   Perl


その他のフリル）。



ただし、Rubyはより簡潔にすることができ、すべての魔法が機能するために括弧のフォレスト（coffescriptリーダー、緊張しない）を必要としません。 さらに、Ruby（およびRails）には、日付から文字列や配列に至るまですべてを囲む、非常に便利な組み込みメソッドが豊富にあります。 



 10.minutes.ago “string”.sqeeze ['array','elements'].include?('array') #true


はい、これはすべてJavascriptで実現できますが、知識とカスタムメソッドが必要です...または_undercoreのようなものが必要です。



100％Javascriptは素晴らしいですが、痛いほど過大評価されています。


言語を理解したら、Webアプリケーションの開発に注意を払うことができます。 Node.jsは非同期のフロントおよびバックJavascript環境に惹かれますが、RailsではSQL、Ruby、およびJavascriptを処理する必要があります。 確かに利点のように思えますが、両方をある程度検討した後、「 過大評価された利点 」のカテゴリにおける言語の統一性を書き留めます。 ほんのいくつかの例とブログ記事を読むだけで、Rubyに慣れることはそれほど難しくありません。 さらに、Rubyを使用すると、ノンブロッキングアーキテクチャのためにJavascriptに付随する「コールバックヌードル」をほぼ完全に忘れることができます（同時並行性が非常に優れているため、間違いなくプラスとして認識しています）。 それでも、私はこれを言いたくはありませんが、名前空間、クラス、およびモジュール用に事前定義された構文があると便利な場合があります。



Railsは必要なものをすべて提供します。


ここで、言語とそれに付随するものは無視します。

Node.js + Express.jsは、コントローラー、ビュー、ヘルパーとともにルーティングを提供します。 ただし、このツールボックスはRailsに比べてまだ色あせています。

類推のために、Node.js + Express.jsの背景のRailsは、アメリカンアパレル（服とホームアクセサリーを販売する2つの大企業-約翻訳者）の背景のMujiに似ています。 どちらも家庭用の服とアクセサリーを提供しますが、アメリカンアパレルの服は特に信頼できるようには見えず、常にニーズを満たしているとは限らず、パンツでかっこよく見えるようにしてください。 一方、無印良品は、世界中のすべてを同じスタイルで、一見すると同じ生地で執obsに装飾しています。



しかし、プログラミングに戻ります。

Railsには、完全なMVCと重要な内反足によってのみ台無しになる便利な機能のセットが組み込まれたエコシステムが組み込まれています。

url_forおよびimage_tagの関数の例をいくつか示します。 url_forは、指定されたデータオブジェクト（ @productsなど）のURLを常に返す組み込み関数であり、 image_tagは常にアセットパイプラインから画像の正しいURLを取得します。

もちろん、これはすべてNode.jsでも実行できますが、これにはモジュールとヘルパーを使用したかなりの作業が必要になります。 さらに、Railsでは、データモデルについてまったく心配する必要はありません。 はい、Node.jsの場合、Mongooseが役立ちますが、それでもデータを完全に制御できると感じることは困難です 



 Product.find(params[:id])


Railsのアプリケーション、モジュール、コントローラー、ビューのどこからでも利用できます。



ただし、これらのチップにはすべてマイナス面があることに注意してください。 私はこれを行うための正しい方法を単に「推測」し、ほとんどの場合最良の方法を逃していることをよく理解しています。 また、何かが故障した場合、特殊な変数名が難しいために故障の原因を特定するのは非常に困難です（たとえば、アクション名+コントローラ名+ "_path"を結び付けると、現在のパスを目的のアクションに戻す魔法のようにnew_product_path ヘルパーを取得できます） 。

Node.jsでは、自分が書いたものだけが表示されます-歪みは最小限です。



多くの人がRailsのモジュールを作成および改善しています。


次：エコシステム。 Node.jsがNPMで素晴らしいスタートを切ったという事実にもかかわらず、さまざまなニーズに対応する最も人気のあるパッケージのほとんどにほこりは収まっておらず、競合他社の数はすでに記録を破っています。 最終的には、壊れた花瓶を接着剤で接着しようとすることがよくあります。

Rails側では、ライブラリとツールの最も強力な無料リポジトリの1つです。 ツールの相対的な人気を示し、機能別に分類したRubyツールボックスも大好きです。 さらに、すでに述べたように、RailsでNodeに手動で入力する必要のある多くのモジュールは、そのまま使用できます。 データベース接続、プロファイリング、キャッシュ、移行、データモデル、パイプライン、ルーティング、コンソールコマンド、テストなどのようなもの...ファビコンさえもすでに配置されています！ もちろん、これはすべて簡単なことですが、すべてが必要であり、正午まで何らかの理由で互いに通信することを拒否するNode.jsモジュールの競合に関するこれらのメッセージをすべて表示しないのはクールです。

とりわけ、Node.jsにあるものよりもはるかに多い、Railsの膨大な数のチュートリアルと例です。 300以上の動画を含むRailscastのみ 。 しかし、Railsは急速に変化しており、多くの教科書はすでに関連性を失っており、他のタスクはクロールでさまざまな記事を公開した後に解決策を得たということを覚えておく価値があります。 その結果、Google < 1 year old



セレクターを使用してRailsに関する質問への回答を見つけ、 Rails Guidesでこの検索を開始することが賢明になりました。



Railsにはジェットパックが含まれています。 デフォルトで。


最後に、同じフレームワーク、データベースがあり、モジュールが構成され、すべてが正常であると仮定します。 Node.jsは、すべてを記述してすべてをまとめるだけの時間を必要とします（これも、サードパーティのプラグインの不整合と厳格な構造の不足によるものです）。 Railsで、完全なRESTfulインターフェースを構築するには、たった1つのコマンド 



 rails generate scaffold Post name:string title:string content:text


必要なコントローラー、ビュー、モデルはそれぞれの場所で待機します。



多くの場合、これには1時間と1時間（データベースエンティティの数の倍数）がかかります。 これはそれほど難しい仕事ではなく、タイプミスに対して脆弱であり、覚えておく必要がある情報が大量にあります。 私はいつもRailsの足場を使用しているわけではありませんが、単純なプロトタイプの場合、より高速な代替案を見つけるのは困難です。



結論として...


Node.jsは、優れたツールキットであり、トレンディで、急成長し、高性能（議論の余地があります）であり、習得が容易で、他のほとんどのWeb開発者ツールよりも優れています。 しかし、彼はいまだにRailsの人のオリンポスの神に対する才能のあるシニアスターのままです。 Node.jsに違反はありませんが、誰を選択しますか？ Rubyの学習に伴う不便さを克服できれば、その選択は明らかです。 覚えておいてください：最良のツールは、仕事をより良く、より速く行えるツールです。
