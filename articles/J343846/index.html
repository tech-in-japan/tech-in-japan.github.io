Telegram、Slack、Facebook向けの.Net Coreのボット

最近、銀行のプロジェクトの1つで、ボットを作成するというタスクが発生しました。 この経験の結果はこのテキストであり、これは主に初心者の開発者を対象としています。







最新のメッセージングツールを使用するほとんどすべての人がボットに出会いました。 ボットの定義の1つは、特定のスケジュールに従って自動的に、および/または特定のスケジュールに従って、ユーザー向けのインターフェースを介してアクションを実行するプログラムです。



ボットは、通知、プロセスの自動化（たとえば、ユーザーの手紙に基づいたTFSでのタスクの自動生成）、およびその他の多くの目的に使用できますが、 1つの記事ではすべてのオプションを検討するのに十分ではないため、コマンドを処理するためのボットの作成方法について説明します。



電報用ボット



最も軽い開発プラットフォームはTelegramです。 プロセスは、特別なBotFatherボットを呼び出すことから始まります。







すべてが非常に簡単です。BotFatherでドライブする検索バーで、スタートボタンを押し、\ newbotコマンドを選択し、質問に連続して答えて、ボットの名前とそのユーザー名を示します。 最終的に、Telegramはボットが正常に作成されたことを報告し、キーを提供し、追加の説明を示したり、チームの名前を入力したりします。



技術的には、メッセージとボットコマンドを処理する2つの方法があります。 1つ目は、サービスが定期的にTelegramサーバーをポーリングして変更がないかどうかです。 デモンストレーションには、Telegram.Bot APIを使用します（パッケージはnugetから入手できます）。 



using Telegram.Bot; using Telegram.Bot.Args; using Telegram.Bot.Types; using Telegram.Bot.Types.Enums; class Program { private static TelegramBotClient client; static void Main(string[] args) { // token,   BotFather client = new TelegramBotClient(token); client.OnMessage += BotOnMessageReceived; client.OnMessageEdited += BotOnMessageReceived; client.StartReceiving(); Console.ReadLine(); client.StopReceiving(); } private async void BotOnMessageReceived(object sender, MessageEventArgs messageEventArgs) { var message = messageEventArgs.Message; if (message?.Type == MessageType.TextMessage) { await client.SendTextMessageAsync(message.Chat.Id, message.Text); } } }


上記の例では、各着信メッセージはユーザーに再送信されます。 Telegramでは、チームもメッセージであり、「/」記号のみが付いています。



2番目の方法では、webhookを設定する必要があります。 変更を処理するhttpsアドレスでホストされるサービス。 この方法では、Telegramサーバーの負荷は軽減されますが、証明書が必要です。 ただし、証明書の問題は、Ngrokサービス（https://ngrok.com/）のおかげで簡単に解決できます。Ngrokサービスは、httpsアドレスからマシンのアプリケーションアドレスにリクエストをトンネルできます。



Ngrokを使用するには、以下を行う必要があります。



  • サイトに登録して個人キーを取得します。
  • ngrok.exeをインストールし、コマンドプロンプトでngrok authtoken Ngrokキーを入力します。
  • ngrok http port serviceコマンドを実行します。
  • Visual Studioでは、Ngrokプラグインをインストールし、メニューの[ツール]-> [Ngrokトンネルの開始]のコマンドを使用できます。


開始後、生成されたアドレスを含むウィンドウが表示されます。 また、すべてのトンネリングイベントに関する情報がここに表示されます。







Ngrokによって返されるHTTPSアドレスは、トンネリングが開始されるたびに変更されますが、アプリケーションをデバッグするときに常にこれを行う必要はありません。 特定のポートにトンネリングを1回設定し、必要な回数だけサービスを停止（開始）します。



Ngrokには管理ページがあり、 127.0.0.1 ：4040で利用できます。このページでは、ユーザーリクエストのパラメーターを確認できます。







メッセージを再度送信して前の例に戻り、次のWebAPIコントローラーを使用してASP.NET Coreプロジェクトを作成します。 



 [Route("bot")] public class BotController : Controller { // token,   BotFather private readonly TelegramBotClient client = new TelegramBotClient(token); [HttpPost] public async void Post([FromBody]Update update) { if (update == null) return; var message = update.Message; if (message?.Type == MessageType.TextMessage) { await client.SendTextMessageAsync(message.Chat.Id, message.Text); } } }


Telegramにサービスの場所を伝えるために、次の簡単なコンソールアプリケーションを作成できます。 



 using Telegram.Bot; using Telegram.Bot.Args; using Telegram.Bot.Types; using Telegram.Bot.Types.Enums; class Program { static void Main(string[] args) { // token,   BotFather var client = new TelegramBotClient(token); client.SetWebhookAsync("https   Ngrok ").Wait(); Console.ReadLine(); client.SetWebhookAsync().Wait(); } }


Slackのボット



別の一般的なメッセージングサービスを検討してください。 Slackでのボットの作成は、単純なフォーム（ https://api.slack.com/apps/new ）から始まります。このフォームでは、アプリケーションの名前を入力し、ワークスペースを指定する必要があります。



[アプリの作成]ボタンをクリックすると、セットアップページが表示されます。[機能の追加]サブセクションで、[スラッシュコマンド]項目を選択してボットコマンドの作成に進む必要があります。 表示されるフォームで、次を入力する必要があります。



  • [コマンド]フィールド-コマンドの名前
  • [リクエストURL]フィールド-サービスのhttpsアドレス
  • 簡単な説明フィールド-コマンドの目的






図は、チャットがAlfaTestNdoボットのgetmoneyコマンドを呼び出す方法の例を示しています。







指定されたhttpsアドレスでSlackコマンドを選択すると、次のメッセージを送信します 



 public class Message { public string channel_id { get; set; } public string channel_name { get; set; } public string command { get; set; } public string response_url { get; set; } public string team_domain { get; set; } public string team_id { get; set; } public string text { get; set; } public string token { get; set; } public string trigger_id { get; set; } public string user_id { get; set; } public string user_name { get; set; } }


ここで、サービスのコードがどのように見えるかを見てみましょう。応答として「hello」メッセージを送信します。 



 [Route("bot")] public class BotController : Controller { [HttpPost] public async void Post(Message message) { var uri = new Uri("https://slack.com/api/chat.postMessage?token=" + token + "&channel=“ + message.channel_id + "&text=hello"); var httpClient = new HttpClient(); await httpClient.GetAsync(uri).ConfigureAwait(false); } }


トークンキーとチャネル識別子message.channel_idを含むchat.postMessageメソッドがサービスで呼び出されます。



Slackは次のキータイプを定義します。



  • ユーザートークン-アプリケーションをインストールしたユーザーのキー。Oauthによる承認が適用されます。 プレフィックスはxoxpです。 詳細はこちらhttps://api.slack.com/docs/oauth
  • ボットユーザートークン-キー、技術仮想ユーザー。 xoxbプレフィックスがあります。 それらの詳細については、 https://api.slack.com/bot-usersをご覧ください 。
  • ワークスペーストークン-ワークスペースキー。 接頭辞xoxaを持っています。
  • レガシートークン-許可されたユーザーキー。 プレフィックスはxoxpです。 それらの詳細については、 https://api.slack.com/custom-integrations/legacy-tokensをご覧ください 。 それらを使用することは推奨されません。
  • 検証トークン-このキースラックは、サービスにアクセスするときに送信されます。 これを使用して、サービスにアクセスするのがアプリケーションであることを確認できます。 APIメソッドの呼び出しには使用されません。


Slack APIメソッドはすべて十分に文書化されており、サポートされるキーに関する情報が含まれています。 メソッドの説明ページには、テスト用の[テスター]タブもあります。



https://api.slack.com/methods/chat.postMessage







Slackトピックの最後に、Telegramと同様に、テキストメッセージだけでなく、ボタン、メニューなどのさまざまなインタラクティブコンポーネントも作成できることを追加します。



Facebook用ボット



私の意見では、Facebookでボットを作成することはTelegramやSlackよりも複雑ですが、一般的にはほとんどすべてが同じです。 developers.facebook.com/appsページにアクセスし、「Add a new application」ボタンをクリックして、フィードバック用の名前とメールアドレスを入力する必要があります。







[アプリケーションIDの作成]ボタンをクリックすると、セットアップページが開きます。ここで、APIメソッドへのアクセスキーを取得するために、ボットの個人ページを指定する必要があります（作成する必要があります）。







[Webhook]セクションで、サービスのhttpsアドレスを指定し、確認キーを見つけて[トークンの確認]フィールドに入力し、購読する情報を選択する必要があります。







「確認して保存」ボタンをクリックすると、Facebookは、hub.mode、hub.challengeおよびhub.verify_tokenの各フィールドを含む確認メッセージを送信します。 キーが実際にhub.verify_tokenフィールドに転送された場合、サービスはこのメッセージを受け入れ、hub.challengeを成功として返すことができるはずです。 



  [Route("bot")] public class BotController : Controller { [HttpGet] public string Verify() { var mode = Request.Query["hub.mode"].FirstOrDefault(); var challenge = Request.Query["hub.challenge"].FirstOrDefault(); var token = Request.Query["hub.verify_token"].FirstOrDefault(); return challenge ?? string.Empty; } }


メッセージのみをサブスクライブし、メッセージ「hello」をボットに送信するとします。 この場合、Facebookは指定したhttpsアドレスに次のメッセージを送信します。







着信メッセージを処理するためのコードはどのように見えますか？ この例では、メッセージはユーザーに再送信されます。

ボットフレームワーク



マイクロソフトのボットフレームワークなどのすばらしいツールに触れない場合、レビューは不完全になります。



まず、Bot Frameworkは、Bing、Cortana、Email、Facebook、GroupMe、Kik、Skype、Skype for Business、Slack、SMS、Microsoft Teamsなどのさまざまなチャネルで作業するための統合APIを提供します。



次に、エミュレータ（https://github.com/Microsoft/BotFramework-Emulator）を取得して、httpsアドレスを必要としないサービスをテストします。 エミュレーターでは、 dev.botframework.com/botsに既に配置されているサービスを確認することもできますが、この場合、アプリケーションIDとパスワードを指定する必要があります。







確かに、多くの不快な瞬間があります。 特に、.Net Coreはサポートされていません。 これについては、ここで詳しく説明します。

github.com/Microsoft/BotBuilder/issues/572

メッセージを再送信して、例に戻りましょう。 Bot Frameworkテンプレートをインストールする必要があります。







この例では、ユーザーからのテキストメッセージ（ActivityTypes.Message型）に応答して、IDialogインターフェイスをサポートするクラスのインスタンスが作成されます。 



  [Route("api/[controller]")] [BotAuthentication] public class MessagesController : Controller { [HttpPost] public async Task<HttpResponseMessage> Post([FromBody]Activity activity) { if (activity?.Type == ActivityTypes.Message) { await Conversation.SendAsync(activity, () => new Dialogs.RootDialog()); } return new HttpResponseMessage(HttpStatusCode.OK); } }


私たちの場合、これはStartAsyncメソッドを持つRootDialogクラスであり、着信メッセージを送信するタスクを編成します。 



 [Serializable] public class RootDialog : IDialog<object> { public Task StartAsync(IDialogContext context) { context.Wait(MessageReceivedAsync); return Task.CompletedTask; } private async Task MessageReceivedAsync(IDialogContext context, IAwaitable<object> result) { var activity = await result as Activity; await context.PostAsync(activity.Text); context.Wait(MessageReceivedAsync); } }


私の意見では、Bot Frameworkの最も興味深い機能はConnectorです。 これは、異なるチャネル間の通信を整理できるAPIです。 詳細については、 docs.microsoft.com / en-us / bot-framework / dotnet / bot-builder-dotnet-connectorをご覧ください。



おわりに



ボットとWebアプリケーションの開発プロセスを比較すると、次の点に注意することができます。どちらの場合も、リクエスト処理の中間層が同様に必要ですが、フロントパートについては、違いが大きいです。 ボットを作成する場合、さまざまなブラウザーやオペレーティングシステムで動作するインターフェイスを開発するためにリソースを探す必要はありません。 インターフェイスはメッセンジャーサービス開発者の責任です。 APIを制御することはできませんが、多くの場合、サイトではなくボットを介したユーザーインタラクションの実装がより有望であり、多くのリソースを節約できます。
