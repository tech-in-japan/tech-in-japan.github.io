デバッグ：デバッグ情報サーバーの展開

作業中のコンピューター上のドキュメントのデポジットを掘り下げて、2、3年前に書いたデバッグ情報サーバーを展開する手順を見つけました。 私はそれをhabro-communityに紹介しようとします。 この命令は、クラッシュダンプ分析（事後デバッグ）だけでなく、製品のリリースバージョンのデバッグを（リモートおよび直接、コンピューターおよびテスターのコンピューターで）使用したいWindowsのC ++開発者に役立ちます。





デバッグ情報ストレージの展開





1.環境の準備


デバッグシンボルのリポジトリを展開するには、次のものが必要です。

  1. インターネットにアクセスできるコンピューター（直接またはプロキシ経由）、Windows 2003 ServerまたはWindows XP Proオペレーティングシステム、インターネットインフォメーションサービスWebサーバー
  2. Windowsディストリビューションのデバッグツール。 配布は次のサイトから取得できます： www.microsoft.com/whdc/DevTools/Debugging/default.mspx 、バージョンx86またはx64
  3. オペレーティングシステムの文字の記号。 配布は次のサイトから取得できます： www.microsoft.com/whdc/devtools/debugging/symbolpkg.mspx
  4. プロキシ配布（インターネットへの直接アクセスの場合）


ディストリビューションキットDebugging Tools for Windows（「C：\ Program Files \ Debugging Tools For Windows」フォルダにインストールされていると仮定します）をインストールし、オペレーティングシステムのデバッグシンボルのすべてのディストリビューションをフォルダにインストールする必要がありますフォルダ「C：\ TempSymbols」）。



2.デバッグ情報のリポジトリの構成


Microsoftデバッグサーバーsymsrv.dllを使用して、シンボルストレージを整理します。 これを行うには、C：\ Symbolsなどのシンボルストア用のフォルダーを作成します。 次に、すべてのコンピューターユーザーに対して読み取り権限を設定する必要があります。

次のステップは、デバッグシンボルファイルをリポジトリに追加することです。 これを行うには、Debugging Tools For Windowsスイートの「symstore.exe」プログラムを使用します。 次のコマンドを実行します。

symstore add /r /3 /fc:\TempSymbols\*.* /sc:\Symbols /compress





このコマンドは、「c：\ TempSymbols」ディレクトリのすべてのサブフォルダーを検索し、そこからデバッグ情報を含むバイナリファイルとファイルを「C：\ Symbols」フォルダーにある3レベルのシンボルストレージにコピーします。 それらをアーカイブし、インデックスファイルを作成してアクセスを高速化し、ストレージを変更するためのアクセストランザクションに関する情報を保存します。

このコマンドのキーの説明：

  • add-ストレージにファイルを追加します。
  • / r-シンボルファイルを含むフォルダーを再帰的にバイパスします。
  • / 3-3層ストレージを編成します（ファイルへのアクセスを高速化するため）
  • / f [path] -リポジトリに追加されたファイルへのパス。
  • / s [path] -リポジトリへのパス。
  • / compress-アーカイブストレージを作成します（ディスクスペースを節約するため）


コマンドは十分に長く実行され、その実行結果は、フォルダー「C：\ Symbols」にデバッグ情報のリポジトリーを作成します。 この手順の後、フォルダー「C：\ TempSymbols」を削除できます。



3. httpプロトコルを介したデバッグ情報のリポジトリへのアクセスの組織


この時点で、ストレージアクセスを提供するようにインターネットインフォメーションサービスを構成する必要があります。 まず、仮想フォルダーを作成します。

  1. 「スタート」->「管理ツール」->「インターネットインフォメーションサービス（IIS）マネージャー」を開きます。
  2. 「Webサイト」フォルダを展開します。
  3. 「Default Web Site」要素のコンテキストメニューで、「New-> Virtual Directory ...」を選択します。
  4. ようこそウィンドウで、「次へ」をクリックします。
  5. 「エイリアス」フィールドに「シンボル」と入力し、「次へ」ボタンをクリックします。
  6. 「パス」フィールドに「C：\ Symbols」と入力し、「次へ」ボタンをクリックします。
  7. 「スクリプトの実行」および「参照」フィールドのチェックを外します。
  8. [次へ]および[完了]ボタンをクリックします。


次に、MIMEタイプを構成します。

  1. 「シンボル」仮想フォルダのコンテキストメニューで、「プロパティ」パントを選択します。
  2. 「HTTPヘッダー」タブを選択します。
  3. 「MIMEタイプ」ボタンをクリックします。
  4. 「次へ」ボタンをクリックします。
  5. [拡張子]フィールドに「*」と入力します。
  6. 「MIMEタイプ」フィールドに「application / octet-stream」と入力します。
  7. [OK]ボタンをクリックして、[MIMEタイプ]ウィンドウを終了します。
  8. [OK]ボタンをクリックして、[プロパティ]ウィンドウを終了します。


4.インターネット経由で文字を更新するためのプロキシフィルターをインストールする


プロキシフィルターは、デバッガーがMicrosoftパブリックストアから取得しようとしたデバッグシンボルファイルを受け取らず、それによってシンボルストアを更新するために必要です。

MicrosoftのDebugging Tools For Windowsスイートで提供されるプロキシフィルターはISAPI拡張機能であり、「C：\ Program Files \ Debugging Tools For Windows \ symproxy」ディレクトリにあります。

プロキシフィルターをインストールします。

  1. ファイル「symproxy.dll」および「symsrv.dll」を「C：\ Program Files \ Debugging Tools For Windows \ symproxy」ディレクトリから「％WINDIR％\ system32 \ inetsrv」フォルダにコピーします。
  2. 空のファイル「％WINDIR％\ system32 \ inetsrv \ symget.yes」を作成します。
  3. ファイル「C：\ Program Files \ Debugging Tools For Windows \ symproxy \ symproxy.reg」からデータをレジストリに追加します。
  4. レジストリエディターを開いて、キー「HKLM /ソフトウェア/ Microsoft / Symbol Server Proxy / Webディレクトリ/シンボル」を見つけます。
  5. 「SymbolPath」の値を「C：\ symbols; http：//msdl.microsoft.com/download/symbols」に変更します。
  6. 次に、レジストリキー「HKLM \ SYSTEM \ CurrentControlSet \ Services \ EventLog \ Application \ SymProxy」を作成し、「EventMessageFilter」という名前と値「％WINDIR％\ system32 \ inetsrv \ symproxy.dll」を含むREG_EXPAND_SZ値を追加して、プロキシフィルターをイベントソースとして登録します」


プロキシフィルターで動作するようにIISを構成します。

  1. 「スタート」->「管理ツール」->「インターネットインフォメーションサービス（IIS）マネージャー」を開きます。
  2. 「アプリケーションプール」項目のコンテキストメニューで、「新規」->「アプリケーションプール」を選択します。
  3. [アプリケーションID]フィールドに「SymSrvプール」と入力します。 OKをクリックします。
  4. 「SymSrvプール」要素のコンテキストメニューで、「プロパティ」を選択します。
  5. [ID]タブを選択し、[事前定義]ラジオボタンを選択し、コンボボックスで[ネットワークサービス]を選択して[OK]を押します。
  6. 「Webサイト->デフォルトWebサイト」ブランチを展開します。
  7. 「シンボル」要素のコンテキストメニューで、「プロパティ」を選択します。
  8. [仮想ディレクトリ]タブで、[作成]ボタンをクリックします。
  9. [アプリケーションプール]コンボボックスで[SymSrvプール]を選択し、[OK]をクリックします。
  10. [既定のWebサイト]コンテキストメニューで、[プロパティ]を選択します。
  11. [ISAPIフィルタ]タブを選択し、[追加]ボタンをクリックします。
  12. 「フィルター名」フィールドに「SymProxy」、「実行可能」フィールドに「％WINDIR％\ system32 \ inetsrv \ symproxy.dll」と入力します。
  13. [OK]をクリックして、[フィルターのプロパティ]ウィンドウを閉じます。
  14. [OK]をクリックして、[既定のWebサイトのプロパティ]ウィンドウを閉じます。


「iisreset.exe」ファイルを実行して、新しい設定でIISを再起動します。



5. symproxy.dllのプロキシサーバー設定の構成


Windows用デバッグツールには、「symproxy.dll」がインターネット（プロキシサーバーなし）で直接動作する場合、「symproxy.dll」が圧縮デバッグシンボルファイルを受信する呼び出しをMicrosoft Webサイトにリダイレクトしないという欠点があります。 この欠陥を解消するには、ローカルプロキシサーバーをインストールし、「proxycfg.exe」ユーティリティを使用してシステムがプロキシサーバーと連携するように構成する必要があります。



6.デバッグ情報サーバーと連携するようにクライアントコンピューターを構成する


各クライアントコンピューター（開発者コンピューター）で、「C：\ LocalSymbols」など、文字をキャッシュするためのフォルダーを作成します。 Windowsディストリビューションのデバッグツールをインストールし、環境変数を作成します。

_NT_SYMBOL_PATH=srv*[local_repository]*http://[symbol_server_ip]/symbols

_NT_IMAGE_PATH=srv*[local_repository]*http://[symbol_server_ip]/symbols







どこで：

  • [local_repository]はローカル文字キャッシュです（例：「C：\ Symbols」）。
  • [symbol_server_ip] -企業デバッグ情報サーバーのIPアドレスまたはドメイン名。


最初の変数はデバッグシンボルの検索パスを担当し、2番目はこれらのシンボルに一致するバイナリを検索します（クラッシュダンプの解析時、つまりクラッシュしたプログラムのバイナリに直接アクセスできない場合に必要）。



まとめ


それで、結果を説明します。 これで、オペレーティングシステムのデバッグシンボルと一部のMicrosoftライブラリと製品が配置されたサーバーができました。 同時に、ソフトウェアの新しいバージョンのキャラクターを受信するためにサーバーにアピールが受信されると、このリクエストはマイクロソフトのWebサイトにリダイレクトされ、キャラクターはそこからダウンロードされ、サーバーに落ちます。 開発者のコ​​ンピューターには、デバッガー（Visual Studio、WinDbgなど）の要求に応じてキャラクターサーバーからダウンロードされるキャラクターのローカルリポジトリがあります。 キャラクターサーバーの作業を完了するには、ビルドシステムに追加するだけです。

  • デバッグ情報ファイル（* .pdb）の作成のセットアップ
  • symstoreを呼び出して、コンポーネントおよびコンポーネント自体に関するデバッグ情報をサーバーに取得します。


おわりに


私が言ったように、この命令は2〜3年前に書かれたので、そこにWin2003を搭載したコンピューターがあります。Win2008とIISの最新バージョンに類推してシンボルサーバーを展開することは難しくないと思います。 また、設定のスクリーンショットを撮影できる仮想マシンはありませんでした。 しかし、説明は非常に詳細なので、理解できると思います。

おそらく、パラグラフ5で説明した問題はもはや関係がないので、チェックしませんでした。

デバッグ情報サーバーの操作に関する詳細情報は、ヘルプファイルにあります。デバッグツールWindowsの場合、シードとしてデバッグ情報サーバーをソースストレージサーバーにバインドできます。クラッシュダンプを解析すると、プログラムクラッシュスタックだけでなく、そして、アセンブリ時に有効なソース内の場所。
