ダミーのSVNマージ

簡単な言葉と多数の写真で、Eclipseの例はsvn mergeの作成方法を示しています。 この記事は、ブランチのマージの実装がまだ日常業務の一部になっていない人にとって有用です。



この記事は、次のソフトウェアを使用するように設計されています。

  • SVN 1.4.4
  • Eclipse 3.4.2 SR2
  • 破壊的0.7.7
  • ネイティブJavaHL 1.5.4 SVNコネクター


ブランチライフサイクル



開発者の1人として、自分に割り当てられたタスクを実行する通常のブランチの寿命の例を考えてみましょう。

最初、リポジトリにはコードの主要部分（トランク、以降「トランク」）のみが含まれ、その現在のリビジョンはr10です。 T1トランクの状態は、ブランチが作成された時点のコードのバージョンに対応しています（ブランチ、以下「ブランチ」）。

リビジョンr11で始まる独自のブランチを作成します。 次に、コードの並行作業が開始され、その間にブランチに変更B1のボリュームを作成します。 同時に、他の開発者がT2の変更をトランクに導入しています。

次に、トランクに加えられた最新の変更をブランチに含めます。 現在、ブランチの状態はT1 + B1であるため、ブランチを作成した瞬間からトランクで発生した変更のみに関心があります。 これらの変更（T2）は、最初と最後のリビジョンの番号によって一意に決定されます。この例では、r10とr12です。 10から12リビジョンのトランクで行われた変更をブランチの現在のバージョンとマージし、必要に応じてエラーを修正し、マージ結果をブランチにコミットします（リビジョンr14）。



これで、トランクからのすべての最新の変更があり、ブランチの状態はT1 + B1 + T2です。

さらなる開発が継続され、ブランチで変更量B2を作成し、他の開発者がトランクで変更量T3を作成します。

再びトランクからの最新の変更をブランチに含めます。 ブランチの状態はT1 + B1 + T2 + B2であるため、必要な変更量はT3だけです。 マージを実行するとき、最初のリビジョンとして、前のマージの最終リビジョンを指定します（r12）。 したがって、12から15リビジョンの間のトランクで行われた変更をブランチの現在のバージョンとマージし、必要に応じてエラーを修正し、マージ結果をブランチにコミットします（リビジョンr17）。

割り当てられたタスクを解決したので、今度はトランクに加えた変更を紹介します。 ブランチを定期的にトランクに合わせて、発生する問題を迅速に解決するため、操作はそれほど複雑である必要はありません。 ブランチの状態はT1 + B1 + T2 + B2 + T3になります（さらに、マージ中に行った修正B1 'およびB2'）。 トランク状態はT1 + T2 + T3になりました。 したがって、11から17リビジョンのブランチで行われた変更をトランクの現在のバージョンとマージし、必要に応じてエラーを修正し、トランクの新しいバージョンをコミットします（r18）。

その後、ブランチを削除し、作業を続行して新しいブランチを作成することをお勧めします。

マージ実行

ステップ2.マージトランクの初期および最終リビジョンの決定


最初のマージの最初のリビジョンとして、ブランチが作成されたトランクのリビジョンを取得する必要があります。 私たちの場合、これは40898です。この数は、ブランチのルートフォルダーの履歴を調べて、[コピー時に停止]ボタンを使用して確認できます。



明確にするために、ブランチを作成するときにフォームのコメントを記述することをお勧めします。

後続の合併では、前のマージで指定されたトランクの初期リビジョンがトランクの初期リビジョンとして取得される必要があります。 その数を忘れないように、合併の結果をコミットするとき、次の形式のコメントを書きます：

この時点で、次のアクションが自動的に発生します。

  1. svn mergeコマンドが実行されます。
  2. トランクで変更されたがブランチで変更されなかったすべてのローカルファイル（ChangedByOther.java）は、トランクのバージョンに置き換えられます。 したがって、ディスク上のトランクからファイルが作成され、作業したこのファイルの以前のバージョンはブランチのsvnにあります。
  3. トランクとブランチの両方で変更されたすべてのローカルファイル（Conflicted.java）は、次の形式のdiffファイルに置き換えられます。

競合ファイルごとに、プロジェクトフォルダーに3つのユーティリティファイルが作成されます。

ステップ4.外部変更の処理


最初に、変更されていない更新済みファイルを処理します。 私たちのタスクは変更をトランクからブランチにアップロードすることであるため、[すべての着信変更を受け入れる]ボタンをクリックすることは理にかなっています。

必要に応じて、各ファイルについて個別に決定し、ダブルクリックします-比較エディターが開きます。 トランクからのファイルのバージョンがウィンドウの左側に表示され、ブランチからのバージョンが右側に表示されます。



変更を破棄するには、比較エディターで双方向比較モードを有効にし（ボタンを押す必要があります）、「競合しないすべての変更を右から左へコピー」ボタンをクリックし、ファイルを保存します（Ctrl-S）。 保存すると、ファイルは「同期」ビューから消えます。

変更を受け入れるには、「同期」ビューでファイルを右クリックし、「受け入れる」を選択します。 ファイルが[同期]ビューから消えます。



ステップ5.競合の処理


私たちのタスクはトランクからブランチに最新の変更をアップロードすることなので、トランクに加えられたすべての変更をそこに保存するように競合ファイルを修正する必要があります。 これらの変更が失敗したと思われる場合は、作成者とこれについて話し合い、トランクを変更して、マージプロセスを繰り返します。

各ファイルを処理してダブルクリックすると、比較エディターが開きます。 エディターの上部パネルで特定の変更を選択すると、トランクの競合するフラグメントがウィンドウの左側に表示され、ブランチの競合するフラグメントが右側に表示されます。



変更を破棄するには、比較エディターで双方向比較モードを有効にし（ボタンを押す必要があります）、エディターの上部パネルでコンパイル単位要素を選択し、「競合しない変更をすべて右から左にコピー」ボタンをクリックして、ファイルを保存します（Ctrl-S）。 次に、「同期」の形式で、ファイルを右クリックし、「マージ済みとしてマーク」を選択します。 ファイルが[同期]ビューから消えます。

変更を受け入れるには、「同期」の形式で、ファイルの右ボタンをクリックして「競合の編集」を選択します-「競合の編集」モードで比較エディターが開きます。 その中に（驚き！）ファイルのバージョンがウィンドウの左側に表示され、トランクのバージョンが右側に表示されます。 その後、比較エディターで、2者間比較モードを有効にし（ボタンを押す必要があります）、「競合しない変更をすべて右から左にコピー」ボタンをクリックし、ファイルを保存します（Ctrl-S）。 さらに、ファイルはウィンドウの左側で手動で編集できます。 次に、「同期」の形式で、ファイルを右クリックし、「マージ済みとしてマーク」を選択します。 ファイルが[同期]ビューから消えます。





ステップ6.失敗した変更のロールバック


たとえば、拒否する代わりに変更を受け入れた（またはその逆）など、誤って何か間違ったことをした場合、すべてをいつでも簡単に修正できます。

「プロジェクトエクスプローラ」の形式でブランチからファイルのバージョンにロールバックするには、プロジェクトを右クリックして、[チーム]-> [元に戻す]を選択します。

マージプロセス中に作成されたサービスファイルは、手動で、または「プロジェクトエクスプローラ」の形式でプロジェクトの右ボタンを押し、チーム->リポジトリと同期を選択して削除できます。 その後、[同期]ビューで、プロジェクトの[オーバーライドと更新...]ボタンをクリックし、[プロジェクトエクスプローラー]ビューを更新（F5）します。

マージを再開するために、[マージ]ダイアログを再度呼び出す必要はありません。 [同期]ビューでF5キーを押すだけです。 必要に応じて、最初に[同期]ビューの左端のツールバーボタンを押して、目的の同期画面を選択する必要があります。





ステップ7.ブランチにコミットする


各ファイルについて決定した後、ブランチに対して次の操作を実行する必要があります。

  • コンパイルエラーを排除します。
  • プロジェクトをコミットするには、フォームのコメントを必ず書いてください：

    Merged with trunk@40898-40906





そのため、トランクからブランチに最新の更新をダウンロードしました。 この操作を定期的に実行すると、マージ時に発生する問題の量が減ります。
