サーバー上のExt JS

Ext JSライブラリに関しては、専門家から非常に多くの否定的意見を聞く必要があります：重い、高価な、バグだらけ。 原則として、ほとんどの問題はそれを調理できないことに関連しています。 Sencha Cmdとすべてのcssを使用して正しく組み立てられたプロジェクトは、1Mbリージョンでの生産の重量を量ります。これは同じAngularに匹敵します。 はい、グリッチはそれほど多くありません...



Senchaの発案がこれとは異なる方法で関連している可能性がありますが、その原則的な反対者でさえ、深刻なイントラネットプロジェクトを構築するための最適なソリューションを見つけることは難しいと認めています。



私の意見では、Ext JSで最も価値があるのはUIコンポーネントのコレクションではなく、優れたOOPアーキテクチャです。 近年のJSの急速な発展を考慮しても、7年前にExt JSで実装された必要なものの多くは、ネイティブクラス（名前空間、ミックスイン、静的プロパティ、親メソッドの便利な呼び出し）にまだありません。 これが、数年前にバックエンドでExt JSクラスの立ち上げを実験するきっかけとなりました。 最初の同様の実験については、すでにHabréに投稿しました。 この記事では、古いアイデアの新しい実装といくつかの新しいアイデアについて説明します。



始める前に、次の質問に注意してください。あなたはどう思いますか、どこで実行され、以下のコードスニペットは何をしますか？ 



Ext.define('Module.message.model.Message', { .... /* scope:server */ ,async newMessage() { ......... this.fireEvent('newmessage', data); ...... } ... })


このコードはサーバーで実行され、サーバーに接続されているすべてのクライアントマシンの「Module.message.model.Message」クラスのすべてのインスタンスで「newmessage」イベントを発生させます。



サーバー側Ext JSを使用する可能性を示すために、簡単なチャットプロジェクトを分析します。 ユーザーがニックネームを入力したときにログインすることはありません。 一般的なメッセージまたはプライベートメッセージを投稿できます。 チャットはリアルタイムで機能するはずです。 希望する人は、ビジネスでこの経済すべてをすぐに試すことができます。

コメントに注意してください/ *スコープ：サーバー* /および/ *スコープ：クライアント* /-これらの構造は、メソッドのタイプを決定するサーバーのラベルです。



testClientMethod-このメソッドはクライアントでのみ実行され、クライアント側でのみ使用可能です。

testGlobalMethod-このメソッドはクライアントとサーバーで実行され、クライアントとサーバーの部分で使用できます。

privateServerMethod-メソッドはサーバー上で実行され、サーバー上でのみ呼び出すことができます。

$ readは、サーバー側でのみ実行される最も興味深いタイプのメソッドですが、クライアントとサーバーの両方で呼び出すことができます。 プレフィックス「$」は、サーバー側のメソッドをクライアント側で使用できるようにします。

fireEventメソッドには、標準のものとは異なり、追加のパラメーターがあり、イベントがトリガーされるクライアントに渡されます。 1つのクライアント識別子、識別子の配列、または文字列「all」を渡すことは許容されます。 後者の場合、イベントは接続されているすべてのクライアントでトリガーされます。 それ以外の場合、これは標準のfireEventです。

可能な質問



クライアントでのサーバーメソッドの可用性はもちろん良好ですが、セキュリティはどうですか？ 悪意のあるハッカーがサーバーコードを見て、バックエンドをクラックしようとすることができますか？



いいえ、彼は成功しません。 まず、すべてのサーバーメソッドは、クライアントブラウザーに送信されるときにクラスコードから削除されます。 このため、コメント/ディレクティブ/ * scope：... * /が意図されています。 第二に、最も公開されているサーバー側メソッドのコードは、クライアント側でリモート呼び出しメカニズムを実装する中間構造によって置き換えられます。



再びセキュリティについて。 クライアントでサーバーメソッドを呼び出すことができる場合、そのようなメソッドを呼び出すことはできますか？ そして、これがデータベースのクリーニング方法である場合は？



クライアントからは、名前に$プレフィックスが含まれるメソッドのみを呼び出すことができます。 このような方法の場合、チェックとアクセスのロジックを自分で決定します。 外部ユーザーは、$なしでサーバーメソッドにアクセスできず、それらも表示されません（前の回答を参照）



クライアントとサーバーが密接にリンクされているモノリシックなシステムを手に入れたようです。 水平スケーリングは可能ですか？



実際、システムはモノリシックに見えますが、そうではありません。 クライアントとサーバーは、異なるマシン上で「ライブ」できます。 クライアントは、サードパーティのWebサーバー（Nginx、Apacheなど）で実行できます。 クライアントとサーバーの分離の問題は、自動プロジェクトビルダーによって非常に簡単に解決されます（これについては別の投稿を書くことができます）。 内部サービスメッセージングメカニズムを実装するために、システムはキューを使用します（これにはRedisが必要です）。 したがって、新しいマシンを追加するだけで、サーバー部分を簡単に水平方向に拡張できます。



通常の開発アプローチでは、原則として、バックエンドはさまざまなクライアントアプリケーション（Webサイト、モバイルアプリケーション）と接続できる特定のAPIセットを提供します。 あなたの場合、Ext JSで書かれたクライアントだけがバックエンドで動作できることがわかりましたか？



サーバー、特にモジュールモデルでは、特定のビジネスロジックが実装されます。 REST API経由でアクセスできるようにするには、小さな「ラッパー」で十分です。 対応する例は、この記事の最初の部分に記載されています。



結論



ご覧のとおり、かなり複雑なアプリケーションを快適にコーディングするために、フロントエンドとバックエンドに1つのライブラリを用意することで十分対応できます。 これには大きな利点があります。



開発プロセスをスピードアップします。 各チームメンバーは、バックエンドとフロントエンドで作業できます。 「このAPIがサーバーに表示されるのを待っている」という理由でのダウンタイムは関係ありません。



少ないコード。 コードの同じセクションをクライアントとサーバーで使用できます（チェック、検証など）。



このようなシステムを維持することは、はるかに簡単で安価です。 2人の多様なプログラマの代わりに、システムは1人（または同じ2人ですが交換可能）をサポートできます。 同じ理由で、チームの離職に伴うリスクも低くなります。
