犬がいないと......

ビジネスでロシア語のライブドキュメントを使用したBDDが必要な場合、テスターの専任の職位がないか、独立した自動化には知識レベルが不十分であり、スクラムチーム間で単一の技術スタックを提供することが重要です。アクション。

秋田図書館とは何かについて話を始める前に、その実装のアイデア、どのような問題を解決したいのか、どのような結果になったのかを説明したいと思います。



この記事は、Anna anna-che 、Alice mutaliska 、およびKsenia KseMishによって作成されました 。



何が欲しかった？



ユーザーに高品質のソフトウェアを継続的に提供したい多くの企業と同様に、テストと関連ドキュメントで完全にカバーされた既製の機能を1つのスプリントで提供できる継続的デリバリーを実装したいと考えました。 しかし、CDへの移行中に、開発プロセスの多くの機能に出会いました。



どうした？



  1. 多数のスクラムコマンド。
  2. 高いリソースコスト。
  3. 長い回帰。


これらの機能を分析した後、私たちはあきらめませんでしたが、連続配信パスを入力できるアクションの2つのオプションを見つけました。



  1. 各チームに1人の自動テスト開発者を引き付けます。
  2. チームテスト担当者（だけでなく）をテストして、自動テストを作成します。


最初のオプションは、自動テストの無料の開発者が市場にあまりいないため高価であるため、私たちには適していませんでした。 しかし、2番目は費用対効果と拡張性に興味がありました。



何をする必要がありますか？



  • テスト自動化のための単一のツールを開発します。
  • テスターに​​低しきい値を設定して、コマンドを入力します。
  • 自動化を開始するには、プログラミング言語の最低限の知識が必要です。
  • 自動テストはテスターだけでなく、チームのメンバーにも書きます（テスターだけでなく製品の品質にも責任があります！）。
  • 自動テストを開発する時間を短縮します。
  • すべてのチームで単一の技術スタックを使用します。


どうしたの？



秋田が生まれました。 映画「ハチ公」の悪名高い犬の品種にちなんで名付けられた秋田図書館。 私たちは、この名前を、常にシステムを支援し、多数の欠陥から保護する準備ができている真の友人に関連付けます。



現時点では、秋田は60を超える実装されたステップで構成されており、これらを使用して、Webアプリケーションのテストを自動化することをすでに想定できます。



どのように機能しますか？



秋田は、テスト自動化のためのBDDステップライブラリです。 BDDは、ユーザーの行動の記述に基づく開発方法論です。 つまり、「ユーザーとして、将来の費用を計画するために[月ごと]ボタンをクリックしたときに月間費用のスケジュールを確認したい」などの説明を書く人がいます。 このような説明は、開発へのエントリポイントとして機能します。 BDDフレームワークでのスクリプトのステップを説明するために、与えられたキーワード、いつ、次に、使用されます（それぞれ、いつ、次に与えられます）。



ライブラリでは、テストはロシア語で記述され、アプリケーションのユーザードキュメントとして機能するユーザースクリプトを表します。 各ユーザースクリプトはステップで構成され、各ステップは、実装が実装される特定のJavaメソッドに関連付けられています。









ライブラリを使用したプロジェクトテンプレートは非常に簡単に構築され、次のようになります。









秋田を使用したテンプレートの主なコンポーネントは、機能ファイル、ステップ（以下、ステップと呼びます）、およびWebページ（以下、ページと呼びます）です。 機能ファイルには、システムの対象となるスクリプト（テスト）のリストが記述されており、将来的に起動される予定です。 Stepsは、機能ファイルに記述されているステップに関連付けられた開発済みのメソッドを保存します。 Webページとその要素の説明は、ページパッケージにあります。



ライブラリを操作する原理を理解するために、次の例を検討することをお勧めします。



スクラムチームのメンバー（この場合、アナリストまたはテスター）はユーザースクリプト（ユーザーストーリー）を作成します。これは、将来、開発タスク、自動テストのチェックセット、およびユーザードキュメントとして機能します。

たとえば、「Alfa-Bankユーザーは、Alfa-Clickシステムにユーザー名/パスワードでログインしてアカウントを表示します」というユーザーストーリーがあります。



テスター（または他のチームメンバー）は以下を行う必要があります。



1. java gradleプロジェクトを作成し、以下に応じて次のエンティティを追加します。



  • 秋田：

    en.alfabank.tests：秋田：3.1.0
  • レポートを生成および添付するためのプラグイン：

    en.alfalab.gradle：cucumber-reporting-gradle-plugin：0.1.0
  • テストを並行して実行するためのプラグイン：

    en.alfalab.gradle：cucumber-parallel-test-gradle-plugin：0.2.1
  • ステップの実装に関する束：

    generateRunner.glue = ["en.alfabank.steps"、 "steps"]


2.完成したユーザースクリプトを、秋田ライブラリの既製の手順を使用してプロジェクトの.featureファイルに転送します。 その後、機能ファイルは次のようになります。 



:   - :    -        " "   "http://alfabank.ru"    ""   "login"    ""   "password"      ""   "-"


３ 。 作業が必要な要素を持つページのページオブジェクトを作成します。ログインページとメインページ「Alpha-click」。 ページパッケージにログインページ（LoginPage）のクラスを作成します。 



 @Name(" ") public class LoginPage extends AkitaPage { @FindBy(id = "login") @Name("") private SelenideElement login; @FindBy(id = "password") @Name("") private SelenideElement password; @FindBy(id = "submit") @Name("") private SelenideElement submitButton; }


AkitaPageを継承し、クラス名とページ要素に@Name



アノテーションを付け、それらのキー記述を.featureファイルで使用することが重要です。 同様に、チェックに参加するすべてのページを記述する必要があります。



すべてのページを説明した後、「しかし、ページがロードされたことをどのように理解しますか？」という疑問が生じます。 ステップ「アルファクリックページがロードされました」では、アルファクリックページが現在のページとして設定され、すべてのSelenideElementsが検索されます。 すべての要素が見つかった場合、ページはロードされています。



ライブラリでのこのステップの実装は次のようになります。 



 /** *  ,   ,        @Name,    @Optional    *   WAITING_APPEAR_TIMEOUT,     "waitingAppearTimeout"  application.properties.    ,    8  */ @("^(?:|||) \"([^\"]*)\" (?:|)$") public void loadPage(String nameOfPage) { akitaScenario.setCurrentPage(akitaScenario.getPage(nameOfPage)); akitaScenario.getCurrentPage().appeared(); }


要素がページに属しているが、読み込み直後に表示されない場合（たとえば、誤って入力されたデータに関するメッセージ）、読み込みの決定に参加してはならず、@ Optionalアノテーションでマークされます 必須ではありません。



4.テストを実行します。



テストを実行するにはいくつかの方法があります。



  • gradleタスク：test、generateCucumberReport
  • VMプロパティのインストールを忘れてはならない特定の機能ファイルのコンテキストメニュー：-Dbrowser = chrome
  • ターミナルから：


> gradlew clean test -i

> gradlew clean test -Pbrowse=chrome -Pprofile=local -PremoteHub=http://remote/wd/hub

> gradlew clean test -Dbrowser=chrome -Ptag=@test







プロジェクトの場合、すべての機能ファイルの並行起動（独自のランナーが作成される機能ファイルごと）、リモートマシンでのテストの起動（-PremoteHub）、ブラウザーの変更（-Pbrowser = chrome）、スクリプトに設定されたタグによるテストの開始（-Ptag） 。



テストの実行後にレポートを受信するには、ターミナルで次のコマンドを実行する必要があります。



> gradlew generateCucumberReport

> gradlew gCR







その結果、標準のCucumberレポート（ビルド/レポート）が生成されます。









そしてそれだけです。 ユーザーストーリーはテストでカバーされ、ユーザードキュメントが生成され、テストの合格に関するレポートが生成されます。

あなたは何をしましたか？



使用されているすべてのテクノロジーとその類似物を研究した結果、次のスタックに到達しました：Java + Gradle + Selenide + Cucumber。



Alfa-BankではJavaとGradleを使用しているため、ここではあまり選択肢がありませんでした。



Cucumberは扱いやすく、かなり大きなコミュニティを持っているため、JBehaveをCucumberに置き換えました。



Selenideライブラリーを使用すると、ブラウザー、タイムアウト、動的ページ、およびAjax要求を簡単に処理できます。 Selenideの著者であるAndrei Solntsevは、彼のLJにあるライブラリに注目し、元のコードをレビューする際にいくつかの貴重なヒントを与えました。



利点は何ですか？



秋田図書館を使用すると、次の結果が得られました。



  1. 新しい従業員の入場基準は下がりました（現在、主要なオンボーディングには1〜2日かかり、プログラミング言語のライブラリ知識の基本的な機能を最小限に抑えるため、ツールの自由なオリエンテーションは2週間で達成されます）。
  2. Webアプリケーションのテストを自動化する単一のツールがインストールされています（現在、Alfa Bankの20以上のチームがライブラリを使用しています）。
  3. 受け入れ+回帰には1時間しかかかりません（ほとんどの機能-約80％が自動テストでカバーされているため、回帰ははるかに高速です）。


秋田は日々、スプリントからスプリントまで、ますます高品質の製品をユーザーに提供するお手伝いをしています。 したがって、ライブラリがある場合は、自動テストの方がはるかに興味深く簡単になると言っても安全です。
