AWSで落ちましたか？ 質問なしで後で立ち寄って、今では前ではない

私が仕事に運転してドルフィンの新しいアルバムを聞いている間、誰かがサブネット全体でAmazonとGoogleのIPアドレスをブロックしていました。 Roskomnadzorは、Telegramに関連していないがメッセンジャーと同じサービスでIPアドレスをレンタルしているサイトのブロックに関する不正確な情報を呼び出し、そのようなメッセージの拡散を防ぐホットラインを作成しました。



そのため、過去1日に複数のサービスをサポートする必要があった理由もわかりません。 誰かがかなり傷ついたが、誰かが移行しなければならなかった。



つまり、AWSからすばやく移行できます。 互換性のあるAPIがあります。 Googleから顧客を移行することもできます。 そして、今後数週間のうちに、無料で移行を行い、契約書や保証書なしで、会社のカードに書かれたクラウドでのテストに1か月を与える準備ができています。 気に入らなければ、何も支払う必要はありません。



以下は、Amazonからのさまざまな種類のVMのアンロードの方法です。



現在、ロシアの雲への関心が急増しています。 ロックが発生したユーザーは、待機するか、サービスをアクセス可能なパブリッククラウドに緊急に移行できることを理解しています。



移行には、アクセス、サポートとの合意、合意の締結が必要です。



これらの手続きをスキップして、MBerezin @ croc.ruに手紙でアクセスできるようにする準備ができています。



適合性



CROCクラウドは、AWSシナリオに従ってさまざまな方法で進化しています。KVMに基づいて仮想化プラットフォームを開発しています。 当然のことながら、私たちは西洋の同僚の経験を取り入れているため、AWSとの互換性も自慢できます。



論文：



  • データセンターの独自のネットワークに基づく2つのアクセスゾーン、
  • タイプに関係なく、すべてのVMで同等に高速なネットワーク、
  • VMとデータセンターにある物理インフラストラクチャ間の高速ネットワーク、
  • IOPSが固定された高性能ドライブ（AWSのアナログio1）、
  • 仮想ディスクのフルサービスインポート/エクスポート、
  • S3でユーザートラフィックを解放します。


カスタマイズされた自動化は、深刻な問題なくクラウドで動作します 。 Elastic Compute Cloud（EC2）、Simple Storage Service（S3）、CloudWatchの3つのサービスの基本機能を実装しました。



移行オプション



1. VMのインポート/エクスポートの標準手順は、ドキュメントに記載されています 。

ただし、重要な制限があります。エクスポートは、以前にAWSにインポートされたマシンに対してのみ可能です。 そうでない場合、エラーが発生します。 



$ aws ec2 create-instance-export-task —instance-id i-XXXXXXXXXXXXXXXX —target-environment vmware —export-to-s3-task DiskImageFormat=vmdk,ContainerFormat=ova,S3Bucket=my-export-bucket An error occurred (NotExportable) when calling the CreateInstanceExportTask operation: Only imported instances can be exported.


インスタンスのエクスポートに成功した場合は、AWS S3からインスタンスをダウンロードする必要があります。



次に、ダウンロードしたイメージをCROC S3にダウンロードします。 これを行うには、CROC Cloud APIアクセス設定ファイルをダウンロードし、S3で動作するようにs3cmdを構成し、ディスクイメージをロードする必要があります。



2.インスタンスをエクスポートできない場合（マシンがAWSテンプレートから作成された場合）、次の制限をバイパスできます。VMディスクをイメージファイルにダンプする必要があります。イメージファイルはS3にロードする必要があります。



a）オリジナルよりも少し大きい一時的なEBSボリュームを作成します

b）一時ボリュームにファイルシステムを作成し、マウントします。 



 # mkfs.xfs /dev/xvdf1 # mkdir /tmp/export # mount /dev/xvdf1 /tmp/export


c）qemu-imgをインストールします

d）お尻とサービス全体を停止し、ディスクイメージを削除します。 



 # qemu-img -O qcow2 -o compat=0.10 /dev/xvda /tmp/export/web1-xvda.qcow2


e）ファイル/tmp/export/web1-xvda.qcow2をCROC S3にダウンロードし、そこからテンプレートを作成して、最初のバージョンと同じ方法でそこからインスタンスを開始します。

f）一時的に作成されたEBSボリュームをアンマウントして削除します



3.他のOSの場合も同様に、Windowsでは、たとえば、ディスクコントロールパネルからVHD / VHDXを作成できます。



4.バックアップのオプションを検討し、別のマシンに復元できます。



5.ファイル（コンテンツなど）の転送で十分な場合、rsyncはLinuxで、robocopyはWindowsで使用できます。



6.ダウンタイムまたは大容量が非常に高価な場合は、Double Take / Carboniteを使用してOSおよびファイルシステムのレプリケーションを操作できます。



一般に、多くのオプションがありますが、すべては特定のインフラストラクチャの配置方法に依存します。 これらのオプションのすべてにおいて、当社のサポートは十分に理解されており、こうした移行の経験があります。 さらに、認定されたAWSスペシャリストがいますので、間違いなくお手伝いできます。
