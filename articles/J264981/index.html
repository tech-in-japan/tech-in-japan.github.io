アスタリスク 開始する

この記事を書くために、私はアスタリスクのセットアップに関するハウツーのほとんど完全な欠如に促され、初心者には明らかな例がありました。 ネットワークでは、IVRの設定、LDAPを介したSIPユーザー認証の設定、内部にアスタリスクを使用したHAクラスターの作成に関するマニュアルなどに関する多くの情報を見つけることができますが、ゼロから取得する方法に関する記事はありません。例。 ほとんどすべての場所で、アスタリスクが提供するすべての機能をすぐに使用することが提案されており、マニュアルで提供されている機能の一部を削除すると、ほとんどの場合、動作不能の設計になります。 この記事は熊手を歩いた結果です...マニュアルを読んでください。 数年前と同じ状況にいるなら、猫へようこそ。



そのため、状況：アスタリスクとその機能を知ったばかりで、自宅で使用したい、または管理者がオフィスでIPテレフォニーを望んでいました。



最初に必要なのは、OSがインストールされたサーバーです。 サーバーにはCentOS 6を選択しましたが、最も安定したサーバーが取得されるのはこのOSであるため、すべての例はこのOSに関連付けられます。 しかし、私はあなたの選択を制限しようとはしていません。 Asterは通常、Debian、Arch、FreeBSDの両方で起動します。 注：「メタリック」な音声、またはその完全な欠如を得ることができるため、初期段階でハイパーバイザーを使用しないことをお勧めします。 タイマーと割り込みについては、次の記事で説明します。これらの説明と設定はこの記事の範囲を超えているためです。 無料のサーバーがなく、仮想マシンなしでは実行できない場合は、x86番目のゲストを必ず配置してください。



サーバーにOSをインストールしたら、アスタリスクのインストールに進むことができます。 Didgiumリポジトリを追加します。 

  rpm -Uvh http://packages.asterisk.org/centos/6/current/i386/RPMS/asterisknow-version-3.0.1-2_centos6.noarch.rpm
ヤムアップデート


dnsmasqをインストールします。 

  yum install dnsmasq


アスタリスクをインストールします。 

  yum install asterisk asterisk-configs --enablerepo = asterisk-12


重要な注意：行末の数字は、アスタリスクのメジャーバージョンを示しています。 この例では、12番目のブランチの最新バージョンがインストールされ、リポジトリで利用できます。



インストール後、サーバーを再起動し、アスタリスクシェルに移動します。 

 アスタリスク-rv


シェルが起動するキーに注意してください。 キーの数「v」は、通話中に表示される情報の量に影響を与えます。その数は1〜14です。



インストールが成功し、アスタリスクが開始された場合、次のプロンプトが表示されます。 

 アスタリスク* CLI>


おめでとうございます、アスタリスクがインストールされ、準備が整いました。 しかし今のところ、シェルは必要ないので、exitを記述します。



時間を節約し、資料を理解しやすくするために、特定の例を使用して理論を検討し、セットアップ時、サポートオペレーターおよび同僚とのコミュニケーション時に使用する必要がある主な用語を分析します。 たとえば、100と101の番号を持つ2人の内部ユーザーと、2人の電話オペレーターがいます。 事業者の1人は、登録付きのトランクを介して都市番号を提供し、もう1人は、登録なしのトランクを使用して長距離方向へのアクセスを提供します。



最初に行う必要があるのは、/ etc / asterisk / sip.confファイルにピアを作成することです。 それを開きます： 

  nano /etc/asterisk/sip.conf


そして、すぐにファイルの最後に移動します。 次のテキストを貼り付けます。 

  [内部]（！）
タイプ=友人
安全でない=招待、ポート
コンテキスト=オフィス
 fromdomain = <ドメイン名またはIP>
ホスト=動的
不許可=すべて
 allow = alaw
資格=はい
 canreinvite = no
 nat = no

 [100]（内部）
シークレット= XXX
 [101]（内部）
シークレット= XXX


100と101の番号を持つ2つの内部ピアを取得しました。これらの設定を理解することが、アスタリスクの実装を成功させる鍵であるため、これらの設定を分析します。



[内部]（！）



[内部]、これはテンプレートの名前であり、（！）テンプレートへのポインタです。

すぐにテンプレートを使用する理由 セットアップ時間を短縮し、構成ファイルからのテキストの量を減らすため、理解するために非常に簡単だからです。



タイプ=友人



利用可能なオプションは、peer、user、およびfriendです。 多くの場合、それらの違いについて誤った意見があります。 多くの人は、「ユーザー」パラメーターは発信通話のみを許可し、「ピア」は着信通話のみを許可し、「友人」は両方向の通話を許可すると考えています。 そうではありません。 「ピア」キーを使用すると、通話中のユーザー名とパスワードの一致が無効になります。 「ピア」パラメーターを使用する場合、アスタリスクは、「ユーザー」を使用する場合、呼び出し元のIPアドレスとポート番号の対応のみをチェックします。ユーザー名フィールドはチェックされ、ソースアドレスはチェックされません。 「friend」パラメーターは、ユーザー名とIPアドレスを強制的にチェックします。



安全でない=招待、ポート

招待-着信コールの認証は無効です。

port-送信元ポートのチェックは無効です。



初期設定時に、両方のキーを書き留めます。



コンテキスト=オフィス



このデバイスからの発信コールが処理されるコンテキスト。 以下の詳細。



fromdomain = <ドメイン名またはIP>



SIPドメインの名前。 初期セットアップでは、サーバーのIPアドレスをアスタリスクで指定します。



ホスト=動的



ピアのIPアドレス。 ログインとパスワードによる認証を使用する場合-動的に設定します。 特定のIPは、ピア設定が登録なしでトランクに使用される場合にのみ示されます。



不許可=すべて

allow = alaw



許可されるコーデックを指定します。



この例では、1行目ですべてのコーデックの使用が禁止され、2行目でg711-aが許可されています。 コーデック設定はそれぞれの場合に個別ですが、ほとんどのロシアとウクライナのプロバイダーはg711aとg729を使用します。 後者は所有権であり、アスタリスクはパススルーモードでのみサポートされます（つまり、トランスコーディングは不可能です）。



資格=はい



この行により、アスタリスクはOPTIONSパケットでデバイスまたはソフトフォンをポーリングします。 監視とトラブルシューティングに必要です。



canreinvite = no



デバイス間でメディアを直接転送することを禁止します。 セットアップを簡素化するために「no」に設定することをお勧めします。



nat = no



アスタリスクに、ごちそうはnatのためではないことを伝えます。 サーバーがその背後にある場合、キーとその使用例の説明は、記事の範囲外です。 回避策については、次の記事で説明します。



[100]（内部）

シークレット= XXX

[101]（内部）

シークレット= XXX



ここで、ごちそうの名前を設定し、テンプレートから設定を取得します。 この例で唯一のパラメーターは、ピア名とパスワードです。

これで、内部ピアの構成が完了しました。次に、オペレーターとのインターフェースの構成に進みましょう。 sip.confの最後に2つのエントリを追加します。

各行の値をペイントしません。なぜなら ホストフィールドを除き、すべての設定は内部ピアの設定と同じです。 アスタリスクがクライアントとして機能する場合（およびオペレーターサーバーの場合、アスタリスクはクライアントです）、オペレーターサーバーのアドレスまたはそのDNS名を指定する必要があります。



2番目の演算子を追加します。

 fromdomain = <ドメイン名またはIP>
ホスト= 5.6.7.8
安全でない=招待、ポート

また、ユーザー名と秘密の行がないことを除いて、違いはありません。2番目のオペレーターは登録を使用しないためです。



オペレーターNo. 1のサーバーで登録を構成します。 これを行うには、sip.confファイルで、オペレーター設定を説明するセクションの前に、次の行を挿入します。 

  register => udp：// YourLogin：YourPass：YourLogin@1.2.3.4/YourLogin


構文は、準備ができていない人にとっては複雑に見えるので、わざわざサポートサービスプロバイダーに電話をかけずに、アスタリスクサーバーへの登録方法を確認するか、設定の送信を依頼してください。 原則として、ほとんどのオペレーターは問題なくソフトスイッチの構成例を送信します。



確かに、制限された権限を持つユーザーとして構成ファイルを編集しました。 その場合、アスタリスクは構成ファイルにアクセスできないため、次のように記述します。

これでピアのセットアップが完了しました。次に、ダイヤルプランのセットアップに進みましょう。 ダイヤルプランはアスタリスクの心臓部であり、それを利用して、絶対にすべての通話が処理されます。 アスタリスクはいくつかの言語を理解しますが、この例では、アスタリスクの最初のリリースで登場した標準の言語を使用します。 設定ファイルは/etc/asterisk/extensions.confファイルに保存されます。



コマンドで開きましょう： 

  nano /etc/asterisk/extensions.conf


多くのデフォルトルールは標準構成ファイルに保存されます。 それらは必要ないので、内容を消去して次のように記述します。 
  [一般]
静的=はい
 writeprotect = no

 [グローバル]


これらは、ダイヤルプランの通常の読み取りに必要なパラメーターであるため、変更しないでください。



簡単なものから始めましょう。 100番から101番まで呼び出す必要があります。これを行うには、グローバルセクションの直後にルールを記述する必要があります。 

  [オフィス]
 exten => _1XX、1、ダイヤル（SIP / $ {EXTEN}）


この行を解析しましょう。

[office] -ピアからの呼び出しが処理されるコンテキストの名前。

exten => -ステップの開始へのポインター。

_1XXはマスクです。 マスクについてさらに詳しく説明します。 これにより、ダイヤルプランコンテキストに該当するすべてのコールがソートされます。



マスクは、一連のパターンを使用してcaller-idでコールをソートします。 

  X-0〜9の任意の数字
 N-2〜9の任意の数字
 [234-6]-番号2、3、4、6
 。  -可能性のあるキャラクター


マスクは記号「_」で始まります。これは、テンプレートであることを意味します。 指定を忘れた場合、アスタリスクは着信番号に1XXを使用し、電話番号の代わりにダイヤルアップパターンを指定し、通話は行われません。

1-アクション番号。



ダイヤルはアプリケーションです。 ダイヤルプランでは、200を超えるさまざまなアプリケーションを使用して、通話を操作できます。 さて、最初の段階では、Dialという1つのアプリケーションのみを使用します。 名前から、それが電話をかけるために使用されることは明らかです。



（SIP / $ {EXTEN}） -アプリケーションの引数。 この例では、内部ピアにSIPプロトコルを使用しているため、ダイヤルに渡す最初の引数は、使用されているシグナリングプロトコルを示しています。 $ {EXTEN}-現在の拡張子。その値はヘッダー（宛先フィールドから）から取得されます。



/渡す引数の区切り文字です。



各パラメーターはコンマで区切られます。 この例では、3桁の数字が使用されているため、マスクには3つのパターンが必要です。 異なる番号の長さを使用する場合は、マスクに目的の数のパターンを書き込みます。 ファイルを保存し、アスタリスクシェルを開いてコマンドを記述します。 

 ダイヤルプランのリロード


ファイルへのアクセスに問題がある場合は、次のように記述します。 

  chownアスタリスク：アスタリスク/etc/asterisk/extensions.conf


そして再び、アスタリスクのシェルを介してダイヤルプランを再起動します。



今すぐ電話してみてください。 すべてを正しく構成すると、番号100のごちそうは番号101のごちそうに到達します。これで、オフィスコンテキストで機能するピア間でオフィス内を呼び出すことができます。 都市に電話をかけるには、最初のオペレーターを介した発信通話のルールを記述する必要があります。 オフィスのコンテキストで記述します。次のようになります。 

  exten => _XXXXXXX、1、ダイヤル（SIP / $ {EXTEN} @ operator1）


ダイヤルプランを再起動して、市に電話をかけます。



次に、着信コールを受け入れる必要があります。 ダイヤルプランに新しいコンテキストを作成して、ファイルの最後に書き込みます。 

  [operator1]
  exten => s、1、Dial（SIP / 100＆SIP / 101）


「s」の意味を説明します。 これは、アスタリスクの標準マスクであり、絶対にすべての課題が該当します。 つまり、登録でトランクを使用する場合、着信caller-idは「s」になります。



ダイヤルプランを保存してリロードします。 これで、固定電話番号を提供するオペレーターを介して着信コールを受信できます。



ここで、連邦番号を携帯電話にかける必要がありました。 [office]コンテキストにもう1行追加します。 

  exten => _89XXXXXXXXX、1、ダイヤル（SIP / $ {EXTEN} @ operator1）


保存し、ダイヤルプランを再起動して呼び出します。 それだけです これで、固定電話番号を提供するオペレーターのサービスを完全に使用できます。



ただし、他の地域の携帯電話で電話をかけるときに生じる小さな問題があります。 電話番号は連邦政府であるため、すべての通話は最初のオペレーターを経由します。 したがって、ダイヤルプランのマスクのサイズを小さくすることが適切です。 例： 

  exten => _8909 [89] [2-9] [1-3] XXXX、1、ダイヤル（SIP / $ {EXTEN} @ operator1）
 exten => _8901 [456] [2-5] 1XXXX、1、ダイヤル（SIP / $ {EXTEN} @ operator1）
 exten => _XXXXXXX、1、ダイヤル（SIP / $ {EXTEN} @ operator1）


この設計では、890982XXXXX-890999XXXXXおよび8901421XXXX-8901651XXXXの範囲のコードを持つ電話へのすべての呼び出しは、最初の演算子を通過します。 あまり便利ではありませんが、それに同意します。 今後の記事では、コールの方向に応じてオペレーターを自動的に選択するためのダイヤルプラン構成をレイアウトします。これもこの記事の範囲を超えているためです。

次に、2番目の演算子を使用して長距離電話をかけるだけです。 [office]コンテキストに行を追加します。 

  exten => _ [78] [3-689] XXXXXXXXX、1、セット（CALLERID（num）= 74991234567）
同じ=> n、ダイヤル（SIP / $ {EXTEN} @ operator2）


お気づきのように、呼び出しを処理する新しいアプリケーションがあります。 設定（CALLERID（num）= 74991234567） -登録なしでトランクを使用するため、これはオペレーターに発信者IDを送信する必要があることを意味します。 この番号は存在する必要はありません。 ここでは、プロバイダーに自己紹介するだけで、それ以上は行いません。 置換する番号は、オペレーターによって異なります。 長距離通話またはゾーン通話のために発信者IDを別のオペレーターの番号に置き換えるサービスを提供するものもあります。 詳細については、オペレーターにお問い合わせください。 same => n-毎回マスクとアクション番号を書き込まないようにするポインター。 例： 

  exten => 78121234567,1、回答
 exten => 78121234567,1、Set（CALLERID（name）= Trunk_1
 exten => 78121234567.1、ダイヤル（SIP / 1000）


次のものに置き換えることができます。 

  exten => 78121234567,1、回答
同じ=> n、設定（CALLERID（name）= Trunk_1
同じ=> n、ダイヤル（SIP / 1000）


2番目のケースでは、すべてがはるかに単純です。
