クリーンシステムでUIのない​​Jenkins for Android

Habréには、Jenkinsを使用してAndroidアプリケーションを構築するトピックに関する同様の記事が既にあります。 現在の主な機能/追加は次のとおりです。



  1. UIがないリモートLinuxマシンにJenkinsをインストールします。
  2. プライベートリポジトリからアプリケーションをビルドします。
  3. 名前がわからないブランチからアプリケーションを構築する問題を解決します。
  4. .apkファイルをアセンブルした後、それらをFabricに送信し、テスターに​​通知します。
  5. Fabricに送信した後、Google Playでアプリを公開します。
  6. アプリケーションを公開するタスクがテスターに​​よって実行されるのを防ぎます。


Jenkinsのインストール



これで、サーバーにアクセスできました。 UIはありませんが、これはまったく問題ありません。

はじめに、jenkinsサーバー自体をインストールしましょう。 



$ wget -q -O - https://pkg.jenkins.io/debian/jenkins-ci.org.key | sudo apt-key add - $ sudo sh -c 'echo deb http://pkg.jenkins.io/debian-stable binary/ > /etc/apt/sources.list.d/jenkins.list' $ sudo apt-get update $ sudo apt-get install jenkins


このパッケージのインストールが行うことの詳細については、 公式ドキュメントを参照してください 。



インストール後、ディレクトリ/var/lib/jenkins



は、Jenkins自体、プロジェクトリポジトリ、およびJenkinsが機能するために必要なその他のファイルが含まれます。 これで、ブラウザーを介してJenkinsにログインし、CIサーバーの構成を開始できます。



ただし、その前に、Androidアプリケーションを構築するために必要なすべてをインストールする必要があります。



Android SDK + JAVAをインストールします



以下の公式Webサイトで 、必要なディストリビューションへのリンクを見つけることができます。 通常はAndroid Studioと一緒にインストールされますが、この場合は必要ないため、目的のリンクを選択してコマンドラインツールを読み込みます。



執筆時点では、Linuxの現在のバージョンはandroid-sdk_r24.4.1でした。 インストールするには、次のコマンドを実行します。 



 $ wget http://dl.google.com/android/android-sdk_r24.4.1-linux.tgz $ tar -xvf android-sdk_r24.4.1-linux.tgz


次に、プラットフォームツール、ビルドツール、およびプロジェクトで使用するアプリケーションをビルドする現在のAPIをダウンロードする必要があります。 これを行うには、プロジェクトのbuild.gradleファイルを（アプリケーションモジュールレベルで）見ます。 たとえば、私のプロジェクトで、私たちが興味を持っている部分は次のとおりです。 



 android { .... compileSdkVersion 25 buildToolsVersion '25.0.0' .... }


これは、どのバージョンをダウンロードする必要があるかを明確に示しています。 必要なコンポーネントをダウンロードできるように、システム変数に次のパラメーターを追加します。 



 $ export ANDROID_HOME=~/android-sdk-linux #     SDK-Tools $ export PATH=$ANDROID_HOME/tools:$PATH $ export PATH=$ANDROID_HOME/platform-tools:$PATH


さらに、ほとんどのガイドでは、次のコマンドを使用してすべてのコンポーネントを単純にダウンロードすることが提案されています。 



 $ android update sdk --no-ui --all


ただし、ハードドライブのスペースを無駄にして申し訳ありませんので、必要なものだけをダウンロードします。 コマンドを実行して、正確にダウンロードする必要があるものを見つけます。 



 $ android list sdk --all


結果は次のとおりです。







これで、コマンドで必要なものだけをインストールできます。 



 $ android update sdk --no-ui --all --filter 1,2,163,164,168,169,170,171


次のコマンドを使用してJAVAを配置します。 



 $ sudo apt-get install openjdk-8-jdk


いいね！ Jenkinsのビルドとアプリケーションのビルドに必要なすべてを提供しました。 それでは、ジェンキンスのセットアップに移りましょう。



プライベートリポジトリへのアクセスを構成する



gitリポジトリが公開されている場合、またはユーザー名とパスワードを使用してアクセスする場合は、このブロックをスキップして次のブロックに直接進むことができます。



そこで、すべてを正しく行い、SSH経由でリポジトリにアクセスすることにしました。 これを行うには、sshキーを生成して/var/lib/jenkins/.sshディレクトリに配置する必要があります。



jenkinsをインストールすると、jenkinsという名前のユーザーがUNIXマシンに自動的に生成されました。 jenkinsがキーを使用できるように、キーを生成する最も簡単な方法は、jenkinsユーザーの下からです。 



 $ sudo su jenkins -s /bin/bash $ ssh-keygen -t rsa -b 4096 -C "your_email@example.com"


その後、2つのファイルが生成されます（標準名はid_rsa、id_rsa.pub）。 id_rsaファイルは/var/lib/jenkins/.sshディレクトリに配置する必要があります。その後、githubリポジトリ設定にデプロイキー（id_rsa.pubファイルのコンテンツ）を追加する必要があります。



次のようになります。







次に、リポジトリURLを既知のリストに追加します。 これを行うには、リポジトリに連絡して、リポジトリブランチのリストを取得してください。 



 $ sudo apt-get install git #  git $ sudo su jenkins -s /bin/bash $ git ls-remote -h git@github.com:[your_repo_url]


そして、既知のホストのリストにgithubを追加することに同意します。 id_rsaファイルにアクセスできない場合は、jenkinsユーザーの下から作成しませんでした。この場合、jenkinsユーザーにこのファイルを操作するためのアクセス権を与える必要があります。 



 $ sudo chmod 700 /var/lib/jenkins/.ssh/id_rsa


これらの手順を完了すると、プライベートリポジトリにアクセスしてAndroidアプリケーションを構築できるようになります。



アプリケーションを構築するタスクを作成する



それでは、ジェンキンスに移りましょう。 デフォルトでは、ポート8080で使用できます。任意のブラウザーでこのポートに切り替えましょう。 最初の起動後、ファイル/ var / lib / jenkins / secrets / initialAdminPasswordにあるパスワードを使用してログインできます。 その後、Jenkinsパネルに入ります。 まず、ANDROID_HOMEディレクトリへのパスを設定する必要があります。



Jenkinsの管理->システム構成：







また、GradleおよびGitのバージョンであるJDKへのパスも構成します。 これを行うには、JAVAが保存されているディレクトリを指定し、アセンブリに必要なgradleのバージョンを追加します。



Jenkinsの管理->グローバルツール設定：











これで、アプリケーションをビルドする最初のタスクを作成できます。 無料の構成でタスクを作成することを選択します。 [ソース管理]タブで[git]を選択します。 次に、ssh経由でアクセスするためのURL gitリポジトリを指定します。 最初のタスクでは、ブランチからウィザードを収集します。 示されているのはデフォルトです。



次に、gradleスクリプトを実行してビルドステップを追加します。 前に追加したgradleバージョンを選択します。 すべての重要なフィールドには、それらが必要な理由に関する手がかりが含まれています。 たとえば、非標準のプロジェクト構造がある場合、ルートbuild.gradleプロジェクトファイルがある場所と、build.gradleと異なる場合はファイル名を指定する必要があります。







次に、.apkファイルをアーカイブして、後でテスターまたは他の人にダウンロードできるようにします。 これを行うには、アセンブリ後のタスクを追加し、保存するファイルの形式を指定します。 初めてタスクを作成したとき、正規表現形式を使用して目的のファイルが検索されたと想定しましたが、JenkinsはAntパススタイルを使用してファイルを検索することがわかりました。 デフォルトでは、すべてのファイルを拡張子.apkで保存することが提案されていますが、ほとんどのファイルは必要ないため、debug.apkおよびrelease.apkで終わるファイルのみを保存します。







build.gradleスクリプトを使用して署名付きapkをビルドする



Google Playでアプリケーションを公開するには、.apkファイルに特別なキーで署名する必要があります 。 ファイルへのパスを指定してAndroid Studioを使用して署名した場合、build.gradleスクリプトからこれを行う方法を学ぶ必要があります。 次に、最も単純な例とそれを改善する方法を検討します。 キーをバージョン管理下に置き、ビルドスクリプトから参照するだけで十分です。 これを実装する方法の例を次に示します。 



 android { signingConfigs { release { keyAlias 'KEY_ALIAS' keyPassword 'KEY_PASSWORD' storeFile file('RELATIVE_PATH_TO_KEYSTORE') storePassword 'STORE_PASSWORD' } } ... buildTypes { release { signingConfig signingConfigs.release } } }


このソリューションの欠点は、アプリケーションのリリースバージョンを構築するためのパスワードだけでなく、プライベートキーがバージョン管理下に保存されることです。 将来的にキーを削除してbuild.gradleスクリプトを修正した場合でも、コミットの履歴を掘り下げて誰かがこの情報を復元する可能性があることに注意してください。 より適切な解決策は、バージョン管理下になく、たとえば、必要なすべてのパラメーターが書き込まれる〜/ホームディレクトリにあるプロパティファイルを保持することです。



たとえば、プロジェクトでは次のアプローチを使用します。 バージョン管理に設定ファイルと署名ファイルがありません。 開発者は自分たちの手を持っています。 したがって、これに従わない人がアプリケーションに署名できるという問題を解決します。 Jenkins自身がアプリケーションに署名できるように、問題を解決する必要があります。 これを行うには、アプリケーションに署名するときに、2つの場所でファイルを確認します。 最初はプロジェクトのルートディレクトリです。 2番目は、このファイルを保存することが予想されるサーバー上の特定の場所です。 次に、このファイルをjenkinsサーバーに添付し、アプリケーションの署名付きバージョンを収集する機会を得ます。 これがどのように実装されているかのgradleスクリプトの例を次に示します。 



 android { ... signingConfigs { if (rootProject.file("release.properties").exists() || new File("/home/ubuntu/release.properties").exists()) { def properties = new Properties() def fileToLoad = rootProject.file("release.properties").exists() ? new FileInputStream(rootProject.file("release.properties")) : new FileInputStream("/home/username/release.properties") properties.load(fileToLoad) release { keyAlias properties.keyAlias keyPassword properties.keyAliasPassword storePassword properties.keyStorePassword storeFile rootProject.file('keystore.jks') } } } ... buildTypes { release { if (signingConfigs.hasProperty("release")) signingConfig signingConfigs.release } } }


Fabricにテストビルドを公開し、テスターに​​警告する



Fabricは、アプリケーションのクラッシュレポートを作成するための優れたシステムです。 標準のGoogleアナリティクスを使用してクラッシュを構築する場合、ファブリックまたはファイヤーベースを調べることを強くお勧めします。 Fabricを使用するには、jenkinsプラグインをインストールする必要があります。



ジェンキンの設定（ジェンキンの管理）→プラグインの管理（プラグインの管理）タブ使用可能、ファブリックと入力します。 Fabric Beta Publisherに興味があります。 インストールしたら、タスクの説明に戻って別のタスクを追加します。











Fabric_api_keyとsecret_keyを指定し、テスター用にアップロードするアプリケーションの.apkファイルへの相対パスを指定する必要があります。 これらの値は、個人アカウントで表示できます。 設定→組織→{Your_app_name}→API_KEY、Build Secret。



さらに、テスターに​​通知を送信できます。 電子メールとテスターのグループの両方を指定できます。 グループを管理する方法は、あなた自身が最高のことをよりよく認識するでしょう。 たとえば、dev、productionの2つのグループを使用することを好みます。 すべてのテストアプリケーションビルドはdevテスターグループに通知し、Google Playにアップロードされるビルドはプロダクションテスターグループに通知します。



さらに、アセンブリが互いに何らかの形で異なる場合、テスター向けのビルドノートでこの情報を指定する必要があります。 これは、環境変数と、この情報を入力するスクリプトを使用して行います。



私たちのプロジェクトは、次のアプローチに従います。 リリース時刻になると、delivery_xxブランチがmasterブランチから作成されます。xxはスプリント番号です。 残念ながら、ジェンキンスはデフォルトでブランチからアセンブリをアセンブルする方法を知りません。ブランチ名は変更できます。 これを自分で教えなければなりません。

これを行うには、別のプラグインEnvInjectをインストールする必要があります。 その後、タスクビューに戻り、新しいオプションを設定します-実行のための環境を準備します。 このプラグインのおかげで、groovyスクリプトを実行し、必要な結果をすべて環境変数に保存できます。



この場合、リポジトリからブランチのリストを取得し、名前がdeliveryで始まるブランチを見つける必要があります。 また、アプリケーションがどのブランチからビルドされたかをテスターが知るために、テスター用のビルドノートをインストールする必要があります。 このスクリプトは次のようになります。 



 def gitURL = "git@github.com:[your_repo_url]" def command = "git ls-remote -h $gitURL" def proc = command.execute() proc.waitFor() if (proc.exitValue() != 0) { println "Error ${proc.err.text}" } def branch = proc.in.text.readLines() .collect { it.replaceAll(/[a-z0-9]*\trefs\/heads\//, '')} .find { it.startsWith("delivery")} if (branch == null) { def build = Thread.currentThread().executable build.doStop() return 0 } def map = [BRANCH: branch, FABRIC_RELEASE_NOTES: "$branch"]


間違いなく、ここで特定のケースに必要なロジックを追加できます。

次に、アセンブリが実行されるブランチの名前を修正します。







そして、上のスクリーンショットに示すように、ファブリックのアプリケーション公開ブロックを更新します。

これで、記事の冒頭で説明したほぼすべてのポイントを実行でき、Google Playでアプリケーションの最終アセンブリを公開する方法を学習する必要があります。



Google Playに公開



Jenkinsを使用してGoogle Playでアプリケーションを公開するためのプラグインをすでに準備しています。 入れてください： Google Play Android Publisherプラグイン 。 次に、Google Playでアプリケーションを公開する特別なサービスアカウントを作成する必要があります。 これを行うには、次の手順を実行します。



  1. アカウント所有者の下で開発者コンソールにアクセスします。
  2. 設定→APIへのアクセス。
  3. [プロジェクトの作成]をクリックします。
  4. [アプリケーションアカウントの作成]をクリックします。
  5. リンクをたどってください。
  6. ドロップダウンリストで[編集]をクリックし、アカウント名を必要な名前（たとえば、「Jenkins」）に変更します。
  7. ドロップダウンリストで、「キーを作成」を選択し、キーの種類「JSON」を選択します。
  8. クリックしてキーを作成します。
  9. ファイルは自動的にダウンロードされ、後でアプリケーションを公開するためにプラグインのデベロッパーコンソールで認証に使用されます。

次に、作成したサービスアカウントに対して開発者コンソールで適切なアクセスを発行し、アプリケーションを公開する必要があります。 これを行うには、次の手順を実行します。



  1. 開発者コンソールに戻る
  2. 作成したアプリケーションアカウントについては、「共有」をクリックします
  3. 作成したアプリケーションアカウントについては、「共有」をクリックします
  4. アプリケーション情報を変更し、アプリケーションの製品版、アルファ版、およびベータ版を管理する権限があることを確認する必要があります
  5. 追加をクリックします


写真の同じもの：











これで、開発者コンソールを終了し、Jenkinsの構成に進んでアプリケーションを公開できます。



  1. ジェンキンスに行く
  2. メニュー項目「資格情報」を選択します
  3. 共有アクセスを選択し、「資格情報の追加」をクリックします
  4. アクセスのタイプを選択します：「プライベートキーからのGoogleサービスアカウント」
  5. android-publishなどのアクセス名を入力します（タスク設定で後で使用されます）
  6. タイプ「JSONキー」を選択します
  7. 以前に「JSONキー」をダウンロードしたファイルをダウンロードします
  8. [OK]をクリックしてアクセスを作成します


写真でも同じこと：











次に、タスクビューに移動し、アセンブリ後のステップ-Android APKをGoogle Playにアップロードします。

前に追加したこのステップの資格情報を指定する必要があります。 apkファイルと情報へのパスに加えて、このバージョンに転送する必要があるユーザーの割合を指定します。 これは、最初は100％になる可能性があります。 また、アップロードする.apkファイルのバージョンを示す必要があります。 jenkinsを介してアルファ版、ベータ版、製品版を公開できます。 その後、タスクを安全に実行し、ジェンキンスがすべての日常的な作業を行って、アプリケーションを組み立て、署名し、それを公開するまで、自分でお茶を飲むことができます。







重要なタスクを保護する



これで、タスクを作成でき、テスターに​​アクセスできるようになり、テスター自身が必要なときにアプリケーションのアセンブリを構築できます。 当然、QAが誤ってアプリケーションを公開することは望ましくありません。 次のように進めることを提案します。

アプリケーションの公開を別のタスクに移動し、QAからアクセスしてこのタスクを起動する必要があります。



  1. Jenkins設定→グローバルセキュリティの設定に移動します
  2. 「認可」をブロック→「プロジェクトベースのマトリックス認可戦略」を選択
  3. すぐにユーザー名を追加し、利用可能なすべてのアイテムを公開します。 この点を説明しないと、ジェンキンからのアクセスを拒否することになります（それでも解決しない場合は、 こちらの解決策を参照してください ）。
  4. 次に、QAユーザー名を追加し、必要と思われる権限を設定します。 彼にタスクを実行させないことが重要です。 ブロック「タスク」、アイテム「ビルド」
  5. その後、QAへのアクセスを許可するタスクに進み、適切なアクセスを許可します。






これで、QAが誤ってアプリケーションをGoogle Playに公開することができなくなります。



おそらくそれだけです。 アプリケーションの自動アセンブリと公開をさらに改善することもできますが、これはこのチュートリアルの範囲外です。
