1. GridLayoutはlayout_weightに応答しません



状況


時々、束を持ったオブジェクトを作成する通常の方法が適切でないことがあります。

正規形




このような状況は、たとえば、フォームの横表示です。 この場合、次のようなものが欲しいです。

横向きのフォーム




50から50の同じアライメントを作成する方法は？ いくつかの基本的なアプローチがあります。

  • ネストされたLinearLayout
  • RelativeLayout
  • TableLayoutまたはGridLayout


ただし、それらにはすべて欠点があります。

  • LinearLayoutの豊富さはxml'kiの巨大さをもたらし、猫の死をもたらします。
  • RelativeLayoutは、将来の変更を複雑にします（フォーム内の複数の行のスワップまたはセパレーターの追加は同じタスクになります。View.setVisibility（View.GONE ）については完全に黙っています。
  • まあ、誰もTableLayoutを使用しません...または彼らはそれを使用しますが、めったにありません。 私はそのような人を知りません


さらに、柔軟な設計を実現するために、 0dpとweight = 1の数値のマジックを使用する必要が非常に頻繁にあります。 ここでは、 TableLayoutもRelativeLayoutも役立ちません。 TextView.setEllipsize（）のようなものを初めて使用しようとすると、問題と痛みが始まります。

そして、おそらくあなたは私が別の要素を見逃していることに気づいたでしょう。 GridLayoutが助けになるように思えますが、 layout_weightプロパティをサポートしていないため、 それでも役に立たないでしょう。 それではどうしますか？



解決策


しばらくの間、何もする必要はありませんでした-RelativeLayoutで苦しめるか、 LinearLayoutを使用するか 、プログラムですべてを埋めます（特に異常な場合）。

ただし、バージョン21では、 GridLayoutがやっとlayout_weightプロパティのサポートを開始し、最も重要なことには、この変更がandroid.support.v7.widget.GridLayoutとしてAppCompatに追加されました！

これについて（そして一般的には通常のGridLayoutが自分の体重を噛みたかったという事実について）知るまでに、 レイアウトが右に泳いだ理由を理解しようとして（ ここのように ）少なくとも1週間を費やしました。 おそらく、これは最も重要な革新の1つであり、何らかの理由で十分な注意を払わずに残されたものです。 幸いなことに、stackoverflow（ 1、2 ）に対する回答はすでに追加され始めています。

また、新しいPercentRelativeLayoutとPercentFrameLayoutのページを確認することをお勧めします-これは本当に爆弾です。 その名前はそれ自体を表しており、非常にレスポンシブなデザインを作ることができます。 iOSは感謝します。 そして、はい、それはAppCompatにあります。



2. Fragment.isRemoving（）とAcitivity.isFinishing（）は等しいですか？



状況


PresenterManagerをシングルトン（ MVPからの挨拶）として記述したいと思ったことがあります。 Presenterを時間内に削除するには、 Activity.isFinishing（）を使用して、アクティビティ内のid Presenterのフラグメントを収集し、それとともに削除します。 当然、このメソッドはNavigationViewの場合はうまく機能しませんでした-フラグメントはFragmentTransaction.replace（）によって変更され、 Presenterが蓄積され、すべてが流出しました。

ラードをグーグルで検索すると、 Fragment.isRemoving（）メソッドが見つかりました。これは同じことですが、フラグメントに対して行われます。 PresenterManagerのコードを書き直して喜んでいます。 終わり...



解決策


...私がそれを機能させようとしたとき、私の静かな生活が始まりました。 正直なところ、私はこの方法とそれを試しましたが、このメソッドの動作はActivity.isFinishing（）とは根本的に異なります。 Googleは間違っていました。 同様の問題が発生した場合は、 Fragment.isRemoving（）を使用する前に3回考えてください。 私は本気です。 画面を回転させるときは、ログに特に注意してください。



ちなみに、 Acitivity.isFinishing（）では、すべてがそれほどスムーズではありません：スタック内の1アクティビティ以上でアプリケーションを最小化し、メモリ不足を待ち、戻ってUp Navigationと* voila *！..を使用します。これは、 アクティビティを作成する簡単なレシピでした。 isFinishing（）== falseは、二度と表示されないアクティビティの場合。

UPD ： bejibxユーザーが正しく指摘しているように、 Activity.isFinishing（）に関するコメントは完全に真実ではありません。 理由を理解するには、コメントスレッドを読むことをお勧めします 。



3. RecyclerViewの ヘッダー/フッター



状況


ページネーションを実装する際の一般的なタスクは、新しいデータのロード中にProgressBarを表示することです。

ListViewとは異なり、 RecyclerViewにははるかに多くの機能があります。ListViewの 頭痛の種と比較して、 RecyclerView.Adapter.notifyItemRangeInserted（）の価値は何ですか 。

ただし、プロジェクトでListViewの代わりに使用しようとすると、すぐに多くのニュアンスに遭遇します。プロパティListView.setDivider（）はどこにありますか？ ListView.addHeaderView（）のようなものはどこですか？ 他に何がRecyclerView.Adapter.getItemViewType（）など、など。

この新しい情報のすべてを処理することは難しくありませんが、不快なことが残っています。 Divider / Headerを追加するには、コードのクラウドを書く必要があります。 複雑なレイアウトについては何と言えますか？ 鳩は、4つの異なるヘッダーとフッターとコントロールを備えたRecyclerViewを実行する必要がありました。 悲しくて落胆したと言ってみましょう、私は非常に長い間歩いた。



解決策


実際、何を探すべきかを知っていれば、すべてがそれほど悪くはありません。 RecyclerViewの主な問題（およびその主な利点）は、 RecyclerViewで何でもできることです。 実質的にスコープはありません。 ここから問題が続きます。 ヘッダーが必要な場合は、自分で実行してください。 しかし、幸いなことに、他の人がすでにやってくれた「自分でやる」ので、それを使いましょう。

典型的な問題とその解決策：

  • 要素のグループの見出し（たとえば、辞書「A」では、この文字で始まるすべての単語の見出しになります）-これを行う最も簡単な方法は、2番目の不要なViewHolderタイプを追加せずに、単一のitem-layoutを使用することです。 現在の要素が、ある文字から別の文字への遷移をマークし、 View.VISIBLEを介してレイアウトに隠されたタイトルを含めることを確認するチェックを追加します 。
  • シンプルな仕切り - このコードをコピーしてプロジェクトに貼り付けます 。 不要な詐欺はありません。 RecyclerView.addItemDecoration（）を介して動作します
  • ヘッダー/フッター/ドラッグアンドドロップなどを追加します。 -ペンを使用する場合は、新しいViewHolderごとに新しいタイプを開始する（お勧めしません）か、 WrapperAdapterを実行します （はるかに優れています）。 しかし、 ここを見て 、お気に入りのライブラリを選択することをお勧めします。 個人的には、 FastAdapterとUltimateRecyclerViewの 2つが同時に好きです
  • ページネーションが必要ですが、 プログレス バーの ヘッダー/フッターを台無しにするのは面倒です-Twitter開発者の1人によるページネーションライブラリです。


私にとってはまだ謎のままです-SimpleDivider / SimpleHeaderAdapterなどを作成することが不可能だった理由 SDKに直接？

6. Gson ： EnumSetとしてのビットマスク



いつ/どこで/なぜ？


（状況は特定のものであり、Androidの問題には直接適用されませんが、次の溝の前に種として追加することにしました）

API要求の1つに応答して、 intの形式のアクセス許可のビットマスクが入ります。 このマスクの要素を処理する必要があります。

最初に頭に浮かぶのは、 intの定数とチェック用のビット演算です。 もちろん、常に機能します。 しかし、もっと欲しい場合はどうしますか？ EnumSetはどうですか？

「問題ありません」-ひげを生やした男が答えて、モデルのアーキテクチャをPOJO、Model、Entity、UiModel 、そして冗談ではないレベルに分割します。 しかし、もしあなたが怠け者で、余計なことをしたくないなら。 クラス？ それで何？



解決策


@SerializedNameの 「ビットのような」名前に注意して、必要な列挙型を作成します。

列挙型アクセス 
public enum Access { @SerializedName("1") CREATE, @SerializedName("2") READ; @SerializedName("4") UPDATE; @SerializedName("8") DELETE; }




jsonからEnumSetへの逆シリアル化用のJsonDeserializerを定義します 。

EnumMaskConverter 
 public class EnumMaskConverter<E extends Enum<E>> implements JsonDeserializer<EnumSet<E>> { Class<E> enumClass; public EnumMaskConverter(Class<E> enumClass) { this.enumClass = enumClass; } @Override public EnumSet<E> deserialize(JsonElement json, Type typeOfT, JsonDeserializationContext context) throws JsonParseException { long mask = json.getAsLong(); EnumSet<E> set = EnumSet.noneOf(enumClass); for (E bit : enumClass.getEnumConstants()) { final String value = EnumUtils.GetSerializedNameValue(bit); assert value != null; long key = Integer.valueOf(value); if ((mask & key) != 0) { set.add(bit); } } return set; } }




そしてGsonに追加します ：

Gsonbuilder 
 GsonBuilder gsonBuilder = new GsonBuilder(); gsonBuilder.registerTypeAdapter((new TypeToken<EnumSet<Access>>() {}).getType(), new EnumMaskConverter<>(Access.class)); Gson gson = gsonBuilder.create();




その結果：

使用する 
 class MyModel { @SerializedName("mask") public EnumSet<Access> access; } /* ...some lines later... */ if (myModel.access.containsAll(EnumSet.of(Access.READ, Access.UPDATE, Access.DELETE))) { /* do something really cool */ }

状況

データは次のようになります。

列挙シーズン

Gsonが通常の@SerializedNameを介して列挙型を解析する機能のおかげで、あらゆる種類の追加レイヤークラスを作成する必要性を取り除くことが可能になりました。 すべてのデータは、要求からすぐにModelに送られます。 すべて順調です：

改造サービス 
 public interface MonthApi { @GET("index.php?page[api]=selectors") Observable<MonthSelector> getPriorityMonthSelector(); @GET("index.php?page[api]=years") Observable<Month> getFirstMonth(@Query("season") Season season); }




申込み 
 class MonthSelector { @SerializedName("season") public Season season; } /* ...some mouses later... */ MonthSelector selector = monthApi.getPriorityMonthSelector(); Season season = selector.season; /* ...some cats later... */ Month month = monthApi.getFirstMonth(season);




そして今、親愛なる専門家、注意は問題です！ 何がうまくいかず、なぜ機能しないのですか？



解決策


ここでは、これが機能していないという情報を特に省略しました。 実際、ログを見ると、 monthApi.getFirstMonth（season）リクエストはindex.phpとして処理されますか？ページ[api] = years＆season_lookup = AUTUMN ...「ええと、何をしていますか？」-私は言います。 あなたの答えは何ですか？ なぜそのような結果ですか？ まだ推測していませんか？ その後、ヒットします。

このタスクに出くわしたとき、1つのことを理解するためにソースで検索するのに数時間かかりました（あるいは覚えていることさえあります）：はい、 Gsonは@GET / @POSTおよび他の同様の_リクエスト_を送信するときに使用されません！ 実際、最後にindex.phpのようなものを見たのはいつですか？ページ[api] = years＆season_lookup = {a：123; b：321} ？ 意味がありません Retrofit 2はBodyの変換時にのみGsonを使用しますが、リクエスト自体には使用しません。 最後に？ ちょうどseason.toString（）が使用されます-したがって、結果です。

ただし、リクエストでGsonを介した変換でenum を使用したい場合（そして私もその1人です） 、 ここに別のコンバーターがあり、すべてがいつものようになります。
