QEMUハイパーバイザーの下でVMWare ESXi 6.5を起動します





世界には素晴らしいVMWare ESXiハイパーバイザーがあり、すべてが順調ですが、動作可能なハードウェアの要件は非常に控えめです。 ESXiは基本的にソフトウェアRAID、100メガバイト、安価なギガビットネットワークカードをサポートしていません。そのため、適切な機器を購入することでしか動作しません。

ただし、クラスタリング、ライブマイグレーション、分散VSAN、分散ネットワークスイッチなど、複数のESXiホストがある場合、ESXiの最も「おいしい」ESXi機能が開きます。 この場合、テスト機器のコストはすでにかなりの量になる可能性があります。 幸いなことに、ESXiはネストされた仮想化、つまり、既に実行中のハイパーバイザーの下から実行する機能をサポートしています。 同時に、外部ハイパーバイザーはゲストがハードウェア仮想化にアクセスする必要があることを理解しており、ESXiはベアメタルでは動作しないことを知っています。 原則として、ESXiはメインハイパーバイザーとしても使用されます。この構成は、かなり前からVMWareでサポートされています。 QEMUハイパーバイザーを使用して、ESXiを起動しようとします。 ネットワークにはこれに関する指示がありますが、以下で見るように、それらは少し時代遅れです。



まず、実験を行うQEMUのバージョンを示します。 



user@debian-pc:~$ QEMU emulator version 2.8.0(Debian 1:2.8+dfsg-2) Copyright (c) 2003-2016 Fabrice Bellard and the QEMU Project developers


現時点では最新バージョンですが、私の焦点は2.4.0でさえありました。

次に、ゲストが実際には存在しないマシン固有のレジスタを読み取ろうとする瞬間に、KVMモジュールの無礼な動作を無効にします。 デフォルトでは、これに応じてKVMはゲスト内で一般保護違反例外を生成します。これが、ゲストが青色（この場合はピンク色）のデススクリーンに表示される理由です。 ルートの下でやってみましょう： 



 root@debian-pc:/> echo 1 > /sys/module/kvm/parameters/ignore_msrs


一部のディストリビューションでは、kvmモジュールがデフォルトで必要なパラメーターとともにロードされますが、一部ではそうではありません。 いずれにせよ、dmesgの文字列を確認する必要があります 



 user@debian-pc:~$ dmesg | grep kvm [ 6.266942] kvm: Nested Virtualization enabled [ 6.266947] kvm: Nested Paging enabled


これらの行が存在しない場合は、/ etc / modprobe.d / kvm.confに行を追加します 



 options kvm-amd npt=1 nested=1


そして再起動します。 Intelプロセッサーの場合、行は次の形式を取ります。 



 options kvm-intel ept=1 nested=1


興味深いことに、dmesgで有効になっているネストされたページング/ネストされた仮想化についてはkvm-amdのみがレポートしますが、kvm-intelはレポートしません。



「額」の問題を解決してみましょう。VMWareWebサイトにアクセスして登録し、VMware-VMvisor-Installer-201701001-4887370.x86_64.isoの最新イメージをダウンロードしてください。



そのような構成では、ESXiが少なくともIDEを認識しないことを想定して、心構えず、16Gbで「フラッシュドライブ」の類似物を作成し、サポートされているe1000ネットワークカードを確保し、4GbにRAMを入れて（ESXiメモリーが少なくなり、起動しないことが保証されます）、インストールを実行しましょう-ドライブ： 



 user@debian-pc:~$ qemu-img create -f qcow2 -o nocow=on /media/storage/VMs/esx_6.5-1.qcow2 16G Formatting '/media/storage/VMs/esx_6.5-1.qcow2', fmt=qcow2 size=17179869184 encryption=off cluster_size=65536 lazy_refcounts=off refcount_bits=16 nocow=on

そして、ここで最初の驚きに直面しています-ESXiはIDEドライブを検出するだけでなく、正常にインストールすることもできますが、インストールの27％で5分間ハングします。

さて、プロセッサは明確です-ホストプロセッサのCPUIDをゲストにコピーする-cpuホストオプションを使用しました。所有しているホストプロセッサは、故人のFM1ソケットのAMD A8-3850 APUです。 ESXiが一般的にそのようなハードウェアに配置されているのは奇妙です。

ただし、すべてがESXiで同等に機能するわけではありません。たとえば、ユーザーモードネットワークでのポート転送は正式にサポートされているe1000eでは機能せず、パケットの半分はvmxnet3で消えます。 それでは、e1000で停止しましょう。



VMを再起動すると、ハイパーバイザーが正常に起動したことがわかります。 実際、これですべてです。一部のマニュアルで推奨されているように、ESXiのQEMUにパッチを適用する必要はありません。



VMディスクはbtrfs上にあるため、nocow = onパラメーターを使用することに注意してください。btrfsは、コピーオンライトの概念を持つファイルシステムです。 これに、シンプロビジョニングされたqcow2形式のディスクもこの原則を実装しているという事実を追加すると、ディスクへの書き込み回数が倍増します。 nocow = onパラメーターを指定すると、qemu-imgはnocow属性を持つファイルを作成し、特定のファイルのbtrfsのコピーオンライトメカニズムをブロックします。



ユーザーネットワークモードでは、軽量のDHCPサーバーがVM内で実行されるため、アドレスを割り当てる必要はありませんが、ポートを転送する必要があります。 Ctrl + Alt + 1を押してQEMUコンソールに移動し、そこにコマンドを入力します 



 > hostfwd_add tcp::4443-:443


ポート443を仮想マシンのネットワークインターフェイスからホストのポート4443に転送します。 次に、ブラウザで次のように入力します



https：// localhost：4443 / ui



セキュリティ例外（もちろん、httpsに自己署名証明書を使用している間にESXi）を確認し、ハイパーバイザーのWebインターフェイスを見てみましょう。







驚くべきことに、ESXiインストーラーは8Gbもの空きディスク領域に「ストレージ」を作成しました。 まず、この便利なコンポーネントの開発はESXiの新しいバージョンよりも速いため、更新されたWebインターフェースを備えたパッケージをインストールします。 Ctrl + Alt + 1でQEMUコンソールに移動し、そこでポート22を転送します。 



 > hostfwd_add tcp::2222-:22


Ctrl + Alt + 2でハイパーバイザーコンソールに切り替え、F2を押します-トラブルシューティングオプション-SSHを有効にしてSSHクライアントに接続します。 



 user@debian-pc:~$ ssh root@127.0.0.1 -p 2222 Password: The time and date of this login have been sent to the system logs. VMware offers supported, powerful system administration tools. Please see www.vmware.com/go/sysadmintools for details. The ESXi Shell can be disabled by an administrative user. See the vSphere Security documentation for more information.

ご覧のとおり、Webインターフェースのサイズは3メガバイトをわずかに超えています。



次に、仮想マシンの改善を試みます。 まず、ディスクコントローラーをIDEからAHCIに変更します。これは、QEMUでの1996 PIIX3コントローラーの実装が、いまだに禁止されているためです。 そして、AHCIコントローラ（Intel ICH9チップセットがエミュレートされます）は、第一に高速で、第二にNCQコマンドキューをサポートします。 



 user@debian-pc:~$ qemu-system-x86_64 --enable-kvm -cpu host -smp 2 -m 4096 -device ich9-ahci,id=ahci -drive file=/media/storage/VMs/esxi_6.5-1.qcow2,if=none,id=drive0 -device ide-drive,drive=drive0,bus=ahci.0 -netdev user,id=hostnet0 -device e1000,netdev=hostnet0,id=net0


ハイパーバイザーのコンポーネントのロード時間を短縮しても、速度が向上したことは明らかです。 喜びのために、私たちはWebインターフェースに入り、...ディスクがないのはどうですか？ 「アダプター」タブにAHCIコントローラーがありますが、ディスクは検出されません。 しかし、ハイパーバイザーはどのようにロードしましたか？ 非常にシンプル-初期段階では、ブートローダーはBIOSを使用してディスクデータを読み取り、ディスクを直接確認する必要はありません。 コンポーネントがメモリにロードされた後、ローダーはそれらに制御を移し、ディスクにアクセスせずにハイパーバイザーの初期化が行われます。







場合によっては、ESXi 6.5はAHCIコントローラー上のディスクを認識しませんが、ESXi 6.0はこれらのディスクを認識しました。 Googleとそのような母親を使用して、 理由を見つけます：ESXi 6.5では、古いahciドライバーが完全に書き直されたvmw_ahciドライバーに置き換えられます。そのため、SSDによって多くの人が遅くなり、ドライブがなくなります。 記事のアドバイスによると、ハイパーバイザーで行います 



 [root@localhost:~] esxcli system module set --enabled=false --module=vmw_ahci


再起動して...何も起こりません。 何が欲しかった？ ディスクがなく、構成を記録する場所がないため、変更は保存されませんでした。 IDEディスクに戻り、このコマンドを実行してからAHCIから起動する必要があります-その後、ディスクが検出されます。



ところで、Webベースのインターフェイスにアクセスすると、インストーラーによって作成された8ギガバイトの「ストレージ」が利用できなくなっていることがわかります。 そのため、さまざまなタイプのコントローラーに対して、VMWareにはさまざまなストレージ定義ポリシーがあります。 次に、ESXiがUSBフラッシュドライブにインストールされ、ストレージがハードドライブにある場合のシステムの実際の構成を表示してみましょう。 USB 3.0エミュレーションを使用します。 



 user@debian-pc:~$ qemu-system-x86_64 --enable-kvm -cpu host -smp 2 -m 4096 -device nec-usb-xhci,id=xhci -drive file=/media/storage/VMs/esxi_6.5-1.qcow2,if=none,id=drive0 -device usb-storage,drive=drive0,bus=xhci.0 -netdev user,id=hostnet0, -device e1000,netdev=hostnet0,id=net0


USB 3.0ドライブも検出されません。 どうやら、ドライバーもここで書き換えられます。 まあ、私たちはすでに何をすべきか知っています。 ハイパーバイザーのコンソールに行き、そこに書きます 



  esxcli system module set -m=vmkusb -e=FALSE


システムが起動したら、ストレージ-デバイスに移動し、フラッシュドライブを確認します。 ところで、USB 3.0コントローラーnec-usb-xhciを使用すると、システムはich9-usb-ehci2を使用した場合よりもはるかに高速に起動します。







そのため、ESXi 6.5では、ESXi 6.0と比較して、少なくとも2つのディスクコントローラードライバーが再度上書きされます。 しかし、バージョン番号のドットの後の数字だけが変更されたように見えるでしょう。つまり、マイナーリリースです。



仮想マシンの構成に1 Tbディスクを追加すると、ハイパーバイザーを備えたディスクに加えて、本格的なストレージを作成できます。 ahciではなくusbドライブからシステムを起動するには、bootindexパラメーターを使用します。 通常、-bootパラメーターはブート順序を制御するために使用されますが、この場合は、ディスクが異なるコントローラーで「ハング」するため、役に立ちません。 同時に、プラットフォームを古い440fxチップセットから新しいQ35 / ICH9に置き換えます。 



 user@debian-pc:~$ qemu-img create -f qcow2 -o nocow=on /media/storage/VMs/esxi_6.5-1.qcow2 1T user@debian-pc:~$ qemu-system-x86_64 -machine q35 --enable-kvm -cpu host -smp 2 -m 4096 -device nec-usb-xhci,id=xhci -drive file=/media/storage/VMs/esxi_6.5-1.qcow2,if=none,id=drive0 -device usb-storage,drive=drive0,bus=xhci.0,bootindex=1 -device ich9-ahci,id=ahci -drive file=/media/storage/VMs/esxi_6.5-1-1T.qcow2,if=none,id=drive1 -device ide-drive,drive=drive1,bus=ahci.0,bootindex=2 -netdev user,id=hostnet0 -device e1000,netdev=hostnet0,id=net0


コンソールに入ります-ここに、ドライブがあります。







実験を続けましょう。ここで、いくつかのハイパーバイザーをネットワークに接続する必要があります。 一部のlibvirtは独立して仮想スイッチを作成し、それにマシンを接続します。これらの操作を手動で実行しようとします。



2つの仮想マシンがあるので、2つの仮想アダプターが必要です。 



 user@debian-pc:~$ sudo ip tuntap add mode tap tap0 user@debian-pc:~$ sudo ip tuntap add mode tap tap1


次に、仮想スイッチが必要です。 長い間、これらの目的のためにbrctlユーティリティによって制御されるLinuxカーネルに組み込まれた仮想スイッチを使用することが慣習でした。 現在、Open vSwitch-仮想環境向けに特別に設計されたスイッチの実装を介して問題を解決するのが慣例となっています。 Open vSwitchには、VLAN、いくつかのスイッチ、そして最も興味深いことにOpenFlowテクノロジーを組み合わせるためのトンネリングプロトコル（GREなど）が組み込まれています。 つまり、L2 / L3フィルタリングルールを読み取り可能な形式でスイッチにロードできます。 以前は、フィルタリングにはiptables / ebtablesの使用が必要でしたが、彼らが言うように、良いことは「ebtables」が呼び出されないことです。



Open vSwitchがまだインストールされていない場合はインストールします。 



 user@debian-pc:~$ sudo aptitude install openvswitch-switch


仮想スイッチを作成します。 



 user@debian-pc:~$ sudo ovs-vsctl add-br ovs-bridge


インターフェイスを追加します。 



 user@debian-pc:~$ sudo sudo ovs-vsctl add-port ovs-bridge tap0 user@debian-pc:~$ sudo sudo ovs-vsctl add-port ovs-bridge tap1


何が起こったのか見てみましょう： 



 user@debian-pc:~$ sudo ovs-vsctl show e4397bbd-0a73-4c0b-8007-12872cf132d9 Bridge ovs-bridge Port "tap1" Interface "tap1" Port ovs-bridge Interface ovs-bridge type: internal Port "tap0" Interface "tap0" ovs_version: "2.6.2"


インターフェイスを実行します。 



 user@debian-pc:~$ sudo ip link set tap0 up user@debian-pc:~$ sudo ip link set tap1 up user@debian-pc:~$ sudo ip link set ovs-bridge up


次のアドレスをスイッチインターフェイスに割り当てます。 



 user@debian-pc:~$ sudo ip addr add 192.168.101.1 dev ovs-bridge


QEMUコマンドラインでネットワークタイプをユーザーからタップに変更するだけで、次のようになります。 



 -netdev tap,ifname=tap,script=no,downscripot=no,id=hostnet0 -device e1000,netdev=hostnet0,id=net0


すべてが機能します。

ここで、ホストから最初のゲストへのpingは、2番目のゲストへのpingを開始するまでしか行われません。 2番目のpingコマンドが中断された後、最初のゲストへのパケットは10秒後に送信を開始しますが、ゲスト間のpingはまったく送信されません。



Macアドレスを台無しにしたことは明らかです。実際、QEMUは、特に指定がない限り、すべてのタップアダプターに同じMACアドレスを割り当てます。



両方のESXiの電源を切り、一意のMacを表示して、再度起動します。

 esxcfg-vmknic -a -i DHCP -p pgName

 esxcfg-vmknic -a -i xxxx -n 255.255.255.0 pgName

 esxcfg-advcfg -s 1 /Net/FollowHardwareMac
