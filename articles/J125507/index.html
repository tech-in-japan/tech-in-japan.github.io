.NETランタイムでのオブジェクトの配置に関する真実の一部

ご存じのとおり、.NETメモリでは、スタック（スタック）とマネージヒープ（マネージヒープ、次にヒープ）の2つのカテゴリに分類されます。 スタック上には、オブジェクト（ObjectInstance）へのリンク（ObjectRef）があり、これはヒープ上にあります。



この記事では、ヒープ上のオブジェクトの場所に焦点を当てます。



読者は次の知識を持っていると想定されます。

1.スタック

2.マネージヒープ

3. GC

4.弱い参照





ObjectInstance


ObjectInstance -ObjectRefが指すアドレスにあるデータ構造（オブジェクト）。

重要なタイプのインスタンスが保存される場合があります。

1.スレッドスタック内

2. GCヒープ内。

サイズに応じて、参照タイプのインスタンスを保存できます。

1.ガベージコレクターヒープ（サイズが最大85000バイトのオブジェクト）

2.ラージオブジェクトヒープ（85,000バイトを超えるオブジェクト）

オブジェクトヘッダー


図1をもう一度見てみましょう。ご覧のように、ObjectRefは、ObjectInstanceの先頭から4バイトのアドレスオフセットを指します。 これは最適化のために行われます-オブジェクトヘッダーには、オブジェクトインスタンスの残りのフィールドほど頻繁には使用されない情報が含まれます。

このフィールドには、同期ブロックエントリテーブルのセル番号が含まれます。 数は1からカウントされます。 ロックステートメントまたはGetHashCode（）メソッドがObjectInstanceに使用されたことがない場合、オブジェクトヘッダーセルに0が格納されます。

同期ブロックエントリテーブルとObjectInstanceはインデックスによってリンクされているため、CLRはメモリ内でこのテーブルのサイズを自由に変更できます。

インデックスがオブジェクトヘッダーに含まれる同期ブロックエントリテーブルのセルには、同じObjectInstanceへの弱いリンクが含まれます。これは、CLRがこのセルの所有者を追跡できるようにするためです（たとえば、スレッドを同期するため）。

また、このセルには、SyncBlock List構造内のSyncBlockオブジェクトへの参照が含まれています。 SyncBlockには、有用ではあるがめったに使用されない情報が含まれています。

1.サンク情報

2. AppDomainインデックス

3.オブジェクトロックデータ

4.ハッシュコード

図2. SimpleClassオブジェクトインスタンスの[メモリ]ウィンドウ



図 図2は、SimpleClassオブジェクトのインスタンスがメモリの内容を表示するウィンドウでどのように見えるかを示しています。

デバッガメモリウィンドウを開く前に（デバッグ-> Windows->メモリ-> Memory1）、returnステートメントにブレークポイントが設定されていました。 次に、ECXレジスタに含まれるsimpleObjectアドレスで、メモリダンプを表示する必要があるオブジェクトのインスタンスのアドレスを決定しました。

1.最初の4バイトブロックはsyncblk番号です。 このオブジェクトの同期コードを記述しなかったため（そしてそのハッシュコードにアクセスしなかったため）、この数は0です。

2.スタックに配置されたsimpleObject変数に格納されたオブジェクトへの参照は、4バイトを示し、その先頭はオフセット4です。

3.次に、変数l1の8バイト値のメモリ内への配置です。

4. String型のstr変数は、GCヒープでホストされる文字列のインスタンスを指す4バイトのObjectRef参照によって表されます。

5.次に、int型の変数i1が8バイトを占有して配置されます。

6. char型の2つの変数-c1およびc2-が並んで配置されています。

7.短いタイプの2つの変数-s1およびs2-も「並んで」配置されます。

8.タイプByteの変数b1、b2、b3、b4はDWORDにパックされ、「サイドバイサイド」に配置されます。

デフォルトでは、オブジェクトのメンバーは、ソースコードで宣言されている順序で必ずしもメモリに配置されるとは限りません。

メモリにフィールドを配置するときに字句シーケンスを観察するには、StructLayoutAttribute属性を使用する必要があります。 この属性は、次の値を持つLayoutKind列挙を入力値として受け入れます。

1.シーケンシャル-オブジェクトのメンバーは、アンマネージメモリにエクスポートされたときの外観の順序で順番に配置されます。

2.明示的-管理されていないメモリ内のオブジェクトの各メンバーの正確な位置は明示的に制御されます。 各メンバーは、FieldOffsetAttribute属性を使用して、タイプ内のこのフィールドの位置を示す必要があります。

3.自動-CLRは、アンマネージメモリ内のオブジェクトのメンバーに適切な場所を自動的に選択します。 この列挙メンバーを使用して定義されたオブジェクトへのアクセスは、マネージコードの外部では許可できません。 このような操作を実行しようとすると、例外がスローされます。



メモリの非構造化コンテンツを調べた後、Son of Strike（SOS）ユーティリティを使用してオブジェクトのインスタンスを調べます。

プロジェクトをアンマネージデバッグモードにし（プロパティ->デバッグ->アンマネージコードデバッグを有効にする）、イミディエイトウィンドウ（ctrl + alt + I）を開き、次のコマンドを実行します。







  1. .load sos
  2. ！DumpHeapタイプのSimpleClass
