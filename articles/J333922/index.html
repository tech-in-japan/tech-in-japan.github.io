多分それは理解できる-全体的なアジャイルと「一度だけの生産」。

この記事で状況を修正しようとします。







短い仕事の説明



この記事では、プラットフォームのプロジェクトのインフラストラクチャソリューションの構成と管理の分野でのソリューションについて説明します。このソリューションは、分析ユニットを除く20の異なるコンポーネントで構成され、合計70の「サーバーユニット」を備えています。







正直に言うと、特にこれらすべてが自動的に制御および構成されている場合、それほど多くはありません。 大規模なシステムについては説明できますが、正直なところ、ほぼ同じです。それは、より多くのインスタンスに広がっているだけで、「企業」は松葉杖のように読みます。

培地の例は、種子用です。







説明したプラットフォームは、SaltStackモジュールによって完全に管理され、SaltStackモジュールによって新しいインスタンスを自然に作成および起動します。 もちろん、各プラットフォームサービスを設定する必要がありますが、システムサービスの設定を忘れないでください。







そのため、たとえば、最も一般的なWebサーバーの構成には、nginx / httpdおよびサイト自体の構成だけでなく、システムサービスも含まれます。







  • ネットワーク構成
  • sshデーモンの構成
  • ホスト認証
  • tcp_wrapper設定
  • iptables
  • selinux / apparmor
  • 時刻同期
  • ロギング
  • 監査


サイトのDNSレコード、証明書の発行と再発行を忘れないでください。

各サービスの構成を個別に説明すると、コピーアンドペーストするだけでも夢中になります。







幸いなことに、SaltStackは、他のほとんどの構成管理システムと同様に、Formulasと呼ばれる再現可能な構成ブロックを備えています。 ただし、式をコピーしたり、リポジトリから直接プルしたりすることは、何らかのタグまたは特別なリリースブランチからであっても、安定化と再現性の観点からは完全に正しくありません-タグを書き換え、ブランチを更新し、チェックおよび検証できます難しすぎます リビジョンに基づいた数式間の対応依存関係のリストを維持する必要があります。 このために、モジュールと依存関係をバージョン管理するプロセスはかなり前に発明されました。 はい、もちろん、リビジョンとともにバージョンの依存関係を追加できますが、VCSはファイルの変更履歴を維持するために開発されたものであり、リリースアーティファクトを修正するためではありません。







顧客の環境で数式を使用する必要がある状況を忘れないでください。顧客の環境から会社のリポジトリサーバーへのアクセスを開くことは、それをやや安全にせず、バイナリアーティファクトへのアクセスを提供する方がはるかに安全です。

SaltStackは、バージョン付きのバイナリパッケージを操作できるコンポーネントを提供します。







このコンポーネントは、 SaltStack Packet Managerまたは略してSPMと呼ばれます。 本質的に、結果のパッケージはtarであり、bzipによってメタ情報を圧縮されており、 FORMULAファイルが責任を負います。 リポジトリを維持するためには、リポジトリ内に保存されたすべてのアーティファクトが存在する場合に生成されるメタ情報ファイルも生成する必要があることに注意してください。原則はYumリポジトリと同じです。 SPMには、この情報を含むメタデータファイルをまとめて、式の依存関係のバージョンを指定するにもかかわらず、リポジトリで使用可能な式を一覧表示できないなど、SPMにいくつかの欠点がありますが、これはSPM自体の若さによるものです。バージョン2016.3でメモリが変更されない場合。

リポジトリでの作業









式がテストされ、タグがリポジトリに配置され、アーティファクトがバイナリリポジトリに埋め込まれていますが、SaltStackではまだ使用できません-メタ情報を更新する必要があります。これには、すべての式を含むリポジトリディレクトリのspm create_repo $DIR



コマンドが必要です。

また、SaltStackを介してNexusを構成するため、SPMはローカルにインストールされます。これを使用します。

Nexusを備えたホストにpython-inotifyモジュールをインストールし、ソルトミニオン用にSaltStack Beaconを設定します。これにより、式でディレクトリの変更を5秒ごとに監視します。 







 #/etc/salt/minion.d/beacons.conf beacons: inotify: interval: 5 disable_during_state_run: True /opt/nexus/data/storage/org.platops.salt.formulas: mask: - create - modify - delete - attrib recurse: True auto_add: True exclude: - /opt/nexus/data/storage/org.platops.salt.formulas/SPM-METADATA - /opt/nexus/data/storage/org.platops.salt.formulas/.nexus/


この段階で、salt-minionを再起動する必要があります。その後、ディレクトリ内のファイルを変更した場合、salt-masterはこれに関するイベントを受け取ります。 







 salt/beacon/nexus.platops.org/inotify//opt/nexus/data/storage/org.platops.salt.formulas { "_stamp": "2017-07-09T10:31:11.283771", "change": "IN_ATTRIB", "id": "nexus.platops.org", "path": "/opt/nexus/data/storage/org.platops.salt.formulas/gitbucket-1.0.0-1.spm"


salt-masterで、構成されたビーコンからのイベントを監視し、構成された状態を起動するリアクターをセットアップします。 







 #/etc/salt/master.d/reactor.conf reactor: - 'salt/beacon/nexus.platops.org/inotify//opt/nexus/data/storage/org.platops.salt.formulas': - /srv/salt/reactor/update-spm-metadata.sls


そして、実際に状態は必要なコマンドを開始します： 







 #/srv/salt/reactor/update-spm-metadata.sls update spm repository metadata: local.cmd.run: - tgt: {{ data['id'] }} - arg: - spm create_repo /opt/nexus/data/storage/org.platops.salt.formulas


salt-master、salt-minion、および実際にはすべてを5秒ごとに再起動することを忘れないでください。ビーコンはディレクトリ内のinotifyを通じてイベントをチェックし、リポジトリメタデータの更新を開始します。

Nexus 3を使用している場合は、これを実装する機会もあります。NexuswebhookとSalt-APIに 注目してください。
