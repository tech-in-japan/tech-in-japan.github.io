ITプロフェッショナルはどのように働いていますか？ マキシム・ゼリンスキー、Sberbank Technologies

私たちは、仕事や余暇の習慣、専門的な習慣、彼らが使用するツールなどについて専門家に質問し続けています。



それらが互いに矛盾しているものを結合するものを見つけることは興味深いでしょう。 おそらく彼らの答えは、私たちの多くを助けるいくつかの一般的なパターン、有用なヒントを特定するのに役立つでしょう。



今日のゲストは、Sberbank-TechnologyのMaxim Zelinskyです。 彼の会社には多くの内部プロジェクトがあります。 マキシムは、天才をマルチステーションと見なさない人に適したライフハックを提供しています。





会社で何をしますか？



統合フロントエンドシステムのタスクのプラットフォームの開発を担当しています。これは、すべてのチャネル（支店、インターネットバンク、モバイルアプリケーション、ATMなど）のすべてのSberbank顧客にサービスを提供するシステムです。 私たちのチームは、高負荷および高可用性アーキテクチャに従事しており、技術ソリューションの選択を担当し、システムおよびインフラストラクチャサービスを作成しています。



あなたがどのように働くかを最もよく説明する言葉（フレーズ）：



ハードコア研究開発（R＆D）。



一日何時間仕事に専念していますか？



地下鉄に乗ると仕事を始めます。 私が仕事に来ると、集会と会議が始まり、夕方どこかで仕事を続けて家で終わることができます。 合計で、11〜12時間かかります。



仕事に出かける途中で他に何をしますか？



地下鉄で仕事用メールをすくい出さないときは、通常、新しい技術に関する本やマニュアルを読みます。 あなたは常に「ナイフの先端」にいる必要があります。 私よりもこれらの技術をよく知っている人がいることは明らかです。 そして、彼らが私のチームで私と一緒に仕事をしてくれてうれしいです。



何時間寝ますか？



私は8時間寝ようとします。



個人的にどのようなマネージャーを使用していますか？



Trelloを試しました。 そして今では時々それを使います。



どのタスクマネージャー/課題追跡/リポジトリーを使用していますか？



SberTechはAtlassian：JIRAスタックを問題追跡ツールとして使用し、WikiのConfluenceとGitリポジトリのBitbucketを使用します。



どの作業環境を使用していますか？ フレームワーク、他のサードパーティ製品？



アーティファクトには、Nexusを使用します。Javaアーティファクトに加えて、NPMモジュールの形式でJavaScriptアーティファクトも保存します。 Jenkinsで一緒に構築しましょう。



一般に、すべては非常にシンプルです。フロントエンドはJavaScript、バックエンドはJavaです。



テクノロジーに関しては、Webとモバイルの両方でReactを使用しています。 現在、銀行の従業員が使用するアプリケーション向けに、React Nativeを積極的に試験運用しています。



チームには内部プロジェクト、ライブラリがありますか？



実際、私たちのチームは、さまざまなチャネル（インターネットバンク、モバイルアプリケーションなど）のプラットフォームに基づくビジネスソリューションを既に実装している開発者向けに設計されたライブラリの作成にのみ従事しています。



同時に、私たちは開業医になり、最初は本物の場合（ YAGNIに訴える必要がある場合もあります）、次に別のチーム間で再利用する場合にのみライブラリを作成しようとします。



バックエンドでSpringプロジェクトを使用しますが、タスク用に徐々にそれらを書き直します。 一般に、Spring Frameworkを使用しない場合、Springの一部として開発しているプロジェクトは品質が非常に異なり、時には完全に互換性がない場合があります。 Spring Web Flow。これも使用しました。



同時に、Spring Web Flow自体はタスクに適合しなくなりました。SPAをサポートするために最初にアップグレードする必要があり（SWFはサーバー側のレンダリングに重点を置いています）、拡張機能）、SPAで動作するように完全に設計されたアナログを作成しました。これは、すぐにオープンソースに投稿することを望みます。



仕事で一番困るのは何ですか？



私のキャリアのすべてにおいて、私は常にあなたがそれを構築するという原則を堅持してきました-あなたはそれを実行します。 開発チームとサポートチームが1つのチームである場合（異なるマネージャーに直線的に報告しますが）、1つの目標を持ち、新機能の市場投入までの時間とその信頼性とパフォーマンスの両方を担当します。



Sbertechには、個別の開発チームと個別の運用チームがあります。 私たちは非常に密接にやり取りし、コミュニケーションをとろうとしますが、KPIは常に一致するとは限りません。 このため、競合が発生します。私たちは何かを迅速に実装したいので、運用部門は完​​全に完全にテストされ、すべてのフォールトトレランスメカニズムを備えたテストに関心があります。 正当化されることもあれば、単にオーバーヘッドになることもあります。 アジャイルの時代では、そのようなデザインは、私の意見では、変化するはずです。 重要なステップとして、SberbankとSberTechでDevOpsプラクティスを実装するプロジェクトが進行中です。最初の結果はすべて2017年にすでに見られます。



たとえば、Googleで使用されているシステム（各チームが制御するシステムの可用性に対する各チームの予算があるとき）に感銘を受けました。 つまり、複数のリリース中にシステムが絶えず低下した場合、予算が修正されるまで、操作は新しいリリースをブロックします。 しかし同時に、予算がある場合、チーム自身が適用する信頼性ソリューションと、リリースをブロックする権利を持たない運用方法を選択します。



どのような専門文献をお勧めしますか？



私の考えでは、アーキテクチャに関する最高の本は、 ソフトウェアシステムアーキテクチャ：視点と視点を使用した利害関係者との共同作業です。 そこでは、すべての側面が考慮されます。アーキテクチャとは何か、顧客とは何か、アーキテクチャではどの側面がカバーされるべきかなどです。



信頼性に焦点を当てたアーキテクチャに関する2番目の本は、 高可用性と災害復旧：概念、設計、実装です。 少し古いですが、フォールトトレランスソリューションの価値の分析というセクションがあります。 私たちは通常、いくつかの単純で比較的安価なソリューションから始めます。たとえば、バランサーを配置します。 同時に、ひどい地獄で終わる可能性がありますが、これはプロジェクトに少しの信頼性を追加しますが、この機能の開発には莫大な費用がかかります。 この本は、合理的なアプローチを教えています：常に単一の障害点があり、それが存在しない場合、私たちはそれについて単純に知りません。



読みやすいから、私はそれをリリースすることをお勧めします。 この本は、開発者、建築家、管理者など、絶対にすべての人に役立つと思います。

私は紙が好きです。 しかし、通常、技術文献は600ページの本です。 したがって、多くの場合、電子リーダーを使用する必要があります-これには第一世代のiPadを使用します。 しかし、私は今でも紙の本を買って、少なくとも私の家の図書館に立つようにしています。

私のモバイルデバイスはすべてAppleです。 少なくとも数年前、彼らの主な利点は話題でした-高品質。 残念ながら、今ではさらに悪化しています。 しかし、私は古い白いプラスチック製のMacBookを持っていますが、これは単に殺すことはできません。そこでバッテリーを定期的に交換するだけです。 HDDでもネイティブです。

私の経験から学んだことは、委任の必要性です。 同時に、あなたよりも賢い人を雇うようにしてください。そうすれば、あなたにとっても大変な仕事で穏やかな魂を彼らに任せることができます。 自分ですべてを管理することは不可能です。

同じ方法で行こうとしている人に何を勧めますか？



独学は最も重要なことです。 あなたはほとんど経験がないかもしれませんが、あなたはまだ研究所で勉強していますが、例えばCore Javaをよく知っていて、自分でさまざまな技術を試し、小さなプロジェクトをいくつかやったことがあります。 この場合、あなたはすでに応募者を人として扱う会社で働く価値があります。 一部の企業は厳格な審査を行っているだけです。経験がなく、見さえしていません。



そして、私たちは常にすべての候補者を見て、面接でのみ結論を導きます。 私は特に履歴書を見てもいませんが、その人に対して偏見のある態度を持たないように、単にひっくり返すだけです。



したがって、私たちは常に独学に従事しなければなりません。 同時に、基本に精通している必要があります-候補者が一部のコンポーネントのAPIを忘れたとしても構いませんが、内部での動作を知ることができる場合（または、少なくとも正しくないが、少なくとも仮定する）、これは100ポイント先です。



同じことはプログラミング言語にも当てはまります。少なくとも2つの言語に精通している必要があります。できれば正反対の言語（JavaとPython、または.NetとErlangなど）に精通している必要があります。 1つの言語または技術のエコシステムが思考を大きく制限する場合があります。
