Pythonには、独自のコメントスタイル（ドキュメント行）があります。 ドキュメント行は、パッケージ、モジュール、クラス、または関数の最初の構成要素である行です。 このような文字列は__doc__オブジェクト属性を使用して自動的にエクスポートでき、pydocで使用されます。 （モジュールでpydocを実行して、どのように見えるかを確認してください。）ドキュメンテーションラインアグリーメントでは、3行の二重引用符を使用して、そのような行を構成します。 ドキュメンテーション行は、空行に続く文字数、疑問符または感嘆符の数が一致するサマリー行（1つの物理行）として編成し、カーソル位置からの残りのドキュメンテーション行を最初の行の最初の引用として配置する必要があります。 以下は、ドキュメント行の設計に関するさらに詳しい情報です。

ファイルとソケット


ファイルとソケットの使用が終了したら、それらを明示的に閉じます
開いているファイル、ソケット、その他のファイル型のオブジェクトを無視すると、次のような多くの副作用があります。

  • ファイル記述子などの限られたシステムリソースを消費する可能性があります。 多くのオブジェクトに関連付けられているコードは、使用後にシステムに返されない場合、それらのリソースが枯渇して無駄になる可能性があります。
  • ファイルを開いたままにしておくと、移動や削除などの他のアクションがファイルに引き継がれるのを防ぐことができます。
  • コードを介して公開されているファイルとソケットは、論理的に「閉じられた」後に誤って読み書きされる可能性があります。


それらが実際に閉じられている場合、読み取りまたは書き込みを試みると例外がスローされるため、問題を非常に迅速に検出できます。 さらに、オブジェクトのコンストラクターが呼び出されると、ファイルとソケットは自動的に閉じられますが、オブジェクトの有効期間をファイルの状態に関連付けることは、次の理由から悪い習慣です。

  • プログラムの実行中にオブジェクトのコンストラクターが呼び出される保証はありません。 さまざまなPython実装は、遅延ガベージコレクションなどのメモリ管理にさまざまなアプローチを使用し、「リモート」オブジェクトの寿命を大幅に延ばすことができます。
  • ファイルへの予期しないリンクは、予想よりも長く「生きている」可能性があります（たとえば、例外スタック、グローバルオブジェクトなど）。
  • ファイルを管理する優先的な方法は、withコンストラクトを使用することです。 


 with open("hello.txt") as hello_file: for line in hello_file: print line




withコンストラクトをサポートせず、contextlib.closing（）を使用するループのようなオブジェクト 



 import contextlib with contextlib.closing(urllib.urlopen("http://www.python.org/")) as front_page: for line in front_page: print line




Python 2.5で書かれた古いコードは、 from __ Future__ import with_statementからのインポートでwithコンストラクトの使用を有効にすることができました 。



TODOによるコメント


タイムコード、短期パッチ、または優れているが完璧ではないソリューションにTODOコメントを使用する
TODOコメントには、各コメントの先頭にTODO行を含め、その後に名前、メールアドレス、またはかっこで示された問題の解決策を提供できる最善の方法のその他の識別子を含める必要があります。 ポイントはオプションです。 そこで何が起こるべきかを説明するコメントは必要ありません。 主な目標は、TODOの一般的な形式です。これにより、問題のより詳細な説明を提供できる特定の人物をすばやく見つけることができます。 TODOは、人がこの問題を解決することを保証するものではありません。 したがって、TODOを作成すると、ほとんどの場合、名前のみが含まれます。 

 # TODO(kl@gmail.com): Use a "*" here for string repetition. # TODO(Zeke) Change this to use relations. If your TODO is of the form "At a future date do something" make sure that you either include a very specific date ("Fix by November 2009") or a very specific event ("Remove this code when all clients can handle XML responses.").

  • 二重アンダースコア（名前の先頭と末尾）は、言語用に予約されています。




命名規則

  • 「内部」-モジュールの内部、保護クラスまたはプライベートクラスを意味します。
  • 先頭のアンダースコア（_）置換は、保護されたモジュール変数および関数（インポート* fromからインポートされない）でサポートされます
  • 二重の先頭のアンダースコア（__）を変数またはメソッドの名前に置き換えると、事実上、このクラスに対してプライベートになります（名前への追加を使用します）。 関連するクラスと最上位レベルをモジュールにまとめる。 Javaとは異なり、1つのモジュールで1つのクラスを作成することに制限する必要はありません。
  • クラスの命名にはラクダ表記を使用し、モジュール名には大文字と下線を使用した表記を使用します。 命名にラクダ表記を使用するモジュールは多数ありますが、誤解を招く可能性があるため、そうすることはお勧めしません。 モジュールはクラスにちなんで命名されます。 （待って、インポートStringIOを書いたのですか、それともStringIOインポートStringIOからですか？）

Python、pychecker、pydoc、および単体テストには、インポートされたモジュールが必要です。 コードは、実行前に__name__ == '__main__'かどうかを常に確認する必要があります。つまり、モジュールを別のプログラムにインポートするときに、モジュールが完全に実行されないことを意味します。 

 def main(): ... if __name__ == '__main__': main()


モジュールがインポートされると、すべての最上位コードが実行されます。 関数を呼び出したり、オブジェクトを作成したり、pycheckerを使用してファイルを検証したり、pydocを使用してドキュメントを収集するときに実行すべきでない他のタイプの操作を実行しないように注意してください。
