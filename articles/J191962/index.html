Windows Server 2012 R2のストレージと高可用性

こんにちは、同僚の皆様、ITテクノロジーファンの皆さん、こんにちは！



今日は、 Windows Sever 2012 R2のディスクサブシステムとデータウェアハウスの管理の分野で発表されるニュースについてお話ししたいと思います。 Windows Server 2012のリリースにより、データストレージおよびディスクアレイとSANの操作に関して多くの改善があったという事実にもかかわらず、バージョンR2には革新の印象的なリストもありますが、一方で、出力に現れる機能にも進化的な改善がありますその前身。





それで、Windows Server 2012 R2の新機能と、すばらしいサーバー製品の新しい化身で急速に発展し続けているものを把握してみましょう。





倉庫が欲しい-大きくて、速くて、かっこいい...そして、とても安い！





原則として、高性能ディスクストレージや、システムをオンザフライでスムーズにスケーリングする機能などは、まず第一に、鋳鉄製ブリッジのコストに匹敵するコストの大きな高価なハードウェアに関連付けられます。

しかし、一方で、すべての企業がそのようなストレージシステムを購入できるわけではなく、 SANネットワークを構築することは困難でリソースを消費するビジネスであり、正直なところ高価です。



要するに、誰もが最低限のお金で満たしたい要件を定式化するために、次の図が得られます。





図1. Windows Server 2012 R2に基づくデータウェアハウスプラットフォームの機能。



図の左側に、ストレージシステムとディスクストレージの実装に関して企業が通常直面する課題のリストを示します。 右側には、これらの問題を解決するために設計された組み込みのWindows Server 2012 R2の機能のリストがあります。 詳細については、いくつかのポイントを見てみましょう。むしろ、現代の高性能な「スマート」データストレージシステムに通常必要なものを理解します。



1）ストレージは、コンポーネント障害（通常、 ディスクマスコントローラー 、ストレージコントローラーに直接接続するディスクシェルフ 、ディスク自体、またはディスクから形成されるアレイとアグリゲート ）に耐性がある必要があります。 通常、 FCアダプターまたはiSCSIアダプターである接続インターフェースも冗長である必要があり、ディスクへのアクセスの観点から、 マルチスレッドまたはMPIOメカニズムを使用します（ マルチパス入力/出力）。 また、実際には、 CNAアダプターが使用されます （ C onverged N etwork A dapter-統合ネットワークアダプター）-データセンターおよびデータネットワークの構築における新しいトレンド。アダプターはイーサネット環境を使用してLANおよびSANタイプの両方を転送し、 LANからSAN、およびその逆のアダプタのタイプ、動作モードは動的に変更されます。 また、CNAアプローチをサポートするために、 DCB標準（ D ata C enter B ridging）が開発され、送信およびデータネットワークとそれらの収束をより便利に管理できるようになりました-ちなみに、DCBプロトコルは2012バージョン以降Windows Serverでサポートされています



2）最新のストレージは「スマート」である必要があります。これは特に、 データ重複排除 、 シンプロビジョニング、またはいわゆる機能などの機能が存在することで明らかになります。 ディスクサブシステムの 「シンプロビジョニング」、「シンスライシング」 、および仮想化 。 ドライブの種類ごとの負荷分散のための知的階層化も頻繁に行われます。



3）最新のストレージシステムは、OSに組み込まれていることが望ましい便利なツールを管理するのに適しています。 実際には、IT環境は異種環境で構成されており、インフラストラクチャマネージャーが異なるメーカーのストレージシステムの異なるモデルを管理することを実際に示唆しており、この事実に基づいてそれらを管理するタスクは複雑です。 管理に中立的な基準を使用するとよいでしょう。 そのような標準の例は、 SMI-S （ S torage Management I nitiative- S pecification）またはSMPです。



ここで、Windows Server 2012 R2の機能の観点からこの図を見てみましょう。 組織、実際にはデータウェアハウスクラスターのマルチスレッドのサポートに関しては、この機能は昔からWindows Serverに存在しており、ネットワークアダプタードライバーとOS機能のレベルで実装されています-これには問題はありません。 収束、CNAおよびDCBのサポートについては、すでに少し前に述べました。

しかし、2番目の部分に進むと、利益が得られるものがあります（つまり、何について話すか-電子メール）。



それでは、データの重複排除から始めましょう。 この機能はWindows Server 2012で初めて導入され、WS2012 / 2012R2のデータ重複排除はブロックレベルで機能します。 同僚たちに、重複排除は3つのレベルで機能することを思い出させてください。3つのレベルは、一方ではその有効性を、他方ではリソースの強度を決定します。 最も軽いバージョンはファイル重複排除です。 ファイル重複排除の例には、 SISテクノロジーが含まれます（ S ingle Single S torage、これはすでに忘却に沈んでいます）。 ご想像のとおり、ファイルレベルで機能し、完全に重複するファイルを元のファイルの場所へのリンクに置き換えます。 実際のデータをリンクに置き換えることが、重複排除の一般的な原則です。 しかし、今、重複排除されたファイルに変更を加えると、そのファイルは既に一意になり、その結果、実際の場所を「むさぼり食い」させるので、スクリプトは最も魅力的ではありません。 したがって、 ブロックレベルでの重複排除はより魅力的なソリューションのようです。元のファイルが変更された場合でも、変更されたブロックのみがその場所を使用し、ファイル全体のブロックは使用しません。 これが、WS2012 / 2012R2のデータ重複排除の仕組みです。 まあ、完全を期すために、その名前から簡単に推測できるように、ビットレベルで重複排除係数が最も高いビット重複排除も動作することに言及しておく必要がありますが、リソース消費が地獄のように単純になっています...原則として、 ビット重複排除はトラフィック最適化システムで使用されます 、地理的に分散した組織が参加するシナリオの場合に使用されます。組織のオフィス間の通信チャネルは、運用コストが非常に高いか、帯域幅が非常に低い 背骨。 基本的に、このようなデバイスの機能は、送信されたトラフィックをデバイスにキャッシュし、一意のデータビットのみを送信することです。 このクラスのソリューションは、ハードウェアまたは仮想アプリケーション（対応する機能を実装するソフトウェアパッケージを備えた仮想マシン）に基づくことができます。



重複排除ではすべてが素晴らしいものになりますが、1つだけでは... WS2012の重複排除はオンラインデータに適用できませんでした。 プロセスで忙しい人は使用中です。つまり、VMでの重複排除の使用が不可能になり、このアプローチの魅力がすべて失われてしまいます。 ただし、WS2012R2の改善により、アクティブなVHD / VHDXファイル上で重複排除を使用できるようになり、VHD / VHDXライブラリ、製品ディストリビューションとの共有リソース、および一般的なファイルボールに対しても効果的です。 次の図は、多数の仮想ディスクとVHD / VHDXディスクを使用したホームサーバーの重複排除のパフォーマンスインジケーターを示しています。





図2. Windows Server 2012 R2の重複排除パフォーマンス



シンプロビジョニングについて話している場合は、VMにこのアプローチを実装することもできます。 一方のプロトタイプ、つまり親ディスクを作成し、それから差分ディスクを作成するために、内部の仮想OSの一貫性と均一性を維持し、他方では、VMディスクが占有するスペースを削減します。 このシナリオの唯一の微妙な点は、親仮想ハードディスクを高速ドライブに配置することです。多くの異なるVMから同じハードディスクに並列アクセスすると、セクターデータとデータブロックの負荷が増加するためです。 つまり SSDを選択するか、仮想ストレージを選択します。 さて、ここで知的引き裂きも役立ちます。



仮想化...そして何が起こるのか？..





ただし、すべてが正常です。

ディスクサブシステムの仮想化の問題に戻りましょう。

歴史的に、ディスクサブシステムの仮想化は非常に古いトピックであり、歴史です。 仮想化は物理レベルからの抽象化であると思い出せば、つまり 基礎となるレベルを隠す -マザーボード上のRAIDコントローラーは、ストレージ、つまりドライブ自体の仮想化機能も実行します。

ただし、WS2012では、Storage Spacesメカニズムが登場しました-実際にはRAIDコントローラーに類似していますが、WS2012 OSデータ用の特別な入力/出力ドライバーのレベルです。 実際、作成される論理アグリゲートのタイプは、RAIDボリュームのタイプに非常に似ています：シンプル（RAID 0）、ミラー（RAID 1）、およびパリティ（RAID 5）。 WS 2012 R2では、このメカニズムにインテリジェントストレージ階層化が追加されました。 つまり、1つのユニットに含まれるさまざまなタイプのドライブ（ SATA 、 SAS 、およびSSD ）の上にユニットを作成できるようになりました。WS2012R2は、負荷のタイプと強度に応じて、ドライブ自体の負荷をインテリジェントに分散します。





図3.ストレージスペースを使用して論理ドライブアグリゲートを作成するときのスマートティアリングの有効化。



信頼性、信頼性、もう一度信頼性





さて、データウェアハウジングに関連する規律について話している場合、データの信頼性の問題、またはむしろそれらの配置の信頼性に注意を払うことはできません。 重要な点がいくつかあります。



WS 2012の出現により、ファイルボールとSMB 3.0ファイルシステムの上に負荷をかけることができます。 さて、ここで、もちろん、ストレージのコンポーネントの重複を思い出してください：組み込みのNICチーミング機能またはデータ転送インターフェイスの集約を使用してネットワークアダプターを予約することができ、SMBマルチチャネルメカニズムはSMBメッセージングレベルでMPIO機能を実行します。 私たちはチャンネルの崩壊から身を守りました-スイッチのみを複製することを忘れないでください。 まあ、VM展開の継続性の観点から、そしてファイル全般だけでなく、 スケーラブルなファイルサーバー（Scale-Out File Server-SOFS）を展開し、重要なデータを配置することができます。しかし、それはデータアクセスの連続性です 。 これに加えて、クラスターを作成するときにCSVボリュームにReFSファイルシステムを使用するWS 2012 R2の機能を追加します。ここでは、作業を停止することなく、ほとんど信頼性とファイルシステムエラー修正をオンザフライで実行できます。 ReFSのWS2012の最初のバージョンには痛みがありましたが、今ではもちろん回復しており、本当に回復力のあるシステムになっています！



SMBを介してデータを暗号化する機能は、データの安全な配置とデータへのアクセス制御を重視する人々にとっても興味深い機能です。

もう1つの興味深い機能は、メカニズムを使用してディスクサブシステムの帯域幅とパフォーマンスを制御する機能です（ ストレージQoS） 。 このメカニズムは興味深く、重要です。このメカニズムを使用してストレージ上に負荷をかけるポリシーをさらに使用し、このデータをSystem Center Virtual Machine Manager 2012/2012 R2に転送できます。



そして最後に、管理タスクの観点から、Windows Server 2012 R2、System Center 2012 R2はSMI-S、SMP、 WMIなどのさまざまなニュートラル制御メカニズムをサポートしていることを思い出してください。
