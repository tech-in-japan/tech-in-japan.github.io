高負荷下のクラウドサービス。 カックル体験

みなさんこんにちは！ Cackleは、2011年からWebサイト向けのクラウドSaaSソリューションを開発しています。 当社の製品は10,000以上のサイトにインストールされており、毎日平均6500万のユニークヒットを処理しています。 ピーク時の帯域幅は780メガビット/秒に達し、データベースは1日あたり最大1億2000万件の読み取り要求と最大30万件の書き込み要求を受信します。 このようなワークロードは、私たちが共有したい難しいソリューションを発明することを困難にします。









最初に、私たちが一般的に開発したもの、すべての技術とアーキテクチャが基づいているもの、高負荷の原因について少し説明します。 さらに-この負荷に対処するために適用する5つの基本的な決定。

フロントエンド



フロントエンドはウィジェットで構成されています。 ウィジェットは、他の共有JavaScriptライブラリに基づく実行可能なJavaScriptライブラリです。 共有ライブラリの例：



  • DOMを操作するためのライブラリ：jQueryは非常に遅く、いくつかのモバイルプラットフォームでは動作しなかったため、彼らはVanilla JSでソリューションを作成しました。 その結果、一部の操作では、速度が600倍に増加しました。このソリューションは、AndroidおよびiOSのすべてのバージョンで機能します。
  • 統合認証ライブラリ（匿名、ソーシャル、SSO）;
  • 対話プロトコル（WebSocket、EventSource、Long-Polling）をスマートに選択できるリアルタイムモードサポートライブラリ。
  • クロスドメインGET / POSTリクエストライブラリ。
  • その他：日付、ブラウザデータベース（cookie、localstorage）、json、画像の読み込み、ビデオ認識（YouTube、Vimeo）、通知トレイ、ページネーションなどを操作します。


すべてのウィジェットはiframeなしで機能します。これにより、CSSをWebサイトのスタイルに合わせて変更できます。

バックエンド



これは、Nginxサーバーの外側にラップされたApache Tomcatコンテナーのクラスターです。 この場合のNginxはプロキシとしてだけでなく、負荷の「シンク」としても機能します。 複数のスレーブへのストリーミングレプリケーションを備えたPostgreSQLデータベース。



すべてのバックエンドは、ロシアとヨーロッパの複数のデータセンター（DPC）に分散されています。 私たちの経験では、すべてのサーバーを1つのデータセンターでホストするのは危険すぎるため、3つの異なるデータセンターに接続しています。



リアルタイム



ブラウザ側でのリアルタイム更新（コメント、いいね、編集、モデレーション、プライベートメッセージ、チャット）のサポートは、サポートされているテクノロジーのいずれかを通じて行われます：WebSocket、EventSource、Long-Polling。 つまり、最初にWebSocketがあるかどうかを確認し、次にEventSource、Long-Pollingを確認します。 切断（エラー）の場​​合、通信はsetTimeoutで接続ステータスを監視する機能によって自動的に復元されます。



サーバーでは、Nginxクラスター+ プッシュストリームモジュールを使用します。 サーバーは3台のみです。2台は共有、1台はオンラインコンサルタント用です。 バックエンド（Tomcat-s）からのリアルタイムメッセージは、すべてのサーバーに送信されます。 また、ブラウザでウィジェットから接続すると、任意のサーバーが（ランダムに）選択されます。 その結果、バランスのようなものが得られます（残念ながら、プッシュストリームはそのまま使用できるバランスをサポートしていません）。

  • SMTPサーバー。

問題



高負荷は2つの問題を引き起こします。



まず、ほとんどすべてのホスティング事業者は、デフォルトで100 Mbpsのサーバー帯域幅を持っています。 上記のすべてが削減されるか、せいぜい追加の帯域を購入するように求められます（価格はサーバー自体のコストよりも数倍高くなります）。



2番目の問題は負荷そのものです。 物理学を欺くことはできません。サーバーがどれほどクールであっても、サーバーには独自の制限があります。



»ソリューション1：JavaScriptでのバランス調整



負荷分散の標準的な方法は、サーバー上の入力要求の分散について説明します。 これにより2番目の問題は解決されますが、最初の（帯域幅）は解決されません。発信トラフィックは同じサーバーを通過するためです。



2つの問題を同時に解決するために、ローダー自体（widget.js）でJavaScriptのバランスをとり、バックエンドをランダムに選択します。 その結果、サーバーウィジェットのクラスターからサーバーにトラフィックをリダイレクトし、それらの間で帯域幅を分割し、負荷を分散します。



この方法のもう1つの大きな利点は、JavaScriptキャッシュです。 ページが更新されると、すべてのライブラリ（widget.jsローダーを含む）がブラウザーキャッシュから受信され、サーバーは新しいリクエストを静かに処理し続けます。

»ソリューション2：Nginx Microcache

»ソリューション3：Tomcat、Javaキャッシュの調整



Tomcat



特にTomcat Tomcatでは動作しません。 実務経験から：



  • コネクタ（Http11Protocol、Nio、4月）の選択にだまされないでください。 負荷が高い場合、これは役割を果たしません。
  • connectionTimeoutをできるだけ小さく、できれば5秒以内にしてください。
  • maxThreadsを大きくしすぎないでください。助けにはならず、状況によっては大きな害を及ぼすだけです。 ロードされたプロジェクトの標準は300〜350です。 これで十分でない場合は、新しいTomcatサーバーを追加します。
  • ただし、acceptCountは数千（2000〜4000）にすることができます。 値が低いと、接続が切断され（接続が拒否されます）、キューに追加されます。 同時に、OSで同じまたはそれ以上のバックログを設定することを忘れないでください。


Javaキャッシュ



一般に、負荷の高いプロジェクトのデータベースは弱点であると受け入れられています。 本当にそうです。 たとえば、サーバーは要求を受け入れ、サービスレベルに送信します。サービスレベルでは、サービスがデータベースにアクセスして結果を返します。 この場合のリンク「サービス-データベース（リレーショナル）」はすべての中で最も遅いため、サービスをキャッシュでラップするのが一般的です。 したがって、データベースからの結果はサービスキャッシュに入れられ、次にキャッシュから取得されます。



キャッシングサービスについては、独自のキャッシュを開発しました。これは、標準のキャッシュ（Ehcacheなど）の動作が遅く、特定のタスクを必ずしもうまく解決できないためです。 特定のタスクのうち、単一の値に対して複数のキーをサポートするキャッシングです。 org.apache.commons.collections.map.MultiKeyMapを使用します。



これはどのタスクに必要です。 たとえば、ユーザーはコメント付きのページにアクセスします。 たくさんのコメントを言ってみましょう、300個。 それらはそれぞれ3ページ（ページネーション）に分割され、それぞれ100ページです。 最初の呼び出しで、最初のページがキャッシュされ（100コメント）、ユーザーが下にスクロールすると、2ページと3ページが順番にキャッシュされます。 ユーザーがこのページにコメントを公開したら、3つのキャッシュをすべてリセットする必要があります。 MultiKeyMapを使用すると、次のようになります。 



 MultiKeyMap cache = MultiKeyMap.decorate(new LRUMap(capacity)); cache.put(chanId, "page1", commentSerivce.list(chanId, 1)); //chanId - id    cache.put(chanId, "page2", commentSerivce.list(chanId, 2)); //commentSerivce.list -     (chanId)   (2) cache.put(chanId, "page3", commentSerivce.list(chanId, 3)); cache.removeAll(chanId); //  3

»ソリューション4：さまざまなデータセンターへのレプリケーションをストリーミングするPostgreSQL



私たちの意見では、PostgreSQLは負荷の高いプロジェクトに最適なソリューションです。 今ではNoSQLを使用するのが流行していますが、ほとんどの場合、適切なアプローチと適切なアーキテクチャを備えているため、PostgreSQLの方が優れています。



PostgreSQLでは、ストリーミングレプリケーションは非常にうまく機能し、同じサブネット上でも、異なるネットワーク上でも、異なるデータセンターでも問題になりません。 たとえば、いくつかの国にデータベースサーバーがあり、深刻な問題はありませんでした。 唯一の注意点は、リリースの大規模なベース変更（ALTER TABLE）です。 一度にすべてのUPDATEを実行しないように、断片的に作成する必要があります。



複製を設定するための多くのリソースがありますが、これはハックされたトピックなので、追加する特別なものはありません：



  • フェイルオーバー（マスターのクラッシュ）の場合は必ず計画（構成）を作成し、実際にテストしてください。
  • WALの原理を理解してください。スレーブとマスターが同期していない場合、アーカイブが必要な理由とアーカイブの場所を知る必要があります。
  • ウィザードのHDD上の場所を追跡します。突然消えると、PostgreSQLがクラッシュする可能性があり、レプリケーションでは大きなトラブルが発生します。
