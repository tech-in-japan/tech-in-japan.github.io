スクローラーは、スライドスタックの観点から、重要なイベント、つまり次のものをレポートするインターフェイスをスライドスタックに提供する必要があります。 



  public interface ScrollingListener { void onScroll(int distance); void onStarted(); void onFinished(); void onJustify(); }




スクロールが開始されたとき、スライドが一定数のピクセルを移動したとき、そのスクロールは終了し、スライドの位置を揃える必要があり、スクロールは完全に完了しました。



スライドスタックの観点から見ると、これ以上簡単なものはありません。すべてのタッチイベントの処理をスクローラーに委任するだけで、実際に何が起こったのかがわかり、必要なコールバックが発生します。



お気づきのとおり、すべての委任は、android.view.View.OnTouchListenerインターフェイスのonTouch（View v、MotionEventイベント）メソッドを介して行われます。 



  @Override public boolean onTouch(View v, MotionEvent event) { switch (event.getAction()) { case MotionEvent.ACTION_DOWN:{ int pointerId = MotionEventCompat.findPointerIndex(event, mActivePointerId); if (pointerId == INVALID_POINTER_ID){ break; } mLastTouchX = MotionEventCompat.getX(event, pointerId); mJustifying = false; forceFinished(true); clearMessages(); break; } case MotionEvent.ACTION_MOVE:{ int pointerId = MotionEventCompat.findPointerIndex(event, mActivePointerId); if (pointerId == INVALID_POINTER_ID){ break; } // perform scrolling float x = MotionEventCompat.getX(event, pointerId); int distanceX = (int)(x - mLastTouchX); if (distanceX != 0) { mTouchScrolling = true; startScrolling(); mListener.onScroll(-distanceX); mLastTouchX = x; } break; } case MotionEvent.ACTION_UP: mTouchScrolling = false; mActivePointerId = INVALID_POINTER_ID; break; } if ((!mGestureDetector.onTouchEvent(event) || ((SlideStackView)v).isOverScrolled()) && (event.getAction() == MotionEvent.ACTION_UP || event.getAction() == MotionEvent.ACTION_CANCEL)){ justify(); } return true; }




順番に行きましょう：



  • ACTION_DOWNは、ユーザーが特定の座標で特定の領域の画面に触れたことを示します。 この場合、そのような場合には完全に標準的な手順を実行する必要があります。つまり、地面を準備し、画面上の後続のユーザーの動きに備えて、それに応じてスライドスタックをスクロールします。 この場合、この手順には、初期位置の保存、すべてのフラグおよび現在のスクロールのリセットが含まれます（実行された場合）。
  • ACTION_MOVEは、移動自体、つまりユーザータッチが新しい座標に移動したことを示します。 座標の差を計算し、ユーザーがスクロールしたことをスライドスタックに通知する必要があることを推測することは難しくありません。 SlideStackScrollerなどのコンポーネントを強調表示することで、モーションイベントの処理とスライドスタックの内部ロジックとの過剰な結合を回避しました。 言い換えれば、スライドスタックの現在の状態、つまり一方向にスクロールできるかどうかは心配していません。 ユーザーが何をしたか、そしてこれに応答するかどうか、そしてどのように反応するかを彼に伝えるだけで、スライドスタックが決定します
  • ACTION_UP そのため、多くの場合、ユーザーは画面との対話を終了します。 しかし、これはリラックスできるという意味ではありません。 おそらく、作業の最も単純な部分だけがここで終わります。 私は誰もが慣れているジェスチャーを意味し、通常は投げ飛ばしと呼ばれます。 リテラル翻訳（「スロー」）では、これが何を意味するのか理解できません。 これに関連して、ジェスチャとは、ユーザーが画面上で指を動かし、あたかも要素を加速し、触れないように実際の身体と同じように要素が動くことを期待するという意味です。 つまり、慣性によるものです。 たとえば、電話をテーブルの上で他の人にプッシュした場合、どのようにスライドするか。
  • 最後に、スライドの位置を調整できるようにジェスチャが終了したかどうか、またはユーザーによって行われたジェスチャがフリングとして解釈できるかどうかを判断する必要があります。その後、適切なアクションを実行し、完了するまで待ってからスライドを調整する必要があります




すぐに言いたいのは、一見すると追加的で重要ではないように思われるフリングが、スライドスクロールの最も頻繁な完了であるということです。 問題は、ほとんどのユーザーがプラットフォーム自体と多くのアプリケーションの両方で水平スワイプに慣れているため、このジェスチャーを直感的かつ迅速に実行することです。
