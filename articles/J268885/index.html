ステロイド上のOpen REST APIまたはiLOを介してHP ProLiantサーバーを管理する

HPからの紹介： この投稿は、勤務中に匿名を希望するお客様の1人によって書かれました。 名前とIPの一致はすべてランダムと見なされます。 モスクワのデモセンターでいつでも同じことを試すことができます。ご希望の方はコメント欄にご記入ください。



ご挨拶。 私の名前はエドワードであり、ITインフラストラクチャの管理を外部委託する小さな会社のシステム管理者です。 Ahem ...すでに問題のあるクラブの匿名の誰かのように見えますか？ さて、大丈夫、私たちは本当に1つの興味深い問題を抱えていました。今日は、どのように問題に遭遇し、どのように誤って解決したかを説明します。









まず、少しの背景。 それにも関わらず、彼らは、5〜8年ほど前に遠隔の鉄を管理するのがどれだけ楽しかったか覚えていますか？ DCのエンジニアにアプリケーションを作成し、KVMの接続、BIOS / CMOSの設定、ネットワークブートの設定、システムの設定を待機します（誰かが既にpreseed / kickstartファイルジェネレータを作成し、DCにDHCP / PXEがあればよいサーバー）。 そして、すべてのサーバーでipmiを取得しました（時間が経つにつれて）。 ああ、最初はどれほど幸せでしたか！ 



ipmitool -U user -P password power up -H XXX ipmitool -U user -P password password bootdev pxe -H XXX ipmitool -U user -P password power reset -H XXX


そして、サーバーはすでにOSを入れているようです。 退屈な場合は、コンソールでこれを見るだけです。 唯一の悲しいことは、BIOSを手動で更新し、構成する必要があったことです（少なくとも、HT / VT-dを有効にします）。 その結果、インストールされた四半期に応じて、すべてのサーバーの構成が異なりました。 さて、あなたは最も若い管理者が常にサーバーを設定することを理解しています。



何か重要なものを見つけたとき、私たちは歩いて、手で設定を切り替えました。 混乱し、唯一の。 しかし、一般的に、これらすべては、私たちに話が起こる前に私たちに合ったものでした。



クライアントの一部をハードウェア上に直接配置することに注意する価値があります（より正確には、このハードウェア上の仮想マシン上にあります。 想像しやすいように、私たちのシステムはOpenStackに非常によく似ています。 そして、ある晴れた日、顧客が私たちのところに来て、「 プライベートクラウドで仮想マシンを管理するシステムを作るために自分のハードウェアで 」と尋ねました 。 私たちは喜んで、契約と作業明細書に署名した後、彼に彼の決定を示し始めました、彼はすべてが好きでした。 私はすべてが好きで、マネージャーはすでにシャンパンを開いています（しかし、私たちの基準による契約は非常に良かったと言わなければなりません）。 そして、ここでクライアントはアイテムTKに指を向けて尋ねます-「 そしてこれでどうやって？ 「。 この時点で、管理者は座って困惑しています 。「 インベントリ、ラックへのインストール、電源ネットワークの接続後、新しいサーバーを自動的にクラウドに追加する必要があります 。」 私たちは見せ始めました-見て、ここにサーバーを記録します（1c）、ここにそのMACアドレスを追加し（dhcpサーバーとpxeを管理するための自己記述Webフェイス）、スクリプトを実行し、ネットワーク経由でサーバーが起動し、システムがインストールされます（私たちは安心しました）少なくともシステムが自動的にインストールされていることに気付いてから息を吐き、その後、サーバーを再起動してキャッチし、Delキーを強く押します...一般的に、クライアントの担当者はそれを見て、頭をかき、会話を歩き回り、考えをまとめて言いました。 ここに自動的に？ さて、1C以外のすべてを投げましょう、すべて同じように、会計士は私ではなくそこに、そしてあなたがMACアドレスを書いた場所にサーバーを置きます 。」 まあ、そしてすぐに彼は去り、私たちがまだ問題を解決することを望んでいると付け加えました。



それから、いつものように、TORを読むことにしました。 私たちは、（私たちにサインして、私たちにそれを書き直した人ではなく、タスクを実行した人たちが）初めて目にしたあらゆる種類の興味深い点を見つけました。 たとえば、OSをバイパスする温度監視、電力消費監視、自動BIOSセットアップなど、考え始めました。 最初は、ipmitoolがこれをすべて実行できると考える傾向がありました。 それから、彼らはこれがどのように見えるかを示しました。 設定を更新する方法、センサーを監視する方法について考えました...一般的に、週末に「可能な限りipmiの代替手段を探してGoogle全体を探索する」というタスクを手放しました。



私たちは月曜日に到着します、誰もが悲しい、怒っています。 そして、一人の管理者が座って笑っています。 私たちがそのような災害に遭ったときに彼がそれを読む理由についての質問で彼を悩ませるのは論理的でした。 判明したように、彼はそれが起こったのではなく、Redfishサーバー管理仕様v 1.0（ ここを読んでください ）を読んだことがうれしかったです。 彼らはすべてを声を出して一緒に読み、怒ったことをやめました。 Redfishはまさに私たちが必要としているものであることがわかりました。 彼らはマネージャーを雇い、Redfishが実装されているハードウェアを見つけるためにすべてのサーバーメーカーに電話をかけました。 彼らは1時間後に来て、笑い、入り口の箱を指して、「 すべて、見つけた」と言います 。 実際、このボックスはHP Proliant Gen9のボックスであることが判明しました。 私たちはサーバーを見つけ、実験室に返しました（そして、DCにすぐにアクセスできる最新の未探索サーバーを推測した人もいます...）、HP REST API（HPサーバーでのRedfishの実装）について知り始めました。



私たちは管理者であるため（これは後にpythonistによって与えられ、彼はすべてを書いて美しいWebインターフェイスを作成しました）、最初にしたことは必要なことをするsh-nicknamesを書くことでした。 もちろん、コンソールcurlを使用してRest APIにアクセスし、エコー経由でJSONを生成することはお勧めしません（まあ、少なくとも後で誰にも表示しない）。ただし、HP ProLiant REST APIとの相互作用の例を共有できます（特に代表者自身から） HPからこれについて尋ねられました）。



実際、そこには多くの可能性があります。ドキュメントには、オブジェクトのリストの200ページとそれらの簡単な説明があります。 概念実証として、主なタスクを達成しようとした最初のこと。 もちろん、それらのいくつかはIPMIを介して実行できますが、1つのツール、つまりAPIを使用することにしました。
