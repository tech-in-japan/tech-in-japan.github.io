リクエストの処理と分類。 パート2：ナビゲーションリクエスト

欲求を表現する主な方法は2つしかありません。 何を取得（または実行）する必要があるかを記述するか 、「テレポート」したい場所を示すかです。 最初のケースでは、システムは最適なネットワーク回答を適切に選択することにより、要求を理解しようとし、決定ツリーでそれらの数百のプロパティを比較します。 2番目のルールでは、正しい答えは原則として1つだけであり、検索エンジンがそれを知っていることを期待しています。



2番目のタイプのクエリ、 どこで 、 どこで質問に答える-ナビゲーションクエリ。 私たちが彼らとどのように働くかについての短い話をあなたの注意にもたらします。



一般的に言って、 何と どこの境界はかなりあいまいです。 「サイクリストフォーラム」と「自転車フォーラムru」という2つの検索クエリを検討してください。 現在、これらの両方の要求に最も関連する回答は、同じサイトvelo-forum.ruです。 違いは、最初のケースでは答えが唯一のものではなく、彼と彼の仲間の間で競争があることです.2番目のサイトでは、サイトは結果のリストに載っているだけでなく、それがどれほど悪いか良いかに関係なく、それを導く必要があります。他の自転車フォーラムと比較して。 2番目の要求はナビゲーションであり、最初の要求はナビゲーションではありません。



「オノマトペwiki」など、混合型のクエリもあります。 今のところそれらを脇に置きますが、それらに戻ります。



そのため、ナビゲーション要求への回答を事前に知る必要があります。 技術的には、このタスクは次のように定式化されます。各ナビゲーションリクエストに対応するインターネットアドレス（以下、ターゲットと呼びます）を関連付けるために ：サイト、サイトのセクション、サイトのページ。 その前に、リクエストがナビゲーションであるかどうかを判断します。



なぜこれが必要なのですか？

同時に、多くのナビゲーションクエリがあります。 あまりにも多くの人が放置するのは非常に軽薄です。 結局、データを処理する上で最も重要なことは、会計と管理です。 それが統計と監視です。 この場合、クエリプロパティの統計と応答の品質の監視。

ホールヘルプ



ユーザーが無関係な回答をクリックすることはほとんどありません。 したがって、ナビゲーション要求の場合、すべてが正しい答えのみを選択することを期待するのは理にかなっています。 作業仮説は、逆も真であるというものでした-リクエストを送信するほとんどのユーザーが同じ結果をクリックすると、リクエストはナビゲーションであり、結果はそのターゲットになります。



このテストでは、仮説全体が真実であることが示されました。非常に関連性の高い回答（「nokiaテーマ」や「icqのダウンロード」など）を含むまれな非ナビゲーションクエリのみが同様の行動パターンを引き起こします。 ただし、ほとんどすべての誤検知では、擬似ターゲットがサイト内のページであることが判明したため、このような要求を（一時的に）考慮から除外しました。



ユーザーの全会一致を測定するためのいくつかの方法（従来のクリック画面を含む）を検討し、次のシンプルで便利な指標を決定しました。



C iを結果R iのクリック数とし、 demandC iをオンデマンドの合計クリック数とします。

N = log C i / log ∑ C iは、リクエストのナビゲーションの程度です。



手動評価では、メトリック値が0.95を超えるクエリは高精度のナビゲーションであることが示されました。 さらに、この精度は、頻度クエリとまれなクエリの両方で等しく高いです。 この方法を使用すると、ナビゲーションの要求/応答のベースは、約80,000のストレージユニットに成長しました。

「デバイス」クエリ



ナビゲーションクエリは、自然言語の文のようにモノリシックではなく、構成で分解することもできます。異なる単語は異なる役割を果たします。 合計で、5つのそのような役割が特定されました（一般に受け入れられている用語がないため、私は自分で作成しなければなりませんでした）。 以下は、すべての「ナビゲーションオファーのメンバー」が存在する複雑なクエリです。

ナビゲーションの役割について説明しましょう。



カーネル -リクエストが導くサイトを一意に識別するフラグメント。 これは、リクエストの最も重要な部分です。 通常、サイトには10​​を超える異なるコアはありません。 たとえば、サイトlib.ruの場合、「lib ru」、「libra」、「libru」、および「midgeのライブラリ」です。



バックグラウンド -サイトに許可されたフラグメント。 リクエスト内でのそれらの存在自体は、そのナビゲーションの性質を示すものではありませんが、同時に適切なカーネルの存在下でターゲットを変更することはありません。 youtube.comの場合、これらは「ビデオ」、「ビデオ」、 headhunter.ruのような単語です-「空室」、「仕事」など。



パス -ターゲットをルートページからサイトに移動する単語。 たとえば、Yandexコアに適用される「カード」という単語は、要求をmaps.yandex.ruにリダイレクトします。



リージョン - パスの一種。リクエストの地理的位置を示します。 その特徴は、地理依存ナビゲーションの場合、クエリテキスト内の領域の明示的な表示は、ユーザーの場所の実際の変更と同等であることです。 たとえば、Kazanからユーザーから受け取ったIKEAリクエストは、他の地域から送信されたIKEA Kazanリクエストと同じ場所につながるはずです。



ノイズ -ナビゲーションに関して何も意味しない言葉。 これらは、「www」、「http」、「website」などの品詞および単語のサービス部分です。



各サイトについて、これらのクエリフラグメントはすべて、多くの場合、異なる順序および組み合わせで繰り返され、データベースに存在します。 重複を排除するために、「複合」クエリを基本部分に自動的に分割して、一意のフラグメントのみを残し、それらの相互作用のロジックをプログラムで実装する方法を探し始めました。



解決策は驚くほど単純でした-私たちの手に渡ったのはデータベースデータの冗長性でした。



同じターゲットを持つクエリをソートし、次に短いクエリを長いクエリからカットすると、クエリのセット全体が2つのタイプのフラグメントに分割されます：個別のクエリとして存在するものと、クエリの一部としてのみ存在するもの。 たとえば、ソースリストがリクエスト「YouTube」と「YouTube video」で構成されている場合、「YouTube」という単語が最初のリストと2番目の「video」に表示されます。 これらはそれぞれコアとバックグラウンドになります。



サイト内の主要なリクエスト（「Raiffeisen ATM」など）を取得し、同様にルートページ（「Raiffeisen」）に到達するリクエストを「減算」すると、 pathが取得されます 。



途中で、最初のリストのさまざまなフラグメントの数をカウントし、データベースに最も頻繁なフラグメントのみを残しました。したがって、多くのクエリがそれらにつながる大規模なサイトでは、小さなもののために何も失わずにすべてのゴミを捨てることができました。
