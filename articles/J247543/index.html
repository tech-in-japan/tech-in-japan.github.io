Impress Application Serverを簡単な言葉で

これは、 Impress onHabréについての最初の入門記事ではありませんが、この1年で多くの質問を受け、このアプリケーションサーバーのアーキテクチャと哲学を説明する経験を得ました。そして、それをマスターし始めた開発者の問題とタスクをよりよく理解し始めたことを願っています。 はい、そしてサーバー自体に、まったく新しい入門記事の緊急性を高めるのに十分な変更がありました。



Impress Application Server （IAS）は、ノードの下での開発の主流ではなく、さまざまな繰り返される典型的なタスクを単純化および自動化し、アプリケーションコードの抽象化レベルを上げ、アプリケーションのフレームワークと構造を設定するように設計された、代替アーキテクチャと哲学を持つNode.jsのアプリケーションサーバーですコードのパフォーマンスと開発者の生産性の両方を最適化します。 IASは現在、サーバータスクのみを対象としていますが、包括的な方法でそれを行います。たとえば、1つのポートでAPI、Webソケット、ストリーミング、静的、サーバー送信イベント、プロキシおよびURL書き換えを組み合わせ、複数のドメインと複数のアプリケーションを1つのように提供できますサーバー、および全体として1つのアプリケーションサーバーとして動作するサーバーのグループ。



はじめに



まず、node.jsの一般に受け入れられているアプローチで、アプリケーションサーバーの開発を開始するように促されたいくつかの問題をリストします。

  1. 多くの場合、アプリケーションコードはシステムコードと混在しています。 実際、ノードと派生フレームワークのほとんどは低レベルであり、各アプリケーションにはサブジェクト領域のタスクに関連しないシステムコードの一部が必ず含まれています。 そのため、たとえば、HTTPヘッダーを追加するとPatientクラスのメソッドになり、この患者にURLをルーティングし、Webソケットを介してイベントを送信するタスクと同じファイルに含まれます。 これは怪しいです。
  2. 野田は、アプリケーションアーキテクチャの面で過度の自由を与えています。 あなたが同意しなければならないミドルウェアの概念に加えて、完全な開発には十分ではありません。一般的なアーキテクチャパターンはありません。 プロジェクトのファイルへの分割、ロジックのクラスへの分割、パターンの適用、アプリケーションでの内部APIの作成、レイヤーの割り当て、さらにはディレクトリ構造さえも、開発者の裁量に任されています。 その結果、プロジェクトの構造とアーキテクチャはチームや専門家によって大きく異なるため、コードの理解とドッキングが複雑になります。
  3. ノードの世界では、RESTの狂信的な崇拝があり、その結果、サーバーメモリに状態を保存することを拒否します。 これは、Node.jsアプリケーションがメモリ内に長期間存在するという事実にもかかわらずです（つまり、すべてのリクエストでロードされず、リクエスト間で終了しません）。 問題のモデルを長期にわたって展開する能力を無視する弱いメモリ使用量（I / Oを最小限に抑えることができたおかげで）はパフォーマンスに対する犯罪です。
  4. npmの多数のモジュールはガベージモジュールです 。いくつかの優れたモジュールの中から、適切なモジュール を見つけるのは簡単ではありません（ダウンロード数とスター数は、コードの品質とトラブルシューティングの速度を常に適切に反映するとは限りません）。 優れた、非常に優れたモジュールのセットから完全なアプリケーションを作成することはさらに困難です。 一緒に、彼らは不安定で予測できない振る舞いをすることができます。 モジュールは相互に十分にシールドされていないため、統合の競合を排除できません（たとえば、一部のモジュールはres.endをオーバーライドしたり、httpヘッダーを送信したり、他のモジュールはこの動作を予期していません）。


まだ多くの小さな問題があり、Node.jsのキャッチエラーに伴う深い悲しみは、涙、血、コーヒー（茶）で満たされた3つのボリュームのトピックです。 上記のすべての結果として、ノードは依然として懸念の原因であり、ほとんどの場合、他のサーバーテクノロジーと組み合わせて追加ツールとして使用され、クライアントアプリケーションのアセンブリのスクリプト作成、プロトタイピング、またはWebソケットでの通知の配信などの補助作業を実行します。 非常にまれに、サーバー部分がノード上にのみある大規模なプロジェクトに会うことができます。



問題の声明



ネガティブな動機付け（リストされている問題）に加えて、IASの開発に対するポジティブな動機付け要因（アイデアとタスク）もありました。

  1. それぞれが独自のクラスター（IPCプロセス間通信に関連付けられたプロセスのクラスター）を持つ複数のサーバー上のnode.jsアプリケーションのスケーリング 。
  2. 複数のアプリケーションを単一のプロセスの一部として、またはプロセスのクラスター、またはそれぞれにプロセスのクラスターを持つサーバーファームとして提供する。
  3. ファイルシステムを監視することにより、アプリケーションを再起動しなくても、ディスク上でコードが変更された場合、メモリ内のコードを自動的に置き換えます 。 アプリケーションによってロードされたファイルが変更されるとすぐに、IASはそれらをメモリに読み込みます。 メモリ内のある時点でコードのいくつかのバージョンがあり、変更前のすべてのリクエストの処理が完了するとすぐに古いバージョンがアンロードされ、新しいリクエストは次のリクエストにすでに使用されています。
  4. プロセス間のメモリ内のデータ構造の同期 。 もちろん、すべてのメモリ構造ではなく、そこに展開されたドメインモデルのグローバルフラグメントのみです。 追加の変更とトランザクションがサポートされています。 いくつかのパラメータが異なるプロセスで並行して増加する場合、順序は重要ではないため、これらの変更はマージされます。


感動の哲学



  1. 最大メモリ使用量 。 非同期I / Oより高速なのは、I / Oがまったくない場合、または要求が発生していないときに最小限に抑えられ、レイジーモードで実行される場合のみです。
  2. モノリシックアーキテクチャと高いコード結合性、すべてのコアモジュールは統合、調整、最適化されて連携して動作します。 これにより、不必要なチェックはなく、典型的な問題を解決する際の動作は常に予測可能です。
  3. ポート、ホスト、IP、プロトコル、サーバー、プロセス、アプリケーション、プロセッサー、メソッドのマルチプレクサ 。 したがって、1つのポートで静的、API、Webソケット、SSE、ストリーミングビデオ、大きなファイルを組み合わせたり、異なるドメインやマルチドメインサイトで複数のアプリケーションを処理したりできます。
  4. サンドボックスを使用して環境から隔離された仮想マシンを適用する原理。 各アプリケーションには、ライブラリとデータがロードされる独自のコンテキスト（スコープ）があります。 アプリケーションは、初期化とファイナライズ、データモデルとデータベース構造、構成とインストール（最初の開始）、更新、移行など、あらゆる種類のハンドラーのアーキテクチャ上の場所を提供します。
  5. アプリケーションとシステムコードの分離。 一般に、アプリケーション内の抽象化レイヤー（高レベルと低レベル）を分離することは、同じ抽象レイヤー内のモデルとビューでロジックを共有するよりもはるかに重要です（それらを混在させるとさらに効率的です）。
  6. ディレクトリツリーでの継承と再定義を使用したURLのファイルシステムへのマッピング 。 ただし、ディレクトリ構造による自動ルーティングに加えて、プロセッサをメモリにプログラムで直接追加し、手動でルーティングを規定する機能があります。
  7. コードの簡潔さ （以下の例を参照）は、ほとんどの場合に必要なすべてを処理する開発済みの組み込みAPIのおかげで実現され、プロジェクト間で拡張および再利用できます。 また、可視性ゾーンを操作する特別なスタイルと、便利なサイズの論理部分を持つファイルに分割するコードにより、簡潔さが促進されます。


応用分野



IASは、いくつかの種類のアプリケーションを作成するように設計されています。

  1. サーバーから再起動せずにクライアント上でAPIと動的ページ変更を行う1ページのWebアプリケーション。
  2. APIを介してページにある程度のダイナミクスを備えたマルチページWebアプリケーション（ロジックはクライアントとサーバーに分割されます）。
  3. 各イベントでページをリロードするマルチページアプリケーション（サーバー上のすべてのロジック）。
  4. 双方向のデータ交換またはサーバーからのイベントのストリームを伴うアプリケーション、対話型アプリケーション（通常、これはオプション1および2のアドオンです）。
  5. ネイティブモバイルおよびウィンドウアプリケーションのサーバーアクセス用のネットワークAPI。


サポートされている複数のAPI作成メソッド

  • RPC API-URLが一連のパラメーターでネットワークメソッドを識別する場合、呼び出しの順序は重要であり、呼び出し間の状態はクライアントとサーバーの両方に保存されます。
  • REST API、URLがリソースを識別する場合、限られた数の操作をリソースに対して実行できます（たとえば、HTTP動詞やCRUD）。リクエストはアトミックであり、呼び出しの順序に違いはなく、呼び出し間に状態はありません。
  • イベントバス：クライアントとサーバー間のオブジェクトの状態を通知または同期するために使用されるWebSocketsまたはSSE（サーバー送信イベント）を介したクライアントサーバー相互作用の単方向または双方向ストリーム。
  • または混合方法。


ハンドラー



IASのミドルウェアの類似物はハンドラー（ハンドラー）です。これは2つのパラメーター（クライアントとコールバック）を持ち、別のファイル（エクスポート元）にある非同期関数です。 コールバックが呼び出されると、IASアプリケーションサーバーは処理が完了したことを検出します。 関数がタイムアウトより長いコールバックを引き起こさない場合、IASはHTTPステータス408（リクエストタイムアウト）を返し、ログの問題を修正します。 ハンドラーが呼び出されたときに例外が発生した場合、IASはクライアントへの回答を想定し、エラーをキャッチして、破損またはリークしたデータ構造を持つサンドボックスを削除および再作成するまで、最適な方法で作業を復元します。

アプリケーション構造



サーバーコードはプロセッサに限定されず、アプリケーションには、ドメインモデル、多くのプロセッサで使用される特殊なライブラリとユーティリティ、およびロジックとデータを配置するためのその他の「場所」も含まれます。 IASで実行されるすべてのアプリケーションは/ applicationsディレクトリにあり、次の構造を持っています。

  • / app-サイトのルートに対応するハンドラーのルートディレクトリ hostname



    hostname



  • / config-アプリケーション構成
  • / doc-ドキュメントとサポート資料、
  • / files-ユーザーがアップロードしたファイルを配置するためのディレクトリ（サブディレクトリの2レベルまたは3レベルのシステムは、多数のファイルでファイルシステムを過負荷にしないように自動的に構築されます）
  • / init-サーバーの起動時に開始する初期化コード（ここでは、プログラムでハンドラーを作成したり、メモリ内のデータ構造を準備したり、tcpポートを開いたりすることができます）。
  • / lib-起動時に（ただし初期化後に）ロードされ、アプリケーションコード全体からアクセス可能なライブラリとユーティリティのディレクトリ。
  • / log-ログのディレクトリ（構成にこのアプリケーションの個別のログがある場合）
  • / model-ドメインモデルのディレクトリ（起動時にロードされますが、初期化後）、
  • / setup-このディレクトリにあるjsファイルは、IASまたはサーバー全体を再起動するときに1回だけ実行されます。この場所は、操作中にアプリケーションのライフサイクル全体を維持するために必要な更新または移行スクリプトに必要です。
  • / tasks-スケジュールされたタスクを配置するためのディレクトリ。2つのタイプがあります。定期的に実行する、特定の間隔で実行する、または指定した時間に実行する、
  • / tmp-一時ファイルのディレクトリ。


次のディレクトリには次のディレクトリが表示されます（ 問題＃195 ）：

  • / client-クライアント部分のソースコード配布のディレクトリ、
  • / static-収集されたクライアントは/static



    に配置され、いくつかの一般的なツールをコレクターとして使用できます。

おわりに



Impress（IAS）は活発に開発されており、4から7のマイナーバージョンが毎週表示されます。 現在、バージョン0.1.195が関連しており、バージョン0.2が近づいています。このバージョンでは、アプリケーション構造と基本APIを修正し、すべての0.2.xバージョンの後方互換性を確認しています。 0.2.xでは、最適化の問題とバグ修正のみを扱い、機能の拡張は、これが0.2.xに基づくアプリケーションの再設計を必要としない場合にのみ可能になります。 0.3.xブランチでは、すべての主要な革新と実験が並行して導入されます。 私は誰もがプロジェクトを開発するように招待します。そして、私にとっては、少なくとも関連性がある限り、コードを維持することを約束します。 バージョン1.0は、独立した開発者がコードを完全にサポートできることを理解したときにのみ表示されます。 ドキュメントは現在準備中です。これまでは、構造とアーキテクチャが頻繁に変更されるという事実により不可能でしたが、バージョン0.2に備えてリンクを公開します。 これに先立って、IASをデフォルトアプリケーションとしてインストールした例を使用して、IASをより詳細に調べることができます。
