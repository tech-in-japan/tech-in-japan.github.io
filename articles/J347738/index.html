1.はじめに



モバイルデバイス（ MU ）のソフトウェア部分の構造を引き続き検討します。 今日は、デバイスのGPTパーティションメモリについて説明します。 これについてコメントを書く代わりに、MBRとGPTの構造を調べることで、このことについて書くことを勧められました。 著者をscったり訂正したりするのではなく、 MUのバイアスで上記の出版物を補います。









したがって、GPT（GUID Partition Table）は次のとおりです。



  • まず、MU のメモリマークアップスキーム （ GPTスキーム ）の名前 。
  • 次に、この回路が物理的に配置されているメモリパーティション （ GPTパーティション ）の名前 。
  • まあ、GPTパーティションのイメージファイルの名前 （ GPTファイル ）。


MBRとGPTの 2つのメモリマークアップスキームがあります。 各回路は、それぞれMBRまたはGPTと呼ばれる個別のメモリセクションに含まれています。



MBRレイアウトのレイアウトは[1]にあります。GPTは、メモリレイアウトを記述するためのもう1つの形式、GUID（GUIDパーティションテーブル）です。 BIOSの代わりにパーティションをロードするために使用されるUEFI標準であるEFI（Extensible Firmware Interface）の一部です。



別の形式に切り替えると、MBR形式の最も重大な欠点である少数のパーティションがなくなりました。 これらのパラメーターが32ビットの数値で記述されているためにパーティションの長さとそのオフセットに制限があるMBRに 4つのレコードのみが配置された場合、128個のパーティションをGPTに配置できます。 さらに、それらのパラメーターはすでに64ビットの数値システムで説明されています...



古いブート標準（BIOS）との互換性のため、およびGPTメモリパーティション記述テーブル自体を保護するために、 GPTパーティションはMBRで始まります。これは、1つのパーティションのみ、つまりすべてのMUメモリを記述します。 MBR自体は、現在「保護」（保護）と呼ばれています。 PMBR また、アドレス0x0000-0x01FF （512バイト）のメモリの最初のセクタにあります。 [ オフセット]フィールドはGPTヘッダーの始まりを示し、パーティションサイズはデバイスメモリ全体の長さに等しく設定されます。 パーティションの種類は0xEE （GPTパーティション）です。



GPTセクションとMUのメモリ全体を保護する本質は次のとおりです。 そのようなセクションがMBRマークアップ図を操作するためのツールを開く場合、マークアップ説明セクションのみで構成される空の未割り当てメモリが表示されます。 したがって、彼はこの記憶で何もすることに成功しません。

2. GPT構造



GPTセクションは、 PMBR 、ヘッダー、およびメモリパーティションを記述するセクションテーブルで構成されます。



PMBRの直後、つまり アドレス0x0200から、長さが0x5C（92）バイトのGPTヘッダーがありますが、セクター全体（512バイト）を占めています。 同じMUでの外観は次のとおりです。



GPTヘッダー




図2 GPTヘッダー



次の構造になっています。 



=============================================================================== |  |  |  |  | |  |  |  |  | |==========|=================|=======|==========================================| | 0x00 | Magic | 8 |   | | 0x08 | Revision | 4 |  GPT- | | 0x0C | HeaderSize | 4 |     | | 0x10 | HeaderCRC32 | 4 | CRC32  | | 0x14 | Reserved1 | 4 | ;   0 | | 0x18 | MyLBA | 8 | LBA  GPT- | | 0x20 | AlternateLBA | 8 | LBA  GPT- | | 0x28 | FirstUsableLBA | 8 | LBA     | | 0x30 | LastUsableLBA | 8 | LBA     | | 0x38 | DiskGUID | 16 | GUID  () | | 0x48 | PartitionsLBA | 8 | LBA   GPT | | 0x50 | NumberParts | 4 |     | | 0x54 | PartitionSize | 4 |       | | 0x58 | PartitionsCRC32 | 4 | CRC32     | | 0x5C | Reserved2 | * |  | ===============================================================================


Magicフィールドには、文字列「EFI PART」-45h 46h 49h 20h 50h 41h 52h 54hが含まれます。これは、図2で青緑色で強調表示され、GPTセクションを一意に識別します。



緑色で強調表示されている[ リビジョン]フィールドには、GPTパーティション構造のバージョン番号（0x00010000）が含まれています。 たとえば、GPTバージョン1.0の場合、00h 00h 01h 00hである必要があります



オレンジ色で強調表示されたHeaderSizeフィールドには、ヘッダーのサイズがバイト単位で表示されます。 これまでのところ、これは92バイトを意味する0x0000005Chです。



ピンクで強調表示されているHeaderCRC32フィールドには、CRC32アルゴリズムを使用して計算されたヘッダーチェックサム（0x09BE8E1F）が含まれています。 計算を直接実行する場合、92バイトのみが考慮され、計算前にこのフィールドに0が入力されます。



次のフィールドReserved1は選択されていません。予約済みであり、0が含まれています。



濃い青色で強調表示されているMyLBAフィールドには、プライマリGPTセクションの配置ブロックのオフセットが含まれています（0x00000001）。



AlternateLBAフィールドには、バックアップGPTセクションの配置ブロックのオフセットが含まれます。 示されていません。



赤で強調表示されているFirstUsableLBAフィールドには、使用が許可されている最初のメモリセクタの配置ブロックのオフセットが含まれています（0x00000022）。 プライマリGPTパーティションが占める最後のセクターのLBA + 1。



LastUsableLBAフィールドには、使用が許可されている最後のメモリセクタの位置オフセットが含まれます。 次のように計算されます。バックアップGPTパーティションの最初のセクターのLBA-1.指定されていません。



青で強調表示されたDiskGUIDフィールドには、ファームウェアGUIDが含まれています。



赤で強調表示されているPartitionsLBAフィールドには、GPTの開始のオフセットが含まれています。 プライマリは常に2で、バックアップはLastUsableLBAです。



明るい緑色で強調表示されたNumberPartsフィールドには、セクションの説明テーブルのサイズが含まれています。 パーティションエントリの数（0x00000018）。



ピンクで強調表示されたPartitionSizeフィールドには、1つのセクションレコードのサイズがバイト単位で含まれています。 UEFI仕様によると、レコードサイズは128バイトまたは0x00000080に固定されています。



赤で強調表示されたPartitionsCRCフィールドには、CRC32アルゴリズムを使用して計算されたパーティション記述テーブル（0x93D54D33）のチェックサムが含まれています。 計算では、PartitionsLBAからFirstUsableLBAまでのすべてのバイトが考慮されます。



Reserved2フィールドには、バックアップフィールドが含まれています。 0からセクターの終わりまで、つまり 512バイトのセクターの場合は420バイト。



アドレス0x400から始まる見出しの直後に、マークアップセクション自体を含む各メモリセクションに関するレコードを含むセクション記述テーブルがあります。 これは次のようなものです。

  ==================================================================================== |  | , |  |  | |  |  |  |  | |===========|========|=====================|=========================================| | 0 | 0x00 | 16 | PartitionTypeGUID | GUID   | | 16 | 0x10 | 16 | UniquePartitionGUID | GUID  | | 32 | 0x20 | 8 | StartingLBA | LBA     | | 40 | 0x24 | 8 | EndingLBA |  LBA    | | 48 | 0x30 | 8 | Attributes |   | | 56 | 0x38 | 72 | PartitionName |     | ====================================================================================


赤で強調表示されているPartitionTypeGUIDフィールドには、このパーティションにデータを保存するために使用されるファイルシステムを定義するパーティションタイプGUIDが含まれています。 各ファイルシステムは、それを一意に識別する独自のGUIDを受け取ります。 UEFI標準では、次のパーティションタイプGUIDのみが厳密に定義されています。 



  ============================================================================= |  GUID |  | |======================================|======================================| | 00000000-0000-0000-0000-000000000000 |   | | C12A7328-F81F-11d2-BA4B-00A0C93EC93B |   UEFI | | 024DEE41-33E7-11d3-9D69-0008C781F39F | ,   MBR | =============================================================================


OSベンダーは、独自のパーティションタイプGUIDを生成してそれらを識別する必要があります。 いくつかの有名なGUIDは[3]にあります。



メモリまたはイメージファイルに書き込む場合、GUID値は異なる順序で書き込まれます。 たとえば、EFIシステムパーティションのGUIDは次のとおりです：C12A7328-F81F-11D2-BA4B-00A0C93EC93B。 GUIDのスペルのバイト順はリトルエンディアンであり、バイトは最初の3ブロックにのみ逆方向に書き込まれます。 上記のGUIDの場合、パーティションテーブルのエントリは次のようになります。 



 28 73 2A C1 1F F8 D2 11 BA 4B 00 A0 C9 3E C9 3B


青で強調表示されているUniquePartitionGUIDフィールドには、セクションのGUIDが含まれています。 これはセクションの一意の識別子であるため、セクションが作成されるたびに作成されます。



緑色で強調表示されているStartingLBAフィールドには、セクションの最初のセクター（0x00020000）によるブロック単位のオフセットが含まれています。



オレンジで強調表示されたEndingLBAフィールドには、セクションの最後のセクター（0x0003FFFF）によるオフセットが含まれています。 パーティションのサイズ（PartitionSize）は、式によって決定されます 



  PartitionSize = EndingLBA - StartingLBA + 1


紫色で強調表示されている属性フィールドには、セクションの属性（フラグ）が含まれています。 8バイト（64ビット）のフラグは次のように割り当てられます。 ビット0〜47（48個）は、パーティションタイプの一般的な属性用に予約されており、残りの16ビット（48〜63）は特定のパーティションを表します。



これらのビットの簡単な説明を次に示します。 



  ================================================ |  |  | |======|=========================================| | 0 |   | | 1 | EFI     | | 2-47 |    UEFI | | ---- | | | 60 | Read-only (  ) | | 61 |     | | 62 | Hidden () | | 63 |    | ================================================


青で強調表示されたPartitionNameフィールドには、UTF-16LEエンコーディングで表された、36文字以下の終端ゼロを持つテキスト行を含むセクションラベル（「モデム」）が含まれます。



GPTがすべてであるということは既に知っていますが、これがGPTセクションの標準構造です。 変更があることがわかります...



なぜなら IntelベースのMUで作業しているときに、異なる構造のGPTセクションを発見しました。Intelの ようなGPTセクションと呼びました。



3. Intel GPTパーティションとは何ですか？



Intelチップに基づく新しいLenovoモバイルデバイス（MU）のファームウェアにはGptメモリレイアウトがありますが、 Gptファイル自体の構造は[1]で説明されている標準のものとは異なります。 これは、たとえばYOGA BOOK YB1-X90デバイスに適用されます。



標準のIntel構造体と比較して、タイプは最大まで削減されます。



  • 最初、すべての標準値を除くすべてのパーティションのガイド、チェックサム、オフセット、または空、たとえば、セクションパラメータレコードのサイズ、プライマリテーブルとバックアップテーブルの配置。
  • セクションパラメーターの説明から属性が削除され、最初と最後のブロックのオフセットがセクションサイズに置き換えられました。


なぜなら このタイプのGpt-ファイルの構造は標準構造とは異なるため、そのファームウェアにはIntelの特別なフラッシャーであるPhoneFlashTool_5.3.2.0を使用する必要があります。



GPTセクションは、標準のセクションと同様に、次のもので構成されます。



  • ヘッダー
  • パーティションパラメータの説明表 。

ヘッダーのサイズはわずか12（ 0x0C ）バイトです（標準バージョンでは512に対して）。

青色でマークされたマジックフィールドには、Intel GPTセクションの画像を識別する番号0x6A8B0DA1が含まれています。



赤でマークされたStartLbaフィールドには、最初のメモリパーティションの配置のオフセット値が含まれています。



緑色でマークされた[ 数値]フィールドには、メモリパーティションの合計数が含まれます。 セクション記述テーブルのエントリの数。 テーブル自体はヘッダーの直後にあります。



3.2。 新しいGPTパーティション記述エントリ構造



セクションの説明テーブルには、各メモリセクションのパラメータを含むエントリが含まれています。 レコードの数はセクションの数と等しく、テーブルの終わりには何もマークされていません。



各レコードのサイズは108（ 0x6C ）バイトですが、 UEFIでは128バイトが含まれています。 新しいセクションの説明エントリは次のようになります。





図6セクションの説明の記録



セクションのパラメータを説明する次のフィールドが含まれます。 



  ============================================================= | № |  |  |  |   | | / |  |  |  | | |=============================================================| | 1 | 0x00 | Size | 4 |   | | 2 | 0x04 | Label | 72 |   | | 3 | 0x4C | GuidType | 16 | GUID   | | 3 | 0x5C | GuidPartition | 16 | GUID  | =============================================================


図6で青色で強調表示されている[ サイズ]フィールドには、パーティションのサイズがMBで表されています（1024 * 1024 = 1048576バイト）。



図6で赤く強調表示された[ ラベル]フィールドには、セクションラベルが含まれています。 UTF-16エンコーディングで表現されたセクション名。



図 6で緑色で強調表示されているGuidTypeフィールドには、セクションタイプGUIDが含まれています。



図 6で黄色で強調表示されているGuidPartitionフィールドには、セクション自体のGUIDが含まれています。
