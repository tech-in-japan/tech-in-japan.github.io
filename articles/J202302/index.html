このシリーズの第4回では、「デスクトップ」アプリケーションの開発に戻り、 LiveBindingをバインドするオブジェクトのテクノロジーをより詳細に理解しようとします。 このメカニズムは普遍的であり、モバイルアプリケーションとデスクトップアプリケーションの両方に適しています。 さらに、このメカニズムはVCLでも機能します。 LiveBindingの知識は、古典的なプロジェクトから始めるのが最適です。



まず最初に、同じタイプのいくつかのタスクを実行する必要がありますが、その作業は日常的な作業に思えるかもしれません。 リストを編集するためのフォーム、つまり測定単位のリストと製品のリストを作成することです。 これらのフォームはメインフォームメニューから呼び出されますが、これも作成する必要があります。



そのため、データモジュールに2つのTFDTableコンポーネントを配置し、tblUnitテーブルとtblFoodstuffテーブルを使用するように構成します（tblRecipeテーブルを使用するためのデータセットと同様）。 コンポーネントをダブルクリックしてフィールドエディターを開き、すべてのフィールドを追加します（コンテキストメニュー項目の[すべてのフィールドを追加]）。



ほとんどすべての実装で、データセットコンポーネントはTDataSetクラスから継承され、いくつかのフィールドセットで機能することを理解することが重要です。 これは、クラスのインスタンス、 TFieldの子孫 （TIntegerField、TStringFieldなど）を指します。 一連のフィールドをデザインタイム（デザインタイム）に作成し、「オブジェクトインスペクター」を使用してフィールドのプロパティを設定できます。 TFDTableを考慮する場合、テーブルの物理フィールドに対応するフィールドに加えて、計算（計算）フィールドまたは置換（ルックアップ）フィールドなど、他のタイプのフィールドを作成できます。 これらのフィールドの値は、プログラム中に動的に計算され、データソースから取得されません。 以下では、そのようなフィールドを作成する例を検討します。



同じ場合、フィールドセットがデザインモードで作成されていない場合、プログラム操作中にデータセットを開くと、フィールドセットが自動的に作成されます。



以下は、uDM.dfmデータモジュールフォームファイルのスニペットで、データセットの1つとその関連フィールドの説明が含まれています。 フォームのコンテキストメニューで[テキストとして表示]を選択すると、フォームのテキスト表現を取得できます。 通常のフォーム送信モードに戻るには、「フォームとして表示」をクリックします。 



object FDTUnits: TFDTable IndexFieldNames = 'Id' Connection = FDConnection1 UpdateOptions.UpdateTableName = 'tblUnit' TableName = 'tblUnit' Left = 112 Top = 208 object FDTUnitsId: TFDAutoIncField FieldName = 'Id' Origin = 'Id' ProviderFlags = [pfInWhere, pfInKey] ReadOnly = True end object FDTUnitsUnitName: TStringField FieldName = 'UnitName' Origin = 'UnitName' Required = True Size = 25 end object FDTUnitsAbbr: TStringField FieldName = 'Abbr' Origin = 'Abbr' Required = True Size = 7 end end




作成されたデータセットは、プログラムの起動時に開く必要があります。 ConnectToDBプロシージャを次のように変更します。 



 function TDM.ConnectToDB: Boolean; begin try FDConnection1.Connected := True; FDTRecipe.Open; FDTUnits.Open; FDTFoodstuff.Open; except end; Result := FDConnection1.Connected; end;




アプリケーション開発の次のステップは、測定単位のテーブル（リスト）を表示および編集するための新しいフォームを作成することです。 当然、これはFM HD形式になります。 このフォームがメインフォームの中央に表示されるように、PositionFormCenterとしてPositionプロパティの値を設定します。



implementationセクションの使用済みモジュールのリストにデータモジュールを追加します。 



 implementation {$R *.fmx} uses uDM;




TGrid、TPanelコンポーネントをフォームに配置します。 パネルに、図に示すように、ボタン、入力フィールド、およびTLabelラベルを配置します。 ボタンの唯一の目的は、フォームを閉じることです。 ModalResultプロパティはmrOkに設定されます。



視覚的なコンポーネントの名前は、明確にするためだけにデフォルトで保持されます。 もちろん、作業中のプロジェクトでは、名前を変更することをお勧めします。







LiveBindingエディターを使用して、ビジュアルコンポーネントをデータソースに関連付けます。 最も一般的な用語では、 LiveBindingは、さまざまなオブジェクトのプロパティをバインドできるメカニズムですが、コンパイルされたハードリンクを使用せず、柔軟な式に基づいています。 LiveBindingメカニズムは、オブザーバーパターンに基づいています。 そのため、LiveBindingを使用してオブジェクトAの一部のプロパティを変更すると、オブジェクトBのプロパティが変更される可能性があります。オブジェクト間の接続は単方向および双方向にできます。 このアプローチは、データセットのデータを表示するのに非常に便利です。 実際、VCL標準アプローチでは、テーブルからのデータを表示するために特別なDB指向（DB対応）ビジュアルコンポーネントが必要でしたが、LiveBindingでは、この目的で最も一般的なコントロール（「コントロール」）を使用できます。 さらに、ビジュアルコンポーネントは、LiveBindingを使用するための特別な準備を必要としません。 このアプローチの利点は明らかです。 このプロセスをもう少し注意深く、もう少し詳しく見てみましょう。



そのため、 LiveBindingデザイナを開き、データセットに対応する要素FDTUnitsを見つけます。 データモジュールの要素がLiveBindingデザイナに表示されない場合は、更新する必要があります（コンテキストメニューで[デザイナの更新]を選択します）。 Edit1入力フィールドのTextプロパティの値を、データセットのUnitNameフィールドの値に接続してみましょう。 通常のドラッグアンドドロップ（drag-n-drop）を使用します。







本当に何が起こったのですか？



デザイナーが接続を視覚的に表示したという事実に加えて、2つの新しい非視覚的なコンポーネントがフォームに表示されました。 TBindSourceDBコンポーネントは、DataSetプロパティを使用して、元のデータセット（この場合はDM.FDTUnits）と通信します。 ビジュアルコンポーネントのプロパティとフィールドの関係が確立されるとすぐに、LiveBindingデザイナで、DM.FDTUnitsオブジェクトが新しいBindSourceDB1オブジェクトにネストされ、このオブジェクトがフォームに表示されます。







このオブジェクトが持つDataSetプロパティの値を推測するのは簡単です。 「クラシックVCL開発」の観点で考える人にとって、 TBindSourceDBを何らかのTDataSourceアナログと考えるのが最も簡単です。 このコンポーネントをフォームにスローして、DataSetプロパティの値を指定するだけです。 そして、コントロールを「アタッチ」します。 今では特別なDBコントロールではなく、視覚的なコンポーネントになります。 したがって、コンポーネントのプロパティと、データセットのデータを表示（および場合によっては変更）する方法を指定する必要があります。 方法を見てみましょう。



BindSourceDB1に加えて、別のオブジェクトがフォームに表示されます-BindingsList1。 オブジェクト間のすべての接続のリポジトリです。 これらの接続にはさまざまなタイプがあります。 図からわかるように、この場合、 TLinkControlToField型のリンクがBindingsListに追加されました。







たとえば、接続を単方向ではなく単方向にするために、そのプロパティを編集できます。







新しい接続を追加しようとすると、実際には「すべてのものとすべて」を接続できる、かなり多くの異なる種類の接続があることがわかります。







次に、GridとBindNavigatorをデータセットに接続します。 BindNavigatorコンポーネントの場合は、DataSourceプロパティを設定するだけです。 ただし、BindingsList1リンクリストに新しいリンクは作成されません。 ただし、グリッドをバインドすると、反対に、タイプTLinkGridToDataSourceBindSourceDBのオブジェクトが作成されます。







オブジェクトインスペクターでは、フィールドのリストを編集できます。 これを行うには、Columnsプロパティを呼び出し、フィールドエディターで[すべてのフィールドを追加]ボタンをクリックします。 ところで、このエディターはTDBGridのフィールドエディターに非常に似ています。 各フィールドについて、プロパティ、特にフィールド幅とタイトルテキストを編集できます。 さらに、新しいフィールドを作成し、テーブル列に表示する式（CustomFormatプロパティ）を設定できます。 したがって、IDと略語の組み合わせを表示するには、次の式を指定できます。 



 DataSet.FieldByName('Id').AsString +" - "+DataSet.FieldByName('Abbr').AsString




上記からわかるように、LiveBindingメカニズムはデータを表示するための十分な機会を提供します。 次に、LiveBindingデザイナを使用すると、視覚コンポーネントをデータソースにリンクするプロセスを視覚化および簡素化できます。

結論として、LiveBindingを使用する別のトリックを検討してください。 デザイナーで、DM要素のAbbrフィールドをアクティブにします。 FDTUnitsおよびコンテキストメニューで[新しいコントロールへのリンク...]を選択します。 表示されるリストで、TEditを選択します。







[OK]ボタンをクリックすると、新しいTEditコントロールとTLabelラベルが「接着」されてフォームに配置されます。 このオブジェクトを目的の位置に配置し、ラベルのTextプロパティに目的の値を設定するだけです。







メニューを使用して、メインフォームからリスト編集フォームの呼び出しを整理します。 メインフォームにTMainMenuコンポーネントを配置します。 ダブルクリックしてメニューエディターを開きます。 図に示すように、いくつかのポイントを作成します。







メニュー項目「測定単位」に対して、次のOnClickイベントハンドラーを定義します。 



 procedure TfMain.aSprUnitExecute(Sender: TObject); begin fSprUnits.ShowModal; end;




当然、リスト形式のモジュールはメイン形式に接続する必要があります。 



 implementation {$R *.fmx} uses uAddRecipe, uSprUnit;
