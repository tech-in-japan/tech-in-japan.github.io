ストレージポリシーベースの管理（SPBM）。 ポリシーベースのストレージ管理



ポリシーベースのストレージ管理テクノロジーにより、VM（仮想マシン）のストレージ要件を設定し、これらのVMの多数のサービスのパフォーマンスを保証できます。 このアプローチを使用した大規模なインフラストラクチャの状況では、VMにストレージを提供するプロセスがより効率的で便利になります。



管理者は、データセンターに必要なポリシーのセットを作成および編集します。必要なポリシーを特定のVMにバインドした後、仮想インフラストラクチャ自体が、ポリシーの要件を満たすデータセンターストレージのセットから限られたストアのセットを選択します。 さらに、ポリシーは、対応するルールが設定され、I / Oフィルター（VAIO）がインフラストラクチャに展開されている場合、VMのデータキャッシングやレプリケーションなどのサービスの実行を保証できます。



ポリシールールのカテゴリ



どのSPBMポリシーが形成されるかに基づいたルールは、3つのカテゴリに分類されます。



一般的なルール -データサービスに基づいており、ストレージのタイプに依存せず、あらゆるタイプのストレージに共通です。 この場合、サービスは、VAIOに基づいて開発されたI / Oフィルター、つまり個々のVMのレベルでのキャッシュとデータ複製によって実行されるデータフローの操作として理解されます。



一般的なルールは、ストレージとその特性の選択に関する要件を確立するものではなく、特定のサービス（I / Oフィルター）がVMに適用されることを保証します。 一般ルールの適用は、vCenterおよびESXiホスト/クラスターにインストールする必要がある適切なI / Oフィルターでのみ使用可能になります。



ストレージ固有のデータサービスに基づくルール -Virtual SANやVVolなど、特定のタイプのストレージを提供するデータサービスによって定義されます。 重要な点は、この特定のストレージに固有のサービスです。



ストレージプロバイダー（ストレージプロバイダー、VASAプロバイダーでもあります）は、ストレージの特性と機能（サービス）に関する情報を収集し、vCenterに提供します。 したがって、すべてのサービスに関する情報を含む特定のストレージプロファイルが形成され、それに応じて、提供できるサービスの品質に関する情報が含まれます。



ポリシーは、一定レベルのサービスとデータサービスの適用を保証するルールに基づいて形成されます。 特定のVMに関連付けられているため、このようなポリシーは、特定のレベルのサービスと特性（容量、パフォーマンス、可用性、フォールトトレランスなど）を備えたストレージへの配置を保証します。



タグに基づくルール -さまざまなリポジトリを手動でマークできるショートカットに基づいて作成されます。



ラベルまたはタグは、主に2つの場合に使用されます。



•VMFSやNFSなどの非プロバイダーボールト（VASA）にタグを付ける場合。

•ストレージ施設の地理的位置や管理上の所属など、VASAで指定されていない機能を示すため。



その結果、リポジトリに割り当てられたタグを使用して、ポリシーのルールを作成できます。



ポリシーの形成と適用



ポリシーには、一般的なルールと、ストレージサービスおよびラベルに基づくルールを含むいくつかのルールセットを含めることができます。 ポリシー内のルールセット間の関係は論理「OR」であり、セット内のルール間の関係は論理「AND」です。 したがって、VMをホストするために、このVMに関連付けられたポリシーセットの1つのルールに完全に準拠するリポジトリの1つを選択できます。 また、このポリシーで指定された一般的なルールがこのVMに適用されます。



VAIO-IOフィルタリング用のvSphere API。 I / Oフィルタリング用のAPIセット



IOフィルタリング用のvSphere API（VAIO）は、I / Oフィルターと呼ばれるサードパーティソフトウェアコンポーネントを作成するように設計されています。 これらのフィルターは、ESXiホストにインストールでき、VMとその仮想ディスク間のI / O要求を処理することにより、VMデータで追加の操作を実行できます。



I / Oフィルターは、使用されるストレージインフラストラクチャに関係なく、VMとその仮想ディスク間のI / O（データ）パスに直接アクセスします。



I / Oフィルターは、サードパーティの開発者が作成し、vCenterサーバーとESXiホストにコンポーネントがインストールされているパッケージとして配布する必要があります。 ESXiホストにフィルターをインストールした後、vCenterサーバーはこれらのホストのI / Oフィルターストレージプロバイダー（I / OフィルターのVASAプロバイダー）を自動的に構成および登録します。 これらのVASAプロバイダーを使用すると、I / OフィルターメカニズムをSPBMに統合できます。その結果、フィルターによって提供されるデータサービスの設定は、ポリシー（VMストレージポリシーインターフェイスの「一般」ルール）を通じて管理できるようになります。 したがって、必要なポリシーをVMディスクにバインドした後、それらはポリシーで指定されたI / Oフィルターアクションの対象になります。



理論的には、サードパーティの開発者は任意のI / Oフィルターを作成できます。 ただし、vSphere 6.2の現在のリリースでは、2つのカテゴリのフィルターのみがサポートされています。



•キャッシュフィルター-ローカルフラッシュドライブを使用して、仮想ディスク上のVMデータをキャッシュできます。

•複製フィルター-特定のVMのすべての書き込み操作を外部ストレージ（ホストまたはクラスター）に複製します。

1つのESXiホストに、1つのカテゴリ（キャッシングなど）であっても、複数のフィルターをインストールできます。



I / Oフィルターコンポーネント：



•VAIO Filter Framework-I / Oフィルター用のVMwareフレームワーク。

•I / O Filter Plug-In-ゲストOSと仮想VMディスクの間のデータを傍受およびフィルタリングするサードパーティソフトウェアコンポーネント。

•CIMプロバイダーは、I / Oフィルタープラグインを構成および管理するためのオプションのサードパーティコンポーネントです。

•vSphere Web Clientプラグイン-vSphere Web ClientからCIMプロバイダーを管理するためのオプションのサードパーティコンポーネント。

I / O Filter Daemon-サードパーティ開発者のオプションのコンポーネントは、ホスト間のネットワーク相互作用を確立する可能性のある特定のフィルターと相互作用するためのサービスとして使用できます。









VMの各仮想マシン実行可能（VMX）コンポーネントには、そのVMの仮想ディスクに関連付けられたI / Oフィルタープラグインを管理するフィルターフレームワークが含まれています。 Filter Frameworkは、VMとその仮想ディスク間でI / O要求が発生するとフィルターを呼び出すため、VMのディスク操作はすべてフィルターによってインターセプトされます。



フィルターは、最初のレプリケーションフィルター、キャッシュフィルターなど、特定の順序で連続して実行されます。 1つの仮想ディスクに複数のフィルターを接続できますが、特定のカテゴリーの1つのみです。



I / O要求が仮想ディスクに関連付けられたすべてのフィルターによって処理された後、この要求は宛先（VMまたはそのディスク）に送信されます。



フィルターの操作によって発生したエラーは、フィルターが接続されているVMの操作に影響を与える可能性がありますが、ホストや他のVMの操作には影響しません。



I / Oフィルター用のローカルフラッシュキャッシュ



キャッシュされたI / Oフィルターを使用するには、*フラッシュドライブ（SSD）をESXiホストにインストールすることにより形成されるローカルフラッシュリソースが必要です。 インストールされたフラッシュドライブは、ホスト内の共有リソース（VFFSでフォーマットされた仮想フラッシュリソース（仮想フラッシュリソース））に結合されます。 VFFS-フラッシュドライブを操作するために最適化されたファイルシステムは、ベースFS vSphere-VMFSの派生物です。



I / O処理中に、フィルターはVMのキャッシュを形成し、VFFSリソースに配置します。









仮想フラッシュリソースの機能：



•1つのホストでは、1つの仮想フラッシュリソース（VFFSパーティション）のみをホストできます。

•仮想フラッシュリソースはキャッシュ専用であり、VMやその他のタスクをホストするために使用することはできません。

•仮想フラッシュリソースは、ローカルフラッシュドライブでのみ使用できます。

•仮想フラッシュリソースは、以下を含むさまざまなデバイスから形成できます。 さまざまなインターフェース（SATA、SAS、PCIe、NVMe）

•フラッシュドライブは完全に仮想フラッシュリソースを使用できるので、Virtual SANやVMFSなどの他の目的に使用することはできません。



注：



仮想フラッシュリソースは、ホストスワップキャッシュおよびフラッシュ読み取りキャッシュメカニズムでも使用でき、1つのVFFSリソースを両方のメカニズムで共有できます。



ホストスワップキャッシュ-VMスワップファイルをホストするように設計されています。 ESXiホストにVMに十分な物理メモリ（RAM）がない場合、VMに必要な不足しているRAMはスワップファイルとしてアレイに転送されます。 ホストスワップキャッシュを使用すると、これらのファイルをVMディスクのメインストレージよりも高速にリソースに移動できます。



フラッシュ読み取りキャッシュ-VAIOフィルターをキャッシュするよりも前の、VM用のデータキャッシュテクノロジは、読み取りキャッシュのみをサポートします。 このテクノロジーはvSphere 5.5に登場し、組み込みのVMwareツールです。 その操作のために、各VMの予約済みキャッシュのサイズを示す必要があります。 テクノロジーの有効性は負荷のタイプに依存します。読み取り操作のフローが大きい場合、パフォーマンスの大幅な向上が期待できます。



* I / OフィルターのキャッシュにVFFSパーティションを使用する必要がないという前提があります; VAIOフィルターの開発者は、VMキャッシュストレージを別の、おそらくより効率的な方法で実装できます。
