TypeScriptについて少し


TSプログラムは、* .tsおよび* .d.tsファイルのセットで構成されています。 これらは、C ++の* .cppおよび* .hファイルの類似物として表すことができます。 つまり 前者には実際のコードが含まれ、後者には同じ名前の実装ファイルが提供するインターフェースが記述されています。 TSヘッダーのみで開発する場合、d.tsファイルは必要ありません。 また、d.tsは実行時にコンパイルおよび使用されません。 ただし、既存のJSコードを記述するにはd.tsファイルが不可欠です。 なぜなら TSは完全にJSに変換され、任意のJSコードを完全に使用できますが、コンパイラはJSで使用される型、変数、および関数について知る必要があります。 D.tsファイルは、この説明の目的を果たすだけです。 これ以上詳しくは述べませんが、すべてTS仕様にあります。



一般に、example.tsをコンパイルするとき、次のファイルを作成できます。

  • example.d.ts-他のプロジェクトで使用するヘッダーファイル
  • example.js-ランタイムで使用するJS
  • example.map.js-デバッグ用のソースマップ

CodePlexで公開するすべてのソース： tsasyncmodulesexampleapp.codeplex.com 。

実践では、共有ライブラリのd.tsファイルは、プロジェクトと同等に最適であることが示されています。 後でテストを記述するときに非常に役立ちますが、別の時間について、および複数のプロジェクトが同じライブラリを使用する場合についても同様です。



ここでは、Visual Studioを使用して、または使用せずにTypeScriptプロジェクトをコンパイルする方法について少し説明する必要があります。 最初のケースでは、スタジオは開発者のためにすべてを行い、ファイルへのリンクがない場合、ファイルを1つずつコンパイラーに渡します。 つまり すべてのtsファイルは常にコンパイルされます。 コマンドラインからプロジェクトをビルドする場合、他のすべてのファイルへのリンクを持つファイルでアセンブリを開始する必要があります。 通常、////>を含むプロジェクトルートにBuild.d.tsファイルを作成します。 アセンブルする必要があるすべてのプロジェクトファイルにリンクし、コンソールを介してコンパイラに渡します。 この方法により、TSコンパイラの設定をスタジオの現在のプラグインよりもはるかに柔軟に管理することができます。これは、今後必要になるでしょう。



Node.exeと完全なTS配布キットが追加され、プロジェクトに慣れる際にスタジオや他のインストール済みソフトウェアに縛られないようになりました。 スタジオとの簡単な起動と統合のためのCmdファイル。後で説明します。

モジュラーロードの構成



すべてが非常に簡単です。 デフォルトで受け取ったクライアントプロジェクトのApp.tsファイルに、次のコードを追加します。 



export class App { public static Main(): void { alert('Main'); } }




プロジェクトのルートにRequireJSConfig.tsファイルを作成します。 



 /// <reference path="../Lib/require.d.ts" /> /// <reference path="App.ts" /> require(["App"], function(App: any) { App.App.Main(); });




Default.htm： 



 <!DOCTYPE html> <html lang="en"> <head> <meta charset="utf-8" /> <title>TSAsyncModulesExampleApp</title> <link rel="stylesheet" href="app.css" type="text/css" /> <script src="js/jquery-1.10.1.min.js"></script> <script src="js/underscore-min.js"></script> <script src="js/backbone.js"></script> <script src="js/require.js" data-main="RequireJSConfig"></script> </head> <body> </body> </html>




アプリケーションを起動します。







おめでとうございます、非同期にロード可能なモジュールを備えたアプリケーションが手に入りました。



私たちが行ったことをさらに詳しく見てみましょう。



まず、TSでファイルがどのようにコンパイルされるかを詳しく説明します。 ファイルにエクスポートディレクティブが含まれている場合、これはコンパイラ設定に応じて、CommonJSまたはAMDモジュールに一意にコンパイルされることを意味します。 VSのデフォルトでは、コンパイルはAMD形式で行われます。これは、Require.jsを使用するコンテキストでは便利です。 TSは、AMDモジュールをラップするための「気味の悪い」手動コードを記述する必要を完全に排除し、依存関係の正しいインストールを処理します。 これは、App.tsで観察される動作です。 



 define(["require", "exports"], function(require, exports) { var App = (function () { function App() { } App.Main = function () { alert('Main'); }; return App; })(); exports.App = App; }); //@ sourceMappingURL=App.js.map




RequireJSConfig.tsにはエクスポートディレクティブが含まれておらず、通常の「フラット」JSにコンパイルされます。 



 /// <reference path="../Lib/require.d.ts" /> /// <reference path="App.ts" /> require(["App"], function (App) { App.App.Main(); }); //@ sourceMappingURL=RequireJSConfig.js.map




コメントは、デバッグ目的でのみコードに残ります。 デバッグアプリケーションビルドモードがあります。 リリース構成では、すべてのコメントが消去されます。



しかし、私たちのバー...モジュールに戻ります。 何が起こった：



  1. default.htmをロードしました
  2. Cssおよび静的に設定されたjsファイルがロードされました。これは、主観的な意見では、非同期にロードする意味がありません。 それらは私たちが作成するほとんどすべてのモジュールに必要です。
  3. アイテム2のjsの中で、require.jsがロードされました
  4. Require.jsは、data-main = "RequireJSConfig"属性の値を読み取り、対応するJSファイルを読み込みました。これは開始ファイルとして扱われます。
  5. RequireJSConfigでは、グローバルコンテキストで最初と最後にrequireメソッドを使用します。 さらに、すべてのモジュール呼び出しは他のモジュールからのものでなければなりません。
  6. require関数では、Appモジュール（最初のパラメーター）を読み込んだ後、同じ名前の変数として読み込んだモジュールを転送するコールバックを呼び出す必要があると言います。 ここでは、良心との取引に進み、TSでそれを類型化しません。 この場所では、TSのモジュール化のイデオロギーと、非同期スクリプト読み込みのマネージャーとしてのRequire.jsの特定の実装との間に矛盾があります。 詳細は以下をご覧ください。
  7. Require.jsはAppモジュールをロードします。 モジュールの命名規則の詳細については、Require.jsのドキュメントをご覧ください。 つまり、ルートからのパスを示します。これは、デフォルトで使用するサイトのルートとは異なるファイル拡張子を指定せずに指定できます。 次に、RequireJSはhead.appendChild（）を使用して、各依存ファイルをスクリプトタグとしてロードします。 必要なモジュールは最後にロードされます、つまり 依存関係の後、つまり、すべての依存関係が常にロードされることを常に確認できることを意味します。 Require.jsとTSは、この問題で完全に一貫した方法で機能します。 TSの構文とコンパイルプロセスは、このシナリオに特に適合しています。
  8. requireメソッドに渡されたコールバックは、AppモジュールのAppクラスであるMainの静的メソッドを呼び出します。
  9. 呼び出されたアラート（「メイン」）;




requireメソッドのコールバックパラメータは、このコンテキストではTSコンパイラに知られていないため、タイプされていません。 類型化するには、次の形式のTSコンストラクトを使用する必要があります。 



 import App = require('App');




この設計は、他のモジュールをロードするためにモジュールのコンテキストでのみ使用でき、AMD / CommonJSモジュールラッパーに対応する依存関係を形成します。 ラッパーなしの通常のフラットコード、つまり 矛盾が生じます。 つまり、TSはモジュールコード生成の分野でRequire.jsと完全に一貫していますが、最初のモジュール読み込みスクリプトをサポートしていません。 したがって、この状況で残っているのは、バニラJSのスタイルで古き良きアプローチを使用することです。



追加モジュールをダウンロードする


そこで、Mainメソッドと呼ばれる最初のモジュールをロードすると、アプリケーションが動作を開始しました。 教育TKによると、任意の数のモジュールをロードできる必要があります。 ただし、ダウンロードを開始する前に、TSのモジュールとは何かを理解する必要があります。



  • まず、アプリケーションの自律的な部分であり、関数と変数を外部にエクスポートするモジュールパターンを実装するコードです。 さらに、実装全体が外部から隠されています。 実際、これは最も純粋な形式のカプセル化です。
  • 第二に、モジュールをロードした後、変数に割り当てられます。 つまり オブジェクトとして使用できます。 実際、モジュールはJSの観点からはオブジェクトです。
  • 第三に、モジュールはファイルです。
  • 第4に、モジュールはTSの名前空間です。 純粋な名前空間を実装する他のモジュール内で、エクスポートできないモジュールが宣言されている場合があります。 つまり モジュールのネストレベルの数に制限はありませんが、トップレベルモジュールのみがAMD / CommonJSモジュールに変換され、スタンドアロンファイルに持ち込まれます。




実際、モジュールは一度にすべてです。 モジュールの使用もあいまいです。 確かに言えることは、モジュールがアプリケーションを自律的で再利用可能なコンポーネントに分割し、それが私たちの目標を完全に満たすということです。



新しいファイルFramework / Events.tsを作成します。これは、将来、抽象イベントモデルの実装の開始点として使用します。 



 export interface IEventPublisher { On(); Off(); Trigger(); } export class EventPublisherBase implements IEventPublisher { On() { } Off() { } Trigger() { } constructor() { alert('EventPublisher'); } }




これまでのところ、すべてのメソッドは単なるスタブです。 仕事の原則は私たちにとって重要です。



App.tsの変更： 



 import Events = require('Framework/Events'); export class App { public static Main(): void { var e = new Events.EventPublisherBase(); } }




モジュール読み込みディレクティブがあります： 



 import Events = require('Framework/Events');




パスはルートから示されます。 上記のように、モジュールは変数に割り当てられます。 次に、アラートを宣言したコンストラクターで、Events名前空間からEventPublisherBaseクラスの新しいインスタンスを作成します。



この場合、Events変数は強く型付けされています。 コンパイラは、ロード可能なモジュールのタイプを厳密に監視します。 コンパイラーはモジュール内の型制御のための参照ディレクティブを必要としないが、正しいコンパイルのためにそれを必要とすることに注意すべきです。 つまり コンパイラはそれなしでは依存関係を追跡できません。 現在、VSはコンパイラの代わりに、明示的なファイル名に置き換えて作業を行います。 これを解決するには、前述のBuild.d.tsファイルを作成し、それに関するコンパイルを開始します。 



 /// <reference path="Framework/Events.ts" /> /// <reference path="App.ts" /> /// <reference path="RequireJSConfig.ts" />




コンパイルに使用できるサンプルcmdファイル（Build-Client.cmd）は、ソリューションのルートに添付されています。



実際、非同期読み込みに関してはそれだけです。 すべてが非常にシンプルでシンプルです。
