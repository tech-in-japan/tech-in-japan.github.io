Wargaming Public API



Wargaming.net Public APIは、ゲームコンテンツ、プレーヤー統計、百科事典データなどを含むWargaming.netプロジェクトへのアクセスを提供する、公開されているプログラミングインターフェイスのセットです。



この記事では、パブリックAPIを作成し、内部コンポーネントの相互作用を整理するための前提条件、APIの構築方法と動作方法、および近い将来開催される開発者コンテストについて少し説明します。



パブリックAPI開発の歴史



Wargaming APIを作成するための前提条件に関するセクションを書く前に、相関関係を探し、現象としてのAPIの進化を評価するために、パブリックAPIの基本的な開発方法に関する理解を更新することにしました。 この情報は私にとって非常に興味深いものだったので、簡単に絞ることができます。



  • 2000年2月7日 -初日からsalesforce.comが開始され、ユーザーにアプリケーションのかなり完全で機能的なAPIを提供します。 実際、これらのユーザーは、サービスとしてのソフトウェア形式をユーザーに提供した最初の人物です。
  • 2000年11月20日 -eBayアプリケーションプログラムインターフェイスがeBay開発者プログラムの一部として開始されました。 事実、かなりの数の異なるアプリケーションが何らかの形でeBayページから情報を引き出したため、これは必要なステップでした。
  • 2002 -Googleが検索用のAPIを発表。 理由は同じです-開発者は、アプリケーションでGoogle検索ツールを積極的に使用し、最終的にはそれを実行しようとしています。 以前のAPIプロバイダーとの大きな違いは、そのようなAPIを効果的に収益化する方法があまり明確ではないことです。 ただし、当時Googleと競合していたAltaVistaは、テクニカルディレクターによると、「需要を待って」APIを急ぐことはありませんでした。
  • 2003-2006-社会サービスは積極的に発展し始めています。 API del.icio.us、Flickr、Facebook、Twitterなどが表示されます。
  • 2006年6月29日 -Google Maps APIの登場。 理由も同じです。Googleマップのジオデータが急速に人気を博したため、内部のJavaScript APIが歯車によって即座に解体され、誰でも好きなように使用されるようになりました。
  • 2006-クラウドサービスの開発、およびそれに応じたAPI：Amazon S3、Amazon EC2など
  • 2007-クラウドベースの音声サービス用のAPIをユーザーに提供する、製品としてのAPIプラットフォームであるTwilioが発売されました。
  • 2009年3月 -Foursquareが開始され、すでに11月に資金調達ラウンドの後、Foursquare APIが開始されています。
  • 2010年10月 -Instagramのローンチ。 すでに12月に、サードパーティの開発者の1人がiPhoneアプリケーションで使用されていたAPIを解体し、非公式のInstagram APIをリリースしました。 2011年1月、Instagramは海賊版APIを閉鎖し、2月にリリースした独自の公式APIに取り組んでいると発表しました。










したがって、最初はAPIの開発の主な要因はオンライン取引プラットフォームでしたが、その後は落ち着きました。 さらに、バトンはソーシャルサービスAPI、クラウドAPI、モバイルAPIで取り上げられました。



2005年以降、さまざまなAPIの数が一般的に爆発的に増加しています。 これを確認するProgrammableWebの写真を次に示します。











Wargaming Public APIを作成するための前提条件



前のセクションの歴史的背景からわかるように、多くの場合、APIを作成する理由はサードパーティの開発者の活動です。 彼らは興味のあるデータを取得するために、持っているツールを使用します。 ただし、APIが不足していることで特に停止するわけではありません。 Webは、常にターゲットサイトを解析して適切な情報を取得できるように編成されています。 はい、不快です。 はい、ページレイアウトが変更されるとすぐにすべてが壊れます。 とにかく、多くの困難。 しかし、それでも。



さらに、この図は、サードパーティの開発者だけに適しているわけではありません。 サイトからの情報をスキャンする多数のリクエストは、サーバーにかなりの負荷をかけるか、サイトを完全に「レイダウン」する可能性があります。 特に、アプリケーションは通常、クローラーではなくユーザーの行動に合わせて最適化されていると考える場合。



実際、これに関しては、Wargamingも例外ではありません。「タンク」が人気になるとすぐに、プレイヤーのプロフィール、氏族、統計、評価、グローバルマップ、百科事典などの情報がすぐに必要になりました。 もちろん、このデータをプルしようとしてサイトの解析が開始されました。 そしてWorld of Tanks Assistantが登場するとすぐに、苦しんでいる人々はすぐにAPIを攻撃しました。



これらすべてから結論を引き出すことは難しくありませんでした。情報はサードパーティの開発者が必要とするため、彼らはまだそれを取得します。 ただし、この場合、パブリックAPIを提供する方が適切です。 管理ははるかに簡単で、開発者にとってより安定した便利で信頼性の高いツールを提供します。少なくとも、このような使用シナリオ向けに最適化されたアルゴリズムにより、トラフィックを節約し、負荷を軽減できます。



さらに、パブリックAPIがある場合、アプリケーションはより機能的で、安定し、信頼できます。 エンドユーザーは満足しており、さらに多くの高品質のアプリケーションがゲームの周りに表示されます。これは開発者にとって重要です。



APIの背後にあるコンポーネントの内部相互作用



APIの仕組みを理解するには、情報フローが内部でどのように構築されているか、データがどのように配信されているか、その関連性などを確認すると役立ちます。 したがって、以下-コンポーネントの相互作用の内部組織について簡単に説明します。



ゲームを取り巻くインフラストラクチャ全体は、サービスモデルに従って構築されます。つまり、コンポーネントは比較的独立しており、APIセットを介して相互に作用します。 これはいくつかの要因によるものです。



  • 何かが落ちても、他のすべてが機能するはずです。
  • 多くのコンポーネントがあります（認証サービス、OpenIDプロバイダー、評価システム、クランサービス、第二次世界大戦、Wargamingリーグ、ゲームポータル、フォーラム、百科事典、CPPなど-約40個のみ）。
  • コンポーネントはさまざまなチームによって開発されます。
  • コンポーネントはすべて一緒にリリースされるわけではありません。
  • 更新が許可されていない場合の完全なダウンタイム。


つまり、大型システムをリリースするよりも、個別に開発された2つまたは3つのコンポーネントのバージョンをリリースする方がはるかに簡単であることは明らかです。 さらに、小さな機能要素としてのコンポーネント開発の管理が簡素化されます。



相互作用の原則



インフラストラクチャには2つの主要なデータストリームがあります。リモートAPIメソッドの呼び出しと、別のサブシステム（ゲームサーバー、他のコンポーネント）のイベントのサブスクライブです。



コンポーネントAPIは、ほとんどの場合HTTP REST APIです。 API呼び出しは、同期的および非同期的に応答を返すことができます。 非同期応答の場合、AMQPプロトコルに従ってメッセージブローカー（この場合はRabbitMQ）が使用されます。



イベントはRabbitMQでも追跡されます。 イベントの例は、プレーヤーのアカウントステータスが変更された後のエクスポートです。 さらに、たとえば、各戦闘の後、この戦闘をプレイしたプレイヤーの戦車関係書類がMQにエクスポートされます。 これにより、ゲームポータルは、最新のプレイヤーの統計情報を表示するようになります（ほとんどの場合）。 同時に、既にロードされているゲームサーバーは追加の負荷を受け取りません。



知覚を簡単にするために現実から少し離れると、Web上のコンポーネントの相互作用の原理を説明する次のような画像が得られます。







現実との違いは、サービスバスがないということです。 各コンポーネントは、必要な他のコンポーネントのAPIの場所を明確に認識しており、中間リンクなしで直接対話します。



イベントバスとほぼ同じ画像-各コンポーネントは、特定のメッセージを受信または公開するためにブローカーに接続する方法を正確に知っています。



Public APIは、このインフラストラクチャの他のコンポーネントとまったく同じコンポーネントです。 いくつかの違いは、おそらく構造設計パターンについて話す場合、比較的多数の他のサービスに関連付けられているという事実、つまり、エコシステム全体に対する一種のファサードであるという事実だけです。



もちろん、パブリックAPIは単なるプロキシサービスではなく、独自の追加ロジックがあり、一部のデータセットをキャッシュおよび保存して、応答速度と内部システムの負荷を最適化できます。 また、アクセス権を確認し、特定のアプリケーションからの呼び出しの制限を監視し、統計を収集する必要があります。 APIは公開されているため、多くのコードは外部契約を維持し、内部コンポーネントとそのAPIへの変更を外部コンシューマーから分離するためにのみ記述されています。

開発者パートナープログラム



パブリックAPIは、単なるデータおよびドキュメントサイトのAPIよりもはるかに大きなストーリーの一部です。 また、フィードバックと開発者サポートを収集するためのツールでもあります。



このプロジェクトの一環として、ゲームmodの開発者を集め、建設的な対話を確立したいと考えています。 現在、データAPIが開発者のニーズの多くをカバーしている場合、モダーの寿命ははるかに複雑です。 少しでも良くなることを願っています。 実際、プロジェクト全体がパブリックAPIだけでなく、Wargaming Developers Partner Programと呼ばれるのはそのためです。



競争



これで、プロジェクトサポートの次の段階に進む準備ができました。ゲームコミュニティとのやり取り、つまり開発者向けのオープンな競争です。



Wargaming Developers Contestは、ゲームの改造、アプリケーション、ゲームに近いサービスの開発者をサポートすることを目的としています。 コミュニティには野心的な人が多く、Wargamingがまだ見当たらない、または実装が遅れているアイデアを実装する完成したプロジェクトを作成することができます。 さまざまなプラットフォームや他のサービスのインターフェースとの統合、クレイジーなユーザーインターフェース、セカンドゲーム画面のコンセプトなど、あらゆることが可能です。



これらすべてのために、Wargaming Developers Contestの一環として、優れた賞金プールを用意しました。 また、数百万人の聴衆にユニークでオリジナルのプロジェクトを確実に紹介します。
