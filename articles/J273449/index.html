モデル変換（またはモデルが作成する変換を作成する変換）の概要





今日は、変換を作成する変換を記述します。 個人的に、これは、人々が夢の中で夢を見たクリストファー・ノーランの「始まり」を思い出させます。



これは、モデル指向開発サイクルの7番目の記事です。 6か月間、私はモデル駆動型開発の深刻な例を使って記事を書き込もうとしてきました。 しかし、毎回、非常に単純な例を整理するために、最初にテクノロジー全般について話す必要があることを理解しています。 そこで今回は、「Hello World」から記事を始めたいと思っただけで、その結果、この単純な例は多額の記事になりました。



はじめに



前回の記事では、モデル、メタモデル、モデルエディター、テキストおよびグラフィック表記について見てきました。 静的から動的に移行する時です。 今日は、いくつかのモデル変換ツールを紹介します。



クエリ/ビュー/変換（QVT）


QVTは、モデル変換を記述するドメイン固有の言語のファミリです。 OMG QVT仕様では、3つの言語について説明しています。



  • QVT Core（QVTc）は、宣言的なモデル変換言語です。 たとえば、UMLおよびERモデルがあります。 QVTコアを使用すると、UMLモデルのクラスがERモデルのエンティティに対応していること、UMLモデルのクラスプロパティがERモデルの属性に対応していることなどを「言う」ことができます。 このようなマッピングについて説明したので、一部のUMLモデルをERモデルに、または逆にUMLモデルのERモデルに変換できます（双方向マッピング）。
  • QVTリレーション（QVTr）はQVTコアとほぼ同じですが、構文糖衣のみで、変換をよりコンパクトに記述できます。
  • QVT運用（QVTo）-以前の2つの言語とは異なり、これはすでに必須であり、一方向のモデル変換のみを記述できます。


QVToは元々Borlandによって実装されていました。 そして2007-2008年から、Eclipseの一部として開発されてきました 。



QVTcおよびQVTrには、 Eclipseの実装もあります 。 Eclipse Neonのリリースまでに使用できるように計画されています。 しかし、それはあまり効果的ではありませんが、したがって、この記事ではそれについて詳しく説明しません。



QVTo変換は、Eclipseだけでなく、スタンドアロンJavaアプリケーションでも実行できることに注意することが非常に重要です。 たとえば、変換はWebサーバー側で正常に機能します。



QVTはOCLのアドオンと見なすことができます。 ナビゲーションには、QVTの演算子「。」および「->」を使用できます。 標準のOCLライブラリを使用できます。



ATL変換言語（ATL）


ATLは、QVTと並行して登場したハイブリッド（宣言的に必須）モデル変換言語です。 QVTによく似ていますが、いくつかの違いがあります。



短所：



  • OMG QVT仕様がある QVTとは異なり、標準化されていません。
  • 少し変わった構文ですが、これは主観的なものです。
  • OCLの独自の簡略化された実装。OMGOCL仕様に適合せず、 Eclipse OCLほど良くない場合があります。
  • エラーメッセージが常に消去されるとは限りません。
  • あまり便利ではないエディターとデバッガー。
  • 変換を開始するには、ANTスクリプトを作成する必要があります。


長所：



  • Eclipse OCL。QVTで使用されますが、優れていますが、比較的重いものです。 ATLは全体としてQVToよりも軽快に見えます。
  • 変換の改良のサポート。 新しいモデルを作成するのではなく、既存のモデルに小さな変更を加える必要がある場合は、ATLの方が便利です。 理論的には、QVTもそのような変換を記述できますが、ほとんどすべてが複雑です。
  • 高次変換のサポート。 ATLを使用すると、モデルと同様にATL変換を使用できます。 つまり これを使用すると、ATL変換を何かに変換したり、逆に何かからATL変換を作成したりできます。 理論的には、QVTでこれを行うことができますが、実際には、すべてがそれほど単純ではありません。


変身


変身はモデル変換言語でもあります。 しかし、QVTやATLとは異なり、テキストではなく視覚的です。 カテゴリー理論に基づいています。 モデルは、ダブルコードカートスクエア（ダブルプッシュアウト-DPO）またはシングルコードカートスクエア（シングルプッシュアウト-SPO）の方法を使用して変換できるグラフと見なされます。



正直なところ、これは非常に興味深いトピックであり、次の記事のいずれかでおそらくカテゴリ理論に戻ります。



Henshinでは、Apache Giraphで変換を実行できます。これにより、非常に大きなモデルを変換できます。 これを行ったことはありませんが、主にQVToを使用しています。



その他のツール


モデル変換のためのその他のツールもあります： Epsilon Transformation Language 、 EMorF 、 AGG（属性付きグラフ文法システム） 、 VIATRA（仮想自動モデル変換）など。 それらはすべて同様の問題を解決し、同様の原則に基づいています。 QVTo、ATL、またはHenshinがあなたに合わない場合、多くの選択肢があります。



それとは別に、XSLTに注目する価値があります。 変革に関しては、多くの人がそれを思い出します。 私自身、2000年代前半、XSLTの出現の夜明けに、XSLTとそのサイトのPHPエンジンについて書きました。 彼は激しくブレーキをかけ、当時は考えられなかった量のRAMを要求しました。 しかし、これはポイントではなく、技術は悪くありません。XSLTにいくつかの簡単な変換を書いています。



XSLTの最初の問題は、XMLドキュメントの変換に焦点を合わせていることです。 もちろん、モデルをXMLドキュメントとしてシリアル化し、XSLTにフィードすることができます。 さらに、対応するOMG XMI仕様もあります（以下、XMIファイルの例があります）。 しかし、最初に、モデルをXMLファイルに入れるさまざまな方法があります。 たとえば、属性はXML属性またはXML要素としてシリアル化できます。 このような変動は、XSLT変換を非常に複雑にします。 第二に、モデルはツリーではなくグラフであり、通常、モデルではモデル間を含む水平接続でいっぱいです。 このようなリンクを使用して目的のオブジェクトを検索するのは、地獄のようです。 そして、UMLモデルでより多くのプロファイルとステレオタイプを思い出せば、これはすでに地獄の9番目のサークルです。



XSLTの2番目の問題はXML構文で、単に便利ではありません。



最後に、通常のモデル変換ツールには、XSLT作成者が夢にも思わなかったことがあります。 たとえば、QVToの変換ログと遅延リンクの解決。これについては、後で（デバッグセクションで）紹介します。



私の議論が十分に納得できない場合は、「「Hello World」変換を記述する変換を記述しています」セクションに、XMI形式のモデルの例があります。 以下は、このようなモデルを生成するATL変換です。 同様のXSLT変換がどのように見えるか想像してください。



ところで、Eclipse Modeling Frameworkを使用すると、XMLスキーマとEcoreメタモデルをリンクできます。 これにより、ほぼ任意のXMLファイルをEcoreモデルに、またはその逆に変換できます。 これにより、QVTo、ATL、およびXMLドキュメントを変換するための他のツールを使用できます。 おそらく、次のいずれかの記事でそのような例を検討します。

モデル指向の「Hello world」を作成します


真のモデル指向の「Hello world」にはテストモデルが必要ですが、以前の記事ではすでに十分なモデルを作成しました。 これに時間を浪費するのをやめて、QVに自分で作成させる： 



 modeltype ECORE 'strict' uses 'http://www.eclipse.org/emf/2002/Ecore'; transformation HelloWorld2(out o : ECORE); main() { object EPackage { name := 'World'; eClassifiers += object EClass { name := 'Alice'; }; eClassifiers += object EClass { name := 'Bob'; }; }; }


最初に（modeltype演算子を使用して）、作成するモデルのメタモデルを指定する必要があります。 次に、3行目で、変換に1つの出力モデルがあることを示します。 最後に、メインステートメントにコードを追加します。



直観的には、変換は2つのクラス（AliceとBob）を含むWorldパッケージを作成します。



クラスまたはプロパティの名前をCtrlでクリックした場合、それらが定義されているメタモデルを開きます。 また、数百または2つの異なるメタモデルがあります。

本当にモデル指向の「Hello world」を書く


次に、テストモデルを新しいモデルに変換します。 3行目では、変換に出力だけでなく入力Ecoreモデルも含まれていることを示します。 そして、いくつかのコードを追加します。 



 modeltype ECORE 'strict' uses 'http://www.eclipse.org/emf/2002/Ecore'; transformation HelloWorld3(in i : ECORE, out o : ECORE); main() { i.rootObjects()[EPackage]->toEPackage(); } mapping EPackage::toEPackage() : EPackage { name := 'Hello' + self.name; eClassifiers := self.eClassifiers->toEDataType(); } mapping EClassifier::toEDataType() : EDataType { name := 'Hello' + self.name; instanceClassName := 'some.ns.' + self.name + 'Class'; serializable := false; }


変換の本質は次のとおりです。 入力モデルですべてのルートパッケージを探し、それらをパッケージに変換しますが、他のパッケージも変換します。 名前にプレフィックス「Hello」を追加し、すべての分類子をデータ型に変換します。 データ型名も接頭辞「Hello」で始まります。ヒープの前に、さらにいくつかのプロパティを設定します。



この変換を開始するための構成を作成します。 以前に作成したモデルを入力として指定することを忘れないでください。出力モデルには、新しいファイル名を指定してください。

デバッグ


Eclipse QVToには、通常のものに加えて、すべてのマッピングの入力および出力オブジェクトを表示する変換デバッガーがあります。 実際、QVToエンジンはモデル変換の詳細なログを保持しています（右上の図）。 そして、これはデバッグだけでなく必要です。 まず、同じオブジェクトが再び表示されると、結果はこのログ（キャッシュ）から取得されます。 ところで、これにより、モデルをインクリメンタルに変換できます。 次に、コードでは、resolve操作を使用してログに明示的にアクセスできます。 また、遅延解決により、まだ存在しないジャーナルエントリを参照できます。 この雑誌はQVToの重要な機能の1つです。







ATL変換言語



truモデル指向の「Hello world」を書き換えます


ここで、ATLの最後の変換を書き換えます。 新しいプロジェクトを作成します（ファイル->新規->その他...-> ATLプロジェクト）。 新しい変換を作成します（ファイル->新規->その他...-> ATLファイル）。 ATLはQVToに非常に似ていることがわかります。 



 -- @nsURI Ecore = http://www.eclipse.org/emf/2002/Ecore module HelloWorld3; create OUT : Ecore from IN : Ecore; rule toEPackage { from package : Ecore!EPackage to newPackage : Ecore!EPackage ( name <- 'Hello' + package.name, eClassifiers <- package.eClassifiers ) } rule toEDataType { from classifier : Ecore!EClassifier to dataType : Ecore!EDataType ( name <- 'Hello' + classifier.name, instanceClassName <- 'some.ns.' + classifier.name + 'Class', serializable <- false ) }


注釈の最初の行で、使用するメタモデルを示しました。 ただし、オートコンプリートがエディターで機能するには、再検出が必要になる場合があります。



ご注意



INRIAはATLの作成に加わりました。率直に言って、これは構文で見ることができます:)彼らのCamlから、私も目を二重にしました。 優れたオフィスですが、彼らの言語は他の言語とは異なっており、フランス語は英語とは異なると感じています。 文字は似ているように見えますが、何かが間違っています。


変換を保存すると、asmファイルがプロジェクトに表示されます。 これは同じ変換ですが、ATL仮想マシンで実行するように設計された形式です。



ご注意



変換で何かを変更することもありますが、古い方法で動作します。 この場合、asmファイルを削除し、自動的に再作成されない場合は、変換に問題があります。 たとえば、変換ではdrop演算子を使用しましたが、これはATLコンパイラの新しいバージョンでのみサポートされており、特別なディレクティブを使用して明示的に有効にする必要があります。 同時に、エラーは発生せず、asmファイルは静かに再生成されませんでした。


[実行]-> [実行構成]を使用してこの変換を開始できます...ただし、この方法にはいくつかの制限があるため、すぐに実行するANTスクリプトを記述します。



プロジェクトに次の内容のbuild.xmlファイルを作成します。 



 <?xml version="1.0"?> <project name="HelloWorldATL"> <target name="HelloWorld3"> <!-- Loading metamodels --> <atl.loadModel name="Ecore" metamodel="MOF" nsURI="http://www.eclipse.org/emf/2002/Ecore" /> <!-- Loading models --> <atl.loadModel name="IN" metamodel="Ecore" path="output/MyModel2.xmi" /> <!-- Transformation --> <atl.launch path="HelloWorld3.atl"> <inmodel name="IN" model="IN" /> <outmodel name="OUT" model="OUT" metamodel="Ecore" /> </atl.launch> <!-- Saving models --> <atl.saveModel model="OUT" path="output/MyModel3.xmi" /> </target> </project>

build.xmlへのパスを指定します。

そして、最も重要なこととして、JREタブで「ワークスペースと同じJREで実行」を選択します。そうしないと、「名前が定義されていません」というエラーが表示されます。

絞り込み変換の作成


新しいモデルを作成するのではなく、既存のモデルに小さな変更を加える必要がある場合があります。 モデルからBobを削除し、残りを歓迎する変換を記述しましょう。 



 -- @atlcompiler atl2010 -- @nsURI Ecore = http://www.eclipse.org/emf/2002/Ecore module HelloWorld4; create OUT : Ecore refining IN : Ecore; rule sayHello { from s : Ecore!ENamedElement (s.name <> 'Bob') to t : Ecore!ENamedElement ( name <- 'Hello' + s.name ) } rule killBob { from s : Ecore!ENamedElement (s.name = 'Bob') to drop }


実行するスクリプト。 
 <?xml version="1.0"?> <project name="HelloWorldATL"> <target name="HelloWorld4"> <!-- Loading metamodels --> <atl.loadModel name="Ecore" metamodel="MOF" nsURI="http://www.eclipse.org/emf/2002/Ecore" /> <!-- Loading models --> <atl.loadModel name="IN" metamodel="Ecore" path="output/MyModel2.xmi" /> <!-- Transformation --> <atl.launch path="HelloWorld4.atl" refining="true"> <inoutmodel name="IN" model="IN" /> </atl.launch> <!-- Saving models --> <atl.saveModel model="IN" path="output/MyModel4.xmi" /> </target> </project>




モデルは別のファイルに保存されますが、構造では、変換で説明されている変更を除き、元のモデルを繰り返します。



私たちは、「Hello World」変換を作成する変換を作成しています


少し脳を除去する時が来たと思います。 私たちがすでに書いたモデルを作成または変更する変換。 変換が私たちのために書く変換を書くことは残っています。



実際、変換自体はモデルです。 これを確認するために、アリスとボブに個人的に挨拶する小さな変換を作成します。 



 -- @nsURI Ecore = http://www.eclipse.org/emf/2002/Ecore module HelloWorld5; create OUT : Ecore from IN : Ecore; rule SayHelloToAlice { from classifier : Ecore!EClassifier ( classifier.name = 'Alice' ) to datatype : Ecore!EDataType ( name <- 'Hello' + classifier.name ) } rule SayHelloToBob { from classifier : Ecore!EClassifier ( classifier.name = 'Bob' ) to datatype : Ecore!EDataType ( name <- 'Hello' + classifier.name ) }
