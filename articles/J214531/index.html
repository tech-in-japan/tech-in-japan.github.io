JavaScriptからAPK。 PhoneGapにうんざりしている人のためのAndroid開発の落とし穴。 JavaからJavaScriptへの橋渡し

こんにちは、Habra！



JavaScriptゲームが大好きで、すべてのプラットフォームに移植するためにコードを防弾化しようとしています。 6か月前、私はすでにAndroidアプリケーションのアセンブリについて書いていましたが、今日はこのトピックをより詳細に明らかにしたいと思います。



PhoneGapを放棄しなければならなかったことをすぐに警告します。 2つのプロジェクトでそれを使用した経験は私を混乱させました。 「Hello World」アプリケーションでは素晴らしい仕事をしますが、パイプライン化されたアセンブリでは、ニュアンスが連続して現れます。



PhoneGapが行かなかった理由：

1.最初は空です。 常に新しいモジュールを接続する必要があります。

2.多くのモジュールは曲がって書かれています。 彼らは多くを過剰に取るか、予期しない動作をします。 たとえば、SMSを送信するためのAndroid用の2つのモジュールのうち、1つは機能しませんでした。

3. EMEI電話の受信などの基本的な問題は解決されていません。 常に終了する必要があります。







私はまだPhoneGapの本質を理解していません。 最初は、1つのボタンが「うまくいく」ことを期待していましたが、実際には何もしません。 各プラットフォームの下に、SDKを配置する必要があります。 各タスクの下で、モジュールを検索してインストールします。 モジュール自体も制限されています。 彼らはプラットフォームの一部でのみ何かをすることができ、他のデバイスで必要な場合は、他のデバイスでこれを行うことができるモジュールを再度探す必要があります。 大量のゴミや不要なものがありますが、最小限のコストでビルドを収集したいと考えています。 これらすべての要因により、ネイティブに記述できます。 そして、ここで落とし穴が出てきます。



CocoonJSが行かなかった理由：

CocoonJSではあまり仕事をしていなかったので、特別な質問はありませんでした。 キャンバスを使用したビルドは本当に速く動作します。 しかし、一般的には、CocoonJSを使用する意味がわかりませんでした。 彼は支払われます。



他のプラットフォームおよび他のニュアンスのアセンブリについては、別の記事があります。このトピックまたはPhoneGapトピックに関するさらなる議論は、この範囲外です。

落とし穴： Androidを閉じる代わりに、アプリケーションを最小化します。 したがって、オーディオを使用する場合は、少なくともサウンドをオフにする必要があります。

 @Override public void onBackPressed() { vw.loadUrl("javascript: windowClose();"); MainActivity.this.finish(); } @Override public void onPause() { super.onPause(); vw.loadUrl("javascript: windowClose();"); MainActivity.this.finish(); } @Override public void onResume() { super.onResume(); vw.loadUrl("javascript: windowOpen();"); } @Override public void onDestroy() { super.onDestroy(); vw.loadUrl("javascript: windowClose();"); MainActivity.this.finish(); }

落とし穴：データを保存するlocalStorageは、アプリケーションを閉じた後に破棄されます。



2つの呼び出しの間にデータを保存するには、ネイティブのSharedPreferencesにデータを保存する必要があります。 保存して読み込むために2つのブリッジをロールしましょう： 



 @JavascriptInterface public void saveSomeThing(String message, String id) { if(numberDataForSave > Integer.parseInt(id)) return; numberDataForSave = Integer.parseInt(id); SharedPreferences preferences = getSharedPreferences("com.example.something", MODE_PRIVATE); SharedPreferences.Editor editor = preferences.edit(); editor.putString("somethingID", message); editor.commit(); } @JavascriptInterface public String loadSomeThing() { SharedPreferences preferences = getSharedPreferences("com.example.something", MODE_PRIVATE); String message = preferences.getString("somethingID", ""); return message; }


落とし穴：メソッドは非同期に動作します（または私には思えましたか？！）。 セーブを頻繁に呼び出すと、データの順序が間違っている可能性があります。



実際、バグは次のようになります。最後の行が保存されたのではなく、前の行が保存されました。 この問題を解決するために、JavaScriptとネイティブコードでデータに番号を付けます。 保存するメソッドには変数numberDataForSaveがあり、保存されたデータのインデックスをチェックします。 インデックスが最後に保存されたものより小さい場合、データは無視されます。

落とし穴： SIMカードなしでSMSを送信しました。 SMSは消えず、コールバックはtrueを返しました。



PhoneGapを使用している場合、Android向けのSMS送信モジュールが曲がって書かれている可能性があります（いずれにしても、私は個人的にこのような問題に遭遇しました）。 どの結果でもtrueを返します。 これを正常に実装するには、JavaからJavaScriptにSMSを送信するための橋を飛び越えましょう。 



 @JavascriptInterface public void sendSms(String phoneNumber, String message) { SmsManager sms = SmsManager.getDefault(); sms.sendTextMessage(phoneNumber, null, message, sendSmsPendingIntent, null); }


そして、MainActivityクラスにメソッドを追加します。 



 private PendingIntent registerSentSmsReceiver() { String SENT = "SMS_SENT"; PendingIntent sentPI = PendingIntent.getBroadcast(MainActivity.this, 0, new Intent(SENT), 0); sendSmsReceiver = new BroadcastReceiver() { public void onReceive(Context arg0, Intent arg1) { switch (getResultCode()) { case Activity.RESULT_OK: vw.loadUrl("javascript: smsSend(true);"); break; default: vw.loadUrl("javascript: smsSend(false);"); break; } } }; registerReceiver(sendSmsReceiver, new IntentFilter(SENT)); return sentPI; }


アプリケーションが新しいクラスでクラッシュするのを防ぐには、onDestroyを更新する必要があります。 



 @Override public void onDestroy() { super.onDestroy(); if (sendSmsReceiver != null) { unregisterReceiver(sendSmsReceiver); } }


設定を解除するときは、sendSmsReceiverを常にオフにする必要があります。



落とし穴： SamsungのAndroidでは、タッチイベントでWebViewが遅くなります。 さらに、彼は指を保持しながら描画を停止することさえできます。



このバグから逃れることはできません。 彼らはそれをやった。 これがCocoonJSプロモーションの理由の1つになりました。 必要に応じて、キャンバス要素を（すべて同じブリッジを使用して）ネイティブに書き換えることができますが、これはゴミです。 すべてを一度にJavaで記述する方が良いでしょう。 ただし、それでも、APKファイルを収集することは理にかなっています。 サムスンに加えて、まだ他のメーカーからの電話の束があり、そこにはすべてがそれほど悪くないことができます。 まあ、この点で同じTizenで、一般的に、おとぎ話。
