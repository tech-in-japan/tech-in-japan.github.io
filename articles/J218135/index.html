世界的に有名なテスターに​​なりたいですか？ 次に、すでにそのような高さに達している人のためにこれを行う方法を尋ねます。 4月20日に、モスクワは、世界で最も有名なソフトウェアテストの専門家の1人であるMichael Boltonのトレーニングを開催します。MichaelBoltonは 、この分野で25年の成功経験があります。











マイケルボルトンはモスクワ訪問を期待して、独占インタビューを行い、ソフトウェアテストの分野に存在する神話と、それを正しく理解することが重要である理由について話し、また、「 テスターのためのクリティカルシンキング 」トレーニングの参加者を待っているものに触れました20モスクワの4月。





テストの分野にはどのような神話が存在し、それらを払拭することが重要なのですか？



最も深刻な神話の1つは、テスターが品質に責任があるということです。 これはそうではありません。製品で働くすべての人が品質に責任を負います。 プロセスの他の参加者は品質を達成したが、テスターはそうではないと言うのはさらに正しいことです。 結局、ソースコード、カタログ、製品範囲、予算、責任者の雇用と解雇、顧客との契約などは管理していません。 テスターに​​は異なる役割があります。品質に対する潜在的な脅威を特定する責任があります。 これについては、 http ： //www.developsense.com/blog/2010/05/testers-get-out-of-the-quality-assurance-business/で詳しく説明しました。



別の神話は、テストがテストに関連するすべてであるという事実にあります。 私の同僚のJames BachとAdam Hodderは最近、この結論に対する非常に奇妙な批判を発表しました：James BachとAaron Hodderによる「テストはテストではありません：パフォーマンステストの文化への途上で」（この本はhttp：//www.testingcircusでダウンロードできます。 com / documents / TestingTrapeze-2014-February.pdf ）。



3番目の神話：製品テストとは、目的の結果が達成されたかどうかを「確実に」チェックすることです。 しかし、私たちにとってのテストは、特別な実験による製品の多目的な評価です。 ジェームズ・バッハと私はすでにこれについて書いています ： http : //www.satisfice.com/blog/archives/856 。



他にも多くの神話があります。 確かに、同時にそれらを過度に暴く問題が既にあります。 確かに、「神話に挑戦する」ためには、テスター、プログラマー、マネージャーはテストの詳細を深く理解し、実験に対する真剣な態度と批判的思考の熟練を必要とします。 そして、あなたはこれに取り組む必要があります。 私はこれを行う準備ができている人々を賞賛します。開発する製品について、どのように、そして何を知っているかを真剣に考えてください。



探索的テストについての神話もあります。 それらのいくつかについて説明しましたが、誰でも検索クエリ「探索的テストではないもの」を入力するだけで、これについて詳しく知ることができます（ロシア語で最初に試すことができるのは「テストではないもの」です）。



神話を払拭することは非常に重要です。なぜなら、テストでできることとできないことを適切に理解する必要があるからです。 結局のところ、テスターは自分の仕事、研究だけの品質を保証することができます。 そして、実際にどのようなテストが可能であるかを人々が理解し、使用する方法を学ぶまで、世界の単一の研究が製品を改善することはありません。



優秀なテスターの主な資質は何ですか？ 「反資質」とは何ですか？



テスターに​​とって多くの重要な資質がありますが、私が思うに、最も重要なことは好奇心です：「私がテストしているこのことは何ですか？ それは何のためですか？ 誰が使用しますか？ ユーザーを理解するための最良の方法は何であり、製品に対する彼の態度は何ですか？」



テストには素晴らしいことが1つあります。すべての種類のスキルが必要です。つまり、まったく異なる多数の人々が製品の作業に貴重な貢献をすることができます。 科学的かつ体系的な思考を持つ人、製品やテストについて書いて話すことができる人が必要です。 また、ソフトウェアは非常に異なる領域や状況の非常に異なる人々によって使用され、エラーは製品のまったく異なる要素にある可能性があるため、数学者、社会学者、心理学者、さらには母親も必要です。



テスターが批判的思考を持ち、それをどのように開発することが重要なのですか？



数年前、私は批判的な思考を「だまされないように考えられていることを考える」能力と定義しました。 ソフトウェア開発プロジェクトに携わっている人々は、何らかの解決策を開発したように見えるため、問題が解決したと確信することは非常に簡単ですが、多くの場合、それは自己欺ceptionにすぎません。 彼らは「うまくいく！」と言いますが、本当に「製品がある程度望ましい要件を満たしている」という意味です。 しかし、そのような解決策には弱点があるか、それ自体が実際の問題になる可能性があります。 これを回避するには、批判的思考が必要です。



批判的思考は一種の自己探求であり、私たちが知っていると思うことです。 そのためには、テスターが自分で開発できる習慣が必要です。 ステートメントに「...」、「...今まで」、「...そして...」、「...一方...」、「...または多分」を追加してみてください。 たとえば、誰かが「これはXです」と言ったら、「これはXである可能性がある」または「これはXである可能性がある」と言い換えます。 テスターであるということは、専門的に安全でないことを意味します。あなたの周りのすべての人が何かを100％確信しているとき、安全でないままでいることです。



テスターは何を学び、成功するために何をすべきか？



調査対象のソフトウェア製品は常に変化しているため、テスターの仕事の特異性は、定期的に新しい何かをテストすることです。 ただし、一般的な原則は同じままです。さまざまな方法で実験し、使用し、調査することで、物事の詳細を学習します。 テストは、調査、発見、調査、および報告で構成されるプロセスです。 これは変わりません。 同時に、私たちは常に自分の仕事を説明し、自分がやっていることと学んだことを他の人に説明することを学びます。 そして、特に理解しようとする専門家にとって、開発が必要なのはまさにこの方向だと思います。



特別なトレーニングや会議などに参加せずに、優れたテスターに​​なることは可能ですか？



トレーニングや会議に参加することなく、本やブログの投稿やインタビューを読むことなく、もちろん、良いテスターに​​なることができます。 しかし、これらはすべて、目標をより速く、より効率的に達成するのに役立ちます。そのため、人々に注意を払うことを強くお勧めします。



テスターのスキルの開発における重要なポイントは、テストスキルの開発と実用化であることを忘れてはなりません。 この分野の文化、実践、価値観に没頭してください。



さらに、人類学、数学、経済学、統計学、歴史、文学、言語学、分析、プログラミング、ジャーナリズムなど、テストの分野とは関係のない分野から多くの素晴らしいアイデアを借りることができます。勉強。 すでにそのようなトレーニングを開始している場合は、必ず続行してください。



テスターとして働いている、またはテスターとして働きたい人が、モスクワのトレーニングを訪問する必要があるのはなぜですか？



私はモスクワでの訓練をテスターの批判的思考に捧げる予定です。 その上で、人々を惑わす可能性のある多くのニュアンス、それらを識別するいくつかの方法、およびソフトウェアテストのスペシャリストがプロジェクトで作業している他の人々の自己欺andと混乱を回避する方法に関するヒントを共有します。 これは重要です。まず第一に、テスターは自分自身や他の人に製品の誤った理解、したがって誤ったプロセスに対して警告しなければならないからです。



さらに、4月18〜19日に開催されるSQA Days-15カンファレンスの一環としてモスクワに滞在中に、問題の性質についてお話しします。 テスターとしての主な仕事は、製品の価値を損なう重要な問題を見つけることです。 しかし、多くの専門家は、このコンセプトのかなり狭い意味でのみ製品の「欠陥」を探していると信じています-私はこれについて非常に心配しています。 私にとって、「問題」の概念はより適切です。なぜなら、問題は誰かが知覚できるものと誰かが望むものとの違いだからです。 言い換えれば、欠陥とは、ソフトウェアで作業するときに遭遇する可能性のあるいくつかのタイプの問題のみを記述するエラーまたは動作不能のコンポーネントです。 問題について考えることは、人とソフトウェアの関係について考えることを意味します。これは、広義の意味では、2人の人の間の明確な関係です。 テスターがこれを理解し、これを理解することが重要です。
