CoreOSとは何ですか？





CoreOSは、簡単かつ柔軟にスケーラブルなクラスターを構築するためのLinuxベースのオペレーティングシステムです。 CoreOSは最小限のディストリビューションです。 インストールISOイメージのボリュームはわずか136MBであり、インストールおよび起動後の最終マシンのメモリには、わずか114MBしか必要ありません。 CoreOSはChromeOSに基づいており、ChromeOS はGentooに基づいています。



実際、CoreOSは条件付きで次の部分に分割できます。



  • Systemd-クラスターマシン上のローカルサービスを管理します
  • Docker-サービスの分離を提供しますが、原則としてその使用は必要ありません
  • Etcd-クラスター構成の分散保存
  • フリート -分散サービス制御を提供します（systemdの「アドオン」）




CoreOSは、必要なクラスターマシンでsystemdサービスを実行し、そのステータスを監視し、構成を保存できます。







システム化


CoreOSは通常のsystemdを使用します。これは現在、多くのLinuxディストリビューションに含まれています。 これは、Linuxサービス用のローカル管理システムです。



Docker


DockerについてHabréで何度も書いています 。 つまり、DockerはホストLinuxオペレーティングシステムのカーネルを取得し、指定したファイルシステムのイメージをマウントし、仮想化環境にいるかのようにこのイメージ内で作業できるようにします。



Dockerは仮想化を使用しないことに注意してください。 プロセスを分離するには、 cgroupと名前空間を使用します 。これにより、KVMなどのツールがもたらすかなり深刻なオーバーヘッドを取り除くことができます。 もちろん、DockerはMacとWindowsの両方で使用できますが、この場合、ホストオペレーティングシステムはVirtualBoxまたはその他の仮想化システム内で動作する必要があります。Dockerが動作するにはLinuxカーネルが必要です（ Boot2Docker



Etcd


Etcdは、CoreOSクラスター内の各マシンで実行される分散Key-Valueリポジトリーであり、クラスター全体のほぼすべてのデータへの共有アクセスを提供します。 サービス設定、それらの現在のステータス、クラスター自体の構成などはetcd内に保存されます Etcdを使用すると、データを階層的に保存（ツリーのようなストレージ）、キーまたはディレクトリ全体への変更をサブスクライブ、キーおよびキーディレクトリのTTL値を設定（実際には「期限切れ」）、キーをアトミックに変更または削除、規則正しい方法で保存（一意のキューを実装できます） ） クラスタースケールで実行されているサービスの構成はetcdに格納されているため、リポジトリ内の対応するキーへの変更にサインアップするだけで、特定のサービスの開始と停止について知ることができます。



EtcdはREST HTTPインターフェースをサポートしているため、curlで十分に動作します。 CoreOSのコマンドラインから、 etcdctlユーティリティを使用して制御されます。



艦隊


フリートは、ローカルコンピュータからクラスタレベルにサービス管理を転送するsystemdのアドオンです。 Fleetは、systemdユニット（etcd内）の形式でサービスの構成を保存し、ローカルマシンに自動的に配信し、クラスターマシン上のサービスを開始、再起動（必要な場合）、停止します。 フリートは、特定のクラスターマシンのワークロードに基づいてサービスの起動を計画できます。 特定のサービスは特定のマシンなどでのみ実行する必要があると彼は言うことができます。



CoreOSは簡単です



CoreOSクラスターでhelloサービスを開始するには、次を実行するだけです。



1. SSHを介して任意のクラスターサーバーにログインします（すべて同等です）。

2.サービスの説明を含むテキストファイルを作成します。 たとえば、hello.service： 



[Unit] Description=My Service After=docker.service [Service] TimeoutStartSec=0 ExecStartPre=-/usr/bin/docker kill hello ExecStartPre=-/usr/bin/docker rm hello ExecStartPre=/usr/bin/docker pull busybox ExecStart=/usr/bin/docker run --name hello busybox /bin/sh -c "while true; do echo Hello World; sleep 1; done" ExecStop=/usr/bin/docker stop hello




3.次のコマンドを実行します。 



 fleetctl submit hello.service fleetctl load hello.service fleetctl start hello.service




4.以上です！ 最後のコマンドfleetを受け取ったので、この件について何も指示しなかったため、負荷の最も少ないクラスターマシンを特定し、ユニットファイルを取得し、選択したマシンに転送し、systemdがサービスを格納している場所に配置して、systemdに起動を依頼します。 そして、起動後、ステータスを監視します。 サービスが実行されていたマシンが予期せずクラッシュした場合、フリートは別のマシンでサービスを再起動します。



この例をより詳細に検討してください。



サービス構成ファイル： 



 [Unit] #    Description=My Service #       docker.service After=docker.service [Service] #       #    ,  docker         TimeoutStartSec=0 #       . #       ,      . ExecStartPre=-/usr/bin/docker kill hello ExecStartPre=-/usr/bin/docker rm hello #          docker    # fleet ,          ExecStartPre=/usr/bin/docker pull busybox #    .      Bash. ExecStart=/usr/bin/docker run --name hello busybox /bin/sh -c "while true; do echo Hello World; sleep 1; done" #     ExecStop=/usr/bin/docker stop hello




実行するコマンド： 



 #               fleetctl submit hello.service # ,             systemd    fleetctl load hello.service #   fleetctl start hello.service




サービスのステータスを表示するには、 fleetctl list-unit-filesコマンド（すべてのサービスの簡単な要約）またはfleetctl status hello.service（サービスに関する詳細情報）を使用します。



CoreOSをインストールする



CoreOSはさまざまな方法でインストールできます 。 ただし、実験することにした場合は、Vagrantからインストールすることをお勧めします。



1. Git 、 VirtualBox 、 Vagrantの最新バージョンをインストールします。

1a。 または、一部のgit cloneフォルダーでgithub.com/coreos/coreos-vagrant.git && cd coreos-vagrant

1b。 または、Gitをダウンロードしたくない場合は、 これをどこかで解凍します。



2.フォルダーに移動し、 user-data.sampleファイルの名前をuser-dataに変更し、 config.rb.sampleをconfig.rbに変更します。 discovery.etcd.io/newに移動し、表示された URLをバッファにコピーし、user-data内のディスカバリ指示の行のコメントを解除して、そこでURLを置き換えます。



3.フォルダに移動し、 そこまで浮浪者を行います

4.ベースイメージがダウンロードされるのを待ちます。

5. vagrant sshを実行します。

パッケージマネージャーとアップデート


CoreOSにはパッケージマネージャーがありません。 また、追加のソフトウェアをアセンブルまたはインストールするためのシステム（wgetおよびcurl :)を除く）。 別の方法として、 統合されたDockerツールボックスコンテナーを使用することをお勧めします。 有効にするには、コマンドラインから/ usr / bin / toolboxを実行するだけです 。 特権コンテナは、yumへのアクセスがあるFedoraで始まります。



CoreOSの更新は、アクティブ/パッシブパーティションシステムを使用してOS全体に対して実行されます。 実際、CoreOSには2つのシステムパーティションがあります。 常にアクティブなのはそのうちの1つだけです。 OS全体の更新は2番目にインストールされ、その後再起動が実行され、アクティブパーティションが変更されます。 オペレーティングシステムは、完全にのみ更新されます。 したがって、以前のバージョンのOSに簡単にロールバックできます。 私のように;）、すでにCoreOSのパッケージマネージャーをすぐに使用したい場合は、 開発者向けドキュメントをお読みください。



サービス発見


CoreOSの「トリック」は、クラスターマシン間でサービスを簡単に転送し、それ自体でサービスを停止および開始することです。 大まかに言って、これは、マシンXで最初に起動されたWebサーバーが、内部ネットワークで完全に異なるIPアドレスを持つマシンYに5分で簡単に到達できることを意味します。 もちろん、Fleetの特別な指示の助けを借りて、特定のマシンにサービスを簡単に「提供」できますが、...では、なぜCoreOSが必要なのですか？ :)これにより、サービスのIPアドレスを解決する問題が発生します。 基本的に、etcdを照会し、サービスが現在実行されているマシンを見つけるのは簡単です。 ただし、これは自動的には行われません。 サービスディスカバリーは非常に深刻な問題であり、 それを解決するために膨大な量のソフトウェアが提供されていますが 、 SkyDNSは私にとって最も魅力的です。 サイドキックユニットを使用してSkyDNSを使用できます。



コンテナの永続性とデータ自体


お気づきのとおり、CoreOSは実際には別のマシンに簡単に転送して実行できるステートレスDockerコンテナを中心に構築されています。 実際、nginxやphp-fpmなどのコンテナー化は非常に簡単です。 それらの構成ファイルは、コンテナーとして配布することもできます。 アプリケーションのソースコードでさえ、データボリューム（これはデータのみを含むdockerコンテナー）の形で配布でき、さまざまなサービスに接続できます。 データベースで何をしたいですか？



いくつかの方法があります。 まず、Flockerなどのサービス。データを使用するサービスを持つコンテナーの後にデータコンテナーを移行します。 次に、DBMSに似たサービスをクラスターモードで起動します。 NoSQLソリューションはこれを非常にうまく行うことができ、リレーショナルソリューションは著しく悪いですが、それはまだ可能です。 第三に、クラスター内でGlusterFSやCephなどの分散ファイルシステムを上げることができます。 ただし、そのようなソリューションはパフォーマンスの観点から評価する必要があります。 ここでこれらの問題について議論しました 。

CoreOSプロジェクトの一環として、代替のコンテナ化とロケットの標準が開発されています。 しかし...今のところ、どうやらDockerを使用したほうが良いようです。 さらに、CoreOSは将来Dockerを放棄する予定はありません。
