この記事は幅広い読者を対象としており、PHPとJavascriptの基本的な知識のみが必要です。 プログラミングを行っていて、略語APIに精通している場合は、その通りです。



当初、この記事では、GraphQLの特徴的な機能と実際に遭遇したRESTful APIの説明のみを想定していましたが、最終的にはいくつかのパートでボリュームチュートリアルになりました。



そしてすぐに、GraphQLをすべての病気の万能薬であり、RESTful APIキラーとは見なさないことを付け加えます。



私たちは誰ですか？



私たちはモバイルアプリケーションを開発する会社であり、原則として、これらのアプリケーションにはiOS（もちろん）、Android、およびWeb用のクライアントがあります。 個人的に、私はこの会社でPHPでサーバー側を書いています。



背景



それはすべて、1つのアプリケーションの開発を完了したという事実から始まりました（完了した時点で、アプリケーションの開発を終了することはできません。アプリケーションは途中で中断されたためです）。 幸いなことに、同社は開発者をテクノロジーの選択に制限しておらず（当然、合理的な範囲内で）、過去に発生した問題を回避するために特定の変更を加えることが決定されました。 結局のところ、あなたが知っているように、あなたがあなたがすることを続ければ、あなたはあなたが受け取るものを受け取り続けます。



RESTful APIの問題



RESTに大きな問題はありませんでしたが、私は定期的にそれらの1つに遭遇しました。 実際、私たちのプロジェクトの開発者は高い資格を持っているため、各開発者が特定のレベルで自分の分野の専門家であると考えているのはこのためです。 APIは、バックエンドとフロントエンドのスペシャリストをつなぐテクノロジーの薄いスレッドであり、開発方法に関する多くの論争の原因です。

バックエンド開発者が作成したいAPIメソッドは何ですか？



もちろん、サーバー側の開発者は、包括的で包括的なオブジェクトと可能な限り一貫性のある方法でAPIメソッドを記述したいと考えています。 例：



GET / api / user /：id 



{ id email firstName lastName createDate modifyDate lastVisitDate status }


GET / api / address /：id 



 { id userId street zip cityId createDate modifyDate status }


GET / api / city /：id 



 { id name }


...など そのようなアーキテクチャを書くことは、単純で高速であるだけでなく（これらすべてが何らかの足場になっていると言わない限り）、それはまた、アーキテクチャ上および美学的により美しく、より正確です（少なくとも開発者自身はそう考えています）。 最良の場合、バックエンドはネストされたオブジェクトに同意するため、addressIdの代わりに、ネストされたアドレスオブジェクトまたは（「1対多」接続の場合）アドレスの配列が返されます。



UI開発者が呼び出したいAPIメソッドは何ですか？



クライアント開発者は、生きている人々（アプリケーションのユーザー）とそのニーズに少し近づいています。その結果、同じデータの異なるセットが必要なアプリケーション内にいくつかの場所（多くを読む必要があります）があります。 したがって、彼は各機能要素がメソッドによって持つことを望んでいます：



GET / api / listOfUsersForMainScreen 



 [ { firstName lastName dateCreated city street } ... ]


...等々。 もちろん、そのような欲求は、作業を削減したいだけでなく、アプリケーションのパフォーマンスを改善したいという欲求によって完全に正当化されます。 まず、UIは3つではなく1つの呼び出しを行います（最初のユーザー、次に住所、次に市）。 第二に、この方法により、多くの（多くの場合かなりの）冗長データを取得する必要がなくなります。 同時に、dateCreatedは、データベースのフィールドから取得した元の形式ではなく、人間の形式で返すことが非常に望ましいです（そうでない場合は、Unix時間も変換する必要があります）。



私は証人であり、時には参加者であった紛争が生まれるのは、これに基づいています。 優れた開発者は、互いのニーズを部分的に満たすために妥協を求め、もちろん妥協を見つけます。 ただし、たとえば、フロントエンドが1つとバックエンドが2つある場合、単独でカリスマ性を最大限に高め、すべての外交スキルを使用して、アプリケーションのライフサイクル全体で1回呼び出される無意味なメソッドの記述を進める必要があります。

ステップ3.プロジェクト構造を作成します。



GraphQLサーバーの実装に関与する主な構造要素：



スキーマ -GraphQLサーバーのエンティティを格納するフレームワークのルートにあるディレクトリ：タイプと突然変異。 ディレクトリの名前と場所は重要ではありません。好きな名前を付けて別の名前空間に配置できます（たとえば、api /またはcomponents /）。



schema / QueryType.php、schema / MutationType.php- 「ルート」タイプ。



schema / Types.phpは、カスタムタイプを初期化する特定のアグリゲーターです。



スキーマ/ミューテーション -ミューテーションは、便宜上、別のディレクトリに保存することが望ましい。



さて、実際にはcontrollers / api / GraphqlController.phpがエントリーポイントです。 GraphQLサーバーへのすべての要求は、1つのエントリポイント-/ api / graphqlを通過します。 したがって、RESTful APIを同時に維持することを妨げるものは何もありません（その点に関して、おおまかに言えば、GraphQLサーバーは、GraphQLクエリをパラメーターとして受け入れるAPIメソッドの1つです）。



ステップ3.1 モデルのタイプを作成します。



新しいスキーマディレクトリとその中のモデルのクラスを作成します。

フィールドの説明には、次のパラメーターが関与します。



type -GraphQLフィールドタイプ（Type :: string（）、Type :: int（）など）。



description-説明（テキストのみ。スキームで使用されるため、リクエストのデバッグが容易になります）。



args-受け入れられる引数（連想配列、keyは引数の名前、valueはGraphQLタイプ）



resolve（$ root、$ args）は、フィールドの値を返す関数です。 引数：$ root-対応するActiveRecordのオブジェクト（この場合、models \ Cityオブジェクトが入ってくる）; $ argsは、引数の連想配列です（$ argsで説明）。

ステップ3.2 スキーマ/ Types.php



実際のところ、GraphQLスキーマには、同じ名前の複数の同一タイプを含めることはできません。 これは、Types.phpアグリゲーターが監視するために設計されたものです。 この名前は、GraphQLライブラリの標準クラスであるTypeとは異なるように見えるように、Typesによって意図的に正確に選択されています。 したがって、Type :: int（）、Type :: string（）、およびカスタムタイプTypes :: query（）、Types :: user（）などを使用して標準タイプにアクセスできます。



schema / Types.php： 



 <?php namespace app\schema; use GraphQL\Type\Definition\ObjectType; class Types { private static $query; private static $mutation; private static $user; private static $address; private static $city; public static function query() { return self::$query ?: (self::$query = new QueryType()); } public static function user() { return self::$user ?: (self::$user = new UserType()); } public static function address() { return self::$address ?: (self::$address = new AddressType()); } public static function city() { return self::$city ?: (self::$city = new CityType()); } }
