サイトへの登録。 長所、短所、落とし穴

昨日の似たような写真の検索サービスのプレゼンテーションと以前のイデオローグのプレゼンテーションの後、かなりの数のユーザビリティーやサービスビルダー（サイトではなく、サービスという言葉を意図的に使用）が1つの簡単なことを実現しなかったことに気付きました：ユーザーは登録せずにサービスを使用します。







各読者は、検索エンジンから共通のエンジンで構築されたフォーラムまたはサービスに切り替えるときに、希望するファイルをダウンロードするか、さらに悪いことに検索テキストを読むように登録するよう求められる状況に直面したと確信しています。



このシナリオを反抗的な例で見てみましょう。 Google経由でHL2 Alyx Nude Modを探しています。 Googleはいくつかのフォーラムへのリンクを提供します。 私はそこに行き、探しているものへのリンクが付いた投稿を見て、それをクリックします...そして、私は登録していないというメッセージとそれをする申し出を受け取ります（怒りに+1）。 登録フォームにアクセスすると、2,000万の必須フィールドがあります（怒りに対して1000）。 このフォームで苦労した結果、登録が成功したことがわかりましたが、メールのリンクからアカウントをアクティブにする必要があります。 （怒り+10）。 さて、私は記録を有効にしています、今、彼らは私のユーザー名とパスワードで入力するように頼みます。 （怒りと、「別の」フォーラムではなく、 古代のミステリーにアクセスできるという感覚に+100。） もう一度入力します ああ、なぜこのリソースに来たのですか？ ああ、ファイルをダウンロードしてください！ 彼はどこですか？ おっと、フォーラムのメインページにいます。 （怒りはすでに符号なし整数の上限を超えており、私は啓蒙の状態にあります）。 さて、検索エンジンに行きましょう、そしてそれを通して再びフォーラムに行きます。 ファイルへのリンクをクリックして... 404 File Not Found ... FFFFFUUUUUUUUUUU ...



ちなみに、「リンクをクリックしました-ダウンロードが開始されました」というフレーズに何を含めるべきかを説明するのに必要な文字数を評価してください。



はい、もちろんこれは最もひどい例ですが、その部分は常に異なるサービスで見られます。 そうすべきだと思うなら、なぜこれをしているのかを自問してください。



-自慢するためにユーザーベースを保存しますか？ 90％は、特定の何かのためにサイトを一度訪れただけで、二度と戻らない死んだ魂です。 データベースおよびユーザーテーブルの不要なエントリ。 登録ユーザーの数を誇りにするのは、もしあなたが彼らに選択肢を残さないなら、一般的な自己欺ceptionです。



-コンテンツを自動的にダウンロードすることを恐れていますか？ 極端な例として、より良いキャプチャ。 ボットはログインできます。 私はcaptchaが無力の兆候であり、問​​題をユーザーの肩に移すことも考えていますが。



別の角度から登録を見てください。 あなたのサービスは当然ユーザーを認識できなければならず、登録/承認はこれだけでこの問題を解決します。 そのため、それなしではできない場合にのみ、ログインするように依頼してください。 投稿やコメントを書いたり、個人用プレイリストを作成したり、サービスに情報を保存するその他の試みを行うために、フォームの前にログインフォームを挿入します。 ユーザーに時間と神経をブタの突っ込みに費やすことを勧めないでください（こんにちはIdyynikとLukeleik）。



時間は人が持つ最も価値のあるものであり、それを補う方法はありません。 ユーザーとその時間を尊重します。



平和とこんにちはラニー！
