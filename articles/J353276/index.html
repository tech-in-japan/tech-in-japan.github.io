ボタンの表示モデルでは、次を設定できます。



  • コンテンツ（タイトル）、

  • デザイン（タイトルの色）、

問題の原因



  • 新しい開発者



プロジェクトの勢いを増すために、新しい開発者が採用されました。



  • 共有リソース



すでに述べたように、すべての国に共通のリソースがあり、それらはパブリックドメインです。 新しい開発者（または古い開発者でも）は、パラメータを変更したり、共有リソースの要素を移動/変更したりできます。 もちろん、誰もがこれを行うべきではないことを理解していますが、それでも事例はありました。



  • デザイナーは、アプリケーションの統一スタイルを忘れていました



すべてのアプリケーションのスタイルは同じです。たとえば、同じテキスト配置、特定のセルの色、同じ背景などです。 時間が経つにつれて、デザイナーはこの単一のスタイルを忘れ始め、ベースセルのスタイルを変更できるようになりました。 しかし彼は、そうすることで、まったく別のアプリケーションに害を及ぼす可能性があるとは考えていませんでした。 そして、それに関与している開発者も、彼が被害を受けたことを知りませんでした。 その結果、手動テスト中に、「左」設計に常にいくつかの問題がありました。



これらの問題の解決策は、Ulテストの導入であることがすぐにわかりました。 Ulテストを導入する前に、iOSプロジェクトでこれを行うことができるツールの市場について少し分析しましたが、他のプラットフォームについては考慮しませんでした。

iOS Ulテストは、2015年に導入されたかなり新しいテクノロジーです。 次の2つのことに基づいています。

スタブマネージャー



何らかの理由で、ネットワークエミュレーションの実装は難しいと考えられています。 実際、すべてはNSURLProtocolクラスを使用して非常に簡単に実行されます。 これは、iOSのURLローディングシステムの動作を事前に定義できるクラスです。



すべては2つのアクションで実行されます。



  1. NSURLProtocolを継承する独自のクラスを作成します。
  2. URL読み込みシステムに登録します。


アプリケーションでネットワークにリクエストを送信すると、ブートシステムはまずNSURLProtocolが登録されているかどうかを確認します。 もしそうなら、彼女は自分の順番でそれぞれに頼り、このリクエストを処理することが可能かどうか尋ねます。 「はい」と言うと、ネットワークへのリクエストが終了し、アプリケーションレベルで処理されます。



カスタムクラスNSURLProtocol



再定義する必要があるのは3つのメソッドのみです。



  1. このリクエストを処理できますか（trueまたはfalseを返すだけです）： 



     +(BOOL)canInitWithRequest:(NSURLRequest *)request
  2. startLoadingメソッド自体。データ置換のすべての魔法が発生します。 



     -(void)startLoading
  3. 最後に、リクエストの処理が終了したことをシステムに伝えます。 



     -(void)stopLoading


すべての魔法が起こるstartLoadingについてもう少し。 



 (void)startLoading NSData *cachedData if (cachedData) { NSHTTPURLResponse *response [self.client URLProtocol: didReceiveResponse: cacheStoragePolicy:]; [self.client URLProtocol:self didLoadData: cachedData]; [self.client URLProtocolDidFinishLoading: self]; } else { [self.client URLProtocol: self didFailWithError:   }


内部では、キャッシュ(NSData *cachedData)



からデータを取得します。 私たちの場合、テストケースごとに同じリクエストに対する異なる回答が存在する可能性があるため、それらは異なるバンドルに保存されます。 そのようなデータがある場合は、独自のカスタム応答を作成してシステムに渡し、キャッシュされたデータを返すと言います。



このデータが見つからなかった場合は、カスタムエラーを作成するだけで、ネットワークから送信されたようにリクエストに戻ります。



スタブマネージャーでは、クラスのみを登録します。 これは以下を介して行われます。 



 NSURLSessionConfiguration setProtocolClasses: @[[MyNSURLProtocol class]]


すべてが非常に迅速かつ簡単に行われ、機能します。 だからこれを恐れないでください。



ページオブジェクトパターン



私が言ったように、私たちはテスト記録のマイナス面と戦い、UIテストを書くためにより責任あるアプローチを取ることを決めました。 コードをきれいにしようとし、Pageオブジェクトパターンを適用しました。







これは非常に単純で、テストとテストが記述された画面の間に、再利用された機能が実装されているレイヤーが必要であるという事実に基づいています。 つまり、要素とのすべての複雑な相互作用、および画面上の要素のすべての検索がそこで保護されます。



ページオブジェクトパターンの利点：



  • コードの読みやすさ。

  • 機能の再利用。

  • ユーザーインターフェイスの一元化。



私たちのアプローチ



各画面にレイヤーを書くのは不要だと判断しました。 XCTestUtilsと呼ばれる3つのコンポーネントで構成される、アプリケーション全体の一般的なレイヤーのみを作成しました。



  1. 画面上のアイテムの検索が保護されるFFElementsクラス。 キーによる検索とインデックスによる検索の2つの機能を実装しています。 



      @interface FFElements : NSObject -(XCUIElement *)objectForKeyedSubscript:(NSString *)key; -(XCUIElement *)objectAtIndexedSubscript:(NSUInteger)index; @end


  2. XCTestCaseのカテゴリ（要素）。 ここでは、UIテストを記述するために必要な特定のFFElementを示しました。セル、ラベル、ボタン。 テキストフィールド。 



      @interface XCTestCase (Elements) @property (nonatomic,readonly) FFElements *textFields; @property (nonatomic,readonly) FFElements *buttons; @property (nonatomic,readonly) FFElements *labels; @property (nonatomic,readonly) FFElements *cells; -(void)wait:(NSTimeInterval)interval; @end


  3. textFieldsの操作に役立つXCUIEIementの小さなカテゴリ： 



      @interface XCUIElement (Utils) @property (nonatomic) NSString *pasteText; +(void)forceTap; @end




デモで見たように、textFieldsにテキストを入力するために、各ボタンをクリックするためのコードが生成されます。 forceTap機能-長押しを導入しました。 textFieldsをクリックすると、「Paste text」または「Copy」ボックスが表示されます。



それはクールになり、誰もが幸せでした。



デモ



Pageオブジェクトパターンの使用方法に関する小さなデモを示します。





プロパティFFElementsを介して、特定のタイプの具象要素にアクセスし、それらと対話するメソッドを呼び出します。 textFieldsは次のように操作します：クリックして長押しし、テキストを貼り付けます。 わかりやすく、読みやすく、そして最も重要なのは、6行だけでした。



スナップショット



UIテストを実行し、デバイス上で実行しますが、デザインがうまくいかないことがわかります。 そして、それを見ることができます。



そのため、スナップショットのようなツールを導入することにしました。 これはFastlaneツールグループの一部であり、非常にシンプルで実装が簡単であり、大きなメリットがあります。



スナップショットの利点：



  • スクリーンショットを撮ります。

  • UIテストを実行し、適切な場所でスクリーンショットを撮ります。

  • 端末から実行されるため、CIと簡単に統合できます。



スナップショットを文字通り実装するには、3つのステップが必要です。



  1. ターミナルで、fastlane snapshot initコマンドを入力します。 その後、Snapfileというファイルが生成されます-これが構成ファイルです。

  2. Snapfileを構成します。

  3. スクリーンショットを取得したい場所を示します。



スナップファイル



通常、UIテストを実行するすべてのデバイスを指定でき、スクリーンショットは異なるデバイスの同じ場所で撮影されます。 



  devices ([ "iPhone 6s" ]) scheme: "our scheme" output_directory: "./path/." stop_after_first_error: Bool reinstall_app: Bool clear_previous_screenshots: Bool Erase_simulator: Bool languages ([ "en-US" ])


さらに、UIテストを実行するスキーム、すべてのスクリーンショットの結果が保存されるフォルダー、および補助設定を示します。



UIテストを実行する言語を指定することもできます。 ローカライズ版が多数ある場合は、ここですべてのローカライズ版を指定し、出力で異なる言語のスクリーンショットを取得できます。



スクリーンショットの時間



技術的な観点からは、コード自体に特定のファイル-SnaphotHelperを追加するだけです。 テストでターゲットに迅速に。 このファイルはプロジェクトフォルダーに生成されます。fastlanesnapshot initを実行するときは、ターゲットに配置するだけです。



次に、スクリーンショットを撮り、そこでXCUIアプリケーションを転送するすべてのテストケース内で、 setup（）メソッド内で[Snapshot setupSnapshot：app]を呼び出します。



最後に残っているのは、スクリーンショットを撮りたい場所でコードを実行し、写真を生成するスクリーンショットの名前を指定することです[スナップショットスナップショット：@ "Name screen" waitForLoadingIndicator：YES] 、ターミナルから実行し、fastlaneスナップショットを作成して満足します。



テストに合格すると、異なる画面の写真だけでなく、HTMLページが生成されます。







その上で、すべての画像は、たとえば、デバイスごと、およびSnapfileで指定したローカライズごとに、カテゴリに分類されます。 確かに、唯一のマイナスは、これらのすべてのスクリーンショットが実行の順序ではなく名前でソートされることです。



注意してください！



UIテストの実装を決定する前に、いくつかの点に注意する必要があります。



  1. スタブマネージャーを恐れる必要はありません。100％実装する必要があります。 実際のネットワークでは、テストは良くありません。

  2. ページオブジェクトパターン -コードがUIテスト用に作成されている場合でも、コードの純度のために戦う必要があります。

  3. スナップショットツールを簡単に実装できます。 すべてのテストケースの準備が整っている場合、すべてのコードが記述されます。これは正午までに行われます。



このプロジェクトでは、次の利点がありました 。



  • 設計バグの数をほぼゼロに減らしました。

  • アプリケーションのすべてのスクリーンショットを迅速に生成するツールと、CIとの統合テストを実装しました。

  • アプリケーションで正しいルーティングを確認しました。



私はこれについては話しませんでしたが、かなり複雑なルーティングがあります。 国によって異なる場合があります。 新しい画面ごとに、この画面のみにある特定の要素をチェックします。それがない場合は、間違った場所にいる、何かが間違っていることを意味します。 そのため、ルーティングを確認します。



しかし、いくつか短所もありました：



  • UIテストの実装期間 。 基本的なフレームワークを作成します。Pag​​eオブジェクトパターンを実装し、Snapshot、Stub Managerを実装します-これは非常に迅速に行われますが、テストケースとデバッグの検討に時間がかかります。

  • また、リファクタリング時にテストをサポートする必要があります。 私たちの場合、これは非常に頻繁に行われます。 もちろん、これに費やす時間はあまりありませんが、忘れないでください。



約束どおり、最後に、UIテストを実装する価値がある場合に考えます。



使用テストは価値があります：



  1. 長期プロジェクトがある場合-100％UIテストを実装する必要があります。 製品を絶えず開発している場合は、設計の品質にも責任を負う必要があります。

  2. Pixel Perfectについて対話を行い、「何かがそこに行きました」という言葉ではなく、アプリケーション全体のスクリーンショットを提供して、修正が必要なものを確認できるようにします。

  3. アプリケーションに複雑なナビゲーションロジックがある場合は、UIテストも実装する必要があります。これは非常に役立ちます。 設計にあまり注意を払うことさえできませんが、ナビゲーションをチェックするためのロジックの実装は非常に簡単です。
