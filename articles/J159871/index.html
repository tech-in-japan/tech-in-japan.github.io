なぜWindows RTはそうですか？

マイクロソフトのトピックへの応答で少し失敗しますか？ 私の意見では、Windows RTはTelemetryを使用する究極の例ですか？ Windows 8とWindows RTがまさにそれであることが判明した理由について。 著者であり、かなり高い地位にあった元Microsoft従業員であるHal Berensonは現在、独立コンサルタントです。 一般に、非常に優れたブログがあり、Microsoftをフォローしているすべての人にお勧めします。



Windows RTがそれだけであることが判明した理由について、多くの人が困惑していますか？ 特に、サードパーティのデスクトップアプリケーションのサポートが不足しているという点で。 Windows RTは、意思決定におけるテレメトリの大規模な使用の結果であると考えています。 新しいマイクロソフト製品を積極的に嫌う人は、「嘘、大きな嘘、統計」があるということわざを確実に思い出すことができます。 しかし、一方で、統計分析に依存することで、Windows 8およびWindows RTに対する標準ユーザーの反応が、専門家や経験豊富なユーザーの反応よりも大幅に優れているように見える理由を説明できます。 製品が設計されている対象読者の外側にいる場合、何かを積極的に評価することは困難です。







各決定は多くの仮定に基づいており、その中で最も重要なのはデータです。 ソリューションには常に十分なデータがありません（したがって、管理は不十分なデータに基づいて意思決定を行う技術であると言われています）。 優れたデータがあれば、答えが明らかになるため、意思決定は簡単だと考える人がいるかもしれません。 ただし、通常は異なるソースからのデータを混在させる必要があります。 ソースが多いほど、分析プロセスの結果として潜在的なエラーが表示される可能性があります。 多くの場合、これらのソース自体は、エラーが発生したプライマリデータの分析の結果です。 そして、最終的には、人生の現実とは関係のないソリューション（例えば、 New Coke ）を手に入れることができます。



しかし、ほぼ完璧なデータがある場合はどうでしょうか？ 少量、限られた収集技術、ランダムデータの代わりに、それが絶対に現実を表していると言える巨大なサンプルがあるとしたらどうでしょう？ これはより良い解決策につながりますか？ 特に、これらのソリューションが非常に複雑でリスクが高い場合はどうでしょうか？ Windows 8での大規模な実験が最終的にこの質問に答えるようです。



Windows Phone 7の設計時に、開発チームは最も人気のある100のiPhoneアプリを調査して、プラットフォームが必要なすべての機能を実現できることを確認しました。 決定の基礎はユースケースの研究でしたが、これは直接的なデータではありません。 さらに、Windows 8チームは、 CEIPプログラムを使用して、自発的にサインアップしたユーザーから収集された膨大な量のデータにアクセスできました（ または、ベータ版を使用している場合、既定で有効になっているベータ版を使用していない場合、約Per ）。



Building Windows 8ブログをフォローしている人なら誰でも、Microsoftが新しいWindows 8インターフェイスを開発する際にこれらのデータを真剣に受け止めていることに気付くでしょう。システム使用パターン。 もちろん、使用パターンが標準とは大きく異なる少数派である場合、Microsoftが行ったことに満足することはできません。 さらに、Microsoftが人々が製品を実際にどのように使用するかを説明するデータを持っているだけでも、ソリューションが正しいことを保証するものではありません。 たとえば、ユーザーが古いインターフェイスと新しいインターフェイスの間でこのような切り替えを望んでいることをデータに伝えることはできません。



しかし、Windows RTに戻り、テレメトリを使用して重要な決定を行う方法について説明します。 ネットブックを思い出してください。 5年前、コンピューターの使用がネットワークに移行しており、ユーザーがこれのために安価な特別なデバイスを必要としていることが理解されるようになりました。 このようなデバイスは、ほとんどのユーザーにとって本格的なPCの代替品ではありませんが、メインコンピューターへの追加と見なされました。 当初、ネットブックは既存のWindowsアプリケーションを実行することを意図していなかったため、メーカーはそれらのメインOSとしてLinuxに注目していました。 この市場の急速な成長により、Microsoftはそれに注意を払い、最初に安価なバージョンのWindows XP、次にWindows 7 Starterを提供しました。 追加の15％のコスト（一部はライセンスのコスト、一部はLinuxと比較してより強力なハードウェアの需要によるもの）にもかかわらず、Windowsのネットブックはすぐに市場の90％を占めました。



ネットブック市場のシェアは急上昇し、PC市場で非常に顕著なニッチになり、その後すぐに3回打撃を受けました。 2009年にApp Storeが導入され、急速に成長し始め、すべてのアプリケーションをWeb上に残す代わりの手段を提供しました（さらにiPhoneにはかなりまともなブラウザーがありました）。 その後、ネットブックの需要は、軽くて薄い11インチの本格的なラップトップにシフトしました。 そして最後に、AppleはiPadを導入しました。これは、Webを使用するためのネットブックのはるかに優れた代替手段であり、App Storeからアプリケーションを実行する機能も備えていました。 ネットブックはほとんど姿を消しました。



Windows 8の開発は、iPadが導入される前から始まりました。 さらに後、iPad 2の発売中、多くのアナリストはタブレットがネットブックの代替品に過ぎないと考えていました。 そして最近まで、タブレットがPCの売り上げに与える影響は、ネットブックからのタブレットへの需要の変化という意味にすぎませんでした。 そのため、2009年末から2010年初頭にWindows 8およびWindows RTの設計に関する決定を下すためのデータの可用性に関して、ネットブックからCEIPを介したテレメトリは、Microsoftが自由に使えるものでした。



次に、Windows RTのみに焦点を当てます。 2009年には、ARMプロセッサを中心に巨大なエコシステムが成長していることはすでに明らかでした。 Microsoftは、10年の初めからWindowsをARMに移植することに取り組み、さらに作業を進めています。 ARMへの移植の決定は、タスクの簡単な部分です。 彼と次に何をするかについての決定は、テレメトリー分析を通じて行われました。 そして、私はそれが本当にそうだったという情報を持っていませんが、私はお金を入れる準備ができています、それは真実から遠くないということです。



Windows Phoneチームが電話を担当しており、ARMが本格的なラップトップとデスクトップを使用するような重いタスクを確実に実行できなかったため、ARM上のWindowsは2つのデバイスのクラスに焦点を当てる必要があることは明らかでした。 その時点では、デバイスは1つだけでした-これはネットブックです。 繰り返しになりますが、iPadはまだ存在せず、iPadに対するユーザーの反応は不明でした。 しかし、iPhoneに似たタブレットの一般的な特性は、明らかに予見できます。



したがって、ARM上のWindowsは、ネットブック、タッチサポートを備えたネットブック、およびネットブックに類似した機能を備えたタブレットのニッチを目指すべきであることは明らかでした。 しかし、後の出来事（ネットブックの消滅とタブレットの台頭）についての知識がなければ、マイクロソフトはどのように決定を下すことができるのでしょうか？ テレメトリー。



なぜユーザーの90％+パーセントがLinuxではなくWindowsを搭載したネットブックにもっとお金を払うことを選んだのですか？ デバイスがWebブラウジングのみに使用される場合、そのようなユーザーの動作は無意味です。 もちろん、さらに推測することしかできません。 UIの習慣、多数のプリンターとの互換性、通常のWindowsアプリケーションを実行する機能（これは、ネットブックの背後にある元のアイデアと矛盾しますが）など。 先ほど言ったように、推測することしかできません。 また、アナリストはユーザーに問い合わせて、独自の結論を引き出すことができます。 しかし、マイクロソフト？ マイクロソフトには、CEIPからの正確なデータがあります。



Microsoftはデータを取得して、ユーザーが印刷する頻度と使用するプリンターを確認できます。 どのくらいの頻度でUSBを使用し、どのように使用しますか。 ネットブックが外部モニター、キーボード、マウスで使用される頻度。 WiFi、有線イーサネット、および3Gが使用される頻度。 Microsoftは、ユーザーがブラウザを使用する頻度と、訪問するサイトの種類を確認できます。 他のアプリケーションが起動するもの、およびそれらの内部で費やす時間。



また、マイクロソフトがテレメトリーから正確に得たものは何だと思いますか？ ネットブックが主にウェブに使用しているものを見たと思う。 次に、Microsoft Officeを使用する際の注目すべき小さな部分があります。 そして、他のアプリケーションの使用が完全に落ちます。 ネットブックは、基本的にWebおよびオフィスアプリケーション用のデバイスです。 次に、サイト訪問の統計を見て、それらのほとんどが、iPhoneで非常に人気があり、新しいMetroアプリケーションモデルを開発する必要があるコンテンツの消費のためのアプリケーションの概念に該当することを確認しました。 また、さまざまな周辺機器のサポート、ネットワークとの連携など、Windowsの標準機能の大規模な使用を見ました。 ここで、ネットブックの使用から得たこのデータを、App Storeのアプリケーションのタコとブームの開発と組み合わせれば、Windows RTが手に入ります。



標準的な質問は、Windows RTがエミュレーションを通じてレガシーx86アプリケーションをサポートしないようにすることです。 一見、これは技術的に不可能ではありません。 DEC Alphaプロセッサはエミュレーションを通じてx86アプリケーションを実行していましたが、当時のAlphaはx86自体よりも高速であったため、十分なパフォーマンスが得られました。 現在のARMプロセッサは、同等のx86よりも著しく低速です（エネルギー効率がはるかに高いにもかかわらず）。 そのため、ARMでのx86エミュレーションにより、ほとんどのアプリケーションが通常の使用に適さなくなります。 しかし、より重要なことは、ネットブックの使用に関するデータが、そのような機会があったとしてもユーザーがまだ何も開始しないことを示している場合、なぜこれを行うのが面倒なのでしょうか？



さて、エミュレーションは困難ですが、サードパーティのアプリケーション開発者にARMに移植するためのツールを提供し、そこにサードパーティのアプリケーションをインストールできるようにしないのはなぜですか。 もちろん、エネルギー消費、メモリ使用量、およびセキュリティに関連する問題があります。 など、古典的なアプリケーションが苦しみ、新しいAPIを解決しなければならなかったもの。 マイクロソフトは、たとえばMicrosoft Officeチームと同様に、サードパーティの開発者にこれらすべてに十分な注意を払わせることができます。 しかし、これにより、新しいAPI用のアプリケーションの作成が妨げられます。 そして、ユーザーが実際にこのタイプのデバイスでそれらを実行するつもりがないのに、なぜこれを気にするのでしょうか？



多くのユーザーがネットブックでPhotoshopを実行することはほとんどありません。 彼らが写真にネットブックを使用している場合、ほとんどの場合、彼らはApp Storeを埋める軽量のアプリケーションを使用しており、それは自分のMicrosoftストアで十分早く表示されます。 そのため、ネットブックの使用に関するテレメトリデータの分析により、実際には、このクラスのデバイスでデスクトップアプリケーションを実行する必要があるのはごく一部のユーザーだけであるという結論に至りました。



それでは、Windows RTを見てみましょう。Surfaceを使って、それが何であるかを見てみましょう。 Surfaceは、ネットブックがiPadと出会う場所です。 それは、ほとんどのユーザーが現代のネットブック上のWindowsについて愛していたものを正確に転送し、ネットブックユーザーが単に使用しなかったものを取り除きます。 これは、これらの長年のネットブックから現代世界に外挿されたテレメトリデータを通じてユーザーがマイクロソフトに伝えたものです。



別の証拠？ ドメイン 一方では、ドメインにWindows RTを入力できないのは奇妙に思えますが、他方では、ドメインにいくつのネットブックがありましたか？ マイクロソフトは、この機能を実行しない他の理由をいくつか持っていたかもしれませんが、主な要因は、テレメトリ分析がこのタイプのデバイスにとって重要ではないことを示したことでした。 ここで、Windows RTの機能に関する質問を思い出してください。ほとんどの場合、ネットブックを使用する一般的なシナリオで答えが見つかります。



テレメトリを使用すると、Windows 8、Windows RT、およびSurfaceの理由を説明できます。

使用事例が平均から大きく逸脱している専門家や経験豊富なユーザーよりも一般ユーザーの方がよく認識されています。 Windows 8、Windows RT、およびSurfaceは、専門家や経験豊富なユーザーによって通常無視されるデバイスのクラスの現在の使用データに基づいて設計されました。 このセグメントは、ポストPC時代が始まる前に約20％を占めていました。 Microsoftが全量のテレメトリを使用してネットブックを非常に人気のあるものにし、さらにAppleがiPadを作成したシナリオを少なくとも部分的にカバーする機能を追加した場合、Windows RTは実際に勝つことができます。



しかし、Windows RTが失敗した場合はどうなりますか？ これは、たとえば、専門家の不満の声が本当のチャンスを得る前に製品を詰まらせるためです。 または、優れたデータにもかかわらず決定が下された結果として。 または、マーケティング、販売、製品に関係のないパートナーのエラーなどの重大な間違いが原因です。 または、統計と呼ばれる嘘の犠牲者になるからです。
