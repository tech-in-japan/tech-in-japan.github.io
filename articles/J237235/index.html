企業ネットワークでMacを管理する方法を学ばなければならなかった方法

Bring Your Own Device（BYOD）は、トップマネージャーのMacbookだけでなく、企業ネットワーク内の代理人のiPad（さらにはiPhone）も持っている管理者にとって、すでに頭痛の種です。 今年、米国の大企業の100人以上のシステム管理者にインタビューしました。彼らは、主にWindows PCが占有する企業ネットワーク上のモバイルガジェットを管理するために何を使用していますか。 「無」と「強い表現/アルコール/涙」が勝ちました。 同時に、45％が、企業がすでにビジネス目的でMacを購入していることを認めました。 これは、マネージャーと主要な従業員が使用するBYODデバイスを介して、数百万ルーブルに相当する情報が通過できるため、問題を解決する必要があることを意味します。



この投稿では、企業環境での独自のMac管理ソリューションであるParallels Mac Managementを自社で導入した方法、開始した場所、およびプロセスで直面したこと（従業員の行動の純粋に心理的な側面を含む）について説明します。 また、企業ネットワークでMacが本当に必要かどうか（そして、それについてのあなたの意見を調べる）、何のために、誰が何を管理に使用するかについて推測することもできます。



怖い現実


企業のすべてのITサービスとは異なり、企業環境でMacが必要な理由の答えを見つけることができるという事実にもかかわらず、その成長は事実です（ Greyhound Researchと独立した専門家による研究を参照）。 企業市場向けの自社のデスクトップおよびモバイル製品（Mac Enterprise EditionのParallels Desktop、同じParallels Mac Management、およびParallels Accessのビジネスバージョン）の売り上げの伸びもこれを裏付けています。 さらに、Windowsで働いている人の95％が、異なるプラットフォームで単一のPC制御システムを使用できるようになり次第、Mac OS Xに移行することを喜んでいるという調査 。



正直に言ってください-多くの企業にとって、コストとリソースを節約するという観点からは、Windows用のPCを選択するほうが収益性が高くなります.Macはより高価であり、Mac OS Xの専用アプリケーションのエコシステムはあまり開発されておらず、大量に習慣的に管理されていません。 Appleが企業環境に積極的に浸透し続けるのはなぜですか？



デバイスは機能していますか？ そしてそれを見つけたら？


同じ調査で、Macプラットフォームはより信頼性が高い-グリッチやウイルスが少なく（回答者の77％が言った）、保守が容易（65％）で、従業員を引き付けるのに役立つ（また65％）と言われました。 しかし、その理由は次の要因によって引き起こされるBYODコンセプト自体の人気の高まりにあるようです。



第一に 、これはモバイルデバイス（電話やタブレット）の数の爆発的な増加であり、プロの仕事への方向転換です。 最近はコンピューターだけができることをすべて実行できるウェアラブルガジェットが増えています。 これは、スマートフォンとタブレットの両方に適用されます。 2013年には、1億9,500万台以上のタブレットが販売されました（Androidの場合は62％、iOSの場合は36％）。 同時に、世界のAndroidおよびiOS Xプラットフォームで使用されている電話の数は9億を超えました（iPhoneのシェア-15.2％、Android -78.6％）。 そして今年、人々はラップトップよりも多くのタブレットを購入するでしょう。



第二に 、BYODの流行は、雇用主企業と従業員の新しい関係を反映している可能性が非常に高いです。 これらの新しい傾向は、西洋から徐々に私たちにやって来ており、従業員が組織の一部として自分自身を考慮する可能性が低い、またはスタッフに含まれていないという事実に表れています。 たとえば、米国の請負業者（「請負業者」）は従業員とは見なされず、会社は何をすべきかを伝える権利を持っていますが、その方法を教えているわけではありません。 スタッフは同じように振る舞います。 このアプローチにより、従業員の制限が少なくなり、雇用主を含むすべての市場参加者に事実上課せられます。 また、タスクがどのように解決されるかは問題ではないため、心から大切なあらゆるデバイスを使用できます。 彼らの技術サポートの問題は、ビジネスに直接関係しないため、めったに議論されません。 さらに、BYODは雇用主からデバイスを取得するコストを取り除き、従業員に転送します。



エンタープライズ環境にMacを含めるのはなぜですか？


おそらく、個々の発生事例を無視する方が簡単ですか？ Windows PCを制御するのと同じ目的で、これを正確に行う必要があると考えています。 たとえば、当社では次のとおりです。



  • 技術的な不具合またはユーザーが退職することによる、ラップトップ上のデータ損失の防止。
  • アップデートの正確な計画、ライセンス要件への準拠、およびITコストの正しい予測のためのソフトウェアおよび機器のインベントリ。
  • リモート管理の簡素化-標準OSイメージの展開、ソフトウェア製品の展開または更新、インシデントを解決するためのリモート管理、ポリシーと一貫した構成。
  • 私たちにとっての具体的な目標は、自社デザインの製品の実用化の経験です。




実装プロジェクトはどのように始まりましたか？


なぜ自分のMacを管理する方法を学ぶ必要があるのか​​を理解するようになった（すべてのIT部門がこの質問をするように思えますが、経験から、Macの場合の答えは予想外かもしれません）、新しいプロジェクトを開きました。 Mac管理ツールを選択する際、定義上、企業環境には単一のMicrosoft Active DirectoryベースのアカウントディレクトリとVPNインフラストラクチャがあり、リモートユーザーに企業ネットワークへのアクセスを提供し、ユーザーがITで問題を解決することに関心があるという前提から進めました。 怠慢で無関心な従業員はMac上のエージェントを削除し、「すべてが機能しない」ことを保証する可能性があるため、これは重要です。



2012年末に、Centrify User Suite Macエディション、Microsoft System Center 2012 SP1（当時はCustomer Technology Preview、CTPの段階にあった）、およびParallels Mac Management（以降-PMM）のバージョン1.0の3つの適切な製品を特定しました。 Centrifyから、ライセンスの支払いが不要になりました。 System Centerライセンスは、マイクロソフトパートナーネットワークプログラムを通じてマイクロソフトパートナーに無料で提供されます（そして、私たちはそのメンバーです）。 System Center Configuration Manager 2007をロシアのオフィスに展開するタスクのために、System Centerの追加のクライアントライセンス（クライアント管理ライセンス、CML）が既に購入されています。 System Center 2012の製品使用権*コンテンツには、エージェントの出所についての予約なしで、各管理対象デバイスにCMLが必要であることが明確に示されているため、これらのライセンスはPMMの展開にも必要です。 ところで、SCCM 2012サーバーの配布は、CMLを購入することによってのみ取得できました。サーバーライセンスはマイクロソフト製品の範囲では利用できませんでしたが、ボリュームライセンスサービスセンターでの配布キットへのアクセスはCMLの取得に関連付けられていました。



プロジェクトの開始時にMacを制御する機能は、SCCM 2012 SP1でのみ発表されました。 彼のリリースのリリースは2013年初頭に予定されていたため、困難は明らかでした。 当時、RMMは会社のいくつかのパートナーをテストすることに同意しました。自社のITサービスがRMMの「内部」の展開を引き受けると、製品開発をスピードアップするためにフィードバックを迅速に得るという新しい目標が現れました。 このプロセスは通常、「セールスエンジニア」-「製品マネージャー」-「開発マネージャー」の通常のチェーンによって遅延されるため、開発者はもちろん、「自分の」からそれらを受け取りました。 さらに、パートナーは米国にありました（この違いは時間と言語にもあります）。 しかし、ITには温室条件はありません：PMMテクニカルサポートに一般的な順序で連絡し、もたらされた収益に応じて、一般的なキューで完了するタスクを設定します（この場合、収益がない場合は、支払いを行う顧客の1人が必要とする場合にのみ変更が完了します） ） その結果、ITサービスは各新機能のユーザーケースの詳細な説明で経験を積み、その実装は顧客の1人がそれを望んでいるかどうかに依存していました。



PMMの展開


このため、製品開発やテストに関与していないすべてのMac従業員が選ばれました。 地理的には、それらは世界中に配置されていました。米国ワシントン州レントン、モスクワ、ノボシビルスク、ミュンヘン、シンガポール、東京のオフィス。 米国、EU諸国、オーストラリア、東南アジアの遠隔地の従業員。

これらのオフィスのシステム管理者は、PMMの機能により、リモート管理、標準ソフトウェアパッケージとアップデートの集中インストール、セキュリティポリシー（Mac OS Xプロファイルに基づく）の集中アプリケーション、標準OSイメージのインストール、機器インベントリにより、業務の遂行を促進できると想定されていました。



特に、Parallels Desktop for Mac（Macユーザーに馴染みのあるデスクトップ仮想化製品で、 ここで説明しました ）のリモート管理のタスクと、そのための標準仮想マシンの配布に焦点を当てています。 リモート管理には、最適なVM設定の設定、更新のアクティブ化とインストールが含まれます。 標準の仮想マシン（Office 2013を搭載したWindows 8）は、ドキュメントの互換性と、Office for Macのインターフェイスとは大幅に異なるWindowsのOfficeインターフェイスのユーザー習慣の両方の理由、および多くのアプリケーションにはまだMac OSX。



一連のタスク全体を解決するために、レントン、ミュンヘン、シンガポール、モスクワ、ノボシビルスクというかなりの数のユーザーがいるオフィスにプライマリサーバーSCCMと1つの配布ポイントをインストールすることが決定されました。 リモートユーザーは、VPNを介して最寄りのデータセンター（レントン、ミュンヘン、モスクワ、またはシンガポール）に接続し、そこからトラフィックがMPLS企業ネットワークを介して、ユーザーが割り当てられている配布ポイントにルーティングされます。



拡張機能は、System Center Configuration Managerインターフェイスに統合されます



PMMのサーバー側がプライマリサーバーにインストールされました。 クライアントのインストールは手動で行われ、一部はユーザー自身が、一部はシステム管理者が行いました。 ユーザーには、エージェントの配布へのハイパーリンクを含む手紙が送られました。システム管理者の介入は、インストールが正常に終了しなかった場合にのみ必要でした。 バージョンPMM 1.7以降、SCCMコマンドによるエージェントバージョンの更新を含むインストールは、約90％のケースで自動モードで正常に実行されました。 以前に失敗しなかったコンピューターへのエージェントのインストールが行われたほとんどの場合、その理由はユーザーがインストール手順に従わなかったためです（VPNに接続せずにインストールを開始できますが、正常に完了するには、VPNからのみアクセス可能なドメインコントローラーにアクセスする必要があります） 現在、PMMの現在のバージョンは3.1であり、小児疾患は克服されたと見なすことができ、さらなる開発は機能の拡大を目指しています。



展開の概要


技術的な観点からは、ParallelsでのPMMの展開は成功したと見なすことができます。 エージェントは、Macによって内部的に選択されたエージェントの95％以上（150以上）にインストールされます。 管理対象Macにソフトウェアパッケージを配布し、Mac OS Xプロファイルを介してパスワードの複雑さと有効期間に関する会社の受け入れられている要件を満たすパスワードポリシーを適用し、仮想マシンの設定を構成することができます。 バージョン2.0からは、OSの自動インストールが可能になりましたが、Mac OSの組み込みツールはOSの更新タスクに対応できるため、この方法での大量のOS更新の経験はまだありません。 また、3番目のバージョンでは、アプリケーションを操作するためのセルフサービスポータル、HTTPS SCCMインフラストラクチャおよびSCCMアプリケーションのサポートなどの新機能が登場しました。 SCCMクライアントのサポートの改善、秘密鍵を使用したFileVault 2暗号化システムもサポートされています。



Parallels Mac Managementのセルフサービスポータル



PMMを実装する上で最も難しい障害の1つは、SCCMについて学ぶことでした。 通常、この製品の経験はなかったため、システム管理者の一人が最初に独自に学習し、次にマイクロソフト認定コースで学習する必要がありました。 後者は、SCCMをインストールして適用しようとする独立した試みの間に蓄積された質問に対する回答を受け取るだけでなく、独立して獲得した知識と経験を体系化することを可能にしました。 その結果、PMMのインストールおよび展開中に発生した問題の大部分は、SCCMの最適でない構成が原因であることが判明しました。 結論： PCMの展開は、組織がSCCMの経験がある場合にのみ開始できます。まだ経験していない場合は、正式な管理者トレーニングが必要です。



2番目の障害は心理的および法的性質でした。 EU諸国の一部の従業員は、この方法でIT部門がプライバシーを侵害すると述べて、エージェントのインストールに反対しました。 この反対は、会社所有のコンピューターは個人情報の場所ではないと説明した司法長官の助けを借りて克服しなければなりませんでした。 ただし、BYODの場合、このような異論ははるかに堅実な根拠を持っている可能性があります。 定義上、それらの技術的な解決策はないため（OSを再インストールできるエージェントはすべてのデータにアクセスできます）、解決策は契約面になければなりません。 結論2： BYOD条件の下で働く従業員は、雇用または契約関係を締結するための条件として、エージェントをインストールすることに書面で同意するよう招待されるべきです。



今後の準備は何ですか


その結果、すべての困難が克服され、Parallels Mac Managementは商業運用され、最大の企業はRackspace、Samsung、およびMcAfeeで実装されています。 同時に、社内プロジェクトは完了していません。次の段階は、テクニカルサポートスタッフが日常業務でPMMを適用するためのトレーニングです。 すべての操作はインターフェースを通じて実行されるため、これにはまずSCCMの基本を教える必要があります（その中のMacはコンピューターと同じ方法で制御されます）。 また、従業員が企業ポリシーに準拠した推奨ソフトウェアをインストールできるように、アプリケーションセルフサービスポータルを独自のセルフサービスポータルに実装することも計画しています。



現在、Parallels Mac Managementは、他のParallelsソリューションと比較して最大の売上成長率を達成しています（年間50％以上）。 これは、Macを企業環境に統合するタスクが非常に重要になったことを示唆しており、ついにそれらを解決する実用的な機会があったとき、実装は間もなく開始されました。



その結果、次の結論を導き出すことができます。まず、「何もない」、「涙」、「強い表現」に加えて、Macの管理の問題（さらには私たち自身の問題さえ）を解決するのに役立つ製品がまだあります。 次に、予期しない心理的なものを含む、同じネットワーク上でのPCとMacの相互作用のさまざまな矛盾する側面を解決する必要があります（すべての従業員が、このD自体の内容に対する責任とともにBYODの利点を考慮するわけではありません）。



そして、私たちはあなたがこれについてどう思うかに本当に興味があります。ビジネス環境でMacが必要なのか、どのような使用シナリオが存在するのか。 コメントにあなたの考えや質問を書いてください、私たちは皆に答えようとします。
