複雑なJavaScriptアプリケーションのシンプルでシンプルな実装

複雑なJavaScriptアプリケーションを開発するためのシンプルでシンプルなアプローチを説明したいと思います。 外部ライブラリからはjQueryとjsテンプレートエンジンのみが使用され、jQueryからは$.ready()



、 $.ajax()



、および$.proxy()



のみが使用されます。 重要なのはライブラリーではなく（お好みのライブラリーに置き換えるのは簡単です）、アプローチ自体です。



このアプローチは、2つのアイデアに基づいています。

  1. JavaScriptウィジェットは小さなモジュールであり、それぞれがWebページの特定の部分を「所有」します（つまり、ページのこの部分のすべての制御は、DOMの直接変更ではなく、このモジュールのメソッドのみによって行われます- カプセル化 ）。 ウィジェットは機能のみに責任がありますが、外観には責任がありません。 したがって、ウィジェットの「所有」するDOMの一部をウィジェットの外部で直接変更することはできますが、純粋に設計タスクの場合のみです（アーキテクチャおよびアプリケーションの全体的な複雑さの場合、CSSまたはjQueryを介して外観を修正することには基本的な違いはありません）
  2. グローバルイベントディスパッチャー。 ウィジェット間の相互作用は、グローバルディスパッチャー（ 弱い接続 、 Mediator / Mediatorパターン）にメッセージを送信することによって実行され、彼はこのメッセージの処理方法を既に決定しています-ウィジェットの作成/削除、他のウィジェットメソッドのプル、デザイナーコードの実行など イベント処理に対する動的なアプローチ（特定のイベントのイベントハンドラーが操作中に追加/削除される場合）とは対照的に、静的ディスパッチャはコードの理解とデバッグを大幅に簡素化します。 もちろん、動的イベントハンドラーが必要なタスクもありますが、ほとんどの場合、これは過度の複雑さであるため、可能なことはすべて静的ハンドラーによって行われます。




自転車について



私は通常、アプリケーションのサーバー側を開発します（さらに、ネットワークサービスほどWebサイト自体ではありません）、クライアント側は私のメインプロファイルではありません。 そのため、かなり洗練された1ページのWebアプリケーションを開発するタスクを最初に行ったとき（約2年前）、そのようなアプリケーションの既製のアーキテクチャアプローチを探しましたが、残念ながら、私は見つけませんでした（レポートから半年が過ぎましたが） Nicholas Zakas「スケーラブルなJavaScriptアプリケーションアーキテクチャ」： ビデオ 、 スライド ）。



私はこの2年間にさまざまなプロジェクトでこのアーキテクチャを開発し、実際のタスクでどのように機能するかを確認しました。 うまくいきます。 :)だから、この記事を書くことにしました。 執筆の過程で、私は再び他のソリューションを探し、Nicholas Zakasのレポートを見つけました。さらに、このトピックの読み物から、記事Addy Osmani Patterns For Large-Scale JavaScript Application Architectureが本当に気に入りました。



そこで説明されているアーキテクチャにすでに精通している人のために、私のアプローチの違いを簡単に説明します。

  • 私の意見では、このアーキテクチャは、Yahoo！、GMailなどの規模のアプリケーションに必要です。 しかし、現代のかなり複雑なWebアプリケーションの大多数にとって、これはまだ多すぎます。 このアーキテクチャの過度の柔軟性には代償が伴います-アプリケーションの複雑さが増します。 以下は、私が実際に必要としなかった機能の例です。

    • ベースライブラリを単純に置き換える機能（例：jQueryとDojo）
    • システム機能に対するモジュールの「アクセス権」の確認と制御
    • ページ上のモジュールを動的に追加/削除
    • 上で書いた動的イベントハンドラーについて
  • これはどこにも明示的に言及されていませんが、私が理解する限り、そのアーキテクチャでは「モジュール」はほぼ全体が大きく複雑なアプリケーションです（GMailページのチャットやYahooの天気ウィジェットなど）。 私の「モジュール」は、独立した独立したエンティティとして区別できる最小限の機能を実装しています。 （ただし、Andrew Burgessのスクリーンキャスト「Writing Modular JavaScript」は、私のものと同じ小さなモジュールを示しています。）例として、1つの入力を持つフォームを担当するモジュール、または同じタグ内で同じ値を出力する担当モジュールを引用できます。 同時に、これらのモジュールはアプリケーションのビジネスロジックを実装するものであり、通常の<select>から非常にクールで美しく、派手な<select>を作成するjQueryプラグインの類似物ではありません。

    • モジュールの機能の規模の違いの結果、それらのモジュールはHTML + CSS + JavaScriptで構成されており、私にとってモジュールの大部分はJavaScriptとこのモジュールに必要な最小限のHTML構造のドキュメントです。
  • モジュールがアクセスできる外部機能を明確に制限する必要があることは完全に同意しますが（実際、これについては記事の最初の段落で書いたので、そのリストにディスパッチャーを追加するだけです）、最も信頼性が高く、効果的だと思います安全なコードはそうではありません。 したがって、コードを規則などに置き換えることができる場合、通常は規則を好みます（もちろん、測定値も知っている必要があります）。 私のアプローチでは、モジュールのアクセスを制限するコードはありませんが、モジュールが登れる場所と登れない場所があります。

ウィジェット



ウィジェットは独自の領域を所有しますが、この領域がページ上のどこにあるかを正確に制御しません-つまり 画面に表示されるかどうか、そして実際にDOMに追加されるかどうかはわかりません。 外部コードは、ページにウィジェット制御エリアを追加/削除する役割を果たします（つまり、一部のウィジェットが他のウィジェットに埋め込まれている場合、グローバルイベントディスパッチャー、または外部ウィジェット）。



ウィジェットクラスの内部では、最低限の外部関数を使用できます。

  • イベントを生成するためのグローバル関数notice()



    。
  • jQuery $



    オブジェクト;
  • テンプレートライブラリ関数。


まれな例外を除き、ウィジェットは何らかの形で画面に表示する必要があります。 彼が「所有する」ページの領域は通常、すでに存在する（そしてその#id



ウィジェットのコンストラクターに渡される）か、ウィジェットによってjs-templateからその場で生成されます（そしてパターンの#id



コンストラクターに渡されます）。



他のすべては基本的ではありません。 私のスタイル：

  • ウィジェットは個別のw.widget_name.jsファイルにあります。
  • ウィジェットは、グローバル名前空間W



    クラスとして実装されます。通常、1つのウィジェットは1つのクラスW.WidgetName



    です。
  • クラスは、JavaScriptの最も単純で自然な方法で実装されます-通常のコンストラクター関数とプロトタイプ継承。
  • プライベートプロパティとメソッドに下線が付き始めます。
  • ウィジェットの最初に、jsテンプレート（またはHTMLブロック）に必要なフォーマットを文書化します。 コンストラクター、パブリックメソッド/プロパティ、生成されたイベント。
  • 慣例により、ウィジェットが制御する領域は、ウィジェットの.$



    プロパティからアクセスできます（ページにウィジェットを表示するためにDOMに挿入されるのは外部コードです）。


少し用語の説明。 多くの場合、JavaScriptウィジェットはHTML + CSS + JavaScriptのセットです。 私のウィジェットは、デフォルトで他のアプリケーションとの接続を容易にするために、HTMLおよびCSSと共に配布することもできますが、絶対に最小限の設計情報を持つようにウィジェットを作成することを好みます-これにより、コードと設計の変更の両方が簡素化されます。 そのため、以下で説明するウィジェットにはデザインに関する言葉はありません。
