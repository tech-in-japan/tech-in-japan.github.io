Microsoft Hyper-V 3.0：新機能

Windows Server 2012のリリースに伴い、仮想化システムのHyper-Vも更新されました。 この記事では、Hyper-V 3.0の新機能について説明します。 3番目のバージョンには以前のバージョンと比較して多くの機能改善が含まれており、最も重要なのは、Hyper-Vの歴史で初めて、Microsoft仮想化システムがVMware vSphere 5に匹敵するエンタープライズレベルの製品レベルに達したことです。



Hyper-Vの主な管理および監視ツールは、 Hyper-Vマネージャーです （図を参照）。 バージョン3.0以降、PowerShellを介して多くの新しい管理機能を利用できるようになりました。 たとえば、VM1仮想マシンの起動は、「Start -VM vm1」コマンドを使用して実行できます。 次に、Hyper-Vの新しいバージョンの主要な機能を検討します。







メモリーバルーニング


仮想化サーバーの個々の仮想マシンに割り当てられたメモリの合計サイズがこの仮想化サーバーで一般的に利用可能なメモリ量を超えるような方法でメモリを割り当てる技術はメモリオーバーコミメントと呼ばれ、Hyper-V 3.0での開発はメモリバルーニングと呼ばれます。 これらのテクノロジーにより、仮想マシンの密度（仮想化サーバーあたりの仮想マシンの数）を増やすことで、サーバー統合の係数を増やすことができます。 この結果は、仮想マシンのアクティビティ、ステータス（オン、オフ）、「最小メモリサイズ」や「仮想マシンの開始段階でのメモリサイズ」などのパラメーターを考慮して、個々の仮想マシンに割り当てられたメモリを動的に再分配することで実現されます管理者を定義します）。 さらに、Hyper-V 2.0でこれら2つのパラメーターが実質的に同等であった場合、Hyper-V 3.0では完全に独立しています。たとえば、マシンの起動段階では、仮想マシンが作業段階で受け取る最小量より多くのメモリを割り当てることができますが、非アクティブの期間。



ライブ移行


ライブ移行 （仮想マシンをシャットダウンせずに仮想化サーバー間で移行する）は、Hyper-V 2.0で登場しました。 当初、制限がありました-一度に移動できる仮想化サーバーの仮想マシンは1つだけでした。 Hyper-V 3.0では、この制限がなくなりました。複数の仮想マシンを並行して移行できますが、管理者は同時に移動する仮想マシンの数に制限を設定して、ネットワークの輻輳を回避できます。





以前は、ライブマイグレーションのフレームワーク内での仮想マシンの移行は、クラスターノード内でのみ可能でした。 Hyper-V 3.0では、この制限はなくなりました。 仮想マシンのこのような「トランスクラスター」移動は、 Shared Nothing Live Migrationsと呼ばれます。 この機能は、従来のクラスターの長期的な代替と見なされるべきではなく、主に必要な仮想化サーバーに仮想マシンを実際に移動する場合にのみ使用されることに注意してください。



シェアードナッシングライブマイグレーションの要件：

  • 最低2台のHyper-V 3.0仮想化サーバー
  • サーバーは同じActive Directoryドメインに含まれている必要があります
  • 最大のパフォーマンスを確保するには、サーバーに同じアーキテクチャのプロセッサが必要です（Hyper-V 3.0のドキュメントの互換性のあるプロセッサの説明を参照）
  • サーバーは、ギガビット以上のネットワーク接続で接続する必要があります
  • 仮想スイッチは同じ構成でなければなりません
  • Microsoftネットワーク用クライアントとMicrosoftネットワーク用ファイルと印刷共有を含める必要があります
ライブ移行がHyper-V 3.0以前のように機能するために、共有ストレージは不要になりました。 ローカルストレージを使用できるようになり、データはイーサネット経由で移行されます。 同じ新しいテクノロジーにより、共通ストレージなしでもクラスターを構築できるようになりました。



ストレージ移行


このテクノロジーはライブマイグレーションとは異なり、仮想マシンファイルはストレージ間で移動しますが、仮想マシン自体は引き続き同じ仮想化サーバー上で実行されます。 このテクノロジーは、Hyper-V 2.0で利用可能なQuick Storage Migrationテクノロジーに取って代わりました。 Hyper-V 3.0では、（以前のバージョンのHyper-Vとは異なり）仮想マシンのストレージはシャットダウンまたは動作を一時停止することなく発生します。





レプリケーション（Hyper-Vレプリカ）


このテクノロジーにより、実行中の仮想マシンを別の仮想化サーバーに複製できます。 増分変更は、バックアップに関してリアルタイムに近い最小起動間隔（「CDPに近い」）であるカスタムスケジュールに従って転送されます（約5〜15分）。 Hyper-Vレプリカは、高い時間遅延と断続的な通信のある回線で動作できます。 マイクロソフトは、この技術を壊滅的な障害から保護する手段として位置付けています。



拡張性






スナップショット


システムにスナップショットが蓄積されると、パフォーマンスが低下します。 これに関連して、Hyper-V 3.0の重要な改善点は、仮想マシンを再起動せずに不要なスナップショットを削除できることです（これはHyper-V 2.0で必要でした）。



ホットアドリソース


Hyper-V 3.0では、仮想マシンを再起動せずにディスクリソースとRAMを追加する機能が導入されました。 VMware vSphereがHyper-Vにまだ実装されていないホットアド仮想プロセッサ機能。



重複排除


Hyper-VはWindows Server 2012の一部であるため、仮想化環境の新しいバージョンではファイルシステムレベルの重複排除が自動的に行われています。 重複排除は、ディスクに保存されているデータの重複ブロックを排除することにより、VHDライブラリ、VHDボリューム、およびディスクのISOイメージが消費するディスク領域を削減します。



仮想スイッチ


Hyper-V 3.0では、仮想スイッチが拡張可能になりました。これにより、ネットワークインフラストラクチャを監視および管理するソフトウェアのメーカーは、Hyper-V仮想スイッチを製品と統合するための拡張モジュールを作成できます。



結論：


Hyper-V 3.0には、エンタープライズセグメントでこのバージョンがVMware vSphereと競合できることを一般的に示唆する多くの拡張機能と新機能が含まれています。
