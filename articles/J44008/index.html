前文



当初、私たちのチームはXPと一般的にアジャイル開発を使用していました。 XPlannerはもともと選ばれました-反復的なソフトウェア開発のために研ぎ澄まされました。

時間が経つにつれて、開発プロセスは変化し、元のプロセスとは少しずつ異なります。



そしてごく最近、私は皆をRedmineにドラッグすることができました。

XPlannerについて



Xplanner — XPlanner is a project planning and tracking tool for eXtreme Programming (XP) teams.





Xplannerは、極端なプログラミングのスタイルで作業するチームのタスクを計画および管理するためのソフトウェアツールです。 プロジェクトは2006年5月に更新されなくなりました。



XPにはいくつかの原則があり、そのうちのいくつかはXPlannerでサポートされています：

  • ソフトウェア開発プロセスは短い反復（通常は1週間または2週間）で構成され、その間に製品機能のかなり完全なブロックを完了する必要があります。 時間通りに終了できないことが判明した場合は、機能を減らす必要がありますが、時間をシフトする必要はありません。
  • 機能のブロックは、「サブストーリー」（ストーリー）-サブシステムまたはモジュールごとに分けられ、独自のストーリーが記述されています。 ストーリーは会議で議論され、記録されます。
  • その後、ストーリーはタスクに分割され、開発を開始する前に評価されます。


このために、XPlannerには適切なツールがあります-反復、ストーリーの反復、タスクのストーリーで構成されるプロジェクト。 パフォーマンスメトリックがあります。 XPlannerの機能のリストは多数ありますが、実際、私たちのチームでは2つしか使用されていません。

  • プロセスを反復、ストーリー、タスクに分割します。
  • 時間の追跡。


短所XPlanner（私の謙虚な意見）


実際、計画ツールから時間追跡ツールに進化しました。 毎週計画を立て、反復から反復へ、履歴から履歴へタスクを転送する方法は非常に長い時間でした。 しかし、さらに悪いことに（私の観点から）、彼は開発者の手によってエミュレートされた基本的なことを実行できませんでした。 たとえば、次の奇妙なプロセスがありました。

  • なぜなら XPlannerはメールでの通知をサポートしていません。各開発者は、特別なメーリングリストに「Task <task id>」という形式の文字を記入する必要がありました。
  • 開発中に、プログラマーが期限に間に合わないと気付いた場合、XPlannerでスコアを変更し、（あなたはそれを推測しましたか？）-はい！ メーリングリストに「問題の評価が変更されました」という手紙を書きます。

    友人はXPlannerに通知があると言っていますが、それらについて正確に言うことはできません
  • 毎晩、開発者はニュースレターにタスク（どのくらいの時間を費やし、何が終了したか）に関するレポートを作成します。
  • コミット時に、タスクのIDはコメントとして書き込まれました。 コミットによってタスクの本質を見つけるには、IDをコピーし、ブラウザーを開いてXPlannerでクロールする必要がありました-を参照してください。
  • さらに、 XPlannerはタスクステータスをサポートしていないため、テスターはどのタスクがまだテストされていないかを知るため、「テスト」履歴が開始されました。 タスクを完了した開発者は、手紙に加えて、そのストーリーで行われたタスクへのリンクを追加する必要がありました。
  • 次に、テスターは、「何か新しいものがあったかどうか」という主題について、それらを調べることを余儀なくされました。
  • タスクに関するコメントは使用されませんでした-誰もが重い「Vasya」を忘れることなく、タスクのテキストに直接それらを追加しました。
  • タスクには優先順位がないため、「（0）モジュール1のエラー」、「（1）それほど重要ではないエラー」のように呼び出されました。
  • タスクには追加のステータスはなく、タスクに関連しています（ブロック、重複）。 したがって、問題の名において、「LOCK」の構成を満たすことが可能でした。
  • 共通のタスクプールや開発中のタスクはありません。 プールには、「Infinity Iteration」が使用されました。 毎週の繰り返しが便利に収集されなかった場所から。
  • 便利なものはありません。タスクのフィルター（非常に重要です！すべてのプロジェクトのすべての重要なエラーの統計を表示できません）、配色（優先度0のタスクがすぐにわかるように）。
  • ワークフローはありません。 誰でもタスクを作成でき、誰でもタスクを閉じることができます。 テストされているかどうかを追跡することはできません。 （そして、かなり厳格な開発プロセスを持ち、作成、承認、実行、ブロック、テスト、クローズ、返品、拒否など多くの状態を持っている人はどうでしょうか？）
長所
  • 開発言語はJavaで、製品を開発します。これにより、原則としてそれを自分で変更できるようになりました。




Redmineについて



Redmine is a flexible project management web application. Written using Ruby on Rails framework, it is cross-platform and cross-database.





Redmineは、柔軟なソフトウェア開発管理ツールです。 RoRプラットフォームで開発され、さまざまなデータベース（SQLite、MySQL、PostgreSQL）を使用できます。

Redmineは一見したところ、インターフェイスからほとんどすべてのものを柔軟に設定できるため、Redmineが気に入っています。既に多くの便利な機能がありますが、同時に不要なものは何もないからです。



Redmineのクイック機能
  • 参考書。 ディレクトリは、インターフェースから変更できる変更不可能なサービスまたはユーザーエンティティのリストです。 次のものを構成できます。
    • タスクタイプのリスト（「エラー」、「リファクタリング」、「サポート」、「新機能」など）
    • タスクステータスのリスト（「作成済み」、「実行中」、「ブロック済み」など）
    • プロジェクトで可能な従業員の役割（マネージャー、開発者、テスター）
    • タスクの優先順位。
    • カテゴリ（プロジェクトのフレームワーク内で定義）。 実際-追加のタスク分割のためのプロジェクトのフレームワーク内のモジュールまたはサブシステム
    • フレックス属性-タスク、プロジェクト、ユーザー、およびタイムレポートのフィールドを追加できます。たとえば、ユーザーの場合は「オフィスの場所」、プロジェクトの場合は「クライアント側の連絡先」などです。
    • タスクのワークフロー -ステータスの関係-タスクの種類-ロール-可能な遷移の。 誰が、どの状態（タスク（:)）でどのタイプのタスクをどのステータスに変換できるかを決定します。たとえば、「マネージャー」と「顧客担当者」のみがタスクを開始でき、「顧客担当者」は「エラー」などのタスクのみを開始できます。 「テスター」のみがタスクを閉じることができます。
  • マルチプロジェクション。 人気のあるTracとは異なり、Redmineはすぐにマルチデザインをサポートします。 さらに、プロジェクトをネストすることができます。 たとえば、使用シナリオは次のとおりです。プロジェクト「CRM会社名」-製品およびCRMシステム実装のプロジェクト。 ネストされたプロジェクト「「ホーン」でのCRMの実装」、「「蹄」でのCRMの実装」
  • 権利の役割システム。 ロールは、ワークフローに参加する以外に、システム内でさまざまな権限を持っている場合があります。 数十個のチェックマークを配置して構成されます。 Redmineは顧客の外部ポータルとしても使用できます-「クライアント」ロールを設定するだけです。
  • カスタマイズ可能なTo Do リスト 。 もちろん、マネージャーのシステムのメインウィンドウはタスクのリストです。 以下を提供します。

    • 列のデフォルトリストの設定（列は不要-チェックなしで非表示）。
    • 列の並べ替え
    • フィルタリング。 フィルタリングは、いくつかのフィールドに従って可能です。例えば、「モジュールに関連するVasily Pupkinに割り当てられた0優先順位エラー」「認証およびアクセス権」ステータス「テスト送信済み」または「終了」。 作成されたフィルターは、自分用に（再度構成しないように）保存するか、すべてのプロジェクト参加者に保存してすぐにアクセスできます。 たとえば、「優先度0のタスクを開く」。
  • RSSまたはメール通知
    • 各リスト（フィルタリングを含む）は、CSV、PDF、RSSの形式で受信できます。 たとえば、シナリオは次のとおりです。テスターは、「テストのために送信され、RSSにサブスクライブします」というステータスのタスクを自分で作成します。それだけです。新しいタスクは自動的に彼に落ちます。
    • メール通知。 タスクの変更、評価、コメントの追加など。 影響を受けるユーザーへのメールのシステムスパム。 開発者は「タスクを閉じました」と書く必要はありません。 Redmineは彼のためにそれを書きます。


  • SCM（コード管理システム）との統合-SVN、CVS、Git、Mercurial、Bazaar、Darcs
    • 各プロジェクトのストレージを構成します。
    • リポジトリのWebベースのビュー。 システムは、コードリポジトリ 、 コミットの形式とファイルの 両方の変更履歴を表示するためのWebインターフェイスを作成し、バージョン（非バイナリファイルの場合）を比較してdiffを表示できます。
    • コミットの追跡。 コミットのコメントに「id 333」を指定することで（テンプレートはカスタマイズ可能）、開発者はコミットをタスクに自動的に添付できます。 コミットを表示すると、タスクの説明に移動でき、コミットのリストがタスクカードに表示されます。
    • コミット時の「クローズ」ステータスへの自動移行。 開発者が「固定333」（テンプレートはカスタマイズ可能）を書き込むと、コミットはタスクに添付されるだけでなく、タスクを目的のステータス（「テスト用に送信済み」など）に変換します。Redmineに移動する必要はありません。
  • タスク タスクカードには、上記に加えて、 ディスカッション履歴 （ RSSの形式で取得可能）、関連するリビジョンのリスト、バージョン変更の履歴、評価、およびその他のパラメーターが表示されます。
  • CSS CSSスタイルには特別な注意を払う必要があります。 システムはテーマ（スキン）をサポートします。テーマはCSSを使用して設定されます。 CSS内で非常に適切に使用されていますが、CSSを使用してデザインを変更できないオブジェクトはまだ見つかりません。 スキンの変更は、主にタスクリストに関連しています（これはプロジェクトマネージャーのメインウィンドウです。作成：ステータスの色表示（ロック-赤、テスト済み-緑、クローズ-グレー）。優先度の色と明るさの表示-（0-明るい、1-少ない）明るく、2番目以降はすべて同じです。タスクの説明の入力フィールドのフォントが増加しました-より便利です（：
  • プラグインシステム。 Redmineには、自分で記述できるプラグインシステムがあります。 現在、プラグインは10個ありますが、1か月前は3または4でした。
  • 良い検索
  • フォーメーションロードマップ 。 一般的な統計が表示され、 各バージョンの詳細ページがあります。
  • プロジェクトサマリー 。 概要には、プロジェクトの問題を解決するための一般的な概要統計が表示されます。
  • 個人用（パーソナライズ）ページ。 個人用ページには、割り当てられたタスクと、従業員が見たい他のブロック（カレンダー、費やした時間、ドキュメント、ニュース、追跡されたタスク）が表示されます。
