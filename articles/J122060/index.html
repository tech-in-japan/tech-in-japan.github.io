こんにちは、Habr。 私は多くの記事、より具体的には-バージョン管理システム（以下、ハードカレンシーと呼ぶ）の非標準（つまり非オリジナル）の使用を多くの方法で説明するトピックに触れることにしました。 この記事はあなたのためではないので、同志のプログラマー、腐ったトマトを隠して通り抜けましょう。 はい、皆さんは既にGit、SVN、CVSのすべての複雑さを研究しており、他の多くの流行語を知っています。 単なる人間である私たちに、SLEを使用するすべての利点を学ばせてください。

ハード通貨に精通したいすべての人と、急速に変化するデータに何らかの方法で対処しているすべての人を招待します。



なぜそれが必要ですか



私自身は工科大学の学生であり、ほぼ常にドキュメント（テキスト、図面、図面）を操作し、1日に3回（10、100）変更しています。 先週行われた編集をキャンセルして、1週間前の状態のドキュメントに戻す必要があることが判明する場合があります。 さて、少しの編集が行われた場合、この場合、Ctrl + Zで50回のヒットが役立ちます。 ただし、今週中にドキュメントの作業が多かれ少なかれ行われた場合、「1週間前に行われた重要なリビジョンに」ステータスを復元できません。 これを行うには、「重要なリビジョンの前」の時点でのドキュメントのコピーと、「別の重要なリビジョンへ」、「疑わしいリビジョンへ」、「キャンセルする必要がありそうな」リビジョンへのコピーがさらに12個必要です。 原則として、このアプローチは可能であり、多くの人によって実践されています。 最近まで、私自身が重要なバージョンのファイルを保持し、「date_time」というプレフィックスを付けて保存していましたが、喜んでいたようです。 この方法の利点は単純さであり、欠点は作業フォルダーの「膨張」と使用の不便さです。 そして、もしそれらのうちの最初のものが何らかの形で戦うことができるなら（大きなハードドライブと7zipを使って）、不便で何かをしなければなりませんでした。



これでできること、またはSLEとは



ウィキペディアから段落を切り取ります。「バージョン管理システム（英語版のバージョン管理システム、VCS、またはリビジョン管理システムから）は、変化する情報の処理を容易にするソフトウェアです。 バージョン管理システムを使用すると、同じドキュメントの複数のバージョンを保存し、必要に応じて以前のバージョンに戻って、誰がいつ変更を加えたかを判断できます。 ウィキペディア自体の原則のように見えます-すべての編集を含む記事のすべてのバージョンが学習に利用できます。

したがって、多くのバージョンのファイルを保存する必要がある状況でハード通貨を使用することが必要です。 このアプローチの利点には、使いやすさと、いわゆるデルタ圧縮による空きディスク容量の節約が含まれます（ファイルは異なるバージョンで保存されず、バージョンごとに変更されるため、保存されるデータ量が削減されます）。 やってみましょう。



SLEとは



同じウィキペディアでは、SLEはローションの有無にかかわらず、大小を問わず一元化および分散されていることが示唆されています。 ハードカレンシー機能の一部のみを（少なくとも最初は）使用するため、これに特に関心はありません。 これがまさに機能だと考えます。

ほとんどすべての硬貨は、何らかの種類のストレージを表し、作業するファイルのすべてのバージョンを保存します。 ここでは、保存されているファイルのバージョンがユーザーによって最も頻繁に決定されることを明確にする必要があります。 たとえば、約12の小さな変更を行い、アクティビティの結果をリポジトリに保存するときだと判断しました。 類推は定期的にCtrl + Sを押すことで頭に浮かびますが、唯一の違いは、このバージョンのファイルに将来アクセスできることです。 当然のことながら、この方法で一気に大量のファイルのバージョンをストレージに保存できます。 このアクションは、単純な方法で「コミット」または「変更のコミット」と呼ばれます。

いつでも、リポジトリに新しいファイルを追加したり、リポジトリ内の既存のファイルを削除したりできます（これはストレージのスマート名です）。ハード通貨は、いつ/何を追加/削除したかを「記憶」します。 また、commit'ahsのコメントのおかげで、この特定のコミットが実際に実行される理由を説明することもできます（「そこに安物の宝石を追加しました」/「そこから必要な部分を削除した」）。

1週間前のバージョンに戻る時が来たことをようやく理解すると、変更の全履歴があります。 そして、ここで何をするかを選択できます。 古いファイルから必要な部分をコピーして現在のバージョンに貼り付ける必要がある場合は、ストレージから古いファイルを抽出し、必要なファイルをそこからコピーします。 完全にロールバックして古いバージョンでの作業を続ける必要がある場合、SLEが再び支援します。以前のバージョンに戻って、いわゆる新しいブランチ（「ブランチ」）を作成できます。 1週間前のバージョン。 したがって、プロジェクトのバージョン履歴は、「ルート」（プロジェクトの開始）から「ブランチ」（編集の成功および失敗）まで、ツリー形式でグラフィカルに表すことができます。 さらに、たとえば、1つのソースファイルから2つの異なるバージョンを開発することを決定した場合など、「ブランチ」を人為的に作成することもできます。 さらに、作業ファイルがテキストドキュメント（およびその他のドキュメント）である場合、異なるブランチを1つに結合することができます-いわゆるマージ（「マージ」）。 ここで、数人の人々がプロジェクトに取り組んでおり、それぞれが独自の「安物の宝石」に従事していると想像してください。 この場合、共通のリポジトリを使用すると、開発が大幅に簡素化されます。



理論から実践まで、またはSLEの使用を開始



ですから、SLEを使用することは良いことだと確信していただければ幸いです。 ハード通貨の使用方法を学ぶことだけが残っています。 これが私たちがやることです。

使用のさまざまな面で互いに異なるさまざまなバージョン管理システムがあります。 私たちは（少なくとも最初は）さまざまなシステムの複雑さに興味がないので、最も単純で最も友好的なシステムについて説明します。 私の謙虚な意見では、奇妙なことに、このようなシステムはMercurialです。TortoiseHgグラフィカルシェルを備えた「非常に大きなコードリポジトリで効果的に動作するように設計されたクロスプラットフォーム分散バージョン管理システム」です。 システムの操作は、Windows、Linux、およびMac OS Xで可能です。

Windowsのシステムでの作業を説明する予約をすぐに行います。 Linuxを習得したので、すべてを類推することは難しくありません。

さらに、Mercurialリポジトリ（bitbucket.org）の無料ホスティングの使用方法についても学習します。これは、プロジェクトだけで作業していない場合に必要なものであり、非常に便利な場合はインターネット経由でプロジェクトのすべてのバージョンにアクセスしたい場合に使用します 実際、これを使用したことがある場合、これは便利なDropboxの代替品です。

開始するには、次の場所からMercurial + TortoiseHgをインストールします：tortoisehg.bitbucket.org。

このシステムはコンソールで機能するため、便宜上、後で通常の操作用にいくつかの* .batファイルを作成します。

すべての操作は、hgコマンドによって実行されます。 パラメータなしで呼び出されると、基本的なコマンドのリストが表示されます。

選択したディレクトリはリポジトリとして機能し（「C：\ project \」フォルダを使用します）、ここに将来のプロジェクトのすべてのファイルを保存する必要があります。 もちろん、同じコンピューター上に複数のリポジトリーを持つことを禁止する人はいません。

システムがリポジトリを作成することを「理解」するために、次のコマンドを実行します。

hg init c:\project





その後、フォルダー「c：\ project \」が作成されます（以前に作成されていない場合）、フォルダー「c：\ project \ .hg \」。Mercurialはすべてのサービス情報を保存します。

コンピューターのローカルリポジトリだけでなく、すべての変更を送信するリモートリポジトリも取得することをすぐに思い出します（または、賢い人が言うように、英語の「プッシュ」からリモートリポジトリに「プッシュ」変更）。 これを行うには、bitbucket.orgにアクセスして登録し、最初のリポジトリを作成します（リポジトリ-新しいリポジトリの作成）。 リポジトリに名前を付け（明確にするためにremote_projectと呼びます）、[リポジトリの作成]をクリックします。

現在、2つのリポジトリがあります。「c：\ project \」フォルダーにあるローカルリポジトリと「bitbucket.org/your_account_name/remote_project/」にあるリモートリポジトリです。your_account_nameはbitbucketで登録するときに指定したリポジトリ、remote_projectはリポジトリの名前、作成時に選択されました。

調査を続行するには、ローカルリポジトリに何かを置く必要があります。 その中に将来のプロジェクトのファイルを作成するか（私の場合、「c：\ project \」フォルダに）、現在のプロジェクトをそこにコピーします。

厳密に言えば、Mercurialに次のように伝える必要があります。「そのようなファイルといくつかの新しいフォルダーをプロジェクトフォルダーに追加しました。」これには「hg add」コマンドが提供されます。 ただし、別のアプローチがより便利です-次のコミットで、Mercurialにプロジェクトフォルダーから新しく作成されたすべてのファイルを取得し、削除されたファイルを忘れるように指示します。「新しいドキュメントを作成するたびにhg add c：\ project \ new_document.doc」を実行するよりもはるかに簡単です」

それでは、最初のコミットに取り掛かりましょう。 次のコマンドで実行されます。

hg commit –A –m “comment to commit”





順番に見てみましょう。 このコマンドは、リポジトリにいるときに入力する必要があります（つまり、最初に「cd c：\ project」を実行する必要があります）。 「-A」オプションは、Mercurialが新たに作成されたファイルを「ピックアップ」するために必要です（上記参照）。「-m」オプションを使用すると、コミットにコメントを追加できます。 これらのコメントは、TortoiseHgとbitbucket.orgのプロジェクトページでバージョン（または変更セット-変更リスト）を表示するときに表示されます。 後で苦しめられないように意味のあるコメントを与えることは非常に重要であり、これまたはその改訂がいつ行われたかを覚えています。

これでリポジトリにプロジェクトの初期バージョンが保存されました。 それ以降のすべてのコミットメントは、現在のバージョンを保存するときだと判断した後、同様に実行されます。

次のコマンドを使用して、コミットをリモートリポジトリに「プッシュ」できます。

hg push https://bitbucket.org/__/remote_project





同時に、リポジトリに対応するフォルダにもいる必要があります。 コマンドを入力すると、bitbucket.orgのアカウントの名前とパスワードが要求されるため、プッシュするたびに入力しないように、コマンドを次のように置き換えることができます。

hg push hg push https://__:__@bitbucket.org/__/remote_project





すべてのコマンドを* .batファイルに変換するため、この場合、パスワードは平文で保存され、セキュリティ上のリスクが生じますが、私にとってはこれで問題ありません。

そのため、便宜上、次の内容のcommit.bat、push.batおよびcommit＆push.batファイルを直接到達ゾーンに作成します。

[commit.batファイルの内容]

IF !%1==! goto exit1

cd C:\project

hg commit -A -m "%*"

goto exit0

:exit1

echo "NO COMMAND-LINE ARG!"

:exit0





引数付きで呼び出されたこのファイルは、コミットコメント内の引数を使用してプロジェクトコミットを実行します。 例：「commit.bat my first commit」を実行し、コメント「my first commit」でコミットを取得します。 FARでは、Ctrl + Enterの組み合わせを使用すると便利です。

[push.batファイルの内容]

cd C:\project





hg push https://__:__@bitbucket.org/__/remote_project





このファイルはリモートリポジトリにプッシュされます。

[commit＆push.batファイルの内容]

IF !%1==! goto exit1

cd C:\project

hg commit -A -m "%*"

goto exit0

:exit1

echo "NO COMMAND-LINE ARG!"

:exit0

call ./push.bat





引数付きで呼び出されるこのファイルは、順次コミットを実行し、コミットするコメントに入力された引数を使用してプロジェクトをプッシュします。

さらに、小さな中間コミットの場合、commit_date_time.batファイルを作成することをお勧めします。

[commit_date_time.batの内容]

cd C:\project

hg commit -A -m "%DATE% %TIME%"





このファイルは、現在の日付と時刻をコメントとしてコミットします。これはしばしば便利です。

誰もが編集の強度と複雑さに応じてcommit'ovとpush'eyの頻度を個別に決定します。 「より頻繁に-より良い」というルールに従うことをお勧めしますが。

リポジトリのファイル/フォルダーを右クリックすると、リポジトリエクスプローラー（TortoiseHg-リポジトリエクスプローラー）を起動できます。これにより、コミットのすべてにコメントが表示されます。 このウィンドウには、リポジトリのツリー構造が表示されます。ここから、以前のバージョンへのコミット、プッシュ、ロールバック（バックアウト）およびその他の操作を行うこともできます。

bitbucket.org/your_account_name/remote_projectで同様の変更セットを見つけることができ、1つのアーカイブにプロジェクトの任意のバージョンをダウンロードできます。これは非常に便利な場合があります。
