WPF：簡単なコンバーターなしのバインド

こんにちは



WPFで新しいプロジェクトの作成を開始するたびに、ブール変数の否定にアタッチするか、ブール変数をVisibility型に変換するために、各バインディングで指定される独自のコンバーターを作成する必要があるという考えに悩まされました。 そして、2つの数値の合計を出力する必要がある場合、または単に数値を2で除算する必要がある場合は、何も追加および除算したくないほど多くのコードを記述する必要があります。



この問題を解決するために、1つまたは複数の標準的なバインディングの類似物を作成しました。これにより、1つ以上のバインディングソースから任意の式にバインドできます。 これがどのように機能し、どのように使用するかについて、さらに詳しく説明したいと思います。



比較して、標準のバインディングと新しいバインディングを使用して行われた同じバインディング（c：バインディング）：



宛先： 



<Label> <Label.Content> <MultiBinding Conveter={x:StaticResource MyCustomConverter2}> <Binding A/> <Binding B/> <Binding C/> </MultiBinding> </Label.Content> <Label>


後： 



 <Label Content="{c:Binding A+B+C }" />


宛先： 



 <Button Visibility="{Binding IsChecked Converter={x:StaticResource NegativeBoolToVisibilityConveter}}" />


後： 



 <Button Visibility="{c:Binding !IsChecked}" />


これらの例からわかるように、新しいバインディングは、Pathプロパティの1つ以上のソースプロパティから任意の式を受け入れることができます。 すべての基本的な算術演算と論理演算、および行追加演算と三項演算子がサポートされています。



他の例：



数学的操作： 

 <TextBox Text="{c:Binding A+B+C}"/> <TextBox Text="{c:Binding ABC}"/> <TextBox Text="{c:Binding A*(B+C)}"/> <TextBox Text="{c:Binding 2*AB*0.5}"/> <TextBox Text="{c:Binding A/B, StringFormat={}{0:n2} --StringFormat is used}"/> <TextBox Text="{c:Binding A%B}"/> <TextBox Text="{c:Binding '(A == 1) ? 10 : 20'}"/>


論理演算： 

 <CheckBox Content="!IsChecked" IsChecked="{c:Binding !IsChecked}"/> <TextBox Text="{c:Binding 'IsChecked and IsFull'}"/> {'and'  '&&' . } <TextBox Text="{c:Binding '!IsChecked or (A > B)'}"/> {'or'  '||',    '||'} <TextBox Text="{c:Binding '(A == 1) and (B less= 5)'}"/> {'less='  '<=' . } <TextBox Text="{c:Binding (IsChecked || !IsFull)}"/>


ブール値と可視性を使用する： 

 <Button Content="TargetButton" Visibility="{c:Binding HasPrivileges, FalseToVisibility=Collapsed}"/> <Button Content="TargetButton" Visibility="{c:Binding !HasPrivileges}"/> <Button Content="TargetButton" Visibility="{c:Binding !HasPrivileges, FalseToVisibility=Hidden}"/>


文字列を使用する： 

 <TextBox Text="{c:Binding (Name + \' \' + Surname)}" /> <TextBox Text="{c:Binding (IsMan?\'Mr\':\'Ms\') + \' \' + Surname + \' \' + Name}"/>


数学クラスでの作業 

 <TextBox Text="{c:Binding Math.Sin(A*Math.PI/180), StringFormat={}{0:n5}}"/> <TextBox Text="{c:Binding A*Math.PI}" /> <TextBox Text="{c:Binding 'Math.Sin(Math.Cos(A))'}"/>

可視性へのブール



新しいバインディングは、プロパティがそのようなタイプに関連付けられている場合、boolタイプをVisibilityタイプに自動的に変換します。 デフォルトでは、falseはVisibility.Collapsedに変換されると見なされますが、この動作はオプションのFalseToVisibilityプロパティ（CollapsedまたはHidden）を設定することで変更できます。 



 <Button Visibility="{c:Binding IsChecked, FalseToVisibility=Hidden}" />


CalcBindingの内部構造



ハムルを拡大する可能性について最初に学んだ人のために、私は少し余談します。 中かっこを使用してxamlからアクセスするものはすべて、C＃で記述された非常に普通のクラスですが、常にMarkupExtensionクラスから継承されます。 このようなクラスはマークアップ拡張機能と呼ばれます。 WPFを使用すると、独自のカスタムマークアップ拡張機能のセットを補完できます。 [1]のように、多くの記事がそのような追加を書くことに専念しており、それらを書くのは本当に簡単です。



Bindingクラス（本来はマークアップ拡張機能）の研究を始めたとき、最初に考えたのは、このクラスから継承し、特定の場所でバインディング動作を変更することでした。 しかし、残念ながら、動作を変更する必要があるProvideValueメソッドは、BindingBase親クラスで封印済み（sealedキーワードで宣言）としてマークされていることがわかりました。 この整理により、新しい空のマークアップ拡張機能を作成し、標準のBindingおよびMultiBindingで使用されるすべてのプロパティをコピーする必要がありました。 もちろん、このようなソリューションでは、BindingおよびMultiBindingの変更のためにクラスを変更する必要がありますが、WPFが長い間開発されていないことを覚えていれば、それは私を脅かすものではないと思います。



CalcBindingはどのように構造化されていますか？ ProvideValueメソッドの開始時に、Pathプロパティが分析され、Pathに1つの変数が含まれる場合、Bindingが作成されます。それ以外の場合、各変数に1つのBindingを含むMultiBindingが作成されます。 すべてのCalcBindingプロパティの値は、作成されたBindingまたはMultiBindingにスローされ、コンバーターはIConverterおよびIMultiConverterインターフェイスを実装し、Pathのコンパイルと結果の匿名関数の実行を行うコンバーターとして転送されます。 結果のBindingまたはMultiBindingメソッドはProvideValueメソッドを呼び出し、外部ProvideValueの結果として結果を返します。 したがって、WPFはバインディングの標準クラスで動作し、私のクラスは一種のファクトリーとして機能します。



前述のように、新しいソリューションが古いソリューションの速度を失わないようにするために、最初にConvertまたはConvertBackメソッドが呼び出されたときに、Pathプロパティで指定された元の式が1回だけコンパイルされます。 コンパイルの結果、ソースプロパティをパラメーターとして受け取り、ターゲットプロパティがバインドされる匿名関数を取得します。 ソースプロパティのいずれかを変更すると、結果の関数が新しいパラメーターで呼び出され、それに応じて新しい値が返されます。



文字列式の解析は2段階で行われます。最初の行では式のツリー（System.Linq.Expressions.Expression）が構築され、2番目の行では結果の式が標準メソッドを使用してコンパイルされます。 文字列からExpressionを作成するために、Microsoftのパーサーがあります。これはDynamicExpressionライブラリ[2]にあります。 残念ながら、このソリューションを変更せずに使用できない問題、たとえばバグが明らかになりました[3] 。 幸いなことに、パーサーはオープンソースにアップロードされ、DynamicExpresso [4]と呼ばれるフォークを使用して、この問題やその他の問題を解決し、サポートされているタイプのリストを拡張しました。



詳細を省略すると、Convertコンバーターメソッドのロジックは次のように表示できます。 



 private Lambda compiledExpression; public object Convert(object[] values, Type targetType, object parameter, CultureInfo culture) { if (compiledExpression == null) { var expressionTemplate = (string)parameter; compiledExpression = new Interpreter().Parse(expressionTemplate, values.Select(v => getParameter(v)).ToList()); } var result = compiledExpression.Invoke(values); return result; }




バックバインディング



ソースからターゲットプロパティへのサイドへのバインドが正常にテストされた後、逆関数を自動的に検索し、反対方向にバインドできるようにしたいと考えました。



逆関数を作成できるようにするためには、引数が1つだけであり、この引数が1回だけ発生する必要があることは明らかです。 これらは必要ですが、不十分な条件です。 学校の数学のコースから、Y = F（X）がY = F 1 （F 2 （F 3 ...（F N （X））））として表現できる複素関数である場合、X = F -1 （ Y）= F N -1 （F N-1 -1 （F N-2 -1 （...（F 1 -1 （Y）））））



したがって、複雑な逆関数を作成するには、この関数を構成するすべての関数の逆関数を見つけて、逆の順序で適用する必要があります。



プログラミングに戻りましょう。 プログラムの観点から、直接関数を実装する既知の式ツリーから逆関数を実装する式ツリーを構築する必要があります。 Expressionに導入された制限により、作成された式ツリーを変更することはできないため、新しい式ツリーを構築する必要があります。



これを達成するために必要なことの例を見てみましょう。 たとえば、元の関数は次のとおりです。 

 Path = 10 + (6+5)*(3+X) - Math.Sin(6*5)




この式を使用して、次のツリーが構築されます。







Xを含むシートからパスの最上部までのパスが直線であり、残りのノードが上下にあるような方法で想像してください。







この図では、式の逆ツリーを構築するために、変数Xを持つリーフからルートへのノードにあるすべての関数を、逆によるPath結果で置き換えてから、アプリケーションの順序を逆にする必要があることが明らかになります。







定数値を計算するブランチを反転する必要はありません。 その結果、逆関数を取得する式の逆ツリーを取得します。 

 X = ((Path - 10) + Math.Sin(6*5)) / (6 + 5) - 3


逆ツリーの変数検索、検証、構築は、たった1つの再帰関数によって実行されます。



現在、次の関数リストがサポートされており、その逆関数は自動的に決定されます。

  • +、-、*、/
  • ！
  • Math.Sin、Math.ASin
  • Math.Cos、Math.ACos
  • Math.Tan、Math.ATan
  • Math.pow
  • Math.log


結果の式ツリーは、フィードバックが最初にトリガーされたときに1回だけ計算され、匿名関数にコンパイルされます。



ソリューションの欠点



他のように、このソリューションにはいくつかの制限と欠点があります。 特定された欠陥を以下にリストします。



1.）sourcePropertyの1つがnullの場合、typeof（null）は何も返さないため、Expressionの作成段階で型を判別できなくなります。 これにより、たとえばそのような式を正しく処理することができなくなります。 

 <Label Content="{c:Binding Path = (A == null) ? 3 : 4}" />


残念ながら、Bindingクラスからソースプロパティタイプを直接見つけることはできないため、唯一可能な解決策は、リフレクションを使用してソースプロパティタイプを認識することですが、この場合は、Source（RelativeSource、ElementNameなど）を検索して、バインディング機能の半分を実装する必要があります。 おそらく、この問題に対する代替ソリューションがあるので、このトピックに関する提案を喜んでいます。



2.）xamlマークアップはxmlマークアップであるため、多数の文字、たとえば開始タグやアンパサンドアイコンは禁止されています。これは、「more」および「logical AND」の記号も意味します。 禁止された文字から逃れるために、Pathはこれらの演算子の代わりに以下に示すいくつかの置換を使用します。



オペレーター パスの置換 ご注意
&& そして
|| または 対称性のために導入、オプション
< 少ない
<= 少ない=




3.）CalcBindingで独自のコンバーターを設定することはできません。 私はそのような機会を必要とするシナリオを思いつきませんでしたので、何か提案があれば、私はそれらを読んでうれしいです。
