TU60テープドライブおよびTA11コントローラー



70年代、DECは、さまざまなシンプルなオーディオテープに基づいた、パンチテープの簡単な代替品を作成しました。 TU60は2つのストリーマーユニットを収容します。 50メートルのテープは最大100Kbを保存できます。







すべてのドライブロジックは2つの6ピンボードに含まれています（ これについては前に書きましたが、統合は良好です ）。







上面図では、モーターを制御し、磁気ヘッドからデータを読み取るアナログロジックを備えたボードを見ることができます。 従来のオーディオカセットとは異なり、TU60ドライブはテンションドラムを使用してテープ速度を制御しません。 代わりに、2つの独立したモーターが使用されます。 1つは、カセットを巻き戻し、張力をかけたままにするためのものです。 2つ目は、サーボによって駆動され、テープを前方に引っ張ります（ 実際、これは計画の地獄です。カセットが再生中または一時停止しているとき、最初のモーターもオンになります。たとえば、1つは100回転を巻き戻そうとし、2つ目は-そして、より強力なものが勝ちますが、もちろん、磁気テープはあまり好きではないので、このステーションを長時間オンにしておくことはお勧めできません。 ） 2番目のモーターの回転速度は一定です（ テープを読み取るときに一定であると完全に正確に言われているわけではありませんが、同じです！速度はタスクによって異なります。たとえば、テープがアイドルのとき、このモーターは最初の逆モーターと組み合わせて張力を与え、ブローチ加工時よりも速度が遅くなります ）、テープのビットレートも同じであるため、テープの全長にわたって情報密度が変化します。 TU60は、マンチェスターコードを使用してデータをエンコードします。



TA11コントローラモジュールは、4ピンM7892 Unibusボードで表されます（ 前の記事で1つのボードの異なる名前についても書きました ）。







コントローラーは、2つのレジスターの単純なインターフェースを提供します：コマンドステータス（ 頻繁に練習しても、1つの形式で読み取り用のレジスターを与え、もう1つの形式で書き込みを受け入れます。ここではすべてが異なります。これは全体的なレジスターですが、1つのフィールドは読み取り専用です他は書き込み専用ですが、読み取り/書き込みが混在しています。 ）、0177500で使用可能、0177502にデータバッファーがあります。ボードはTU60とPDP-11の間の仲介者であるだけでなく、追加のロジック（作業のチェーン）も含みます。割り込みやアドレスデコーダ（レジスへの参照をキャッチする必要があります フレーム）。



このデバイスで遊んでいると、すぐに問題に遭遇しました。 巻き戻しボタンは正常に機能するように見えました。 しかし、その後、突然、彼女は故障しました。 モーターの軸とカセットを接続する三角形の機構が軸から外れることは明らかでした。 図面によれば、＃2-56 1/8 "（ インチサイズ ）の止めネジがあるはずです。しかし、それはそこにはありませんでした。そして、それがどのように消えるのかは明確ではありませんでした。2番目のドライブにも同じ問題がありました。スウェーデンでは、メトリックシステムを使用しているためです。）私はより詳細に図を見ると、下の三角ヒッチに「止めネジ」の代わりに「ロクタイト」という碑文があります。いずれにせよ、私は接着剤を買いました 素晴らしい！



ただし、MaindecのZTAA診断テストは失敗しました。 私が最初に発見した問題は、制御ボードにあるDEC8881ドライバーで、EOTおよびBOT信号の通過を妨げていました。 私はそれを7439に交換しました。これはどこか中国で3ドル/個で購入しました。 興味深いのは、DEC8881のはんだ付けを解除したとき、下から7439というラベルが付けられていたことです！ その後、診断は2番目のストリーマーで行われ始めましたが、最初のストリーマーでは行われませんでした。 前進駆動はまったく機能しませんでした。 段階的に、障害のあるオープンコレクタードライバー75452を追跡しました。最後に、両方のドライブがZTAAを通過しました。 しかし、ZTABではありません...

CAPS-11を起動します



CAPS-11オペレーティングシステムでテープイメージを取得しました。 ブートカセットには、CAPS-11 OS自体の初期ロードを提供するCTLOAD.SYSファイルが含まれていました。 たぶん、「オペレーティングシステム」は大声で言っているかもしれませんが、CAPS-11はファイルをテープにロードおよび保存する機能を備えたかなり単純なコマンドラインパーサーです。 システム自体はCAPS11.S8Kファイル内にあり、8キロメモリの環境で動作するようにリンクされていました（ PDPのメモリはプロセッサワードで測定されました。ちなみに、8進数システムは非常によく使用されました。 。 このOSの起動プロセスは非常に複雑です。 ブート領域（01000にある）はCBOOTと呼ばれます。 CTLOAD.SYSファイルの最初の32バイト（テープの最初のバイト）をスキップします。 次に、次の128バイトをメモリのアドレス0に読み込みます。CTLOAD.SYSのこの部分はPRELDRと呼ばれます。 PRELDRは使用可能なメモリのサイズを見つけ、すべてのCTLOAD.SYSを可能な限り高いアドレスのメモリにロードします。 ロードユニットにはCABLDRとCBOOT（ミラー）が含まれています。 CABLDRはカセットアブソリュートローダーですが、名前にもかかわらず、読み取り可能なファイルの形式はパンチテープの場合とまったく同じです。 したがって、CAPS11.S8Kにはパンチテープの形式があり、PDP11GUIを介して簡単にダウンロードできます（ このプログラムについては、前の記事を参照してください ）。 ただし、CABLDRはプログラマのコンソールによって提供されるスイッチレジスタを使用して、何をどのようにロードするかを決定するため（たとえば、このレジスタでシステムリベースアドレスを設定し、ユーザーアドレスにロードできます ）、いいえ、CABLDRは存在しないメモリを除いてマシンを停止します。



021510の開始アドレスでPDP11GUIを使用すると、CAPS-11プロンプトを表示できます。 70年代初頭からのシステムは2014年に開始されました。40年：

ベアシステムでのCコードの実行



TU60の問題に対処するには、記録された一連の操作を目的の順序で繰り返すだけのこのデバイスが必要です。 残念ながら、XXDPでの診断の作業はまったく予測できません。 読み取り操作と書き込み操作の両方で同じエラーコードを出力できます。 少なくとも、何が間違っているのかを特定することはできません。 さらに、ロジックアナライザーを使用すると、同じ測定を何度も実行できますが、研究のテストポイントをわずかに変更するだけです。 先に進むための最良の方法は、独自の簡単なテストプログラムを書くことだと決めました。 それ以前は、データをテープに読み書きする小さなアセンブラーコードを書きました。 しかし、PDP-11の経験豊富なアセンブラープログラマーではないため、コードの量がわずかに増えたとしても、ますます多くのデバッグ作業が必要になりました。 私はCが大好きです。



Cプログラムを作成するには何が必要ですか？ もちろん、クロスコンパイラーです！ ダイアン・ナイサスのページに触発されました。 GCC 4.8.2とともにBinutils 2.24をダウンロードしてコンパイルしました。



Binutils： 

src/configure --target=pdp11-aout --disable-nls make all sudo make install


GCC： 

 src/configure --target=pdp11-aout --disable-nls --without-headers --enable-languages=c make all-gcc sudo make install-gcc


次に、CRTのエントリポイントであるcrt0.sが必要です。 私はちょうど働いた最小限のバージョンを書いた： 

  .text .globl _main, ___main, _start _start: mov $400,sp jsr pc, _main halt ___main: rts pc


フォーマットされた行の出力は非常に有用なものなので、 組み込み開発用の小さなprintf実装を検索して見つけました。 ライブラリをコンパイルしてリンクしようとすると、多くの関数が見つかりませんでしたというメッセージを受け取りました-___ mulhi3、___ udivhi3など。 関数は、ネイティブPDP-11アセンブラー命令の代わりにgccによって使用されました。 乗算、除算、および残りの計算を実行するこれらの単純な関数の独自のバージョンを作成する必要がありました。



現在、PDP-11 / 10と互換性のある出力ファイルの作成に使用される-m10フラグを使用したコンパイル中に、短から長に変換するコードの生成中にコンパイラがクラッシュしました。 私が見つけた唯一の解決策は、PDP-11 / 40用にコンパイルし、SXTやASHCなど、PDP-11 / 04でサポートされていない命令を手動で置き換えることでした。



その後、私は本当のHello Worldを書く準備ができました。 



 pdp11-aout-gcc -m10 -Ttext 1000 -msoft-float -nostartfiles -nodefaultlibs -nostdlib crt0.s printf.c test.c divmulmod.s pdp11-aout-objcopy -O binary a.out hello.dmp


コードをアドレス0x1000または010000にリベースする必要があることをリンカーに通知し、objdumpを使用してオブジェクトをクリーンなバイナリファイルに変換しました。



最初に、実際のハードウェアをオンにする前に、このファイルをE11エミュレーターで実行しようとしました。 ただし、実際のマシンでPDP11GUIを使用して起動した場合のスクリーンショットは次のとおりです。







いいね！ フォーマットされた出力は正常に機能します。 乗算、除算、および剰余の取得も正しい結果になります（まあ、実際には、実際にはそうではありません。剰余を取得する機能には、修正しなければならないバグが含まれていることがわかりました）。 これで、古いTU60をロードするCテストプログラムを簡単に作成できました。 最後に、シリアルポートを介してファイルを読み取り、TU60にファイルを送信するプログラムを作成して、ブートカセットを作成できると思います。 いずれにせよ、CTLOAD.SYS内でCABLDRにパッチを適用して、スイッチレジスタを読み取ろうとしないようにします。これにより、例外が必ずスローされます。 （マシンにはプログラミングコンソールがありません）。



TU60EXERCISER



TU60EXERCISERという小さなプログラムを書くのに数時間費やしました。







これで、標準のZTA **プログラムを使用するよりも簡単な方法で、ドライブに対する操作を実行できます。 READコマンドをテストすると、オープンコレクタードライバーDEC8881（ これは以前に交換されたものではなく、別のDEC8881です）に問題が見つかりました。 これは 、ロジックアナライザーをTU60データバッファーに直接接続したときに気付かなかったものです。 問題は、CRC生成、または逆にCRC検証が失敗することでした。 データを正しく読み取っていた場合でも、データを読み取るときに常にCRCエラーが発生しました。

CRCの操作ロジックのエラーを検索します



以下は、テープに「AB」文字を書き込むとどうなるかを示す4枚の写真で、16ビットのチェックサムが生成されます。 データのLSBが最初に書き込まれ、WRITED信号がアクティブLowレベルに設定されます。



重要な点は、CRCは、ビット15、13、および0の入力信号がXORゲートを通過するフィードバック付きのシフトレジスタによって生成されることです。







エラーは明らかですか？ 私はすぐに彼女に気付きませんでしたが、数時間後に再び写真を調べて見つけました！



ビット5は高レベルでスタックしています。 しかし、常にではありませんが、鋭い目をした同志がどのように私を修正することができます。 最初は、すべてのビットがゼロです。 そのとおりですが、クリア信号を受信したという事実により、シフトレジスタ全体がリセット状態になります（ 他の多くの例と同様に8271にはクリア状態ピンがあります ）。 この場合、出力は正しいですが、データが連続してシフトレジスタに入力されると、誤った結果が返されます。 常に1シフトされます。 この要素は、Texas Instruments SN74179と同様のSignetics N8271チップによって実装されます。

いくつかの失敗が待っていました。 最終的にブルガリアから74179を入手してはんだ付けし、適切なエミュレーター設定でTU60EXERCISERプログラムをE11でテストした後、カセットが故障し始めました。 これは、ヘッドとテープの間の摩擦が非常に大きいため、ストリーマーがそれらを正しく巻き戻すことができないという事実に現れました。 さらに、テープの断続的な移動のために、ドライブが何かを読み書きできない状況が発生しました。



これは、古いカセットの一般的な問題であることが判明しました。 Deoxit D5を手元に持っていませんでした（ 参考文献では、電気機械化合物用の特別な潤滑剤が使用されていました ）。 CRCからシリコンスプレーを試しましたが、目立った効果はありませんでした。

コンソールエミュレータで「CT」と入力した後、ストリーマーは最初に短時間カセットを回しましたが、その後停止しました。 CAPS-11のマニュアルを読み直し始めたところ、次のことに気付きました。

PRELDRは、システムテープの最初のファイルの最初のエントリである必要があります。 このプリローダーは、CBOOT Loader Formatに従って書かれた小さなプログラムです。CBOOTLoader Formatは、メモリのサイズを決定し、後続のプログラムを上位メモリアドレスにロードするのに十分なほど高度です。 CBOOTはアドレス0のメモリにリンクし、自動的に起動します。 このブート段階のCAPS-11メモリカードは、図E-3にメモリマップ＃2として示されています。



PRELDRは、スキームによって判断されるCBOOTが01000にあるという事実に依存していますか？ もちろん、ROMから起動します。



編集：問題の本当の原因は、仲間のコレクターのおかげで見つかりました：CBOOTは、TA11 CSR（ ストリーマーコントローラーのコマンドステータスレジスタ ）をR0（ ユーザープロセッサレジスタ ）に配置し、M9312 CT（ M9312はコンソールエミュレーター、CTはこのエミュレーターのコマンドですが、同時に、これはテープからのブートコードを含むEPROMチップの名前です。コマンドはこのEPROMのコードのみを起動しますが、システムの起動時に自動的に起動を試みます。 ）ロード時に、TA1 CSRをR1（およびR0のストリーマー番号）に入力します。 PRELDRは、レジスタTA11の値をR0で正確に見ることを期待しています。 CT-ROMから直接起動できるようにPRELDRにパッチを適用する必要があります。



E11を使用してソースからCBOOTを構築し、結果をCBOOT.BINファイルに書き込みました 。 その後、pdp11guiを介してメモリにロードし、アドレス01000から起動しました。はい！ カセットは約30分間再生を開始し、「。」記号が1つ現れました。 CAPS-11を起動しました！
