sudo rm -rf、または2017/01/31からのGitLab.comデータベースでのインシデントの記録



彼はゆっくりと酔っていたが、それでもジャンプで酔っ払った。 そして、悟りの瞬間に、彼は目の前で完全になじみのない部屋で切り刻まれたオークのテーブル、彼の手に描かれた剣、周りのお金のないドンをたたくのを見たとき、彼は家に帰る時だと思った。 しかし、手遅れでした。



アルカディとボリス・ストルガツキー

GitLab.comの管理者の1人がレプリケーションを修正し、コンソールを混同し、メインのPostgreSQLデータベースを削除したため、OpenSourceの世界にとって重要なイベントが2017年1月31日に発生しました。その結果、大量のユーザーデータが失われ、サービス自体がオフラインになりました。 さらに、5つの異なるバックアップ/レプリケーション方法はすべて動作不能であることが判明しました。 データベースが削除される6時間前に誤ってLVMイメージから回復しました。 彼らが言うように、それは起こります。 しかし、私たちはプロジェクトチームに敬意を表さなければなりません。彼らはすべてをユーモアで扱う強さを見出し、頭を失いませんでした。Twitterですべてについて書いて、実際には、チームがリアルタイムで投稿した内部ドキュメントを共有することで、驚くべきオープン性を示しました展開イベントの説明。







それを読んでいると、あなたは文字通り貧弱なYPのように感じます。苦しい一日の後の午後11時にPostgresとの闘争に失敗し、疲れて目を細めて、致命的なsudo rm -rf



を戦闘サーバーのコンソールに押し込み、Enterを押します。 しばらくして、彼は自分がしたことを理解し、削除をキャンセルしますが、手遅れです-これ以上のベースはありません...







このケースの重要性と多くの点で有益なため、インシデントの作業中にGitLab.comスタッフが作成した彼のジャーナルレポートをロシア語に完全に翻訳することにしました。 カットの下で結果を見つけることができます。







それでは、それがどのようであったかを詳細に調べましょう。

被った損失



  1. 約6時間でデータが失われました。
  2. 4,613の通常プロジェクト、74のフォーク、350の輸入品が（大体）失われました。 Gitリポジトリは失われないため、データが失われる前にユーザー/グループが存在していたプロジェクトを再作成できますが、これらのプロジェクトの問題を復元することはできません。
  3. 約4979（約5000）のコメントを失いました。
  4. 潜在的に失われた707人のユーザー（Kibanaのログによると、より正確に言うのは困難です）。
  5. 1月31日17:20より前に作成されたWebフックは復元され、後に作成されます-は失われます。


タイムライン（UTCで示される時間）



  1. 2017/01/31 16：00/17：00-21:00

    • YPはステージングでpgpoolとレプリケーションの設定に取り組んでおり、LVMスナップショットを作成して戦闘データをステージングにロードします。また、このデータを使用して他のレプリカへのデータベースのロードを高速化できることを期待しています。 これは、データ損失の約6時間前に発生します。
    • レプリケーションの設定には問題があり、非常に時間がかかります（初期同期pg_basebackupの場合のみ約20時間）。 YPはLVMスナップショットを使用できませんでした。 この段階での作業は中断されました（YPは、その日は働いていなかった別の同僚の助けが必要だったため、またGitLab.comのスパム/高負荷のために）。
  2. 2017/01/31 21:00- スパマーによるサイトの負荷の急増 -Twitter | たるみ

    • IPアドレスでユーザーをブロックします。
    • CDNとしてリポジトリを使用するためのユーザーの削除。その結果、47,000人のIPユーザーが同じアカウントでログインしました（データベースの負荷が高くなります）。 情報は、テクニカルサポートおよびインフラストラクチャチームに転送されました。
    • スパムのユーザーを削除する（スニペットを使用） -Slack
    • データベースの負荷は通常に戻り、いくつかのPostgreSQLテーブルのバキュームが手動で開始され、多数の空の行が削除されました。
  3. 2017/01/31 22:00- レプリケーションラグに関する警告を受け取りました -Slack

    • db2の修正を試みますが、この段階では4 GB遅れています。
    • db2.clusterは複製を拒否し、ディレクトリ/ var / opt / gitlab / postgresql / dataはクリーンな複製を確保するためにクリーンアップされます。
    • db2.clusterはdb1への接続を拒否し、max_wal_sendersが低すぎると誓います。 この設定は、WAL（レプリケーション）クライアントの数を制限するために使用されます。
    • YPはdb1によってmax_wal_sendersを32にインクリメントし、PostgreSQLを再起動します。
    • PostgreSQLは、開いているセマフォが多すぎて起動しないことを誓います。
    • YPはmax_connectionsを8000から2000に削減し、PostgreSQLが起動します（8000でほぼ1年間正常に動作したという事実にもかかわらず）。
    • db2.clusterはまだ複製を拒否しますが、接続について不平を言うことはなくなり、代わりにハングし、何もしません。
    • この時点で、YPは絶望感を抱き始めます。 その日前に、彼は遅すぎたので（現地時間の23:00頃）作業を終了することを発表しましたが、予期しない複製の問題のためにそのままでした。
  4. 2017/01/31 23:00頃

    • YPは、おそらくpg_basebackupがデータディレクトリのクリーンさについてあまりにもtoo慢すぎると考え、それを削除することにしました。 数秒後、彼はdb2.cluster.gitlab.comではなくdb1.cluster.gitlab.comでコマンドを実行したことに気付きました 。
    • 2017/01/31 23:27：YPは削除をキャンセルしますが、手遅れです。 約310 GBのうち、残っているのは4.5- Slackだけです。

    問題点



    1. デフォルトでは、LVMスナップショットは24時間に1回のみ取得されます。 幸いなことに、YPはクラッシュの6時間前に手動で作成しました。
    2. 定期的なバックアップも1日に1回しか行われていないようですが、YPはまだそれらの保存場所を把握していません。 JNによると、それらは機能しません。サイズが数バイトのファイルが作成されます。

      • SH：9.6ではなくPostgreSQL 9.2のバイナリが実行されるため、pg_dumpは正しく機能しないようです。 これは、データ/ PG_VERSIONが9.6に設定されているが、このファイルが作業ノード上にない場合、オムニバスはPg 9.6のみを使用するためです。 その結果、9.2はデフォルトで起動し、何もせずに静かに終了します。 その結果、SQLダンプは作成されません。 Fog-gemは古いバックアップをクリアした可能性があります。
    3. Azureのディスクスナップショットは、NFSサーバー用、データベースサーバー用に含まれています-いいえ。
    4. 同期プロセスは、ステージングでデータを同期した後、Webフックを削除します。 24時間以内に作成された通常のバックアップからそれらを引き出すことができない場合、それらは失われます。
    5. レプリケーション手順は非常に壊れやすく、ランダムなシェルスクリプトに依存するエラーが発生しやすく、文書化が不十分であることが判明しました。

      • SH：後で、gitlab_replicatorディレクトリのスナップショットを作成し、レプリケーション構成を削除し、別のPostgreSQLサーバーを起動することで、ステージングデータベースの更新が機能することがわかりました。
    6. S3バックアップも機能しません。フォルダーは空です。
    7. バックアップの作成に失敗した場合の信頼できる通知システムはありませんが、dev-hostで同じ問題が発生するようになりました。


    言い換えると、使用されている5つのバックアップ/レプリケーション方法のうち、機能するものはありません。 =>現在、6時間前に作成された作業用バックアップを復元しています。
