Proxmox 3.1：すぐに使えるインストールのニュアンス

Habr上のProxmox仮想化システムに関する記事がいくつかありました。もちろん、この記事を書く前に、自分の実験のためにそれらを勉強しなければなりませんでした。 私の研究は、ベアメタルへのProxmox 3.1のインストール、構成、および既存の仮想マシンのVMWareから新しいシステムへの移行にまで及びました。 OSのインストールは、投稿の名前が示すように、Proxmox 3.1配布キットから「現状のまま」実行されました。 仮想マシンのインストール、構成、およびインポートのプロセスで、興味深い瞬間が生じました。この瞬間をコミュニティと共有したいと思います。



最初の瞬間：システムとメッセージ「GRUB LOADING ...」のインストール


OSをディスクからインストールした後、Proxmoxが再起動して初めて起動します。この時点で、「GRUB LOADING ..」というメッセージが表示され、起動がフリーズします。

「GRUB LOADING ..」の問題は、proxmoxをインストールするときに考慮すべき2つの要因に関連しています。

  1. OSのインストールが実行されるハードドライブは、起動時に常に最初でなければなりません（残念ながら、BIOSでのブート順序オプションの対応するインストールでは、望ましい結果、パラドックスが得られませんが、目的のsata0ポートを見つけて、正しい構成でシステムを起動する必要があります）
  2. ドライブは常に接続されている必要があります（ドライブがない場合、「GRUB LOADING ..」を観察します。明らかに、ドライブデバイスは起動スクリプトに表示されます）


2番目の瞬間：openvzコンテナー（proxmox）でネットワークを構成するか、ブリッジなどのインターフェースをアクティブにする


Webインターフェースでコンテナーを作成した後、ネットワーク構成に問題があります。 設定時に、IPアドレスを指定することは可能ですが、それは仮想NATになります。 コンテナーへのアクセスはvhostで可能ですが、ネットワーク全体からではありません。 ネットワーク全体からコンテナにアクセスできるようにするには、イーサネット（eth0）を構成する必要があります。 インストール段階で、eth0も設定できますが、IPアドレスは割り当てられません。 さらに、コンテナ自体にはeth0がありません（ifconfigコマンドでは表示されません）。 ここで問題が発生します。ブリッジインターフェイスの設定方法です。

OpenVZネットワークのセットアップに関する資料によると、次のことを行う必要があります。コンテナを入力します（「100」-コンテナID）

vzctr enter 100





コンテナシステムでeth0インターフェイスを定義します（この手順を行わない場合、次の手順はコンテナがリロードされるまで有効です）

ifconfig eth0 0





eth0インターフェイスを構成します 

  ifconfig eth0 192.168.XXX.XXXネットマスク255.255.255.0
 ip route add 192.168.XXX.YYY dev eth0
 ipルートは192.168.XXX.YYYを介してデフォルトを追加します


または永続的に/ etc / network / interfaceに貢献する 

 自動eth0
 iface eth0 inet static
		アドレス192.168.XXX.XXX
		ネットマスク255.255.255.0
		ゲートウェイ192.168.XXX.YYY


ポイント3：VMWare仮想マシンをKVM Proxmoxにインポートする


  1. freebsd.vmdkをインポートする

    既存のVMWare仮想マシンをFreeBSDでインポートする前に、Proxmox KVMで同じパラメーターを使用してプロトタイプを作成し、「vmdk」ディスク形式を選択してIDEと入力する必要があります。 プロトタイプを作成したら、「freebsd.vmdk」ディスクを/ var / lib / vz / images / id /にコピーし、名前を変更して（作成したディスクの名前を指定）、仮想マシンを起動します。 ダウンロードがルートパーティションの検索で終了する場合は、狂乱ディスクをマシンのドライブにロードし、hdrwモードでlive-cdからの起動を繰り返します（ディスクへの書き込み）。 マウントされたdf -hスライスを見て、ディスクラベル（adまたはda）を見つけ、/ etc / fstabファイルを編集し、ディスクラベルを前のコマンドで受け取ったものに変更します。 OSを正常にロードした後、ネットワークインターフェイスを編集する必要があります。
  2. windows.vmdkをインポートする

    インポートしたVMWare仮想マシンをネイティブソフトウェアで最初にダウンロードしてから、ロードしたマシンのレジストリにIDEドライバー情報を追加し、pciide.sysを％SystemRoot％System32 / Driversにコピーする必要があります。 VMWare Toolsを仮想システムからアンインストールし、仮想マシンを切断して、vmdk仮想ディスクファイルをProxmox（KVM）にコピーします。 インポートされたディスク（IDEタイプ）で仮想システムをロードします。


瞬間4：ProxmoxでのVMDK形式の仮想ディスクの圧縮


私の意見では、vmdkディスクの圧縮手順は非常に重要です。 物理ディスク容量は常に制限されています。

  1. WINDOWS.vmdkを圧縮する

    仮想ディスクの圧縮は、ゲスト仮想システムの実行中のvmdkファイルの「膨張」の影響（ファイルサイズの増加）と関連しています。 MS WindowsでVMDK形式の仮想ディスクを圧縮するには、VMWareのユーティリティをいくつか使用できます：vmware-mountおよびvmware-vmdiskmanager-VDDKパッケージの一部として、からダウンロードできます。 サイト なぜなら ProxmoxはDebianで実行され、linux64アーカイブをダウンロードしてインストールします。

    tar -xzf VMware-vix-disklib-5.1.0-774844.x86_64.tar.gz





    cd ./vmware-vix-disklib-distrib





    ./vmware-install.pl





    インストール後、/ etc / ld.so.confに次の行を追加します。

    /usr/lib/vmware-vix-disklib/lib64







    さらなるユーティリティは、ディスクで動作する準備ができています。 ディスクが圧縮の対象となるゲスト仮想システムをオフにします。 vmdk-disk圧縮手順は、3つのフェーズで構成されます。

    • vmdkディスクデフラグツール

      vmware-vdiskmanager -d /path/to/name.vmdk



    • ディスク圧縮の準備

      vmware-mount /path/to/name.vmdk /mnt/vmdk/





      マウントフェーズ中に不明なNTFSファイルシステムに関するエラーが発生した場合、Debianでこのシステムのサポートをロードする必要があります。

      apt-get install ntfs-3g





      vmware-vdiskmanager -p /mnt/vmdk/



    • ディスク圧縮

      vmware-mount -x # vmdk





      losetup -d /dev/loop0 # : "device is busy"





      vmware-mount -x #





      vmware-vdiskmanager -k /path/to/name.vmdk #







    vmdkディスク圧縮の効果は、ゲスト仮想システムの実際のディスクサイズまでです。

  2. 圧縮FREEBSD.vmdk

    FREEBSD OSで仮想ディスクを圧縮する場合、上記のユーティリティは機能しません。 ホストシステムにvmdkディスクをマウントする問題があります。DebianはUFSファイルシステムを認識しません。 ただし、解決策があります。FreeBSDの組み込み機能を使用できます。元のゲストOSのファイルシステムのダンプを作成し、新しく作成した仮想ディスクに復元し、ディスクを変更します-freeBSDを新しいhddに転送する手順



ポイント5：コンソールを介してProxmoxで仮想ゲストシステムを復元する


Webインターフェイスを介してProxmoxで仮想ゲストシステムを復元する手順はよく知られていますが、コンソールでこれがどのように発生するかは明確ではありません。 ゲストシステム（KVMについて話す）は、ファイルvzdump-qemu-id-date-time.vma.gzにバックアップされます。 これは、事前にパッケージ化された仮想ディスクとゲストOS構成ファイルのアーカイブです。

最初に.vmaファイルを取得することは論理的です：

gzip -d vzdump-qemu-103-date-time.vma.gz





103はProxmoxのゲストシステム番号です

次に、ファイルvzdump-qemu-103-date-time.vmaを配布する必要があります。 このため、Proxmox 3.1には特別なユーティリティvmaがあります

vma extract vzdump-qemu-103-date-time.vma /var/103





注：最初にディレクトリ「103」が/ varディレクトリに作成されていない場合、コマンドはエラーなしで機能します。 その結果、/ var / 103 /にはいくつかのファイルがあります。

vm-103-disk-1.raw





qemu-server.conf





注：vmdkディスクを使用してゲストシステムを復元する場合、復元されたディスクは「raw」形式になります。 これを修正するには、目的の形式に変換します。

qemu-img -f raw -O vmdk vm-103-disk-1.raw vm-103-disk-1.vmdk







モーメント6：Proxmox上のWindowsゲストの正しいシャットダウン（KVM）


Proxmoxゲストシステム（KVM）の「OSの正しいシャットダウン」とは、「物理マシンのシャットダウンボタンを押す」というイベントをシミュレートすることを意味します。 UNIXシステムでは、このようなシャットダウンは正常に機能します。 これは、QEMUのチームによって達成されます。

qm shutdown 103





ただし、バックグラウンドのWindowsゲストシステムでは、コマンドはエラーを返し、システムはシャットダウンしません。 切断は、ログインが成功した場合にのみ「正しく」、または別のコマンド（qm stop 103）によって「強制的に」（イベント「電源ケーブルの切断」）発生します。 ゲストWindows OSのローカルポリシーを調整することにより、この状況を修正できます。
