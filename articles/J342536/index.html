コンテナ管理ツール





開発者にとって、アプリケーションのデプロイは常に頭痛の種でした。 Windows COMの時代に偶然コーディングを行ったOldfagsは、おそらく「DLL Hell」を覚えています。これは、開発者やシステム管理者にとって本当の悪夢です。 しかし、何年も経ちましたが、日々増加する新しいテクノロジーの流れは、しばしば混乱と不確実性を生み出します。



ソフトウェア開発のほとんどすべての場合、開発環境はアプリケーションが実際に動作する環境とは大きく異なります。 異なるコンピューターが異なるように構成されているという事実は明白で予測可能ですが、これらのコンピューター上のアプリケーションの異なる動作は受け入れられません。



記載されている難しさを考えると、コンテナテクノロジーは実際に展開の革命となり、Dockerなどの開発により人気が爆発的に成長しました。 コンテナは何をしますか？ すべての必要な依存関係とともに、アプリケーションをユーザーランタイムで正常に実行するために必要な構成でソフトウェアを展開できます。 したがって、コンテナ化されたイメージにより、オペレーティングシステムの高価なエミュレーションを回避しながら、「パッケージ化された」最終ランタイムを開発環境とほぼ同一にすることができます。



その結果、非常に軽量なパッケージが作成され、多くの場合、いくつかのアプリケーション固有のプロセスのみが実行されます。 このようなパッケージは、さまざまなインスタンスで簡単に配布、展開、実行できます。 上記を要約すると、コンテナは異なるソフトウェア環境でのアプリケーションの同じ動作を保証します。



この記事は、いくつかの重要なコンテナ管理ツールの簡単な概要です。 コンテナテクノロジを使用する予定がある場合は、少なくともこの一連のアプリケーションに慣れる必要があります。



これは詳細なレビューではなく、比較や評価も行いません。 コンテナ管理ツールを探索する際の出発点と考えてください。 以下は、さらなる詳細な調査のためのツールの選択に役立つことを願って、短いリストと概要です。



Docker







Dockerは最も人気のあるプラットフォームの1つです。 コンテナーテクノロジーの最近の人気の高まりの理由は、Dockerである可能性があります（テクノロジー自体はそれほど新しくなく、10年以上存在しています）。



Dockerは、コンテナアプリケーションをデプロイおよび実行するための機能を提供します。 コンテナは仮想マシンに似ていますが、OSよりも高いレベルで実装され、分離度は低くなっています。 必要なすべてのソフトウェアでオペレーティングシステム全体をパックする代わりに、コンテナはアプリケーションとその直接の依存関係のみをパックします。 これは、Dockerによって実装される基本的な概念です。



Docker形式は通常のパッケージと似ていますが、パッケージがスタンドアロンであり、同じホスト（または異なるホスト）で複数のコピーを実行できる点が異なります。 このアプローチは、リソースの使用を最大化し、生産性を向上させ、アプリケーションのサイズを縮小します。 同時に、Dockerは高度な分離を提供し、マシンを傷つけることなく、コンテナー自体の内部のアプリケーションの問題を制限します。



クベルネテス







Dockerは、単一のホストでコンテナーを操作するのに最適で、この目的に必要なすべての機能を提供します。 しかし、今日の分散サービスの現実では、重要なタスクは、サーバー上および複雑なインフラストラクチャ内のリソースとワークロードを管理することです。



そのようなツールの1つがGoogle Kubernetesです。 この珍しいギリシャ語は「知事」または「指揮官」と訳されています。 名前が示すように、Kubernetesは、多くのノードでコンテナーを編成（編成）するという大変な仕事を引き受け、幅広い便利な機能を提供します。



サービス発見



最近、コンテナサービスのエコシステムは、Microservicesアーキテクチャスタイルの人気の高まりとともに成長しました。 Microservices Architectureの背後にある考え方は、重いモノリシックアプリケーションを、分散システムで相互接続される小さな（半）自律サービスに分割することです。 また、このようなシステムではサービスの数が何度も増加するため、アプリケーションが「リッスン」するポートを手動で割り当てて考慮することは意味がありません。 コンテナの展開後にポートを自動的に割り当てる方が賢明です。 また、このアプローチはポート間の競合の問題を解決しますが、適切なサービスを見つけて他のサービスに提供することも不可能にします。



この問題の解決策は、Service Discoveryテクノロジーです。 サービスディスカバリは次のように機能します。サービスの開始時に、レジストリは分散キー/値をIPアドレスとポート番号で登録します。他のサービスは簡単に目的のサービスを見つけて呼び出すことができます。



スケジューラー（スケジューラー）



Kubernetesが提供するもう1つの重要な機能は、スケジューリングです。 スケジューラは、コンテナ化されたアプリケーションを起動するタスクを引き受け、利用可能なリソース、現在のタスク、リクエストを考慮します。 つまり、スケジューラはスケーリングを実行し、タスクを異なるホストに再割り当てし、問題が発生した場合に能力間でワークロードを移動します。

多くの便利な機能と大きなコミュニティのおかげで、Kubernetesの人気は最近急速に高まっています。



Coreos







CoreOSは、コンテナアプリケーションの展開と実行に必要な機能を備えたLinuxベースのオペレーティングシステムです。 比較的小さな負荷分散機能にもかかわらず、CoreOSは単一サーバー環境とクラスターコンピューティング環境の両方に適しています。



CoreOSは、コンテナを操作する組み込み機能と、パブリッククラウドアプリケーションを操作するための適応機能により、他の一般的なLinuxディストリビューションの深刻な競争相手のようです。



CoreOSは、Dockerを抽象化の追加レベルとして、およびコンテナーの標準形式として使用することから始まりましたが、後にリリースされ、独自のコンテナーエンジンRocket（rkt-読み続けます）をサポートします。



CoreOSはオープンソースですが、Tectonicと呼ばれる商用バージョンのシステムも利用できます。 テクトニックは「フリーミアム」であり、その大きな利点は、コンテナオーケストレーション用のKubernetesディストリビューションです。



Etcd







CoreOSのEtcdは、オープンソースの軽量な分散キー/値リポジトリです。 Etcdは、サーバークラスターにデータを保存するための信頼できるツールをアプリケーションに提供し、値の変化に適切に対応できるようにします。 Kubernetesは、すべてのクラスターノードが構成データの読み取り/書き込みを行うことができる中央リポジトリとして、またサービスの発見（Service Discovery）およびリーダーの選択の制御（Leader Election）として使用します。



rkt







rktは、Linuxクラスター用のコンテナー管理プラットフォームです。 独自のOSの設計とDockerとの対話の経験に基づいて、CoreOSは、コンテナアプリケーションを分離して作成、検証、実行するためのツールを開発しました。



Dockerの代替としてのrktは、ほぼ1年後にリリースされ、それ以来人気を集めています。 Rktは、少なくとも2つの理由でdockerの改善と見なすことができます。



-署名の検証と特権の分離をデフォルトで使用するため、セキュリティが強化されています。



-移植性を提供：Dockerイメージを実行する機能に加えて、rktは、App Container Images（ACI）-オープンなApp Container Specification（appc）で指定された形式のイメージを実行するように設計されています。



メソス







Apache Mesosは、分散コンピューティングシステム用のカーネルであり、Linuxカーネルと同じ原理に基づいて構築されていますが、抽象化レベルが異なります。 その主な目標は、複数の物理マシンまたは仮想マシンのコンピューティングリソースを高レベルの分離で統合し、1つの大きなリソースプールのように中央管理サービスで利用できるようにすることです。 簡単に言うと、Mesosを使用すると、サーバークラスターを単一のコンピューティングパワーとして使用できます。



Mesosは、各マシンのリソースの可用性を測定およびレポートするエージェントをホストすることにより、タスクの実行を計画および編成し、タスクの完了を確認できます。 したがって、タスクは複数のサーバーで同時に実行され、その利用可能なリソースは単一のプール（いわゆるMesosフレームワーク）と見なされます。



領事







Consulは多くの操作を含む分散システムですが、その主な価値はサービスディスカバリー機能（サービスディスカバリー）とキー/バリューストレージの提供です。



Consulでは、サービス検出ツールとして、アプリケーションをAPIなどのサービスを提供するクライアントとして登録できます。 他のシステムは、Consulに直接アクセスして利用可能なサービスを見つけることができます。 ビルトインヘルスチェックは、脅威が発生したときにアクションを実行する機能を提供します。たとえば、テストに失敗したノードからのトラフィックの受け入れを拒否します。



キー/値ストアとして、Consulはアプリケーションに、ノード全体に構成データを保存し、etcdと同様にそのデータの変更に応答する機能を提供します。 さらに、Consulは、いくつかのデータセンター、ゴシッププールに基づくより包括的なヘルスチェックシステム、独自のサービス検索プール、および監視サービス用のWebインターフェイスをネイティブサポートしています。



Googleコンテナエンジン







Google Container Engine（GKE）は、Dockerコンテナー用のクラスターマネージャーおよびオーケストレーションシステムです。 GoogleのKubernetesベースのキャンペーンで作成されたオープンソースシステムは、次の目的に使用できます。



  • コンテナクラスタの作成、サイズ変更、デバッグ
  • 自動スケーリングアプリケーション
  • ポッドの作成と管理


Kubernetesと比較して、Google Container Engineは開発者とシステム管理者に追加のサービスセットを提供します。 たとえば、ロードバランシング、メトリックトラッキング、およびロギングはすべてGKEによって提供されます。
