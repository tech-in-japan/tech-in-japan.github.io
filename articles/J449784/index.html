Djangoフレームワークが次のプロジェクトに適しているかどうかを開発者が理解できるようにしましょう。 可能性が高い-適切です。



前のプロジェクトで特定のプログラミング言語またはフレームワークを使用したという理由だけで、または単にそれをよく知っているという理由だけで、特定のプログラミング言語またはフレームワークをつかまないでください。 だから物事は行われていません。



新しいプロジェクトに着手する前に、目的の結果を達成するのに最適な言語またはフレームワークを評価する必要があります。 あなたにとって最も重要なことは何ですか？ セキュリティ、開発速度、スケーラビリティ、汎用性、サポート？

後で急いで悔い改めるよりも、作業を開始する前に十分な情報に基づいて決定することをお勧めします（または、さらに悪いことに、実装の過程でプロジェクトに松葉杖を掛けます-事前にサポートを気にしなかったためです）。



私は長年にわたってさまざまなテクノロジーを使用し、モバイル開発とWeb開発の両方に取り組んでおり、Djangoは他のWebフレームワークにはない機能の完全なセットを提供すると考えています。



これは大きな声明であると理解しています。 それを正当化させてください。

一般的なビュー：Djangoを使用する場合



以下の点の少なくともいくつかがあなたに関するものである場合（そして、あなたが強く反対する点がリストにない場合）、Djangoはあなたのプロジェクトに適している可能性が高いです。



  • WebアプリケーションまたはバックエンドAPIを開発する必要があります。
  • 作業中にプロジェクトをすばやく展開し、迅速に展開して変更する必要があります。
  • デフォルトでは、アプリケーションは最も一般的な脆弱性と攻撃、特にCSRF、SQLインジェクション、XSS、クリックジャッキングなどから保護される必要があります。
  • アプリケーションではいつでも、拡張と縮小の両方のスケーリングが必要になる場合があります。
  • 将来的には、機械学習などの最新のテクノロジーを統合する予定です。
  • 積極的に開発され、世界中の多くのトップ企業や主要なWebサイトで使用されている信頼できるフレームワークを使用する必要があります。
  • WebアプリケーションとAPIのサーバー側の両方が、「単一の真実のソース」（DRY原則）と一致する同じコードベースにある必要があります。
  • データベースクエリを直接操作したくない場合、ORMサポートが必要です。
  • フリーソフトウェアを使用します。
  • 行き詰まった場合は、自分で解決策を探す必要があるため、優れたドキュメントと応答性の高い開発者コミュニティが必要です。


上記の要因に加えて、自分（またはチーム）のスキルを考慮する必要があります。



あなたがWeb開発者であり、Webがどのように機能するかをすでに知っている場合、Djangoとの連携は比較的スムーズに機能します。 もちろん、Djangoがどのように構成されているか、その他のことも理解する必要があります-そして、準備ができていることを考慮してください。

Djangoを使用しない場合



  • あなたは巨大なアプリケーションを扱っており、1つのコードベースに収まりません。 アプリケーションをマイクロサービスに分割することをお勧めします。 各レベルは、専任のチームによって適切に処理されます。 特定のユースケースごとに、他のテクノロジーが適しています。 これらのシナリオのいくつかでは、Djangoが便利な場合がありますが、Django（および他の個別のフレームワーク）でそのようなアプリケーションを完全に開発することは実用的ではありません。
  • データベースを操作したり、ファイル操作を実行したり、少なくとも少し複雑なことをしたりする必要のない単純なアプリケーションを作成する必要があります。

    マイクロフレームはこれらの状況に適しています。 最も人気のあるマイクロフレームの1つ-DjangoのようなFlaskはPythonで書かれています。 同様のマイクロフレームは、他のテクノロジーでも利用できます。 PHPのSlim、JavaのApache Spark、Node.jsのExpress.jsなど
  • 自分ですべてをゼロから書き、何をしているかを知りたい。
  • あなたまたはあなたの同僚はDjango / Pythonにまったく慣れていないため、必要なスキルを身に付けるための時間とリソースがありません。

    後者の場合の最良の解決策は、あなたが最もよく知っているものを扱うことです。 新しいテクノロジーやフレームワークを採用すると、失敗する可能性が何度も高まります。


上記のすべてがあなたのプロジェクトに関するものではない場合、Djangoがあなたに合うでしょう。

乾燥原理（繰り返さないでください）



私は、支持者が「DRY」原則に本当に準拠していると主張する多くのフレームワークを知っています。 私はそのような多くのフレームワークを使用してきましたが、それらのどれにも「DRY」の原則が実装されていません。



残念ながら、ほとんどのフレームワークでは、「DRY」の原則に十分な注意が払われていません。 私の意見では、定期的に更新する予定のアプリケーション（ほとんどの最新のアプリケーションについて言えます）を書いている場合、問題を避けるためにDRYの原則に従うべきです。



そのため、Laravelでは、各プロシージャの検証を個別に記述する必要があります。 他のほとんどのフレームワークでも同じことが言えます。 コードをDRYの原則に準拠させるには、一生懸命働く必要があります。 特にチームとして働いている場合、追跡するのは難しいです。



次に、Djangoフレームワークは、DRYの原則に違反するのが通常、意図的にのみ行われるように設計されています。



そうではないはずですよね？ 例を考えてみましょう。



Djangoがデータベースの検証と移行でどのように機能するかを以下に示します



必須フィールドを使用してModel



クラスを作成します。 必要な追加の検証と制限をすべて示します。



移行は、単一のCLIコマンド「python manage.py makemigrations」によって生成されます。

変更は、「python manage.py migrate」という1つのCLIコマンドでデータベースに加えられます。

検証と制限は、CRUD操作（Django AdminまたはDjango REST Framework）のたびに自動的にチェックされます。 検証を再度記述する必要はありません。

Django Admin CRUDビューの生成には同じモデルクラスが使用されます。 カスタムHTML / CSSは必要ありません。



これらの条件を他のフレームワークと比較してください。次の数行のコードでは、このようなことは決してできないと思います。 



 class Employee(models.Model): name = models.CharField(max_length=127) email = models.EmailField(null=True, blank=True) created_at = models.DateTimeField(blank=True, null=True, auto_now_add=True) updated_at = models.DateTimeField(blank=True, null=True, auto_now=True)


これは単に「繰り返さない」ことだけではありません。 このアプローチは、将来のバグからあなたを救います。 私たちは皆、ある場所で何かを変更し、別の場所で交換するのを忘れていた状況で自分自身を見つけました-そして、これは多くのユーザーが問題を抱え始めて初めて明らかになりました。



Djangoでは、上記のコードに戻り、 `max_length`



フィールドを別のものに置き換える必要がある場合は、ここで実行します。 変更は、すべてのルートの検証とデータベースに自動的に適用されます。



Djangoのオブジェクトリレーショナルマッピング



Djangoは、フル機能のすぐに使えるORMエンジンを提供します。



Eloquent、greenDAO、Yii ARなど、さまざまなテクノロジーで多くのORMツールを使用しました。 それらのすべてで、最も単純なリクエストは非常にうまく処理されますが、遅かれ早かれ、これらのリクエストまたはそれらのリクエストをゼロから書く必要がありました。



Django ORMでは、私はまだそのような状況にはありません。 データベースクエリを使用していることを忘れてしまうほどうまく機能しました。 これは、まさにオブジェクトリレーショナルマッピングのはずです。 以下に、Django ORMの例を示します。
