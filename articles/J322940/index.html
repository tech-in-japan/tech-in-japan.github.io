SharePoint 2013/2016監視：主要なパフォーマンスカウンター

こんにちは 私の名前はリュウボフ・ボルコバです。ビジネスソリューション開発部門のシステムアーキテクトです。 私の主な専門分野は、企業のSharePointポータルの実装、ソリューション開発、技術サポートです。 長年の経験により、一般的なファームに含まれるサーバーのパフォーマンスに影響する主なパターンを特定できます。



この投稿の目標は、エンタープライズSharePointポータルの管理者が効果的なサーバーメンテナンスプランを開発できるようにすることです。 以下のテキストは、SharePoint 2013/2016ファームサーバーの毎日のメンテナンスプランに含めることをお勧めするパフォーマンスカウンターと実用的な例をまとめたものです。 メーターに関するデータを使用して、エクスプレスモニタリングパネルのインジケーターを手動で構成および分析したり、メーターが一定期間にわたってしきい値を超えた場合に通知の受信を自動化したりすることができます。この期間は、組織が採用する要件と標準によって異なります。







システムカウンター



SharePointファームの一部であるサーバーシステムを監視する必要があるユニバーサルパフォーマンスカウンターがいくつかあります。



CPU



CPU使用率（_Total）\％プロセッサ時間



プロセッサが、アイドルを除くすべてのコマンドフローの処理に費やす時間の割合。 この値は、100％と、プロセッサがアイドルスレッドの完了に費やす時間の割合との差に等しくなります。 （アイドル命令ストリームは、他の命令フローがない場合、プロセッサーの作業時間を占有します。）このカウンターは、プロセッサー使用率の主要な指標です。 測定間隔中のプロセッサー使用率の平均値を示します。



パフォーマンスを監視し、すべてのプロセッサのワークロードを75％以下のレベルに維持する必要があります。 より高いレベルの輻輳では、システムはアクティビティの突然のバーストに対処できなくなります。 また、1つのコンポーネントの障害が他のコンポーネントの誤動作を引き起こす場合の「ドミノ効果」を回避します。 たとえば、3つのWebサーバーがある場合、すべてのサーバーの平均CPU負荷が60％未満であることを確認する必要があります。これにより、1つが失敗した場合、他の2つのプロセッサが追加の負荷を処理できます。



割り込み/ s（割り込み/秒）



プロセッサがハードウェア割り込みを受信して​​処理する平均速度（1秒あたりのイベント数）。 この値には、個別に計算される遅延プロシージャ呼び出しは含まれません。 この値は、システムタイマー、マウス、ディスクドライバー、データライン、ネットワークアダプター、その他の周辺機器など、ハードウェア割り込みを形成するデバイスのアクティビティの間接的なインジケーターです。 通常、これらのデバイスは、シャットダウンしたとき、または要求を処理する必要が生じたときに、プロセッサを中断します。 この場合、コマンドフローの通常の実行は中断されます。 通常、システムタイマーは10ミリ秒ごとにプロセッサに割り込み、ハードウェア割り込みの「バックグラウンド」を作成します。 したがって、この値は、最後の2つのサンプルの値の差をサンプリング間隔の持続時間で割った値を表示します。



カウンタの読み取り値はプロセッサに依存しています。 適切な開始値は、1秒あたり1,000割り込みです。 システムアクティビティの対応する増加を伴わないこのカウンタの値の大幅な増加は、問題を示しており、ネットワークアダプタ、ディスク、またはその他の機器が中断を引き起こしている可能性があります。



システム



プロセッサキューの長さ



待機中のスレッドの数で測定される現在のプロセッサキューの長さ。 すべてのプロセッサは、スレッドがプロセッササイクルを待機する1つの共通キューを使用します。 このカウンタには、現在実行中のスレッドは含まれません。 プロセッサー時間については、複数のプロセッサーを搭載したコンピューター上でも1つのキューがあります。 したがって、コンピューターに複数のプロセッサーがある場合は、この値を負荷を処理するプロセッサーの数で割る必要があります。



このカウンターは現在の値を反映し、特定の時間間隔での平均値ではありません。



プロセッサあたり5スレッド未満の連続プロセッサキューは一般に許容され、ワー​​クロードにも依存します。 しきい値を超える値は通常、プロセッサの輻輳を示します。



プロセス



_Totalインスタンスのワーキングセット



プロセスのワーキングセットキャッシュの現在のサイズをバイト単位で表示します。 ワーキングセットは、プロセススレッドによって最近使用されたメモリページのセットです。 コンピューターの空きメモリの量がしきい値を超えると、未使用のページがプロセスイベントのワーキングセットに格納されます。 空きメモリがしきい値を下回ると、ワーキングセットからページが削除されます。 必要な場合、RAMからアンロードされる前にRAMエラーが解決されると、ワーキングセットに転送されます。



このカウンタは、システム全体の問題と特定のプロセスに関連する問題の両方を示すことができます。 ワーキングセットのサイズが大幅に増減すると、スワップが発生します。



スワップファイルの推奨設定は「RAM + 10」です。



ワークセットからの削除が発生した場合、カウンター「Process（*）\ Work set」を追加して、問題の影響を受けるプロセスを見つける必要があります。



さらに、このカウンターの読み取り値をカウンター "Memory \ Resident bytes of system cache"の値と比較して、ワーキングセットからのシステム全体のページ削除が発生するかどうかを判断することをお勧めします。

このカウンタの読み取り値は、_Totalオブジェクトのシステムカウンタ「プロセッサ\％CPU使用率」のデータとともに分析する必要があります。 すべてのプロセッサのワークロードがしきい値を超える場合、ASP.Netプロセスのロードに関するデータは、このプロセスが問題の原因であるかどうかを判断するのに役立ちます。

論理ディスク



平均ディスクキューの長さ



ディスク要求キューの平均長。 特定の時間間隔中に処理を待機しているディスク要求の数を表示します。 単一ディスクのキューが2以下である場合は、正常と見なされます。 キューに3つ以上の要求がある場合、おそらくディスクが過負荷になり、着信要求を処理する時間がありません。 「読み取り要求の平均数」カウンターと「ディスクへの書き込みの平均長さ」カウンターを使用して、ディスクが処理できない正確な操作を指定できます。

記憶



使用可能なMバイト



このカウンタは、割り当てに使用可能な物理メモリの量を示します。 十分なメモリがない場合、ページファイルがより集中的に使用され、1秒あたりのページエラー数が増加します。 このカウンターの値がWebサーバーで2 GB未満の場合、メモリを無駄にする必要があります。



割り当てられたメモリ使用率（％使用中のコミット済みバイト数）



専用メモリー制限（Commit Limit）に対するコミット済みメモリー（Committed Bytes）の割合。 この値は、使用可能な仮想メモリの実際の使用量を反映しています。 ページファイル（ページファイル）が増加すると、割り当てられたメモリの制限が変更される可能性があることに注意してください。 この値は特定の現在値であり、特定の時間間隔での平均値ではありません。



しきい値：警告は70％、90％以上-クリティカル。 より高い値では、メモリの量を増やすだけで十分です。



ページフォールト/秒（ページフォールト/秒）



カウンターは、1秒あたりのページエラーの平均数を示します。 1秒あたりの失敗したページ読み取り数で測定されます。 ページエラーは、メモリ内のページを要求するプロセスがあるときに発生し、システムは要求された場所でそれを見つけることができません。 要求されたページがメモリに見つからない場合、このエラーはソフトページフォールトと呼ばれます 。 要求されたページをディスクから復元する必要がある場合、このエラーはハードページフォールトと呼ばれます 。



ページ入力/秒



入力ページ/秒は、リンクが処理された時点でメモリにないページへのリンクを解決するときにディスクから読み取られたページの数です。 スレッドがRAMのワーキングセットにない仮想メモリページを参照すると、ページエラーが発生します。 このカウンターは、システムキャッシュによって実行されるスワップ（ページ交換）も考慮して、アプリケーションによって要求されたデータにアクセスします。 これは、過剰なメモリ負荷を特定し、過剰なページ交換を引き起こすための重要な情報源です。 推奨される警告しきい値は1000です。



読み取りページ/秒（ページ読み取り/秒）



カウンター値は、ページ割り込みの処理中に、ページに関係なく、単位時間あたりの読み取り操作が実行された回数を示します。 このカウンターは、プロセスのワーキングセットが物理メモリに対して大きすぎること、およびディスクとのページごとの交換があることを示します。 読み取り操作の数のみを表示します。各操作で取得されたページ数は除きます。 大きな読み取り値は、メモリのボトルネックを示しています。 過度のスワッピングは、応答と不安定性を低下させる可能性があります。



カウンタページ入力/秒と読み取りページ/秒は一緒に考慮する必要があります。 最初のものにはディスクから読み取られたページの数が含まれ、2番目にはスワップ中に実行された読み取り操作の数が含まれます。 これらのカウンタは、ハードページフォールト—目的のデータページが物理メモリにないメモリアクセス操作を考慮します。 したがって、Exchangeページ/秒、読み取りページ/秒、ページの入力/秒が常に高レベルである場合、オペレーティングシステムがページファイルをアクティブに操作していると見なすことができます。これは、メモリ不足を示しています。 ページ数/秒を入力このカウンターの値は、ページ数/秒の値以上でなければなりません。



キャッシュエラー/ s（キャッシュフォールト）



このカウンタは、ファイルシステムキャッシュでページを検索する際のエラーの頻度を示します。 これは、ページがメモリ内で見つかった場合はソフトウェアエラー、ページがディスク上にある場合はハードウェアエラーの可能性があります。



読み取りおよび書き込み操作にキャッシュを積極的に使用すると、サーバーのパフォーマンスに大きく影響する可能性があります。 キャッシュエラーの数の増加を監視する必要があります。これは、 非同期高速読み取り/ sまたは順方向読み取り/ sの値の減少によって示されます。



ページ交換/秒（ページ/秒）



深刻なページクラッシュを解決するために、ページをディスクに読み書きする速度。 この値は、入力ページ/秒と出力ページ/秒の値の合計です。 カウンター読み取り値は、システム全体で遅延を引き起こす障害の種類の主要な指標です。 ファイルシステムキャッシュ内のページフォールトを補正するために受信したページ数を示します。 これらのページは通常、アプリケーションで必要です。 このカウンターの値は10を超えてはなりません。



非ページプールのバイト（プールの非ページバイト）



非ページプールのサイズ（バイト単位）。 非ページプールは、ディスクに書き込むことができないオブジェクトに使用されるシステムの仮想メモリの領域であり、存在する間は物理メモリに残る必要があります。 このカウンタは、平均値ではなく現在の値のみを反映します。



オペレーティングシステムのコンポーネントが機能するために必要な場所を要求する、メモリの特別なシステム領域にスペースを割り当てる要求。 非ページページプールのページは、ディスク上のページファイル（ページファイル）にアップロードできず、その使用期間全体にわたってRAMに残ります。 このカウンターは現在の値を反映し、特定の時間間隔での平均値ではありません。



カウンターは、2つの最小値-2x RAMおよび128 GBを超えてはなりません。



メモリカウンターのパフォーマンスを分析するための一般的な推奨事項



\ Memory \ Exchange pages / sec、\ Memory \ Read pages / sec、Memory \ Enter pages / secが常に高レベルで、\ Memory \ Cache errors / secが低い場合、オペレーティングシステムがアクティブであると想定できます。スワップファイルで動作します。これは、メモリ不足を示しています。 ただし、\ Memory \ Cache Errors / sも高い場合、ほとんどの場合、メモリに表示される大きなファイルのアクティブな作業が原因です。 通常、これにはそれほど時間はかかりません。

SharePoint Serverパフォーマンスカウンター



ASP.NetおよびASP.Netアプリケーション



ASP.Net。 アプリケーションの再起動



Webサーバーの存続期間中にアプリケーションが再起動する回数。 このカウンターの値は、Application_OnEndイベント（Webアプリケーションの終了）が発生するたびに増加します。 アプリケーションの再起動は、Web.configファイルの変更、アプリケーションの\ Binディレクトリ内のアセンブリの変更、またはWebフォームページの多数の変更の結果として発生する可能性があります。 この値の予期しない増加は、予期しない状況の結果としてWebアプリケーションのシャットダウンが発生したという事実による可能性があります。 この場合、問題の原因をできるだけ早く分析する必要があります。 このカウンタの値はゼロになるはずです。



ASP.Net。 拒否されたリクエスト



要求キューのオーバーフローにより拒否された要求の総数。 多くの場合、リクエストを処理するにはサーバーリソースが不十分なため、リクエストは拒否されます。 この値は、返されるHTTPエラーコード503の数に対応します。これは、サーバーがビジーであることを意味します。 リソース不足の理由の例：Webサーバーへの多数のリクエスト、DBMSへの多数の低速リクエスト（最適化されていない）、サードパーティ開発者によって書かれたソリューションのコンポーネントが正しく実行されません。 したがって、サーバー上のリソース不足の原因を特定して解消するには、追加のツールを使用してWebサーバーのより詳細な分析を実行する必要があります。



ASP.Net。 キューに入れられたリクエスト



MS SharePoint Webアプリケーションは、HTTPを介してユーザーのブラウザーに表示され、予備のデータ処理と処理（Webパーツ、ユーザーコントロールなど）を必要とするHTMLページの標準ブロックを提供します。 企業ポータルWebページでユーザーに表示される最終的なデータ処理結果を準備するには、データベース、ファイルシステムなどに対する1つまたは複数のクエリが必要になる場合があります。 このカウンターの最大デフォルト値は5000です。このパラメーターはMachine.configファイルで変更できます。 このカウンタの値は、しきい値の70〜75％を超えてはなりません。 3500-3750。

SQL Serverの監視



ディスクサブシステム制御



Microsoft SQL Serverは、Microsoft Windows ServerオペレーティングシステムのI / Oシステムコールを使用して、ディスクの読み取りおよび書き込み操作を実行します。 SQL ServerはディスクI / Oを実行するタイミングと方法を決定しますが、基本的なWindows I / OはWindowsによって実行されます。

以下は、毎日の保守計画で監視することが推奨されるメーターの最小セットを説明しています。



物理ディスク\平均ディスク読み取り時間（物理ディスク\平均ディスク秒/読み取り）



ディスク読み取り操作ごとに平均で費やした時間（秒単位）。 ディスクの読み取りが完了した平均時間を示します。 このカウンタの基本的なインジケータは15ミリ秒を超えてはなりません。 読み取り操作の実行時間が平均でこの値を十分に長い時間超えている場合、これは入出力デバイスの問題を示している可能性があります。



物理ディスク\平均ディスク書き込み時間（物理ディスク\平均ディスク秒/書き込み）



平均ディスク書き込み時間は、ディスクへの書き込み操作ごとに平均で費やされる秒単位の時間です。 このカウンタの基本的なインジケータは15ミリ秒を超えてはなりません。



「ディスクからの読み取りの平均時間」および「ディスクへの書き込みの平均時間」カウンターは、ストレージデバイスディスクがオペレーティングシステムで使用可能になるソフトウェアアドインで遅延を直接測定します。 これらを使用すると、関連するハードウェアまたはソフトウェアに関係なく、ドライブおよびハードウェアがI / O要求の処理に費やした時間を正確に測定できます。



OLTPシステムの場合、平均値は15ミリ秒未満で、有効なピークは最大25ミリ秒です。 データの読み取りまたは書き込みにかかる時間が短いほど、システムの機能は速くなります。



CPU使用率制御



Microsoft SQL Serverインスタンスの監視により、CPU使用率レベルが標準範囲内にあるかどうかを判断できます。 一定レベルのCPU使用率は、CPUをアップグレードする必要があるか、複数のプロセッサーを追加する必要があることを示している場合があります。 アプリケーションを最適化すると、CPU使用率を削減できます。 プロセッサシステムには次の操作が大量にロードされています。



  • 実行計画のコンパイルと再コンパイル。
  • ソート;
  • ハッシュ。

<SQL Serverインスタンス> SQL Statistics \ SQL Compilations / sec



SQLコンパイルはすぐに完了しました。 コードコンパイルパスが入力された回数を示します。 SQL Serverのステートメントレベルでの再コンパイルによるコンパイル操作が含まれます。 SQL Serverの「Microsoft SQL ServerのSQL統計」オブジェクトは、SQL Serverのインスタンスに送信されるクエリのコンパイルと種類を監視するためのカウンターを提供します。 クエリのコンパイルと再コンパイルの数、およびSQL Serverのインスタンスが受信したパッケージの数を監視すると、SQL Serverがユーザークエリを実行する速度と、クエリオプティマイザーがそれらを効率的に処理する方法がわかります。 コンパイルでは、リクエストの処理にかなりの時間がかかります。 コンパイルコストを節約するために、データベースエンジンはコンパイルされたクエリプランをクエリキャッシュに保存します。 キャッシングの目的は、将来の再利用のために既にコンパイルされたリクエストを保存することでコンパイルの数を減らし、後で来る可能性のある同様のリクエストを再コンパイルする必要をなくすことです。 ただし、各一意の要求は少なくとも1回コンパイルする必要があります。 要求のコンパイルは、次の要因によって発生する可能性があります。



  • 基本的な変更（テーブルへの列またはインデックスの追加）または統計スキーマ（テーブル内の多数の行の挿入または削除）を含むスキーマの変更。
  • 環境の変更（SETステートメント）。 セッションパラメータの変更（たとえば、ANSI_PADDINGまたはANSI_NULLS句により、リクエストが再コンパイルされる場合があります）。
