レジストラREG.RUは、 テンプレート::ツールキットで記述されたWebサイトの速度を上げる必要に直面し、この問題を解決するように指示しました。 考えられるソリューションを分析した社内の人々が、 Project :: Toolkit（以降TT）の完全な構文を維持しながら、プロジェクトを別のテンプレートエンジンText :: Xslateに移行することに決めたのは嬉しい驚きでした。



Xslateについて聞いたことがありますが、以前は取引していませんでした。 ウェブサイトに記載されている広告データによると、彼はTTを158回追い抜いており、「彼が言ったことの少なくとも半分が正しい」場合（c）、これは優れた選択肢のように見えます。 そのため、疑う余地はほとんどありませんでした。ほとんどのプロジェクトはCで書かれており、少なくとも真珠自体よりも高速でした。 さらに、XslateはTT構文を部分的にサポートしています。 コードを見ると、すべてが「OK」であるように思われました-おそらく著者がテンプレートエンジンに焦点を当てていたため、この方向にXslateを拡張することに興味がある人には基本的なヒントを残したため、完全なサポートはありません（プロジェクトに参加するための一種の招待）。 これはすべて非常に有望に見えたので、Xslateを終了してTTを完全にサポートするか、顧客にとって十分にサポートするまで決めました。REG.RUと私は喜んで同意しました。 すべてが勝者であるように見えました-クライアントは158倍速くコードを受け取り、Xslateは改善された構文と新しいユーザーを受け取り、コミュニティはTT2に代わる優れた代替手段を受け取ります。

Cで書き換えられたものは同じアナログですが、可能な限り軽量です。 また、すべての処理もCで行われます。テンプレートエンジンのアトミック操作（たとえば、上記の連結または[％IF％]、[％CALL％]などのTTに関する操作）は、Cだけでなく、テンプレートファイルのループサイクルが書き込まれます。 これは、（Cで書かれていても）真珠から手順を呼び出すのに必要なクロックサイクルが節約されたことを意味します-これは非常に高価な操作です。 たとえば、TTで書かれたような非常に一般的なテンプレートは
