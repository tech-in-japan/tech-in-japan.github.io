サポートされるブラウザーの限られた範囲





以前のプロトタイプに対するブラウザーの弱いサポートは、情報の更新を配信するために選択されたテクノロジーであるサーバー送信イベントによるものでした。 そこで今回は、古くから実績のあるAjaxテクノロジーを使用することにしました。 Ajaxを使用するとWebトラフィックが増加しますが、私の意見では、最大数のブラウザーで機能するはずです。

Ajaxのもう1つの欠点は、特別な措置を講じない場合、COMポート経由で送信される行をスキップできることです。つまり、行がシリアルポートに非常に早く到着し、Ajax要求があまり頻繁に届かない場合、リクエスト間のすべての行はクライアントに表示されません。 しかし、たとえば現在の温度を表示するタスクの場合、これはまったく怖いことではありません。



おそらくWebSocketsテクノロジーを使用できますが、IEで理解しているように、IEでは10番目のバージョンからしかサポートされておらず、IE 8を持っているので、この方向を考え出すことすらしませんでした。

先日、HabrまたはGiktaymsの記事の1つで、WebSocketの不足を回避できるように思われるSockJSライブラリーについて言及しましたが、まず、特別なライブラリーが必要です。 サポートはサーバー側で行われます。2番目に、この時点でAjaxは正常に機能していたため、気付かれないままでした。

しかし、何よりもZMQライブラリについてお話ししたいと思います-その驚くべき機能を実証するために。

デモZMQサーバーのコードは次のとおりです。 

 import zmq import time context = zmq.Context.instance() pub_sock = context.socket(zmq.PUB) pub_sock.bind( 'tcp://*:12345' ) count = 0 def get_full_line_from_serial(): global count count += 1 return time.strftime( '%Y.%m.%d %H:%M:%S' ) + ': string N %d' % count while True: line = get_full_line_from_serial() print line pub_sock.send( line ) time.sleep(1)


COMポートからの読み取りをシミュレートします。 実際、彼は1秒ごとに新しい行を生成し、それを「公開」（つまり、サブスクライブされたすべてのクライアントに送信）し、デバッグ用に出力します。



水クライアントコード： 

 import zmq context = zmq.Context.instance() zmq_sub_sock = context.socket(zmq.SUB) zmq_sub_sock.setsockopt(zmq.SUBSCRIBE, '') zmq_sub_sock.connect( 'tcp://localhost:12345' ) poller = zmq.Poller() poller.register( zmq_sub_sock, zmq.POLLIN ) while True: socks = dict(poller.poll(timeout=5000)) if zmq_sub_sock in socks: print zmq_sub_sock.recv() else: print 'No data within 5 sec'


ZMQソケットを開き、考えられるすべてのメッセージをサブスクライブし、サーバーに接続します。 次に、無限ループで新しいデータを予期して表示します。

スタンドアロンアプリ





私は、Pythonがスタンドアロンのアプリケーションを作成するのにあまり適応していないことをすでに述べたようです。 これはインタプリタ言語であり、インタプリタプログラムが正常に機能する必要があります。 彼にとって残念なことに、ネイティブバイナリコードを生成できるコンパイラはありません。 スクリプトから一種の独立したアプリケーションを作成できる2つのプログラム、py2exeとpyInstallerを知っています。 私自身は2番目のものをよく使用するため、このプロジェクトで使用することにしました。



pyInstallerはPythonスクリプトを取得し、依存するすべてのファイルを分析し（インポートされたすべてのモジュールとプラグインの動的ライブラリを決定します）、それらをすべて個別のフォルダーに収集するか、自己解凍アーカイブの類似物にパックします。 インタープリター自体のコードと、起動時にランタイムを準備してターゲットスクリプトの解釈を開始する小さな開始部分が追加されます。



Pythonの作成者もライブラリの作成者も、Pythonのそのような使用の可能性について考えていなかったため、すべては「切り株デッキを介して」機能します。 主な問題は、必要なすべてのライブラリとモジュールのリストを準備することです。 多くのライブラリ作成者が暗黙的なインポートを使用しているため、どのモジュールがスクリプトまたは別のモジュールにインポートされるかを明示的に示すことはありません。 したがって、多くの場合、pyInstallerによるスクリプトの自動分析の後、追加のモジュールの名前および/またはそれらを検索するパスを構成ファイル（拡張子.spec）に手動で追加する必要があります。 pyInstallerと一緒に、そのようなライブラリのための既製の関数の大規模なセットがありますが、人生はまだありません。 特に、これはZMQの現在のバージョンで起こりました-pyZMQ（PythonへのZMQバインディング）の開発者は、補助ライブラリのインポートメカニズムを変更しました。その結果、pyInstallerによって構築されたアプリケーションは起動しません。 人々はすでにこの問題に対処しており、pyInstallerに対応するパッチを準備しています。 このパッチは機能していますが、公式リリースにはまだ含まれていないため、pyInstallerに自分の手でパッチを適用する必要がありました。 github.com/pyinstaller/pyinstaller/pull/110/filesにあるパッチコードを参照してください。



ただし、スタンドアロンアプリケーションがフォルダー全体ではなく1つの実行可能ファイルのみで構成される場合、このパッチでさえ十分ではありませんでした。 .exeファイルを解凍するときにZMQディストリビューションのlibsodium.pydライブラリが正しい場所に来るように、手動で.specファイルを修正する必要がありました。



2番目の問題は、マルチプロセッシングモジュールでした。 バックエンドを2つの部分に分割しました。これらは並行して動作するはずです。 バックエンドを実行するために2つの別個のプログラムを実行する必要がある場合、それは間違っているように思えました。 また、マルチスレッドを使用することは、GIL（Global Interpretation Lock）の存在のために間違っているように思われました。 起動時に単純に2つの新しいプロセスを生成する単純なスクリプトを作成する方が簡単に思えます。1つはCOMポートからデータを読み取るZMQサーバー用、もう1つはWebクライアントからのリクエストに応答するHTTPサーバー用です。 実際、通常モードで作業するとき（つまり、インタープリターが「手動で」開始されるとき）、次のスクリプトは正常に機能します。 



 # -*- coding: utf-8 -*- from multiprocessing import Process import serial_to_zmq import zmq2web_using_flask import time def main(): args = get_command_line_params() p1 = Process( target=serial_to_zmq.work, args=(args.serial_port_name, args.serial_port_speed, args.zmq_pub_addr) ) p1.start() p2 = Process(target=zmq2web_using_flask.work, args=(args.zmq_sub_addr,)) p2.start() print 'Press Ctrl+C to stop...', while True: time.sleep(10) def get_command_line_params(): ... if __name__ == '__main__': main()




しかし、pyInstallerで処理した後、結果の.exeは正常に機能しませんでした。 いくつかのZMQサーバーとFlaskアプリケーションが、「ソケットがすでに使用されています」というメッセージとその他の奇妙なメッセージとともに起動されました。 Flaskアプリケーションを起動するときに、debug = Falseパラメーターを渡す必要があることが判明しました。 



  app.run(host='0.0.0.0', debug=False, threaded=True)




また、マルチプロセッシングモジュールの場合、pyInstallerで作成されたスタンドアロンアプリケーションの場合にスクリプトが解釈されるときに作成される、そのモード（froozen）で必要な特殊関数freeze_support（）を呼び出す必要があります。



一般的に、この項目の結果は次のとおりです。Pythonスクリプトからスタンドアロンアプリケーションを作成できますが、簡単ではありません。
