残念ながら、非常に強力なツールが一般に無視され、間違いなく何らかの普及が必要なPowerShellで奇妙な状況が発生しました。 さらに、最近ではWindows 7の一部であり、すぐにかなりの数のユーザーの職場に置かれる予定です。 そして、情報ストアの順序を復元するための、一方では簡潔ですが、他方では興味深い管理タスクという形のこのような機会があります。 それでは始めましょう。



私は少し余談から始めます。 コマンドプロセッサのような一見シンプルなツールについて話す場合でも、崇高なものが必要です。 そして彼を見つけたと思う。 あなたは笑いますが、私はコマンドプロセッサの開発をマルチパラダイムとして定義しました。 最初のパラダイムは必須です。 ほとんどすべてのバッチファイルで見られ、Pythonの元のタスクの例でも見られました。 別のパラダイムは機能的です。 関数型プログラミング言語で使用されるアプローチと類似しているため、これを呼び出しました。 日常生活では、私たちはそれをチームパイプ、単なるパイプ、および他の多くの愛情深い用語として知っています:)簡単に、簡単な例を使用して、その外観を思い出させます：



X:\> (dir /b folder1\*.txt && dir /b folder2\*.txt) | find "text" | sort







ここでは、垂直バーで区切られた3つの指示が表示されます。 後続の各コマンドは、前のコマンドの結果を取得し、それらに対して特定の操作を実行し、パイプキュー内の次のコマンドを転送します。 この例では、標準のcmd.exeは最初の命令を使用して、2つのフォルダーからテキストファイルのリストを収集します。 このリストはfindコマンドに渡されます。このコマンドは、サブストリング「text」を含む行のみを残し、それらはすでにsortコマンドに送信されてソートされます。 関数型プログラミング言語では、これは次のようになります。



sort(find((dir /b folder1\*.txt && dir /b folder2\*.txt), "text"))







いくつかの類似点はありませんか？ 実際、すべてのパイプ要素は関数に似ています。 ちょうど別のタイプのパイプエントリが既存のタイプの関数エントリに追加されました（ポストフィックス、プレフィックス、インフィックスエントリなど）-パイプエントリ:)



例からわかるように、コマンドパイプはMS-DOSの時代からのものであるにもかかわらず、MicrosoftのるつぼのPowerShellの起源を考えると、奇妙に聞こえるUNIXコミュニティに個別に感謝したいと思います。 しかし、これには簡単な説明があります。 UNIXのようなシステムでは、これらのメカニズムが芸術のランクに昇格し、さまざまなチームを組み合わせて最も珍しくて非常に便利な組み合わせにすることができました。



私の意見では、PowerShellがこれを正確に描いて非常に成功した機能を引き出し、それを別の劣らず成功したソリューションと組み合わせました。 同じLinuxのコマンドパイプのラインフィードとは異なり、PowerShellはオブジェクトを操作します。 すべてを明確にするために、タスクの実装を進めることを提案します。 もちろん、パイプを介した「機能的」アプローチを使用してこれを行います。命令型アプローチは、Pythonでの実装の場合の既存のアプローチとほとんど変わらないからです。 そして、ツールだけでなく、パラダイムも比較したいと思います。



私たちのアプローチの機能性により、問題の解決策をこれらのまさに機能のいくつかの見た目に分割し、私たちの場合は、それぞれ独自の厳密に定義された役割を果たします。 それとは別に、PowerShellの機能をさらに表示するために、実装を意図的に少し複雑にします。 実際の条件では、何かを単純化できますが、何かを完全に排除できます。

ステップ1.タスクの実行条件が整ったら、写真のふりをするファイルのリストを作成することから始めます。 問題のステートメントはまさにこれです。ファイル拡張子が有用な情報であるという仮定から進めますが、ファイルが画像であることを保証するものではないからです。 ほとんどの場合、これは必要ありませんが、前述したように、意思決定は意図的に複雑です。 したがって、指定されたフォルダーとそのすべてのサブフォルダー内の指定された拡張子のすべてのファイルを選択します。 最初のパイプステップは次のようになります。



PS X:\> dir $source -r -include *.jpg, *.png, *.gif







「-r」オプションは、ディレクトリの再帰的な走査を意味します。「-include」では、インクルードファイルのマスクをリストできます（または「-exclude」オプションで除外されたマキシをリストできます）。 このコマンドに応答して、ファイルのリストを取得します。



ステップ2.作成しようとしているパイプの次の要素。前の命令から受け取ったファイルごとに、ラスターイメージオブジェクトを作成します。 この手順は、PowerShellの複数の機能を同時に説明するものですが、わかりやすくするために、例を示します。



PS X:\> dir $source -r -include *.jpg, *.png, *.gif | select FullName, @{Name="Image"; Expression={New-Object System.Drawing.Bitmap $_.FullName}} -ErrorAction SilentlyContinue







2番目のパイプ命令で最初に表示されるのは、selectコマンドです。 その目的は、新しいオブジェクトを作成して渡すことです。 これを行うには、関心のある新しいオブジェクトのすべてのプロパティをコンマで区切って選択します。 最初はFullNameです。 この形式で示されるプロパティは、パイプで取得したオブジェクトから取得し、同じ名前と値で新しいオブジェクトに渡すことを意味します。 この例では、ファイルへのフルパスを返すFileInfoクラスのFullNameプロパティについて説明しています。



次の構成はもう少し複雑です。 新しいプロパティが作成され、その名前がNameに渡され、値がExpressionに渡されます。 値として、図（System.Drawing.Bitmap）を記述するクラスのインスタンスを作成し、同じFullName値を図ファイルの場所とともにコンストラクターに渡します。 別に、FullNameプロパティにアクセスするための構文の違いに注意してください。 selectステートメントは、これを簡単な方法で行います。 他のほとんどの場合、変数$ _はパイプで渡されたオブジェクトを意味し、そのプロパティはドットとプロパティの名前でアクセスできます。



使用するファイルがビットマップ図ではない場合、System.Drawing.Bitmapオブジェクトを作成しようとするとエラーが発生します。 これらのエラーを無視するために、ErrorActionオプションを追加して、それらを無視できるようにしました。 このオプションはselectコマンドに固有のものではなく、他のほとんどの指示で使用できるいわゆる共通パラメーターのカテゴリーを指していることに注意してください。



手順3.前の手順の結果に従って、オブジェクトのリストを取得します。各オブジェクトには、ファイル名へのフルパスを持つFullNameと、ビットマップクラスのインスタンスとしての図を持つImageという2つのプロパティがあります。 いずれかのファイルでビットマップイメージのクラスを作成できなかった場合、Imageプロパティは空になります。 そのため、イラストではないすべてのオブジェクトを除外するためのステップが必要です。 新しい命令の追加の結果は次のようになります。



PS X:\> dir $source -r -include *.jpg, *.png, *.gif | select FullName, @{Name="Image"; Expression={New-Object System.Drawing.Bitmap $_.FullName}} -ErrorAction SilentlyContinue | where { $_.Image }







すべてが非常にシンプルで簡潔です。 私たちは、財産に対するおなじみの訴えに会います。 この場合、Imageプロパティに。 新しいwhereステートメントを満たします。これにより、指定された条件を満たすオブジェクトのみをパイプで転送できます。 同時に、空の値の単純なチェックに慣れます。 空ではない場合、条件を制御します！$ _。イメージ、比較演算、論理演算などは、より複雑な条件に関与します。 たとえば、{$ _。Image.Width -gt 1000 -and $ _。Image.Height -gt 1000}の場合、幅と高さが1000を超えるすべてのイラストを取得します。



ステップ4.イラストのリストを受け取った後、それぞれのイラストに対して、イラストを保存するフォルダーの名前を作成します。 このようにします：



PS X:\> dir $source -r -include *.jpg, *.png, *.gif | select FullName, @{Name="Image"; Expression={New-Object System.Drawing.Bitmap $_.FullName}} -ErrorAction SilentlyContinue | where { $_.Image } | select FullName, @{Name="ImageFolder"; Expression={"{0}\{1}x{2}" -f $target, $_.Image.Width, $_.Image.Height}}







選択コマンドと新しいオブジェクトの形成についてはすでにご存じでしょうが、ここでは文字列のフォーマットがより重要です。 この例からわかるように、すべてがフォーマットのある行で始まり、フォーマットに使用される値がリストされます。 ますます、.NET string.Formatメソッドに従い、 MSDNでフォーマット規則を見つけることができます。



ステップ5.この関数の実行結果を見ると、必要なものがほぼすべて揃っていることがわかります。 つまり、元のイラストへの完全なパスはすべて同じFullNameプロパティにあり、宛先パスには新しいImageFolderプロパティのサイズに応じたフォルダがあります。 フォルダーを作成し、そこにファイルをコピー/移動するための継続的な命令がありました。 これを行うには、foreach命令を使用します。これにより、パイプで受信した各オブジェクトに対して他の命令を実行できます。 これはすべて一緒にこのようになります：



PS X:\> dir $source -r -include *.jpg, *.png, *.gif | select FullName, @{Name="Image"; Expression={New-Object System.Drawing.Bitmap $_.FullName}} -ErrorAction SilentlyContinue | where { $_.Image } | select FullName, @{Name="ImageFolder"; Expression={"{0}\{1}x{2}" -f $target, $_.Image.Width, $_.Image.Height}} | foreach {if (-not (test-path $_.ImageFolder)) {md $_.ImageFolder}; copy $_.FullName -destination $_.ImageFolder; $_}







ご覧のとおり、セミコロンで区切られた3つの命令がforeachに配置されました。 2番目は間違いなく単純なコピーなので、長いコメントは必要ありません。 ところで、これは、イラストファイルを移動するためのmoveコマンドで置き換えることができます。 最初の命令はもう少し長くなりますが、それほど複雑ではありません。 条件付きif構造では、論理否定とtest-path構造を使用して、フォルダーが存在しないことを確認します。 フォルダーが見つからない場合は、この場合のみ作成します。 3番目の命令は完全に回避できたかもしれませんが、foreachはパイプの最終命令ではなく、処理を続行できることを彼女に示したかったのです。 最初のステップで、コンソールに変数の値を表示したことを覚えていますか？ したがって、ここで、$ _命令は、パイプによってパイプにさらに受け取ったオブジェクトを表示します。 代わりに、他の何かを印刷できます。 たとえば、foreach {...。;などの変数を定義して出力します。 $ result = ...; $ result}。



要約すると、これらのいくつかの短い行は、この問題に対するわずかに積み上げられたソリューションです。 良いですか、悪いですか-判断するのは難しいです。 私はここにいると思います、あなたの貯金箱にあなたの集合的な知恵の少しを追加するものを含めて:)
