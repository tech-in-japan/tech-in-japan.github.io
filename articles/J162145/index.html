Windows Server 2012でHyper-Vレプリカを構成する

すべての燃えるような気分！



今日、 Windows Server 2012に登場した非常に興味深い革新、つまりHyper-Vレプリカについてお話ししたいと思います。 この技術は実用的です-それをより詳しく知りましょう。









Hyper-Vレプリカの意味は何ですか？





ポイントは、ご想像のとおり、Hyper-Vレプリカイン... VMレプリケーションです。

複製の意味は、誰でも知っていると思います。 この特定のケースでは、Hyper-Vレプリカ機能は、Windows Server 2012に組み込まれた耐障害性メカニズムと見なされ、位置付けられます。

正しく聞いた-耐障害性についてであり、高可用性についてではない-Windows Server 2012では、クラスタリング機能が高可用性に使用されますが、Hyper-Vレプリカは、データセンターサイト外でVMインスタンスを複製できるメカニズムですリモートサイトへ。

WS2012まで、WS2008R2SP1およびSC2012バンドル、つまりOrchestrator + DPMを使用して、耐障害性シナリオが実装されていたことにすぐに注意してください。 VMインフラストラクチャの観点から見ると、Hyper-Vレプリカは寿命を大幅に簡素化しますが、 System Center 2012 SP1 Orchestratorと組み合わせて最適な効果が得られます。



このメカニズムがどのように構成され動作するかを見てみましょう。



Hyper-Vレプリカを構成する





まず、このメカニズムがポイントツーポイントで機能すること、つまり特定のVMがメインホストまたはクラスターからリモートバックアップホストに複製されることを理解する必要があります。両方のオブジェクトは、当然ながらHyper-Vの役割を持ちます。

まず、複製されたデータを受信するようにターゲットホストを構成する必要があります。



1）Hyper-Vマネージャーを起動します。

2）Hyper-Vマネージャーで必要なターゲットホストを選択し、その設定に移動します。

3）「レプリカ構成」項目を選択します。

4）[このコンピューターをレプリカサーバーとして有効にする]オプションと、データ複製の方法を選択します-Kerberos（HTTP）または証明書に基づくより安全な方法（HTTPS）。







5）次に、VMのレプリカを受信できるサーバーとそのターゲットの場所を選択できます。 デフォルトでは、認証されたサーバーからのレプリカの受け入れを許可します。

6）要件に応じて、Hyper-VレプリカHTTPリスナー（TCP-In）またはHyper-VレプリカHTTPSリスナー（TCP-In）の受信ホストのファイアウォールで例外が許可されていることを確認してください。



次に、レプリケーション用にVMを構成する必要があります。



1）レプリケーションに必要なVMを選択して右クリックし、[ レプリケーションを有効にする]を選択します。

2）表示されるレプリケーションセットアップウィザードで、最初のダイアログボックスと次のダイアログボックスで[ 次へ ]をクリックします。

3）VMを複製するターゲットサーバーのFQDN名を指定し、[ 次へ ]をクリックします。

4）認証方法（Kerberosまたは証明書ベース）および送信中にデータを圧縮するかどうかを選択します。 次へをクリックします。

5）複製するVM仮想ディスクを選択し、[ 次へ ]をクリックします。

6）復旧ポイントパラメータの設定- 最新かつ唯一の復旧ポイントのみを使用するか（最新の復旧ポイントのみ） 、特定の周期で複数のポイントを作成する機能を使用することを決定します（追加の復旧ポイント） -決定します。 2番目の場合、必要な復旧ポイントの数と、作成間の間隔を指定できます。 WS2012は、VSSを増分VMスナップショットとともに使用して、レプリケーションメカニズムを提供します。 選択したら、 「次へ 」をクリックします。







7）次に、ネットワーク、外部メディア、またはターゲットサーバー上の既存のVMを使用して、プライマリレプリケーションを実行するメカニズムを選択する必要があります。 複製開始時間を選択して、 「次へ」をクリックします。

8）設定の概要を確認し、[ 完了 ]をクリックして構成プロセスを完了します。



レプリケーション統計を表示するには、必要なVMの[ レプリケーション ]タブをクリックします。 また、VMのネットワーク設定で、VMレプリカのバックアップ起動の場合の代替ネットワーク構成を指定できます。

複製に関するかなり詳細なビデオもあります



いくつかの注意事項





VM内にドメインコントローラーがある場合、VMレプリケーションを使用しないでください。レプリカが復元されると、コントローラー内でSIDセキュリティキーを生成するシーケンスが失敗する可能性があるためです。 これは、レプリケーションメカニズムが非同期で5分ごとに更新を送信するために発生する可能性があります。この間、セキュリティキーは更新できますが、レプリカは更新されません。 このようなシナリオでは、バックアップサイトにドメインコントローラーの別のインスタンスを保持することをお勧めします。

そして一般的に-これはアドバイスです：下位レベルとは別のレプリケーションメカニズムがある場合（たとえば、アプリケーション自体がレプリケートされ、アプリケーションレベルがVMのコンテナレベルよりも高い場合-これはインフラストラクチャレベルであるため）-これらを使用する方が良いネイティブメカニズム。



一般に、VMレプリカの構成に複雑なことはありません。

実際の環境ですべてがどのように機能するかを確認するだけです-メガ管理者からのフィードバックを喜んで聞きます！
