JBoss 7 Cluster-Apacheを使用した負荷分散

前回の記事では、JBossクラスターのすべてのノードへのユーザーセッションレプリケーションを構成しました。 このようなアクションだけではフォールトトレランスは向上しません。受信した機能を使用するには、クラスターのノード間で外部呼び出しを分散するロードバランサーが必要です。 JBossドキュメントの推奨事項に従って、Apache Webサーバーによってノード間で負荷が分散されます。 記事のすべての情報はさまざまなソースで入手できますが、散在しており、実際に発生する可能性のある問題の解決策の説明とともに、すべてが1か所で収集されるリソースに出くわしませんでした（私は個人的に持っていたので、レシピを共有しています） 。 この記事は完全な情報のふりをするのではなく、逆のことです-最小構成について説明します。 特定の構成と設定に関心がある専門家と、このトピックから遠く離れているが、クラスタリングとは何かに関心がある人々の両方を対象としています。



仕事の一般原則


クラスタリングの原則に精通している専門家は、このセクションをスキップできます。 明確にするために、多くの人が直面している特定の状況を考えます。1つのアプリケーションサーバーで構成される既存のシステムがクラスターソリューションに切り替わります。 そのような場合の最良の選択肢は、広大さをすぐに把握するのではなく、最小限の作業構成を構築することです：クラスター内の2つのアプリケーションサーバー、およびその前にクライアントへのインターフェイス（フロントエンド）としてのWebサーバー。



作業のスキームは次のとおりです。以前にJBoss http / httpsコネクターに直接送信されたすべてのリクエストは、Webサーバー（Apache）にアクセスする必要があります。 Apacheは、2つのアプリケーションサーバーの「背後」での存在を「知る」ように構成されています。 クライアントが最初にシステムにアクセスすると、Webサーバーはアプリケーションサーバー（ Node-1 ）の1つを選択し、要求をそれにリダイレクトします。 セッションが作成され、Cookieが追加されます。このCookieは、同じクライアントの後続のすべての要求を選択したアプリケーションサーバーに「固定」するためにWebサーバーによってさらに使用されます。 Node-1は 、クライアントリクエストを処理し、セッションを作成/変更するときに、その状態をクラスターの他のノードに複製します。

セッションの詳細
簡略化。 セッション-JBossの側で作成され、さまざまな属性を持つオブジェクト。 セッションには、セッションの作成時にブラウザーに送信される識別子があり、クライアントがサーバーにアクセスするたびに、ブラウザーによってサーバーに送信されます（通常はCookieの形式で）。 サーバーは識別子によってセッションを識別し、見つかったセッションのデータのコンテキストで要求を処理します。 クラスターでのレプリケーションのタスクは、クライアントがセッションにアクセスしたときに、他のノードが識別子でセッション（およびその中に保存されたデータ）を見つけ、ノードのようにクライアントのリクエストを処理できるように、セッションデータを他のノードに送信することです彼女の文脈で。
Host-1の 「クラッシュ」が発生した場合、Apacheは次のクライアント呼び出しで、サーバーが利用できないことを検出し、リクエストを別のホスト（ Host-2 ）にリダイレクトします。 これは、非常に「デッドウェイト」が使用され始める場所です。その状態は、クラスタのすべてのノードに既に存在しています。 リクエストはNode-2によって処理され、Apacheはセッションを「固定」します。つまり、 以降のすべてのユーザーリクエストは、すぐにNode-2に送信されます。



基本設定


セッションレプリケーションがJBoss側ですでに構成されている場合、追加の設定は必要ありません。必要なもの（ AJPコネクタ）はすべてデフォルト構成ですでに構成されています。 Apache HTTPサーバーをダウンロードしてインストールする必要があります（インストールプロセスはこの記事の範囲外です）。 すべての構成は、conf / httpd.confファイルを編集することにより行われます。

  1. モジュールの接続 。 次の行をコメント解除または追加します。 

    LoadModule proxy_module modules/mod_proxy.so LoadModule proxy_ajp_module modules/mod_proxy_ajp.so LoadModule proxy_balancer_module modules/mod_proxy_balancer.so LoadModule headers_module modules/mod_headers.so


    ロードバランサーには、mod_proxy.so、mod_proxy_ajp.so、およびmod_proxy_balancer.soモジュールが必要です。 システムに「セッション」の概念がない場合は、十分です。 たとえば、Webサービスを介してリクエストを処理するシステムを使用しました。これらは相互に関連しない単一の呼び出しであるため、セッションは作成されませんでした。 この場合、ユーザーとそのセッションが存在するため、mod_headers.soモジュールを追加する必要があります。これにより、ブラウザーによって送信されたセッションデータを追跡できます。

  2. バランス調整 。 次の行を追加します。 

     NameVirtualHost 192.168.1.0:80 <VirtualHost 192.168.1.0:80> Header add Set-Cookie "ROUTEID=.%{BALANCER_WORKER_ROUTE}e; path=/" env=BALANCER_ROUTE_CHANGED <Proxy balancer://ajp-cluster> BalancerMember ajp://192.168.1.1:8009 route=node1 BalancerMember ajp://192.168.1.2:8009 route=node2 ProxySet stickysession=ROUTEID </Proxy> <Location /> ProxyPass balancer://ajp-cluster/ </Location> </VirtualHost>


    192.168.1.0の代わりに、Apacheアドレスを指定します。

    [ プロキシ]セクションで、クラスターノードのAJPコネクターのアドレスのリストを指定します。ノードごとにBalancerMemberが1回出現します。 JBoss 7構成では、デフォルトで、AJPコネクターはポート8009でリッスンします。 ルートパラメーターは、Apache自体でのみ使用される一意のホスト名を割り当てます。 JBossの開始時に指定された名前（ パラメーターjboss.node.name ）と一致する必要はありません。 これ以上変更する必要はありません;クラスターは起動とテストの準備ができています。
    構成の詳細
    Header add Set-Cookieディレクティブは、次のようにProxySet stickysessionディレクティブと連携して機能します。 クライアントが最初にApacheを呼び出すときに、バランサーがリダイレクト先のノードを選択した後、「 ROUTEID 」という名前のCookieがユーザーセッションに追加され、その値は選択したノードの識別子（上記の構成のnode1 / node2）です。 その後のクライアント呼び出しで、Apacheは以前にどのノードが選択されたかを「考える」必要がなくなり、CookieはリクエストをリダイレクトするノードにWebサーバーに明示的に指示します。 Headerディレクティブはセッション内のCookieの存在を担当し、ProxySet stickysessionディレクティブはセッションを特定のノードに「固定」するためにルーティングするときに使用します。 サーバーがセッションで機能しない場合（たとえば、上の例のようにWebサービスのみ）、HeaderおよびProxySetディレクティブは必要ありません。

    ジョシオニド
    多くの構成例では、セッション識別子JSESSIONIDをノードマーカーとして使用することをお勧めします。 これは論理的なようです、なぜなら この識別子は、ほとんどのJava Webアプリケーションで使用されます。 apache.orgの例： 
     ProxyPass /test balancer://mycluster stickysession=JSESSIONID|jsessionid scolonpathdelim=On <Proxy balancer://mycluster> BalancerMember http://192.168.1.50:80 route=node1 BalancerMember http://192.168.1.51:80 route=node2 </Proxy>
    すぐに言っておく必要があります。JBossの場合、これは少なくともデフォルトの構成では機能しません。 このメソッドは、アプリケーションのサーバー側で形成されるJSESSIONIDという名前のCookie値が2つの部分で構成されていることを前提としています。セッションIDはサーバーIDで補完され、ドットで区切られます。 JBoss 7は、セッションIDのみを含む単純なクッキーJSESSIONIDを形成します。セッションIDの値はランダムに生成され、意味を持ちません。

    したがって、説明した構成との主な違いは次のとおりです。

    • この記事で説明されている構成では、ノード識別子とセッション識別子は異なるCookieにあります。ここでは、2つの値が1つのCookieに結合されています。
    • この記事で説明されている構成では、ノード識別子はバランサーの側で形成され（論理的です：使用者は誰でも）、アプリケーションサーバー側のセッション識別子（同じ原則：セッション識別子はApacheではなくJBossで必要です）。 ここでは、両方の値がアプリケーションのサーバー側（またはバックエンドのWebサーバー）で生成されます。






SSLセットアップ（HTTPS）


クラスタリングの前に、クライアントとデータを交換するときに暗号化を使用し、HTTPSプロトコルを使用してクライアントと通信した場合、HTTPSコネクターの構成でJBoss側で証明書設定が実行されました。 クラスター化されたソリューションに移行する場合、Apacheは証明書を管理する必要があります。



一般原則
プロセスは次のとおりです。クライアントからのHTTPSリクエストは、AJPプロトコルを使用して、Apache Webサーバーに送信され、復号化され、クリア形式でアプリケーションサーバーに送信されます。 アプリケーションサーバーからの応答もWebサーバーに平文で送信され、そこで暗号化され、HTTPSプロトコルを介してクライアントに送信されます。 したがって、ApacheとJBossの間には、いわゆる「非武装地帯」があります。 外部の不正アクセスから閉じられ、アプリケーションレベルで不正アクセスに対する追加の保護を必要としないサーバー間に信頼関係が確立される環境。 HTTPSポート（通常は443）を介して外部からアクセスできるのはApacheのみであり、他のすべては密に閉じる必要があります。







conf / httpd.confのWebサーバー設定：

  1. ポート443盗聴 。 デフォルト設定のApache Webサーバーは、ポート80のみをリッスンします。ポート443をリッスンするには、次の行を追加します。 

     Listen 443
  2. モジュールの接続 。 次の行をコメント解除するか、追加します。 

     LoadModule ssl_module modules/mod_ssl.so


  3. バランス調整 。 次の行を追加します。 

     NameVirtualHost 192.168.1.0:443 <VirtualHost 192.168.1.0:443> Header add Set-Cookie "ROUTEID=.%{BALANCER_WORKER_ROUTE}e; path=/" env=BALANCER_ROUTE_CHANGED <Proxy balancer://ajp-cluster> BalancerMember ajp://192.168.1.1:8009 route=node1 BalancerMember ajp://192.168.1.2:8009 route=node2 ProxySet stickysession=ROUTEID </Proxy> <Location /> ProxyPass balancer://ajp-cluster/ </Location> ProxyRequests off SSLEngine on SSLCertificateFile c:/Apache2/conf/my.cer SSLCertificateKeyFile c:/Apache2/conf/my.key </VirtualHost>




前の構成に加えて、ここでは、サーバーがトラフィックの暗号化に使用する証明書と秘密キーへのパスを指定する必要があります。



Apache 2.2の設定は上記のバージョン2.4以降で説明されており、mod_lbmethod_byrequests.soおよびmod_slotmem_shm.soモジュールを追加し、NameVirtualHost宣言を削除します。これらは不要になりました。

証明書の問題とその解決策
JBoss側で暗号化を設定する場合、たとえばPKCS＃12形式のキーと証明書を含む1つのファイルが必要でした。 Apacheでは、2つのファイルに分割する必要があります。1つのファイルには証明書があり、もう1つのファイルには秘密鍵があります。 さらに、Apacheには2つの制限があります。

  1. PEM形式のみが認識されます。 キーファイルをテキストエディタで開くと、キーがこの形式に一致するかどうかを確認できます。 コンテンツには「----- BEGIN PRIVATE KEY -----」および「----- END PRIVATE KEY -----」という行が含まれている必要があり、それらの間の行の長さは64文字を超えてはなりません。 同じことが証明書にも適用され、それぞれ「----- BEGIN CERTIFICATE -----」および「----- END CERTIFICATE -----」の文字列が使用されます。
  2. WindowsでApacheをビルドすることは、パスワードで保護されたキーストアを開く方法を知りません。


上記の問題はすべて、 Opensslユーティリティを使用して解決されます。 JBossで使用されるキーと証明書のストレージファイルから、キーと証明書を個別のファイルに抽出し、必要な形式に変換してパスワード保護を解除する必要があります。 PKCS＃12形式からキーを変換する例を次に示します（他の形式もサポートされています。Googleがお手伝いします）。 

 openssl pkcs12 -nocerts -nodes -in C:\key.p12 -out C:\Apache2\conf\my.key openssl pkcs12 -clcerts -nokeys -nodes -in C:\key.p12 -out C:\Apache2\conf\my.cer


注意！ 受信したmy.keyファイルを産業用サーバーに転送した後、コンピューターから削除することを忘れないでください。 彼は何によっても保護されておらず、侵害される可能性があります。


おわりに


記事に記載されているソリューションを適用した後、原則として、JBoss設定からHTTP / HTTPSコネクターの説明を不要として削除できます。 これで、すべての外部コールはAJPコネクタを介して着信します。 これにより、特にこれらのプロトコルに特に注意を払うスキャンユーティリティの場合、サーバーのセキュリティが大幅に向上します。
