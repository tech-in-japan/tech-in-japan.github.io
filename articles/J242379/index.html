高度なGulpとBrowserify：興味深いトリック

数週間前、私は非営利の音楽プロジェクトを行うサイクルを開始しました（最初の投稿は「I PR」にありますが、リンクは付けません）が、残念ながら最初の記事に夢中になり、彼が具体的にどのようにそれをしたか、彼は他のプロジェクトから効果的なトリックを思い出し始めました。 どうやら、これは、プロジェクト自体への所定の重点とともに、UFOが私の背後に飛んで来たという事実とポストにつながった。



しかし、記事にあるすべては少なくともほとんど知られておらず、その半分は私が確信しているように一般的に一意であり、これらの各ヒントはgulpでの作業を大幅に促進することができるので、この資料が取り消せない場合は本当に申し訳ありませんになります。



そのため、プロジェクトへのすべての参照を削除し、実際には誰も見たことのない記事を（改訂および修正を加えて）再公開しようとしました。 あなたが無愛想なファンなら、少なくとも2番目の部分を読んでください。gulpが好きではないという事実は、browserifyが好きではないという意味ではありません。



要約：

  1. エラーを処理する簡単な方法。
  2. ソースファイルを格納するためのユニバーサル構造。
  3. 複数のストリーム（たとえば、コンパイルされたコーヒーとjs）を1つに結合します。
  4. テキストのストリームを作成します。
  5. Browserify用の独自のプラグインを作成します。
  6. BrowserifyのGulpプラグインからプラグインを作成します。








1つのビデオポータルで作業しているときに、 gulpに切り替えました。 今、彼らは順調に発展しており、私のビルドシステムはまだ彼らと一緒にいるようです。

その後、 スタイラスに切り替えました。



なんで？ 非常に高速です。

私は記憶から多くの数字を与えるので、私は少し嘘をつくことができますが、比率は同じです、私はそれらを正確に覚えています。



入門：

  • うなり声;
  • 相互にほとんど関係のない多数のスタイルファイル（混乱しない場合は73）があります。アセンブリ-接着によるものです。 基本的に、スタイルはコンポーネントであり、ページフックおよびレイアウトであるため、互いに独立しています。ほとんどの場合、変数のセットのみが接続されています。
  • 1つにまとまる多数のスクリプトがあります。 CoffeeScriptがコンパイルされ、通常のjsは変更されません。 各ファイルはクロージャーでラップされます。
  • グローバルスペースのフォルダーからすべてのテンプレートを使用してオブジェクトを作成する特別なプラグインを使用して、多数のプリコンパイルされたヒスイテンプレートからアセンブルされたアプリケーションファイルがあります。 実際、出力もjsファイルです。
  • 各アイテムの組み立て時間は、Macbook Air 2013、i5では合計で約10秒、約30〜40秒です。 PCIeを介して接続されているMBAにはSSDがあり、ハードドライブの操作速度を提供します。これは、通常のSATA接続では原理的に達成できないようです。 HDDを搭載したコンピューターでは、組み立てに1分以上かかる場合があります。




すべては、ハードドライブでの作業にギャグがあることを示しており、3つのタスクのうち1つだけで、何らかのキャッシュですべてを解決できます。



問題の研究はまた、スタイルが最も長くコンパイルされたことを示しました。



この問題の解決策は次のとおりです。



  • gruntをgulpに置き換えます。 コンパイルとスタイルとスクリプトの接着の問題は解決されました-個々のファイルのディスクへの書き込みのステップは削除されました。
  • sassをスタイラスに置き換え、すべてをグローバルファイルのインクルードに変換します。 スタイルのコンパイルは1秒未満に加速されます。 どうやら、ノードからルビーに各ファイルを転送すると、多くのリソースを消費しました。 そして、ルビーサスは非常に高速ではありません。 ちなみに、転送はまったく問題なく行われました-sassは基本的なものを使用し、mixinと関数を使用せず、特定の観点から、sass形式はスタイラス形式のサブセットと呼ぶことができます。
  • すべてのcoffeescriptをJSに変換してコンパイルを高速化するのは良いことです。これらはほとんどが古いウィジェットでした。
  • アプリケーションのjsアセンブリを翻訳して、キャッシング用にブラウザー化します。




その結果、プロジェクト全体のアセンブリには、個々のファイルタイプの約3〜4秒、約1秒かかりました。

もちろん、ウォッチドッグは一般的に即座に解決しました。



それ以来、gulp + browserifyスタックを使用しています。



ブラウザーブラウザーの利点-ウィジェットをクロージャーでラップすることと、本質的に最も単純なコード検証-解析できないコードを見逃しません。

エラー処理



最初の解決策は次のようなものでした： 



gulp.task('build-html', function () { ... .pipe(plugins.jade()).on('error', console.log.bind(console)) ...




しかし、判明したように、これはあまり良くありません。ストリームは「フリーズ」し、更新時に再起動しません。

なぜこれが起こっているのですか？ 実際には、タスクは終了せず、この状態でハングしたままであり、一言で言うと、前の実行が終了しないと再起動は不可能であるようです。



私は長い間この問題の良い解決策を探していましたが、やがてこの問題を解決するための独自の関数を書きました： 



 function log(error) { console.log([ '', "----------ERROR MESSAGE START----------".bold.red.underline, ("[" + error.name + " in " + error.plugin + "]").red.bold.inverse, error.message, "----------ERROR MESSAGE END----------".bold.red.underline, '' ].join('\n')); this.end(); } gulp.task('build-html', function () { ... .pipe(plugins.jade()).on('error', log) ...




また、ストリーム（this.end（））を閉じて、タスクを完了させます。



必要に応じて、たとえばうなり声のアラートを追加できますが、個人的には十分です。



この関数は、事前定義されたカラーnpmパッケージを必要とし、非常に良い結論を与えます。 余分なパッケージを配置したくない場合は、花からメソッドを削除できます。



ここで最も重要なことは、最後の行です。



this.end（）を実行すると、特定のgulpタスクが終了します。 はい、これはメモリ内で少しがらくたですが、ウォッチドッグタスクを更新すると、スタイルアセンブリを再実行できるようになります。

プラグイン接続



10〜20個のプラグインがある場合、それらを手動で記述するのは多少面倒になります。



このための別のプラグインがあり、プラグインメソッドでプラグインオブジェクトを作成しますが、同じことをより明確かつ簡単に行うことができます。 



 var gulp = require('gulp'), plugins = {}; Object.keys(require('./package.json')['devDependencies']) .filter(function (pkg) { return pkg.indexOf('gulp-') === 0; }) .forEach(function (pkg) { plugins[pkg.replace('gulp-', '').replace(/-/g, '_')] = require(pkg); });




誰かがこのコードの正確な動作を理解していない場合、package.jsonのdevDependenciesのコンテンツを開き、gulp-で始まるすべての要素をプラグイン[pluginName]として接続します。 プラグインがgulp-css-base64のような名前の場合、plugins.css_base64で入手できます。
