Acronis Backup 12.5（現在および）Advanced：待望のリリース

背景



Acronis Backupソリューションバージョン11.5 / 11.7は5年以上前にリリースされました。この製品はまだ世界中のデータを正常に保護していますが、すでに時代遅れになっており、根本的に新しいアプローチを必要とする客観的に必要な変更が成熟しています。 新しいトレンドに対応したAcronis Backup 12プラットフォームが約1年前にリリースされたにもかかわらず、「Advanced」エディションは常にバージョン11.7のままであり、互換性の問題と非常に異なるアーキテクチャソリューションを同時にサポートする複雑さをもたらしました。 最後に、Acronis Backup製品ライン全体が単一のプラットフォームで利用できるようになりました。 この記事は、Acronis Backup 12.5のリリースと、実際、このバージョンの新機能について説明しています。



Acronis Backup 12.5には、標準と高度の2つのフレーバーがあり、ハイブリッド環境向けの本格的なバックアップシステムとしての価値があることを思い出してください。 シンプルで便利なWebインターフェースは、世界中のどこからでも物理サーバー、仮想サーバー、クラウドサーバーの信頼性の高いバックアップとリカバリを提供します。







なぜ新しいバージョンですか？ そして同様に！ （いいえ）



問題点：バージョン11.7にはすべての利点がありますが、増え続けるデータの世界で使用できない多くの欠点があります。 わずか5年前、組織全体のデータ量は数テラバイトを超えていませんでしたが、今日では、請求書はすでに数十から数百に上っています。



現代の現実に適応するために最初に必要な変更は、すべてのコンポーネントを含むx32ビットプレーンからx64へのアプリケーションの変換です。 これにより、インフラストラクチャを拡張しながら大量のシステムリソースをサポートし、x32形式では使用できなくなったサードパーティライブラリをサポートできます。



ライブラリの例
このようなサードパーティのコンポーネントの例として、バージョン6.0からx64形式へのVMware VDDKの移行を挙げることができます 。これにより、Acronis Backup 11.7 Advancedの以前のバージョンでVMware vSphereの新しいバージョン（特に6.5）をサポートすることが非常に難しくなりました。



新しいプラットフォームの2番目の明らかな前提条件は、IT分野でのクラウドコンピューティングへの一般的な動きです（少なくとも今日は）。



新機能



Acronis Backup 12.5のすべての改良点をリストするつもりはありません（これは常に公式Webサイトで確認できます ）が、最も重要なもののみに焦点を当てます。これは、製品を使用するとき、または注目に値するノウハウを表すときに最も需要があると考えられます。



機能の点でのAcronis Backup 12.5 StandardとAdvancedの違いはこちらにあります 。



独立したデータプラン



バックアップの複製、検証、仮想マシンへの変換、古いバックアップのクリーンアップ（クリーンアップ）など、バックアップに関連する操作を実行するための独立した独立した計画を作成できるようになりました。







以前のバージョン11.7でこれらの操作が全体的なバックアップ計画の一部であり、次々に実行されていた場合、より細かい調整の可能性があります。



たとえば、平日に基本的なバックアップを作成し、週末に複製することができます。 同時に、この複製は、バックアップを直接作成するプロセスに関与しない独立した独立したエージェント（たとえば、DMZに立つ）によって実行できます。



これらの変更の一部は、バックアップ検証機能の拡張でした。 バージョン12.5より前では、検証は次の方法でのみ実行されていました。



  • バックアップを作成するとき、特定の数のセクターのチェックサムがアーカイブに書き込まれます
  • 検証中に、これらのチェックサムはアーカイブからセクターを差し引くことにより再計算され、アーカイブに記録されたものと比較されます
  • チェックサムが一致する場合、すべてが正常であり、一致しない場合、アーカイブは破損していると認識されます


拡張は、アーカイブから仮想マシンを起動することによる新しい検証オプションで構成されます 。 これは、仮想マシンだけでなく、 物理マシンのバックアップを含むVMware vSphereまたはMicrosoft Hyper-Vハイパーバイザーがある場合に機能します。 つまり、イメージの取得元のプラットフォームに関係なく、選択したハイパーバイザーで仮想マシンを実行できます。これは、Acronis Universal Restoreテクノロジーがオンザフライで適用されるためです。



最初のバージョンでは、実行中の仮想マシンの有効性は、開始後にVMware ToolsまたはHyper-V Integration Servicesからの「ハートビート」応答をチェックすることによってチェックされます。 少なくとも、リカバリ中にゲストOSが起動することを確認してください。

次の更新では、利用可能なチェックが拡張され、「ハートビート」に加えて、実行中のVM内でスクリプトが実行されます。

階層管理モデル



専任の管理者に管理を委任して、組織内に「部門」（いわゆる「ユニット」）を作成する機能を追加しました。



次のように機能します。主任管理者は、Acronis Backup 12.5をインストールし、モスクワとサンクトペテルブルクのオフィスなど、組織のWebコンソールインターフェイスに「部門」を作成します。







次に、管理者は、ドメインからアカウントを選択することにより、作成された「部門」に任命されます。



部門管理者を追加する




その結果、「Departments」の管理者の下でWebコンソールで認証する場合、この「Division」に属するリソースのみが使用可能になります。 リソースは、仮想または物理マシン、アプリケーションデータベース、Office 365メールボックスなどの保護されたエンティティ、およびバックアップストレージの場所です。



リソースのバインドは、バックアップエージェントのレベルで発生します。 有効なエージェントfor VMware（またはエージェントfor Hyper-V）を特定のグループに登録すると、このエージェントに「見える」すべての仮想マシンがこの「部門」に自動的に含まれます（仮想マシンを個別のサブグループに分割することはまだできません。近い将来これを実装する予定です）。



「部門」へのエージェントの登録は、インストールの段階で実行されます。インストールがWebコンソールからリモートで実行される場合、エージェントは、管理が実行されるコンテキストで対応する「部門」に自動的に登録されます。 「部門」間の切り替えは、組織の「メイン」管理者が使用でき、部門管理者は使用できません。



部門間の切り替え




それ以外の場合、バックアップおよびデータ管理タスクの管理は、どの部門の管理者にとっても変わりません。



監査は、「部門」の管理者によって実行されるすべての操作に対しても実行されます。 「メイン」管理者。さらに、部下の行動を監視する機能を含む、すべてのリソースを管理できます。

ダッシュボード、アラート、およびレポート



バージョン11.7のダッシュボードのレビューを聞いて、情報が不十分で、時には「スパム」アラート（アラート、別名「アラート」）であることがわかりました。したがって、バージョン12.5ではこのパネルが大幅に再設計されました。



特に、「アラート」のシステムに対して次のことが行われました。



  • 「アラート」を解決するためのスキームが変更されました。「アラート」の発生につながった条件が満たされなくなった場合、警告は自動的に無効になります。 たとえば、スペースが解放された場合、バックアップストレージの警告と空き容量の不足は自動的に削除されます。



  • 「予想されるバックアップがシャットダウンエージェントのために作成されなかった」、「バックアッププロセスの凍結」、「別のタスクとのスケジュールの競合によりバックアップがスキップされた」など、以前に検出されなかった状況を解決できる新しいタイプのアラートが追加されましたその他。


ダッシュボードも大幅に変更され、「ウィジェット」を追加することで好みや色にカスタマイズできるようになりました。「ウィジェット」には追加の設定が含まれ、インタラクティブで、最大限の使いやすさが提供されます。







また、Acronis Monitoringは同じ「エンジン」で動作し、レポートを作成します。レポートは、各「部門」ごとにカスタマイズ可能で、定期的に電子メールで通知を送信したり、.pdfとして保存したりできます。

Acronis NotaryおよびAcronis ASign



また、Acronis Backup 12.5のコンテキストでは、ブロックチェーンテクノロジーのサポートと、それに基づくAcronis NotaryおよびAcronis ASignツールのサポートに言及する価値があります。これらは、それぞれデータの不変性とドキュメントのデジタル署名を保証することを目的としています。



これらのツールは、Acronis True Image 2017 New Generationですでに利用可能でしたが、現在では企業向けソリューションで使用されています。



Acronis Notary、その仕組み：個々のファイル/フォルダーのバックアップコピーを作成すると、ハッシュツリーが作成されます-保護されたファイルごとにツリー内に1つのエントリがあり、このツリーがアーカイブに書き込まれます。 次に、このツリーのハッシュが計算され、 notary.acronis.com Webサービスを介してイーサリアムデータベースに追加されます。 ファイルをチェックするとき、目的のハッシュがイーサリアムに要求され、アーカイブ内のツリーのハッシュに関して計算されたハッシュと比較されます。

すぐに機能に注意してください。



  • 「公証」は、ファイルアーカイブに対してのみ機能します。 つまり ディスクまたはセクションを含むアーカイブは「公証」の対象ではありません。
