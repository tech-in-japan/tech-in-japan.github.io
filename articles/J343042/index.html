HANAの初期計画



HANAの最初のSAP計画は非常に楽観的でした。 Hasso Plattnerによって擁護され、計画は非常に広範であったため、これらすべての説明は別の記事になります。 ただし、次のものが含まれていました。



  • 書籍： Hasso Plattnerが記憶の分野で書いた4冊の本。
  • マーケティングとPR：エンタープライズソフトウェアの歴史の中で最大の努力とコストの1つであったに違いない、HANAのマスマーケティングとPR。
  • 誇張された利点： 記事がHANAの利点を誇張している場合 、HANAに多大な利点を提供します。
  • 単純化の幻想的な概念：モデルが単純化されているため、HANAは実際にはSAPの複雑さを大幅に減らし、プロジェクトのビジネスプロセスを減らすというメッセージ。
  • HANAの人為的なS / 4HANA制限の作成：ここでSAPはベストセラーのERPシステムを採用し、S / 4HANAと呼ばれる新しいバージョンではHANAのみに制限しました。 この制限を作成する真の技術的理由はなかったため、これは常に反競争的な動きでした。 しかし、いつものように、連邦取引委員会と欧州貿易委員会はこれに注意を払わず、声明を出しませんでした。 コンセプトは、多くの顧客がS / 4HANAに移行し、その後、HANAの購入を余儀なくされることでした。 しかし、記事「SAPが顧客数の認識を管理する方法」で説明されているように、S / 4HANA市場の採用は信じられないほど低かったため、このアプローチはまだ開発されていません。
  • 普及と配布： SAPは、データベース層でHANAを受け取った後、次の戦略を採用しました。 SAPには、HANAの購入後に顧客に提供されるあらゆる種類のデータベースと製品を操作するためのツールがあります。 HANAの問題の1つは、SAPがHANAのコピーを購入するときに、HANAを実行するために必要なすべてのコンポーネントを顧客に開示しないことです。 顧客は、購入後に追加の製品について学習します。 第二に、SAPは、HANAのデータをアーカイブするために、SAP IQ、正式にはSybase IQなどの遅延データベースを配置しました。 議論は、SAP IQがHANAとのやり取りがはるかに良好であり、計画によれば、企業がサードパーティ企業のデータベースをプッシュすることを強制することであるはずでした。


データベースの市場シェアを使用してOracleがアプリケーション層に侵入したため、SAPはデータベースレベルでの成長（買収ではなく）を計画していました。 したがって、彼らはOracleを「彼らの」アカウントから追い出すつもりでした。 サードパーティのソフトウェアプロバイダーとして、SAPアカウントにソフトウェアを販売する場合、このアカウントも所有していると考えられることに注意してください。



SAPのデータベースレベルへの移行は、不正確な情報に基づいていました。 Oracleはアプリケーション層に侵入しました。 これはいくつかの理由で私が好きではないものです。買収は市場集中につながり、市場集中は常に競争と妨害の減少につながるからです。 しかし、Oracleには実際の価値を提供する実際のアプリケーションがありました。



市場の非効率。 通常、購入は同じスペースを支配する企業によって行われます。 データベース内のOracle。 大企業向けERPのSAP。 オペレーティングシステムおよびオフィススイートのマイクロソフト。 これらの「過剰」利益は、関連分野の企業を購入するためにしばしば使用されます。 ただし、これは利益の低下につながります。 オラクルの利益の半分以上はまだデータベースから得られています。 マイクロソフトの収益の4分の3は、WindowsとOfficeからのものです（XBox、電話などからではありません）。 官僚制が増加しており、追加は中核よりも効果が低いため、買収（および中核以外の地域での成長）は企業の効率の低下を意味します。 このルールに反対していると思われる唯一の会社はAmazonです。 Amazonはあらゆる方向に動くことができ、同時に非常に効果的であるようです。 さらに、AWSなどの地域に移行します。AWSは、開始した地域（オンライン小売）よりも高い利益をもたらします。



市場の集中：獲得は市場の集中につながり、市場の集中は常に競争とブロッキングの減少につながります。 このレッスンは、すべての世代が検討する必要があります。 しかし、Oracleには実際の価値を提供する実際のアプリケーションがありました。 したがって、私はこれに同意しませんが、連邦取引委員会（ドーナツの送信にほとんどの時間を費やしているようです）は、Oracleの大規模な買収を承認すべきでした。 これが、ほとんどの場合、ソフトウェアの取得が政府の承認を得るのに良い考えではない理由です。

HANAのSAP計画は非常に贅沢だったため、ファンタジーのカテゴリーに分類されました。 数年前、私はコンサルティング会社に勤務し、HANAとS / 4HANAで教育を受けた数人のSAP男性が参加した会議に出席しました。 教育的なプレゼンテーションとして編成されたものは、HANA、S / 4HANA、または私が今まで目にした他の製品の最も積極的な取引プラットフォームの1つになりました。



SAPマーケティングからの砂糖油の窒息



ある時点で、これらのシニア男性の1人は、「HANAとS / 4HANA以外のSAPにはイノベーションはない」と述べました。



「SAPにはHANAとS / 4HANA以外にイノベーションはありませんでした。」



これは馬鹿げた声明です。これは、他のSAP製品が開発を行わないことを意味します（SAPでは、「イノベーション」は「開発」のe曲表現です）。 そして、もちろん、HANAおよびS / 4HANA以外のすべてのSAP製品のすべての開発の停止は、これは非常に悪い、非常に非論理的なステップになります。 ただし、HANAを囲む双曲線を過大評価することは困難であり、ある程度までそれを囲みます。 HANAはSAP会議に参加し、SAPコンサルティング会社のマーケティングコミュニケーションを取り、それぞれがHANAからお互いを押し出そうとしました。 私が契約していたあるコンサルティング会社では、売り手は彼について何もチェックせずに、素晴らしいHANAトランザクションパイプラインで受け入れられました。 HANAの専門知識に対する熱意は非常に高かったので、これは、おそらく他のコンサルティング会社が、履歴書にHANAを持つ人を、非常に有利な条件でつかむことができました。 LinkedInでは、ほぼ一晩で、人々はプロジェクトでの経験以上のものにHANAを追加しました。 HANAソリューションアーキテクトとHANAマーケティングスペシャリストがいました。

SAPはどのように評価しますか？

SAPとOracleは、グローバルリセラーとSAPサポートの長期的な拡大に同意します。 20年以上にわたり、SAPとOracleは協力して、サポートされるSAP / Oracle環境、SAPアプリケーション、およびOracleデータベースを顧客に提供しています。 この拡張中、SAPのお客様は引き続きSAPまたはOracleからOracleライセンスを取得してSAPビジネスアプリケーションをサポートできますが、SAPとOracleは引き続きOracle / SAPの組み合わせのサポートを提供します。



2025年から、SAPはすべてのOracleおよびSAP環境（フルタイムおよびリードタイム）の統合サポートを提供し続けます。 SAPは継続的な再販プロセスを継続します。 2023年12月31日までの更新期間の最初の部分のOracleランタイムライセンス。 2024年1月1日から2025年12月31日までの過去2年間の更新期間中、SAPは既存のOracleランタイムクライアントに対してのみOracleライセンスランタイムを販売します。


これらは144ワードです。 契約条件の大幅な変更に関する144の言葉。これらはすべて、SAP HANAポリシーの大幅な変更を表しています。



HANAの未来は？



SAPが抱えているもう1つの問題を次に示します。 HANAによって確立された大規模なデータベースがないと、人々の数とそれらを使用する「コミュニティ」の点で、はるかに人気のあるデータベースに比べて不利になります。 私の意見では、SAPは「あなたがそれをするまでそれを偽造する」（「あなたがそれを作るまでそれを偽造する」）ことを望んで、HANAを誇張したと考えています。 しかし、この成長がなければ、マーケティングの推進力は薄れ、HANAは急成長している製品ではないことが明らかになり、そのため慣性が失速しています。







これは成長しているデータベースのように見えますか？ データベースの人気を推定するDB-Enginesによると、HANAは8か月減少しています。 確かに、SAPは非常に多くのマーケティングドルとエネルギーをHANAの宣伝に注ぎ込んでいますが、実際には他のSAP製品を損ないます。 HANAの未来が本当によく見えないのはそのためです。 HANAはアプリケーションの主流を離れ、実際には合理的な価値を追加し、実際の現実と比較してSAPの誇張から利益を得ています。 IBMとOracleはSAPにますます同意しなくなっていますが、両社はSAPと関係があり、SAPにある程度依存しています。



私の最近の記事でお気づきかもしれませんが、SAPはBW向けに販売されたときにすでにコーヒーの表面からクリームを剥いでいます。 BW後、HANAは単なる何かを装った分析データベースであるため、価値提案ははるかに弱くなります。

もちろん、これはSAPのお客様にとって過激です。 しかし実際には、ERP以外のこれらの新しいシステムの利点に関して、SAPが言ったことは何も起こりません。 全体として、失敗した戦略でのリソースの大規模な不合理な使用でした。 しかし、これを行った顧客は間違いを認めることができません。 興味深いことに、意思決定者が悪い決定に対して責任を負うことはほとんどありません。

誰が間違っていますか？



Hasso Plattner： HassoはHANAの背後にある主要な原動力でした。 HassoはSAPで非常に豊かで強力なので、誰も彼にノーと言うことはできません。 ただし、Hassoは、HANAのようなものがSAPにもたらすメリットを大きく過大評価していました。 Hassoは、SAPに新たな命を吹き込む魔法の薬を見つけたと本当に思っていましたが、安価なデータベースとオープンソースデータベースがマージンとプロプライエタリデータベースプロバイダーの販売を弱体化させようとしていたときに、データベースに多額の投資をしました、これは記事「SQLの衰退時にAlphaがどれほど正確に求めているか？」 SAPは、データベースにアクセスしてそのようなプレミアム製品を発売することにより、明らかに間違ったことに投資しました。 ハッソは、ハッソが予測したように市場が反応しなかっただけでなく、ハッソがハナについて語ったことのほとんどにおいて、それが単に真実ではなかったというだけでなく、ハナで誤解しました。



ビル・マクダーモット：ビル・マクダーモットは、CEOよりもSAPのPR担当者です。 ビル・マクダーモットが話すとき、私は彼が何について話しているのかさえ知らないという印象を決して受けません。 ビルは優れた営業担当者ですが、本質的には、リスナーが獲得したい目標を非常に簡単に達成できるように、物事の説明を提示します。 たとえば、BillはFortune Magazineのインタビュアーに、トランザクションとビッグデータの両方を単一のデータベースで実行できると話しました。 ただし、HANAをHadoopに接続するには、単一のデータベースを使用しませんか？ キャリアアドバイスを求める人のために、あなたは多くの仕事をすることなく年間120百万ドル以上を稼ぐことができます。 あなたがそれを得ることができるならば、これは良い取引です。
