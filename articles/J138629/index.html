構成


デフォルトでは、Node.JSはシングルスレッドで実行されますが、特にマルチコアプロセッサの場合、あまり便利ではなく、最適でもありません。 ほとんどすぐに、モジュールはいくつかのプロセス（Web Workerのさまざまな実装）を便利に起動するように見えました。 標準のNode.JS APIでこれを行うことは難しくありませんでした。 バージョン0.6.0のリリースで、Node.JSは新しいモジュールClusterを導入しました。 複数のNode.JSプロセスを開始するタスクを大幅に簡素化します。 このモジュールのAPIを使用すると、net / httpサーバーが共通のTCPポートを使用するノードプロセスを分岐できます。 親プロセスは子プロセスを管理できます：停止、新しいプロセスの開始、予期しない終了への応答。 子プロセスは、親とメッセージを交換できます。



クラスターを使用すると必要な数のプロセスを開始するのに便利であるにもかかわらず、異なるTCPポートでノードの2つのインスタンスを実行します。 更新中のダウンタイムを回避するためにこれを行います。そうしないと、再起動中（ちなみに数秒しかかかりません）、ユーザーはサイトにアクセスできなくなります。 ノードのインスタンス間で、負荷はHttpUpstreamModule Nginxを使用して分散されます。 再起動中にインスタンスの1つが使用できなくなると、2番目のインスタンスがすべてのリクエストを引き継ぎます。



Nginxは、許可されていないユーザーに対して、サイトのすべてのページが短時間（1分間）キャッシュされるように構成されています。 これにより、Node.JSの負荷を大幅に軽減すると同時に、関連するコンテンツを非常に迅速に表示できます。 許可ユーザーの場合、キャッシュ時間は3秒に設定されます。 これは一般のユーザーには完全に見えませんが、許可Cookieを含む多数のリクエストでサイトをロードしようとする侵入者を防ぐことができます。

つなぐ
github.com/senchalabs/connect

このモジュールはNode.JS httpサーバーのアドオンであり、その機能を大幅に拡張します。 彼はルーティング、Cookieサポート、セッションサポート、リクエスト本文の解析などの機能を追加します。これなしでは、Node.JSでのWebアプリケーションの開発は悪夢になりそうです。 ほとんどの接続機能はプラグインとして実装されます。 また、標準パッケージには含まれていない接続用の同様に便利なプラグインも多数あります。 独自のプラグインを開発して不足している機能を追加することも非常に簡単です。



このモジュールの人気と急速な開発にもかかわらず、NginxがNode.JSからの回答をキャッシュすることを妨げる問題がそこにありました。 デフォルトでは、Nginxのproxy_cacheディレクティブは、次のヘッダーの少なくとも1つが含まれている場合、バックエンドの応答をキャッシュしません。

  • クッキーを設定する
  • 値が「no-cache」、「no-store」、「private」、または「max-age」で、数値またはゼロ以外の値を含むCache-Control。
  • 過去の日付で失効します。
  • X-Accel-Expires：0。
接続セッションでは、すべての応答でSet-Cookieヘッダーが送信されるように実装されました。 これは、ブラウザセッションの寿命よりも長い寿命を持つセッションをサポートするために行われました。 PHPでは、セッション時間を特定の値に設定すると、ユーザーがサイトでアクティブになっている場合でも、この時間後に終了します。 Connectは異なるポリシーを使用します-Cookieはすべてのリクエストで更新され、その有効期間は現在のリクエストからカウントされ始めます。 ユーザーがアクティブな間、セッションは終了しません。 PHPのアプローチは、私にはもっと本当のようです。 セッションは、まだ長期のデータストレージを目的としていません。 コードに適切な変更を加え、プルリクエストを送信しました。 さらに、 短い議論の後（私の英語に反してはいけません）、妥協案が見つかりました-期限切れセッションがインストールされていないため、Cookieは1回だけ送信されるようになりました。 有効期間が厳密に指定されたセッションの場合、この問題はまだ解決されていません。

ああ
github.com/baryshev/aop

このモジュールはまだAOPパターンを完全に実装すると主張していません。 現在は、必要に応じて動作を変更できるアスペクトで関数をラップできる単一のメソッドが含まれています。 この手法は、関数の結果をキャッシュするために非常に便利です。



たとえば、ある種の非同期関数があります。 



var someAsyncFunction = function(num, callback) { var result = num * 2; callback(undefined, result); };




この関数は頻繁に呼び出され、結果をキャッシュする必要があります。 通常、次のようになります。 



 var someAsyncFunction = function(num, callback) { var key = 'someModule' + '_' + 'someAsyncFunction' + '_' + num; cache.get(key, function(error, cachedResult) { if (error || !cachedResult) { var result = num * 2; callback(undefined, result); cache.set(key, result); } else { callback(undefined, cachedResult); } }); };




プロジェクトには、このような機能が多数存在する場合があります。 コードは大きく膨らみ、読みにくくなり、大量のコピー＆ペーストが表示されます。 そして、ここでaop.wrapでも同じことができます。 



 var someAsyncFunction = function(num, callback) { var result = num * 2; callback(undefined, result); }; /** *   -    this   *   - ,      *   - ,        *   -  ,    (    ) */ someAsyncFunction = aop.wrap(someAsyncFunction, someAsyncFunction, aspects.cache, 'someModule', 'someAsyncFunction');




それとは別に、アスペクトライブラリを作成し、そこにキャッシュ関数を定義します。この関数は、すべてをキャッシュする役割を果たします。 



 module.exports.cache = function(method, params, moduleName, functionName) { var that = this; //         var key = moduleName + '_' + functionName + '_' + params[0]; cache.get(key, function(error, cachedResult) { //    callback- (   ) var callback = params[params.length - 1]; if (error || !cachedResult) { //     ,    ,  callback- params[params.length - 1] = function(error, result) { callback(error, result); if (!error) cache.set(key, result); }; method.apply(that, params); } else { callback(undefined, cachedResult); } }); };




必要に応じて、機能面を増やすことができます。 大規模プロジェクトでは、このアプローチによりコードの量が大幅に節約され、すべてのエンドツーエンド機能が1か所にローカライズされます。



将来的には、AOPパターンの残りの機能を実装して、このライブラリを拡張する予定です。





形
github.com/baryshev/form

このモジュールのタスクは、入力データをチェックしてフィルタリングすることです。 ほとんどの場合、これらはフォームですが、外部APIなどからのリクエストに応じて受信したデータでもあります。 このモジュールにはノード検証ライブラリが含まれており、その機能を完全に使用できます。



このモジュールの動作原理は次のとおりです。各フォームは、フィルターが掛けられている一連のフィールド（フィールドの値に影響を与える関数）およびバリデーター（フィールドの値が条件に適合しているかどうかをチェックする関数）によって記述されます。 データを受信すると、それらはプロセスフォームメソッドに渡されます。 コールバックでは、エラーの説明（データがフォームの基準を満たしていない場合）またはフィルター処理されたフィールドセットを含むオブジェクトを取得し、さらに使用する準備ができています。 小さな使用例： 



 var fields = { text: [ form.filter(form.Filter.trim), form.validator(form.Validator.notEmpty, 'Empty text'), form.validator(form.Validator.len, 'Bad text length', 30, 1000) ], name: [ form.filter(form.Filter.trim), form.validator(form.Validator.notEmpty, 'Empty name') ] }; var textForm = form.create(fields); textForm.process({'text' : 'some short text', 'name': 'tester'}, function(error, data) { console.log(error); console.log(data); });




この場合、テキストフィールドのエラー「Bad text length」が表示されます。 送信されるテキストの長さは30文字未満です。



フィルターとバリデーターは順番に実行されるため、行末に多くのスペースを追加しても、エラーが発生します。 チェックする前に、トリムフィルターでスペースが削除されます。



node-validatorページで独自のフィルターとバリデーターの作成方法を読んだり、ソースコードを参照したりすることができます。将来の計画では、このモジュールの移植をブラウザーで使用できるようにし、その機能を十分に文書化します。

ハブ
github.com/baryshev/hub/blob/master/lib/index.js

おそらく、このモジュールはNode.JSの最小モジュールであると主張できます。 それは1行のみで構成されています：module.exports = {}; ただし、それがないと、開発ははるかに複雑になります。 このモジュールは、アプリケーションの実行中にオブジェクトを格納するためのコンテナです。 Node.JS機能を使用します-接続されると、モジュールは一度だけ初期化されます。 require（ 'moduleName'）へのすべての呼び出しは、アプリケーションにいくつあっても、最初の言及で初期化されたモジュールの同じインスタンスへの参照を返します。 実際、アプリケーションの各部分間でリソースを共有するためのグローバルスペースの使用に取って代わります。 このようなニーズは非常に頻繁に発生します。 例：DBMS接続プール、キャッシュ接続プール、ロードされた構成へのリンク、およびロケール。 これらのリソースはアプリケーションの多くの部分で必要であり、それらへのアクセスは簡単でなければなりません。 リソースが初期化されると、ハブオブジェクトのプロパティに割り当てられ、将来的には、ハブを事前に接続することで他のモジュールからアクセスできるようになります。

アプリケーション構造



アプリケーションはフレームワークを使用しないため、Node.JSでアプリケーションを記述する一般的なスタイルと常識を除き、その構造はルールに関連付けられていません。 アプリケーションはMVCモデルを使用します。



server.js起動ファイルには、httpサーバーの起動、接続の構成、構成と言語のロード、MongoDBおよびMemcachedとの接続の確立、コントローラーの接続、ハブ内のモジュール間で共有する必要のあるリソースへのリンクの設定など、アプリケーションの主要なリソースの初期化が含まれています。 ここでは、必要な数のプロセスが分岐されます。 マスタープロセスでは、node-cronが起動され、スケジュールされたタスクが実行されます。子プロセスでは、http-serversが起動されます。



各コントローラーで、connectはURLをハンドラーメソッドに接続します。 各リクエストは、接続の初期化中に作成される一連のメソッドを通過します。 例： 



 var server = connect(); server.listen(port, hub.config.app.host); if (hub.config.app.profiler) server.use(connect.profiler()); server.use(connect.cookieParser()); server.use(connect.bodyParser()); server.use(connect.session({ store: new connectMemcached(hub.config.app.session.memcached), secret: hub.config.app.session.secret, key: hub.config.app.session.cookie_name, cookie: hub.config.app.session.cookie })); server.use(connect.query()); server.use(connect.router(function(router) { hub.router = router; }));




これは、要求を処理して応答を生成するアルゴリズムに新しい動作を非常に簡単に追加できる便利なメカニズムです。



コントローラーは、必要に応じて、MongoDbまたはMemcachedからデータを受け取るモデルのメソッドを呼び出します。 応答のすべてのデータの準備ができると、コントローラーはテンプレートエンジンにコマンドを与えてページを形成し、生成されたhtmlをユーザーに送信します。
