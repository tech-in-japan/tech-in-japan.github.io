「資産」が「電子会話」を整理した方法

当社は、あらゆる意味で常に成長しています。 ある時点で、施設のスタッフだけでなく、会議、会議、ディスカッション、プレゼンテーションの数が大幅に増加したと感じました。 また、会議室は、時間、人などの限られたリソースになりました。







しかし、これは問題の半分にすぎません。 誰もが知っているように、衝突が発生しないように、リソースを適切に管理できる必要があります。 パートナーとの会議中に、隣のユニットの長が「そして、これは忙しい！？」という言葉で会議室に突入する状況は受け入れられません。













2番目のポイントは、会議室を予約するためのツール（OutlookやそのWebインターフェイスなど）が手元にない場合ですが、貴重なリソースを獲得するには「管理」することが非常に必要です。 計画を実施するには、職場に行き、そこから予約をする必要があります。 ここで、次の会議から会議室を離れる時間がないと想像してください。







もちろん、「理由」にはさまざまなオプションがあります。 私たちにとって、これらの2つのポイントは、問題の解決策を見つける方向で研究を始めるのに十分でした。







どうやって自分の決断に至ったのか



最初に、標準の2つのオプションが考慮されました。







  • 準備ができているものを見つけます。
  • 独自に実装します。


市場に出ている製品、その機能、解決する問題を理解するために、既製のソリューションを探し始めました。







実装の可能性があると考えられるいくつかのソリューションを特定しました。





  • スケジュール
  • エボコルームマネージャー ;
  • BookITルームスケジューリングソリューション ;
  • CUE touchONE 。


それぞれを見て、機能と最終コストを分析して、製品を開発することにしました。 そして、機能に満足していなかったからではありません。 コストと特定の要件に合わせてソリューションをカスタマイズできないことに満足できなくなりました。

私たちが何を必要としていたかは明らかでした。目の前には既製の製品がすでにあったので、「創造」を開始することだけが残っていました。







製品によって解決される主なタスク



製品が最初に解決することになっている主なタスクを特定しました。







  • 管理の容易さ;
  • 会議室のステータスの表示。
  • 会議室をすばやく便利に予約する機能。
  • 会議室の予約のスケジュールを表示します。
  • MS Exchangeとの統合。


実装



割り当て、インターフェイスレイアウト、および必要なグラフィックのレンダリングを行った後、開発を開始しました。 最終製品には開発だけでなく、必要なことに注意してください。 それでも、ハードウェアを選択する必要がありました。 当然、最終デバイスとしてタブレットを使用することをすぐに決定しました。 市場で販売されているタブレットの価格と特性を調査した結果、Lenovo TAB2 A10-30 10.1を選択しました。 価格/品質比の面では、彼は私たちを満足させました。 同時に、技術、価格、画像、その他のパラメーターに適した他のデバイスを実際に使用できることを理解する必要があります。 しかし、それだけではありません。 タブレットは会議室の近くで何らかの形で固定する必要があります。 特殊なケースがこれに使用されていることを知っており、インターネットで検索を開始しました。 結局のところ、適切なケースを見つけることは容易ではありません。 市場には多くのオファーがありません。また、入手可能なものからは、外観に私たちに合ったものは多くありません。 しかし、彼らが言うように、「水は石を研ぐ」。 ケースが見つかり、タブレットが購入され、上記の要件をすべて満たす製品が開発されました。







どうした



ソリューションアーキテクチャについて少し



  • クライアント部分は、AngularJSを使用して実装されたWebアプリケーションです。 デバイスにネイティブクライアントを実装しませんでしたが、そのようなオプションを検討しました。製品の深刻な開発が継続している場合は、そのような開発の可能性が高くなります。
  • 現時点では、補助製品のキオスクモードブラウザーを使用しています。これにより、実行中のブラウザーモードから他のアプリケーションへの移行をブロックし、任意のアドレスに移動する機能を「ロックアップ」できます。 この製品の機能については、 こちらをご覧ください 。
  • バックエンドは、ASP.NET Coreフレームワークを使用してC＃で実装されます。 この場合、バックエンドをデプロイできるプラットフォームに縛られていません。 現時点では、Windows Server上で動作しますが、Linux / Unixマシンでの展開を妨げるものはありません。
  • DBMSとしてMS SQLを使用します。 同時に、開発ではORM Entity Frameworkを使用します。これにより、ほぼすべてのsql指向のDBMSにシームレスに切り替えることができます。

    現在使用しているデータベース自体は、非常に限られた形式で使用しています。 その中に、管理者アカウントと会議室のパラメーターを保存します。 スケジュールに関する主要なデータソースである会議室のリストは、MS Exchange Serverです。 将来、Exchangeとバックエンドの間に中間層を実装することにより、データベースの役割が拡大します。





  • アプリケーションサーバーがAPIを介してやり取りするMS Exchange Server。


バックエンドとExchangeの間に中間層を実装する必要があるのはなぜですか？ 実際、この方法で解決したい主なタスクは次のとおりです。

必ずしもすべての人がITインフラストラクチャにExchangeサーバーを持っているわけではないと考えています。 したがって、少なくとも情報のソースから抽象化する必要があります（インターフェイスのレベルに進みます）。







そして、私たちが遭遇した技術的な問題は、控えめに言っても、リクエストに対する即座のExchange反応ではありません。 リクエストキューと中間キャッシングレイヤーが必要です。これらのレイヤーに対して、バックエンド呼び出しが直接行われ、Exchangeと直接対話することはありません。







インターフェース



管理者



エンドデバイス（タブレット）での作業は、特定の会議室へのバインドから始まります。 これを行うために、単純な「管理パネル」が実装され、会社で利用可能な会議室のリストが表示されます。













このリスト内のチャットルームの1つを選択し、管理パスワードを入力することにより、バインドが実行されます。 これで設定が完了し、タブレットを壁に取り付けることができます。







追加機能には、会議室のパラメーターを設定する機能があります。







  • 快適な宿泊のために会議室が収容する人数。
  • 会議機器の可用性;
  • コンピューターの存在;
  • オフィスの存在;
  • ボードの存在;
  • テレビの存在。










このリストは拡張可能であり、必要に応じて、必要なアイテムを簡単に追加できます。 現在の段階では、パラメーターは参考になりますが、将来的には、指定された特性に応じて会議室の検索に使用されます。







メイン画面



メイン画面には、会議室の名前、雇用状況の表示、現在の日時が表示されます。 メイン画面から、会議室のスケジュールと予約を表示したり、会議室の詳細情報を表示したりできます。







これまでに、会議室の3つの重要な職業ステータスを特定しました。







  1. 会議室は無料です。
  2. 会議室は無料で15以下です。
  3. 会議室は忙しいです。


会議室は無料です 会議室は間もなく混雑します。 忙しい交渉














1番目と3番目のステータスの目的が直感的に明確でなければならない場合、2番目のステータスには少しコメントが必要です。 従業員が今すぐ会議室を使用できるようにするためにこのステータスを実装しましたが、15分以内であることを警告します（たとえば、「バックグラウンド」がないように重要な電話をかける必要があります）。







ご予約



予約プロセスについて詳しく説明します。 私たちの問題は会議への招待ではなく、会議室のタイムリーな予約だったことを思い出させてください。













このプロセスを可能な限り高速化および簡素化するために、会議の主催者と参加者を選択する必要があるだけでなく、トピックを導入することでインターフェイスが過負荷にならないようにしました（必要な場合、これはまったく難しくありませんが）。 ほとんどの場合、会議室の可用性に関する問題は、突然の必要性の日に感じられます。 したがって、簡素化の道をたどりました。会議の開始時間と期間のみを選択できます。 日付は常に現在と見なされます。













同時に、会議の開始時刻の選択を開始すると、会議の期間を選択する要素がアクティブまたは非アクティブになります。 ロジックは単純です-開始の間隔と期間がビジーでない場合、指定された期間の要素がアクティブになり、会議が指定された範囲に該当する場合、要素は非アクティブになります。













会議の開始とその期間を選択したら、予約できます。 これでプロセスは終了です。予約スケジュールを表示すると、実行したアクションの結果をすぐに確認できます。







スケジュール



予約スケジュール画面では、雇用範囲と「会議室」の自由度がリストとして表示されます。













ご覧のとおり、サービスを介して予約すると、会議のテーマは「エクスプレス予約」に設定されます。 未使用の範囲をクリックすると、予約モードに入ります。そのプロセスは上記で説明されています。







計画について



現時点では、基本的なニーズは閉じています。 しかし同時に、私たちは栄冠にとどまり、サービスを開発する計画を持ちたくありません。 また、ユーザー機能の拡張だけではありません。 これは、実装を計画しているものの完全なリストではありません。







  • 中間ソフトウェア「レイヤー」。その助けを借りて、会議室、スケジュールなどに関する情報のソースから抽象化することが可能になります。 これにより、必要に応じて、外部サービスにコネクタを実装することにより、MS Exchange製品に縛られることなくソリューションを実装できます。
  • 特定の期間の特定のパラメーターの会議室の選択。
  • 会議室から会議をキャンセルまたは完了する可能性（会議がより早く終了し、他の人にこれを知らせる必要がある場合があります）。 従業員を「冗談」に誘惑しないために、この機能を外部インターフェイスに導入しませんでした。
  • 会議を迅速に延長するためのメカニズム。
  • 会議、参加者、開催者の日付を選択する機能、および予約インターフェイスでトピックを入力する機能。
  • コードのアーキテクチャをリファクタリングし、さらには変更します。 現在、コードアーキテクチャに埋め込まれているボトルネックを認識しており、それらの修正に取り組んでいます。


私たちは、人生をより安全で便利なものにするよう常に努めてきました。 同時に、私たちはオープンソースでレイアウトされた製品を愛し、感謝するだけでなく、この方向への一歩を踏み出します。 この製品を実装したことで、利便性と快適性の向上に向けて新たな一歩を踏み出したため、ベストプラクティスとアイデアをさらに共有して、製品を一般に公開したいと考えています。 これを行うには、機能開発とおそらく使いやすさの提案を含むフィードバックを受け取り、製品が潜在的なユーザーに直面するより多くの問題を解決できるようにします。
