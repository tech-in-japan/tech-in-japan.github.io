RubyのパーサーOOXML（docx、xlsx、pptx）：エラーと発見

オープンソースのRubyでOOXML形式のパーサーをレイアウトしました。 GitHubおよびRubyGems.orgで入手でき、AGPLv3ライセンスの下で無料で配布されています。 すべてがファッショナブルなRuby開発者のようなものです。









サードパーティのパーサーを使用しなかった理由





パーサーがRubyの最初のOOXMLパーサーではないことは秘密ではありません。 サードパーティ製品を使用することもできますが、使用しないことにしました。 私たちが見つけた解決策には、いくつかの問題があります。



a）開発者によって長い間放棄されてきた。

b）基本的な機能のみをサポートします。

c）通常、3つの個別のライブラリとして配布されます。 多くの場合、docxパーサーとxlsxパーサーは異なる人々によって作成されたため、それらのインターフェースは劇的に異なる可能性があります。 同意します、これは不便です。

私たちは自分自身と私たちのタスク（ドキュメントエディタのテスト）のために書きましたが、その後、おそらく他のRuby開発者を助けることができることに気付きました。



a）積極的に開発中です。

b）エディターのすべての機能をサポートしており、それはたくさんあります。 ここで読むことができます。

c）docx、xlsx、pptxで動作するため、OOXMLパーサーを呼び出しました。



機能b）-機能について説明します。 標準の可能な機能をすべて実装していますか？ いや。 ECMA-376標準は4ボリュームで、合計9000ページを超えています（いいえ）。 実際、約7000です。 息を吐くことができます。

なぜパーサーが必要なのですか

自動テストの最初から、機能テストという単一のコンセプトを採用しました。



最も簡単なテストを実施してください。



1.新しいドキュメントを作成します。

2.テキストを印刷し、その太字プロパティを設定します。

3.太字が設定されていることを確認します。



ONLYOFFICEエディターはCanvasで記述されています。つまり、文書内のテキストは画像です。 画像内のフォントの太さを確認することは非常に困難です。 ただし、任意のフォントに太字を適用できます！







一部のフォント（Arial Blackなど）では、ボールドが視覚的にまったく表示されない場合があります。 画像をimagemagickと比較することは最良の選択肢ではないことに同意します。



したがって、テストの検証手順は別の段落で強調表示されました。つまり、



4.結果のファイルをdocx形式でダウンロードし、テキストに太字パラメーターが設定されていることを確認します。



そのようなパラメーターは数百あります。 同時に、既存のソリューションはどれも、テキスト、テーブル、およびその他のいくつかの最も単純なグラブ以外のものをサポートしていませんでした。 そこで、独自のライブラリを作成することにしました。



これらのフォーマットをすべて編集用に開くことができるドキュメントエディターを開発しています。 エディターから既製のソリューションを使用して、それを通してテストを検証してみませんか？

パーサーの仕組み





docxファイルをアーカイブしようとしたことがある場合、圧縮率が非常に小さいことに気づいたかもしれません。 なぜそう 簡単です。ooxmlファイルは、xmlファイルのアーカイブセットにすぎません。 それらの構造は非常に簡単です。







たとえば、ONLYOFFICEエディターで簡単なグリーティングファイルを作成し、docxでダウンロードします。 次に、それをzipファイルとして解凍し、このドキュメントの重要な部分が保存されている場所を確認します。



次の構造が表示されます。 



#tree ├── [Content_Types].xml ├── docProps │ ├── app.xml │ └── core.xml ├── _rels └── word ├── document.xml ├── fontTable.xml ├── _rels │ └── document.xml.rels ├── settings.xml ├── styles.xml ├── theme │ ├── _rels │ │ └── theme1.xml.rels │ └── theme1.xml └── webSettings.xml






私たちは内部を掘り下げ始めます。 順番に。



[Content_Types] .xml-ドキュメント内のMIMEタイプのリスト。 寒いです。



app.xml-ドキュメントのメタデータ、作成者アプリケーション、統計。 すでに暖かく、情報は興味深く有用です。



core.xml-最新の変更に関するメタデータ。



document.xml-ああ、それはビンゴです。 ドキュメントのコンテンツはこのファイルに隠されています。後で検討します。



fontTable.xml-ドキュメント内のフォントの表。 便利です。



document.xml.rels-アーカイブ内のすべてのファイルのリスト。このリストは、写真やグラフを含む複雑なドキュメントの場合に非常に役立ちます。



settings.xml-名前から、デフォルトのズーム、数字の区切りなど、さまざまなドキュメントパラメータがそこに保存されていることが明らかです。



styles.xml、theme1.xml、およびtheme1.xml.relsは、ドキュメントのスタイルとテーマのパラメーターを含む非常にかさばる非常に詳細なファイルです。 これらのドキュメントを理解する機能は、製品の重要な機能の1つです。



webSettings.xml-ドキュメントのWebバージョンに関する設定。 docxの最も一般的な機能ではありません。省略してください。



そのため、単純なドキュメントでは、 word / document.xmlが興味深いことがわかりました。







シンプルなxml。 幸いなことに、Rubyでのxmlの解析に問題はありません。 のこぎりを取り、DOMツリーを取得します。 さて、それは技術の問題であり、標準を読んでください（怠notではない場合、ドキュメントは非常に大きいです）、または古き良きリバースエンジニアリングで必要なパラメーターがドキュメントのどこに隠れているかを理解するだけです。

RuboCopを使用する

RuboCopはRuby用の静的コードアナライザーであり、それが気に入っています。 本当に、本当に。 そして、私たちは常に彼の意見に耳を傾けます。 コードを適切な状態に保ち、愚かな間違いを避け、次のコミット後にコードが汚れたり悪化したりしないように厳密に監視するのに役立ちます（ overcommitによる統合のおかげ）。



彼の仕事は次のようになります。苦労して一日を過ごした後、Rubyの変数は通常大文字で始まることを忘れて、次の形式のコードをコミットしようとします。



-path_to_zip_file = copy_file_and_rename_to_zip（path_to_file）

+ ZIP_file = copy_file_and_rename_to_zip（path_to_file）


この場合、エラーが発生します。

RuboCopで分析........................................ [RuboCop]が失敗しました

変更された行のエラー：

ooxml_parser / lib / ooxml_parser / common_parser / parser.rb：8：7：E：動的定数の割り当て


追加の操作なしでこのコードをコミットすることはできません（ `SKIP = RuboCop git commit -av`）。 これは愚か者に対する大きな防御です。
