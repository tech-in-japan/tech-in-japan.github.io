比較的最近、使用されているバージョン管理システムに関する調査が公開されました。 予想通り、Gitは大きなマージンを獲得し、Fossilはリストに含まれていませんでしたが、コメントでは数回しかフラッシュしませんでした。 Habrでの検索では、化石についてはほとんど何も書かれていなかったことが示されました。 したがって、私はこの記事を公開することにしました-特に、化石に関するロシア語の情報は非常に乏しく単調であるためです。



オープンソース開発への長年の参加を通じて、CVS、Subversion、およびGitを使用する機会がありましたが、個人および仕事関連のプロジェクトでは、ランダムリンク経由でFossilに会うまでバージョン管理システムを使用しませんでした。 ただダウンロードして適切な場所にコピーする必要があるという事実にすぐに夢中になります-唯一の実行可能ファイル。 簡単な場合は、試してみませんか？ また、リポジトリ全体が1つのファイル（SQLiteデータベース）であり、コピーして自宅に持ち帰ったり、職場の別のコンピューターにインストールしたりすることもできました。 私は一般的にミニマリストのアプローチが好きです-たぶん3-4人のチームにSubversionのようなものを置くために手が上がっていなかったのかもしれません。



判明したように、Fossilは完全に「大人」のシステムであり、その有名な競合他社に比べて基本的な機能の点で劣っていません。 確かなボーナスとして、Webインターフェイス、バグ追跡システム、Wikiがあり、エラー通知とWikiページはプロジェクトファイルもバージョン管理システムなどの管理下にある同じリポジトリにあります。 e。 それらの変更も追跡されます。 操作の過程で、ドキュメントが研究されたとき、選択の正確性に対する信頼を追加した、小さいながらも楽しい「グッズ」が明らかになりました。 私はすぐに強調したい：私にとっての正しさ。 誰もが異なる要件、異なる習慣を持ち、他の誰かが他の何かに適しています。 私にとって、Fossilは控えめに言っても完全に適切なソリューションです。



化石のインストール

そのため、 http：//www.fossil-scm.org/download.htmlにアクセスして、オペレーティングシステムに対応するパッケージをダウンロードします。 実行可能ファイルを実行するのに便利な場所、できればSET PATHでアクセスできるディレクトリにある実行可能ファイルを解凍します。 すべて、システムは作動する準備ができています。



リポジトリの作成と設定

さらに、明確にするために、Windowsがあるという前提から進めます。 また、明確にするために、すべてのリポジトリが個別のディレクトリc：\ fossilに格納されることを前提としていますが、もちろん、個別のスペースを割り当てることはできませんが、制御を与えるファイルのすぐ隣に配置します化石。 さらに、C：\ projects \ castle \ source \にソースがあり、Fossilの制御下に置きたいCastleプロジェクトがあるとします-それらを作業ファイルと呼び、ソースディレクトリが作業ディレクトリになります。 最後に、これらのソース用に空のcastle.fossilリポジトリを作成します。 



fossil new c:\fossil\castle.fossil


Fossilはcastle.fossilファイルを作成し、管理者の名前（通常、OSにログインしているユーザー名）とパスワードを教えてくれます。 化石は一般に非常に「おしゃべり」であり、彼のメッセージを読む必要があります;重要な情報がそこに含まれている可能性があります。 すでに理解しているように、これはすべてコンソールで行われるため、Fossilを使用するには、コマンドラインスキルを持ち、英語に精通することをお勧めします。

次のステップは、c：\ projects \ castleディレクトリに移動し、作成されたリポジトリを開くことです。 



 c: cd \projects\castle fossil open c:\fossil\castle.fossil


ほとんどのFossil操作はオープンリポジトリで実行されるため、作成後すぐに実行する必要があります。 しかし、それを閉じること（ 化石close ）はほとんど意味がありません-プロジェクトの完了後を除いて。 fossil openコマンドは、現在のディレクトリにデータベースサービスファイルを作成します。その名前は、FossilのバージョンとOSに依存する場合があります-Windowsでは_FOSSIL_です。 このファイルは、開いているリポジトリに関する情報を保存し、ファイルの変更を追跡します。 リポジトリを開く前に、ワーカーの親ディレクトリに移動したことに注意してください。 これは重要です。リポジトリは、作業ディレクトリまたは親のいずれか（ツリーの任意のレベル）で開いている必要があります。開いていない場合、Fossilは作業ディレクトリからリポジトリへのファイルの追加を拒否します。

実際、 fossil openはサービスファイルを作成するだけでなく、作業ディレクトリとリポジトリ内のファイル間の対応をチェックし、リポジトリのファイルが見つからないか、作業ディレクトリと異なる場合、作業ディレクトリのファイルをリポジトリから上書きします。 （すべて/はい/いいえを警告した後）。 しかし、これまでのリポジトリは空なので、このようなことは何も起こりません。



ファイルをリポジトリに追加する前に、何かを事前に構成することをお勧めします。 これを行うには2つの方法があります。

1） fossil settingsコマンドを使用してコンソールから： 



 fossil settings crnl-glob '*' fossil settings encoding-glob '*'


crnl-globを '*'（any）に設定すると、行末の文字のチェック、 encoding-glob-ファイルのエンコーディングのチェックが無効になります。 これが行われず、ファイルがWindowsの標準CRNLシーケンスまたはUTF-8以外のエンコードを使用する場合、Fossilは警告を発行し、 コミット操作中に確認を要求します。 さらに、作業ディレクトリに存在する可能性があるが、リポジトリに含めるべきではないファイルマスクのリストを指定することをお勧めします。たとえば、オブジェクトモジュール、実行可能ファイルなどです。 



 fossil settings ignore-glob '*.o,*.obj,*.exe,*.bak,*.log'


fossil設定 コマンドに--globalオプションを追加できます。この場合、このパラメーターはこのコンピューター上のすべてのリポジトリーに対してグローバルに設定されます。 ところで、設定値が指定されていない場合、 fossil settingsコマンドは応答としてパラメーターの現在の値を返します。パラメーター名を指定しない場合、すべてのプリセットパラメーターとその値のリストを取得します。



2）パラメーターを設定する2番目の方法-グラフィカルインターフェイスを使用： 



 fossil ui


このコマンドの結果、Fossil Webサーバー、この場合はローカルサーバー（デフォルトのポートは8080、別のオプション--portを指定できます）が起動され、ブラウザーはURL 127.0.0.1:8080を使用します。 パラメータを設定するには、管理メニュー項目を開き、設定を変更して、crnl-glob、encoding-glob、ignore-globを変更します。 同時に、管理者/構成にアクセスできます-プロジェクトの名前と管理者/ユーザーを設定します-パスワードを変更します。



最後に、addコマンドでファイルを追加し、 commitコマンドでリポジトリに送信します。 



 fossil add source fossil commit -m "Initial commit"


追加では、個々のファイルとディレクトリの両方を追加でき、ファイルマスクを使用できます。 チーム 

 fossil add .
現在のディレクトリとサブディレクトリからすべてのファイルを追加します。



リポジトリが開いてファイルが追加された、つまりFossilの制御下に置かれたので、実際にプロジェクトの作業を続けることができます。 プログラム、記事、書籍を作成し、重要な変更をリポジトリに送信するたびに、意味のあるコメントを付けてcommitを作成します。 



 fossil commit -m "Another brick in the wall"


ファイルを削除/移動/追加するためのいくつかのコマンドを次に示します。 



 fossil rm source\wall\brick.c
リポジトリからbrick.cを削除します。 この場合、このファイルの以前のバージョンはすべてリポジトリに残り、単に削除済みとしてマークされ、リポジトリでのそれ以降の変更は修正されません。 このファイルは作業ディレクトリから削除されません-必要であれば、私たち自身でそれをしなければなりません。 



 fossil mv source\bridge\item* source\wall\
アイテム*マスクを使用して、あるディレクトリから別のディレクトリにファイルを移動します。 Fossilは作業ディレクトリ自体のファイルを移動しないことに注意してください。これは手動で行う必要があります。 



 fossil addremove
リポジトリにない作業ディレクトリのリポジトリファイル（つまり、 fossil extrasコマンドが提供するリスト）にリポジトリファイルを追加し、作業ディレクトリから以前に削除したファイルをリポジトリから削除します。



そして、リポジトリと個々のファイルに関する情報を提供するいくつかのコマンド： 



 fossil ls source\wall
リポジトリにあるsource \ wallディレクトリのファイルをリストします。 -vパラメーターはファイルステータス（EDITED、UNCHANGED）の列を追加し、 -ageパラメーターは各ファイルの最終コミット時間を追加します。 



 fossil status
リポジトリの現在の状態を確認します。 



 fossil changes
最後のコミット以降に変更されたファイルをリストします。 



 fossil extras
「追加」ファイルをリストします-作業ディレクトリには存在するが、リポジトリには含まれていないファイル。



ファイルバージョンを操作するためのコマンド-変更の表示、元に戻す 



 fossil timeline fossil timeline after 2014-09-01 fossil timeline before 2014-07-15
リポジトリの変更履歴を表示します。- vパラメーターは、各変更の影響を受けるファイルのリストを出力に追加します。 -tを使用すると、出力する変更の種類を指定できます。ファイル内の-t ci 、イベント内の-te 、チケット内の-tt 、Wiki内の-tw 。 



 fossil finfo source\wall\brick.c
デフォルトでは、brick.cファイルの変更履歴を表示します。 -sパラメーターが指定されている場合、ファイルに関する簡単な情報、 -pがファイルの内容を表示する場合、および-pに加えてバージョンID（ -r VERSION ）も指定されている場合は、指定されたバージョンのファイルが表示されます。



ファイルバージョン識別子は、詳細に説明する必要がある重要な概念であり、ファイルの特定のリビジョンまたはリポジトリ全体で何かを行う必要がある多くのコマンドで使用されます。 バージョンを識別するための次の方法があります。

  • SHA-1ハッシュ
  • ラベル名（タグ）
  • タイムスタンプ-作成日時
  • 識別名：tip、current、next、previous、またはprev


Fossilリポジトリの各要素（ファイルバージョン、チケット、Wikiページなど）には、一意の識別子、SHA-1を使用して計算された40桁のハッシュがあります。 コマンドに40文字すべてを入力するのはあまり便利ではないため、最初の数文字（少なくとも4文字）に制限できます。 そのため、ファイルのバージョンを識別する最初の方法は、ハッシュの初期フラグメントです。 別の方法は、YYYY-MM-DD、YYYY-MM-DD HH：MM、YYYY-MM-DD HH：MM：SSのいずれかの形式で記録されたファイル作成日時です。 詳細はこちらをご覧ください。 



 fossil diff source\wall\brick.c fossil diff --from VERSION1 --to VERSION2 source\wall\brick.c fossil diff --from previous --to current source\wall\brick.c
指定されたファイルの異なるバージョン間の差（diff）をコンソールに表示します。 ここで、VERSION1とVERSION2は、上記の説明に従って、 fossil finfoコマンドに対するバージョン識別子です。 指定しない場合、リポジトリの最新バージョンと作業ディレクトリ内のファイルの違いが表示されます。 Tcl / Tkがインストールされている場合（通常はLinuxシステムにもインストールされますが、Windows用のポートもあります）、- tkオプションを使用できます-この場合、Tcl / Tkに基づく組み込みのグラフィカルツールを使用して違いを示します。 テキストファイルの違いをグラフィカルに表示するためのGUIプログラムがある場合（Windowsではexamdiff.exeを使用）、Webインターフェイスまたはfossil settings diff-commandコマンドを使用して、Fossil diff-commandパラメーターの値として事前に設定できます -その後このプログラムはfossil diffが実行されるときに呼び出されます。 



 fossil gdiff
fossil diffと同じですが、違いを表示するためだけに、Fossil gdiff-commandパラメーターの値として登録されているプログラムが使用されます。 



 fossil tag add TAGNAME VERSION fossil tag cancel TAGNAME VERSION
指定されたバージョンのリポジトリのタグを追加 （ add ）または削除（ cancel ）します。 このラベルは、バージョン識別子のエイリアスと見なすことができ、識別子の代わりに適切なコマンドで、できればtag：プレフィックスとともに使用できます。 通常、ラベルはプロジェクトの最も重要な段階的なバージョンに配置されます。 



 fossil revert fossil revert source\wall\brick.c fossil revert -r VERSION source\wall\brick.c
指定されたバージョンのファイル（またはファイル名が指定されていない場合はすべてのファイル）を作業ディレクトリに戻します。 バージョン（ -r VERSION ）が指定されていない場合、最後のバージョン（最後のコミット中にリポジトリーに入ったバージョン ）が取得されます。 バージョン識別子は、上記の規則に従って（ fossil finfoコマンドに対して ）指定されます。 



 fossil undo
revert 、 merge、またはupdateコマンドによって行われた作業ディレクトリの変更をキャンセルします（最後の2つについては、後の部分で検討します）。



さて、もちろん、助けてください-それなしでどのようにできますか： 

 fossil help fossil help add
引数なしのヘルプは、コマンドのリストを表示し、コマンド名を引数として（たとえば、例のようにadd ）、このコマンドの説明を表示します。

list-リポジトリのリスト、 変更 -すべてのリポジトリで変更されたファイルのリストなどを表示します。



Webインターフェイス、チケット、Wiki

リポジトリの設定について話したときに、 化石UIチームによって起動されたFossil Webインターフェースについてはすでに言及しました。 次のようになります。









設定の表示と変更（管理）に加えて、ここではプロジェクト履歴（タイムライン）、ファイルのリスト（ファイル）、各ファイルを個別に表示できます。プロジェクトブランチの構造（ブランチ）、タグ（タグ）、バグ追跡システム（チケット）およびWiki-ドキュメント。



履歴（タイムライン）を見ると、識別子（角かっこで囲まれたハッシュコード）をクリックして任意のバージョンを選択でき、その時点でのファイルのリスト、変更されたファイルのリスト、各ファイルの以前のバージョンとの違い（[ diff]）、このバージョンのzipまたはtarアーカイブをダウンロードし、履歴内のプレゼンテーションを編集できます。 色合いなど



チケットとWikiの使用について簡単に説明したいと思います。 チケット-この場合、この単語はおそらく「カード」として正しく翻訳されます-バグ追跡システム、バグ追跡を実装する方法ですが、これらのカードは一般的に言えば、エラーメッセージだけでなく、プロジェクトの作業計画にも使用できます。 そこで、チケットを選択し、次に-新しいチケットとカードに記入します-簡単な概要、タイプ、このエントリが属するバージョン番号、重大度レベル、電子メール、およびwikiマークアップを使用できる詳細な説明。これは、任意のオブジェクトを参照できることを意味します他のカード、Wikiページ、ファイルバージョンなどの識別子によるリポジトリ内。 ちなみに、リンクとして識別子を使用するこの可能性は、リポジトリ内の情報の一貫性を高め、プロジェクトのすべての部分の最大の統合に貢献するため、非常に貴重に思えます。 入力すると、チケットは[すべてのチケット]リストとタイムラインの両方に表示されます。 開いて、追加のコメントを追加し、解決策を適用できます（修正、拒否、Unable_To_Reproduce、Works_As_Designedなど）。 チケットが閉じている場合、対応する修正をコミットするとき、このカードの識別子を角括弧で示すことをお勧めします。対応する行のバージョン履歴には、チケットへのリンクがあります。



Wikiの使用を開始する前に、プロジェクトの名前を指定することをお勧めします（まだ行っていない場合）：管理/構成-[プロジェクト名]フィールドに入力し、[変更を適用]ボタンをクリックします。 Wikiを選択し、Castleプロジェクトのホームページ（Castleプロジェクトと呼びます）をクリックして、このページを編集します。 Formatting Rulesで使用されているwikiマークアップのルールを確認します。 [変更のプレビュー]を使用して、[これらの変更を適用]をクリックして保存します。 すべてが正常な場合、作成されたテキストがメニューバーの下のメインページ（ホーム）に表示されます。 Wikiページへの変更はバージョン管理システムによってキャプチャされ、タイムラインに表示されることを思い出させてください。

通常の名前付きFossilに加えて、時間関連のWikiページを作成できます。これらのページはイベントと呼ばれ、タイムラインに表示されます。 このようなイベントを作成するには、Wikiを選択してから、新しいイベントを作成します。 イベントをアタッチする時間、タイムラインコメント（タイムラインの行）を入力し、色を選択してページコンテンツ（実際にはページのコンテンツ）を編集します。 Fossilドキュメントでは、イベントを次のように使用することを推奨しています。

  • プロジェクトのマイルストーン-メジャーリリースなど
  • プロジェクトの現在のステータスを説明する開発者ブログエントリ、さらなる開発のロードマップ
  • プロジェクトのマイルストーン
  • プロジェクト関連のニュース
  • お知らせ。
