真実は何ですか？



真実-検索エンジン結果の最初のページ。 同社はすべてのマーケティング活動を推進し、多くの労働者に技術を「販売」しようとすることができますが、最初の10個のリンク（+ Wikipedia）には特定のソリューションを評価するメカニズムがあります。 しかし、決定について話すのは時期尚早ですが、印象の前にすべきです。 フロアから選択されたリンクで武装して、「Windows 8インターフェイス設計」について話しましょう。



ねえ、なぜそんなに一方的なの？ なぜインターフェイスだけなのですか？ 適用される開発は、主にコンポーネントプログラミングに基づいています。 大きな誤解は、プログラミング言語を使用していることです。 もちろん違います。 システム構築の事前定義されたパターンに関して、さまざまな程度の可変性を備えた必要な機能の手動生成にコンポーネント言語を使用します。 ユーザーは、「Petrovich、親愛なる、ボタンを追加してください。クリックすると、請負業者が良心を失った契約が表示されるウィンドウが表示されるはずです...」などのインターフェイス機能の観点から推論に慣れています。



Microsoft Corporationは長い間（非常に長い間）、開発者を注意深く開発したインターフェイスの安定性に夢中にさせていました（デザイナーは手を離したようです）。 しかし、タブレットの大規模な出現、切り株の動き、そして世界には（ほぼ）新しいWindowsがありました。 審美的に休止状態のDelphi開発者の群衆は、さまざまな度合いでひらめきました。 Windows 8での開発にDelphiを使用することはできますか？ 質問は純粋に修辞的なように思えますが、いくつかの明確化が必要です。

はい、habrahabr.ruに登録することで、 この研究を参照することができます。 一方で、結果は予測可能です。 他方-役に立たない。 いいえ、自尊心を感じるためには、世論に対するあなたの0.02％の貢献を見ることはクールです。 しかし、そのような数値の有用性は、ある程度の精度で0です。 まあ、新しいWindows 8インターフェイスは、開発者の好みを対象としておらず、あなたではなく、私の愛する人を対象としています。 しかし、私はまだそのような死んだ数字を、新しいメタファーとプログラミングする人の非互換性ではなく、説得する有名な企業の恥ずかしがり屋の方法で消し去ります。

Delphiはそれと何の関係がありますか？



XE3のリリース以降、様式化されたアプリケーションを作成できます。 WinRTアプリケーションではない実行可能モジュールは、Windows 8のスタイルをエミュレートします。もちろん、新しい隠phorのために、Microsoftは新しいOSを作成し、Embarcaderoは古典的なアプリケーションの新しいスタイルであることを笑い飛ばすことができます。 しかし、Windows 8の普及のこの段階で、エンバカデロ社がDelphiプログラマーの観点から最も効果的な決定を下したという説は、より正確と言えます。 Windows 8に非WinRTアプリケーション（従来のアプリケーション）を迅速に導入するためのMetropolis専門技術が利用可能になりました。



これは、FireMonkeyとVCLのスタイリングメカニズムに基づいています。 Windows 8のスタイルでアプリケーションを作成することが優先順位ではない場合でも、そのような機会はWindows 8を学習する確実な方法です。インストールしてクリックする、同様のインターフェイスを備えた電話を取得する、またはWindows 8向けのエンタープライズビジネスラインアプリケーションの作成を訓練することができます後者の選択肢は、プログラマーの精神、つまり開発を通じた積極的な学習に沿ったものです。



仮想マシンでは、Windows 8をインストールするときに「ドライブA」で問題が発生しましたが、これが何を意味するのかを覚えておく必要がありました。



新しいAPIを使用せずにスタイリングについて説明しているため、Metropolisを使用して作成されたアプリケーションは（Windows 8に加えて）Windows 7、Windows Vista（これとは何ですか）、Windows XPで実行されます。 しかし、Windows 8でMacOSアプリケーションを様式化することは不可能です。 これらのプラットフォームの支持者の両方のグループは、Mac上のWindows 8を熟考することを特に喜んでいません。
