Jelasticクラスタアーキテクチャ：インフラストラクチャとしての高レベルプラットフォームの概要

以前の出版物で、私たちの製品の新しいポジショニングについて、IaaSの柔軟性とPaaSの利便性を組み合わせたより包括的なPlatform-as-Infrastructureソリューションへの変換について話しました。 この記事では、Jelasticクラスターとその主要コンポーネントの構造について説明します。



Jelasticの主なタスクは、複雑な技術的ソリューションを簡素化し、管理者と開発者の日常業務を自動化することです。 たとえば、今日では、Java、PHP、およびRubyで記述された複雑なアプリケーションを簡単にデプロイできます。 多言語のサポートは、もともとJelasticアーキテクチャに組み込まれていました。 今後の計画には、Node.js、Python、および.Netのサポートも含まれます。



Jelasticプラットフォームのアーキテクチャ機能と、開発者とクラスタ管理者に開かれる可能性を詳しく見てみましょう。



一般的な情報



インストールJelastic Platform-as-Infrastructureは、単一システムとして相互作用するサーバーとサービスのグループで構成される分離クラスターであり、本番環境でアプリケーションを簡単にデプロイ、テスト、保守、およびスケーリングする機能を提供します。



Jelasticクラスタは、ユーザー環境専用の仮想分離コンテナをホストする物理サーバーと、内部システムサービス用の管理サーバーで構成されます。









管理サーバー



管理サーバー -リソースの管理、リクエストの処理、ユーザーアクティビティの分析、およびJelasticクラスターのユニバーサルサポートのための一連の内部コンポーネントが含まれています。 通常、別の専用サーバーを使用することをお勧めします。これにより、より高いレベルの安定性とアップタイムが提供されるためです。 管理サーバーのコンポーネントは次のことを担当します。

  • 必要なリソースの割り当て（プロビジョニング）
  • テンプレートのカスタマイズ（クラスタリング、負荷分散など）
  • アプリケーションライフサイクル管理
  • アプリケーションの展開
  • アプリケーションのスケーリング
  • 共通のリゾルバーを介したユーザーリクエストの処理
  • ログと統計
  • 課金
  • ビジネスインテリジェンスツール
  • 健全性の監視と検証


上記の各プロセスは、個別のサブシステムによって処理されます。サブシステムの詳細については、以降の出版物で説明します。



インフラストラクチャ層



Jelasticは基本的なリソースについてはあまり気にせず、 ベアメタルとOpenStack、AWS Amazon、Azure、RackspaceなどのIaaSソリューションの両方にインストールできます。



クラスターが完全に機能するために必要な物理サーバーの数は、予想される負荷によって異なり、インストール前にネゴシエートされます。



物理サーバーまたは大規模な仮想マシン（KVM、ESXi、Hyper-Vなどを使用して仮想化）は、小さな分離された仮想コンテナーにさらに分割されます 。 特定のアプリケーションに必要なすべてのスタックを備えたこのようなコンテナのセットは、ユーザー環境を形成します。



環境を作成するときに、事前に準備されたテンプレートを使用して、要求されたコンテナがプールからプルされます。 これにより、環境の作成とセットアップのプロセスを数秒で短縮できます。



分離されたコンテナ内に異なるスタックが配置されている場合、アプリケーショントポロジの異なるレイヤー（ロードバランサー、サーバー、データベースなど）を完全に独立してスケーリングできます。 これにより、あるスタックが他のスタックに影響を与える可能性がなくなります。







コンテナ仮想化



JelasticはParallels Virtuozzo Containersを使用します。これにより、1つの物理サーバー上のコンテナ数で非常に良い密度を実現できます。 これは、オペレーティングシステムレベルで仮想化を使用することで実現されます 。



このタイプの仮想化により、物理サーバーを小さなコンテナーに分割することができ、その結果、割り当てられた物理リソース内により密に配置されます。 物理サーバーでホストされるコンテナーは、CentOS上の独立したLinuxマシンです（他のバージョンもサポートされている場合があります）。



完全な分離と高レベルのコンテナ保護により、稼働時間が最大化され、相互のリソースの不正な借用が防止されます。 すべてのコンテナがインストールされた環境は完全なプライバシーを取得し、他のユーザーがアクセスできなくなります。



Jelasticが最初の日から完全なコンテナ仮想化を使用し始めた最初のプラットフォームであることは注目に値します。 最も近い競合他社であるOpenShiftとCloudFoundryは、cgroups + SElinuxとWarden疑似コンテナー仮想化レイヤーをそれぞれ実装して、ソリューションを再設計しました。 近い将来、彼らはアーキテクチャの再設計の別の段階を待っています。この段階では、すでに現在のJelasticソリューションに少し近づいています。両方のプラットフォームがDockerに移行します。 ただし、これでもJelasticにはいくつかの重要な利点があります。リソースの利用効率が同じコンテナの分離、ライブマイグレーションの存在、プラットフォーム内でコンテナを使用する3年の貴重な経験です。



アプリケーションの高可用性



機能的な特徴の1つは、異なる物理サーバー間で同じ環境のコンテナーを均一に分散させることです。 これは、可能な場合は同じ環境のコンテナが同じ物理サーバーに到達しないように、クラスタ全体のコンテナの分散を規制する非アフィニティルールを使用して実現されます。





その結果、クラスター化されたアプリケーションを作成するときに、物理サーバーの1つで問題が発生した場合にアプリケーションが動作を停止するリスクが軽減されます。



ライブ移行



Jelasticは、物理サーバーが過負荷になった場合にクラスターのバランスを取るユニークな機会を提供します。 これはライブマイグレーションによって発生します 。 このようなソリューションにより、物理サーバーからコンテナを自動的に避難させることができ、必要に応じて定期的なメンテナンスが可能になります。 同時に、ライブマイグレーションはアプリケーションのパフォーマンスに影響を与えません-アプリケーションはアクティブなままで、必要なリソースを受け取ります。







また、自動垂直スケーリングの独自の機能がライブマイグレーションと完全に組み合わされているため、サーバー内で遅延や生産性を損なうことなくアプリケーションをスケーリングできます。 1つのアプリケーションがスケーリングする（サーバー上のリソースをますます消費する）場合、他のアプリケーションを他のサーバーに自動的に移行することができ、アクティブに成長するアプリケーションのためにリソースを解放できます。 同時に、アプリケーションコンテナをリロードする必要がないため、アプリケーションのダウンタイムリスクが軽減されます。



ハイバネーションモード（スリープモード）



高度なリソース使用率は、特別なアプリケーションハイバネーションモードを使用して実現されます 。 たとえば、過去2日間アクセスされていない非アクティブなアプリケーションは中断され、スリープモードになり、すべてのRAMがディスクにアンロードされます。 その結果、解放されたリソースは後で使用するためにクラスターに返されます。



コンテナはスリープモードになっているため、リソースを消費せず（ディスク領域のみ）、最終消費者に節約の機会を与えます。 アプリケーションが再び要求されると、プラットフォームは、文字通り数秒で、休止状態になる前のアクティブ状態にアプリケーションを戻します。 将来のリリースでは、この機能が大幅に拡張され、最終消費者が冬眠および「覚醒」のスケジュールをカスタマイズできるようになります。



コンテナレベルのキャッシング



リソース利用をさらに合理的にするために、Jelasticはメモリ重複排除メカニズムを使用します 。 このメカニズムは、異なるファイル（システムライブラリ、アプリケーションサーバー、データベースファイルなど）から最も頻繁に要求される同じファイルに関する統計を収集し、そのようなファイルをキャッシュに入れます。 その結果、コンテナはディスクではなくキャッシュに直接アクセスしてデータを受信できます。 これにより、I / O操作の数が減り、生産性が向上します。



アクセスレベル



Jelasticは、インターネットアプリケーションの開発者、中小企業（標準アプリケーションまたはSaaS用のスケーラブルなホスティングを必要とする）、大規模なソフトウェア開発会社、ITインテグレーター、IT部門が開発された金融組織、そしてもちろんプロバイダーなど、さまざまなユーザーグループに有用ですホスティングサービス。 各タイプのJelasticユーザーには、特定のアクセスレベルと、プラットフォームと対話する特定の方法があります。



Jelasticクラスタへのアクセスには3つのレベルがあります：

  • 管理者
  • 開発者またはSMB
  • エンドユーザー






Jelasticは2つの管理パネルを提供します：クラスタ管理者向けのJelasticクラスタパネル（JCA）（今月はこのパネルに関する大量のドキュメントへの公開アクセスを開く予定です）、および開発者向けのコントロールパネル（ダッシュボード）。 機能が分割された2つのコントロールパネルの存在により、Jelasticシステムは開発チームと運用チームにとって便利なソリューションになります。



管理者



Jelasticプラットフォームの管理者は、インストールされたクラスターを所有する会社のOps部門です。 インストール後、プラットフォームは完全に廃棄されます。



管理者はJelastic Cluster Admin Panelを使用して、基本構成（制限、クォータ、料金、ローカリゼーションなど）を構成し、更新中またはスケジュールされた保守中にクラスターの保守を実行できます。



管理者がJelasticクラスターのライフサイクル中に実行する4つの主なタスクは次のとおりです。

  • 設置
  • 打ち上げ
  • 運営管理
  • 更新する






Jelasticは、上記のすべてのタスクの高品質な実装のための完全なツールセットと詳細な手順を提供します。



開発者とSMB



このユーザーグループは、環境を作成し、アプリケーションを展開し、アプリケーションのライフサイクルを成功させるために必要なすべてのアクションを実行できます 。 環境とアプリケーションの操作は、主に開発者向けのコントロールパネルを介して行われます。



Jelasticシステムの開発者が実行できる基本的なアクションは次のとおりです。

  • シンプルで複雑な環境の作成
  • コードを変更せずにさまざまな方法でアプリケーションをデプロイする
  • ソフトウェアスタック設定の変更
  • アプリケーションの垂直および水平スケーリング
  • テストおよびリモートデバッグアプリケーション
  • アプリケーションのライフサイクル管理
  • クローン作成と共有
  • 環境を一時停止および開始する機能
  • 統計分析とログ処理
  • フルバージョンへの移行とアカウントの補充
  • などなど。


インストールをホストする場合、Jelasticシステムでの開発者のアクティビティは、試用期間中の機能の登録とテストから始まります。 ライフサイクルの主な段階を図に示します。







試用期間の終わりに、開発者はフルバージョンへの切り替えを選択できます。そうしないと、テスト期間中に作成された環境が時間の経過とともに自動的に削除されます。



エンドユーザー



開発者が各自の環境にデプロイしたアプリケーションを使用することにより、エンドユーザーは間接的にJelasticクラスターに関連付けられます。



すべての着信ユーザー要求は、対応するアプリケーションのドメイン名に送信され、次のいずれかの方法で処理されます。



  • 一般的なリゾルバを介して


Jelasticは複数の同期リゾルバを使用してリクエストを同時に処理し、リクエスト間で負荷を分散します。 リゾルバーの数は、クラスター内のアプリケーションの合計負荷によって異なります。







  • 環境のエントリポイント（ロードバランサー、アプリケーションサーバー、またはデータベース）に追加された場合、 外部IPを介して。


本番アプリケーションには外部IPアドレスをお勧めします。これにより、一部のアプリケーションが他のアプリケーションに影響を及ぼす可能性のあるリスクが大幅に軽減されます。 また、外部IPアドレスを使用すると、アプリケーションのリモートデバッグ、リモートバックアップ、JMX、FTPの使用、カスタムSSL証明書の使用、Webソケットの操作などの機能にアクセスできます。
