「決して言わない」またはタイムゾーンを正しく操作する

この記事では、プログラマーがタイムゾーンで作業するのを待つ問題について説明します。 理論的には、すべてが素晴らしく、シンプルで明確なように見えますが、人生は複雑なものであり、実際には、時には完全に予期しない状況が発生します。



TL; DR：タイムゾーンの操作は痛みと屈辱です。 タイムゾーンを使用しないでください！



そのため、 周りの人 は 、ユーザーから時間を取得したら、すぐにUTCに転送する必要があり、時間をUTCで処理するだけで、時間を厳密にUTCで保存する必要があることを伝えます。 アドバイスは一見合理的であり、それに従うことで生活が楽になります...プログラムに日付の複雑な作業が含まれていない限り。 サイトのユーザー登録の日付と時刻をデータベースに記録しますか？ オンラインストアにメッセージの送信時間または注文作成日を保存しますか？ 日時を示すメッセージをログに表示しますか？ UTCを使用すると、すべて正常になります。この記事をさらに読むことはできません。 現在の時刻はすべてUTCに完全に変換でき、問題を忘れることができます。 しかし、将来、時間とともに働きたいとしたらどうでしょうか？ または過去に？ たとえば、カレンダーサービス、またはメッセージの遅延送信サービスを作成する場合はどうでしょうか？



UTCは万能薬ではありません

例で説明します。 同じ保留中のメッセージサービスを作成したとします。 私たちのサイトを訪れたユーザーは、メールまたはSMSでいつでも（もちろん、将来的に）自分のリマインダーを作成できます。 私たちのサイトは非常にシンプルです：日付、時刻を設定し、リマインダーテキストと通信チャネル（電子メールアドレスまたは電話番号）を入力し、ユーザーから受信したデータをデータベースに入れてから、定期的に選択してメッセージを送信します。 感謝する人々のすべて、利益と敬意！



いいえ、すべてではありません。 すべての場所を常にUTCで保存するというアドバイスに従って、ユーザーから受け取った日付と時刻をUTCに変換し、データベースに格納しました。 2014年3月2日にモスクワのユーザーがサイトにアクセスし、2014年11月3日の午前9時にリマインダーを作成できるようにします。 したがって、2014年3月2日の2014年11月3日のヨーロッパ/モスクワタイムゾーンのオフセットは「UTC + 4」であったため、データベースに値「2014-11-03 05:00:00」を設定しました。



私が得ているものを参照してください？



はい、2014年7月21日、ロシア連邦下院は夏時間を廃止する法案を採択しました。 この法律によると、2014年10月26日から、ヨーロッパ/モスクワのタイムゾーンへの移行は、「UTC + 4」ではなく「UTC + 3」になりました（夏時間も取り消されましたが、現在はそうではありません）。 したがって、ユーザーに11月3日午前5時にUTCに通知を送信すると、ユーザーはモスクワの午前8時にその通知を受け取ります。



結論は簡単です。時間をUTCで保存できますが、現在および最近の過去のイベント、つまり、タイムゾーンが変更されない日付のイベントのみを保存できます。 将来の日付にUTCで時間を保存するのは危険です。なぜなら、どの国の政府が他の法律を採用し、10年、5年、さらには1年でタイムゾーンに何が起こるかわからないからです。



一方、ユーザーのローカル時間とユーザーのタイムゾーンをデータベースに保存すると、そのようなデータを操作することは事実上不可能になります。 通知サービスの例に戻ります：通知によって作成された2人のユーザー。 モスクワの最初のユーザーは、2014年12月15日15:00にSMSを送信するように要求しました（データベースにローカル時間「2014-12-15 15:00:00」とタイムゾーン「ヨーロッパ/モスクワ」を書き込みます）。 ニューヨークの2番目のユーザーは、2015年12月15日午後7時に電子メールを送信するように要求しました（データベースに彼の現地時間「2014-12-15 19:00:00」とタイムゾーン「America / New_York」を記述します） ） これまでのところ、ユーザーが通知を受け取りたい現地時間を記録しており、これらの国の政府がこれらのタイムゾーン（オフセット、夏時間、何でも）。

それでは、どのように時間を正しく保つのですか？

では、どのようにしてデータベースに時間を正しく保存することができますか？ もちろん、これを行わない方が良いですが、本当に必要な場合は、個人的な推奨事項があります（批判や提案を聞いてうれしいです）：

  1. 発生したイベントの時刻、特定のアクションの後の現在時刻を保存する必要がある場合は、UTCで保存します。 これらは、ログエントリ、ユーザー登録の時間、手紙の注文または送信です。
  2. 時刻がユーザーまたはユーザーのタイムゾーンに関連付けられていない場合は、UTCで保存します。 これは、たとえば、次の日食の時間かもしれません。
  3. 過去または未来の時間を保存する必要がある場合は、ユーザーの現地時間を保存し、タイムゾーンを近くに保存します。 そしてさらに良いので、確かに、ユーザーの地理的な位置を保持します。 この時間のサンプルを作成する必要がある場合は、時間を近くのUTCで保存し、タイムゾーン情報を変更するときにこの時間を更新します。
  4. 特定の地理的位置の任意の日付の時刻を正確に知る必要がある場合（たとえば、天文計算の場合）-ユーザーのタイムゾーンではなく、ユーザーの正確な座標を保存します。 ただし、そのようなタスクに直面している場合は、正しい方法をすでに知っています。

違います。 少なくともすべての場合ではなく、ここに具体的な例を示します。



遅延メッセージのサービスを使用して例に戻りましょう。 すべては順調で、サービスは開発中です。ユーザーは満足していますが、繰り返し通知の機能を追加するよう求めています。 また、繰り返しは単純（「毎日」、「隔日」、「毎月」）だけでなく、非常に複雑です（「毎週火曜日の週」、「毎月最後の金曜日の毎月」など）。 これらの繰り返しのために自転車を書かないために、既製のソリューションを研究します。 「定期的なイベント」などがあります。 繰り返しの規則を記述するための特別な形式があります。もちろん、すべての可能なオプションを考慮していません（たとえば、「2 in 2」を指定することはできません）が、ほとんどの場合をカバーします。 この形式のアプリケーションの例は、 iCalendar仕様の RRULEフィールドの説明と 、Pythonのpython-dateutilモジュールのrruleオブジェクトのドキュメントで見ることができます。



python-dateutilモジュールをコードで使用します。 すべてがうまくいくように見えますが、ユーザーは文句を言います、そして、これらの不満の研究はかなり予期しない結果で私たちを導きます。



定期的なイベントの1つのオプションは、曜日ごとに繰り返すことです。 たとえば、毎週火曜日と金曜日の12:00に繰り返されるイベントを記述できます。 実際のコードで実際にどのように見えるかを次に示します。 

 >>> import datetime >>> from dateutil import rrule >>> list(rrule.rrule(rrule.WEEKLY, count=4, byweekday=(rrule.TU, rrule.FR), dtstart=datetime.datetime(2014, 11, 3, 12, 0))) [datetime.datetime(2014, 11, 4, 12, 0), datetime.datetime(2014, 11, 7, 12, 0), datetime.datetime(2014, 11, 11, 12, 0), datetime.datetime(2014, 11, 14, 12, 0)]


すべてが順調に進んでいるようです。 ここで、モスクワのユーザーが朝の1時に発生する定期的なイベントを作成したとします。 彼から「2014-11-03 01:00:00」という時間を得るとすぐに、私たちは賢い人の推奨に従って、すぐにそれをUTCに転送します（今は翻訳プロセスに興味がありません。実際には3時間かかることを知っておく必要があります）受信した時間から）、UTCの次の時間を取得します：datetime.datetime（2014、11、2、23、0）。 これまでのところ良い。 受信した時間のリプレイを取得しましょう： 

 >>> list(rrule.rrule(rrule.WEEKLY, count=4, byweekday=(rrule.TU, rrule.FR), dtstart=datetime.datetime(2014, 11, 2, 23, 0))) [datetime.datetime(2014, 11, 4, 23, 0), datetime.datetime(2014, 11, 7, 23, 0), datetime.datetime(2014, 11, 11, 23, 0), datetime.datetime(2014, 11, 14, 23, 0)]


何かが間違っているようです。 取得した値をユーザーの現地時間に変換すると（それぞれに3時間追加されます）、繰り返しが移動し、イベントは午前1時ですが、水曜日と土曜日にすべて同じように繰り返されることがわかります。 これはpython-dateutilモジュールのエラーではなく、コードは正しく機能しました。 これは私たちの間違いです。この特定のケースでは、ユーザーの現地時間で作業する必要がありました。



ところで、多くのカレンダーサービスにはこのバグがあります。たとえば、OS XのiCalプログラムは、特定の場合、再試行が完全に間違っていると見なします。
