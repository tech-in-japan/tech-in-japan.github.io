イソトキシン：Toxをサポートする無料のマルチプロトコルメッセンジャー

あなたはトックスのことを聞いたに違いありません。 思い出させてください：これは、Toxネットワークの参加者間でメッセージ、オーディオ、ビデオストリームを送信するための無料の安全なp2pプロトコルです。 実際、これはskypeの代替です。 Toxについて初めて聞いたとき、スカイプに対する嫌悪感はまだそれほど高くはありませんでしたが、すでに代替手段を探し始めていました。 Toxの根底にあるアイデアが大好きでした。必要なすべての機能を備えたプロトコルを作成し、それをクライアントが作成します。 Toxネットワークの最初のクライアントが登場したとき、私は「地獄、私はそれをもっと悪くすることはできない！」と考えました。 一般に、Skypeへの敵意に押されて、夢のメッセンジャーのプロジェクトを始めました。 さて、私のローカルhgリポジトリで最初のコミットが19か月で、414番目のコミットが行われたとき、Habrにこのクライアントについて幅広い聴衆に伝える前に、私は最終的に熟成しました。







何をする



  • ビデオコールを含むtoxプロトコルの現在のすべての機能を完全にサポート（nospamの変更を除きますが、この機会を本当に必要としている人が世界中にいることを強く疑います）
  • ローカルネットワーク内の通信用の独自のプロトコル（主にプラグインシステムのデバッグ用に作成されましたが、非常に機能的です。ビデオを除き、すべてtoxと同じです）
  • 複数のプロトコルの同時操作のサポート（たとえば、一度に異なるIDで2つのtox接続を使用できます）
  • 履歴、アバター、顔文字、メッセージの到着通知などの保存など、球状メッセンジャーの標準機能
  • メタコンタクト、音声/ビデオ通話、デスクトップ共有、グループチャット、メッセージ検索、ファイル転送などの高度な機能
  • パスワードを使用したプロファイル暗号化。 プロファイルは、すべての設定とメッセージ履歴を含むファイル、sqliteデータベースです。
  • インターフェイスでの「スキン」のサポート
  • 自動更新


より完全なリスト
  • 連絡先リストで連絡先を検索する
  • ライブメッセージ検索
  • メッセージ履歴を保存しない機能
  • 連絡先タグとタグによるフィルタリング（最新バージョンで行われます）
  • クライアントバージョン（Isotoxinの場合）およびそのTox ID（追加リクエストを送信した場合にのみスパムが表示されない）を含む連絡先の詳細を確認できます。
  • 連絡先にコメントを書くことができます（説明だけで、どこにも送信されません）
  • 個々の接続にアバターと名前を指定できます
  • Webカメラからアバターを作成できます
  • サウンドアラート。 アラートの音を選択する機能
  • スマイルセットのサポート。 アニメーションGIFのサポート
  • メッセージ作成時間が経過しました（toxプロトコルを介したアドオン）
  • メッセージと名前のbbコードのサポート。 反対側のクライアントがbbコードをサポートしていない場合、名前は彼に送信されますclean
  • 多言語主義。 再起動せずに言語を切り替えることができます
  • 「退席中」のステータスは自動的にオンになります（カスタマイズ可能）
  • メッセージを送信するさまざまな方法：Enter / Ctrl + Enter / Shift + Enter / Double Enter
  • メッセージの日付と時刻を表示するためのカスタマイズ可能な形式
  • メッセージ履歴をtxtまたはhtmlファイルにエクスポートする機能（必要に応じて、エクスポート用に任意のファイル形式を構成できます）
  • 入力通知の表示と送信
  • ファイル拡張子に基づいた自動ファイル受信
  • メッセージリストにアニメーション画像を含む画像を表示します
  • 送信する前に画像をトリミングできます。 画像自体は、ファイル、クリップボード、Webカメラ、またはデスクトップから取得できます。 トリミング後、最終サイズに応じてjpgまたはpng形式が自動的に選択されます
  • 自動更新はプロキシを介して起動できます
  • 自動起動セットアップ
  • トレイに最小化
  • 個々の連絡先については、マイクのミュートで音声通話の自動受信を有効にできます
  • ビデオを送信するとき、イメージはソースでレシーバーのサイズにスケーリングされます。 ストリップを保存しました
  • メッセージテキスト、リンク、または連絡先IDまたは現在のTox IDからのQRコードの生成


もちろん、やるべきことはまだたくさんありますが、快適に使用するにはこれで十分です。



開発



私はC ++とWindowsでのプログラミングでかなりの経験を持っています。 この経験を無視するのは愚かなことなので、開発はC ++およびWindowsで開始されました。 しかし、さまざまなプラットフォームをサポートすることの重要性を認識し、私はすぐにwinapiおよび他のWindows専用機能への依存を最小限に抑えることにしました。 私のテキストレンダリングではGDIも使用していませんが、自分で作成します。 フォントを読み取るには、FreeTypeライブラリが使用されます。 すべてのGUI、すべてのコントロール-すべてがゼロから記述され、システムに関連付けられていません。 winapiウィンドウのすべての作業は別の場所で行われ、インターフェイスロジックから可能な限り抽象化されます。 小さな背後にある問題-Linuxで書くことを学ぶ。 これまでのところ、私はこれのための時間がありません。 移植したい人がいるかもしれません。

最初から、私はモジュラーアーキテクチャを考案しました。 実行可能ファイルはGUIのみに対応している必要があります。 すべてのネットワーク業務はモジュールに委ねられました。 簡単に言えば-通常のプラグインシステム。

開発プロセス中に発生する問題の1つは、Visual Studioでビルドするtoxカーネルの完全な不本意です。 すべてがC99標準の動的配列にあることが判明しました。 コアカーネル開発者であるirungentooは、これらの配列をコードから削除する要求を拒否します。「C99は標準です。 msvcがサポートしていない場合、これが彼の問題です。」 あなたが私の個人的な意見に興味があるなら、動的配列は悪であると考えます。 それらは、sizeof演算子の不変性を殺します。 したがって、msvcコンパイラーの開発者と、動的配列と通信することをためらうことを完全に理解しています。 そのため、2つの方法がありました。cygwinを介してtoxcoreを個別に収集する（他のToxクライアント開発者がWindowsで行うように）か、コードから動的配列を自分で削除します。 2番目のパスを選択した結果、 このリポジトリが表示されました。 これは、 toxcoreの直接のフォークではなく、必要なライブラリ（libsodium、opus、vpx）からの一種のアセンブリであり、msvcコンパイラ（2013および2015でサポート）によって簡単かつ簡単にアセンブルされます。

Isotoxinのコードのほとんどすべては、サードパーティのライブラリと同様に友人によって書かれたいくつかの点を除いて、私だけによって書かれました。 C ++ 11の機能が積極的に使用されています。 2015thスタジオのアップデートのリリースといくつかの重大なバグの修正に関連して、私はconstexprとC ++ 14を最大限に活用するためにこのコンパイラに完全に切り替える予定です。



建築



Isotoxinのコードの最初の行を書き始めたばかりのとき、私の計画にはオープンソースへの欲求が完全に欠けていました。 ただし、 toxカーネルが提供される GPL3ライセンスと競合がありました 。 矛盾とは何ですか？ つまり、このライセンスでは、このライセンスの下でライブラリを使用するプログラムのソースコードを開く必要があります。 このジレンマの明らかな解決策は、プロトコル実装を別の動的ライブラリ（dll）に移動し、そのdllのソースを開くことです。

しかし、私はもう少し先へ進むことにし、dllだけに限定しませんでした。 その結果、次のアーキテクチャが誕生しました。

1. isotoxin.exeはクリーンなGUIです。 最初は、そのソースを開きたくありませんでした。 このファイルの信頼度を高めるために、ネットワークを操作する機能を完全に奪うことにしました。 後で、それにもかかわらず閉鎖を拒否したとき、ネットワークでの作業が返されましたが、それはただ1つの目的でした-アップデートの確認とダウンロードです。

2. plghost.exe-プロトコルdllをロードする小さなexe'shnik。 より正確に-唯一のdll。 isotoxin.exeから直接起動し、プロセス間通信の最新の手段と通信し、コマンドをプロトコルdllに変換して結果を送り返します。

3. proto。*。 Dll-実際にはプロトコルdll。 単一のインターフェースを実装します。



このアーキテクチャには長所と短所があります。



メリット。

  • コード責任の差別化。 GUIからプロトコル関数を取得して呼び出したり、その逆を行うことはできません。 これにより、エラーを最小限に抑えて、明確で安全なコードを作成できます。
  • 信頼性 クラッシュを引き起こすプロトコルエラーはGUIをドロップしません。 plghost.exeがクラッシュした場合、後者は単に再ロードされます。 ユーザーは問題に気付かないことさえあります。
  • セキュリティ。 ユーザー-偏執病-は、ディスクの操作に関するさまざまなルールを使用してplghost.exeに課すことができます。たとえば、書き込みと読み取りを完全に制限できます。 何も起こらなかったようにすべてが機能し、攻撃者がプロトコル実装の脆弱性を使用してコードにアクセスできたとしても、サンドボックスを離れることはできません。
  • 将来-32ビットと64ビットの両方のプロトコル実装の同時操作のサポート。 plghost.exeの64ビットバージョンを収集するだけで十分です。 T.O. Isotoxinの64ビットバージョンのリリース（および、これが発生することを除外しません）は、サードパーティプロトコルの64ビットバージョンの緊急リリース（必要な場合）を必要としません。


欠点。 そして、それらなしではどうでしょう。

  • コードとそのデバッグの重大な複雑さ。 いくつかのexe-shnikの同時デバッグはシュガーではありません。
  • 追加ファイル（plghost.exe）
  • システムリソースの要件はわずかに高くなります。 それにもかかわらず、各動作プロトコルで追加のexeを起動することは無料ではありません。


欠点のリストは、多少なりとも小さく、深刻ではないことが判明しました。 これ以上のことは思いつきませんでした。 したがって、アーキテクチャの選択は正しいと考えています。

メリット



すべての開発者が自分の製品を賞賛しようとしていることは明らかです。 できる限り客観的になるよう努めます。 現在、qToxは、toxネットワークをサポートするすべてのクライアントの中で最も人気があり、機能が豊富なものとして認識されるべきです。 ただし、IsotoxinはほとんどすべてのカウントでqToxをバイパスします（もちろん、非ウィンドウサポートを除きます）。

  • 移植性。 ある時点で、qToxはポータブルではなくなりました。 おそらくポータビリティが彼に再び返されたのだろう、私は知らない、チェックしなかった。
  • XPサポート。 qToxはWindows XPで実行できなくなりました。 一部の人にとって、これは問題になる可能性があります。 イソトキシンは問題なくXPで動作します
  • 安定性。 イソトキシンはほとんど落ちません。 すべての秋は私の個人的な悲劇です。 転倒の原因を突き止めるまで、落ち着きません。 qToxは、積極的に使用しなくても落ちました
  • プロトコルにない追加機能。 toxcore forkでは、もちろん、下位互換性のある新機能を紹介しています。 イソトキシンがワイヤーのもう一方の端でイソトキシンを検出すると、これらのオプションがオンになります。 これらの機能の1つは、メッセージの長さが無制限であることです（他のクライアントは、カーネルの作成者の推奨に従って、長いメッセージをスペースで切り取ります）


まだまだありますが、記事を煩雑にせず、広告に変えないために、すべてを説明するつもりはありません。



計画



将来の多くの計画があります。 プロトコルを追加することに加えて（私は本当にTelegramとjabberを追加したい）、Linuxに移植したいという大きな要望があります。 知識のある人にとっては、これは問題にならないはずです。 上記で書いたように、Windowsの特別な機能は使用していません。 一般的には、Skypeを完全に置き換えることです。
