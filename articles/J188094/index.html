テクニックといくつかのスクリプトを使用した写真とビデオのアーカイブ内の順序

私はプロの写真家とはほど遠いですが、写真を撮るのが好きです。定期的にさまざまな品質の数万枚の写真のアーカイブに、DSLR、「ソープボックス」、3台の電話からの写真、電話やビデオカメラからのビデオ録画が補充されます。 写真やビデオのアーカイブを順番に維持しようとしていないソフトウェアです！ Adobe Lightroom、Apple iPhoto、Google Picasa ...



その結果、それらのどれも問題をうまく解決できず、私は自分で書かなければなりませんでした。 私の経験は、この記事が生まれた誰かに役立つと思いました。



あなたが写真家であるだけでなく、小さなプログラマでもあるなら、30分であなたのために同様のシステムを作成することができます。







問題の声明

だから、どのような問題を解決する必要がありましたか？ カタログ作成者に期待することはどれほど珍しいですか？



  • 年、月、日ごとに写真を解析するメカニズムが必要です 。 必要に応じて時間ごとにイベントをすばやく見つける。
  • インポート中に重複を特定し、新しいもののみをアーカイブに追加する必要があります。
  • フリーソフトウェアが必要です。できればインストールは不要ですが、理想的にはマルチプラットフォームも必要です 。 アイデアは、ソフトウェアが写真アーカイブと一緒に外部ドライブにあり、このドライブが接続されているマシンから実行する必要があるということです。 その理由は、信頼性とハードドライブ容量の要件です。
  • すべての再インストール、オペレーティングシステムの変更 （別のものにさえ）に耐えるソフトウェアが必要です。
  • バックアップをローカルに作成するための便利なメカニズムが必要ですが、バックアップはソフトウェアに依存せず（つまり、単なるフォルダーのコピーでした）、全体または一部（サブフォルダー）を簡単に復元できることが重要です。
  • 写真のRAWバージョンを （理想的には一般に別のディスクに） 個別に保持し、JPGバージョンと共有する必要があります（高解像度であっても）。 さらに、必要に応じて、 目的のファイルのRAWバージョンを迅速かつ簡単に上げることができます。 ええ、はい、どのファイルにこのRAWバージョンがあり、どのファイルにないのかを理解する必要があります。
  • 同様に、 Canonカメラ（.MTS）からAVCHD録画用のビデオプレビュー（.AVI）を作成し、 FullHDビデオ（.MTS）をプレビュービデオ（.AVI）とは別に保持する必要があります。目的のフラグメントを検索してブラウズしますが、編集に高品質が必要な場合は、どこで入手できますか
  • 写真をフォルダ全体でネットワークにアップロードする必要があります （Yandex.Fotki、Facebook）
  • 携帯電話、タブレット、ノートパソコンでプレビューの最新の「モバイルアーカイブ」を作成します 。




最も一般的な要件。 おそらく、世界にはこれをすべて実行できる既製のソフトウェアがありますが、私はそれを見つけることができませんでした。 したがって、私は自分で書く必要がありました。



ソリューションコンセプト

だから私のアプローチは何ですか？



  • データベースなし、専用ソフトウェア 。 すべての写真は、順序付けられた方法でオープンファイルシステムに保存されます。
  • 任意のオペレーティングシステム用に作成でき、必要に応じて変更できる単純なスクリプトは、整理する責任があります。
  • さまざまなタスクにさまざまなハードドライブを使用する 。 外付けHDD-重いファイル（.MTS、.RAW、おそらくJPGのオリジナル）、およびラップトップ自体の再生可能な軽いアーカイブ（プレビュー）用。 RAWおよびVIDEOのオリジナルは、外付けハードドライブに保存できます。これらはすべてのディスク領域を使い果たす傾向があり、非常にまれにしか使用されず、重要であり紛失してしまうためです。 JPGのオリジナルは、外付けハードドライブ、または自信がある場合はローカルドライブ（Raidアレイ、デスクトップハードドライブ）に保存されます。 プレビュー（写真とビデオは別々に）はラップトップまたはデスクトップPCのローカルハードドライブに保存され、ネットワークへのアップロードに便利です。

    IMPORTフォルダー

    カメラと携帯電話の新しいものはすべてここにコピーされます。 このヒープには、さまざまな解像度のJPG写真、RAW写真、AVIビデオ、MOVビデオがあります。 処理中に、写真とビデオはこのフォルダーから他のフォルダーに転送され（以下を参照）、古いインポート済みの写真はこのフォルダーから削除されます。 その結果、手動で処理される何らかの種類のゴミが残る場合があります。



    ORIGフォルダー

    元の写真をJPG形式で保存するためのフォルダー。 年月日ごとに写真とビデオを構成するには、このフォルダーと後続のフォルダーで年月日階層を使用します。 つまり、2004年9月12日の写真はORIG / 2004/09/12 /ブランチに保存されます。 私のファイル名はIMPORTの元のファイル名と同じですが、たとえば、タイムスタンプYYYYMMDDHHMMII.JPGの形式でファイル名を一意にして、異なるデバイスからインポートするときに重複しないようにするのが正しいです。



    RAWフォルダー

    RAW写真が検出されると、ネイティブJPGバージョンがあるかどうかが確認され、存在しない場合はORIG / YYYY / MM / DD /で自動的に作成され、RAW写真自体はRAW / YYYY / MM /に転送されますDD /。 同時に、ORIGのJPGバージョンの名前で、このファイルのIMG_5905_RAW.JPGのRAWバージョンがあることが示されます。 通常、RAWファイルはサイズが大きいため、このフォルダーを外部ドライブに保存することは理にかなっています。



    VIDEOフォルダー

    ビデオファイルのオリジナルはここに転送されます。 通常、それらは非常に大きいため、このフォルダーを外部ドライブに保持することは理にかなっています。



    PREVIEWおよびPREVIEW_VIDEOフォルダー

    サムネイルはインポート時に自動的に作成され、PREVIEWフォルダーに配置され、PREVIEW / 2004/09/12 /のような階層が保持されます。 モバイルフォトカタログのモバイルデバイスで破棄され、フォトホスティングにアップロードされます。



    プレビューのアーカイブをモバイルデバイスにダンプし、定期的に更新します-日付順にフォルダをコピーするだけです。 RAWおよびVIDEOフォルダーを外部ドライブに保存しておくと便利です。これらのディスクには空きディスク容量を埋めることができ、ラップトップのプレビューでフォルダーを保持できるためです-何らかのハードディスク障害またはラップトップの紛失により突然失われた場合、それらは常にもう一度行うことができますが、場合によっては、適切な写真をすばやく見つけたり、友人に見せたりすることができます。



    これは、ビデオカメラからのFullHDビデオに関して特に重要です。 ホームアーカイブの場合は、画面品質のビデオを保存するだけで十分ですが、インストールの場合は、元のMOVをフルHDで撮影することをお勧めします。 さらに、キヤノンのカメラからフルHD MOVを再生すると（少なくともロード中は少なくとも最初の数秒間）、すべてのラップトップの速度が完全に低下するため、国内の目的で元のビデオを積極的に使用することは困難です。 したがって、ビデオプレビューがローカルで、FullHDバージョンが外部ドライブにある場合、非常に便利です。



    しかし、イベントはどうですか？

    イベントごとにグループ化するには、日ごとにフォルダーへのシンボリックリンクを使用します。 つまり、マデイラ島への旅行は、EVENTS / Madeira /というフォルダーの形式で保存されます。このフォルダーには、2012/05 / 01、2012 / 05/02、...、2012/05/09のフォルダーが含まれています。 しかし、このアプローチでは解決できないいくつかの困難があります。



    最初の理由は、3500枚の写真ではなく、数百枚の写真を選んだマデイラへの旅行に関連付けたい場合があるためです。 2番目は、トピックが日ごとに明確に打ち勝つとは限らないことです。 多くの場合、日中はいくつかのトピックがあり、一部のトピックは1日の後半と次の前半に影響を与えますが、これら2日間の残りの時間は別のイベントに関連します。



    したがって、上記の方法に加えて、1）ファイルへのシンボリックリンクを作成するか、2）これらのファイルをイベントフォルダーに物理的にコピーします。 オペレーティングシステムにはシンボリックリンクの大量作成はありません。 それにもかかわらず、そのような方法を実装することは依然として現実的で複雑ではありません。 物理コピーには欠点があります-同じファイルが2つの場所に保存され始めます。 おそらくこれもプラスです-同様の写真を殺したり、最高のものだけを残したりすることができます。 両方のフォルダー内のファイルを検索し、完全な識別情報を使用して、ファイルをシンボリックリンクに置き換えるスクリプトを作成できます。



    アーカイブ全体がハードドライブに保存され、MacBook、Windowsを搭載したコンピューター、Ubuntuを搭載したコンピューターの3台のコンピューターに接続できます。 したがって、ソフトウェアは3つのOSすべてと互換性があることが理想的です。



    それは概念であり、以下はすべて（決して理想的ではない）実装です。



    実装

    このブロックは、プログラミングや自動化に馴染みのない写真家向けに既に用意されています。 タスクは非常に単純であり、bashスクリプトに関する最小限の知識で理解できます。



    私が持っているソフトウェアはすべて2つのスクリプトです。 IMPORTフォルダーのすべてを処理し、必要に応じてORIG / YYYY / MM / DD /、VIDEO / YYYY / MM / DD /、RAW / YYYY / MM / DD /フォルダーを同時に作成します。 2番目のスクリプトは、ORIG、VIDEO、RAWをスキャンし、写真とビデオのプレビューを作成し、プレビュー/ YYYY / MM / DDフォルダーに配置します。



    パパによるファイルのインポートとソート





    最初のスクリプトを機能させるには、exiftoolというユーティリティが必要です。 EXIFファイル情報から撮影日時を抽出できます。 3つのすべてのオペレーティングシステムで見つけることができます。 ubuntuでは、apt-get install libimage-exiftool-perlパッケージからインストールされます。 WindowsおよびMacOSの場合は、ここからダウンロードしてください： http : //www.sno.phy.queensu.ca/~phil/exiftool/



    ロジックを開発するためのツールとして、bashスクリプトとperlを選択しました。 第一に、MacOSとUbuntuで同じであり、Cygwinポータブルでも同じであるため、Windowsで大幅な編集を行わなくてもすべて動作します。 あなたの許可を得て、私はMacOSとUbuntu、そして画像処理に集中し、まったくスプレーされないようにします。



    以下に、タスクを実行する非常に単純化されたスクリプトを示します。



    まず、指定された.JPGファイルをライブラリに追加するスクリプトを分析しましょう。 



    #./add_file_to_media_library.sh <filename>




    呼び出し構文からわかるように、パラメーターで指定された1つのファイルで機能します。 利便性のための小さな制限-パスは相対的である必要があります。 つまり、 "IMG_5949.JPG"（現在のディレクトリ内）またはMYPHOTOS / IMG_5949.JPG "（現在のディレクトリにあるMYPHOTOSフォルダ内）のいずれかです。



    1. export PREFIX = "/ Volumes / Elements /" ＃これは写真アーカイブへのパスです。 このフォルダにはORIGが含まれています
    2. export FILENAME = " ` pwd` / $ 1 " ＃＃ファイル名にフルパスを追加します。

    4. echo $ OUT | grep "日付の作成" ;
    5. ＃次に、日付からORIG形式のパスを作成します/ 2013/05/01 /-最終的にファイルを配置する場所
    6. export FOLDER = ` echo $ OUT | perl -i -npe "s /^(.*？）：（\ d +）\：（\ d +）\：（\ d +）（。*？）$ / ORIG \ / \ 2 \ / \ 3 \ / \ 4 \ // g "、
    7. ＃そのような方法がない場合は作成する
    8. echo mkdir -p " $ PREFIX $ FOLDER "
    9. mkdir -p " $ PREFIX $ FOLDER "
    10. echo -n $ FILENAME
    11. ＃パスはパラメーター内で相対パスにすることができるため、ファイル名のみを引き出します
    12. export NEWFILENAME = ` echo $ FILENAME | perl -i -npe "s / ^（。*？）\ /（[^ \ /] +？）\。JPG $ / \ 2.JPG / g" ; `
    13. ＃パスタイプORIGにファイルが既に存在するかどうかを確認する/ 2013/05/01 /
    14. if [ -f " $ PREFIX $ FOLDER $ NEWFILENAME " ]
    15. それから
    16. ＃は、その後削除
    17. echo "exist ... $ PREFIX $ FOLDER $ NEWFILENAME "
    18. ＃必要に応じて、IMPORTフォルダーからの削除を有効にできます
    19. ＃しかし、これは危険なビジネスです。なぜなら、ターゲットファイルが破られたり、サイズがゼロになる可能性があるからです。 その後、写真を失う可能性があります。 どこかにファイルを転送できますが、気にしませんでした

    21. 他に

    23. echo mv \ " $ FILENAME \" \ " $ PREFIX $ FOLDER \"
    24. mv " $ FILENAME " " $ PREFIX $ FOLDER "
    25. fi








    その結果、すべてのソフトウェアはexiftool、bash、perlのみ必要であり、bashで数行を取ります。 上記のフラグメントを呼び出し、パラメータ「ファイル名」を渡すスクリプトを作成することは今のままです。 これを行うには多くの方法があります、私は使用します 



     # find . -name «*.JPG» | perl -i -npe "s/^(.*?)[\n\r]+$/\.\/add_file_to_media_library.sh \"\1\"\n/g" ./add_file_to_media_library.sh «.//IMG_5790.JPG» ./add_file_to_media_library.sh «.//IMG_5802.JPG» ./add_file_to_media_library.sh «.//IMG_5794.JPG»




    したがって、add2media.shを作成します。 



     #!/bin/bash find . -name «*.JPG» | perl -i -npe "s/^(.*?)[\n\r]+$/\.\/add_file_to_media_library.sh \"\1\"\n/g" > /tmp/run.sh bash /tmp/run.sh rm /tmp/run.sh




    （BTWはrun.shを作成する非常にエレガントなソリューションではありません。findではxargsと-execの両方を使用できますが、すべてにマイナスとプラスがあります）

    可能な開発

    FacebookとGoogle Picasaで写真を共有します。 この点に関して、PREVIEWからの計算を自動化する開発もいくつかありました。 FacebookではFBCMDユーティリティが使用され、Picasaでアルバムを作成して画像をバッチアップロードするには 、 Googleコマンドラインツールが使用されます。



    私の意見では、写真の説明の追加、タグ付け、顔など、既存のカタログ作成者のさまざまな「機能」。 1つのイベントで1000発のショットが生成され、それらの1つを選択するだけで重要な作業が深刻な作業になるような状況にはあまり適していません。 しかし、EXIF情報の使用と変更は、非常に興味深い方向に見えます。 たとえば、5月1日から10日までマデイラにいたことを示すことができ、対応するEXIFフィールドのORIGのすべての画像が地理座標を取得します。 その後オンラインホスティングにインポートすると、この情報を読み取ることができ、写真が自動的にその場所に添付されます。



    また、夜間の写真とビデオのアーカイブ全体が、安全なクラウドバックアップのどこかにドロップオフするのを見たいです。 上記のアプローチで自動化することも非常に簡単ですが、まだ手が届きません。 どのカタログ作成ツールを使用していますか？ 私のアプローチで考慮されていないそれらに必要なものはありますか？
