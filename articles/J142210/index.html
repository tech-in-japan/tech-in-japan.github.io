この名前の関数（「worker」）のサーバーにハンドラーが登録されており、現在ハンドラーが空いている場合、処理用のデータと関数の名前がタスク（「job」）の形式で送信されます。

Gearmanのインストールと制御



大きなニーズがある場合は、WindowsでGearmanサーバーを起動できます。Javaでの実装があります。

https://launchpad.net/gearman-java



しかし、Linuxでの作業を検討します（次の例はDebian用です）。

サーバーのインストールはスムーズに実行され、機能はありません。



aptitude install gearman-job-server



インストールが正常に完了すると、サーバーはetc / init.d / gearman-job-serverからシンプルで明確なコマンドで制御されます。

{開始| 停止| 再起動| 強制リロード}

サーバーのデフォルトのホストはlocalhost、ポートは4730です





サーバー状態監視


サーバーで何が起こっているのかを常に確認する必要があります。登録されているタスク、キューにあるタスクの数、各タスクに登録されているワーカーの数。

これは、次のコマンドを使用してコンソールから実行できます。

（エコーステータス）| netcat 127.0.0.1 4730





ギアマンを再起動したときのキューの運命


すぐに問題が発生します： hara-kiriで実行中のサーバーを再起動すると、実行中のサーバーのキューはどうなりますか？

デフォルトのインストールでは、キューはメモリに保存されます。サーバーを再起動すると、クライアントがサーバーに送信したすべてのタスクがキャンセルされ、サーバー上のすべてのワーカーの登録がキャンセルされます。 同時に、ワーカーに対して「サーバーとの接続が失われました」などの例外が生成されます。

キューはデータベースに保存でき、MySQL、PostgreSQL、SQLiteがサポートされています。 そのようなキューを整理する方法は、オフィスでよく書かれています。 ウェブサイト：

http://gearman.org/index.php?id=manual:job_server#persistent_queues



キューを保存しても、サーバーの再起動時にタスクハンドラー（ワーカー）がサーバーから "脱落"するため、ワーカーを再起動するか、ワーカー自体でこのような状況を予測する必要があることに注意してください。





キュー全体をリセットし、特定のタスクのキューをリセットします


キューがデータベースに保存されていない場合、上記のようにサーバーを再起動すると、キュー全体がリセットされます。 キューがデータベースに保存されている場合、サーバーの再起動に加えて、対応するテーブルをすべてまたは選択的にクリアする必要があります。 サーバーを再起動すると、クライアントとワーカーの両方で例外がスローされることを思い出してください。



ただし、1つのタスクのみのキューをリセットする必要がある場合に状況が発生する可能性があります。 例：100のサイトを解析し、1つが応答を停止し、このサイトからデータを受信するためにキューに1,000のタスクがあり、他のすべてのサイトからのデータは問題なくキューサーバーを介して受け入れられます。 停止したサイトでのみキューを削除する必要があります。

これを行う最も簡単で最も簡単な方法は、キューサーバーに同じ名前のタスクを登録する偽のワーカーを開始することです。ただし、NULL、空の文字列、または好きなものをすばやく返します。主なことは高速です。 このワーカーはすべての保留中のタスクを許可し、キューはクリアされます。





ワーカーの再起動


なぜこれが必要なのでしょうか？ さて、労働者が凍結している疑いのない明らかな状況に加えて、もう1つ、非常に頻繁にあります。

状況：ワーカーが実行されており、サーバーにタスクを登録しました。 この時点でワーカーコードを変更すると、どれだけ保存しても、タスクが登録された時点でキューサーバーからのコードが処理されます。

コードの変更を有効にするには、つまり、サーバーのタスクを既に変更されたコードで処理するには、ワーカーを再起動する必要があります。つまり、現在のワーカーのスクリプトを停止して再実行する必要があります。 メソッドは簡単です。bashスクリプトを使用して、手動でexit_workerなどのワーカーで関数を処理し、クラウンなどから関数を起動します。





Gearmanと連携するためのPECLおよびREARのPHP拡張



PHPでGearmanキューサーバーを操作するには、 pecl gearmanとpear Net_Gearmanの 2つの拡張オプションを使用できます。



これらの拡張機能には根本的な違いがあります。





ナシNet_Gearman


pear Net_Gearmanは、いくつかのphpファイルです。 そしてそれだけです。 サーバーへのインストールは非常に簡単です。

pear install Net_Gearmanチャネル：//pear.php.net/Net_Gearman-0.2.3

サーバーにインストールすることもできません。アーカイブを解凍し、接続accを実装するだけです。 クラスとそれだけです-例えば、それを使用することができます 



function __autoload($className){ //  pear Net_Gearman,  Net_Gearman_Job if(strstr($className, 'Net_Gearman_') and !strstr($className, 'Net_Gearman_Job_')){ $className = str_replace('Net_Gearman_', '', $className); include_once PATH_TO_PEAR_NET_GEARMAN.$className.'.php'; } //  pear Net_Gearman_Job if(strstr($className, 'Net_Gearman_Job_')){ $className = str_replace('Net_Gearman_Job_', '', $className); include_once PATH_TO_PEAR_NET_GEARMAN.'Job/'.$className.'.php'; } }


ちなみに、これにより、同じDenwerまたはOpenServerを使用して、PHPでGearmanを操作できます。



現在のバージョンは2009年のアルファリリースです。 しかし、これは恐ろしいことではありません。必要なファイルを編集する手間はなく、追加のコンポーネント/ライブラリは使用されません。

pear Net_Gearmanの重要な詳細に注意してください。ライブラリにはサーバーを監視し、（部分的に）サーバーを管理する手段があります。これにより、たとえば上記の監視用スクリプトを作成できます。

バックグラウンドタスクと優先度


タスクのステータスは通常またはバックグラウンド（同期/非同期）であり、その実行の優先度はクライアントで設定されます。 タスクがバックグラウンドとしてサーバーに追加された場合、クライアントは実際にサーバーに単純に「スロー」し、それ以上の運命には関心がありません。 タスクが正常な場合、クライアントはサーバーからの完了結果を期待します。

バックグラウンドタスクの場合、プレフィックスのバックグラウンドが使用されます。

優先順位には3つのレベルがあります-通常（プレフィックスなし）、低/高比率。 低/高、たとえばaddTaskHighBackgroundメソッド-バックグラウンドタスクを高い優先度で追加します。
