サードパーティのTFTPサーバーでLTSPを上げるときに苦しんだこと





少し前、Linuxでターミナルサーバーを作成することに関する詳細な記事がHabrahabrに掲載されました。 そのコンテンツは非常に魅力的であり、そのようなソリューションのカスタマイズのシンプルさを示しています。



私たちのネットワークには、Windows Server 2008 R2と別のTFTPサーバーに基づく同様のソリューションが既にあります。 それが私の問題でした。 以下は、PXEの構成の一部です。 これで汗をかいた猫にお願いします。 



LABEL ltsp KERNEL 10.0.20.8::/ltsp/i386/vmlinuz APPEND initrd=10.0.20.8::/ltsp/i386/initrd.img root=/dev/nbd0 nbdroot=10.0.20.8,2000 ip=dhcp selinux=0 init=/sbin/init-ltsp —


サードパーティのtftpサーバーを使用しているため、別の画像アップロード形式を使用する必要があります。 「oldstyle」という言葉で簡単にグーグル検索されます。 したがって、サーバーとクライアントのコンテンツをインストールして構成した直後に、 dpkg-reconfigure nbd-serverを作成し、質問に答えます 。



  • サポートポート：はい
  • 部数：1（もっとあるかもしれません）
  • イメージへのパスを指定します：/opt/ltsp/images/i386.img
  • ポート番号：2000
  • エクスポート名：any（私の場合はltsp）


これで、ネットワークからマシンをロードする際にエラーが発生せず、マシンにデプロイされているクライアントのイメージをNBD経由で取得できます。 認証用のデータを入力しようとすると、「 サーバーからの応答がありません 。 再起動します。」 そして、全体のキャッチ。



繰り返しますが、サードパーティのtftpサーバーを使用している場合、 / var / lib / tftpboot / ltsp / i386 / aフォルダーの内容はそれぞれ無視され、クライアントのすべてのデータをリストするlts.conf構成は無視されます。 これを修正するには、イメージで使用するlts.confを更新する必要があります。 これを行うには、 / opt / ltsp / i386 / etc / lts.confに移動し、すべてを指定します。 私の場合、これはサーバーアドレスです：



  • サーバー= 10.0.20.8
  • オリジナルの記事で残りのlts.confを使用できます


手順の後、 ltsp-update-imageコマンドを使用してイメージを更新し、端末から接続できます。
