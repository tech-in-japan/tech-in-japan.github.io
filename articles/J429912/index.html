SpeechKit分



Yandexの内部でさえ、誰もがそれが何であるかを知っているわけではないので、SpeechKitについて聞いたかどうかは尋ねません。



SpeechKitは、すべてのYandexスピーチテクノロジーへの扉です 。 このライブラリを使用すると、音声テクノロジーをアプリケーションに統合できます：音声認識と合成、音声アクティベーション。



おそらく音声アシスタントのアリスについて聞いたことがあるでしょう-彼女はSpeechKitに基づいて作業しているだけです。 SpeechKit自体には認識や合成は含まれず、サーバー上で発生します。 しかし、すべてがアプリケーションに統合できるのは、ライブラリを介してです。



通常、次の質問があります-サーバー上のすべてが発生した場合、ライブラリは何をしますか？ なぜ必要なのですか？



ライブラリは多くのことを行います：



  1. すべてのプロセスの同期。 たとえば、音声アシスタントを使用して、ユーザーはボタンをクリックし、何かを言って、アシスタントを中断し、リクエストを行います。すべてはライブラリを通過します。 これはライブラリのユーザーにとって透過的であり、ユーザーはこれらすべてを心配する必要はありません。
  2. ネットワーキング すべてはサーバー上で行われるため、そこからデータを取得して処理し、ユーザーに提供する必要があります。 SpeechKitは、同じネットワーク接続内の複数の異なるサーバーにアクセスできるようになりました。1つは認識に関与し、もう1つは意味の割り当てに、3つ目は音楽認識に、などです。 これはすべてライブラリ内に隠されており、ユーザーは心配する必要はありません。
  3. オーディオソースを操作します。 私たちは人間のスピーチを扱っており、音声の処理もSpeechKit内で行われます。 また、標準デバイスから書き込むだけでなく、どこからでもデータを受信できます。 ファイルでもストリームでもかまいません。これらすべてを処理できます。


SpeechKitは内部チームで使用されます。 現在、16のYandexチームによって統合されています。 また、これを行ったいくつかの外部チームについても知っています。

Case SpeechKit API



例としてSpeechKitを使用して、バージョン2をバージョン3にリファクタリングした方法を説明します。APIを大幅に変更し、これらすべての原則を使用しようとしました。



APIが複雑で「理論的」だったため、ニーズが生じました。 その中に最初に呼び出さなければならないグローバルコンポーネントがありました-呼び出されていません-すべてが機能していませんでした。 非常に奇妙な設定が設定されました。 SpeechKitはもともとNavigatorの一部だったため、APIは非常に「理論的」でしたが、この作品はライブラリに取り込まれました。 APIは基本的に、Navigatorで使用されるケースで機能しました。



徐々にユーザーの数が増え、実際に必要なもの、つまりメソッド、コールバック、パラメーターを理解し始めました。 彼らは、APIが実装を許可しなかったリクエストとともに私たちに来ました。 これが何度も繰り返され、APIが限界に達していることが明らかになりました。 そこで、リファクタリングに関与しました。



リファクタリングプロセスは長く（半年）、苦痛でした（誰もが不幸でした） 。 主な難点は、大量のコードを取得して書き直さないことでした。 単純にリファクタリングを行うことは不可能でしたが、使用されていたすべてのアクティブバージョンをサポートする必要がありました。 ユーザーに次のように言うことはできません。「みんな、はい、機能しません。はい、この機能が必要です。バージョン3ですべてを行います。6か月お待ちください！」



その結果、リファクタリングには長い時間がかかり、プロセスもユーザーにとっても苦痛でした。 結局、後方互換性なしにAPIを変更したからです。 彼らは彼らにやって来て、「ここに新しい美しいSpeechKitがあります。どうぞお試しください！」と答えました。 たとえば、1年以内にこのバージョンに切り替えたチームがありました。 そのため、私たちは丸1年間、以前のバージョンをサポートしました。



しかし、結果はそれだけの価値がありました。 統合が簡単になり、バグが少なくなりました 。 これは、APIの基本的な設計原則で述べたものです。 APIが正しく使用されていることが確実な場合、この部分には間違いなく問題はありません。すべてのクラスが正しく呼び出され、すべてのパラメーターが正しいです。 バグを見つけるのがはるかに簡単で、何かがおかしくなる場合が少なくなります。

開発



まず第一に、ライブラリはあなたが書いたものをするべきです-その機能をうまく実行するために。 ただし、コードを非常に優れたライブラリにすることができます。 SpeechKitの開発中に収集したいくつかのコメントを提供します。



コードはあなただけのものではありません



ライブラリが原因でサービスが機能していないとユーザーに言わせたくないので、 デバッグ情報を収集することは絶対に必要です。



IOSには、収集する必要がある情報を示すデバッグ情報レベルがあります。 デフォルトでは、すべての呼び出し、すべての値など、検出可能なすべてのものを完全に収集します。 これはすばらしいことですが、非常に大量のデータです。 -gline-tables-onlyを設定すると、関数呼び出しに関する情報を収集できます。 これで問題を見つけて修正するには十分です。



これはXcode設定（ビルド設定）に含まれており、デバッグ情報レベルと呼ばれます。 たとえば、この設定を有効にすることで、SpeechKitバイナリファイルのサイズを600 MBから90 MBに削減しました。 これはあまり必要な情報ではなく、ただ捨てました。



2番目の重要なことは、 プライベートキャラクターを隠すことです。 ライブラリにiTunesをアップロードするたびに、何かを追加するのではなく、何か間違っていることを使用しているという新しい警告が表示されるリスクがあります。 したがって、Appleがプライベートと見なすライブラリを使用する場合は、必ず非表示にしてください。 これはあなたにとって何の意味もありません。あなたは彼らと一緒に仕事をすることもできますが、ユーザーがあなたのライブラリを使ってアプリケーションをiTunesにアップロードしようとするとすぐにエラーを受け取ります。 誰もがあなたにそれを修正するように頼むわけではなく、ほとんどはあなたの解決策を使うことを単に拒否するでしょう。



文字の競合を避ける ：所有するすべてのもの、クラス、カテゴリにプレフィックスを追加します。 ライブラリにカテゴリUIColor + HEXがある場合、ユーザーが正確に同じカテゴリを使用していることを確認してください。ライブラリを統合すると、キャラクターの競合が発生します。 繰り返しになりますが、誰もがあなたに言ってそう言いたいとは限りません。



別の質問は、自分がライブラリでサードパーティライブラリを使用する場合です。 覚えておく価値のあるニュアンスがいくつかあります。 まず、ライブラリよりも古いバージョンにあるものを使用する場合は、必ず弱いリンクを使用してください（Xcode-> Build Phases-> Link Binary With Libraries-> Status is on）。 これにより、突然このライブラリが落ちない場合、落ちないようにすることができます。



Appleのドキュメントには、この仕組みの詳細が記載されています。 しかし、弱いリンクは、ライブラリが使用されない場合、ライブラリがロードされないことを意味しません。 つまり、アプリケーションの開始時間がユーザーにとって重要であり、サードパーティのライブラリを使用して開始に時間がかかるライブラリの一部が不要な場合、ウィークリンクは役に立ちません。 これにより、ライブラリは使用されているかどうかに関係なくロードされます。



ランタイムにロードする場合、これは起動時のリンクの問題を取り除くのに役立ちます。その後、dlopenと動的ロードを使用する必要があります。 これには多くの手間がかかります。まず、これが理にかなっているかどうかを理解する必要があります。 Facebookには、動的にリンクする方法の例として、かなり興味深いコードがあります。



最後に、 内部でグローバルエンティティを使用しないようにしてください 。 各プラットフォームにはいくつかのグローバルコンポーネントがあります。 ライブラリにドラッグしないことをお勧めします。 これはグローバルオブジェクトであり、ライブラリユーザーはそれを使用して好きなように構成できるため、これは明らかなようです。 ライブラリで使用する場合、何らかの方法でその状態を保存し、再構成してから状態を復元する必要があります。 多くのニュアンスがあり、間違いを犯す場所があります。 これを覚えて、避けてください。



たとえば、SpeechKitでは、3番目のバージョンの前に、ライブラリ内でオーディオを操作し、オーディオセッションを明示的に設定およびアクティブ化しました。 iOSでのオーディオセッションは、すべてのアプリケーションが持っているものです。持っていないと言ってはいけません。 開始時に作成され、アプリケーションとシステムメディアデーモンの相互作用を担当し、アプリケーションがオーディオで何をしたいかを伝えます。 これは、単語の真の意味でのシングルトンオブジェクトです。 私たちは落ち着いてそれを必要に応じて設定しましたが、これはユーザーに音量の変更などの小さな問題があるという事実につながりました。 設定の設定を担当するオーディオセッションのもう1つの方法は、非常に長い方法です。 約200ミリ秒かかります。これは、アクティブ化または非アクティブ化の顕著な低下です。



3番目のバージョンでは、ライブラリからオーディオセッションを引き出しました。 その後、SpeechKitが統合されたすべてのサービスのほとんどすべてのユーザーは、非常に不幸だと言いました。 これで、SpeechKit用に特別に設定する必要のある種類のオーディオセッションがあることを理解するはずです。



これによる結論は次のとおりです。すべて同じように、グローバルエンティティを使用しないようにしますが、ユーザーが常に決定に満足するとは限らないという事実に備えてください。



ユーザーにとって便利なものにします



他にどのようにユーザーを支援できますか？



  • ログの追加：さまざまなレベル、動的な包含 。



最も簡単な方法は、ファイルを添付することです。ファイルがあると、メガデバッグモードが起動します。 ユーザーにエラーのあるユーザーがいて、正確に何が起こったのかを理解する必要がある状況でデバッグを行うことは本当に役立ちます。



  • すべてのユーザーOSバージョンをサポートします。



ライブラリでのバージョンサポートについて話すとき、それは通常のアプリケーションでのバージョンサポートと同じではないことに注意してください。 通常のアプリケーションでは、たとえば、ユーザーの2％のみがiOS 8を使用し、iOS 8のサポートを停止できることを意味する統計を調べます。これはライブラリーには当てはまりません。ここでは、OSバージョンを放棄すると、ユーザーとすべてのユーザーが完全に放棄されます。 これは原則としてユーザーの半分かもしれません。



したがって、ライブラリを使用するアプリケーションがどのバージョンを使用しているかを監視する必要があり、これに基づいて、何かをサポートするかどうかをすでに結論付けています。 私たちはiOS 7を長い間放棄しませんでした。iOS8を放棄し、iOS 9を放棄する準備ができている人が既にいたようです。そして私たちは彼と密接に協力し、そのような状況に彼を残すことはできませんでした。



繰り返しになりますが、ユーザーは「サポートしていないバージョンでこの機能をオフにしましょう」とは言いません。いいえ、単にライブラリを削除し、一連のバージョン全体をサポートするライブラリを見つけます。



  • 新しいバージョンで最小増分を追加します。


これはライブラリ開発者にとって非常に「それほどではない」です。 リリースの準備ができているすべてのものをリリースしたい。 機能を作成し、バグを修正しました。次に、パケット全体を入れてリリースにロールアウトします。 リリースもプロセスです。 これはユーザーにとってはそうではありません。 製品のテストとリリースの準備を行っているとき、追加のテストが必要な新機能を備えたアセンブリを受け取りたくありません。



いくつかのリリースをロールバックし、それらをパーツに分割し、すでにバラバラに展開しているケースが本当にありました。 その後、変更を実装したチームは、一度にすべてではなく、小さな変更があるバージョンを正確に使用できます。



これは実際には開発にはあまり便利ではありませんが、バージョンの最小の増分により、ユーザーは少し幸せになります。



多くのテストはありません



これは、通常のアプリケーションとライブラリの両方に当てはまります。 しかし、ライブラリの場合も機能があります。



もちろん、 自動テストが必要ですが、それらに加えて、ライブラリ用のテストアプリケーションがあることは素晴らしいことです。 自分で書いたものを統合し、どのような問題や落とし穴が生じるかを理解するのに役立ちます。 ユーザーが何であるかを自分で感じることができます。



ライブラリが何らかの方法でネットワークと相互作用し、暗号化を含む場合、少なくともデータとセキュリティに関連するものがあり、 検証のためにセキュリティガードに渡してください。 あなたは絶対に脆弱性が見つかったライブラリになりたくありません-それは人生の汚名です。 ほとんどすべての大企業には、製品の安全性のチェックを扱う部門全体があります。 持っていない場合は、外部監査があります。 外部に余裕がない場合は、ネットワークでテストを見つけて実行し、ライブラリがユーザーデータの漏洩を許可していないことを確認してください。



テストで非常に重要な最後のことは、時間、消費電力、特定のライブラリに典型的なすべてなど、 可能なすべての測定値を最初から追加しようとすることです 。 最終的にはこれを行う必要がありますので、最初から測定について考えてみてください。



これは、変更に対する保護やライブラリの高速化の必要性から保護するものではありませんが、何が間違っていたかを把握するのに役立ちます。 グラフがある場合、これは、どの機能が時間遅延を追加したか、または消費電力が増加したかをリアルタイムで監視するのに役立ちます。



これはライブラリの機能ではないため、これを行う時間はほとんどありません。これはあなたが開発するためのものではありません。 しかし、これは、良好な状態と品質でそれを維持するのに役立ちます。



ここでは、Yandexでモバイルデバイスのエネルギー消費量を測定する方法を読むことができます。 時間測定については面白い話でした。 ライブラリ開発者として、すべてのチームがすべてのSpeechKitスクリプトを使用するわけではないため、特定の場合の動作を測定することは困難です。 時間までは、テストアプリケーションを使用しました。 音声合成用のレコグナイザーまたはコンポーネントなどの特殊な使用例が記述され、ログが記録されてすべてのステップが保存され、その結果、クールなグラフィックが作成されました。



すべては問題ありませんが、オーディオを操作し、すべてをチェックするために、特定のケースではオーディオトラックが実際に再生されます。 さらに、多くの測定を行う必要があるため、彼らはテストを夜に残しました。スピーカーを配置し、何らかのデバイスを近くに配置し、オーディオファイルを開始しました。 朝、すべてがオフになり、翌夜、それが繰り返され、その後再び繰り返されました。 オフィスを歩き回ったのは魔法の生き物ではありませんでした-クリーナーだけが怖かったです。 時々、非常に奇妙なテキストが時々読まれていました。



その結果、ローカルテストベンチを作成することになりました。これをキャビネットと呼びます。 これは自然なキャビネットで、防音のみが施されています。 多くのデバイスが含まれており、ファーム全体にデバイスがあります。各デバイスは、誰も傷つけないため、就業日に何度も起動できます。

このプロセスは通常の段階で構成されていますが、ライブラリを開発する場合、これらの各ポイントには独自の特性があります。
