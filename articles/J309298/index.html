本「ES6とだけではない」

すでにJavaScriptを使用した経験がある場合でも、おそらくあなたが完全に知らない言語です。 この本は、JavaScript標準の最新バージョンであるEcmascript 6（ES6）で導入された新機能に焦点を当てています。



ES6は、ほとんどのJavaScriptプログラマには馴染みのない言語の複雑さについて語っています。 この知識を武器に、真のマスタリーを達成できます。 新しい構文を学ぶ イテレーター、ジェネレーター、モジュール、クラスを正しく使用することを学ぶ。 データをより効率的に使用できます。 Array、Object、Math、Number、Stringなどの新しいAPIをご覧ください。 メタプログラミングの助けを借りてプログラムの機能を拡張します。



目標と目的



JavaScriptで奇妙または理解できないと思われるすべてのものをブラックリストに載せる傾向がある場合（および一部はそれを行うのに慣れている場合）、ある時点でリッチ言語からの空のシェルしか得られません。



誰もがアクセスできるJavaScriptメカニズムのこのようなサブセットは、この言語の長所と考えられていますが、学習しやすい、安全な、または最小限の機能と呼ぶ方が正しいです。



私（著者）は、逆のことを行うことをお勧めします。JavaScriptを徹底的に研究して、最も複雑な機能を理解することです。 この本で議論されるのはそれらについてです。



JS開発者が特定の問題を解決するために最低限必要なことだけを学ぶ傾向があることは知っていますが、私の本では困難を避けるための一般的な推奨事項は見つかりません。



何かが私が必要とする方法で機能する場合でも、私はこの事実自体に満足する準備ができていません-それがなぜそしてどのように機能するかを理解することが重要です。 私のアプローチを共有してほしい。 私は控えめに電話して、ほとんどの人が歩いていない厄介な道を進み、JavaScriptが何であり、それがどんな機会を提供するかを完全に理解しています。 そして、あなたがこの知識を持っているとき、単一のテクニックでも、単一のプラットフォームでも、新しいアプローチでも、あなたの理解を超えています。



You Do n’t Know JSシリーズの各本は、誤解されているか表面的なことが多い言語の特定の重要な要素を深く包括的に明らかにしています。 この文献を読んだ後、あなたは理論的だけでなく実用的なニュアンスも理解しているという確固たる信念を得るでしょう。



JavaScriptの知識は、ほとんどの場合、自分自身が理解不足のtrapに陥った人から得たものです。 これは、JavaScriptが実際に何であるかを示す単なる影です。 あなたはまだ彼を本当に知らないが、あなたがJSを知らないシリーズの本はあなたが追いつくのを助けるでしょう。 さあ、JavaScript言語があなたを待っています！

バージョンのサポート



JavaScript標準は公式にはECMAScript（または略してES）と呼ばれ、最近まで、すべてのバージョンは整数のみで示されていました。 ES1とES2は、名声と大量導入を受けていません。 JavaScriptで最初に広く使用されたフレームワークは、Internet Explorer 6から8およびAndroid 2.xモバイルブラウザーのこの言語の標準であるES3でした。 私が言及しない政治的理由のために、ES4の不幸なバージョンは決して日の目を見ませんでした。



2009年にES5は正式に完成し（2011年にES5.1が登場）、Firefox、Chrome、Opera、Safariなどの多くの最新ブラウザーの標準として人気が高まっています。



JSの次のバージョン（外観は2013年から2014年、次に2015年に延期されました）は、ES6という名前で議論されています。 しかし、その後、次のバージョンの年に基づいて命名方式に切り替える提案が到着し始めました。たとえば、2016年末までに完了するES2016（別名ES7）です。 誰もがこのアプローチに同意するわけではありませんが、ES6標準がES2015という名前でユーザーに知られるようになる可能性があります。 また、ES2016バージョンの外観は、新しい命名スキームへの最終的な移行の証拠となります。



さらに、JSの進化率は1年に1つのバージョンを超えていることがわかりました。 標準の議論で新しいアイデアが生まれるとすぐに、ブラウザ開発者は新しい機能のプロトタイプを提供し、先駆者のプログラマーはコードの実験を開始します。



通常、公式の承認のかなり前に、エンジンとツールの初期のプロトタイプのおかげで、新しい機能が事実上の標準になります。 したがって、JSの将来のバージョンを、新しい機能の出現に関連するものとして検討するのは理にかなっています。基本機能の任意のセット（現在のように）や1年（計画どおり）ではありません。



この場合、バージョン番号は以前の重要性を失い、JavaScriptは常に変化する標準になります。 また、コードを「このような標準に従って作成された」と言うのではなく、サポートされている機能のサポートされている機能に応じて考慮することをお勧めします。



トランスコンパイル



機能の急速な進化は、サイトまたはアプリケーションがイノベーションをサポートしない古いブラウザーで動作する状況で新しい機能を使用したい開発者にとって深刻な問題を引き起こします。



どうやら、ES5は多くの場所で定着しなかったようです。これは、基本的に、以前のプラットフォームのすべてではないにしてもほとんどのプラットフォームのサポートが停止するまで、コードベースが新しい標準に準拠しなかったためです。 その結果、多くの開発者は、たとえば5年以上前にES5に登場した厳格モードなどを使用し始めたばかりです。



このような長期の遅延は、一般的に将来のJSエコシステムに有害であると考えられています。 言語の開発に携わる人々は、仕様が承認された直後に、開発者が新しい機能的な機能とテンプレートを考慮してコードを作成し始めることを夢見ており、ブラウザはこれをすべて実装できます。



矛盾を解決するには？ ここでは、特別なツール、特にトランスコンパイル手法2が役立ちます。 大まかに言えば、特別なツールを使用して、ES6コードをES5環境で動作する同等の（またはそれに近いもの）に変換します。



例として、短縮されたプロパティ定義を使用します（第2章の「オブジェクトリテラルの拡張」セクションを参照）。 ES6で行う方法は次のとおりです。 



var    foo = [1,2,3]; var    obj = {     foo     //  'foo: foo' };  obj.foo;     // [1,2,3]     ()  : var    foo = [1,2,3];  var    obj = {     foo: foo }; obj.foo;    // [1,2,3]




このような小さくても便利な変換により、同じ名前の場合、オブジェクトリテラルfoo：fooの宣言をfooに減らすことができます。 この場合のトランスコンパイラのアクションは、リンティング、縮小、およびその他の同様の操作に似た組み込みワークフローです。



シムライブラリ



すべての新しいES6機能にトランスコンパイラが必要なわけではありません。 Shimライブラリとも呼ばれるポリフィルは、新しい環境から古い環境への動作を定義するためのパターンです。 ポリフィリングは構文では許可されていませんが、さまざまなAPIに使用できます。



2つの値の厳密な等価性をテストするように設計された新しいObject.is（..）メソッドを見てみましょう。ただし、===演算子がNaNおよび-0に対して持つ詳細な例外はありません。 Object.is（..）メソッドのポリフィルは非常に簡単です。 



 if (!Object.is) {    Object.is = function(v1, v2) {        //    '-0'        if (v1 === 0 && v2 === 0) {            return 1 / v1 === 1 / v2;        }        //    'NaN'        if (v1 !== v1) {            return v2 !== v2;        }        //          return v1 === v2;    }; }




すべての新しいJSプロジェクトに含めるべき素晴らしいES6 Shimコレクションがあります。



JSは継続的な開発を待っており、ブラウザの新しい機能のサポートは、大きなフラグメントではなく、利用可能になった時点で徐々に実装されると想定されています。 そのため、関連性を維持するための最良の戦略は、データベースにコードを追加し、ビルドプロセスにトランスコンパイルを含め、開発者が常に変更に対応できるようにすることです。



保守的に考え、新しい機能を使用せずに動作するすべてのブラウザーが消えるまで延期する人は、常にはるかに遅れをとるでしょう。 JavaScriptコードの記述をより効率的で合理的で信頼性の高いものにするすべての革新は、それらをバイパスします。
