Strata + Hadoop 2016レビュー





昨年、BadooはHadoop + Sparkバンドルを積極的に使用し始め、Spark Streamingを使用して数千万のメトリックを収集および処理する独自のシステムを構築しました。

まず、私たちが使用している技術スタックに興味がありました。 Sparkについての講演者は、Cloudera、Databricks、Hortonworks、IBMの書籍の講演者、クリエイター、アクティブな「寄稿者」、著者の流れ全体を強調しました。



会議の主催者は、表紙に豪華な動物の本を出版するIT出版社として知られるO'Reillyと、Hadoopベースのソリューションを専門とするIT会社のClouderaです。



今年の会議の会場は、巨大なロンドンのエクセル展示ホールでした。 そのサイズは素晴らしいです。 DLR列車の2つの駅の間にあり、必要なイベントがどの部分で行われるかわからない場合は、15〜20分かけて建物の一方の端からもう一方の端まで移動できます。



イベント自体の規模も同様に野心的でした。 会議は火曜日から金曜日までの4日間続きました。 最初の2日間で、入門講義、セミナー、マスタークラスが開催され、翌日には会議自体が開催されました。 おそらく、ビッグデータを扱うことに専念する会議はそうであると思われるので、そこで提示される情報の量は膨大でした。 一連の8つのレポート（！）で始まり、その後10を超える並列セクションが開き、それぞれに6時間のレポートがありました。

Spark 2.0次は何ですか？



Apache Sparkセクションの最初は、Databricksの開発者であるTathagata Dazでした（彼はすぐに誰も彼の名前を発音できないことに気付いたので、誰もが単にTDと呼びました）。 彼のレポートでは、Apache Spark 2.0のリリースで何を期待するかについて話しました。

TDは、メジャーリリースは今年6月に予定されていると述べました。 この記事の執筆時点では、 不安定なプレビューリリースバージョンのみがすべてのユーザーが利用できます。 スピーカーはまた、リリースの「大部分」にもかかわらず、1.xとの後方互換性がほぼ完全に維持されることを保証しました。

次に、このリリースの約束された画期的な機能について直接説明します。



  1. タングステンフェーズ2 。 Tungstenプロジェクトは、Sparkフレームワークでの鉄のメモリと使用率を改善することを目的とした一連の最適化です。 更新されたバージョンでは、タングステンの作業を5〜10倍加速することを約束します。 これは、コード生成を最適化し、メモリアルゴリズムを改善することで達成されました。 以前にいくつかの連続した操作からの要求が仮想呼び出しのチェーンを必要としていた場合、今では単一のコードにコンパイルされます。

  2. 構造化ストリーミング 。 開発者から多くのフィードバックを受け取ったSparkチームは、ストリーミングモデルを大幅に作り直しました。 更新されたバージョンでは、構造化ストリーミングと呼ばれる「インタラクティブ」ストリーミングが使用されます。このストリーミングでは、既存のストリームに対してさまざまなクエリを実行し、機械学習モデルを構築するランタイムを実行できます。 本質的に、これはSQL APIの上に構築された高レベルAPIです。 これにより、ストリーミングはタングステンからすべての最適化を受け取るはずです。



  3. データセットとデータフレーム 。 新しいバージョンでは、DataFrame = Dataset [Row]オブジェクトを作成するときに、これら2つのAPIがマージされます。 これにより、DatesetのDataFrameオブジェクトでマップ、フィルターなどの操作を実行できます。 機能性データセットは実験的なものとしてマークされており、1.xバージョンとの互換性が崩れる場所の1つです。









Sparkでアプリケーションを開発している場合は、夏に新しいバージョンを準備してアップグレードするのに時間がかかることを確認してください。 APIの改善とパフォーマンスの向上は見事です。



Matei Zaharia、CTO Databricks、Sparkの作成者によるSpark Summitの非常によく似たレポートは、 Spark 2.0で見ることができます。



Sparkでのストリーミングの未来。 構造化ストリーミング



会議でのTDからの2番目のレポートは、Spark Streamingの方向性がどのように発展するかについてのより詳細な話でした。 スピーカーによると、Sparkを使用している開発者の半数以上が、Spark Streamingがシステムの最も重要なコンポーネントであると考えています。



開発者がストリーミングの存在の3年間にわたって行った主な結論：このプロセスは単独で発生するべきではありません。 ユーザーは、データストリームを取得して処理し、後で使用するためにデータベースに格納するだけでなく、 ほとんどの場合、監視をこのストリームに接続し、機械学習用のデータを収集するなども必要です。



この点で、開発者はより広く考え始め、上記のすべての可能性が追加されるフレームワーク内で、ストリーミングだけでなく継続的アプリケーションという新しいプロジェクトを呼び始めました。



TDは、現在のD-Streamsモデルの主要な問題領域を調査し、構造化ストリーミングと呼ばれる新しいソリューションを導入しました。

構造化ストリーミングは、無限のテーブルを持つ作品としてストリーミングを見ることができる新しい概念です。







このテーブルのデータは、DataFrames APIを介したSQLクエリを使用してクエリできます。 ユーザーが必要とするものに応じて、すべてのデータに対して、および受信したデータのデルタに対してのみリクエストを呼び出すことができます。

API dstreamsとDataFramesを組み合わせることにより、ストリームからのデータを静的セットと組み合わせる操作を実行できるようになりました。

このレポートでは、新しいシステムが「内部」でどのように機能するか、最も重要なこととして、フォールトトレランスがどのように達成されるかについても検討しました。



Spark Streamingの上に複雑なシステムを構築する場合、開発者によると、最も単純化されたAPIで高速で耐障害性のあるストリーミングを得ることができるため、新しいStructured Streamingコンセプトを必ず検討する必要があります。



仕様は、 構造化ストリーミングプログラミング抽象化セマンティクスとAPIにあります 。

彼女の首の金属リングのチャイムの下にある贅沢なスピーカーは、Sparkの内部について多くの興味深いことを語った。

そして、私は幸せとシールが欲しいです。 猫について言えば、報告書には猫がたくさんいました。 まあ、彼女はそれらが好きです。 重要ではなく、「mi-mi-mi」だけで、それだけです。



では、過剰なメモリ消費とパフォーマンスの低下はどこから来るのでしょうか？

  • キーによる不要なグループ化の実行。 可能であれば、groupByKeyではなくreduceByKeyを使用する必要があります。 これにより、データ量がすぐに削減されます。
  • 不均一に分散されたデータ。 1つのキーあたりのデータが他のキーよりもはるかに多い場合、シャッフルを実行すると、このデータはすべて1つのレデューサーに送られ、... OOM-killerがすぐそこにあります！ したがって、シャッフルの交換が役立つ場合があります。
  • 他のデータセットとの結合を使用する必要性。 まあ、これは一般的にすべてのmap-reduceアルゴリズムにとって災害です。データ量を減らすことは不可能であり、増やすことしかできないからです。 爆発的な増加がないことに注意する必要があります。




これはすべて理解され、説明され、解決策が提案されます。 さらなる最適化を待つ際にどのような問題が発生する可能性があるかが示されます。
