星が空になったとき：ジムワイリッヒとのインタビュー

翻訳者から：Jim WeirichはRubyコミュニティの伝説的な人格です。 残念ながら、彼は私たち全員と同じように人間です。 少し前に彼は私たちを去った。 この翻訳は、世界を少し変えたこの男へのオマージュです。


私はジムについて長い間聞いていました。結局のところ、彼は実際にルービストの間で事実上の標準になったレーキユーティリティの著者です。 私の幸運なことに、私はGoRuCo 2012での彼のスピーチのリスナーの1人で、rakeの高度な機能について話しました。 ジムが複雑なトピックについて語った敏ility性と容易さにどれほど感銘を受けたか覚えています。 その年の後半、RubyConfカンファレンスで、彼のもう1つの注目すべきプレゼンテーションを見ました。これは、ラムダ計算の基本原則のいくつかを説明しました。 ジムは、これを最も簡単なトピックではなく、簡単に説明しただけでなく、途中でリスナーを失いませんでした。



ジムからこのインタビューを受けた理由を理解できたと思います。これは、彼についてもう少し学び、何かを学ぶユニークな機会だからです。 開発世界での彼の道の始まり、Rubyでの彼の最初のステップ、関数型プログラミング、Rubyのスレッドのモデル、さらに彼のテストフレームワークであるRSpec-Givenについても話しました。 だから、このインタビューを開いて、最も先進的でカリスマ的な開発者の1人を見つけるのを怠らないでください！









プログラマーの短い要約



どのようにして開発者になりましたか？ この職業を選んだ理由は何ですか？



私は古き良き70年代に学校でプログラミングに出くわしました。 それから私は、BCDシステムの開発に捧げられた本を図書館で取りました。 アセンブラーに似た言語がプログラミング言語として使用されました。 一般的に-もちろん、アセンブラーでBCDの下で書くのは少しクレイジーですが、私はそれを理解しました。



そして、頭字語BCDはどういう意味ですか？



BCDはBinary Coded Decimalであり、すべての数値が10進表記で格納されるシステムです（従来のバイナリとは異なります）。



これは非常に珍しいマシンアーキテクチャですが、プログラミングの初心者でも受け入れられます。 私はコードを書くことを学んでいたときにどのように抜けたのかを覚えています。 職業の選択に関して、私がアドバイザーに正確に宣言したことを推測していると思います。 確かに、パンフレットで典型的なプログラマーの説明を読んだ後、私は特に落ち込んでいた。



そこに書かれたものを苦しめませんか？



さて、次のようなもの：



「...あなたは深い暗闇の中で完全に一人で座っています。 ...仕事の何時間もの間、誰ともコミュニケーションを取りません！”



その後、プログラミングは素晴らしいと思いましたが、そのような仕事は...いや、ありがとう！



私の大学時代、コンサルタントは私にアドバイスしました。「Fortran for Beginnersコースにサインアップしませんか？ 気に入るかもしれません。」 私が最初のレッスンに来たとき、講師は黒板に行き、その過程でそれを説明するコードを書き始めました。 彼が書いた最初の関数の名前-メンバー。 私は講義を聞いていて、それが私に気づきました。「これは、Fortranのある種のあまりにも奇妙な方言です。 ブラケットが多すぎます！」 そうそう、それはLispだった！ 結局のところ、私はリトル・リスパーの著者であるダン・フリードマンの最初で唯一のコースに参加しました。



まあ、パンフレットからプログラマの不吉な説明はどうですか？



ハ、プログラミングは非常に社会的な活動であることが判明しました。 今日、一人で仕事をしていても、Twitter、IRC、githubの問題、またはメーリングリストを通じて他の人といつでもチャットすることができます...私たちは1970年代に説明された唯一のオオカミではありません。



幸せな発見



どのようにしてRubyに来たのですか？



私の記憶が私に正しく役立つなら、それは2000年でした。 その後、C ++とJavaで作成し、Perlでも作成しました（日常的な作業を減らすための小さなスクリプトです）。 そして、残念ながら、しばらくしてパールは私を怒らせ始めました。それは、高速なハックに最適でしたが、美しい抽象化を書くのが難しいからです。 ここでは、たとえば、リストが必要です-パールでそれを吐き出してください、しかしリストのリストがすでに必要な場合-そして、すべて、言語はぎこちなく動作し始めます。



そして、私は自分に合った新しい言語を探し始めました。 もっと表現力のあるものが欲しかったので、抽象化をより積極的に使用できるようになりました。 また、オブジェクト指向プログラミングが大好きだったので、対応する言語も必要でした。 私はPythonを非常に慎重に研究し、少なくとも3つのアプローチを行いました。 しかし、ご存知のように、私の頭の中にはそのような「バム！」はありませんでした。 そのため、しばらくすると、Pythonの進歩が止まりました。







Pythonを学ぼうとする3回の試みのどこかで、デイブ・トーマスから手紙を受け取りました。彼は次のように書いています。「Rubyという小さな言語を見つけました。 「 The Pragmatic Programmer 」という本を読んで、デイブ自身が何かをお勧めするなら、これは少なくとも一見の価値があると断言しました。 私はRubyをダウンロードし、ほんの数分間彼と遊んだところ、突然気づいた。 これはまさに私が探していた言語です！」 3日後、Perlでの書き込みを停止し、完全にRubyに切り替えました。



それはどのバージョンのRubyでしたか？



それはRuby 1.6でした（庭で2000年夏だったので）。



Rubyはあなたに何を夢中にさせましたか？



彼は真珠と同じようにすべてをしましたが、オブジェクト指向の方法でのみでした。 「さて、私はこの問題をPerlで書きますが、オブジェクトがあれば、この方法で問題を解決します」と考えたことを覚えています。 その後、Rubyに座って、オブジェクトに基づいて構築されたソリューションが機能するようになりました！ もちろん、Pearlでできることと同じように、素早く（そして汚い）トリックを使用できましたが、しばらくすると、Rubyには優れた抽象化があり、言語自体が非常に正しく開発されていることがわかりました。 私は彼に会ったことだけで幸せでした。



Rubyは機能しますか？



それでは、Rubyはオブジェクト指向言語です。 しかし、同時に機能しますか？ そして、「機能的」という言葉はどういう意味ですか？



これは本当に良い質問です。 今日、関数型言語はますます注目を集めています。たとえば、今夜、シンシナティ市で関数型言語のプログラマーとミーティングを行います。 この決定に対する私の妻は、「まあ、大丈夫、そして明日、あなたは単なる人間の開発者と会っているのですか？」と言いました。



一般に、言語を機能させるには2つの要件があります。最初の要件は、関数を最初のクラスのオブジェクトとして操作することです。ある場所から別の場所への関数の転送、変数との関連付け、クロージャーを使用した新しい関数の作成などです。 Rubyは最初の要件を満たします。ラムダとクロージャがあり、それらを変数にバインドし、動的に新しい関数を作成することもできます。 そのため、こちら側から見ると、Rubyは非常に機能的な言語です。



しかし、2番目の要件があります！ 関数型言語は非常に興味深いものになります。 それは、エンティティの状態を変更することの容認できないことにあります。 Rubyはオブジェクトの状態を常に変化させるため、この要件を満たしていません！



関数型プログラミングは、そのマルチスレッド実装にとって興味深いものです。 マルチスレッドプログラミングの重要な問題の1つは、たとえば、同じデータを処理する2つのスレッドがある場合の競合状態です。 このレースでは、誰が勝つかわかりません。 たとえば、カウンターの増加。 2つのスレッドがあり、最初のスレッドが現在の値を読み取り、メモリレジスタ内の値をインクリメントしてから戻します。 この時点で、2番目のストリームは値を読み取り、値を増やして書き込みます。 それらが厳密な順序で次々に交互になり、10から始まる場合、11、12などになります。 スレッド共有リソースが変更される可能性がある場合は、非常に注意する必要があります。



この問題を解決するには、2つの方法があります。共有リソースを使用しないか、この同じリソースを変更しないでください。 関数型プログラミングは2番目のアプローチを使用します。データは変更されず、作成されるだけです。 このアプローチにより、マルチスレッドプログラミングが著しく簡単になります。 私たちの技術社会では、新しい鉄/プロセッサのそれぞれで、マルチスレッドの必要性が高まるだけだと思います。 また、Clojureのような言語には、スレッド操作を容易にするための非常に優れた構造が既に組み込まれています。 Clojureでは、データを変更しても、安全なときに非常に特殊な条件でそれを行います。



フローの彼のモデルは1960年から1970年に由来するため、Rubyはこれに苦しんでいるのではないかと心配しています。



Rubyスレッドモデル



はい、特にMRI Ruby。 個人的には、インタープリター自体はスレッドセーフではないと考えています。 これは本当にそうですか？



インタープリターはスレッドセーフですが、これはGIL（Global Interpreter Lock）の助けを借りて達成され、開発者はRubyConfでそれについて多くのことを話しました。 このアプローチの本質は、インタープリターに入ると、他のスレッドが同時にインタープリターに入ることができないようにロックが設定されることです。 もちろん、これはすべてスレッドの安全性につながりますが、ある程度は、スレッド自体の作業を複雑にします。







しかし、JRubyとRubiniusはスレッドセーフですか？



はい！ 下位レベルでロックを課します。 そのようなことがあります。最初に必要なレベルでアーキテクチャにロックインすれば、最終的にこれを行うのは簡単です。 既存の言語にスレッドセーフを追加することは、MRI開発者が対処しなければならない本当の問題です。 彼らは最後にGILを追加しましたが、それは少し表面的な解決策です。



Matzは、マイクロロック（マイクロロック）、つまり共有データにロックをかけるという別のアプローチをテストしたと述べました。 また、この場合、コードはGILを使用したバージョンよりもはるかに遅く動作し始めたと述べました。 ここにそのようなソリューションの価格があります。 したがって、スレッドを使用する必要がある場合は、JRubyまたはRubiniusを詳しく調べる必要があります。 しかし、それらを使用する場合でも、共有リソースは変更される可能性があることを覚えておく必要があります。Rubyはこの問題に対する優れたソリューションをまだ提供していないためです。

私は、新しいモダンなアイデアがしばしば研究と過去からの遠い日々の基盤に基づいているという事実が好きです。 例えば、RubyConfでのプレゼンテーションで話した関数型プログラミングやLambda計算のアイデア。



これらは80歳のものです！



あなたはまだあなたの袖にそのようなものを持っていますか？



2009年、Michael Feathersは「すべてのプログラマが最低2回読むべき10の論文」という記事を書きました（ 残念ながら、元の記事はもう利用できません ）。



最初の本はDavid Parnanによって書かれ、 「システムをモジュールに分解する際に使用される基準について」と題されています 。 彼は、簡単なプログラム、コンテキストのキーワードを書きました。







プログラムはテキストファイルを実行し、すべての単語を見つけ、これらの単語が見つかったテキスト領域を見つけ、見つかったテキストブロックを記録して、単語が後でそれらが置かれているテキストとともにディレクトリに表示されるようにします。 それは実験でした。 Davidは異なるアプローチを使用して2つのプログラムを作成し、これらのプログラムに新しい要件が加えられたときに何が起こるかを調べました。 実験中に、プログラムを書く最良の方法は、モジュールに分割することであることがわかりました。各モジュールは内部コンテンツを隠します。



これはオブジェクト指向プログラミング（OOP）のアイデアではありませんか？



はい、それらのうちの1つ、OOPでは、各オブジェクトは情報の一部と、世界が内部で何が起こるかを気にかけないものを格納します。 したがって、保管方法および内部部品との相互作用の方法を変更しても、他の部品には影響しません。 そして、この非常に単純な実験は、何をする必要があるかを示しました。 この記事全体を読むことを強くお勧めします 。 デビッドが実際に実験を行ったのが気に入っています。彼はコードを書いて、新しい条件下での動作を確認しました。



もう1つのクールな記事は、ジョンバッカスが書いた「プログラミングをフォンノイマンスタイルから解放できますか？ 」です。 これは彼が「ML」、つまり関数型プログラミング言語と呼んだ最初の出版物の1つです。 MLは、HaskellやOCalmなどの言語の基礎となっています。



私が共有したいもう一つの素晴らしい研究は、ケン・トンプソンのTrusting Trustに関するReflectionsです。 彼は、ソースコードにないバグをUnixに追加できるコンパイラのハックについて説明しました。

あなたには自然な才能があります。あなたは非常に複雑なトピックを説明することができます。 どうやってやるの？



実は、パフォーマンスに関する本をたくさん読んでいます。適切な準備方法、実際のパフォーマンス方法などです。 すべての本で、メモや他の同様のものをとることをお勧めします-しかし、このアプローチは私にはうまくいきません！



パフォーマンスの準備をするとき、Mac OS用のOmnigraffleのようなホワイトボードまたはユーティリティを使用するのが好きです。 アイデアが書かれている小さな長方形から始めます。 その後、対応するアイデアを線でつなぎます。 たとえば、パフォーマンスがRubyに関連している場合は、話したいことをすべて書き留めます。 その後、「マインドマッピング」のように、それらを接続します。 確かに、有向グラフを取得します。



その後、紙の上で面白いストーリーを作って、得られたグラフに沿って歩きます。 もちろん、必ずしもアイデアが整理される順番ではありません。



つまり、テクニカルライターのようなものですか？



はい、私はその説明が好きです！



ラムダ計算のような難しいトピックを興味深いストーリーに変えることは芸術です。 80年前に書かれたこのトピックに関する元の科学的研究を読もうとすると、多くの人が目を失ったと思います。



パフォーマンスを成功させる秘keyは、自分で目を燃やすことだと思います。 あなたが何を言っているのかがあなた自身にとって何よりもまず面白いなら、それは聴衆の残りの部分に興味をひくことが判明します。 勉強中に教授にどれほどひどく単調な声で会ったかご存知ですか？ これは静かな恐怖です。 この忘れられない経験の後、私が演じるなら、講師のようには決してやらないと決めました。 パフォーマンスのためのこれらすべての準備-これはもちろん良いことであり、助けにはなりますが、話すときはあなたの目の炎がより重要です。



Rspecで指定



もう演奏する予定はありますか？



1月に、私の頭脳について、 Given / When / Thenフレームワークについてお話します。 これは私のお気に入りの子です。テストを書くたびに、フレームワークを使用します。これは、私の考えを非常にエレガントに表現しているからです。 それについてはCodeMashで説明します。



このフレームワークには、BDDの哲学と共通点がありますか？ キュウリはどうですか？ または、同じ3つのキーワードを使用しているという事実は単なる偶然ですか？



これは単なる偶然であり、私のフレームワークはCucumberとは関係ありませんが、Given / When / Thenを使用したアプローチでは本当に気に入っています。



Rspec-Givenを使用すると、次のように簡単に指定できます。これは与えられたもので、テストするコードを以下に示します。これはテストが合格するために当てはまります。 私の意見では、これはテストを書く素晴らしい方法です。 Test :: Unit：データが与えられたときでも、同様のアプローチを使用しました。ここにテストコードがあり、ここにテスト自体があります。 しかし、Test :: UnitもRspecも必要なメソッドを提供していないため、よりエレガントなソリューションを見つけようとしました。



約2〜3年前、私はこの問題を振り返り、アイデアを紙に書きました。 当時、私はナッシュビルで開催されたRuby Hoedownカンファレンスに参加していました。 この紙切れを近所の開発者に渡して、フィードバックと意見を集めました。 ジョーオブライエンは私に向き直り、「別のテストフレームワークを書くつもりはないのですか？」と言いました。



Rspecで非常にシンプルで非常に機能する実装に出会うまで、私は1〜2年間アイデアを試しました。 Rspecが、私が望んでいたものを実装するのに十分な柔軟性と抽象性を備えているのは素晴らしいことです。 その結果、Rspecの上に小さなライブラリが作成されました。 したがって、テストをより明確にすることで、Rspecで記述している間に得た知識を使用できます。
