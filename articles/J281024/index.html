DevOpsの展開方法：GitHubおよびVisual Studio Team Services Build Systemとの継続的な統合





みなさんこんにちは！ Vorlon.JSチームによる一連の記事「DevOpsのデプロイ方法」を継続します。

Vorlon.JSはnode.jsベースのツールで、Web開発者が、特にモバイルシステムや組み込みシステムで、Webアプリケーションをリモートでテスト、制御、デバッグする便利な方法を提供します。 MSDNのブログで、チームはVorlon.JSの作業を整理する際のDevOpsプラクティスの段階的な実装と、日常のタスクを解決するためのツールの選択について詳しく説明しました。 Vorlon.JSはオープンソースプロジェクトです。
マイクロソフトは、DevOpsプラクティスの実装に使用できる多数のツールを提供しています。このシリーズでは、Visual Studio Team Services、Visual Studio Code、およびMicrosoft Azureとの連携について詳しく説明します。 もちろん、Microsoft Azureにアプリケーションをデプロイする場合を含め、マイクロソフト以外の多くのツールを使用する可能性についても触れます。



サイクルコンテンツ：

  • Visual Studio Team Servicesを使用したプロジェクトの計画と管理
  • GitHubおよびVisual Studio Team Servicesビルドシステムとの継続的な統合
  • Microsoft AzureおよびAzure Resource Managerを使用したASAフレームワーク
  • Visual Studio Team Servicesのリリース管理
  • Microsoft Azure App ServiceとWeb Appsを使用した実稼働環境でのテスト


Visual Studio Team Services （VSTS）がGITまたはTFVSC（Team Foundationバージョンソース管理）をサポートするソース管理ツールを提供していることを既にご存知かもしれません。



この記事では、Visual Studio Team Servicesを使用して、GitHubでホストされるコードの継続的統合をいかに簡単に実装できるかを詳細に説明します。



シリーズの前の記事に注意してください 。VSTSアカウントをまだお持ちでない場合は、無料で入手する方法を詳しく説明しています。



GitHubをVisual Studio Team Servicesに接続する



GitHubでの作業を開始するには、まず、VSTSで外部サービスを構成するためのページでアカウントを接続する必要があります。



ウィンドウの右上隅にある6をクリックして、チームのプロジェクト設定に移動し、[サービス]タブを選択します。 [新しいサービスエンドポイント]ボタンをクリックすると、GitHubを含むサービスに新しい接続を追加できます。







[認証]ボタンをクリックしてGitHubアカウントに接続し、GitHubアカウント情報を認証に使用します。



アセンブリ定義の作成



GitHubアカウントを設定したら、チームプロジェクトに戻り、「ビルド」を押してビルド定義にアクセスします。



新しいビルドサービスは、さまざまなビルドエージェント（Linux、Windowsなど）を実行し、MSBuild、M​​aven、Antなどの多数のビルドタスクを処理し、Androidアプリケーションのパッケージを作成し、NPMまたはGulpタスクを実行できるオープンシステムです。など



実際、ビルドシステムは完全にオープンであり、ボックスから提供されたものが目的に適さない場合は、独自のタスクを作成することができます。 既存のタスクはすべて、GitHubリポジトリで公開されています 。 ここでは、アプリケーションパッケージを作成して展開するタスク（Azureリソースマネージャー、Webアプリ、Docker、Tom Catなど）も紹介しますが、VSTSでのリリース管理について説明する記事の後半でそれらについて説明します。



一般に、新しいアセンブリ定義を作成するには、いくつかの手順を実行するだけで済みます。



組み立てプロセスを形成するためのタスクを定義する



これは最も重要なステップであり、構築するソリューションによっては時間がかかる場合があります。 Vorlon.JSの場合、次のものが必要でした。

  • アプリケーションのすべての依存関係を追加するためにNPMインストールを実行し、TypeScript、LessなどからJavaScriptおよびCSSファイルを生成するデフォルトのGulpタスクを実行します。

  • Mocha.JSで単体テストを実行する「テスト」 Gulpタスクを起動する

  • アプリケーションを含むzipアーカイブを作成する「zip」Gulpタスクを起動する

  • アセンブリアーティファクト（前の手順で作成されたzipファイルと、次の記事で説明する他の展開スクリプト）を公開します。 これらのアーティファクトは、アプリケーションをさまざまな環境（開発、QA、運用前、運用など）に公開するためにリリース管理サービスによって使用されます。



[ビルド]タブで、左側のプラス記号をクリックして、新しいビルド定義を作成します。 既存のテンプレートまたは空の定義のいずれかを選択できます。







この場合、空の定義を選択しました。 [次へ]をクリックして[設定]ページにアクセスします。ここで、新しいアセンブリに関連付けられるソースコードを格納するリポジトリのタイプ、アセンブリ開始時にそれらを使用するエージェントキュー、および継続的統合を使用する場合は、コミットが行われるたびにアセンブリを開始するには、対応する設定を確認する必要があります。







使用できるビルドエージェントには2つのタイプがあります。

  • ホストされたビルドエージェント -Visual Studio Team Servicesのサービスとして提供され、数分で使用料を支払います

  • プライベートビルドエージェント -独自の仮想マシンで構成できるエージェント 。WindowsまたはLinuxで実行できます。



このステップでは、ビルドプロセスに使用するすべてのツールでネイティブにサポートされているHostedエージェントを使用できます。



[作成]をクリックして、アセンブリ定義を作成します。 アセンブリ定義コンストラクターにリダイレクトされます。これにより、アセンブリー中に実行するプロセスを決定できます。 アセンブリは、次々に実行されるステップで構成されます。



[ビルドステップの追加]をクリックします。 プロセスに追加するさまざまなステップを選択できます。







Vorlon.JSビルド用に定義したプロセスは次のとおりです。







ご覧のとおり、各ステップは右側の領域で構成できます。 たとえば、リポジトリが複製された後、最初の「NPMインストール」タスクが実行されます。 次に、2つのgulpタスクが実行されます。1つ目はMocha.JSフレームワークを使用してGulpでユニットテストを実行し、2つ目はすべてのコードとルートモジュールを含むzipアーカイブを生成します。 後者のタスクは、リリース管理段階でこれらのアーティファクトを使用してアプリケーションを展開できるようにするために、アセンブリアーティファクトをアセンブリエージェントに公開するために使用されます。







リポジトリ定義



ビルドプロセスに必要なすべての手順を追加したら、[リポジトリ]タブに戻り、このビルド定義で使用するソースコードリポジトリを構成します。 Vorlon.JSの場合、GitHubリポジトリを使用しました。







アセンブリを開始します



このプロジェクトでは、GitHubリポジトリ内のソースコードに変更が加えられたらすぐに新しいビルドを開始したかったのです。 これを設定するには、スクリーンショットに示すように、「トリガー」タブに移動して、継続的インテグレーションを選択する必要があります。







さまざまなリポジトリブランチのスケジュールビルドを開始することもできます。たとえば、夜間ビルドを開始できます。







パネルの「保存」をクリックします。 初めて、ポップアップでアセンブリの名前を指定するように求められます。



次のステップでは、コードを変更し、リポジトリにコミットして、ビルドプロセスが開始されているかどうか、すべてが正常に機能しているかどうかを確認します。 ここではGitHubを使用しますが、アセンブリ定義を作成するために実行されるすべてのステップは、VSTS（GITまたはTFVSC）の他のプロバイダーの参加でも実行できるため、GitHubの使用はオプションです！

この場合、Visual Studio Codeツールを使用します。これはGITとのネイティブ統合を提供するため、プログラミングが非常に簡単になり、すぐに変更をGitHubに送信します。



ビルド結果と出力を確認する



最新のアセンブリにアクセスするには、左側のアセンブリ定義をクリックします。 キュー内のアセンブリと完了したアセンブリを表示することを選択できます。

アセンブリが完了すると、VSTSポータルのログに直接アクセスして、すべてのアセンブリ手順が正しく完了したかどうかを確認できます。

ポータルからアセンブリ成果物をダウンロードすることもできます。そのため、プロセス全体でアセンブリエージェントに公開されたものを確認できます。 これらのアーティファクトは、今後のリリース管理の記事で使用します。







しかし、リリース管理のトピックに取り掛かる前に、シリーズの次の記事では、「インフラストラクチャとしてのコード」アプローチとは何か、それが環境の作成とアプリケーションのデプロイのプロセスを改善する方法を理解します。
