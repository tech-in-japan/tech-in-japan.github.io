3つの 命名規則





意味のある名前を選択することは、読みやすいコードを書く上で重要な部分です。 命名規則を一貫して使用すると、コードの可読性が向上し、人がプログラムの構造をすばやく理解できるようになります。 別の要件は、名前の競合のリスクを最小限に抑えるための順序を確立することです。 このセクションでは、読みやすさ、一貫性を促進し、大規模開発をサポートするために設計された命名規則について説明します。 以下のドキュメントでは、タイプはクラス、列挙型、またはタイプ定義です。



3. 1ライブラリプレフィックス



すべてのファイルとその内容（クラス、関数など）は特定のライブラリに属します。







r-nam-libpx：各ライブラリにプレフィックスを割り当てます。

各ライブラリは、このライブラリの一意の2文字または3文字のプレフィックスに関連付ける必要があります。 プレフィックス文字は小文字でなければなりません。



r-nam-glob：すべてのグローバル名にライブラリプレフィックスを使用します。

ライブラリ内のオブジェクトがグローバルネームスペースの一部である場合、その名前はライブラリプレフィックスで始まる必要があります。







名前をプレフィックスで構成する方法に関する規則は、次のセクションで説明します。 著者またはサポート担当者の責任は、ライブラリ内で名前の競合がないことを確認することです。 プレフィックスは、ライブラリの外部での名前の競合を（できれば！）防ぎます。

3. 2つのスタイルの推奨事項







s-nam-abbrev：略語の使用頻度を減らします。

略語が広く知られている場合にのみ（会社内または世界で）単語を省略します。

たとえば、「DDX」は、Delcamで理解される略語です。 ループ変数「i」は「標準」プログラマのショートカットであり、「array_index」よりもはるかに優れています。 読みにくい名前は悪いと見なされます。



s-nam-length：名前の長さを無理にしないでください。



一部の環境では、認識される文字の数に制限があるため、名前を長くしすぎないでください。 たとえば、unix 'ar'コマンドは、15文字より長いファイル名を切り捨てます。



s-nam-desc：単純でわかりやすい名前を使用します。

変数の名前は、その使用について説明する必要があります。 変数名に1つ、2つの単語が含まれている場合が最適です。 3の場合、これはすでに注意を引き、4の場合、名前は読みにくい場合があります。 2単語の名前を作成するときは、最も重要な単語を最初に置きます。



s-nam-dif：名前は大幅に異なる必要があります。

1、2文字だけ異なる変数名を使用しないでください。 文字の場合にのみ異なるタイプ名を使用しないでください。 同時に、同様の機能を実行する変数には同様の名前を使用します。





s-nam-std：標準ライブラリで使用される名前の使用を避けます。

マクロ定義はルールによって制限されないため、これはローカルオブジェクトにも推奨されます（標準ライブラリ内のオブジェクトにプレフィックスがある場合、問題は軽減されます）。 標準ライブラリでは非常に一般的な（したがって広く使用されている）単語（たとえば、「インデックス」）が使用されることがあるため、これはルールではなく推奨事項です。





3.4ファイル名





ファイルには4つのタイプがあります。



  1. ヘッダーファイルには型定義、関数プロトタイプなどが含まれます。ソースファイルにはクラス実装などが含まれます。インラインファイルにはインライン関数が含まれます。
  2. アウトラインファイルにより、インライン関数の非インラインバージョンがデバッグ目的で使用できるようになります。






ファイルの命名規則を以下に示します。





r-nam-clfile：クラス実装を含むファイルには、クラスと同じ名前が必要です。

ファイル名は、クラス名と同じ大文字と小文字で記述する必要があります。 これは、メインクラスにも適用されます。 このルールは、ヘッダー、インライン、およびソースファイルに適用されます。





r-nam-utfile：一連のユーティリティを実装するファイルには、ライブラリプレフィックスが前に付いた小文字の名前が必要です。

プレフィックスは、アンダースコア1つで名前と区切る必要があります。 このルールは、ヘッダー、インライン、およびソースファイルに適用されます。



r-nam-enum：列挙型を1つだけ含むヘッダーファイルには、列挙型と同じ名前を付けることができます。



r-nam-ext：ファイル拡張子：ヘッダー.h、ソーステキスト/アウトライン.c、インライン.i。



Ellemtelは、CコンパイラとC ++コンパイラの両方で受け入れ可能なファイルに対して、「。hh」、「。cc」、「。icc」を予約し、「。h」と「.c」を予約します。 すべてのDelcamコードは、ルールに従ってC ++をコンパイルする必要があるため、このような区別は行いません。



r-nam-outline：アウトラインファイルには、対応するヘッダーと同じ名前が付けられ、「ファイル拡張子」の前に「I」が追加されます。



r-nam-split：ソーステキストファイルが大きくなり、分割する必要がある場合、新しい各ファイルの名前は、古いファイルの名前にアンダースコアと小文字の単語が続いたものになります。

たとえば、zzClass.cファイルに、別のファイルに実装されているメンバー関数zzClass :: member（）が含まれている場合、そのファイルはzzClass_member.cと呼ばれる場合があります。





例：zzClass.c：zzClassクラスを実装するソースファイル。 zz_utils.h：ユーティリティヘッダーファイル。 zz_utils.i：zz_utils.hで定義されているインライン関数の実装。

zzClassI.c：zzClass.hで定義されているインライン関数のアウトラインバージョン。















4 コード形式とテキスト形式



このセクションでは、コードの外観を定義します。

4. 1一般的なコード形式







r-fmt-brace：有効なブラケットスタイルのみを使用します。

ブラケットは、K＆Rの「短い」形式に従って配置する必要があります。開始ブラケットは、前のコードの直後の行末にあります。 閉じかっこは、開きかっこが配置されている行の最初の非空白文字に対応する列の別の行にあります。 2つの角括弧の間のコードはインデントされています。

例：





if（function_returns_true（））{

execute_this_code（）;

}







例外：関数の開始ブラケット（{）は別の行になければなりません。

例外：関数の引数は別の行を占める場合があります。



r-fmt-lines：文字列は79文字を超えてはなりません。

この制限は、私たちが使用している標準の「emacs」設定が長い行が折りたたまれているウィンドウを作成するという事実のために選択されました。 さらに、80x24（80x25）は古い端末からの一般的なウィンドウサイズであり、最小化された行は読みにくくなります。



r-fmt-indent：インデントは2つのスペースです。



ロジックの新しいレベルごとに1レベル戻ります。

例外：要素が整列している場合、if-elseシーケンスは読みやすくなります-r-if-elseifを参照してください。



r-fmt-tabs：タブ文字を使用しないでください。

コードのサポート中にインデントを維持するのが難しくなります。 他のエキゾチックなキャラクターの使用は避けてください。



s-fmt-sep：不要なセパレータを使用しないでください。



例：列挙の最後の値の後に余分なコンマを使用しないでください（「strict」コンパイラーは、IBMなどのエラーをスローします）。





4. 2オペレーター







s-fmt-long：非常に長いステートメントの使用を避けます。

代わりに、いくつかの短いステートメントを使用してください。



s-fmt-stat：各ステートメントを別々の行に配置します。

これを行わない場合、プログラムの明瞭さが改善されることを確認する必要があります。



s-fmt-col：同じ行に複数のステートメントがある場合は、ステートメントを列にグループ化します。





s-fmt-split：改行するときは、もう1レベル戻ります。

1行を複数の行に分割する必要がある場合は、最初の行を除くすべての行を1レベル追加します。 例外：s-fmt-splitexprを参照してください。

例外：管理構造は、対応するセクション（§6）で説明されているように分割されます。





4. 3つの式







s-fmt-splitexpr：式を分割するとき、同じ論理レベルの部分式は垂直に整列する必要があります。

ネストレベルが高い場所で式を壊さないでください。 算術演算子と論理演算子は、次の行の先頭ではなく、行の末尾に配置してください。 これにより、長い式のロジックを簡単にトレースできます。

例：





結果=（（（x1 + 1）*（x1 + 1））-

（（y1 + 1）*（y1 + 1）））;





例：割り当て式は次のように分解できます。







結果= a_long *斜辺+

2.0 * cos（アルファ）;





壊れた式を整列させる場合、サブ式のいずれかまたは両方を括弧で囲む必要があります。

例：





if（（a && b）||

c）{//これをしない



次のように書く方が良いです。







if（（a && b）||

©）{//はるかに良い







s-fmt-prec：操作を明確にするために、より頻繁に括弧を使用します。

覚えていない限り、テーブルの優先順位を探しないでください。代わりに括弧を使用してください。 操作の優先順位を思い出せない場合、コードを読み取った人もそれを覚えていない可能性が高くなります。 原則として、操作+、/と+、-の間の優先順位。 他のすべての場合には括弧を使用します。





4. 4つのスペース





これらのガイドラインは、インデントに使用されるスペースではなく、行内のスペースに適用されます。



  1. 複数のスペースを使用しないでください。

    （たとえば、 'class <class>'で検索すると、クラス<clas s>の定義が見つかります ）。
  2. ポインター変数と「[」の間にスペースを使用しないでください（たとえば、「array [」で検索すると、配列の要素への参照が見つかるはずです）。 関数名と '（'の間にスペースを使用しないでください（たとえば、 'func（'で検索すると、すべての関数呼び出しが見つかります）。常に '（'と制御ステートメントの間にスペースを入力します（たとえば、 'while（x）'を使用し、 「while（x）」ではありません）常に「{」と「}」の前にスペースを入力します（行の先頭にある場合を除く）。
  3. 常に「{」と「}」の後にスペースを入力します（行末にある場合を除く）。
  4. 文字 '、;）]'の前にスペースを入れ ないでください 。
  5. '（'または ']'の後にスペースを入力しないでください。定義と式のコンマの後にスペースを配置することをお勧めしますが、関数呼び出しの引数の間にスペースを配置することは推奨されません。単項演算とオペランドの間にスペースを入力します。「::」、「。」、「->」の前後にスペースを使用しないでくださいキーワード「演算子」の後にスペースを入力しないでください。
  6. 上記の推奨事項を除き、演算子の前後のスペースはオプションです。






s-fmt-space：スペースの使用に関するガイドラインに従ってください。
