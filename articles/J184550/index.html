完璧主義者の哲学者の観点から、これはひどいです。 サーバー側でデータを整理する数百の方法がすでに考案されている場合（MVCアプローチ、DBMSは基本的に情報を整理するための方法論です）、次のステップは、クライアント側でデータを整理することです 。



Backbone.jsは何を提供できますか？ 私の意見では、最初の最も重要なことは、データの順序付けです。これは、使い慣れたMVCを介してBackbone.jsに実装されます。 それがどうなるかについて簡単に：



1.アプリケーションにモデルがあります。 ユーザー、製品、注文...すばらしい。Backbone.jsを使用すると、これらのモデルをクライアント側で説明できます。 そして、これらのモデルは本物です。新しいインスタンスを作成し、既存のインスタンスのフィールドを編集し、不要なインスタンスを削除できます。 後でスローされる情報を取得したり、サーバーから注文クラスのインスタンスを取得したりするための関数を作成することはできません。 注文には多くの不要な情報が含まれている可能性があります。 たぶん。 ただし、クライアント側で必要な順序フィールドのみをJSONで送信することを禁止する人はいません。 そして、あなたの仮想マネージャーは、注文数に加えて、それぞれの合計に加えて、突然、表示したいかもしれません-そして、私たちはすでにすべてを行っています、情報はすでにそこにあります、それを表示するだけで、オブジェクトはすでにクライアント上にあり、サーバーと同期しています。 そして、マネージャーがその場でこの金額を変更したい場合はどうなりますか？ 問題ありません。 新しい自家製の関数を記述するよりも、コードにorder.save（{：cost => this.input.val（}）を記述する方がはるかに優れています。モデルは、GET、POST、PUT、およびDELETEリクエストを送信するURLを知っており、たとえば、必要な場合、product.delete（）はDELETEを/ api / v1 / products / 75に送信します（75はこのモデルのIDです）。



2.多くの同様のブロックが必要なアプリケーションのページがあります。 製品検索結果は一例です。 各製品は、div class = 'product'内に配置できます。 サーバー上でこのページを作成した後、このエリアと記載されている製品との接続は失われます。 このページで何かをしたい場合は、このエリアに関連する製品を再度知らせる方法を考案する必要があります。たとえば、製品IDでデータ属性を指定し、フォームやリクエストで使用します。Backbone.jsはそのようなバインディング自体を実装します。 これは「ビュー」と呼ばれ、さらに多くのことが行われます。たとえば、製品エリアは製品の変更に応答して、それ自体を再描画できます。 この領域内では、Productクラスのインスタンス（まさにそのクラス）にアクセスでき、クライアント側で何でも実行できます。 Backbone.jsがサーバーへの変更を保存するように、後で保存することを忘れないでください。



ところで。 製品テーブルのいくつかのフィールドでフィルタリングしたいですか？ これは追加のアクションを必要としないことをすでに推測しましたか？ サーバーに何かを送信する必要はありません。同じテーブルを取得し、再度フィルタリングするだけです。 すでにオブジェクトがあります。1行のJavaScriptコードで製品のコレクションをフィルタリングするだけです。 そして、はい、彼らも自分自身を再描画します。製品の変更時にこの製品に関する情報があるページの一部を更新することをBackboneに伝える必要があるのは一度だけです。



3.また、データの整理のトピックでも。 クライアント側のMVCフレームワークを使用すると、場合によっては、クライアントとサーバー間のトラフィックを減らしてjsonの小さな断片を交換することができ、これにより大量のトラフィックを節約できます。 顧客に送信し、顧客からのデータのみを受信します。顧客にページ全体を常に送信するわけではありません。 すべてのjavascriptとcssを一度ロードしてキャッシュすることができます。そうすると、好きなように注文されたデータの交換のみが行われます。
