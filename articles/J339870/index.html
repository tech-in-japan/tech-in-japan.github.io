DBMSインターフェイスをすばやく作成するためのソリューションを見つける

ほとんどすべての人は、Excelのテーブルの使用からクライアントバンキングアプリケーションでのデータの操作まで、あらゆる種類の会計、データ収集、分析に直面しています。 さまざまなデータベース管理システム（DBMS）などの会計に広く使用されています。







この記事では、このようなシステムを見つける方法についてお話したいと思います。







ユーザーの観点から、類似のDBMS（クライアントバンク、FSZNシステム、現金決済アプリケーション）に直面して、私は常に、十分な財務能力を備えた大企業でさえ、アプリケーションに本当に便利なインターフェイスを実装できないことが多いのではないかといつも思っていました。







明らかに、アプリケーションインターフェイスが便利であるためには、データに対する一連の標準操作の綿密な実装に基づいて開発する必要があります。 このような基盤がないと、ソフトウェア製品の品質に悪影響を及ぼすか、ソフトウェア製品の開発時間が大幅に長くなります。







1C：エンタープライズ



2003年に、あらゆる種類のアカウンティングのアプリケーションが必要になったときに、1Cの構成を作成しました。













当時、このアプローチではすべてがうまくいきました。 しかし、しばらくして、会計システムへのネットワークアクセスの必要性に直面しました。 サーバーパーツ1Cの価格を分析した後、私にとっては価格が高すぎることに気づき、代替手段、できれば無料のオープンソースを探し始めました。







Django管理者



レビューを分析および調査した後、管理インターフェイスジェネレーターを備えたDjangoフレームワークを選択しました。 同時に、1Cに埋め込まれたアイデアの一部をPythonコードに転送する必要がありました。 結果は、ほぼ次のインターフェイスになりました。













ただし、大量のデータでは、このアプローチの欠点が現れ始めました。







まず、Django Adminインターフェースの機能は非常に限られているため、Djangoで提供されていない機能が必要な場合、作成の利便性が無駄になります。 この場合、Djangoコードを調べて、コードを埋め込み、クラスを継承し、テンプレートをやり直すことができる場所を探す必要があります。 このような変更は本質的に非体系的であり、しばらくしてからそれらに戻って動作を覚えるのは困難です（特に、Djangoの次のバージョンを更新した後に行う必要があります）。







第二に、Django Adminによって作成されたインターフェイスを操作することは十分に便利ではなく、データ入力は困難で効率的ではありませんでした。 データが送信されるたびにインターフェースがページをオーバーロードせず、AjaxやWebSocketなどのテクノロジーを使用するように、少なくとも1Cレベルでの対話性が必要でした。







MySQL、Navicatなど



その結果、Django Adminモジュールの使用を完全に放棄し、上記の技術を使用してJavaScriptでクライアント部分を自分で記述する必要がありました。 したがって、対話性の問題は解決されましたが、インターフェースの作成にかかった時間は不当に長くなりました。 時として、時間を節約するために、Navicatの形式のクライアント部分で純粋なMysqlを使用してデータを収集および分析しました。 判明したように、トリガーとビューのおかげで、これは悪い解決策ではなく、膨大な数のタスクがこの方法で非常に簡単に解決されます（Wikipediaによると、Mysqlはそのような問題を解決するために最初に作成されたためです）。







最適なDBMS開発ツールの基準



時間が経つにつれて、DBMSを開発するためのツールに対する主観的な希望のリストを自分で作成しました。 彼は：







  • モデルに基づいてデータを操作するためのインターフェイスを迅速かつ自動的に作成できます。
  • オブジェクト同士の相互作用のメカニズムだけでなく、これらの相互作用をインターフェースに表示するメカニズムも提案します。
  • 選択肢を提供する：インターフェイスまたはその一部をより高速に（ただし柔軟性は低く）またはより機能的に（ただし低速に）開発します。
  • さまざまなデータソースを操作します。
  • ウェブ技術に基づいて、クロスプラットフォームであり、オープンソースを持っています。


これらの要件を満たすものは見つかりませんでした。 したがって、タスクを解決する独自のフレームワークを作成することにしました。 TypeScriptとAngular2が主要なテクノロジーとして選択されました。 同時に、フレームワークは、Angularを使用するだけでなく、他のJavaScriptライブラリを使用するアプリケーションの開発にも使用できるように設計されました。 私は何が起こったかを共有したいと思います：多分誰かが役に立つでしょう。







新しいフレームワークの構造



フレームワークは、DBMSのインターフェイスの迅速な作成に焦点を当てています。 いくつかの部分（モジュール）で構成されています。 一部は個別に使用でき、一部は他とのみ使用できます。







コアモジュールには、モデルを記述するメカニズム、データオブジェクト（レコード）の相互作用、およびデータベースクエリを記述するメカニズムが含まれています。 コアモジュールは、バックエンドモジュールを介してデータソースにアクセスします。







バックエンドモジュールは、コアモジュールとデータのデータベース（ソース）の間のレイヤーです。 データソースは、SQLなどのデータベースサーバー自体、またはDjangoやSequelizeなどの他のフレームワークのモデルにアクセスするためのレイヤーのいずれかです。







model-uiモジュールは、インターフェースの生成を担当します。uiモジュールが提供するコントロールを使用して、コアモジュールが提供するデータを視覚化します。







uiモジュールには、model-uiモジュールがインターフェイスを生成するために使用する基本的なコントロールが含まれています。 これらの要素は、フレームワークから独立して使用することもできます。







windows-managerモジュールは、ユーザーインターフェイスを表示するためのコンテナーを管理します。 Windowsマネージャーの種類に応じて、アプリケーションはコンピューターとモバイルデバイスの両方に展開できます。







モデルとデータ操作



フレームワークの機能は、同じDjangoとは異なり、レコードのリストとレコード自体を記述するために異なるタイプのモデルが使用されることです： ListModelとRecordModel 。 このアプローチにより、1つのレコードだけでなく、異なるモデルのレコード、および編集不可能な行（たとえば、これらのレコードに対する作業の結果）のリストにレコードを表示できます。







フレームワークにはこれに必要なメカニズムが含まれていますが、アプリケーション開発中にモデルを記述する必要はありません。 バックエンドモジュールは、既に存在し、他の環境（Django、Sequelize、SQLなど）で記述されているものに基づいて内部モデルを自動的に生成します。







このフレームワークは、Djangoでの作業と類似しています。 たとえば、選択とクエリ（ QuerySetとQuery ）を操作するためのクラスは、PythonコードからTypeScriptコードに適応した同じ名前のDjangoクラスと同等です。 たとえば、データソースからデータを取得するには、およそ次のコードを記述する必要があります。 







backend .getListModel('product') .getQueryset() .filter({name__icontains: ''}) .orderBy('-price') .limit(0,10) .getRows() .subscribe(products => {…});


フレームワークのもう1つの特徴的な機能は、仮想フィールドのサポートです。 仮想フィールドが変更されると、オブジェクトの実フィールドが変更され、実フィールドの値が変更されると、仮想フィールドが変更されます。 Djangoにも同様のことがあります。オブジェクトを介して、このオブジェクトで利用可能な形式でデータベースに保存されていないデータにアクセスする場合-xxxxxx_setプロパティを介してオブジェクトを参照する他のオブジェクトを取得するか、このオブジェクトのIDのみがデータベースに保存されている場合のプロパティを介したオブジェクト。







次の図では、 product_idフィールドとproduct_nameフィールドは実在し、 productフィールドは仮想です。













フレームワークは「依存レコードの遅延読み込み」を実装します。 Djangoとは異なり、ここで開発者はこの場合、このメカニズムを適用せずに、すぐにデータを受信するほうがよいため、クライアントとサーバー間のリクエストの数を減らすことができます。 したがって、上記の例では、製品productにはサプライヤーを参照するサプライヤーフィールドがあります。 デフォルトでは、 製品['supplier']フィールドにアクセスする場合にのみ、データベースからサプライヤが要求されます。 ただし、上記の例を次のように変更した場合： ... getRows（ 'supplier'）。Subscribe（products => {...}）; -製品リストの各製品にはすでにサプライヤに関するデータが含まれており、それらにアクセスするとき、データベースへのクエリはありません。







インターフェース



アプリケーション開発を加速するために、このフレームワークでは、model-uiおよびuiモジュールに基づいた組み込みインターフェースを使用できます。 これは、特定のインターフェイスが不要な場合、またはアプリケーションをできるだけ迅速に展開する必要がある場合に便利です。 ただし、これらのモジュールを放棄すると、任意のインターフェイスを使用できます（たとえば、BootstrapまたはAngular Materialに基づいて作成されます）。







標準インターフェイスのすべてのデータ入力コントロールは、フォームとテーブルセルの両方で使用できるように作成されています。 要素には、データの操作を簡素化するプロパティも備わっています。 たとえば、テキストフィールドは次のようになります。













拡張ボタンをクリックすると、拡大された入力フィールドが開きます。これは、入力フィールドの幅がそこに配置されたテキストよりも小さい場合に便利です。 このような状況は、要素がフォーム上に密集して配置されている場合、またはテーブルの形式でデータを編集している場合に発生する可能性があります。







数値の入力フィールドを検討する場合は、数値自体に加えて、その中に数式を入力できます。













別のオブジェクトの選択フィールドには、いくつかの入力メソッドもあります。単語の出現に応じてドロップダウンリストから、または別のポップアップフォームからオブジェクトを選択できます。













誤った入力のエラーについては、コントロール下のテキストではなく、特別なマーカーで表示されます。 このアプローチにより、フォーム要素の密度を維持し、エラーの導出においてその位置を変更することはできません。 このコントロールを使用したテーブルセルのエラーも同様に表示されます。













データへのインターフェース



フレームワークでは、レコード（オブジェクト）の相互作用に多くの注意が払われます。







フォーム上のレコードを変更すると、リスト内のレコードが自動的に更新されます。













レコードが異なるコンピューターから同時に編集された場合、ユーザーは警告を受け取ります。













フレームワークインターフェイスは、依存レコードの行をソートするためのメカニズムも実装します（モデルとバックエンドモジュールによって提供される場合）。













指定されたインターフェイス要素の例は、フレームワークに埋め込まれているもののほんの一部です。 インターフェイスのすべての要素と原則のより詳細な説明は、個々の記事のトピックです。 また、インターフェイスの指定された例が課されるのではなく、基礎としてのみ提供され、開発者の裁量でカスタマイズできることを繰り返す価値があります。







今後の計画



フレームワークには、私が見たかった主な機能が含まれています。 それは完全に動作しますが、それでも、現在の段階では、概念のみと呼びます。







そのため、たとえば、現時点では、フレームワークがMySQLデータベースでのみ動作することを許可するバックエンドがあります。そのため、Electronでのみ実行できます。 また、モバイルデバイスで動作するためのインターフェイスやその他の多くの機能は実装されていません。







フレームワークの開発のための最も近い計画：







  • Queryクラスのクエリに結合およびグループ化メカニズムを実装します。
  • 関連付けおよびグループ化を操作するためのコントロールを追加します。
  • Queryオブジェクトをjsonまたはxmlに変換するバックエンドを開発し、Djangoモデルを操作するためのサーバー側を開発します。
  • データサーバーへの要求をキャッシュするためのメカニズムを実装します。
  • 他の多くのアイデアを実現します。


どうやって試す？



作業中のフレームワークを試してみたい場合は、フレームワークに記述されたアプリケーションの例を見ることができます。 次のようにインストールできます。 







 git clone https://github.com/astoniocom/astonio-demo.git


次に、依存関係をインストールする必要があります。 







 npm install


次に、app.module.tsファイルを開いて、MySQLデータベースへのアクセスを構成し、アプリケーションをコンパイルします。 







 npm run watch


次のコマンドで実行します： 







 npm run app


このフレームワークはgithubで入手できます。
