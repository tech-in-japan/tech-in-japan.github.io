IDAO。 第一段階



タスクは、ミューオン/非ミューオン粒子をそれらの特性に従って分類することでした。 このタスクの重要な特徴は、トレーニングデータに重み列が存在することでした。主催者自身がこの列の回答に対する自信として解釈しました。 問題は、かなりの数の行に負の重みが含まれていたことです。













ヒントを使って数分間考えて（ヒントは重み列のこの機能に注目を集めました）、このグラフを作成した後、3つのオプションをチェックすることにしました。







1）負の重み（およびそれぞれ重み）を持つ行のターゲットを反転します

2）重みを最小値にシフトして、0から開始するようにします

3）行に重みを使用しない







3番目のオプションは最悪でしたが、最初の2つは結果を改善し、最良のオプションは1番目のオプションで、すぐに1番目のタスクで現在の2位に、2番目に1番目になりました。



次のステップは、欠損値のデータを調べることでした。 オーガナイザーは、すでにいくつかの欠損値が存在するコーミングデータを提供し、それらは-9999に置き換えられました。







MatchedHit_ {X、Y、Z} [N]およびMatchedHit_D {X、Y、Z} [N]列に欠損値が見つかりました。N= 2または3の場合のみです。理解したように、一部の粒子は4つの検出器すべてを通過しませんでした、3または4プレートで停止しました。 データにはLextra_ {X、Y} [N]列も含まれており、MatchedHit_ {X、Y、Z} [N]と同じようですが、何らかの外挿を使用しています。 これらのわずかな推測は、MatchedHit_ {X、Y、Z} [N]の欠損値の代わりに、Lextra_ {X、Y} [N]（XおよびY座標のみ）に置き換えることができることを示唆しました。 MatchedHit_Z [N]は中央値で十分に満たされていました。 これらの操作により、両方のタスクについて1つの中間の場所に移動できました。













最初のステージでの勝利のために彼らは何も与えなかったことを考えると、私たちはこれでやめることができましたが、私たちは続けていくつかの美しい写真を描き、新しい機能を思いつきました。













たとえば、検出器の4つのプレートのそれぞれから粒子の交差点を作成すると、各プレート上の点が4対5のアスペクト比と（0,0）の中心を持つ5つの長方形にグループ化されることがわかります。最初の長方形にはポイントがありません。

これらのサイズを決定したら、粒子ごとに4つの新しいカテゴリフィーチャ（各プレートと交差する長方形の数）を追加しました。













また、粒子が中心から離れて散乱しているように見え、この散乱の「品質」を何らかの形で評価するというアイデアが生まれました。 理想的には、おそらく、エントリポイントに応じて何らかの「理想的な」放物線を考え出し、それからの逸脱を推定することができますが、「理想的な」線に限定しました。 各エントリポイントにこのような理想的なラインを構築したので、このラインからの各粒子の軌跡の平均二乗偏差を計算することができました。 ターゲット= 1の平均偏差は152であり、ターゲット= 0の平均偏差は390であるため、この機能を暫定的に良好と評価しました。 確かに、この機能はすぐに最も有用なもののトップに当たりました。







私たちは喜んで、追加の4つの特徴として理想的なラインからの各粒子の4つの交点すべての偏差を追加しました（そして、それらもうまく機能しました）。







主催者から提供されたコンテストのトピックに関する科学記事へのリンクは、この問題を解決するのが最初とはほど遠いことを示唆しており、おそらくいくつかの特殊なソフトウェアがあります。 IsMuonSimple、IsMuon、IsMuonLooseメソッドが実装されているgithubでリポジトリを発見したので、それらをわずかな変更を加えて自分自身に転送しました。 方法自体は非常に簡単でした。たとえば、エネルギーがしきい値を下回る場合、これはミューオンではなく、ミューオンではありません。 このような単純な兆候は、明らかに勾配ブースティングを使用した場合に増加を与えることができなかったため、しきい値に別の兆候「距離」を追加しました。 これらの機能も少し改善されました。 おそらく、既存のメソッドをより徹底的に分析して、より強力なメソッドを見つけて属性に追加することができます。







競争の終わりに向かって、私たちは2番目のタスクに少し「クイック」ソリューションを傷つけました。その結果、次の点でベースラインとは異なりました。







  1. 負の重みを持つ行では、ターゲットが反転しました

SNA Hackathon 2019-テキスト。 第一段階



タスクは、Odnoklassnikiソーシャルネットワーク上のユーザーの投稿を、そこに含まれるテキストに従ってランク付けすることでした。テキストに加えて、投稿のいくつかの特性（言語、所有者、作成日時、表示日時）がまだありました。







テキストを操作する古典的なアプローチとして、次の2つのオプションを選びます。







  1. 同様の単語が同様のベクトルを持つように各単語をn次元のベクトル空間にマッピングし（詳細は記事で確認できます）、テキストの中間の単語を見つけるか、単語の相対位置を考慮したメカニズム（CNN、LSTM / GRU）を使用します。
  2. 文全体をすぐに処理できるモデルを使用します。 たとえば、Bert。 理論的には、このアプローチはうまく機能するはずです。


これは私のテキストの最初の経験だったので、誰かを教えるのは間違っているでしょう、それで私は自分自身を教えます。 コンテストの最初に自分自身に与えるいくつかのヒントを以下に示します。







  1. 何かを学ぶために走る前に、データを見てください！ テキスト自体に加えて、データにはいくつかの列があり、それらよりも多くの列を絞り出すことができました。 最も簡単なことは、列の一部に対してターゲットエンコーディングを行うことです。
  2. すべてのデータから学習しないでください！ 大量のデータ（約1700万行）があり、それらすべてを使用して仮説をテストすることは完全にオプションでした。 トレーニングと前処理は非常に遅かったため、より興味深い仮説をテストする時間があることは明らかです。
  3. < 物議を醸すアドバイス >キラーモデルを探す必要はありません。 私はエルモとバートに長い間対処し、すぐに私を高い場所に導くことを望み、その結果、ロシア語にFastTextの事前訓練された埋め込みを使用しました。 Elmoでは、より良い速度を達成することはできませんでしたが、Bertではそれを理解することができませんでした。
  4. < 物議を醸すアドバイス > 1つのキラー機能を探してはいけません。 データを見ると、テキストの1％の領域には実際にはテキストが含まれていないことに気付きました！ しかし、その後、いくつかのリソースへのリンクがあり、サイトを開いて名前と説明を引き出す簡単なパーサーを作成しました。 それは良い考えのように思えますが、私は夢中になり、すべてのテキストのすべてのリンクを解析することに決め、再び多くの時間を失いました。 これらはすべて、最終結果に大きな改善を与えるものではありませんでした（たとえば、ステミングで把握しましたが）。
  5. クラシック機能が動作します。 たとえば、Googleは「text features kaggle」で、すべてを読んで追加します。 TF-IDFは、テキストの長さ、単語、句読点の量などの統計機能も改善しました。
  6. DateTime列がある場合、それらをいくつかの個別の機能（時間、曜日など）に解析する必要があります。 どの機能を強調するかは、グラフ/いくつかの指標で分析する必要があります。 ここでは、すべてを思いつきで行い、必要な機能を強調しましたが、通常の分析では問題はありません（たとえば、最終版で行ったように）。
