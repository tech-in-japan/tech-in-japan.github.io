それでは、XMLの利点は何ですか？ 簡単に解析できますか？ しかし、コレクターファイルのパーサーをどのくらいの頻度で記述しますか？ XML configを記述し、NAntにフィードするだけです。 自分で何かを解析する必要はありません。 コンパイルは必要ありませんか？ さて、プロジェクト自体をビルドする前にビルドスクリプトをビルドしたくないので、最初の議論に値します。 そうでない場合は、ビルドスクリプト用のビルドスクリプト用のビルドスクリプトを記述する必要があります...など。 「ニワトリ、満足の卵」（ translate.google.comによる問題「ニワトリ<->卵」の翻訳に満足しています。約Transl。 ） 結局のところ、これは一部の言語の制限です。 コンパイルを必要としない言語はどうでしょうか。NAntやMSBuildと同様に、解釈された言語へのリンクを実行可能ファイルに渡すことで実行できますか？ あなたは私がリードしていることを理解しています。 このように機能する言語は多数あり、そのうちのいくつかは.Netプラットフォームで実行されます。

レーキを実行



Rakeは上記で説明したツールです。 XMLは使用しません。 これは、完全にRubyで記述された単なるスクリプトです。 インタープリター言語で記述する場合、理論的にはビルドの問題を解決するためにXMLを使用する必要はありません。 コードを書くだけです 。 IronRubyには独自のバージョンのRakeが付属しています。すべてが正しくセットアップされていることを確認するには、コマンドラインで「rake」と入力するだけです。 数秒後に、単一のrakeファイルが見つからなかったことを示すメッセージが表示されます。 それでは、プロジェクトとそのための簡単なrakeファイルを作成しましょう。

最初に行うことは、プロジェクトのディレクトリを作成することです（c：\ development \ RakeTestとします）。 指定したディレクトリで、Rakefile.rbファイルを作成します。 次に、お気に入りのテキストエディター（ たとえば、NotePad ++、約Transl。 ）で開き、次のコードを挿入します。 

 require 'rake' task :default => [:congratulate_me] desc "Tells you that you are awesome" task :congratulate_me do puts "Congrats, you have rake running. Wasn't that easy?" end


その後、コマンドラインでこのファイルがあるディレクトリに移動し、「rake」と入力するだけです。 応答として、「おめでとう、rakeを実行しています。 簡単ではなかったですか？」

そして、それは本当に簡単ですよね？ これで、完全なコレクターを自由に使用できます。 上記のコードでは、Rakeライブラリの接続、「congratulate_me」タスク起動のデフォルトのインストール、およびタスク自体の定義とその説明を確認できます。 単純なナントスクリプトのように見えますが、ハリネズミの猫であるため、コードを作成しました。

次に、タスクを追加して1つのタスクに結合するか、既定のタスクからではなく他のタスクからアセンブリを開始するのにかかるコストを見てみましょう。 そのため、スクリプトにいくつかのタスクを追加します。 

 task :congratulate_team do puts "Congrats to the team!" end task :congratulate_everyone => [:congratulate_me,:congratulate_team] do puts "Phew, that was a lot of congrats" end


物事がもう少し複雑になったことがわかります。 コンソールに文字列をポップするだけのタスクと、名前の後に何らかの配列を持つ別のタスクがあります。 これは、特定のタスクの依存関係を他のタスクに割り当てることができるNAnt機能の類似物です。 これにより、このタスクを完了する前に他のユーザーを実行できます。 ここで、Rakefile.rbでデフォルトタスクの名前を「congratulate_everyone」に変更すると、ビルドスクリプトの実行時に、他の2つのタスクが正常に完了した後にcongratulate_everyoneタスクが実行されることがわかります。 ところで、デフォルトのタスクを指定する必要はありません。 特定のタスクを名前で呼び出すことができます。 これは、「rake congratulate_team」のように行われます。

NAntまたはMSBuildフレームワークでできることの多くは既にエミュレートしています。 これは、これらのフレームワークを使用すると、特定のアクションを実行するために複数のタスクをシーケンス内で単純に結合できるためです。 また、スクリプトには一連のタスクがあり、特定の順序でタスクを実行できます。 そして、特別な設定は必要ありません。 スクリプトで変数を宣言するか、どこからでもデータを読み取ることができます。 さらに、他のルビーファイルをスクリプトに接続して、さまざまな場所から設定、コード、タスクにアクセスできます。 XMLでできることの多くを、無料で実行できたようです。

残っている問題は1つだけです-彼らが知っている方法と同じことをすることですよね？ 結局のところ、スタジオソリューションの構築、テストの実行、バージョン管理システムの活用、ファイルのコピーと圧縮、FTPへのファイルのアップロードなどの機会が欲しいのです。 たぶん私たちは何かを見逃しましたか？ それでは、MSbuildとNAntに追いつくことができるでしょうか？ まあそれは尋ねた。



Friends Rakeおよび.Net



不思議に思われるかもしれませんが、これらのスクリプトタスクをすべて実行するにはどうすればよいでしょうか。 偶然、私はそれらをすべて手で打つ必要がありますか？ この問題はすでに解決できますか？ そして、私があなたに答えるだろうと思います！ もちろん、この問題はすでに解決されています。 頭痛薬は「 アルバコア 」と呼ばれます。 これはGitHubの小さなプロジェクトです。 急いでインストールしないでください。 これを2つの方法で行う方法を説明します。

IronRubyには、特別なigemツール（MRI Rubyのgemの類似物）が付属しています。 このツールを使用すると、追加のIronRubyパッケージをマシンにインストールできます。 このツール自体は、パッケージをリポジトリからマシンに依存関係とともにコピーします。 そのため、1つのigem install albacoreコマンドで、Albacoreが機能するために必要なすべてがコンソールにインストールされます。 rake-fileに1行だけ追加する必要があります。 

 require 'albacore'


次に、コンソール.Netアプリケーション「RakeTestApp」用の単純なプロジェクトを作成し、Albacoreを介したアセンブリがその例でどのように機能するかを確認します。 「AutomatedRelease」などの名前のプロジェクトに構成を追加することから始め、ソリューションの各プロジェクトでアセンブリの作業ディレクトリを「../build_output」に変更します。 このようにして、Albacoreが自由に使える特別なmsbuildタスクを実行して、.slnファイルを処理し、ソリューションをコンパイルできます。

最初に、残りのすべてを順番に実行するタスクを作成します。 

 task :full => [:clean,:build_release,:run_tests]


次に、アセンブリ「build_output」の作業ディレクトリを削除するタスクを作成します。 クリーニングを行います。 

 task :clean do FileUtils.rm_rf 'build_output' end


ご覧のとおり、ファイルを削除するだけの特別なタスクは必要ありません。 必要な機能は、標準のRubyライブラリに既に実装されています。 次に、Albacoreタスクから継承した最初のタスクを作成します。 このタスクはソリューションをコンパイルします： 

 msbuild :build_release do |msb| msb.properties :configuration => :AutomatedRelease msb.targets :Build msb.solution = "RakeTestApp/RakeTestApp.sln" end


そのため、「task」の代わりに「msbuild」を使用し、タスクに「build_release」という名前を付けました。 このタスクは1つのパラメーター-構成を取ります。 ここでは、「AutomatedRelease」で構成を設定し、収集する必要のある目的の目標とソリューション記述子へのパスを指定します。 また、アセンブリ中に実行する必要があるテストもあります。 AlbacoreのNUnitタスクからタスクを継承できます。 次のようになります。 

 nunit :run_tests do |nunit| nunit.path_to_command = "tools/nunit/nunit-console.exe" nunit.assemblies "build_output/RakeTestAppTests.dll" end


繰り返しになりますが、このタスクは1つの引数を取ります。この引数を使用して、タスクのいくつかのパラメーターを設定できます。 ここでは、実行可能ファイルとテスト付きアセンブリのリストを指定します（区切り文字-コンマ）。



今、すべて一緒に 



 require 'rake' require 'albacore' task :default => [:full] task :full => [:clean,:build_release,:run_tests] task :clean do FileUtils.rm_rf 'build_output' end msbuild :build_release do |msb| msb.properties :configuration => :AutomatedRelease msb.targets :Build msb.solution = "RakeTestApp/RakeTestApp.sln" end nunit :run_tests do |nunit| nunit.path_to_command = "tools/nunit/nunit-console.exe" nunit.assemblies "build_output/RakeTestAppTests.dll" end




以上です！ IronRuby、Rake、およびAlbacoreを使用して簡単なビルドツールを作成しました。 しかし、これはAlbacoreでできることのほんの一部です。 このライブラリを使用すると、ndepend、ncover、sftp、sqlコマンド、zipを使用したディレクトリのアーカイブ、sshの呼び出し、unzipなどを使用できます。 これらのレゴアクションはすべてAlbacoreで実行されます。



おわりに



私がXML-configsを使用してアセンブリにうんざりしている場合は、IronRuby、Rake、およびAlbacoreを試してみることをお勧めします。 最悪のシナリオでは、これはわずかな時間になります。 しかし、その後、燃え尽きた場合、XML構成のタタール語-モンゴル語イグテ抑圧から永遠に自由になります。 楽しんでいただけましたでしょうか！
