Windows 1.0 SDK







Windowsの最初のバージョンのSDKは、7つの5インチフロッピーディスクで配布されました。



ウィキペディアが Charles Petzoldを引用しているように、「Windows 1.0 SDKのHello Worldのオリジナルバージョンは少しスキャンダラスでした。 HELLO.Cファイルの長さは約150行で、HELLO.RCリソーススクリプトはさらに20行でした...ベテランのプログラマは、Windows向けのhello-worldプログラムに直面すると、しばしば恐怖に顔をしかめたり、逆に笑ったりします。



10年前、チャールズはそれぞれSDKの20周年を想起し、 20年後に新しいプログラムを作成しようとする感想と試みについての歴史愛好家向けの非常においしい記事を書きました。



当時、IBM、Digital Research、およびMicrosoftは、それぞれのシステムのSDKをリリースしました。 彼らは約500ドルの費用がかかり、それらはすべて非常に複雑でしたが、正直なところ、Windowsは最も際立っていました。 それは完璧ではありませんでしたが、MicrosoftはWindowsアプリケーションの作成に投資するために、当時のように独立したソフトウェア開発者を本当に望んでいたように見えました。


7枚のディスケットに配置されました。



  • ユーティリティ-インストールおよび実行可能ファイルを含む2枚のフロッピーディスク

    • EXEHDR、IMPLIB、LIB、LINK4、MAKE、MAPSYM、RC、RCPP、SYMDEB、WINSTUB
    • ダイアログ、フォント編集、ヘップウォーク、アイコン編集、シェーカー、SLAPJR


  • 80Kbの有名なWindows.hを含むライブラリとインクルードファイル！ -フロッピーディスク2枚
  • デバッグ用の文字ファイルとライブラリ-1枚のディスケット
  • コード例-ソースに13個のアプリケーションを含む2枚のフロッピーディスク：

    • こんにちは-ウィンドウの作成、
    • TYPE-キーボード入力とテキスト表示、
    • 形状-メニューの操作とウィンドウ内の図形の描画、
    • トラック-マウスを使用する（マウスが使用され始めたばかりで、大衆ユーザーには馴染みがなかったことを覚えていますか？）、
    • FONTTEST-ダイアログボックスの作成とユーザー入力の処理、フォントの操作、
    • TEMPLATE-他のすべての人の基盤としての空のアプリケーション
    • MAPMODES-位置とサイズを変更するさまざまなデータ表示モード、

    • CARDFILE-Windowsのファイルキャビネットを備えたアプリケーションの類似物で、フルスクリーンモードで動作します。
    • MUZZLEはpascalのアプリケーションで、ウィンドウを使用した作業の作成を示します。


最後の例を除くすべての例はCで記述されており、アセンブラーに挿入される場合もあります。



ユーティリティには、「Microsoft Cコンパイラ3.0の特別バージョンと、合計約1000ページの2つの分厚いマニュアルが含まれており、Windowsアプリケーションを作成するために必要なすべてが記載されています。」



コスト（$ 500）について話すと、Charlesは80Kヘッダーファイルが最も価値があり、当時驚異的だったと書いています。 実際、彼はWindowsを操作するためのプログラムインターフェイス（API）を定義しました。これには、多数の新しいデータ型、定数、構造、およびアプリケーションで使用可能なすべての関数の説明（合計約400）が含まれます。 そして、すべてがマニュアルに文書化されていました。

Windows 10 SDK



30年以上にわたり、ソフトウェアシステムは大きく進歩しました。 内部コードの複雑さと量が劇的に増加しました。 最新のオペレーティングシステムのインストーラは、ギガバイトの情報を含むDVD-ROM全体を占有できます。 同程度の開発ツールとSDKと必要なすべてのライブラリの重量を量ります。



たとえば、 Windows 10 SDKには以下が含まれます。

•ヘッダーファイル、ライブラリ、メタデータ（バージョン管理！）

•関連するプラットフォームファイルとのWindowsコントラクト（Windowsランタイムのメタデータ）

•Windowsプラットフォームの拡張機能-Windowsデスクトップ、Windowsモバイル、Windows IoT、およびWindows PPIのSDK。

•WindowsApp.lib-ユニバーサルアプリケーション用の汎用ライブラリ

•ユナイテッドCRT

•Windowsアプリ認定キット

•DirectXアップデート

•Windowsパフォーマンスツールキット

•.NET Framework 4.6 SDK

•Windowsアクセシビリティツール

•サンプルコードとドキュメントへのリンク



長年にわたって、アプリケーションのコンパイルとコンパイルのための比較的単純なユーティリティは、アプリケーションを効率的に記述し、コードを操作し、デバッグし、チームワークを提供するための便利なVisual Studio環境に変わりました。







フロッピーディスクは、主にオンライン配信に置き換えられました。 たとえば、SDKをVisual Studioとは別にダウンロードする場合、最初にこれら7つのフロッピーディスクの半分の重量の小さなブートローダーをダウンロードし、次に約700 MBのインストールファイルをダウンロードするように求められます。 Windows 10 Mobileのエミュレーターはさらに1.5GBです。



APIが進化するにつれて、すべてのドキュメントが妥当な量の紙の量に収まらないことがすぐに明らかになりました。 Windowsのドキュメントが別のCD / DVDで提供されていた時代がありましたが、今ではインターネット経由で、したがってMSDN Webサイトから入手できます。 さらに、ドキュメントの多くのセクションは、ロシア語を含む広範な言語に翻訳されています。



サンプルコードは、 MSDNコードギャラリー、または今日の開発者コミュニティで慣例となっているGitHubにあります 。 今日、多くのITコミュニティで見られる興味深い点は、関連する例の形成へのコミュニティの参加と、ドキュメントの修正と推奨事項の導入です。



誰かがそのような変化を30年前に予測できたのだろうか？



Windows 10：Hello World



私は、新しいUWPプラットフォームとJavaScript言語（私に最も近いもの）を使用して、Windows 10用の「Hello World」アプリケーションの一種のアナログを作成しようとすることにしました。 一般に、これは非常に興味深い点です。30年前に純粋なCが主要言語であり、ほとんどすべての例が記述されていた場合、今日ではC / C ++だけでなく、C＃、VB、 JavaScript（およびObjCでもUWP向けに記述できます！）。



最初に目を引くのは、ファイルのコレクションから、非常に多くのアイコンを含むプロジェクトへの移行です。また、リソースファイルとアプリケーションコードを含むファイルに加えて、アプリケーション自体とその設定（マニフェスト）の個別の大きな説明もあります。







また、プロジェクト自体の設定でアセンブリ手順が「説明」されていることにも注意してください（メイクファイルの一種）。 また、30年前とは異なり、はるかに複雑になっています。 C ++でコードを記述している場合、競合しない16ビットではなく、x86、x64、またはARMのアセンブリを選択する必要がありました。もちろん、F5またはShift + F5を押すことで通常慎重に隠される他の多くのパラメーターは言うまでもありません

ページレイアウトファイルdefault.html



アプリケーションはコードから「アセンブル」することができますが、インターフェースの重要な部分をより単純でより具体的な宣言的な方法で説明できるようになりました。 さらに、次のものを含むデータバインディングをすぐに接続できます。

（自動）データバインディングは、モダンデザインの巨大なバンです。



アプリケーションロジックファイルdefault.js



単純なアプリケーションの基本的なロジックは、1つのファイルに集中させることができます（30年前のようなより複雑な例では、さまざまな種類のモジュールを分解し始める可能性が高いでしょう）。 



 (function () { "use strict"; var app = WinJS.Application; var activation = Windows.ApplicationModel.Activation; var appView = Windows.UI.ViewManagement.ApplicationView.getForCurrentView(); // Called on app activation app.onactivated = function (args) { // Initiate resources and UI -- all joined into a chain of promises var promise = new WinJS.Promise.join([ WinJS.Resources.processAll(), WinJS.UI.processAll() // Add basic logic ]).then(function () { // Update window title appView.title = WinJS.Resources.getString('idstitle').value; // Add click listener for menu "about" item document.getElementById("aboutMenu").addEventListener("click", (e) => showAboutDialog()); }); args.setPromise(promise); }; // Show"About" MessageDialog function showAboutDialog() { var messageDialog = new Windows.UI.Popups.MessageDialog( WinJS.Resources.getString('aboutbox.text').value, WinJS.Resources.getString('idsabout').value); // Dummy event handler for commands in dialog box var aboutHandler = (command) => { return true; }; messageDialog.commands.append( new Windows.UI.Popups.UICommand(WinJS.Resources.getString('aboutbox.button').value, aboutHandler)); messageDialog.showAsync(); } app.start(); })();


アプリケーションコードは4倍短くなりました（いくつかのコメントと空白行も追加しました）が、技術開発の最も重要な利点は、APIの表現力が大幅に向上したことです。 実際、「ペン」で行われていたことの多くは、システムまたはライブラリ関数への便利な呼び出しの後ろに隠れています。



矢印関数のような構文糖とpromiseを使用する機能により、コードの記述がより簡単で読みやすくなります。 ところで、重要な変更は非同期コードモデルへの移行です。したがって、約束と非同期です。







もちろん、私の「30年後の更新」は逐語的ではありませんが、アプリケーションを作成するアプローチの変更の全体的なダイナミクスは非常に追跡可能だと思います。 今日では、これは完全に新しいAPIのように見えますが、古いWin32チューブや同じWindows.hの耳は時々スキップします。
