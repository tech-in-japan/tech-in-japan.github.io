TypeScriptアプリケーションでのメモリリークの診断と修復

はじめに



最近、かなり複雑な高度なUIを使用した大規模なプロジェクトを完了しました。 詳細を説明することなく、ブラウザ内にデスクトップのようなものがウィンドウ、オーバーラップ、およびそのすべてが実装されているとしましょう。 もちろん、メモリリークの問題は解決しませんでした。 率直に言って、当分の間、私たちはビジネス結果を得ることに焦点を合わせました。 手がメモリリークに達すると、ブラウザウィンドウがギガバイトのRAMを占有していることが発見されました。 私たちはエラーを分類し、一般にそれらの除去へのアプローチを策定しました。 このアプローチを皆さんと共有したいと思います。



クライアントアプリケーションのメモリリークのトピックについては、すでに多くのことが書かれています。 当初、主な問題はブラウザIE8およびそれ以前のバージョンでした（例：

http://habrahabr.ru/post/141451/

http://habrahabr.ru/post/146784/

https://learn.javascript.ru/memory-leaks ）。

しかし、今、IE8が過去にあると言うことができるとき、問題は残ります。 TypeScriptなどの言語を使用しても、それらが存在しないことは保証されません。 また、Webアプリケーションのフロントエンドがより複雑になっているという事実を考えると、問題の関連性は高まるだけです。

1. jQuery-ウィジェット



現代生活ではjQueryライブラリーを放棄する傾向がありますが、多くの場合、jQueryライブラリーなしでは実行できません。 百万の強力なウィジェットの軍隊は大きな役割を果たし、その多くは非常に重要で便利な機能を実装しています。

ウィジェットを使用する一般的な方法は、ラッパーオブジェクトでウィジェットを「ラップ」することです。次に例を示します。 

 export function toCheckbox( $element: JQuery, options: CheckboxOptions ) : JQuery { return $element.jqxCheckBox(options); }


問題は、ウィジェットの構築後、jQueryセレクター（$要素）への参照がラッパーオブジェクトに格納されることです。 このようなリンクは、オブジェクトと対応するDOM要素の間に循環依存関係を作成できます。 したがって、このようなリンクは、メモリクリーニングメカニズムがトリガーされたときに「クリーンアップ」する必要があります。 このケースでは、jQueryセレクターとjQueryセレクターリンクの「ゼロ化」のために呼び出される特別なnullify関数がこれに使用されます。 

  //  -c jQuery- export function toCheckbox( beDisposed: ed.Disposables, $element: JQuery, options: CheckboxOptions ) : JQuery { ed.append(beDisposed, function disposeJqxCheckbox() { if ($element != null) { let instance = $element.jqxCheckBox('getInstance'); //   jQuery- $element.jqxCheckBox('destroy'); // ''  jQuery- ed.nullify(instance, 1); // ''  $element = null; options = null; beDisposed = null; } }); return $element.jqxCheckBox(options); }


したがって、ウィジェットのすべての可能性を排除し、そのDOM表現は「クリーンアップされていない」ままです。



2.カスタムノックアウトバインディング



一般に、DOM要素が表示される場合は常に、リークを慎重に監視する必要があります。 Konckoutの主な目的はモデルとビューを関連付けることなので、DOM要素を慎重に管理することが重要です。



次の例では、オブジェクトはメモリ内で「ハング」したままです。 イベント処理をDOM要素にバインドします。 

 init: function (element: HTMLElement, …) { let beDisposed = xko.toBeDisposed(element); $(document).on('keypress', 'input,textarea,select', function (e) { $(element).on('keypress', 'input, textarea, select', function (e) { ... }); }); }


したがって、それに応じてそれらをほどく必要があります。 

 init: function (element: HTMLElement, …) { let beDisposed = xko.toBeDisposed(element); $(document).on('keypress', 'input,textarea,select', function (e) { $(element).on('keypress', 'input, textarea, select', function (e) { ... }); }); ed.appendUntied(beDisposed, () => { $(element).off('keypress'); $(document).off('keypress'); }); }

4.約束と競合状態（クライアントコードの競合状態）



JavaScriptアプリケーションで、サーバーへのアクセスなどの非同期操作を使用する必要が生じる場合がよくあります。 そのような問題を解決するために、原則として、彼らはいわゆる約束を使用します。 jQueryの約束または一般的なQライブラリの約束のいずれかを使用できますが、どのライブラリを使用するかにかかわらず、そのようなオブジェクトを操作する方法は似ています。 promise自体を使用するのは簡単ですが、実装されているシナリオによっては、副作用が発生する可能性があります。これは、デバッグおよび分析ツールの助けを借りて初めて確認できます。

例として次のコードを使用して、いわゆる競合または競合状態の場合を考えます。 

 //   then some.willGetLegendImage(this.beDisposed).then((image) => { model.setLegendImage(image); });


ここでの問題は、呼び出し元のコードがプロミス（その後）返されたオブジェクトを無視し、その結果、関数の実行がいつ完了するかわからないことです。 関数の実行はメインスレッドから脱落するようであり、使用されるオブジェクトはコールバックまでメモリにロックされたままです。 さらに、元のオブジェクトを削除した後に関数が機能する場合に、undefinedという形式の例外が発生する可能性があります。



より安全なコードを取得するには、後で使用するためにpromiseを返す必要があります。 

 //     promise,     then.  promise   promise,   willGetLegendImage. var promise = some.willGetLegendImage(this.beDisposed).then((image) => { //   model  callback-     model.setLegendImage(image); });


したがって、ソースプロミスの状態を常に監視し、ソースプロミスから継承されたthen（）呼び出しの結果を常に返す必要があります。 これらのオブジェクトは、イベントハンドラーの削除メカニズムを使用して削除することもできます。

プロジェクトに要件を実装するために、利用可能なhtml要素のみでは実装が困難なグラフと図を表示することが必要になりました。 D3ライブラリの機能に触発され、SVG要素と組み合わせて使用​​することが決定されました。 ライブラリのAPIは比較的単純であり、開発されたドキュメントが付属しており、多くの実用的な例がありますが、アプリケーションのコンテキストでは、言及したい一連のニュアンスが生じました。



5.1。 D3選択の更新



D3を使用して、グラフ、チャート、その他の関連要素（グラフの軸など）を表示するのは非常に簡単であることがわかりました。 また、独自のデータバインディングメカニズムのおかげで、ビューを更新するためのコードは読みやすく簡潔に見えました。 ただし、D3を使用する一般的な方法は、削除された要素をクリーニングする機能で希釈する必要がありました。



データを更新するときに、更新されたデータと一致しない、つまり不要になるsvg要素のサブセットを取得できると仮定するのは論理的です。 多くの場合、ビューを構成する要素は、ビューを構成するときに、他のコンポーネントで使用できます。そのような要素は、削除する前に何らかの方法でクリーンアップする必要があります。 データバインディング後、D3 UpdatedSelectionオブジェクトを取得します。 このようなオブジェクトには、削除するアイテムのサブセットへのアクセスを許可するexit（）メソッドがあります。 このメソッドは要素の配列を返し、ループ内で並べ替えを行い、クリーニングコードを実行できます。 そのような機能がなく、以前に生成された要素へのリンクがある場合、後でメモリにハングしているDOM要素（いわゆる分離DOM要素）を見つけることができます。

5.2。 D3イベント



D3は、イベントを操作するための独自の抽象化を提供します。 たとえば、データをバインドするとき、またはd3.behaviors（ドラッグ、移動、ズームなど）を使用するときに、D3を介してイベントを使用する必要が生じる場合があります。

6. Googleマップ



6.1。 マップオブジェクトのライフサイクルの詳細



このコンポーネントのすべての利点を活用するには、マップオブジェクトのライフサイクルの機能を考慮する必要があります。一度マップオブジェクトのインスタンスを作成すると、それを削除することはできません。 Google Map APIは、このオブジェクトのデストラクタ関数を提供しません。 この側面については、 https ： //code.google.com/p/gmaps-api-issues/issues/detail？id = 3803で詳しく説明しています 。

不要なメモリリークを回避するには、アプリケーションアーキテクチャの最初からこのニュアンスを考慮する必要があります。



したがって、メモリリークを最小限に抑えるために、開発者はそのような特定のオブジェクトの慎重な使用に注意する必要があります。



アプリケーションの詳細に応じて、一度作成されたカードのインスタンスがあれば、それを再利用することをお勧めします。 または、複数のカードを同時に使用する必要がある場合は、マップオブジェクトのプールを用意します。
