リアルタイム更新でFacebook Graph APIクエリを最適化する

Facebookアプリケーションにはさまざまな機能があります。たとえば、多くの場合、ユーザーがアプリケーションを使用している間、アプリケーションはAPIを介して十分な情報を受け取ります。 しかし、1か月以上開いていなくても、アプリケーションが最新のユーザーデータで動作する場合はどうでしょうか。



ユーザーがアプリケーションで作業している間だけでなく、データを受信する方法は2つあります。

  1. ユーザーからoffline_access許可を取得し（「永遠の」ユーザーaccess_tokenを保存し）、「スケジュールに従って」必要なデータを取得します（cronスクリプトをプルします）。
  2. Facebookからすべてのデータ変更を受信し、リアルタイム更新を介して更新を構成およびサブスクライブするスクリプトを記述します。


猫の下で、リアルタイム更新を使用することで、状況によっては1日あたりのAPIリクエストの数を100倍以上減らす方法の仮想的な例を学ぶことができます。 更新にサブスクライブするためのスクリプトを作成し、その動作を確認して、Facebook自体からオブジェクトの変更に関するデータを受け取ります。



記事の冒頭には多くの理論が含まれており、Facebookのドキュメントページのロシア語の翻訳に似ています 。 英語で読みやすい場合は、2番目の方法のすべてをスキップして、「 リアルタイム更新の例」セクションに進んでください 。 しかし、何らかの理由で、最初の読書の後、この機能を構成できませんでした。 この記事があなたがより早く理解するのに役立つことを願っています。



最初の方法



一部のアプリケーションの機能は、スケジュールに従ってデータを受信する必要性に関連している場合があります。 たとえば、特定のディレクトリからページのいいね！を収集します。 これを行うには、1時間に1回間隔でデータを取得します。 これは良い使用例です。そうしないと、このデータを取得できなくなります。 しかし、多くの場合、この方法は、「スケジュールどおり」ではなく、Facebook自体からリアルタイムで更新を受信できるオブジェクトにも使用されます。



最初の方法の重大な欠点のうち、次のものを区別できます。

  • データのチェック頻度が低いほど、信頼性は低くなります。
  • 頻繁な更新による高いリソースコスト。
  • ユーザーは、offline_access-いつでも自分のデータへのアクセスを許可したがりません。


機能がユーザーの友人に関連付けられているアプリケーションを作成したと想像してください。最新の情報を入手することは非常に重要です。 アプリケーションが1000人のユーザーによってインストールされたとします。 そして、友人データの許容されるキャッシュ時間を1時間とします。 その結果、ユーザーフレンドのデータベースが最新になるように、毎日24,000のリクエストを送信します！ また、ユーザーは1000人のみであり、1日にすべての人が友人に変更を加えるのは数百に過ぎないと想定できます（これは個人的な経験に基づく推測です）。



最初の方法の欠点を解決するために、FacebookはGraph APIのリアル​​タイム更新を行いました。



第二の方法



リアルタイム更新のサポートにより、アプリケーションでFacebookデータの変更をサブスクライブできます。 アプリケーションは、Facebookサーバーから「スケジュールどおりに」ポーリングする代わりに、Facebookから更新を受信して​​データを保存します。 データをキャッシュしてこのAPIを使用すると、アプリケーションの信頼性が向上し、ロード時間が短縮されます。 購読している変更が発生するたびに、Facebookは変更のリストを使用してデータ更新処理スクリプトにHTTP POST要求を行います。 サーバーは、変更が表示されてから数分以内に変更の通知を送信します。



オブジェクト


現時点では、次のタイプのオブジェクトをサブスクライブできます。

  • user-特定のフィールドおよび対応するGraph APIノードとのユーザー関係に関する通知を受け取ります。
  • 許可-ユーザーがアプリケーションの許可を変更したときに通知を受け取ります。
  • ページ-アプリケーションを追加（インストール）したページが一般的なプロパティを変更したときに通知を受け取ります。


*サブスクライブできるUserオブジェクトのすべてのプロパティと関係。 たとえば、フィード、友人、アクティビティ、興味、音楽、本、映画、テレビ、いいね、チェックインなどのリンクが利用できます。 ただし、同時に、ホーム、タグ付き、投稿、写真、アルバム、ビデオ、グループ、メモ、イベント、受信トレイ、送信トレイ、更新、アカウントなどのリンクの変更を購読することはできません



サブスクリプション


アップデートのサブスクリプションを設定するには、次のものが必要です。

  1. FacebookからHTTP GET（サブスクリプションを確認する）要求とPOST（データの変更に関する）要求の両方を受信するスクリプト（URL）を設定します。
  2. Graph APIにPOSTリクエストを行う 
      https://graph.facebook.com/<app-id>/subscriptions
    サブスクライブし、検証リクエストを処理する準備をします。


リクエストを送信することにより 
  https://graph.facebook.com/ <app-id> / subscriptions？access_token = ...
送信するHTTP POST、GET、またはDELETEリクエストに応じて、次の3つのことを実行できます。

  1. サブスクリプションの追加と変更（POST）。
  2. サブスクライブしたすべてのオブジェクトとそのフィールドのリストを取得します（GET）
  3. サブスクリプションの削除（DELETE）


リクエストにはaccess_tokenアプリケーションが含まれている必要があります。これは、アプリケーションID（app-id）とアプリケーションシークレットキー（app-secret）を使用して取得できます。 HTTP GETを送信するだけです 
  https://graph.facebook.com/oauth/access_token?client_id=<app-id>&client_secret=<app-secret>&grant_type=client_credentials
そして答えが来るでしょう：access_token = ...



サブスクリプションの追加と変更


したがって、サブスクリプションを追加するリクエストを送信する必要がある場所を上記で読みました。 急いでこれらの要求を満たす必要はありません。資料のプレゼンテーションの構造は、必要な手順の順序とは異なります。 これは、材料を連続的に研究するために特に行われます。 ただし、この例では、例の前の最後の部分で説明されているクエリを最初に実行します。




アプリケーションは、可能なオブジェクトのタイプごとに1つのサブスクリプションのみを持つことができます。 すでにサブスクリプションがあるオブジェクトを追加すると、既存のサブスクリプションは新しいサブスクリプションに置き換えられます。 サブスクリプションを追加または変更するには、フィールドを含むPOSTリクエストを送信する必要があります。

  • オブジェクト-オブジェクトの種類：ユーザー、権限、またはページ。 たとえば、アプリケーションのすべてのユーザーなど、このタイプのすべてのオブジェクトを監視します。
  • fields-選択したオブジェクトのプロパティと関係のリスト（コンマで区切られた要素）。 たとえば、ユーザーの名前、写真、友達フィールド、およびニュースフィードリンクの変更を監視するには、値「名前、写真、友達、フィード」を指定する必要があります
  • callback_url-アプリケーションが購読している変更をFacebookが投稿するURL。
  • verify_tokenは、リクエストの信頼性を検証するために設定する「シークレットコード」です。 サブスクリプション処理スクリプトに渡されます。

サブスクリプションを削除


すべてのオブジェクトに対するサブスクリプション全体を削除するには、DELETEリクエストを送信します。 特定のオブジェクトのサブスクリプションを解除する場合は、オブジェクトパラメーターを指定します（たとえば、object = user）。



コールバックサーバー



Facebookサーバーは、サブスクリプションを追加または変更しようとすると、callback_urlにHTTP GETリクエストを送信します。 サブスクリプションが成功すると、FacebookサーバーはHTTP POSTを使用して同じURLに変更通知を送信します。



サブスクリプション検証


サブスクリプションが最終的に追加または変更される前に、Facebookサーバーは次のパラメーターを使用して、callback_urlでHTTP GETを作成します。

  • hub.mode-文字列「subscribe」がこのパラメーターに渡されます。
  • hub.challenge-ランダムな文字列。
  • hub.verify_tokenは、Facebookに送信したリクエストの信頼性を検証するために設定する「シークレットコード」です。


データ更新を処理するためのスクリプトは、最初にhub.verify_tokenの値（Facebookに送信した「シークレットコード」）を確認してから、hub.challengeパラメーターを返します。 これは、サーバーでDDoS攻撃を実行する手段としてスクリプトを保護するために特に行われます。

開発者向けの注意：PHPでは、パラメーターのピリオドは自動的にアンダースコアに変換されます。 したがって、 $_GET["hub_mode"]



、 $_GET["hub_challenge"]



および$_GET["hub_verify_token"]



を参照する必要があります。

結論


これで、Facebook Real-Time Updatesのすばらしい機能について知ったので、ユーザーデータと権限の変更に関する情報を送信できます。 残念ながら、何らかの理由で、ユーザーがアプリケーションを削除しても、データは変更されませんでした。 したがって、新しいメソッドが時間とリソースを節約しないように-結局、特定のタスク（たとえば、アプリケーションのすべてのユーザーにメールを送信する）を実装するときに、通常のAPIを使用してデータベース内のすべてのユーザーを事前に調べて、アプリケーションがインストールされているかどうかを確認することはまだ価値があります。
