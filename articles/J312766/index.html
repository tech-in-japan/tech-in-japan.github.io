Xcode：おそらくストーリーボードを操作する最良の方法







この投稿は、 Xcodeの無料翻訳です：ストーリーボードを扱うためのより良い方法 、 スタンオストロフスキー







元の記事の一部のコード例は古くなっており（Swift 3のリリースにより）、翻訳で変更されています。







Interface Builderを使用するためのヒントとコツ。







Appleは、新しいXcode 8でInterface Builderを大幅に改善しました。サイズクラスの使用がより直感的になり、ストーリーボードを拡大縮小する機能が非常に便利になり、Interface Builderでの完全なプレビューが素晴らしいです。 Interface Builderの使用に疑問を抱いている人にとっては、これはプラスになります。







一方、多くの開発者は、複雑なナビゲーションを備えた大規模なマルチスクリーンアプリケーションを作成する際に、Interface Builderでまだ問題を抱えています。







この記事では、プロジェクトでストーリーボードを操作するためのベストプラクティスをいくつか紹介します。 Interface Builderをすでに使用していますか、それともこの方向への最初の一歩を踏み出しているだけですか？ -いずれにせよ、これらのヒントはあなたに役立ちます。







1.チームで作業している場合は、画面ごとに個別のストーリーボードを使用します。 あなたが一人で働いていても、それは確かに良い習慣になります。


プロジェクトには、このようなmain.storyboardファイルが1つありますか？













デザイナーの観点から見ると、すべてが問題ありません。UIとナビゲーションは完全に表示されます。 そして、これはまさにInterface Builderが作成されたものです。

しかし、開発者にとって、これには多くの問題が伴います。







  • バージョン管理： ストーリーボードのマージの競合を解決するのは非常に難しいため、別々のストーリーボードで作業すると、チームの生活が楽になります。
  • ストーリーボードファイルは膨大になり、操作が困難になります。 間違ったView Controllerでマウスをクリックして制約を誤って変更する頻度はどれくらいですか？
  • ストーリーボードIDを各View Controllerに割り当てる必要があり、これによりエラーが発生する可能性があります。コードでこのView Controllerを使用するたびに、このIDを「ハードコード」する必要があります。


プロジェクト内のさまざまなストーリーボードを接続する方法は？ 2つの方法があります。







  1. Xcode 7に登場したストーリーボード参照へのリンクを使用します。







  2. ストーリーボードをコードで直接リンクします。


最初の方法の詳細については、 こちらをご覧ください 。







複雑なプロジェクトで広く使用されているため、2番目の方法について説明します。







2.ストーリーボードファイルと関連するコントローラークラス（UIViewControllerの子孫）に同じ名前を使用します。


これにより、命名規則が簡素化されるだけでなく、段落3で説明するいくつかの「利点」が得られます。







3.コントローラークラスでストーリーボードを直接初期化します。


ストーリーボードを介してView Controllerを初期化する場合、次のコードがよく表示されます。 







let storyboard = UIStoryboard(name: “Main”, bundle: nil) let homeViewController = storyboard.instantiateViewController(withIdentifier: “HomeViewController”)


少し汚い：ストーリーボードに名前を付ける必要があり、View ControllerのストーリーボードIDが必要であり、HomeViewControllerを作成するたびにこのパターンを使用する必要があります。







このコードをコントローラークラス自体に転送し、ストーリーボードを使用して静的メソッドを使用してコントローラーを初期化することをお勧めします。 







 class HomeViewController: UIViewController { static func storyboardInstance() -> HomeViewController? { let storyboard = UIStoryboard(name: “HomeViewController”, bundle: nil) return storyboard.instantiateInitialViewController() as? HomeViewController } }


前のヒント（同じファイル名）に従えば、ストーリーボード名の「ハーコード」を避けて、 ストリング（記述:)を使用できます。 







 let storyboard = UIStoryboard(name: String(describing: self), bundle: nil)


ストーリーボードファイルの名前がコントローラークラスと同じであることを確認してください。 そうしないと、このようなストーリーボードへのリンクを作成しようとすると、アプリケーションがクラッシュします。

これにより、コードが読みやすく、フォールトトレラントになります。 







 class HomeViewController: UIViewController { static func storyboardInstance() -> HomeViewController? { let storyboard = UIStoryboard(name: String(describing: self), bundle: nil) return storyboard.instantiateInitialViewController() as? HomeViewController } }


instantiateInitialViewController（）を介してView Controllerにアクセスする場合は、Interface BuilderでこのView ControllerをinitialViewControllerとして指定してください。 同じストーリーボードに複数のView Controllerがある場合、 instantiateViewController（withIdentifier：_）を使用する必要があります

さて、このようなView Controllerの初期化には1行かかります： 







 let homeViewController = HomeViewController.storyboardInstance()


シンプルでわかりやすいでしょう？







同じアプローチを使用して、nibからビューを初期化できます。 







 class LoginView: UIView { static func nibInstance() -> LoginView? { let nib = Bundle.main.loadNibNamed(String(describing: self), owner: nil, options: nil) return nib?.first as? LoginView } }


4.ストーリーボード上のトランジションでプロジェクトをオーバーロードしないでください。


ポイント1からのアドバイスに従えば、移行はありません。ただし、同じストーリーボードに複数のView Controllerがある場合でも、セグエを使用してそれらの間を移動することはお勧めできません。







  • 各遷移（セグエ）に名前を付ける必要があり、それ自体がエラーにつながる可能性があります。 名前の文字列をハードコーディングするのは悪い習慣です。
  • if / elseまたはswitch分岐演算子を使用して複数のセグエを操作する場合 、 prepareForSegueメソッドは単純に読み取り不可になります。


代替手段は何ですか？ ボタンをクリックして次のView Controllerに移動したい場合は、このボタンにIBActionを追加し、コードでView Controllerを初期化します。これは、ポイント3から覚えているように、1行です。 







 @IBAction func didTapHomeButton(_ sender: AnyObject) { if let nextViewController = NextViewController.storyboardInstance() { // initialize all your class properties // nextViewController.property1 = … // nextViewController.property2 = … // either push or present the nextViewController, // depending on your navigation structure // present(nextViewController, animated: true, completion: nil) // or push navigationController?.pushViewController(nextViewController, animated: true) } }


5.セグエをほどきますか？ いいえ、聞いていません。


ナビゲーションには、ユーザーを前の画面に戻すことが含まれる場合があります。







よくある間違い：新しいトランジションを使用して、前のView Controllerに移動します。 このような遷移は、現在のView Controllerを削除して以前のView Controllerに戻る代わりに、すでにスタックにあるView Controllerの新しいインスタンスを作成します。

iOS 7以降、Interface Builderでは、ナビゲーションスタックを「巻き戻す」ことができます。













セグエの巻き戻しにより、前の画面への戻りを指定できます。 とても簡単に聞こえますが、実際にはいくつかの追加手順が必要であり、開発者を混乱させるだけです。







  • 通常、ボタンのアクション（アクション）を作成すると、Interface Builderがコードを作成します（ IBAction ）。 この場合、「Ctrl」を押しながらボタンから「Exit」にアクションをドラッグする前に、コードがすでに記述されていることが予想されます。
  • 通常、ボタンのアクションを作成すると、このアクションのコードは、ボタンが属する同じクラスに作成されます。 巻き戻しセグエの場合、この遷移が発生するView Controllerのクラスでコードを記述する必要があります。
  • prepareForUnwindメソッドには、 prepareForSegueメソッドと同じ欠点がすべてあります（前の段落を参照）。


もっと簡単な方法は何ですか？







コードでこれを行う方が簡単です。ボタンの巻き戻しアクションを作成する代わりに、通常のIBActionを作成し、 dismissViewControllerまたはpopViewControllerを使用します（ナビゲーションに応じて）。 







 @IBAction func didTapBackButton(_ sender: AnyObject) { // if you use navigation controller, just pop ViewController: if let nvc = navigationController { nvc.popViewController(animated: true) } else { // otherwise, dismiss it dismiss(animated: true, completion: nil) } }
