Ruby on Rails用のサイト訪問者ソーシングモジュール

この投稿は主にWeb分析に関するものです。サイトへの訪問者のソースを正しく識別する方法、およびこの困難なタスクに役立つRuby on Railsのモジュールについてです。 最後に、Railsコミュニティに注意を払うようにお願いする小さな部分があります。それは私とRailsに関するものです。 しかし、順番に見てみましょう。



パート1 Web分析と訪問者のソースの決定について



問題


かなり些細な作業があります：私たちのサイトに来た訪問者のソースを特定することです。 私はあなたのことは知りませんが、長い間、GoogleアナリティクスのCookieの本体に寄生していました。 私たちの分析は、ソースの書き換え、セッションのアカウンティングの問題を解決し、一般に、解析リファラーのすべての費用を排除しました。 しかし、良いことはすべて終わります。



GoogleがUniversalのベータ版をリリースしたとき、遅かれ早かれ、Google Cookieに別れを告げ、それを自分で行う方法を学ぶ必要があることが明らかになりました。 しかし、それ以来、彼はClassicとUniversalの非互換性を宣言しました。これまでのところ、あなたは正確に座ることができました。Classicは長い間サポートされます。



GA Cookieの新しいバージョンでは、ユーザーIDが1つだけ残っています。 分析はそれをサーバーに送信し、そこですべての計算を既に実行します。 そして今では、Cookieやjsを介してソースからソースに関する情報を取得することはできません。


しかし最近、Googleはユーザーをスティックで優しく突くようになりました。プロファイルコンバーター-Classic→Universalを展開しました。 そして、ここでロールのウィグリングを開始する必要があります。リマーケティングリストがUniversalに追加された時点で、クラシックアナリティクスは最終的なリーダーを受け取ります。 そして、これはそれほど遠くないと思います。



この点で、私は自作のutmz cookieジェネレーターに生まれました。 そしてそれをソースバスターと呼びました。



最初にフォームについて


ジェネレーターは、 Ruby on Rails用のマウント可能なエンジン形式で作成されています 。 すべてのRailsアプリケーションにgemとして素早く調整し、コンソールからの単一コマンドで更新できます。 モジュールは独立しており、それ自体ですべてを実行します。 ソースに関するデータを使用して、電話番号、サイト上のコンテンツを置き換え、アプリケーションとともに保存し、さらなる分析に使用できます。 特定のルールに従って、モジュールはソース（および多数のパラメーター）を計算し、訪問者のCookieに保存します。



GitHubへの直接リンク： https : //github.com/alexfedoseev/sourcebuster

データ構造


合計で、4つの主要な種類のトラフィックがあります。

  1. utm -utmタグでマークされたトラフィック
  2. オーガニック -オーガニック検索エンジンの結果からのトラフィック
  3. 紹介 -紹介トラフィック（サードパーティのリソースからのリンク）
  4. typein-直接遷移

次に、1人の訪問者が異なるソースから異なる時間にサイトにアクセスできるため、ソースを書き換えるためのルールを作成する必要があります。



ソースを書き換えるロジック


書き換えロジックは、Googleアナリティクスのロジックに従います。





参照リンクは、現在のセッションで何も上書きしないことに注意してください。 理由-例で説明しましょう。多くの場合、現在の訪問中に、訪問者は実際のソースではないサードパーティのリソースからサイトにアクセスします。たとえば、登録を有効にするリンクがあったメールサービスからです。

このシステムでは、現在の訪問に関するデータを上書きすることに加えて、最初の訪問に関するデータを保存することにしました。 つまり、変換時に、訪問者の最初と現在のソースに関するデータが得られます。



モジュールを使用して正確に引き出すことができるもの：

  • 最初のソースのデータ：

    utm_source、utm_medium、utm_campaign、utm_content、utm_term
  • 現在のソースに関する同じデータ

    （ユーザーが別のソースから繰り返し移行した場合）
  • 最初の訪問日
  • エントリーポイント
  • ソースが書き換えられた完全リファラー
  • IPおよびユーザーエージェントユーザー




モジュールのインストール
モジュールを固定したいRailsアプリケーションが既にあると仮定します。



Gemfileアプリケーションで、次を追加します。 

gem 'sourcebuster', :git => "git@github.com:alexfedoseev/sourcebuster.git"


インストール： 

 bundle install


これはマウント可能なエンジンであるため、分離された名前空間に存在します。

routes.rbを追加して、アプリケーションにマウントします。 

 mount Sourcebuster::Engine => "/sourcebuster"


次に、すべての移行をコピーして実行する必要があります。

コピー： 

 bundle exec rake sourcebuster:install:migrations


そして実行： 

 bundle exec rake db:migrate


データベースには3つの新しいテーブルがあります。

  • sourcebuster_referer_sources

    カスタムソースデータ。
  • sourcebuster_referer_types

    紹介の種類に関するデータ（紹介トラフィックの場合は基本的にutm_medium）。
  • sourcebuster_settings

    アプリケーション設定（セッション期間とサブドメイン処理）。


それらを使用して何もする必要はありません。既にデータがすぐに使用でき、それらのインターフェイスがあります。



マウント可能なエンジンの詳細-http://guides.rubyonrails.org



モジュールはほとんど接続されていますが、最後のタッチは残ります。アプリケーションのどこにでもCookieを設定できます。 これを行うには、 アプリケーションのapplication_controller.rbに追加します。 

 class ApplicationController < ActionController::Base include Sourcebuster::CookieSettersHelper before_filter :set_sourcebuster_data helper_method :extract_sourcebuster_data # some code private def set_sourcebuster_data set_sourcebuster_cookies end end


準備完了のようです。 エンジンはメインアプリケーションテンプレートを使用するため、スタイルを自分でカスタマイズできます（これを変更する可能性があります）。 何かがうまくいかない場合、私は何かを見逃す可能性があります-書き込み。

サブドメイン処理

これは基本的にGAの_setDomainNameに類似しています。 例で説明します。

サブドメインがあるサイトがあるとしましょう：

  • site.com
  • blog.site.com
  • shop.site.com


また、 site.comページからblog.site.comへの遷移を内部的な非参照遷移と見なします （つまり、1つのサブドメインから別のサブドメインに切り替えても、ソースは上書きされません）。 これを行うには、設定で「サブドメインがあり、それらの間のトラフィックはリフェラルであってはならない」をチェックする必要があり、 「メインホスト」フィールドでサイトのルートホストを追加します。そのサブドメインはすべて、モジュールによって1つのサイトと見なされます。 ここでは、 「site.com」が示されています。



フィールドにblog.site.comを指定すると、 alex.blog.site.comからblog.site.comへの移行は非参照になり、 alex.blog.site.comからshop.site.comへの移行は既に参照になりますトラフィック。



追加のソース


インターフェース： http : //sandbox.alexfedoseev.com/sourcebuster/custom_sources



システムには、多数の追加ソースの処理をカスタマイズする機能があります。

設定は、次のパラメーターに従って行われます。

  • ドメイン

    処理しているソースはそれに一致します。
  • エイリアス

    美しい/わかりやすいソース名。
  • チャンネル

    referral 、 organic、またはsocialを指定できます。
  • リクエストパラメータ

    検索エンジンのURLのキーワードパラメーター。


この表の例は、例で説明する最も簡単な方法が必要です。



例1

システムでBing検索からオーガニックトラフィックへの移行を検討する必要があります（これは事実です）。

bing.comにアクセスし、検索ボックスに「apple」というクエリを入力すると、次の形式のアドレスを含む検索ページにリダイレクトされます。

www.bing.com/search ？ q = apple ＆go =＆qs = n＆form = QBLH＆pq = apple＆sc = 8-5＆sp = -1＆sk =＆cvid = 718ad07527244c319ecebf44aa261f64



それに基づいて、新しい特別なソースを作成します。

  • ドメイン： bing.com
  • エイリアス： bing

    または、必要に応じて、何も記述しなくても、リファラーホストが置き換えられます。
  • チャンネル： オーガニック
  • キーワードパラメータ： q

    これは、検索結果を含むページのURLにある「？」と「= your_query」の構造の間の記号です。


これで、そのようなページに由来するすべてのものがオーガニックトラフィックと見なされます。



例2

ソーシャルからの移行を強調する必要があります。 別のグループのネットワーク。

同様のパターンに従います。

  • ドメイン： facebook.com
  • エイリアス： facebook
  • チャンネル： ソーシャル
  • キーワードパラメータ： 不要


できた これで、facebookからのすべてのクリックリンク（utmタグでマークされたものを除く）は、ソーシャルチャネルの価値を持つようになります。



ドメインフィールドでは、ゾーン（.com、.com.ruなど）を完全に指定する必要があります。 値facebook.comを指定すると、ドメインfacebook.com.ruからのトラフィックはこのフィルターに該当しません。 そして、ドメインm.facebook.comから-取得します。
