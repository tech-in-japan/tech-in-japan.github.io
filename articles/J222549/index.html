2014年のAPIの未来

2013年は、APIテクノロジーの開発と、この分野での普及と投資における印象的な成果によって特徴付けられました。 ProgrammableWebで初めてマークされたAPIの数は10,000を超え、この技術に関連する深刻なキャッシュインジェクションと買収があり、 APIエバンジェリストと共同で開催された独自の会議API戦略と実践を含む多くの新しい会議が開催されました。 2013年は、参加者が経済のさまざまな部門の長期的な課題を解決しようとしているAPI Commonsなどの新しいイニシアチブによって特徴付けられました。



API管理プラットフォームの一部としてサポートする既存のユーザーAPIの数は、過去1年間で倍増しました。これは、APIトラフィックの増加と同じです。 したがって、このような印象的な2013年の後、私たちは疑問に思いました。新しい2014年に向けて何を準備しているのでしょうか。 そして、ここに私たちの予測があります！



1.（予想どおり）APIの成長： 2014年に多くの新しいAPIが登場することになる、明らかな最初の予測-すべての指標は、その数が増えるだけであることを示しています。 既知のリストに記載されているパブリックAPIの数は10,000に達しましたが、プライベートAPIまたは半パブリックAPIの数はまだ多く、パブリックAPIを大幅に超えています。 これらのプライベートAPIには、モバイルバックエンドなど、急速に成長しているカテゴリに属する​​インターフェイスが含まれています。 SAAS統合用のAPI。 重量級のB2Bソリューションの統合に代わるAPI、およびPhilips Hue電球などの家電ハードウェア向けのAPIでさえも。



2014年には、これらの領域内で作成されるパブリックおよびプライベートAPIの数が100,000から200,000に増加すると予想しています-プライベートAPIとパブリックAPIの比率は9であるとの推定によると、すでにこの数の下限に近づいています： 1。 APIの成長は2つの方向で行われると予想されます。まず、新しいAPIがリリースされます（そのような成長の指標として、3スケールの数百の新規登録と毎月数十のAPIの起動に注意します）。企業がはるかに幅広いオーディエンスにリーチするために努力するプライベートAPI



2. APIの一般に受け入れられている使用のオプションの拡張：APIの数の増加に伴い、 2番目でおそらく最も重要なのは、APIリリースを開始する企業の種類の増加に関連しています-現在、スポーツ機器メーカー、小売業者、ハイテク機器メーカー建設会社、政府、慈善団体、経済のほぼすべての部門の代表者。 この傾向は今後も続くと考えています。 APIの経済に関する最近公開された電子書籍の研究では、モバイルプラットフォーム機能からビジネストランザクションの高速化まで、APIを使用するための5つの重要なオプションについて説明しました。これら5種類のAPI値はすべて相互接続されているようで、実際に適用できます経済のあらゆる部門で。



3. 2014年の重要な問題は、APIの著作権の問題です 。OracleとGoogle Javaの間の著作権訴訟の再開に伴い、 API仕様の要件の将来の厳しさについて再び予感がつきました。 -特定のインターフェイスパターンが著作権によって再利用から保護される範囲。 このような保護は正当化される場合もありますが、広く使用されている重要なパターンをサードパーティ企業から「成功裏に」保護するため、APIへの著作権の適用はイノベーションの開発に悪影響を与える可能性があります。 API Commonsなどのイニシアチブは、再利用可能なパターンの基盤を構築するのに役立ちますが、同様のプロジェクトは開発の初期段階にあるため、著作権紛争は2014年の重要な部分になります。



4.サービス記述技術に焦点が当てられます。RESTAPIの重要な課題の1つは、APIプロバイダーが、検索および処理できるAPIについて割り当てられ移動されたメタデータをほとんど公開しないことです。 WADLやSwaggerなどの一部の開発フォーマット、およびBlueprintやRAMLなどの新しいフォーマットが現在利用可能ですが、それらはせいぜい部分的に問題を解決し、一般的ではありません。 このように自動化された説明がないため、スタブライブラリの生成からAPIの検索まで、APIを操作するプロセス全体が困難になります。たとえば、このような電子説明の公開が標準になる前に検索エンジンAPIが表示されることはほとんどありません。



その結果、どの形式を借りるかに関する業界およびベストプラクティスに関連する議論の出現、およびそれらの公開/検索方法に関する合意の出現が予想されます。 これがないと、おそらくProgrammableWebなどのAPIのパブリックプラットフォームを整理することはできません。



5.新しいAPIアグリゲーターが追加されます。APIの数が増えると、 Segment.IOやZyprなどの中間レベルのプロジェクトが表示され、特定のセグメントの複数のAPIを単一のアドレス可能なAPIに結合できます。 このタイプのアグリゲーターは、複数のバックエンドAPIの単一の統合ポイントを作成します。 場合によっては、アグリゲーターが提供するソリューションは、それらが結合するAPIの機能のために信頼できないように見えるかもしれませんが、アグリゲーターのソリューションは開発者のプロジェクトの複雑さを軽減し、直接サポートのコストを削減するので、他の場合には、オープンアームで待たれます APIの数が増えると、アグリゲーターが表示されるエリアの数も増えます。 APIの使用ペースは作成の速度よりも先であり、組み合わせることで次の予測につながります...



6. API開発ツールは積極的に開発されます。現在、Mashery、ApigeeなどのさまざまなベンダーのAPIを開発するための多くのツールは、API所有者にAPIをサードパーティに提供する機会を提供することに焦点を当てています。 これは、現在どこでも使用されている重要な技術です。 ただし、最近まで、API「消費者」向けのツールはほとんどありませんでした。特に、開発者がアプリケーション内でAPIの使用をデバッグ、監視、追跡するためのツールはほとんどありませんでした。 2013年には、この問題を解決するためにいくつかの企業が登場しました。その中にはRunscopeやAPI Scienceなどの企業がありました。 今年は、3scaleからの新しいオファーを含む、このような組織がさらに登場します。 十分に文書化されたAPIを使用する開発者でさえ、目標に向かって困難と不満を経験しているため、APIを使用する開発者のイノベーションが重要になっています。



7.シングルページアプリケーションはAPI開発の新しいエンジンになります。HTML5の出現で実装が可能になったHTMLで記述されたWebアプリケーションのシングルページアプリケーションアーキテクチャ（またはSPA）のアーキテクチャは、現在大幅に成長しています。 SPAの概念では、HTML、CSS3、およびJavascriptを使用して、単一のHTMLページをWebブラウザーにアップロードし、バックエンドサーバーへのAPIリクエストによりアプリケーションに埋め込まれたすべての機能を実行することで動作するアプリケーションを作成します。このアプリケーションでは、新しいWebページに移動します。 このテクノロジーがAPIに与える影響についてはすでに記述しており[ 1、2 ]、この傾向の進展を観察しています。 特に、 Angular.jsなどのJavascriptフレームワークの人気が高まり始め、開発ツールが進化し、 Famo.usなどの新しいフレームワークが複雑なJS開発プロセスをWebにとって非常に魅力的にしています。



その結果、SPAアーキテクチャは2014年にビジネスアプリケーション/機能アプリケーションの標準になり始め、eコマースや小売でも徐々に使用されると予想されます。



8.ハイパーメディアのAPIの人気が高まります：ハイパーメディアで動作するAPIは、そのようなAPIからのリクエストに返されるレスポンスに特定の「許可された」アクションが含まれるため（このプロセスの詳細はこちら ）、APIに従来のAPIよりも柔軟性があります。要求の内容を動的に変更し、それによって現在の状況に適応します。 ただし、現在、静的URL APIよりもハイパーメディアAPIを好む人は依然として妥協する必要がありますが、現在このテクノロジーの大規模な展開は、 パブリックメディアプラットフォームおよび最近リリースされたAPIで見られますAmazon Appstream API



この技術が定着するには時間がかかりますが、2014年には、その使用のためのより鮮明なオプションと、それを普及させることを目的としたツールが増えると予想されます。



9.市民機関からのデータを処理するAPIは、今年の突破口となります。どのカテゴリのAPIが他のAPIよりも速く成長するかを予測するために、市民機関からの情報を使用してAPIに戻り続けました。 ニューヨーク、シカゴ、サンフランシスコ、アムステルダム、ヘルシンキなどの大都市のリーダーシップにより、すでに大量のデータが利用可能になっているため、今こそ都市インフラストラクチャにデータを提供するAPIの時代です。 そして、町民は、順番に、アプリケーションとマッシュアップを作成し、受け取った情報を視覚化することができます。 この傾向はますます一般的になっています-特に、政府機関は市民が必要とするすべてのWebおよびモバイルアプリケーションを作成するのに法外に費用がかかるという事実により、したがって、そのような情報へのアクセスはイノベーションと都市住民の機会への新しい方向を開きます助けて。 この分野の人気が高まっているもう1つの理由は、 open311 、 CitySDK 、 CityProtocolなどのイニシアチブが多数の都市でこれらのインターフェイスの一部を積極的に標準化しようとしていることです。これにより、テクノロジーの適応を促進し、同時にそのようなデータを使用するアプリケーションの消費市場を拡大する必要があります。



10. openAuth、JSONなどのネイティブWebテクノロジーは成長と発展を続けます。2012年から2013年にかけて、XMLはパブリックAPIのサポートに関して人気を失い始めました。JSONはこの方向への上昇傾向になりました。 以前にXMLを扱っていた企業がこのテクノロジのサポートを停止した場合もあります。 この傾向は2013年にも特徴的であり、主にJSONを指向し、XMLをはるかに少ない範囲で指向した多くの新しいAPIの出現が特徴です。 この傾向は、モバイルおよびHTML5 + JavascriptがAPIの数の増加にプラスの影響を与えることを考えると、他のWeb指向の標準の普及とともに、2014年もおそらく有効です。 APIが主にB2B統合（以前は以前のSOAP / XMLの特権）で使用されていた場合でも、モバイルスクリプトは追加のオプションとして積極的に考慮され、最終的なテクノロジーの選択に影響します。



そのため、2014年は私たちにとってエキサイティングな展望を切り開き、間違いなく多くの驚きを準備します！ 私たちは、仮定を確認または反論することを楽しみにしています。また、新しい予想外の何かについて学ぶことを楽しみにしています。
