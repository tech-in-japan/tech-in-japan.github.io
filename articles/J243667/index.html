ASP.NET 5の概要

ASP.NET 5は、ASP.NETプラットフォームに大きな変更をもたらします。 この記事では、ASP.NET 5の新しい概念について説明し、それらが最新のWebアプリケーションの開発にどのように役立つかを説明します。



ASP.NET 5の紹介。



ASP.NET 5は、最新のWebアプリケーションを構築するためのすべての冗長な.NETがないスタックです。 クラウドにデプロイするか、独自のサーバーで実行するアプリケーションを開発するための最適化されたプラットフォームを提供するためにゼロから構築しました。 ソリューション構築の柔軟性をサポートするために、このプラットフォームは最小限のオーバーヘッドでモジュラーコンポーネントで構成されています。

ASP.NET 5には次の機能が含まれています。



  • 新しい柔軟でクロスプラットフォームなランタイム
  • HTTPリクエスト用の新しいモジュラーパイプライン
  • クラウド構成を使用する準備ができました
  • MVC、Web API、およびWebページを組み合わせた統合プログラミングモデル
  • プロジェクトを再構築せずに変更を確認する機能
  • .NET Frameworkの複数のバージョンを並べて使用する
  • セルフホスティングまたはIISでのホスティングの可能性
  • Visual Studio 2015の新しいツール
  • GitHubオープンソース


ASP.NET 5に対して行った変更は、顧客の要求とフィードバックに基づいていました。 これらの変更により、開発、ホスティング、およびメンテナンスが簡素化され、最新のWebアプリケーションに重点が置かれます。



レガシーアプリケーションは、変更なしでASP.NETの新しいバージョンで実行されます。 ただし、ASP.NET 5の新機能を利用するには、既存のコードを新しいフレームワークに移植する必要があります。 ASP.NET 5と以前のバージョンのASP.NETには多くの類似点があるため、コードの移植は、アプリケーション全体を書き換えるのではなく、特定の問題領域を修正することです。



この記事では、ASP.NET 5の方向性と変更点について説明します。

ASP.NETが再設計されたのはなぜですか？



柔軟なクロスプラットフォームランタイムの必要性


これまで、.NET Frameworkは単一の包括的なインストールとして使用されてきました。 新しい.NETの各リリースでは、新しい機能が追加されましたが、古い機能が削除されることはめったにないため、フレームワークのサイズは絶えず増大していました。 このアプローチにより、.NETがインストールされたマシンがあらゆるタイプの.NETアプリケーションをサポートできるようになりますが、これは各アプリケーションが実際には使用されない機能に依存していることを意味します。



.NETのテクノロジーの1つに重要な更新が必要な場合は、.NETのこの部分がアプリケーションで使用されていなくても、重要な更新が利用可能であるという通知を受け取ります。 更新が実際に必要ないことを期待して、この更新をインストールしてアプリケーションを中断するか、更新を無視するかを決定する必要があります。



ASP.NET 5は、ランタイム環境の3つのバリアントで作業できるため、柔軟性が向上します。

  1. フル.NET CLR

    フル.NET CLRは、Visual Studioのプロジェクトのデフォルトランタイムです。 APIの完全なセットを提供し、後方互換性のための最良の選択です。
  2. コアCLR（クラウド最適化ランタイム）

    Core CLRはASP.NET 5プロジェクトのランタイム環境であり、不要なものは一切なく、完全にモジュール化されています。 このCLRはコンポーネントモデル用に再作成されました。現在、アプリケーションに必要な機能のみを含めることができます。 コンポーネントをNuGetパッケージとして追加するだけです。 その結果、アプリケーションは必要な機能のみに依存します。 ランタイム環境を開発するこのアプローチでは、各コンポーネントが独自のスケジュールで更新されるため、コンポーネントの更新をより迅速に提供できます。 .NET CLRのフルバージョンでは、Core CLRは約200メガバイトではなく、約11メガバイトを占有します。 Core CLRはアプリケーションと共に直接展開でき、Core CLRのさまざまなバージョンを並べて機能させることができます（その両方については以下で詳細に説明します）。
  3. クロスプラットフォームCLR

    LinuxおよびMac OS X用のクロスプラットフォームランタイムをリリースします。このランタイムにより、MacおよびLinuxデバイスで.NETアプリケーションを開発および実行できます。 Monoコミュニティと密接に連携しています。 リリース前は、Mono CLRを使用してクロスプラットフォーム開発を行うことができます。 詳細については、 ASP.NET vNext Application Development on Macを参照してください（ 翻訳者のメモ ：8月の記事は参照により入手できます。Habréには現在の状況を説明する投稿があります）。




デフォルトでは、新しいVisual StudioプロジェクトはFull .NET CLRを使用します。 プロジェクトの構成プロパティでコアCLRを指定できます。







どこでもホスティング


ASP.NET 5では、IISまたはセルフホスティングモードでアプリケーションをホストできます。 Core CLRを使用すると、展開パッケージで収集されたすべての依存関係を使用してアプリケーションを展開できます。 したがって、アプリケーションとその依存関係は完全に独立しており、システムへの.NETのインストールに依存しなくなりました。 アプリケーションは、あらゆるタイプのデバイスまたはホスティングプラットフォームで実行できます。



この新しい機会は多くの自由を与えます。 ホスティングに最適なオプションとしてIISを引き続きお勧めしますが、必要に応じて別のホスティングプラットフォームを使用できます。 ホスティング設定により、開発に使用するフレームワークが決定されなくなり、逆もまた同様です。



IISの外部でアプリケーションをホストする例： MVC 6でWeb APIを作成します 。



異なるバージョンの.NETを並べて使用する


サーバー上のアプリケーションが.NET Frameworkの単一のシステム全体のインストールに依存している場合、すべてのアプリケーションは同じバージョンの.NETで実行されます。 この状況では、.NET Frameworkの新しいバージョンへのアップグレードを検討する際に懸念が生じます。 一部のアプリケーションで最新バージョンの.NETを使用したい場合がありますが、古いアプリケーションがすべて新しいバージョンで適切に動作するかどうかはわかりません。



幸いなことに、ASP.NET 5はこの問題を修正します。 展開パッケージ内で依存関係を定義し、各アプリケーションの.NETバージョンを指定できます。 一部のアプリケーションでは最新バージョンのすべての利点が得られますが、残りのバージョンでは古いバージョンを使用し続けるだけです。 これらの異なるバージョンはすべて、問題なく並んで機能します。 異なるバージョンのアプリケーションを並べて実行するには、それらのターゲットコアCLRを選択する必要があります。



依存関係管理を簡素化する


ASP.NET 5には、プロジェクトの依存関係を管理するための新しい簡単な方法が導入されています。 プロジェクトにアセンブリ参照を追加する必要がなくなり、代わりにNuGetパッケージを参照して依存関係を管理します。 NuGetパッケージマネージャーを使用してNuGetパッケージを追加するか、プロジェクトで使用されるNuGetパッケージとバージョンをリストするJSONファイル（project.json）を編集できます。 他の依存関係を追加するには、NuGetパッケージの名前とバージョン番号をproject.jsonファイルに書き込むだけです。







Visual Studio 2015では、IntelliSenseを使用して手頃な価格のNuGetパッケージを見つけることができます。







project.jsonファイルには、プロジェクトに直接追加されたNuGetパッケージのみが含まれます。 他のパッケージに依存するNuGetパッケージを追加すると、これらの2次依存関係はロードされますが、project.jsonファイルにはリストされません。 このアプローチにより、project.jsonファイルを整理して管理しやすくすることができます。 project.jsonからNuGetパッケージを削除すると、それらを必要とする他のパッケージがない場合は、2次依存関係も削除されます。



JSON形式のおかげで、Visual Studioがインストールされていなくても、依存関係の管理は簡単です。 任意のテキストエディターでproject.jsonファイルを開き、変更を加えることができます。たとえば、クラウドにデプロイされたアプリケーションの依存関係を更新します。



MVC、Web API、およびWebページでの重複を排除


これまで、MVC、Web API、およびWebページには、同様の機能のさまざまな実装が含まれていました。 たとえば、MVCとWeb APIはルーティングを提供しますが、MVCルーティングクラスはSystem.Web.Mvc.Routing名前空間にあり、同様のWeb APIクラスはSystem.Web.Http.Routingにあります。 または、WebページとMVCは両方ともRazor構文を使用しますが、一部のNuGetパッケージはいずれかの実装とのみ互換性があります。



ASP.NET 5では、MVC、Web API、およびWebページがMVC 6と呼ばれる単一のフレームワークにマージされます。このマージにより、フレームワークから重複が削除され、アプリケーション開発が容易になります。 MVC、Web API、またはWebページのいずれを使用しているかによって、わずかに異なるコードを記述する必要はなくなりました。



このプレビューバージョンでは、MVCとWeb APIがMVC 6に統合されました。Webページは、後のバージョンでMVC 6に追加されます。



HTTPパフォーマンスの改善


ASP.NET 5では、HTTP要求用の新しいソフトウェアパイプラインが導入され、不要なものがすべてなくなりました。 このコンベアはモジュール式であり、必要なコンポーネントのみを追加できます。 オーバーヘッドを減らすことで、アプリの帯域幅が大きくなります。 新しいパイプラインはOwinもサポートしています。



クラウド対応


新しいASP.NET 5プロジェクトを作成すると、このプロジェクトはクラウドで簡単に展開できるように構成されます。 Visual Studio 2015は、Web.configファイルを置き換える新しい環境構成システムを提供します。 新しいシステムでは、さまざまなソース（JSON、XML、環境変数など）から名前付きの値を照会できます。 各環境に値を指定し、デプロイ後、アプリケーションは正しい値を読み取るだけです。



また、クラウド内のアプリケーションの問題を簡単に検出できる診断ツールとトレースツールも提供しています。



依存性注入の統合


依存関係の注入はASP.NET 5に組み込まれています。IoCコンテナーを使用して、依存関係を登録できます。 インジェクションの依存関係により、使用環境に適したサービスを簡単に提供できます。 詳細については、「 ASP.NET vNextのDependecyインジェクション 」を参照してください。





オープンソースと透明化


ASP.NET 5のすべてのコードはGitHubで入手できます 。 コードを使用できるだけでなく、開発全体でGitHubリポジトリを使用します。 どのような変更が行われたか、いつ行われたかを正確に確認できます。 コードをダウンロードして、変更を提供できます。



GitHubでASP.NET 5を開発することにより、コードと意図した開発を理解しやすくします。 編集を提供したり、ASP.NETのカスタム実装を開発したりできます。



柔軟な開発環境を提供する


Visual Studio 2015は、ASP.NETアプリケーションを開発するための軽量なアプローチを提供します。 コードを変更し、変更を保存して、ブラウザーでページを更新するだけです。 プロジェクトを再構築せずにブラウザの変更が表示されます。







Webプロジェクトのコード、またはプロジェクトが参照するクラスライブラリのコードを変更できます。 変更を確認するには、デバッグモードではなく、プロジェクト（CTRL + F5）を実行できます。



Visual Studioは、動的コンパイルにRoslynコンパイラを使用します。 コンパイルされたフレームワークのフルパワーは引き続き利用できますが、開発によりインタープリター言語を使用する感覚が生まれます。



Visual Studioのグ​​ラフィカルインターフェイスの各機能は、コマンドライン操作に対応しています。 インターフェイスの使用とコマンドラインでのスクリプトの作成を簡単に切り替えることができます。



最後に、ASP.NET 5プロジェクトに他のコードエディターを使用できます。



Webフォームはどうですか？



Webフォームを使用して引き続きアプリケーションを開発し、Webフォームが.NET Web開発プラットフォームの不可欠な部分であることを完全に確信できます。 開発経験が現在のWebプラクティスと一致するように、Webフォームに新機能を追加することに引き続き注力しています。



Web Forms 4.6には、次の新機能が含まれています。

  • HTTP 2
  • 非同期バインディングモデル
  • Roslyn CodeDOMコンパイラ


既存のWebフォームアプリケーションは、.NET 4.6を使用するIIS上で変更なしで引き続き実行されます。 Core CLRでWebフォームアプリケーションを使用することはできません。



Web Forms 4.6の新機能に関するビデオについては、ここをクリックしてください。 Visual Studio 2013 Update 2でのWebフォームの多くの変更の説明については、「 ASP.NET Webフォームの改善」を参照してください。



古いアプリはどうですか？



ASP.NET 5で多くの変更が行われたため、すべてのアプリケーションを書き換える必要があるのではないかと心配するかもしれません。 心配しないで。 ASP.NETの以前のバージョンで構築されたアプリケーションは、新しい.NET Frameworkで引き続き機能します。 ASP.NET 5の新機能が必要な場合を除き、これらのアプリケーションを更新または移植する必要はありません。



たとえば、現在Webフォーム、MVC 5、Web API 2、SignalR 2、Web Pages 3、またはEntity Framework 6を​​使用しているアプリケーションは、変更されていない新しいフレームワークで完全にサポートされます。 ただし、以前のバージョンとの完全な互換性を提供するのはこのCLRのみであるため、古いアプリケーションを実行するには完全な.NET CLRを使用する必要があります。



コアCLRのAPIは多少制限されています。 Core CLRを使用するには、アプリケーションは、このランタイムで使用可能な型とメンバーのみを使用する必要があります。



アプリケーションがCore CLRランタイムで実行できることを確認するには、Portability Analyzer APIを使用します 。 このツールは、アプリケーションを実行できるプラットフォームと、他のプラットフォームでのアプリケーションの起動をブロックする依存関係を示します。 これは、必要な変更の範囲を理解するのに役立ち、サポートされていないものを置き換える新しいタイプまたはメンバーを提供します。



MVC 6およびSignalR 3アプリケーションは新しいHTTPパイプラインを使用するため、System.Webを使用するアプリケーションと互換性がありません。 既存のアプリケーションをMVC 6またはSignalR 3にアップグレードするには、Visual Studio 2015で新しいプロジェクトを作成し、コードを新しいプロジェクトに転送する必要があります。 移植するときは、サポートされていないコードを変更する必要があります。
