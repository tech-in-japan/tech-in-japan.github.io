Android用のシンプルなScala SBTアプリケーションを作成する方法

こんにちは、友達。 開発者向けのツールを開発しているときに、前述のツール、プログラミング言語、およびプラットフォームの相互作用に関する質問に出会いました。 最初の2つのエンティティが単純に相互に作成された場合（SBTがScalaでの開発に最もよく使用される主要なビルドツールである場合）、Andriodプラットフォームへのそれらの適用性は認められません。 それにもかかわらず、巨大で絶えず進化しているソフトウェア業界では、この一連のテクノロジーが合理的な選択になる状況があります。 問題は、この選択が合理的であるかどうかであり、この記事の主題ではありません。 そして、その方法を正確に説明します。



開発環境



Android StudioとIntelliJ IDEAで動作するこのアプローチの適用性を調査しました。 この目的のために、実際には環境に大きな違いはありません。 微妙な違いがあります-それらを使って、記事の過程で知り合います。 しかし、一般に、使用するテクノロジーは、StudioとIdeaの両方に等しく統合されています。 結局、最初のものはIdeaと同じIntelliJプラットフォームに基づいています。 したがって、選択はプロジェクトの詳細に基づいて行う必要があります。



基本的なプロジェクトを作成する



私たちのビジネスの最初のツールは、IntelliJ IDEAとAndroid StudioのScala言語で開発するために設計されたScalaプラグインです。 このプラグインはSBTも提供します。 開発環境には付属していないため、JetBrainsリポジトリからダウンロードする必要があります。









これで、基本的なScala SBTプロジェクトを作成できます。 そして、ここに環境間の最初の違いがあります。Scalaプラグインは、Ideaで利用可能な新しいSBTプロジェクトを作成するためのウィザードを提供します： [File → New → Project → Scala → SBT]



。 しかし、Android Studioはウィザードを選択する機会を与えず、すぐに（別のビルドツールであるGradleに基づいて）ウィザードを起動します。 したがって、プロジェクトはSBT Studioにのみインポートできます。 基本的に、これは20秒の問題です。

ProGuardツール-ScalaをAndroidに適合させる



現時点では、最も単純なアプリケーションが実行されています。 たとえば、拡張を開始する場合は、少なくともbuild.sbt

問題が発生します。 Androidプラットフォームでは、参照の数に制限があります-64K。 Scalaコードの性質により、この制限にすぐに到達します。 そして、このライブラリをオンにするだけで、プロジェクトの開始時にエラーが発生します。



Error:Android Pre Dex: [scaloid_2.11-4.2.jar] trouble writing output: Too many method references: 81694; max is 65536.







この問題を解決するための標準的なアプローチ：



  1. ProGuardツールを使用してプロジェクトから未使用のコードを切り取ります。
  2. プロジェクトソースをいくつかの.dexファイルに分離します。各ファイルには、許容される数のメソッドが含まれます。 プロシージャの名前はMultiDexingです。


原則として、最初のステップでしか問題を解決できない場合は、それだけが使用されます。 最初のものでは十分でない場合、MultiDexingはすでに存在します。 このタスク内では、ProGuardで十分です。

このツールは、Androidファセットオプションでオンになります： [Project Structure → Facets → Android → Proguard → Run proguard ...]



。 デフォルトでは、基本設定のテキストファイルが既に指定されていることに注意してください：StudioにはSDK .../Android/sdk/tools/proguard/proguard-android.txt

 -dontwarn scala.** -dontwarn org.scaloid.** -keep class com.example.**
