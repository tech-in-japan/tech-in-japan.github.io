String.Internとは何ですか？ （収容プールに飛び込む...）


C＃で文字列を使用する場合、CLRはトリッキーなことを行います。これは、文字列インターンと呼ばれるものです。 これは、任意の文字列のコピーを1つ保存する方法です。 100個、さらに悪い場合は100万個の文字列変数に格納する場合、同じ値を使用すると、文字列値を格納するためのメモリが何度も割り当てられます。 文字列インターンは、この問題を回避する方法です。 CLRは、収容プールと呼ばれるテーブルをサポートします。 このテーブルには、プログラムの実行中にプログラムで宣言または作成された各行への一意のリンクが1つ含まれています。 .NET Frameworkは、インターンプールと対話するための2つの便利なメソッドを提供します： String.Intern（）およびString.IsInterned（） 。



String.Intern（）メソッドは非常に簡単な方法で機能します。 引数として文字列を渡します。 この行が既に抑留プールにある場合、メソッドはこの行へのリンクを返します。 まだ存在しない場合は、プールに行を追加し、そのリンクを返します。 以下に例を示します。 



  Console.WriteLine(object.ReferenceEquals( String.Intern(helloWorld), String.Intern(helloWorld2)));


HelloWorldとHelloWorld2が2つの異なる文字列オブジェクトを参照している場合でも、このコードはTrueを表示します。これらは両方とも文字列「Hello World」を含むためです。

しばらく停止します。 String.Intern（）を使用して少し理解する価値があります。これは、このメソッドが一見わずかに非論理的な結果をもたらす場合があるためです。 この動作の例を次に示します。 



  string a = new string(new char[] {'a', 'b', 'c'}); object o = String.Copy(a); Console.WriteLine(object.ReferenceEquals(o, a)); String.Intern(o.ToString()); Console.WriteLine(object.ReferenceEquals(o, String.Intern(a)));


コードを実行すると、コンソールに2行が出力されます。 最初のWriteLine（）メソッドはFalseを表示しますが、これはString.Copy（）メソッドが文字列の新しいコピーを作成し、新しいオブジェクトへのリンクを返すため理解できます。 しかし、最初にString.Intern（about.ToString（））を実行してからString.Intern（a）がaboutへのリンクを返すのはなぜですか？ それについて考えるために少し立ち止まってください。 さらに3行追加すると、これはさらに直感に反します。 



  object o2 = String.Copy(a); String.Intern(o2.ToString()); Console.WriteLine(object.ReferenceEquals(o2, String.Intern(a)));


これらのコード行は、新しいo2オブジェクト変数でのみ同じように見えます。 しかし、最後に、 WriteLine（）はFalseを出力します。 それで何が起こっているのでしょうか？



この小さな混乱は、 String.Intern（）と収容プールの内部で何が起こっているかを把握するのに役立ちます。 自分で理解する最初のことは、 ToString（）の文字列オブジェクトメソッドが常に自身への参照を返すことです。 変数oは、値 "abc"を含む文字列オブジェクトを指しているため、独自のToString（）メソッドを呼び出すと、この文字列へのリンクが返されます。 だからここで何が起こるかです。

最初は、「abc」を含む行番号1のオブジェクトを指します。 変数oは、「abc」も含む行番号2の別のオブジェクトを指します。 String.Intern（o.ToString（））を呼び出すと、行番号2への参照がインターンプールに追加されます。 これで、行番号2のオブジェクトが収容プールにある場合、いつでもパラメーター「abc」で呼び出したString.Intern（）は行番号2のオブジェクトへの参照を返します。

したがって、変数oとString.Intern（a）をReferenceEquals（）メソッドに渡すと、 String.Intern（a）は行番号2のオブジェクトへの参照を返したため、 Trueを返します。 ここで、新しい変数o2を作成し、 String.Copy（）メソッドを使用して別のStringオブジェクトを作成しました。 これは、行番号3のオブジェクトになり、文字列「abc」も含まれます。 String.Intern（o2.ToString（））の呼び出しは、「abc」がすでに存在するため、今回は収容プールに何も追加しませんが、行番号2へのポインターを返します。

したがって、このIntern（）の呼び出しは実際には行番号2への参照を返しますが、変数に割り当てる代わりに破棄します。 次のようなことができます： string q = String.Intern（o2.ToString（）） 。これにより、 q変数が行＃2のオブジェクトへの参照になります。 これが、最後のWriteLine（）が Falseを出力する理由です。これは、3行目のリンクと2行目のリンクの比較であるためです。



String.IsInterned（）を使用して、文字列がインターンプールにあるかどうかを確認します


もう少し逆説的な名前のメソッドがあります。これは、インターンされた文字列を操作するときに便利です： String.IsInterned（） 。 文字列オブジェクトへの参照を取ります。 この文字列が収容プールにある場合、収容された文字列の文字列への参照を返します;収容プールにまだない場合、メソッドはnullを返します 。

名前が少し非論理的に聞こえる理由は、このメソッドは「Is」で始まるが、多くのプログラマが期待するようにブール型を返さないためです。

IsInterned（）メソッドを使用する場合、null合体演算子- ??を使用して、文字列が収容プールにないことを表示すると便利です。 。 たとえば、次のように書く： 



  string o = String.IsInterned(str) ?? "not interned";


現在、 IsInterned（）の結果は、ゼロでない場合は変数oに戻り、収容プールに行がない場合はストリング「not interned」に戻ります。

これが行われない場合、 Console.WriteLine（）メソッドは空の行を出力します（ nullが検出されるとこのメソッドが行います ）。

String.IsInterned（）の仕組みの簡単な例を次に示します。 



  string s = new string(new char[] {'x', 'y', 'z'}); Console.WriteLine(String.IsInterned(s) ?? "not interned"); String.Intern(s); Console.WriteLine(String.IsInterned(s) ?? "not interned"); Console.WriteLine(object.ReferenceEquals( String.IsInterned(new string(new char[] { 'x', 'y', 'z' })), s));


最初のWriteLine（）ステートメントは、「xyz」がインターン メントプールにまだないため、コンソールに「not interned 」と表示されます。 2番目のWriteLine（）ステートメントは、 インターンメントプールに既に「xyz」が含まれているため、「xyz」を出力します。 また、3番目のWriteLine（）は、オブジェクトsが収容プールに追加されたオブジェクトを指すため、 Trueを出力します。



リテラルインターンを自動的に


メソッドの最後に1行だけ追加して、プログラムを再度実行することにより： 



  onsole.WriteLine(object.ReferenceEquals("xyz", ));


完全に予期しないことが起こります！

プログラムは「インターンしない」と表示することはなく、最後の2つのWriteLine（）メソッドはFalseを表示します！ 最後の行をコメントアウトすると、プログラムは期待どおりに動作します。 なぜ？！ プログラムの最後にコードを追加すると、その上のコードプログラムの動作がどのように変わりましたか？ これは非常に奇妙です！



これに初めて出会ったときは本当に奇妙に思えますが、本当に理にかなっています。 プログラム全体の動作を変更する理由は、コードにリテラル「xyz」が含まれているためです。 そして、プログラムにリテラルを追加すると、CLRはプログラムが開始する前であっても、それを自動的にインターンプールに追加します。 この行にコメントして、プログラムからリテラルを削除すると、収容プールには文字列「xyz」が含まれなくなります。

「xyz」はプログラムの起動時に収容プールに既にあることを理解します。この文字列はコード内でリテラルとして表示されるため、プログラムの動作のこのような変更はすぐに明らかになります。 String.IsInterned（s）はnullを返さなくなりました。 代わりに、リテラル「xyz」への参照を返します。これは、 ReferenceEquals（）がFalseを返す理由も説明します。 これは、文字列sが収容プールに追加されないためです（「xyz」はすでにプール内にあり、別のオブジェクトを指している）。



コンパイラはあなたが思っているよりも賢いです！


コードの最終行を次のように変更します。 



  Console.WriteLine( object.ReferenceEquals("x" + "y" + "z", s));


プログラムを実行します。 リテラル「xyz」を使用しているかのように機能します。 +は演算子ではありませんか？ これは、実行時にCLRで実行されるメソッドではありませんか？ その場合、リテラル「xyz」の内在化を防ぐコードが必要です。

実際、これは"x" + "y" + "z"をString.Format（ "{0} {1} {2}"、 'x'、 'y'、 'z'）に置き換えると起こります。 コードの両方の行は「xyz」を返します。 なぜ、 +演算子を連結して、文字列 "xyz"を使用しているように振る舞うのか、実行時にString.Format（）が実行されるのはなぜですか？

この質問に答える最も簡単な方法は、コード“ x” +“ y” +“ z”をコンパイルしたときに実際に得られるものを確認することです。



Program.cs： 



 using System; class Program { public static void Main() { Console.WriteLine("x" + "y" + "z"); } }


次のステップは、コンパイラーが実行可能なタイプのアプリケーションをコンパイルしたことを確認することです。 このために、MSIL逆アセンブラーであるILDasm.exeを使用します。 このツールは、Visual Studioのすべてのバージョン（Expressエディションを含む）と共にインストールされます。 また、ILの読み方がわからなくても、何が起こっているのかを理解できます。



Ildasm.exeを実行します。 64ビットバージョンのWindowsを使用している場合、次のコマンドを実行します。 "％ProgramFiles（x86）％\ Microsoft SDKs \ Windows \ v7.0A \ Bin \ Ildasm.exe" （引用符を含む）、 [スタート] >> [実行]ウィンドウから 、またはコマンドラインから。 32ビットバージョンのWindowsを使用している場合は、 「％ProgramFiles％\ Microsoft SDKs \ Windows \ v7.0A \ Bin \ ildasm.exe」コマンドを実行する必要があります。

Programクラスを展開し、 Main（）メソッドをダブルクリックします。 これらの手順の後、逆アセンブルされたメソッドコードが表示されます。







コード内のリテラル「xyz」の存在を確認するためにILを知る必要はありません。 ILDasmを閉じてから、「x」+「y」+「z」の代わりに「xyz」を使用するようにコードを変更すると、ILコードは完全に同じように解析されます。 これは、コンパイラがコンパイル時に「x」+「y」+「z」を「xyz」に置き換えるのに十分なため、「xyz」が常に返すメソッド呼び出しに余分な操作を費やす必要がないためです。 そして、リテラルがプログラムでコンパイルされると、CLRはプログラムの起動時にそれを収容プールに追加します。
