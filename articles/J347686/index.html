提案されたソリューション



コンパイルオプションの新しいグループ/ external：*を導入します。これは、「外部」ヘッダーファイルで機能します。 既存のサードパーティライブラリの多様性をよりよく反映するため、他のコンパイラで採用されている「 システムヘッダーファイル 」という名前より「 外部ヘッダーファイル 」という名前を優先しました。 さらに、C ++標準は[lex.header]セクションの外部ヘッダーファイルに既にアピールしているため、私たちの選択は非常に自然です。 ユーザーがそれらをマスターするのが簡単になるため、新しいキーを個別に説明するのではなくグループに結合しました。新しいキーの完全な構文は、既知のキーとの類推によって予測できます。 現時点では、グループは5つのパラメーターで構成され、2つのカテゴリーに分けられています（以下の関連セクションを参照）。



外部ヘッダーファイルのセットを定義するパラメーター



  • /外部：I <パス>
  • /外部：山括弧
  • /外部：env：<var>


外部ヘッダーファイルの診断動作を定義するパラメーター



  • /外部：W <n>
  • /外部：テンプレート-


2番目のグループには、 / external：w、/ external：Wall、/ external：Wv：<version>、/ external：WX [-]、/ external：w <n> <warning>、/などのパラメーターを追加できます。 external：wd <warning>、/ external：we <warning>、/ external：wo <warning>など これらは警告レベルの類似物として、または外部（カスタムではなく）ヘッダーファイルの標準キーとして使用できます。 これは実験的な機能であるため、有効にするには、最後までデバッグするまで/ Experimental：externalオプションを追加する必要があります。 新しいキーの機能を見てみましょう。



外部ヘッダーファイル



現在、ライブラリの作成者とユーザーは、ヘッダーファイルの場所を4つの方法で指定できます。アセンブリスクリプトへの追加の複雑さ、埋め込み可能性、および制御の程度が異なります。



  • / external：I <path> -GCC、Clang、およびEDGコンパイラーの-isystemまたは単に-i（小文字）のイデオロギー的類似物。外部ヘッダーファイルを含むディレクトリが設定されます。 パス内のすべての再帰サブディレクトリも外部と見なされますが、含まれるファイルを検索するディレクトリのリストに追加されるのはパスのみです。
  • / external：env：<var> -セミコロンで区切られた外部ヘッダーファイルがあるディレクトリのリストが保存される環境変数の名前を設定します。 このメソッドは、INCLUDEやCAExcludePathなどの環境変数に依存するビルドシステムを使用する場合に役立ちます。これらの環境変数は、 / includeスイッチでチェックする必要のない外部インクルードファイルとファイルのリストをそれぞれ設定します。 ユーザーは、単純に/ external：env：INCLUDEおよび/ external：env：CAExcludePathを記述でき、 / external：Iオプションを使用してディレクトリの長いリストを渡すことはできません。
  • / external：山括弧 - #include <>コマンド（ #include ""ではなく ）でインクルードされたすべてのヘッダーファイルを外部として解釈できます。
  • #pragma system_header-ヘッダーファイルの埋め込みラベル-ライブラリ作成者は、特定のヘッダーファイルを外部としてマークできます。


外部ヘッダーファイルの警告レベル



/ external：W <n>オプションにより、ユーザーは外部ヘッダーファイルのデフォルトの警告レベルを設定できます。 このような包含物を設計の類似物でラップします。 



#pragma warning (push, n) //   n    #pragma warning (pop)


このキーと外部ヘッダーファイルを指定する優先方法を組み合わせて使用​​すると、これらのファイルに対して発行された警告を完全にオフにできます。



例：



外部ヘッダーファイル：some_lib_dir / some_hdr.hpp 



 template <typename T> struct some_struct { static const T value = -7; // W4: warning C4245: 'initializing': // conversion from 'int' to // 'unsigned int', signed/unsigned // mismatch };


ユーザーコード：my_prog.cpp 



 #include "some_hdr.hpp" int main() { return some_struct<unsigned int>().value; }


このコードを次のようにコンパイルする場合：



cl.exe / I some_lib_dir / W4 my_prog.cpp



コメントに記載されている第4レベルの警告メッセージC4245がヘッダーファイルに発行されます。 パラメーターを使用したコンパイル：



cl.exe /実験的：外部/外部：W0 / I some_lib_dir / W4 my_prog.cpp



外部ヘッダーファイルを指定しなかったため、効果はありません。 パラメーターを使用したコンパイル：



cl.exe /実験的：外部/外部：I some_lib_dir / W4 my_prog.cpp



また、外部ヘッダーファイルの警告レベルが設定されておらず、デフォルトで/ Wパラメータで指定されたレベルに対応しているため（この場合は4）、効果はありません。 外部ヘッダーファイルの警告を抑制するには、これらのファイルへのパスとそれらの警告のレベルの両方を指定する必要があります。



cl.exe /実験的：外部/外部：I some_lib_dir /外部：W0 / W4 my_prog.cpp



このコマンドを使用すると、some_hdr.hppファイルのすべての警告を取り除き、my_prog.cppファイルの警告のみを残すことができます。



内部コードと外部コードの両方に影響する警告



外部ヘッダーファイルのアラートレベルを設定することができれば素晴らしいと思いますが、内部ユーザーファイルにも関連する一部のメッセージを除外するリスクがあります。 includeとともにpush / popプラグマディレクティブのみを使用すると、ユーザーコードでテンプレートをインスタンス化する際に多くの有用な警告が消えます。 このような警告は、ユーザー側の問題の存在を示している可能性があり、テンプレート内の特定の型のみを置換する場合に明示されます（たとえば、<type_traits>からconstまたは＆を削除する型変換を使用するのを忘れた場合）、この問題を報告する必要があります。 このバージョンより前は、メッセージが発行された時点での警告レベルは語彙分析の​​みに基づいて決定されていましたが、問題領域は別の範囲にある可能性があります。 どうやら、テンプレートのインスタンス化ポイントで警告レベルを比較して、どの警告を発行するべきか、どの警告を発行すべきでないかを判断することは理にかなっています。



定義が外部ヘッダーファイルにあるテンプレートの警告を誤って消さないようにするために、上記の単純化されたロジックからテンプレートを除外できるようにしました。これは、 / external：templates-オプションと/ external：W <n>を渡すことで実行できます。 同時に、テンプレート定義が含まれて警告が発行される時点での現在の警告レベルだけでなく、テンプレートのインスタンス化のシーケンスのすべての時点での警告レベルも調べます。 警告レベルは、すべてのレベルのメッセージセット全体に対してラティスを形成します（ただし、一度に複数のレベルで警告を発行することがあるため、理想的ではありません）。 このラティスに関して特定のプログラムポイントで許可されるべき警告を定義するスーパーセットは、インスタンス化のチェーンを通じて各プログラムポイントで許可されるメッセージを組み合わせることによって取得されます。 これはまさに/ external：template-キーの目的であり、外部ヘッダーファイルに保存され、ユーザー（つまり、内部）コードに実装されているテンプレートに警告を表示できます。



cl.exe / Experimental：external / external：I some_lib_dir / external：W0 / external：templates- / W4 my_prog.cpp



このコマンドを使用すると、外部ヘッダーファイルの警告レベルが0であっても警告を表示できます。



警告の抑制と強制



上記のメカニズムは、それ自体では警告をオンまたはオフにしません。指定されたファイルセットのデフォルトレベルを設定するだけで、警告をオン、オフ、および抑制するための標準メカニズムが作用します。



  • / wdNNNN、/ w1NNNN、/ weNNNN、/ Wv：XX.YY.ZZZZなど
  • #pragma warning（disable：4507 34; once：4385; error：4164）
  • #pragma warning（push [、n]）/ #pragma warning（pop）


また、 / external：templates-を使用する場合、インスタンス化ポイントで警告を抑制することができます。 前述の例では、ユーザーは/ external：templates-キーのために発行された警告を明示的に抑制することができます。 



 int main() { #pragma warning( suppress : 4245) return some_struct<unsigned int>().value; }


反対側で作業しているライブラリ作成者は、同じメカニズムを使用して、特定のレベルの警告の一部またはすべてを強制的に含めることができます。これらの警告は非常に重要であり、 / external：W <n>スイッチで抑制すべきではないと考えている場合です。



例：



外部ヘッダーファイル：some_lib_dir / some_hdr.hpp 



 #pragma warning( push, 4 ) #pragma warning( error : 4245 ) template <typename T> struct some_struct { static const T value = -7; // W4: warning C4245: 'initializing': // conversion from 'int' to // 'unsigned int', signed/unsigned // mismatch }; #pragma warning( pop )


上記のようにライブラリヘッダーファイルを変更することにより、ライブラリの作成者は、ユーザーが/ external：W <n>パラメーターで指定したレベルに関係なく、このファイルがレベル4でチェックされることを確認できます-コンパイラーは引き続きすべてのレベル4警告を発行しますとアップ。 さらに、そこに示されているように、特定の警告を強制的に設定して、常にエラーと見なされるようにしたり、このヘッダーファイルに対して一度オフにしたり、抑制したり、一度だけ発行したりすることができます。この場合も、ユーザーはこの設定をバイパスできません。



制限事項



現在のバージョンでは、コンパイラオプティマイザー（スキャナーではなく）によって発行された場合、外部ヘッダーファイルで警告をトリガーできます。 このような警告は通常C47XX形式ですが、オプティマイザーによってすべてのC47XXメッセージが発行されるわけではありません。 原則として、何らかの警告がデータフローまたは制御の分析を必要とする場合、おそらく実装のオプティマイザーからのものであり、これまでのところ新しいメカニズムを使用して抑制することはできません。 この問題については知っていますが、中間コードの表示に大きな変更が伴うため、ソリューションはVisual Studioの次のメジャーアップデートの前には表示されません。 ただし、このような警告は、/ wd47XXスイッチを使用して、標準的な方法で引き続き無効にすることができます。



さらに、最初にユーザーのフィードバックを調べるため、この実験的な機能はまだ警告に統合/分析されていません。 /分析の警告はレベルで分割されないため、現在のロジックとそれらを組み合わせる最適な方法も探しています。

壊れたウィンドウの理論との類推に戻ると、ライブラリ全体のエコシステムに対するイノベーションの影響を曖昧に評価していることに注意する必要があります。 一方で、ユーザーはライブラリのエラーを問題と見なさなくなり、ユーザーに報告したり、自分で編集したりする可能性が低くなるため、ユーザーに損害を与える可能性があります。 一方、ユーザーは自分のコードをより細かく制御できるようになり、コードに対してより厳しい要件を設定できるようになります。 以前は、制御できないライブラリがそれらを阻止していましたが、現在は緩和できます。



前述の副作用の可能性を認識していますが、サードパーティのコードのエラーを修正することはプログラマーの主なタスクではありません。なぜなら、彼は自分で作業する必要があるコードを持っているからです。プロセス。ファイルに対してのみ/ WXオプションを有効にできないため。 しかし、さらに重要なことは、ライブラリに対するマイナスの影響が別の有用な結果によって相殺されることを確信しています。



開発者がサードパーティライブラリへの警告を非表示にできるようにすることで、独自のコードにさらに注意を払い、改善し、可能な限り最大限のレベルで警告を完全にクリアすることを推奨します。 サードパーティライブラリの作成者は同じ開発者であり、このチェーン内の別のリンクです。つまり、サードパーティコードの依存関係に気を取られないようにする機会を与え、独自のコードの品質を注意深く監視し、可能な限り最大の警告レベルでコンパイルすることをお勧めします。 などなど。 なぜこれが重要なのですか？ 実際、ソフトウェア開発プロセスの現在の構造では、依存関係チェーンに沿って移動するにつれて警告の数が雪崩のように増加し 、このチェーンに進むにつれて警告に対処することが難しくなります。 開発者は自重で「故障」し、状況を修正する試みを停止します。 コードを他の人のコードから分離できる場合、チェーン内の各開発者は、この警告の雪崩を止める（結果をブロックする）ために自由に資金を利用できます。 彼は自分のサイト、したがってチェーン全体への影響を最小限にするインセンティブを持っています。 もちろん、これは投機的な結論ですが、この影響は図書館への悪影響に劣らないと考えています。
