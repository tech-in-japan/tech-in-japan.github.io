QtアプリケーションをMac OS Xに統合する（CocoaとObjective-C ++を使用）

すべての人に良い一日を！



最近、Mac OS Xでウィンドウタイトルをカスタマイズすることについて書いたのですが、QtとCocoaの相互作用についてさらに詳しく書くようにというリクエストがありました。 トピックを少し広げて、Qtを使用して作成されたアプリケーションのMac OS Xへの統合について説明できると思います。Qtfor Cocoaがこの場合に使用されることに言及します。Qtfor Carbonを使用する場合は、炭素のみを使用する必要があります。 しかし、それは時代遅れであり、極端な場合にのみ使用する価値があります。



通常のQtプログラムには、Apple HIGとの多くの矛盾があります。 より正確には、すべてのプログラムが追加機能を必要とするわけではないので、持っているかもしれません。 たとえば、すべてのプログラムでドックのアイコンの上にバッジを付けたり、ドックメニューを展開したり、ポピーメニューの一部の機能を削除/複製する必要はありません。



しかし、この機能が必要な場合はどうでしょうか？ ドック（Skypeなど）で通知の数を表示する必要がある場合は、ドックのアイコンをクリックして処理し、メニュー項目をドックに追加し、通常のメニューを使用して、一般的に、プログラムをMac OSのネイティブのようにしますか？ これのいくつかは、通常または半文書化されたQt関数を使用して実行できます。また、Cocoaと、したがってObjective-Cを使用してのみ実行できるものもあります。



Objective-C ++が役立ちます！



それはどんな動物で、何と一緒に食べますか？ 本質的に、これはObjective-CとC ++クラスを1つのソースファイルに結合する機能です。 さらに、見出しの拡張子は標準（.h）のままですが、ソースについては、拡張子.mmを指定する必要があります。これにより、コンパイラはそれを食べてチョークしません。



ここで、Qtで作成された（おそらく大きな）プロジェクトがあると想像してください。 当初はWindowsまたはLinuxで書かれていましたが、今ではマクロに転送する必要があります。そのため、ケシの栽培者がこの怪物を見たときに鼻にしわが寄らないように、美しく便利です。



手始めに、Apple HIGのプログラムインターフェースを調整する必要があることは明らかです。ただし、これはこの記事の範囲外です。異なるOS向けに異なるコードをコンパイルできる便利なQ_WS_ *定義についてのみ言及します。 アプリケーションを新しい環境に適応させる方法について説明します（またはQtでMacアプリケーションをゼロから作成する方法-目標によって異なります）。



順番に行きましょう。



一般的な統合





まず、プログラムに名前とアイコンを付けます。 いいえ、バンドルの名前ではなく、アプリケーションメニューに表示される名前です。 これを行うには、qmakeが生成するものではなく、独自のInfo.plistファイルが必要です。 これは、.proの1行で実行されます。 



macx: QMAKE_INFO_PLIST = MyInfo.plist


.plistでは、次のように記述します。 



 <?xml version="1.0" encoding="UTF-8"?> <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd"> <plist version="1.0"> <dict> <key>LSHasLocalizedDisplayName</key> <true/> <key>CFBundleIconFile</key> <string>myicon.icns</string> <key>CFBundlePackageType</key> <string>APPL</string> <key>CFBundleGetInfoString</key> <string>Created by Qt/QMake</string> <key>CFBundleSignature</key> <string>????</string> <key>CFBundleExecutable</key> <string>MyAppName</string> <key>CFBundleIdentifier</key> <string>com.mycompany.myapp</string> <key>NOTE</key> <string>This file was generated by Qt/QMake.</string> </dict> </plist>


もちろん、「MyAppName」と「com.mycompany.myapp」の代わりに、プログラムの英語名とそのバンドル識別子を記述します。 なぜ英語？ はい、別のファイルにローカライズするため。 これは、リストの最初のパラメーターLSHasLocalizedDisplayNameによって示されます。 ディレクトリ「ru.lproj」とその中にファイルInfoPlist.stringsを作成します。 このファイルでは、次のように記述します。 



 /* Localized versions of Info.plist keys */ CFBundleName = "  "; CFBundleDisplayName = "";


ここでは、バンドルのローカライズされた名前と、アプリケーションメニューに表示される名前を指定する必要があります。 これを機能させるには、プログラムをインストールするときにこのディレクトリをAppName.app/Contents/Resourcesにコピーする必要があります。.proファイルでINSTALLSを指定することをお勧めします。詳細については、qmakeのドキュメントを参照してください。 スクリーンショットは、バンドル自体にラテン語の名前があるにもかかわらず、アプリケーションメニューの名前がロシア語であることを示しています。





プログラムアイコン（MyInfo.plistファイルのmyicon.icnsを参照）を設定するには、グラフィックエディターで自分で作成するか、 プログラムまたはオンラインサービスを使用して.icoまたは一連の.pngから変換できる.icnsファイルが必要です。 アイコンの見栄えを良くするには、512x512、256x256、128x128、64x64、48x48、32x32、16x16のいくつかのサイズにすることをお勧めします。 システム自体が、どの状況でどのサイズを表示するかを選択します。 アイコンのあるファイルもリソースにインストールする必要があります。 インストールする最も簡単な方法は、.proファイルに以下を書き込むことです。 



 macx: ICON = myicon.icns


コード分​​離





Objective-C ++には多くの制限があります。 たとえば、.cppソースファイルにObjective-Cクラスインターフェイスの宣言にヘッダーを含めることはできません。コンパイラがチョークするからです。 Mac OS X専用に設計された新しいプロジェクトの場合、これは制限ではありません。すべてのC ++コードを.mmファイルに保持し、楽しむことができるからです。 しかし、Qtの意味はクロスプラットフォームなので、ソースコードはまだ.cppファイルにあると想定します。



出口は簡単です。 C ++でObjective-C呼び出しとクラスのラッパーを作成する必要があります。 私はこの構造を選択しました。MacOS Xとの統合を提供するQt / C ++クラスがあります。それ自体でいくつかの作業を行い、Cocoa / Objective-Cを積極的に使用するプライベートクラス「デリゲート」へのデリゲートを行います。 同時に、次のファイルを取得します。



* myclass.h-メインクラスのヘッダー

* myclass.cpp-実装

* myclass_p.h-プライベートクラスヘッダー（Objective-Cインターフェイスなし）

* myclass_p.mm-プライベートクラスソース、Objective-Cクラスのインターフェイスと実装、ヘッダーなどを含めることができます。



したがって、C ++とObjective-C ++を明確に区別します。



ところで、Objective-C ++が機能するためには、.proファイルでそれを使用するすべてのヘッダー/ソースをOBJECTIVE_HEADERSおよびOBJECTIVE_SORCESセクションに配置する必要があります。 そしてもちろん、macxブロック内でそれを行います：{}。 また、Cocoaを使用する場合は、このフレームワークをプロジェクトに追加する必要があります。 .proで書き込みます。 



 macx: QMAKE_LFLAGS += -framework Cocoa


そして今、楽しみが来るでしょう。



Dockを使用する





ドックを操作する5つの主要な機能を考えてみましょう。バッジの追加、オーバーレイの追加、ドック内のアイコンのクリック処理、ドック内のプログラムアイコンの「トス」、独自のメニューの追加。 Qtを使用すると、最後の問題だけでなく、文書化が不十分な関数qt_mac_set_dock_menu（QMenu *）でも解決できます。 また、外部から宣言する必要があります。 



 extern void qt_mac_set_dock_menu(QMenu *); // Qt internal function


個人的な経験に基づいて、「ネイティブ」マックメニューと比較して、メニューにはいくつかの制限が課されています。



*非アクティブ（無効）メニュー項目は何らかの理由でアクティブになります

* QMenuはaboutToHideおよびaboutToShow信号を出力しません

*要素をインデントすることはできません

*メニューの最初のQActionがセパレーターである場合、それは表示されます（他のすべてのQMenuマニフェストとは異なります）



したがって、それに適応する必要があります。 もちろん、Objective-C / Cocoaですべてを実行できますが、QActionとネイティブメニュー項目をマッピングする独自のメカニズムを作成する必要があります。 ただし、これらの制限を排除する必要性が非常に高い場合にのみ、これを行うことは理にかなっています。





次に、ドックのクリックを検討します。 Cocoaで記述した場合、問題はありません。AppDelegateでapplicationShouldHandleReopen：hasVisibleWindows：メソッドを実装すれば十分です。 しかし、Qtの腸で作成されたプログラムデリゲートに直接アクセスすることはできません。 したがって、 ランタイムのマジックを使用して、このメソッドの実装を追加します 。 まず、必要なものを実装する関数を宣言します。 これを行うには、プライベートクラスをシングルトンに変換し（とにかく、このクラスの複数のオブジェクトは必要ありません）、次の関数を記述します。 



 void dockClickHandler(id self, SEL _cmd) { Q_UNUSED(self) Q_UNUSED(_cmd) MyPrivate::instance()->emitClick(); }


関数は、デリゲートに埋め込まずに死んでいます。 これをためらわないでください！ 



 MyPrivate::MyPrivate() : QObject(NULL) { Class cls = [[[NSApplication sharedApplication] delegate] class]; if (!class_addMethod(cls, @selector(applicationShouldHandleReopen:hasVisibleWindows:), (IMP) dockClickHandler, "v@:")) NSLog(@"MyPrivate::MyPrivate() : class_addMethod failed!"); } void MyPrivate::emitClick() { emit dockClicked(); }


ここで、アプリケーションのデリゲートクラスを取得し、メソッドをその場で追加します。 同時に、emitClick（）メソッドを実装します。このメソッドは、クリックに関するQtシグナルを発行します。 実際、これですべてです。たとえば、メインプログラムウィンドウなどを表示できるドックをクリックするだけです。



次に、ドックのプログラムアイコンを放り投げることができます。 私の最初の考え：「QApplication :: alert（QWidget *）は同じことをすることができます！」その考えは真実ですが、時期尚早に楽観的です。 問題は、Mac OS X 10.6の場合です。*この関数は正常に機能しますが、10.7の場合です。*何らかの理由で不要です（これはQt 4.7.xが公式にLionをサポートしていないためかもしれませんが、 4.8では修正されます）。 私はこれがなぜ起こるのか理解できず、Cocoaにトスを書いただけです。 



 void MyPrivate::requestAttention() { [NSApp requestUserAttention: NSInformationalRequest]; }


このコードはQt-shn alert（）を複製しますが、Mac OS Xのすべてのバージョンで動作します。他のシステムでも、alert（）を使用できます。



次に、バッジを扱いましょう。 誰かが知らない場合、これはドックのプログラムアイコンの上にテキストが表示された赤い円です。 たとえば、Mail.appの未読メッセージの数を表示するために使用されます。 Appleのドキュメントでは、これは次のように行う必要があると書かれています。 



 [[NSApp dockTile] setBadgeLabel: badgeString];


ここで、badgeStringはNSString *型です。 ええ、これは最初の不便です！ Qtでは、QStringは通常操作されます。つまり、文字列の特定の「コンバーター」を記述する必要があります。 



 // warning! nsstring isn't released! NSString * nsStringFromQString(const QString & s) { const char * utf8String = s.toUtf8().constData(); return [[NSString alloc] initWithUTF8String: utf8String]; }


コード（およびコメント）からわかるように、返された文字列は解放されないため、呼び出しコードでそれを行う必要があります（QtはARCを使用しないため、メモリを監視します）。



これで、ドックのバッジに必要な行を表示するプライベートクラスの関数を作成できます。 



 void MyPrivate::setDockBadge(const QString & badgeText) { NSString * badgeString = nsStringFromQString(badgeText); [[NSApp dockTile] setBadgeLabel: badgeString]; [badgeString release]; }




ドックを使用する最後の側面は、任意のオーバーレイをドックに追加することです。 これは、次のコードを使用して行われます。 



 [[NSApp dockTile] setContentView: view];


ここでのビューはNSView *です。 しかし、私たちはQtで作業しています！ そのため、オーバーレイにQWidget *を配置する必要があります。 NSViewをQWidgetから取得する方法は？ 簡単な関数を書きます： 



 NSView * nsViewFromWidget(QWidget * w) { return (NSView *)w->winId(); }


簡単です。Qtの作成者は、ほぼすべての作業を行ってくれました。



しかし、残念ながら、残念なことに私たちは待っています。QWidgetから取得したNSViewは、ドックへのインストールには適していません。 つまり、それは置かれ、NSDockTileはそれを食べますが、ドック内のウィジェットのコンテンツの代わりに、空のスペースが形成されます。 理由はわかりません。 しかし、Qt Creatorでさえ、ドックの進行状況バーは、このために特別に作成された純粋なNSViewを介して正確にハングアップします。 そのため、独自のオーバーレイが必要な場合は、Cocoaでビューを作成してもかまいません。 ああ。



メニューを操作する





ポピーメニュー（トップパネルのメニュー）に進みましょう。 デフォルトでは、Qtプログラムは作成しません（システム機能を備えた標準のアプリケーションメニューを除く）。 Qt開発者にとって最初に思い浮かぶのはQMenuBarです。 確かに、それに関するドキュメントは、ポピーメニューの機能を実行できると述べています。 ただし、2つのオプションがあります。プログラムウィンドウごとに個別のQMenuBarを作成するか、1つのグローバルを作成します。 否定できない利点があるため、2番目のオプションを選択します。



繰り返しになりますが、ドキュメントに基づいて、親がゼロのQMenuBar、つまりQMenuBar * macMenuBar = new QMenuBar（NULL）を作成する必要があります。これはグローバルになります。 より正確には、こうして作成された最初のメニューバーがグローバルになります。



そして今-手で単調な仕事がたくさん。 「編集」、「ファイル」、「ヘルプ」メニューなどを作成します。 これはかなり退屈な仕事ですが、それなしではプログラムは良く見えません。 さらに、標準のショートカット⌘Wおよび⌘Mは、それぞれウィンドウを閉じたり最小化したりしません。 また、自分で行う必要があります（少なくとも、wellQはすぐに動作します）。



Mac OS XのQActionのいくつかの機能に注目します。それらにショートカットを指定する場合、「Ctrl + M」をQKeySequenceコンストラクターに渡して⌘Mショートカットを作成する必要があります。 そして一般に、ケシの下のQtの⌘キーはCtrlのようにどこにでも行き、CtrlキーはMetaのように行きます。 おそらく、プログラムの移植性を高めるためです。



さて、メニューを作成するこのアプローチの制限について。



*メニュー項目をインデントすることはできません

* QMenuはaboutToHide信号を送信しません（aboutToShowのみ）

*次のバグが発生します。[ヘルプ]メニューが「ヘルプ」と呼ばれる場合、メニュー項目を検索するためのシステムブロックが自動的に作成されますが、ロシア語で命名されている場合、現在のロケールがロシア語であっても、これは発生しません。 このグリッチを取り除く方法は、私はまだ見つけていません。

 id _class = [NSThemeFrame class];
