AERODISK ENGINE N2ストレージクラッシュテスト、強度テスト



アクトン！ 記事を読んだ後、あなたは言うことができます：もちろん、ベンダーは、すべてが「強打」、温室条件などでうまくいくように自分自身をチェックします 私は答えます：種類の何もない！ 私たちの外国の競合他社とは異なり、私たちはあなたの近くにいます。あなたはいつでも（モスクワまたは中央委員会で）私たちに来て、どんな方法でもストレージシステムをテストできます。 したがって、結果を世界の理想的な絵に合わせるのはあまり意味がありません。 とても簡単に確認できます。 歩くのが面倒で時間がない人のために、リモートテストを組織することができます。 このための特別なラボがあります。 お問い合わせ。







ACHTUNG-2！ このテストは負荷テストではありません ここでは、フォールトトレランスのみに関心があります。 数週間以内に、より強力なスタンドを準備し、ここに結果を公開することでストレージシステムの負荷テストを実施します（ところで、テストの希望は受け入れられます）。







それでは、休憩に行きましょう。







テストスタンド



スタンドは次の鉄で構成されています。







  • 1 x Aerodisk Engine N2ストレージ（2コントローラー、64GBキャッシュ、8xFCポート8Gb / s、4xイーサネットポート10Gb / s SFP +、4xイーサネットポート1Gb / s）; 次のディスクがストレージシステムにインストールされます。
  • 4 x SAS SSDディスク900 GB;
  • 12 x SAS 10kドライブ1.2 TB;
  • 1 x Windows Server 2016搭載物理サーバー（2xXeon E5 2667 v3、96GB RAM、2xFCポート8GB / s、2xイーサネットポート10GB / s SFP +）;
  • 2 x SAN 8Gスイッチ;
  • 2 x LAN 10Gスイッチ;


FCおよびイーサネット10Gを介してスイッチを介してサーバーをストレージに接続しました。 下のスタンドのスキーム。









MPIOやiSCSIイニシエーターなどの必要なコンポーネントは、Windows Serverにインストールされます。

ゾーンはFCスイッチで構成され、対応するVLANはLANスイッチで構成され、MTU 9000はストレージポート、スイッチ、およびホストにインストールされます（これを行う方法はドキュメントに記載されているため、ここではこのプロセスを説明しません）。







テスト方法



クラッシュテスト計画は次のとおりです。







  • FCおよびイーサネットポートの障害チェック。
  • 電源障害チェック。
  • コントローラー障害のチェック。
  • グループ/プールのディスク障害を確認します。


すべてのテストは、IOMTERで生成する合成負荷条件で実行されます。 並行して、同じテストを実行しますが、ストレージシステムに大きなファイルをコピーする条件で行います。







IOmeterの構成は次のとおりです。







  • 読み取り/書き込み-70/30
  • ブロック-128k（ストレージシステムを大きなブロックで濡らすことにしました）
  • スレッドの数は128です（これはワークロードに非常に似ています）
  • 完全ランダム
  • ワーカー数-4（FC用に2、iSCSI用に2）








テストには次のタスクがあります。

  1. 合成ロードとコピープロセスが中断されず、さまざまな障害モードでエラーが発生しないことを確認します。
  2. ポート、コントローラーなどの切り替えプロセスが十分に自動化されており、障害が発生した場合に管理者のアクションを必要としないことを確認してください（つまり、フェールオーバーでは、もちろん、フェールバックの話はありません）。
  3. 情報がログに正しく表示されていることを確認してください。

所有するすべてのドライブを使用して、ハイブリッドグループをセットアップします。 2つのSSDディスクがキャッシュに追加され、2つのSSDディスクが追加のストレージレベル（オンライン層）として追加されます。 グループ内の3台のディスクの障害を一度にチェックするために、12台のSAS10kディスクをRAID-60P（トリプルパリティ）にグループ化しました。 1つのディスクがオートコレクト用に残されました。













2つのLUN（FC上に1つ、iSCSI上に1つ）を接続しました。









両方のLUNはEngine-0コントローラーによって所有されています。

テスト番号4。グループ内の3つのディスクの障害



このテストの前に、追加の準備手順を実行しました。 事実、ENGINEストレージは非常に便利な機能を提供します-異なる再構築（再構築）ポリシー。 以前、TS Solutionはこの機能について書きましたが、その本質を思い出してください。 ストレージ管理者は、再構築中にリソース割り当ての優先順位を指定できます。 または、I / Oパフォーマンスの方向、つまり、より長い再構築ですが、パフォーマンスの低下はありません。 または、再構築速度の方向では、生産性が低下します。 またはバランスの取れたオプション。 ディスクグループの再構築中のストレージのパフォーマンスは常に管理者にとって頭痛の種なので、I / Oパフォーマンスの方向に偏りを持たせ、再構築の速度を犠牲にしてポリシーをテストします。









次に、ドライブの障害を確認します。 また、LUN（ファイルとIOMETER）での記録も有効にします。 トリプルパリティグループ（RAID-60P）があるため、システムは3台のディスクの障害に耐える必要があり、障害後は自動交換が機能する必要があり、1台のディスクが障害が発生したディスクの代わりにRDG内に存在し、再構築が開始される必要があります。







始めます。 最初に、ストレージインターフェイスを使用して、引き出したいディスクを強調表示します（自動交換ディスクを取り外さないように）。













アイロンの表示を確認してください。 すべてが正常で、強調表示された3つのドライブが表示されます。

コピーインジケーター（キャッシュがウォームアップされているため、最初よりも高い）とIOMETERは、ディスクを引き出して再構築を開始するときに（5〜10％以内）あまり変化しません。







ストレージを見ます。









グループのステータスでは、再構築のプロセスが開始され、完了に近づいていることがわかります。













RDGスケルトンは、2つのディスクが赤のステータスにあり、1つがすでに交換されていることを示しています。 オートコレクトディスクはなくなり、3番目に故障したディスクを置き換えました。 Rebildは数分間実行され、3台のディスクに障害が発生してもファイルの記録は中断されず、I / Oパフォーマンスはそれほど変化しませんでした。















ドライブ障害テストは間違いなく成功しています。







おわりに

  • FCポート障害チェック-成功
  • イーサネットポート障害チェック-成功
  • コントローラー障害チェック-成功
  • 電源障害チェック-成功
  • グループ\プールのディスク障害を確認します-正常に
