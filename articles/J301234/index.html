TierIIIデータセンターでの顧客のバックアップの配置方法





最もシンプルなのはS3ストレージです。 この場合、コピーのロジックおよび編成なしで、スペースのみを提供します。 暗闇の中でクラブを打つような、安くて、シンプルで、信頼できる。 S3では、Veritas NetBackup / Backup Exec、CommVault、Veeam、EMC Networkerなど、ほとんどのバックアップ製品をコピーできます。このソリューションを完了するには約2時間かかります。



2番目のアプローチは、完全バックアップインフラストラクチャです。 実質的にハードウェアとソフトウェアの一部をレンタル用に顧客に提供します。 この場合、顧客はストレージだけでなく、コピーロジックのソフトウェア組織を含むバックアップサービス全体を受け取ります。 たとえば、顧客がVolochaevskayaのデータセンターにあり、サービスを使用している場合、Avamarコンプレックスはインフラストラクチャからバックアップを収集し、必要に応じて同じデータセンターのコピーと別のサイトへのコピーの両方を作成します。 さらに、システムにはAvamarの内部保護機能があります。システムの金型は1日2回撮影され、重複排除、圧縮されます。



もちろんすべてが優れていますが、客観的な理由（誰もが危機をなんとか感じることができた）のために、より予算的で機能的なオプションを探し始めました。



言及したので、Avamarから始めましょう。



ローカルインフラストラクチャ外のバックアップストレージタスクを解決するために、EMC Avamar PAKをインストールしました。 スキームは古典的です。クライアントアプリケーションがコピーされたマシンにインストールされ、オペレーティングシステムおよびソフトウェアと統合され、データを収集し、重複排除し、データセンターに送信します。 コピーは常にインクリメンタルです。最初に、システムはデータの完全なコピーを作成し、次にブロックのみを変更します（通常、合計ボリュームの1〜3％）。 これにより、チャネルの負荷が大幅に削減され、バックアップウィンドウが短縮されます。 アクセスに関しては、これらは合成完全バックアップです。



長所：

  • 重複排除と圧縮による高速コピー。
  • 良い価格;
  • データセンターのインフラストラクチャを仮想マシンに復元する機能。

  • バックアップコストは、資本コストで急増するのではなく、要件が増えるにつれて増加します。
  • 顧客側の最小限の不安とサポート作業。 お客様と一緒に、初期セットアップを実行し、システムのステータスを監視します（これは無料です）。

Compressorデータセンターは、Avamar PACを含むインフラストラクチャを展開し、vcloud（CROCクラウドインフラストラクチャの肩の1つ）およびCROCクラウド（KVMベースのクラウドソリューション）と統合されています。



短所：

  • ローカルサイトでのバックアップと復元は、インターネットチャネルの品質に完全に依存します。




顧客は多くの場合、昨日削除された別の手紙、特定のファイル、または何か他のものなど、バックアップに小さなものが必要です。 ユーザーが誤ってデスクトップで上書きした「1.txt」ファイルがあるため、画像全体を展開するのはかなり愚かなことです。 PAC Avamarは、ソフトウェア管理者に便利なコンソールと、アプリケーションおよびユーザーファイルを操作するための優れたGUIを提供します。 彼ら自身が、バックアップ管理者を引き出さずにデータを見つけて復元できます。 当然、アクセス権は区別されます。



安くて陽気な



さまざまな実装オプションを評価して、次のソリューションに徐々に移行しました-ターンキーターンキーバックアップソリューションを顧客のインフラストラクチャに提供するとき：







CrocBackAppは、ディスク容量が大きく、OEMのCommVaultがインストールされている外部ディスクシェルフを接続できるサーバーです。 顧客の要求に応じて、サーバーを1つのシャーシにクラスター化し、一般に、顧客のニーズに合わせてほぼすべてのアーキテクチャソリューションを構築できます。



テストが進むにつれて、このようなソリューションのもう1つの重要な利点が明らかになりました。セットアップの容易さと企業環境への統合です。 つまり、顧客はCrocBackAppをインストールした状態でアイロンを取り、サイトに配置して起動し、保護されたマシンでウィザードを使用してクライアントをインストールします。



さらに、ハードウェアとソフトウェアの購入量によりおいしい割引を実現しています。このようなソリューションを大規模ベンダーから自分で組み立てたり購入しようとすると、より高価になります。



バックアップをリモートサイトに保存するタスクに加えて、頻繁に発生する別のタスクは、バックアップだけでなく、リモートデータセンターの柔軟なインフラストラクチャの編成です。 ここでは、「サービスとしてのバックアップ」ではなく、実際には「サービスとしての災害復旧」をお客様に提供しています。 サービスの魅力は、通常モードでは、顧客が実際にデータの保存とリモートデータセンターへの送信に対してのみ支払うことであり、必要に応じてインフラストラクチャ全体を消費し始め、数分で復元できることです。



同様のサービス-「サービスとしての災害復旧」-さまざまな方法で行います。



  1. クラウドとプライベートクラウドまたは顧客インフラストラクチャの間には直接レプリケーションがあります。 VMware、Hyper-V、KVMクラウドがあり、ハイパーバイザー（vSphereレプリケーションなど）を使用してデータを直接複製できます。 これは、最も簡単で手頃な複製方法です。
  2. バックアップからクラウドインフラストラクチャにインフラストラクチャを復元します。 さまざまなメーカーのソリューションがあります。 Veeam Cloud Connectレプリケーションがあり、WANチャネルを介してインフラストラクチャにデータまたはバックアップをレプリケートできます。 メインバックアップソリューション-Avamar-の一部として、異なるクラウド間（たとえば、KVMのVmware、およびその逆）の異なる環境-p2v、v2pからデータをコピーおよび復元できます。
  3. 3番目のソリューションはより興味深いものです。


一部のお客様から、仮想マシン上のクラウドにハードウェア（OS）を複製するように依頼されました。 これはまさに、クライアントがバックアップからクラウドから何かを直接約10分で取得および起動できる「フォールアウト」オフィスのタスクです。ZertoおよびDouble-Takeソフトウェアを使用したレプリケーションにより、最小限の遅延でIPによってオペレーティングシステムとアプリケーションのデータをミラーリングできます。 技術面では、オペレーティングシステムまたはESXハイパーバイザー内で実行され、クラウドですべての変更を転送するサービスのように見えます。 実用的な面では、これはクライアントがアプリケーションを復元して可能な限り短い時間で動作する機会です。







このアプローチでは、技術的なソリューションに加えて、お客様および災害復旧のための災害復旧計画と協力しています。 必要に応じて、顧客自身がマシンの一部またはインフラストラクチャ全体の復元をテストまたは実行できるようにします。



練習する



Avamarベースのソリューションは5年前から使用されており、CrocBackAppにはいくつかの例があります。 どういうわけか、1つの小売チェーンがやって来ましたが、これにはインフラストラクチャ全体にわたる集中管理されたバックアップがなく、個々のオフィス用に設計された十分な「小規模」がありませんでした。 ハードウェアとソフトウェアはどこでも異なっていました、そして何かが落ちたとき、サーカスと動物園はすべてをロールバックする試みで始まりました。 いいえ、これは全体としては機能しましたが、個々のケースごとに、この環境での8年の経験があり、考古学層が何であるかを正確に知っているシステム管理者の能力が必要でした。 地域のこれらの悪夢を完全に取り除くために、私たちにCrocBackAppキューブを注文したのは彼でした。

それから政府機関が来ました。彼らのデータは最大ではありませんが重要です。 それらの「キューブ」は単にFCまたはイーサネットスイッチに固執するだけで、クライアントはすべてのローカルマシンにインストールされます。 次に、バックアップスケジュールとストレージ計画を構成するだけです。



その結果、非常にシンプルなスキームが形成されました。

  • 多数のオフィスを持つビジネス、または逆にSMBが、個別のハードウェアをローカルにインストールすることを望まない/できない場合、Avamarバックアップを実装します。 ストレージのコストはかなり低く、初期コストはゼロです。 つまり、OPEXは小さく、CAPEXはゼロです。
  • サイトに機器を配置する機会がある人のために、CrocBackAppを提供しています。 OPEXはAvamarよりも小さいですが、CAPEXがあります。
  • また、すでに独自のバックアップインフラストラクチャをお持ちの場合は、S3を使用してリモートストレージを整理することをお勧めします。
