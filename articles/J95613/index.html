nginxを使用したブロックキャッシュ

多くの開発者は、たとえば1つの小さなブロックのために5〜10分間、サイトのページをキャッシュできない状況に精通しています。リアルタイムでなくても、その緊急性を維持する必要があります。 同時に、サイトのトラフィックは増加し続け、ページの生成時間は増加しています。これを行うには何かが必要です...
  • オプション1： 過去6か月間に手が届かなかった部分を締めます。 誰もがあなたを理解し、用語を他のタスクに移動します。 サイトを法外な負荷から救うために「スーパーマン」だけの役割を果たし、「無料」で問題を解決します（機器への追加の注入なし）。 記事「Tuning nginx」が役立つかもしれません 。

  • 解決策2： 技術ベースを改善する（サーバーで頭脳を購入し、ディスクシステムを改善し、データベースの下に別のサーバーを配置する）。 原則として、問題は解決されておらず、むしろ延期されています。 これで、「掘り込み」、負荷の流入の第2波に備える時間ができました。負荷は大きくなり、より強力にカバーされます。

  • 解決策3： あなたのオプション。おそらくコメントから学びます。

Web開発で最も古いテクノロジーの1つに基づいた、実績のある比較的単純なソリューションを提供します。




仕組み



サイトは常に複数の独立したブロックに分割することができ、その生成は（必要に応じて）異なるサーバーで実行できます。



同時に、特定の「コレクター」がブロックを単一の全体にアセンブルします。割り当てられた時間内に何らかの理由でブロックが作成されなかった場合、これはクライアントに「ゲートウェイタイムアウト」または「内部サーバーエラー」を与える理由にはなりません。 正常に作成されたブロックを収集し、「失敗」のサイトでキャッシュから古いコンテンツを表示できます。



このようなモデルを実装するには、Web開発テクノロジーのベテランであるssiが必要です。 「コレクター」として、記事のタイトルから明らかなように、 nginxが機能します。 fastcgi_cacheモジュールのおかげで「奇跡」が可能になります。



それでは、行きましょう：





余分なリンクを除外します



Apacheは必要ありません。Apacheの存在は、原則としてRewriteRulesの使用によって説明されます。 nginxには、mod_rewriteの類似物、または正規表現を使用した場所/エイリアスの組み合わせがあり 、その機能により、ApacheのRewriteRuleに類似物を書き込むことができます。 さらに、最新のフレームワークでは、エンジン自体が入力URLを解析できます（たとえば、 Zend Frameworkの Zend_Controller_Router_Rewrite ）



fastcgiバックエンドとして、任意のプラットフォームを使用できます。 例はphpにありますが、これはPythonやperlで同様のコードを書くことができないという意味ではありません。



fastcgiモードでphpを実行します。

# /bin/su -m www_user -c "PHP_FCGI_CHILDREN=8 /usr/bin/php-cgi -q -b 127.0.0.1:7777 &"







php.ini（error_log = /var/log/fastcgi/fastcgi.log）でログファイルへのパスを指定することもできますが、php-cgiを再起動する必要があります。



私たちはやる：

# killall php-cgi





すべてを新しい方法で開始します



fastcgiを開始するより高度な方法は、 php- fpmをインストールすることです。





nginxをインストールする



リポジトリ/ポートから標準を設定できます...しかし、キャッシュ内のファイルを「クリーン」に動作させるには、コンパイルする必要があります。



モジュールが必要です： ngx_cache_purge



redhatのようなシステムでこれをどのように行うことができるかを詳細に説明し、類推によりシステム用にコンパイルします。 



 ＃cd〜/ rpmbuild / SRPMS
 ＃yumdownloader --source nginx
 ＃rpm -ivh nginx-0.7.65-1.fc12.src.rpm


./configureリストのどこかにあるnginx.specファイルを編集し、--add -module = / root / rpmbuild / BUILD / ngx_cache_purge-1.0 \という行を挿入します 。 不要なモジュールを含む行をすぐに削除できます（たとえば、-with-ipv6 \、-with-http_dav_module \、-with-mail \、-with-mail_ssl_module \ ...）



http://labs.frickle.com/files/ngx_cache_purge-1.0.tar.gzの内容をフォルダー/root/rpmbuild/BUILD/ngx_cache_purge-1.0に解凍します。

すべてをコンパイルできます： 



 ＃cd〜/ rpmbuild / SRPMS
 ＃rpmbuild -ba nginx.spec


これはきれいな方法ではありません、なぜなら 結果の.src.rpmには、ngx_cache_purgeモジュールのファイルは含まれません。 ただし、これが重要な場合は、ここでブランチ8.xx用の「正しい」バージョンのnginx .src.rpmをダウンロードできます。 確かに、必要のないモジュールのいくつかについてコメントしました。



再構築されたnginxをサーバーにインストールします。



＃rpm -ivh nginx-0.7.65-1.fc12.x86_64.rpm

キャッシュ時間を管理するためのバックエンドを学びます。



実際、nginxでは、キャッシュ時間はfastcgi_cache_valid 200 0mパラメーターで指定されています。 タイトルが再定義されていないすべてのページに適用されます。



デフォルトの構成では、キャッシュ時間を0に設定しています。 キャッシュは無効です。 ただし、バックエンドが次のようなヘッダーを生成する場合： 



キャッシュ制御：public、max-age = 20
どちらか 
有効期限：木、2010年3月18日20:57:07 GMT


そのページnginxは20秒間キャッシュされます。 phpでは、header（）関数を使用してヘッダーを変更できます（ nginxの作成者によれば、「X-Accel-Cache-Control」、「Cache-Control」、「Expires」が最優先です） 。



小さな関数を書きましょう。 キャッシュ時間を制御します： 

 function cacheHeaders（$ lifetime = 0）{
 ＃$ date = gmdate（ "D、d MYH：i：s"、time（）+ $ lifetime）;
 ＃header（ 'Expires：'。$ date。 'GMT'）;
         header（ 'Cache-Control：public、max-age ='。$ lifetime）;
 }

キャッシュからページを削除する



残念ながら、nginxにはキャッシュからページを削除するネイティブ（通常の）方法がまだありません。 これは不便な場合があります。



コンパイル中にngx_cache_purgeモジュールを追加した場合、config（/etc/nginx/conf.d/myproject.conf）で、「location / {...」セクションの前に次のセクションを追加します。 



場所〜^ /パージ（/.*）{
         #allow 127.0.0.1;
         #allow 10.1.1.0/24;
         #deny all;
         fastcgi_cache_purge mycache $ 1 $ is_args $ args;
 }


キャッシュされたページを削除するには、http：//myproject/mypage.php？Lang = ru、http：//myproject/purge/mypage.php？Lang = ruのページをダウンロードするだけです。



phpでは、これはfile_get_contentsコマンド（ "http：//myproject/purge/mypage.php？Lang = ru"）で実行できます。



allowおよびdenyディレクティブを使用して、キャッシュを「クリア」できるホストの範囲を制限できます。
