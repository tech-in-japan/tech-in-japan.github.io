PHPを真剣に考える



ソユーズロケットが列車で発射台に届けられました。 NASAのパブリックドメインからの写真。



これは有名なSlackアプリケーションのエンジニアの1人であるPHP Seriouslyの翻訳です。 彼は、Slackが移行をほぼ完了したHack言語とHHVM仮想マシンだけでなく、PHPの欠点と利点についても語っています。



SlackはほとんどのサーバーロジックにPHPを使用していますが、これは最近では最も一般的な選択肢ではありません。 この言語で新しいプロジェクトを書くことにしたのはなぜですか？ あなたも同じことをすべきですか？



PHPで少し遊んだほとんどのプログラマーは、PHPについて2つのことを知っています。それは、選択の余地があるときには決して使用しない悪い言語であり、世界の歴史で非常に成功したプロジェクトのいくつかはそれを使用していることです。 これは矛盾ではありませんが、この事実はあなたに考えさせるべきです。 つまり、Facebook、Wikipedia、Wordpress、Etsy、Baidu、Box、そして最近Slackです。これらはすべて、PHPを使用しているにもかかわらず、問題を解決できますか？ Rubyを使用した場合、より成功するでしょうか？ アーラン？ ハスケル？



そうではない可能性があります。 PHP言語には、間違いなく開発を遅らせる多くの欠点がありますが、PHP環境には、これらの欠点を補う以上の利点があります。 また、彼は自分の言語の問題を解決する方法も持っています 。これは非常に印象的です。 収益に基づいて、PHPは、競合する環境と比較して、成功するWebプロジェクトを作成、変更、および運用するための最適な基盤を提供します。 今日、私はPHPで新しいプロジェクトを始めたいと思いますが、いくつかの予約がありますが、これについて謝罪する理由はありません。



歴史的遠足



PHPは、現代言語向けの独自のWebサーバー環境で生まれました。 その強みは、サーバー上の要求コンテキストに関連しています。



PHPは元々「 パーソナルホームページ 」と呼ばれていました。 1995年に公開されました。 Rasmus Lerdorfは、インターネットのd明期に人気のあるゲストブックや訪問者カウンターなど、小さくシンプルな動的Webアプリケーションのサポートに重点を置いています。



PHPのリリース以来、PHPは作成者が当初予想していたよりもはるかに複雑なプロジェクトで使用されてきました。 いくつかの大きな変更が行われ、それぞれがこれらの複雑なアプリケーションを抑制する新しいメカニズムをもたらしました。 現在、2016年、彼はJavaScript、Python、Ruby、およびLuaを含むMixed Productive Language Paradigm（MPDPL）ファミリー[1]の機能豊富なメンバーです。 2000年代前半にPHPを試した場合、最新のPHPコードベースは、 特性 、 クロージャー、 ジェネレーターで驚くかもしれません。



PHPの美徳



PHPには、非常に深い、そして間違いなく真の機能があります。



まず、 条件 。 各Web要求は、完全にクリーンな状態で始まります。 その名前空間とグローバル変数は初期化されていませんが、いくつかの標準的なグローバル変数、関数、およびプリミティブ機能とライフサポートを提供するクラスは例外です。 既知の状態で各リクエストを開始すると、起こりうるエラーから一種の分離が得られます。 要求tがソフトウェアの問題に遭遇して失敗した場合、このバグは後続の要求t + 1の実行に影響を与えません。 実際、もちろん、ソフトウェアヒープに加えて、アプリケーションの状態は他の場所にあり、データベース、memcache、またはファイルシステムを完全に破壊する可能性が十分にあります。 しかし、PHPは、この弱点を、状態を維持できるすべての考えられる環境と共有しています。 同時に、要求間でソフトウェアヒープを分割すると、ほとんどのソフトウェアエラーの価格が下がります。



2つ目は並行性です。 個々のリクエストは1つのPHPスレッドで機能します。 一見、これはばかげた制限のように見えます。 しかし、アプリケーションはWebサーバーのコンテキストで実行されるため、自然な並行性のソースはWebリクエストです。 非同期の要求を自分自身に送信するだけで、分離された状態とバックアップと復元の呼び出しで並行性を簡単に取得できます。 実際には、これは、他の汎用言語で使用されるロックアンドシェアアプローチよりも安全でエラー耐性があります。



結論として、PHPプログラムがクエリレベルで動作するという事実は、プログラマのワークフローが高速で効率的であり、アプリケーションを変更した後も高速であることを意味します。 多くの生産的な開発言語がこれを主張していますが、各リクエストで状態をクリアせず、イベントのメインストリームがプログラム状態レベルをリクエスト間で分割する場合、ほとんどの場合、開始に時間がかかります。 典型的なPythonアプリケーションサーバーの場合、典型的なデバッグサイクルは「サーバーを考え、編集し、再起動し、テストリクエストを送信する」ようなものになります。 「サーバーの再起動」が合計時間のうち数秒しかかからない場合でも、現在の状態の最も不必要な部分を念頭に置くには 、人間の脳の15〜30秒の大きなスライスが必要です。



「ページを考え、編集し、再読み込みする」スタイルのPHP開発は、開発者の生産性を高めると主張します。 長く複雑なプロジェクト開発サイクルでは、これによりさらに大きく増加します。



長所とPHP



これがすべて真実なら、なぜ彼はそんなに嫌われているのでしょうか？ すべてのカラフルな双曲線を削除すると、PHPクラスターに関する主な不満は次の問題に帰着します。



  1. 変容の驚き。 最近のほとんどすべての言語では、たとえば、整数と浮動小数点を> =演算子で比較できます。 地獄、Cでも可能です。 ここで何を意味するかは明らかです。 文字列と数値を==と比較することはそれほど明白ではなく、言語によって選択が異なります。 この状況でのPHPの選択は最も悪質であり、驚きと不快なエラーにつながります。 たとえば、123 == '123foo'はtrueと評価されます（そこで何をしているのでしょうか？）、しかし0123 == '0123foo'はうそ（うーん）です。



  2. リンクとセマンティクスの意味に関する矛盾。 PHP 3には、引数を渡し、すべてを値で返し、リクエスト内のデータの論理コピーを作成するという明確なセマンティクスがありました。 プログラマは、＆記号[2]とともに参照セマンティクスを選択できます。 これは、PHP 4および5でのオブジェクト指向プログラミングツールの導入とともに発生しました。ほとんどのPHP OOアノテーションはJavaから借用されており、Javaにはオブジェクトが参照で渡されるセマンティクスがあり、プリミティブ型は値で渡されます。 その結果、PHPセマンティクスの現在の状態では、オブジェクトは参照によって渡され（C ++などの代わりにJavaを選択）、プリミティブ型は値によって渡されます（ここではJava、C ++、およびPHPが同意します）が、古い参照セマンティクスと＆記号はあいまいな方法で新しい世界とやり取りすることが時々ありました。



  3. 失敗を無視するコンピューティングの哲学 。 PHPは、完全に奇妙なことを既に実行していても、クエリを実行し続けるように非常に懸命に努力しています。 そのため、たとえば、ゼロで除算しても例外はスローされず、INFは返されず、致命的な要求は終了しません。 デフォルトでは、警告を発し、値をfalseに設定するだけです。 falseは数値コンテキストでは暗黙のうちに0と見なされるため、多くのアプリケーションがデプロイされ、ゼロによる未診断の除算で実行されます。 具体的には、この問題はPHP 7で解決されましたが、あいまいさを処理するための設計の推進力は、意味がある場合でも、ライブラリにも浸透しています。



  4. 標準ライブラリの矛盾。 PHPが若かったとき、その聴衆はCに最も精通しており、多くのAPIは標準C言語ライブラリの設計を使用していました：6文字の小文字の名前、成功/失敗の応答、呼び出されたパラメーターに実際の値を返すさらに。 PHPが進化するにつれて、接頭辞_を使用して名前空間に分割するCスタイルのスタイルがより一般的になりました：mysql _...、json _...など。 最近では、CamelCaseクラスのcamelCase Javaスタイルの命名方法が、新しい機能を導入する最も一般的な方法になりました。 そのため、最終的に、DirectoryIterator（$パス）のような混合式とif（！（$ F = fopen（$ p、 'w +'））...のような複雑なロジックのコード例を見ることがあります。


PHPの無反射的な謝罪者のように見えないようにするために、これらはすべて重大な問題であり、欠陥を引き起こす可能性が高くなります。 それらは強制されていないエラーです。 PHPの優れた部分とこれらの問題の間には、本質的な妥協はありません。 すべての良い面を維持しながら、これらの欠点を解決するPHPを作成する機会を実現する必要があります。



HHVMとハック



このPHPシステムの後継はHackと呼ばれます[3]



ハックは、人々がPHPの「漸進型システム」と呼ぶプログラミング言語です。 「タイプシステム」とは、プログラマーがコードを流れるデータについて自動的に検証された不変式を作成できることを意味します。この関数は、JavaまたはC ++またはHaskellなどの静的に型付けされた言語で文字列または数値を受け取り、Fribblesシートを返します。どちらを選ぶか。 「漸進的」とは、コードベースの一部を静的に型付けできる一方で、それ以外の部分は依然として乱雑で動的なPHPにあることを意味します。 これらのアプローチを組み合わせることにより、大規模なコードベースを徐々に移行することができます。



ここでは、Hackタイプシステムとその仕組みの説明で大量のインクを注ぐのではなく、 単に試してみてください 。 あなたが戻ったとき、私はここにいます。



これはきちんとしたシステムであり、表現できるという点で非常に野心的です。 また、プロジェクトが当初の予想よりも強くなった場合に、プロジェクトを段階的にハックに移行できる可能性は、PHPエコシステムのユニークな利点です。 ハックタイプチェックは、バックグラウンドで実行されるため、「思考、編集、リロードページ」のスタイルでワークフローを保存します。ファイルシステムに変更があると、コードベースモデルが徐々に更新されます。 Hackプロジェクトは、すべての一般的なエディターおよびIDEとの統合を提供するため、Webデモの場合と同様に、コードの入力を終了すると、既にタイプエラーに関するフィードバックを確認できます。



ハックに照らしてPHPがもたらす実際のリスクの全体を見てみましょう。



  1. 変換の驚きは、Hackファイルのエラーになります。 データの問題のクラス全体がなくなります。
  2. Hack のリンクと値のセマンティクスは 、古いスタイルのリンクの使用を単に禁止することでクリアされるため、新しいコードベースでは不要になります。 これにより、JavaまたはCの場合と同様に、セマンティクスの動作が、参照によるオブジェクトおよびその他の値によるオブジェクトのスタイルに似たものになります。
  3. 失敗を無視する PHPベースの計算は、ランタイム環境のプロパティであり、これらのシステムに直接注入するために、Hackなどのセマンティクスアナライザーで処理するのがより困難です。 ただし、実際には、障害を無視するほとんどの形式のコンピューティングでは、非常に驚​​くべき変換が必要です。 たとえば、ゼロで除算した後にfalseになったために発生する問題は、タイプ[4]のチェック境界の交点では最終的に発生せず、ブール値を数値に変換しようとして失敗します。 これらの境界は、Hackコードベースではより一般的です。 これらのタイプを説明する簡単な機能と合わせて、ハックは実際には多くの不正確なスタートの「ブレーキング距離」を短縮します。
  4. 結論として、 論争は標準ライブラリに残っています 。 ハックのほとんどは、この問題をより安全な抽象化でラップすることで、この問題を軽減できることを望んでいます。


ハックは、他の一般的なMPDPLファミリメンバーにはない機会を提供します。メイン開発の後に型システムを導入する機能で、価格が価格を上回るシステムの部分に部分的にのみ導入できます。



Hhvm



Hackはもともと、 HipHop仮想マシン 、またはPHP用のオープンソース仮想環境であるHHVMの一部として開発されました。 HHVMは、プロジェクトを成功させるためのもう1つの重要なオプションを提供します。それは、サイトをより速く、より経済的に実行する能力です。 Facebook は 、PHPインタープリターよりもプロセッサー効率が11.6倍向上したと報告し、 Wikipedia は 6倍の加速を報告しています。



Slackは最近、Web環境をHHVMに移行し、すべての出口点で大幅な遅延の削減を受け取りましたが、このテキストを書いている時点では、プロセッサの負荷に関するリンゴごとのスタイルの測定値がありません。 また、コードベースをHackに移行する過程にあり、ここで経験を報告します。



楽しみにして



PHPは多くの成功したプロジェクトで使用されている非常に貧弱な言語であるという明白なパラドックスから始めました。 私たちは、貧しい言語としての彼の評判は、単独で、十分に価値があると信じています。 これを使用するプロジェクトの成功は、PHP環境の基本的な特性、およびPHPが提供する加速化された開発機能と多くの共通点を持っています。 そして、この環境の利点（エラー分離によるバグの減少、安全な並列処理、プログラマーの高スループット）は、言語の欠陥のために生じる問題よりも価値があります。



さらに、MPDPLファミリーの他のメンバーとは異なり、HackおよびHHVMの形で、より生産的で安全で維持された環境に移行するための明確な道筋を提供します。 SlackはHHVMへの移行の最終段階にあり、Hackへの移行の初期段階にあります。より速い時間枠でより良いソフトウェアを作成できるので楽観的です。
