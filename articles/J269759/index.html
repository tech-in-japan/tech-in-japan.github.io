PyTestの長所と短所



1）APIからの独立（定型なし）。 同じunittestでコードがどのように見えるか：



コード 
import unittest class TestUtilDate(unittest.TestCase): def setUp(self): #init_something() pass def tearDown(self): #teardown_something() pass def test_upper(self): self.assertEqual('foo'.upper(), 'FOO') def test_isupper(self): self.assertTrue('FOO'.isupper()) def test_failed_upper(self): self.assertEqual('foo'.upper(), 'FOo') if __name__ == '__main__': suite = unittest.TestLoader().loadTestsFromTestCase(TestUtilDate) unittest.TextTestRunner(verbosity=2).run(suite)




PyTestでも同じこと：



コード 
 import pytest def setup_module(module): #init_something() pass def teardown_module(module): #teardown_something() pass def test_upper(): assert 'foo'.upper() == 'FOO' def test_isupper(): assert 'FOO'.isupper() def test_failed_upper(): assert 'foo'.upper() == 'FOo'




2）詳細レポート。 JUnitXMLへのアップロードを含む（Jenkinsとの統合用）。 レポート自体のタイプは、追加のモジュールによって変更できます（色を含む）（それらについては後で別途説明します）。 まあ、一般的には、コンソールのカラーレポートの方が便利です。赤のFAILEDはすぐに表示されます。







3）便利なアサート（Pythonの標準）。 異なるassert'ovの束全体を覚えておく必要はありません。



4）すべてのレベルの動的なフィクスチャ。これは、自動的に特定のテストのために呼び出すことができます。



5）フィクスチャの追加機能（戻り値、ファイナライザ、スコープ、リクエストオブジェクト、自動使用、ネストされたフィクスチャ）



6）テストのパラメーター化、つまり、異なるパラメーターセットを使用した同じテストの起動。 一般的に、これはパラグラフ5「追加機能フィクスチャ」を指しますが、別のパラグラフに値するほど機会が多いのです。



7）マーク。テストをスキップしたり、テストを失敗としてマークしたり（これは開発中に便利な予想される動作です）、テストスイートに名前を付けて名前でのみ実行できるようにします。



8）プラグイン。 このモジュールには、個別にインストールできる追加モジュールのかなり大きなリストがあります。



9）unittestおよびnoseで記述されたテストを実行する機能、つまり、それらとの完全な下位互換性。



欠点については、多くはありませんが、次のように言えます。



1）追加レベルのネストの欠如：テストのモジュール、クラス、メソッド、関数には、適切なレベルがあります。 ただし、同じ関数が複数のテストケースを持つことができる場合（たとえば、戻り値とエラーのチェック）、ロジックには追加レベルのテストケースが必要です。 これは、追加のモジュール（プラグイン）pytest-describeによって部分的に補われますが、適切なレベルのフィクスチャーがないという問題が発生します（scope =“ describe”）。 もちろん、これで生きることはできますが、状況によっては、PyTestの主な原則「単純さと便利さのためのすべて」に違反する可能性があります。



2）生産を含む、モジュールの個別のインストールの必要性。 それでも、unittestとdoctestは基本的なPythonツールキットの一部であり、追加のジェスチャーは必要ありません。



3）PyTestを使用するには、同じユニットテストよりもPythonの知識が少し必要です（「必要な知識の概要」を参照）。

テストを実行する



unittestの場合、メイン関数呼び出しが使用されます。 したがって、起動は「python unittest_example.py」という形式になります。 同時に、テストスイートを実行するには、TestSuitでそれらを個別に組み合わせて実行する必要があります。 PyTestは、フォルダ内のすべてのファイル（再帰的にサブフォルダをトラバースする）または指定されたファイルについて、独自の名前test_ *（クラス名のTest_ *）ですべてのテストを収集します。 つまり、呼び出し例は「py.test -v pytest_example.py」のようになります



基本的な備品



この場合、フィクスチャは、テストに適切な環境を作成するために起動される関数とメソッドを呼び出します。 unittestと同様に、PyTestにはすべてのレベルのフィクスチャの名前があります。

拡張フィクスチャ



問題は、一部のテストで特定の環境が必要であるが、他のテストでは必要ない場合はどうすればよいかということです。 異なるモジュールまたはクラスに広がりますか？ それは非常に快適で美しく見えません。 拡張PyTestフィクスチャが役立ちます。



したがって、PyTestで拡張フィクスチャを作成するには、次のものが必要です。



1）pytestモジュールをインポートする

2）@ pytest.fixture（）デコレータを使用して、この関数がフィクスチャであることを示します

3）フィクスチャ（スコープ）のレベルを設定します。 可能な値は、「関数」、「cls」、「モジュール」、「セッション」です。 デフォルト値=「関数」。

4）このフィクスチャのティアダウンを呼び出す必要がある場合、ファイナライザを追加する必要があります（フィクスチャに渡されたリクエストオブジェクトのaddfinalizerメソッドまたはyield構成体を使用して）

5）このフィクスチャの名前を関数パラメーターのリストに追加します 



 import pytest @pytest.fixture() def resource_setup(request): print("resource_setup") def resource_teardown(): print("resource_teardown") request.addfinalizer(resource_teardown) def test_1_that_needs_resource(resource_setup): print("test_1_that_needs_resource") def test_2_that_does_not(): print("test_2_that_does_not") def test_3_that_does_again(resource_setup): print("test_3_that_does_again")

2）フィクスチャに自動使用フラグを使用します。 ただし、予期しないテスト動作が発生する可能性があるため、この機能は注意して使用する必要があります。

3）テストパラメータを介した上記の方法

分解拡張器具



上記のように、特定の拡張フィクスチャに対してティアダウンを呼び出す必要がある場合、2つの方法で実装できます。



1）ファイナライザをフィクスチャに追加します（フィクスチャに渡されたリクエストオブジェクトのaddfinalizerメソッドを介して）

2）yieldコンストラクトを使用して（PyTestバージョン2.4以降）



拡張フィクスチャを作成する例で検討した最初の方法。 次に、yieldを使用して同じ機能を実証します。 関数をフィクスチャとして装飾するときにyieldを使用するには、@ pytest.fixture（）ではなく@ pytest.yield_fixture（）デコレータを使用する必要があることに注意してください。 



 import pytest @pytest.yield_fixture() def resource_setup(): print("resource_setup") yield print("resource_teardown") def test_1_that_needs_resource(resource_setup): print("test_1_that_needs_resource") def test_2_that_does_not(): print("test_2_that_does_not") def test_3_that_does_again(resource_setup): print("test_3_that_does_again")


ファイナライザオプションと一致するため、テキストに結論を追加しません。

パラメータ化



パラメータ化は、異なる一連の入力パラメータを使用して同じテストを実行する方法です。 たとえば、文字列が5文字より長い場合は疑問符を、5文字より短い場合は感嘆符を、文字列に正確に5文字がある場合はドットを追加する関数があります。 したがって、3つのテストを作成する代わりに、1つのテストを作成できますが、異なるパラメーターで呼び出します。



テストのパラメーターを設定するには、2つの方法があります。



1）値の配列を渡すフィクスチャのparamsパラメーターの値を介して。

つまり、この場合のフィクスチャは、パラメーターを渡すラッパーです。 そして、それらは上記のリクエストオブジェクトのparam属性を介してテスト自体に渡されます。

2）デコレータ（ラベル）@ pytest.mark.parametrizeを介して、変数名のリストとその値の配列が渡されます。



だから最初の方法。 



 import pytest def strange_string_func(str): if len(str) > 5: return str + "?" elif len(str) < 5: return str + "!" else: return str + "." @pytest.fixture(scope="function", params=[ ("abcdefg", "abcdefg?"), ("abc", "abc!"), ("abcde", "abcde.") ]) def param_test(request): return request.param def test_strange_string_func(param_test): (input, expected_output) = param_test result = strange_string_func(input) print "input: {0}, output: {1}, expected: {2}".format(input, result, expected_output) assert result == expected_output

追加モジュール



PyTestには多数のアドオンモジュールがあります。

興味のあるモジュールとその理由のみに言及することができます（モジュールの詳細については、上記のリンクを参照してください）：



pytest-describe-別のレベルの抽象化を追加します（同等のmodule-function-testcaseとしてmodule-description-function）。



pytest-instafail-モジュールの基本的な動作を変更して、セッション全体の終わりではなく、実行中にすべてのエラーとクラッシュが表示されるようにします。



pytest-marks-テストに複数のラベルを同時に設定できます： 



 @pytest.mark.red @pytest.mark.green @pytest.mark.blue def some_test_method(self): .....


pytest-ordering -Runラベルでテストを実行する順序を手動で設定できます。 



 import pytest @pytest.mark.run(order=2) def test_foo(): assert True @pytest.mark.run(order=1) def test_bar(): assert True


pytest-pep8 -pep-8規約に準拠しているかどうかテストコードを確認できます。



pytest-smartcov-完全または部分的なテストのコードのカバレッジを確認できます。



pytest-timeout-コマンドラインパラメーターまたは特別なラベルを使用して、タイムアウトでテストを完了できます。



pytest-sugar -PyTestの出力の外観を変更し、進行状況バーと完了率を追加できます。それは場所にかかわらず、美しく見え、あまり有益ではありません。
