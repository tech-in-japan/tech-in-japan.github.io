クライアントでのクラスタリング、または地図上に10,000ポイントを表示する方法

プロジェクトにはいくつかの種類があります

地理情報の出力は

必要：不動産サイト、会社名簿、

アトラクション、掲示板などのカタログ。

これらのサイトでは、goolge API 3およびYandex API 2のプラグインを開発することにしました。



基本的な要件：

  1. サーバーでの深いデータ準備なしで接続する機能。つまり、オブジェクトの既存の地理座標のみを使用します。
  2. データを準備および送信するためのシンプルなプロトコル。
  3. 2つのタイプのラベル（クラスターとグループ）を使用した高速クライアントクラスタリング。 クラスター-近くにあるいくつかのオブジェクト。 クリックすると、クラスターが「開き」ます。つまり、オブジェクトがクラスタリング距離よりも遠いときにズームが増加します。 グループ-1つのポイント（最小距離）にある複数のオブジェクト。 ズームでグループをクリックすると、グループ内のオブジェクトのリストが表示されます。
  4. クラスタリングを使用して最大10,000ポイントを表示します。
  5. IE7のモバイルデバイス（第1世代のiPad）で表示します。
  6. グループクラスターとオブジェクト自体の2つの情報ウィンドウのクライアントでのテンプレート化。
  7. ラベルにスプライトを使用します。
  8. さまざまな種類のオブジェクトに対して無制限の数の種類のラベル（アイコン）を使用する機能。
  9. さまざまなズーム範囲に対して複数のアイコンサイズを使用する機能。


この記事では、このプラグインの実装に使用されたいくつかのアプローチについて説明します。



機能の動作時間の測定は、いくつかのデバイスで実行されました。 便宜上、それらの一部2で書き込みます-Google Chrome 19.0 @ AMD Athlon 64 X2 Dual Core 5600+ 2.8 GHz RAM 2Gb Windows-7 32bitが平均で、Safari @ Apple iPad 16GB 1.0 GHz iOS 5.1.1が最も遅く不機嫌です。



サーバー側



サーバー部分は次の2つの要求で構成されます。



1.カードデータの要求。


ラベルを選択するためのパラメーターを含むリンクがこのリクエストに送信されます。 たとえば、都市= 1および部屋= 1,2。 つまり、1ベッドルームと2ベッドルームのアパートメントで都市ID = 1のみを選択します（不動産サイトの例）。

応答として、マップのjsonパラメーター（マップ、マップの中心とズーム、アイコンパラメーター、クラスタリング距離、スプライトなど）とすべてのポイントに関するデータを含むjsonを含むスクリプトが返されます。 これらのポイントには、オブジェクトID、地理座標、アイコンタイプ、ツールチップラインが含まれます。 非圧縮状態の10,000ポイントの場合、応答サイズは1.2 MBですが、gzip圧縮では350 KBのみです。

頻繁に繰り返されるリクエストは、nginxによって適切にキャッシュされます。



2.オブジェクトの要求。


1つのマーカーをクリックすると、情報ウィンドウにオブジェクトに関するデータがポップアップ表示されます。 AjaxリクエストにはオブジェクトのIDが含まれます。 応答はjsonです。 jsonフィールドの名前は情報ウィンドウテンプレート内のタグと一致する必要があり、後者は受信したコンテンツに置き換えられます。



クライアント部



ポイントデータの解析


受信したデータは次のように処理されます。

1.ゼロズームでのグローバルピクセル座標は、各ポイントに対して計算されます。

2.グローバル座標に基づいて、14番目のズームのタイルIDが計算され、対応するタイルIDの配列にポイントが配置されます。 将来、どのズームでも、必要なタイルのIDを簡単に計算して、それらのすべてのポイントを選択できます。 ポイントサンプリング速度-10,000ポイントの場合、100/350ミリ秒以下。

クラスタリング


Yandex API 2のネイティブClustererなど、いくつかのクラスターをテストしました。それらのいずれも、いくつかの理由で私に適していませんでした。 まず、クラスターのみが存在し、グループは存在しません。 クラスターをクリックすると、ズームは最大まで増加しますが、リストは表示されません。 リストを取得するには、もう一度クリックする必要があります。 これが、YandexのClustererの配置方法です。 Google Maps v3用のMarkerClustererにもこのタイプのマーカーはありません。

第二に、クラスターはグリッドによって計算されます。グリッドは高速ですが、多数のラベルを持つクラスターが「グリッド上」に配置され、見苦しくなります。

その結果、独自のClusterizeクラスタリングが作成されました。 その中で、クラスタリング方法は非同期ハイブリッドです。タイルのデータを要求すると、次のことが起こります。

-タイルは4つのサブタイル（グリッド）に分割され、各サブタイルは距離によってクラスター化されますが、クラスター自体はセルの中央ではなく、「折りたたまれた」ポイントの領域の中央に配置されます。

-受信したポイントとクラスターはタイルに収集され、再びクラスター化されます。 この時点で、すでにいくつかのポイントがあります。

-受け取ったポイントは、隣接するタイルを考慮して「間引かれ」ます（既にクラスター化されている場合）。 つまり、タイトルのマーカーと重なる境界線上の境界線タイルにクラスターまたはマーカーがある場合、これらのマーカーはクラスターに結合されます。 これにより、タイルの境界にマーカーが「付着」しないようにします。

クラスタリング速度は非常に高いことが判明しました。10,000のうち、60/500ミリ秒で580ポイントを取得します。 まったく問題ありません。



マーカーアイコン


スプライトを使用してアイコンを表示します。 アイコンの種類の数に制限はありません。 例では、単一マーカー+クラスターおよびグループアイコンに7種類のアイコンを使用しています。 2つのバージョンの例2の各タイプのアイコンは、14を超えるズームでは大きく、14未満では小さいです。クラスタリング距離は17ピクセルであるため、小さいアイコンは小さいスケールできれいに見えます。
