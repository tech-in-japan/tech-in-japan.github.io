しかし、それが置かれているページに応じて特定のグループにサブスクライブする動的なフォームが必要でした。



APIを使用します。

ユーザーのサブスクリプションは、2つのメソッドで表されます。listSubscribe-新しいユーザーのサブスクリプションを作成し、listUpdateMember-ユーザーのサブスクリプションを更新します。

apidocs.mailchimp.com/api/1.3/listsubscribe.func.php

apidocs.mailchimp.com/api/1.3/listupdatemember.func.php



メソッドはほとんど同じです。たとえば、キーupdate_existing = trueを指定すると、listSubscribe（）がlistUpdateMember（）に変わります。



より完全なlistSubscribeメソッドのみを考慮します。 

 $mailchimpAPI->listSubscribe($listId, $email, $mergeVars, $emailType, $doubleOptin, $updateExisting, $replaceGroups, $sendWelcome);


重要なパラメーター：

•$ listId-ユーザーリストID

•$ email-サブスクリプションを作成または更新する必要があるユーザーの郵送先住所

•$ mergeVars-グループなどの可変データを含む配列

•$ updateExisting-サブスクリプションが既に存在する場合は更新します

•$ replaceGroups-$ mergeVarsにあるグループを置き換えるか、既存のグループとマージします



ユーザーグループを更新するには、$ mergeVarsパラメーターが必要です。idキーはルートグループ（ベンダーとカテゴリ）のIDで、groupsキーはコンマで区切られたグループ名です： 



 $mergeVars = array('GROUPINGS' => array( array( 'id' => '4444', 'groups' => '99# , 77# ' ), ));


$ replaceGroups = falseを設定して、既存のグループを$ mergeVarsの新しいグループと結合します。



グループを完全に新しいグループに置き換えるには、最初に$ replaceGroups = trueを設定し、次に$ mergeVarsで、ルートグループの配列全体をグループ自体の空の値で転送する必要があります。 

 $mergeVars = array('GROUPINGS' => array( //    array( 'id' => '4444', 'groups' => '99# , 77# ' ), //  array( 'id' => '5555', 'groups' => '' ), //  array( 'id' => '7777', 'groups' => '' ), ));


つまり、$ replaceGroups = trueパラメーターは、すべてではなく、同じルートグループ内で機能します。



この方法を使用して、2つのフォームオプションを作成しました。

1.すべてのユーザー-現在のページに応じて、電子メールフィールドと配布グループの非表示フィールドを含む通常のフォーム。

2.購読/購読解除できるグループのチェックボックスのリストが表示されるアカウントのフォーム。



listSubscribeメソッドの機能：

1.サブスクリプションが新規の場合、その処理はすべてmailchimpサービスに送られ、確認メールが送信されます。確認後、サブスクライバーのプロファイルへのリンクが提供され、データを編集できます。プロファイルは再びmailchimp Webサイトにあります。

2.ユーザーの電子メールが既にメーリングリストにある場合、メソッドはエラーを生成し、個人データとサブスクライバーグループを手動で編集することを提案します。プロファイルへのリンクは、エラー$ mailchimpAPI-> errorMessageのテキストに含まれます。 サブスクリプションを更新するには、$ updateExisting = trueを指定する必要がありますが、このキーは他の誰かのサブスクリプションを上書きするためにも使用できるため、権限のないユーザーには適していません。



その結果、2つの問題があります。

1.新しいサブスクライバーのすべての処理は私たちとは行われません。たとえば、配布グループを公開するなど、それに影響を与えることはできません。

2.許可されていないユーザーへのサブスクリプションを更新する方法は明確ではありません。指定された電子メールがそのユーザーのものであることを確認できないためです。



そのため、listSubscribeによるサブスクリプション確認レターの送信を無効にしました。パラメーターは$ doubleOptin = false、$ sendWelcome = falseで、次のメカニズムを記述しました。



1-当社のウェブサイトでサブスクリプションフォームを送信する場合、一意の確認リンクを記載したメールを送信します。

2-$ doubleOptin = false、$ sendWelcome = false、$ updateExisting = trueを使用してサブスクリプションを厳密に作成または更新するリンクハンドラー。 これで、指定された電子メールは、許可されていないユーザーのものであることが確実になりました。なぜなら、彼は手紙のリンクをクリックしたからです。



パラメーター$ doubleOptin = falseを使用すると、確認せずに任意のユーザーのサブスクリプションに署名/更新するため、この方法は、このサブスクリプションが現在のユーザーに属していることが確実な場合にのみ使用する必要があります。たとえば、個人アカウントまたは一意のリンクをクリックする。
