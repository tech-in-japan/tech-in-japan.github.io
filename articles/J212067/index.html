ニューススタンドアプリ。 iOSマガジンを作成

最近、私は1つの紙の雑誌のiOSバージョンを開発していました。 実際、これはまさにこのトピックを明らかにする試みです。



はじめに紹介します。 ニューススタンドとは何ですか？ そのようなエンティティはどこから来て、何に変わりましたか？ 考えてみると、私は次のことに気付きました。iOSプログラムに包まれたこのバージョンの雑誌は、単に無責任なサイズという点でpdfとは異なります。 その理由の1つは、膨大な写真の山です。 ただし、このヒープはiOSマガジンの光沢を生み出します。 ニューススタンドのアイデアが生まれた理由は、私が理解しているように、特定のコンテンツの所有権に関するAppleの立場でした。 つまり タスクは定期的に配布するために正直に（多くの場合、支払われます）、コピーペーストが困難でした。 これらの人たちはこの仕事に対処しました-ニューススタンドよりも高い現金回転率を持つ電子出版社は存在しないと思います（そう言えば）。



ユーザーレベルでいくつかの雑誌を確認したら、技術的な解決策を探します。 まず、ニューススタンドキットが登場しました。



ニューススタンドキットは、次のギズモで構成されています。雑誌の個々の号の特別なライブラリ管理メカニズム（NKLibrary）（NKIssue）とそのバックグラウンドダウンロードを含むダウンロード（NKAssetDownload）。 かっこいい！ しかし、その当時、これらのギズモを雑誌に変える方法は私には完全に不明確でした。



すべて同じです。 明らかに、プログラム雑誌は2つの部分で構成されています：雑誌の問題の管理と別の雑誌の表示。 つまり ニューススタンドキットは、最初のコンポーネントである管理に焦点を当てています。 2番目のコンポーネント-雑誌の1つの号の表示-Appleはテンプレートに負担をかけませんでした。 2番目に彼女から始めます。



理論的には、サーバーから必要な情報をすべてダウンロードして表示する必要があります。 つまり それはパンフレットの補足についてでした。 ダウンロードされた情報は、条件付きで、原材料（画像、フィルム、テキスト、html-in、pdf）と設定に分けられ、それによってすべてをまとめることができます。 誰もがパンフレットについて読むことに興味があるとは思わないので、経営に関する部分にこれらをさらに送ります。



上で述べたように、Appleは特定の雑誌の表示に関して開発者を制限していません。 ただし、スワイプで雑誌をページングして、おそらく良い形で受け入れられます。 このために、UIScrollViewと無限スクロールのアイデアを使用しました（dev apple-> WWDC-> ScrollView Techniques-> Infinite scrolling）。 簡単に言うと、メインビュー全体にUIScrollViewを配置する必要があります。 UIScrollViewの場合、contentSizeを3つの画面幅に設定し、3つのUIViewを配置します（それぞれ1つの画面のサイズ）。 さらにいくつかの設定があり、スワイプで美しいページングがあります。 フィニッシュラインでは、UIScrollViewデリゲートメソッド-scrollViewDidEndDecelerating：を実装し、それに応じて、内部UIViewを更新します。 無限スクロールの複雑さ-別の目に見えるUIVeiw、同様のUIScrollViewを埋め込むと、水平および垂直の反転につながります。

次の質問：ライブラリーの更新-issues.plistをリロードし、既存のメソッドで処理することになります。 ここで、ジャーナルがライブラリに追加されたかどうかを確認しました。 確かに、実質的な問題が発生します。 多くの雑誌では、ユーザーは新しい雑誌がリリースされるまでにそれを忘れることができます。 そして、ここでは、不適切なことに、手近にAPNSがあります。 これは別のトピックです。 私は認めます、私はそれを初めて習得しませんでした。 とにかく、私は私のジャーナルでこれらの通知をしませんでした。 鍵となる障害は証明書であり、むしろその変換の必要性でした。



それにもかかわらず、APNS：まず、通知送信サービスです。 プログラムは、実行中のデバイスのIDを取得して、サーバーに転送できます。 まず、これらの通知をサブスクライブする必要があります。 

 @implementation AppDelegate - (BOOL)application:(UIApplication *)application didFinishLaunchingWithOptions:(NSDictionary *)launchOptions { // Register device for receiving push notifications [[UIApplication sharedApplication] registerForRemoteNotificationTypes: (UIRemoteNotificationTypeBadge | UIRemoteNotificationTypeAlert)];


その後、UIApplicationDelegateプロトコルにはアプリケーションdidRegisterForRemoteNotificationsWithDeviceToken：メソッドがあり、DeviceToken（id）があり、そこからサーバー情報をマージできます。 近くにアプリケーションがあります：didReceiveRemoteNotification：メソッド。通知を受け取った後に何かを行うことができます。



サーバー側では、DeviceTokenであるリクエスト（HTTP）を受け入れて、データベースに保存する必要があります。 さらに、購入したチケットに応じて、またはサーバーに展開されているものに応じて、APNSに送信するメッセージを探す必要があります。 APIによれば、証明書とパスワードを添付することにより、ペイロードとDeviceTokenの両方を含むメッセージを接続して送信します。 まだ微妙な違いがありますが、これは別の問題です。



とりわけ、iOSマガジンにはさらに2つの制限があります。 羽-プライバシーポリシーのURL。 技術面から見ると、非常に簡単です。 2番目の制限は、iTunesConnect内部購入を使用してコンテンツを配布する必要があることです。 私が理解しているように、これはユーザーが私の無料の雑誌を0お金で買わなければならないことを意味します。



合計すると、iTunesConnectのアプリ内購入は、ストアキットの使用を意味します。 UserDefaultsでは、isFreeSubscribedプロパティを取得しました。 私はタプで確認し、NOの場合はアラートを表示します。 同意により、署名する

さらに、デリゲートメソッドを実装し、isFreeSubscribedをYESに設定して、In-App Purchaseを忘れる必要があります。 SKProductsRequestのProductIdは、「アプリ内購入の管理」の下のiTunesConnectから取得する必要があります。
