Glibcライブラリのテスト実験



PVS-Studioを使用してglibcライブラリを確認する実験を実施しました。 実験の目的は、アナライザーがLinuxプロジェクトをどの程度正常にチェックできるかを確認することです。 これまでのところ悪いことができます。 非標準の拡張機能を使用しているため、多くの誤検知があります。 しかし、それでも面白いものを見つけることができました。



glibc



glibc -GNU Cライブラリ（GNUライブラリ）。 Glibcは、システムコールと、open、malloc、printfなどの基本機能を提供するCライブラリです。 Cライブラリは、すべての動的にリンクされたプログラムに使用されます。 GNUオペレーティングシステム用のFree Software Foundationによって作成されています。 glibcはGNU LGPLライセンスの下でリリースされています。



ウィキペディアからの改編： glibc 。



少し前に、新しいバージョンのglibcライブラリがリリースされたというニュースがインターネットに登場しました。 このため、 PVS-Studioアナライザーを使用してこのライブラリをテストする必要がありました。 つまり、glibc-2-19-90バージョンがチェックされました。 残念ながら、数週間は気が散りましたので、今だけ記事を書く時間を見つけました。 私はいくつかの静的アナライザーの大規模な比較に忙しかった。 これは、CppcheckやVisual Studio 2013の静的アナライザーよりも優れているものを尋ねることを止めないため、私たちにとって非常に重要なタスクです。したがって、glibcは少し待たなければなりませんでした。



ひどいものを見つけるとは思っていませんでしたが、実際には見つかりませんでした。 glibcライブラリは非常に高品質であり、多くのパーサーによって検証されています。 少なくとも、これらは次のとおりです。
  • コベリティ
  • Clang;
  • Cppcheck。
したがって、少なくとも何かを見つけることは、すでに大きな成果です。

未定義の動作



glibcライブラリはできるだけ移植性の高いものにすべきだと思われます。 ただし、携帯性の観点から安全とは言えない多くのせん断構造があります。



シフトに関するC標準が示すことは次のとおりです。



整数のプロモーションは、各オペランドで実行されます。 結果の型は、昇格された左オペランドの型です。 右のオペランドの値が負であるか、昇格した左のオペランドの幅以上である場合、動作は未定義です。



E1 << E2の結果は、E1の左シフトE2ビット位置です。 空きビットはゼロで埋められます。 E1に符号なしの型がある場合、結果の値はE1 * 2 pow E2で、結果の型で表現可能な最大値よりも1を法として減じられます。 E1に符号付きのタイプと非負の値があり、E1 * 2 pow E2が結果のタイプで表現できる場合、それが結果の値です。 それ以外の場合、動作は未定義です。



5 E1 >> E2の結果は、E1を右シフトしたE2ビット位置です。 E1に符号なしの型がある場合、またはE1に符号付きの型と非負の値がある場合、結果の値はE1 / 2 pow E2の商の整数部になります。 E1に符号付きタイプと負の値がある場合、結果の値は実装定義です。



標準から、負の数をシフトするのは間違っているということです。 ただし、これはglibcライブラリでは非常に一般的な操作です。



左シフトの例： 

 static void init_cacheinfo (void) { .... count_mask = ~(-1 << (count_mask + 1)); .... }


V610未定義の動作。 シフト演算子 '<<を確認してください。 左のオペランド '-1'は負です。 cacheinfo.c 645



右シフトの例： 

 utf8_encode (char *buf, int val) { .... *buf = (unsigned char) (~0xff >> step); .... }


式「〜0xff」のタイプは「int」で、-256です。
