なぜ仮想化が必要なのですか？

「仮想化」という言葉は、最近IT環境で一種の「ファッション」になっています。 ハードウェアとソフトウェアのすべてのベンダー、すべてのIT企業は、仮想化がクールでモダンであり、誰もが必要としていることを満場一致で叫んでいます。 しかし、マーケティングのスローガンを追うのではなく（そして、時にはゲーベル自身がen望で死ぬこともあります）、この流行語を単純な「技術者」の観点から見て、必要かどうかを判断しましょう。 。







仮想化の種類





したがって、まず、仮想化は3つのタイプに分けられます。

  • 仮想化を表示
  • アプリケーションの仮想化
  • サーバー仮想化

このようなシステムには2つの欠点があります：第1に、より強力なサーバーを購入する必要がある（これは、アプリケーションをローカルで実行するのに十分なTTXを備えた多くのクライアントワークステーションよりも安くなる可能性がある）と、第2に、ターミナルサーバーの形で単一障害点が発生することです。 この問題は、クラスターまたはサーバーファームを使用することで解決されますが、これによりシステムのコストがさらに高くなります。



アプリケーションの仮想化は非常に興味深い、比較的新しい方向です。 ここでは彼について詳しく説明しません。これは別の記事全体のトピックだからです。 つまり、アプリケーションの仮想化により、独自のサンドボックス（サンドボックスと呼ばれることもあります）で個別のアプリケーションを実行できます。 この方法は、多くの問題を解決するのに役立ちます。 まず、セキュリティ：隔離された環境で実行されているアプリケーションは、OSや他のアプリケーションに害を及ぼすことはできません。 次に、すべての仮想化アプリケーションを1つのソースから一元的に更新できます。 第三に、アプリケーションの仮想化により、同じ物理PC上で、互いに競合する複数の異なるアプリケーション、または同じアプリケーションの複数の異なるバージョンで実行することができます。 アプリケーションの仮想化の詳細については、たとえば、次のWebキャストを参照してください。www.techdays.ru/ videos / 1325.htmlいつか、このトピックに関する記事を書くことになるかもしれません。



最後に、 サーバーの仮想化に進み、詳細に説明します。

サーバー仮想化は、プロセッサ、メモリ、ハードドライブなどの特別なコンピューターハードウェアソフトウェアを使用したソフトウェアシミュレーションです。 さらに、このような仮想コンピューターにオペレーティングシステムをインストールすることができ、単純な「鉄」コンピューターと同じように動作します。 このテクノロジーの最も興味深い利点は、同じハードウェア内で複数の仮想コンピューターを実行できることです。一方、すべての仮想コンピューターは互いに独立して動作できます。 何に使用できますか？

最初に思い浮かぶのは、サーバー仮想化をトレーニングとテストの目的に使用できることです。 たとえば、新しいアプリケーションやオペレーティングシステムは、仮想環境で商用運用する前にテストできます。これ専用のハードウェアを購入したり、何か問題が発生した場合にITインフラストラクチャを麻痺させるリスクはありません。



ただし、これに加えて、サーバー仮想化は実稼働環境でも使用できます。 これには多くの理由があります。

仮想化を使用すると、統合によりサーバーの数を減らすことができます。つまり、以前は複数のサーバーが必要でしたが、仮想環境で1つのサーバーを配置して必要な数のゲストオペレーティングシステムを実行できます。 これにより、機器の取得コストが節約されるだけでなく、エネルギー消費、ひいてはシステムの熱放散が削減されます。したがって、より強力ではなく、したがってより安価な冷却システムを使用できます。 しかし、このコインには表裏があり、表裏にはありません。 実際、仮想化に基づいたソリューションを実装する場合、新しいサーバーを購入しなければならない可能性が高いです。 実際には、仮想サーバーは物理サーバーのハードウェアリソースを使用するため、より強力なプロセッサ、大量のRAM、およびより高速なディスクサブシステムが必要になります。 さらに、一部の仮想化システム（特にMS Hyper-V）では、ハードウェア仮想化技術（Intel VTまたはAMD-V）およびその他のプロセッサー機能をサポートするためにプロセッサーが必要です。 特に最近までリリースされた多くのプロセッサ、特にx86_32bitはすべてこれらの要件を満たしていないため、非常に機能している古いサーバーを放棄する必要があります。 ただし、1つのより強力なサーバーは、いくつかのより強力ではないサーバーや古いサーバーよりもはるかに低コストである可能性が高く、ほとんどの場合、陳腐化のために変更する時間です。



もう1つの非常に重要なポイントがあります。北部の仮想化により、インフラストラクチャの管理を極限まで簡素化できます。 すべてのシステム管理者が評価する主な利点は、インストールされているゲストOSとそのステータスに関係なく、「ハードウェア」レベル、または「仮想ハードウェア」レベルで仮想サーバーコンソールにリモートアクセスできることです。 したがって、「凍結」サーバーを再起動するために、サーバールームに移動したり、IP-KVMスイッチなどの高価な機器を購入したりする必要はありません。仮想サーバーコンソールに移動して[リセット]ボタンを押します。 さらに、仮想サーバーはスナップショットテクノロジをサポートし（以前の記事を参照）、仮想システムのバックアップと復元がはるかに簡単になりました。



別の議論の余地のない利点は、仮想マシン内で実行されているOS（ゲストOS）が、それが動作する物理サーバー（ホスト）にインストールされている機器を認識できないことです。 したがって、ハードウェアを交換するとき、アップグレードするとき、または新しいサーバーに移動するときでも、ホストOS（ホストOS）でのみドライバーを更新する必要があります。 仮想デバイスのみがゲストOSを「見る」ため、ゲストOSは以前と同じように動作します。



また、仮想環境では、特別なソフトウェアライセンスルールが適用される場合があることを思い出してください（特に、Microsoft Windows Server 2008 Enterpriseのライセンスを購入すると、ゲストとしてOSの4つのコピーを無料で使用でき、Microsoft Windows Server 2008 Datacenterでは通常、無制限の数のゲストを使用できます） OSはプロセッサのフルライセンスの対象です）。



フォールトトレランステクノロジに言及することは間違いありません。 仮想マシンが起動される物理サーバーはクラスターに結合でき、サーバーの1つに障害が発生した場合、自動的に別のサーバーに「移動」できます。 完全なフォールトトレランスを実現することは常に可能とは限りません（特に、MS Hyper-Vでは、このような「突然の移動」は同じように見え、サーバーの突然の停電と同じ結果をもたらす可能性があります）、可能性のあるダウンタイムは大幅に削減されます：「移動」には数分かかります、サーバー自体の修復または交換には数時間、場合によっては数日かかることもあります。 仮想マシンの「移動」が通常モードで発生した場合、ユーザーに気付かれることなく完全に通過できます。 このようなテクノロジーは、ベンダーごとに異なる呼び方があります。たとえば、MSは「ライブ移行」と呼び、VMwareはVmotionと呼びます。 このようなテクノロジーを使用すると、営業時間中にユーザーをお気に入りのアプリケーションから追放することなく、サーバーのシャットダウン（一部のハードウェアコンポーネントの交換、重要な更新プログラムのインストール後のOSの再起動など）に関連する作業が可能になります。 さらに、インフラストラクチャがそれに応じて構築されている場合、実行中の仮想マシンは、負荷の少ないサーバーに自動的に移動でき、逆の場合は、最も負荷の高いサーバーを「アンロード」できます。 Microsoftテクノロジに基づいたインフラストラクチャでは、System Center Virtual Machine ManagerとOperations Managerがこれに使用されます。



結論として、サーバー仮想化のトピック-仮想化は常に同じように役立つわけではないことに注意してください。 特に、負荷の高いサーバー、特にディスクサブシステムの負荷が高いサーバーを仮想環境に転送することは必ずしも良い考えではありません。これらは「重い」DBMS、Exchange Server、特にメールボックスサーバーの役割、およびその他の負荷の高いアプリケーションです。 ただし、負荷の少ないサーバー（ドメインコントローラーAD、WSUS、あらゆる種類のSystem Center * Manager、Webサーバー）を仮想化して、必要とすることもできます。 ところで、少なくとも1つのコントローラーが「鉄」、つまり仮想ではないことが非常に望ましいのは、ドメインコントローラーの場合です。 これは、インフラストラクチャ全体を正常に動作させるために、他のすべてのサーバーの起動時に、少なくとも1枚のCDがネットワーク上ですでに利用可能であることが望ましいためです。



まとめ





それでは、要約してみましょう。どの仮想化が有用である場合と、それが賛否両論あります。

多くのユーザーが同じソフトウェアセットを使用しており、システムが非常に地理的に分散している場合は、ビューの仮想化、またはターミナルサービスの使用を検討する必要があります。



このようなシステムの利点：

  • クライアント側のハードウェア要件の削減
  • ネットワーク帯域幅要件の削減
  • セキュリティを強化する
  • 管理とサポートを大幅に簡素化


短所：

  • パフォーマンスと信頼性の両方に対するサーバー要件の増加
  • 単一障害点の可能性




新しいOSで正しく動作しない、または互いに競合する多くのアプリケーションがある場合、または同じコンピューターで同じプログラムの複数のバージョンを実行する必要がある場合は、アプリケーションレベルの仮想化が必要です。



利点：

  • 安全性
  • 管理の容易さ-アプリケーションへのアクセス権の集中更新と制限


短所：

  • 技術を理解し、実際に実装するのが難しい。




ラック内のスペースを解放し、システムの消費電力を削減し、「サーバー動物園」を取り除く必要がある場合は、ソリューションはサーバー仮想化です。



このようなソリューションの利点：

  • ラックスペースの節約
  • エネルギー消費と熱放散の減少
  • 管理を簡素化
  • サーバーの展開と管理のための広範な自動化機能
  • フェールオーバークラスターとライブマイグレーションによるシステムダウンタイムの削減
  • （Microsoft Windows Serverを使用する場合）ゲストOSのライセンスを保存できます




欠点は、原則として、ターミナルソリューションの場合と同じです。



  • サーバーのハードウェア要件の増加
  • シングルポイント障害の可能性-物理ホストとホストOS
