SSAS 2012：多次元から表形式のデータモデルへ

はじめに



ビジネスインテリジェンスソリューションの基礎としての表形式データモデルは、MicrosoftがPowerPivotと呼ばれるコンポーネントで提案したもので、Microsoft Office Excelにわずかに追加されました。 それ以来、このモデルの重要性に関する議論は今日に至っていません。 新技術に対する謝罪者はその革新的な性質を確信しており、懐疑論者はこれは進化的変化に過ぎないと信じています。 SQL Server 2012 Analysis Services（SSAS 2012）で、Microsoftはテーブルモデルの原則に基づいた分析データベースのサーバー側バージョンを導入しました。 当然、これは議論に新鮮なストリームを追加せざるを得ません。



一見、懐疑論者は正しい。 実際、以下に示すように、純粋に技術的な観点から、この技術には目立ったものは含まれていません。 それでも、表形式データモデルは革新的なステップと見なすことができます。 簡単に言えば、主な革命的な成果は技術分野に全くありません。

このエッセイでは、まず、このモデルの出現につながった原因と状況を示し、次に、そのモデルの革命の問題を明らかにしようとします。



背景



重要な技術ソリューションにはそれぞれ独自の前提があります。 もちろん、表形式モデルの外観にも独自の理由があります。



主な（唯一ではありませんが）前提条件は、ITビジネス分析手法における中小企業への関心が急激に高まったことです。 その結果、インタラクティブなビジネス分析のための便利なツールと方法の検索。 この検索の過程で、いわゆる多次元データ分析がこの分野でほぼ独占的な地位を占めていることが判明しました。 同時に、大企業の分析問題を解決する際に顕著な柔軟性と生産性を提供するこのテクノロジーは、小規模企業のビジネス分析には扱いにくく、不当に複雑であることが明らかになりました。



したがって、小規模企業のビジネスタスクに適合した簡素化された技術が必要になりました。 表形式モデルに基づく表形式データ分析が可能になったのは、非常に素晴らしい決断でした。



（自然な結果は、インタラクティブなビジネス分析の分野での多次元分析の独占の「転覆」でした-転覆せずに革命的な変換は何をしますか？しかし、多次元分析はキャンセルされておらず、わずかに絞られているため、ビジネス分析の分野を共有する必要があります別の非常に有望な技術。）



ビジネスインテリジェンスの別の紹介



残念ながら、さらなるプレゼンテーションの一貫性を保つために、ビジネス分析の簡単な紹介なくしてはできません。 ビジネス分析の詳細な紹介は、インターネットで簡単に見つけることができるので、できるだけ簡潔になるようにします。



まず 、定義を行います。 ビジネス分析は、結論を導き、ビジネス上の意思決定を提案し、予測を行うプロセスです。 もちろん、これはすべて、金融、物流、マーケティングなど、関連するビジネス分野の専門家である人が行うべきです。 実際、そのような専門家はアナリストです。 分析プロセスの初期情報は伝統的にIT構造に配置されるため、アナリストはITスペシャリストのサービスなしにそれを抽出することはできません。 したがって、ビジネス分析のプロセスには、アナリストと情報のプロバイダーとしてのITスペシャリストという2つの関係者が関与します。 プロセスの全体的な効率は、「共生」の成功に依存します。



第二に 、アナリストが必要とする材料は何ですか？ より正確には、ITスペシャリストからどのような資料が必要ですか？ ここでの答えは予想外に簡単です-サマリーテーブルとチャートが必要です。 多くの異なる。



第三に、ITスペシャリストとアナリストの関係は、よく知られているジレンマ「貧しい人に何を与えるべきか-魚か釣り竿か」に従って構築されます。 つまり、ITスペシャリストがアナリストに必要なテーブルと図を作成するか（「魚」オプション）、アナリストが独自のテーブルとダイアグラムを作成し、ITスペシャリストが必要なデータとツールのみを提供します（「釣り竿」オプション）。 これらのアプローチのそれぞれの長所と短所を説明することなく、現在の傾向が明らかに2番目のオプションを支持していることに注目します。



第4に、アナリストは自分でテーブルとチャートを設計するため、便利で比較的単純なツールを提供する必要があります。 そのようなソフトウェアツールの選択は十分に大きいので、あなたが望むなら、あなたはあらゆる好みのためにそれを見つけることができます。 特に、Microsoftはこれらのタスクのために、Pivo​​tTable機能を備えたすばらしい人気のMicrosoft Office Excelツールを提供しています（PowerPivotと混同しないでください。



最後に、 5番目 。 ITスペシャリストにとっての主な問題は、アナリストが必要とするソースデータを最も便利で快適な方法で提供することです。 このタスクは、いくつかの単純なアプローチに基づいています。a）作業に本当に必要なデータのみを選択します。 b）それらの間の論理接続を設定します。 c）このデータに明確な名前を付けます。 d）それらをカテゴリに分類します。 後続のビジネス分析のためのこのような便利で明確なデータ配置は、 セマンティックモデルと呼ばれます。 従来、このようなモデルは、「星」または「雪片」の形で接続されたテーブルのセットです。



したがって、ビジネス分析を専門とするITプロフェッショナルの主なタスクは、次のように簡単に定式化されます：セマンティックデータモデルの作成。



長年、多次元キューブはそのようなセマンティックモデルとして機能していました。 そして、少なくとも特定のビジネス分野では、新しいセマンティックモデルである表データモデルを置き換えることが求められています。



典型的なプロセスチェーン



この移行のロジックを示すために、いくつかの図を示します。

最初に、大企業のビジネスインテリジェンスの典型的なプロセスチェーンを示す簡略図を見てみましょう。







  • 従来、データの主なソースはOLTP（Online Transactional Processing）データベースであり、現在のビジネスプロセス（たとえば、発注および昇格）を表示します。 詳細には触れませんが、これらのデータベースの詳細は、通常、直接的なビジネス分析にはほとんど役に立たないことに注意してください。
  • したがって、これらのデータベースのデータは特別なデータウェアハウスにエクスポートされます。 このようなストレージは、さまざまなソースからのデータを蓄積および結合できます。 重要な点に注意してください-通常、リポジトリのデータ構造は、ソースOLTPデータベースの構造を繰り返しません。 それどころか、彼らは比較的便利で理解可能な形式で情報を入れようとします-通常は「星」または「雪片」の形式で、このレベルですでにセマンティックモデルのプロトタイプを見ることができます。
  • リポジトリは、いわゆる多次元OLAPデータベース（オンライン分析処理-インタラクティブ分析処理）のソースとして機能します。 これらのデータベースでは、データソースビュー（DSV）と多次元キューブ自体の2つの論理レベルを区別できます。 DSVの目的は、後続のキューブ作成のための中間モデルを提供することです。まず、DSVは表示に必要なテーブルのみを選択（除外）します。 第二に、異なるソースから選択されたこれらのテーブルは、リンクによって単一のモデルに結合されます。 DSVは、究極のセマンティックモデルである多次元キューブを作成するための初期構造です。 図からわかるように、1つのOLAPベースに複数のDSVを含めることができ、1つのDSVが複数のキューブを作成するための基盤として機能できます。
  • Excel PivotTableなどのクライアントアプリケーションを使用するアナリストはOLAPデータベースに接続し、目的のキューブを選択し、その構造とデータを使用してピボットテーブルとチャートを作成します。
  • このチェーンには、OLTPデータベース構造、ストレージ構造、DSV、キューブの4つのレベルのデータモデルがあります。 最後の3つのモデル（DW、DSV、およびCube）は通常、アナリストにとって非常に理解しやすい形式であり、1つまたは別の予約があればセマンティックと見なせることに注意することが重要です。




多次元モデルから表形式へ



質問をするのは自然です-このシーケンスにはあまりにも多くのセマンティックモデルがありますか？ 答えは企業の規模によって異なります。 数百、場合によっては数千のOLTPデータベース、数十のキューブおよびストレージを持つ大企業の場合、このチェーンは決して冗長ではありません。 それどころか、それは決定の最大の柔軟性を提供し、したがって、完全に正当化されます。



OLTPアカウントがユニットに移動し、分析のために1つのストレージで十分であり、多くの場合、1つのキューブである小規模企業の状況は異なって見えます。 ここでは、3つの連続したセマンティック（またはほぼセマンティック）モデルの存在が明らかに冗長であることが判明する場合があります。



この場合の簡略図は次のようになります。







ここで、多分、多次元キューブ自体を「内部」で見る時が来ました。これは、これまでの議論では「ブラックボックス」でした。



キューブの基礎は、いわゆる多次元モデルです。 このモデルでは、最終的な目標は集約データのセットであるピボットテーブルを作成することであると想定しています。 したがって、モデルでは、すべてのデータは最初に2つの種類に分割されます。それ以降の集計を目的とするデータ（いわゆるメジャー）と、集計コンテキストを指定するデータ（ディメンション）です。 たとえば、店舗の名前が垂直方向に、年が水平方向に、年間および店舗ごとの総売上が表示されるテーブルを作成します。 この場合、モデル内の販売量はメジャーとして表示され、年とストアは測定の属性として表示され、合計は集計の特殊なケースです。 利便性を高めるために、メジャーとメジャー間の論理関係はモデルで直接指定され、集計関数はモデル内のメジャーにも直接関連付けられます。 そのため、ピボットテーブルを作成するには、必要なメジャーと測定属性を選択するだけで十分です。残りは選択したコンポーネント間の論理接続であり、集約方法は、いわばユーザーに対して「透過的」です。



したがって、多次元モデルは、実際には、 論理的に関連する多くのメジャーとメジャーの組み合わせです 。 ちなみに、「多次元」という名前の由来は、メジャーと測定自体が完全に表形式の「2次元」形式になっていることです。



モデル自体に加えて、キューブにはエンドユーザーにとって興味深い追加機能があります。たとえば、測定属性から階層を作成する機能や、最近では非常に人気のあるコンポーネントが主要業績評価指標（KPI）であることに注意してください。



したがって、キューブは次のように想像できます。







さて、中小企業のプロセスチェーンに戻りましょう。 コンパクトにするために、図には、関心のあるセマンティックモデル（DW、DSV、およびキューブ）を含むコンポーネントのみが示されています。







ここでは、実際に頻繁に遭遇する状況に直面しています-3つのモデルすべて、特にDSVと多次元モデルは非常に似ています。 したがって、解決策はそれ自体を示唆しています-何かを捨てます。 問題は正確に何ですか？



そしてここで、マイクロソフトの開発者は一見、逆説的な決定を下しました。チェーンから多次元モデルを削除し、それによって多次元キューブの概念そのものを削除します！ キューブに残っている機能（階層、KPIなど）は、DSVに直接付加されます！ パラドックスは、DSVを技術的に簡単に削除し、キューブをその確立され実証されたすべての機能とともにそのまま残すことです。 それにもかかわらず、彼らは彼らがしたようにした！



この逆説的な決定の意味を説明する前に、分析ベースがどのように見えるか見てみましょう。 現在では、すでにテーブルモデルデータベースと呼ばれています。 Cubeの機能を継承したDSV自体は、別の名前-Tabular Data Modelとも呼ばれます。







技術チェーンの全体像は次のようになります。







ここでは、お客様にとってほとんど変更されていないことに注意してください。 通常の方法を使用して、たとえば前述のExcelピボットテーブルツールを使用して、表形式モデルのデータベースに接続できます（わずかな技術的な違い-データベースには1つのモデルしか存在できません）。



それでは、革命とは何ですか？



そして今、私たちのエッセイの主な質問に戻ります-この決定の革命的な性質は何ですか？ これまで検討されてきたすべてが、懐疑論者の工場に水を注いでいるようです。 実際、エンドユーザー（アナリスト）にとっては、実際には何も変わっていません。 ITテクノロジーの観点からも、目立ったものは何もないようです。テクノロジーチェーンの過剰なリンクを削除することは、もちろん技術的な些細なことです。



答えとして、私たちがすでに注意を払ってきたパラドックスに再び戻りましょう：マイクロソフトの開発者は、最小の抵抗の経路を取り、追加のリンクとしてDSVを捨てる代わりに、予期せぬ行動を起こしました-多次元モデルを削除しました。 また、ベースから多次元モデルを削除すると、多次元キューブが消え、その後多次元分析の概念が消えました。



逆説的に見えるかもしれませんが、この「多次元性」の除去が革命的な主な成果です！



簡単に言うと、マイクロソフトの開発者は、これらの「多次元」用語を取り除く機会を得ました。これらの用語は、初心者にとっては本当に恐ろしい話であり、ビジネスインテリジェンスの道を歩もうとしている人々にとって深刻な心理的障壁を作り出します。



実際、高等数学の研究所のコースの内容をまだ覚えている初心者にとって、 「多次元データ分析」は「数学的な多次元分析」との明確な関連を引き起こします。 後者は、二重、三重、曲線積分などの素晴らしい概念を持つ数学の非常に尊敬されている分野ですが、ビジネス分析とは何の関係もありません。 それにもかかわらず、ビジネス分析、したがって、未経験者による「多次元分析」に従事するという提案は、再び分析と線形代数の世界に突入するための招待として（しばしば無意識に）知覚されます。 これは通常、熱意からほど遠い感情を引き起こします。



はい、そして、そのような協会に負担をかけられていない人にとって、技術の研究が通常始まる「N次元の立方体を精神的に想像しましょう」というフレーズは、劣等感の発展に大きく貢献するかもしれません。



「多次元キューブ」、「多次元」モデル、「多次元分析」という用語は、普通の人の精神に圧力をかけ、心理的な障壁を作り出し、したがって、この分野の発展を阻むものです。これはマイクロソフトの開発者によって発見された単純な真実です。 「多次元」の用語を取り除き、表や列などの単純で馴染みのある「フラット」な用語に置き換えれば、心理的障壁はなくなります。



「多次元」の用語を取り除き、それに応じて、ビジネス分析の新しいフレッシュフォースの到来を妨げる心理的障壁を取り除くことが、テーブルモデルの新しいテクノロジーの真の革新的なアイデアです。



おわりに



結論として、心理的障壁を手に負えないと考える人のために、私は隣接する地域からの1つの実例を示します-PowerShellを使用したスト​​ーリー。



長年にわたり、Microsoftは管理タスクを自動化する手段としてVBScript（Visual Basicの切り捨てバージョン）を浸透させようとしてきました。 そして、ここ数年、システム管理者（もちろんすべてではありませんが、このような多くのものは一般的に統計的にのみ表示されます）は、この原始的なツールに頑固に反対し、はるかに原始的なコマンドライン言語で自動化スクリプトを作成することを好みました。 この対立は何年も続き、誰かがシステム管理者に新しい高度な自動化メカニズムを提供するアイデアを思い付くまで、それらの通常のコマンドラインであるPowerShellに似ていました 。 そして、物事はスムーズに進みました。率直に言って、正直なところ、PowerShellはVBScriptほど単純ではありませんでしたが、新しいツールキットは熱狂的に受け入れられました。 ここで心理的な瞬間が正しく考慮されたというだけでした。
