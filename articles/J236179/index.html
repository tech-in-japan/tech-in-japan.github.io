Intel IOT開発キットの内容

2月に、 Galileo gen1の Yoctoビルドについて書きました 。これにより、 Galileoの開発が少し簡単になります。 それからほぼ1年が経過し、ガリレオには追随者が現れ始めました- ガリレオgen2 、 エジソン 。 Edisonについて（すでに2か月販売中 ）は、Intel IOT Development Kitについての同じ投稿である別個に作成する必要があります。







私のデスクトップの腺の（混乱のためごめんなさい）、それはGalileo gen1、gen2およびEdisonと互換性があります。

2013年12月から2014年10月まで、私はこの製品のアーキテクトでしたが、プロジェクトの進捗状況、既に起こったこと、計画されていることについて説明します。



なぜこれを始めたのですか？ ネイティブ（プラットフォームの作成者による）GalileoのYoctoアセンブリは8 MBに収まる必要があるため、uClibcを使用します。 8GBのSDカードから起動する場合でも。 また、主にArduino IDEを介してのみプログラミングするために、シャープ化されています。 ただし、GalileoはArduino Uno互換のボードであるだけでなく、パフォーマンスも15年前のスタッフに匹敵するコンピューターですが、最新の周辺機器をサポートしています。 したがって、PCとArduino周辺機器を備えたLinuxコンピューターとしてのプログラミングを最大限に促進することは非常に論理的です。



そのため、開発ツール、ライブラリ、一般的な周辺機器のドライバーを含むYocto 1.6 ビルドを作成することから始めました。 箱から出して、C / C ++、Python、Node.JS、シェルで作成できます。 ディスクイメージの重量はわずか200 MB ですが 、opkg用のマルチギガバイトリポジトリがまだあり、そこからさらに数千のパッケージをインストールできます（/ルートファイルシステムに十分な大きさのSDカードがある場合）。 2月のアルファ版と比較して、ダウンロードに含まれる新しいサービスがたくさんあるにもかかわらず、現在のビルドの読み込みは3倍高速です。 そして、sysvinitの代わりにsystemdだけです。



Arduino IDEとの互換性を維持するために、ディスクイメージにはuclibcとeglibcの両方が含まれています。 残念なことに、スケッチから有用なライブラリを取り出すことは不可能であり、 IPCを使用する必要があります。



PCの周辺に十分なライブラリがある場合、最も単純で安価なセンサー/アクチュエータには、Arduinoまたはmbedライブラリのみがあります。 高度なセンサーのドライバーには多くのプラットフォーム固有のコードがあるため、それらは移植する必要があります。 そこで、自転車を発明することを決め、GithubでMITライセンスを持つ2つの完全にオープンなプロジェクト-libmraaとUPMを公開しました。 libmraaはCで、UPMはC ++（ネイティブC ++別名クラス付きC）で記述されています。 どちらもビルド時にライブラリを生成し、PythonおよびNode.JSでプログラミングするときに使用できます。



libmraaは、GPIO、ADC、PWM、I2C、SPI、RS232として使用できるGalileo / Edisonピンへのインターフェースを提供します。 これはドライバーではなく、すべてがユーザーモードで行われます（さようなら、リアルタイム！）。 Arduinoスケッチのように、Galileo gen1、gen2、Edisonでは、/ sys / class / *デバイスの上で実行されます。 つまり、これらのデバイスのArduinoピンへのバインドが単調で些細なものであれば、それは単純なラッパーになります。 残念ながら、これらのプラットフォームはそれぞれ、入出力の分野で独自の特性と制限があります。 たとえば、Galileo gen2の詳細についてはこちらをご覧ください 。 Minnow maxとBaytrail NUCのサポートは近日中にサポートされます。



UPMはlibmraaの上で実行され、センサー/アクチュエータロジックを実装します。 したがって、センサー/アクチュエータコードには、それらが動作するプラットフォームに関連するロジックはありません。 （#ifdef UNO / DUO / TREが眼球であるArduinoと比較してください。）UPMはこれまで40台のデバイスしかサポートしていませんが、コミュニティの助けを借りて私たちのチームは1年で数百人を書きます。



通常のプログラミングツールに加えて、Intel GalileoのIOT devkitディスクイメージには、あまり一般的ではないものも含まれています。 ルーマニアのスタートアップWyliodrinは、Intel Galileo、Edison、およびRaspberry Pi上のセンサー/アクチュエーターのグラフィカルプログラミングを容易にするBlocklyベースのプログラミング環境を開発しました。 WyliodrinのメンバーとIntel Galileo所有者への特別オファーは、別の記事に値します。



アセンブリには、node.jsデーモンと、 Intel IoT分析クラウドとの連携を簡素化するAPIも含まれています 。 一般的に、IoTのクラウドバックエンドは非常にファッショナブルなトピックであり、多くの有名なベンダーが既に発表（Azure Intelligent Systems Service）または独自のバージョンをリリースしています。 いくつかのテストを行いましたが、ガリレオまたはエジソンに接続されたセンサーからのデータは、さらに処理するためにクラウドに簡単に送信されます。 もちろん、Intel IOT Analyticsは最も簡単に使用できるようになりました-既にIOT devkitに組み込まれているため、例があります。



IOT開発キットの一部であるGalileoのディスクイメージについて何度も言及しました。 一部の読者は、2つの質問があります。devkitには他にどのようなパーツがあり、Edisonはどうですか？



ディスクイメージ、opkgパッケージリポジトリ、IoT AnalyticsおよびWyliodrinのクラウドサービスに加えて、ホストでの開発を容易にする2つの要素を追加しました（それでも、GalileoまたはEdisonで直接大きなものを開発するのはあまり快適ではないかもしれません）。 IOT開発キットは、Node.JSおよびLinux、Windows、Mac OS向けのEclipseの特別なビルドで、C / C ++開発用のホストとして構成されたIntel XDKをサポートします。



さらに、IOT開発キットは、ハッカソン（例えば、 モスクワで間もなく開催されます）、一部のアカデミックプログラムに参加するか、ガリレオgen1キットまたはgen2キットを購入するだけで、具体的な形で入手できます。



2番目の質問に関しては、すべてがシンプルです。Edisonプラットフォームを作成したチームは、uClibcを使用する必要性に制限されず、Yocto 1.6.1に基づいてEdisonのディスクイメージを作成しました。 libmraaとUPMをアセンブリに含め、上記のすべての利点（XDK、IOT devkit build Eclipse、IoT分析、Wyliodrin、およびArduino IDE）との互換性を確保し、ターゲット上のすべてのライブラリーと互換性がありました。



秋のインテル開発者フォーラムで、開発キットサイトとフォーラムを開きました 。 しかし、ガリレオとエジソンはすでに独自のサイトとフォーラムを持っているので、混乱がありました。 情報の一部はメーカーのこのサイトにのみあり、一部は当社のサイトにのみあり、一部は複製されています。 たとえば、最近、彼らはドブス博士の開発キットについて書いたが、記事の著者はこの混乱を完全に理解できなかったように思えた。 この記事の読者がプログラマ向けの最も古い雑誌の編集者よりもよく理解することを願っています。



このサイトでは、ダウンロード用にディスクイメージとEclipse IDEをアップロードし、すべてIOT devkit 1.1ベータ版と呼びました。 ベータは、チームにパートタイムのQAエンジニアが1人しかいなかったためです（ただし、全員が自分のドッグフードを食べたため、明らかなバグが一掃されました）。 数か月以内に、すべてのバグを修正し、新しい興味深いセンサーのサポートを追加し、Eclipse IDEにウィザードを追加して新しいプロジェクトを作成し、ハッカソン参加者の多くの要望を考慮して、攻撃的な単語「ベータ」を取り除きます。
