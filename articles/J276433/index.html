Veeam Backup FREE Editionの新バージョンがリリースされました：NFRキーについての簡単な概要と有用な情報がフルバージョンに

Veeamソリューションの新しいバージョンをレビューする伝統を支持し、最近リリースされた無料のVeeam Backup FREE Editionのバージョン9.0について簡単に説明します。 （前回のリリースでは、 Habréの別の投稿が商用版と無料版のニュースに捧げられましたが、今回はVeeam Availability Suite 9.0に含まれるVeeam ONEのレビューなど、いくつかの記事が計画されています）。

それで、新しいバージョンの開発者はユーザーに何を喜ばせましたか？









Veeam BitLooker



バックアップの変更から始めましょう。 ページングファイル（ pagefile.sys ）がバックアップに含まれていないという事実に加えて、現在は休止状態ファイル（ hiberfil.sys ）、および削除されたファイルとブロックもデフォルトでは含まれません。

これが発生すると、次のことが起こります。

  1. 仮想マシンのデータのコピーを開始する前に、Veeamはゲストハウスのマスターファイルテーブルを分析してpagefile.sysおよびhibernate.sysブロックの存在を検出し、検出されたブロックをゼロで詰まらせます。
  2. 次に、VMイメージをターゲットの場所にコピーするとき（つまり、実際のバックアップで）、チェックが実行されます。VMイメージのデータブロックにそのような「null」ブロックのみが含まれる場合、このブロックはバックアップに移動せず、他のブロックが含まれる場合-役立つ-それは来ています。


写真の左側には、ステップ1の後に何が起こったのか（「ヌル」ブロックはオレンジ色でマークされています）、右側にはステップ2の結果（つまり、バックアップに行くもの）が表示されます。







Veeamはダーティブロックでも同じことを行います。 削除済みとマークされたもの：最初（データ転送の開始前）に、MFTを使用してそのようなブロックを識別し、ゼロで埋めてから、VMイメージを分析し、意味のある有用なブロックのみを保管場所に転送します。 「ヌル」ブロックはバックアップされません。 そのため、Veeam BitLookerと呼ばれる新機能は、トラフィックの削減とバックアップパフォーマンスの向上に役立ちます。

重要な注意：デフォルトのブロック除外設定は、ゲストファイルシステムがMicrosoft NTFSであるマシンにのみ適用されます。



ダイレクトNFSアクセスのデータ転送方法（VMwareプラットフォームの場合）



NFSストレージにファイルが保存されているマシンの場合、特別に設計されたVeeam NFSクライアントの助けを借りて、ハイパーバイザーをバイパスする新しいデータ転送方法が提案されます。 この方法では、Veeam Backup Proxyコンポーネントを使用します（デフォルトではVeeamバックアップサーバーで実行されます）。そのためには2つの条件を満たす必要があります。

  1. NFSストレージへのアクセス
  2. 最小の読み取り専用/書き込み権限とNFSストレージへのルートアクセス。


メソッドは次のように機能します。

  1. Veeam Backup Proxyコンポーネントは、ストレージ上のVMの場所を見つけるために、ESX（i）にリクエストを送信します。
  2. ハイパーバイザーが目的のVMを見つけます。
  3. ストレージシステム上のVMディスクの場所に関するメタデータ（VMデータブロックの物理アドレス）を受け取ります。
  4. 次に、このメタデータをVeeam Backup Proxyコンポーネントに送信します...
  5. ...それらを使用して、ローカルネットワークを使用してストレージからVMデータブロックを直接コピーするユーザー。
  6. 次に、Veeam Backup Proxyはこれらのブロックを処理し、ターゲットストレージに送信します。


新しい方法は（上記の条件が満たされると）自動的にアクティブ化され、ハイパーバイザーの負荷を最小限に抑え、仮想マシンのバックアップ、リカバリ、移行の速度を向上させることができます。 ここで詳細に説明します （ただし、当面は英語のみです）。

重要！ この方法がバックアップ中に正しく機能するには、VMware Toolsの静止が使用されないことを確認する必要があります-VeeamZIP設定を開き、 ゲスト処理セクションでゲスト静止を無効にする（クラッシュの一貫したバックアップを実行する）がチェックされていることを確認する必要があります







新しいテープアーカイブ



まず、並列記録をサポートしています。 ファイルをテープにアーカイブするデバイスに複数の書き込みヘッドがある場合、アーカイブタスクの並列処理を使用できます。この場合、複数のヘッドが同時にテープに書き込みます。 これは、たとえば、非常に限られたバックアップウィンドウ時間で大量のデータを記録する必要がある場合に便利です。

このオプションは、カスタムメディアプールで機能します。 そのようなプールごとに、並列記録に使用できる書き込みヘッドの最大数を設定できます。 これを行うには、プールのプロパティを開き、ウィザードの[オプション]という手順に進み、[ このメディアプールを使用するジョブの並列処理を有効にする ]オプションを選択します。

次に、最大数のヘッドをいくつ使用できるかを示します（デフォルトは2）。







第二に、ストレージプールについて話しているため、v9ではすべてのプールがグローバルになることをお知らせします。 これは、プールが以前のようにライブラリ（書き込みデバイス）に関連付けられなくなったことを意味します-新しいバージョンでは、デフォルトのストレージプールに異なるライブラリのカセットを含めることができるため、たとえば、毎回カタログを作成する必要がなくなりましたカセットをあるデバイスから別のデバイスに再配置しています。 ご覧のとおり、 テープのステップで、プールにカセットを提供するライブラリのリストで、 複数のライブラリがデフォルトで指定されています。







重要な注意：アーカイブ中にデバイスレベルで組み込みの暗号化（ハードウェア暗号化）を使用する場合、暗号化されたカセットを読み取るには、ライターと同様のデバイス（つまり、同じ標準に従って組み込みの暗号化が有効になっているデバイス）が必要になることに注意してください

第三に、「不明な」ライブラリ、つまりWindowsドライバーが見つからなかったライブラリのサポートが実装されています。 そのようなデバイスを使用するには、SCSI経由で接続する必要があり、Microsoftデバイスマネージャーのデバイス管理コンソールでは、そのようなデバイスがUnknown Medium Changerとして表示されることに注意してください。 もちろん、メーカー（OEM）の "ネイティブ"ドライバーがあるデバイスを使用することをお勧めします。

サポートされているデバイスの詳細については、 システム要件を読むことをお勧めします 。



スタンドアロンコンソール



これで、コンソールをサーバーとは別にインストールでき、RDPセッションや、接続を使用してバックアップを変更したい他の人が落ち着いて作業できないため、緊張することがなくなります。 デフォルトでは、コンソールはバックアップサーバーとともにインストールされますが、セットアップでコンソールを個別にインストールすることを選択できます。







コンソールのシステム要件については、 こちらをご覧ください （Microsoft .NET Framework 4.5.2がセットアップに含まれています）。 もちろん、Veeamバックアップサーバーへの適切な接続（少なくとも1 Mbit / s）を確保する必要があります。そうしないと、スタンドアロンコンソールのユーザーの喜びがすぐに消えてしまいます。

インストール時に、コンソールをインストールするフォルダーへのパスを指定します。 インストールと起動後、接続するサーバーを通知する必要があります。 ローカルサーバーでない場合は、サーバー、ポート（デフォルトでは9392 ）、ユーザー名、およびパスワードを指定します。 このWindowsセッションで作業するアカウントを使用し、 [Windowsセッション認証を使用する]にチェックマークを付けます 。 いずれの場合でも、アカウントはVeeamサーバー上のローカルユーザーグループ、またはアクセス権を持つドメイングループのメンバーである必要があります。

有用：接続パラメーターを入力するたびにVeeam Backupとのセッションを開始しないようにするには、デスクトップのショートカットを作成して保存します（つまり、 ショートカットの保存をクリックします ）。







セットアップ時に、Veeam PowerShellスナップインがコンソールと共にインストールされ、 Connect -VBRServerコマンドレットを使用してVeeamサーバーに接続できることも便利です。

通常どおり、さらに作業が行われます。 スタンドアロンコンソールの便利な機能には、ローカルリカバリ中に（つまりローカルサイトに）マウントサーバーとして機能する機能があります。 たとえば、コンソールから仮想マシンのゲストOSファイルのリカバリを開始すると、 現状のままのバックアップがこのマウントサーバーにマウントされます。「マウントポイント」はフォルダーC：\ veeamflr \ VM_nameに作成されますが、サーバーには作成されませんVeeamバックアップ。ただし、コンソールが実行されているローカルマシン上。 その後、すべてがいつものようになります。VeeamBackup Browserで必要なファイルを選択し、保存先を指定します。







このようにして、Veeamバックアップサーバーは、以前のバージョンで実行していた「アセンブラー」の追加の役割から解放されます。 また、アプリケーションオブジェクトを復元する場合、Veeam Explorersツールはスタンドアロンコンソールからも起動できることに注意してください（ローカルリカバリによりトラフィックが節約されます）。 修復の詳細については、 こちらをご覧ください 。

注：コンソールを備えたマシンに本格的なVeeamバックアップサーバーを配置するには、まずコンソールを分解する必要があります。
