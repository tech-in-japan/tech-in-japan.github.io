Mobius 2016：新しい方法でのモバイル開発

今年、初めてのiPhoneの販売は成長を止め、モバイル市場全体もかつての急速な成長を誇っていません。また、新しいスマートフォンとOSの新しいバージョンはもはや驚くべきものではありません。 これは、モバイル開発に停滞が来ていることを意味しますか？ それに関する報告は、同じ会議で前の年に言われたことを完全に繰り返し始めませんか？



サンクトペテルブルクで開催されたメビウス会議は、この質問と他の多くの質問の両方に答えることを可能にしました。 カットの下-何がそこにあったかについての詳細な物語。











会議は「開発者の夏祭り」の一部となりました。1日前に DotNext が 、1日後にHolyJSが開催されました 。 3つの異なる世界を比較する機会であることが判明しました..



トピック#UXmattersはVibe Elsingのオープニング基調講演で述べられており、観客は完全な「キャプテンシー」を待っていたと思うでしょう。 そうですね 、はい、UXは重要です。 しかし、「悪いユーザーエクスペリエンス」の例の中には、非自明で全体的なものさえありました。 スピーカーは巨人を大胆に攻撃しました。たとえば、起動時にアニメーションをやりすぎたとしてUberを批判しました：「私はこれを行った人と話しました。彼は、データがバックグラウンドでロードされている間に何かでユーザーを楽しませる必要があると説明しました。 しかし、あなたが空港にいるとき、時差ぼけに苦しんでいて、インターネット接続が不十分な状態でタクシーを呼ぼうとしていると、アニメーションはまったく話題になりません。」







さらに、Google Developer Expertの栄誉ある称号は、VibaがGoogleの最新の再設計を批判することを止めませんでした。 最近、Googleフォトアプリケーションでは、メニューを左側に残す「ハンバーガー」に加えて、3つのメインセクションを選択できる下のパネルもあります。 スピーカーによると、両方の同時存在はユーザーを混乱させます。最初は誰もがクリックする場所を理解せずに目を走らせます。







「アプリケーションストアで製品のレビューを検討する」という講演者の建設的な提案でも、誰もが明確に同意することはほとんどありません。 多くの人々は、そのようなレビューは最も知的なユーザーではなく、最も大声で書かれていると考えているので、特別な注意を払うべきではありません。 しかし、Vibeによれば、ここには矛盾はありません。「はい、レビューは愚かである可能性がありますが、有用な情報はそれらから抽出できます。」



次に、別のGoogle Developer ExpertであるDenis Neklyudovが 、5月のGoogle I / Oでのプレゼンテーションについて話し、個人的に訪問しました。 報告書の冒頭で、講演者は「I / Oからのニュースをフォローすれば別の部屋に行くことができます」と率直に言ったが、彼のスピーチは他人の言葉の記憶された繰り返しではなかった。







Googleは、基調講演のI / Oが世界中で視聴されることを知っており、デニスが開発者にとって重要なことについて話している間、普通のユーザーを感動させようとします。 したがって、彼は大声で提示されたGoogleアシスタントについては話さず（SDKはありません）、VRプラットフォームDaydreamのようなトピックをすばやく実行しました（まだ全盛期まで生きる必要があります）が、すぐにインスタントアプリについて話し始めました（インストールせずにAndroidアプリケーションを起動します） ） Googleが基調講演で一般的に無視したものであることが判明しました。ChromeOSでAndroidアプリケーションを実行する機能で、モバイル開発者は一晩でデスクトップになります。



そして、Firebaseに関するニュースは、開発者にとって最も興味深いものになりました（Neklyudovのスライドに書かれていたので、「実際、肉体はすべてここにあります」）。 プラットフォームの多数の革新についてコメントし、彼はGoogle自身が考えていないような事例を引用しました。「リモート構成を使用すると、特定のカテゴリのユーザーのアプリケーションの外観をリモートで制御できるようになりました。 たとえば、3月8日にアプリをピンクに変えるには」 一般に、モバイル開発には目立った変化があります。YouTubeI / Oブロードキャストを視聴した一般ユーザーでさえ、このことに気付きませんでした。







その後、 Kirill Popov （Odnoklassniki）は、 FrescoライブラリがAndroidでの画像の操作にどのように役立つかについて話しました。 惨めな卵のトピックは価値がないと思われるかもしれません。写真を表示するだけなら、1時間話すことは何ですか？ しかし、ポドフはOdnoklassnikiのアプリケーション例を使用して、実際のプロジェクトで「単純なタスク」が非常に難しいタスクに変わる方法を明確に説明しました。



「これがユーザーのアバターです。 写真をリクエストし、キャッシュをチェックし、写真をダウンロードして、キャッシュに入れる必要があります。 同時に、アバターは正方形ではなく、特別な係数を使用したベジェ曲線を使用して切り抜きます。 そのため、画像を変更し、再度キャッシュに入れてから表示する必要があります。 ただし、この間、ユーザーは既に何かを見る時間があるため、プレースホルダーが必要です。 ここで背景を見てみましょう：アバターで作られていますが、この場合は切り取られていませんが、ぼやけています（このアバターは既にダウンロードされているため、サーバーに新しいリクエストをしたくありません）。 そして、スクロールするときは、新しい画像をすばやく読み込む必要があります。これはすべて、少量のメモリを搭載したデバイスで開くことができます... "







強力なツールが必要であることは驚くことではありませんでした。そして、フレスコは、必要なすべてのものをすぐに見つけた唯一のものであることが判明しました：最適なメモリ使用、プログレッシブjpegとの連携、マルチレベルキャッシング、画像の後処理、使用の柔軟性。 しかし、シリルが認めたように、小規模プロジェクトの場合、この豊富さはすべて「スズメの大砲」の状況である可能性があります。 だれかがもっと簡単なグライドを必要とし、フレスコを習得することの難しさは、その使用の利点を超えるでしょう。
