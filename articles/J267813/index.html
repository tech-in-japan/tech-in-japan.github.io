InfoboxCloudクラウドで開発用のインフラストラクチャを開発しています。 パート1：Gitlabを起動する

この一連の記事では、InfoboxCloudクラウドに開発インフラストラクチャを展開するプロセスについて説明します。 アプリケーションスタックを簡単にデプロイするために、Dockerを使用します。



最初の記事では、 Gitlabをデプロイします 。これには以下が含まれます。



  • gitソースコード管理システムのウェブインターフェース 。これはgithubに最もよく似ています
  • ユーザーアクティビティの便利な表示
  • ファイルブラウザ
  • Wiki
  • コードレビュー機能
  • バグトラッカー
  • コードスニペットを作成する機能
  • Webフックを挿入する機能
  • ビルドサーバー


などなど。







無料のリポジトリを無制限に作成し、アクセスおよびセキュリティポリシーを構成できます。 Gitlabの前にVPNを展開する場合、パブリックIPサーバーにGitlabを提供し、企業ネットワークからVPNを介してクラウドの内部ネットワークに接続することでそれを操作することはできません。



環境の準備



1. CentOS 7でサーバーを作成して、 DocboxをInfoboxCloudにインストールします 。 Dockerが機能するためには、まさに今必要な仮想マシンです。そのため、サーバーを作成するときは、 「OSカーネル制御を許可する」ボックスを必ずチェックしてください。



InfoboxCloud for Dockerでサーバーを作成する方法
InfoboxCloudにまだアクセスできない場合は、 注文してください 。



登録後、電子メールでコントロールパネルにアクセスするためのデータを受け取ります。 https://panel.infobox.ruでコントロールパネルにログインします。



サブスクリプションの[クラウドインフラストラクチャ]セクションで、[新しいサーバー]をクリックします（必要に応じて、ドロップダウンメニューの右上隅にあるサブスクリプションが変更されます）。







必要なサーバーパラメータを設定します。 下のスクリーンショットに示すように、サーバーに1つのパブリックIPアドレスを与え、 「OSカーネル制御を許可する」ボックスをチェックしてください。







使用可能なオペレーティングシステムのリストで、CentOS 7を選択し、サーバーの作成を完了します。







その後、サーバーにアクセスするためのデータが電子メールで送信されます。



CentOS 7でサーバーを作成した後、 SSHを介してサーバーに接続します。



このようなサーバーでDockerとDockerを操作するための便利なユーティリティをインストールできるスクリプトを準備しました。 必要な設定は自動的に行われます。



コマンドを実行してDockerおよびComposeをインストールします。 



bash <(curl -s http://repository.sandbox.infoboxcloud.ru/scripts/docker/centos7/install.sh)


Dockerとcomposeがインストールされます。



サーバーをクリックして「イメージを作成」をクリックすると、コントロールパネルに dockerがインストールされたイメージを作成できます。 その後、Dockerを使用してイメージから新しいサーバーを作成し、このステップを再度実行しないでください。



クラウドサーバーのシステム要件



Gitlabの快適な操作のために、100〜500ユーザーの2つのCPUコアと2Gb Ramが推奨されます。

テスト中に可能な限り保存する必要がある場合は、Gitlabサーバーに1 CPUコア1 GHzを指定できますが、少なくとも2 GBのメモリが必要です。



多数のユーザーと連携する必要がある場合：



CPU

  • 4 CPUコア-最大2,000ユーザー
  • 8 CPUコア-最大5,000ユーザー
  • 16 CPUコア-最大10,000ユーザー
  • 24 CPUコア-最大15,000ユーザー


RAM

  • 4GB RAM-最大1,000ユーザー
  • 8GB RAM-最大2,000ユーザー
  • 16GB RAM-最大4,000ユーザー
  • 32GB RAM-最大8,000ユーザー
  • 64GB RAM-16,000ユーザー


さらにユーザーが必要な場合は、複数のサーバーで Gitlab を実行できます 。



Gitlabをインストールする



Gitlabの最新バージョンを迅速に展開するためのファイルはすでに準備されています。 GitlabはDockerコンテナーにデプロイされます。 その前に、Gitlabで公式イメージを更新します。Gitlabの最新バージョンを含むOS上のすべての更新プログラムを受け取ります（公式イメージは遅れて更新されます。最新の安定版が以前にあります）。



次のコマンドでgitをインストールします。 



 yum install -y git


ユーザーディレクトリに移動します。 



 cd ~


gitlabを展開するために必要なファイルをダウンロードします。 



 git clone https://github.com/trukhinyuri/gitlab-docker.git


次に、展開用のファイルがあるディレクトリに移動します。 



 cd ~/gitlab-docker


次のファイルとディレクトリがディレクトリに存在します。



  • Dockerfile ：公式のgitlabイメージで実行されるアクションについて説明しています。
  • docker-compose.ymlは、結果のイメージをデプロイする方法、転送するポート、ホストにマウントするコンテナーのフォルダーを記述します。
  • configフォルダー：gitlabはこのフォルダーに構成ファイルを保存します
  • データフォルダー：gitlabはこのフォルダーにデータファイルを保存します
  • logsフォルダー：gitlabはこのフォルダーにログを保存します


Dockerfileの内容： 

 FROM gitlab/gitlab-ce:latest MAINTAINER Yuri Trukhin <yuri@trukhin.com> ENV REFRESHED_AT 2015.09.27.004 ENV GITLAB_SHELL_SSH_PORT 8005 RUN apt-get update RUN apt-get -y upgrade EXPOSE 80 EXPOSE 443 EXPOSE 22


コマンドの目的をさらに詳しく見てみましょう。



  • FROMは、どのイメージからどのバージョンのコンテナをビルドするかを示します。 この場合、最新バージョンのGitlab Community Editionの公式イメージを使用します。
  • ENVは環境変数を設定します。 ENV REFRESHED_AT 2015.09.27.004-イメージが作成された日付。 コンテナを更新する必要がある場合は、日付を変更し、イメージを再構築して（以下で行います）、展開します。 ENV GITLAB_SHELL_SSH_PORT 8005は、ホストのSSHがポート8005にあり、このポートで作業する必要があることをGitlabに伝えます（そして、これをWebインターフェースで考慮します）。
  • RUNは、コンテナ内でコマンドを実行します。 OSを更新します。
  • EXPOSEは、ホストへの転送に使用可能にする必要があるポートを示します。


Docker-compose.ymlには以下が含まれます。 



 gitlab: build: . ports: - "8004:443" - "8003:80" - "8005:22" volumes: - ./config:/etc/gitlab - ./logs:/var/log/gitlab - ./data:/var/opt/gitlab restart: always


最初の行は、結果のコンテナの名前を示します。 buildは、イメージをビルドするためのDockerfileのあるパスを示します。 ポートセクションで、どのホストポートをどのコンテナポートに転送するかを指定します。 ボリュームセクションで、適切なフォルダー内のコンテナーに転送するホストフォルダーを指定します。 restart：alwaysポリシーは、システムの起動時およびプロセスがその中に入ったときにコンテナが自動的に起動することを意味します。



これはすべて準備済みです。最初にコマンドで画像を収集する必要があります。 



 docker-compose build


次に、コマンドを使用してコンテナをデプロイします。 



 docker-compose up -d


Gitlabの前にバランサーを配置する必要がある場合は、この記事のバランサーで関連するコンテナーを作成する例を参照できます。 Gitlabに安全にログインするには、NGINXバランサー側でトラフィックの暗号化を有効にすることをお勧めします。



ファイルの編集方法がわからない場合は、この記事の詳細について 「テキストファイルを編集する方法」セクションを参照してください 。 Docker Composeの処理に興味がある場合は、ここに詳細があります 。



これで、gitlabにログインできます： 



 http://ip–


Gitlabが正常にインストールされました。







デフォルトのログインデータ：



  • ログイン ：ルート
  • パスワード ：5iveL！fe


最初のログイン時に、ログイン用の新しいパスワードを設定するように求められます。



パスワードを変更してGitlabにログインすると、スタートページが表示されます。







初期設定



gitlabが使用されるドメインのAレコードを、クラウド内のホストの専用IPアドレスに向けます。 強調表示されたアドレスは、「クラウドインフラストラクチャ」セクションのコントロールパネルで表示できます。







メインのgitlabパラメーターは、ホストの〜/ gitlab / config / gitlab.rbファイルで指定する必要があります 。



指示するパラメーターを持つ行がコメント化されている場合（＃はパラメーターの前に設定されます）-＃を削除する必要があります。 これは、設定されているパラメーターに対してのみ行う必要があります。



まず、構成ファイルをバックアップします。 パラメーターが変更されるたびにこれを行うことをお勧めします。 



 cp gitlab.rb gitlab.rb.old




gitlab.rbのパラメーター設定


external_url


パラメーターで、Gitlabを使用できるサーバーに向けられたドメインを指定します。







time_zone


このパラメーターは時間帯を指定します。 



 gitlab_rails['time_zone'] = 'Europe/Moscow'


gitlab.rbのメール設定


このセクションでは、 Yandex Mailに 関連付けられたドメインのgitlab.rbの作業パラメーターを示します。 GmailとMailgunの設定はここにリストされています 。



これらのパラメーターをgitlab.rbファイルに貼り付け、 git.alm @ plugndo.comをドメインのYandexアドレスに置き換えます。 ここでメールパスワードをメールパスワードに置き換えます 。 plugndo.comの代わりに、メールドメインの名前を挿入します。 残りの設定は変更しないでください。 



 gitlab_rails['smtp_enable'] = true gitlab_rails['smtp_address'] = "smtp.yandex.ru" gitlab_rails['smtp_port'] = 587 gitlab_rails['smtp_user_name'] = "git.alm@plugndo.com" gitlab_rails['smtp_password'] = "    " gitlab_rails['smtp_domain'] = "plugndo.com" gitlab_rails['smtp_authentication'] = "login" gitlab_rails['smtp_enable_starttls_auto'] = true #gitlab_rails['smtp_openssl_verify_mode'] = 'peer' gitlab_rails['gitlab_email_from'] = 'git.alm@plugndo.com' gitlab_rails['gitlab_email_reply_to'] = 'git.alm@plugndo.com'


変更を保存した後、gitlabコマンドでコンテナーをリロードします。 



 docker restart CONTAINER_ID


、ここで、 CONTAINER_IDの代わりに、gitlabでコンテナーの一意の番号を指定します。 次のコマンドを使用して表示できます。 



 docker ps


メール設定の正確性を確認するには、Gitlabでユーザーを作成します。すべてが正しく行われた場合、ユーザーはパスワードを設定するためのメールリンクを受け取ります。







おわりに



この記事の手順を適用すると、独自のクラウドサーバーに最新バージョンのGitlabが届きました。 プロジェクト用にシステムの構成を開始できます。 InfoboxCloudの分散型フォールトトレラントストレージシステムのトリプルレプリケーションのおかげで、データの安全性について心配することはできません。
