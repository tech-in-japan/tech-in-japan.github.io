言語の変更



私（および他の多くの人々）がJDK 12で最も注目すべきと考える機能は、新しいswitchステートメント（ JEP 325 ）です。 これは、「プレビュー」の機能として使用される最初の言語変更でもあります。 「プレビュー」のアイデアは、 JEP 12の一部として2018年初頭に導入されました。 これは基本的に、コマンドラインオプションを使用して新機能のベータ版を有効にする方法です。 プレビューを使用すると、ユーザーのフィードバックに基づいて変更を加えることができます。最悪の場合、正しく受信されなかった場合は、関数を完全に削除します。 機能をプレビューするための鍵は、それらがJava SE仕様に含まれていないことです。 新しいスイッチについては、Habré に非常に優れた翻訳があります。

JDK 12では、スイッチは「コンテンツ」を評価して結果を生成する式になりました。 これは、下位互換性に影響を与えないことをすぐに説明します。したがって、スイッチを演算子として使用するコードを変更する必要はありません。







JEPの例を使用します。これは単純で明確なためです。







古いスイッチ 
int numLetters; switch (day) { case MONDAY: case FRIDAY: case SUNDAY: numLetters = 6; break; case TUESDAY: numLetters = 7; break; case THURSDAY: case SATURDAY: numLetters = 8; break; case WEDNESDAY: numLetters = 9; break; default: throw new IllegalStateException("Huh? " + day); }


ご覧のとおり、曜日を変数day



名前にマッピングしてから、値numLetters



割り当てます。 switchは演算子になったので、switchステートメントの結果を使用して割り当てを1回実行できます（エラーコードの可能性を大幅に減らします）。 







 int numLetters = switch (day) { case MONDAY, FRIDAY, SUNDAY -> 6; case TUESDAY -> 7; case THURSDAY, SATURDAY -> 8; case WEDNESDAY -> 9; default -> throw new IllegalStateException("Huh? " + day); };


2つの構文の変更がすぐにわかります。 OpenJDK開発者は、カンマ区切りリストと呼ばれるあまり知られていない構文関数に出くわしました。 また、ラムダ式演算子->



すると、値を簡単に返すことができます。 本当にこれが必要な場合は、値を指定してbreak

JVMの変更



JEP 189：シェナンドー ：一時停止時間の短いガベージコレクター



Shenandoahは2014年にRed Hatによって発表された研究プロジェクトで、JVMでのメモリ管理のための低レイテンシアプリケーション要件に焦点を当てています。 その目標は、20 GBを超えるヒープの最大休止時間1..10ミリ秒（ したがって、小規模なアプリケーション向けではありません -Shenandoahの開発者の一人が答えたように 、これはそうではなく、小さなアプリケーションで素晴らしい仕事をします）。 このコレクターは、アプリケーションスレッドと並行して動作するように設計されているため、ほとんどのガベージコレクターで発生する問題を回避できます。







JEP 344：混合G1コレクション



この変更は、G1コレクターが設定された遅延目標に到達したときのG1コレクターの動作を改善することを目的としています。 G1は、ヒープスペース（古いものと古いものの両方）を領域に分割します。 古い世代では、1回の操作でガベージを収集する必要がないという考え方です。 G1がガベージを収集する必要がある場合、G1は定義する領域を選択します。 これはコレクションキットと呼ばれます。 JDK 12より前のセットで作業が開始されたとき、すべての作業は基本的にアトミック操作として完了しなければなりませんでした。 問題は、アプリケーションのヒープ領域の使用法の変更により、コレクションセットが大きすぎて収集に時間がかかりすぎて、一時停止時間が達成されないことがあったことです。







JDK 12では、G1がこの状況を識別した場合、これがアプリケーションが新しいオブジェクトにスペースを割り当て続ける能力に影響しない場合、データ収集を途中で中断します。 G1の最終的な効果は、短い一時停止時間に達すると改善されます。







JEP 346：G1から未使用の割り当て済みメモリをすばやく返す



これはG1のもう1つのパフォーマンス向上ですが、もう1つは、JVMがシステムの他の部分と対話する方法に関連しています。 明らかに、JVMヒープにはメモリが必要であり、起動時にオペレーティングシステムの仮想メモリアロケータにメモリを要求します。 アプリケーションの起動時に、ヒープに必要なメモリ量が低下し、割り当てられたメモリの一部が他のアプリケーションで使用するためにオペレーティングシステムに返される場合があります。







G1はすでにこれを行っていますが、これを行うことができるのは2つの場所のうちの1つだけです。 最初に、完全なコレクション中に、次に、並列サイクルの1つ中に。 G1は完全な収集を行わないようにし、メモリ使用量が少ないと、収集サイクルの間にかなりの期間が生じる可能性があります。 これは、G1が固定メモリを長期間保持できるという事実につながります。







JDK 12では、G1はアプリケーションがアイドル状態のときに定期的に継続または並列ループの実行を試み、Javaヒープの全体的な使用状況を判断します。 未使用のメモリは、よりタイムリーで予測可能な方法でオペレーティングシステムに返すことができます。







新しいコマンドラインフラグ-XX:G1PeriodicGCInterval



を使用して、チェック間のミリ秒数を設定できます。







この機能により、長時間アイドル状態になっているアプリケーションでJVMメモリをより保守的に使用できます。







その他の新しいJDK機能



JEP 230：マイクロベンチマークキット



Java Microbenchmarking Harness（JMH）は、Alexei ShipilevがOracleで働いていたときに開発され、Javaアプリケーションのパフォーマンステストを開発するための広範なプラットフォームを提供します。 Alexeyは、アプリケーションのパフォーマンスを分析する際に発生する多くの単純なミス（ウォームアップ、例外の回避など）を回避するのに役立つ素晴らしい仕事をしました。







これで、JMHをOpenJDKに含めることができます。 JDK自体の作業とコードの変更に興味がある人は誰でも、これを使用して、変更前後のパフォーマンスを比較したり、異なるリリースのパフォーマンスを比較したりできます。 テストを可能にするために、多くのテストが含まれています。 JMHの設計では、新しいテストを必要な場所に簡単に追加できます。







JEP 340：2つではなく1つのAarch64ポート



OpenJDKにはArm64アーキテクチャ用の2つのポートがあり、1つはOracleによって提供され、もう1つはRed Hatです。 これは不要であり、OracleはJDKバイナリのArmのサポートを停止したため、Red Hatポートのみを使用することが決定されました。これはまだサポートおよび開発されています。







JEP 341：デフォルトのCDSアーカイブ



データ共有（CDS）クラスは、以前はOracle JDKの商用機能でした。 Oracle JDKとOpenJDKの機能の違いをすべて排除するためにJDK 11で最近行われた移行により、OpenJDKに含まれました。







CDSを使用するには、アプリケーションの起動時にロードされるクラス用に作成されたアーカイブが必要です。 64ビットプラットフォーム用のJDK 12では、 lib/server



ディレクトリにclasses.jsa



ファイルがありlib/server



。 これは「デフォルトクラス」のCDSアーカイブです。 JDKモジュールのすべてのパブリッククラスを意味すると思います。 開梱して確認する方法が見つかりませんでした。 CDSはデフォルトで有効になっているため、コマンドラインの-Xshare:auto
