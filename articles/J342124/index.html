Java Enterprise-最初のビジネスロジック！



そのため、ビジネスロジック（ドメインロジック）の実装に特別な注意を払っています。 ロジックの準備が整った場合にのみ、より狭い技術的な問題が発生します。 Java Enterpriseはこのアプローチを好みます。 サブジェクト領域-その本質-は、最初にPure Javaで実装されます。 私たちは企業部門で働いているため、通信やデータ保持などのエンドツーエンドの機能は非常に重要です。 現代のJava EEでは、それらは宣言的に完全に実装され、実質的に最小限のオーバーヘッドです。 基本的に、Java EEプラットフォームでは、「構成規則」の原則が定められており、このプログラミング手法をサポートしています。



Java仕様は、いくつかの注釈とともにビジネスロジック（純粋なJavaクラス）に追加されます。 次に、単体テストを使用してビジネスロジックを簡単に検証します。 ほとんどすべての機能は最新のJava EEのアノテーションで宣言されているため、クラスはテストで直接インスタンス化できます-エンタープライズコンテナなしでも可能です。

Java EEでの通信：JAX-RS、JMS、WebSocket、およびJSON

Java EEおよびゼロ依存アプリケーション：DockerおよびKubernetesとの連携



プログラマが可能な限り生産的に作業する必要があるという事実は、ビルドおよびデプロイメントモデルレベルにも反映されます。 Java EEアプリケーションでは、アプリケーションは伝統的に実装から分離されています。 クラスはAPIのみをインポートします-実装はエンタープライズコンテナーを提供します。 このアプローチにより、アセンブリ、配信、展開の時間が最小限に抑えられます。 連続配信パイプラインのすべての段階で、アプリケーションコンテンツのみがコンパイル、アセンブル、配信されますが、フレームワークの実装はそうではありません。 後者は実行時にすでに発生しています。



これが、Java EEがコンテナーとオーケストレーションの世界で、つまりDockerおよびKubernetesとの組み合わせで最も利益をもたらす理由です。 書き込みのためにファイルシステムでコピーが使用され、個別のキャッシュが個々のコンテナイメージのレベルで動作する場合、インフラストラクチャは最も必要な操作のみを実行します。 各反復で、アプリケーションのコンパイルされたクラスのみが配信されますが、技術的な部分は配信されません。 このアプローチは、WARアーカイブにパッケージ化されたいわゆる「ゼロ依存アプリケーション」によって提供されます。 さらに、アプリケーションには、 provided

通常、エンタープライズプロジェクトでは、アプリケーションサーバーに関連する複雑さ、バージョン、および構成を少なくとも部分的に処理する必要がありました。 サーバーには多くのアプリケーションがあり、展開、構成、および新規インストール操作を複雑にし、場合によっては調整も必要になります。 この問題は、Dockerなどのコンテナーを使用して解決されます。 コンテナーには、ランタイム環境や構成など、アプリケーションの実行と構成に必要なものがすべて含まれているため、テクノロジーの選択は他のアプリケーションに依存しません。 たとえば、Java EE 7からJava EE 8への切り替えなどのアップグレードは、それほど面倒で危険ではありません。 したがって、デプロイされたバージョンでは、スタック全体とメガバイト単位の依存関係を提供する必要はありません。これらはすべて、対応するベースイメージの一部です。



Java EE 8以降



そのため、Java EE 8には新しい機能がありますが、その中で最も注目すべきは、JSON-Bの新しい標準です。 以前は、JSONなどのJSONを表示するためのフレームワークは、サードパーティの依存関係、特に推移的な依存関係とアプリケーションサーバーライブラリ間のバージョンの競合を解決する役割を担っていました。 競合の解決にはリソースと時間がかかりました。 Java EE 8は、これらすべての追加コストなしで実行できます。



JSON-Bに加えて、セキュリティを提供する別の新しい標準であるセキュリティAPIが導入され、CDI、JSON-P、Bean Validation、JPA、Servlet、JSFなどの古いAPIが更新されました。 さらに、サーバー送信イベント、JAX-RSのリアクティブクライアント、新しいJSON-P 1.1機能などの革新に特に注目しています。 Bean Validation 2.0、およびさまざまな標準の統合における多数の改善。 Java EE APIがここ数か月および数年で進化した方向は、開発者の観点から非常に楽観的であり、Java EEプラットフォームのEclipse Foundationへの移行後（Eclipse Enterprise for Java（EE4J）に準拠）、Java EEのさらなる開発はよりオープンになりますモードとおそらくスピードアップ。

最新のJava EEにより、プログラマーはアプリケーションの最も重要な側面に集中し、技術コストを最小化できます。 展開可能なバージョンが可能な限りコンパクトになったため、Java EEはDocker、Kubernetes、OpenShiftなどのコンテナ環境と最適に統合されるようになりました。
