独自のソリューションの開発：リスクと責任

こんにちは、Habr！ この記事では、Mail.Ru Groupでのコード作成のアプローチについて説明します。 既製のソリューションを使用する場合、および自分で書く方が良い場合。 まあ、そして最も重要なこと-あなたの仕事が効果的でなく、他の人に利益をもたらさないようにするために、どのような措置を講じる必要があるか。 これらのニュアンスはすべて、内部JSSDKを作成するタスクの例で考慮されます。これは、2つのプロジェクトのコードベースを組み合わせる必要があるために発生しました。





マイケル・パークスによるイラストレーション



自転車を再発明するのは良くないと聞いていますが、自転車と完成品の境界線はどこにあるのでしょうか？ Backbone 、 Ember、またはAngular はどの段階でそうではなくなりましたか？ これはめったに語られません。 過去4年間、私はさまざまな種類の「自転車」を継続的に開発してきました-私が好きだからではなく（しかし私は本当に好きです）、一部のソリューションは時代遅れであり、他は特定の技術に結び付けられています（たとえば同じjQueryで ）これは必要ありません。それを切り離すことは、最初から書くことに相当します。 しかし、主な問題は狭い専門性と拡大の機会の欠如です。 同じgithubには多くの決定がありますが、誰もが未来を持っているわけではありません。 したがって、あなたが考えているように、素晴らしいことを書いて、タスクを緊急に完了することを決定した場合、時間を無駄にせず、あなたの後にそれをサポートしなければならない他の人に同情します。 99％の確率で、すべてを書き換えます。 それで、いつ自分の自転車を発明することが可能であり、必要なのでしょうか？



タスクから始めて、それを評価します：

  • 可能性（スコープと開発の見通しはありますか、おそらく明日は必要なくなるでしょう）;
  • 汎化（他のタスクやプロジェクトへの適用の可能性）;
  • 疎外（内部インフラからの独立）。


これらの単純なポイントは、フレームワークまたはjQueryプラグインの開発を問わず、ほぼすべてのタスクに適用できます。



私たちの話は3年前に始まりました。タスクは「タッチデバイス用のメールを開発する」ことでした。そのためには、すべてを行う必要がある技術を選択する必要がありました。 次の3つのオプションがありました。



  1. ビッグメールの経験を活用します。
  2. 人気のあるフレームワークを使用します。
  3. 自分で書いてください。


ビッグメールのコードを使用することはできませんでした-17年の歴史が自分を感じさせます。 したがって、自分で書くか、既製のツールを探すかのいずれかでした。 このようなタスクのフレームワークを開発することは、私たちの経験を考慮しても非常に困難です。これには実際には潜在的な可能性はありません。これは手押し車に厳密に関連付けられた高度に専門化されたソリューションになる可能性が非常に高いです。 必要なものを大まかに想像して、タスク（および最も重要なことにはチーム）に適切なソリューションを選択しました。



  • Grunt-プロジェクトをビルドします。
  • RequireJS-モジュールの編成。
  • バックボーン-モデル、ビュー、ルーティング。
  • Festはテンプレートエンジンです。


これらのシンプルなツールにより、プロジェクトを迅速に開発し、その実装を開始できました。 タッチメールが大きな機能に追い付くまで、すべてがうまくいきました。 このため、多くの製品機能は2回作成されました。最初はメールで、次にtouch.mail.ruでしたが、実装の違いは最小限であり、構成に適していました。 この状況は、新しいバックエンドAPIの導入によって悪化しました。これは、答えを「プルして取得」するだけでは不十分でした。







これをすべて検討した結果、もう二度と開発、二重のバグ、二重のテスト、バグに陥ることはできないと判断しました。さらに、メール機能の一部を統合したい内部プロジェクトもあります。



すべてが共通のコードベースの必要性について言及しました。これは別のリポジトリにあり、その中に共通のコンポーネントが実装されます。



プロジェクトに関する知識を要約して、基本的なパッケージセットを決定しました。



  • エミッタ-イベントのエミッタ。
  • 約束-約束;
  • リクエスト-HTTPリクエストをサーバーに送信します。
  • RPC-サーバーAPIを操作するロジックを担当します。
  • モデル-モデルクラス。
  • RPCModel-RPCを介して作業するための高度なモデル。
  • Model.List-モデルのリスト（コレクション）を操作するためのクラス。


そして、それは中小企業です。これらのコンポーネントを構築するものについて：



  • 既製のライブラリ/フレームワークを選択します。
  • 自分で書いてください。


同様の質問に答えるために、私は自分で次の手順を策定しました。



  1. 既製のソリューションのリストをコンパイルする（完全に適切でない場合でも）。
  2. リストを調べます（約1週間後、コード、サポート、githubのタスク（ある場合）などを調べます）。
  3. ソリューションがタスクに適合しない場合、タスクを変更しようとします（マネージャー/デザイナーに移動し、「これは不可能です、すべての愚か者」ではなく、代替手段を提供します）。
  4. あなたにふさわしくないものがあれば、準備はできていますか...（後で詳しく説明します）。




既製のソリューションを見つける



まず最初に、ソリューションの要件を定義することから始めます。 これは、特定のタスクに必要な機能のリスト（その前に既に整理済み）と拡張性が必要です。 過剰なエンジニアリングに関与しないでください。それは良いことにはつながりませんが、混乱させてターゲットから遠ざけるだけです。







そのため、最初にすべきことは、モデルの作成方法に基づいて、モデルの処理方法を決定することでした。 ただし、ソリューションには次の機能が必要であることを考慮する必要があります。



  • ドット表記-ドット表記によるモデルプロパティへのアクセス。たとえば、model.get（ 'foo.bar.baz'）;
  • ゲッター-`get`、model.fooなしのプロパティへのアクセス// {bar：{baz：true}};
  • キャッシング-localStorageまたはIndexedDBからデータを回復する機能。
  • モデルの保持-モデルの整合性。


前述したように、Backboneで構築された多くのプロジェクトと同様に、タッチメールは、Emitter、Model、Collection、Router、およびViewを提供する優れた基盤です。 これですべてのニーズをカバーできます。



すべてがバックボーンのない大きなメールにのみかかっていましたが、同様のインターフェース（get / set）を持っていたモデルにありました。

モデルの完全性とは何ですか？



手紙を受け取る例を考えてみましょう。 

function findOne(id) { var dfd = $.Deferred(); var model = new Backbone.Model({id: id}); model.fetch({ success: dfd.resolve, error: dfd.error }); return dfd.promise(); } // -   #1 findOne(123).then(function (model) { model.on("change:flag", function () { //   console.log(model.get("flag")); }); }); // - #2 findOne(123).then(function (model) { model.set("flag", true); //     });




一見すると、たとえばfindOneメソッドを変更してプロミスを記憶し、それを返すことで問題を修正できます。 

 var _promises = {}; //   //   function findOne(id) { if (_promises[id] === undefined) { var dfd = $.Deferred(); var model = new Backbone.Model({id: id}); model.fetch({ success: dfd.resolve, error: dfd.reject }); _promises[id] = dfd.promise(); } return _promises[id]; }




しかし、IDでモデルを検索することに加えて、モデル（コレクション）のリストもあります。 また、コレクションを取得する場所はどこでも、同じモデルのインスタンスへの参照で構成し、アプリケーションの任意の時点で整合性を維持する必要があります。



もちろん、これはバックボーンの上に巻き付けることができますが、問題はこれだけではありません。 たとえば、コレクションメソッドを実行した後、出力で配列を取得します。 

 //     id var ids = collection .where({ flag: true }) .pluck("id"); // TypeError: undefined is not a function




したがって、Backboneで必要なことを行うには、次のものが必要です。



  • ドット表記 -ネスト/ディープモデルを接続するか、自分で書きます。
  • achaching-わかりやすいものが見つかりませんでした。
  • モデルを永続化 -自分で記述します。
  • また、ロギング、moki、その他のささいなこと


必要な機能を実装する拡張機能を見つけたとしても、このホッジポッドで何かを構築するリスクはありません-これらの拡張機能の間でバグや競合が発生する可能性が非常に高く、重大なパフォーマンス障害が発生します。 このような機能は、フレームワーク自体のコアに統合する必要があります。

リクエスト



すべてが1つに似ており、まったくないソリューションの無限の海を次に示します。



  • イベント（開始、終了、エラー、WiFi認証の喪失など）;
  • タイミング（開始時刻と終了時刻、要求期間）;
  • エラー処理と結果の変更の可能性。
  • エラーの場合などに、要求を再試行します。


最も近い適切なオプションはjQuery.ajaxのみです。



そのため、さまざまな理由で見つかった各ソリューションは、要件に適合しません。 例：



  • エミッタ-handleEventおよび/またはイベントオブジェクトをサポートしません。
  • 約束-jQueryと互換性がありません。
  • リクエストはjQueryに最も近いものです。


もちろん、ソリューションの1つを採用し、機会を減らしてjQueryに関与することができました。 しかし、これらのモジュールはそれほど大きくなく、jQueryの存在は期待できませんでした。

準備はいいですか...

  1. 狭い問題を解決するのではなく、一般的な解決策を書きます。
  2. テストとドキュメントを書きます。
  3. 7/24を維持します。
  4. すべてを無料で行います。


最後の点は誰かには奇妙に思えるかもしれませんが、時間がかかります-これは実際に重要な点です。 実際、あなたのフードの下にあるものをビジネスすることは重要ではありません-それは利益を心配しています（私は現在一般的に言っています）ので、あなたが主張して実装の時間を得ることができたなら、サポートはあなたの費用で提供され、それは正直です-それはですあなたの選択、あなたの決定でした。 多くの人がこの点を過小評価しており、そのために、GitHubが翌日にサポートが停止した決定で詰まっているようです。 愚かではない週に2、3のタスク（そして1日）を準備する必要があります。そして、彼らがあなたに感謝する最大値は、すでに良いことです（そして、それはテストでさえバグになることは別です）。

私たちのために、私は次のスタックを選択しました ：



  1. プロジェクトをビルドするGruntJS 。
  2. JSHintおよび.editconfig-コーディングスタイルまたはタブ対に関するすべての質問と余分なホリバーを削除します。 ロボットと議論することはできません。
  3. QUnit + Istanbul-テストは製品の品​​質を向上させるだけでなく、開発およびリファクタリングプロセスを加速します。 カバレッジは、テストがAPIにある機能をどの程度カバーしているかを確認する機会を提供します。 CIはTravisで 、現在はBambooです。
  4. ES5 + Polyphilsは最も重要なポイントの1つです。 TS 、 CS、またはES6は単なるテクノロジーではありません。 この選択は、ソリューションを別の開発者が使用するかどうかの意思決定に大きく影響します。
  5. git pre-commit-hook（JSHint）+ git pre-push-hook（QUnit + Istanbul）-package.jsonのpreinstallまたはpostinstallでフックをインストールするなど、自動化できるものを自動化します。
  6. JSDoc3-ドキュメントおよびコメントコード、最新のIDEはJSSDKに従ってオートコンプリートを構築できますが、最も重要なことは、コメントまたはパラメーターの説明を読んだ後、別の開発者がコードとそのロジックにすばやく侵入することです。

開発に直接進みます。



JSSDKでは、各モジュールは4つのファイルを含む個別のフォルダーです。 たとえば、モデル：

  • Model.js-モジュールコード。
  • Model.tests.js-テスト;
  • Model.bench.js-パフォーマンステスト（必要な場合）;
  • README.md-ドキュメント（JSDoc3により生成）。


すでに書いたように、自動化できるすべてのものを自動化します。 したがって、モジュールを作成するために、個別の単調なタスクがあります。



したがって、たとえば、RPCModelを継承するmail.Folderモデルの作成は次のようになります。 

 > grunt model:create:mail/Folder:RPCModel   «mail/Folder»..OK  «Folder»  boot.js .. OK JSSDK/mail/Folder/Folder.js .. OK JSSDK/mail/Folder/Folder.test.js .. OK JSSDK/mail/Folder/Folder.bench.js .. OK JSSDK/mail/Folder/README.md .. OK




開発中に、テストが最初に記述され、次にコードが記述されます。 変更を行うか、新しいモジュールを作成した後、楽しい部分が始まります-コミットとプッシュ：







git commit -am"..."



- grunt jshint

タスクが失敗すると、コミットまたはプッシュが失敗します。これにより、マスター内のコードを常に機能させることができます。 動作しないコードは、マスター以外のブランチでのみコミットできます。 他のブランチでは、エラーが表示されるだけです。 また、プッシュは弱いテストカバレッジから抜け出せない場合があります。 100％未満の弱点（現在1,635件のアサーション）をすべて考慮します。
