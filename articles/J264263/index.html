ダミーではなく、BeagleBoneまたはrunitの起動を高速化する





当社が設計および製造するデバイスは、コールドスタート後にクイックスタートが必要です。 本格的なオペレーティングシステムを搭載していないデバイス（NutOSを使用しています。EtherNut）では、このような問題はありません。電源を入れてから数秒で動作するようになります。 しかし、Linuxが内部にある、特にポータブルな測定システムでは、より複雑で高度なものでは、初期化アルゴリズムを高速化するという問題は重要です。

10Gイーサネットスイッチのパイロットバージョンでは、よく知られているBeagleboneボードを使用し、qemuエミュレーターを除くブートプロセスを喜んでデバッグしました。 ちなみに、ビーグルボーン制御ボードを備えた10ギガビットスイッチのこのパイロットバージョン（記事の写真）はサーバールームにインストールされており、数年間正常に機能しています。

runitへの移行により、500MHzアームプロセッサでのシステムの起動が1分半で6秒に加速されたとすぐに言わなければなりません 。



免責事項：このメモは当社の内部ウィキ用に書かれたものであり、すべてのソフトウェア開発者がシステム管理者ではないため、いくつかの点を最も簡単で最も理解しやすい言語で説明する必要があると考えました。





はじめに：現状のままで何を変更できるか



私は問題を簡単に説明しようとします。

通常、Linuxシステムは次のように起動します。

  1. ブートローダー（lilo、grub、u-boot、...）
  2. コア
  3. 初期化プログラム
  4. 他のすべて（/ etc / inittabで説明）は、このinitプログラムによって開始されます。




ランレベルについては説明しません。 これにはmanがあります：man runlevel、man init、man inittab。



initはシステムのメインプロセスであり、何らかの方法で他のすべてのプログラムを管理します。プロセス番号も1です。非常に単純化されている場合、タスクは/ etc / inittabファイルにリストされているプログラムを開始および再起動します。



しかし、ブートプロセスに戻ります。



いわゆるSystem V（sysv）ブート順序に準拠しているディストリビューションの作成者は、何らかの理由で、initがランレベルレベルを決定するパラメーターを使用して同じrcスクリプトを呼び出すことを決定しました。 これは、たとえば、すべての子供が1つの教科書で学校に行き、勉強しているクラスに応じて、異なるページでのみ開いた場合と同じように考えることができます。 このような教科書がどれほど厚いはずか想像できます！



Linuxシステムの起動シーケンスはほぼ同じです。



たとえば、/ etc / inittabの一部です：

スクリプト/etc/init.d/rcには、シェルの300行（！）行以上のコードが含まれています。 しかし、一般的には何もしません。実行ステージ（ランレベル）に応じて/etc/rc?.dからプログラムを順番に起動します。



さらに進んでいます。 初期段階（ランレベルS、上記または/ etc / inittabを参照）では、/ etc / rcS.d /ディレクトリーにあるすべてのものが実行されます。 ホスト名からnfsのマウントおよびランダムジェネレーターの初期化までの20個のスクリプトです。



ブートプロセスのこの部分を完了すると、initはマルチユーザーモードに入り、/ etc / rc2.d /からスクリプトを実行します。 最も単純な場合、約10個あります。



新しいプログラム（サービス、デーモンなど）を追加する場合、インストールパッケージは通常1つまたは2つのスクリプトをブートプロセスに追加し、システムの起動時間が再び長くなります。



デスクトップコンピューターやラップトップ、またはサーバーでさえもダウンロードが急ではない場合、できる限り早く電源を入れた後、私たちが生産するデバイスは動作する準備ができているはずです。 Bercut-ETなどのデバイスの場合、問題はありません。そこでプロセスを完全に制御します。Bercut-MMTの場合、クイックスタートの問題は最近まで開いていました。



そのため、一連のrunitプログラムにより、システムの起動手順を最も簡単かつ完全に制御できます。



実際、runitを使用しているときにLinux（Linuxだけでなく、一般にbsdのように読み込むこともできます）を起動すると、カーネルがrunit-initプログラムを実行し、3つのシェルスクリプトの動作を制御します。

  1. / etc / runit / 1-初期起動
  2. / etc / runit / 2-マルチユーザーモード（ランレベル2と同様）
  3. / etc / runit / 3-シャットダウン/リブート（電源オフ、シャットダウン、停止）




通常、各スクリプトはコマンドの1つまたは2つの画面を占有し、必要なことだけを行います。 そのシンプルさ、透明性、そして最も重要なこととして、少量のスクリプトにより、システムのインストールイメージの管理者または開発者は、プロセスを簡単かつ完全に制御できます。



他のプログラムおよびサービスは通常、 runsvdirプログラムの下で実行を開始し、依存関係、優先順位、および起動順序は管理者によって決定されます。 Runsvdirの使用例はすでに Habréにあるため、ここでは詳しく説明しません。



Bercut.BeagleBoneダウンロードログでは、システムが初期スクリプト（/ etc / runit / 1）の実行に費やした時間と、「login：」プロンプトが表示されるまでの時間に注意を払うことができます。
