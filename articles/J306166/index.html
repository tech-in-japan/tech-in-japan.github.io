URLの秘cks



Githubの哲学はシンプルであると同時に、非常に柔軟なツールになることです。 このため、多くの機能はユーザーインターフェイスから使用できませんが、URLパラメーターを介してアクセスできます。





公開鍵アクセス



githubアカウントを持っている同僚のsshアクセスを構成する場合、最も簡単な方法は、 https： //github.com/ <user_name> .keys（たとえば、 https：/ /github.com/defunkt.keys ）。 この機能を使用してCIプロセスを設定している場合、常に最新の公開キーを使用できます。





差分とパッチへのアクセス



コミットまたはプルリクエストURLページの最後に.diff



または.patch



を追加すると、この出力をdiffまたはパッチUNIXユーティリティの形式で取得できます（たとえば、 https：//github.com/tars/の形式のURLを使用します） tars / commit / 07902a956da92e6a616a69d3b3f0f9276f0c13fe.diff-末尾の「.diff」に注意してください）。 このように、キャッシュデスクを離れずに一連の変更を取得してSlackまたはメールで送信すると便利な場合があります。





リポジトリ内のファイルへのURLの共有



ここでは、一見、すべてが簡単です。ブラウザーのアドレスバーからファイルへのリンクをコピーし、必要に応じて貼り付けました。 しかし、プロジェクトの動的な性質により、コミットは常にウィザードに追加され、ファイルの内容は将来変更されるか、ファイルが完全に削除される可能性があるため、ブラウザのアドレスバーの通常のコピーはここでは適切ではありません。 ファイルの現在のバージョンへの永続的なリンク（ 永続的なリンク -Githubが呼び出すように）を取得するには、URLのブランチ名の代わりにコミットハッシュを使用する必要があります。 はい、これはかなり不便なので、githubのメンバーがホットキーを作成しました。 ファイルを表示しているときに「y」を押します-ブラウザのURLはパーマネントリンクに変更されます。





差分を表示するときに空白文字を除外する



ファイルの違いを表示するときにURLに?w=1



を追加し?w=1



違いがスペースのみであった場合、これは気を散らしません。









特定のコードブロックを強調表示する



ファイルを表示するときに、その行のいずれかをクリックして、指定された場所へのリンクを送信できます。 一連の行を選択するには、次のように、範囲の最後の行番号をマイナス記号から行番号の後にURLに追加します（URLの最後でL30-L32に注意してください ）： https : //github.com/torvalds/linux/blob/9256d5a308c95a50c6e85d682492ae1f86a70f9b /include/uapi/linux/netfilter_bridge/ebt_among.h#L30-L32





リポジトリ内のブランチリビジョンの比較



プルリクエストを作成するとき、マージの結果としてどの変更がターゲットブランチに分類されるかをすぐに確認できます。 ただし、ブランチ比較ページには明らかな機能があります 。たとえば、2週間など、特定の期間のブランチのすべての変更を確認できます-https: //github.com/github/linguist/compare/master@%7B2week%7D...master 。 ブラウザーに比較ページをブックマークしておくと便利な場合があります。何か問題が発生した場合は、たとえば過去24時間にわたってコードの変更点をすばやく確認します（「master @ {1day} ... master）。 標準のgit diffはこの機能の内部で使用されるため、gitが受け入れる任意の時間形式を使用できます。 ブランチ、コミットなどの比較の詳細については、ドキュメントを参照してください。

チケットとプルリクエスト



問題とプルリクエストは、プロジェクト管理の絶え間ない支援者です。 Githubには、それらと連携するための多くのツールが用意されています。 ラベルシステムを使用して、各チケットまたはプルリクエストに対応する属性（機能、バグ、ドキュメントなど）を付加できます。 また、マイルストーンを作成して、どのバージョンのどのチケットで作業が実行されるかを示すこともできます。 そして、楽しくて役に立つトリックがいくつかあります。

コマンドラインとGithub



ハブ



ハブは、githubの作成者によるコンソールユーティリティ（git + hub = github）であり、その目的は、シェルからのサービスの使用を容易にすることです。 基本的に、ハブは標準のgitをラップし、リポジトリ、プルリクエスト、チケットを操作するための追加コマンドを提供します。



これは、リポジトリを複製する方法です。 

 $ hub clone github/hub


フォークを作ります： 

 $ hub fork


プルリクエストを開く： 

 $ hub pull-request


ユーティリティの作成者は、ハブコマンドは標準のgitコマンドと競合しないため、インストール後にgitを呼び出すときにハブでエイリアスを作成することをお勧めします（alias git = hub）。



ユーティリティの使用に関する詳細は、manまたはプロジェクトのWebサイトで見つけることができます。





プルリクエストは既にリポジトリにあります。



少数の人が知っている興味深い点。 プルリクエストは概して特別なブランチであるため、プルリクエストを作成すると、それに含まれるすべての変更が自動的にリポジトリに分類されます。 この機能により、元のフォークが削除された場合でも、githubはプルリクエストの変更を表示できます。 したがって、常にそれらにアクセスできます。 プルリクエストからの変更を新しいブランチ（new_branch）に分類するには、次のコマンドを実行する必要があります。 

 $ git fetch origin pull/<pr_num>/head:new_branch


ここで、<pr_num>はURLからのプルリクエスト番号です。
