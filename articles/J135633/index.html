ジュレベル

Habréでは、 JRebelが単に言及されているか、新しいバージョンがリリースされたという情報が掲載されているいくつかの記事が公開されました 。 同時に、すべての読者がそれが何であり、このソフトウェアがどのように機能するかを理解しているわけではありません。



この製品の開発への直接の参加者として、JRebelが存在する理由と、それがJava開発者にどのように役立つかを明確にしたいと思います。



足はどこから成長しますか？





最初の問題は、Javaを使用するほとんどすべての開発者に知られています。プロジェクトを変更した後、結果を確認するには、コンテナでビルドおよびデプロイするのに多くの時間がかかります。 開発プロセスを加速または自動化する方法に関するHabréの優れた記事がすでに公開されているので、ここでは繰り返しません。 しかし、実際には、前述のメソッドには独自の欠点があります。すべての変更がデプロイされたアプリケーションに通常の方法でオーバーロードされることはありません。 コンテナを再起動する必要があるメモリリークを取得するのは非常に簡単です。 技術的な詳細は、当社のウェブサイトの一連の記事によく描かれています-好奇心the盛な読者にぜひお読みください。



時間はどこに行きますか？




Webアプリケーション開発サイクルは、古典的な形式ではどのように見えますか：

1.コード（またはリソース）を変更しました

2.組み立てられたJAR / WAR / EAR

3.受信したアーカイブをコンテナに展開しました

4.デプロイされたアプリケーションを開き、いくつかの操作の後、それらの作業の結果を確認しました。



アプリケーションのサイズ、使用されるコンテナ、およびその他のいくつかの要因に応じて、手順2、3、および4が数秒から完全に狂った数になることがあります。 当社は、使用されている技術とアプリケーションの展開に要する時間に関する開発者への調査を実施しました。 結局のところ、平均的な展開には一度に約3分、1時間に約10分かかります。 展開に30分以上かかる嘆かわしいケースでは、このプロセスを1時間に何回繰り返すことができるかを尋ねるのも意味がありません。 答えは明らかです。



コンテナ/アプリケーションの再起動に数秒かかる場合、上記の問題はそれほど感じられません。 ただし、プロジェクトが成長し、より複雑になると、不便が感じられます。 これはあなたが考えることができる場所です：多分JRebelはあなたが必要なものですか？



JRebelが助けに！





したがって、JRebelは、開発中にアプリケーションを再デプロイする問題を排除するように設計されたツールです。 多くの時間を節約できます。



JRebelを使用する場合の利点は次のとおりです。

1. JRebelはワークスペースからリソースを直接ロードできるため、完全なアプリケーションアーカイブ（JAR / WAR / EAR）をコンパイルする必要はありません。 変更されたコードをコンパイルするだけで、アーカイブの完全なアセンブリよりもはるかに短い時間で済みます。

2.アプリケーションの再デプロイはありません-ここでも時間を節約できます。

3.新しいクラスローダーは作成されないため、更新中にメモリリークのリスクが少なくなります。 したがって、コンテナ自体を強制的に再起動する時間を節約できます。

4.オブジェクトおよびユーザーセッションの状態が保存されます。 したがって、理想的には、同じページにとどまり、変更の結果を確認できます-F5キーを押すだけです。



インストールとセットアップ




いくつかのインストールオプションがあります。 なぜなら ほとんどの開発者はまだEclipse、NetBeans、IntelliJなどのIDEを使用して作業しているため、インストールする自然な方法は、単一のIDEにプラグインをインストールすることです。



Eclipseユーザーは、インストールのために、Eclipse Marketplaceサービスを使用できます。 NetBeansおよびIntelliJIDEAユーザーは、プラグインのリストでJRebelを見つけることができます。 詳細な手順については、 こちらをご覧ください 。 プラグインをインストールした後、彼らは登録してライセンスを取得するのがいいとあなたに知らせます







登録後、 JRebelを食べる準備ができました。



実際、JRebelは特定の開発環境に関連付けられていません。 IDEではなく、アプリケーションが実行されている場所で動作します-つまり 次のような-javaagent引数を使用してJVMプロセスに関連付けられます。



java -javaagent：/opt/jrebel/jrebel.jar -cp my.awesome.Application



構成




ほとんどの場合、JRebelは追加の構成を必要とせず、IDEプラグインを使用して自動的に生成できる構成ファイル-rebel.xmlのみを必要とします。







一番下の行は、JRebelであるエージェント（javaagent）が、コンパイルされたクラスの場所と静的ファイル（html、cssなど）を知っている必要があるということです。 これにより、必要なすべてのリソースを、デプロイされたアーカイブからではなく、プログラマーが変更を加えるプロジェクトから直接ダウンロードできます。



すべてのプロジェクトが「標準」ディレクトリ構造に従い、通常のIDEツールのみを使用してコンパイルされた場合、おそらく構成ファイルは必要ないでしょう。IDEからすべてのプロジェクト情報を取得するか、慣例に従って動作するだけです。 しかし、すべての開発者が標準について独自のアイデアを持っているため、JRebelにはいくつかのヒントが必要です。



構成ファイル自体の構造はかなり単純です。クラスパスと静的リソースの場所を設定するだけです。

ご覧のとおり、このようなファイルを使用すると、最も非標準的なプロジェクト構造を表示できます。私の経験では、このような要件は頻繁に発生します。 rebel.xmlのオプションの詳細な説明は、 ドキュメントに記載されています 。



重要！ JRebelが構成を正しく読み取るためには、rebel.xmlをデプロイ済みアプリケーションに適用する必要があります。Webアプリケーションの場合、rebel.xmlはWEB-INF / classesにある必要があります。 JARファイルの場合、rebel.xmlはアーカイブのルートにある必要があります。







理想的なケースでは、web.xmlとrebel.xmlのみがアーカイブに含まれ、他のリソースはrebel.xmlのパスを介して表示できます。



Mavenプロジェクトの場合、構成ファイルを生成して適切な場所に保存できるプラグインを使用することができます。



打ち上げ




IDEからの起動は非常に簡単です。 IntelliJIDEAの場合、プラグインのインストール後に表示される新しいスタートボタンを使用するだけです。







Eclipseでは、新しいブックマークが起動構成に追加され、JRebelがこの起動と他のいくつかのオプションで有効になるかどうかを確認できます。







開発環境外でコンテナを実行する場合は、選択したコンテナのスクリプトの-javaagentパラメータでjrebel.jarへの正しいパスを追加するだけです。 構成ガイドには、さまざまなJVMおよびOSで実行するサンプルスクリプトを含むコンテナのリストが含まれています。



特徴





JRebelは、便利な自動化ツールであるデプロイ済みアプリケーションでプロジェクトを表示することに加えて、さらに多くの機能を提供します。 まず第一に、これは主な機能です-コードの変更をオーバーロードします。 作業の過程で、かゆみをリファクタリングするような方法につまずいたとしましょう。 言われるとすぐに、extractメソッドを呼び出します。これにより、クラスに新しいメソッドが追加されます。 コードをオーバーロードするための標準ツールHotSwapは、このような変更を処理できません。 次に、JRebelはクラスをオーバーロードし、それについてコンソールに書き込みます。 オーバーロードは、変更されたクラスが使用される瞬間に行われ、それにより、Javaに固有の遅延動作をシミュレートします。



さらに、フレームワーク内の設定に注釈を使用するとします。 たとえば、 @RequestMapping



の値@RequestMapping



、コントローラーが使用できるパスを決定するために使用されます。 アノテーションの値を変更することにより、アプリケーションで新しい方法でリソースを見つけることが期待されます。

次のように機能します。コンパイル後、アノテーション値はクラスの新しいバージョンで使用可能になります。 注釈は実行可能コードではなく、フレームワーク自体がその動作を変更するメタデータに応じたメタデータです。 クラスの新しいバージョンがロードされるようになったので、メタデータが変更され、動作を更新する価値があることをフレームワークに知らせる必要があります。 そのような場合、JRebelには各フレームワークまたはコンテナーの特別な統合が必要です。



注釈と同様に、フレームワークが外部構成に依存している場合、特別な統合が必要です。 たとえば、Spring Frameworkの場合、構成は注釈とXMLファイルの両方で指定できます。

たとえば、XML構成に新しいコンポーネント（ bean ）を追加し、 @Autowired



アノテーションを使用して、新しく作成したコンポーネントをコントローラーに転送します。



制限事項




HabréおよびJRebelで説明されているすべてのアプローチと同様に、いくつかの技術的な制限があります。



この記事の執筆時点では、クラス階層の変更はサポートされていません。 プログラムで1つのクラスがすでに「A extends B」として記述されている場合、「A extends C」に変更することはできません。 インターフェイスのリストの変更にも同じことが当てはまります。クラス宣言からインターフェイスを追加または削除することはできません。



考慮すべき点がいくつかあります。



静的初期化。 JRebelは、すでにメモリに作成されているオブジェクトの状態を保持しようとします。 したがって、コンストラクターまたは静的初期化ブロックの再起動はありません。 これからいくつかの結果が生じます。



静的フィールドの値を変更することにより、クラスの新しいバージョンでこの新しい値が表示されることが期待されます。 実際、この値は、JRebelが再起動しなかった静的ブロックで割り当てられます。 したがって、新しい意味は表示されません。 現在、JRebelは、新しい静的フィールドが追加された場合にのみ静的ブロックを再起動します。



この動作の理由は、静的ブロックで発生することがオブジェクトの状態に無期限に影響を与える可能性があるため、JRebelは最も極端な場合にのみオブジェクトを再起動しようとするためです。



JRebelがコンストラクターを再起動しないという事実から生じる2番目の結果は、新しいフィールドがクラスに追加されると、このタイプのデフォルト値が割り当てられることです。 つまり タイプがプリミティブではないフィールドを追加すると、割り当てられた値はnullになります。このフィールドが既存のオブジェクトに対して逆参照されると、一般にNullPointerExceptionが発生する可能性があります。



JRebel SDK





前述のように、JRebelのフレームワーク構成をサポートするには特別な統合が必要です。 すでにあらゆる種類の統合を実装するためにかなりの労力と時間を費やしており、サポートされているフレームワークのリストは非常に印象的です。 一部のフレームワークを使用する場合、統合は不要であり、JRebelはすぐに使用できます。 例に行く必要はありません。Vaadinの特別な統合がなければ 、JRebelは非常にうまく機能します。フィンランドの同僚は、JRebelを独自のフレームワークで使用できることを非常に喜んでいます。

残念ながら、そのような例は多くありません。 ほとんどのフレームワークは外部構成に依存しているため、追加の統合が必要です。 私たちはすべての人とすべてのサポートを喜んで実現しますが、すべての未知のフレームワークに追いつくことはできず、多くの企業が独自の「バイク」を作成します。 そのような場合、JRebelには独自の統合を作成する機能があります。 弊社のサイトには、ニーズに合わせてJRebelサポートを実装する方法に関する小さなガイドがあります。
