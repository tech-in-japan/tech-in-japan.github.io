Android Oreo：開発者は何を期待すべきですか？

Android Studio 3.0に革新的な機能はありますか？ Java 9の主な欠点は何ですか？また、Kotlinの利点は何ですか？ これについて、およびプロファイラー、コンパイラー、IoTなどについては、ロシアからGDEステータスを取得した最初の開発者であるDenis Neklyudovとのインタビューで取り上げました。









Denis Neklyudov nekdenisは、Google Developer Expertのステータスを取得した最初のロシアの開発者です。 彼は、ソーシャルネットワーク、金融機関、運輸会社、広告ネットワークおよび代理店向けのアプリケーションの開発で豊富な経験を持っています。 彼は多数のプロジェクトに参加し、アーキテクチャ、UI、UXに取り組んでいます。 彼は、さまざまな会議で講演し、Habréを含むコースやブログを実施しています。

Android Oreoのさまざまな開発機能について彼に尋ねました。



-一般に、Android Studio 3.0の最近のリリースがAndroid開発者の世界における一種の小さな革命であることは秘密ではありません。 多くの場合、Java 8機能のサブセットをサポートすることを意味します（ただし、完全ではありませんが、将来のリリースで修正される予定です）。 Java 8のどのサブセットがすぐに使用できるようになり、APIレベルに依存関係がありますか？ この理由は何ですか？ 少なくとも、ドキュメントには、Streamのインターフェイスまたは機能的なインターフェイスがAPIレベル24以上のAndroidで利用可能であることが記載されており、これについて経験豊富な開発者の意見に興味があります。



Denis Neklyudov：実際、バージョンが3番目に上げられたのは理由がないわけではありません。 メジャースタジオリリースは、何か新しいことを試す大きな理由です。 私は、Android開発からKotlinへの移行を熱心に支持しています。まず、新しいAndroid StudioのJetBrainsからの言語サポートの改善に注意することをお勧めします。 しかし、ラムダ、デフォルトメソッドの参照メソッド、およびその他のJava 8バンの「脱糖化」は試みませんでした。 機能の詳細リストはサイトに表示されます 。 しかし、実際には、ストリームは利用できません。 なぜ彼らがそれらを移植しなかったのは良い質問です。 ただし、これでサードパーティのポート、たとえばthisまたはthisを使用できますが、自分の危険とリスクがあります。



-Kotlin言語に対するGoogleの公式サポートで結果が得られましたか？ あなたの意見は何が変わりましたか？



Denis Neklyudov： JetBrainsとのGoogleの相互作用が変更され、Kotlinサポートにより密接に連携できるようになりました。 特に、将来の言語とそのサポートの問題に対処するコトリン財団の設立に関する作業が始まりました。 IDEAまたはASでプラグインを使用して住んでいた人にとって、新機能に関してはほとんど変更されていません。 しかし、Kotlinへの切り替えや試用を恐れていた開発者にとって、公式サポートは、これが単なる実験や多くのオタクではなく、公式にサポートされている言語であるという事実の象徴になりました。 そして、はい、今はヨーダ記法について覚えておく必要があります（悲しいことに微笑む）。



-Kotlin-新しい言語と新しいテクノロジー。 企業からの最も一般的な質問の1つは、問題が発見された場合の対処方法です。 開発者は、他の組織からのプルリクエストをどの程度自由に受け入れますか？ コミットメントが見られるまで10年待たなければなりませんか？ これは、ライブラリ、コンパイラなどの全体的な品質の観点から、要するに、新しい開発者の参入しきい値でさえ可能ですか？



Denis Neklyudov： KotlinはJava、特にC ++に関しては若い男ですが、すでに7歳であり、ほぼ2年間リリースされています。 したがって、本番環境での本格的な実装に十分安定しています。 言語サポートは非​​常に優れているため、PRを行う必要はありません。バグトラッカーで問題を開始するだけで、JetBrainsのスタッフが次のビルドで理由のあるバグを修正します。



言語に不慣れな開発者にとって、エントリーのしきい値は非常に低いです。 この言語は、Javaと、Swiftを含む多くの現代言語の両方に似ています。 最初の知り合いには、常に公式のトレーニング資料をお勧めします -迅速かつわかりやすくあなたを知ることができます。



-Java 9が最近リリースされましたが、これはAndroidの開発にどの程度正確に影響を与えますか？ ライブラリが9に更新され、再び使用することができなくなる可能性がありますか？



Denis Neklyudov： Kotlinへの移行は、JavaからのAndroidの（開発の面での）遅れの問題の解決策の1つだと思います。 Java 8コードさえ実行できないデバイスの大部分。 また、Kotlinのおかげで、開発者は多くの最新の言語ガジェットを使用してJava 7バイトコードにコンパイルできます。Androidライブラリの世界は常にJava 7に住んでおり、Java 8に移植する人はいません。 Project Reactorに代わるJava 8の代替品。Androidでは誰も使用していません。現在、両方のプロジェクト（KotlinおよびRxJava2）が積極的にサポートおよび開発されている理由も含まれています。



-言語の数とバージョンが増えたため、プラットフォームの断片化についてどう思いますか。 古いバージョンをサポートし、デバイス生成ごとにセグメント化するための最も合理的な戦略は何ですか？



Denis Neklyudov：私にとっては、長年にわたってほとんど変わっていません。 幸いなことに、SDKの多くの新機能はサポートライブラリにバックポートされており、残りはアプリケーションが現在実行されているAndroidのバージョンを確認することでサポートできます。 現時点では、Android 4.1の最小バージョンを維持することが重要です。 そして、Android 5以降のプラットフォーム機能に焦点を当てます。多額のお金があれば、Facebookのように2つのバージョンのアプリケーションを用意します。新しいプラットフォームのすべての機能と、最小限のセットで、シンプルでトラフィックの少ないバッテリー機能が、1.5ルーブルの中国のプラスチックのガラガラに取り組んでいます。



-Android Profiler-特に気に入っている点、注目すべき利点は何ですか？



Denis Neklyudov：アプリケーションのネットワークアクティビティを本当に見たいと思っています。Stethoを忘れていました 。 また、メモリの割り当てを確認するための優れたツールでもあります：誰がどの程度実装されているか。 CPUの動作を分析するにはsystraceほど優れたものはありませんが、プロファイラーを使用してStudioを終了することなく、すぐに確認できます。



-Android Profilerとは何ですか？ これはAndroid Studioに関連したものですか、それとも特定のサブシステムのAPIを呼び出しているだけですか？ 何らかの種類のバックエンドがある場合、自動テストなどに使用できます。 現在、モバイルプラットフォーム用の開発ツールチェーンの構築には非常に大きな問題がありますが、それを正しく行う方法を理解している人はほとんどいません。



Denis Neklyudov：プロファイラーはAndroid Studioの一部であり、UIなしで使用するためのヘッドレスAPIはありません。 はい、ほとんど必要ありません。 メモリリークを検索し、プロセッサの負荷を分析し、ヒューマンネットワークアクティビティを表示するために作成されました。 これはCIで自動化する必要があるプロセスの一部だとは思いません。 夢を見るなら、リークに関するレポートや最適でないアルゴリズムをビルドサーバーから受け取るのはいいことですが、今ではそれは遠い未来のユートピアです。



-デスクトップJavaの開発者には、プロファイラーの作成とデバッグという特別な規律があります。 Android用に10種類のプロファイラーも必要ですか？ Android Profilerを使用する価値があるのはどのような状況ですか？ 彼はいつも信じる価値がありますか？



Denis Neklyudov：正直に言うと、モバイルの世界は急速に変化しているため、すべてのビジネスがアプリケーションの綿密なプロファイリングに携わるエンジニアに余裕があるわけではありません。 プロファイラーの電力がトレースを調査するのに十分でない場合、systraceの電力は確かに十分なはずです。 このテーマに関する優れた報告書があります。



-インスタントアプリのサポート-このテクノロジーの主な利点は何ですか？ 開発者とエンドユーザーにとってのメリットは何ですか？ このテクノロジーを使用する場合、機能に制限はありますか？



Denis Neklyudov： Instant Appsが安定し、どこでも動作し、開発ツールとビルドシステムの不注意なサポートを受けてすぐに、新しい時代が到来します。インストールのないアプリケーションです。 Webページのように、必要なものだけを画面ごとにアプリケーション画面にロードすることを想像してください。



もちろん、スタンドアロンアプリケーションはどこにも行きません。 インスタントメッセンジャー、ソーシャルネットワーク、金融アプリケーション、ゲームをページ上で開く人はいませんが、ここでは、たとえば、カードからUberの一部を開き、ポスター、ショッピングセンターカード、またはQRまたはリンクを使用して遊園地からチケットの支払いをする方が、インストールするよりもはるかに便利です1日のアプリケーション、さらには4Gの配布パッケージ全体をダウンロードすることもできます。



開発者にとって、Instant Appsはアプリケーションのアーキテクチャと構造をクリーンアップする機会です。 独立したモジュールまたは疎結合モジュールに分割します。 しかし、今では多くのツールがサポートされていません。 たとえば、ルートモジュールにないDataBindingレイアウトを使用することはできません。 また、インスタントアプリのサイズを監視する必要があります。apkは4 MBを超えてはなりません。



機能の面でも制限があり、インスタントアプリはインストールを必要としないため、システム、その状態、およびユーザーデータへのアクセスが少なくなります。 しかし、すべての制限は非常に論理的です。



-Android Things-「通常の」Androidと何か違いはありますか？ このシステムを消費者に提供する上でのGoogleの究極の目標は何ですか？ この製品はあなたにとってどのような意味がありますか？ 同じRaspberry PiのLinuxとの機能上の違いは何ですか？



Denis Neklyudov： Android Thingsは、コンテンツプロバイダーやカスタムキーボードなど、モバイルデバイスでのみ必要な要素がないという点で、Androidのモバイルバージョンとは異なります。 代わりに、入力/出力デバイスを操作するためのAPIを追加しました。



もちろん、ATは、何らかのプログラムを備えたマイクロコントローラーだけでなく、モノのインターネット専用に作成されました。 デバイスのパワーと幅広い既製ツールにより、ネットワークに接続されたデバイスの可能性を最大限に活用できます。 鮮明な例は、センサーとコントローラーの中央ハブとしてのATデバイスの使用です。ATでTensorFlowを使用することもできるため、ネットワークからの応答を待たずに自分で意思決定を行うことができます。



たとえば、Raspberry Pi 3でRaspbianを使用する場合、Firebase Cloud Messages、Firestore、Fuse Location Providerなど、さまざまなGoogleサービスを使用することはできません。 また、Raspberry Piは家庭用のデバイスであり、工業用ではありません。 次に、Android Thingsは、NXP pico i.mx7dなどの商用SoMで動作します。



Android Thingsについては、私のポッドキャストで、 Googleのチーム代表であるDave Smithとの全号インタビューを行っています 。 個人的には、趣味とお気に入りのライブラリ、使い慣れた言語、SDKを組み合わせる機会は、宿題を自動化して夕方に得られる大きな喜びです。 たとえば、 ポーランドのGoogle Developer Daysで、中央ユニットがATで働いていたエアコンの制御を公開しました。

-コンパイラに関する質問があります。 明らかに、ジャックツールチェーンはついに忘却に陥り、その代わりに新しいDexコンパイラが登場します（Android Studio 3.0ではまだプレビューのままですが、3.1ではデフォルトですでに使用されています）。 彼はジャックよりもどのように優れていますか、これは最終的に開発にどのように影響しますか？ どのような最適化が使用されていますか？



Denis Neklyudov： D8と呼ばれる新しいDexコンパイラ。これは、Dexファイルのコンパクト化によりアプリケーションのサイズをわずかに縮小し、Jackはあまり接続されていません。 しかし、Googleは不完全なコンパイラを放棄しました、それは事実です。 新しいDexは、最後までデバッグされていないため、予期しない場所でバグを整理できない限り、実際には開発に影響しません。



-GoogleのスタッフがJava 8の機能のネイティブサポートをどのように達成したと思いますか？ Androidランタイムにはどのような変更が加えられましたか？



Denis Neklyudov：まあ、これはネイティブサポートではなく、言語「砂糖」のバイトコードへの変換です。 これらはデサカロシスを生成し、Java 8で作成されるものを模倣するバイトコードを形成します。しかし、もちろん、すべてをバックポートすることは不可能ですが、これはレトロラムダを放棄する大きな理由です。



-Java 8の機能のサポートを考えると、これにより開発にScala> = 2.12を使用できますか？ 一般に、ロジックは次のとおりです。新しいツールチェーンはjavac-> desugar-> dxを使用します。 Javaは、開発会社によって公式にサポートされている言語です。 ただし、desugar＆dxはjvmバイトコードに依存しているため、それを生成する他の言語を接続できるようにする必要があります。 理論的には、Scalaはjavacとは異なるコードを生成するため、jvm-bytecodeからdalvikバイトコードへの変換に問題がある可能性があります。 たとえば、バイトコードinvokedynamic命令に問題がある可能性があります。 それで、desugarは将来的にすべての有効なJavaバイトコードをサポートするのでしょうか？



Denis Neklyudov： Invokedynamicは古いSDKにはないため、すべてのJava 8チップのポートを作成することは不可能でした。 ScalaとAndroidはめったに交差せず、Kotlinのサポートが与えられているため、モバイル開発の世界ではなく、そのセグメントで優れている言語の誰かの分野に耳を引っ張って短い人生を過ごすのはなぜか私には明確ではありません。



-Android 8の素晴らしい革新には、java.nio.fileとjava.lang.invokeの追加があります。 これらの新しいAPIの利点は何ですか？また、どのシナリオで最適に使用されますか？



Denis Neklyudov：開発者がこれらのAPIに依存するためにアプリケーションにminsdk 26を配置するには、数年が経過することを忘れないでください。 しかし、一般的に、それらは特定のライブラリを書く人にとって有用です。
