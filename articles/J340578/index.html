DBMSリリースInterSystemsCaché2017.2





先週、CachéDBMSの新しいバージョンが番号2017.2でリリースされました。

ロシア語の変更点のリストをご紹介します。 変更点の完全なリストと英語のアップグレードチェックリストは、 こちらにあります 。



ミラーリングおよびログ回復中の並列ログ解除



Chaché2017.2では、ログのミラーリングとログからの回復のログ速度が向上しました。 これにより、データベースの更新頻度が高いミラーシステムのスケーラビリティが向上します。



ジャーナリング解除とは、ジャーナルエントリをデータベースに適用するプロセスです。 以前は、単一のプロセス（および一連の補助プリフェッチプロセス）がデータベースの更新とログの削除に従事していました。 このリリースでは、複数のプロセス（最大4つ）がジャーナルエントリを異なるデータベースに並行して適用できます。 この機能は、十分なプロセッサと共有メモリが使用可能な場合に機能します。



ミラーリングする場合、新しい構成設定を使用して、ノードのタイプによって並列デジャーナリングを制限できます。 デフォルトでは、この機能は「 レポート非同期 」タイプのノードに対して無効になっています。 レポートにロギング解除による更新処理中の複数のデータベースのデータが含まれている場合、異なるデータベースの更新のタイミングの違いにより、これらのレポートの予測は以前のバージョンよりも変動する可能性があります。 この変動の性質は、本質的に、変化するデータからコンパイルされた他のレポートの変動の性質と変わりません。



並列デジャーナリングの詳細については、ドキュメントの「 ミラーリング 」の章の「並列デジャーナリングの構成 」セクションを参照してください。



ログから回復する場合、非標準設定が選択されていると、並列デジャーナリングは機能しません。 特に、エラーが発生して操作がキャンセルされたとき、またはジャーナルの更新を使用しているときは機能しません。 同時デジャーナリングの詳細については、ドキュメントのジャーナリングの章の「 ^ JRNRESTOを使用したジャーナルファイルからのグローバルの復元 」セクションを参照してください。



また、システムの起動時に、シャドウサーバーで並列ログ解除が機能しません。



最後に、新しいCachéリリースには、同時デジャーナリングが機能するかどうかに関係なく適用される2つの機能強化が含まれています。 ロギング解除プリフェッチプロセスの効率が改善され、ロギング解除キューのメモリ使用量がプロセスごとに50MBに制限されました。 以前のバージョンでは、デロギングキューはかなり多くのメモリを消費していました。



DeepSeeの新しい機能



DeepSee Folder Manager



DeepSeeコンポーネントの展開を簡素化するために、関連オプションをエクスポートするための新しいオプションがFolder Managerに追加されました。 コンテナクラスにエクスポートする場合、このオプションは、選択した要素だけでなく、選択した要素に関連する他の要素もエクスポートします。 表示パネルの関連項目には、表示パネルで使用されるピボットテーブルと用語のリストが含まれます。 ピボットテーブルの関連アイテムには、名前付きフィルター、ピボット変数、および一般的な計算アイテムが含まれます。



以前のバージョンでは、Folder Managerは常にサーバーファイルシステムを使用してファイルをエクスポート/インポートしていました。 現在、Folder Managerでは、ローカルまたはサーバーのファイルシステムを選択できます。



ダッシュボードフィルター-名前付きフィルターをダッシュ​​ボードフィルターのデフォルト値として使用できるようになりました。



新しいiFindおよびiKnow機能



iFindは、隣接する場所による検索をサポートするようになりました。検索語が互いに近いが、必ずしも厳密な位置検索ではなく、その順序であるエントリを検索できます。 隣接する場所による検索を有効にするには、角括弧内のコンマで区切られた検索語のリストを取得します。 オプションで、これらの用語が発生する範囲を指定できます。 たとえば、クエリ[Boston、New York、5]は、「Boston」と「New York」という用語が互いに5桁以内にあるすべてのエントリを検索します。



iKnowは、ドメインを作成するとき、特に多数のコアを持つシステムを使用するとき、いくつかのパフォーマンスの改善を受けました。 機器とデータセットによっては、合計処理時間が10％から30％に短縮されることがあります。

拡張されたiKnow REST API。 たとえば、ほとんどの既存のAPIエンドポイントについて、結果の数のみ、または結果の完全なリストを持つ数のみを取得できます。



SQLの機能強化



SQLクエリの監査



このリリースでは、SQLクエリの実行を監査する機能が追加されています。 SQLクエリを監査するための3つの新しいシステムイベントがあります。

％System /％SQL / DynamicStatement-動的SQLクエリ用

％System /％SQL / EmbeddedStatement-埋め込みSQLクエリ用

％System /％SQL / XDBCStatement-xDBCクエリ用



これらのイベントを有効にするには、システム管理→セキュリティ→監査→システムイベントの設定を選択して、システム監査イベントページのコントロールポータルに移動します。



クエリの最適化



SQLクエリはいくつかの最適化を受けました。 クエリオプティマイザーは、接続条件の選択性を計算する際に排出の選択性を考慮するようになりました。これにより、一部のクエリのプランが改善されます。 外部結合は、内部結合で利用可能なすべての最適化を使用できるようになりました。 特に、インデックスによって部分的に満たされる外部結合条件では、一時インデックスを作成する必要がなくなり、クエリパフォーマンスが大幅に向上することがよくあります。



オプションのANSI SQL文の優先度



新しいオプションにより、CachéSQLで既定で使用される左から右へのステートメントの優先度の代わりに、ANSI SQL標準に従ってステートメントの優先度を設定できます。 キャッシュSQL優先度は引き続きデフォルトで使用されますが、APIまたはControl Portalの一般SQL設定ページから、システム管理→設定→SQLおよびオブジェクト設定→一般SQL設定を選択して優先度を変更できます。



固定計画の「進化」



Cachéの以前のバージョンでは、Cachéの新しいバージョンにアップグレードするときに、SQLクエリプランを記録し、クエリプランを自動的に記録する機能が追加されました。 新しいバージョンにクエリプランの最適化が含まれていた場合、アップグレード中に修正プランに適用されませんでした。 現在のリリースでは、Cachéは、新しい最適化の恩恵を受ける可能性のあるクエリを識別してメモします。 SQL式のメインページでは、そのようなクエリは[新しいプラン]列に表示されます。 このような計画を凍結解除して、新しい最適化を活用できます。



Atelierの新機能に関する情報の入手先-EclipseベースのIDE



Atelierは、CachéのEclipseベースのIDEであり、Cachéのリリースサイクルに依存しないリリースとして利用できます。 新しいAtelier機能については、新しいバージョンごとに提供されているドキュメントに記載されています。 Atelierを使用すると、クライアントシステムでCachéアプリケーションを開発し、アプリケーションをCachéサーバにアップロードして、実行およびデバッグできます。



今後の開発では、新しいIDEに焦点を当てます。 Studioは引き続きインストールに使用でき、開発者は引き続き開発に使用できます。 ただし、Studioはサポート製品と見なされ、新しい機能は提供されません。



必要なリソースと問題の重大度によっては、Studioのマイナーなエラーが解決されない場合があります。



Atelierは、CachéまたはEnsembleに加えて、スタンドアロンのダウンロードとして入手できます。 既存のEclipseインストール用に別個のAtelierインストーラーまたはプラグインを使用することを選択できます。 AtelierはEclipseの自動更新メカニズムを使用して、ユーザーがタイムリーに更新を受信できるようにします。 Atelierは別のダウンロードページからダウンロードできます 。
