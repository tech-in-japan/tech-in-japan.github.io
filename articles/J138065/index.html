情報アーキテクチャ



情報アーキテクチャを作成する目的は、情報、タスク、およびそれらの間の関係を定義することです。 これは、ユーザーがデジタル形式で受け取るものです。情報と、この情報を使って何かをする能力です。情報を消費して、意思決定や何かについての知識を得る、または情報を作成します。



情報アーキテクチャ（IA）はまったく別の分野です（ 情報アーキテクチャ研究所もあります）。 情報アーキテクチャの目的は、情報を整理することです。



アイデアとコンセプトを生成する段階で、素晴らしいアイデアを作成しました。 これらのユーザーシナリオは、名前、日付、価格、画像、温度範囲などの情報の形のないクラスターに関連付けられています。情報アーキテクチャの段階で、この形のない部分を取り、情報構造を形成します。 初めてこれを行うことは不可能です。 これには多くのパスが必要です。



IAを定義するのに役立つ2つの非常に便利なツールがあります。 アプリケーションフローチャート（アプリケーションフロー図）とラフプロトタイピングです。







したがって、最初のアイデアを情報アーキテクチャに適用し、アプリケーションフロー図を作成してテストします。 成熟度にはさまざまなレベルがあります。通常、タスクのフローから始めて、詳細画面 + コンテンツビュー + ナビゲーション図に進みます 。



ソフトウェア開発（またはその他のプロセス）の古き良きフローチャートを覚えていますか？ これらはアプリケーションの同じフロー図であり、使用する視覚的表現のみがユーザーフローとユーザーエクスペリエンスと相互作用を設計するタスクに焦点を合わせています。 アプリケーションのフローチャートを作成したら、対応するユーザースクリプトのストーリーを伝えることができます。 その結果、情報アーキテクチャを処理するためのフィードバックとアイデアを受け取り、その説明に戻って磨きをかけることができます。



その後、戻ってもう一度確認し、より正確なアプリケーションフロー図などを作成します。 少しずつ、アプリケーションフロー図がより詳細になり、単純なタスクフローから、コンテンツやナビゲーションの表示のアイデアを示す画面に移行します。 高品質のアプリケーションフローチャートをワイヤフレームとは呼びませんが、多くの人はそうします。 もちろん、低精度の回路。

インタラクションデザイン



情報の構造と、この情報を使用してユーザーが実行できるタスクを決定しました。 ここで、これらすべてを復活させるために、これらすべてのユーザーインターフェイスの開発を開始する必要があります。



インタラクションデザインの本質は次のとおりです。適切に構成された情報を表示できるユーザーインターフェイス要素とインタラクションエクスペリエンスのセットを作成し、ユーザーがこの情報に関連するタスクを正常に完了できるようにします。 この段階で達成したいのは、情報とタスクを実装する可能性を最大限に高めることです。 インタラクティブなデザインが完全に開発されていない場合、情報アーキテクチャは完璧かもしれませんが、これは情報がアプリケーションに完全に表示されず、ユーザーが目的のタスクを完了できないという事実につながります。



私の意見では、デフォルトでは、対話型設計（対話型設計）は情報とタスクのフィルターです。 定義上、これは情報でもタスクでもないため、これはまさにフィルターであり、単なる手段にすぎません 。 インタラクションデザイン（またはユーザーインターフェイス）は、ユーザーと情報の間です。 言い換えれば、ユーザーインターフェイス（インタラクティブデザインの結果）は、他の方法で証明されるまでは有罪でなければなりません。 :-)



このコンセプトは、Metro-designの原則が示すものとよく一致していると思います。仮想シーンの主役はユーザーインターフェイスではなく情報です。 ユーザーインターフェイスは、情報を投稿し、タスクを完了できるようにするためだけに存在します。 これは、ユーザーインターフェイスレイヤーが薄くてほとんど見えないカーテンであるか、 厚くて重いメンブレンであるかに関係なく、すでに適切な（または不十分な）デザイン実行の問題です。 そして、私は視覚的なデザインについても話していません-インタラクションデザインについてだけです：ユーザーがこの接続に関連して可能な情報とタスクと対話する方法。



Windows Phone用のMetroデザイン言語がなかった場合、やり取りのメタファーをゼロから作成する必要があります。次の記事では、Metroの原則に基づいて前進する方法について詳しく説明し、独自のイノベーションを導入する方法についていくつかのアイデアを説明しますが、この記事では、そのMetro言語に基づいたインタラクションデザインの定義に焦点を当てます。 Windows Phone用に既に用意されており、私たちの味方として使用できるデザイン。



この取り組みでは、デザインパターンが友人になります。シームレスな情報アーキテクチャを持つことは、Metroコントロールに情報構造を投影するのに役立ちます。情報アーキテクチャとタスクにより、ページ、ピボットページ、またはパノラマなどのコントロールが表示されます。情報の階層と構造が投影されますコンテンツの提出。異なる情報、個々のタスク間の関係、および情報とタスク間の関係によりナビゲーションが生成されます（アプリケーションパネルを思い出してください）。情報アーキテクチャの説明のすべてが、特定のWindows Phoneコントロールに変換されます。不確実性はないはずです。不確実性がある場合、できることはいくつかあります。ニーズに合わせて設計パターンをカスタマイズするか、独自のパターンを作成します。







繰り返しになりますが、まず、情報アーキテクチャの説明に基づいて、Windows Phoneに適したいくつかの組み込み設計パターンを選択します、たとえば、検索、ローカル検索のカード、または「連絡先」ハブのメール、プレイリスト、連絡先カードなど。さらに、情報アーキテクチャのニーズを完全または一般的に満たすデザインパターンが見つからない場合は、かなり近いデザインパターンをカスタマイズするか、独自のパターンを作成することもできます。
