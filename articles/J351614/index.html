TP-Link T2600G-28MPS：PoE、LLDP、および音声VLANと連携

約6か月前に、ワイヤレス機器（アクセスポイントとコントローラー）に基づいてWi-Fiネットワークを構築する例を公開しました 。 本日、T2600G-28MPSレベル2+スイッチについて詳しく説明します。これは、ビデオ監視システム、分散型ワイヤレスネットワーク、IPテレフォニーを提供する有線ネットワークセグメントを構築し、パケットスイッチングを実行するために使用できます。







T2600G-28MPSのすべての機能の完全な説明を提供するという目標を設定するのではなく、サポートされるワイヤレスネットワークとIPテレフォニーの動作を直接サポートする機能に焦点を当てます。



Power over Ethernetのセットアップと操作



端末機器に電力を供給することは、ネットワークエンジニアが解決するのが最も簡単な作業ではありません。 もちろん、ほとんどのモデルのアクセスポイント、IPカメラ、電話に付属の外部電源を使用できます。 ただし、このソリューションはスケーラブルではなく、便利なソリューションと考えることは困難です：多数の追加ブロックとワイヤ、ネットワーク展開時間の増加、施設の外観の悪化、集中型電源管理の不可能性、電源の品質を保証できないなど。 代わりに、ますます多くのネットワーク管理者がPoEテクノロジーに注意を向けています。これにより、これらすべての問題をすぐに解決できます。



TP-Link T2600G-28MPSのエネルギー収支は増加しています（最大384ワット）。 今日では、802.3af-2003と802.3at-2009の2つのIEEE標準が広く使用されており、テクノロジーの動作が記述されています。 それらの最初は、15.4ワットに等しい最大利用可能電力を決定します。 2番目によって提供される最大電力は30ワットです。 簡単な計算により、このスイッチを使用すると、デバイスのすべてのポートに接続された802.3afコンシューマーに同時に電力を供給できることがわかります。 より強力な消費者を接続する場合（802.3at標準をサポート）、すべての端末デバイスに最大許容電力を提供することはできませんが、すべての802.3atデバイスが動作するために常に25.5ワットの電力を必要とするわけではないことに注意する必要があります。 そのような消費は、むしろ例外であるか、比較的短い期間必要となる場合があります。 さらに、スイッチには、エネルギー収支不足のモードで状況を解決できる設定があります。 ネットワーク管理者が管理できるものを見てみましょう。



PoEテクノロジーに関連するすべての設定は、Webインターフェイスの同じメニューグループに収集されます。 管理者は、「PoE Con​​fig」アイテムを使用して、PoEクライアントに電力を供給するためにスイッチが消費する最大電力を指定し、ポートの現在の状態を表示および変更し、接続デバイスのインターフェース優先度とクラスを設定し、時間間隔とPoEプロファイルを選択できます。

同じメニュー項目の[PoEプロファイル]タブでは、エネルギープロファイルを作成できます。これにより、一般的なコンシューマーを接続する際のスイッチ構成が大幅に簡素化されます。

LLDP

ネットワークでのLLDPプロトコルの主な用途の1つは、その拡張LLDP-MED（メディアエンドポイントディスカバリー）の使用である可能性があります。これにより、サービス情報が音声端末機器と交換されます。 もちろん、前述のように「音声」はRTPにカプセル化されています。 シグナリングには、SIPまたはH.323の2つのプロトコルのいずれかが従来使用されています。 それでは、なぜLLDP-MEDが必要なのですか？ この拡張により、IP電話と音声ゲートウェイの構成コストを大幅に削減できます。 通常、音声データを送信するために、スイッチ上に個別の仮想ネットワーク（VLAN）が作成されます。 何のために？ 通常、2つの理由があります。セキュリティの必要性と音声トラフィックに優先順位を付けることです。



当然、IP電話が接続されているすべてのスイッチポートをこの仮想ネットワークに配置するだけでもかまいませんが、ネットワーク管理者はスイッチング機器のネットワークインターフェイスの不足に対処する必要があります。 スイッチに十分な数の空きインターフェイスがあるネットワークにIP電話を接続する非常に簡単な例を以下に示します。







キャンパスのスイッチにL2ポートがないという問題は、ユーザーのコンピューターを電話の特別なポートに接続することで解決されることがよくあります。 IP電話自体には、3つのポートを備えた統合イーサネットスイッチがあります。1つは内部、2つは外部（ネットワーク機器とユーザーのPCへの接続用）です。 2つの仮想ネットワークからのデータは、電話とネットワークスイッチ間のチャネルを介して送信されるため（access-vlanのユーザーデータとvoice-vlanの音声データ）、たとえば802.1qを使用してフレームにタグを付ける必要があります。 このようなタグ付けは非常に一般的なソリューションであり、ネットワーク管理者にとって問題を引き起こすことはありません。 ただし、正しく機能するためには、電話機自体を特定の方法で構成する必要があります-仮想ネットワーク識別子の対応する値-VID（VLAN Identifier）を登録する必要があります。







したがって、すべての電話を設定する必要があります。 プロセスはまったく速くありません、同意します。 このルーチンタスクを解決するために、LLDP-MED拡張が使用されます。これにより、スイッチは、使用されている仮想ネットワーク番号をIP電話に通知できます。



LLDP-MED拡張機能を構成するには、同じ名前のメニュー項目を参照してください。

これで設定を完了できるようです。 ただし、電話とスイッチがどのように相互作用するかについてもう少し詳しく説明したいと思います。 テストモデルの例-Avaya IP Deskphone 9620Lについて説明します。 私たちの生活を単純化しないために、スイッチのネットワークインターフェイスが不足している状況、つまり、ユーザーのPCが電話で接続されている状況を考慮します。 2つの仮想ネットワーク（VLAN 2-音声、VLAN3-データ）を作成し、対応する仮想SVIインターフェイス（VLAN 2-192.168.2.1/24、VLAN 3-192.168.3.1/24）を構成し、DHCPサーバー用に2つのプールを構成しました。



そのような電話機を接続するとすぐに、スイッチは受電PoEデバイス（PD-受電装置）を検出し、ポートに電圧を供給します。これにより、IP電話がオンになり、ダウンロードを開始できます。 下のスクリーンショットからわかるように、電話は大食いではありません（メーカーによると、このモデルは最悪の場合最大5.3ワットを消費する可能性があります）。 原則として、ほとんどすべてのAvaya IP Phoneモデルの最大消費電力は7ワットを超えません。 他のベンダーにも同様の状況があります。 その結果、TP-Link T2600G-28MPSは24台のIP電話に余裕を持って電力を供給します。







しかし、LLDPに戻ります。 スイッチは、このプロトコルでネイバーを検出しました。 一見すると、このページに示されている結論は少し奇妙に見えます。1つの電話についての2つのエントリです。 これが起こる理由を理解してみましょう。







電話機がスイッチに2つの異なるシャーシIDを伝えることがわかります。 実際、これは、DHCPサーバーから電話機が受信したIPアドレスです。つまり、電話機は各プールから1つのアドレスを要求します。 ただし、DHCPサーバーは、電話に対して発行された1つのアドレスのみを表示します。







この段階では、すべてが完全に混乱し、理解できないように見えます。 しかし、答えは非常に簡単です。 電話とスイッチ間でLLDPプロトコルを使用してパラメータをネゴシエートする手順のみを考慮する必要があります。



電話機に電力が供給されて初めてロードするとき、どの仮想ネットワークが使用されているかはまだわかりません。つまり、フレームをマークするために必要なVIDタグに関する情報はまだありません。 この時点で、IP電話はタグなしフレームを送信します。 これらのフレームは、メニューの「VLAN」グループの「802.1Q VLAN」項目の「Port Config」タブの「PVID」フィールドで定義された仮想ネットワークに分類されます。







そのため、ユーザーデータの送信に使用する予定の仮想ネットワークの電話のMACアドレスに関するレコードがブリッジブリッジテーブルに表示されます。





この時点で、電話機はDHCPプロトコルを使用してIPアドレスとその他のネットワークパラメーターを取得します。 このプロセスと並行して、LLDPメッセージング（LLDP-MEDを含む）が発生し、その結果、電話機は独自のフレームを配置する仮想ネットワークの番号を認識します。











正しい仮想ネットワークの番号を確認すると、IP電話は以前に受信したDHCPアドレスを解放し、同じプロセスを繰り返しますが、既に独自のフレームにタグ付けしているため、新しい仮想ネットワークに既にあるIPアドレスやその他のネットワーク情報を取得します そのため、DHCPアドレスでリースされているのは1つだけです。 さて、ブリッジテーブルのエントリは、その有効期限が切れるまで「ハングアウト」します（Webインターフェイスメニューの「スイッチング」グループの「MACアドレス」項目の「エージングタイム」タブ「動的アドレス」オプション）。 私たちの言葉を確認するために、説明されたアクションを含むダンプの小さな断片を持参することにしました。







ダンプは、ポートミラーリング機能を使用して削除されました。 残念ながら、このオプションを使用してすべてのサービスフレームを転送できるわけではありません。







結論として、IP電話にステーションまたは音声ゲートウェイ/サーバーのアドレスを伝える方法はいくつかあることに注意してください。 電話自体の設定およびさまざまな独自のソリューションでアドレスの静的な表示を考慮しない場合、多くのオプションはありません。 これらには、たとえば、TFTP / FTP / HTTP / HTTPSを介して電話でダウンロードできる構成ファイルを使用して、この設定を（他のパラメーターのホストと一緒に）転送する非常に明白な方法が含まれます。 やや目立たない方法は、さまざまなDHCPオプションを使用することです。 たとえば、Avaya IP Phoneはオプション＃176を使用して、H323ゲートキーパーの機能を実行するデバイスのアドレスを指定します。 次の記事（ http://foxnetwork.ru/index.php/component/content/article/207-dhcp.html ）の助けを借りて、DHCPプロトコルの世界に飛び込むことができます。 ゲートウェイアドレスについて電話に通知する上記の方法に加えて、LLDPプロトコルという別の方法を使用できます。 現在まで、当社のスイッチは、後者の2つの機能のサポートをまだ誇っていません。

音声VLAN



現代のネットワークでは、特定の仮想ネットワークに属するフレームは、フレームがスイッチによって受信されたインターフェース（ポートベースのVLAN）、またはプロトコルタグ802.1q（タグベースのVLAN）に基づいて決定されます。 たとえば、送信者のMACアドレスに基づいて、フレームが仮想ネットワークに属しているかどうかを判断する方法がいくつかあります。 イーサネットネットワークのMACアドレスは48ビット長で、2つの等しい部分で構成されます。最初の部分には、各ネットワーク機器メーカーに集中的にIEEEによって割り当てられたOUI（Organizationally Unique Identifier）が含まれます。 送信者アドレスのOUI値に基づいて、スイッチによってフレームを音声仮想ネットワークに割り当てることができます。 OUIに基づいて最初から最後まで音声仮想ネットワークを設定するプロセス全体を見てみましょう。 公平に言えば、スイッチがIP電話だけでなく、デバイスのMACアドレスに基づいて仮想ネットワークのメンバーシップを決定できることは注目に値します。 対応する設定は、メニューの「VLAN」グループの「MAC VLAN」項目で使用できます。







したがって、OUI値を管理することから始めて、それに基づいてフレームが音声仮想ネットワークに分類されます。 これは、同じメニュー項目の「OUI構成」タブを使用して行われます。







何らかの理由でまだ作成されていない場合は、音声トラフィック用の仮想ネットワークを作成する必要があります。 この手順は、Webインターフェイスメニューの「VLAN」グループの「802.1Q VLAN」項目を使用して実行できます。 この段階で、この仮想ネットワークにスイッチインターフェイスを追加します。











通常の仮想ネットワークが作成された後、音声トラフィックの伝送に使用する予定のものを正確に示す必要があります。 スイッチで作成できる音声仮想ネットワークは1つだけです。 対応する設定は、QOSグループのVoice VLAN項目のGlobal Configタブで利用できます。







最後の仕上げは、ポート構成タブを使用して物理インターフェースを構成することです。







「セキュリティモード」オプションを使用して追加の保護を有効にする可能性について言及するしかありません。 この機能を有効にすると、スイッチは、送信者が構成済みのOUIのリストにない音声仮想ネットワークにフレームを渡しません。
