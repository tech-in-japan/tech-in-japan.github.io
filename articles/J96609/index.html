HTC Hero GSMの例を使用したAndroidフォン用のファームウェアの開発と変更。 パート1

ファームウェアの修正バージョンをインストールする理由は異なります。 面白いローディングアニメーションで友人を驚かせたい、誰かが特定の機能（vpnなど）を欠いている、誰かがプロセッサをオーバークロックして携帯電話の最大パフォーマンスを絞りたい、誰かがAndroidオペレーティングシステムの新しいバージョンを5か月待っているお気に入りのHTCヒーロー。

現時点では、さまざまなAndroidスマートフォン向けの非常に多くの想像を絶するアセンブリがすでに存在しています。 時々、それらはHabréで何らかの形で現れることさえあります。

公式のファームウェアに基づいてカスタムファームウェアを作成するプロセスと機能について説明します。 この知識は、HTC Hero GSM用のAndroid 2.1に基づく数少ない国内ファームウェアの1つの開発中に得られました。 そして、ロシアのある主要なフォーラムで、自分自身と他の立派なユーザーで多かれ少なかれ正常にテストされました。

次のすべてがHTC Heroに対して行われたという事実にもかかわらず、これらのルールと機能はすべての電話、特にHTCによって開発され、独自のSenseシェルを使用するものに対して完全に有効です。

実験には、次のものが必要です。

  • できれば最新バージョンのAndroid SDK
  • システムアプリケーションをリエンジニアリングするためのApktoolユーティリティ
  • システムアプリケーションの最適化を解除するためのSmali / baksmaliユーティリティ
  • イメージからシステムファイルを抽出するUnyaffsユーティリティ
  • カーネルとRAMディスクを抽出するsplit_bootimg.plスクリプト
  • サービスパックおよび個々のアプリケーションに署名するためのTestsignユーティリティ
  • インストールおよび構成されたJRE
  • Android携帯
  • スーパーユーザー権限と修正されたリカバリルーチン（ リカバリROM ）


上記のすべては、LinuxとWindowsの両方のバージョンで利用可能です。 しかし、私の例では、Linuxの使用に焦点を当てます。

もちろん、開発を開始するためにルート権限やリカバリは必要ありませんが、作成を試してみたい場合は必要になります。 HTC Heroの場合、 RA-hero-v1.6.2を使用できます。

おそらく、非公式のファームウェアを使用すると保証が失われますが、私たちのファームウェアは消えませんでした。 そして、ほとんどの操作が安全であるという事実にもかかわらず-あなたは常にあなたのアンドロイドに不可逆的な害を引き起こさないように、何がなぜ行われているのかを明確に理解する必要があります



ベース





ファームウェアの開発には、いくつかの異なるアプローチがあります。

  • Android Open Source Projectソースからビルドする
  • CyanogenModプロジェクトのソースからのアセンブリ
  • 電話メーカーが提供するファームウェアの変更


Androidプラットフォームが開いているように見えるという事実にもかかわらず、実際の電話は閉じたコンポーネントを使用します。 これらは、バイナリ形式（wifi / gps / fm）で配布されるドライバーと、Marketやその他のGoogleサービスなどの主要なシステムコンポーネントです。 また、ここでは、サムスンのHTC Sense 、Motoblur、 TouchWizなど、インターフェースの分野の企業の開発を追加する必要があります。 これは、ソースコードからファームウェアを開発する際に克服できない困難をいくつかの場所で生み出します。

私は、電話ベンダーが提供する既製のファームウェアの修正について説明することを提案します。

HTC電話のファームウェアには、次の2つの形式があります。

  • RUU。 Rom更新ユーティリティ 。 電話を更新するWindows用ユーティリティ
  • オタ。 Over the Air 。 wifi / gprsネットワークを介して電話自体によってダウンロードされたパッケージ。コンピューターの関与なしに電話にインストールされます。


最近、OTA更新はバイナリdiffで構成されているため、それらをベースとして使用するのは多少困難になっています 。 これには特定のファームウェアバージョンの特定の電話が必要であるため、ある電話から別の電話にファームウェアを適合させたい場合、これは単純に不可能です。 または、電話識別子の長くて苦痛な操作に続いて、更新プログラムをインストールせずにダウンロードします。 さらに、この手順を数回繰り返す必要がある場合があります。

RUU更新を使用します。



rom.zipを抽出する





1.興味のある電話に適したバージョンのRUUをダウンロードします。 HTC Webサイトまたは他のソースで見つけることができます。 HTC Heroの場合、6月上旬にリリースされたAndroid 2.1のバージョンをオペレーターのChunghwa（台湾）に使用します

2.電話機を更新するときのRUUユーティリティは、いくつかの領域を同時にフラッシュします。

  • ブートローダー（hboot）
  • linux kernel + ramdisk（ブート）
  • 無線モジュール（無線）のファームウェア
  • 回復ルーチン
  • システムパーティション（/システム）
  • ユーザーセクション（/データ）


ただし、愛情を込めてインストールされたブートローダーとリカバリをRUUユーティリティで上書きすることはできません。 公式ファームウェアだけでなくインストールを継続できるようにするため。 これを行うには、ラジオ/ブート/システム/データを抽出する必要があります。

本質的に、RUUは、rom.zipに必要なイメージを保持するInstallShieldインストーラーです。

それを起動すると、メインのウェルカム画面が表示されます。 さらに進むことなく、％TEMP％システムフォルダーを開きます。このフォルダーには2つの新しいフォルダーがあり、そのうちの1つにrom.zipファイルがあります。 人里離れた場所にコピーし、アンインストールしてRUUを閉じます。

コア





また、ブートパーティションも必要です。これは、基本的にはramdiskを備えたLinuxカーネル（選択したファームウェアの場合は2.6.29 armv6l）であり、次の形式です 。 

  ** + ----------------- +
 ** | ブートヘッダー|  1ページ
 ** + ----------------- +
 ** | カーネル|  nページ
 ** + ----------------- +
 ** | ラムディスク|  mページ
 ** + ----------------- +
 ** | 第二段階|  oページ
 ** + ----------------- +
 **
 ** n =（kernel_size + page_size-1）/ page_size
 ** m =（ramdisk_size + page_size-1）/ page_size
 ** o =（second_size + page_size-1）/ page_size


カーネルを交換する場合、または初期化スクリプトを変更する場合は、 ブートイメージからそれらを抽出する必要があります 。 これを行うには、 William Enckによって作成された素晴らしいperlスクリプトsplit_bootimg.plが必要です 。
  1. $ split_bootimg.pl .. / rom / boot.img $ ls boot.img-kernel boot.img-ramdisk.gz data system



  2. $ split_bootimg.pl .. / rom / boot.img $ ls boot.img-kernel boot.img-ramdisk.gz data system



  3. $ split_bootimg.pl .. / rom / boot.img $ ls boot.img-kernel boot.img-ramdisk.gz data system





カーネル自体を別のものと交換することを除いて、カーネル自体で何もすることはできません。また、ramdiskはその後の変更と設定のために解凍できます。
  1. $ mkdir ramdisk && cd ramdisk && gzip -dc .. / boot.img-ramdisk.gz | cpio -i $ ls data default.prop dev init init.goldfish.rc init.hero.rc init.rc logo.rle proc sbin sys system



  2. $ mkdir ramdisk && cd ramdisk && gzip -dc .. / boot.img-ramdisk.gz | cpio -i $ ls data default.prop dev init init.goldfish.rc init.hero.rc init.rc logo.rle proc sbin sys system



  3. $ mkdir ramdisk && cd ramdisk && gzip -dc .. / boot.img-ramdisk.gz | cpio -i $ ls data default.prop dev init init.goldfish.rc init.hero.rc init.rc logo.rle proc sbin sys system



この記事では、カーネルまたはramdiskで何もしたくないので、すべてをバックパックします（または、ステップに戻ってブートにまったく触れません）
  1. find . | cpio --quiet -o -H newc | gzip > .. / new-ramdisk.gz $ cd .. && mkbootimg --kernel boot.img-kernel --ramdisk new-ramdisk.gz --cmdline "no_console_suspend=1 console=null" -o newboot --base 0x19200000



  2. find . | cpio --quiet -o -H newc | gzip > .. / new-ramdisk.gz $ cd .. && mkbootimg --kernel boot.img-kernel --ramdisk new-ramdisk.gz --cmdline "no_console_suspend=1 console=null" -o newboot --base 0x19200000



他の電話機では、ベースオフセット設定が異なる場合があります。 解凍すると、split_bootimgからコマンドラインを取得します。

ファームウェア





受け取ったファームウェアにまだ新しい機能を追加していないという事実にもかかわらず、フラッシュして起動しようとしています。

このために
  1. サービスパックをマップのルートにコピーします
  2. 復旧する
  3. nanroidバックアップの作成
  4. 拭く
  5. 点滅
  6. 過負荷です


私たちの患者が複雑な手術を生き延びたことを嬉しく思います。





このレビューには、準備された基本ファームウェアの変更に直接影響する以下の重要な問題は含まれていません。 カスタムフェイスの作成者が遭遇する質問、およびHTC Hero GSMのロシア語バージョンのファームウェア2.1を作成するときに私が直接遭遇した質問

  • ルート、busyboxの追加
  • パッケージの最適化解除。 smali / baksmaliを使用します。 Odex依存関係の解決
  • リモートodexアプリケーションの最適化
  • システムパッケージの変更。 署名パッケージ：必要な場合と無視できる場合。 アプリケーションのパッケージ化。 再構築せずにアプリケーションリソースを更新する
  • システムの大規模な変更の例として、システムのローカリゼーション。
  • システムアプリケーションリソース/バイトコードの逆コンパイルとアセンブリ。 システムアプリケーションエラーの修正
  • zip-align、png最適化
  • 共有uidを使用する場合のアプリケーション署名とコラボレーションの問題
  • あるパーティションから別のパーティションにデータを転送する例としてのinitスクリプトの使用（dalvik-cache2cache、app2sd）


それまでの間、 FOTAクライアントの更新の後に、システム自体のバージョン2.1-update1への更新が行われることを楽しみにしています 。
