なぜ、なぜ、これらの＃？@！ バカはviを使用しますか？

私は「Habrahabr」の読者に記事の翻訳を提供します「なぜ、ああ、なぜ、それらの＃？@！Nutheadsはviを使用するのですか？」 ジョン・ベルトラン・デ・エレディアが執筆。







はい、信じられないかもしれませんが、30年以上前に光を見たviエディター（そして彼の若い、約15歳の最高のクローンと大きな改善-vim）には多くのファンがいます。







いいえ、彼らは最新の状態を保ちたくない恐竜ではありません。viユーザーコミュニティは成長を続けています。私は、わずか2年前に（プログラマーとして10年働いた後）始めました。 友達は今viに切り替えています。 地獄、ほとんどのviユーザーはそれが書かれたときに生まれさえしなかった！







はい、vi / vim編集モデルが他のどのモデルよりも優れている特定の理由があります。 viを使用するためにUnixの専門家である必要はありません。ほとんどの既存のプラットフォームで無料で利用できます。 ほとんどのIDEには、その機能を使用できるプラグインがあります。 いくつかの誤解を払拭し、その優位性を示すいくつかの例を考えてみましょう。







誤解＃1：モーダル編集


最初にviまたはvimに出会ったとき、入力を開始するために「i」を押す必要があることにショックとうんざりします。 どの「hjkjl」ボタンがどの方向を担当しているのか思い出せません。 そして、行の最後に印刷するには「a」を押す必要があります。 おそらくすでに他のエディターを使用しており、矢印ナビゲーションは入力モードで機能するため、絶対に必要な場合にのみコマンドモードに切り替えて使用します。 たぶん、あなたはこれに20分を費やし、絶えず不平を言うでしょう。 -二度とviを使いたくない。







しかし、これはviを使用する絶対に間違った方法です。 適切な使用はモーダルではありません。 常にコマンドモードで、短い「爆発的な」タイピングの場合のみ入力モードに入り、その後すぐにコマンドモードに戻ります。 だから、問題は覚えていません-どのモードが存在しないかです。入力モードでは電話に出ませんが、戻ったとき、自分がどこにいたかを思い出そうとします。 コードを書き、電話が鳴り、コマンドモードに戻り、コールに応答します。 または、同意したらそこに戻りますが、入力モードを通常の作業モードとは決して思いません。







この背後にある哲学を説明させてください。







viコマンドは結合されるように設計されています：「d」は削除を意味し、「e」はカーソルを単語の末尾に移動することを意味するため、「de」は「現在のカーソル位置から次の単語の末尾までをすべて削除する」ことを意味する完全なコマンドです（何かほとんどの通常のエディターのCtrl-Shift-Right、Left、Delなど）。







これの最初の肯定的な結果は、コマンド「。」 最後に完了した結合編集コマンドを繰り返します（ただし、移動コマンドは繰り返しません）。 「dw」、「。」を押した後 「現在のカーソル位置から次の単語開始まで削除」コマンドのように動作します。 ナビゲーションコマンドのすべての機能を使用してファイルをナビゲートし、「。」を押すことができます。 次の単語が始まる前にすべてを削除する必要がある場合は、非常に迅速に。 この機能は非常に強力です。







そして、今度は入力コマンドに到達しました。 これらのコマンドは、それぞれに固有のアクションを実行した後、エディターを入力モードにして、テキストを入力できるようにします。 そのため、これらのコマンドの場合、コマンドの全文には、「i」（またはそこで使用した入力コマンド）からコマンドモードに戻るまでの間に入力したすべてのテキストが含まれます。







これは、現在のカーソル位置に「Hello」と入力する「iHello [ESC]」と入力すると、「。」になります。 現在のカーソル位置に「Hello」と入力するコマンドです。 それだけです。 この機会がいかに強力か想像できます。 「A」はカーソルを行末に移動し、入力モードをアクティブにします。 [ESC]を押してダイヤルを完了したら、「。」を押します。 行末で入力を繰り返す任意の場所。







別のさらに印象的な例：「ce」コマンドは、「c」コマンドと「e」コマンドの組み合わせです。 'c'hange'コマンドは、次の移動コマンドで示される範囲を削除し、入力モードをアクティブにします。 コマンド 'd'に似ていますが、コマンドモードを終了します。 さらに、次の（短い）入力セッション中に入力されたテキストもコマンドの一部です。 したがって、「ceHello [ESC]」と入力すると、現在のカーソル位置から単語の最後までの単語の一部が「Hello」に置き換えられ、その後コマンド「。」に置き換えられます。 このエントリを必要な回数繰り返すことができます。







移動コマンドは、より複雑で、さまざまな編集コマンドで補完することもできます（「o」-「現在の下に新しい行を作成」、「O」-「現在の上に行を作成」、「S」- 「行末まですべてを削除する」など。すべて入力モードになります。 「。」を押すだけで、どの編集パターンを作成して繰り返すことができるか想像してみてください！

誤解＃2：正規表現ではありません


Vi / Vimは正規表現処理で素晴らしい仕事をします。 多かれ少なかれ真面目なテキストエディターは正規表現をサポートしています-検索、置換など。しかし、vi（私が知っているものの）のみが「procを含む行の後に」「2番目の包含を検索して置換」のような複雑な組み合わせを使用できます。または他に思い浮かぶことは何でもあります。







したがって、正規表現の価値や、vi / vimがそれらを処理する能力を軽視したくありません。 しかし、viの真の力、それを使用する方法を学んだ後になければ生きられない力は、基本的なテキスト編集モデルの力です。







  • 1行または2行のコマンドで、行または画面上で必要な場所に正確に移動できます。
  • 'd'や 'c'などのコマンドは、移動コマンドと組み合わせると、目的のコードを直接変更でき、クリックするだけで編集を繰り返すことができます。
  • キーボード上の手は常に快適な位置にあります。 編集は、PCとラップトップの両方で同様に快適です。

誤解＃4：「hjkl」はテキスト内を移動しますか？


多くの人々は、矢印の代わりにhjklを使用することを奇妙に感じています。 この実装の本当の理由は、このような端末に矢がなくて、hjklキーが余分な負荷を運んだことのようです。 しかし、この配置の副作用は、快適な位置から手を離す必要がないことです。これは素晴らしいことです。







いずれにせよ、最初はhjklを使用してテキスト内を移動しますが、vi / vimを習得した場合、「h」および「l」キーの使用を拒否する可能性が高くなります。 なんで？ カーソルをテキスト内の適切な場所にすばやく移動できる、より強力なコマンドがあるためです。 行を編集するときは、常に希望の文字に直接移動する移動コマンドがあり、次を使用することがわかりました：「f」は、次の出現を検索したい文字で補完され、「％」は括弧間で移動するなど ファイルをナビゲートするときは、コマンドを使用して画面の上下/中央/下に直接移動できます。「/」でクイック検索、「]]」およびその兄弟を使用して機能をナビゲートできます。

それでは、正しいvi / vimの一般化された概念について説明しましょう。







**正しい概念＃1：急な学習曲線


vi / vimを効果的に使用する方法を学ぶことは容易ではないという事実です。 あなたは数週間から数ヶ月を費やす必要があり、最初の経験は快適ではありません。 これが、vi編集モデルが人気がなく、決して普及しない主な理由だと思います。 他の編集者のユーザーとのスピードで勝ち始める前に、約30チームを翻訳、学習、そしてワークアウトする必要があります。 ほとんどの場合、これらは単一ボタンのコマンドですが（ほとんどすべてに記憶を簡単にする簡単なニーモニックがあります）、これはまだ簡単な作業ではありません。 あきらめて、使い慣れたエディターに戻るのは非常に簡単です。 しかし、これらの条件を満たしていれば、古い習慣に戻ったであろう人は誰も知りません。







おわりに


あなたがしたいことをしてください。 あまりにも多くの労力を費やし、成果が得られないと思われる場合は、vi / vimの使用方法を学ばないでください。 viではなくemacsを学びます。 または、IDEの無駄なコードエディターを引き続き使用します。 しかし、いずれにしても、ユーザーviがバカだと宣言することはありません。 私は彼らが彼をとても好きな理由を説明することに成功したことを望みます、そして、たとえあなたがそれを使いたくなかったとしても、少なくとも彼の力に感謝すべきです。
