3Dモデル









別の興味深いこと。 モデルの読み込み。 私のライブラリは、テクスチャ付きの3Dモデルを受け入れ、名前に応じていくつかのパラメーターを個々の要素（メッシュ）に自動的に設定します。 実際、たとえばモデルの場合、影を落とすように設定されている場合、そのコンポジションに含まれる各メッシュによってキャストされます。 モデル全体が完全に影を落としたり、パフォーマンスに強く影響する他のプロパティを取得したりする必要は必ずしもありません。 したがって、各メッシュを個別に考慮する必要があることを示す特定のフラグをオンにすると、ロード時に、このメッシュまたはそのプロパティを持つメッシュと持たないメッシュを決定できます。 たとえば、家の平らな水平な屋根や、大きな背景に対してモデルの小さな無関係な細部の多くによって影を落とす必要はまったくありません。 それでも、プレーヤーはこれらの影を見ることができず、ビデオプロセッサの電力がそれらの影を処理するために使用されます。



これを行うには、グラフィックエディター（Blender、Maxなど）で、特定のルールに従って（オブジェクト名フィールドで）メッシュの名前をすぐに指定できます。 アンダースコア（_）が必要です。 条件付き制御文字は左側に配置する必要があります。たとえば、d-ダブルサイド（メッシュは双方向、そうでない場合は-片方向）、c（キャストシャドウ）-シャドウをキャスト、r（シャドウを受信）-シャドウを取得します。 つまり、たとえば、家のパイプメッシュの名前はcr_tubeになります。 他の多くの文字も使用されます。 たとえば、「l」はコライダーです。つまり、crl_wall01という名前の家の壁は、プレーヤーが自分自身を通り抜けることができず、影を落とします。 屋根やドアハンドルなどのコライダーを作成する必要はなく、それによってパフォーマンスが低下します。 既に理解しているように、モデルをロードするとき、私のライブラリはメッシュの名前を解析し、それらにシーン上の対応するプロパティを与えます。 ただし、このためには、3Dエディターからモデルをエクスポートする前に、すべてのメッシュに正しく名前を付ける必要があります。 これにより、パフォーマンスが大幅に節約されます。

アニメーション









ゲームオブジェクトの動きをアニメーション化するために、最も単純なエンジンが使用されます。これは、特定のフレームレート（60番目に制限されます）で、特定の時間間隔の数値パラメーターを変更し、中間値をハンドラーに渡します。 また、ハンドラーでは、たとえば、モデルが表示されます。



たとえば。 空間内のオブジェクトobjを位置（10; 10; 50）からポイント（100; 300; 60）に移動する必要があります。 初期値と最終値を指定して、3つのパラメーターを設定します。 x座標は10から100まで、yは10から300まで、zは50から60まで変化します。これはすべて、たとえば4秒で発生します。 

 m3d.lib.anim.add( 'moveobj', 1, 4000, 'and', {userpar1:111, obj:my3DObject}, [ {lim1:10, lim2:100, sstart:10, sfin:100, t:0}, {lim1:10, lim2:300, sstart:10, sfin:300, t:0}, {lim1:50, lim2:60, sstart:50, sfin:60, t:0} ], function(){ myPeriodicFun(this); }, function(){ myFinishFun(this); } ); m3d.lib.anim.play();


5つのパラメーターの最初の行：moveobj-アニメーションの名前（任意）、1-ストリーム番号（無制限の数のストリームでオブジェクトを並行してアニメーション化できます）、4000-アニメーション時間4秒、および-未使用のパラメーター（将来的に遷移ロジックを担当します）まで同じストリーム内のアニメーション間で、userparはパラメーターとしてハンドラーに渡される連想配列です。たとえば、計算された半径、正弦、余弦、および一般にこのアニメーションに対して計算された値で、これらを計算しない 各反復の時間について。 または、実際にアニメーション化する3Dオブジェクトを参照します。



次は、可変パラメーターを持つ配列です。 最初のパラメーターはx座標の変化、2番目はy、3番目はzであることがわかります。 lim1とlim2に、それぞれのサイズとサイズを変更します。 sstartとsfinでは、同じ値を指定します。 ここで、たとえば他の値から開始を指定できます。パラメータは、そこから円に「スクロール」し、lim2をバイパスし、lim1で新しい「回転」を開始します。 たとえば、アニメーションがいくつかの値（lim1とlim2）の間でループする場合、これは必要ですが、最初から（つまり、lim1からではなく）中間値から開始する必要があります。

ダイナミックシャドウ









シーン全体に影を表示することは非常に無駄であることが判明したため、影をオンにすると、fpsが数回低下しました。これはダメです。プレーヤーの周囲の特定の小さな正方形にオブジェクトの影を表示します。そのような手法は、フレームレートを大幅に向上させるとすぐに言います。



ここで複雑なことはありません。Three.jsシャドウは、ボックスで定義された特定のシャドウカメラ（shadowCameraLeft、shadowCameraRight、shadowCameraTop、shadowCameraBottom）内に投影されます。シーン全体に引き伸ばすことも、メインカメラに追従してその周辺のみに影を落とすように作成することもできます。このシステムに追加したのは、シャドウを更新するステップだけです。システムに計算がロードされるため、プレイヤーがひきつるたびにこの更新を行う必要はまったくありません。影が更新されるように、3つの軸のいずれかに沿って特定の最小距離を超えてください。



私のライブラリでは、3Dワールドを初期化するときに、contrオブジェクトが作成されます。このオブジェクトには、常にカメラの座標が含まれます。そこにカスタムパラメータcontr.cameraForShadowStepを設定することもできます。これには、シャドウカメラの位置が変化するカメラ位置のステップが含まれます。



たとえば、シャドウカメラの平行六面体の寸法が700x700x700の場合、contr.cameraForShadowStepは、たとえば20に設定できます。また、プレーヤーがいずれかの軸に沿って20移動すると、初期位置が再び記憶され、プレーヤーの周囲のシャドウが更新されます。 3Dエディターですべてのモデルが作成されたスケールに応じて、3Dワールドのスケールは任意です。また、700x700x700と20の代わりに、7000x7000x7000と200を使用する必要がある可能性があります。しかし、これは本質を変えるものではありません。



ところで、太陽が空を移動するとき、影の方向はそこで変化するはずなので、影はこのシステムとは無関係に更新されます。つまり、プレーヤーが移動せずに立っている場合でも更新されます。そこでは、空を更新する期間までに影を更新する機能が愚かに呼ばれています。
