かんばんボードの実験または物事を成し遂げる方法



長いダウンタイムの後、リソースが現れ始め、専門家、経営陣が変化と革新をサポートし、奨励することがあります。 これに加えて、会社のビジネスをどれだけ改善、最適化、真にサポートできるかという要望やアイデアが現れます。 もちろん、最初の呼吸で多くのことを行うことができます。特に最初の段階では、ほとんど生けられていないフィールドがあります。 しかし、後になって、ITインフラストラクチャが十分に成長し、変更や実装に関する決定が当初ほど迅速かつ容易に行われない瞬間が訪れます。 その理由はまさに実装と革新であり、今ではそれらをサポート、維持する必要があり、IT部門が成長したという事実にもかかわらず、十分なリソースがありません。大規模で複雑で機能するシステムがあります。 しかし、計画はどうですか？ それらのアイデアはどうですか？ まだたくさんあります。



次に、かんばんボードを使用した実験について説明します。かんばんボードは、IT企業から遠く離れたIT部門で優れた結果をもたらしました。 このようなツールを使用したワークフローは、より制御され管理しやすくなり、各専門家へのタスクの依存性がより透明になりました。 今、あなたは常に自信を持って部門が100％で働いていると言うことができ、達成される可能性が高い特定の結果に向かっています。



初期状態



異なる方向と資格の専門家が何人かいるとします。 私たちはいくつかの主要なタスクやプロジェクトに直面しており、私たちの目標は、スペシャリストとしての能力に応じて、すべての人が最大限の効率で参加できるようにプロセスを整理することです。 また、異なる頻度の従業員の雇用に影響を与える一定のルーチンが存在することにも留意する必要があります。



上記の現実は私たちの出発点です。 準備に移る...



設計



最初の段階では、各プロジェクトをアトミックコンポーネントに分割する必要があります。 理想的には、これらのタスクは、問題の一般的な本質を掘り下げることなく完了できるように区別して設定する必要があります。 さらに、各タスクは、特定のメトリックと指定された最終結果で、最初から最後まで形成される必要があります。



さらに、各プロジェクトのフレームワーク内で、各タスクの相互依存関係を設定します。 これを行うために、各タスクに番号を付け、依存している他のタスクを示します。 結果は、各プロジェクトの実装計画になります。



例：

  1. ESXi #2, Name: ServerABC, Guest: Ubuntu x64, RAM: 2 Gb, HDD: 20Gb Pre-allocated



  2. (after: #1) Ubuntu Server 12.04 x64 ServerABC (ESXi #2); HDD: root 2Gb, swap 2Gb, home ; IP: 192.168.0.7; hostname: ABC.com; user: abc pass: abc



  3. ...


プロジェクトの開発と計画の作業は、専門の専門家が簡単に実行できます。 この後、プロジェクトは頭に移り、専門家は現在の当面の責任の実施に従事し続けます。



チケットステッカー



準備が完了したら、いくつかのタスクの配列を手に入れ、その実装によって特定の目標を達成することができます。 視覚的な制御と管理としてかんばんボードを使用し、追跡と詳細な制御のためにチケットシステム（例： trac ）を使用します。



これらのツールを使用して、プロジェクトの実行を準備します。 各プロジェクトのチケットシステムで、準備された計画を配置する個別のチケットを作成します。 さらに、各タスクの作業を開始すると、その実行に付随するサブステッカーを作成します。



私たちの計画はすでに詳細に説明されており、チケットシステム（メインプロジェクトチケット内）に配置されているため、ボード上で視覚化するために、各タスクを簡単にステッカーに書き換えるだけです。 準備されたステッカーをボードに配置し、プロジェクトごとにグループ化します。



ステッカーの例：





アクションフィールド



準備が完了したら、イベントの開発を管理するかんばんを検討します。 左から右への「従来の」フローとは異なり、このステッカーは「プール」から始まる次の手順で低下します。







  • プールは、プロジェクトごとにグループ化されたタスクが配置されるボード上の最大の場所です。 プロジェクトは特定の色のエリアを形成するため、プロジェクトは必要なスペースのみを占めるように、エリアは意図的にセクションにマークされていません。 さらに、IT部門の日常生活における「プロジェクト」という用語は、多くの場合、サブタスクに分解できる大きなタスクを意味する場合があります。
  • キューは、即時の短期実行計画です。 ボードのこのセクションには、近い将来の実行が予定されているすべてのタスクが含まれており、主に高速プロジェクトまたは優先プロジェクトの実装で使用されます。 実際には、ほとんどのタスクではセクションはスキップされ、プールのタスクはエグゼキューターに直接送られます。
  • 実行 -この段階は、特定のアーティストに対応する領域に明確に分割されています。 請負業者を選択する際には、高度な資格を持つスペシャリストと初心者の両方にとって、いくつかのタスクの解決に同時に時間がかかる可能性があるという事実を考慮する必要があります。 ただし、パフォーマンスの最適化の目標は異なる場合があり、この段階で目標を達成するための戦略を明確に選択できます。 たとえば、初心者のスペシャリストを介して、要求の少ない資格や緊急のタスクをスキップして、自己開発の機会を与えることができます。 当然、緊急かつ重要な作業は訓練された専門家に向けられるべきです。 一般に、多くの戦略がありますが、最も重要なことは、プロセスがどのように進行しているかをリアルタイムで確認し、現在の作業状況にすばやく適応できることです。
  • 検証 -前の段階が実行者に分割されるのと同じように、それは本質的に補助的であり、品質保証プロセスに似ています。 たとえば、別のスペシャリストからのフィードバックを受け取ったり、最も能力のあるスペシャリストによる検証のために、タスクが配置されます。 プロジェクトが別のタスクで明示的にQAを提供する場合、またはサードパーティの検証が不要な場合も、この手順をスキップできます。
  • 完了 -完成したすべてのステッカーがプロジェクトの小さな本に集められる最終段階。




専門家の雇用と計画されたプロジェクトの作業への専門家の参加が不可能であることを明確に示すために、特定の色、たとえば赤の磁石を使用できます。 また、専門家が無料で、あらゆるプロジェクトに参加できることを緑色の磁石で示すこともできます。 多くの場合、緑色の磁石は、特定の瞬間に適切なタスクがない専門家にかかっています。そのようなタスクが見つかったらすぐに占有したいと考えています。



完了



私が提案したツールまたはその改善されたバージョンが、あなた自身またはビジネスがあなたの前に置いた興味深く楽しいプロジェクトの流れを適切かつ効率的に管理するのに役立つことを願っています。 頑張って
